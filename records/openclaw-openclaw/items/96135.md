## Summary

Keep open: current main still routes OpenAI batch audio through the `openai-audio-transcriptions` auth contract and rejects OAuth profiles before transcription. The open linked fix PR is a candidate path, but it is unmerged, so this issue remains necessary until maintainers land, revise, or decline that contract change.

## What This Changes

The issue asks OpenClaw to restore OAuth-backed OpenAI/Codex `tools.media.audio` batch transcription or explicitly validate and document that OpenAI batch STT is API-key-only.

## Best Possible Solution

Review the linked fix PR and choose the OpenAI batch STT contract explicitly: restore OAuth-backed batch transcription with regression coverage, or keep API-key-only batch STT and add targeted validation, doctor guidance, and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an OpenAI `tools.media.audio` entry with an OAuth `profile` resolves `modelApi: "openai-audio-transcriptions"`, then the shared OpenAI auth guard rejects that OAuth mode before transcription runs.

## Solution Assessment

Unclear. The linked PR is a plausible narrow repair, but the best final behavior depends on maintainer direction for whether OpenAI batch STT should support OAuth profiles or be documented and validated as API-key-only.
