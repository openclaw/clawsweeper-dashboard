## Summary

Keep open: current main and the latest release still route OpenAI batch media audio through the API-key-only `openai-audio-transcriptions` auth contract, so OAuth-profile `tools.media.audio` configs still need an explicit maintainer decision, validation/docs/doctor treatment, or restored OAuth-backed batch behavior.

## What This Changes

The issue reports OAuth-backed OpenAI batch audio transcription failing after provider migration made the media-understanding OpenAI audio path API-key-only.

## Best Possible Solution

Make the OpenAI batch STT contract explicit: either restore an owner-approved OAuth-backed batch transcription path, or keep API-key-only batch STT and add targeted validation, docs, or doctor guidance for OAuth-profile audio configs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an OpenAI `tools.media.audio` entry with an OAuth `profile` resolves `modelApi: "openai-audio-transcriptions"`, then `model-auth` rejects non-API-key modes before transcription runs.

## Solution Assessment

Unclear. The failure is real, but the best fix depends on maintainer direction: restore OAuth-backed batch transcription if that remains a supported contract, or document, validate, and migrate the API-key-only OpenAI batch STT behavior.
