## Label Justifications

- `P2`: This is a real regression in a bounded OpenAI OAuth audio-transcription path, but it is limited to users relying on OAuth-only batch STT rather than all provider or channel traffic.

- `impact:auth-provider`: The failure is caused by OpenAI provider auth-mode routing rejecting OAuth profiles for the audio transcription model API.

## Summary

Keep open: current main and the latest release still use the API-key-only guard for OpenAI batch audio transcription, and the issue is now paired with an open closing PR that needs review rather than a separate cleanup close.

## What This Changes

The issue asks OpenClaw to either restore OAuth-backed OpenAI batch audio transcription for `tools.media.audio` or clearly validate, document, and migrate users to an API-key-only contract.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/97280 or an equivalent fix that restores native OpenAI OAuth batch STT with endpoint trust bounds, or explicitly document, validate, and migrate users to an API-key-only contract.

## Reproduction Assessment

Yes from source inspection, but not from a live local OAuth run in this read-only review. Current main and `v2026.6.10` route OpenAI audio through `openai-audio-transcriptions`, then reject OAuth profiles through the shared OpenAI API-key-only guard.

## Solution Assessment

No final solution is on main yet. The open candidate PR is the right work item to review, but the best fix still needs maintainer acceptance of the OAuth batch-STT contract and endpoint-trust bounds.
