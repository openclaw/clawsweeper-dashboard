## Summary

Keep open: current main still rejects the reported OAuth-profile OpenAI batch audio path before transcription, and the remaining fix requires a maintainer decision on whether batch STT should support OAuth or stay API-key-only with explicit validation/docs/doctor guidance.

## What This Changes

The issue reports that OAuth-profile OpenAI batch audio transcription through `tools.media.audio` now fails after the provider/auth migration requires an OpenAI Platform API-key profile.

## Best Possible Solution

Make the OpenAI batch STT contract explicit: either restore and prove an owner-approved OAuth-backed batch route, or keep API-key-only batch STT and add targeted validation/docs/doctor guidance for OAuth-profile audio configs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an OpenAI `tools.media.audio` entry with an OAuth `profile` resolves `modelApi: "openai-audio-transcriptions"`, then `model-auth` rejects non-API-key modes before transcription runs.

## Solution Assessment

Unclear. The failure is real, but the best fix depends on maintainer direction: restore and prove OAuth-backed batch transcription if that remains supported, or document/validate/migrate the API-key-only OpenAI batch STT behavior.
