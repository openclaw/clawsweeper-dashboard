## Label Justifications

- `P2`: The issue is a bounded regression in OpenAI media-audio auth rather than a core runtime outage.
- `impact:message-loss`: Affected users lose usable transcript text from inbound voice notes or batch audio when the configured OAuth profile is rejected before transcription.
- `impact:auth-provider`: The failure is specifically in OpenAI provider auth routing between OAuth profiles and API-key-only model API contracts.

## Summary

Keep open: current main and the latest release still route OpenAI batch audio through the API-key-only `openai-audio-transcriptions` auth guard, and the only direct fix candidate is the open, unmerged PR at https://github.com/openclaw/openclaw/pull/97280.

## What This Changes

The issue requests restored or explicitly documented handling for OAuth-backed OpenAI/Codex batch audio transcription after the provider migration made `tools.media.audio` require an OpenAI Platform API-key profile.

## Best Possible Solution

Review and resolve https://github.com/openclaw/openclaw/pull/97280: either restore OAuth-backed batch STT with endpoint/auth safeguards, or keep batch STT API-key-only and add explicit validation, docs, and doctor guidance.

## Reproduction Assessment

Yes. Current main resolves OpenAI audio to `modelApi: "openai-audio-transcriptions"`, passes that into `resolveApiKeyForProvider`, and `model-auth` only permits OAuth for `openai-chatgpt-responses`, so an OAuth audio profile is rejected before transcription dispatch.

## Solution Assessment

Unclear. The failure is real, but the best fix depends on maintainer direction: restore OAuth-backed batch STT with endpoint/auth safeguards, or keep API-key-only batch STT and add validation/docs/doctor guidance for existing OAuth-profile audio configs.
