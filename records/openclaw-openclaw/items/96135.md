## Summary

Keep open: current main and the latest release intentionally route OpenAI batch audio through the API-key-only `openai-audio-transcriptions` contract, but users with existing OAuth-backed `tools.media.audio` configs still lack clear docs or doctor/config diagnostics before runtime failure.

## What This Changes

The issue asks OpenClaw to either restore OAuth-backed OpenAI batch STT or clearly document and warn/migrate that `tools.media.audio` OpenAI batch transcription is API-key-only.

## Best Possible Solution

Keep API-key-only OpenAI batch STT as the current contract, then add focused docs and doctor/config diagnostics for OAuth-backed `tools.media.audio` OpenAI entries so upgraded users get an actionable warning before runtime failure.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure an explicit OpenAI OAuth profile for `tools.media.audio` and current main passes `openai-audio-transcriptions` to `resolveApiKeyForProvider`, which rejects that profile before transcription.

## Solution Assessment

Yes for the narrow repair path. The current OpenAI Platform audio contract is already established by merged source and tests, so the maintainable fix is docs plus doctor/config diagnostics rather than adding a second OAuth batch STT route without new maintainer direction.
