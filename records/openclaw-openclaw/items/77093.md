## Summary

Keep open: current main still delegates public Gmail Pub/Sub push processing to `gog`, and neither current OpenClaw source nor the changelog shows a fix for the reported watcher-side 500. The reporter's follow-up usefully confirms direct `/hooks/gmail` succeeds while the tokenized watcher endpoint returns 500, so the remaining investigation belongs at the watcher/gog boundary.

## What This Changes

The issue reports that real Gmail Pub/Sub pushes reach Google Pub/Sub but fail to process through OpenClaw's Gmail watcher endpoint in a Docker plus Tailscale Funnel deployment.

## Best Possible Solution

Capture the watcher-side `gog` error from the same Docker plus Tailscale Funnel setup, then fix the narrow watcher invocation/config/state diagnostics in OpenClaw or route an upstream `gogcli` fix if the dependency owns the failure.

## Reproduction Assessment

No; there is no high-confidence live reproduction against current main because it requires Gmail, Google Pub/Sub, Tailscale Funnel, and the watcher error log. Source inspection and the reporter's direct POST results do give a clear failing boundary at `gog gmail watch serve`.

## Solution Assessment

Unclear; changing Gateway `/hooks/gmail` is not the best target because direct internal posts already return 200 and are tested. The narrow fix should target the watcher invocation, config/state diagnostics, or upstream `gogcli` after the exact `handlePush` failure is known.
