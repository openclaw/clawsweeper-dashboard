## Summary

Keep open: current main still relies on the external `gog gmail watch serve` process for Pub/Sub delivery, and the reporter's follow-up further narrows the failure to that watcher boundary rather than proving it is fixed.

## What This Changes

The issue requests that real Gmail Pub/Sub pushes delivered through Docker and Tailscale Funnel be accepted by the OpenClaw Gmail webhook path instead of failing at the local watcher endpoint.

## Best Possible Solution

Keep this open and narrow follow-up to the Gmail watcher boundary: capture the watcher error, then patch OpenClaw watcher args/diagnostics/docs or coordinate a `gogcli` fix based on that failure mode.

## Reproduction Assessment

No; there is not a high-confidence live reproduction against current main. The source path and reporter's live evidence point to the watcher boundary, but confirming the bug requires Gmail, Pub/Sub, Tailscale Funnel, and the exact `gog` error line.

## Solution Assessment

Unclear; a Gateway hook change is probably not the narrow fix because `/hooks/gmail` already accepts the internal payload. The best next repair depends on whether `gog` fails while validating the push, reading Gmail history, fetching messages, or forwarding to OpenClaw.
