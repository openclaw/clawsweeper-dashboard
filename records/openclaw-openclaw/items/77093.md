## Summary

Keep open: current main still routes Gmail Pub/Sub pushes through `gog gmail watch serve`, and the added reporter evidence narrows the failure to that watcher boundary rather than proving the Docker plus Tailscale Funnel path is fixed.

## What This Changes

The issue reports that real Gmail Pub/Sub pushes reach Google Pub/Sub but the OpenClaw Gmail watcher endpoint returns 500 before forwarding to the internal Gmail hook in a Docker plus Tailscale Funnel deployment.

## Best Possible Solution

Keep Gmail Pub/Sub in core and narrow the repair to the watcher boundary: capture the gog handle-push error, then update OpenClaw integration, diagnostics, docs, or the upstream dependency as indicated.

## Reproduction Assessment

No; we do not have a high-confidence live reproduction against current main. The provided Docker/Tailscale evidence and current source make the watcher path clear, but the exact current-main `gog` handle-push failure is still missing.

## Solution Assessment

Unclear; changing the Gateway hook handler is probably not the best fix because `/hooks/gmail` works and is covered by tests. The narrow solution should start with watcher stderr and then patch OpenClaw watcher integration or upstream gogcli as appropriate.
