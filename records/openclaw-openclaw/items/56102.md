## Summary

Keep open. Current main still passes ACP initialize input directly into the ACP SDK, the pinned SDK rejects date-string `protocolVersion` values before OpenClaw's translator runs, and an open closing PR still owns the proposed fix path.

## What This Changes

The issue asks OpenClaw's ACP bridge to accept VS Code/Cursor initialize requests that send MCP-style date-string `protocolVersion: "2025-11-25"`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56176 with a current-main ACP initialize stream normalizer before SDK validation, plus focused regression coverage and redacted VS Code or Cursor proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: VS Code/Cursor send `protocolVersion: "2025-11-25"`, current main passes initialize directly into SDK 0.21.0, and that SDK schema requires a numeric uint16 version before `AcpGatewayAgent.initialize()` runs.

## Solution Assessment

Yes. A stream-boundary normalizer before `AgentSideConnection` is the narrow maintainable fix, but this issue should stay open until the linked PR lands or is replaced.
