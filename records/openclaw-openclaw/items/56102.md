## Summary

Keep open. Current main still passes the ACP stdio stream directly into the ACP SDK, whose current schema requires an integer `protocolVersion`, so the reported VS Code/Cursor date-string initialize request remains a plausible current failure. The issue is also paired with open PR #56176 using closing syntax, so it should stay open until that PR or an equivalent fix is reviewed and landed.

## What This Changes

The issue asks OpenClaw's ACP bridge to interoperate with VS Code/Cursor initialize requests that send `protocolVersion: "2025-11-25"` instead of the ACP SDK's numeric protocol version.

## Best Possible Solution

Keep the issue open while maintainers review PR #56176 or an equivalent patch. If ACP compatibility is intended, the fix should adapt only the incoming `initialize.protocolVersion` before ACP SDK validation, preserve valid numeric ACP requests unchanged, and add regression coverage; if not, OpenClaw should provide a clear MCP-client path through `openclaw mcp serve` instead of failing with an SDK validation error.

## Reproduction Assessment

Yes. The issue log plus current-main code and the ACP SDK 0.21.0 schema provide a high-confidence reproduction path: send an `initialize` request with `protocolVersion: "2025-11-25"` into `openclaw acp`, and SDK validation rejects it before `AcpGatewayAgent.initialize()` can respond.

## Solution Assessment

Unclear until maintainer review of the product boundary. A narrow pre-validation compatibility layer is the most direct code fix for the reported command, but the documented MCP-server path means maintainers should explicitly choose compatibility versus steering MCP clients to `openclaw mcp serve`.
