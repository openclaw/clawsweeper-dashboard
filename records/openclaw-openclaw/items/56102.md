## Summary

Keep open. Current main still has the source-level rejection path the prior ClawSweeper comment described: `openclaw acp` feeds the raw stdio stream into `AgentSideConnection`, while SDK 0.21.0 still validates `initialize.params.protocolVersion` as a number, and PR #56176 remains open/unmerged with closing syntax.

## What This Changes

The issue asks OpenClaw's ACP bridge to accept VS Code/Cursor initialize requests that send MCP-style date-string `protocolVersion: "2025-11-25"`.

## Best Possible Solution

Land or replace PR #56176 with a current-main ACP stream compatibility layer before SDK validation plus focused regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an `initialize` request with `protocolVersion: "2025-11-25"` reaches SDK `zInitializeRequest` validation before OpenClaw's translator runs, and SDK 0.21.0 requires a numeric protocol version. I did not run a live VS Code/Cursor session in this checkout.

## Solution Assessment

Yes, the narrow maintainable fix is to adapt only the incoming ACP initialize stream before `AgentSideConnection` validation, while preserving valid numeric ACP requests unchanged. A broader MCP-loopback or gateway protocol change would touch the wrong surface.
