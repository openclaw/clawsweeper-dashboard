## Summary

Keep open. Current main still has the same source-level rejection path: `openclaw acp` passes the raw stdio stream into the ACP SDK connection, the pinned SDK still validates `initialize.params.protocolVersion` as an integer, and the linked fixing PR remains open rather than merged.

## What This Changes

The issue reports that `openclaw acp` rejects VS Code/Cursor initialize requests using MCP date-string `protocolVersion: "2025-11-25"`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56176 with a current-main ACP initialize stream normalizer before SDK validation, plus focused regression coverage and real VS Code/Cursor proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a VS Code/Cursor initialize request with string `protocolVersion: "2025-11-25"` reaches the SDK `zInitializeRequest` parser before OpenClaw's translator can respond, and SDK 0.21.0 requires an integer.

## Solution Assessment

Yes, the narrow maintainable direction is to normalize only the incoming ACP initialize stream before `AgentSideConnection` validation while preserving valid numeric ACP requests unchanged. The existing PR is the active candidate, but it still needs review and proof before the issue should close.
