## Summary

Keep open. Current `main` still has the same ACP initialize rejection path: `openclaw acp` passes the raw NDJSON stream into the ACP SDK, the pinned SDK still requires numeric `initialize.protocolVersion`, and the closing PR remains open rather than merged.

## What This Changes

The issue reports that `openclaw acp` rejects VS Code/Cursor MCP initialize requests using date-string `protocolVersion: "2025-11-25"`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56176 with a current-main ACP initialize normalizer before SDK validation, focused regression coverage, and redacted VS Code or Cursor proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: VS Code/Cursor send string `protocolVersion: "2025-11-25"`, current `openclaw acp` passes initialize into SDK 0.21.0 unchanged, and that SDK schema requires a uint16 number.

## Solution Assessment

Yes. The narrow maintainable fix is a current-main stream-boundary normalizer before `AgentSideConnection` validation while preserving valid numeric ACP initialize requests; the issue should stay open until the linked PR is landed or replaced.
