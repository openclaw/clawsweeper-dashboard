## Summary

Keep open. Current main still passes the ACP initialize message into the ACP SDK before any OpenClaw normalization, the pinned SDK still requires numeric initialize.protocolVersion, and the linked closing PR remains open rather than merged.

## What This Changes

The issue requests VS Code/Cursor compatibility when MCP initialize sends date-string protocolVersion values to the ACP server.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56176 with a current-main ACP initialize normalizer before SDK validation, matching the SDK integer/range contract and backed by focused regression coverage plus redacted VS Code or Cursor proof.

Yes. Source inspection gives a high-confidence path: VS Code/Cursor send protocolVersion "2025-11-25", current openclaw acp passes initialize into SDK 0.21.0 unchanged, and that SDK schema requires a uint16 number.

## Solution Assessment

Yes. A stream-boundary normalizer before AgentSideConnection validation is the narrow maintainable fix, but this issue should stay open until the linked PR is landed or replaced.
