## Summary

Keep open. Current main and the latest release still pass ACP initialize frames directly into the ACP SDK, whose 0.21.1 schema rejects the reported date-string `protocolVersion` before OpenClaw's initialize handler runs; the open closing PR remains the right follow-up path but is not merged.

## What This Changes

The issue reports that VS Code 1.113 and Cursor send MCP date-string `protocolVersion: "2025-11-25"`, which OpenClaw ACP rejects during initialize.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56176 with a current-main ACP initialize stream normalizer before SDK validation, matching the SDK integer/range contract and adding focused regression coverage plus redacted VS Code or Cursor proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: VS Code/Cursor send `protocolVersion: "2025-11-25"`, current main passes initialize params directly into SDK validation, and SDK `0.21.1` requires a numeric integer in the 0..65535 range before OpenClaw's initialize handler runs.

## Solution Assessment

Yes. A stream-boundary normalizer before `AgentSideConnection` is the narrow maintainable fix, but this issue should stay open until the linked PR lands or is replaced with equivalent tested behavior.
