## Summary

Keep open. Current main and v2026.5.22 still pass ACP initialize frames directly into the SDK, where string `protocolVersion` values are rejected before OpenClaw's handler runs, and the linked closing PR remains open and conflicting.

## What This Changes

The issue reports that `openclaw acp` rejects VS Code/Cursor initialize requests when `protocolVersion` is the MCP date string `2025-11-25`.

## Best Possible Solution

Refresh, land, or replace the linked PR with a current-main stream-boundary normalizer before `AgentSideConnection`, matching the SDK uint16 contract and adding focused regression coverage plus redacted VS Code or Cursor proof.

## Reproduction Assessment

Yes, source-level. Reporter logs show VS Code/Cursor sending a string `protocolVersion`, current main passes initialize params directly to `AgentSideConnection`, and the SDK requires a uint16 number before OpenClaw's handler runs.

## Solution Assessment

Yes. The narrow maintainable fix is a pre-`AgentSideConnection` stream normalizer that accepts MCP date-string initialize versions while preserving valid ACP numeric clients and matching the SDK contract.
