## Summary

Keep open: the PR targets a valid user-visible auth-error bug and has focused tests/changelog, but the diff overmatches the existing `auth` failover reason and would show hard-coded HTTP 401 copy for 403 or no-status auth failures.

## What This Changes

The PR adds an `auth_invalid_token` runtime-failure kind, formats matching embedded assistant errors as a re-authentication hint, adds formatter regression tests for 401 variants, and updates the changelog.

## Best Possible Solution

Keep the fallback-copy fix, but gate the new classification to actual HTTP 401/token-expired shapes or make the copy status-neutral, while leaving provider-owned token refresh as a separate follow-up.

## Reproduction Assessment

Yes for the narrow formatter path from source: current main sends `HTTP 401: "Invalid token"` through the raw HTTP formatter, and the PR adds direct unit coverage for that payload. I did not establish a live Feishu/AutoGLM expired-token run.

## Solution Assessment

No as written. Sanitizing the fallback copy is the right narrow fix, but mapping every `auth` failover reason to an HTTP-401-specific kind is too broad; constrain it to real 401/token-expired shapes or use status-neutral copy.
