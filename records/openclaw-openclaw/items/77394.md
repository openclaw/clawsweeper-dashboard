## Summary

Keep open: the patch is directionally right for the reported raw 401 leak, but the new auth branch also catches existing generic 403 auth classifications and would show HTTP 401/token-expired copy for non-401 provider failures.

## What This Changes

The PR adds an `auth_invalid_token` runtime-failure kind for plain provider auth errors, maps it to friendly embedded-agent re-authentication copy, adds formatter coverage, and updates the changelog.

## Best Possible Solution

Keep the narrow user-visible sanitizer, but make the new classification status-aware so 401 token/auth failures get the re-auth hint while existing 403 access failures retain accurate copy or receive their own tested message.

## Reproduction Assessment

Yes from source inspection: current main has no plain HTTP 401 auth-copy branch, and the raw HTTP fallback is reachable for the reported provider error shape. The PR-introduced 403 regression is also source-reproducible through the existing 401/403 generic auth classifier.

## Solution Assessment

No, not as submitted. The maintainable fix is to guard the new `auth_invalid_token` path to HTTP 401, or split 401 and 403 into separate copy with regression tests for both statuses.
