The PR updates reply-payload normalization and tests so outbound replies can map filePath, path, and media aliases into mediaUrl.

## Best Possible Solution

## Reproduction Assessment

Yes, by source inspection. recordChannelMessageReplyDispatch normalizes loose reply payloads before delivery, and current main only preserves mediaUrls/mediaUrl, so alias-only filePath, path, media, or fileUrl payloads are dropped at that boundary.

## Solution Assessment

No, not as written. The SDK helper is the right boundary, but the patch should mirror the established send-action alias contract and include fileUrl instead of defining a conflicting reply-only precedence rule.
