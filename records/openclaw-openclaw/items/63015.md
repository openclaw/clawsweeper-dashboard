## Summary

Keep this PR open: current main still has the outbound reply media-alias gap, and the PR targets the right helper, but the proposed fallback contract diverges from the shared send action and omits the current `fileUrl` alias before merge.

## What This Changes

The PR changes `normalizeOutboundReplyPayload` to populate `mediaUrl` from `mediaUrl`, `filePath`, `path`, or `media`, and adds unit coverage for those fallback cases.

## Best Possible Solution

Finish this as a narrow plugin SDK reply-normalization fix: make final reply media aliases match the shared send-action contract, cover each supported alias and precedence case in `reply-payload.test.ts`, and add the required changelog entry without changing channel-specific handlers.

## Reproduction Assessment

Yes. On current main, `normalizeOutboundReplyPayload({ filePath: "/tmp/photo.jpg" })` normalizes to a payload with `mediaUrl: undefined`, and inbound final reply dispatch uses that normalizer before delivery.

## Solution Assessment

No, not as written. The right repair boundary is `src/plugin-sdk/reply-payload.ts`, but the proposed alias order should match the shared send action or be explicitly documented and tested as a different contract, and `fileUrl` parity should be included.
