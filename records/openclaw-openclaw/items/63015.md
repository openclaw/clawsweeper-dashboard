## Summary

Keep this PR open. Current main still drops loose outbound reply media aliases, so the PR is not obsolete, but the proposed fallback order diverges from the shared send-action path and should be resolved before merge.

## What This Changes

The PR changes `normalizeOutboundReplyPayload` to populate `mediaUrl` from `mediaUrl`, `filePath`, `path`, or `media`, and adds unit tests for those fallback cases.

## Best Possible Solution

Finish the PR as a narrow plugin SDK reply-normalization fix: use one canonical media-alias order that matches the shared send action unless maintainers explicitly choose a different contract, include regression coverage for every supported alias, and keep the change limited to reply payload normalization and tests.

## Reproduction Assessment

Yes. On current main, a static reproduction is `normalizeOutboundReplyPayload({ filePath: "/tmp/photo.jpg" })`, which leaves `mediaUrl` undefined because the implementation reads only `payload.mediaUrl`; inbound reply dispatch uses that normalizer before channel delivery.

## Solution Assessment

No, not as written. Normalizing the aliases in `src/plugin-sdk/reply-payload.ts` is the right repair boundary, but the patch should align or deliberately document/test precedence against the shared send-action path and consider `fileUrl` parity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Align media alias precedence with send action:** `src/plugin-sdk/reply-payload.ts:40-49`
  - body: The new fallback chain chooses `mediaUrl -> filePath -> path -> media`, but the shared message-send path currently resolves `media -> mediaUrl -> path -> filePath -> fileUrl`. If a final reply contains multiple aliases, auto-reply delivery and tool-send delivery can select different attachments, and `fileUrl` remains unsupported. Please either use the shared order here or explicitly lock and test the intended different contract.
