## Summary

Keep open: current main still drops alternate media aliases in loose outbound reply normalization, but this PR is not merge-ready because its alias order and missing `fileUrl` support diverge from the established send-action contract, and contributor real behavior proof is still missing.

## What This Changes

The PR changes `normalizeOutboundReplyPayload` and unit tests so loose `filePath`, `path`, and `media` fields can populate `mediaUrl`.

## Best Possible Solution

Refresh or replace the branch with a narrow reply normalizer fix that mirrors the send-action alias contract, keeps current rich reply fields, adds precedence and `fileUrl` regression coverage, and includes redacted real behavior proof.

## Reproduction Assessment

Yes by source inspection, not by an executed test: a loose reply payload with only `filePath` or `path` reaches `recordChannelMessageReplyDispatch`, and current main only preserves `mediaUrl`/`mediaUrls` before delivery.

## Solution Assessment

No, not as written. The helper is the right ownership boundary, but the patch should align with the existing send-action alias contract, include `fileUrl`, preserve newer rich reply fields after rebase, and provide real behavior proof.
