## Summary

Keep open: current main still has the reply-normalizer attachment gap, but this PR cannot merge as written because its alias precedence diverges from the send-action contract, omits `fileUrl`, and still lacks real behavior proof.

## What This Changes

The PR changes `src/plugin-sdk/reply-payload.ts` and tests so loose outbound reply payloads can populate `mediaUrl` from `mediaUrl`, `filePath`, `path`, or `media`.

## Best Possible Solution

Refresh or replace the branch with a narrow SDK reply-normalizer fix that mirrors the send-action alias contract, includes `fileUrl`, preserves current rich reply fields, adds focused regression coverage, and includes redacted real gateway/channel proof.

## Reproduction Assessment

Yes, by source inspection rather than an executed test: `recordChannelMessageReplyDispatch` normalizes loose reply payloads before delivery, and current main only preserves `mediaUrls`/`mediaUrl` there.

## Solution Assessment

No, not as written. The helper is the right boundary, but the diff should match the established send-action alias contract and include `fileUrl` instead of defining a conflicting precedence rule.
