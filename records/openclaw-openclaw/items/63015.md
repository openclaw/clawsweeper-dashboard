## Summary

Keep open: the PR targets a real reply-normalization bug on current main, but the proposed alias contract still diverges from the established send path and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR changes `normalizeOutboundReplyPayload` to derive `mediaUrl` from `mediaUrl`, `filePath`, `path`, or `media`, and adds unit tests for those fallback cases.

## Best Possible Solution

Finish this as a narrow SDK reply-normalization fix that matches or shares the send-action media alias contract, including `fileUrl` and precedence coverage, then add redacted real gateway proof.

## Reproduction Assessment

Yes. By source inspection, a loose final reply payload with only `filePath`, `path`, `media`, or `fileUrl` reaches `normalizeOutboundReplyPayload`, and current main leaves `mediaUrl` undefined for those aliases.

## Solution Assessment

No, not as written. The change belongs in the right helper, but it should align with the current send-action alias order and `fileUrl` support instead of creating a second reply-only precedence rule.
