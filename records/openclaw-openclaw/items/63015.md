## Summary

Keep open: the bug is real on current main, but this PR is not merge-ready because its alias precedence diverges from the existing send-action contract, omits `fileUrl`, and lacks after-fix real behavior proof.

## What This Changes

The PR changes `normalizeOutboundReplyPayload` to derive `mediaUrl` from `mediaUrl`, `filePath`, `path`, or `media`, and adds unit tests for those fallbacks.

## Best Possible Solution

Finish this as a narrow plugin SDK reply-normalization fix that mirrors or shares the send-action media alias contract, covers `media`, `mediaUrl`, `path`, `filePath`, `fileUrl`, and precedence, then verify a real gateway attachment send.

## Reproduction Assessment

Yes, by source inspection. On current main, a loose final reply payload with only `filePath` reaches `normalizeOutboundReplyPayload` through inbound reply dispatch and leaves `mediaUrl` undefined.

## Solution Assessment

No, not as written. The change is in the right file, but it should match the existing send-action alias order and include `fileUrl` instead of creating a second reply-only precedence rule.
