## What This Changes

The PR changes outbound reply normalization and tests so loose reply payloads can use `filePath`, `path`, or `media` as media fallbacks in addition to `mediaUrl`.

## Best Possible Solution

Port the alias handling to the shared outbound reply normalizer, align or explicitly document alias precedence against the message-tool send path, and add real gateway/channel proof that alias-only media replies deliver attachments.

## Reproduction Assessment

Yes at source level: current main normalizes loose outbound replies through a shared normalizer that reads only `mediaUrl`, while docs and the message-tool path accept `filePath`, `path`, and `media`. I did not run a live gateway or WhatsApp reproduction in this read-only review.

## Solution Assessment

No as submitted: the fix idea is narrow and useful, but the branch targets the old inline SDK implementation and lacks real behavior proof. The best fix is a current-main port to the shared normalizer with precedence tests and channel proof.
