## Summary

Keep open: this PR targets a real source-reproducible outbound attachment drop, but the patch diverges from the established send-action media alias contract, omits `fileUrl`, is dirty against current main, and still lacks real behavior proof.

## What This Changes

The branch changes `src/plugin-sdk/reply-payload.*` to map loose outbound reply `filePath`, `path`, and `media` fields into `mediaUrl`, with unit tests for those aliases.

## Best Possible Solution

Refresh or replace this branch with a narrow reply normalizer fix that mirrors the send-action alias order including `fileUrl`, preserves current rich reply fields, coordinates with https://github.com/openclaw/openclaw/pull/67652, and includes redacted real attachment-send proof.

## Reproduction Assessment

Yes, by source inspection. `recordChannelMessageReplyDispatch` normalizes loose final reply objects before delivery, and current main only preserves `mediaUrls`/`mediaUrl`, so alias-only attachment fields are dropped there.

## Solution Assessment

No, not as written. The SDK helper is the right boundary, but the fix should mirror the established send-action alias contract and include `fileUrl` instead of adding a conflicting reply-only precedence rule.
