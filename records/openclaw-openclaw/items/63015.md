## Summary

Keep open: this PR targets a real source-reproducible attachment drop in the reply normalizer, but the proposed patch still diverges from the existing send-action media alias contract and lacks after-fix real behavior proof.

## What This Changes

The PR changes `normalizeOutboundReplyPayload` to derive `mediaUrl` from `mediaUrl`, `filePath`, `path`, or `media`, and adds unit tests for those fallback cases.

## Best Possible Solution

Finish this as a narrow plugin SDK reply-normalization fix that mirrors or shares the send-action media alias contract, covers `media`, `mediaUrl`, `path`, `filePath`, `fileUrl`, and precedence, then verify a real gateway attachment send.

## Reproduction Assessment

Yes, by source inspection. On current main, a loose final reply payload with only `filePath` reaches `normalizeOutboundReplyPayload` through inbound reply dispatch and leaves `mediaUrl` undefined.

## Solution Assessment

No, not as written. The change is in the right file, but the alias order and missing `fileUrl` support should be corrected so reply delivery matches the existing send-action contract instead of creating a second precedence rule.
