## Summary

Keep open. Current main still converts OpenRouter `reasoning_details` and `reasoning` deltas into literal field-name `thinkingSignature` values and then preserves those fields on OpenRouter replay; that matches the reporter's poisoned-history shape and is not covered by the earlier empty-`reasoning_content` fix.

## What This Changes

The issue reports OpenRouter `openai-completions` reasoning history being persisted with field-name sentinels and replayed on the next turn, causing reasoning models to fail.

## Best Possible Solution

Fix the OpenRouter passthrough parser/replay boundary so valid full `reasoning_details` arrays survive when required, raw `reasoning` strings are replayed only for models that support them, and literal field-name sentinels are stripped or downgraded before history replay.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only pass. Source inspection gives a high-confidence reproduction path: current main writes `reasoning_details`/`reasoning` field names into `thinkingSignature` and tests assert those values are replayed on OpenRouter follow-up requests.

## Solution Assessment

Yes, the best direction is a narrow OpenRouter/openai-completions parser and replay-policy fix, not a new config option. A blanket sentinel strip is a useful defensive fence, but the durable fix should preserve real OpenRouter `reasoning_details` structures where the provider contract requires them.
