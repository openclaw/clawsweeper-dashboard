## Summary

Keep open: current main still has a source-level OpenRouter completions path that stores/replays reasoning field names as reasoning metadata, and the upstream OpenRouter contract distinguishes plaintext `reasoning` from structured `reasoning_details` arrays that should be preserved unmodified when needed.

## What This Changes

Fix OpenRouter `openai-completions` passthrough so reasoning metadata is not persisted or replayed as literal field-name signatures that break multi-turn reasoning models.

## Best Possible Solution

Add a narrow OpenRouter completions parser/replay fix that preserves real structured `reasoning_details` when valid, strips or avoids replaying field-name sentinels for plaintext reasoning, and covers Anthropic/xAI plus related passthrough replay cases with regression tests.

## Reproduction Assessment

Yes at source level: the issue gives a concrete two-turn OpenRouter reasoning-model path, and current main still maps `reasoning_details`/`reasoning` into stored `thinkingSignature` sentinels that are replayed. I did not run the live WhatsApp/OpenRouter repro in this read-only pass.

## Solution Assessment

No, the current implementation is not the best solution because it conflates OpenRouter field names with replayable reasoning payloads. The maintainable fix should align the OpenRouter parser and replay sanitizer with OpenRouter's documented `reasoning` versus `reasoning_details` contracts.
