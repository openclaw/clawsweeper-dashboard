## Summary

Keep open: current main still has a source-level OpenRouter completions path that can store and replay reasoning field names as reasoning metadata, and the related merged fixes only cover adjacent empty/unsupported reasoning replay cases.

## What This Changes

The issue requests fixing OpenRouter openai-completions reasoning normalization so `reasoning` and `reasoning_details` field names are not stored or replayed as `thinkingSignature` values.

## Best Possible Solution

Fix the OpenRouter parser and replay sanitizer so plaintext reasoning is stored without a replayable field-name signature, structured `reasoning_details` preserves only valid provider payloads, and sentinel values are stripped before replay with focused regression coverage.

## Reproduction Assessment

Yes at source level: the issue gives a concrete two-turn OpenRouter reasoning-model path, and current main still maps `reasoning_details` or `reasoning` into replayed sentinel metadata. I did not run the live WhatsApp/OpenRouter repro in this read-only review.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it conflates OpenRouter field keys with replayable reasoning payloads. A narrow parser/replay fix should align with OpenRouter's `reasoning` versus `reasoning_details` contract.
