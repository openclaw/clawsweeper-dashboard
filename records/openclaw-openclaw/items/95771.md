## Summary

Keep open: the warning-only cron observability change is useful, but one new warning is misleading on cron schedules and the PR body only provides mocked Vitest proof rather than real behavior proof.

## What This Changes

The PR adds warning logs and regression coverage for cron paths where `nextRunAtMs` becomes `undefined` after completion, transient-error retry, or maintenance recompute.

## Best Possible Solution

Keep the observability work, but make the transient-error warning schedule-specific and require redacted real runtime or terminal proof before merge.

## Reproduction Assessment

Yes. Source inspection of the proposed diff shows a cron error with `normalNext === undefined` reaches the new warning before current main clears the schedule through `resolveCronNextRunWithLowerBound`.

## Solution Assessment

No. Adding warnings is the right direction, but the transient-error warning should be moved to the actual non-cron fallback or reworded/gated so cron keeps one accurate unresolved-schedule diagnostic.
