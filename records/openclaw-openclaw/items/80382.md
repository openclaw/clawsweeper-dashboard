## Summary

Keep open: the linked Telegram leak is source-reproducible and the PR is aimed at the right dispatch path, but the patch currently broadens a Telegram-specific hard-suppression need into a global change that can regress documented `/verbose` tool bubbles in other preview channels; it also still lacks real Telegram behavior proof.

## What This Changes

The PR changes shared reply dispatch so `suppressDefaultToolProgressMessages` is honored even when session verbose mode is on, and updates two dispatcher tests to expect suppressed tool/status messages.

## Best Possible Solution

Narrow the hard opt-out so Telegram final-only or explicit tool-progress-disabled cases stop leaking internal messages while preview-channel suppression still allows documented `/verbose` diagnostics, then attach current-head Telegram proof before merge.

## Reproduction Assessment

Yes. Source reproduction is high-confidence: current main combines Telegram direct context, `verboseLevel: "on"`, and `suppressDefaultToolProgressMessages: true`, then the existing tests expect default tool/status output despite suppression.

## Solution Assessment

No. The PR fixes the implicated predicate for the Telegram leak, but making the existing flag unconditional globally is broader than the reported Telegram final-only problem and conflicts with the documented `/verbose` behavior restored by merged history.
