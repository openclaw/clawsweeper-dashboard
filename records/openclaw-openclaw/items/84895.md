## Summary

Keep this PR open: current main and the latest release still drop cron payload `toolsAllow` on the CLI-backed isolated cron path, while the PR is a focused handoff into the existing CLI fail-closed contract. The remaining blocker is maintainer handling for the current branch conflict and the intentional compatibility change for existing CLI cron jobs with `toolsAllow`.

## What This Changes

The PR forwards isolated cron `agentPayload.toolsAllow` into the CLI `runCliAgent` call and adds a regression test asserting the CLI runner receives that allowlist.

## Best Possible Solution

Refresh the focused handoff against current main and land it only after maintainers explicitly accept the fail-closed behavior for restricted CLI-backed cron jobs.

## Reproduction Assessment

Yes. Source inspection on current main shows cron payload `toolsAllow` reaches prompt/delivery logic and the embedded runner but is omitted from the CLI runner call, while the CLI preparation layer already fails closed when it receives `toolsAllow`.

## Solution Assessment

Yes, with a compatibility caveat. Forwarding the existing payload field into the existing `RunCliAgentParams.toolsAllow` contract is the narrowest maintainable fix; the landing decision is whether maintainers accept the stricter fail-closed behavior.
