## Summary

Keep open. This is a draft, maintainer-labeled experimental feature PR with useful real-run proof for the context-engine warning, but source review found a blocker where worker-emitted agent events are not re-emitted in the parent process.

## What This Changes

The PR adds an experimental Node worker path for agent attempts with config/docs, permission flag plumbing, a build entry, and context-engine initialization coverage.

## Best Possible Solution

Keep the worker runtime behind experimental config, repair parent event forwarding with focused coverage, then let maintainers decide when the draft experiment is ready.

## Reproduction Assessment

Yes for the review finding from source inspection: worker thread event emissions stay in the worker, while the bridge only invokes a callback that does not re-emit parent `AgentEventPayload`s. I did not run a live subagent/control UI stream because this was a read-only sweep.

## Solution Assessment

No. The experimental direction is plausible, but the bridge should preserve the existing parent-process agent event contract, likely by posting full worker event data with `runId`/`sessionKey` and calling `emitAgentEvent` in the parent while still invoking the callback for bookkeeping.

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Re-emit worker events from the parent bridge:** `src/agents/worker-runtime/agent-runtime.ts:172-173`
