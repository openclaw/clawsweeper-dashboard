## What This Changes

Adds CLI and runtime validation, docs, tests, and changelog text to reject cron announce delivery to the internal webchat channel.

## Best Possible Solution

Keep the PR focused, but validate the effective announce delivery mode at CLI/RPC/runtime boundaries, reject webchat before cron fallback loses the requested channel, and align the docs examples and links with accepted commands.

## Reproduction Assessment

Yes. The PR body supplies live before/after terminal evidence, and current-main source gives a high-confidence path through implicit announce creation and raw cron delivery resolution.

## Solution Assessment

No. Rejecting webchat as an outbound announce target is the right direction, but this patch checks too narrowly and documents an invalid main-session command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Reject webchat when announce is implicit:** `src/cli/cron-cli/register.cron-add.ts:210-214`
  - body: This guard only runs when `hasAnnounce` is true, but isolated agentTurn jobs default to announce when neither `--announce` nor `--no-deliver` is passed. `cron add --session isolated --message hello --channel webchat` can still create the same forbidden announce-to-webchat delivery config.
  - confidence: 0.94
- **[P2] Reject webchat when edit preserves announce mode:** `src/cli/cron-cli/register.cron-edit.ts:156-160`
  - body: This edit guard only rejects when `--announce` is present. Editing an existing announce job with `cron edit <id> --channel webchat` preserves `delivery.mode="announce"` through `mergeCronDelivery`, so the forbidden config can still be written.
  - confidence: 0.88
- **[P2] Reject cron webchat before it is normalized away:** `src/infra/outbound/channel-selection.ts:222-227`
  - body: Putting the typed error only in `resolveMessageChannelSelection` does not catch raw cron jobs. `resolveDeliveryTarget` first calls `resolveSessionDeliveryTarget`, which drops non-deliverable `webchat`, then invokes `resolveMessageChannelSelection({ cfg })` without the original channel, so the cron runtime can still surface the generic no-channel fallback.
  - confidence: 0.91
- **[P2] Remove the invalid --no-deliver main-session example:** `docs/automation/cron-jobs.md:189`
  - body: The new default-agent example uses `--session main --system-event ... --no-deliver`, but cron add rejects any `--no-deliver` flag outside non-main agentTurn jobs. Users following this documented shape will get the existing validation error instead of creating the reminder.
  - confidence: 0.95
- **[P3] Fix the assertSupportedJobSpec source link:** `docs/automation/cron-jobs.md:175`
  - body: The new docs link points at `src/cron/jobs.ts`, but the current source lives under `src/cron/service/jobs.ts`, so readers following the reference hit a missing file. Use the current path and the docs site's internal-link style where possible.
  - confidence: 0.86
