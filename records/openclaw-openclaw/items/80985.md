## Summary

Keep this PR open. The bug-fix direction is valid and the supplied terminal proof is sufficient, but source review still finds blocker gaps that allow announce-to-webchat through and one documented command conflicts with current CLI validation.

## What This Changes

The PR adds cron CLI/runtime validation, tests, docs, and a changelog entry to reject cron announce delivery to the internal webchat channel with a typed error.

## Best Possible Solution

Reject webchat against the effective cron delivery mode at create, edit, gateway, and runtime resolution boundaries, preserve the original requested channel until the typed error can fire, and document only commands the CLI accepts.

## Reproduction Assessment

Yes. The PR body supplies live before/after terminal proof, and current-main source inspection shows a high-confidence path through implicit announce creation, edit-mode preservation, and raw cron delivery resolution.

## Solution Assessment

No. Rejecting webchat as an outbound announce target is the right fix direction, but this patch places the guards too narrowly and documents a main-session command that current CLI validation rejects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Reject webchat when announce is implicit:** `src/cli/cron-cli/register.cron-add.ts:210-214`
  - body: This guard only runs when `hasAnnounce` is true, but isolated agentTurn jobs default to `delivery.mode: "announce"` when neither `--announce` nor `--no-deliver` is passed. `cron add --session isolated --message hello --channel webchat` can still create the forbidden announce-to-webchat delivery config.
  - confidence: 0.94
- **[P2] Reject webchat when edit preserves announce mode:** `src/cli/cron-cli/register.cron-edit.ts:156-160`
  - body: This edit guard only catches `--announce --channel webchat`. Editing an existing announce job with `cron edit <id> --channel webchat` sends a channel-only delivery patch, and the cron service merge preserves the existing announce mode, so the same broken config can still be created.
  - confidence: 0.9
- **[P2] Reject webchat before cron normalizes it away:** `src/infra/outbound/channel-selection.ts:222-227`
  - body: Putting the typed error only in `resolveMessageChannelSelection` does not catch raw cron jobs. `resolveDeliveryTarget` first calls `resolveSessionDeliveryTarget`, which drops non-deliverable `webchat`, then calls `resolveMessageChannelSelection({ cfg })` without the original channel, so cron runtime can still surface the generic no-channel fallback.
  - confidence: 0.92
- **[P2] Remove the invalid main-session --no-deliver example:** `docs/automation/cron-jobs.md:191`
  - body: The new default-agent example uses `--session main --system-event ... --no-deliver`, but cron add rejects any `--no-deliver` flag outside non-main agentTurn jobs. Users following the documented shape will hit the existing `--announce/--no-deliver require a non-main agentTurn session target` error.
  - confidence: 0.95
