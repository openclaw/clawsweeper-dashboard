## Summary

Keep open: the fix direction is useful and the PR now supplies real behavior proof, but the patch misses an implicit announce create path, does not put the runtime webchat rejection on the cron raw-jobs path, and adds a docs example that the current CLI rejects.

## What This Changes

The PR adds CLI/runtime validation, tests, docs, and a changelog entry to reject cron announce delivery to the internal webchat channel with a clearer typed error.

## Best Possible Solution

Keep the PR focused, but move the guard to the effective cron delivery mode and cron delivery resolver path so every announce-to-webchat shape fails before or during validation with the typed actionable error, then align docs with commands the CLI accepts.

## Reproduction Assessment

Yes. The PR body includes live before/after terminal evidence, and current source inspection shows the bad shape is source-reproducible through implicit announce creation and raw cron delivery resolution.

## Solution Assessment

No. Rejecting webchat as an outbound announce target is the right direction, but this patch places two guards too narrowly and documents a command that current CLI validation rejects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Reject webchat when announce is implicit:** `src/cli/cron-cli/register.cron-add.ts:210-214`
  - body: This guard only runs when `hasAnnounce` is true, but isolated agentTurn jobs default to `deliveryMode: "announce"` later when neither `--announce` nor `--no-deliver` is passed. `cron add --session isolated --message hello --channel webchat` still builds the forbidden `delivery: { mode: "announce", channel: "webchat" }`, so the create-time guardrail misses the same bad shape.
  - confidence: 0.94
- **[P2] Reject cron webchat before it is normalized away:** `src/infra/outbound/channel-selection.ts:222-227`
  - body: Putting the typed error only in `resolveMessageChannelSelection` does not catch raw cron jobs. `resolveDeliveryTarget` first calls `resolveSessionDeliveryTarget`, which drops non-deliverable `webchat`, then invokes `resolveMessageChannelSelection({ cfg })` without the original channel, so `delivery.channel="webchat"` can still surface the generic no-channel fallback instead of `WebchatNotDeliverableError`.
  - confidence: 0.91
- **[P2] Remove the invalid --no-deliver main-session example:** `docs/automation/cron-jobs.md:189`
  - body: The new default-agent example uses `--session main --system-event ... --no-deliver`, but cron add currently rejects any `--no-deliver` flag outside non-main agentTurn jobs. Users following this documented supported shape will get the existing `--announce/--no-deliver require a non-main agentTurn session target` error; omit `--no-deliver` here or intentionally change the CLI contract.
  - confidence: 0.95
