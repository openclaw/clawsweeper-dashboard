## Summary

Keep open: the PR targets a real cron/webchat foot-gun, but the patch is not ready because the cron raw-job/runtime path still bypasses the new typed error, one added docs command is rejected by the existing CLI, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds CLI and runtime rejection, tests, docs, and a changelog entry for cron announce delivery to `webchat`.

## Best Possible Solution

Reject explicit `webchat` announce delivery in the cron validation/resolution path as well as the CLI, document only commands the CLI accepts, and then add redacted real CLI/runtime proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current `main` lets cron delivery carry `channel: "webchat"` into paths where `webchat` is not deliverable and then falls back or emits generic channel errors. I did not run a live command in this read-only review, but the invalid added docs command is also source-proven by the existing CLI guard.

## Solution Assessment

No. The direction is right, but the runtime guard needs to live in the cron validation/resolver path that sees raw jobs and updates, and the docs need to avoid `--no-deliver` on main `systemEvent` examples.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject webchat before cron fallback resolution:** `src/infra/outbound/channel-selection.ts:221-226`
  - body: Placing the typed error only in `resolveMessageChannelSelection` does not catch raw cron jobs: `resolveDeliveryTarget` sends `delivery.channel` through `resolveSessionDeliveryTarget`, which drops non-deliverable `webchat` and then calls `resolveMessageChannelSelection({ cfg })` without the original channel. A raw `jobs.json`/RPC cron with `delivery.channel: "webchat"` can still fall back to the last or single configured channel instead of failing fast, so add the explicit webchat rejection to the cron delivery validation/resolution path too.
  - confidence: 0.86
- **[P2] Drop `--no-deliver` from the main example:** `docs/automation/cron-jobs.md:190`
  - body: The added default-agent example uses `--session main --system-event ... --no-deliver`, but `cron add` rejects `--no-deliver` for non-isolated/non-agentTurn jobs with `--announce/--no-deliver require a non-main agentTurn session target.` Main `systemEvent` jobs already have no announce delivery, so this documented command would fail for users.
  - confidence: 0.94
