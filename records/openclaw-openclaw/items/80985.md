## Summary

Keep open: this is a plausible focused bug fix with sufficient terminal proof, but the current patch still misses implicit announce creation and the cron runtime resolver path, and one new docs example conflicts with current CLI validation.

## What This Changes

The PR adds cron CLI and outbound selector validation, tests, docs, and a changelog entry for rejecting announce delivery to the internal webchat channel.

## Best Possible Solution

Keep the fix focused, but validate the effective cron delivery mode and preserve the original requested channel in the cron resolver so every announce-to-webchat shape fails with the typed actionable error, then align docs with accepted CLI commands.

## Reproduction Assessment

Yes. The PR body supplies live before/after terminal evidence, and current-main source inspection gives a high-confidence reproduction path through implicit announce creation and raw cron delivery resolution.

## Solution Assessment

No. Rejecting webchat as an outbound announce target is the right direction, but this patch places two guards too narrowly and documents a command that the current CLI rejects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Reject webchat when announce is implicit:** `src/cli/cron-cli/register.cron-add.ts:210-214`
  - body: This guard only runs when `hasAnnounce` is true, but isolated agentTurn jobs default to `deliveryMode: "announce"` when neither `--announce` nor `--no-deliver` is passed. `cron add --session isolated --message hello --channel webchat` can still create the same forbidden announce-to-webchat delivery config.
  - confidence: 0.94
- **[P2] Reject cron webchat before it is normalized away:** `src/infra/outbound/channel-selection.ts:222-227`
  - body: Putting the typed error only in `resolveMessageChannelSelection` does not catch raw cron jobs. `resolveDeliveryTarget` first calls `resolveSessionDeliveryTarget`, which drops non-deliverable `webchat`, then invokes `resolveMessageChannelSelection({ cfg })` without the original channel, so the cron runtime can still surface the generic no-channel fallback.
  - confidence: 0.91
- **[P2] Remove the invalid --no-deliver main-session example:** `docs/automation/cron-jobs.md:189`
  - body: The new default-agent example uses `--session main --system-event ... --no-deliver`, but cron add rejects any `--no-deliver` flag outside non-main agentTurn jobs. Users following this documented shape will get the existing `--announce/--no-deliver require a non-main agentTurn session target` error.
  - confidence: 0.95
