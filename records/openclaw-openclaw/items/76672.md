## Summary

Keep open. The latest PR head is still protected by the `maintainer` label and still has P1 sudo carrier parsing gaps in the exec approval security boundary.

## What This Changes

The PR adds a shared command-analysis module and routes exec approval parsing, strict inline-eval detection, approval display summaries, fallback forwarding text, and related tests through it.

## Best Possible Solution

Keep the shared command-analysis boundary, but make sudo/doas carrier parsing match real command-running forms, add regression coverage for clustered and mode flags, then rerun the changed gate and security review.

## Reproduction Assessment

Yes. Source inspection of head `7fb6a25c` plus `sudo -h` gives a high-confidence path for `sudo -EH python3 -c 'print(1)'`, `sudo -i python3 -c 'print(1)'`, `sudo -s python3 -c 'print(1)'`, and `sudo -k python3 -c 'print(1)'` to return null before the carried interpreter is analyzed.

## Solution Assessment

No. The shared boundary is a maintainable direction, but this parser needs to cover real sudo command forms before it is safe for the approval path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Handle grouped sudo short options:** `src/infra/command-analysis/risks.ts:237-247`
  - body: `sudo` accepts grouped standalone flags such as `-EH`, but this parser treats the whole token as one option and returns `null` when it is not in the exact option sets. A valid command like `sudo -EH python3 -c 'print(1)'` is no longer unwrapped, so strict inline-eval detection and approval payload checks can miss the inner interpreter command.
  - confidence: 0.9
- **[P1] Unwrap command-running sudo modes:** `src/infra/command-analysis/risks.ts:59-89`
  - body: The sudo option tables omit command-running modes such as `-i`, `-s`, and `--shell`, and classify `-k` as non-executing even though `sudo -k <command>` still runs the command after resetting credentials. These forms return `null` before the carried command can be inspected.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff changes exec approval analysis and can miss inner commands behind valid sudo carrier forms.

Concerns:

- **[medium] Strict inline-eval carrier bypass:** `src/infra/command-analysis/risks.ts:367`
  - body: Valid sudo carrier forms can return `null` from `resolveSudoLikeCarriedArgv`, so `detectPolicyInlineEval` may not see an inner `python3 -c` or similar approval-only command.
  - confidence: 0.9
- **[medium] Approval guard carrier bypass:** `src/infra/command-analysis/risks.ts:315`
  - body: `buildCommandPayloadCandidates` depends on the same carrier resolver; when it fails to unwrap valid sudo forms, the existing `/approve` shell-command guard no longer sees the carried payload candidate.
  - confidence: 0.82
