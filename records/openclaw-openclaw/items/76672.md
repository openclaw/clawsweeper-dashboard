## Summary

Keep open: this maintainer-labeled PR changes the exec approval security boundary, and latest head still has a source-reproducible sudo carrier bypass that should block merge.

## What This Changes

This PR introduces a shared command-analysis boundary for exec approval parsing, inline-eval and command payload detection, approval display summaries, and gateway/node approval wiring.

## Best Possible Solution

Keep the shared command-analysis direction, but make sudo carrier parsing match real command-running grammar and add regression coverage for those forms before merge.

## Reproduction Assessment

Yes. Source inspection of PR head shows resolveCarrierCommandArgv returns null for valid sudo command-running forms, and sudo -h confirms those forms can carry a command.

## Solution Assessment

No. The shared boundary is maintainable, but the current parser is not safe enough for the exec approval security path until it recognizes real sudo execution modes and aliases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Treat command-running sudo modes as carriers:** `src/infra/command-carriers.ts:47-77`
  - body: `sudo -i`, `sudo -s`/`--shell`, and `sudo -k <command>` are valid command-running forms, but this table either omits them or classifies them as non-executing. When `resolveSudoLikeCarriedArgv` returns `null`, the new shared `/approve` and strict inline-eval checks only see the outer sudo argv, so carried payloads like `/approve ...` or `python3 -c ...` can bypass approval parsing.
  - confidence: 0.92
