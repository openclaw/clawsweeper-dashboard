## Summary

Keep open. The refactor is useful, but the new carrier parser can miss valid grouped `sudo` standalone options, which weakens strict inline-eval approval checks; the protected `maintainer` label also requires explicit maintainer handling.

## What This Changes

The branch moves exec approval command analysis into a shared `src/infra/command-analysis` boundary, wires summaries into approval UI and forwarding, and updates strict inline-eval carrier detection tests.

## Best Possible Solution

Keep the shared command-analysis direction, but fix the carrier parser to preserve strict inline-eval enforcement for valid `sudo`/carrier option forms and add focused regression coverage before merge.

## Reproduction Assessment

Yes. Source inspection plus `sudo -h` gives a high-confidence path: `sudo -EH python3 -c 'print(1)'` is valid, but the new resolver returns null on `-EH` before reaching the inner interpreter.

## Solution Assessment

No. The shared boundary is maintainable, but the proposed parser needs clustered carrier-option support and regression tests before it is the safest implementation for the approval boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Handle clustered sudo flags before unwrapping:** `src/infra/command-analysis/risks.ts:211-220`
  - body: `sudo` accepts grouped standalone flags such as `-EH`, but this parser only recognizes exact option tokens and returns null for the cluster. That means a valid command like `sudo -EH python3 -c 'print(1)'` is no longer unwrapped, so `detectPolicyInlineEval` can miss the inner inline eval in strict mode. Please split supported clustered standalone flags, or otherwise preserve detection for these valid carrier forms and add coverage.
  - confidence: 0.87
