## Summary

Keep open. This protected maintainer-labeled PR still has high-confidence P1 parser bugs in the exec approval security boundary: valid sudo forms can fail to unwrap the carried command, so strict inline-eval and `/approve` guard analysis can miss it.

## What This Changes

The PR adds a shared command-analysis boundary and routes exec approval parsing, strict inline-eval detection, and approval display/forwarding summaries through it.

## Best Possible Solution

Keep the shared command-analysis direction, but align sudo/doas carrier parsing with real command-running forms, add regression coverage, and rerun the changed gate plus security review.

## Reproduction Assessment

Yes. Source inspection of PR head `789851f6d9af5fc61c17e61d08cf624d4be53305` plus local `sudo -h` gives a high-confidence path for `sudo -EH python3 -c 'print(1)'`, `sudo -i python3 -c 'print(1)'`, `sudo -s python3 -c 'print(1)'`, and `sudo -k python3 -c 'print(1)'` to return null before the carried interpreter is analyzed.

## Solution Assessment

No. The shared boundary is maintainable, but the proposed carrier parser must cover real sudo command-running forms and clustered flags before it is safe for this approval path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Handle clustered sudo flags before unwrapping:** `src/infra/command-analysis/risks.ts:207-220`
  - body: `sudo` accepts grouped standalone flags such as `-EH`, but this parser compares the whole token against exact option names and returns `null` for the cluster. A valid command like `sudo -EH python3 -c 'print(1)'` is no longer unwrapped, so strict inline-eval detection and approval payload checks can miss the inner interpreter command.
  - confidence: 0.9
- **[P1] Treat sudo command-running modes as carriers:** `src/infra/command-analysis/risks.ts:59-89`
  - body: The sudo option tables omit valid command-running modes such as `-i`, `-s`, and `--shell`, and classify `-k` as non-executing even though `sudo -k <command>` still runs the command after resetting credentials. These forms return `null` before the carried command can be inspected.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch changes exec approval analysis and can miss inner commands behind valid sudo carriers.

Concerns:

- **[medium] Strict inline-eval carrier bypass:** `src/infra/command-analysis/risks.ts:333`
  - body: Valid sudo carrier forms can return `null` from `resolveSudoLikeCarriedArgv`, so `detectPolicyInlineEval` may not see an inner `python3 -c` or similar approval-only command.
  - confidence: 0.9

  - body: `buildCommandPayloadCandidates` depends on the same carrier resolver; when it fails to unwrap valid sudo forms, the existing `/approve` guard no longer sees the carried payload candidate.
  - confidence: 0.82
