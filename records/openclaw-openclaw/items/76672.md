## Summary

This PR should stay open: the exact automerge head fixes the earlier sudo -i/-s gap, but still treats sudo -k/--reset-timestamp as non-executing even though sudo can run commands with -k, leaving the shared approval parser unsafe.

## What This Changes

This PR adds a shared src/infra/command-analysis boundary, rewires exec approval risk and allowlist analysis through it, and updates gateway/node approval forwarding plus tests.

## Best Possible Solution

Keep the shared command-analysis direction, but make sudo reset-timestamp options unwrap carried commands when a command follows and add regression coverage for /approve plus strict inline-eval forms.

## Reproduction Assessment

Yes. Source inspection of head c57c2c114f5a shows the parser returns null for -k/--reset-timestamp, and sudo help shows -k is valid in the command-running usage.

## Solution Assessment

No. The shared boundary is maintainable, but the current parser is not safe enough for automerge until sudo reset-timestamp command forms are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Treat sudo -k as a command-carrying option:** `src/infra/command-carriers.ts:74-83`
  - body: `sudo -k <command>` and the long reset-timestamp form are valid command-running forms, but putting them in `SUDO_NON_EXEC_OPTIONS` makes `resolveSudoLikeCarriedArgv` return `null` before unwrapping the carried command. The `/approve` shell guard and strict inline-eval detection then only see the outer `sudo`, so payloads like `sudo -k /approve ...` or `sudo -k python3 -c ...` can bypass approval parsing.
  - confidence: 0.94

## Security Review

Status: needs_attention

  - body: Because -k/--reset-timestamp currently forces the sudo carrier resolver to return null, the shared command-analysis checks can miss carried /approve commands and inline-eval interpreter payloads behind those valid sudo execution forms.
