## Summary

Keep open. The protected maintainer-labeled PR still has high-confidence exec approval boundary regressions: the new sudo carrier resolver rejects valid command-running sudo forms and grouped flags, so strict inline-eval and approval-command guard checks can miss the carried command.

## What This Changes

This PR adds a shared `src/infra/command-analysis` boundary, routes inline-eval and payload extraction through it, and forwards command-analysis summaries through exec approval UI/protocol surfaces.

## Best Possible Solution

Keep the shared command-analysis direction, but make sudo/doas carrier parsing match valid command-running forms and option clusters, add regression coverage, then rerun the changed gate and security review.

## Reproduction Assessment

Yes. Source inspection of head `084c19fd17627655102afde36b95974519d9957d` plus local `sudo -h` gives a high-confidence path: valid `sudo -EH python3 -c 'print(1)'`, `sudo -i python3 -c 'print(1)'`, `sudo -s python3 -c 'print(1)'`, and `sudo -k python3 -c 'print(1)'` forms return null before the carried interpreter is checked.

## Solution Assessment

No. The shared boundary is maintainable, but the proposed carrier parser must cover command-running sudo forms and clustered flags before it is safe for this approval boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Handle clustered sudo flags before unwrapping:** `src/infra/command-analysis/risks.ts:207-220`
  - body: `sudo` accepts grouped standalone flags such as `-EH`, but this parser compares the whole token against exact option names and returns `null` for the cluster. A valid command like `sudo -EH python3 -c 'print(1)'` is no longer unwrapped, so strict inline-eval detection and approval payload checks can miss the inner interpreter command.
  - confidence: 0.9
- **[P1] Treat sudo command-running modes as carriers:** `src/infra/command-analysis/risks.ts:59-89`
  - body: The sudo option tables omit valid command-running modes such as `-i`, `-s`, and `--shell`, and classify `-k` as non-executing even though `sudo -k <command>` still runs the command after resetting credentials. These forms return `null` before the carried command can be inspected.
  - confidence: 0.88
