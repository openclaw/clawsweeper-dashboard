## Summary

Keep open. Current main still starts the optional model-pricing refresh from the early Gateway runtime-services path, pricing remains enabled by default, and the 60s OpenRouter/LiteLLM fetch timeout is still present; the existing `models.pricing.enabled: false` setting is a mitigation, while open PR #73486 remains the focused implementation candidate for deferring the refresh until after readiness.

## What This Changes

The issue reports macOS Gateway startup still delaying readiness by about 75 seconds when default OpenRouter/LiteLLM model-pricing fetches hit 60s timeouts, despite the earlier manifest-reuse pricing fix.

## Best Possible Solution

Land or adapt #73486 so the default pricing refresh starts only after Gateway sidecars and channels are ready, keep `models.pricing.enabled: false` as the restricted-network mitigation, and continue the broader process/network degradation investigation under #73323.

## Reproduction Assessment

Yes. The issue includes concrete macOS restart steps and timing logs, and current main preserves the same default early pricing-refresh ordering with 60s remote-fetch timeouts; a focused repro can force the pricing endpoints to hang and verify whether sidecar/channel readiness is delayed.

## Solution Assessment

Yes, as a direction. Deferring optional pricing refresh until after the ready boundary is the narrowest maintainable fix for this startup bug; the config opt-out helps affected users but does not make the default boot path robust.
