## Summary

Keep open. Current main still starts the optional model-pricing refresh from the early Gateway runtime-services path while pricing is enabled by default, before the post-attach sidecar/channel readiness boundary; the documented `models.pricing.enabled: false` toggle is a mitigation, not a fix for the default startup behavior. The narrow implementation path is already represented by open PR #73486, while #73323 remains the broader runtime/network degradation tracker.

## What This Changes

Request that Gateway pricing catalog refresh stop delaying default startup/readiness when OpenRouter or LiteLLM requests hang or hit their 60s timeout.

## Best Possible Solution

Land or adapt #73486 so the default pricing refresh starts only after Gateway sidecars/channels are ready, keep `models.pricing.enabled: false` as the documented restricted-network mitigation, and leave the broader long-running runtime degradation investigation in #73323.

## Reproduction Assessment

Yes. The issue includes concrete macOS restart steps and logs, and current main preserves the same default early pricing-refresh ordering with 60s remote-fetch timeouts; a focused maintainer repro can force the pricing endpoints to hang under default config and verify whether sidecar/channel readiness is delayed.

## Solution Assessment

Yes. Deferring optional pricing refresh until after the ready boundary is the narrowest maintainable fix for this startup bug; the existing config opt-out helps affected users but does not make the default boot path robust.
