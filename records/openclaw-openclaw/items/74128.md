## Summary

Keep open. Current main still starts the optional model-pricing refresh during early Gateway runtime setup before the sidecars/channels-ready boundary, so the default startup-readiness problem reported here is not resolved; #73486 remains the active narrow implementation candidate, while #73323 tracks the broader runtime/network degradation.

## What This Changes

Report that Gateway boot on macOS still waits roughly 75 seconds on OpenRouter/LiteLLM pricing fetch timeouts after the 7d9dc8cf24 partial fix.

## Best Possible Solution

Land or adapt #73486 so the default pricing refresh starts only after Gateway post-attach work and sidecars/channels reach the ready path, keep `models.pricing.enabled: false` as a documented mitigation, and leave the deeper runtime/network investigation in #73323.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to make the OpenRouter/LiteLLM pricing fetches hang under default config, restart the Gateway, and verify whether sidecar/channel readiness waits; current source and tests still show pricing begins before the ready boundary with 60s fetch timeouts.

## Solution Assessment

Yes. Deferring the optional pricing refresh until after sidecars/channels are ready is the narrowest maintainable fix for this issue; relying only on the opt-out leaves the default startup path vulnerable to remote catalog stalls.
