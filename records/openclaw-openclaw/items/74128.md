## Summary

Keep open. Current main still starts the optional model-pricing refresh from early Gateway runtime setup before the post-attach sidecars/channels-ready boundary; the documented `models.pricing.enabled: false` setting is a mitigation, but it does not fix the default startup-readiness path reported here. Open PR #73486 remains the narrow implementation candidate, while #73323 tracks the broader runtime/network degradation.

## What This Changes

The issue requests that Gateway startup stop letting optional OpenRouter/LiteLLM pricing fetch timeouts delay sidecars/channels readiness on macOS after the earlier 7d9dc8cf24 partial fix.

## Best Possible Solution

Land or adapt #73486, or an equivalent narrow change, so optional pricing refresh cannot begin until after sidecars/channels reach the ready path; keep `models.pricing.enabled: false` documented as a mitigation and leave the deeper runtime/network investigation in #73323.

## Reproduction Assessment

Yes. The issue provides repeated macOS boot logs and same-host network counterexamples, and current source/tests give a focused reproduction path: make the pricing fetches hang under default config and verify the refresh starts before the sidecars/channels-ready path.

## Solution Assessment

Yes. Deferring optional pricing refresh until after Gateway readiness is the narrowest maintainable fix for this issue; relying only on the opt-out leaves default installs exposed to remote catalog stalls.
