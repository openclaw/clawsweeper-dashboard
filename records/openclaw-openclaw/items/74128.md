## Summary

Keep open. Current main still starts the optional model-pricing refresh from the early Gateway runtime-services path before the post-attach sidecars/channels-ready boundary; the later `models.pricing.enabled: false` opt-out is a useful mitigation, but it does not fix the default startup-readiness problem reported here. Open PR #73486 remains the narrow implementation candidate, while #73323 covers the broader runtime/network degradation investigation.

## What This Changes

The issue asks for Gateway startup to stop waiting on optional OpenRouter/LiteLLM pricing fetch timeouts before sidecars/channels become ready on macOS after the earlier 7d9dc8cf24 partial fix.

## Best Possible Solution

Land or adapt #73486, or an equivalent change, so the optional pricing refresh cannot begin until after sidecars/channels reach the ready path; keep `models.pricing.enabled: false` documented as a mitigation and leave the deeper network degradation investigation in #73323.

## Reproduction Assessment

Yes. The issue provides concrete boot logs across repeated macOS restarts, and current source/tests give a focused reproduction strategy: make the pricing fetches hang under default config and verify that the refresh is started before the sidecars/channels-ready path.

## Solution Assessment

Yes. Deferring optional pricing refresh until after Gateway readiness is the narrowest maintainable fix for this issue; relying only on the opt-out leaves default installs exposed to remote catalog stalls.
