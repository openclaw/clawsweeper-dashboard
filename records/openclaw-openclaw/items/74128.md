## Summary

Keep open. Current main still launches the optional model-pricing refresh from early Gateway runtime setup with pricing enabled by default, so the reported startup ordering is not proven fixed; open #73486 remains the focused deferral candidate while #73323 tracks broader runtime network degradation.

## What This Changes

The issue reports that Gateway boot on macOS still waits roughly 75 seconds for OpenRouter and LiteLLM pricing fetch timeouts despite the earlier manifest-reuse fix.

## Best Possible Solution

Land or adapt #73486 so optional pricing refresh cannot run before sidecar/channel readiness, keep `models.pricing.enabled: false` as the mitigation, and leave broader process-wide network degradation under #73323.

## Reproduction Assessment

Yes. The issue provides concrete restart steps and timing logs, and current main still has a default-enabled early pricing-refresh path with 60-second remote fetch timeouts; a focused repro can force the pricing endpoints to hang during Gateway restart and observe readiness timing.

## Solution Assessment

Yes, as a direction. Deferring optional pricing refresh until after the ready boundary is the narrowest maintainable fix for this startup bug; the existing opt-out helps affected users but does not make the default boot path robust.
