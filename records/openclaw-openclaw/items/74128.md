## Summary

Keep open. Current main still starts the optional model-pricing refresh from the early Gateway runtime-services path, before sidecars/channels reach the ready boundary, so the reported default startup-readiness bug is not implemented away. The config opt-out is present, but it is only a mitigation; #73486 remains the active narrow implementation candidate and #73323 remains the broader runtime/network tracker.

## What This Changes

The issue reports that Gateway startup can still block for roughly 75 seconds on OpenRouter/LiteLLM pricing fetch timeouts after the earlier 7d9dc8cf24 partial fix.

## Best Possible Solution

Land or adapt #73486 so the default pricing refresh starts only after Gateway sidecars/channels are ready, keep `models.pricing.enabled: false` as the documented mitigation for restricted networks, and leave the broader runtime degradation investigation in #73323.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to make the OpenRouter/LiteLLM pricing fetches hang under default config, restart the Gateway, and observe whether sidecar/channel readiness waits; current source and tests still show pricing starts before the ready boundary with 60s fetch timeouts.

## Solution Assessment

Yes. Deferring the optional pricing refresh until after sidecars/channels are ready is the narrowest maintainable fix for this issue; relying only on the opt-out leaves the default startup path vulnerable to remote catalog stalls.
