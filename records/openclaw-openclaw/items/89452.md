## Summary

Keep open: the reported WebRTC stop/getUserMedia race is real, but this PR is not merge-ready because the new guard still converts an intentional stop during microphone setup into an app-level error and the contributor provided only mock/unit-test proof.

## What This Changes

The PR adds a post-getUserMedia null-peer guard in the WebRTC Realtime Talk transport and a unit test for stop() racing with microphone setup.

## Best Possible Solution

Handle stop during getUserMedia as an idempotent cancellation: stop the resolved stream, leave the dashboard idle, and reserve errors for real microphone/provider setup failures.

## Reproduction Assessment

Yes. Source inspection on current main shows stop() can null this.peer while start() is awaiting getUserMedia, and the next addTrack dereferences this.peer.

## Solution Assessment

No. The guard is a plausible mitigation, but the best fix is to treat an already-closed setup as cancellation and clean up the resolved media stream instead of throwing into the UI error path.
