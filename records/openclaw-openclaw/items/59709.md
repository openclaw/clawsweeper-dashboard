## Summary

Keep open. Current main still has a source-reproducible QQ Bot cron false-positive delivery path: QQ Bot failures can be returned as value results with no platform id, shared text delivery still counts that result, and cron still turns a non-empty result list into delivered status.

## What This Changes

The issue reports that cron jobs on v2026.3.31 mark QQ Bot delivery as delivered while no QQ notification is received.

## Best Possible Solution

Make no-identity QQ Bot text failures fail or suppress the durable delivery contract before cron can record the run as delivered, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: mock QQ Bot sendText to return an error result with no message id, then the current adapter/shared text path can still produce a non-empty delivery result that cron counts as delivered. I did not run a live QQ Bot credential test in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to enforce the existing delivery-identity/error contract on QQ Bot text results or the shared text result path, then add regression coverage for cron delivered status. This should not add a feature, config option, or product policy.
