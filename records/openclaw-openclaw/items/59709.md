## Summary

Keep open. Current main still has a QQ Bot cron false-positive delivery path: QQ Bot send failures can return data without a platform message identity, the text delivery path still counts that result, and cron marks the run delivered from a non-empty result list.

## What This Changes

Cron jobs configured for QQ Bot delivery should only record delivered after the QQ notification is actually accepted with a durable outbound identity.

## Best Possible Solution

Make no-identity QQ Bot send failures fail or suppress the durable delivery contract before cron can record them as delivered, with focused regression coverage for QQ Bot text delivery and isolated cron status.

## Reproduction Assessment

Yes, at source level: mock QQ Bot sendText to return an error result with no message id, then observe the current text delivery path produce a non-empty result that cron can mark delivered. I did not run a live QQ Bot credential test in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable repair is to enforce the existing delivery-identity/error contract on QQ Bot text results, or shared text results if nearby tests show the shape is not QQ-specific. This should not add a new feature, config option, or product policy.

On current main, QQ Bot sendText can return { channel: "qqbot", error } with no platform message id; the channel adapter maps that to an empty-id message send result, shared text delivery counts the result, and isolated cron records delivered=true from a non-empty result list. Implement the narrowest repair so no-identity/error QQ Bot text sends fail or suppress durable delivery before cron marks them delivered. Prefer fixing the QQ Bot adapter/engine contract first; add a shared text-result identity guard only if tests show other adapters can return the same no-identity success-shaped result. Add focused regression coverage for QQ Bot message adapter/sendText failure shape and isolated cron direct delivery status. Do not add a config option or change unrelated channels. Changelog is expected because user-visible cron delivery status changes.
