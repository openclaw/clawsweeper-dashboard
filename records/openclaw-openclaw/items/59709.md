## Summary

Keep this open. Current main still has a plausible false-positive delivery path for QQ Bot cron delivery: QQ Bot engine send failures can be returned as result metadata, shared text delivery counts the returned result, and cron marks direct delivery as delivered when the result list is non-empty.

## What This Changes

The issue reports that cron jobs configured for QQ Bot delivery record successful delivery while no QQ notification is received, even though manual message delivery works.

## Best Possible Solution

Fix the QQ Bot outbound adapter so engine error results fail delivery instead of producing successful outbound results, then add a cron regression proving run history only reports delivered after a real QQ outbound success. If live QQ accepts but suppresses proactive messages, surface that as a clearer diagnostic rather than a green delivered state.

## Reproduction Assessment

Unclear for the full end-to-end QQ notification because no live QQ credentials were used. There is a high-confidence code-level reproduction path: mock QQ Bot outbound to return an error with no message id and verify cron can currently treat the non-empty outbound result as delivered.

## Solution Assessment

No, current main is not the best solution because it still allows QQ send errors to pass through as non-throwing results. The narrowest maintainable fix should start in the QQ Bot adapter, with a shared outbound guard only if tests show other channel adapters can return error/no-identity results the same way.
