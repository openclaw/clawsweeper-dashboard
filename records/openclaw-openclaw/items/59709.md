## Summary

Keep this open. Current main still has a plausible QQ Bot cron false-positive delivery path: QQ Bot send failures can become non-throwing outbound results, text delivery counts those results, and cron marks direct delivery as delivered when the result list is non-empty.

## What This Changes

The issue reports that cron jobs configured for QQ Bot delivery can record successful delivery while no QQ push notification is received.

## Best Possible Solution

Make QQ Bot outbound send failures fail the delivery contract instead of returning success-shaped results, and add a cron regression that only records delivered after a real QQ outbound identity is returned.

## Reproduction Assessment

Unclear for the full end-to-end QQ notification symptom because this read-only review did not use live QQ credentials. There is a high-confidence code-level reproduction path: mock QQ Bot outbound to return an error with no message id and verify cron can currently count the non-empty result as delivered.

## Solution Assessment

No, current main is not the best solution because it lets QQ Bot send errors pass through as non-throwing outbound results. The narrowest maintainable fix should start in the QQ Bot adapter, with a shared text-result identity/error guard only if tests show other adapters can return the same shape.
