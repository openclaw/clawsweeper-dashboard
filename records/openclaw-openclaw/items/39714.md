## Summary

Keep open. Current main still lacks the Dockerized sandbox browser bridge split, and this draft is not mergeable as-is because key browser changes target the deleted old core browser path instead of the active plugin SDK/browser plugin runtime path. The maintainer discussion already identified the same current-path rewrite requirement, and local inspection confirms it remains true on current main.

## What This Changes

The PR branch adds advertised sandbox browser bridge/CDP host plumbing, embedded sandbox URL selection, attach-only remote tab handling, related config/help/schema/tests, and docker-compose/secrets-baseline updates.

## Best Possible Solution

Rewrite the fix on the active browser plugin path: add a deliberate advertised/container-reachable bridge URL contract through src/plugin-sdk/browser-bridge.ts and the browser plugin runtime API, keep the gateway-side bridge URL local, expose the advertised URL on SandboxBrowserContext, inject it from buildEmbeddedSandboxInfo(), resolve Dockerized sandbox CDP host/port separately, preserve attach-only remote tab behavior where appropriate, and add focused Docker/non-Docker regression coverage plus docs for any new config knob.

## Reproduction Assessment

Yes for a code-level reproduction against current main: the sandbox path still builds CDP and bridge URLs from 127.0.0.1/baseUrl and passes that same URL into embedded sandbox info. A live Docker/DooD reproduction was not run because this review is read-only.

## Solution Assessment

No. The intent is correct, but the proposed implementation targets obsolete src/browser files and mixes in security-sensitive compose/baseline churn; the safer solution is a current-path plugin SDK/runtime rewrite with narrow tests and explicit networking review.
