## Summary

Keep open. The bug class is real, but this draft is not merge-ready because it targets legacy browser paths after the active browser bridge moved behind the SDK facade and bundled browser plugin runtime, and it still lacks inspectable real Docker behavior proof.

## What This Changes

The branch splits sandbox browser bridge listen and advertised URLs, adds Docker host/CDP host handling and config fields, changes attach-only remote tab creation, and updates tests plus Docker Compose and secret-baseline files.

## Best Possible Solution

Port the listen/advertise bridge contract through the SDK browser-bridge facade and bundled browser runtime, keep host-local and sandbox-reachable URLs separate, preserve auth/SSRF boundaries, document any approved config knobs, and prove Docker/DooD plus non-Docker behavior.

## Reproduction Assessment

Yes, at source level: current main starts the sandbox browser bridge through the SDK facade without advertised-host parameters and passes only bridgeUrl into embedded sandbox info. I did not run a Docker/DooD E2E in this read-only review.

## Solution Assessment

No. The draft is not the best implementation path because it edits removed core browser files while the active runtime lives behind src/plugin-sdk/browser-bridge.ts and extensions/browser/src/browser; the safer path is a current-runtime rewrite with explicit network/security review.
