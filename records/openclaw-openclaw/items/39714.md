## Summary

Keep open: the Docker sandbox browser problem is real on current main, but this draft cannot merge because part of the patch targets the removed legacy `src/browser` tree instead of the active plugin SDK/runtime browser path.

## What This Changes

The PR adds separate local and advertised sandbox browser bridge/CDP host plumbing, Docker host-gateway reachability, embedded sandbox URL selection, attach-only tab-opening changes, config fields, and tests.

## Best Possible Solution

Rework the fix on the active browser plugin SDK/runtime path, keep host-local and container-advertised URLs separate, resolve sandbox CDP reachability deliberately, update docs for any config knob, and validate with focused tests plus Docker/DooD smoke.

## Reproduction Assessment

Yes, at source level: current main still builds sandbox CDP readiness and embedded `browserBridgeUrl` from loopback/local bridge paths. I did not run a live Docker/DooD smoke because this review is read-only.

## Solution Assessment

No. The intent is correct, but the current PR is not the best fix because it edits removed core browser files instead of the active plugin SDK/runtime contract and still needs a scoped security review for browser-control reachability.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Port the bridge contract to the active browser runtime:** `src/browser/bridge-server.ts:20`
  - body: The advertised bridge contract is added under `src/browser/bridge-server.ts`, but current main has no `src/browser` tree and starts the bridge through `src/plugin-sdk/browser-bridge.ts` backed by `extensions/browser/src/browser/bridge-server.ts`. Without moving this contract to the active facade and runtime, the Docker sandbox path still exposes only the local `baseUrl`.
  - confidence: 0.95
- **[P2] Move attach-only tab opening to active tab ops:** `src/browser/server-context.tab-ops.ts:130`
  - body: The attach-only tab creation change is made in old `src/browser/server-context.tab-ops.ts`, while current browser routes use `extensions/browser/src/browser/server-context.tab-ops.ts`, where persistent Playwright tab creation still runs before the raw CDP path. A current-path Docker CDP rewrite would still hit the timeout-prone path unless this is ported.
  - confidence: 0.86
