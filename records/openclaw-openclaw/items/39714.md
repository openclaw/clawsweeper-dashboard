## Summary

Keep open: the Docker sandbox browser problem is still real on current main, but this draft is not mergeable as-is because it changes stale core browser paths while the active bridge and tab operations now run through the plugin SDK/runtime, and it also changes browser-control networking in a way that needs maintainer review.

## What This Changes

The PR adds host-local versus advertised sandbox browser bridge/CDP plumbing, sandbox browser host config fields, Docker host-gateway reachability, embedded sandbox URL selection, attach-only tab creation changes, tests, and secrets-baseline updates.

## Best Possible Solution

Rework the fix on the active plugin path: expose separate host-local and advertised bridge URLs through the browser SDK/runtime, resolve sandbox CDP host/port separately, port attach-only tab handling to extensions/browser, drop incidental baseline churn, document any new config knob, and validate with focused tests plus Docker/DooD smoke.

## Reproduction Assessment

Yes, at source level: current main still builds sandbox CDP readiness and embedded browserBridgeUrl from loopback/local bridge paths. I did not run a live Docker/DooD smoke because this review was read-only.

## Solution Assessment

No. The PR's intent is correct, but the patch is not the best current solution because it targets removed core browser files and mixes security-sensitive Docker/browser-control networking into a stale draft instead of the active plugin SDK/runtime contract.
