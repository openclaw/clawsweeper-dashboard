## Summary

Keep open: the underlying Docker sandbox browser bug is still source-reproducible on current main, but this draft is not mergeable because it targets removed legacy browser files instead of the active plugin SDK/runtime path and lacks inspectable Docker behavior proof.

## What This Changes

The PR adds separate local and advertised sandbox browser bridge/CDP host handling, Docker host-gateway plumbing, attach-only tab-opening changes, config fields, and regression tests.

## Best Possible Solution

Rework the fix on the active plugin SDK and browser plugin runtime path, keep host-local and container-advertised URLs separate, resolve Dockerized CDP reachability deliberately, document any new config surface, and validate with focused tests plus Docker/DooD smoke.

## Reproduction Assessment

Yes, at source level: current main still builds sandbox CDP readiness and embedded `browserBridgeUrl` from loopback/local bridge paths. I did not run a live Docker/DooD smoke because this review is read-only.

## Solution Assessment

No. The intent is correct, but the current PR is not the best fix because it edits removed core browser files instead of the active plugin SDK/runtime contract and still needs browser-control networking review.

  - body: The advertised bridge contract is added under removed legacy `src/browser` code, but current main starts the bridge through `src/plugin-sdk/browser-bridge.ts` backed by `extensions/browser/src/browser/bridge-server.ts`. A current-path Docker CDP rewrite would still hit the timeout-prone path unless this is ported.
