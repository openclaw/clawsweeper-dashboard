## Summary

Keep open: the underlying Docker sandbox browser bridge problem is still visible on current main, but this draft is stale against the pluginized browser runtime and cannot merge as-is.

## What This Changes

The PR adds separate local and advertised sandbox browser bridge/CDP host plumbing, Docker host-gateway reachability, embedded sandbox URL selection, attach-only tab-opening changes, config fields, and tests.

## Best Possible Solution

Rework the fix on the active browser plugin SDK/runtime path, keep host-local and container-advertised URLs separate, resolve sandbox CDP reachability deliberately, update docs for any config knob, and validate with focused tests plus Docker/DooD smoke.

## Reproduction Assessment

Yes, at source level: current main still builds sandbox CDP readiness and embedded browserBridgeUrl from loopback/local bridge paths. I did not run a live Docker/DooD smoke because this review is read-only.

## Solution Assessment

No. The PR's intent is correct, but the patch is not the best current solution because it edits removed core browser files instead of the active plugin SDK/runtime contract and still needs security review for the Docker/browser-control trust boundary.

## Review Findings

Overall correctness: patch is incorrect

  - body: The advertised bridge contract is added under src/browser/bridge-server.ts, but current main deleted src/browser and starts the bridge through src/plugin-sdk/browser-bridge.ts backed by extensions/browser/src/browser/bridge-server.ts.   - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Needs attention because the PR expands Docker/container reachability to browser-control services and changes bridge binding behavior.

Concerns:

- **[medium] Non-loopback bridge binding changes the trust boundary:** `src/browser/bridge-server.ts:67`
  - body: The PR introduces allowNonLoopbackHost and permits the browser bridge to bind on 0.0.0.0 for sandbox browser use. The bridge is authenticated, but browser-control routes are sensitive and this should be reviewed on the active plugin runtime path with SSRF and sandbox-networking policy in mind.
  - confidence: 0.76

  - confidence: 0.72
