## Summary

Keep open: the PR targets a real Feishu message-loss bug that is still present on current main, but the current branch is not merge-ready because it catches all card failures, can resend partial/block content as text, and has only mocked/unit proof rather than redacted real Feishu/Lark delivery proof.

## What This Changes

The PR wraps Feishu reply-dispatcher structured-card delivery in a catch that retries failed card sends through the plain-text message path and adds two dispatcher tests.

## Best Possible Solution

Implement a Feishu 230099/11310-only final-reply fallback in the reply dispatcher that uses the existing text path only before any card chunk has been delivered, preserves current thread fallback semantics, and includes redacted real Feishu/Lark proof.

## Reproduction Assessment

Yes, source-level. The linked issue includes concrete Feishu 230099/11310 logs, and current main still routes markdown-table auto/card replies into sendStructuredCardFeishu without a table-limit catch or retry; I did not run a live Feishu send.

## Solution Assessment

No for the current PR head. Plain-text fallback is the right direction, but the patch should classify the Feishu table-limit error, limit fallback to final replies, avoid replaying already delivered card chunks, and rebase without dropping current reply/thread fallback behavior.
