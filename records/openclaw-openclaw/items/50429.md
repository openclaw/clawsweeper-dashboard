## Summary

Keep this PR open for revision. The Feishu table-limit reply bug is real and current main still lacks a reply-dispatcher fallback, but the proposed catch path is too broad: it retries every card error as text, can resend already delivered chunks, and can expose block preview payloads as ordinary Feishu messages.

## What This Changes

The PR wraps non-streaming Feishu reply card delivery in a try/catch, retries failed card sends through the plain-text Feishu message path, and adds two reply-dispatcher tests for fallback and successful card delivery.

## Best Possible Solution

Revise the Feishu reply-dispatcher fallback so it degrades only known 230099 / 11310 table-limit failures, only for final user-visible reply payloads, and without duplicating content after prior card chunks have already been sent. Add focused regression coverage for non-table card errors, block payloads, and multi-chunk partial delivery; reuse the classifier from #73651 if available.

## Reproduction Assessment

Yes. The linked bug has concrete Feishu 230099 / 11310 logs and current main's non-streaming card path still lets sendStructuredCardFeishu errors propagate; the PR's own mocked rejection test also reproduces the intended fallback path.

## Solution Assessment

No. The direction is useful, but this patch is not the safest implementation because it catches every card send failure and retries the full reply as text without constraining payload kind or partial delivery state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid retrying after delivered card chunks:** `extensions/feishu/src/reply-dispatcher.ts:419-445`
  - body: The catch wraps the whole card-mode sendChunkedTextReply call. If chunk 2 or later fails, earlier card chunks have already been sent, and the fallback resends the entire reply as text, including duplicate first-chunk mentions. Track delivered chunks and only fall back before the first send, or resume from the failed chunk.
  - confidence: 0.88
- **[P2] Limit fallback to final reply payloads:** `extensions/feishu/src/reply-dispatcher.ts:445-448`
  - body: When streaming startup has failed, info.kind === "block" can still reach this catch path. Sending those failed card chunks through sendMessageFeishu bypasses the existing block suppression and can surface partial or preview text before the final answer. Keep this fallback limited to final payloads or explicitly drop block payloads here.
  - confidence: 0.84
- **[P2] Classify table-limit errors before fallback:** `extensions/feishu/src/reply-dispatcher.ts:438-445`
  - body: The catch handles every sendStructuredCardFeishu failure. Auth, permission, malformed-card, and rate-limit failures should still propagate; only the known 230099 / 11310 table-limit rejection should degrade to plain text.
  - confidence: 0.8
