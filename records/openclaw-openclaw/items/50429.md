## Summary

Keep open: the PR targets a real Feishu message-loss bug, but the current head is not merge-ready because the fallback catches all card failures, can replay or expose messages, lacks real Feishu/Lark proof, and is dirty against current main.

## What This Changes

The PR wraps non-streaming Feishu structured-card reply delivery in a try/catch and retries failed card sends as plain text with two dispatcher tests.

## Best Possible Solution

Implement a Feishu 230099/11310-only final-reply fallback that retries through the existing text path only before any card chunk has been delivered, preserves thread fallback semantics, and proves delivery in a redacted live Feishu/Lark environment.

## Solution Assessment

No for the current PR head. Plain-text fallback is the right direction, but it should be table-limit-specific, final-only, avoid replaying already delivered card chunks, and preserve current thread fallback behavior.
