## Summary

Keep open: the Feishu table-limit reply bug is still valid on current main, and the PR has addressed the earlier double-conversion feedback, but the latest diff still falls back for all card-send failures and can resend unsafe or duplicate content.

## What This Changes

The PR wraps Feishu non-streaming reply card sends in a try/catch that logs failures and retries the reply as plain text, with two dispatcher tests for failure and success paths.

## Best Possible Solution

Constrain the reply-dispatcher fallback to final Feishu table-limit failures before any card chunk has been delivered, reusing a shared 230099 / 11310 classifier if the outbound PR lands first.

## Reproduction Assessment

Yes. Source inspection shows current main has an uncaught non-streaming reply card send path, #43690 provides concrete Feishu 230099 / 11310 logs, and the PR test simulates that sendStructuredCardFeishu rejection.

## Solution Assessment

No. Retrying as plain text is the right direction, but this patch is broader than the safe fix because it retries every card failure, can replay already-delivered chunks, and can expose non-final block payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid retrying after delivered card chunks:** `extensions/feishu/src/reply-dispatcher.ts:417-439`
  - body: If chunk 2 or later fails, sendChunkedTextReply has already delivered earlier card chunks, but the catch retries the whole reply as text. That can duplicate content and first-chunk mentions; only retry before any card chunk is sent, or resume from the failed chunk.
  - confidence: 0.88
- **[P2] Limit fallback to final replies:** `extensions/feishu/src/reply-dispatcher.ts:439-448`
  - body: When streaming startup has failed, block payloads can reach this catch and be sent as normal Feishu text. That bypasses the existing block suppression for internal preview content and can send a partial snippet before the final answer, so the fallback should ignore non-final payloads.
  - confidence: 0.84
- **[P2] Classify table-limit errors before falling back:** `extensions/feishu/src/reply-dispatcher.ts:439-445`
  - body: The catch handles every sendStructuredCardFeishu failure. Auth, permission, malformed-card, and rate-limit errors should continue to fail normally; only the known Feishu 230099 / 11310 table-limit rejection should degrade to plain text.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `extensions/feishu/src/reply-dispatcher.ts:416`
  - body: This is a user-facing Feishu delivery fix, but the PR only changes dispatcher code and tests. OpenClaw policy expects user-facing fixes to add a single-line CHANGELOG.md entry so the shipped behavior change is visible in release notes.
  - confidence: 0.78
