## Summary

Keep open: the linked Feishu table-limit bug is still real, but this PR head is not merge-ready because the fallback is too broad, can change message delivery semantics, and lacks real Feishu/Lark proof.

## What This Changes

The branch wraps Feishu reply-dispatcher structured-card delivery in a catch that retries failed card sends as plain text and adds two dispatcher tests.

## Best Possible Solution

Add a Feishu 230099/11310-only final-reply fallback that reuses the existing plain-text table-conversion path only before any card chunk has been delivered, with redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes, source-level. The linked issue includes concrete Feishu 230099/11310 logs, and current main still routes Markdown-table auto/card replies into sendStructuredCardFeishu without a table-limit fallback; I did not run a live Feishu send.

## Solution Assessment

No for the current PR head. Plain-text fallback is the right direction, but the patch should classify the Feishu table-limit error, limit fallback to final replies, and avoid replaying already-delivered card chunks.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Classify table-limit errors before falling back:** `extensions/feishu/src/reply-dispatcher.ts:438-445`
  - body: The catch handles every sendStructuredCardFeishu failure and retries as text. Auth, permission, malformed-card, and rate-limit errors should continue to fail normally; only the known Feishu 230099 / 11310 table-limit rejection should degrade to plain text.

  - body: When streaming startup has failed, `info.kind === "block"` payloads can reach this catch and be sent as normal Feishu text. That bypasses the current block suppression for internal preview content, so the fallback should ignore non-final payloads.
  - confidence: 0.84
- **[P2] Avoid replaying already delivered chunks:** `extensions/feishu/src/reply-dispatcher.ts:419-438`
  - body: If chunk 2 or later fails, sendChunkedTextReply has already delivered earlier card chunks, but this catch retries the whole reply as text. That can duplicate content and first-chunk mentions; retry only before any chunk is sent, or resume from the failed chunk.
  - confidence: 0.88
