## Summary

Keep open: the PR targets a real Feishu delivery bug that is still present on current main, but the current patch is not ready because it falls back on any card error, can expose block preview payloads, can duplicate already delivered chunks, and lacks real Feishu after-fix proof.

## What This Changes

The PR wraps Feishu reply-dispatcher card delivery in a catch that retries failed card sends as plain text and adds two fallback-focused dispatcher tests.

## Best Possible Solution

Add a Feishu table-limit-only final-reply fallback that retries through the existing plain-text/table-conversion path only before any card chunk has been delivered, reusing a shared 230099 / 11310 classifier if the outbound work lands first.

## Reproduction Assessment

Yes, at source level: current main routes Markdown-table Feishu auto/card replies to sendStructuredCardFeishu without a table-limit catch, and the linked issue includes concrete 230099 / 11310 logs. I did not run a live Feishu send.

## Solution Assessment

No. Plain-text fallback is the right direction, but this patch should classify the table-limit error, limit fallback to final replies, and avoid replaying the whole reply after any card chunk was delivered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Classify table-limit errors before falling back:** `extensions/feishu/src/reply-dispatcher.ts:437-445`
  - body: The catch handles every sendStructuredCardFeishu failure. Auth, permission, malformed-card, and rate-limit errors should continue to fail normally; only the known Feishu 230099 / 11310 table-limit rejection should degrade to plain text.
  - confidence: 0.86
- **[P2] Limit the fallback to final replies:** `extensions/feishu/src/reply-dispatcher.ts:437-448`
  - body: When streaming startup has failed, info.kind === "block" payloads can reach this catch and be sent as normal Feishu text. That bypasses the existing block suppression for internal preview content, so the fallback should ignore non-final payloads.
  - confidence: 0.84
- **[P2] Avoid replaying chunks after partial card delivery:** `extensions/feishu/src/reply-dispatcher.ts:419-437`
  - body: If chunk 2 or later fails, sendChunkedTextReply has already delivered earlier card chunks, but this catch retries the whole reply as text. That can duplicate content and first-chunk mentions; only retry before any chunk is sent, or resume from the failed chunk.
  - confidence: 0.88
