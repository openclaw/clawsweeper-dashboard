## Summary

Keep open: this PR targets a valid Feishu message-loss bug, but the current head is conflicting, lacks real Feishu/Lark proof, and its broad fallback can duplicate delivered content or expose non-final block payloads.

## What This Changes

The PR wraps Feishu reply-dispatcher structured-card sends in a catch that retries failed card delivery through the existing plain-text message path and adds two mocked dispatcher tests.

## Best Possible Solution

Add a Feishu 230099/11310-only final-reply fallback that reuses the existing plain-text table-conversion path only before any card chunk has been delivered, with redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes, source-level. The linked issue includes concrete Feishu 230099/11310 logs, and current main still routes markdown-table auto/card replies into sendStructuredCardFeishu without a table-limit catch or retry; I did not run a live Feishu send.

## Solution Assessment

No for the current PR head. Plain-text fallback is the right direction, but the patch should classify the Feishu table-limit error, limit fallback to final replies, avoid replaying already delivered card chunks, and rebase cleanly onto current main.
