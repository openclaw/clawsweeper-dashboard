## Summary

Keep open: current main still implements only the light, REM, and deep text-memory dreaming sweep, while the requested parametric consolidation phase is a broad new feature with unresolved memory/plugin boundary, provider, training-data, and rollback decisions. The reporter's follow-up usefully narrows the security requirements, but it reinforces that this needs maintainer design review rather than cleanup closure.

## What This Changes

Add an optional post-deep-sleep dreaming phase that distills memory/session experience into LoRA or provider fine-tuning checkpoints with opt-in config, provenance, regression guards, and rollback.

## Best Possible Solution

Define a maintainer-reviewed design that decides whether parametric consolidation belongs in memory-core or a new plugin/API seam, then specify opt-in data boundaries, provider behavior, evaluation, checkpoint activation, and rollback before implementation.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source and docs inspection confirm current main only supports light, REM, and deep text-memory dreaming phases.

## Solution Assessment

Unclear: the requested LoRA/fine-tuning phase is a plausible direction to evaluate, but it is not yet a narrow maintainable solution. The safer path is a design review around memory/plugin boundaries, training-data safety, provider integration, checkpoints, and rollback semantics.
