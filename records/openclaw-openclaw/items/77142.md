## Summary

Keep open: current main still implements the documented three-phase text-memory dreaming pipeline, while this issue requests a broad new parametric learning capability. The reporter's follow-up usefully narrows security requirements, but the work still needs maintainer product, memory/plugin API, provider, consent, evaluation, checkpoint, and rollback design before implementation.

## What This Changes

The issue requests an optional post-deep-sleep consolidation phase that distills dreaming output into LoRA or provider fine-tuning checkpoints.

## Best Possible Solution

Define a maintainer-reviewed design for plugin-owned or memory-core-owned parametric consolidation, including stable data boundaries, safety gates, provider behavior, evaluation, and rollback before adding phase/config support.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source and docs inspection show current main supports only light, REM, and deep text-memory dreaming phases.

## Solution Assessment

Unclear: LoRA/provider fine-tuning may be a valid direction, but the proposed phase is not yet the narrowest maintainable implementation path. The safer route is a design review around memory ownership, training-data safety, provider integration, checkpoints, and rollback semantics.
