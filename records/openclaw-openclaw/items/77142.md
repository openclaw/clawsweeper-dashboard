## Summary

Keep open: current main still implements the documented light/REM/deep text-memory dreaming pipeline only. The reporter's follow-up security requirements are useful, but this remains a broad new parametric training feature that needs maintainer product, security, provider, and plugin/API design before code work.

## What This Changes

Add an optional post-deep dreaming consolidation phase that distills memory/session output into LoRA or provider fine-tuned adapters with checkpoints, regression guards, rollback, and new config.

## Best Possible Solution

Define a maintainer-reviewed design for opt-in parametric consolidation, including memory/plugin ownership, training-data filters, dry-run review, provenance, provider behavior, evaluation gates, checkpoint retention, and rollback before adding phase/config support.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source and docs inspection show current main supports light, REM, and deep text-memory dreaming phases only.

## Solution Assessment

Unclear: the proposed LoRA/fine-tuning phase is not yet a narrow maintainable implementation path. The safer route is a design review around memory/plugin boundaries, opt-in data handling, provider integration, evaluation, and rollback semantics before code changes.
