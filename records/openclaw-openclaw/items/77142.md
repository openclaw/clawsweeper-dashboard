## Summary

Keep open: current main only implements the documented light/REM/deep text-memory dreaming pipeline, while the requested parametric consolidation phase is a broad new feature needing maintainer product, plugin/API, provider, and security design. The reporter’s follow-up adds useful safety requirements, but it does not make this a narrow repair candidate.

## What This Changes

Add an optional post-deep-sleep consolidation phase that distills dreaming output into LoRA/adapters or provider fine-tunes with checkpoints and regression guards.

## Best Possible Solution

Define a maintainer-reviewed, opt-in consolidation design around memory/plugin boundaries, training-data selection, redaction/consent, provider-local execution, checkpoint activation, regression evaluation, rollback, config, and docs before implementation.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source and docs inspection give high confidence that current main exposes only light, REM, and deep dreaming phases.

## Solution Assessment

Unclear: LoRA or provider fine-tuning may be a viable direction, but the proposed implementation is not yet the narrowest maintainable path. The safer route is a maintainer design review for plugin/memory boundaries and training-data safety before code work.
