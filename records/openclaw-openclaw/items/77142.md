## Summary

Keep open: current main only supports text-memory dreaming phases, while this asks for a broad new parametric learning capability that needs maintainer product, plugin/API, provider, and safety design before implementation.

## What This Changes

Add an optional fourth dreaming phase that distills memory and session experience into LoRA or fine-tuned adapter checkpoints after deep sleep.

## Best Possible Solution

Keep the issue open for a maintainer design that decides the memory/plugin/provider boundary, then defines opt-in consolidation config, training-data controls, provenance, evaluation, checkpointing, and rollback before code work.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source and docs inspection show current main has only light, REM, and deep text-memory dreaming phases.

## Solution Assessment

Unclear: adding the proposed phase directly to memory-core is not yet proven to be the safest maintainable path. The narrower next step is a maintainer-reviewed design for memory/plugin APIs, provider fine-tuning behavior, and training-data safety.
