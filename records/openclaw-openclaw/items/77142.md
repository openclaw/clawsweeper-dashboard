## Summary

Keep open: current main still implements the documented light/REM/deep text-memory dreaming pipeline only, while the requested parametric consolidation phase is a broad new feature requiring maintainer product, plugin/API, provider, and security design. The reporter's follow-up usefully sketches opt-in, filtering, dry-run, provenance, and rollback requirements, but those are not current OpenClaw contracts yet.

## What This Changes

Add an optional post-deep-sleep dreaming consolidation phase that distills memory/session output into LoRA or provider fine-tuning checkpoints with safety gates and rollback.

## Best Possible Solution

Define a maintainer-reviewed design for memory-owned or plugin-owned parametric consolidation, including stable data boundaries, opt-in controls, provider/local training interfaces, dry-run review, provenance, evaluation, and rollback before implementation.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source and docs inspection show current main supports only light, REM, and deep text-memory dreaming phases.

## Solution Assessment

Unclear: the proposed LoRA/fine-tuning phase is not yet a narrow maintainable implementation path. The safer route is maintainer design review around memory/plugin boundaries, training-data safety, provider integration, checkpoint activation, and rollback semantics.
