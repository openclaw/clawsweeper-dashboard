## Summary

Keep open: current main still implements the documented light/REM/deep text-memory dreaming pipeline only, while this request adds a broad parametric training phase that needs memory/plugin boundary, provider/runtime, config, evaluation, privacy, and rollback design. The reporter's follow-up security requirements are useful, but they reinforce that this needs maintainer design review rather than a narrow cleanup close.

## What This Changes

Add an optional post-deep dreaming consolidation phase that distills memory/session output into LoRA or provider fine-tuning checkpoints with safety guards and rollback.

## Best Possible Solution

Define a maintainer-reviewed design for memory-core or plugin-owned parametric consolidation, including public data inputs, opt-in config, provider/local training boundaries, dry-run review, provenance, checkpoint activation/rollback, and regression evaluation before implementation.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source, schema, host types, and docs on current main consistently show only light, deep, and REM dreaming phases.

## Solution Assessment

Unclear: the proposed LoRA/fine-tuning phase is not yet a narrow maintainable implementation path. The safer route is maintainer design review around memory plugin boundaries, training-data safety, provider integration, evaluation, and rollback semantics.
