## Summary

Keep open. Current main has adjacent session routing, dashboard session creation/deletion, compaction restore, transcript archival-on-delete, and memory/wiki tooling, but it still does not implement the RFC's central shared-LLM/isolated-session/public-KB architecture or first-class archived/pinned/tagged session lifecycle. The discussion is useful prior art, especially Artyomkun's comments and linked fork, but this remains a broad maintainer product/API decision rather than a cleanup-close candidate.

## What This Changes

The issue requests a multi-session architecture with shared inference, isolated per-channel/task sessions, shared public knowledge retrieval, and Control UI/API support for creating, archiving, restoring, pinning, tagging, and binding sessions.

## Best Possible Solution

Keep this as a maintainer-owned umbrella or split it into reviewable design slices: canonical session identity and main-session policy, additive archive/restore/pin/tag protocol and storage fields with migration tests, Control UI lifecycle workflows, and a scoped shared memory/wiki retrieval contract. Use the discussion and linked fork as input rather than trying to land the whole architecture in one broad rewrite.

## Reproduction Assessment

Not applicable for the full RFC because it is an architecture proposal, not a single bug. The remaining gaps are source-verifiable: current docs, routing tests, session types, protocol method lists, UI session controller, and memory docs still do not expose the requested default architecture or lifecycle contract.

## Solution Assessment

No; a single broad re-architecture branch is not the best way to solve this safely. The maintainable path is additive, versioned protocol/storage slices with migration and compatibility tests, followed by channel, UI, and memory follow-through once maintainers approve the contract.
