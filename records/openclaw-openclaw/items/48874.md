## Summary

Keep open. Current main has adjacent session routing, dashboard session creation, deletion-time transcript archiving, memory search, and plugin session extension slots, but it still does not implement the RFC's central multi-session architecture, archive/restore/pin/tag lifecycle, or public knowledge-base contract; Artyomkun's linked fork work is useful prior art, not merged behavior.

## What This Changes

The issue proposes a broad multi-session architecture with shared LLM inference, isolated per-channel/task sessions, shared RAG knowledge, canonical session routing, and WebUI session create/archive/restore/pin workflows.

## Best Possible Solution

Keep this as a maintainer-owned design umbrella or split it into explicit slices: canonical session identity and migration, session lifecycle state/API, Control UI lifecycle workflows, and shared memory/RAG policy.

## Reproduction Assessment

Not applicable for the full RFC because it is an architecture proposal, not a single bug. The remaining gaps are source-verifiable in current main across routing defaults, SessionEntry, protocol schemas, UI controllers, and memory docs.

## Solution Assessment

No, landing this as one broad re-architecture is not the safest solution. The maintainable path is additive, reviewed slices with migration and plugin-compatibility tests before changing defaults.
