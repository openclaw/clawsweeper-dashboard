- `merge-risk: 🚨 session-state`: The PR changes how persisted session-delivery recovery state classifies pending restart-continuation entries after failures or crashes.

## Summary

Keep this PR open, but not merge-ready: the recovery-state direction is useful and proof-backed, yet the marker is currently set before the channel turn clears pre-dispatch setup, so a no-send setup failure can be moved to failed instead of retried.

## What This Changes

The PR wires session-delivery recovery onto the shared SQLite recovery_state marker so recovered agent-turn deliveries that may have run are refused instead of blindly replayed.

## Best Possible Solution

Move the durable marker to a post-record, pre-dispatch boundary that only fires once non-idempotent agent work can run, add regression coverage for pre-dispatch failures staying retryable, then have maintainers accept the crash-window no-replay tradeoff.

## Reproduction Assessment

Yes. Source inspection shows current main delivers a pending session entry before acking it, and the PR body includes a SQLite-backed before/after probe where the same unacked agentTurn is delivered twice on base and once on the PR head.

## Solution Assessment

No as submitted. The recovery_state approach is the right owner boundary, but the hook fires before pre-dispatch setup succeeds; the marker should move to the first point where the agent turn can actually perform non-idempotent work.

- **[P2] Mark the send attempt after pre-dispatch setup succeeds:** `src/gateway/server-restart-sentinel.ts:321`
  - body: `onSendAttemptStart` runs before `dispatchAssembledChannelTurn`, but the turn kernel first awaits `recordInboundSession` before starting `runDispatch`. If that pre-dispatch import/session-record setup throws, `sendAttempted` remains true and recovery moves the entry to failed even though no LLM, tool, or message-send work ran. Move the marker to a post-record/pre-dispatch boundary or clear it on pre-dispatch failure, and add a regression for this no-send failure staying retryable.
  - confidence: 0.84
- **[P3] Set sendAttempted only after the marker persists:** `src/infra/session-delivery-queue-recovery.ts:176-177`
  - body: The hook flips `sendAttempted` before `markSessionDeliveryPlatformSendAttemptStarted` completes. If that marker write throws, the catch path treats the turn as possibly sent and moves the entry to failed even though delivery never started. Flip the in-process flag only after the marker is durably written, or keep separate attempted-versus-marker-write state.
  - confidence: 0.77
