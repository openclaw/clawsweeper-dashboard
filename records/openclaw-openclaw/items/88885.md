## Summary

Keep open: this is a focused reliability fix, but the current head still marks the session delivery as send-attempted before the turn kernel reaches its post-record/pre-dispatch boundary, so a pre-dispatch failure can suppress a retry that has not sent a reply.

## What This Changes

The PR adds a session-delivery recovery marker on the shared SQLite queue, refuses marked recovered entries, clears busy-deferred attempts, and adds recovery regressions.

## Best Possible Solution

Use the existing SQLite `recovery_state` column, but persist the marker through the turn kernel's `afterRecord` seam after inbound recording succeeds and before dispatch starts, then merge only after maintainers accept the fail-safe replay-refusal semantics.

## Reproduction Assessment

Yes. Source inspection and the supplied proof give a high-confidence path: enqueue an `agentTurn`, let delivery succeed without completing `ackSessionDelivery`, then run recovery again and observe the same pending entry replay on current main.

## Solution Assessment

No, not yet. The recovery-state approach is the right bounded direction, but the marker must move to the post-record/pre-dispatch seam so pre-dispatch failures remain retryable.
