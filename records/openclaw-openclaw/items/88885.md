## Summary

Keep open: this is a focused candidate for the linked crash-recovery bug, but the patch still persists the send marker before inbound session recording succeeds, so a pre-dispatch failure can move a queued turn to failed even though no reply was sent.

## What This Changes

The PR wires the session-delivery recovery drain to persist a `send_attempt_started` SQLite recovery marker, refuse marked recovered entries, clear busy-deferred attempts, and add recovery tests.

## Best Possible Solution

Use the existing SQLite `recovery_state` column, but mark only after successful inbound recording and immediately before dispatch, then merge only after maintainers accept the fail-safe replay-refusal semantics.

## Reproduction Assessment

Yes. Source inspection and the PR’s supplied proof give a high-confidence path: enqueue an `agentTurn`, let delivery succeed without completing `ackSessionDelivery`, then run recovery again and observe the same pending entry replay on current main.

## Solution Assessment

No, not yet. The recovery-state approach is the right bounded direction, but the marker must move after inbound recording succeeds so pre-dispatch failures remain retryable.
