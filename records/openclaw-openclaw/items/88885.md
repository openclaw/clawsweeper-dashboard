## Summary

Keep open: this is a useful recovery fix with strong proof for the reported duplicate-replay path, but the current patch marks the entry before channel-turn pre-dispatch recording succeeds, which can convert a retryable pre-dispatch failure into a failed session delivery.

## What This Changes

The PR wires session-delivery recovery to the existing SQLite `recovery_state` column, adds send-attempt hooks around restart-continuation delivery, refuses marked recovered entries, and adds focused recovery tests.

## Best Possible Solution

Move the marker boundary to after inbound session recording succeeds and before non-idempotent dispatch begins, preserve busy/pre-send retries, and then land the recovery-state fix if maintainers accept the fail-safe replay policy.

## Reproduction Assessment

Yes. Current main delivers a pending session entry and then acks it with no marker, and the PR body plus tests show the crash-before-ack replay path; the new marker-boundary defect is also source-reproducible from `dispatchAssembledChannelTurn` recording before `runDispatch`.

## Solution Assessment

No, not yet. Reusing `recovery_state` is the right bounded direction, but the marker must be placed after pre-dispatch recording succeeds or cleared on pre-dispatch failure before this is the safest fix.
