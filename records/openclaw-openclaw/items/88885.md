## Summary

Keep open: current main still has the session-delivery blind-replay gap, and this focused PR is the active linked fix path. It should remain in maintainer review because the patch intentionally changes ambiguous crash-window recovery from retry/replay to fail-safe refusal.

## What This Changes

The PR wires session-delivery recovery hooks into the restart-continuation delivery path and persists a send-attempt marker in SQLite recovery state to refuse blind replay after crash recovery.

## Best Possible Solution

Keep this PR open and have maintainers decide whether the fail-safe no-duplicate policy is the desired session-delivery recovery contract before landing a current-main-refreshed branch.

## Reproduction Assessment

Yes. Current main source shows drainQueuedEntry awaits deliver(entry) and only then ackSessionDelivery, with no session recoveryState, so a crash after deliver but before ack leaves a pending entry that recovery treats as fresh.

## Solution Assessment

Unclear. The implementation is a plausible bounded fix and matches the outbound no-blind-replay direction, but the best permanent answer depends on maintainer acceptance of fail-safe refusal versus adding adapter reconciliation.
