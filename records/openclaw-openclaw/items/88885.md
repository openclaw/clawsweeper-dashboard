## Summary

Keep open: the PR targets a valid crash-recovery replay bug and supplies useful real-behavior proof for the crash-before-ack path, but it is not merge-ready because the new catch path clears the marker for every thrown delivery even though the delivery seam can include message sends before it throws.

## What This Changes

The PR adds a session-delivery recovery marker on the shared SQLite queue state, refuses recovered marked entries instead of blind replaying them, and adds focused recovery tests.

## Best Possible Solution

Preserve the durable marker whenever send outcome is uncertain, clear it only for explicit pre-send/no-send retryable cases, and keep SQLite-backed regression coverage for both crash-before-ack and sent-before-error paths.

## Reproduction Assessment

Yes. Source inspection shows current main delivers then acks without a session recovery marker, and the PR's own live-output proof measures deliverCount dropping from 2 to 1 for the crash-before-ack case.

## Solution Assessment

No. The marker-and-refuse direction is the right layer, but this implementation is incomplete because it clears the marker for all thrown deliveries even though the deliver seam can send before it throws.
