## Summary

Keep open: the recovery-state direction is useful and proof-backed, but this patch still writes the send marker before the channel turn reaches the post-record/pre-dispatch boundary, so a retryable record/setup failure can be moved to failed as if a reply may have been sent.

## What This Changes

The PR adds session-delivery recovery hooks and storage helpers to persist a SQLite send-attempt marker during restart-continuation recovery, plus regression tests for crash, sent-before-error, pre-send failure, and busy-deferred paths.

## Best Possible Solution

Use the existing SQLite recovery_state column through the channel turn afterRecord boundary after inbound recording succeeds and before dispatch starts, add the missing retryable-failure regression, then have maintainers explicitly accept fail-safe refusal or pause for adapter reconciliation design.

## Reproduction Assessment

Yes. Source inspection and the supplied terminal proof give a high-confidence path: enqueue an agentTurn, let delivery succeed without completing ackSessionDelivery, then run recovery again and observe the pending entry replay on current main.

## Solution Assessment

No, not yet. The recovery-state approach is the right bounded direction, but the marker must move to the post-record/pre-dispatch seam so pre-dispatch failures remain retryable.
