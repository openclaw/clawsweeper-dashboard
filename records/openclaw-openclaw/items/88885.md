## Summary

Keep open: this PR targets a real current-main and latest-release replay bug, and the patch appears technically sound, but it intentionally chooses fail-safe failed-state handling over blind replay in an uncertain crash window, so maintainer acceptance is still needed before merge.

## What This Changes

The branch wires session-delivery recovery into the existing SQLite `recovery_state` column, adds send-attempt hooks at the restart-continuation turn boundary, and covers replay/refusal paths with focused recovery tests.

## Best Possible Solution

Land this as the bounded no-duplicate recovery fix only after a maintainer accepts the fail-safe tradeoff; track adapter-side reconciliation separately if full at-least-once recovery is required later.

## Reproduction Assessment

Yes. Source inspection and the linked proof show a high-confidence path: enqueue an `agentTurn`, let `deliver` run without completing `ackSessionDelivery`, then run recovery again and current main can deliver the same pending entry again.

## Solution Assessment

Yes, as a bounded repair. Reusing the existing SQLite `recovery_state` column and matching the outbound no-reconcile refusal pattern is the narrow maintainable fix; full adapter reconciliation would be a larger follow-up.
