## Summary

Keep open: the PR targets a real source-backed session-state bug, but the patch is not safe to merge because it suppresses the archive while still replacing the canonical session entry, leaving the active run's transcript disconnected and not actually deferred for later archive.

## What This Changes

The PR adds an auto-reply session guard that skips transcript archiving for implicit stale rollovers while a reply run is active, plus a regression test for daily rollover during an in-flight run.

## Best Possible Solution

Defer the whole implicit stale rollover while the old session has an active reply run, or persist an explicit pending rollover/archive state that keeps the old transcript reachable until post-run archive completes.

## Reproduction Assessment

Yes at source level: the linked issue and current code show daily staleness is evaluated and previous transcript archive is committed inside `initSessionState` before active-run admission. I did not run a live wall-clock/provider rollover in this read-only review.

## Solution Assessment

No: suppressing `previousSessionEntry` alone is not the best fix because the canonical session entry is still replaced with a new session, so the active run's transcript remains disconnected and the promised deferred archive has no old entry to act on.
