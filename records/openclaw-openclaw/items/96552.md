## Summary

Keep open: source review confirms the proposed guard is too narrow because it suppresses archiving while still committing a new session id, and the PR still has only test/grep proof rather than runtime transcript-file proof.

## What This Changes

The PR changes auto-reply session rollover logic to skip archiving a stale prior session transcript when that session has an active reply run, and adds a regression test for that case.

## Best Possible Solution

Defer the whole implicit stale rollover while the old session has an active reply run, or persist a real pending rollover/archive state that keeps the old transcript reachable until post-run archive completes.

## Reproduction Assessment

Yes at source level: current main evaluates daily staleness and archives the previous transcript inside `initSessionState` before `runPreparedReply` reaches active-run admission. I did not run a live wall-clock/provider rollover in this read-only review.

## Solution Assessment

No: suppressing `previousSessionEntry` alone avoids the rename but still commits a new session entry, so the active run's transcript remains disconnected and the promised deferred archive has no old entry to act on.
