## Summary

Keep open: the reported gateway reasoning gap is still credible on current main, but this PR is not mergeable as-is because it mixes the focused reasoning fix with unrelated session-memory work and introduces blocking regressions in command execution, reset hooks, and visibility/auth gates.

## What This Changes

This PR changes embedded Pi reasoning stream gating, replaces the `src/commands/agent.ts` barrel with a copied agent command implementation that forwards session reasoning, and adds scheduled-reset/session-memory metadata handling with tests.

## Best Possible Solution

Land a focused gateway reasoning fix in the owner modules: pass stored session `reasoningLevel` only where the command path still needs it, allow callback-less `thinking` agent-event broadcasts for WebSocket clients while preserving silent/visibility gates, and split scheduled-reset/session-memory cleanup into a separate reviewed change.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to set `reasoningLevel: "stream"` on a webchat/WebSocket session and send a turn: current main resolves/stores the mode, but the embedded subscribe path returns before broadcasting `stream: "thinking"` when no channel callback is present.

## Solution Assessment

No. The PR addresses a real gap, but the implementation is not the best path because it copies stale command code, combines unrelated session-memory behavior, and weakens existing authorization and reasoning-visibility contracts.
