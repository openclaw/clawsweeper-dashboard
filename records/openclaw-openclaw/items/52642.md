## Summary

This is a valid gateway reasoning-stream bug, but this PR should stay open for maintainer handling because the branch mixes unrelated session-memory work and introduces concrete correctness and security regressions. Current main still callback-gates `thinking` broadcasts when WebSocket/webchat does not provide `onReasoningStream`, while narrower open PRs #47613 and #61469 cover the broadcast side more directly.

## What This Changes

The PR forwards stored session reasoning into embedded Pi runs, loosens embedded reasoning stream callback gating, and adds scheduled-reset/session-memory metadata handling with tests.

## Best Possible Solution

Land a focused gateway reasoning fix in the owner modules: allow callback-less `thinking` agent-event broadcasts for WebSocket clients while preserving `canShowReasoning` and silent-reply gates, forward stored `reasoningLevel` only where the command ingress path still needs it, and split scheduled-reset/session-memory cleanup into a separate reviewed change.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to set `reasoningLevel: "stream"` via `sessions.patch` and send over WebSocket/webchat: current main stores and resolves the setting, but `emitReasoningStream()` still returns before broadcasting when no `onReasoningStream` callback exists.

## Solution Assessment

No. The proposed branch addresses the right user-visible gap, but it is not the best fix because it copies a stale command implementation, mixes unrelated session-memory work, and weakens existing authorization and reasoning-visibility safeguards.
