## Summary

Keep open. Current main still preserves some stale rollover context for typed `session_end`/`session_start` plugin hooks, but it does not run the bundled `session-memory` internal command hook or `before_reset` path for daily/idle lazy resets, and maintenance pruning still deletes stale entries without a pre-prune memory hook. The related #61675 and #71792 PRs are useful daily/idle candidates, while the prune half of this issue remains unaddressed.

## What This Changes

The issue requests that session-memory preservation run for daily/idle session reset and before maintenance pruning, not only for compaction or explicit reset-style paths.

## Best Possible Solution

Keep this issue open as the umbrella until maintainers either land or supersede the daily/idle PR work and add a documented preservation hook before stale entries are pruned, or split the prune behavior into a canonical narrower tracker linked from here.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to enable `session-memory`, create a stale daily or idle session and send the next non-system message, or run enforced maintenance against a stale store entry; current source shows those paths do not dispatch the internal session-memory hook before rollover/prune.

## Solution Assessment

No current close path is the best solution. The maintainable direction is to reuse the existing reset hook machinery for daily/idle rollover and add a documented pre-prune lifecycle or session-memory-compatible dispatch before store maintenance deletes stale entries.
