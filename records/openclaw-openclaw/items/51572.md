## Label Justifications

- `P2`: The issue describes a real memory/session lifecycle gap with meaningful but bounded impact and overlapping implementation discussions.
- `impact:data-loss`: The reported failure mode is session context being reset or pruned without a memory preservation opportunity.
- `impact:session-state`: The affected behavior is session rollover, pruning, transcript archival, and memory continuity across session boundaries.

## Summary

Keep open. Current main still keeps bundled session-memory scoped to command:new/command:reset, automatic daily/idle rollover emits session lifecycle hooks instead of that memory hook, and session-store maintenance prunes/archives stale entries without a pre-prune memory preservation contract.

## What This Changes

The issue requests firing the session-memory preservation path before automatic session resets and maintenance pruning, not only during command-triggered resets.

## Best Possible Solution

Choose one lifecycle memory contract that covers daily/idle rollover and prune-before-delete exactly once, with bounded maintenance behavior, privacy/retention rules, docs, and focused regression coverage.

## Reproduction Assessment

Yes for the current behavior gap. Source inspection shows automatic daily/idle rollover has the previous session entry but only emits session_end/session_start, while session-store pruning archives/deletes stale entries without dispatching session-memory.

## Solution Assessment

Unclear as a single direct fix. Reusing session-memory is plausible, but maintainers need to choose whether reset/prune preservation belongs in session-memory, before_reset, session_end, or a memory-core ingestion contract before wiring maintenance paths.
