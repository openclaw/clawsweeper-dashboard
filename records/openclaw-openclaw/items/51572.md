## Summary

Keep open: current main still has command-scoped session-memory capture, while automatic daily/idle rollover and session-store maintenance pruning do not invoke that memory-finalization path before context can be archived or removed.

## What This Changes

Request to fire bundled session-memory when a session is reset or pruned automatically, not only on compaction or explicit reset commands.

## Best Possible Solution

Define a maintainer-approved lifecycle contract that invokes session-memory before automatic daily/idle rollovers and before store maintenance deletion, with bounded async behavior, privacy/dedupe rules, docs, and regression tests.

## Reproduction Assessment

Yes. At source level, current main can rotate stale daily/idle sessions and prune stale store entries while bundled session-memory still handles only internal command:new and command:reset events; I did not run a live rollover because this review is read-only.

## Solution Assessment

Unclear. The requested memory-finalization behavior is valid, but the best implementation depends on maintainer choice of hook/API shape, timeout behavior, and privacy semantics.
