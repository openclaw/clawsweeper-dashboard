## Summary

Keep open: current main and v2026.5.22 still leave the bundled session-memory hook command-only, while automatic daily/idle rollover emits typed lifecycle hooks and maintenance pruning archives/deletes stale entries without a pre-prune memory flush contract. Two open PRs cover part of the daily/idle reset path, but prune-before-delete and the hook contract still need maintainer direction.

## What This Changes

Requested behavior is for session-memory to capture context before automatic session reset or maintenance pruning, not only on manual reset/new or compaction-related paths.

## Best Possible Solution

Define one lifecycle contract for memory capture: review or replace the linked daily/idle rollover PRs, then add an explicit pre-prune path with dedupe, transcript-retention, and security tests before closing this issue.

## Reproduction Assessment

Yes, by source inspection: automatic daily/idle rollover only emits typed session lifecycle hooks, and pruning deletes stale entries through store maintenance without a memory hook dispatch. I did not run a live gateway repro because this review was read-only, but the relevant current-main paths are clear.

## Solution Assessment

No, not as a direct command-hook reuse without maintainer design: firing command:new/reset for automatic maintenance would blur command and lifecycle semantics. A typed lifecycle/pre-reset or explicit session-memory listener with dedupe, retention, and upgrade tests is the safer direction.
