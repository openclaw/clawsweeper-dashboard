## Label Justifications

- `P2`: This is a real session memory lifecycle gap with meaningful user impact, but it needs normal-priority maintainer contract selection rather than emergency handling.
- `impact:data-loss`: The reported failure mode is session context being reset or pruned without any memory preservation opportunity.
- `impact:session-state`: The affected behavior is session rollover, transcript archival/deletion, maintenance pruning, and cross-session memory continuity.
- `impact:security`: The requested automatic background persistence can carry private transcript content into durable memory files and needs privacy/retention boundaries.

## Summary

Keep open: current main and the latest release still limit bundled session-memory to manual command resets, automatic daily/idle rollover only emits session lifecycle hooks, and session-store pruning archives/deletes stale entries without a pre-prune memory-preservation contract.

## What This Changes

Request to make the session-memory preservation path run for automatic session resets and maintenance pruning, not only manual reset/new command flows.

## Best Possible Solution

Choose one lifecycle memory contract that covers daily/idle rollover and prune-before-delete exactly once, with bounded maintenance behavior, privacy/retention rules, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Configure daily or idle reset, let an existing session go stale, then send the next message: initSessionState captures the previous session but only emits session_end/session_start, while maintenance pruning archives/deletes entries without invoking session-memory.

## Solution Assessment

Unclear as a direct hook-only fix. Reusing session-memory is plausible, but maintainers need to choose whether reset/prune preservation belongs in session-memory, before_reset/session_end, a new lifecycle event, or a memory-core ingestion contract.
