## Summary

Keep open. Current main still keeps bundled session-memory command-only, automatic daily/idle rollover only emits plugin lifecycle hooks, and session-store pruning archives/deletes removed sessions without a memory hook. Related reset PRs cover part of the request, but not the prune-before-delete behavior, so this is not safe to close as implemented or duplicate.

## What This Changes

Request that session-memory fire when sessions roll over by idle/daily reset and before session-store pruning deletes stale entries.

## Best Possible Solution

Choose one lifecycle memory contract, land one daily/idle reset implementation or replacement, and explicitly decide whether prune-time preservation belongs in session-memory, session_end/before_reset, or memory-core transcript ingestion with bounded maintenance behavior.

## Reproduction Assessment

Yes. Source inspection is enough: a stale daily/idle session rotates through session_end/session_start without session-memory, and session-store pruning deletes/archives stale entries without hook dispatch.

## Solution Assessment

Unclear. Expanding session-memory directly is plausible for reset, but prune-time work and existing Dreaming ingestion make this a lifecycle/memory product boundary rather than a narrow bug fix.
