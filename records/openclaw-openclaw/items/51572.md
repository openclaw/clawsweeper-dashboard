## Summary

Keep open: current main and the latest stable release still keep bundled session-memory limited to command:new/command:reset, while automatic daily/idle rollover and session-store maintenance pruning can archive or delete old session context without that memory-finalization path. Related issues and PRs cover parts of the same area, but none currently supersede the combined reset-plus-prune request.

## What This Changes

Issue #51572 requests session-memory finalization before automatic session reset and session-store prune paths, not only existing command-triggered memory capture.

## Best Possible Solution

Define one maintainer-approved lifecycle memory-finalization contract before automatic daily/idle rollover and destructive store prune, with bounded async behavior, privacy/dedupe rules, docs, and regression tests.

## Reproduction Assessment

Yes at source level: current main has automatic stale rollover and prune paths with previous session data, while bundled session-memory only handles command:new and command:reset; no live rollover was run because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract; the requested memory-finalization behavior is valid, but the permanent fix depends on event shape, deadline behavior, and privacy semantics.

## Risks / Open Questions

- Maintainers still need to choose the contract shape: synthetic command reset events, before_reset/session_end reuse, or a new typed reset/prune lifecycle event.

- Related open work covers overlapping reset, context-prune, and agentic-memory-flush ideas, so a fix should avoid creating competing lifecycle contracts.
