## Label Justifications

- `P2`: The report describes a real session-memory lifecycle gap with meaningful but bounded impact and active overlapping implementation work.
- `impact:data-loss`: The reported failure mode is session context being reset or pruned without a memory preservation opportunity.
- `impact:session-state`: The affected behavior is session rollover, transcript archival, pruning, and memory continuity across session boundaries.

## Summary

Keep open. Current main and v2026.5.19 still keep bundled session-memory scoped to manual command hooks while automatic daily/idle rollover only emits session lifecycle hooks, and session-store pruning archives/deletes stale entries without a pre-prune memory contract; the open reset PRs are useful but do not settle the prune path or privacy policy.

## What This Changes

Request that bundled session-memory run not only for manual `/new` or `/reset`, but also automatic idle/daily rollovers and before session-store pruning.

## Best Possible Solution

Choose one lifecycle memory contract that captures exactly once for automatic daily/idle rollover and before prune, with bounded maintenance behavior, privacy/retention rules, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Configure daily or idle reset with session-memory enabled, let an existing session go stale, then send the next message: current main captures `previousSessionEntry` but only emits `session_end`/`session_start`, while maintenance pruning archives/deletes stale entries without a memory hook.

## Solution Assessment

Unclear as a direct one-line fix. Reusing session-memory is plausible, but maintainers need to select the reset/prune contract and privacy bounds because the open reset PRs do not cover prune-before-delete.
