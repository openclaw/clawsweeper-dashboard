## Label Justifications

- `impact:data-loss`: The reported failure mode is prior session context being reset or pruned without a memory preservation opportunity.
- `impact:session-state`: The affected surface is session rollover, transcript archival, pruning, and memory continuity across session boundaries.

## Summary

Keep open: current main and v2026.5.22 still limit bundled session-memory to command reset/new events, while automatic daily/idle rollover only emits lifecycle hooks and maintenance pruning archives/deletes stale entries without a pre-prune memory contract.

## What This Changes

Request session-memory preservation on automatic daily/idle session reset and before session maintenance prune, not only on existing command-driven reset/new or compaction-related paths.

## Best Possible Solution

Choose one bounded lifecycle memory contract that captures exactly once for automatic daily/idle rollover and before prune, with privacy/retention rules, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Configure daily or idle reset with session-memory enabled, let an existing session go stale, then send the next message: current main captures previousSessionEntry but only emits lifecycle hooks, while maintenance pruning archives/deletes stale entries without a memory hook.

## Solution Assessment

Unclear as a direct one-line fix. Reusing session-memory is plausible, but maintainers need to select the reset/prune contract and privacy bounds because the current reset PRs do not cover prune-before-delete.
