## Summary

Keep open: current main and the latest release still keep bundled session-memory on command:new/command:reset, while automatic daily/idle rollover and session-store maintenance pruning do not run that memory-finalization path before old session context is archived or removed.

## What This Changes

The issue asks for session-memory to run before idle/daily session resets and maintenance pruning so context can be preserved before the session state is discarded.

## Best Possible Solution

Define a maintainer-approved lifecycle contract that invokes memory finalization before automatic daily/idle rollovers and before destructive store maintenance, with bounded async behavior, privacy/dedupe rules, docs, and regression tests.

## Reproduction Assessment

Yes. At source level, current main can replace stale daily/idle sessions and prune stale store entries while bundled session-memory handles only internal command:new and command:reset events; I did not run a live rollover because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the hook/API contract. The requested memory-finalization behavior is valid, but the best permanent fix depends on reset/prune event shape, timeout behavior, and privacy semantics.
