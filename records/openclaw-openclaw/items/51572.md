## Label Justifications

- `P2`: The issue affects durable memory/session continuity for long-running agents but is bounded to session lifecycle and hook behavior.
- `impact:data-loss`: The reported failure mode is conversation context being discarded or pruned without a memory extraction opportunity.

## Summary

Keep open: current main and the latest release still limit bundled session-memory to command:new/command:reset, while automatic daily/idle rollover and destructive session maintenance pruning do not run that memory-finalization path; related reports and PRs cover slices of the problem but no merged or viable canonical item fully resolves this reset-plus-prune request.

## What This Changes

The issue requests firing session-memory or an equivalent memory-finalization hook before automatic session reset and maintenance prune paths, not only on the existing command/reset or compaction-adjacent paths.

## Best Possible Solution

Define a maintainer-approved lifecycle contract that finalizes memory before automatic daily/idle rollover and destructive maintenance prune, with bounded async execution, privacy/dedupe rules, docs, and regression coverage across reset and prune paths.

## Reproduction Assessment

Yes, at source level. Current main can replace stale daily/idle sessions and prune stale store entries while bundled session-memory reacts only to command:new and command:reset; I did not run a live rollover because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the hook/API contract. The requested memory-finalization behavior is valid, but the best permanent fix depends on reset/prune event shape, timeout behavior, and privacy semantics.
