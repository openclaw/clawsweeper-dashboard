## Summary

Keep open. Current main still has the old outbound delivery contract, so the PR’s intended fix is neither implemented nor superseded; the related delivery-status work remains dependent on this contract change.

## What This Changes

The PR changes outbound delivery to return a DeliveryOutcome with hook-cancellation metadata, adds DeliveryError partial-send metadata, and updates result-consuming gateway, cron, reply, message callers, and tests.

## Best Possible Solution

Keep this PR open and either land it after maintainer review and rebase, or implement the same core contract on main: expose delivery outcome metadata from outbound delivery, preserve partial-send metadata for non-bestEffort failures, and update gateway/cron/status callers so hook cancellations are no longer treated as delivery failures.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the ambiguity: hook cancellation is a continue path that returns only an empty result array, and partial non-bestEffort sends rethrow the original error without sentBeforeError metadata.

## Solution Assessment

Yes in direction. The PR fixes the ambiguity at the owning outbound delivery boundary and updates the result-consuming callers; the safer landing path is to rebase onto current main and run the targeted delivery/gateway/cron checks before merge.
