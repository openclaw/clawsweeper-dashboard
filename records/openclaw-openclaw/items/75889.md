## Summary

Keep open: current main still emits the reported warning from the deferred startup retry path, even though direct `gateway_start` reconciliation is already quieted. The existing regression test explicitly reproduces cron becoming available after an initial retry warning, so the startup-noise problem remains actionable.

## What This Changes

The issue asks memory-core to avoid a misleading managed dreaming cron startup warning when cron becomes available shortly after gateway startup.

## Best Possible Solution

Keep startup-window retry attempts debug-level or silent, then emit the warning only after the bounded retry window is exhausted or on a clearly non-startup runtime reconciliation failure.

## Reproduction Assessment

Yes. The current `extensions/memory-core/src/dreaming.test.ts` retry case reproduces a missing cron at `gateway_start`, advances the startup retry, observes the warning, then confirms cron can appear later and create the managed job.

## Solution Assessment

Yes, the narrow maintainable fix is in memory-core's retry reason/log policy using the existing `getCron` seam; adding a new core plugin-ready event is not necessary for this report.
