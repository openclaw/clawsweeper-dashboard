## Summary

Keep open: current main demotes the immediate gateway_start warning, but the deferred startup retry still calls the runtime reconciliation path and can emit the same cron-unavailable warning before cron becomes available.

## What This Changes

The issue requests eliminating a harmless memory-core startup warning when dreaming cron reconciliation runs before the cron service is registered.

## Best Possible Solution

Keep startup retries debug-level or silent while the bounded startup retry loop is still waiting for cron, and warn only after retry exhaustion or a clearly non-startup runtime failure.

## Reproduction Assessment

Yes. The current memory-core unit test reproduces a startup-missing-cron path, advances the 5-second retry, observes the warning, then verifies the managed cron job is created once cron appears.

## Solution Assessment

Yes, as narrowed. Adjusting the existing memory-core startup retry log policy is a smaller and safer solution than adding a new core plugin-ready event for this one timing issue.
