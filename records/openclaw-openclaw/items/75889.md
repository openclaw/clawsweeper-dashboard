## Summary

Keep open: current main still emits the exact warning during the deferred startup retry path when cron is unavailable for the first retry, so the startup-noise bug is not fully fixed.

## What This Changes

The issue asks memory-core to avoid misleading startup warnings when managed dreaming cron reconciliation races cron service availability.

## Best Possible Solution

Keep startup retries debug-level or silent while inside the bounded startup retry window, and warn only after cron remains unavailable past that window or during a clear non-startup runtime failure.

## Reproduction Assessment

Yes. The current `extensions/memory-core/src/dreaming.test.ts` retry case reproduces cron-unavailable startup, advances the retry timer, observes the warning, then confirms cron can appear later and the managed job is created.

## Solution Assessment

Yes, with a narrower implementation than adding a new core ready event. The best fix is to adjust memory-core's existing retry warning policy around startup retries while preserving a genuine runtime failure signal.
