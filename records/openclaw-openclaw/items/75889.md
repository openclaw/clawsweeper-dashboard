## Summary

Keep open: current main still reaches the warn-level cron-unavailable path during the startup retry flow, and the focused closing PR #76099 is open and unmerged.

## What This Changes

memory-core should avoid a warn-level cron-unavailable startup message while dreaming is enabled and the cron service attaches shortly after boot.

## Best Possible Solution

Land PR #76099 or an equivalent narrow memory-core change that keeps startup retry-window cron absence at debug level while preserving warn-level logging after retry exhaustion or genuine runtime failure.

## Reproduction Assessment

Yes. Current source shows the startup retry calls runtime reconciliation while cron is still unavailable, and the existing retry test asserts the warn-level `cron service unavailable` message after one retry delay.

## Solution Assessment
