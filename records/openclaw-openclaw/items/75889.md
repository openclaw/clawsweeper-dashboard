## Summary

Keep open: current main still has a startup retry path that can emit the reported warn-level memory-core cron warning, and open PR #76099 is the linked unmerged fix candidate.

## What This Changes

The issue requests suppressing or deferring a harmless memory-core dreaming startup warning when the cron service becomes available shortly after gateway startup.

## Best Possible Solution

Review and land PR #76099, or an equivalent narrow memory-core change that demotes retry-window cron unavailability while preserving warnings after retry exhaustion or genuine runtime failure.

## Reproduction Assessment

Yes. Current `extensions/memory-core/src/dreaming.test.ts` reproduces cron being unavailable at gateway start, advances the startup retry timer, and asserts the reported warning before cron later becomes available.

## Solution Assessment

Yes. Handling this in memory-core's bounded retry/logging policy is narrower than adding a new core plugin-ready event because the existing `gateway_start` `ctx.getCron?.()` seam is already documented.
