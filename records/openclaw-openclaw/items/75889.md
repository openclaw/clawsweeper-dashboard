## Summary

Keep open: current main still emits the warn-level cron-unavailable message during the bounded startup retry path, and the linked fix PR is open and unmerged.

## What This Changes

The report asks `memory-core` to avoid a warn-level cron-unavailable startup message while the managed dreaming cron service is still attaching.

## Best Possible Solution

Land a narrow `memory-core` fix that treats bounded startup retry attempts as startup-window noise, keeps debug visibility, and preserves warn-level logging after retry exhaustion or genuine runtime cron failure.

## Reproduction Assessment

Yes. Source and existing tests show that with dreaming enabled and `getCron()` initially unavailable, the startup retry calls runtime reconciliation after 5 seconds and currently expects the cron-unavailable warning before cron becomes available.

## Solution Assessment

Yes, directionally. The linked PR's approach is narrow because it suppresses only the bounded startup retry noise while preserving the genuine runtime warning path, but the issue should stay open until that PR is reviewed and merged.
