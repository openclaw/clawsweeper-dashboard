## Summary

Keep open: current main still has a deferred startup retry path that logs the exact cron-unavailable warning before cron later becomes available, so the reported startup-noise bug is not fully resolved.

## What This Changes

The issue requests quieter or deferred memory-core dreaming cron reconciliation when gateway startup reaches memory-core before the cron service is available.

## Best Possible Solution

Adjust memory-core's existing startup retry/logging policy so bounded startup retries stay debug-level or silent, while preserving a warning after retry exhaustion or a genuine non-startup cron failure.

## Reproduction Assessment

Yes. `extensions/memory-core/src/dreaming.test.ts` simulates startup with cron unavailable, advances the startup retry timer, observes the warning, then makes cron available and verifies the managed cron job is created.

## Solution Assessment

Yes, with the narrower alternative already available in memory-core. The maintainable fix is to refine the existing retry classification/log policy rather than add a new core plugin-ready API for this single startup race.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75889. When `memory-core` dreaming is enabled and cron is temporarily unavailable during gateway startup, the deferred startup retry currently calls runtime reconciliation and emits `memory-core: managed dreaming cron could not be reconciled (cron service unavailable).` before cron later appears. Keep the fix scoped to memory-core retry/logging policy unless code inspection proves a generic seam is required. Preserve managed cron creation, runtime config refresh, and the genuine warning for post-startup/runtime cron failure. Update `extensions/memory-core/src/dreaming.ts` so bounded startup retries are debug-level or silent while cron is still attaching, and warn only after retry exhaustion or a true non-startup runtime failure. Update `extensions/memory-core/src/dreaming.test.ts` to cover immediate `gateway_start` no-warn, first deferred retry no-warn while cron is still unavailable, later retry creating `Memory Dreaming Promotion` when cron appears, and a genuine failure/exhaustion warning. Add a user-facing `CHANGELOG.md` fix entry if log behavior changes. Do not change cron service contracts, external cron job storage, or introduce a new core plugin-ready API unless strictly necessary.
