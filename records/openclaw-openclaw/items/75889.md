## Summary

Keep open: current `main` demotes the immediate `gateway_start` warning, but the deferred startup retry path still logs the same warning before cron can become available, so the reported startup-noise problem is not fully implemented away.

## What This Changes

The issue asks `memory-core` to avoid warning-level startup noise when managed dreaming cron reconciliation runs before the cron service is ready.

## Best Possible Solution

Keep startup retries at debug or silent while cron is still inside the bounded startup retry window, and emit a warning only when cron remains unavailable after that window or on a clearly non-startup runtime failure.

## Reproduction Assessment

Yes. The current `extensions/memory-core/src/dreaming.test.ts` retry case reproduces a cron-unavailable startup, advances the 5-second retry, observes the warning, then confirms cron can appear later and the managed job is created.

## Solution Assessment

The requested direction is mostly right, but the narrowest maintainable fix is to adjust the existing startup retry warning policy rather than add a new plugin-ready event or broader core API.

## Repair Work Prompt

Fix the `memory-core` managed dreaming cron startup warning noise reported in https://github.com/openclaw/openclaw/issues/75889. Keep the existing `gateway_start`/`getCron` retry architecture; do not add a new core plugin-ready API unless unavoidable. Adjust `extensions/memory-core/src/dreaming.ts` so transient missing cron during the bounded startup retry window does not emit the warning-level `managed dreaming cron could not be reconciled (cron service unavailable)` message when cron may still become available. Preserve a warning/error signal when cron remains unavailable after retries are exhausted or when a clearly non-startup runtime reconciliation fails. Update `extensions/memory-core/src/dreaming.test.ts` to cover: no warning on immediate startup, no warning on deferred retry that later succeeds, and a warning when cron remains unavailable beyond the startup retry policy or on genuine runtime failure. Add a single-line `CHANGELOG.md` fix entry; do not touch unrelated memory, cron, gateway startup, or plugin SDK behavior.
