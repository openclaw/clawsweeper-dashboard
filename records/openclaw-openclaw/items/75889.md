## What This Changes

The issue asks memory-core to avoid warn-level startup noise when managed dreaming cron reconciliation runs before the cron service is available.

## Best Possible Solution

Review and land or supersede the focused fix in https://github.com/openclaw/openclaw/pull/76099 so startup retry-window cron absence stays below warn level while genuine post-window failures still warn.

## Reproduction Assessment

Yes. Current source and the existing retry test show that with dreaming enabled and cron initially unavailable, the startup retry calls runtime reconciliation after the retry delay and still expects the cron-unavailable warning.

## Solution Assessment

Yes. The narrow maintainable fix is to suppress only bounded startup retry-window noise and preserve warnings after retry exhaustion or genuine managed runtime cron failure.
