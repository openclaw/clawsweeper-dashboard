## Summary

Keep open: current main still has a startup retry path that logs the exact `cron service unavailable` warning before cron later becomes available, so the reported startup-noise bug is not fully fixed.

## What This Changes

The issue asks `memory-core` to avoid misleading startup warnings when dreaming is enabled and cron is temporarily unavailable during gateway startup.

## Best Possible Solution

Keep startup retries debug-level or silent while inside the bounded startup retry window, and warn only if cron remains unavailable after that window or during a clear non-startup runtime failure.

## Reproduction Assessment

Yes. The current `extensions/memory-core/src/dreaming.test.ts` retry case reproduces cron-unavailable startup, advances the retry timer, observes the warning, then confirms cron can appear later and the managed job is created.

## Solution Assessment

Yes. The narrowest maintainable fix is to adjust `memory-core`'s existing retry warning policy while preserving a genuine runtime failure signal; a new core plugin-ready event is not required by the current code path.
