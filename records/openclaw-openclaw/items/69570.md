## Summary

Keep this open. Current main still treats each approved async exec completion as a separate approval-id-scoped agent follow-up, while the remaining fix needs an explicit scheduling policy for coalescing, cancellation, user-input priority, and audit preservation.

## What This Changes

The issue requests preventing repeated async exec approval completions from monopolizing a session by adding cancellation, batching, stale-follow-up suppression, and user-input priority behavior.

## Best Possible Solution

Keep this as the canonical approved-completion cascade report and design a core exec-approval scheduling rule that can coalesce or cancel stale follow-ups while preserving result/audit traceability and letting stop or newer user input preempt safely.

## Reproduction Assessment

Yes, at source level. Multiple same-session gateway or node exec approvals still create one approval-id-scoped follow-up agent turn each, and the issue discussion includes a recent deterministic production reproduction.

## Solution Assessment

Unclear. The current per-approval follow-up design is not enough, but the best fix requires a maintainer policy decision on pausing, batching, dropping, direct-delivering, or auditing stale completion messages.
