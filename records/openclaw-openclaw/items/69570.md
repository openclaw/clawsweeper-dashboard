## Summary

Keep open. Current main still creates one approval-id-scoped detached exec follow-up turn per approval result, and the live discussion plus current docs/tests show this is still the active contract; solving the cascade needs an explicit exec approval scheduler and cancellation policy.

## What This Changes

The issue requests cancellable or coalesced async exec approval completions so stale completion messages do not block newer user commands.

## Best Possible Solution

Keep this as the canonical approved-completion cascade report and define a core scheduling rule that can coalesce or cancel stale exec approval follow-ups, prioritize stop or newer user input, and preserve per-approval auditability.

## Reproduction Assessment

Yes, at source level. Configure exec approvals, approve multiple same-session foreground commands, and current main still returns `approval-pending` while creating one detached approval-id follow-up agent turn after each command finishes.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix requires maintainer policy on whether to pause, batch, drop, direct-deliver, or audit stale exec completion messages.
