## Summary

Keep this open. Current main still creates a separate agent continuation for each resolved exec approval, keyed by approval id, and the existing stop/queue cleanup paths do not cancel async exec producers that may dispatch later completion follow-ups.

## What This Changes

The issue asks OpenClaw to prevent approved async exec completion cascades from monopolizing a session and blocking newer user input.

## Best Possible Solution

Keep this as the canonical completion-cascade bug and design core scheduling semantics that coalesce or cancel stale approval-completion follow-ups, let /stop and newer user input preempt them, and preserve per-approval audit/result traceability.

## Reproduction Assessment

Yes. The source-level reproduction is multiple same-session gateway or node exec approvals that later complete; current main creates one follow-up agent turn per approval id, and the discussion includes a recent deterministic production reproduction on OpenClaw 2026.4.21.

## Solution Assessment

No. The current per-approval follow-up path is not the best solution for the reported UX; the safer direction is an explicit scheduler/circuit-breaker policy rather than relying on existing queue clearing or message dedupe.
