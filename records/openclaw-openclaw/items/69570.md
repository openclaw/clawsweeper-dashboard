- `P1`: The reported cascade can make an active agent/channel session effectively unusable for affected real users.
- `impact:session-state`: The issue is about delayed detached follow-up turns and session queue state continuing after the original task is done.
- `impact:message-loss`: The report says new user commands are ignored or blocked behind pending completion turns during the cascade.

## Summary

Keep open. Current main and the latest shipped release still model each approved async exec as its own approval-id-scoped detached follow-up agent turn, and the related reports do not fully supersede this approved-completion cascade problem.

## What This Changes

Make async exec approval completions cancellable, batched, or lower priority so multiple approvals do not create a delayed follow-up cascade that blocks user input.

## Best Possible Solution

Define a core exec approval completion scheduler that can coalesce or cancel stale follow-ups, prioritize stop and newer user input, and preserve a trustworthy audit trail for every approved command outcome.

## Reproduction Assessment

Yes, at source level. Configure gateway exec approvals, approve multiple same-session foreground commands, and current main still returns approval-pending while creating one detached approval-id follow-up agent turn after each command finishes.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix needs maintainer policy on pausing, batching, direct delivery, stale-drop behavior, and audit preservation; https://github.com/openclaw/openclaw/issues/70953 covers the broader approval-pending pause contract.
