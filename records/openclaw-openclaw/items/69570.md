## Summary

Keep this open. Current main still models exec approval completion as one session follow-up per approval id, and the related discussion plus source path show that /stop and queue cleanup do not define a safe policy for later async producers, batching, stale suppression, or user-input preemption.

## What This Changes

The issue asks OpenClaw to prevent multiple async exec approval completions from cascading into delayed agent turns that block newer user input.

## Best Possible Solution

Keep this as the canonical completion-cascade bug and design core scheduling semantics that coalesce or cancel stale exec approval follow-ups while preserving auditability and letting /stop or newer user input preempt the cascade.

## Reproduction Assessment

Yes, at source level. Multiple same-session gateway or node exec approvals still create one delayed follow-up agent turn per approval id, and the discussion includes a recent deterministic production reproduction on OpenClaw 2026.4.21.

## Solution Assessment

No, the current per-approval follow-up design is not the best solution for the reported UX. The maintainable path needs an explicit scheduler/circuit-breaker policy rather than relying on existing queue clearing or per-id dedupe.
