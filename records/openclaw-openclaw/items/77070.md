## Summary

Keep open: current main does not show a clear fix for the reported v2026.5.2 OpenAI timeout regression, and source inspection confirms the reporter’s attempted `openai-completions` workaround is normalized back onto Responses for the built-in OpenAI route.

## What This Changes

The issue reports that direct OpenAI API-key agent runs stopped replying after the v2026.5.2 update, with slow `models.list`, event-loop starvation, and `/v1/responses` fetch timeouts.

## Best Possible Solution

Diagnose the direct OpenAI API-key path on current main and fix the narrow transport or gateway hot path that causes agent turns to hang, with regression coverage for explicit route selection and the timeout symptom.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the complete outage: I did not live-reproduce the v2026.5.2 OpenAI timeout and event-loop delay. Source inspection does reproduce one important sub-behavior: an explicit built-in OpenAI `api: "openai-completions"` configuration is rewritten to Responses on `api.openai.com`, matching the reported failed workaround.

## Solution Assessment

Is this the best way to solve the issue? Unclear: keeping it open is the right outcome, but the narrowest fix is not proven yet. The strongest lead is route normalization or a provider/gateway hot path around OpenAI Responses, but maintainers should confirm with a live or synthetic regression before changing native OpenAI transport semantics.
