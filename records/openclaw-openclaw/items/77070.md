## Summary

Keep open: current main has a partial SSE default for the OpenAI provider, but post-fix confirmations and source inspection still show direct OpenAI Responses paths that can select the WebSocket transport when SSE is not explicit.

## What This Changes

The issue reports that OpenAI-backed gateway and embedded agent replies hang with idle/fetch timeouts after upgrading to 2026.5.2, while direct OpenAI curl succeeds.

## Best Possible Solution

Make every direct OpenAI Responses embedded, subagent, and compaction path resolve to SSE by default unless an explicit WebSocket override is intentionally honored, then update the tests, docs, and changelog to match.

## Reproduction Assessment

No live high-confidence reproduction was performed in this review. Source inspection gives a concrete current-main path where direct OpenAI Responses with `auto` or missing explicit SSE can still select WebSocket, and the issue has post-fix user confirmations.

## Solution Assessment

No, the current partial provider default is not enough to close this. The narrow maintainable fix is to centralize direct OpenAI Responses transport resolution on SSE by default while preserving explicit WebSocket only as an intentional override.
