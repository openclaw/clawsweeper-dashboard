## Summary

Keep open: current main contains a partial SSE default for the OpenAI provider, but the embedded direct OpenAI Responses path still has a current-main route that selects the OpenAI WebSocket manager unless SSE is explicit, and this issue was reopened after multiple post-fix confirmations.

## What This Changes

The issue requests fixing OpenAI gateway calls that hang after upgrade by routing direct OpenAI Responses embedded/subagent traffic through the working HTTP/SSE path instead of the stalled WebSocket path.

## Best Possible Solution

Make every direct OpenAI Responses embedded/subagent path resolve to SSE by default, preserve WebSocket only as an explicit override with fast failure, and align docs, tests, and changelog with that contract.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a medium-confidence current-main path where direct OpenAI Responses can still select WebSocket unless SSE is explicit, matching the reopened issue reports.

## Solution Assessment

Yes, the narrow maintainable solution is to fix transport resolution for direct OpenAI Responses embedded/subagent flows; removing all WebSocket code would be broader than needed unless maintainers choose that direction.
