## Summary

Keep open: the provider-level SSE default is present, but current main still has direct OpenAI embedded and compaction paths that select the OpenAI WebSocket transport unless SSE is explicit, matching the reopened post-fix reports and related #77570 context.

## What This Changes

The issue reports a regression where OpenAI-backed gateway and embedded agent runs time out indefinitely after updating to 2026.5.2.

## Best Possible Solution

Make every direct OpenAI Responses embedded, subagent, and compaction path resolve to SSE by default, while preserving an explicit `transport: "websocket"` override only where that behavior is intentionally supported.

## Reproduction Assessment

Yes, from source inspection: current main still routes direct OpenAI Responses attempts and compaction into the WebSocket path unless SSE is explicit, and post-fix issue comments report the same idle-timeout signature. I did not perform a live OpenAI/VPS run, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes, a narrow repair is the best next step: align the embedded and compaction transport selectors with the provider SSE default and update the conflicting tests/docs. Closing as implemented would be premature because the remaining WebSocket selection path is still present.
