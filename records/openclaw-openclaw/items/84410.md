- `merge-risk: 🚨 compatibility`: The PR changes upgrade/runtime behavior for existing `browser.defaultProfile` config writes from restart-applied to hot-applied.

## Summary

Keep open: the profiles hot-reload direction is plausible, but this PR also makes `browser.defaultProfile` hot even though implicit-profile selection reads the old default before request-time refresh, so it can run a browser action on the wrong profile.

## What This Changes

The PR changes browser reload metadata from a broad `browser` restart prefix to hot `browser.profiles` and `browser.defaultProfile` prefixes, with matching planner and plugin metadata tests.

## Best Possible Solution

Ship `browser.profiles` hot reload for the linked config.patch workflow, and include `browser.defaultProfile` only after the runtime refreshes before choosing the implicit profile.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: the PR classifies `browser.defaultProfile` as hot while current `forProfile()` chooses `current.resolved.defaultProfile` before the refresh. I did not run a live gateway test because this review is read-only.

## Solution Assessment

No, not as written. The `browser.profiles` portion is the right narrow fix, but `browser.defaultProfile` should either keep restart behavior or refresh before default selection with runtime coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not mark defaultProfile hot before refreshing selection:** `extensions/browser/plugin-registration.ts:59`
  - body: `forProfile()` reads `current.resolved.defaultProfile` before `resolveBrowserProfileWithHotReload()` refreshes the browser config. With this line, a `config.patch` that changes `browser.defaultProfile` no longer restarts the gateway, so the next browser request without an explicit profile can still run against the old default profile. Either leave `browser.defaultProfile` restart-required, or refresh before choosing the fallback default and add a runtime regression for that path.
  - confidence: 0.88
