## Summary

Keep open. The browser.profiles part addresses the linked gateway restart bug, but adding browser.defaultProfile as hot is unsafe because current route context selects the default profile before refreshing resolved browser config, so the first implicit browser request after a default-profile change can still use the stale profile.

## What This Changes

The branch replaces the browser plugin's broad restart prefix with hot prefixes for browser.profiles and browser.defaultProfile, with matching planner and plugin metadata tests.

## Best Possible Solution

Ship granular hot reload for browser.profiles, and include browser.defaultProfile only if the runtime refreshes before implicit default selection and has a regression plus real runtime proof.

## Reproduction Assessment

Yes. Source inspection shows the linked browser.profiles restart bug is addressed at the planner level, and it also shows the PR-introduced browser.defaultProfile issue because default selection happens before resolved config refresh.

## Solution Assessment

No as proposed. The maintainable path is to keep browser.defaultProfile restart-backed for this PR or make the default-profile runtime path genuinely hot before classifying it that way.
