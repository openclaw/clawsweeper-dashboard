## Summary

Keep open. The cron retry change addresses a real current-main behavior gap, but the PR is not ready because it also changes direct subagent/cron fallback delivery outside the cron restart scope and lacks the required user-facing changelog entry.

## What This Changes

The branch reschedules interrupted isolated agent-turn cron jobs on Gateway startup, documents that restart behavior, adds restart-catchup coverage, exports the startup deferral constant, and changes direct completion fallback text for subagent/cron task completions.

## Best Possible Solution

Land the cron retry implementation, docs, regression test, and changelog as one focused fix; keep direct fallback delivery result selection unchanged here unless maintainers decide on that policy in a separate agent-delivery PR.

## Reproduction Assessment

Yes. Persist an enabled one-shot isolated `agentTurn` cron job with stale `state.runningAtMs`, then start `CronService`; current main follows `markInterruptedStartupRun`, disables the `at` job, clears `nextRunAtMs`, and emits the restart-interrupted error, while the PR’s new test models the intended deferred retry state.

## Solution Assessment

No, not as the current combined PR. The cron retry approach is narrow and maintainable, but the fallback-delivery behavior change is unrelated and the required changelog entry is missing.
