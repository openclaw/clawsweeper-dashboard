## Summary

Keep open. The cron retry path itself appears valid, but the branch still mixes in a separate direct fallback-delivery behavior change and is missing the required user-facing changelog entry.

## What This Changes

The branch reschedules stale interrupted isolated agent-turn cron jobs on Gateway startup, documents and tests that retry behavior, exports the startup deferral constant, and also changes direct subagent/cron fallback completion text to use task/status summaries.

## Best Possible Solution

Land the isolated cron retry behavior with its docs, regression test, and a one-line Unreleased Fixes changelog entry; split the subagent/cron fallback-delivery behavior into a separate agent-delivery PR if maintainers want that change.

## Reproduction Assessment

Yes. Persist an enabled one-shot isolated `agentTurn` cron job with stale `state.runningAtMs` and a past `nextRunAtMs`, then start `CronService`; current main marks it interrupted/failed and disables it, while the PR test covers the desired deferred retry state.

## Solution Assessment

No, not as the current combined PR. The cron implementation is a narrow maintainable fix, but the fallback-delivery change is a separate behavior change and the changelog entry is still missing.
