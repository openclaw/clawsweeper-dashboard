## Summary

Keep open: the report is source-plausible and affects the compatibility-sensitive session/provider/auth state path, but current main does not yet provide enough proof that Codex model/auth-profile changes cannot still persist as ordinary OpenClaw session overrides for the originating Telegram session.

## What This Changes

The issue reports that a Codex-selected model/auth-profile override is persisted to a Telegram direct session key and later affects that unrelated chat session.

## Best Possible Solution

Codex model/auth-profile selections should persist only in the Codex app-server binding or an explicit Codex session key, while Telegram /status should show a session override only after an intentional OpenClaw session-level model selection.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the relevant persistence path and the report gives concrete steps, but the exact Codex-to-Telegram trigger still needs a focused regression test or live proof.

## Solution Assessment

Unclear. The maintainable direction appears to be using Codex's existing app-server binding for Codex-only model/auth-profile state, but maintainers should decide how generic session_status model mutations should behave inside Codex turns.
