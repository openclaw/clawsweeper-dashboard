## Summary

Keep open: the PR still retries after the shared exec helper has already emitted the transient verbose failure log, so it does not fully address the reported scary startup symptom, and the supplied proof is injected rather than a real Tailscale gateway run.

## What This Changes

Adds a retrying `tailscale status --json` helper for `getTailnetHostname`, missing-binary short-circuit coverage, and a simulated repro script.

## Best Possible Solution

Use a quiet or delayed status probe that logs only after all retries fail, preserve the fast missing-binary fallback, and prove it on a real Tailscale gateway restart.

## Reproduction Assessment

Yes, at source level: current main immediately calls `getTailnetHostname` after `enableTailscaleServe`, and `runExec` logs failures before callers can catch them. I did not establish a live Tailscale daemon reproduction in this read-only review.

## Solution Assessment

No. Retrying near the status probe is the right area, but this implementation catches the failure after `runExec` has already logged the scary transient line; a quiet retry path or a short delay before the first probe is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Delay the first status probe or use a quiet retry path that logs only after all attempts fail.
