## Summary

Keep open: the PR still routes intermediate retry attempts through the normal verbose `runExec` failure path, so the reported transient `Command failed: tailscale status --json` line can still be emitted before a later retry succeeds. The branch also has only injected-exec proof, not real Tailscale gateway restart proof.

## What This Changes

The PR adds a three-attempt 500ms retry wrapper for `getTailnetHostname`'s `tailscale status --json` call, tests transient and missing-binary cases, and adds an injected-exec repro script.

## Best Possible Solution

Keep the retry localized to the optional status probe, but make intermediate attempts quiet or final-attempt-only and prove transient retries stay quiet while final failures still surface.

## Reproduction Assessment

Yes at source level: current main runs `getTailnetHostname` immediately after `enableTailscaleServe`, and a failing status probe reaches `runExec` verbose failure logging before the caller catch swallows the optional hostname failure. I did not run a live macOS Tailscale restart.

## Solution Assessment

No. Retrying the status probe is the right shape, but this implementation must suppress or defer intermediate `runExec` failure logging and still surface the final failure normally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Suppress transient status logs before retry succeeds:** `src/infra/tailscale.ts:140-143`
  - body: `readTailscaleStatusJson` retries only after awaiting the normal exec helper, but `runExec` emits `Command failed: tailscale status --json` under verbose logging before it rethrows. The first startup-race failure can still produce the reported scary line even when a later retry returns a host, so use a quiet probe or defer logging until all retry attempts fail.
  - confidence: 0.91
