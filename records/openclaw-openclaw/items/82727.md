## Summary

Keep open: the PR targets the right Tailscale startup race and has addressed the missing-binary retry delay, but it still retries after the default exec helper can emit the verbose failure line the linked report is about, and the supplied proof is an injected-exec simulation rather than a real Tailscale restart.

## What This Changes

Adds a retrying Tailscale `status --json` reader for `getTailnetHostname`, tests transient and missing-binary paths, and adds an injected-exec repro script.

## Best Possible Solution

Keep the localized retry around the hostname status probe, but make retried status executions quiet until the final failure and prove that with focused coverage plus real macOS/Tailscale restart output.

## Reproduction Assessment

Yes at source level: current main enables Tailscale Serve and immediately calls `getTailnetHostname`, while `runExec` logs command failures before throwing under verbose logging. I did not run a live macOS Tailscale daemon restart in this read-only review.

## Solution Assessment

No. Retrying the status probe is the right shape, but the current patch catches after the default exec path can already log the transient failure; a quiet or final-attempt-only logging path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Suppress transient status logs before retry succeeds:** `src/infra/tailscale.ts:140-145`
