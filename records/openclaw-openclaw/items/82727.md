## Summary

Keep this PR open: it targets the right Tailscale startup path and now avoids retrying missing-binary errors, but it still catches after the default exec helper has already logged the transient `status --json` failure, so the reported scary verbose line can remain. The after-fix proof is also an injected-exec simulation rather than a real Tailscale daemon restart.

## What This Changes

The PR adds a retrying Tailscale `status --json` helper used by `getTailnetHostname`, unit coverage for transient and missing-binary cases, and an injected-exec repro script.

## Best Possible Solution

Keep the localized retry and non-executable short-circuit, but make transient status probes quiet until the final failure and add coverage proving recovered retries do not emit the verbose failure line.

## Reproduction Assessment

Yes at source level: current main enables Serve, immediately probes `tailscale status --json`, and `runExec` logs failures before the caller catch can hide them. I did not establish the transient daemon failure against a live macOS Tailscale setup in this read-only review.

## Solution Assessment

No. Retrying the hostname status probe is the right shape, but this implementation catches after the default exec helper has already logged the transient failure, so a quiet probe or final-attempt-only log path is the narrower maintainable fix.

## Review Findings
