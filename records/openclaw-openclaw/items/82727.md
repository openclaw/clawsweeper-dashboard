## Summary

Keep open: the PR is a plausible focused fix, but it is draft with changes requested and still lets the transient `tailscale status --json` failure log before a later retry can succeed. The supplied behavior proof is an injected simulation rather than real Tailscale gateway restart output.

## What This Changes

The PR adds a retrying Tailscale status JSON reader for `getTailnetHostname`, transient and missing-binary tests, and an injected-exec repro script.

## Best Possible Solution

Make the optional status probe retry quietly or log only on the final failed attempt, while preserving final failure visibility and adding focused tests plus redacted live restart proof.

## Reproduction Assessment

Yes at source level: Serve mode immediately calls `getTailnetHostname` after `enableTailscaleServe`, and `runExec` logs a failed `tailscale status --json` before rethrowing. I did not establish a live macOS Tailscale daemon reproduction in this read-only review.

## Solution Assessment

No. Retrying the status probe is the right narrow shape, but this patch still needs quiet or final-attempt-only logging for intermediate failures and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Suppress intermediate status retry logs:** `src/infra/tailscale.ts:140-145`
  - body: `readTailscaleStatusJson` retries after awaiting the normal `runExec` helper, but `runExec` emits `Command failed: tailscale status --json` under verbose logging before it rethrows. A first startup-race failure can therefore still show the user-visible scary line even when the second attempt succeeds, so intermediate attempts need to be quiet or logged only after all retries fail.
  - confidence: 0.91
