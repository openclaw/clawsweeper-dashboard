## Summary

Keep open: the PR is a focused fix for a real source-level Tailscale startup race, but the latest patch still lets the reported verbose `Command failed: tailscale status --json` line be emitted before the retry wrapper catches the failure, and the supplied proof is an injected-exec simulation rather than a real Tailscale daemon run.

## What This Changes

The PR adds retry handling for `tailscale status --json` in tailnet hostname discovery, plus unit coverage and a simulated repro helper.

## Best Possible Solution

Keep the retry near the optional hostname probe, but make that probe quiet or defer logging until all retry attempts fail, while preserving the fast missing-binary fallback and adding regression coverage for the noisy-log case.

## Reproduction Assessment

Yes at source level: current main immediately calls hostname discovery after enabling Tailscale Serve, and `runExec` logs command failures before the optional caller catch can hide them. I did not establish a live macOS launchd/Tailscale daemon reproduction in this read-only review.

## Solution Assessment

No. Retrying the status probe and short-circuiting ENOENT are the right area, but this implementation still catches after `runExec` has already logged the transient failure; a quiet probe or delayed final-failure log is the safer maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Suppress transient status logs before retry succeeds:** `src/infra/tailscale.ts:140-143`

A first startup-race failure can therefore still emit the reported scary line even if a later retry succeeds; use a quiet status probe or log only after all retry attempts fail.
