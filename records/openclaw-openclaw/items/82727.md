## Summary

Keep open: the branch is aimed at the right bug, but it still retries through the verbose `runExec` path that emits the reported `Command failed: tailscale status --json` line before a retry can succeed, and the supplied proof is an injected simulation rather than real Tailscale startup evidence.

## What This Changes

Adds a retrying `tailscale status --json` helper for `getTailnetHostname`, tests transient and missing-binary paths, and adds a simulated repro script.

## Best Possible Solution

Keep a localized retry around optional post-Serve hostname discovery, but defer or suppress status failure logging until retries are exhausted and validate it against a real macOS/Tailscale restart.

## Reproduction Assessment

Yes at source level: Serve startup immediately calls `getTailnetHostname` after `enableTailscaleServe`, and default `runExec` logs failed status commands before throwing under verbose/debug logging. I did not run a live macOS Tailscale daemon restart.

## Solution Assessment

No. Retrying the status probe is the right boundary, but this patch retries after the logging helper has already emitted the transient failure; the narrower maintainable fix is a quiet or final-attempt-only status probe plus real proof.
