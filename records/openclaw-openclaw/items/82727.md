## Summary

Keep open. The follow-up addresses the earlier missing-binary fallback concern, but the retry still happens after `runExec` has already logged the transient `tailscale status --json` failure, so the main user-visible symptom remains; the supplied proof is also simulated/injected rather than real Tailscale daemon behavior.

## What This Changes

The PR adds a three-attempt `tailscale status --json` retry helper for `getTailnetHostname`, skips retries for missing binaries, adds unit coverage, and adds an injected repro script.

## Best Possible Solution

Retry or delay the status probe in a way that prevents transient verbose error logs until all attempts fail, while preserving fast missing-binary fallback and adding redacted real gateway/Tailscale proof.

## Reproduction Assessment

Yes, at source level: current main runs Serve setup and then `tailscale status --json`, and `runExec` logs command failures before callers can catch them. I did not run a live Tailscale daemon reproduction in this read-only review.

## Solution Assessment

No. Retrying near hostname/status probing is the right area, but this implementation catches the failure after the shared exec helper has already logged the scary transient line; the safer fix is a quiet retry path or a delay before the first status probe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Avoid logging transient failures before retry succeeds:** `src/infra/tailscale.ts:141-144`
  - body: `runExec` logs `Command failed: tailscale status --json` in its catch before rethrowing, so a first-attempt startup race still emits the exact scary verbose line even if `readTailscaleStatusJson` catches it and the second attempt succeeds. Delay before the first status probe or use a quiet retry path that logs only after all attempts fail.
