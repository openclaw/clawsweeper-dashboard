## Summary

Keep this PR open: it targets a valid Tailscale startup-race bug, but the latest patch still catches after `runExec` has already emitted the noisy `Command failed` line, and the supplied proof is injected-exec/mock-only rather than a real daemon startup proof.

## What This Changes

The PR retries `tailscale status --json` inside tailnet hostname discovery, skips retries for missing-binary errors, adds unit coverage, and adds an injected-exec repro script.

## Best Possible Solution

Keep the retry near the optional hostname probe, but make that status probe quiet or defer logging until all retry attempts fail, while preserving the fast missing-binary fallback and adding regression coverage for noisy-log suppression.

## Reproduction Assessment

Yes at source level: current `main` enables Tailscale Serve, immediately probes `tailscale status --json`, and `runExec` logs failures before the caller catch can hide them. I did not establish a live Tailscale daemon reproduction in this read-only review.

## Solution Assessment

No. Retrying the status probe is the right area, but this implementation catches after the default exec helper has already logged the transient failure; a quiet probe or final-attempt-only log path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91
