## Summary

Keep open: this PR targets the linked Tailscale startup-noise bug in the right area, but the patch retries deterministic missing-binary failures before falling back to the macOS app-path candidate, and the supplied proof is injected/mock-only rather than a real Tailscale or gateway run.

## What This Changes

Adds a three-attempt 500ms retry helper for `tailscale status --json`, wires `getTailnetHostname` through it, adds a unit test, and adds an injected-exec repro script.

## Best Possible Solution

Retry only plausible transient status failures while preserving fast fallback to the macOS app-path candidate, then verify with focused tests and redacted real gateway/Tailscale output.

## Reproduction Assessment

Yes, at source level: current main still shells out to `tailscale status --json` immediately after Serve setup and logs failures before the caller swallows optional hostname discovery errors. The PR-introduced fallback delay is also source-reproducible because the helper retries every caught status error before moving to the next candidate.

## Solution Assessment

No: the retry belongs near hostname/status probing, but retrying every command failure is broader than the bug. The narrower fix is to retry only transient status failures or short-circuit non-executable launch errors so candidate fallback remains fast.

## Review Findings
