## Summary

Keep open: the retry is in the right area and the latest head now skips missing-binary retries, but the patch still catches after `runExec` has already emitted the transient verbose failure line, so it does not yet fix the reported noisy startup behavior. The supplied after-fix proof is still an injected-exec simulation rather than a real Tailscale gateway restart.

## What This Changes

The PR adds a retrying `readTailscaleStatusJson` helper, unit coverage, and an injected-exec repro script for transient `tailscale status --json` failures.

## Best Possible Solution

Keep the retry near the optional hostname probe, but make the status probe quiet or defer logging until all retry attempts fail, while preserving the missing-binary short-circuit and adding regression coverage for suppressed transient verbose logs.

## Reproduction Assessment

Yes at source level: current main enables Serve, immediately probes `tailscale status --json`, and `runExec` logs failures before the caller catch can hide them. I did not establish the transient daemon failure against a live macOS Tailscale setup in this read-only review.

## Solution Assessment

No. Retrying the status probe is the right shape, but this implementation catches after the default exec helper has already logged the transient failure; a quiet probe or final-attempt-only log path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Suppress transient status logs before retry succeeds:** `src/infra/tailscale.ts:140-143`
