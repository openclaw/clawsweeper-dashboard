## Summary

Keep open: current main and the latest release still expose only string-based CLI backend output replacements, while this asks for a new public plugin SDK hook that needs maintainer API direction before implementation.

## What This Changes

Request to add an optional `CliBackendPlugin.transformAssistantText` hook that rewrites parsed assistant text before existing static output replacements.

## Best Possible Solution

Choose the stable SDK contract first, then implement the accepted narrow hook or callback-replacement design with runner wiring, public docs, API baseline updates, and focused tests that preserve existing static replacement behavior.

## Reproduction Assessment

Yes, at source level: current main exposes only string-based `textTransforms.output`, and the CLI runner applies those static replacements after parsing. No live current-main Antigravity reproduction is available because the motivating backend remains in open PRs.

## Solution Assessment

Unclear until maintainers choose the public SDK contract. The dedicated backend-owned hook is plausible, but callback-capable replacements or a broader CLI event hook should be compared before codifying a new plugin API.
