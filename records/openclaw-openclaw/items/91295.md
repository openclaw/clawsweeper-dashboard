## Summary

Keep open: current main and the latest release still only expose static string output replacements for CLI backends, while this asks for a new public plugin SDK hook that needs maintainer API direction before implementation.

## What This Changes

The issue requests an optional `CliBackendPlugin.transformAssistantText` hook for backend-owned post-parse assistant text rewrites in CLI backends.

## Best Possible Solution

Choose the stable SDK contract first, then implement the accepted hook or replacement-callback design with runner wiring, public SDK exports/docs, API baseline updates, and focused tests that preserve existing static replacement behavior.

## Reproduction Assessment

Yes, at source level: current main exposes only string-based `textTransforms.output`, and the CLI runner applies those static replacements after parsing. I did not run a live Antigravity reproduction because the motivating backend remains in open PR work.

## Solution Assessment

Unclear until maintainers choose the public SDK contract. A dedicated backend-owned hook is plausible, but callback-capable replacements and the broader JSONL/event hook direction should be compared before codifying a new plugin API.
