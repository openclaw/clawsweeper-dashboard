## Summary

Keep open: the production fix is pointed at a real memory-core data-loss path, but the current PR still has a blocking regression-test issue because the injected read error does not hit the fs-safe read path that production now uses.

## What This Changes

The branch rethrows operational phase-signal read errors, removes stale short-term recall/phase-signal temp files during repair, reports cleanup counts in memory CLI/dreaming summaries, syncs the SDK facade type, and updates tests plus the changelog.

## Best Possible Solution

Land the production behavior after the regression test injects the read failure at the real fs-safe boundary, doctor repair output/tests include removedTempFiles, and the unrelated dependency-audit gate is resolved or explicitly waived by the appropriate owner.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction path: make privateFileStore(...).readJsonIfExists fail with a non-not-found error while phase-signals.json contains prior entries, and recordDreamingPhaseSignals will fall back to an empty store and write it back.

## Solution Assessment

No, not yet. The production fix is the narrow maintainable direction, but the test currently mocks the wrong read API and the doctor repair surface has not been kept in sync with the new removedTempFiles result.

Overall correctness: patch is incorrect

- **[P2] Inject the read failure through fs-safe:** `extensions/memory-core/src/short-term-promotion.test.ts:855-859`

  - confidence: 0.91
