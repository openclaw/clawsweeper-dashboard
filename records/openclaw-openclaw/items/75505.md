## Summary

Keep open. This is a narrow status-rendering PR, but the proposed precedence change is not safe as written: it can pair the session-selected provider with the active runtime model, while the current status renderer already treats selected overrides and active runtime/fallback identity as separate concepts. The PR needs a focused repair plus regression coverage before merge.

## What This Changes

The PR changes `src/agents/tools/session-status-tool.ts` so the `session_status` card provider/model fallback chain prefers runtime identity, then session overrides, then configured defaults.

## Best Possible Solution

Revise the PR so `session_status` resolves selected and active model identities as coherent pairs, preserving the selected override pair for selected-card inputs while allowing active runtime/fallback display through the existing active-model path. Add focused tests for runtime-only, override-only, and override-plus-runtime/fallback cases, plus a changelog entry credited to the contributor.

## Reproduction Assessment

Yes for the PR defect. A static trace with `providerOverride`/`modelOverride` plus `modelProvider`/`model` shows the new `defaultModelForCard` can take the runtime model while `providerForCard` still keeps the selected provider, producing a mismatched provider/model pair.

## Solution Assessment

No. The patch changes fallback variables locally instead of reusing or extending the selected/active identity contract, so it is not the narrowest maintainable fix; the safer repair is to compute selected and active identities explicitly and keep provider/model pairs together.
