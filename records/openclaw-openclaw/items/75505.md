## Summary

Keep open. The PR is aimed at a real `session_status` display gap, but the latest patch still regresses current status semantics: it ignores providerless persisted runtime models and collapses selected-card inputs with active runtime identity, while current code/docs/tests keep selected overrides and active runtime/fallback display separate.

## What This Changes

The PR modifies `src/agents/tools/session-status-tool.ts` so `session_status` resolves provider/model display inputs from a runtime-first paired identity before falling back to session overrides or configured defaults.

## Best Possible Solution

Revise the PR so `session_status` computes selected-card identity and active runtime identity as separate values, preserves providerless runtime-model display when no explicit override exists, and adds regression coverage plus a changelog entry for the user-visible fix.

## Reproduction Assessment

Yes for the PR defects. A static trace of the PR branch shows a session entry with only `model: "legacy-runtime-model"` falls through to the configured default, and an explicit selected override plus runtime provider/model makes `buildStatusText` compute selected auth/default-thinking from the active runtime pair before final rendering re-reads the override.

## Solution Assessment

No. The paired runtime-first branch avoids one mixed-pair case but is not the best fix because it collapses selected and active identity responsibilities; the safer path is to keep selected override/default inputs coherent and let the existing active-model path use `sessionEntry.modelProvider`/`sessionEntry.model`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve providerless runtime models:** `src/agents/tools/session-status-tool.ts:535-545`
  - body: The new branch only treats the runtime identity as usable when both `runtimeProviderForCard` and `runtimeModelForCard` are present. Existing behavior deliberately supports a bare persisted runtime model by passing that model through and clearing `providerOverride`; with this patch a session whose entry only has `model: "legacy-runtime-model"` falls back to the configured default instead, regressing the current status-card test.
  - confidence: 0.92
- **[P2] Keep selected and active identities separate:** `src/agents/tools/session-status-tool.ts:535-537`
  - body: When a session has an explicit selected override and also has active runtime fields, this branch passes the runtime pair as the selected `provider`/`model` inputs to `buildStatusText`. That makes selected auth/default-thinking and the active-diff calculation use the active runtime identity before `buildStatusMessage` later re-reads the override, so selected and fallback labels can get the wrong auth/thinking context. Keep selected-card inputs from the selected override/default pair and let `sessionEntry` drive active runtime display.
  - confidence: 0.84
