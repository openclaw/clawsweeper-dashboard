## Summary

Keep open. The PR targets a real status-rendering area, but the current diff regresses providerless runtime-model display and mixes selected-model identity with active runtime identity, so it should not merge as-is.

## What This Changes

The PR changes `src/agents/tools/session-status-tool.ts` so `session_status` chooses a runtime-first provider/model pair for the status card before falling back to session overrides or configured defaults.

## Best Possible Solution

Keep the status card's selected identity derived from the explicit session override or configured default, preserve the existing providerless-runtime display path for legacy/current runtime-only sessions, and let `sessionEntry.modelProvider`/`sessionEntry.model` drive active runtime or fallback display. Add focused regression coverage for both cases plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A static trace of the PR diff shows a session entry with only `model: "legacy-runtime-model"` falls into the configured-default branch, and an explicit selected override plus runtime provider/model passes the runtime pair into `buildStatusText` as the selected identity.

## Solution Assessment

No. The runtime-first paired-identity branch fixes one mixed-pair symptom but is not the safest design because selected status inputs and active runtime state have different responsibilities; the narrower fix is to preserve the selected/default path and keep active runtime display separate.
