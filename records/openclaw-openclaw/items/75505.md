## Best Possible Solution

Keep selected/default identity and active runtime identity separated, but preserve the existing runtime resolver and providerless-session synthesis for no-override sessions; remove dependency/artifact churn and add a regression that asserts the actual status-builder contract.

No. The requested selected-versus-active split is a reasonable direction, but this patch replaces too much of the runtime identity path and mixes in unrelated install-surface changes; a narrower override-only adjustment is safer.
