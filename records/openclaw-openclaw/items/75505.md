## What This Changes

The PR changes `session_status` model/provider resolution, adds one regression test and a changelog entry, and currently adds an accidental root-level patch-fragment file.

## Best Possible Solution

Keep the selected/default identity and active runtime identity separated: preserve the existing runtime resolver for runtime/provider inference, keep the providerless legacy-model synthesis, add the explicit override regression from this PR, remove the artifact file, and retain an accurate changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available from current-main tests: providerful runtime sessions, configured custom-provider runtime-only sessions, and providerless legacy runtime sessions all have expected status-card assertions that the PR’s latest resolution chain would violate.

## Solution Assessment

No. The proposed fallback chain is not the best fix because it removes the resolver that keeps runtime provider/model identity coherent; the safer repair is to keep the current selected-vs-active split and add only the missing explicit-override coverage.
