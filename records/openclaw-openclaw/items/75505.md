## Summary

Keep open. The PR targets a real status-rendering problem, but the current diff still changes the wrong contract: it regresses runtime-only sessions with no provider and conflates the selected model identity with the active runtime identity before the shared status renderer runs.

## What This Changes

The PR changes `src/agents/tools/session-status-tool.ts` so `session_status` selects a provider/model pair from runtime fields first, then complete session overrides, then configured defaults.

## Best Possible Solution

Keep the status card's selected identity derived from explicit session overrides or configured defaults, preserve the current providerless runtime-model display path, and let `sessionEntry` runtime fields drive active or fallback status only when that state is intentionally shown.

## Reproduction Assessment

Yes. A static trace of the PR diff is enough: a session entry with only `model: "legacy-runtime-model"` falls through to the configured default, and an explicit override plus runtime provider/model passes the runtime pair into selected status inputs.

## Solution Assessment

No. Runtime-first paired identity is not the narrowest maintainable fix because selected status and active runtime state have separate contracts; the safer fix is to preserve the selected/default path and keep runtime state as active/fallback context.
