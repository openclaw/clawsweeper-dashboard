## Summary

Keep open. The latest branch still violates the status identity contract: current main keeps selected/default identity separate from active runtime state and has regression coverage for providerless runtime models, while this PR can still render mismatched provider/model pairs and also includes an accidental top-level diff artifact.

## What This Changes

This PR updates `session_status` status-card provider/model resolution in `src/agents/tools/session-status-tool.ts` and currently adds a stray top-level diff-artifact file.

## Best Possible Solution

Keep the current selected/default versus active-runtime split, preserve the providerless runtime-model display path, remove the accidental artifact file, then add focused regression coverage and the required user-facing changelog entry for any corrected status behavior.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: with a session entry that has `model` but no `modelProvider`, the PR falls back to the configured provider and leaves the session entry untouched, while current main's regression test expects a providerless `legacy-runtime-model` card.

## Solution Assessment

No. The latest PR shape is not the best fix because it mixes selected identity and runtime identity; the safer repair is to keep selected overrides/defaults and active runtime state as separate inputs to the shared status renderer.
