## Summary

Keep open: the selected-vs-runtime status bug is real, but this PR is not safe to merge because it removes resolver-backed runtime identity handling, adds an assertion that does not match the local status mock contract, and still includes unrelated lockfile plus root-file artifacts.

## What This Changes

This PR changes `session_status` provider/model precedence for runtime and session overrides, adds a focused regression test and changelog entry, and modifies `pnpm-lock.yaml` plus two root files.

## Best Possible Solution

Preserve `resolveSessionModelIdentityRef` for runtime identity, layer explicit selected provider/model display on top of it, and keep the PR diff limited to status code, focused tests, and the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a session entry with runtime `modelProvider`/`model` plus `providerOverride`/`modelOverride`, then current `session_status` label construction uses the override provider but not the override model; the PR's added test sketches that path, though it asserts the mock incorrectly.

## Solution Assessment

No. The requested selected-vs-active fix is the right direction, but replacing the shared resolver with direct fallback logic is not the narrow maintainable solution because it regresses custom-provider inference and providerless runtime identity.
