## Summary

Keep open. The PR has useful security-hardening intent, but it is not mergeable because the self-hosted discovery SSRF policy is broader than current main, the new alias scanner rule still leaves a common bypass, and no after-fix real behavior proof is present.

## What This Changes

The PR adds JavaScript/TypeScript skill-scanner rules, switches self-hosted model discovery to guarded fetch, and caps plugin typed-hook priority by plugin origin.

## Best Possible Solution

Rebase and split the work: preserve current main's host-pinned SSRF discovery policy, land scanner hardening with focused positive and false-positive tests, and handle hook priority caps as a documented security contract after secops review.

## Reproduction Assessment

Yes for the blocking PR findings: source inspection of the PR head shows the unpinned allowPrivateNetwork SSRF policy and an alias regex that misses non-leading ESM import aliases. I did not run live exploits or tests because this was a read-only review.

## Solution Assessment

No. Current main's host-pinned guarded-fetch policy is narrower, and the scanner plus hook-policy changes need rebasing, tests, and security ownership review rather than landing as a combined patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Pin self-hosted discovery to the configured host:** `src/plugins/provider-self-hosted-setup.ts:71-72`
  - body: The guarded fetch call only passes `allowPrivateNetwork: true` and no hostname allowlist. In the SSRF guard, that skips private-network checks and an empty allowlist accepts any hostname, so metadata or other private hosts remain reachable; preserve current main's host-pinned policy instead.
  - confidence: 0.9
- **[P2] Match aliases after earlier import specifiers:** `src/security/skill-scanner.ts:203`
  - body: The alias rule only matches when `exec` or another dangerous function is the first item after `{`. `import { readFile, exec as run } from "node:child_process"` can still call `run(...)` without matching this rule or `dangerous-exec`, leaving the advertised ESM alias hardening incomplete.
  - confidence: 0.85
