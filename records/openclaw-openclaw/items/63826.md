## Summary

Keep open: this external security PR still contains potentially useful scanner and hook hardening, but the current head is not mergeable as-is because it weakens main's self-hosted discovery SSRF policy, leaves an advertised scanner bypass, lacks real behavior proof, and includes a hook-priority policy decision that needs owner/security review.

## What This Changes

The branch adds JavaScript/TypeScript skill-scanner heuristics, switches self-hosted OpenAI-compatible model discovery to guarded fetch, and clamps typed hook priorities for bundled versus external plugins.

## Best Possible Solution

Land a rebased, focused security change that preserves current self-hosted SSRF protections, fixes scanner gaps with positive and false-positive tests, and makes any hook-priority limits through an explicit reviewed plugin contract.

## Reproduction Assessment

Yes for the blocking PR-review defects via source inspection: the PR head omits the configured-host allowlist and the alias regex misses non-leading ESM import aliases. I did not run tests or live exploits because this is a read-only review.

## Solution Assessment

No. The safer path is to rebase and split the work: retain main's host-pinned SSRF guard, add scanner hardening with focused tests, and review hook priority caps as a security contract rather than landing this combined branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pin self-hosted discovery to the configured host:** `src/plugins/provider-self-hosted-setup.ts:71-72`
  - body: The guarded fetch policy only sets `allowPrivateNetwork` and leaves the hostname allowlist empty. Since an empty allowlist is treated as allow-all while current main pins discovery to the configured host, rebasing this hunk would weaken the SSRF policy and allow redirects or proxy-mediated discovery to leave the configured host while private-network checks are bypassed.
  - confidence: 0.91
- **[P2] Match aliases after earlier import specifiers:** `src/security/skill-scanner.ts:203`
  - body: The alias rule only matches when `exec` or `spawn` is the first import specifier after `{`. A common form such as `import { readFile, exec as run } from "node:child_process"` can still invoke `run(...)` without matching this rule or the existing direct-call rule, leaving the advertised alias hardening incomplete.
  - confidence: 0.86
