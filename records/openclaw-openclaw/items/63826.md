## Summary

Keep this PR open for maintainer/security review. Current main already has a stronger self-hosted provider discovery SSRF fix, but the scanner hardening and hook-priority policy are still not implemented on main; the PR also needs reconciliation so it does not weaken the shipped guarded-fetch policy.

## What This Changes

The PR adds additional skill-scanner regex rules, changes self-hosted model discovery to use `fetchWithSsrFGuard`, and clamps typed hook priority by bundled versus external plugin origin.

## Best Possible Solution

Split or revise this security work into focused follow-ups: preserve current main's stronger self-hosted provider guard, land tested scanner rules with explicit false-positive coverage, and decide any hook-priority cap as a documented plugin contract/security policy change.

## Reproduction Assessment

Yes for the reviewed behavior: source inspection and `rg` on current main reproduce that the scanner rules and hook cap are absent, while the PR diff plus `ssrf.ts` show why the provider policy is too broad. Full exploit execution was not run in this read-only review.

## Solution Assessment

No, the current PR is not the best merge shape. The provider fix is already shipped more safely on main, while scanner and hook-policy changes need targeted tests, compatibility review, and security-owner sign-off.
