## Summary

Keep open: current main still lacks an alias/profile-scoped route identity, and the Anthropic CLI auth migration path can still collapse a colliding `claude-cli/*` allowlist entry into an existing `anthropic/*` entry. The related model-profile and `routeTo` work overlaps, but it is open/unmerged and does not yet settle the Anthropic CLI/API migration contract.

## What This Changes

Request a public config/runtime contract that lets one Anthropic upstream model be selected through both CLI and API routes without losing either alias during migration.

## Best Possible Solution

Define one selectable route/profile identity contract, then make Anthropic CLI/API migrations preserve colliding routes with clear warnings, docs, and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established because this was a read-only review. Source inspection does reproduce the risky migration path: a colliding `claude-cli/*` key is deleted after canonicalization, while the current schema has no alias-scoped route identity to preserve both routes.

## Solution Assessment

Unclear. Per-alias runtime overrides are plausible, but maintainers need to choose between alias metadata, sibling keys, `routeTo`, provider/name/API-id decoupling, and narrower migration collision guards before implementation.
