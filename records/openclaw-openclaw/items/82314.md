## Summary

Keep open: current main still lacks alias-scoped route identity for one upstream model via both Anthropic API and Claude CLI, and the remaining fix is a public config/runtime product decision. The HammerMei `poor-claude` comment is useful stopgap context, but it does not replace a core route-selection contract.

## What This Changes

Request a per-alias or sibling-key runtime override so one Anthropic model can be configured through both direct API and Claude CLI routes in one agent.

## Best Possible Solution

Define one route-identity contract for model aliases or sibling keys, then make doctor and Anthropic CLI auth migrations preserve colliding CLI/API routes with clear warnings and regression coverage.

## Reproduction Assessment

No, not as a live end-to-end command in this read-only pass. Source inspection shows the Anthropic CLI auth migration can collapse colliding claude-cli/* and anthropic/* entries, while current doctor tests only prove partial preservation for the core doctor normalizer.

## Solution Assessment

Unclear. Per-alias runtime overrides are plausible, but maintainers need to choose the public route-identity shape, runtime precedence, auth-profile behavior, and non-destructive migration semantics before implementation.
