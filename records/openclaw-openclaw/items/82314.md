## Label Justifications

- `P2`: This is a meaningful config/runtime improvement with possible billing and migration consequences, but it needs product design rather than emergency repair.
- `impact:data-loss`: The reported migration path can remove a user-configured `claude-cli/*` model entry when it collides with a canonical Anthropic entry.
- `impact:auth-provider`: The issue is about preserving model choice, auth-profile intent, and runtime routing between Anthropic API and Claude CLI paths.

## Summary

Keep open: current main still lacks an alias-scoped route identity for selecting the same Anthropic model through both API and Claude CLI routes, and the remaining fix is a public config/runtime decision. The linked `poor-claude` workaround is useful context, but it does not replace the OpenClaw route-identity contract requested here.

## What This Changes

Request support for per-alias or sibling-key runtime overrides so one agent can expose both Anthropic API and Claude CLI routes for the same model without migration collisions.

## Best Possible Solution

Define one route-identity contract for model aliases or sibling keys, then make Anthropic CLI/API migrations preserve colliding routes with warnings and regression coverage.

## Reproduction Assessment

No for a full live `doctor --fix` reproduction in this read-only review. Yes at source level for the Anthropic CLI provider-auth migration path that rewrites and deletes colliding `claude-cli/*` entries while current schema still lacks alias-scoped route identity.

## Solution Assessment

Unclear. Per-alias runtime overrides are plausible, but maintainers should first choose whether route identity belongs in alias metadata, sibling model keys, auth-profile selection, or a narrower migration collision guard.
