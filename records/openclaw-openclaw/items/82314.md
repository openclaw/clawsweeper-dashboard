## Summary

Current main fixed the narrower gateway fallback-to-Anthropic-API regression, but it still lacks a route identity that can keep Claude CLI and Anthropic API variants of the same upstream model selectable in one defaults model map. The remaining work is a public config/runtime design decision, not conservative cleanup.

## What This Changes

The issue requests a per-alias or sibling-key route identity so one configured Anthropic model can keep distinct Claude CLI and direct API paths without destructive migration collisions.

## Best Possible Solution

Define one backward-compatible model route-identity contract, then update schema/help/docs, picker/listing, runtime policy resolution, auth-profile selection, migrations, and regression coverage together.

## Reproduction Assessment

No high-confidence live reproduction was performed because this review is read-only. Source inspection does show the provider-auth migration still collapses colliding `claude-cli/*` and `anthropic/*` entries, while current schema still lacks any alias-scoped route contract.

## Solution Assessment

Unclear. Per-alias runtime overrides are plausible, but current related work shows maintainers still need to choose between alias metadata, sibling keys, `routeTo`, provider/name keying, or a narrower non-destructive migration guard.
