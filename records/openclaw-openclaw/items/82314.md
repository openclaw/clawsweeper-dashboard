## Summary

Keep open: current main has a partial Claude CLI runtime migration fix, but it still lacks a route identity that can represent API and Claude CLI transports for the same Anthropic model in one agent config, and the remaining choice is a public config/runtime product decision.

## What This Changes

The issue requests per-alias or equivalent route-scoped model runtime configuration so one Anthropic model can be selected through both direct API and Claude CLI transports without destructive migration collisions.

## Best Possible Solution

Define one maintainer-approved route-identity contract for model selection, then update migration, picker, runtime policy, auth-profile routing, docs, and regression tests to preserve distinct API and Claude CLI routes without resurrecting ambiguous legacy defaults.

## Reproduction Assessment

No high-confidence live reproduction was run for the exact reported command. Source inspection does show the current schema lacks alias-scoped route metadata and the Anthropic CLI auth migration can still collapse colliding legacy and canonical model entries.

## Solution Assessment

Unclear. Per-alias runtime overrides are plausible, but maintainers need to choose between alias metadata, sibling route keys, or a narrower non-destructive migration guard before implementation.
