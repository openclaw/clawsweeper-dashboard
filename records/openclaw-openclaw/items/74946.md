## Summary

Keep open. Current main still lacks runtime Moonshot provider alias normalization, and this PR only adds a manifest catalog alias plus catalog-planner coverage, so the reported direct `moonshotai/kimi-k2.6` runtime path can still resolve under an unmatched provider id.

## What This Changes

The PR adds a `modelCatalog.aliases.moonshotai -> moonshot` entry to the Moonshot plugin manifest and adds a manifest-planner unit test for alias-filtered `moonshotai/kimi-k2.6` catalog rows.

## Best Possible Solution

Add `moonshotai` and likely `moonshot-ai` as aliases in the runtime provider-id normalization path while preserving `moonshot` as the canonical plugin/provider/auth/config id. Keep the manifest catalog alias only as complementary model-listing support, and add focused direct-resolution tests plus the required changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: `moonshotai/kimi-k2.6` parses to provider `moonshotai` on current main, and `resolveModelWithRegistry` keeps that provider for registry/config/runtime lookup, matching the linked issue's `Unknown model` failure.

## Solution Assessment

No. This PR is not the best fix because it covers catalog planning only; the narrow maintainable solution is runtime provider alias normalization with regression coverage for parsing/direct resolution, optionally accompanied by the manifest alias for model-picker rows.
