## Summary

Keep open. ## What This Changes

The PR adds a `moonshotai -> moonshot` `modelCatalog.aliases` entry to the Moonshot plugin manifest and a manifest-planner unit test for provider-filtered alias catalog rows.

## Best Possible Solution

Add a runtime provider-id alias so `moonshotai/<model>` and preferably `moonshot-ai/<model>` canonicalize to the existing `moonshot` provider while keeping `moonshot` as the canonical plugin, auth, config, and provider id. The manifest catalog alias can stay only if maintainers also want alias-filtered model-list rows.

Yes. The linked issue gives concrete config and log output, and the current source confirms the static path: `moonshotai/kimi-k2.6` can remain a provider id that is not registered as `moonshot`, so registry lookup misses the direct Moonshot model.

## Solution Assessment

No. A `modelCatalog.aliases` entry is useful for catalog planning, but it is not the narrow runtime fix for the reported `Unknown model` path; provider-id normalization plus model-resolution regression coverage is the maintainable fix.
