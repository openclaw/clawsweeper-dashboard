## Summary

Keep open: current main still lacks the runtime Moonshot provider alias, and this PR’s manifest-only alias addresses catalog planning but not the direct model resolution failure reported in the linked bug.

## What This Changes

The PR adds a Moonshot `modelCatalog.aliases` entry mapping `moonshotai` to `moonshot` and a manifest-planner test for alias-filtered catalog rows.

## Best Possible Solution

Keep `moonshot` as the canonical provider/plugin/auth/config id, but accept `moonshotai` and `moonshot-ai` in runtime provider-id normalization with focused direct-resolution tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The linked bug gives concrete configuration steps and logs, and current main source shows `moonshotai/kimi-k2.6` still parses and resolves as provider `moonshotai` rather than canonical `moonshot`.

## Solution Assessment

No. Catalog aliases are useful for provider-filtered catalog rows, but the reported failure is runtime direct model resolution, so the narrow maintainable fix belongs in provider-id normalization plus regression coverage.
