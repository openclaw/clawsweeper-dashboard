## Summary

Keep open. Current main still lacks the requested post-compaction validator, and this stacked draft PR remains useful but not merge-ready because the public config schema is stale and transcript-derived failure signatures enter compaction instructions without untrusted-data wrapping.

## What This Changes

This PR adds a stacked loop-aware compaction guard series culminating in a pure post-compaction validator, plus config scaffolding, transcript-tail detection, guard scoring, instruction augmentation, and focused tests.

## Best Possible Solution

Land the loop-aware compaction stack from the lower PRs upward, regenerate the public schema artifacts, wrap transcript-derived guard evidence as untrusted data, and make an explicit maintainer call on whether validation feeds checkpoint restore or remains diagnostic-only.

## Reproduction Assessment

Not applicable as a user-facing bug reproduction; this is a feature PR. The current-main gap is reproducible by source search and by inspecting the existing pre-persistence quality audit, post-compaction side effects, and checkpoint paths.

## Solution Assessment

No, not in the current form. The pure validator direction is plausible, but the stacked branch needs a rebase/stack decision, generated schema output, and prompt-data hardening before it is the narrowest maintainable solution.

Full review comments:

- **[P2] Regenerate the public config schema:** `src/config/zod-schema.agent-defaults.ts:97-106`
  - body: The PR adds `agents.defaults.compaction.guard` to the zod schema and help/label metadata, but it does not update `src/config/schema.base.generated.ts`. That generated module is imported by `src/config/schema.ts` for public config schema responses and schema checks, so consumers and gates stay stale until the generated output is committed.
  - confidence: 0.91
- **[P2] Wrap guard failure signatures as untrusted data:** `src/agents/pi-extensions/compaction-instructions.ts:91-94`
  - body: `failure.signature` comes from transcript/tool error text and is appended directly into the guard instruction block. Tool failures can contain attacker-controlled strings, and this repo already has untrusted prompt-data wrappers for instruction-adjacent runtime text, so these signatures should be escaped or wrapped before insertion.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Needs attention: the diff adds no dependency, CI, secret, or code-execution surface, but it does introduce a prompt-injection-sensitive instruction path for transcript-derived failure text.

Concerns:

  - body: The guard instruction builder interpolates `failure.signature` from tool/transcript error text into an instruction block. A malicious or compromised tool output could shape compaction instructions unless this value is escaped or wrapped as untrusted data.
