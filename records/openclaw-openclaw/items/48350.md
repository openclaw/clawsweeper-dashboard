## Summary

Keep this PR open: current main does not contain the requested post-compaction validator, and the related loop-aware compaction guard stack is still open. The branch is not merge-ready because it is draft/stale against the current compaction path, lacks real behavior proof, and has concrete validator/prompt-safety issues in the proposed code.

## What This Changes

The branch adds a stacked loop-aware compaction guard surface, including guard config, transcript-tail detection, guard scoring, guarded compaction instructions, and a pure post-compaction validator with tests.

## Best Possible Solution

Rebase the stack onto the current `pi-hooks` compaction path, fix the validator and prompt-safety issues, add focused regression coverage, and then review the validator alongside the open guard/recommend-reset design thread.

## Reproduction Assessment

Yes for the review findings: source inspection shows the collapse-marker early return and the untrusted signature interpolation paths clearly. The feature request itself is not a bug reproduction; current main simply lacks this validator.

## Solution Assessment

No. A pure validator is a reasonable layer, but this branch should not let collapse wording mask raw repeated failures or place tool-error text inside trusted compaction instructions before it is considered mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Check raw failures before accepting collapse markers:** `src/agents/post-compaction-validator.ts:249-250`
  - body: Because this returns `false` as soon as any collapse marker appears, a summary can keep repeated raw error lines while adding wording like "repeated tool failures were summarized" and avoid `failure-pattern-not-collapsed`. Count raw failure lines/signatures before treating collapse phrasing as healthy.
  - confidence: 0.86
- **[P2] Keep tool-error text out of trusted instructions:** `src/agents/pi-extensions/compaction-instructions.ts:95-96`
  - body: `failure.signature` is derived from transcript/tool error text, but the hook writes it directly into the guard instruction block. A malicious or noisy tool error can be promoted from untrusted runtime data into compaction instructions; quote/wrap it as untrusted data or omit the raw signature.
  - confidence: 0.74
