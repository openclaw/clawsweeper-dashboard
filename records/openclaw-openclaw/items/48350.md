## Summary

Keep open. This is a draft stacked feature PR, and source review still shows blocking current-main rebase, Unicode retention, prompt-boundary, docs-contract, recovery-policy, and real behavior proof gaps.

## What This Changes

The branch adds a loop-aware compaction guard stack ending in a pure post-compaction validator, with config/schema/docs, tail detection/scoring, guarded compaction instructions, and tests.

## Best Possible Solution

Rebase or consolidate the guard stack onto current pi-hooks, reuse the existing Unicode-aware summary-quality machinery, separate trusted guard policy from bounded untrusted transcript data, align config docs with actual behavior, and settle checkpoint-backed recovery semantics before merge.

Not applicable as a current-main bug reproduction because this is a new reliability feature PR.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the patch needs current-main rebase, Unicode retention fixes, prompt-boundary hardening, docs alignment, real behavior proof, and maintainer judgment on recovery semantics before it is the narrow maintainable solution.

Full review comments:

- **[P2] Move guard logic onto pi-hooks:** `src/agents/pi-extensions/compaction-safeguard.ts:9`
  - body: Current main loads the active compaction safeguard from `src/agents/pi-hooks`, not `src/agents/pi-extensions`. Adding the scorer hook under the old path means the new behavior will miss the runtime path or require conflict-heavy rework; rebase the guard logic onto the active hook files.
  - confidence: 0.9
- **[P2] Preserve non-Latin goals in the validator:** `src/agents/post-compaction-validator.ts:180-184`
  - body: `normalizeText` strips every non-ASCII letter and `isReflectedInSummary` treats an empty normalized source as already reflected. A latest goal written in Chinese, Japanese, or another non-Latin script can be omitted from the summary while validation still passes; use the existing Unicode-aware overlap machinery and add a regression test.
  - confidence: 0.91
- **[P2] Wrap failure signatures as untrusted prompt data:** `src/agents/pi-extensions/compaction-instructions.ts:96`
  - body: `failure.signature` comes from recent transcript or tool error text and is interpolated directly into instruction-like compaction guidance. Tool errors can contain attacker-controlled text, so wrap or escape these signatures with the existing untrusted prompt-data helper before placing them near instructions.
  - confidence: 0.84
- **[P2] Keep guard directives out of the custom block:** `src/agents/pi-extensions/compaction-safeguard.ts:872-893`
  - body: The guard augmentation is passed as `customInstructions` and then sent through `buildCompactionStructureInstructions`, where custom text is wrapped as untrusted prompt data. That demotes fixed guard policy to data, so risky compactions may not reliably follow it; add a trusted guard-instruction seam and keep transcript signal details bounded as data.
  - confidence: 0.78
- **[P2] Align guard docs with runtime behavior:** `docs/gateway/configuration-reference.md:1036-1039`
  - body: The config reference says `guard` is reserved and has no runtime effect, but this stacked branch wires `guard.enabled` into safeguard compaction behavior. Operators would get the wrong contract if these docs ship unchanged, so update the docs/help copy or keep runtime behavior out of this PR.
  - confidence: 0.9
