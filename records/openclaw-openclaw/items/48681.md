## Summary

Keep open. ## What This Changes

The PR adds `agents.list[].multipleWorkspaces` and uses a symlink-backed composite workspace directory for agents configured with multiple physical workspaces.

## Best Possible Solution

Pick one explicit multi-root filesystem model, then wire it through agent config/schema, workspace resolution, all workspace setup callers, path guards, sandbox/media/tool handling, docs, and real upgrade proof before merging.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows the new `multipleWorkspaces` setup depends on callers passing the list, while current main has separate agent-command, cron, and sandbox workspace setup paths. For the central request, reproduction is not applicable because this is a new capability rather than a broken documented behavior.

## Solution Assessment

No: this is not yet the best way to solve the issue because the symlink-composite model competes with the active included-root/path-policy design and is not wired through all runtime paths. A safer solution is a single maintainer-approved multi-root contract with explicit path-guard, sandbox, docs, tests, and proof coverage.
