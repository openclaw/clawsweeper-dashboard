## Summary

Keep open: current main still has a single-workspace contract, and this PR is a meaningful new feature, but it is not merge-ready because the symlink-composite design needs maintainer product/security direction, complete runtime propagation, and real behavior proof.

## What This Changes

The PR adds `agents.list[].multipleWorkspaces`, resolves those agents to a generated composite workspace directory, and populates that directory with symlinks to configured workspace paths.

## Best Possible Solution

Choose one maintainer-approved multi-root workspace model, then wire it through config/schema, every runtime setup caller, path guards, sandbox/media/tool handling, docs, tests, and real multi-repo proof before merging.

## Solution Assessment

No, not in its current merge shape. The symlink-composite approach may be useful, but the safer path is one maintainer-approved multi-root contract with complete runtime propagation, security-boundary coverage, docs, tests, and real behavior proof.

Full review comments:

- **[P1] Pass multi-workspaces through every setup flow:** `src/auto-reply/reply/get-reply.ts:174`
  - body: `resolveAgentWorkspaceDir()` makes every agent with `multipleWorkspaces` run from the composite directory, but this patch only passes the workspace list in the reply flow. Agent command, isolated cron, and sandbox setup still call `ensureAgentWorkspace` without `multipleWorkspaces`, so those executions can start in an empty composite workspace and lose access to the configured repos.
  - confidence: 0.9
- **[P2] Reject root paths before naming symlinks:** `src/agents/workspace.ts:733`
  - body: When a configured workspace is a filesystem root such as `/` or a Windows drive root, `path.basename(ws)` is empty and this branch uses `""` as the link name. That makes `path.join(compositeDir, linkName)` address the composite directory itself, so setup cannot expose that workspace as a child link and only logs/skips after trying to replace the directory entry.
  - confidence: 0.82
