## Summary

Keep open. Current main still lacks this multi-workspace config surface, but this branch is not mergeable as-is because PR-head source inspection shows incomplete workspace setup propagation, an unhandled root-path edge case, missing real behavior proof, and an unsettled filesystem-root product/security contract that overlaps newer work.

## What This Changes

The PR adds `agents.list[].multipleWorkspaces` and a symlink-backed composite workspace directory so one agent can operate across multiple physical repositories.

## Best Possible Solution

Define one first-class multi-root workspace/access contract that consistently covers reply, CLI, cron, reverse lookup, docs, sandbox/path-guard behavior, and the relationship between composite workspaces, `includedWorkDirs`, and PathGuard-style filesystem policy.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows composite workspace resolution without multi-workspace propagation in agent-command and isolated cron paths, and the root-path empty-basename case is visible in `ensureCompositeWorkspace`. For the central request, reproduction as a bug is not applicable because this is a new capability rather than broken existing behavior.

## Solution Assessment

No. The symlink-composite idea may be viable, but this branch is incomplete across execution paths and should not become the chosen contract until maintainers settle the broader multi-root/filesystem policy direction and the contributor supplies real behavior proof.
