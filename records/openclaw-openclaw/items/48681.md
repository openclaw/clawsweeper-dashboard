## Summary

Keep open. Current main still has no multi-workspace agent implementation, so the PR is not obsolete, but the proposed branch is not mergeable: the composite workspace can be selected without being populated in non-reply execution paths, and root workspace entries can derive an empty symlink name. Earlier review comments appear to have helped fix several collision and stale-link bugs, but the latest head still has blocking correctness gaps.

## What This Changes

The PR adds an optional `multipleWorkspaces` agent config field, a composite symlink workspace resolver/populator, reverse-lookup and workspace-directory listing updates, and reply-path forwarding for the configured workspace list.

## Best Possible Solution

Keep the PR as an implementation candidate, but require it to centralize or consistently pass multi-workspace setup through every execution path, reject or safely name invalid/root workspace entries, and add focused tests plus docs/changelog coverage for config, sandbox, backup, and reverse-lookup behavior.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure `multipleWorkspaces`, then run CLI or isolated cron agent execution on a fresh setup; those paths can resolve to the composite directory without passing the workspace list that populates it.

## Solution Assessment

No. The symlink-composite approach may be acceptable, but this patch is not the best mergeable form until setup is centralized or all setup callers receive the workspace list, and invalid link names are handled deliberately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Populate composite workspaces in every setup path:** `src/agents/agent-scope.ts:295-296`
  - body: The new resolver sends every multi-workspace agent to `workspace-composite-<id>`, but only the reply path passes `multipleWorkspaces` into `ensureAgentWorkspace`. CLI and isolated cron runs still call setup without the list, so fresh runs create or use an empty composite workspace and cannot see the configured repos.
  - confidence: 0.9
- **[P2] Reject root workspace paths before deriving link names:** `src/agents/workspace.ts:732`
  - body: For a configured filesystem root, `path.basename()` is empty and this branch assigns that empty string as the link name. `path.join(compositeDir, "")` points at the composite root, so setup cannot create a usable repo link and may target the composite directory itself.
  - confidence: 0.84
