## Summary

Keep open: current main still has the single-workspace contract, and this PR is not merge-ready because PR-head source still leaves composite workspaces unpopulated in CLI/cron flows, mishandles filesystem-root workspace entries, lacks inspectable real behavior proof, and changes the agent filesystem boundary in an area with active related design work.

## What This Changes

This PR adds `agents.list[].multipleWorkspaces`, resolves those agents to a symlink-backed composite workspace, includes both composite and physical workspace dirs in discovery, and extends the agent config type/schema.

## Best Possible Solution

Define one first-class multi-root workspace/access contract that consistently covers reply, CLI, cron, reverse lookup, docs, sandbox/path-guard behavior, and the relationship between composite workspaces, `includedWorkDirs`, and filesystem policy before merging an implementation.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows composite workspace resolution without multi-workspace propagation in agent-command and isolated cron paths, and the root-path empty-basename case is visible in `ensureCompositeWorkspace`. For the central request, reproduction as a bug is not applicable because this is a new capability rather than broken existing behavior.

## Solution Assessment

No. The symlink-composite approach may be viable, but this branch is incomplete across execution paths and should not become the chosen contract until maintainers settle the broader multi-root/filesystem policy direction and the contributor supplies real behavior proof.
