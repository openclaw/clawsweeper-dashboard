## Summary

Keep open. Current main still does not implement multi-workspace agent config, so the PR is not obsolete, but the proposed branch still has blocking correctness issues around workspace setup parity and composite link-name edge cases.

## What This Changes

The PR adds a `multipleWorkspaces` agent config/schema field, resolves those agents to a composite symlink workspace, updates reverse lookup/workspace-dir listing, and forwards the workspace list through the reply path.

## Best Possible Solution

Keep this as an implementation candidate, but require the branch to populate composite workspaces in every execution path it redirects, reject or safely name invalid/root workspace entries, and add focused tests plus docs for config, sandbox, backup, and reverse-lookup behavior before merge.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure `multipleWorkspaces`, then exercise CLI or cron agent execution; those paths resolve the agent to the composite directory but do not pass the workspace list into setup, so the composite is not populated.

## Solution Assessment

No. The symlink-composite design may be acceptable, but this patch is not the best mergeable form until workspace setup is centralized or every setup caller receives the multi-workspace list, and invalid link names are handled deliberately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Populate composite workspaces outside the reply path:** `src/agents/agent-scope.ts:295`
  - body: The new resolver sends every multi-workspace agent to `workspace-composite-<id>`, but only `getReplyFromConfig` passes `multipleWorkspaces` into `ensureAgentWorkspace`. CLI and cron paths still call setup with only `dir`, so they create or use an empty composite workspace and cannot see the configured repos.
  - confidence: 0.9
- **[P2] Reject empty composite link names for filesystem roots:** `src/agents/workspace.ts:733`
  - body: A configured root workspace such as `/` or `C:\` has an empty basename, so the non-dedup branch sets `linkName` to `""`. `path.join(compositeDir, linkName)` then targets the composite directory itself, so setup skips the root workspace or tries to replace the composite root instead of creating a usable link.
  - confidence: 0.84
