## Summary

Keep open: current main still has only the single-workspace agent model, and this feature PR remains useful but is not mergeable until the composite workspace setup works across all agent execution paths and the public config surface is documented/changeloged.

## What This Changes

This PR adds an `agents.list[].multipleWorkspaces` config field and resolves those agents to a symlink-backed composite workspace directory.

## Best Possible Solution

If maintainers want this core feature, ship a rebased implementation that treats multi-workspace as a first-class agent config: populate composite links in every setup path, reject invalid roots, document the config, add a changelog entry, and cover reply, CLI, cron, reverse lookup, and symlink-name edge cases.

## Reproduction Assessment

Yes. Static source inspection of the PR gives a high-confidence path: configure `multipleWorkspaces`, then run CLI or isolated cron on a fresh setup; those callers resolve to the composite directory but do not pass the workspace list needed to populate links.

## Solution Assessment

No. The symlink-composite approach may be viable, but this patch is not the best mergeable form until setup is centralized or propagated, root workspace entries are handled deliberately, and docs/changelog/tests are added.
