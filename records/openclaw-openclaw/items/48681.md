## Summary

Keep open: current main does not implement multi-workspace agents, but this branch still cannot merge because composite setup is incomplete across execution paths, root path handling is unsafe, real behavior proof is missing, and the new core config/filesystem contract needs maintainer product review.

## What This Changes

The PR adds `agents.list[].multipleWorkspaces` and creates a symlink-backed composite workspace directory, with config/schema, workspace discovery, reverse lookup, and reply setup changes.

## Best Possible Solution

If maintainers want this capability, define one first-class multi-root workspace contract that consistently covers reply, CLI, cron, reverse lookup, docs, validation, sandbox/path-guard behavior, and its relationship to `includedWorkDirs` or allowed-roots designs.

## Reproduction Assessment

Yes for the PR defects: static PR-head inspection shows composite workspace resolution without multi-workspace propagation in CLI and cron paths, and the root-path empty-basename case is visible in source. For the central request, reproduction as a bug is not applicable because this is a new capability rather than broken existing behavior.

## Solution Assessment

No. The symlink-composite direction may be viable, but this branch is incomplete across execution paths and needs maintainer approval for the new config/filesystem contract before it can be considered the best solution.

  - confidence: 0.86
