## Summary

Keep open: current main does not implement multi-workspace agents, and the PR still has source-level blockers plus missing real behavior proof before it can be considered for merge.

## What This Changes

The PR adds `agents.list[].multipleWorkspaces` and routes those agents through a symlink-backed composite workspace directory with reverse workspace lookup support.

## Best Possible Solution

If maintainers want this feature, implement it as a first-class agent config across every workspace setup path, reject invalid root entries, align schema/help/docs/changelog, and prove reply, CLI, cron, reverse lookup, and symlink edge cases.

## Reproduction Assessment

Yes. Static inspection of the PR head gives a high-confidence path: configure `multipleWorkspaces`, then run agent CLI or isolated cron on a fresh setup; those callers resolve the composite directory but do not pass the workspace list needed to populate symlinks.

## Solution Assessment

No. The symlink-composite approach may be viable, but this branch is not the best mergeable form until setup propagation, invalid-root handling, config contract docs, and real behavior proof are addressed.

  - confidence: 0.92

  - confidence: 0.86
