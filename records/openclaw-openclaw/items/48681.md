## Summary

Keep open: current main still has only the single-workspace contract, while this PR is a new multi-root feature with open correctness findings, missing real behavior proof, merge conflicts, and an unresolved filesystem/security product decision. The related includedWorkDirs PR is relevant and may become the better contract, but it is not merged or clearly maintainer-designated as the canonical replacement yet.

## What This Changes

The PR adds `agents.list[].multipleWorkspaces`, resolves those agents to a symlink-backed composite workspace, threads that through reply workspace setup and workspace discovery, and extends the agent config type/schema.

## Best Possible Solution

Choose one explicit multi-root workspace/access model for core, preferably one that covers config schema, docs, sandbox/path-guard behavior, reply, CLI, cron, reverse lookup, and upgrade safety together before any implementation lands.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows agent-command and isolated cron omit `multipleWorkspaces` when ensuring the composite workspace, and the root-path empty-basename case is visible in `ensureCompositeWorkspace`. For the central request, reproduction as a bug is not applicable because this is a new capability.

## Solution Assessment

No. The symlink-composite approach may be viable, but this branch is incomplete across execution paths and overlaps with the active includedWorkDirs direction, so maintainers should settle the filesystem contract first.
