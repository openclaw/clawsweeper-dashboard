## Summary

Keep open: current main still lacks this multi-workspace capability, and the PR is not merge-ready because source inspection shows unpopulated composite workspaces outside the reply path plus an invalid root-path edge case; the new core config and symlink workspace model also need maintainer product review and real behavior proof.

## What This Changes

The PR adds an `agents.list[].multipleWorkspaces` config key and routes those agents through a symlink-backed composite workspace directory with reverse workspace lookup support.

## Best Possible Solution

If maintainers want this core feature, implement it as a first-class agent config across every workspace setup path, reject invalid root entries, align schema/help/docs/changelog, and cover reply, CLI, cron, reverse lookup, and symlink-name edge cases.

## Reproduction Assessment

Yes. Static source inspection of the PR head gives a high-confidence path: configure `multipleWorkspaces`, then run CLI or isolated cron on a fresh setup; those callers resolve to the composite directory but do not pass the workspace list needed to populate links.

## Solution Assessment

No. The symlink-composite approach may be viable, but this branch is not the best mergeable form until setup propagation, invalid-root handling, config contract docs, and real behavior proof are addressed.
