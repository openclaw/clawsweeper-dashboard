## Summary

Keep open for maintainer review: current main still has a single-workspace contract, the competing included-root design is open but not an explicit superseding decision, and this branch still has correctness, proof, conflict, and security-boundary blockers.

## What This Changes

The branch adds `agents.list[].multipleWorkspaces`, resolves such agents to a generated composite workspace directory, and populates that directory with symlinks to the configured workspace paths.

## Best Possible Solution

Pick one explicit multi-root workspace model, then wire it through config/schema, every runtime setup caller, path guards, sandbox/media/tool handling, docs, tests, upgrade behavior, and real multi-repo proof before merging.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows relative `multipleWorkspaces` entries are symlinked as relative targets and several runtime setup paths omit the new parameter. For the central request, reproduction is not applicable because this is a new feature/config capability.

## Solution Assessment

No: the current symlink-composite implementation is not yet the best way to solve this because it competes with the active included-root design and is not threaded through all runtime and boundary surfaces. The safer direction is a single maintainer-approved multi-root contract with explicit security and upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve multi-workspace entries before symlinking:** `src/agents/agent-scope.ts:284`
  - body: `resolveUserPath()` only expands home-relative paths, so a config entry like `"./repo-a"` is passed to `fs.symlink` as a relative target. Because symlink targets are interpreted relative to the composite directory, this can create a self-referential or wrong link and leave the configured workspace unusable.
  - confidence: 0.9
- **[P1] Populate composites for every workspace setup path:** `src/auto-reply/reply/get-reply.ts:174`
  - body: This only passes `multipleWorkspaces` in the reply flow, but `resolveAgentWorkspaceDir()` makes all multi-workspace agents use the composite directory. Agent command, isolated cron, and sandbox setup still call `ensureAgentWorkspace` without the new list, so those runs can start in an unpopulated composite workspace and lose access to the configured repos.
  - confidence: 0.9
- **[P2] Reject root workspaces before deriving link names:** `src/agents/workspace.ts:733`
  - body: When a configured workspace is a filesystem root, `path.basename(ws)` is empty and this branch uses the empty string as the link name. `path.join(compositeDir, "")` addresses the composite directory itself, so setup cannot expose that workspace as a child link and only logs/skips after trying to replace the directory entry.
  - confidence: 0.82
