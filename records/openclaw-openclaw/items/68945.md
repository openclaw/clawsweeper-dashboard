## Summary

Keep open: current main still omits Telegram humanDelay wiring, and the open linked implementation PR has not merged, so the reported regression remains source-reproducible.

## What This Changes

Telegram block streaming should pass route-scoped `humanDelay` into the shared reply dispatcher so configured pauses apply between streamed block replies.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/69022 after rebase and redacted real Telegram proof, or merge an equivalent Telegram-only patch that forwards `resolveHumanDelayConfig(cfg, route.agentId)` into `dispatcherOptions.humanDelay` with focused regression coverage.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: Telegram block streaming on current main never supplies `dispatcherOptions.humanDelay`, while the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Yes. The narrow Telegram-only wiring is the best fix because it reuses the existing resolver and shared dispatcher contract instead of adding Telegram-specific sleep logic or a new config surface.
