## Summary

This issue should stay open: current main and the latest release still omit Telegram humanDelay wiring, and the linked implementation pull request is open but not merged.

## What This Changes

The issue asks Telegram block-streamed replies to honor existing agent humanDelay settings between coalesced block replies.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/69022 or an equivalent Telegram-boundary patch that forwards resolveHumanDelayConfig(cfg, route.agentId) into dispatcherOptions.humanDelay with focused Telegram regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: Telegram block streaming never supplies dispatcherOptions.humanDelay on current main, while the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Yes, the narrow Telegram-only wiring is the best fix because it reuses the existing resolver and shared dispatcher contract instead of adding Telegram-specific sleep logic or a new config surface.
