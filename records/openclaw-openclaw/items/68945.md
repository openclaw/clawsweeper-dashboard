## What This Changes

The issue asks for Telegram streamed block replies to honor configured `agents.defaults.humanDelay` and per-agent `humanDelay` pauses between coalesced blocks.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/69022 after conflict resolution and real Telegram proof, or merge an equivalent Telegram-only patch that forwards `resolveHumanDelayConfig(cfg, route.agentId)` into `dispatcherOptions.humanDelay` with focused regression coverage.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main never supplies `dispatcherOptions.humanDelay` in Telegram block streaming, while the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Yes. The narrow Telegram wiring in the linked PR is the best fix because it reuses the existing resolver and shared dispatcher contract instead of adding Telegram-specific delay logic or a new config surface.
