## Summary

Keep open: current main still omits the Telegram `humanDelay` wiring, and the linked closing PR is open rather than merged. The report remains a narrow, source-reproducible Telegram parity bug.

## What This Changes

The issue asks Telegram streamed block replies to honor `agents.defaults.humanDelay` and per-agent overrides by passing the resolved delay config into the shared block reply dispatcher.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/69022 or an equivalent Telegram-only patch after focused regression coverage and redacted live Telegram proof show configured pauses between streamed block replies.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current main shows Telegram never supplies `dispatcherOptions.humanDelay`, while the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Is this the best way to solve the issue? Yes. Reusing the existing resolver through the Telegram local runtime barrel and passing it to the shared dispatcher is the narrow maintainable fix; adding Telegram-specific sleep logic or a new config surface would be worse.
