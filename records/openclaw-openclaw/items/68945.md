## Summary

Current main still reproduces the Telegram humanDelay gap, and open PR #69022 is already linked as the implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests wiring resolved per-default/per-agent humanDelay into Telegram block streaming so coalesced block replies are paced like other channels.

## Best Possible Solution

Land or replace #69022 with the same narrow Telegram plugin-boundary fix: expose/import resolveHumanDelayConfig through the existing runtime seam, pass it as dispatcherOptions.humanDelay, and keep focused Telegram regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main’s Telegram dispatcher never supplies dispatcherOptions.humanDelay, while the shared dispatcher only delays later block replies when that option is present.

## Solution Assessment

Yes. The narrow Telegram-only wiring is the best fix because it matches established channel behavior and avoids changing the shared dispatcher contract.
