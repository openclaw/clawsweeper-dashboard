## What This Changes

The issue reports that Telegram block-streaming replies do not receive the configured humanDelay between coalesced messages.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/69022 after rebase and real Telegram proof, or merge an equivalent Telegram-only patch that forwards resolveHumanDelayConfig(cfg, route.agentId) into dispatcherOptions.humanDelay with focused regression coverage.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: Telegram block streaming on current main never supplies dispatcherOptions.humanDelay, while the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Yes. The narrow Telegram-only wiring is the best fix because it reuses the existing resolver and shared dispatcher contract instead of adding Telegram-specific sleep logic or a new config surface.
