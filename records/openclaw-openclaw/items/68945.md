## Summary

This issue should stay open. Current main still omits Telegram humanDelay wiring in the inbound block-streaming dispatcher, while the shared dispatcher only applies the pause when callers pass `dispatcherOptions.humanDelay`; open PR #69022 is already the explicit implementation candidate and should be reviewed or replaced before closing this issue.

## What This Changes

Telegram block streaming should pass the resolved per-agent/default humanDelay configuration into the shared block reply dispatcher so coalesced Telegram block replies are paced like other messaging channels.

## Best Possible Solution

Keep the fix in the Telegram plugin boundary: expose or import `resolveHumanDelayConfig` through the existing SDK/runtime seam, pass `resolveHumanDelayConfig(cfg, route.agentId)` as `dispatcherOptions.humanDelay` for Telegram inbound block streaming, and add focused Telegram regression coverage before closing this issue.

## Reproduction Assessment

Yes. The high-confidence source reproduction is that Telegram block streaming on current main never supplies `dispatcherOptions.humanDelay`, while the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Yes. The narrow Telegram-only wiring described by the issue and PR #69022 matches the established channel pattern and avoids changing the shared dispatcher contract; this issue should remain open until that PR, or an equivalent fix, lands.
