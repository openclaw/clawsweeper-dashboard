## Summary

Keep open: current main still omits the Telegram `humanDelay` wiring, and the open closing PR at https://github.com/openclaw/openclaw/pull/69022 already owns the narrow implementation path. This issue should close only after that PR or an equivalent patch lands.

## What This Changes

Apply configured `agents.defaults.humanDelay` and per-agent `humanDelay` overrides to Telegram streamed block replies by forwarding the resolved config into the block reply dispatcher.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/69022 or an equivalent Telegram-only patch after focused regression coverage and redacted live Telegram proof demonstrates configured pauses between streamed block replies.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: Telegram never supplies `dispatcherOptions.humanDelay` on current main, while the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Yes. Passing `resolveHumanDelayConfig(cfg, route.agentId)` into Telegram dispatcher options reuses the existing resolver and shared dispatcher contract, which is narrower than Telegram-specific delay logic or a new config surface.
