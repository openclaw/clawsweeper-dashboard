## Summary

Keep open: current main and the latest release still omit the Telegram humanDelay wiring, and the linked closing PR is open rather than merged, so the reported regression remains source-reproducible.

## What This Changes

Telegram block-streamed replies should honor agents.defaults.humanDelay and per-agent humanDelay overrides by forwarding the resolved config into the shared reply dispatcher.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/69022 after rebase and redacted real Telegram proof, or merge an equivalent Telegram-only patch with focused regression coverage.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current Telegram block streaming never supplies dispatcherOptions.humanDelay, while the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Yes. The narrow maintainable fix is to forward resolveHumanDelayConfig(cfg, route.agentId) into Telegram dispatcherOptions.humanDelay through the local runtime barrel, reusing the existing dispatcher contract instead of adding Telegram-specific delay logic.
