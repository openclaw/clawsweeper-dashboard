## Summary

Keep open. Current main still rejects the reported Slack message-action shape before any Slack delivery, and the adjacent two-pass outbound payload path is not wired into `send` message actions. The related open Slack file PR is inbound media hydration work and does not claim to change outbound Slack sends.

## What This Changes

The issue requests that Slack message action `send` support file attachments together with interactive or presentation blocks instead of throwing.

## Best Possible Solution

Fix this inside the Slack plugin message-action path by using the existing media-sequence/final-blocks behavior, or an equivalent Slack-owned helper, for `send` with media plus presentation/interactive blocks while preserving a single logical action for result and transcript semantics.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call Slack message action `send` with `to`, `media`, and `interactive` or `presentation`; `handleSlackMessageAction` renders blocks and throws before invoking Slack delivery, and the lower runtime has the same guard.

## Solution Assessment

Unclear as currently proposed. The two-pass direction matches existing Slack outbound behavior, but the maintainable fix should share or adapt that path rather than only deleting the guard in one layer.
