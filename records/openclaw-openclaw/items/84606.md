## Summary

Keep open: the PR fixes the main memory-wiki palace summary problem and includes real terminal proof, but the legacy-payload fallback still lets filtered cluster counts be rendered as a full vault breakdown when authoritative `pageCounts` is absent.

## What This Changes

The branch adds vault-level `totalPages` and `pageCounts` to `wiki.palace`, normalizes them in Control UI, and updates the palace view/tests to show vault totals plus section-local metrics.

## Best Possible Solution

Keep the backend vault totals and clearer palace rendering, but make the full vault breakdown render only from authoritative `pageCounts` or clearly label legacy fallbacks as visible/estimated section counts.

## Reproduction Assessment

Yes. Source inspection of current main shows the backend returns only filtered `totalItems` and the UI headline uses selected cluster counts, matching the linked mixed-vault report.

## Solution Assessment

Not yet. The main fix shape is right, but the compatibility fallback should not present cluster-derived counts as authoritative full-vault counts.

- **[P2] Avoid labeling cluster-derived counts as full vault totals:** `ui/src/ui/controllers/dreaming.ts:643-649`
