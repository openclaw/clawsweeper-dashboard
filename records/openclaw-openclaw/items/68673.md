## Summary

Keep open: the PR targets a real `update_plan` loop-detection gap, but the current patch over-collapses material plan text into presence bits, which can make legitimate plan rewrites look like identical no-progress outcomes. It also needs the required changelog entry for a user-visible fix before merge.

## What This Changes

The PR special-cases `update_plan` loop detection by normalizing `update_plan` arguments and outcome details before hashing, with regression tests for wording churn and structural status progress.

## Best Possible Solution

Keep the `update_plan`-specific loop signature, but include a stable content signal for each step so cosmetic churn can collapse without making unrelated plan rewrites collide; add regression coverage for both wording-only churn and material step changes.

## Reproduction Assessment

Yes. A source-level reproduction is clear: record 30 `update_plan` calls whose non-empty step text changes while statuses and plan length stay fixed, then call `detectToolCallLoop`; the PR normalization makes those records share the same args and result shape and can trip the global breaker.

## Solution Assessment

No. Normalizing `update_plan` is the right narrow area, but reducing steps to `status` and `hasStep` is too coarse; a safer fix should hash or canonicalize step content while still ignoring purely cosmetic explanation or echo churn.
