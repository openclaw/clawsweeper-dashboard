## Summary

Keep open. The PR addresses a real no-progress loop-detection gap for `update_plan`, and earlier review feedback about outcome normalization was addressed, but the current normalization over-collapses materially different plan steps into the same loop signature.

## What This Changes

The PR normalizes `update_plan` arguments and outcome details before loop hashing, and adds regression tests for wording-only plan churn and status-change progress.

## Best Possible Solution

Keep a special `update_plan` loop signature, but preserve a content signal for material step changes while ignoring only clearly cosmetic noise such as explanation-only or whitespace-only churn; add regression coverage for both cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to record enough `update_plan` calls whose step text changes materially while the plan length and statuses stay the same; under the PR normalization, those calls share the same args/result shape and can reach the global breaker.

## Solution Assessment

No. Special-casing `update_plan` is a reasonable direction, but this implementation is too broad because unrelated non-empty plan steps with the same status pattern collide.
