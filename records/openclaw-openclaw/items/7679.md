## Summary

Keep open. Current `main` still defaults Telegram DMs to pairing, still documents pairing as the default, and setup still relies on manual numeric `allowFrom` entry rather than token-verified owner ID capture; the existing maintainer discussion correctly keeps this as access-control follow-up work.

## What This Changes

The issue requests changing new Telegram setup to default to `dmPolicy: "allowlist"` and populate `allowFrom` from the owner Telegram ID captured during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that generates a short verification token, captures only the sender who sends that token, writes explicit `dmPolicy: "allowlist"` plus `allowFrom`, updates docs/schema/runtime/tests, and preserves existing explicit pairing-mode configs.

## Reproduction Assessment

Yes. Source inspection on current `main` shows omitted Telegram `dmPolicy` still resolves to pairing, and the current tests still cover pairing-code responses for unauthorized DMs.

## Solution Assessment

Unclear as a default-only change. The maintainable solution needs token-verified owner capture plus explicit `allowFrom`; a schema/runtime default flip alone would fail validation or create an unsafe capture race.
