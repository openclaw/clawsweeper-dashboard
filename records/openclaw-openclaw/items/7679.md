## Summary

Keep open: current main and the latest release still model Telegram DM pairing as the default, and the requested owner-ID capture would change authorization defaults, so it needs maintainer security/product review rather than cleanup closure.

## What This Changes

Request fresh Telegram setup to use `dmPolicy="allowlist"` with the owner Telegram ID captured during onboarding instead of defaulting to `dmPolicy="pairing"`.

## Best Possible Solution

Implement Telegram-owned onboarding hardening: display a short verification token, capture only a matching sender ID, write `dmPolicy="allowlist"` plus `allowFrom` for fresh setup, and preserve existing explicit pairing configs.

## Reproduction Assessment

Yes for the current behavior, though this is a defaults/onboarding feature request rather than a failing-test bug. Source inspection shows omitted Telegram `dmPolicy` still defaults and falls back to pairing, unknown DMs still receive pairing challenges, and setup/docs still use manual numeric `allowFrom` entry.

## Solution Assessment

Yes directionally, but only with token-verified owner capture. A schema/runtime default flip alone is not the best solution because allowlist mode requires a sender ID and first-message capture would introduce an ownership race.
