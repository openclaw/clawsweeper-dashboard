## Summary

Keep open: current `origin/main` and `v2026.6.5` still default Telegram DM access to pairing, setup/docs/tests still model manual numeric `allowFrom`, and the broader open bootstrap/default issues do not replace this token-verified Telegram owner-capture request.

## What This Changes

The issue asks fresh Telegram setup to default to `dmPolicy="allowlist"` with the intended owner’s Telegram ID captured during onboarding.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, capture only the matching numeric sender ID, write `dmPolicy="allowlist"` plus `allowFrom` for fresh owner setup, preserve explicit pairing configs, and update schema/runtime/docs/tests.

## Reproduction Assessment

Yes, at source level. Current main still defaults omitted Telegram `dmPolicy` to pairing, sends pairing challenges to unknown DMs, and documents/tests that behavior; I did not run live Telegram setup in this read-only review.

## Solution Assessment

No for a default-only change. The maintainable fix needs token-verified owner capture during Telegram setup plus compatibility-aware schema/runtime/docs/tests, not merely flipping the default or trusting the first inbound sender.
