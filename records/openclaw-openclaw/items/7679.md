## Summary

Current main and v2026.6.5 still use pairing-oriented Telegram DM defaults and manual numeric allowlist setup; the narrower token-verified owner enrollment remains unimplemented and needs a maintainer product/security decision rather than cleanup.

## What This Changes

Default new Telegram DM setup to an owner allowlist by capturing the intended owner’s numeric ID through a verified onboarding handshake.

## Best Possible Solution

Add a Telegram-owned one-time challenge-token enrollment step for fresh onboarding, persist the matching numeric sender ID with dmPolicy="allowlist", and preserve existing explicit configurations and omitted-policy upgrade behavior unless maintainers approve a migration.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 still use pairing-oriented behavior and manual numeric allowlist entry, so verified automatic owner capture is demonstrably absent. This is a requested new onboarding/security mode rather than a failing established contract.

## Solution Assessment

Yes for the enrollment mechanism: an exact one-time token is safer than first-sender capture and belongs in the Telegram plugin setup surface. The unresolved product decision is whether only fresh onboarding changes or the omitted-policy runtime default also changes.
