## Summary

Keep open: current main still defaults Telegram DMs to pairing, still sends pairing challenges for unknown DM senders, and setup/docs still rely on manual numeric allowFrom entry rather than token-verified owner capture. This remains a security-sensitive Telegram onboarding/defaults decision, not a conservative cleanup close.

## What This Changes

Request fresh Telegram setup/defaults to use dmPolicy="allowlist" with the owner Telegram ID captured during onboarding.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, require the intended owner to send it to the bot, capture that matching numeric Telegram user ID, write dmPolicy="allowlist" plus allowFrom for fresh setup, and preserve explicit existing pairing configs.

## Reproduction Assessment

Yes for the current behavior, though this is a feature/defaults request rather than a failing test bug. Source inspection on current main shows omitted Telegram dmPolicy still defaults and falls back to pairing, unknown DMs still get pairing challenges, and setup/docs still use manual numeric allowFrom entry.

## Solution Assessment

Yes directionally, but only with token-verified owner capture. A schema/runtime default flip alone is not the best solution because allowlist mode requires a sender ID and first-message capture would introduce an ownership race.
