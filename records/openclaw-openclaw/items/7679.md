## Summary

Keep open: current main still defaults Telegram DMs to pairing and uses manual numeric allowFrom setup, while the requested owner-ID capture changes Telegram authorization defaults and needs maintainer security/product review.

## What This Changes

Request fresh Telegram setup to default DMs to allowlist by capturing the owner's numeric Telegram ID during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding hardening: display a short verification token, capture only the matching sender's numeric Telegram ID, write dmPolicy="allowlist" plus allowFrom for fresh setup, and preserve explicit pairing-mode configs.

## Reproduction Assessment

Yes for the current behavior: source inspection shows omitted Telegram dmPolicy still defaults and falls back to pairing, unknown DMs still receive pairing challenges, and setup/docs still use manual numeric allowFrom entry. This is a defaults/onboarding feature request rather than a failing-test bug.

## Solution Assessment

Yes directionally, but only with token-verified owner capture and upgrade-safe handling. A schema/runtime default flip alone is not the best solution because allowlist mode requires a sender ID and first-message capture would introduce an ownership race.
