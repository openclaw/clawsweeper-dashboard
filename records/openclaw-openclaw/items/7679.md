## Summary

Keep open: current main still documents and implements Telegram DM pairing as the default, and setup still relies on manual numeric allowFrom entry rather than token-verified owner ID capture. The request is valid security-default work, but it needs maintainer product/security approval because a default-only flip could break setup or capture the wrong owner.

## What This Changes

The issue requests that fresh Telegram setup default to dmPolicy="allowlist" and add the owner's Telegram user ID to allowFrom during onboarding.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, capture only the matching sender ID, write dmPolicy="allowlist" plus allowFrom for fresh owner setup, and preserve explicit pairing configs.

## Reproduction Assessment

Yes, at source level. Current main still resolves omitted Telegram dmPolicy to pairing, sends pairing challenges to unknown DMs, and documents/tests that behavior; I did not run a live Telegram setup because the source path directly shows the current state.

## Solution Assessment

No for an automated default-only change. The maintainable fix needs token-verified owner capture during Telegram setup plus schema/runtime/docs/tests, not merely flipping the default.
