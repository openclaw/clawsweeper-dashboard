## Summary

Keep open. Current main still defaults Telegram DMs to pairing in schema, runtime, docs, and tests, while setup still relies on manual numeric `allowFrom` entry rather than token-verified owner ID capture.

## What This Changes

The issue requests changing Telegram onboarding/defaults so new setups use `dmPolicy: "allowlist"` with the owner's Telegram ID captured during setup instead of defaulting to pairing.

## Best Possible Solution

Implement this as Telegram-owned onboarding/access-control work: generate a short verification token, capture only the matching sender ID, write explicit `dmPolicy: "allowlist"` plus `allowFrom`, update docs/tests/schema expectations, and preserve existing explicit pairing configs.

## Reproduction Assessment

Yes. Source inspection on current main is enough: omitted Telegram `dmPolicy` still resolves to pairing, and existing tests cover unauthorized DM pairing-code responses.

## Solution Assessment

Unclear as written. The direction is sound only with token-verified owner capture; changing the schema default alone would fail allowlist validation, and first-message capture would be unsafe.
