## Summary

Keep open: current main still defaults and documents Telegram DMs as pairing, and setup only supports manual numeric `allowFrom` entry, while the requested safer default needs token-verified owner capture and maintainer product/security judgment.

## What This Changes

The issue requests changing Telegram onboarding/defaults so new setups use `dmPolicy="allowlist"` with the owner Telegram ID captured during setup.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender ID, writes explicit `dmPolicy="allowlist"` plus `allowFrom`, updates docs/schema/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Yes, for the current behavior: source inspection on current main shows missing Telegram `dmPolicy` resolves to pairing, and tests assert pairing-code replies for unauthorized DMs. This is a safer-default/security hardening request rather than a broken existing contract.

## Solution Assessment

Yes if implemented with token-verified owner capture and explicit non-empty `allowFrom`; no for changing only the default. The current validation and setup flow make a default-only fix unsafe or incomplete.
