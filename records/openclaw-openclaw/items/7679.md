## Summary

Keep open: current main and the latest release still default Telegram DMs to pairing when `dmPolicy` is omitted, setup still uses manual numeric `allowFrom` entry, and no token-verified owner-ID capture exists. Broader bootstrap/default issues are related but do not supersede this Telegram-specific security-default request.

## What This Changes

The issue requests Telegram onboarding to default fresh owner setup to `dmPolicy: "allowlist"` with the owner Telegram ID captured during setup.

## Best Possible Solution

Implement this as Telegram-owned onboarding hardening: generate a short verification token, capture only the matching numeric sender ID, write `dmPolicy="allowlist"` plus `allowFrom` for fresh owner setup, preserve explicit pairing configs, and update schema/runtime/docs/tests.

## Reproduction Assessment

Yes, at source level. Current main still resolves omitted Telegram `dmPolicy` to pairing, sends pairing challenges to unknown DMs, and documents/tests that behavior; I did not run live Telegram setup because the source path directly proves the current state.

## Solution Assessment

No for a default-only change. The maintainable fix needs token-verified owner capture during Telegram setup plus schema/runtime/docs/tests, not merely flipping the default or trusting the first inbound sender.
