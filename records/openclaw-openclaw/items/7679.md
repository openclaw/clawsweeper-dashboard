## Summary

Keep open: current main and the latest release still default Telegram DMs to pairing, still document pairing as the default, and still use manual numeric `allowFrom` entry rather than token-verified owner capture. The related open issues and PRs overlap but do not supersede this focused Telegram setup/security-default request.

## What This Changes

The issue asks Telegram onboarding to default new setups to `dmPolicy="allowlist"` by securely capturing the owner Telegram ID during setup.

## Best Possible Solution

Implement a maintainer-approved Telegram setup flow that generates a short verification token, captures only the matching sender ID, writes `dmPolicy="allowlist"` plus `allowFrom`, and preserves explicit pairing-mode compatibility with docs and regression tests.

## Reproduction Assessment

Yes, at source level: current main still defaults omitted Telegram DM policy to pairing, sends pairing challenges to unknown DMs, documents pairing as default, and uses manual numeric `allowFrom` entry instead of verified owner capture. I did not run tests or a live Telegram bot because this review is read-only.

## Solution Assessment

No, not as a blanket default flip or first-message capture. The maintainable solution is verified fresh-setup owner enrollment plus a deliberate compatibility decision for existing omitted-policy installs.
