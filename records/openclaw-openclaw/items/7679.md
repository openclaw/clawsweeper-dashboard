## Summary

Current main and v2026.6.5 still default Telegram DMs to pairing and require manual numeric allowlist entry, so the secure owner-enrollment behavior remains missing; this needs an explicit maintainer security and compatibility decision rather than being folded into the broader bootstrap request.

## What This Changes

Make fresh Telegram onboarding securely verify the intended owner, store that numeric sender ID, and configure direct messages for allowlist access by default.

## Best Possible Solution

Implement Telegram-owned fresh-setup enrollment that displays a short-lived nonce, accepts it only from the matching inbound DM, records that sender's numeric ID, and writes explicit `dmPolicy: "allowlist"` plus `allowFrom`, while leaving existing configurations unchanged unless maintainers approve and document a migration.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.5 default missing Telegram DM policy to pairing, send pairing challenges on that path, and expose only manual numeric allowFrom setup. This is a requested security and onboarding behavior change rather than a broken documented contract.

## Solution Assessment

No, a blanket default flip or first-sender capture is not the best solution; verified numeric owner enrollment scoped to fresh setup is safer, and maintainers must separately decide whether omitted-policy upgrades should retain pairing.

## Risks / Open Questions

- Blindly capturing the first inbound sender during onboarding creates a race in which another Telegram user could be enrolled as owner; enrollment needs proof of a wizard-generated nonce or equivalent explicit verification.
- Changing the schema or runtime fallback globally could make existing installations with omitted dmPolicy stop accepting pairing requests, so fresh-install behavior and upgrade behavior need separate, intentional treatment.
- Username-based enrollment would conflict with the maintained numeric-ID authorization contract because Telegram usernames are mutable and are not a reliable arbitrary-user lookup key.
