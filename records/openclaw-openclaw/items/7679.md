## Summary

Keep open: current main still implements and documents Telegram DM pairing as the default, while the requested safer allowlist onboarding needs a trust-model decision and token-verified owner capture rather than a default-only flip.

## What This Changes

Request fresh Telegram setup to default DMs to allowlist mode by capturing the owner's Telegram ID during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender ID, writes explicit dmPolicy="allowlist" and allowFrom for fresh setup, updates docs/schema/tests, and leaves existing explicit pairing configs intact.

## Reproduction Assessment

Yes for the reported current behavior: source inspection on current main shows missing Telegram dmPolicy still resolves to pairing, and the DM access test asserts a pairing-code reply for unauthorized DMs under pairing policy. This is a safer-default feature request rather than a broken existing contract.

## Solution Assessment

No for a default-only flip. The maintainable solution needs token-verified owner capture plus explicit non-empty allowFrom; otherwise it is either unsafe or rejected by current validation.
