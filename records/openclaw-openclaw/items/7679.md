## Summary

Keep open. Current main still implements and documents Telegram DM pairing as the default, while setup only writes allowlist mode after manual numeric `allowFrom` entry; the remaining work is a security-sensitive onboarding/default change that needs token-verified owner capture.

## What This Changes

Request Telegram setup to default new DMs to allowlist by capturing the owner's Telegram user ID during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that token-verifies the intended owner, writes explicit `dmPolicy: "allowlist"` plus `allowFrom`, updates schema/runtime/docs/tests, and preserves explicit pairing configs.

## Reproduction Assessment

Yes. Source inspection on current main is enough: omitted Telegram `dmPolicy` still resolves to pairing, and the DM access test still covers unauthorized pairing-code responses.

## Solution Assessment

Unclear as written. The safer direction is sound only with token-verified owner capture; changing the schema default alone would fail allowlist validation, and first-message capture would introduce a race.
