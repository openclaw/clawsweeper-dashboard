## Summary

Keep open. Current main still defaults Telegram DMs to pairing, still sends pairing challenges to unknown DM senders, and setup/docs/tests still model manual numeric allowFrom entry rather than token-verified owner ID capture; the token-capture warning in the discussion remains relevant.

## What This Changes

Fresh Telegram onboarding should default DMs to allowlist mode and record the owner’s numeric Telegram ID in allowFrom.

## Best Possible Solution

Implement Telegram-owned onboarding that generates a short verification token, captures only the sender who sends that token, writes explicit dmPolicy="allowlist" plus allowFrom, and updates schema/runtime/docs/tests while preserving explicit pairing-mode configs.

## Reproduction Assessment

Yes, for the reported current behavior. Source inspection on current main shows omitted Telegram dmPolicy still resolves to pairing, and existing tests assert pairing-code responses for unauthorized DMs under pairing policy.

## Solution Assessment

Unclear as a default-only change. The maintainable fix needs token-verified owner capture plus explicit allowFrom; flipping the default alone is not safe or complete.
