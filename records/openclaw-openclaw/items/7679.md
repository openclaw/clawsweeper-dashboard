## Summary

Keep open: current main and v2026.5.22 still default Telegram DMs to pairing, still issue pairing challenges to unknown DM senders, and still lack token-verified owner ID capture during Telegram setup. The broader bootstrap and post-bootstrap default issues are related, but they do not supersede this Telegram-specific security-default request.

## What This Changes

Request fresh Telegram setup to default to owner allowlist mode by capturing a verified Telegram owner ID instead of leaving omitted dmPolicy in pairing mode.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, capture only the matching numeric sender ID, write dmPolicy="allowlist" plus allowFrom for fresh owner setup, preserve explicit pairing configs, and update schema/runtime/docs/tests.

## Reproduction Assessment

Yes, at source level. Current main still resolves omitted Telegram dmPolicy to pairing, sends pairing challenges to unknown DMs, and documents/tests that behavior; I did not run live Telegram setup because the source path directly proves the current state.

## Solution Assessment

No for a default-only change. The maintainable fix needs token-verified owner capture during Telegram setup plus compatibility-aware schema/runtime/docs/tests, not merely flipping the default or trusting the first inbound sender.
