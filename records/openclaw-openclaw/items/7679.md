## Summary

Keep open: current main still defaults Telegram DMs to pairing in schema, runtime, docs, and tests, while setup only writes allowlist after manual numeric ID entry. The requested safer default needs a Telegram-owned onboarding/security decision, especially the token-verified owner capture called out in the discussion.

## What This Changes

The issue asks Telegram onboarding to default fresh direct-message access to an owner allowlist instead of pairing.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, capture only the matching sender ID, write dmPolicy="allowlist" plus allowFrom for fresh owner setup, and preserve explicit pairing configs.

## Reproduction Assessment

Yes at source level. Current main still resolves omitted Telegram dmPolicy to pairing, sends pairing challenges to unknown DMs, and documents/tests that behavior; I did not run a live Telegram setup because the source path directly shows the current state.

## Solution Assessment

No for an automated default-only change. The maintainable fix needs token-verified owner capture during Telegram setup plus docs/tests, not merely flipping the schema default.
