## Summary

Keep open: current main and the latest tagged release still default Telegram DMs to pairing, still issue pairing challenges for unknown DM senders, and still use manual numeric allowFrom setup rather than token-verified owner capture. This remains a security-sensitive defaults/onboarding decision, not conservative cleanup.

## What This Changes

The issue asks for fresh Telegram setup to default DMs to allowlist mode and automatically record the owner Telegram user ID during onboarding.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, require the intended owner to send it to the bot, capture that matching numeric Telegram user ID, write dmPolicy="allowlist" plus allowFrom for fresh setup, and preserve existing explicit pairing configs.

## Reproduction Assessment

Yes for the current behavior, though this is a defaults/onboarding feature request rather than a failing-test bug. Source inspection on current main shows omitted Telegram dmPolicy still defaults and falls back to pairing, unknown DMs still receive pairing challenges, and setup/docs still use manual numeric allowFrom entry.

## Solution Assessment

Yes directionally, but only with token-verified owner capture. A schema/runtime default flip alone is not the best solution because allowlist mode requires a sender ID and first-message capture would introduce an ownership race.
