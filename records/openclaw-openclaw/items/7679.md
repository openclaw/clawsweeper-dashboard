## Summary

Keep open. Current main still defaults Telegram DMs to pairing and still documents/setup-tests manual numeric allowFrom entry, while the requested safer default needs token-verified owner capture and maintainer judgment on Telegram onboarding/security behavior.

## What This Changes

The issue requests fresh Telegram setup to default to dmPolicy="allowlist" and populate allowFrom with the owner's Telegram ID captured during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the sender who sends that token, writes explicit dmPolicy="allowlist" plus allowFrom for fresh setup, updates docs/schema/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Yes for the reported current behavior: source inspection on current main shows missing Telegram dmPolicy still resolves to pairing, and the DM access test asserts a pairing-code reply for unauthorized DMs under pairing policy. This is a safer-default feature request rather than a broken existing contract.

## Solution Assessment

No for a default-only flip. The maintainable solution needs token-verified owner capture plus explicit non-empty allowFrom; otherwise it is either unsafe or rejected by current validation.
