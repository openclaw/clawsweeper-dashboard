## Summary

Keep open: current main and the latest release still default Telegram DMs to pairing, while setup/docs remain centered on manual numeric allowFrom entry instead of token-verified owner capture. This is a security-sensitive product/default change, not a strict bug-fix automation candidate.

## What This Changes

Request fresh Telegram setup to default DMs to allowlist by capturing the owner Telegram user ID during onboarding instead of defaulting to pairing.

## Best Possible Solution

Implement this in the Telegram plugin setup flow: generate a short verification token, capture only the matching sender ID, write dmPolicy="allowlist" plus allowFrom for fresh setup, update docs/schema/tests, and preserve existing explicit pairing-mode configs.

## Reproduction Assessment

Yes, as a source-level check: current main still defaults Telegram config/runtime to pairing, docs say pairing is the default, and tests cover pairing-code replies for unauthorized DMs. I did not run tests because this review was read-only.

## Solution Assessment

Yes directionally, but only with token-verified owner capture and an upgrade-safe defaulting plan. Changing the schema/runtime default alone is not the best solution because allowlist mode requires an allowFrom entry and existing omitted-policy configs may rely on pairing.
