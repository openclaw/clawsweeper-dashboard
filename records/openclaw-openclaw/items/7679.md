## Summary

Keep open: current main and the latest tagged release still default Telegram DMs to pairing, still issue pairing challenges to unknown DM senders, and still rely on manual numeric allowFrom setup rather than token-verified owner capture. This remains a security-sensitive Telegram onboarding/defaults decision; the broader open follow-up at https://github.com/openclaw/openclaw/issues/81876 does not replace the Telegram-specific fresh-setup capture work.

## What This Changes

Request fresh Telegram setup to default DMs to allowlist and populate allowFrom with a verified owner Telegram ID instead of using pairing by default.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, require the intended owner to send it to the bot, capture that matching numeric Telegram user ID, write dmPolicy="allowlist" plus allowFrom for fresh setup, and preserve existing explicit pairing configs.

## Reproduction Assessment

Yes for the current behavior, though this is a defaults/onboarding feature request rather than a failing-test bug. Source inspection shows omitted Telegram dmPolicy still defaults and falls back to pairing, unknown DMs still receive pairing challenges, and setup/docs still use manual numeric allowFrom entry.

## Solution Assessment

Yes directionally, but only with token-verified owner capture. A schema/runtime default flip alone is not the best solution because allowlist mode requires a sender ID and first-message capture would introduce an ownership race.
