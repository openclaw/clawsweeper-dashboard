## Summary

Keep open: current main still defaults Telegram DMs to pairing, still issues pairing challenges to unknown DM senders, and setup/docs still use manual numeric allowFrom entry instead of token-verified owner capture. This remains a Telegram onboarding security/product decision, not a conservative cleanup close.

## What This Changes

Request fresh Telegram setup/defaults to use dmPolicy="allowlist" and capture the owner Telegram user ID into allowFrom instead of starting in pairing mode.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, require the intended owner to send it to the bot, capture that matching numeric Telegram user ID, write dmPolicy="allowlist" plus allowFrom for fresh setup, and preserve explicit existing pairing configs.

## Reproduction Assessment

Yes for the current behavior. Source inspection on current main shows omitted Telegram dmPolicy still defaults/falls back to pairing, unknown DMs still get pairing challenges, and setup/docs still model manual numeric allowFrom entry rather than token-verified owner capture.

## Solution Assessment

Yes directionally, but only with token-verified owner capture. A default-only schema/runtime change is not sufficient because allowlist mode requires a non-empty allowFrom and first-message capture would introduce an ownership race.
