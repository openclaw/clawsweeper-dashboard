## Summary

Keep open. Current main still implements and documents Telegram DMs as pairing by default, and setup still asks for manual numeric `allowFrom` entry rather than token-verified owner ID capture, so the safer-default request remains unresolved and needs maintainer design review.

## What This Changes

Default new Telegram DM setup to `dmPolicy: "allowlist"` by capturing the owner Telegram ID during onboarding.

## Best Possible Solution

Implement this as Telegram-owned onboarding/access-control work: generate a short verification token, capture the sender ID only from a matching message, write explicit `dmPolicy: "allowlist"` plus `allowFrom`, update docs/tests, and preserve existing explicit pairing configs.

## Reproduction Assessment

Yes. Source inspection on current main is enough: omitting Telegram `dmPolicy` leaves schema and runtime on `pairing`, and the existing DM access test covers the unauthorized-sender pairing-code path.

## Solution Assessment

Unclear as written. The safer default is maintainable only with token-verified owner capture; changing the schema default alone or capturing the first sender would be incomplete and unsafe.
