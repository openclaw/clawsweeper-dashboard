## Summary

Keep open: current main still defaults Telegram DMs to pairing, still sends pairing challenges to unknown DM senders, and setup/docs still require manual numeric allowFrom entry. The request is a valid security-hardening feature, but the safe path needs token-verified owner capture and maintainer product/security review rather than a default-only flip.

## What This Changes

Telegram fresh setup should default DMs to allowlist mode by capturing the owner's Telegram user ID during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender's numeric Telegram ID, writes explicit dmPolicy="allowlist" plus allowFrom, and updates schema/runtime/docs/tests while preserving existing explicit pairing configs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a safer-default feature request. Current behavior is source-reproducible from schema, runtime, setup, docs, and tests that still model pairing by default with manual numeric allowFrom entry.

## Solution Assessment

Yes if implemented with token-verified owner capture and explicit non-empty allowFrom; no for a default-only flip. The current schema and setup flow make a default-only change unsafe or incomplete.
