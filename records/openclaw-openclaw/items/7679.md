## Summary

Keep open: current main still defaults Telegram DMs to pairing, documents pairing as the default, and only supports manual numeric allowFrom setup; the requested safer default needs a Telegram trust-model and token-verified owner-capture decision before implementation.

## What This Changes

Fresh Telegram setup should default DMs to allowlist and automatically add a verified owner Telegram ID to allowFrom.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender ID, writes explicit dmPolicy="allowlist" and allowFrom for fresh setup, updates docs/schema/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Yes for the current behavior: source inspection shows missing Telegram dmPolicy still resolves to pairing, and the DM access test asserts a pairing-code reply for unauthorized DMs under pairing policy. This is a safer-default feature request rather than a broken existing contract.

## Solution Assessment

No for a default-only flip. The maintainable path needs token-verified owner capture plus explicit non-empty allowFrom; otherwise the change is unsafe or rejected by current validation.
