## Summary

Keep open: current main still defaults Telegram DMs to pairing, still sends pairing challenges for unknown DM senders in pairing mode, and setup/docs still use manual numeric allowFrom entry rather than token-verified owner capture. This is a valid security-hardening feature request, but a default-only automated fix would be unsafe and incomplete.

## What This Changes

The issue requests Telegram onboarding to default new DM access to allowlist mode by token-verifying and capturing the owner’s Telegram ID.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender’s numeric Telegram ID, writes explicit dmPolicy="allowlist" plus allowFrom, updates docs/schema/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Yes for the current behavior: source inspection on origin/main shows missing Telegram dmPolicy resolves to pairing, and existing tests assert pairing-code replies for unauthorized DMs under pairing policy. This remains a safer-default/security-hardening request rather than a broken existing contract.

## Solution Assessment

Yes if implemented with token-verified owner capture and explicit non-empty allowFrom; no for changing only the default. The current validation and setup flow make a default-only fix unsafe or incomplete.
