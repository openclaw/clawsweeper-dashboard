## Summary

Keep open. Current main still defaults Telegram DMs to pairing, still sends pairing challenges to unknown DM senders, and setup/docs/tests still model manual numeric allowFrom entry rather than token-verified owner ID capture; the issue discussion’s token-capture warning remains relevant.

## What This Changes

Change Telegram setup and defaults so a fresh owner setup uses dmPolicy="allowlist" with the owner’s Telegram user ID captured during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that generates a short verification token, captures only the sender who sends that token, writes explicit dmPolicy="allowlist" plus allowFrom, and updates schema/runtime/docs/tests while preserving existing explicit pairing-mode configs.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted Telegram dmPolicy still resolves to pairing, and existing tests assert pairing-code responses for unauthorized DMs under pairing policy.

## Solution Assessment

Unclear as a default-only change. The maintainable fix needs token-verified owner capture plus explicit allowFrom; a schema/runtime default flip alone is not safe or complete.
