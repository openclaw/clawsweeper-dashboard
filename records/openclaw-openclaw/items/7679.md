## Summary

Keep open: current main still defaults Telegram DMs to pairing, still sends pairing challenges to unknown DM senders, and still uses manual numeric allowFrom setup rather than token-verified owner capture. Related bootstrap/default issues are broader and open, so they do not supersede this Telegram-specific security-default request.

## What This Changes

The issue asks for fresh Telegram setup to default DMs to allowlist mode by token-safely capturing the owner's Telegram user ID and writing it to allowFrom.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, capture only the matching numeric sender ID, write dmPolicy="allowlist" plus allowFrom for fresh owner setup, preserve explicit pairing configs, and update schema/runtime/docs/tests.

## Reproduction Assessment

Yes, at source level. Current main still resolves omitted Telegram dmPolicy to pairing, sends pairing challenges to unknown DMs, and documents/tests that behavior; I did not run live Telegram setup because the source path directly proves the current state.

## Solution Assessment

No for a default-only change. The maintainable fix needs token-verified owner capture during Telegram setup plus schema/runtime/docs/tests, not merely flipping the default or trusting the first inbound sender.
