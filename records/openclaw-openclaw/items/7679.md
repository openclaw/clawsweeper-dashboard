## Summary

Keep open: current main and the latest release still default Telegram DMs to pairing, still issue pairing challenges to unknown DM senders, and still rely on manual numeric allowFrom entry rather than token-verified owner ID capture during setup.

## What This Changes

The issue requests changing fresh Telegram setup/defaults to use allowlist mode with an auto-captured owner Telegram ID instead of pairing mode.

## Best Possible Solution

Implement this as Telegram-owned onboarding work: generate a short verification token, capture only the matching numeric Telegram sender ID, write dmPolicy="allowlist" plus allowFrom for fresh owner setup, preserve explicit pairing configs, and update schema/runtime/docs/tests.

## Reproduction Assessment

Yes, at source level. Current main still resolves omitted Telegram dmPolicy to pairing, sends pairing challenges to unknown DMs, and documents/tests that behavior; no live Telegram run is needed to prove the requested behavior is still missing.

## Solution Assessment

No for a default-only change. The maintainable fix needs token-verified owner capture during Telegram setup plus compatibility coverage, not merely flipping the schema/runtime default or trusting the first inbound sender.
