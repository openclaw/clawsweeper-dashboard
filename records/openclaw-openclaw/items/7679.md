## Summary

Keep open: current main still defaults, resolves, and documents Telegram DMs as pairing, and setup only supports manual numeric allowFrom entry. The requested safer default needs token-verified owner capture plus maintainer product/security review before implementation.

## What This Changes

Default fresh Telegram setup to allowlist mode by capturing the owner Telegram user ID into allowFrom during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender ID, writes explicit dmPolicy="allowlist" plus allowFrom, updates docs/schema/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Yes for the current behavior: source inspection on current main shows missing Telegram dmPolicy resolves to pairing, and existing tests assert pairing-code replies for unauthorized DMs. This is a safer-default/security hardening request rather than a broken existing contract.

## Solution Assessment

Yes if implemented with token-verified owner capture and explicit non-empty allowFrom; no for changing only the default. The current validation and setup flow make a default-only fix unsafe or incomplete.
