## Summary

Keep open: current main still defaults Telegram DMs to pairing, setup still asks for manual numeric allowFrom entry, and docs still present pairing as the default. This is a valid security-hardening feature request, but the safe path needs token-verified owner capture and maintainer product/security review rather than a default-only change.

## What This Changes

The issue requests Telegram onboarding to default direct messages to allowlist mode by capturing the owner's Telegram user ID during setup.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender’s numeric Telegram ID, writes explicit dmPolicy="allowlist" plus allowFrom, updates docs/schema/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a safer-default feature request. The current behavior is high-confidence source-reproducible: schema, runtime, setup, docs, and tests still model pairing by default with manual allowFrom entry.

## Solution Assessment

Yes if implemented with token-verified owner capture and explicit non-empty allowFrom; no for changing only the default. The current schema and setup flow make a default-only fix unsafe or incomplete.
