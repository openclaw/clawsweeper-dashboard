- `impact:security`: The issue concerns how the initial Telegram owner is authenticated and who may initiate DM access.

## Summary

Current main and v2026.6.5 still use pairing-oriented Telegram defaults and manual owner-ID entry. The newer cross-channel proposal at https://github.com/openclaw/openclaw/issues/81876 overlaps post-bootstrap hardening but deliberately preserves first-run pairing, so it does not supersede this issue’s distinct token-verified first-owner enrollment request.

## What This Changes

Securely verify and capture the intended Telegram owner during onboarding, then configure new setups with owner-only allowlist DM access.

## Best Possible Solution

Add a Telegram-owned one-time challenge during fresh onboarding, persist the matching sender’s numeric ID and allowlist policy atomically, preserve all explicit existing policies, and coordinate post-bootstrap cross-channel behavior with https://github.com/openclaw/openclaw/issues/81876.

## Reproduction Assessment

Yes at source level. Leave Telegram dmPolicy unset on current main and setup/runtime resolve it to pairing, while onboarding requires manual numeric allowFrom entry rather than verified sender capture.

## Solution Assessment

The token-verified owner-enrollment direction is the safest narrow implementation, but maintainers must approve its fresh-install and upgrade semantics before coding because it changes an authorization default.
