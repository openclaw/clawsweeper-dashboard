## Summary

Keep open: current main still defaults Telegram DMs to pairing, setup still asks for manual numeric allowFrom entry, and docs still describe pairing as the default. This is a valid security-hardening feature request, but the safe implementation needs token-verified owner capture rather than a default-only flip.

## What This Changes

Request Telegram setup/defaults to use allowlist mode with a verified captured owner Telegram ID instead of pairing mode.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender’s numeric Telegram ID, writes explicit dmPolicy="allowlist" plus allowFrom, updates docs/schema/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a safer-default feature request. The current behavior is high-confidence source-reproducible: schema/runtime/docs/tests still model pairing by default and manual allowFrom entry.

## Solution Assessment

Yes if implemented with token-verified owner capture and explicit non-empty allowFrom; no for changing only the default. The current schema and setup flow make a default-only fix unsafe or incomplete.
