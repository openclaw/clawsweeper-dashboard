## Summary

Keep this issue open. Current main still defaults Telegram DM access to pairing in both schema and runtime, still sends pairing challenges to unknown direct-message senders, and setup/docs still model manual numeric allowFrom entry rather than token-verified owner ID capture during onboarding. The discussion’s warning about verifying the captured sender with a short token remains relevant, so the safer-default request is still unresolved.

## What This Changes

Telegram should default fresh setup to `dmPolicy: "allowlist"` by capturing the owner’s Telegram user ID during onboarding instead of defaulting to pairing mode.

## Best Possible Solution

Keep the issue open and implement this as Telegram-owned onboarding/access-control work: generate a short verification token, ask the intended owner to send that exact token to the bot, capture the matching numeric Telegram user ID, write explicit `dmPolicy: "allowlist"` plus `allowFrom`, and preserve existing explicit pairing-mode configs.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static on current main: omit Telegram `dmPolicy`, observe the schema/runtime default to pairing, and follow the existing DM access test showing unauthorized senders receive pairing codes under pairing policy.

## Solution Assessment

Partially. The requested allowlist default is the right direction only with token-verified owner capture; changing the schema default alone or capturing the first sender would be unsafe and incomplete.
