## Summary

Keep open. Current main still defaults Telegram DMs to pairing and setup/docs still use manual numeric `allowFrom`; the remaining work changes Telegram's onboarding/default trust model and needs token-verified owner capture.

## What This Changes

Default fresh Telegram setup to `dmPolicy: "allowlist"` by capturing the owner's Telegram user ID during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that token-verifies the intended owner, writes explicit `dmPolicy: "allowlist"` plus `allowFrom`, updates schema/runtime/docs/tests, and preserves explicit pairing configs.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted Telegram `dmPolicy` still resolves to pairing, and the DM access test still covers unauthorized pairing-code responses.

## Solution Assessment

Unclear as written. The safer direction is to pair the allowlist default with token-verified owner capture; a schema default flip alone would fail allowlist validation, and first-message capture would be unsafe.
