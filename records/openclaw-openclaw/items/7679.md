## Summary

Keep open. Current main still defaults Telegram DM access to pairing and setup/docs still model manual numeric `allowFrom` entry, while the requested safer default needs token-verified owner capture and maintainer judgment on the Telegram trust model.

## What This Changes

Requests Telegram onboarding to default fresh DM access to `dmPolicy="allowlist"` by capturing the owner's Telegram user ID and writing it to `allowFrom`.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the sender who sends that token, writes explicit `dmPolicy="allowlist"` plus `allowFrom` for fresh setup, updates docs/schema/tests, and preserves existing pairing configs.

## Reproduction Assessment

Yes for the reported current behavior: source inspection on current main shows missing Telegram `dmPolicy` still resolves to `pairing`, and the DM access test asserts a pairing-code reply for unauthorized DMs under pairing policy. This is a safer-default feature request rather than a broken existing contract.

## Solution Assessment

Unclear as a default-only change. The maintainable solution needs token-verified owner capture plus explicit non-empty `allowFrom`; flipping `dmPolicy` alone is unsafe and incomplete.
