## Summary

Keep open: current main still defaults and documents Telegram DMs as pairing, while the requested safer owner-only onboarding needs a deliberate token-verified capture flow rather than a default-only flip.

## What This Changes

The issue requests Telegram fresh setup/defaults to use `dmPolicy="allowlist"` with the owner Telegram ID captured into `allowFrom` instead of pairing by default.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender ID, writes explicit `dmPolicy="allowlist"` plus `allowFrom`, updates docs/schema/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Yes for the current behavior: source inspection on current main shows missing Telegram `dmPolicy` resolves to pairing, and the DM access test asserts a pairing-code reply for unauthorized DMs under pairing policy. This is a safer-default feature request rather than a broken existing contract.

## Solution Assessment

Yes if implemented with token-verified owner capture and explicit non-empty `allowFrom`; no for changing only the default. The current code and validation make a default-only flip unsafe or incomplete.
