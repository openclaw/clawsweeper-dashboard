## Summary

Keep open: current main still defaults Telegram DMs to pairing and the setup/docs path still relies on manual numeric allowFrom entry. This is a valid security-hardening feature request, but the safe implementation needs token-verified owner capture and maintainer product/security review rather than a default-only flip.

## What This Changes

Request fresh Telegram setup to default DMs to allowlist by capturing the owner's Telegram ID into allowFrom during onboarding.

## Best Possible Solution

Implement Telegram-owned onboarding that displays a short random verification token, captures only the matching sender's numeric Telegram ID, writes explicit dmPolicy="allowlist" plus allowFrom, updates schema/runtime/docs/tests, and preserves existing explicit pairing configs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a safer-default feature request. The current behavior is high-confidence source-reproducible from schema, runtime, setup, docs, and tests that still model pairing by default with manual allowFrom entry.

## Solution Assessment

Yes if implemented with token-verified owner capture and explicit non-empty allowFrom; no for changing only the default. The current schema and setup flow make a default-only fix unsafe or incomplete.
