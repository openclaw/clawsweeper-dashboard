## Summary

Keep open: the hardening goal is plausible, but the current PR is stale and unsafe because two added logger calls can throw on the exact fallback paths they are meant to protect, TLS chmod failures are logged too quietly, current main moved important fs-safe and allow-from behavior, and no real behavior proof is attached.

## What This Changes

The PR adds logging for TLS chmod failures, introduces fs-safe close-handle cleanup logging, logs malformed pairing allow-from JSON, and removes a non-null assertion in internal hook registration.

## Best Possible Solution

Replace or rebase with a current-main patch that keeps cleanup and malformed allow-from fallback non-throwing, logs TLS chmod failures at warn level, places allow-from JSON diagnostics in the shared reader, and adds focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, at source level. The PR passes `{ error }` where current `logWarn` expects a `RuntimeEnv`, and current main still suppresses TLS chmod failures with `catch(() => {})`.

## Solution Assessment

No. The requested hardening is narrow and maintainable, but this implementation is stale and can turn tolerated cleanup or allow-from fallback paths into thrown logger errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a real logger runtime to safeCloseHandle:** `src/infra/fs-safe.ts:29`
  - body: `logWarn` takes a `RuntimeEnv`, not metadata. Passing `{ error: err }` makes the close-failure path call `runtime.log(...)` on an object without `log`, so cleanup can throw and mask the original filesystem error.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave generated TLS key or cert files with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warning/error monitoring.
  - confidence: 0.88
- **[P2] Move allow-from logging to the shared reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main delegates allow-from parsing to `readAllowFromFileSyncWithExists` and `readAllowFromFileWithExists`, so pairing-store-only logging will miss the rebased/current path. Put the diagnostic in the shared file reader instead.
  - confidence: 0.86
