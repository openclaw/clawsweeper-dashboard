## Summary

Keep open: the hardening direction is useful, but the branch is not mergeable as-is because it misuses the logger contract in fallback paths, logs TLS chmod failures too quietly, is stale against current fs-safe and allow-from boundaries, and lacks required real behavior proof.

## What This Changes

The PR adds logging around gateway TLS chmod failures, file-handle cleanup, and malformed pairing allow-from JSON, and removes a non-null assertion in internal hook registration.

## Best Possible Solution

Rebase or replace with a current-main patch that uses the logger API correctly, warns for TLS chmod failures, places allow-from diagnostics in shared readers, and includes focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes, at source level: on the PR branch, triggering a close failure or malformed allow-from JSON reaches `logWarn(..., { error })`, while current main still swallows TLS chmod errors at `src/infra/tls/gateway.ts:59`. I did not run a runtime reproduction in this read-only review.

## Solution Assessment

No. The hardening direction is reasonable, but this patch is not the best solution because it can make cleanup/fallback paths throw and it targets stale boundaries instead of current main's shared readers and fs-safe facade.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a valid logger runtime to safeCloseHandle:** `src/infra/fs-safe.ts:29`
  - body: `logWarn` takes a `RuntimeEnv`, but this passes `{ error: err }`. If `handle.close()` rejects, the logger tries to call `runtime.log(...)` on that object and throws, so cleanup can mask the original file error.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS chmod failures at warning severity:** `src/infra/tls/gateway.ts:64-69`
  - body: The PR converts silent chmod failures into `info` logs. A failed chmod can leave generated TLS key or cert permissions weaker than intended, and info-level output can be missed by warn/error monitoring.
  - confidence: 0.88
- **[P2] Move allow-from logging into the shared readers:** `src/pairing/pairing-store.ts:443`
  - body: Current main routes allow-from parsing through shared async and sync readers. Keeping the warning in this pairing-store sync path would miss active shared-reader surfaces and should live with the fallback logic.
  - confidence: 0.84
