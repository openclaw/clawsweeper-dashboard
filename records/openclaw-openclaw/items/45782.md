## Summary

Keep open: the hardening goal is plausible, but this PR is stale against current main, lacks after-fix real behavior proof, and introduces concrete security-sensitive regressions in the logger error paths.

## What This Changes

The PR adds logging around TLS chmod failures, file-handle close cleanup failures, malformed pairing allow-from JSON, and removes a non-null assertion in internal hook registration.

## Best Possible Solution

Rebase or replace this with a current-main patch that uses the logger contract correctly, logs TLS chmod failures at warn level, places malformed allow-from diagnostics in the shared reader, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. The PR diff routes FileHandle.close() failures and malformed allow-from JSON through logWarn(message, { error }), while current main's logWarn contract treats that second argument as a RuntimeEnv and calls runtime.log.

## Solution Assessment

No. The requested hardening is narrow and useful, but this implementation is stale and changes tolerated fallback/cleanup paths into potentially throwing paths; the safer fix is a current-main patch with warn-level TLS logging and shared-reader allow-from diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass a real runtime to logWarn:** `src/infra/fs-safe.ts:28`
  - body: `logWarn` takes a `RuntimeEnv`, not metadata. Passing `{ error: err }` makes the close-failure path call `runtime.log(...)` on an object without `log`, so cleanup can throw and mask the original filesystem error instead of staying non-throwing.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:444`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty existing-store fallback, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Use warn for TLS permission failures:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave the generated TLS key or cert with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warning/error monitoring; widen the log type and use `warn`.
  - confidence: 0.88
- **[P2] Move allow-from logging into the shared reader:** `src/pairing/pairing-store.ts:444`
  - body: Current main delegates allow-from parsing to `readAllowFromFileWithExists` and `readAllowFromFileSyncWithExists`, so adding malformed JSON logging in `pairing-store.ts` will miss the rebased/current path. Put the diagnostic in the shared file reader instead.
  - confidence: 0.86
