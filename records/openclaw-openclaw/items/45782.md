## Summary

Keep this PR open. Current main still has the silent TLS chmod and file-handle close paths the PR is trying to improve, but the submitted patch is not merge-ready because two new `logWarn(..., { error })` calls can throw in the very error paths they are meant to observe, the TLS permission warning is logged at `info`, and the allow-from JSON change targets a stale parser location relative to current main.

## What This Changes

The PR adds logging for TLS chmod failures, file-handle close failures, and malformed allow-from JSON, and replaces one internal-hook non-null assertion with a local handler array variable.

## Best Possible Solution

Revise or replace the PR with a current-main patch that keeps the same narrow observability goal: warn on TLS chmod failures, log file-handle cleanup failures without changing cleanup control flow, add malformed allow-from JSON logging in the shared helper, preserve the hook type cleanup, and cover those error paths with targeted tests.

## Reproduction Assessment

Yes. The two throwing regressions are reproducible by exercising the newly added `logWarn(..., { error })` calls, because current `logWarn` treats the second argument as a `RuntimeEnv` and calls `runtime.log`; the stale allow-from target is reproducible by following current main from `pairing-store.ts` into `allow-from-store-file.ts`.

## Solution Assessment

No. The intended cleanup is narrow and reasonable, but this implementation should be rebased and moved to the current helper paths, use warning-level TLS logging, and call the logger through a valid runtime or message-only API so error handling remains non-throwing.

Full review comments:

- **[P1] Pass a real runtime to logWarn:** `src/infra/fs-safe.ts:29`
  - body: `logWarn` takes a `RuntimeEnv` as its second argument and calls `runtime.log`, so passing `{ error: err }` makes close-failure cleanup throw a TypeError. That masks the original open/read error instead of preserving the previous non-throwing cleanup behavior.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call also passes `{ error }` as the runtime argument. When malformed JSON is encountered, the logger can throw before the cache is updated and before the empty `exists: true` fallback returns.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave the generated key or cert with looser permissions than intended. Emitting that only through `info` can hide the security degradation from normal warning/error monitoring.
  - confidence: 0.88
- **[P2] Move allow-from logging to the current reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main delegates sync allow-from parsing to `readAllowFromFileSyncWithExists`, so adding malformed JSON logging to the old `pairing-store.ts` parser does not cover the current fallback path after a rebase.
