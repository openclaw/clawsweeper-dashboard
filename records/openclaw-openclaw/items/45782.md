## Summary

Keep open: the hardening direction is useful, but the proposed patch has concrete logger-runtime regressions, logs TLS permission failures too quietly, lacks required real behavior proof, and is stale against current main's fs-safe and allow-from reader boundaries.

## What This Changes

The PR adds logging around gateway TLS chmod failures, fs-safe handle close failures, malformed pairing allow-from JSON, and removes a non-null assertion in internal hook registration.

## Best Possible Solution

Replace this with a current-main patch that warns on TLS chmod failures, puts allow-from diagnostics in the shared readers, preserves non-throwing cleanup/fallback behavior, and includes focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes, at source level. The PR diff routes cleanup and malformed allow-from failures into logWarn(..., { error }), while current main's logger expects a RuntimeEnv and current main still suppresses TLS chmod failures at src/infra/tls/gateway.ts:59.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best solution because it can make cleanup/fallback paths throw, logs TLS permission failures at info severity, and misses current main's shared allow-from reader boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a valid logger runtime to safeCloseHandle:** `src/infra/fs-safe.ts:29`
  - body: `logWarn` takes a `RuntimeEnv`, but this call passes `{ error: err }`. If `handle.close()` rejects, the logger tries to call `runtime.log(...)` on that object and throws, so cleanup can mask the original file error instead of safely suppressing close failures.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS chmod failures at warning severity:** `src/infra/tls/gateway.ts:64-69`
  - body: The PR reports key/cert chmod failures with `info`. A failed permission restriction can leave generated TLS material weaker than intended, and info-level output can be missed by warning/error monitoring.
  - confidence: 0.88
- **[P2] Move allow-from diagnostics into the shared readers:** `src/pairing/pairing-store.ts:443`
  - body: Current main reads allow-from JSON through shared async and sync readers. Adding this warning only in the old pairing-store sync path would miss active reader surfaces after rebase, so the diagnostic belongs with the shared fallback logic.
  - confidence: 0.84
