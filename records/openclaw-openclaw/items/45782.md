## Summary

Keep open: the hardening direction is useful, but the branch is stale/conflicting, lacks required real behavior proof, and the current diff introduces concrete logging regressions in security-sensitive fallback paths.

## What This Changes

The PR adds logging for TLS permission restriction failures, file-handle close failures, malformed pairing allow-from JSON, and rewrites internal hook handler registration to avoid a non-null assertion.

## Best Possible Solution

Replace this with a current-main patch that warns on TLS chmod failures, puts allow-from diagnostics in the shared reader if desired, preserves non-throwing cleanup/fallback behavior, and includes focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes, at source level. The PR diff passes `{ error }` where `logWarn` expects a `RuntimeEnv`, and current main still suppresses TLS chmod failures in `src/infra/tls/gateway.ts`.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best solution because it can make cleanup/fallback paths throw, logs TLS permission failures at info severity, and misses current main's shared allow-from reader boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a valid runtime to cleanup logging:** `src/infra/fs-safe.ts:29`
  - body: `logWarn` takes a `RuntimeEnv`, but this passes `{ error: err }`. If `handle.close()` rejects, the logger tries to call `runtime.log(...)` on that object, so cleanup can throw and mask the original filesystem error instead of safely suppressing close failures.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call passes `{ error }` as the logger runtime. On malformed JSON that can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS chmod failures at warning severity:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod on generated TLS key/cert material can leave permissions weaker than intended. Reporting it through `info` can be missed by warning/error monitoring, so the generated-cert helper should accept and use `warn`.
  - confidence: 0.88
- **[P2] Move allow-from diagnostics into the shared readers:** `src/pairing/pairing-store.ts:443`
  - body: Current main reads allow-from JSON through the shared async and sync reader helpers. Adding the warning only in the old pairing-store-local sync path would miss active reader surfaces after rebase.
  - confidence: 0.84
