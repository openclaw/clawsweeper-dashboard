## Summary

Keep open: the observability hardening is still potentially useful, but this stale branch is conflicting, lacks after-fix real behavior proof, and introduces concrete runtime regressions in security-sensitive logging paths.

## What This Changes

The PR adds logging for TLS chmod failures, file-handle cleanup failures, malformed allow-from JSON, and replaces one internal-hook non-null assertion.

## Best Possible Solution

Rebase or replace with a current-main patch that warns on TLS chmod failures, adds allow-from diagnostics in the shared reader, preserves non-throwing fallbacks, updates focused tests, and includes redacted runtime proof.

## Reproduction Assessment

Yes, at source level. The PR passes { error } where current logWarn expects a RuntimeEnv, while current main still silently suppresses TLS chmod failures and preserves malformed allow-from fallback behavior in the shared reader.

## Solution Assessment

No. The intended observability hardening is useful, but this branch is not the best fix because it is stale, can make fallback/cleanup paths throw, and logs TLS permission failures below warning severity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a valid runtime to cleanup logging:** `src/infra/fs-safe.ts:28`
  - body: `logWarn` takes a `RuntimeEnv`, but this passes `{ error: err }`. If `handle.close()` rejects, the logger tries to call `runtime.log(...)` on that object, so cleanup can throw and mask the original filesystem error instead of safely suppressing close failures.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:444`
  - body: This `logWarn` call passes `{ error }` as the logger runtime. On malformed JSON, that can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Warn when TLS chmod restriction fails:** `src/infra/tls/gateway.ts:63-68`
  - body: A failed chmod on generated TLS key/cert material can leave permissions weaker than intended. Reporting it through `info` can be missed by warning/error monitoring, so the generated-cert helper should accept and use `warn`.
  - confidence: 0.88
