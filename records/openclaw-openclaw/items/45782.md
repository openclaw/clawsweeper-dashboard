## Summary

Keep open: the observability goal is still not implemented on current main, but this PR is not safe to merge because two new logging paths can throw and the allow-from change targets an obsolete parsing location.

## What This Changes

The PR adds logging for TLS chmod failures, file-handle close cleanup failures, malformed pairing allow-from JSON, and refactors internal hook registration to avoid a non-null assertion.

## Best Possible Solution

Revise or replace the PR with a current-main patch that keeps cleanup and malformed allow-from fallback non-throwing, logs TLS chmod failures at warning severity, updates the shared allow-from reader, and adds focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: force a `FileHandle.close()` rejection or malformed allow-from JSON on the PR branch, and the new `logWarn(..., { error })` calls can throw because the logger expects a runtime with `log`.

## Solution Assessment

No. The intended logging hardening is narrow and maintainable, but this implementation uses the logger contract incorrectly, logs TLS permission failures too quietly, and patches a stale allow-from parser instead of the current shared reader.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass a real logger runtime to safeCloseHandle:** `src/infra/fs-safe.ts:29`
  - body: `logWarn`'s second argument is a `RuntimeEnv`, not metadata. Passing `{ error: err }` makes the close-failure path call `runtime.log(...)` on an object without `log`, so cleanup can throw and mask the original filesystem error.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty existing-store fallback, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave the generated TLS key or cert with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warning/error monitoring; widen the log type and use `warn`.
  - confidence: 0.88
- **[P2] Move allow-from logging to the shared reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main delegates allow-from parsing to `readAllowFromFileSyncWithExists` and `readAllowFromFileWithExists`, so adding malformed JSON logging to the older `pairing-store.ts` parser will miss the current path after rebase.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/infra/tls/gateway.ts:64`
  - body: This is user-facing security-hardening/operator-observability work, but the PR does not update `CHANGELOG.md`. Add a concise single-line `### Fixes` entry before merge.
  - confidence: 0.78
