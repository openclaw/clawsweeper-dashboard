## Summary

Keep open for maintainer/contributor handling: the observability hardening is useful, but this PR is not mergeable because it introduces logger-contract regressions, keeps TLS permission failures at info severity, lacks real behavior proof, and conflicts with current-main code that moved the fs-safe and allow-from boundaries.

## What This Changes

The branch adds logging around TLS chmod failures, file-handle close failures, malformed allow-from JSON, and removes a non-null assertion in internal hook registration.

## Best Possible Solution

Replace or rebase this with a small current-main patch that warns on TLS chmod failures, adds allow-from diagnostics in the shared reader without changing fallback behavior, and includes focused regression coverage plus redacted real-run proof.

Yes, at source level: the PR passes { error } where current main's logWarn expects a RuntimeEnv, and the current main TLS chmod and shared allow-from fallback paths are still visible in source. I did not run a live reproduction because this was a read-only review.

## Review Findings

Full review comments:

- **[P1] Pass a valid runtime to cleanup logging:** `src/infra/fs-safe.ts:28`
  - body: `logWarn` takes a `RuntimeEnv`, but this call passes `{ error: err }`. If `handle.close()` rejects, logging tries to call `runtime.log(...)` on that object, so cleanup can throw and mask the original filesystem error instead of safely suppressing close failures.
  - confidence: 0.96
- **[P1] Keep malformed allow-from JSON handling non-throwing:** `src/pairing/pairing-store.ts:444`
  - body: This `logWarn` call passes `{ error }` as the logger runtime. On malformed JSON, that can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Warn when TLS chmod restriction fails:** `src/infra/tls/gateway.ts:63-68`
  - body: A failed chmod on generated TLS key/cert material can leave permissions weaker than intended. Reporting it through `info` can be missed by warning/error monitoring, so the generated-cert helper should accept and use `warn`.
  - confidence: 0.88
