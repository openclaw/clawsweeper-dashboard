## Summary

Keep open. The PR targets a real remaining observability gap, but it should not merge as-is because two new logging calls can throw in fallback/error-cleanup paths, the TLS permission failure is logged too quietly, and the allow-from change is stale against current main's shared reader.

## What This Changes

The PR adds logging around TLS chmod failures, file-handle close cleanup, and malformed allow-from JSON, and rewrites internal hook registration to avoid a non-null assertion.

## Best Possible Solution

Revise or replace the PR with a narrow current-main patch that preserves non-throwing cleanup/fallback behavior, emits warning-level TLS permission logs, places malformed allow-from JSON logging in the shared reader, keeps the hook cleanup if still useful, and includes focused tests plus any required changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: force a `handle.close()` rejection or malformed allow-from JSON in the PR branch and the new `logWarn(..., { error })` path can throw because `logWarn` expects a runtime with `log`. The stale allow-from target is also reproducible by following current main from `pairing-store.ts` into `allow-from-store-file.ts`.

## Solution Assessment

No. The intended observability cleanup is narrow and useful, but this implementation is not the best fix because it can make fallback paths throw, logs TLS permission failures at `info`, and patches an old allow-from parsing location instead of the current shared helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass a real runtime to logWarn:** `src/infra/fs-safe.ts:29`
  - body: `logWarn`'s second argument is a `RuntimeEnv`, not metadata. Passing `{ error: err }` means a close failure calls `runtime.log(...)` on an object without `log`, so cleanup can throw and mask the original filesystem error.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty existing-store fallback, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave the generated TLS key or cert with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warning/error monitoring; widen the log type and use `warn`.
  - confidence: 0.88
- **[P2] Move allow-from logging to the current reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main delegates allow-from parsing to `readAllowFromFileSyncWithExists`, so adding malformed JSON logging to the older `pairing-store.ts` parser will not cover the current fallback path after rebase. Put the logging in the shared helper while preserving its fallback result.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/infra/tls/gateway.ts:64`
  - body: This is a user-facing fix/security-hardening change for operator-visible logs, but the PR does not update `CHANGELOG.md`. Add a concise entry before merge, with allowed attribution only.
  - confidence: 0.78
