## Summary

Keep open: the security-observability goal is plausible, but this PR is not mergeable as-is because it is stale/conflicting, lacks real behavior proof, and introduces logger-contract regressions in security-sensitive fallback paths.

## What This Changes

The branch adds logging for TLS chmod failures, file-handle close cleanup, and malformed pairing allow-from JSON, plus a small internal hook non-null assertion cleanup.

## Best Possible Solution

Rebase or replace the patch against current main, keep cleanup and malformed allow-from fallback non-throwing, log TLS chmod failures at warn level, place allow-from diagnostics in the shared readers, and add focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. The PR passes { error } where current logWarn expects a RuntimeEnv, and current main still suppresses TLS chmod failures; no runtime test was run because this review was read-only.

## Solution Assessment

No. The hardening direction is narrow and maintainable, but this stale implementation can turn tolerated cleanup or allow-from fallback paths into thrown logger errors and logs TLS permission failures too quietly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a real logger runtime to safeCloseHandle:** `src/infra/fs-safe.ts:29`
  - body: logWarn takes a RuntimeEnv, not metadata. Passing { error: err } makes the close-failure path call runtime.log on an object without log, so cleanup can throw and mask the original filesystem error.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This logWarn call has the same invalid { error } runtime argument. On malformed JSON it can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave generated TLS key or cert files with looser permissions than intended. Emitting that only through info can hide the security degradation from warning/error monitoring.
  - confidence: 0.88
- **[P2] Move allow-from logging to the shared reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main routes allow-from parsing through readAllowFromFileWithExists and readAllowFromFileSyncWithExists, so pairing-store-only logging will miss the rebased/current path. Put the diagnostic in the shared file reader instead.
  - confidence: 0.86
