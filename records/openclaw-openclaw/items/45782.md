## Summary

Keep open. The observability hardening is useful and not fully present on current main, but this branch is stale/conflicting, introduces runtime regressions in two logging paths, logs TLS permission failures below warning severity, and still lacks required real behavior proof.

## What This Changes

The PR adds logging for TLS chmod failures, file-handle cleanup failures, malformed allow-from JSON, and refactors internal hook registration to avoid a non-null assertion.

## Best Possible Solution

Replace or rebase this with a current-main patch that warns on TLS chmod failures, adds allow-from diagnostics in the shared reader without changing fallback behavior, avoids stale fs-safe edits, adds focused tests, and includes redacted real-run proof.

## Reproduction Assessment

Yes, at source level. The PR passes `{ error }` where current logWarn expects a RuntimeEnv, and current main still has observable source gaps for TLS chmod logging and shared allow-from diagnostics.

## Solution Assessment

No. The intent is reasonable, but this branch is not the best fix because it is stale, conflicts with current main, can make cleanup/fallback paths throw, and logs TLS permission failures at info severity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a valid runtime to cleanup logging:** `src/infra/fs-safe.ts:28`
  - body: `logWarn` takes a `RuntimeEnv`, but this passes `{ error: err }`. If `handle.close()` rejects, the logger tries to call `runtime.log(...)` on that object, so cleanup can throw and mask the original filesystem error instead of safely suppressing close failures.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:444`
  - body: This `logWarn` call passes `{ error }` as the logger runtime. On malformed JSON, that can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Warn when TLS chmod restriction fails:** `src/infra/tls/gateway.ts:63-68`
  - body: A failed chmod on generated TLS key/cert material can leave permissions weaker than intended. Reporting it through `info` can be missed by warning/error monitoring, so the generated-cert helper should accept and use `warn`.
  - confidence: 0.88
