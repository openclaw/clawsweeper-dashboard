## Summary

Keep open: the hardening goal is still useful and current main has not fully implemented it, but this branch has source-level correctness/security blockers and missing contributor real behavior proof.

## What This Changes

The PR adds logging around TLS chmod failures, file-handle close cleanup failures, malformed pairing allow-from JSON, and removes a non-null assertion in internal hook registration.

## Best Possible Solution

Replace or rebase this as a current-main patch that uses the logger contract correctly, logs TLS permission failures at warn level, keeps malformed allow-from fallback non-throwing, and includes focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes, at source level: the PR diff passes `{ error }` as the second argument to `logWarn`, while current main's logger treats that argument as a RuntimeEnv and calls `runtime.log`. I did not run a live reproduction because this was a read-only review.

## Solution Assessment

No. The intended observability hardening is narrow and useful, but this implementation can make cleanup and malformed allow-from fallback paths throw, and it logs TLS permission failures below warning severity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass a real runtime to logWarn:** `src/infra/fs-safe.ts:28`
  - body: `logWarn` expects a runtime argument, not metadata. Passing `{ error: err }` means the close-failure logging path can throw while handling cleanup, masking the filesystem error this helper is supposed to preserve.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:444`
  - body: This logging call passes `{ error }` where the logger expects a runtime, so malformed JSON can throw before the empty existing-store fallback is cached and returned.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave generated TLS material with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warning/error monitoring.
  - confidence: 0.88
- **[P2] Move allow-from logging into the shared reader:** `src/pairing/pairing-store.ts:444`
  - body: Current main delegates allow-from parsing to the shared reader, so adding malformed JSON logging in `pairing-store.ts` will miss the rebased path. Put the diagnostic in `allow-from-store-file.ts` instead.
  - confidence: 0.86
