## Summary

Keep open: the security-hardening intent is useful, but this external PR should not merge as-is because it lacks real behavior proof, is conflicting/stale against current main, and introduces concrete logging regressions.

## What This Changes

The PR adds logging for TLS chmod failures, file-handle cleanup failures, malformed pairing allow-from JSON, and removes a non-null assertion in internal hook registration.

## Best Possible Solution

Replace or rebase with a current-main patch that keeps cleanup and malformed allow-from fallback non-throwing, logs TLS chmod failures at warn level, places allow-from JSON diagnostics in the shared reader, and adds focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, at source level. The PR passes `{ error }` where current `logWarn` expects a `RuntimeEnv`, and current main still shows TLS chmod failures are silently suppressed.

## Solution Assessment

No. The requested hardening is narrow and maintainable, but this implementation is stale and can turn tolerated cleanup/allow-from fallback paths into thrown logger errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass a real logger runtime to safeCloseHandle:** `src/infra/fs-safe.ts:29`
  - body: `logWarn` takes a `RuntimeEnv`, not metadata. Passing `{ error: err }` makes the close-failure path call `runtime.log(...)` on an object without `log`, so cleanup can throw and mask the original filesystem error.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty existing-store fallback, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave the generated TLS key or cert with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warning/error monitoring; widen the log type and use `warn`.
  - confidence: 0.88
- **[P2] Move allow-from logging to the shared reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main delegates allow-from parsing to `readAllowFromFileSyncWithExists` and `readAllowFromFileWithExists`, so adding malformed JSON logging in `pairing-store.ts` will miss the rebased/current path. Put the diagnostic in the shared file reader instead.
  - confidence: 0.86
