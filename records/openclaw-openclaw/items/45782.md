## Summary

Keep open: the security-logging goal is still useful, but this stale/conflicting PR introduces fallback-path regressions, logs TLS permission failures too quietly, and lacks required real behavior proof.

## What This Changes

The PR adds logging for TLS chmod failures, file-handle close cleanup failures, malformed pairing allow-from JSON, and refactors internal hook registration to avoid a non-null assertion.

## Best Possible Solution

Revise or replace this with a current-main patch that keeps cleanup and allow-from fallbacks non-throwing, logs TLS chmod failures at warn level, and puts malformed allow-from diagnostics in the shared reader with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: force a FileHandle.close rejection or malformed allow-from JSON on the PR branch, and the new logWarn calls can throw because they pass a metadata object where the logger expects a runtime.

## Solution Assessment

No. The hardening direction is narrow and maintainable, but this implementation uses the logger contract incorrectly, emits TLS permission failures at info level, and patches a stale allow-from parser instead of the current shared helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass a real runtime to logWarn in safeCloseHandle:** `src/infra/fs-safe.ts:29`
  - body: `logWarn` expects a RuntimeEnv as its second argument. Passing `{ error: err }` means a close failure can throw `runtime.log is not a function`, masking the original filesystem error instead of keeping cleanup non-throwing.
  - confidence: 0.96
- **[P1] Keep malformed allow-from JSON handling non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn(..., { error })` call has the same invalid runtime argument. On malformed JSON it can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave generated TLS key/cert files with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warn/error monitoring.
  - confidence: 0.88
- **[P2] Move allow-from logging to the shared reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main routes pairing allow-from parsing through the shared allow-from reader, so adding malformed JSON logging only to the older pairing-store parser will miss the current path after rebase.
  - confidence: 0.86
