## Summary

Keep open. The security-observability intent is still useful, but the current branch is not mergeable as-is because it misuses the logger contract, logs TLS chmod failures too quietly, and is stale against current shared fs-safe and allow-from boundaries.

## What This Changes

The PR adds logging for TLS chmod, file-handle cleanup, and malformed pairing allow-from JSON paths, plus removes a non-null assertion in the internal hook registry.

## Best Possible Solution

Rebase or replace the patch on current main, log TLS chmod failures at warning severity, add allow-from diagnostics in the shared reader, use the logger API correctly, and include focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes, at source level: the PR passes `{ error }` where current `logWarn` expects a `RuntimeEnv`, and current main still suppresses TLS chmod failures. I did not run a runtime reproduction in this read-only review.

## Solution Assessment

No. The hardening direction is reasonable, but this branch misuses the logger contract and targets stale file-local boundaries instead of current main's shared readers and fs-safe facade.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a real logger runtime to safeCloseHandle:** `src/infra/fs-safe.ts:29`
  - body: `logWarn` takes `(message, runtime?: RuntimeEnv)`, but this passes `{ error: err }`. If `handle.close()` fails, the logger calls `runtime.log(...)` on an object without `log`, so cleanup can throw and mask the original I/O error instead of staying safe.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty fallback return, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave generated TLS key or cert files with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warning/error monitoring.
  - confidence: 0.88
- **[P2] Move allow-from logging to the shared reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main routes allow-from parsing through `readAllowFromFileWithExists` and `readAllowFromFileSyncWithExists`, so keeping the diagnostic in `pairing-store` would miss the active shared reader path. Put the malformed JSON warning where those readers already preserve the fallback.
  - confidence: 0.84
