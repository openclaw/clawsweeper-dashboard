## Summary

Keep open. Current main still has the intended observability gaps, but this PR is not safe to merge as-is: prior Greptile/ClawSweeper comments correctly identified invalid `logWarn` usage, stale allow-from placement, too-quiet TLS permission logging, and missing release-note coverage.

## What This Changes

The PR adds logging for TLS chmod failures, file-handle close failures, and malformed pairing allow-from JSON, plus refactors internal hook registration to avoid a non-null assertion.

## Best Possible Solution

Revise or replace the PR with a current-main patch that preserves non-throwing cleanup and allow-from fallback behavior, emits warning-level TLS permission logs, adds malformed JSON logging in the shared allow-from reader, and includes focused tests plus a changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: force a `handle.close()` rejection or malformed allow-from JSON on the PR branch and the new `logWarn(..., { error })` path can throw because `logWarn` expects a runtime with `log`.

## Solution Assessment

No. The intended observability cleanup is narrow and useful, but this implementation is not the best fix because it can make fallback paths throw, logs TLS permission failures at `info`, and patches an obsolete allow-from parsing location for current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass a real runtime to logWarn:** `src/infra/fs-safe.ts:29`
  - body: `logWarn`'s second argument is a `RuntimeEnv`, not metadata. Passing `{ error: err }` means a close failure calls `runtime.log(...)` on an object without `log`, so cleanup can throw and mask the original filesystem error.
  - confidence: 0.96
- **[P1] Keep malformed allow-from fallback non-throwing:** `src/pairing/pairing-store.ts:443`
  - body: This `logWarn` call has the same invalid `{ error }` runtime argument. On malformed JSON it can throw before the cache update and empty existing-store fallback, turning a tolerated allow-from file into a fatal path.
  - confidence: 0.95
- **[P2] Log TLS permission failures at warn level:** `src/infra/tls/gateway.ts:64-69`
  - body: A failed chmod can leave the generated TLS key or cert with looser permissions than intended. Emitting that only through `info` can hide the security degradation from warning/error monitoring; widen the log type and use `warn`.
  - confidence: 0.88
- **[P2] Move allow-from logging to the shared reader:** `src/pairing/pairing-store.ts:443`
  - body: Current main delegates allow-from parsing to `readAllowFromFileSyncWithExists`, so adding malformed JSON logging to the older `pairing-store.ts` parser will not cover the current fallback path after rebase. Put the logging in the shared helper while preserving its fallback result.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/infra/tls/gateway.ts:64`
  - body: This is user-facing security-hardening/operator-observability work, but the PR does not update `CHANGELOG.md`. Add a concise `### Fixes` entry before merge, with allowed attribution only.
  - confidence: 0.78
