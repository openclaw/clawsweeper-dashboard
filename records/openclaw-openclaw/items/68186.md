## Summary

Keep open: the PR targets a real current-main macOS screen.snapshot bug, but the latest head still falsely rejects some in-budget near-limit snapshots and the external PR proof is only injected Swift tests, not a real macOS node/gateway run.

## What This Changes

The PR hardens macOS screen.snapshot handling, plumbs nodeId through bridge invoke requests for frame budgeting, adds Swift regression coverage, and updates the changelog.

## Best Possible Solution

Repair or remove the raw precheck so in-budget responses reach the projected frame check, add an accepted near-limit regression, collect redacted real macOS node/gateway proof, then run focused Swift checks before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source inspection: current main still falls back from malformed screen.snapshot params to defaults and emits inline base64 payloadJSON without a size guard; the PR-head near-limit regression is also reproducible from the raw precheck arithmetic.

## Solution Assessment

Is this the best way to solve the issue? No, not yet; the fail-closed and projected-frame direction is maintainable, but the raw precheck should not reject frames that the later projected-size check would accept, and the external PR still needs real behavior proof.
