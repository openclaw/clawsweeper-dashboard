## Summary

Keep open: current main still has the reported macOS `screen.snapshot` validation and payload-bound gaps, and this PR is the active fix path, but the branch still has a P2 false-rejection bug in its raw-byte precheck that should be fixed before merge.

## What This Changes

The PR hardens macOS `screen.snapshot` by rejecting malformed params, sanitizing capture failures, bounding inline snapshot responses against a projected `node.invoke.result` frame size, plumbing `nodeId` into Swift bridge requests, adding runtime regression tests, and updating the changelog.

## Best Possible Solution

Keep this PR as the active fix path, raise or remove the raw precheck so the projected frame calculation decides near-limit payloads, then land the hardened implementation with focused Swift regression coverage and the existing changelog entry.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction for the original bug, and the PR-branch regression is reproducible with an injected snapshot just above the 19,660,032-byte raw cutoff that still fits under the 25 MiB frame budget with short ids.

## Solution Assessment

No, not yet. The hardening direction is narrow and maintainable, but the raw precheck should use a true base64-only overflow cutoff or defer near-limit cases to `projectedOuterFrameBytes`.
