## Summary

Keep this PR open. Current main still has the reported macOS screen.snapshot malformed-params and unbounded-payload behavior, and this branch is the active implementation for the linked bug, but the latest diff still introduces a near-threshold false rejection before its precise frame-size projection can run.

## What This Changes

The PR hardens macOS `screen.snapshot` by rejecting malformed params, sanitizing capture failures, adding payload/frame-size guards with `id`/`nodeId` projection, plumbing `nodeId` through the Swift bridge request path, adding runtime regression tests, and updating the changelog.

## Best Possible Solution

Keep the PR as the active fix path, adjust the raw precheck so it only short-circuits snapshots whose base64 expansion alone cannot fit or remove it in favor of the projected frame check, then land the hardened implementation with the linked regression coverage.

## Reproduction Assessment

Yes. Current-main source shows the original fallback and missing guard, and the PR regression can be reproduced with an injected 19,660,100-byte snapshot payload plus short ids: it fits under the 25 MiB serialized frame budget but is rejected by the branch's raw precheck.

## Solution Assessment

No, not yet. The overall hardening direction is narrow and maintainable, but the raw precheck should not subtract envelope reserve before the authoritative projected-frame check runs for near-limit payloads.
