## Summary

Keep open. Current main has substantial legacy OpenAI Codex migration, auth-profile, and session cleanup coverage, but it still does not fully settle the reported mixed-provider upgrade case where canonical openai/gpt-5.5 can select provider-level openai routing that is not the ChatGPT subscription backend; the closest remaining work is split across open migration-policy and route-visibility items that need maintainer direction.

## What This Changes

The issue requests safer handling and diagnostics for OpenClaw 2026.6.10 upgrades where openai-codex subscription-backed GPT routes are canonicalized to openai/gpt-5.5 while provider-level openai still points elsewhere.

## Best Possible Solution

Adopt one maintainer-approved migration policy for unsafe legacy OpenAI Codex provider/auth/session state, likely through https://github.com/openclaw/openclaw/pull/93180, and route remaining display diagnostics through the existing route-visibility work without adding steady-state legacy runtime shims.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live Windows multi-agent run; source inspection shows the mixed-provider migration and missing diagnostic paths are plausible on current main, but the exact OAuth upgrade incident was not replayed.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the migration policy: the shared migration-plan PR is the strongest current direction, while route-display work covers only part of the operator confusion.
