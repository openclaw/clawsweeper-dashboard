## What This Changes

The issue reports that after OpenClaw 2026.6.10 provider unification, `openai/gpt-5.5` agent routing can fail when provider `openai` still points at a non-subscription backend while the subscription route remains under legacy `openai-codex`.

## Best Possible Solution

Keep canonical `openai/gpt-*` Codex routing and add a focused doctor/status diagnostic for configured `openai` provider transport/auth mismatches plus stale fallback/session pins.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection supports the mixed-provider path, but I did not replay the reported Windows multi-agent OAuth upgrade end to end.

## Solution Assessment

Is this the best way to solve the issue? The durable direction is to preserve canonical `openai/gpt-*` routing and improve doctor/status diagnostics; switching users back to durable `openai-codex/*` routing would conflict with current repository policy.
