## Summary

Keep open. Current main has meaningful migration, warning, SQLite auth, and stale-session repair coverage, but it still does not clearly close the reported mixed-provider upgrade case where canonical `openai/gpt-5.5` is selected while provider-level `openai` routing/auth points at the wrong backend.

## What This Changes

The issue requests safer handling and diagnostics for OpenAI provider unification after `openai-codex/*` routes are canonicalized to `openai/gpt-5.5`.

## Best Possible Solution

Keep canonical `openai/gpt-*` routing, then add or land one maintainer-approved diagnostic/migration path that detects mixed OpenAI backend/auth/session state without reintroducing unsupported live `openai-codex/*` routing by accident.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection supports the missing-diagnostic path for mixed canonical `openai` and legacy subscription-backed state, but I did not replay the reported Windows multi-agent OAuth upgrade end to end.

## Solution Assessment

Unclear until maintainers choose the intended product boundary. The best direction is a focused doctor/status migration diagnostic that preserves canonical OpenAI routing; switching users back to durable live `openai-codex/*` routing would conflict with current policy and docs.
