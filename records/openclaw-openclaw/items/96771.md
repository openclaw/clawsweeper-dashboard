## Summary

Keep open. Current main has meaningful legacy `openai-codex` migration, auth-profile repair, and stale-session cleanup coverage, but I did not find a current doctor/status check that fully covers the reported mixed-provider case where canonical `openai/gpt-5.5` is selected while provider-level `openai` routing still points at a non-subscription backend.

## What This Changes

Request better migration/doctor/status diagnostics for `openai/gpt-5.5` after `openai-codex` provider unification when provider-level routing, auth storage, or session overrides make the selected route resolve unexpectedly.

## Best Possible Solution

Keep canonical `openai/gpt-*` Codex routing, then add a focused doctor/status diagnostic for mixed `openai` and legacy subscription backend state plus stale fallback/session pins instead of reintroducing live `openai-codex` routing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection supports the missing-diagnostic path for mixed `openai` and legacy subscription-backed provider state, but I did not replay the reported Windows multi-agent OAuth upgrade end to end.

## Solution Assessment

Is this the best way to solve the issue? The durable direction is to preserve canonical `openai/gpt-*` routing and improve doctor/status diagnostics; switching users back to durable `openai-codex/*` routing would conflict with current repository policy and docs.
