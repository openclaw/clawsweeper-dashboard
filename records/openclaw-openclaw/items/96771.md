## Summary

Keep open. Current main has meaningful legacy `openai-codex` migration, stale override warnings, SQLite auth storage, and status-session visibility, but it still does not fully cover the reported 2026.6.10 mixed-provider case where canonical `openai/gpt-5.5` resolves through a provider-level `openai` backend that is not the subscription/Codex backend.

## What This Changes

The issue requests upstream diagnostics and recovery tooling for provider-unification cases where `openai/gpt-5.5` is selected but provider-level `openai` routing, auth storage, or session overrides still point at the wrong backend or fallback.

## Best Possible Solution

Keep canonical `openai/gpt-*` Codex routing, then add a maintainer-scoped doctor/status diagnostic that reports the resolved model, provider backend/API, auth profile source, and session override state without reviving live `openai-codex/*` routing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection supports the missing-diagnostic path for mixed `openai` and legacy subscription-backed provider state, but I did not replay the reported Windows multi-agent OAuth upgrade end to end.

## Solution Assessment

Is this the best way to solve the issue? The best direction is to preserve canonical `openai/gpt-*` routing and improve doctor/status route diagnostics; switching users back to durable `openai-codex/*` routing would conflict with current docs and repository policy.
