## Summary

Keep open. Current main still does not extract legacy object-shaped `.to` results, and this PR is a plausible compatibility fix, but the branch regresses existing string normalization and lacks contributor-owned real behavior proof.

## What This Changes

The branch adds a legacy `{ ok, to }` compatibility shim in outbound target normalization, resolver/send regression tests, an oauth test mock, and `.secrets.baseline` line metadata updates.

## Best Possible Solution

Port the compatibility shim onto current main so all selected target candidates flow through `normalizeOptionalString`, keep focused regression tests, and merge only after redacted real CLI/plugin send proof is attached.

## Reproduction Assessment

Yes, at source level for the remaining compatibility gap: current main discards object-shaped normalizer results instead of extracting `.to`. The exact old `to required` failure path appears partially mitigated on current main because non-string results now coerce to undefined and fall back to raw input.

## Solution Assessment

No as written. A narrow compatibility shim is the right boundary, but the PR must preserve the existing `normalizeOptionalString` trimming and blank rejection behavior when selecting either a string result or legacy `.to`.
