## Summary

Current main still has the string-only outbound normalizer, so the compatibility fix is still useful, but this PR should not merge as written because it changes existing trim/blank rejection semantics and lacks after-fix real behavior proof.

## What This Changes

The PR adds a runtime compatibility path for legacy object-shaped channel `normalizeTarget` results, plus resolver/send regression tests and secret-baseline line updates.

## Best Possible Solution

Keep the compatibility shim at `target-normalization`, but run every selected candidate through `normalizeOptionalString`, retain focused resolver/core-send coverage, and require redacted real CLI/plugin proof before merge.

## Reproduction Assessment

Yes at source level: current main rejects object-shaped `normalizeTarget` results through `normalizeOptionalString` and never reads `to`. I did not establish a live CLI `ToolInputError: to required` reproduction in this read-only review.

## Solution Assessment

No as written. A narrow compatibility shim is the right boundary, but the implementation must preserve existing string normalization and still needs real after-fix behavior proof.
