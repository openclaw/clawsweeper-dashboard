## Summary

Keep open: current main still lacks the requested legacy object-shaped normalizer tolerance, but the PR is not merge-ready because the shim changes existing string normalization behavior and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR adds a runtime compatibility path that reads `to` from legacy `{ ok, to }` target normalizer results, adds outbound resolver/send regression coverage, and updates secret-baseline line offsets.

## Best Possible Solution

Keep a small compatibility shim in `target-normalization`, normalize every selected string through the existing helper, place regression coverage in the current outbound test layout, and require redacted real CLI/plugin proof before merge.

## Reproduction Assessment

Yes at source level: current main feeds a runtime object-shaped normalizer result into `normalizeOptionalString`, which rejects non-strings and cannot extract `to`. I did not run a live CLI send with an installed legacy plugin in this read-only review.

## Solution Assessment

No as written. A runtime shim in `target-normalization` is the right narrow boundary, but the PR must preserve existing trim/blank rejection for string results and be rebased into the current test layout before it is merge-ready.
