## Summary

Keep open: current main and the latest release still ignore `.to` on object-shaped `normalizeTarget` results, so the compatibility gap is not implemented on main. This PR is not merge-ready as written because it lacks contributor-owned real behavior proof and the shim must preserve current string trimming/blank rejection when ported.

## What This Changes

The PR adds a legacy `{ ok, to }` target-normalizer compatibility shim plus resolver and send-path regression tests for outbound message sends.

## Best Possible Solution

Port the shim onto current `normalizeTargetForProvider`, pass the selected string candidate through `normalizeOptionalString`, keep focused resolver/send coverage, and merge only after redacted real CLI or plugin proof is added.

## Reproduction Assessment

Yes, at source level for the compatibility gap: current main treats object-shaped normalizer results as non-strings and never reads `.to`. I did not run a live CLI reproduction of the exact old error in this read-only review.

## Solution Assessment
