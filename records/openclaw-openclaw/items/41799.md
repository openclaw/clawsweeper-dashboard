Keep this PR open because the compatibility bug is still real on current main, but the proposed branch is stale and should not merge as-is. ## What This Changes

The PR adds outbound target-normalization compatibility for legacy channel plugins that return `{ ok, to }` from `messaging.normalizeTarget`, plus resolver and send-path regression coverage and incidental test/baseline changes.

## Best Possible Solution

Land a narrow rebased compatibility patch in `src/infra/outbound/target-normalization.ts` that treats the normalizer result as `unknown`, extracts only a trimmed nonblank string `to` from legacy object results, and otherwise preserves existing fallback/validation behavior. Put coverage in `target-normalization.test.ts` plus one current split send-path regression, without changing the SDK contract, QQ-specific parsing, or secret-scan baseline metadata.

## Reproduction Assessment

Yes. A high-confidence source-traced reproduction is to register a channel plugin whose `messaging.normalizeTarget` returns `{ ok: true, to: "qqbot:c2c:..." }` and call `resolveMessagingTarget` or `runMessageAction("send")`; current main drops that object before the send handler requires `to`.

## Solution Assessment

No, not as currently proposed. The production shim is the right direction, but the PR should be rebased onto current main, moved into the current split tests, and stripped of unrelated baseline/mock churn before it is the maintainable fix.
