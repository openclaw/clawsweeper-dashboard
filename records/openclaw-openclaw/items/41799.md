## Summary

Keep open: current main still has the legacy-object target-normalizer failure path, but this PR should not merge as written because the shim bypasses current string cleanup and contributor-owned real behavior proof is still missing.

## What This Changes

The PR adds a runtime compatibility path for outbound target normalization when a channel plugin returns legacy `{ ok, to }` results, plus resolver/send regression tests and secret-baseline line updates.

## Best Possible Solution

Port a narrow runtime-only legacy object extraction shim into `src/infra/outbound/target-normalization.ts`, feed the selected value through `normalizeOptionalString`, keep focused resolver/send coverage, and merge only after redacted real CLI/plugin send proof is attached.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a plugin normalizer returning `{ ok: true, to: "qqbot:..." }` is passed to `normalizeOptionalString`, which rejects non-strings before `.to` can be used.

## Solution Assessment

No as written. A compatibility shim in target normalization is the right boundary, but the selected string should still be normalized through the existing helper and the PR needs real behavior proof before merge.
