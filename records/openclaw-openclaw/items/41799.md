## Summary

Keep open: current main still does not extract `.to` from legacy object-shaped target normalizer results, but this PR needs a small correctness fix and external real behavior proof before merge.

## What This Changes

The PR adds a compatibility shim for legacy `{ ok, to }` channel target normalizer results, resolver/send regression tests, and secret-baseline line-number updates.

## Best Possible Solution

Port the compatibility shim onto current `target-normalization`, run selected candidates through `normalizeOptionalString`, keep focused resolver/core-send coverage, and merge only after redacted real CLI/plugin proof is added.

## Reproduction Assessment

Yes, at source level for the remaining compatibility gap: current main treats object-shaped normalizer returns as non-strings and never reads `.to`. I did not establish a live current-main CLI reproduction of the exact old error.

## Solution Assessment

No as written. A narrow compatibility shim is the right boundary, but the PR must preserve current string normalization semantics and still needs after-fix real behavior proof.
