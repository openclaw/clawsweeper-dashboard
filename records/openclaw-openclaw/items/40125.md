## Summary

Keep open. Current main has preflight compaction, memory flush, post-overflow recovery copy, `/context`, and a Control UI high-context notice, but it still lacks the requested generic pre-provider chat warning with one-time warning/urgent state for persistent channel sessions.

## What This Changes

Request proactive context-window pressure warnings in persistent chat sessions, with `/reset` or `/new` recovery guidance before overflow.

## Best Possible Solution

Add a generic auto-reply/session UX path that evaluates fresh context usage before provider invocation, persists one-time warning and urgent threshold state, and sends concise persistent-chat guidance naming `/reset` and `/new` without making it Telegram-specific.

## Reproduction Assessment

Not applicable as a traditional bug reproduction; this is a feature gap. Source inspection on current main gives a high-confidence verification path: token/context data exists, but there is no persisted threshold-warning state or outbound warning before the provider call.

## Solution Assessment

Yes, the requested generic session-pipeline warning is the narrowest maintainable direction. It should reuse existing token accounting and delivery hooks rather than adding Telegram-only transport logic or changing provider failover policy.
