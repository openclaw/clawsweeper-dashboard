## Summary

Keep open. Current main has context accounting, preflight compaction, mid-turn context-pressure recovery, post-overflow recovery copy, `/context` reporting, and a browser Control UI notice, but it still lacks the requested per-session, one-time proactive warning delivered into persistent chat before provider invocation.

## What This Changes

The issue asks OpenClaw to warn persistent chat users before a session reaches the active model context limit and to provide `/reset` or `/new` recovery guidance.

## Best Possible Solution

Add a generic auto-reply/session feature that evaluates fresh context usage before provider invocation, persists warning and urgent threshold state on the session, and sends concise persistent-chat guidance naming `/reset` and `/new`; keep it channel-agnostic and separate from model failover policy.

## Reproduction Assessment

Not applicable as a traditional bug reproduction because this is a feature gap. The high-confidence verification path is source-based: current main has context/token data and delivery primitives, but no persisted warning state or outbound threshold warning path before provider invocation.

## Solution Assessment

Yes. The requested generic session-pipeline warning is the narrowest maintainable direction; implementing it as Telegram-only transport logic or as fallback-model policy would miss other persistent chat surfaces and conflate context overflow with transient provider errors.

## Repair Work Prompt

Implement proactive context-pressure warnings for persistent chat sessions for #40125. Add a generic auto-reply/session feature, not Telegram-specific logic: before provider invocation on normal non-heartbeat turns, estimate/project current prompt/context usage against the active model context window using existing session token/context primitives. Persist per-session warning state for at least warning and urgent thresholds so each threshold is announced once, and clear that state on `/new` and `/reset` and when the session is reset/rotated as appropriate. Send concise channel-generic guidance through the existing reply/block-reply delivery path before the model call; the warning should name `/reset` to clear current session history and `/new` to start fresh. Reuse existing compaction and context accounting helpers where possible; do not change model fallback semantics, do not make this Telegram-only, and do not add a broad plugin API. Update focused tests and user-facing docs/changelog if the behavior is user-facing.
