## Summary

Current main still has the reported mismatch: the read-only doctor warning path emits a blanket `openai-codex/*` rewrite warning, while the repair path already has separate Codex-plugin and usable-OAuth checks. Provider docs also describe `openai-codex/*` as an explicit PI Codex OAuth route, so this should stay open for a focused diagnostic fix.

## What This Changes

The issue asks for `openclaw doctor` read-only warnings about `openai-codex/*` model refs to account for Codex OAuth auth profiles before suggesting `openai/*` rewrites.

## Best Possible Solution

Make the read-only doctor diagnostic route-aware: preserve the legacy rewrite warning when there is no usable Codex OAuth path, but avoid or reword the scary `should become` message when the current `openai-codex/*` PI OAuth route is valid and intentional.

## Reproduction Assessment

Yes, from source inspection: `collectCodexRouteWarnings` emits the rewrite warning for any collected `openai-codex/*` ref and has no auth-profile branch. I did not run doctor live because this review is read-only and must not mutate local state.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing Codex readiness/OAuth eligibility logic in the warning path and update focused tests and docs wording. The fix should not change `doctor --fix` migration behavior unless maintainers explicitly decide to change that policy too.
