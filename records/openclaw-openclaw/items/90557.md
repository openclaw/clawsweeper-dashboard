## Label Justifications

- `P1`: The issue reports a released Telegram outbound regression that blocks generated HTML attachment delivery in a security-sensitive host-read path.
- `impact:security`: The affected behavior is the trusted local HTML host-read gate that controls which HTML files may be read and delivered.

## Summary

Keep open: current main and v2026.6.9 still trust generated host-read HTML only when the file being loaded is under the OpenClaw temp root, while the reported Telegram flow stages the file under media/outbound and reloads it there. The open closing PR at https://github.com/openclaw/openclaw/pull/90579 owns the repair path, so this issue should remain canonical until that PR or an equivalent fix lands.

## What This Changes

The issue requests preserving trusted generated host-read HTML through outbound media staging so Telegram can deliver staged HTML attachments instead of dropping them.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90579 or an equivalent provenance-preserving repair, then close this issue after the fix is merged and validated.

## Reproduction Assessment

Yes, at source level. Current main stages local reply media into media/outbound and Telegram later reloads that staged .html through a host-read gate that only trusts temp-root realpaths; I did not run a live Telegram repro in this read-only review.

Yes for keeping the issue open, but not for closing it yet. The best path is to review and land the linked PR or a focused replacement that preserves the trusted-HTML security boundary through staging.
