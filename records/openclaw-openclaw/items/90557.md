## Label Justifications

- `P1`: This is a released Telegram regression that can drop generated HTML attachments from real channel replies.
- `impact:message-loss`: The issue reports that Telegram delivery fails and no file or media-failure placeholder reaches chat.
- `impact:security`: The failing behavior is inside the host-read HTML trust boundary for local generated files.

## Summary

Keep open: current `main` still matches the reported failure path, because final-reply local media is staged into the outbound media store and the later host-read HTML load only trusts the current file path when it is under the OpenClaw temp root.

## What This Changes

Fix Telegram delivery of trusted generated HTML attachments after final-reply staging moves them outside the trusted temp root.

## Best Possible Solution

Carry trusted generated-HTML provenance through outbound staging, then add focused coverage for final MEDIA reply staging into Telegram delivery while keeping arbitrary host-local and staged HTML blocked.

## Reproduction Assessment

Yes. Source inspection corroborates the two-step path: final MEDIA normalization stages local media into `media/outbound`, and the later host-read HTML load rejects `.html` whose current realpath is not under the preferred temp root.

## Solution Assessment

Yes on the bug direction, but the safest fix is provenance-carrying rather than simply allowing every staged outbound HTML file. The regression test should exercise the staged final-reply path, not only direct `loadWebMedia` on the original temp file.
