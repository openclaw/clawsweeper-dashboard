## Summary

Keep this open: current main still has a source-backed WebChat ordering gap for merged-session history because the UI sorter uses only top-level numeric `timestamp`, while Gateway history can carry canonical transcript time in `__openclaw.recordTimestampMs`. The related WebChat ordering PRs cover live stream/tool and preamble ordering, but they do not currently own this merged-session metadata case.

## What This Changes

The issue reports WebChat rendering messages out of order after WebChat and WeChat share the same agent main session, even though session history API order is correct.

## Best Possible Solution

Make WebChat visible-time resolution consume canonical transcript metadata such as `__openclaw.recordTimestampMs`, add a merged-session regression test, and coordinate that rule with the active WebChat ordering PRs.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection gives a medium-confidence path: Gateway history preserves canonical record time in metadata while WebChat render sorting only consumes top-level numeric timestamps.

## Solution Assessment

Yes, the likely best fix is a narrow Control UI timestamp-normalization/render-order repair. Gateway history already exposes the needed metadata, so a broader Gateway or WeChat plugin rewrite would be the wrong first fix.
