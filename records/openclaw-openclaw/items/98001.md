## Summary

Keep this open: current main still has a source-backed WebChat ordering gap for transcript history because the renderer sorts visible history by top-level numeric `timestamp`, while Gateway history can carry the canonical transcript record time in `__openclaw.recordTimestampMs`. The open WebChat ordering PRs overlap the same renderer area but do not specifically own this merged-session history-metadata case.

## What This Changes

The issue reports WebChat rendering user and assistant messages out of order after WebChat and WeChat traffic are merged into the same agent main session.

## Best Possible Solution

Make WebChat render ordering use the same canonical transcript time as Gateway history, especially `__openclaw.recordTimestampMs`, and cover the merged-session case with focused regression tests.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection gives a medium-confidence path: Gateway history preserves canonical record time in metadata, while WebChat render sorting only consumes top-level numeric timestamps.

## Solution Assessment

Yes, the likely best fix is a narrow Control UI timestamp-normalization/render-order repair. Gateway history already exposes the needed metadata, so a broader Gateway or WeChat plugin rewrite would be the wrong first fix.
