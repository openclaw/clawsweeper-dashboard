## Summary

Keep this open: current main and v2026.6.10 have the general WebChat visible-time sorter, but source inspection still shows a plausible merged-session gap because the WebChat renderer sorts only top-level numeric message timestamps while Gateway history can carry canonical transcript time in `__openclaw.recordTimestampMs`.

## What This Changes

The issue reports WebChat rendering messages out of order after WebChat and WeChat messages are merged into the same agent main session.

## Best Possible Solution

Make WebChat render ordering consume canonical transcript time, including `__openclaw.recordTimestampMs` and parseable timestamp strings where appropriate, with a focused merged-session regression test.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection gives a medium-confidence path: Gateway history preserves canonical record time in metadata, while WebChat render sorting only consumes top-level numeric timestamps.

## Solution Assessment

Yes, the likely best fix is a narrow Control UI timestamp-normalization/render-order repair. Gateway history already exposes the needed metadata, so a broader Gateway or WeChat plugin rewrite would be the wrong first fix.
