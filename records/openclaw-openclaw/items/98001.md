## Summary

Keep this open: current main and v2026.6.10 still have a source-backed WebChat history ordering gap where visible sorting reads top-level numeric `timestamp`, while Gateway transcript hydration carries canonical record time in `__openclaw.recordTimestampMs`. The nearby open WebChat ordering PR covers live tool-card causal order, not this merged-session history metadata case.

## What This Changes

Report that WebChat renders messages out of order after WebChat and openclaw-weixin sessions merge into the same agent main session.

## Best Possible Solution

Make WebChat render ordering consume canonical transcript time such as `__openclaw.recordTimestampMs` for history messages, add a focused merged-session regression, and keep the live tool causal-order rule consistent with the adjacent open PR.

## Reproduction Assessment

No live high-confidence reproduction has been established. Source inspection gives a medium-confidence path: Gateway history preserves canonical record time in metadata, while WebChat render sorting only consumes top-level numeric timestamps.

## Solution Assessment

Yes, the likely best fix is a narrow Control UI timestamp-normalization/render-order repair. Gateway history already exposes the needed metadata, so a broader Gateway or WeChat plugin rewrite would be the wrong first fix.
