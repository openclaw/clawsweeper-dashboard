## Summary

Keep this open: current main still has a source-backed WebChat ordering gap for merged-session history because the renderer sorts history rows by top-level numeric `timestamp`, while Gateway history can carry canonical record time in `__openclaw.recordTimestampMs`. The related WebChat tool-order PR overlaps the sorter but does not own this merged WebChat/WeChat history case.

## What This Changes

The issue reports that merged WebChat and WeChat sessions can render messages out of order in WebChat even though the history API returns the correct order.

## Best Possible Solution

Make WebChat render ordering consume canonical transcript time where available, then cover the merged-session history case with a focused Control UI regression test.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a medium-confidence path: WebChat sorting ignores canonical record-time metadata that Gateway history can carry.

## Solution Assessment

Yes, likely. The narrowest maintainable fix is in the Control UI timestamp normalization/render-ordering path, not a WeChat plugin or Gateway rewrite.
