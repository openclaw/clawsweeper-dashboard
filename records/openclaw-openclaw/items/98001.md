## Summary

Keep this open: current source and the latest release already have the general visible-time sorter, but the reported released-build behavior still has a plausible source-backed gap because WebChat render ordering only uses top-level numeric timestamps while session history can carry canonical transcript time in metadata.

## What This Changes

The issue reports WebChat rendering user and assistant messages out of order after WebChat and openclaw-weixin share the same agent main session.

## Best Possible Solution

Make WebChat render ordering use canonical transcript time, including __openclaw.recordTimestampMs and parseable timestamp strings, then add a focused merged-session regression test.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection gives a medium-confidence path: transcript history can preserve canonical record time in metadata while the WebChat renderer sorts only top-level numeric timestamps.

## Solution Assessment

Yes, the likely best fix is a narrow Control UI timestamp-coercion/render-order repair. The gateway history contract should not need a rewrite if the UI consumes the canonical time it already receives.
