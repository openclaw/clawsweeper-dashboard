## Summary

Keep this open: current main still has a source-backed WebChat ordering gap because the renderer sorts history messages by top-level numeric `timestamp`, while Gateway history can carry the canonical transcript record time in `__openclaw.recordTimestampMs`. The adjacent merged and open WebChat ordering PRs cover numeric live/history ordering and live tool causal ordering, but neither currently owns this merged-session transcript metadata case.

## What This Changes

The issue reports that after WebChat and WeChat sessions merge into the same agent main session, WebChat sometimes renders a user message after its corresponding assistant reply.

## Best Possible Solution

Make WebChat render ordering consume the same canonical transcript time as Gateway history, especially `__openclaw.recordTimestampMs`, and cover the merged-session case with a focused regression test.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection gives a medium-confidence path: Gateway history preserves canonical record time in metadata, while WebChat render sorting only consumes top-level numeric timestamps.

## Solution Assessment

Yes, the likely best fix is a narrow Control UI timestamp-normalization/render-order repair. Gateway history already exposes the needed metadata, so a broader Gateway or WeChat plugin rewrite would be the wrong first fix.
