## Summary

Keep open: current main and the latest release still expose TUI delivery as a launch-time/runtime choice with no config-backed default, and the related config-default PRs are closed unmerged. The adjacent stuck-spinner fix was merged, but it explicitly left this delivery-default/config ergonomics work out of scope.

## What This Changes

Allow `openclaw tui` users to set a config-backed default for `--deliver` so replies can be delivered without passing the flag every run.

## Best Possible Solution

Define an approved TUI delivery default config contract with schema, exported types, help/docs, focused tests, and explicit CLI override semantics.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves omitted `--deliver` to false, sends that static value with each TUI message, and exposes no config key for changing the default. I did not run a live TUI session in this read-only review.

## Solution Assessment

Unclear. A config-backed default matches the requested direction, but maintainers still need to choose the public namespace and flag-over-config semantics before this becomes a mechanical fix.
