## Summary

Keep open: current main and the latest release still have `openclaw tui --deliver` as an invocation-time flag with no config-backed default, while the direct implementation PRs are closed unmerged. The merged adjacent TUI spinner/abort work addressed the lifecycle failure around delivery-off runs, but it explicitly left this config/default request as separate product work.

## What This Changes

The issue requests a config-backed default for the TUI `--deliver` flag so users do not need to pass it on every `openclaw tui` invocation.

## Best Possible Solution

Define an approved TUI delivery-default config contract with schema, exported types, help/docs, focused tests, and explicit CLI override semantics that preserve existing behavior unless users opt in.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves omitted `--deliver` to false, forwards that value to every TUI send, and has no config key that can change the default. I did not run a live TUI session in this read-only review.

## Solution Assessment

Unclear. A config-backed default matches the requested direction, but maintainers still need to choose the public namespace, override semantics, and upgrade behavior before this is a mechanical fix.
