## Summary

Keep open: current main still lacks any config-backed default for TUI delivery. The prior config PRs are closed unmerged, while the later merged TUI lifecycle fix addressed the stuck-spinner path and explicitly left this delivery-default request as separate.

## What This Changes

Allow `openclaw tui` to read a config-backed default for the `--deliver` flag so users do not need to pass it every run.

## Best Possible Solution

Add a documented config-backed TUI delivery default under an approved namespace, with explicit CLI override precedence, regression tests, and docs/settings alignment.

## Reproduction Assessment

Yes, for the missing requested feature. Source inspection on current main shows the CLI still defaults `--deliver` to false, forwards a boolean into `runTui`, and has no config type/schema key for a TUI delivery default; I did not run a live TUI session in this read-only review.

## Solution Assessment

Unclear. A config-backed default is plausible and narrow, but maintainers need to choose the namespace and override semantics before treating a patch as mechanical.
