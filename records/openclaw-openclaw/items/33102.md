## Summary

Keep open: current main still has no config-backed TUI delivery default. The CLI/runtime hard-default omitted delivery to false, strict config/docs expose no `tui.deliver` or equivalent key, and the linked implementation PRs were closed unmerged.

## What This Changes

Allow `openclaw tui` delivery to default from OpenClaw config so users do not have to pass `--deliver` every time.

## Best Possible Solution

Add a narrow public TUI config contract, preferably `tui.deliver?: boolean`, and resolve delivery as explicit CLI choice first, then config, then the current false fallback, with docs/schema/tests/changelog updated together.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` with no `--deliver` becomes `deliver: false`, `runTui` keeps that fallback, and the strict config schema has no supported delivery default key.

## Solution Assessment

Yes. The requested configurable default is the narrow maintainable direction; flipping delivery on globally or relying on shell aliases would either change broader behavior or fail to provide a public config contract.
