## Summary

Keep open: current main and the latest release still make TUI reply delivery flag-only, with no config-backed default. The adjacent stuck-spinner/abort work was handled separately; this remaining request needs a public config contract decision.

## What This Changes

Allow `openclaw tui` users to configure the default value for `--deliver` instead of passing the flag on every invocation.

## Best Possible Solution

Define an approved opt-in TUI delivery-default config key with schema/types, config help, docs, focused tests, and explicit CLI override semantics that preserve off-by-default behavior unless users configure it.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves omitted `--deliver` to `false`, forwards that value into TUI sends, and has no config key that can change the default. I did not run a live TUI session because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public config contract. A config-backed opt-in default matches the request, but the namespace and CLI override semantics are product/API decisions rather than a strict bug fix.
