## Summary

Keep open: current main still has no configurable TUI delivery default, and the related fix PRs were closed unmerged. The request is valid, but adding a new public config/default surface needs maintainer product direction before repair automation.

## What This Changes

The issue asks for a configurable default for the TUI `--deliver` behavior so users do not need to pass the flag on every invocation.

## Best Possible Solution

A maintainer-approved TUI delivery-default model should define the canonical config path, preserve explicit CLI overrides, update schema/help/docs, and add focused tests for default, opt-in, and opt-out behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still defaults `--deliver` to false, forwards only the CLI boolean into TUI runtime, and the canonical config shape has no `tui.deliver` key. I did not run the interactive TUI because the missing config/default behavior is directly source-provable in this read-only review.

## Solution Assessment

Unclear. A config-backed default is plausible, but it adds public configuration surface and touches prior default-delivery intent, so maintainers should first decide whether `tui.deliver`, `cli.tui.deliver`, a runtime-persisted setting, or the existing `--deliver` control is the desired product shape.
