## Summary

Keep open: current main still lacks a config-backed TUI delivery default, and the related stuck-spinner fix was merged separately while leaving this delivery-default request out of scope.

## What This Changes

The issue requests a user-configurable default for `openclaw tui --deliver` so users do not have to pass the flag every time.

## Best Possible Solution

Add an approved config-backed TUI delivery default with exported types, schema/help/labels, docs, focused tests, and explicit CLI override semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` defaults `--deliver` to false, passes a boolean into `runTui`, and the config type/schema/docs have no TUI delivery default key.

## Solution Assessment

Unclear. A config-backed default is a plausible narrow solution, but maintainers need to choose the public namespace and CLI/config precedence before this becomes mechanical.
