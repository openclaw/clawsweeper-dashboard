## Label Justifications

- `P2`: This is a normal-priority user-facing TUI configuration improvement with clear source evidence and limited blast radius.
- `impact:message-loss`: The requested default affects whether TUI-originated assistant replies are delivered through configured channels without repeatedly passing `--deliver`.

## Summary

Keep open: current main and the latest release still make TUI delivery flag-only with no config-backed default, while the adjacent stuck-spinner failure is a separate closed item. This needs a maintained public config namespace and CLI precedence contract before implementation.

## What This Changes

Add a config-backed default for `openclaw tui --deliver` so users can opt into delivery without passing the flag every run.

## Best Possible Solution

Define an approved opt-in TUI delivery-default config key with schema/types, docs, help text, focused tests, and explicit CLI override semantics that preserve off-by-default behavior unless users opt in.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves omitted `--deliver` to `false`, forwards that value into TUI sends, and has no config key that can change the default. I did not run a live TUI session because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public config contract. A config-backed default matches the request, but the namespace, `--deliver`/`--no-deliver` precedence, and docs contract are product/API decisions rather than a mechanical bug fix.
