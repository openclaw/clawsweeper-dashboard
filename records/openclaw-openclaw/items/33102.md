## Summary

Keep open. Current main still hard-defaults omitted TUI delivery to false, has no public `tui.deliver` or equivalent config contract in types/schema/help/docs, and the earlier implementation PRs for this request were closed unmerged. The later streaming watchdog work appears to reduce the stuck-spinner symptom reported in the follow-up, but it does not provide a configurable default for `--deliver`.

## What This Changes

The issue requests a public OpenClaw config option that can set the default value used by `openclaw tui --deliver` when the CLI flag is omitted.

## Best Possible Solution

Add a narrow public TUI config contract, preferably `tui.deliver?: boolean`, and resolve delivery as explicit CLI choice first, then config, then the current false fallback. Keep the runtime send semantics unchanged apart from the new default source, add explicit false override support, and update config docs/metadata plus focused regression tests.

## Reproduction Assessment

Yes. The high-confidence source reproduction is `openclaw tui` with no `--deliver`: Commander supplies false, `runTui` keeps false, and `sendChat` receives `deliver: false`; adding `tui.deliver` to config is also unsupported by the current strict config schema.

## Solution Assessment

Yes, the requested configurable default is the narrow maintainable solution. A shell alias does not solve config-driven behavior, and flipping the product default globally would change delivery policy more broadly than this issue requires.

## Repair Work Prompt

Fix openclaw/openclaw issue #33102 with a narrow public TUI delivery-default config contract. Current behavior: `openclaw tui` masks omitted `--deliver` as false in `src/cli/tui-cli.ts`, `runTui` resolves `deliverDefault` as `opts.deliver ?? false`, and `src/tui/tui-command-handlers.ts` passes that fixed value to `client.sendChat`. Add optional config support, preferably `tui.deliver?: boolean`, so omitted CLI input resolves to `config.tui?.deliver ?? false`; preserve absent-config behavior as false; preserve `--deliver` as explicit true; add `--no-deliver` as explicit false; do not flip the global default to true; do not change unrelated delivery/session routing semantics. Update exported config types, strict zod schema, schema help/labels, public docs (`docs/cli/tui.md`, `docs/web/tui.md`, and config reference if generated/manual coverage requires it), focused tests for config default and CLI override behavior, and a single-line CHANGELOG entry under the active release with appropriate thanks to the reporter. Preserve local embedded TUI mode and the `/deliver`/Settings runtime toggle behavior. Related context: #33190, #33198, #33208 were prior unmerged attempts; #66991 documents the stuck-spinner symptom when delivery is off but should not broaden this patch unless a focused regression assertion is needed.
