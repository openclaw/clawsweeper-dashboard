## Summary

Keep open: current main still treats TUI delivery as a launch-only flag and exposes no public config key for a default; the related spinner/abort fix addressed an adjacent lifecycle bug but explicitly left this config request unresolved.

## What This Changes

The issue requests a config-backed default for `openclaw tui --deliver` so users do not need to pass the flag on every TUI launch.

## Best Possible Solution

Add an approved config-backed TUI delivery default with schema, exported types, help/labels, docs, focused tests, and explicit flag-over-config precedence.

## Reproduction Assessment

Yes for the current behavior: source inspection on current main shows omitted `--deliver` becomes `false`, that value is passed to `sendChat`, and config schema/types/docs expose no delivery default. I did not run the interactive TUI in this read-only review.

## Solution Assessment

Unclear. A config-backed default is plausible and matches the request, but maintainers still need to choose the public namespace and CLI/config precedence before implementation is mechanical.
