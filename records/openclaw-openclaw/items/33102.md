## Summary

Keep open. Current main still has no config-backed TUI delivery default: the CLI/runtime path hard-resolves delivery to false when omitted, the send path uses that fixed startup value, and the public config contract has no `tui.deliver` or equivalent. The related implementation PRs are closed unmerged, and the later stuck-spinner discussion reinforces the same delivery-default problem rather than replacing this request.

## What This Changes

The issue requests a public OpenClaw config option for the TUI `--deliver` default so users can enable reply delivery without passing `--deliver` on every invocation.

## Best Possible Solution

Add a narrow public TUI config contract, preferably root `tui.deliver?: boolean`, wire it through the config schema/help/docs surfaces, resolve startup delivery as explicit CLI value first then config then false, and keep existing absent-config behavior unchanged while adding an explicit false CLI override.

## Repair Work Prompt

Implement a narrow fix for openclaw/openclaw#33102. Current `openclaw tui` cannot use config for the delivery default. Add an optional public config key, preferably root `tui.deliver?: boolean`, and wire it through exported config types, zod schema, schema help/labels/generated docs metadata, CLI/TUI docs, and focused tests. Preserve absent-config behavior as false. Preserve `--deliver` as an explicit true override and add `--no-deliver` as an explicit false override; omitted CLI input must reach `runTui` as undefined so `runTui` can resolve `opts.deliver ?? config.tui?.deliver ?? false`. Keep normal TUI sends using the resolved default. Do not flip the global default to true, do not change gateway chat delivery semantics, do not broaden channel/provider delivery, and do not rely on the currently documented `/deliver` or settings toggle unless the PR deliberately implements and tests that runtime toggle too. Add tests for config true, config false/absent, CLI true override, and explicit false override. Update `CHANGELOG.md` with a single-line user-facing entry and appropriate Thanks attribution, never `@codex`.
