Keep open: current main and the latest release still lack any config-backed default for TUI delivery, and the direct contributor PRs for `tui.deliver` were closed unmerged. The remaining work is a real TUI ergonomics/message-visibility request, but it adds a new config/default surface and needs maintainer product approval before an automated fix lane should touch it.

## What This Changes

Requested support for configuring the default value of the TUI `--deliver` flag so users do not need to pass it on every invocation.

## Best Possible Solution

A maintainer should decide whether TUI delivery deserves a persisted config default, and if approved implement it through the canonical config schema, docs, and focused TUI CLI/runtime tests.

## Reproduction Assessment

Yes. Source inspection of current main shows `--deliver` defaults to false, `runTui` falls back to false, and the config schema/type has no `tui.deliver` field; I did not run the interactive TUI because the missing config path is directly source-provable.

## Solution Assessment

Unclear. A `tui.deliver` config default is a plausible narrow solution, but the repository’s config-surface policy means maintainers should first confirm that a persisted default is the desired product shape rather than a runtime-only setting or different UX change.
