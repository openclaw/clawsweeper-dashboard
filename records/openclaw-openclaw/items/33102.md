## Summary

Keep open: current main and the latest release still make TUI delivery flag-only with no config-backed default. The adjacent stuck-spinner failure was fixed separately, and the remaining work needs a public config namespace and CLI precedence decision before implementation.

## What This Changes

The issue asks for a config-backed default for `openclaw tui --deliver` so users do not need to pass the flag on every invocation.

## Best Possible Solution

Define an approved opt-in TUI delivery-default config key with schema/types, config help, docs, focused tests, and explicit CLI override semantics that preserve off-by-default behavior unless users configure it.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves omitted `--deliver` to `false`, forwards that value into TUI sends, and has no config key that can change the default. I did not run a live TUI session because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public config contract. A config-backed default matches the request, but the namespace and `--deliver` versus explicit opt-out precedence are product/API decisions rather than a strict bug fix.
