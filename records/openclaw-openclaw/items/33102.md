## Summary

Keep open: current main and v2026.6.8 still hard-code the TUI `--deliver` default to false and the config schema only accepts `tui.footer.showRemoteHost`; the linked stuck-spinner bug was fixed separately, and the direct config PRs closed unmerged.

## What This Changes

The issue requests a user-configurable default for TUI `--deliver` so users do not have to pass the flag on every run.

## Best Possible Solution

Decide the canonical persistent surface first, then implement it through config types, strict schema, help/docs, and focused TUI CLI/runtime tests with explicit CLI override behavior and no regression to the already-fixed stuck-spinner path.

## Reproduction Assessment

Yes, at source level: current main still declares `--deliver` with default `false`, forwards only the CLI boolean to `runTui`, and the strict config schema has no `tui.deliver` or `cli.tui.deliver` field. I did not run a live TUI session because the missing config surface is directly visible in source and docs.

## Solution Assessment

Unclear until maintainers choose the product contract. A raw `tui.deliver` key is plausible, but the best fix must account for the high config-surface bar, external delivery privacy semantics, CLI override behavior, and the current docs/source mismatch around `/deliver` and settings.
