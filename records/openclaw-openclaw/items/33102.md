## Summary

Keep open: current main still lacks a config-backed TUI delivery default. The related config PRs are closed unmerged, and the later merged abort-gap fix at https://github.com/openclaw/openclaw/pull/77199 addressed adjacent lifecycle behavior while explicitly leaving this request out of scope.

## What This Changes

Request adds a config setting so `openclaw tui --deliver` can default on without passing the flag on every invocation.

## Best Possible Solution

Add a documented config-backed TUI delivery default under a maintainer-approved namespace, with explicit CLI precedence, regression tests, and aligned TUI docs/settings behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` still defaults delivery from the CLI option to false, passes that through to `sendChat`, and has no config schema key for a TUI delivery default; I did not run a live TUI session in this read-only review.

## Solution Assessment

Unclear. A config option is plausible and narrow, but the exact namespace, CLI override semantics, and relationship to documented Settings or `/deliver` controls need maintainer direction before treating this as a mechanical fix.
