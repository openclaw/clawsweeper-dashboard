## Summary

Current main still has no configurable TUI delivery default: the CLI forces `false` when `--deliver` is omitted, and each send uses that captured launch value. The merged abort repair addresses a different pre-event gap and explicitly leaves this request out of scope, while both direct config PRs closed unmerged only for inactivity, so the issue remains necessary but needs a maintainer product decision on default and persistence ownership.

## What This Changes

Add a persistent user-selectable default for the TUI `--deliver` behavior so delivery need not be enabled on every launch.

## Best Possible Solution

Define delivery-on and non-delivery as coherent TUI modes, use one mutable delivery state throughout the process, and persist the preference through the canonical TUI preference owner; add a global config field only if maintainers intentionally want a machine-wide policy with explicit positive and negative CLI overrides.

## Reproduction Assessment

Yes for the issue as written: current main deterministically supplies `false` when `--deliver` is omitted and has no config-backed launch default. The later indefinite-spinner symptom is adjacent and should be validated separately from this feature gap.

## Solution Assessment

Unclear. Persisting the preference is reasonable, but the proposed new global config key is not yet proven better than a safer default or an existing/canonical TUI preference store, and maintainers must decide the intended non-delivery UX.
