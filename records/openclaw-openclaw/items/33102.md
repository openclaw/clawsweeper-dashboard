## Summary

Keep open: current main and the latest release still treat TUI delivery as a launch/runtime flag, and the config-backed default remains unresolved after the related config PR attempts closed unmerged.

## What This Changes

The issue asks for a config-backed default for `openclaw tui --deliver` so users do not have to pass the flag on every TUI launch.

## Best Possible Solution

Define an approved config-backed TUI delivery default with schema, types, help text, docs, focused tests, and explicit CLI precedence semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted `--deliver` becomes `false`, that value is passed into each TUI send, and the config schema/types/docs expose no delivery default.

## Solution Assessment

Unclear. A config-backed default matches the request, but maintainers still need to choose the public namespace and flag-over-config semantics before this becomes a mechanical fix.
