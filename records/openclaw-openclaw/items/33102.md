## Summary

Keep open: current main and the latest stable release still lack a `tui.deliver` config default, while the linked stuck-spinner bug was fixed separately and the two direct implementation PRs closed unmerged.

## What This Changes

The issue asks for a configurable default for `openclaw tui --deliver` so users can persist their preferred TUI delivery behavior.

## Best Possible Solution

Decide the product policy first; if persistent TUI delivery is wanted, add a canonical `tui.deliver` schema/help/docs surface with tri-state CLI override coverage and tests, otherwise document the current flag/settings-panel behavior as intentional.

## Reproduction Assessment

Yes at source level: current main still defaults `--deliver` to false, forwards only the CLI boolean into TUI sends, and has no `tui.deliver` config schema. I did not run a live TUI session because this review is read-only and the source path is sufficient for the missing-config claim.

## Solution Assessment

Unclear until maintainers decide the config policy. The requested config key is a plausible solution, but OpenClaw's config bar means the best fix should include tri-state `--deliver`/`--no-deliver` behavior, schema/help/docs alignment, and tests if the product direction is accepted.
