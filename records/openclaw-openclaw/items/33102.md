## Summary

Keep open: current main and the latest release still have no configurable TUI delivery default, and the related config PRs were closed unmerged. The adjacent stuck-spinner bug was fixed separately, but the requested config/default behavior remains a product/config-surface decision.

## What This Changes

Allow `openclaw tui` users to configure a persistent default for the `--deliver` flag instead of passing it on every launch.

## Best Possible Solution

Choose the supported TUI delivery preference surface, then implement one canonical default path with schema/types/help/docs/tests and explicit CLI override semantics.

## Reproduction Assessment

Yes, as a source-level reproduction: current main parses `--deliver` with default `false`, passes it into the TUI, and sends `deliverDefault` to `sendChat`; the config schema does not accept a delivery default key. I did not run a live TUI session because this cleanup review is read-only.

## Solution Assessment

Unclear. A config default is plausible, but maintainers still need to choose the best product shape, such as `tui.deliver`, `cli.tui.deliver`, a runtime settings preference, or keeping delivery explicitly per launch.
