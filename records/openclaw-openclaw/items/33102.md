## Summary

Keep open: current main and the latest release still make TUI reply delivery flag-only, with no config-backed default. The adjacent stuck-spinner/abort work was handled separately, so the remaining request needs maintainer approval for a public config key and CLI precedence contract.

## What This Changes

The issue requests a configurable default for `openclaw tui --deliver` so users can opt into TUI reply delivery without passing the flag every time.

## Best Possible Solution

Approve and implement an opt-in TUI delivery-default config key with schema/types, config help, docs, focused tests, and explicit CLI override semantics that preserve off-by-default behavior unless users configure it.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves omitted `--deliver` to `false`, forwards that value into TUI sends, and has no config key that can change the default. I did not run a live TUI session because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public config contract. A config-backed opt-in default matches the request, but the namespace and CLI override semantics are product/API decisions rather than a strict bug fix.
