- `impact:message-loss`: With the current default, assistant output can be stored without being visibly delivered in the TUI unless delivery is enabled.

## Summary

Keep open. Current main and the latest release still default TUI delivery off through the CLI and do not expose a `tui.deliver` config schema; the related config/default PRs closed unmerged, while the merged stuck-spinner fix explicitly left this ergonomics/config request as the canonical follow-up.

## What This Changes

The issue requests a persistent OpenClaw config default for TUI reply delivery so users do not need to pass `openclaw tui --deliver` every time.

## Best Possible Solution

Choose one canonical TUI delivery-default contract; if global config is approved, add `tui.deliver` through types, schema, labels/help, CLI override semantics, docs, and focused TUI/config tests, otherwise update docs and runtime UX to the chosen non-config path.

## Reproduction Assessment

Yes for source-level behavior: current CLI source still defaults `--deliver` to false and forwards `Boolean(opts.deliver)` into `runTui`. I did not run an interactive TUI session during this read-only review.

## Solution Assessment

Unclear as submitted: `tui.deliver` is plausible, but OpenClaw's config-surface policy means maintainers should first decide whether the persistent default belongs in global config, per-session state, or a different TUI setting path.
