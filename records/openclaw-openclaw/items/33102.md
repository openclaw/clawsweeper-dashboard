## Summary

Keep open: current main and `v2026.6.10` still default TUI delivery off and do not expose a `tui.deliver` config default; the related stuck-spinner failure was fixed separately, but the requested persistent default remains a product/config-surface decision.

## What This Changes

The issue requests a persistent OpenClaw config default for TUI `--deliver` behavior so users do not have to pass `openclaw tui --deliver` on every run.

## Best Possible Solution

Decide the accepted TUI delivery-default policy, then land one focused PR that updates schema/types/help/docs/tests and preserves explicit per-run override behavior, preferably without silently changing existing users' default delivery mode.

## Reproduction Assessment

Yes, at source level: current main and the latest release still default `--deliver` to false and do not define a `tui.deliver` config key. I did not run the TUI because this review was read-only, but the source path is direct.

## Solution Assessment

Unclear: a config default is a plausible small fix, but OpenClaw's config-surface bar is high and maintainers should first confirm the key, default, persistence model, and `--no-deliver` override semantics.
