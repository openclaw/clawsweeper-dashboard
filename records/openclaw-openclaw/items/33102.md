## Summary

Keep open: current `main` and the latest release still default TUI delivery off and do not expose a `tui.deliver` config key. The related config PRs were closed unmerged, while the merged stuck-spinner work fixed an adjacent lifecycle bug and explicitly left this default/config ergonomics request separate.

## What This Changes

The issue requests a persistent OpenClaw config default for `openclaw tui --deliver` so users do not need to pass the flag on each TUI launch.

## Best Possible Solution

Choose the canonical TUI delivery-default model first; if accepted, implement it through the current config schema/help/labels, CLI override semantics, TUI docs, and focused tests in one aligned change.

## Reproduction Assessment

Yes for the central source behavior: current `main` and `v2026.6.6` still default `--deliver` to false and do not accept a `tui.deliver` config key. I did not run a live TUI session in this read-only review.

## Solution Assessment

Unclear as a product choice: adding `tui.deliver` is a plausible fix, but the repo policy requires maintainers to approve new config/default surface first. A narrower implementation should follow only after that canonical direction is chosen.
