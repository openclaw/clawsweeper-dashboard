## Label Justifications

- `P2`: The missing default is a normal-priority TUI usability and delivery problem with a bounded CLI/config surface, not an emergency or urgent runtime outage.
- `impact:message-loss`: With delivery off by default, assistant replies can be stored without appearing where the user expects, which is a message-delivery visibility failure.

## Summary

Keep open: current main and latest v2026.6.8 still hard-code TUI delivery off unless `--deliver` is passed, and config/schema/docs only support TUI presentation settings. The requested fix would add a new config/default surface, so it needs maintainer product direction rather than an automated fix PR.

## What This Changes

The issue requests a persistent OpenClaw config default for `openclaw tui --deliver` so users do not need to pass the flag every run.

## Best Possible Solution

Decide the canonical persistent delivery-default owner, then implement one path with `--deliver` and `--no-deliver` override semantics, schema/help/docs alignment, and focused CLI/TUI tests.

## Reproduction Assessment

Yes. Current main source shows `--deliver` defaults to false, the CLI forwards only `Boolean(opts.deliver)`, and the config schema has no delivery-default field.

## Solution Assessment

Unclear. A config option is a plausible solution, but OpenClaw policy treats new config/default surfaces as high-bar product decisions, and a persisted TUI preference or changed built-in default may be the better owner.
