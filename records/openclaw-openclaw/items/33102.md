## Summary

Keep open: current main and the latest release still treat TUI delivery as a launch/runtime flag rather than a config-backed default. The related stuck-spinner bug is closed/merged separately, and both config-default PR attempts are closed unmerged, so the requested config behavior remains unresolved.

## What This Changes

Request to let `openclaw tui` use a config-backed default for `--deliver` so users do not have to pass the flag on every launch.

## Best Possible Solution

Define an approved config-backed TUI delivery default with schema/types/help/docs/tests and explicit CLI precedence before implementing it.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted `--deliver` becomes `false`, that value is passed into each TUI send, and the config schema/types/docs expose no delivery default.

## Solution Assessment

Unclear. A config-backed default is plausible, but maintainers still need to choose the public namespace and flag-over-config semantics before this becomes a mechanical fix.
