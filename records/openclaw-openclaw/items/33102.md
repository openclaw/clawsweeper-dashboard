## Summary

Keep open: current main and the latest release still make TUI delivery a launch flag only; related lifecycle fixes addressed the stuck-spinner path but did not add the requested config contract.

## What This Changes

Add a public config-backed default for `openclaw tui --deliver` so users can opt into delivery without passing the flag on every launch.

## Best Possible Solution

Add an approved config-backed TUI delivery default with exported types, Zod schema/help/labels, docs, focused tests, and explicit `--deliver` / `--no-deliver` precedence.

## Reproduction Assessment

Yes, for the current behavior: source inspection on current main shows omitted `--deliver` becomes `false`, `sendChat` receives that static value, and config schema/types/docs expose no delivery default. I did not run the interactive TUI in this read-only sweep.

## Solution Assessment

Unclear. A config-backed default is plausible, but maintainers still need to choose the public namespace and CLI/config precedence before implementation is mechanical.
