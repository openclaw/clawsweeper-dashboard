## Summary

Keep open: current main and the latest release still expose `openclaw tui --deliver` only as an invocation-time flag with no config-backed default. The related stuck-spinner bug was handled separately, and the direct config-default PRs are closed unmerged, so the remaining work is a public config/CLI contract decision.

## What This Changes

The issue requests a config-backed default for the TUI `--deliver` flag so users do not need to pass it on every invocation.

## Best Possible Solution

Define an approved TUI delivery-default config contract with schema/help/docs/tests and explicit CLI override semantics that preserve current behavior unless users opt in.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves omitted `--deliver` to `false`, forwards that value into TUI sends, and has no config key that can change the default. I did not run a live TUI session because this was a read-only review.

## Solution Assessment

Unclear. A config-backed default matches the request, but maintainers still need to choose the public config namespace, CLI override behavior, docs contract, and upgrade semantics before this becomes a mechanical fix.
