## Summary

Keep open: current main still lacks a config-backed TUI delivery default. The two related config PRs were closed unmerged, and the later merged abort-gap fix addressed adjacent TUI lifecycle behavior while leaving this delivery-default request out of scope.

## What This Changes

Add a config-backed default for `openclaw tui --deliver` so users do not need to pass the flag on every TUI launch.

## Best Possible Solution

Add a documented config-backed TUI delivery default under a maintainer-approved namespace, with explicit CLI precedence, regression tests, and aligned `/deliver` or Settings documentation.

## Reproduction Assessment

Yes. Source inspection on current main shows the TUI still defaults delivery from the CLI option to `false` and the config schema has no delivery-default key; I did not run a live TUI session in this read-only review.

## Solution Assessment

Unclear. A config option is plausible and narrow, but the exact namespace, CLI precedence, Settings behavior, and relationship to the documented `/deliver` surface need maintainer direction before treating this as a mechanical fix.
