## Label Justifications

- `P2`: The absent default setting creates recurring TUI confusion and can hide successful replies, but the impact is bounded and has an explicit flag workaround.
- `impact:message-loss`: With delivery off, a reply can be generated and persisted without appearing through the user-expected TUI or channel delivery flow.

## Summary

The requested capability remains absent on current main and in v2026.6.6: TUI delivery still defaults off, the documented workaround is `--deliver` or `/deliver on`, and the config schema has no delivery preference. Keep this issue open for a maintainer product decision because prior config/default PRs closed unmerged, while the merged run-abort fix explicitly treated this request as separate remaining work.

## What This Changes

Add a persistent configuration default for TUI reply delivery so users do not need to pass `--deliver` on each invocation.

## Best Possible Solution

Choose and document one canonical policy: either add a narrowly scoped `tui.deliver` preference with `--deliver` and `--no-deliver` overrides, or deliberately change the default with upgrade-impact proof; cover fresh and existing configurations plus delivery-on/off TUI behavior.

## Reproduction Assessment

Yes, source-reproducible with high confidence: run `openclaw tui` without `--deliver`; the CLI and runtime resolve delivery to false, and no current config field can override that default. This review did not execute a live Gateway turn.

## Solution Assessment

Unclear. Persisted config is a reasonable option, but it is not automatically the best fix because changing the default or using a TUI-owned persisted preference has different delivery and upgrade consequences that maintainers must choose explicitly.
