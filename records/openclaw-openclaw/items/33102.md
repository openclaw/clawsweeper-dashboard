## Summary

Keep open: current main and v2026.5.12 still treat TUI delivery as a launch/runtime toggle, not a config-backed default, and the merged stuck-spinner fix explicitly left this ergonomics/config request unresolved.

## What This Changes

The issue requests a config-backed default for the TUI `--deliver` behavior so users do not have to pass `openclaw tui --deliver` on every launch.

## Best Possible Solution

Define an approved config-backed TUI delivery default with schema/types/help/docs/tests and explicit CLI precedence before implementing it.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted `--deliver` becomes `false`, that value is passed into each TUI send, and the config schema/types/docs expose no delivery default.

## Solution Assessment

Unclear. A config-backed default is plausible, but maintainers still need to choose the public namespace and override semantics before this becomes a mechanical fix.
