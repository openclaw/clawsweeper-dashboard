## Summary

Keep open: the reported media proxy bug is still source-reproducible on current main, but this external PR is not merge-ready because it lacks inspectable real behavior proof, is currently conflicting, and would regress current proxy/media contracts if rebased mechanically.

## What This Changes

The PR makes media-understanding audio/video proxy fetch resolution target-aware by passing backend `baseUrl` into proxy-fetch and adding custom `NO_PROXY`/CIDR matching coverage.

## Best Possible Solution

Rebase and implement target-aware media proxy selection by reusing the current `proxy-env` matcher, `resolveEnvHttpProxyAgentOptions`, multipart normalization, and provider request override flow, then add focused regression tests plus redacted live STT proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves the proxy fetch before the configured media backend `baseUrl` is known, while target-aware `NO_PROXY`/CIDR matching lives in `proxy-env.ts` and is not applied at that runner fetch seam. I did not run a live self-hosted STT backend in this read-only review.

## Solution Assessment

No. Moving proxy resolution after `baseUrl` is the right direction, but this branch duplicates stale proxy matching and drops current proxy-fetch and media request contracts instead of reusing the shared helpers.
