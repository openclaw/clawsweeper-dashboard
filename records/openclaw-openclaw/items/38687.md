## Summary

Keep open: the reported media proxy bug is still source-reproducible on current main, but this PR is not mergeable as-is because it is stale against current proxy/media contracts, has blocking proxy-routing regressions, and still lacks inspectable real behavior proof.

## What This Changes

The branch makes media audio/video proxy fetch selection target-aware by passing provider baseUrl into resolveProxyFetchFromEnv and adding local NO_PROXY/CIDR coverage.

## Best Possible Solution

Rebase the target-aware media proxy selection onto current main by reusing shared proxy-env matching and managed proxy options, preserving media request overrides and multipart normalization, then attach redacted live STT proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves the media proxy fetch before provider baseUrl is known, so a NO_PROXY match for the configured STT backend cannot prevent fetchFn injection at the runner seam. I did not run a live self-hosted STT backend in this read-only review.

## Solution Assessment

No. Moving proxy selection after baseUrl resolution is the right direction, but this branch duplicates stale proxy logic and drops current proxy-fetch and media execution contracts instead of reusing the shared helpers.
