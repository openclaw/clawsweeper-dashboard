## Summary

Keep this PR open: it targets a real media proxy bug on current main, but the branch still changes security-sensitive egress routing and is stale against the current shared proxy helpers, so it should not be merged as-is.

## What This Changes

The PR makes media-understanding resolve proxy fetches after provider baseUrl resolution and adds target-aware NO_PROXY bypass logic plus proxy regression tests for media backends.

## Best Possible Solution

Keep the baseUrl-aware media runner change, but implement target decisions in the shared proxy layer, preserving ALL_PROXY behavior and requiring explicit NO_PROXY/CIDR support or a maintainer-approved documented loopback exception.

## Reproduction Assessment

Yes. Static source inspection shows current main resolves the media proxy fetch before the backend baseUrl is known and passes that fetchFn into audio/video provider requests; I did not run tests because this review was read-only.

## Solution Assessment

No. The PR targets the right surface, but the current patch is not the best final fix because it duplicates proxy matching, changes loopback egress policy, and drops current ALL_PROXY-backed option handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep media loopback behind explicit proxy opt-out:** `src/infra/net/proxy-fetch.ts:136-140`
  - body: The new matcher returns direct fetch for localhost, ::1, and 127.0.0.0/8 before checking NO_PROXY. Managed proxy mode documents loopback bypass only for Gateway control-plane traffic and clears NO_PROXY so normal runtime traffic stays behind the filtering proxy, so media transcription to a loopback URL would bypass operator egress controls without an explicit media opt-out.
  - confidence: 0.88
- **[P2] Preserve ALL_PROXY-backed proxy fetches:** `src/infra/net/proxy-fetch.ts:205-216`
  - body: Current main maps ALL_PROXY into explicit EnvHttpProxyAgent options and has coverage for ALL_PROXY-only environments. This branch gates proxy fetch creation on hasEnvHttpProxyConfigured("https") and then constructs EnvHttpProxyAgent without those options, so ALL_PROXY-only deployments stop receiving a proxy fetch for affected callers.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff changes network egress routing and introduces direct loopback media bypass behavior that needs maintainer/security approval before merge.

Concerns:

- **[medium] Unconditional loopback media proxy bypass:** `src/infra/net/proxy-fetch.ts:136`
  - body: The PR bypasses proxy wrapping for loopback media backend targets even when the operator did not set NO_PROXY. That conflicts with the managed proxy model where normal runtime traffic remains behind the filtering proxy and NO_PROXY is cleared to prevent destination-based bypasses.
  - confidence: 0.88
- **[low] ALL_PROXY-only environments can lose proxy routing:** `src/infra/net/proxy-fetch.ts:205`
  - body: By replacing current explicit EnvHttpProxyAgent option construction with a hasEnvHttpProxyConfigured gate and no constructor options, ALL_PROXY-only deployments can silently fall back to direct routing for affected fetches.
  - confidence: 0.8
