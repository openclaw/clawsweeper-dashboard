## Summary

Keep open. Current main still has the reported media proxy ordering problem, and the PR is aimed at a real narrow bug, but the branch is not the best merge shape because it introduces direct loopback media egress under managed proxy mode and drops current ALL_PROXY handling.

## What This Changes

The PR makes media-understanding proxy-fetch resolution target-aware by moving audio/video fetch selection after backend baseUrl resolution and adding proxy-fetch plus media-runner regression coverage for NO_PROXY-covered backends.

## Best Possible Solution

Resolve media provider baseUrl before choosing fetchFn, but centralize the target-aware decision in the shared proxy-env/proxy-fetch layer, preserve resolveEnvHttpProxyAgentOptions including ALL_PROXY, and require an explicit NO_PROXY or documented operator opt-in for any direct media path.

## Reproduction Assessment

Yes. Static reproduction is high confidence: with HTTPS_PROXY set and a media backend baseUrl covered by NO_PROXY, current main still constructs the proxy fetch before it knows baseUrl and passes it into the audio transcription request. I did not run tests because this review was read-only.

## Solution Assessment

No. The PR targets the right bug, but the current patch is not the safest final solution because it duplicates NO_PROXY semantics, bypasses loopback media targets without explicit opt-in, and drops current ALL_PROXY support; reusing the existing proxy-env helpers is the narrower maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep media loopback behind explicit proxy opt-out:** `src/infra/net/proxy-fetch.ts:136-140`
  - body: The new matcher returns direct fetch for localhost, ::1, and 127.0.0.0/8 before checking NO_PROXY. Managed proxy mode documents loopback bypass only for Gateway control-plane traffic and clears NO_PROXY so normal runtime traffic cannot skip the filtering proxy, so media transcription to a loopback URL would bypass operator egress controls without an explicit media opt-out.
  - confidence: 0.88
- **[P2] Preserve ALL_PROXY-backed proxy fetches:** `src/infra/net/proxy-fetch.ts:205-216`
  - body: Current main maps ALL_PROXY into explicit EnvHttpProxyAgent options and has coverage for ALL_PROXY-only environments. This branch gates proxy fetch creation on hasEnvHttpProxyConfigured("https") and then constructs EnvHttpProxyAgent without options, so ALL_PROXY-only deployments stop receiving a proxy fetch for media and other resolveProxyFetchFromEnv callers.
  - confidence: 0.82

## Security Review

  - body: The PR bypasses proxy wrapping for loopback media backend targets even when the operator did not set NO_PROXY. That conflicts with the managed proxy model where normal runtime traffic remains behind the filtering proxy and NO_PROXY is cleared to prevent destination-based bypasses.
