## Summary

Keep open: the underlying media proxy bug is still source-reproducible on current main, but this stale branch should not merge because it diverges from current proxy/media contracts and lacks inspectable live proof.

## What This Changes

The branch makes media audio/video proxy fetch selection target-aware by passing provider baseUrl into proxy resolution and adding local NO_PROXY/CIDR matching coverage.

## Best Possible Solution

Rework the target-aware media proxy selection on top of current main by reusing shared proxy-env matching and proxy-fetch options/normalization while preserving media request overrides, then attach redacted live STT proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves media fetchFn before the provider baseUrl is available, so a configured NO_PROXY backend cannot bypass proxy injection at that runner seam. I did not run a live self-hosted STT backend in this read-only review.

## Solution Assessment

No. Moving proxy selection after baseUrl resolution is the right direction, but this branch duplicates stale proxy logic and drops current proxy-fetch and media request contracts instead of reusing the shared helpers.

  - body: This path now gates on `hasEnvHttpProxyConfigured("https")` and constructs `EnvHttpProxyAgent` without the options current main builds through `resolveManagedEnvHttpProxyAgentOptions`. That can skip `ALL_PROXY`, HTTP fallback, or active managed proxy TLS for media requests.

  - confidence: 0.88
