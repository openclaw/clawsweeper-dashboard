## Summary

Keep open: the underlying media proxy bug is source-reproducible, but this stale/conflicting branch would regress current proxy and media-provider contracts, and the external PR still lacks inspectable real behavior proof.

## What This Changes

The PR makes media audio/video proxy resolution target-aware by passing provider baseUrl into proxy-fetch and adding local NO_PROXY/CIDR matching tests.

## Best Possible Solution

Rebase the target-aware media proxy idea onto current main by reusing the shared proxy-env matcher/options, preserving multipart normalization and media request overrides, then add focused regression tests plus redacted live STT proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves fetchFn before provider baseUrl is known, while target-aware NO_PROXY matching exists elsewhere and is not applied at that runner seam. I did not run a live self-hosted STT backend in this read-only review.

## Solution Assessment

No. Moving proxy selection after baseUrl resolution is the right direction, but this branch duplicates stale proxy matching and drops current proxy-fetch/media request contracts instead of reusing the shared helpers.
