## Summary

Keep open: current main still lacks target-aware media proxy fetch resolution, but this branch cannot merge as-is because it duplicates newer shared proxy matching and regresses current egress-routing behavior. It also still lacks inspectable real behavior proof for the live STT path.

## What This Changes

The PR makes media-understanding audio/video proxy fetch resolution depend on the configured backend baseUrl, adding NO_PROXY/loopback bypass logic and related proxy-fetch and runner tests.

## Best Possible Solution

Rework the branch against the current shared proxy helpers so media passes backend `baseUrl` into target-aware NO_PROXY logic while preserving existing ALL_PROXY options, FormData normalization, and managed-proxy policy boundaries.

## Reproduction Assessment

Yes. Source inspection shows current main resolves media `fetchFn` before `baseUrl` is known, so a backend covered only by CIDR/target-aware NO_PROXY semantics still receives the proxy-aware fetch.

## Solution Assessment

No. The PR targets the right media runner surface, but the current branch duplicates proxy matching and regresses current proxy-fetch behavior; the safer solution is a small target-aware extension that reuses `matchesNoProxy`/`resolveEnvHttpProxyAgentOptions`.
