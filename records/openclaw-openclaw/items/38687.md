## Summary

Keep open: current main still has the reported baseUrl-blind media proxy path, but this PR should not merge as-is because it conflicts with current shared proxy semantics and changes loopback media egress policy.

## What This Changes

The PR makes media-understanding resolve proxy fetches after audio/video backend baseUrl resolution and adds target-aware NO_PROXY/loopback proxy-bypass tests.

## Best Possible Solution

Rewrite this around the shared proxy-env helpers: keep baseUrl-aware media routing, preserve ALL_PROXY option mapping, require explicit NO_PROXY for direct media paths unless a loopback exception is approved and documented, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main resolves the media proxy fetch before the backend baseUrl is known and passes that wrapper into audio/video provider calls; I did not run tests because this review was read-only.

## Solution Assessment

No. The PR targets the right surface, but the current patch is not the best final fix because it duplicates proxy matching, changes loopback egress policy, and can drop current ALL_PROXY-backed EnvHttpProxyAgent options.
