## Summary

This PR addresses a real media proxy bug, but the current patch is not ready because it changes loopback egress bypass behavior and regresses current ALL_PROXY handling in the shared proxy-fetch path.

## What This Changes

This PR makes media-understanding resolve proxy fetches after the configured audio/video backend baseUrl is known and adds target-aware NO_PROXY bypass logic with proxy regression tests.

## Best Possible Solution

Rewrite the fix on current main so media resolves proxy behavior after baseUrl resolution, reuses shared proxy-env/proxy-fetch semantics, preserves ALL_PROXY, and only adds CIDR or loopback bypass rules after explicit maintainer/security approval.

## Reproduction Assessment

Yes, from source inspection. Current main resolves a proxy fetch before the media backend baseUrl is known and passes that wrapper into audio/video provider requests, while the reported PR diff moves the decision to the backend target.

## Solution Assessment

No. The PR targets the right surface, but the implementation should be rebuilt around the shared proxy helpers and current ALL_PROXY behavior instead of adding a separate matcher with direct loopback bypass policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep media loopback behind explicit proxy opt-out:** `src/infra/net/proxy-fetch.ts:136-140`
  - body: The new matcher returns direct fetch for localhost, ::1, and 127.0.0.0/8 before requiring a NO_PROXY match. The managed proxy docs reserve direct loopback for Gateway control-plane traffic and keep normal runtime HTTP/WebSocket traffic behind the configured proxy, so this can bypass operator egress controls for media transcription targets.
  - confidence: 0.88
- **[P2] Preserve ALL_PROXY-backed proxy fetches:** `src/infra/net/proxy-fetch.ts:205-216`
  - body: Current main maps ALL_PROXY/all_proxy into explicit EnvHttpProxyAgent options for resolveProxyFetchFromEnv. This branch gates creation on HTTP/HTTPS proxy vars and constructs EnvHttpProxyAgent without those options, so ALL_PROXY-only deployments can silently lose proxy routing for affected callers.
  - confidence: 0.84
