## Summary

Keep open: the reported media proxy bug is plausible and not solved at the runner seam on current main, but this PR is not mergeable as-is because it lacks inspectable real behavior proof and would regress current proxy policy/transport behavior.

## What This Changes

The PR changes media-understanding proxy fetch resolution so audio/video backend requests can bypass the proxy when their configured baseUrl matches loopback or NO_PROXY rules.

## Best Possible Solution

Rebase or rewrite the fix so media resolves proxy fetch after baseUrl is known, reuses the current shared proxy-env matcher/options and FormData normalization, and bypasses only explicit NO_PROXY/CIDR targets unless maintainers approve a loopback policy change.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a generic media proxy fetch before the backend baseUrl is known, while upstream EnvHttpProxyAgent does not implement OpenClaw's CIDR NO_PROXY extension. I did not run a live self-hosted STT backend in this read-only review.

## Solution Assessment

No. Target-aware resolution is the right direction, but this PR duplicates stale proxy matching and loses current proxy-fetch behavior; the safer path is to reuse the current shared helpers and avoid implicit loopback bypass without owner approval.
