## Summary

Keep open: the target-aware media proxy seam is a valid bug-fix direction, but this PR head should not merge because it broadens loopback proxy bypass policy, drops current proxy-fetch behavior, and still lacks inspectable real behavior proof.

## What This Changes

The PR makes media audio/video proxy fetch resolution target-aware by passing the configured backend baseUrl into resolveProxyFetchFromEnv and adding custom NO_PROXY, loopback, and CIDR coverage.

## Best Possible Solution

Move the target-aware decision into the shared proxy contract or media runner using existing matchesNoProxy and resolveEnvHttpProxyAgentOptions, bypass only explicit NO_PROXY matches, preserve multipart normalization, and add redacted live STT proof.

## Reproduction Assessment

Yes, source-reproducible for the target-aware gap: current main computes fetchFn before the media backend baseUrl is available, so the runner cannot apply OpenClaw's target-specific NO_PROXY/CIDR policy there. I did not have an inspectable live self-hosted STT artifact.

## Solution Assessment

No. Passing baseUrl into the media proxy decision is the right seam, but this branch duplicates NO_PROXY parsing and drops current proxy-fetch behavior instead of reusing the shared proxy-env and proxy-fetch contracts.
