## Summary

Keep open. The reported media/proxy bug is still source-reproducible on current main, but this branch is not merge-ready because it changes security-sensitive egress behavior, drops current proxy option handling, and lacks inspectable real-behavior proof.

## What This Changes

The PR makes media-understanding proxy fetch resolution target-aware for configured audio/video backend URLs and adds NO_PROXY-focused proxy-fetch and runner tests.

## Best Possible Solution

Land a rebased fix that passes the resolved backend baseUrl into a shared target-aware proxy decision, reuses proxy-env semantics, preserves EnvHttpProxyAgent options and FormData normalization, and keeps loopback direct routing behind explicit NO_PROXY unless maintainers approve a policy change.

## Reproduction Assessment

Yes, from source inspection. Current main resolves the media proxy fetch before the backend baseUrl is available, so configured CIDR NO_PROXY coverage cannot be applied to that media backend path; I did not run a live STT setup.

## Solution Assessment

No. Moving fetch resolution after baseUrl lookup is the right direction, but duplicating proxy matching and changing loopback/ALL_PROXY behavior is not the narrowest maintainable fix against current main.
