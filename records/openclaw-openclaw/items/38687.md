## Summary

Keep open: the reported media/proxy bug is plausible and current main does not yet make media runner proxy-fetch resolution target-aware, but this branch needs changes before merge because its NO_PROXY matcher widens wildcard-list bypass semantics and the PR still lacks posted real behavior proof.

## What This Changes

The branch changes proxy-fetch resolution and media runner wiring so provider fetch functions are resolved against the configured media backend baseUrl and bypassed for loopback or NO_PROXY-matched targets.

## Best Possible Solution

Rebase onto current main, make media proxy-fetch selection target-aware by reusing the shared proxy-env NO_PROXY matcher, and add focused regression coverage for CIDR, wildcard-octet, host:port, and wildcard-list cases while preserving recent media request handling.

## Reproduction Assessment

No live reproduction was established. Source inspection gives a clear targeted reproduction for the review finding with `NO_PROXY=localhost,*`, and the original media CIDR path remains plausible because current media runner still creates a proxy fetch before it knows the target baseUrl.

## Solution Assessment

No. The target-aware direction is maintainable, but the branch should reuse current main's shared proxy-env matcher instead of carrying a divergent NO_PROXY parser that already disagrees with repository tests.
