## Summary

Keep this PR open, but it is not mergeable as-is. The core bug remains plausible and source-reproducible, but the current branch duplicates proxy matching, bypasses loopback media targets without explicit NO_PROXY, drops current proxy-fetch behavior, and still lacks inspectable real behavior proof.

## What This Changes

The PR makes media audio/video provider proxy fetch resolution target-aware so configured backend base URLs can avoid proxy wrapping for loopback or NO_PROXY matches.

## Best Possible Solution

Rework the PR to pass the media backend baseUrl into a shared proxy decision that preserves current proxy-env/proxy-fetch contracts and add redacted live or log proof for the self-hosted STT path.

## Reproduction Assessment

Yes, source-reproducible: current main resolves the media proxy fetch before the backend baseUrl is available, and undici's NO_PROXY matcher does not support CIDR entries. I did not run a live self-hosted STT backend in this read-only review.

## Solution Assessment

No. Moving fetch resolution after baseUrl is the right direction, but this branch duplicates proxy matching and drops current proxy-fetch behavior instead of reusing the shared proxy-env and proxy-fetch contracts.

  - body: The matcher returns direct fetch for localhost, ::1, and 127.0.0.0/8 before consulting NO_PROXY.   - confidence: 0.9

  - body: Current proxy-fetch converts global FormData to undici FormData and strips stale multipart headers before proxy dispatch. This branch forwards raw init through undiciFetch, which can reintroduce multipart upload failures for proxied audio transcription.
  - confidence: 0.84
