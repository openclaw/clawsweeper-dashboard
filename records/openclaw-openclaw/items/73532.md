## Summary

Keep open. Current main has substantial unshipped runtime-deps and plugin-loader mitigations that directly target the reported hot loop, but the issue is still an active canonical regression tracker and there is not yet affected-host proof that the packaged gateway, request-time plugin paths, and repeated staging paths are all quiet on current main.

## What This Changes

The issue reports a gateway starvation regression where repeated bundled plugin runtime-deps mirror preparation, runtime-deps repair, and manifest loading can saturate CPU, grow memory, and block Telegram, WS RPC, and chat-completions traffic.

## Best Possible Solution

Keep this as the canonical runtime-deps/plugin-loader starvation tracker until maintainers retest an affected packaged host on current main or the next release. If it still reproduces, split the remaining work into the narrowest proven subpath, such as request-time plugin registry loading, runtime-deps materialization, dist-mirror preparation, or channel startup.

## Reproduction Assessment

Yes for the affected released range: the thread gives concrete Linux, Docker, and Raspberry Pi setups with plugin counts, profiles, logs, and version ranges. It is unclear for current main at 4cbd1b53 because this read-only review did not run an affected packaged host after the latest unshipped mitigations.

## Solution Assessment

Unclear. The current-main direction is maintainable because repair is moved into startup/config planning and hot paths are verify-only, but closing this tracker is not the best outcome until affected-host proof shows the gateway stays responsive after startup and during chat-completions/tool-provider paths.
