## Summary

Keep open. Current main contains substantial runtime-deps and plugin-loader mitigations, but the thread is still the active canonical tracker and there is not high-confidence affected-host proof that the packaged Gateway, request-time plugin paths, and chat-completions trigger are all quiet on current main.

## What This Changes

The issue requests fixing a Gateway event-loop starvation regression where bundled plugin runtime mirror preparation and JSON5 manifest parsing repeatedly run after startup.

## Best Possible Solution

Keep this as the canonical runtime-deps/plugin-loader starvation tracker until maintainers retest an affected packaged host on current main or the next package with CPU, RSS, event-loop delay, runtime-deps staging logs, and plugin-loader timing. If it still reproduces, split the remaining work into the narrowest proven subpath, such as request-time tool registry loading, runtime-deps convergence, dist-mirror preparation, or channel startup.

Yes for the affected released range: the issue thread gives concrete Linux, Raspberry Pi, Docker, versions, plugin counts, profiles, logs, and trigger patterns. Unclear for current main because this read-only review inspected source/tests/history but did not run an affected packaged host after the latest unreleased mitigations.

## Solution Assessment

Unclear. The current-main direction is maintainable because repair is moved into startup/config planning, hot paths are verify-only, manifests are cached, and mirrors are memoized, but closing is not the best outcome until affected-host proof shows startup and chat-completions paths no longer starve the event loop.
