## Summary

Keep open. Current main has several targeted runtime-deps, mirror, and manifest-cache mitigations, but the discussion still contains post-mitigation/current-main-style reports and current source still has reachable startup prestage, per-plugin runtime-deps staging, dist-mirror locking, and request-time registry paths. Closing would need affected-host proof on exact current main or the next package, not source inspection alone.

## What This Changes

The issue reports a bundled plugin/runtime-deps loader hot loop where repeated runtime dist mirror preparation, staged installs, and JSON5 manifest parsing starve the Gateway event loop and break Telegram, WS probes, and chat requests.

## Best Possible Solution

Keep this as the canonical runtime-deps/plugin-loader starvation tracker until maintainers retest an affected packaged host on current main or the next package. If it still reproduces, split or fix the smallest proven hot path with timing/cache-hit instrumentation around runtime-deps materialization, dist-mirror locks, manifest registry loading, chat-completions preparation, and channel startup.

## Reproduction Assessment

Yes for the affected released range: the issue gives concrete Linux, Docker, and Raspberry Pi setups with plugin counts, profiles, logs, and v2026.4.24-v2026.4.29 behavior. It is unclear for exact current main because this read-only review did not run an affected host and newer mitigations have not been proven against the same reproducer.

## Solution Assessment

Unclear. The current-main fixes are sensible targeted mitigations, but the best solution is not to close this tracker until startup, repeated-staging, and request-time chat-completions paths are quiet on an affected packaged host.
