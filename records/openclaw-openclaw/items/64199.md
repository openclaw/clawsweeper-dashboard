## Label Justifications

- `P1`: The issue can route sibling Discord threads into one persistent ACP process, breaking a real channel workflow with cross-thread context exposure.
- `impact:session-state`: The core failure is persistent ACP session identity collapsing multiple Discord threads into shared state.

## Summary

Keep open: current main and the latest release still materialize inherited Discord thread configured ACP bindings with the parent channel id before building the persistent ACP session key, and the paired fix PR is open but conflicting.

## What This Changes

The issue reports that persistent configured ACP bindings for Discord parent channels give sibling threads the same ACP session key, causing shared Claude Code process state.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/64322 or an equivalent current-main patch that keys inherited Discord parent-channel ACP bindings by the live thread id while preserving direct bindings and channel-owned canonical remaps.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Discord preflight passes a live thread id plus parent id, the matcher returns the parent id, the registry materializes that match, and the ACP key hashes only conversationId.

## Solution Assessment

Yes for the fix direction, but not as a close decision. The maintainable solution is a narrow current-main change to return or materialize the live thread identity for Discord parent fallback while preserving channel-specific canonical remaps.
