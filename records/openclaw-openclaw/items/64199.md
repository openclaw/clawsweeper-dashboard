## Summary

Current main still has the reported configured ACP session-key collision path: inherited Discord thread routing returns the parent channel as the configured ACP match, then the persistent ACP key hashes only that materialized conversation id. The paired open fix PR #64322 should be reviewed or replaced before this issue closes.

## What This Changes

Persistent ACP Discord parent-channel bindings should give each runtime thread its own configured ACP session key instead of sharing the parent channel key.

## Best Possible Solution

Review and land #64322 or an equivalent narrow fix that preserves parent-channel route inheritance while materializing inherited Discord thread ACP bindings with the live thread id and regression coverage for two sibling threads.

## Reproduction Assessment

Yes. Resolve two Discord configured binding routes with different messageChannelId values and the same earlyThreadParentId; current main returns the parent match and therefore derives the same targetSessionKey.

## Solution Assessment

Yes. The maintainable direction is to keep parent-channel inheritance for matching but make the materialized configured ACP identity thread-scoped for inherited thread contexts; the open PR should be reviewed against that boundary.
