## Summary

Current main still matches the reported configured ACP session-key collision: inherited Discord thread routing returns the parent channel as the configured ACP conversation, the registry materializes that match, and the persistent ACP key hashes only that materialized conversation id. The same-author implementation PR #64322 is still the appropriate path to review or replace before this issue can close.

## What This Changes

The issue requests that persistent ACP configured bindings inherited from a Discord parent channel use per-thread session keys so sibling threads do not share one Claude Code process.

## Best Possible Solution

Keep parent-channel route inheritance, but materialize inherited Discord thread ACP bindings with the live thread conversation id so persistent configured ACP sessions are isolated per thread. Add regression coverage proving two thread ids under one parent produce different targetSessionKey values while direct parent-channel and direct thread bindings keep their current priority behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to resolve two different Discord thread conversation ids with the same parent-channel configured ACP binding; current main returns the parent match, materializes the parent id, and hashes that id into the same targetSessionKey.

## Solution Assessment

Yes. The narrow maintainable direction is to keep the issue open and review/land #64322 or an equivalent fix that scopes only inherited Discord thread materialization and adds the missing regression coverage.
