## Summary

Keep open: current main still materializes inherited Discord thread configured ACP bindings through the parent channel id, so sibling threads can share one persistent ACP process; the same-author fix PR is still open and conflicting.

## What This Changes

The issue reports that persistent ACP configured bindings for Discord parent-channel routes key by the parent channel instead of each live thread, causing sibling threads to share Claude Code state.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/64322 with a current-main patch that keys inherited Discord parent-channel ACP bindings by the live thread id, preserves direct bindings and channel-owned canonical remaps, and adds regression coverage for two thread ids under one parent.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by resolving two Discord configured ACP bindings with different live thread conversation ids and the same parentConversationId; current main returns the parent id and hashes it into the same configured ACP key.

## Solution Assessment

Yes for the fix direction. The narrow maintainable solution is to change only the Discord inherited parent-channel match so the live thread id drives the persistent ACP key, while preserving direct matches and other channel canonical remaps.
