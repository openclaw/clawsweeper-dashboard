## Summary

Keep open: current main still materializes inherited Discord thread configured ACP bindings through the parent channel id, so sibling threads can share one persistent ACP key; the paired fix PR at https://github.com/openclaw/openclaw/pull/64322 is still open and unmerged.

## What This Changes

The issue asks OpenClaw to give each Discord thread under a configured parent-channel persistent ACP binding its own session key and ACP process.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/64322 or an equivalent current-main patch that makes inherited Discord parent-channel ACP bindings key by the live thread id while preserving direct bindings and channel-owned canonical remaps.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by resolving two Discord configured ACP bindings with different thread conversation ids and the same parentConversationId; current main materializes both through the parent id that is hashed into one configured ACP key.

## Solution Assessment

Yes. The narrow maintainable fix is to change the Discord inherited parent-channel match/materialization path so the live thread conversation id drives the persistent ACP key, while preserving the existing registry contract for channel-owned canonical remaps.
