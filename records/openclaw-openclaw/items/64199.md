## Summary

Keep open: current main still has the configured ACP collision path for inherited Discord thread routing, and the paired open fix PR at https://github.com/openclaw/openclaw/pull/64322 remains unmerged.

## What This Changes

The issue reports that configured persistent ACP Discord parent-channel bindings materialize sibling thread conversations to the same session key, causing one Claude Code process to be shared across threads.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/64322 or an equivalent current-main fix that uses the live Discord thread id for inherited configured ACP session keys while preserving direct bindings and channel-owned remaps.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by resolving two Discord configured ACP bindings with different thread conversation ids and the same parentConversationId; current main materializes both through the parent id that is hashed into one configured ACP key.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the Discord inherited parent-channel match/materialization path so the live thread conversation id drives the persistent ACP key, while avoiding a global key-hash change that could break channel-owned canonical remaps.
