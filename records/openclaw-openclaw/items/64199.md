## Label Justifications

- `P1`: This can route multiple Discord threads into one persistent ACP process, breaking a real channel workflow with cross-thread context exposure.
- `impact:session-state`: The issue is about persistent ACP session identity collapsing sibling Discord threads into shared state.
- `impact:security`: Shared persistent ACP context can expose one Discord thread's conversation context to another thread under the same parent channel.

## Summary

Keep open: current main and the latest shipped release still materialize inherited Discord thread ACP bindings with the parent channel identity before building the persistent ACP key, and the same-author fix PR remains open and conflicting.

## What This Changes

The issue reports that persistent ACP configured bindings for Discord parent channels key sessions by parent channel instead of thread, causing sibling threads to share one Claude Code process.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/64322 or an equivalent current-main patch that keys inherited Discord parent-channel ACP bindings by the live thread id while preserving direct bindings and other channel-owned canonical remaps.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Discord preflight passes a live thread id plus parent id, the matcher returns the parent id, the registry materializes that match, and the configured ACP key hashes only conversationId.

## Solution Assessment

Yes for the fix direction, but not as an already-closed issue. The maintainable solution is a narrow current-main change that materializes the live Discord thread identity for inherited parent-channel ACP bindings while preserving channel-specific canonical remaps.
