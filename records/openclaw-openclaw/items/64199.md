## Summary

Current main still has the configured ACP collision path for inherited Discord thread routing, and the paired same-author fix PR at https://github.com/openclaw/openclaw/pull/64322 is still open and unmerged. This issue should remain open until that PR is reviewed, rebased/replaced, and landed or an equivalent fix ships.

## What This Changes

The issue reports that persistent ACP configured bindings for Discord parent-channel routes materialize sibling threads to the same parent-derived session key, causing shared Claude Code process state.

## Best Possible Solution

Keep this issue open until the paired PR or an equivalent fix lands: inherited Discord parent-channel ACP bindings should produce distinct persistent session keys per runtime thread while preserving direct parent/direct thread bindings and other channel-canonical remaps.

## Reproduction Assessment

Yes. A high-confidence source path exists: resolve two Discord configured binding routes with different messageChannelId values and the same earlyThreadParentId; current main matches the parent id and then hashes that parent id into the configured ACP key.

## Solution Assessment

Yes for the narrow product direction. The maintainable fix is to make Discord parent-channel fallback materialize inherited thread bindings with the runtime thread id, with regression coverage, rather than changing the global key hash and risking other channel remapping semantics.
