## Summary

Keep open. Current main still routes Discord voice through a voice-channel-derived session with text delivery disabled, and the config/schema/docs still do not expose a bridge-to-text mode; the maintainer discussion also explicitly kept this as dedicated Discord voice feature work rather than cleanup.

## What This Changes

Request adding a Discord voice bridge mode so spoken transcripts are posted to a selected text channel, routed through that text-channel session, and replies are both delivered as Discord text and spoken in voice.

## Best Possible Solution

Implement the bridge in the bundled Discord plugin with typed config/schema/docs, explicit bridge target resolution, permission checks for posting transcripts to text, speaker-attributed transcript messages, text-channel session routing, normal Discord text delivery, and TTS fan-out of the same final reply; keep realtime audio provider work as a separate follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice joins resolve a route from the voice channel id, voice turns call agentCommandFromIngress() with deliver:false, and strict config/schema/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested owner boundary is right for the bundled Discord plugin, but the exact default behavior, permission model, and target-channel selection need maintainer design before a narrow automated fix would be safe.
