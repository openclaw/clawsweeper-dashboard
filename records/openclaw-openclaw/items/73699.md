## Summary

Keep open. Current main still runs Discord voice through the voice-channel session with text delivery suppressed, and the maintainer discussion explicitly left voice-as-I/O bridging as remaining Discord plugin feature work.

## What This Changes

Request to make Discord voice channels act as microphone/speaker I/O for the bound text-channel agent session, with transcripts posted to text and replies both posted and spoken.

## Best Possible Solution

Add an opt-in bundled Discord plugin bridge with typed config, validated text-channel targeting, speaker-attributed transcript posting, normal text delivery, TTS fan-out, docs, changelog, and focused voice tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: /vc join and autoJoin carry only the voice channel, route resolution uses that channel, processDiscordVoiceSegment passes deliver:false, and strict config/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested bridge direction is plausible, but the exact config shape and permission/privacy contract need maintainer decisions before this is a safe narrow implementation.
