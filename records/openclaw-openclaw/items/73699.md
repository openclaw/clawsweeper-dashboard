## Summary

Keep open. Current main still routes Discord voice through a voice-channel session with text delivery disabled, and the maintainer discussion explicitly says the requested voice-as-I/O bridge remains dedicated feature work.

## What This Changes

Request an opt-in Discord voice mode that posts voice transcripts into a bound text channel, runs the normal text-channel agent session, and speaks the text reply back in voice.

## Best Possible Solution

Implement a maintainer-approved, opt-in Discord plugin bridge with typed config, validated text-channel targeting, speaker-attributed transcript posting, normal text-channel delivery, voice TTS fan-out, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: voice joins resolve the session from the voice channel, `processDiscordVoiceSegment` runs with `deliver:false`, and the strict Discord voice config exposes no bridge fields.

## Solution Assessment

Unclear. The requested bridge direction is plausible, but the exact config and privacy/permission contract need maintainer decisions before this is the narrowest maintainable fix.
