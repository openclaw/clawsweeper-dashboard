## Summary

Keep open. Current main still routes Discord voice turns through the joined voice channel session with text delivery disabled, and the issue discussion includes a maintainer pass explicitly leaving this as dedicated Discord voice feature work.

## What This Changes

The issue requests an opt-in Discord voice bridge mode where voice transcripts and spoken replies use a paired text-channel agent session instead of an isolated voice-channel session.

## Best Possible Solution

Implement this in the bundled Discord plugin after maintainer design: add typed opt-in bridge config, validate target-channel visibility and permissions, post speaker-attributed transcripts, run the normal text-channel session with delivery enabled, fan out the final reply to TTS, and update Discord voice docs/config metadata/tests while keeping realtime audio provider work separate under #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: join() resolves the route from the voice channel id, processDiscordVoiceSegment() uses deliver:false, and the strict voice config/docs expose no bridge fields.

## Solution Assessment

Unclear. The voice-as-I/O direction is plausible and belongs in the Discord plugin, but the exact config shape, transcript visibility rules, and permission/tool-access contract need maintainer decisions before a narrow automated fix is safe.
