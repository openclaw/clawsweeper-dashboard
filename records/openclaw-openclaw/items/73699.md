## Summary

Keep open. Current main still routes Discord voice through a voice-channel session with text delivery suppressed, and the maintainer discussion explicitly left voice-as-I/O bridging as dedicated Discord plugin feature work.

## What This Changes

Request a Discord voice bridge mode that sends STT transcripts into a paired text-channel session and posts plus speaks the same agent reply instead of using an isolated voice session.

## Best Possible Solution

Implement this in the bundled Discord plugin as an opt-in, typed bridge mode with target validation, speaker-attributed transcript posting, normal text delivery, TTS fan-out, docs, changelog, and focused voice tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: /vc join and autoJoin resolve a voice-channel route, processDiscordVoiceSegment uses deliver:false, and the strict config/docs expose no bridge-to-text fields.

## Solution Assessment

Unclear. Voice-as-I/O is a plausible product direction, but the exact config, bridge target, permission, privacy, and speaker-attribution contract needs maintainer design before an autonomous fix is safe.
