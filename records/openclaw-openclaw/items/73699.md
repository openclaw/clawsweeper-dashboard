## Summary

Keep open. Current main still implements Discord voice as its own voice-channel-routed ingress path: auto-join only passes the voice guild/channel, the route peer is the voice channel id, voice turns call `agentCommandFromIngress()` with `deliver: false`, and the strict config/docs expose no bridge-to-text option. The maintainer discussion explicitly left this as dedicated Discord voice feature work, not cleanup.

## What This Changes

Request a Discord voice bridge mode where spoken input is transcribed into a bound text-channel session and the same agent reply is both posted to text and spoken in voice.

## Best Possible Solution

Implement the bridge inside the bundled Discord plugin as an explicit typed mode: add config/schema/docs, resolve and validate the target text channel, post speaker-attributed transcripts, run the normal text-channel session with delivery enabled, and fan out the final reply to both Discord text and voice TTS. Keep realtime audio provider work separate.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice joins resolve routing from the voice channel id, voice segment processing passes `deliver: false`, and the schema/docs expose no bridge fields.

## Solution Assessment

Unclear. The voice-as-I/O direction appears maintainable in the Discord plugin, but the proposed exact config shape is not yet proven to be the safest API because permission, privacy, target-channel, and attribution semantics need maintainer design first.
