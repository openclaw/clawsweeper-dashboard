## Summary

Keep open. Current main still implements isolated Discord voice-channel turns, and the maintainer discussion explicitly keeps voice-as-I/O bridging as dedicated feature work rather than a cleanup close.

## What This Changes

The issue requests an opt-in Discord voice bridge that treats voice as microphone/speaker I/O for an existing text-channel agent session, with transcript posting and spoken replies.

## Best Possible Solution

Implement a maintainer-approved, opt-in Discord plugin bridge with typed config, validated text-channel targeting, speaker-attributed transcript posting, normal text-channel delivery, voice TTS fan-out, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows voice joins build a route from the voice channel, processDiscordVoiceSegment runs agentCommandFromIngress with deliver:false, and the strict config schema exposes no bridge fields.

## Solution Assessment

Unclear. The requested opt-in bridge is plausible, but the exact config shape and permissions/privacy contract need maintainer/channel-owner design before this is the narrowest maintainable fix.
