## Summary

Keep open. Current main still routes Discord voice turns through the joined voice channel's own session, suppresses normal text delivery, and exposes no bridge-to-text configuration; maintainer discussion also explicitly left this as dedicated Discord voice feature work rather than cleanup.

## What This Changes

Request a Discord voice bridge mode that posts STT transcripts into a bound text-channel session, uses that session for agent context and delivery, and also speaks replies back into the voice channel.

## Best Possible Solution

Implement this in the bundled Discord plugin after maintainer design: add typed opt-in bridge config, validate the text target and permissions, post speaker-attributed transcripts, run the normal text-channel session with delivery enabled, fan out the final reply to TTS, and update Discord voice docs/config metadata/tests. Keep realtime audio provider work separate under #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: autoJoin and /vc join only select a voice channel, join() resolves routing from that voice channel id, processDiscordVoiceSegment() uses deliver:false, and strict config/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested voice-as-I/O direction is plausible and belongs in the Discord plugin, but the exact config shape and permission/privacy contract need maintainer decisions; a narrow sessionChannelId-only change would not cover the full transcript posting and TTS fan-out behavior requested here.
