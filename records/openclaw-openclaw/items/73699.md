## Summary

Keep open. Current main still routes Discord voice through the voice-channel session with text delivery disabled, and the maintainer discussion explicitly leaves voice-as-I/O bridging as dedicated feature work rather than cleanup-closeable work.

## What This Changes

Add an opt-in Discord voice bridge that routes spoken transcripts through an existing text-channel session, posts transcripts and replies to text, and also speaks replies in voice.

## Best Possible Solution

Implement a maintainer-approved, opt-in Discord plugin bridge with typed config, validated text-channel targeting, speaker-attributed transcript posting, normal text-channel delivery, voice TTS fan-out, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: voice joins resolve the route from the voice channel, voice turns run with deliver:false, and the strict Discord voice config exposes no bridge fields.

## Solution Assessment

Unclear. The requested bridge direction is plausible, but the exact config, target-selection, privacy, permission, and speaker-attribution contract needs maintainer approval before it is the narrowest maintainable fix.
