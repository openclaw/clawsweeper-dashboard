## Summary

Keep open. Current main still routes Discord voice through the voice-channel session with text delivery suppressed, and the maintainer discussion explicitly left this as dedicated Discord voice feature work.

## What This Changes

Add an opt-in Discord voice bridge so spoken transcripts use a bound text-channel session, appear in text, and have replies posted to text while also playing over TTS.

## Best Possible Solution

Implement this inside the bundled Discord plugin after owner design: typed opt-in bridge config, target validation, speaker-attributed transcript posting, normal text delivery, TTS fan-out, docs/config metadata, and focused voice tests; keep realtime provider work under #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: /vc join and autoJoin resolve a voice-channel route, processDiscordVoiceSegment uses deliver:false, and the strict config/docs expose no bridge-to-text fields.

## Solution Assessment

Unclear. The voice-as-I/O direction is plausible, but the exact configuration and privacy/permission contract need maintainer decisions; a narrow sessionChannelId-only change would not satisfy the full requested behavior.
