## Summary

Keep open. Current main still routes Discord voice turns through the joined voice channel's own session, suppresses normal text delivery, and exposes no bridge-to-text configuration; the maintainer discussion also explicitly left this as dedicated Discord voice feature work. The related #53562/#53741 work overlaps but is narrower and does not supersede the broader voice-as-text-I/O request, while #72891 tracks the separate realtime-provider latency direction.

## What This Changes

Request to make Discord voice channels act as audio input/output for an existing text-channel agent session, with speaker-attributed transcripts and agent replies posted to text while replies are also spoken by TTS.

## Best Possible Solution

Implement the bridge in the bundled Discord plugin after maintainer design: add typed opt-in bridge config, validate the target text channel and permissions, post speaker-attributed transcripts, run the normal text-channel session with delivery enabled, fan out the final reply to TTS, and update Discord voice docs/config metadata/tests. Keep realtime audio-provider support tracked separately under #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: auto-join and `/vc join` only select a voice channel, `join()` resolves the agent route from that voice channel, `processDiscordVoiceSegment()` uses `deliver: false`, and strict config/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested voice-as-I/O direction is plausible, but the exact config shape and permission/privacy contract need maintainer decisions; a narrow `sessionChannelId`-only change would not cover the full transcript posting, normal text delivery, and TTS fan-out behavior requested here.
