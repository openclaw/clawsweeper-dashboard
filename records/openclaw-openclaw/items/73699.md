## Summary

Keep open. Current main still runs Discord voice through the joined voice-channel route with text delivery suppressed, and the issue discussion explicitly leaves the voice-as-I/O bridge as dedicated Discord voice feature work rather than a completed fix or duplicate cleanup.

## What This Changes

Request a Discord voice bridge mode that posts STT transcripts into a bound text channel, runs the text-channel agent session, posts replies to text, and speaks them back over voice.

## Best Possible Solution

Implement the bridge inside the bundled Discord plugin after owner design, with typed opt-in config, validated target resolution, speaker-attributed transcript posting, normal text delivery, TTS fan-out, docs/config metadata, and focused voice tests; keep realtime provider work under #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: /vc join and autoJoin resolve a voice-channel route, processDiscordVoiceSegment uses deliver:false, and strict config/docs expose no bridge-to-text fields.

## Solution Assessment

Unclear. The voice-as-I/O direction is plausible, but the exact config and privacy/permission contract need maintainer decisions; a sessionChannelId-only change would be narrower than the requested behavior.
