## Summary

Keep open. Current main still routes Discord voice through the voice-channel session with `deliver: false`, exposes no bridge-to-text config fields, and the maintainer discussion explicitly left this as dedicated Discord plugin feature work.

## What This Changes

Request an opt-in Discord voice bridge so spoken transcripts and replies use an existing text-channel agent session instead of a separate voice-channel session.

## Best Possible Solution

Add an opt-in Discord plugin bridge with typed config, validated text-channel targeting, speaker-attributed transcript posting, normal text delivery, TTS fan-out, docs, changelog, and focused voice regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `/vc join` and autoJoin carry only the voice channel, the route is resolved from that channel ID, `processDiscordVoiceSegment()` uses `deliver: false`, and strict config/docs expose no bridge option.

## Solution Assessment

Unclear. The requested bridge direction is plausible, but the exact config shape, default behavior, channel permission model, and tool-access contract need maintainer design before this is the safest implementation path.
