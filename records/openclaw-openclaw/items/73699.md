## Summary

Keep open. Current main still routes Discord voice through a voice-channel-derived session, suppresses text delivery for voice turns, and exposes no strict config/schema/docs option for bridging spoken I/O into a bound text-channel session. The maintainer discussion also explicitly left this as dedicated Discord voice feature work rather than cleanup.

## What This Changes

Request a Discord voice bridge mode where spoken transcripts feed a selected text-channel agent session and the reply is both posted to text and spoken back in the voice channel.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed config/schema/docs, explicit bridge target selection, permission checks for the text target, speaker-attributed transcript posting, normal Discord text delivery, and TTS playback of the same final reply; keep realtime audio provider support tracked separately.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice sessions resolve from the voice channel id, voice turns call agentCommandFromIngress() with deliver:false, and the strict Discord voice schema/docs expose no bridge fields.

## Solution Assessment

Unclear. The voice-as-I/O direction belongs in the Discord plugin, but the exact config shape and permission/privacy model should be decided by maintainers before an autonomous implementation is the safest path.
