## Summary

Keep open. Current main still routes Discord voice through the joined voice channel session with text delivery suppressed, and the maintainer discussion explicitly left the requested voice-as-I/O bridge as dedicated Discord plugin feature work.

## What This Changes

Request an opt-in Discord voice bridge where spoken transcripts feed the bound text-channel session, text replies are posted, and replies also play through TTS.

## Best Possible Solution

Implement this in the bundled Discord plugin after owner design: typed opt-in bridge config, target validation, speaker-attributed transcript posting, normal text delivery, TTS fan-out, config metadata, docs, changelog, and focused voice tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: /vc join and autoJoin resolve a voice-channel route, processDiscordVoiceSegment uses deliver:false, and the strict config/docs expose no bridge-to-text fields.

## Solution Assessment

Unclear. The voice-as-I/O direction is plausible, but the exact configuration, privacy, permission, and speaker-attribution contract need maintainer decisions before a narrow automated fix would be safe.
