## Summary

Keep open. Current main now provides partial voice-as-I/O behavior through `voice.agentSession.target` and realtime agent-proxy routing, but the requested automatic text-channel transcript posting and text reply delivery are still not implemented, and the live maintainer discussion says this needs a dedicated feature decision.

## What This Changes

Request to bridge Discord voice input/output into an existing Discord text-channel agent session, including speaker-attributed text transcripts, text-channel replies, and TTS playback.

## Best Possible Solution

Build any text-channel mirroring as an opt-in layer on top of `voice.agentSession.target`, with validated Discord text targets, explicit delivery context, mention policy, speaker-attributed transcripts, TTS fan-out, docs, changelog, and focused tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main path: voice turns can route through `voice.agentSession.target`, but the batch path still uses `deliver: false` and no inspected source posts transcripts or replies to the target text channel.

## Solution Assessment

Unclear. `voice.agentSession.target` is the right foundation for shared context, but completing automatic text mirroring needs explicit product and safety decisions before a narrow implementation is clearly best.
