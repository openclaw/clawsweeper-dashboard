## Summary

Keep open. Current main still implements Discord voice as a separate voice-channel-routed session with suppressed text delivery, and the maintainer discussion explicitly left this as dedicated Discord voice feature work rather than cleanup.

## What This Changes

Add a Discord voice mode where spoken transcripts are routed into a bound text-channel agent session and replies are both posted to text and spoken through TTS.

## Best Possible Solution

Implement the bridge in the bundled Discord plugin with typed config/schema/docs, explicit text target selection, Discord permission checks, speaker-attributed transcript posting, normal text delivery for the routed agent reply, and TTS playback of that same final reply; keep realtime audio provider support tracked separately.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice sessions route through the joined voice channel, voice turns call agentCommandFromIngress() with deliver:false, and the strict Discord voice schema/docs expose no bridge fields.

## Solution Assessment

Unclear. Voice-as-I/O belongs in the Discord plugin, but the exact config shape, bridge target fallback, permission/privacy behavior, and transcript formatting should be settled by maintainers before a safe implementation PR is queued.
