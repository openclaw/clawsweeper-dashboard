## Summary

Keep open. Current main still implements Discord voice as a voice-channel-routed session with text delivery disabled, and maintainer discussion on this issue explicitly says the bridge-to-text behavior was not included in recent Discord fixes. The related transcript-routing PR was closed unmerged, so there is no implemented or superseding core fix to close against.

## What This Changes

Request an opt-in Discord voice mode where voice transcripts are posted into a bound text-channel session and the text-channel agent reply is both posted to text and spoken back over Discord voice.

## Best Possible Solution

Implement this as a bundled Discord plugin feature after maintainer design: add typed opt-in bridge config, validate the text target against Discord permissions and channel bindings, post speaker-attributed transcripts into the chosen text channel, run the normal text-channel session with delivery enabled, and fan out the final response to Discord voice TTS while keeping realtime-audio provider work tracked separately in #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice autoJoin cannot specify a text target, `join()` routes via the voice channel id, `processDiscordVoiceSegment()` uses `deliver: false`, and the strict schema/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested voice-as-I/O direction is plausible and belongs in the Discord plugin, but the proposed config is not yet the full maintainable solution until maintainers decide the target, visibility, permission, transcript, and reply fan-out semantics.
