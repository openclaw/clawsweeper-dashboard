## Summary

Keep open. Current main still routes Discord voice turns through the joined voice channel's session with delivery suppressed, and the maintainer discussion explicitly left the text bridge as dedicated Discord feature work.

## What This Changes

Add a Discord voice-as-I/O mode that posts voice transcripts into a bound text channel, runs the existing text-channel agent session, posts replies, and also speaks them in voice.

## Best Possible Solution

Implement this inside the bundled Discord plugin with typed opt-in bridge config, validated text-channel target selection, speaker-attributed transcript posting, normal text delivery plus voice TTS fan-out, docs/config metadata, and focused voice tests; keep realtime provider work under #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: /vc join or autoJoin resolves a voice-channel route, processDiscordVoiceSegment uses deliver:false, and strict config/docs expose no bridge-to-text fields.

## Solution Assessment

Unclear. The voice-as-I/O direction is plausible, but the exact config and privacy/permission contract need maintainer decisions before a narrow implementation is clearly best.
