## Summary

Keep open. Current main still lacks a Discord voice-as-text-channel-I/O bridge: voice sessions route through the joined voice channel, voice turns suppress text delivery, and the strict config/schema/docs have no bridge target option. Maintainer and related PR history also point to explicit Discord voice feature design rather than cleanup closure.

## What This Changes

Request Discord voice channels to act as microphone and speaker I/O for an existing or bound text-channel agent session, posting transcripts and replies to text while also playing replies over TTS.

## Best Possible Solution

Implement the bridge in the bundled Discord plugin after channel-owner design: add typed opt-in config, validate the text target, post speaker-attributed transcripts, run the text-channel session with normal delivery, fan out the final reply to TTS, update docs/config metadata/tests, and keep realtime audio provider support tracked separately in #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: autoJoin cannot carry a text target, join() routes via the voice channel id, processDiscordVoiceSegment() uses deliver:false, and the strict schema/docs expose no bridge fields.

## Solution Assessment

Unclear. The voice-as-I/O direction is plausible and belongs in the Discord plugin, but the proposed config shape is only partial until maintainers decide the permission, privacy, transcript posting, and target-channel semantics; the prior sessionChannelId PR was closed for that owner direction.
