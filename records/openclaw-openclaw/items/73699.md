## Summary

Keep open. Current main still lacks the requested Discord voice-as-text-session bridge, and the maintainer discussion explicitly left this as dedicated feature work rather than cleanup. The related sessionChannelId PR was closed unmerged, while the narrower related issue remains open, so there is no implemented or superseding fix to close against.

## What This Changes

Request to add a Discord voice bridge mode so spoken transcripts use a bound text-channel agent session, post transcript/reply text, and also play replies back through TTS.

## Best Possible Solution

Implement this in the bundled Discord plugin after maintainer design: add typed opt-in config for bridge mode and target selection, validate the text target and permissions, post speaker-attributed transcripts, run the normal text-channel session with delivery enabled, fan out the final reply to TTS, and update Discord voice docs/config metadata/tests. Keep realtime audio provider work separate under #72891.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: autoJoin and /vc join can only select a voice channel, join() resolves the route from that voice channel id, processDiscordVoiceSegment() uses deliver:false, and strict config/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested voice-as-I/O direction is plausible and belongs in the Discord plugin, but the proposed config shape is not yet clearly the best fix until maintainers settle target-channel, transcript visibility, permission/privacy, and related #53562/#53741 semantics.
