## Summary

Keep open: this is a focused fix for a valid WhatsApp session data-loss regression, but the patch is not merge-ready because it can report a successful backup after swallowing backup failures and then delete the only auth state. The contributor proof is also insufficient because it is a simulated temp-file transcript rather than a real WhatsApp/plugin-update logout path.

## What This Changes

The PR adds timestamped WhatsApp Web Baileys credential backups in `logoutWeb()` before clearing or deleting the managed auth directory.

## Best Possible Solution

Keep the WhatsApp-owned backup-before-clear mitigation, but make backup success/failure explicit, cover both owned and legacy auth-dir branches with focused tests, and require real after-fix proof for the plugin-update-triggered logout path before merge.

## Reproduction Assessment

Yes, at source level: current `main` reaches `logoutWeb()` from logged-out/reconnect cleanup and deletes managed WhatsApp auth directories, and no current pre-logout backup exists. I did not run a live WhatsApp plugin update in this read-only review.

## Solution Assessment

No for the current patch shape. Backup-before-clear is an acceptable owner-local mitigation, but it must not swallow copy failures and then log a successful backup before deleting credentials.
