## Summary

Keep open. The crash class is real, but this PR is not safe to merge as-is because it moves HTTP-header ByteString sanitization into the generic SecretInput boundary, where current code also handles non-header credentials and presence checks. Current main still needs a narrower header-bound fix rather than the global credential mutation proposed here.

## What This Changes

The PR changes `normalizeSecretInputString` to delegate to `normalizeOptionalSecretInput` and adds direct tests for trimming, blank inputs, non-Latin1 stripping, and Latin-1 preservation.

## Best Possible Solution

Keep `normalizeSecretInputString`, `normalizeResolvedSecretInputString`, and `hasConfiguredSecretInput` trim-only for generic credential semantics, then apply `normalizeSecretInput` or a clearly named header-safe helper only where values are assembled into HTTP headers, starting with Comfy Cloud `X-API-Key` and shared provider request auth/header paths.

## Reproduction Assessment

Yes. The ByteString failure reproduces with Node's `Headers` constructor for U+2502, and current main traces a Comfy Cloud config API key from trim-only normalization into `X-API-Key`. The PR regression is also source-reproducible by applying the proposed sanitizer to non-header Unicode credentials such as a Matrix password.

## Solution Assessment

No. The PR uses the right sanitizer for the wrong boundary: ByteString constraints apply to HTTP headers, not to every generic secret value or presence check. The safer alternative is header-bound sanitization with regression tests proving non-header Unicode credentials remain unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Unicode secrets outside HTTP headers:** `src/config/types.secrets.ts:116`
  - body: Delegating `normalizeSecretInputString` to `normalizeOptionalSecretInput` strips every code point above `0xff` for all SecretInput consumers. Matrix password auth resolves `channels.matrix.password` through this generic path and sends it in a JSON login body, so a valid password like `пароль123` would become `123`. Keep this helper trim-only and sanitize only values that are assembled into HTTP headers.
  - confidence: 0.91
- **[P2] Keep presence checks independent from header sanitization:** `src/config/types.secrets.ts:116`
  - body: `hasConfiguredSecretInput` calls this helper as a presence check. After this change, an all-non-Latin1 configured value such as `пароль` normalizes to `undefined`, so gateway auth can treat a real password as absent and take unrelated token-ref or missing-credential paths. Presence checks should not reuse a header-value sanitizer.
  - confidence: 0.86
