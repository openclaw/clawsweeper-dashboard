## Summary

Keep this PR open, but do not merge it as-is. The reported ByteString crash is real on current main for header-bound secret values, but this patch moves header sanitization into the generic SecretInput helper, which would silently mutate valid non-header credentials and change auth presence checks.

## What This Changes

The PR changes `normalizeSecretInputString` to delegate to `normalizeOptionalSecretInput` and adds direct tests for trimming, blank input, non-Latin1 stripping, and Latin-1 preservation.

## Best Possible Solution

Keep `normalizeSecretInputString`, `normalizeResolvedSecretInputString`, and `hasConfiguredSecretInput` trim-only for generic credential semantics, then apply `normalizeSecretInput` or a clearly named header-safe helper only where configured values are assembled into HTTP headers.

## Reproduction Assessment

Yes. The ByteString failure reproduces with Node's `Headers` constructor for U+2502, and current main has a concrete Comfy Cloud path from trim-only API key normalization into `X-API-Key`. The PR regression is also source-reproducible from generic consumers such as Matrix password login and gateway presence checks.

## Solution Assessment

No. The patch uses the right sanitizer at the wrong boundary: ByteString constraints apply to HTTP headers, not every generic secret value. The safer alternative is header-bound sanitization with regression coverage proving non-header Unicode credentials remain unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Unicode secrets outside HTTP headers:** `src/config/types.secrets.ts:116`
  - body: Delegating `normalizeSecretInputString` to `normalizeOptionalSecretInput` strips every code point above `0xff` for all SecretInput consumers. Matrix password auth resolves `channels.matrix.password` through this generic path and sends it in a JSON login body, so a valid password like `–ø–∞—Ä–æ–ª—å123` would become `123`. Keep this helper trim-only and sanitize only values assembled into HTTP headers.
  - confidence: 0.91
- **[P2] Keep presence checks independent from header sanitization:** `src/config/types.secrets.ts:116`
  - body: `hasConfiguredSecretInput` calls this helper as a presence check. After this change, an all-non-Latin1 configured value such as `–ø–∞—Ä–æ–ª—å` normalizes to `undefined`, so gateway auth and setup surfaces can treat a real password as absent and take unrelated token-ref or missing-credential paths. Presence checks should not reuse a header-value sanitizer.
  - confidence: 0.87
