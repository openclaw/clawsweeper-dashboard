## Summary

Keep this PR open for maintainer follow-up, but do not merge it as written. Current main still has a header-bound ByteString crash path for pasted non-Latin1 API-key values, while the PR fixes it by changing generic SecretInput normalization in a way that can mutate valid Unicode credentials and alter configured-presence checks outside HTTP headers.

## What This Changes

The PR changes `normalizeSecretInputString` to delegate to `normalizeOptionalSecretInput` and adds direct tests for trimming, blank inputs, non-Latin1 stripping, and Latin-1 preservation.

## Best Possible Solution

Adopt a narrow header-bound hardening path: keep `normalizeSecretInputString`, `hasConfiguredSecretInput`, and non-header credentials trim-only, and apply `normalizeSecretInput` or a clearly named header-safe helper where values are assembled into HTTP headers. Start with Comfy Cloud `X-API-Key` and shared provider request auth/header assembly, with regression tests for pasted U+2502 plus guard tests proving Unicode Matrix and gateway password values remain unchanged.

## Reproduction Assessment

Yes. Current main can be traced from a Comfy Cloud config API key through trim-only normalization into `X-API-Key`, and a read-only Node smoke check confirms `new Headers({ 'X-API-Key': 'â”‚sk-test' })` throws the ByteString TypeError. The PR regression is also reproducible by applying the proposed sanitizer semantics to non-header Unicode credentials such as `ĐżĐ°Ń€ĐľĐ»ŃŚ123`.

## Solution Assessment

No. The PR uses an existing sanitizer, but applies it at the generic SecretInput boundary instead of the HTTP-header boundary where ByteString constraints actually apply. The safer solution is to preserve generic trim-only semantics and sanitize only header values before constructing request headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Unicode secrets outside HTTP headers:** `src/config/types.secrets.ts:116`
  - body: Delegating `normalizeSecretInputString` to `normalizeOptionalSecretInput` strips every code point above `0xff` for all SecretInput consumers. That includes non-header auth paths such as Matrix password login, where a valid password like `ĐżĐ°Ń€ĐľĐ»ŃŚ123` is sent in a JSON body but would become `123` after this change. Keep this helper trim-only and apply ByteString hardening only where values are assembled into HTTP headers.
  - confidence: 0.91
- **[P2] Keep presence checks independent from header sanitization:** `src/config/types.secrets.ts:116`
  - body: `hasConfiguredSecretInput` calls this helper as a presence check. After this change, a configured all-non-Latin1 value such as `ĐżĐ°Ń€ĐľĐ»ŃŚ` normalizes to `undefined`, so gateway auth can treat a real password as absent and resolve or require unrelated token refs. Presence checks should not reuse a header-value sanitizer.
  - confidence: 0.86
