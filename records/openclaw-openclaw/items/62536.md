## Summary

Keep open: current main still has the source-level log leak, but this PR cannot land as-is because it is based on the old delivery flow and still lets free-form labels reach info logs.

## What This Changes

The PR routes subagent announce output and announce retry/error logs through an `agents/announce` subsystem logger with a focused delivery test.

## Best Possible Solution

Land a rebased announce-specific logger change on the current payload-plan delivery path, keeping info logs to stable ids, status, and counts while moving body/task/label text to sanitized debug logs.

## Reproduction Assessment

Yes, at source level. Direct announce calls carry `sourceTool: "subagent_announce"`, current delivery logging still sends visible formatted payload text to `runtime.log(output)`, and gateway console capture writes that path at info level.

## Solution Assessment

No, not as submitted. The subsystem logger boundary is the right direction, but the patch needs to target the current payload-plan path and keep free-form labels out of info logs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep free-form labels out of info logs:** `src/agents/subagent-announce.ts:597-600`
  - body: `params.label` is caller-provided session/task text, and interpolating it into an info-level announce log can leak sensitive or multiline content into default operational logs. Keep info logs to stable metadata and move any label/task text to sanitized debug output.
  - confidence: 0.9
- **[P2] Rebase announce logging onto payload plans:** `src/agents/command/delivery.ts:296-305`
  - body: This change is inserted into the old `normalizeOutboundPayloads` delivery flow, but current main builds an outbound payload plan and handles no-visible payload status before logging. Rebase the announce split onto `projectOutboundPayloadPlanForOutbound` so it applies to current delivery semantics.
  - confidence: 0.86
