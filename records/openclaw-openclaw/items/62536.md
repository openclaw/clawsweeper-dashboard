## Summary

Keep this PR open: the announce logging fix is still useful on current main, but the branch is not mergeable as-is because it logs free-form labels at info level and is based on the pre-payload-plan delivery flow.

## What This Changes

The PR routes subagent announce delivery output and announce warnings/errors through an `agents/announce` subsystem logger, with metadata at info level and full announce bodies at debug level.

## Best Possible Solution

Land a rebased announce-specific logging change that keeps default info logs to stable metadata only, moves free-form announce/body text to debug, and preserves normal non-announce delivery logging behavior.

## Reproduction Assessment

Yes, source-level. Current main carries `sourceTool: "subagent_announce"` into direct announce agent calls, gateway runs use `defaultRuntime`, and non-json delivery payload text still reaches `runtime.log(output)` before console capture writes info/file logs.

## Solution Assessment

No, not as-is. The subsystem logger boundary is the right direction, but the PR needs a current-main rebase and must remove free-form label text from info-level operational logs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep free-form labels out of info logs:** `src/agents/subagent-announce.ts:600`
  - body: `params.label` comes from the optional `sessions_spawn.label` string and is not constrained for content or control characters, so putting it in the default `announce delivery` info line can persist user/model-authored context and pollute operational logs. Keep info logs to stable ids/status/lengths and move labels or task text to sanitized debug output only.
  - confidence: 0.9
- **[P3] Rebase announce routing onto payload plans:** `src/agents/command/delivery.ts:299`
  - body: This branch adds the announce logging split around the older `normalizeOutboundPayloads` flow, but current main now builds an outbound payload plan and handles no-visible-payload statuses before logging. Rebase the change onto the current `projectOutboundPayloadPlanForOutbound` flow so the fix is mergeable and does not bypass current delivery semantics.
  - confidence: 0.78
