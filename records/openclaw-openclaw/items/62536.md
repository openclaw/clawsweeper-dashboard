## Summary

Keep open: the source-level announce logging leak path still exists on current main, but this branch should not merge as-is because it is stale against the current payload-plan delivery flow and still emits free-form label text at default info level. The contributor supplied useful real log proof, so the right next step is a focused repair or rebase rather than closing the PR.

## What This Changes

The PR routes subagent announce delivery output and announce warn/error messages through an `agents/announce` subsystem logger, adds a regression test, and leaves normal delivery logging unchanged.

## Best Possible Solution

Land a rebased announce-specific logging fix that keeps info logs to stable ids, status, and counts only, moves announce body and task text to debug, and preserves current outbound payload-plan/no-visible-payload semantics.

## Reproduction Assessment

Yes, source-level. Direct announce calls mark `sourceTool: "subagent_announce"`, current delivery still formats visible payload text and calls `runtime.log(output)`, and gateway console capture records that at info level.

## Solution Assessment

No, not as-is. The subsystem logger boundary is the right direction, but the patch must be rebased onto the current payload-plan path and must keep free-form label/task text out of info logs.
