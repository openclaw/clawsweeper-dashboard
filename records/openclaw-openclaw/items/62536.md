## Summary

Keep open: current main still has the reported announce-log leak path, and this PR is useful but not merge-ready because it leaves a free-form label in default info logs, misses the empty-payload path, lacks the required changelog entry, and is dirty against current main.

## What This Changes

The PR routes subagent announce delivery output and announce-related warning/error logging through an `agents/announce` subsystem logger, with announce body text intended for debug-level logs and metadata for info-level logs.

## Best Possible Solution

Rebase or replace the PR on current main, keep announce info logs to stable metadata only, route empty announce replies through the same subsystem boundary, add a changelog entry, and land with focused agent logging tests plus the changed gate.

## Reproduction Assessment

Yes. Source inspection shows `subagent_announce` delivery reaches `deliverAgentCommandResult()`, which currently formats payload text and calls `runtime.log(output)`, while `runtime.log` delegates to console capture at info level; I did not run a live gateway in this read-only review.

## Solution Assessment

No, not as-is. The subsystem logger boundary is the right fix direction, but the PR still needs to omit free-form labels from info logs, handle empty announce payloads, add the changelog entry, and adapt to current main.
