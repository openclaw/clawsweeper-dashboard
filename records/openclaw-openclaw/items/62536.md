## Summary

Keep open: the reported announce-body log leak is still source-reproducible on current main, and this PR is useful but not mergeable as-is because it is conflicting against the current delivery flow and still writes a free-form spawn label into default info logs.

## What This Changes

The PR routes subagent announce delivery logging through an `agents/announce` subsystem logger with metadata at info level and full announce bodies at debug level.

## Best Possible Solution

Land a rebased implementation that routes announce-run delivery output through subsystem logging with stable metadata-only info entries, sanitized/debug-only free-form text, and regression tests that preserve non-announce delivery behavior.

## Reproduction Assessment

Yes, source-level. Current main carries `sourceTool: "subagent_announce"` into direct announce agent calls, gateway runs use `defaultRuntime`, and non-json delivery payload text still reaches `runtime.log(output)` before console capture writes info/file logs.

## Solution Assessment

No, not as-is. The subsystem logger boundary is the right approach, but the PR must be rebased onto the current outbound payload-plan flow and keep default info logs to stable metadata only.
