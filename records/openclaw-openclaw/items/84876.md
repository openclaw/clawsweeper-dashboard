## What This Changes

The PR adds a Telegram outbound guard that blocks product/customer updates containing source paths or dense technical vocabulary and adds regression tests for blocked and allowed examples.

## Best Possible Solution

Keep the transport layer generic and enforce product/customer wording only from an explicit product-update metadata or hook path, with tests proving ordinary technical Telegram messages still send.

## Reproduction Assessment

Yes by source inspection, if the PR diff is applied: a Telegram message containing a product-context word plus three jargon terms reaches `assertTelegramProductPlainLanguage` and throws before `sendText` or `sendPayload`. I did not execute tests because this review is read-only.

## Solution Assessment

No. A lexical guard in the shared Telegram transport is broader than the documented `message_sending` hook policy surface; the maintainable path is an explicit product-update signal or hook-based policy with compatibility coverage.
