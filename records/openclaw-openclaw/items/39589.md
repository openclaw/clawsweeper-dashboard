## Summary

Keep this issue open. Current main still rejects the provider-only `models.providers.openai` shape before memory embeddings can use the configured `baseUrl`, and open PR #39766 already targets this issue with closing syntax.

## What This Changes

The issue asks for memory embeddings to honor a custom `models.providers.openai.baseUrl` when the provider entry is used only for endpoint/auth configuration.

## Best Possible Solution

Keep the issue open until #39766 or an equivalent focused fix lands, aligning the Zod schema, generated schema, TypeScript config type, and regression tests around provider-only entries defaulting `models` to `[]`.

## Reproduction Assessment

Yes. The source-level path is high confidence: the reporter's config omits `models`, current schema/type/generated contracts require it, and validation rejects the config before the embedding runtime can read `models.providers.openai.baseUrl`.

## Solution Assessment

Yes. The narrow maintainable fix is to accept missing provider `models` at the config boundary with a default empty array and keep generated/type/test surfaces aligned; routing users to `memorySearch.remote.baseUrl` would only be a workaround.
