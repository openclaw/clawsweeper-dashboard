## Summary

Keep this issue open. Current main still rejects the reporter's provider-only `models.providers.openai` entry because the config schema, generated schema, and TypeScript config contract still require `models`, while the embedding runtime would honor the provider `baseUrl` once validation lets the entry through. The issue is also paired with open PR #39766 using closing syntax, so it should remain open until that or an equivalent fix lands.

## What This Changes

The reporter wants `memory-core` OpenAI embeddings to use `models.providers.openai.baseUrl` for custom OpenAI-compatible endpoints even when that provider entry contains only endpoint/auth configuration and no `models` array.

## Best Possible Solution

Keep this issue open until #39766 or an equivalent focused fix lands. The end state should make `models.providers.<id>.models` optional with a validation-time default of `[]`, align generated schema and TypeScript config types, and cover provider-only OpenAI-compatible embedding config in regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is the reporter's config with `models.providers.openai` omitting `models`: current `OpenClawSchema.safeParse` rejects it because `ModelProviderSchema` and the generated schema still require `models`.

## Solution Assessment

Yes. The narrowest maintainable fix is to default missing provider `models` to an empty array at the config boundary and update all generated/type surfaces; using only `memorySearch.remote.baseUrl` is a workaround, not a fix for the provider config contract.
