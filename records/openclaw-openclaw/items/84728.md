## Summary

Keep open: the bug and repair direction are valid, and the supplied real-rollout proof is sufficient, but the patch adds an unbounded whole-file read path before every bound Codex resume. Current main already treats large Codex rollouts carefully with optional byte guards and streamed token scans, so the repair needs to preserve that runtime behavior before merge.

## What This Changes

The PR adds a Codex app-server transcript repair helper that inserts synthetic custom_tool_call_output records for orphaned custom_tool_call items before resuming a bound native Codex thread.

## Best Possible Solution

Land a bounded or streaming Codex transcript repair that modifies only corrupted rollouts before resume while preserving the existing large-rollout startup guard semantics.

## Reproduction Assessment

Yes. The linked issue and PR body include live 2026.5.19 restart logs plus a redacted native rollout shape, and upstream Codex 0.132.0 source confirms that a custom_tool_call without custom_tool_call_output triggers the reported missing-output error.

## Solution Assessment

No, not as implemented. Repairing the bound Codex rollout before resume is the right boundary, but the implementation should stream or bound the scan and only rewrite after detecting corruption.
