## Summary

Current main still uses a native Codex hook relay with a separate helper invocation for configured hook events. The broader performance report at https://github.com/openclaw/openclaw/issues/84037 remains related, but this issue contributes focused active-turn CPU samples, relay timing data, and a useful falsification of the workspace-hook explanation, so it remains actionable rather than redundant.

## What This Changes

Reduce CPU-heavy short-lived `openclaw-hooks` relay helper processes during active native Codex turns.

## Best Possible Solution

Instrument per-event helper startup and execution time, reproduce under an active current-main Codex turn, then optimize the existing command path or elide only events proven to have no registered work; pursue an upstream persistent hook transport only if subprocess startup is confirmed as the dominant cost.

## Reproduction Assessment

No, not at high confidence: source inspection confirms the helper mechanism and the report contains detailed live measurements, but this review did not independently reproduce the CPU saturation on current main.

## Solution Assessment

Unclear. Reducing helper startup or provably empty invocations is likely the narrowest fix, while pooling helpers changes the upstream Codex execution model and requires maintainer and dependency-owner judgment.
