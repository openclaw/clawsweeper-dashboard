## Summary

Keep open: current main and the latest release still contain the reported macOS location-permission polling shape, and all three linked fix PRs were closed unmerged. The issue is source-reproducible enough to route to a narrow fix, but any landing PR should include signed macOS app TCC-count and permission-freshness proof.

## What This Changes

The issue reports repeated macOS TCC location permission requests because OpenClaw recreates `CLLocationManager` during permission polling instead of reusing a retained manager.

## Best Possible Solution

Land a narrow macOS permission fix that reuses a retained or delegate-owned `CLLocationManager` for authorization-status reads, with signed-app before/after `TCCAccessRequest` counts and grant/revoke/Settings freshness proof.

## Reproduction Assessment

Partially. The reporter supplied concrete `log show` commands, and current source shows the one-second permission monitor reaches fresh `CLLocationManager` authorization reads, but this review did not run a signed macOS app to count TCC events.

## Solution Assessment

Likely yes, with proof. Reusing a retained manager is the narrow maintainable fix; if live proof shows stale authorization state, route status reads through a delegate-owned location owner instead of a separate static status manager.
