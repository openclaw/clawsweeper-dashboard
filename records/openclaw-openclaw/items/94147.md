- `impact:security`: The issue concerns repeated CoreLocation/TCC permission-boundary activity that can look like excessive location permission access.

## Summary

Keep open: current main and the latest release still have the one-second macOS permission monitor reaching fresh `CLLocationManager()` authorization reads, and the same-change PRs linked from the issue were closed unmerged rather than landing a fix.

## What This Changes

The issue asks OpenClaw’s macOS app to stop rebuilding `CLLocationManager` during permission polling because that reportedly floods macOS TCC permission logs.

## Best Possible Solution

Land a narrow macOS permission fix that reuses a retained or delegate-owned location manager for authorization-status reads, with signed OpenClaw.app before/after TCC counts and grant/revoke Settings refresh proof.

## Reproduction Assessment

Yes for a source-backed path, but not from a live signed-app run in this review. Current main shows the one-second monitor calling `PermissionManager.status()` and fresh `CLLocationManager()` authorization reads; a live repro should use the reporter's `log show --process OpenClaw` TCC count commands on a signed app.

## Solution Assessment

The requested direction is likely the right class of fix, but current main is not fixed. A retained manager is the narrow mitigation; if live proof shows stale authorization state, the safer shape is a delegate-owned shared location owner.
