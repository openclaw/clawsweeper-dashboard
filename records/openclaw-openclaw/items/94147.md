## Summary

Keep open: current main and the latest release still route macOS permission polling through fresh `CLLocationManager().authorizationStatus` reads, and the macOS cache-fix PRs linked from the issue are all closed unmerged. The reporter added concrete v2026.6.9 live TCC log evidence, so this remains actionable but should stay in maintainer/security/live-repro review before another fix attempt.

## What This Changes

The issue reports that the macOS app repeatedly rebuilds `CLLocationManager` during permission polling, producing frequent TCC permission IPC/log activity.

## Best Possible Solution

Land a focused macOS permission-status fix that avoids per-poll TCC-sensitive work in `PermissionManager.status()` and proves, on a signed OpenClaw.app, that TCC events no longer burst while permission reporting remains correct.

## Reproduction Assessment

Yes for the source path: current main has the 1-second permission monitor and node-mode permission snapshot path reaching fresh `CLLocationManager` authorization reads. The reporter also supplied v2026.6.9 live TCC log counts, but this review did not independently run the signed macOS app.

## Solution Assessment

Yes, the maintainable direction is a lifecycle-owned CoreLocation manager and/or permission-status cache in the macOS permission owner, not repeated framework queries from polling callers. The fix should cover both settings and node-mode/status callers and include signed before/after proof.
