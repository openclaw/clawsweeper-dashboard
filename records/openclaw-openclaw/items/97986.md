## Summary

Keep this PR open, but it needs one compatibility fix before merge: the QQBot filename restore should be scoped to known OpenClaw media-store paths instead of applying to every UUID-shaped filename.

## What This Changes

The PR adds a QQBot outbound filename helper and routes file sends through it so staged media-store attachment names are displayed without the internal UUID suffix.

## Best Possible Solution

Keep the QQBot transport fix, but mirror the current shared-media guard: strip the UUID suffix only for paths proven to be under the OpenClaw media store and preserve sanitized basenames everywhere else.

## Reproduction Assessment

Yes, from source inspection: current QQBot document paths pass `sanitizeFileName(path.basename(...))` into QQ file sends, so a media-store staged basename remains recipient-visible today. I did not run a live QQBot send in this read-only review.

## Solution Assessment

No, not yet. The PR targets the right QQBot file-send boundary, but it needs the same media-store scoping used by current shared media code before it is the safest fix.

- **[P1] Preserve UUID-shaped user filenames outside the media store:** `extensions/qqbot/src/engine/messaging/outbound-file-name.ts:21-22`
  - body: `extractOriginalFilename` removes any basename ending in `---<uuid>` before the extension, but this helper applies it to every QQBot file path and URL. Scope the restore to known media-store paths and keep the sanitized basename elsewhere.
