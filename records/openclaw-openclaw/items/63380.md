## Summary

Keep open. Current main still rejects account-level Telegram `agentId`, and this PR is the paired implementation candidate, but it is not merge-ready because real Telegram/gateway proof is missing and the branch needs rebase, generated metadata, and SDK import cleanup.

## What This Changes

The PR adds `channels.telegram.accounts.<id>.agentId` to Telegram config schema/types, routes unmatched named-account traffic to that account agent, and adds schema/routing tests.

## Best Possible Solution

Rebase onto current main, then either land a documented compatibility restoration for account-level `agentId` with generated metadata and live proof, or close this path in favor of a `bindings[]` migration/doctor guidance decision.

## Reproduction Assessment

Yes, at source level. Current main's strict Telegram account schema/type omit account-level `agentId`, and the runtime routing path only consumes topic `agentId` plus bindings, matching the linked regression report.

## Solution Assessment

Unclear. Restoring account-level `agentId` may be the right compatibility fix, but this branch still needs rebase cleanup, generated metadata/docs alignment, and real Telegram or gateway proof before it is the maintainable solution.
