## Summary

Keep open: current main still lacks account-level Telegram `agentId`, the PR is the active linked fix candidate, and the branch still has correctness blockers plus missing real behavior proof before it can merge.

## What This Changes

The PR adds Telegram account-level `agentId` schema/type support, routes it as a default when no explicit binding matches, rejects top-level `channels.telegram.agentId`, and adds schema/routing tests.

## Best Possible Solution

Land this PR or an equivalent focused patch only after maintainers accept the compatibility direction, the branch is rebased onto current Telegram config tests, generated metadata/docs are aligned, and real Telegram or gateway proof is provided.

## Reproduction Assessment

Yes, at source level. Current main uses a strict Telegram account schema and account type without `agentId`, while route resolution consumes topic `agentId` plus bindings only, so the reported account config remains rejected or unrouted without a live gateway run.

## Solution Assessment

No, not as submitted. The compatibility direction is plausible, but generated metadata, stale test placement, narrow SDK imports, account precedence wording/docs, and real behavior proof need to be fixed before this is the maintainable solution.
