## Summary

Keep open. Current main still rejects Telegram account-level `agentId`, and this PR is the active paired fix candidate, but it still needs real behavior proof plus a few concrete alignment fixes before merge.

## What This Changes

The PR adds Telegram account-level `agentId` schema/type support, default routing from account config when no explicit binding matches, top-level rejection coverage, and schema/routing regression tests.

## Best Possible Solution

Land this PR or an equivalent focused patch only after the compatibility direction is accepted: account-level `agentId` should validate, route as a fallback after explicit bindings, update generated metadata/docs, and include real Telegram or gateway proof.

## Reproduction Assessment

Yes, at source level. Current main uses a strict Telegram account schema without `agentId`, and route resolution only consumes topic-level `agentId` plus bindings, so the reported account config remains rejected or unrouted without a live gateway run.

## Solution Assessment

No, not as submitted. The compatibility direction is plausible, but generated metadata, public docs/comment wording, narrow SDK import usage, rebase state, and real behavior proof need to be aligned before this is the maintainable fix.
