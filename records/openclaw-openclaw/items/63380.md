## Summary

Keep open. The PR is a plausible fix for the Telegram account-level `agentId` regression, but it still lacks after-fix real behavior proof and has blocking metadata/import/docs follow-ups before it is safe to merge.

## What This Changes

The branch adds Telegram account-level `agentId` schema/type support, default routing when no explicit binding matches, top-level rejection tests, and schema/routing regression tests.

## Best Possible Solution

Land this PR or an equivalent focused patch only if maintainers choose compatibility: account-level `agentId` should validate, route only as a default after explicit bindings, update generated metadata and docs, use narrow SDK imports, and include real Telegram or gateway proof.

## Reproduction Assessment

Yes, at source level. Current main parses Telegram accounts with a strict schema that lacks `agentId`, and Telegram runtime only consumes topic-level `agentId`, so the reported account config remains rejected or unrouted without a live gateway run.

## Solution Assessment

No, not as submitted. The compatibility direction is plausible, but generated metadata, public docs/comment wording, narrow SDK import usage, rebase state, and real behavior proof need to be aligned before this is the maintainable fix.
