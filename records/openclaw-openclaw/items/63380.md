## Label Justifications

- `P1`: This PR targets an upgrade regression that can prevent Telegram multi-account gateways from starting or routing messages correctly.
- `merge-risk: 🚨 compatibility`: The PR changes accepted Telegram config shape and upgrade behavior for account-level `agentId`, which overlaps with the currently documented `bindings[]` route.
- `merge-risk: 🚨 message-delivery`: Incorrect precedence between account defaults, explicit bindings, and topic overrides could route Telegram messages to the wrong agent or suppress them.
- `merge-risk: 🚨 session-state`: The patch changes route agent IDs and session keys for Telegram accounts, which can split or move existing conversation state.

## Summary

Keep open. The regression remains source-reproducible on current main and this PR is the active linked fix candidate, but it is not merge-ready because the branch is conflicting, lacks real Telegram/gateway proof, and still needs config metadata/test-location cleanup.

## What This Changes

The PR adds account-level `channels.telegram.accounts.<id>.agentId` schema/types, applies it as a Telegram default route when no explicit binding matches, and adds schema/routing tests.

## Best Possible Solution

Rebase the PR onto current main, then either land a fully documented compatibility restoration for account-level `agentId` with generated metadata and live proof, or close it in favor of a bindings-based migration plus clearer doctor/docs guidance.

## Reproduction Assessment

Yes, at source level. Current main's strict Telegram account schema/type omit account-level `agentId`, and the runtime routing path only consumes topic `agentId` plus bindings, matching the linked regression report.

## Solution Assessment

Unclear. Restoring account-level `agentId` may be the right compatibility fix, but this branch still needs rebase cleanup, generated config metadata/docs, and real Telegram or gateway proof before it is the maintainable solution.
