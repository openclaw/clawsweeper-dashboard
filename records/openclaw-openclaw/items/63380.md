## Summary

Keep open. The PR addresses a real source-reproducible Telegram multi-account regression, but it still needs real behavior proof plus generated metadata, docs/comment, and import-boundary follow-ups before merge.

## What This Changes

The PR adds account-level Telegram `agentId` schema/type support, routes it as an account default when no binding matches, rejects top-level Telegram `agentId`, and adds schema/routing regression tests.

## Best Possible Solution

Land this PR or an equivalent focused patch only if maintainers choose compatibility: account-level `agentId` should validate, route only as a default after explicit bindings, update generated config metadata and docs, and include real Telegram or gateway proof; otherwise improve doctor/docs to migrate users to `bindings[]`.

## Reproduction Assessment

Yes, at source level. Current main parses Telegram accounts with a strict schema that lacks `agentId`, and Telegram runtime only consumes `topicConfig?.agentId`, so the reported config remains rejected or unrouted without a live gateway run.

## Solution Assessment

No, not as submitted. The runtime direction is plausible, but generated metadata, public docs/comment wording, narrow SDK import usage, and real behavior proof need to be aligned before this is the maintainable fix.
