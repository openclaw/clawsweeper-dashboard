## Summary

Keep this PR open. The provider addition is useful and has real live proof, but the OpenRouter audio streaming implementation currently decodes streamed base64 chunks in a way that can corrupt audio when chunks are split at arbitrary base64 boundaries.

## What This Changes

The PR adds fal and OpenRouter music-generation providers plus SDK helpers, onboarding scopes, docs, tests, live coverage, and a small run-node test type cleanup.

## Best Possible Solution

Keep the provider work, but fix OpenRouter streaming assembly to collect base64 strings and decode once after completion, with a regression test for non-aligned chunk splits.

## Reproduction Assessment

Yes, at source level: split a single base64 audio payload at non-4-character boundaries and independent per-chunk decoding corrupts the bytes, which matches the changed OpenRouter stream reader path.

## Solution Assessment

No. The PR should follow OpenRouter's documented stream contract by joining `delta.audio.data` chunks before one base64 decode; the rest of the provider shape can remain focused.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Decode streamed audio after joining chunks:** `extensions/openrouter/music-generation-provider.ts:134`
  - body: OpenRouter documents audio streaming as collecting `delta.audio.data` strings, joining them, then base64-decoding once. Decoding every SSE chunk independently corrupts audio whenever the provider splits the base64 stream on a non-4-character boundary, so generated music can be invalid even though aligned test chunks pass. ([openrouter.ai](https://openrouter.ai/docs/guides/overview/multimodal/audio?utm_source=openai))
