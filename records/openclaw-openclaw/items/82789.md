## Summary

Keep open: this is a new bundled provider feature with the protected `maintainer` label, and the latest head still has a concrete OpenRouter streaming bug that can corrupt generated audio.

## What This Changes

The PR adds fal and OpenRouter music-generation providers for `music_generate`, expands provider manifests/onboarding/docs/SDK helpers, and updates related tests and live coverage.

## Best Possible Solution

Keep the provider feature, but repair OpenRouter audio stream assembly to join base64 fragments before decoding and add a non-aligned chunk regression test before maintainer review.

## Reproduction Assessment

Yes, at source level: latest PR head decodes each OpenRouter audio SSE fragment separately, and a non-4-character base64 split reproduces corrupted bytes without needing live API access.

## Solution Assessment

No, the PR should follow OpenRouter's stream contract by collecting `delta.audio.data` strings and decoding once after completion; that is narrower than changing provider selection or the shared music runtime.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Decode streamed audio after joining chunks:** `extensions/openrouter/music-generation-provider.ts:134`
  - body: OpenRouter documents audio streaming as collecting `delta.audio.data` strings, joining them, then base64-decoding once. Decoding every SSE fragment independently corrupts audio when the provider splits base64 on a non-4-character boundary, so generated tracks can be invalid even for a valid provider response. ([openrouter.ai](https://openrouter.ai/docs/guides/overview/multimodal/audio?utm_source=openai))
