## Summary

Keep open: the hardening target is valid, but the current head has blocking security/API-boundary issues and only mock-based proof for a Telegram authorization change. The protected maintainer label also means this PR should stay in explicit maintainer review rather than cleanup close.

## What This Changes

The PR changes Telegram target writeback to require `operator.admin` or an internal authority marker, threads that marker through shared send and queue paths, and adds regression coverage.

## Best Possible Solution

Keep the hardening, but make writeback authority core-owned and non-forgeable at the Telegram/core boundary, ensure gateway scopes win whenever present, and add redacted live Telegram or equivalent terminal/log proof before merge.

## Reproduction Assessment

Yes from source inspection: current main treats missing `gatewayClientScopes` as allowed for Telegram target writeback, so a scoped-origin path that loses scope metadata can persist config/cron target normalization. I did not run tests because this review was read-only.

## Solution Assessment

No: requiring admin for gateway writeback is the right behavior, but this implementation is not the best boundary because it exposes a self-asserted internal authority marker through public/shared channel APIs. A safer fix keeps that authority core-owned and non-forgeable, then proves the real Telegram path.

Full review comments:

- **[P1] Keep internal writeback authority out of public channel contexts:** `src/channels/plugins/outbound.types.ts:38`
  - body: This adds `targetWritebackAuthority` to `ChannelOutboundContext`, which is exported through `openclaw/plugin-sdk/channel-contract` for channel plugins. That makes a Telegram-specific internal persistence authority part of the shared plugin API, so external or non-Telegram callers can receive or set a marker that should be a core-owned authorization fact instead of a public send option.
  - confidence: 0.88
- **[P1] Make gateway scopes override internal writeback authority:** `src/channels/message/send.ts:177-178`
  - body: `targetWritebackAuthority` wins before checking `gatewayClientScopes`, so a caller can provide both `gatewayClientScopes: ["operator.write"]` and `targetWritebackAuthority: "internal"` and still reach Telegram writeback as internal. For this admin gate, scoped gateway facts need to dominate, or the internal authority must be derived from a non-forgeable core origin instead of caller input.
  - confidence: 0.86
