## Summary

## What This Changes

The PR requires admin scope or internal authority for Telegram target writeback and threads scope/authority data through Telegram sends, channel sends, cron delivery, CLI sends, and queue replay tests.

## Best Possible Solution

Keep the admin writeback hardening, but derive authority from a closed core/Telegram origin, make explicit gateway scopes deny without `operator.admin` regardless of internal markers, and add real Telegram or equivalent redacted runtime proof before merge.

## Reproduction Assessment

Yes from source inspection: current main treats missing `gatewayClientScopes` as allowed for Telegram target writeback, so a scoped-origin path that loses scope metadata can persist config/cron target normalization. I did not run tests because this review was read-only.

## Solution Assessment

No: requiring admin for gateway writeback is the right behavior, but this implementation exposes a caller-supplied internal authority marker through public/shared channel APIs. A safer fix keeps authority core-owned and makes explicit gateway scopes dominate.
