## Summary

This PR should stay open for maintainer review: the Telegram authorization hardening is useful, but the current patch turns a Telegram target-writeback decision into a shared channel/plugin/queue contract and still lacks real Telegram or equivalent runtime proof.

## What This Changes

The PR requires `operator.admin` or an internal authority before Telegram target resolution writes back config or cron targets, and threads that authority through Telegram send/action paths, shared outbound delivery, cron delivery, queue replay, and tests.

## Best Possible Solution

Keep the writeback authorization decision scoped to Telegram-owned runtime boundaries, or explicitly approve and document a generic persisted channel authority contract, then require redacted real Telegram or equivalent runtime proof before merge.
