## Summary

Keep open: current main still has a source-reproducible path for inline Google/Gemini service secrets, and the linked fixing PRs are still open rather than merged.

## What This Changes

The issue reports that generated systemd service units can inline Google/Gemini API keys instead of routing them through the managed service environment path.

## Best Possible Solution

Land one canonical fix, preferably the staged-systemd EnvironmentFile approach in https://github.com/openclaw/openclaw/pull/96149 if review confirms it, then close this issue as implemented with release or main-sha proof.

## Reproduction Assessment

Yes. Source inspection shows config/auth env SecretRef values can enter the install-plan environment with inline valueSource, and systemd unit rendering writes inline env entries as `Environment=` lines.

## Solution Assessment

Yes. The narrow maintainable solution is to mark Google/Gemini env-ref service values as file-backed before systemd rendering, so the generated EnvironmentFile carries the secret and the unit file omits it.
