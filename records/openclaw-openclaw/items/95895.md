## Summary

Keep open: current `main` still has a credible service-env planning gap where env-backed provider/auth-profile secrets can be carried as inline gateway service environment unless the same key is already durable in the state `.env`. The exact reported non-SecretRef setup needs a focused regression fixture, but this is not already fixed or superseded.

## What This Changes

The issue reports that generated Linux systemd gateway service units can inline `GEMINI_API_KEY` instead of treating it as a managed service secret.

## Best Possible Solution

Implement a generic service-env policy that renders provider/config/auth-profile env-backed secrets through the systemd EnvironmentFile-managed path, with `GEMINI_API_KEY` as the regression case and sibling provider coverage to prevent one-off drift.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source and tests do show a reproducible nearby path where env-backed auth-profile/config secrets are planned inline unless the key is already durable in the state `.env`, which is enough to keep this open.

## Solution Assessment

No, a static Gemini-only managed-key addition is probably not the best fix. The maintainable fix is to make the service-env planner classify env-backed provider secrets generically and verify the exact Gemini systemd output.
