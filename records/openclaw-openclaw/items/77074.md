## Summary

Keep open: the report is valid and the patch targets the right runtime/plugin boundary, but it introduces a no-provider-selected regression against current main’s runtime-provider discovery guard and needs the required changelog entry.

## What This Changes

The PR updates the agent `web_search` wrapper to fall back to captured runtime metadata/configured provider IDs and lets web-provider runtime resolution fall through from empty active registries to scoped plugin loads, with focused tests.

## Best Possible Solution

Land a narrow fix that preserves runtime-provider discovery for unpinned auto-detect, uses configured provider IDs for pinned late-bound calls, falls through from empty compatible registries, and records the user-facing fix in the changelog.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current main can return no providers from an otherwise compatible active registry and the related issue gives the Brave capability-vs-agent failure path; I did not run a live Gateway reproduction in this read-only pass.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is: the late-bound metadata/config fallback and empty-registry fall-through are appropriate, but the no-selection `preferRuntimeProviders` change should preserve current main’s runtime auto-detect behavior and the PR needs a changelog entry.
