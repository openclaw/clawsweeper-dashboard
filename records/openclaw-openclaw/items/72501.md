## Summary

Keep open: this is a plausible fix for the current Telegram poll-answer routing gap, but the latest branch still has blocking correctness/security findings and no after-fix real Telegram proof.

## What This Changes

The branch adds a persistent Telegram poll registry, records non-anonymous poll sends, registers `poll_answer`, and routes poll-answer updates into Telegram session system events with tests.

## Best Possible Solution

Keep the Telegram-owned registry design, mark poll-vote events untrusted, align the handler with current authorization helpers, fail-soft on registry persistence, and merge only with redacted live Telegram vote proof.

## Reproduction Assessment

Yes by source inspection: current main can send polls and return a poll id, and `poll_answer` is in the default update set, but there is no poll-answer handler or persistent registry. I did not run a live Telegram reproduction because this review is read-only.

## Solution Assessment

No, not yet. The registry approach is maintainable, but the implementation needs the untrusted-event boundary, current authorization helper contract, fail-soft persistence, and real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll votes as untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram voter identity and selected options, then enqueues it without `trusted: false`. `enqueueSystemEvent` defaults omitted trust metadata to trusted, so a voter-controlled name can enter trusted system context; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.92
- **[P2] Await the current authorization helper:** `extensions/telegram/src/bot-handlers.runtime.ts:1048-1056`
  - body: Current main's `authorizeTelegramEventSender` is async and returns a boolean, but this handler keeps the old object-shaped check. After rebasing, this will fail type checking or skip poll answers at runtime; pass `senderId` into the context, await the helper, and check the boolean like the reaction handler.
  - confidence: 0.86
- **[P2] Keep completed poll sends successful:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry write failure before returning the successful send result.
  - confidence: 0.88
