## Summary

Keep open: the targeted Telegram poll-answer fix is useful, but this PR is not mergeable yet because it still queues external vote text as trusted system context, can fail an already-created poll send on local registry persistence errors, and lacks live Telegram proof for the transport path.

## What This Changes

The PR adds a Telegram poll registry, records public poll metadata on send, routes `poll_answer` updates into agent sessions, updates allowed-update handling, and adds focused tests.

## Best Possible Solution

Keep the registry-based route, mark poll-vote system events untrusted, fail-soft and log registry persistence after successful sends, update assertions, and merge only after redacted live Telegram poll-vote proof.

## Reproduction Assessment

Yes, by source inspection: current main can send Telegram polls and return a `pollId`, and the dependency contract supports `poll_answer`, but current main has no poll-answer handler or registry route. I did not run live Telegram in this read-only review.

## Solution Assessment

No, not yet. The registry/session-routing design is the right shape, but the patch still needs the untrusted event boundary, fail-soft persistence, and live Telegram proof before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Mark poll votes untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1586`
  - body: The handler builds `text` from Telegram voter identity and poll option labels, then enqueues it without `trusted: false`. `enqueueSystemEvent` treats omitted trust metadata as trusted, so voter-controlled names or option text can enter trusted system context; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.93
- **[P2] Keep poll sends successful after registry failures:** `extensions/telegram/src/action-runtime.ts:482-490`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry write failure before returning the successful send result.
  - confidence: 0.9
