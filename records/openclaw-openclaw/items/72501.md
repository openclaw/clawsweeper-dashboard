## Summary

Keep this PR open, but it is not merge-ready: the implementation records poll registry entries only for Telegram action-runtime polls while the shared outbound path used by `openclaw message poll` remains unchanged, so public poll votes from that path can still be silently skipped. The available real proof is also not sufficient for latest head because the Mantis comparison ran on an older commit and marked the candidate as not fixed.

## What This Changes

The PR adds a persistent Telegram poll registry, records public polls from the action runtime, routes `poll_answer` updates into sessions, and adds focused Telegram tests.

## Best Possible Solution

Move poll registry recording to a shared Telegram poll-send path that covers both action runtime and outbound adapter sends, then rerun focused tests and fresh Telegram Desktop proof on latest head.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main can send Telegram polls and its default update set includes `poll_answer`, but there is no registry-backed poll-answer session routing. I did not run a fresh live Telegram repro against today’s main.

## Solution Assessment

No, not yet. The PR is in the right Telegram plugin boundary, but the maintainable fix should cover the shared outbound poll path as well as action-runtime sends before it claims public poll votes route reliably.
