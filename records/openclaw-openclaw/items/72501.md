## Summary

Keep open: current main still lacks Telegram `poll_answer` session routing, and this PR is a plausible targeted fix, but the latest patch still has blocking trust and non-idempotent-send handling issues plus only mock/test proof.

## What This Changes

The PR adds a Telegram poll registry, records public poll metadata on send, and routes `poll_answer` updates into the originating session.

## Best Possible Solution

Keep the Telegram-owned registry design, mark poll-vote system events untrusted, fail-soft on registry persistence after send, update focused tests, and merge only after redacted live Telegram vote proof.

## Reproduction Assessment

Yes, by source inspection: current main can send polls and exposes `pollId`, and `poll_answer` is delivered in default updates, but no current-main `poll_answer` handler or persistent poll registry exists. I did not run live Telegram because this was a read-only review.

## Solution Assessment

No, not yet. The registry approach is the right shape, but the patch needs the untrusted event boundary, fail-soft persistence, and live Telegram proof before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Mark poll votes untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1586`
  - body: The handler builds `text` from Telegram voter identity and poll options, then enqueues it without `trusted: false`. `enqueueSystemEvent` treats omitted trust metadata as trusted, so voter-controlled names/options can enter trusted system context; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.93
- **[P2] Keep poll sends successful after registry failures:** `extensions/telegram/src/action-runtime.ts:482-490`
  - body: This write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry write failure before returning the successful send result.
  - confidence: 0.9
