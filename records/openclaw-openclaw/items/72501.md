## Summary

Keep open: the PR is a focused fix for a real Telegram poll-answer gap, but the current head still has blocking trust, topic-routing, and non-idempotent-send issues, and the supplied proof is test-only rather than live Telegram proof.

## What This Changes

The PR adds a Telegram poll registry, records non-anonymous poll sends, registers a `poll_answer` handler that enqueues session events, updates allowed updates, and adds Telegram tests.

## Best Possible Solution

Keep the registry/handler direction, but require untrusted system-event metadata, topic-aware registry correlation, fail-soft registry persistence, and redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main can send public Telegram polls and exposes a `pollId`, while the Telegram runtime has no poll-answer handler or registry to route votes into a session. I did not run live Telegram in this read-only review.

## Solution Assessment

No, not yet. The persistent registry and handler are the right shape, but the patch still needs trust downgrade, topic-target preservation, fail-soft persistence, and live Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Mark poll votes untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1586`
  - body: The handler builds text from Telegram voter identity plus poll question and option labels, then calls `enqueueSystemEvent` without `forceSenderIsOwnerFalse: true` or `trusted: false`. Omitted trust metadata is treated as trusted, so external vote text can enter trusted system context; pass the untrusted flag and assert it in the routing test.
  - confidence: 0.94
- **[P2] Preserve topic targets in the poll registry:** `extensions/telegram/src/action-runtime.ts:502`
  - body: Only the explicit `messageThreadId` parameter is stored, but Telegram poll sends also accept topic IDs embedded in `to` such as `-100...:topic:42`. Those polls are sent to the topic, then later votes are registered without topic 42 and route to the wrong forum session; store the resolved target thread ID or have `sendPollTelegram` return it, and cover the target-form case.
  - confidence: 0.9
- **[P2] Keep successful poll sends successful:** `extensions/telegram/src/action-runtime.ts:497-505`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry failure before returning the successful send result.
  - confidence: 0.91
