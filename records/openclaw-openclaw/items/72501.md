## Summary

## What This Changes

The PR adds a Telegram poll registry, records sent public polls, routes incoming `poll_answer` updates into Telegram sessions, and adds focused Telegram tests.

## Best Possible Solution

Keep the persistent Telegram-owned registry/handler direction, but require untrusted system-event metadata, effective topic correlation, fail-soft registry persistence, focused tests for those cases, and redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main can send public Telegram polls and expose `pollId`, while the Telegram runtime has no `poll_answer` handler or registry. I did not run live Telegram in this read-only review.

## Solution Assessment

No, not yet. The registry and handler are the right shape, but the current patch needs trust metadata, topic-aware registry writes, fail-soft post-send persistence, and live Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Mark poll votes untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1586`
  - body: The handler builds the system-event text from Telegram user names plus poll question/option strings, then enqueues it without `forceSenderIsOwnerFalse: true` or `trusted: false`. Omitted trust metadata is not downgraded by `enqueueSystemEvent`, so external vote text can enter the next turn with trusted owner semantics; pass the untrusted flag and assert it in the test.
  - confidence: 0.94
- **[P2] Store the effective poll topic:** `extensions/telegram/src/action-runtime.ts:502`
  - body: Only the explicit `messageThreadId` parameter is stored in the registry, but poll sends also accept topic ids embedded in `to` such as `-100...:topic:42`. Those polls are sent to the topic, then later votes are registered without that topic and route to the wrong forum session; store the effective parsed thread id or return it from the send helper.
  - confidence: 0.9
- **[P2] Fail soft after a poll is created:** `extensions/telegram/src/action-runtime.ts:498-505`
  - body: The registry write runs after Telegram has already created the poll. If local persistence throws, `handleTelegramAction` reports the action as failed and a caller may retry the non-idempotent send, creating duplicate polls; catch and log the registry failure before returning the successful send result.
  - confidence: 0.91
