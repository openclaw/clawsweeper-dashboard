## Summary

Keep open: this is a useful Telegram capability, but the current PR head is not mergeable because it trusts external vote text, can misroute topic-target poll votes, can report failure after successfully creating a poll, and still lacks live Telegram proof.

## What This Changes

The PR adds a Telegram poll registry, records sent public polls, routes `poll_answer` updates into agent sessions, and adds focused Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry and handler direction, but fix trust metadata, fail-soft persistence, topic-target correlation, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main can send public Telegram polls and exposes `pollId`, and grammY delivers `poll_answer`, but the Telegram runtime has no poll-answer handler or registry. I did not run live Telegram in this read-only review.

## Solution Assessment

No, not yet. The persistent registry is the right shape, but the patch needs an untrusted system-event boundary, fail-soft post-send persistence, topic-aware registry entries, and live Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Mark poll votes untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1586`
  - body: The handler builds text from Telegram voter identity plus poll question/option labels, then calls `enqueueSystemEvent` without `forceSenderIsOwnerFalse: true` or `trusted: false`. Omitted trust metadata is treated as trusted, so external vote text can enter trusted system context; pass the untrusted flag and assert it in the routing test.
  - confidence: 0.94
- **[P2] Preserve topic targets in the poll registry:** `extensions/telegram/src/action-runtime.ts:487`
  - body: Only the explicit `messageThreadId` parameter is stored, but Telegram poll sends also accept topic IDs embedded in `to` such as `-100...:topic:42`. Those polls are sent to the topic, then later votes are registered without topic 42 and route to the wrong forum session; store the resolved target thread ID or have `sendPollTelegram` return it, and cover the target-form case.
  - confidence: 0.9
- **[P2] Keep successful poll sends successful:** `extensions/telegram/src/action-runtime.ts:482-490`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry failure before returning the successful send result.
  - confidence: 0.91
