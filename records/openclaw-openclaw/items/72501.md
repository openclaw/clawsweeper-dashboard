## Summary

Keep open. The PR targets a real Telegram gap on current main, but the latest patch still has merge-blocking prompt-trust and reliability defects before it should land.

## What This Changes

The PR adds a persistent Telegram poll registry, records public poll sends, routes `poll_answer` updates into sessions, forces `poll_answer` into allowed updates, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry approach, but land it only after vote events are explicitly untrusted, registry persistence is fail-soft, fallback routing records the effective poll location, and the user-facing changelog is updated.

## Reproduction Assessment

Yes. Source inspection on current main shows public poll sends can return a `pollId`, while no `poll_answer` handler or poll registry exists to route votes into a session.

## Solution Assessment

No, not yet. The extension-local registry is the right shape, but the current patch weakens the system-event trust boundary and can report or route successful poll sends incorrectly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram user names plus poll question/options and enqueues it without `trusted: false`. Since system events default to trusted `System:` framing, external vote-derived text can be promoted into trusted prompt context; pass `trusted: false` and assert that in the routing test.
  - confidence: 0.91
- **[P2] Do not fail completed poll sends on registry writes:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This write runs after Telegram has already accepted and created the poll. If the local registry write fails, the action throws even though the non-idempotent send succeeded, which can make callers report failure or retry and create duplicate polls; catch and log registry persistence failures before returning the successful send result.
  - confidence: 0.87
- **[P2] Record the effective poll thread after fallback:** `extensions/telegram/src/action-runtime.ts:480`
  - body: `sendPollTelegram` can retry without `message_thread_id` when Telegram reports the topic missing, but the registry stores the originally requested `messageThreadId`. In that fallback path later votes route to a stale topic session instead of the chat-level session where the poll was actually posted; return/store the effective thread location.
  - confidence: 0.78
- **[P3] Add the user-facing changelog entry:** `extensions/telegram/src/action-runtime.ts:475`
  - body: This is a user-facing Telegram fix, but the PR file list does not include `CHANGELOG.md`. Add a single-line entry under the active release fixes section with appropriate contributor credit before merge.
  - confidence: 0.82
