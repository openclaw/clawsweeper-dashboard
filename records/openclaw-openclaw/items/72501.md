## Summary

Keep open. The PR addresses a real Telegram gap on current main, and the latest head has cleared the earlier CODEOWNERS overwrite and poll-registry read-error concerns, but the current diff still queues vote-derived Telegram text as trusted system context and can report a successful non-idempotent poll send as failed if local registry persistence throws.

## What This Changes

The PR adds a Telegram poll registry, records non-anonymous poll sends, ensures `poll_answer` is in allowed updates, routes `poll_answer` updates into session system events, and adds Telegram tests.

## Best Possible Solution

A mergeable end state is a Telegram-owned persistent poll-id registry with authorization-preserving `poll_answer` routing that queues external vote summaries as untrusted system events, treats registry persistence as best-effort after a successful send, and carries the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `poll_answer` is allowed and `sendPollTelegram` returns a `pollId`, but main has no poll registry or `poll_answer` handler, so public poll votes have no route into a session.

## Solution Assessment

No, not yet. The owner-boundary design is appropriate, but the current implementation must preserve the prompt trust boundary and avoid throwing after a successful non-idempotent Telegram send.
