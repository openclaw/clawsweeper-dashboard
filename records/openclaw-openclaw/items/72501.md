## Summary

Keep open: current main still lacks Telegram poll-answer routing, but this PR is not ready because the latest head still needs real Telegram proof and has correctness/security blockers in the new handler path.

## What This Changes

The PR adds a persistent Telegram public-poll registry, records sent public polls, and routes Telegram `poll_answer` updates into session system events.

## Best Possible Solution

Keep the Telegram-owned registry approach, but rebase it onto current main, mark poll-vote events untrusted, use the current authorization helper correctly, fail-soft on registry persistence, and merge only with redacted live Telegram vote proof.

## Reproduction Assessment

Yes, by source inspection: current main can send polls and expose a poll id, and grammY defaults include `poll_answer`, but there is no poll-answer handler or persistent registry. I did not run a live Telegram reproduction because this was a read-only review.

## Solution Assessment

No, not yet. The registry design is plausible, but the patch needs the untrusted event boundary, current authorization-helper contract, fail-soft persistence, and real Telegram proof before it is the right merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll votes as untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram voter identity and selected options, then enqueues it without `trusted: false`. `enqueueSystemEvent` defaults omitted trust metadata to trusted, so a voter-controlled name can enter trusted system context; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.92
- **[P2] Await the current authorization helper:** `extensions/telegram/src/bot-handlers.runtime.ts:1048-1056`
  - body: Current main's `authorizeTelegramEventSender` is async and returns a boolean, but this handler keeps the old object-shaped check. After rebasing, this will fail type checking or skip poll answers at runtime; pass `senderId` into the context, await the helper, and check the boolean like the reaction handler.
  - confidence: 0.86
- **[P2] Keep completed poll sends successful:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry write failure before returning the successful send result.
  - confidence: 0.88
