## Summary

Keep this PR open: it targets a real Telegram poll-answer gap, but the current patch is not mergeable as-is because it still trusts external poll-vote text, can report a completed Telegram poll send as failed after local registry persistence errors, needs rebasing onto current main's async authorization helper, and lacks after-fix real Telegram proof.

## What This Changes

The branch adds a persistent per-account Telegram poll registry, records non-anonymous poll sends, registers `poll_answer`, and routes poll-answer updates into Telegram session system events with tests.

## Best Possible Solution

Keep the Telegram-owned registry design, adapt it to current authorization, mark poll-vote events untrusted, fail-soft with logging on registry persistence, and require redacted live Telegram poll-vote proof before merge.

## Reproduction Assessment

Yes for the current gap by source inspection: current main can send Telegram polls and return a poll id, but it has no `poll_answer` handler or persistent registry for vote routing. I did not run live Telegram reproduction because this review is read-only.

## Solution Assessment

No, not yet. The registry direction is maintainable, but the implementation must preserve the untrusted-content boundary, fail softly after completed sends, and be rebased onto current main's auth helper before live proof can validate it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll votes as untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram voter identity plus poll question/options and enqueues it without `trusted: false`. System events default to trusted framing, so external vote-derived text can enter trusted prompt context; pass `trusted: false` and assert that in the routing test.
  - confidence: 0.92
- **[P2] Fail softly after Telegram creates the poll:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This registry write runs after Telegram has already accepted and created the poll. If the local write throws, callers will see the action as failed and may retry, creating duplicate polls; catch and log persistence failures before returning the successful send result.
  - confidence: 0.88
- **[P2] Rebase poll authorization onto current main:** `extensions/telegram/src/bot-handlers.runtime.ts:1048-1056`
  - body: The new handler uses the older `{ allowed }` authorization result shape, but current main's helper is async and returns a boolean. Rebase this block to await the helper and check the boolean, or poll answers will fail type checking or be skipped after integration.
  - confidence: 0.84
