## Summary

Keep open: current main still lacks poll-answer routing, and the PR is a plausible owner-local fix, but the latest branch needs source changes before merge for prompt-trust handling, fail-soft persistence, and current-main authorization compatibility.

## What This Changes

The PR adds a Telegram poll registry, records non-anonymous poll sends, registers `poll_answer`, and routes poll-answer updates into Telegram session system events with tests.

## Best Possible Solution

Keep the Telegram-owned registry design, adapt it to current main's authorization contract, mark poll-vote system events untrusted, fail-soft and log registry persistence failures, then require redacted live Telegram poll-vote proof before merge.

## Reproduction Assessment

Yes for the current gap by source inspection: current main can send polls and return a poll id, but it has no `poll_answer` handler or persistent poll registry. I did not run a live Telegram reproduction because this review is read-only.

## Solution Assessment

No, not yet. The owner-local registry approach is maintainable, but the PR must preserve the external-content trust boundary, avoid failing completed sends on local persistence errors, and rebase onto the current authorization helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll votes as untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram voter identity plus poll question/options and enqueues it without `trusted: false`. System events default to trusted framing, so external vote-derived text can enter trusted prompt context; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.92
- **[P2] Do not fail completed poll sends on registry writes:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This write runs after Telegram has already accepted and created the poll. If the local registry write throws, callers will see the action as failed and may retry, creating duplicate polls; catch and log persistence failures before returning the successful send result.
  - confidence: 0.88
- **[P2] Rebase poll authorization onto the current helper:** `extensions/telegram/src/bot-handlers.runtime.ts:1048-1057`
  - body: The new handler uses the older `{ allowed }` authorization result shape, but current main's helper is async and returns a boolean with existing callers awaiting it. Rebase this block and check the awaited boolean, otherwise poll answers will either fail type checking or be skipped after integration with current main.
  - confidence: 0.84
