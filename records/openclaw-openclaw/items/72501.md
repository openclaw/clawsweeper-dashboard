## Summary

Keep this PR open. The missing Telegram public-poll vote route is real on current main and the latest head removed the earlier CODEOWNERS overwrite concern, but the patch still has a prompt-trust security regression, can report a successful non-idempotent poll send as failed if local registry persistence breaks, and lacks the required user-facing changelog entry.

## What This Changes

This PR adds a Telegram poll registry, records non-anonymous poll sends, force-includes `poll_answer` in Telegram allowed updates, routes `poll_answer` updates into session system events, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry-backed design, but make poll vote notifications explicitly untrusted, treat registry persistence as best-effort after a successful send with logging, add the active `CHANGELOG.md` fix entry, and re-run the focused Telegram tests plus changed-gate CI.

## Reproduction Assessment

Yes. Source inspection on current main is enough: poll sends can return a `pollId`, but there is no persisted poll registry and no `poll_answer` handler, so public poll votes have no route into a session.

## Solution Assessment

No, not yet. The owner-boundary approach is appropriate, but the current patch must preserve OpenClaw's external-content trust boundary and avoid turning local persistence failure into a failed non-idempotent Telegram send.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram user names plus poll question/options, then enqueues it without `trusted: false`. `enqueueSystemEvent` defaults to trusted `System:` framing, so external vote-derived text can enter later prompts as trusted context; pass `trusted: false` and assert that in the routing test.
  - confidence: 0.91
- **[P2] Do not fail successful poll sends on registry writes:** `extensions/telegram/src/action-runtime.ts:476-483`
  - body: This await runs after Telegram has already accepted and created the poll. If the local registry write fails, the action throws even though the non-idempotent send succeeded, which can make callers report failure or retry and create duplicate polls; catch and log registry persistence failures before returning the successful send result.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `extensions/telegram/src/action-runtime.ts:475`
  - body: This is a user-facing Telegram fix, but the latest head does not change `CHANGELOG.md`. Add a single-line entry under the active `Unreleased` fixes section with appropriate contributor credit before merge.
  - confidence: 0.83
