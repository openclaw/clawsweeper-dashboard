## Summary

Keep open: this PR addresses a real Telegram public-poll routing gap and now has visible Telegram Desktop proof, but the reviewed head still has a security blocker plus topic-routing and non-idempotent send failure bugs.

## What This Changes

The PR adds Telegram poll_answer routing by persisting public-poll metadata, handling poll_answer updates as session events, and adding focused Telegram tests.

## Best Possible Solution

Land the persistent registry and poll_answer handler only after vote events are explicitly untrusted, topic correlation uses the actual send target, and registry persistence is fail-soft after successful sends.

## Reproduction Assessment

Yes. Current main can send public Telegram polls and return pollId but has no poll_answer handler or registry, and Mantis baseline proof at the PR base showed the vote was not routed into a visible follow-up.

## Solution Assessment

No, not yet. The registry-plus-handler direction is maintainable, but this head needs owner-downgraded system events, resolved-topic preservation, and fail-soft registry writes before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1584-1587`
  - body: The event text is built from Telegram voter identity plus poll question and option labels, then enqueued without `forceSenderIsOwnerFalse: true` or `trusted: false`. System events default missing trust metadata to trusted, so a public poll vote can promote external text into trusted prompt context.
  - confidence: 0.94
- **[P2] Keep sent polls successful if registry persistence fails:** `extensions/telegram/src/action-runtime.ts:497-505`
  - body: This write runs after Telegram has already created the poll. If local registry persistence throws, the action reports failure and callers may retry a non-idempotent send, creating duplicate polls even though the first poll exists.
  - confidence: 0.91
- **[P2] Preserve target topic IDs in the poll registry:** `extensions/telegram/src/action-runtime.ts:502`
  - body: Only the explicit `messageThreadId` option is stored, but `sendPollTelegram` also accepts topic IDs embedded in targets like `chatId:topic:42`. Those polls are sent to the topic, then later votes are registered without topic 42 and can route to the base group session.
  - confidence: 0.9
