## Summary

Keep open. The PR addresses a real Telegram gap on current main, but the latest patch still has blocking security and reliability issues before it can merge: poll vote text is queued as trusted system context, and registry persistence can turn a successful non-idempotent Telegram poll send into an action failure.

## What This Changes

The PR adds a Telegram poll registry, records non-anonymous poll sends, force-adds `poll_answer` to allowed updates, routes `poll_answer` updates into session system events, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry-backed design, but mark poll vote notifications explicitly untrusted, make registry persistence best-effort with logging after successful sends, add the required active-release changelog entry, then rerun the focused Telegram tests and changed gate.

## Reproduction Assessment

Yes. Source inspection on current main is enough: Telegram poll sends return a `pollId`, and `poll_answer` is listed in allowed updates, but no registry or `poll_answer` handler exists to route vote updates into sessions.

## Solution Assessment

No, not yet. The PR chooses the right owner boundary and mostly the right shape, but it must preserve the external-content trust boundary and avoid failing a successful non-idempotent send because a local registry side effect failed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram user names plus poll question/options and enqueues it without `trusted: false`. `enqueueSystemEvent` defaults to trusted `System:` framing, so vote-derived external text can be promoted into trusted prompt context; enqueue these events as untrusted and assert that in the routing test.
  - confidence: 0.91
- **[P2] Keep poll sends successful when registry writes fail:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This awaited registry write runs after Telegram has already accepted and created the poll. If local persistence fails, the action throws even though the non-idempotent send succeeded, which can make callers report failure or retry and create duplicate polls; catch and log registry persistence failures before returning the successful result.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `extensions/telegram/src/action-runtime.ts:475`
  - body: This is a user-facing Telegram fix, but the latest PR file list does not include `CHANGELOG.md`. Add a single-line entry under the active unreleased fixes section with appropriate contributor credit before merge.
  - confidence: 0.82
