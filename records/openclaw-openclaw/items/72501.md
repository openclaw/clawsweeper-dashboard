## Summary

Keep open. The PR fixes a real Telegram gap on current main, and the earlier CODEOWNERS overwrite is no longer present in the latest diff, but the current head still has merge-blocking issues: poll vote text is queued as trusted system context, registry persistence can turn an already-sent non-idempotent poll into a reported failure, and the user-facing fix lacks the required changelog entry.

## What This Changes

The PR adds a Telegram poll registry, records non-anonymous poll sends, force-adds `poll_answer` to allowed updates, routes `poll_answer` updates into Telegram session system events, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry-backed approach, but make poll vote notifications explicitly untrusted, treat registry persistence as best-effort after a successful send with logging, add the active-release changelog entry, and then re-run the focused Telegram tests plus the changed gate.

## Reproduction Assessment

Yes. Source inspection on current main is enough: poll sends can return a `pollId`, but there is no persisted poll registry and no `poll_answer` bot handler, so public poll votes have no route into a session.

## Solution Assessment

No, not yet. The design is in the right owner boundary and covers the main behavior, but the implementation must preserve the external-content trust boundary and avoid failing a successful non-idempotent poll send because local registry persistence failed.

## Review Findings

  - body: The handler builds `text` from Telegram user names and poll question/options, then enqueues it without `trusted: false`. `enqueueSystemEvent` defaults to trusted `System:` framing, so vote-derived external text can be promoted into trusted prompt context; enqueue these poll vote events as untrusted and assert that in the routing test.

  - body: This `await` runs after Telegram has already accepted and created the poll.
