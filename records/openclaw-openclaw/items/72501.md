## Summary

Keep open: current main still lacks Telegram `poll_answer` session routing, and this PR is a plausible targeted fix, but the latest patch still has a trusted-context security bug, a non-idempotent send failure path, and only synthetic proof.

## What This Changes

The PR adds a Telegram poll registry, records public poll metadata after sends, routes `poll_answer` updates into sessions, and adds focused Telegram tests.

## Best Possible Solution

Keep the Telegram-owned persistent registry approach, but make poll-vote system events explicitly untrusted, make registry persistence fail soft after successful sends, update assertions, and merge only after redacted live Telegram proof.

## Reproduction Assessment

Yes, by source inspection: current main can send public Telegram polls and returns `pollId`, and Telegram exposes `poll_answer`, but current main has no poll-answer handler or persistent registry. I did not run live Telegram in this read-only review.

## Solution Assessment

No, not yet. The registry design is the right shape, but the patch still needs an untrusted system-event boundary, fail-soft persistence after successful sends, and live Telegram proof before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Mark poll votes untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1586`
  - body: The handler builds `text` from Telegram voter identity and poll data, then enqueues it without `trusted: false`. `enqueueSystemEvent` treats omitted trust metadata as trusted, so voter-controlled names can enter trusted system context; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.93
- **[P2] Keep poll sends successful after registry failures:** `extensions/telegram/src/action-runtime.ts:482-490`
  - body: This write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry write failure before returning the successful send result.
  - confidence: 0.9
