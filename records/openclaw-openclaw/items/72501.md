## Summary

Keep open. Current main still lacks Telegram poll-answer routing, but this PR is not merge-ready because the new handler queues external Telegram vote text as trusted system context and the post-send registry write can turn an already-created poll into a reported action failure.

## What This Changes

The PR adds a Telegram poll registry, records public poll sends, forces `poll_answer` into allowed updates, routes poll-answer updates into Telegram session system events, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry-backed design, but make vote notifications explicitly untrusted, make registry persistence best-effort and logged after a successful send, add the required changelog entry, and then validate the Telegram registry/action/bot tests plus the changed gate.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing behavior: a public Telegram poll send can return a `pollId`, but there is no persisted poll registry and no `poll_answer` handler to route later vote updates into a session.

## Solution Assessment

No, not yet. The PR chooses the right owner boundary and a plausible registry design, but it must preserve the external-content trust boundary and avoid reporting failure after a successful non-idempotent Telegram send.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1098-1101`
  - body: The new handler builds `text` from Telegram user data plus poll question/options, then enqueues it without `trusted: false`. System events default to trusted `System:` prompt framing, so voter-controlled names or poll text can be promoted into trusted context; enqueue vote notifications as untrusted and cover that in the routing test.
  - confidence: 0.9

  - body: This write runs after `sendPollTelegram` has already created the poll. If the local registry write fails, the action throws even though Telegram already sent the poll, which can cause callers to report failure or retry a non-idempotent send; catch and log registry persistence failures as best-effort before returning success.
  - confidence: 0.86
