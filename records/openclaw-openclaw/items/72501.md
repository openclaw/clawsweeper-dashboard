## Summary

Keep open. Current main still lacks Telegram `poll_answer` session routing, so the PR addresses a real gap, but the latest head should not merge because it queues vote-derived text as trusted system context and can fail a successful Telegram poll send when local registry persistence fails.

## What This Changes

The PR adds a Telegram poll registry, records public poll sends, forces `poll_answer` into allowed updates, routes poll-answer updates into Telegram session system events, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry design, mark poll vote notifications explicitly untrusted, catch and log registry persistence failures after successful sends, add the active-release changelog entry, then rerun the focused Telegram tests plus the changed gate.

## Reproduction Assessment

Yes. Source inspection on current main is enough: public poll sends can return a `pollId`, but `poll_answer` has no handler or persisted poll registry, so votes have no current path into a session.

## Solution Assessment

No, not yet. The owner-local registry approach is the right shape, but the implementation must preserve the external-content trust boundary and keep completed poll sends successful when local persistence fails.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram user names plus poll question/options and then enqueues it without `trusted: false`. Because system events default to trusted `System:` framing, vote-derived external text can be promoted into trusted prompt context; pass `trusted: false` and assert that in the routing test.
  - confidence: 0.91
- **[P2] Do not fail completed poll sends on registry writes:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This `await` runs after Telegram has already accepted and created the poll. If the local registry write fails, the action throws even though the non-idempotent send succeeded, which can make callers report failure or retry and create duplicate polls; catch and log registry persistence failures before returning the successful send result.
  - confidence: 0.87
