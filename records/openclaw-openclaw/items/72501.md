## Summary

Keep open: the registry and handler direction is useful and the earlier security findings are addressed, but the current diff only records Telegram action-runtime poll sends. The documented outbound/CLI poll path still bypasses the registry, so part of the stated poll-vote routing behavior remains unfixed.

## What This Changes

The PR adds a Telegram poll registry, records non-anonymous poll sends from the Telegram action runtime, handles `poll_answer` updates as untrusted session events, and adds focused Telegram tests.

## Best Possible Solution

Use one shared registry-recording path for all Telegram public poll sends while keeping the handler authorization, topic routing, fail-soft persistence, and untrusted event metadata.

## Reproduction Assessment

Yes. Source inspection shows current main can send Telegram public polls and return a `pollId`, but it has no `poll_answer` handler or registry; the remaining PR defect is also source-reproducible because outbound `sendPoll` still calls `sendPollTelegram` without recording the registry entry.

## Solution Assessment

No, not yet. The persistent registry plus untrusted authorized handler is the right shape, but the registry write must be shared with the Telegram outbound/CLI poll path before this solves the stated behavior.
