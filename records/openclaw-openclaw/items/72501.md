## Summary

Keep open. Current main still lacks Telegram poll-answer routing, but the latest PR head does not yet satisfy the stated `openclaw message poll --poll-public` flow, and the Mantis Telegram Desktop comparison marks the candidate as not fixed.

## What This Changes

The PR adds a Telegram poll registry, records public poll metadata on Telegram action sends, and routes `poll_answer` updates into agent sessions.

## Best Possible Solution

Record public poll metadata in the shared Telegram poll send/outbound path, preserve the resolved topic id, enqueue vote events as untrusted external context, fail soft on registry persistence errors, and rerun live Telegram proof.

## Reproduction Assessment

Yes. Current main has poll sending but no `poll_answer` handler or registry, and the Mantis baseline run captured the public-poll vote without routed follow-up.

## Solution Assessment

No. The registry/handler shape is reasonable, but this PR misses the message-tool/CLI send path, trusts external vote text, loses topic correlation in some targets, and can fail after a successful non-idempotent poll send.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Record polls from the message send path:** `extensions/telegram/src/action-runtime.ts:497-506`
  - body: The stated `openclaw message poll --poll-public` flow goes through the gateway `send` method and Telegram outbound adapter, which call `sendPollTelegram` directly rather than `handleTelegramAction`. Because the registry write is only added here, those polls still have no registry entry and `poll_answer` is skipped; the Mantis candidate run reports `fixed: false` for this exact real Telegram path.
  - confidence: 0.95
- **[P1] Mark poll votes untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1584-1587`
  - body: The handler builds prompt text from Telegram voter identity plus poll question/option labels and queues it without `forceSenderIsOwnerFalse: true` or `trusted: false`. Omitted trust metadata remains trusted system-event context, so external poll-vote text can be promoted into trusted prompt context.
  - confidence: 0.94
- **[P2] Preserve topic targets in the poll registry:** `extensions/telegram/src/action-runtime.ts:502`
  - body: Only the explicit `messageThreadId` parameter is stored, but Telegram sends also accept topic ids embedded in `to` such as `-100...:topic:42`. Those polls are sent to the topic, then later registered without topic 42 and can route votes to the wrong forum session.
  - confidence: 0.9
- **[P2] Keep successful poll sends successful:** `extensions/telegram/src/action-runtime.ts:497-506`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry failure before returning the successful send result.
  - confidence: 0.91
