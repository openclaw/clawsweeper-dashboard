## Summary

Keep this PR open: the latest branch removes the earlier CODEOWNERS problem and updates the authorization path, but it still enqueues external Telegram poll-vote text as trusted system context, still lets post-send registry persistence failures turn a successful non-idempotent poll send into a failed action, and only supplies mocked test output as proof.

## What This Changes

The branch adds a persistent Telegram poll registry, records public poll sends, routes `poll_answer` updates into sessions, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry approach, but downgrade poll-vote events, fail-soft on registry writes after send, and merge only after redacted live Telegram poll-vote proof.

## Reproduction Assessment

Yes, by source inspection: current main can send Telegram polls and receive `poll_answer` update types, but it has no poll-answer handler or persistent registry. I did not run a live Telegram reproduction because this was a read-only review.

## Solution Assessment

No, not yet. The registry/session routing design is plausible, but the patch needs the untrusted event boundary, fail-soft persistence, and real Telegram proof before it is the right merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll votes as untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1586`
  - body: The handler formats `text` from Telegram voter identity and selected options, then enqueues it without `trusted: false`. `enqueueSystemEvent` defaults omitted trust metadata to trusted, so voter-controlled names or option text can enter trusted system context; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.93
- **[P2] Keep completed poll sends successful:** `extensions/telegram/src/action-runtime.ts:482-490`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry write failure before returning the successful send result.
  - confidence: 0.88
