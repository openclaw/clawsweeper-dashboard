## Summary

Keep open. Current main still lacks Telegram `poll_answer` session routing, so the PR targets a real gap, but the latest head still queues vote-derived text as trusted system context, can report failure after a poll has already been sent if registry persistence fails, and lacks the required changelog entry.

## What This Changes

The PR adds a persistent Telegram poll registry, records non-anonymous sent polls, forces `poll_answer` into allowed updates, routes vote updates into session system events, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry-backed design, but revise the PR so vote notifications are explicitly untrusted or otherwise structured as untrusted metadata, registry persistence is best-effort after a successful poll send, and the user-facing fix has a changelog entry before the focused Telegram tests and changed gate run.

## Reproduction Assessment

Yes. Source inspection on current main shows poll sends can return a `pollId` but there is no `poll_answer` handler or registry lookup; source inspection on the PR head shows the trusted enqueue and post-send awaited registry write paths.

## Solution Assessment

No, not yet. The PR chooses the right owner boundary and a plausible architecture, but it should preserve the existing external-content trust boundary and avoid turning a successful non-idempotent Telegram send into a reported failure because local state persistence failed.
