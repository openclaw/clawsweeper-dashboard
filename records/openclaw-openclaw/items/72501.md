## Summary

Keep this PR open: it targets a real Telegram poll-vote routing gap, but the current head still has blocking trust, topic-routing, non-idempotent send, and real-behavior-proof issues before it is mergeable.

## What This Changes

The PR adds a persistent Telegram public-poll registry, records non-anonymous poll sends, routes `poll_answer` updates into agent sessions, updates allowed updates, and adds Telegram tests.

## Best Possible Solution

Keep the registry-and-handler direction, but make vote events untrusted, preserve the resolved topic target, make registry persistence fail-soft, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main can send public Telegram polls and returns a `pollId`, and Telegram types expose `poll_answer`, but the Telegram runtime has no poll-answer handler or registry. I did not run a live Telegram vote in this read-only review.

## Solution Assessment

No, not yet. The persistent registry and handler are the right shape, but the current patch still needs trust downgrade, topic-target preservation, fail-soft persistence, and live Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Mark poll votes as untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1584-1587`
  - body: The handler builds system-event text from Telegram voter identity plus poll question and option labels, then enqueues it without `forceSenderIsOwnerFalse: true` or `trusted: false`. Omitted downgrade metadata keeps sender authority trusted, so external vote text can enter trusted system context; pass the untrusted flag and assert it in the poll-answer routing test.
  - confidence: 0.94
- **[P2] Keep successful poll sends successful:** `extensions/telegram/src/action-runtime.ts:497-505`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry failure before returning the successful send result.
  - confidence: 0.91
- **[P2] Preserve topic targets in the poll registry:** `extensions/telegram/src/action-runtime.ts:502`
  - body: Only the explicit `messageThreadId` parameter is stored, but `sendPollTelegram` also sends polls to topics embedded in `to` such as `-100...:topic:42`. Those votes later route without topic 42 and can wake the wrong session; store the resolved target thread id or return it from the send path.
  - confidence: 0.9
