## Summary

Keep open. The PR fixes a real Telegram poll-answer gap, and the latest file list no longer contains the earlier CODEOWNERS overwrite, but the current patch still promotes vote-derived text into trusted system context and can fail an already-created Telegram poll when local registry persistence fails.

## What This Changes

Adds a Telegram poll registry, records public poll sends, routes `poll_answer` updates into the originating session, ensures `poll_answer` stays in allowed updates, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-owned registry design, but mark poll vote notifications explicitly untrusted, make registry persistence non-fatal after successful sends, and add the active-release changelog entry before running the focused Telegram tests and changed gate.

## Reproduction Assessment

Yes. Source inspection on current main is enough: public poll sends can return a `pollId`, `poll_answer` is allowed, but no current handler or persisted poll route context exists, so votes cannot wake the originating session.

## Solution Assessment

No, not yet. The owner-local registry is the right shape, but the implementation needs to preserve the external-content trust boundary and avoid turning local persistence failures into failed non-idempotent sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The handler builds `text` from Telegram voter identity and poll question/options, then enqueues it without `trusted: false`. System events default to trusted `System:` framing, so vote-derived external text can enter later agent context as trusted; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.91
- **[P2] Do not fail completed poll sends on registry writes:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This write runs after `sendPollTelegram` has already created the Telegram poll. If the local registry write fails, the action throws despite the non-idempotent send succeeding, which can make callers retry and create duplicate polls; catch and log the persistence failure before returning success.
  - confidence: 0.88
