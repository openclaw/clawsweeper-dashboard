## Summary

Keep this PR open: it targets a real Telegram poll-answer gap, but the latest patch still has blocking correctness/security issues and lacks required real Telegram behavior proof.

## What This Changes

The PR adds a Telegram poll registry, records public poll sends, registers `poll_answer`, and routes poll-answer updates into Telegram session system events with tests.

## Best Possible Solution

Keep the Telegram-owned registry design, rebase it onto current async authorization, mark poll-vote events untrusted, make registry persistence fail-soft, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes by source inspection: current main can send public Telegram polls and retain a poll id, but there is no `poll_answer` handler or persistent registry to route votes back into a session. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not yet. The registry approach is a reasonable narrow fix, but the patch must preserve the current authorization helper contract, avoid trusted prompt context for external vote text, and fail-soft after Telegram has accepted a poll send.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the current authorization helper:** `extensions/telegram/src/bot-handlers.runtime.ts:1048-1057`
  - body: `authorizeTelegramEventSender` on current main is async and returns a boolean, but this new handler stores the unresolved promise and checks `.allowed`. That will fail type checking after rebase and would not apply the sender authorization result correctly; await the helper and test the boolean like the existing reaction handler does.
  - confidence: 0.9
- **[P1] Mark poll votes as untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1100-1103`
  - body: The text enqueued here includes Telegram voter identity plus poll question/option labels, all of which are external chat content. `enqueueSystemEvent` defaults to trusted context unless `trusted: false` is supplied, so a poll answer can enter the agent prompt as trusted system text; pass `trusted: false` and assert it in the routing test.
  - confidence: 0.92
- **[P2] Do not fail completed poll sends on registry writes:** `extensions/telegram/src/action-runtime.ts:475-483`
  - body: This registry write runs after Telegram has already accepted and created the poll. If the local state write throws, the action reports failure and callers may retry, creating duplicate polls; catch and log the persistence failure before returning the successful send result.
  - confidence: 0.88
