## Summary

Keep open: the PR targets a real Telegram poll-vote message-loss gap, but the current patch is not merge-ready because it trusts external vote text, can lose topic routing, can turn a successful non-idempotent poll send into a failure, and the Mantis Telegram Desktop candidate proof failed.

## What This Changes

The PR adds a persistent Telegram poll registry, records public poll sends, registers poll_answer updates, and routes received votes into Telegram sessions with new tests.

## Best Possible Solution

Land a topic-aware, fail-soft, untrusted poll-answer routing implementation with passing targeted Telegram tests, green checks, and passing live Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: current main can send public Telegram polls and returns pollId, while the Telegram runtime still has no poll_answer handler or registry. The Mantis baseline also captured the before state, though it ran on the PR base rather than today’s current main.

## Solution Assessment

No, not yet. The registry/handler direction is the right maintainable shape, but it must downgrade external vote text, preserve the resolved topic target, make registry persistence fail-soft, and pass live Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Mark poll-answer text as untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1584-1587`
  - body: The generated event text includes Telegram voter identity plus poll question/option text, then enqueues it without `forceSenderIsOwnerFalse: true` or `trusted: false`. System events without that metadata preserve owner trust into the next prompt, so external vote text can be promoted as trusted context.
  - confidence: 0.94
- **[P2] Store the resolved topic for sent polls:** `extensions/telegram/src/action-runtime.ts:502`
  - body: Poll sends can get their topic from the `to` target, such as `-100...:topic:42`, and `sendPollTelegram` sends to that parsed topic. The registry records only the explicit `messageThreadId` parameter, so votes for target-encoded topic polls can route to the wrong session.
  - confidence: 0.9
- **[P2] Keep successful poll sends successful:** `extensions/telegram/src/action-runtime.ts:497-505`
  - body: This registry write runs after Telegram has already created the poll. If local persistence throws, callers see a failed action and may retry a non-idempotent send, creating duplicate polls; catch and log the registry failure before returning the successful send result.
  - confidence: 0.91
