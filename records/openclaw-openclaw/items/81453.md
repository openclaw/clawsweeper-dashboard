## Summary

Keep open: current `main` still makes `agents.*.heartbeat.model` string-only in the config schema and runtime path, so the reported rejection of object fallback form is source-reproducible. The array form needs maintainer contract judgment because current agent model docs describe string or `{ primary, fallbacks }`, not array fallback chains.

## What This Changes

The issue asks for `agents.*.heartbeat.model` to accept fallback model shapes instead of rejecting array and object forms.

## Best Possible Solution

Make a maintainer decision on the heartbeat model contract, then implement heartbeat fallback parity by reusing the existing agent model object schema and runtime fallback override path, with docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `HeartbeatSchema.model` is `z.string().optional()` and the runtime only normalizes a string heartbeat override, matching the reported object/array validation failure.

## Solution Assessment

Unclear. Object-form parity with `AgentModelSchema` looks like the maintainable path, but the requested array form is not a documented shape for normal agent models and should be confirmed before implementation.
