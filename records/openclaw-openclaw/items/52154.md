## Summary

Keep open: current main does not implement this config surface, and the contributor supplied useful Windows proof, but PR head is conflicting and still has blocking issues around default shell startup behavior and a failed lint check.

## What This Changes

The PR adds a `commands.shellProfile` config option, generated config metadata, exec/chat/PTY plumbing, allowlist handling, and shell utility tests to source an explicit profile before host-shell command execution.

## Best Possible Solution

Land a rebased, maintainer-approved implementation that sources an explicit profile only for supported host-shell exec paths while preserving deterministic defaults, allowlist bypass protection, and documented sandbox/node exclusions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug because this is a feature PR; yes for the PR defects, which are source-reproducible from PR head and supported by the failed `check` job log.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The feature direction may be viable, but the patch must preserve current default shell behavior, pass checks, and get maintainer product/security review for the new exec surface.
