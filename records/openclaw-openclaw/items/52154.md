## Summary

Keep open: current main still lacks the requested opt-in shell profile feature, and the PR remains the canonical implementation thread, but the current head is not mergeable as-is because it regresses default shell startup isolation and has a strict-check failure risk.

## What This Changes

The PR adds an optional `commands.shellProfile` config key, validation/docs, and exec/supervisor wiring to source a configured shell profile before host shell command execution.

## Best Possible Solution

Land a maintainer-approved opt-in profile overlay that preserves deterministic no-profile defaults, scopes host-only behavior clearly, and refreshes schema, docs, and focused shell tests after rebase.

## Reproduction Assessment

No, not as a current-main bug; this is a feature PR. The blocking PR defects are source-reproducible from PR head by comparing `getShellConfig(shellProfile?)` with current main's default shell args and by observing the unused supervisor import.

## Solution Assessment

No. The direction is plausible, but this patch must layer profile loading onto current shell resolution/default args, remove stale code, rebase cleanly, and receive product/security approval.
