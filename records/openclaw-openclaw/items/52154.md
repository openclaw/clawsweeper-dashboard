## Summary

Keep open. The feature is not implemented on current main, and the PR has useful Windows proof, but the branch is conflicting and still has blocking correctness/security issues around default shell startup plus a failing check for an unused import.

## What This Changes

The PR adds an optional `commands.shellProfile` config key plus exec/chat/PTY plumbing, generated config metadata, and tests to source a user-specified profile before host shell command execution.

## Best Possible Solution

Land a rebased, maintainer-approved implementation that sources an explicit profile only for supported host-shell exec paths while preserving deterministic defaults, allowlist bypass protection, sandbox/node exclusions, and aligned docs/generated metadata.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug because this is a feature PR; yes for the PR defects, which are source-reproducible from PR head and supported by the failed `check` job log.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The feature direction may be viable, but the patch must preserve current default shell behavior, pass checks, and receive maintainer product/security review for the new exec surface.
