## Summary

The PR is still the canonical open implementation for this shell-profile feature, but it is not mergeable as-is because the branch changes default shell startup behavior for users who did not opt in and CI is failing on an unused import.

## What This Changes

The PR adds an optional `commands.shellProfile` config field, schema/types, docs/generated config metadata, exec/chat/PTY plumbing, allowlist handling, and shell utility tests to source a configured profile before supported host-shell command execution.

## Best Possible Solution

Land a rebased implementation that sources an explicit profile only on supported host-shell paths while preserving current no-profile defaults, allowlist bypass protection, and documented sandbox/node exclusions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug because this is a feature PR; yes for the PR defects, which are source-reproducible from PR head and supported by the failed `check` job log.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The feature direction may be viable, but the patch must preserve current default shell behavior, pass checks, and get maintainer product/security review for the new exec surface.
