## Summary

Keep open: current main still has no `commands.shellProfile` support, and this feature PR has useful proof, but PR head still regresses the existing default shell fallback behavior and fails the `check` CI job on an unused import.

## What This Changes

The PR adds a `commands.shellProfile` config option and threads explicit profile sourcing through host exec, chat `/bash`, PTY supervisor spawn, config schema/help, generated baselines, and shell utility tests.

## Best Possible Solution

Land a rebased implementation that sources an explicit profile only on supported host-shell paths while preserving deterministic no-profile defaults, allowlist enforcement, and documented sandbox/node exclusions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug because this is a feature PR; yes for the PR defects, which are source-reproducible from PR head and supported by the failed `check` job log.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The feature direction may be viable, but the patch must preserve current default shell behavior, pass checks, and get maintainer product/security review for the new exec surface.
