## Summary

Keep open: current main still lacks the requested `commands.shellProfile` capability, but the PR is not merge-ready because PR head changes default shell startup behavior when the option is unset and leaves an unused strict-module import.

## What This Changes

The PR adds an optional `commands.shellProfile` config key and threads profile sourcing through host exec, PTY supervisor, chat bash, config metadata, generated docs, and shell utility tests.

## Best Possible Solution

Land a maintainer-approved opt-in shell-profile overlay that composes with existing shell resolution/default args, preserves allowlist isolation, and refreshes schema/docs/tests after rebase.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug; this is a feature PR. The blocking PR defects are source-reproducible from PR head at `src/agents/shell-utils.ts:108` and `src/process/supervisor/supervisor.ts:3`.

## Solution Assessment

Is this the best way to solve the issue? No. The direction is plausible, but this patch should layer profile loading on top of current shell defaults/fallbacks and remove stale code before maintainer security/product review.
