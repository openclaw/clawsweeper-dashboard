## Summary

Keep this PR open, but it still needs changes before merge: current main does not have `commands.shellProfile`, the feature remains useful, and PR head still regresses default shell startup behavior while also failing lint on an unused import.

## What This Changes

The PR adds an optional `commands.shellProfile` config key and threads profile loading through exec tool setup, shell selection, supervisor PTY spawning, chat `/bash`, schema/help metadata, and focused shell utility tests.

## Best Possible Solution

Land a rebased implementation that sources an explicitly configured profile only on supported host shell paths while preserving main's default shell isolation, allowlist enforcement, and documented sandbox/node exclusions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug; this is a feature PR. The PR defects are source-reproducible from PR head, and the unused import is confirmed by the failed `check` job log.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The direction may be viable, but the patch must preserve current shell defaults, remove stale code, and receive maintainer product/security approval for the new exec surface.
