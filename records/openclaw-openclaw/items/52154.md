## Summary

Keep open. Current main still lacks the requested shell-profile config and this PR remains the canonical implementation thread, but PR head is not ready because it regresses default no-startup shell behavior and has a failing unused import; the new host command-execution config also needs maintainer product/security review and a rebase.

## What This Changes

Adds optional `commands.shellProfile` config, schema/docs metadata, exec/chat/PTY plumbing, and tests to source a user-provided shell profile for supported host exec shells.

## Best Possible Solution

Land a rebased, maintainer-approved implementation that sources explicit profile paths only in supported host exec paths while preserving deterministic default shell startup, allowlist bypass protections, and generated config/docs alignment.

## Reproduction Assessment

No, not as a current-main bug because this is a feature PR and current main intentionally lacks `commands.shellProfile`. Yes for PR defects: source inspection of PR head shows the default fallback loses no-startup args, and the failed `check` job reports the unused import.

## Solution Assessment

No, not as-is. The direction may be viable, but the patch must preserve default shell behavior, remove the failing unused import, rebase cleanly, and receive maintainer product/security review for the new exec surface.
