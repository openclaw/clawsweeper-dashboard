## Summary

Keep open: current main still lacks this config surface and the PR contains meaningful feature work, but the patch is not mergeable as-is because it changes default shell startup behavior for users who did not opt in, leaves an unused import, and needs maintainer product/security judgment.

## What This Changes

The PR adds optional `commands.shellProfile` config and threads profile sourcing through host exec, PTY supervisor, chat bash, config schema/help, and shell utility tests.

## Best Possible Solution

Land a maintainer-approved opt-in profile overlay that preserves current deterministic default shell args and fallback behavior, documents host exclusions clearly, and keeps allowlist-enforced exec from sourcing profile code.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a user bug; this is a feature PR. The blocking PR defects are high-confidence from source inspection of PR head against current main shell defaults and supervisor imports.

## Solution Assessment

Is this the best way to solve the issue? No. The direction is plausible, but the patch should layer profile sourcing on current `getShellConfig()` defaults, remove the unused import and stale generated artifacts, and get maintainer product/security approval before merge.
