## Summary

Keep this PR open: current main still does not implement `commands.shellProfile`, but the branch is conflicting and still has correctness/security blockers around preserving current shell startup defaults plus a failed check from an unused import.

## What This Changes

The PR adds an optional `commands.shellProfile` config key and threads profile loading through host exec, chat bash, PTY supervisor, config metadata, and shell utility tests.

## Best Possible Solution

Land a rebased, maintainer-approved implementation that sources an explicit profile only on supported host-shell exec paths while preserving deterministic defaults, allowlist bypass protection, and aligned config docs/metadata.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug, because this is a new config feature. The PR defects are source-reproducible by comparing PR head `getShellConfig(shellProfile?)` with current-main shell defaults and by the failed `check` job.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The feature direction may be viable, but the patch must preserve current default shell behavior, pass checks, rebase cleanly, and receive maintainer approval for the new exec surface.
