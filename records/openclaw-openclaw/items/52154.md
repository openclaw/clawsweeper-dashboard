## Summary

Keep open: current main still lacks the requested `commands.shellProfile` config, and this PR is the canonical implementation thread, but the current branch is conflicting and still has correctness/security blockers in the host exec startup path.

## What This Changes

The PR adds an optional `commands.shellProfile` config field and threads it through host exec, chat bash, PTY supervisor, config metadata, and shell utility tests.

## Best Possible Solution

Land a rebased, maintainer-approved implementation that sources an explicit profile only for supported host-shell exec paths while preserving deterministic defaults, allowlist bypass protection, and aligned config docs/metadata.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug because this is a feature PR; yes for the PR defects, which are source-reproducible by comparing the PR fallback path with current main and by the failed `check` job.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The feature direction may be viable, but the patch must preserve current default shell behavior, pass checks, and get maintainer approval for the new exec surface.
