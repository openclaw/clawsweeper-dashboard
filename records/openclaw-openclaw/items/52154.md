## Summary

Keep open. Current main still does not implement `commands.shellProfile`, and this PR is the canonical implementation thread, but the branch is currently conflicting and still has merge blockers in the shell fallback path plus a failing check from an unused import.

## What This Changes

The PR adds an optional `commands.shellProfile` config key, config metadata, exec/chat/PTY plumbing, and tests to source a user-provided profile before supported host shell command execution.

## Best Possible Solution

Land a rebased, maintainer-approved implementation that sources an explicit profile only for supported host-shell exec paths while preserving deterministic defaults, allowlist bypass protection, sandbox/node exclusions, and aligned docs/generated metadata.

## Reproduction Assessment

No, not as a current-main bug because this is a feature PR. The PR defects are source-reproducible from PR head and one is also backed by the failed `check` job log.

## Solution Assessment

No, not as-is. The feature direction may be viable, but the patch must preserve current default shell behavior, pass checks, rebase cleanly, and receive maintainer product/security review for the new exec surface.
