## Summary

Keep open: current main does not implement `commands.shellProfile`, and this PR still contains the unique implementation path, but the latest head should not merge because it regresses default shell startup suppression and fails the check job on an unused import.

## What This Changes

The PR adds an optional `commands.shellProfile` config key and threads it through host exec, chat bash, supervisor PTY, config schema/help, generated baselines, and shell utility tests.

## Best Possible Solution

Land a rebased implementation that sources an explicitly configured profile only on supported host shell paths while preserving main's no-profile/no-config defaults, allowlist enforcement, and documented sandbox/node exclusions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug; this is a feature PR. The PR defects are source-reproducible from PR head, and the unused import is confirmed by the failed `check` job log.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The direction may be viable, but the patch must preserve current shell defaults, remove stale code, and receive maintainer product/security approval for the new exec surface.
