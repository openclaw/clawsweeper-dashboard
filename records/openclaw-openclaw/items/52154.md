## Summary

Keep open: current main still does not implement `commands.shellProfile`, and the thin linked feature issue was already closed in favor of this PR, but the PR head is not merge-ready because it regresses default shell startup behavior, leaves an unused import, and changes host exec startup policy.

## What This Changes

The PR adds an optional `commands.shellProfile` config field and threads profile sourcing through exec runtime, PTY supervisor, chat bash, config schema/help, generated config docs, and shell utility tests.

## Best Possible Solution

Land a maintainer-approved opt-in profile overlay that preserves deterministic shell defaults and allowlist isolation, with refreshed schema, docs, and tests after rebase.

## Reproduction Assessment

No, not as a current-main bug; this is a feature PR. The blocking PR defects are source-reproducible from PR head by comparing `getShellConfig(shellProfile?)` with current main's default shell args and by observing the unused supervisor import.

## Solution Assessment

No. The direction is plausible, but this patch should layer profile loading onto current shell resolution/default args, remove stale code, rebase/regenerate cleanly, and receive product/security approval.

Overall correctness: patch is incorrect
