## Summary

Keep open: current main does not implement `commands.shellProfile`, and the PR still contains meaningful feature work, but the current head is not mergeable because it regresses default shell startup behavior and has a failing lint check.

## What This Changes

The PR adds an optional `commands.shellProfile` config key and threads explicit shell-profile sourcing through host exec, chat bash, PTY spawning, schema/help, generated baselines, and shell utility tests.

## Best Possible Solution

Land a rebased implementation that sources an explicit profile only on supported host-shell exec paths while preserving current no-profile defaults, allowlist bypass protection, and documented sandbox/node exclusions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug because this is a feature PR; yes for the PR defects, which are source-reproducible from PR head and supported by the failed `check` job log.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The feature direction may be viable, but the patch must preserve current default shell behavior, pass checks, and get maintainer product/security review for the new exec surface.
