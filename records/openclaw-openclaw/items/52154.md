## Summary

Keep open: current main still does not implement `commands.shellProfile`, and this PR remains the canonical implementation thread, but the branch still has source-level blockers in shell default preservation and a strict TypeScript unused import.

## What This Changes

The PR adds an optional `commands.shellProfile` config key and threads profile sourcing through host exec, PTY supervisor, chat bash, config schema/help, generated docs, and shell utility tests.

## Best Possible Solution

Land a maintainer-approved opt-in profile overlay that preserves deterministic default shell args and fallbacks, keeps allowlist-enforced exec from sourcing profile code, and updates schema/docs/tests without stale generated artifacts.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug; this is a feature PR. The blocking defects are source-reproducible from PR head: `shell-utils.ts` drops current defaults at line 108 and `supervisor.ts` imports an unused symbol at line 3.

## Solution Assessment

Is this the best way to solve the issue? No. The feature direction is plausible, but this patch should preserve current unset-profile behavior, remove the unused import, rebase/regenerate cleanly, and receive maintainer product/security approval before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve current shell defaults when profile is unset:** `src/agents/shell-utils.ts:108`
  - body: When `shellProfile` is unset or unsupported, this fallback returns bare `-c`, dropping current main's bash/zsh/fish startup-file suppression and placeholder-shell fallback. That changes default exec behavior for users who did not opt in, so profile sourcing should be layered on top of the existing shell resolution and args.
  - confidence: 0.91

  - confidence: 0.99
