## Summary

Keep this PR open because current main does not implement `commands.shellProfile`, but the proposed patch is not merge-ready: it regresses current default shell startup behavior and still has a CI-blocking unused import. The feature also changes host command startup policy, so it needs maintainer product/security approval after the branch is repaired and rebased.

## What This Changes

The PR adds an optional `commands.shellProfile` config key and threads a profile-sourcing prefix through host exec, PTY supervisor, chat bash, config schema, help metadata, and tests.

## Best Possible Solution

Land a rebased or replacement implementation that sources an explicit profile only on supported host-shell paths while preserving deterministic no-profile defaults, allowlist enforcement, sandbox/node exclusions, and config documentation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main bug, because this is a feature PR. The blocking PR defects are source-reproducible from PR head, and the unused import is also confirmed by the failed `check` job log.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The feature direction may be viable, but the patch must preserve current main's default shell behavior, remove the stale import, rebase cleanly, and receive maintainer approval for the new exec startup surface.
