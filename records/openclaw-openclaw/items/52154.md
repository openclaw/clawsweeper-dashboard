## Summary

Keep open: this is a plausible feature PR, but the current patch is not correct against current main because it changes default shell startup behavior when the new option is unset, leaves an unused supervisor import, and introduces a host command startup config that needs maintainer product/security approval.

## What This Changes

The PR adds an optional `commands.shellProfile` config field and threads profile sourcing through host exec, PTY, chat bash, config metadata, generated docs, and shell utility tests.

## Best Possible Solution

Land a maintainer-approved opt-in profile overlay that preserves current deterministic default shell args and fallback behavior, with clear documented exclusions for sandbox, node host, and enforced allowlist paths.

## Reproduction Assessment

Not applicable as a user bug reproduction; this is a feature PR. The blocking PR defects are high-confidence from source inspection of PR head against current main's shell defaults and supervisor imports.

## Solution Assessment

No. The direction is plausible, but the patch should layer profile sourcing on top of current `getShellConfig()` defaults, remove the unused import and stale generated artifacts, and get maintainer product/security approval before merge.
