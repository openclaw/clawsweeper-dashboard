## Summary

Keep this PR open: it targets a real plugin gateway RPC bug, but the current patch replaces the attached dispatch registry instead of preserving it, so it can regress current-main plugin method dispatch and authorization.

## What This Changes

The PR changes `handleGatewayRequest` to rebuild the gateway method registry from live active plugin state for every request and adds a stale-startup-registry regression test.

## Best Possible Solution

Land one refreshed gateway dispatch fix that uses the supplied registry when it owns the method, falls back to the live registry for stale plugin registrations, tests both cases, and includes redacted live CLI proof.

## Reproduction Assessment

Yes. Current-main source and tests show both the stale startup-registry miss the PR targets and the attached-registry case the PR would regress.

## Solution Assessment

No, not as written. Gateway dispatch is the right layer, but the best fix must preserve the supplied registry path and use live lookup as a fallback instead of replacing the registry source.
