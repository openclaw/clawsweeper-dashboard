## Summary

Keep this PR open for maintainer review: the code shape looks like the right gateway-layer fix and I found no blocking patch bug, but registry selection is compatibility-sensitive and the supplied proof still stops at an in-process dispatch harness rather than the reported install/restart/inspect/`openclaw gateway call` path.

## What This Changes

The PR changes `handleGatewayRequest` to prefer an attached gateway method registry only when it owns the requested method, otherwise falling back to a live registry, and adds regression tests for stale live plugin dispatch, attached-registry dispatch, and fail-closed behavior.

## Best Possible Solution

Land this branch or an equivalent attached-registry-first live-fallback fix after maintainers either accept the chokepoint proof or receive redacted end-user gateway-call proof, then close https://github.com/openclaw/openclaw/issues/94127.
