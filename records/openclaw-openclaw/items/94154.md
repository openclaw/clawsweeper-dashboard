## What This Changes

The PR changes gateway request dispatch to prefer an attached method registry only when it owns the requested method, otherwise falling back to the live plugin registry, and adds a regression test plus a PR-specific proof script.

## Best Possible Solution

Land one clean gateway dispatch fix that preserves attached registries, falls back to the live registry for stale misses, keeps focused regression coverage, removes or generalizes the proof script, and includes redacted user-path terminal/log proof.

## Reproduction Assessment

Yes. Source inspection shows current main passes a startup attached registry into `handleGatewayRequest`, while `registerGatewayMethod` writes late plugin methods into the live active registry; I did not run the full CLI repro in this read-only review.

## Solution Assessment

Yes for the core code shape: dispatch is the right layer, and using the attached registry when it owns the method with live fallback for stale misses is the narrow maintainable fix. The branch still needs proof cleanup and user-path validation before merge.
