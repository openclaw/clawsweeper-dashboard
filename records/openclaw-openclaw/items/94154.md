## Summary

Keep this PR open, but the current patch is not merge-ready: it fixes the stale live-registry miss by discarding the supplied request registry, which the current merge result would use for both authorization and dispatch on current main. The contributor also still supplied only Vitest output, not live plugin install/restart/inspect/`openclaw gateway call` proof.

## What This Changes

The PR changes gateway request dispatch to always rebuild the method registry from the live active plugin registry and adds a regression test for plugin RPC dispatch with a stale supplied registry.

## Best Possible Solution

Land one refreshed gateway dispatch fix that uses the supplied attached registry when it owns the method, falls back to the live active plugin registry for stale misses, covers both directions, and includes redacted live CLI proof.

## Reproduction Assessment

Yes. Source inspection shows the reported stale-startup registry miss and also shows the current-main attached-registry case that the PR's live-only merge result would violate; I did not run live gateway proof in this read-only review.

## Solution Assessment

No. Gateway dispatch is the right layer, but the best fix must compose the supplied attached registry with the live registry instead of replacing the attached-registry contract.
