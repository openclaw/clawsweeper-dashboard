Keep open: this is a concrete routing regression with multiple matching user reports, current main still has the stale WebChat-origin precedence hazard, and https://github.com/openclaw/openclaw/pull/77611 is an open linked fix that closes this issue. ## What This Changes

Telegram group/topic assistant replies should route back to Telegram delivery context instead of falling through to stale WebChat session-origin metadata.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/77611; the desired end state is explicit Telegram deliveryContext/last-route data winning over stale WebChat origin metadata without broadening WebChat delivery inheritance.

## Reproduction Assessment

Yes, at source level: deliveryContextFromSession on current main can prefer stale WebChat channel/origin metadata over an explicit Telegram deliveryContext, and the issue plus duplicate production report describe the same observable leak. I did not run a live Telegram E2E in this read-only review.

## Solution Assessment

Yes: the linked PR is the narrowest maintainable direction I see because it makes explicit persisted delivery routes authoritative before legacy origin fallback, while keeping WebChat no-inherit behavior covered separately.
