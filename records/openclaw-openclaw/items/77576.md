## Summary

Keep open: the report remains a valid current-main routing regression, and an open PR already references this issue with closing syntax, so the issue should stay open until that PR lands or is rejected.

## What This Changes

Telegram group/topic session replies should route back to Telegram even after WebChat/Control UI interaction leaves stale webchat origin metadata.

## Best Possible Solution

Review and land the linked focused route-precedence PR, or replace it with an equivalent fix that makes explicit Telegram deliveryContext/last* data authoritative while preserving WebChat non-inheritance behavior.

## Reproduction Assessment

Yes, source-level: a session with stale WebChat channel/origin metadata and explicit Telegram deliveryContext can flow through the current deliveryContextFromSession precedence and prefer the stale WebChat fallback. I did not run a live Telegram group E2E in this read-only review.

## Solution Assessment

Yes: the linked PR is the narrowest maintainable fix path I found because it prefers explicit persisted Telegram route data before legacy origin metadata and adds coverage for both the bug and WebChat no-inheritance behavior.
