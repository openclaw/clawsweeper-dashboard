## Summary

Keep open: the report has an open linked implementation PR with closing syntax, and current main still shows the stale WebChat-origin delivery-context shape that the PR is meant to fix.

## What This Changes

The issue reports Telegram group/topic session replies being routed to WebChat instead of back to Telegram when stale WebChat session-origin metadata exists.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77611, or close that PR with a replacement only if maintainers choose a different narrow fix that makes explicit Telegram delivery metadata authoritative without weakening WebChat route-safety behavior.

## Reproduction Assessment

Yes, source-level: current main can be exercised through `deliveryContextFromSession` with stale WebChat `channel`/`origin.provider` and explicit Telegram `deliveryContext`, and the linked PR adds that regression case. I did not run a live Telegram group E2E in this read-only review.

## Solution Assessment

Yes: the linked PR is the narrowest visible path because it changes explicit route precedence, adds focused regression coverage, and preserves WebChat non-inheritance. The issue should stay open until that PR or an equivalent fix merges.
