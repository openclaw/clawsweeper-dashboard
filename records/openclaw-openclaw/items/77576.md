## Summary

Keep open: this issue is the canonical report for a real Telegram group/topic routing regression, current main still shows the stale-origin route-precedence hazard, and https://github.com/openclaw/openclaw/pull/77611 is an open implementation candidate that explicitly targets this issue.

## What This Changes

The issue asks for automatic assistant replies in Telegram group/topic sessions to route back to Telegram instead of leaking to WebChat or another stale session surface.

## Best Possible Solution

Land a narrow route-precedence fix that makes explicit Telegram deliveryContext/last-route metadata authoritative for Telegram-originated turns while preserving WebChat chat.send isolation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run, but source inspection and the linked PR’s regression tests give a clear source-level reproduction around stale WebChat origin metadata overriding explicit Telegram route data.

## Solution Assessment

Is this the best way to solve the issue? Yes: the open PR’s direction is the narrow maintainable path, because it changes route precedence without broadening WebChat clients’ ability to inherit external routes.
