## Summary

Keep open: current main still contains the stale WebChat-origin route precedence that matches the report, and an open fix PR already closes this issue. This should not be cleaned up before that PR lands or is replaced.

## What This Changes

Telegram group/topic replies should route back to Telegram rather than WebChat when a reused session has stale WebChat origin metadata.

## Best Possible Solution

Review and land or replace the linked fix PR so explicit persisted `deliveryContext` and `last*` routes win over stale `origin.provider` metadata while preserving WebChat route-safety behavior.

## Reproduction Assessment

Yes, source-level: current main can derive a WebChat route from stale `entry.channel` or `origin.provider` even when the stored delivery context is Telegram. I did not run a live Telegram group E2E in this read-only review.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix I found: make explicit persisted delivery route data authoritative and keep legacy origin metadata as fallback only. The issue should stay open until that PR merges or maintainers choose a replacement.
