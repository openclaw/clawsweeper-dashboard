## Summary

Keep open: current main still lacks the cron timeline feature, and this PR remains the active implementation candidate, but PR head has blocking correctness/i18n findings and lacks current-head real behavior proof.

## What This Changes

The PR adds a 24-hour Control UI cron timeline with markers, clustering, high-frequency chips, zoom controls, timezone status plumbing, styles, locale keys, and a changelog entry.

## Best Possible Solution

Keep this PR as the active timeline candidate, but require marker computation to mirror or share scheduler stable per-job stagger semantics, fix untranslated timeline strings, add focused coverage, and provide current-head real behavior proof before merge.

## Reproduction Assessment

Yes for the blocking patch issue: PR head source shows staggered markers use schedule.staggerMs / 2, while current main's scheduler hashes job.id for a stable offset modulo the stagger window.

## Solution Assessment

No. The timeline direction is useful, but approximating stagger with a midpoint and shipping English labels in non-English locales are not the narrowest maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the scheduler's per-job stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: `getTodayRunHours` adds `schedule.staggerMs / 2` to every cron marker, but runtime scheduling hashes the job id and uses that stable offset modulo the stagger window. Most staggered jobs will be plotted at the wrong time, which can flip past/pending state and send users to history for a run that has not happened yet.
  - confidence: 0.92

  - body: Several non-English locale files add `cron.timeline` labels in English while neighboring cron strings are localized, so users in those locales will see English timeline controls. Regenerate or translate these keys before shipping the new UI.
  - confidence: 0.86
