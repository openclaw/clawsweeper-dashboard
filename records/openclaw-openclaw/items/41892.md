## Summary

## What This Changes

This PR adds a Control UI Cron Jobs timeline strip with helper scheduling logic, CSS, i18n bundles, an additive cron status timezone field, and a changelog entry.

## Best Possible Solution

Repair this PR so timeline markers use the same scheduler/stagger semantics as runtime, regenerate generated locale output, and require current real Control UI proof before maintainer UX review.

## Reproduction Assessment

Yes. Source inspection of PR head a95e062067decc18c7b2762d0db3647add4c8d67 shows the timeline adds schedule.staggerMs / 2, while current main scheduler code and tests use a stable job-id-derived offset; locale source inspection also shows English strings/fallbacks in generated bundles.

## Solution Assessment

No. The feature direction matches https://github.com/openclaw/openclaw/issues/40644, but the implementation should derive marker times from runtime scheduling semantics and complete generated localization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the scheduler's stable stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: The timeline shifts cron markers by `schedule.staggerMs / 2`, but runtime scheduling hashes `job.id` and applies that stable offset inside the stagger window. For staggered jobs this can place dots minutes away from the real run and flip past/pending behavior; derive the same per-job offset or avoid shifting exact markers.
  - confidence: 0.9
- **[P3] Regenerate localized timeline strings:** `ui/src/i18n/locales/de.ts:1240-1251`
  - body: The PR advertises full i18n, but generated bundles such as German still contain English `cron.timeline.*` controls, and some locale metadata still lists those keys as fallbacks. Users in those locales will see mixed-language UI, so complete/regenerate the locale output before shipping.
  - confidence: 0.86
