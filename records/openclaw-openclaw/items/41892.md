## Summary

Keep open: this is the active implementation candidate for the cron calendar/timeline request and the screenshots are useful proof, but the patch still has a correctness issue where timeline markers do not match runtime cron stagger semantics.

## What This Changes

The PR adds a 24-hour Cron Jobs timeline strip to the Control UI with markers, clusters, zoom modes, always-on chips, i18n strings, and a gateway timezone field.

## Best Possible Solution

Keep this PR open, fix the timeline to use the same cron timing facts as the runtime scheduler, then let maintainers decide whether the timeline slice is enough to close the broader calendar request.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: compare a cron job with explicit or default stagger whose stable hashed offset is not half the window, and the PR helper will plot the marker at the midpoint instead of the runtime execution time.

## Solution Assessment

No. The feature direction is plausible, but the implementation should share or carry the scheduler's exact per-job stagger offset instead of approximating with `staggerMs / 2`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the runtime stagger offset for timeline markers:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: The new timeline adds `staggerMs / 2` to every cron marker, but runtime scheduling uses `resolveStableCronOffsetMs(job.id, resolveCronStaggerMs(schedule))`, including the default 5-minute top-of-hour stagger. Any staggered job whose hashed offset differs from the midpoint is plotted at the wrong time and can be marked past/pending incorrectly, so the marker computation needs to use the same per-job offset as the scheduler.
  - confidence: 0.91
