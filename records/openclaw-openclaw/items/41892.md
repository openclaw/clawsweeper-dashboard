## Summary

Keep open. Current main still lacks the requested Cron Jobs timeline/calendar surface, #40644 remains open, and this PR is the active implementation candidate, but the current patch is not merge-ready because the timeline occurrence math still diverges from runtime cron scheduling semantics.

## What This Changes

The PR adds a Control UI cron timeline strip with marker computation, CSS, i18n keys/metadata, a cron.status gatewayTimezone field, and a changelog entry.

## Best Possible Solution

Keep this PR open as the implementation candidate for #40644, but land it only after the occurrence calculation uses a shared/server-backed scheduler-compatible seam or exact shared helper, applies the real per-job stagger offset, adds focused timeline helper/render tests, updates Control UI docs, and coordinates any file-split conflicts with #41954.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: a top-of-hour cron job with default stagger is executed using a deterministic job-id hash offset in src/cron/service/jobs.ts, while the PR plots it at staggerMs/2; a Croner expression using L/W/# is accepted by runtime Croner but the PR parser skips those tokens and treats the restricted field as wildcard.

## Solution Assessment

No. The feature direction is useful, but the current solution is not the narrowest maintainable fix until it stops approximating runtime cron semantics in a separate UI parser or proves parity with focused tests; the safer path is to share occurrence computation with the scheduler or expose a server-side occurrence summary to the Control UI.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Match the scheduler's per-job stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: Runtime cron scheduling hashes the job id to pick a stable offset inside staggerMs, but this helper applies the same average staggerMs/2 to every cron job. Default top-of-hour jobs can therefore be plotted minutes away from their actual execution slot, which flips past/pending state and sends users to the wrong history window.
  - confidence: 0.9
- **[P2] Don't wildcard Croner calendar modifiers:** `ui/src/ui/helpers/cron-timeline.ts:433-435`
  - body: Croner supports calendar modifiers such as L, W, #, and +, and runtime scheduling delegates the full expression to Croner. This parser skips those tokens and callers treat an empty restricted field as wildcard, so jobs like last-day or nth-weekday schedules can show markers on days when they will not run.
  - confidence: 0.86
