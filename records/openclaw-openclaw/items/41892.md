## Summary

Keep open: current main still has the list/history-only cron UI, and this PR remains the active implementation candidate, but latest PR head still misaligns timeline markers with runtime cron stagger semantics and lacks current-head real behavior proof.

## What This Changes

The PR adds a Control UI cron timeline strip with markers, clusters, zoom controls, high-frequency chips, styling, translations, changelog text, and cron status timezone plumbing.

## Best Possible Solution

Keep this as the active implementation candidate, fix marker generation to mirror or share runtime scheduler stagger semantics, add focused coverage, and require current-head browser proof before merge.

## Reproduction Assessment

Yes. Source inspection at PR head shows the timeline uses an average stagger offset without job id, while current scheduler code uses a stable per-job hash and default top-of-hour stagger windows.

## Solution Assessment

No. The timeline direction is useful, but marker computation should mirror or share runtime scheduler logic for staggered cron jobs instead of approximating a midpoint that cannot account for job id or default stagger behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the scheduler's per-job stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: `getTodayRunHours` only receives `job.schedule` and applies `schedule.staggerMs / 2`, but runtime scheduling resolves the stagger window and hashes `job.id` to a stable offset.
