## Summary

Keep open: current main still lacks the requested cron timeline and this PR is the active implementation candidate, but it should not merge yet because marker generation still approximates staggered cron times instead of using the scheduler's stable per-job/default stagger behavior, and latest-head real behavior proof is insufficient.

## What This Changes

The PR adds a Control UI cron timeline with markers, clusters, zoom controls, high-frequency chips, styling, locale strings, changelog text, and a cron status gatewayTimezone field.

## Best Possible Solution

Keep this as the active implementation candidate, fix marker generation to use scheduler-equivalent stable offsets/default stagger behavior, add focused coverage, and require current-head browser proof; split broader calendar/week/drag scope from the linked issue if needed.

## Reproduction Assessment

Yes. Source inspection at PR head shows getTodayRunHours adds schedule.staggerMs / 2 and receives only job.schedule, while current runtime scheduling hashes job.id and applies default top-of-hour stagger windows.

## Solution Assessment

No. The timeline direction is useful, but marker computation should mirror or share runtime scheduler logic for staggered cron jobs instead of approximating a midpoint that cannot account for job id or default stagger behavior.
