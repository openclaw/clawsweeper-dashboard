## Summary

Keep open: current main still lacks the cron timeline, and this PR is a useful implementation for the linked feature request, but its client-side occurrence generation still diverges from runtime cron scheduling for staggered jobs and Croner calendar modifiers.

## What This Changes

The PR adds a Control UI Cron Jobs timeline strip with high-frequency chips, CSS/i18n/changelog updates, and a cron status gateway timezone field.

## Best Possible Solution

Land the timeline only after occurrence generation matches runtime scheduling, preferably by sharing/exposing scheduler-compatible helpers or returning server-computed occurrence summaries, with focused tests for staggered jobs and Croner calendar modifiers.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR plots cron occurrences with average stagger and wildcarded unsupported calendar tokens, while current runtime code/tests use per-job hash staggering and Croner accepts those modifiers.

## Solution Assessment

No. The feature direction is useful, but this implementation should not merge until the timeline stops approximating scheduler behavior with a divergent parser or proves parity with runtime-backed tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match the scheduler's stable cron stagger:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: Runtime cron scheduling resolves a job-specific SHA-256 offset inside the effective stagger window, including the default top-of-hour window. This helper applies the same average `staggerMs / 2` to every cron marker and cannot account for job id or default stagger, so markers can be minutes away from actual execution and flip past/pending state or history navigation.
  - confidence: 0.92
- **[P2] Do not wildcard Croner calendar modifiers:** `ui/src/ui/helpers/cron-timeline.ts:431-436`
  - body: Croner supports calendar modifiers such as `L`, `W`, `#`, and `+`, but this parser skips unrecognized restricted tokens and callers treat empty parsed fields as match-any. Last-day, nearest-weekday, nth-weekday, or explicit DOM/DOW AND schedules can therefore show timeline markers on days when runtime Croner will not run them.
  - confidence: 0.88
