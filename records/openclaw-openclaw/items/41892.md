## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Croner modifiers instead of wildcarding them:** `ui/src/ui/helpers/cron-timeline.ts:319-322`
  - body: For valid Croner syntax that this helper cannot parse, such as `L`, `W`, `MON#1`, or the `+` AND modifier, `parseCronField` returns an empty list and this path treats the empty restricted field as a match. Runtime uses Croner with the full expression, so the timeline can show runs on days when the job will not execute; use Croner/runtime projection or fail closed for unsupported fields instead of wildcarding them.
  - confidence: 0.94
- **[P1] Use real job status for past timeline markers:** `ui/src/ui/helpers/cron-timeline.ts:542`
  - body: This assigns every past marker `ok` solely from the marker time. Cron jobs already carry `lastRunStatus`/`lastStatus`, and the linked request asks for status-colored markers, so a job whose latest run failed or was skipped can be displayed as a healthy green past dot and send users to misleading history.
  - confidence: 0.91
- **[P2] Use the deterministic runtime stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:385-388`
  - body: Runtime cron scheduling hashes the job id to a stable offset inside the stagger window, but the timeline adds `staggerMs / 2` for every job. That can place markers at times the job will never use and flip past/pending state; pass the job id through and mirror the runtime offset, or use a shared projection helper.
  - confidence: 0.9
- **[P3] Remove the release-owned changelog entry:** `CHANGELOG.md:1988`
  - body: Normal PRs should not edit `CHANGELOG.md`; release generation owns that file. Keep the release-note context in the PR body or squash message instead, and drop this added entry from the branch.
  - confidence: 0.88
