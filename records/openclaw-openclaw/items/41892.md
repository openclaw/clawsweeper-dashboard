## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve Croner semantics for timeline markers:** `ui/src/ui/helpers/cron-timeline.ts:319-323`
  - body: The helper treats an empty parsed restricted DOM/DOW field as a match, while `parseCronField` drops Croner-supported tokens such as `L`, `W`, `#`, and `+`. Runtime scheduling uses Croner for the full expression, so jobs like `0 9 L * *` or `0 12 1 * +MON` can be plotted on days when they will not run; use the scheduler/Croner contract or fail closed for expressions the helper cannot evaluate.
  - confidence: 0.9
- **[P2] Use real job status for past timeline markers:** `ui/src/ui/helpers/cron-timeline.ts:542`
  - body: Every marker before the current clock time is labeled `ok`, even when the job's last run is `error`, `skipped`, or `unknown`. The existing Cron UI already resolves `lastRunStatus ?? lastStatus ?? unknown`, so this timeline can present failed jobs as healthy and send users to misleading run history.
  - confidence: 0.88
- **[P2] Apply the runtime stagger offset to markers:** `ui/src/ui/helpers/cron-timeline.ts:386-388`
  - body: This uses `staggerMs / 2` as an average offset, but runtime scheduling hashes the job id to a deterministic offset within the effective stagger window and also applies the default top-of-hour stagger. Timeline dots can therefore be minutes away from the actual run slot and flip `ok` versus `pending` around the boundary.
  - confidence: 0.86
