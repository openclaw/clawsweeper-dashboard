## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use the runtime cron contract for markers:** `ui/src/ui/helpers/cron-timeline.ts:431-436`
  - body: This parser only models a subset of Croner and then treats unsupported restricted fields as wildcard. Runtime scheduling uses Croner directly and supports seconds/year plus `L`, `W`, `#`, `+`, and `?`, so valid jobs can be plotted on the wrong days or shown far more often than they actually run. Please derive timeline occurrences from the same Croner/runtime contract or fail closed for unsupported expressions.
  - confidence: 0.9
- **[P2] Preserve exact cron stagger offsets:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: Runtime cron scheduling resolves a default top-of-hour stagger and then applies a stable per-job hash offset. The timeline only adds `schedule.staggerMs / 2` when an explicit value exists, so default-staggered jobs and explicitly staggered jobs can be shown at different times than they actually execute. Use the shared stagger resolution and stable offset logic, or surface only schedule windows without claiming exact run markers.
  - confidence: 0.86
- **[P2] Regenerate non-English timeline strings:** `ui/src/i18n/locales/de.ts:1240-1248`
  - body: Several non-English locale bundles add the new `cron.timeline` keys as English strings while their metadata reports no fallback keys. That makes localized Cron pages display English copy for this feature and hides the fallback from the i18n report. Regenerate through the UI i18n pipeline or mark the keys as fallback output consistently.
  - confidence: 0.82
- **[P3] Remove the release-owned changelog edit:** `CHANGELOG.md:11`
  - body: Normal feature PRs should not edit `CHANGELOG.md`; release generation owns that file. Keep the release-note context in the PR body or squash message instead so this branch does not conflict with release automation.
  - confidence: 0.9
