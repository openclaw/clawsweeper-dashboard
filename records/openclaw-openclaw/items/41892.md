## Summary

Keep open. Current main still lacks a Cron Jobs calendar/timeline view, and this open PR is the active implementation candidate for the linked feature request, but the patch is not merge-ready because the new UI occurrence calculator still diverges from runtime scheduler semantics for staggered cron jobs and Croner calendar modifiers.

## What This Changes

The PR adds a Control UI Cron Jobs timeline strip and high-frequency chips, plus supporting cron.status timezone metadata, CSS, i18n locale keys/metadata, and a changelog entry.

## Best Possible Solution

Land the timeline only after occurrence generation is scheduler-compatible: reuse or expose a shared runtime-backed occurrence helper, or enumerate through Croner with the same timezone, modifier, DOM/DOW, seconds/year, and deterministic stagger rules, then cover those cases with focused helper/render tests and coordinate the cron view split before merge.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: a top-of-hour cron job with default stagger is executed using a deterministic job-id hash offset in src/cron/service/jobs.ts, while the PR plots it at staggerMs/2; a Croner expression using L, W, #, or + is accepted by runtime Croner but the PR parser skips those tokens and treats the restricted field as wildcard.

## Solution Assessment

No. The feature direction is useful, but the current patch is not the narrowest maintainable solution until it stops approximating runtime cron semantics in a separate UI parser or proves parity with focused tests; the safer path is a shared scheduler-compatible occurrence seam or server-provided occurrence summary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Match the scheduler's per-job stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: Runtime cron scheduling hashes the job id to pick a stable offset inside staggerMs, but this helper applies the same average staggerMs/2 to every cron job. Default top-of-hour jobs can therefore be plotted minutes away from their actual execution slot, which flips past/pending state and sends users to the wrong history window.
  - confidence: 0.9
- **[P2] Don't wildcard Croner calendar modifiers:** `ui/src/ui/helpers/cron-timeline.ts:431-436`
  - body: Croner supports calendar modifiers such as L, W, #, and +, and runtime scheduling delegates the full expression to Croner. This parser skips those tokens and callers treat an empty restricted field as wildcard, so last-day, nearest-weekday, nth-weekday, or AND-constrained schedules can show markers on days when they will not run.
  - confidence: 0.86
