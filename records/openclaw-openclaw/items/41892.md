## What This Changes

This PR adds a Control UI cron timeline strip, high-frequency chips, gateway timezone exposure, timeline styles, i18n strings, and a changelog entry.

## Best Possible Solution

Keep this PR as the active implementation candidate, but align timeline occurrences with Croner/runtime scheduling and status semantics before maintainer UX acceptance.

## Solution Assessment

No. The UI direction is plausible, but the implementation should reuse or mirror runtime-compatible Croner occurrence calculation plus exact per-job stagger/status facts instead of wildcarding and approximating them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor Croner calendar modifiers when filtering dates:** `ui/src/ui/helpers/cron-timeline.ts:321`
  - body: Runtime scheduling gives the full expression to Croner, which supports `L`, `W`, `#`, and `+` calendar modifiers. Here an unparsed restricted DOM/DOW field becomes an empty array and is treated as a match, so schedules like `MON#1`, `15W`, or `L` can render markers on every day instead of only their real run dates.
  - confidence: 0.93
- **[P2] Use the same per-job stagger offset as runtime:** `ui/src/ui/helpers/cron-timeline.ts:386`
  - body: The runtime uses a stable hash of `job.id` modulo the stagger window, but the timeline always adds `staggerMs / 2`. Staggered cron jobs will usually be plotted at the wrong minute and can be marked past or pending incorrectly; compute the same per-job offset or expose the actual scheduled slots.
  - confidence: 0.91
- **[P2] Do not label every elapsed marker as ok:** `ui/src/ui/helpers/cron-timeline.ts:542`
  - body: This marks any elapsed schedule time as `ok` without checking the job's run status. A cron job whose latest run failed will still show an `ok` marker and popup once its time passes, contradicting the timeline's status cue; use `job.state?.lastStatus` where appropriate or separate time state from run result.
  - confidence: 0.86
