## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve Croner semantics for timeline markers:** `ui/src/ui/helpers/cron-timeline.ts:319-323`
  - body: The helper treats an empty parsed restricted month/day field as a wildcard, but `parseCronField` returns empty for syntax it cannot evaluate. Since runtime cron execution is delegated to Croner, this can plot jobs on dates where runtime will not run them; delegate to the runtime contract or fail closed for expressions this helper cannot prove.
  - confidence: 0.9
- **[P2] Use real job state for past marker health:** `ui/src/ui/helpers/cron-timeline.ts:542`
  - body: Every marker before the current clock time is labeled `ok`, regardless of `job.state.lastRunStatus` or `lastStatus`. A failed, skipped, or unknown last run can therefore appear healthy in the timeline and route users to misleading history.
  - confidence: 0.88
- **[P2] Apply the runtime stagger offset to markers:** `ui/src/ui/helpers/cron-timeline.ts:386-388`
  - body: This uses `staggerMs / 2` as an average offset, while runtime scheduling hashes the job id into a deterministic offset and also applies the default top-of-hour stagger. Timeline dots can land minutes away from the actual slot and flip past versus pending near the boundary.
  - confidence: 0.86
