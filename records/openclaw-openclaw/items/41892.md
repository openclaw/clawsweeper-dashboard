## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the runtime stagger offset for markers:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: Runtime scheduling hashes `job.id` to a stable offset inside `staggerMs`, but this helper shifts every staggered cron by the midpoint. For default top-of-hour stagger or explicit larger windows, the timeline can place dots before or after the actual run and misclassify them as past/pending.
  - confidence: 0.92
- **[P2] Do not wildcard unsupported Croner tokens:** `ui/src/ui/helpers/cron-timeline.ts:319-330`
  - body: The runtime evaluates full Croner syntax, including `L`, `W`, and `#`, but this parser skips those tokens and treats an empty parsed constraint as a match. A valid schedule like `0 9 L * *` would appear every day instead of only the last day, so the timeline becomes unreliable for supported cron expressions.
  - confidence: 0.9
- **[P2] Do not label elapsed markers as successful runs:** `ui/src/ui/helpers/cron-timeline.ts:542`
  - body: A scheduled time being in the past is not the same as a successful run. With `status: hour <= nowH ? "ok" : "pending"`, a job whose latest run errored still renders a green OK marker, even though the existing job state already distinguishes ok/error/skipped.
  - confidence: 0.86
