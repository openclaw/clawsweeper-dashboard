## Summary

Keep open. Current main does not include the cron timeline work, and this PR is a useful implementation path for the linked calendar-view request, but it is not merge-ready because the timeline still approximates scheduler behavior instead of matching the Croner/runtime contract.

## What This Changes

This PR adds a Control UI cron timeline strip with high-frequency chips, zoom/hover/click interactions, gateway timezone metadata, CSS, i18n keys, and a changelog entry.

## Best Possible Solution

Keep the PR open, but make the timeline derive occurrences from the same Croner/runtime semantics as the scheduler, including supported cron modifiers, optional year fields, timezone resolution, and deterministic per-job stagger offsets.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows the custom UI parser wildcarding Croner-supported fields and averaging stagger offsets while current runtime code/tests use Croner and stable hash-derived offsets. I did not run browser tests in this read-only review.

## Solution Assessment

No. The current PR direction is useful, but the maintainable fix is to share or reuse scheduler-grade occurrence calculation rather than keeping an approximate UI-only cron parser.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match Croner semantics for timeline occurrences:** `ui/src/ui/helpers/cron-timeline.ts:431-436`
  - body: The helper skips Croner-supported tokens such as `L`, `W`, `#`, and `+`, then treats an empty restricted field as a wildcard. Runtime evaluates the full expression with Croner, so schedules like `0 12 1 * +MON` or `0 9 L * *` can be plotted on days they do not actually run. Please derive today's occurrences from Croner or a shared scheduler helper instead of wildcarding unsupported syntax.
  - confidence: 0.93
- **[P2] Use the stable per-job stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: Adding `staggerMs / 2` only plots the average of the stagger window, but runtime hashes the job id to a stable offset. Default top-of-hour jobs can run anywhere in the 0..5m window while the timeline always shows +2.5m, so status and history navigation can be wrong near boundaries. Reuse/export the runtime offset logic or have the backend return effective run times.
  - confidence: 0.91
