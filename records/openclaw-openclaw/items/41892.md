## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Build the timeline from the complete job set:** `ui/src/ui/views/cron.ts:539`
  - body: `props.jobs` is only the currently visible cron table data, and current main loads cron jobs in 50-row pages with `jobsHasMore`. With more matching jobs than the loaded page, the timeline silently omits scheduled tasks and stops being the advertised daily overview; fetch or expose a complete bounded occurrence dataset instead of using the table page.
  - confidence: 0.97
- **[P1] Use the scheduler's exact stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: Adding `staggerMs / 2` does not match runtime scheduling: the service hashes the job id to choose a stable offset anywhere in the stagger window, and it also applies a default top-of-hour stagger when `staggerMs` is absent. This can place markers minutes away from actual runs and flip past/pending state; use scheduler-owned occurrence data or the same offset helper.
  - confidence: 0.98
- **[P1] Do not wildcard Croner calendar modifiers:** `ui/src/ui/helpers/cron-timeline.ts:431-436`
  - body: Croner 10.0.1 supports `L`, `W`, and `#`, but this parser skips unrecognized tokens and documents that callers treat an empty restricted field as unrestricted. A last-day or nth-weekday job can therefore appear on ordinary days, sending users to empty history or hiding the actual schedule; enumerate with Croner or fail closed for unsupported expressions.
  - confidence: 0.96
- **[P1] Honor Croner's plus day-of-week AND modifier:** `ui/src/ui/helpers/cron-timeline.ts:314-324`
  - body: This OR branch applies whenever both DOM and DOW are restricted, but Croner uses a `+` prefix on day-of-week to require AND logic. For `0 9 15 * +1`, runtime only runs when the 15th is Monday, while the timeline would plot every Monday or every 15th; detect the modifier or delegate occurrence calculation to Croner.
  - confidence: 0.93
