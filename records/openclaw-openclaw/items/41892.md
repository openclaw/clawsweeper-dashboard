## Reproduction Assessment

Yes. The blocking finding is source-reproducible: the PR plots staggered cron jobs at `staggerMs / 2`, while current runtime scheduling uses a stable per-job hash offset; no live browser repro was run in this read-only review.

## Solution Assessment

No. The timeline direction is useful, but the implementation should mirror or share the scheduler's stable per-job stagger calculation instead of approximating it with a midpoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the scheduler's per-job stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:386`
  - body: `getTodayRunHours` adds `schedule.staggerMs / 2` to every cron marker, but runtime scheduling hashes the job id and uses `hash(job.id) % staggerMs`. Most staggered jobs will be plotted at the wrong time, which can flip past/pending state and send users to history for a run that has not happened yet. Pass the job id into the helper and mirror or share the scheduler's stable offset calculation.
  - confidence: 0.92
