Keep open: this is an active feature PR for the linked cron calendar request, but the current patch still has a correctness bug in staggered cron marker placement and needs current-head real behavior proof before merge.

## What This Changes

The PR adds a 24-hour cron timeline and always-on chip UI to Control UI, threads gateway timezone through cron status, adds timeline styles/i18n keys, and updates the changelog.

## Best Possible Solution

Keep reviewing this PR as the active timeline implementation, but require exact scheduler-aligned marker computation, focused helper coverage, and current-head real behavior proof before merge.

## Reproduction Assessment

Yes, source-reproducible for the blocking review finding: current runtime uses a stable per-job stagger offset, while the PR plots staggered cron jobs at `staggerMs / 2`. A live browser repro was not run in this read-only review.

## Solution Assessment

No. The timeline direction is useful, but the implementation should mirror or share the scheduler's per-job stagger calculation instead of approximating it with a midpoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the scheduler's per-job stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:386`
  - body: `getTodayRunHours` adds `schedule.staggerMs / 2` to every cron marker, but runtime scheduling hashes the job id and uses `hash(job.id) % staggerMs` before computing next/previous runs. Any staggered job whose stable offset is not exactly half the window will be plotted at the wrong time, which can flip past/pending state and send users to history for a run that has not actually happened. Pass the job id into the helper and mirror or share the stable offset calculation instead of averaging.
  - confidence: 0.92
