## Summary

Keep open: current main still lacks the Control UI cron timeline, and this PR remains the active implementation candidate for https://github.com/openclaw/openclaw/issues/40644, but the patch still misaligns staggered cron markers and lacks current-head real behavior proof.

## What This Changes

This PR adds a Control UI cron 24-hour timeline strip with markers, clusters, zoom controls, high-frequency chips, styling, locale strings, changelog text, and a cron status gatewayTimezone field.

## Best Possible Solution

Keep this PR as the active timeline candidate, but make marker computation mirror or share the scheduler's stable per-job/default stagger logic, add focused helper coverage, and require current-head real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection at PR head shows getTodayRunHours adds schedule.staggerMs / 2, while current runtime scheduling hashes job.id for a stable offset and applies default top-of-hour stagger windows.

## Solution Assessment

No. The timeline direction is useful, but the stagger implementation should mirror or share scheduler logic instead of approximating with a midpoint that cannot account for job id or default stagger behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the scheduler's per-job stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: `getTodayRunHours` applies `schedule.staggerMs / 2` and is called with only `job.schedule`, while runtime scheduling resolves the stagger window and hashes `job.id` to a stable offset. Staggered cron jobs, including default top-of-hour jobs without explicit `staggerMs`, can be plotted at the wrong time, flipping pending/past state and sending users to the wrong history/details flow.
  - confidence: 0.92
