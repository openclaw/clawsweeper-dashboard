## Summary

Keep open: current main still lacks the cron timeline feature, and this PR still has a source-backed scheduler-stagger mismatch plus insufficient current-head behavior proof.

## What This Changes

The PR adds a Control UI cron timeline strip with marker computation, clustering, zoom controls, styles, locale keys, changelog text, and cron status timezone plumbing.

## Best Possible Solution

Land a focused timeline slice only after marker generation mirrors runtime scheduler offsets, locale bundles are regenerated correctly, focused coverage is added, and current-head browser proof is supplied.

## Reproduction Assessment

Yes. Source inspection at PR head shows marker generation passes only the schedule to getTodayRunHours and applies an average stagger, while current scheduler code uses job.id plus the resolved default or explicit stagger window.

## Solution Assessment

No. The UI direction is plausible, but marker computation should share or mirror runtime scheduler stagger semantics instead of approximating a midpoint that cannot account for job identity or default top-of-hour stagger behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use scheduler stagger offsets for timeline markers:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
  - body: `getTodayRunHours` only receives `job.schedule` and applies `schedule.staggerMs / 2`, but runtime scheduling resolves default top-of-hour stagger and hashes `job.id` to a stable offset. Staggered cron jobs can be plotted at the wrong time and misclassified as past or pending.
  - confidence: 0.93
- **[P3] Regenerate translated timeline locale strings:** `ui/src/i18n/locales/de.ts:1241-1251`
  - body: This adds English cron timeline strings to a non-English bundle while the locale metadata reports no fallback keys and all keys translated. German users will see untranslated controls, and i18n reporting cannot flag them as fallbacks.
  - confidence: 0.86
