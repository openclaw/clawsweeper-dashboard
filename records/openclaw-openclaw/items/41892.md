## Summary

Keep open: current main still lacks the cron calendar/timeline feature and this PR remains the active implementation candidate for https://github.com/openclaw/openclaw/issues/40644, but the patch is not merge-ready because timeline markers still diverge from runtime cron stagger behavior and current-head real behavior proof is insufficient.

## What This Changes

The PR adds a 24-hour Control UI cron timeline with markers, clusters, high-frequency chips, zoom controls, timezone status plumbing, styles, translations, and changelog text.

## Best Possible Solution

Keep this PR as the active timeline candidate, fix marker computation to share or exactly mirror runtime cron scheduling, add focused helper coverage, and require current-head real behavior proof before merge.

## Reproduction Assessment

Yes. The blocking patch issue is source-reproducible: PR head applies schedule.staggerMs / 2 without job identity, while current runtime scheduling resolves the effective stagger window and hashes job.id for a stable offset.

## Solution Assessment

No. The timeline direction is useful, but the implementation should mirror or share the scheduler's resolveCronStaggerMs plus stable per-job hash offset instead of approximating marker times.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Use the scheduler's stable cron stagger offset:** `ui/src/ui/helpers/cron-timeline.ts:384-388`
