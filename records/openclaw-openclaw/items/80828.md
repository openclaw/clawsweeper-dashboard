## Label Justifications

- `P2`: This is meaningful auth/security hardening for remote Control UI access, but it is a new feature/design request rather than an active outage or confirmed bypass.
- `impact:security`: The issue proposes changing the browser authentication and device-pairing trust boundary by using a messaging channel as an additional factor.
- `impact:auth-provider`: The request affects Control UI authentication, device pairing, trusted identity, and SecretRef-adjacent operator authorization flows.

## Summary

Keep open. Current main still uses shared auth plus requestId-based Control UI device pairing, with no WhatsApp or messaging-delivered OTP handshake; the remaining work is security-sensitive product design, not conservative cleanup.

## What This Changes

Request to authenticate unrecognized Control UI browser devices by sending a short-lived one-time code through WhatsApp or another messaging channel and using it to pair the device.

## Best Possible Solution

Define an opt-in, channel-agnostic device-pairing approval or OTP design in core, with WhatsApp as one transport plus explicit target config, expiry, replay protection, rate limits, audit logs, scope-bound approval, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create a pending device-pair request and must be approved by existing requestId-based paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp-delivered OTP may fit single-operator deployments, but the safer maintainable direction is a channel-agnostic, opt-in pairing approval design rather than hardcoding WhatsApp into gateway auth.
