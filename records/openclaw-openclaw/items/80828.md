## Summary

Keep open. Current main still has requestId-based Control UI device pairing and existing gateway auth modes, but no WhatsApp or channel-delivered browser OTP handshake; this is a security-sensitive product feature, not a cleanup-close candidate.

## What This Changes

Request to let an unrecognized Control UI browser complete device pairing with a short one-time code delivered through WhatsApp or another messaging channel.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or OTP flow with WhatsApp as one transport, explicit target config, expiry, replay protection, rate limits, audit logs, and scope-preserving approval.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create a pending device-pair request and must be approved by existing requestId-based paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp-only OTP may help single-operator deployments, but the safer maintainable direction is an opt-in, channel-agnostic pairing approval design rather than hardcoding WhatsApp into gateway auth.

## Security Review

Status: needs_attention

Summary: The request proposes a new browser authentication factor over a messaging channel, so it needs explicit security design before implementation.

Concerns:

- **[medium] Define the messaging auth trust boundary:** `docs/channels/whatsapp.md:220`
  - body: WhatsApp `allowFrom` is currently a DM sender access-control list, not a Control UI authentication factor; using it for browser pairing would broaden that boundary and needs opt-in target, expiry, replay, rate-limit, and audit decisions.
  - confidence: 0.9
- **[medium] Preserve pairing approval authority:** `src/gateway/server-methods/devices.ts:258`
  - body: Device approval currently validates requestId and caller scopes before minting approved access; a messaging OTP or approval path must not bypass those scope checks or silently grant broader operator access.
