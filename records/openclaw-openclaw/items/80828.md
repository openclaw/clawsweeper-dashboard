## Summary

Keep open. Current main and the latest release still use requestId-based Control UI device pairing, Tailscale/trusted-proxy auth, and channel pairing; they do not provide a WhatsApp-delivered browser OTP handshake, which would be a new security-sensitive auth feature.

## What This Changes

The issue requests a WhatsApp-delivered one-time-code handshake for pairing unrecognized Control UI browsers without SSH or CLI approval.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or notification flow, with WhatsApp as one possible transport and explicit target config, threat model, expiry, rate limits, audit logs, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create a pending requestId and require existing approval paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp OTP may help single-operator deployments, but the maintainable path should be an opt-in channel-agnostic approval design rather than hardcoding WhatsApp into gateway auth.
