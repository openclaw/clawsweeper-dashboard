## Summary

Keep open. Current main still uses Control UI auth modes plus requestId-based device pairing, and there is no WhatsApp-delivered browser OTP handshake; the request is a security-sensitive product feature, not a cleanup-close candidate.

## What This Changes

Request to let unrecognized Control UI browsers authenticate or pair with a one-time code delivered over WhatsApp or another messaging channel.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval flow with WhatsApp as one transport, preserving request scope checks and documenting the security boundary.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create a pending device-pair request and must be approved through existing requestId-based paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp-only OTP may fit the reporter's single-operator setup, but the maintainable direction is a channel-agnostic, opt-in pairing approval design rather than hardcoding WhatsApp into gateway auth.

## Security Review

Status: needs_attention

Summary: The request would turn a messaging channel identity into a Control UI authentication factor, so it needs explicit security design before implementation.

Concerns:

- **[medium] Define the messaging auth trust boundary:** `docs/channels/whatsapp.md:220`
  - body: WhatsApp `allowFrom` is documented as a DM sender access-control list, not a Control UI authentication factor; using it for browser pairing needs opt-in target selection, expiry, replay protection, rate limits, and audit behavior.
  - confidence: 0.9
- **[medium] Preserve pairing approval authority:** `extensions/device-pair/pair-command-auth.ts:13`
  - body: Messaging commands currently map command owners to `operator.pairing`, while device approvals remain scope-bounded in gateway methods; any OTP or message approval path must not bypass those approval-time checks.
  - confidence: 0.87
