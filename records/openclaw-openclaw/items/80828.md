## Summary

Keep open. Current main still has manual device-pair approval, Tailscale/trusted-proxy identity paths, and a device-pair notification plugin, but it does not implement a WhatsApp-delivered Control UI OTP handshake; that is a new security-sensitive auth feature rather than conservative cleanup.

## What This Changes

Request a WhatsApp or messaging-delivered OTP flow that lets an unrecognized browser authenticate and pair with the Control UI without SSH or per-device tunnel setup.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or notification flow, likely through the device-pair plugin or a versioned SDK seam, with explicit auth semantics before any WhatsApp-specific OTP implementation.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs provide a high-confidence current-behavior check: current main supports manual device-pair approval and trusted identity bypasses, but no WhatsApp or messaging OTP handshake.

## Solution Assessment

Unclear as requested. A WhatsApp-only OTP flow may address the reporter's workflow, but the narrower maintainable direction is probably a generic, opt-in device-pair approval channel with a documented security model rather than hardcoding WhatsApp into gateway auth.
