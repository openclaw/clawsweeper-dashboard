## Summary

Keep open. Current main still uses existing Control UI auth modes plus requestId-based device pairing, and there is no WhatsApp-delivered browser OTP handshake; this is a security-sensitive product feature, not a cleanup-close candidate.

## What This Changes

Request a WhatsApp or messaging-delivered one-time-code flow that pairs unrecognized Control UI browsers without SSH or per-device setup.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or notification flow with WhatsApp as one transport, explicit targets, expiry, replay protection, rate limits, audit logs, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source, docs, and tests give a high-confidence current-main behavior check: unpaired Control UI browsers create pending requestId-based pairing requests and must be approved through existing paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp-only OTP may fit the single-operator use case, but the narrower maintainable direction is an opt-in, channel-agnostic pairing approval design rather than hardcoding WhatsApp into gateway auth.
