## Summary

Keep open. Current main still supports requestId-based Control UI device pairing plus token/password/Tailscale/trusted-proxy auth, but no WhatsApp-delivered OTP browser handshake; this is a security-sensitive feature request that needs maintainer product design rather than cleanup closure.

## What This Changes

The issue requests a WhatsApp-delivered one-time-code flow for pairing and authenticating new Control UI browser devices.

## Best Possible Solution

Design an opt-in, channel-agnostic pairing approval or notification flow with WhatsApp as one transport while preserving existing device-pair scope checks, auditability, expiry, replay protection, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create a pending requestId-based pairing request and must be approved through existing pairing paths, not WhatsApp OTP.

## Solution Assessment

Unclear. The requested WhatsApp-only OTP may fit the reporter's single-operator deployment, but the more maintainable direction is likely a channel-agnostic, opt-in approval design instead of hardcoding WhatsApp into gateway auth.
