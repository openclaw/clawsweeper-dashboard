## Summary

Keep open. Current main still uses shared/trusted gateway auth plus requestId-based device pairing/device tokens for Control UI access; there is no WhatsApp or messaging-delivered OTP handshake, and the remaining work is security-sensitive product design rather than conservative cleanup.

## What This Changes

Requested a social-identity MFA flow that delivers a one-time code over WhatsApp or another messaging channel to authenticate or pair a new Control UI browser/device.

## Best Possible Solution

Adopt a maintainer-approved, opt-in, channel-agnostic device-pairing approval or OTP design with explicit transport targets, expiry, replay protection, rate limiting, audit semantics, and docs/tests before adding any WhatsApp-specific implementation.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers use requestId/device-token pairing and Gateway-host approval, not WhatsApp or messaging OTP.

## Solution Assessment

Unclear pending maintainer design. The problem fits OpenClaw's security direction, but a WhatsApp-only gateway auth path is not the best final shape without a channel-agnostic approval boundary and explicit security policy.
