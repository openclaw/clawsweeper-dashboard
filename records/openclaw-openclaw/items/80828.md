## Summary

Keep open. Current main still implements requestId-based Control UI device pairing and existing gateway auth modes, but not a WhatsApp-delivered browser OTP handshake; this is a security-sensitive product feature, not a conservative cleanup close.

## What This Changes

Request a WhatsApp or messaging-delivered one-time-code flow to authenticate and pair unrecognized Control UI browser devices without SSH or local CLI approval.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or OTP flow with WhatsApp as one transport, preserving approval scopes, expiry, replay protection, rate limits, audit logs, and public docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create a pending device-pair request and must be approved through existing requestId-based paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp-only OTP may fit single-operator deployments, but the more maintainable direction is an opt-in, channel-agnostic pairing approval design rather than hardcoding WhatsApp into gateway authentication.
