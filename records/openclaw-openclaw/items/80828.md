## Summary

Keep open. Current main still has requestId-based Control UI device pairing and existing gateway auth modes, but no WhatsApp-delivered browser OTP handshake; the request is a security-sensitive product feature rather than a cleanup-close candidate.

## What This Changes

Request adding a WhatsApp or messaging-delivered one-time code flow to authenticate and pair Control UI browser sessions without SSH or CLI approval.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or notification flow with WhatsApp as one transport, explicit target config, scope-preserving approval, expiry, replay protection, rate limits, audit logs, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create a pending device-pair request and must be approved by existing requestId-based paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp-only OTP may be useful for single-operator deployments, but the safer maintainable direction is a channel-agnostic, opt-in pairing approval design rather than hardcoding WhatsApp into gateway auth.

## Security Review

Status: needs_attention
