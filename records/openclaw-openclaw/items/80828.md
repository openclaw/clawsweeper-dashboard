## Summary

Keep open. Current main has manual device-pair approval, Tailscale/trusted-proxy auth paths, and chat-based device-pair helpers, but it does not implement a WhatsApp-delivered Control UI OTP handshake; this is a new security-sensitive auth/product feature.

## What This Changes

Request a WhatsApp or messaging-delivered one-time code flow that lets an unrecognized browser authenticate and pair with the Control UI without SSH or an already-paired device.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or notification flow with explicit target config, threat model, expiry, replay/rate-limit controls, audit logging, and scope-preserving approval semantics.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-behavior check: remote Control UI pairing is approved through existing device-pair paths or skipped by existing trusted identity modes, not by a WhatsApp OTP handshake.

## Solution Assessment

Unclear. A WhatsApp-only OTP flow is plausible, but the narrower maintainable direction is likely a generic device-pair approval/notification seam or device-pair plugin extension rather than hardcoding WhatsApp into gateway auth.
