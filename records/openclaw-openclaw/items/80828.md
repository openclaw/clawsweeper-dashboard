## Summary

Keep open. Current main still has requestId-based Control UI device pairing and Telegram-only device-pair request notifications, but no WhatsApp-delivered browser OTP handshake; this is a security-sensitive product feature rather than a cleanup-close candidate.

## What This Changes

Request to authenticate unrecognized Control UI browsers by sending a short-lived WhatsApp OTP to an allowlisted operator number and pairing the device after code entry.

## Best Possible Solution

Design an opt-in, channel-agnostic pairing approval or auth challenge with explicit transport config, scoped approval authority, expiry, replay/rate limits, audit logs, and docs; add WhatsApp as one transport only after the security model is agreed.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create pending device-pair requests and must be approved by existing requestId-based paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp-only OTP may fit single-operator deployments, but the safer maintainable direction is an opt-in channel-agnostic pairing approval/auth design rather than hardcoding WhatsApp into gateway auth.
