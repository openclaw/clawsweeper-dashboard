## Summary

Keep open. Current main and the latest shipped release support manual Control UI device-pair approval, Tailscale/trusted-proxy auth paths, and channel pairing, but not a WhatsApp-delivered browser OTP handshake; this is a new security-sensitive auth/product feature.

## What This Changes

Request a short-lived WhatsApp-delivered one-time code flow that lets unrecognized Control UI browsers authenticate and become paired devices.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or notification flow, with WhatsApp as one possible transport, explicit target config, threat model, rate limits, expiry, audit logs, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers get a pending device-pair request and must be approved by existing requestId-based paths, not WhatsApp OTP.

## Solution Assessment

Unclear. A WhatsApp-only OTP may be useful for single-operator deployments, but the safer maintainable direction is a channel-agnostic device-pair approval design rather than hardcoding WhatsApp into gateway auth.

## Security Review

Status: needs_attention
