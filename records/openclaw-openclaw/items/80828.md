## Summary

Keep open. Current main documents and implements manual/identity-based Control UI pairing, but not a WhatsApp or messaging OTP authentication handshake; the request is a new security-sensitive auth feature that needs maintainer product and threat-model design.

## What This Changes

Add a WhatsApp or messaging-delivered one-time-code flow for authenticating and pairing new Control UI browser devices.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or authentication-factor flow with explicit config, threat model, delivery target selection, expiry, replay/rate limits, and audit logging, then implement WhatsApp through the plugin seam if the design supports it.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: remote Control UI pairing is manual or identity-header based, not a WhatsApp OTP handshake.

## Solution Assessment

Unclear. The requested WhatsApp-only OTP flow is plausible, but the safer maintainable direction is a channel-agnostic device-pair approval/auth seam or a deliberate device-pair plugin extension rather than hardcoding WhatsApp into gateway authentication.
