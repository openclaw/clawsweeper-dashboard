## Summary

Keep open. Current main has manual device-pair approval, Tailscale/trusted-proxy auth paths, and a device-pair plugin, but it does not implement a WhatsApp-delivered Control UI OTP handshake; this is a new security-sensitive product decision rather than conservative cleanup.

## What This Changes

Add an opt-in WhatsApp or messaging one-time-code flow for approving unrecognized Control UI browser/device pairing without local SSH or CLI access.

## Best Possible Solution

Design this as an opt-in, channel-agnostic device-pair approval or notification flow, likely extending the device-pair plugin or a generic SDK seam, with explicit target config, threat model, rate limits, expiry, and audit logging.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs confirm the current-main behavior: remote Control UI pairing is approved through existing device-pair paths or skipped only by existing trusted identity modes, not by a WhatsApp OTP handshake.

## Solution Assessment

Unclear. A WhatsApp-only OTP flow is plausible, but the narrower maintainable direction is probably a channel-agnostic device-pair approval/notification seam or a device-pair plugin extension rather than hardcoding WhatsApp into gateway auth.
