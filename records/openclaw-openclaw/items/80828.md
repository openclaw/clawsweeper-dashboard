## Summary

Keep open. Current main and the latest release still use shared/trusted gateway auth plus requestId-based device pairing/device tokens for Control UI access; there is no WhatsApp or messaging-delivered OTP handshake, and the remaining work is security-sensitive product design rather than conservative cleanup.

## What This Changes

Request to authenticate unrecognized Control UI browsers with a one-time code delivered through WhatsApp or another messaging channel.

## Best Possible Solution

Keep the issue open for a maintainer-approved, opt-in, channel-agnostic remote device-pairing approval design that preserves device identity, scope bounds, replay protection, rate limits, audit semantics, and WhatsApp/plugin boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs provide a high-confidence current-main behavior check: remote Control UI pairing uses device identity, requestId approval, and device tokens, not a messaging OTP.

## Solution Assessment

Unclear until maintainers choose the auth model. A WhatsApp-only OTP may be a useful delivery mechanism, but the durable solution should be channel-agnostic and preserve device identity, scopes, expiry, replay prevention, rate limiting, and audit semantics.
