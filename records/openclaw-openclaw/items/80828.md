## Summary

Keep open. Current main still uses manual device-pair approval, Tailscale/trusted-proxy identity paths, and a device-pair plugin, but it does not implement a WhatsApp-delivered Control UI OTP handshake; this is a new security-sensitive auth feature rather than conservative cleanup.

## What This Changes

Request to authenticate new Control UI browsers with a short-lived one-time code delivered over WhatsApp to an allowlisted operator number.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or MFA seam with explicit target config, expiry, replay/rate limits, audit logging, and scope rules, then decide whether the WhatsApp plugin should implement that seam.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-behavior check: new Control UI browsers are approved through existing device-pair paths or existing trusted identity modes, not by a WhatsApp OTP handshake.

## Solution Assessment

Unclear. A WhatsApp OTP flow may be useful, but hardcoding WhatsApp into gateway auth is not the narrowest maintainable direction; a generic, opt-in approval/MFA seam would be safer.
