## Summary

Keep open. Current main still has requestId-based Control UI pairing and existing gateway auth modes, but no WhatsApp or messaging-delivered OTP handshake; this is a security-sensitive product feature, not a cleanup-close candidate.

## What This Changes

Request to add a WhatsApp or messaging-delivered one-time-code handshake for remote Control UI browser authentication and device pairing.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or OTP flow with WhatsApp as one transport, explicit configuration, scoped approval, expiry, replay protection, rate limits, audit logs, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source, docs, and tests give a high-confidence current-main behavior check: unpaired Control UI browsers use requestId-based pairing and no WhatsApp OTP branch exists.

## Solution Assessment

Unclear. WhatsApp OTP may be useful for single-operator deployments, but the safer maintainable direction is an opt-in, channel-agnostic pairing approval design rather than hardcoding WhatsApp into gateway auth.
