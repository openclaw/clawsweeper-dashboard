## Summary

Current main now generates macOS Gateway LaunchAgent plists with `ProcessType=Interactive`, which addresses the launchd classification boundary implicated by the macOS LAN/SSH `No route to host` reports; the fix is present in the first beta after the merge.

## What This Changes

The issue reports macOS LaunchAgent-backed gateway exec failing to reach an SSH/LAN host while Terminal and direct runtime execution succeed.

## Best Possible Solution

Keep the shared `ProcessType=Interactive` LaunchAgent generation and route any remaining Tailscale Funnel-specific LAN egress behavior to the narrower open tracker.

## Reproduction Assessment

Yes for the pre-fix behavior: the issue and related LAN reports give concrete macOS LaunchAgent plus `tools.exec.host=gateway` SSH/nc/curl steps, and the missing generated `ProcessType` key was source-visible. I did not rerun the fixed current-main path on a macOS LAN host.

## Solution Assessment

Yes. Adding `ProcessType=Interactive` in the shared LaunchAgent plist builder is the narrowest maintainable fix for the macOS launchd classification boundary behind these gateway exec reachability reports; the Tailscale Funnel-specific PID/audit-token case remains separate.
