## Summary

Keep open: current main still routes voice-call realtime WebSocket upgrades with a raw prefix check, and an open closing PR already targets the bug but has not merged.

## What This Changes

The issue asks voice-call realtime WebSocket upgrades to accept only the exact configured stream path or slash-delimited child routes, not sibling prefixes.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79919 or an equivalent focused fix that enforces exact-or-slash-child realtime stream path matching with regression coverage.

## Reproduction Assessment

Yes, at source level: current main uses `pathname.startsWith(pattern)`, and the same predicate returns true for `/voice/stream/realtime-extra/token` against `/voice/stream/realtime`. I did not run a live server because this was a read-only review.

## Solution Assessment

Yes: exact-or-slash-child matching is the narrow maintainable fix because generated stream URLs append the token as a slash-delimited child. The issue should stay open until the linked PR or an equivalent patch lands.
