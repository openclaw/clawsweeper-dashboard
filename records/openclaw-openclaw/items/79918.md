## Summary

Keep open: current main still uses raw prefix matching for realtime voice-call WebSocket upgrades, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks OpenClaw to reject sibling realtime voice-call WebSocket paths such as `/voice/stream/realtime-extra/<token>` when the configured stream path is `/voice/stream/realtime`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79919 or an equivalent focused exact-or-slash-child route-boundary fix with regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level: current main checks `pathname.startsWith(pattern)`, so `/voice/stream/realtime-extra/token` matches the configured `/voice/stream/realtime` route. I did not run a live server because this was a read-only review.

## Solution Assessment

Yes: exact-or-slash-delimited matching is the narrow maintainable fix because generated realtime TwiML uses the configured stream path plus a slash child token. The issue should stay open until the linked PR or an equivalent patch lands.
