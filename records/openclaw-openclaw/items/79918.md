## Summary

Keep open: current main still uses raw prefix matching for realtime voice-call WebSocket upgrades, and the linked focused fix PR remains open with closing syntax, so this issue should not be closed before that PR or an equivalent patch lands.

## What This Changes

The issue asks to tighten voice-call realtime WebSocket route matching so sibling paths like `/voice/stream/realtime-extra/<token>` no longer reach the realtime handler.

## Best Possible Solution

The route classifier should require the exact configured stream path or a slash-delimited child path, preserve root stream-path behavior, and carry focused webhook regression tests.

## Reproduction Assessment

Yes, at source level. Current main still checks `pathname.startsWith(pattern)`, so `/voice/stream/realtime-extra/token` matches the configured `/voice/stream/realtime` route boundary without needing a live provider to see the bug.

## Solution Assessment

Yes. The linked PR's exact-or-slash-child matching is the narrow maintainable fix because it changes only the realtime route predicate and adds focused regression coverage while preserving token, TwiML, and provider behavior.
