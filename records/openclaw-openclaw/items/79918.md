## Summary

Keep open: current main still uses raw prefix matching for realtime voice-call WebSocket upgrades, and the open closing PR already owns the focused fix path.

## What This Changes

The issue reports that voice-call realtime WebSocket upgrades treat sibling stream paths as realtime routes because the router uses raw prefix matching.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79919, or an equivalent focused patch, so realtime upgrades only match the exact configured stream path and slash-delimited token child paths with regression coverage.

## Reproduction Assessment

Yes, at source level: current main checks `pathname.startsWith(pattern)`, so `/voice/stream/realtime-extra/token` is classified as realtime for the configured `/voice/stream/realtime` path. I did not run a live server because this was a read-only review.

## Solution Assessment

Yes: exact-or-slash-child matching is the narrow maintainable fix because generated realtime URLs append the token as a child path. The issue should stay open until the linked PR or an equivalent patch lands.
