## Summary

Keep open: current main still routes realtime voice-call WebSocket upgrades with a raw prefix match, and the active linked fix PR should be reviewed or landed before this issue closes.

## What This Changes

The issue asks voice-call realtime WebSocket upgrades to match only the configured stream path or slash-delimited child paths, rejecting sibling prefixes.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79919 or an equivalent owner-scoped fix that uses exact-or-slash-child realtime route matching with regression coverage.

## Reproduction Assessment

Yes, at source level: current main checks `pathname.startsWith(pattern)`, so `/voice/stream/realtime-extra/token` is classified as realtime for `/voice/stream/realtime`. I did not run a live server in this read-only review.

## Solution Assessment

Yes: exact-or-slash-child matching in the voice-call webhook router is the narrow maintainable fix, and the linked PR keeps token format, TwiML generation, and provider behavior unchanged.
