## Summary

Keep open: this is a useful Windows memory bug-fix direction, but the branch is outdated against current main. Current main has a platform-variant sqlite-vec fallback that still loads `vec0.dll` directly, while this PR only retries the bundled `sqliteVec.load()` path and is currently merge-conflicting.

## What This Changes

The PR adds a Windows sqlite-vec loader retry without the `.dll` suffix, focused loader tests, and a changelog entry.

## Best Possible Solution

Rebase onto current main and introduce a small shared loader helper so both bundled and platform-variant sqlite-vec paths get the Windows suffixless retry while explicit `extensionPath` remains untouched.

## Reproduction Assessment

Yes for source-level review: current main passes suffixed Windows sqlite-vec paths through both bundled and variant load paths, and the PR body includes Windows before/after terminal output. I did not independently run the Windows scenario.

## Solution Assessment

No, not as the branch stands. The suffixless retry is the right narrow direction, but it must be rebased and applied to current main's platform-variant fallback before merge.
