## Summary

Keep open: the Windows sqlite-vec suffix retry is useful, but this branch is stale against current main and only covers the bundled `sqliteVec.load()` path. Current main also has a platform-variant fallback that can load `sqlite-vec-windows-x64/vec0.dll` directly, so the retry needs to be shared before merge.

## What This Changes

The PR adds a Windows-only `.dll` suffixless retry for bundled sqlite-vec loading, returns the effective suffixless path, adds focused loader tests, and adds a changelog entry.

## Best Possible Solution

Use one small loader helper for auto-resolved sqlite-vec paths that leaves explicit `extensionPath` untouched, retries Windows `.dll` paths without the suffix for both bundled and platform-variant loads, persists the effective path, and keeps the existing config-hint errors.

## Reproduction Assessment

Yes for source-level review: the PR body gives Windows `openclaw memory index` and `openclaw doctor` before/after terminal output, and current main still has auto-resolved `.dll` paths in both sqlite-vec load paths. I did not independently rerun the Windows smoke in this read-only review.

## Solution Assessment

No, not as the branch stands. The suffixless retry is the right narrow direction, but it should be shared with current main's platform-variant fallback before merge.
