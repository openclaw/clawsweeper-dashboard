## Summary

Keep open. Current main still lacks the Windows suffixless `.dll` retry this PR is trying to add, so the contribution is useful, but the branch is not correct as-is because it is stale against current main's explicit `extensionPath` contract and is missing the required user-facing changelog entry.

## What This Changes

This PR adds a Windows-only sqlite-vec load retry that strips a bundled `.dll` suffix after `sqliteVec.load(db)` fails, returns the suffixless path on success, and wraps dual-failure errors in both memory-host sqlite-vec loader copies.

## Best Possible Solution

Reapply the Windows suffixless retry on top of current main in the package-owned loader while preserving the explicit `extensionPath` early-return behavior, returning the actual loaded path, adding targeted tests for both override and retry paths, and leaving broader sqlite ABI or function-registration failures to separate tracked issues.

## Reproduction Assessment

No, not for the original Windows `.dll` runtime failure in this Linux checkout. Yes for the PR regression: the existing package test demonstrates that explicit `extensionPath` must work without importing bundled `sqlite-vec`, while the PR head imports it first.

## Solution Assessment

No, this branch is not the best merge shape as-is. The right fix is the same Windows retry idea, but rebased onto current main with the explicit override branch before `loadSqliteVecModule()` and with a changelog entry.
