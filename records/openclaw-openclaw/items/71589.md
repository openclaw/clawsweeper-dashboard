## Summary

Keep this PR open: it targets a real user-visible tool-use/commentary leak, but the current patch can still emit phase-pending text before it later learns the message should be suppressed, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR expands embedded agent output suppression for unphased tool-use/commentary assistant turns, clears pending visible buffers, preserves `final_answer` streaming, and updates adjacent regression, qa-lab, status, and lockfile test surfaces.

## Best Possible Solution

Hold phase-pending assistant text until user-visible classification is known, suppress commentary/tool-use output before any partial, block, or final delivery, and preserve explicit `final_answer` streams.

## Reproduction Assessment

Yes, by source inspection: emit an assistant `text_delta` without phase/tool metadata while partial replies or `text_end` block delivery are enabled, then classify the same assistant message as commentary/tool use at `text_end` or `message_end`. I did not run live Feishu/Telegram proof in this read-only review.

## Solution Assessment

No. Expanding the suppression predicate and clearing buffers after classification helps, but it cannot retract already emitted partial or block output; the safer fix is to quarantine phase-pending chunks before visibility callbacks.
