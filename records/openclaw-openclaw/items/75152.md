## Summary

Keep open: current main and v2026.6.10 still do not normalize XML-RPC-style `{ item: { oldText, newText } }` edit entries, and https://github.com/openclaw/openclaw/pull/96700 is the open linked fix candidate for this issue.

## What This Changes

The issue reports that the edit tool rejects valid `edits` payloads when the provider/XML-RPC path wraps edit entries instead of passing direct `oldText` and `newText` fields.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/96700 so edit argument preparation and direct execution normalize single-key `{ item: ... }` entries with type-clean regression tests and redacted real provider proof, then close this issue.

## Reproduction Assessment

Yes for the source-level wrapped-entry shape: current main leaves `{ item: ... }` edit entries unchanged and validates only direct `oldText` and `newText` fields. No for the full live DeepSeek/XML-RPC environment because this review did not run that provider path.

## Solution Assessment

Yes, keeping this issue open behind the linked fix candidate is the right outcome now. The narrow normalizer in https://github.com/openclaw/openclaw/pull/96700 is the plausible fix path, but it still needs merge readiness before the issue is implemented.
