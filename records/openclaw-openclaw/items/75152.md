## Summary

Keep open: current main and v2026.6.8 now forward edit `prepareArguments` and accept direct `edits[]`, but the reported native-array/XML-RPC object-wrapped entry shape is still not normalized or covered by tests.

## What This Changes

The issue reports that the Edit tool rejects both JSON-string and native-array `edits` payloads in an OpenClaw 2026.4.25 npm install, making the tool unusable in that provider environment.

## Best Possible Solution

Normalize the verified object-wrapped `edits[]` transport shape before edit validation while preserving the canonical direct `[{ oldText, newText }]` schema and existing JSON-string/top-level compatibility.

## Reproduction Assessment

Yes for a source-level reproduction of the suspected wrapped-entry shape: current validation requires direct `oldText`/`newText` fields and the edit normalizer does not unwrap `{ item: ... }` entries. No for the full reporter environment, because I did not run a live DeepSeek/XML-RPC edit call.

## Solution Assessment

No, current main is only a partial solution. Forwarding `prepareArguments` and parsing JSON-string edits is right, but the remaining native-array transport shape needs a narrow normalizer or a live repro that disproves it.
