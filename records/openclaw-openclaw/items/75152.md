## Summary

Keep open: current main still does not normalize XML-RPC-style `{ item: { oldText, newText } }` edit entries before edit validation/execution, and the linked open fix candidate is not merged yet.

## What This Changes

The issue reports that the Edit tool rejects DeepSeek/XML-RPC edit payloads when `edits` arrives as a string or wrapped array entry instead of direct `{ oldText, newText }` objects.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/96700 so one shared pre-validation edit argument normalizer handles wrapped entries in prepared and guarded edit paths, then close this issue after proof.

## Reproduction Assessment

Yes from source, but not as a live run: current main leaves `{ item: ... }` edit entries unchanged while guarded edit validation accepts only direct `oldText` and `newText` fields. The full DeepSeek/XML-RPC environment still needs live proof.

## Solution Assessment

Yes for keeping it open: the issue is not implemented yet, and the best path is to review and land or replace the linked fix candidate. The fix should normalize before guarded validation, not only inside base edit execution.
