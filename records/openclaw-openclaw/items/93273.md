## Label Justifications

- `P2`: This is a normal-priority Control UI session continuity fix with limited blast radius but user-visible session/message-state impact.
- `merge-risk: 🚨 session-state`: The PR can mis-associate active Control UI run state by overwriting a newer local `chatRunId` with a stale history snapshot.
- `merge-risk: 🚨 message-delivery`: If run state is overwritten, later deltas or finals for the newer run can be ignored or handled as a different active run.

## Summary

Keep open. The PR targets the right existing `inFlightRun` contract, but the proposed call site can overwrite a newer local run when history resolves after a send, and the external proof is still mock/unit-only.

## What This Changes

The branch adds Control UI adoption of `chat.history.inFlightRun` during `loadChatHistory` and tests buffered and empty in-flight history reload state.

## Best Possible Solution

Guard `inFlightRun` adoption so it only runs when the history load still owns stream adoption, add a stale-history/newer-run regression test, then add redacted real Control UI browser or gateway proof.

## Reproduction Assessment

Yes. Source inspection shows `chat.history` can return `inFlightRun` and the TUI consumes it, while current Control UI history loading ignores it; I did not run a live browser reproduction.

## Solution Assessment

No as submitted. Reusing the existing `chat.history.inFlightRun` contract is the right layer, but the implementation must preserve newer local runs during in-flight history reloads and still needs real behavior proof.
