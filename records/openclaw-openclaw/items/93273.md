## Summary

Keep open: this PR targets the right Control UI gap, but the new adoption helper stores a performance-clock value in `chatStreamStartedAt`, while the renderer treats that field as a wall-clock message timestamp. The external proof is also mock-only, so this needs a small code fix plus real Control UI/Gateway behavior proof before merge.

## What This Changes

The branch adds Control UI adoption of `chat.history.inFlightRun` in `loadChatHistory` and regression tests for buffered and empty in-flight runs.

## Best Possible Solution

Keep the Control UI `inFlightRun` adoption, but pass a wall-clock stream timestamp, add focused coverage for that timestamp contract, and verify switch-back in a real Control UI/Gateway run.

## Reproduction Assessment

Yes. Source-level evidence shows current main has a gateway `inFlightRun` response and TUI consumer while Control UI history reload does not consume it; the PR-introduced timestamp bug is also source-reproducible from `controlUiNowMs()` versus the renderer timestamp contract.

## Solution Assessment

No, not as submitted. The owner boundary and general fix shape are right, but the helper should use the existing wall-clock `startedAt` value rather than `startedAtMs`, and the PR still needs real browser/Gateway proof.
