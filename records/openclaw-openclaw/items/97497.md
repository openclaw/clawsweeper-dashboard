## Summary

Keep open: this PR targets a real Claude CLI auth-provider bug, but it is not merge-ready because it recognizes `apiKeyHelper` only in the Anthropic synthetic-auth seam while the shared Claude CLI credential resolver and its other callers still treat the same host as unauthenticated.

## What This Changes

The PR adds an Anthropic-local `apiKeyHelper` settings probe and synthetic auth marker so `claude-cli` can pass the pre-spawn auth gate without stored Claude CLI credentials.

## Best Possible Solution

Represent `apiKeyHelper` as a canonical Claude CLI credential in `src/agents/cli-credentials.ts`, consume that shape from Anthropic synthetic auth, and cover setup/status/auth-epoch behavior plus redacted real OpenClaw helper-auth proof.

## Reproduction Assessment

Yes, source-level. Current main returns `undefined` synthetic auth when the shared Claude CLI credential reader finds no Keychain or `.credentials.json` credential, so the auth gate can throw `missing-provider-auth`; I did not run a live proxy-backed OpenClaw turn.

## Solution Assessment

No. The direction is useful, but the best fix is to add an `api_key_helper` credential shape to the shared Claude CLI credential reader instead of duplicating settings parsing inside the Anthropic synthetic-auth seam.
