## Summary

Keep open: the reported Claude CLI auth-gate bug is valid, but this PR is not the best fix yet because it teaches only the Anthropic synthetic-auth gate about `apiKeyHelper` instead of the shared Claude CLI credential resolver, and the PR body only provides unit/mock proof rather than a real OpenClaw CLI turn.

## What This Changes

The PR adds an Anthropic-local `~/.claude/settings.json` `apiKeyHelper` check and returns a non-secret synthetic auth marker for `claude-cli`, with mocked Anthropic provider tests.

## Best Possible Solution

Represent `apiKeyHelper` as a canonical Claude CLI credential in `readClaudeCliCredentialsCached`, then have Anthropic synthetic auth, setup/doctor/status, and CLI auth epoch consume that one shape with focused tests and redacted live proof when feasible.

## Reproduction Assessment

Yes, source-level. Current main returns `undefined` synthetic auth when the shared Claude CLI credential reader finds no keychain or `.credentials.json` credential, so the auth gate can throw `missing-provider-auth`; I did not run a live proxy-backed Claude CLI turn.

## Solution Assessment

No. The direction is right, but the best fix is to add an `apiKeyHelper` credential shape to the canonical Claude CLI credential reader rather than duplicating settings parsing inside the Anthropic synthetic-auth seam.
