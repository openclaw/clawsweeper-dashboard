## Summary

Keep open: the report is recent, concrete, and current main only shows scoped cleanup for known bundle-MCP runtimes, not cleanup for orphaned downstream `claude mcp serve` processes or a generic way to reuse a local Hermes daemon across logical scopes.

## What This Changes

The issue asks OpenClaw to reap orphaned Claude/Hermes MCP subprocesses after session end and avoid spawning a full Hermes MCP child fleet for every Hermes-as-tool invocation.

## Best Possible Solution

Fix owned process cleanup generically, including conservative orphan handling for OpenClaw-owned workspaces, and design Hermes reuse through a generic opt-in MCP daemon or HTTP reuse contract rather than hard-coded Hermes process probing in core.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the relevant cleanup paths are scoped to known runtime handles and no startup orphan sweep exists, but the reported WSL2/Hermes process tree was not recreated.

## Solution Assessment

Unclear as a single fix. Reaping owned or orphaned subprocesses is a plausible lifecycle bug fix, while cross-scope Hermes daemon sharing needs a maintainer design decision around generic MCP reuse and isolation.
