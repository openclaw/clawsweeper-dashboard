Keep open: the report is recent and concrete, and current main still has a plausible OpenClaw process-lifecycle gap around downstream MCP grandchildren even though the mirror discussion narrows part of the original fleet-spawn claim to external/configured behavior.

## What This Changes

The issue asks OpenClaw to reap orphaned `claude mcp serve` children after session end and avoid spawning separate Hermes MCP fleets when a reusable local Hermes daemon is available.

## Best Possible Solution

Fix OpenClaw-owned process cleanup generically where the direct CLI process can leave MCP descendants, and handle Hermes reuse through documented HTTP MCP configuration or a generic opt-in daemon reuse contract.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a plausible current-main cleanup boundary for downstream MCP grandchildren and no generic orphan sweep, while the reporter's mirror comments narrow but do not fully eliminate the OpenClaw-side concern.

## Solution Assessment

Unclear as one combined fix. A narrow process-lifecycle repair may be appropriate, but hard-coded Hermes daemon probing is not the best core direction while existing generic HTTP MCP transport support and cross-scope isolation concerns exist.
