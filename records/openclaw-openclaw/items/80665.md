## Summary

Keep open: the OpenClaw-owned duplicate stdio MCP child part overlaps the open canonical leak thread, but this report adds a Hermes/Claude mutual-tooling case and a proposed cross-daemon reuse behavior that is not fully owned by the existing issue.

## What This Changes

The issue reports orphaned Claude MCP server processes after OpenClaw/Hermes tool sessions end and repeated Hermes MCP child-fleet startup when Hermes is invoked as a tool.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/75621 for the generic OpenClaw-owned bundled stdio MCP runtime sharing fix, and separately decide whether Hermes should be consumed through an explicit HTTP/daemon transport or a coordinated Hermes-side lifecycle contract.

## Reproduction Assessment

Partially. ## Solution Assessment

Unclear as written. Fixing OpenClaw-owned duplicated stdio children should follow the canonical runtime-sharing work, while daemon reuse and orphan sweeps need a narrower ownership-safe design with Hermes rather than ad hoc process probing.
