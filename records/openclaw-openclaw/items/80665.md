## Summary

Keep open: current main still shows cleanup for known MCP runtime handles and direct stdio transports, but not a proven OpenClaw-side answer for detached downstream MCP grandchildren or a generic cross-scope Hermes daemon reuse contract. The linked Hermes discussion usefully narrows part of the report, but it does not make this OpenClaw issue safely closeable yet.

## What This Changes

The issue asks OpenClaw to prevent orphaned downstream `claude mcp serve` processes after OpenClaw/Hermes MCP sessions end and to avoid separate Hermes MCP child fleets across logical scopes.

## Best Possible Solution

Split the work: keep generic OpenClaw-owned stdio child/grandchild cleanup honest, use existing remote MCP URL configuration for daemon reuse where possible, and only add a broader daemon-sharing contract after maintainer design review.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection plus the mirror issue comments show a plausible lifecycle gap for detached downstream MCP grandchildren, but the reported WSL2/Hermes tree was not recreated against current main.

## Solution Assessment

Unclear as a single fix. Reaping owned subprocess trees is a plausible bug fix, while automatic Hermes daemon reuse should be handled through existing remote MCP config or a generic opt-in contract rather than hard-coded process probing.
