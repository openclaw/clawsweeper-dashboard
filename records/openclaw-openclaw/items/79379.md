## Summary

Keep open. Current main still has the duplicate silent-final fallback behavior this PR targets, and the branch is a focused fix, but it should not merge as-is because the new suppression drops final `NO_REPLY` payloads that carry media or rich content after a prior tool/block reply.

## What This Changes

The PR adds dispatcher state to skip rewriting an exact final `NO_REPLY` after a queued tool/block reply, adds a regression test, and records the Telegram fallback change in the changelog.

## Best Possible Solution

Repair the dispatcher guard so only text-only silent finals are suppressed after prior delivery, while preserving media/rich payloads, then land one converged fix with focused coverage.

## Reproduction Assessment

Yes. Source inspection shows current main rewrites an exact final `NO_REPLY` in direct sessions without considering a prior queued tool/block reply, and the PR adds a focused test for that sequence; I did not execute it in this read-only pass.

## Solution Assessment

No. The dispatcher-level location is plausible, but the current `return null` suppresses the whole final payload whenever the text is exact `NO_REPLY`; it should first prove there is no media or rich reply content to deliver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve final media when skipping silent text:** `src/auto-reply/reply/reply-dispatcher.ts:149`

Gate this suppression on there being no other sendable payload content, or normalize the silent text instead of dropping the payload.
