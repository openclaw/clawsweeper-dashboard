## Summary

Current main already solves the update false-positive this PR targets, using a narrower OpenClaw-release comparator in front of the shared generic SemVer comparator, with regression coverage and release-note provenance.

## What This Changes

The PR changes update-version comparison and tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Keep the mainline OpenClaw-specific comparator fix and close this PR as superseded by the shipped implementation.

## Reproduction Assessment

Yes. The old v2026.5.3 update comparator delegated directly to generic SemVer comparison, while current main and v2026.5.4-beta.1 show the added OpenClaw-release comparison and regression tests for correction releases.

## Solution Assessment

Yes. The current main fix is the narrower maintainable solution because it applies OpenClaw correction-release ordering only before the generic SemVer fallback instead of globally reclassifying numeric prerelease suffixes.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.88

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores arbitrary numeric suffixes with `Number.parseInt()`. Registry-provided versions above `Number.MAX_SAFE_INTEGER` can collapse adjacent revisions to the same JS number and produce the wrong update ordering; keep the suffix as a normalized string, use `BigInt`, or enforce a documented safe bound before comparing.
  - confidence: 0.78
