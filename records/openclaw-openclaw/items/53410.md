## Summary

Keep this PR open. The underlying update-comparison bug still exists on current main, but the patch needs adaptation after the shared comparator refactor and should avoid lossy numeric suffix parsing before merge.

## What This Changes

The PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and prereleases.

## Best Possible Solution

Apply correction-release ordering only to OpenClaw package update/status/startup/update-runner/update-command comparisons, keep generic plugin SemVer behavior unchanged, and compare numeric suffixes precision-safely with focused regressions and a changelog entry.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence source-level reproduction: the update comparator treats numeric suffixes as prereleases, so `2026.3.23-1` compares older than `2026.3.23` and can drive a false update-available path.

## Solution Assessment

Unclear as-is. The requested behavior is the right direction, but the PR should be adapted to current main and should avoid precision loss without changing generic ClawHub/plugin SemVer semantics by accident.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.78

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. Registry-provided versions above `Number.MAX_SAFE_INTEGER` can collapse adjacent revisions to the same JS number and produce the wrong update ordering; keep the suffix as a normalized string, use `BigInt`, or enforce a documented safe bound before comparing.
