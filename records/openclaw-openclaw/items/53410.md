## Summary

Keep this PR open. Current main still has the update/status comparator on the generic SemVer path, so numeric correction suffixes are still treated as prereleases there; the PR is useful but needs adaptation to the refactored comparator layout and precision-safe numeric suffix comparison before merge.

## What This Changes

The PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Adapt the fix on current main so OpenClaw package update/status/startup/update-runner/update-command comparisons use correction-release ordering for `YYYY.M.D-N`, while generic plugin/ClawHub SemVer checks keep standard prerelease semantics unless maintainers explicitly opt them into a mode. The numeric suffix comparison should avoid precision loss and include focused regression coverage for correction-vs-stable, correction-vs-beta, revision ordering, and affected update callers.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence source-level reproduction: the update comparator delegates to generic SemVer parsing, and the existing regression still asserts `1.0.0-1` is older than `1.0.0-beta.1`, which explains the false update-available path for `2026.3.23-1` versus `2026.3.23`.

## Solution Assessment

Unclear as-is. The requested behavior is the right direction for OpenClaw package versions, but the current PR should be adapted to the refactored comparator layout and should compare numeric suffixes precision-safely without changing generic ClawHub/plugin SemVer semantics by accident.
