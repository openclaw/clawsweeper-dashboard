## Summary

Keep open: current main still invalidates a saved `claude-cli` session from a one-shot transcript-content check, and the reported Telegram DM race is distinct from the earlier phantom-binding fix.

## What This Changes

The issue reports Telegram DM context loss when `claude-cli` transcript validation runs before Claude Code has flushed the previous turn transcript.

## Best Possible Solution

Keep the stale-transcript guard, but add a bounded freshness-aware retry before clearing `claude-cli` resume state and cover delayed transcript visibility with regression tests.

## Reproduction Assessment

Yes, source-reproducible: the current helper does a single transcript scan and both caller paths invalidate or clear a saved Claude CLI binding when that immediate check returns false. I did not run a live Telegram/Claude CLI scenario in this read-only review.

## Solution Assessment

Yes: a narrow retry/backoff or freshness guard around the Claude CLI transcript probe is the best direction, with the existing phantom-binding invalidation still applying once the bounded wait expires.
