## Summary

Keep open: the branch is aimed at the right Workboard lifecycle problem, but it can now suppress task-backed lifecycle advancement when a task payload lacks a usable `updatedAt`, and the supplied proof is still a mocked controller harness rather than real UI/Gateway/browser behavior.

## What This Changes

The branch adds a timestamp-aware Workboard lifecycle status guard, selected-option binding in the Workboard edit modal, and controller/view regressions for stale and fresh session/task sync.

## Best Possible Solution

Represent unknown task/session freshness as no staleness evidence, keep fresh lifecycle following as the default, add missing/invalid timestamp coverage, and attach redacted real Workboard UI/Gateway proof before merge.

## Reproduction Assessment

Yes for the patch blocker from source inspection: a card with `updatedAt > 0` and a completed/failed task lacking a usable `updatedAt` will pass `sourceUpdatedAt: 0` into the new guard and skip the status patch. I did not run tests because this review was read-only.

## Solution Assessment

No, not yet. The timestamp-aware guard is a reasonable fix shape, but it needs to treat unknown freshness as non-stale and prove the visible Workboard path in a real UI/Gateway/browser setup.

- **[P1] Preserve task sync when freshness is unknown:** `ui/src/ui/controllers/workboard.ts:1331-1332`
