## Summary

Keep open: the Workboard fix is close, but the PR can suppress task-driven lifecycle status sync when `task.updatedAt` is missing or unparseable, and the supplied proof is still a direct controller harness rather than real Control UI/gateway/browser proof.

## What This Changes

The PR updates Workboard Control UI select rendering and adds a timestamp-aware lifecycle guard with controller/view regression tests so stale linked session/task updates do not overwrite newer card status changes.

## Best Possible Solution

Keep the timestamp-based stale protection, but leave unknown task freshness out of the staleness guard, add regression coverage for missing/unparseable `task.updatedAt`, and provide redacted real Control UI/gateway proof before merge.

## Reproduction Assessment

Yes, at source level: on current main, a linked card moved to `running` can be moved again by `syncWorkboardLifecycle` because `shouldSyncCardStatus` ignores lifecycle freshness. A live browser reproduction was not run in this read-only review.

## Solution Assessment

No, not yet. The timestamp guard is the right owner-boundary direction, but it must not treat unknown task timestamps as stale, and the PR still needs real Control UI/gateway proof.
