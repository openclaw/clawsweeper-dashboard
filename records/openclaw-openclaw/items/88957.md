## Summary

Keep open: the patch is a bounded Workboard bug fix and the controller-layer approach matches the current ownership, but this external PR still lacks real Control UI/gateway proof and changes lifecycle timestamp semantics that maintainers should accept before merge.

## What This Changes

The PR updates Workboard Control UI controller/view behavior so edit-modal selections reflect saved card state and linked lifecycle sync skips status writes when session/task timestamps are older than the persisted card update while preserving fresh lifecycle updates.

## Best Possible Solution

Merge only after redacted real Control UI/gateway proof shows edit reopen and drag-to-running persistence while fresh linked session/task lifecycle updates still move active cards.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the controller path: current main writes lifecycle status without a timestamp freshness check, and the linked issue plus PR tests describe edit/drag cases where stale lifecycle state rewrites newer card state; I did not run a live browser repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Likely yes: the controller owns both draft state and lifecycle writeback, and the plugin/gateway already provide persisted updatedAt values, making a timestamp guard cleaner than host-local memory; merge readiness still depends on real setup proof.
