## Summary

Keep open: the PR is aimed at the right Workboard lifecycle race, but it still treats unknown task freshness as stale and can suppress valid task-driven status updates; the external-contributor proof is controller-level terminal output with mocked requests, not real Control UI/Gateway behavior.

## What This Changes

The PR updates Workboard edit-modal select rendering and adds timestamp-aware lifecycle synchronization so stale linked session/task updates do not overwrite newer card edits or moves.

## Best Possible Solution

Apply the stale-status guard only when the lifecycle source has a parseable real timestamp, preserve prior task sync for unknown freshness, add focused regression coverage, and attach redacted real Control UI/Gateway proof.

## Reproduction Assessment

Yes for the PR-introduced regression from source inspection: TaskSummary.updatedAt is optional, the PR converts unknown freshness to 0, and the new stale guard skips lifecycle updates for persisted cards with updatedAt greater than 0; no live browser/Gateway repro was run.

## Solution Assessment

No. Timestamp-aware lifecycle guarding is the right shape for stale linked-session overwrites, but unknown task freshness must preserve prior task lifecycle sync instead of being treated as stale.
