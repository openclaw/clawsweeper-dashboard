## Summary

Keep open: the PR addresses a real Workboard persistence bug, but the current timestamp guard is too broad and can suppress legitimate linked lifecycle status updates after non-status card edits; proof is still harness-only rather than a real Control UI/gateway run.

## What This Changes

The PR updates the Control UI Workboard controller/view so linked card status changes and edit-modal select values persist across lifecycle sync and modal reopen, with focused controller and view tests.

## Best Possible Solution

Preserve only explicit operator status overrides while allowing non-status edits and same-status moves to keep following linked lifecycle sync, then add real Control UI proof for edit persistence and linked-card drag/status behavior.

## Reproduction Assessment

Yes, source-reproducible: current main’s syncWorkboardLifecycle derives a target status from linked session/task state and writes it whenever shouldSyncCardStatus returns true, so stale lifecycle data can rewrite an operator status change. I did not execute the UI path because this review stayed read-only.

## Solution Assessment

No, not as written: the timestamp guard is a plausible direction, but using card.updatedAt for every update means non-status edits can suppress legitimate lifecycle status sync. The better fix needs an explicit status-override signal or equivalent narrower check.
