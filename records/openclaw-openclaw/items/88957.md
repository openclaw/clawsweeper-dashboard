## Best Possible Solution

Apply the stale guard only for parseable real source timestamps, preserve current task lifecycle sync for unknown freshness, cover it with regression tests, and add redacted Control UI/Gateway proof.

## Reproduction Assessment

Yes for the PR-introduced regression by constructing a task-backed card with updatedAt > 0 and a completed or failed task missing updatedAt; source shows the new guard skips the status update current main would make. The original browser report still lacks live proof in this review.

## Solution Assessment

No. Timestamp-aware lifecycle guarding is the right shape for stale linked-session overwrites, but unknown task freshness must preserve prior task lifecycle sync instead of being treated as stale.
