## Summary

Keep open: current main still suppresses Slack `isReasoning` replies while Slack progress-draft reasoning only receives `onReasoningStream`, and core only enables that callback for reasoning mode `stream`; the mode-`on` Slack visibility regression remains source-reproducible.

## What This Changes

The issue requests restoring a transient Slack pre-final signal for reasoning mode `on` after Slack stopped delivering separate persisted reasoning messages.

## Best Possible Solution

Preserve durable Slack reasoning suppression, add or document a transient Slack progress-draft signal for mode-`on` reasoning, and prove it live without restoring persisted reasoning messages by default.

## Reproduction Assessment

Yes, at source level: mode `on` emits `isReasoning` block replies, Slack suppresses those payloads, and Slack progress reasoning only receives `onReasoningStream` from mode `stream`. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

Unclear as a final product choice: a transient progress-draft bridge is the safer likely fix because it restores a pre-final signal while preserving the leak suppression. Maintainers still need to decide whether Slack should be a documented exception or get a Slack-specific mode-`on` bridge.
