## Summary

Keep open: current `main` still suppresses Slack `isReasoning` block replies while only stream-mode reasoning reaches the transient Slack progress path, so the reported `/reasoning on` visibility gap remains source-reproducible and is not fully owned by the newer Slack-safe progress-visibility issue.

## What This Changes

Restore or document a Slack-safe pre-final signal for tool-less Slack turns when reasoning visibility is `on` after the Slack reasoning-message suppression fix.

## Best Possible Solution

Keep durable reasoning suppression in Slack, bridge mode-`on` reasoning into a transient Slack progress snapshot or document Slack as an exception, and coordinate the disclosure policy with the broader Slack-safe progress visibility work.

## Reproduction Assessment

Yes at source level: `/reasoning on` emits `isReasoning` block replies, Slack suppresses those payloads, and Slack progress reasoning only receives `onReasoningStream` callbacks from `reasoningMode === "stream"`. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer policy: the transient progress-draft bridge is safer than a suppression opt-out because it preserves the leak fix, but the final Slack reasoning visibility contract needs product/security judgment.
