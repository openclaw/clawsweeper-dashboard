## Summary

Keep open: current main still suppresses Slack `isReasoning` payloads while Slack progress-draft reasoning is fed only by `onReasoningStream`, and core only enables that callback for reasoning mode `stream`. The reported mode-`on` Slack visibility regression remains source-reproducible and needs a maintainer decision on the safe UX contract.

## What This Changes

The issue asks to restore a Slack pre-final reasoning signal for reasoning mode `on` on tool-less turns after Slack reasoning suppression, or add an explicit Slack suppression opt-out.

## Best Possible Solution

Preserve durable Slack reasoning suppression, add or document a transient Slack progress-draft signal for mode-`on` reasoning, and prove it live without restoring persisted reasoning messages by default.

## Reproduction Assessment

Yes at source level: mode `on` emits `isReasoning` block replies, Slack suppresses those payloads, and Slack progress reasoning only receives `onReasoningStream` from mode `stream`. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

Unclear as a final product choice: the transient progress-draft bridge looks safer than a suppression opt-out because it restores a pre-final signal while preserving the leak fix. Maintainers still need to decide whether Slack should be an explicit docs exception or get a Slack-specific bridge.
