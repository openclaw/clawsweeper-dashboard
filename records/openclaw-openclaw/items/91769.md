## Summary

Keep open: current main still has the reported behavior split where Slack suppresses `/reasoning on` block-reply payloads while its progress-draft reasoning path only receives stream-mode callbacks, so the central Slack visibility gap remains real from source inspection.

## What This Changes

The issue requests restoring Slack pre-final visibility for `reasoningDefault: "on"` tool-less turns after Slack reasoning-message suppression, either through a transient progress draft or an explicit suppression opt-out.

## Best Possible Solution

Keep durable Slack reasoning suppression, then either bridge `/reasoning on` reasoning into a transient Slack progress-draft snapshot or document Slack as an explicit exception; avoid re-enabling permanent reasoning messages by default.

## Reproduction Assessment

Yes at source level: `/reasoning on` emits `isReasoning` block replies and Slack currently suppresses them, while the progress-preview callback only fires for stream-mode reasoning. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

The transient progress-draft bridge looks like the safer direction than a suppression opt-out, because it preserves the leak fix while restoring a pre-final signal. The exact Slack reasoning visibility contract still needs maintainer judgment.
