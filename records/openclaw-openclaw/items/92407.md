## Summary

The report is a concrete current-main message-delivery bug: `image` is recognized during media-source handling but omitted from the generic send payload, so text can be delivered with `ok: true` while the attachment is lost. Three open closing PRs already own the repair, so this issue should remain open until maintainers select and land one canonical fix.

## What This Changes

Honor the message tool’s top-level `image` argument as send media, or reject it instead of silently returning success after text-only delivery.

## Best Possible Solution

Land one narrow shared-outbound fix that treats top-level `image` as the lowest-precedence send media alias while preserving non-send image semantics, with focused regression coverage and real Telegram after-fix proof.

## Reproduction Assessment

Yes, at source level: current main recognizes top-level `image` as media-related input but omits it from send-payload media selection, matching the reported text-only success. This review did not independently execute a live Telegram send.

## Solution Assessment

Yes for a narrow top-level send alias repair. Extending structured attachments or the public schema is not required to fix the reported call and should only be included if maintainers intentionally adopt that broader contract.
