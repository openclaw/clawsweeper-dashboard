## Label Justifications

- `P1`: A real Telegram workflow repeatedly lost requested photos because the tool falsely reported successful delivery.
- `impact:message-loss`: The requested attachment is silently suppressed while only the text portion reaches the recipient.

## Summary

Current main and v2026.6.5 retain a split contract: `image` is recognized only for Discord event-cover sandbox normalization, but a send call can carry it through without producing outbound media, matching the reported successful text-only Telegram delivery. The issue is valid, narrow, and not owned by another canonical item.

## What This Changes

Prevent message-tool sends using an `image` argument from silently returning success after delivering only the accompanying text.

## Best Possible Solution

Keep `media` as the canonical send parameter and add an early actionable validation error when the event-specific `image` field is supplied to `send`, so unsupported attachment intent can never degrade to successful text-only delivery.

## Reproduction Assessment

Yes, from source at high confidence. Current main normalizes `image` but excludes it from send payload media extraction, and the reporter provides a concrete Telegram path showing `ok: true` with text-only delivery; this read-only review did not execute a live Telegram send.

## Solution Assessment

Yes, with a narrower choice than adding an alias: reject `image` for `send` before transport and direct the model to canonical `media`. That preserves its intentional Discord event-cover meaning without creating another undocumented send parameter.
