## Label Justifications

- `P1`: A real channel workflow silently loses requested media while the tool reports successful delivery, preventing automatic recovery.

## Summary

The report remains necessary: current main has a deterministic media-parameter mismatch that can return success after silently omitting the requested image, and three linked closing PRs are still open. The issue should remain the canonical tracker until maintainers select, verify, and merge one narrow repair.

## What This Changes

Fix the message tool so a top-level `image` parameter on `send` is delivered as media or rejected instead of silently producing a successful text-only delivery.

## Best Possible Solution

Merge a send-scoped, lowest-precedence `image` media fallback with focused regression coverage and redacted live Telegram proof, while preserving non-send event-cover semantics and keeping structured attachments canonical on `media`.

## Reproduction Assessment

Yes at high confidence from source inspection: `image` participates in shared media-source handling but is absent from generic send payload extraction, deterministically leaving a text-only payload when no canonical media field is supplied. The reporter also provides a concrete live Telegram reproduction, although this review did not independently execute it.

## Solution Assessment

Yes for the narrow send-only fallback proposed by https://github.com/openclaw/openclaw/pull/92416, subject to live Telegram proof. Expanding structured attachment schema aliases is not required to solve the reported call and would create avoidable parallel API surface.
