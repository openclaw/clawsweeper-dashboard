## Label Justifications

- `P1`: A false-success tool result silently drops requested media and caused repeated broken deliveries in a real Telegram workflow.

## Summary

The report matches a concrete current-main bug: `image` is recognized during media-source validation but omitted when the send payload chooses its attachment, allowing text-only delivery to return success. Three open pull requests already target the issue, including one with Telegram Desktop proof, so the issue should remain open until one canonical fix lands.

## What This Changes

Make the message tool either send the attachment supplied through the top-level `image` parameter or reject that parameter instead of silently delivering text only.

## Best Possible Solution

Choose one canonical pull request that makes the already-recognized top-level `image` input produce outbound media while preserving non-send event-cover semantics; add structured attachment or schema aliases only if maintainers explicitly want those additional public inputs.

## Reproduction Assessment

Yes from source inspection: current main accepts `image` during media-source handling but omits it from send-payload media selection, matching the reported text-only success. This review did not independently execute a live Telegram send, so the status is source-reproducible rather than directly reproduced.

## Solution Assessment

Yes for a narrow top-level send alias repair. Expanding structured attachments or the public tool schema is not required to fix the reported call and should be treated as a separate contract choice.
