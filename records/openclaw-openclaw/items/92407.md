## Label Justifications

- `P1`: A real user-facing channel workflow can repeatedly report success while silently omitting requested media, preventing automatic recovery.
- `impact:message-loss`: The requested image attachment is silently suppressed while the text portion is delivered.

## Summary

Current main and v2026.6.5 both contain a deterministic mismatch: shared outbound handling recognizes `image`, but generic send builds delivery media without it, allowing a successful text-only result. The issue is actionable and no canonical duplicate or open fixing PR was found.

## What This Changes

Prevent the message tool from returning success after silently dropping an attachment supplied through the `image` argument.

## Best Possible Solution

At the shared action-aware boundary, normalize `image` to canonical `media` only for generic send or reject it with a clear retryable validation error, while preserving explicitly declared action-specific `image` semantics and adding focused regression coverage.

## Reproduction Assessment

Yes at source level. Current main deterministically accepts and normalizes `image` as a media-source hint, then omits it from the generic send media payload; a live Telegram run was not performed during this read-only review.

## Solution Assessment

Yes, if implemented at the shared action-aware boundary. A Telegram-specific branch or a global reinterpretation of action-owned `image` fields would be less maintainable than canonicalizing or rejecting the field specifically for send.
