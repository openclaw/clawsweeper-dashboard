## Summary

Keep open: current main still drops RPC wizard progress and still has OAuth user-action URL paths that are only logged, while the QR/media portion is partly addressed by the existing `web.login.start`/`web.login.wait` QR RPC and should be narrowed rather than duplicated.

## What This Changes

Request protocol-visible onboarding parity for non-terminal clients by adding RPC progress delivery, QR/media or channel-login handling, a documented auth-challenge prompter capability, and guardrail tests.

## Best Possible Solution

Confirm the protocol direction, reuse the existing `web.login.*` QR flow unless a wizard media field is explicitly needed, then implement queued non-interactive progress steps and a documented auth-challenge capability with focused seam tests.

## Reproduction Assessment

Yes, by source inspection: an RPC wizard runner that calls `prompter.progress().update()` or `stop()` will emit nothing because the current session prompter returns no-op callbacks. I did not run a live Windows companion repro.

## Solution Assessment

Unclear: the issue identifies valid gaps, but the QR/media proposal needs narrowing because current main already has a dedicated `web.login.*` QR RPC. The best fix is likely progress queue semantics plus a documented auth-challenge seam while reusing or explicitly replacing the existing QR login contract.
