## Summary

Keep open: current main still drops RPC wizard progress, has no companion-facing pairing setup-code/QR gateway RPC, and still has OAuth URL paths that are only logged; the open candidate PRs at https://github.com/openclaw/openclaw/pull/94664 and https://github.com/openclaw/openclaw/pull/94300 address sub-gaps but are not merged and do not close the umbrella work.

## What This Changes

The issue requests protocol-visible wizard onboarding parity for non-terminal clients: progress frames, connection pairing QR/setup-code access, a stable auth-challenge capability, and guardrail tests.

## Best Possible Solution

Keep this as the canonical umbrella until maintainers approve the protocol shape, then land or replace the focused progress/OAuth PRs and add a scoped pairing setup-code RPC that reuses existing pairing helpers without duplicating `web.login.*`.

## Reproduction Assessment

Yes, by source inspection: an RPC wizard runner that calls `prompter.progress().update()` or `stop()` emits nothing, shared OAuth can still log the action URL outside the prompt, and no pairing setup-code RPC is registered on current main. I did not run a live companion app repro in this read-only review.

## Solution Assessment

Unclear; the progress and in-band auth directions are valid, but the pairing setup-code method and non-interactive progress polling contract are new client-visible surfaces. The safer path is maintainer-approved API shape first, while preserving the existing `web.login.*` channel QR contract.
