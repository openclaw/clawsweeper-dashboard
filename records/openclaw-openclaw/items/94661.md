## Summary

Keep open: current main and v2026.6.8 still drop RPC wizard progress and still have auth URL paths that are only logged, while QR login is already partly covered by `web.login.*` and needs protocol/product narrowing rather than cleanup closure.

## What This Changes

The issue requests non-terminal wizard parity for progress updates, QR/media login artifacts, and a documented in-band auth-challenge capability.

## Best Possible Solution

Confirm the protocol direction, reuse or explicitly replace the existing `web.login.*` QR contract, then implement queued non-interactive progress steps and a documented auth-challenge capability with seam tests.

## Reproduction Assessment

Yes, by source inspection: an RPC wizard runner that calls `prompter.progress().update()` or `stop()` emits nothing because the current session prompter returns no-op callbacks, and shared OAuth still logs the authorization URL outside the prompt. I did not run a live Windows companion repro in this read-only review.

## Solution Assessment

Unclear; the issue identifies valid gaps, but the QR/media part should be narrowed around the existing `web.login.*` RPC before adding a parallel wizard media field. The progress and auth-challenge pieces need maintainer-approved protocol semantics before implementation.
