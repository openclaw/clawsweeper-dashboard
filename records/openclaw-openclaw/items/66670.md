## Label Justifications

- `P2`: The issue affects task attribution and dashboards for spawned work, but the blast radius is limited to task ledger metadata rather than core runtime availability.
- `impact:session-state`: The persisted task ledger can mis-associate active subagent work with the wrong agent/session state.

## Summary

Keep open. Current main still resolves the child agent for native and ACP `sessions_spawn`, but the task-run creation paths do not pass that resolved agent into the persisted task record, so `task_runs.agent_id` can still be missing or parent/requester-attributed.

## What This Changes

Populate `task_runs.agent_id` for `sessions_spawn` task records with the child agent that actually executes the spawned task.

## Best Possible Solution

Pass the already-resolved child `targetAgentId` into native and ACP task-run creation, then cover persisted task attribution for both spawn paths.

## Reproduction Assessment

Yes, at source level. Native and ACP spawn compute a child-agent session key, but task creation only resolves `agentId` from explicit params or requester/owner keys, so the child agent is not persisted unless it is passed explicitly.

## Solution Assessment

Yes. Plumbing the resolved child `targetAgentId` through the existing task-run APIs is the narrowest maintainable fix; plugin-side `child_session_key` parsing should remain a fallback, not the primary contract.
