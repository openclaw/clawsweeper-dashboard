## Summary

Keep open. Current `main` still exposes `agentId` on task-run records and APIs, but native and ACP `sessions_spawn` task creation do not pass the already-resolved child agent id, so `task_runs.agent_id` can remain attributed to the requester instead of the executing child agent.

## What This Changes

The issue requests that tasks spawned through `sessions_spawn` store the executing child agent id in `task_runs.agent_id`.

## Best Possible Solution

Carry the resolved target agent into native subagent registration and ACP task creation, then cover both in task registry and SQLite persistence tests.

## Reproduction Assessment

Yes, at source level. A cross-agent spawn creates a child session key for the target agent, but task creation receives no explicit `agentId` and the registry fallback reads the requester-owned keys instead.

## Solution Assessment

Yes. Passing the already-resolved target agent at the spawn/task boundary is the narrowest maintainable fix; parsing `child_session_key` in plugin code should remain only a workaround.
