## Summary

Keep open. Current main still resolves the target child agent for `sessions_spawn`, but the native and ACP task creation paths do not pass that target as `agentId`, so `task_runs.agent_id` is derived from the requester/owner session instead of the executing child agent.

## What This Changes

Request that `sessions_spawn` persist the executing child agent in SQLite `task_runs.agent_id` so plugins and dashboards can attribute background subagent work correctly.

## Best Possible Solution

Carry the already-resolved target agent through the spawn boundary into task creation for both native and ACP runtimes, then rely on the existing `TaskRecord.agentId` to SQLite mapping rather than parsing labels or `child_session_key` in plugins.

## Reproduction Assessment

Yes. A source-level reproduction is a cross-agent spawn from `agent:main` to a worker: current code creates `agent:worker:*` child sessions but task attribution resolves from explicit `agentId`, owner, or requester only, so the row is not attributed to the worker.

## Solution Assessment

Yes. Passing explicit target-agent metadata is the narrowest maintainable fix; changing plugins to parse `child_session_key` would preserve a convention-dependent workaround instead of fixing the core task ledger contract.
