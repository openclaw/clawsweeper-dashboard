## Summary

Keep open. Current main still resolves a target child agent for native and ACP `sessions_spawn`, but the task-run creation path does not pass that target as `agentId`, so `task_runs.agent_id` can still be attributed to the requester instead of the executing child agent.

## What This Changes

The issue requests that spawned subagent task runs persist the executing child agent in `task_runs.agent_id`.

## Best Possible Solution

Carry the resolved target agent through native and ACP spawn task creation, then add regression coverage proving `TaskRecord.agentId` and SQLite `task_runs.agent_id` contain the executing child agent.

## Reproduction Assessment

Yes. A source-level reproduction is a cross-agent spawn from `agent:main:main` to `agent:worker`: the child key is built for `worker`, but task creation receives no explicit `agentId` and falls back to the requester/owner path.

## Solution Assessment

Yes. Passing explicit target-agent metadata at the spawn boundary is the narrowest maintainable fix; plugin-side parsing of `child_session_key` should remain only a workaround.
