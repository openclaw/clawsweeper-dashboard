## Summary

Keep open. Current main still resolves the target child agent for native and ACP spawns, but the task-run creation paths do not pass that target as `agentId`, so task ledger attribution can still resolve to the requester instead of the executing child agent; no duplicate or linked fix PR was found.

## What This Changes

Populate `task_runs.agent_id` for `sessions_spawn` task records with the executing child agent instead of relying on `child_session_key` parsing.

## Best Possible Solution

Pass the resolved `targetAgentId` through native subagent registration and both ACP task creation branches as `agentId`, with focused coverage proving SQLite and SDK task filters attribute spawned tasks to the child agent while requester ownership stays intact.

## Reproduction Assessment

Yes, at source level. A cross-agent spawn creates a child session key for the target agent, but task creation receives no explicit `agentId` and the registry falls back to requester-owned values.

## Solution Assessment

Yes. Passing the already-resolved target agent at the spawn/task boundary is the narrowest maintainable fix; parsing `child_session_key` should remain only a compatibility workaround for readers.
