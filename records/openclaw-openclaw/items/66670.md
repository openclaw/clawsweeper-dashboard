## Summary

Keep open. Current main still resolves the target child agent for native and ACP spawns, but the task-run creation path does not pass that target as `agentId`, so `task_runs.agent_id` can still point at the requester/parent instead of the executing child agent.

## What This Changes

Populate `task_runs.agent_id` for `sessions_spawn` task runs with the executing child agent instead of requiring plugin readers to parse `child_session_key`.

## Best Possible Solution

Carry the resolved target agent through native and ACP spawn task creation, then add regression coverage proving both `TaskRecord.agentId` and SQLite `task_runs.agent_id` contain the executing child agent.

## Reproduction Assessment

Yes, at source level. A cross-agent spawn from `agent:main:main` to `agent:worker` creates a child key for `worker`, but task creation receives no explicit `agentId` and falls back to requester ownership.

## Solution Assessment

Yes. Passing the resolved target agent at the spawn/task boundary is the narrowest maintainable fix; plugin-side parsing of `child_session_key` should remain only a workaround.
