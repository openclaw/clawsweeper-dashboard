## Summary

Keep open. Current `main` still knows the target child agent for native and ACP `sessions_spawn`, but the task-run creation path does not pass that target as `agentId`, so `task_runs.agent_id` and task-ledger agent filtering can still attribute spawned work to the requester or leave it unhelpfully unset.

## What This Changes

Populate `TaskRecord.agentId` and SQLite `task_runs.agent_id` for native and ACP `sessions_spawn` child runs with the executing child agent.

## Best Possible Solution

Carry the resolved target agent into native subagent registration and ACP task creation, then add focused registry/store coverage proving `TaskRecord.agentId` and SQLite `task_runs.agent_id` contain the executing child agent.

## Reproduction Assessment

Yes, at source level. A cross-agent spawn from a parent session to a worker session creates a child key for the worker, but task creation receives no explicit `agentId` and `resolveTaskAgentId` falls back to requester-owned values.

## Solution Assessment

Yes. Passing the already-resolved target agent at the spawn/task boundary is the narrowest maintainable fix; parsing `child_session_key` in plugins should remain only a workaround.
