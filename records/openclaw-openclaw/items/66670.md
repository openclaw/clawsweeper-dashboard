## Summary

Keep open. Current main still has the target child agent available during native and ACP `sessions_spawn`, but the task-run creation paths do not pass that target as `agentId`, so SQLite `task_runs.agent_id` continues to be populated from requester/owner context instead of the executing child agent; no duplicate or closing PR was found.

## What This Changes

Request that `sessions_spawn` write the executing child agent into `task_runs.agent_id` for spawned native subagent and ACP task runs.

## Best Possible Solution

Pass the already-resolved child `targetAgentId` into native and ACP task-run creation, preserving requester `ownerKey`/delivery semantics and adding regression coverage for SQLite persistence plus SDK/Gateway task attribution.

## Reproduction Assessment

Yes, at source level. A cross-agent spawn builds a child session key containing the target agent, but task creation receives no explicit `agentId` and the registry falls back to requester/owner attribution.

## Solution Assessment

Yes. Passing the already-resolved target agent at the spawn/task boundary is the narrowest maintainable fix; parsing `child_session_key` should remain a reader compatibility fallback, not the primary attribution contract.
