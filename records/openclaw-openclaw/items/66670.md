## Summary

Keep open. Current main still resolves the executing child agent during native and ACP `sessions_spawn`, but the task-run creation paths do not pass that target as `agentId`, so persisted `task_runs.agent_id` can still reflect requester/owner attribution instead of the child agent; no duplicate or closing PR was found.

## What This Changes

Request that `sessions_spawn` task records persist the executing child agent in `task_runs.agent_id`.

## Best Possible Solution

Pass the already-resolved child `targetAgentId` into native and ACP task-run creation, then add focused regression coverage for persisted task attribution.

## Reproduction Assessment

Yes, at source level. Native and ACP spawn build child session keys from the resolved target agent, but task creation does not pass `agentId` and the registry falls back to owner/requester attribution.

## Solution Assessment

Yes. Passing `targetAgentId` at the spawn/task boundary is the narrowest maintainable fix; parsing `child_session_key` should remain a reader fallback, not the primary attribution contract.
