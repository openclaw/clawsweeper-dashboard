## Summary

Keep this issue open: current main still records spawned sub-agent task runs without the executing child agent in the first-class task_runs.agent_id field, and the remaining work is already owned by the open closing PR at https://github.com/openclaw/openclaw/pull/88872.

## What This Changes

The issue requests that sessions_spawn persist the executing child agent in task_runs.agent_id for spawned sub-agent task rows.

## Best Possible Solution

Land the focused fix in https://github.com/openclaw/openclaw/pull/88872 after normal maintainer review and checks, preserving parent owner/requester fields while storing the executing child agent in task_runs.agent_id.

## Reproduction Assessment

Yes, at source level: current main builds childSessionKey from targetAgentId but registers the task without targetAgentId, and task creation falls back to owner/requester agent ids. I did not run a live SQLite repro because this sweep is read-only.

## Solution Assessment

Yes. The narrow maintainable solution is to pass or infer the child agent during core task creation; plugin readers parsing child_session_key should remain a workaround, not the canonical persisted attribution field.
