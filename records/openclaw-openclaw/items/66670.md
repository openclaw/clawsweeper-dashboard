## Summary

Keep open. Current main still does not write the executing target agent into `task_runs.agent_id` for cross-agent `sessions_spawn`; the task layer can persist `agentId`, but native and ACP spawn task creation still omit the resolved target agent, so registry inference falls back to the requester/owner session instead of the child agent.

## What This Changes

The issue requests that `sessions_spawn` task rows populate `task_runs.agent_id` with the agent that actually executes the spawned work.

## Best Possible Solution

Add explicit target-agent attribution at the spawn boundary: carry `targetAgentId` through native `RegisterSubagentRunParams`, pass it into native and ACP `createRunningTaskRun`, and add regression coverage for both runtimes proving cross-agent spawns persist the child agent in `TaskRecord.agentId` and SQLite `task_runs.agent_id`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a cross-agent `sessions_spawn` from `agent:main` to a worker agent: current spawn code builds a child session key for the target but creates the task row without `agentId`, so the registry derives the parent/requester instead of the executing child.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the already-resolved `targetAgentId` into task creation, not parse labels or rely on plugin-side `child_session_key` conventions; any historical repair should be handled separately.
