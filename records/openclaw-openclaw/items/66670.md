## Summary

Keep open. Current main still resolves a target child agent for native and ACP spawns, but that target is not carried into task creation, so `task_runs.agent_id` can still reflect the requester/owner instead of the executing child agent.

## What This Changes

Populate `task_runs.agent_id` for `sessions_spawn` task rows with the agent that actually executes native subagent and ACP child sessions.

## Best Possible Solution

Carry the resolved target agent through native and ACP spawn task creation, then let the existing task registry and SQLite persistence store it without plugin-side parsing.

## Reproduction Assessment

Yes. A source-level reproduction is a cross-agent spawn from `agent:main:main` to a worker: the child session key is created for the worker, while task creation receives no explicit `agentId` and falls back to requester/owner attribution.

## Solution Assessment

Yes. Passing explicit target-agent metadata at the spawn boundary is the narrowest maintainable fix; parsing `child_session_key` in plugins would keep the workaround convention-dependent.
