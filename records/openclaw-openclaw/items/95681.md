## Summary

Keep open: this is a maintainer-authored, maintainer-labeled CI workflow change, so it should stay in normal maintainer handling. I found no blocking patch defect, but the new main-push admission gate is an intentional automation tradeoff that green PR CI cannot fully prove before merge.

## What This Changes

Adds a `runner-admission` CI job that waits 90 seconds only on canonical `main` pushes, gates `preflight` and `security-fast` on it, and updates CI docs and workflow guard coverage.

## Best Possible Solution

Land the admission gate only after current CI finishes and maintainers accept the intentional main-push delay; otherwise tune the window before merge.

## Reproduction Assessment

Not applicable: this is a CI workflow change rather than a user-facing bug report. Source inspection and live PR CI prove the non-main path, while the canonical `main` debounce path is inherently post-merge behavior.

## Solution Assessment

Yes, this is an acceptable bounded fix: placing the wait before `preflight` and `security-fast` is the narrow workflow layer that can avoid Blacksmith matrix registration. A sleep inside the Blacksmith jobs would miss the stated goal, and existing max-parallel/concurrency guards already cover a different part of the pressure problem.
