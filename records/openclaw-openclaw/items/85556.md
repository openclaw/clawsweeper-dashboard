## Summary

Keep this PR open, but do not merge as-is: the patch adds the right kind of liveness signal, but starts the monitor only from spawn argv, which misses fresh Claude live sessions where the Claude session id is learned later from JSONL.

## What This Changes

The PR adds a Claude CLI sub-agent transcript activity monitor, wires it into Claude live-session startup/exit, and adds focused monitor tests.

## Best Possible Solution

Track the Claude CLI session id as live-session state and start the sub-agent monitor when either resume argv or the parsed JSONL `session_id` supplies it, with regression coverage for fresh and resumed live sessions.

## Reproduction Assessment

Yes. The PR body provides before logs for the watchdog abort, and source inspection shows current main has no sub-agent `run.progress` path while the diagnostic watchdog relies on `lastProgressAgeMs`.

## Solution Assessment

No. Polling the sub-agent transcript can be a reasonable narrow fix, but the implementation should be driven by the live session's actual parsed Claude session id rather than only by spawn argv.

Full review comments:

- **[P1] Start the monitor after fresh session ids are parsed:** `src/agents/cli-runner/claude-live-session.ts:742-751`
  - body: Fresh Claude live sessions do not keep `--session-id` in the spawned argv: `buildClaudeLiveArgs` strips it before `createClaudeLiveSession`, and the actual Claude session id is learned later from JSONL in `handleClaudeLiveLine`. Because this block starts the sub-agent monitor only from `extractCliSessionIdFromArgv(params.argv)`, first-turn/fresh live sessions still get no `cli:live:subagent` progress and can hit the same `abort_embedded_run` path while a sub-agent is active. Store the parsed session id on the live session and start the monitor when it first becomes available, with a regression test for the fresh-session path.
