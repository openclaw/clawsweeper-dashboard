## Summary

Keep open. Current main still has no `OPENCLAW_SESSION_KEY` marker, and the PR addresses a real integration need, but the patch is not merge-ready because it can preserve stale inherited markers and applies before the node-host branch despite documenting node execution as out of scope.

## What This Changes

The PR injects `defaults.sessionKey` as `OPENCLAW_SESSION_KEY` into exec child environments and adds gateway-focused tests for injection, absence, and caller override behavior.

## Best Possible Solution

Define one documented runtime marker resolver, scope it explicitly to supported hosts, distinguish explicit `params.env` overrides from inherited ambient values, and land docs/changelog plus gateway, sandbox, PTY, and node-scope regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by setting an ambient gateway or sandbox `OPENCLAW_SESSION_KEY` to a stale value and observing that the proposed merged-env guard preserves it over `defaults.sessionKey`; the node mismatch follows from current request-env versus requested-env code paths.

## Solution Assessment

No. The capability is useful, but this patch is not the best final shape because it injects before node dispatch and treats inherited env as equivalent to caller override instead of using a scoped, documented runtime-marker resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Override inherited session markers:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: The guard checks the already-merged environment, so an ambient `OPENCLAW_SESSION_KEY` from the gateway process or sandbox defaults prevents `defaults.sessionKey` from being written. That breaks the promised runtime guarantee by letting a stale session marker reach the child process unless the caller explicitly overrides it.
  - confidence: 0.9
- **[P2] Keep session markers out of node host env:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: This block runs before the `host === "node"` branch, but node execution forwards only `requestedEnv` to remote `system.run` while approval analysis reads the full `request.env`. A node command can therefore be analyzed with `OPENCLAW_SESSION_KEY` present and run remotely without it, so the injection needs to be gated to gateway/sandbox or moved after the node branch.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/agents/bash-tools.exec.ts:1605-1617`
  - body: This introduces a user-facing `feat` runtime marker but only changes code and tests. OpenClaw policy requires user-facing feature changes to include a single-line `CHANGELOG.md` entry so the new env contract ships in release notes.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency, CI, or supply-chain change, but it intentionally exposes a session identifier through subprocess environment variables and needs maintainer acceptance of that runtime contract.

Concerns:

- **[medium] Session key becomes subprocess environment data:** `src/agents/bash-tools.exec.ts:1615`
  - body: The new variable is inherited by affected child processes and descendants, where it can be printed, logged, captured in diagnostics, or exposed to tools that inspect process environments; that may be intended, but it should be explicitly documented and accepted before merge.
  - confidence: 0.88
- **[medium] Ambient env can spoof the runtime marker:** `src/agents/bash-tools.exec.ts:1615`
  - body: Because the guard preserves an inherited `OPENCLAW_SESSION_KEY`, a stale or operator-injected ambient value can be passed to commands as if it were the current session marker, undermining the proposed guarantee for downstream session-aware services.
  - confidence: 0.86
