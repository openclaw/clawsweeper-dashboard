## Summary

Keep open. Current main does not expose `OPENCLAW_SESSION_KEY`, and the PR addresses a real integration need, but the patch is not merge-ready because the env injection can preserve stale inherited markers and runs before the node-host branch despite the PR documenting node execution as out of scope.

## What This Changes

The PR injects `defaults.sessionKey` as `OPENCLAW_SESSION_KEY` into exec child environments before host dispatch and adds gateway-focused tests for default injection, absence, and caller override behavior.

## Best Possible Solution

Define one supported runtime marker contract, then land a consolidated implementation that scopes injection to intended hosts, distinguishes explicit `params.env` overrides from inherited ambient values, documents the security model, updates changelog/docs, and adds gateway, sandbox, PTY, and node-scope regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: set an ambient `OPENCLAW_SESSION_KEY` on the gateway process or sandbox defaults and the proposed merged-env guard preserves that stale value instead of `defaults.sessionKey`; for `host=node`, analysis sees the injected full env while remote execution forwards only `requestedEnv`.

## Solution Assessment

No. The proposed direction is useful, but this patch is not the best final shape because it injects before node dispatch and treats inherited env as equivalent to caller overrides; a scoped resolver shared with the broader marker contract would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Override inherited session markers:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: The guard checks the already-merged environment, so an ambient `OPENCLAW_SESSION_KEY` from the gateway process or sandbox defaults prevents `defaults.sessionKey` from being written. That breaks the promised runtime guarantee by letting a stale session marker reach the child process unless the caller explicitly overrides it.
  - confidence: 0.9
- **[P2] Keep session markers out of node host env:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: This block runs before the `host === "node"` branch, but node execution forwards only `requestedEnv` to remote `system.run` while approval analysis reads the full `request.env`. A node command can therefore be analyzed with `OPENCLAW_SESSION_KEY` present and run remotely without it, so the injection needs to be gated to gateway/sandbox or moved after the node branch.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No dependency, CI, or supply-chain change is present, but the patch intentionally exposes a session identifier through child-process environment variables and needs maintainer acceptance of that runtime contract.

Concerns:

- **[medium] Session key becomes subprocess environment data:** `src/agents/bash-tools.exec.ts:1615`
  - body: The new variable is inherited by affected child processes and descendants, where it can be echoed, logged, captured in diagnostics, or exposed to tools that inspect process environments. That may be the intended integration contract, but it should be explicitly documented and accepted before merge.
  - confidence: 0.88
