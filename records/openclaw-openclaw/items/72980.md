## Summary

Keep open. Current main still has no `OPENCLAW_SESSION_KEY` runtime marker, and this PR is useful but not merge-ready because it preserves inherited or sandbox-stale markers, injects before the node-host dispatch despite scoping node out, lacks the required user-facing changelog entry, and overlaps with open PR #40423.

## What This Changes

The PR injects `defaults.sessionKey` as `OPENCLAW_SESSION_KEY` for exec child environments and adds gateway-focused tests for injection, absence, and caller override behavior.

## Best Possible Solution

Adopt one documented runtime-marker contract with explicit gateway, sandbox, PTY, and node-host behavior, shared helper coverage where practical, docs/changelog updates, and coordination with #40423.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by setting an inherited or sandbox-default `OPENCLAW_SESSION_KEY` to a stale value while `defaults.sessionKey` has the expected value; the PR's post-merge guard preserves the stale value. The node mismatch is also statically visible from the current analysis env versus `requestedEnv` execution path.

## Solution Assessment

No. The capability is useful, but the implementation should only let explicit `params.env` override the runtime marker, keep node-host scoping explicit, document the security contract, and align with the broader marker work in #40423.
