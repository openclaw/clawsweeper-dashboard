The issue reports a webchat gateway crash when sending image attachments, with the stack pointing at base64 attachment validation during `chat.send`.

## Best Possible Solution

Replace large-payload regex validation in the gateway attachment parser with linear, non-recursive validation and add regression coverage for webchat image attachments under the existing size and media contracts.

Fix https://github.com/openclaw/openclaw/issues/82182. The reported webchat image upload crash maps to `src/gateway/chat-attachments.ts:isValidBase64`, which currently uses a whole-payload regex under `parseMessageWithAttachments`. Replace that validation with a linear/non-recursive base64 check that preserves the existing contract: reject empty content, require length divisible by 4, allow only valid base64 characters with padding only at the end, keep existing max-byte/MIME/offload behavior, and do not change attachment API shape or model capability policy. Add focused regression coverage for a large webchat image/base64 payload and for invalid trailing characters/padding without triggering `RangeError`. If the UI `dataUrlToBase64` regex proves part of the same failure path, adjust it narrowly without moving payloads back into Lit/chat state. Add a user-facing changelog fix entry.
