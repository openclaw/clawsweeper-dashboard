## Summary

Keep open: current main and latest release still do not make proactive cron/heartbeat sends available to the next provider replay, and the related implementation PR remains open, conflicting, and explicitly blocked on the context-contract decision.

## What This Changes

The issue asks OpenClaw to keep proactive outbound messages in the relevant agent session context so follow-up replies do not lose conversation continuity.

## Best Possible Solution

Settle the maintainer-approved context representation, then rework or replace https://github.com/openclaw/openclaw/pull/86491 so proactive deliveries are visible to the next relevant agent turn without replaying transcript-only delivery-mirror rows as provider-authored output.

## Reproduction Assessment

Yes, source-reproducible: current main stores delivery-mirror rows as transcript bookkeeping but filters them before provider replay, and the cron/heartbeat proactive sends currently pass no mirror metadata at all. I did not run a live Telegram, Discord, or cron scenario in this read-only pass.

## Solution Assessment

No final solution is on main yet. Transcript-only mirroring is a plausible partial repair, but the best fix needs an explicit next-turn context design rather than blindly replaying delivery-mirror assistant rows.
