## Summary

Keep open: current main and the latest release still do not make the central proactive-send context problem go away; some isolated cron paths now have transcript mirroring, but normal cron announce, heartbeat delivery, and next-provider replay still leave the reported desynchronization unresolved.

## What This Changes

The issue asks for proactive outbound messages such as heartbeat updates, scheduled reports, reminders, and cron sends to be visible in the agent session context used by follow-up replies.

## Best Possible Solution

Define the maintainer-approved proactive context representation, then land or replace the related mirroring PR so normal cron, heartbeat, and routed outbound proactive sends are visible to the next relevant agent turn without duplicating transcript-only delivery rows.

## Reproduction Assessment

Yes, source-reproducible: current main still sends normal cron announce and heartbeat deliveries without mirror metadata, and even delivery-mirror transcript rows are filtered before provider replay. I did not run a live Telegram, Discord, or WhatsApp scenario in this read-only review.

## Solution Assessment

No final solution is on main yet. The related PR is a plausible implementation candidate, but the best fix needs an explicit maintainer decision about what proactive delivery context should be replayed to the model versus kept as transcript bookkeeping.
