## Summary

Keep open: current main still does not mirror proactive cron/heartbeat outbound deliveries into the session transcript that the next user reply will replay, and the related implementation remains an open PR rather than a merged fix. The issue is source-reproducible from the current cron and heartbeat send paths, but the safest landing path needs the session-context product decision and review of the existing related PR.

## What This Changes

The issue requests that proactive outbound messages, such as heartbeat checks, scheduled reports, and reminders, be represented in session context so later user replies are not desynchronized.

## Best Possible Solution

Land a narrow session-aware mirror for proactive cron and heartbeat deliveries that records the exact delivered assistant text in the transcript the next reply will replay, while avoiding provider-owned thread/topic sessions and duplicate delivery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction shape: proactive cron and heartbeat paths deliver outward through durable/outbound messaging without appending the delivered assistant text to the next replayed session transcript. I did not run a live Telegram/Discord/WhatsApp scenario in this read-only review.

## Solution Assessment

Unclear as a product decision: mirroring the delivered text into the destination transcript is the right direction, but maintainers still need to approve the exact session ownership rule to avoid duplicate mirrors or writes into provider-owned thread/topic sessions.
