The PR replaces selected `existsSync()` plus `readFileSync()` file reads with direct reads and `ENOENT` handling in device auth, device identity, and exec approvals code.

## Best Possible Solution

Rebase against current main, preserve the private-store auth and identity paths, narrow the direct-read plus `ENOENT` handling to the remaining exec approvals reads, add focused regression coverage, and provide redacted real setup proof.

Yes. Current main still has `existsSync(filePath)` immediately before `readFileSync(filePath, "utf8")` in both exec approvals read paths, so the issue is source-reproducible without mutating the checkout.

## Solution Assessment

No as-is. Direct read plus `ENOENT` handling is the right narrow shape for the remaining exec approvals reads, but the auth and identity hunks should be dropped or rewritten to preserve current private-store behavior.
