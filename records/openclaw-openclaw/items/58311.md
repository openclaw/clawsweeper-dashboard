## Summary

Keep open: current main still has the exec approvals check/read race, but this branch is conflict-stale, lacks real behavior proof, and would regress newer auth/identity storage if carried through rebase as-is.

## What This Changes

The PR replaces selected device auth, device identity, and exec approvals check-then-read file reads with direct `readFileSync` calls plus `ENOENT` handling.

## Best Possible Solution

Preserve current private-store auth and identity handling, then land a narrow exec approvals direct-read hardening with focused tests and an explicit maintainer decision on non-ENOENT read failures.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main has `fs.existsSync(filePath)` immediately before `fs.readFileSync(filePath, "utf8")` in both exec approvals read paths. I did not run a concurrent delete/swap harness, so runtime reproduction is not established.

## Solution Assessment

Is this the best way to solve the issue? No as-is: direct read plus `ENOENT` handling is the right narrow shape for exec approvals, but the auth and identity hunks are stale and the fail-closed approvals behavior needs explicit policy, tests, and proof.
