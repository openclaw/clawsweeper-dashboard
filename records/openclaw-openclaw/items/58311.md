## Summary

Keep open: current main still has the exec-approvals check-then-read race this PR targets, but the branch is stale against current auth/identity private-store hardening and lacks real behavior proof.

## What This Changes

The PR changes device auth, device identity, and exec approvals file reads from `existsSync()`/`readFileSync()` to direct reads with `ENOENT` handling.

## Best Possible Solution

Rebase to keep fs-safe device auth/identity storage, apply direct-read plus `ENOENT` handling only to the remaining exec-approvals reads, and add focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main still checks `fs.existsSync(filePath)` immediately before `fs.readFileSync(filePath, "utf8")` in both exec approvals read paths. I did not run tests because this review had to keep the checkout read-only.

## Solution Assessment

No as-is. Direct read plus `ENOENT` handling is the right narrow shape for exec approvals, but the auth and identity hunks should be dropped or rewritten against current private-store code before merge.
