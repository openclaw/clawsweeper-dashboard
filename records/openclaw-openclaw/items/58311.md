## Summary

Keep open: current main still has the exec-approvals check/read race this PR targets, but the branch is stale against the newer private file-store auth/identity paths and lacks external real-behavior proof.

## What This Changes

The PR replaces `existsSync()`-before-`readFileSync()` reads in device auth, device identity, and exec approvals with direct reads and explicit `ENOENT` handling.

## Best Possible Solution

Land a rebased hardening patch that fixes the remaining exec-approvals race, preserves the current private file-store device paths, defines fail-closed read-error semantics, and adds focused regression coverage.

## Reproduction Assessment

Yes, as source-reproducible: current main still performs `existsSync()` immediately before `readFileSync()` in the two exec-approvals read paths. I did not run tests because this review had to keep the checkout read-only.

## Solution Assessment

No as-is. Direct read plus `ENOENT` handling is the right narrow shape for exec approvals, but the PR needs to be rebased away from stale auth/identity hunks and needs maintainer confirmation of read-error security semantics.
