## Summary

Keep this PR open for maintainer review. Current main still contains the targeted check-then-read patterns, and the proposed direction is plausible, but the patch changes security-sensitive read-error behavior and needs explicit maintainer sign-off plus a changelog entry before merge.

## What This Changes

The PR refactors exec approvals, device identity, and device auth store reads to use direct `readFileSync()` calls with explicit `ENOENT` handling instead of `existsSync()` pre-checks.

## Best Possible Solution

Land a focused hardening patch that covers the selected approvals/auth/identity reads consistently, defines the missing-file versus unreadable-file contracts, adds targeted regression coverage, and records the user-visible security hardening in the changelog.

## Reproduction Assessment

Yes. Current main has `existsSync()` immediately before `readFileSync()` in the affected paths, and a mocked filesystem can make the existence check succeed while the read sees a different result or throws `ENOENT`.

## Solution Assessment

Mostly yes. Direct read plus explicit `ENOENT` handling is the narrow fix for the check/read gap, but the mergeable solution should include maintainer-approved error semantics, identity-helper scope, tests, and changelog coverage.
