## Summary

Keep open: the central exec-approvals TOCTOU problem is still source-reproducible on current main, but this PR is conflicting against newer main, lacks after-fix real behavior proof, and its auth/identity hunks would replace newer fs-safe private-store code if carried forward.

## What This Changes

The PR replaces selected security-sensitive file reads in device auth, device identity, and exec approvals with direct `readFileSync` calls that handle `ENOENT` instead of prechecking `existsSync()`.

## Best Possible Solution

Rebase against current main, keep the fs-safe auth/identity storage, apply direct-read plus `ENOENT` handling only where it still belongs, and add focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, source-reproducible: current main still checks `fs.existsSync(filePath)` immediately before `fs.readFileSync(filePath, "utf8")` in both exec approvals read paths. I did not run tests because this review had to keep the checkout read-only.

## Solution Assessment

No as-is. Direct read plus `ENOENT` handling is the right narrow shape for the remaining exec approvals reads, but the auth and identity hunks should be dropped or rewritten to preserve current private-store behavior.
