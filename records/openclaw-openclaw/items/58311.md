## Summary

Keep this PR open. Current main still has the targeted check-then-read patterns, and the proposed direct-read/ENOENT shape is plausible, but the security-sensitive error-semantics change and missing focused coverage need maintainer review before merge.

## What This Changes

The PR replaces selected `existsSync()` plus `readFileSync()` reads with direct reads and `ENOENT` handling in exec approvals, device identity, and device auth storage.

## Best Possible Solution

Land a focused hardening patch that uses one read plus explicit missing-file handling, adds regression coverage for missing files, invalid JSON, and non-ENOENT failures, and documents or fixes the remaining read-only identity helper scope.

## Reproduction Assessment

Yes. Current main has `existsSync()` immediately before `readFileSync()` in the affected paths, and the race can be reproduced with a mocked or swapped filesystem state between the check and read; direct missing reads report `ENOENT`.

## Solution Assessment

Mostly yes. Direct read plus `ENOENT` handling is the narrow maintainable fix for the TOCTOU gap, but the final solution should include maintainer-approved error semantics and focused tests.
