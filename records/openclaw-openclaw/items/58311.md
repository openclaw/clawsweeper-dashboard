## Summary

Keep this PR open. Current main still has the targeted check-then-read file access in exec approvals, device auth storage, and device identity, and the proposed patch is a plausible narrow hardening change; because it changes security-sensitive error semantics and adds no focused tests in the supplied diff, it should stay open for maintainer review rather than cleanup closure.

## What This Changes

The PR replaces selected `existsSync()` followed by `readFileSync()` reads in exec approvals, device identity, and device auth storage with direct reads that handle `ENOENT` as the missing-file case.

## Best Possible Solution

Land a focused hardening patch that removes the check-then-read pattern from the security-sensitive stores, preserves intended missing-file behavior, explicitly validates non-ENOENT read failures, and documents or fixes the remaining read-only identity helper scope before merge.

## Reproduction Assessment

Yes. Current main has adjacent `existsSync()` and `readFileSync()` calls in the targeted functions, and the race can be exercised at source level by making the file disappear or become unreadable between the check and the read.

## Solution Assessment

Yes, with a scope caveat. Direct read plus explicit `ENOENT` handling is the narrow maintainable fix for the missing-file race, but it is not a complete file-swap defense and the non-ENOENT semantics need focused validation.
