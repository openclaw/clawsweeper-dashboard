## Summary

Keep open: current main still applies the Control UI allowed-origin seed before normalizing legacy gateway bind aliases, so a legacy `gateway.bind: "0.0.0.0"` config can be migrated to `lan` without receiving the required seeded origins.

## What This Changes

The issue asks doctor legacy config migration to seed `gateway.controlUi.allowedOrigins` for host-alias bind values that normalize to non-loopback bind modes.

## Best Possible Solution

Resolve host aliases before the non-loopback origin seed, or make that seed resolve aliases internally, while preserving existing allowed-origin and Host-header fallback no-op behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run legacy doctor migration on `gateway.bind: "0.0.0.0"` with no configured `gateway.controlUi.allowedOrigins`; the seed predicate returns before the later alias migration maps the bind to `lan`.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize or resolve bind aliases before checking whether Control UI origins need seeding, with focused regression coverage for non-loopback and loopback aliases.
