## Summary

Keep open: the PR has strong live proof for the reported stale runtime-model cases, but the changed persisted-context cap still trusts unproven legacy `contextTokens` when an active model window resolves, so one stale `/status` path remains.

The branch filters stale persisted and live context-window values in `/status` rendering and status summaries, threads the existing agent `contextTokens` cap into resolved config, and adds regression tests.

## Best Possible Solution

Gate persisted `contextTokens` behind matching model or fallback provenance before any capping, preserve explicit configured caps, and add regression coverage for a legacy session entry without runtime model metadata.

## Reproduction Assessment

Yes. Source inspection shows current main can mix persisted/live `contextTokens` with a different displayed model, and PR head still has a no-provenance legacy branch that can keep an old context window.

## Solution Assessment

No. The fix direction is right, but the persisted-context cap must use the same provenance gate before `Math.min` so unproven legacy values cannot remain status overrides.
