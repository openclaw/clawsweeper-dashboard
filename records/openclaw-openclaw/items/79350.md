## Summary

Keep open: current main has a related zero-output retry for Ollama, but the reported local-daemon `glm-5.1:cloud` stall/failover path remains source-reproducible because the idle watchdog still treats the loopback Ollama base URL as local before considering cloud model identity.

## What This Changes

The issue asks OpenClaw to fail over or surface a clear error when Ollama `glm-5.1:cloud` stalls after tool results during agent runs through a local Ollama daemon.

## Best Possible Solution

Treat Ollama `:cloud` model refs routed through a signed-in local daemon as cloud-like for idle timeout and failover, while preserving explicit timeout overrides and the disabled watchdog for truly local models.

## Reproduction Assessment

Yes, source-reproducible for the timeout-policy path: the reporter supplied concrete tool-loop logs, and current main disables the implicit idle watchdog for `127.0.0.1` without receiving the `glm-5.1:cloud` model id needed to keep cloud-model stalls on the cloud path. I did not run a live Ollama Cloud GLM session in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to thread model identity into the idle-timeout decision and classify Ollama `:cloud` refs as cloud-like before applying the local-base-url watchdog opt-out. A new global timeout knob or broad tool-loop rewrite would be a less targeted solution.
