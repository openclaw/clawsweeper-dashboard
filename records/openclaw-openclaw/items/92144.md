## What This Changes

The PR adds additive `storage` and `sqlitePath` fields to the cron status payload, keeps `storePath` as a deprecated compatibility field, updates the disabled-scheduler CLI warning, and extends focused cron tests.

## Best Possible Solution

Land the additive status fields while retaining `storePath` as the logical legacy key until maintainers choose a separate deprecation/removal plan.

## Reproduction Assessment

Yes. Source inspection shows current main returns the legacy `state.deps.storePath` from `cron.status` while cron persistence reads from the shared SQLite state database; I did not run the live CLI in this read-only sweep.

## Solution Assessment

Yes. Adding explicit `storage` and `sqlitePath` fields while preserving `storePath` is the narrow compatibility-safe fix; replacing `storePath` outright would risk shipped scripts that parse the existing field.
