## Summary

No blocking findings. Current main and v2026.6.5 still expose the legacy cron jobs path even though cron persistence is SQLite-backed, and this PR fixes that at the status/reporting boundary while preserving the legacy field for compatibility.

## What This Changes

This PR adds `storage` and `sqlitePath` to `cron.status`, prefers `sqlitePath` in the disabled-scheduler CLI warning, and updates cron/plugin contract tests.

## Best Possible Solution

Merge a narrow status/reporting fix that exposes the canonical SQLite path while keeping the legacy `storePath` field available for existing consumers.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron.status()` returns the logical legacy jobs path while the cron store reads from the shared SQLite state database.

## Solution Assessment

Yes. Adding explicit `storage` and `sqlitePath` at the status boundary while keeping `storePath` avoids changing a shipped field's semantics and matches the existing SQLite storage ownership.
