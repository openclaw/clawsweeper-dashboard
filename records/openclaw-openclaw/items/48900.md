## Summary

Keep open for maintainer review: current main still lacks native audio/video forwarding, and this PR has blocking schema, session-key, prompt-media contract, and raw-media egress policy issues before it is mergeable.

## What This Changes

The PR adds opt-in `tools.media.audio/video.nativeForwarding` handling that captures raw audio/video bytes during media understanding and injects them into the embedded runner prompt media array.

## Best Possible Solution

Define a typed native audio/video media contract with strict schema/help/docs, provider capability and fallback gates, one resolved session key, payload limits, and explicit raw-media egress/audit policy, then converge this PR with the canonical native-audio issue and alternate implementation PR.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict schema rejection path, redirected session-key mismatch, and image-only dependency boundary are traceable from the diff and current main. The requested native forwarding behavior itself is a feature, not a current-main bug reproduction.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best mergeable implementation while audio/video blocks are cast through an image-only prompt boundary and raw-media egress/provider support policy remains unsettled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the resolved runner key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: In native-command redirects, the run can target `CommandTargetSessionKey`, but this stores raw media under `ctx.SessionKey` while the runner consumes and discards under `params.sessionKey`. Redirected media will not be forwarded and up-to-25 MiB base64 blocks can remain in the process map.
  - confidence: 0.92
- **[P2] Register nativeForwarding in the strict config schema:** `src/config/types.tools.ts:105-114`
  - body: This adds a public config key only to the TypeScript type. `ToolsMediaUnderstandingSchema` is strict and still omits that key, so user config containing the new option would be rejected and the option remains undocumented in generated schema/help surfaces.
  - confidence: 0.91
- **[P2] Add a typed audio/video prompt media contract:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: The pinned prompt boundary still accepts only text and image content, while these new blocks use `type: "audio" | "video"`. Casting them through `images` can serialize them as unsupported image payloads or lose media until the dependency contract and provider adapters land together.
  - confidence: 0.9
