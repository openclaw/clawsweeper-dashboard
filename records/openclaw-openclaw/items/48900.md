## Summary

Keep open: the feature is still absent from current main and the PR is a plausible implementation candidate, but the patch is not mergeable as-is because it misses the strict config schema, stores native media under the wrong key for redirected native-command runs, and forwards audio/video through an image-only prompt contract while raw-media provider egress still needs maintainer policy review.

## What This Changes

The PR adds opt-in `tools.media.audio/video.nativeForwarding` support that captures raw attachment bytes during media understanding and injects them into embedded-runner prompt media.

## Best Possible Solution

Land a revised native media design that uses a typed audio/video prompt-media contract, provider capability and fallback rules, schema/help/docs updates, raw-media egress/audit policy, and one resolved key for capture, consume, and cleanup.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict schema omits the new config key, native-command retargeting can use a different consume key, and the pinned prompt content contract only accepts text/image content. The user-facing feature itself is not a bug reproduction; current main remains transcription-oriented for audio.

## Solution Assessment

No. The feature direction is plausible, but this patch should be revised around a typed audio/video media contract and explicit raw-media egress/capability policy instead of casting through the image path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the resolved runner key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: In native-command redirects, the run can be retargeted to `CommandTargetSessionKey`, but this stores raw media under `ctx.SessionKey` while the runner consumes and discards under `params.sessionKey`. Redirected voice/video blocks are then not forwarded and can remain in the process store.
  - confidence: 0.92
- **[P2] Register nativeForwarding in the strict schema:** `src/config/types.tools.ts:100`
  - body: Adding `nativeForwarding` to the TypeScript config type is not enough because `ToolsMediaUnderstandingSchema` is strict. User config containing the new audio/video key will be rejected until the schema and generated config/help metadata are updated.
  - confidence: 0.91
- **[P2] Add a typed audio/video prompt-media contract:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: The prompt boundary and pinned pi-ai types are still image-shaped, while these new blocks use `type: "audio" | "video"`. Until the upstream prompt contract and adapters accept typed audio/video content, this cast can pass unsupported shapes to providers or drop the media.
  - confidence: 0.88
