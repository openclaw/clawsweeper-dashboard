## Summary

Keep this PR open: current main still lacks native raw audio/video forwarding, but this patch is not mergeable as-is because source review confirms session-key cleanup, strict config schema, upstream prompt-media typing, changelog, and raw-media egress blockers.

## What This Changes

The PR adds optional `tools.media.audio/video.nativeForwarding`, captures raw attachment bytes during media understanding, stores them by session key, and merges them into embedded-runner prompt media.

## Best Possible Solution

Land a revised feature around typed prompt-media support, provider capability/fallback rules, documented raw-media egress, schema/help/docs/changelog updates, one resolved session key for capture and cleanup, and focused redirect/config tests.

## Reproduction Assessment

Yes for the PR blockers. Source inspection shows native command redirects can use `CommandTargetSessionKey` while the PR stores blocks under `ctx.SessionKey`, and the strict media config schema lacks the proposed key.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best implementation until raw media egress, config validation/docs, and the upstream prompt-media type/adapter contract are explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the resolved runner key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: In native command redirects, the agent runs under `CommandTargetSessionKey`, but this stores raw media under the source `ctx.SessionKey`. The runner consumes and discards under its resolved `params.sessionKey`, so redirected voice/video blocks are not forwarded and can remain in the process store. Thread the same resolved run/session key into media understanding and use it for push, consume, and discard.
  - confidence: 0.92
- **[P2] Register nativeForwarding in the config schema:** `src/config/types.tools.ts:100`
  - body: Adding `nativeForwarding` to the TypeScript type is not enough because `ToolsMediaUnderstandingSchema` is strict. User config containing `tools.media.audio.nativeForwarding` or `tools.media.video.nativeForwarding` will be rejected unless the zod schema, generated config metadata/help, and docs are updated with the new key.
  - confidence: 0.91
- **[P2] Keep audio/video blocks out of ImageContent casts:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: The prompt boundary is still typed as `ImageContent[]`, while these new blocks use `type: "audio" | "video"`. Until the pi-ai contract and adapters accept typed audio/video content, this cast can send unsupported prompt shapes to providers or silently lose the media instead of failing clearly.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/config/types.tools.ts:100`
  - body: This exposes a user-facing feature and config option, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-visible feature work before merge.
  - confidence: 0.88
