## Summary

Keep open: the feature is not implemented on current main, and the PR still has concrete merge blockers around config validation, session-key cleanup, typed audio/video prompt contracts, and raw-media egress policy.

## What This Changes

The PR adds an opt-in `tools.media.audio/video.nativeForwarding` path that captures raw audio/video attachments and injects them into embedded-runner prompt media alongside image blocks.

## Best Possible Solution

Land a revised native audio/video design with typed prompt-media support, provider capability and fallback semantics, schema/help/docs updates, raw-media egress policy, and one resolved key for capture, consume, and cleanup.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict schema omits the new key, native-command session retargeting can diverge from the PR storage key, and the pinned prompt-media types only accept text/image content.

## Solution Assessment

No. The feature direction is plausible, but this patch should be revised around a typed audio/video media contract and explicit raw-media egress/capability policy instead of casting through the image path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the resolved runner key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: In native command redirects, reply setup can retarget the run to `CommandTargetSessionKey`, but this stores raw media under `ctx.SessionKey` while the runner consumes and discards under `params.sessionKey`. Redirected voice/video blocks are then not forwarded and can remain in the process store.
  - confidence: 0.92
- **[P2] Register nativeForwarding in the strict schema:** `src/config/types.tools.ts:100`
  - body: Adding `nativeForwarding` to the TypeScript config type is not enough because `ToolsMediaUnderstandingSchema` is strict. User config containing the new audio/video key will be rejected until the schema and generated config/help metadata are updated.
  - confidence: 0.91
- **[P2] Add a typed audio/video prompt-media contract:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: The prompt boundary and pinned pi-agent/pi-ai types are still image-shaped, while these new blocks use `type: "audio" | "video"`. Until the upstream prompt contract and adapters accept typed audio/video content, this cast can pass unsupported shapes to providers or drop the media.
  - confidence: 0.88

## Risks / Open Questions

- Raw audio/video forwarding changes provider egress from text transcript/description to original media bytes and still needs explicit capability, fallback, payload, audit, and documentation policy.
- The pinned prompt-media contract is image-shaped, so casting audio/video blocks through the image path can fail or be silently mishandled until the upstream content contract is extended.
- Native command retargeting can leave up-to-25 MiB base64 media blocks stored under the wrong key for the lifetime of the process.
- A newer open implementation candidate at https://github.com/openclaw/openclaw/pull/78797 covers a related native audio/video direction, but it has similar contract and transport blockers rather than a merged replacement.
