## Summary

Keep open. Current main still lacks the native audio/video forwarding feature, and the PR remains relevant to open native-audio feature requests, but the proposed patch has blocking correctness and data-egress contract issues before it is mergeable.

## What This Changes

The PR adds opt-in `tools.media.audio/video.nativeForwarding`, captures base64 audio/video attachments during media understanding, stores them by session key, and merges them into embedded-runner prompt media alongside images.

## Best Possible Solution

Revise the PR around a typed prompt-media contract that supports native audio/video, then add the strict config schema/help/generated docs, provider capability and fallback rules, one resolved run/session key for capture and cleanup, and focused tests for native-command redirects and config validation.

## Reproduction Assessment

Yes for the current-main gap and PR bugs. Current main can be verified by code search and schema inspection, and the PR keying bug follows from `getReplyFromConfig` resolving native command targets into `agentSessionKey` while the PR stores blocks under the source `ctx.SessionKey`.

## Solution Assessment

No. The feature direction is plausible, but this patch is not yet the best implementation because it exposes a public config key without schema/docs support, relies on an unlanded dependency contract, and uses mismatched storage keys for native media cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use the resolved runner key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: In native command redirects, `getReplyFromConfig` runs the agent under `CommandTargetSessionKey`, but this code stores native blocks under the source `ctx.SessionKey`. The runner then consumes and discards under `params.sessionKey ?? params.sessionId`, so redirected voice/video blocks are never forwarded and can retain up to the capped media payload for the process lifetime. Thread the same resolved session/run key into media understanding and use it for push, consume, and discard.
  - confidence: 0.92
- **[P2] Add the config schema for nativeForwarding:** `src/config/types.tools.ts:100`
  - body: Adding `nativeForwarding` to the TypeScript type is not enough because `ToolsMediaUnderstandingSchema` is `.strict()`. User config containing `tools.media.audio.nativeForwarding` or `tools.media.video.nativeForwarding` will be rejected unless the zod schema, generated config metadata/help, and docs are updated with the new key.
  - confidence: 0.9
- **[P2] Do not cast native media blocks through ImageContent:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: The current prompt boundary is still typed and handled as `ImageContent[]`, while these new blocks use `type: "audio" | "video"`. Until the pi-ai/pi-coding-agent contract and adapters accept typed audio/video content, this cast can send unsupported prompt shapes to providers or silently lose the media instead of failing clearly.
  - confidence: 0.83

## Risks / Open Questions

- The PR sends raw audio/video bytes to the selected model provider when enabled, so the data-egress and provider-capability contract needs explicit schema/docs/runtime behavior before merge.
- The proposed module-level store can leak or drop native media blocks in native-command redirect flows because the capture and consume keys can diverge.
- The patch depends on a companion `@mariozechner/pi-ai`/adapter media contract that is not present in current main, so casting audio/video blocks through `ImageContent[]` is unsafe.
- The new public config key is added only to TypeScript types, not to the strict zod schema, field metadata, generated schema/help, or docs.
