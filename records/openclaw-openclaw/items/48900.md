## Summary

This PR should stay open because current main still lacks native audio/video forwarding, but the branch needs revision before merge for schema coverage, session-key lifecycle, typed prompt-media support, and raw media egress policy.

## What This Changes

The PR adds an opt-in `nativeForwarding` path that captures raw audio/video attachment bytes during media understanding and injects them into embedded-runner prompt media alongside images.

## Best Possible Solution

Land a revised native audio/video design with schema/help/docs coverage, typed prompt-media support, provider capability and fallback behavior, one resolved key for capture/consume/cleanup, and explicit raw-media egress policy.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the strict schema omits the new key, redirected native-command runs can use a different consume key, and the current prompt-media contract is image-only. The happy-path Telegram voice-note scenario also has copied runtime-log proof in the PR body.

## Solution Assessment

No. The feature direction is plausible, but the patch should be revised around a typed audio/video media contract, schema-visible config, aligned session-key lifecycle, and explicit raw-media egress/capability policy.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Use the resolved runner key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: In native-command redirects, reply setup can retarget the run to `CommandTargetSessionKey`, but this stores raw media under `ctx.SessionKey` while the runner consumes and discards under `params.sessionKey`.   - confidence: 0.92

  - body: Adding `nativeForwarding` to the TypeScript config type is not enough because the runtime schema is strict. User config containing the new audio/video key can be rejected until the Zod schema and generated config/help metadata are updated.
  - confidence: 0.91
- **[P2] Add a typed audio/video prompt-media contract:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: The prompt boundary and pinned pi-ai types are still image-shaped, while these new blocks use `type: "audio" | "video"`. Until the upstream prompt contract and adapters accept typed audio/video content, this cast can pass unsupported shapes to providers or drop the media.
  - confidence: 0.9
