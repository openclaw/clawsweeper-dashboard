## Summary

Keep open. The feature is plausible and the PR body includes real Telegram voice-note proof, but the branch is not mergeable as-is because the new config key is not accepted by the strict runtime schema, raw media blocks can be stored under a different key than the runner consumes for native command targets, and the prompt-media contract still depends on an absent pi-ai companion change.

## What This Changes

Adds optional `tools.media.audio/video.nativeForwarding` that captures raw audio/video attachment buffers and merges native media blocks into embedded runner prompt media.

## Best Possible Solution

Define a typed native media prompt contract, update the config schema/help/tests, and key capture/consume/cleanup on the same resolved target run key before merging the feature.

## Reproduction Assessment

Yes for the review findings by source inspection: `nativeForwarding` is absent from the strict zod schema, and the native-command target path clearly retargets after media understanding runs. The contributor also provided copied Telegram runtime logs for the primary feature path.

## Solution Assessment

No. The requested feature direction may be useful, but this patch routes new media through an image-only prompt option and a process-global side channel before the config and session-key contracts are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Add the config key to the runtime schema:** `src/config/types.tools.ts:105-113`
  - body: The PR exposes `tools.media.*.nativeForwarding`, but OpenClaw validates media config with a strict zod schema that this diff does not update. A user who adds the documented option will get an unknown-key config validation error before the forwarding code can run. Add the key to the runtime schema and cover it with a config regression test.
  - confidence: 0.94
- **[P2] Use the resolved target key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: For native slash/menu commands that target an existing chat session, media understanding runs on the inbound slash context, but the embedded runner consumes using the target session key. Storing blocks under `ctx.SessionKey` leaves raw audio/video unconsumed and undiscarded for that path. Thread the resolved target key into media understanding and use one key for push, consume, and cleanup.
  - confidence: 0.88
- **[P2] Gate forwarding on a real prompt media contract:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: This sends `{ type: "audio" | "video", data, mimeType }` through the existing `images` option with a force-cast, but the branch does not update the pinned pi-ai dependency or any local prompt type to accept those blocks. Until the prompt API actually maps native audio/video, enabling the option can send malformed content or fail provider calls. Update the dependency/contract or fail clearly before injecting blocks.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff stores raw user audio/video in a process-global map, and one review finding can leave those sensitive buffers resident indefinitely.

Concerns:

- **[medium] Raw media can remain in memory after key mismatch:** `src/media-understanding/apply.ts:599`
  - body: When native command retargeting makes the push key differ from the consume/discard key, base64 audio/video blocks stay in the module-level store for the process lifetime. That is both a memory leak and unnecessary retention of user media.
  - confidence: 0.87
