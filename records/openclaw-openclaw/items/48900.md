## Summary

Keep open. The feature is still absent from current main, but the PR branch is not merge-ready because the public config key is not in the strict schema/docs, redirected native-command runs can drop and retain media blocks, and the prompt-media boundary still relies on an unlanded pi-ai audio/video contract.

## What This Changes

The PR adds opt-in `tools.media.audio/video.nativeForwarding`, captures raw attachments during media understanding, stores them by session key, and merges those blocks into embedded-runner prompt media.

## Best Possible Solution

Revise the feature around a typed prompt-media contract with provider capability and fallback rules, schema/help/generated-doc updates, one resolved run/session key for capture and cleanup, focused redirect/config tests, and a changelog entry.

## Reproduction Assessment

Yes for the PR blockers. Code inspection shows current main has no native-forwarding implementation, and the keying bug follows from a native command with `CommandTargetSessionKey` where the PR stores under the source session key but consumes under the resolved runner key.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best implementation until raw media egress, config validation/docs, and the upstream prompt-media type/adapter contract are made explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the resolved runner key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: In native command redirects, `getReplyFromConfig` runs the agent under `CommandTargetSessionKey`, but this code stores native blocks under the source `ctx.SessionKey`. The runner consumes and discards under `params.sessionKey ?? params.sessionId`, so redirected voice/video blocks are never forwarded and can retain the capped media payload for the process lifetime. Thread the same resolved session/run key into media understanding and use it for push, consume, and discard.
  - confidence: 0.92
- **[P2] Add the config schema for nativeForwarding:** `src/config/types.tools.ts:100`
  - body: Adding `nativeForwarding` to the TypeScript type is not enough because `ToolsMediaUnderstandingSchema` is `.strict()`. User config containing `tools.media.audio.nativeForwarding` or `tools.media.video.nativeForwarding` will be rejected unless the zod schema, generated config metadata/help, and docs are updated with the new key.
  - confidence: 0.91
- **[P2] Do not cast native media blocks through ImageContent:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: The prompt boundary is still typed as `ImageContent[]`, while these new blocks use `type: "audio" | "video"`. Until the pi-ai contract and adapters accept typed audio/video content, this cast can send unsupported prompt shapes to providers or silently lose the media instead of failing clearly.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/config/types.tools.ts:100`
  - body: This exposes a user-facing feature and config option, but the PR does not touch `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-visible feature work before merge.
  - confidence: 0.88
