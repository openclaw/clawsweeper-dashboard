The issue reports that cron stale-future repair can misclassify exact-second persisted cron slots and rebase a valid future `nextRunAtMs`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81731 or an equivalent narrow fix that probes past the candidate second and keeps focused cron regression coverage.

## Reproduction Assessment

Yes, source-level: current main classifies candidate slots by probing Croner at `runAtMs + 1`, while Croner previous-run enumeration works at second granularity. I did not run a local failing test in this read-only checkout, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, the linked PR's `+ 1_000` probe is the narrowest maintainable fix because it preserves the existing repair logic while moving the previous-run lookup past the candidate second. The added focused regression test targets the exact multi-interval future slot that current tests miss.
