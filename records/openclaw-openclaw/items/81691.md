## Summary

Keep open: current main and v2026.5.12 still use the 1ms previous-run probe, the Croner 10.0.1 boundary behavior reproduces the reported exact-second mismatch, and the open closing PR at https://github.com/openclaw/openclaw/pull/81731 should land before this issue closes.

## What This Changes

The issue requests that stale-future cron repair preserve exact-second persisted cron slots instead of rebasing them as stale.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81731 or an equivalent patch with the regression test, then close this issue as implemented after merge.

## Reproduction Assessment

Yes, at source/dependency level: current main uses the `runAtMs + 1` probe, and Croner 10.0.1 returns the prior scheduled slot for the reported `slot + 1ms` probe while `slot + 1000ms` returns the candidate slot. I did not run a full OpenClaw test in this read-only pass.

## Solution Assessment

Yes. Probing one second past the candidate slot and adding the Honolulu regression test is the narrow maintainable fix; the issue should stay open until the existing closing PR lands.
