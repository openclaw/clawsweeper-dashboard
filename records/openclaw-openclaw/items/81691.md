## Summary

Keep open: current main and the latest shipped tag still use the `runAtMs + 1` previous-run probe, and a focused Croner 10.0.1 check reproduces the exact-second mismatch described in the report. No open canonical fix was found.

## What This Changes

The issue requests preserving exact-second persisted cron `nextRunAtMs` slots during stale-future repair.

## Best Possible Solution

Update the cron slot predicate to use second-granular schedule semantics, then add a focused regression test for the Honolulu exact slot and nearby non-slots.

## Reproduction Assessment

Yes. The current source uses the reported `runAtMs + 1` probe, and a focused Croner 10.0.1 check shows that `slot + 1ms` returns the prior day while `slot + 1000ms` returns the candidate slot.

## Solution Assessment

Yes. Probing beyond the candidate second or otherwise using Croner’s second-granular match semantics is the narrow maintainable fix; a broader scheduler refactor is not needed for this bug.
