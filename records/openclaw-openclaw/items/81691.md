## What This Changes

The issue reports that cron stale-future repair can rebase valid exact-second persisted cron slots because `isStaggeredCronRunAtMs` checks the previous run at `runAtMs + 1ms`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81731 or an equivalent narrow fix with exact-second regression coverage, then close this issue from the merged fix.

## Reproduction Assessment

Yes, at source level: current main still probes `runAtMs + 1`, and Croner 10.0.1 previous-run enumeration operates at second precision. I did not run a full failing runtime test because this review was read-only.
