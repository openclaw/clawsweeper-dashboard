## Summary

Keep open: current main still uses the reported `runAtMs + 1` cron-slot probe, and the issue is now paired with an open closing PR that specifically addresses this bug. Per repo policy, the right cleanup is to review and land or close the linked fix PR, not close the issue before that PR merges.

## What This Changes

The issue reports that stale future cron-slot repair treats valid exact-second persisted cron `nextRunAtMs` values as stale because the schedule-slot predicate probes the previous run at `runAtMs + 1`.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/81731, confirm the one-second probe and regression coverage are acceptable, then close this issue automatically or manually after the fix merges.

## Reproduction Assessment

Yes. Source inspection shows the current helper still probes `runAtMs + 1` on the stale-future repair path, and the existing ClawSweeper review plus the linked fix PR provide a focused Croner contract reproduction for the exact-second slot.

## Solution Assessment

Yes. The narrow maintainable fix is to probe beyond the candidate second and add the focused regression test; the open linked PR is already shaped around that approach.
