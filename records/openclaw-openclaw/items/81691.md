## Summary

Current main still has the reported cron slot-classification bug: the repair path probes an exact scheduled second with `+1ms`, while Croner’s previous-run lookup is second-granular enough that `slot + 1ms` still misses the candidate slot.

## What This Changes

The issue asks OpenClaw to preserve exact second-aligned persisted cron `nextRunAtMs` values during stale-future repair.

## Best Possible Solution

Change the scheduled-slot check to use second-granular semantics, then add focused regression coverage for exact, minus-30-second, and plus-30-second Honolulu cron timestamps.

## Reproduction Assessment

Yes. Current source shows the `+1ms` probe, and a standalone probe against the pinned `croner@10.0.1` package reproduces `slot + 1ms` resolving to the prior day and the current repair condition returning `shouldRepair:true`.

## Solution Assessment

Yes. Probing past the candidate second, or otherwise normalizing the check to Croner’s second-granular schedule semantics, is the narrowest maintainable fix and should be paired with the suggested regression test.
