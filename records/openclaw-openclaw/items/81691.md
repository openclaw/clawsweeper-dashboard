## Summary

Keep open: current `main` and `v2026.5.12` still use the reported 1 ms cron-slot probe, and the issue now has an open closing PR that should be reviewed or landed before the issue is closed.

## What This Changes

The issue reports that stale future cron repair can rebase valid exact-second cron slots because the schedule-slot predicate probes the previous run at `runAtMs + 1`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81731 or an equivalent patch with the regression test, then close this issue after the fix reaches `main`.

## Reproduction Assessment

Yes. Current `main` still has the `runAtMs + 1` probe, and the existing ClawSweeper comment plus linked PR proof show the Croner 10.0.1 exact-second mismatch for the Honolulu repro input.

## Solution Assessment

Yes. The linked PR’s one-second probe and focused regression test are the narrow maintainable fix; closing the issue before that PR merges would be premature.
