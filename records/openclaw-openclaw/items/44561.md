## Summary

Keep open: the reported gateway/UI display-name bug is still source-reproducible on current main, but this PR is incomplete because it stabilizes only the older `listSessionsFromStore` path while current main also emits rows through the shared `buildGatewaySessionRow` path used by live `sessions.changed` updates.

## What This Changes

The PR changes `src/gateway/session-utils.ts` so `listSessionsFromStore` recognizes configured agent/global main session keys and returns `entry.displayName` or `Main Session` instead of transient labels.

## Best Possible Solution

Move the stabilization into `buildGatewaySessionRow` or a shared display-name helper used by list and live row callers, preserve explicit `entry.displayName`, recognize configured per-agent and global main keys, and add focused gateway/UI regression coverage.

## Reproduction Assessment

Yes, by source inspection: project a main session row with no explicit `displayName`, no `entry.label`, and `origin.label: "heartbeat"` through `buildGatewaySessionRow`; current main will expose `heartbeat` as row display metadata before the UI fallback can return `Main Session`. Tests were not run because this review is read-only.

## Solution Assessment

No. The intended behavior is right, but the PR puts the guard in a stale list-only location; the maintainable fix is a shared row-projection guard with regression coverage for both `sessions.list` and live `sessions.changed` callers.
