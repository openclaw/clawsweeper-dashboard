## Summary

Keep this PR open: it targets a real main-session display-name bug, but the branch is stale against current main and the patch is now too narrow. Current main routes both session-list rows and live session-change snapshots through `buildGatewaySessionRow`, while this PR only patches the older inline `listSessionsFromStore` display-name path.

## What This Changes

The PR changes `src/gateway/session-utils.ts` so `listSessionsFromStore` treats configured agent and global main session keys as stable main sessions, preserving explicit `entry.displayName` and otherwise returning `Main Session` instead of transient origin labels.

## Best Possible Solution

Put the stabilization in `buildGatewaySessionRow` or a shared display-name helper used by both `listSessionsFromStore` and `loadGatewaySessionRow`, preserve explicit `entry.displayName`, recognize configured per-agent and global main session keys, and cover stale heartbeat-origin labels with focused gateway/UI regression tests.

## Reproduction Assessment

Yes. A high-confidence reproduction is to project a main session row with no explicit `displayName`, no `entry.label`, and `origin.label: "heartbeat"` through `buildGatewaySessionRow`; current main will produce a row display name that the UI prefers before its `Main Session` fallback.

## Solution Assessment

No. The intended behavior is right, but the proposed patch applies it in an older list-only path; the narrower maintainable fix is a shared row-projection guard plus regression coverage for list and live session-change callers.
