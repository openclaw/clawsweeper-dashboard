## Summary

Keep open: this is a focused planner callback seam, but it does not fix the linked descriptor-registration/startup diagnostic behavior. Current main has no production `buildToolPlan` caller to wire the callback, and plugin descriptor capture still does not preserve tool-authored availability.

## What This Changes

Adds an optional `BuildToolPlanOptions.onHiddenDiagnostic` callback, invokes it when `buildToolPlan` hides a descriptor, and adds a planner unit test for hidden descriptors.

## Best Possible Solution

Surface `unsupported-signal` authoring diagnostics once at descriptor registration or plugin resolution while keeping the evaluator pure; keep the exported planner callback only if a real owner wires and documents it.

## Reproduction Assessment

Yes, from source inspection: a descriptor with `availability: { anyOf: [] }` reaches `buildToolPlan`, is moved to `hidden` with an `unsupported-signal` diagnostic, and current main has no production caller that emits it.

## Solution Assessment

No. The callback is a plausible helper, but the best fix is to surface malformed descriptor diagnostics at a lifecycle-owned registration or plugin resolver boundary before claiming the linked bug is fixed.
