## Summary

Keep open: the latest PR is still not merge-ready because the new plugin-resolution warning loop evaluates cached descriptors that do not preserve plugin-authored `availability`, so the linked registration warning still will not fire, and the supplied proof remains unit-test-only.

## What This Changes

The branch adds an exported `BuildToolPlanOptions.onHiddenDiagnostic` callback, invokes it for hidden planner diagnostics, adds a plugin-resolution warning loop, and adds a planner unit test.

## Best Possible Solution

Preserve and validate plugin-authored availability at the descriptor-registration or plugin-resolution boundary, emit `unsupported-signal` authoring warnings there, and keep the planner callback only if maintainers want that exported API.

## Reproduction Assessment

Yes. Source inspection shows current descriptor capture drops plugin-authored `availability`, and missing availability evaluates as always available, so the PR's new plugin-resolution loop will not warn for the reported empty-group plugin descriptor path.

## Solution Assessment

No. The PR is a plausible support hook, but as written it evaluates descriptors after the relevant metadata has already been dropped; the best fix is a lifecycle-owned registration or plugin-resolution diagnostic with a deliberate API decision for the callback.
