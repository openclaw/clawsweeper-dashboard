## Summary

Keep open: the latest patch still misses the linked registration-path diagnostic because the new plugin-resolution loop evaluates cached descriptors after plugin-authored availability has already been dropped, so malformed empty plugin availability can still produce no warning.

## What This Changes

Adds an optional `BuildToolPlanOptions.onHiddenDiagnostic` callback, invokes it from `buildToolPlan`, adds planner test coverage, and adds a plugin-resolution warning loop over cached descriptors.

## Best Possible Solution

Preserve or validate plugin-authored availability at the descriptor-registration or plugin-resolution boundary, warn once only for malformed authoring diagnostics, and keep the planner callback only if maintainers want that exported API.

## Reproduction Assessment

Yes. Source inspection shows `capturePluginToolDescriptor` omits plugin-authored `availability`, and `evaluateToolAvailability` treats missing availability as available, so the PR's new post-capture loop cannot warn for the reported plugin descriptor path.

## Solution Assessment

No. The callback is a plausible support hook, but the best fix for the linked behavior is at descriptor registration or plugin resolution before availability is dropped, with warning filtering for malformed authoring diagnostics.
