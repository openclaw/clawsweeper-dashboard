- `P2`: The affected `/btw` workflow is deterministically broken or unsafe for CLI aliases, but normal agent turns remain usable and the blast radius is limited.
- `merge-risk: 🚨 compatibility`: Merging changes existing CLI-alias behavior from attempting direct fallback to immediately reporting that `/btw` is unsupported.

## Summary

The patch correctly prevents `/btw` from silently crossing from a configured CLI runtime into direct provider authentication, preserves the generic fallback for non-CLI harnesses, and now has credible production-handler proof. It should remain open for explicit agent/harness owner approval because the safety fix intentionally changes affected existing setups to an immediate unsupported error and does not implement the linked issue’s requested same-runtime side-question capability.

## What This Changes

The PR adds a fail-closed guard before `/btw` direct-provider fallback for CLI-runtime aliases and adds focused regression coverage while retaining fallback for non-CLI custom harnesses.

## Best Possible Solution

Keep `/btw` on the active session’s runtime and authentication boundary; approve this guard only as a documented interim state, then implement `runSideQuestion` in the appropriate CLI harness so side questions use the same backend as the main turn.

## Reproduction Assessment

Yes. Current source provides a high-confidence path: a CLI-runtime alias whose selected harness lacks `runSideQuestion` falls through to generic direct-provider model, auth, and stream handling; the branch deterministically blocks that transition.

## Solution Assessment

Yes as a narrow interim safety fix, but only with explicit owner approval of the compatibility tradeoff. The permanent best fix is a harness-owned side-question path that answers through the active CLI runtime.
