## Summary

The patch is a focused, technically sound interim mitigation with strong real-handler proof, but it intentionally changes existing CLI-alias behavior and carries the protected `maintainer` label. The comment describing “maintainer policy approval” was posted by the contributor, so an agent/harness owner still needs to explicitly accept the fail-closed compatibility policy before merge.

## What This Changes

The PR prevents `/btw` from falling back to direct provider authentication and streaming for matching CLI-runtime aliases without a side-question hook, while preserving fallback for non-CLI harnesses.

## Best Possible Solution

Keep ephemeral side-question execution owned by each runtime harness; for matching CLI aliases without a safe hook, fail closed rather than silently changing backend, after an agent/harness owner explicitly accepts that interim user-visible behavior.

## Reproduction Assessment

Yes. Current source deterministically sends a CLI alias without a side-question hook into direct-provider fallback, and the submitted Crabbox scenario exercises the production command handler with that configuration.

## Solution Assessment

Yes as the narrowest interim safety mitigation: same-runtime answering belongs in a harness hook, while generic fallback must not silently change backend. Explicit owner approval is still required because the patch deliberately changes existing user-visible behavior.
