## Summary

This PR is a focused safety guard against silently changing backend identity, but it intentionally makes configured CLI-runtime `/btw` requests fail and does not satisfy the linked issue’s expected same-runtime side question. It should remain open for explicit owner approval and real-session proof; the protected `maintainer` label also prevents cleanup closure.

## What This Changes

The PR makes `/btw` reject direct-provider fallback for models mapped to CLI runtime aliases while preserving the generic fallback for non-CLI custom harnesses and adding regression tests.

## Best Possible Solution

Define `/btw` at the harness boundary: add a CLI-runtime `runSideQuestion` implementation when the runtime can safely fork or run ephemeral context, and otherwise use an explicitly approved, documented fail-closed error without direct-provider substitution.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a CLI alias selected through harness policy has no side-question hook and currently falls through to direct model/auth streaming; the patch deterministically throws before that fallback. The contributor has not shown this in a real configured session.

## Solution Assessment

No, not yet as the permanent fix: failing closed protects backend identity, but it does not provide the same-runtime `/btw` behavior requested by the linked issue. The maintainable end state is a harness-owned side-question path or an explicitly approved unsupported contract.
