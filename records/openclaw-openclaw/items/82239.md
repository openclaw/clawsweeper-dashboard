## Summary

Keep open: this is a fresh external regression report with concrete 2026.5.12 evidence, and current main still has the Telegram partial-stream finalization path that can treat preview-finalized text as delivered and mirror it into the transcript. The related Telegram delivery issues are adjacent, but none clearly owns this exact truncated-final plus second delivery-mirror symptom.

## What This Changes

The issue reports that Telegram partial streaming can finalize to a truncated visible reply and append that shorter delivery-mirror message after the full assistant answer is already in the transcript.

## Best Possible Solution

Keep partial streaming, but make finalization authoritative: use the assembled final assistant answer for the final Telegram edit or a failure-confirmed fallback, and suppress or dedupe delivery-mirror transcript appends that would shorten an already-recorded assistant turn.

## Reproduction Assessment

Yes at source level: the report gives concrete Telegram partial-streaming steps, and current source still has the preview-finalized transcript mirror path that matches the second assistant entry. I did not run a live Telegram group-topic reproduction, and the exact truncation trigger still needs a focused regression test.

## Solution Assessment

No for the current implementation: treating preview-finalized content as final without reconciling it with the full assistant output can leave Telegram and the transcript with a shorter answer. The maintainable fix is a narrow finalizer guard/fallback plus regression coverage, not a new setting or default-mode change.
