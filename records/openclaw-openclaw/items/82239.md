## Summary

Keep open: this is a fresh external regression report with concrete 2026.5.12 evidence, and current Telegram code still has a preview-finalized path that mirrors the finalized preview text back into the transcript. The related issues are adjacent, but none clearly supersedes this exact truncated-final plus second delivery-mirror transcript symptom.

## What This Changes

The issue asks OpenClaw to make Telegram `streaming.mode: "partial"` finalize with the complete assistant answer and avoid appending a shorter delivery-mirror assistant message after the full transcript entry.

## Best Possible Solution

Keep partial streaming, but make finalization authoritative: use the assembled final assistant answer for the final Telegram edit or fall back to normal `sendMessage`, and suppress or dedupe delivery-mirror transcript appends that would shorten an already-recorded assistant turn.

## Reproduction Assessment

Yes, there is a credible reproduction path from the report: Telegram group topic, `channels.telegram.streaming.mode = "partial"`, and a normal multi-paragraph answer on 2026.5.12. I did not live-reproduce it, but source inspection confirms the preview-finalized transcript mirror path that matches the second truncated assistant entry.

## Solution Assessment

No, the current behavior is not the best solution: treating the preview-finalized text as final without reconciling it with the full assistant output can leave Telegram and the transcript with a shorter answer. The safer fix is a narrow finalizer guard/fallback plus regression coverage, not a new setting or default-mode change.
