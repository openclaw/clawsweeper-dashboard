## Label Justifications

- `P2`: This is a normal-priority Telegram agent/channel workflow improvement with real routing value but bounded blast radius and an active PR candidate.
- `impact:session-state`: The requested behavior binds persistent spawned subagent sessions to Telegram topics, so unresolved routing can hide or mis-associate session state.
- `impact:message-loss`: Without the bound Telegram child-topic path, subagent follow-ups and completion messages can be unavailable or routed away from the intended topic.

## Summary

Keep open. Current main and the latest release have Telegram topic-binding infrastructure, but they still do not expose Telegram as a model-visible `sessions_spawn(thread:true, mode:"session")` subagent child-topic spawn path; the direct implementation candidate remains open and unmerged.

## What This Changes

The issue requests Telegram support for `sessions_spawn` with `mode="session"` and `thread=true`, binding spawned subagent sessions to Telegram conversations or forum topics.

## Best Possible Solution

Land or replace the active Telegram PR with a narrow capability seam that lets Telegram advertise explicit subagent child-topic spawns while preserving normal current-topic placement and using the core session-binding adapter path.

## Reproduction Assessment

Yes, source-level. Current main keeps Telegram as current-placement while `sessions_spawn` only exposes thread/session parameters for automatic child-placement channels, so the requested Telegram child-topic subagent spawn path remains hidden from the model-facing tool contract.

## Solution Assessment

No, not as originally proposed. Adding new primary `subagent_spawning` routing is no longer the best layer; the maintainable path is an explicit Telegram spawn capability that uses the existing core session-binding adapter path.
