## Summary

Keep open. Current main still lacks Telegram native subagent lifecycle hook wiring for `sessions_spawn({ mode: "session", thread: true })`, even though Telegram has separate conversation/thread binding infrastructure and ACP topic binding support. The open implementation candidate remains PR #42707, so this issue should stay open for maintainer review of that PR or an equivalent fix.

## What This Changes

Request Telegram support for persistent thread-bound native subagent sessions created with `sessions_spawn` using `mode: "session"` and `thread: true`.

## Best Possible Solution

Review and land PR #42707 or a narrow equivalent in the Telegram plugin boundary: add Telegram `registerFull` lifecycle hooks for native subagent spawning, delivery target resolution, and cleanup; reuse the existing Telegram thread-binding manager/session-binding adapter; honor `channels.telegram.threadBindings.spawnSubagentSessions`; and keep ACP topic binding behavior covered separately in tests and docs.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a Telegram-origin `sessions_spawn({ mode: "session", thread: true })` reaches the core hook gate, but current Telegram never registers the native lifecycle hooks needed to return `threadBindingReady`.

## Solution Assessment

Yes, the best direction is a Telegram plugin-boundary implementation rather than a core special case. The current issue proposal is directionally correct, but it should align with the existing session-binding adapter and current `idleHours`/`maxAgeHours` config style instead of adding a separate Telegram-only lifecycle model.
