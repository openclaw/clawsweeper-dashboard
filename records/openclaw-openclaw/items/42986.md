- `P2`: This is a normal-priority Telegram channel and agent workflow improvement with real routing value but bounded blast radius and an active implementation candidate.
- `impact:session-state`: The requested behavior keeps persistent spawned subagent sessions associated with Telegram topics, so the unresolved gap can leave session state unavailable or mis-associated.

## Summary

Keep open. Current main and v2026.6.10 have Telegram session-binding primitives, including child topic creation, but Telegram still resolves as `current` placement and is not exposed as a model-visible `sessions_spawn(thread:true, mode:"session")` child-topic subagent path. The direct candidate remains https://github.com/openclaw/openclaw/pull/82023, which is open and conflicting, while https://github.com/openclaw/openclaw/pull/83274 only partially overlaps the remaining work.

## What This Changes

The issue requests Telegram support for persistent thread-bound subagent sessions created with `sessions_spawn(thread:true, mode:"session")`.

## Best Possible Solution

Keep this issue open as the canonical tracker, then land or replace the active Telegram work with a narrow explicit subagent child-topic capability that preserves Telegram's normal current-topic placement and uses the core session-binding adapter path.

## Reproduction Assessment

Yes, source-level. Current main keeps Telegram as `current` placement while `sessions_spawn` only exposes thread/session fields for automatic child-placement channels, so the requested Telegram child-topic subagent spawn path remains hidden from the model-facing tool contract.

## Solution Assessment

No, not as originally proposed. Adding new primary `subagent_spawning` routing is no longer the best layer; the maintainable path is an explicit Telegram spawn capability that uses the existing core session-binding adapter path.
