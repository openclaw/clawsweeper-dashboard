## Label Justifications

- `P2`: This is a normal-priority Telegram session-routing capability gap with real user-visible impact but bounded to one channel and subagent spawning.
- `impact:session-state`: The requested behavior is about keeping spawned subagent sessions persistently bound to the correct Telegram conversation or forum topic.
- `impact:message-loss`: Without a valid Telegram thread-bound spawn path, subagent follow-ups or completion output can become invisible, misrouted, or unavailable to the requester.

## Summary

Keep this open: current main and the latest release still do not expose Telegram as a supported `sessions_spawn(thread:true, mode:"session")` subagent spawn channel, and the open implementation candidate is still conflicting and needs maintainer direction.

## What This Changes

Request Telegram support for persistent thread-bound native subagent sessions, ideally binding spawned subagents to Telegram conversations or forum topics.

## Best Possible Solution

Expose Telegram native subagent child-topic spawning through the current core `SessionBindingService` path with an explicit capability contract, refresh focused tests and live Telegram proof, then close this after the fix lands.

## Reproduction Assessment

Yes, source-level. Current main gates model-facing thread/session spawn availability on child-placement channels, while Telegram keeps top-level placement as current even though its adapter can create child forum topics.

## Solution Assessment

No for the issue's original hook-based sketch. The maintainable path is to expose a Telegram-specific automatic spawn capability through the current core session-binding path rather than adding new deprecated `subagent_spawning` routing.
