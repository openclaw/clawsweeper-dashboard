## Summary

Keep open because current main still lacks Telegram native subagent lifecycle hook wiring for thread-bound persistent subagent sessions. Telegram has separate conversation/thread-binding infrastructure and ACP topic binding support, but `sessions_spawn({ mode: "session", thread: true })` still depends on channel `subagent_spawning` hooks that Telegram does not register; PR #42707 remains the open implementation candidate.

## What This Changes

The issue requests native Telegram support for thread-bound persistent subagent sessions via `sessions_spawn({ mode: "session", thread: true })`.

## Best Possible Solution

Review and land PR #42707 or a narrow equivalent in the Telegram plugin boundary, reusing the existing session-binding adapter and `threadBindings.spawnSessions` policy rather than adding a core special case.

## Reproduction Assessment

Yes. Source inspection shows a Telegram-origin `sessions_spawn({ mode: "session", thread: true })` reaches the core hook gate, while current Telegram never registers the native lifecycle hooks needed to return `threadBindingReady`.

## Solution Assessment

Yes. The maintainable direction is Telegram plugin-boundary hook wiring that reuses the existing Telegram thread-binding/session-binding infrastructure; a core Telegram special case would violate the channel/plugin boundary.
