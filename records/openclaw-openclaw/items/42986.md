## Label Justifications

- `P2`: This is a normal-priority Telegram channel/agent workflow improvement with real routing value but bounded blast radius and an active implementation candidate.
- `impact:session-state`: The requested behavior keeps persistent spawned subagent sessions associated with Telegram topics, so the unresolved gap can leave session state unavailable or mis-associated.
- `impact:message-loss`: Without the bound Telegram child-topic spawn path, subagent follow-ups and completion messages can be hidden or routed away from the intended topic.

## Summary

Keep open. Current main and v2026.6.10 have Telegram thread-binding primitives, but Telegram is still not exposed as a model-visible `sessions_spawn(thread:true, mode:"session")` child-topic subagent spawn path, and the direct implementation PR remains open and conflicting.

## What This Changes

Request Telegram `sessions_spawn(thread:true, mode:"session")` support that binds persistent subagent sessions to Telegram conversations or forum topics.

## Best Possible Solution

Land or replace the active Telegram work with a narrow explicit subagent child-topic capability that preserves Telegram's normal current-topic placement and uses the core session-binding adapter path.

## Reproduction Assessment

Yes, source-level. Current main keeps Telegram as current-placement while `sessions_spawn` only exposes thread/session fields for automatic child-placement channels, so the requested Telegram child-topic subagent spawn path remains hidden from the model-facing tool contract.

## Solution Assessment

No, not as originally proposed. Adding new primary `subagent_spawning` routing is no longer the best layer; the maintainable path is an explicit Telegram spawn capability that uses the existing core session-binding adapter path.
