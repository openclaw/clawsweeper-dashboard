## Summary

Keep open because current main still lacks Telegram native subagent lifecycle hook wiring for `sessions_spawn` with `mode:"session"` and `thread:true`, even though Telegram already has separate conversation/thread-binding infrastructure for ACP/topic routing.

## What This Changes

Request Telegram support for persistent thread-bound native subagent sessions created by `sessions_spawn({ mode: "session", thread: true })`.

## Best Possible Solution

Review and land PR #42707 or a narrow equivalent in the Telegram plugin boundary, reusing the existing Telegram session-binding adapter and `threadBindings.spawnSessions` policy without adding a core Telegram special case.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a Telegram-origin `sessions_spawn({ mode: "session", thread: true })` reaches the core `subagent_spawning` hook gate, but Telegram does not register the native lifecycle hooks needed to return `threadBindingReady`. I did not run a live Telegram spawn in this read-only review.

## Solution Assessment

Yes. The best direction is Telegram plugin-boundary hook wiring that reuses the existing Telegram thread-binding/session-binding infrastructure; a core channel-specific special case would cut across the plugin boundary.
