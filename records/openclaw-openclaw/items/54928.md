## Summary

Keep open. Current main still rejects Exa `webSearch.baseUrl`, the Exa runtime still sends requests to the fixed Exa endpoint, and open PR #54939 already claims this work with closing syntax, so the issue should stay open until that PR lands, is closed, or is replaced.

## What This Changes

The issue requests support for `plugins.entries.exa.config.webSearch.baseUrl`, or clearer Exa-specific validation/docs guidance when custom Exa endpoints are unsupported.

## Best Possible Solution

Land a reviewed Exa-plugin implementation that adds `webSearch.baseUrl` to the manifest/UI metadata, resolves and validates it in the runtime, uses the trusted web-search endpoint path, partitions cache keys by resolved endpoint, and updates Exa/web-search docs plus focused tests. If maintainers reject custom Exa endpoints, document that capability gap and improve the validation guidance instead.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: adding `baseUrl` under `plugins.entries.exa.config.webSearch` hits the strict Exa manifest schema, and the validation code emits `invalid config: ...` for schema errors.

## Solution Assessment

Unclear for the open PR because its diff was not reviewed in this issue pass. The best maintainable direction is provider-owned Exa plugin work, not a core migration, with explicit endpoint validation and cache isolation.
