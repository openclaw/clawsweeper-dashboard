## Summary

Keep open. Current main still rejects Exa `webSearch.baseUrl`, the runtime still posts to the fixed Exa endpoint, docs still only describe `apiKey`, and open PR #54939 is already the implementation candidate for this issue.

## What This Changes

Requests support for `plugins.entries.exa.config.webSearch.baseUrl`, or explicit schema/docs guidance if Exa custom endpoints remain unsupported.

## Best Possible Solution

Land or replace the open Exa-plugin implementation with a reviewed patch that adds `webSearch.baseUrl` to the manifest/UI metadata, validates and resolves the endpoint in the runtime, keeps requests inside the trusted web-search path, partitions cache keys by endpoint, and updates Exa docs and tests. If maintainers decide Exa should not support endpoint overrides, document that capability gap and improve the provider-specific validation guidance instead.
