## Summary

Keep open. Current main still composes the reported unsafe path: `current` can persist a live chat session key, `session:*` targets are used as the cron run session key, and delete-after-run direct cleanup deletes that same key. Open PR #57831 targets the same root cause, but it is still unmerged, so the issue should remain open until that PR or a successor lands.

## What This Changes

Prevent one-shot `deleteAfterRun` cron reminders targeting `current` or a live `session:*` DM key from deleting or rebinding the live non-default direct-message session.

## Best Possible Solution

Land a narrow cron/session safety fix that prevents delete-after-run cleanup from deleting user-facing direct-message session keys. `current` and explicit direct-chat `session:*` targets should either downgrade to a cron-owned isolated run session or reuse the live session without cleanup, with regression coverage for non-default Signal/DM direct delivery and `delivery.mode: "none"` message-tool sends.

## Reproduction Assessment

Yes. The issue includes concrete logs, and the current source gives a high-confidence reproduction path: create a one-shot `agentTurn` job from a direct session with `sessionTarget: "current"`, let it persist as `session:<live key>`, and direct cleanup will call `sessions.delete` on that key when `deleteAfterRun` is true.

## Solution Assessment

No for current main: deleting whatever concrete session key the cron run used is not the narrow maintainable solution. The safer solution is to prove the session is cron-owned before deletion and to guard normalization/runtime fallback for live direct-chat targets.
