## Summary

Keep open: current main still composes the reported unsafe cron/session path, and the remaining fix is represented by open cron PRs that need review, combination, or replacement before this issue can close.

## What This Changes

The issue reports that one-shot cron jobs can reuse and then delete a live non-default direct-message session key, causing later DM replies to lose transcript context.

## Best Possible Solution

Land or replace the overlapping cron fixes with one narrow implementation that keeps live direct-message session keys out of reusable cron runs, deletes only cron-owned sessions, and adds regression coverage across normalization, runtime execution, and direct delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows `current` can persist a live key, one-shot jobs default cleanup on, runtime reuses `session:*`, and direct cleanup deletes that supplied key; I did not run the destructive live-DM repro.

## Solution Assessment

Yes. The narrow maintainable fix is a cron-session ownership guard plus direct-session target fallback/downgrade, with tests proving safe group/named session reuse remains intact.
