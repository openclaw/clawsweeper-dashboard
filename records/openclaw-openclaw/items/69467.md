## Summary

Keep open. Current main still composes the reported unsafe path: `current` can persist a live direct-message session key, explicit `session:*` targets become the cron run key, and `deleteAfterRun` direct cleanup deletes that same key; open PR #57831 targets the same root cause but is not merged in the provided GitHub context.

## What This Changes

The issue reports that one-shot cron jobs created from a live non-default direct-message session can persist that live session key and delete or rebind it after `deleteAfterRun` cleanup.

## Best Possible Solution

Review and land PR #57831 or an equivalent narrow successor that blocks direct-message `current`/`session:*` reuse at create/update time, falls back safely for already-persisted dirty jobs at runtime, and adds regression coverage for direct delivery plus `delivery.mode: "none"` message-tool sends.

## Reproduction Assessment

Yes. The issue includes a concrete log timeline, and current source gives a high-confidence path: create a one-shot `agentTurn` from a direct session with `sessionTarget: "current"`, let it persist as `session:<live key>`, and direct cleanup can call `sessions.delete` on that key when `deleteAfterRun` is true.

## Solution Assessment

No for current main. The maintainable fix is to prove the run session is cron-owned before cleanup and to guard direct-message `current` or explicit `session:*` targets before they reach runtime.
