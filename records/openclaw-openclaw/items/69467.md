## Summary

Keep open. Current main still allows `sessionTarget: "current"` or explicit `session:*` cron targets to carry a live DM session key into one-shot `deleteAfterRun` direct cleanup, and the two related fix PRs are still open rather than merged.

## What This Changes

The issue reports that one-shot cron jobs using live direct-message session targets can delete and rebind a non-default agent DM session when `deleteAfterRun` cleanup runs.

## Best Possible Solution

Land or replace the linked cron fixes with one narrow implementation that keeps live direct-message session keys out of reusable cron runs, deletes only cron-owned sessions, and adds regression coverage across normalization, runtime, and direct delivery.

## Reproduction Assessment

Yes. The issue includes a detailed sanitized incident timeline, and current main still composes the path: `current` persists a live key, one-shot jobs default cleanup on, runtime reuses `session:*`, and direct cleanup deletes that key.

## Solution Assessment

Yes. Keeping this open for the linked cron-session ownership guard and direct-session target fallback is the narrow maintainable path; current main has not implemented it yet.
