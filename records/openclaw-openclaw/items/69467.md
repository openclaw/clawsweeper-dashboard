## Summary

Keep open. Current main still composes the unsafe cron path: `current` can persist a live session key, one-shot jobs default cleanup on, runtime reuses explicit `session:*` targets, and direct cleanup deletes the supplied session key; the related fix PRs remain open and unmerged.

## What This Changes

The issue reports live direct-message session loss when one-shot cron jobs targeting `current` or `session:<live-key>` run with `deleteAfterRun` cleanup.

## Best Possible Solution

Land or replace the linked cron fixes with one narrow implementation that keeps live direct-message session keys out of reusable cron runs, only deletes cron-owned sessions, and adds regression coverage across normalization, runtime, and direct delivery.

## Reproduction Assessment

Yes. The issue includes a detailed sanitized incident timeline, and current main source still composes the path: `current` persists a live key, one-shot jobs default cleanup on, runtime reuses `session:*`, and direct cleanup deletes that key.

## Solution Assessment

Yes. A cron-session ownership guard plus a direct-session target fallback is the narrow maintainable direction; current main has not implemented it, so the open fix PRs or a combined successor should be reviewed before closing this issue.
