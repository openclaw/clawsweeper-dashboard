## Summary

Keep open. Current main still composes the unsafe cron path, and open fix PRs already target the root cause, so the issue should remain open until one of those fixes or a combined successor lands.

## What This Changes

The issue asks for one-shot cron jobs using `deleteAfterRun` not to delete or rebind a live non-default direct-message session when `sessionTarget` resolves to the current live session key.

## Best Possible Solution

Land or replace the linked cron fixes with a narrow implementation that proves `deleteAfterRun` cleanup only targets cron-owned sessions, prevents live direct-message `current` or explicit `session:*` targets from being reused, and adds regression coverage across normalization, runtime execution, and direct delivery.

## Reproduction Assessment

Yes. The issue includes a concrete log timeline, and current main still composes the path: `current` can persist a live session key, one-shot jobs default cleanup on, runtime reuses `session:*`, and direct cleanup deletes that key.

## Solution Assessment

Yes, a cron-session ownership guard plus direct-session target fallback is the narrow maintainable direction. Current main has not implemented it yet, so the linked open PRs or a combined successor should be reviewed before closing this issue.
