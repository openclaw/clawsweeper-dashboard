## Summary

Keep open. Current main still composes the reported unsafe path, and the remaining fix work is already represented by open cron PRs that need review or replacement before this issue can close.

## What This Changes

One-shot cron jobs targeting `current` or `session:<live DM key>` with `deleteAfterRun` should not delete or rebind a live non-default Signal DM session.

## Best Possible Solution

Land or replace the linked cron fixes with a narrow implementation that only cleans up cron-owned sessions, keeps live direct-message session keys out of reusable cron runs, and adds regression coverage across normalization, runtime execution, and direct delivery.

## Reproduction Assessment

Yes. The issue includes a detailed sanitized incident timeline, and current main source still composes the path: `current` persists a live key, one-shot jobs default cleanup on, runtime reuses `session:*`, and direct cleanup deletes that key.

## Solution Assessment

Yes. A cron-session ownership guard plus a direct-session target fallback is the narrow maintainable direction; current main has not implemented it, so the open fix PRs or a combined successor should be reviewed before closing this issue.
