## Summary

Keep open. The default Docker-image root cause was shipped separately, but current main still lacks this PR's remote-shell/custom-image missing-python guidance, and the PR branch is dirty against current main with a local detector that no longer matches the current helper launch contract.

## What This Changes

The PR adds a sandbox mutation error helper, wraps local and remote sandbox fs-bridge mutation execution to translate missing-`python3` failures, and adds a local bridge unit test.

## Best Possible Solution

Land a fresh current-main patch that preserves the shipped default-image behavior while adding missing-interpreter guidance for the current Docker mutation helper and the remote-shell helper, preserving the original error as `cause` and covering both paths with targeted tests.

## Reproduction Assessment

Yes. The default Docker-image case is covered by `ensureDockerImage()` tests, and the remaining remote-shell gap can be reproduced by mocking `runRemoteShellScript` to throw `python3: not found` during `createRemoteShellSandboxFsBridge().writeFile()` while current main has no catch around `runRemoteScript`.

## Solution Assessment

No. The PR direction is useful, but the current diff is not the best fix because its local detector targets the old `python3 /dev/fd/3` helper marker instead of the current `python_cmd` helper contract; rebasing and mapping the current local and remote mutation failures is safer.
