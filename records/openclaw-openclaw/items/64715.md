## Summary

Keep this issue open. Current main still exposes Telegram outbound interactive controls as inline keyboards only, while native reply keyboards and reply-keyboard removal are absent from the Telegram send/action/message-tool surfaces; the related closed items cover inline-button behavior, not this separate capability.

## What This Changes

Add Telegram-native reply keyboard and keyboard removal support to the agent/message outbound send surface.

## Best Possible Solution

Implement an additive Telegram plugin-owned reply_markup path with typed reply keyboard rows, removal, resize, one-time behavior, docs, and regression coverage while preserving existing inline-keyboard behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the outbound Telegram path only builds inline keyboards, and the message tool/docs expose no native reply-keyboard or keyboard-removal fields.

## Solution Assessment

Yes. The narrow maintainable direction is a Telegram-specific additive send/action/schema path; changing the shared presentation contract should be deferred unless maintainers explicitly want portable persistent-keyboard semantics.
