## Summary

Keep this issue open. Current main still exposes Telegram outbound keyboard support as inline keyboards only, while Telegram and grammY support reply keyboards and keyboard removal on send-style `reply_markup`.

## What This Changes

The issue requests native Telegram reply keyboard and keyboard-removal support through the agent/message send surface.

## Best Possible Solution

Add an additive Telegram plugin-owned reply-markup path with typed reply keyboard rows, keyboard removal, resize and one-time flags, docs, and regression tests while preserving existing inline keyboard behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the Telegram outbound send, action runtime, message-tool schema, docs, and tests only expose inline keyboard handling, while upstream Telegram/grammY supports reply keyboard markup on send paths.

## Solution Assessment

Yes. The narrow maintainable solution is an additive Telegram-owned send/action/schema implementation; changing the shared portable presentation contract should be deferred unless maintainers explicitly choose that broader design.
