## Summary

Keep this issue open. Current main still exposes Telegram outbound controls as inline keyboards only, while Telegram and the locked @grammyjs/types contract support reply keyboards and keyboard removal on send methods. The related items are about inline buttons/callbacks, so they do not supersede this separate reply-keyboard feature gap.

## What This Changes

Add Telegram-native reply keyboard rows/buttons, keyboard removal, resize_keyboard, and one_time_keyboard support to the agent/message outbound send surface.

## Best Possible Solution

Implement this as an additive Telegram plugin-owned reply_markup path for outbound sends, with typed reply keyboard rows and remove-keyboard semantics, while preserving existing inline keyboard behavior and leaving any broader portable keyboard contract to an explicit maintainer design decision.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: searches for ReplyKeyboardMarkup, ReplyKeyboardRemove, reply keyboard, resize_keyboard, one_time_keyboard, and remove_keyboard find no Telegram runtime/schema/docs support, while the send path visibly normalizes only inline buttons into reply_markup.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned additive runtime/schema/docs support for reply keyboards through send/message surfaces; changing the generic presentation contract is not required for this issue.
