## Summary

Keep open: current main still exposes `max` for OpenRouter DeepSeek V4 thinking and the DeepSeek V4 payload wrapper can send `reasoning_effort: "max"`, while OpenRouter documents `xhigh`, `high`, `medium`, `low`, `minimal`, and `none` as effort values. ([openrouter.ai](https://openrouter.ai/docs/guides/best-practices/reasoning-tokens?utm_source=openai))

## What This Changes

The issue asks OpenClaw to stop sending an OpenRouter-invalid `reasoning_effort` value for DeepSeek V4 Pro after the 2026.5.3-1 upgrade.

## Best Possible Solution

Keep this open and fix the OpenRouter-owned DeepSeek V4 thinking profile/wire mapping so OpenClaw only sends OpenRouter-accepted effort values while preserving reasoning replay compatibility.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the `xhigh`/`max` failure: the current wrapper emits `reasoning_effort: "max"` for OpenRouter DeepSeek V4 and an existing test asserts that payload. I did not establish a live-provider failing run from this checkout.

## Solution Assessment

Yes, the best fix is a narrow OpenRouter/DeepSeek V4 mapping correction plus regression tests for `high` and `xhigh`. Adding a new config knob or changing global thinking semantics would be broader than needed.
