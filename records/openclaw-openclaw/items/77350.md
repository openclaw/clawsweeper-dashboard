## Summary

Keep open: current main still exposes `max` for OpenRouter DeepSeek V4 thinking and the OpenRouter stream wrapper can emit `reasoning_effort: "max"`. The linked open PR #77380 uses closing syntax for this issue, so the issue should stay open until that PR is reviewed and merged or replaced.

## What This Changes

Requests a fix for OpenRouter DeepSeek V4 Pro so OpenClaw sends only OpenRouter-accepted reasoning effort values instead of the invalid `max` value after 2026.5.3-1.

## Best Possible Solution

Land a narrow OpenRouter-owned fix that removes `max` from the OpenRouter DeepSeek V4 profile, maps OpenRouter `xhigh` to the accepted wire value, and updates regression coverage plus changelog/docs if the user-facing behavior changes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the `xhigh`/`max` failure: current main advertises `max`, routes OpenRouter DeepSeek V4 through the shared wrapper, and an existing test asserts `reasoning_effort: "max"`; I did not run a live provider call.

## Solution Assessment

Yes. A provider-local OpenRouter mapping/profile correction is the narrow maintainable fix; changing global thinking semantics or adding a new config option would be broader than needed.
