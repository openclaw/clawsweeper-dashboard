## Summary

Keep open: current main still has a source-backed path that sends `reasoning_effort: "max"` for OpenRouter DeepSeek V4, while OpenRouter's chat-completions contract does not accept `max` for that field.

## What This Changes

The issue asks OpenClaw to stop failing OpenRouter `deepseek/deepseek-v4-pro` requests after the 2026.5.3-1 upgrade by sending only accepted `reasoning_effort` values.

## Best Possible Solution

Fix the OpenRouter-owned DeepSeek V4 thinking profile and wire mapping so OpenClaw sends `xhigh` rather than `max` to OpenRouter, while preserving direct DeepSeek V4 `max` behavior and reasoning replay backfill.

## Reproduction Assessment

Yes for the invalid `max` wire payload: current main emits `reasoning_effort: "max"` for OpenRouter DeepSeek V4 when the selected level is `xhigh` or `max`, and an existing test asserts that payload. The configured-`high` part of the report is not yet independently reproduced.

## Solution Assessment

Yes: a narrow OpenRouter-specific mapping correction plus regression tests is the best fit. Changing global thinking semantics would be too broad because the direct DeepSeek provider still appears to use `max` intentionally.
