## Summary

Keep open. The linked bug is real and the PR targets the right persistence path, but the current patch can create an invalid preserved model/provider pair for legacy model-only session entries and the changelog line does not satisfy the repository PR gate.

## What This Changes

The branch adds a `preserveRuntimeModel` option to session-store persistence, enables it for heartbeat runs, adds focused session-store regression tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Land a corrected narrow patch that keeps heartbeat model overrides scoped to heartbeat runs, preserves only valid pre-existing runtime identity without inventing a provider for model-only entries, leaves non-heartbeat persistence unchanged, and satisfies the active changelog gate.

## Reproduction Assessment

Yes. Configure a shared-session heartbeat model different from the primary, run a heartbeat, and inspect the session store/status path; current main persists the completed heartbeat run model into shared runtime metadata. The PR-specific bug is reproducible with an entry that has `model` but no `modelProvider` and `preserveRuntimeModel: true`.

## Solution Assessment

No, not as currently written. The preserve flag is a reasonable boundary, but the branch must avoid pairing a legacy model-only entry with the heartbeat provider and must fix the changelog line before merge.
