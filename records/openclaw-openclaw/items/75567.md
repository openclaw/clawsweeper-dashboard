## Summary

Keep open. The PR targets the real heartbeat runtime-model bleed reported in the linked issue and the changelog/security surface look fine, but the proposed preservation branch still writes heartbeat model/context for shared sessions with no prior runtime model and can pair a legacy model-only row with the heartbeat provider, so it is not mergeable as-is.

## What This Changes

This PR adds a heartbeat-only `preserveRuntimeModel` option to session-store persistence, wires it from `agent-command.ts`, adds session-store regression tests, and updates the changelog.

## Best Possible Solution

Keep `heartbeat.model` as a per-run selection, but shared-session heartbeat persistence should not create or mutate selected runtime model/context fields with the heartbeat identity. The fix should preserve a complete pre-existing runtime identity when present, leave runtime fields unset or derivable when none exists, avoid provider/model mismatches for legacy rows, and cover fresh-session, explicit-override, legacy model-only, existing-runtime, and normal non-heartbeat overwrite cases.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `updateSessionStoreAfterAgentRun` with `preserveRuntimeModel: true` and a heartbeat result on a session entry with no runtime `model`; the proposed code writes the heartbeat model/context, and existing resolution code then treats persisted runtime fields as selected state.

## Solution Assessment

No. The flag is a reasonable seam, but the proposed fallback should not persist heartbeat identity when there is no matched pre-existing runtime model, and missing providers should be resolved or left unset rather than filled from the heartbeat run.

  - confidence: 0.88

Existing model resolution treats both fields as an explicit runtime identity, so a row like `model: claude-opus` can become `ollama/claude-opus` instead of preserving or inferring the original provider.
  - confidence: 0.84
