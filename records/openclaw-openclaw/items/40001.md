## Label Justifications

- `P1`: The issue describes a live silent data-loss path in cron and ordinary agent workflows, and current main still lacks a general append primitive.
- `impact:data-loss`: Overwrite-only `write` calls can replace existing memory or workspace files without built-in recovery.
- `impact:session-state`: The affected shared memory files preserve operational and session context across runs.

## Summary

Keep open: current main and latest release still expose ordinary `write` as create/overwrite only, and https://github.com/openclaw/openclaw/pull/77127 is an open closing PR rather than a merged fix.

## What This Changes

Request an append-capable file write primitive so cron and ordinary agent sessions can add to shared memory/workspace files without truncating existing content.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77127 or a maintainer-approved equivalent append-capable write path that preserves overwrite defaults, workspace/sandbox guards, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows ordinary `write` still accepts only `path` and `content` and routes to overwrite operations, while cron runs use `trigger: "cron"` outside the memory-flush append-only wrapper.

## Solution Assessment

Yes as a direction, but not as completed main behavior. A first-class append-capable write path is the durable fix; maintainers still need to approve the exact public tool API and concurrency behavior in the linked PR.
