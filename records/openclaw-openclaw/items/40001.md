## Summary

Keep open. Current main still exposes the normal model-facing `write` tool as create/overwrite-only; the existing append path is scoped to memory-flush runs, so the reported isolated-cron shared-file data-loss class remains valid.

## What This Changes

Add a safe append capability for model-facing file writes so isolated cron jobs can accumulate shared workspace files without overwriting prior content.

## Best Possible Solution

Add a backwards-compatible append primitive at the file-tool boundary, reuse root-enforced append helpers for workspace paths, define sandbox behavior explicitly, and update cron/memory guidance plus changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an existing workspace file and invoke the normal `write` tool with new content; current main routes that call through overwrite paths, and cron-triggered runs do not activate the memory-flush append wrapper.

## Solution Assessment

Yes. Keeping this open for a narrow append-primitive fix is the best current path; prompt guards, backups, or `exec >>` reduce damage but do not remove the overwrite-only primitive that causes the data loss.

## Repair Work Prompt

Fix the model-facing file write data-loss issue tracked by #40001. Add a backwards-compatible append primitive for standard agent file writes, preferably by extending the existing `write` tool with explicit append intent such as `mode: "overwrite" | "append"` defaulting to overwrite; if the underlying schema/wrapper makes that unsafe, add a dedicated `append` file tool instead. Preserve existing `write` overwrite behavior by default. For host workspace-only paths, reuse `appendFileWithinRoot` so workspace path policy remains enforced. For unrestricted host writes, create parent directories and append without truncating. For sandboxed writes, either implement real bridge append semantics or reject append intent loudly rather than silently overwriting. Update tool catalog/parameter validation, focused tests for host/workspace/sandbox/memory-flush interactions, cron or tools docs that describe append usage/workarounds, and add a user-facing CHANGELOG entry. Do not weaken path guards, do not document `exec >>` as the preferred long-term answer, and do not change unrelated cron delivery behavior.
