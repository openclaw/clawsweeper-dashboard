- `impact:data-loss`: Overwrite-only `write` calls can replace existing memory or log files without a built-in recovery path.

## Summary

Keep this open: current main still exposes ordinary agent and cron `write` as overwrite-only, while the only append-only behavior is scoped to memory-flush runs and an open linked PR already owns the implementation path.

## What This Changes

The issue requests a safe append-capable file-writing primitive because overwrite-style agent writes can silently destroy shared cron or memory files.

## Best Possible Solution

Review and land a maintainer-approved append-capable file primitive that preserves overwrite as the default, keeps workspace and sandbox guards, serializes same-file mutations, and then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Current main is source-reproducible: ordinary host and sandbox write paths still replace file contents with only `path` and `content`, while append-only behavior is gated to memory-flush runs.

## Solution Assessment

Yes as a direction, but not yet as completed repository state. A first-class append-capable write path is the narrow durable fix; maintainers still need to approve the exact public API shape in the open linked PR.
