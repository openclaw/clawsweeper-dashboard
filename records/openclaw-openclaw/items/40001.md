## Summary

Keep open. Current main and v2026.6.6 still expose the normal agent `write` tool as create/overwrite only, while the existing append-only wrapper is scoped to memory-flush runs; the open linked PR is the active fix path but has not merged.

## What This Changes

The issue asks for a safe append-capable file write path so agent and isolated cron sessions do not silently overwrite shared workspace memory files.

## Best Possible Solution

Land a maintainer-approved append path for the existing write-tool surface, keeping overwrite as the default and preserving host, workspace, sandbox, and validation behavior.

## Reproduction Assessment

Yes. Source inspection shows the normal `write` tool only accepts `path` and `content` and delegates to overwrite-style write operations; issue comments add multiple real cron and single-agent incidents, but I did not run a mutating local repro because this review is read-only.

## Solution Assessment

Yes for keeping the issue open. The linked append-mode PR is the right review target because prompt guards and `exec >>` workarounds do not preserve the write tool's workspace policy boundary, but maintainers still need to approve the exact API shape.
