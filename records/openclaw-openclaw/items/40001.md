## Summary

Keep open: current main and the latest release still expose ordinary `write` as path/content overwrite only, while the append-only behavior on main is scoped to memory-flush runs. The open linked implementation PR is the right current path to review, so this issue should stay open until that PR lands or maintainers choose a replacement API shape.

## What This Changes

The issue requests a first-class append-capable write path so cron and agent sessions can add to shared workspace files without truncating existing memory or log content.

## Best Possible Solution

Keep this issue open as the canonical tracker until https://github.com/openclaw/openclaw/pull/77127 lands or maintainers choose a replacement append API that preserves overwrite-by-default behavior and existing workspace/sandbox guards.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 expose only path/content overwrite semantics for ordinary `write`, while cron runs use `trigger: "cron"` and do not enter the memory-flush append-only wrapper; public comments add concrete cron and single-agent truncation examples.

## Solution Assessment

Yes, pending maintainer API acceptance. A first-class append primitive is the maintainable direction compared with prompt guards or `exec >>` workarounds, but maintainers still need to choose whether the permanent contract is `append?: boolean`, a mode field, or a separate tool.
