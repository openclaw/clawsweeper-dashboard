## Summary

Keep open. Current main still exposes the normal `write` tool as overwrite-only for ordinary and isolated cron runs; the keepmyclaw ClawHub backup skill is a useful mitigation, but it does not fix the core file-tool contract, and the directly related append-mode PR is closed unmerged.

## What This Changes

Request a safe append-capable file primitive or write mode so isolated cron sessions can accumulate shared files without destructive overwrites.

## Best Possible Solution

Add a backwards-compatible append primitive at the OpenClaw file-tool boundary, reuse the existing safe append helper for workspace paths, keep overwrite as the default, and update cron/tool guidance so agents no longer emulate append with read-concatenate-overwrite.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create an existing file, invoke the normal model-facing `write` tool with shorter content, and current main routes ordinary and isolated cron writes through overwrite paths rather than the memory-flush append wrapper.

## Solution Assessment

Yes. Keeping this issue open for a first-class append primitive is the narrow maintainable fix; prompt guards, backup skills, and `exec >>` redirects reduce damage but do not fix the overwrite-only tool contract or preserve path-policy boundaries.
