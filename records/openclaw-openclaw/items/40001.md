## Label Justifications

- `P1`: The issue describes repeated silent data loss in real agent and cron workflows, and current main still lacks a general append primitive.
- `impact:data-loss`: Overwrite-only `write` calls can replace existing memory or workspace files without built-in recovery.
- `impact:session-state`: The affected shared memory files preserve operational context and session continuity across agent runs.

## Summary

Keep open: current main and the latest release still expose ordinary `write` as overwrite-only outside the memory-flush-specific append wrapper, and the open implementation path is still https://github.com/openclaw/openclaw/pull/77127.

## What This Changes

The issue requests a safe append-capable write primitive so agent and cron writes do not overwrite shared memory or workspace files.

## Best Possible Solution

Land or replace the open append-mode PR with a maintainer-approved write path that preserves overwrite defaults, workspace and sandbox guards, mutation-queue behavior, docs, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current ordinary `write` still routes to overwrite operations with only `path` and `content`, and recent comments provide production byte-shrink examples of the same failure mode.

## Solution Assessment

Yes as a direction, but not as completed main behavior. A first-class append-capable write path is the durable fix; maintainers still need to approve the exact model-facing API and concurrency behavior.
