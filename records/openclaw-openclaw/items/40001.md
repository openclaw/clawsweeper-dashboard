## Label Justifications

- `P1`: The issue describes a live silent data-loss path in cron and ordinary agent workflows, and current main still lacks a general append primitive.
- `impact:data-loss`: Overwrite-only `write` calls can replace existing memory or workspace files without built-in recovery.
- `impact:session-state`: The affected shared memory files preserve operational and session context across runs.

## Summary

Keep open: current main and the latest release still expose ordinary `write` as create/overwrite only, and the issue has fresh production data-loss evidence plus an open linked PR with closing syntax, so it should remain open until that PR or an equivalent append-capable contract lands.

## What This Changes

The issue requests an append-capable write path so cron and ordinary agent sessions can add to shared workspace memory files without destructive overwrite.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77127 or a maintainer-approved equivalent append-capable write contract that preserves overwrite defaults, workspace/sandbox guards, queued mutation behavior, docs, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows ordinary `write` still accepts only `path` and `content` and routes to overwrite operations, while cron runs use `trigger: "cron"` outside the memory-flush append-only wrapper.

## Solution Assessment

Yes as a direction, but not as completed main behavior. A first-class append-capable write path is the durable fix; maintainers still need to approve the exact public tool API and concurrency behavior in the linked PR.
