## Summary

Keep this issue open: current main and v2026.6.10 still expose ordinary `write` as overwrite-only, and the open closing PR is the active append-mode implementation path rather than a reason to close before merge.

## What This Changes

The issue requests an append-safe file write primitive because overwrite-only agent writes can truncate shared memory files used by cron and long-running sessions.

## Best Possible Solution

Land a narrow append-capable write contract through the open linked PR or a direct successor while preserving overwrite as the default and keeping workspace/sandbox guards intact.

## Reproduction Assessment

Yes. Current main and v2026.6.10 define ordinary `write` as `path` plus `content` and execute overwrite-style `writeFile`; only the memory-flush wrapper appends.

## Solution Assessment

Yes as a direction, but not as completed work. A first-class append mode is the maintainable boundary because prompt-only guards and read-concatenate-overwrite flows are racy; the linked PR should be reviewed, landed, or explicitly replaced.
