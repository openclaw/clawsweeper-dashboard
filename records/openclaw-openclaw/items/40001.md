## Summary

Keep this issue open: current main and the latest release still define ordinary `write` as overwrite-only, only the memory-flush wrapper has scoped append behavior, and the open linked append-mode PR is the active candidate fix rather than a reason to close early.

## What This Changes

Request first-class append semantics for the agent `write` tool so isolated cron and agent sessions can add to shared memory/workspace files without overwriting existing content.

## Best Possible Solution

Review and land a narrow append-capable write contract through https://github.com/openclaw/openclaw/pull/77127 or a direct successor, preserving overwrite as the default and keeping workspace/sandbox guards intact.

## Reproduction Assessment

Yes. Current main and v2026.6.10 define ordinary `write` as `path` plus `content` and execute overwrite-style `writeFile`; the issue and comments provide concrete cron and single-agent overwrite incidents.

## Solution Assessment

Yes as a direction, but not as completed work. A first-class append mode or append tool is the maintainable boundary because prompt-only guards and read-concatenate-write flows remain racy; the linked PR should be reviewed, landed, or explicitly replaced.
