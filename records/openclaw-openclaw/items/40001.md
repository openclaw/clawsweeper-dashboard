## Label Justifications

- `P1`: The report describes a real broken cron/agent workflow that can silently truncate shared workspace and memory files for current users.
- `impact:data-loss`: The central failure mode is existing file content being overwritten when the user expected accumulation.
- `impact:session-state`: The affected files include memory and operational context artifacts that preserve agent continuity across sessions and scheduled runs.

## Summary

Keep open. Current main still exposes the ordinary write tool as create-or-overwrite only, the append-only wrapper is scoped to memory-flush runs, and the open linked implementation has not merged yet.

## What This Changes

Request a first-class append mode or append tool for agent file writes so cron and ordinary sessions can accumulate shared memory/workspace files without truncating existing content.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/77127 or choose a different public write-tool API; after the chosen append primitive lands with host/workspace/sandbox coverage, close this issue as implemented.

## Reproduction Assessment

Yes, at source level. Current main still routes ordinary write calls through path/content-only overwrite implementations, and cron-triggered runs do not activate the memory-flush append-only wrapper; I did not run a live mutating cron repro in this read-only review.

## Solution Assessment

Yes on the direction, but not yet on the exact API. A guarded first-class append primitive belongs at the write-tool boundary; maintainers still need to confirm whether https://github.com/openclaw/openclaw/pull/77127's optional append field is the preferred shape or whether a dedicated append tool is better.
