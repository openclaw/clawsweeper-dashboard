## Label Justifications

- `P1`: The issue reports a real broken agent/cron workflow where shared memory files can be silently overwritten.
- `impact:data-loss`: The core failure mode is truncating existing workspace or memory files with no built-in recovery.
- `impact:session-state`: The affected files hold persistent agent memory and operational context across sessions.

## Summary

Keep open. Current main and the latest release still expose ordinary `write` as overwrite-only, while https://github.com/openclaw/openclaw/pull/77127 is the open implementation candidate that uses closing syntax for this issue.

## What This Changes

The issue asks for a first-class append path for the agent write tool so isolated cron and other sessions do not silently overwrite shared memory/workspace files.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77127, or explicitly choose a different append-tool API, while preserving overwrite as the default and keeping workspace/sandbox guards intact.

## Reproduction Assessment

Yes. Source inspection shows ordinary `write` still accepts only `path` and `content` and calls the overwrite operation, while isolated cron runs use `trigger: "cron"` rather than the memory-flush append wrapper.

## Solution Assessment

Yes for the direction, pending maintainer API acceptance. A first-class append primitive is safer than prompt guards or `exec >>` because it can preserve OpenClaw path guards and mutation serialization; the exact public API shape remains the decision.
