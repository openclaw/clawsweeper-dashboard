## Label Justifications

- `P1`: The issue reports recurring user-facing data loss in ordinary agent and cron workflows, with an open fix path but no landed general append primitive yet.
- `impact:data-loss`: The failure mode silently replaces existing workspace and memory files with only the latest write content.
- `impact:session-state`: The affected shared memory and daily journal files carry long-lived agent/session context that can be corrupted or lost across isolated runs.

## Summary

Keep open: current main and the latest release still expose ordinary agent `write` as overwrite-only, while the linked append-mode PR remains open and owns the proposed fix path.

## What This Changes

The issue requests an append-capable write primitive so agents and cron jobs can add to shared workspace memory or log files without truncating existing content.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77127 if maintainers accept the append API, or replace it with an equivalent explicit append primitive that preserves overwrite defaults, workspace/sandbox guards, serialization, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows ordinary `write` accepts only path/content and calls the overwrite path, while isolated cron uses `trigger: "cron"` and does not receive the memory-flush append wrapper.

## Solution Assessment

Mostly yes. A first-class append primitive is safer than prompt guards or `exec >>`, but maintainers still need to choose the exact public write-tool contract before merge.
