## Label Justifications

- `P1`: Multiple reports describe real cron and memory workflows silently truncating durable files, and the issue affects a core agent/channel workflow despite an open fix candidate.
- `impact:data-loss`: The reported behavior can replace existing workspace memory files with shorter content and lose prior notes without built-in recovery.
- `impact:session-state`: The affected files include durable memory and operational context used to preserve agent continuity across sessions and cron runs.

## Summary

Keep this issue open: current main and the latest release still expose ordinary `write` as overwrite-only, and the open linked PR is the active implementation candidate rather than a reason to close early.

## What This Changes

The issue requests a first-class append mode or append primitive for agent file writes so cron and ordinary agent sessions do not overwrite shared memory files.

## Best Possible Solution

Review and land, revise, or explicitly close https://github.com/openclaw/openclaw/pull/77127; the end state should keep overwrite as the default while adding a guarded first-class append path.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.8` expose ordinary `write` as `path` plus `content` and route it through overwrite operations; I did not run a mutating live Gateway reproduction in this read-only review.

## Solution Assessment

Yes on direction, with one product decision still open. A first-class append primitive at the write-tool boundary is narrower and safer than prompt guards or `exec >>`, but maintainers still need to accept the exact public API shape through the linked PR.
