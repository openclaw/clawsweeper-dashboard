## Label Justifications

- `P1`: The issue describes repeated silent data loss in real agent and cron workflows, and current main still lacks a general append primitive.
- `impact:data-loss`: Overwrite-only `write` calls can replace existing memory or workspace file contents without built-in recovery.
- `impact:session-state`: The affected memory files preserve operational context and session continuity across agent runs.

## Summary

Keep open: current main and v2026.5.20 still expose ordinary `write` as path/content overwrite-only, append behavior is scoped to memory-flush runs, and https://github.com/openclaw/openclaw/pull/77127 remains the open closing implementation path.

## What This Changes

The issue requests a first-class append-capable file write primitive so cron and agent sessions can add to shared memory/workspace files without truncating existing content.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/77127 with a maintainer-approved append-capable write path that preserves overwrite defaults, workspace/sandbox guards, mutation-queue behavior, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows ordinary `write` still accepts only `path` and `content` and routes to overwrite operations, while issue comments provide production byte-shrink examples of the same failure mode.

## Solution Assessment

Yes as a direction, but not as completed main behavior. A first-class append-capable write path is the durable fix; maintainers still need to approve the exact model-facing API and concurrency contract.
