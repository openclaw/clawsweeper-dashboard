## Label Justifications

- `P1`: The issue describes repeated silent data loss in real agent and cron workflows, and current main still lacks a general append primitive.
- `impact:data-loss`: Overwrite-only `write` calls can replace existing memory or workspace files without built-in recovery.
- `impact:session-state`: The affected shared memory files preserve operational context and session continuity across agent runs.

## Summary

Keep open: current main still exposes ordinary `write` as overwrite-only outside the memory-flush-specific wrapper, and the linked append-mode implementation at https://github.com/openclaw/openclaw/pull/77127 is still open rather than merged.

## What This Changes

This issue requests a safe append-capable file-write primitive because ordinary agent and cron writes can replace shared memory/workspace files.

## Best Possible Solution

Land or replace the open append-mode PR with a backwards-compatible write path that preserves overwrite defaults, workspace and sandbox guards, mutation-queue behavior, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current ordinary `write` exposes only `path` and `content` and routes to replacement writes, while cron runs use `trigger: "cron"` and do not enter the memory-flush append-only wrapper.

## Solution Assessment

Yes as a direction, but not as completed main behavior. A first-class append-capable write path is the durable fix; maintainers still need to approve the exact public tool API and concurrency behavior.
