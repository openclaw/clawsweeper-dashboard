## Summary

Keep open. Current main still exposes `write` as create/overwrite, while append behavior is limited to the memory-flush wrapper; the keepmyclaw backup skill and `exec >>` workaround are useful mitigations, not a core file-tool fix.

## What This Changes

The issue requests an append-safe file write primitive for isolated cron and general agent sessions so shared files are not silently overwritten.

## Best Possible Solution

Ship a backward-compatible append capability at the file-tool boundary using the existing workspace-safe append helper, with cron/memory docs and regression tests; keep backup skills as mitigation rather than the primary fix.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the core issue: ordinary `write` still routes through overwrite/truncate paths, while append behavior is gated to memory-flush runs only.

## Solution Assessment

Yes for the direction: a tool-layer append primitive is the maintainable fix. The exact public API shape still needs maintainer choice before implementation.
