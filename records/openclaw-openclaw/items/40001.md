## Label Justifications

- `P1`: The issue reports a broken agent/cron file-writing workflow that can silently overwrite user memory files.
- `impact:data-loss`: The central failure mode is loss of existing workspace memory file content.
- `impact:session-state`: The affected files carry persistent agent memory and operational context across sessions.

## Summary

Keep open: current main and the latest release still expose the general `write` tool as create/overwrite only. OpenClaw has a memory-flush-only append wrapper, but ordinary agent and cron writes still lack the requested append primitive, and the open linked PR is the active fix path.

## What This Changes

Request a first-class append path for agent file writes so shared memory files are not destroyed by overwrite-style `write` calls.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77127, or replace it with an equivalent maintainer-approved core write-tool API that preserves workspace guards and serialized file mutation behavior.

## Reproduction Assessment

Yes, at source level. Current main exposes only `path` and `content` for `write`, overwrites through `ops.writeFile`, and limits append behavior to memory-flush maintenance turns; I did not run mutating tests because this review is read-only.

## Solution Assessment

Yes in direction, but the exact shape needs maintainer review. A first-class append primitive fixes the model-facing gap better than prompt guards or `exec >>`, which either remain probabilistic or bypass the write tool's workspace boundary.
