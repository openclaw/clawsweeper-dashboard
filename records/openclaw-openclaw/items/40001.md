## Label Justifications

- `P1`: The issue describes a live silent data-loss path in real cron and agent workflows, and current main still lacks a general append primitive.
- `impact:data-loss`: Overwrite-only `write` calls can replace existing memory or log files without built-in recovery.
- `impact:session-state`: The affected shared memory files preserve operational/session context across runs.

## Summary

Keep this open: current main still leaves ordinary agent and cron `write` calls as overwrite-only, and the append-mode PR that uses closing syntax for this issue is open rather than merged.

## What This Changes

The issue requests an append-capable file write primitive so isolated cron and agent sessions can add to shared workspace files without silently overwriting existing content.

## Best Possible Solution

Land a maintainer-approved append-capable write path that preserves default overwrite behavior, keeps workspace and sandbox guards, serializes same-file mutations where practical, and updates focused tests and docs.

## Reproduction Assessment

Yes. The current source shows ordinary `write` accepts only `path` and `content` and routes to overwrite operations, while cron runs use `trigger: "cron"` and do not activate the memory-flush append wrapper.

## Solution Assessment

Yes as a direction, but not as completed main behavior. A first-class append-capable write path is the durable fix; maintainers still need to approve the exact public tool API in the linked PR.
