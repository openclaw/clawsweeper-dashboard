- `impact:data-loss`: The concrete failure mode is truncation of existing daily memory files to only the latest write content.

## Summary

Keep this canonical issue open: current main and the latest release still expose ordinary `write` as overwrite-only, while the linked append-mode pull request is open and explicitly references this issue as its fix path.

## What This Changes

Request an append-capable write primitive so ordinary agent and isolated cron writes can append to shared workspace memory files without truncating existing content.

## Best Possible Solution

Keep this canonical issue open while maintainers review https://github.com/openclaw/openclaw/pull/77127; the desired end state is an append primitive that preserves overwrite as the default, keeps workspace and sandbox guards intact, serializes same-file appends, and has focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 ordinary write tooling accept only `path` and `content` and call overwrite-style write operations, while the issue discussion adds multiple production truncation reports.

## Solution Assessment

Mostly yes. A first-class append primitive is the right layer because prompt guards and read-concatenate-write emulation remain racy, but the exact public API shape needs maintainer acceptance before the linked PR lands.
