## Label Justifications

- `P1`: The issue describes repeated silent data loss in real agent and cron workflows, and current main still lacks a general append primitive.
- `impact:data-loss`: Overwrite-only `write` calls can replace existing memory or workspace files without built-in recovery.
- `impact:session-state`: The affected shared memory files preserve operational context and session continuity across agent runs.

## Summary

Keep open: current main still exposes the general `write` tool as overwrite-only outside the memory-flush-specific append wrapper, and this issue is paired with an open closing pull request at https://github.com/openclaw/openclaw/pull/77127.

## What This Changes

Request to add an append-capable file-write primitive so agent and cron writes can add to shared memory/workspace files without truncating existing content.

## Best Possible Solution

Expose a maintainer-approved append-capable write primitive that preserves the overwrite default, workspace and sandbox path guards, mutation-queue behavior, docs, and focused regression coverage; then close this after the linked PR or replacement lands.

## Reproduction Assessment

Yes. Source inspection on current main shows ordinary `write` still accepts only `path` and `content` and routes to overwrite operations, while multiple comments provide production byte-shrink examples of the same failure mode.

## Solution Assessment

Yes as a direction, but not as completed main behavior. A first-class append-capable write path is the durable fix; maintainers still need to approve the exact model-facing API and concurrency behavior.
