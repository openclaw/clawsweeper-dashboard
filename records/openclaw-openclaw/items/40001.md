## Summary

Keep open. Current main still exposes the general model-facing `write` tool as create/overwrite only; append behavior exists only in the memory-flush wrapper, so the keepmyclaw backup skill and `exec >>` workaround are useful mitigations but not a core file-tool fix.

## What This Changes

Request a safe append capability for the model-facing file write path so isolated cron sessions can accumulate shared workspace logs without overwriting existing content.

## Best Possible Solution

Define and ship a backward-compatible model-facing append primitive at the file-tool boundary, preserving workspace path enforcement and documenting cron/memory append guidance.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: an existing file written through the ordinary `write` tool is routed to overwrite code paths, while append-only behavior is gated to memory-flush runs only. I did not execute a live cron reproduction because this review was read-only.

## Solution Assessment

Yes, the requested tool-layer append primitive is the right direction; prompt guards, backup skills, and shell redirection reduce damage but do not fix the overwrite-only contract or preserve workspace path policy. The exact public API shape still needs maintainer choice.
