## Summary

This issue should stay open: current main and the latest release still expose ordinary `write` as create-or-overwrite, while the append-mode fix is tracked by an open unmerged PR. The current memory-flush wrapper only covers the special memory trigger path, so the reported cron and normal-agent data-loss path still has no merged core fix.

## What This Changes

The issue requests a first-class append path for the agent `write` tool so shared memory and workspace files are not silently truncated by isolated cron or ordinary agent sessions.

## Best Possible Solution

Review and land, or intentionally replace, https://github.com/openclaw/openclaw/pull/77127 with a maintainer-approved append path that preserves overwrite defaults, workspace/sandbox guards, same-file serialization, and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows ordinary `write` still accepts only `path` and `content` and calls the overwrite write path; multiple recent comments add production byte-shrinkage evidence, but I did not run a live cron session because this review is read-only.

## Solution Assessment

Yes in direction: a first-class append path at the write-tool boundary is the maintainable fix because prompt guards and read-concatenate-write workarounds remain racy. The exact API shape still needs maintainer approval through the linked PR.
