## Summary

Keep this open as the canonical file-tool contract issue. Current main still exposes general `write` as create-or-overwrite, while append-safe behavior is only wired for memory-flush runs; the latest comments and closed duplicate/PR context show the cron/general-agent data-loss case remains unresolved.

## What This Changes

Request a general append-capable file primitive for agent and cron writes so shared workspace files can accumulate content without destructive `write` overwrites.

## Best Possible Solution

Keep this canonical issue open, choose a public append-capable file-tool surface that preserves workspace path policy, then land a narrow implementation with docs, changelog, and cron/general-write regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main's general `write` path overwrites existing content and only memory-flush-triggered runs get append-only wrapping. I did not run a live cron because the review was read-only.

## Solution Assessment

Yes on direction, unclear on exact API shape. A public append-capable file primitive is the maintainable fix, but maintainers still need to choose between an `append` tool, a `write` mode, or a boolean write parameter before automation should implement it.
