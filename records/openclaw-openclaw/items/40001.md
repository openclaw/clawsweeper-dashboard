## Label Justifications

- `P1`: The issue reports active silent overwrites of shared memory/workspace files in cron and agent workflows, affecting real users now.
- `impact:data-loss`: The central failure mode is truncating existing shared files such as daily memory logs without backup or recovery.
- `impact:session-state`: The affected files are used for durable memory and cross-session operational context, so overwrites cause agent/session state drift.

## Summary

Keep this issue open: current main and v2026.6.8 still expose ordinary `write` as overwrite-only, while https://github.com/openclaw/openclaw/pull/77127 is the open closing PR for the append-capable fix.

## What This Changes

The issue asks for append-safe write behavior so isolated cron and agent sessions can add to shared memory/workspace files without overwriting existing content.

## Best Possible Solution

Land a maintainer-approved append-capable write contract through https://github.com/openclaw/openclaw/pull/77127 or a direct successor, preserving workspace guards, sandbox behavior, mutation ordering, and the existing memory-flush append-only wrapper.

## Reproduction Assessment

Yes, at source level. Current main's ordinary write tool accepts only `path` and `content`, describes create/overwrite behavior, and calls `ops.writeFile`; the issue comments add multiple production reports of append-intended writes truncating existing memory files.

## Solution Assessment

Yes as a product direction, but not yet as a completed fix. A first-class append path is the maintainable boundary because prompt-only guards and read-concatenate-overwrite flows are racy; the open PR still needs maintainer API/recovery review before this issue can close.
