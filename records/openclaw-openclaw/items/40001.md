## Label Justifications

- `P1`: The issue reports repeated silent overwrites of durable memory and workspace files in real cron and agent workflows, and current source still lacks general append support.
- `impact:data-loss`: The central failure mode is truncating or replacing existing user memory and workspace files with new write content.
- `impact:session-state`: The affected files include durable memory and operational context that sessions rely on across cron and main-session boundaries.

## Summary

Keep open: current main and the latest release still expose ordinary `write` as create/overwrite only, while the linked open PR is the active fix candidate and should be reviewed before this issue closes.

## What This Changes

The issue requests a safe append-capable write primitive so cron and ordinary agent sessions do not overwrite shared memory or workspace files when they intend to append.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/77127 with a backwards-compatible append path that keeps overwrite as the default and preserves workspace and sandbox guards.

## Reproduction Assessment

Yes. Source inspection shows the ordinary `write` tool accepts only `path` and `content` and routes through overwrite-style file writes, while append-only behavior is limited to the special memory-flush wrapper.

## Solution Assessment

Mostly yes. The write-tool boundary is the right layer for a durable fix, but maintainers still need to confirm whether the permanent API should be `append: true`, a mode field, or a separate append tool.
