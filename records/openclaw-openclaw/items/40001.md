## Summary

Keep this issue open: current `main` and `v2026.6.8` still expose ordinary `write` as overwrite-only, while https://github.com/openclaw/openclaw/pull/77127 is the open closing PR that owns the active append-mode fix path.

## What This Changes

The issue requests a safe append-capable agent file-write primitive so shared memory/workspace files are not truncated by overwrite-only `write` calls.

## Best Possible Solution

Review and land, revise, or explicitly reject https://github.com/openclaw/openclaw/pull/77127; the target end state should keep overwrite as the default while providing a first-class append primitive that preserves host, workspace, sandbox, and SDK contracts.

## Reproduction Assessment

Yes, at source level. Current `main` and `v2026.6.8` expose ordinary `write` as `path` plus `content` and route it through overwrite operations; I did not run a mutating live Gateway/cron reproduction in this read-only review.

## Solution Assessment

Yes on direction, with maintainer acceptance still needed for the exact API. A first-class append primitive at the write-tool boundary is the maintainable fix because prompt guards and `exec >>` workarounds either fail probabilistically or bypass the workspace path-policy surface.
