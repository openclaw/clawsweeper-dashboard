- `P1`: The issue reports active silent overwrites of shared memory/workspace files in cron and agent workflows affecting real users now.
- `impact:data-loss`: The central failure mode is truncating existing shared files such as daily memory logs without built-in backup or recovery.

## Summary

Keep this issue open: current main and v2026.6.10 still expose ordinary `write` as overwrite-only, and https://github.com/openclaw/openclaw/pull/77127 is the open closing PR for the append-capable fix.

## What This Changes

The issue requests a first-class append-capable write path so cron and agent sessions can add to shared memory/workspace files without truncating existing content.

## Best Possible Solution

Land a maintainer-approved append-capable write contract through https://github.com/openclaw/openclaw/pull/77127 or a direct successor while preserving workspace guards, sandbox behavior, mutation ordering, and the memory-flush wrapper.

## Reproduction Assessment

Yes, at source level. Current main's ordinary write tool accepts only `path` and `content`, describes create/overwrite behavior, and calls `ops.writeFile`; the issue thread adds multiple production reports of append-intended writes truncating existing memory files.

## Solution Assessment

Yes as a direction, but not yet as a completed fix. A first-class append path is the maintainable boundary because prompt-only guards and read-concatenate-overwrite flows are racy; the linked PR should be reviewed, landed, or explicitly replaced rather than queueing a competing fix.
