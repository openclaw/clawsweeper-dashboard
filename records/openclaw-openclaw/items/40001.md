## Summary

Keep open. Current main still exposes normal model-facing `write` as create/overwrite-only for ordinary and isolated cron runs; the append behavior present today is scoped to memory-flush runs for one configured memory file, so it does not resolve the reported shared-file data-loss class.

## What This Changes

The issue requests a safe append-capable file write primitive or equivalent guardrails so isolated cron sessions cannot clobber shared workspace files when they intended to add content.

## Best Possible Solution

Add a backwards-compatible append primitive at the OpenClaw file-tool boundary, either an explicit `write` append mode or a dedicated append file tool, using existing safe append helpers where possible and updating cron/memory guidance so agents no longer emulate append with read-concatenate-overwrite.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create an existing workspace file, invoke the normal model-facing `write` tool with shorter content, and current main routes through overwrite paths; isolated cron runs use `trigger: "cron"`, which does not activate the memory-flush append wrapper.

## Solution Assessment

Yes. Keeping this open for a narrow append-primitive fix is the best current path; prompt warnings and backup skills reduce damage but do not remove the overwrite-only primitive that causes the data loss.
