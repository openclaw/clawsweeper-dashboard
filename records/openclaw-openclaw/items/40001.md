## Summary

Keep this open. Current main still exposes general `write` as create-or-overwrite, while the only append-safe wrapper is scoped to memory-flush runs; the newer comments and the closed duplicate add useful production evidence that this remains the canonical core/tool-contract issue.

## What This Changes

The issue requests an append-safe file-writing primitive or equivalent guardrails so isolated cron sessions do not overwrite shared workspace memory/log files.

## Best Possible Solution

Keep this canonical issue open, choose a public append-capable file primitive that preserves workspace path policy, then update docs, changelog, and regression coverage for cron/general agent writes.

## Reproduction Assessment

Yes, from source inspection: current main's general `write` path overwrites existing content and only memory-flush-triggered runs get append-only wrapping. I did not run a live cron because the review was read-only.

## Solution Assessment

No. The current memory-flush-only wrapper is useful but does not solve isolated cron or general agent append-intent writes; a maintainer-approved public append surface is the narrower maintainable fix.
