## Summary

Keep open. Current main and v2026.5.3 still acknowledge Telegram inline-button callbacks without visible text, and open PR #76975 already targets this issue with closing syntax, so the issue should wait for that PR to be reviewed and either landed or closed.

## What This Changes

Request optional Telegram callback acknowledgement feedback for inline keyboard clicks, such as configurable `answerCallbackQuery` text/showAlert or pre-editing the message before model work.

## Best Possible Solution

Review and land or close PR #76975; the durable end state is an opt-in Telegram plugin setting with schema, types, docs, and regression coverage, while #42156 remains the narrower timing follow-up.

## Reproduction Assessment

Yes, at source level. Any Telegram inline-button callback reaching the current handler is acknowledged with no `text` payload, and the upstream type contract says omitted text shows nothing to the user; I did not run a live Telegram client check.

## Solution Assessment

Unclear until maintainer review. The linked PR's opt-in config path is a narrow maintainable approach, but the user-visible default/copy and toast-versus-alert policy still need approval before the issue is resolved.
