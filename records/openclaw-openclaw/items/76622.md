## Summary

Keep open. Current main still acknowledges Telegram callback queries without a text/options payload, and an open focused PR already references this issue with closing syntax, so this should remain open until that PR lands or maintainers choose a different Telegram UX policy.

## What This Changes

Request immediate user-visible Telegram inline-button acknowledgement, likely through configurable callback acknowledgement text/showAlert or pre-editing the clicked message.

## Best Possible Solution

Review and land #76975 if its opt-in Telegram acknowledgement config is accepted, or close that PR with an explicit alternative UX decision while keeping this issue open until the path resolves.

## Reproduction Assessment

Yes. A source-level reproduction is any Telegram inline-button callback reaching `bot.on("callback_query")` on current main: it calls `ctx.answerCallbackQuery()` or `bot.api.answerCallbackQuery(callback.id)` without a text/options payload before routing work.

## Solution Assessment

Unclear. The opt-in config shape in #76975 is a plausible narrow implementation, but maintainers still need to decide whether configurable toast text/showAlert is the desired Telegram UX versus pre-editing, default copy, or timing work in #42156.
