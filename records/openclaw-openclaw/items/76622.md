## Summary

Keep open. Current main still acknowledges Telegram inline-button callbacks without a text/options payload, and open PR #76975 is explicitly paired with this issue via closing syntax, so this should stay open until that PR is reviewed and either landed or rejected.

## What This Changes

The issue requests immediate visible feedback for Telegram inline keyboard callbacks, either through configurable callback acknowledgement text or by pre-editing the message before model handling finishes.

## Best Possible Solution

Review PR #76975 as the focused implementation path; if accepted, keep the change in the Telegram plugin with schema, type, docs, and regression coverage rather than adding core Telegram-specific policy.

## Reproduction Assessment

Yes at source level. Any Telegram `callback_query` reaching the current handler is acknowledged with `ctx.answerCallbackQuery()` or `bot.api.answerCallbackQuery(callback.id)` and no text/options payload; I did not run a live Telegram client check.

## Solution Assessment

Unclear until PR #76975 is reviewed. Its opt-in `callbackAck` approach is narrow and preserves existing defaults, but the new user-visible Telegram UX/config surface needs maintainer approval.
