## Summary

Keep open. The report is fresh, concrete, and still matches a risky current-main path: the Gateway `update.run` handler runs package-manager updates from the live Gateway request path, then schedules restart only after the package swap. Current main has useful mitigations for clean staged installs, dist inventory verification, and the separate `openclaw update` CLI guard, but those do not prove the macOS LaunchAgent `update.run` TUI hang and missing Control UI chunk regression is fixed. Related updater-safety work is active in #75691 and #75729, so this should stay open for maintainer lifecycle review rather than be closed or sent to a duplicate repair lane.

## What This Changes

The issue asks OpenClaw to fix a macOS npm-global LaunchAgent `update.run` regression where upgrading to v2026.4.29 leaves TUI unable to connect because the Gateway reports missing hashed Control UI dist chunks and WebSocket handshake timeouts.

## Best Possible Solution

Make package-manager `update.run` fail-safe for managed Gateway installs: either hand off to an external updater that stops or exits the managed Gateway before replacing the package tree, or refuse live package-manager swaps with clear `openclaw update` and recovery guidance. The accepted path should verify packaged dist inventory, restarted `/health`, and `openclaw tui` connectivity for the npm-global macOS LaunchAgent artifact.

## Reproduction Assessment

Yes. The issue gives high-confidence reproduction steps and logs: upgrade a macOS npm-global LaunchAgent install from 2026.4.11 to 2026.4.29 via Gateway `update.run`, then run `openclaw tui` and inspect Gateway logs for the missing hashed Control UI chunk plus WebSocket handshake timeout.

## Solution Assessment

No, not as a standalone pre-clean. Current main already has staged package replacement and dist inventory verification, so the safer solution is to remove or refuse the live-Gateway package swap boundary and validate the restarted packaged Gateway end to end.
