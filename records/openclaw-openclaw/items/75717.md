## Summary

Keep open. The report is detailed, recent, and names a latest-release upgrade path that can still fail before current updater hardening gets a chance to protect the install. Current main and v2026.4.29 have useful stale-dist pruning and staged npm install verification, but the reported path starts from v2026.4.11, whose updater performs an in-place global install and only verifies a narrow runtime-sidecar set before restart.

## What This Changes

The issue reports that upgrading from OpenClaw 2026.4.11 to 2026.4.29 via `openclaw gateway update.run` can leave hashed Control UI dist assets inconsistent, causing gateway health failures and TUI WebSocket handshake timeouts.

## Best Possible Solution

Keep this open and reproduce the exact `2026.4.11 -> 2026.4.29/current` npm-global update in a disposable install prefix. The maintainable fix should make the pre-restart path fail closed when any `dist/postinstall-inventory.json` entry is missing, preserve the staged-install cleanup for newer updaters, and provide a clear recovery path through reinstall or doctor instead of restarting a gateway with missing hashed chunks.

## Reproduction Assessment

Yes, the issue gives concrete version, install, OS, gateway, and log details, and the source comparison identifies the old updater path that lacks the later staged full-dist verification. This review did not run the macOS global install live because the task is read-only.

## Solution Assessment

No, the existing main/v2026.4.29 hardening is not enough to confidently solve this report because it protects future staged npm updates and prunes stale files, while this failure starts from a pre-hardening updater and reports a missing current chunk after restart. A safer fix is a package-level or pre-restart inventory-existence check that old updaters can trigger through the target package lifecycle.
