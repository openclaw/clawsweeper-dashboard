## Summary

Keep open. The report is fresh, concrete, and targets the latest shipped release; current main still performs `update.run` package-manager work inside the live Gateway process and only schedules the update restart afterward. Existing mitigations such as staged npm install, dist inventory verification, and hard update respawn were already present around this release path, so this is not strong enough to close as implemented.

## What This Changes

The issue requests a fix for `openclaw gateway update.run` upgrades that leave `openclaw tui` stuck on WebSocket handshake timeout after the Gateway references a missing hashed `dist` file.

## Best Possible Solution

Make package-manager `update.run` fail-safe for managed Gateway installs: either hand off to an external updater that stops/restarts the service before replacing the package tree, or refuse live package swaps with a clear recovery path, then add packaged-dist and post-restart health validation that catches missing hashed chunks before users reach the TUI.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path: upgrade a macOS npm-global LaunchAgent install from 2026.4.11 to 2026.4.29 via `openclaw gateway update.run`, then launch `openclaw tui` and observe the handshake timeout plus missing hashed dist module in Gateway logs; I did not execute it because this review was read-only.

## Solution Assessment

Unclear. The proposed pre-clean is plausible but likely incomplete because current main already stages and verifies npm installs; the safer fix probably needs an out-of-process service handoff or a product decision to disallow live package swaps from `update.run`.
