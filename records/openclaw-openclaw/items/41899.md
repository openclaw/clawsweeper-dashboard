## Summary

Keep open. Current main has static hook timeouts/error policy and plugin-local or channel-local breakers, but it still lacks the requested framework-level plugin circuit breaker with rolling budgets, plugin/hook lock attribution, persisted open/half-open/forced-open state, status/reset controls, and per-plugin degraded fail modes.

## What This Changes

Request a framework-level plugin circuit breaker that can quarantine failing plugin hooks while the rest of the gateway continues operating.

## Best Possible Solution

Implement a phased host-side breaker around plugin hook execution and session-lock telemetry, starting with timing/error attribution and passive evaluation before enabling active open/half-open behavior behind approved policy.

## Reproduction Assessment

Not applicable as a conventional bug reproduction; this is a feature request. Source inspection and current issue discussion give high-confidence evidence that the requested framework breaker surfaces are still absent on current main.

## Solution Assessment

Yes at the architecture boundary: core runtime participation is needed because external plugins cannot reliably gate every plugin hook or attribute host session locks to active plugin handlers. The maintainable path is telemetry and lock attribution first, passive breaker evaluation second, then active behavior behind maintainer-approved policy.
