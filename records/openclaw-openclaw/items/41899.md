## Summary

Keep open. Current main has static hook timeouts/error policy and plugin-local breakers, but it still lacks the requested framework-level plugin circuit breaker with rolling plugin/hook budgets, lock attribution, persisted open/half-open/forced-open state, status/reset controls, and per-plugin fail modes.

## What This Changes

The issue requests a core plugin-runtime circuit breaker that can quarantine unhealthy plugin hooks while the rest of the gateway continues to run.

## Best Possible Solution

Implement this as a phased core runtime feature: timing/error attribution first, plugin/hook lock attribution second, passive breaker evaluation third, then active open/half-open behavior and operator reset/status surfaces behind approved policy.

## Reproduction Assessment

Not applicable as a conventional bug reproduction; this is a feature request. Current-main source inspection gives high-confidence evidence that the requested framework breaker surfaces are still absent.

## Solution Assessment

Yes at the architecture boundary: core runtime participation is needed because external plugins cannot reliably gate all host hook execution or attribute host session locks to active plugin handlers. The exact budgets, persistence format, fail modes, and rollout defaults still need maintainer design.
