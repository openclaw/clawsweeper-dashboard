## Summary

Keep open. Current main has useful partial safeguards, including plugin hook timeouts, fail-open/fail-closed hook policy, an Active Memory-local timeout breaker, and some channel/provider-specific breakers, but it still lacks the requested framework-level plugin circuit breaker with plugin/hook lock attribution, persisted state, status/reset operator controls, and per-plugin degradation policy.

## What This Changes

Request to add a framework-level plugin circuit breaker with hook timing/error capture, lock attribution, persisted breaker state, status/reset controls, half-open recovery, forced-open quarantine, and per-plugin fail modes.

## Best Possible Solution

Design a core plugin-runtime resilience feature that adds plugin/hook attribution, passive telemetry, persisted breaker state, operator status/reset controls, and explicit per-plugin fail modes while keeping plugin-local safeguards where they are safer.

Not applicable. This is a framework feature/design request; source inspection gives a high-confidence current-main gap check rather than a failing bug reproduction path.

## Solution Assessment

Unclear. The circuit-breaker direction is plausible, but budgets, fail modes, persistence, reset semantics, and lock attribution need maintainer runtime/product design before implementation.
