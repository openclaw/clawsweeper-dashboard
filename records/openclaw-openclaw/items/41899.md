## Summary

Keep open: current main has hook timeout handling, channel health restarts, generic runtime health storage, and a plugin-local Active Memory breaker, but not the requested framework-level plugin circuit breaker with rolling per-plugin state, lock attribution, open/half-open/forced-open lifecycle, operator visibility/reset controls, and per-plugin degradation policy.

## What This Changes

Request a framework-level plugin circuit breaker that can quarantine misbehaving plugin hooks/tools and gracefully degrade gateway operation.

## Best Possible Solution

Keep this as a maintainer-owned design item and define the SQLite-backed breaker state, plugin API semantics, degradation policies, and operator status/reset surfaces before implementation.

## Reproduction Assessment

Yes from source at the capability level: current main has direct hook/tool invocation and partial timeout/health mitigations but no generic plugin breaker state or recovery lifecycle; no live incident was reproduced in this review.

## Solution Assessment

Unclear as a final design: a framework breaker is directionally appropriate, but the exact persistence, per-plugin fail modes, operator commands, and API compatibility need maintainer decisions first.
