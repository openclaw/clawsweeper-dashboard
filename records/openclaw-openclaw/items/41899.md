## Summary

Keep open. Current main has partial hook timeout, plugin inspection, and session-lock watchdog behavior, but it does not implement the requested plugin-scoped breaker state machine, persisted registry, status/reset surface, or per-plugin degradation policy; the related open report at https://github.com/openclaw/openclaw/issues/55253 reinforces the same gap while the closed Telegram transport report is a narrower channel-specific case.

## What This Changes

The issue requests a framework-level plugin circuit breaker that isolates failing plugin hooks with telemetry, persisted state, status visibility, manual reset, and configurable degradation behavior.

## Best Possible Solution

Design this as a host-owned plugin runtime resilience feature: passive telemetry first, then plugin-scoped breaker state, explicit lock/hook attribution, plugin-authored fail-mode semantics, and operator status/reset controls.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested plugin-scoped breaker and operator surfaces, but I did not run a live lock-overrun scenario.

## Solution Assessment

Unclear as an implementation plan. The requested direction is reasonable, but the best solution needs maintainer design across plugin runtime contracts, config/schema/docs, persistence, status UI, and per-plugin degradation policy before code work starts.
