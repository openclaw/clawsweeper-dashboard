## Summary

Keep open: current main and v2026.6.10 still provide plugin hook timeouts, plugin-local breakers, and status quarantines, but not the requested framework-level plugin circuit breaker with rolling per-plugin state, lock attribution, half-open/forced-open recovery, operator status/reset controls, and per-plugin degradation policy.

## What This Changes

The issue requests a framework-level plugin circuit breaker for graceful degradation when one plugin repeatedly times out, holds session locks, or throws errors.

## Best Possible Solution

Design a staged core/plugin-runtime breaker with passive telemetry first, SQLite-backed runtime health state, status visibility, and maintainer-approved skip/fail-request/manual quarantine semantics before enabling active degradation.

## Reproduction Assessment

Yes at source level: current hook execution has timeout/logging behavior but no generic per-plugin breaker state, half-open recovery, forced-open quarantine, or operator reset path. I did not run a live multi-agent freeze reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation: the direction is valid, but the best fix needs maintainer policy on defaults, fail modes, status/reset UX, and SQLite-backed runtime state rather than the proposed JSON sidecar.
