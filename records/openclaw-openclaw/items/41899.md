## Summary

Keep open. Current main has hook timeouts, fail-open/fail-closed hook policy, an Active Memory plugin-local timeout breaker, and generic runner idle-timeout safeguards, but it still lacks the requested framework-level plugin circuit breaker with lock attribution, persisted state, status/reset controls, and per-plugin degradation policy.

## What This Changes

Request a core plugin-runtime circuit breaker that can quarantine misbehaving plugin hooks while keeping the gateway healthy.

## Best Possible Solution

Design a core plugin-runtime resilience feature that adds plugin/hook attribution, passive telemetry, persisted breaker state, operator status/reset controls, and explicit per-plugin fail modes while preserving plugin-local safeguards where they are safer.

## Reproduction Assessment

Not applicable. This is a feature/design request; source inspection gives a high-confidence current-main gap check rather than a bug reproduction path.

## Solution Assessment

Unclear. The circuit-breaker direction is reasonable, but budgets, fail modes, persistence, reset semantics, and lock attribution need maintainer runtime/product design before implementation.
