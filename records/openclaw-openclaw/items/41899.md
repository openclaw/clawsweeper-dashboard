- `P1`: The request targets gateway-wide freezes, queue growth, and restart churn from one misbehaving plugin, which can break real multi-agent/channel workflows.

## Summary

Keep open: current main and the latest release have useful partial mitigations, but they still lack the requested generic plugin-scoped circuit breaker with persisted state, lock attribution, half-open/forced-open states, operator reset/status surfaces, and per-plugin fail modes. The 31-agent production freeze comment is relevant evidence that this needs maintainer runtime/product follow-up rather than cleanup closure.

## What This Changes

The issue requests a framework-level plugin circuit breaker that can isolate failing plugin hooks/tools while the rest of the gateway continues operating.

## Best Possible Solution

Design this as staged core resilience work: passive plugin/hook timing and lock attribution first, then persisted breaker state plus status/reset controls, then active skip, half-open, and forced-open behavior with explicit per-plugin fail modes.

## Reproduction Assessment

Not applicable as a strict bug reproduction path: this is a framework feature/design request. Source inspection gives a high-confidence current-main gap check for the missing generic breaker, attribution, persistence, and operator surfaces.

## Solution Assessment

Unclear as a final implementation. A circuit breaker is a plausible direction, but fail modes, thresholds, persisted state, reset semantics, lock attribution, and safe degradation behavior need maintainer runtime/product design before code work.
