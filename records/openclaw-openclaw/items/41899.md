## Summary

Keep open: current main and v2026.6.10 still have hook timeouts, plugin-local circuit breakers, and runtime health/quarantine records, but not the requested framework-level plugin circuit breaker with per-plugin rolling state, hook/session-lock attribution, half-open/forced-open recovery, status visibility, reset controls, and per-plugin degradation policy.

## What This Changes

Request adds a framework-level plugin circuit breaker so one failing or slow plugin can degrade gracefully without freezing the gateway or sessions.

## Best Possible Solution

Design a staged core/plugin-runtime breaker with passive telemetry first, SQLite-backed runtime health state, status visibility, and maintainer-approved skip/fail-request/manual quarantine semantics before enabling active degradation.

## Reproduction Assessment

Yes at source level: current main runs plugin hooks with timeout/error policies but has no generic per-plugin breaker state, lock attribution, half-open recovery, forced-open quarantine, or operator reset path. I did not run a live multi-agent freeze reproduction in this read-only pass.

## Solution Assessment

Unclear as a final implementation: the reliability direction is valid, but the best fix needs maintainer policy on defaults, fail modes, status/reset UX, and SQLite-backed runtime state rather than the proposed JSON sidecar.
