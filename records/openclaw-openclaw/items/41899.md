## Summary

Keep open: current main and the latest release have hook timeouts, session-lock cleanup, plugin state storage, and a plugin-local Active Memory breaker, but they still lack the requested generic plugin-scoped breaker with rolling failure state, plugin/hook lock attribution, persisted breaker registry, operator status/reset controls, and plugin-declared degradation policy.

## What This Changes

The issue requests a framework-level plugin circuit breaker that gracefully isolates misbehaving plugin hooks while preserving the rest of the gateway.

## Best Possible Solution

Keep this as the canonical design item, stage hook timing and plugin/hook lock attribution first, then add a maintainer-approved SQLite-backed breaker registry with status/reset controls and plugin-declared degradation policy.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only pass. Source inspection does confirm current main lacks the generic breaker state machine, plugin/hook lock attribution, and operator reset/status surface requested here.

## Solution Assessment

Unclear as a concrete implementation plan. The direction is valid, but the best maintainable solution needs maintainer agreement on plugin API boundaries, persistence, default budgets, fail modes, and upgrade behavior.
