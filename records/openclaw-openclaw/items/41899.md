## Summary

Keep open: current main and the latest release have partial mitigations such as hook timeouts, plugin health diagnostics, context-engine quarantine, and an Active Memory-local timeout breaker, but they still do not provide the requested generic plugin-scoped circuit breaker with rolling state, lock attribution, half-open recovery, operator reset/status controls, and per-plugin degradation policy.

## What This Changes

Request a framework-level plugin circuit breaker that isolates misbehaving plugin hooks while preserving the rest of the gateway.

## Best Possible Solution

Design a staged core/plugin-runtime breaker that records plugin and hook failure/lock telemetry in SQLite-backed runtime health state, starts with passive visibility, and only enables skip/fail-request/manual quarantine behavior after maintainer-approved policy and operator controls are defined.

## Reproduction Assessment

No live reproduction was run, but source-level reproduction of the missing capability is high confidence: current main has timeout/quarantine pieces but no generic plugin breaker registry, lock attribution, or operator reset/status path.

## Solution Assessment

Unclear as a final implementation: the request is valid, but the best fix should be staged and policy-led, with passive telemetry first and SQLite-backed breaker state instead of the proposed JSON sidecar.
