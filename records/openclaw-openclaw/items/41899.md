## Summary

Keep open: current main has hook timeouts, session-lock watchdogs, plugin state storage, and diagnostics that partially reduce stalls, but it still does not implement the requested per-plugin circuit-breaker state machine, plugin/hook lock attribution, persisted breaker registry, status/reset command surface, or per-plugin degradation policy. Related discussion makes this issue the canonical broad plugin-runtime breaker thread rather than a duplicate to close.

## What This Changes

The issue requests a framework-level plugin circuit breaker that times and attributes plugin hook failures, opens or half-opens per-plugin breakers, persists operator-visible state, and supports graceful plugin-scoped degradation.

## Best Possible Solution

Keep this issue open as the canonical design item, then stage the work through maintainer-approved telemetry and attribution first, followed by a SQLite-backed plugin breaker registry, status/reset surfaces, and typed per-plugin degradation policy that avoids plugin-specific defaults in core.

## Solution Assessment

Unclear as an implementation plan. A maintainable solution needs maintainer agreement on the generic plugin API/runtime boundary, persistence model, default budgets, and how per-plugin degradation policies are declared.
