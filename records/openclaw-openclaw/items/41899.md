## Label Justifications

- `P1`: The request targets gateway-wide freezes and restart churn from one plugin, which can break real multi-agent/channel workflows.
- `impact:session-state`: The reported failure mode is session locks held by plugin work until sessions freeze or become stale.
- `impact:message-loss`: The issue and related production comments describe inbound messages not being processed while sessions are locked or the gateway is degraded.
- `impact:crash-loop`: The requested breaker is meant to stop repeated plugin failures from cascading into gateway restart churn or frozen runtime availability.

## Summary

Keep open. Current main has hook timeouts, session-lock cleanup, and an Active Memory-only breaker, but it still lacks the requested framework-level plugin circuit breaker with plugin/hook lock attribution, persisted state, operator status/reset controls, half-open/forced-open behavior, and per-plugin degradation policy.

## What This Changes

Add a framework-level plugin circuit breaker so one misbehaving plugin can be degraded without freezing the gateway or other sessions.

## Best Possible Solution

Design a staged core plugin-runtime resilience feature: passive timing/error telemetry and plugin/hook lock attribution first, then persisted breaker state plus status/reset controls, then active skip/half-open/forced-open behavior with explicit per-plugin fail modes.

## Reproduction Assessment

Not applicable as a strict bug reproduction path: this is a framework feature/design request. Source inspection gives a high-confidence current-main gap check for the missing generic breaker, attribution, persistence, and operator surfaces.

## Solution Assessment

Unclear as a final implementation. The circuit-breaker direction is plausible, but budgets, fail modes, persisted state, lock attribution, reset semantics, and safe degradation behavior need maintainer runtime/product design before code work.
