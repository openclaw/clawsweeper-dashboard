## Summary

Keep open. Current main has useful resilience pieces, including static hook timeouts, fail-open/fail-closed hook policy, session-lock cleanup, and an Active Memory-local timeout breaker, but it still lacks the requested framework-level plugin circuit breaker with rolling budgets, plugin/hook lock attribution, persisted open/half-open/forced-open state, status/reset controls, and per-plugin degraded fail modes.

## What This Changes

Request a framework-level plugin circuit breaker that gracefully degrades misbehaving plugin hooks using telemetry, persisted state, operator controls, and per-plugin fail modes.

## Best Possible Solution

Implement a phased host-side breaker around plugin hook execution and session-lock telemetry, with status/reset controls and coordination with #55253 for any crash-loop auto-recovery follow-up.

## Reproduction Assessment

Not applicable as a conventional bug reproduction; this is a feature request. The implementation gap is high-confidence from current source/docs inspection of hook execution, lock ownership, operator surfaces, and the related discussion.

## Solution Assessment

Yes at the architecture boundary: core runtime participation is needed because external plugins cannot reliably gate every plugin hook or attribute host session locks to active plugin handlers. The safer path is telemetry and lock attribution first, passive breaker evaluation second, then active open/half-open behavior behind maintainer-approved policy.
