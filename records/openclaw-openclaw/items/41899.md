## Summary

Keep open: current main has hook timeouts and several plugin-local circuit breakers, but it does not have the requested framework-level plugin breaker with plugin/hook lock attribution, persisted open/half-open/forced-open state, status visibility, reset controls, or per-plugin fail modes. The open related #55253 overlaps, but it is broader and does not supersede the detailed circuit-breaker design and operator context here.

## What This Changes

The issue requests a framework-level plugin circuit breaker that can isolate misbehaving plugin hooks, record breaker state, and expose operator controls for graceful degradation.

## Best Possible Solution

Design a staged core/plugin-runtime resilience layer that starts with telemetry and plugin-attributed lock/run events, then adds passive evaluation, persisted breaker state, status output, and explicit operator controls before enabling active skipping for selected hooks.

## Reproduction Assessment

Not applicable. This is a feature/design request, and source inspection confirms only partial existing mitigations rather than a failing current-main bug path.

## Solution Assessment

Unclear as an immediate implementation plan. The staged rollout in the issue is directionally sound, but budgets, fail modes, persistence, and plugin-owned degradation need maintainer design before a narrow PR is safe.
