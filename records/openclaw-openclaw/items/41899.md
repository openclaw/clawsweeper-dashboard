## Label Justifications

- `P1`: The request targets gateway-wide freezes, queue growth, and restart churn from one misbehaving plugin, which can break real multi-agent/channel workflows.

- `impact:crash-loop`: The requested breaker is intended to stop repeated plugin errors from cascading into restart churn or frozen runtime availability.

## Summary

Keep open: current main has partial resilience work, but it still lacks the requested generic plugin-scoped circuit breaker with persisted state, plugin/hook lock attribution, half-open/forced-open states, status/reset controls, and per-plugin fail modes.

## What This Changes

Request a framework-level plugin circuit breaker that gracefully degrades only unhealthy plugin hooks/tools instead of letting one plugin freeze gateway sessions or message processing.

## Best Possible Solution

Design this as a staged core resilience feature: passive plugin/hook timing and lock attribution first, then persisted breaker state plus status/reset controls, then active skip, half-open, and forced-open behavior with explicit per-plugin fail modes.

## Reproduction Assessment

Not applicable as a strict bug reproduction path: this is a framework feature/design request. Source inspection gives a high-confidence current-main gap check for the missing generic breaker, attribution, persistence, and operator surfaces.

## Solution Assessment

Unclear as a final implementation. A circuit breaker is a plausible direction, but fail modes, thresholds, persisted state, reset semantics, lock attribution, and safe degradation behavior need maintainer runtime/product design before code work.
