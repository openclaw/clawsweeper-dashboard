## Summary

Keep open: current main has per-hook timeouts, session-lock watchdog cleanup, plugin state storage, and a plugin-local Active Memory breaker, but it still lacks the requested generic plugin-scoped breaker state machine, plugin/hook lock attribution, persisted breaker registry, status/reset operator surface, and typed per-plugin degradation policy.

## What This Changes

The issue requests a framework-level plugin circuit breaker that times and gates plugin hooks, attributes lock overruns to plugins, persists breaker state, exposes status/reset controls, and supports graceful per-plugin fail modes.

## Best Possible Solution

Keep this as the canonical design item, stage telemetry and lock attribution first, then add a maintainer-approved SQLite-backed plugin breaker registry with status/reset controls and typed plugin-declared degradation policy that keeps plugin-specific defaults out of core.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only pass. Source inspection does confirm current main lacks the requested generic plugin breaker state machine, plugin/hook lock attribution, and operator reset/status surface.

## Solution Assessment

Unclear as an implementation plan: the direction is valid, but the best maintainable solution needs maintainer agreement on the plugin API/runtime boundary, persistence model, default budgets, and per-plugin degradation declarations.
