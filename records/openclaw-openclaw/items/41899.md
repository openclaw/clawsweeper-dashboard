## Summary

Keep open. Current main has partial guardrails, including plugin hook timeouts, plugin-specific active-memory timeout breaking, invalid-config quarantine, and session-lock cleanup, but it still lacks the requested framework-level plugin-scoped circuit breaker with rolling failure state, lock attribution to plugin/hook, persisted breaker registry, status/reset surfaces, and per-plugin degradation policy.

## What This Changes

Request a framework-level plugin circuit breaker that can skip or quarantine unhealthy plugin hooks while the rest of the gateway continues running.

## Best Possible Solution

Design this as a generic plugin-runtime resilience feature with abortable hook/tool timeouts, plugin/hook/session lock attribution, persisted breaker state, status/reset controls, metrics, and tests, while leaving plugin-specific degradation behavior explicit and configurable.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main lacks the requested generic breaker, but no concrete failing plugin scenario was reproduced.

## Solution Assessment

Unclear as a final design, though the direction is credible. The maintainable path is a maintainer-owned design across plugin runtime, lock telemetry, persistence, operator UX, and per-plugin degradation policy rather than a narrow one-file repair.
