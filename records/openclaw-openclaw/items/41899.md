## Summary

Keep open. Current main has several resilience pieces, but it still does not implement the requested host-side plugin circuit breaker with rolling budgets, plugin/hook/session lock attribution, persisted open/half-open/forced-open state, operator status/reset controls, and per-plugin degraded fail modes.

## What This Changes

The issue requests a framework-level plugin circuit breaker that degrades only misbehaving plugin hooks while keeping the rest of the gateway available.

## Best Possible Solution

Use this as the dedicated plugin circuit-breaker tracker unless maintainers choose #55253 as canonical. The end state should be a host-side breaker around plugin hook execution and session-lock telemetry, with plugin/hook/session attribution, rolling error and latency budgets, persisted states, status/reset controls, per-plugin fail modes, and regression tests proving one failing plugin does not block unrelated plugins.

## Reproduction Assessment

Not applicable as a conventional bug reproduction; this is a feature request. The implementation gap is high-confidence from static inspection of current hook execution, lock ownership, CLI/docs surfaces, and plugin-local breaker examples.

## Solution Assessment

Yes, the requested direction belongs in core runtime rather than a plugin alone, because external plugins cannot reliably gate all host hook execution or attribute session locks to active plugin handlers. The maintainable path is phased: telemetry and attribution first, passive evaluation second, then active open/half-open behavior behind a gateway flag.
