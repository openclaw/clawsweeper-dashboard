## Summary

Keep open. Current main has useful resilience pieces, including hook timeouts, static fail-open/fail-closed policy, invalid plugin config isolation, lock cleanup, diagnostics, and an Active Memory-local timeout breaker, but it still lacks the requested framework-level plugin circuit breaker with rolling budgets, plugin/hook lock attribution, persisted open/half-open/forced-open state, status/reset controls, and per-plugin degraded fail modes.

## What This Changes

Request a host-level plugin circuit breaker that degrades only failing plugin hooks while preserving the rest of the gateway.

## Best Possible Solution

Use this as the broad gateway/plugin-runtime breaker tracker unless maintainers make #55253 canonical, then implement a phased host-side breaker around hook execution and session-lock telemetry with status/reset controls and focused regression tests.

## Reproduction Assessment

Not applicable as a conventional bug reproduction; this is a feature request. The implementation gap is high-confidence from static inspection of current hook execution, lock ownership, docs, tests, and the related discussion.

## Solution Assessment

Yes at the architecture boundary: this needs core runtime participation because external plugins cannot reliably gate all plugin hooks or attribute host session locks to active plugin handlers. The safer implementation path is telemetry and lock attribution first, passive evaluation second, then active breaker behavior behind an operator-controlled rollout.
