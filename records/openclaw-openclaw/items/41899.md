## Label Justifications

- `P1`: The reported failure class can freeze active sessions or trigger repeated plugin failure loops when one plugin stalls or crashes under real gateway load.
- `impact:crash-loop`: The issue is about repeated plugin failures causing gateway/session availability degradation and restart-style recovery pressure.
- `impact:message-loss`: The related reports describe user messages not being processed while plugin work blocks or destabilizes the session path.
- `impact:session-state`: The requested fix needs session lock attribution and plugin isolation to avoid stuck or zombie session state.

## Summary

Keep open: current main and v2026.6.8 include hook timeouts, context-engine quarantine, status diagnostics, and an Active Memory-local timeout breaker, but they still do not provide the requested framework-level plugin breaker with rolling per-plugin state, lock attribution, half-open/forced-open recovery, operator reset/status controls, and per-plugin degradation policy.

## What This Changes

Request a framework-level plugin circuit breaker that isolates unhealthy plugin hooks, tracks lock and failure signals, persists state, exposes status/reset controls, and supports graceful per-plugin degradation.

## Best Possible Solution

Design a staged core/plugin-runtime breaker that records plugin and hook failure/lock telemetry in SQLite-backed runtime health state, starts with passive visibility, and enables skip/fail-request/manual quarantine behavior only after maintainer-approved policy and operator controls are defined.

## Reproduction Assessment

No live reproduction was run, but source-level reproduction is high confidence because hook execution has timeouts/logging without a generic plugin breaker registry, lock attribution, half-open/forced-open state, or operator reset/status path.

## Solution Assessment

Unclear as a final implementation: the request is valid, but the best fix should be staged and policy-led, with passive telemetry first and SQLite-backed breaker state instead of the proposed JSON sidecar.
