- `P1`: The request targets gateway-wide freezes, queue growth, and restart churn from one misbehaving plugin, which can break real multi-agent/channel workflows.

## Summary

Keep open. Current main has partial mitigations such as bounded hook timeouts, session-lock cleanup, diagnostics, and an Active Memory-local timeout breaker, but it still lacks the requested framework-level plugin circuit breaker with generic plugin/hook attribution, persisted breaker state, status/reset controls, half-open/forced-open behavior, and per-plugin degradation policy.

## What This Changes

Request a gateway/plugin-runtime circuit breaker that can isolate misbehaving plugins and degrade only affected plugin hooks or tools while the rest of the gateway continues running.

## Best Possible Solution

Design a staged core resilience feature: passive plugin/hook timing and lock attribution first, then persisted breaker state plus status/reset controls, then active skip, half-open, and forced-open behavior with explicit per-plugin fail modes.

## Reproduction Assessment

Not applicable as a strict bug reproduction path: this is a framework feature/design request. Source inspection gives a high-confidence current-main gap check for the missing generic breaker, attribution, persistence, and operator surfaces.

## Solution Assessment

Unclear as a final implementation. A circuit breaker is a plausible direction, but fail modes, thresholds, persisted state, reset semantics, lock attribution, and safe degradation behavior need maintainer runtime/product design before code work.
