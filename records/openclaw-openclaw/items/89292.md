Keep open: this maintainer draft fixes a real source-visible crash path, but the proposed cloning step introduces a compatibility regression for supported class-backed plugin tools before the PR is safe to merge.

## What This Changes

The PR hardens auto-reply context prompt bundle construction so unreadable tool `name` descriptors are skipped while healthy tool names continue into `/context` and `/export-session` metadata.

## Best Possible Solution

Revise the name guard to filter unreadable tools without replacing healthy tool instances, or use a forwarding wrapper that keeps the original tool as receiver for non-`name` properties, with regression coverage for a class-backed healthy sibling.

## Reproduction Assessment

Yes from source: current main can crash when `tool.name` throws, and the PR’s replacement-object approach can be checked against the existing class-backed plugin tool pattern with private accessors.

## Solution Assessment

No: guarding unreadable names is the right layer, but cloning every readable tool is not the best fix because it breaks supported object identity/private-field behavior.
