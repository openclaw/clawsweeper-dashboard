## Summary

The PR is still useful because current main still reads command prompt tool names directly, but it should not merge as written: the helper clones every readable tool, which can break existing class-backed plugin tools whose accessors rely on private fields. A reviewer already pointed out the same receiver/identity problem, and the branch is currently draft and conflicting.

## What This Changes

The PR filters out command prompt tools whose `name` getter throws and adds a regression test that keeps a readable sibling tool in the prompt bundle.

## Best Possible Solution

Keep the unreadable-name guard, but return the original readable tool instances while carrying the resolved names separately for prompt generation, with a regression covering a class-backed healthy tool beside an unreadable-name sibling.

## Reproduction Assessment

Yes from source: current main can crash when `tool.name` throws, and the PR's clone-based repair can be checked against existing class-backed plugin tool coverage with private-field-backed accessors.

## Solution Assessment

No: guarding unreadable names is the right boundary, but cloning readable tools is not the best fix because downstream metadata readers still access other tool fields on objects that may need their original receiver.
