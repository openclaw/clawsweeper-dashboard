## What This Changes

## Best Possible Solution

Land once maintainers explicitly accept the stricter invalid-stop behavior, or adjust the gateway to preserve current tolerance for malformed `stop` while still forwarding valid stop sequences.

## Reproduction Assessment

Yes, from source inspection: current main does not parse or forward `payload.stop`, so a chat-completions request cannot carry stop sequences to streamParams. The PR adds focused tests for the new forwarding and rejection behavior, but I did not run tests in this read-only review.

## Solution Assessment

Mostly yes: using the existing stream-param channel and provider transport mappings is the narrow maintainable path. The unresolved part is whether malformed `stop` values should now fail closed or remain tolerated for compatibility.
