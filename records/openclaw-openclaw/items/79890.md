## Summary

Current main already addresses the central doctor/upgrade routing problem by migrating legacy CLI-prefixed model refs to canonical provider refs while adding model-scoped CLI runtime policy. This PR should not merge as-is because it preserves the legacy prefix instead of the canonical migration path, carries unrelated Active Memory behavior changes, and lacks real behavior proof.

## What This Changes

The PR skips doctor model.primary migration when a matching CLI agentRuntime is present, adds migration tests, and also changes Active Memory caching and zero-hit timeout behavior.

## Best Possible Solution

Keep current main’s canonical provider/model migration with model-scoped CLI runtime policy; any remaining live billing failure should be reported with the post-upgrade config showing the missing runtime policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a high-confidence migration path for the reported upgrade behavior, but I did not run a live doctor/provider reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: current main’s canonical anthropic/* model ref plus model-scoped claude-cli runtime policy matches the documented config contract, while this PR keeps a legacy prefix and includes unrelated Active Memory changes.
