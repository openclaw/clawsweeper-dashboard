## Summary

Keep open: the patch is aimed at a real Telegram plugin-state row-limit bug, but the proposed budget still does not cover multi-account topic-cache namespaces, and the external PR has no real behavior proof yet.

## What This Changes

The PR lowers Telegram message, topic-name, and bot-info plugin-state cache caps, adds focused tests, and also moves a Codex app-server oversized-session token check before rollout-file scanning.

## Best Possible Solution

Bound Telegram plugin-state usage across all supported account/topic namespace shapes, keep or split the Codex startup guard with its own proof, and land only after real runtime evidence shows the row cap is no longer exceeded.

## Reproduction Assessment

Yes from source inspection: fill the 800-row message cache, 32 bot-info rows, and two 160-row account-scoped topic-name namespaces, then a Telegram topic write can still cross the 1,000 live-row plugin cap. I did not run this because the review is read-only.

## Solution Assessment

No: lowering per-namespace caps is the right direction, but the current PR does not budget across all supported topic namespaces. The safer fix is a global Telegram plugin-state budget or one bounded shared topic namespace plus multi-account coverage.
