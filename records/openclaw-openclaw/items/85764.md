## Summary

Keep open: this is a focused source-backed fix for a real session write-lock timeout path, but it changes live-holder reclaim semantics and still needs maintainer acceptance plus matching docs/help text before merge.

## What This Changes

The PR records `maxHoldMs` in session lock payloads, makes acquisition-time stale removal respect holder max-hold for non-self-held locks, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land the narrow max-hold acquisition fix once maintainers accept cross-process live-holder reclaim semantics and align config docs/help with the new behavior.

## Reproduction Assessment

Yes at source level: current main's acquisition-time inspection ignores `maxHoldMs`, so a live lock older than max-hold but younger than `staleMs` remains non-stale. I did not run a live multi-process repro in this read-only review.

## Solution Assessment

Mostly yes, if maintainers accept the existing max-hold policy as authoritative across process boundaries. The safer alternative is to keep acquisition watchdog-only or add an explicit strict mode, but that would not solve the reported timeout as directly.
