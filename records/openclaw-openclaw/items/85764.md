## Summary

Keep open for maintainer review: source inspection supports the bug and the PR direction, but acquisition-time reclaim of a live holder after `maxHoldMs` is a session-state policy change that needs explicit owner acceptance before merge.

## What This Changes

The PR stores the holder's session write-lock `maxHoldMs` in the lock payload, makes acquisition-time stale inspection respect it for other processes, adds regression coverage, and records a changelog fix.

## Best Possible Solution

Land the narrow max-hold acquisition fix after maintainers accept the cross-process live-lock reclaim semantics and align config docs/help with the new behavior.

## Reproduction Assessment

Yes at source level: current main's acquisition-time `shouldReclaim` path inspects live locks against `staleMs` but not `maxHoldMs`, so a live lock older than `maxHoldMs` and younger than `staleMs` remains non-stale. I did not run a live multi-process repro in this read-only review.

## Solution Assessment

Yes in direction: recording the holder's max-hold policy in the lock payload and using fs-safe's unchanged-snapshot removal is the narrow maintainable path. The remaining questions are maintainer acceptance of live-holder reclaim semantics and docs/help alignment.
