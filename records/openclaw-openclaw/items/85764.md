- `P1`: The PR targets an active gateway session-lock failure that can make real agent workflows unresponsive and block transcript writes.
- `merge-risk: 🚨 session-state`: The diff changes session transcript lock reclamation, where incorrect deletion or duplicate writers can corrupt or mis-associate session state.
- `merge-risk: 🚨 availability`: The diff changes the lock acquisition path that determines whether blocked gateway writes recover, time out, or fail with stale-lock errors.

## Summary

Keep open: the intended maxHoldMs acquisition fix is directionally valid, but the patch does not thread that policy through stale-lock recovery, so a maxHold-exceeded live lock can still fail instead of being removed and retried. The new dead-PID fast path also adds a path-only unlink on a session-state lock, overlapping an already tracked stale-lock race concern.

## What This Changes

The PR makes session write-lock acquisition classify locks older than maxHoldMs as stale and adds a pre-retry dead-PID lock cleanup shortcut.

## Best Possible Solution

Use one shared reclaim policy for shouldReclaim and stale removal, preferably integrated with fs-safe's unchanged-snapshot recovery, then add focused coverage for a live PID older than maxHoldMs but younger than staleMs.

## Reproduction Assessment

Yes for the PR defect: source inspection shows shouldReclaim can now return true for hold-exceeded, but the file_lock_stale recovery helper rechecks without maxHoldMs and returns false. I did not run tests because this review is read-only.

## Solution Assessment

No. The maxHoldMs check needs to be applied consistently through both the reclaim decision and the stale-removal retry path, and the new direct unlink should be removed or snapshot-guarded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Thread maxHoldMs through stale recovery:** `src/agents/session-write-lock.ts:662`
  - body: When this new `maxHoldMs` check makes `shouldReclaim` return true, fs-safe uses the default fail-closed mode and throws `file_lock_stale`. `acquireSessionWriteLock` then calls `removeReportedStaleLockIfStillStale`, but that helper still re-inspects with only `staleMs`, so a live PID older than `maxHoldMs` but younger than `staleMs` is not removed and the raw stale-lock error escapes instead of retrying acquisition. Please pass the same `maxHoldMs` policy through the removal path or use one shared reclaim decision.
  - confidence: 0.95
- **[P2] Avoid unlinking locks outside snapshot recovery:** `src/agents/session-write-lock.ts:647`
  - body: This fast path reads a lock payload and then deletes `lockPath` directly if the recorded PID is dead. If the sidecar changes between that read and the unlink, this can remove a fresh holder's lock; this is the same class of path-only stale cleanup that the existing fs-safe `remove-if-unchanged` recovery is designed to avoid. Please route dead-PID cleanup through the normal snapshot-guarded stale recovery path or remove this shortcut.
  - confidence: 0.86
