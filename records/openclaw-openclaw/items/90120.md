## Summary

Keep this maintainer-labeled draft open: the alias and hookAliases guard is useful, but the stated throwing-id hardening is still incomplete because real provider public artifacts read id before the new registry helper runs.

## What This Changes

The branch adds guarded provider id, aliases, and hookAliases readers in the plugin contract registry plus a regression test for poisoned provider metadata during alias resolution.

## Best Possible Solution

Keep the registry-entry guards, but either add a public-artifact loader guard with a production-path regression for unreadable id metadata or narrow the PR claim and test to alias/hookAliases poisoning only.

## Reproduction Assessment

Yes for the alias/hookAliases crash: current main directly reads those provider metadata fields during alias resolution. The throwing-id public-artifact path is not reproduced by the new test because the real loader reads id earlier.

## Solution Assessment

No, not fully. Guarding registry entries is the right local fix for alias/hookAliases poisoning, but the throwing-id case needs a guard at the public-artifact loader or a narrower stated/tested scope.
