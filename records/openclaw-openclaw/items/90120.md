## What This Changes

The branch adds guarded provider id/alias readers in the plugin contract registry and a regression test for poisoned provider metadata during alias resolution.

## Best Possible Solution

Keep the registry-entry guards, then either guard the public-artifact provider validation path with a production-path regression or narrow the PR scope and test to alias/hookAliases poisoning only.

## Reproduction Assessment

Yes for source reproduction of the alias/hookAliases crash: current main directly reads those metadata fields during alias resolution. The throwing-`id` public-artifact path is not covered by the new test because the mock bypasses the real loader validation.

## Solution Assessment

No, not fully. Guarding registry-entry reads is the right local mitigation for alias poisoning, but the claimed throwing-`id` case needs a guard at the public-artifact validation path or a narrower stated/tested scope.
