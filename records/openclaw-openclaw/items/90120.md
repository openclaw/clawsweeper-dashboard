## Summary

Keep open: the PR is a useful provider-contract hardening patch, but one claimed path is still uncovered. The new guard handles provider entries that already reach `registry.ts`, while a real public-artifact provider with a throwing `id` can still fail earlier in the public-artifact type guard before the new helper runs.

## What This Changes

The branch adds guarded provider id/alias readers in the plugin contract registry and a regression test for poisoned provider metadata during alias resolution.

## Best Possible Solution

Keep the registry-entry guards, then either guard the real public-artifact loader for unreadable `id` metadata with a production-path regression or narrow the PR scope to aliases/hookAliases poisoning only.

## Reproduction Assessment

Yes for the alias/hookAliases crash: current main directly reads those provider metadata fields during alias resolution. The throwing-`id` public-artifact path is not fully reproduced by the new test because the real loader reads `id` earlier.

## Solution Assessment

No, not fully. Guarding provider entries in `registry.ts` is the right local fix for alias/hookAliases poisoning, but the throwing-`id` case needs the guard at the public-artifact loader or a narrower stated/tested scope.
