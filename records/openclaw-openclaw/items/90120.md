## Best Possible Solution

Keep the registry-entry guards, then either guard the real public-artifact provider validation path with a production-path regression or narrow the PR claim and test to alias and hookAlias poisoning only.

## Reproduction Assessment

Yes for the alias and hookAlias crash: current main directly reads `entry.provider.id`, `aliases`, and `hookAliases` in the provider-alias resolver. The throwing-id public-artifact path is also source-reproducible as uncovered by this patch because `isProviderPlugin` still reads `value.id` first.

## Solution Assessment

No, not fully. Guarding provider entries in `registry.ts` is the right local fix after entries exist, but the throwing-id case needs the guard at the public-artifact validator or a narrower stated and tested scope.
