## Summary

Keep open: current source confirms the reported `[object Object]` path, but the PR only extracts `.path` while current docs and downstream evaluation still treat config requirements as string OpenClaw config keys, so it needs contract review or a fuller fix before merge.

## What This Changes

The PR adds `normalizeConfigRequirementList` in `src/shared/frontmatter.ts` so object-format `requires.config` entries emit their `path` string instead of being passed to the generic string-list normalizer.

## Best Possible Solution

Decide the skill metadata contract first, then either support object-format config requirements end to end with parser, type, status, docs, and regression coverage, or reject them with a clear diagnostic instead of emitting `[object Object]`.

## Reproduction Assessment

Yes, for the reported `[object Object]` symptom: source inspection shows object array entries flow into a normalizer that calls `String(entry)`. I did not run a live CLI fixture because this review was read-only.

## Solution Assessment

No: extracting `.path` alone does not settle whether `requires.config` objects are supported or whether their `path` is a config key or filesystem path. The maintainable fix should update the full contract and downstream checks together.
