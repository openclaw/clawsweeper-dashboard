## Summary

Keep this PR open: current main still has the source-reproducible `[object Object]` symptom, but the revised patch now drops unsupported object config requirements, which can make a skill eligible instead of preserving a clear unsatisfied diagnostic. The earlier supply-chain concern was resolved by removing the dependency override from the latest diff.

## What This Changes

The PR changes shared skill frontmatter parsing so object-format `requires.config` entries emit an `UnsupportedSkillConfigRequirement` warning and are skipped, and adds a changelog entry.

## Best Possible Solution

Keep the string-key contract unless maintainers approve object requirements; this PR should either implement object support end to end or report unsupported object entries without making the requirement disappear.

## Reproduction Assessment

Yes for the visible serialization symptom: source inspection shows current main passes object array entries through `String(entry)`, and Node confirms that produces `[object Object]`. A full CLI reproduction was not run during this read-only review.

## Solution Assessment

No: skipping object entries is not the best fix because it can mark a skill eligible after dropping a declared requirement. A safer narrow fix would preserve a visible unsupported-requirement diagnostic that remains unsatisfied, or implement the object contract across parser, evaluator, docs, and tests.
