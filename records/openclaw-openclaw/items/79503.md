## Summary

Keep open: the PR fixes the visible `[object Object]` normalization symptom, but it does not implement the requested object-format config requirement semantics end to end and it mixes in a supply-chain change.

## What This Changes

The PR extracts `.path` from object entries in `metadata.openclaw.requires.config` and also pins `fast-uri` to 3.1.2 in package metadata and the lockfile.

## Best Possible Solution

Define the `requires.config` object contract first, then either implement object entries across parser types, evaluator/status output, docs, and tests, or reject unsupported objects with a clear diagnostic.

## Reproduction Assessment

Yes for the visible serialization symptom: current main stringifies object entries through `String(entry)`, which source inspection shows will produce `[object Object]`. The broader readable-file expectation is not reproducible as an existing documented contract because current docs and types define `requires.config` as string config paths.

## Solution Assessment

No: extracting `.path` is too narrow because the rest of the requirement pipeline still treats the value as an OpenClaw config key. The safer solution is a complete object-contract implementation or an explicit unsupported-shape diagnostic.
