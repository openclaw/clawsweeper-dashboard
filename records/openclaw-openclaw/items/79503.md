## Summary

Keep open: the PR addresses a real `[object Object]` parser symptom, but it expands an undocumented object-format skill config contract and only extracts `.path` without making downstream eligibility or file-access semantics work.

## What This Changes

The PR adds a config requirement normalizer in `src/shared/frontmatter.ts` that extracts `.path` from object entries and leaves string entries unchanged.

## Best Possible Solution

Decide the `requires.config` object contract, then either implement object entries end to end with types, parser, evaluator/status output, docs, and regression coverage, or reject object entries with a clear diagnostic.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the `[object Object]` symptom: object entries flow into a normalizer that calls `String(entry)`. I did not run a live CLI fixture in this read-only review.

## Solution Assessment

No: extracting `.path` alone is not the best fix because downstream still treats the result as an OpenClaw config key and the object contract is not documented. The safer path is to settle and implement the full contract or reject unsupported objects clearly.
