## Summary

Keep open: the latest PR still only extracts `.path` into a string, while current docs, types, and eligibility checks define `requires.config` as OpenClaw config keys rather than filesystem-readable objects. It also mixes in an unrelated dependency override that needs separate review.

## What This Changes

The PR adds a config requirement normalizer that extracts `.path` from object entries and pins `fast-uri` to 3.1.2 in package metadata and the lockfile.

## Best Possible Solution

Decide the skill metadata contract, then implement object config requirements end to end with parser/types/evaluator/status/docs/tests or reject unsupported objects with a clear diagnostic; handle the `fast-uri` pin separately.

## Reproduction Assessment

Yes for the visible serialization symptom: current main stringifies object entries via `String(entry)`, which source inspection shows produces `[object Object]`. The broader filesystem-readable object expectation is not a documented current-main contract.

## Solution Assessment

No: the PR is too narrow because extracting `.path` leaves the rest of the requirement pipeline treating that value as an OpenClaw config key. A maintainable fix needs either a full object-contract implementation or an explicit unsupported-shape diagnostic.
