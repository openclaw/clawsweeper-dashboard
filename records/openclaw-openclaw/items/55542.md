## Summary

Keep open. The migration is not safe to merge as written: current main still relies on the sharp-backed media-understanding-core runtime, and the PR changes exported image behavior, backend selection, installer docs, and build approvals before the graph is sharp-free.

## What This Changes

The PR removes sharp from the root package/install docs, adds @napi-rs/image, and ports core image operations plus image-related tests to Transformer APIs.

## Best Possible Solution

Rebase or replace the migration around current media-understanding-core ownership, preserve SDK media semantics and documented backend/install compatibility, and only remove sharp once the dependency graph is actually sharp-free.

Yes for the PR regressions: the branch diff, current main source, and @napi-rs/image/sharp type contracts are enough to reproduce the behavior mismatches by inspection. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The dependency swap may be worth pursuing, but the branch changes runtime semantics and install guardrails before the current owner module and package graph are ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve source format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: `normalizeExifOrientation` now returns `transformer.rotate().jpeg(90)` for every non-sips input. The current sharp path uses `rotate().toBuffer()`, whose default output format matches the input, so PNG/WebP callers can now receive lossy JPEG bytes and lose alpha after requesting only orientation normalization.
  - confidence: 0.95
- **[P2] Preserve resize semantics when using Transformer:** `src/media/image-ops.ts:344-345`
  - body: The Transformer resize call drops sharp's `withoutEnlargement` behavior and switches to `fit: 1` when enlargement is allowed, which is `Fill` in @napi-rs/image rather than sharp's aspect-preserving `inside`. This can upscale small PNGs during optimization or stretch images into squares instead of preserving the existing media contract.
  - confidence: 0.9
- **[P2] Keep the legacy image backend override working:** `src/media/image-ops.ts:28`
  - body: Changing the Bun/macOS non-sips override from `OPENCLAW_IMAGE_BACKEND=sharp` to only `napi-rs` makes existing deployments with the documented `sharp` override fall back to the sips path after upgrade. Keep `sharp` as a compatibility alias or add a deliberate migration path.
  - confidence: 0.88
- **[P2] Keep sharp build approval until the graph is sharp-free:** `package.json:964`
  - body: The PR removes `sharp` from `pnpm.onlyBuiltDependencies`, but current main still declares sharp in media-understanding-core and the lockfile still resolves it. Dropping the build approval before the dependency graph is migrated can leave installs without the native sharp package setup they still need.
  - confidence: 0.84
