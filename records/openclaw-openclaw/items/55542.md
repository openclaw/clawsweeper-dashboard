## Summary

Keep open: this native image dependency swap is still not safe to merge because current main now routes image processing through the media-understanding-core plugin, while this branch changes the older root implementation/package boundary and still lacks real behavior proof.

## What This Changes

## Best Possible Solution

Rebase onto current main and migrate the media-understanding-core image runtime/package policy only if maintainers want this native dependency swap, preserving format, alpha, backend-override compatibility, and real install/media proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move the image dependency swap into the media plugin:** `package.json:862`
  - body: Current main loads image ops from `media-understanding-core`, whose package still declares and imports `sharp`. Adding `@napi-rs/image` at root and editing the old root implementation will leave the current runtime path on `sharp` after rebase, so the migration needs to update the plugin-owned runtime/dependency instead.
  - confidence: 0.86
- **[P2] Preserve format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: The branch returns `transformer.rotate().jpeg(90)` for the non-sips path, so a PNG or WebP input can become lossy JPEG and lose alpha even though the helper only promises orientation normalization. Keep same-format behavior or make the conversion explicit in the API contract.
  - confidence: 0.91
- **[P2] Keep the existing backend override working:** `src/media/image-ops.ts:28`
  - body: Changing the old `OPENCLAW_IMAGE_BACKEND=sharp` override to only accept `napi-rs` means existing Bun/macOS installs using the documented override would start taking the sips path after upgrade. Keep `sharp` as a compatibility alias or provide an explicit migration path.
  - confidence: 0.86
