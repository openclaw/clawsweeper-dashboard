## Summary

Keep open: current main has not implemented this sharp replacement, and the PR is stale against the current plugin-owned image runtime. Main still routes image operations through media-understanding-core, that plugin still depends on sharp, and the installer/docs still expose sharp/libvips behavior, while the PR removes compatibility guardrails and changes media semantics before that graph is migrated.

## What This Changes

The PR replaces root sharp usage with @napi-rs/image in media image operations and related image tests, docs, package metadata, and lockfile entries.

## Best Possible Solution

Rebase or rewrite the migration around media-understanding-core, keep sharp compatibility until all plugin/runtime/docs/install ownership is migrated, and add regression coverage for EXIF format preservation plus alpha/no-upscale behavior.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main still delegates to a sharp-backed media-understanding-core runtime and documents/installs sharp behavior, while the PR diff forces JPEG for EXIF normalization and removes sharp guardrails. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The dependency swap may be a valid direction, but this branch targets an older core-owned implementation and removes compatibility before the current plugin-owned sharp graph is migrated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve source format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: normalizeExifOrientation now returns transformer.rotate().jpeg(90) for every non-sips input. The current sharp-backed contract uses rotate().toBuffer(), so PNG/WebP callers can receive lossy JPEG bytes and lose alpha after requesting only orientation normalization.
  - confidence: 0.95
- **[P2] Keep legacy backend override value working:** `src/media/image-ops.ts:28`
  - body: Changing the Bun/macOS non-sips override from OPENCLAW_IMAGE_BACKEND=sharp to only napi-rs makes existing deployments with the documented sharp override fall back to the sips path after upgrade. Keep sharp as a compatibility alias or add an explicit migration path.
  - confidence: 0.9
- **[P2] Keep sharp build policy until the graph is sharp-free:** `package.json:964-965`
  - body: The diff removes sharp from pnpm.onlyBuiltDependencies and deletes installer docs, but current main still declares sharp in media-understanding-core and installer scripts still pass SHARP_IGNORE_GLOBAL_LIBVIPS. Removing this approval/doc path before migrating the plugin dependency can break installs and remove the documented workaround.
  - confidence: 0.88
