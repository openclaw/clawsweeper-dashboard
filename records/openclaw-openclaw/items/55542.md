## Summary

Keep open: current main has not replaced sharp with @napi-rs/image, and the PR still has merge-blocking media behavior and install-policy regressions that need maintainer direction before repair.

## What This Changes

The PR removes sharp from root install docs/package metadata, adds @napi-rs/image, and ports core image operations plus image-related tests to Transformer APIs.

## Best Possible Solution

Rebase or replace the migration in media-understanding-core, preserve sharp backend/config/install compatibility until the graph and docs are actually sharp-free, and add media format/alpha regression coverage.

## Reproduction Assessment

Yes for the PR defects: the diff, current main media delegation, and upstream @napi-rs/sharp type contracts show the format/backend/build-policy mismatches by inspection. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The dependency swap may be worth pursuing, but the current branch changes public media behavior and removes sharp install guardrails before the current plugin-owned dependency graph is ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve input format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: `normalizeExifOrientation` still returns `transformer.rotate().jpeg(90)` for every non-sips input. The current sharp-backed contract uses `rotate().toBuffer()`, whose default output format matches the input, so PNG/WebP callers can now receive lossy JPEG bytes and lose alpha after requesting only orientation normalization.
  - confidence: 0.95
- **[P2] Keep the sharp backend override compatible:** `src/media/image-ops.ts:28`
  - body: Changing the Bun/macOS non-sips override from `OPENCLAW_IMAGE_BACKEND=sharp` to only `napi-rs` makes existing deployments with the documented `sharp` override fall back to the sips path after upgrade. Keep `sharp` as a compatibility alias or add an explicit migration path.
  - confidence: 0.9
- **[P2] Keep sharp build policy until the graph is sharp-free:** `package.json:964-965`
  - body: The diff removes `sharp` from `pnpm.onlyBuiltDependencies` and drops installer docs, but current main still declares sharp in `extensions/media-understanding-core/package.json` and installers still pass `SHARP_IGNORE_GLOBAL_LIBVIPS`. Removing this approval/doc path before migrating the plugin dependency can break installs and remove the documented workaround.
  - confidence: 0.86
