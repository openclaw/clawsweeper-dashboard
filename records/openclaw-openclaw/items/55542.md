## Summary

Keep open. Current main has not implemented this dependency swap: the live image path still delegates to the bundled media-understanding-core plugin, which depends on sharp, and the PR branch still has merge-blocking behavior regressions already called out in the review discussion.

## What This Changes

The PR removes the root sharp dependency and related install docs, adds @napi-rs/image with lockfile/build-approval changes, and ports core image operations plus image-related tests from sharp to @napi-rs/image Transformer APIs.

## Best Possible Solution

Rebase or replace the migration against current main by treating media-understanding-core as the owner of the image backend, preserving the plugin SDK media contract, keeping OPENCLAW_IMAGE_BACKEND=sharp as a compatibility alias or documenting a migration, preserving input format during EXIF normalization, and removing sharp docs/build approvals only after the dependency graph is actually sharp-free.

## Reproduction Assessment

Yes for the PR regressions: the diff and dependency docs are enough to reproduce the API mismatch and EXIF format-loss paths by inspection, and current main confirms the legacy sharp behavior being changed. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The proposed direction may be worth pursuing, but this branch is not the best current fix because it ports an older core image implementation while main now owns the sharp backend in media-understanding-core and it removes compatibility/install guardrails too early.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve source format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: `normalizeExifOrientation` now returns `transformer.rotate().jpeg(90)` for every non-sips input. The previous sharp path used `rotate().toBuffer()`, so PNG/WebP callers can now receive lossy JPEG bytes and lose alpha even though they only requested orientation normalization.
  - confidence: 0.94
- **[P2] Use positional resize arguments with Transformer:** `src/media/image-ops.ts:344`
  - body: `@napi-rs/image` documents `Transformer.resize(width, height?, filterType?)`, but this branch passes a Sharp-style options object in the JPEG and PNG paths. Runtime image optimization can throw or fail to resize oversized media instead of returning capped output.
  - confidence: 0.9
- **[P2] Keep the legacy image backend override working:** `src/media/image-ops.ts:28`
  - body: Changing the Bun/macOS non-sips override from `OPENCLAW_IMAGE_BACKEND=sharp` to only `napi-rs` makes existing deployments with the documented `sharp` override fall back to sips after upgrade. Keep `sharp` as a compatibility alias or provide an explicit migration path.
  - confidence: 0.87
- **[P2] Keep sharp build approval while sharp remains resolved:** `package.json:964`
  - body: The PR removes `sharp` from `pnpm.onlyBuiltDependencies`, but the lockfile still resolves sharp through Baileys and current main also has the media-understanding-core sharp runtime path. Removing the approval before the graph is sharp-free can break installs that still need sharp's native package setup.
  - confidence: 0.82
