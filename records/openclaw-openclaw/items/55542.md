## Summary

Keep open: the PR is not merge-ready because it targets the old root image runtime rather than current `media-understanding-core`, still has a format-loss regression in EXIF normalization, lacks real behavior proof, and changes native dependency trust/build policy.

## What This Changes

The PR replaces root `sharp` usage with `@napi-rs/image` across image ops, related tests, install docs, package metadata, and the lockfile.

## Best Possible Solution

If maintainers approve replacing `sharp`, migrate the `media-understanding-core` runtime/dependency and workspace build policy while preserving format, alpha, and legacy backend override behavior.

## Reproduction Assessment

Yes for the PR review defects by source inspection: the branch edits the old root image implementation while current main loads plugin-owned image ops, and line 299 re-encodes orientation normalization to JPEG. I did not run tests because this review is read-only.

## Solution Assessment

No. The branch is not the best path because current main owns the image runtime in `media-understanding-core`; the safer path is a plugin-owned migration after maintainer dependency approval and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move the image dependency swap into the media plugin:** `package.json:862`
  - body: Current main loads active image ops from `media-understanding-core`, whose package still declares and imports `sharp`. Adding `@napi-rs/image` at the root package while changing the old root implementation will not replace the active runtime after rebase; migrate the plugin-owned runtime/dependency and build policy instead.
  - confidence: 0.9
- **[P2] Preserve format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: The non-sips path now returns `transformer.rotate().jpeg(90)`, so PNG/WebP inputs can become lossy JPEG and lose alpha even though the helper only promises orientation normalization. Preserve the input format or make the conversion an explicit API contract.
  - confidence: 0.93
- **[P2] Keep the legacy backend override value working:** `src/media/image-ops.ts:28`
  - body: Current main treats `OPENCLAW_IMAGE_BACKEND=sharp` as the non-sips override on Bun/macOS, but this condition changes the override to only `napi-rs`. Existing installs with the old value would fall back to sips after upgrade; keep `sharp` as a compatibility alias or provide an explicit migration path.
  - confidence: 0.84
