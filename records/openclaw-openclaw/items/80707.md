## Summary

This PR is a narrow, plausible fix for a real startup race and should stay open for maintainer review rather than cleanup-close. Current main can still treat an empty post-restart cron active-job set as authoritative before cron startup reconciliation completes, and the branch adds a focused authority bit with regression coverage plus terminal proof.

## What This Changes

The branch adds cron active-job liveness authority tracking, marks cron startup reconciliation boundaries, and teaches task-registry maintenance to preserve mapped cron tasks while cron liveness is non-authoritative.

## Best Possible Solution

Land this focused liveness-authority guard after maintainer review and required exact-head checks, while leaving recovery of already-terminal lost cron rows to https://github.com/openclaw/openclaw/pull/75036.

## Reproduction Assessment

Yes. Source inspection shows current main starts authoritative task maintenance before cron.start and then trusts an empty activeJobIds set for mapped stale cron tasks, making the false lost path reachable during startup.

## Solution Assessment

Yes. The PR adds a narrow authority bit owned by cron active-job state and consumes it only in the cron backing check, preserving the existing lost behavior once startup reconciliation exits.
