## Summary

## What This Changes

The PR adds cron active-job liveness authority tracking and uses it so task maintenance preserves mapped cron task rows until cron startup reconciliation exits.

## Best Possible Solution

Merge the narrow liveness-authority guard after maintainer acceptance; if maintainers want more confidence, request one full gateway-restart proof rather than changing the fix shape.

## Reproduction Assessment

Yes. Source inspection shows current main configures gateway task maintenance as runtime-authoritative before cron startup, and the cron backing-session check treats an empty `activeJobIds` set as missing backing state.

## Solution Assessment

Yes, this appears to be the best narrow fix: cron startup owns when the process-local active-job set becomes authoritative, while task maintenance only preserves mapped cron rows during that window. The remaining choice is proof/contract acceptance, not a different obvious code path.
