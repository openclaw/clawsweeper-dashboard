## Summary

Keep open for maintainer review: the PR addresses a real cron/task-registry startup race and I found no concrete patch defect, but it intentionally changes task ledger liveness semantics during cron startup.

## What This Changes

The PR adds a cron active-job liveness authority flag, toggles it around cron startup, and teaches task-registry maintenance to preserve mapped cron tasks while that liveness is non-authoritative.

## Best Possible Solution

Land the guard after a maintainer accepts the startup liveness contract, or request one full gateway-restart proof if the focused state-dir proof is not enough.

## Reproduction Assessment

Yes, source-reproducible: current main starts authoritative task maintenance before cron startup and trusts only activeJobIds for mapped cron liveness once runtime-authoritative. I did not run a live gateway restart in this read-only review.

## Solution Assessment

Yes, this is an acceptable owner-boundary fix: the new state lives with the cron active-job singleton and task maintenance consumes it only for mapped cron tasks. The durable run-log recovery alternative already exists, but it cannot prevent the pre-finish startup false-positive window by itself.
