## What This Changes

The PR adds a cron active-job liveness authority flag, toggles it around cron startup, and updates task-registry maintenance/tests to preserve mapped cron tasks while startup liveness is non-authoritative.

## Best Possible Solution

Land the guard after maintainer acceptance of the startup liveness/audit-ledger contract, or request one full gateway-restart proof if isolated terminal proof is not enough.

## Reproduction Assessment

Yes, source-reproducible: current main starts authoritative task maintenance before post-ready cron startup and trusts only activeJobIds for mapped cron liveness once runtime-authoritative. I did not run a live gateway restart in this read-only review.

## Solution Assessment

Yes, this is an acceptable owner-boundary fix: the new state lives with the cron active-job singleton and task maintenance consumes it only for mapped cron tasks. Durable cron history recovery exists, but it does not cover the pre-start liveness window by itself.
