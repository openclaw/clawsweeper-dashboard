## Summary

Keep open for maintainer review: current main still trusts the empty process-local cron active-job set during authoritative task maintenance, while this PR adds a focused startup liveness guard with real behavior proof. The remaining question is maintainer acceptance of the temporary task-ledger session-state semantics during cron startup, not a code blocker I can point to.

## What This Changes

This PR adds cron active-job liveness authority tracking, toggles it around cron startup, and makes task-registry maintenance preserve mapped cron task rows until startup reconciliation exits.

## Best Possible Solution

Land the narrow liveness-authority guard after maintainers accept the startup-window task-ledger behavior, or ask for one full gateway-restart proof before merge.

## Reproduction Assessment

Yes from source inspection, without executing a live repro: current main starts authoritative task maintenance before post-ready cron startup and hasBackingSession relies on the empty activeJobIds set for cron rows.

## Solution Assessment

Yes, this is the narrowest maintainable fix I found: expressing active-job authority in cron state is tighter than disabling cron lost-marking broadly or delaying all task maintenance, while still restoring normal lost-marking after startup exits.
