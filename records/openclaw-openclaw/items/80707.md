## Summary

Keep open: current main still trusts the process-local cron active-job set for authoritative cron task liveness, and this PR is the focused startup-window guard for that remaining false-lost path. I found no discrete patch defect, but merging intentionally changes task-ledger session-state semantics and needs maintainer acceptance.

## What This Changes

The PR adds a cron active-job liveness authority flag, marks cron startup reconciliation as non-authoritative until `cron.start()` exits, and teaches task maintenance to preserve mapped cron task rows only during that window.

## Best Possible Solution

Land this focused guard only after maintainers accept the startup liveness contract, and keep broader terminal-lost cron recovery tracked by the canonical issue rather than expanding this PR.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: task maintenance can run as runtime-authoritative while cron startup has not yet populated `activeJobIds`, so current main can mark mapped stale cron rows `lost`. I did not run a live gateway restart reproduction in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance: the PR is a narrow owner-boundary fix for the startup window and avoids broad durable-store changes. The safer alternative is to require full restart proof or replace it with a durable recovery design if maintainers reject the liveness-authority contract.
