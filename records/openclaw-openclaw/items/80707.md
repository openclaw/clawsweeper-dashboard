## What This Changes

The PR adds a cron active-job liveness authority flag, marks cron startup reconciliation as non-authoritative until `start()` exits, and makes task maintenance preserve mapped stale cron tasks during that window.

## Best Possible Solution

Rebase the guard onto current main's cron active-marker generation model, keep the startup liveness window narrow, and land it alongside focused regression coverage after task/session-state owner review.

## Reproduction Assessment

Yes. Source inspection shows current gateway task maintenance can be runtime-authoritative before cron active liveness has its own reconciled signal, and `hasBackingSession()` still uses process-local cron active-job state for lost marking.

## Solution Assessment

Yes, directionally, but not merge-ready as-is. A narrow cron liveness-authority guard is the right layer for the pre-start false-lost race; the branch now needs a current-main rebase that preserves newer active-marker and startup-stop behavior.
