## What This Changes

The PR adds cron active-job liveness authority state, toggles it around cron startup, and updates task-registry maintenance/tests so mapped cron tasks are not marked lost while startup liveness is non-authoritative.

## Best Possible Solution

Land the narrow liveness-authority guard after maintainer acceptance of the startup ledger contract and required checks; keep durable historical cron-row recovery in the canonical related work.

## Reproduction Assessment

Yes, from source inspection: current main makes authoritative cron task backing depend on process-local activeJobIds, while gateway startup can start task maintenance before cron.start finishes. I did not execute a live restart repro in this read-only review.

## Solution Assessment

Yes, this is an acceptable narrow fix: the authority guard is scoped to cron startup and restores normal lost-task reconciliation in finally. The broader historical durable-recovery problem remains separate.
