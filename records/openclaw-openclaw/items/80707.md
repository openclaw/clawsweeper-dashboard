## Summary

Keep open for normal review and merge: current main still has the startup race, the PR targets the right path, proof is sufficient, and I found no blocking correctness or security issue.

## What This Changes

The branch adds cron active-job liveness authority tracking around startup and uses it in task maintenance so mapped cron tasks are preserved only while liveness is non-authoritative.

## Best Possible Solution

Land this narrow startup-liveness guard after maintainer review and final required checks, while leaving durable recovery of already-terminal lost cron rows to https://github.com/openclaw/openclaw/pull/75036.

## Reproduction Assessment

Yes. Source inspection shows current main starts authoritative task maintenance before cron.start and then trusts an empty activeJobIds set for mapped stale cron tasks, so the false lost path is reachable during startup.

## Solution Assessment

Yes. The PR adds a narrow liveness authority bit owned by the cron active-job state and consumes it only in the cron task-maintenance backing check, preserving existing lost behavior once startup exits.
