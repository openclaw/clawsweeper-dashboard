## Summary

Keep open: current main and v2026.6.1 still trust only the process-local cron active-job set when reconciling stale cron task rows, while this PR adds a narrow startup liveness guard with focused coverage and sufficient terminal proof. I did not find a blocking patch defect; the remaining decision is maintainer acceptance of the task-ledger/session-state contract and proof depth.

## What This Changes

The PR adds a cron active-job liveness authority flag, toggles it around cron startup, and teaches task-registry maintenance to preserve mapped cron task rows while startup liveness is non-authoritative.

## Best Possible Solution

Land the guard if maintainers accept that mapped cron rows should be retained until cron startup declares active-job liveness authoritative; otherwise replace it with a durable cron-run-state reconciliation design.

## Reproduction Assessment

Yes, source-reproducible: current main starts authoritative task maintenance before post-ready cron startup and trusts only activeJobIds for mapped cron liveness. I did not run a live gateway restart in this read-only review.

## Solution Assessment

Yes, this is an acceptable owner-boundary fix: the new state lives with the cron active-job singleton and task maintenance consumes it only for mapped cron tasks. Durable cron history recovery exists, but it does not cover the pre-start empty-activeJobIds window by itself.
