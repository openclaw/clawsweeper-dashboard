## Summary

Keep open. The code direction appears sound for the linked fallback-delivery bug, but this external PR is not merge-ready because its after-fix proof is tests/mocks only and the live TUI/WebChat behavior has not been shown.

## What This Changes

The PR emits fallback progress lifecycle events, defers embedded TUI lifecycle-error finalization during fallback recovery, and adds regression coverage for fallback response delivery after an initial timeout/error.

## Best Possible Solution

Land the narrow fallback lifecycle/delivery fix once a redacted real TUI/WebChat timeout-to-fallback run proves the fallback answer is delivered; keep any unconditional recovery notification as a separate UX decision.

## Reproduction Assessment

Yes from source and the linked bug evidence, but not from a live run in this review. The current-main path emits per-attempt lifecycle errors while fallback success can arrive later, and WebChat/TUI need a non-terminal progress signal to avoid premature terminal handling.

## Solution Assessment

Yes for the code direction: emitting a fallback progress lifecycle event and deferring TUI error finalization is narrower than a broad new notification system. It is not merge-ready until real behavior proof is added and the exact-head gates are resolved or waived as unrelated.
