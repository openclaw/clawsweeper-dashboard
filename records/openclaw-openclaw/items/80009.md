## Summary

Keep open: this is a focused, recent PR for a source-provable fallback delivery bug that is not fully implemented on current main. ## What This Changes

The PR emits `fallback_step` lifecycle events from auto-reply fallback, defers embedded TUI lifecycle-error finalization with a 15s grace, adds fallback delivery regression tests, and switches a Bedrock test to the public plugin SDK test-env import.

## Best Possible Solution

Land the narrow lifecycle-progress fix once maintainers accept the controlled proof or the contributor adds redacted real TUI/WebChat/provider fallback output showing the final fallback response delivered.

## Reproduction Assessment

Yes from source, not from a live run. Current main's embedded TUI handles lifecycle `error` as terminal while the fallback runner has an unused `onFallbackStep` hook, and the linked bug report shows a successful fallback with no user-visible delivery.

## Solution Assessment

Yes, with the proof caveat. Emitting fallback progress through the existing model-fallback observation hook and aligning embedded TUI with the existing Gateway lifecycle-error grace is a narrow maintainable fix without adding config or broad rediscovery.
