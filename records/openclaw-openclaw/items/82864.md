## Summary

Keep open: current main’s docs and tests define canonical `openai/*` run-provider refs with `openai-codex` used as the Codex harness auth provider, while this branch changes a helper whose caller can mutate the active run provider. The PR also still lacks inspectable after-fix real behavior proof.

## What This Changes

The PR changes `resolveSelectedOpenAIPiRuntimeProvider` to return `openai-codex` for OpenAI Codex harness selections and adds two routing unit tests.

## Best Possible Solution

Preserve `openai/*` as the model/run provider for native Codex harness turns, route auth/profile selection through `openai-codex`, and add integrated runner coverage plus redacted real-run proof before merge.

## Reproduction Assessment

Yes for the PR regression: source inspection shows the new branch returns `openai-codex` for Codex harness runs, and the caller can assign that as the active provider. The original 401 report is not high-confidence reproducible from the PR body alone because no inspectable runtime proof is attached.

## Solution Assessment

No: changing `resolveSelectedOpenAIPiRuntimeProvider` for every Codex runtime run is not the narrowest maintainable fix. The safer direction is to keep provider rewriting limited to the PI compatibility path and handle Codex harness auth through the runtime auth/profile-selection contract.
