## Summary

Keep open: the proposed routing change conflicts with the current Codex harness contract by changing the active run provider instead of only selecting the Codex auth provider, and the PR still lacks viewable real behavior proof.

## What This Changes

The PR changes OpenAI `harnessRuntime: "codex"` selections to return `openai-codex` from `resolveSelectedOpenAIPiRuntimeProvider` and adds two focused routing tests.

## Best Possible Solution

Preserve canonical `openai/*` run/provider refs for native Codex harness turns, route `openai-codex` through the existing auth-plan/profile-selection seam, and add integrated runner coverage for the reported 401 path.

## Reproduction Assessment

Yes, source-reproducible for the PR regression: current `runEmbeddedPiAgent` assigns the helper's alternate provider back to the active provider, and existing Codex harness tests assert that forced `openai/*` Codex runs stay on provider `openai` while forwarding `openai-codex` auth.

## Solution Assessment

No, this is not the best fix because it rewrites the run provider for all OpenAI Codex runtime selections. The narrower maintainable direction is to keep provider rewriting limited to PI compatibility and fix any missing Codex OAuth selection in the harness auth/profile path.
