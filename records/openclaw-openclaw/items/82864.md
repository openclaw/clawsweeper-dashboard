## Summary

Keep open: this is a focused routing fix attempt, but the proposed change appears to move native Codex harness runs away from the documented canonical OpenAI provider path, and the PR still lacks real behavior proof from a real Codex runtime setup.

## What This Changes

The PR changes OpenAI Codex routing so `provider: "openai"` with `harnessRuntime: "codex"` resolves to `openai-codex`, and adds two focused unit tests for OpenAI and non-OpenAI providers.

## Best Possible Solution

Preserve the documented split: keep native Codex harness model/runtime refs under `openai/*`, route auth/profile selection through `openai-codex`, and add integrated run coverage before merge.

## Reproduction Assessment

Yes, source-reproducible for the PR regression: the changed helper feeds a caller that mutates the active run provider, while current docs and tests require Codex harness attempts to keep provider `openai` and use `openai-codex` only for auth. I did not run the test suite because this review is constrained to read-only inspection.

## Solution Assessment

No: returning `openai-codex` from `resolveSelectedOpenAIPiRuntimeProvider` for all Codex runtime runs is not the narrowest maintainable fix. The safer direction is to keep provider rewriting limited to the PI compatibility path and cover the real Codex harness route at the `runEmbeddedPiAgent` level.
