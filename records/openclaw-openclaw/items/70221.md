## Summary

Keep open. Current main and the latest shipped tag still carry height-only `[embed]` parsing and iframe rendering, while `gateway.controlUi.chatMessageMaxWidth` only changes the global chat message column and does not satisfy the requested per-embed width control.

## What This Changes

Add a per-embed `width` parameter to the Control UI `[embed]` directive so design previews can render wider than the chat message column.

## Best Possible Solution

Add typed, validated `preferredWidth` handling through shortcode parsing, normalized preview types, message normalization, iframe rendering/CSS, prompt/docs, and focused tests while preserving current sandboxing and URL allow rules.

## Reproduction Assessment

Yes. A Control UI assistant message like `[embed ref="cv_demo" title="Demo" height="600" width="900" /]` follows the existing shortcode parser path, but current source and the latest release only parse/render `height`, so `width` is dropped before UI rendering.

## Solution Assessment

Yes for the feature direction, with a narrower boundary than arbitrary inline styles. The maintainable solution is a validated `preferredWidth` presentation hint carried through the existing embed contract, while the existing `chatMessageMaxWidth` setting remains the global chat-column control.
