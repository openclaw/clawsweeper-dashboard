## Summary

Keep open. Current main still treats `[embed]` iframe sizing as height-only: width is not parsed into the canvas preview contract, not carried through normalized UI/sidebar types, and the iframe renderer/CSS still fills the message container. The issue is narrow, unprotected, and has no paired implementation PR in the supplied context, so it is a good ClawSweeper repair candidate.

## What This Changes

The issue requests a `width` parameter for `[embed]` so Control UI hosted iframe previews can render wider than the chat message column.

## Best Possible Solution

Add a bounded `preferredWidth` presentation hint end to end for hosted Control UI embeds: parse `width`, carry it through the canvas preview and sidebar/UI types, render it as a responsive preferred iframe/panel width, and document/test the behavior without changing sandboxing or non-web channel semantics.

## Reproduction Assessment

Yes. A static reproduction is `[embed ref="cv_demo" title="Demo" height="600" width="900" /]`: current main will parse and apply the height, but the width is ignored and the iframe remains `width: 100%` of the message container.

## Solution Assessment

Yes. The narrowest maintainable solution is to extend the existing canvas preview contract with a bounded width hint rather than accepting arbitrary inline styles or changing iframe security policy.
