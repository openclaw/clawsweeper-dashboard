## Summary

Keep open. Current main still has height-only `[embed]` parsing/rendering, while the shipped `chatMessageMaxWidth` setting only changes the global chat message column and does not provide a per-embed width contract.

## What This Changes

The issue requests a per-embed width parameter for the Control UI `[embed]` directive so iframe previews can render wider than the chat column.

## Best Possible Solution

Add typed, validated `preferredWidth` handling through shortcode parsing, normalized preview types, message normalization, iframe rendering/CSS, prompt/docs, and focused tests while preserving current sandboxing and URL allow rules.

## Reproduction Assessment

Yes. A Control UI assistant message like `[embed ref="cv_demo" title="Demo" height="600" width="900" /]` follows the existing parser path, but current source only reads and renders `height`; `width` has no parser, type, normalizer, renderer, CSS, docs, or test path.

## Solution Assessment

Yes for the feature direction, with a safer boundary than the issue sketch. The maintainable fix is a bounded `preferredWidth` presentation hint carried through existing embed contracts, not arbitrary inline style passthrough or sandbox policy changes.
