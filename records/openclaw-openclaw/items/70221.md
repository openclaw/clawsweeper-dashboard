## Summary

Keep open. Current main still has height-only `[embed]` handling and a separate global chat-message max-width setting, so the requested per-embed width contract is not implemented.

## What This Changes

Request that Control UI `[embed]` directives accept a per-embed width parameter for inline iframe previews.

## Best Possible Solution

Add a typed, validated `preferredWidth` presentation hint end to end for hosted Control UI embeds while preserving existing sandboxing, URL allow rules, and the global chat-message width setting.

## Reproduction Assessment

Yes. Source inspection on current main shows an assistant message like `[embed ref="cv_demo" title="Demo" height="600" width="900" /]` would preserve height but has no parser, type, normalizer, renderer, CSS, docs, or test path for width.

## Solution Assessment

Yes for the feature direction, with a safer implementation boundary. The maintainable solution is a bounded `preferredWidth` field carried through existing embed contracts, not arbitrary style passthrough or sandbox policy changes.
