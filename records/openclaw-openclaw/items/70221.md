## Summary

Keep open. Current main still ignores per-embed width: the parser, preview types, normalizer, renderer, CSS, docs, and focused tests carry height only, while the merged chat-message max-width setting is a separate global layout control.

## What This Changes

The issue requests a typed `width` parameter for Control UI `[embed]` directives so individual iframe previews can render wider than the chat message column.

## Best Possible Solution

Add a typed, validated `preferredWidth` presentation hint end to end for hosted Control UI embeds while preserving existing sandboxing, URL allow rules, and the global chat-message width setting.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: an assistant message like `[embed ref="cv_demo" title="Demo" height="600" width="900" /]` would preserve height but has no parser, type, normalizer, renderer, or CSS path for width.

## Solution Assessment

Yes for the feature direction, with a safer implementation boundary. The maintainable solution is a bounded `preferredWidth` field carried through existing embed contracts, not arbitrary style passthrough or sandbox policy changes.
