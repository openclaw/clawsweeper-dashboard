## Summary

Keep open. Current main still ignores per-embed width: the parser, typed preview shape, normalizer, renderer, CSS, docs, and focused tests carry only height, while the merged chat-message max-width setting is a separate global layout control.

## What This Changes

Add a user-facing `width` parameter to Control UI `[embed ...]` directives so individual iframe previews can request a wider desktop-sized presentation.

## Best Possible Solution

Add a typed, validated `preferredWidth` presentation hint end to end for hosted Control UI embeds while preserving existing sandboxing, URL allow rules, and the global chat-message width setting.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: an assistant message like `[embed ref="cv_demo" title="Demo" height="600" width="900" /]` preserves height but has no parser, type, normalizer, renderer, or CSS path for width.

## Solution Assessment

Yes for the feature direction, with a safer implementation boundary. The maintainable solution is a bounded `preferredWidth` field carried through existing embed contracts, not arbitrary style passthrough or sandbox policy changes.
