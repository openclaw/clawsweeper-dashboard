## Summary

Keep open: the bug is valid, but this PR needs a tighter fix and stronger proof because it makes the direct migrate runtime calls persisted-aware while another migration-provider ownership lookup remains demand-only, and the supplied terminal proof does not exercise a global-only provider setup.

## What This Changes

The PR adds an optional persisted-registry mode to manifest contract resolution, enables it for migration-provider runtime lookup, and adds a regression test for a persisted global Codex provider.

## Best Possible Solution

Land one canonical resolver-level fix that makes migration-provider contract ownership and runtime resolution use the default persisted metadata snapshot, carrying over this PR's regression coverage if useful.

## Reproduction Assessment

Yes. Source inspection shows current main passes preferPersisted:false through the standalone manifest-contract lookup and registry snapshot loading then disables persisted install-record reads; the linked issue also has shipped-release user proof on 2026.6.6.

## Solution Assessment

No. The direct migrate calls are a plausible partial fix, but the best fix is the canonical resolver/default behavior in https://github.com/openclaw/openclaw/pull/89612 or an equivalent update that also covers post-install ownership lookup.
