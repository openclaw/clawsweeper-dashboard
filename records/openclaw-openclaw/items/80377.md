## Summary

Keep open for maintainer handling: current main encodes preserve-on-reset behavior as intentional, and the same-author linked cleanup PR remains open while explicitly reframing the reported behavior as by design rather than a runtime bug.

## What This Changes

The issue asks for `/reset` to clear session-level thinking, verbose, reasoning, TTS, model, provider, and auth-profile overrides so agent/global defaults are consulted again.

## Best Possible Solution

Keep the current preserve-on-reset contract unless maintainers intentionally redefine it; handle the misleading no-op fast-path ternaries through the linked cleanup PR.

## Reproduction Assessment

Yes: source and tests show `/reset` preserves user-set overrides on current main. That reproduces the reported observation, but the same evidence shows it is the currently encoded behavior contract.

## Solution Assessment

No: clearing those fields to force defaults would conflict with current regression tests and the merged preserve-on-reset change in https://github.com/openclaw/openclaw/pull/10881. The safer path is maintainer product judgment plus optional cleanup via https://github.com/openclaw/openclaw/pull/80414.
