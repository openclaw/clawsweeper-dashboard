## Summary

Keep open. Current main treats preservation across `/new` and `/reset` as intentional for user-set behavior overrides and user/legacy model/auth selections, while the no-op fast-path ternaries are still confusing; changing that contract needs maintainer product judgment and should be coordinated with https://github.com/openclaw/openclaw/issues/51251 for the model override portion.

## What This Changes

Requests that `/reset` clear session-level thinking, verbose, reasoning, TTS, model, provider, and auth overrides so configured defaults apply again, or simplify the no-op ternaries if preservation is intended.

## Best Possible Solution

Make an explicit maintainer decision for reset semantics, then either simplify/document preservation in both session paths or update full session, fast path, gateway reset, docs, and tests to clear the chosen override fields consistently.

## Reproduction Assessment

Yes. Source inspection and current tests reproduce that reset preserves these session override fields; the unresolved part is whether that should remain the expected behavior.

## Solution Assessment

No, not as a direct bug fix. Replacing the reset branches with `undefined` would conflict with current preservation tests/docs and needs a product decision before implementation.
