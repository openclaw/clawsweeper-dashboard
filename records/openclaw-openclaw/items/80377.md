## Summary

Keep open. Current main still preserves explicit session-level behavior overrides across `/reset`, and the exact no-op ternary pattern still exists in the fast test bootstrap path, but production tests and comments now encode a deliberate user-vs-auto override policy that needs maintainer judgment before changing reset semantics.

## What This Changes

The issue asks for `/reset` to clear persisted thinking, verbose, reasoning, TTS, model, provider, and auth-profile session overrides so agent/global defaults apply again.

## Best Possible Solution

Decide and document reset semantics, then either update production and fast-bootstrap code/tests to clear explicit overrides or keep the current preserve policy and replace the no-op fast-path ternaries with explicit mirrored logic.

## Reproduction Assessment

Yes from source inspection: current main preserves explicit behavior overrides on `/reset`, and the fast bootstrap still contains the exact no-op ternary pattern. I did not run the gateway because this was a read-only review, but the assignments and tests are direct.

## Solution Assessment

Unclear. Clearing every field is not obviously the best fix because current main deliberately preserves user-set behavior overrides and user-sourced model/auth selections while clearing auto-sourced pins; maintainers should choose the intended reset contract first.
