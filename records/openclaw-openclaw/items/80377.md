## Summary

Keeping this open for maintainer judgment: current main intentionally preserves user-set behavior overrides and user-selected model/auth overrides across `/reset`, while clearing auto-sourced selections. The report is a reset-policy disagreement rather than a clear bug, and the same author still has an open cleanup PR tied to this issue.

## What This Changes

The issue asks for `/reset` to clear session-level thinking, verbose, reasoning, TTS, model, provider, and auth-profile overrides so configured agent defaults are consulted again.

## Best Possible Solution

Keep the current user-vs-auto reset policy unless maintainers deliberately choose new reset semantics; if they do, change `initSessionState`, docs, and tests directly rather than treating the fast-path ternaries as the root cause.

## Reproduction Assessment

Yes, the preservation behavior is source-reproducible: `/reset` creates a new session while carrying user-set behavior overrides and selected user model/auth overrides forward. That reproduces the observation, but it does not establish a bug because current tests and comments expect that behavior.

## Solution Assessment

No, not as a bug fix: clearing every listed field would regress the current tested policy and the merged auto-vs-user reset distinction. The safer path is a maintainer product decision, with any cleanup PR kept separate from behavior changes.
