## Summary

Current `main` no longer supports treating this as a narrow bug fix: the production reset path intentionally preserves user-set behavior overrides, and model/provider/auth reset semantics already distinguish user-sourced choices from auto-sourced fallback pins. The issue should stay open for maintainer judgment while the paired cleanup PR remains open.

## What This Changes

The issue requests changing `/reset` so persisted session-level thinking, verbosity, reasoning, TTS, model, provider, and auth-profile overrides are cleared and configured agent defaults apply again.

## Best Possible Solution

Keep the current reset policy unless maintainers explicitly choose a new full-reset behavior or command; handle the misleading fast-path ternaries through the paired cleanup PR or close both as by-design.

## Reproduction Assessment

Yes for the observed behavior: source and tests show `/reset` preserves user-set behavior overrides and only clears auto-sourced model/auth selections. That is not high-confidence evidence of a bug because current main defines this as intentional behavior.

## Solution Assessment

No as an automated fix: clearing all session overrides would reverse an intentionally merged reset policy and needs maintainer product judgment. The narrow cleanup is to remove misleading no-op ternaries without changing runtime behavior.
