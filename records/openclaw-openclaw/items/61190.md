## Summary

Keep open. Current main still has the resolver bug this PR targets, and the contributor patch is useful, but it is not ready as written because it normalizes authoritative `channel` directory entries to `group` and lacks the required changelog entry for a user-facing fix.

## What This Changes

The PR changes outbound target resolution to derive directory-match target kind from the matched directory entry and adds regression tests for a user entry returned through group lookup plus channel-entry normalization.

## Best Possible Solution

Land the narrow resolver fix after preserving authoritative directory entry kinds for `user`, `group`, and `channel` in both single-match and non-error ambiguous paths, with focused regression coverage and a changelog entry crediting the contributor.

## Reproduction Assessment

Yes. A focused unit scenario can reproduce the current-main bug by returning `{ kind: "user", id: "staff_bob", name: "Bob" }` from `listGroups` for bare input `Bob`; current main returns the guessed `group` kind. The PR's channel regression is reproducible with a `{ kind: "channel" }` directory entry plus a channel-preferred or channel-inferred lookup where routing distinguishes `channel` from `group`.

## Solution Assessment

No, not as currently written. The PR is the right narrow surface for the user/group bug, but the helper should preserve `entry.kind === "channel"` and the PR should add the required changelog entry before merge.
