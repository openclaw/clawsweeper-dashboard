## Summary

Keep open: this PR addresses the real user-via-group directory bug, but the current patch still coerces authoritative `channel` directory entries to `group`, which conflicts with the resolved-target contract and downstream session routing; it also needs the required changelog entry.

## What This Changes

The PR adds resolver logic to derive directory-match target kind from the matched entry and adds target-resolver regression coverage for user entries returned by group lookup plus channel normalization.

## Best Possible Solution

Land the narrow resolver fix after preserving authoritative `user`, `group`, and `channel` directory entry kinds in both directory-match result paths, with focused regression coverage and an active-version changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: return a `{ kind: "user" }` entry from `listGroups` for a bare name and current main returns the guessed `group`; a `{ kind: "channel" }` entry shows why preserving the matched kind matters for routing.

## Solution Assessment

No, not as currently written. The PR is on the right narrow resolver surface, but the helper should preserve `entry.kind === "channel"` and the PR should add the required changelog entry before merge.
