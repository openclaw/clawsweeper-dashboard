## Summary

Keep this PR open. The user/group resolver fix is narrow and useful, but the proposed helper still collapses matched `channel` directory entries to `group` even though the current channel contract and outbound session routing treat `channel` as a distinct resolved target kind.

## What This Changes

The PR changes outbound target resolution to derive directory-match target kind from the matched entry and adds regression coverage for a user entry returned through group lookup plus a channel-entry normalization case.

## Best Possible Solution

Land the narrow resolver/test fix after preserving authoritative directory entry kind for `user`, `group`, and `channel` in both single-match and non-error ambiguous-match paths. If channel-to-group normalization is intentional, document it as a resolver/session-routing contract and cover the downstream routing behavior explicitly.

## Reproduction Assessment

Yes. A focused unit scenario can return `{ kind: "user", id: "staff_bob", name: "Bob" }` from `listGroups` for bare input `Bob`; current main returns the guessed `group` kind. The PR's remaining issue is reproducible with a `{ kind: "channel" }` directory entry and a plugin whose routing distinguishes `channel` from `group`.

## Solution Assessment

No, not as currently written. The PR uses the right narrow resolver surface for the user/group bug, but the helper should preserve `entry.kind === "channel"` instead of collapsing it to `group` unless maintainers explicitly change that public routing contract.
