## Summary

Keep open: current main still has a source-reproducible Nextcloud Talk no-reply path where the runner creates an incomplete-turn error payload, but group/channel `message_tool_only` delivery suppression can prevent that internal error from being posted back to the room. ## What This Changes

The issue reports Nextcloud Talk webhooks reaching the LLM while OpenClaw 2026.5.4-2026.5.6 never posts a reply or visible error back to the room.

## Best Possible Solution

Keep the issue open and make internal runner failure notices, especially incomplete-turn errors, visible through Nextcloud Talk even when normal group-room replies remain `message_tool_only`.

## Reproduction Assessment

Yes, as a source-reproducible path: the issue log matches the runner's incomplete-turn error payload creation, and current dispatch skips final payload delivery under `message_tool_only` unless a special internal-error metadata flag is present. I did not run a live Nextcloud plus llama.cpp stack.

## Solution Assessment
