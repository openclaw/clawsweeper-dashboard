## Summary

Keep open. Current main still has global/list-based WhatsApp DM controls, session send-policy delivery suppression, and prompt/history maps, but not a WhatsApp-owned per-contact policy that separately controls inbound handling, explicit outbound sends, and autonomous replies. The related closed report at https://github.com/openclaw/openclaw/issues/73740 confirms the same outbound-only contact use case should stay centralized here.

## What This Changes

Request a WhatsApp config/runtime feature for per-contact inbound, explicit outbound, and autonomous-reply permissions at root and account scope.

## Best Possible Solution

Design this as a WhatsApp-owned typed contact policy with schema/docs, root/account field-level merge semantics, inbound admission enforcement before side effects, explicit outbound checks, autonomous-reply dispatch guards, and focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request, but source inspection is high-confidence: current schema, docs, inbound access, outbound target resolution, and sendPolicy tests show no per-contact inbound/outbound/autonomous policy.

## Solution Assessment

Unclear until a maintainer chooses the public config shape. The maintainable direction is hard WhatsApp runtime enforcement in config/schema/runtime, not prompts, memory, dummy agents, or session sendPolicy alone.
