## Summary

Keep open. Current main still has only coarse WhatsApp DM controls and direct-chat prompt/history maps, while the requested per-contact inbound/outbound/autonomous-reply policy remains unimplemented; the closed related request confirms the same use case but leaves this as the canonical tracker.

## What This Changes

The issue requests per-contact WhatsApp policy controls for inbound handling, explicit outbound sends, and autonomous replies, including root/account merge semantics.

## Best Possible Solution

Design this as a WhatsApp-owned plugin/config change with typed per-contact policy, root/account field merging, inbound pre-side-effect enforcement, outbound authorization, autonomous-reply guards, schema/docs metadata, and focused tests.

## Reproduction Assessment

Not applicable as a feature request, but the static verification path is high-confidence. Current schema, docs, inbound admission, outbound resolution, and auto-reply dispatch show no per-contact inbound/outbound/autonomous policy today.

## Solution Assessment

Unclear until a maintainer chooses the public config shape. The maintainable direction is hard WhatsApp runtime policy in config/schema/runtime, not prompts, memory, or the dummy-agent workaround described in the related discussion.
