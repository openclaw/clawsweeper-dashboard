## Summary

Keep open. Current main still exposes only coarse WhatsApp direct-message access controls plus prompt/history maps, while the requested per-contact inbound/outbound/autonomous-reply policy remains unimplemented; the related #73740 was closed as the narrower duplicate of this broader tracker, and 3sken's dummy-agent workaround confirms the use case but does not replace native enforcement.

## What This Changes

Add root-level and account-level WhatsApp per-contact policy controls for inbound admission, explicit outbound sends, and autonomous replies with field-level account overrides.

## Best Possible Solution

Design this as a WhatsApp-owned plugin/config change: add a typed per-contact policy model, define root/account field-merge semantics, enforce it before inbound side effects, check explicit outbound targets, guard autonomous delivery paths, update generated config/docs, and add focused tests around the three policy directions.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence static verification path. Current schema, docs, inbound admission, outbound resolution, and auto-reply dispatch show no per-contact inbound/outbound/autonomous policy today.

## Solution Assessment

Unclear until a maintainer chooses the public config shape. The maintainable direction is hard WhatsApp runtime policy in config/schema/runtime, not prompts, memory, or the dummy-agent workaround described in the related discussion.
