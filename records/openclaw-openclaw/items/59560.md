## Summary

Keep open. Current main still lacks the requested per-contact WhatsApp policy for inbound handling, explicit outbound sends, and autonomous replies; the related open implementation at https://github.com/openclaw/openclaw/pull/82039 only addresses the outbound allowlist split and does not cover the broader inbound/autonomous policy in this issue.

## What This Changes

Add root/account WhatsApp per-contact messaging policy so individual contacts can independently allow or block inbound handling, explicit outbound sends, and autonomous replies.

## Best Possible Solution

Design this as a WhatsApp-owned config/runtime feature with typed contact policy, root/account merge semantics, enforcement before inbound side effects, explicit outbound checks, autonomous-reply dispatch guards, docs/schema updates, and focused tests.

## Reproduction Assessment

Not applicable as a feature request, but the static verification path is high confidence. Current main’s schema, account resolution, inbound admission, outbound authorization, docs, and reply dispatch do not expose or enforce per-contact inbound/outbound/autonomous policy.

## Solution Assessment

Unclear until maintainers choose the public config shape. The maintainable direction is hard WhatsApp runtime enforcement in config/schema/runtime, not prompts, memory, or dummy-agent routing workarounds.
