## Summary

Keep open. Current main still has only global/list-based WhatsApp DM controls plus prompt/history maps; the requested per-contact inbound, outbound, and autonomous-reply policy is not implemented, and the related closed report confirms the same outbound-only-contact use case should remain centralized here.

## What This Changes

Request per-contact WhatsApp config to separately control inbound handling, explicit outbound sends, and autonomous replies at root and account scopes.

## Best Possible Solution

Design this as a WhatsApp-owned config/runtime feature with typed contact policy, root/account merge semantics, inbound admission enforcement before side effects, explicit outbound checks, autonomous-reply dispatch guards, docs/schema updates, and focused tests.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence static verification path. Current schema, docs, inbound admission, outbound authorization, and reply dispatch show no per-contact inbound/outbound/autonomous policy on current main.

## Solution Assessment

Unclear until a maintainer chooses the public config shape. The maintainable direction is hard WhatsApp runtime enforcement in config/schema/runtime, not prompts, memory, or the dummy-agent workaround from the related discussion.
