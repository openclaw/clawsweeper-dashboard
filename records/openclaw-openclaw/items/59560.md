## Summary

Keep open. Current main still only has WhatsApp global/list-based DM controls plus direct prompt/history maps; the requested per-contact inbound, outbound, and autonomous-reply policy is not implemented, and the related #73740 discussion plus 3sken's dummy-agent workaround confirm this is a real use case rather than a solved configuration question.

## What This Changes

Request to add root-level and account-level WhatsApp per-contact policy controls for inbound admission, explicit outbound sends, and autonomous replies with field-level account overrides.

## Best Possible Solution

Add a WhatsApp-owned typed contact policy model, define root/account field-merge semantics, enforce it before inbound side effects, check explicit outbound targets, guard reply dispatch for autonomous sends, update config docs/generated schema, and cover the three policy directions with focused tests.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence static verification path. Current schema, docs, inbound admission, outbound resolution, and auto-reply dispatch show no per-contact inbound/outbound/autonomous policy today.

## Solution Assessment

Unclear until a maintainer chooses the final public config shape. The maintainable direction is hard WhatsApp runtime policy in config/schema/runtime, not prompts, memory, or the dummy-agent workaround from the related discussion.
