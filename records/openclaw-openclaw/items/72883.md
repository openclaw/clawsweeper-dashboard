## Summary

Keep this issue open. Current main still blocks the safe cache-trace config patch, and PR #73098 is open, unmerged, and uses closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that agent-facing gateway config.patch/config.apply allow enabling diagnostics.cacheTrace only when message, prompt, and system content capture are disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so gateway config writes can enable cache trace only when the resulting diagnostics config has includeMessages, includePrompt, and includeSystem all disabled, with regression coverage for safe and unsafe attempts.

## Reproduction Assessment

Yes. Static inspection is enough: the reported patch changes diagnostics.cacheTrace leaves that are not allowlisted, and the guard rejects non-allowlisted changed paths before forwarding config.patch to the gateway write.

## Solution Assessment

Yes. A conditional allow based on the resulting cacheTrace config is the narrowest maintainable direction; unconditionally allowing the subtree would be less safe because the raw content flags default to true.
