## Summary

Keep this issue open: current main still rejects the reported safe `diagnostics.cacheTrace` patch, and the linked open PR is the active closing path rather than evidence that the issue is already resolved.

## What This Changes

Allow `gateway config.patch` to enable `diagnostics.cacheTrace` when message, prompt, and system content capture remain disabled.

## Best Possible Solution

Land PR #73098 or an equivalent narrow guard that permits cache-trace observability only when all content capture flags are false.

## Reproduction Assessment

Yes. Static inspection of current main is high-confidence: the reported patch changes `diagnostics.cacheTrace` leaves, those leaves are not allowlisted, and the guard rejects them before forwarding `config.patch`.

## Solution Assessment

Yes. The conditional-safe approach is the narrowest maintainable fix; unconditionally allowing the subtree would weaken the current content guard because the payload-capture flags default to true.
