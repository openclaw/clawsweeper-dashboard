Keep this issue open: current main still rejects the reported safe `diagnostics.cacheTrace` patch, and PR #73098 remains the active closing implementation path rather than a reason to close early.

## What This Changes

The issue requests that `gateway config.patch` allow safe cache-trace decision observability while keeping prompt, message, and system content capture protected.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow guard that allows cache-trace decision observability only when all content capture flags resolve disabled.

## Reproduction Assessment

Yes. Static inspection is high-confidence: the reported patch changes `diagnostics.cacheTrace` leaves, those leaves are not allowlisted, and the agent-facing `config.patch` branch rejects them before forwarding the write.

## Solution Assessment

Yes. The conditional-safe direction is the best fit; unconditionally allowing the cache-trace subtree would weaken the existing content guard because the payload-capture flags default to true.
