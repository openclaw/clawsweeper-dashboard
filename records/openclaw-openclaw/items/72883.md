## Summary

Keep this open: current main still rejects `diagnostics.cacheTrace.*` through the agent-facing gateway config mutation guard, and PR #73098 is already open with closing syntax for this issue but is not merged. The requested change is valid but security-sensitive because cache-trace content capture defaults to enabled for messages, prompt text, and system prompts.

## What This Changes

The issue asks for `gateway config.patch` to support enabling `diagnostics.cacheTrace.enabled` through a safe path that keeps raw message, prompt, and system-prompt capture disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so the gateway tool permits cache-trace enablement only when the resulting diagnostics config keeps all raw-content capture disabled, with regression tests for safe enablement and unsafe attempts.

## Reproduction Assessment

Yes. Static current-main inspection reproduces the rejection path: a patch touching `diagnostics.cacheTrace.*` produces changed leaf paths that are absent from `ALLOWED_GATEWAY_CONFIG_PATHS`, and `assertGatewayConfigMutationAllowed` throws before the gateway `config.patch` RPC is called.

## Solution Assessment

Yes. A conditional safe-cache-trace exception based on the resulting config is the narrowest maintainable direction; unconditionally allowing the subtree would broaden access to content-capture flags whose defaults are sensitive.
