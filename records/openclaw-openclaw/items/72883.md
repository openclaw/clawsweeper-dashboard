## Summary

Keep this issue open. Current main still rejects the reported safe cache-trace config patch, and the linked PR that uses closing syntax for this issue is still open and unmerged, so closure should wait for that PR or an equivalent fix to land.

## What This Changes

Allow gateway config.patch to enable diagnostics.cacheTrace when content capture for messages, prompts, and system prompts is explicitly disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so gateway config.patch/config.apply can enable cache trace only when the resulting diagnostics config disables all raw content capture, with regression coverage for safe and unsafe attempts.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: the reported merge patch changes diagnostics.cacheTrace leaves that are not allowlisted, so the guard rejects them before config.patch reaches the gateway write call.

## Solution Assessment

Yes. A conditional allow based on the resulting cacheTrace config is the narrowest maintainable fix; unconditionally allowing the subtree would be unsafe because the content capture flags default to true.
