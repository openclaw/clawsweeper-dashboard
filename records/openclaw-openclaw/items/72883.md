## Summary

Keep this issue open: current main still rejects the reported safe `diagnostics.cacheTrace` patch, and PR #73098 remains the open implementation path with closing syntax rather than a reason to close early.

## What This Changes

Allow agent-facing `gateway config.patch` to enable `diagnostics.cacheTrace` only when sensitive content capture is explicitly disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow guard that permits cache-trace decision observability only when all content-capture flags resolve disabled.

## Reproduction Assessment

Yes. Source inspection is high-confidence: the reported patch changes `diagnostics.cacheTrace` leaves, those leaves are not allowlisted, and the agent-facing `config.patch` branch rejects them before the gateway write.

## Solution Assessment

Yes. The conditional-safe direction is the narrowest maintainable fix; unconditionally allowing the subtree would weaken the existing content guard because the payload-capture flags default to true.
