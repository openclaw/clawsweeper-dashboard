## Summary

Keep this issue open: current main still blocks the reported safe cache-trace patch, and PR #73098 is open and unmerged with closing syntax for the same behavior.

## What This Changes

The issue requests a supported `gateway config.patch` path to enable cache-trace decision logging while keeping message, prompt, and system capture disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow guarded change, so cache tracing can be enabled only when content capture is disabled.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the rejection: the patch changes `diagnostics.cacheTrace` leaf paths, none are allowlisted, and the guard throws before forwarding `config.patch`.

## Solution Assessment

Yes, the conditional-safe direction is the best fix. The guard should allow cache-trace enablement only when the resulting configuration keeps all content capture flags false; unconditionally allowing the subtree would weaken the current safety boundary.
