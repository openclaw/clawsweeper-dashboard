## Summary

Keep this issue open because current main still blocks the reported safe `diagnostics.cacheTrace` patch through the agent-facing gateway tool, and PR #73098 is already open as the linked implementation path.

## What This Changes

Allow gateway `config.patch`/`config.apply` to enable `diagnostics.cacheTrace` only when message, prompt, and system content capture remain disabled.

## Best Possible Solution

Land PR #73098 or an equivalent narrow guard change that permits cache-trace enablement only when the resulting cache-trace config keeps all content capture flags disabled.

## Reproduction Assessment

Yes. Static inspection of current main is enough: the reported patch changes `diagnostics.cacheTrace` leaves, those leaves are not allowlisted, and the guard rejects them before `config.patch` is forwarded.

## Solution Assessment

Yes, the conditional-safe approach is the best direction. Unconditionally allowing the cache-trace subtree would weaken the current model-to-operator trust boundary because the content flags default to capture.
