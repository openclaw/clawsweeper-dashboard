## Summary

Keep this issue open. Current main still has no cache-trace exception in the agent-facing gateway config mutation guard, and the linked implementation PR remains open and unmerged, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that agent-facing gateway config.patch/config.apply allow safe diagnostics.cacheTrace enablement when raw content capture is explicitly disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so agent-facing gateway config writes can enable cache trace only when the resulting cacheTrace configuration has includeMessages, includePrompt, and includeSystem all disabled, with regression coverage for safe and unsafe attempts.

## Reproduction Assessment

Yes. Static inspection of current main is enough: the reported patch changes diagnostics.cacheTrace leaves that are not allowlisted, and the guard rejects non-allowlisted changed paths before forwarding config.patch to the gateway write.

## Solution Assessment

Yes. A conditional allow based on the resulting cacheTrace config is the narrowest maintainable direction; unconditionally allowing the subtree would be less safe because the raw content flags default to true.
