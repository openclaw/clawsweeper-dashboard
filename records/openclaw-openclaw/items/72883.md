## Summary

Keep this open. Current main still rejects the reported `diagnostics.cacheTrace.*` mutation through the agent-facing gateway config guard, and the provided GitHub context shows PR #73098 is open, unmerged, and explicitly targets this issue.

## What This Changes

Request to let gateway `config.patch` enable `diagnostics.cacheTrace` safely when `includeMessages`, `includePrompt`, and `includeSystem` remain disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so gateway config mutation permits safe cache-trace observability only when the resulting diagnostics config cannot capture raw content, with regression coverage for safe enablement and unsafe attempts.

## Reproduction Assessment

Yes. Static inspection is enough to reproduce the failure path: the reported patch changes `diagnostics.cacheTrace` leaves, those leaves are absent from the allowlist, and the guard throws before forwarding `config.patch`.

## Solution Assessment

Yes. A conditional allow based on the resulting `cacheTrace` config is the narrowest maintainable direction; unconditionally allowing the subtree would be less safe because the content-capture fields default to true.
