## Summary

Keep this issue open. Current main still rejects the reported safe `diagnostics.cacheTrace` patch through the agent-facing gateway tool, and linked PR #73098 is open, unmerged, and uses closing syntax for this issue.

## What This Changes

The issue asks for `gateway config.patch` or an equivalent gateway action to enable cache-trace observability only when message, prompt, and system content capture are disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so the gateway tool can enable cache tracing only when the resulting cache-trace configuration keeps `includeMessages`, `includePrompt`, and `includeSystem` false, with regression coverage for safe and unsafe attempts.

## Reproduction Assessment

Yes. Static inspection of current main gives a high-confidence reproduction: the reported patch changes `diagnostics.cacheTrace` leaves, those leaves are absent from the gateway tool allowlist, and the guard rejects them before forwarding `config.patch`.

## Solution Assessment

Yes. A conditional allow based on the resulting cache-trace config is the narrowest maintainable direction; unconditionally allowing the subtree would be less safe because raw content capture defaults to enabled.
