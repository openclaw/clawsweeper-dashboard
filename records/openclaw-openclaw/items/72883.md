## Summary

Keep this issue open. Current main still rejects the reported safe `diagnostics.cacheTrace` patch through the agent-facing gateway tool, and PR #73098 is open and unmerged with closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Allow gateway `config.patch` or `config.apply` to enable cache-trace observability only when sensitive cache-trace content capture is disabled, or provide an equivalent safe enablement path.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so the gateway tool can enable cache tracing only when the resulting cache-trace configuration has `includeMessages`, `includePrompt`, and `includeSystem` all disabled, with regression coverage for safe and unsafe attempts.

## Reproduction Assessment

Yes. Static inspection of current main is enough: the reported patch changes `diagnostics.cacheTrace` leaf paths, those paths are absent from the gateway tool allowlist, and the guard rejects non-allowlisted changed paths before forwarding `config.patch`.

## Solution Assessment

Yes. A conditional allow based on the resulting cache-trace config is the narrowest maintainable direction; unconditionally allowing the subtree would be less safe because the raw content flags default to true.
