Keep open: current main confirms the reported UX tension, but the requested changes cross config startup policy and the agent-facing config-write security boundary, so this is not a closeable cleanup or narrow automated bug fix.

## What This Changes

The issue requests clearer handling for deprecated `agentRuntime.fallback`, degraded read-only CLI behavior when config is invalid, and documentation or allowlist changes for agent-driven heartbeat timeout config patches.

## Best Possible Solution

Split the item into maintainer-approved config UX work: decide which invalid-config states can safely degrade for read-only commands, keep doctor as the canonical migration writer, and either document the supported heartbeat edit path or deliberately expand the agent-facing allowlist with tests.

Yes. Source inspection gives a high-confidence current-main path: `agentRuntime.fallback` fails the strict schema, `sessions` is not invalid-config allowlisted, and agent-driven `config.patch` rejects heartbeat timeout because it is outside the narrow allowlist.
