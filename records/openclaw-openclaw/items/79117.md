## Summary

Keep open: current main still exhibits the reported source-level behavior, but the request aggregates legacy config UX, read-only CLI invalid-config policy, and a protected agent config boundary, so it needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue asks for clearer handling of removed `agentRuntime.fallback`, graceful read-only CLI behavior when config is invalid, and a supported path for changing `agents.defaults.heartbeat.timeoutSeconds` without raw JSON edits.

## Best Possible Solution

Split the outcome: keep legacy config cleanup in `openclaw doctor --fix`, decide which read-only CLI commands may run on invalid config, and route protected config edits through a documented direct-operator path or the broader owner-approved design in https://github.com/openclaw/openclaw/issues/77886.

## Reproduction Assessment

Yes, source-reproducible: `agentRuntime.fallback` is strict-invalid with doctor repair, `sessions` is not invalid-config allowlisted, and the agent-facing gateway tool rejects paths outside its narrow allowlist. I did not live-replay the macOS update session.

## Solution Assessment

Unclear; auto-migrating or unprotecting paths is not obviously the best fix. The safer direction is to preserve doctor-owned legacy repair and make an explicit maintainer decision on invalid-config read-only CLI scope and protected config approval.
