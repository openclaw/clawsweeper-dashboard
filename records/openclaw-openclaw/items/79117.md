## Summary

Keep open: current main still matches the reported config UX behavior, and resolving it spans startup/deprecation policy plus the agent-facing config-write security boundary rather than a conservative cleanup close.

## What This Changes

Requests clearer handling for removed `agentRuntime.fallback` config, graceful read-only CLI behavior when config is invalid, and documentation or relaxation for protected heartbeat timeout writes through `config.patch`.

## Best Possible Solution

Split this into maintainer-approved config UX work: decide which invalid-config states can degrade for read-only commands, keep doctor as the canonical migration writer, and either document the heartbeat edit path or deliberately allowlist it with focused tests.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: legacy `agentRuntime.fallback` is strict-schema invalid and doctor-owned, `sessions` is not invalid-config allowlisted, and the gateway tool blocks heartbeat timeout because it is outside the protected write allowlist.

## Solution Assessment

Unclear. The requested direction is plausible, but migration warnings, read-only CLI degradation, and agent-facing config write permissions need separate maintainer policy decisions before a narrow implementation is safe.
