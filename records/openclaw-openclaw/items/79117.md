## Summary

Keep open: current main still matches the reported config UX tension, but the requested changes span startup invalid-config policy, read-only CLI degradation, legacy migration ownership, and the agent-facing persistent config-write boundary, so this needs maintainer direction rather than conservative cleanup.

## What This Changes

The issue requests clearer config upgrade handling for legacy `agentRuntime.fallback`, graceful read-only CLI behavior with invalid config, and documentation or relaxation of the `config.patch` guard for `agents.defaults.heartbeat.timeoutSeconds`.

## Best Possible Solution

Split this into maintainer-approved config UX work: keep doctor as the canonical legacy-config writer, decide which invalid-config states can safely degrade for read-only commands, and either document the supported heartbeat edit path or deliberately expand the gateway-tool allowlist with targeted tests.

## Reproduction Assessment

Yes, source-reproducible. Current main rejects stale `agentRuntime.fallback`, does not allowlist `sessions` for invalid config, and rejects agent-facing `config.patch` writes outside the narrow gateway-tool allowlist.

## Solution Assessment

Unclear. The requested direction is plausible, but startup warnings, read-only invalid-config degradation, and agent-driven heartbeat config writes have separate compatibility and security tradeoffs that need maintainer policy first.
