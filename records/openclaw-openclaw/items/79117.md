## Summary

Keep open: current main and the latest release have partially improved legacy `agentRuntime` detection/doctor handling, but the reported `openclaw sessions` invalid-config failure and the agent-facing `config.patch` protection for heartbeat timeout still appear to remain.

## What This Changes

The issue requests clearer config-upgrade handling for removed `agentRuntime.fallback`, graceful read-only CLI behavior when config is invalid, and a documented or allowed path for patching `agents.defaults.heartbeat.timeoutSeconds`.

## Best Possible Solution

Keep strict runtime config validation, but make the safe recovery path explicit: preserve doctor migration for legacy `agentRuntime`, decide whether `sessions` should be an invalid-config read-only command, and either document heartbeat timeout as operator-only or deliberately add it to the agent-facing config patch allowlist with tests.

## Reproduction Assessment

Yes, from source inspection: invalid config still exits for non-allowlisted commands and `sessions` is not allowlisted, while the gateway tool still rejects unallowlisted config paths including heartbeat timeout. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear as a single change: doctor-based migration matches repository policy, but blanket invalid-config tolerance and unprotecting gateway config paths need maintainer scoping. A narrower fix for safe read-only `sessions` plus a documented heartbeat timeout decision is likely the better path.
