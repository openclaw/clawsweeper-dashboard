## Summary

Keep open: current main still matches the reported config UX tension, and the requested changes span legacy-config migration policy, invalid-config CLI degradation, and the protected agent-facing config-write boundary.

## What This Changes

The issue asks for clearer handling of removed `agentRuntime.fallback`, read-only CLI behavior when config is invalid, and a documented or allowed path for changing heartbeat timeout through `config.patch`.

## Best Possible Solution

Split this into maintainer-approved config UX work: keep doctor as the canonical legacy-config writer, define which invalid-config states can degrade for read-only commands, and coordinate heartbeat protected writes with the broader owner-approved config-change design.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main rejects stale `agentRuntime.fallback`, excludes `sessions` from the invalid-config allowlist, and blocks heartbeat timeout edits through the agent-facing gateway tool.

## Solution Assessment

Unclear. The requested direction is plausible, but the three parts involve separate migration, CLI-degradation, and security-boundary decisions that should not be handled as one narrow automated fix.
