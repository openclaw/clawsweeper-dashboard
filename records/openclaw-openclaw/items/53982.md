## Summary

Keep this issue open: current main still documents and surfaces a Nextcloud Talk bot install command without the required `--feature response`, and outbound HTTP 401 handling still only points at a bot-secret mismatch. Upstream Nextcloud Talk stable32 source confirms message sends are filtered by `FEATURE_RESPONSE`, and the two related fix PRs are closed unmerged with no open replacement found.

## What This Changes

Document and diagnose that Nextcloud Talk bot registration must include `--feature response` so OpenClaw can send replies back to Nextcloud Talk conversations.

## Best Possible Solution

Update the Nextcloud Talk plugin-owned docs and setup wizard to register bots with `--feature webhook --feature response --feature reaction`, then extend the outbound 401 diagnostic and focused tests so missing response-feature setup is called out without losing the real secret-mismatch guidance.

## Reproduction Assessment

Yes. The high-confidence reproduction path is to follow the current documented `talk:bot:install` command without `--feature response`, then trigger an outbound reply; upstream stable32 filters bot sends by `FEATURE_RESPONSE` while current OpenClaw guidance omits that feature.

## Solution Assessment

Yes. The narrow maintainable solution is a Nextcloud Talk plugin/docs repair plus targeted send-error diagnostic coverage; no core API or broader channel architecture change is needed.
