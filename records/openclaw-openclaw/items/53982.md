## Summary

Keep this issue open: current main still omits the required Nextcloud Talk `response` bot feature in both docs and setup wizard text, and the outbound 401 diagnostic still points only at a bot-secret mismatch. Upstream Nextcloud Talk stable32 source confirms message sends are filtered through `FEATURE_RESPONSE`, and both related fix PRs are closed unmerged with no open implementation candidate replacing them.

## What This Changes

The issue requests documenting `--feature response` for Nextcloud Talk bot registration and improving the outbound 401 error when that feature is missing.

## Best Possible Solution

Update the Nextcloud Talk plugin-owned docs and setup wizard to register bots with `--feature webhook --feature response --feature reaction`, and extend the outbound 401 diagnostic plus regression tests so users see the missing-response-feature setup pitfall without losing the real secret-mismatch guidance.

## Reproduction Assessment

Yes. The high-confidence reproduction path is to follow the current documented `talk:bot:install` command without `--feature response`, then trigger an outbound reply; upstream stable32 filters bot sends by `FEATURE_RESPONSE` while current OpenClaw guidance omits that feature.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-owned docs/setup copy plus a targeted send-error diagnostic and tests; no core API or broader channel architecture change is needed.
