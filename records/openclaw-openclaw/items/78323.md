## Summary

Keep open: current main still has a source-backed warning path during the bounded startup retry, and the discussion has expanded this into the canonical tracker for healthy-cron runtime false positives rather than a fully resolved startup-only report.

## What This Changes

The issue asks memory-core to stop emitting misleading managed dreaming cron-unavailable warnings during gateway startup or runtime when the cron service is expected to recover or is already healthy.

## Best Possible Solution

Keep this as the canonical memory-core cron warning tracker, land or supersede the focused startup retry fix, then decide the minimal runtime-context or warning-policy change that removes healthy-cron false positives without hiding genuine cron-management failures.

## Reproduction Assessment

Yes, at source level. With dreaming enabled and cron unavailable during the startup retry window, current main calls runtime reconciliation after `STARTUP_CRON_RETRY_DELAY_MS` and the focused test expects the warn-level cron-unavailable log.

## Solution Assessment
