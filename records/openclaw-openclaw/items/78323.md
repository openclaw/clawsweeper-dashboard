## Summary

Current main already suppresses the startup-only memory-core dreaming cron warning and keeps a runtime warning only for a genuinely unavailable cron service; the reported version was v2026.4.21, while the exact warning suppression shipped in v2026.4.22 and later startup retry/live-cron fixes are also present.

## What This Changes

Suppress or defer the benign memory-core managed dreaming cron unavailable warning during gateway startup when the cron service has not attached yet.

## Best Possible Solution

Keep the shipped startup/runtime reconciliation behavior and have users on v2026.4.21 upgrade to v2026.4.22 or newer, preferably the latest release.

## Reproduction Assessment

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: startup misses are demoted to debug and retried, live cron state is refreshed, and the warning is preserved for a real runtime cron outage.
