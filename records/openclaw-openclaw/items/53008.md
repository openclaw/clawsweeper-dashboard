## Summary

Keep open. Current main still runs preflight compaction and memoryFlush inside the foreground reply path, and the maintenance work still shares same-session serialization with inbound messages; recent stuck-session recovery only releases stale lanes and deliberately avoids active embedded/reply/lane work.

## What This Changes

The issue asks OpenClaw to prevent memory compaction and memoryFlush maintenance from blocking inbound Telegram/main-session processing for minutes, with shorter bounds and recovery behavior.

## Best Possible Solution

Define a maintainer-owned compaction scheduling contract where preflight compaction and memoryFlush are bounded maintenance work that cannot starve interactive inbound messages, with explicit same-session ordering rules, cancellation semantics, transcript-rotation safety, and regression coverage for queued inbound messages during active compaction.

## Reproduction Assessment

Yes. The issue includes a concrete log timeline, and current source gives a high-confidence source-level reproduction path: trigger preflight compaction or memoryFlush during a foreground reply, then observe later same-session inbound messages queue behind the active session/global lane while compacting runs reject steering.

## Solution Assessment

No. Current main is not the best solution yet, and the closed #53154-style lane move is too narrow because same-session serialization, active compaction cancellation, and transcript/session rotation behavior still need an explicit policy.
