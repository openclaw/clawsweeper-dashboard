- `P1`: The issue describes real channel users seeing Telegram inbound messages queue behind memory compaction for minutes until a manual gateway restart.
- `impact:session-state`: The affected code path is session memory, transcript compaction, active session lanes, and compaction metadata.
- `impact:message-loss`: Inbound messages are accepted into the queue but not processed while compaction occupies the lane, making them appear lost to users.
- `impact:crash-loop`: The reported failure is an availability hang/unresponsive gateway session that required manual restart to restore interactivity.

## Summary

Keep open: current main and the latest release still run memory-flush and preflight compaction through the main/session lane path, the direct fix PR was closed unmerged, and the related open issues only cover overlapping scheduler or recovery pieces rather than superseding this memory-compaction lane-isolation bug.

## What This Changes

The issue requests that memory compaction and memoryFlush no longer block the main inbound message processing lane for long-running Telegram sessions.

## Best Possible Solution

Define a maintainer-approved foreground/background session scheduling policy, then move memory compaction and memoryFlush onto an isolated or preemptible background path with regression coverage for Telegram-style inbound messages and session-state ordering.

## Reproduction Assessment

Yes, source-level: current main still awaits preflight compaction before the reply turn and still runs memoryFlush with `lane: CommandLane.Main`. I did not run a live 10-minute Telegram stall in this read-only review.

## Solution Assessment

Unclear as an implementation choice: a dedicated lane is plausible, but the best fix needs a maintainer-approved scheduler policy for foreground priority, background abort/preemption, and transcript ordering safety.
