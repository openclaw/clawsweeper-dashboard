## Summary

Keep open. Current main still runs preflight compaction and memoryFlush through the foreground reply/session serialization path, so same-session Telegram messages can still sit behind active maintenance work; related open items are adjacent but do not supersede this concrete report.

## What This Changes

The issue asks OpenClaw to prevent memory compaction from blocking interactive message processing when a compaction run hangs or takes minutes.

## Best Possible Solution

Define bounded maintenance scheduling so preflight compaction and memoryFlush cannot starve interactive same-session messages, with explicit cancellation/session-rotation semantics and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session Telegram message; current code can queue that message behind the active reply/session path while compaction rejects steering.

## Solution Assessment

No current implementation solves the issue yet. A simple global-lane swap is not sufficient because same-session serialization, active compaction steering, cancellation, and transcript rotation need one coherent policy.
