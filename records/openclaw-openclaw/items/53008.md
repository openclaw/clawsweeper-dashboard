## Summary

Keep open. Current main improved stalled-session recovery, but memory maintenance still runs in the foreground reply/session path, with memoryFlush explicitly using the main lane and same-session serialization still able to starve inbound messages until completion or recovery timeout.

## What This Changes

Prevent memory compaction and memoryFlush maintenance from blocking inbound Telegram or other interactive messages for the same session for up to the compaction timeout.

## Best Possible Solution

Define bounded maintenance scheduling so preflight compaction and memoryFlush cannot starve same-session interactive messages, while preserving transcript rotation/memory metadata and using diagnostics recovery only as a fallback.

## Reproduction Assessment

Yes, by source inspection. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message: current main awaits the maintenance path, uses the main lane for memoryFlush, serializes by session first, and rejects steering while compacting.

## Solution Assessment

No, the current main behavior is not the best solution because recovery after a long stall does not prevent foreground maintenance from starving interactive messages. A dedicated or otherwise bounded maintenance scheduling policy is needed, with explicit same-session cancellation and transcript-rotation semantics.
