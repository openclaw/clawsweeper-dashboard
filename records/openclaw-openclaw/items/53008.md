## Summary

Keep open. Current main still runs preflight compaction and memoryFlush inside the foreground reply/session path, and same-session inbound work remains serialized while compacting runs reject steering.

## What This Changes

The issue reports that memory compaction can block Telegram session processing long enough for the bot to appear unresponsive for 10 or more minutes.

## Best Possible Solution

Design bounded maintenance scheduling so compaction and memoryFlush cannot starve interactive same-session messages, with explicit cancellation, transcript rotation, and channel acknowledgment semantics.

## Reproduction Assessment

Yes, by source inspection. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message; the message remains serialized behind the active reply/session lane while compacting runs reject steering.

## Solution Assessment

Unclear as a specific implementation. A dedicated lane may reduce global contention, but same-session serialization and transcript locks mean the maintainable fix needs a broader maintainer-owned scheduling and rotation design.
