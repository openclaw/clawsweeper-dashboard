## Summary

Keep open. Current main still runs preflight compaction and memoryFlush inside the foreground reply/session path, and the related lane-only PR was closed unmerged, so the broader starvation bug remains source-reproducible.

## What This Changes

The issue reports that memoryFlush/preflight compaction can occupy the main/session processing lane long enough to queue Telegram messages and require a gateway restart.

## Best Possible Solution

Design bounded maintenance scheduling so compaction and memoryFlush cannot starve interactive same-session messages while preserving cancellation, transcript rotation, and channel acknowledgment semantics.

## Reproduction Assessment

Yes, source-level. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message; current main still awaits maintenance in the reply path and only queues steering once the reply operation is running with a streaming backend.

## Solution Assessment

Unclear as a specific implementation. A dedicated lane may reduce global contention, but maintainers still need to choose the same-session scheduling, cancellation, transcript rotation, and user acknowledgment contract.
