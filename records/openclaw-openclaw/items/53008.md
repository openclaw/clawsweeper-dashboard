## Label Justifications

- `P1`: The report describes a real agent/channel workflow becoming unresponsive for more than 10 minutes while user messages queue behind compaction housekeeping.
- `impact:session-state`: The affected behavior is session memoryFlush/compaction lifecycle work and transcript state rotation in the active agent session.

- `impact:crash-loop`: The user-visible failure is a long hang requiring gateway restart to recover, which falls under process availability and stuck-runtime impact.

## Summary

Keep open: current main and the latest release still route memoryFlush housekeeping through the main embedded-run lane and per-session lane, the direct lane-isolation PR was closed unmerged, and no narrower open item supersedes this exact memoryFlush lane-isolation bug.

## What This Changes

The issue requests that memoryFlush compaction housekeeping stop blocking the main message-processing lane long enough to starve Telegram replies.

## Best Possible Solution

Land a maintainer-approved dedicated compaction or memoryFlush maintenance lane that preserves per-session state consistency while keeping foreground channel messages from waiting behind long housekeeping work.

## Reproduction Assessment

Yes, source-reproducible: current main still sends memoryFlush through `CommandLane.Main` and the embedded runner still enqueues main-lane work before the session work. I did not run a live Telegram outage reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice: a separate maintenance lane is the likely right direction, but the exact lane, timeout, and ordering semantics need maintainer approval because session compaction mutates active session state.
