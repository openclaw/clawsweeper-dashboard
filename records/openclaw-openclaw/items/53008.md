## Summary

Keep open: current main and the latest release still run memoryFlush and preflight compaction through foreground main/session-lane paths, the direct lane-isolation PR was closed unmerged, and related open issues cover overlapping recovery/cancel cases rather than superseding this lane-starvation bug.

## What This Changes

The issue reports that memoryFlush compaction can block the interactive processing lane long enough for inbound Telegram messages to queue and appear unprocessed until timeout or gateway restart.

## Best Possible Solution

Define a maintainer-approved foreground/background session scheduling policy, then isolate or preempt memoryFlush and required compaction with regression coverage for queued Telegram-style inbound messages and transcript ordering.

## Reproduction Assessment

Yes, source-level: current main still awaits preflight compaction before the reply turn and runs memoryFlush with `lane: CommandLane.Main`, while queued compaction serializes through the same session lane. I did not run a live Telegram 10-minute stall in this read-only review.

## Solution Assessment

Unclear as an implementation choice: a dedicated lane or preemptible/background path is plausible, but the best fix needs maintainer direction on scheduler policy, abort semantics, and transcript rotation safety.
