## Summary

Keep open: current main and the latest release still run memory-flush/preflight compaction through the same session/global lane machinery that can block interactive replies, while the only direct lane-isolation PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to prevent memoryFlush/session compaction from blocking the main interactive processing lane for inbound Telegram messages.

## Best Possible Solution

Choose and implement a maintainer-approved compaction/memory-flush isolation strategy that keeps user messages responsive while preserving session-state ordering, with focused regression coverage for lane selection and queued reply behavior.

## Reproduction Assessment

Yes, source-level: current main still sends memoryFlush through CommandLane.Main and queued preflight compaction through the session/global lane path. I did not run a live 10-minute Telegram reproduction in this read-only review.

## Solution Assessment

No, current main is only partially mitigated; it tracks some flush failures but does not isolate the long-running compaction work from interactive lane admission. The best fix needs a maintainer-approved lane or sequencing design rather than reopening the closed PR unchanged.
