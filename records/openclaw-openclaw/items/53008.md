## Summary

Keep open. Current main still runs preflight compaction and memoryFlush inside the foreground reply/session path, and the remaining fix needs a maintainer-owned scheduling and cancellation policy rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent session compaction and memoryFlush maintenance from blocking inbound Telegram message processing for minutes.

## Best Possible Solution

Define bounded maintenance scheduling so preflight compaction and memoryFlush cannot starve interactive same-session messages, with explicit cancellation/session-rotation semantics and regression tests.

## Reproduction Assessment

Yes, by source inspection. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message and it can remain queued behind the active reply/session lane while compaction rejects steering.

## Solution Assessment

No. Current main still keeps the maintenance work in the foreground reply/session serialization path, and a narrow global-lane switch is not enough without defining same-session ordering, cancellation, and transcript-rotation safety.
