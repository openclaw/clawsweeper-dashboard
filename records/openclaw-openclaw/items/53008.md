## Label Justifications

- `P1`: The issue describes a real Telegram workflow becoming unresponsive for minutes while user messages queue behind compaction work.
- `impact:message-loss`: Inbound Telegram messages are reported as queued and not processed while compaction occupies the active lane.
- `impact:session-state`: The affected behavior is session memoryFlush/compaction scheduling, transcript state, and per-session lane serialization.
- `impact:crash-loop`: The reported recovery required manually restarting the gateway after the session processing lane stayed wedged.

## Summary

Keep open: current main and v2026.6.10 still run preflight compaction and memoryFlush inside the active reply/session scheduling path. The compaction timeout has been reduced from the reported 600s default to 180s, but there is still no dedicated or preemptible compaction lane that would let foreground Telegram messages proceed promptly.

## What This Changes

The issue reports memoryFlush/session compaction occupying the foreground processing lane, leaving inbound Telegram messages queued until compaction times out or the gateway is restarted.

## Best Possible Solution

Land a maintainer-approved bounded or preemptible maintenance-lane design for memoryFlush and compaction that preserves session ordering while allowing foreground channel messages to be acknowledged or processed promptly.

## Reproduction Assessment

Yes, at source level: current main still awaits preflight compaction, runs memoryFlush on CommandLane.Main, and queues compaction through the same session/global lane structure. I did not run a live Telegram timeout reproduction.

## Solution Assessment

No, current main is only a partial mitigation because the shorter timeout does not isolate background compaction from foreground messages. The best fix needs maintainer agreement on scheduler and session-state semantics before implementation.
