## Summary

Keep open. Current main still executes preflight compaction and memoryFlush inside the foreground reply/session serialization path, and the remaining fix needs maintainer policy for maintenance scheduling, cancellation, and session rotation rather than cleanup closure.

## What This Changes

The issue reports that session memory compaction can monopolize the interactive agent lane for minutes, leaving inbound Telegram messages queued until timeout or gateway restart.

## Best Possible Solution

Define bounded background maintenance scheduling so preflight compaction and memoryFlush cannot starve same-session user messages, with explicit cancellation, queue ordering, and transcript-rotation semantics plus focused regressions.

## Reproduction Assessment

Yes, by source inspection. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message; current main can keep that message behind the active reply/session lane while compacting work rejects steering.

## Solution Assessment

No, a narrow global-lane switch is not enough. The maintainable fix needs same-session maintenance scheduling and cancellation semantics because runEmbeddedPiAgent still serializes through the session lane before the global lane.
