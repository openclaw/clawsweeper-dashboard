## Summary

Keep open. Current main and the latest release still await preflight compaction and memoryFlush in the foreground reply path, memoryFlush still dispatches through `CommandLane.Main`, and the prior lane-isolation PR was closed unmerged; related open work only covers narrower overflow or flush-failure cases.

## What This Changes

The issue reports that compaction and memoryFlush can block interactive channel processing so inbound Telegram messages queue without responses for 10+ minutes.

## Best Possible Solution

Design bounded maintenance scheduling so compaction and memoryFlush cannot starve interactive same-session messages while preserving transcript locks, rotation, cancellation, and channel acknowledgment semantics.

## Reproduction Assessment

Yes, source-level. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message; current main still awaits maintenance in the reply path and does not steer messages while the operation is compacting/flushing.

## Solution Assessment

Unclear as a specific implementation. A dedicated lane may reduce one source of contention, but maintainers still need to choose the same-session scheduling, cancellation, transcript rotation, and user acknowledgment contract.
