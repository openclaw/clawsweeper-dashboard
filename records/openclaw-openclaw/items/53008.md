## Label Justifications

- `P1`: The issue reports a real channel workflow becoming unresponsive for 10+ minutes while inbound Telegram messages queue behind compaction.
- `impact:message-loss`: The reported user-visible symptom is inbound Telegram messages queued and not processed until restart.
- `impact:session-state`: The failure centers on session compaction, memoryFlush, transcript rotation, and same-session lane state.
- `impact:crash-loop`: The affected gateway/session can hang long enough to require manual gateway restart for recovery.

## Summary

Keep open. Current main still awaits preflight compaction and memoryFlush inside the foreground reply/session flow, same-session work remains serialized through the session lane, and compacting active runs reject steering; related PR work has not merged a fix for this broader starvation path.

## What This Changes

Memory compaction and memoryFlush should not starve interactive Telegram/session message processing for minutes.

## Best Possible Solution

Design bounded maintenance scheduling so compaction and memoryFlush cannot starve interactive same-session messages, with explicit cancellation, transcript rotation, and channel acknowledgment semantics.

## Reproduction Assessment

Yes, source-level. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message; current main still serializes the maintenance work through the reply/session path and rejects steering while compacting.

## Solution Assessment

Unclear as a specific implementation. A dedicated global lane alone is not sufficient because same-session serialization and transcript locking/rotation remain, so maintainers need to choose the scheduling and acknowledgment contract first.
