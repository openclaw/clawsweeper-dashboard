- `impact:message-loss`: The user-visible failure is inbound Telegram messages being queued and not processed until timeout or restart.

## Summary

Keep open. Current main still runs preflight compaction and memoryFlush in the foreground reply/session path, same-session work is serialized through the session lane, and the only linked closing PR was closed unmerged after the author called out that lane choice may need different sequencing or a dedicated lane.

## What This Changes

The issue asks OpenClaw to prevent session compaction and memoryFlush from starving inbound Telegram messages for minutes while foreground processing is blocked.

## Best Possible Solution

Design bounded maintenance scheduling so compaction and memoryFlush cannot starve interactive same-session messages, with explicit contracts for cancellation, transcript locking/rotation, queued followups, and channel acknowledgments.

## Reproduction Assessment

Yes, source-level with high confidence. Trigger preflight compaction or memoryFlush during a foreground reply and then send another same-session message; current main still awaits maintenance work and serializes the same session through one lane.

## Solution Assessment

Unclear as a concrete implementation. A dedicated global lane alone is not sufficient because the same-session lane, transcript write lock, compaction rotation, and channel acknowledgment behavior all need an explicit maintainer-approved contract.
