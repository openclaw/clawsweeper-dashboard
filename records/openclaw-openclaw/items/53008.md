## Summary

Keep open. Current main still runs preflight compaction and memoryFlush as awaited foreground reply work, both calls still lack an explicit maintenance lane, omitted embedded lanes still default to `main`, and same-session serialization still happens before global lane choice. The linked #53154 attempted part of this fix but is closed unmerged, while #73581 is related broader lane-stall work rather than a clean replacement for this compaction-specific report.

## What This Changes

Prevent memory compaction and memoryFlush from blocking Telegram and other inbound message processing for minutes when maintenance work hangs.

## Best Possible Solution

Define an explicit maintenance scheduling contract for preflight compaction and memoryFlush: maintenance work should be bounded, unable to starve interactive inbound messages, observable in diagnostics, and covered by tests for queued same-session input during active compaction and transcript rotation.

## Reproduction Assessment

Yes. The issue includes a concrete journalctl timeline, and current source gives a high-confidence reproduction path: trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message while the awaited compaction run occupies the session/global lane path.

## Solution Assessment

No. Current main does not solve the issue, and the closed #53154-style lane move is not sufficient by itself because same-session serialization, compaction steering rejection, and active-work recovery policy still need a coherent maintainer-owned contract.
