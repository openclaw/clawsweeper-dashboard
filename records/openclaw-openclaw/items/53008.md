## Summary

Keep open. Current main still awaits preflight compaction and memoryFlush inside the foreground reply/session path; the newer diagnostics recovery path can abort some stalled work later, but it does not make compaction non-blocking for interactive messages. The closed dedicated-lane PR was not merged, and the broader stuck-session issue remains open for adjacent recovery work rather than fully superseding this compaction-isolation report.

## What This Changes

The issue asks for memory compaction and memoryFlush maintenance work to stop blocking inbound Telegram messages on the main processing lane for minutes.

## Best Possible Solution

Define bounded maintenance scheduling so preflight compaction and memoryFlush cannot starve same-session interactive messages, with explicit cancellation/session-rotation semantics and regression coverage for queued user messages during active maintenance.

## Reproduction Assessment

Yes, by source inspection and reporter logs. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message; current main keeps that work in the active reply/session serialization path and compacting runs reject steering.

## Solution Assessment

No for the current state. A narrow global-lane switch like the closed unmerged PR is not enough because same-session serialization still applies; the maintainable fix needs a deliberate maintenance scheduling and abort policy.
