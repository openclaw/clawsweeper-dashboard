- `impact:session-state`: The problem is rooted in compaction and memoryFlush work holding session state and per-session lane execution while later turns wait.

- `impact:crash-loop`: The symptom is a process-level availability hang requiring manual gateway restart to recover in the reported incident.

## Summary

Keep open: current main still runs memoryFlush through the main global lane and the same per-session lane that serializes inbound replies, so a slow compaction/flush can still starve user messages. The linked fix PR was closed unmerged, and the broader lane-starvation work remains open rather than merged.

## What This Changes

Prevent memoryFlush and compaction work from blocking Telegram or other user-message processing for minutes.

## Best Possible Solution

Define a state-safe user-vs-background session scheduling policy so inbound user messages can interrupt or bypass memoryFlush/compaction background work without racing transcript writes, then cover memoryFlush and broader scheduler fairness with focused regression tests.

## Reproduction Assessment

Yes, at source level. An over-threshold memoryFlush path awaits a provider-backed embedded run while the embedded runner serializes the same session lane with maxConcurrent 1; I did not run a live 10-minute Telegram stall in this read-only review.

## Solution Assessment

Unclear. A lane-param-only fix is not enough because embedded runs also acquire the per-session lane; maintainers should first choose the user/background lane policy and abort/yield semantics.
