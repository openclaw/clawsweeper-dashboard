## Summary

Keep open. Current main still awaits preflight compaction and memoryFlush inside the foreground reply path, and the prior lane-isolation PR was closed unmerged, so the Telegram unresponsiveness report remains source-reproducible.

## What This Changes

The issue asks OpenClaw to keep inbound channel messages responsive when memory compaction or memoryFlush runs for a long time.

## Best Possible Solution

Design bounded maintenance scheduling so compaction and memoryFlush cannot starve interactive same-session messages while preserving transcript locks, rotation, cancellation, and channel acknowledgment semantics.

## Reproduction Assessment

Yes, source-level. Trigger preflight compaction or memoryFlush during a foreground reply, then send another same-session message; current main still awaits maintenance in the reply path and only queues steering once the reply operation is running with a streaming backend.

## Solution Assessment

Unclear as a specific implementation. A dedicated lane may reduce one source of contention, but maintainers still need to choose the same-session scheduling, cancellation, transcript rotation, and user acknowledgment contract.
