## Label Justifications

- `P1`: The issue describes a real user-facing Telegram workflow becoming unresponsive for minutes while messages queue behind compaction work.
- `impact:message-loss`: Inbound Telegram messages are reported as queued and not processed while compaction occupies the active lane.
- `impact:session-state`: The affected behavior is session memoryFlush/compaction scheduling, transcript state, and per-session lane serialization.
- `impact:crash-loop`: The reported recovery was a manual gateway restart after the processing lane stayed wedged.

## Summary

Keep open: current main and v2026.6.10 still route memoryFlush/preflight compaction through the interactive reply path and main/session lanes, while the open related PR only mitigates tool-heavy overflow ordering and does not isolate compaction from foreground messages.

## What This Changes

The issue reports that memoryFlush/session compaction can occupy the interactive agent lane long enough to make Telegram messages queue without responses for minutes.

## Best Possible Solution

Land a maintainer-approved compaction or memoryFlush maintenance lane, or an equivalent bounded/preemptible design, that lets foreground channel messages be acknowledged or processed promptly while preserving session transcript ordering and rotation safety.

## Reproduction Assessment

Yes. Source inspection on current main shows preflight compaction is awaited in the reply path, memoryFlush still uses `CommandLane.Main`, and compaction still queues through the same session/global lane structure; I did not perform a live Telegram timeout reproduction.

## Solution Assessment

Not yet. A separate maintenance lane or bounded/preemptible compaction path is the likely maintainable direction, but the final design needs maintainer approval because it changes scheduler and session-state ordering semantics.
