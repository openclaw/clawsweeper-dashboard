Keep open: current main still lacks an `upstreamRes` error listener, and this is a focused debug-proxy crash fix, but the PR should not merge yet because the claimed 502 behavior is not actually exercised and the supplied proof does not run the real proxy path.

## What This Changes

Adds an upstream response `error` listener in `src/proxy-capture/proxy-server.ts` and a proxy test for an origin that destroys the response socket.

## Best Possible Solution

Keep the upstream response error listener, but make the status behavior and regression test truthful: either delay committing upstream headers so pre-body aborts can return 502 and assert it, or explicitly test/document that mid-body aborts only close the stream without crashing.

## Reproduction Assessment

Yes, source-reproducible with medium confidence: current main reads from `upstreamRes` without an `error` listener, and Node throws on unhandled EventEmitter `error` events. I did not run a full real proxy repro in this read-only review.

## Solution Assessment

No, not yet. The handler addresses the crash class, but the claimed 502 path is undermined by the immediate `writeHead()` call and the new test does not assert the response status.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make the 502 branch reachable before sending headers:** `src/proxy-capture/proxy-server.ts:223-225`
  - body: The new handler only sets 502 when `!res.headersSent`, but this response callback immediately calls `res.writeHead(upstreamRes.statusCode ?? 502, upstreamRes.headers)` after registering the listener. Since `writeHead()` marks headers sent, later upstream response errors will keep the origin status/headers, and the new test does not catch that because it only checks that some data arrived. Delay committing upstream headers until a successful body/end path, or narrow the behavior and test to the stream-close semantics.
  - confidence: 0.88
