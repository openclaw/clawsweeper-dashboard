## Summary

Keep open: current main still logs root-help fast-path failures, sets `process.exitCode = 1`, and returns as handled, so a failed help render can still rely on an empty Node event loop instead of an explicit termination path.

## What This Changes

The issue asks for the root help fast-path error handler to terminate or drain reliably instead of only setting `process.exitCode`.

## Best Possible Solution

Fix the fast-path error boundary with a minimal explicit termination path, review the root-version sibling for the same invariant, and add child-process coverage proving a failed fast path cannot be held open by an active handle.

## Reproduction Assessment

No high-confidence real OpenClaw hang repro was run. Source inspection shows the fast path only sets `process.exitCode` and returns handled, and a focused Node probe confirms active handles keep such a process alive.

## Solution Assessment

Yes, the best fix belongs at the fast-path error boundary in `src/entry.ts`. The repair should stay narrow, preserve fast startup, and either cover or explicitly rule out the similar root-version fast-path failure branch.
