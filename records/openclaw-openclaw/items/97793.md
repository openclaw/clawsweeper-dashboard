## Summary

Keep open: current main and the latest release still set `process.exitCode` in the root-help fast-path error handler and then return as handled, so the requested termination behavior is not implemented. The report is narrow and plausible, but I did not reproduce a real `openclaw --help` hang against current main.

## What This Changes

Make root help fast-path error handling terminate or fully drain instead of only setting `process.exitCode`.

## Best Possible Solution

Fix `src/entry.ts` at the root-help fast-path error boundary with a minimal termination path and add a child-process regression proving a dangling handle cannot keep failed root help alive.

## Reproduction Assessment

No high-confidence real OpenClaw repro yet. Source inspection shows the fast-path handler only sets `process.exitCode`, and a focused Node probe confirms that active handles keep such a process alive.

## Solution Assessment

Yes, the best fix belongs at the fast-path error boundary in `src/entry.ts`. Use a minimal explicit exit or injected exit helper rather than importing the full runtime teardown path.
