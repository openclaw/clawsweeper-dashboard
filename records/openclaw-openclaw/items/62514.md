## Summary

Keep this open. Current main still leaves the mixed inline-image plus HTTP(S) prompt-reference path unresolved, and the issue is already paired with open PR #63710 using closing syntax, so cleanup should wait for that PR to land, be replaced, or be explicitly scoped down by a maintainer.

## What This Changes

The issue asks OpenClaw to handle Web UI image turns so uploaded images and prompt-referenced images reach vision-capable models natively instead of relying on weaker models to call the `image` tool correctly.

## Best Possible Solution

Treat PR #63710 as the canonical implementation candidate for the mixed-source HTTP image path, verify it preserves media fetch safety and covers the Web UI reproduction, and split any remaining local-path or tool-exposure behavior into a narrower follow-up before closing this issue.

## Reproduction Assessment

Yes for the deterministic mixed-source part: current main can be traced through `detectAndLoadPromptImages()` with `existingImages` plus an HTTPS URL, where the URL is not converted into a native image input. The exact hallucinated tool call is model-dependent, but the issue includes concrete logs from `glm-4.6v`.

## Solution Assessment

Unclear until PR #63710 is reviewed. Preloading explicit HTTP(S) image refs into the native vision payload is a narrow likely fix for the mixed-source case, but the maintainer should confirm whether the original uploaded-image tool-misuse case also requires hiding or gating the fallback `image` tool.
