## Summary

Keep this open. Current main still ignores explicit HTTP(S) image URLs during native prompt-image detection, and open PR #63710 is already the linked implementation candidate using closing syntax.

## What This Changes

Make Web UI mixed-image turns reliably include intended image inputs for vision-capable models without weak multimodal models falling into hallucinated image-tool calls.

## Best Possible Solution

Review and either land or supersede #63710 for the HTTP(S) mixed-source path, then split local-path drops or image-tool exposure into narrower follow-ups if maintainers want those changed.

## Reproduction Assessment

Yes for the deterministic mixed-source path: trace `detectAndLoadPromptImages()` with an existing inline image plus an HTTPS image URL and the URL is not added to native image input. The hallucinated tool call itself is model-dependent, but the issue includes concrete `glm-4.6v` logs.

## Solution Assessment

Unclear until #63710 is reviewed. Preloading explicit HTTP(S) image refs through the existing media loader is the narrow likely fix for mixed-source prompts, but fallback image-tool exposure may be a separate policy decision.
