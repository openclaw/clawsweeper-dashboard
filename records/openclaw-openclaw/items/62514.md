## Label Justifications

- `P2`: The report describes a normal-priority multimodal image-input bug with limited blast radius to mixed WebUI vision prompts and weaker tool-calling models.
- `impact:message-loss`: The affected path can silently omit user-provided image context from the native model request, so part of the user message is lost before response generation.

## Summary

Keep open: current main and the latest release still intentionally ignore HTTP(S) prompt image URLs before native vision assembly, and the same-repo closing PR is still open, conflicting, and awaiting maintainer review of the media-safety boundary.

## What This Changes

The issue reports that mixed WebUI image input can cause weaker multimodal models to ignore images or hallucinate `image` tool calls instead of reasoning over all user-provided image context.

## Best Possible Solution

Resolve mixed-image assembly in the runner by deterministically including safe prompt-referenced images before the first vision request, with the HTTP(S) media policy and regression/live proof settled on the linked PR or a maintainer-chosen replacement.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main ignores HTTP(S) prompt image refs before native vision assembly, and focused tests assert that local-only behavior; I did not run a live WebUI/GLM session.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a final merge path: deterministic mixed-image assembly before the first vision request is the right class of fix, but the safe HTTP(S) preload boundary still needs maintainer decision.
