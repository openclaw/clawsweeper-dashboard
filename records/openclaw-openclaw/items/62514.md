## Summary

Keep open: current main and the latest release still intentionally ignore remote HTTP(S) prompt image URLs before native vision assembly, while the same-repo closing fix at https://github.com/openclaw/openclaw/pull/63710 is open, conflicting, and still needs maintainer review for the HTTP preload safety boundary.

## What This Changes

The issue reports that WebUI mixed image inputs can leave prompt image sources out of native vision input, causing weaker multimodal models to ignore images or call the fallback image tool with hallucinated paths or URLs.

## Best Possible Solution

Keep this issue as the canonical report until maintainers land a repaired version of https://github.com/openclaw/openclaw/pull/63710 or choose another deterministic mixed-image assembly design with explicit HTTP(S), sandbox, and failure-message semantics.

## Reproduction Assessment

Yes at source level: current main still ignores HTTP(S) prompt image refs before native vision assembly, and focused tests assert that behavior. I did not run a live WebUI/glm-4.6v session.

## Solution Assessment

Unclear as a final product choice: deterministic mixed-image assembly before the first vision request is the right direction, but maintainers still need to decide the safe HTTP(S) preload, SSRF, sandbox, and user-visible failure boundary.
