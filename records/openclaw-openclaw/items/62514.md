## Summary

Keep this open. Current main still keeps HTTP(S) prompt image refs out of native prompt-image detection, so mixed inline upload plus remote image URL input can still leave the first vision request with incomplete image context; the linked implementation candidate remains open and unmerged.

## What This Changes

The issue reports WebUI multimodal image turns where uploaded images plus prompt-referenced images can be ignored or lead weak tool-calling models to call the image tool with hallucinated paths or URLs.

## Best Possible Solution

Review, rework, and land or supersede https://github.com/openclaw/openclaw/pull/63710 for safe remote image native preloading, then split local-path drops or tool-exposure policy into narrower follow-ups if needed.

## Reproduction Assessment

Yes for the deterministic mixed-source path: source inspection shows an existing inline image plus an HTTPS image URL leaves the URL out of `detectImageReferences()` and therefore out of the first native vision image payload. The exact hallucinated tool call still depends on the selected model.

## Solution Assessment

Unclear until the linked PR is reworked or reviewed. Preloading explicit remote image refs through the existing native image merge path is likely the narrow fix, but it must avoid arbitrary URL fetching and preserve the existing media-fetch security contract.
