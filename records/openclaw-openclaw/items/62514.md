## Summary

Keep open: current main and v2026.6.10 still intentionally ignore HTTP(S) prompt image URLs before native vision assembly, and the same-repo closing fix remains open, conflicting, and needs maintainer review for the HTTP preload boundary.

## What This Changes

The issue requests that WebUI mixed image turns with inline uploads and prompt-declared images or URLs be assembled so vision models receive the intended images instead of relying on fallback image-tool calls.

## Best Possible Solution

Keep this as the canonical issue until maintainers land a repaired deterministic mixed-image assembly fix or choose another design with explicit HTTP(S), sandbox, and failure-message semantics.

## Reproduction Assessment

Yes at source level: current main still ignores HTTP(S) prompt image refs before native vision assembly, and focused tests assert that behavior. I did not run a live WebUI/glm-4.6v session.

## Solution Assessment

Unclear as a final product choice: deterministic mixed-source assembly before the first vision request is the right direction, but maintainers still need to decide the safe HTTP(S) preload, SSRF, sandbox, and failure boundary.
