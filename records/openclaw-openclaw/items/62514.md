## Summary

Keep open: current main and v2026.6.10 still intentionally ignore HTTP(S) prompt image URLs before native vision assembly, while the same-repo closing PR remains open, conflicting, and needs maintainer review for the safe HTTP preload boundary.

## What This Changes

The issue requests deterministic WebUI multimodal image handling so uploaded images and prompt-declared image refs are assembled for vision models without relying on hallucination-prone fallback image-tool calls.

## Best Possible Solution

Keep this as the canonical issue until maintainers land a repaired version of https://github.com/openclaw/openclaw/pull/63710 or choose another deterministic mixed-image assembly design with explicit HTTP(S), sandbox, and failure-message semantics.

## Reproduction Assessment

Yes at source level: current main still ignores HTTP(S) prompt image refs before native vision assembly, and focused tests assert that behavior. I did not run a live WebUI/glm-4.6v session.

## Solution Assessment

Unclear as a final product choice: deterministic mixed-image assembly before the first vision request is the right direction, but maintainers still need to decide the safe HTTP(S) preload, SSRF, sandbox, and user-visible failure boundary.
