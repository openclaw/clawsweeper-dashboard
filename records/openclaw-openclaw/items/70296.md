## Summary

Keep open. Current main still has no restart-sentinel channel-notification suppression path, and BOOT startup still appends the canonical `NO_REPLY` footer after arbitrary BOOT.md content; the Control UI model read-back portion appears fixed by current code and tests.

## What This Changes

The issue asks to suppress restart-sentinel channel notices, fix Control UI model read-back display, and resolve a BOOT.md silent-reply token conflict.

## Best Possible Solution

Keep the issue open but narrow the remaining work: add a documented restart-sentinel notification policy that suppresses only user-channel notices while preserving internal wake/continuation behavior, and separately decide whether BOOT.md should keep the canonical `NO_REPLY` contract or gain a documented configurable silent-token mechanism.

## Reproduction Assessment

Yes. The restart-sentinel behavior is reproducible from current source and tests by scheduling a routed sentinel wake, which enqueues and delivers the notice; the BOOT behavior is reproducible by inspecting `buildBootPrompt`, which appends `NO_REPLY` after arbitrary BOOT.md content.

## Solution Assessment

No. Closing is not the best way to solve this because two reported areas still need source-level or product follow-up; the maintainable path is to split the narrow sentinel config change from the BOOT token-contract decision.
