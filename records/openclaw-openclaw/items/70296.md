## Summary

Keep open. Current main appears to have fixed the Control UI model read-back part, but the restart-sentinel channel notice still has no suppression surface and the BOOT.md silent-token behavior still injects the canonical `NO_REPLY` footer after arbitrary BOOT.md content.

## What This Changes

Requests suppression for restart-sentinel channel notices, correct Control UI model read-back after save, and resolution of BOOT.md silent-token conflicts.

## Best Possible Solution

Keep this open but split the remaining scope: add a documented restart-sentinel notification policy that suppresses only user-channel notices while preserving internal wake/continuation behavior, and separately decide whether BOOT.md should always use canonical `NO_REPLY` or expose a configurable silent-token contract.

## Reproduction Assessment

Yes. Source and focused tests show the sentinel path still enqueues and delivers routed notices, while `buildBootPrompt` still appends `NO_REPLY` after BOOT.md content; the UI read-back portion has a current render test showing the configured default model is selected.

## Solution Assessment

Unclear for the bundled issue as filed. The model read-back fix on main looks appropriate, but the sentinel suppression and BOOT token behavior need separate maintainer direction before the whole issue can be considered solved.
