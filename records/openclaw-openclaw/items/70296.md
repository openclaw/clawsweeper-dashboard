## Summary

Keep open. Current main appears to have fixed the Control UI model read-back portion, but restart-sentinel notices are still delivered to routed channels with no suppression config, and the BOOT.md path still appends the canonical silent token after workspace content, so the combined report is not fully implemented.

## What This Changes

Request to suppress restart-sentinel channel notices, fix Control UI model read-back display, and resolve BOOT.md silent-token footer conflicts.

## Best Possible Solution

Split or narrow the remaining work: add a documented policy/config surface that suppresses only user-channel restart-sentinel notices while preserving internal wake/continuation behavior, treat the Control UI read-back as already fixed, and make a maintainer call on whether BOOT.md should always use canonical NO_REPLY or expose a documented override contract.

## Reproduction Assessment

Yes. Source and focused test inspection on current main show the sentinel path still enqueues and delivers routed channel notices, while buildBootPrompt still appends NO_REPLY after BOOT.md content; the UI read-back portion has a render test for the configured default model.

## Solution Assessment

Unclear for the bundled issue as filed. The Control UI fix on main looks appropriate and sentinel suppression is narrow, but the BOOT.md token behavior should be split or explicitly scoped because current public docs define NO_REPLY as the supported silent token.
