## Summary

Keep open: current main receives Telegram’s new `guest_message` update type through grammY’s defaults but has no guest-specific ingress, authorization, session, or `answerGuestQuery` delivery path. The open implementation at https://github.com/openclaw/openclaw/pull/83632 remains the canonical path, has positive live Telegram proof and operator support, but currently conflicts with main and still needs maintainer approval of its security and session contracts.

## What This Changes

Add opt-in Telegram Guest Mode so authorized users can mention a bot outside its joined chats and receive a one-shot guest-query response.

## Best Possible Solution

Implement this in the Telegram plugin as an explicit opt-in capability that requests guest updates only when enabled, authorizes senders through one documented account-level rule, uses isolated non-persistent guest conversation identity, preserves configured account-to-agent routing, and delivers one supported response through `answerGuestQuery`.

## Reproduction Assessment

Yes at the source level: current main’s grammY update list includes `guest_message`, while exhaustive repository search finds no guest handler or `answerGuestQuery` delivery call. A live current-main Telegram run was not performed during this read-only review.

## Solution Assessment

No, a config toggle alone is not sufficient; the best solution also needs explicit authorization, isolated session identity, account routing, supported-output, and exactly-once delivery contracts. The linked Telegram-plugin PR is the appropriate implementation path once those decisions and its conflicts are resolved.
