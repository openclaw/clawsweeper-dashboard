## Summary

Keep open: current main still routes Bedrock adaptive-thinking Claude requests through a branch that preserves the caller maxTokens but does not derive a larger cap from the resolved model metadata, so the reported 4096-token default can still make long adaptive-thinking turns non-deliverable.

## What This Changes

The issue asks OpenClaw to stop hard-capping Amazon Bedrock adaptive-thinking Claude requests at 4096 output tokens and to honor resolved model or configured output-token limits.

## Best Possible Solution

Make the Bedrock adaptive-thinking path honor resolved model maxTokens by default while preserving explicit per-run or per-agent maxTokens overrides and the existing adaptive thinking request fields.

## Reproduction Assessment

Yes at the source level, but not via a live Bedrock credential run here. Current main shows adaptive-thinking Bedrock requests keep `options.maxTokens` while non-adaptive Claude requests already use `model.maxTokens`, matching the reported cap path.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing model-cap/thinking-budget logic for adaptive Bedrock Claude requests rather than adding a new config surface. Existing `agents.defaults.params`, per-agent params, and per-model params already provide override paths that should remain higher precedence.
