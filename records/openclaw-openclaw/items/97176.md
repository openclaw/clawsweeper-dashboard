## What This Changes

The issue asks Bedrock adaptive-thinking Claude requests to honor resolved model maxTokens or existing explicit overrides instead of remaining capped at 4096 output tokens.

## Best Possible Solution

Make Bedrock adaptive-thinking and Fable option resolution populate maxTokens from resolved model metadata when no explicit caller cap is set, preserving existing explicit maxTokens overrides and adaptive thinking payload behavior.

## Reproduction Assessment

Yes, at source level: current main shows adaptive Bedrock requests keep `options.maxTokens` while the non-adaptive Claude path derives a cap from `model.maxTokens`. I did not run a live AWS Bedrock repro in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse existing model-cap maxTokens semantics in the Bedrock provider runtime, not to add a new config surface. Existing `params.maxTokens` behavior should remain the explicit override path.
