## Summary

Keep open: current main and v2026.6.10 still leave Bedrock adaptive-thinking/Fable option resolution on the base maxTokens path while only the non-adaptive Claude branch applies the model maxTokens cap, and no merged duplicate fix was found.

## What This Changes

The issue reports that Bedrock Converse adaptive-thinking Claude models ignore resolved model maxTokens and remain capped at the 4096 base output budget.

## Best Possible Solution

Make Bedrock adaptive-thinking and Fable option resolution populate maxTokens from resolved model metadata when no explicit caller cap is set, preserving explicit maxTokens overrides and adaptive thinking payload behavior.

## Reproduction Assessment

Yes, at source level: current main shows adaptive Bedrock requests keep options.maxTokens unset while the non-adaptive Claude path derives a cap from model.maxTokens. I did not run a live AWS Bedrock repro in this read-only pass, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse existing model-cap maxTokens semantics in the Bedrock provider runtime, not to add a new config surface. Existing params.maxTokens behavior should remain the explicit override path.
