## Summary

Keep open: current main still has a Bedrock adaptive-thinking branch that preserves the caller/default maxTokens but does not derive the larger cap from resolved model metadata, so long adaptive-thinking Claude turns can still hit the reported 4096-token path.

## What This Changes

Report asks OpenClaw to stop hard-capping Bedrock adaptive-thinking Claude requests at the default output-token budget when the resolved model supports a larger maxTokens value.

## Best Possible Solution

Make the Bedrock adaptive-thinking option path honor resolved model maxTokens by default while preserving explicit per-request, per-model, and per-agent maxTokens overrides and the existing adaptive thinking payload fields.

## Reproduction Assessment

Yes at the source level: current main shows adaptive-thinking Bedrock requests keep `options.maxTokens` while the non-adaptive Claude path already derives a cap from `model.maxTokens`. I did not run a live AWS Bedrock repro in this read-only pass.

## Solution Assessment

Yes: the best narrow fix is to reuse the existing model-cap thinking-budget behavior for adaptive Bedrock Claude requests, not to add a new config surface. Existing `params.maxTokens` paths should remain explicit overrides.
