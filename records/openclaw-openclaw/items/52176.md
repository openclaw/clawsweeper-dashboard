## Summary

Keep open. The request is useful and the existing generic Docker APT build arg is only a workaround, but this patch is not safe to merge as-is because it adds root-level unpinned remote installer execution and still leaves uv-installed skill executables outside the runtime PATH; it also needs the required changelog entry.

## What This Changes

The PR adds an optional Docker build argument that runs the Astral uv installer during image build and moves uv and uvx into /usr/local/bin.

## Best Possible Solution

Add a dedicated uv Docker option only if it follows the repository's Docker supply-chain bar: pin a uv release or artifact, verify it before execution, install uv/uvx in a shared executable location, set `UV_TOOL_BIN_DIR` or PATH so uv-installed skill binaries are visible to OpenClaw, and document/changelog the user-facing build arg. Until then, keep the generic `OPENCLAW_DOCKER_APT_PACKAGES` workaround as the unsupported escape hatch.

## Reproduction Assessment

Yes. A static reproduction path is high-confidence: the PR diff shows unpinned `curl | sh`, current main shows stricter Docker artifact verification patterns, and current skill eligibility code requires uv-installed tool entrypoints to be found on PATH. I did not run a Docker build because this review is read-only.

## Solution Assessment

No. Installing latest uv with an unverified remote script and only relocating uv/uvx is not the narrowest maintainable solution for Docker skills; a safer fix should pin and verify uv and make uv tool executables visible to the non-root runtime user.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Pin and verify the uv installer:** `Dockerfile:203`
  - body: This build step runs a network-delivered installer as root with `curl ... | sh`, so rebuilding the same OpenClaw commit can execute different code later. The Dockerfile already uses pinned images and verifies the Docker apt key fingerprint; the uv path should likewise pin a release/artifact and verify it before execution.
  - confidence: 0.94
- **[P2] Put uv-installed tools on the node user's PATH:** `Dockerfile:203-206`
  - body: Moving uv and uvx to `/usr/local/bin` lets the launcher run, but `uv tool install` writes tool entrypoints to the user's executable directory unless `UV_TOOL_BIN_DIR` or PATH is configured. Skills such as nano-pdf are only eligible when their required executable is on PATH, so uv-backed skills can still install but remain unavailable in this image.
  - confidence: 0.9
- **[P3] Add the Docker feature to the changelog:** `Dockerfile:199`
  - body: This is a user-facing `feat(docker)` build argument, but the PR only changes the Dockerfile. Repo policy expects user-facing feature changes to add a `CHANGELOG.md` entry with contributor credit.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete Docker build supply-chain concern by executing an unpinned remote installer as root.

Concerns:

- **[high] Unverified root-level installer execution:** `Dockerfile:203`
  - body: The new Dockerfile block pipes `https://astral.sh/uv/install.sh` directly to `sh` during image build. If that endpoint or any transitively trusted release path changes or is compromised, the resulting OpenClaw image can silently bake in unexpected executable code; this is inconsistent with the Dockerfile's existing pinned image and Docker apt key verification posture.
