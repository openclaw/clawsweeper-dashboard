## Summary

Keep open: current main still lacks first-class uv Docker support, but this PR is not merge-ready because it executes an unpinned remote installer as root, leaves uv-installed tool executables off the runtime PATH, does not wire the supported Docker setup path, and lacks real behavior proof. The discussion usefully confirms an existing `OPENCLAW_DOCKER_APT_PACKAGES` workaround, so the need is real but still needs maintainer/security direction.

## What This Changes

Adds an optional Dockerfile build arg `OPENCLAW_INSTALL_UV` that runs Astral's uv installer and moves `uv` and `uvx` into `/usr/local/bin`.

## Best Possible Solution

If OpenClaw accepts this Docker feature, implement uv/uvx installation from a pinned or digest-verified source, configure a node-writable uv tool bin directory on `PATH`, wire setup/docs coverage, and validate the real Docker build/runtime path.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking PR defects are source-reproducible from the added Dockerfile block, current setup forwarding, OpenClaw's `uv tool install` path, PATH-based skill eligibility, and upstream uv storage docs.

## Solution Assessment

No. The direction is plausible, but this patch is not the best mergeable shape while it trusts a mutable remote installer as root and omits uv tool PATH/setup integration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pin the uv install artifact:** `Dockerfile:200`
  - body: This build path executes whatever `https://astral.sh/uv/install.sh` serves at build time as root, so rebuilding the same OpenClaw commit can trust different code. Use a pinned uv release artifact or installer URL with digest/signature verification before adding this production image knob.
  - confidence: 0.93
- **[P2] Expose uv-installed tools on PATH:** `Dockerfile:198-204`
  - body: Installing `uv` and `uvx` does not make later `uv tool install` executables visible to the non-root `node` runtime. Configure a node-writable `UV_TOOL_BIN_DIR` and include it in `PATH`, otherwise uv-backed skills such as `nano-pdf` can still fail the PATH-based `requires.bins` check.
  - confidence: 0.89
- **[P2] Wire the uv arg through Docker setup:** `Dockerfile:197`
  - body: The supported `scripts/docker/setup.sh` local-build path forwards an explicit build-arg list, and this PR only adds the Dockerfile arg. Operators setting `OPENCLAW_INSTALL_UV=1` around setup would still build an image without uv unless setup/docs are updated or the feature is explicitly raw `docker build` only.
  - confidence: 0.86
