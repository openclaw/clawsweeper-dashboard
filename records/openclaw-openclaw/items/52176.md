## Summary

Keep open: current main does not implement the uv Docker build arg, and the PR is not mergeable as-is because the added Docker path trusts an unpinned root installer and leaves setup/docs/runtime PATH gaps.

## What This Changes

The PR adds an opt-in Dockerfile build arg that installs uv and uvx into `/usr/local/bin` during image builds.

## Best Possible Solution

If core accepts this knob, install uv/uvx from a pinned or digest-verified source, configure a node-writable uv tool bin directory on PATH, wire setup/docs/changelog, and validate the Docker build path; otherwise document the existing custom-image workaround.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking review issues are source-reproducible from the PR-added Dockerfile block plus current Docker setup, skill installer, and skill eligibility code.

## Solution Assessment

No. The direction is plausible, but this is not the best mergeable shape while it trusts an unpinned installer as root and leaves uv tool PATH, setup, docs, and changelog surfaces incomplete.

Full review comments:

- **[P1] Pin the uv install artifact:** `Dockerfile:200`
  - body: This Docker build path executes whatever `https://astral.sh/uv/install.sh` serves at build time as root, so rebuilding the same OpenClaw commit can trust different code. Use a pinned uv release image/artifact with digest or checksum verification before adding this production image knob. ([docs.astral.sh](https://docs.astral.sh/uv/guides/integration/docker/))
  - confidence: 0.92
- **[P2] Expose uv-installed tools on PATH:** `Dockerfile:198-204`
  - body: Moving only `uv` and `uvx` to `/usr/local/bin` does not expose executables installed later by `uv tool install` when OpenClaw runs as `node`. Skills such as `nano-pdf` are gated by required bins on PATH, so set a node-writable `UV_TOOL_BIN_DIR` and include it in runtime PATH. ([docs.astral.sh](https://docs.astral.sh/uv/guides/integration/docker/))
  - confidence: 0.88
- **[P2] Wire the uv arg through Docker setup:** `Dockerfile:197`
  - body: The supported `scripts/docker/setup.sh` local build path passes a fixed list of build args, and this PR only adds the Dockerfile arg. Operators setting `OPENCLAW_INSTALL_UV=1` around setup would still build an image without uv unless setup/docs are updated or the feature is explicitly raw `docker build` only.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `Dockerfile:197`
  - body: This is a user-facing Docker feature, so repo policy requires a `CHANGELOG.md` entry for the `feat` change before merge. Add a single-line entry under the active release without inventing contributor attribution.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete supply-chain concern by executing an unpinned remote uv installer as root during optional Docker image builds.

Concerns:

- **[high] Unpinned remote installer execution:** `Dockerfile:200`
