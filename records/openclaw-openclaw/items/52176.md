## Summary

Keep open for maintainer review: current main does not implement this Docker uv knob, and the PR still adds an unpinned root-executed installer path with incomplete setup/PATH integration and no real Docker runtime proof. The workaround discussed in the comments shows a real user need, but it also reinforces that this is a product/security choice rather than a cleanup close.

## What This Changes

The PR adds an opt-in `OPENCLAW_INSTALL_UV` Docker build argument that installs `uv` and `uvx` into `/usr/local/bin`.

## Best Possible Solution

If OpenClaw accepts this Docker feature, install uv/uvx from a pinned and verified source, configure a node-writable uv tool bin directory on `PATH`, wire setup/docs/tests, and prove a real Docker build plus uv-backed skill flow.

## Reproduction Assessment

Not applicable for the requested feature itself. The blocking PR defects are source-reproducible from the added Dockerfile block, current Docker setup wiring, skill installer behavior, and PATH-based skill eligibility checks.

## Solution Assessment

No. The direction may be useful, but this patch is not the best mergeable form while it trusts a mutable remote installer as root and leaves node-visible uv tool paths plus supported setup wiring incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Pin the uv install artifact:** `Dockerfile:205`
  - body: This build path executes whatever `https://astral.sh/uv/install.sh` serves at build time as root, so rebuilding the same OpenClaw commit can trust different code. Use a pinned uv release URL or release artifact with digest/checksum verification before adding this production image knob.
  - confidence: 0.94
- **[P2] Expose uv tool executables to the node runtime:** `Dockerfile:199-207`
  - body: Installing `uv` and `uvx` does not make later `uv tool install` executables visible to the non-root runtime. Add a node-writable uv tool bin directory and include it on `PATH`, otherwise uv-backed skills such as `nano-pdf` can still fail the PATH-based `requires.bins` check after installation.
  - confidence: 0.9
- **[P2] Wire the uv arg through Docker setup:** `Dockerfile:199`
  - body: The supported `scripts/docker/setup.sh` local-build path forwards a fixed build-arg list, and this PR only adds the Dockerfile arg. Operators setting `OPENCLAW_INSTALL_UV=1` around setup would still build an image without uv unless setup/docs are updated or the feature is explicitly raw `docker build` only.
  - confidence: 0.87
