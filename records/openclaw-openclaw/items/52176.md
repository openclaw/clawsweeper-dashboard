## Summary

Keep open. The PR is useful, and the second commit addressed the missing `uvx` move, but the patch is still not merge-ready because it adds an unpinned root-level remote installer and does not make `uv tool install` executables visible to OpenClaw's PATH-based skill eligibility checks.

## What This Changes

The PR adds an `OPENCLAW_INSTALL_UV` Dockerfile build arg that runs the Astral uv installer and moves `uv` and `uvx` into `/usr/local/bin`.

## Best Possible Solution

If core should support this, implement uv installation from a pinned or digest-verified source and set a deterministic `UV_TOOL_BIN_DIR`/PATH for the non-root runtime, with Docker docs and focused tests updated; otherwise document the existing custom-image or `OPENCLAW_DOCKER_APT_PACKAGES` workaround.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking review concerns are source-reproducible from the PR diff, current Dockerfile/runtime PATH, OpenClaw's `uv tool install` path, and the official uv Docker/tool-bin documentation.

## Solution Assessment

No. The current PR shape is narrower than a full custom image, but it should not merge while it trusts the latest remote installer during a root image build and leaves uv-installed tool executables outside the runtime eligibility path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pin the uv install source:** `Dockerfile:200`
  - body: This adds a root Docker build path that runs whatever `https://astral.sh/uv/install.sh` serves at build time. Rebuilding the same OpenClaw commit can produce a different image or trust a compromised/moved installer; Astral's Docker guide recommends pinning a uv version and notes SHA256 pinning as best practice for reproducible environments. Use a pinned uv image digest, verified release artifact, or pinned installer URL with verification instead. citeturn1view0.
  - confidence: 0.91
- **[P2] Put uv tool executables on the runtime PATH:** `Dockerfile:199-203`
  - body: Installing `uv` and `uvx` into `/usr/local/bin` is not enough for uv-backed skills: OpenClaw runs `uv tool install`, then checks `requires.bins` by scanning `PATH`, while uv links installed tool executables into the uv tool bin directory unless `UV_TOOL_BIN_DIR` or related XDG/HOME settings say otherwise. Set a deterministic node-writable tool bin directory that is on PATH, or uv-installed skills such as `nano-pdf` can remain ineligible after installation. citeturn1view2.
  - confidence: 0.88
