## Summary

Keep open. The PR is useful source work and the discussion already found a workable custom-image workaround through `OPENCLAW_DOCKER_APT_PACKAGES`, but the patch should not merge as-is because it adds an unpinned root-level installer path, leaves uv-installed tool executables outside a deterministic runtime PATH, and does not wire the new build arg through the supported Docker setup/docs surface.

## What This Changes

The PR adds a Dockerfile-only `OPENCLAW_INSTALL_UV` build arg that downloads uv's installer and moves `uv` and `uvx` into `/usr/local/bin`.

## Best Possible Solution

If core supports this, install `uv`/`uvx` from a pinned or digest-verified source, set a node-writable `UV_TOOL_BIN_DIR` on `PATH`, wire the arg through Docker setup/docs, and add changelog plus focused Docker validation; otherwise document the existing custom-image workaround.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking review issues are source-reproducible from the PR-added Dockerfile block, current skill installer/eligibility code, Docker setup script, and official uv Docker/tool-bin documentation.

## Solution Assessment

No. The direction is plausible, but this is not the best mergeable shape while it trusts an unpinned installer as root, omits a deterministic uv tool-bin PATH, and leaves the supported Docker setup/docs surface incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pin the uv install source:** `Dockerfile:200`
  - body: This Docker build path executes whatever the uv installer endpoint serves at build time as root, so rebuilding the same OpenClaw commit can trust different code or fail if the installer layout changes. Use a pinned uv image/tag with digest verification or a pinned installer/release artifact before adding this production image knob.
  - confidence: 0.92
- **[P2] Expose uv tool installs on PATH:** `Dockerfile:198-204`
  - body: Moving `uv` and `uvx` to `/usr/local/bin` does not make executables installed by `uv tool install` visible to OpenClaw. Skills such as `nano-pdf` are gated by required bins on `PATH`, so define a node-writable `UV_TOOL_BIN_DIR` and include it in the runtime `PATH`.
  - confidence: 0.88
- **[P2] Wire the uv build arg through Docker setup:** `Dockerfile:197`
  - body: The supported `scripts/docker/setup.sh` build path passes an explicit set of build args, and this PR only adds the Dockerfile ARG. An operator setting `OPENCLAW_INSTALL_UV=1` for the setup script would still build an image without uv unless the script and Docker docs are updated or the feature is scoped to raw `docker build` only.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete supply-chain concern by executing an unpinned remote installer as root during optional Docker image builds.

Concerns:

- **[high] Unpinned remote installer execution:** `Dockerfile:200`
  - body: `curl -LsSf https://astral.sh/uv/install.sh | sh` trusts the current network response during a Docker build without a version pin, digest, signature, or release-artifact verification, making rebuilds nondeterministic and unsafe if the endpoint or installer changes.
