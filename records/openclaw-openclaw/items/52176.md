## Summary

Keep open. The request is plausible, but this PR is not merge-ready because it adds mutable root remote-script execution to the Docker build and still does not make uv-installed tool shims visible to OpenClaw's runtime skill eligibility checks. The discussion usefully confirms an existing custom-image workaround, but current main has not implemented this feature.

## What This Changes

The PR adds an optional `OPENCLAW_INSTALL_UV` Docker build argument that downloads Astral's uv installer and moves `uv` and `uvx` into `/usr/local/bin` for skills and plugins.

## Best Possible Solution

If maintainers want first-class uv support in the official Docker image, use a pinned and verifiable uv artifact or pinned official uv image copy, expose both launchers and tool-installed executables through `UV_TOOL_BIN_DIR` or PATH for the `node` runtime user, then add Docker docs plus a focused Dockerfile/container smoke. If they do not want that surface in the official image, document the existing custom build-arg workaround instead.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: the PR diff executes a mutable installer and only moves `uv`/`uvx`, while current main installs uv-backed skills with `uv tool install` and rejects skills whose required command is not PATH-visible.

## Solution Assessment

No. The proposed patch is not the best way to solve the request because official uv Docker guidance supports pinning and explicit tool-bin handling, while this PR leaves both reproducibility and installed-tool PATH behavior incomplete. citeturn0search2

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pin and verify the uv installer before executing it:** `Dockerfile:203`
  - body: This build step runs whatever `https://astral.sh/uv/install.sh` serves at build time as root whenever the arg is enabled. Rebuilding the same OpenClaw commit can execute different installer bytes, and a compromised upstream endpoint would be baked into the image; use a pinned, verifiable uv artifact instead.
  - confidence: 0.94
- **[P2] Expose uv tool shims to the runtime PATH:** `Dockerfile:198-203`
  - body: Installing only `uv` and `uvx` into `/usr/local/bin` does not make commands later created by `uv tool install` visible. OpenClaw checks `requires.bins` on PATH, so a uv-backed skill can install and still remain unavailable unless the image sets `UV_TOOL_BIN_DIR` or adds the node tool-bin directory to PATH.
  - confidence: 0.88
