## Summary

Keep open. The PR addresses a real Docker gap, but the current patch is not ready to merge because it adds a supply-chain-sensitive uv installer path without pinning/verification and does not make uv-installed tool executables available to the non-root runtime PATH.

## What This Changes

The PR adds an optional `OPENCLAW_INSTALL_UV` Docker build argument that runs Astral's uv installer and moves `uv` and `uvx` into `/usr/local/bin`.

## Best Possible Solution

Revise the Docker feature so uv installation is reproducible and safe: pin the uv version, verify the downloaded installer or release artifact before executing it, expose a node-owned uv tool executable directory on PATH, wire the build arg through the standard Docker setup/docs, and add the required changelog entry before merge.

## Reproduction Assessment

Yes. Source inspection reproduces the merge blockers: the PR's Dockerfile executes the moving uv installer, while current OpenClaw installs uv-backed skill dependencies with `uv tool install` and checks resulting binaries through PATH under the non-root `node` runtime.

## Solution Assessment

No. Adding `uv` and `uvx` is the right direction, but the current implementation is not the narrowest maintainable solution until it pins/verifies the installer and makes uv tool-installed entrypoints discoverable by OpenClaw's skill gating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pin the uv installer before executing it:** `Dockerfile:203`
  - body: This runs the moving `https://astral.sh/uv/install.sh` script during a root Docker build. Astral documents version-specific installer URLs, so rebuilding the same OpenClaw commit can fetch a different uv version or install layout; pin the version and verify the installer or release artifact before running it.
  - confidence: 0.9
- **[P2] Expose uv tool executables on PATH:** `Dockerfile:198-203`
  - body: `uv tool install` creates tool entrypoints in uv's executable directory, but this image later runs as `node` and the patch only moves the `uv` and `uvx` launchers into `/usr/local/bin`. A Python-backed skill can still fail `requires.bins` after installation unless a writable node-owned tool bin directory is configured on PATH.
  - confidence: 0.86
- **[P3] Wire the build arg through Docker setup:** `Dockerfile:199`
  - body: The documented Docker flow builds through `scripts/docker/setup.sh`, which currently forwards the existing Docker build args but not this new one. Without adding that pass-through and docs, setting `OPENCLAW_INSTALL_UV` for the standard setup path has no effect.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a new root Docker build path that executes a remote, moving installer script without an external pin or verification step.

Concerns:

- **[medium] Unverified moving installer execution:** `Dockerfile:203`

Even though the current uv installer verifies downloaded archives internally, OpenClaw should pin and verify the installer or a release artifact before adding this optional supply-chain path.
