## Summary

Keep open: current main still lacks this uv Docker build arg, and the PR has concrete merge blockers around supply-chain pinning, runtime PATH behavior, setup wiring, and changelog coverage.

## What This Changes

The PR adds an opt-in `OPENCLAW_INSTALL_UV` Docker build argument that downloads uv during image build and moves `uv`/`uvx` into `/usr/local/bin`.

## Best Possible Solution

If OpenClaw accepts a first-class uv Docker knob, install uv/uvx from a pinned or digest-verified source, configure a node-writable uv tool bin directory on PATH, wire setup/docs/changelog, and validate the Docker build path.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking PR defects are source-reproducible from the added Dockerfile block plus current setup, docs, skill install, and skill eligibility code.

## Solution Assessment

No. The direction is plausible, but this patch is not the best mergeable shape while it trusts a mutable remote installer as root and leaves uv tool PATH, setup/docs, and changelog surfaces incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pin the uv install artifact:** `Dockerfile:200`
  - body: This build path executes whatever `https://astral.sh/uv/install.sh` serves at build time as root, so rebuilding the same OpenClaw commit can trust different code. Use a pinned uv release image/artifact with digest or checksum verification before adding this production image knob.
  - confidence: 0.93
- **[P2] Expose uv-installed tools on PATH:** `Dockerfile:198-204`
  - body: Installing `uv` and `uvx` does not make later `uv tool install` executables visible to the non-root `node` runtime. Configure a node-writable `UV_TOOL_BIN_DIR` and include it in `PATH`, otherwise uv-backed skills such as `nano-pdf` can still fail the PATH-based `requires.bins` check.
  - confidence: 0.89
- **[P2] Wire the uv arg through Docker setup:** `Dockerfile:197`
  - body: The supported `scripts/docker/setup.sh` local-build path forwards an explicit build-arg list, and this PR only adds the Dockerfile arg. Operators setting `OPENCLAW_INSTALL_UV=1` around setup would still build an image without uv unless setup/docs are updated or the feature is explicitly raw `docker build` only.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `Dockerfile:197`
  - body: This is a user-facing Docker feature, but the PR changes only `Dockerfile`. Add a single-line `CHANGELOG.md` entry under the active release without inventing contributor attribution.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete supply-chain risk by executing an unpinned remote uv installer as root during optional Docker image builds.
