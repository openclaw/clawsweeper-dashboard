## Summary

Current main still has the embedded-run abort, wait, active-state, and queue helpers internally, but the plugin runtime `api.runtime.agent` surface does not expose them; the current Gateway `chat.abort` work is adjacent and does not replace this in-process plugin API gap.

## What This Changes

Expose `abortEmbeddedPiRun`, `waitForEmbeddedPiRunEnd`, `isEmbeddedPiRunActive`, and `queueEmbeddedPiMessage` through `api.runtime.agent` for bundled plugins.

## Best Possible Solution

Add a focused additive runtime contract that preserves the existing embedded-run helper semantics and documents it as the supported plugin path instead of the current `globalThis` side door.

## Reproduction Assessment

Yes. This is a feature/API gap with a high-confidence static reproduction path: current main contains the internal helpers under `src/agents` but omits them from the lazy runtime module, `createRuntimeAgent()`, the `PluginRuntimeCore` type, runtime docs, mocks, and shape coverage.

## Solution Assessment

Yes. Exposing the existing embedded-run helpers through `api.runtime.agent` is the narrowest maintainable solution for trusted in-process plugins; the broader runId/Gateway abort API tracked in #52412 should remain a separate contract.
