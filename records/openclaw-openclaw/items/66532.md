## Summary

Keep open. Current main still has the embedded-run abort, wait, active-state, and queue helpers internally, but `api.runtime.agent` only exposes the embedded run starters and session helpers; the related Gateway abort work and broader runId abort request do not satisfy this in-process plugin runtime API gap.

## What This Changes

Expose `abortEmbeddedPiRun`, `waitForEmbeddedPiRunEnd`, `isEmbeddedPiRunActive`, and `queueEmbeddedPiMessage` through `api.runtime.agent` for bundled plugin cancellation and coordination flows.

## Best Possible Solution

Add the missing plugin runtime contract as a focused additive API: re-export the helpers through the lazy embedded runtime module, lazy-bind them on `createRuntimeAgent()`, declare their types, update the runtime mock and shape coverage, refresh the Plugin SDK API baseline if it drifts, and document the supported cancellation and queueing semantics for plugin authors.

## Reproduction Assessment

Yes. This is a feature/API gap rather than a runtime crash; the high-confidence reproduction path is static current-main inspection showing the helpers exist under `src/agents` but are absent from the plugin runtime barrel, runtime object, type contract, docs, mock, and shape test.

## Solution Assessment

Yes. The requested additive runtime exposure is the narrowest maintainable solution for trusted in-process plugins that already start embedded runs, while the broader runId/Gateway abort API tracked in #52412 can remain separate because it has different identification and RPC semantics.
