## Summary

Keep open: current main still lacks the requested injected `api.runtime.agent` helper family, even though the internal sessionId-based helpers and partial public SDK aliases already exist. The related gateway abort work and the open runId-based abort request do not cover this narrower `api.runtime.agent` surface.

## What This Changes

Expose sessionId-based embedded-run abort, wait, active-state, and queue helpers through `api.runtime.agent` for plugins.

## Best Possible Solution

Add a narrow, documented, additive `api.runtime.agent` surface for the existing sessionId-based embedded-run helpers while keeping the runId abort API separate.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the API gap: the internal helpers exist under `src/agents`, while the injected plugin runtime module, builder, type, docs, mock, and shape test omit them.

## Solution Assessment

Yes. A focused lazy runtime binding plus type, mock, docs, and shape coverage is the narrowest maintainable path for trusted in-process plugin callers; it should not expand into the separate runId abort API.

## Repair Work Prompt

Implement the feature requested in https://github.com/openclaw/openclaw/issues/66532. Add only the existing sessionId-based embedded-run coordination helpers to `api.runtime.agent`: `abortEmbeddedPiRun`, `waitForEmbeddedPiRunEnd`, `isEmbeddedPiRunActive`, and `queueEmbeddedPiMessage`. Preserve lazy loading through `runtime-embedded-pi.runtime.ts`, declare the helpers in `PluginRuntimeCore['agent']`, update `createRuntimeAgent()`, the runtime shape test, the plugin runtime mock, and public SDK runtime docs. Keep this separate from the broader runId-based `api.runtime.agent.abort` design tracked at https://github.com/openclaw/openclaw/issues/52412, and do not change gateway `chat.abort` semantics or add config. Add a changelog entry if maintainers want this user-facing plugin API change recorded during landing.
