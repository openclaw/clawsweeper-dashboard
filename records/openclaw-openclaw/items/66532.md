## Summary

Keep open. Current main still has the embedded-run abort, wait, active-state, and queue helpers internally, and it now has adjacent Gateway abort support, but the requested in-process plugin runtime contract is still missing from `api.runtime.agent`. The CoClaw plugin context remains a valid example of why plugins should not depend on the process-global embedded-run state shape.

## What This Changes

Expose `abortEmbeddedPiRun`, `waitForEmbeddedPiRunEnd`, `isEmbeddedPiRunActive`, and `queueEmbeddedPiMessage` through the plugin runtime agent API for bundled plugin cancellation and coordination flows.

## Best Possible Solution

Add the missing plugin runtime contract in a focused PR: re-export the internal helpers from the lazy embedded runtime module, lazy-bind them on `createRuntimeAgent()`, declare them on `PluginRuntimeCore["agent"]`, update the runtime mock and runtime shape coverage, refresh the Plugin SDK API baseline, and document the supported cancellation/coordination semantics for plugin authors.

## Reproduction Assessment

Yes. This is a feature/API gap rather than a runtime bug; the high-confidence check is a static current-main inspection showing the helpers exist under `src/agents` but are absent from the plugin runtime barrel, runtime object, type contract, docs, and shape test.

## Solution Assessment

Yes, the requested additive runtime exposure is the narrowest maintainable path for in-process plugins that already start embedded agent runs. The broader runId-based `api.runtime.agent.abort` idea in #52412 can remain separate because it has different identification and Gateway/RPC semantics.
