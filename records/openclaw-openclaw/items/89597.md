## Summary

This PR is still useful because current main and v2026.6.6 still do runtime QQBot credential-backup JSON import, but it is not merge-ready: the proposed doctor/setup detector imports HOME/USERPROFILE QQBot credential backups into the active plugin state root, which can reintroduce cross-profile credential recovery after the linked isolation fix. The PR also still needs real after-fix migration proof and currently reports a dirty merge state.

## What This Changes

This PR removes QQBot runtime JSON/JSONL cache imports, moves credential-backup JSON import into setup/doctor plugin-state migration, and adds startup preflight detection plus tests/docs.

## Best Possible Solution

Keep the SQLite-only runtime/cache cleanup, but constrain credential-backup migration and preflight detection to the active state root unless maintainers explicitly approve a documented cross-root recovery mode, then add regression coverage and real migration proof.

## Reproduction Assessment

Yes. Source inspection shows the PR adds HOME/USERPROFILE as a QQBot credential-backup source, imports that backup into the active plugin state root, and current startup restores appId/clientSecret from plugin state into config; I did not run a mutating migration because this review is read-only.

## Solution Assessment

No. Moving legacy import out of runtime is the right direction, but automatic OS-home import into arbitrary active state roots is not the safest fix; the migration should stay scoped to the active state root or become an explicit maintainer-approved recovery mode.
