## Summary

Keep open: the SQLite-only runtime cleanup is useful, but this PR is not merge-ready because the new QQBot credential migration still scans HOME/USERPROFILE as a legacy credential source for an arbitrary active state root, and the PR body still lacks real doctor/setup migration proof.

## What This Changes

This PR removes QQBot runtime JSON/JSONL cache imports, adds a setup/doctor migration for QQBot credential-backup JSON files into plugin SQLite KV, updates startup preflight detection, docs, and tests.

## Best Possible Solution

Keep the runtime SQLite-only cleanup, but restrict QQBot credential-backup migration and startup preflight detection to the active state root unless maintainers explicitly approve a separate cross-root recovery mode with tests and redacted real migration proof.

## Reproduction Assessment

Yes, source-reproducible from the PR source: the detector adds HOME/USERPROFILE as a credential-backup root and the doctor migration persists detected entries into plugin KV, which QQBot startup later uses for credential restore. I did not execute the migration because this review is read-only.

## Solution Assessment

No. Moving legacy reads out of runtime is the right direction, but automatic HOME/USERPROFILE credential import into arbitrary active state roots is not the safest fix; the migration should stay state-root scoped or become an explicit maintainer-approved recovery mode.
