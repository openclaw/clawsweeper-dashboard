## Label Justifications

- `P1`: This PR touches QQBot credential recovery and can affect real custom-state gateway users by restoring appId/clientSecret from the wrong profile.

- `merge-risk: 🚨 security-boundary`: The new OS-home import path can cross the state-root isolation boundary established by the prior credential security fix.

## Summary

Keep this PR open: it is a useful QQBot SQLite migration cleanup, but the proposed migration still imports HOME/USERPROFILE QQBot credential backups into the active plugin KV state, which keeps the credential-boundary concern unresolved and needs proof from a real doctor/setup migration path before merge.

## What This Changes

The PR removes QQBot runtime JSON/JSONL imports, adds QQBot credential-backup legacy-state migration through setup/doctor, updates startup preflight detection, and adjusts docs/tests for SQLite-only runtime state.

## Best Possible Solution

Keep the runtime SQLite-only cleanup, but constrain doctor/setup migration and startup preflight to the active state root unless maintainers explicitly approve a documented cross-root recovery mode, then require real migration proof.

## Reproduction Assessment

Yes. Source inspection shows the PR can detect/import HOME or USERPROFILE QQBot credential backups into the active plugin KV state, and QQBot startup later restores those credentials into config when appId/clientSecret are missing.

## Solution Assessment

No. Moving legacy import out of runtime is the right direction, but automatic OS-home import into arbitrary active state roots is not the safest fix; it should stay scoped to active state or be an explicit maintainer-approved recovery path.
