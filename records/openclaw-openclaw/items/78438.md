Keep open for maintainer review: the patch is a plausible focused Telegram transport escape hatch with sufficient after-fix logs, but it adds a new user-facing env override that overlaps the existing Telegram network config surface and needs product/owner acceptance.

## What This Changes

The PR adds `OPENCLAW_TELEGRAM_FORCE_IPV4` to pin Telegram dispatcher policy to IPv4 from startup, with Telegram docs and changelog updates.

## Best Possible Solution

If maintainers want a process-level Telegram IPv4 escape hatch beyond existing network config, land it with focused fetch coverage; otherwise fix the existing network config contract tracked at https://github.com/openclaw/openclaw/issues/75574 instead.

Source inspection does confirm current main starts the default Telegram dispatcher without a force-IPv4 startup policy and only builds IPv4-only transport for sticky fallback after eligible failures.

## Solution Assessment
