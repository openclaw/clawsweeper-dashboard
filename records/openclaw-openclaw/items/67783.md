## Summary

Keep this PR open, but do not merge it as written: the main Feishu typing-target bug is real and the PR direction is narrow, yet the unconditional `ctx.messageId` threading would make card-action callbacks use synthetic `card-action-*` ids as reaction targets.

## What This Changes

The PR passes Feishu inbound message ids into the reply dispatcher so typing reactions can target the user’s message instead of the thread root.

## Best Possible Solution

Keep the root/thread reply routing unchanged, target typing reactions at a real inbound Feishu message only for actual message events, and preserve the existing no-target suppression for synthetic card actions.

## Reproduction Assessment

Yes, from source. A topic or `replyInThread` event where `ctx.rootId` differs from `ctx.messageId` currently routes replies to the root and uses that same id for the typing reaction; the PR regression is also source-reproducible because card actions synthesize `card-action-*` message ids that the PR would pass as `inboundMessageId`.

## Solution Assessment

No, not as written. Splitting reply target from typing target is the right narrow fix, but the implementation must avoid treating synthetic card-action ids as real reaction anchors and should keep the old no-target guard before selecting an inbound target.

## Review Findings
