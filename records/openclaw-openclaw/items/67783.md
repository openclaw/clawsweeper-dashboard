## Summary

Keep open: the PR targets a real Feishu/Lark typing-reaction bug and has useful proof, but the current diff would also make synthetic card-action callbacks send non-existent `card-action-*` ids to the Feishu reaction API.

## What This Changes

The PR adds an optional Feishu `inboundMessageId`, passes `ctx.messageId` into the reply dispatcher, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused tests.

## Best Possible Solution

Preserve the reply/thread target split, but only provide a real inbound typing target for actual Feishu message events; synthetic card-action callbacks should fall back to the real reply target or skip typing, with regression coverage.

## Reproduction Assessment

Yes, from source. Current main routes topic or configured `replyInThread` replies to the root while the dispatcher uses that same id for `addTypingIndicator`, so the reaction can land on the topic root instead of the user's child message.

## Solution Assessment

No, not as written. Splitting reply placement from typing-reaction targeting is the right narrow fix, but the branch must avoid using synthetic card-action ids as real Feishu reaction targets.
