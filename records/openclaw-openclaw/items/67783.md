## Summary

Keep open. The PR addresses a real Feishu/Lark thread-mode typing-reaction bug that is still present on current main, but the current patch would regress card-action callbacks by using synthetic card-action ids as Feishu reaction targets.

## What This Changes

The PR adds an optional Feishu inbound message id, threads `ctx.messageId` into the reply dispatcher, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused Feishu tests.

## Best Possible Solution

Land the typing-target split after ensuring only real inbound Feishu message ids can override the reply target, with regression coverage for both topic/replyInThread messages and card-action callbacks.

## Reproduction Assessment

Yes, from source. Current main routes topic or configured `replyInThread` replies to the root while the dispatcher uses that same id for `addTypingIndicator`, so the reaction can land on the thread root instead of the inbound child message.

## Solution Assessment

No, not as written. Separating reply placement from typing-reaction targeting is the right narrow fix, but the branch must keep synthetic card-action ids out of the Feishu reaction API and preserve existing fallback/no-target behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`
  - body: This passes `ctx.messageId` as `inboundMessageId` for every Feishu dispatch path, including card-action callbacks whose message id is synthesized as `card-action-${event.token}`. Because the dispatcher now prefers `inboundMessageId`, those callbacks would call the Feishu reaction API with a non-existent synthetic id instead of using the real reply target or preserving the current no-target suppression path.
