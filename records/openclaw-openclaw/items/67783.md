## Summary

Keep open. Current main still reuses the Feishu reply target as the typing-reaction target, and this PR addresses a real user-visible Feishu/Lark bug, but the patch currently introduces a card-action regression by letting synthetic callback IDs become reaction targets.

## What This Changes

The PR adds an optional Feishu inbound message id, passes `ctx.messageId` into the reply dispatcher, uses it for typing-reaction targeting with fallback to `replyToMessageId`, and adds focused tests.

## Best Possible Solution

Keep reply/thread placement anchored to the Feishu topic root, derive typing reactions from a real inbound Feishu message id only, and make synthetic card-action callbacks fall back to the real reply target or skip typing with regression coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible: topic or `replyInThread` messages route replies to the root while the dispatcher sends the typing reaction to that same reply target; the PR screenshots also show the before/after UI behavior.

## Solution Assessment

No, not as written. Separating reply placement from typing targeting is the right narrow fix, but the inbound typing target must be limited to real Feishu message events so card-action synthetic ids are not sent to the reaction API.
