## Summary

Keep open. The Feishu typing-target bug is valid and the PR has a plausible narrow fix, but the current diff also makes synthetic card-action IDs eligible for Feishu reaction calls, which needs a small guard and regression coverage before merge.

## What This Changes

The PR adds an optional Feishu inbound message target, passes `ctx.messageId` into the reply dispatcher, uses it for typing-reaction targeting with fallback, and adds focused Feishu tests.

## Best Possible Solution

Preserve the split between reply placement and typing-reaction targeting, but only pass a real inbound Feishu message ID for actual message events; synthetic card-action callbacks should fall back to the real reply target or skip typing, with regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main routes topic or configured `replyInThread` replies to the root while the dispatcher uses that same ID for typing, and the proposed diff would route synthetic card-action IDs into the same Feishu reaction API.

## Solution Assessment

No, not as written. Splitting reply placement from typing targeting is the right narrow fix, but the PR should guard `inboundMessageId` to actual Feishu message events so synthetic card actions fall back or skip typing.
