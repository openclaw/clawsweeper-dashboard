## Summary

Keep open. The PR targets a real Feishu/Lark topic typing-reaction bug that is still present on current main, but the branch still has a source-backed card-action regression where synthetic callback ids can become Feishu reaction targets.

## What This Changes

The PR threads Feishu `ctx.messageId` into the reply dispatcher and uses it as the preferred typing-reaction target, with focused topic/replyInThread tests.

## Best Possible Solution

Preserve topic/root reply placement, target typing reactions at real inbound Feishu message ids, and make synthetic card-action callbacks fall back to the real reply target or skip typing with regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main routes topic replies to the root while the dispatcher uses that same id for typing reactions. The PR body also provides live Feishu/Lark screenshots for the visible before/after behavior.

## Solution Assessment

No, not as written. Splitting reply placement from typing targeting is the right narrow fix, but the preferred typing target must be limited to real Feishu message events so synthetic card-action ids are not sent to the reaction API.
