## Summary

Keep open: the PR fixes a real Feishu/Lark typing-reaction targeting bug, but the branch currently makes synthetic card-action events prefer non-existent `card-action-*` message ids as typing targets.

## What This Changes

The PR adds an optional Feishu `inboundMessageId`, passes `ctx.messageId` through Feishu dispatch, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused tests.

## Best Possible Solution

Land the Feishu typing-target split after gating the new inbound target to real inbound message events and preserving existing card-action reply-target or no-target behavior.

## Reproduction Assessment

Yes, from source. Current main can route Feishu topic or configured `replyInThread` replies to the root while the dispatcher uses that same id for `addTypingIndicator`, so the reaction can land on the topic root instead of the child inbound message.

## Solution Assessment

No, not as written. Separating reply placement from typing-reaction targeting is the right narrow fix, but the branch must avoid using synthetic card-action ids as real Feishu reaction targets.
