## Summary

Keep open: the bug is valid and the PR fixes the main topic-thread path, but the current diff would make Feishu card-action callbacks prefer synthetic `card-action-*` ids as typing targets and send them to the external reaction API.

## What This Changes

The PR adds an optional Feishu inbound message id, passes `ctx.messageId` into reply dispatch, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused Feishu tests.

## Best Possible Solution

Land a narrow Feishu plugin fix that keeps reply/thread routing unchanged, targets typing reactions at real inbound message ids for real message events, and preserves card-action fallback or no-target suppression.

## Reproduction Assessment

Yes, from source. Current main routes topic or configured `replyInThread` replies to the thread root while typing uses `replyToMessageId`, and the PR regression is also source-reproducible because card actions synthesize `card-action-*` message ids that would be preferred as inbound typing targets.

## Solution Assessment

No, not as written. Splitting reply target from typing target is the right narrow fix, but the branch should only pass a real message id as the inbound typing target and add card-action coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`
  - body: This passes `ctx.messageId` as `inboundMessageId` for every Feishu dispatch path, including card-action callbacks that synthesize ids like `card-action-${event.token}`. Gate the inbound typing id to real message events.
