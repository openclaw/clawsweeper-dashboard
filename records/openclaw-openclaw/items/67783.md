## Summary

Keep open. Current main still conflates Feishu reply placement with typing-reaction targeting, and this PR addresses a real thread/topic UX bug, but the patch needs one correctness fix before merge because it would route synthetic card-action ids into the Feishu reaction API.

## What This Changes

This PR adds optional Feishu inbound-message targeting for typing reactions, passes `ctx.messageId` from Feishu dispatch, and adds focused thread/topic targeting tests.

## Best Possible Solution

Keep the reply/thread target split, but compute a real typing target only for actual inbound Feishu message events; synthetic card-action callbacks should fall back to their real reply target or suppress typing.

## Reproduction Assessment

Yes, from source. Current main routes topic or configured `replyInThread` replies to the root while the dispatcher uses that same id for `addTypingIndicator`, so the reaction can land on the topic root instead of the user's child message.

## Solution Assessment

No, not as written. Separating reply placement from typing-reaction targeting is the right narrow fix, but the branch must avoid using synthetic card-action ids as real Feishu reaction targets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`

Because the dispatcher now prefers `inboundMessageId`, those callbacks would call the Feishu reaction API with a non-existent synthetic id instead of preserving the current real reply target or no-target suppression behavior.
