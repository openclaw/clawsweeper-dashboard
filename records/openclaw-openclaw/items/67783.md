## Summary

Keep open. The PR addresses a real Feishu/Lark topic-thread typing-reaction bug and has sufficient before/after screenshot proof, but the patch still needs a narrow repair because it would use synthetic card-action message ids as Feishu reaction targets.

## What This Changes

The PR threads Feishu `ctx.messageId` into the reply dispatcher as `inboundMessageId`, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused tests.

## Best Possible Solution

Preserve the split between reply/thread placement and typing-reaction targeting, but only pass a real inbound Feishu message id for actual message events; synthetic card-action callbacks should fall back to the real reply target or skip typing, with regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main routes topic or configured `replyInThread` replies to the root while the dispatcher uses that same id for `addTypingIndicator`, so the reaction can land on the topic root instead of the user's child message.

## Solution Assessment

No, not as written. Splitting the reply target from the typing target is the right narrow fix, but the branch must avoid treating synthetic card-action ids as real Feishu message ids.
