## Summary

Keep this PR open, but it should not merge as written because the useful Feishu typing-target fix also sends synthetic card-action ids to the reaction API.

## What This Changes

The PR threads an optional Feishu inbound message id into the reply dispatcher, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused Feishu tests.

## Best Possible Solution

Land the Feishu typing-target split after gating the new inbound target to real inbound message events and preserving existing card-action fallback or no-target behavior.

## Reproduction Assessment

Yes, from source. Current main routes topic or configured `replyInThread` replies to the root while the dispatcher uses that same reply target for `addTypingIndicator`, so the reaction can land on the topic root instead of the child inbound message.

## Solution Assessment

No, not as written. Separating reply placement from typing-reaction targeting is the right narrow fix, but the branch should only use real inbound message ids and preserve card-action fallback/no-target behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`

Because the dispatcher now prefers `inboundMessageId`, those callbacks would call the Feishu reaction API with a non-existent synthetic id instead of preserving the current real reply target or no-target suppression behavior.
