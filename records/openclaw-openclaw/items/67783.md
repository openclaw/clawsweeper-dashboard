## Summary

Keep open: the PR addresses a real Feishu/Lark typing-reaction target bug, but the branch still appears to prefer synthetic card-action message ids as external reaction targets, which should block merge until guarded.

## What This Changes

The branch adds an optional inbound Feishu message id, passes `ctx.messageId` into reply dispatch, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused tests.

## Best Possible Solution

Land a narrow Feishu plugin fix that keeps reply/thread routing unchanged, targets typing reactions at real inbound message ids for real message events, and preserves card-action fallback or no-target suppression.

## Reproduction Assessment

Yes, from source. Current main routes topic or `replyInThread` replies to the thread root while typing uses `replyToMessageId`, and the PR regression is also source-reproducible because card actions synthesize `card-action-*` message ids that would be preferred as inbound typing targets.

## Solution Assessment

No, not as written. Splitting reply target from typing target is the right narrow fix, but the branch should only pass a real message id as the inbound typing target and add card-action coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`

Since the dispatcher now prefers `inboundMessageId`, those callbacks would call the Feishu reaction API with a non-existent synthetic id instead of preserving the current real reply-target fallback or no-target suppression behavior. Gate the inbound typing id to real message events.
