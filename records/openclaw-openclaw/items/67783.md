## Summary

Keep open: this is a real Feishu/Lark typing-reaction bug with sufficient visible proof, but the current diff still needs a guard so synthetic card-action message IDs are not sent to the Feishu reaction API.

## What This Changes

The PR adds an optional Feishu inbound message target, passes `ctx.messageId` into the reply dispatcher, uses it for typing reactions with fallback, and adds focused Feishu tests.

## Best Possible Solution

Preserve separate reply placement and typing-reaction targets, but pass a real inbound Feishu message ID only for actual message events; synthetic card actions should fall back to the real reply target or skip typing, with regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main routes topic/configured thread replies to a root reply target while typing uses that same ID, and the PR diff would additionally route synthetic card-action IDs into the typing reaction path.

## Solution Assessment

No, not as written. Splitting reply placement from typing targeting is the right narrow fix, but the PR should only prefer the inbound ID when it is a real Feishu message and should cover card-action fallback behavior.
