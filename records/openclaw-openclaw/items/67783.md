## Summary

Keep open: the reported Feishu/Lark typing-reaction target bug is real and the screenshots are sufficient proof, but the patch currently passes synthetic card-action message IDs into the reaction path and needs a narrow guard before merge.

## What This Changes

The PR adds an optional Feishu inbound message ID to the reply dispatcher, passes `ctx.messageId` from Feishu dispatch, uses it for typing-reaction targeting with fallback, and adds focused unit coverage.

## Best Possible Solution

Keep reply/thread placement anchored to the Feishu topic root while deriving typing reactions only from a real inbound Feishu message ID, with card-action fallback or skip behavior covered by regression tests.

## Reproduction Assessment

Yes, source-reproducible. Current main routes topic/configured thread replies to a root reply target while typing uses that same ID, and the proposed patch's card-action regression is also source-reproducible from the synthetic `card-action-*` message path.

## Solution Assessment

No, not as written. Separating reply placement from typing targeting is the right narrow fix, but the inbound typing target must be limited to real Feishu message events or card actions must fall back/skip typing.
