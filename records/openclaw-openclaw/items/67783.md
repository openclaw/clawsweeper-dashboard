## Summary

Keep open: current main still lacks a separate Feishu inbound typing target, and the PR has useful after-fix Lark screenshots, but the patch should not merge until it avoids sending synthetic card-action ids to the Feishu reaction API.

## What This Changes

The PR adds an optional Feishu `inboundMessageId`, passes `ctx.messageId` into reply dispatch, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused tests.

## Best Possible Solution

Land a narrow Feishu plugin fix that keeps reply/thread routing unchanged, targets typing reactions at real inbound message ids for real message events, and preserves card-action fallback or no-target suppression.

## Reproduction Assessment

Yes, from source and screenshots. Current main routes topic or `replyInThread` replies to the thread root while typing still uses `replyToMessageId`, and the PR screenshots show the reported Lark before/after behavior; the card-action regression is also source-reproducible from synthetic `card-action-*` message ids.

## Solution Assessment

No, not as written. Separating reply target from typing target is the right narrow fix, but the branch should only use real Feishu message-event ids as typing targets and keep card-action fallback or suppression behavior intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`
  - body: This passes `ctx.messageId` as `inboundMessageId` for every Feishu dispatch path, including card-action callbacks where `message_id` is synthesized as `card-action-${event.token}`. Since the dispatcher now prefers `inboundMessageId`, those callbacks would call Feishu's reaction API with a non-existent synthetic id, or start typing even when reply targets are suppressed. Gate the inbound typing id to real message events.
