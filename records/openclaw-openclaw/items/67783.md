## Summary

Keep open: the PR addresses a real Feishu/Lark typing-reaction bug and has sufficient visual proof, but the patch currently prefers `ctx.messageId` unconditionally and can send synthetic card-action ids to Feishu's reaction API.

## What This Changes

The PR adds an optional Feishu inbound message id, passes it from inbound handling into the reply dispatcher, targets typing reactions at that id with fallback to the reply target, and adds focused tests.

## Best Possible Solution

Preserve reply/thread placement on the Feishu topic root, target typing reactions only at real inbound Feishu message ids, and make synthetic card-action callbacks fall back to a real reply target or skip typing with regression coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible: topic or configured `replyInThread` messages route replies to the root while the dispatcher sends the typing reaction to that same reply target, and the inspected PR screenshots show the before/after UI behavior.

## Solution Assessment

No, not as written. Separating reply placement from typing targeting is the right narrow fix, but the inbound typing target must be limited to real Feishu message events so card-action synthetic ids are not sent to the reaction API.
