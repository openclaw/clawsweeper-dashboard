## Summary

Keep open. Current main still targets the Feishu typing reaction at the reply/thread root, and this PR is the right narrow direction, but the proposed fallback can re-enable typing reactions for intentionally no-target synthetic card-action events by using `ctx.messageId` even when there is no real Feishu message anchor.

## What This Changes

The PR adds an optional Feishu `inboundMessageId`, passes `ctx.messageId` from Feishu inbound dispatch into the reply dispatcher, uses it for typing-reaction targeting with `replyToMessageId` fallback, and adds focused Feishu tests.

## Best Possible Solution

Keep the separate inbound typing target, but preserve the existing no-reply-target suppression: require a real `replyToMessageId`/reaction anchor before adding a typing reaction, then target `inboundMessageId ?? replyToMessageId`. Add tests for the topic child-message target and for suppressed card-action events, plus the required Unreleased changelog entry.

## Reproduction Assessment

Yes. The reported bug is reproducible from source by combining topic/replyInThread routing, where `replyTargetMessageId` becomes `ctx.rootId`, with the dispatcher path that reacts to `replyToMessageId`; the PR regression is reproducible with an inbound id present and `replyToMessageId` absent, matching card-action no-target events.

## Solution Assessment

No, not as written. The main direction is the narrow maintainable fix, but the implementation should keep the old no-target guard and use the inbound id only after confirming there is a real message-reaction anchor.
