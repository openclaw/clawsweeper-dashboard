## Summary

Keep open: the Feishu Topic-mode Typing target bug is real and the supplied screenshot proof supports the central fix, but the current branch still passes `ctx.messageId` as the preferred Typing target for every Feishu turn, including synthetic callback/menu/reaction events.

## What This Changes

The PR threads an optional Feishu `inboundMessageId` from `handleFeishuMessage` into the reply dispatcher and makes Typing reactions prefer that id over `replyToMessageId`, with topic-mode regression tests.

## Best Possible Solution

Land a guarded version that keeps final replies anchored to topic roots while Typing reactions target real inbound Feishu messages, and leaves synthetic callback/menu/reaction turns on their existing real reply target or without a Typing target.

## Reproduction Assessment

Yes. Source inspection on current main shows topic sessions choose the root as `replyTargetMessageId` while the dispatcher adds Typing to that same id, and the PR screenshots show the live before/after behavior; the remaining synthetic-event regression is also source-reproducible from the event builders.

## Solution Assessment

No, not yet. Separating the inbound Typing target is the right fix shape, but this implementation should only set that target for real Feishu message events and should cover synthetic callback/menu/reaction siblings before merge.
