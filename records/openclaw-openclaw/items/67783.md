## Summary

Keep open: the PR addresses a real Feishu/Lark topic typing-target bug and the supplied screenshots prove the intended visible change, but the current diff still lets synthetic card-action ids become Feishu reaction targets.

## What This Changes

The PR adds an optional Feishu inbound message id, passes `ctx.messageId` into reply dispatch, prefers it for typing reactions with `replyToMessageId` fallback, and adds focused tests.

## Best Possible Solution

Preserve topic/root reply placement, target typing reactions at real inbound Feishu message ids, and make synthetic card-action callbacks fall back to the real reply target or skip typing with regression coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible because topic or configured `replyInThread` flows can route replies to the root while the dispatcher uses that same id for typing reactions; the PR screenshots show the visible before/after behavior.

## Solution Assessment

No, not as written. Splitting reply placement from typing targeting is the right narrow fix, but the preferred typing target must be limited to real Feishu message events so synthetic card-action ids are not sent to the reaction API.
