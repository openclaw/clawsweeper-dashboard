## Summary

Keep open: this PR targets a real Feishu/Lark typing-reaction bug, but the current diff has a blocking card-action regression because it can send synthetic card-action ids to the Feishu reaction API.

## What This Changes

The PR adds optional Feishu inbound-message routing so typing reactions target the user's inbound message instead of the reply/thread root, with focused bot and dispatcher tests.

## Best Possible Solution

Preserve the reply/thread routing split, but pass a separate typing target only for real inbound Feishu message events and keep card-action callbacks on the existing real reply-target or no-target behavior.

## Reproduction Assessment

Yes, from source. Current main routes topic or configured `replyInThread` replies to the root while the typing reaction uses that same reply target, so the reaction can land on the thread root instead of the user's child message.

## Solution Assessment

No, not as written. Splitting reply placement from typing reaction target is the right narrow fix, but the branch should only use real inbound message ids and must preserve card-action fallback/no-target behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`
  - body: This passes `ctx.messageId` as `inboundMessageId` for every Feishu dispatch path, including card-action callbacks that synthesize ids like `card-action-${event.token}`. Since the dispatcher now prefers `inboundMessageId`, those callbacks would call the Feishu reaction API with a non-existent synthetic id instead of preserving the current real reply-target fallback or no-target suppression behavior. Gate the inbound typing id to real message events and add card-action coverage.
  - confidence: 0.88
