## Summary

Keep open. The PR addresses a real Feishu/Lark thread-mode typing-reaction bug and has sufficient screenshot proof, but the current diff still introduces a card-action regression by treating synthetic callback ids as valid Feishu message ids for typing reactions.

## What This Changes

The PR adds an optional Feishu `inboundMessageId` typing target, passes `ctx.messageId` into the reply dispatcher, and adds focused Feishu tests.

## Best Possible Solution

Preserve the split between reply/thread placement and typing-reaction targeting, but only pass a real inbound Feishu message id for actual message events; synthetic card-action callbacks should fall back to the real reply target or skip typing, with regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main routes topic or configured `replyInThread` replies to the root while the dispatcher uses that same id for `addTypingIndicator`, and the proposed diff would route synthetic card-action ids directly into the same Feishu reaction API.

## Solution Assessment

No, not as written. Splitting reply placement from typing targeting is the right narrow fix, but the PR must avoid treating synthetic card-action ids as real Feishu message ids.
