## Summary

Keep open. The PR addresses a real Feishu/Lark typing-reaction target bug that current main still has, but the patch as written would make synthetic card-action callbacks use non-existent `card-action-*` ids as reaction targets.

## What This Changes

The PR adds an optional Feishu inbound message id, passes it from inbound dispatch, uses it as the typing-reaction target with reply-target fallback, and adds focused Feishu tests.

## Best Possible Solution

Preserve root/thread reply routing, target typing reactions at a real inbound Feishu message only for actual message events, and keep synthetic card-action no-target suppression intact.

## Reproduction Assessment

Yes, from source. A topic or `replyInThread` event where `ctx.rootId` differs from `ctx.messageId` currently routes replies to the root and uses that same id for the typing reaction; the PR regression is also source-reproducible because card actions synthesize `card-action-*` message ids.

## Solution Assessment

No, not as written. Splitting reply target from typing target is the right narrow fix, but the implementation must avoid treating synthetic card-action ids as real reaction anchors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`
  - body: This passes `ctx.messageId` as `inboundMessageId` for every Feishu dispatch path, including card-action callbacks that synthesize ids like `card-action-${event.token}`. Since the dispatcher now prefers `inboundMessageId`, those callbacks would send typing reaction API calls to non-existent synthetic ids instead of preserving the current real reply-target or no-target suppression behavior; gate the inbound typing id to real message events in both call sites.
  - confidence: 0.88
