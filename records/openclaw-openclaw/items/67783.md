## Summary

Keep open. The PR addresses a real Feishu/Lark typing-reaction target bug, but the patch still prefers `ctx.messageId` unconditionally, which can send synthetic card-action ids to Feishu's reaction API.

## What This Changes

The PR threads `ctx.messageId` as `inboundMessageId` through Feishu reply dispatch and uses it as the preferred target for the typing reaction, with focused tests.

## Best Possible Solution

Preserve topic/root reply placement, target typing reactions at real inbound Feishu message ids, and make synthetic card-action callbacks fall back to the real reply target or skip typing with regression coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible: topic or configured `replyInThread` flows route replies to the root while the dispatcher uses that same root id for typing reactions; the PR screenshots show the visible before/after behavior.

## Solution Assessment

No, not as written. Splitting reply placement from typing targeting is the right narrow fix, but the preferred typing target must be limited to real Feishu message events so synthetic card-action ids are not sent to the reaction API.
