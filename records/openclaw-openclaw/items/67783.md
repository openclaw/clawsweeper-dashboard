## Summary

This PR still addresses a real Feishu/Lark user-visible bug on current main, but the proposed patch introduces a concrete card-action regression by preferring synthetic callback ids as reaction targets, so it should stay open for a narrow repair before merge.

## What This Changes

The PR threads an optional `inboundMessageId` through Feishu message handling into the reply dispatcher, uses it as the typing-reaction target with `replyToMessageId` fallback, and adds focused dispatcher and bot tests.

## Best Possible Solution

Land the Feishu fix after gating the inbound typing target to real inbound message events, preserving existing reply/thread routing and card-action fallback behavior with regression coverage.

## Reproduction Assessment

Yes, from source. Current main routes topic or configured `replyInThread` replies to the root while the typing reaction uses that same root id, so the reaction can land on the thread root instead of the user's child message.

## Solution Assessment

No, not as written. Splitting reply placement from typing reaction target is the right narrow fix, but the branch should only use real inbound message ids and preserve card-action fallback/no-target behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep synthetic card-action ids out of typing targets:** `extensions/feishu/src/bot.ts:1101`
  - body: This passes `ctx.messageId` as `inboundMessageId` for every Feishu dispatch path, including card-action callbacks that synthesize ids like `card-action-${event.token}`. Since the dispatcher now prefers `inboundMessageId`, those callbacks would call the Feishu reaction API with a non-existent synthetic id instead of preserving the current real reply-target fallback or no-target suppression behavior. Gate the inbound typing id to real message events and add card-action coverage.
