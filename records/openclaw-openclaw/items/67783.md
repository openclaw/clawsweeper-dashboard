## Summary

## What This Changes

The PR adds an optional Feishu inbound message id, passes `ctx.messageId` to the reply dispatcher, prefers it for typing reactions, and adds focused tests.

## Best Possible Solution

Preserve root/thread reply routing while targeting typing reactions only at real inbound Feishu message ids, with synthetic card-action callbacks falling back to the real reply target or skipping typing and tests for both paths.

## Reproduction Assessment

Yes for source-level reproduction: current main routes topic or configured `replyInThread` replies to the root while the dispatcher uses that same id for `addTypingIndicator`. I did not run live Feishu or tests in this read-only review, but the inspected before/after screenshots show the visible behavior.

## Solution Assessment

No, not as written. Splitting reply placement from typing targeting is the right narrow fix, but the preferred typing target must be limited to real Feishu message events so synthetic card-action ids fall back or skip typing.
