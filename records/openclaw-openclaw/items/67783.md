## Summary

Keep open; the reported Feishu/Lark typing-reaction bug is valid and the branch is directionally right, but the current patch would route card-action typing reactions to synthetic `card-action-*` ids before merge.

## What This Changes

The PR adds an optional Feishu inbound message id, passes `ctx.messageId` into the reply dispatcher, prefers it for typing-reaction targeting, and adds focused Feishu tests.

## Best Possible Solution

Land a narrow Feishu plugin fix that keeps reply/thread routing unchanged, targets typing reactions at a real inbound message for real message events, and preserves existing card-action fallback or no-target suppression behavior.

## Reproduction Assessment

Yes, from source. A topic or `replyInThread` event where `ctx.rootId` differs from `ctx.messageId` uses the root as the reply target today, and the PR regression is also source-reproducible because card actions synthesize `card-action-*` message ids.

## Solution Assessment

No, not as written. Splitting reply target from typing target is the right narrow fix, but the implementation must avoid treating synthetic card-action ids as real Feishu message anchors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Since the dispatcher now prefers `inboundMessageId`, those callbacks would send typing reaction API calls to non-existent synthetic ids instead of preserving the current real reply-target fallback or no-target suppression behavior; gate the inbound typing id to real message events.
