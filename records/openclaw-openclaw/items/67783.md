## Summary

Keep this PR open, but it needs changes before merge: the visible Feishu/Lark typing-target bug is real and the screenshots are useful proof, while the current diff would also route synthetic card-action ids into the reaction API.

## What This Changes

The PR adds an optional Feishu `inboundMessageId`, passes `ctx.messageId` into reply dispatch, prefers it for typing reactions, and adds focused Feishu tests.

## Best Possible Solution

Preserve the reply-vs-typing target split, but only use a real Feishu inbound message id as the typing target and add regression coverage for card-action synthetic events.

## Reproduction Assessment

Yes, from source. Current main can route topic or configured `replyInThread` replies to the root while the dispatcher uses that same id for `addTypingIndicator`, so the reaction can land on the topic root instead of the user's child message.

## Solution Assessment

No, not as written. Separating reply placement from typing-reaction targeting is the right narrow fix, but the branch must avoid using synthetic card-action ids as real Feishu reaction targets.
