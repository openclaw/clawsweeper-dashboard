## Summary

Keeping this open: current main still has a source-level sessions counter mismatch, and the requested archive-exclusion behavior conflicts with intentional archive-indexing tests and comments, so maintainers need to decide the product/privacy policy before changing active search semantics.

## What This Changes

The issue requests aligning session memory status/indexing file enumeration so reset/deleted archive files are not counted as live indexed session files.

## Best Possible Solution

Define one shared session-file enumeration contract for status and indexing, use it for the counter, and make a separate explicit maintainer decision on whether reset/deleted archives remain active-search sources or move to an archive-only corpus.

## Reproduction Assessment

Yes for the counter mismatch by source inspection: a sessions directory with live `.jsonl` plus `.jsonl.reset.*` or `.jsonl.deleted.*` files will index through the usage-counted helper while the status denominator counts only names ending in `.jsonl`. The dirty-state race is unclear because I did not establish a current-main failing path for it.

## Solution Assessment

No for the requested archive-skipping fix as an automatic repair: current tests and comments intentionally keep reset/deleted archives searchable. The narrow maintainable fix is to align the status denominator with the same helper used for indexing, then separately decide and document archive recall semantics.
