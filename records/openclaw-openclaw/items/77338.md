## Summary

Keep this open: current `main` and latest `v2026.6.6` still show the status denominator using a strict `.jsonl` suffix count while the session indexer uses the usage-counted session helper. The archive-indexing part of the report has been superseded by intentional current behavior, but the counter mismatch remains the canonical open work.

## What This Changes

The issue reports that `openclaw memory status` can show session file-count overflow because status scanning and session indexing use different file filters for reset/deleted archives and snapshots.

## Best Possible Solution

Keep this issue open as the canonical tracker and align `openclaw memory status` session counts with the current usage-counted session file contract while preserving reset/deleted archive recall.

## Reproduction Assessment

Yes, from source: current `memory status` counts only `*.jsonl` files while indexing uses the usage-counted session file helper that includes reset/deleted archives and excludes checkpoints. I did not run a live SQLite reproduction because this review is read-only.

## Solution Assessment

No for the original proposed exclusion of reset/deleted archives; current code intentionally keeps those archives searchable. The best fix is to share or reuse the canonical usage-counted session file enumeration for status counts without changing archive recall semantics.
