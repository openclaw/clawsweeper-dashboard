## Summary

Keep this open: current `main` and the latest `v2026.6.6` release still have the session-status denominator using a strict `.jsonl` suffix count while the session indexer uses the usage-counted session helper. The original archive-exclusion fix direction is now stale because reset/deleted archives are intentionally indexed for post-reset recall, but the counter mismatch remains real.

## What This Changes

Fix `openclaw memory status` session source counts so status and indexing use the same session artifact filter.

## Best Possible Solution

Align the memory status session denominator with the current usage-counted session transcript helper, preserving reset/deleted archive recall and excluding backup/checkpoint/trajectory artifacts consistently.

## Reproduction Assessment

Yes. Source inspection shows `memory status` counts only filenames ending in `.jsonl`, while session indexing uses the usage-counted helper that includes `.jsonl.reset.*` and `.jsonl.deleted.*`; I did not run a live CLI repro because this review is read-only.

## Solution Assessment

No for the issue body's preferred archive-exclusion path; current tests protect reset/deleted archive recall. The narrow maintainable fix is to make status share the indexer's usage-counted session-file contract.
