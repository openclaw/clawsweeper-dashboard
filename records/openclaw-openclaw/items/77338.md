## Summary

Keep open: current main still has a source-level mismatch where `memory status` counts only filenames ending in `.jsonl`, while the sessions indexer deliberately includes usage-counted `.jsonl.reset.*` and `.jsonl.deleted.*` archives. The counter overflow is valid, but the requested archive-skipping behavior conflicts with current tests/comments and needs maintainer policy direction.

## What This Changes

The issue asks for sessions memory status and indexing to treat reset/deleted transcript archives consistently so source counts do not overflow and active search semantics are clear.

## Best Possible Solution

Make the status denominator share the same session file enumeration contract as indexing, then separately decide whether reset/deleted archives should remain searchable or move to a different corpus.

## Reproduction Assessment

Yes for the counter mismatch by source inspection: a sessions directory containing `active.jsonl` plus `active.jsonl.reset.*`/`.deleted.*` will be indexed through `listSessionFilesForAgent` but counted by `scanSessionFiles` only if the filename ends in `.jsonl`. I did not establish a live current-main reproduction for the nondeterministic dirty-state report.

## Solution Assessment

No: the preferred suggestion to skip reset/deleted archives is not currently the safest fix because current code and tests explicitly preserve those archives for search. The narrow maintainable repair is to share enumeration for status/counting while leaving archive search semantics to a maintainer decision.
