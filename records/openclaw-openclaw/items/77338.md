## Summary

Keep open: current main still has the source-level `memory status` sessions counter mismatch, while reset/deleted archive recall is intentional and partly covered by the separate archive-retention policy thread.

## What This Changes

The issue reports that `openclaw memory status` can show more indexed session files than scanned session files because reset/deleted session archives are counted by indexing but not by the status denominator.

## Best Possible Solution

Align `memory status` with the usage-counted session transcript enumeration used by indexing, and leave archive recall/retention semantics to the explicit policy work.

## Reproduction Assessment

Yes for the counter mismatch by source inspection: construct a sessions directory with `active.jsonl` plus `active.jsonl.reset.<timestamp>` and current status scanning counts only the active file while the indexer enumerates both. The dirty-state race remains unclear without a runtime reproduction.

## Solution Assessment

No for the requested archive-skipping fix as an automatic path because current main intentionally keeps reset/deleted archives searchable. The safer narrow fix is to share or mirror the usage-counted session file enumeration for status totals while maintainers decide archive policy separately.
