## Summary

Keep open. The reported Feishu bug is still real on current main, but the PR's module-lifetime registration guard is unsafe because plugin reloads clear active registry state while the flag would stay set. The config-preservation direction is correct, but the branch needs to be rebased onto the current manifest-gated setup-promotion contract.

## What This Changes

The PR adds a Feishu full-registration idempotency guard, Feishu named-account promotion keys, a Feishu promotion fallback, and regression tests for duplicate registration and top-level groupPolicy preservation.

## Best Possible Solution

Keep the Feishu groupPolicy preservation fix, but rework it onto the current setup-promotion contract by exposing Feishu named-account promotion through the setup adapter and package metadata. Replace the process-wide registration guard with registry-scoped or replacement-safe idempotency so reloads can re-register Feishu hooks and tools.

## Reproduction Assessment

Yes. The duplicate-registration path is source-reproducible by calling the Feishu bundled entry registration twice, and the groupPolicy rewrite path follows from a named-account Feishu config entering the legacy normalizer without a Feishu named-account promotion filter.

## Solution Assessment

No. The named-account promotion direction is appropriate, but the process-wide guard is not the best fix because plugin reloads intentionally clear active registrations and require re-registration into the new registry.
