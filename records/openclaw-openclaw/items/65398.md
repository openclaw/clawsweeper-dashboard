## Summary

Keep open: this targets a real Feishu regression, but the branch still has merge-blocking correctness issues around registry-scoped registration and setup-promotion metadata, and it lacks inspectable real behavior proof.

## What This Changes

The branch adds Feishu full-registration idempotency, Feishu named-account promotion filtering/fallback handling, and regression tests for duplicate registration and top-level groupPolicy preservation.

## Best Possible Solution

Keep the Feishu bug fix, but make registration idempotency registry-scoped, declare Feishu config-promotion support through package metadata, and merge only after focused tests plus redacted live proof.

## Reproduction Assessment

Yes, at source level. Current main promotes static `groupPolicy` without a visible Feishu named-account filter, and the registration path can be rerun while active plugin runtime state is cleared on activating reloads.

## Solution Assessment

No. The PR fixes the right behavior in principle, but the process-lifetime guard should be scoped to the active registry/API and the promotion metadata should be exposed through the manifest-gated setup contract.

  - body: The module-level `feishuToolsRegistered` flag survives beyond a plugin registry rebuild. Add the Feishu package feature hint so the normalizer can see this filter before preserving top-level policy keys.
