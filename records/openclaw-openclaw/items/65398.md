## Summary

Keep open. Current main still lacks the Feishu named-account promotion and duplicate-registration protection, so this PR is not obsolete, but the proposed branch should not merge as-is because the process-wide registration guard can skip re-registration after an activating plugin reload clears registry state, and the config-promotion fix needs to align with the current manifest-gated setup contract.

## What This Changes

The PR adds a Feishu full-registration idempotency guard, Feishu named-account promotion keys plus a core fallback, and regression tests for duplicate registration and preserving top-level `groupPolicy` in named-account configs.

## Best Possible Solution

Preserve the Feishu named-account `groupPolicy` fix through the current manifest-first setup promotion contract, and make Feishu registration idempotent without using process-lifetime state that survives active registry reloads.

## Reproduction Assessment

Yes. The registration issue is source-reproducible by re-running Feishu full registration and by following the loader path that clears active registration state before reload; the config rewrite follows from static `groupPolicy` promotion without a Feishu named-account promotion filter.

## Solution Assessment

No. The named-account promotion direction is right, but the implementation should use current package setup metadata and a reload-safe idempotency mechanism instead of a process-wide guard plus a stale core fallback.
