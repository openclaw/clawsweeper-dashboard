## Summary

Keep open: the Feishu bug is valid on current main, but this PR is not mergeable as-is because the module-level registration guard can skip a fresh active registry, current main requires package metadata for config-promotion setup loading, the new test imports use stale helper paths, and the external PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds Feishu full-registration deduplication, Feishu named-account config-promotion keys/fallbacks, and regression tests for duplicate registration and preserving top-level groupPolicy.

## Best Possible Solution

Revise the PR to key Feishu registration dedupe to the active API/registry, declare Feishu configPromotion metadata through the package/setup contract, rebase tests to current SDK helper subpaths, then merge after focused tests and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main still calls every Feishu hook/tool registration each time registerFull runs, and the legacy normalizer can move top-level groupPolicy into accounts.default for named-account Feishu configs because no Feishu named-account promotion filter is visible on the early path.

## Solution Assessment

No. The PR targets the right symptoms, but the module-level guard is not registry-safe, the current manifest-gated promotion path needs Feishu package metadata, and the new test imports need current SDK helper subpaths.
