## Summary

Keep open. Current main still has a source-backed QQBot availability failure shape: official C2C streaming awaits slow platform stream sends through turn finalization, while diagnostic recovery does not release active embedded work by default.

## What This Changes

Prevent QQBot C2C streaming from freezing a session when slow official stream sends interact with stuck-session recovery.

## Best Possible Solution

Add a QQBot-owned bounded and coalesced official C2C streaming path that always reaches a terminal state, leaving global active-run abort policy to the diagnostics recovery thread.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully live: source inspection gives a credible current-main path through QQBot C2C streaming and non-recoverable active work, but the QQ platform pacing and second embedded run need real credentials to prove end to end.

## Solution Assessment

Is this the best way to solve the issue? Yes for a narrow first repair: bound and coalesce QQBot official C2C stream sends inside the plugin and preserve terminal finalization. A global diagnostic active-abort change is a separate maintainer policy path.
