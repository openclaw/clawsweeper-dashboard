## Summary

Keep open. Current main still has the reported combination: QQBot C2C streaming is routed through awaited `stream_messages` callbacks, while normal stuck-session recovery still logs `active_embedded_run` as observe-only and does not pass the opt-in active-abort flag.

## What This Changes

Reports a QQBot C2C streaming availability bug where slow official `stream_messages` delivery can keep an active embedded run past the stuck-session threshold, leaving later messages queued until gateway restart.

## Best Possible Solution

Add a focused QQBot streaming recovery/backpressure fix so slow official C2C stream sends cannot hold the session lane indefinitely: batch or coalesce stream updates for the QQ API cadence, ensure slow/error finalization reaches a terminal or fallback state, and cover the stuck-session interaction with tests before considering any broader core active-run abort policy.

## Reproduction Assessment

Yes at the code-path level: current source and tests reproduce the reported `active_embedded_run` observe-only recovery skip while QQBot streaming awaits `stream_messages` send work. Full live reproduction of QQ's roughly 50-second send cadence still needs a configured QQBot environment.

## Solution Assessment

Unclear for the issue's broad suggested fixes because it mixes QQBot batching, recovery thresholds, and possible core abort behavior. The narrowest maintainable solution is a QQBot-local streaming backpressure/finalization fix first, with any core active-run abort path kept explicit and covered by diagnostics tests.
