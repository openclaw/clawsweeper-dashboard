## What This Changes

The issue requests recovery for QQBot C2C streaming sessions that freeze when slow streamed sends keep an active embedded run from being recovered.

## Best Possible Solution

Keep the shipped progress-aware stuck-session recovery, and track any remaining QQBot slow-stream batching concern as a separate focused performance issue if it still reproduces after v2026.5.3.

## Reproduction Assessment

Yes at the source-path level: the original report combined QQBot C2C streaming with stuck-session recovery that observed active embedded runs. I did not live-test QQ rate limiting, but current main source and tests show the deadlock path now has progress tracking and active abort recovery.

## Solution Assessment

Yes. The maintainable fix is the shared diagnostic progress and stalled-run recovery path because the freeze was caused by recovery refusing active embedded runs; QQBot-specific batching can be handled separately if slow but recoverable streaming remains painful.
