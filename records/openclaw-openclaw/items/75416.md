## Summary

Keep open. Current main still matches the reported failure shape: diagnostics warn after the default 120s stuck-session window, the heartbeat calls recovery without enabling active-run abort, recovery stays observe-only for active embedded runs, and QQBot official C2C streaming awaits serialized `stream_messages` sends and finalization on the reply path.

## What This Changes

The issue reports that QQBot C2C streaming with `channels.qqbot.streaming: true` can keep a session in processing long enough for stuck-session recovery to skip intervention, leaving later messages queued until the Gateway is restarted.

## Best Possible Solution

Add a QQBot-owned streaming backpressure and terminal-state repair so official C2C stream sends cannot keep the session lane active indefinitely: coalesce or pace stream updates for QQ API limits, make slow/error finalization reach a bounded terminal or fallback state, and only add an explicit tested core recovery opt-in if the plugin cannot solve it locally.

## Reproduction Assessment

Yes at the code-path level: current main still combines the 120s stuck-session warning, observe-only recovery for active embedded runs, and QQBot C2C streaming finalization that awaits official stream-send promises. Full live reproduction of the exact 50s QQ send cadence requires a configured QQBot environment.

## Solution Assessment

Unclear for the issue's broad suggested fixes because they mix QQBot batching, global stuck-session thresholds, and active embedded-run abort policy. The narrowest maintainable solution is a QQBot streaming backpressure/finalization fix first, with a small core opt-in only if the QQBot path needs it.
