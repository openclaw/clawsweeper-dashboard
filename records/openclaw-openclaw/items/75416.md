## Summary

Keep open. Current main still matches the source-backed failure shape: QQBot official C2C streaming awaits serialized `stream_messages` sends and finalization, while diagnostics warn after 120 seconds and active embedded-run recovery remains observe-only by default.

## What This Changes

The issue reports QQBot C2C streaming rate-limit delays interacting with stuck-session recovery so active embedded runs stay observe-only and subsequent messages queue indefinitely.

## Best Possible Solution

Add a QQBot-owned streaming backpressure and bounded terminal-state path so official C2C stream sends cannot keep the session lane active indefinitely, with core recovery opt-in only if the plugin cannot resolve it locally.

## Reproduction Assessment

Yes at the code-path level: current main combines the 120s stuck-session window, observe-only active-run recovery, and QQBot C2C streaming that awaits official stream-send promises. Full live reproduction of the QQ rate-limit cadence needs real QQBot credentials and platform behavior.

## Solution Assessment

Unclear for the broad suggested fixes because they mix QQBot batching, global recovery policy, and second-run investigation. The safest first repair is a QQBot streaming pacing/finalization fix with focused tests, then a narrow core seam only if still required.

The observable problem is that QQBot official C2C streaming can await slow/rate-limited `stream_messages` sends long enough for the 120s stuck-session detector to fire while recovery remains observe-only for an active embedded run, leaving later messages queued.
