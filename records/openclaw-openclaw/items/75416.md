## Summary

Keep open. Current main still has the reported availability failure shape: QQBot official C2C streaming can leave dispatch waiting on the streaming controller while diagnostics recovery observes an active embedded run and intentionally skips intervention.

## What This Changes

The issue reports that QQBot official C2C streaming can interact with stuck-session recovery so slow stream sends keep the session active, queue later messages, and require a gateway restart.

## Best Possible Solution

Add a QQBot-owned streaming backpressure and finalization fix so official C2C stream sends cannot keep the session lane active indefinitely: coalesce or rate-limit stream updates for the QQ API cadence, ensure slow/error finalization reaches a terminal or fallback state, and only add a core recovery seam if the QQBot fix needs an explicit, tested opt-in.

## Reproduction Assessment

Yes at the code-path level: current main still combines a 120s stuck-session warning, observe-only recovery for active embedded runs, and QQBot streaming finalization that waits behind official stream send promises. Full live reproduction of the reported send cadence requires a configured QQBot environment.

## Solution Assessment

Unclear for the issue's broad suggested fixes because it mixes QQBot batching, recovery thresholds, and active-run abort policy. The narrowest maintainable repair is a QQBot streaming backpressure/finalization change first, with diagnostics tests guarding any explicit core recovery hook.
