## Summary

Keep open. The prior ClawSweeper review correctly kept this issue open, and current main still matches the reported failure shape: QQBot official C2C streaming awaits serialized stream sends/finalization while diagnostic recovery still avoids active embedded runs by default.

## What This Changes

The issue requests QQBot streaming and stuck-session recovery changes so slow QQ `stream_messages` delivery cannot permanently freeze a session lane.

## Best Possible Solution

Add a QQBot-owned bounded streaming path that coalesces or times out slow official C2C stream sends and reaches a terminal state without keeping the session lane active indefinitely, adding a generic core recovery opt-in only if plugin-local bounds are insufficient.

## Reproduction Assessment

Yes at the code-path level: current main combines QQBot official C2C streaming, awaited `stream_messages` sends, and no heartbeat recovery path that can break active streaming work. Full live reproduction of the exact QQ rate-limit cadence still needs real QQBot credentials and platform behavior.

## Solution Assessment

Unclear for the broad suggested fixes because they mix QQBot batching, global recovery policy, and second-run investigation. The narrowest maintainable first fix is QQBot streaming pacing and bounded finalization, with core recovery changes only behind a generic seam if still required.

Start with a QQBot-owned repair: official C2C `stream_messages` sends must not keep the session lane active indefinitely when QQ is very slow or a stream request hangs.
