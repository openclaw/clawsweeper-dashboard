## Summary

Keep open. Current main still preserves the reported failure shape: QQBot official C2C streaming is awaited through the reply path, while stuck-session recovery observes active embedded runs by default instead of clearing them.

## What This Changes

QQBot C2C streaming can leave sessions frozen when slow official stream message sends keep an embedded run active and stuck-session recovery skips active runs.

## Best Possible Solution

Bound QQBot C2C streaming sends and finalization so slow or rate-limited platform chunks cannot keep a session active indefinitely; keep global active-run recovery policy unchanged unless a generic opt-in seam is proven necessary.

## Reproduction Assessment

Yes at the code-path level: current main combines QQBot official C2C streaming, awaited stream callbacks/finalization, and recovery that skips active embedded runs by default. Full live proof of the exact QQ rate-limit cadence still needs real QQBot credentials.

## Solution Assessment

Yes for a narrow first repair: fix the QQBot-owned streaming path with bounded sends, coalescing, and terminal cleanup. The broader suggestions to change global stuck-session abort policy or add configurable thresholds should wait unless the plugin-local fix is insufficient.
