## Summary

Keep open. Current main has newer diagnostics that classify active embedded work as long-running or stalled rather than immediately recovering it, but it still has no QQBot-owned bound on official C2C stream sends/finalization and the active-run recovery path remains observe-only unless explicitly opted in. The prior ClawSweeper comment correctly identified the main failure path; the remaining repair should be in QQBot streaming/backpressure first, with only a small generic recovery seam if the plugin cannot safely bound the active run itself.

## What This Changes

The issue reports that QQBot official C2C streaming under QQ platform rate limiting can keep an embedded run active long enough to block later messages in the same session indefinitely.

## Best Possible Solution

Add a QQBot-owned official C2C streaming backpressure and bounded terminal-state path: coalesce or pace partial updates for QQ limits, avoid awaiting an unbounded platform send for every model delta, make finalization reach a fallback/abort state within a tested timeout, and only add a generic core recovery opt-in if the plugin cannot solve the active-run lifetime locally.

## Reproduction Assessment

Yes at the code-path level: configure a C2C QQBot account with streaming enabled and mock sendC2CStreamMessage to be very slow or never resolve while a second message queues in the same session. Full live reproduction of the exact QQ rate limit and event-loop metrics requires a configured QQBot environment.

## Solution Assessment

Unclear for the issue's broad suggested fixes because global active-run abort policy, QQ batching, and stuck-session thresholds are mixed together. The narrowest maintainable repair is a QQBot streaming backpressure/finalization fix first, with a small generic recovery seam only if tests prove the plugin cannot bound the active run by itself.
