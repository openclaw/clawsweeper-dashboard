## Summary

Keep open. Current main still matches the reported availability failure shape: QQBot official C2C streaming is awaited through the outbound turn, and diagnostic stuck-session recovery still observes active embedded runs by default instead of breaking or bounding them.

## What This Changes

The issue reports that QQBot C2C streaming can hold a session active long enough for stuck-session recovery to skip the active embedded run, leaving later messages queued indefinitely.

## Best Possible Solution

Add a QQBot-owned bounded official C2C streaming path that coalesces or times out slow stream sends and always reaches a terminal state, while leaving global active-run abort policy unchanged unless a generic opt-in seam proves necessary.

## Reproduction Assessment

Yes at the code-path level: current main combines QQBot official C2C streaming, awaited stream sends/finalization, and recovery that skips active embedded runs by default. I did not establish live proof of the QQ platform rate-limit cadence in this checkout.

## Solution Assessment

Unclear for the broad suggested fixes because they mix QQBot batching, global recovery policy, and second-run investigation. The narrowest maintainable first fix is plugin-local bounded C2C streaming/finalization, with any core recovery change deferred to the broader stuck-session recovery track.
