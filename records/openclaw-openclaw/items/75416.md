## Summary

Keep open. Current main still has the reported availability failure shape: QQBot official C2C streaming awaits slow platform sends through turn finalization, while diagnostics either treats active work as non-recoverable or observes active embedded runs by default.

## What This Changes

QQBot C2C streaming should not leave sessions permanently stuck when official QQ stream sends are very slow and stuck-session recovery sees the turn as still active.

## Best Possible Solution

Add a QQBot-owned bounded/coalesced official C2C streaming path that always reaches a terminal state, and leave global active-run abort policy to the diagnostics recovery thread.

## Reproduction Assessment

Yes at the code-path level: current main combines QQBot official C2C streaming, awaited stream sends/finalization, and diagnostics that do not recover active work by default. Full live reproduction of the QQ platform pacing still needs real QQBot credentials and platform behavior.

## Solution Assessment

Yes for a narrow first repair: bound and coalesce QQBot C2C stream sends inside the plugin and preserve terminal finalization. A global diagnostic active-abort change is a separate maintainer policy path under PR #76115.
