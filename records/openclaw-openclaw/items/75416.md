## Summary

Keep open. Current main still has the source-backed availability risk described here: QQBot official C2C streaming is enabled by config, stream sends are awaited inside the streaming controller path, and normal stuck-session recovery still treats active embedded runs as observe-only instead of releasing or aborting them.

## What This Changes

The issue reports that slow QQBot official C2C streaming can keep a session in processing long enough for stuck-session recovery to fire, skip recovery because the embedded run is active, and leave later messages queued indefinitely.

## Best Possible Solution

Add a focused QQBot streaming backpressure and finalization fix so slow official C2C stream sends cannot keep the session lane occupied indefinitely. Prefer coalescing/batching or timeout-bounded stream updates in the QQBot plugin first, and only add broader core active-run abort behavior if it is explicit, opt-in, and covered by diagnostics tests.

## Reproduction Assessment

Yes at the code-path level: current main can be traced from QQBot streaming config to awaited `sendC2CStreamMessage` calls while stuck-session recovery remains observe-only for active embedded runs. Full live reproduction still requires a configured QQBot account and QQ's slow stream-send behavior.

## Solution Assessment

Unclear for the broad suggested fixes as written because they mix QQBot batching, diagnostic thresholds, and core abort policy. The narrowest maintainable solution is a QQBot-local streaming backpressure/finalization fix with tests before changing default core recovery semantics.
