## Summary

Keep open. Current main still matches the reported failure shape: QQBot official C2C streaming awaits slow `stream_messages` sends/finalization, while diagnostic recovery still treats active embedded work as non-recoverable by default.

## What This Changes

The issue requests QQBot streaming and stuck-session recovery changes so slow QQ `stream_messages` delivery cannot freeze a session lane indefinitely.

## Best Possible Solution

Add a QQBot-owned bounded streaming path that coalesces or times out slow official C2C stream sends and reaches a terminal state without keeping the session lane active indefinitely; add a generic core recovery opt-in only if plugin-local bounds are insufficient.

## Reproduction Assessment

Yes at the code-path level: current main combines QQBot official C2C streaming, awaited `stream_messages` sends, and no heartbeat recovery path that can break active streaming work. Full live reproduction of the exact QQ rate-limit cadence still needs real QQBot credentials and platform behavior.

## Solution Assessment

Unclear for the broad suggested fixes because they mix QQBot batching, global recovery policy, and second-run investigation. The narrowest maintainable first fix is QQBot streaming pacing and bounded finalization, with core recovery changes only behind a generic seam if still required.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75416. Start with a QQBot-owned repair: official C2C `stream_messages` sends must not keep the session lane active indefinitely when QQ is very slow or a stream request hangs. Work primarily in `extensions/qqbot/src/engine/messaging/streaming-c2c.ts` and `extensions/qqbot/src/engine/gateway/outbound-dispatch.ts`. Add bounded send/finalization behavior around `sendC2CStreamMessage` and pending flush waits, coalesce or pace queued partials so slow platform sends do not serialize unboundedly, and transition to a terminal fallback/abort state with useful logs on timeout. Add regression coverage for slow/hung stream sends and finalization unblocking, update `CHANGELOG.md` as a user-facing fix, and do not change QQ credentials, auth, API routes, or unrelated QQBot media behavior.
