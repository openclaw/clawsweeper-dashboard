## Summary

Keep open. Current main still has the reported availability failure shape: normal stuck-session recovery observes active embedded runs without aborting them, while QQBot official C2C streaming awaits serialized `stream_messages` sends and finalization behind the same controller path.

## What This Changes

The issue reports that slow QQBot C2C streaming can keep an embedded run active past the stuck-session threshold, causing recovery to skip intervention and later messages to queue indefinitely.

## Best Possible Solution

Add a QQBot-owned streaming backpressure and finalization fix so official C2C stream sends cannot keep the session lane active indefinitely: coalesce or rate-limit stream updates for the QQ API cadence, ensure slow/error finalization reaches a terminal or fallback state, and only add a core recovery seam if the QQBot fix needs an explicit, tested opt-in.

## Reproduction Assessment

Yes at the code-path level: current main still combines a 120s stuck-session warning, observe-only recovery for active embedded runs, and QQBot streaming finalization that waits behind official stream send promises. Full live reproduction of the reported send cadence requires a configured QQBot environment.

## Solution Assessment

Unclear for the issue's broad suggested fixes because they mix QQBot batching, recovery thresholds, and active-run abort policy. The narrowest maintainable repair is a QQBot streaming backpressure/finalization change first, with diagnostics tests guarding any explicit core recovery hook.
