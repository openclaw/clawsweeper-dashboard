## Summary

Keep open. The existing ClawSweeper comment is confirmed by current source: normal diagnostic recovery still observes active embedded runs instead of aborting them, and QQBot official C2C streaming still awaits stream-send callbacks, so the reported freeze path is not implemented or superseded on current main.

## What This Changes

The issue requests recovery and streaming changes so slow QQBot official C2C streaming cannot leave an active embedded run and queued session permanently stuck after the stuck-session threshold fires.

## Best Possible Solution

Add a QQBot-focused streaming backpressure or recovery path that prevents slow official C2C `stream_messages` sends from holding the session lane indefinitely, with tests for slow partial sends and stuck-session recovery. Keep any generic active-run abort behavior conservative, explicit, and separately tested.

## Reproduction Assessment

Yes at the code-path level: current main still reproduces the reported observe-only recovery skip for active embedded runs, and QQBot official streaming still awaits partial stream sends. Full end-to-end reproduction of QQ's 50-second send cadence needs a live QQBot environment.

## Solution Assessment

Unclear for a final implementation because the issue mixes QQBot batching, diagnostic recovery, and an unexplained second run. The narrowest maintainable direction is to fix QQBot streaming backpressure/finalization first and avoid making broad core active-run abort behavior the default.
