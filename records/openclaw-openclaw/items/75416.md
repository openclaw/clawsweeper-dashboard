## Summary

Keep open. Current main still has the reported warning-only active embedded run recovery path, and QQBot official C2C streaming still awaits the stream-send callback chain, so the session-freeze path is neither fixed nor fully superseded by the earlier QQBot streaming work or the generic stuck-session abort PR.

## What This Changes

The issue requests recovery and QQBot streaming changes so slow official C2C `stream_messages` sends cannot keep an embedded run active and leave later messages queued indefinitely.

## Best Possible Solution

Add a QQBot-focused streaming backpressure or recovery path that prevents slow official C2C `stream_messages` sends from holding the session lane indefinitely, with tests for slow partial sends and stuck-session recovery. Keep any generic active-run abort behavior conservative, explicit, and coordinated with #71127/#73243 rather than baking QQBot policy into core.

## Reproduction Assessment

Yes at the code-path level: current main still reproduces the reported observe-only recovery skip for active embedded runs, and QQBot official streaming still awaits partial stream sends. Full end-to-end reproduction of QQ's slow send cadence needs a live QQBot environment.

## Solution Assessment

Unclear for the exact implementation because the issue mixes QQBot batching, diagnostic recovery, and an unexplained second run. The narrowest maintainable direction is to fix QQBot streaming backpressure/finalization first and coordinate any broader core abort behavior with the existing stuck-session recovery work.
