## Summary

Keep open. Current main still lacks a bounded QQBot official C2C streaming send/finalization path, and the diagnostics/recovery path still does not break active embedded work by default, so the reported availability failure remains plausible even though newer diagnostics may classify it as active/stalled work rather than the exact old stuck-recovery log.

## What This Changes

The issue asks OpenClaw to prevent QQBot official C2C streaming from freezing a session when slow platform sends interact with stuck-session recovery.

## Best Possible Solution

Add QQBot-owned bounds around official C2C streaming sends and finalization so slow or rate-limited `stream_messages` calls coalesce, time out, or fall back to a terminal non-streaming delivery without leaving the session lane active indefinitely.

## Reproduction Assessment

Yes at the source-path level: enable QQBot C2C streaming and make `sendC2CStreamMessage` hang or take longer than the diagnostics threshold while a follow-up queues. Full live confirmation of the reported 50-second QQ cadence still requires real QQBot credentials and platform behavior.

## Solution Assessment

Unclear for the broad suggested core recovery changes because active-run abort policy is shared and risky. The narrowest maintainable first fix is plugin-local QQBot streaming pacing/bounds, with a generic recovery seam only if that cannot make the turn terminal.
