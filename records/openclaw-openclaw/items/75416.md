## Summary

Keep open. Current main still matches the reported availability failure shape: QQBot official C2C streaming awaits unbounded platform sends and finalization, while diagnostics classify active embedded work as non-recoverable or observe-only instead of freeing the queued session; a later v2026.5.2 user comment reports similar QQBot streaming trouble.

## What This Changes

The issue reports that slow QQBot C2C streaming can keep an embedded run active long enough for stuck-session recovery to skip intervention, leaving later messages queued until restart.

## Best Possible Solution

Add a QQBot-owned bounded C2C streaming path that coalesces or times out slow stream_messages sends, always reaches a terminal delivery state, and clarifies streaming:true versus streaming.mode/c2cStreamApi semantics in docs/tests.

## Reproduction Assessment

Yes at the source-code level: current main enables QQ official C2C streaming for streaming:true, awaits each stream_messages call and final DONE send without a local bound, and active embedded work is not recovery-eligible. I did not live-verify the QQ platform rate limit cadence.

## Solution Assessment

Yes for a narrow QQBot-local repair that bounds/coalesces C2C streaming and guarantees finalization. No for a first-pass global stuck-session abort policy, which should remain out of scope unless the plugin-local fix cannot cover the failure.
