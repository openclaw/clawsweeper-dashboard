## Summary

Keep open. Current main still matches the reported availability failure shape: QQBot official C2C streaming is awaited through the reply path, and diagnostic recovery still observes active embedded runs by default instead of breaking them.

## What This Changes

The issue reports that QQBot C2C streaming can keep a session active past the stuck-session threshold, causing recovery to skip the run and later messages to queue until restart.

## Best Possible Solution

Add a QQBot-owned bounded streaming path that coalesces or times out slow official C2C stream sends and always reaches a terminal state, with any core recovery change kept behind a generic opt-in seam only if needed.

## Reproduction Assessment

Yes at the code-path level: current main combines QQBot official C2C streaming, awaited stream sends/finalization, and recovery that skips active embedded runs by default. Full live proof of the reported 50-second QQ send cadence still needs real QQBot credentials and platform behavior.

## Solution Assessment

Unclear for the broad suggested fixes because they mix QQBot batching, global recovery policy, and second-run investigation. The narrowest maintainable first repair is bounded QQBot C2C streaming and finalization, leaving global active-run abort policy out of scope unless plugin-local bounds are insufficient.
