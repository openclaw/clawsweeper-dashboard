## Summary

Keep open: current main still has a source-level path where an `at` agent-turn cron can persist `runningAtMs` before execution and then wait indefinitely if the isolated-agent setup never reaches the execution-start callback, leaving no run result or useful diagnostics.

## What This Changes

Investigate silent drops for one-shot `at` cron jobs targeting `--session-key session:cody-coder`, where `runningAtMs` is set but no agent session starts.

## Best Possible Solution

Add a scheduler-level pre-execution watchdog and diagnostics path so an agent-turn cron that never reaches execution start records a failed or retryable run, clears `runningAtMs`, and leaves useful run evidence without requiring an external watchdog.

## Reproduction Assessment

Yes, source-level: a due isolated `agentTurn` cron whose `runIsolatedAgentJob` promise never resolves and never calls `onExecutionStarted` will keep the persisted `runningAtMs` marker without applying an outcome. I did not run a live congestion repro, so the exact production race is not fully reproduced.

## Solution Assessment

Yes. The narrowest maintainable fix is to bound and record failures in the cron scheduler handoff before execution starts, rather than relying on two-hour stale-marker cleanup, external sentinel files, or a new operator config knob.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79475. Current main persists `runningAtMs` before isolated agent execution and defers the agent-turn timeout until `onExecutionStarted`; if `runIsolatedAgentJob` hangs before that callback, the cron job can stay running with no `lastRunAtMs`, run log, session file, or diagnostics. Add a focused scheduler recovery path for pre-execution/setup hangs: bound the handoff, record an error diagnostic/run outcome, clear `runningAtMs`, and let existing one-shot retry/disable/delete semantics handle the result. Preserve the existing behavior that long-running executions get their timeout measured after execution starts, and do not add a new config option or change `--session` / `--session-key` semantics unless the regression test proves it is necessary. Add a regression test with an isolated `agentTurn` at-job whose runner never calls `onExecutionStarted`, plus coverage that list/status remain responsive and the persisted job leaves `runningAtMs` with an error or retry state. Add a user-facing changelog fix entry.
