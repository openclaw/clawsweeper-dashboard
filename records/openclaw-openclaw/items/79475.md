## Summary

Keep open: current main still has a plausible pre-start cron hang path where `runningAtMs` is persisted before detached agent execution starts, while the timeout is deferred until `onExecutionStarted`.

## What This Changes

The issue asks for cron reliability diagnostics and recovery for one-shot `at` jobs targeting a session key that enter `runningAtMs` but never spawn or record a run.

## Best Possible Solution

Keep the reservation, add a bounded pre-start timeout and structured diagnostics around the scheduler-to-agent handoff, then record a normal failed run/task and clear `runningAtMs` when startup never begins.

## Reproduction Assessment

No. The reporter cannot reproduce on demand and I did not establish a live current-main failure, but source inspection shows a current-main path that can remain pending before `onExecutionStarted` arms the timeout.

## Solution Assessment

No, setting `runningAtMs` only after spawn completes is not the safest fix because it weakens duplicate-run reservation. The narrower solution is a separate pre-start timeout/diagnostic path that preserves the existing execution-timeout deferral semantics.
