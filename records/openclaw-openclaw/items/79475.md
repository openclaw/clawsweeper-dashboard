## What This Changes

Investigate one-shot `at` cron jobs targeting a custom session key that can remain stuck with `runningAtMs` set and no spawned session or run evidence under gateway load.

## Best Possible Solution

Add a bounded pre-execution/startup timeout and structured diagnostics around the cron scheduler to detached-agent execution-start handoff, then record a normal error/skipped run and clear `runningAtMs` when startup never begins.

## Repair Work Prompt

Fix the at-cron silent-drop class reported in https://github.com/openclaw/openclaw/issues/79475. Current main can persist `runningAtMs` for a non-main `agentTurn` cron before the detached agent runtime calls `onExecutionStarted`; because `executeJobCoreWithTimeout` defers the timeout until that callback, lane/setup congestion can leave a one-shot job stuck with no run completion. Keep `runningAtMs` as the duplicate-start reservation, but add a bounded pre-execution/startup timeout or equivalent guard that records a durable error/skipped run, clears `runningAtMs`/active task state, and emits structured diagnostics/logs when execution never starts. Preserve the existing post-start timeout/cleanup behavior. Add focused regression coverage, especially a non-main `agentTurn` job whose `runIsolatedAgentJob` never calls `onExecutionStarted` and never resolves. Do not add a config option, plugin API, or broad cron store migration. Add a user-facing changelog entry for the reliability fix.
