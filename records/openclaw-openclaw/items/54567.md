## Summary

Keep this PR open, but it should not merge as-is: the reordered linger call still runs after the Linux user-systemd availability skip, and the helper itself returns on the same unavailable-user-bus probe, so the claimed headless-server path remains unfixed.

## What This Changes

The PR moves `ensureSystemdUserLingerNonInteractive` before `service.install()` in the non-interactive gateway daemon installer and adds a comment explaining the headless systemd case.

## Best Possible Solution

Distinguish true systemd absence from a disabled or unreachable user bus, perform any linger/status remediation before the early skip, and cover the path with focused seam tests plus redacted headless Ubuntu proof.

## Reproduction Assessment

Yes, source-level. On Linux, a `Failed to connect to bus` result makes `isSystemdUserServiceAvailable()` false, so the installer exits before the PR's moved call can run, and the helper would return on that same probe anyway.

## Solution Assessment

No. Reordering the helper only before `service.install()` is not sufficient because the reported unavailable-user-bus path is blocked by the earlier availability gate and by the helper's own self-gate.
