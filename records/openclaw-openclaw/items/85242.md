## Summary

This is still actionable on current `main`: the Codex app-server terminal-idle watchdog is still a 30-minute default that is not derived from the effective run timeout, while the fallback timeout text can still point users at `agents.defaults.timeoutSeconds` instead of the terminal-idle limiter.

## What This Changes

The issue asks for the Codex app-server terminal-idle watchdog to respect the effective run timeout or expose the actual limiting timeout, and for diagnostics to stop pointing only at `agents.defaults.timeoutSeconds`.

## Best Possible Solution

Derive the terminal-idle watchdog from the effective Codex app-server run budget, keep the shorter completion/post-tool idle guards separate, and surface terminal-idle-specific user guidance that names the actual watchdog and timeout involved.

## Reproduction Assessment

Yes, at source level: a Codex app-server turn that makes progress and then waits for `turn/completed` longer than the terminal watchdog will abort at the 30-minute constant even when `params.timeoutMs` is higher. I did not live-wait a 30-minute scheduled job in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to make the terminal-idle ceiling follow the effective run timeout and improve the terminal-idle error text, without changing the separate short completion-idle and dynamic-tool watchdogs.
