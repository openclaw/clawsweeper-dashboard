## Summary

Keep open: current main and v2026.6.8 still leave the Codex app-server terminal-idle watchdog independent of the effective run timeout, and terminal-watch timeouts still fall through to generic timeout guidance. The open related PR is a useful partial fix for the early-abort path, but it is not merged and intentionally leaves the diagnostic wording half tracked here.

## What This Changes

The issue asks OpenClaw to make the Codex app-server terminal-idle watchdog follow the effective run timeout or a supported override, and to report terminal-idle-specific guidance when that watchdog fires.

## Best Possible Solution

Refresh or replace the focused runtime fix, then add terminal-idle-specific timeout guidance that names the watchdog/effective budget while avoiding a new public config knob unless maintainers explicitly choose one.

## Reproduction Assessment

Yes, at source level: current main resolves the terminal-idle watchdog from an explicit override only, does not pass the effective run timeout, and skips terminal-specific prompt timeout outcomes. I did not live-wait for a 30-minute scheduled Codex turn.

## Solution Assessment

Yes, the best fix is the narrow runtime-plus-diagnostics path: derive the existing terminal watchdog from the effective run budget and report terminal-idle-specific guidance. A new public config/env surface is not required unless maintainers deliberately choose that policy.
