## Summary

Keep open: current main still has a separate 30-minute Codex app-server terminal-idle watchdog that is not derived from the effective run timeout, and the timeout fallback can still point users at `agents.defaults.timeoutSeconds` rather than the terminal-idle limiter.

## What This Changes

The issue asks OpenClaw to prevent Codex app-server's terminal-idle watchdog from silently overriding longer run-specific timeouts and to improve timeout diagnostics.

## Best Possible Solution

Derive the terminal-idle watchdog from the effective Codex app-server run budget, keep the shorter completion and post-tool idle guards separate, and surface terminal-idle-specific timeout guidance.

## Reproduction Assessment

Yes, at source level: a Codex app-server turn that stays active but emits no terminal event past 30 minutes can hit `turn_terminal_idle_timeout` even when `params.timeoutMs` is higher. I did not live-wait a 30-minute scheduled run in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to make the terminal-idle ceiling follow the existing effective run timeout and improve its error text, without adding a new public setting or changing the shorter completion-idle guards.
