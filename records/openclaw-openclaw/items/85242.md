## Summary

Keep open: current main and v2026.6.8 still keep the Codex app-server terminal-idle watchdog independent of the effective run timeout, and terminal-idle failures still fall through to generic timeout guidance. The open related PR is a good candidate for the runtime-budget half, but it is not merged and intentionally leaves the diagnostic wording tracked here.

## What This Changes

The issue asks OpenClaw to stop letting Codex app-server terminal-idle timeouts silently override longer run-specific timeouts and to report the actual terminal-idle limiter in user guidance.

## Best Possible Solution

Land or refresh the focused runtime-budget fix, then add terminal-idle-specific timeout guidance without adding a new public config or environment knob unless maintainers deliberately choose that policy.

## Reproduction Assessment

Yes at source level: current main resolves the terminal-idle watchdog from an explicit override only, does not pass `params.timeoutMs`, and skips terminal-specific timeout outcomes. I did not live-wait for a 30-minute scheduled Codex turn.

## Solution Assessment

Yes: the narrow maintainable fix is to derive the existing internal watchdog from the effective run budget and report terminal-idle-specific guidance. Adding a new public config or environment knob is not necessary unless maintainers explicitly choose that surface.
