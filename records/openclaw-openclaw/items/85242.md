## What This Changes

The issue requests that the Codex app-server terminal-idle watchdog stop imposing a lower hardcoded timeout than the effective run timeout and report accurate timeout guidance.

## Best Possible Solution

Land or update the linked repair PR so the terminal-idle watchdog follows the effective run budget and the surfaced timeout guidance identifies the terminal-idle watchdog and timeout that fired.

## Reproduction Assessment

Yes. Current main still resolves the terminal-idle watchdog from the explicit override only, so a Codex app-server turn that waits for `turn/completed` beyond 30 minutes can abort at the internal constant even when `params.timeoutMs` is higher.

## Solution Assessment

Yes for the issue direction: deriving the watchdog from the effective run timeout and adding terminal-idle-specific guidance is the narrow owner-local fix. The open PR is a good runtime start, but it needs either diagnostic coverage or a separate tracked follow-up before this issue is closed.
