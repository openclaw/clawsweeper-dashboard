## Summary

Keep open: current main still has memory/session-history consumers that enumerate and flatten session JSONL files directly, and the public context-engine contract still lacks a complete durable-history provider or completeness marker for those consumers.

## What This Changes

The issue asks OpenClaw to stop treating rotated tail-only session JSONL files as complete raw history for memory-core indexing, dreaming ingestion, session markdown export, and the bundled session-logs skill.

## Best Possible Solution

Add a typed session-history provider or completeness-aware read seam, then route memory-core session indexing, dreaming ingestion, session markdown export, and session-logs guidance through that contract when a context engine owns durable transcript history.

## Reproduction Assessment

Yes from source for the OpenClaw side: current main still enumerates and flattens only session JSONL files that would be tail-only after context-engine rotation. I did not live-run lossless-claw, so this is source-reproducible rather than a real setup reproduction.

## Solution Assessment

Yes in direction, but not as a one-off JSONL workaround. The maintainable solution is a public provider/completeness contract used by all affected consumers, coordinated with the active SQLite/session seam work.
