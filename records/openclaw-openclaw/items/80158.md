## Summary

Keep open: current main still has several memory/session-history consumers that enumerate and flatten session JSONL files directly, while the public context-engine contract does not expose a complete durable-history provider they could use. The related SQLite/session-seam issues are open design tracks, not merged fixes, and this issue captures a concrete affected memory-core/session-logs behavior.

## What This Changes

The issue reports that memory-core, dreaming, markdown export, and the bundled session-logs skill treat rotated tail-only session JSONL files as complete history.

## Best Possible Solution

Add a typed session-history provider/read seam that can report completeness, then route memory-core indexing, dreaming ingestion, markdown export, and session-logs guidance through that seam when a context engine owns durable history.

## Reproduction Assessment

Yes from source for the OpenClaw side: current main still enumerates and flattens only the JSONL files that would be tail-only after rotation. I did not live-run lossless-claw, so this is source-reproducible rather than fully reproduced in a real setup.

## Solution Assessment

Yes in direction, but it needs maintainer contract design: a canonical provider/completeness marker is safer than adding more JSONL scraping branches in each consumer.
