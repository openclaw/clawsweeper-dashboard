## Summary

Keep open: the documentation gap is plausible, but this PR currently contradicts current main on both protocol compatibility and trusted-proxy WebSocket scope handling, and the external PR has no real behavior proof yet.

## What This Changes

The PR updates gateway protocol and trusted-proxy auth docs to describe trusted-proxy WebSocket missing-scope behavior and adjusts protocol version examples/constants.

## Best Possible Solution

Revise the docs to preserve v3-to-v4 protocol range compatibility and accurately describe current trusted-proxy WebSocket behavior: browser JavaScript cannot set arbitrary WebSocket headers itself, but the gateway does read proxy-injected upgrade headers and bounds scopes when that header is present.

## Reproduction Assessment

Yes, source-reproducible: current main protocol constants, native client call sites, and WebSocket handler code show the PR's changed statements are false. I did not run a live gateway because this was a read-only review.

## Solution Assessment

No; docs-only is the right surface, but this diff should be rewritten to match current source and tests instead of changing protocol compatibility or claiming WebSocket upgrade headers are unavailable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Keep the documented protocol lower bound at v3:** `docs/gateway/protocol.md:47`
  - body: Current main still exports `MIN_CLIENT_PROTOCOL_VERSION = 3`, native clients send min=3/max=4, and the server accepts any range containing protocol 4. Changing the examples to `minProtocol: 4` and saying the server rejects mismatches tells third-party/native clients to drop supported v3 gateway compatibility.
  - confidence: 0.96
- **[P2] Align scope-clearing docs with trusted-proxy behavior:** `docs/gateway/trusted-proxy-auth.md:67`

This would document the exact troubleshooting path incorrectly.
  - confidence: 0.93
