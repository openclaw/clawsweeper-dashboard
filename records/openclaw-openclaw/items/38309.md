## Summary

Keep open: current main still defines `sandbox.docker.env` as a string-only Docker env map, and SecretRef support would add a new supported secret materialization surface inside sandbox containers rather than fix broken existing behavior.

## What This Changes

The issue requests allowing SecretRef objects as `sandbox.docker.env` values for defaults and per-agent Docker sandbox config while keeping the resolved runtime Docker env map as strings.

## Best Possible Solution

Keep the string-only Docker env contract until secops/product approves this surface; if approved, implement it end-to-end through the existing SecretInput schema, collectors, target registry, docs, and tests so Docker receives only resolved strings.

## Reproduction Assessment

Yes for the unsupported config shape: source inspection on current main shows the schema/type accept only string env values and there is no Docker env SecretInput collector or target-registry entry. I did not run a live Docker container because this is a read-only review and the source path is decisive.

## Solution Assessment

Unclear until maintainers approve the new secret materialization surface. If approved, the narrow maintainable path is to reuse the existing SecretInput runtime, active-surface policy, target registry, docs, and regression tests rather than bespoke Docker-only secret handling.
