## Summary

Keep open: current main and the latest release still accept only string values for `sandbox.docker.env`, and the SecretRef runtime collector/target registry do not cover Docker env paths. The request is source-reproducible and aligned with OpenClaw's security direction, but it needs maintainer product/security review because resolved Docker env values are metadata-visible via Docker inspection.

## What This Changes

Request SecretRef support for values in `sandbox.docker.env` so sandbox Docker environment variables can be resolved from secret references instead of only plaintext strings.

## Best Possible Solution

Add static config-time SecretRef support only after maintainers approve the Docker-env exposure contract, then align schema/types, runtime resolution, secret target registry, docs, and focused config/secrets/sandbox tests.

## Reproduction Assessment

Yes from source inspection: sandbox Docker env is still a string-only config surface, while SecretInput collection and target registry coverage omit `agents.defaults/list` sandbox Docker env paths.

## Solution Assessment

Unclear. The requested direction is plausible and matches existing SecretRef patterns, but the best implementation depends on a maintainer decision about resolving secrets into Docker metadata-visible container env.
