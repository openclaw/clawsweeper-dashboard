## Summary

Keep open: current main and v2026.5.20 still treat `sandbox.docker.env` as string-only, and enabling SecretRef here needs maintainer security/product approval because resolved values become Docker container environment metadata.

## What This Changes

Requests allowing SecretRef objects as values in `sandbox.docker.env` while keeping resolved sandbox Docker runtime env as `Record<string,string>`.

## Best Possible Solution

After security/product approval, extend the existing SecretInput schema, collector, target registry, docs, and sandbox resolution path while documenting Docker env metadata exposure and preserving resolved runtime `Record<string,string>` output.

## Reproduction Assessment

Yes, by source inspection: the schema and TypeScript contract only accept string env values, runtime config merges env maps directly, and the SecretRef collectors/registry do not include Docker env paths.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable path is to reuse the existing SecretInput collector/registry/runtime snapshot machinery instead of adding Docker-only secret handling.
