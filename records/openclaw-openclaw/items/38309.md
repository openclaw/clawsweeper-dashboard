## Summary

Keep open: current main still rejects object-shaped SecretRefs in `sandbox.docker.env`, and adding it would expand a security-sensitive secret materialization surface into Docker container environment metadata.

## What This Changes

Allow `sandbox.docker.env` values to use SecretRef objects while resolving the runtime Docker env map to plain strings.

## Best Possible Solution

After security/product approval, extend the existing SecretInput schema, collector, registry, audit, docs, and focused tests while keeping the resolved runtime Docker env type as `Record<string, string>`.

## Reproduction Assessment

Yes. Source inspection shows current main accepts only string Docker env values and has no Docker env SecretRef collector or registry target; I did not run a live sandbox because the object form is rejected before launch.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable path is to reuse the existing SecretInput/collector/target-registry machinery rather than adding Docker-only secret handling.
