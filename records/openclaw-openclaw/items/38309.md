## Summary

Keep open: current main and the latest release still treat `sandbox.docker.env` as plain string env values, while the exact linked implementation PR is closed unmerged. The request is valid but needs maintainer product/security review because resolving SecretRefs into Docker env would still expose resolved values through Docker container metadata.

## What This Changes

Requested support for SecretRef objects in `sandbox.docker.env` so sandbox Docker environment values can reference managed secrets instead of only plain strings.

## Best Possible Solution

Approve a clear sandbox Docker secret-delivery contract, then either add this as an explicit SecretRef surface with metadata-exposure docs and tests or choose a safer mounted-file/per-exec delivery path.

## Reproduction Assessment

Yes, from source: current main and v2026.6.6 both validate `sandbox.docker.env` values as strings and omit Docker env from the canonical SecretRef target list, so a SecretRef object is not a supported config value.

## Solution Assessment

Unclear as an implementation path: the requested direction is coherent, but maintainers need to decide whether Docker env is acceptable secret delivery despite Docker metadata exposure or whether another secret delivery mechanism should be preferred.
