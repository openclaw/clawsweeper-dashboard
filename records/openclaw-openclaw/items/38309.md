The request is valid, but it adds a new secret materialization path into Docker sandbox environment variables and needs explicit maintainer/secops approval before implementation.

## What This Changes

Request support for SecretRef objects as values in `sandbox.docker.env`.

## Best Possible Solution

Approve the Docker-env secret surface first, then implement end-to-end SecretInput support using existing collector and target-registry patterns with tests proving only resolved strings reach Docker create args.

## Reproduction Assessment

Yes, source-level. A SecretRef object under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` still hits the current string-only schema, and no collector resolves that path before Docker launch.

## Solution Assessment

Unclear until maintainers approve this secret materialization surface. If approved, the narrow maintainable path is to reuse existing SecretInput resolution, active-surface policy, and target registry rather than adding bespoke Docker secret handling.
