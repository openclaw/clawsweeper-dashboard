## Best Possible Solution

After owner/security approval, implement SecretInput support end-to-end through schema, collectors, target registry, docs, and tests while keeping the resolved Docker runtime env as `Record<string, string>`.

## Solution Assessment

Unclear until maintainers approve the new secret materialization surface. If approved, the narrow maintainable path is to reuse the existing SecretInput activation, active-surface policy, target registry, docs, and regression tests rather than adding Docker-only secret handling.
