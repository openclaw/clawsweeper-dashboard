## Summary

Keep open: current main and v2026.6.1 still model `sandbox.docker.env` as plain string values, and the linked implementation PR is closed unmerged. The request is source-reproducible, but it needs maintainer product/security approval because resolved SecretRefs would still be injected into Docker container environment metadata.

## What This Changes

Allow SecretRef object values in default and per-agent `sandbox.docker.env` so sandbox containers can receive secrets without plaintext config values.

## Best Possible Solution

Add maintainer-approved SecretInput support for Docker env that resolves only active sandbox configs before container creation, updates secret target/audit/configure coverage, and documents the Docker metadata exposure clearly.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a SecretRef object under `sandbox.docker.env` hits a schema that accepts only string values. I did not run a live config validation because this was a read-only review.

## Solution Assessment

Unclear until maintainers decide the security policy. The requested direction is plausible, but the best implementation must reuse the existing secrets runtime and preserve Docker env as resolved strings only after explicit approval of the metadata exposure.
