## Summary

Keep open: current main still has string-only Docker sandbox env config and no Docker-env SecretRef collector or target-registry surface. The closed linked PR is useful context, but it was not merged and this remains a security-sensitive feature request that needs maintainer/secops approval before implementation.

## What This Changes

Support SecretRef objects as values in `sandbox.docker.env` for default and per-agent Docker sandbox configuration.

## Best Possible Solution

Approve the Docker-env SecretRef surface first, then implement it end to end through the existing SecretInput schema, collector, target-registry, and resolved-runtime snapshot patterns.

## Reproduction Assessment

Yes, source-level. A SecretRef object under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` still hits the current string-only schema, and no collector resolves that path before Docker launch.

## Solution Assessment

Unclear until maintainers approve this secret materialization surface. If approved, the narrow maintainable solution is to reuse the existing SecretInput resolver, active-surface policy, and target registry instead of adding bespoke Docker secret handling.
