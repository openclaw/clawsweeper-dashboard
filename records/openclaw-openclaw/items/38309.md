## Summary

Keep open: current main still rejects SecretRef objects in sandbox Docker env and lacks the collector/registry paths needed to resolve them. The linked implementation PR was closed without merge, so the feature remains unresolved and needs maintainer/security approval before implementation.

## What This Changes

Support SecretRef objects in `sandbox.docker.env` for global and per-agent sandbox Docker configuration while preserving plain string env values.

## Best Possible Solution

After owner/security approval, implement SecretInput support end-to-end through schema, public types, collectors, target registry, docs, and focused tests while keeping resolved Docker runtime env as `Record<string, string>`.

## Reproduction Assessment

Yes for the unsupported config shape: current main’s schema and public type accept only string env values, and the SecretRef collector/target-registry paths do not include Docker env. I did not run a live Docker container because source inspection proves object-shaped SecretRefs are not accepted.

## Solution Assessment

Unclear until maintainers approve this new secret materialization surface. If approved, the narrow maintainable path is to reuse the existing SecretInput collector and registry machinery rather than adding Docker-only resolution logic.
