## Summary

Keep open: current main and the latest release still only accept string-valued Docker sandbox env entries, and SecretRef support here needs maintainer/security approval because it would materialize resolved secrets into Docker container environment metadata.

## What This Changes

The issue requests allowing `sandbox.docker.env` values to use SecretRef objects while preserving resolved Docker runtime env as `Record<string, string>`.

## Best Possible Solution

After security approval, implement SecretInput support end to end through schema, public config types, collectors, target registry, docs, and focused tests while keeping resolved Docker runtime env as strings.

## Reproduction Assessment

Yes, source-reproducible: current main’s schema and public type accept only string Docker env values, and the SecretRef collector/registry paths do not include Docker env. I did not run a live container because object-shaped SecretRefs are rejected before sandbox launch.

## Solution Assessment

Unclear until maintainers approve this new secret materialization surface. If approved, the maintainable path is to reuse the existing SecretInput schema, runtime collectors, target registry, audit, and docs machinery rather than adding Docker-only resolution logic.
