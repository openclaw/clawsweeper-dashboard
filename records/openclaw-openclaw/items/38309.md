## Summary

Keep this open. Current main still treats `sandbox.docker.env` as a plain string map: schema validation, generated schema, public config types, Docker config resolution, secret collectors, and the target registry do not accept or resolve SecretRef values for this surface, and the paired implementation PR was closed unmerged.

## What This Changes

Allow `SecretRef` objects as values in `agents.defaults.sandbox.docker.env` and `agents.list[].sandbox.docker.env` while preserving resolved Docker env values as strings.

## Best Possible Solution

Implement an end-to-end SecretInput addition for Docker sandbox env: widen config and generated schemas, type unresolved settings separately from resolved Docker runtime config, collect defaults and per-agent env entries through the secrets runtime and target registry, resolve before Docker create/backend handoff, preserve plain strings, document the supported surface, and add focused schema, runtime, registry, merge, and create-args tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add a structured `{source, provider, id}` SecretRef under `agents.defaults.sandbox.docker.env`; the current Zod and generated schemas require string values, so validation rejects it before Docker launch.

## Solution Assessment

Unclear because there is no current patch to review. The requested direction is plausible, but the maintainable fix should use the existing SecretInput resolver, active-surface policy, and target registry rather than adding Docker-only ad hoc resolution.
