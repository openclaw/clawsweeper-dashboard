## Summary

Keep open: current main still treats Docker sandbox env values as plain strings, and the linked implementation PR is closed unmerged. The request is valid, but it adds a new secret materialization path into Docker container environment variables and needs maintainer/secops approval before implementation.

## What This Changes

Allow `sandbox.docker.env` values to use `SecretRef` objects while resolving them to strings before Docker sandbox launch.

## Best Possible Solution

Approve the Docker-env secret surface first, then implement end-to-end SecretInput support using existing collector and target-registry patterns with tests proving only resolved strings reach Docker args.

## Reproduction Assessment

Yes, source-level. `SandboxDockerSchema.env` and `SandboxDockerSettings.env` are string-only, and the core SecretInput collectors/registry do not include Docker env paths on current main.

## Solution Assessment

Unclear until maintainers approve this secret materialization surface. If approved, the narrow maintainable solution is to reuse the existing SecretInput resolution, active-surface policy, and target registry rather than adding bespoke Docker secret handling.
