## Summary

Keep open: current main still rejects SecretRef objects in `sandbox.docker.env`, and the remaining work needs maintainer security/product approval because resolved values would become Docker container environment metadata.

## What This Changes

Allow `sandbox.docker.env` values to use SecretRef objects while keeping existing plain string env values working.

## Best Possible Solution

After security/product approval, extend the existing SecretInput schema, runtime collector, target registry, docs, and focused tests while keeping the resolved Docker runtime env type as `Record<string, string>` and documenting Docker metadata exposure.

## Reproduction Assessment

Yes for the current unsupported behavior. Source inspection shows object-shaped SecretRefs are rejected by `SandboxDockerSchema`, the public Docker env type is string-only, and no Docker env SecretRef collector exists on current main.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable path is to reuse the existing SecretInput collector and registry machinery rather than adding Docker-only secret handling.
