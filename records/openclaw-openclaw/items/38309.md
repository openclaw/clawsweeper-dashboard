## Summary

Keep open: current main still treats Docker sandbox env values as plain strings only, and the related implementation PR #38310 is closed unmerged. The request is valid, but it expands the supported SecretRef credential surface into Docker container environment variables and should get maintainer/secops approval before implementation.

## What This Changes

The issue asks OpenClaw to accept SecretRef values in defaults and per-agent `sandbox.docker.env`, resolve them before Docker sandbox launch, and include those paths in secrets tooling metadata.

## Best Possible Solution

After approval, add end-to-end SecretInput support for Docker sandbox env through schema/types, runtime collectors, target registry, generated docs, and focused tests while preserving plain string compatibility and avoiding resolved-secret persistence.

## Reproduction Assessment

Yes, source-level. A structured `{source, provider, id}` value under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` hits the current string-only schema, and there is no Docker env SecretRef collector before launch.

## Solution Assessment

Unclear until maintainers approve the surface. If approved, the narrow maintainable fix is to reuse the existing SecretInput resolver and target registry rather than adding Docker-specific secret handling.
