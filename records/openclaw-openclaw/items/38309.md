## Summary

Keep this issue open: current main and the latest release still keep `sandbox.docker.env` as a plain string map, while the exact implementation PR was closed unmerged. The request is valid but needs maintainer product/security approval before repair because it expands secret-handling config and injects resolved values into Docker environment metadata.

## What This Changes

The issue requests allowing SecretRef objects as values in `sandbox.docker.env` while keeping resolved Docker runtime env values as strings.

## Best Possible Solution

Have maintainers approve the security/product direction, then implement the narrow core config change through the existing SecretInput schema, secrets collector, target registry, docs, and tests while keeping Docker launch input as a resolved string-only env map.

## Reproduction Assessment

Yes. Source inspection on current main shows the Docker sandbox env type and runtime schema still require string values, and the secrets registry/collector has no `sandbox.docker.env` target.

## Solution Assessment

Yes as a technical direction, with maintainer approval. The narrow maintainable path is to reuse the existing SecretInput/SecretRef resolution pipeline and preserve a resolved `Record<string, string>` before Docker launch, while explicitly accepting the Docker metadata exposure tradeoff.
