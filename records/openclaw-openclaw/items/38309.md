## Summary

Keep open: current main still does not accept SecretRef objects in `sandbox.docker.env`, and the remaining work is a security-sensitive feature/config surface rather than a cleanup close candidate.

## What This Changes

Allow `sandbox.docker.env` values to use SecretRef objects while keeping existing plain string env values working.

## Best Possible Solution

After owner/security approval, implement SecretInput support end-to-end through schema, public types, collectors, target registry, docs, and focused tests while keeping resolved Docker runtime env as plain strings.

## Reproduction Assessment

Yes for the unsupported config shape: current main’s schema and public type accept only string env values, and the SecretRef collector/target-registry paths do not include Docker env. I did not run a live Docker container because source validation proves object-shaped SecretRefs are not accepted.

## Solution Assessment

Unclear until maintainers approve this new secret materialization surface. If approved, the narrow maintainable path is to reuse the existing SecretInput activation and registry machinery rather than adding Docker-only resolution logic.
