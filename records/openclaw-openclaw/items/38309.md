## Summary

Keep open: current main still accepts only plain string values for Docker sandbox env, and the requested SecretRef support would add a new secret materialization surface inside Docker containers that needs maintainer/secops approval before implementation.

## What This Changes

This issue requests allowing SecretRef objects in `sandbox.docker.env` for global and per-agent Docker sandbox configuration.

## Best Possible Solution

Keep the string-only Docker env contract until secops/product approves this surface; if approved, implement it end-to-end through existing SecretInput schema, collectors, target registry, docs, and tests so Docker receives only resolved strings.

## Reproduction Assessment

Yes, source-level: current main has a string-only schema/type for `sandbox.docker.env`, no Docker env SecretInput collector, and no target-registry entry for Docker env paths. I did not run a live Docker reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers approve the new secret materialization surface. If approved, the narrow maintainable path is to reuse the existing SecretInput runtime, active-surface policy, and target-registry patterns rather than bespoke Docker secret handling.
