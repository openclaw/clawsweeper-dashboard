## Summary

Keep open: current main still only accepts string values for `sandbox.docker.env` and has no Docker-env SecretRef collector or target-registry coverage. The request is valid, but adding a new place where secrets materialize into Docker container environment variables needs maintainer/secops approval first.

## What This Changes

Allow `agents.defaults.sandbox.docker.env` and `agents.list[].sandbox.docker.env` values to accept SecretRef objects and resolve them to strings before Docker launch.

## Best Possible Solution

Approve the Docker env secret surface first, then implement end-to-end `SecretInput` support in schema, types, collectors, target registry, docs, and tests while keeping resolved values only in the runtime snapshot and Docker create args.

## Reproduction Assessment

Yes, source-level. A SecretRef object under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` still hits the string-only schema, and no current collector resolves that path before Docker launch.

## Solution Assessment

Unclear until maintainers approve this new secret materialization surface. If approved, the narrow maintainable fix is to reuse the existing SecretInput collector, active-surface diagnostics, target registry, and runtime snapshot patterns instead of adding bespoke Docker-only secret handling.
