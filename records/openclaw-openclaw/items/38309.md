## Summary

Keep open: current main still treats `sandbox.docker.env` as string-only and does not include Docker env in the SecretRef collector or target registry. The linked contributor PR at https://github.com/openclaw/openclaw/pull/38310 is useful context but was closed unmerged, and the remaining change needs maintainer security/product approval because resolved secrets would be exposed as Docker container environment metadata.

## What This Changes

Allow `sandbox.docker.env` entries to accept SecretRef objects while preserving plain string env values.

## Best Possible Solution

After security/product approval, extend the existing SecretInput schema, runtime collector, target registry, docs, and focused tests while keeping the resolved Docker runtime env type as `Record<string, string>`.

## Reproduction Assessment

Yes for the unsupported config shape. Source inspection shows object-shaped SecretRefs are rejected by `SandboxDockerSchema` before launch and no Docker env SecretRef collector or registry target exists on current main.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable approach is to reuse the existing SecretInput collector and target-registry machinery rather than adding Docker-only secret handling.
