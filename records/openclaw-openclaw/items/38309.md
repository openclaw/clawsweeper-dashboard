## Summary

Keep open: current main and the latest release still treat `sandbox.docker.env` as string-only, and the focused implementation PR was closed unmerged. The request is source-reproducible and well-scoped, but it needs explicit maintainer security/product approval because it would materialize SecretRefs into Docker container environment metadata.

## What This Changes

Support SecretRef values in `sandbox.docker.env` while keeping resolved Docker runtime env values as plain strings.

## Best Possible Solution

After security/product approval, extend the existing SecretInput schema, collector, target registry, docs, and sandbox resolution path while preserving `SandboxDockerConfig.env` as `Record<string, string>` and documenting Docker metadata exposure.

## Reproduction Assessment

Yes for the missing capability: current source shows the schema and TypeScript contract only accept string env values, and the runtime secret collector has no Docker env assignment path. I did not run tests because this cleanup review is read-only and the source path is unambiguous.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable fix is to reuse the existing SecretInput collector, target registry, and runtime snapshot machinery rather than adding Docker-only secret handling.
