## Summary

Keep open: current main still rejects object-shaped SecretRefs in `sandbox.docker.env`, does not collect or resolve that path, and does not list it in the SecretRef target surface. The related implementation PR at https://github.com/openclaw/openclaw/pull/38310 was closed without merge; the later explicit Docker env fix at https://github.com/openclaw/openclaw/pull/82763 only fixed plain configured env keys, not SecretRef support.

## What This Changes

The issue requests SecretRef support for `sandbox.docker.env` so Docker sandbox environment values can reference external secrets instead of only plain strings.

## Best Possible Solution

After owner/security approval, implement SecretInput support end to end through schema, public types, collectors, target registry, docs, and focused tests while keeping resolved Docker runtime env as `Record<string, string>`.

## Reproduction Assessment

Yes, source-reproducible: current main’s schema and public type accept only string Docker env values, and the SecretRef collector/registry paths do not include Docker env. I did not run a live Docker container because object-shaped SecretRefs are rejected before runtime use.

## Solution Assessment

Unclear until maintainers approve this new secret materialization surface. If approved, the maintainable path is to reuse the existing SecretInput schema, collector, registry, and audit machinery rather than adding Docker-only resolution logic.
