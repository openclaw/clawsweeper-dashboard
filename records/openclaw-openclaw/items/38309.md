## Summary

Keep open: current main still does not accept or resolve SecretRef objects in `sandbox.docker.env`, and the request remains a security-sensitive feature decision because approved support would materialize secrets as Docker container environment variables.

## What This Changes

The issue requests SecretRef support for `sandbox.docker.env` values while preserving plain-string environment variables and resolving secrets before Docker sandbox launch.

## Best Possible Solution

After maintainer security/product approval, extend the SecretInput schema, runtime collector, target registry, docs, and focused tests while keeping the resolved Docker runtime env type as `Record<string, string>` and documenting Docker metadata exposure.

## Reproduction Assessment

Yes, for the current unsupported behavior by source inspection: the schema and public settings type accept only string Docker env values, and the runtime merge path has no Docker env SecretRef resolver.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable path is to reuse the existing SecretInput collector and target-registry machinery rather than adding a Docker-only secret path.
