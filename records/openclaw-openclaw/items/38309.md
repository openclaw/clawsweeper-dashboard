## Summary

Keep open: current main and v2026.6.5 still reject SecretRef objects in `sandbox.docker.env`, the linked implementation PR closed unmerged, and no canonical replacement owns the work. The feature fits the security roadmap, but maintainers must approve resolving secrets into Docker-visible environment metadata and define active-sandbox resolution semantics.

## What This Changes

Allow default and per-agent Docker sandbox environment entries to accept SecretRef values that resolve to strings before container creation.

## Best Possible Solution

Approve and document the Docker secret-exposure contract, then add SecretInput support through the canonical secrets snapshot, resolve only the applicable sandbox configuration, retain a string-only normalized runtime type, and cover audit, plan, configure, schema, and live Docker behavior.

## Reproduction Assessment

Yes at source level. Current main's Zod schema requires string values for `sandbox.docker.env`, so a SecretRef object is predictably rejected; this read-only review did not run a live configuration or Docker scenario.

## Solution Assessment

The proposed input-versus-normalized type split is directionally correct, but the best implementation must use canonical snapshot resolution, avoid unnecessarily resolving inactive-agent secrets, and preserve the explicit Docker metadata warning.

## Security Review

Status: needs_attention

Summary: The feature is security-sensitive because externally stored secrets would still become plaintext Docker environment values visible to sandbox workloads and daemon operators.
