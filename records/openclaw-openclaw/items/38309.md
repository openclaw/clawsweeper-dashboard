## Summary

Keep open: current main still treats Docker sandbox env values as plain strings, and no runtime collector or target-registry entry resolves `SecretRef` values for this path. The request is clear and useful, but it adds a new secret materialization surface inside Docker sandbox containers, so it needs maintainer/secops approval before implementation.

## What This Changes

Allow `SecretRef` objects as values in `sandbox.docker.env` while keeping the resolved Docker runtime config as `Record<string, string>`.

## Best Possible Solution

Keep the current string-only Docker env contract until secops/product approves this SecretRef surface; if approved, implement it end-to-end through the existing SecretInput schema, collectors, target registry, and tests so Docker receives only resolved strings.

## Reproduction Assessment

Yes, source-level. Current main still has a string-only schema/type for `sandbox.docker.env`, no Docker env SecretInput collector, and no target-registry entry for Docker env paths.

## Solution Assessment

Unclear until maintainers approve the new secret materialization surface. If approved, the narrow maintainable path is to reuse the existing SecretInput runtime, active-surface policy, and target-registry patterns rather than bespoke Docker secret handling.
