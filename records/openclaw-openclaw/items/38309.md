## Summary

Keep open: current main still rejects SecretRef objects in `sandbox.docker.env`, and the remaining work needs maintainer security/product approval because resolved values would become Docker container environment metadata.

## What This Changes

The issue requests allowing SecretRef objects in `sandbox.docker.env` while preserving existing plain string values and resolving the final Docker runtime env to strings.

## Best Possible Solution

After maintainer security/product approval, extend the existing SecretInput schema, collector/target registry, sandbox resolution path, docs, and focused tests while documenting Docker env metadata exposure.

## Reproduction Assessment

Yes for the current unsupported behavior by source inspection: the schema only accepts string env values, runtime config merges env maps directly, and the SecretRef collectors do not walk Docker env paths.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable path is to reuse the existing SecretInput collector and registry machinery rather than adding Docker-only secret handling.
