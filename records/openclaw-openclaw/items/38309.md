## Summary

Keep open: current main still treats `sandbox.docker.env` as string-only and has no Docker-env SecretRef collector or registry entries. The request is valid, but it adds a new secret materialization path into Docker sandbox environment variables and needs maintainer/secops approval before implementation.

## What This Changes

Allow SecretRef objects in default and per-agent `sandbox.docker.env`, resolving them to strings before Docker sandbox launch and exposing them through secrets audit/plan/configure.

## Best Possible Solution

Approve the secret surface first, then implement end-to-end SecretInput support using the existing collector and target-registry patterns with tests proving resolved strings reach Docker create args and unresolved refs are not persisted.

## Reproduction Assessment

Yes, source-level. A SecretRef object under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` still hits the current string-only schema, and no collector resolves that path before Docker launch.

## Solution Assessment

Unclear until maintainers approve this secret materialization surface. If approved, the narrow maintainable path is to reuse existing SecretInput resolution, active-surface policy, and target registry rather than adding bespoke Docker secret handling.
