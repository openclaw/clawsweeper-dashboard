## Summary

Keep open: current main and v2026.6.5 still model and validate `sandbox.docker.env` as plain strings, the linked implementation PR was auto-closed unmerged, and no viable canonical replacement owns the remaining work.

## What This Changes

Support `SecretRef` objects as values in `sandbox.docker.env` while preserving existing plain string Docker environment entries.

## Best Possible Solution

Land a maintainer-approved SecretRef design for Docker sandbox env that resolves only through the existing secrets pipeline, keeps resolved Docker config as strings, updates audit/plan/configure/docs, and preserves the Docker metadata warning.

## Reproduction Assessment

Yes, at source level: the current type and Zod schema accept only string values for `sandbox.docker.env`, so an object-shaped SecretRef would be rejected before sandbox launch. I did not run a live parser because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the security model. Widening the schema to `SecretInput` is plausible, but the best fix must explicitly preserve Docker metadata warnings and define default/per-agent resolution semantics.
