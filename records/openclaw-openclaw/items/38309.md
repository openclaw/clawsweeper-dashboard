## Summary

Keep open: current main still accepts only string values for `sandbox.docker.env` and has no Docker-env SecretRef collector or target-registry entries. This is a valid feature request, but it expands secret materialization into Docker container environment variables and needs maintainer/secops approval before implementation.

## What This Changes

Allow `sandbox.docker.env` under agent defaults and per-agent sandbox config to accept `SecretInput` values, resolve them before Docker launch, and include them in secrets audit/plan/configure surfaces.

## Best Possible Solution

Approve the secret surface first, then implement end-to-end SecretInput support using the existing collector and target-registry patterns, resolving to strings before Docker create and keeping resolved values out of persisted config.

## Reproduction Assessment

Yes, source-level. A SecretRef object under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` still hits the current string-only schema, and no collector resolves that path before Docker launch.

## Solution Assessment

Unclear until maintainers approve this secret materialization surface. If approved, the narrow maintainable solution is to reuse the existing SecretInput resolver, active-surface policy, and target registry instead of adding bespoke Docker secret handling.
