## Summary

Keep open: current main still treats `sandbox.docker.env` as string-only and has no Docker-env SecretRef collector or target-registry coverage. The request is valid, but it adds a new secret materialization path into Docker sandbox environment variables and needs maintainer/secops approval before implementation.

## What This Changes

Support SecretRef objects as values in `agents.defaults.sandbox.docker.env` and `agents.list[].sandbox.docker.env` while keeping resolved Docker env values as strings.

## Best Possible Solution

Approve the secret surface first, then implement end-to-end SecretInput support for default and per-agent Docker env using the existing collector, target-registry, docs, and test patterns while ensuring only resolved strings reach Docker create args.

## Reproduction Assessment

Yes, source-level. A SecretRef object under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` still hits the current string-only schema, and no collector resolves that path before Docker launch.

## Solution Assessment

Unclear until maintainers approve this secret materialization surface. If approved, the narrow maintainable path is to reuse existing SecretInput resolution, active-surface policy, and target-registry patterns rather than adding bespoke Docker secret handling.
