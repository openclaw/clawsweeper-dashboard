## Summary

Keep open: current main and the latest release still keep `sandbox.docker.env` string-only, and the linked implementation PR is closed unmerged. The request is valid but security/product-sensitive because resolved SecretRefs would still be injected as Docker env vars, which are visible through Docker metadata to daemon-level observers.

## What This Changes

The issue requests support for SecretRef objects in sandbox Docker environment configuration values.

## Best Possible Solution

Approve a clear sandbox Docker secret-delivery contract, then add this as an explicit SecretRef surface with metadata-exposure docs and schema/collector/registry/tests, or keep Docker env string-only and steer secrets to a safer mounted-file/custom-image path.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the gap: current main validates `sandbox.docker.env` as string-only and has no Docker env SecretRef collector or target-registry entry.

## Solution Assessment

Unclear as a product direction. The requested implementation is a plausible additive SecretInput surface, but maintainers should first decide whether Docker env metadata exposure is acceptable or whether mounted files/custom images should remain the recommended secret path.
