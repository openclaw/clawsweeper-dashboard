## Summary

Keep open: current main still rejects SecretRef values on `sandbox.docker.env`, lacks a runtime collector/registry entry for that path, and the related implementation PR #38310 is closed unmerged. The request is valid, but it adds a security-sensitive Docker container environment secret surface that should get maintainer/secops approval before implementation.

## What This Changes

Support SecretRef objects in global and per-agent Docker sandbox environment maps while resolving them to plain string environment values before container launch.

## Best Possible Solution

Add an approved end-to-end SecretInput surface for Docker sandbox env through the existing schema, resolver, target registry, generated docs, and focused tests; keep plain strings compatible and avoid persisting resolved secrets to config, logs, or generated artifacts.

## Reproduction Assessment

Yes, source-level. A structured `{source, provider, id}` value under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` hits the current string-only Zod/generated schema, and the resolved sandbox config has no SecretRef materialization path for Docker env.

## Solution Assessment

Unclear because there is no active patch to review. The requested direction is plausible, but the narrow maintainable solution should use the existing SecretInput resolver, target registry, generated schema/docs pipeline, and sandbox/config regression tests after secops approval.
