## Summary

Keep open: current main and the latest release still keep `sandbox.docker.env` string-only, and there is no Docker-env SecretRef collector or target-registry entry. The remaining question is a maintainer security/product decision about extending SecretRef support to Docker env while preserving the warning that resolved values remain visible in Docker metadata.

## What This Changes

Request support for SecretRef objects in default and per-agent `sandbox.docker.env` while preserving existing plain string env values and resolving Docker runtime env to strings.

## Best Possible Solution

Adopt a maintainer-approved implementation that adds Docker env SecretInput support in schema/types, resolves defaults and per-agent entries before Docker config use, updates the target registry/docs/tests, and keeps the resolved runtime Docker env string-only with the Docker metadata warning intact.

## Reproduction Assessment

Yes at source level: current types and Zod schema require string values for `sandbox.docker.env`, and the SecretRef runtime collectors/registry do not cover Docker env paths. I did not run a live Docker repro during this read-only review.

## Solution Assessment

Unclear until maintainers accept the product/security direction. The proposed shape is plausible if it preserves string-only resolved Docker runtime config, active-surface SecretRef resolution, target-registry coverage, tests, and the Docker metadata exposure warning.
