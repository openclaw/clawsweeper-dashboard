## Summary

Current main still rejects the requested SecretRef shape because `sandbox.docker.env` is string-only, and no merged or viable open PR owns the work; the feature fits OpenClaw’s security direction but needs explicit maintainer security/product approval before a fresh focused implementation.

## What This Changes

Allow `SecretRef` values in default and per-agent sandbox Docker environment maps, resolving them to strings before container launch.

## Best Possible Solution

Approve a fresh focused core PR that accepts `SecretInput` only in raw Docker sandbox settings, collects and resolves default and per-agent entries in the canonical secrets snapshot, preserves string-only effective runtime types, updates audit/plan/configure targets, documents Docker metadata exposure, and proves a real container launch.

## Reproduction Assessment

Yes, at high-confidence source level: current main declares the exact env map as string-only, so the example SecretRef object is unsupported. A live Docker run is still needed to validate a future fix, not to establish the current gap.

## Solution Assessment

Yes in direction, but not via the closed broad branch: the narrowest maintainable solution is to extend the existing core SecretInput collector/resolved-snapshot pattern while keeping sandbox launch string-only.
