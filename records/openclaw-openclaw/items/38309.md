## Summary

Keep open: current main and the latest release still treat `sandbox.docker.env` as a plain string map, and the supported SecretRef surface does not include Docker sandbox env values. The request is plausible and source-reproducible, but it needs maintainer product/security review because resolved values would still enter Docker environment metadata.

## What This Changes

Allow `SecretRef` objects as values in `agents.defaults.sandbox.docker.env` and `agents.list[].sandbox.docker.env`, resolving them before Docker sandbox use while preserving plain string env values.

## Best Possible Solution

Decide the Docker metadata exposure contract first, then add a narrow SecretInput path for Docker sandbox env with schema, runtime collector, target registry, docs, and tests that keep resolved runtime env string-only.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main and `v2026.6.6` still accept only string values for `sandbox.docker.env` and do not collect Docker env SecretRefs.

## Solution Assessment

Unclear until maintainers choose the security contract. The proposed shape is plausible, but the best implementation must explicitly account for Docker metadata exposure and active-surface semantics.
