## Summary

Keep open: current main still keeps `sandbox.docker.env` string-only, and adding SecretRef support would be a new security-sensitive config/runtime surface rather than an already-implemented bug fix.

## What This Changes

Allow SecretRef objects in sandbox Docker env maps and resolve them before Docker container creation.

## Best Possible Solution

Keep the string-only Docker env contract until owner/security approval, then implement SecretInput schema, collectors, target registry, docs, and tests end-to-end so Docker receives only resolved strings.

## Reproduction Assessment

Yes for the unsupported config shape: source inspection on current main shows the schema/type accept only string env values and there is no Docker env SecretInput collector or target-registry entry. I did not run a live Docker container because the validation and runtime paths are decisive for this feature request.

## Solution Assessment

Unclear until maintainers approve the new secret materialization surface. If approved, the narrow maintainable path is to reuse the existing SecretInput runtime, active-surface policy, target registry, docs, and regression tests rather than adding Docker-only secret handling.
