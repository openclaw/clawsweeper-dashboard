## Summary

Keep open: current main and the latest release still treat `sandbox.docker.env` values as plain strings and do not collect Docker env SecretRefs, while the exact implementation PR was closed unmerged. The issue is valid, but it needs maintainer product/security review because resolved SecretRefs would still become Docker environment variables visible through Docker metadata.

## What This Changes

The issue requests static `SecretRef` support for Docker sandbox environment variables in default and per-agent sandbox config.

## Best Possible Solution

Make an explicit maintainer security decision: either support `SecretInput` for Docker sandbox env with schema/runtime/registry/docs coverage and documented Docker metadata exposure, or document this path as string-only and recommend a non-env secret delivery mechanism.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro path: put a SecretRef object under `agents.defaults.sandbox.docker.env` and current schema validation rejects it because env values are strings only.

## Solution Assessment

Unclear. The requested implementation is coherent, but the best solution depends on whether maintainers accept Docker metadata exposure or prefer mounted files, custom images, or another secret delivery path.
