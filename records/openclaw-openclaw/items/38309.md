## Summary

Keep open: current `main` and `v2026.5.20` still treat `sandbox.docker.env` as string-only config, and the paired implementation PR is closed unmerged. The implementation shape is clear, but this needs maintainer security/product approval because SecretRefs would resolve into Docker container environment metadata.

## What This Changes

Allow SecretRef objects in `sandbox.docker.env` so Docker sandbox environment variables can reference external secret sources instead of only literal strings.

## Best Possible Solution

After security/product approval, extend the existing SecretInput schema, runtime collector, target registry, docs, and sandbox resolution path while keeping resolved Docker config as `Record<string,string>` and documenting Docker metadata exposure.

## Reproduction Assessment

Yes for the missing capability: current source shows the schema and TypeScript contract only accept string env values, and the runtime path has no Docker env SecretRef collector/resolution step. I did not run validation because this review is read-only and the source path is unambiguous.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable approach is to reuse the existing SecretInput collector, target registry, and runtime snapshot machinery rather than adding Docker-only secret handling.
