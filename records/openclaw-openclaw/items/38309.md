## Summary

Keep open: current main still rejects and does not resolve SecretRef values on the Docker sandbox env surface, and the paired implementation PR is closed unmerged. The request is valid but security-sensitive because it would materialize external secrets into container environment variables.

## What This Changes

Request support for SecretRef objects in defaults and per-agent `sandbox.docker.env` while keeping the resolved Docker runtime env as strings.

## Best Possible Solution

Add this as an end-to-end supported SecretInput surface through the existing secrets resolver, active-surface policy, target registry, generated schemas, docs, and focused sandbox/config tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a structured `{source, provider, id}` SecretRef under `agents.defaults.sandbox.docker.env`; current Zod and generated schemas require string values before Docker launch.

## Solution Assessment

Unclear because there is no current patch to review. The requested direction is plausible, but the maintainable fix should use the existing SecretInput resolver, target registry, and docs matrix instead of Docker-only ad hoc resolution.
