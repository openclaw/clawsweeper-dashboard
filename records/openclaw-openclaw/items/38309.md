## Summary

Keep open: current main still does not support SecretRef values in Docker sandbox env, and the linked implementation PR is closed unmerged. This is a valid feature request, but it expands the supported secret materialization surface into Docker container environment variables and needs maintainer/secops approval before implementation.

## What This Changes

Allow `sandbox.docker.env` values to use SecretRef objects while preserving literal string env values.

## Best Possible Solution

After approval, add end-to-end SecretInput support for Docker sandbox env through schema/types, runtime collectors, target registry, generated docs, and focused tests while preserving string compatibility and avoiding resolved-secret persistence.

## Reproduction Assessment

Yes, source-level. A structured SecretRef value under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` still hits the current string-only schema, and there is no current Docker env SecretRef collector before launch.

## Solution Assessment

Unclear until maintainers approve the surface. If approved, the narrow maintainable path is to reuse the existing SecretInput resolver, active-surface policy, and target registry instead of adding Docker-specific secret handling.
