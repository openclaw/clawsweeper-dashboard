## Summary

Keep open: current main and the latest release still validate and type Docker sandbox env values as plain strings, and the linked implementation PR was closed unmerged. The request is source-reproducible, but it needs security/product approval because SecretRefs would intentionally materialize into Docker container environment metadata.

## What This Changes

Allow `sandbox.docker.env` entries to accept SecretRef objects and resolve them before Docker sandbox launch while preserving existing plain-string env values.

## Best Possible Solution

After security/product approval, extend the existing SecretInput schema, runtime collector, target registry, docs, and sandbox resolution path with explicit Docker metadata exposure documentation.

## Reproduction Assessment

Yes for the missing capability: current source shows the schema and TypeScript contract only accept string env values, and the runtime secret collector has no Docker env assignment path. I did not run tests because this cleanup review is read-only and the source path is unambiguous.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable fix is to reuse the existing SecretInput collector, target registry, and runtime snapshot machinery rather than adding Docker-only secret handling.
