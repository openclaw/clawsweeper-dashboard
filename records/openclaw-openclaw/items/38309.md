## Summary

Keep open: current main still does not support object-shaped SecretRefs in `sandbox.docker.env`, and the remaining work is a security-sensitive feature decision rather than implemented cleanup.

## What This Changes

Request support for `SecretRef` values in `sandbox.docker.env` while preserving existing plain string Docker env behavior.

## Best Possible Solution

After maintainer/security approval, implement SecretInput support end to end through schema, public config types, collectors, target registry, docs, and focused tests while keeping resolved Docker runtime env as `Record<string, string>`.

## Reproduction Assessment

Yes, source-reproducible: current main’s schema and public type accept only string Docker env values, and the SecretRef collector/registry paths do not include Docker env. I did not run a live container because object-shaped SecretRefs are rejected before sandbox launch.

## Solution Assessment

Unclear until maintainers approve this new secret materialization surface. If approved, the maintainable path is to reuse the existing SecretInput schema, runtime collectors, target registry, audit, and docs machinery rather than adding Docker-only resolution logic.
