## Summary

Keep open: current main and the latest release still treat `sandbox.docker.env` values as plain strings, while the linked implementation PR was closed unmerged. The request is valid, but it needs maintainer product/security review because resolved SecretRefs would still become Docker environment variables visible through Docker metadata.

## What This Changes

Allow global and per-agent `sandbox.docker.env` entries to use `SecretRef`/`SecretInput` values instead of only plaintext strings.

## Best Possible Solution

Make an explicit maintainer security decision: either support `SecretInput` on Docker sandbox env with documented Docker metadata exposure and full schema/runtime/registry coverage, or document that this path stays string-only and recommend a non-env secret delivery path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro path: put a SecretRef object under `agents.defaults.sandbox.docker.env` and current config validation will reject it because the schema requires string values.

## Solution Assessment

Unclear. The requested implementation shape is technically coherent, but the best solution depends on whether maintainers accept Docker metadata exposure for SecretRef-resolved values or prefer a different secret delivery path.
