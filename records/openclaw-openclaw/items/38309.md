## Summary

Keep open: current main still does not support SecretRef values in `sandbox.docker.env`. The requested feature is concrete, but it expands secret materialization into Docker container environment variables and should get maintainer/secops approval before implementation.

## What This Changes

Support SecretRef objects as values in sandbox Docker env config and resolve them to strings before Docker container launch.

## Best Possible Solution

Approve the new secret surface first, then implement end-to-end SecretInput support through schema/types, runtime collectors, target registry, docs, and focused regression tests.

## Reproduction Assessment

Yes, source-level. A SecretRef object under `agents.defaults.sandbox.docker.env` or `agents.list[].sandbox.docker.env` still conflicts with the string-only schema, and no collector resolves that path before Docker launch.

## Solution Assessment

Unclear until maintainers approve this secret materialization surface. If approved, the narrow maintainable path is to reuse the existing SecretInput collector/registry patterns and ensure only resolved strings reach Docker create.
