## Summary

Keep open: current main and the latest release still treat `sandbox.docker.env` values as strings, and the only linked implementation PR was closed unmerged. The request is source-reproducible, but it needs maintainer security/product approval because it would intentionally resolve SecretRefs into Docker container environment metadata.

## What This Changes

Support SecretRef objects as values in `sandbox.docker.env` while keeping resolved Docker sandbox env as `Record<string, string>`.

## Best Possible Solution

After security/product approval, extend the existing SecretInput schema, runtime collector, target registry, docs, and sandbox resolution path while documenting Docker metadata exposure.

## Reproduction Assessment

Yes for the missing capability: current source shows the schema and TypeScript contract only accept string env values, and the runtime secret collector has no Docker env assignment path. I did not run tests because this cleanup review is read-only and the source path is unambiguous.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable fix is to reuse the existing SecretInput collector, target registry, and runtime snapshot machinery rather than adding Docker-only secret handling.
