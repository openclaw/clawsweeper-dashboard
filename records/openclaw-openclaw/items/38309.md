## Summary

Keep open: current main and the latest release still treat `sandbox.docker.env` as plain string-only config, and enabling SecretRef here needs an explicit maintainer security/product decision because the resolved values become Docker container environment metadata.

## What This Changes

The issue requests SecretRef support for `sandbox.docker.env` values so Docker sandbox environment variables can reference configured secrets.

## Best Possible Solution

After maintainer security/product approval, extend `sandbox.docker.env` through the existing SecretInput schema, collector, target registry, docs, and sandbox resolution path while preserving resolved runtime `Record<string,string>` output and documenting Docker metadata exposure.

## Reproduction Assessment

Yes, by source inspection: the schema and TypeScript contract only accept string env values, runtime config merges env maps directly, and the SecretRef collectors/registry do not include Docker env paths. I did not run a live config validation command because this review is read-only and the source path is unambiguous.

## Solution Assessment

Unclear until maintainers approve Docker env as a SecretRef materialization surface. If approved, the maintainable solution is to reuse the existing SecretInput collector/registry/runtime snapshot machinery rather than adding Docker-only secret handling.
