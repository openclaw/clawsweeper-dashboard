## Summary

Keep this PR open: the status usage fix is targeted and source-backed, but the patch needs a small test-isolation cleanup and the external contributor has not provided inspectable after-fix real behavior proof yet.

## What This Changes

This PR maps `/status` usage resolution to `openai-codex` when an OpenAI model runs through the Codex harness and adds status regression assertions.

## Best Possible Solution

Keep the focused usage-provider mapping, restore the scoped fetch mock in the regression test, then merge after redacted real `/status` output and normal touched-surface checks are available.

## Reproduction Assessment

Yes, from source inspection: current main maps Codex-harness auth labels to `openai-codex`, but the usage lookup still calls `resolveUsageProviderId(provider)` with `openai`, which is not a usage provider. I did not run a live gateway reproduction during this read-only sweep.

## Solution Assessment

Mostly yes: a status-only usage-provider remap is the narrow maintainable fix and preserves the separate auth label behavior. The PR still needs scoped mock cleanup in the test and inspectable real behavior proof before merge.
