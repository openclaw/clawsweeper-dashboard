## Summary

Keep open: current main does not implement the weighted-recall feature, and this PR still needs contributor proof plus maintainer decisions before it is mergeable because the feedback selector, active-backend behavior, and persistence semantics are not sound as submitted.

## What This Changes

The PR adds SkillFoundry-derived bundled skill docs and a memory-core weighted recall feature with feedback CLI commands, a JSON weight store, ranking integration, tests, and changelog entries.

## Best Possible Solution

Split or narrow the work: publish the new skills through ClawHub unless core bundling is approved, and land weighted recall only with a stable feedback selector, backend-aware semantics, merge-safe persistence, docs/tests, and real CLI proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection: the PR requires a chunk id that current supported search contracts and normal CLI output do not expose, and the active QMD backend path does not consult the new weight store. I did not execute the PR because this review kept the checkout read-only.

## Solution Assessment

No, not as submitted. Weighted recall may be useful, but the maintainable path needs a supported user-facing selector, backend-aware application or rejection, concurrency-safe state updates, and a separate maintainer decision for bundled skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Expose a supported feedback selector:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback commands require `<chunk-id>`, but the supported `MemorySearchResult` contract and normal `memory search` output do not expose an id. Users cannot discover a valid argument through the CLI/API, so the feedback loop is unusable without inspecting internals.
  - confidence: 0.93
- **[P2] Gate feedback on the active memory backend:** `extensions/memory-core/src/cli.runtime.ts:815`
  - body: The command writes `WeightedRecallStore` directly, but current main can route searches to `QmdMemoryManager`, which returns rankings without consulting `memory-weights.json`. In that backend the CLI can print success while recall/correction has no ranking effect.
  - confidence: 0.86
- **[P2] Merge on-disk weights before flushing:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the entire cached map back to `memory-weights.json`. Two feedback commands that load the same starting file and update different chunks can silently lose one update when the later writer wins; use locking or an atomic read-merge-write path.
  - confidence: 0.88
- **[P3] Remove the replacement-weight promise:** `extensions/memory-core/src/cli.runtime.ts:828`
  - body: The correction command tells users new entries start at `0.70`, but it never records or associates a replacement chunk id, and the comment above says replacements start at default weight. This promises behavior the command does not implement.
  - confidence: 0.8
