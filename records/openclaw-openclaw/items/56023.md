## Summary

Keep open: current main does not implement weighted recall feedback, and the PR still has source-level UX and persistence blockers plus a bundled-skills scope decision before it is mergeable.

## What This Changes

The PR adds three bundled agent skills plus memory-core weighted recall scoring, feedback CLI commands, local weight persistence, tests, and changelog entries.

## Best Possible Solution

Split or narrow the PR: route the new skills through ClawHub unless maintainers approve bundling, and land weighted recall only after it has a supported feedback selector, concurrency-safe persistence, docs, tests, and a current-main rebase.

## Reproduction Assessment

Yes for the PR review findings from source inspection: the proposed commands require chunk ids that supported search output/API do not expose, and two store instances can overwrite each other's distinct updates. I did not execute the PR because this was a read-only review and pnpm is unavailable in the checkout.

## Solution Assessment

No, not as submitted. Weighted recall may be useful, but the maintainable path needs a stable user-facing feedback target, locked read-merge-write persistence, current-main rebase work, and a separate maintainer decision on bundled skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Expose a feedback selector before requiring chunk IDs:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback subcommands require `<chunk-id>`, but normal `memory search` output only prints path/line/snippet and `MemorySearchResult` has no `id`. Users cannot discover a valid argument through the supported CLI/API, so recall/correct is unusable in the main workflow.
  - confidence: 0.92
- **[P2] Preserve concurrent feedback updates:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the cached map over `memory-weights.json`. Two feedback processes that load the same starting file and update different chunks will lose one update when the later flush wins; use a lock/read-merge-write pattern like the existing short-term recall state.
  - confidence: 0.88
- **[P3] Remove the unrecorded replacement-weight promise:** `extensions/memory-core/src/cli.runtime.ts:828`
  - body: `recordCorrection()` returns 0.70 and this message says new entries start at that weight, but the command never records a replacement chunk id. A user-created corrected memory entry will still default to 1.0, so the CLI promises behavior it does not implement.
  - confidence: 0.8
