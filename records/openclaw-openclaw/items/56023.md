## Summary

Keep this PR open. Current main still does not contain the weighted-recall feature or these SkillFoundry skills, but the branch should not merge as submitted because the feedback CLI lacks a usable target contract, feedback persistence can drop concurrent updates, and bundled skills need maintainer scope judgment under the ClawHub-first policy.

## What This Changes

The PR adds three SkillFoundry-style repo agent skills plus memory-core weighted recall scoring, feedback CLI commands, JSON persistence, tests, and changelog entries.

## Best Possible Solution

Split the work before merge: keep weighted recall as a focused memory-core design PR with a supported feedback selector, consistent ranking behavior, concurrency-safe persistence, docs, and tests; route the three skills through ClawHub or explicit maintainer approval.

## Reproduction Assessment

Yes for the review findings. On the PR head, the CLI requires chunk ids that normal search results do not expose, and the lost-update issue follows from two WeightedRecallStore instances loading the same file, changing different ids, and flushing last-writer-wins state.

## Solution Assessment

No, not as submitted. Weighted recall may be useful, but the maintainable path needs a stable feedback target, locked read-merge-write persistence, a current-main rebase, and a separate decision on bundled skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Expose a feedback selector before requiring chunk IDs:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback commands require `<chunk-id>`, but normal `memory search` output and the public `MemorySearchResult` type do not expose chunk ids. Users have no supported way to supply this argument, making recall/correct unusable outside internal inspection.
  - confidence: 0.92
- **[P2] Preserve concurrent feedback updates:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the whole cached map directly to `memory-weights.json`. If two feedback commands update different chunks from the same starting snapshot, the later write silently drops the earlier update; use the repo's lock/atomic read-merge-write pattern for this shared per-agent state file.
  - confidence: 0.86
- **[P3] Align correction replacement weight messaging:** `extensions/memory-core/src/cli.runtime.ts:817-828`
  - body: `recordCorrection()` returns `0.7` and the CLI tells users new entries start at that weight, but the command never records a replacement chunk id, so a corrected memory file entry still defaults to `1.0`. Either store a replacement selector or remove the `0.70` promise.
  - confidence: 0.78
