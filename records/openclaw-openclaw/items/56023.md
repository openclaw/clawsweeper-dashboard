## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Expose a supported feedback selector:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback commands require `<chunk-id>`, but supported search output does not expose a stable chunk id and the public result type omits it. Normal users cannot discover a valid argument without inspecting internals, so add a stable selector to search output or accept an existing visible selector such as path/range and cover hybrid search too.
  - confidence: 0.9
- **[P2] Gate feedback on the active backend:** `extensions/memory-core/src/cli.runtime.ts:783-784`
  - body: This handler writes `memory-weights.json` directly under the agent state dir without checking which memory backend is active. When `memory.backend = "qmd"`, searches go through QMD and never read this file, so the CLI can print success while ranking is unchanged; resolve the active backend and either implement parity or fail clearly.
  - confidence: 0.88
- **[P2] Merge on-disk weights before flushing:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the entire cached weight map back to disk. Two concurrent `memory feedback` processes can load the same old file, update different chunks, and have the last writer discard the first update; use file locking or an atomic read-merge-write path before writing.
  - confidence: 0.86
- **[P3] Coalesce weight file loads during search:** `extensions/memory-core/src/memory/weighted-recall.ts:143-145`
  - body: After the manager invalidates the cache before each search, `applyWeightsToResults()` starts one `getWeight()` per result concurrently. Because `load()` has no in-flight read memoization, a single search can issue many duplicate reads/parses of the same JSON file; preload once per weighting pass or share a pending load.
  - confidence: 0.8
