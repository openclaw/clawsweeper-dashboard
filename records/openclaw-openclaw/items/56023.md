## Summary

Keep open: the PR is a real feature proposal, but current `main` does not already implement it and the branch is not merge-ready because the feedback CLI lacks a discoverable selector, can report success on a backend that ignores its weights, and still has no contributor real-behavior proof.

## What This Changes

The PR adds three SkillFoundry-derived bundled skills plus memory-core weighted recall scoring, `openclaw memory feedback recall/correct` subcommands, local weight persistence, tests, and changelog entries.

## Best Possible Solution

Split or narrow the work: publish the new skills through ClawHub unless maintainers approve bundling, and land weighted recall only after it has a stable feedback selector, backend-aware ranking, merge-safe persistence, focused coverage, docs, and real CLI proof.

## Solution Assessment

No, not as submitted. Weighted recall may be useful, but the maintainable path needs a stable selector, backend-aware application, concurrency-safe state updates, and a separate maintainer decision for bundled skills.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Expose a supported feedback selector:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback commands require `<chunk-id>`, but the supported `MemorySearchResult` contract and normal `memory search` output do not expose any id. Users cannot discover a valid argument through the CLI/API, so the feedback loop is unusable without inspecting internals.
  - confidence: 0.92
- **[P2] Gate feedback on the active backend:** `extensions/memory-core/src/cli.runtime.ts:815`
  - body: The commands always write `WeightedRecallStore`, but current `main` can route memory search to `QmdMemoryManager`, which returns rankings without consulting `memory-weights.json`. In that backend, the CLI can print success while recall/correction has no effect.
  - confidence: 0.84
- **[P2] Merge feedback writes before flushing:** `extensions/memory-core/src/memory/weighted-recall.ts:123-124`
  - body: `flush()` writes the entire cached map back to `memory-weights.json`. Two feedback commands that load the same starting file and update different chunks can silently lose one update when the later writer wins; use locking or a read-merge-write pattern before replacing the file.
  - confidence: 0.88
- **[P3] Remove the replacement-weight promise:** `extensions/memory-core/src/cli.runtime.ts:828`
  - body: The correction command reports that corrected replacement entries should start at weight `0.70`, but it never creates or associates a replacement chunk id. That promises behavior the command does not implement.
  - confidence: 0.8
