## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement the weighted-recall feedback loop or the three proposed SkillFoundry skills, but the branch is not ready as-is because it mixes a bundled-skill product decision with memory-core API/ranking changes and still lacks a usable feedback target contract.

## What This Changes

The PR adds three repo-root agent skill documents plus a memory-core weighted-recall store, feedback CLI commands, search-score weighting integration, tests, and changelog entries.

## Best Possible Solution

Split the work. Keep the memory-core portion as a focused design/implementation PR that exposes a stable feedback selector, applies weights consistently across FTS/vector/hybrid ranking, preserves current-main search behavior, writes state with existing locking/atomic patterns, updates CLI docs, and adds targeted tests. Move the three SkillFoundry skills to ClawHub or a bundle-style plugin unless maintainers explicitly approve bundling them in core.

## Reproduction Assessment

Yes for the review issues: on the PR branch, `openclaw memory feedback recall <chunk-id>` requires an id, while `memory search` output and the public result type do not provide one. The concurrent-write concern is reproducible with two WeightedRecallStore instances loading the same file, writing different ids, and flushing in last-writer-wins order.

## Solution Assessment

No, not as submitted. Weighted recall may be a reasonable memory-core feature, but the current PR needs a narrower memory-only path or a maintainer-approved skills split, plus a real feedback-target contract and concurrency-safe persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Expose a feedback selector before requiring chunk ids:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback commands require `<chunk-id>`, but the user-facing `memory search` output still shows only score and path/line/snippet, the public `MemorySearchResult` type has no id, and hybrid results strip ids before returning JSON. Typical users have no supported way to obtain a valid argument, so the new recall/correct flow is effectively unusable.
  - confidence: 0.92
- **[P2] Preserve concurrent feedback updates:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the whole cached weight map directly to `memory-weights.json`. If two feedback commands update different chunks from the same starting snapshot, the later flush overwrites the earlier update, silently losing feedback. Use the repo's locking/atomic read-merge-write pattern for this per-agent state file.
  - confidence: 0.84
