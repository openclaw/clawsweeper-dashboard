## Summary

Keep this PR open: current main does not implement weighted recall feedback, and the PR still has source-backed usability/persistence defects plus missing real behavior proof and a maintainer scope decision for bundled skills.

## What This Changes

The PR adds three SkillFoundry-derived bundled skills, memory-core weighted recall scoring, feedback CLI commands, local weight persistence, tests, and changelog entries.

## Best Possible Solution

Split or narrow the work: publish the new skills through ClawHub unless maintainers approve bundling, and land weighted recall only after it has a supported feedback selector, locked or merge-safe persistence, focused tests/docs, real CLI proof, and a current-main rebase.

## Reproduction Assessment

Yes for the PR review findings from source inspection: the proposed commands require chunk ids that current supported search output/API do not expose, and the new store writes a whole cached map over disk state. I did not execute the PR because this review kept the checkout read-only.

## Solution Assessment

No, not as submitted. Weighted recall may be useful, but the maintainable path needs a stable user-facing feedback target, concurrency-safe state updates, and a separate maintainer decision for bundled skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Expose a supported feedback selector:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback commands require `<chunk-id>`, but current `MemorySearchResult` has no id and normal `memory search` output prints only score, path/lines, and snippet. Users cannot discover a valid argument through the supported CLI/API, so the feedback flow is effectively unusable without inspecting internals.
  - confidence: 0.92
- **[P2] Merge feedback writes before flushing:** `extensions/memory-core/src/memory/weighted-recall.ts:123-124`
  - body: `flush()` writes the entire cached map back to `memory-weights.json`. Two feedback commands that load the same starting file and update different chunks can silently lose one update when the later writer wins; use a lock or read-merge-write pattern like the existing short-term recall state.
  - confidence: 0.88
- **[P3] Remove the unrecorded replacement-weight message:** `extensions/memory-core/src/cli.runtime.ts:828`
  - body: The correction command tells users new entries start at the returned 0.70 weight, but it never records or associates a replacement chunk id. A user-edited corrected memory entry will still default to 1.0, so the CLI promises behavior it does not implement.
  - confidence: 0.8
