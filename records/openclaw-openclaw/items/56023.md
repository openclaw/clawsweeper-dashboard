## Summary

Keep open: current main does not already implement the proposed feedback feature or bundled skills, and the PR still needs maintainer scope review plus fixes for unusable feedback selectors and lossy persistence before merge.

## What This Changes

The PR adds three bundled agent skills and memory-core weighted recall feedback commands with per-agent JSON weight persistence and search-score weighting.

## Best Possible Solution

Split or narrow the PR: route the new skills through ClawHub unless maintainers approve bundling, and land weighted recall only with a discoverable feedback selector, concurrency-safe persistence, docs/tests, and a current-main rebase.

## Reproduction Assessment

Yes for the PR review findings from source inspection: PR head requires chunk ids that current supported search output and `MemorySearchResult` do not expose, and the proposed store can overwrite concurrent feedback updates. I did not execute tests because this was a read-only review.

## Solution Assessment

No, not as submitted. The weighted recall idea may be useful, but the maintainable path needs a stable user-facing feedback target, locked read-merge-write persistence, current-main rebase work, and a separate maintainer decision on bundled skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Expose a feedback selector before requiring chunk IDs:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback subcommands require `<chunk-id>`, but the supported search contract does not expose an id and normal `memory search` output prints only score, path/line range, and snippet. Users cannot discover a valid argument through the CLI workflow, so recall/correct is effectively unusable for typical operators.
  - confidence: 0.92
- **[P2] Preserve concurrent feedback updates:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the cached map over `memory-weights.json`. Two feedback processes that load the same starting file and update different chunks will lose one update when the later flush wins; use a lock/read-merge-write pattern like the existing short-term recall state.
  - confidence: 0.88
- **[P3] Remove the unrecorded replacement-weight promise:** `extensions/memory-core/src/cli.runtime.ts:828`
  - body: This message says corrected new entries start at the returned 0.70 weight, but the command only demotes the original chunk id and never records a replacement chunk id. A user-created corrected memory entry will still default to 1.0, so the CLI promises behavior it does not implement.
  - confidence: 0.8
