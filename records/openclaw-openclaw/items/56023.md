## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Expose a supported feedback selector:** `extensions/memory-core/src/cli.ts:107`
  - body: The feedback commands require `<chunk-id>`, but current `MemorySearchResult` and normal `memory search` output do not expose any id. Users cannot discover a valid argument through the supported CLI/API, so the feedback flow is unusable without inspecting internals.
  - confidence: 0.92
- **[P2] Merge feedback writes before flushing:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the entire cached map back to `memory-weights.json`. Two feedback commands that load the same starting file and update different chunks can silently lose one update when the later writer wins; use a lock or read-merge-write pattern before replacing the file.
  - confidence: 0.88
- **[P2] Handle QMD before reporting feedback success:** `extensions/memory-core/src/cli.runtime.ts:815`
  - body: The commands always write `WeightedRecallStore`, but current main can route memory search to `QmdMemoryManager`, which returns rankings without consulting `memory-weights.json`. In that backend, the CLI can print success while recall/correction has no effect.
  - confidence: 0.84
- **[P3] Remove the replacement-weight promise:** `extensions/memory-core/src/cli.runtime.ts:828`
  - body: The correction command tells users new entries start at the returned 0.70 weight, but it never records or associates a replacement chunk id. A corrected memory-file entry will still default to 1.0, so this line promises behavior the command does not implement.
  - confidence: 0.8
