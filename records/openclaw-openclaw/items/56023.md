## What This Changes

The PR adds SkillFoundry-derived agent skills plus memory-core weighted recall scoring, feedback recall/correct CLI commands, JSON weight persistence, tests, and changelog entries.

## Best Possible Solution

Split the work: publish the new skills through ClawHub unless core bundling is approved, and land weighted recall only with a stable user-facing feedback selector, backend-aware behavior, merge-safe persistence, focused tests/docs, and real CLI proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection: current supported search outputs expose no stable chunk id, current QMD search bypasses the proposed weight store, and the PR writes cached weights wholesale. I did not execute the PR because this was a read-only review.

## Solution Assessment

No, not as submitted. The weighted-recall direction may be useful, but the maintainable solution needs a supported selector, explicit QMD/builtin backend semantics, concurrency-safe persistence, and a separate decision on bundled skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Expose a supported feedback selector:** `extensions/memory-core/src/cli.ts:107`
  - body: The feedback commands require `<chunk-id>`, but current `MemorySearchResult` and normal `memory search` output do not expose a stable id. Users cannot discover a valid argument through supported CLI/API surfaces, so the feedback loop is unusable without inspecting internals.
  - confidence: 0.92
- **[P2] Gate feedback on the active backend:** `extensions/memory-core/src/cli.runtime.ts:783-784`
  - body: These commands write `WeightedRecallStore` directly, but current main can route memory search to `QmdMemoryManager`, which returns rankings without consulting `memory-weights.json`. In QMD mode the CLI can print success while recall/correction has no effect.
  - confidence: 0.86
- **[P2] Merge on-disk weights before flushing:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the whole cached map to `memory-weights.json`. Two feedback commands that load the same starting file and update different chunks can silently lose one update when the later writer wins; use locking or a read-merge-write update.
  - confidence: 0.88
- **[P3] Align corrected-entry output with implemented behavior:** `extensions/memory-core/src/cli.runtime.ts:828`
  - body: The correction command tells users new entries start at `0.70`, but the code only demotes the supplied chunk and does not create or associate a replacement entry. A manually added replacement will not receive that weight, so the CLI promises behavior it does not implement.
  - confidence: 0.8
