## Summary

Keep open: current main does not implement weighted recall, and the PR still has source-backed correctness blockers plus missing real CLI proof and an unresolved product/scope question around bundling new skills in core.

## What This Changes

This PR adds three SkillFoundry-derived skills, memory-core weighted recall scoring, feedback recall/correct CLI commands, a per-agent weight store, tests, and changelog entries.

## Best Possible Solution

Split or narrow the work: publish the new skills through ClawHub unless core bundling is approved, and land weighted recall only with a stable feedback selector, backend-aware ranking semantics, merge-safe persistence, focused tests/docs, and real CLI proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection: current main exposes no stable chunk id in the supported result contract or normal CLI output, QMD search bypasses the PR's weight store, and the PR flush path can overwrite concurrent updates. I did not execute the PR because this review kept the checkout read-only.

## Solution Assessment

No, not as submitted. Weighted recall may be useful, but the maintainable path needs a supported user-facing selector, explicit backend behavior, concurrency-safe persistence, and a separate maintainer decision for bundled skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Expose a supported feedback selector:** `extensions/memory-core/src/cli.ts:107`
  - body: The new feedback commands require `<chunk-id>`, but the supported `MemorySearchResult` contract and normal `memory search` output do not expose a stable id. Users cannot discover a valid argument through the CLI/API, so the feedback loop is unusable without inspecting internals.
  - confidence: 0.92
- **[P2] Gate feedback on the active backend:** `extensions/memory-core/src/cli.runtime.ts:783-784`
  - body: The feedback commands write `WeightedRecallStore` directly, but current main can route memory search to `QmdMemoryManager`, which returns rankings without consulting `memory-weights.json`. In that backend the CLI can print success while recall/correction has no effect.
  - confidence: 0.84
- **[P2] Merge on-disk weights before flushing:** `extensions/memory-core/src/memory/weighted-recall.ts:123`
  - body: `flush()` writes the entire cached map back to `memory-weights.json`. Two feedback commands that load the same starting file and update different chunks can silently lose one update when the later writer wins; use locking or a read-merge-write pattern before replacing the file.
  - confidence: 0.88
- **[P3] Remove the replacement-weight promise:** `extensions/memory-core/src/cli.runtime.ts:828`
  - body: The correction command tells users new entries start at `0.70`, but it never records or associates a replacement chunk id, and the comment above says replacements start at the default weight. This promises behavior the command does not implement.
  - confidence: 0.8
