## Label Justifications

- `P1`: The linked regression prevents explicitly enabled external chat channel plugins from loading at gateway boot, breaking a real channel workflow for affected users.
- `merge-risk: 🚨 compatibility`: Existing configs with `plugins.entries.<id>.enabled=true` for external channel plugins can start executing that plugin at gateway boot after this change.
- `merge-risk: 🚨 availability`: Starting third-party channel plugin runtime during gateway boot can affect startup latency or stability, and CI does not prove the real installed-plugin path.

## Summary

Keep open: the patch is a plausible and likely correct startup-planner fix for the linked external channel regression, but the PR still needs real after-fix gateway boot proof and maintainer acceptance of the startup/upgrade behavior before merge.

## What This Changes

The branch adds a startup-plan predicate and regression test so explicitly enabled non-bundled channel plugins can load at gateway boot without a `channels.<id>` config block, while tool-only plugins stay lazy.

## Best Possible Solution

Land this startup-planner fix after redacted real boot proof shows the external channel plugin loads from explicit plugin entry enablement, while preserving denylist, allowlist, bundled-channel, tool-laziness, and deferred-load boundaries.

## Reproduction Assessment

Yes at source level: current main excludes an explicitly enabled external channel plugin from the gateway startup plan when no `channels.<id>` config exists. I did not run a live installed-plugin gateway boot in this read-only review.

## Solution Assessment

Yes, this appears to be the best code layer: it fixes the generic startup planner rather than adding a Weixin-specific workaround or relying on a CLI restart fallback. The remaining blocker is proof and explicit acceptance of the startup behavior change, not a line-level patch defect.
