## Label Justifications

- `P1`: The issue reports source-backed cross-user cron job disclosure and unauthorized schedule mutation in real shared deployments.
- `impact:data-loss`: Current id-only remove and update paths can destroy or overwrite another agent's persisted scheduled-job state when a caller has cron access.

- `impact:message-loss`: Unauthorized cron edits, removals, or delivery-target changes can suppress or misroute scheduled messages.

## Summary

Keep open: current main only partially scopes cron listing by agentId, while cron get/update/remove/run/runs still use bare job ids or global job-name enrichment, so the reported multi-agent isolation problem remains security-sensitive and needs maintainer product direction.

## What This Changes

The issue requests per-agent or otherwise caller-scoped cron job isolation so agents cannot list, mutate, run, or inspect other agents’ scheduled jobs in shared deployments.

## Best Possible Solution

Design a trusted cron ownership model over the current SQLite-backed cron store, derive caller scope from Gateway/tool context, enforce list/get/update/remove/run/runs by default, preserve explicit operator all-jobs behavior, and document legacy unowned-job handling with focused tests.

## Reproduction Assessment

Yes by source inspection: create jobs with different agentId/sessionKey values, then get/update/remove/run a foreign job by id or request all-scope run history; current main lacks trusted caller ownership checks on those paths.

## Solution Assessment

No for the literal per-agent jobs.json shape because cron is now SQLite-backed; the right direction is a service/protocol/Gateway ownership boundary with explicit operator behavior.
