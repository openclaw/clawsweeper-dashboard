## Summary

Keep open. Current main has a partial list-side agentId filter, but cron mutation and run-history paths still lack a trusted caller ownership check, and the discussion contains concrete production leakage/tampering reports plus recent maintainer-facing comments keeping the broader ownership design open.

## What This Changes

Request per-agent or per-session isolation for cron scheduled jobs in shared multi-user deployments.

## Best Possible Solution

Design and land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service, storage/history paths, CLI/UI/docs, with explicit admin access and safe legacy handling.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create jobs with distinct agentId/sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs scope all from another caller; current main has no caller ownership guard on those paths.

## Solution Assessment

No. The current list-side agentId filter is only a partial mitigation, and per-agent jobs.json or a list-only filter would be incomplete without trusted enforcement across mutation, run, history, and admin paths.
