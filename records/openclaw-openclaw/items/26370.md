## Summary

Keep this issue open: current main partially limits agent-tool listings by agent ID, but agent-facing get, update, remove, run, and run-history paths do not consistently enforce trusted caller ownership, so the documented production disclosure and tampering risk remains.

## What This Changes

Enforce cron job ownership isolation so callers in shared deployments cannot view, modify, delete, execute, or inspect jobs outside their trusted agent or session scope.

## Best Possible Solution

Define a closed trusted caller-scope model at the agent/Gateway boundary, enforce it before every agent-facing cron read, mutation, execution, and history lookup, keep operator-wide management explicitly separate, and migrate legacy ownership metadata in canonical SQLite storage.

## Reproduction Assessment

Yes at source level: current main only applies an agent filter to listing, while agent-facing get, update, remove, and run operations remain globally addressable by job ID. Independent production comments report the corresponding disclosure and cross-user modification behavior.

## Solution Assessment

No, separate per-agent jobs.json files are not the best solution because they conflict with the repository’s canonical-state policy and still cannot isolate multiple users sharing one agent. Trusted caller-derived authorization over the canonical store is narrower and covers the full problem.
