## Summary

Keep open. Current main has only partial list-side cron scoping, while cron mutation paths and all-scope run history still operate without a trusted caller ownership guard; the discussion also includes concrete production leakage and tampering reports.

## What This Changes

Add agent-scoped cron storage and access control so agents in shared deployments can only see and manage their own scheduled jobs, with explicit admin/global handling.

## Best Possible Solution

Land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service, storage/run-history paths, CLI/UI/docs/tests, with explicit admin access and safe legacy job handling.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct agentId or sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs with scope=all as another caller; current main has no trusted caller ownership guard on those paths.

## Solution Assessment

No, the current list-side agentId filter is only a partial mitigation. The maintainable fix needs trusted caller scope across list, mutation, run, history, admin, and legacy/global job paths rather than only per-agent files or a list-only filter.
