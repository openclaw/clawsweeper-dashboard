## Summary

Keep open: current main still only partially mitigates the reported cron isolation problem, with list-side agentId filtering but no trusted caller ownership guard for mutation and all-scope run-history paths. The issue has concrete production reports, a recent maintainer comment keeping it open, and a narrower open follow-up for one run-history name leak, so this remains an access-control/product design item rather than cleanup-close material.

## What This Changes

The issue requests agent-scoped cron job storage and access control so shared deployments cannot list, edit, delete, run, or inspect other agents' scheduled jobs by default.

## Best Possible Solution

Land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, run-history paths, CLI/UI/docs/tests, with explicit admin access and safe legacy job handling.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct agentId/sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs with scope=all as another caller; current main has no trusted caller ownership guard on those paths.

## Solution Assessment

No, the current list-side agentId filter is only a partial mitigation. The maintainable solution needs caller-scoped ownership checks across list, mutation, run, history, admin, and legacy/global job behavior rather than only per-agent files or a list-only filter.
