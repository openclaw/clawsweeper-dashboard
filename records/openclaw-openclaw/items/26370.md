## Summary

Keep open: current main still lacks a complete trusted cron ownership boundary for shared deployments; list has partial agent filtering, but mutation and run-history paths remain id/global-scope based, and the issue has concrete production reports plus product/security-review labels.

## What This Changes

Request per-agent or equivalent trusted-scope isolation for cron scheduled jobs so agents in multi-user deployments cannot view, update, delete, run, or inspect other agents' jobs by default.

## Best Possible Solution

Define one trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI, docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct agentId/sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs with scope=all from another caller; current main does not compare those requests to trusted caller ownership.

## Solution Assessment

No, the proposed per-agent jobs.json split or a list-only filter is incomplete by itself. The maintainable solution needs trusted caller-scoped checks for visibility, mutations, run history, admin access, and legacy/global jobs.
