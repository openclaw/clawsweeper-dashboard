## Summary

Keep open: current main still only partially scopes cron listing by agentId, while update, remove, run, and all-scope run-history paths lack trusted caller ownership checks. The issue has concrete production leakage/tampering reports, a prior keep-open maintainer review, and active product/security-review labels, so it is not cleanup-close material.

## What This Changes

The issue requests an agent/session-scoped cron ownership model so scheduled jobs are not globally visible or mutable in shared deployments.

## Best Possible Solution

Define and land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI/docs, and tests, with explicit admin access and safe legacy-job handling.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct agentId/sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs with scope=all as another caller; current handlers and service methods do not compare the target job owner to trusted caller scope.

## Solution Assessment

No, a per-agent jobs.json split or list-only filter is not a complete solution. The maintainable fix needs trusted caller-scoped ownership checks across list, mutation, run, history, admin, and legacy/global-job behavior.
