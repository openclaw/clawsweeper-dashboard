## Summary

Keep open: current main only partially scopes cron listing by `agentId`, while cron mutations and all-scope run-history enrichment still lack a trusted caller ownership boundary. The issue has concrete production reports of leakage and tampering plus active P1/product/security-review labels, so it is not conservative cleanup-close material.

## What This Changes

Request to isolate cron scheduled jobs per agent/session so shared deployments cannot list, modify, delete, or run other users' jobs, with a safe admin and legacy-job model.

## Best Possible Solution

Define one trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: create jobs with distinct `agentId` or `sessionKey` metadata, then call `cron.update`, `cron.remove`, `cron.run`, or `cron.runs(scope=all)` as another caller. Current handlers and service methods do not compare the target job owner to trusted caller scope.

## Solution Assessment

No, a per-agent `jobs.json` split or the current list-only `agentId` filter is not complete by itself. The maintainable fix needs trusted caller-scoped ownership checks across list, mutation, run, history, admin, and legacy/global-job behavior.
