## Label Justifications

- `P1`: Shared cron ownership gaps can expose or modify scheduled jobs in real multi-user deployments.
- `impact:data-loss`: The report and comments describe unauthorized modification or deletion of persisted scheduled job data.
- `impact:security`: The item is about cross-agent and cross-user visibility and mutation across tenant boundaries.
- `impact:message-loss`: Mis-scoped or tampered scheduled jobs can misdeliver, suppress, or redirect scheduled channel messages.

## Summary

Keep open: current `main` still has only partial `agentId` list filtering, while cron mutations, manual runs, and all-scope run-history enrichment are not protected by a trusted caller ownership check. The discussion includes concrete production leakage/tampering reports, and the open related PR only addresses the narrower all-scope run-history job-name leak.

## What This Changes

Request per-agent or per-session cron job isolation so shared deployments cannot list, modify, delete, run, or leak other agents' scheduled jobs.

## Best Possible Solution

Define one cron ownership contract across agent tool calls, Gateway protocol/handlers, service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes. Source inspection shows jobs carry agentId/sessionKey metadata, but current main still lets cron.update, cron.remove, cron.run, and all-scope cron.runs operate without comparing the target job to a trusted caller scope.

## Solution Assessment

No for the literal per-agent jobs.json proposal by itself. The maintainable fix needs trusted caller-scoped authorization across list, mutation, manual run, run history, admin override, and legacy/global-job behavior; storage layout is only one part of that model.
