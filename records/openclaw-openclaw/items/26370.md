## Summary

Keep open: current main only partially scopes cron listing by agentId, while cron mutation and all-scope run-history paths still lack trusted caller ownership checks. The discussion includes concrete production leakage/tampering reports and active product/security-review labels, so this is not cleanup-close material.

## What This Changes

Request to isolate cron scheduled jobs per agent or user by scoping storage and CRUD access, with legacy migration and optional admin aggregation.

## Best Possible Solution

Define and land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI/docs, and tests, with explicit admin access and safe legacy-job handling.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct agentId/sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs with scope=all as another caller; current handlers and service methods do not compare the job owner to a trusted caller scope.

## Solution Assessment

No, a per-agent jobs.json split or list-only filter is not a complete fix. The maintainable solution needs trusted caller-scoped ownership checks across list, mutation, run, history, admin, and legacy/global-job behavior.
