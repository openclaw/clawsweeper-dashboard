## Summary

Keep open: current main and the latest release still only provide partial `agentId` list filtering, while trusted caller ownership is not enforced for cron mutation, manual run, or all-scope run-history paths. The issue has concrete production reports of cross-user visibility and tampering, and the related open PR only covers the narrower all-runs job-name leak.

## What This Changes

Request per-agent or otherwise caller-scoped cron job isolation so agents in multi-user deployments cannot view or modify other agents' scheduled jobs.

## Best Possible Solution

Define one trusted cron ownership contract across agent tool calls, Gateway protocol/handlers, service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes. Source inspection on current main shows jobs carry `agentId`/`sessionKey`, but update, remove, manual run, and all-scope run-history paths still operate without comparing the target job to trusted caller scope.

## Solution Assessment

No for the literal per-agent `jobs.json` proposal by itself. The maintainable fix needs caller-scoped authorization across list, mutation, manual run, run history, admin override, and legacy/global-job behavior; storage layout is only one part of that model.
