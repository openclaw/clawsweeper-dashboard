## Summary

Keep open. Current main has partial `agentId` filtering for `cron.list`, but the broader reported ownership boundary is still missing because update, remove, run, and run-history paths remain addressable by bare job id or unfiltered job lists; related items do not supersede this canonical tracker.

## What This Changes

Request to isolate cron scheduled-job visibility and mutations by agent/session in multi-user deployments.

## Best Possible Solution

Keep this issue open as the canonical maintainer-designed core cron ownership tracker: enforce trusted caller scope for visibility and mutations over the canonical SQLite-backed store, define explicit admin/operator all-jobs behavior, and handle legacy/global jobs through migration or doctor policy.

## Reproduction Assessment

Yes. Source inspection on current main shows the remaining management paths still operate by bare job id or unfiltered job lists, and the issue discussion includes production reports of matching cross-user leakage and tampering; I did not run a live multi-user Gateway repro in this read-only review.

## Solution Assessment

No for the literal per-agent `jobs.json` proposal. Current cron state is SQLite-backed, and the narrower maintainable solution is a trusted caller-scope authorization model over the canonical store with explicit admin/operator behavior.
