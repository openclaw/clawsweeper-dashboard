## Label Justifications

- `P1`: The issue reports source-backed cross-user cron disclosure and unauthorized schedule mutation in real shared deployments.
- `impact:data-loss`: Current id-only cron update/remove paths can overwrite or delete another agent's persisted scheduled-job state when a caller has cron access.
- `impact:security`: The central problem is a cross-user authorization and metadata-visibility boundary for scheduled jobs.
- `impact:message-loss`: Unauthorized cron edits, removals, or delivery-target changes can suppress or misroute scheduled messages.

## Summary

Keep open: current main and the latest release still only partially scope cron listing, while cron get/update/remove/run/runs remain security-sensitive global or id-only paths that need maintainer-owned cron ownership design.

## What This Changes

Request per-agent/per-caller isolation for scheduled cron jobs so agents cannot view, mutate, delete, run, or inspect another agent's scheduled work by default.

## Best Possible Solution

Design a trusted cron ownership model over the current SQLite-backed scheduler: derive caller scope from Gateway/tool context, enforce list/get/update/remove/run/runs by default, keep an explicit operator all-jobs path, and document legacy unowned-job handling with focused tests.

## Reproduction Assessment

Yes, at source level: create jobs with different agentId/sessionKey values, then get/update/remove/run a foreign job by id or request all-scope run history; current main lacks trusted caller ownership checks on those paths. I did not run a live gateway repro in this read-only review.

## Solution Assessment

No for the literal per-agent jobs.json storage shape because cron is now SQLite-backed. The maintainable solution is a trusted service/protocol/Gateway ownership boundary with explicit operator behavior and migration handling.
