## Summary

Keep open. Current main only partially addresses the report with agentId filtering for cron.list, while cron.update, cron.remove, cron.run, and all-scope cron.runs still lack trusted caller ownership checks; the discussion also includes concrete production leakage/tampering reports and a maintainer note keeping the ownership model open.

## What This Changes

Request isolated cron job visibility and management per agent/session so shared deployments do not expose or allow mutation of other users' scheduled jobs.

## Best Possible Solution

Design and land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service, storage/history paths, and docs, with explicit admin access and safe legacy handling.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists for the remaining issue: create jobs with distinct agentId/sessionKey metadata, then call update/remove/run by id or all-scope run history from another context; current main has no caller ownership guard on those paths.

## Solution Assessment

No. Current main is only a partial list-side fix, and per-agent jobs.json or a list-only filter would be incomplete; the maintainable fix needs trusted enforcement across list, mutation, run, and history paths with an explicit admin model.
