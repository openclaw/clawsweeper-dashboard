## Label Justifications

- `P1`: Shared cron ownership gaps can expose or mutate scheduled jobs in real multi-user deployments.
- `impact:data-loss`: The reported behavior includes unauthorized modification or deletion of persisted scheduled jobs.
- `impact:security`: The item is about cross-agent and cross-user visibility and control over scheduled job data.
- `impact:message-loss`: Tampered or mis-scoped cron jobs can misdeliver, suppress, or redirect scheduled channel messages.

## Summary

Keep open: current main still has only partial cron isolation through an `agentId` list filter, while trusted caller ownership is missing for mutations, manual runs, all-scope run-history enrichment, and the broader admin/legacy-job contract. The issue discussion includes concrete production leakage/tampering reports, and the related open PR only covers the narrower all-run job-name leak.

## What This Changes

Request agent-scoped cron job storage or equivalent ownership enforcement so agents in multi-user deployments cannot list, edit, delete, run, or inspect other agents' scheduled jobs by default.

## Best Possible Solution

Define one trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create jobs with different agent/session metadata, then call update/remove/run by another job id or request all-scope run history; current main lacks trusted caller ownership checks on those paths.

## Solution Assessment

No for the literal per-agent `jobs.json` proposal by itself. The maintainable fix needs trusted caller-scoped authorization across list, mutation, manual run, history, admin override, and legacy/global-job behavior; storage layout is only one implementation detail.
