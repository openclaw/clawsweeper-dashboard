## Summary

Keep open. Current main still has global cron list, mutation, manual-run, and run-history paths without trusted caller agent/session ownership checks, and the issue discussion includes concrete shared-deployment leakage/tampering reports plus a maintainer comment keeping this work open.

## What This Changes

Request to isolate cron scheduled jobs per agent/session, including scoped storage or equivalent access control, migration handling, and optional admin-wide visibility.

## Best Possible Solution

Add a trusted cron ownership model that enforces default caller agent/session visibility and mutation checks across list, update, remove, run, and runs while preserving explicit admin/operator access and safe legacy handling.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: create jobs with different agentId/sessionKey metadata, then call cron.list or cron.update/remove/run/runs from another agent context; current main has no caller-scope parameter or ownership guard on those paths.

## Solution Assessment

No. Current main does not solve the issue, and a list-only sessionKey filter or storage-only split would be incomplete; the maintainable fix needs trusted enforcement at the tool/Gateway/service boundary with an explicit admin path.
