## Summary

Keep open. Current main only has a partial `cron.list` agentId filter; cron mutation paths still use bare job ids and all-scope run history still reads global job metadata, while the discussion includes concrete production leakage and tampering reports.

## What This Changes

Request per-agent/session isolation for cron scheduled jobs so shared deployments cannot list, modify, run, or inspect other agents' jobs.

## Best Possible Solution

Land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service, storage/run-history paths, CLI/UI/docs/tests, with explicit admin access and safe legacy job handling.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct `agentId` or `sessionKey` metadata, then call `cron.update`, `cron.remove`, `cron.run`, or `cron.runs` all-scope from another caller; current main has no trusted caller ownership guard on those paths.

## Solution Assessment

No, the current list-side agentId filter is only a partial mitigation. The maintainable fix needs trusted caller scope across list, mutation, run, history, admin, and legacy/global job paths rather than only per-agent files or a list-only filter.
