## Summary

Keep open. Current main still exposes cron management through global list, update, remove, run, and run-history paths keyed by job id rather than trusted caller agent/session scope, and the production reports plus maintainer comment point to a real multi-user isolation gap.

## What This Changes

The issue requests agent-scoped cron scheduled-job storage and default access control so agents in shared deployments can only see and manage their own scheduled jobs.

## Best Possible Solution

Add a trusted cron ownership model that derives caller agent/session scope, enforces default visibility and mutation checks across cron.list/update/remove/run/runs, preserves explicit admin/operator access, handles legacy global jobs safely, and updates protocol, service, UI, docs, and tests together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: create jobs with different agentId/sessionKey metadata, then call cron.list or cron.update/remove/run/runs from another agent context; current main has no caller-scope parameter or ownership guard on those paths.

## Solution Assessment

No. Current main does not solve the issue, and a list-only sessionKey filter would leave update, remove, manual run, and run-history access exposed; the safer solution is trusted ownership enforcement at the tool/Gateway/service boundary with an explicit admin bypass.
