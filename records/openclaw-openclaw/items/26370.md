## Summary

Keep open. Current main still leaves the reported cron management surfaces effectively global: jobs can carry agent/session metadata and `cron.add` stamps caller metadata, but list/update/remove/run/runs do not receive or enforce trusted caller ownership. The issue also has concrete production reports and a recent maintainer comment keeping the ownership model open.

## What This Changes

Request per-agent or trusted caller-scope cron isolation so agents in multi-user deployments cannot globally list, update, remove, run, or inspect other agents' scheduled jobs.

## Best Possible Solution

Design a cron ownership model that threads trusted caller agent/session scope into cron management calls, enforces default job visibility and mutation by owner, preserves an explicit admin/operator all-jobs mode, handles legacy global jobs safely, and consolidates the overlapping #35447 discussion without losing this issue's production context.
