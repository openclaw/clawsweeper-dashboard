## Summary

Keep open: current main only partially scopes cron listing by `agentId`, while cron mutations and all-scope run-history enrichment still lack a trusted caller ownership boundary. The issue has concrete production reports of leakage/tampering and current P1/product/security-review labels, so it is not conservative cleanup-close material.

## What This Changes

The issue requests agent-scoped cron job storage and access control so scheduled jobs are visible and mutable only within the caller agent/user scope in shared deployments.

## Best Possible Solution

Define one trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct `agentId` or `sessionKey` metadata, then call `cron.update`, `cron.remove`, `cron.run`, or `cron.runs` with `scope=all` as another caller; current handlers and service methods do not compare the target job owner to trusted caller scope.

## Solution Assessment

No, a per-agent `jobs.json` split or list-only filter is not complete by itself. The maintainable fix needs trusted caller-scoped ownership checks across list, mutation, run, history, admin, and legacy/global-job behavior.
