## Summary

Keep open: current main and v2026.5.20 still have only partial cron list filtering, while trusted caller ownership is missing for cron mutations, manual runs, and all-scope run-history name enrichment. The issue has concrete production reports of cross-user leakage and tampering, and the open related PR only addresses a narrower run-history job-name leak.

## What This Changes

The issue requests per-agent or equivalent caller-scoped cron job storage and CRUD isolation for multi-user deployments.

## Best Possible Solution

Define one cron ownership contract across agent tool calls, Gateway protocol/handlers, service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes. Source inspection on current main shows jobs can carry agentId and sessionKey, but cron.update, cron.remove, cron.run, and all-scope cron.runs still operate without comparing the target job to a trusted caller scope.

## Solution Assessment

No for the literal per-agent jobs.json proposal by itself. The maintainable fix needs trusted caller-scoped authorization across list, mutation, manual run, run history, admin override, and legacy/global-job behavior; storage layout is only one part of that model.
