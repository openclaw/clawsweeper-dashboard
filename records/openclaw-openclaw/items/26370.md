## Label Justifications

- `P1`: The issue reports source-backed cross-user cron job disclosure and unauthorized schedule mutation in real shared deployments.
- `impact:data-loss`: Current id-only remove and update paths can destroy or overwrite another agent's persisted scheduled-job state when a caller has cron access.
- `impact:security`: The central report is a cross-user authorization and metadata-visibility boundary for scheduled jobs.
- `impact:message-loss`: Unauthorized cron edits, removals, or delivery-target changes can suppress or misroute scheduled messages.

## Summary

Keep open: current main still only partially scopes cron listing by agentId, while cron get/update/remove/run/runs remain id-based or globally enriched, and the issue discussion includes concrete production leakage and tampering reports that need maintainer security/product direction.

## What This Changes

Request per-agent/per-session cron scheduled-job isolation so agents can only see, run, update, and remove their own jobs by default.

## Best Possible Solution

Design a trusted cron ownership model over the existing SQLite cron store, derive caller scope from Gateway/tool context, enforce list/get/update/remove/run/runs by default, preserve explicit operator all-jobs behavior, and document legacy unowned-job handling with focused tests.

## Reproduction Assessment

Yes at source level: create jobs with different agentId/sessionKey values, then update/remove/run a foreign job by id or request all-scope run history; current main lacks trusted caller ownership checks on those paths.

## Solution Assessment

No for the literal per-agent jobs.json shape because cron is now SQLite-backed; the right direction is a service/protocol/Gateway ownership boundary with explicit operator behavior.
