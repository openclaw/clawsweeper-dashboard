## Summary

Keep open: current main partially mitigates cron list visibility with an agentId filter, but trusted caller ownership is still missing for mutation and all-scope run-history paths. The issue also has production reports of cross-user leakage/tampering and a recent maintainer-side comment keeping the broader ownership model open.

## What This Changes

Request per-agent/per-session cron job isolation so shared deployments cannot list, inspect, mutate, or delete scheduled jobs across agents by default.

## Best Possible Solution

Land a trusted cron ownership contract across the cron tool, Gateway protocol/handlers, service storage, run-history paths, CLI/UI, docs, and tests, with explicit admin access and safe legacy-job handling.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct agentId/sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs with scope=all as another caller; current main has no trusted caller ownership guard on those paths.

## Solution Assessment

No, the current agentId list filter is only a partial mitigation. The maintainable solution needs caller-scoped ownership checks across list, mutation, run, history, admin, and legacy/global job behavior rather than only per-agent files or a list-only filter.
