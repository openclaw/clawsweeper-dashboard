## Summary

Keep open. Current main still exposes cron management through global list, update, remove, run, and run-history paths keyed only by job id; jobs can carry agent/session metadata, but those fields are not used as trusted caller scope. The production comments and the recent maintainer comment both point to a real multi-user isolation gap, and related #35447 overlaps but does not supersede the richer context here.

## What This Changes

Request per-agent/session isolation for cron job storage and management so multi-user agents cannot view, mutate, run, or inspect each other's scheduled jobs by default.

## Best Possible Solution

Add a maintainer-designed cron ownership model: derive trusted caller agent/session scope from createCronTool or Gateway request context, enforce default visibility and mutation checks across cron.list, cron.update, cron.remove, cron.run, and cron.runs, preserve an explicit admin/operator all-jobs mode, handle legacy global jobs safely, and update protocol, service, UI/CLI, docs, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: create jobs with different agentId/sessionKey values, then call cron.list or cron.update/remove/run/runs from another agent context; current main has no caller-scope parameter or ownership check on those paths, so the service operates over the global store or bare job id.

## Solution Assessment

No. Current main does not solve the issue, and the lightweight sessionKey-only list filter suggested in discussion would not protect update, remove, manual run, or run-history access; the safer direction is a trusted ownership guard at the tool/Gateway/service boundary with an explicit admin bypass.
