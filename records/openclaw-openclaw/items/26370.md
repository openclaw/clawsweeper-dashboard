## Summary

Keep open: current main only has partial list-side cron scoping by agentId, while cron mutations and all-scope run history still lack a trusted caller ownership guard; the discussion includes concrete production leakage and tampering reports.

## What This Changes

Request to isolate scheduled cron jobs per agent/session in shared multi-user deployments, including scoped storage or equivalent access control, migration handling, and optional admin visibility.

## Best Possible Solution

Land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, run-history paths, CLI/UI/docs/tests, with explicit admin access and safe legacy job handling.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct agentId or sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs with scope=all as another caller; current main has no trusted caller ownership guard on those paths.

## Solution Assessment

No, the current list-side agentId filter is only a partial mitigation. The maintainable solution needs caller-scoped ownership checks across list, mutation, run, history, admin, and legacy/global job behavior rather than only per-agent files or a list-only filter.
