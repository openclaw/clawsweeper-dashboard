## Label Justifications

- `P1`: The issue reports a real shared-deployment cron isolation failure where users can see or mutate other users' scheduled jobs.
- `impact:data-loss`: Id-only update and remove paths can overwrite or delete another agent's persisted scheduled-job state when cron access is available.
- `impact:security`: The central report is a cross-user authorization and sensitive metadata visibility boundary for scheduled jobs.
- `impact:message-loss`: Unauthorized cron edits, removals, or delivery-target changes can suppress or misroute scheduled messages.

## Summary

Keep open: current main and v2026.6.10 still only partially scope cron listing by agentId, while get/update/remove/run/runs remain id-based or globally enriched, and the production leakage reports need maintainer security/product direction.

## What This Changes

Request per-agent cron job isolation so agents in multi-user deployments can only view and mutate their own scheduled jobs by default.

## Best Possible Solution

Keep the shared SQLite cron store, but add a maintainer-approved trusted cron ownership boundary across the agent tool, Gateway protocol, service contract, run-history enrichment, docs, and tests, with an explicit operator all-jobs path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create jobs with different agentId/sessionKey values, then get/update/remove/run a foreign job by id or request all-scope run history; current main lacks trusted caller ownership checks on those paths.

## Solution Assessment

No for the literal per-agent jobs.json shape because cron is now SQLite-backed. The best fix is a service/protocol/Gateway ownership model with explicit operator behavior and legacy-job handling.
