## Label Justifications

- `P1`: Production reports describe cross-user cron visibility and mutation in shared deployments, which is an urgent broken agent/channel workflow with security impact.
- `impact:data-loss`: Cross-agent update/remove access can modify or delete another user's scheduled jobs.
- `impact:security`: Cross-tenant cron listing and run-history names can expose job names, content, schedules, and delivery targets.
- `impact:message-loss`: Changing another job's delivery target or ownership can suppress or misroute scheduled messages.

## Summary

Keep open: current main and the latest release only partially address the report with `agentId` list filtering, while trusted caller ownership is still not enforced for cron get/update/remove/run/run-history paths. The issue has concrete production reports of cross-user visibility and tampering, and the related open PR covers only the narrower all-runs job-name leak.

## What This Changes

The issue requests default per-agent/per-session isolation for cron scheduled jobs so one agent cannot list, modify, delete, run, or inspect another agent's jobs in multi-user deployments.

## Best Possible Solution

Keep this open for a core cron ownership model: derive trusted caller agent/session scope server-side, enforce default job visibility and mutation/run-history access by owner, preserve an explicit operator/admin all-jobs mode, handle legacy unowned jobs safely, and update Gateway protocol, CLI/UI, docs, and tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create jobs with different agentId/sessionKey values, then call get/update/remove/run/runs by id or all-scope run history; current main lacks trusted caller ownership checks on those paths.

## Solution Assessment

No. The exact per-agent `jobs.json` shape is no longer the best fit because cron now uses shared SQLite, but the requested isolation should be solved by owner-scoped access control in the canonical cron store plus explicit admin behavior.
