## Label Justifications

- `P1`: The issue reports source-backed cross-user cron job disclosure and unauthorized schedule mutation in multi-user deployments.
- `impact:data-loss`: A caller can still remove or overwrite another agent's scheduled job by id on current main, which can destroy persisted user automation state.
- `impact:security`: The central problem is a cross-tenant visibility and authorization boundary for cron job metadata and mutations.
- `impact:message-loss`: Unauthorized cron edits/removals and cross-app delivery confusion can suppress or misroute scheduled messages.

## Summary

Keep open: current main and v2026.6.6 still only have partial agentId list filtering, while cron mutation and run-history paths are not trusted-caller scoped. The discussion includes concrete production leakage/tampering reports and the item still needs maintainer product and security review before a broad ownership model lands.

## What This Changes

Request per-agent or per-session ownership isolation for cron scheduled jobs so agents can only view and mutate their own schedules by default.

## Best Possible Solution

Design a trusted cron ownership model over the SQLite-backed cron store, deriving caller scope from Gateway/tool context, enforcing list/get/update/remove/run/runs by default, preserving explicit operator/admin all-jobs behavior, and documenting legacy unowned-job handling with focused tests.

## Reproduction Assessment

Yes at source level. Create jobs with different agentId/sessionKey values: cron.list receives an agentId filter from the tool, but update/remove/run still operate by bare id and cron.runs(scope="all") builds names from an unfiltered list.

## Solution Assessment

No for the literal per-agent jobs.json storage shape, because current cron state is SQLite-backed. The access-control goal is the right direction, but the maintainable solution belongs in the existing service/protocol/Gateway ownership boundary with an explicit admin mode.
