## Summary

Keep open. Current main still exposes cron management through global list, update, remove, run, and run-history paths keyed by job id rather than trusted caller agent/session scope; the production reports and recent maintainer comment make this a real shared-deployment isolation gap, not a stale duplicate of #35447.

## What This Changes

The issue requests per-agent cron job isolation so multi-user deployments cannot list or mutate other agents’ scheduled jobs.

## Best Possible Solution

Add a trusted cron ownership model that derives caller agent/session scope, enforces default visibility and mutation checks across cron.list/update/remove/run/runs, preserves explicit admin/operator access, handles legacy global jobs safely, and updates protocol, service, UI, docs, and tests together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: create jobs with different agentId/sessionKey metadata, then call cron.list or cron.update/remove/run/runs from another agent context; current main has no caller-scope parameter or ownership guard on those paths.

## Solution Assessment

No. Current main does not solve the issue, and a list-only sessionKey filter would be incomplete; the safer path is trusted ownership enforcement at the tool/Gateway/service boundary with an explicit admin bypass.
