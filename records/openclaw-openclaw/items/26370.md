## Label Justifications

- `P1`: The issue reports source-backed cross-user cron job disclosure and unauthorized schedule mutation in real shared deployments.
- `impact:data-loss`: A caller can still remove or overwrite another agent's scheduled job by id on current main, which can destroy persisted user automation state.

- `impact:message-loss`: Unauthorized cron edits, removals, or delivery-target changes can suppress or misroute scheduled messages.

## Summary

Keep open: current main still has only partial list-side agent filtering, while cron mutation, manual-run, and run-history paths are not scoped by trusted caller ownership. The issue has concrete production leakage/tampering reports, and the remaining fix needs maintainer security/product direction because current cron state is shared SQLite rather than per-agent jobs.json.

## What This Changes

The issue requests per-agent or otherwise caller-scoped cron job storage and access control so users in multi-agent deployments cannot see or mutate other users' scheduled jobs.

## Best Possible Solution

Design a trusted cron ownership model over the SQLite-backed cron store, deriving caller scope from Gateway/tool context, enforcing list/get/update/remove/run/runs by default, preserving explicit operator all-jobs behavior, and documenting legacy unowned-job handling with focused tests.

## Reproduction Assessment

Yes at source level. Create jobs with different agentId/sessionKey values: cron.list can be agentId-filtered, but update/remove/run still operate by bare id and cron.runs(scope="all") builds names from an unfiltered list.

## Solution Assessment

No for the literal per-agent jobs.json storage shape because current cron state is SQLite-backed. The access-control goal is the right direction, but the maintainable solution belongs in the existing service/protocol/Gateway ownership boundary with an explicit operator mode.
