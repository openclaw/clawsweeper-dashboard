## Summary

Keep open: current main only partially mitigates cron visibility with an agentId list filter, while trusted caller ownership is still missing for list overrides, mutations, manual runs, and all-scope run-history enrichment. The discussion includes production leakage/tampering reports and active product/security labels, so this is not conservative cleanup-close material.

## What This Changes

Request caller-scoped cron job isolation so shared multi-agent deployments cannot expose, modify, delete, or manually run another agent or user's scheduled jobs.

## Best Possible Solution

Define one trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: create jobs with distinct agentId/sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs(scope=all) as another caller; current handlers and service methods do not compare target job ownership to trusted caller scope.

## Solution Assessment

No for the literal per-agent jobs.json proposal by itself. The maintainable fix needs trusted caller-scoped authorization across list, mutation, run, history, admin override, and legacy/global-job behavior; storage layout is only one part of that model.
