## Summary

Keep open: current main has only partial agentId list filtering, while cron mutation and all-scope run-history paths still lack trusted caller ownership checks. The issue has concrete production reports, active security/product-review labels, and a narrower open follow-up for one name-leak subset, so this is not cleanup-close material.

## What This Changes

Request per-agent or per-session isolation for scheduled cron jobs so agents cannot view, modify, delete, or run jobs owned by other agents or users.

## Best Possible Solution

Define and land a trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI/docs, and tests, with explicit admin access and safe legacy-job handling.

## Reproduction Assessment

Yes, at source level. Create jobs with distinct agentId/sessionKey metadata, then call cron.update, cron.remove, cron.run, or cron.runs with scope=all as another caller; current handlers and service methods do not compare the job owner to a trusted caller scope.

## Solution Assessment

No, a per-agent jobs.json split or list-only filter is not the complete fix. The maintainable solution needs trusted caller-scoped ownership checks across list, mutation, run, history, admin, and legacy/global-job behavior.
