## Summary

Keep open: current main only partially scopes `cron.list` by a supplied `agentId`, while cron mutation, manual run, and all-scope run-history paths still lack trusted caller ownership enforcement. The discussion includes concrete production reports of cross-user cron visibility and tampering, and the related open PR only addresses a narrower run-history name leak.

## What This Changes

The issue requests agent-scoped cron job storage and access control so one agent or user cannot list, modify, delete, or run another agent's scheduled jobs in shared deployments.

## Best Possible Solution

Define one cron ownership contract across agent tool calls, Gateway protocol/handlers, service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes. Source inspection on current main shows jobs carry `agentId`/`sessionKey`, but update, remove, manual run, and all-scope run-history paths still operate without comparing the target job to trusted caller scope.

## Solution Assessment

No for the literal per-agent `jobs.json` proposal by itself. The maintainable fix needs caller-scoped authorization across list, mutation, manual run, run history, admin override, and legacy/global-job behavior; storage layout is only one part of that model.
