## Label Justifications

- `P1`: Shared cron ownership gaps can expose or modify scheduled jobs in real multi-user deployments.
- `impact:data-loss`: The reported behavior includes unauthorized deletion or mutation of persisted scheduled job data.
- `impact:security`: The item is about cross-agent visibility and modification across tenant/user boundaries.
- `impact:message-loss`: Tampered or mis-scoped scheduled jobs can misdeliver or suppress scheduled channel messages.

## Summary

Keep open: current main has only partial cron visibility mitigation through an `agentId` list filter, while trusted caller ownership is still missing for list overrides, mutations, manual runs, and all-scope run-history enrichment. The discussion includes production leakage/tampering reports and an open narrow follow-up PR that would not close the broader ownership model by itself.

## What This Changes

Request isolated cron scheduled-job ownership for multi-user deployments, including per-agent storage or equivalent access control, safe migration, and optional admin visibility.

## Best Possible Solution

Define one trusted cron ownership contract across the agent tool, Gateway protocol/handlers, cron service/storage, CLI/UI/docs, and tests, with explicit admin access and safe handling for existing global jobs.

## Reproduction Assessment

Yes. Source inspection shows jobs can carry agent/session metadata, but `cron.update`, `cron.remove`, `cron.run`, and `cron.runs(scope=all)` still operate without comparing target job ownership to a trusted caller scope.

## Solution Assessment

No for the literal per-agent `jobs.json` proposal by itself. The maintainable fix needs trusted caller-scoped authorization across list, mutation, run, history, admin override, and legacy/global-job behavior; storage layout is only one part of that model.
