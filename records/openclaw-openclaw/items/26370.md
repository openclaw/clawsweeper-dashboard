## Label Justifications

- `P1`: Production reports and source inspection show cross-user cron visibility and mutation gaps in shared agent/channel workflows.
- `impact:data-loss`: Unscoped update/remove access can modify or delete another user's scheduled jobs.
- `impact:security`: The issue is about cross-tenant exposure and unauthorized mutation of scheduled job metadata and delivery targets.
- `impact:message-loss`: Changing another job's delivery target or ownership can suppress or misroute scheduled messages.

## Summary

Keep open: current main and v2026.6.5 only partially address the report with optional `agentId` list filtering, while trusted caller ownership is still missing for get/update/remove/run/run-history paths. The closed broad PR is unmerged, and the open run-history PR is narrower and currently conflicting, so neither supersedes this canonical security issue.

## What This Changes

Request per-agent cron job isolation so multi-user deployments cannot list, modify, delete, or run scheduled jobs owned by other agents/users.

## Best Possible Solution

Keep this issue open as the canonical tracker for a SQLite-backed cron ownership model that enforces default owner-scoped visibility and id-based operations, with explicit admin behavior and migration handling for legacy jobs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create jobs under different `agentId`/`sessionKey` values, then call `cron.get`, `cron.update`, `cron.remove`, `cron.run`, or `cron.runs` by id from another agent path; current main does not enforce caller ownership on those operations.

## Solution Assessment

No for the literal per-agent `jobs.json` proposal. Current docs and repository policy indicate cron state is now shared SQLite, so the maintainable solution is ownership enforcement in the canonical cron store, Gateway protocol, and agent tool contract with an explicit admin mode.
