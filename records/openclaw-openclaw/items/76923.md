## Summary

Keep open: the PR is useful and follows existing restart deferral/coalescing machinery, but the new safe-restart RPC bypasses the current `commands.restart=false` restart-disable contract and the user-facing change is missing a changelog entry.

## What This Changes

The PR adds a safe Gateway restart coordinator, `gateway.restart.preflight`/`gateway.restart.request` RPC methods, `--safe` restart CLI handling, tests, and CLI docs.

## Best Possible Solution

Preserve the existing restart-disable contract in the new RPC, add focused regression coverage and a changelog entry, then rerun the targeted gateway/daemon restart checks.

## Reproduction Assessment

Yes. From source inspection, a Gateway with `commands.restart=false` would still accept the new admin-scoped `gateway.restart.request` handler and schedule a restart because the handler does not check the existing restart-enabled policy.

## Solution Assessment

No, not as written. The coordinator should keep using the existing deferral/coalescing path, but the RPC entry point needs the same restart-disabled guard used by `/restart` and the gateway tool before it schedules anything.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate safe restart requests with the restart flag:** `src/gateway/server-methods/restart.ts:11-15`
  - body: The new `gateway.restart.request` method schedules a SIGUSR1 restart unconditionally. Existing manual restart surfaces reject when `commands.restart=false`, and the docs define that flag as disabling gateway restart tool actions, so an admin-scoped RPC can now bypass the configured restart-disable policy. Please check the running config before calling `requestSafeGatewayRestart` and return an error when restarts are disabled.
