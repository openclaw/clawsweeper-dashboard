## Summary

Keep this PR open; the direction is useful, but the safe restart path is not merge-ready because the new preflight can report a restart as safe while the scheduler still defers on the older queued-task count. It also needs the required changelog entry for a user-facing CLI/RPC change.

## What This Changes

Adds `--safe` gateway/daemon restart handling, restart preflight/request RPC methods, a safe restart coordinator, docs, and tests around active restart blockers.

## Best Possible Solution

Make safe preflight, scheduler deferral, and shutdown drain share the same blocker definition, add regression coverage for queued and stale-ended task records, and add a `CHANGELOG.md` entry for the new CLI/RPC behavior.

## Reproduction Assessment

Yes, by source inspection. A queued task record makes the PR's preflight safe because queued records are excluded, but the existing scheduler still defers via `getInspectableTaskRegistrySummary().active`, which counts queued records.

## Solution Assessment

No. The coordinator shape is reasonable, but the implementation should first align the actual restart deferral check with the new preflight semantics and include the required changelog entry.
