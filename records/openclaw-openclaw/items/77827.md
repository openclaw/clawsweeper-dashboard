## Summary

Keep open: the LM Studio/OpenAI-compatible Responses gap is valid and the PR has useful real behavior proof, but the patch can duplicate or reindex reasoning blocks when `response.reasoning_text.done` appears with a normal Responses output item.

## What This Changes

The PR adds a `response.reasoning_text.done` branch in `src/agents/openai-transport-stream.ts` that turns `event.text` into a thinking block and emits thinking stream events.

## Best Possible Solution

Handle reasoning text events through a stable reasoning block keyed by the Responses item/content metadata, merge with an active reasoning item when present, avoid retroactively changing emitted block indexes, and add regression coverage for both LM Studio done-only and standard item-associated streams.

## Reproduction Assessment

Yes. The PR body gives a concrete LM Studio setup with before/after screenshots, and current main has no handler for the upstream `response.reasoning_text.done` stream event; I did not rerun LM Studio locally.

## Solution Assessment

No. The requested direction is the right bug fix, but this implementation should merge the done event into the existing Responses item flow instead of unshifting a standalone block.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Merge reasoning_text.done with the active reasoning block:** `src/agents/openai-transport-stream.ts:500`
  - body: The OpenAI stream event is tied to an item/content index, but this branch always creates a new block at the front. If a provider emits the normal `output_item.added` reasoning item before `response.reasoning_text.done`, the original `currentBlock` remains open and `output_item.done` can emit a second, often empty, thinking block; if text has already started, unshifting also changes previously emitted block indexes. Track or reuse the reasoning block for the event's item instead of inserting a standalone block.
  - confidence: 0.88
