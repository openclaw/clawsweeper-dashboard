## Summary

Keep open: the patch is a plausible collision mitigation, but it changes the persisted bridge page path for every existing bridge artifact and does not remove the old page, so upgraded vaults can retain stale duplicate source pages; the supplied proof is also code-inspection only, not a real multi-workspace bridge run.

## What This Changes

This PR changes memory-wiki bridge artifact slug hashing so the hash input includes `workspaceDir` as well as `relativePath`.

## Best Possible Solution

Make the bridge path change upgrade-safe by deleting or migrating the previous tracked page path when an active source's derived page path changes, and prove the two-workspace same-relative-path case with a regression test plus real behavior proof.

## Reproduction Assessment

Unclear: source inspection confirms the bridge imports active memory artifacts across configured workspaces and currently hashes `relativePath` for `artifactSlug`, but this review did not run a live two-workspace bridge repro and current filenames already include `workspaceSlug`.

## Solution Assessment

No: including `workspaceDir` in the hash is a plausible mitigation, but the current patch is not upgrade-safe because it renames all bridge source pages without cleaning up the previous tracked path.
