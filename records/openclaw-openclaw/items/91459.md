## Summary

This PR should stay open: the hash change is a plausible mitigation, but it changes deterministic bridge source page paths without cleaning up old active pages, and the supplied proof is code inspection rather than a real multi-workspace bridge import.

## What This Changes

The PR changes memory-wiki bridge artifact slug hashing so `workspaceDir` is included with `relativePath`.

## Best Possible Solution

Keep the workspace-aware hash only if the PR also removes the previous bridge page path for the same active sync key and proves the two-workspace import path with focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? ## Solution Assessment

Is this the best way to solve the issue? No, not yet; adding workspace to the artifact hash is plausible, but the best fix must also handle existing source-sync/page-path cleanup and prove the collision path with a focused regression or live import proof.
