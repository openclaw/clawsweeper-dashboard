## Summary

Close: current main fixes the stale WhatsApp runtime-config boundary that could accept/log a group message on v2026.5.7 and then continue dispatch-adjacent work with startup config. The related https://github.com/openclaw/openclaw/issues/80704 was closed on the same current-main fix; this is fixed on main but not yet in a shipped release.

## What This Changes

The issue reports that WhatsApp group messages received after upgrading to v2026.5.7 are logged by the gateway but not dispatched to the configured agent.

## Best Possible Solution

Keep the current-main hot-reload plumbing and ship it in the next release; reopen only if a fresh current-main WhatsApp group flow still logs inbound messages without an agent turn.

## Reproduction Assessment

Yes for the reported release regression by source comparison: v2026.5.7 used startup config past the fresh route lookup, while current main passes a per-message config through gating and dispatch. I did not run a live WhatsApp group flow.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it refreshes and carries the existing runtime config at the inbound boundary instead of adding a restart-only policy or a new config option.
