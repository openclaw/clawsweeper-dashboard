## What This Changes

The issue asks for Windows gateway scheduled-task installation to launch through a windowless helper instead of running `gateway.cmd` directly.

## Best Possible Solution

Land a narrow fix equivalent to the closed direct PR: default gateway service installs to the existing hidden VBS launcher and add focused gateway env/install coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: gateway service env does not enable the hidden launcher, while schtasks only switches to `.vbs` when that env value is truthy. I did not run a live Windows desktop repro in this read-only review.

## Solution Assessment

Yes: reusing the existing VBS hidden launcher as the default for gateway Scheduled Tasks is the narrowest maintainable solution. Adding a second launcher mechanism or broad Windows launch refactor is unnecessary for this issue.
