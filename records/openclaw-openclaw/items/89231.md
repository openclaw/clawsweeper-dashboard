## Summary

Keep open: current main still defaults the Windows Gateway Scheduled Task to the `gateway.cmd` action, while the existing VBS launcher is only used when an internal hidden-launcher env flag is set. The related open heartbeat issue is the same console-window class, but it is a different launch path and does not replace this scheduled-task default bug.

## What This Changes

Make the Windows installer-created `OpenClaw Gateway` Scheduled Task use a windowless launcher instead of launching `gateway.cmd` directly.

## Best Possible Solution

Use the existing VBS hidden-launcher path as the default for the Gateway Windows Scheduled Task, cover fresh install and existing-task update behavior, and keep the heartbeat-console work tracked separately.

## Reproduction Assessment

Yes, source-level: the normal gateway service env does not set the hidden-launcher flag, so Scheduled Task creation resolves to `gateway.cmd`. I did not run the Windows desktop UI path live.

## Solution Assessment

Yes, the narrowest maintainable fix is to reuse the existing `gateway.vbs` hidden launcher for gateway tasks by default and update the daemon tests. Adding a new user config knob would be unnecessary because the hidden launcher already exists.
