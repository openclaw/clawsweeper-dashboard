## Summary

Keep open: this PR is not merge-ready because it makes an undefined wildcard channel config tier active at runtime, lacks after-fix live channel proof, and only falls back when the whole channel object is missing rather than when `ackReaction` is missing.

## What This Changes

This PR changes `resolveAckReaction()` to look up `channels["*"].ackReaction` when no concrete channel config object exists.

## Best Possible Solution

Define or reject `channels["*"]` as one coherent wildcard config contract across validation, schema/docs, runtime precedence, sibling scanners, tests, and live proof; if accepted, make fallback field-level while preserving explicit empty-string suppression.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.8` have no wildcard lookup in `resolveAckReaction()`, and current validation rejects `channels["*"]`. Live runtime proof is blocked until the broader wildcard config contract is accepted and implemented.

## Solution Assessment

No. The standalone lookup is too narrow and too early; the maintainable fix is a contract-level change that defines wildcard validation/docs/scanners and falls back when `ackReaction` is missing, not only when the channel object is missing.
