## Summary

Keep open: current main still lacks the runtime wildcard lookup, but this PR is not merge-ready as a standalone change because it creates a new `channels["*"]` config tier without validation/docs/test contract or real behavior proof.

## What This Changes

The PR changes `resolveAckReaction()` to use `channels["*"].ackReaction` when the concrete channel config lookup returns no channel object.

## Best Possible Solution

Define `channels["*"]` as one explicit wildcard config contract across runtime, validation, docs, schema/scanners, tests, and live proof, or keep `messages.ackReaction` as the supported global fallback.

## Reproduction Assessment

Yes, at source level: current main has no wildcard lookup in `resolveAckReaction`, and current validation rejects `channels["*"]` as an unknown channel id. I did not mark live reproduction because the real config path is blocked without a broader contract change.

## Solution Assessment

No, not as a standalone branch. The narrow lookup is only acceptable after maintainers define or reject `channels["*"]`, and the fallback should be based on a missing `ackReaction` field rather than only a missing channel object if wildcard defaults are accepted.
