## Summary

Keep this PR open: it addresses a real config-loss path on current main and the general fix direction is sound, but the patch leaves an ordinary onboard writer with the same size-drop bypass and the contributor proof is still test-harness-only.

## What This Changes

The PR changes onboard config writes to stop allowing size-drop writes by default, preserves existing `agents.list` and `bindings` in local onboard reruns, and adds schema/tests for dangling binding agent IDs.

## Best Possible Solution

Land a complete guard where every ordinary onboard rerun preserves existing agents and bindings, shrink bypasses are limited to explicit reset/import flows, and dangling binding agent IDs fail validation compatibly.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main hardcodes `allowConfigSizeDrop: true` in ordinary onboard writes while the config writer only blocks `size-drop:*` when that flag is absent. I did not run the CLI because this review is read-only, so the status is source-reproducible rather than locally reproduced.

## Solution Assessment

Not yet. The proposed direction is the right owner-boundary fix, but it should apply the same shrink policy to non-interactive remote onboarding and include real behavior proof before merge.
