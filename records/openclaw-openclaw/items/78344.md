## Label Justifications

- `P1`: The WebChat restart path can turn a documented continuation into a visible system event and leave the intended follow-up agent turn unanswered.
- `impact:message-loss`: The continuation intended to resume the agent can be converted into a system-event wake rather than a dispatched agent turn, causing no assistant response.
- `impact:session-state`: The issue concerns runtime continuation state crossing the wrong audience/provenance boundary into visible chat/session history.

## Summary

Keep open. Current main still routes `continuationMessage` through an `agentTurn` only when a routable `channel` plus `to` exists; WebChat/session-only restart continuations can still fall back to `System:` wake events instead of an internal agent turn.

## What This Changes

The issue asks Control UI/WebChat to treat restart/runtime continuation messages as agent-internal state or clearly separate runtime/system content rather than ordinary visible user chat.

## Best Possible Solution

Define a first-class restart/runtime continuation contract: WebChat/session-only `agentTurn` continuations should dispatch internally by session key, while intentionally visible runtime notices render outside user-authored chat.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a WebChat/session-only restart continuation without `to` makes the route undefined, then the queued `agentTurn` falls back to a `System:` wake. I did not rerun a live WebChat restart in this read-only review.

## Solution Assessment

Unclear as a final implementation. The issue’s direction is valid, but the best fix is not a UI-only hide or badge change; it should first settle session-only restart dispatch plus runtime audience/provenance semantics.
