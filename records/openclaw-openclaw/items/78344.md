## Label Justifications

- `P1`: A restart/recovery workflow can expose internal continuation text and leave the intended WebChat follow-up unanswered.
- `impact:message-loss`: The no-route fallback can prevent the continuation prompt from dispatching as the resumed agent turn.
- `impact:session-state`: Continuation prompts are restart/session state that can cross into visible transcript state with the wrong audience.

## Summary

Keep open. Current main and the latest release still route WebChat restart continuations without a deliverable route through the system-event wake path, so the continuation can remain visible as `System:` text and fail to run as a hidden agent turn.

## What This Changes

The issue requests that Control UI/WebChat continuation prompts and runtime handoff messages be delivered as agent-only introspection or clearly separate runtime/system events instead of normal visible user chat.

## Best Possible Solution

Define the WebChat restart-continuation path as an internal agent turn keyed by session, and render only intentional runtime notices through a distinct system/audience surface.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: a WebChat continuation cannot resolve a deliverable route, then the queued `agentTurn` falls back to a system-event wake that formats as `System:` text; I did not run a live WebChat restart scenario.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the audience contract, but the current source path is not sufficient; the safer fix is gateway/session-key dispatch plus distinct visible runtime treatment, not a display-only filter.
