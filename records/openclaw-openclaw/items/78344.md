## Summary

Keep open. Current main still has a source-backed path where restart `agentTurn` continuations without a routable delivery target fall back to queued system events, and WebChat history projection does not currently treat `internal_system` user-role provenance as hidden or diagnostic-only; the open recovery-notice PR is adjacent, not a full implementation of this request.

## What This Changes

The issue requests that Control UI/WebChat continuation and restart-recovery messages be treated as agent/runtime-internal state rather than rendered as ordinary visible user chat.

## Best Possible Solution

Define one core continuation audience and routing contract: non-deliverable WebChat restart continuations should dispatch as agent-only work by session key, while any intentionally visible runtime notice should render through a distinct diagnostic/system treatment rather than a normal user bubble.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: an `agentTurn` restart continuation with no resolved `to` gets no route, falls back to `enqueueRestartSentinelWake`, and is then formatted as a `System:` event. I did not run a live WebChat restart in this read-only review.

## Solution Assessment

No, a UI-only treatment is not the best fix because it can hide the unanswered-continuation path. The safer solution needs both the session-key dispatch behavior for WebChat continuations and a product decision for hidden versus diagnostic rendering of runtime continuations.
