## Label Justifications

- `P1`: The issue describes a restart/recovery workflow where internal continuation text can appear in user-visible WebChat and the intended agent continuation may not run.
- `impact:session-state`: The reported continuation prompt is session/recovery state that can be misrepresented in the visible transcript.
- `impact:message-loss`: The related restart test reports the continuation being enqueued as a system event without producing the intended assistant follow-up.

## Summary

Keep open. Current main still has a source-supported gap: restart continuations only dispatch as routed agent turns when a delivery target exists, while WebChat-style sessions can lack `to`; the fallback is a queued `System:` event, and the display projection does not generally hide or re-lane `internal_system` continuation inputs.

## What This Changes

The issue asks Control UI/WebChat to stop rendering runtime continuation messages as normal human-authored chat and to route continuation prompts as agent-only/internal state or a clearly distinct system lane.

## Best Possible Solution

Use a structured internal-event/provenance lane for restart continuations, including a WebChat session-only agent-turn path for sessions without `to`, then keep truly internal continuation text out of visible history or render intentional notices as a distinct runtime block.

## Reproduction Assessment

Yes, at source level: current restart continuation routing requires a routable `to`, and current chat projection has no generic `internal_system` hiding or distinct-lane rule. I did not run a live WebChat restart reproduction in this read-only review.

## Solution Assessment

No for a UI-only fix; the safer solution must combine runtime dispatch semantics with transcript projection/audience handling so WebChat continuations both wake the agent and avoid false user attribution.
