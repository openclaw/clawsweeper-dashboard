## Label Justifications

- `P1`: The reported WebChat restart path can leave a documented continuation unanswered and show runtime text in the transcript, breaking a real agent/channel recovery workflow.
- `impact:message-loss`: The continuation intended to resume the agent can be converted into a system-event wake instead of a dispatched agent turn, leaving no assistant response.
- `impact:session-state`: The issue concerns runtime continuation state being projected into visible chat with the wrong audience/provenance boundary.

## Summary

Keep open. Current main and the latest release still document `continuationMessage` as a one-turn restart continuation, but WebChat/internal sessions without a routable `to` fall back to queued `System:` events instead of an agent-turn continuation; the broader open trackers do not replace this restart-specific dispatch and audience problem.

## What This Changes

The issue asks Control UI/WebChat to treat restart and runtime continuation prompts as agent-internal continuation context rather than ordinary visible user chat.

## Best Possible Solution

Define a first-class restart/runtime continuation contract: WebChat/internal `agentTurn` continuations should dispatch as session-only agent turns with internal provenance, while intentionally visible runtime notices render outside user-authored chat.

## Reproduction Assessment

Yes, from source with high confidence. A WebChat/internal session without a routable `to` makes `resolveRestartContinuationRoute` return `undefined`, and the queued `agentTurn` path falls back to `enqueueSystemEvent`; a live WebChat restart was reported but not rerun here.

## Solution Assessment

Unclear as a final implementation. The issue's direction is valid, but the best fix is not a UI-only hide or badge change; it should first settle session-only restart dispatch plus runtime audience/provenance semantics.
