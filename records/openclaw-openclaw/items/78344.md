## Label Justifications

- `P1`: The issue affects restart recovery in WebChat by showing internal continuation text and potentially failing to resume the agent turn.
- `impact:session-state`: Continuation text is session/runtime state that can be projected as ordinary conversation state.
- `impact:message-loss`: The WebChat no-route path can leave the intended continuation unanswered instead of dispatching a resumed agent turn.

## Summary

Keep open. PR #88161 shipped a partial routed-continuation fix, but current main still falls back to a system-event wake for agentTurn continuations without a route, matching the WebChat/no-to follow-up rather than proving this issue fully fixed.

## What This Changes

The issue requests that Control UI/WebChat continuationMessage and runtime continuation or handoff messages be treated as agent-internal state instead of visible user-authored chat.

## Best Possible Solution

Give restart continuations an internal agent-turn dispatch path keyed by session when no external route exists, and render any intentional runtime notice through a distinct system/audience surface.

## Reproduction Assessment

Yes, source-reproducible: current main still falls back from no-route agentTurn continuations to a system-event wake, and the issue comment reports that exact WebChat restart shape.

## Solution Assessment

No, the current shipped fix is only partial: routed continuations are internal/message-tool-only, but no-route WebChat continuations are not proven to dispatch as hidden internal agent turns.
