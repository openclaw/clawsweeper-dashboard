## Summary

Keep open: current main still turns restart/update continuations into an inbound agent turn with the continuation text as `Body`/`RawBody`. The turn now carries `InputProvenance.kind: "internal_system"`, but the current WebChat display projection does not hide or specially render that provenance, so the central visible-chat concern is not resolved.

## What This Changes

The issue asks Control UI/WebChat to treat `continuationMessage` and similar runtime continuations as agent-only or clearly system-originated content, not ordinary human-authored chat.

## Best Possible Solution

Use the existing internal-system provenance or a narrow continuation audience contract so the agent receives the continuation while visible history either suppresses it or renders it through the shared runtime/system block.

## Reproduction Assessment

Yes, at source level. A restart or update sentinel with `continuationMessage` and a routable session becomes an `agentTurn` whose continuation text is dispatched as `Body`/`RawBody`, and current display projection has no `internal_system` handling.

## Solution Assessment

No, the current implementation is not the best solution because it labels the turn as internal only in provenance while still carrying the raw continuation through the visible user-message body. The safer solution is a provenance/audience-aware transcript/display boundary shared with the related runtime-system-message work.
