## Label Justifications

- `P1`: The issue affects active WebChat restart/recovery continuations and can leave the intended agent turn unanswered.
- `impact:session-state`: Runtime continuation state can leak into visible transcript state instead of remaining internal to the agent/session.
- `impact:message-loss`: The related WebChat restart observation shows the continuation can be routed as a system event and fail to wake an agent response.

## Summary

Keep open. Current main still has a source-backed path where WebChat restart continuations with no routable `to` fall back to session-scoped system events, while the broader agent-only audience/display behavior requested here remains unresolved.

## What This Changes

Control UI/WebChat should treat `continuationMessage` and runtime continuation prompts as agent-only runtime/introspection input instead of visible user-authored chat.

## Best Possible Solution

Define the restart-continuation audience and route contract so non-deliverable WebChat sessions dispatch the continuation as agent-only work by session key, then render any intentionally visible runtime notices through a distinct diagnostic/system treatment.

## Reproduction Assessment

Yes. Source inspection shows that an agent-turn continuation without a routable delivery target falls back to the system-event wake path, and existing test coverage expects that fallback instead of WebChat agent dispatch.

## Solution Assessment

No. A UI-only treatment is not the best fix because it can hide that WebChat continuations did not wake the agent; dispatch and audience/display semantics need to be settled together.
