## Label Justifications

- `P1`: The report concerns WebChat restart/recovery continuations being misrouted or visibly misattributed, which can break an active agent/channel workflow.
- `impact:session-state`: The issue is about restart/recovery continuation state being treated as visible transcript content instead of internal runtime state.
- `impact:message-loss`: The related WebChat restart observation shows the continuation can surface as a system event and fail to wake the intended agent turn.

## Summary

Keep open. Current main and the latest release still fall back from an unroutable restart continuation to a session-scoped system-event wake, while the WebChat-specific agent-only continuation/audience behavior requested here remains unresolved.

## What This Changes

The issue requests that WebChat restart continuation and runtime handoff messages be treated as agent-only/internal runtime context, or clearly separated system UI, instead of visible human-authored chat.

## Best Possible Solution

Define the restart-continuation audience/route contract so non-deliverable WebChat sessions dispatch the continuation as an agent-only turn by session key, and render any intentionally visible runtime notices through a distinct shared system/diagnostic treatment.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: an agent-turn continuation with no resolved route falls back to `enqueueRestartSentinelWake`, and the test suite currently expects system-event enqueue instead of agent dispatch. I did not run a live WebChat restart in this read-only pass.

## Solution Assessment

No, a simple UI-only treatment is not the best fix. The safer shape is to fix WebChat restart-continuation dispatch as agent-only internal work first, then apply a shared audience/display policy for runtime notices that are intentionally visible.
