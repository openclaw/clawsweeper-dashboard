## Summary

Keep open. Current main still clears self-declared scopes for device-less token/password WebSocket operators outside the narrow local backend and Control UI exceptions, and the open related PR is an unmerged security-sensitive policy change rather than shipped proof.

## What This Changes

Requests Docker/LAN token-authenticated WebSocket operator connections from an A2A plugin to retain `operator.admin`/`operator.write` scopes like loopback connections.

## Best Possible Solution

Define the intended Docker/A2A WebSocket trust contract first, then encode any approved exception in the central connect policy with regression coverage for handshake negotiation and an `operator.write` RPC.

## Reproduction Assessment

Yes, from source inspection and current tests: a device-less token-auth WebSocket operator outside the reserved local-backend path has its declared scopes cleared, and `agent` then requires `operator.write`. I did not run the Docker/A2A setup live in this read-only pass.

## Solution Assessment

No, the issue's suggested unconditional LAN token scope grant is not clearly the best fix because current docs/tests reserve device-less omission to narrower trusted paths. A maintainer-approved pairing, device-token, or explicitly trusted backend contract would be safer.
