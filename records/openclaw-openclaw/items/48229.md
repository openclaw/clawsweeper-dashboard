## Summary

Keep open. Current main and v2026.5.20 still clear self-declared scopes for device-less shared-token WebSocket operators, checked-in tests assert the missing-scope failure, and the open fix paths still need maintainer/security policy approval.

## What This Changes

The issue asks for Docker/LAN WebSocket operator sessions authenticated by gateway token to retain requested operator.write/admin scopes instead of connecting with cleared scopes and failing write RPCs.

## Best Possible Solution

Land one maintainer-approved WebSocket shared-token scope contract, likely the bounded gateway.auth.tokenScopes path with privileged opt-in, docs, regression tests, and redacted Docker/LAN proof.

## Reproduction Assessment

Yes, at source level. Current main accepts the relevant device-less shared-token operator path with cleared scopes, and checked-in tests expect a follow-up scoped RPC to fail with missing scope.

## Solution Assessment

No for the blanket requested behavior. The safer maintainable solution is an explicit configured token-scope or device-identity policy rather than trusting remote client-declared scopes by default.
