## Summary

Keep open. Current main and the latest release still clear self-declared scopes for device-less shared-token WebSocket operator sessions outside narrow trusted paths, and the remaining work is part of an active gateway auth-policy cluster with open PRs that need maintainer/security judgment.

## What This Changes

The issue requests that token-authenticated WebSocket operator clients retain `operator.write` when `gateway.bind` is `lan`, including when `operator.admin` is requested.

## Best Possible Solution

Keep the issue open while maintainers settle the WebSocket auth trust boundary, then either land a scoped fix with regression coverage or document that external A2A/custom WS clients must use `GatewayClient` with device identity and challenge signing.

## Reproduction Assessment

Yes, from source and checked-in tests. Current main connects a valid shared-token operator without device identity, clears its scopes, and expects a scope-gated RPC to fail with `missing scope`; I did not run a live Docker reproduction because this review is read-only.

## Solution Assessment

Unclear. Preserving scopes would address the report, but current docs/tests intentionally limit device-less scope preservation to narrow trust paths, so the maintainable fix needs an explicit maintainer-approved security policy.
