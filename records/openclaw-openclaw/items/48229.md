## Summary

Keep open. Current main and the latest release still clear scopes for device-less shared-token WebSocket operator sessions outside narrow trusted paths, and open related PRs now carry the remaining security/product decision instead of making this issue obsolete.

## What This Changes

The issue asks for LAN-bound, token-authenticated WebSocket operator clients to retain `operator.write`/admin-equivalent scopes instead of connecting with cleared scopes.

## Best Possible Solution

Land a maintainer-reviewed gateway auth policy, likely an explicit token-scope/configured trust path or a documented device-identity requirement, with source, tests, and protocol docs aligned.

## Reproduction Assessment

Yes, from current source and checked-in tests. Current main accepts a valid shared-token operator without device identity, clears its scopes, and expects a scope-gated RPC to fail with `missing scope`; I did not run a live Docker reproduction in this read-only review.

## Solution Assessment

Unclear. Preserving all requested scopes would address this report, but it changes the remote device-less operator trust boundary, so a reviewed explicit policy/config path is safer than a blanket LAN token exception.
