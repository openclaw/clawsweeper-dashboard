## Summary

Keep open. Current main and v2026.5.20 still clear scopes for device-less shared-token/password WebSocket operator sessions, and the linked fix paths are still open because the remaining question is an auth/security policy decision.

## What This Changes

The issue requests that token-authenticated LAN WebSocket operator connections retain operator.write/admin scopes instead of connecting with cleared scopes.

## Best Possible Solution

Land one maintainer-approved gateway shared-token scope policy, likely configured token scopes with privileged-scope guards, plus docs, regression tests, and redacted real Docker/LAN proof.

## Reproduction Assessment

Yes, at source level. Current main accepts a valid device-less shared-token operator, clears requested scopes, and checked-in tests expect a follow-up scoped RPC to fail with missing scope.

## Solution Assessment

Unclear for the requested blanket scope retention. A bounded configured token-scope or device-identity policy is safer than trusting client-declared remote scopes by default.
