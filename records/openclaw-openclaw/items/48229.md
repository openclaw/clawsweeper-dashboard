## Summary

Keep open: current main still clears self-declared operator scopes for device-less token/password WebSocket operator sessions outside narrow local backend exceptions, and the linked fix attempts need maintainer/security policy review before this issue can be closed.

## What This Changes

The issue asks OpenClaw to grant requested operator scopes to valid token-authenticated WebSocket operator clients when `gateway.bind` is `lan`, matching the loopback behavior.

## Best Possible Solution

Maintain one explicit WebSocket auth policy: either land a scoped, secops-approved fix with regression coverage for remote shared-token operators, or document that LAN token-auth clients must present approved device identity.

## Reproduction Assessment

Yes, from source and checked-in tests. A valid shared-token operator connect with `device: null` is accepted, its scopes are cleared, and a scope-gated RPC is expected to fail with `missing scope`.

## Solution Assessment

Unclear. Granting scopes to all valid LAN token-auth WebSocket operators would address the report, but current docs and tests reserve device-less scope preservation for narrow trusted paths, so maintainers need to approve the exact trust boundary first.
