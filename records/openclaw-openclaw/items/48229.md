## Summary

Keep open: current main and the latest release still clear scopes for the relevant device-less shared-token WebSocket operator path, and the remaining fix is a security-sensitive Gateway auth policy decision already represented by open PRs.

## What This Changes

The issue asks for LAN-bound token-authenticated WebSocket operator connections to retain operator.write/operator.admin scopes instead of connecting with empty scopes.

## Best Possible Solution

Land one maintainer-approved WebSocket shared-token scope contract, likely bounded gateway.auth.tokenScopes with privileged opt-in, docs, regression tests, and redacted Docker/LAN proof.

## Reproduction Assessment

Yes, at source level. Current main accepts the relevant device-less shared-token operator path with cleared scopes, and checked-in tests expect a follow-up scoped RPC to fail with missing scope.

## Solution Assessment

No for the blanket requested behavior. The safer maintainable solution is an explicit configured token-scope or device-identity policy rather than trusting remote client-declared scopes by default.
