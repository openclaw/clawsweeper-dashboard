## Summary

Keep open. Current main and v2026.5.22 still clear self-declared scopes for device-less shared-token WebSocket operators, checked-in tests assert the missing-scope failure, and open linked PRs still need maintainer/security policy reconciliation.

## What This Changes

The issue requests that token-authenticated LAN WebSocket operator clients retain or receive operator.write/operator.admin-derived scopes instead of failing scoped gateway calls.

## Best Possible Solution

Land one maintainer-approved WebSocket shared-token scope contract, likely the bounded gateway.auth.tokenScopes path with privileged opt-in, docs, regression tests, and redacted Docker/LAN proof.

## Reproduction Assessment

Yes, at source level. Current main accepts the relevant device-less shared-token operator path with cleared scopes, and checked-in tests expect a follow-up scoped RPC to fail with missing scope.

## Solution Assessment

Unclear for the blanket requested behavior. The safer maintainable solution is an explicit configured token-scope or device-identity policy rather than trusting remote client-declared scopes by default.
