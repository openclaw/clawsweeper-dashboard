## Summary

Keep open: current main and the latest release still clear self-declared scopes for device-less shared-token operator WebSocket sessions, and the open linked PR plus canonical tracker have not resolved the regular token-auth policy question.

## What This Changes

The issue requests that LAN/Docker WebSocket operator clients authenticated with a valid gateway token retain `operator.admin`/`operator.write` scopes instead of failing write-scoped RPCs with `missing scope`.

## Best Possible Solution

Resolve the gateway auth contract once: either land a maintainer-approved, tested scope-retention policy for non-local token-auth operators, or document that device-less shared-token WebSocket scopes are intentionally cleared and close/update the linked items accordingly.

## Reproduction Assessment

Yes from source inspection: current main clears scopes for the reported device-less shared-token operator WebSocket path, and the default-token suite asserts the resulting `missing scope` behavior. I did not run the Docker/LAN flow live in this read-only review.

## Solution Assessment

Unclear/no. Preserving token-authenticated non-local operator scopes may fix the Docker/A2A workflow, but it changes an intentional security-sensitive baseline and the linked PR has narrowed to Tailscale/trusted-proxy mismatch.
