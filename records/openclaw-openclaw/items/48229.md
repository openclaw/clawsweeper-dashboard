## Summary

Keep open. Current main still clears requested WebSocket operator scopes for ordinary device-less shared-token clients, so the Docker LAN token-auth A2A path remains a real auth-policy gap rather than an implemented fix; the linked open PR is relevant but appears scoped to Tailscale/trusted-proxy behavior and explicitly leaves regular token-auth operators unchanged.

## What This Changes

The issue asks OpenClaw to grant requested `operator.write`/`operator.admin` WebSocket scopes to token-authenticated operator clients when the Gateway is bound to LAN for Docker deployments.

## Best Possible Solution

Define the intended A2A/plugin WebSocket contract explicitly: require paired device identity/device tokens for write-scoped operator RPCs, or add a narrow documented trusted backend/plugin exception with regression coverage for LAN bind, token auth, requested admin/write scopes, and a write-scoped RPC.

## Reproduction Assessment

Yes, at source level. Current tests already model a shared-token, device-less operator WebSocket connect with requested admin scope and assert a follow-up privileged RPC fails with `missing scope`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear. A broad rule that shared LAN tokens preserve all self-declared WebSocket operator scopes is not clearly the safest fix; a paired-device flow or a narrow trusted backend/plugin exception would better match the current documented boundary.
