## Label Justifications

## Summary

Keep open. Current main and v2026.5.18 still clear self-declared scopes for device-less token/password WebSocket operator sessions, checked-in tests expect the resulting scoped RPC to fail with `missing scope`, and open related PRs are still resolving the gateway auth policy boundary.

## What This Changes

The issue asks for WebSocket operator clients authenticated with a valid gateway token on `gateway.bind: "lan"` to retain requested operator scopes such as `operator.write`.

## Best Possible Solution

Define and land one maintainer-approved WebSocket shared-token scope policy, likely a bounded `gateway.auth.tokenScopes` path or documented device-identity requirement, with protocol/security docs, regression tests, and real gateway proof aligned.

## Reproduction Assessment

Yes, at source level. Current main accepts a valid shared-token operator without device identity, clears its requested scopes, and checked-in tests expect the follow-up scoped RPC to fail with `missing scope`.

## Solution Assessment

Unclear for the requested blanket scope retention. Preserving client-declared scopes would address the symptom, but current docs and tests keep device-less exceptions narrow, so the safer maintainable direction is an explicit configured token-scope or device-identity policy.
