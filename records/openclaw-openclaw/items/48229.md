## Summary

Keep open. Current main and the latest release still clear self-declared scopes for device-less shared-token WebSocket operator sessions, and open related PRs are carrying the remaining gateway auth/security policy decision rather than making this issue obsolete.

## What This Changes

The issue requests that LAN-bound token-authenticated WebSocket operator clients retain requested operator scopes, including `operator.write`, without depending on loopback binding.

## Best Possible Solution

Reconcile the open gateway auth PRs and land one maintainer-approved policy, likely an explicit configured token-scope path or a documented device-identity requirement, with code, tests, and protocol docs aligned.

## Reproduction Assessment

Yes, from current source and checked-in tests. Current main accepts a valid device-less shared-token operator, clears its scopes, and expects a scope-gated RPC to fail with `missing scope`; I did not run the full Docker LAN setup.

## Solution Assessment

Unclear. Preserving all requested client-declared scopes would address the report, but an explicit configured token-scope or device-identity policy is safer than a blanket LAN token exception.
