## Summary

Keep open. Current main still clears requested scopes for device-less shared-token WebSocket operator sessions, the latest release still carries that expectation, and an open PR still references this issue with closing syntax while a maintainer-labeled auth-policy PR proposes the safer configured-scope direction.

## What This Changes

The issue asks for LAN-bound token-auth WebSocket operator clients to retain `operator.write`/admin scopes without requiring device identity.

## Best Possible Solution

Land one maintainer-approved Gateway auth policy, likely the configured token-scope path in https://github.com/openclaw/openclaw/pull/81792 or a documented device-identity requirement, then close this issue and the overlapping reports after merge.

## Reproduction Assessment

Yes, from current source and checked-in tests: a valid shared-token operator without device identity is accepted with cleared scopes, and scoped RPCs are expected to fail with `missing scope`. I did not run the live Docker/LAN setup in this read-only review.

## Solution Assessment

Unclear for the requested blanket behavior. Preserving all client-declared scopes would address the symptom, but current docs/tests keep device-less exceptions narrow, so a configured token-scope policy with explicit privileged-scope bounds is the safer maintainable direction.
