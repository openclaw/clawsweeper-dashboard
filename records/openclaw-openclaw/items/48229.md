## Label Justifications

- `impact:security`: The requested fix changes whether device-less WebSocket clients can receive privileged operator scopes after shared-token auth.

## Summary

Keep open. Current main still clears self-declared scopes for device-less token/password WebSocket operator sessions, checked-in tests expect the follow-up scoped call to fail with `missing scope`, the latest release still carries that behavior, and open related PRs are still resolving the gateway auth policy boundary.

## What This Changes

The issue requests that LAN-bound token-authenticated WebSocket operator clients retain `operator.admin`/`operator.write` scopes for Docker/A2A gateway use.

## Best Possible Solution

Land one maintainer-approved Gateway auth policy, likely bounded `gateway.auth.tokenScopes`/privileged-scope rules or a documented device-identity requirement, then close this issue and overlapping reports after merge.

## Reproduction Assessment

Yes, from current source and checked-in tests: a valid shared-token operator without device identity is accepted with cleared scopes, and scoped RPCs are expected to fail with `missing scope`. I did not run the live Docker/LAN setup in this read-only review.

## Solution Assessment

Unclear for the requested blanket scope retention. `operator.admin` satisfies write only after scopes survive the handshake, and current docs/tests keep device-less exceptions narrow, so the safer maintainable direction is an explicit configured token-scope or device-identity policy.
