## Summary

Keep open. Current main still strips self-declared WebSocket operator scopes for ordinary device-less shared-token/password clients, so the reported Docker LAN token-auth path can still connect but then fails write-scoped RPC authorization. The HTTP default-scope fix mentioned in discussion is a separate HTTP path, and the linked open PR needs maintainer review because it appears focused on Tailscale/trusted-proxy behavior rather than the ordinary token-auth Docker case.

## What This Changes

The issue requests that token-authenticated WebSocket operator clients on `gateway.bind: "lan"` receive `operator.write`/admin scopes for Docker A2A plugin deployments even without loopback device identity.

## Best Possible Solution

Define the intended contract for A2A/plugin WebSocket clients explicitly: either require a paired device identity/device token for write-scoped operator RPCs, or add a narrow, documented gateway exception for trusted plugin/backend clients with regression coverage for `bind: "lan"`, token auth, requested write/admin scopes, and a write-scoped RPC.

## Reproduction Assessment

Yes. The current test baseline already models a shared-token, device-less operator WebSocket connect with requested admin scope and asserts the follow-up privileged RPC fails with a missing-scope error; I did not run the test because this review is read-only.

## Solution Assessment

Unclear. Granting full operator scopes solely from a LAN shared token may solve the Docker A2A use case, but it is security-sensitive and may be less maintainable than requiring device identity or adding a narrower authenticated backend/plugin exception.
