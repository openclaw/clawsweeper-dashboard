## Summary

Keep open. Current main and v2026.5.19 still clear self-declared scopes for device-less shared-token/password WebSocket operator sessions, and the remaining fix path is an open gateway auth policy decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to grant `operator.write`/admin-equivalent WebSocket scopes to token-authenticated operator clients when `gateway.bind` is `lan`.

## Best Possible Solution

Land one maintainer-approved WebSocket shared-token scope contract, preferably the bounded configured token-scope path with docs, regression tests, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main accepts a valid device-less shared-token operator, clears its requested scopes, and checked-in tests expect a follow-up scoped RPC to fail with missing scope.

## Solution Assessment

No for blanket scope retention as requested. The safer maintainable direction is an explicit configured token-scope or device-identity policy rather than trusting client-declared remote scopes by default.
