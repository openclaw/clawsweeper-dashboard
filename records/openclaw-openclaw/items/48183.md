## Summary

Keep open: current main and v2026.5.22 still clear Feishu HTTP server state immediately after `server.close()`, and the focused fix remains open at https://github.com/openclaw/openclaw/pull/48588 rather than merged.

## What This Changes

The issue asks for Feishu monitor shutdown to wait for HTTP server close before clearing `httpServers` state to avoid leaked resources and restart port conflicts.

## Best Possible Solution

Review, refresh, and land or replace https://github.com/openclaw/openclaw/pull/48588 with a narrow Feishu-owned awaitable HTTP shutdown helper and regression coverage, while keeping broader lifecycle architecture in https://github.com/openclaw/openclaw/issues/48793 or another design tracker.

## Reproduction Assessment

Yes, at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop cleanup and webhook abort cleanup. A live restart loop would only be needed to measure the external `EADDRINUSE` or memory-growth symptom.

## Solution Assessment

Yes for the immediate bug: the maintainable fix is a narrow Feishu cleanup change that awaits bounded HTTP shutdown before clearing state. The broader ResourceManager proposal discussed in comments is useful follow-up work but should stay separate from this user-facing bug fix.
