## Summary

Keep open: current main and the latest release still clear Feishu HTTP server state immediately after `server.close()`, and the focused open fix at https://github.com/openclaw/openclaw/pull/48588 uses closing syntax for this issue.

## What This Changes

The issue reports that Feishu monitor shutdown deletes HTTP server Map entries before Node finishes asynchronous server shutdown, risking retained resources and restart port conflicts.

## Best Possible Solution

Review, refresh, and land or replace https://github.com/openclaw/openclaw/pull/48588 with a narrow Feishu-owned awaitable HTTP shutdown helper and regression coverage, while keeping broader resource-lifecycle architecture in https://github.com/openclaw/openclaw/issues/48793 and https://github.com/openclaw/openclaw/issues/49804.

## Reproduction Assessment

Yes, at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop cleanup and webhook abort cleanup. A live restart loop would only be needed to measure the external `EADDRINUSE` or memory-growth symptom.

## Solution Assessment

Yes, the best immediate solution is a narrow Feishu cleanup fix that awaits bounded HTTP shutdown before clearing state. The broader ResourceManager proposal is useful follow-up work but should stay separate from this user-facing bug fix.
