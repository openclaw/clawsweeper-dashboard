## Summary

Keep open: current `main` and the latest release still clear Feishu HTTP server state immediately after `server.close()`, and the focused linked fix PR is still open rather than merged.

## What This Changes

The issue asks Feishu monitor cleanup to wait for HTTP server shutdown before clearing tracked state so repeated restarts do not leak resources or hit port-binding failures.

## Best Possible Solution

Review, refresh, and land or replace `https://github.com/openclaw/openclaw/pull/48588` with a narrow Feishu-owned awaitable HTTP shutdown helper and regression coverage, while keeping broader lifecycle architecture in separate design issues.

## Reproduction Assessment

Yes, at source level: current `main` still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop cleanup and webhook abort cleanup. A live restart loop would only be needed to measure the external `EADDRINUSE` or memory-growth symptom.

## Solution Assessment

Yes for the immediate bug: the maintainable fix is a narrow Feishu cleanup change that awaits bounded HTTP shutdown before clearing state. The broader ResourceManager proposal discussed in comments is useful follow-up work but should stay separate from this user-facing bug fix.
