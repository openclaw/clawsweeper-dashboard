## Label Justifications

- `P2`: This is a normal-priority Feishu plugin restart/resource-cleanup bug with limited scope but real operational impact.
- `impact:message-loss`: If Feishu webhook restart or reload cannot reclaim its port promptly, incoming Feishu channel events can fail to reach the monitor.

## Summary

Keep open: current main and the latest release still close Feishu HTTP servers and clear `httpServers` before the close callback, while the direct linked fix at https://github.com/openclaw/openclaw/pull/48588 remains open.

## What This Changes

The issue asks for Feishu monitor shutdown to await HTTP server closure before clearing state so repeated restarts do not leak resources or leave ports bound.

## Best Possible Solution

Land or replace the linked Feishu-local fix so state and transport cleanup await bounded HTTP server shutdown, expose an awaitable stop path, add focused close sequencing tests, and leave broader lifecycle architecture to https://github.com/openclaw/openclaw/issues/48793 or https://github.com/openclaw/openclaw/issues/49804.

## Reproduction Assessment

Yes at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop cleanup and webhook abort cleanup. I did not run a live Feishu restart loop to measure external memory growth or `EADDRINUSE`.

## Solution Assessment

Yes, the narrow Feishu-local awaited close path is the best immediate fix; a central ResourceManager may be useful later, but it is a separate plugin SDK design decision and should not replace the direct bug fix.
