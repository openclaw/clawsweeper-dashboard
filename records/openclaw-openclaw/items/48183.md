## Summary

Keep open: current main and the latest release still clear Feishu HTTP server state immediately after `server.close()`, and the linked open fix PR is the right implementation path before this issue can close.

## What This Changes

The issue requests awaitable cleanup for Feishu monitor HTTP servers so server references are not cleared before HTTP shutdown completes.

## Best Possible Solution

Review, refresh, and land or replace https://github.com/openclaw/openclaw/pull/48588 with a narrow Feishu-owned awaitable HTTP shutdown helper and regression coverage, while keeping the broader ResourceManager design in https://github.com/openclaw/openclaw/issues/48793.

## Reproduction Assessment

Yes, at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop and webhook-abort cleanup paths. A live restart loop would only be needed to measure the external EADDRINUSE or memory-growth symptom.

## Solution Assessment

Yes, the best immediate solution is a narrow Feishu cleanup fix that awaits bounded HTTP shutdown before clearing state. The broader ResourceManager proposal is useful follow-up work but should remain separate from this user-facing bug fix.
