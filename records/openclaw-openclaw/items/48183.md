## Summary

Keep open: current main and the latest release still clear Feishu HTTP server state immediately after `server.close()`, and the linked open fix PR is the right implementation path before this issue can close.

## What This Changes

The issue reports that Feishu monitor shutdown clears HTTP server Map entries before the async close callback completes, risking lingering ports and resource leaks.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/48588 with a narrow Feishu-owned awaitable HTTP shutdown helper and focused regression coverage, while keeping the broader ResourceManager design in the separate architecture issues.

## Reproduction Assessment

Yes, at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop and webhook-abort cleanup paths. A live restart loop would only be needed to measure the external memory or port symptom.

## Solution Assessment

Yes, the best immediate solution is a narrow Feishu cleanup fix that awaits server close and bounds stuck closes. The broader ResourceManager proposal is useful but should remain separate from this user-facing bug fix.
