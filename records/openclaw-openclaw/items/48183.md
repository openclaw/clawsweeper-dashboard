## Summary

Keep open: current main and the latest release still clear Feishu HTTP server state immediately after `server.close()`. The open focused fix at https://github.com/openclaw/openclaw/pull/48588 uses closing syntax for this issue, so this should stay open until that PR is reviewed, refreshed, landed, or replaced; the broader ResourceManager discussion is already separated into follow-up issues.

## What This Changes

Ensure Feishu monitor shutdown waits for HTTP server close before clearing `httpServers` state to avoid port/resource leaks on restart.

## Best Possible Solution

Review, refresh, and land or replace https://github.com/openclaw/openclaw/pull/48588 with a narrow Feishu-owned awaitable HTTP shutdown helper and regression coverage, while keeping ResourceManager architecture work in https://github.com/openclaw/openclaw/issues/48793 and https://github.com/openclaw/openclaw/issues/49804.

## Reproduction Assessment

Yes, at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop cleanup and webhook abort cleanup. A live restart loop would only be needed to measure the external EADDRINUSE or memory-growth symptom.

## Solution Assessment

Yes, the best immediate solution is a narrow Feishu cleanup fix that awaits bounded HTTP shutdown before clearing state. The broader ResourceManager proposal is useful follow-up work but should remain separate from this user-facing bug fix.
