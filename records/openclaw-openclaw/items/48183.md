## Summary

Keep open. Current main still calls `server.close()` in the Feishu monitor cleanup paths and removes tracked state immediately, the latest release tag contains the same behavior, and open PR #48588 is already the paired implementation candidate using closing syntax for this issue.

## What This Changes

The issue requests Feishu monitor shutdown to await HTTP server close/drain before clearing `httpServers` and related bot state.

## Best Possible Solution

Land or replace #48588 with a narrow Feishu plugin fix that makes HTTP server teardown awaitable, drains or bounds active connections, preserves existing WebSocket cleanup behavior, and adds focused regression coverage for targeted stop, global stop, and webhook abort cleanup. Keep the broader ResourceManager design in #48793 and #49804 as separate architecture work.

## Reproduction Assessment

Yes. The high-confidence reproduction path is code-level: start Feishu webhook monitoring, then trigger account stop, global stop, or abort/reload; current main calls `server.close()` before immediately removing tracked server state. Live Feishu credentials would only be needed to observe the external memory or port symptom.

## Solution Assessment

Yes. A narrow Feishu-specific async HTTP cleanup fix is the best immediate solution; the broader cross-plugin ResourceManager proposal may be worthwhile, but it should not block this plugin bug fix.
