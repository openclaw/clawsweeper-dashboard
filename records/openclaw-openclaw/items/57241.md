## Summary

Keep open. Current main still unconditionally re-applies Tailscale Serve on gateway startup and still enforces password-only Funnel, so the reported behavior is not fixed on main. Open PR #73755 explicitly closes this issue and is the current implementation candidate, so this issue should stay open until that PR is reviewed, landed, or rejected.

## What This Changes

The issue asks OpenClaw to preserve externally configured Tailscale Funnel routes across gateway restarts for token-auth deployments, or otherwise provide a supported persistent Funnel path.

## Best Possible Solution

Review #73755 as the current implementation candidate. The desired end state is a documented, tested path that prevents OpenClaw-managed Serve startup from clobbering externally managed Funnel routes without weakening public Funnel auth unless maintainers explicitly approve that security tradeoff.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `serve` mode reaches `tailscale serve --bg --yes <port>` on startup with no preservation check, while `funnel` mode with token auth reaches the exact password-only startup error.

## Solution Assessment

No final solution is on main yet. The narrow maintainable direction appears to be an opt-in preserve-existing-Funnel behavior or clear `mode: "off"` guidance for externally managed Funnel, while any token-auth Funnel change needs explicit public-exposure security review.
