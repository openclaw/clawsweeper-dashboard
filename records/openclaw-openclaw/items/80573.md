## Summary

Keep open: current main and v2026.6.10 still emit the generic `potential-exfiltration` plugin code-safety warning, and neither the plugin manifest nor audit aggregation path implements egress intent, rationale, downgrade, suppression, or operator acknowledgement semantics. The related items are adjacent only: https://github.com/openclaw/openclaw/issues/48510 is broader per-tool descriptors, while https://github.com/openclaw/openclaw/issues/82469 and https://github.com/openclaw/openclaw/pull/82535 address test-file false positives rather than legitimate production data export.

## What This Changes

The issue requests a plugin manifest capability/rationale field so legitimate data-egress plugins can downgrade, annotate, acknowledge, or suppress `plugins.code_safety` potential-exfiltration warnings.

## Best Possible Solution

Keep this open for a maintainer-approved security design: add an additive manifest egress-intent/rationale contract that annotates findings and only downgrades or suppresses after explicit operator-owned acknowledgement.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.10 still emit `potential-exfiltration` and aggregate it into generic `plugins.code_safety` findings, while the manifest contract lacks egress-intent, rationale, or operator acknowledgement fields.

## Solution Assessment

Unclear as a final implementation shape. A manifest declaration is plausible, but the best fix needs maintainer-approved security semantics for annotation, severity downgrade, and any operator acknowledgement or suppression path.
