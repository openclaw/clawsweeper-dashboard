## Summary

Keep open: current main and v2026.6.8 still emit the generic plugin `potential-exfiltration` warning and only support operator-owned suppressions; the requested plugin manifest egress-intent/rationale contract is not implemented and needs maintainer security/product direction before a fix PR.

## What This Changes

The issue asks for plugin manifest fields that declare legitimate data-egress intent and rationale so `security audit --deep` can distinguish intentional export plugins from suspicious exfiltration patterns.

## Best Possible Solution

Define a narrow security-reviewed plugin manifest egress-intent/rationale contract that annotates matching audit findings and only suppresses after explicit operator acknowledgement.

## Reproduction Assessment

Yes, source-level: current main still has the `potential-exfiltration` rule and generic `plugins.code_safety` warning output, while plugin manifest types/docs do not include the proposed egress-intent field. Runtime reproduction is not applicable because this is a feature/API request rather than a broken existing behavior report.

## Solution Assessment

Unclear. A plugin manifest declaration is a plausible direction, but the best fix needs maintainer-approved security semantics for annotation, downgrade, and explicit operator suppression.
