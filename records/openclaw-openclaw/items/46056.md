## Summary

Keep open. Current main and the latest release still lack builtin-specific allowlist handling or docs, and the open linked implementation PR uses closing linkage for this issue, so the issue should remain the canonical tracker until maintainers land or reject that policy change.

## What This Changes

The issue requests documented or opt-in handling for pathless shell builtins in exec allowlist mode so commands such as `cd /path && git status` do not unexpectedly require approval.

## Best Possible Solution

Review and either land `https://github.com/openclaw/openclaw/pull/79363` after security/product approval and configured exec-flow proof, or reject builtin auto-allow and add focused docs for the current limitation and workaround.

## Reproduction Assessment

Yes, source-reproducible. Current main has no builtin satisfaction branch, and pathless command segments cannot match normal allowlist paths, so `cd /tmp && git status` fails allowlist evaluation for the `cd` segment.

## Solution Assessment

Unclear. A closed-set opt-in `safeBuiltins` config is plausible, but it is a new exec approval policy knob that needs maintainer security/product review; docs-only is the safer fallback if that policy is rejected.
