## Summary

Keep open. Current main still treats pathless shell builtins as unsatisfied allowlist segments, and the shipped docs still do not explain why commands such as `cd /path && git status` can prompt even when the shell and later binaries are allowlisted.

## What This Changes

Clarify or change exec approval allowlist behavior for shell builtins such as `cd`, `export`, and shell-provided `echo`/`pwd`.

## Best Possible Solution

Add a focused documentation clarification for pathless shell builtins now, and treat runtime builtin auto-allow as a separate opt-in exec-approval policy design if maintainers want it later.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level on current main: shell-chain parts are evaluated independently, pathless resolutions cannot satisfy ordinary allowlist entries, and no builtin satisfaction branch exists.

## Solution Assessment

Yes for a docs-only repair. Runtime auto-allow is not the narrowest maintainable default without explicit policy review because shell builtins can alter later command execution context.

## Repair Work Prompt

Treat https://github.com/openclaw/openclaw/issues/46056 as a docs-only exec approvals clarification. Update `docs/tools/exec-approvals.md` in the allowlist section, and optionally `docs/tools/exec-approvals-advanced.md` if the shell-chain wording fits better there, to explain that POSIX shell builtins/pathless commands such as `cd`, `export`, and shell-provided `echo`/`pwd` do not satisfy ordinary resolved-path or bare-command allowlist entries because there is no resolved binary path. Mention that every top-level shell-chain segment must satisfy allowlist/safeBins/skill rules, that allowlisting `/usr/bin/bash` does not by itself allow a builtin segment, and that users can avoid the prompt by using tool-native options such as `git -C <path>` where available or by approving the chained command. Do not add `safeBuiltins`, config, runtime policy changes, migrations, or tests for changed runtime behavior. Keep examples generic with no personal paths. No changelog is expected unless behavior or config changes are introduced.
