## Summary

Keep open. Current main still treats pathless shell builtins as unsatisfied allowlist segments, and the exec approval docs still do not explain why commands such as `cd /path && git status` can prompt even when the shell and later binaries are allowlisted.

## What This Changes

Request to document or change exec allowlist handling so shell builtins such as `cd` and `export` do not unexpectedly trigger approval prompts when other binaries are allowlisted.

## Best Possible Solution

Add a focused documentation clarification to the exec approvals allowlist section explaining that pathless POSIX shell builtins cannot satisfy ordinary path or bare-name allowlist entries, with safer workarounds such as `git -C`. Treat runtime builtin auto-allow as a separate opt-in policy design with tests.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level on current main: shell-chain segments are evaluated independently, pathless resolutions cannot match ordinary allowlist entries, and `cd` is reported by the checked shell as a builtin rather than a filesystem executable.

## Solution Assessment

Yes for a docs-only first fix. Runtime auto-allow is not the best default fix without explicit policy and security review because builtins such as `cd` and `export` can change later command execution context.

## Repair Work Prompt

Fix the exec allowlist documentation for https://github.com/openclaw/openclaw/issues/46056 as a docs-only change. In `docs/tools/exec-approvals.md`, update the allowlist section to explain that pathless POSIX shell builtins such as `cd`, `export`, and `unset` cannot satisfy normal resolved-path or bare command-name allowlist entries, so commands like `cd /path && git status` may still prompt even if `/usr/bin/bash` and `git` are allowlisted. Include a safer workaround such as using tool-native directory flags (`git -C /path status`) where available. Do not change runtime allowlist semantics, safeBins, approval persistence, or add a `safeBuiltins` option. Do not add a changelog entry unless repo docs policy requires one for this clarification.
