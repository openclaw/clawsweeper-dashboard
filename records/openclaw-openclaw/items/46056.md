## Summary

Keep open. Current main still lacks a builtin-specific allowlist path, and the open implementation PR at https://github.com/openclaw/openclaw/pull/79363 uses closing linkage for this issue, so this should stay open until that PR lands or maintainers reject the runtime direction.

## What This Changes

The issue asks OpenClaw to document or optionally allow selected shell builtins in exec allowlist mode so chained commands such as `cd /path && git status` do not always prompt when the later binary is allowlisted.

## Best Possible Solution

Review and either land the closed-set opt-in implementation in https://github.com/openclaw/openclaw/pull/79363 after its remaining concerns are resolved, or reject that policy and add a focused docs clarification with the builtin workaround.

## Reproduction Assessment

Yes. Source inspection on current main shows pathless builtin segments cannot satisfy normal allowlist, safeBins, or skill auto-allow, and shell-chain evaluation fails at the first unsatisfied segment.

## Solution Assessment

Unclear. The linked opt-in `tools.exec.safeBuiltins` direction is plausible, but it changes exec approval policy and should be decided through that PR rather than closing this issue early.
