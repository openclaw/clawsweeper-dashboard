## Summary

Keep open: current main and the latest shipped v2026.5.18 release still make Claude thinking profiles provider-hook scoped, so a custom provider id using `api: "anthropic-messages"` falls back to the base thinking levels and misses `adaptive`/`max`. The report is source-reproducible, but the fix needs provider-boundary review rather than a direct Claude fallback in generic thinking code.

## What This Changes

The issue asks OpenClaw to expose Claude `adaptive` and `max` thinking profiles for custom `anthropic-messages` providers that proxy Claude models.

## Best Possible Solution

Add a provider-boundary-safe thinking-profile path for configured Anthropic-compatible providers, cover `adaptive` and Claude 4.6 `max`, and keep broader opaque-id opt-in design aligned with https://github.com/openclaw/openclaw/issues/72890.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are not valid levels even when the configured provider uses `api: "anthropic-messages"`; I did not run a live proxy because this is a read-only review and no proxy credentials were provided.

## Solution Assessment

No for the suggested direct fallback. The maintainable fix should preserve the provider-policy boundary by carrying the needed transport/model-family capability into the thinking profile path or by extending the existing custom-provider capability contract.
