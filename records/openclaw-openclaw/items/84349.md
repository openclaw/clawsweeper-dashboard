## Summary

Keep open: current main and v2026.5.18 still route thinking profiles through provider-owned hooks, so a custom provider id using the Anthropic Messages transport falls back to the base thinking ladder and will not advertise `adaptive` or `max`. The report is source-reproducible, but the exact fix needs provider-boundary review because the suggested core fallback would move Claude-specific policy into generic thinking resolution.

## What This Changes

The issue asks OpenClaw to give custom `api: "anthropic-messages"` Claude providers the Claude adaptive/max thinking profiles instead of the base thinking levels.

## Best Possible Solution

Add a provider-boundary-safe way for configured Anthropic-compatible custom providers to expose Claude thinking profiles, likely by carrying transport/family metadata into thinking policy or extending catalog compat, then update profile tests and docs after confirming the 4.6 `max` contract.

## Reproduction Assessment

Yes, from source inspection: a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are not valid levels even when the configured model uses `api: "anthropic-messages"`. I did not run a live proxy because this was a read-only review and no proxy credentials were provided.

## Solution Assessment

No, the suggested direct Claude fallback in generic thinking resolution is probably not the best maintainable fix because it bypasses the provider-policy boundary. A safer fix should expose the needed transport/model-family fact to thinking policy or make the custom catalog compat contract expressive enough to advertise `adaptive` and `max`.
