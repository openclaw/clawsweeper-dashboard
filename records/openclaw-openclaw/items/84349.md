## Summary

Keep open: current main still routes custom `anthropic-messages` provider ids through the generic thinking profile unless the provider has its own thinking hook, and the shared Claude 4.6 profile still omits `max` despite the current Anthropic effort contract. The report is source-reproducible, but the durable fix needs maintainer judgment on the provider-policy boundary rather than a generic Claude fallback in core.

## What This Changes

The issue asks for custom `api: "anthropic-messages"` providers to receive Claude adaptive/max thinking profiles and for Claude Sonnet/Opus 4.6 to advertise `max` effort.

## Best Possible Solution

Add a provider-boundary-safe way for configured Anthropic-compatible providers to get the right Claude thinking profile or declared capability, update Claude 4.6 `max` support, and cover validation, transport behavior, session defaults, and docs with focused tests.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are unavailable even when the model uses the Anthropic Messages transport.

## Solution Assessment

No for the suggested direct fallback in generic thinking resolution. A safer solution should keep Claude policy provider-owned, carry or declare the relevant Anthropic-compatible model-family fact, and align the shared Claude 4.6 profile with the upstream contract.
