## Summary

Keep open. Current main still resolves custom provider thinking profiles by provider hook/provider id, so a custom `api: "anthropic-messages"` Claude proxy falls back to the base `off` through `high` profile, and the current Claude 4.6 profile/tests still omit `max` despite current Anthropic effort docs listing `max` for Opus 4.6 and Sonnet 4.6.

## What This Changes

The issue asks OpenClaw to expose Claude `adaptive` and `max` thinking profiles for custom Anthropic Messages-compatible providers, especially Claude 4.6 models behind proxies.

## Best Possible Solution

Carry enough configured model API or capability metadata into thinking profile resolution so known Claude models on `anthropic-messages` custom providers expose the shared Claude profile, update Claude 4.6 `max` support consistently through profile and transport mapping, and leave opaque alias opt-in to the existing custom capability discussion.

## Reproduction Assessment

Yes, at source level. A custom provider id has no matching provider thinking hook, so current `resolveThinkingProfile` falls back to base levels; the same current code and v2026.5.18 release profile also omit `max` for Claude 4.6.

## Solution Assessment

No, not exactly as proposed. Calling the Claude helper directly from the generic fallback would solve the symptom, but the safer maintainer path is to preserve provider boundaries by using configured model API/capability metadata or a decided provider-family contract.
