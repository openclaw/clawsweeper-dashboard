## Summary

Keep open: current main and the latest release still leave custom `anthropic-messages` providers on the generic thinking profile unless the custom provider supplies its own thinking hook, so `adaptive` and `max` are not advertised for Claude proxy models. The report is source-reproducible, but the fix needs maintainer product review because the suggested generic Claude fallback would cross the current provider-owned policy boundary.

## What This Changes

The issue requests Claude thinking profiles, including `adaptive` and `max`, for custom providers configured with the `anthropic-messages` transport.

## Best Possible Solution

Add a provider-boundary-safe way for configured Anthropic-compatible custom providers to expose Claude thinking profiles, then update source, tests, and docs for custom `anthropic-messages` providers and the approved 4.6 `max` behavior.

## Reproduction Assessment

Yes, from source inspection: a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, and the configured catalog does not carry `api: "anthropic-messages"` into thinking policy. I did not run a live Bifrost or LiteLLM proxy because this read-only review has no proxy endpoint or credentials.

## Solution Assessment

No, the suggested direct Claude fallback in generic thinking resolution is probably not the best maintainable fix because it embeds Claude-specific policy in generic core. A safer path should expose the needed Anthropic-compatible model-family capability through the provider/catalog contract or custom provider profile metadata.
