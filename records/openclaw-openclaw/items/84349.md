## Summary

Keep open: current main still leaves custom `anthropic-messages` provider IDs on the base thinking profile unless the provider supplies its own hook, and the shared Claude 4.6 profile still omits `max` despite the current Anthropic effort contract.

## What This Changes

The issue requests Claude adaptive/max thinking profiles for custom Anthropic Messages-compatible providers.

## Best Possible Solution

Add a provider-boundary-safe profile path for configured Anthropic-compatible custom providers, align Claude 4.6 `max` support with the upstream contract, and coordinate the opaque-model capability question with the related open issue.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with `api: "anthropic-messages"` can reach the Anthropic transport, but `resolveThinkingProfile()` only receives the custom provider id and falls back to base levels when no provider hook is present.

## Solution Assessment

No. The suggested direct Claude fallback in generic thinking resolution would solve part of the symptom, but the maintainable fix should preserve provider-owned policy by carrying the right Anthropic-compatible/profile fact through a declared seam.
