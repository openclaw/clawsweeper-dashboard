## Summary

Keep open: current main still leaves custom `anthropic-messages` provider ids on the base thinking profile unless a provider-owned hook exists, and Claude 4.6 still lacks `max` in the shared Claude profile even though the current Anthropic effort contract includes it. The report is source-reproducible, but the fix needs maintainer judgment on the provider-policy boundary.

## What This Changes

Custom Anthropic-compatible providers should expose Claude `adaptive` and `max` thinking profiles for Claude model ids instead of falling back to base thinking levels.

## Best Possible Solution

Add a provider-boundary-safe profile path for configured Anthropic-compatible custom providers, align Claude 4.6 `max` support with the upstream contract, and cover validation, session defaults, transport behavior, and docs with focused tests.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base profile, so `adaptive` and `max` are unavailable even when the configured model uses `api: "anthropic-messages"`.

## Solution Assessment

No. The suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it cuts across provider-owned policy; the safer path should carry the right transport/model-family or declared capability fact into provider-aware thinking policy.
