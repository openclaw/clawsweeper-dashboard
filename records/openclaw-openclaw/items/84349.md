## Summary

Keep open: current main still gives custom provider ids without a provider thinking hook only the base thinking profile, while the shared Claude 4.6 profile and adjacent tests still omit `max`. The related open custom-provider capability issue and Anthropic xhigh PR overlap this area, but neither makes this issue obsolete or fully implemented.

## What This Changes

Custom `anthropic-messages` providers are reported to miss Claude `adaptive` and `max` thinking profiles for proxied Claude models.

## Best Possible Solution

Define a provider-boundary-safe Claude profile path for configured Anthropic-compatible providers, update the shared Claude 4.6 profile if maintainers accept `max`, and keep the opaque-alias capability shape aligned with https://github.com/openclaw/openclaw/issues/72890.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base profile, so `adaptive` and `max` are unavailable even when the configured model uses the Anthropic Messages transport.

## Solution Assessment

No for the suggested direct fallback. The issue is valid, but the maintainable fix should preserve provider ownership by carrying the right transport/model-family or explicit capability fact into provider-owned thinking policy.
