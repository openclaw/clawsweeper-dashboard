## Summary

Keep open: current main still routes custom `anthropic-messages` provider IDs without a provider thinking hook to the base thinking profile, and the shared Claude 4.6 profile still omits `max`. The report is source-reproducible, but the fix needs maintainer judgment because a generic Claude fallback in core would cross the provider-policy boundary.

## What This Changes

The issue asks for custom `api: "anthropic-messages"` providers to receive Claude adaptive/max thinking profiles and for Claude 4.6 profiles to expose `max`.

## Best Possible Solution

Add a provider-boundary-safe profile path for configured Anthropic-compatible custom providers, align Claude 4.6 `max` support with the upstream effort contract, and coordinate opaque custom-model capability handling with https://github.com/openclaw/openclaw/issues/72890.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are unavailable even when the configured model uses the Anthropic Messages transport.

## Solution Assessment

No for the suggested direct core fallback. The maintainable fix should preserve the provider-policy boundary by carrying the right transport/model-family fact or provider-owned profile into thinking policy instead of hardcoding Claude policy in generic core resolution.
