## Summary

Keep open: current main still routes thinking profiles through provider-owned hooks, so a custom provider id using the Anthropic Messages transport falls back to the base thinking ladder and will not advertise Claude `adaptive` or `max`. The report is source-reproducible, but the fix needs provider-boundary review before a safe implementation path is chosen.

## What This Changes

The issue asks for custom `api: "anthropic-messages"` providers serving Claude models to expose Claude thinking profiles, including `adaptive` and `max` where supported.

## Best Possible Solution

Expose Claude thinking profiles for Anthropic-compatible custom providers through a provider-boundary-safe metadata or hook path, then cover custom provider `adaptive` and 4.6 `max` behavior with focused tests.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base profile, so `adaptive` and `max` are not valid levels even when the configured model uses `api: "anthropic-messages"`.

## Solution Assessment

No. The suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it bypasses the current provider-policy boundary; a safer solution should carry the relevant transport/model-family fact through a supported provider metadata or hook contract.
