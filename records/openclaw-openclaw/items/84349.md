## Summary

Keep open: current main still does not give a custom provider id with `api: "anthropic-messages"` the Claude-specific thinking profile, and the shared Claude 4.6 profile still omits `max`. The issue is source-reproducible, but the fix should be maintainer-shaped because a generic Claude fallback in core would cross the provider-policy boundary and overlaps the existing custom-provider capability discussion.

## What This Changes

The issue requests Claude `adaptive` and `max` thinking profiles for custom Anthropic Messages providers, including Claude 4.6 `max` support.

## Best Possible Solution

Add a provider-boundary-safe profile path for configured Anthropic-compatible custom providers, align Claude 4.6 `max` support across shared profile, transport behavior, docs, and tests, and coordinate any opaque-id capability surface with https://github.com/openclaw/openclaw/issues/72890.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches only the base thinking profile, so `adaptive` and `max` are unavailable even when the configured model uses `api: "anthropic-messages"`.

## Solution Assessment

No for the suggested direct fallback. Calling Claude policy directly from generic thinking resolution would solve the symptom but is not the best maintainable shape because provider-owned policy should stay behind a typed provider/API boundary.
