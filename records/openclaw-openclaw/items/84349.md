## Summary

Keep open: current main still gives custom `anthropic-messages` provider IDs only the base thinking levels unless that provider has its own thinking hook, and Claude 4.6 still lacks `max` in the shared Claude profile despite the current Anthropic effort contract. The report is source-reproducible, but the fix needs provider-boundary judgment because the suggested generic Claude fallback would put provider-specific policy into core thinking resolution.

## What This Changes

The issue asks OpenClaw to expose Claude `adaptive` and `max` thinking profiles for custom providers configured with `api: "anthropic-messages"`, including `max` for Claude Sonnet/Opus 4.6.

## Best Possible Solution

Add a provider-boundary-safe profile path for configured Anthropic-compatible custom providers, align Claude 4.6 `max` support with the confirmed upstream contract, and cover profile validation, transport behavior, session defaults, and docs with focused tests.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are unavailable even when the configured model uses the Anthropic Messages transport.

## Solution Assessment

No, the suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it bypasses the provider-policy boundary. A safer solution should carry the right transport/model-family or compat fact into provider-owned thinking policy.
