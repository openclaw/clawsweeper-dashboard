## Summary

Keep open: current main still validates thinking levels through provider-owned thinking profiles, so a custom provider id using the Anthropic Messages transport cannot inherit Claude `adaptive`/`max` from the bundled Anthropic provider. The report is source-reproducible, but the fix needs provider-boundary and capability-policy review rather than a direct Claude fallback in generic thinking resolution.

## What This Changes

The issue requests that custom `api: "anthropic-messages"` providers and proxies expose Claude-specific thinking levels such as `adaptive` and `max` instead of falling back to the base thinking profile.

## Best Possible Solution

Expose Claude thinking profiles for custom Anthropic-compatible providers through a provider-boundary-safe contract, then update profile tests and docs after maintainers decide the 4.6 `max` user-facing policy.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base profile, so `adaptive` and `max` are not valid levels even when the configured provider uses `api: "anthropic-messages"`.

## Solution Assessment

No. The suggested generic Claude fallback is not the best maintainable fix because it bypasses the provider-policy boundary; the safer path is to carry the needed transport/model-family fact into thinking policy or add a profile-capable custom provider contract.
