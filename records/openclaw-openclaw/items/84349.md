## Summary

Keep open: current main still gives custom `anthropic-messages` provider IDs only the base thinking profile unless a provider-owned hook handles them, and Claude 4.6 profiles still omit `max` despite the current Anthropic effort contract. The related custom-provider capability issue overlaps but does not supersede this issue’s exact known-Claude-model and 4.6 `max` gaps.

## What This Changes

Custom `anthropic-messages` providers should expose Claude adaptive/max thinking profiles, including `max` for Claude Sonnet 4.6 and Opus 4.6.

## Best Possible Solution

Add a provider-boundary-safe thinking-profile path for configured Anthropic-compatible custom providers, update Claude 4.6 `max` support, and keep the opaque-ID capability work aligned with https://github.com/openclaw/openclaw/issues/72890.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are unavailable even when the configured model uses the Anthropic Messages transport.

## Solution Assessment

No. The suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it crosses the provider-policy boundary; the safer path is a provider/API-aware profile mechanism plus an explicit Claude 4.6 `max` profile update.
