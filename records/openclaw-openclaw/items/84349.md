## Summary

Keep open: current main still lets custom Anthropic Messages-compatible provider ids miss the Claude thinking profile hook, so the reported `adaptive`/`max` policy gap remains source-reproducible. The fix needs maintainer product/API judgment because the direct fallback suggested in the issue would put Claude-specific policy into generic thinking resolution instead of a provider-boundary contract.

## What This Changes

The issue asks OpenClaw to expose Claude `adaptive` and `max` thinking profiles for custom providers configured with `api: "anthropic-messages"`.

## Best Possible Solution

Add a provider-boundary-safe way for configured Anthropic-compatible custom providers to advertise Claude thinking capabilities, then cover known Claude ids, `adaptive`, and verified 4.6 `max` behavior in focused tests.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are not advertised even when the configured model uses the Anthropic Messages transport.

## Solution Assessment

No. The suggested direct Claude fallback is not clearly the best maintainable fix because it bypasses provider-policy ownership; the safer solution is to carry the needed transport/model-family capability through a documented provider or catalog contract.
