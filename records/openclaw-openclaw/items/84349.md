## Summary

Keep open: current main and v2026.5.18 still leave custom `anthropic-messages` provider IDs without the Claude thinking profile, and Claude 4.6 `max` support is not implemented consistently. The report is source-reproducible, but the repair needs provider-boundary/API judgment rather than cleanup closure.

## What This Changes

Report that custom configured `api: "anthropic-messages"` providers do not expose Claude `adaptive` or `max` thinking profiles for Claude 4.6 models.

## Best Possible Solution

Expose Claude thinking profiles to configured Anthropic-compatible custom providers through a provider-boundary-safe contract, then add confirmed Claude 4.6 `max` support across profile resolution, transport effort mapping, docs, and focused regression tests.

## Reproduction Assessment

Yes, from source inspection: a custom provider ID with no `resolveThinkingProfile` hook reaches the base profile, so `adaptive` and `max` are not valid even when the configured model uses `api: "anthropic-messages"`. I did not run a live proxy because this review is read-only and no proxy setup was provided.

## Solution Assessment

No, the suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it bypasses the provider-policy boundary. A better fix should carry a configured Anthropic-compatible model/family fact into thinking policy or expose an approved custom-provider profile contract, then align `max` transport behavior.
