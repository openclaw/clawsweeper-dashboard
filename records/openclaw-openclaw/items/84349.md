## Summary

Keep open: current main and shipped v2026.5.19 still resolve custom Anthropic-compatible provider thinking through provider-owned hooks and then fall back to the base ladder, so custom `api: "anthropic-messages"` providers do not get Claude `adaptive` or `max` profiles. The source path is clear, but the fix needs maintainer provider-boundary review before automation.

## What This Changes

Custom providers configured with `api: "anthropic-messages"` should expose the correct Claude thinking profile instead of falling back to base thinking levels.

## Best Possible Solution

Add a provider-boundary-safe contract for custom Anthropic-compatible providers to advertise Claude thinking profiles, likely by carrying configured `api` and model-family facts or extending catalog compat, then update tests/docs after confirming the Claude 4.6 `max` mapping.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are not valid levels even when the configured provider uses `api: "anthropic-messages"`; I did not run a live proxy in this read-only review.

## Solution Assessment

No. The suggested direct Claude fallback in generic thinking resolution is not the best maintainable path because it bypasses the provider-policy boundary; the safer fix is to expose a clear provider/profile contract for custom Anthropic-compatible Claude models.
