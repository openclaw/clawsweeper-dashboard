## Summary

Keep open: current main still reproduces the reported provider-policy gap, and the Claude 4.6 `max` effort mismatch remains. The root cause is clear from source, but the fix needs maintainer judgment so custom Anthropic-compatible providers do not get provider-specific Claude policy through an unsafe generic core fallback.

## What This Changes

The issue asks for custom providers configured with `api: "anthropic-messages"` to receive Claude `adaptive` and `max` thinking profiles instead of falling back to base thinking levels.

## Best Possible Solution

Carry the configured Anthropic-compatible provider/model fact into a provider-owned thinking profile path, add Claude 4.6 `max` support, and align tests/docs with the related opaque-ID capability decision.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base profile, so `adaptive` and `max` are unavailable even when the configured model uses the Anthropic Messages transport.

## Solution Assessment

No, the suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it bypasses the provider-policy boundary. A safer solution should preserve provider ownership while using configured transport/model-family facts.
