## Summary

Keep open: current main still falls back to base thinking levels for custom `anthropic-messages` provider ids without a provider-owned thinking hook, and the shared Claude 4.6 profile still omits `max` despite the current Anthropic effort contract.

## What This Changes

Requests that custom providers configured with `api: "anthropic-messages"` receive Claude thinking profiles and that Claude 4.6 profiles include `max`.

## Best Possible Solution

Add a provider-boundary-safe way for configured Anthropic-compatible custom providers to receive the correct Claude thinking profile, align Claude 4.6 `max` support with the upstream effort contract, and cover validation, session defaults, transport payloads, and docs together.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id without a `resolveThinkingProfile` hook falls through to base thinking levels, so `adaptive` and `max` are unavailable even when the model uses the Anthropic Messages transport.

## Solution Assessment

No. The suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it bypasses provider-owned policy; a safer solution should carry or declare the Anthropic-compatible profile fact at the provider boundary.
