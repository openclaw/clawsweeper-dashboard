## Summary

Keep open: current main and the latest release still route custom provider IDs without a provider thinking hook to the base thinking profile, while Claude 4.6 still omits `max`; the report is source-reproducible but needs a provider-boundary decision before implementation.

## What This Changes

The issue asks OpenClaw to give custom `api: "anthropic-messages"` providers the Claude thinking profile, including `adaptive` and Claude 4.6 `max` support.

## Best Possible Solution

Add a provider-boundary-safe profile path for configured Anthropic-compatible providers, align Claude 4.6 `max` support with the upstream effort contract, and cover validation, transport payloads, session defaults, and docs with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a custom provider id with `api: "anthropic-messages"` and no `resolveThinkingProfile` hook reaches the base profile, so `adaptive` and `max` are not available even for Claude-family model ids.

## Solution Assessment

No. The suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it bypasses provider-owned policy; the safer solution should carry an approved transport/model-family or explicit compat fact into the provider thinking profile path.
