## Summary

Keep open: current main still makes Claude adaptive/max thinking profiles provider-id owned, so a custom provider using the Anthropic Messages transport falls back to the base thinking ladder. The report is source-reproducible, but the fix needs maintainer/provider-boundary review because the direct fallback proposed in the issue would put Claude-specific policy into generic thinking resolution.

## What This Changes

Request that custom providers configured with `api: "anthropic-messages"` expose Claude adaptive/max thinking profiles, including `max` for Sonnet/Opus 4.6.

## Best Possible Solution

Add a provider-boundary-safe way for custom Anthropic-compatible providers to advertise Claude thinking profiles, then update 4.6 `max` expectations, tests, and docs together.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no thinking profile hook reaches the base thinking profile, so `adaptive` and `max` are not valid levels even when the configured model uses the Anthropic Messages transport.

## Solution Assessment

No. The suggested direct fallback would likely solve the symptom but is not the best maintainable boundary; the safer path is to expose the needed transport/model-family fact to provider-owned thinking policy or the configured catalog contract.
