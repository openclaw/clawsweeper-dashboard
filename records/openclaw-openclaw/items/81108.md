## Summary

Keep open: the disk-discovery direction and supplied proof are solid, but the current head still adds synchronous checkpoint directory scans to single-row/session-event row loads, which is a merge-blocking availability regression.

## What This Changes

The PR adds gateway disk discovery for `{session}.checkpoint.<uuid>.jsonl` files across session rows, compaction list/get/branch/restore, and targeted gateway tests.

## Best Possible Solution

Keep disk checkpoint discovery for explicit list/describe and compaction APIs, but move single-row event snapshots to a prepared, cached, or async-safe checkpoint preview path with regression coverage proving no sync filesystem scan on that hot path.

Yes. Source inspection shows current main reads checkpoint metadata from `sessions.json`, while the linked report and PR proof cover checkpoint files present on disk without metadata; source inspection also reproduces the PR regression path through single-row sync fs scans.

## Solution Assessment

No. Disk discovery is the right repair direction for the user-visible checkpoint bug, but the current implementation is not the best fix because it puts synchronous directory scanning on session event snapshots.
