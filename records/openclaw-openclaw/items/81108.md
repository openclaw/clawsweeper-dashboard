## Summary

Keep open: the disk checkpoint recovery is useful and has real behavior proof, but the current head still adds synchronous checkpoint directory scanning/statting to the `sessions.list` row path, so it needs one more hot-path repair before merge.

## What This Changes

The PR makes gateway session compaction APIs and session rows discover `{session}.checkpoint.<uuid>.jsonl` files from disk when `sessions.json` lacks usable checkpoint metadata.

## Best Possible Solution

Preserve disk-discovered checkpoint recovery for list/get/branch/restore, but move row preview discovery to an async, bounded, or lifecycle-owned index and merge only after focused tests and exact-head CI pass.

## Reproduction Assessment

Yes, at source level: current main lists checkpoints only from `compactionCheckpoints` metadata, while the linked report shows valid `.checkpoint.<uuid>.jsonl` files on disk with missing historical visibility. I did not run a live repro in this read-only review.

## Solution Assessment

No, not yet: recovering disk checkpoints is the right bug-fix direction, but the session-row preview path should not add synchronous request-time directory/stat polling to a gateway hot path.
