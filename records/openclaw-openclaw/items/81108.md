## Summary

Keep this PR open: the disk-checkpoint discovery fix is useful and mostly wired through the gateway, but the latest head still introduces an availability regression for one-row `sessions.list` calls by creating the shared row context, and therefore the subagent registry index, solely to carry checkpoint preview data.

## What This Changes

The PR discovers `{session}.checkpoint.<uuid>.jsonl` files when store metadata is missing and exposes them through session rows, `sessions.describe`, and compaction list/get/branch/restore paths with gateway regression tests.

## Best Possible Solution

Keep the disk-discovery behavior, but pass checkpoint preview data through a narrow path that does not force `SessionListRowContext` or `buildSubagentRunReadIndex` for one-row lists.

## Reproduction Assessment

Yes, from source inspection: current main only reports stored checkpoint metadata, while the PR source shows disk discovery for checkpoint files and also shows the remaining one-row list context regression. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the disk-discovery direction is the right bounded fix, but the implementation should not make one-row `sessions.list` build the shared subagent registry context just to carry checkpoint previews.
