## Summary

Keep open: this is a focused gateway bug fix with sufficient terminal proof, but the current PR head still dedupes disk discovery against raw stored checkpoint metadata. The latest maintenance pass fixed restore persistence and session-row preview filtering, but malformed stored metadata can still hide a valid checkpoint transcript from the async list/get/branch/restore path.

## What This Changes

The PR adds gateway-side discovery of `{session}.checkpoint.<uuid>.jsonl` files so session rows and `sessions.compaction.*` APIs can list, get, branch, and restore disk checkpoints when `sessions.json` lacks checkpoint metadata.

## Best Possible Solution

Keep disk discovery in the gateway session helper/API boundary, but validate stored checkpoint metadata before using it for async dedupe or output and cover the malformed-metadata recovery case before merge.

## Reproduction Assessment

Yes, source-backed. Current `main` derives session rows and `sessions.compaction.*` responses from stored checkpoint metadata, while the linked report and PR proof cover valid checkpoint transcript files that exist on disk without historical metadata.

## Solution Assessment

No, not yet. Gateway-side disk discovery is the right maintainable boundary, but the async helper should validate stored checkpoint metadata before dedupe/output so malformed persisted records cannot hide valid disk files.
