## Summary

Keep open: this PR addresses a valid metadata-missing checkpoint bug and has sufficient terminal proof, but the latest head still lets restore select a disk-discovered checkpoint while preserving only stored metadata, which drops checkpoint history after restore.

## What This Changes

The PR updates gateway session compaction APIs and session rows to discover checkpoint transcript files from disk when stored session metadata lacks compaction checkpoint entries.

## Best Possible Solution

Keep the gateway disk-discovery approach, but have restore materialize or preserve the discovered checkpoint list for the restored entry and add a disk-discovered restore regression test.

## Reproduction Assessment

Yes, source-backed. Current main uses only `entry.compactionCheckpoints` for list/get/row projection, so a metadata-missing session with valid checkpoint transcript files returns empty checkpoint APIs/UI; I did not run a live gateway repro in this read-only pass.

## Solution Assessment

No, not as-is. Gateway-side disk discovery is the right boundary, but restore must preserve disk-derived checkpoint history the same way metadata-backed restore does.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:
