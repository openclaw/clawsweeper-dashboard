## Summary

Keep open: this PR addresses a real gateway checkpoint-list bug, but the current patch still lets malformed stored checkpoint metadata suppress discovery of a valid checkpoint file with the same id, so list/get/branch/restore can remain broken for a recoverable session.

## What This Changes

The branch adds gateway disk discovery of checkpoint transcript files and wires discovered checkpoints into session rows, compaction list/get/branch/restore, with focused regression tests and small cron/pairing test adjustments.

## Best Possible Solution

Land the gateway disk-discovery fix after the async helper ignores or repairs malformed stored checkpoint metadata instead of letting it hide a valid disk-backed checkpoint.

## Reproduction Assessment

Yes, source-backed. Current main reads checkpoint rows and compaction APIs from stored metadata only, while the linked bug provides sessions with valid checkpoint files on disk but missing historical metadata.

## Solution Assessment

Not yet. Gateway-side disk discovery is the right boundary, but the async dedupe needs to validate stored checkpoint entries before using their ids to suppress disk discovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Skip malformed metadata when deduping disk checkpoints:** `src/gateway/session-compaction-checkpoints.ts:526`
  - body: The new disk scan dedupes by every raw stored checkpoint id before validating the stored checkpoint shape. If `sessions.json` contains a malformed entry with `checkpointId` X and a valid `{session}.checkpoint.X.jsonl` exists, the scan skips the disk file, so `sessions.compaction.list/get/branch/restore` keep returning or using the malformed metadata instead of the recoverable checkpoint. Current main already filters malformed checkpoint metadata for row projection; the async helper should do the same or otherwise allow disk discovery when the stored entry lacks usable checkpoint data.
  - confidence: 0.87
