## Summary

This PR is still the right fix path for the linked checkpoint visibility bug, but the latest patch still lets partially malformed stored checkpoint metadata hide or break recovery from a valid disk checkpoint with the same id.

## What This Changes

The branch adds gateway disk discovery for `{session}.checkpoint.<uuid>.jsonl` files across session rows, compaction list/get, branch, and restore flows.

## Best Possible Solution

Land the disk-discovery approach after the async API path validates stored checkpoints for full restore/list usability before dedupe/output and covers partial stored metadata with a matching disk checkpoint.

## Reproduction Assessment

Yes, source-backed. Current main derives session rows and compaction APIs from stored checkpoint metadata only, while the linked report and PR proof show valid checkpoint transcript files can exist on disk without usable metadata.

## Solution Assessment

No, not yet. Gateway-side disk discovery is the right boundary, but the async list/get/branch/restore path should validate stored checkpoint records more strictly before deduping or returning them.
