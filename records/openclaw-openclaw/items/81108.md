## Summary

This PR still appears necessary: current main reads compaction checkpoints from stored session metadata only, while the branch recovers valid checkpoint transcript files from disk for session previews and compaction list/get/branch/restore. I did not find a concrete blocking code defect in the current diff, but the change intentionally expands session-state recovery semantics and still needs maintainer acceptance plus green exact-head checks before merge.

## What This Changes

The PR adds gateway discovery of valid `{session}.checkpoint.<uuid>.jsonl` files, uses discovered checkpoints in session preview and compaction list/get/branch/restore paths, and adds regression coverage for missing or incomplete checkpoint metadata.

## Best Possible Solution

Land the bounded disk-discovery fallback if maintainers accept valid checkpoint transcript files as authoritative recovery state; otherwise keep the metadata-only model and route the linked bug to an explicit session-store migration or product decision.
