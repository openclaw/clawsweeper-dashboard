## Summary

Keep open: the gateway disk-discovery direction is correct and the supplied terminal proof is useful, but latest head still feeds raw stored checkpoint metadata into the async discovery merge, so malformed persisted entries can mask a valid disk checkpoint file.

## What This Changes

The PR adds gateway discovery of `{session}.checkpoint.<uuid>.jsonl` files to session list and compaction list/get/branch/restore flows, with regression tests.

## Best Possible Solution

Land the gateway disk-discovery approach after the async helper validates stored checkpoint metadata before dedupe/output and adds malformed-metadata coverage for the disk-backed API path.

## Reproduction Assessment

Yes, source-backed. Current main derives session rows and compaction APIs from stored metadata, while the linked report and PR proof show valid checkpoint transcript files can exist on disk without matching `compactionCheckpoints` metadata.

## Solution Assessment

No, not yet. Gateway-side disk discovery is the right maintainable boundary, but the async helper should validate stored metadata before it dedupes or returns checkpoint entries.
