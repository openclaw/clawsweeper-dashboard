## Summary

Current main and v2026.6.8 still intentionally isolate ACPX Codex `CODEX_HOME`; the requested plugin/skill inheritance remains a security-boundary feature decision, with https://github.com/openclaw/openclaw/pull/77006 as an unmerged and currently conflicting candidate path rather than a reason to close.

## What This Changes

The issue requests that ACPX Codex background tasks inherit or explicitly use user-installed Codex plugins/skills from the user's personal Codex home instead of only the isolated ACPX Codex home.

## Best Possible Solution

Keep ACPX isolated by default and decide whether to land a documented, explicit opt-in path for trusted Codex home, marketplace, or skill-root use with focused tests and security/upgrade proof.

## Reproduction Assessment

Yes. Source inspection shows current main still generates an isolated ACPX `codex-home`, passes it as `CODEX_HOME`, and has tests/docs that preserve that boundary rather than inheriting personal Codex plugins; I did not run a live WSL2/Superpowers reproduction.

## Solution Assessment

Unclear until maintainers choose the trust boundary. Silent inheritance is not the best default because it weakens isolation; the maintainable path is an explicit opt-in Codex-home, marketplace, or skill-root design with proof.
