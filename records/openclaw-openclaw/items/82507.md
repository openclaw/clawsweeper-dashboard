## Summary

Keep open: current main still keeps ACPX Codex on an isolated generated `CODEX_HOME` and has no merged plugin inheritance, marketplace allowlist, or explicit Codex-home override for ACPX. The request is a real feature and policy decision, not an implemented fix or a ClawHub-only bundle, because the missing behavior sits in the ACPX wrapper boundary.

## What This Changes

The issue requests that ACPX Codex background tasks inherit or explicitly receive user-installed Codex plugins such as Superpowers from the user's normal Codex home.

## Best Possible Solution

Keep ACPX isolated by default, and if maintainers accept the feature, add an explicit allowlisted Codex home/plugin/marketplace path with public docs, focused ACPX tests, and redacted real ACPX/Codex proof.

## Reproduction Assessment

Source-reproducible, but not live-reproduced here. Current main forces the generated ACPX `codex-home`, the schema has no override, and docs say native Codex plugin config does not apply to ACP bindings.

## Solution Assessment

No, automatic inheritance of the operator's personal Codex plugins is not clearly the safest default. An explicit opt-in home or allowlist path, such as the related open PR after maintainer/security review, is the safer direction.
