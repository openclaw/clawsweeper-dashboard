## Summary

Keep open for maintainer product/security review: current main still launches Codex ACPX with an isolated generated CODEX_HOME and has no ACPX plugin-inheritance, marketplace allowlist, or Codex-home override path on main; the related explicit-home override PR is open and conflicting, not a shipped fix.

## What This Changes

The issue asks ACPX Codex background tasks to inherit or explicitly access user-installed Codex plugins, marketplaces, or skill directories such as Superpowers.

## Best Possible Solution

Keep ACPX isolated by default, and if maintainers accept the capability, add an explicit allowlisted Codex ACP home/plugin/marketplace path with public docs, focused ACPX tests, and redacted real-runtime proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source, tests, and docs do confirm current main uses an isolated generated CODEX_HOME for ACPX Codex and does not apply native Codex plugin config, marketplaces, or skills.load.extraDirs to ACP bindings.

## Solution Assessment

No, implicit inheritance of the operator’s personal Codex plugins is not clearly the safest default. The maintainable direction is an explicit, reviewed allowlist or Codex-home override path, or a clear maintainer decision to keep ACPX isolated and route users to migration/ClawHub-managed skills instead.
