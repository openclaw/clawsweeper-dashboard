## Summary

Current main and the latest release still intentionally isolate ACPX Codex `CODEX_HOME`, and the requested plugin inheritance/marketplace path remains a feature request with a security boundary decision. The issue should stay open while maintainers decide whether the explicit override work in https://github.com/openclaw/openclaw/pull/77006 is the right lane.

## What This Changes

The issue requests that ACPX-managed Codex sandboxes inherit or otherwise expose user-installed Codex plugins such as Superpowers from the operator's personal Codex home.

## Best Possible Solution

Keep ACPX isolated by default and decide whether to land an explicit, documented opt-in Codex-home or marketplace/plugin path with focused tests and upgrade/security proof.

## Reproduction Assessment

Yes, from source inspection rather than live WSL2/Superpowers execution: current main generates `state/acpx/codex-home`, sets `CODEX_HOME` to it, and has tests asserting source Codex home state is not copied wholesale. This is a feature gap, not a broken documented behavior.

## Solution Assessment

No, silent inheritance is not clearly the best solution because it weakens a deliberate isolation boundary. The best fix is an explicit maintainer-approved opt-in path, likely evaluated through the existing Codex-home override PR or a narrower migration/config design.
