## Summary

Keep open for maintainer product/security review: current main intentionally isolates Codex ACPX `CODEX_HOME` and does not expose an ACPX plugin-inheritance knob, so the requested behavior is a real feature request rather than an already-fixed bug or ClawHub-only bundle.

## What This Changes

The issue requests that ACPX Codex background tasks inherit or install user Codex plugins such as Superpowers inside the OpenClaw-managed Codex home.

## Best Possible Solution

Keep ACPX isolated by default, and if accepted, add an explicit allowlisted Codex ACP plugin/marketplace migration path with docs and focused ACPX tests.

## Reproduction Assessment

No high-confidence live reproduction was run. Source and docs do confirm the current main behavior: Codex ACPX gets an isolated generated `CODEX_HOME`, while existing native Codex plugin config does not apply to ACP bindings.

## Solution Assessment

No, automatic inheritance of the operator's `~/.codex` plugins is not clearly the best default. A safer maintainable direction is an explicit allowlist or migration/install command that preserves the current isolation boundary unless the operator opts in.
