## Summary

Keep open for maintainer product/security review: current main intentionally runs Codex ACPX with an isolated generated `CODEX_HOME`, and the existing native Codex plugin configuration is documented as not applying to ACP conversation bindings. The related explicit-home override PR is a plausible candidate direction, but it is still open and not a reason to close this feature request as implemented.

## What This Changes

Request that ACPX Codex background tasks inherit, sync, or explicitly configure user-installed Codex plugins/skills/marketplaces such as Superpowers.

## Best Possible Solution

Keep ACPX isolated by default; if accepted, add an explicit allowlisted Codex ACP home/plugin/marketplace configuration path with public docs, focused ACPX tests, and redacted real Codex ACP proof.

## Reproduction Assessment

No high-confidence live reproduction was run for WSL2 plus Superpowers. Source, tests, and docs do confirm the reported current-main behavior: ACPX Codex uses an isolated generated `CODEX_HOME`, and native `codexPlugins` config does not apply to ACP bindings.

## Solution Assessment

No, automatic inheritance of the operator's `~/.codex` plugins is not clearly the best default. A safer maintainable direction is an explicit allowlist, explicit home override, or migration/install flow that preserves isolation unless the operator opts in.
