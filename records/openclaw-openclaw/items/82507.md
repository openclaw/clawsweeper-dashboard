## Summary

Keep open for maintainer product/security review: current main still gives Codex ACPX an isolated generated `CODEX_HOME` and does not expose a plugin-inheritance, marketplace allowlist, or explicit plugin-home path for ACP background tasks.

## What This Changes

The issue requests that ACPX Codex background tasks inherit or explicitly load user-installed Codex plugins, skills, and marketplaces from the user's Codex home.

## Best Possible Solution

Keep ACPX isolated by default, and if maintainers accept this direction, add an explicit allowlisted Codex ACP home/plugin/marketplace path with public docs, focused ACPX tests, and redacted real-runtime proof.

## Reproduction Assessment

No live reproduction was run, but source inspection gives a high-confidence current-main path: ACPX forces a generated Codex home and has no plugin inheritance or marketplace allowlist config. This is a feature gap rather than a broken existing contract.

## Solution Assessment

No, implicit inheritance of the operator's personal Codex plugins is not clearly the safest default. A narrower maintainable solution would be explicit, allowlisted, documented, and reviewed against the ACPX isolation and credential boundary.
