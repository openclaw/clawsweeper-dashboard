## Summary

Keep open for maintainer product/security review: current main still isolates Codex ACPX `CODEX_HOME` and has no ACPX plugin-inheritance or marketplace allowlist path, so this is a real feature request rather than an already-fixed bug or ClawHub-only bundle.

## What This Changes

The issue requests a way for Codex ACPX background tasks to see user-installed Codex plugins such as Superpowers from the operator's personal Codex environment.

## Best Possible Solution

Keep ACPX isolated by default, and if accepted, add an explicit allowlisted Codex ACP plugin/home/marketplace path with public docs, focused ACPX tests, and redacted real-runtime proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source, tests, docs, and the pinned adapter contract do confirm current main uses an isolated generated Codex home and does not apply native Codex plugin config to ACP bindings.

## Solution Assessment

No, automatic inheritance of the operator's personal Codex plugins is not clearly the safest default. A maintainable solution should be explicit and allowlisted, or use a reviewed Codex-home override path such as the related open PR after maintainer/security review.
