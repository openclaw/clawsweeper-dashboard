## Summary

Keep open: current main and the latest release still intentionally isolate ACPX Codex in a generated `CODEX_HOME`, and the overlapping explicit-home PR is open but not merged. The remaining work is a product/security decision about whether ACPX may trust personal Codex plugin, marketplace, skill, and credential state.

## What This Changes

The issue requests a way for ACPX Codex background tasks to use user-installed Codex plugins or skills from the user’s personal Codex home instead of only the isolated ACPX Codex sandbox.

## Best Possible Solution

Keep ACPX isolated by default and land only a documented, maintainer-approved opt-in path for trusted Codex home, marketplace, or skill-root use with focused tests and redacted live proof.

## Reproduction Assessment

Yes, at source level. Current main still generates an isolated ACPX `codex-home`, passes it as `CODEX_HOME`, and has docs/tests that preserve that boundary rather than inheriting personal Codex plugins; I did not rerun a live WSL2/Superpowers reproduction.

## Solution Assessment

No for silent inheritance, unclear for the final approved design. The maintainable solution is an explicit opt-in Codex-home, marketplace, or skill-root path after maintainers choose the trust boundary and proof expectations.
