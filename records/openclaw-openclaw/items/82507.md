## Summary

Keep open for maintainer product/security review: current main still isolates ACPX Codex `CODEX_HOME` and has no ACPX plugin inheritance, marketplace allowlist, or explicit Codex-home override on main. The related explicit-home PR is open and conflicting, so it is a candidate path rather than proof that this request is implemented.

## What This Changes

Request ACPX Codex background tasks to inherit user-installed Codex plugins or configured skill/plugin locations such as Superpowers instead of seeing only the isolated sandbox defaults.

## Best Possible Solution

Keep ACPX isolated by default, and if accepted, add an explicit allowlisted Codex ACP home/plugin/marketplace path with public docs, focused tests, and redacted real-runtime proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms current main uses an isolated generated Codex home and does not apply native Codex plugin config or OpenClaw `skills.load.extraDirs` to ACPX Codex sessions.

## Solution Assessment

No, automatic inheritance of the operator's personal Codex plugins is not clearly the safest default. The narrower maintainable path is an explicit, documented, allowlisted Codex-home or marketplace override after maintainer/security review.
