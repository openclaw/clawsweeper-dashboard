## Summary

Keep open. The duplicate-greeting path is still present on current main and in the latest shipped release: Voice Call queues an explicit initial greeting while the OpenAI realtime bridge also leaves server-VAD automatic response creation enabled by default, and no linked PR or duplicate currently owns the fix.

## What This Changes

The issue reports duplicate outbound OpenAI realtime voice-call greetings when both Voice Call’s explicit initial greeting and OpenAI server-VAD auto response create responses.

## Best Possible Solution

Keep one opening source of truth for OpenAI outbound realtime calls: preserve the configured Voice Call initial greeting, suppress only the competing first OpenAI server-VAD auto response, and leave Google plus later OpenAI turns unchanged.

Yes, at source level: a non-empty outbound initial message makes Voice Call call triggerGreetingOnReady, while the OpenAI realtime session defaults server_vad.create_response to true. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes for a narrow provider-aware repair; the best path is first-turn collision handling for OpenAI, not dropping the explicit greeting or disabling auto responses globally. Regression coverage should assert the startup sequence creates only one opening response while normal later turns still auto-respond.

## Repair Work Prompt

Fix the duplicate outbound OpenAI realtime voice-call greeting reported in https://github.com/openclaw/openclaw/issues/85846. Current main queues Voice Call initialGreetingInstructions and triggerGreetingOnReady, while extensions/openai/realtime-voice-provider.ts defaults server_vad.create_response to true, so a caller’s first audio can trigger an OpenAI auto response while the explicit greeting also sends response.create. Implement a provider-aware first-turn collision fix that preserves the configured explicit initial greeting, suppresses only the competing first OpenAI server-VAD auto response, leaves Google realtime behavior unchanged, and keeps normal later OpenAI turns auto-responding. Do not solve this by disabling autoRespondToAudio for the whole bridge or by dropping initialGreetingInstructions globally. Add focused regression coverage for the startup sequence and the later-turn compatibility path; add a CHANGELOG.md fix entry unless maintainers decide release notes will be handled separately. If live Twilio/OpenAI credentials are available, capture redacted proof that an outbound conversation call with --message speaks exactly one greeting.
