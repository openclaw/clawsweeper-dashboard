## Summary

Keep open: current main and v2026.6.9 still combine OpenAI server-VAD auto-response with the explicit voice-call ready-trigger greeting path, while the linked fix candidate remains open and unmerged.

## What This Changes

The issue reports duplicate outbound OpenAI realtime voice-call greetings when an initial message is queued and both server VAD auto-response and the explicit greeting trigger create responses.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/86285, or an equivalent narrow OpenAI-provider-owned fix with live Twilio/OpenAI proof, then close this issue from the merged implementation.

## Reproduction Assessment

Yes, at source level: current main queues an explicit ready-trigger greeting while OpenAI server VAD auto-response remains enabled by default. I did not place a fresh live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes, the best fix direction is narrow OpenAI-specific first-turn collision handling while preserving the shared explicit greeting path for non-OpenAI providers such as Google. Broadly dropping initial greetings or disabling auto-response for the whole call would be riskier.
