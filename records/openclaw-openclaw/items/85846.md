## What This Changes

Outbound OpenAI realtime voice calls with a non-empty initial message can produce two startup greetings because Voice Call triggers an explicit greeting while OpenAI server VAD auto-response is enabled.

## Best Possible Solution

Land a narrow OpenAI-only first-turn suppression fix with redacted live-call proof, then close this issue from the merged implementation.

## Reproduction Assessment

Yes, at source level and with the reporter's five-call live repro: current main queues triggerGreetingOnReady while OpenAI VAD create_response defaults true. I did not place a fresh Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes, the best fix direction is narrow OpenAI-only first-turn collision handling around the explicit greeting. Broadly dropping greetings or disabling auto responses for the whole call would risk Google behavior or later OpenAI turns.
