## Summary

Keep open. Current main and v2026.5.20 still have both startup paths: Voice Call fires an explicit initial greeting on provider ready and OpenAI defaults server-VAD auto responses to enabled, so the reported double greeting remains source-reproducible.

## What This Changes

The issue reports duplicated outbound OpenAI Realtime Voice Call greetings when an initial message is queued because both Voice Call `triggerGreeting` and OpenAI `server_vad.create_response` can request a response.

## Best Possible Solution

Keep one opening source of truth for OpenAI outbound realtime calls by preserving the explicit Voice Call initial greeting and suppressing only the competing first OpenAI server-VAD auto response, while leaving Google and later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: a non-empty outbound initial message makes Voice Call fire `triggerGreetingOnReady`, while the OpenAI realtime bridge sends `server_vad.create_response: true` by default. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes on the end state: the maintainable fix is provider-aware first-turn collision handling for OpenAI only. Dropping explicit greetings globally or disabling OpenAI auto responses for the whole session would be broader than needed.
