## Summary

Keep open. Current main still has the reported first-turn collision: Voice Call queues an explicit initial greeting, and the OpenAI realtime bridge defaults server VAD to create an automatic response on caller audio.

## What This Changes

The issue reports duplicate spoken greetings on outbound OpenAI realtime voice calls when both an initial message and OpenAI server-VAD auto-response are active.

## Best Possible Solution

Keep one opening source of truth for OpenAI outbound realtime calls by preserving the configured Voice Call greeting, suppressing only the competing first OpenAI server-VAD auto-response, and leaving Google plus later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: a non-empty initial message makes Voice Call fire triggerGreetingOnReady, while OpenAI realtime sessions default server_vad.create_response to true. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes for a narrow first-turn collision fix; dropping the explicit greeting or disabling auto responses globally would trade this bug for weaker configured greetings or broken later turns.
