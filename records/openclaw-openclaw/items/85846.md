## Summary

Keep open. Current main and the latest release still have the source-level collision: Voice Call queues an explicit initial realtime greeting, while the OpenAI realtime bridge leaves server-VAD auto responses enabled by default.

## What This Changes

The issue asks for outbound OpenAI realtime voice calls with a non-empty initial message to produce one greeting instead of both the plugin-triggered greeting and OpenAI server-VAD auto response.

## Best Possible Solution

Keep one opening source of truth for OpenAI outbound realtime calls: preserve the explicit configured initial greeting, suppress only the competing first OpenAI server-VAD auto response, and leave Google plus later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: a non-empty initial message makes Voice Call fire triggerGreetingOnReady, while OpenAI realtime sessions default server_vad.create_response to true. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes for a narrow first-turn collision fix, not for dropping greetings or disabling OpenAI auto responses globally. The maintainable path is to keep the configured opener and suppress only the competing first automatic OpenAI response with regression coverage.
