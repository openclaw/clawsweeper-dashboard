## Summary

Keep open. The report is recent, source-supported, and still present on current main: Voice Call queues an explicit OpenAI opening response while the OpenAI realtime bridge also enables server-VAD auto responses by default.

## What This Changes

Prevent OpenAI realtime outbound voice-call startup from emitting both the server-VAD auto response and the explicit initial greeting response for one initial message.

## Best Possible Solution

Keep one opening source of truth for OpenAI outbound realtime calls: preserve the explicit Voice Call initial greeting, suppress or avoid only the first competing OpenAI server-VAD auto response, and leave Google plus later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: a non-empty initial message makes Voice Call fire triggerGreetingOnReady, while OpenAI realtime sessions default server_vad.create_response to true. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

No to the broad workarounds: dropping the explicit greeting loses the configured opener, and disabling auto responses globally risks later turns. The narrow maintainable fix is first-turn collision handling for OpenAI only, with regression coverage around the startup sequence.
