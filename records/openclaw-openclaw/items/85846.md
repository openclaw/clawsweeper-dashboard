## Summary

Keep open. Current main still has both response sources for this outbound OpenAI realtime path: Voice Call queues an explicit initial greeting, while the OpenAI realtime provider enables server-VAD automatic responses by default.

## What This Changes

The issue reports duplicate outbound voice-call greetings when OpenAI realtime server VAD auto-response and Voice Call's explicit initial greeting both request a response.

## Best Possible Solution

Keep one opening source of truth for OpenAI outbound realtime calls by preserving the configured initial greeting while suppressing only the first competing server-VAD auto response; leave Google and later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: a non-empty initial message makes Voice Call fire triggerGreetingOnReady, while the OpenAI realtime session defaults server_vad.create_response to true. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes for a narrow provider-aware first-turn fix. Dropping the explicit greeting loses the configured opener, and disabling auto responses globally risks later turns, so the maintainable path is scoped first-turn collision handling for OpenAI only.
