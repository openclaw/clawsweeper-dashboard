## Summary

Keep open. Current main and the latest shipped tag still have two independent opening paths for outbound OpenAI realtime calls with an initial message: Voice Call triggers an explicit greeting on provider ready while the OpenAI realtime bridge leaves server-VAD automatic responses enabled by default.

## What This Changes

The issue asks OpenClaw to prevent outbound Voice Call OpenAI realtime sessions with an initial message from speaking both the OpenAI VAD auto-response and OpenClaw's explicit initial greeting.

## Best Possible Solution

Keep one opening source of truth for OpenAI outbound realtime calls by suppressing only the first competing OpenAI server-VAD auto response when an explicit initial greeting is queued, while leaving Google greetings and later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: a non-empty initial message queues Voice Call triggerGreetingOnReady, while OpenAI realtime sessions default server_vad.create_response to true. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes, the maintainable fix is provider-aware first-turn collision handling. Dropping all explicit greetings would regress Google, and disabling OpenAI auto responses globally would change later conversation turns unnecessarily.
