## Summary

Keep open: current main still has the reported OpenAI realtime startup collision, with Voice Call queuing an explicit greeting while the OpenAI bridge also enables server-VAD automatic responses by default.

## What This Changes

The issue asks OpenClaw to prevent outbound Voice Call OpenAI realtime sessions from speaking two greetings when an initial message is provided.

## Best Possible Solution

Keep one opening source of truth for OpenAI outbound realtime calls: preserve the configured initial greeting, suppress or avoid only the first competing OpenAI server-VAD auto response, and leave Google plus later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: a non-empty initial message makes Voice Call fire triggerGreetingOnReady, while OpenAI realtime sessions default server_vad.create_response to true. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes for a narrow provider-aware first-turn collision fix. Dropping explicit greetings or disabling auto responses globally would be broader than needed and risks breaking Google greetings or later OpenAI turns.
