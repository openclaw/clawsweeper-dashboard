## Summary

Keep open. Current main still wires the Voice Call explicit initial greeting together with OpenAI Realtime server-VAD auto responses, and the open closing PR has not merged yet.

## What This Changes

The issue reports duplicate outbound OpenAI realtime voice-call greetings when a non-empty initial message triggers both Voice Call's explicit greeting path and OpenAI server VAD auto-response.

## Best Possible Solution

Land a narrow OpenAI-specific first-turn collision fix, preferably through the linked PR after proof, preserving the configured explicit greeting while leaving Google and later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: current main still queues an explicit Voice Call greeting and configures OpenAI server_vad.create_response as true by default. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes, the maintainable direction is to keep one startup response owner for OpenAI by suppressing the first competing VAD auto-response, then restoring normal later turns. Dropping the explicit greeting globally would regress Google or configured opener behavior.
