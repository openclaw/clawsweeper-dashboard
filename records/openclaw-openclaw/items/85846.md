The issue reports duplicate outbound OpenAI realtime voice greetings when voice-call initial greeting triggering and OpenAI server VAD auto-response can both create responses.

## Best Possible Solution

Review and land the linked OpenAI-only first-turn suppression fix after redacted live Twilio/OpenAI proof, then close this issue from the merged implementation.

## Reproduction Assessment

Yes, at source level and from the reporter's five-call live repro: current main queues `triggerGreetingOnReady` while OpenAI VAD `create_response` defaults true. I did not place a fresh Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes, the best fix direction is narrow OpenAI-only first-turn collision handling around the explicit greeting path. Broadly dropping greetings or disabling auto responses for the whole call would risk other providers or later OpenAI turns.
