## Summary

Keep open. Current main and the latest release still have both startup paths: Voice Call queues an explicit initial greeting, while the OpenAI realtime bridge defaults server-VAD response creation on, so the reported duplicate greeting remains a valid source-reproducible bug.

## What This Changes

The issue reports duplicate outbound OpenAI realtime voice-call greetings when a non-empty initial message combines Voice Call's explicit greeting trigger with OpenAI server-VAD automatic response creation.

## Best Possible Solution

Keep a single opening source of truth for OpenAI outbound realtime calls: preserve the configured explicit initial greeting, suppress only the competing first OpenAI server-VAD auto response, and leave Google plus later OpenAI turns unchanged.

## Reproduction Assessment

Yes, at source level: a non-empty initial message makes Voice Call trigger an explicit greeting on provider ready, while OpenAI realtime sessions default server_vad.create_response to true. I did not place a live Twilio/OpenAI call in this read-only review.

## Solution Assessment

Yes, the maintainable direction is narrow first-turn collision handling for OpenAI only. Dropping the explicit greeting or disabling auto-response globally would be broader than the reported bug and could change Google or later OpenAI turns.
