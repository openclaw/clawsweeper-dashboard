I found no blocking correctness or security findings, but the PR intentionally rewrites legacy cron config, so maintainers should explicitly accept that compatibility impact before merge.

## What This Changes

The branch changes cron doctor's legacy notify migration to drop top-level `notify: true` when `cron.webhook` is unset, adds regression tests, and updates cron doctor docs.

## Best Possible Solution

Land the doctor cleanup with the updated docs once maintainers accept the compatibility impact of deleting the inert legacy marker.

## Reproduction Assessment

Yes. Source inspection on current main shows doctor counts `notify === true` for the preview and then preserves that marker when `cron.webhook` is unset, so a store with `delivery.mode: none` will be re-detected on the next doctor pass.

## Solution Assessment

Yes. Doctor is the right owner for legacy config repair, runtime delivery is controlled by `delivery` rather than top-level `notify`, and the patch preserves delivery semantics while keeping the invalid configured-webhook warning.
