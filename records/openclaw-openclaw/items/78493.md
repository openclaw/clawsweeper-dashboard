## Summary

This should stay open: current main has stronger clobber detection, but source inspection still shows a write path can proceed from an EACCES-derived fallback snapshot, and the sudo/update state-ownership guard requested by the report is not present.

## What This Changes

The issue asks OpenClaw to fail closed on unreadable config writes and prevent sudo-run update/plugin phases from creating root-owned user state.

## Best Possible Solution

Add a fail-closed config write guard for unreadable/EACCES source snapshots and reject sudo/root update plugin-state mutations against a user-owned OpenClaw state directory, with regression coverage for both paths.

## Reproduction Assessment

Yes, from source inspection rather than a live macOS run: an EACCES read produces an invalid fallback snapshot, and the final config write guard does not treat that unreadable base as a blocking condition.

## Solution Assessment

Yes, the requested fail-closed behavior is the narrow maintainable fix; recovery commands can still restore last-good explicitly, but ordinary doctor/update writes should not replace an unreadable existing config or mutate user state as root.

On current main, config EACCES/read failures produce an invalid fallback snapshot, and later doctor/config writes can still persist a skeletal config because unreadable-base/missing-meta-before-write is not a hard blocking reason.
