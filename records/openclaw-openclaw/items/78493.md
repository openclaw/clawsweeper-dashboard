## Summary

Keep open: current main now has a root CLI guard that mitigates the sudo mutation path, but the unsafe config-write path is still present on main and the linked fix is an open PR, not merged. The fresh Linux/systemd comment corroborates the ownership class, so this should wait for maintainer review of the fix PR and any remaining ownership follow-up.

## What This Changes

The issue requests fail-closed handling for unreadable OpenClaw config writes and sudo/root commands that leave user state or plugin runtime files root-owned.

## Best Possible Solution

Land the focused config write-guard fix from https://github.com/openclaw/openclaw/pull/79299 after maintainer/CI review, then close this issue only if current root-guard coverage is accepted for the sudo ownership path or a narrower follow-up tracks any remaining plugin-runtime ownership gap.

Yes, source-reproducible. EACCES in readConfigFileSnapshotInternal produces a valid:false fallback snapshot, and the current write guard does not treat unreadable-base or last-known-good shape loss as blocking reasons; the issue comments add real macOS and Linux observations.

## Solution Assessment

Yes for the config half: extending the existing rejected-payload write guard is the narrow maintainable fix. The sudo ownership half is partly mitigated by current main's root guard, but maintainers should confirm whether explicit root escape hatches need separate handling.
