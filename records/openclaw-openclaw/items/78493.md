## Label Justifications

- `P1`: The issue reports a real update/doctor workflow that can clobber config and leave the gateway unable to start.
- `impact:data-loss`: The central failure mode is overwriting a rich user config with a skeletal fallback-derived config after an EACCES read failure.
- `impact:crash-loop`: The missing gateway.mode and root-owned plugin/runtime-deps artifacts can prevent gateway startup or cause repeated systemd restarts.

## Summary

Keep open: current main and v2026.6.9 still allow the unreadable-config fallback to feed later config writes, and the sudo/wrong-UID update/plugin-state mutation policy is still unresolved.

## What This Changes

The issue requests fail-closed config writes and sudo/wrong-UID state mutation guards for update, doctor, and plugin repair workflows.

## Best Possible Solution

Define a maintainer-approved recovery contract that rejects unreadable/last-good-destructive config writes before commit and prevents sudo/root update or plugin repair paths from mutating user-owned state unless they run as the target owner or an explicit recovery mode.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: EACCES returns a valid:false fallback snapshot, the writer can continue after reread failure, and update doctor passes allowConfigSizeDrop=true.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the wrong-UID policy; a config write guard is the narrow fix for the data-loss half, but update/plugin-state ownership needs an explicit product decision.
