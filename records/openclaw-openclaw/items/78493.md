## Summary

Keep open: current main now blocks ordinary root CLI entrypoints, but the EACCES/unreadable-config overwrite path is still source-reproducible because invalid fallback snapshots are not a hard write blocker.

## What This Changes

The issue asks OpenClaw to fail closed when sudo/root-owned state or unreadable config snapshots could let doctor/update overwrite a user-owned config or plugin state directory.

## Best Possible Solution

Add a fail-closed write guard for unreadable/EACCES config snapshots, save rejected payloads without replacing the live config, and keep sudo/root state mutations blocked or explicitly diagnosed with regression coverage.

## Reproduction Assessment

Yes. A focused source-level reproduction can mock an EACCES config read, observe `snapshot.valid === false`, and then exercise doctor/replaceConfigFile to show current write guards do not block the unreadable base before commit.

## Solution Assessment

Yes. The narrow maintainable fix is to make unreadable existing config a hard blocker for ordinary full-file writes while leaving explicit recovery commands responsible for restoring last-good state.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78493 as a narrow bug fix. Current main already has a root CLI guard, so focus on the remaining data-loss path: when an existing `openclaw.json` cannot be read due to EACCES/unreadable permissions, ordinary config writes from `writeConfigFile`/`replaceConfigFile`/doctor must fail closed, leave the live config untouched, and save/log a rejected candidate when a write was attempted. Do not add startup/runtime migrations or broad cache layers. Keep explicit recovery/doctor repair commands responsible for restoring last-good config. Add regression coverage for unreadable existing config plus doctor writes, severe last-good clobber signatures, and root/sudo plugin-state behavior to the extent the existing root guard should block it. Preserve `OPENCLAW_ALLOW_ROOT=1` and container bypass semantics unless an existing test contract says otherwise. Add a changelog entry because this is user-facing data-loss protection.
