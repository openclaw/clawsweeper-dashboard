## Summary

Keep open: current main has clobber rejection for valid snapshots, but the EACCES fallback snapshot and doctor write path still lack a hard fail-closed guard, and the update plugin phase still lacks a sudo/user-state ownership guard.

## What This Changes

The issue requests fail-closed config writes and safer update behavior when `sudo openclaw update` creates mixed ownership and `doctor` later cannot read the existing config.

## Best Possible Solution

Add a core fail-closed write guard for unreadable/EACCES source snapshots, and add an update/plugin-state guard that refuses root-run mutations against a user-owned OpenClaw state directory unless an explicit recovery path owns the repair.

## Reproduction Assessment

Yes, from source inspection rather than a live macOS run: an EACCES read produces an invalid fallback snapshot, and the final write guard does not treat that unreadable base as a blocking condition.

## Solution Assessment

Yes, fail-closed behavior in the existing config writer and update state mutation boundary is the narrowest maintainable fix; explicit recovery commands can remain the path for restoring last-good config.
