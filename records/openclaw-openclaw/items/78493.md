## Summary

Keep open: current main and v2026.6.10 still allow an unreadable-config fallback snapshot to reach later config writes, and the sudo/wrong-UID update/plugin-state mutation policy remains unresolved.

## What This Changes

The issue reports that sudo or wrong-UID OpenClaw maintenance commands can leave mixed-owned state and let doctor/config repair overwrite a valid config after an EACCES read fallback.

## Best Possible Solution

Add a fail-closed config write boundary for unreadable/EACCES snapshots, then settle and implement the wrong-UID maintenance policy so update/plugin commands do not create root-owned user state.

## Reproduction Assessment

Yes at source level: EACCES returns a valid:false fallback snapshot, and later write paths can still proceed without an unreadable-config blocker. The latest issue comment also reports the same path still reproduces on v2026.6.10.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is a hard unreadable-config write blocker plus a maintainer-approved wrong-UID update/plugin mutation policy.
