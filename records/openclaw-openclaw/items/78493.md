## Summary

Keep open: current main still does not block config writes after an unreadable config fallback, and the sudo/wrong-UID plugin/state mutation policy remains unresolved despite adjacent fixes for npm cache ownership and update recovery guidance.

## What This Changes

The issue asks OpenClaw to prevent sudo-driven update or doctor paths from creating mixed root/user-owned state and from overwriting a previously valid config after EACCES/read-failure fallback.

## Best Possible Solution

Add a maintainer-approved guard that rejects unreadable/last-good-destructive config writes before commit and prevents sudo/root update or plugin repair paths from mutating user-owned state unless they run as the target owner or enter an explicit recovery mode.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at the source level: current main converts EACCES into a valid:false fallback snapshot, then the write path can continue after reread failure without the last-good blockers proposed in the closed PR; I did not perform a live sudo update rerun.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the wrong-UID policy; the config half has a clear narrow guard shape, but the sudo update/plugin-state half needs an explicit fail-closed, re-exec, or ownership-repair decision.
