## Label Justifications

- `P1`: The issue describes a real update/doctor workflow that can clobber config and leave the gateway unable to start.
- `impact:data-loss`: The central failure mode is overwriting a rich user config with a skeletal fallback-derived config after an EACCES read failure.
- `impact:crash-loop`: The missing gateway.mode and root-owned plugin/runtime-deps artifacts can prevent gateway startup or cause repeated service restarts.

## Summary

Keep open: current main still lets an EACCES/unreadable config snapshot flow into later config writes without a dedicated unreadable-config blocker, and the sudo/wrong-UID update/plugin-state policy remains unresolved despite fresh reporter confirmation on v2026.6.10.

## What This Changes

Prevent sudo or wrong-UID OpenClaw update/doctor paths from creating root-owned user state and prevent config writes after EACCES from overwriting a previously valid config.

## Best Possible Solution

Define a maintainer-approved recovery contract that rejects config writes after an unreadable config and prevents sudo/root update or plugin doctor paths from mutating user-owned state unless they run as the target owner or an explicit recovery mode.

## Reproduction Assessment

Yes, at source level: EACCES returns a valid:false fallback snapshot, the writer can continue after reread failure, and update doctor can run repair writes during update. I did not run a live sudo/macOS or systemd reproduction in this read-only review.

## Solution Assessment

Unclear as a full fix until maintainers choose the wrong-UID policy; a config write guard is the narrow fix for the data-loss half, but sudo/plugin-state ownership needs an explicit product decision.
