## Summary

Keep open. This protected maintainer PR still targets a real current-main mismatch: `src/agents/agent-tools.ts` exposes `write`, `edit`, and `apply_patch` for `workspaceAccess: "none"`, while the sandbox mount and bridge layers only allow writes for `"rw"`; however, the branch is now conflicting after the `pi-tools` to `agent-tools` rename and needs a current-main port before it can land.

## What This Changes

The PR hides sandboxed `write`, `edit`, and `apply_patch` tools unless `workspaceAccess` is `"rw"`, adds focused tool-surface tests, and adds one changelog entry.

## Best Possible Solution

Port the narrow `workspaceAccess === "rw"` tool-surface gate to current `src/agents/agent-tools.ts`, add current tests for `none`/`ro`/`rw` including `apply_patch`, drop the changelog edit, and leave writable-`none` or writable-bind semantics to the canonical sandbox policy issues.

## Reproduction Assessment

Yes, source-level. Current `main` exposes write-capable tools for `workspaceAccess: "none"` via `allowWorkspaceWrites !== "ro"`, while the sandbox bridge and mount metadata reject writes unless the access mode is `"rw"`.

## Solution Assessment

Yes for the linked closed bug at https://github.com/openclaw/openclaw/issues/65316, but not as a land-ready branch. The narrow gate is the right layer for the misleading tool surface, while writable-`none` and writable-bind semantics need separate maintainer policy decisions.
