- `impact:auth-provider`: The requested behavior changes API-key selection, provider quota classification, cooldown scope, and model/auth failover routing.

## Summary

Keep open: current main and the latest release support generic multi-key collection, caller-scoped key rotation, auth-profile cooldowns, and embedded profile failover, but they still do not expose a shared Gemini project/quota-bucket identity that can cool all keys in one bucket while selecting another bucket.

## What This Changes

Request first-class provider key pools with bucket-aware cooldown/backoff for 429 and RESOURCE_EXHAUSTED failures, especially Gemini project-scoped quotas.

## Best Possible Solution

Define an additive provider-owned quota-scope contract on auth candidates and apply cooldown/selection against that identity in the existing auth-profile health and embedded failover paths while preserving current profile/model fallback defaults.

## Reproduction Assessment

Yes at source level: current main has key/profile rotation and profile/model cooldown state, but no quota-bucket identity that groups multiple Gemini keys or projects for cooldown and selection. No live exhausted Gemini project was exercised in this read-only review.

## Solution Assessment

No for the proposed log-watching sidecar and auth-file rewrite; the maintainable fix is a typed quota-scope contract integrated with existing auth candidates, cooldown state, and embedded failover. The exact quota identity and config/API surface need maintainer product judgment first.
