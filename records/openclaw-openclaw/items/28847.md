## Summary

Keep open: current main has partial multi-key and profile-cooldown support, but the requested Gemini quota-bucket contract and embedded-runner key-list rotation gap remain unresolved and require maintainer API/product decisions.

## What This Changes

The issue requests provider key pools with bucket-aware cooldown/backoff on rate-limit errors, especially grouping Gemini keys by Google project to avoid retry storms.

## Best Possible Solution

Define an additive quota-bucket/auth key-pool contract, keep provider-specific Gemini project metadata in the provider layer where possible, and wire embedded runtime auth selection to cool down shared buckets while reusing existing key-list support with docs and regression coverage.

## Reproduction Assessment

Not applicable as a feature/config design item, not a broken pre-existing bucket contract. Source inspection gives high confidence that current main lacks quota-bucket config and embedded key-list rotation, but no live Gemini 429 repro was run.

## Solution Assessment

Unclear as submitted. The sidecar demonstrates a useful operational direction, but the upstream solution should be a typed auth/profile quota-bucket contract plus accepted embedded-runner key-list behavior, not log-tailing auth-file rewrites.
