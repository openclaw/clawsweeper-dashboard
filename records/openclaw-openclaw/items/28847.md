## Summary

Keep open: current main and v2026.6.8 cover flat API-key rotation and per-profile/model cooldowns, but they still do not model a shared quota bucket such as a Gemini project or billing account. The sidecar workaround and related provider-pool/circuit-breaker items are useful context, but none fully supersedes this narrower bucket-aware key-pool request.

## What This Changes

Request first-class provider key pools with bucket/quota-group-aware cooldown and rotation on 429/RESOURCE_EXHAUSTED, especially for Gemini projects, to avoid retry storms.

## Best Possible Solution

Define an additive provider-owned quota-scope identity on auth candidates, integrate it with existing auth-profile/key rotation and cooldown state, and keep log-watching auth-file rewrites as external workarounds rather than the core design.

## Reproduction Assessment

Yes, at source level: current main exposes flat key rotation and per-profile/model cooldowns but has no shared quota-bucket identity. I did not run a live exhausted Gemini project in this read-only review.

## Solution Assessment

No, the sidecar log-watcher and auth-file rewrite are not the best permanent OpenClaw solution. The maintainable path is a typed provider/auth quota identity integrated with existing cooldown and embedded failover paths, but the public surface needs maintainer direction.
