## Summary

Keep open: current main and v2026.6.10 cover flat API-key rotation plus profile/model cooldowns, but they still do not model a shared Gemini project or quota bucket that can be cooled down as a group. The linked Keymanager sidecar and aashari fork are useful evidence of workarounds, while the open provider-pool and quota-circuit-breaker items only partially overlap this narrower bucket-aware key-pool request.

## What This Changes

The issue requests provider key pools with Gemini project or quota-bucket-aware cooldown/backoff on 429 errors to avoid retry storms.

## Best Possible Solution

Define an additive provider-owned quota-scope identity on key/auth candidates, route 429 cooldown through that scope, preserve current defaults, and reconcile the API shape with the broader provider-profile routing design before implementation.

## Reproduction Assessment

Yes, at source level: current main exposes flat key rotation and per-profile/model cooldowns but has no shared quota-bucket identity. I did not run a live exhausted Gemini project in this read-only review.

## Solution Assessment

Partly: native bucket-aware behavior targets the right gap, but the sidecar log-watcher/auth-file rewrite should remain a workaround rather than the core design. The best fix needs a typed provider/auth quota identity and maintainer-approved config or plugin API shape.
