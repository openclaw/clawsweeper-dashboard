## Summary

Keep open: current main has generic multi-key collection, rate-limit-shaped retry helpers, auth-profile cooldowns, and profile failover, but it still lacks the requested Gemini project/quota-bucket grouping and embedded-runner key-pool contract. The reporter's Keymanager sidecar, aashari's fork note, and Jackten's embedded-runner analysis are useful prior art; the broader provider-pool issue and Z.AI PR do not fully cover this focused Gemini bucket/key-list gap.

## What This Changes

Request native provider key pools with rate-limit cooldown/backoff and Gemini project/quota-bucket grouping so exhausted buckets are avoided without restarting the gateway.

## Best Possible Solution

Define an additive provider auth-pool/quota-bucket contract, then wire embedded runtime key/profile selection and cooldown state through that contract with docs and focused regression coverage while preserving current profile-fallback defaults.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature/config design request. Source inspection gives high confidence that current main lacks bucket-aware key-pool behavior, but no live Gemini 429 project-bucket repro was run.

## Solution Assessment

Unclear as submitted. The sidecar proves useful operational direction, but the maintainable upstream solution should be a typed quota-bucket/auth-pool contract and embedded-runner key-list behavior, not log-tailing auth-file rewrites.
