## Summary

Keep open: current main covers parts of the request with multi-key collection, rate-limit-shaped key rotation helpers, and auth-profile cooldown/failover, but it still lacks a quota-bucket/key-pool contract for Gemini projects and the embedded runner still advances auth profiles rather than provider key-list entries. The reporter's Keymanager sidecar and the aashari fork are useful prior art, while related open issues cover adjacent slices rather than a merged replacement.

## What This Changes

Request native provider key pools with Gemini project/quota buckets and 429 cooldown/backoff rotation to avoid retry storms.

## Best Possible Solution

Define an additive provider-pool/quota-bucket contract, likely coordinated with https://github.com/openclaw/openclaw/issues/41135 and https://github.com/openclaw/openclaw/issues/81902, then wire embedded runtime key/profile selection through it with docs and regression coverage while preserving current profile-fallback defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config design request. Source inspection gives high confidence current main lacks quota-bucket config and embedded key-list rotation, but no live Gemini 429 repro was run.

## Solution Assessment

Unclear as submitted. The sidecar proves useful operational direction, but the maintainable upstream solution should be a typed quota-bucket/auth-pool contract and embedded-runner key-list behavior, not log-tailing auth-file rewrites.
