## Summary

Keep open: current main has substantial auth-profile rotation, cooldown, and rate-limit fallback behavior, but it does not implement the requested Gemini project/quota-bucket grouping. The linked sidecar/fork context is useful, but it does not make the remaining core/provider design obsolete.

## What This Changes

The issue requests provider API key pools with bucket-aware cooldown and backoff for quota errors, especially Gemini project-level 429 handling.

## Best Possible Solution

Design quota-bucket metadata as a first-class auth/provider contract, then cooldown and select profiles by bucket while preserving the existing profile/model fallback behavior and documenting the config surface.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence gap: current main has profile/model cooldowns but no quota-bucket/project grouping to reproduce as existing behavior.

## Solution Assessment

No, the current profile-level cooldown path is only a partial solution for the Gemini same-project multi-key case. The maintainable fix needs an explicit bucket/grouping design in auth profile metadata/state and provider selection before implementation.
