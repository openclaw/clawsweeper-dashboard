## Summary

Keep open: current main has multi-key env collection, rate-limit key-rotation helpers, auth-profile cooldowns, and Google project metadata, but it still lacks a shared quota-bucket contract and the embedded runner appears to rotate auth profiles rather than provider key-list entries.

## What This Changes

Request first-class provider key pools with quota-bucket-aware cooldown/backoff and hot rotation for Gemini/project-level 429s, plus consistent same-run fallback for configured key lists.

## Best Possible Solution

Define a generic quota-bucket auth/profile contract, then wire Gemini project-aware shared cooldown and any approved embedded LLM key-list rotation through prepared runtime auth with focused docs and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established for a strict bug. Source inspection does show the remaining gap: bucket-aware cooldown is absent, and embedded LLM auth currently resolves one credential/profile path rather than using the provider key-list rotation helper.

## Solution Assessment

Unclear as submitted. The sidecar proves the operational need, but the maintainable upstream path is a generic quota-bucket/provider-auth contract plus a separately scoped decision on embedded key-list retries.
