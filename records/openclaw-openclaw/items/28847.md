## Summary

Keep open: current main and the latest release have flat API-key rotation plus per-profile/model cooldowns, but still lack the requested quota-bucket identity that would cool all Gemini keys from the same project and select a different project bucket.

## What This Changes

The issue requests first-class provider API-key pools with bucket/quota-group-aware cooldown and rotation for Gemini project-scoped 429 errors.

## Best Possible Solution

Define an additive provider-owned quota-scope identity on auth candidates, integrate it with existing auth-profile cooldown/failover state, and keep the sidecar log watcher/auth-file rewrite as an external workaround rather than the core design.

## Reproduction Assessment

Yes, at source level: current main exposes flat API-key rotation and per-profile/model cooldowns but has no shared quota-bucket identity. I did not run a live exhausted Gemini project in this read-only review.

## Solution Assessment

No, the proposed log-watching sidecar and auth-file rewrite are not the best permanent core solution. A typed provider/auth quota identity integrated with existing cooldown and embedded failover paths is cleaner, but the public surface needs maintainer direction.
