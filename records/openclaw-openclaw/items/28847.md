## Summary

Keep open: current main and the latest release support flat multi-key rotation plus auth-profile cooldown/failover, but they still lack the requested quota-bucket/project identity needed to cooldown a whole Gemini project and rotate away from that bucket. Related provider-pool and quota circuit-breaker work overlaps, but no merged or viable open item fully supersedes this narrower bucket-aware key-pool request.

## What This Changes

Issue requests provider key pools with quota-bucket-aware cooldown and rotation for Gemini/project-scoped 429s to avoid retry storms.

## Best Possible Solution

Define an additive provider-owned quota-scope identity on auth/key candidates, integrate it with existing cooldown and fallback state, and reconcile the API shape with the broader provider-profile routing tracker before implementation.

## Reproduction Assessment

Yes, at source level: current main exposes flat key rotation and per-profile/model cooldowns but has no shared quota-bucket identity. I did not run a live exhausted Gemini project in this read-only review.

## Solution Assessment

Partly: the requested native bucket-aware behavior targets the right product gap, but the sidecar log-watcher/auth-file rewrite should remain a workaround rather than the core design. The best fix needs a typed provider/auth quota identity and maintainer-approved config or plugin API shape.
