## Summary

Keep open: current main covers part of the provider key-pool request through documented multi-key collection, explicit helper-based key rotation, and auth-profile cooldown/failover, but it still lacks a Gemini project quota-bucket contract and the embedded runner still advances auth profiles rather than provider key-list entries. The aashari fork and the reporter's sidecar are useful prior art, and the May 9 discussion correctly narrows a remaining embedded-runner gap that needs maintainer API/product scoping.

## What This Changes

Request provider key pools with bucket-aware cooldown/backoff and rotation across Gemini project quota groups after 429s without a gateway restart.

## Best Possible Solution

Define an additive quota-bucket/key-pool contract, keep provider-specific Gemini project metadata in the owning provider layer where possible, and wire embedded runtime auth selection to cool down shared buckets while reusing existing key-list support with docs and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config design request. Source inspection gives high confidence that current main lacks quota-bucket config and embedded key-list rotation, but no live Gemini 429 repro was run.

## Solution Assessment

Unclear as submitted. The sidecar proves useful operational direction, but the maintainable upstream solution should be a typed quota-bucket/auth-pool contract plus accepted embedded-runner key-list behavior, not log-tailing auth-file rewrites.
