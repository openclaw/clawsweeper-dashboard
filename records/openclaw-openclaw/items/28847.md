## Summary

Keep open: current main covers generic provider key collection, per-operation key rotation, and auth-profile cooldowns, but it still lacks the requested Gemini/project quota-bucket metadata and bucket-wide cooldown/selection behavior. Related work is adjacent, and https://github.com/openclaw/openclaw/issues/80451 was already closed into this issue as the canonical narrower quota-bucket tracker rather than replacing it with the broader pool-routing design in https://github.com/openclaw/openclaw/issues/41135.

## What This Changes

The issue requests first-class provider key pools with rate-limit cooldown/backoff and Gemini project or quota-bucket grouping so 429s cool down the exhausted bucket and route new requests elsewhere.

## Best Possible Solution

Define an additive provider/auth quota-bucket contract, feed bucket health into key/profile selection and embedded runtime auth preparation, document Gemini project grouping, and preserve existing plain key rotation and auth-profile fallback defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API design request. Source, docs, issue discussion, and related item state give high confidence that current main lacks quota-bucket metadata and bucket-wide cooldown, but no live Gemini 429 scenario was run.

## Solution Assessment

Unclear as submitted. The sidecar and public fork are useful prior art, but the maintainable OpenClaw path should be a typed provider/auth quota-bucket contract rather than log tailing and rewriting auth files.
