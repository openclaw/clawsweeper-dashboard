## Summary

Keep open. Current main has adjacent Gemini key-list discovery, 429/RESOURCE_EXHAUSTED classification, per-profile cooldowns, and fallback rotation, but it still lacks the central quota-bucket/project grouping needed to cool sibling Gemini keys that share the same Google project.

## What This Changes

Request native provider key pools with bucket-aware cooldown/backoff and hot rotation so Gemini project-level 429s do not cause retry storms.

## Best Possible Solution

Add an explicit provider-owned quotaBucket/quotaGroup concept on auth profiles or credential metadata, persist bucket-level health alongside profile usage, and make Gemini 429/RESOURCE_EXHAUSTED handling cool every profile in the same bucket before selecting a different bucket. Cover config/help docs plus auth-profile and embedded-runner rotation tests.

## Reproduction Assessment

Yes for the missing feature by source inspection: current main cannot represent a Gemini project bucket and failure state is updated per profile/key, not per bucket. A live quota-exhaustion run would further demonstrate it, but is not necessary to confirm the missing repository behavior in this read-only review.

## Solution Assessment

No, current main is not yet the best solution because per-key/per-profile cooldown cannot distinguish multiple keys in the same Gemini project from keys in different projects. The sidecar and fork are useful prior art, but native provider-owned bucket metadata and health are the maintainable direction.
