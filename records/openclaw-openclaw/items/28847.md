## Summary

Keep open: current main and the latest release support plain API-key list collection, per-operation key rotation, auth-profile ordering, and profile/model cooldowns, but they still do not provide quota-bucket metadata or bucket-wide cooldown/selection for Gemini project-level 429s. The external sidecar and public fork comments are useful prior art, and the narrower embedded-runner comment remains relevant, but the remaining work needs maintainer API/product scoping rather than cleanup closure.

## What This Changes

The issue requests provider key pools with quota-bucket metadata and bucket-wide cooldown/backoff for Gemini project-level 429s to avoid retrying keys from the same exhausted quota group.

## Best Possible Solution

Define one additive provider/auth quota-bucket contract, feed bucket health into key/profile selection and embedded runtime auth preparation, document Gemini project grouping, and preserve existing plain key rotation plus auth-profile fallback defaults.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction: this is a feature/API design request. Source inspection gives high confidence that current main lacks quota-bucket metadata and bucket-wide cooldown, but no live Gemini 429 scenario was run.

## Solution Assessment

Is this the best way to solve the issue? Unclear as submitted. The sidecar and public fork are useful prior art, but the maintainable OpenClaw path should be a typed provider/auth quota-bucket contract rather than log tailing and rewriting auth files.
