## Summary

Keep open: current main and the latest release support plain multi-key collection, per-operation key rotation, auth-profile ordering, and profile/model cooldowns, but they still lack quota-bucket metadata and bucket-wide cooldown/selection for Gemini project-level 429s. Related issues and PRs are useful prior art, but this remains the canonical tracker for the bucket-aware part.

## What This Changes

Request first-class provider key pools with Gemini project/quota-bucket-aware cooldown, backoff, and rotation on 429-like quota failures.

## Best Possible Solution

Define one additive provider/auth quota-bucket contract, feed bucket health into key/profile selection and embedded runtime auth preparation, document Gemini project grouping, and preserve existing plain key rotation and auth-profile fallback defaults.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/API design request. Source inspection gives high confidence that current main lacks quota-bucket metadata and bucket-wide cooldown, but no live Gemini 429 scenario was run.

## Solution Assessment

Unclear as submitted. The sidecar and public fork are useful prior art, but the maintainable OpenClaw path should be a typed provider/auth quota-bucket contract rather than log tailing and rewriting auth files.
