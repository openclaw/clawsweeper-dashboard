## Summary

Keep open: current main and the latest release support generic multi-key collection, per-operation key rotation, auth-profile ordering, and profile cooldowns, but they still do not provide Gemini/project quota-bucket metadata or bucket-wide cooldown/selection behavior. The linked sidecar, public fork, and related open items are useful prior art, while https://github.com/openclaw/openclaw/issues/28847 remains the canonical tracker for this specific bucket-aware provider/auth contract.

## What This Changes

Request first-class provider key pools plus Gemini project/quota-bucket-aware cooldown and backoff on 429-style failures to avoid retry storms without gateway restarts.

## Best Possible Solution

Define an additive provider/auth quota-bucket contract, feed bucket health into key/profile selection and embedded runtime auth preparation, persist cooldown state safely, document Gemini project grouping, and preserve existing plain key rotation and auth-profile fallback defaults.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature/API design request. Source, docs, tests, and related issue state give high confidence that current main lacks quota-bucket metadata and bucket-wide cooldown, but no live Gemini 429 scenario was run.

## Solution Assessment

Unclear as submitted. The sidecar and public fork are useful prior art, but the maintainable OpenClaw solution should be a typed provider/auth quota-bucket contract rather than log tailing and rewriting auth files.
