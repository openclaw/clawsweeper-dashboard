## Summary

Keep open: current main implements generic multi-key collection and profile/model cooldown behavior, but it still lacks the requested quota-bucket contract and bucket-wide cooldown/rotation needed for Gemini project-scoped 429 handling. Recent comments and related items confirm the remaining work is a product/API design gap rather than a stale or already-shipped bug fix.

## What This Changes

Request provider key pools with quota-bucket metadata and bucket-aware cooldown/backoff so Gemini project-scoped 429s rotate to a healthy bucket instead of retrying the same exhausted quota group.

## Best Possible Solution

Define an additive provider/auth-pool quota-bucket contract, route embedded/runtime key selection through bucket-aware health state, document Gemini project grouping, and preserve current profile fallback defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API design request. Source and docs inspection gives high confidence that current main lacks bucket/quota-group metadata, but I did not run a live Gemini 429 scenario.

## Solution Assessment

Unclear as submitted. The sidecar demonstrates useful behavior, but the maintainable OpenClaw path should be a typed provider/auth-pool contract rather than log-tailing and rewriting auth files.
