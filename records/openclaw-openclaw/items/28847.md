## Summary

Keep open: current main covers generic key-list collection, rate-limit-shaped helper rotation, auth-profile rotation, and profile cooldowns, but it still lacks a quota-bucket contract for Gemini projects and the embedded runner does not rotate through provider key-list entries before profile/model fallback.

## What This Changes

Add provider key pools with rate-limit-aware cooldown/backoff, Gemini project quota buckets, and hot rotation to avoid retry storms.

## Best Possible Solution

Define an additive generic quota-bucket/auth key-pool contract, keep provider-specific project metadata in the owning provider layer where possible, and wire embedded runtime auth selection to cool down shared buckets while reusing existing key-list support with docs and regression coverage.

## Reproduction Assessment

Not applicable as a feature/config design item, not a broken pre-existing bucket contract. Source inspection gives high confidence that current main lacks quota-bucket config and embedded key-list rotation, but no live Gemini 429 repro was run.

## Solution Assessment

Unclear as submitted. The sidecar demonstrates a useful operational direction, but the upstream fix should be a typed auth/profile quota-bucket contract plus an accepted embedded-runner key-list path, not log-tailing auth-file rewrites.
