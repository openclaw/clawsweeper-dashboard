## Summary

Keep open: current main supports API-key list collection, caller-specific key rotation, auth-profile cooldowns, and embedded profile failover, but it still lacks a shared quota-scope identity that can cool all keys from one Gemini project while selecting another project. The related provider-pool issue and open PRs cover adjacent or broader behavior, but none is a safe canonical replacement for this exact gap.

## What This Changes

Add provider API-key pools with quota-bucket-aware cooldown, jittered 429 backoff, rotation throttling, and hot key switching.

## Best Possible Solution

Define one additive provider-owned quota-scope contract, attach it to canonical auth candidates, and apply cooldown and selection against that identity in the existing SQLite auth-health and embedded failover paths while preserving current profile/model fallback defaults.

## Reproduction Assessment

Yes at source level: embedded failover and persistent health are profile-scoped and expose no quota-scope identity, so profiles sharing one Gemini project cannot be cooled as a group. No live current-main quota exhaustion was run.

## Solution Assessment

No for the proposed log-watching sidecar and auth-file rewrite; the maintainable solution is a typed quota-scope extension to the existing auth-candidate, cooldown, and embedded failover paths.
