## Summary

Keep open: current main supports plain provider key lists, rate-limit key rotation, and auth-profile cooldown/failover, but it still lacks the requested Gemini project/quota-bucket metadata and bucket-wide cooldown/rotation contract. The linked sidecar and fork are useful prior art, and open related issues overlap, but none supersede this exact provider key-pool bucket design.

## What This Changes

The issue requests native provider key pools with bucket-aware cooldown/backoff so Gemini project-scoped 429s cool down the whole quota bucket and rotate to another bucket without restarting the gateway.

## Best Possible Solution

Define an additive provider/auth-pool quota-bucket contract, route embedded/runtime key selection through bucket-aware health state, document Gemini project grouping, and preserve current profile fallback defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API design request. Source and docs inspection gives high confidence that current main lacks quota-bucket metadata and bucket-wide cooldown, but I did not run a live Gemini 429 scenario.

## Solution Assessment

Unclear as submitted. The sidecar demonstrates useful behavior, but the maintainable OpenClaw path should be a typed provider/auth-pool quota-bucket contract rather than log-tailing and rewriting auth files.
