## Summary

Keep open. Current main covers key lists, rate-limit classification, per-profile cooldown/backoff, and fallback rotation, but it still cannot represent or cool a Gemini quota bucket shared by multiple keys, so the central request remains missing and needs maintainer design.

## What This Changes

Request first-class provider key pools with 429/RESOURCE_EXHAUSTED backoff and Gemini project/quota-bucket-aware rotation without gateway restarts.

## Best Possible Solution

Add provider-owned quotaBucket/quotaGroup metadata plus bucket-level health, then cool all profiles in the same Gemini quota bucket on 429/RESOURCE_EXHAUSTED before selecting a different bucket, with docs and focused regression coverage.

## Reproduction Assessment

Yes for the missing feature by source inspection: current main can represent multiple keys and per-profile cooldowns, but not a Gemini project/quota bucket shared by sibling keys. I did not run a live Gemini quota-exhaustion test because the source/schema gap is already definitive for the requested behavior.

## Solution Assessment

No, the current per-key/per-profile cooldown path is not the best solution for Gemini project quotas because it can retry another key in the same exhausted project. The maintainable direction is provider-owned bucket metadata and bucket-level health rather than a log-tailing sidecar rewriting credentials.
