## Summary

Keep open: current main and v2026.6.6 have flat caller-scoped key rotation and auth-profile cooldown/failover, but still lack a shared quota identity that cools equivalent Gemini project credentials and selects a different project. Related open work covers broader pool routing or narrower provider-specific behavior, but no viable canonical replacement implements the full remainder.

## What This Changes

Add provider key pools with shared Gemini project quota buckets, bounded cooldown/backoff, and hot cross-bucket failover after quota errors.

## Best Possible Solution

Define an additive provider-owned quota-scope identity on canonical auth candidates, reuse existing auth-profile cooldown and failover state to suppress equivalent credentials, and make embedded runs consistently consume supported key lists without introducing a log watcher or separate state store.

## Reproduction Assessment

Yes, at source level: current main separates caller-scoped flat key rotation from auth-profile cooldown/failover and has no shared quota-bucket identity. This review did not exercise a live exhausted Gemini project.

## Solution Assessment

No, the proposed log-watching sidecar and auth-file rewrite are not the best permanent solution. A typed provider/auth contract integrated with the existing cooldown and embedded failover paths is narrower and avoids duplicate state, but its quota identity and public surface require maintainer judgment.
