## Summary

Keep open: current main has provider API-key list helpers, per-profile/model cooldowns, and profile rotation caps, but it still lacks shared Gemini project/quota-bucket cooldown semantics; the latest discussion also identifies a narrower embedded-runner gap where env-list provider keys are not rotated within the same run before auth-profile/model fallback.

## What This Changes

Request first-class provider key pools with quota-bucket-aware cooldown/backoff and hot rotation for Gemini project-level 429s.

## Best Possible Solution

Define a generic auth-profile/provider quota-bucket contract, then implement Gemini project-aware shared cooldown plus focused embedded-runner key-list rotation coverage without weakening current profile fallback semantics.

## Reproduction Assessment

Not applicable for a strict bug reproduction: this is an enhancement/design request. Source inspection does show the remaining capability gap on current main, and I did not run a live Gemini quota exhaustion scenario.

## Solution Assessment

Unclear until maintainers choose the contract. The sidecar and current per-profile/key-list behavior are useful workarounds, but the upstream solution should be a generic quota-bucket/provider-auth design rather than provider-specific ad hoc log watching.
