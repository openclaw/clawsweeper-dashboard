## Summary

Keep open: current main and the latest release cover generic key-list collection, caller-scoped key rotation, auth-profile cooldown, and model fallback, but they still do not provide the requested quota-bucket/key-pool contract for Gemini project-scoped 429 handling. The Keymanager sidecar, aashari fork comment, Jackten embedded-runner note, and broader provider-routing issue are useful context, but none supersede this narrower bucket-aware request.

## What This Changes

The issue asks for provider API-key pools with Gemini project/quota-bucket-aware cooldown and backoff to avoid retry storms after 429 or RESOURCE_EXHAUSTED responses.

## Best Possible Solution

Define an additive provider/auth-pool quota-bucket contract, wire embedded runtime key/profile selection and cooldown state through it, document Gemini project grouping, and preserve current profile fallback defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config design request. Current source and docs give high confidence that the requested quota-bucket contract is absent, but no live Gemini 429 run was attempted.

## Solution Assessment

Unclear as submitted. The sidecar proves useful operational behavior, but the upstream shape should be a typed auth/provider-pool contract rather than log-tailing and rewriting auth files.
