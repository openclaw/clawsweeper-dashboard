## Summary

Keep open: current main supports generic provider key-list collection, rate-limit-only key rotation, auth-profile cooldowns, and model/profile fallback, but it still lacks the requested quota-bucket/key-pool contract for Gemini project-scoped 429 handling. The broader provider-profile routing tracker remains open and related, while the closed multi-key issues and open Z.AI-only PR do not supersede this narrower bucket-aware request.

## What This Changes

The issue requests first-class provider API-key pools with quota-bucket-aware cooldown/backoff for Gemini project-scoped 429 handling.

## Best Possible Solution

Define an additive provider/auth-pool quota-bucket contract, wire embedded runtime key/profile selection and cooldown state through it, document Gemini project grouping, and preserve current profile fallback defaults.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API design request. Source and docs inspection give high confidence that current main lacks bucket/quota-group metadata, though no live Gemini 429 run was attempted.

## Solution Assessment

Unclear as submitted. The sidecar proves useful operational behavior, but the upstream shape should be a typed provider/auth-pool contract rather than log-tailing and rewriting auth files.
