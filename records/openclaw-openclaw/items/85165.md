## Summary

Keep open: current main still has a plausible local OpenAI-compatible token-accounting regression. The transport requests streaming usage for known local endpoints, but the completions stream parser only reads OpenAI-shaped `usage` on `chunk.usage` or `choice.usage` and does not normalize llama.cpp-style top-level `timings`, so the session context snapshot can stay missing or too low.

## What This Changes

The issue reports that local OpenAI-compatible llama.cpp routing undercounts context token usage in the Control UI and `/status` after upgrading from 2026.5.12 to 2026.5.18/2026.5.20.

## Best Possible Solution

Normalize local OpenAI-compatible streaming usage at the completions transport boundary, then persist a valid prompt/context snapshot without changing the broader `SessionEntry.totalTokens` semantics.

## Reproduction Assessment

No live high-confidence repro was run. Source inspection gives a medium-confidence fixture path: feed the OpenAI-completions stream parser a llama.cpp-style final chunk with response-level `timings` and the current parser will not update the prompt/context usage snapshot from it.

## Solution Assessment

Yes, the best fix is narrow transport normalization plus regression coverage. UI and `/status` should continue consuming the existing session snapshot contract rather than adding a separate display workaround.
