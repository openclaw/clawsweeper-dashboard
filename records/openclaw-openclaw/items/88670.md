## Summary

Keep this PR open, but it is not merge-ready: the new retry helper can still surface `SessionWriteLockStaleError` when the stale lock disappeared and a fresh lock appeared before the path probe, which preserves part of the linked stale-lock failure mode.

## What This Changes

The PR adds bounded retry for transient stale session-lock acquire failures, preserves one acquire timeout budget across retries, and adds focused session-write-lock regression coverage.

## Best Possible Solution

Keep the bounded retry behavior, but make the retry decision depend on the diagnostic payload or stale snapshot that was actually inspected, preserving typed diagnostics for still-present live stale locks and retrying disappeared or fresh-replaced stale reports within the original timeout budget.

## Reproduction Assessment

Yes, from source inspection: `readLockPayload()` can return `null` for a vanished stale lock, `inspectLockPayload(null, ...)` remains stale, and the PR then re-probes the path rather than the stale snapshot. I did not run a live multi-tool reproduction in this read-only review.

## Solution Assessment

No. The bounded retry direction is right, but the retry classification should be based on the diagnostic read or snapshot identity instead of a later `fs.access()` probe that can observe a different lock.
