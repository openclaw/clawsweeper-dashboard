## Summary

Keep open. Current main has several iMessage echo-loop mitigations, including `is_from_me` drops, echo-cache checks, reflected-content drops, multi-scope group echo lookup, and a new persisted echo cache, but it still lacks a configured local account-handle/own-sender guard for the reported `is_from_me=false` local-handle echo shape; the open fix PR remains the right place to resolve that gap.

## What This Changes

The issue reports iMessage agent replies being re-ingested as inbound messages, causing duplicate replies or self-triggered loops.

## Best Possible Solution

Review and refresh https://github.com/openclaw/openclaw/pull/48200 or an equivalent iMessage-plugin-owned guard that reliably identifies the local sending account, keeps the behavior backwards-compatible, updates config/docs/generated surfaces, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still has no configured own-account-handle guard, so an inbound iMessage from the local handle with `is_from_me=false` and an echo-cache scope miss can proceed toward dispatch.

## Solution Assessment

Unclear as a final product choice. The open account-handle PR is a narrow plausible fix, but adding a new public config key requires maintainer review and aligned schema, docs, metadata, UI hints, and tests before this issue should close.
