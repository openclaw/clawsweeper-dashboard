## Summary

Keep open. Current main has substantial iMessage echo-loop hardening, but it still intentionally lets text-only reflected replies fall through after the 4-second fallback window, which overlaps the delayed echo reports in this issue and is also present in the latest release tag checked.

## What This Changes

The issue reports that iMessage agent replies are reflected from the Messages database as inbound messages, causing duplicate processing or reply loops.

## Best Possible Solution

Keep the shipped ID/from-me/prefix/reflection guards, then add a precise late-echo signal or narrowly scoped fallback that catches delayed text-only agent echoes without suppressing legitimate self-chat messages.

## Reproduction Assessment

Yes at source level: current tests and code show a text-only echo after 4.5 to 5 seconds is not matched and will dispatch as a duplicate. I did not run live iMessage traffic in this read-only review.

## Solution Assessment

Unclear. The current partial implementation is safer than broad from-me or long-text-TTL filtering, but the remaining fix needs iMessage maintainer judgment because the obvious TTL increase conflicts with known self-chat false-positive risk.
