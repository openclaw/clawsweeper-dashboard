## Summary

Keep open. Current main contains real iMessage echo-loop hardening, including from-me drops, ID-backed echo matching, leading corruption-marker normalization, and assistant-marker reflection guards, but it still intentionally allows text-only echoed replies after the 4s fallback window to dispatch as duplicates. That remaining path overlaps the delayed iMessage echo reports, so the broader user problem is not proven fixed.

## What This Changes

The issue asks for iMessage inbound monitoring to ignore or deduplicate agent-sent messages so replies are not re-ingested as new inbound user messages.

## Best Possible Solution

Keep the shipped ID/from-me/prefix/assistant-marker guards, then make an explicit maintainer decision on late text-only echoes: either add a more precise identity signal from imsg, a narrowly scoped late-echo cache that cannot suppress real self-chat messages, or close only after live/current-main evidence proves delayed text-only echoes are no longer reachable.

## Reproduction Assessment

Yes for the remaining code-level path: the current unit tests reproduce a text-only echo miss after 4.5 to 5 seconds by asserting the cache returns false and duplicate delivery is the safe failure mode. Live iMessage reproduction was not run in this read-only review.

## Solution Assessment

Unclear. The current partial implementation is safer than the original broad filtering request, but simply widening the text TTL is not clearly the best fix because prior history shows it can cause self-chat message loss; a precise late-echo identity or narrowly scoped fallback would be safer.
