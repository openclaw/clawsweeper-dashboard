## Summary

Keep open. Current main and the latest release include several iMessage echo-loop guards, but the remaining source path still lets delayed text-only reflected replies miss the sent-message cache, and the echo cache is still scoped to the inbound sender rather than an account-wide outbound identity.

## What This Changes

The issue asks the legacy iMessage plugin to stop re-ingesting agent-sent Messages database rows or reflected replies as inbound user messages.

## Best Possible Solution

Keep the shipped ID, from-me, self-chat, prefix-normalization, and reflection guards, then add a precise agent-sent echo signal or bounded account-aware fallback that catches delayed and scope-mismatched agent echoes without dropping legitimate self-chat or repeated user messages.

## Reproduction Assessment

Yes, at source level: current tests and code show a text-only agent echo after 4.5 to 5 seconds is not matched and can continue toward dispatch. The issue discussion also describes a sender-scope mismatch that current source still lacks an account-global fallback for.

## Solution Assessment

No. The current partial implementation is safer than broad text suppression, but it does not fully solve this issue; the original simple `is_from_me` filter is also insufficient because self-chat rows can represent legitimate user messages.
