## Summary

Keep this PR open, but do not merge it as-is: the bug is plausible from source inspection, while the patch changes caller-specific failure semantics and still lacks real Tailscale behavior proof.

## What This Changes

This PR catches Tailscale JSON parse failures in the shared parser and routes Funnel status parsing through that helper.

## Best Possible Solution

Make Tailscale parse handling caller-aware: preserve noisy valid JSON support, keep malformed whois output on the error-TTL path, report malformed Funnel status as unknown command output, and add focused regression tests.

## Reproduction Assessment

Yes, by source inspection rather than a live Tailscale run. Malformed non-empty output through readTailscaleWhoisIdentity or ensureFunnel exercises the current raw JSON.parse paths and the PR's changed fallback semantics.

## Solution Assessment

No. Returning {} from the shared parser is not the narrowest maintainable fix because empty output, malformed output, command errors, and missing identity/status data have different meanings at different callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve whois parse failures on the error TTL path:** `src/infra/tailscale.ts:23-24`
  - body: Catching parser failures in the shared helper and returning {} means malformed tailscale whois --json output no longer reaches readTailscaleWhoisIdentity's catch; it is cached as a successful null identity with the default 60s TTL instead of the 5s error TTL, prolonging transient Serve auth failures.
  - confidence: 0.93
