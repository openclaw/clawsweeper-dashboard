## Summary

Keep this PR open. Current main has not already implemented the Tailscale JSON-parse resilience, but the proposed broad `{}` fallback introduces correctness regressions in malformed whois handling and Funnel status diagnostics, so it should be revised before merge.

## What This Changes

The PR changes `src/infra/tailscale.ts` so the shared noisy JSON helper catches parse failures by returning `{}` and `ensureFunnel` parses `tailscale funnel status --json` through that helper.

## Best Possible Solution

Keep the resilience fix caller-aware: accept noisy valid JSON where intended, preserve malformed whois JSON on the error-TTL path, report malformed Funnel status as unknown command output instead of disabled setup state, and add focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Current main reproduces the original crash path with malformed non-empty Tailscale JSON, and the PR regressions are reproducible by feeding malformed `tailscale whois --json` and `tailscale funnel status --json` output through the changed paths.

## Solution Assessment

No. Returning `{}` from the shared parser is not the narrowest maintainable fix because callers attach different meanings to empty objects; caller-level parse handling is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve whois parse failures on the error TTL path:** `src/infra/tailscale.ts:23-24`
  - body: Catching parser failures here and returning `{}` changes `readTailscaleWhoisIdentity`: malformed `tailscale whois --json` output no longer reaches that function's `catch`, so it is cached as a successful null identity with the default 60s TTL instead of the 5s error TTL. That can prolong transient Tailscale Serve auth failures.
  - confidence: 0.93
- **[P2] Keep Funnel parse failures out of the disabled-state branch:** `src/infra/tailscale.ts:333`
  - body: With the helper returning `{}` on malformed output, invalid or partial `tailscale funnel status --json` is treated the same as an empty status response and enters the “Funnel is not enabled” setup flow. In that case the Funnel state is unknown, so the code should preserve a parse or command diagnostic instead of prompting remediation for a disabled tailnet.
  - confidence: 0.91
