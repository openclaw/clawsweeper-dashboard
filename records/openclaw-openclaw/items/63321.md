## Summary

Keep open: the resilience goal is plausible, but the current patch changes malformed Tailscale output into an empty object in places where callers currently distinguish parse errors from empty or disabled state, and the external PR still lacks real Tailscale proof.

## What This Changes

The PR changes `src/infra/tailscale.ts` so the shared Tailscale JSON parser catches parse errors and returns `{}`, and `ensureFunnel` uses that helper for `tailscale funnel status --json`.

## Best Possible Solution

Make Tailscale parse handling caller-aware: preserve noisy valid JSON support, keep malformed whois output on the error-TTL path, report malformed Funnel status as unknown command output, and add focused regression tests plus real Tailscale proof.

## Reproduction Assessment

Yes, by source inspection rather than a live Tailscale run. Malformed non-empty output through `readTailscaleWhoisIdentity` or `ensureFunnel` exercises the current raw parse paths and the PR's changed fallback semantics.

## Solution Assessment

No. Returning `{}` from the shared parser is not the narrowest maintainable fix because empty output, malformed output, command errors, and missing identity/status data mean different things at different callers.

## Review Findings

Overall correctness: patch is incorrect

  - body: Catching parser failures in the shared helper and returning `{}` means malformed `tailscale whois --json` output no longer reaches `readTailscaleWhoisIdentity`'s catch; it is cached as a successful null identity with the default 60s TTL instead of the 5s error TTL, prolonging transient Serve auth failures.
  - confidence: 0.93

  - confidence: 0.91
