## Summary

Keep this PR open. The crash-prevention goal is valid on current main, but the proposed shared `return {}` fallback erases parse failures in callers where an empty object has real meaning, especially Tailscale whois auth caching and Funnel diagnostics.

## What This Changes

The PR catches `JSON.parse` failures inside `parsePossiblyNoisyJsonObject` by returning `{}` and changes `ensureFunnel` to parse Funnel status through that helper.

## Best Possible Solution

Keep the resilience fix caller-aware: allow noisy valid JSON where intended, but preserve malformed JSON as an error for whois/auth, report Funnel status parse failures as unknown command output rather than disabled state, and cover those cases with focused tests.

## Reproduction Assessment

Yes. A malformed `tailscale whois --json` payload and malformed `tailscale funnel status --json` payload are enough to trace the changed paths statically: the PR's helper returns `{}`, which then uses the wrong whois cache path and the empty-Funnel branch.

## Solution Assessment

No. The broad shared fallback is not the narrowest maintainable fix; the safer solution is explicit caller-level parse handling that distinguishes empty output, noisy valid JSON, and malformed JSON.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve whois parse failures on the error TTL path:** `src/infra/tailscale.ts:23-24`
  - body: Catching parser failures here and returning `{}` changes `readTailscaleWhoisIdentity`: malformed `tailscale whois --json` output no longer reaches that function's `catch`, so it is cached as a successful null identity with the default 60s TTL instead of the 5s error TTL. That can prolong transient Tailscale Serve auth failures.
  - confidence: 0.93
- **[P2] Keep Funnel parse failures out of the disabled-state branch:** `src/infra/tailscale.ts:333`
  - body: With the helper returning `{}` on malformed output, invalid or partial `tailscale funnel status --json` is treated the same as an empty status response and enters the “Funnel is not enabled” setup flow. In that case the Funnel state is unknown, so the code should preserve a parse/command diagnostic instead of prompting remediation for a disabled tailnet.
  - confidence: 0.91
