## Summary

Keep this PR open, but it is not ready to merge: the branch improves one noisy-output path but also changes malformed Tailscale output into normal empty state in caller paths that currently distinguish parse failures from disabled or missing identity state.

## What This Changes

The PR changes `src/infra/tailscale.ts` so the shared Tailscale JSON parser catches parse failures and returns `{}`, and Funnel status parsing uses that helper.

## Best Possible Solution

Make Tailscale parse handling caller-aware: preserve noisy valid JSON support, keep malformed whois output on the error-TTL path, report malformed Funnel status as unknown command output, and add focused regression tests plus real Tailscale proof.

## Reproduction Assessment

Yes, by source inspection rather than a live Tailscale run. Malformed non-empty stdout from `tailscale whois --json` or `tailscale funnel status --json` exercises the current parse paths and the PR's changed fallback semantics.

## Solution Assessment

No. Returning `{}` from the shared parser collapses malformed output into normal empty status or missing identity data; the safer fix is caller-specific error handling.
