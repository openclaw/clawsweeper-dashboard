## Summary

Keep open: the underlying Tailscale parse-crash report is plausible on current main, but this PR is not mergeable because it converts malformed command output into normal empty state in caller-sensitive paths and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/infra/tailscale.ts` so Tailscale JSON parse failures return `{}` and Funnel status parsing uses the shared noisy-JSON helper.

## Best Possible Solution

Handle Tailscale parse failures at the caller: preserve noisy valid JSON support, keep malformed whois on the error-TTL path, report malformed Funnel status as unknown command output, and add focused regression tests plus redacted real Tailscale proof.

## Reproduction Assessment

Yes, by source inspection: malformed non-empty stdout from `tailscale whois --json` or `tailscale funnel status --json` reaches the current and changed parse paths. I did not run a live Tailscale reproduction in this read-only review.

## Solution Assessment

No. Returning `{}` from the shared parser is too broad because it hides malformed output as normal empty state; the safer fix is caller-specific error handling with regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

- **[P2] Preserve whois parse failures on the error TTL path:** `src/infra/tailscale.ts:23-24`
  - body: Catching parser failures here and returning `{}` means malformed `tailscale whois --json` output no longer reaches `readTailscaleWhoisIdentity`'s catch path, so a transient parse failure is cached with the normal `cacheTtlMs` instead of the shorter `errorTtlMs`.
  - confidence: 0.93
