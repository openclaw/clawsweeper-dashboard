## Summary

Keep open: the Tailscale malformed-output problem is plausible, but this patch is not merge-ready because it turns parse failures into normal empty-state results in two caller-visible paths and still lacks real after-fix Tailscale proof.

## What This Changes

The PR wraps the shared Tailscale JSON parser in a catch that returns `{}` and routes `ensureFunnel` status parsing through that helper.

## Best Possible Solution

Make Tailscale parse handling caller-aware: preserve noisy valid JSON support, keep malformed whois output on the error-TTL path, report malformed Funnel status as unknown command output, and add focused regression tests plus real Tailscale proof.

## Reproduction Assessment

Yes for source-level reproduction: malformed non-empty stdout from `tailscale whois --json` or `tailscale funnel status --json` exercises the current parse paths and the PR's changed fallback semantics. I did not run a live Tailscale repro in this read-only review.

## Solution Assessment

No. Returning `{}` from the shared parser collapses malformed output into normal empty status or missing identity data; the safer fix is caller-specific error handling.
