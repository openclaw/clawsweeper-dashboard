## Summary

Keep open. The PR targets a real Tailscale parsing hardening gap, but the current patch is not mergeable because it changes caller semantics: malformed `tailscale whois --json` becomes a long-lived null identity cache entry, and malformed Funnel status becomes the disabled/setup branch instead of an unknown parse/command failure.

## What This Changes

The branch changes `src/infra/tailscale.ts` so the shared noisy JSON parser returns `{}` on parse failure and `ensureFunnel` uses that helper for `tailscale funnel status --json`.

## Best Possible Solution

Keep the resilience improvement caller-aware: accept noisy valid JSON where intended, preserve malformed whois JSON on the short error-TTL path, report malformed Funnel status as unknown parse/command output rather than disabled state, and add focused tests plus the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: feed malformed non-empty output through `readTailscaleWhoisIdentity` and the PR caches null identity with the success TTL; feed malformed `tailscale funnel status --json` output through `ensureFunnel` and the PR enters the disabled/setup branch.

## Solution Assessment

No. Returning `{}` from the shared parser is not the narrowest maintainable fix because each caller assigns different meaning to an empty object; caller-level parse handling or a typed parse result is safer.
