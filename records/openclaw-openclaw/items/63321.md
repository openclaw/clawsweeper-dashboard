## Summary

Keep open: the resilience goal is plausible, but the patch masks malformed Tailscale output as empty state, changes whois cache and Funnel guidance semantics, and still lacks contributor real-behavior proof.

## What This Changes

The PR wraps Tailscale JSON parsing in try/catch returning empty objects and routes Funnel status parsing through the shared noisy-JSON helper.

## Best Possible Solution

Make Tailscale parse handling caller-aware: preserve noisy valid JSON support, keep malformed whois output on the error-TTL path, report malformed Funnel status as unknown command output, and add focused regression tests plus real Tailscale proof.

## Reproduction Assessment

Yes, by source inspection rather than a live Tailscale run. Malformed non-empty stdout from tailscale whois --json or tailscale funnel status --json exercises the current parse paths and the PR's changed fallback semantics.

## Solution Assessment

No. Returning {} from the shared parser collapses malformed output into normal empty status or missing identity data; the safer fix is to catch parse failures at the caller with caller-specific error handling.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve whois parse failures on the error TTL path:** `src/infra/tailscale.ts:23-24`
  - body: Catching parser failures here and returning {} means malformed tailscale whois --json output no longer reaches readTailscaleWhoisIdentity's catch; it is cached as a successful null identity with the normal cacheTtlMs instead of the shorter errorTtlMs.
