## Summary

Keep open: the PR targets a plausible Tailscale parse crash, but the current patch changes caller-specific parse-failure semantics and still lacks required real Tailscale proof.

## What This Changes

The PR wraps the shared Tailscale JSON parser in try/catch returning `{}` and routes `ensureFunnel`'s status JSON through that helper.

## Best Possible Solution

Make Tailscale parse handling caller-aware: preserve noisy valid JSON support, keep malformed whois output on the error-TTL path, report malformed Funnel status as unknown command output, and add focused regression tests plus real Tailscale proof.

## Reproduction Assessment

Yes, by source inspection rather than a live Tailscale run. Malformed non-empty output from `tailscale whois --json` or `tailscale funnel status --json` exercises the current raw parse paths and the PR's changed fallback semantics.

## Solution Assessment

No. Returning `{}` from the shared parser collapses malformed output into normal empty status or missing identity data, so the fix should preserve caller-specific error handling instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve whois parse failures on the error TTL path:** `src/infra/tailscale.ts:23-24`
  - body: Catching parser failures in the shared helper and returning `{}` means malformed `tailscale whois --json` output no longer reaches `readTailscaleWhoisIdentity`'s catch; it is cached as a successful null identity with the default 60s TTL instead of the 5s error TTL, prolonging transient Serve auth failures.
  - confidence: 0.93

  - body: With malformed `tailscale funnel status --json` converted to `{}`, the next branch treats unknown or partial command output as Funnel not enabled and prompts setup. Preserve a parse or command diagnostic instead of sending users down the disabled-tailnet remediation path.
  - confidence: 0.91
