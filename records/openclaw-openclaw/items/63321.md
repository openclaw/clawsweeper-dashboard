## Summary

Keep open: this is a focused Tailscale parse-hardening PR, but the current diff converts malformed CLI output into successful empty state in caller-visible paths and still lacks after-fix real Tailscale proof.

## What This Changes

The PR wraps Tailscale CLI JSON parsing in `src/infra/tailscale.ts` so malformed output returns `{}` and routes Funnel status parsing through the shared helper.

## Best Possible Solution

Make Tailscale parse handling caller-aware: preserve noisy valid JSON support, keep malformed whois output on the error-TTL path, report malformed Funnel status as unknown command output, and add focused regression tests plus real Tailscale proof.

## Reproduction Assessment

Yes for source-level reproduction: malformed non-empty stdout from `tailscale whois --json` or `tailscale funnel status --json` exercises the current parse paths and the PR's changed fallback semantics. I did not run a live Tailscale repro.

## Solution Assessment

No. Returning `{}` from the shared parser collapses malformed output into normal empty status or missing identity data; the safer fix is caller-specific error handling.
