## Summary

Keep open: the PR targets a real source-reproducible mismatch, but it changes `tools.exec.pathPrepend` from exec-scoped behavior into process-wide PATH mutation and still has reload/casing defects plus missing real behavior proof.

## What This Changes

The PR adds config-load logic in `src/config/io.ts` to prepend trimmed `tools.exec.pathPrepend` entries to `deps.env.PATH`.

## Best Possible Solution

Choose and document the contract: either make binary discovery use a config-derived effective PATH, or implement deliberate global PATH mutation with owned-entry tracking, shared helper/casing support, reload tests, and docs updates.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, then run a `hasBinary()`-backed skill/status check for a binary in that directory; current main never merges that setting before scanning PATH.

## Solution Assessment

No. The PR fixes the immediate symptom, but it is not the best current solution unless maintainers deliberately accept global PATH mutation with owned-entry cleanup, PATH-key casing support, tests, and docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track owned PATH entries across config reloads:** `src/config/io.ts:1840-1846`
  - body: This prepends directories into `deps.env.PATH` but never records which entries came from the current config. If a live config update removes or changes `tools.exec.pathPrepend`, old directories remain visible to `hasBinary()` and later PATH lookups until process restart.
  - confidence: 0.89
- **[P2] Preserve the active PATH key casing:** `src/config/io.ts:1841-1845`
  - body: The patch reads and writes `deps.env.PATH` directly instead of using the existing PATH helper. Env objects can carry `Path` or another PATH key casing, so this can create a second key or miss the active PATH value.
  - confidence: 0.83
