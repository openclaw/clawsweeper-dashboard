## Summary

Keep open. The underlying PATH mismatch is still present on current main, but this PR should not merge as-is because it changes an exec-scoped setting into process-wide binary resolution while leaving concrete reload and PATH-key-casing defects.

## What This Changes

The PR adds config-load logic in `src/config/io.ts` to trim, dedupe, and prepend `tools.exec.pathPrepend` entries into `deps.env.PATH`.

## Best Possible Solution

Choose the contract first: either compute a config-aware effective PATH for skill/media binary discovery without globally mutating `process.env`, or deliberately make the setting process-wide through the shared helper with owned-entry cleanup, PATH-key casing coverage, and docs/help/test updates.

## Reproduction Assessment

Yes. Set `process.env.PATH` without a temp bin directory, configure `tools.exec.pathPrepend` to that directory, load config, and call `hasBinary` for an executable there; current main will not find it. The PR’s reload defect is also reproducible by loading with a prepend and then reloading after removing it, because the added code only prepends.

## Solution Assessment

No. The PR addresses the immediate symptom, but it is not the best current shape because it broadens the documented trust boundary and leaves stale reload semantics plus Windows-style PATH-key handling broken. A safer solution would use an explicit effective-PATH seam or a stateful global mutation helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Remove stale PATH entries on reload:** `src/config/io.ts:1840-1846`
  - body: This only prepends new entries into `deps.env.PATH` and never records which entries came from the loaded config. If a live config reload removes or changes `tools.exec.pathPrepend`, `hasBinary()` and later PATH-based resolution will keep seeing the old directory until process restart, so disabled or removed tools can remain eligible.
  - confidence: 0.89
- **[P2] Preserve existing PATH key casing:** `src/config/io.ts:1840-1844`
  - body: The patch reads and writes `deps.env.PATH` directly instead of using the existing `applyPathPrepend` helper. Env objects copied from Windows-style environments may only have `Path`, so this can miss the existing value, fail dedupe, or create a second PATH key.
  - confidence: 0.82
