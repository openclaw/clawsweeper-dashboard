## Summary

Keep open. The source-level symptom is real, but this PR broadens an exec-scoped PATH setting into process-wide binary resolution and still needs maintainer/security acceptance plus reload and PATH-key casing fixes before merge.

## What This Changes

The PR prepends trimmed `tools.exec.pathPrepend` entries into `deps.env.PATH` during config load so process-wide binary detection sees those directories.

## Best Possible Solution

Choose and document the contract: either make binary discovery use a config-derived effective PATH, or implement deliberate global PATH mutation with owned-entry tracking, shared helper/casing support, reload tests, and docs updates.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, then call a `hasBinary()`-backed skill/status check for a binary in that directory; current main never merges that setting before scanning PATH.

## Solution Assessment

No. The PR fixes the immediate symptom, but it is not the best current solution because it mutates global PATH without owned-entry removal on reload, bypasses existing PATH-key casing semantics, and broadens the documented exec-only behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track and remove owned PATH entries on reload:** `src/config/io.ts:1840-1846`
  - body: This prepends directories into `deps.env.PATH` but never records which entries came from the current config. If a live config update removes or changes `tools.exec.pathPrepend`, the old directories remain visible to `hasBinary()` and later PATH lookups until process restart.
  - confidence: 0.89
- **[P2] Preserve existing PATH key casing:** `src/config/io.ts:1841-1845`
  - body: The patch reads and writes `deps.env.PATH` directly instead of using the existing PATH helper. Env objects can carry `Path` or another PATH key casing, and this code can create a second key or miss the existing value instead of updating the active PATH entry.
  - confidence: 0.83

## Security Review
