## Summary

Keep this PR open, but it is not merge-ready: the underlying source path is plausible, yet the patch broadens an exec-scoped PATH setting into process-wide command resolution without reload cleanup, PATH-key casing handling, real behavior proof, or explicit security/product acceptance.

## What This Changes

The PR prepends `tools.exec.pathPrepend` entries to `deps.env.PATH` during config load so `hasBinary()`-backed skill checks can see the same directories as exec runs.

## Best Possible Solution

Define the contract deliberately: either pass a config-derived effective PATH into binary discovery, or implement process-wide PATH mutation with owned-entry cleanup, casing-safe helper reuse, focused tests, docs, and security acceptance.

## Reproduction Assessment

Yes, by source inspection: configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, then run a `hasBinary()`-backed skill/status check. Current main keeps that setting scoped to exec environments while `hasBinary()` reads the process PATH directly.

## Solution Assessment

No. Direct global PATH mutation addresses the symptom, but this patch should not merge without owned-entry reload semantics, PATH key casing support, tests/docs, real behavior proof, and explicit acceptance of the widened command-resolution surface.

## Review Findings

Overall correctness: patch is incorrect

  - body: This prepends configured directories into `deps.env.PATH` but never records which entries came from the current config. When an in-process config reload removes or changes `tools.exec.pathPrepend`, old directories remain visible to `hasBinary()` and later PATH-based lookups until restart.
