## Summary

Keep open. Current main still has the reported PATH visibility gap, but this PR is not merge-ready because it expands an exec-scoped setting into process-wide PATH mutation, bypasses the shared PATH helper, and leaves stale prepended entries after config reloads.

## What This Changes

The PR adds config-load logic in `src/config/io.ts` that trims and dedupes `tools.exec.pathPrepend` entries, then prepends them to `deps.env.PATH`.

## Best Possible Solution

Choose the contract first. Either make `hasBinary` and media/skill discovery use a documented effective PATH derived from config, or implement deliberate global PATH mutation through the shared helper with owned-entry tracking, removal on reload, Windows/path-key coverage, tests, and docs updates.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, load config, then call `hasBinary` for a binary in that directory; current main never merges that setting before `hasBinary` scans PATH. The PR reload defect is also reproducible by loading with one prepend and reloading after removing it, because the added code only prepends.

## Solution Assessment

No. The PR fixes the immediate symptom, but it is not the best current shape because it widens an exec-scoped PATH setting into global process resolution and misses reload/casing semantics already covered by the shared helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Remove owned PATH entries when config changes:** `src/config/io.ts:1840-1846`
  - body: This prepends missing directories into `deps.env.PATH` on every config load, but never records or removes entries from an earlier `tools.exec.pathPrepend`. If a live config update removes or changes the list, `hasBinary()` and later PATH lookups still see the old directory until restart.
  - confidence: 0.89
- **[P2] Reuse the PATH prepend helper for env casing:** `src/config/io.ts:1840-1844`
  - body: The patch reads and writes `deps.env.PATH` directly instead of using `applyPathPrepend`. Env objects can carry `Path` or another PATH key casing, and the existing helper preserves that casing while deduping; this code can create a second PATH key or miss the existing value.
  - confidence: 0.83
