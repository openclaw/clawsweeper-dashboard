## Summary

Keep open: current main still does not make `tools.exec.pathPrepend` visible to process-wide binary discovery, but this PR widens an exec-scoped PATH setting into global PATH without a documented contract, has concrete reload and PATH-casing bugs, and lacks after-fix real behavior proof.

## What This Changes

The PR changes config loading to prepend trimmed `tools.exec.pathPrepend` entries into `deps.env.PATH` so `hasBinary()`-backed checks can see exec-configured directories.

## Best Possible Solution

Define the contract deliberately: either pass a config-derived effective PATH into binary discovery, or implement global PATH mutation with owned-entry tracking, casing-safe helper reuse, tests, docs, and changelog.

## Reproduction Assessment

Yes, by source inspection: configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, then run a `hasBinary()`-backed skill/status check; current main loads the config without merging that setting while `hasBinary()` scans only global PATH.

## Solution Assessment

No. The PR addresses the immediate symptom, but direct global PATH mutation is not the best current solution without owned-entry cleanup on reload, existing PATH-key casing support, explicit security acceptance, and docs/tests for the widened contract.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Track and remove owned PATH entries on reload:** `src/config/io.ts:1840-1846`
  - body: This prepends configured directories into `deps.env.PATH` but never records which entries came from the current config. When a live config update removes or changes `tools.exec.pathPrepend`, old directories remain visible to `hasBinary()` and later PATH lookups until process restart.
  - confidence: 0.89
