## Summary

The PR targets a real-looking PATH discovery gap, but it is not merge-ready: it lacks after-fix real behavior proof, mutates an exec-scoped setting into daemon-wide PATH state, and still has concrete reload/casing defects.

## What This Changes

The branch prepends `tools.exec.pathPrepend` entries to `deps.env.PATH` during config loading so readiness checks that call `hasBinary()` can see configured exec tool paths.

## Best Possible Solution

Define an explicit effective-PATH contract for binary discovery, then implement it with existing PATH helpers, owned-entry cleanup, focused tests/docs, and maintainer acceptance of any widened semantics.

## Reproduction Assessment

Yes, by source inspection: current `hasBinary()` reads `process.env.PATH`, while `tools.exec.pathPrepend` is applied only to exec environments. A closed stale report also described the same skill status versus runtime PATH mismatch.

## Solution Assessment

No. Direct global PATH mutation addresses the symptom, but the maintainable fix should either pass an explicit effective PATH into discovery or intentionally document and harden daemon-wide PATH semantics.
