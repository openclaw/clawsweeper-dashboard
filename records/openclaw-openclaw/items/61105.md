## Summary

Keep open: the motivating PATH mismatch is still present on current main, but this PR still has blocking correctness/security concerns and lacks required after-fix real behavior proof.

## What This Changes

The PR adds config-load logic in `src/config/io.ts` to prepend `tools.exec.pathPrepend` entries into the process environment PATH.

## Best Possible Solution

Define the contract deliberately: either pass a config-derived effective PATH into binary discovery, or implement process-wide PATH mutation with owned-entry cleanup, casing-safe helper reuse, tests, docs, and security acceptance.

## Reproduction Assessment

Yes, by source inspection: configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, then run a `hasBinary()`-backed skill/status check; current main never folds that setting into the process PATH while `hasBinary()` scans only `process.env.PATH`. I did not run a live macOS app repro.

## Solution Assessment

No. The direct global PATH mutation addresses the symptom, but it should not merge without owned-entry reload semantics, PATH key casing support, tests/docs, and explicit acceptance of the widened command-resolution surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track and remove owned PATH entries on reload:** `src/config/io.ts:1840-1846`
  - body: This prepends configured directories into `deps.env.PATH` but never records which entries came from the current config. When an in-process config reload removes or changes `tools.exec.pathPrepend`, old directories remain visible to `hasBinary()` and later PATH-based lookups until restart.
  - confidence: 0.89
- **[P2] Preserve existing PATH key casing:** `src/config/io.ts:1841-1845`
  - body: The patch reads and writes `deps.env.PATH` directly instead of using the existing PATH helper. Env objects can carry `Path` or another PATH key casing, so this can create a second key or miss the active PATH value instead of updating it.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Needs attention because the PR broadens an exec-scoped PATH setting into process-wide command resolution.
