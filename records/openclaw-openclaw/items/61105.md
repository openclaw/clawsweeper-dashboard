## Summary

Keep open: the requested behavior is still a source-visible gap on current main, but this PR should not merge as-is because it widens an exec-scoped PATH setting into daemon-wide process state without reload cleanup, PATH-key handling, real behavior proof, tests/docs, or explicit security/product acceptance.

## What This Changes

The PR changes config loading so `tools.exec.pathPrepend` entries are prepended to `deps.env.PATH` for `hasBinary()`-based skill and status checks.

## Best Possible Solution

Define an explicit effective-PATH contract for binary discovery, then implement it with existing PATH helpers, owned-entry cleanup, docs, focused tests, and security acceptance.

## Reproduction Assessment

Yes, by source inspection: configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, then run a `hasBinary()`-backed skill/status check. Current main keeps `pathPrepend` scoped to exec environments while `hasBinary()` reads `process.env.PATH`.

## Solution Assessment

No. Direct global PATH mutation addresses the symptom, but a maintainable fix needs either a config-derived effective PATH passed into binary discovery or daemon-wide mutation with casing-safe helpers, owned-entry cleanup, tests/docs, and explicit security acceptance.
