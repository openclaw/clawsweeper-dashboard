## Summary

Keep open: current main still leaves `tools.exec.pathPrepend` scoped to exec runs while `hasBinary()` reads the process PATH, but this PR should not merge as-is because it introduces one-way global PATH mutation without reload cleanup, PATH-key casing support, real behavior proof, or explicit security/product acceptance.

## What This Changes

The PR adds config-load logic in `src/config/io.ts` to prepend `tools.exec.pathPrepend` entries to `deps.env.PATH` so `hasBinary()` callers can discover those binaries.

## Best Possible Solution

Define the contract deliberately: either pass a config-derived effective PATH into binary discovery, or implement process-wide PATH mutation with owned-entry cleanup, casing-safe helper reuse, focused tests, docs, and security acceptance.

## Reproduction Assessment

Yes, by source inspection: configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, then run a `hasBinary()`-backed skill/status check. Current main keeps that setting scoped to exec environments while `hasBinary()` reads the process PATH directly.

## Solution Assessment

No. Direct global PATH mutation addresses the symptom, but this patch needs owned-entry reload semantics, PATH key casing support, tests/docs, real behavior proof, and explicit acceptance of the widened command-resolution surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track and remove owned PATH entries on reload:** `src/config/io.ts:1841-1843`
  - body: This prepends configured directories into `deps.env.PATH` but never records which entries came from the current config. When an in-process config reload removes or changes `tools.exec.pathPrepend`, old directories remain visible to `hasBinary()` and later PATH-based lookups until restart.
  - confidence: 0.89
- **[P2] Preserve the active PATH key casing:** `src/config/io.ts:1839-1843`
  - body: The patch reads and writes `deps.env.PATH` directly instead of using the existing PATH helper. Env objects can carry `Path` or another PATH key casing, so this can create a second key or miss the active PATH value instead of updating it.
  - confidence: 0.84
