## Summary

Keep open: current main still keeps `tools.exec.pathPrepend` scoped to exec environments, and this PR addresses a real binary-discovery mismatch, but the branch is not merge-ready because it broadens PATH semantics, leaves stale prepends across reloads, bypasses existing PATH-key helpers, and lacks inspectable real behavior proof.

## What This Changes

The PR prepends `tools.exec.pathPrepend` entries to `deps.env.PATH` during config loading so `hasBinary()`-based readiness checks see configured exec tool paths.

## Best Possible Solution

Define an explicit effective-PATH contract for binary discovery, then implement it with the existing PATH helpers, owned-entry cleanup, docs, focused tests, and maintainer acceptance of any widened semantics.

## Reproduction Assessment

Yes. By source inspection, current main's `hasBinary()` reads `process.env.PATH` while `tools.exec.pathPrepend` is applied only to exec environments, so a configured directory absent from the daemon PATH is invisible to readiness checks.

## Solution Assessment

No. Direct global PATH mutation addresses the symptom, but the maintainable fix should either pass an explicit effective PATH into discovery or intentionally document and harden daemon-wide PATH semantics.
