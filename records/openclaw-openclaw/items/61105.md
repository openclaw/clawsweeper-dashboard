## Summary

Keep open: the PR addresses a source-visible mismatch, but the proposed process-wide PATH mutation is not mergeable as-is because it leaves stale entries on reload, ignores PATH key casing, lacks real behavior proof, and needs maintainer/security acceptance of the widened command-resolution contract.

## What This Changes

The PR prepends `tools.exec.pathPrepend` entries into the config I/O environment so process-wide binary checks can see the same directories as exec runs.

## Best Possible Solution

Define the contract deliberately: either pass a config-derived effective PATH into binary discovery, or implement process-wide PATH mutation with owned-entry cleanup, casing-safe helper reuse, focused tests, docs, and security acceptance.

## Reproduction Assessment

Yes. By source inspection, configure `tools.exec.pathPrepend` with a directory absent from `process.env.PATH`, then run a `hasBinary()`-backed skill/status check; current main keeps that setting scoped to exec environments while `hasBinary()` reads the process PATH.

## Solution Assessment

No. Direct global PATH mutation addresses the symptom, but this patch needs owned-entry reload semantics, PATH key casing support, tests/docs, real behavior proof, and explicit acceptance of the widened command-resolution surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track owned PATH entries across reloads:** `src/config/io.ts:1843-1845`
  - body: This only prepends configured directories and never records which entries came from the current config. If an in-process config reload removes or changes `tools.exec.pathPrepend`, old directories remain visible to `hasBinary()` and later PATH-based lookups until restart.
  - confidence: 0.89
- **[P2] Preserve the active PATH key casing:** `src/config/io.ts:1839-1843`
  - body: The patch reads and writes `deps.env.PATH` directly. Env objects can carry `Path` or another PATH key casing, so this can create a second key or miss the active PATH value instead of updating it through the existing PATH helper.
  - confidence: 0.84

## Security Review
