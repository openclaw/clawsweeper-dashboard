## Summary

Keep open. Current main and the latest release still have explicit channel-deny paths that use verbose/debug-only logging, and the open linked PR is an implementation candidate rather than a reason to close before merge.

## What This Changes

Request structured warning-level observability when an inbound channel route explicitly denies a message through legacy `allow:false` or current `enabled:false`, without changing fail-closed semantics.

## Best Possible Solution

Land a maintainer-approved PR that emits warning-level denial diagnostics at the plugin boundary for explicit route disables, either as an accepted Slack-first fix with a follow-up for cross-channel structure/dedupe or as one broader structured implementation.

## Reproduction Assessment

Yes. Source-level inspection shows Slack `enabled:false` resolves to `allowed=false`, returns false in `isChannelAllowed`, and only logs verbose output before the inbound handler drops the message; Discord has a parallel debug/verbose-only explicit-deny path.

## Solution Assessment

Unclear. Warning-level plugin-local logging is directionally right, but maintainers still need to decide whether a Slack-only unstructured warning is acceptable or whether this should be a structured cross-channel diagnostic with dedupe.
