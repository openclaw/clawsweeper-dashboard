## Summary

Keep this open: current main and the latest release still use verbose-only Slack channel-denial logs, the same-repository fix attempts were closed unmerged, and the remaining decision is the right logging boundary rather than whether the observability gap exists.

## What This Changes

The issue requests structured warning logs when a resolved channel route drops a message because channel config denies it.

## Best Possible Solution

Add warning-level, structured, rate-limited policy-denial logging at the maintainer-chosen Slack or shared channel ingress boundary while preserving fail-closed channel semantics.

## Reproduction Assessment

Yes from source: a Slack channel can resolve to `allowed: false`, `ctx.isChannelAllowed(...)` returns false, and the inbound caller drops the message after verbose-only logs. I did not run a live Slack gateway reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation shape: warning-level observability is the right direction, but maintainers still need to choose Slack-only logging versus a shared structured channel ingress helper with dedupe or rate limiting.
