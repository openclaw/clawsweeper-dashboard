## Summary

Keep open. Current main still drops explicit Slack channel-deny messages with verbose-only logging, Discord has the same class of verbose-only explicit-deny path, and https://github.com/openclaw/openclaw/pull/58640 is still open with closing syntax for this issue.

## What This Changes

The issue requests structured warning logs when inbound channel messages are dropped because an explicit channel route is configured with `allow: false`.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/58640 or a replacement merges with plugin-owned warning logs for explicit channel-deny decisions, preserving fail-closed routing semantics.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a Slack channel entry as disabled, send a room message, and current main returns false from `isChannelAllowed` before dropping the message after verbose-only logging.

## Solution Assessment

Unclear. Warning at the plugin denial boundary is the right direction, but maintainers should decide whether a Slack-only fix satisfies the broader gateway/channel request while a similar Discord path remains.
