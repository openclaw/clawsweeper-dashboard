## Summary

Keep open. Current main still drops explicit channel denials without the requested warning on the inspected Slack and Discord paths, and the linked open PR #58640 uses closing syntax for this issue, so the issue should remain open until that PR is reviewed, merged, or replaced with the broader provider-aware fix.

## What This Changes

Emit a structured warning when an inbound message is dropped because the resolved channel route has `allow: false`.

## Best Possible Solution

Keep the issue open and use PR #58640 as the active implementation candidate, but require the final fix to log explicit `allow:false` denials with enough structured context to diagnose the policy decision and with tests that assert warning behavior on the affected channel path.

## Reproduction Assessment

Yes. A code-level reproduction is clear on current main: configure a Slack room or Discord guild channel route with `allow:false`, send an inbound message on that route, and the inspected gating code returns a drop while emitting only debug/verbose logging.

## Solution Assessment

Unclear. A warning at the explicit-deny branch is the right narrow direction, but the current linked PR appears Slack-only while the issue requests gateway/channel observability more generally, so maintainer review should confirm scope before landing.
