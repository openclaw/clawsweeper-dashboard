## Summary

Keep this open: current main and v2026.6.10 still only emit verbose Slack channel-denial logs before dropping the inbound message, the same-repository fix attempts were closed unmerged, and the open fork PR is dirty/unproven rather than a safe canonical fix.

## What This Changes

The issue requests structured warning logs when a gateway/channel route drops a message because its resolved config has allow: false.

## Best Possible Solution

Add warning-level, structured, rate-limited policy-denial logging at the maintainer-chosen Slack or shared channel ingress boundary while preserving fail-closed channel semantics.

## Reproduction Assessment

Yes from source: a Slack channel config can resolve to allowed:false, ctx.isChannelAllowed returns false after verbose-only logs, and the caller drops the message. I did not run a live Slack gateway reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation shape: warning-level observability is the right outcome, but maintainers still need to choose Slack-only logging versus a shared structured channel ingress helper with dedupe/rate limiting.
