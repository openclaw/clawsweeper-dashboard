## What This Changes

Request warning-level structured observability when gateway/channel routing drops a message because the resolved route is explicitly disallowed.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/58640 or a replacement lands, then either close with proof or split any remaining cross-channel structured warning/rate-limit work into a narrower follow-up.

## Reproduction Assessment

Yes, source-level: configure a Slack room channel entry with enabled:false under groupPolicy open, then ctx.isChannelAllowed returns false after only logVerbose output and the inbound handler drops the message. I did not run a live Slack or Discord gateway session in this read-only review.

## Solution Assessment

Unclear. A plugin-boundary warning is the right direction, but maintainers still need to decide whether a Slack-only warning satisfies the broader structured cross-channel observability and dedupe request.
