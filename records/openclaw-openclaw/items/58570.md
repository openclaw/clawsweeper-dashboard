Keep this open: current main and latest release still use verbose-only Slack policy-denial logging, and the remaining linked fork PR is open but conflicting and unmerged. ## What This Changes

The issue requests structured warning logs when a resolved channel route drops an inbound message because its config has `allow: false`.

## Best Possible Solution

Add warning-level, structured, rate-limited policy-denial logging at the maintainer-chosen Slack or shared channel ingress boundary, without changing fail-closed channel semantics.

## Reproduction Assessment

Yes from source inspection: Slack channel config can resolve `allowed:false`, `ctx.isChannelAllowed(...)` returns false after only verbose logs, and the inbound caller then drops the message. I did not run a live Slack gateway reproduction in this read-only review.

## Solution Assessment

Unclear as a product choice: warning-level observability is the right direction, but maintainers still need to choose Slack-only versus shared structured logging and dedupe before implementation.
