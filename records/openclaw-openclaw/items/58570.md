## What This Changes

The issue requests warning-level, structured, preferably deduped logging when inbound channel policy denies a message because a route resolves to allow/enabled false.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/58640 or a replacement lands, then close with proof or split remaining cross-channel structured denial logging into a narrower follow-up.

## Reproduction Assessment

Yes, source-level: configure a Slack room channel entry with enabled:false under groupPolicy open, and current main returns false after only logVerbose output before the inbound handler drops the message. Discord has a parallel source-level explicit-deny path with logDebug/logVerbose only.

## Solution Assessment

Unclear. A plugin-boundary warning is the right direction, but maintainers still need to decide whether a Slack-only unstructured warning satisfies this broader structured cross-channel observability request.
