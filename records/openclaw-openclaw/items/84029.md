## Summary

Keep open: the fix is useful and targets the right CLI path, but the branch is not merge-ready because explicit out-of-range `--port` values currently get the config-port diagnostic and the supplied proof is still source-probe/test output rather than a real CLI run.

## What This Changes

The PR replaces `node run`'s silent port fallback with explicit validation through `defaultRuntime` and adds `registerNodeCli` regression tests for invalid, valid, and config-fallback port cases.

## Best Possible Solution

Land one canonical CLI validation fix that rejects explicit invalid `--port` values with the existing `formatInvalidPortOption("--port")` path, reports saved config-port problems separately, preserves omitted-port fallback, and includes real CLI proof.

## Reproduction Assessment

Yes from source inspection: current main parses invalid explicit `opts.port` to null, then `parsePortWithFallback` substitutes the saved/default port before `runNodeHost`. I did not run the foreground node host during this read-only review.

## Solution Assessment

No as currently patched: the overall direction is the right narrow fix, but explicit out-of-range `--port` values should use the `--port` diagnostic instead of blaming `node.gateway.port` config.
