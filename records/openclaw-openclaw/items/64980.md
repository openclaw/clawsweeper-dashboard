## Summary

Keep open. Current main still drops Matrix replacement events before routing and does not expose inbound streaming, finality, live-marker, or replacement metadata through the Matrix agent context, shared hook context, or core message context; choosing the public contract is a maintainer product/API decision.

## What This Changes

Expose Matrix streaming or finality metadata to agents so multi-agent rooms can avoid acting on partial message chunks.

## Best Possible Solution

Add a backward-compatible typed inbound draft/final/replacement state, or explicitly suppress non-final Matrix draft events, then propagate the chosen behavior through Matrix routing, `MsgContext`, plugin hook mappers, focused tests, and Matrix docs.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from this review. Source inspection on current main does confirm that inbound Matrix replacement/finality state is dropped or absent before it can reach agents or plugin hooks.

## Solution Assessment

Unclear as proposed. A finality signal would address the user problem, but a typed public inbound state or deliberate suppression policy is safer than ad hoc raw `is_streaming`, `is_final`, or full Matrix relation exposure.
