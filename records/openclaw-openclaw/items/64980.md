## Summary

Keep open. Current main still has the Matrix outbound draft/live machinery, but inbound Matrix replacement/live/finality state is not exposed to agents or plugin hooks, and choosing whether to expose typed metadata or suppress draft events is a public contract decision.

## What This Changes

Expose inbound Matrix streaming, finality, or replacement metadata to agents so multi-agent rooms can avoid acting on partial streamed message chunks.

## Best Possible Solution

Add a backward-compatible typed inbound Matrix/core contract for draft/final/replacement state, or explicitly suppress Matrix live draft events until finalization, then propagate the chosen state through Matrix routing, `MsgContext`, plugin hook mappers, and focused contract tests.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection on current main does confirm the requested metadata is absent from agent/plugin contexts and Matrix replacements are dropped before routing.

## Solution Assessment

Unclear as proposed. A finality signal would address the user problem, but a typed public inbound state or deliberate routing suppression policy is safer than ad hoc raw `is_streaming`, `is_final`, or Matrix relation fields.
