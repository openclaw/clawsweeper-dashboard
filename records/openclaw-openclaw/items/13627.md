## Summary

Keep open. Current main still serializes Slack sends without enforcing the requested configurable minimum delay, and the related broad cross-channel throttle PR was closed unmerged.

## What This Changes

Add a Slack-specific configurable outbound send throttle so rapid sequential Slack messages receive distinct timestamps and display in order.

## Best Possible Solution

Add a Slack-owned minimum inter-send delay at the outbound post/follow-up layer, with schema/types/UI hints/docs/changelog and coverage for normal sends, DMs, thread replies, block replies, chunked media/text, and slash response_url follow-ups.

## Reproduction Assessment

Yes for the source-level gap: current main queues Slack sends but has no configurable minimum delay between outbound Slack posts. I did not rerun the live Slack same-second display ordering behavior in this read-only review.

## Solution Assessment

Yes. A Slack-owned configurable delay is narrower than the closed broad cross-channel rate-limit PR and fits the existing Slack send/config surfaces if it covers every Slack post and response_url follow-up path.
