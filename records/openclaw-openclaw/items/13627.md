## Summary

Keep open. Current main and the latest shipped release serialize Slack sends per target but still do not expose or apply a configurable minimum inter-send delay; the linked broad implementation was closed unmerged, and the remaining choice is the Slack-scoped config/default and latency tradeoff.

## What This Changes

The issue requests a Slack-specific, configurable minimum delay between outbound messages so rapid sends do not display out of order in Slack.

## Best Possible Solution

Add a Slack-owned configurable minimum inter-send delay covering normal sends, DMs, threads, chunks, media follow-ups, block replies, and slash response_url follow-ups, with schema/types/UI hints/docs and focused regression tests.

## Reproduction Assessment

Yes for the source-level gap: current main queues Slack sends per target but has no config field or delay between post calls, and slash follow-ups still loop without spacing. I did not establish a live Slack workspace UI repro for same-second display ordering.

## Solution Assessment

Yes directionally, but the exact Slack config key, default delay, and latency tradeoff need maintainer product judgment. A focused Slack plugin change is narrower and safer than reviving the closed broad cross-channel throttle PR.
