## Summary

Keep open. Current main and the latest release still serialize Slack sends without the requested configurable minimum-delay throttle, and the only related implementation PR is closed unmerged.

## What This Changes

Request a Slack-specific configurable outbound send throttle so rapid sequential Slack messages get distinct timestamps and display in send order.

## Best Possible Solution

Add a Slack-owned throttle setting only after the default and config shape are approved, then apply it consistently to normal sends, DMs, thread replies, block replies, text chunks, media follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes for the missing implementation path: current source shows Slack postMessage and slash follow-up paths have no minimum-delay throttle. No live Slack workspace proof was run for the display-order symptom itself.

## Solution Assessment

Partly. A Slack-owned throttle is the right boundary, but the proposed always-on one-second default needs maintainer approval because it changes reply latency and adds a new config surface.
