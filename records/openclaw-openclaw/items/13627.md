## Summary

Keep open. Current main still serializes Slack sends without a configurable minimum delay, and the only linked implementation PR is closed unmerged; the remaining decision is the Slack config/default latency tradeoff, not cleanup closure.

## What This Changes

Request a Slack-specific configurable outbound send throttle so rapid sequential Slack messages get distinct timestamps and preserve display order.

## Best Possible Solution

Add a Slack-owned configurable minimum inter-send delay covering normal sends, DMs, threads, chunks, media follow-ups, block replies, and slash response_url follow-ups, with schema/types/UI hints/docs and focused regression tests.

## Reproduction Assessment

Yes for the source-level gap: current main queues Slack sends per target but has no config field or delay between post calls, and slash follow-ups still loop without spacing. I did not reproduce Slack's same-second display ordering in a live workspace.

## Solution Assessment

Yes directionally, but not ready for automatic implementation until maintainers choose the Slack config key, default delay, and latency tradeoff. A focused Slack plugin change is narrower than the closed broad cross-channel throttle PR.
