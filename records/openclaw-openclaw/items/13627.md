## Summary

Current main serializes Slack sends per account, token, destination, and thread, while Slack documents fractional, conversation-unique message timestamps; the requested one-second configurable delay therefore remains an unproven product change rather than a demonstrated bug fix. Keep this open for maintainer judgment, but implementation should not begin without a real workspace reproduction showing reordered `ts` values.

## What This Changes

Add a configurable Slack-only minimum delay between sequential outbound messages so multipart replies receive distinct timestamps and display in order.

## Best Possible Solution

Reproduce the behavior in a real Slack workspace and capture the exact message `ts` values and send path; only then choose a Slack-plugin-owned ordering fix, preferring an internal scheduling correction over a new user setting unless operators demonstrably need control.

## Reproduction Assessment

No. The report has no live Slack transcript or reordered `ts` pair, current main already serializes canonical sends, and Slack documents fractional conversation-unique message timestamps rather than one-second-resolution IDs.

## Solution Assessment

No. A configurable one-second throttle is not the best established solution because it adds latency and configuration before the underlying display-ordering failure or affected send surface has been proven.
