## Summary

Keep open. Current main serializes Slack sends per target but still has no Slack-specific configurable minimum inter-send delay, and the two related outbound-throttle PRs are closed unmerged.

## What This Changes

Request a Slack-specific configurable minimum delay between outbound messages so Slack timestamp ordering stays stable for rapid multi-message replies.

## Best Possible Solution

Add a Slack-owned minimum inter-send delay covering normal sends, blocks, DMs, threads, chunks, media follow-ups, and slash response_url follow-ups, with schema/types/UI hints/docs/changelog and focused regression coverage.

## Reproduction Assessment

Yes for the source-level gap: current main queues Slack sends but does not enforce a configurable delay between Slack posts. I did not live-test Slack's same-second display ordering in this read-only review.

## Solution Assessment

Yes for direction, pending maintainer approval of the new config surface and default. A focused Slack plugin throttle is narrower than the closed broad cross-channel PR and matches the owner boundary if it covers every Slack post surface.
