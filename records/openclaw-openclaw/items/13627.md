## Summary

Keep open. Current main still only serializes Slack sends per target; it does not expose or enforce the requested configurable minimum delay, and the related broad throttle PR was closed unmerged.

## What This Changes

Add a Slack-specific configurable outbound send throttle so rapid multi-message replies get distinct Slack timestamps.

## Best Possible Solution

Add a Slack-owned configurable minimum inter-send delay covering normal sends, blocks, DMs, threads, chunks, media follow-ups, and slash response_url follow-ups, with schema/types/UI hints/docs/changelog and focused regression coverage.

## Reproduction Assessment

Yes for the source-level gap: current main queues Slack sends but does not expose or enforce a configurable delay between Slack posts. I did not live-test Slack's same-second display ordering in this read-only review.

## Solution Assessment

Yes for the requested direction, pending maintainer approval of the config surface and default. A focused Slack plugin throttle is narrower than the closed broad cross-channel PR and matches the owner boundary if every Slack post surface is covered.
