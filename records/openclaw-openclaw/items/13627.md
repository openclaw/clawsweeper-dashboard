## Summary

Keep open. Current main serializes Slack sends per account/token/recipient/thread, but it still has no configurable minimum-delay throttle for Slack timestamp ordering, and the only linked implementation PR was closed unmerged.

## What This Changes

Issue requests a Slack-specific configurable outbound send throttle so rapid sequential messages receive distinct Slack timestamps and display in order.

## Best Possible Solution

Add a focused Slack plugin throttle setting and apply it consistently to normal sends, DMs, thread replies, block replies, text chunks, media-caption follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes for the source-level gap: current main queues Slack sends but does not enforce any minimum time between outbound posts. The live Slack same-second display ordering behavior was not rerun in this read-only review.

## Solution Assessment

Yes. A Slack-owned configurable delay is narrower than the closed broad cross-channel throttle PR and fits the existing Slack send/config surfaces.
