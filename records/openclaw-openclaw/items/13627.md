## Summary

Keep open. Current main serializes Slack sends per account/token/recipient/thread, but it still lacks the requested configurable minimum-delay throttle that would force distinct Slack message timestamps; the related broad outbound throttling PR is closed unmerged.

## What This Changes

Request a Slack-specific configurable outbound send throttle so rapid sequential channel, thread, and DM messages are spaced enough for Slack timestamp ordering.

## Best Possible Solution

Add a Slack-owned account-level throttle setting with strict schema/types/UI hints/docs and consume one shared delay helper across normal sends, DMs, thread replies, block/presentation sends, chunked text, media-caption follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call sendMessageSlack twice rapidly, or send a chunked Slack reply with a fake Slack client, and observe that current main serializes calls without any timer-enforced minimum delay.

## Solution Assessment

Yes. A focused Slack plugin delay is the narrow maintainable fix; the closed broad multi-channel rate-limiting PR was larger than this Slack timestamp-ordering issue requires.
