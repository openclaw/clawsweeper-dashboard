## Summary

Keep open. Current main serializes Slack writes and same-target sends, but it still has no configurable minimum inter-send delay for Slack timestamp ordering; the only linked implementation PR is closed unmerged and the remaining choice is a Slack config/default product decision.

## What This Changes

Request a Slack-specific configurable outbound send throttle so rapid sequential Slack messages receive distinct timestamps and display in order.

## Best Possible Solution

Add a Slack-owned minimum inter-send delay that covers normal sends, blocks, DMs, threads, chunks, media follow-ups, and slash response_url follow-ups, with schema/types/UI hints/docs/changelog and focused regression coverage after maintainers choose the config shape and default.

## Reproduction Assessment

Yes for the source-level gap: current main queues Slack sends but does not enforce a configurable delay between Slack posts. I did not live-test Slack's same-second display ordering in this read-only review.

## Solution Assessment

Yes, pending maintainer approval of the new config surface and default. A focused Slack plugin throttle is narrower than the closed broad cross-channel PR and matches the owner boundary if it covers all Slack post surfaces.
