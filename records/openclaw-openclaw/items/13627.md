## Summary

Keep open: current main and v2026.6.5 still serialize same-target Slack sends without the requested configurable minimum inter-send delay, and the only linked implementation PR is closed unmerged.

## What This Changes

The issue requests a Slack-specific configurable outbound send throttle so rapid Slack messages keep timestamp/display order.

## Best Possible Solution

Add a Slack-plugin-owned configurable minimum inter-send delay and apply it consistently to normal sends, DMs, thread replies, block replies, text chunks, media-caption follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes for the source-level gap: current main queues same-target Slack sends but has no configurable delay between post calls, and slash follow-ups still loop without spacing. I did not establish a live Slack workspace UI reproduction of same-second display ordering.

## Solution Assessment

Yes directionally: a focused Slack plugin setting is narrower than reviving the closed broad cross-channel throttle PR. The exact config key, default delay, and latency tradeoff still need maintainer product judgment.
