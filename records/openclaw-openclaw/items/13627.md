## Summary

Keep open. Current main serializes Slack sends per account/token/recipient/thread, but it still has no configurable minimum delay to force distinct Slack message timestamps, and the related broad throttle PR #26732 was closed unmerged.

## What This Changes

Request a configurable Slack outbound send throttle, defaulting around one second, so rapid channel, thread, and DM messages keep Slack timestamp ordering.

## Best Possible Solution

Add a Slack-owned account-level throttle setting with strict schema/types/UI hints/docs and apply one shared delay helper across normal sends, DMs, thread replies, block sends, text chunks, media-caption follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call sendMessageSlack twice rapidly, or send a chunked Slack reply with a fake Slack client, and observe that current main serializes calls without any timer-enforced minimum delay.

## Solution Assessment

Yes. A focused Slack plugin delay is the narrow maintainable fix; the closed broad multi-channel rate-limiting PR #26732 was larger than this Slack timestamp-ordering issue requires.
