## Summary

Keep open: current main and v2026.6.5 still serialize Slack writes without a configurable minimum inter-send delay; the only linked throttle PR is closed unmerged and no safer canonical replacement is open.

## What This Changes

The issue requests a configurable Slack outbound send throttle so rapid multi-message replies get distinct Slack timestamps and preserve display order.

## Best Possible Solution

Approve a Slack-specific config shape/default, then implement the delay inside the Slack plugin across normal sends, DMs, thread replies, block replies, chunked text, media-caption follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes for the source-level gap: current main queues same-target Slack sends but has no configurable delay between post calls, and slash follow-ups still loop without spacing. I did not run a live Slack workspace UI reproduction for same-second display ordering.

## Solution Assessment

Unclear until maintainers decide the latency tradeoff; a Slack-plugin-owned throttle is the right boundary, but the exact config key, default delay, and opt-out behavior need product judgment.
