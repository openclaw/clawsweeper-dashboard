## Summary

Keep open. Current main still discards delivery-result status in the normal agent-command and Gateway path, so a recovered or fallback run can look successful even when channel delivery returned no result. The isolated cron path has narrower delivery-status handling, but that does not cover the thread-bound chat path; #53961 is related but still an open draft.

## What This Changes

The issue reports that recovered or fallback LLM runs can produce a valid transcript response while channel delivery is silently skipped.

## Best Possible Solution

Keep this as the canonical reliability bug for recovered/fallback agent-run delivery drops. Land or revise #53961, or an equivalent structured delivery-status contract, then add a narrow post-condition at the agentCommandFromIngress/deliverAgentCommandResult boundary that retries only when delivery was requested but not actually attempted or was definitively failed using fresh session delivery context.

## Reproduction Assessment

Yes. The issue has high-confidence field reproductions across Discord and Telegram, and current main exposes a focused source-level reproduction seam by making deliverOutboundPayloads return [] after a successful agent result and observing that the Gateway still treats the run as completed.

## Solution Assessment

Yes. The best direction is a structured delivery status plus a carefully scoped fresh-context retry; a provider-specific overloaded_error fix would miss the model-fallback case, while a blanket retry after any empty result risks duplicate sends.
