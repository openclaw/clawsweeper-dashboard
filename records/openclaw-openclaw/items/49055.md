## Summary

This issue should stay open: current main has better delivery-status reporting and pending-final-delivery markers, but the inspected paths still do not perform a fresh-context retry or general replay for recovered/fallback final replies that were generated but not sent.

## What This Changes

Report that recovered Anthropic overload or model-fallback runs can record a final assistant reply in the session transcript without delivering it to Discord or Telegram.

## Best Possible Solution

Add a route-safe final-delivery post-condition that refreshes session/route context and retries or resumes only unambiguous no-send final replies, with duplicate-delivery protection and live Discord/Telegram or maintainer-accepted QA proof.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. The historical reports are concrete, and current source still shows the plausible no-retry boundary after a generated final reply, so this is source-reproducible but not fully live-reproduced.

## Solution Assessment

No, the current shipped status/pending-marker work is only a foundation. The best fix still needs an explicit, idempotent retry or replay rule for no-send final replies after recovery/fallback paths.
