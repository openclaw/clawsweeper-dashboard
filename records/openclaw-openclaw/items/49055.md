## Summary

Keep open. Current main still lacks a status-aware post-condition for the general agent-command delivery path, and the related delivery-status work is still an open draft rather than shipped behavior.

## What This Changes

The issue requests that recovered or fallback agent runs that produce valid responses still deliver their outbound channel message and surface delivery failures.

## Best Possible Solution

Keep this as the canonical bug, land or revise #53961 or an equivalent delivery-status contract, then add a scoped fresh-context retry/post-condition at the agent-command delivery boundary with regression coverage.

## Reproduction Assessment

Yes. Field reports cover Discord and Telegram, and current main has a focused source reproduction by making `deliverOutboundPayloads` return `[]` after a successful agent result while `deliverAgentCommandResult` and Gateway still report normal completion.

## Solution Assessment

Yes, the best direction is structured delivery status plus a carefully scoped fresh-context retry. A provider-specific `overloaded_error` fix would miss the model-fallback reproduction, and blanket retries risk duplicate sends.
