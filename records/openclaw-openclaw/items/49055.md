## Label Justifications

- `P1`: Successful agent runs can still leave users without a Discord or Telegram final reply after overload or fallback recovery, breaking a live channel workflow.
- `impact:message-loss`: The reported failure is generated final replies being absent from Discord or Telegram despite transcript completion.
- `impact:session-state`: The suspected failure mode is stale delivery/session context after recovery or fallback, with transcript state and channel delivery state disagreeing.

## Summary

This issue should stay open: current `main` has durable delivery status and pending-final-delivery markers, but the source still shows final delivery is resolved and attempted once from one session/route context, with heartbeat-only replay for pending final text and no fresh-context retry for recovered/fallback no-send outcomes.

## What This Changes

The issue asks OpenClaw to prevent final channel replies from being silently dropped after Anthropic overload recovery or model fallback when the session transcript records a successful response.

## Best Possible Solution

Add a narrow delivery completion post-condition that refreshes session/route context and resumes or retries only unambiguous no-send final replies, with idempotency protection and live Discord/Telegram or maintainer-accepted QA-channel proof.

## Reproduction Assessment

No high-confidence live reproduction was run against current `main`. Source inspection still shows the plausible boundary: final delivery is attempted once from one resolved session/route context, while the issue comments provide matching Discord and Telegram field observations.

## Solution Assessment

No, current `main` is only a partial foundation. Durable status and pending markers are useful, but the maintainable fix still needs an explicit route-safe retry/resume rule for unambiguous no-send outcomes with duplicate-send protection.
