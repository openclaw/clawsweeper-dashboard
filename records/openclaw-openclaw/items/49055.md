## Summary

Keep open. Current `main` and `v2026.5.12` have durable delivery status and pending-final-delivery markers, but the recovered/fallback final-reply path still resolves delivery context once, sends once, and returns without a fresh-context retry or user-message replay post-condition.

## What This Changes

Ensure recovered or fallback-completed assistant replies are delivered to Discord or Telegram when the transcript records a clean final response.

## Best Possible Solution

Add a narrow delivery completion post-condition that retries or resumes only route-safe no-send outcomes after refreshing session context, with duplicate-send protection and live channel or qa-channel proof.

## Reproduction Assessment

No high-confidence live reproduction against current main is available. Source inspection still shows the relevant gap, and the issue comments provide matching Discord and Telegram field reports.

## Solution Assessment

No, current main is only a partial solution. Durable status and pending markers are the right foundation, but maintainers still need to define the safe retry/resume rule for no-send outcomes with route preservation and idempotency.
