## Summary

Keep open. Current main now exposes durable delivery status and keeps pending final-delivery state, but the normal agent-command path still makes one delivery attempt and returns without re-resolving fresh session delivery context or retrying the recovered/fallback reply.

## What This Changes

Ensure recovered or fallback LLM replies that complete successfully after provider overload are delivered to Discord/Telegram instead of only being recorded in the transcript.

## Best Possible Solution

Add a narrow delivery post-condition that uses deliveryStatus to retry or resume only unambiguous no-send outcomes with freshly resolved, route-safe session context and duplicate-send protection.

## Reproduction Assessment

No live high-confidence current-main reproduction was run in this read-only review. The remaining gap is source-reproducible: agent-command makes one delivery call and returns without a fresh-context retry, while the issue discussion provides matching Discord and Telegram field reports.

## Solution Assessment

No, current main is only a partial solution. Durable status and pending-final-delivery state are the right foundation, but the maintainable fix still needs a retry/idempotency rule at the delivery completion boundary.
