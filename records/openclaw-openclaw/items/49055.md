## Summary

Keep open. Current `main` has durable delivery status and pending-final-delivery markers, but the recovered/fallback final-reply path still performs one delivery attempt from the supplied session entry and has no route-safe fresh-context retry or user-message replay post-condition.

## What This Changes

Reported bug where overload or model-fallback recovery records a successful assistant response in the transcript but silently fails to deliver it to Discord or Telegram.

## Best Possible Solution

Add a narrow delivery completion post-condition that uses `deliveryStatus` to retry or resume only route-safe no-send outcomes after refreshing session context, with duplicate-send protection and live Discord/Telegram or accepted QA-channel proof.

## Reproduction Assessment

No high-confidence live reproduction was run against current `main`. The issue has concrete Discord and Telegram field reports, and source inspection still shows one-shot delivery without fresh-context retry, but a live recovered-overload/fallback channel path was not exercised.

## Solution Assessment

No, current `main` is only a partial solution. Durable status and pending markers are the right foundation, but the remaining fix needs an explicit retry/resume rule for unambiguous no-send outcomes with idempotency and route preservation.
