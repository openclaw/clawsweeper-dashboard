## Label Justifications

- `P1`: Successful agent runs can still leave users without a Discord or Telegram final reply after overload/fallback recovery, breaking a real channel workflow.
- `impact:message-loss`: The reported user-visible failure is generated final replies being absent from Discord or Telegram despite transcript completion.
- `impact:session-state`: The suspected failure mode is stale delivery/session context after recovery or fallback, with transcript state and channel delivery state disagreeing.

## Summary

Keep open. Current main and v2026.5.18 have durable delivery status and pending-final-delivery markers, but the recovered/fallback path still resolves a delivery route once, sends once, and returns without a fresh-context retry or user-message replay post-condition.

## What This Changes

The issue asks OpenClaw to stop dropping successful Discord and Telegram final replies after Anthropic overload recovery or model fallback invalidates delivery context.

## Best Possible Solution

Add a narrow delivery completion post-condition that uses deliveryStatus to retry or resume only route-safe no-send outcomes after refreshing session context, with duplicate-send protection and live channel or accepted qa-channel proof.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection still shows the plausible current boundary: final delivery is attempted once from one resolved sessionEntry/route and pending replay is heartbeat-only, while the issue comments provide matching Discord and Telegram field observations.

## Solution Assessment

No, current main is only a partial foundation. Durable status and pending markers are useful, but the maintainable fix still needs an explicit route-safe retry/resume rule for unambiguous no-send outcomes with idempotency proof.
