## Summary

Keep open. Current main has better delivery-status projection and durable pending-final-delivery state, but the recovered/fallback reply path still returns after one delivery attempt without re-resolving fresh session context or applying a route-safe retry/post-condition.

## What This Changes

The issue asks OpenClaw to deliver a valid recovered assistant response to Discord or Telegram after an earlier provider overload or fallback instead of recording it only in the transcript.

## Best Possible Solution

Add a narrow delivery completion post-condition that uses deliveryStatus to retry or resume only route-safe no-send outcomes after refreshing session context, with duplicate-send protection and live channel or qa-channel proof.

## Reproduction Assessment

No, not with a live current-main run. The source path remains reproducible by inspection: final delivery is attempted once and returns without fresh-context retry, while the issue discussion provides matching Discord and Telegram field reports.

## Solution Assessment

No, current main is only a partial solution. Durable delivery status and pending markers are the right foundation, but the maintainable fix still needs an explicit retry/resume rule for unambiguous no-send outcomes.
