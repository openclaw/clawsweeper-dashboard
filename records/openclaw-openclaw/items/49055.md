## Summary

Keep this open. Current main has added useful delivery status and durable pending-final-delivery plumbing, but the normal agent-command path still performs one delivery attempt and does not re-resolve fresh session delivery context or retry the recovered/fallback reply before returning.

## What This Changes

The issue asks for cleanly recovered Anthropic overload or model-fallback runs to still deliver their final reply to Discord/Telegram instead of only recording it in the transcript.

## Best Possible Solution

Add a narrow agent-command delivery post-condition that retries once with freshly resolved session delivery context only for unambiguous no-send outcomes, while preserving pendingFinalDelivery until a confirmed send or intentional suppression.

## Reproduction Assessment

Yes at source level, but not with a live channel run in this read-only review. Current main still has a single agent-command delivery call with no fresh-context retry, while the issue discussion provides multiple Discord and Telegram field reproductions.

## Solution Assessment

No, current main is only a partial solution. Structured delivery status and pending-final-delivery state are the right foundation, but the missing retry/idempotency rule is still the maintainable fix boundary.
