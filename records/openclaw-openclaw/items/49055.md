## Summary

Keep open. Current main and the latest release have better durable delivery status and pending-final-delivery markers, but the recovered/fallback reply path still resolves delivery context once, sends once, and returns without a route-safe fresh-context retry or user-message replay post-condition.

## What This Changes

The issue asks OpenClaw to deliver final replies to Discord or Telegram after an overloaded_error recovery or model fallback succeeds and records transcript output.

## Best Possible Solution

Add a narrow delivery completion post-condition that uses deliveryStatus to retry or resume only route-safe no-send outcomes after refreshing session context, with duplicate-send protection and live channel or qa-channel proof.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection still shows the relevant path: final delivery is attempted once from the current sessionEntry and returns without fresh-context retry, while issue comments provide matching Discord and Telegram field reports.

## Solution Assessment

No, current main is only a partial solution. Durable status and pending markers are the right foundation, but the safe fix still needs an explicit retry/resume rule for unambiguous no-send outcomes with idempotency and route preservation.
