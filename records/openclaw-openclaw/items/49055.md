## Summary

Keep open. Current main has durable delivery status and pending-final-delivery markers, but the relevant path still resolves delivery context once and lacks a route-safe fresh-context retry or user-message replay for recovered/fallback replies after a no-send outcome.

## What This Changes

The issue reports recovered provider/fallback completions being written to the session transcript but silently not delivered to Discord or Telegram.

## Best Possible Solution

Add a narrow delivery completion post-condition that uses deliveryStatus to retry or resume only route-safe no-send outcomes after refreshing session context, with duplicate-send protection and live channel or accepted qa-channel proof.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection still shows a credible path: final delivery is attempted once from the current sessionEntry without fresh-context retry, while issue comments provide Discord and Telegram field reports.

## Solution Assessment

No, current main is only a partial solution. Durable status and pending markers are the right foundation, but the safe fix still needs maintainer-approved retry scope, idempotency, and route-preservation semantics.
