## Summary

Keep open. The report is fresh, has concrete Docker logs plus corroborating follow-up comments, and current main still shows plausible Discord gateway READY race surfaces: IDENTIFY is launched fire-and-forget after HELLO, the post-limiter closed-socket branch returns silently, and the lifecycle READY observer still only reacts to a debug "websocket opened" event rather than an already-open socket. The linked fork PR is still open and targets this issue, so this issue should stay open until that work is reviewed, landed, or replaced.

## What This Changes

The issue reports an intermittent Discord startup race where the gateway READY event never arrives after login, leaving the bot online but unresponsive.

## Best Possible Solution

Keep this open until the linked implementation is reviewed and landed, or replaced by an equivalent target-repo Discord plugin fix. The durable fix should cover both missed startup READY observation and the IDENTIFY limiter/socket-close path with focused regression tests.

## Reproduction Assessment

Source inspection gives a plausible code-level path matching the report rather than an executable local reproduction.

## Solution Assessment

Unclear: current main's READY timeout/restart behavior is useful mitigation, but it is not enough to close while the IDENTIFY path can still silently return and an open targeted PR exists. The safer path is to review, land, or replace that PR with tests for the remaining race surface.
