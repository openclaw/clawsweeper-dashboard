## Summary

Keep open. The report is fresh, corroborated by follow-up comments, and still paired with an open unmerged implementation candidate; current main has Discord READY timeout/restart mitigation, but source inspection still shows the lower-level IDENTIFY limiter/socket-close path can silently return and lacks focused regression coverage.

## What This Changes

The issue reports an intermittent Discord gateway startup race where READY never reaches OpenClaw, leaving the Discord bot online but unresponsive after startup or reconnect.

## Best Possible Solution

Keep this open until the linked implementation is reviewed and landed, or replaced by an equivalent target-repo Discord plugin fix. The durable fix should cover both missed startup READY observation and the IDENTIFY limiter/socket-close path with focused regression tests.

## Reproduction Assessment

Source inspection still shows plausible race surfaces matching the discussion, and later comments suggest more than one Discord connection failure mode may be involved.

## Solution Assessment
