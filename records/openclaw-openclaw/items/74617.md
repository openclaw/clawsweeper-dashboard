## Summary

Keep open. Current main still has the lower-level Discord IDENTIFY limiter path that can return without surfacing an error when the socket is no longer open, and the issue is already tied to an open implementation candidate; the existing READY timeout/restart logic is useful mitigation but not strong enough evidence to close this regression.

## What This Changes

The issue reports an intermittent Discord gateway startup race where READY never reaches OpenClaw, leaving the Discord bot online but unresponsive after startup or reconnect.

## Best Possible Solution

Land or replace the focused Discord plugin fix in the target repository, with regression coverage for both missed startup READY observation and the identify-limiter socket-close path. The already-fixed #52372 registration race should stay separate from this remaining READY/IDENTIFY behavior.

## Reproduction Assessment

Unclear: the reporter provided concrete Docker steps, logs, and a raw WebSocket control, but I did not run a live Discord reproduction against current main. Source inspection still shows plausible race surfaces matching the discussion.

## Solution Assessment

Unclear: current main's startup polling and one-time restart is a reasonable mitigation, but it is not enough to close while the IDENTIFY path can still silently return and an open fix candidate exists. The safer path is to review, land, or replace that candidate with focused gateway tests.
