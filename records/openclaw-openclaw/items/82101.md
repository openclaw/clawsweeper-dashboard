## Summary

Keep open: the repro is credible, but the patch currently strips documented valid reasoning pass-back fields from assistant history, so it needs a shape-aware fix before merge.

## What This Changes

This PR adds a Chat Completions request-builder sanitizer that deletes `reasoning_details`, `reasoning_content`, and `reasoning` from assistant replay messages and adds a focused unit test.

## Best Possible Solution

Preserve supported reasoning replay fields and only remove or convert invalid shapes that OpenRouter rejects, with regression coverage for both the failing string case and valid array/string pass-back.

## Reproduction Assessment

Yes. The PR supplies a live curl reproduction for `assistant.reasoning_details` as a string returning 500, and the current `pi-ai` converter source shows how that invalid string shape can be emitted from a persisted thinking block.

## Solution Assessment

No. The narrow fix should target the invalid `reasoning_details` string shape while preserving documented `reasoning`, `reasoning_content`, and array-shaped `reasoning_details` pass-back fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve valid reasoning replay fields:** `src/agents/openai-transport-stream.ts:2393-2397`
  - body: OpenRouter supports passing `message.reasoning` strings, `reasoning_content` aliases, and `reasoning_details` arrays back on assistant history, and `pi-ai` can emit valid `reasoning_details` arrays from tool-call thought signatures. Deleting all three fields fixes the bad string shape but also drops valid reasoning continuity for supported models, so this should only remove or convert invalid shapes such as string-valued `reasoning_details`. ([openrouter.ai](https://openrouter.ai/docs/guides/best-practices/reasoning-tokens?utm_source=openai))
