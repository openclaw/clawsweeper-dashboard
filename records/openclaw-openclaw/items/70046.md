## Summary

Keep open. Current main still rejects bare time-only `--at` values, and the discussion identifies this PR as the clean active respin for #59441 after #59444 picked up unrelated churn, but the PR still needs a changelog entry and deterministic date-boundary tests before merge.

## What This Changes

The PR adds `HH:MM` / `HH:MM:SS` handling to CLI `parseAt`, resolves time-only values against today's UTC or `--tz` local date, validates invalid times/timezones, and adds focused parser tests.

## Best Possible Solution

Land this focused parser fix after adding an Unreleased changelog entry and freezing the new time-dependent tests, while leaving help-text wording to the separate #69975 review path.

## Reproduction Assessment

Yes. On current main, `cron add --at 09:00 --tz Asia/Shanghai` reaches `parseAt`, which has no time-only branch; `Date.parse('09:00')` is `NaN` in this Node runtime, so the CLI throws `Invalid --at`.

## Solution Assessment

Yes, with small amendments. Handling time-only input in CLI `parseAt` is the narrow maintainable fix because that layer has the `--tz` context and can reuse the existing offset-less zoned datetime helper; the lower-level absolute parser cannot infer an unambiguous date or timezone.
