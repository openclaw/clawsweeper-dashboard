## Summary

Keep open: this is the active clean respin for https://github.com/openclaw/openclaw/issues/59441 and current main still lacks bare time-only `--at` parsing. The patch looks functionally sound, but it still needs contributor real behavior proof and a small clock-stability fix in the new tests.

## What This Changes

The PR adds time-only `HH:MM`/`HH:MM:SS` handling to cron `parseAt`, applies `--tz` to those values, rejects invalid time/timezone inputs, and adds focused parser tests.

## Best Possible Solution

Land the focused parser support after the contributor adds redacted real cron behavior proof and the tests freeze or capture the clock; track the `+duration` help/parser mismatch separately unless maintainers expand scope.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main routes `--at` through `parseAt`, has no time-only branch, and the fallback `Date.parse('09:00')` returns `NaN` in this Node runtime. I did not run a full live `openclaw cron add` command in this read-only review.

## Solution Assessment

Yes, with one small test hardening. `parseAt` is the narrow layer that has access to `--tz`, and reusing the existing offset-less zoned datetime helper avoids pushing ambiguous time-only parsing into the lower-level absolute timestamp parser.
