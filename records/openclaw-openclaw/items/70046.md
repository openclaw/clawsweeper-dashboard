## What This Changes

The branch adds `HH:MM`/`HH:MM:SS` handling to cron CLI `parseAt`, resolves those values using `--tz` or UTC today, and adds parser tests for valid and invalid cases.

## Best Possible Solution

Land the focused parser change with redacted real CLI proof and a small help/docs update that names the new time-only `--at` forms without changing lower-level absolute timestamp parsing.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main routes `--at` through `parseAt`, has no time-only branch, and local Node reports `Date.parse('09:00')` as `NaN`. I did not run a live `openclaw cron add` command in this read-only review.

## Solution Assessment

Yes, the parser-layer approach is the best fit because `parseAt` has access to `--tz` and can reuse the existing offset-less zoned datetime helper. It should be paired with user-facing help/docs and real CLI proof before merge.
