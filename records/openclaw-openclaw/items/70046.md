## Summary

Keep open: current main still lacks the time-only `--at` parser path, and the PR is a plausible focused fix, but the external PR still needs after-fix real behavior proof and a small test hardening before merge.

## What This Changes

The PR adds `HH:MM`/`HH:MM:SS` handling to cron `parseAt`, applies `--tz` to those time-only values, returns `null` for invalid zones/times, and adds focused parser tests.

## Best Possible Solution

Land the focused parser support after the contributor adds redacted real cron behavior proof and the tests freeze or capture the clock; track the `+duration` help/parser mismatch separately unless maintainers expand scope.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main routes `--at` through `parseAt`, has no time-only branch, and `Date.parse('09:00')` returns `NaN` in this Node runtime. I did not run a full live `openclaw cron add` command in this read-only review.

## Solution Assessment

Yes, with one small test hardening. `parseAt` is the narrow layer that has access to `--tz`, and reusing the existing offset-less zoned datetime helper avoids pushing ambiguous time-only parsing into the lower-level absolute timestamp parser.
