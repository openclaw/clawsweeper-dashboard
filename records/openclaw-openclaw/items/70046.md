## Summary

Close for now: the patch is focused, technically plausible, and well-proven, but it adds a new public `cron --at` time-only grammar plus “today in UTC or --tz” semantics that current main does not implement or document and no maintainer has explicitly sponsored.

## What This Changes

The PR adds `HH:MM[:SS]` parsing for cron one-shot `--at` values, applies `--tz` to those time-only inputs, rejects invalid time-only cases, and updates CLI help/docs/tests.

## Best Possible Solution

Keep core cron on the documented ISO/duration contract unless a maintainer explicitly sponsors time-only scheduling and its exact “today” semantics.

## Reproduction Assessment

Yes by source inspection: current main `parseAt` has no time-only branch, so `09:00` falls through to absolute timestamp and duration parsing and returns null. I did not run a local CLI command because this read-only review is using source and PR proof.

## Solution Assessment

Unclear as a product decision: the implementation is a clean parser-layer solution, but accepting bare `HH:MM[:SS]` adds a new public CLI scheduling grammar with “today” semantics. A maintainer needs to decide whether that grammar belongs in core before merge.
