## Summary

Current main still lacks the time-only `--at` parser path, and this PR is the active scoped fix for the linked cron bug rather than obsolete cleanup. The patch looks directionally correct, but it should stay open because the external contributor has not provided after-fix real behavior proof and the new date-boundary assertions can flake around midnight.

## What This Changes

The PR adds time-only `HH:MM`/`HH:MM:SS` parsing to `cron add --at`, applies `--tz` to those inputs, rejects invalid time parts/timezones, and adds focused parser tests.

## Best Possible Solution

Land this focused parser fix after adding real CLI behavior proof and making the date-boundary tests deterministic.

## Reproduction Assessment

Yes by source inspection and a focused runtime check: current main has no time-only branch in `parseAt`, and `Date.parse('09:00')` is `NaN` in this Node runtime, so the CLI reaches the `Invalid --at` path.

## Solution Assessment

Yes, with a small test fix. `parseAt` is the narrow layer that has the `--tz` context and can reuse the existing offset-less zoned datetime helper; the branch should keep the tests deterministic and include real behavior proof before merge.
