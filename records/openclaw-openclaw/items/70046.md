## Summary

Keep open: current main still lacks a time-only `--at` parser path, and this PR is a focused fix candidate, but it still needs after-fix real behavior proof plus small test/help hardening before merge.

## What This Changes

The PR adds `HH:MM`/`HH:MM:SS` handling to cron `parseAt`, applies `--tz` to those values, returns `null` for invalid zones/times, and adds focused parser tests.

## Best Possible Solution

Land the focused `parseAt` support after contributor proof is added, the time-sensitive tests share a stable clock, and CLI help/scope is kept in sync.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main routes `--at` through `parseAt`, has no time-only branch, and `Date.parse('09:00')` returns `NaN` in this Node runtime. I did not run a full live `openclaw cron add` command in this read-only review.

## Solution Assessment

Yes, with small follow-ups. `parseAt` is the narrow layer that has `--tz`, and reusing the existing offset-less zoned datetime helper is maintainable; the tests and CLI help should be hardened before merge.
