## Summary

Keep open: current `main` and the latest release still lack bare `HH:MM`/`HH:MM:SS` handling for cron `--at`, and this refreshed PR is the active clean fix path, but it still needs real CLI behavior proof before merge.

## What This Changes

The PR adds time-only parsing for cron `--at`, resolves those inputs against UTC or the supplied IANA `--tz`, and adds focused parser tests for valid, invalid, and existing duration cases.

## Best Possible Solution

Land the parser fix with the same PR or a narrow follow-up that updates cron add/edit help and the cron docs to advertise `HH:MM`/`HH:MM:SS` semantics.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main routes cron `--at` through `parseAt`, has no time-only branch, and this Node runtime returns `NaN` for `Date.parse('09:00')`. I did not run a live `openclaw cron add` command in this read-only review.

## Solution Assessment

Yes for the code shape: `parseAt` is the narrow layer with access to `--tz`, and reusing the existing offset-less zoned datetime helper avoids pushing ambiguous time-only parsing into lower-level absolute timestamp parsing. It is not merge-ready until real behavior proof is added.
