## Summary

Keep open: current main still lacks bare time-only `--at` parsing, and this focused PR is the active implementation path with sufficient updated terminal proof. The patch looks functionally correct, but it should update public help/docs for the new accepted format before merge.

## What This Changes

The PR adds `HH:MM` and `HH:MM:SS` parsing to cron `--at`, resolves those values against UTC or the supplied `--tz`, and adds parser regression tests.

## Best Possible Solution

Land the parser fix with the same PR or a narrow follow-up that updates cron add/edit help and the cron docs to advertise `HH:MM`/`HH:MM:SS` semantics.

## Reproduction Assessment

Yes at source level: current main's `parseAt` has no bare time-only branch, so `09:00` reaches neither offset-less ISO handling nor duration parsing. I did not run the CLI because this review is read-only.

## Solution Assessment

Mostly yes: `parseAt` is the right layer because it receives `--tz` and can reuse the existing zoned offset-less datetime helper. The remaining improvement is to update public help/docs for the new format.
