## Summary

Keep open. Current main and the latest release still reject bare time-only `--at` input, while this PR is the active clean respin for the cron parser fix linked from #59441 and #59444. The implementation direction is narrow and appropriate, but the PR still needs normal maintainer review plus a required changelog entry and a small test-stability cleanup before merge.

## What This Changes

The PR adds a time-only branch to cron CLI `parseAt` and new parser tests for timezone resolution, invalid zones, out-of-range values, offset-less ISO datetimes, and durations.

## Best Possible Solution

Land the parser fix in the cron CLI layer, where `--tz` context exists, while keeping the lower-level absolute-time parser date-agnostic. Before merge, add the active-version changelog entry, harden the date-sensitive assertions, and either update help/docs here or explicitly leave that follow-up to the existing help-text track.

## Reproduction Assessment

Yes. The current-main path is source-backed: `cron add --at 09:00 --tz Asia/Shanghai` reaches `parseAt`, which lacks a time-only branch, and this runtime reports `Date.parse('09:00')` as `NaN`, so `resolveDirectSchedule` throws `Invalid --at`.

## Solution Assessment

Yes, with merge-prep caveats. Handling bare time-only strings in CLI `parseAt` is the narrow maintainable fix because it has the timezone option and can reuse `parseOffsetlessIsoDateTimeInTimeZone`; changing `parseAbsoluteTimeMs` would make a shared parser infer an ambiguous date without caller context.
