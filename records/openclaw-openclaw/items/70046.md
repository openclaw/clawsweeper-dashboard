## Summary

The patch is technically focused and well-proven, but it expands the documented public cron CLI grammar to accept `HH:MM[:SS]` without maintainer-confirmed product acceptance. That makes this a product-direction close, not a negative code review.

## What This Changes

The PR adds time-only `HH:MM[:SS]` parsing for cron `--at`, applies `--tz` to that input form, and updates CLI help, docs, and focused tests.

## Best Possible Solution

Leave the shipped ISO/duration cron `--at` contract in place unless a maintainer explicitly sponsors time-only input, then land a focused version with the same parser, docs, and regression coverage.

## Reproduction Assessment

Yes at source level: current main `parseAt` has no time-only branch, so `09:00` cannot become a schedule before `resolveDirectSchedule` throws. I did not execute CLI/tests because this review is read-only.

## Solution Assessment

The code shape is good if maintainers want the feature, because `parseAt` has the needed `--tz` and current-date context. As a repository action, merging is not the best path without maintainer acceptance of the new public CLI grammar and today-at-time semantics.
