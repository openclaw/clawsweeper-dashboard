## Summary

Close as unconfirmed product direction: the patch is focused, technically sound, and proven, but it adds a new documented public cron `--at` input family that current main does not define and no maintainer has explicitly sponsored.

## What This Changes

The PR adds `HH:MM[:SS]` handling to cron CLI `--at` parsing, applies `--tz` to those time-only inputs, and updates related help text, docs, and tests.

## Best Possible Solution

Keep the existing ISO/duration cron `--at` contract unless a maintainer explicitly sponsors `HH:MM[:SS]` as an accepted public CLI format and reopens or lands a sponsored branch.

## Reproduction Assessment

Yes from source inspection: current main routes `--at` through `parseAt`, and that function has no `HH:MM` time-only branch, so a bare `09:00` falls through to invalid input. I did not run a live CLI repro because this cleanup review is read-only.

## Solution Assessment

Technically yes if maintainers want the new contract: the CLI `parseAt` layer already has `--tz` and duration context, and the PR reuses the existing offset-less timezone helper. As a merge path, no: the remaining blocker is maintainer product acceptance, not the implementation mechanics.
