## Summary

Keep open: current main still rejects the reported time-only cron input and this PR is a focused, likely-correct parser change, but it is not merge-ready until the external contributor adds real behavior proof and the new date assertions stop depending on two live clock reads.

## What This Changes

The PR adds support for bare `HH:MM`/`HH:MM:SS` values in `cron add --at`, resolves them with `--tz`, handles invalid IANA zones, and adds focused parser tests.

## Best Possible Solution

Land the focused parser support after the contributor adds redacted real cron behavior proof and the tests freeze or capture the clock; track the `+duration` help/parser mismatch separately unless maintainers expand scope.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main routes `--at` through `parseAt`, has no time-only branch, and a focused Node check returned `NaN` for `Date.parse('09:00')`.

## Solution Assessment

Yes, with one small test hardening. `parseAt` is the narrow layer that receives `--tz`, and reusing the existing offset-less zoned datetime helper avoids pushing ambiguous time-only parsing into the lower-level absolute timestamp parser.
