## What This Changes

The PR adds `HH:MM`/`HH:MM:SS` parsing for cron `--at`, resolves those values with UTC or the supplied `--tz`, and adds parser regression tests.

## Best Possible Solution

Land the parser fix with the same PR or a narrow follow-up that updates cron add/edit help and cron docs to advertise `HH:MM`/`HH:MM:SS` time-only semantics.

## Reproduction Assessment

Yes at source level: current main's `parseAt` lacks a bare time-only branch, so `09:00` reaches neither offset-less ISO handling nor duration parsing. I did not run the CLI because this review is read-only.

## Solution Assessment

Mostly yes: `parseAt` is the right layer because it receives `--tz` and can reuse the existing zoned offset-less datetime helper. The safer finish is to update public help/docs for the new accepted format before merge.
