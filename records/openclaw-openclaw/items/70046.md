## What This Changes

The PR adds time-only `HH:MM[:SS]` parsing for cron `--at`, applies `--tz` to that form, and updates cron help, docs, and tests.

## Best Possible Solution

Land the focused parser/help/docs/test update if maintainers accept `HH:MM[:SS]` as an additive cron `--at` input.

## Reproduction Assessment

Yes at source level: current main `parseAt` has no `HH:MM[:SS]` branch, so time-only input falls through to absolute timestamp and duration parsing before `resolveDirectSchedule` throws. I did not execute CLI/tests because this review was read-only.

## Solution Assessment

Yes for the code shape: `parseAt` is the right layer because time-only input needs the CLI `--tz` option and today's date before delegation to the existing offset-less zoned parser. Changing `parseAbsoluteTimeMs` instead would make ambiguous time-only strings valid in lower cron/storage paths that lack that context.
