## Summary

Keep open for maintainer review: the parser change is focused and the PR body now provides sufficient terminal proof, but the new time-only `--at` format remains missing from public help/docs.

## What This Changes

The PR adds `HH:MM`/`HH:MM:SS` parsing for cron one-shot `--at` values, applies `--tz` to those time-only inputs, and adds parser regression tests.

## Best Possible Solution

Land the parser fix with the same PR or a narrow follow-up that updates cron add/edit help and cron docs for `HH:MM`/`HH:MM:SS` time-only `--at` semantics.

## Reproduction Assessment

Yes at source level: current main's `parseAt` has no time-only branch, so `09:00` reaches neither offset-less ISO handling nor duration parsing and returns null. I did not run the CLI because this review is read-only.

## Solution Assessment

Mostly yes: `parseAt` is the right layer because it receives `--tz` and can reuse the existing zoned offset-less datetime helper. The remaining safer path is to update public help/docs with the new accepted format.
