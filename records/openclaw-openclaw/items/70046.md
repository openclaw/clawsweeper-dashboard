## What This Changes

The PR adds `HH:MM[:SS]` parsing for cron `--at`, applies `--tz` to those time-only values, and updates cron help, docs, and tests.

## Best Possible Solution

Land a refreshed version only if maintainers accept `HH:MM[:SS]` as a supported cron `--at` shorthand, keeping resolution at the CLI schedule parser boundary and storing canonical ISO timestamps.

## Reproduction Assessment

Yes for the current behavior: source inspection shows current main routes `--at` through `parseAt(options.at, options.tz)`, and `parseAt` has no time-only branch before absolute timestamp and duration parsing. The expected `HH:MM[:SS]` behavior is an additive CLI capability rather than an already-documented current contract.

## Solution Assessment

Yes for implementation shape: the CLI parser has the `--tz` context and can continue persisting canonical ISO timestamps, while changing lower cron runtime parsing would broaden ambiguous timestamp semantics. The remaining question is product acceptance of the new public input format, not a line-level patch defect.
