## Summary

Keep open: current main still lacks the central time-only cron `--at` behavior, and the PR’s CLI-layer implementation is the right owner boundary with sufficient terminal proof. I found one non-blocking help-text mismatch for bare duration wording before landing.

## What This Changes

The PR changes cron one-shot schedule parsing so `--at` accepts `HH:MM[:SS]` time-only values, applies `--tz` for those values, and updates CLI help, docs, and parser tests.

## Best Possible Solution

Land the CLI parser/docs change after aligning `cron edit --help` duration wording with the accepted `20m` and `+20m` forms.

## Reproduction Assessment

Yes from source: current `parseAt` handles offset-less ISO datetimes, absolute timestamps, and durations, while bare `HH:MM` falls through to `parseAbsoluteTimeMs` and then returns `null`, causing `resolveDirectSchedule` to throw.

## Solution Assessment

Yes with one small copy fix: adding the time-only branch in the CLI `parseAt` layer is the right boundary because it has `--tz` context and can delegate zoned conversion to the existing offset-less datetime helper.
