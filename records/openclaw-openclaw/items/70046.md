## What This Changes

The branch adds bare `HH:MM`/`HH:MM:SS` handling in cron `parseAt`, resolves time-only values using UTC or `--tz`, rejects invalid time/timezone input, and adds parser unit tests.

## Best Possible Solution

Land the focused `parseAt` support after the contributor adds redacted real cron CLI proof and the date-boundary tests use a frozen or captured clock; track the `+duration` mismatch separately unless maintainers intentionally fold it into this parser cleanup.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main routes `--at` and `--tz` into `parseAt`, `parseAt` has no time-only branch, and this Node runtime returns `NaN` for `Date.parse('09:00')`, which becomes the current `Invalid --at` path.

## Solution Assessment

Yes, with one small test hardening. `parseAt` is the narrow layer that has the `--tz` context and can supply today's date, while the lower-level absolute timestamp parser should not guess a date for bare times.
