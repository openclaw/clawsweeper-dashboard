## Summary

Keep open. Current main still rejects bare time-only `--at` values, while this open PR is a focused implementation candidate; it needs small mechanical follow-up for the required changelog entry and date-boundary-sensitive tests before maintainer landing.

## What This Changes

The PR adds a CLI `parseAt` branch for `HH:MM` / `HH:MM:SS`, resolves it against today's UTC date or the requested IANA timezone, rejects invalid time-only/timezone input, and adds parser unit tests.

## Best Possible Solution

Land this focused parser fix after adding the required changelog entry and hardening the date-boundary-sensitive tests, while keeping help/docs alignment coordinated with the existing help-text PR.

## Reproduction Assessment

Yes. On current main, `cron add --at 09:00 --tz Asia/Shanghai` reaches `parseAt`, which lacks a time-only branch; `Date.parse('09:00')` is `NaN` on this Node runtime, so the CLI throws `Invalid --at`.

## Solution Assessment

Yes, with small follow-ups. Handling time-only input in CLI `parseAt` is the narrow maintainable fix because that layer has the `--tz` context and can reuse the existing offset-less zoned datetime helper; changing `parseAbsoluteTimeMs` would make the lower-level parser infer an ambiguous date.
