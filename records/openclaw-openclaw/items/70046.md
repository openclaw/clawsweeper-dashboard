## What This Changes

The PR adds time-only `HH:MM[:SS]` parsing for `cron --at`, applies `--tz` to those values, and updates cron CLI help, docs, and tests.

## Best Possible Solution

Keep the current ISO/duration `--at` contract unless a maintainer sponsors time-only syntax; if sponsored later, revive this focused parser/docs/test patch.

## Reproduction Assessment

Yes by source inspection for the unsupported current-main behavior: `parseAt` has no time-only branch, so `HH:MM` falls through to invalid input. I did not run parser commands in this read-only review.

## Solution Assessment

Technically yes if maintainers choose this behavior: `parseAt` is the right CLI boundary and the patch reuses existing timezone conversion. As a merge path, no: it should not land without maintainer sponsorship because it creates a new public CLI syntax.
