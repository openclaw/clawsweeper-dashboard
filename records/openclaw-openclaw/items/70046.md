## What This Changes

The PR adds `HH:MM`/`HH:MM:SS` parsing for cron `--at`, applies `--tz` when resolving those time-only values, and aligns cron help, docs, and tests with the new accepted format.

## Best Possible Solution

Rebase the branch onto current main, preserve the CLI-boundary time-only parser behavior, and land it with the focused parser/help/docs coverage intact.

## Reproduction Assessment

Yes, at source level. Current main has no time-only branch in `parseAt`, so `09:00` falls through the absolute timestamp and duration paths instead of being resolved with `--tz`.

## Solution Assessment

Yes. Handling time-only input in CLI `parseAt` is the best fit because the CLI has the `--tz` option and can continue storing one-shot schedules as canonical ISO timestamps; changing lower cron parsing would broaden runtime state semantics unnecessarily.
