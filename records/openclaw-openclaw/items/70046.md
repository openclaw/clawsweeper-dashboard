## Summary

Keep open. Current main still lacks the bare `HH:MM` / `HH:MM:SS` parser branch, and this PR is the active clean respin for the closed cron report, but it still needs contributor real-behavior proof before merge and one low-risk test hardening fix.

## What This Changes

The PR adds time-only `cron add --at` parsing, applies `--tz` to those values by resolving today in that timezone, rejects invalid time parts/timezones, and adds focused parser tests.

## Best Possible Solution

Land the focused parser/test change after making the date-boundary tests deterministic and adding real CLI behavior proof, with help text kept aligned through the related CLI wording PR.

## Reproduction Assessment

Yes by source inspection, with a small runtime check: current main falls through to `Date.parse('09:00')`, which is `NaN` here, so `resolveDirectSchedule` throws `Invalid --at`. I did not run the full CLI command because this review is read-only and avoids test/build artifacts.

## Solution Assessment

Yes for the parser approach, with small follow-ups. `parseAt` is the narrow layer that has `--tz` context and can reuse the existing offset-less zoned datetime helper; the PR should also freeze/capture time deterministically in tests and provide real behavior proof.
