## What This Changes

The PR adds `HH:MM[:SS]` handling for cron `--at`, resolves those values with `--tz`, and updates cron help, docs, and focused tests.

## Best Possible Solution

Land the focused parser/help/docs/test update if maintainers accept `HH:MM[:SS]` as an additive cron `--at` input; otherwise close with a product rationale and keep the documented ISO/duration behavior.

## Reproduction Assessment

Yes at source level: current main `parseAt` handles offset-less ISO, absolute timestamps, and durations, while `HH:MM` matches none of those paths and `resolveDirectSchedule` throws the invalid `--at` error. I did not execute the CLI because this was a read-only checkout review.

## Solution Assessment

Yes for the code shape: `parseAt` is the right layer because time-only input needs the CLI `--tz` option and today's date, and the PR reuses the existing zoned parser instead of changing lower absolute timestamp parsing. Product acceptance of the new public input grammar remains separate from code correctness.
