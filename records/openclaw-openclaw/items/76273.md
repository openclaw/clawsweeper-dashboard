## Summary

Keep open. Current main still has the reported Codex-vs-PI prompt split: PI resolves workspace bootstrap files including SOUL.md into Project Context, while the Codex app-server path builds developer instructions and optional context-engine projection without loading workspace bootstrap files; maintainers still need to choose the public contract.

## What This Changes

Clarify or align Codex harness behavior so users know whether SOUL.md should be injected like PI or durable persona should live in AGENTS.md for Codex.

## Best Possible Solution

Pick one public contract and make it executable: either project resolved workspace bootstrap into Codex with targeted tests/snapshots, or document Codex as AGENTS.md-first and update onboarding/templates so SOUL.md is not promised for that runtime.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to compare the Codex app-server path, which has no bootstrap resolver call sites, with the PI path that resolves contextFiles and passes them into buildEmbeddedSystemPrompt; a focused Codex unit test with a workspace SOUL.md should expose the gap.

## Solution Assessment

Unclear. The issue deliberately asks maintainers to choose between two plausible solutions, and the narrowest maintainable fix depends on whether Codex should preserve PI bootstrap compatibility or prefer Codex-native AGENTS.md behavior.
