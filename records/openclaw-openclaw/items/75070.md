## Summary

Keep open. Current main still routes bare `/new` and `/reset` through reset startup prompts that do not include the requested anti-fabrication or tool-verification guardrail, and the issue is paired with open PR #75127 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Session reset prompts should prevent the model from presenting invented calendar, email, JSON, or other contextual data as real user-provided input and require verification before acting on such data.

## Best Possible Solution

Review and land PR #75127, or an equivalent narrow prompt-and-test change, so every bare reset prompt variant explicitly forbids presenting invented data as real and requires tool verification before acting on calendar, email, task, memory, file, or JSON claims.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence path showing reset prompts lack the requested guardrail, but the reported fabricated first response depends on a specific model/provider and no transcript evidence was provided.

## Solution Assessment

Yes. A narrow reset prompt contract with direct prompt regression tests is the smallest maintainable repository-level fix supported by the evidence; broader runtime permission or routing changes are not justified by this report alone.
