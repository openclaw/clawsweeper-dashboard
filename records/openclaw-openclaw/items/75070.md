## Summary

Keep open. Current main still builds bare `/new` and `/reset` startup prompts without the requested anti-fabrication/tool-verification guardrail, and open PR #75127 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Prevent bare session resets from prompting models to fabricate calendar, email, JSON, or other structured data as if it came from the user.

## Best Possible Solution

Land PR #75127, or an equivalent narrow prompt and test change, so every bare reset prompt variant explicitly forbids presenting invented structured data as real and requires tool verification before acting on calendar, email, task, memory, or JSON claims.

## Reproduction Assessment

Unclear. There is a high-confidence static path showing bare `/new` and `/reset` send the current reset prompt, but the reported fabricated response depends on the Xiaomi model/provider and lacks transcript evidence.

## Solution Assessment

Yes. A narrow reset-prompt contract guardrail with direct prompt regression tests is the smallest maintainable repo-level fix supported by the evidence; broader runtime permission or routing changes are not justified by this report alone.
