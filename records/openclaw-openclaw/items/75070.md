## Summary

Keep open. Current main still builds model-running reset prompts without the requested anti-fabrication or tool-verification guardrail; bare text reset commands now acknowledge without invoking the model, but the shared reset prompt still reaches model-running reset paths, and PR #75127 remains open with closing syntax for this issue.

## What This Changes

Prevent session-reset startup prompts from leading models to present invented calendar, email, JSON, or similar data as user-provided context; the first reset response should greet/no-op or verify real data with tools before acting.

## Best Possible Solution

Review and land PR #75127, or an equivalent narrow prompt-and-test change, so every model-running reset prompt variant explicitly forbids presenting invented data as real and requires tool verification before acting on calendar, email, task, memory, file, or JSON claims.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence path showing the current model-running reset prompt lacks the requested guardrail, but reproducing the fabricated output itself depends on the Xiaomi model/provider behavior and no transcript or logs were provided.

## Solution Assessment

Yes. A narrow shared reset prompt contract with direct prompt regression tests is the smallest maintainable repository-level fix supported by the evidence; broader routing, permissions, or provider changes are not justified by this issue alone.
