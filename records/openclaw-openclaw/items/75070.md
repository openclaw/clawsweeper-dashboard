## Summary

Keep open. Current main still routes bare `/new` and `/reset` through reset prompt text that does not explicitly forbid invented user data or require tool verification before acting, and open PR #75127 already targets the same issue with closing syntax.

## What This Changes

The issue asks OpenClaw to prevent bare session resets from prompting models to fabricate structured calendar, email, or JSON-like data as if it were user-provided input.

## Best Possible Solution

Land PR #75127, or an equivalent narrow prompt and test change, so every bare reset prompt variant explicitly says reset startup contains no user-provided structured data, forbids presenting invented calendar/email/task/JSON content as real, and requires tool verification before acting on external facts.

## Reproduction Assessment

Unclear. There is a high-confidence static reproduction path showing current main builds and sends the bare reset prompt without the requested guardrail, but the reported fabricated model output is provider/model-dependent and has no transcript evidence attached.

## Solution Assessment

Yes, the linked direction is the best repo-level fix. A reset-prompt contract update with direct prompt regression tests is narrower and more maintainable than changing runtime permissions or routing based on this report alone.
