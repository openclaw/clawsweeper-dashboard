## Summary

Keep open. Current main still sends bare session reset startup prompts that do not explicitly forbid fabricated user data or require tool verification before acting, and an open unmerged PR already targets this issue with closing syntax.

## What This Changes

The issue requests a reset-startup guardrail so bare `/new` and `/reset` prompts do not let models present invented calendar, email, JSON, or other contextual data as real.

## Best Possible Solution

Review and land PR #75127, or an equivalent narrow prompt-and-test change, so every bare reset prompt variant explicitly forbids presenting invented data as real and requires verification before acting on structured claims.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence path showing the reset prompts lack the requested guardrail and are used by both auto-reply and Gateway reset flows, but the specific fabricated JSON response depends on the reported model/provider and no transcript was provided.

## Solution Assessment

Yes. A narrow reset prompt contract with direct prompt regression tests is the smallest maintainable repository-level fix supported by the evidence; broader runtime permission or routing changes are not justified by this report alone.
