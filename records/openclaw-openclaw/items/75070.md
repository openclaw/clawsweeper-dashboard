## Summary

Keep open. Current main still routes bare `/new` and `/reset` through reset startup prompts without an explicit anti-fabrication or tool-verification guardrail, and open PR #75127 references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to prevent session reset startup prompts from letting models present invented calendar, email, or JSON-style data as real user context.

## Best Possible Solution

Land PR #75127, or an equivalent narrow prompt-and-test change, so all bare reset prompt variants forbid presenting invented data as real and require verification before acting on external data claims.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence path showing current main lacks the guardrail on both reset dispatch surfaces, but the reported fabricated response depends on a specific provider/model and no live transcript was provided.

## Solution Assessment

Yes. A focused reset prompt contract with direct prompt regression tests is the narrowest maintainable repository-level fix supported by the evidence; broader runtime or tool-permission changes are not justified by this report alone.
