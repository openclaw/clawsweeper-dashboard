## Summary

Keep open. Current main still sends the bare reset startup prompt without an explicit anti-fabrication or tool-verification guardrail, and open PR #75127 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests a session-reset guardrail so bare `/new` and `/reset` prompts do not lead models to present invented calendar, email, JSON, memory, or other structured data as user-provided facts.

## Best Possible Solution

Review and land PR #75127, or an equivalent narrow prompt and test change, so every bare reset prompt variant explicitly forbids presenting invented data as real and requires tool verification before acting on calendar, email, task, memory, file, or JSON claims.

## Reproduction Assessment

Unclear. There is a high-confidence static path showing bare `/new` and `/reset` send the current reset prompt, but the reported fabricated first response depends on a specific model/provider and lacks transcript evidence.

## Solution Assessment

Yes. A narrow reset-prompt contract guardrail with direct prompt regression tests is the smallest maintainable repo-level fix supported by the evidence; broader runtime permission or routing changes are not justified by this report alone.
