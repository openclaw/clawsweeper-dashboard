## Summary

Keep open. Current main still generates macOS service PATHs from user/version-manager directories and still merges existing plist PATH entries during install planning, and open PR #75246 targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request to make macOS Gateway LaunchAgent installs write a canonical minimal PATH instead of capturing or preserving interactive-shell Node version-manager and package-manager PATH segments.

## Best Possible Solution

Review and land PR #75246, or an equivalent macOS-only fix, that makes LaunchAgent PATH generation canonical, skips old plist PATH preservation on macOS, updates focused service-env/install-plan regression tests and user-facing docs, and leaves Linux service PATH compatibility as a separate policy surface.

No live launchd install was run, so the timeout/version-mismatch symptoms remain report-backed rather than independently reproduced.

## Solution Assessment

Yes, the requested macOS-only canonical PATH direction is the narrowest maintainable fix for this report. Any opt-in support for version-manager service PATHs should be handled separately from the deterministic launchd default.
