## Summary

Keep open. Current main has meaningful partial observability plumbing, but the public plugin-hook contract still lacks the full requested agent identity, first-class agent span aliases, outbound message_sending correlation, and requester/child subagent run hierarchy.

## What This Changes

The issue requests richer plugin hook trace context, including runId, messageId, senderId, span parentage, and subagent parent/child run identifiers for observability plugins.

## Best Possible Solution

Define one backward-compatible plugin-hook correlation contract, then land focused slices for agent inbound identity, outbound delivery correlation, subagent requester/child run IDs, and trace/span aliases with SDK docs and contract tests.

## Reproduction Assessment

Yes for the missing-contract claim: source inspection on current main shows the remaining gaps in public hook types and call sites. I did not run a live Feishu or Opik trace because the unresolved behavior is visible at the SDK/type boundary.

## Solution Assessment

Unclear for immediate implementation: the requested direction is valid, but the maintainable solution needs agreement on field names, optional versus required compatibility, and how DiagnosticTraceContext maps to hook-level aliases.
