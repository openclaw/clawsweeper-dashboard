## Summary

Keep open: current main and v2026.6.8 only partially address the plugin-hook observability request. runId and trace plumbing now exist for some agent and message paths, but outbound message_sending still lacks run/message identity, agent hook context still lacks message/span aliases, and subagent hooks still do not expose requesterRunId for run-based trace hierarchy.

## What This Changes

The issue requests a complete plugin-hook trace correlation contract exposing run, message, sender, span, and subagent parent/child identifiers across hook families.

## Best Possible Solution

Define one backward-compatible plugin-hook correlation contract, then land focused slices for agent inbound identity, outbound delivery correlation, subagent child/requester run IDs, and trace/span aliases with SDK baseline, docs, and tests aligned.

## Reproduction Assessment

Yes, at source level. Current hook types and outbound/subagent call sites still omit requested message, outbound run, span alias, and requester-run identifiers even though runId/trace plumbing exists in some paths.

## Solution Assessment

No for a one-shot mechanical patch. The direction is valid, but the best fix is a maintainer-approved public SDK contract across agent, message, diagnostics, and subagent hooks before implementation.
