## Summary

Keep open. Current main has useful partial trace plumbing, including runId backfill and diagnostic trace context, but the public plugin-hook contract still lacks the full agent identity, message_sending event correlation, and requester/child subagent run hierarchy requested for reliable observability traces.

## What This Changes

The issue requests consistent message, sender, run, and span correlation fields across plugin hooks so observability plugins can build accurate trace trees.

## Best Possible Solution

Define one backward-compatible plugin-hook correlation contract, then land focused slices for agent inbound identity, outbound delivery correlation, subagent requester/child run IDs, and trace/span aliases with SDK docs and contract tests.

## Reproduction Assessment

Yes for the missing-contract claim: source inspection on current main shows the remaining gaps in public hook types and call sites. A live Feishu or Opik run was not needed to prove the SDK/type boundary is still incomplete.

## Solution Assessment

Unclear for immediate implementation: the requested direction is valid, but maintainers still need to approve field names, optional versus required compatibility, and how DiagnosticTraceContext maps to first-class hook aliases.
