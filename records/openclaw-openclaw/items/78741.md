## Summary

Keep open. The PR targets the right Control UI storage boundary, but the current sanitizer still misses colon-rich routed group session keys that the existing session grammar supports, and the supplied proof is unit/jsdom output rather than a real Control UI restore run.

## What This Changes

The PR adds Control UI settings restore sanitization for persisted routed direct/group session keys and extends storage regression tests.

## Best Possible Solution

Keep the storage-layer reset, but make the routed-key detector follow the shared session-key grammar for direct/group peer sessions without an arbitrary length cap, then prove a real Control UI reload from seeded localStorage falls back to main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed `openclaw.control.settings.v1:<gateway>` / `sessionsByGateway` with a routed key, call `loadSettings()`, and current main restores it unfiltered; on the PR branch, colon-rich group keys still bypass the sanitizer because of the `parts.length <= 7` cap.

## Solution Assessment

No. The storage restore boundary is the right place to fix this, but the current implementation should cover the shared routed direct/group key grammar and add a regression for Feishu/topic/sender or equivalent colon-rich group keys before merge.
