## Summary

Keep open: the PR targets a real Control UI restore bug, but the proposed detector misses current routed direct-session key shapes and the external real-behavior proof is only test/type/lint output, not a real UI reload path.

## What This Changes

The PR adds Control UI settings-load sanitization for persisted routed direct/group session keys and storage tests expecting restored selections to fall back to main.

## Best Possible Solution

Use storage-layer restore sanitization backed by the shared routed-session key contract, preserve explicit URL/session-picker selections, and keep the linked issue open until a complete fix lands.

## Reproduction Assessment

Yes, source-reproducible: seeding scoped Control UI local settings with a routed sessionsByGateway entry flows through loadSettings() and then survives applySessionDefaults() on current main. I did not run tests because this was a read-only review.

## Solution Assessment

No. Resetting persisted routed sessions during storage restore is the right layer, but this implementation should cover all current routed direct/group key shapes or use a shared parser instead of the narrower regex.
