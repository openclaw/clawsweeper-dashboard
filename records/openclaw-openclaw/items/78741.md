## Summary

Keep open: the PR is aimed at a real Control UI restore bug, but the latest sanitizer still misses routed group keys whose peer id contains colon-qualified topic/sender data, and the external proof is still unit/jsdom-only.

## What This Changes

The PR changes Control UI settings restore so persisted routed direct/group session selections fall back to `main`, with storage regression tests for routed key shapes.

## Best Possible Solution

Keep the storage-layer sanitizer, but make it follow the routed session-key contract: reset all generated direct/group peer keys, including colon-qualified group peer IDs, while preserving subagent, cron, custom, main, and other non-routed selections.

## Reproduction Assessment

Yes for source-level reproduction: current `main` returns scoped `sessionsByGateway` selections unchanged from `resolveScopedSessionSelection()`, and `OpenClawApp` initializes settings from `loadSettings()`. I did not run tests because this review is read-only.

## Solution Assessment

No. The storage layer is the right boundary, but the detector should cover colon-qualified group peer IDs from the current session-key contract before this is the best fix.
