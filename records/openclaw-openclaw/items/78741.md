## What This Changes

The PR adds Control UI storage restore logic that resets legacy top-level historical routed session keys to `main` while preserving current scoped `sessionsByGateway` selections, with storage regression coverage.

## Best Possible Solution

Land a narrow legacy-restore sanitizer only after maintainers accept scoped-session preservation and the PR has current-head browser reload proof for both reset and preserve paths.

## Reproduction Assessment

Yes from source inspection: current main loads persisted session keys through `loadSettings()` into the app's active `sessionKey`, and scoped or legacy routed values can therefore become the restored UI session. I did not run a browser repro in this read-only review.

## Solution Assessment

Unclear until maintainers accept the policy split: legacy-only sanitization is a narrow, compatible fix if scoped `sessionsByGateway` is treated as explicit user selection. If the intended product behavior is to reset all historical routed startup restores, the patch is too narrow.
