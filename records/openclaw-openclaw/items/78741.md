## Summary

Keep open: the PR targets the right Control UI settings-load boundary and the supplied connected-browser proof is useful, but the detector still misses backward-compatible `:dm:` direct session keys, so older persisted routed selections can still restore stale sessions.

## What This Changes

The PR adds Control UI settings-load sanitization that resets persisted routed direct/group session selections to `main` and adds storage regression tests.

## Best Possible Solution

Keep the storage-layer restore fix, but make the historical routed-session detector cover the same direct markers as the session-key contract and then explicitly accept the reset-to-main upgrade behavior.

## Reproduction Assessment

Yes, at source level: seeding Control UI localStorage with a scoped routed session key reaches `resolveScopedSessionSelection()` on current main, which returns the persisted key unchanged. I did not run a browser repro locally, but the PR discussion includes connected-browser after-fix proof.

## Solution Assessment

No, not yet: the storage-load boundary is the right place to fix this, but the detector should cover the existing `dm` direct-session compatibility contract before merge. The safer path is to add `dm` detection and regression tests while preserving subagent, cron, and custom session keys.
