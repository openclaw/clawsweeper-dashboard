## Summary

Keep this PR open: it is still trying to fix a real Control UI session-restore bug, but the latest patch only sanitizes legacy top-level persisted routed keys while intentionally preserving routed keys from `sessionsByGateway`, which the linked report also names as a problematic restore source. Current-head real browser reload proof is also still insufficient for the latest scoped-vs-legacy behavior.

## What This Changes

The PR adds a Control UI settings-restore predicate that falls legacy top-level routed session keys back to `main`, while adding storage regression tests that preserve scoped `sessionsByGateway` routed selections.

## Best Possible Solution

Sanitize persisted routed session selections consistently during storage restore, keep explicit `?session=` URL selection as the opt-in route, and land only after current-head browser reload proof demonstrates the accepted scoped-vs-legacy policy.

## Reproduction Assessment

Yes, source-level: current main restores persisted scoped and legacy session keys directly from localStorage, so pre-seeding a routed key reproduces the restore path without needing backend routing. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not yet: sanitizing only legacy top-level settings is too narrow for the linked report because current persisted selections live in `sessionsByGateway`. The narrower maintainable fix is to sanitize stored restore values consistently while leaving explicit URL session selection as the deliberate override.
