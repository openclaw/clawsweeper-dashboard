## Summary

Keep open. The patch appears correct for the linked Control UI restore bug and the after-fix proof is sufficient, but it intentionally changes browser-local persisted session restore semantics and needs maintainer acceptance before merge.

## What This Changes

The PR sanitizes legacy top-level routed Control UI session selections to `main` on settings restore while preserving scoped `sessionsByGateway` selections and adding storage regression coverage.

## Best Possible Solution

Merge the scoped/legacy split if maintainers accept resetting only legacy top-level routed selections to `main`; otherwise adjust the restore contract before landing.

## Reproduction Assessment

Yes from source inspection: current main returns legacy top-level routed `sessionKey` values unchanged during `loadSettings()`, and the PR adds regression cases showing those values now fall back to `main`. I did not run tests because this was a read-only review.

## Solution Assessment

Yes, with a maintainer decision: applying the sanitizer only to legacy top-level restore data is the narrowest implementation I found that avoids breaking scoped `sessionsByGateway` selections. The unresolved question is whether resetting those legacy routed keys is the desired compatibility boundary.
