## Summary

Keep open. Current main and v2026.6.6 have useful groundwork for classified cron errors, bounded transient retries, and failure alerts, but recurring cron still lacks the requested reason-specific circuit breaker, Retry-After-aware guard state, and job-level pause/disable recovery contract.

## What This Changes

Request a reason-aware cron guard layer that uses quota/auth/rate-limit/error classification to back off, pause, alert, and expose guard state for automated cron jobs.

## Best Possible Solution

Define one recurring-cron guard policy that reuses existing failover classification, retry hints, provider cooldowns, and failure alerts, then add the minimal persisted job guard state plus protocol/UI/docs/tests needed for safe pause, recovery, and operator visibility.

## Reproduction Assessment

Yes, for the policy gap: current source and tests show classified errors and bounded transient retries followed by generic recurring scheduling, with no reason-specific recurring circuit breaker or guard state. I did not run a live provider retry-storm reproduction.

## Solution Assessment

Partly. Reusing `FailoverReason`, retry hints, auth-profile cooldowns, and failure alerts is the right foundation, but maintainers still need to decide thresholds, recovery semantics, config exposure, and provider-versus-job ownership.

## Risks / Open Questions
