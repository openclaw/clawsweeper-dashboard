## Summary

Keep open. Current main has useful groundwork for this request, including `lastErrorReason`, one-shot retry handling, failure alerts, and local-provider preflight skips, but recurring cron still lacks the requested reason-aware guard layer, circuit breaker, Retry-After-aware scheduling, and persisted cooldown/disabled guard state.

## What This Changes

The issue requests reason-aware cron guardrails so billing/auth/format failures can circuit-break, rate limits can honor Retry-After/backoff without disabling jobs, and job state can expose guard reason/cooldown data.

## Best Possible Solution

Add a coordinated cron guard layer that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with safe configurable defaults and one alert, respects Retry-After for rate limits, and keeps transient timeout/network paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is a feature request. The missing behavior is source-reproducible: current main records `lastErrorReason`, then schedules recurring errors through the same fixed consecutive-error backoff path.

## Solution Assessment

Yes as a direction, but the exact defaults and config shape need maintainer decision. The maintainable solution should span cron policy, state, Gateway protocol, UI/macOS surfaces, docs, and tests rather than adding a timer-only special case.
