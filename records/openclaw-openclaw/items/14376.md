## Summary

Keep open. Current main and v2026.6.8 include classified cron error observability, one-shot transient retry, recurring transient retry, and failure alerts, but recurring cron still lacks the requested reason-specific circuit breaker, Retry-After-aware guard state, and job-level pause/disable recovery contract.

## What This Changes

The issue requests reason-aware cron guardrails that classify billing, auth, rate-limit, timeout, and format failures and apply circuit breaking, backoff, alerting, and persisted state for scheduled jobs.

## Best Possible Solution

Define one recurring-cron guard contract that reuses failover classification, retry hints, provider cooldowns, and failure alerts, then add the minimal persisted job state plus protocol, UI, docs, and tests for safe pause, recovery, and operator visibility.

## Reproduction Assessment

Yes, at source level: current main shows classified errors and transient retry handling followed by generic recurring scheduling, with no reason-specific circuit breaker or guard state. I did not run a live provider retry-storm reproduction.

## Solution Assessment

Partly. Reusing FailoverReason, retry hints, auth-profile cooldowns, and failure alerts is the right foundation, but maintainers still need to choose thresholds, recovery semantics, config exposure, and provider-versus-job ownership.
