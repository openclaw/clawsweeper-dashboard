## Summary

Keep open. Current main has partial error observability and generic retry/backoff, but recurring cron still does not apply reason-aware guardrails, circuit breaking, Retry-After scheduling, or job-level cooldown/disabled state; the issue discussion also contains real retry-storm reports and a recent maintainer review that reached the same keep-open conclusion.

## What This Changes

Request a reason-aware cron guard layer that classifies quota/auth/rate-limit/timeout/format failures and applies safe backoff, circuit-breaker, alerting, and observability behavior for background jobs.

## Best Possible Solution

Add a coordinated cron guard layer that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with safe configurable defaults and one alert, respects Retry-After for rate limits, and keeps transient timeout/network paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is a feature request, but the missing behavior is source-reproducible. Current main classifies lastErrorReason and then schedules enabled recurring errors through the same fixed consecutive-error backoff branch.

## Solution Assessment

Yes, the requested direction is the right maintainable shape, but it should be implemented as a coordinated cron policy/state/protocol/UI/docs change rather than a timer-only special case.
