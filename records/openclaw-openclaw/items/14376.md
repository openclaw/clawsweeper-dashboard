## Summary

Keep open: current main and v2026.6.5 have `lastErrorReason` observability and bounded transient retry, but recurring cron still lacks job-level reason-aware circuit breakers, Retry-After scheduling, and guard state. The open provider circuit-breaker PR only covers part of provider quota handling, not the cron guard layer requested here.

## What This Changes

The issue requests reason-aware cron guardrails that classify quota/auth/rate-limit/timeout/format failures, apply reason-specific backoff or circuit breaking, and expose job guard state.

## Best Possible Solution

Implement a core cron guard layer that classifies terminal run failures once, persists/display guard state, coordinates with auth-profile/provider cooldowns, respects Retry-After where available, and covers recurring cron behavior with docs and regression tests.

## Reproduction Assessment

Yes from source inspection: a recurring job that returns a classified rate-limit, billing, auth, or format error still flows through generic retry/backoff scheduling, with no job-level guard window or circuit-break state. I did not run a live provider exhaustion scenario.

## Solution Assessment

No, this is not solved by the current implementation; the existing `lastErrorReason` field is useful groundwork but not the full guardrail. The best fix is a cross-surface cron policy/state change, not a timer-only tweak.
