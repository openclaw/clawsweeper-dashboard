## Summary

Keep open. Current main and v2026.6.5 still only partially address the request: cron records lastErrorReason and has transient retry/backoff behavior, but recurring jobs still lack billing/auth/format circuit breaking, Retry-After-aware scheduling, and job-level guard state.

## What This Changes

The issue requests reason-aware cron guardrails so quota, auth, rate-limit, timeout/network, and format failures produce different backoff, disable, alerting, and state behavior.

## Best Possible Solution

Define an additive recurring cron guard contract, then implement reason-specific scheduling, persisted guard state, user-visible status, docs, and tests together so quota/auth/format stop safely while transient and rate-limit paths remain bounded and recoverable.

## Reproduction Assessment

Yes, at source level. Current main records lastErrorReason and has transient retry branches, but recurring non-transient failures still fall through to generic fixed backoff without circuit-breaker state.

## Solution Assessment

Yes as a product direction, but not as a narrow timer-only fix. The maintainable solution needs explicit guard policy, config/schema additions, persisted state, UI/protocol/docs exposure, regression tests, and provider/auth cooldown coordination.
