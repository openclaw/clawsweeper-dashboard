## Summary

Keep open. Current main still has partial cron error observability and generic recurring backoff, while the requested reason-aware cron guard layer, circuit breaker, Retry-After-aware recurring scheduling, and persisted guard state remain unimplemented.

## What This Changes

Request reason-aware cron guardrails that classify cron failures and apply quota/auth/format circuit breaking, rate-limit backoff, retry-storm prevention, and job-visible error state.

## Best Possible Solution

Add a coordinated cron guard layer that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with safe configurable defaults and one alert, respects Retry-After for rate limits, and keeps transient paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is an enhancement request. The missing behavior is source-reproducible: current main records lastErrorReason, then schedules recurring errors through the same fixed consecutive-error backoff path.

## Solution Assessment

Yes as a product direction, but not as a narrow automated timer-only fix. The maintainable solution needs explicit guard policy, config/schema additions, persisted state, UI/protocol/docs exposure, and regression tests.
