## Summary

Keep open. Current main has partial cron error observability and one-shot transient retry handling, but recurring cron still uses generic consecutive-error backoff and lacks the requested reason-specific circuit breaker, Retry-After scheduling, persisted guard state, and client-visible cooldown/disabled semantics.

## What This Changes

Request reason-aware cron guardrails that classify quota/auth/rate-limit/timeout/format failures, apply safer backoff or circuit-break behavior, persist guard state, alert once, and expose the result to users and clients.

## Best Possible Solution

Keep this open until recurring cron has an explicit reason-aware guard policy with persisted state, additive protocol/client exposure, docs, and regression tests coordinated with auth-profile/provider cooldown behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source inspection against origin/main confirms the current recurring cron path is still generic backoff, and the supplied comments provide concrete real-world retry-storm examples motivating the feature.

## Solution Assessment

Unclear until maintainers choose the policy shape. The requested direction is credible, but the narrow maintainable solution needs a designed contract across timer policy, persisted state, Gateway schema, UI/macOS, docs, and provider cooldown semantics.
