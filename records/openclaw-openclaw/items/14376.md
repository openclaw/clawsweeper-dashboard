## Summary

Keep open. Current main has partial cron error observability, generic recurring backoff, one-shot transient retry, and failure alerts, but it still lacks the requested reason-aware recurring cron guard layer with billing/auth/format circuit breaking, Retry-After-aware scheduling, persisted guard state, and client-visible cooldown/disabled status.

## What This Changes

Request reason-aware cron guardrails that classify cron failures and apply quota/auth/rate-limit-aware backoff, alerts, and circuit breaking for background jobs.

## Best Possible Solution

Design and implement a coordinated cron guard contract that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with safe configurable defaults and one alert, respects Retry-After for rate limits, and keeps transient paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is an enhancement request. The missing behavior is source-reproducible: current main records lastErrorReason, then routes recurring errors through the same fixed consecutive-error backoff path with no reason-specific guard state.

## Solution Assessment

Yes as a product direction, but not as a narrow automated timer-only fix. The maintainable solution needs explicit guard policy, config/schema additions, persisted state, UI/protocol/docs exposure, and regression tests coordinated with provider/auth cooldown behavior.
