## Summary

Keep open. Current main and the latest release include classified cron error observability, transient recurring retry, and failure alerts, but they still do not implement the requested reason-specific cron guard layer with job-level circuit breaker, Retry-After-aware guard state, and pause/disable recovery semantics.

## What This Changes

The issue requests reason-aware cron guardrails so recurring jobs classify quota, auth, rate-limit, timeout, and format failures and apply safer backoff, notification, or circuit-break behavior.

## Best Possible Solution

Define one recurring-cron guard contract that reuses failover classification, retry hints, provider cooldowns, and failure alerts, then add the minimal persisted job state plus protocol, UI, docs, and tests for safe pause, recovery, and operator visibility.

## Reproduction Assessment

Yes, at source level: current main shows classified errors and transient retry handling followed by generic recurring scheduling, with no reason-specific circuit breaker or job-level guard state. I did not run a live provider retry-storm reproduction.

## Solution Assessment

Partly. Reusing FailoverReason, retry hints, auth-profile cooldowns, and failure alerts is the right foundation, but maintainers still need to choose thresholds, recovery semantics, config exposure, and provider-versus-job ownership.
