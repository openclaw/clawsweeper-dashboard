## Summary

Keep open: current main and the latest release still provide classified cron error observability and transient retry behavior, but recurring cron lacks the requested reason-specific circuit breaker, Retry-After-aware guard state, and job-level pause or disable recovery contract.

## What This Changes

Request reason-aware cron guardrails that classify quota, auth, rate-limit, timeout, format, and unknown failures to drive backoff, circuit-break, observability, and optional configuration.

## Best Possible Solution

Define one recurring-cron guard policy that reuses failover classification, retry hints, provider cooldowns, and failure alerts, then add minimal persisted job guard state plus protocol, UI/client, docs, and regression coverage for safe pause, recovery, and operator visibility.

## Reproduction Assessment

Yes at source level: current cron classifies errors but recurring scheduling still uses generic consecutive-error backoff and has no job-level guard state. I did not run a live provider retry-storm reproduction.

## Solution Assessment

Partly. Reusing FailoverReason, retry hints, auth-profile cooldowns, and failure alerts is the right foundation, but maintainers still need to choose thresholds, recovery semantics, config exposure, and provider-versus-job ownership.
