## Summary

Keep open. Current main and the latest release have classified cron error observability, transient retry behavior, and failure alerts, but recurring cron still lacks the requested reason-specific circuit breaker, Retry-After-aware guard state, and job-level pause or disable recovery contract.

## What This Changes

Request a reason-aware cron guard layer that classifies quota, auth, rate-limit, timeout, format, and unknown failures, then applies safer backoff, pause, alert, and state-exposure policy for scheduled jobs.

## Best Possible Solution

Define one recurring-cron guard policy that reuses failover classification, retry hints, provider cooldowns, and failure alerts, then add the minimal persisted job guard state plus protocol, UI, docs, and tests needed for safe pause, recovery, and operator visibility.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 show classified errors and transient retry handling followed by generic recurring scheduling, with no reason-specific recurring circuit breaker or guard state. I did not run a live provider retry-storm reproduction.

## Solution Assessment

Partly. Reusing FailoverReason, retry hints, auth-profile cooldowns, and failure alerts is the right foundation, but maintainers still need to decide thresholds, recovery semantics, config exposure, and provider-versus-job ownership.
