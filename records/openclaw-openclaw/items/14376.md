## Summary

Keep open. Current main records cron error reasons and has one-shot retry/failure-alert pieces, but recurring cron still applies generic consecutive-error backoff without reason-specific circuit breaking, Retry-After scheduling, or persisted guard state.

## What This Changes

Request a reason-aware cron guard layer that classifies cron failures and applies quota/auth/format circuit breakers, rate-limit backoff, guard state, alerts, and configuration.

## Best Possible Solution

Design and implement a coordinated cron guard policy across timer logic, persisted state/config, Gateway schema, client surfaces, docs, and regression tests, aligned with provider/auth cooldown semantics.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is an enhancement request. The missing behavior is source-reproducible: current main records lastErrorReason, then schedules recurring errors through the same fixed consecutive-error backoff path.

## Solution Assessment

Yes as a product direction, but not as a narrow automated timer-only fix. The maintainable solution needs explicit guard policy, config/schema additions, persisted state, client-visible state, docs, and regression tests.
