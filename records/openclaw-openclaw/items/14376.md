## Summary

Keep open. Current main has useful groundwork (`lastErrorReason`, one-shot retry, failure alerts), but recurring cron still lacks the requested reason-aware guard layer, circuit breaker, Retry-After-aware recurring scheduling, and persisted cooldown or disabled guard state.

## What This Changes

The issue requests reason-aware cron guardrails that classify quota/auth/rate-limit/timeout/format failures and choose backoff, alert, cooldown, or circuit-break behavior accordingly.

## Best Possible Solution

Add a coordinated cron guard layer that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with safe configurable defaults and one alert, respects Retry-After for rate limits, and keeps transient timeout/network paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is a feature request, but the missing behavior is source-reproducible: current main records `lastErrorReason` and then schedules recurring errors through the same fixed consecutive-error backoff branch.

## Solution Assessment

Yes as a direction, but the exact defaults and config shape need maintainer decision. The maintainable solution should span cron policy, state, Gateway protocol, UI/macOS surfaces, docs, and tests rather than adding a timer-only special case.
