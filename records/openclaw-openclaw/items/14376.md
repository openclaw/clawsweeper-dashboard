## Label Justifications

- `P2`: The missing guardrails can cause costly retry storms in automated workloads, but the scheduler and core runtime remain operational.
- `impact:auth-provider`: The requested behavior directly concerns authentication failures, quota exhaustion, rate limits, provider failover, and cooldown handling.

## Summary

Current main partially addresses the request with classified error state, bounded transient retries for one-shot and recurring jobs, and failure alerts. However, recurring jobs still use generic scheduling after retry exhaustion and lack reason-specific circuit breaking or persisted recovery state, so the central safety feature remains necessary and requires maintainer product judgment.

## What This Changes

Add reason-aware recurring cron guardrails that distinguish quota, authentication, rate-limit, timeout, format, and unknown failures for backoff, alerting, and circuit-break behavior.

## Best Possible Solution

Reuse the existing failover classification, transient retry, provider cooldown, and failure-alert machinery; add only the minimal job-level circuit state needed for repeated permanent failures, with explicit pause and recovery semantics plus complete protocol, UI/CLI, docs, migration, and regression coverage.

## Reproduction Assessment

Yes for the policy gap: current source and regression tests show bounded transient retries followed by generic recurring backoff, with no reason-specific recurring circuit breaker. A live multi-provider retry storm was not run, but the missing behavior is directly source-reproducible.

## Solution Assessment

Partly. Reusing failover reasons and current retry machinery is the right foundation, but maintainers must decide thresholds, recovery behavior, provider-versus-job ownership, and whether additional configuration is warranted.
