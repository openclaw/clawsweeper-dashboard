## Summary

Keep open. Current main and the latest release still have partial cron error observability plus generic recurring backoff, but the requested recurring reason-aware guard layer, Retry-After-aware scheduling, job-level circuit-breaker state, and billing/auth/format auto-pause policy remain missing.

## What This Changes

The issue requests reason-aware cron guardrails for quota, auth, rate-limit, timeout, and format failures, including backoff, circuit-breaking, persisted state, and visibility.

## Best Possible Solution

Define an additive recurring cron guard contract, then implement reason-specific scheduling, persisted guard state, client-visible status, docs, and tests together so quota/auth/format stop safely while transient and rate-limit paths remain bounded and recoverable.

## Reproduction Assessment

Yes, at source level, though not as a live bug reproduction because this is a feature request. Current main records lastErrorReason, then schedules recurring failures through the same fixed consecutive-error backoff path without reason-specific guard state.

## Solution Assessment

Yes as a product direction, but not as a narrow timer-only fix. The maintainable solution needs explicit guard policy, config/schema additions, persisted state, UI/protocol/docs exposure, regression tests, and provider/auth cooldown coordination.
