## Summary

Keep open. Current main and the latest release still have partial cron error observability plus fixed consecutive-error backoff, but not the requested reason-aware recurring guard policy, circuit breaker, Retry-After scheduling, or job-level guard state.

## What This Changes

The issue requests reason-aware cron guardrails that classify quota, auth, rate-limit, timeout/network, and format failures to choose backoff, circuit breaking, alerts, and visible job state.

## Best Possible Solution

Add a coordinated cron guard layer that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with safe configurable defaults and one alert, respects Retry-After for rate limits, and keeps transient paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is a feature request. The missing behavior is source-reproducible: current main records lastErrorReason, then schedules recurring failures through the same fixed consecutive-error backoff path without reason-specific guard state.

## Solution Assessment

Yes as a product direction, but not as a narrow timer-only fix. The maintainable solution needs explicit guard policy, config/schema additions, persisted state, UI/protocol/docs exposure, regression tests, and provider/auth cooldown coordination.
