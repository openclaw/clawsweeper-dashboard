## Summary

Keep open. Current main still records cron error reasons and applies generic consecutive-error backoff, but it does not implement the requested recurring cron guard layer, Retry-After scheduling, job-level circuit breaker state, or billing/auth/format auto-pause policy.

## What This Changes

Request reason-aware cron guardrails that classify quota, auth, rate-limit, timeout, format, and unknown failures for backoff, circuit breaking, and observability.

## Best Possible Solution

Ship a coordinated cron guard layer that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with configurable safe defaults and one alert, respects Retry-After for rate limits, and keeps transient paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is a feature request, but the missing behavior is source-reproducible. Current main records lastErrorReason, then schedules recurring failures through the same fixed consecutive-error backoff path without reason-specific guard state.

## Solution Assessment

Yes as a product direction, but not as a narrow timer-only fix. The maintainable solution needs explicit guard policy, config/schema additions, persisted state, UI/protocol/docs exposure, regression tests, and provider/auth cooldown coordination.
