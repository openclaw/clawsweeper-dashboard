## Summary

Keep open. Current main has partial cron error observability and one-shot transient retry behavior, but recurring cron still uses a generic consecutive-error backoff and lacks the requested reason-specific guard policy, Retry-After scheduling, circuit breaker, and persisted client-visible cooldown/disabled state.

## What This Changes

Request reason-aware cron guardrails that classify job failures and apply quota/auth/format circuit breakers, rate-limit-aware backoff, and last-error observability for background jobs.

## Best Possible Solution

Add an explicit recurring-cron guard policy with reason classification, persisted guard state, additive protocol/client exposure, docs, and regression tests coordinated with provider/auth cooldown behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source inspection against current main confirms recurring cron still uses generic consecutive-error backoff, while the supplied comments provide concrete real-world retry-storm examples motivating the feature.

## Solution Assessment

Unclear until maintainers choose the policy shape. The requested direction is credible, but the maintainable solution needs a designed contract across scheduler policy, persisted state, Gateway schema, UI/macOS, docs, and provider cooldown semantics.
