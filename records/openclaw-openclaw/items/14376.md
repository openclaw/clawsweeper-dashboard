## Summary

Keep open. Current main has the observability groundwork, but recurring cron still applies a fixed consecutive-error backoff instead of the requested billing/auth/format circuit breaker, Retry-After-aware rate-limit handling, or job-level guard state.

## What This Changes

Request reason-aware cron guardrails that classify quota, auth, rate-limit, timeout, format, and unknown failures, then apply safer backoff, circuit-break, alerting, and state visibility for recurring jobs.

## Best Possible Solution

Add an explicit recurring cron guard layer that classifies terminal run errors once, persists/display guard state, respects Retry-After when available, auto-pauses or circuit-breaks billing/auth/format failures with configurable safe defaults and one alert, and keeps timeout/network/rate-limit jobs enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Yes, for source-level reproduction: a recurring job result with `status: "error"` currently flows through `applyJobResult` and receives the same `errorBackoffMs(consecutiveErrors)` scheduling path regardless of classified reason. The supplied bradz86 and kitcommerce reports add real-world retry-storm evidence, but a live provider storm is not required to verify the missing policy branch.

## Solution Assessment

No, the current partial solution is not enough: `lastErrorReason` improves observability but does not drive mitigation. The maintainable fix should be additive across cron state/config/protocol/client surfaces and tested as recurring cron policy, not only one-shot retry or raw error classification.
