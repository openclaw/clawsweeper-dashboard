## Summary

Keep open: current main and v2026.6.5 classify cron failures and retry transient recurring errors, but permanent billing, authentication, and format failures still leave recurring jobs enabled under generic backoff without persisted circuit-breaker state.

## What This Changes

Add reason-aware recurring cron guardrails with failure-specific retry, circuit breaking, persisted state, observability, and configurable safe defaults.

## Best Possible Solution

Build on the existing transient retry classifier with one cron-owned persisted guard state for permanent failures, coordinated with provider/profile cooldowns and exposed consistently through protocol, UI, alerts, documentation, and regression coverage.

## Reproduction Assessment

Yes, at source level: a recurring billing, authentication, or format failure is classified as permanent but remains enabled and returns to generic scheduling and backoff. This review did not run a live provider retry storm.

## Solution Assessment

No, the timer-only implementation sketch is not sufficient by itself. The narrowest maintainable solution is a canonical guard-state decision coordinated with existing provider cooldown ownership and reflected across persisted state and client contracts.
