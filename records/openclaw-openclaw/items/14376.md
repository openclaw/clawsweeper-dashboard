## Summary

Keep open. Current main and v2026.6.10 have classified cron error observability plus generic retry/backoff, but still lack the requested reason-specific recurring cron guard layer, persisted job-level cooldown/disable state, Retry-After-aware scheduling, and full policy/UI/docs coverage.

## What This Changes

Request a reason-aware cron guard layer that classifies quota, auth, rate-limit, timeout, and format failures and applies safer backoff, circuit-breaker, alerting, and state visibility behavior for cron jobs.

## Best Possible Solution

Keep this as the canonical umbrella until maintainers approve and land a cron guard policy that covers recurring scheduling, persisted guard state, Gateway/UI visibility, docs, and regression tests while coordinating with provider/auth cooldown behavior.

## Reproduction Assessment

Yes for the source-level gap: current main records lastErrorReason but recurring cron still uses generic retry/backoff and public state lacks guard fields. I did not rerun a live provider retry-storm scenario.

## Solution Assessment

No current partial path is sufficient yet. The best fix needs a maintainer-approved guardrail design; https://github.com/openclaw/openclaw/pull/95787 and https://github.com/openclaw/openclaw/pull/64127 should be reviewed as partial slices, not treated as full replacements for this issue.
