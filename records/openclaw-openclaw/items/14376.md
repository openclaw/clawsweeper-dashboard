## Summary

Keep open. Current main has partial cron error observability and one-shot retry handling, but recurring cron still uses generic consecutive-error backoff and lacks the requested reason-specific circuit breaker, Retry-After scheduling, and job-level guard state.

## What This Changes

Request reason-aware cron guardrails that classify quota, auth, rate-limit, timeout, and format failures and apply safer backoff, circuit-breaker, and observable job-state behavior.

## Best Possible Solution

Add an explicit recurring cron guard policy with persisted state, additive protocol/client exposure, docs, and regression tests coordinated with auth-profile/provider cooldown behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature and policy request. Source inspection on current main confirms recurring cron still uses generic consecutive-error backoff, while the issue comments provide concrete retry-storm examples motivating the feature.

## Solution Assessment

Unclear until maintainers choose the policy shape. The requested direction is credible, but the maintainable solution needs a designed contract across scheduler policy, persisted state, Gateway schema, UI/macOS/plugin-visible surfaces, docs, and provider cooldown semantics.
