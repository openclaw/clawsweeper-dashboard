## Summary

Keep open. Current main has cron error-reason observability and bounded one-shot retry, but recurring cron still uses a fixed consecutive-error backoff and lacks the requested billing/auth/format circuit breaker, Retry-After-aware rate-limit scheduling, and job-level guard state; the discussion includes real retry-storm reports and a maintainer review already keeping this open.

## What This Changes

Request reason-aware cron guardrails that classify quota/auth/rate-limit/timeout/format failures and apply safer recurring-job backoff, circuit-break, alerting, and state visibility policies.

## Best Possible Solution

Add an explicit recurring cron guard layer that classifies terminal run errors, persists and displays guard state, respects Retry-After for rate limits, auto-pauses terminal billing/auth/format failures with one alert window, and coordinates with provider/auth-profile cooldowns.

## Reproduction Assessment

Yes, at source level: a recurring job result with `status: "error"` still flows through the same `errorBackoffMs(consecutiveErrors)` scheduling path regardless of classified reason. The real-world comments add impact evidence, but the missing policy branch is visible in current main without live provider traffic.

## Solution Assessment

No for current main: `lastErrorReason` is useful groundwork but not the requested mitigation. The proposed direction is sound, but the maintainable fix needs explicit policy across cron state/config/protocol/client surfaces rather than only a timer-local branch.
