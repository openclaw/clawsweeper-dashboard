## Summary

Keep open. Current main still has partial cron error observability and generic consecutive-error backoff, but it does not implement the requested reason-aware recurring cron guard layer, Retry-After scheduling, job-level circuit breaker state, or billing/auth/format auto-pause policy.

## What This Changes

The issue requests reason-aware cron guardrails that classify quota, auth, rate-limit, timeout, format, and unknown failures and apply distinct recurring-job backoff, alert, cooldown, or circuit-breaker behavior.

## Best Possible Solution

Add a coordinated cron guard layer that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with configurable safe defaults and one alert, respects Retry-After for rate limits, and keeps transient paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Yes at source level, though not as a live bug reproduction because this is a feature request. Current main records lastErrorReason, then schedules recurring failures through the same fixed consecutive-error backoff path without reason-specific guard state.

## Solution Assessment

Yes as a product direction, but not as a narrow timer-only fix. The maintainable solution needs explicit guard policy, config/schema additions, persisted state, UI/protocol/docs exposure, regression tests, and provider/auth cooldown coordination.
