## Summary

Keep open. Current main has partial cron error observability, but recurring cron still uses one fixed consecutive-error backoff path and lacks the requested reason-aware circuit breaker, Retry-After scheduling, and job-level guard state; the 2026-04-26 maintainer review in the issue reached the same conclusion.

## What This Changes

The issue requests reason-aware cron guardrails that classify quota, auth, rate-limit, timeout, format, and unknown failures and apply safer backoff or circuit-break behavior with observable job state.

## Best Possible Solution

Add an explicit cron guard layer that classifies terminal run errors once, persists/display guard state, circuit-breaks billing/auth/format failures with safe configurable defaults and one alert, respects Retry-After for rate limits, and keeps transient timeout/network paths enabled with bounded backoff.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is a feature request, but the missing behavior is source-reproducible. Current main classifies lastErrorReason and then schedules recurring errors through the same fixed backoff branch.

## Solution Assessment

Yes, the requested direction is the right maintainable shape, but it should be implemented as a coordinated cron policy/state/protocol/UI/docs change rather than a timer-only special case.
