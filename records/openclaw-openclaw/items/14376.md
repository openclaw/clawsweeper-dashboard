## Summary

Keep open. Current main and v2026.6.5 still only partially address the request: cron records and exposes lastErrorReason and has bounded transient retry behavior, but recurring jobs still lack the requested billing/auth/format circuit breaker, Retry-After-aware scheduling, and job-level guard state.

## What This Changes

Request reason-aware cron guardrails that classify provider/tool failure reasons and apply reason-specific backoff, circuit breaking, user notification, and observable job state.

## Best Possible Solution

Define an additive recurring cron guard contract, then implement reason-specific scheduling, persisted/displayed guard state, docs, and tests so quota/auth/format stop safely while transient and rate-limit paths remain bounded and recoverable.

## Reproduction Assessment

Yes, at source level. Current main records lastErrorReason and has transient retry branches, but recurring billing/auth/format-style failures still fall through to generic backoff without circuit-breaker state.

## Solution Assessment

Yes as a product direction, but not as a narrow timer-only fix. The maintainable solution needs an explicit guard policy, additive state/config/protocol design, UI/docs exposure, regression tests, and provider/auth cooldown coordination.
