## Summary

Keep open. Current main now handles part of the request—classified, bounded transient retries for recurring cron jobs plus last-error observability—but still applies generic backoff to permanent failures and has no billing/auth/format circuit breaker, provider Retry-After scheduling, or persisted job-level guard state. The real-world reports in the discussion support the remaining cost and availability problem, while the existing product-decision labels correctly reflect that automatic pausing and recovery semantics need maintainer ownership.

## What This Changes

Add reason-aware recurring cron backoff and circuit breaking for quota, authentication, rate-limit, timeout, format, and unknown failures.

## Best Possible Solution

Extend the existing cron retry decision into one canonical terminal guard outcome that honors provider retry timing, distinguishes transient retries from permanent-failure pauses, defines recovery and alert semantics, and exposes only stable job state needed by clients.

## Reproduction Assessment

Yes at source level: current main clearly demonstrates bounded transient recurring retries while lacking permanent-failure pause decisions, Retry-After scheduling, and persisted guard state. No controlled live provider retry storm was run, though the discussion includes two concrete production incidents.

## Solution Assessment

Partly. Reusing the current classifier, timer transition, failure-alert path, and provider cooldown concepts is the best foundation, but the broad original sketch needs a maintainer-approved policy for pause defaults, provider-wide coordination, recovery, configuration scope, and client state.
