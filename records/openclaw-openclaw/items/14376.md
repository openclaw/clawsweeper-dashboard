## Summary

Keep open. Current main has shipped partial cron error observability and one-shot retry handling, but recurring cron still lacks the requested reason-aware guard layer, Retry-After-aware scheduling, circuit breaker, and persisted cooldown/disabled guard state.

## What This Changes

Request reason-aware cron guardrails so quota/auth/format errors can circuit-break, rate limits can back off with Retry-After, and job state can expose guard status.

## Best Possible Solution

Add a coordinated cron guard layer that classifies terminal run errors once, persists and exposes guard state, circuit-breaks billing/auth/format failures with safe configurable defaults and one alert, respects Retry-After for rate limits, and keeps transient timeout/network paths enabled with bounded backoff coordinated with provider/auth cooldowns.

## Reproduction Assessment

Not applicable as a live bug reproduction because this is an enhancement request. The missing behavior is source-reproducible: current main records `lastErrorReason`, then schedules recurring errors through the same fixed consecutive-error backoff path.

## Solution Assessment

Yes as a product direction, but the exact defaults and config/schema shape need maintainer decision. The maintainable solution should span cron policy, persisted state, Gateway protocol, UI/macOS/plugin-visible surfaces, docs, and regression tests rather than adding a timer-only special case.
