## Summary

Keep open. Current main has partial cron error observability and generic recurring backoff/failure alerts, but recurring cron still does not apply reason-aware circuit breaking, Retry-After scheduling, or persisted job-level guard state; the real-world comments and the recent maintainer review still describe remaining product work.

## What This Changes

Requests reason-aware cron guardrails that classify quota, auth, rate-limit, timeout, and format failures, then back off or circuit-break recurring jobs with persisted state and user visibility.

## Best Possible Solution

Add a recurring cron guard layer around applyJobResult that uses shared failover classification and Retry-After/cooldown inputs, persists guard state, exposes it through Gateway/UI/docs, and tests billing/auth/format circuit-breaks separately from rate-limit/timeout backoff.

## Reproduction Assessment

Yes, at source level: current main classifies lastErrorReason but the recurring cron error branch still schedules from a fixed consecutive-error backoff. I did not run a live provider retry-storm scenario.

## Solution Assessment

Unclear as an implementation until maintainers choose defaults and ownership boundaries. A timer-only patch would be too narrow; the maintainable solution needs scheduler policy, state/protocol/client visibility, docs, and tests.
