Full review comments:

- **[P1] Use the in-memory session before rejecting native /name:** `src/auto-reply/reply/commands-name.ts:100-102`
  - body: When `/name <title>` is the first native slash command in a fresh Telegram or Discord session, `initFastReplySessionState` has already created `params.sessionEntry` and placed it in `params.sessionStore`, but it has not necessarily persisted a disk row yet. This branch only checks `resolved.existing` from the store and returns “no active session to name,” so the command can fail for a valid active native session; seed or persist the fallback entry before rejecting.
  - confidence: 0.88
- **[P1] Persist renamed aliases through the canonical key:** `src/auto-reply/reply/commands-name.ts:114-115`
  - body: This mutates the resolved entry object in place, but it never writes `store[resolved.normalizedKey]` or removes `resolved.legacyKeys`. Current helpers such as `updateSessionStoreEntry` do that via `persistResolvedSessionEntry`, so a rename against a legacy or folded alias can preserve the stale key and leave duplicate or stale rows in `sessions.list`; persist the renamed entry through the canonical helper/path.
  - confidence: 0.9
- **[P1] Notify session subscribers after saving the new name:** `src/auto-reply/reply/commands-name.ts:124-125`
  - body: A successful `/name` currently returns a chat reply after a direct store write, but it never emits the `sessions.changed` event that `sessions.patch` emits after label changes. Subscribed Control UI/TUI/SDK session lists can therefore keep showing the old label until reload; route the successful mutation through the command metadata notification seam or emit the equivalent scoped update before merge.
  - confidence: 0.86
