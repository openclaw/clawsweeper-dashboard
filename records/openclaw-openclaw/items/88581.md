Full review comments:

- **[P2] Seed native slash sessions before renaming:** `src/auto-reply/reply/commands-name.ts:99-102`
  - body: When `/name <title>` is the first native slash command in a fresh chat, the fast path has only created `params.sessionEntry` in memory. This branch checks only the persisted store entry and returns “no active session to name,” so the command cannot name the current session until some later turn persists it. Use the in-memory session entry as the fallback/upsert target before rejecting.
  - confidence: 0.88
- **[P2] Persist renamed aliases through the canonical key:** `src/auto-reply/reply/commands-name.ts:114-115`
  - body: If `resolveSessionStoreEntry` resolves a legacy or case-folded alias, mutating `entry.label` saves the row under that legacy key and leaves `resolved.legacyKeys` in place. Existing store helpers persist through `resolved.normalizedKey` and delete aliases, so this path can leave `sessions.list` exposing the stale alias row instead of the canonical session.
  - confidence: 0.86
- **[P2] Notify session subscribers after renaming:** `src/auto-reply/reply/commands-name.ts:125`
  - body: A successful `/name` write returns only a chat reply, while the existing `sessions.patch` rename path emits `sessions.changed`. Connected session lists and SDK subscribers can therefore keep showing the old label until a reload, so this command should trigger equivalent refresh behavior or be explicitly accepted as reload-only before merge.
  - confidence: 0.79
