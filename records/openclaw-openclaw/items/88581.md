## Summary

Keep this PR open. The Telegram Desktop proof is now sufficient and the feature is useful, but the current patch still has source-backed session-state blockers around first native slash-command sessions, canonical legacy-key persistence, and live session-list notification parity.

## What This Changes

Adds a built-in `/name <title>` chat command, handler tests, command registry entry, and slash-command docs for renaming the current session label from chat.

## Best Possible Solution

Land the command only after its write path matches current session-store semantics: seed fresh native sessions, persist through the canonical key, and either emit `sessions.changed` like `sessions.patch` or get explicit maintainer acceptance for the reload-only tradeoff.

## Reproduction Assessment

Yes. Source inspection gives high-confidence paths for the remaining defects: native slash handlers receive an in-memory session before disk persistence, while the new `/name` mutator reads only the disk store; the same source pass shows direct label mutation bypassing canonical persistence and `sessions.changed`.

## Solution Assessment

No. The command is a reasonable feature, but the current direct store-write shape is not the safest implementation until it preserves canonical session-key behavior and handles live session-list refresh semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Seed native slash sessions before renaming:** `src/auto-reply/reply/commands-name.ts:99-102`
  - body: When `/name <title>` is the first native slash command in a new chat, the native fast path has only prepared `params.sessionEntry` in memory. This mutator reads the on-disk store, sees no entry, and returns “no active session to name,” so first-turn native `/name` fails despite having an active session object available.
  - confidence: 0.9
- **[P1] Persist resolved aliases through the canonical key:** `src/auto-reply/reply/commands-name.ts:114-115`
  - body: `resolveSessionStoreEntry` can return an entry found under a legacy or folded key, but this branch only mutates that object in place. Unlike the existing persistence helpers, it never writes `store[resolved.normalizedKey]` or removes `resolved.legacyKeys`, so a rename can leave the stale alias row as the saved session entry.
  - confidence: 0.88
- **[P1] Notify session subscribers after renaming:** `src/auto-reply/reply/commands-name.ts:116-125`
  - body: A successful `/name` saves the label and replies in chat, but it never emits the `sessions.changed` event that the existing `sessions.patch` rename path emits. Connected session lists can therefore continue showing the old label until a manual or unrelated reload, which is a visible session-state regression unless maintainers explicitly accept the reload-only behavior.
  - confidence: 0.86
