## Summary

Current main and the v2026.5.3 tag already authorize the reported Telegram native command path when the sender is present in both `commands.allowFrom.telegram` and `commands.ownerAllowFrom`; `/restart` then passes the separate owner gate.

## What This Changes

The issue asks for Telegram `/restart` native command authorization to accept a sender configured in Telegram command and channel allowlists.

## Best Possible Solution

Keep the shipped v2026.5.3 behavior; users still seeing this should update and restart the actual Telegram-serving gateway process.

## Reproduction Assessment

No. The reported config maps to an authorized path in current main and in v2026.5.3: the Telegram sender id matches `commands.allowFrom.telegram`, and the same id in `commands.ownerAllowFrom` satisfies the `/restart` owner gate.

## Solution Assessment

Yes. The current split between command authorization and owner authorization is the narrow maintainable design for a dangerous command like `/restart`, and the provided config satisfies both without adding a new fallback or policy knob.
