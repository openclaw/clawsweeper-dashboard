## Summary

Current main and the v2026.5.3 release contain the Telegram native-command authorization path needed for the reported config: `commands.allowFrom.telegram` authorizes the sender, and `commands.ownerAllowFrom` then satisfies the owner-only `/restart` gate.

## What This Changes

The issue asks for Telegram `/restart` to run for a DM sender listed in `commands.allowFrom.telegram` and `commands.ownerAllowFrom`.

## Best Possible Solution

Keep the shipped command authorization behavior and treat any remaining report as a new diagnostics issue with exact installed version, effective config, `/whoami`, and Telegram native-command logs.

## Reproduction Assessment

No high-confidence failing reproduction remains on current main. Source inspection of the reported config path shows the Telegram sender ID should match `commands.allowFrom.telegram` and `commands.ownerAllowFrom`, with v2026.5.3 containing the same implementation.

## Solution Assessment

Yes. The narrow maintainable solution is the current shared command-auth path plus Telegram native-command handoff, because it preserves `commands.allowFrom` as the command allowlist and keeps `/restart` owner-only without adding Telegram-specific special cases.
