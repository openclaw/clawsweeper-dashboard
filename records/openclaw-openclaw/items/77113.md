## Summary

Current main and the latest shipped release already implement the requested Telegram native command authorization path: provider-specific `commands.allowFrom.telegram` is used for native command sender authorization, and `/restart` then separately checks the sender against owner authorization.

## What This Changes

Telegram `/restart` from a Telegram DM sender listed in `commands.allowFrom.telegram` and `commands.ownerAllowFrom` should execute instead of returning “You are not authorized.”

## Best Possible Solution

Keep the shipped command authorization model; if rejection still happens in a deployment, the next report should include the running gateway version, effective runtime config, and Telegram command debug logs showing the resolved sender and owner candidates.

## Reproduction Assessment

No. The reported config path is not a failing reproduction against current source: the Telegram sender ID is passed as `SenderId`, normalized against `commands.allowFrom.telegram`, and covered by native command authorization tests.

## Solution Assessment

Yes. The existing implementation keeps `commands.allowFrom` as the command sender allowlist and `commands.ownerAllowFrom` as the `/restart` owner gate, which is the narrowest maintainable behavior for this issue.
