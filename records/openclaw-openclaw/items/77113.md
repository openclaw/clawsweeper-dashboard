## Summary

Current main already authorizes the reported Telegram native-command path: `commands.allowFrom.telegram` is checked against the numeric Telegram sender ID before the native command rejection, and paired DM senders are covered by existing native-command tests.

## What This Changes

The issue reports that a Telegram DM sender listed in pairing/channel and command allow lists is rejected as unauthorized when sending `/restart`.

## Best Possible Solution

Keep the current shared command-auth implementation; if a released install still rejects after upgrading/restarting, open a fresh issue with the exact OpenClaw version, runtime config snapshot, and Telegram native-command logs.

## Reproduction Assessment

No high-confidence failing reproduction remains on current main. Source inspection of the exact Telegram DM native-command path shows the reported numeric sender ID is normalized and matched through `commands.allowFrom.telegram`, so the rejection branch should not run for that config.

## Solution Assessment

Yes. The existing shared command authorization path with Telegram-specific allowlist formatting is the narrow maintainable solution; a `/restart`-specific bypass would be less safe and duplicate the owner/command auth model.
