## Summary

Current main and the shipped v2026.5.3 source already authorize Telegram native commands from `commands.allowFrom.telegram`, and `/restart` additionally passes when the same sender is in `commands.ownerAllowFrom`.

## What This Changes

Allow a paired Telegram DM sender listed in `commands.allowFrom.telegram` and `commands.ownerAllowFrom` to run the native `/restart` command.

## Best Possible Solution

Keep the shipped authorization path; if the symptom persists on a specific install, the next report should include the installed OpenClaw version, native-command debug logs, and the effective runtime config snapshot.

## Reproduction Assessment

No. With the provided config shape, current source normalizes the Telegram numeric sender ID and matches it against `commands.allowFrom.telegram`, while `commands.ownerAllowFrom` makes the sender an owner for `/restart`.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable solution: native Telegram command auth uses the command-specific allowlist when configured, and `/restart` separately enforces owner-only status.
