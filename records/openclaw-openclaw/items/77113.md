## Summary

Current main and the shipped v2026.5.3 source already route Telegram native commands through `commands.allowFrom.telegram` and then re-check `commands.ownerAllowFrom` for `/restart`, so the reported numeric sender is no longer rejected by the current implementation.

## What This Changes

The issue reports that a Telegram sender listed in command and channel allowlists is rejected when sending `/restart`.

## Best Possible Solution

Keep the shipped command-auth path and ask for a new report only if an updated v2026.5.3-or-newer install still rejects the sender with `/whoami`, effective config, and version proof.

## Reproduction Assessment

No: the reported failure does not reproduce by current-main source trace. A Telegram DM sender id in `commands.allowFrom.telegram` becomes an authorized command sender, and the same id in `commands.ownerAllowFrom` satisfies the `/restart` owner gate.

## Solution Assessment

Yes: the current generic command-auth resolver plus Telegram native command wrapper is the narrow maintainable solution. A Telegram-only bypass would be less appropriate because the shared resolver already handles provider-specific command allowlists and owner lists.
