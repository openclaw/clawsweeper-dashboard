## Summary

The issue is reproducible and has an active linked implementation PR, so it should remain open until that PR is reviewed and resolved; current main enforces a 160-character TypeBox bound, while workshop persistence accepts the resulting description without a second length check.

## What This Changes

Requests a larger or configurable skill proposal description limit, or a clearer validation error when the current limit is exceeded.

## Best Possible Solution

Adopt one maintainable character-based description bound in the existing tool schema, correct the byte/character wording, add boundary regression coverage, and keep the error actionable; avoid a configuration option without a demonstrated operator policy need.

## Reproduction Assessment

Yes. On current main, any `skill_workshop` description longer than 160 characters is deterministically rejected by the TypeBox argument schema before the handler runs.

## Solution Assessment

Partly. Raising the existing character limit and correcting the message is the narrowest direction, but the exact 500-character bound needs maintainer agreement and focused boundary coverage; a new config key is not justified by the reported ergonomics problem.
