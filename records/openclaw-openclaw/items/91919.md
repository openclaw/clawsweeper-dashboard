## Label Justifications

- `P1`: Affected deployments cannot start the gateway at all when their state filesystem rejects chmod.
- `impact:crash-loop`: The synchronous permission exception terminates every gateway startup attempt on an affected filesystem.

## Summary

Current main still propagates synchronous `chmodSync` failures from the shared state database open and repair paths, so the reported gateway outage remains valid. The issue should stay open while the focused closing PR is reviewed; that PR covers the missing paths but still needs an explicit maintainer decision about tolerating every chmod failure versus only unsupported-filesystem errors.

## What This Changes

Prevent gateway startup and state-schema repair from crashing when state database permission hardening runs on filesystems that reject POSIX chmod operations.

## Best Possible Solution

Land the linked helper-level fix after deciding the tolerated chmod error policy, preserving containment failures and proving open, repair, and transaction behavior under rejected chmod operations.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence reproduction path: make `fs.chmodSync` throw EPERM, EINVAL, or ENOTSUP while opening the default shared state database, and the exception propagates before gateway startup completes; the linked PR records the matching before-fix stack.

## Solution Assessment

Yes, conditionally. Catching at the individual chmod operation is narrower and more complete than wrapping open call sites because it also fixes repair while retaining containment validation, but maintainers must choose whether all chmod errors or only unsupported-filesystem errors may continue.
