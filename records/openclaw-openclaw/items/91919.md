## Label Justifications

- `P1`: Affected storage can make the gateway unusable because a synchronous chmod exception escapes during required state database initialization.
- `impact:crash-loop`: The failure deterministically aborts gateway startup and can repeatedly restart the process under container health management.

## Summary

Keep open: current main and v2026.6.5 still allow chmod failures to escape during SQLite startup, and the related NFS WAL PR addresses journal mode rather than permission errors. The bug is narrow and queueable, but the repair should cover the sibling per-agent opener and must not swallow path-containment validation.

## What This Changes

Make SQLite state database permission hardening non-fatal when the backing filesystem rejects POSIX chmod operations.

## Best Possible Solution

Separate mandatory path validation from advisory mode tightening, then apply one consistent best-effort chmod policy to shared and per-agent database opens with focused EPERM/EINVAL regression coverage.

## Reproduction Assessment

Yes at source level: make chmod throw EPERM or EINVAL while opening either default SQLite database, and the exception escapes the unguarded startup call. This review did not execute a real incompatible storage mount, so the status is source-reproducible rather than reproduced.

## Solution Assessment

The recommended intent is correct, but wrapping the entire current helper is not the best implementation because it would hide path-containment validation. The maintainable fix separates validation from chmod handling and applies the policy consistently to both database owners.
