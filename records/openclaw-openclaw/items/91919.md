## Summary

Current main and the latest release still propagate open-time chmod failures from the shared state database, while the linked implementation PR remains open and unmerged. The issue should stay open until that repair is reviewed, proven on affected storage, and landed or replaced.

## What This Changes

Prevent the shared state database opener from crashing gateway startup when its filesystem rejects POSIX chmod operations.

## Best Possible Solution

Keep state-path validation and database creation failures fatal, make only unsupported chmod hardening non-fatal and observable, and prove the production gateway starts on a real incompatible mounted volume.

## Reproduction Assessment

Yes, at source level with high confidence: forcing chmodSync to fail during the production shared-state opener propagates an exception before startup completes. A real affected-filesystem run has not yet been supplied.

## Solution Assessment

Yes in direction: narrowly tolerating unsupported chmod operations is preferable to adding another environment variable or database-path setting. The implementation should preserve fatal validation and database errors while adding focused regression coverage and real storage proof.
