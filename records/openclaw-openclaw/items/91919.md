## Summary

Current `main` and stable release `v2026.6.5` still have the unhandled startup path, while the clean open PR https://github.com/openclaw/openclaw/pull/91934 already owns the focused repair. Keep this issue open until that PR is merged or rejected; the independent rootless-Docker report adds useful confirmation, although its database-open error may include an additional cause.

## What This Changes

Prevent unsupported POSIX `chmod` failures from crashing gateway startup and state-database repair while preserving fatal handling for genuine permission and I/O faults.

## Best Possible Solution

Land the linked helper-level fix after validating its closed errno contract and coverage for database open, schema repair, committed writes, and unexpected errors; then let the merged PR close this issue while separately following up if the rootless-Docker reporter still sees database-open failures.

## Reproduction Assessment

Yes, from source at high confidence. Make `chmodSync` throw an unsupported-filesystem error during `openOpenClawStateDatabase`, and the exception escapes the unguarded permission helper before SQLite can open; no live affected mount was exercised in this review.

## Solution Assessment

Yes, the linked PR's current approach is the narrowest maintainable fix: classify known unsupported `chmod` errors at the filesystem-call boundary rather than swallowing the entire permission helper or adding environment and database-path configuration.
