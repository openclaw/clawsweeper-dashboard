## Summary

Current main and the latest release still contain the source-reproducible startup crash. Keep this issue open because the viable closing PR is open and specifically owns the narrow repair.

## What This Changes

Prevent gateway startup failure when shared-state database permission hardening uses chmod on a backing filesystem that does not support POSIX permissions.

## Best Possible Solution

Merge https://github.com/openclaw/openclaw/pull/91934 with its chmod-only recovery boundary, deduplicated warning, and regression coverage, then let the closing reference resolve this issue.

## Reproduction Assessment

Yes. Current main directly reaches throwing `chmodSync` calls during database open, and the linked PR supplies a before/after ENOTSUP injection that demonstrates the startup failure and successful recovery.

## Solution Assessment

Yes. Handling failure at the individual chmod operation is the narrowest maintainable solution because it repairs all permission-helper callers without swallowing the path-containment guard or database errors.
