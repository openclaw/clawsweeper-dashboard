## Label Justifications

- `P1`: This shipped regression can prevent affected installations from starting the gateway or serving any channel.
- `impact:crash-loop`: The unhandled synchronous chmod error terminates startup and recurs on every restart against the affected storage.

## Summary

Current main and v2026.6.5 both contain the crashing shared-state chmod path, while the focused closing PR remains open and unmerged; this issue is still necessary as the canonical regression report.

## What This Changes

Prevent unsupported `chmodSync` permission hardening from crashing shared state-database open and schema repair on non-POSIX-capable storage.

## Best Possible Solution

Land a helper-level shared-state fix that preserves containment and database errors, makes the intended chmod-failure policy explicit, and covers open, repair, and write paths; then evaluate the separate agent-database helper as a follow-up.

## Reproduction Assessment

Yes at source level: injecting a throwing `chmodSync` while opening the shared database on current main propagates through the unguarded permission helper. The linked PR also provides matching before-and-after ENOTSUP output, although a real Azure Files or NFS mount was not tested.

## Solution Assessment

Yes in overall shape: the helper-level boundary is better than broad caller catches because it reaches open and repair while preserving path-containment checks. Maintainers still need to decide whether all chmod errors or only unsupported-filesystem error classes should be tolerated.
