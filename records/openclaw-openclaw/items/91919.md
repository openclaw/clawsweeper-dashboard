## Summary

Keep open: current main and the latest release still allow a synchronous permission-hardening failure to abort shared database initialization, and no viable open PR owns the repair. The closed attempted fix was not a safe landing path because it mixed unrelated changes and suppressed every `chmodSync` failure.

## What This Changes

Prevent gateway startup from failing when shared-state permission hardening invokes `chmodSync` on storage without POSIX permission support.

## Best Possible Solution

Introduce one observable best-effort permission path for filesystem capability failures during database open and post-write maintenance, while preserving strict path validation, directory-creation failures, and normal 0700/0600 enforcement.

## Reproduction Assessment

Yes at source level with high confidence: a throwing `chmodSync` propagates through two unguarded permission calls in `openOpenClawStateDatabase`. This review did not execute the gateway on a real mount that rejects chmod.

## Solution Assessment

The report's narrow best-effort direction is appropriate, but a catch-all that silently suppresses every chmod error is not the best fix. The repair should handle the intended unsupported-permission condition with safe diagnostics and retain actionable failures or an explicit warning when private modes cannot be enforced.
