## Summary

Current main still exposes the reported startup failure, and the open linked PR directly owns the repair. The remaining question is the safe degraded-permission policy, not whether the issue is valid or actionable.

## What This Changes

The issue requests that shared state database startup tolerate filesystems that reject POSIX chmod operations instead of terminating the gateway.

## Best Possible Solution

Merge a chmod-local best-effort repair that preserves containment and SQLite errors, emits an actionable degraded-permission warning, and verifies database open, schema repair, and post-write behavior.

## Reproduction Assessment

Yes at source level: a thrown `chmodSync` propagates from the open and repair paths before a usable database is returned. The linked PR also records a controlled ENOTSUP before-and-after run, though not on a real NFS, SMB, or Azure Files mount.

## Solution Assessment

Yes in direction: the linked PR’s chmod-local boundary is narrower than wrapping the complete permission helper because it preserves path-containment and database failures while covering all relevant callers. Maintainers should still confirm the intended error classification and warning policy.
