## Risks / Open Questions

- A broad boot-time plugin preflight framework can conflict with existing doctor lint checks, config dry-run validation, safe-restart preflight, and plugin lifecycle hooks unless maintainers choose one owner boundary.
- Running plugin-provided checks at boot can inspect env/config/state and affect fail-open/fail-closed startup behavior, so redaction, timeout, side-effect, and recovery semantics need explicit security design.
- A partial implementation that only wraps config.patch/config.apply or gateway startup would miss sibling paths such as doctor checks, shared config mutation helpers, plugin runtime loading, and state migrations.
