## Summary

Keep open. Current main has adjacent manifest validation, setupEntry metadata, gateway lifecycle hooks, SecretRef startup checks, and doctor invalid-plugin quarantine, but it does not expose the requested reusable fail-closed preflight framework for plugins. The remaining work changes plugin SDK/control-plane and startup semantics, so it needs maintainer design rather than cleanup closure.

## What This Changes

Request to add a reusable plugin preflight framework for boot-time environment, config, and state validation with plugin-registered checks and consistent reporting/failure behavior.

## Best Possible Solution

Define a narrow additive plugin preflight contract aligned with manifest validation, setupEntry, doctor repair, and gateway preflight before adding plugin-specific env/config/state checks.

## Reproduction Assessment

## Solution Assessment

Unclear until maintainers settle the SDK/control-plane design. A generic preflight surface may be the right direction, but an ad hoc startup hook would be less maintainable than a documented contract coordinated with existing gateway and config preflight work.
