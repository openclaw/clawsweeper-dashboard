## Summary

Keep open. Current main has manifest-based config validation, metadata startup planning, and lifecycle hooks, but it does not have a reusable rejectable plugin preflight contract; related gateway/config preflight items are adjacent and still open, so this is feature/API design rather than cleanup closure.

## What This Changes

The issue requests a reusable plugin preflight framework with plugin-level checks for environment variables, config files, and state integrity before startup.

## Best Possible Solution

Define a narrow additive plugin preflight contract aligned with manifest validation, gateway preflight, setupEntry, and doctor, then document the hook/result semantics before adding plugin-specific checks.

## Reproduction Assessment

Not applicable: this is a new plugin API/startup framework request, not broken existing behavior. Source inspection gives high confidence that current main lacks the requested boot-time check registration and fail-closed result contract.

## Solution Assessment

Unclear: a reusable preflight surface may be useful, but the maintainable solution needs an SDK/control-plane design rather than an ad hoc gateway_start hook or parallel validator.
