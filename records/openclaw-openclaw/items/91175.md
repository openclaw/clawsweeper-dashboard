## Summary

## What This Changes

The PR adds a memory-core Deep Sleep `DREAMS.md` managed-block writer, calls it after the deep sweep report write, and adds helper-level tests for creation, replacement, lowercase filename reuse, and symlink rejection.

## Best Possible Solution

Move Deep Sleep summary updates into a shared `DREAMS.md` update helper that validates before read, serializes all `DREAMS.md` edits, and uses atomic replacement, then prove the deep sweep and concurrent diary/backfill paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Guard DREAMS.md before reading it:** `extensions/memory-core/src/dreaming-markdown.ts:173`
  - body: The new helper resolves `DREAMS.md` and immediately reads it, then calls `assertSafeDreamsPath` only after building the replacement. A symlinked or non-file `DREAMS.md` is therefore opened before the refusal path runs, so a deep sweep can read or block on an unexpected target instead of rejecting it at the workspace boundary. Move the lstat/regular-file check ahead of the read, ideally in the shared DREAMS.md updater.
  - confidence: 0.86
- **[P1] Serialize the Deep Sleep DREAMS.md update:** `extensions/memory-core/src/dreaming-markdown.ts:189`
  - body: This is a read-modify-write of `DREAMS.md`, but it bypasses the process-wide `dreamsFileLocks` path that `appendNarrativeEntry`, backfill, reset, and dedupe use. A concurrent diary or backfill update can read the old file while this helper prepares its replacement, and whichever atomic rename runs last drops the other's section. Route the Deep Sleep block through the shared locked updater and add a race regression.
  - confidence: 0.9
