## Summary

Keep open: current main still persists automatic fallback model overrides, the requested no-persist policy is not implemented, and the overlapping open issues/PRs are alternatives rather than a merged or clearly canonical replacement.

## What This Changes

Request a config flag or policy that lets automatic model fallback apply to the current turn without persisting `modelOverrideSource: "auto"` into session state.

## Best Possible Solution

Pick one fallback-persistence contract, preserve current sticky fallback as the compatibility default, then implement the opt-out with schema/types/docs and regression coverage for default sticky mode, opt-out per-turn mode, primary-probe recovery, and stale/unreachable fallback providers.

## Reproduction Assessment

Yes for the persistence path: current source, docs, and tests show automatic fallback selections can persist as `modelOverrideSource: "auto"`. The full outage scenario still needs a controlled live primary-failure plus unreachable-fallback reproduction for high-confidence runtime proof.

## Solution Assessment

Unclear: the requested flag is plausible, but maintainers need to choose the API scope and semantics before implementation. A per-agent fallback policy, gateway-level flag, TTL, or narrow stale-pin cleanup each has different compatibility behavior.
