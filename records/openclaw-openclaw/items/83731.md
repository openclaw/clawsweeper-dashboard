- `P2`: This is a normal-priority routing/session improvement with real production impact, but it is not a core outage, security emergency, or already-approved narrow bug fix.
- `impact:session-state`: The reported problem is stale per-session fallback state pinning later turns to an automatic model override.
- `impact:auth-provider`: The issue concerns provider/model fallback routing and auth-provider drift when the persisted fallback target becomes unreachable.

## Summary

Keep open: current main and the latest release still persist automatic fallback selections, the supported recovery paths are primary probing, reset, and doctor cleanup rather than the requested opt-out policy, and the newer discussion adds a concrete drift case that needs maintainer product review.

## What This Changes

Requests a `gateway.routing.persist_auto_fallback` config option so fallback can be used for the current turn without persistently pinning `modelOverrideSource: "auto"` in session state.

## Best Possible Solution

Maintainers should decide whether OpenClaw should offer an opt-out or TTL policy for automatic fallback persistence; if accepted, implement it through typed config, schema/help/docs, and tests that preserve default stickiness while covering opt-out recovery and existing cleanup paths.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. Source, docs, and tests show the persistence path clearly, but a full repro would need a controlled primary failure, fallback success, degraded fallback target, and later primary recovery.

## Solution Assessment

Unclear: the requested flag is plausible, but it is a new core routing policy knob and not the only possible solution. Maintainers should choose between an opt-out flag, a TTL such as https://github.com/openclaw/openclaw/pull/79678, narrower cleanup such as https://github.com/openclaw/openclaw/pull/84476, or documenting the current probe/reset/doctor behavior.
