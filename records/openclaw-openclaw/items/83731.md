## Summary

Keep open: current main still persists automatic fallback model overrides and has no supported `gateway.routing.persist_auto_fallback` config surface. Related work improves primary probing or proposes TTL/narrow cleanup, but it does not replace the requested opt-out policy decision.

## What This Changes

Request adding `gateway.routing.persist_auto_fallback` so operators can keep current sticky fallback behavior by default or opt out of persisting automatic fallback model overrides.

## Best Possible Solution

If maintainers want this operator policy, add a typed config option with schema/help/docs coverage and regression tests that preserve current default stickiness while making auto-fallback persistence opt-out; otherwise choose and document the supported recovery model through primary probing, TTL, reset, or doctor cleanup.

## Reproduction Assessment

No high-confidence end-to-end repro was run in this review. The core persistence behavior is source-reproducible from current main and covered by tests, and the fresh comment adds real-world stuck-session state evidence.

## Solution Assessment

Unclear: the requested flag is plausible, but it is a new core routing policy knob rather than the only maintainable recovery path. Maintainers should choose between opt-out persistence, TTL, narrow ephemeral cleanup, and the existing primary-probe/reset/doctor mechanisms.
