## Summary

Keep open: current main still persists automatic fallback model overrides, has recovery probes and reset cleanup, but does not provide the requested no-persist routing policy. The May 20 real-world comment strengthens the case that this remains a maintainer product/config decision rather than an already-implemented cleanup.

## What This Changes

The issue requests a `gateway.routing.persist_auto_fallback` config flag so automatic model fallback can apply to the current turn without persisting a sticky session override.

## Best Possible Solution

If maintainers want this operator policy, add a typed config option with schema/help/docs coverage and tests that preserve current default stickiness while making auto-fallback persistence opt-out; otherwise choose the TTL or cleanup approach as the canonical policy and document that no flag will be added.

## Reproduction Assessment

Yes, source-reproducible at medium confidence: current code, docs, and tests show auto fallback selections are persisted as session overrides and only later probed or reset. I did not run a live provider outage/recovery reproduction, but the May 20 comment supplies a real-world stuck `sessions.json` case.

## Solution Assessment

Unclear: the requested flag is plausible, but it is a new core routing policy knob rather than the only possible fix. Maintainers should choose between opt-out persistence, TTL expiry, narrow ephemeral-failure cleanup, or documenting the existing probe/reset behavior.
