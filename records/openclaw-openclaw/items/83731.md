## Summary

Keep open: current main has auto-fallback recovery and cleanup paths, but it still persists automatic fallback model overrides and there is no supported config surface for the requested no-persist policy.

## What This Changes

The issue requests a `gateway.routing.persist_auto_fallback` config option so automatic fallback model selections can apply only to the current turn instead of being written to session state.

## Best Possible Solution

If maintainers want this operator policy, add a typed config option with schema/help/docs coverage and tests that preserve current default stickiness while making auto-fallback persistence opt-out; otherwise document the existing primary-probe, reset, and doctor cleanup behavior as the supported recovery path.

## Reproduction Assessment

Yes, source-reproducible at medium confidence: current code and tests show auto fallback selections can be persisted with `modelOverrideSource: "auto"`. A high-confidence live repro would require a controlled primary timeout, fallback success, and later primary recovery run.

## Solution Assessment

Unclear: the requested flag is plausible, but it is a new core routing policy knob rather than the only possible fix. Maintainers should choose whether opt-out persistence is preferable to the existing primary-probe, reset, and doctor cleanup behavior.
