## Summary

Keep open: current main still intentionally persists automatic fallback model pins, the requested config surface is absent, and the latest issue comment adds a concrete real-world stuck-state case that the existing primary-probe recovery does not fully cover.

## What This Changes

The issue requests a `gateway.routing.persist_auto_fallback` config option to let operators opt out of persisting automatic fallback model pins across turns.

## Best Possible Solution

Maintainers should choose the fallback recovery policy, then either rebase and land the TTL approach in https://github.com/openclaw/openclaw/pull/79678 or add a typed opt-out config with current sticky behavior preserved by default, schema/help/docs coverage, and tests for persistence, no-persist, primary-probe, and upgrade cases.

## Reproduction Assessment

No high-confidence live repro was run. The issue is source-reproducible at medium confidence because current code and tests show automatic fallback pins are persisted and reused, and the latest comment supplies a real captured stuck session state.

## Solution Assessment

Unclear. The requested flag is plausible and bounded, but it is a new core routing policy knob and overlaps with the open TTL proposal, so maintainers should choose the desired persistence/failback policy before implementation.
