## Summary

Keep this open. Current main and v2026.4.29 still route the skills watcher through root directory watches with an ignored callback and no traversal-depth bound, while the skills loader only needs root skills, direct child skills, and one grouped level. The issue is also paired with open closing PR #75518, so it should remain open until that PR is reviewed/merged or replaced.

## What This Changes

The issue reports that the skills watcher in OpenClaw 2026.4.29 can hit macOS EMFILE on startup when a watched workspace or skills root contains hundreds of thousands of files.

## Best Possible Solution

Keep the issue open and fix the watcher so its traversal matches skill discovery: root skill, direct child skill, and one grouped level only. The linked PR should either be narrowed into that fix with focused tests, or replaced by a small PR that preserves default ignores, adds bounded traversal coverage, and includes the required changelog entry.

## Reproduction Assessment

No, not as a live reproduction in this read-only review. The source and history provide a strong static reproduction path: current main recursively watches skill roots without a depth cap, and the latest release shipped the same watcher behavior reported by the user.

## Solution Assessment

No, adding only more directory names or reordering the existing `!stats` check is not the best fix. The maintainable fix is to bound watcher traversal to the loader's actual discovery depth; PR #75518 should not be treated as sufficient until it addresses that and drops unrelated changes.
