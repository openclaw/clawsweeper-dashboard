## Summary

Keep open. Current main still has a plausible skills-watcher EMFILE path because the watcher observes whole configured skill roots without an explicit traversal depth cap, while skill discovery itself is bounded; additionally, open PR #75518 references this issue with closing syntax, so this issue should remain open until that PR is reviewed, landed, or superseded.

## What This Changes

The issue requests preventing the skills watcher from exhausting file descriptors on large macOS workspaces by ignoring large non-source directories or exposing custom watch excludes.

## Best Possible Solution

Make the skills watcher match the bounded discovery contract: watch only skill roots deeply enough for root skills, direct child skills, and one grouped level, preserve the existing ignore list, and add regression coverage for large non-skill subtrees before deciding whether #75518 should land or be superseded.

## Reproduction Assessment

No high-confidence live reproduction was performed. The report provides concrete EMFILE symptoms and current source gives a focused static reproduction target, but the exact watched directory layout and `skills.load.extraDirs` values are still missing.

## Solution Assessment

No, the linked PR's stated reorder is not clearly the best solution because current main and the latest release already check ignore patterns before the directory early return. A safer fix is to bound watcher traversal to the skill discovery depth and cover that behavior with a targeted test.
