## Summary

Keep open: current main still lacks the WhatsApp config field, strict schema support, resolved account state, and Baileys send-option plumbing needed for outbound disappearing-message expiration. The related implementation PRs #73080 and #73083 were useful context, but both are closed unmerged, and the latest source still has no equivalent implementation.

## What This Changes

Request support for channel-level and account-level WhatsApp `disappearingMessagesSeconds` config that forwards positive values to Baileys outbound sends as `ephemeralExpiration`.

## Best Possible Solution

Implement this inside the bundled WhatsApp plugin: add typed and schema-validated channel/account config, resolve a positive effective expiration with account override semantics, forward it through all outbound text, media, quoted, voice-note companion text, poll, and tool send paths, then document the setting and add focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the proposed config key is absent from strict schema/types, resolved account state cannot carry it, and current WhatsApp send paths never pass Baileys `ephemeralExpiration` despite the pinned dependency exposing that option.

## Solution Assessment

Yes. A WhatsApp-plugin-owned config and send-options fix is the narrowest maintainable solution; no new core channel abstraction is needed unless multiple channel plugins later need a shared disappearing-message contract.

## Repair Work Prompt

Fix the WhatsApp disappearing-message expiration issue for outbound replies. Implement the feature inside the bundled WhatsApp plugin, using #73080 and #73083 only as source context, not as a wholesale patch. Add `channels.whatsapp.disappearingMessagesSeconds?: number` and `channels.whatsapp.accounts.<accountId>.disappearingMessagesSeconds?: number` to the typed config and strict zod schema. Account-level config must override channel-level config. Positive finite values should be floored to whole seconds and forwarded to Baileys as `MiscMessageGenerationOptions.ephemeralExpiration`; `0`, `null`, omitted, negative, NaN, and Infinity should mean no explicit OpenClaw override. Preserve existing behavior unless a positive override is configured. Carry the effective expiration through `ResolvedWhatsAppAccount`, `ActiveWebSendOptions`, `createWebSendApi`, `sendMessageWhatsApp`, and `sendPollWhatsApp` so text, media, quoted replies, voice-note companion text, polls, and channel/tool sends use the same policy. Update WhatsApp docs and config reference docs, and add a user-facing `CHANGELOG.md` entry with proper contributor thanks, including `Thanks @dougvk`; credit #73080/#73083 and @hclsys in the PR body if borrowing from that source work. Do not introduce a core channel abstraction or bundled-id logic in core. Do not change unrelated WhatsApp auth, account selection, media handling, reply threading, or Baileys dependency versions.
