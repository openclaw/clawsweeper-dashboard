## Summary

Keep open: current main and v2026.6.10 still use per-session run serialization plus a transcript-file write lock with timeout tuning; existing DM isolation helps some deployments but does not implement the requested read/write locking, batching, session pooling, or broader multi-user session-affinity model.

## What This Changes

The issue requests a production-scale session concurrency redesign so multiple Slack and Telegram users do not contend on one long-held per-session writer lock.

## Best Possible Solution

Maintainers should decide the intended core session concurrency model, then implement it with upgrade-safe session-state semantics and channel delivery proof rather than relaxing the lock opportunistically.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not yet: source and docs show the single-writer session lock model, but the reported production Slack and Telegram contention was not live-reproduced against current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the issue lists plausible directions, but choosing read/write locks, per-sender sharding, batching, or session pools is an architecture and product decision that needs maintainer scoping.
