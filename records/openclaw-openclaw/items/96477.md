## Summary

Keep open: current main and v2026.6.10 still expose an exclusive per-session transcript write-lock model with timeout/stale/max-hold tuning, while this issue asks for a broader multi-user concurrency architecture. The active related work addresses narrower lock timeout and delivery-loss symptoms, but it explicitly does not resolve read/write locking, per-sender sharding, batching, or session-pool direction.

## What This Changes

The issue asks OpenClaw to relax or redesign the per-session single-writer lock so multi-user Slack and Telegram deployments do not queue or fail behind long-running agent pipelines.

## Best Possible Solution

Keep this issue open as the maintainer-owned design tracker for core session concurrency, then split the chosen read/write-lock, sharding, queueing, or bounded-status design into focused PRs with live channel and upgrade proof.

## Reproduction Assessment

Yes at source level: current main still serializes transcript mutations through an exclusive session write lock with a timeout path. I did not reproduce the exact 3-5 user Slack/Telegram production load pattern live.

## Solution Assessment

Unclear: the issue correctly identifies a core bottleneck, but the best fix depends on maintainer choice between lock granularity, session affinity, queueing, and user-facing busy semantics. A one-off timeout tweak or the open narrow delivery PR would not solve the full request.
