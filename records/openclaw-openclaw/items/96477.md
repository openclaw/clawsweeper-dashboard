## Summary

Keep open: current main and v2026.6.10 still use a per-transcript/session write-lock model with timeout knobs, while the request asks for a broader core session-concurrency architecture decision that is not fully owned by any narrower open tracker.

## What This Changes

The issue asks OpenClaw to relax per-session write-lock bottlenecks for multi-user Slack and Telegram deployments through a model such as read/write locking, session sharding, batching, or session pools.

## Best Possible Solution

Keep this as a maintainer-owned session concurrency design tracker, choose one core model, then split the chosen implementation into focused PRs with upgrade, session-state, and channel-delivery proof coordinated with the narrower lock/delivery issues.

## Reproduction Assessment

Yes for the current lock model: source and docs show transcript writes serialize through a per-session transcript write lock with a 60000 ms default acquire timeout. The exact production Slack/Telegram multi-user failure pattern was not live-reproduced in this read-only review.

## Solution Assessment

No final solution has been selected yet. Timeout tuning and narrower delivery fixes address symptoms, but read/write locking, sharding, batching, and session pools require a maintainer-owned architecture and compatibility decision.
