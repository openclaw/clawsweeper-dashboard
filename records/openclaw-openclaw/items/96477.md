## Summary

Keep open: current main and v2026.6.10 still use a per-session transcript write-lock model with a 60s acquire timeout, and the new reporter comment says the production Slack/Telegram friction remains after the narrower abort-timeout fix. Existing `dmScope` and per-sender routing can reduce some multi-user contention, but they do not answer the broader read/write-lock, batching, or shared-session concurrency design question.

## What This Changes

The issue requests a more production-friendly session concurrency model for multi-user Slack and Telegram deployments, considering read/write locks, session sharding, write batching, or session pools.

## Best Possible Solution

Keep this as a maintainer-owned production scaling design tracker, clarify the supported path for multi-user isolation versus intentional shared-session workloads, then split any chosen concurrency change into focused PRs with live channel proof.

## Reproduction Assessment

Yes for the source-level contention model: current code and docs show per-session transcript writes wait on one file lock and fail after the configured acquire timeout. No live current-main Slack/Telegram 3-5 user reproduction was established in this read-only review.

## Solution Assessment

No single proposed path is proven best yet. Existing DM/session isolation is a partial mitigation, while read/write locking, prioritized batching, or session pooling need maintainer architecture choice and upgrade-safety proof before implementation.
