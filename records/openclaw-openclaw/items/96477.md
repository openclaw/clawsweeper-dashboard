## Summary

Keep open: current main and v2026.6.10 still use exclusive per-session transcript write locks, while existing per-sender/per-channel-peer routing only mitigates part of the reported multi-user contention. The remaining work is a core session-concurrency product decision, not a cleanup-close candidate.

## What This Changes

The issue requests production-friendly session concurrency by relaxing the current per-session single-writer lock model through read/write locking, per-sender sharding, batching, or session pooling.

## Best Possible Solution

Keep this as the broader session-concurrency design tracker, document/configure existing DM isolation as the short-term mitigation, and split any chosen read/write-lock, batching, or shared-session affinity change into focused PRs with upgrade and channel-delivery proof.

## Reproduction Assessment

Yes for the underlying lock model: current source shows embedded runs and transcript mutations still coordinate through exclusive session write locks. The exact production Slack/Telegram multi-user failure was not live-reproduced in this read-only review.

## Solution Assessment

Unclear which proposed direction is best. Current per-sender/per-channel-peer routing mitigates some DM contention, but read/write locking, shared-session batching, or pooling requires maintainer architecture selection.
