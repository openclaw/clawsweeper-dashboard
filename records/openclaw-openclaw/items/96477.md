## Summary

Keep open: current main and v2026.6.10 still document and implement per-session serialized lanes plus a file-based session write lock with timeout tuning, while the requested read/write locking, per-sender sharding, batching, or session-pool model remains a core session architecture decision rather than an implemented cleanup item.

## What This Changes

Request to make the per-session single-writer transcript lock more production-friendly for concurrent multi-user Slack and Telegram deployments.

## Best Possible Solution

Keep this open for maintainer-owned design scoping of the core session concurrency and affinity model, then split the chosen approach into focused PRs with channel-delivery and upgrade proof.

## Reproduction Assessment

Partly. Source and docs show the per-session lane plus file-based write-lock model with a 60000 ms default acquire timeout, but the exact multi-user Slack/Telegram production failure was not live-reproduced in this read-only pass.

## Solution Assessment

No current implementation is the best solution yet. Timeout tuning and the linked lock/delivery PR address symptoms, while this issue needs a maintainer-selected concurrency model before code changes are safe.
