## Summary

Keep open: current main and v2026.6.10 still use the per-session transcript write-lock and timeout model, and the existing `dmScope` isolation path only covers one slice of the broader production scaling request. Related lock-delivery issues cover narrower failures, but this issue still needs a maintainer architecture/product decision.

## What This Changes

The issue asks OpenClaw to make per-session locking more production-friendly for concurrent Slack and Telegram users without removing write serialization.

## Best Possible Solution

Keep this as maintainer-owned session concurrency design work, then split the selected model into focused core/session and channel-delivery changes with upgrade proof.

## Reproduction Assessment

Yes for the lock model itself: current source and docs show transcript mutations serialize through a per-session write lock with a 60s default acquire timeout. No live multi-user Slack/Telegram deployment was reproduced in this read-only review.

## Solution Assessment

No current implementation is the best solution yet; `dmScope` covers some DM isolation, while this issue needs a maintainer-selected lock granularity or session-affinity model before implementation.
