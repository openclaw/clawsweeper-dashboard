## Summary

Keep open: current main and v2026.6.10 still use a per-transcript session write-lock model with timeout tuning, while this issue asks for a broader multi-user concurrency design. Existing DM/session isolation knobs and narrower lock-delivery trackers partially overlap, but they do not settle the requested read/write lock, session affinity, batching, or session-pool direction.

## What This Changes

The issue requests a production-friendly session concurrency model for multi-user Slack and Telegram deployments that currently contend on one session write lock.

## Best Possible Solution

Keep this open for maintainer-owned design scoping, choose the supported session concurrency model, then split implementation into focused PRs with upgrade-safe config/docs and live Slack or Telegram proof.

## Reproduction Assessment

Yes for the current lock model: source and docs show transcript writes serialize through one per-session write lock with a 60s default acquire timeout. The exact multi-user Slack/Telegram production failure was not live-reproduced in this read-only review.

## Solution Assessment

Unclear as an implementation path: the issue identifies plausible directions, but the best fix needs a maintainer-selected concurrency model rather than another timeout-only patch.
