## Summary

Keep open: current main still resolves Control UI config labels from server uiHints, schema titles, or English key humanization, and the request is a narrow i18n enhancement rather than an already-fixed bug.

## What This Changes

The issue requests that schema-driven Control UI config field labels resolve through the existing i18n system before falling back to English labels.

## Best Possible Solution

Add a small optional i18n lookup layer for config field and section labels, keyed by stable config paths, with English fallback preserved and generated locale workflow respected.

## Reproduction Assessment

Yes from source inspection: the config form label resolver never consults the i18n translation map before rendering labels. I did not run the browser UI in this read-only review.

## Solution Assessment

Yes, the requested direction is maintainable if implemented as an optional i18n lookup that falls back to existing uiHints/schema/humanize behavior. The safest shape is to centralize the lookup in the config-form label helper rather than scattering calls through render code.
