## Summary

Keep open: current main still resolves Control UI config labels from server uiHints, schema titles, or English key humanization, so the requested i18n label override path is not implemented yet.

## What This Changes

The issue requests i18n-backed labels for Control UI config page fields, with the existing generated English labels preserved as fallback.

## Best Possible Solution

Add a centralized optional i18n lookup for stable config field and section paths, preserving existing uiHints/schema/humanize fallback and the generated locale workflow.

## Reproduction Assessment

Yes, from source inspection: the config form label resolver never consults the i18n translation map before rendering labels. I did not run the browser UI in this read-only review.

## Solution Assessment

Yes, the requested direction is maintainable if implemented as an optional i18n lookup that falls back to existing uiHints/schema/humanize behavior. The safest shape is to centralize the lookup in the config-form label helper rather than scattering calls through render code.
