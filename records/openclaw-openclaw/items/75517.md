## Summary

Keep open. Current main still has a high-confidence path where `models list --provider <id>` without `--all` stays on the configured-only row path and can print `No models found`, while the full provider catalog path is only exercised by `--all --provider`; the report is recent and plausible enough that this needs a maintainer decision rather than cleanup closure.

## What This Changes

The issue asks for `openclaw models list --provider google` to return Google model rows instead of `No models found`.

## Best Possible Solution

Decide the CLI contract explicitly. If the old UX should be preserved, route `models list --provider <id>` with no configured rows through a provider-scoped catalog path or add a clear fallback hint, then cover Google/provider-without-configured-rows behavior and update docs/changelog to match.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: plain `--provider` skips the all-source plan and registry path, then appends only configured rows; the existing forward-compat test already expects `No models found` for a provider with no configured rows.

## Solution Assessment

Unclear. Restoring provider catalog rows for plain `--provider` may be the user-friendly fix, but current docs and tests intentionally reserve full catalog listing for `--all --provider`, so maintainers should choose between restoring the shortcut, adding a hint, or keeping the current contract.
