## Summary

Keep open. Current main still has the reported configured-only `models list --provider <id>` path, and the issue is now paired with open fix PR #75615 using closing syntax, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue requests restoring `openclaw models list --provider google` so it returns provider catalog models instead of `No models found` when the provider has no manually configured model rows.

## Best Possible Solution

Review and land PR #75615 if its behavior matches the intended CLI contract, or replace it with an equivalent narrow fix that routes non-`--all` provider filters through the provider catalog cascade while preserving the no-filter configured-only fast path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `models list --provider <id>` without `--all` skips source planning and uses only configured rows, and the existing forward-compat test expects `No models found.` for an unconfigured provider filter.

## Solution Assessment

Yes, the linked PR direction appears to be the narrow maintainable path: reuse the existing provider-scoped catalog cascade for non-`--all` provider filters while keeping ordinary `models list` registry-free. Maintainers still need to review the exact PR semantics before merge.
