## Summary

Keep open. Current main still routes `models list --provider <id>` without `--all` through configured-only rows, so a provider such as Google can still print `No models found` when no configured model rows are present; the issue is recent and conflicts with an intentional current docs/test contract rather than being cleanup-closeable.

## What This Changes

The issue asks for `openclaw models list --provider google` to return Google model rows instead of `No models found`.

## Best Possible Solution

Define the CLI contract explicitly. If the old UX should be preserved, route `models list --provider <id>` with no configured rows through a provider-scoped catalog/registry fallback or show a clear hint, then update focused tests, docs, and the changelog to match.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: plain `--provider` skips the all-source plan and registry path, then appends only configured rows; the existing forward-compat test already expects `No models found` for a provider with no configured rows.

## Solution Assessment

Unclear. Restoring provider catalog rows for plain `--provider` may be the user-friendly fix, but current docs and tests intentionally reserve full catalog listing for `--all --provider`, so maintainers should choose between restoring the shortcut, adding a hint, or keeping the current contract.
