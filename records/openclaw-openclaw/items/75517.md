## Summary

Keep open. Current main still has the reported empty-result path for plain provider-filtered model lists, and the issue is already paired with open PR #75615 using closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests restoring `openclaw models list --provider <id>` so provider filters such as `google` return catalog/model rows instead of `No models found` when the user has not manually configured provider models.

## Best Possible Solution

Review #75615 and either land it or adjust it so the CLI contract is explicit: plain `--provider` should either use the provider catalog cascade consistently, or the command/docs should clearly require `--all --provider` and give users a better hint than an empty result.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `modelsListCommand({ provider: "moonshot", json: true })` is tested to avoid registry/catalog loading and log `No models found.`, matching the same unconfigured-provider path reported for Google.

## Solution Assessment

Unclear. The linked PR targets the right boundary by reusing the existing provider catalog/source-plan cascade for provider-filtered non-`--all` lists, but its behavior changes around configured rows and registry fallback should be reviewed before treating it as the final fix.
