## Summary

Keep open: the Telegram bug is still source-reproducible on current main, but this PR targets a stale reply-media helper instead of the current reply-chain fetch path and still lacks required real Telegram proof.

## What This Changes

The PR attempts to prevent Telegram replies to bot-authored media from re-injecting the bot's own media as an inbound user attachment.

## Best Possible Solution

Add the numeric bot-id guard in resolveReplyMediaForChain before bot.api.getFile, preserve reply media from users and other bots, keep focused regression coverage, and attach redacted live Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows a Telegram text update with bot-authored reply_to_message.photo can enter buildTelegramReplyChain and reach resolveReplyMediaForChain, where bot.api.getFile is called without a self-authored reply-target check.

## Solution Assessment

No as-is. The numeric sender-id guard is the right narrow fix, but it needs to protect the current reply-chain fetch path and be backed by real Telegram proof before merge.
