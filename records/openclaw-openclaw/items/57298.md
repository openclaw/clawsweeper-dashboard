## Summary

Keep open: the Telegram bug is still present on current main, but this PR is not mergeable as-is because its guard targets an obsolete reply-media helper instead of the active reply-chain fetch path and it still lacks required real Telegram proof.

## What This Changes

The PR adds a Telegram bot-id guard, regression tests, and a changelog entry intended to stop bot-authored reply-target media from being fetched as inbound user media.

## Best Possible Solution

Refresh or replace the branch with a numeric `ctx.me.id` guard inside `resolveReplyMediaForChain` before fetching each reply-chain node, while preserving user and other-bot reply media with focused tests and redacted live Telegram proof.

## Reproduction Assessment

Yes, via source inspection. A Telegram text update whose `reply_to_message.photo` was authored by `ctx.me` can be normalized into a reply-chain node and reach `bot.api.getFile` in `resolveReplyMediaForChain` before any nested sender-id check.

## Solution Assessment

No as-is. The numeric bot-id guard is the right narrow solution, but this branch needs to move it to the active reply-chain fetch path and provide real Telegram proof.
