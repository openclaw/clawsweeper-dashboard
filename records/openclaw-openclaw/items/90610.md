## Summary

The feature is still necessary and the branch follows the right owner boundaries, but one concrete lifecycle bug prevents merge: a `final_answer` item that starts and completes without a text delta never becomes an activity candidate or selected record. This is a recent, focused contribution with useful tests and renderer proof; it should stay open for a narrow repair rather than cleanup closure.

## What This Changes

The PR emits Codex final-answer candidate lifecycle item events, records and renders them in Control UI activity, suppresses them from channel progress, and adds localized activity labels.

## Best Possible Solution

Seed or update final-answer candidates from canonical completed item/turn snapshots as well as deltas, keeping emission idempotent, then prove both delta-rich and snapshot-only sequences produce the same candidate/superseded/selected activity while channels still receive only the selected final payload.

## Reproduction Assessment

Yes, at source level: start and complete a `phase: "final_answer"` agent message with text present only in the completion snapshot and no text-delta notification; the final reply is collected, but no answer-candidate activity is emitted.

## Solution Assessment

Mostly yes: projecting provider-native candidate lifecycle at the Codex owner boundary and consuming generic item events in Control UI is the narrow maintainable design, but snapshot-only completion must share the same canonical candidate update path as streaming deltas.
