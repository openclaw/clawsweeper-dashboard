## Summary

This PR remains useful, but it is not merge-ready: the happy-path lifecycle is well covered, while `buildResult()` can falsely mark a streamed draft as `selected` after an aborted or failed Codex turn. The Control UI renderer proof is useful, but it does not prove the changed live Codex-to-Gateway event path.

## What This Changes

Adds Codex final-answer candidate lifecycle events, renders candidate/superseded/selected records in localized Control UI activity, and suppresses those records from channel progress.

## Best Possible Solution

Mark a candidate `selected` only after a successfully completed turn establishes an authoritative final answer, define and test the failed/cancelled candidate outcome, then demonstrate a redacted real Gateway-to-Control-UI run.

## Reproduction Assessment

Yes at source level: existing aborted and failed projector paths call `buildResult()`, while this PR unconditionally marks any active final-answer candidate selected at that boundary.

## Solution Assessment

No in its current form; candidate selection must be tied to successful authoritative turn completion rather than every terminal projection.
