## Summary

This PR remains useful and is not implemented on current `main`, but it is not merge-ready: the new projector marks a final-answer candidate `selected` before the run’s authoritative terminal outcome is known, so failed, aborted, timed-out, or client-closed attempts can be displayed as successful selections. The supplied Vite fixture demonstrates rendering but does not prove the complete Gateway/Codex activity path or this non-success behavior.

## What This Changes

The PR emits Codex final-answer candidate lifecycle events, suppresses them from channel progress, and renders candidate, superseded, and selected states in localized Control UI activity entries.

## Best Possible Solution

Derive `selected` only from an authoritative successfully completed Codex turn, terminate non-success candidates without selection, and then provide inspectable proof of the real Gateway-to-Control-UI activity flow.

## Reproduction Assessment

Yes. Stream a `final_answer` item, terminate the attempt through failure, abort, timeout, or client close, and call the existing result path; this branch emits `selected` before that non-success outcome is finalized.

## Solution Assessment

No as currently implemented. The Codex projector is the correct ownership boundary, but candidate selection must be gated by authoritative successful completion rather than by the existence of final-answer text during generic result construction.
