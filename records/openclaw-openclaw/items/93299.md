## Summary

Keep open: this is a focused candidate fix for the Windows Scheduled Task foreground-listener bug, but PR head still reports fallback success when the verified port owner is a different PID and the after-fix proof is fixture-based rather than a native Scheduled Task handoff.

## What This Changes

The branch changes Windows Scheduled Task fallback launch detection to baseline-filter process evidence and makes install/start/restart consume a boolean fallback launch result, with startup-fallback regression tests.

## Best Possible Solution

Require fallback success to prove the spawned fallback or installed managed task owns the port, cover the mismatched-PID case, and add redacted live Windows Scheduled Task handoff proof before merge.

## Reproduction Assessment

Yes, at source level: current main accepts listener-backed launch evidence without a pre-run baseline, and PR head still treats a mismatched verified listener PID as fallback success. I did not run a live Windows Scheduled Task handoff in this read-only review.

## Solution Assessment

No. Baseline-aware launch evidence is the right layer, but this implementation must fail or keep waiting when the verified port owner is not the spawned fallback or installed managed task.
