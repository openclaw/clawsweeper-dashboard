## Summary

Keep open: the latest patch is not merge-ready because it only filters the baseline in `hasLaunchEvidence`, while the foreground OpenClaw listener path can still return `running` from `readLaunchObservation` and skip the fallback before that filter is reached.

## What This Changes

This PR changes Windows Scheduled Task fallback launch detection in `src/daemon/schtasks.ts` to capture baseline gateway listener PIDs and ignore baseline PIDs when checking post-run launch evidence.

## Best Possible Solution

Make the launch-evidence flow baseline-aware before any listener-backed runtime is accepted as `running`, then add focused tests for both a pre-existing foreground OpenClaw listener and an unrelated WSL listener on the task port.

## Reproduction Assessment

Yes, at source level: a pre-existing foreground OpenClaw listener can make `readScheduledTaskRuntime` report `running`, and the PR still exits before the new baseline filter is used. I did not run live Windows Task Scheduler proof in this read-only review.

## Solution Assessment

No. Baseline filtering is the right direction, but the current patch applies it too late; the maintainable fix must prevent listener-backed runtime status from suppressing fallback before baseline-aware launch evidence runs.
