## Summary

Keep open: the patch improves the baseline launch-evidence path, but it still reports fallback success when a different PID owns the gateway port, so the foreground-window Scheduled Task failure can survive the change. The supplied proof is Windows terminal output for mocked tests and netstat/tasklist, not a live Scheduled Task handoff after closing the foreground gateway.

## What This Changes

The PR makes Windows Scheduled Task launch evidence baseline-aware, changes fallback launch to return a boolean consumed by three call sites, and updates daemon startup-fallback tests.

## Best Possible Solution

Require fallback success to prove the spawned or managed task owns the port or passes durable health, add the mismatched-PID regression, and provide redacted live Windows Scheduled Task handoff proof before merge.

## Reproduction Assessment

Yes at source level: the linked issue and current main show listener-backed launch evidence can accept a foreground gateway, and PR head still treats a different verified listener PID as fallback success. I did not run a live Windows Scheduled Task handoff in this read-only review.

## Solution Assessment

No. Baseline-aware launch evidence is the right layer, but the fallback result must fail when another PID owns the port and the proof should exercise a real Task Scheduler handoff.
