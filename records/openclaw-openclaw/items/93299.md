## Summary

Keep open: this PR targets a real Windows Scheduled Task availability bug, but PR head still reports fallback success when a different verified PID owns the gateway port, and the supplied proof does not show a live after-fix Windows Scheduled Task handoff.

## What This Changes

The branch changes Windows schtasks fallback launch detection to use raw task state plus baseline-filtered process evidence, returns a boolean from fallback launch, and adds startup-fallback regression tests.

## Best Possible Solution

Require fallback success to prove the spawned fallback or installed managed task owns the port, add the mismatched-PID regression, rebase on current main, and attach redacted live Windows Scheduled Task handoff proof before merge.

## Reproduction Assessment

Yes, at source level: current main can accept listener-backed runtime or port PID evidence during Scheduled Task fallback launch, matching the foreground-listener false positive. I did not run a live Windows Task Scheduler repro in this read-only review.

## Solution Assessment

No. Baseline-aware launch evidence is the right layer, but this implementation must fail or continue waiting when the verified port owner is not the spawned fallback or installed managed task.
