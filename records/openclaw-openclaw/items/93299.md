## Summary

Keep open: the PR is still necessary for the linked Windows Scheduled Task bug, but the current patch does not fix the foreground-listener path because `readScheduledTaskRuntime` can still return `running` before the new `hasLaunchEvidence` filter runs.

## What This Changes

The PR changes `src/daemon/schtasks.ts` to capture a baseline of configured-port listener PIDs and ignore those PIDs when checking Scheduled Task launch evidence.

## Best Possible Solution

Move the baseline or raw task-state decision into the launch observation path, add focused regression tests for the foreground-listener case, and require redacted native Windows Scheduled Task handoff proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level: current main can report listener-backed Scheduled Task runtime as `running` and skip fallback launch when a foreground gateway already owns the port. I did not run a live Windows Task Scheduler handoff in this read-only pass.

## Solution Assessment

No. Baseline-aware launch evidence is the right layer, but this patch applies the baseline only in `hasLaunchEvidence`; the initial runtime observation can still short-circuit the fallback before that filter runs.
