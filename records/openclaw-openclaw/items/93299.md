## Summary

Keep open: this remains the active candidate for the linked Windows Scheduled Task bug, but the current head is not merge-ready because the new baseline is captured after `/Run` and after a listener-backed runtime check, leaving direct schtasks fallback tests failing and live Windows proof still insufficient.

## What This Changes

The PR changes Windows Scheduled Task fallback launch detection to filter process launch evidence against a baseline snapshot and adds a startup-fallback regression test for foreground gateway listeners.

## Best Possible Solution

Capture a pre-/Run process baseline in the caller, make launch polling use raw schtasks state instead of listener-backed runtime status, preserve genuine fast-starting task evidence, and then add redacted live Windows proof.

## Reproduction Assessment

Yes, at source level. The current PR head has direct CI failures in `src/daemon/schtasks.startup-fallback.test.ts` showing the foreground-listener case still does not spawn fallback and the already-starting task case now spawns a duplicate fallback.

## Solution Assessment

No. Baseline-aware launch evidence is the right direction, but the baseline must be captured before `/Run` and launch polling must avoid listener-backed runtime status so the fix distinguishes pre-existing foreground listeners from the managed Scheduled Task process.

Capture the baseline in `runScheduledTaskOrThrow()` before `/Run`, or compare only against a true pre-run snapshot.
