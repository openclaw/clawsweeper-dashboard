## Summary

Keep open: the patch is not merge-ready because the new baseline filter is applied only in the later launch-evidence check, while the initial launch observation can still accept a pre-existing listener-backed runtime as `running` and skip fallback first.

## What This Changes

The PR modifies Windows Scheduled Task fallback launch detection to ignore gateway listener PIDs that were already present before the fallback poll.

## Best Possible Solution

Make launch observation baseline-aware before accepting listener-backed `running` status, add focused regression tests for pre-existing foreground and unrelated listeners, then require current-head Windows terminal or log proof.

## Reproduction Assessment

Yes, at source level: a pre-existing verified gateway listener can make `readScheduledTaskRuntime()` report `running`, and this PR still exits before the new baseline filter is consulted. I did not run live Windows Task Scheduler proof in this read-only review.

## Solution Assessment

No. Baseline filtering is the right direction, but this implementation applies it too late; the launch observation itself needs raw schtasks or baseline-aware handling before listener-backed runtime status is accepted.
