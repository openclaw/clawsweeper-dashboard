## Summary

Keep open: this PR is a useful candidate for the focused Windows Scheduled Task listener-evidence bug, but the latest source-only patch still captures the listener baseline after `schtasks /Run`, so a quickly started managed task can be misclassified as baseline and trigger the fallback unnecessarily; native after-fix Windows handoff proof is also still insufficient.

## What This Changes

The PR changes `src/daemon/schtasks.ts` to filter Windows Scheduled Task fallback listener evidence through a baseline PID set.

## Best Possible Solution

Capture the listener baseline before `/Run`, pass it into the fallback decision, keep listener-backed status reporting separate from launch proof, add focused regression coverage, and land only after redacted native Windows handoff proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: the fallback launch decision trusts listener-backed runtime state, and the PR now captures listener baseline only after `schtasks /Run` has already been issued. I did not run a live Windows Task Scheduler handoff in this read-only pass.

## Solution Assessment

No. Baseline-aware launch evidence is the right layer, but this patch captures the baseline too late and lacks focused regression coverage at the live head.
