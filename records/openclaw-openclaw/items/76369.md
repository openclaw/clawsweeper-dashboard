## Summary

Keep this issue open. Current main still reads the BlueBubbles webhook password as a raw string even though the schema and public SecretRef surface allow SecretRef objects, and the related Discord SecretRef PRs do not cover this plugin path.

## What This Changes

The issue reports dropped inbound BlueBubbles/iMessage webhooks when `channels.bluebubbles.password` is configured as a SecretRef object.

## Best Possible Solution

Make BlueBubbles resolve or carry a secret-aware string password for webhook auth and catchup, then add regression coverage for SecretRef-shaped password config without changing core SecretRef policy.

## Reproduction Assessment

Yes, from source inspection: a registered BlueBubbles target with a SecretRef-shaped `password` reaches `target.account.config.password?.trim()` and throws, and catchup reaches a strict unresolved SecretRef resolver. I did not execute the live webhook path in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-local secret-aware password handling for webhook auth and catchup. A guard that only avoids `.trim()` on objects would stop the TypeError but would still reject valid SecretRef-backed webhook requests.
