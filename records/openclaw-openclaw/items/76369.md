## Summary

Keep this issue open. Current main still has the BlueBubbles webhook auth crash path, and the linked open PRs are implementation candidates rather than a reason to close before merge.

## What This Changes

Fix BlueBubbles inbound webhook authentication so SecretRef-backed `channels.bluebubbles.password` values do not throw and drop inbound messages.

## Best Possible Solution

BlueBubbles should resolve or safely normalize SecretRef-backed passwords before webhook auth and any catchup/API use, with regression coverage for object-shaped SecretRef config.

## Reproduction Assessment

Yes from source inspection. A registered BlueBubbles webhook target with a SecretRef object in `account.config.password` reaches `target.account.config.password?.trim()` in `monitor.ts` and throws because the object has no `trim` method.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-local SecretRef resolution/normalization before comparing webhook credentials, without changing core SecretRef policy; maintainers should choose between the linked PRs and settle the catchup scope.
