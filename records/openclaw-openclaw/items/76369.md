## Summary

Keep this issue open. Current main still treats the BlueBubbles webhook auth password as a string at the request gate, while the same channel explicitly declares `channels.bluebubbles.password` as a SecretRef-capable credential, so the reported TypeError is source-reproducible and belongs in the BlueBubbles plugin.

## What This Changes

The issue asks for BlueBubbles webhook, server-info, and catchup password paths to handle `channels.bluebubbles.password` SecretRefs instead of treating them as plain strings.

## Best Possible Solution

Fix this inside the BlueBubbles plugin by resolving or carrying the runtime-resolved password into webhook auth, server-info probing, and catchup paths, then add focused regression coverage for SecretRef-backed passwords.

## Reproduction Assessment

Yes, at source level: a registered BlueBubbles webhook target whose `account.config.password` is a SecretRef object will hit `.trim()` in `extensions/bluebubbles/src/monitor.ts` and throw before auth can complete. I did not execute the gateway because this review is read-only and `pnpm` is unavailable in the checkout environment.

## Solution Assessment

Yes, the maintainable direction is a narrow plugin-owned fix using the existing SecretInput/runtime-resolution helpers and tests around the webhook auth boundary. A core migration or new config option is unnecessary unless implementation proves the plugin loader is handing source snapshots to channel runtime incorrectly.
