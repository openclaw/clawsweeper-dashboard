## Summary

Keep open: the Telegram bug is still present on current main, and the source PR has the right product direction, but the branch is stale against the current `createTelegramBotCore`/`TelegramPollingSession` structure and needs to move the abort scoping into the live bot-core path before it can merge.

## What This Changes

The PR changes Telegram abort handling so polling restart/shutdown cancellation is intended to apply only to `getUpdates`, with regression tests that keep non-polling sends from being aborted.

## Best Possible Solution

Land a current-main Telegram plugin fix in the bot-core/polling-session path that scopes polling-session abort propagation to `getUpdates`, leaves outbound sends governed by their own request signal and method timeout, updates regression tests for `getUpdates`, `sendMessage`, and preferably `sendChatAction`, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: current polling restarts abort the shared `fetchAbortController`, current bot-core applies that signal to every Telegram fetch, and the existing fetch-abort test still asserts broad cancellation.

## Solution Assessment

No, not as this branch currently stands. The concept is the narrow maintainable fix, but the implementation must move from the old `bot.ts` middleware shape into the current `bot-core.ts` fetch wrapper or an equivalent current-main seam.
