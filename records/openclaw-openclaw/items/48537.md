## Summary

Keep open. The reported Telegram abort bleed is still plausible on current main, and production comments support the failure pattern, but this PR branch is stale against the current `bot-core.ts` refactor and needs a rebased implementation before it can merge.

## What This Changes

The PR moves Telegram polling abort handling from the shared fetch wrapper to a `getUpdates`-only API middleware and updates fetch-abort tests for `getUpdates` versus `sendMessage`.

## Best Possible Solution

Land a current-main Telegram plugin fix in `bot-core.ts` or an equivalent current seam so polling-session restarts still cancel `getUpdates`, while `sendMessage`, `sendChatAction`, and other outbound calls depend only on their own request signal and timeout guard.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: polling stall recovery aborts the active fetch controller, and current `bot-core.ts` forwards that abort signal into every Telegram fetch; comments provide matching production logs.

## Solution Assessment

No. The concept is the narrow maintainable fix, but this branch implements it in the old `bot.ts` shape instead of the active `bot-core.ts` fetch wrapper on current main.
