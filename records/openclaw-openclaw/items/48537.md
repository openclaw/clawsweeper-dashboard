## Summary

Keep this PR open, but do not merge it as-is. The bug is valid and production comments support the direction, but current main moved Telegram bot creation and fetch abort handling into `bot-core.ts`, while this branch edits the older `bot.ts` shape, so the proposed patch would not fix the current shared-abort path after rebase.

## What This Changes

The PR scopes Telegram polling restart/shutdown abort handling to `getUpdates` via a grammY API middleware in `bot.ts` and updates fetch-abort regression tests for `getUpdates` versus non-polling sends.

## Best Possible Solution

Land a current-main Telegram plugin fix in the `bot-core.ts` fetch wrapper or an equivalent current-main seam: polling-session abort propagation should still cancel `getUpdates`, while `sendMessage`, `sendChatAction`, and other outbound methods should rely only on their own request signal and method timeout. Add regression coverage for `getUpdates`, `sendMessage`, and ideally `sendChatAction`, plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: current polling restarts abort the shared `fetchAbortController`, current `bot-core.ts` attaches that signal to every Telegram fetch, and current tests still assert broad cancellation; production comments provide matching runtime symptoms.

## Solution Assessment

No, not as this branch currently stands. The concept is the narrow maintainable fix, but the implementation needs to move from the old `bot.ts` middleware shape into the current `bot-core.ts` fetch wrapper or a current-main equivalent.
