## Summary

Keep open: the reported Telegram abort-scope bug is still present on current main and latest release, but this PR branch targets the old `bot.ts` implementation shape instead of the current `bot-core.ts` fetch wrapper, so it is not mergeable as-is.

## What This Changes

The PR scopes Telegram polling restart abort handling to `getUpdates` and updates fetch-abort tests so non-polling sends are not cancelled by the polling restart signal.

## Best Possible Solution

Land a current-main Telegram plugin fix in `bot-core.ts` that scopes the polling-session `fetchAbortSignal` to `getUpdates` only, preserves request signals and method timeouts for outbound sends, and adds regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes, by source inspection. Current main passes one polling-session abort signal into `createTelegramBotCore`, aborts it during stall recovery, and the fetch wrapper applies it to all Telegram API methods without a `getUpdates` guard.

## Solution Assessment

No, not as this branch currently stands. The design direction is right, but the implementation needs to move from the old `bot.ts` middleware shape into the current `bot-core.ts` fetch wrapper or an equivalent current-main seam.
