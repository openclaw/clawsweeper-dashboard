## Summary

Keep open for normal review and landing. The PR is a focused fix for the linked SQLite WAL stall path: current main runs the default timer through a synchronous TRUNCATE checkpoint, while the branch changes only the periodic default to PASSIVE and preserves TRUNCATE for manual/close checkpoints with targeted coverage.

## What This Changes

The PR changes SQLite WAL maintenance so the default periodic checkpoint uses PASSIVE, close/manual checkpoints keep the TRUNCATE default, and tests cover defaults plus explicit checkpointMode overrides.

## Best Possible Solution

Land the narrow hybrid checkpoint change with regression coverage after normal CI/maintainer review; track a separate follow-up only if production data shows PASSIVE checkpoint I/O still stalls the event loop.

## Reproduction Assessment

Yes, source-reproducible: current main's default timer calls DatabaseSync.exec with a TRUNCATE checkpoint, and DatabaseSync is synchronous while SQLite documents the heavier TRUNCATE lock/truncation behavior. ([nodejs.org](https://nodejs.org/api/sqlite.html))

## Solution Assessment

Yes: using PASSIVE only for the default periodic path while keeping TRUNCATE for close/manual checkpoints is the narrowest maintainable fix for the reported timer stall path.
