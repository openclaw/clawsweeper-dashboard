## Summary

Keep open. Current main and `v2026.4.29` contain the Telegram read-only accessor fix, but the same author still has open PR #74833 with closing syntax for this issue, so conservative cleanup should wait until that PR is reviewed, merged, or explicitly closed as superseded.

## What This Changes

The issue reports that `openclaw status` can fail when Telegram `botToken` is a file-backed SecretRef because read-only allowlist metadata accessors strict-resolve the runtime token.

## Best Possible Solution

Resolve PR #74833 first, either by closing it as superseded by the shipped implementation or by landing an equivalent reviewed patch; then this issue can close as implemented with release evidence.

## Reproduction Assessment

Yes. The report gives a concrete file-backed Telegram `botToken` SecretRef config, and current `extensions/telegram/src/shared.test.ts` covers that shape for `resolveAllowFrom` and `resolveDefaultTo`; no test command was run because this review was required to stay read-only.

## Solution Assessment

Yes. The current config-only accessor is the narrow maintainable fix because it reads `allowFrom` and `defaultTo` metadata without dereferencing `botToken`, while runtime token resolution remains strict.
