## Summary

Keep open. Current main and v2026.4.29 already contain the Telegram config-only accessor behavior for read-only status metadata, but this issue is paired with open PR #74833 using closing syntax, so conservative cleanup should leave it open until that PR is reviewed, merged, or explicitly closed as superseded.

## What This Changes

The issue reports that `openclaw status` can fail when Telegram `botToken` is a file-backed SecretRef because read-only allowlist/default-target metadata paths dereference the strict runtime token resolver.

## Best Possible Solution

Resolve #74833 first: either close it as superseded by the v2026.4.29/main implementation or land an equivalent reviewed patch, then close this issue as implemented with the release and commit evidence.

## Reproduction Assessment

Yes. The report gives a concrete file-backed Telegram `botToken` SecretRef config, and current `extensions/telegram/src/shared.test.ts` exercises the same read-only accessor path without resolving the token.

## Solution Assessment

Yes. A config-only Telegram `resolveAccessorAccount` is the narrow maintainable fix because status metadata reads avoid secret dereference while runtime `resolveTelegramAccount` remains strict.
