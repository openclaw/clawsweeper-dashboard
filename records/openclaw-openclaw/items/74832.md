## Summary

Keep open. Current main and v2026.4.29 contain the intended Telegram read-only accessor behavior, but this issue is still paired with open PR #74833 using closing syntax, so conservative cleanup should leave the issue open until that PR is reviewed, merged, or explicitly closed as superseded.

## What This Changes

The issue reports that `openclaw status` can crash when Telegram `botToken` is a file-backed SecretRef because read-only allowlist/default-target metadata paths dereference the strict runtime token resolver.

## Best Possible Solution

Resolve #74833 first: close that PR as superseded by the v2026.4.29/current-main implementation if maintainers accept the shipped code, or land an equivalent reviewed patch, then close this issue as implemented with the release evidence.

## Reproduction Assessment

Yes. The report gives a concrete file-backed Telegram `botToken` SecretRef config, and current `extensions/telegram/src/shared.test.ts` covers the same failure boundary by exercising `allowFrom` and `defaultTo` through the read-only config adapter.

## Solution Assessment

Yes. The config-only `resolveAccessorAccount` is the narrow maintainable fix because it keeps status metadata reads from dereferencing secrets while leaving runtime `resolveTelegramAccount` token resolution strict.
