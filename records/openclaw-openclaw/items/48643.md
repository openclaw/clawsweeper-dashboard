## Summary

Keep this PR open, but it is not merge-ready: the current branch still bypasses the established `sessions.create` contract, has rollback bugs, is missing contributor-supplied real behavior proof, and overlaps ongoing Control UI parallel-chat UX work.

## What This Changes

The PR adds a Control UI chat toolbar button and modal for creating a labeled parallel thread with a selected agent and first message, plus `handleCreateThread` wiring and chat modal CSS.

## Best Possible Solution

Adopt one coherent Control UI parallel-session flow that uses `sessions.create` with parent linkage, label, and initial message support, preserves the active chat state on failure, handles dialog focus robustly, and aligns with the pinned-chat UX direction before landing.

## Reproduction Assessment

Yes, for PR review: source inspection of the PR head shows the direct `chat.send` path and rollback defects, while current main shows the `sessions.create` and `loadChatHistory` contracts. This is not a current-main bug reproduction because the dedicated create-thread modal is not on main.

## Solution Assessment

No. The branch should be rebuilt around `sessions.create` and a safer rollback/focus model, and the UI direction should be reconciled with the open pinned-chat parallel-session work.

Full review comments:

- **[P2] Route thread creation through sessions.create:** `ui/src/ui/app-chat.ts:181-186`
  - body: This generates a session key, switches local state, and calls `chat.send` directly. Current main's `sessions.create` owns parent linkage, label/model metadata, transcript creation, and optional first-message startup, so this creates sessions outside the established contract.
  - confidence: 0.9
- **[P2] Restore the previous transcript on create failure:** `ui/src/ui/app-chat.ts:189-198`
  - body: The catch path switches back to the previous session but clears `chatMessages`, tool messages, stream state, and run state before calling `loadChatHistory`. If the gateway is disconnected or the history reload fails, the user returns to the previous session with an empty conversation after a failed create.
  - confidence: 0.88
- **[P2] Preserve the create failure message after rollback:** `ui/src/ui/app-chat.ts:205-206`
  - body: After setting the specific create-thread error, the following `loadChatHistory` call clears `lastError` on entry and leaves it null when history reload succeeds. The dialog then reports the generic failure message instead of the actionable reason.
  - confidence: 0.91
- **[P3] Keep Escape handling active for the dialog:** `ui/src/ui/views/chat.ts:904`
  - body: The Escape handler is attached to the overlay, so it only receives events while focus remains inside a descendant. If focus leaves the dialog, Escape no longer closes or guards it; use a document-level listener or focus trap while the modal is open.
  - confidence: 0.78
