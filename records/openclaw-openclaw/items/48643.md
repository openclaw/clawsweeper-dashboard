## Summary

Keep open: the feature is still absent from current main, but this PR should not merge as-is because it bypasses the current session-creation contract, has rollback defects, lacks contributor real-behavior proof, and overlaps the active pinned-chat UX direction.

## What This Changes

The PR adds a Control UI New thread modal plus app handler and CSS to create a parallel chat session with a selected agent, optional label, and first message without resetting the current session.

## Best Possible Solution

Rework the feature into one coherent Control UI parallel-session flow that uses sessions.create with parent linkage, label, and first-message support, preserves existing chat state on failure, and aligns with the pinned-chat UX direction.

## Reproduction Assessment

Yes for PR review: source inspection of the PR head shows the direct chat.send path and rollback defects, while current main shows the sessions.create and loadChatHistory contracts. This is not a current-main bug reproduction because the dedicated create-thread modal is not on main.

## Solution Assessment

No. The branch should route creation through sessions.create, preserve previous chat state and the actionable failure message on rollback, and settle the visible UX alongside the pinned-chat work before merge.

  - confidence: 0.88

  - confidence: 0.9

  - confidence: 0.78
