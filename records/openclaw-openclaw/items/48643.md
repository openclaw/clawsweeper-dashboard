## What This Changes

The PR adds a Control UI New thread dialog, handler wiring, and styles for creating a parallel chat session with a selected agent, optional label, and first message.

## Best Possible Solution

Rework this into one Control UI parallel-session flow that uses `sessions.create` for parent linkage, label, and first-message startup, preserves previous session state on failure, and fits the pinned/threaded chat UX direction.

## Reproduction Assessment

Yes for PR review: source inspection of the PR head shows the direct `chat.send` path and rollback defects. This is not a current-main bug reproduction because the dedicated New thread modal is not on main.

## Solution Assessment

No. The branch should use the existing `sessions.create` contract, preserve the previous transcript and actionable error on failure, and align the visible affordance with the broader pinned/threaded chat direction before merge.

Full review comments:

- **[P1] Route thread creation through `sessions.create`:** `ui/src/ui/app-chat.ts:181-186`
  - body: This creates a generated key, switches local state, and calls `chat.send` directly.   - confidence: 0.89

  - confidence: 0.9

  - body: The Escape handler is attached to the overlay, so it only receives key events while focus remains inside a descendant. If focus leaves the dialog, Escape no longer closes or guards it; use a document-level listener or a focus trap while the modal is open.
  - confidence: 0.78
