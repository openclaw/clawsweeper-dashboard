## Summary

Keep open. Current main still lacks the dedicated Control UI flow for creating a parallel chat thread, and the PR remains a useful implementation candidate, but the branch is not merge-ready against current main because it bypasses the newer `sessions.create` contract and still has concrete rollback/error and dialog-keyboard defects called out in review discussion.

## What This Changes

The PR adds a Control UI chat “New thread” button and dialog, wires an `onCreateThread` callback through app state, creates a generated session key, sends the first message, optionally labels the session, and adds modal styling.

## Best Possible Solution

Implement the Control UI feature on top of the current gateway contract: call `sessions.create` with the active session as `parentSessionKey` and the optional first task/message, keep the active transcript and actionable error intact on failure, make the modal keyboard/focus behavior robust, and cover success, rollback, and in-flight dismissal with focused UI tests.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspection of the chat props, toolbar wiring, and reset-focused tests; the PR defects are reproducible from the diff paths where `chat.send` is used instead of `sessions.create`, rollback clears prior messages, and `loadChatHistory` clears `lastError` on successful reload.

## Solution Assessment

No, not as proposed. The dedicated UI is the right product direction, but the maintainable implementation should be rebased onto `sessions.create` rather than creating a client-side session key and sending the first message through `chat.send`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Route thread creation through sessions.create:** `ui/src/ui/app-chat.ts:181-186`
  - body: This creates the new thread by sending the first message to a client-generated session key. On current main, `sessions.create` is the gateway contract that records `parentSessionKey`, label/model metadata, transcript creation, and optional first task/message; bypassing it leaves the new conversation unlinked from the active session and races label setup. Please call `sessions.create` with the active session as `parentSessionKey` instead of sending to a synthetic key.
  - confidence: 0.88
- **[P2] Restore the previous transcript on create failure:** `ui/src/ui/app-chat.ts:192-198`
  - body: The rollback path clears `chatMessages` and stream/tool state before reloading history. If `chat.send` fails because the gateway disconnected, `loadChatHistory` returns early while disconnected, leaving the UI back on the previous session with an empty transcript even though no reset was requested. Preserve the previous chat state or only replace it after a successful history reload.
  - confidence: 0.86
- **[P2] Preserve the create failure message after rollback:** `ui/src/ui/app-chat.ts:205-206`
  - body: After setting `host.lastError` to the specific create-thread failure, the next `loadChatHistory` call clears `lastError` on entry and leaves it null when history reload succeeds. The dialog wrapper then shows only the generic fallback, so users lose the actionable error. Capture and restore the create failure after the history reload.
  - confidence: 0.9
- **[P3] Keep Escape handling attached while the dialog is open:** `ui/src/ui/views/chat.ts:903-904`
  - body: The Escape handler is attached to the overlay, so it only runs while focus remains inside a descendant. Tabbing out of the dialog means Escape no longer closes or guards the modal. Add a document-level listener or a focus trap while the dialog is open.
  - confidence: 0.78
