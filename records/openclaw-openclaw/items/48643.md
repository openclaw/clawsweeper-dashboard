## Summary

Keep open for maintainer/contributor follow-up: the feature is plausible, but the current PR still bypasses the established session creation contract, has rollback bugs, lacks contributor real behavior proof, and overlaps the open pinned-chat direction.

## What This Changes

The PR adds a Control UI New thread modal, handler wiring, and styles to create a parallel chat session with a chosen agent, optional label, and first message.

## Best Possible Solution

Rework the feature into one Control UI parallel-session flow that uses sessions.create for parent linkage, label, and initial message support, preserves previous chat state and errors on rollback, and aligns the visible affordance with the pinned-chat UX direction.

## Reproduction Assessment

Yes for PR review: source inspection of the PR head shows the direct chat.send path and rollback defects. This is not a current-main bug reproduction because the dedicated New thread modal is not on main.

## Solution Assessment

No. The branch should use the existing sessions.create contract, preserve the previous transcript and actionable error on failure, and settle the UI alongside the pinned-chat work before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route thread creation through sessions.create:** `ui/src/ui/app-chat.ts:181-186`
  - body: This creates a generated key, switches local state, and calls chat.send directly. Current main's sessions.create owns session row creation, transcript setup, parent linkage, label/model metadata, and optional initial message startup, so this bypasses the established session contract.
  - confidence: 0.91
- **[P2] Restore the previous transcript on create failure:** `ui/src/ui/app-chat.ts:189-198`
  - body: The error path switches back to the previous session but clears messages and stream/tool state before reloading history. If the gateway is disconnected or the reload fails, the user returns to the previous session with an empty transcript.
  - confidence: 0.88
- **[P2] Preserve the create failure message after rollback:** `ui/src/ui/app-chat.ts:205-206`
  - body: After setting the specific create-thread error, the following loadChatHistory call clears lastError on entry and leaves it null when history reload succeeds, so the dialog reports the generic failure instead of the actionable reason.
  - confidence: 0.9
- **[P3] Keep Escape handling active for the dialog:** `ui/src/ui/views/chat.ts:904`
  - body: The Escape handler is attached to the overlay, so it only receives key events while focus remains inside a descendant. If focus leaves the dialog, Escape no longer closes or guards it; use a document-level listener or a focus trap while the modal is open.
  - confidence: 0.78
