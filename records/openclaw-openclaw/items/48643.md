## Summary

Keep open for contributor revision: the feature is useful, but the branch bypasses the current session-creation contract, has rollback defects, lacks real behavior proof, and overlaps the newer pinned-chat work in https://github.com/openclaw/openclaw/pull/79747.

## What This Changes

The PR adds a Control UI new-thread modal and wiring that switches to a generated session key, sends the first message, optionally patches a label, and styles the dialog.

## Best Possible Solution

Adopt one coherent Control UI parallel-session flow that delegates creation to sessions.create, preserves the current chat state on failure, handles dialog focus robustly, and includes focused tests plus contributor-supplied runtime proof.

## Reproduction Assessment

Yes, for PR review: source inspection of the PR head reproduces the chat.send bypass and rollback defects, while current main shows the sessions.create and loadChatHistory contracts. This is not a failing current-main bug because the dedicated new-thread dialog is not on main.

## Solution Assessment

No. The feature should be rebuilt around sessions.create with parentSessionKey, label, and task/message support, while preserving rollback state and resolving the dialog focus/Escape behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route thread creation through sessions.create:** `ui/src/ui/app-chat.ts:181-186`
  - body: This creates the new thread by switching to a client-generated session key and calling chat.send. Current main's sessions.create owns parent linkage, label/model metadata, transcript creation, and first-message startup, so this bypasses the session contract used by dashboard-created sessions.
  - confidence: 0.9
- **[P2] Restore the previous transcript on create failure:** `ui/src/ui/app-chat.ts:189-198`
  - body: The catch path switches back to the previous session but clears chatMessages, tool messages, stream state, and run state before calling loadChatHistory. If the gateway is disconnected or chat.history cannot reload, the user returns to the previous session with an empty transcript after a failed create.
  - confidence: 0.88
- **[P2] Preserve the create failure message after rollback:** `ui/src/ui/app-chat.ts:205-206`
  - body: After setting the specific create-thread error, the following loadChatHistory call clears lastError on entry and leaves it null when history reload succeeds. The dialog then shows the generic failure message instead of the actionable error.
  - confidence: 0.91
- **[P3] Keep Escape handling active for the open dialog:** `ui/src/ui/views/chat.ts:904`
  - body: The Escape handler is attached to the overlay, so it only sees events while focus remains inside a descendant. If focus leaves the dialog, Escape no longer closes or guards it; use a document-level listener or focus trap while the modal is open.
  - confidence: 0.78
