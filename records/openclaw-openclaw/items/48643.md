## Summary

Keep open. Current main now has a gateway-backed New Chat path, but it does not fully implement this PR's dedicated dialog and first-message flow, and the PR branch still has concrete correctness blockers around stale session creation and rollback behavior.

## What This Changes

The PR adds a Control UI New thread dialog, app callback wiring, modal styles, and a client-generated session/send flow for starting a parallel chat without resetting the active session.

## Best Possible Solution

Rework the PR on top of current main so the dialog uses `sessions.create` with parent linkage and an initial message, keeps rollback state intact, adds robust modal keyboard behavior, and includes focused tests plus a changelog entry.

## Reproduction Assessment

Yes, for this PR review. Source inspection and the provided PR comments reproduce the blockers: the branch creates a client key and calls `chat.send` instead of the current `sessions.create` contract, and its rollback path can drop transcript state and the actionable error.

## Solution Assessment

No. The UI direction is useful, but this branch is not the best implementation because current main already has a first-class `sessions.create` contract for parent-linked dashboard sessions and initial messages.

Full review comments:

- **[P2] Route thread creation through sessions.create:** `ui/src/ui/app-chat.ts:181-186`
  - body: This creates the new thread by sending the first message to a client-generated session key. Current main has `sessions.create` for dashboard-created sessions, including `parentSessionKey`, label/model metadata, transcript creation, and optional first task/message, so this flow should use that contract instead of bypassing it with `chat.send`.
  - confidence: 0.9
- **[P2] Restore the previous transcript on create failure:** `ui/src/ui/app-chat.ts:192-198`
  - body: The rollback path clears `chatMessages` and stream/tool state before trying to reload the previous session. If the create request fails during a disconnect or the history reload cannot complete, the UI returns to the previous session with an empty transcript even though no reset was requested.
  - confidence: 0.86
- **[P2] Preserve the create failure message after rollback:** `ui/src/ui/app-chat.ts:205-206`
  - body: After setting `host.lastError` to the create-thread failure, the following `loadChatHistory` call clears `lastError` on entry and leaves it null when history reload succeeds. Capture and restore the create failure after the history reload so the dialog shows the real error.
  - confidence: 0.91
- **[P3] Keep Escape handling attached while the dialog is open:** `ui/src/ui/views/chat.ts:903-904`
  - body: The Escape handler is attached to the overlay, so it only runs while focus remains inside a descendant. If focus leaves the dialog, Escape no longer closes or guards it; use a document-level listener or focus trap while the modal is open.
  - confidence: 0.78

  - body: This is a user-facing `feat(ui)` change, but the diff does not include `CHANGELOG.md`. OpenClaw policy requires an active-version changelog entry before merge.
  - confidence: 0.87
