Keep open: the feature is plausible, but this PR is not merge-ready because the branch bypasses the current `sessions.create` contract, has rollback defects, lacks contributor real-behavior proof, and overlaps active pinned-chat UX work.

## What This Changes

The PR adds a Control UI dialog and handler for creating a parallel chat thread with a selected agent, label, first message, and modal styling.

## Best Possible Solution

Adopt one coherent Control UI parallel-session flow that uses `sessions.create` with parent linkage, label, and initial message support, preserves existing chat state on failure, and reconciles the UX with pinned-chat work.

## Reproduction Assessment

Yes for PR review: source inspection of the PR head shows the direct `chat.send` path and rollback defects, while current main shows the `sessions.create` and `loadChatHistory` contracts. This is not a current-main bug reproduction because the dedicated create-thread modal is not on main.

## Solution Assessment

No. The branch should be rebuilt around the existing `sessions.create` contract and safer rollback/focus handling, then aligned with the active pinned-chat UX direction before merge.
