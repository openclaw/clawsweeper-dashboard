## Summary

Keep open. Current main still lacks the parallel-thread Control UI flow, so this PR is not obsolete, but the proposed patch is not merge-ready because it bypasses the current `sessions.create` contract and has concrete rollback/error-handling defects.

## What This Changes

The PR branch adds a Control UI New thread button/dialog and app callback, creates a client-generated session via `chat.send`, patches an optional label, and adds modal CSS so users can start a parallel chat without resetting the active session.

## Best Possible Solution

Implement the Control UI feature on top of current `sessions.create`: pass the active session as `parentSessionKey`, send the first task/message through that method, preserve the active transcript and actionable error on failure, add robust modal focus/Escape behavior, and cover success, rollback, and in-flight dismissal with focused UI tests plus a changelog entry.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting `ChatProps`, the toolbar button, and `renderApp` reset wiring; the PR defects are reproducible from the proposed `chat.send` create path and rollback code interacting with current `loadChatHistory`.

## Solution Assessment

No. The dedicated UI is the right direction, but the proposed implementation is not the best fix because the gateway now owns dashboard child-session creation through `sessions.create`, which is narrower and preserves parent linkage, transcript setup, metadata, and initial-run semantics.

  - body: This creates the new thread by sending the first message to a client-generated session key.   - body: The rollback path clears `chatMessages` and stream/tool state before trying to reload the previous session. If the create request fails during a disconnect or the history reload cannot complete, the UI returns to the previous session with an empty transcript even though no reset was requested. Preserve the previous chat state or only replace it after a successful reload.
  - confidence: 0.86
