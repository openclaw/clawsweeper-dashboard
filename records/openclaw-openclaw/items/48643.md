## Summary

Keep open: the branch adds a real Control UI feature that is not implemented on current main, but the proposed implementation still has blocking session-contract and rollback bugs, no contributor-supplied real behavior proof, and UX overlap with the open pinned-chat work.

## What This Changes

The PR adds a Control UI new-thread modal, handler, state wiring, and CSS for creating a parallel chat thread with a selected agent, optional label, and first message.

## Best Possible Solution

Rework the feature into one Control UI parallel-session flow that uses sessions.create with parent linkage, label, and first-message support, preserves existing chat state on failure, and aligns the visible affordance with the pinned-chat direction.

## Reproduction Assessment

Yes for PR review: source inspection of the PR head shows the direct chat.send path and rollback defects, while current main shows the sessions.create and loadChatHistory contracts. This is not a current-main bug reproduction because the dedicated create-thread modal is not on main.

## Solution Assessment

No. The branch should route creation through sessions.create, preserve the previous chat state and actionable failure message on rollback, and settle the visible UX alongside the pinned-chat work before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route thread creation through sessions.create:** `ui/src/ui/app-chat.ts:181-186`
  - body: This constructs a client-side session key, switches local UI state, and calls chat.send directly. Current main's sessions.create owns session row creation, parent linkage, label/model metadata, transcript creation, and optional initial message startup, so this bypasses the established session contract.
  - confidence: 0.91
- **[P2] Restore the previous transcript on create failure:** `ui/src/ui/app-chat.ts:189-198`
  - body: On chat.send failure, this switches back to the previous session but clears messages and stream/tool state before loadChatHistory. If the gateway is disconnected or the reload fails, the user returns to the previous session with an empty conversation.
  - confidence: 0.88
- **[P2] Preserve the create failure message after rollback:** `ui/src/ui/app-chat.ts:205-206`
  - body: After setting the specific create-thread error, the following loadChatHistory call clears lastError on entry and leaves it null when history reload succeeds, so the dialog reports the generic failure instead of the actionable reason.
  - confidence: 0.9
- **[P3] Keep Escape handling active for the dialog:** `ui/src/ui/views/chat.ts:904`
  - body: The Escape handler is attached to the overlay, so it only receives events while focus remains inside a descendant. If focus leaves the dialog, Escape no longer closes or guards it; use a document-level listener or a focus trap while the modal is open.
  - confidence: 0.78
