## Summary

This PR should stay open for contributor revision: the feature is not implemented on current main, but the branch bypasses the current session-creation contract, has rollback defects, and lacks required real behavior proof.

## What This Changes

The PR adds a Control UI modal and app wiring for creating a parallel chat thread from an initial message without resetting the active session.

## Best Possible Solution

Rework this PR or a replacement branch on current main so the new-thread UI uses sessions.create with parentSessionKey, label, and initial message support, preserves rollback state and error text, handles dialog focus/Escape robustly, and includes focused coverage plus contributor-supplied real proof.

## Reproduction Assessment

Yes, for PR review: source inspection of the PR head reproduces the chat.send bypass and rollback defects, while current main source shows the sessions.create and loadChatHistory contracts. This is not a failing current-main bug because the new-thread dialog is not implemented on main.

## Solution Assessment

No. The UI goal is useful, but the proposed path should be rebuilt around current sessions.create helpers rather than client-generating a session key and treating chat.send as the creation API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route thread creation through sessions.create:** `ui/src/ui/app-chat.ts:181-186`
  - body: This creates the new thread by switching to a client-generated session key and calling `chat.send`. Current main's `sessions.create` already owns parent linkage, label/model metadata, transcript creation, and optional first-message startup, so this bypasses the session contract that dashboard-created sessions now use.
  - confidence: 0.9
- **[P2] Restore the previous transcript on create failure:** `ui/src/ui/app-chat.ts:192-198`
  - body: The catch path clears the active transcript and stream state before trying to reload the previous session. If the gateway is disconnected or `chat.history` cannot reload, the user returns to the previous session with an empty transcript after a failed create.
  - confidence: 0.88
- **[P2] Preserve the create failure message after rollback:** `ui/src/ui/app-chat.ts:205-206`
  - body: After setting the specific create-thread error, the following `loadChatHistory` call clears `lastError` on entry and leaves it null when history reload succeeds. The dialog then shows the generic failure message instead of the actionable error.
  - confidence: 0.91
- **[P3] Keep Escape handling active for the open dialog:** `ui/src/ui/views/chat.ts:904`
  - body: The Escape handler is attached to the overlay, so it only sees events while focus remains inside a descendant. If focus leaves the dialog, Escape no longer closes or guards it; use a document-level listener or focus trap while the modal is open.
  - confidence: 0.78
