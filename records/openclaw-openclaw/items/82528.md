## Summary

Keep open: current main still rejects `/active-memory status|on|off` for `main` when `config.agents` is omitted or empty, and the only linked fix PR was closed unmerged. The source-visible symptom is real, but the docs and manifest still describe `config.agents` as an explicit targeting allowlist, so the remaining choice is a maintainer contract decision rather than a safe cleanup close.

## What This Changes

The issue asks Active Memory session slash commands to treat omitted or empty `config.agents` as the default `main` agent instead of reporting the session as off.

## Best Possible Solution

Maintainers should decide the empty-agent contract, then either implement the shared default-`main` gate with tests and docs/manifest updates, or preserve fail-closed targeting and improve setup guidance plus the command response.

## Reproduction Assessment

Partly. Source inspection gives a high-confidence reproduction for the slash-command response with `config.enabled: true`, empty `config.agents`, and agent `main`, but current source does not reproduce the claimed mismatch where per-turn recall continues under that same config.

## Solution Assessment

Unclear. The proposed `[DEFAULT_AGENT_ID]` fallback is the narrow code fix if empty config should mean `main`, but the docs/manifest currently describe an explicit allowlist and need maintainer-approved contract alignment.
