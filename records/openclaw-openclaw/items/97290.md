## Summary

Keep open: current main and v2026.6.10 still leave GitHub Copilot onboarding with no default fallback candidate, and the issue has both a protected `maintainer` label and an open closing PR that should be reviewed before resolution.

## What This Changes

Issue requests a default same-provider GitHub Copilot fallback candidate so idle-timeout failover has another Copilot model to try when no fallback list is configured.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/97291 as the candidate fix, and land a provider-local Copilot fallback default only if maintainers accept the setup-default compatibility tradeoff and explicit user fallback choices remain preserved.

## Reproduction Assessment

Yes for the missing fallback setup path: current source and tests show Copilot onboarding writes no fallback candidate when none is configured, and embedded failover only switches models when configured fallbacks exist. I did not rerun a live Copilot idle-timeout reproduction against current main.
