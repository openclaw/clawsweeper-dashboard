## Label Justifications

- `P2`: This is a normal-priority plugin dispatch performance bug with real first-message latency impact but limited blast radius.
- `merge-risk: 🚨 compatibility`: The PR changes plugin loader cache compatibility, where an overly broad match can reuse the wrong registry across existing runtime configurations.
- `merge-risk: 🚨 security-boundary`: Dropping loadOptions can ignore plugin trust, load-path, and activation changes, which are security-sensitive plugin loading boundaries.

## Summary

Keep open: the performance bug is source-traceable and the PR has useful real-host proof, but the current patch broadens active registry reuse past the loader cache compatibility boundary and should not merge as written.

## What This Changes

This PR adds a standalone runtime registry fallback after strict cache-key misses, adds focused regression tests, and changes the root protobufjs override and lockfile.

## Best Possible Solution

Land one canonical compatibility-aware registry-reuse fix that covers the gateway-startup versus dispatch load-options shape while preserving workspace, plugin-scope, config, trust, activation, and runtime-mode reload boundaries.

## Reproduction Assessment

Yes, source-level: current main dispatch builds lean standalone load options, active lookup enforces strict cache compatibility, and the standalone loader reloads after that miss. I did not rerun the live heap benchmark in this read-only review.

## Solution Assessment

No, not as written. Reusing the warm startup registry is the right direction, but dropping loadOptions after the strict miss is broader than the existing cache contract; the fix should add a compatibility-aware reuse path instead.
