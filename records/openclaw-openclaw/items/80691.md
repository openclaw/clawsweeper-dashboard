## Summary

Keep open: the PR targets a valid source-traceable performance bug, and its real behavior proof is sufficient, but the fallback reuses the active registry after dropping `loadOptions`, which can cross config, trust, activation, load-path, or runtime-mode boundaries.

## What This Changes

The PR adds a standalone runtime registry fallback plus regression tests to reuse a warm active plugin registry after startup-vs-dispatch cache keys diverge, and bumps the `protobufjs` override.

## Best Possible Solution

Keep the warm-registry fast path, but encode a compatibility-aware startup-rich versus dispatch-lean equivalence that preserves config, trust, workspace, plugin-scope, activation, load-path, and runtime-mode reload boundaries.

## Reproduction Assessment

Yes, source-level. Current main shows dispatch building lean runtime load options and the active lookup enforcing compatible registry reuse, while the PR body supplies live patched-container logs for the after-fix performance path; I did not run a live heap profile locally.

## Solution Assessment

No, not as written. Reusing the warm registry is the right direction, but retrying without `loadOptions` is broader than the current cache contract; the safer fix is a compatibility helper for the known startup/dispatch equivalence.
