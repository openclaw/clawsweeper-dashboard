## Summary

Keep open: the performance bug is credible, but this PR is not merge-safe because its fallback drops the existing load-options compatibility guard and can reuse stale plugin runtime state; the unrelated root dependency override also needs maintainer approval or separation.

## What This Changes

The PR adds a fallback in the standalone runtime registry loader to reuse the active registry after a strict cache-key miss, adds focused tests, and bumps the root protobufjs override.

## Best Possible Solution

Keep the warm-registry fast path, but make the startup-rich versus dispatch-lean equivalence compatibility-aware while preserving config, trust, load-path, activation, runtime-mode, workspace, and plugin-scope reload boundaries.

## Reproduction Assessment

Yes, source-level. Current main still shows dispatch building lean runtime load options while the active lookup enforces strict compatibility, and the PR body supplies live patched-container logs for the after-fix performance path; I did not rerun the live heap profile.

## Solution Assessment

No, not as written. Reusing the warm registry is the right direction, but retrying without loadOptions is broader than the current cache contract; a safer fix should encode the known compatible startup/dispatch equivalence and add stale-config regression coverage.

## Review Findings

- **[P2] Preserve load-options compatibility on fallback:** `src/plugins/runtime/standalone-runtime-registry-loader.ts:82-87`
