## Summary

Keep open: this PR targets a valid source-reproducible plugin-registry performance bug, but the fallback added by the patch drops the existing load-options compatibility guard and can reuse stale runtime state; the dependency override also needs to be split out or explicitly approved.

## What This Changes

The PR adds a warm active-registry fallback and regression tests for dispatch-time plugin registry reuse, plus a root protobufjs override bump.

## Best Possible Solution

Keep the warm-registry fast path, but make it compatibility-aware for the known gateway-startup-rich versus dispatch-lean equivalence while preserving config, trust, workspace, plugin-scope, activation, and runtime-mode reload boundaries.

## Reproduction Assessment

Yes, source-level. Current main still shows dispatch building lean runtime load options and the active lookup enforcing strict cache compatibility, while the PR body supplies patched-container runtime logs for the after-fix path.

## Solution Assessment

No, not as written. Reusing a warm startup registry is the right direction, but retrying without loadOptions is broader than the current compatibility contract and needs stale-config/trust regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve load-options compatibility on fallback:** `src/plugins/runtime/standalone-runtime-registry-loader.ts:82-87`
  - body: After the strict cache-key check misses, this lookup retries without `loadOptions`, so an active registry with the same workspace and plugin IDs can satisfy requests whose config, trust, load path, activation, or runtime-mode inputs changed. That crosses the existing cache compatibility boundary and can dispatch with stale plugin runtime state instead of loading for the new options.
  - confidence: 0.91
