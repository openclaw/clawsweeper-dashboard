## Summary

Keep open: the reported dispatch hot-path reload is source-reproducible and the after-fix proof is convincing, but this PR still bypasses the current load-options compatibility boundary after a strict cache miss and mixes in a root dependency override that needs separate maintainer approval.

## What This Changes

This PR reuses the active runtime plugin registry after a standalone loader cache-key miss, adds focused regression tests, and bumps the workspace protobufjs override.

## Best Possible Solution

Keep the warm-registry fast path, but make it compatibility-aware for the specific gateway-startup-rich versus dispatch-lean equivalence while preserving config, trust, load-path, activation, runtime-mode, workspace, and plugin-scope reload boundaries.

## Reproduction Assessment

Yes, source-level. The linked report and PR body give concrete fresh-dispatch steps and live logs, and current main still shows dispatch building lean load options while the active lookup enforces strict compatibility before falling through to loadOpenClawPlugins.

## Solution Assessment

No, not as written. Reusing the warm registry is the right direction, but the fallback should preserve load-options compatibility or add a narrow equivalence helper rather than dropping loadOptions entirely.

## Review Findings
