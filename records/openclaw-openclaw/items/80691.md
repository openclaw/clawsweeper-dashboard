## Summary

## What This Changes

## Best Possible Solution

Land one canonical compatibility-aware registry reuse fix that preserves config, trust, load-path, plugin-scope, workspace, and runtime-mode reload boundaries, with the dependency override split out or explicitly approved.

## Reproduction Assessment

Yes. Current main still shows dispatch building lean runtime load options and strict active lookup returning undefined on cache-key incompatibility, and the PR body supplies live after-fix Docker logs for the performance path.

## Solution Assessment

No as written. Reusing the warm startup registry is the right direction, but the fallback must remain load-options-compatible and should add stale config/trust regression coverage instead of relying only on workspace and plugin IDs.
