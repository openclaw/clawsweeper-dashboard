## Summary

Keep open: the performance bug is valid and the PR now supplies real runtime proof, but the patch still weakens the plugin loader compatibility contract by retrying without `loadOptions`, which can reuse stale runtime state across config, trust, activation, load-path, or runtime-mode changes.

## What This Changes

The branch adds a fallback in standalone runtime registry loading to reuse the active plugin registry after a strict load-options cache-key miss, plus focused regression tests.

## Best Possible Solution

Keep the warm-registry fast path, but make it compatibility-aware for the specific gateway-startup-rich versus dispatch-lean equivalence while preserving config, trust, load-path, activation, runtime-mode, workspace, and plugin-scope reload boundaries.

## Reproduction Assessment

Yes, source-level. The linked issue and PR body give concrete gateway boot/fresh-DM steps, and current main shows `ensureRuntimePluginsLoaded` calling the standalone loader with lean dispatch options that can miss the strict active-registry cache key and fall through to `loadOpenClawPlugins`; I did not run the live heap profile.

## Solution Assessment

No, not as written. Reusing the warm registry is the right direction, but dropping `loadOptions` makes the fallback broader than the current cache contract; the safer fix is a compatibility helper that recognizes the known startup/dispatch equivalence without ignoring other cache-shaping inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve load-options compatibility on fallback:** `src/plugins/runtime/standalone-runtime-registry-loader.ts:82-87`

That bypasses the current cache-key contract and can dispatch with stale plugin runtime state instead of loading the registry for the new options.
