## Summary

Keep open: the performance problem is real and the PR has useful live proof, but the current fallback is broader than the loader cache contract and can reuse stale plugin runtime state after config, trust, load-path, activation, or runtime-mode changes.

## What This Changes

The PR adds a standalone runtime registry fallback after cache-key misses, adds focused regression tests, and changes the root protobufjs override and lockfile.

## Best Possible Solution

Land one canonical compatibility-aware registry-reuse fix that preserves config, trust, workspace, plugin-scope, activation, and runtime-mode reload boundaries, with the dependency override handled separately or explicitly approved.

## Reproduction Assessment

Yes. Current source still shows dispatch using lean load options and the active lookup enforcing strict cache compatibility, and the PR body supplies live after-fix container logs for the performance path.

## Solution Assessment

No, not as written. Reusing the warm registry is the right direction, but retrying without loadOptions is broader than the existing cache contract; the safer fix is an explicit compatibility helper plus regression coverage for stale config and trust changes.
