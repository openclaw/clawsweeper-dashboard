## Summary

Keep open: the PR targets a real current-main reload path and improves on the earlier workspace-unsafe attempt, but it still needs after-fix real behavior proof and the fallback relaxes load-options compatibility too far.

## What This Changes

The PR adds a strict-cache-miss fallback in `ensureStandaloneRuntimePluginRegistryLoaded`, adds three focused regression tests, and adds a changelog entry for the plugin registry reuse performance fix.

## Best Possible Solution

Keep the warm-registry reuse direction, but make the fallback compatibility-aware for the intended gateway-startup versus dispatch option difference and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes for a source-level reproduction: current main shows dispatch can miss strict load-options compatibility and then call `loadOpenClawPlugins`. I did not run the live heap-profile reproduction in this read-only review.

## Solution Assessment

No as written. Reusing the warm registry is the right direction, but the patch should preserve load-options compatibility for config, activation, runtime mode, and discovery differences instead of dropping to workspace plus plugin ids only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve load-options compatibility after the miss:** `src/plugins/runtime/standalone-runtime-registry-loader.ts:82-87`
  - body: The new fallback re-queries `getLoadedRuntimePluginRegistry` without `loadOptions`, so any active registry with the same workspace and required plugin ids is returned even when config, activation inputs, runtime options, or tool-discovery mode differ. Current loader coverage intentionally rejects same-workspace/same-plugin reuse when the load context changes; bypassing that can keep stale plugin runtime state after config or activation changes. Keep this fallback tied to the specific startup/dispatch-compatible inputs, or add the missing compatibility guard and regression test.
