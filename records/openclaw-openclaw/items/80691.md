## Summary

Keep open: the PR targets a valid, source-traceable dispatch performance regression, but the current diff still weakens the active-registry compatibility boundary after a cache-key miss and mixes in a root dependency override change.

## What This Changes

The PR adds a fallback to reuse an active runtime plugin registry after a standalone loader cache-key miss, adds regression tests, and changes the protobufjs workspace override.

## Best Possible Solution

Keep the warm-registry fast path, but make it compatibility-aware for the gateway-startup-rich versus dispatch-lean equivalence while preserving config, trust, load-path, activation, runtime-mode, workspace, and plugin-scope reload boundaries.

## Reproduction Assessment

Yes, source-level. Current main still shows dispatch building lean runtime load options, the active lookup enforcing strict cache compatibility, and the standalone loader falling through to loadOpenClawPlugins after a miss; the PR body supplies after-fix live Docker logs.

## Solution Assessment

No, not as written. Reusing the warm registry is the right direction, but retrying without loadOptions is broader than the current cache contract; a narrower compatibility helper plus stale-config regression coverage is safer.
