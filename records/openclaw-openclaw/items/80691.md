## Summary

Keep open: this targets a real source-traceable plugin-registry performance bug, but the proposed fallback bypasses the loader's load-options compatibility contract and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a standalone runtime loader fallback that reuses an active registry after a strict cache-key miss when workspace and required plugin IDs match, plus regression tests and a changelog entry.

## Best Possible Solution

Implement compatibility-aware reuse for the specific gateway-startup-rich versus dispatch-lean equivalence while preserving config, activation, runtime mode, workspace, plugin ID, load path, and trust changes as reload boundaries, then add regression coverage and after-fix real behavior proof.

## Reproduction Assessment

Yes, source-level. The linked issue gives concrete gateway/fresh-dispatch steps, and current main still shows dispatch calling the standalone runtime loader with lean options that can miss strict compatibility and fall through to loadOpenClawPlugins; I did not run the live heap profile.

## Solution Assessment

No. Reusing the warm registry is the right direction, but this implementation drops loadOptions on the fallback and therefore ignores compatibility inputs that current main intentionally keeps in the loader cache key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve load-options compatibility on fallback:** `src/plugins/runtime/standalone-runtime-registry-loader.ts:82-87`
  - body: After the strict compatibility check misses, this fallback calls `getLoadedRuntimePluginRegistry` again without `loadOptions`, so any active registry with the same workspace and plugin IDs is reused even if the requested config, activation metadata, runtime mode, load paths, or other cache-shaping inputs changed. Current main intentionally includes those fields in the loader cache key, and `active-runtime-registry.test.ts` asserts a config change must miss; with this patch, a dispatch after changing a plugin config/path but keeping the same plugin ID can keep using the stale registry instead of reloading.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or CI changes, but the fallback can keep stale plugin runtime state across security-relevant plugin config, trust, or load-path changes.

Concerns:

- **[medium] Stale registry reuse can ignore plugin trust changes:** `src/plugins/runtime/standalone-runtime-registry-loader.ts:82`
  - body: Because the second lookup omits `loadOptions`, a strict cache-key miss caused by changed plugin config, activation source, load paths, or trust settings can still return the previous active registry when workspace and plugin IDs match. That can continue using old plugin runtime code or settings after a security-relevant configuration change.
