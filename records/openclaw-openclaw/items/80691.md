## Summary

Keep open: the performance problem is source-reproducible and the PR has usable real-host proof, but the current patch still broadens registry reuse by retrying without `loadOptions`, which can cross config, trust, load-path, activation, or runtime-mode compatibility boundaries.

## What This Changes

The branch adds a fallback in the standalone runtime registry loader to reuse an active registry after a strict load-options cache-key miss, plus focused regression tests.

## Best Possible Solution

Keep the warm-registry fast path, but make it compatibility-aware for the startup-rich versus dispatch-lean equivalence while preserving config, trust, load-path, activation, runtime-mode, workspace, and plugin-scope reload boundaries.

## Reproduction Assessment

Yes, source-level. The linked issue and PR body give concrete gateway boot/fresh-DM steps, and current source shows the dispatch ensure path can miss strict active-registry compatibility and fall through to `loadOpenClawPlugins`; I did not rerun the live heap profile.

## Solution Assessment

No, not as written. Reusing the warm registry is the right direction, but the fallback should recognize the specific compatible startup/dispatch option shape instead of ignoring other cache-shaping inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve load-options compatibility on fallback:** `src/plugins/runtime/standalone-runtime-registry-loader.ts:82-87`
  - body: After the strict compatibility check misses, this fallback retries without `loadOptions`, so any active registry with the same workspace and plugin IDs can be reused even when the requested config, activation metadata, runtime mode, trust, or load paths changed. That bypasses the current cache-key contract and can dispatch with stale plugin runtime state instead of loading the registry for the new options.
  - confidence: 0.91
