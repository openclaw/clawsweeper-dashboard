## Summary

Keep open. Current main still has the reported producer/validator mismatch: browser screenshot and labeled snapshot CLI output can still emit `MEDIA:~/...`, while the shared media parser intentionally rejects `~`-prefixed media sources before final channel delivery. The related fix PR was closed unmerged, so the user-facing bug remains actionable.

## What This Changes

The issue requests that browser screenshot media paths be delivered to chat instead of being rejected when the agent emits `MEDIA:~/...` paths.

## Best Possible Solution

Preserve the shared parser's traversal and home-dir rejection, and update machine-readable `MEDIA:` producers to emit real absolute paths while reserving `shortenHomePath` for human-readable display output. Add focused regression tests and a user-facing changelog entry for the browser screenshot/snapshot case, with a small audit of sibling media CLI producers using the same pattern.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: browser inspect output can still emit `MEDIA:~/...`, `splitMediaFromOutput` rejects `~`-prefixed candidates, and final reply parsing depends on that parser before channel delivery.

## Solution Assessment

Yes for a producer-side fix; no for relaxing the shared parser by expanding `~/` inside validation. The safer maintainable path is to stop machine-readable media producers from shortening paths and keep the parser's existing file-read guard intact.
