## Summary

Keep open. Current `main` still has the browser CLI emitting home-shortened machine-readable `MEDIA:` lines while the shared media parser intentionally rejects `~` paths; the linked fix PR is closed unmerged, so the reported delivery failure remains actionable.

## What This Changes

The issue reports that browser screenshots saved under the user home directory are emitted as `MEDIA:~/...`, then rejected by the shared media parser before Telegram can receive the attachment.

## Best Possible Solution

Keep the shared media parser’s home-dir/traversal rejection intact, and update the browser inspect CLI so agent-consumed `MEDIA:` lines use the absolute paths already returned by the browser route while human-facing path summaries can remain shortened.

## Reproduction Assessment

Yes. The source-level path is high-confidence: browser inspect emits `MEDIA:` with `shortenHomePath`, home paths become `~`, final replies call `splitMediaFromOutput`, and current parser tests reject `MEDIA:~/...`.

## Solution Assessment

No for the issue’s preferred parser expansion; the safer and narrower fix is the issue’s producer-side alternative: emit absolute browser screenshot paths for machine-readable `MEDIA:` lines and leave parser rejection unchanged.
