## Summary

Keep open. Current main still has the browser producer/media-parser mismatch: the browser CLI emits machine-readable `MEDIA:` lines through `shortenHomePath`, while the shared media parser deliberately rejects `~`-prefixed paths; the earlier direct fix PR was closed unmerged and a newer related browser media-hint PR is still open.

## What This Changes

The issue requests reliable chat delivery for browser screenshots when a browser screenshot result is emitted as a `MEDIA:~/...` path.

## Best Possible Solution

Keep the parser’s tilde rejection, emit absolute paths for machine-readable browser screenshot/snapshot media hints, and cover the path with focused browser/media regression tests plus a changelog fix entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: browser screenshot output under HOME can become `MEDIA:~/...`, and current parser code/tests reject that exact path shape before outbound delivery.

## Solution Assessment

Yes for a producer-side repair. Emitting absolute browser `MEDIA:` paths is narrower and safer than expanding `~` in the shared parser, which is currently a deliberate guard against home-directory media access.
