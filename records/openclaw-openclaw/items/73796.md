## Summary

Keep open. Current main still has the producer/validator mismatch: browser CLI `MEDIA:` output home-shortens paths, while the shared media parser intentionally rejects `~` paths before channel delivery.

## What This Changes

The report asks OpenClaw to deliver browser screenshot attachments when an agent emits `MEDIA:~/...` instead of showing `Media failed`.

## Best Possible Solution

Emit absolute paths only in machine-readable browser CLI `MEDIA:` lines, keep `shortenHomePath` for human-readable output, preserve parser home-dir rejection, and add focused regression tests plus a changelog fix entry.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current browser CLI code emits `MEDIA:~/...` when paths are under HOME, and current parser tests explicitly reject that form.

## Solution Assessment

No for the issue's preferred parser expansion. The narrower maintainable fix is producer-side absolute browser `MEDIA:` output, because the shared parser's home-dir rejection is a tested security boundary.
