## Summary

Keep open. Current main still has a producer/validator mismatch for browser `MEDIA:` paths: browser CLI screenshot/snapshot output can emit `MEDIA:~/...`, while the shared media parser deliberately rejects home-dir-prefixed paths and strips them from visible text.

## What This Changes

The issue reports that browser screenshot `MEDIA:~/...` paths are rejected by media validation, preventing captured screenshots from being delivered over Telegram.

## Best Possible Solution

Keep the parser’s tilde rejection, make browser screenshot media hints use absolute paths on the producer side, and land focused regression coverage plus the existing changelog entry through the open browser screenshot PR or an amended replacement.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a browser screenshot path under the home directory is shortened to `MEDIA:~/...`, then `parseReplyDirectives`/media parsing rejects and strips that token.

## Solution Assessment

Yes. Producer-side absolute `MEDIA:` output is the narrowest maintainable fix because the parser’s home-dir rejection is an intentional local-file safety boundary.
