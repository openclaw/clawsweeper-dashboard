## Summary

Keep open. Current main still has the browser producer emitting home-dir-shortened machine-readable `MEDIA:` paths while the shared media parser intentionally rejects `~` prefixes before final reply delivery; the related fix PR was closed unmerged, so the user-facing Telegram failure remains actionable.

## What This Changes

The issue requests that browser screenshot media be delivered to chat instead of failing when the agent emits a `MEDIA:~/...` screenshot path.

## Best Possible Solution

Keep the parser's traversal and home-dir rejection intact, and fix the browser plugin producer so only human-facing logs are home-shortened while agent-consumed `MEDIA:` directives use the absolute paths already returned by the browser route.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: browser inspect can emit `MEDIA:~/...`, final replies parse through `splitMediaFromOutput`, and current parser tests confirm `~/...` media candidates are rejected.

## Solution Assessment

Yes. The narrowest maintainable solution is the producer-side browser CLI change; accepting `~` in the shared parser would touch a broader security-sensitive boundary unnecessarily.
