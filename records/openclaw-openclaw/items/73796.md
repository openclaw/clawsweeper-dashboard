## Summary

Keep open. Current main still has the browser CLI emitting machine-readable `MEDIA:` directives through `shortenHomePath`, while the shared media parser still intentionally rejects `~`-prefixed paths; the related fix PR #73798 was closed unmerged, so the reported Telegram/browser screenshot delivery bug remains actionable.

## What This Changes

The issue asks OpenClaw to deliver browser screenshot media when an agent reply includes a `MEDIA:~/...` path produced from browser screenshot output.

## Best Possible Solution

Update the browser CLI producer so only machine-readable `MEDIA:` lines use the already-absolute `result.path` and `result.imagePath` values, while human-readable path summaries may remain home-shortened. Add focused browser CLI regression tests and a user-facing changelog fix entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current browser CLI output can produce `MEDIA:~/...`, and current parser tests verify that `MEDIA:~/...` is rejected before reply media delivery.

## Solution Assessment

Yes. The narrowest maintainable fix is producer-side absolute-path emission in the browser CLI, because the browser route already returns absolute paths and the parser's tilde rejection is an intentional security behavior.
