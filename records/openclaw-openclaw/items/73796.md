## Summary

Keep open. Current main still has the producer/validator mismatch for browser CLI screenshot media: machine-readable `MEDIA:` lines shorten HOME to `~`, while the shared media parser intentionally rejects `~` paths and strips invalid local media tokens.

## What This Changes

The issue reports that browser screenshot delivery fails when a browser screenshot result is surfaced as `MEDIA:~/...`, because the outbound media parser rejects the tilde-prefixed local path.

## Best Possible Solution

Emit absolute paths for browser CLI machine-readable `MEDIA:` lines, keep `shortenHomePath` only for human-readable output, and preserve the shared parser's tilde rejection.

## Reproduction Assessment

Yes. The source-level reproduction is a browser screenshot or AI snapshot saved under HOME: `shortenHomePath` makes the CLI print `MEDIA:~/...`, and `splitMediaFromOutput` rejects and strips that token.

## Solution Assessment

Yes. A producer-side repair is the narrowest maintainable fix because the route already returns absolute paths and parser tests show `~` rejection is an intentional local-file safety guard.
