## Summary

The report matches a real class of prior inter-session loop bugs, but current main intentionally supports a finite reply-back exchange and already suppresses exact control replies; without the offered transcripts, there is not yet a high-confidence current-main reproduction of an unbounded loop or unrelated webchat output being routed.

## What This Changes

Investigate and prevent agent output in a webchat session from being automatically echoed between sessions after a `sessions_send` delivery.

## Best Possible Solution

Reproduce from a redacted two-session transcript, then fix the central `sessions_send` A2A flow only if the trace proves a turn-limit, control-token, or provenance-boundary failure, with regression coverage for that exact exchange.

## Reproduction Assessment

No, not for the alleged infinite or unrelated-output routing behavior. Source inspection reproduces the intended bounded A2A reply loop, but the missing transcripts are needed to show that current main exceeds the configured cap, ignores an exact stop token, or routes text outside that flow.

## Solution Assessment

No, the proposed webchat filter, `LOCAL_ONLY` flag, and per-agent config are not yet the best fix. The narrow maintainable solution is to correct the existing central A2A loop after the failing exchange is identified, without adding another routing/config surface.
