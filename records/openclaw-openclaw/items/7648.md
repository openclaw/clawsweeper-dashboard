## Summary

Keep open. Current main still lacks the requested Control UI/WebChat auto-scroll preference or Scroll to Top action: browser chat scrolling remains bottom-oriented with only manual scroll-up preservation, native WebChat still pins to the bottom, and neither UI settings nor gateway config expose a scroll mode.

## What This Changes

Add a Control UI/WebChat option or action to prevent long agent responses from forcing the viewport to the bottom.

## Best Possible Solution

Add a UI-level chat scroll preference and/or top-jump action for browser Control UI and native WebChat, preserving the existing manual scroll-up behavior, bottom/new-message indicator semantics, initial load behavior, send behavior, refresh behavior, and streaming behavior. Keep TUI-specific scroll work in #44130 unless maintainers deliberately merge the UX designs.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send or load a chat in current main and the browser path calls resetChatScroll/scheduleChatScroll toward the bottom, while native WebChat pins scrollPosition to the bottom; no UI or config field can disable that behavior.

## Solution Assessment

Yes for keeping this open and routing a focused UI fix. The narrowest maintainable path is a persisted UI preference/action first, with gateway-level config added only if maintainers want operator policy rather than per-user UI behavior.
