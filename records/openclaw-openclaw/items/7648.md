## Summary

Keep open. Current main still lacks a Control UI/WebChat auto-scroll preference or Scroll to Top action; browser and native chat paths remain bottom-oriented with only manual scroll-up preservation, and the related TUI issue does not supersede this request.

## What This Changes

Request adding a Control UI/WebChat option or affordance to stop bottom auto-scroll so long responses can be read from the top.

## Best Possible Solution

Add a persisted UI-level chat scroll preference for browser Control UI and native WebChat, preserving manual-scroll, initial-load, send, streaming, refresh, and new-message indicator behavior; leave gateway-level config out unless maintainers request operator policy.

## Reproduction Assessment

Yes. This is a feature request rather than a broken contract, but current-main source inspection clearly reproduces the missing capability: the browser and native chat paths bottom-follow and no UI setting, gateway config field, or Scroll to Top action exists.

## Solution Assessment

Yes for keeping this open and routing a focused UI feature. The narrowest maintainable path is a per-user UI preference/action first, with gateway-level config added only after maintainer direction.
