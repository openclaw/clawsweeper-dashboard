## Summary

Keep open. Current main still lacks the session-store hardening because `updateSessionStoreAfterAgentRun` merges the full stale in-memory entry over the latest disk row, while this PR still needs repair or narrowing before it can merge cleanly.

## What This Changes

The PR bundles session-store terminal-status preservation with Mattermost DM threading, generic tool-loop escalation, MiniMax usage parsing, Xiaomi reasoning payload handling, and regression tests.

## Best Possible Solution

Land a narrow repaired or replacement PR that removes stale `status` from the after-run merge patch, adds the terminal-status regression test, and keeps unrelated provider/channel changes out unless they are separately fixed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pre-seed the disk store with `status: "done"`, `endedAt`, and `runtimeMs`, pass a stale in-memory row with `status: "running"` to `updateSessionStoreAfterAgentRun`, and observe the current merge can restore `running`.

## Solution Assessment

No, not as a whole PR. The session-store patch is the right narrow direction, but the branch combines unrelated changes and leaves the MiniMax nested `usage_percent` candidate path broken; a repaired or narrowed branch is safer.
