## Summary

Current main and v2026.6.6 already solve the central execution problem with first-class command cron jobs plus conservative doctor conversion and warnings for recognizable shell-style agent turns. A meaningful unique remainder remains: manifest-grade ownership, purpose, mutation, approval, and rollback metadata is absent from the persisted cron/API contract and requires a maintainer product-boundary decision.

## What This Changes

Add governance metadata to deterministic command cron jobs, flag narrow deterministic agent-turn jobs, and expose manifest-ready fields through cron list JSON.

## Best Possible Solution

Keep the shipped command-job and conservative doctor paths, then define a minimal generic governance contract only if maintainers confirm multiple OpenClaw consumers need it; otherwise support-control should own and join its manifest metadata against stable cron job IDs.

## Reproduction Assessment

Yes, at source level: current main clearly shows the shipped command-job and doctor behavior, while the cron contracts lack the five requested governance fields. This is a contract-gap verification rather than a failing runtime reproduction.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. Adding fields directly to core is plausible, but an external manifest keyed by cron job ID is narrower if the semantics are specific to support-control.
