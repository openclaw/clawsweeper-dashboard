## Summary

Keep open. Current main and the latest release still use forgeable sentinel text for inbound metadata stripping, and two open closing PRs already propose different trusted-contract shapes for this issue.

## What This Changes

The issue requests a trusted inbound-decoration marker and structured bare-body contract so OpenClaw and plugin consumers can strip, replay, export, and deduplicate decorated inbound user messages without parsing sentinel text.

## Best Possible Solution

Land one maintainer-approved additive contract that marks decorated inbound user turns out of band, preserves the host-known bare body for trusted consumers, keeps legacy sentinel fallback for old rows, and closes this issue through the merged PR.

## Reproduction Assessment

Yes from source inspection, though not from a live failing run in this read-only review. Current main decorates inbound content as sentinel text and strips user-role text by those same literal headers, so a user-authored sentinel-shaped body follows the same code path as host decoration.

## Solution Assessment

Unclear until maintainers choose the durable contract shape. A trusted out-of-band marker plus bare body is the right class of solution, but the permanent field location and redaction/upgrade boundaries should be settled through one linked PR.
