## Summary

Keep open. Current main and the latest release still rely on forgeable inbound metadata text stripping, while the two PRs that would add a trusted contract are still open and need maintainer/API-shape review before this issue can close.

## What This Changes

The issue requests a trusted out-of-band inbound-decoration marker and bare-body contract so consumers do not strip or deduplicate user-authored text via forgeable sentinel strings.

## Best Possible Solution

Define one host-owned trusted inbound-decoration contract on persisted/runtime user turns, update UI/history/CLI/memory/replay consumers to prefer it, and keep sentinel stripping only as a legacy fallback for old rows.

## Reproduction Assessment

Yes. Source inspection shows a user-role message containing an exact inbound metadata sentinel followed by a JSON fence follows the same strip path as injected decoration in UI extraction, CLI dedupe, memory corpus, and historical replay consumers.

## Solution Assessment

Unclear which open PR is the best final implementation. The trusted out-of-band marker and bare-body direction is the right layer, but maintainers need to choose the permanent metadata shape and legacy fallback boundary.
