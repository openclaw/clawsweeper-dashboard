## Label Justifications

- `P2`: This is a real content-preservation and session-history correctness bug with active fix candidates, but it is not an emergency runtime outage.
- `impact:data-loss`: Forgeable sentinel stripping can silently remove genuine user-authored text or collapse distinct turns during dedupe.
- `impact:session-state`: The affected paths include history replay, transcript/UI normalization, memory export, and dedupe behavior that shape session state.

## Summary

Keep open. Current main and the latest release still use forgeable sentinel text for inbound metadata stripping, and two linked open PRs propose competing trusted-contract shapes for the same cross-consumer message API.

## What This Changes

Request a trusted out-of-band inbound message marker plus bare body so UI, history, memory, replay, and plugin consumers do not strip or dedupe user content by parsing sentinel text.

## Best Possible Solution

Land one maintainer-approved additive trusted inbound-decoration contract that marks decorated user turns out of band, exposes the host-known bare body to trusted consumers, and keeps legacy sentinel stripping only for old unmarked rows.

## Reproduction Assessment

Yes, from source inspection. Current main decorates inbound user content with literal sentinel text and strips or compares user-role content by the same literal headers, so a user-authored sentinel-shaped body follows the same path as host decoration.

## Solution Assessment

Unclear until maintainers choose the durable contract shape. A trusted out-of-band marker plus bare body is the right class of solution, but the field location, marker semantics, legacy fallback, and redaction boundary should be settled through one linked PR.
