## Label Justifications

- `P2`: The report describes a real content-integrity and plugin-contract problem with bounded blast radius, but it needs contract design rather than urgent runtime recovery.
- `impact:data-loss`: A user-authored message that matches the decorated block shape can be stripped or false-deduplicated, silently dropping real user content.
- `impact:session-state`: The issue affects how current and historical user turns are normalized for replay, memory, and downstream session consumers.

## Summary

Keep open. Current main and the latest release still decorate inbound user content as forgeable text blocks, strip them with literal sentinel heuristics, and expose no trusted decorated marker or bare-body contract for core or plugin consumers.

## What This Changes

Add a trusted out-of-band inbound-decoration marker and bare-body field so core and plugin consumers can strip and deduplicate without parsing sentinel text.

## Best Possible Solution

Define one host-owned inbound decoration contract on current and persisted user turns, expose the bare body to plugin/UI/memory consumers, and keep sentinel stripping only as a legacy path for old transcript rows.

## Reproduction Assessment

Yes. Source inspection shows a user message containing an exact inbound metadata sentinel followed by a JSON fence follows the same strip path as injected decoration; I did not run a live channel repro in this read-only review.

## Solution Assessment

Yes as a direction, but not as an automation-ready patch. A trusted out-of-band marker and bare-body field is the right layer; the exact storage, plugin API, and legacy fallback shape needs maintainer design.
