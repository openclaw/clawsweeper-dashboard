## Summary

Keep open: current main and v2026.6.9 still use forgeable sentinel-text stripping for inbound decoration, and two open closing PRs are active fix candidates rather than merged implementations.

## What This Changes

The issue requests a trusted out-of-band inbound-decoration marker and bare-body contract so core and plugin consumers do not strip or dedupe user-authored text by matching forgeable metadata headers.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land one marker-gated trusted inbound-decoration contract, preserve legacy sentinel stripping only for unmarked historical rows, and close or supersede the competing candidate PR.

## Reproduction Assessment

Yes. Source inspection shows a user message containing exact inbound metadata sentinel lines plus fenced JSON follows the same `stripInboundMetadata` paths used by UI extraction, CLI dedupe, memory export, and historical replay; I did not run tests because this review is read-only.

## Solution Assessment

Yes as a direction, but the exact PR shape is still a maintainer decision. The durable solution should be a host-owned marker plus bare body that consumers trust before falling back to sentinel parsing for legacy rows.
