## Summary

Keep open. Current main and the latest release still use forgeable inbound-metadata text stripping with no trusted bare-body marker, and two open linked PRs are candidate fixes rather than merged implementations.

## What This Changes

The issue requests an out-of-band trusted marker and bare-body contract for inbound-decorated user messages so UI, replay, memory, and plugin consumers do not strip or deduplicate user-authored sentinel-looking text by parsing forgeable content.

## Best Possible Solution

Land one maintainer-approved marker-gated trusted inbound-decoration contract, preserve legacy sentinel stripping only for unmarked historical rows, and close this issue after the chosen PR merges.

## Reproduction Assessment

Yes. Source inspection shows an exact inbound metadata sentinel followed by a JSON fence is stripped from user-role text by the same path used for host decoration; no trusted field exists on current main to distinguish the two.

## Solution Assessment

Yes as a direction, but not as a completed solution yet. A trusted out-of-band marker plus bare-body field is the right layer; maintainers still need to choose and land the durable storage/API shape through one of the linked PRs.
