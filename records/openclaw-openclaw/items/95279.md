## Label Justifications

- `P2`: The issue describes real content-integrity risk across history and plugin consumers, but the fix is a bounded contract design rather than an urgent runtime outage.
- `impact:data-loss`: A user-authored decorated-looking block can be stripped or false-deduplicated, silently dropping real user content.
- `impact:session-state`: The requested contract affects how current and historical user turns are normalized for replay, UI history, memory, and downstream consumers.

## Summary

Keep open. Current main and the latest release still use forgeable sentinel text to identify inbound decoration, while no trusted decorated marker or bare-body field exists for plugin, UI, history, or memory consumers.

## What This Changes

The issue requests a host-owned inbound-decoration contract that marks decorated user turns and carries the bare body out of band from user-controlled message text.

## Best Possible Solution

Add a narrow host-owned inbound-decoration contract on current and persisted user turns, expose a trusted bare body or marker to plugin/UI/memory consumers, and keep text-shape stripping only as a legacy fallback for old rows.

## Reproduction Assessment

Yes, from source inspection: exact inbound sentinel headers followed by json fences are stripped by stripInboundMetadata, and there is no trusted out-of-band marker to distinguish user-authored text from injected decoration. I did not run a live channel or plugin reproduction in this read-only review.

## Solution Assessment

Yes as a direction, but not as an automation-ready patch. A host-owned out-of-band marker or bare-body field is the right layer; maintainers still need to choose the message, persistence, plugin API, and legacy fallback shape.
