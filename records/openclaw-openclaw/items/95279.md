## Label Justifications

- `P2`: The issue describes a real content-integrity/session problem, but the fix is a bounded contract design rather than an urgent core outage.
- `impact:data-loss`: A user-authored decorated-looking message can be stripped or false-deduplicated, silently dropping real user content.
- `impact:session-state`: The issue affects how current and historical user turns are normalized for replay, UI history, memory, and downstream consumers.

## Summary

Keep open. Current main and v2026.6.8 still expose sentinel-based inbound metadata stripping to UI/history/memory consumers, and the existing runtime-only context split does not provide the trusted decorated-marker plus bare-body contract requested here.

## What This Changes

The issue asks OpenClaw to add a trusted out-of-band inbound decoration marker and bare-body contract so consumers do not strip or deduplicate by forgeable sentinel text.

## Best Possible Solution

Add a narrow host-owned inbound-decoration contract on current and persisted user turns, expose a trusted bare body or marker to plugin/UI/memory consumers, and keep sentinel stripping only as a legacy path for old transcript rows.

## Reproduction Assessment

Yes, from source inspection: a user message containing an exact inbound metadata sentinel followed by a JSON fence takes the same path as injected decoration through `stripInboundMetadata`. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Yes as a direction, but not as an automation-ready patch: an out-of-band marker and bare-body field is the right layer, while the exact message, persistence, plugin API, and legacy fallback contract needs maintainer design.
