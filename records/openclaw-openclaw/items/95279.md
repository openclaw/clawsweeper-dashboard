## Label Justifications

- `P2`: The issue describes real content-integrity and session-state risk, but the fix is a bounded contract design rather than an urgent outage.
- `impact:data-loss`: A user-authored decorated-looking message can be stripped or false-deduplicated, silently dropping real user content.
- `impact:session-state`: The issue affects how current and historical user turns are normalized for replay, UI history, memory, and downstream consumers.

## Summary

Keep open. Current main still exposes only text-shaped inbound decoration plus sentinel-based stripping, and no trusted decorated-marker or bare-body contract exists for UI, history, memory, or plugin consumers.

## What This Changes

Request a trusted out-of-band inbound decoration marker and bare-body contract so consumers do not strip or dedupe user text by forgeable sentinel headers.

## Best Possible Solution

Design and land a host-owned inbound decoration contract that carries a trusted marker plus bare body or bare-body hash, then update UI/history/memory/plugin consumers to prefer it and retain sentinel stripping only for legacy rows.

## Reproduction Assessment

Yes, at source level: a user message containing an exact inbound metadata sentinel followed by a json fence takes the same path as injected decoration through stripInboundMetadata. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Yes as a direction, but not as an automation-ready patch: an out-of-band marker and bare-body field is the right layer, while maintainers still need to choose the message, persistence, plugin API, and legacy fallback contract.
