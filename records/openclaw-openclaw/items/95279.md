## Label Justifications

- `P2`: This is a real content-preservation and session-history correctness bug with active fix candidates, but it is not an emergency outage or crash-loop.
- `impact:data-loss`: Forgeable sentinel stripping can silently remove genuine user-authored text or collapse distinct turns during dedupe.
- `impact:session-state`: The affected paths include transcript display, historical replay, memory export, and imported-session dedupe state.

## Summary

Keep open. Current main and the latest release still depend on forgeable inbound-metadata sentinel text, while two open linked PRs are competing implementation candidates for the trusted contract.

## What This Changes

Request a trusted out-of-band inbound-decoration marker plus bare user body so UI, history, memory, replay, and CLI dedupe consumers do not strip or dedupe user text by matching forgeable sentinel strings.

## Best Possible Solution

Land one maintainer-approved additive, marker-gated trusted inbound-decoration contract, update all shared consumers to prefer it, and keep sentinel stripping only as a legacy fallback for unmarked historical rows.

## Reproduction Assessment

Yes. Source inspection shows a user-authored sentinel-plus-fenced-JSON block follows the same stripInboundMetadata path as host decoration in UI, CLI dedupe, memory export, and replay; no write-producing live test was run in this read-only review.

## Solution Assessment

Yes on direction, but unclear on final field shape. A trusted marker plus bare body is the right class of fix, while maintainers still need to select one durable contract shape and verify redaction and upgrade boundaries.
