## Summary

Keep open. Current main and v2026.6.10 still rely on literal inbound-metadata sentinel text for stripping and comparable text, while two unmerged closing PRs propose competing trusted-contract shapes for the same message/session boundary.

## What This Changes

Request a trusted out-of-band inbound-decoration marker and bare body so UI, history, replay, memory, and plugin consumers do not strip or dedupe by forgeable text headers.

## Best Possible Solution

Land one maintainer-approved additive trusted inbound-decoration contract, keep legacy sentinel fallback only for old unmarked rows, and let the merged closing PR close this issue.

## Reproduction Assessment

Yes, from source inspection. Current main decorates user content with literal sentinel headers and the UI/history/memory/replay consumers strip user-role text by those same headers, so sentinel-shaped user text can follow the same code path as host decoration.

## Solution Assessment

Unclear for the exact field shape. A trusted out-of-band marker plus trusted bare body is the right class of fix, but the permanent message metadata location and redaction/upgrade behavior should be decided through one linked PR.
