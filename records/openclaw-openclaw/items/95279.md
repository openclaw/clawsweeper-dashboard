## Summary

Keep open. Current main and the latest release still use forgeable sentinel text for inbound metadata stripping/dedupe, and two open closing PRs already propose competing trusted-contract shapes for the same session-message boundary.

## What This Changes

The issue requests a trusted out-of-band inbound-decoration marker and bare-body contract so consumers do not strip or dedupe user text by matching forgeable sentinel headers.

## Best Possible Solution

Choose one additive, host-owned message metadata contract and land it through the canonical PR so literal sentinel-shaped user text is preserved across transcript persistence, UI display/extraction, import dedupe, memory export, and historical replay while legacy rows keep their fallback behavior.

## Reproduction Assessment

Yes. Source inspection shows a user-role message containing an exact inbound sentinel line followed by a fenced JSON block will be passed to `stripInboundMetadata`, which removes that block without any trusted out-of-band marker check.

## Solution Assessment

Yes on the fix layer, but unclear on the final contract shape. A trusted out-of-band marker and bare-body value is the maintainable solution, while the specific top-level versus `__openclaw` metadata shape needs maintainer selection between the open PRs.
