## Label Justifications

- `P2`: The report describes a real content-integrity bug, but the fix is a bounded contract/design change rather than an urgent broken core runtime.
- `impact:data-loss`: A user-authored decorated-looking message can be stripped or false-deduplicated, silently dropping real user content.
- `impact:session-state`: The issue affects how current and historical user turns are normalized for replay, memory, and downstream session consumers.

## Summary

Keep open. Current main still strips inbound decoration by forgeable sentinel text, several core/UI/memory consumers depend on that stripper, and the message/plugin-facing types do not expose a trusted decorated marker or bare-body field.

## What This Changes

Request a trusted out-of-band inbound decoration marker and bare-body field so core and plugins can strip or deduplicate decorated inbound user messages without parsing sentinel text.

## Best Possible Solution

Add a narrow host-owned inbound decoration contract on current and persisted user turns, expose the bare body to plugin/UI/memory consumers, and keep sentinel stripping only as a legacy path for old transcript rows.

## Reproduction Assessment

Yes, from source inspection: a user message containing an exact inbound metadata sentinel followed by a json fence takes the same path as injected decoration through stripInboundMetadata. I did not run a live channel repro in this read-only review.

## Solution Assessment

Yes as a direction, but not as an automation-ready patch: a trusted out-of-band marker and bare-body field is the right layer, while the exact storage/plugin contract needs maintainer design.
