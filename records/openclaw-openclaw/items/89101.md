## Summary

Keep open for maintainer review. Current main and the latest release still expose only terminal-looking session `status` values, while this PR is a focused additive fix with sufficient screenshot proof; the remaining decision is whether maintainers accept `resumable` as model-visible session-routing metadata for all non-running states, including killed sessions.

## What This Changes

Adds a `resumable?: boolean` session-list field, passes it from gateway session rows into the agent `sessions_list` output, updates model-facing guidance, and refreshes focused tests plus prompt snapshots.

## Best Possible Solution

Land the additive signal after maintainer confirmation that `resumable` is the canonical `sessions_list` contract, and align any public client/docs row types in the same PR if maintainers treat it as gateway metadata.

## Reproduction Assessment

Yes at the source-contract level: current main exposes terminal-looking statuses while `sessions_send` reactivation accepts ended subagent runs, which explains the duplicate-spawn failure mode. I did not run a deterministic LLM repro in this read-only review, but the PR screenshots show the after-fix same-session follow-up path.

## Solution Assessment

Yes, this is an acceptable narrow fix if maintainers accept the contract: additive metadata avoids breaking existing `status` consumers and matches the current reactivation path. The main alternative is a broader lifecycle/status redesign, which is more invasive than this bug fix.
