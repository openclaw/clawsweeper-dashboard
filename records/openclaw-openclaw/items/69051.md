## Summary

Keep this PR open, but do not merge the current head as-is. Current main still lacks the source/plain TTS metadata and provider-capability routing this PR is trying to add, while the latest PR head has two concrete fallback-routing defects in the new preflight path.

## What This Changes

The PR adds TTS source/plain reply metadata, speech-provider capability types and SDK exports, ElevenLabs v3 expressive-tag capability metadata, and provider-specific speech-text routing through the speech-core fallback loop with tests.

## Best Possible Solution

Keep the PR open and fix the preflight route handling so resolveFirstRoutableText carries the selected provider and resolved providerConfig forward, then use that selected route consistently for initial text-tier decisions and fallback-safe synthesis setup. After that, validate the speech-core tests plus SDK/API and changed-lane gates before landing it ahead of the dependent stack.

## Reproduction Assessment

Yes. A focused regression test can configure a broken primary TTS provider and a ready fallback provider, then call maybeApplyTtsToPayload with source/plain metadata; the current PR head either returns text-only after resolving effectiveProvider or can throw while recomputing initialRouteUsesExpressiveText. I did not run tests because this review was read-only.

## Solution Assessment

No, not as-is. The capability-layer direction is a reasonable narrow foundation, but the implementation should reuse the provider/config selected during preflight instead of re-resolving the primary provider after fallback routing has already succeeded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the preflight-selected provider for initial config:** `extensions/speech-core/src/tts.ts:1842-1849`
  - body: `resolveFirstRoutableText` can select a fallback provider after the primary provider's config resolution fails, but this block then resolves `effectiveProvider` again and returns `nextPayload` on failure. That means TTS is skipped even though a fallback provider was already proven routable; carry the selected provider/config out of preflight or resolve `initialRoute.provider` here.
  - confidence: 0.88
- **[P2] Reuse resolved config when checking the routed text tier:** `extensions/speech-core/src/tts.ts:1912-1914`
  - body: `initialRouteUsesExpressiveText` calls `resolveRoutedSpeechText` without the provider config resolved during preflight, so it can re-enter provider config resolution and throw after the preflight loop already handled that failure. Reuse the selected route's providerConfig or guard this check so a bad primary provider cannot abort reply handling.
  - confidence: 0.84
