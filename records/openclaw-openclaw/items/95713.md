## Summary

Keep open: the PR targets the right timeout/overflow compaction call sites, but it switches the compaction provider/model without carrying the matching live auth profile, so cross-provider `/model` switches can still compact with stale credentials. The external PR also provides only mocked Vitest proof, not real behavior proof.

## What This Changes

The PR adds a regression test and changes timeout/overflow compaction in `run.ts` to use the pending live `/model` selection's provider/model instead of the run-start provider/model.

## Best Possible Solution

Carry the live selection as one provider/model/auth-profile unit into both timeout and overflow compaction contexts, add focused regression coverage for auth-profile changes, and require redacted runtime proof for `/model` switching followed by compaction.

## Reproduction Assessment

Yes, source-reproducible: current main sets `requestedSelection` from a pending live `/model` switch, but timeout and overflow compaction still use stale run-start context when the attempt cannot restart cleanly. I did not run a live multi-provider compaction session in this read-only review.

## Solution Assessment

No as submitted: the changed surface is the right narrow place for provider/model routing, but the fix must carry or clear the matching auth profile too. The PR also needs real behavior proof beyond mocked Vitest output.
