## Summary

Keep this open: current main still routes spawned Codex startup through the shared app-server client and can clear that singleton on startup failure, while PR #72812 remains the open implementation candidate for the same isolation bug.

## What This Changes

Invalid or expired bearer-token failures from `btw spawn codex 5.5` should fail the helper spawn without disconnecting the active gateway/session.

## Best Possible Solution

Land PR #72812, or an equivalent narrow Codex app-server fix, so spawned/helper runs use isolated clients while normal Codex runs keep shared-client reuse.

## Reproduction Assessment

Yes. The issue provides concrete invalid-token steps and logs, and current main shows a source path where a spawned Codex `thread/start` failure can use and then clear the shared app-server client.

## Solution Assessment

Yes. Isolating spawned/helper Codex app-server clients and scoping shared-client teardown to runs that own the singleton is narrower than changing gateway restart behavior or global session lifecycle.
