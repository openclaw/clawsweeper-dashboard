## Summary

Keep this open. Current main still routes Codex app-server startup through the shared client factory and clears the shared singleton on startup errors, while PR #72812 is the open, unmerged fix vehicle for this issue.

## What This Changes

The issue reports that `btw spawn codex 5.5` with an invalid or expired bearer token returns a 401 and then disconnects the active gateway/session, requiring `openclaw gateway restart`.

## Best Possible Solution

Review and land PR #72812, or an equivalent narrow fix, so spawned/helper Codex app-server startup uses an isolated client while ordinary Codex runs keep the shared-client lifecycle and cleanup behavior.

## Reproduction Assessment

Yes. The reporter provided concrete steps and logs, and the current source path shows an auth failure during `thread/start` can flow through the startup catch that clears the shared client.

## Solution Assessment

Yes in direction. Isolating only spawned/helper Codex startup is narrower than changing the global shared-client contract, but the fix should land through PR #72812 or an equivalent reviewed patch.
