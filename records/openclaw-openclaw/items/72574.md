## Summary

Keep this open. Current main still uses the shared Codex app-server client for spawned/helper startup and still clears the shared singleton on startup failure, while the open linked PR already targets this exact isolation bug.

## What This Changes

The issue reports that `btw spawn codex 5.5` with an invalid or expired Codex bearer token returns 401 and then destabilizes the active gateway/session instead of failing locally.

## Best Possible Solution

Keep this issue open until PR #72812, or an equivalent narrow Codex app-server change, lands and verifies that spawned/helper runs use isolated clients and do not clear the shared singleton on child auth failure.

## Reproduction Assessment

Yes. The report includes concrete steps and logs, and current source shows a clear path where a `thread/start` 401 can throw through `runCodexAppServerAttempt` startup cleanup and clear the shared client.

## Solution Assessment

Yes. Isolating only spawned/helper Codex app-server clients is the narrow maintainable direction because normal Codex runs can retain shared-client reuse while child auth failures stop at the child boundary.
