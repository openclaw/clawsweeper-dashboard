## Summary

Keep this open because current main still routes spawned Codex app-server startup through the shared-client path that can clear the singleton on startup failure, and PR #72812 is an open, unmerged implementation candidate for the same issue.

## What This Changes

Invalid or expired Codex bearer tokens during `btw spawn codex 5.5` should fail gracefully without disconnecting the active gateway session.

## Best Possible Solution

Review and land PR #72812, or an equivalent narrow Codex app-server change, so spawned/helper runs use isolated clients while normal Codex runs keep shared-client reuse.

## Reproduction Assessment

Yes, source-backed rather than live-observed. The report gives concrete invalid-token steps and logs, and current main still lets spawned Codex startup use the shared app-server singleton before cleanup can clear it.

## Solution Assessment

Yes. Isolating spawned/helper Codex app-server clients and limiting shared-client teardown to runs that own the singleton is narrower than changing gateway restart behavior or global session lifecycle.
