## Summary

Keep this open. Current main still lets spawned/helper Codex app-server startup use the shared client path and still clears the shared singleton on startup failure, while the linked fix PR is open and unmerged with closing syntax for this issue.

## What This Changes

The issue reports that `btw spawn codex 5.5` with an invalid or expired bearer token returns 401 and then disconnects the active gateway session until restart.

## Best Possible Solution

Review and land PR #72812, or an equivalent narrow change, so spawned/helper Codex app-server startup uses an isolated client and only cleans up that isolated client on auth failure while ordinary Codex runs keep the shared singleton behavior.

## Reproduction Assessment

Yes. The issue includes concrete steps and logs, and current main has a source-level reproduction path where a `thread/start` 401 during spawned/helper startup can flow into shared-client cleanup.

## Solution Assessment

Yes. Isolating only spawned/helper app-server clients is the narrow maintainable fix because it preserves the shared-client contract for normal Codex runs while preventing child auth failures from tearing down the active session.
