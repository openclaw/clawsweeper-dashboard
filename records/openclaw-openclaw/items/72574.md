## Summary

Keep this open. Current main still routes Codex app-server startup through the shared client and clears that shared singleton on startup errors, and the linked closing PR is still open/unmerged.

## What This Changes

The issue requests that invalid or expired bearer-token failures from auxiliary Codex spawn/helper runs fail locally without disconnecting the active gateway/session.

## Best Possible Solution

Land PR #72812 or an equivalent narrow Codex plugin fix that uses an isolated app-server client for spawned/helper lineage, keeps ordinary runs on the shared client, and limits shared-client teardown to non-isolated startup failures.

## Reproduction Assessment

Yes. The issue includes concrete steps and logs, and current main has a source-level reproduction path where a spawned/helper `thread/start` 401 can propagate through shared app-server startup and clear the shared singleton.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix because it isolates spawned/helper startup failures inside the Codex plugin without changing the broader gateway lifecycle or normal shared-client contract.
