## Summary

Keep this open because current main still has the reported failure isolation gap: Codex app-server startup uses the shared client path and clears that shared singleton on startup failure. The linked PR #72812 is open, unmerged, and explicitly targets this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that invalid or expired bearer-token failures from `btw spawn codex 5.5` fail locally without disconnecting or requiring restart of the active OpenClaw gateway/session.

## Best Possible Solution

Land PR #72812, or an equivalent narrow change, so spawned/helper Codex app-server startup uses an isolated client and only cleans up that isolated client on auth failure while normal Codex runs keep the shared singleton behavior.

## Reproduction Assessment

Yes. The issue includes concrete user steps and logs, and current main has a source-level path where a `thread/start` 401 from a spawned/helper run can bubble into startup cleanup that clears the shared client.

## Solution Assessment

Yes. Isolating only spawned/helper Codex app-server clients is the narrow maintainable fix because it preserves the shared-client contract for ordinary Codex runs while preventing child auth failures from tearing down the live session.
