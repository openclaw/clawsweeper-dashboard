## Summary

Keep this open. Current main still routes Codex app-server startup through the shared client and clears that singleton on startup failure, while PR #72812 is an open, unmerged fix vehicle linked with closing syntax.

## What This Changes

The issue asks OpenClaw to keep the active gateway/session stable when `btw spawn codex 5.5` fails with an invalid or expired Codex bearer token.

## Best Possible Solution

Review and land PR #72812, or an equivalent narrow change, so spawned/helper Codex app-server startup uses an isolated client and only that isolated client is cleaned up on auth failure while normal Codex runs keep using the shared singleton.

## Reproduction Assessment

Yes. The issue includes concrete user steps and logs, and current main has a source-level reproduction path where a `thread/start` 401 from a spawned/helper run can propagate into shared app-server startup cleanup.

## Solution Assessment

Yes. Isolating only spawned/helper app-server clients is the narrow maintainable fix because it preserves the shared-client contract for ordinary Codex runs while preventing child auth failures from tearing down the live session.
