## Summary

Keep this open: current main still routes Codex app-server startup through the shared client and can clear that singleton on startup failure, while open PR #72812 is the explicit implementation candidate for the spawned/helper isolation fix.

## What This Changes

The issue asks for invalid-token `btw spawn` failures to stay isolated so the active OpenClaw gateway/session remains usable.

## Best Possible Solution

Review and land PR #72812, or an equivalent narrow Codex app-server change, so spawned/helper runs use isolated clients and failed auth cannot clear the shared live-session client.

## Reproduction Assessment

Yes. The issue provides concrete invalid-token steps and logs, and current source shows a path where a spawned Codex `thread/start` 401 can flow into startup cleanup after using the shared singleton.

## Solution Assessment

Yes. Isolating spawned/helper Codex app-server clients and limiting shared-client teardown to runs that own the singleton is narrower than changing global gateway restart or session lifecycle behavior.
