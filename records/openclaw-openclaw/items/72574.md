## Summary

Keep this open: current main still lets spawned/helper Codex app-server startup use the shared client and clear it on startup failure, while PR #72812 remains the open implementation candidate for this exact isolation bug.

## What This Changes

The issue asks for invalid or expired bearer-token failures in `btw spawn codex 5.5` to fail locally without disconnecting the active gateway/session.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72812, or an equivalent narrow Codex app-server change, so spawned/helper runs use isolated clients while normal Codex runs preserve shared-client reuse.

## Reproduction Assessment

Yes. The issue provides concrete invalid-token steps and logs, and current source shows a high-confidence path where a spawned Codex `thread/start` 401 can reach startup cleanup after using the shared singleton.

## Solution Assessment

Yes. Isolating spawned/helper Codex app-server clients and scoping shared-client teardown to runs that own the singleton is narrower than changing gateway restart behavior or global session lifecycle.
