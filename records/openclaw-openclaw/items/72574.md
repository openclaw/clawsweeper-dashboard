## Summary

Keep open: current main still routes Codex app-server startup through the default shared-client factory and can clear that singleton on startup failure, while PR #72812 remains the open implementation candidate for the same isolation bug.

## What This Changes

The report asks for invalid or expired Codex bearer-token failures from `btw spawn` to fail locally without disconnecting the active OpenClaw gateway/session.

## Best Possible Solution

The end state should be isolated spawned/helper Codex app-server clients with shared-client teardown limited to non-isolated runs that own the singleton.

## Reproduction Assessment

Yes, source-backed. The issue gives concrete invalid-token steps and logs, and current main shows a path where a spawned Codex `thread/start` startup failure can use and then clear the shared app-server client.

## Solution Assessment

Yes. Isolating spawned/helper Codex app-server clients and scoping shared-client teardown is narrower than changing gateway restart behavior or global session lifecycle.
