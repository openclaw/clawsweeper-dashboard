## Summary

Keep this open: current main still starts Codex app-server attempts through the shared client factory and clears the shared singleton on startup failure, while PR #72812 is open and explicitly targets this isolation bug.

## What This Changes

The issue requests isolating failed Codex helper spawns so invalid or expired bearer-token auth errors do not disconnect the active OpenClaw gateway/session.

## Best Possible Solution

Review and land PR #72812, or an equivalent narrow Codex app-server change, so spawned/helper runs use isolated clients and child auth failures do not clear the shared singleton.

## Reproduction Assessment

Yes. The issue provides concrete steps and logs, and current source shows a source-level reproduction path where a `thread/start` 401 can propagate into startup cleanup that clears the shared client.

## Solution Assessment

Yes. The narrow maintainable fix is to isolate spawned/helper Codex app-server clients and scope shared-client teardown to runs that actually own the shared singleton, while preserving shared reuse for normal Codex runs.
