## Summary

Keep this open: current main still routes spawned/helper Codex app-server startup through the shared client factory, and the linked open PR is the active implementation candidate for isolating invalid-token failures from the live session.

## What This Changes

The issue asks for invalid or expired bearer-token failures in `btw spawn codex 5.5` to fail locally without disconnecting the active gateway/session.

## Best Possible Solution

Review and land PR #72812, or an equivalent narrow Codex app-server change, so spawned/helper runs use isolated clients while normal Codex runs keep shared-client reuse.

## Reproduction Assessment

Yes. The issue provides concrete steps and logs, and current source shows a high-confidence source-level path where a spawned Codex `thread/start` 401 can reach startup cleanup after using the shared app-server singleton.

## Solution Assessment

Yes. The best fix direction is to isolate spawned/helper Codex app-server clients and scope shared-client teardown to runs that actually own the shared singleton, rather than changing gateway restart or global session behavior.
