## Summary

Keep this open: current main still has the shared Codex app-server startup path that can clear the singleton after a spawned/helper auth failure, and #72812 is an open, unmerged implementation candidate for this exact isolation bug.

## What This Changes

Invalid or expired Codex bearer tokens during `btw spawn codex 5.5` should fail the helper run without disconnecting the active gateway/session.

## Best Possible Solution

Review and land #72812, or an equivalent narrow Codex app-server change, so spawned/helper runs use isolated clients while normal Codex runs retain shared-client reuse.

## Reproduction Assessment

Yes. The issue gives concrete invalid-token steps and logs, and current main shows a high-confidence source path where a spawned `thread/start` 401 can reach shared-client startup cleanup.

## Solution Assessment

Yes. Isolating spawned/helper Codex app-server clients and limiting singleton teardown to runs that own the shared client is narrower than changing gateway restart or global session lifecycle behavior.
