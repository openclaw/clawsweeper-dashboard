## Summary

Keep this open. Current main still lets spawned/helper Codex app-server startup use the shared client and still clears that shared singleton on startup failure, while open PR #72812 already targets the same invalid-token isolation bug with closing syntax.

## What This Changes

Prevent invalid or expired bearer-token failures from `btw spawn codex 5.5` from disconnecting or destabilizing the active gateway/session.

## Best Possible Solution

Land PR #72812 or an equivalent narrow Codex app-server change that uses isolated clients for spawned/helper runs and limits shared-client teardown to non-isolated startup failures.

## Reproduction Assessment

Yes. The report includes concrete steps and logs, and current source shows a high-confidence path where a `thread/start` 401 can throw through startup cleanup and clear the shared singleton.

## Solution Assessment

Yes. Isolating spawned/helper Codex app-server clients is the narrow maintainable fix because normal Codex runs can keep shared-client reuse while child auth failures stop at the child boundary.
