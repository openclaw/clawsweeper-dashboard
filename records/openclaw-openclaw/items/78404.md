## Summary

Keep open: current main still has the WhatsApp no-op reload path that leaves an absent WhatsApp plugin undiscovered, and the issue is paired with an open fixing PR using closing syntax.

## What This Changes

The issue reports that externally hot-enabling `channels.whatsapp.enabled=true` can be silently reverted when WhatsApp was not loaded at gateway cold boot.

## Best Possible Solution

Review, revise, and land https://github.com/openclaw/openclaw/pull/78414 or a replacement that makes unloaded-channel activation observable and recoverable while preserving the reload-mode contract, then close this issue after merge.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run in this read-only review. The source path is clear: `channels.whatsapp` is a no-op reload prefix and current reload code returns before rebuilding or loading an absent plugin, while the production demotion writer remains untraced.

## Solution Assessment

Unclear. Restart-on-activation is a plausible narrow repair, but the open PR still needs maintainer judgment on whether this should auto-restart in hot mode, warn, or become a fuller dynamic hot-load path.
