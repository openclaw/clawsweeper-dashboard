## Summary

Keep open: current main still treats `channels.whatsapp.*` as a no-op reload path with no unloaded-channel activation guard, and a live open PR with closing syntax already targets the exact behavior.

## What This Changes

The issue reports that hot-enabling `channels.whatsapp.enabled` while WhatsApp is absent from the running gateway registry can be silently ignored or persisted back to false.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/78414 or a replacement that makes unloaded-channel activation observable and recoverable while preserving the reload-mode contract, then close this issue after merge.

## Reproduction Assessment

No high-confidence end-to-end current-main reproduction was run. Source inspection verifies the no-op reload path and missing activation guard, while the exact writer that persists `enabled=false` remains untraced.

## Solution Assessment

Unclear. Restart-on-activation is a plausible narrow repair, but maintainers still need to settle whether hot mode should auto-restart, warn, or grow a fuller dynamic hot-load path.
