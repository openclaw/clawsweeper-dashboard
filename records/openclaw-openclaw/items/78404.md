## Summary

Keep open: current main still has the WhatsApp no-op reload rule and generic docs still promise channel hot reload, while source review does not show a shipped fix for the silent false-to-true wedge. The related reports cover adjacent wording and auto-enable churn, not this symmetric hot-enable path.

## What This Changes

Hot-enabling WhatsApp with `channels.whatsapp.enabled=true` should either load/reload the WhatsApp plugin or produce a clear restart requirement instead of silently losing the external config change.

## Best Possible Solution

Define one supported contract for WhatsApp false-to-true channel activation: either dynamically discover/reload the plugin like other channel hot reloads, or preserve the external config value and emit an explicit restart-required diagnostic with docs updated.

## Reproduction Assessment

No. The production report is detailed and current source proves `channels.whatsapp` edits are classified as a no-op when the plugin is absent, but I did not establish a current-main live reproduction of the later true-to-false demoting write.

## Solution Assessment

Unclear. A warning/restart requirement is the conservative near-term repair, but maintainers should decide whether the documented `channels.*` hot reload contract instead requires WhatsApp false-to-true to trigger plugin discovery and channel startup.
