Keep open: the PR addresses a real exec failure-message ambiguity, but it is not merge-ready because the patch deletes semantic exec action labels and the added proof is still a one-off simulation rather than a real cron/chat/CLI failure path.

## What This Changes

The branch adds `stripLeadingExecDisplayVerb`, applies it to exec/bash failure-warning metadata, and updates payload/tool-display tests to expect leading exec summary verbs to be removed.

## Best Possible Solution

Render the tool failure label separately from the command or subject while preserving the semantic exec action summary, then add focused regression tests and real cron/chat/CLI proof.

## Reproduction Assessment

Yes, at source level: current main stores exec display metadata in `lastToolError`, formats it with `formatToolAggregate`, and appends `failed`, so command-shaped summaries can remain visually ambiguous. I did not run a live cron/chat/CLI reproduction in this read-only review.

## Solution Assessment

No. The patch fixes the ambiguity by deleting the first display-summary word, but the best fix is to preserve the action label and change the failure-rendering shape so action and literal command remain visually distinct.
