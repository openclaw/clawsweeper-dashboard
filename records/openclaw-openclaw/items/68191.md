## Summary

Keep open. Current main has partial durable cron recovery, but cron lost reconciliation still falls back to the process-local `activeJobIds` set, terminal `lost` rows are not recovered by the active-task helper, and the startup catch-up/manual-run active-job symmetry gaps are still visible in current source. Related PRs #71040 and #75036 remain open rather than merged.

## What This Changes

Prevent cron task maintenance from marking completed or still-running cron task ledger records `lost` solely because transient `activeJobIds` state is absent after a gateway restart.

## Best Possible Solution

Land a coordinated fix that keeps cron tasks live across every execution path while running, recovers active and terminal false-positive cron rows from durable run history/job state where possible, and leaves genuinely unrecoverable cron rows with clear diagnostics instead of hiding all cron lost detection.

## Reproduction Assessment

Yes for the core failure mode: the current maintenance test harness can reproduce a stale cron task becoming `lost` when `isCronJobActive` is false and no durable terminal history is resolved. The live reports from OpenClaw 2026.4.26 add supporting evidence for accumulated `runtime=cron` rows with `error="backing session missing"`.

## Solution Assessment

Yes for the durable-state direction, but not for the unconditional cron exemption fallback. The narrowest maintainable path is to complete active-job mark/clear symmetry and durable terminal recovery while preserving real lost-task detection for cron rows that cannot be recovered.
