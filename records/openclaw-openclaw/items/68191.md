## Summary

Keep open. Current main has partial durable cron recovery, but cron lost reconciliation still depends on the process-local active-job set for active rows, terminal false-positive lost rows are still excluded from durable recovery, and related PRs #71040 and #75036 remain open rather than merged.

## What This Changes

The issue reports false-positive cron task ledger rows marked `lost` with `backing session missing` after gateway restart because task maintenance treats cron liveness as transient `activeJobIds` membership.

## Best Possible Solution

Land a coordinated fix that keeps every cron execution path active while its task row is non-terminal and recovers backing-session false-positive terminal cron rows from exact durable run history without disabling real lost-task detection.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an expired active cron task whose `sourceId` is absent from `activeJobIds`; the current test harness asserts that it becomes `lost`, and current source shows terminal `lost` rows cannot enter durable cron recovery.

## Solution Assessment

Yes for the durable-state and active-job-symmetry direction, but no for an unconditional cron exemption. The maintainable fix is to complete active-job ownership coverage and constrained durable terminal recovery so genuine unrecoverable cron rows can still be diagnosed.
