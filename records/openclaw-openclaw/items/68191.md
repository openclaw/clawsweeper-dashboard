## Summary

Keep open. Current main has partial durable cron recovery and docs now describe that path, but the source still allows Gateway-owned cron tasks to fall through to process-local `activeJobIds` and become `lost` when durable recovery cannot match the run; the issue also has two related open fix PRs rather than a merged implementation. Reviewed: https://github.com/openclaw/openclaw/issues/68191

## What This Changes

Prevent cron task maintenance/audit from recording false-positive `lost` rows with `backing session missing` solely because transient cron active-job memory was cleared across a Gateway restart.

## Best Possible Solution

Land a targeted cron/task maintenance fix that keeps true lost-task detection, marks all live cron execution paths active while running, uses durable run/job state before lost-marking, and can recover or clearly diagnose terminal `lost` cron rows created by the false-positive path.

## Reproduction Assessment

Yes. Current main has a focused test harness where a stale cron task with no active cron job is reconciled to `lost`, and the source path still reaches that behavior when durable recovery misses. I did not run tests because this review was required to keep the checkout byte-clean.

## Solution Assessment

No. Current main's durable recovery is the right direction, but it is incomplete for the reported surface; the safer maintainable route is to review the active-job symmetry and terminal-row recovery PRs rather than disabling cron lost detection wholesale.
