## Summary

Keep open: the proposed idea is plausible, but this patch is underimplemented because it only adds a field to an internal context type while current health-check runner paths still call checks once and never set or consume `retries`.

## What This Changes

The PR adds an optional `retries?: number` property to `HealthCheckRunContext` in `src/flows/health-check-runner-types.ts`.

## Best Possible Solution

Define retry behavior at the runner boundary only if maintainers want a generic policy, with retry count source, transient-error classification, backoff behavior, and focused tests; otherwise keep retries inside individual health probes.

## Reproduction Assessment

Not applicable as a user bug reproduction; this is a feature PR. Source inspection gives high confidence that the claimed retry behavior is absent because the runner and lint paths call each check once and never pass a nonzero retry value.

## Solution Assessment

No. A type-only field is not the best way to solve transient health-check failures because it neither defines retry ownership nor changes runtime behavior; the maintainable path is a complete retry contract or check-local retry where transient semantics are known.
