This PR should stay open, but it is not merge-ready: the branch only adds a `retries` type field while current doctor health-check paths still execute each check once, so the advertised transient retry behavior is not implemented.

## What This Changes

The PR adds an optional `retries?: number` property and comment to `HealthCheckRunContext` in the doctor health-check runner types.

## Best Possible Solution

Define a complete runner-level retry contract only if maintainers want generic doctor health-check retries; otherwise keep transient retry handling inside checks that understand their own failure modes.

## Reproduction Assessment

Not applicable as a user bug reproduction; this is a feature PR. Source inspection gives high confidence that the claimed retry behavior is absent because runner, validation, and lint paths each call the health check once.

## Solution Assessment

No. A type-only field is not the best way to solve transient health-check failures because it neither defines retry ownership nor changes runtime behavior.
