## Summary

Keep open: current `main` still lacks the requested Teams Graph tenant override, and this PR addresses a real source-level gap, but the latest PR head still retains a raw Teams client secret in a module-level cache key and has no inspectable real split-tenant proof.

## What This Changes

The branch adds `graphTenantId`/`MSTEAMS_GRAPH_TENANT_ID` for Teams Graph token acquisition, wires a separate Graph token provider through Teams monitor/message handling, and updates docs, generated config metadata, and focused tests.

## Best Possible Solution

Land a tenant-aware Teams Graph token provider after replacing the secret-bearing cache key with a non-secret discriminator or one-way fingerprint, keeping config/docs/tests aligned, and attaching redacted proof from a real split-tenant deployment.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level gap: current main has no `graphTenantId` config/env surface and Graph thread calls use the existing Teams token provider. A live 403 reproduction still requires a split Azure/M365 tenant setup that was not available in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently written. The split-provider direction is narrow and plausible, but the cache key must avoid plaintext secret retention and the contributor needs to attach inspectable real behavior proof before merge.
