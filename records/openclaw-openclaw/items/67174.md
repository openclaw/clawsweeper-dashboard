## Summary

Keep this PR open: current main still lacks the split-tenant Teams Graph token surface, and the branch is a plausible implementation path, but the latest diff is not merge-ready because it retains a raw Teams client secret in a module-level cache key and lacks inspectable real behavior proof.

## What This Changes

The PR adds `channels.msteams.graphTenantId` and `MSTEAMS_GRAPH_TENANT_ID`, then routes Microsoft Teams Graph token acquisition through a separate tenant-aware provider for split-tenant deployments.

## Best Possible Solution

Use a tenant-aware Teams Graph token provider without retaining plaintext secrets, keep config schema, docs, generated metadata, and regression tests aligned, and require redacted proof from a real split-tenant deployment before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level gap: current main has no `graphTenantId` config/env surface and uses the existing Teams token provider for Graph thread calls. A live 403 reproduction would require a split Azure/M365 tenant setup, which was not available in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently written. The split-provider direction is narrow and plausible, but the cache key must avoid raw secret retention and the contributor needs to attach real deployment proof before merge.
