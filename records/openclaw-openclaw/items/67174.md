## Summary

Keep open: current main still lacks a split-tenant Teams Graph override, and the PR direction is plausible, but the latest head has a blocking security finding and no inspectable real split-tenant proof.

## What This Changes

The PR adds `channels.msteams.graphTenantId`/`MSTEAMS_GRAPH_TENANT_ID`, wires a dedicated Teams Graph token provider, and updates docs, generated config metadata, changelog, Graph error handling, cache behavior, and tests.

## Best Possible Solution

Land tenant-aware Teams Graph token acquisition after replacing the secret-bearing cache key with a non-secret discriminator and attaching redacted proof from a real split-tenant deployment.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level gap: current main has no `graphTenantId` config/env surface and channel thread Graph calls use the normal Teams token provider. A live 403 reproduction was not established because it requires a split Azure/M365 tenant setup.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently written. The split-provider direction matches the existing Teams SDK token-provider seam, but the raw-secret cache key and missing real deployment proof need to be resolved before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid caching raw Teams client secrets:** `extensions/msteams/src/graph.ts:49`
  - body: `buildGraphTokenProviderCacheKey` serializes `creds.appPassword`, so the resolved Azure client secret is retained in a module-level cache key for the process lifetime. Use a non-secret discriminator or one-way fingerprint so the cache does not keep the plaintext secret.
  - confidence: 0.86
