## Summary

Keep open: this is a focused Teams auth/config feature that is still missing from current main, but the latest PR head still needs a secret-handling fix and inspectable real split-tenant proof before merge.

## What This Changes

The PR adds `graphTenantId`/`MSTEAMS_GRAPH_TENANT_ID` for Microsoft Teams Graph token acquisition, wires separate Graph token providers through Teams runtime paths, and updates docs, generated config metadata, changelog, and focused tests.

## Best Possible Solution

Land tenant-aware Teams Graph token acquisition after replacing the secret-bearing cache key with a non-secret discriminator or one-way fingerprint and attaching redacted proof from a real split-tenant deployment.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level gap: current main has no `graphTenantId` config/env surface and Graph thread context uses the normal Teams token provider. A live 403 reproduction still requires a split Azure/M365 tenant setup, which was not available in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently written. The split-provider direction is narrow and matches the Teams SDK token contract, but the raw-secret cache key and missing real behavior proof need to be resolved before merge.
