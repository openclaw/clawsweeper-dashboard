## Summary

Keep open. Current main still lacks the cross-tenant Microsoft Teams Graph tenant surface, and this PR is the active implementation vehicle, but the latest head is not merge-ready because it still retains a Teams client secret in a module-level cache key, lacks the required changelog credit, and GitHub reports the branch as dirty against its base.

## What This Changes

The PR adds `channels.msteams.graphTenantId` and `MSTEAMS_GRAPH_TENANT_ID`, routes Microsoft Teams Graph token acquisition through a graph-tenant token provider, updates config/docs/plugin metadata/generated baselines, and adjusts thread Graph error handling tests.

## Best Possible Solution

Land a narrow Microsoft Teams plugin change that keeps Bot Framework reply tokens on the bot tenant, uses `graphTenantId` only for Graph app-only token acquisition, avoids retaining raw secrets in cache keys, keeps config schema/docs/generated metadata aligned, and ships with focused token/thread-context regression coverage.

## Reproduction Assessment

No full live reproduction was run for a split-tenant Teams deployment. Static verification is strong: current main has no `graphTenantId` config/env path and uses the normal Teams token provider for Graph thread context, while the PR author's body reports manual cross-tenant verification.

## Solution Assessment

No, this PR is not the best mergeable form yet. The split-provider direction is narrow and maintainable, but the raw-secret cache key, missing changelog attribution, and dirty merge state should be fixed before merge.
