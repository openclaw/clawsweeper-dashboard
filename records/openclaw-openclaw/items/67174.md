## Summary

Keep open: current main still lacks the Teams split-tenant Graph token surface, and this PR remains the focused implementation path, but it is not merge-ready because it lacks inspectable real split-tenant proof, has a raw-secret cache-key regression, and is currently conflicting against main.

## What This Changes

The PR adds `channels.msteams.graphTenantId` and `MSTEAMS_GRAPH_TENANT_ID`, routes Teams Graph token acquisition through a tenant-specific provider, and updates Teams docs, plugin metadata, config metadata, changelog, and tests.

## Best Possible Solution

Land a rebased Teams plugin change that adds the Graph tenant override while preserving current secret/token storage and Graph fetch guards, avoids secret-bearing cache keys, and includes redacted proof from a real split-tenant deployment.

## Reproduction Assessment

Yes for the source-level gap: current main has no `graphTenantId`/`MSTEAMS_GRAPH_TENANT_ID` surface and uses the normal Teams token provider for Graph thread context. The full live 403 reproduction still requires a split Azure/M365 tenant setup, and no artifact is attached.

## Solution Assessment

No, not as currently written. A dedicated Graph token provider is the right narrow direction, but the raw-secret cache key, stale/conflicting head, and missing real behavior proof need to be resolved first.
