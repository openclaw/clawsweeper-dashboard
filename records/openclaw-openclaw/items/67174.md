Keep open: current main still lacks the requested Teams split-tenant Graph token configuration, so the PR is not obsolete. The branch remains useful, but it is not merge-ready because it lacks inspectable real behavior proof and still has concrete secret/token-storage blockers.

## What This Changes

The PR adds `channels.msteams.graphTenantId` and `MSTEAMS_GRAPH_TENANT_ID`, routes Teams Graph token acquisition through a tenant-specific provider, and updates Teams docs, metadata, config schema, changelog, and tests.

## Best Possible Solution

Land a rebased Teams plugin change that adds the Graph tenant override while preserving current secret/token storage and Graph fetch safeguards, with redacted proof from a real split-tenant deployment.

## Reproduction Assessment

Yes for the source-level gap: current main has no `graphTenantId` / `MSTEAMS_GRAPH_TENANT_ID` surface and still uses the normal Teams token provider for Graph thread context. The full live 403-to-success path still requires a split Azure/M365 tenant setup and attached proof.

## Solution Assessment

No, not as currently written. A dedicated Graph token provider is the right narrow direction, but the branch must avoid secret-bearing cache keys, preserve current private token-store behavior, and include real split-tenant proof.
