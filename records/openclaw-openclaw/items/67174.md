## Summary

Keep open: current main does not implement the split-tenant Teams Graph tenant override, and the PR is a plausible focused feature, but the latest head still needs real split-tenant proof and has two secret-handling blockers before merge.

## What This Changes

The PR adds `channels.msteams.graphTenantId`/`MSTEAMS_GRAPH_TENANT_ID`, routes Microsoft Teams Graph token acquisition through a separate provider when configured, and updates docs, generated config metadata, changelog, and focused Teams tests.

## Best Possible Solution

Keep the split-provider approach, but rebase onto current main, preserve private token storage, replace the secret-bearing cache key with a non-secret discriminator or one-way fingerprint, and attach redacted proof from a real split-tenant Teams deployment.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level gap: current main has no `graphTenantId`/`MSTEAMS_GRAPH_TENANT_ID` surface and uses the normal Teams token provider for Graph thread context. A full live 403 reproduction still requires a split Azure/M365 tenant setup and is not attached.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently written. A dedicated Graph token provider is a narrow maintainable direction, but the raw-secret cache key, stale delegated-token write path, and missing real behavior proof need to be resolved first.

  - body: `buildGraphTokenProviderCacheKey` serializes `creds.appPassword`, so the resolved Azure client secret is retained in a module-level string for the process lifetime. Use a non-secret discriminator or one-way fingerprint for the provider cache key instead.

  - confidence: 0.82
