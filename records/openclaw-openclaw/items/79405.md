## Summary

Keep open: the bug-fix direction is plausible, but the current PR is not merge-ready because source inspection still shows two blocking delivery regressions and the supplied proof does not demonstrate the fallback path it changes.

## What This Changes

The PR adds task-aware subagent completion fallback delivery, persists taskId in subagent run/pending delivery state, adds fallback delivery statuses, and extends lifecycle coverage.

## Best Possible Solution

Refresh against current main, preserve requester-agent completion delivery, scope task-aware fallback to terminal give-up paths, forward the fallback flag through the delivery boundary, add boundary coverage, and attach redacted runtime proof of the fallback path.

## Reproduction Assessment

Yes. Source inspection reproduces the PR blockers: task-aware lookup can stop existing taskId-free completion delivery, and the fallback flag is accepted by deliverSubagentAnnouncement but not forwarded to the helper that gates fallback sends.

## Solution Assessment

No. The intended hardening is reasonable, but this patch is not the narrow maintainable solution until it preserves normal completion delivery and wires the fallback flag through the actual send boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep task lookup out of normal completion delivery:** `src/agents/subagent-announce.ts:556-570`
  - body: This lookup now runs for every non-nested completion and returns `false` when `taskId` or task delivery context is missing. Existing direct completion callers already pass a requester origin without a task id, so they would stop before the normal announce path instead of only using task context for terminal give-up fallback.
  - confidence: 0.92
- **[P1] Forward the fallback flag to direct delivery:** `src/agents/subagent-announce-delivery.ts:1177-1194`
  - body: `deliverSubagentAnnouncement` accepts `allowCompletionFallback`, but the direct delivery call never passes it to `sendSubagentAnnounceDirectly`. That helper gates fallback sends on `params.allowCompletionFallback === true`, so the terminal give-up retry path still cannot send the fallback this PR is meant to add.
  - confidence: 0.95
