## Summary

Keep open; the current PR is not merge-ready because source inspection shows the fallback flag is dropped and normal non-nested completion announces can be blocked by the new task lookup, while the supplied proof does not show the failure-to-fallback path.

## What This Changes

The PR adds task-aware subagent completion fallback delivery, taskId persistence on subagent runs, new fallback delivery status values, and lifecycle tests for give-up fallback handling.

## Best Possible Solution

Rebase onto current main, preserve normal requester-agent completion delivery, scope task-aware fallback to terminal give-up paths, forward the fallback flag through the delivery boundary, add boundary coverage, and provide redacted runtime proof of the fallback path.

## Reproduction Assessment

Yes. Source inspection reproduces the PR blockers: the new task context gate can stop existing completion delivery before direct announce, and allowCompletionFallback is accepted but not forwarded into the helper that gates fallback sends.

## Solution Assessment

No. The intended fallback hardening is plausible, but the patch is not the narrow maintainable fix until it preserves existing completion delivery and wires the fallback flag through the public delivery boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep task lookup out of normal completion delivery:** `src/agents/subagent-announce.ts:556-570`
  - body: This task-aware lookup now runs for every non-nested completion and returns `false` when the task id or delivery-state origin is missing. Existing direct completion callers already pass a requester origin but no task id, so they would stop before the normal announce path instead of only using task context for the terminal fallback case.
  - confidence: 0.92
- **[P1] Forward the fallback flag to direct delivery:** `src/agents/subagent-announce-delivery.ts:1157`
  - body: `deliverSubagentAnnouncement` accepts `allowCompletionFallback`, but the direct delivery call never passes it to `sendSubagentAnnounceDirectly`. That helper gates every fallback send on `params.allowCompletionFallback === true`, so the terminal give-up retry path still cannot send the fallback this PR is meant to add.
  - confidence: 0.95
