Full review comments:

- **[P1] Keep requester-origin fallback when task context is absent:** `src/agents/subagent-announce.ts:564-570`
  - body: This early return makes user-facing subagent completion delivery fail whenever the task id or task delivery requester origin is missing. `createRunningTaskRun` is best-effort and existing pending delivery records will not have this new `taskId`, while this flow already resolved `directOrigin`; use task status context only when present or fall back to the existing requester origin instead of returning `task_id_missing`/`context_missing`.
  - confidence: 0.86
- **[P1] Constrain direct fallback to mirrored direct-message delivery:** `src/agents/subagent-announce-delivery.ts:784-795`
  - body: The new fallback sends raw completion text to any channel/thread and reports `direct-fallback` as delivered without recording a requester-session mirror. For group/thread or message-tool-only routes this bypasses the requester-agent rewrite and leaves session state unaware of the visible send, so retries or later prompts can duplicate or miss the delivered context; reuse the current mirrored direct-message fallback shape instead.
  - confidence: 0.84
- **[P1] Keep delivery-status parsing in sync with new statuses:** `src/tasks/task-registry.types.ts:19-23`
  - body: These new delivery statuses can be written through `setDetachedTaskDeliveryStatusByRunId`, but current main restores SQLite rows through a closed `TASK_DELIVERY_STATUSES` parser. A rebased version needs to update the parser/store/audit tests or avoid persisting new status strings, otherwise an upgraded checkout can fail to read task rows containing `fallback_sent` or the new failure codes.
  - confidence: 0.78
