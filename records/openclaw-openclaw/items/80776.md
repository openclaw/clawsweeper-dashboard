## Summary

Keep open: the PR addresses a plausible Codex app-server lane hang, but the fallback can now release and interrupt a turn after a completed `agentMessage` that the protocol explicitly allows to be mid-turn commentary.

## What This Changes

The PR adds a Codex app-server idle-release fallback for quiet completed assistant-message items, updates focused harness tests, and documents the new lane-release behavior.

## Best Possible Solution

Keep the narrow Codex app-server fallback, but only treat a completed assistant item as terminal when protocol metadata and current-turn state rule out commentary or other intermediate output.

## Reproduction Assessment

Yes, source-level: the new branch arms release on any completed `agentMessage` with text, while the generated schema says commentary messages are mid-turn and may precede more tool calls or output.

## Solution Assessment

No, not as written: the Codex app-server lifecycle is the right place for this fallback, but it must preserve `phase: "commentary"` semantics before interrupting and resolving the native turn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not release commentary agentMessage items:** `extensions/codex/src/app-server/run-attempt.ts:1262-1263`
  - body: The fallback arms for every completed `agentMessage` with text, but the Codex schema defines `phase: "commentary"` as mid-turn text where more tool calls or assistant output may follow, and the projector already treats multiple `agentMessage` items as intermediate until `turn/completed`. A commentary item followed by more than 10s of model silence would now trigger `turn/interrupt` and resolve the run before the next tool call or final answer. Please ignore commentary-phase agent messages and add a regression test before using this as a terminal fallback.
  - confidence: 0.9
