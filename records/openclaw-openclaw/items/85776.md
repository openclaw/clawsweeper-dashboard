## Summary

Keep open for maintainer review: current main and the latest release still forward OpenAI `output_compression` on PNG/default-PNG image paths, while this focused PR gates that field to JPEG/WebP and covers both direct and Codex request builders. No blocking review or security findings remain; the only residual validation gap is that the proof verifies the deployed guard rather than showing a fresh live PNG generation.

## What This Changes

The PR centralizes OpenAI image output-compression filtering, omits `output_compression` for PNG/default-PNG direct and Codex image requests, adds request-body regression tests, and updates the image generation docs.

## Best Possible Solution

Land the provider-local guard so OpenAI PNG/default-PNG requests omit `output_compression` while JPEG/WebP requests keep compression.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.5.22` attach `output_compression` whenever config supplies it, including PNG/default-PNG paths, while the OpenAI SDK contract limits that parameter to JPEG/WebP outputs.

## Solution Assessment

Yes. Centralizing the format gate and reusing it at both OpenAI request construction sites is the narrowest maintainable fix; no new config or public API direction is needed.
