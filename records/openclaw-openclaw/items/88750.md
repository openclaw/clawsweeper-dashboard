## What This Changes

## Best Possible Solution

Land only after runtimeSettings is treated as a versioned plugin contract: preserve strict legacy plugins, expose and document the SDK surface, populate accurate requested/resolved/fallback fields or leave them null, and add redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection shows the PR always passes runtimeSettings while current registry compatibility only strips sessionKey/prompt; a strict schema engine like the existing strict sessionKey test helper would reject the new key and enter the quarantine fallback path.

## Solution Assessment

No. The direction is plausible, but the current implementation is not the best merge shape until compatibility stripping, accurate fallback semantics, SDK/docs exposure, and real configured-engine proof are all aligned.

Full review comments:

- **[P1] Preserve strict legacy engines when adding runtimeSettings:** `src/agents/harness/context-engine-lifecycle.ts:156`
  - body: This now passes `runtimeSettings` to plugin lifecycle methods, but the current registry wrapper only strips `sessionKey`/`prompt` for strict legacy schemas. The latest commit that says it fixes this is empty, so a strict engine that rejects unknown keys will still throw, be quarantined, and fall back to the default engine instead of running.
  - confidence: 0.96
- **[P2] Thread requested and fallback state into the contract:** `src/agents/embedded-agent-runner/run/attempt.ts:2915-2917`
  - body: The production call sites only pass the current provider/model into the builder, leaving `requestedModelId`, `fallbackActive`, `fallbackReason`, and `degradedReason` unset. That means fallback turns report `requested: null` and `fallbackActive: false`, which undercuts the main purpose of this contract.
  - confidence: 0.9
- **[P2] Stop reporting model ids as model families:** `src/agents/harness/context-engine-lifecycle.ts:51`
  - body: `modelFamily` is populated from `modelId`/`requestedModelId`, so plugins can receive a concrete model id in a field named `family`. Please plumb a real family value or keep this field `null` until that value exists.
  - confidence: 0.93
- **[P2] Publish the runtime settings contract through the SDK:** `src/context-engine/types.ts:67-95`
  - body: This adds a plugin-facing lifecycle type, but the public plugin SDK exports and docs still describe only the old assemble fields. External plugin authors need the new type and docs through the supported SDK surface, not internal imports.
