## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve strict legacy engines when adding runtimeSettings:** `src/agents/harness/context-engine-lifecycle.ts:156`
  - body: Every assemble hook now receives `runtimeSettings`, but the current registry compatibility wrapper only retries after stripping `sessionKey` and `prompt`. Strict-schema context-engine plugins that reject unknown lifecycle keys will throw on their first turn and get quarantined into fallback, so add `runtimeSettings` to the legacy strip/gate path and cover bootstrap/assemble/afterTurn/maintenance.
  - confidence: 0.95
- **[P2] Stop reporting model ids as model families:** `src/agents/harness/context-engine-lifecycle.ts:51`
  - body: This sets `model.family` from `modelId` or `requestedModelId`, so consumers can receive a concrete id such as `gpt-5.5` in a field named family. That can mislead plugin diagnostics or branching; plumb a real family value or leave the field null until one exists.
  - confidence: 0.91
- **[P2] Make fallback diagnostics internally consistent:** `src/context-engine/runtime-settings.ts:52-54`
  - body: A caller can pass `fallbackReason` with `fallbackActive: false`, producing `mode: "normal"` and `fallbackActive: false` while still exposing a fallback reason. Consumers of this new contract can make opposite decisions from the same object, so either coerce a reason into active fallback or drop/reject the reason when fallback is explicitly false.
  - confidence: 0.86
- **[P2] Publish the runtime settings type through the SDK:** `src/context-engine/types.ts:67`
  - body: The plugin-facing hook params now expose `runtimeSettings`, but the named contract and errors are not added to the public SDK export/docs surfaces plugin authors use. That makes adoption depend on internal paths for a new public lifecycle contract; re-export and document it with the SDK baseline checks.
  - confidence: 0.88
