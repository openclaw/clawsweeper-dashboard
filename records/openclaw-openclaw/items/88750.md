## Summary

Keep the PR open: the runtime-settings lifecycle direction is useful, but the current patch expands the context-engine plugin contract without preserving strict legacy engines, mislabels model family data, leaves the public SDK/docs surface incomplete, and lacks required real configured behavior proof.

## What This Changes

The PR adds a ContextEngineRuntimeSettings contract and passes runtimeSettings through context-engine bootstrap, assemble, afterTurn, and maintenance paths across embedded, CLI, and Codex app-server runs.

## Best Possible Solution

Land the runtime-settings contract only after preserving strict legacy compatibility, correcting or omitting model family data, exposing the SDK/docs contract, and adding real configured run proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence repro path: a strict context-engine plugin that rejects unknown lifecycle keys receives runtimeSettings, the current registry only strips sessionKey/prompt, and the runtime can quarantine the engine after the thrown error.

## Solution Assessment

The shared lifecycle is the right layer for runtime settings, but this is not the best merge shape until legacy compatibility, SDK exposure, model-family semantics, and real runtime proof are handled.
