## Summary

## What This Changes

The PR adds a `ContextEngineRuntimeSettings` contract and passes runtime settings into context-engine lifecycle calls across harness, CLI, embedded, and Codex app-server paths.

## Best Possible Solution

Land the runtime-settings contract only after preserving legacy strict-engine compatibility, aligning the intended SDK surface, and adding redacted real behavior proof from a configured context-engine run.

## Reproduction Assessment

Yes, by source inspection: current `main` strips only `sessionKey` and `prompt` for strict legacy context engines, while this PR passes a new `runtimeSettings` property into lifecycle params. A strict engine that rejects unknown keys follows the existing tested strict-engine pattern and would enter the runtime quarantine fallback path.

## Solution Assessment

No, not yet. The general owner boundary is reasonable, but the patch needs the existing compatibility wrapper and regression tests updated before the new lifecycle contract is safe to merge.

- **[P1] Preserve legacy engines when adding runtimeSettings:** `src/agents/harness/context-engine-lifecycle.ts:156`
  - body: Existing context-engine plugins may validate lifecycle params with strict schemas.
