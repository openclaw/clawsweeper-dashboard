## Summary

Keep this PR open, but it is not merge-ready: the runtime-settings contract is useful, yet the patch currently breaks strict legacy context-engine plugins and only provides test/static validation, not real configured-engine proof.

## What This Changes

The PR adds a ContextEngineRuntimeSettings type/builder and threads a runtimeSettings object into context-engine bootstrap, assemble, afterTurn, and maintenance paths for embedded, CLI, and Codex app-server runs.

## Best Possible Solution

Land a compatibility-preserving runtime-settings contract: strip or gate runtimeSettings for legacy strict engines, correct or omit model family until a real family is available, align SDK exports/docs, and include redacted proof from a configured context-engine run.

## Reproduction Assessment

Yes, source-level. A strict context engine that rejects unknown lifecycle keys will receive runtimeSettings from this patch, while the compatibility wrapper only strips sessionKey/prompt and otherwise routes lifecycle throws into quarantine/fallback.

## Solution Assessment

No. Passing runtime settings through the shared lifecycle is the right layer, but this is not the best merge shape until the legacy compatibility, model-family semantics, SDK exports, and real runtime proof are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve legacy engines when adding runtimeSettings:** `src/agents/harness/context-engine-lifecycle.ts:156`
  - body: This now always passes `runtimeSettings` into `assemble`, and the same new key is also threaded through bootstrap/afterTurn/maintenance. The existing compatibility wrapper only retries after stripping `sessionKey`/`prompt`, so strict-schema engines that reject unknown keys will throw, be quarantined, and fall back instead of using the configured engine. Extend the legacy compat stripping or gate this field by support before sending it unconditionally.
  - confidence: 0.9
- **[P2] Do not report model ids as model families:** `src/agents/harness/context-engine-lifecycle.ts:51`
  - body: `modelFamily` is populated from `modelId`, so runtime settings will report values like `gpt-5.5` as the family. Plugins and diagnostics that consume a `family` field need a family-level value, not a concrete model id; either plumb a real family value or leave this field null until one is available.
  - confidence: 0.82
- **[P2] Export the runtime settings type through the plugin SDK:** `src/context-engine/types.ts:67`
  - body: The PR adds a named `ContextEngineRuntimeSettings` contract, but the public `openclaw/plugin-sdk` barrel still re-exports only `ContextEngineRuntimeContext`. Context-engine plugin authors should not need internal paths or type inference tricks to consume the new lifecycle contract; add the SDK export and matching contract/docs coverage in this PR.
  - confidence: 0.78
