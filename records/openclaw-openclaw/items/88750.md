## Summary

Needs changes before merge: the new runtime settings lifecycle contract is useful and the supplied proof covers configured-engine plumbing, but current head captures the requested model before hook-based model selection, so normal hook-routed runs can be reported as fallback/mismatched-model runs.

## What This Changes

The PR adds an optional ContextEngineRuntimeSettings payload and threads it through context-engine lifecycle hooks, SDK exports, Codex app-server, CLI, compaction paths, docs, and tests.

## Best Possible Solution

Keep the optional lifecycle payload and legacy strip/retry path, but derive requested/resolved semantics from the post-hook selected model unless a real fallback decision occurred, with regression coverage for hook-routed runs.

## Reproduction Assessment

Yes, source-reproducible: a before_model_resolve or before_agent_start hook can change modelId after requestedModelId is captured, making the new runtimeSettings payload report a false requested/resolved mismatch.

## Solution Assessment

No: threading a versioned payload through the lifecycle is the right layer, but the current implementation conflates hook-selected models with fallback by capturing the requested model too early.
