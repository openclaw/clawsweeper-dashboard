## Summary

Keep this PR open for review: the accepted RFC and linked implementation issue now support the direction, and the supplied terminal proof is adequate, but the patch still derives the new public `contextEngineSelection` payload from engine metadata instead of the registry-selected slot id.

## What This Changes

The PR adds an optional `ContextEngineRuntimeSettings` payload to selected context-engine lifecycle hooks, exports the contract through the plugin SDK, wires it through embedded, CLI, and Codex app-server paths, and updates docs and tests.

## Best Possible Solution

Carry the registry-selected context-engine id and selection source through resolution into runtimeSettings across harness, CLI, and compaction paths, then keep the RFC v1 docs and focused tests aligned.

## Reproduction Assessment

Not applicable as a user bug, but the API contract issue is source-reproducible: the new builder reads `contextEngine.info.id` while the registry explicitly allows that metadata to differ from the selected slot id.

## Solution Assessment

No as-is: the accepted RFC is the right product path, but the implementation should use registry selection metadata rather than engine-owned metadata for `contextEngineSelection`.
