## Summary

Keep open. The Firecrawl SSRF concern is still real on current main, but this PR is not safe to merge as-is because its new Firecrawl client import points at a plugin SDK subpath that current main does not export. The remaining action is branch revision plus maintainer/security review, not cleanup closure.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper, calls it before scrape requests, exports it for tests, and adds Firecrawl client tests for allowed and blocked scrape target URLs while the body still describes a cache-size cap not present in the final diff.

## Best Possible Solution

Revise this as a narrow Firecrawl security hardening change: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF types from exported plugin SDK subpaths, decide and document the hosted versus self-hosted Firecrawl SSRF policy, keep focused client and caller-surface regression tests, and align the PR body with the actual shared cache behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main forwards `params.url` as the Firecrawl `/v2/scrape` request body without local Firecrawl SSRF validation, and the PR defect is visible by comparing its `browser-security-runtime` import with current main's package exports and SDK entry files.

## Solution Assessment

No as-is. The enforcement point and tests are the right shape, but the patch must import `SsrFBlockedError` from an exported subpath such as `openclaw/plugin-sdk/ssrf-runtime` or `openclaw/plugin-sdk/security-runtime` before it is safe to merge.
