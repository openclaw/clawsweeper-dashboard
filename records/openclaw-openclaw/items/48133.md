## Summary

Keep open. The Firecrawl SSRF issue is still valid on current main, but this PR is not safe to merge as-is because the proposed Firecrawl client imports `SsrFBlockedError` from a plugin SDK subpath that current main does not export, so the plugin would fail module resolution before the guard can run.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper and tests for blocked internal, metadata, malformed, and non-HTTP scrape targets while the body also describes a cache-size cap that is not present in the final diff.

## Best Possible Solution

Revise this as a narrow Firecrawl security hardening change: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF types from exported plugin SDK subpaths, keep focused client and caller-surface regression tests, and align the PR body with the actual shared cache behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main forwards `params.url` into the Firecrawl `/v2/scrape` request body without local SSRF validation, and the PR defect is visible by comparing its `browser-security-runtime` import with current main's package exports and SDK entry files.

## Solution Assessment

No as-is. The enforcement point and tests are the right shape, but the patch must import `SsrFBlockedError` from an exported subpath such as `openclaw/plugin-sdk/ssrf-runtime` or `openclaw/plugin-sdk/security-runtime` before it is safe to merge.
