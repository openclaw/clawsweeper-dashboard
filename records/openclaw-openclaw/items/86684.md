## Label Justifications

- `P1`: Unexpected parent-session compaction during yielded subagent workflows can break active orchestration and materially increase cost for real users.
- `impact:session-state`: The report concerns parent branch, transcript leaf, and context-state drift during subagent wake handling.
- `impact:data-loss`: The observed early compactions replace exact parent-session history even when visible context usage is low.

## Summary

Keep open: current main and the latest release still have a source-backed path where inflated successful-turn usage can trigger overflow compaction, `sessions_yield` persists a hidden parentable context marker, and persisted compaction entries still do not carry the requested trigger cause.

## What This Changes

The issue requests that subagent completion or `sessions_yield` wake handling stop compacting parent sessions at low real context usage and persist compaction trigger metadata for future triage.

## Best Possible Solution

Fix the core compaction/accounting contract so successful-turn auto-compaction uses a trustworthy active-context snapshot, add trigger metadata to persisted compaction entries, and prove the subagent/yield wake path with focused regression coverage plus a real long-running session check.

## Reproduction Assessment

Yes for a source-level reproduction: a successful assistant message with inflated `usage.input + cacheRead` over the window still enters the overflow compaction path even if `totalTokens` is lower. I did not run a current-main live subagent reproduction in this read-only review.

## Solution Assessment

Yes, keeping this open for a core sessions/compaction fix is the best direction; a provider-only patch or one-file clamp would be too narrow without deciding trigger metadata and yielded-parent wake semantics.
