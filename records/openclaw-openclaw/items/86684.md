## Label Justifications

- `P1`: The issue reports a high-impact regression in subagent-heavy workflows where parent sessions can compact unexpectedly during completion wake handling.
- `impact:session-state`: The affected behavior changes parent session branch continuity, compaction state, and yielded-session wake semantics.
- `impact:data-loss`: Unexpected compaction can remove exact parent transcript history even when the visible session reports low context usage.

## Summary

Keep open: current main and v2026.6.6 still have a source-backed path where a successful assistant turn with inflated `usage.input + cacheRead` can trigger overflow compaction, and the requested parent-wake trigger metadata is not represented in the current compaction reason set. The adjacent cron last-call usage fix is merged, but it touched cron session totals only and does not settle this subagent/session wake behavior.

## What This Changes

The issue asks OpenClaw to prevent `sessions_yield` subagent completion wakes from compacting parent sessions at low context usage and to record the compaction trigger cause.

## Best Possible Solution

Define the successful-turn overflow and compaction-trigger contract, then fix parent wake handling so subagent completion can notify/resume without compacting unless real context pressure exists, with focused source tests and one live subagent proof.

## Reproduction Assessment

No high-confidence minimized live repro was established in this review. The source path is reproducible: a successful same-model assistant message whose `usage.input + cacheRead` exceeds the context window will still drive overflow auto-compaction, matching the reporter's impossible usage evidence.

## Solution Assessment

Unclear. The narrowest maintainable fix likely needs both a defensive corrupted-usage policy and trigger metadata, but maintainers should decide how that interacts with providers that rely on successful-turn overflow detection.
