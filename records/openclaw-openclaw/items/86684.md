## Label Justifications

- `P1`: The issue reports a high-impact regression that can unexpectedly compact active parent-session history during real subagent workflows.
- `impact:data-loss`: Unexpected compaction can remove exact parent-session transcript history even when visible context usage is far below the model window.
- `impact:session-state`: The affected behavior is yielded parent transcript state, hidden branch leaves, and compaction parentage during subagent wake-up.

## Summary

Keep open: current main and v2026.6.6 still contain the source-backed overflow and compaction metadata gaps described here, and the merged cron accounting fix only covers an adjacent isolated-agent path.

## What This Changes

Prevent subagent completion and `sessions_yield` wake-ups from compacting a parent session at low context usage, and persist the compaction trigger when compaction is legitimate.

## Best Possible Solution

Make one owner-approved fix that separates parent final-call usage from subagent aggregate usage, ignores impossible usage snapshots for overflow compaction, records compaction trigger cause, and proves yielded-parent wake does not compact below budget.

## Reproduction Assessment

Yes: this is source-reproducible from the detailed live logs and current code paths where inflated successful-turn usage can still drive overflow compaction. I did not run a live current-main subagent scenario in this read-only pass.

## Solution Assessment

Yes on the requested end state, but the best fix is not a narrow caller patch. It should cover overflow sanity, correct parent usage attribution, compaction trigger metadata, and yielded-parent wake proof together.
