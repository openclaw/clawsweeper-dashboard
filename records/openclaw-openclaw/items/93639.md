## Summary

Keep open: current main and the latest release still lack the fix, but this PR is not merge-ready because the latest diff still prepends raw configured prefixes and can leak template placeholders that normal replies resolve with context.

## What This Changes

The PR applies the existing `messages.responsePrefix` resolution to outbound `message(action=send)` payloads and adds core-send tests for prefixed, idempotent, and media-only sends.

## Best Possible Solution

Apply the prefix in the shared send path only after resolving the same cascade and template context used by visible replies, with coverage for static, idempotent, media-only, and templated prefixes.

## Reproduction Assessment

Yes. Source inspection shows current main sends `message(action="send")` text without any `responsePrefix` handling, while normal replies resolve and prepend the configured prefix through the reply normalizer.

## Solution Assessment

No. The PR is in the right shared send layer for static prefixes, but the best fix must also resolve templated prefix values with the same context normal replies use or intentionally avoid applying unresolved templates.
