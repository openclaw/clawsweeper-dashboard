## Summary

Keep open: this is a focused fix in the right shared outbound path for static prefixes, but it is not merge-ready because it prepends the raw configured prefix and would render documented responsePrefix template variables literally on message-tool sends.

## What This Changes

The PR prepends the configured `messages.responsePrefix` to outbound `message(action=send)` payloads in `handleSendAction` and adds core-send tests for static prefixing, idempotence, and media-only sends.

## Best Possible Solution

Keep the shared `handleSendAction` fix shape, but resolve the effective prefix through the same template/context behavior as direct replies before merge, with regression coverage for static, account/channel, and model/provider template prefixes.

## Reproduction Assessment

Yes. Source inspection shows current main direct replies resolve and apply `messages.responsePrefix`, while message-tool sends build and dispatch text without that prefix path; the PR’s remaining defect is also source-reproducible from the missing template context.

## Solution Assessment

No. The chosen shared outbound path is the right layer for static prefixes, but the implementation must resolve documented responsePrefix templates with current attempt context before it is the best fix.
