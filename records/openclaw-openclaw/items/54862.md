## Summary

Keep open: current main still has no OpenRouter `autoRouter.allowedModels` support, but this PR remains stale/dirty and has concrete correctness and boundary blockers before it can merge.

## What This Changes

The branch adds an OpenRouter stream wrapper that injects `plugins: [{ id: "auto-router", allowed_models: [...] }]` from `params.autoRouter.allowedModels`, exports the helper, and adds OpenRouter extra-param tests.

## Best Possible Solution

Land a rebased OpenRouter plugin implementation that gates the auto-router plugin to canonical Auto requests, preserves current OpenRouter wrapper behavior, exposes any shared helper through `extensions/openrouter/api.ts`, and documents the new provider config.

## Reproduction Assessment

Yes. Under the PR head, source inspection shows `autoRouter.allowedModels` is consumed from merged OpenRouter extra params and injected without an Auto-route guard, so a shared config can affect fixed OpenRouter model calls.

## Solution Assessment

No. The feature direction matches OpenRouter's documented API, but this patch is not the safest merge shape until Auto-only gating, wildcard reasoning handling, API-seam usage, current-main wrapper preservation, docs, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard auto-router injection to Auto model requests:** `extensions/openrouter/stream.ts:106`
  - body: `autoRouter.allowedModels` is an Auto Router setting, but this block runs for every OpenRouter model with a non-empty allowlist. Since shared/default params are merged before provider wrapping, fixed-model requests can receive the `auto-router` plugin and have reasoning behavior changed; gate this wrapper to canonical Auto model ids before injecting.
  - confidence: 0.91
- **[P2] Import the helper through the plugin API seam:** `src/agents/pi-embedded-runner-extraparams.test.ts:4`
  - body: This core agent test imports `injectAutoRouterPlugin` from the OpenRouter extension entrypoint. Core tests that need bundled plugin helpers should use a stable plugin `api.ts` seam or keep the assertion inside the plugin tests, otherwise entrypoint refactors can break unrelated core tests.
  - confidence: 0.88
- **[P2] Treat provider-agnostic Grok wildcards as reasoning unsafe:** `extensions/openrouter/stream.ts:57-59`
  - body: The PR advertises wildcard allowlists and OpenRouter documents provider-agnostic patterns, but this helper only recognizes explicit `x-ai/` entries. A pattern like `*/grok-*` can still route Auto Router to Grok while leaving proxy reasoning enabled, so requests can fail on the same unsupported-reasoning path this guard is meant to avoid.
  - confidence: 0.85
- **[P2] Preserve current OpenRouter stream wrappers when rebasing:** `extensions/openrouter/stream.ts:128`
  - body: Current main wraps OpenRouter streams with Anthropic prefill cleanup, DeepSeek V4 thinking replay handling, and the Hunter Alpha reasoning guard. This PR head only forwards `allowlistBlocksReasoning` into the shared OpenRouter wrapper, so the rebase should add auto-router injection inside the current wrapper chain instead of replacing those current-main protections.
  - confidence: 0.82
- **[P3] Document the canonical OpenRouter Auto config:** `extensions/openrouter/stream.ts:14-16`
  - body: This adds a user-facing provider config option, but the PR has no provider-docs change and the new JSDoc example uses the stale duplicated `openrouter/openrouter/auto` ref. Document `params.autoRouter.allowedModels` in `docs/providers/openrouter.md` using the canonical `openrouter/auto` ref so users can discover the setting safely.
  - confidence: 0.8
