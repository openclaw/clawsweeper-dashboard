## Summary

Keep this PR open: current main still lacks OpenRouter `autoRouter.allowedModels`, but the branch is not ready because it applies the payload to non-auto OpenRouter models, drops current main's OpenRouter wrapper chain if rebased as-is, crosses the plugin/core test boundary, lacks public docs, and still has no inspectable real behavior proof.

## What This Changes

The PR adds OpenRouter `params.autoRouter.allowedModels` handling that injects an `auto-router` plugin payload, exports the helper, and adds OpenRouter plus agent extra-params tests.

## Best Possible Solution

Land a rebased provider-local implementation that injects `allowed_models` only for `openrouter/auto`, preserves existing OpenRouter stream wrappers, uses `api.ts` or plugin-owned tests for the helper seam, documents the params shape, and includes redacted live OpenRouter proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows the unconditional injection path, stale wrapper composition, and direct core-test import across the plugin boundary. No for the claimed live OpenRouter behavior, because no inspectable after-fix artifact is attached.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best merge path until it is gated to Auto Router requests, rebased over current wrappers, moved through the proper plugin seam or plugin-owned tests, documented, and backed by live proof.

  - body: `autoRouter.allowedModels` is read from merged params and injected for every OpenRouter model.   - confidence: 0.9

  - body: The new core agent test imports `injectAutoRouterPlugin` from the OpenRouter extension entrypoint. Core tests should use a plugin `api.ts` seam or keep this assertion in plugin-owned tests, so entrypoint refactors do not break unrelated core coverage.
  - confidence: 0.88
