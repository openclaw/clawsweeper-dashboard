## Summary

Keep open: current main still lacks the requested OpenRouter Auto Router allowlist surface, and the PR is useful source work, but the current branch changes OpenRouter request shaping too broadly and still has concrete provider-plugin, test-boundary, docs, and changelog blockers.

## What This Changes

The PR adds an OpenRouter `params.autoRouter.allowedModels` option that injects an OpenRouter `auto-router` plugin payload and adds provider extra-param tests.

## Best Possible Solution

Land a revised OpenRouter plugin implementation that gates allowlist injection to documented Auto routes, keeps existing reasoning safeguards, exposes shared helpers through the plugin API seam or moves tests to the plugin, and documents the new config with a changelog entry.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a clear current-branch reproduction for the blocking defect: configure a shared `autoRouter.allowedModels` value and call a fixed OpenRouter model, and the unguarded wrapper can still inject the Auto Router plugin and alter reasoning.

## Solution Assessment

No. The feature direction matches OpenRouter’s documented API, but the current PR is not the safest merge shape until Auto-only gating, API-seam usage, wildcard-safe reasoning handling, current-main guard preservation, docs, and changelog are addressed.
