## Summary

Keep open: current main still has the source-visible gateway close/response race, but this external PR is conflicting, lacks real behavior proof, fails formatting, and needs to preserve newer typed gateway close-error handling before merge.

## What This Changes

The branch adds request-in-flight deferral for normal gateway closes in `src/gateway/call.ts` plus a mocked `cron.list` close-race regression test.

## Best Possible Solution

Land a rebased narrow gateway close-race fix that lets confirmed successful responses win over a normal close while preserving `GatewayTransportError` metadata and diagnostics for no-response closes, with focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current main sets `ignoreClose` only after the `await client.request()` continuation, and the PR's test models `onClose(1000, "")` during request settlement. I did not run tests because this review is read-only.

## Solution Assessment

No for the current patch. The deferral direction is plausible, but it needs to be rebased onto current main, keep typed wrapper close errors, and avoid letting code-1000 no-response closes fall through as raw client errors.
