## Summary

Keep open: the performance goal is valid, but the branch currently gates Gateway startup on a duplicated static migration probe that misses an existing Microsoft Teams feedback-learning migration, so existing users could skip automatic upgrade migration.

## What This Changes

The PR makes `gateway run` skip the doctor state-migration preflight unless a new lightweight detector finds known legacy state, and adds detector-focused tests.

## Best Possible Solution

Keep the Gateway fast path aligned with the existing doctor/plugin migration detector, or add a shared cheap plugin-owned legacy-input contract with regression coverage for every current plugin doctor migration source before skipping the preflight.

## Reproduction Assessment

Yes, from source inspection: a setup with only Microsoft Teams `*.learnings.json` legacy files has a current plugin doctor migration, while the PR's Gateway gate does not recognize that source and would skip the preflight. I did not execute the path because this review is read-only.

## Solution Assessment

No; the static detector is not the best fix because it duplicates plugin doctor contracts and already drifts from them. The safer repair is to share plugin-owned legacy detection or leave Gateway preflight enabled until coverage is complete.
