## Summary

Keep this PR open, but do not merge as-is: the prior Greptile comments appear addressed, while current review still finds security and config-contract blockers plus stale exec-host integration against current main.

## What This Changes

The PR adds optional default-on tirith pre-exec scanning for gateway, sandbox, and node exec through a new `tools.exec.commandSecurity` config object, tests, and config schema/type plumbing.

## Best Possible Solution

A maintainer-approved design should decide whether this belongs in core or a plugin/API hook, then integrate with current exec phase boundaries using a least-privilege scanner environment, explicit daemon/network/cwd behavior, complete config docs, tests, and changelog.

## Reproduction Assessment

Not applicable. This is a security hardening feature PR rather than a current-main bug report, and source inspection confirms current main has no `tools.exec.commandSecurity` surface.

## Solution Assessment

No. The direction may be useful, but the current patch is not the narrowest maintainable solution until scanner privilege, upstream tirith runtime behavior, config contract drift, and the current node-exec refactor are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Sanitize the tirith child-process environment:** `src/agents/bash-tools.exec.ts:414`
  - body: The scanner is launched before approval using `coerceEnv(process.env)`, so the optional tirith binary can see provider tokens and host-control variables that normal host exec removes with `sanitizeHostExecEnvWithDiagnostics`. Build a least-privilege env for both gateway/sandbox and node scanner invocations.
  - confidence: 0.9
- **[P2] Disable tirith daemon and background update paths:** `src/security/command-security.ts:39`
  - body: The wrapper runs `tirith check --json --non-interactive` without `--no-daemon` or an equivalent update/network opt-out. Upstream `tirith check` can spawn background threat-db updates and delegate to a daemon, which contradicts the PR's no-network/no-cloud security claim before approval.
  - confidence: 0.84
- **[P2] Run the scanner from the exec workdir:** `src/security/command-security.ts:40-42`
  - body: `execFile` receives only timeout/env, and the wrapper has no cwd parameter. Because tirith discovers policy from its current directory/cwd, commands with an explicit exec workdir can be scanned under the gateway cwd and miss the matching project policy.
  - confidence: 0.78
- **[P2] Update the public config contract:** `src/config/zod-schema.agent-runtime.ts:414-424`
  - body: This adds user-facing `tools.exec.commandSecurity` keys but only wires types, zod, and labels. Schema help, generated config baselines, docs, and the changelog still omit the new surface, so `openclaw config schema` and user docs will drift.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a new pre-approval local binary invocation and needs security changes before merge.

Concerns:

- **[high] Scanner receives unsanitized OpenClaw environment:** `src/agents/bash-tools.exec.ts:414`
  - body: The proposed scanner child process inherits `process.env`, which can expose credentials and host-control variables to an optional external binary before the normal host exec env sanitizer runs.
  - confidence: 0.9
- **[medium] External scanner may use daemon or network update paths:** `src/security/command-security.ts:39`
  - body: The invocation does not pass `--no-daemon` or disable tirith background threat-db update behavior, so the scanner can spawn additional local work or network fetches before approval despite the stated no-network scope.
  - confidence: 0.84
- **[medium] Project tirith policy can be ignored:** `src/security/command-security.ts:40`
  - body: The wrapper does not pass the effective command cwd into tirith, so project-local tirith policy discovery can run from the gateway process cwd instead of the requested exec workdir.
  - confidence: 0.78
