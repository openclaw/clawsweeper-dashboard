## Summary

Keep open. Current main does not implement the Tirith/commandSecurity surface, so the PR is not obsolete, but the current diff still has a blocking security-boundary regression: gateway exec can launch the configurable scanner before the effective `security=deny` guard runs. Earlier review feedback about node ordering and bounded findings appears addressed by PR commits, but this PR still needs maintainer/secops review before merge.

## What This Changes

The PR adds default-on optional Tirith pre-exec command scanning to gateway, sandbox, and node exec paths, plus `tools.exec.commandSecurity` config schema/types/labels, warning-to-approval plumbing, and command-security tests.

## Best Possible Solution

A mergeable version should resolve effective gateway and node deny policy before invoking any scanner process, keep the scanner additive after existing hard denial and host-env guards, add a regression proving `security=deny` does not start Tirith, document default/fail-open behavior, and get secops agreement on whether this belongs in core by default.

## Reproduction Assessment

Yes for the blocking PR regression. Set effective gateway exec policy to `security=deny` with command security enabled/default-on and invoke gateway exec; by diff inspection, `checkCommandSecurity` runs before `processGatewayAllowlist` reaches the current deny resolver.

## Solution Assessment

No. Pre-exec scanning is a plausible defense-in-depth direction, but launching a configurable scanner before the established deny guard is not the safest implementation; the scanner should run only after effective deny policy has been resolved.
