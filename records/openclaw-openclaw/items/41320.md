Full review comments:

- **[P1] Make tirith scanning opt-in or prove migration:** `src/agents/bash-tools.exec.ts:405`
  - body: The scanner runs whenever `commandSecurity` is omitted because this condition treats missing config as enabled, and the node hook does the same. Any existing user who already has `tirith` on PATH can start seeing blocked or warning-gated exec commands after upgrade without opting in, so gate this on explicit config or include a maintainer-approved migration/default with fresh-install and upgrade proof.
  - confidence: 0.9
- **[P1] Sanitize the scanner process environment:** `src/agents/bash-tools.exec.ts:413`
  - body: The pre-approval tirith process inherits full `process.env` before the command runs. Current host exec strips dangerous inherited keys, so this can expose provider tokens, package tokens, SSH agent sockets, proxy settings, and scanner-control variables to an optional external binary; build a minimal sanitized scanner env and apply it to both gateway and node hooks.
  - confidence: 0.92
- **[P2] Constrain tirith to side-effect-free preflight:** `src/security/command-security.ts:44`
  - body: The wrapper calls `tirith check` with only `--non-interactive`; upstream `check` can delegate to the daemon, trigger background threat DB update checks, write audit/session state, fetch configured remote policy, and dispatch configured webhooks. Add an explicit no-daemon/offline/no-audit style contract or equivalent env policy before using this as a pre-approval gate.
  - confidence: 0.88
- **[P2] Run tirith from the effective command cwd:** `src/security/command-security.ts:45`
  - body: `execFile` is called without `cwd`, and callers do not pass the resolved command workdir. Tirith policy discovery can therefore use OpenClaw's process cwd instead of the command's effective directory, missing project `.tirith` policies or loading the wrong policy before deciding allow, warn, or block.
  - confidence: 0.87
