## Summary

Keep open. The hardening idea is not implemented on current main and remains useful, but this PR still changes the exec security boundary by default, runs a third-party scanner before approval with raw environment and unconstrained side effects, lacks posted real behavior proof, and is currently conflicting.

## What This Changes

Adds a `tools.exec.commandSecurity` config and tirith-backed pre-exec scanning hooks across gateway, sandbox, and node exec paths with approval-warning tests.

## Best Possible Solution

Use a maintainer-approved command-scanner contract that is opt-in or safely migrated, runs with minimal sanitized env and effective cwd, disables scanner side effects for preflight, and is proven with real OpenClaw runtime output after rebase.

## Solution Assessment

Is this the best way to solve the issue? No. A tirith-backed scanner may be useful, but this implementation should not merge until the default policy, sanitized environment, cwd-aware scanning, and side-effect-free preflight contract are settled and proven in a real OpenClaw run.
