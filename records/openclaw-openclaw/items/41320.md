## Summary

Keep open: current main still lacks the Tirith command-security surface, and this PR is not mergeable as-is because it can launch a configurable scanner before effective gateway `security=deny` is enforced.

## What This Changes

The PR adds default-on optional Tirith pre-exec scanning across gateway, sandbox, and node exec paths, plus config schema/types/labels, warning-to-approval plumbing, and tests.

## Best Possible Solution

Resolve effective gateway and node deny policy before any scanner process, keep scanning additive after hard-denial and host-env guards, document default/fail-open behavior, and get maintainer/secops review before merge.

## Reproduction Assessment

Yes for the blocking PR regression. With gateway exec effectively set to `security=deny` and command security enabled, the PR diff shows `checkCommandSecurity` running before `processGatewayAllowlist` reaches the current deny resolver.

## Solution Assessment

No. Pre-exec scanning is plausible defense in depth, but the current placement is not the safest implementation because it starts the scanner before the established deny guard.
