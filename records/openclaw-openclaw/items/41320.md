## Summary

The PR is not obsolete because current main still lacks this feature, but it should stay open for contributor proof and maintainer security/product review. The pre-approval scanner boundary still exposes concrete security concerns around inherited environment, side effects, and policy discovery.

## What This Changes

The PR adds a `tools.exec.commandSecurity` config path and invokes tirith before gateway, sandbox, and node exec commands, with warning approval plumbing and tests.

## Best Possible Solution

Land a maintainer-approved scanner seam only after it is opt-in or explicitly approved as default-on, uses least-privilege env, scans from the effective command cwd, invokes tirith in a side-effect-free/offline mode, aligns docs/tests/config, and includes redacted real behavior proof.

## Reproduction Assessment

For the feature itself, reproduction is not applicable because current main has no `commandSecurity` surface. For the review blockers, yes: PR-head and upstream tirith source inspection show the scanner env, side-effect, and cwd paths with high confidence.

## Solution Assessment

No, not as written. The direction may be useful, but a mergeable core solution needs a least-privilege and side-effect-free scanner contract, effective-cwd policy discovery, real behavior proof, and maintainer security/product approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Sanitize the scanner process environment:** `src/agents/bash-tools.exec.ts:413`
  - body: The pre-approval tirith process is launched with a full `coerceEnv(process.env)`, while current host exec hardening strips sensitive inherited keys before running commands. This can expose provider tokens, package tokens, SSH agent sockets, and tirith control env to an optional external binary; build a minimal/sanitized scanner env and apply it to both gateway and node hooks.
  - confidence: 0.91
- **[P2] Disable scanner daemon and network side effects:** `src/security/command-security.ts:44`
  - body: The wrapper calls default `tirith check` without `--no-daemon` or an offline/no-update contract. Upstream `check` can spawn background threat DB updates, delegate to a daemon, fetch remote policy, and perform daemon network enrichment, which is too much side effect for a local pre-approval gate.
  - confidence: 0.88
- **[P2] Scan from the command's effective cwd:** `src/security/command-security.ts:45-50`
  - body: `execFile` is called without `cwd`, and callers do not pass the resolved command workdir. Tirith policy discovery can therefore use OpenClaw's process cwd instead of the command's effective directory, missing project `.tirith` policies or loading the wrong policy before deciding allow, warn, or block.
  - confidence: 0.86
