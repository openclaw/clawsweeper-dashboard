## Summary

Keep open. Current main still lacks the Tirith command-security surface, and the PR is a security-sensitive core exec change with a blocking ordering defect: the gateway path can launch the configurable scanner before the existing effective `security=deny` guard runs.

## What This Changes

The PR adds default-on optional Tirith pre-exec command scanning across gateway, sandbox, and node exec paths, plus `tools.exec.commandSecurity` config schema/types/labels and command-security tests.

## Best Possible Solution

A mergeable version should resolve effective gateway and node deny policy before invoking any scanner process, add a regression test proving `security=deny` does not start Tirith, document the operator-facing default/fail-open behavior, and get secops agreement on the core trust model.

## Reproduction Assessment

Yes for the blocking PR regression. Set effective gateway exec policy to `security=deny` with command security enabled/default-on and invoke a gateway exec; by diff inspection, `checkCommandSecurity` runs before `processGatewayAllowlist` reaches the current deny resolver.

## Solution Assessment

No. The security-in-depth direction is plausible, and the earlier node-ordering/output-bounding feedback appears addressed, but the current placement is not the safest implementation because it launches a configurable scanner before the existing effective deny guard.

## Review Findings

Overall correctness: patch is incorrect
