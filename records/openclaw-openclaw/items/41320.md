## Summary

Keep open. Current main still lacks this Tirith command-scanning surface, and the PR has a blocking exec security regression because gateway hard-deny policy is evaluated after the new scanner can be launched.

## What This Changes

Adds optional default-on Tirith pre-exec command scanning, `tools.exec.commandSecurity` config, warning-to-approval plumbing, and tests across gateway, sandbox, and node exec paths.

## Best Possible Solution

Keep the PR open for security review; if pursued, resolve effective hard-deny before any scanner process starts, preserve existing approval semantics, document the trust and fail-open model, and rerun targeted exec approval coverage plus the changed gate.

## Reproduction Assessment

Yes, for the blocking PR regression from source inspection. The raw PR patch invokes `checkCommandSecurity` before gateway `processGatewayAllowlist`, while current main throws for effective `security=deny` inside `resolveExecHostApprovalContext`.

## Solution Assessment

No. Pre-exec scanning may be useful defense in depth, but this placement is not the narrowest safe implementation because it starts the scanner before the established hard-deny guard.
