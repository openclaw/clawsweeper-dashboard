## Summary

Current main still lacks the requested intentional STOP-driven repeated-run mode. The recently added per-agent `runRetries` setting is a failure-recovery guard only, and the discussion identifies unresolved interruption, safety, and ownership semantics that require maintainer product judgment.

## What This Changes

Add a per-agent default iteration limit for intentionally rerunning an agent on the same prompt until it emits an explicit STOP signal.

## Best Possible Solution

Define a typed intentional-run lifecycle mode, separate from failure-recovery `runRetries`, that shares canonical terminal-outcome, interruption, transcript, and cancellation machinery; then either implement it generically in the embedded runner with bounded per-agent defaults or expose the precise plugin runtime hook needed to own it outside core.

## Reproduction Assessment

Yes for the feature-gap claim: current source, schema, and docs show only failure-recovery retries, not intentional same-prompt reruns until STOP. This is source-reproducible rather than a failing current behavior test because the item requests a new capability.

## Solution Assessment

No, the proposed direct Kimi port is not yet the best OpenClaw solution. Copying its field names, `-1` sentinel, and precedence would bypass OpenClaw's canonical terminal-outcome and interruption contracts; maintainers should first choose an explicit core lifecycle mode or a generic plugin seam.

## Risks / Open Questions

- The STOP signal format, whether successful output is persisted or delivered on every iteration, and how queued user input interrupts or replaces the loop are undefined.
- An unlimited mode could create unbounded token, tool, and side-effect execution unless separate operator-visible safety limits and cancellation behavior are specified.
