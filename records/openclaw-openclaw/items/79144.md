## Summary

Keep this PR open: the patch is in the right central update-check path, but it still only honors registry environment variables and misses the npmrc-backed `npm config get registry` case from the linked report.

## What This Changes

The PR adds a shared npm registry URL resolver for OpenClaw update metadata checks, wires `fetchNpmPackageTargetStatus` through it, adds regression coverage, and adds a changelog entry.

## Best Possible Solution

Keep the centralized resolver approach, but extend it to match npm/pnpm registry precedence for both registry env vars and npmrc-backed registry config, with focused tests for userconfig and fallback cases.

## Reproduction Assessment

Yes. Current main source shows the update metadata helper hardcodes `registry.npmjs.org`, and startup/status/update preflight callers flow through that helper; this read-only review did not run repo tests.

## Solution Assessment

No. The central resolver is the right direction, but this implementation is incomplete because it misses npmrc-backed registry configuration while adding an env key that npm and pnpm do not honor as a registry setting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor npmrc registry config in the resolver:** `src/infra/update-channels.ts:41-44`
  - body: The resolver only checks `npm_config_registry`, `NPM_CONFIG_REGISTRY`, and `npm_config_userconfig_registry`, so a user whose mirror is set by npmrc or an npm userconfig file still falls through to `registry.npmjs.org`. That leaves the `~/.npmrc` and `npm config get registry` case from https://github.com/openclaw/openclaw/issues/79140 broken, and `npm_config_userconfig_registry` is not a registry key npm or pnpm honor.
  - confidence: 0.88
