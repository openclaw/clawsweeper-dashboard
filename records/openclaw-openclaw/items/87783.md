Keep open: current main and the latest release still rethrow Slack dispatch errors without the requested explicit-mention timeout fallback, and the linked implementation PR is open and not merged.

## What This Changes

The issue requests a visible Slack fallback for explicit human room/channel mentions when the routed handler times out before producing any visible reply.

## Best Possible Solution

Land an accepted dispatcher-level Slack fallback, or an equivalent replacement, that posts one deduped visible timeout reply only for explicit human mentions with no prior visible output.

## Reproduction Assessment

Yes at source level: current main catches `dispatchChannelInboundReply` failures, computes visible delivery state, and rethrows without posting a timeout fallback. I did not run a live Slack workspace timeout reproduction in this read-only review.

## Solution Assessment

Yes as a narrow owner-boundary fix shape, but it is not implemented on main yet. The Slack dispatcher has the mention facts, visible-delivery state, reply planner, and retry boundary needed for the fallback; the active PR still needs refresh, proof, and maintainer acceptance.
