## Label Justifications

- `P1`: The linked bug can leave a real user's channel session unable to process messages until manual reset.
- `merge-risk: 🚨 session-state`: The diff mutates durable session recovery markers and can make interrupted session state unrecoverable after retries are exhausted.
- `merge-risk: 🚨 message-delivery`: If the recovery marker is cleared without releasing the stale owner, queued or subsequent channel messages can remain unprocessed.

## Summary

Keep open: the patch is not safe to merge because it erases the durable restart-recovery marker after failed recovery attempts while leaving the session row running, so later startup recovery can no longer retry the interrupted session. The linked report remains valid, but the safer repair boundary is the stale owner or lane release path, plus real restart proof.

## What This Changes

The PR adds a startup-recovery helper that clears `abortedLastRun` from running session-store rows after restart recovery retries are exhausted.

## Best Possible Solution

Keep `abortedLastRun` as the retry marker until recovery succeeds or explicitly marks the session failed, and solve the stuck behavior by releasing the stale run/lane owner with restart regression proof.

## Reproduction Assessment

Yes at source level: current recovery uses `abortedLastRun` as the durable selector for restart recovery, and the linked report shows a real post-restart stuck session. I did not establish a full live gateway-restart reproduction in this read-only review.

## Solution Assessment

No. The proposed marker clear is too broad; the maintainable fix should make the stale owner or lane dispatchable, or mark the session failed visibly, while preserving retry state on transient recovery failures.
