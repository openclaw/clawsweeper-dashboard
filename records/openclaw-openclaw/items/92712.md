## Summary

Keep open: the linked restart busy-state bug is valid, but this PR should not merge as written because it clears the durable restart-recovery marker after recovery has failed and can make still-running wedged sessions invisible to the next recovery pass.

## What This Changes

The PR adds a post-retry cleanup helper in `src/agents/main-session-restart-recovery.ts` that clears `abortedLastRun` from running sessions when startup recovery exhausts retries or throws.

## Best Possible Solution

Preserve `abortedLastRun` until resume succeeds or the session is explicitly failed, and fix the stale run or command-lane owner path with real restart proof.

## Reproduction Assessment

Yes at source level: current recovery selects marked running rows by `abortedLastRun`, and the linked report provides a concrete post-restart stuck-session scenario. I did not run a destructive live mid-turn restart in this read-only review.

## Solution Assessment

No. The PR is a plausible symptom clear, but it is not the best fix because it removes retry state after failed recovery instead of releasing the stale run/lane owner or marking the session failed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the recovery marker after failed retries:** `src/agents/main-session-restart-recovery.ts:893-896`
  - body: This branch calls `clearStaleAbortedRunFlags` when recovery still reports failures. Current recovery only revisits rows with `status === "running"` and `abortedLastRun === true`, so deleting the marker on a still-running row after a transient resume/read/gateway failure can make the row invisible to the next startup recovery pass while it remains wedged.
  - confidence: 0.9
- **[P1] Use the recovery eligibility filters before clearing markers:** `src/agents/main-session-restart-recovery.ts:840-850`
  - body: The helper loops every resolved store and clears any running aborted row without the skip, routability, active-owner, or duplicate-session checks that `recoverStore` applies. If one failed row exhausts retries, this can mutate skipped current-process or non-routable rows that recovery deliberately preserved.
  - confidence: 0.86
