## Summary

Keep open: the branch should not merge as-is because it clears the restart-recovery marker after recovery has failed, which can hide a still-running wedged session from later recovery and bypass the filters current recovery uses before mutating rows.

## What This Changes

The PR adds a helper that clears `abortedLastRun` from running sessions after main-session restart recovery exhausts retries or gives up.

## Best Possible Solution

Preserve `abortedLastRun` until resume succeeds or the session is explicitly failed, fix the stale owner or lane-release path, and prove an actual restart plus follow-up message.

Full review comments:

- **[P1] Preserve the recovery marker after failed retries:** `src/agents/main-session-restart-recovery.ts:890-896`
  - body: This branch calls `clearStaleAbortedRunFlags()` when recovery still reports failures. Current recovery only revisits rows with `status === "running"` and `abortedLastRun === true`, so deleting the marker on a still-running row after a transient resume, read, or gateway failure can make the row invisible to the next startup recovery pass while it remains wedged.
  - confidence: 0.9
- **[P1] Use the recovery eligibility filters before clearing markers:** `src/agents/main-session-restart-recovery.ts:844-848`
  - body: The helper loops every resolved store and clears any running aborted row without the skip, routability, active-owner, or duplicate-session checks that `recoverStore()` applies. If one failed row exhausts retries, this can mutate skipped current-process or non-routable rows that recovery deliberately preserved.
  - confidence: 0.86
