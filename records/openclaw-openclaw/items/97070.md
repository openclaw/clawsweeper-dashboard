## Summary

This PR should stay open: the SDK fetch hook is the right layer and partially addresses the unbounded SSE stream risk, but the helper skips non-ok SSE responses that the OpenAI SDK still reads with an unbounded `response.text()` path, and the PR body only provides mocked/Vitest proof.

## What This Changes

The branch adds a shared `createSdkBoundedFetch` helper, tests it, and injects it into both Azure OpenAI Responses SDK client constructors to cap successful SSE response bodies at 16 MiB.

## Best Possible Solution

Keep the SDK fetch-hook approach, but cap SSE bodies regardless of HTTP status, add redacted runtime proof for normal and oversized Azure Responses paths, and have maintainers explicitly accept or tune the 16 MiB stream cutoff before merge.

## Reproduction Assessment

Yes, from source inspection. Current main constructs Azure Responses SDK clients without a custom fetch, and `openai@6.39.1` has both SSE accumulation and an unbounded non-ok `response.text()` error-body read path; I did not run a live hostile Azure endpoint.

## Solution Assessment

No. The SDK fetch hook is the right layer, but this implementation is too narrow because it leaves non-ok SSE bodies unbounded and still needs real runtime proof plus maintainer acceptance of the stream cutoff.
