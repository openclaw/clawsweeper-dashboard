## Summary

Keep open: the PR is a focused and useful Azure Responses hardening attempt, but it is not merge-ready because the new wrapper explicitly bypasses non-ok SSE responses even though the OpenAI SDK reads non-ok bodies with an unbounded `response.text()`, and the submitted proof is mock-only.

## What This Changes

The PR adds a shared SDK fetch wrapper, tests it, and injects it into both Azure OpenAI Responses SDK client constructors to cap successful SSE response bodies at 16 MiB.

## Best Possible Solution

Keep the SDK fetch-hook approach, but cap SSE bodies regardless of HTTP status, add real local-runtime proof for both Azure client branches, and make the 16 MiB cutoff an explicit maintainer-accepted tradeoff before merge.

## Reproduction Assessment

Yes from source inspection. Current main constructs Azure Responses SDK clients without a custom fetch, and `openai@6.39.1` has both SSE accumulation and unbounded non-ok error-body reads; I did not run a live hostile Azure endpoint.

## Solution Assessment

No. The SDK fetch hook is the right layer, but this implementation is too narrow because it leaves non-ok SSE bodies unbounded; the best fix is to cap all SSE statuses and prove the production provider path.
