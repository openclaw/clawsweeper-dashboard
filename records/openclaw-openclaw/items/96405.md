## What This Changes

The PR changes `parseCommandArgs` to track token presence separately from token text so quoted empty strings remain positional prompt-template arguments, with a focused regression test.

## Best Possible Solution

Land one final shared-parser change that preserves quoted empty arguments, keeps the overlapping apostrophe-parser work from regressing it, and includes after-fix real prompt-template expansion proof.

## Reproduction Assessment

Yes at source level: current main only pushes non-empty `current` strings, so `parseCommandArgs('first "" third')` drops the blank argument before substitution. I did not run a live repro because this read-only review avoided commands that could create artifacts.

## Solution Assessment

Yes, with merge gates: tracking token presence separately is the narrowest fix for empty quoted arguments. If the broader apostrophe parser PR lands first, this behavior should be folded into that final parser rather than maintained as a competing branch.
