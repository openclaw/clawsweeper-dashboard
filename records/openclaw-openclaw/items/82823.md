## Summary

Keep open: the feature direction is plausible, but the patch currently misses the motivating Telegram delivery path when a caller already provides `beforeDeliver`, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds a public `reply_payload_sending` plugin hook, SDK/docs exports, dispatch wiring, and tests for mutating or canceling normalized `ReplyPayload`s before delivery.

## Best Possible Solution

Keep the PR open, compose plugin hook execution with caller-provided `beforeDeliver`, add regression coverage for that case, and require real plugin/channel proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: on the PR head, `dispatcherOptions.beforeDeliver` short-circuits the new hook chain, and current Telegram callers provide `beforeDeliver`. A focused regression test can reproduce this by registering `reply_payload_sending` and passing a caller `beforeDeliver` through `dispatchInboundMessageWithBufferedDispatcher`.

## Solution Assessment

No. The hook phase is a reasonable seam, but the implementation should compose the caller `beforeDeliver` with the built-in message and reply-payload hooks so existing channel delivery hooks do not bypass the new API.
