## Summary

Keep this PR open: the hook is a plausible SDK feature, but the current wiring skips the new hook on channel paths that already provide `beforeDeliver`, including the Telegram path the PR cites, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a `reply_payload_sending` plugin SDK hook, wires it into auto-reply before-delivery dispatch, exports and documents the new types, and adds hook/dispatch tests.

## Best Possible Solution

Compose caller-provided `beforeDeliver`, `message_sending`, and `reply_payload_sending` in a defined order for both dispatcher wrappers, add regression coverage for a custom-beforeDeliver path such as Telegram, and include real channel proof for the visible payload mutation.

## Reproduction Assessment

Yes. Source inspection shows Telegram supplies an identity `beforeDeliver`, while the PR's nullish-coalescing hook wiring would skip `reply_payload_sending` whenever a caller already provides `beforeDeliver`.

## Solution Assessment

No. The maintainable shape is to compose the existing/custom before-delivery hook with the global plugin hooks instead of making them mutually exclusive.
