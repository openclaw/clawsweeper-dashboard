## Summary

## What This Changes

The PR changes cron next-run recompute so errored recurring jobs use the built-in error-backoff schedule after reload instead of `cron.retry.backoffMs`, with a regression test for that behavior.

## Best Possible Solution

Refresh or replace the draft so reload/recompute and live recurring scheduling preserve configured `cron.retry.backoffMs`, or explicitly align docs, help text, types, and upgrade expectations if maintainers choose a narrower retry contract.

## Reproduction Assessment

Yes. Source inspection reproduces the patch defect: the changed line drops `cron.retry.backoffMs`, and the new test expects the default 30s backoff even when state config sets a 60-minute retry delay.

## Solution Assessment

No. Current main and the merged configured-backoff work make recurring retry backoff config-backed, so removing that argument is not the narrow maintainable fix unless maintainers explicitly change the config contract.
