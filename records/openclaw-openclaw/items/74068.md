## Summary

Keep open: this is a member-authored draft and should not be auto-closed, but the patch remains incorrect because it would make reload/recompute ignore configured `cron.retry.backoffMs` even though current main now treats recurring cron backoff as config-backed.

## What This Changes

The PR changes cron recompute for errored recurring jobs to use the built-in backoff schedule instead of `state.deps.cronConfig.retry.backoffMs`, and adds a regression test expecting the default 30s behavior.

## Best Possible Solution

Refresh or replace the draft so reload/recompute and live recurring scheduling preserve configured `cron.retry.backoffMs`; if maintainers want a narrower contract, update docs, types, help text, tests, and upgrade expectations explicitly.

## Reproduction Assessment

Yes. Source inspection reproduces the patch defect: the changed line drops `cron.retry.backoffMs`, and the added test expects the default 30s backoff even when state config sets a 60-minute retry delay.

## Solution Assessment

No. Current main and the merged configured-backoff work make recurring retry backoff config-backed, so removing that argument is not the narrow maintainable fix unless maintainers explicitly change the config contract.
