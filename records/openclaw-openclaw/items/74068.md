## Summary

Keep open, but this draft is not merge-ready: current main now treats recurring cron retry backoff as config-backed, and this patch would regress custom `cron.retry.backoffMs` during reload/recompute.

## What This Changes

The PR removes configured retry backoff from cron recompute for errored recurring jobs and adds a regression expecting the default 30s backoff despite custom retry config.

## Best Possible Solution

Refresh or replace the draft so reload/recompute and live recurring scheduling both honor configured `cron.retry.backoffMs`, or explicitly align docs, help text, types, and upgrade expectations if maintainers choose a narrower retry contract.

## Reproduction Assessment

Yes. Source inspection reproduces the patch defect: PR head drops the configured schedule argument and its new test expects 30s even with a 60-minute custom `cron.retry.backoffMs`.

## Solution Assessment

No. Current main and the merged configured-backoff work make recurring retry backoff config-backed, so removing that argument is not the narrow maintainable fix without an explicit contract change.
