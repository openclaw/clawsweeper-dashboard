## Summary

## What This Changes

The PR changes cron next-run recompute so recurring errored jobs use the fixed recurring error-backoff schedule after reload and adds a regression test for a recurring `every` job with custom one-shot retry config.

## Best Possible Solution

Land the scoped recompute/test change with a single-line active changelog entry after the draft is ready and validation is green; keep broader recurring transient retry semantics in the existing follow-up items.

## Reproduction Assessment

Yes. On current main, an enabled recurring job with `lastStatus: "error"`, finite `lastRunAtMs`, `consecutiveErrors: 1`, and `cronConfig.retry.backoffMs: [3600000]` flows through `recomputeNextRuns` and uses the one-shot hour-long delay instead of the fixed 30s recurring backoff.

## Solution Assessment

Yes for the code path, with a process gap. Removing the custom retry schedule from recurring recompute is the narrowest maintainable fix because it matches the live recurring timer path and the documented one-shot-only `cron.retry` contract, but the PR still needs the required changelog entry.
