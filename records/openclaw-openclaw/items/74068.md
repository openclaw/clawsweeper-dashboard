## What This Changes

The PR removes one-shot `cron.retry.backoffMs` from recurring cron recompute backoff and adds a regression test for an errored recurring job with a custom one-shot retry backoff.

## Best Possible Solution

Land this scoped cron service/test fix after adding a single-line Unreleased changelog entry and getting green targeted plus changed-gate validation; keep the broader recurring transient-retry semantics in the existing recurring retry follow-ups rather than expanding this PR.

## Reproduction Assessment

Yes. On current main, an enabled recurring job with `lastStatus: "error"`, finite `lastRunAtMs`, `consecutiveErrors: 1`, and `cronConfig.retry.backoffMs: [3600000]` will have `recomputeNextRuns` floor the next run at one hour instead of the fixed 30s recurring backoff.

## Solution Assessment

Yes for the code path, with one process gap. Removing the custom retry schedule from recurring recompute is the narrow maintainable fix because it matches the live recurring timer path and the documented one-shot-only `cron.retry` contract; the safer remaining change is only the changelog entry.
