## Summary

Keep this PR open for maintainer handling because it is a member-authored draft, but it is not merge-ready: current main and the latest release now treat `cron.retry.backoffMs` as the recurring cron retry/backoff contract, while this draft would make recompute ignore that custom config and fall back to the built-in default.

## What This Changes

The PR changes cron next-run recomputation so errored recurring jobs use the default error-backoff schedule after reload and adds a regression test for that recompute behavior.

## Best Possible Solution

Retire this default-only direction or rebase it so recompute preserves the configured `cron.retry.backoffMs` contract already present on main and in `v2026.6.9`.

## Reproduction Assessment

Yes, by source inspection. The PR diff removes the configured retry schedule from recompute and its test demonstrates that a job with custom `cron.retry.backoffMs` would be re-armed at the built-in 30s default.

## Solution Assessment

No. The current main and release behavior is the better fix because it keeps recompute, runtime scheduling, docs, and durable regression coverage aligned on the configured cron retry backoff contract.
