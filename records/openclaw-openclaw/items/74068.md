## Best Possible Solution

## Reproduction Assessment

Yes. On current main, an enabled recurring `every` job with `lastStatus: "error"`, finite `lastRunAtMs`, `consecutiveErrors: 1`, and `cronConfig.retry.backoffMs: [3600000]` will recompute using the one-shot hour-long backoff instead of the fixed 30-second recurring backoff.

## Solution Assessment

Yes for the code path, with one process gap. Calling `errorBackoffMs(consecutiveErrors)` in recurring recompute matches the live recurring path and the documented one-shot-only `cron.retry` contract; the safer merge path is to add the changelog entry before validation/landing.
