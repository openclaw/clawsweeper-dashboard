## Summary

Keep this member-authored draft open for maintainer handling: current main now treats recurring cron retry backoff as config-backed, while this PR would make reload/recompute ignore custom `cron.retry.backoffMs`. The branch is also draft and currently conflicting, so it is not a cleanup-close candidate or an automation-ready repair path.

## What This Changes

The PR changes cron next-run recompute for errored recurring jobs to use the built-in recurring backoff schedule instead of configured `cron.retry.backoffMs`, with a regression test for that behavior.

## Best Possible Solution

Refresh or replace the draft so reload/recompute and live recurring scheduling preserve configured `cron.retry.backoffMs`, or explicitly align docs, help text, types, tests, and upgrade expectations if maintainers choose a narrower retry contract.

## Reproduction Assessment

Yes. Source inspection reproduces the patch defect: the changed line drops `cron.retry.backoffMs`, and the new test expects the default 30s backoff even when state config sets a 60-minute retry delay.

## Solution Assessment

No. Current main and the merged configured-backoff work make recurring retry backoff config-backed, so removing that argument is not the narrow maintainable fix unless maintainers explicitly change the config contract.
