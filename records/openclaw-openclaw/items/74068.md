## Summary

Keep open. Current main still applies the one-shot `cron.retry.backoffMs` schedule when recomputing errored recurring jobs after reload, while the live recurring path and docs use the fixed recurring backoff schedule; the PR is a narrow, plausible fix but needs a changelog entry and normal draft/CI review before merge.

## What This Changes

The PR changes recurring cron recompute to ignore one-shot retry backoff config and adds a regression test for an errored recurring `every` job with custom `cron.retry.backoffMs`.

## Best Possible Solution

Land the scoped service/test fix with a single-line changelog entry once the draft is ready and validation is green; keep broader recurring transient-retry semantics tracked separately in #46341/#46423.

## Reproduction Assessment

Yes. On current main, construct an enabled recurring job with `lastStatus: "error"`, finite `lastRunAtMs`, `consecutiveErrors: 1`, and `cronConfig.retry.backoffMs: [3600000]`, then call `recomputeNextRuns`; the recompute path uses the one-shot hour-long delay instead of the fixed 30s recurring backoff.

## Solution Assessment

Yes for the code path, with one process gap. Removing the custom retry schedule from recurring recompute is the narrowest maintainable fix because it aligns reload behavior with the live recurring timer path and the documented one-shot-only `cron.retry` contract.
