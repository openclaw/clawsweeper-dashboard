## Summary

## What This Changes

The PR changes cron recompute to use the fixed recurring error-backoff schedule for recurring jobs and adds a regression test covering a custom one-shot retry backoff after reload.

## Best Possible Solution

Land the scoped cron service/test fix with a single-line changelog entry, while keeping broader recurring transient-retry behavior tracked separately from this reload/recompute consistency bug.

## Reproduction Assessment

Yes. On current main, construct an enabled recurring `every` job with `lastStatus: "error"`, finite `lastRunAtMs`, `consecutiveErrors: 1`, and `cronConfig.retry.backoffMs: [3600000]`, then call `recomputeNextRuns`; the recompute path uses the one-shot hour-long delay while the live recurring path uses the fixed 30s backoff.

## Solution Assessment

Yes for the code path, with one process gap. Removing the configured retry schedule from recurring recompute is the narrowest maintainable fix because it aligns reload behavior with the live recurring timer path and the documented one-shot-only `cron.retry` contract; the missing changelog should be added before merge.
