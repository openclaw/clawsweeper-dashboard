## Summary

Keep open for maintainer handling: this is a member-authored draft, and the current patch is not merge-ready because it removes configured recurring cron retry backoff from reload/recompute behavior that current main and the latest release now preserve.

## What This Changes

The PR changes cron recompute for errored recurring jobs to use the built-in default backoff instead of configured `cron.retry.backoffMs`, with a regression test for that default-over-config behavior.

## Best Possible Solution

Refresh or replace the draft so reload/recompute and live recurring scheduling preserve configured `cron.retry.backoffMs`, or explicitly align docs, help text, types, tests, and upgrade expectations if maintainers choose a narrower retry contract.

## Reproduction Assessment

Yes. Source inspection reproduces the patch defect: the changed line drops `cron.retry.backoffMs`, and the added test expects the default 30-second backoff even when state config sets a 60-minute retry delay.

## Solution Assessment

No. Current main and the latest release make recurring retry backoff config-backed, so removing that argument is not the narrow maintainable fix unless maintainers explicitly change the config contract.
