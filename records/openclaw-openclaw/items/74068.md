## Summary

Keep open. Current main still applies one-shot `cron.retry.backoffMs` during recurring reload/recompute, while the PR’s code change is a narrow fix for that mismatch; the remaining actionable gap is the required changelog entry before merge.

## What This Changes

The PR changes recurring cron recompute to use the default recurring error-backoff schedule instead of `cron.retry.backoffMs`, and adds a regression test for reload/recompute behavior.

## Best Possible Solution

Land the scoped cron service/test fix with a single-line changelog entry once the draft is ready and targeted plus changed-gate validation are green; keep broader recurring transient-retry behavior on its separate tracking item.

## Reproduction Assessment

Yes. From current-main source, an enabled recurring job with `lastStatus: "error"`, finite `lastRunAtMs`, and `cron.retry.backoffMs: [3600000]` will have recompute use the one-shot hour-long backoff even though the live recurring path uses the fixed recurring backoff.

## Solution Assessment

Yes for the code direction, but not complete as a merge-ready PR. Removing the configured one-shot retry schedule from recurring recompute is the narrowest maintainable fix; the branch still needs the required changelog entry and normal validation.
