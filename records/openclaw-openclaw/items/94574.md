## What This Changes

The PR removes the explicit Slack bot token argument from the monitor startup `auth.test` call so the configured WebClient token is used through the Authorization header.

## Best Possible Solution

Land the narrow Slack monitor fix after redacted real OpenClaw/Slack startup proof confirms `auth.test` still succeeds without serializing the token body.

## Reproduction Assessment

Source-reproducible: current main passes `{ token: botToken }` into the Slack WebClient call, and the SDK serializes options data while already setting the constructor token as the Authorization header. I did not run a live OpenClaw Slack monitor reproduction with real credentials.

## Solution Assessment

Yes: removing the per-call token override is the narrowest owner-local fix because the Bolt app already owns the bot token and sibling Slack paths use `auth.test()` without arguments. The remaining gap is proof, not an obviously better implementation layer.
