## Summary

Keep this PR open for normal review: the patch is the right narrow fix and I found no code or security findings, but the external-PR real behavior proof is still insufficient because the PR body only shows a standalone curl and the Mantis Slack smoke failed before exercising OpenClaw.

## What This Changes

The PR removes the explicit `token` argument from the Slack monitor boot `auth.test` call and adds a Vitest regression test asserting no bot token is passed in the call arguments.

## Best Possible Solution

Land the narrow Slack provider change after redacted after-fix proof shows OpenClaw's Slack `auth.test` request no longer includes the token in the serialized body, while keeping the focused regression test.

## Reproduction Assessment

Yes from source and dependency contract: current main calls `app.client.auth.test({ token: botToken })`, and the Slack SDK serializes call options into the POST body while also setting Authorization. I did not intercept a live OpenClaw Slack request with a real token.

## Solution Assessment

Yes, this is the narrowest maintainable fix: remove the redundant call argument at the Slack monitor boot call while keeping the WebClient constructor token and existing header auth path. Slack's docs explicitly allow Authorization-header auth for `auth.test`. ([docs.slack.dev](https://docs.slack.dev/reference/methods/auth.test/))
