## What This Changes

The PR removes the explicit `botToken` argument from the Slack monitor startup `app.client.auth.test` call so the token-bound Slack WebClient supplies authentication through its Authorization header.

## Best Possible Solution

Land the narrow Slack monitor fix after redacted OpenClaw Slack startup or request-adapter proof confirms `auth.test` still succeeds and the token is not serialized in the request body.

## Reproduction Assessment

Yes, source-reproducible: current main constructs the Bolt app with the bot token and then passes the same token into `auth.test` options, while the Slack SDK serializes options into the request body. I did not run a live OpenClaw Slack monitor with real credentials.

## Solution Assessment

Yes: removing the per-call token override is the narrowest maintainable fix because the Bolt app client already owns the token and sibling Slack paths use `auth.test()` without arguments. A request interceptor or log redaction would be broader and would not remove the duplicate credential transmission at the call site.
