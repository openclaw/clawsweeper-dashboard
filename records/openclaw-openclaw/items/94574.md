## Summary

Keep open: current main and the latest release still send the Slack bot token as an `auth.test` method argument, and the PR is a narrow, dependency-backed fix. The remaining blocker is proof, not code: the contributor's curl shows Slack accepts header-only auth, but it does not show patched OpenClaw behavior, and the linked Mantis run failed during Crabbox warmup before any Slack gateway proof was captured.

## What This Changes

The PR removes the Slack bot token from the `app.client.auth.test` call argument and adds a Slack monitor regression test that rejects passing `token` in that call.

## Best Possible Solution

Keep the narrow provider change, but require redacted after-fix OpenClaw proof or a successful Slack Mantis rerun before merge; no config, migration, or broader Slack startup rewrite is needed.

## Reproduction Assessment

Yes at source level: current main passes `botToken` as an `auth.test` method argument while the Slack app client already has the token-backed Authorization path. I did not run a live Slack token repro in this read-only review.

## Solution Assessment

Yes: removing only the duplicate method argument is the narrowest maintainable fix, and it matches the existing `probeSlack` pattern that calls `client.auth.test()` on a token-backed client. A broader client wrapper or config option would add unnecessary surface for this bug.
