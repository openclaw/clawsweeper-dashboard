## Summary

Keep open: current main still auto-runs embedded fallback after Gateway timeout/transport failures, the requested hard-fail diagnostic control is not implemented, and the linked draft PR is still open rather than merged.

## What This Changes

The issue requests clearer embedded-fallback diagnostics plus an additive `openclaw agent` hard-fail mode such as `--no-fallback` or `OPENCLAW_AGENT_NO_FALLBACK=1`.

## Best Possible Solution

Preserve default fallback compatibility, add an explicit gateway-only hard-fail mode with focused docs/tests/changelog, and decide separately whether broader embedded stderr prefixing is worth the log-shape change.

## Reproduction Assessment

Yes, at source level: current `agentCliCommand` catches Gateway timeout/transport errors and runs the embedded fallback, while CLI registration and docs expose no no-fallback control. I did not run a live Gateway scenario in this read-only review.

## Solution Assessment

Yes, directionally: an additive no-fallback flag/env mode is the narrow maintainable solution because it lets diagnostics fail closed without breaking scripts that rely on default fallback. The linked draft PR is the active implementation path, but it is not merged and still needs proof/review.
