## Summary

Keep open: current main still automatically runs embedded fallback after Gateway transport or timeout failures, and the requested gateway-only diagnostic control has not landed. The open draft PR that targets this behavior is still unmerged and needs review/proof, so this issue should remain the canonical tracker.

## What This Changes

Request an additive `openclaw agent` diagnostic mode that fails closed on Gateway dispatch failure instead of masking it with embedded fallback.

## Best Possible Solution

Land an additive gateway-only no-fallback flag/env mode with clearer diagnostics, focused tests, and docs while preserving default fallback compatibility unless maintainers explicitly change it.

## Reproduction Assessment

Yes, at source level: current `agentCliCommand` catches Gateway transport and timeout failures and invokes embedded fallback, and focused tests assert that behavior. I did not run a live Gateway probe in this read-only review.

## Solution Assessment

Yes for the additive fail-closed control: it is the narrowest maintainable direction for diagnostic probes. Changing default fallback behavior or prefixing all embedded output should remain a separate maintainer decision.
