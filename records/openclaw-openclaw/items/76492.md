## Summary

Keep open: current main still catches Gateway dispatch failures and runs the embedded agent afterward; the existing JSON marker and stderr line help identify fallback runs, but there is still no fail-closed diagnostic mode for probes that must verify Gateway behavior.

## What This Changes

Request an explicit no-fallback diagnostic mode and louder embedded-fallback warnings for `openclaw agent`.

## Best Possible Solution

Preserve the default fallback, but add an explicit Gateway-only fail-closed CLI/env control with clearer fallback diagnostics, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes, at source level: current main catches Gateway failures in `agentCliCommand`, logs one `EMBEDDED FALLBACK` line, and invokes the embedded agent. I did not run tests because this review had to keep the checkout read-only and `pnpm` was unavailable in the environment.

## Solution Assessment

Yes: an additive `--no-fallback`/env control plus a clearer stderr preamble is the narrow maintainable direction while preserving default compatibility. The optional full stderr prefix should be weighed separately because it can change log shape.
