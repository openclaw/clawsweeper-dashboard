## Summary

Keep open: current main has the prior JSON fallback marker work, but it still automatically runs the embedded agent after Gateway dispatch failures and has no hard-fail diagnostic mode for probes that must verify Gateway behavior.

## What This Changes

Add a louder Gateway-to-embedded fallback diagnostic and an explicit `openclaw agent` no-fallback control for diagnostic or CI runs.

## Best Possible Solution

Preserve the existing default fallback, but add an explicit fail-closed CLI/env control plus focused diagnostics, tests, docs, and a changelog entry so Gateway probes can prove they did not run embedded.

## Reproduction Assessment

Yes, at source level: current main catches Gateway failures in `agentCliCommand`, logs one `EMBEDDED FALLBACK` line, and then invokes the embedded agent. I did not execute tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, current main is not the best complete solution for this issue because it only marks fallback runs after they happen. The narrow maintainable direction is an explicit no-fallback control plus clearer fallback diagnostics while keeping default compatibility.

## Repair Work Prompt

Implement the diagnostic fallback control requested by https://github.com/openclaw/openclaw/issues/76492. Keep the existing default behavior for compatibility, but add an additive `openclaw agent --no-fallback` flag and an `OPENCLAW_AGENT_NO_FALLBACK=1` environment override that fail the command when Gateway dispatch fails instead of invoking the embedded agent. Make the fallback stderr preamble clearly state that the embedded run uses the current shell environment, not the Gateway process environment, while preserving JSON stdout cleanliness. Do not prefix every embedded stderr line unless the implementation can do so without changing JSON stdout or broad runtime logging behavior. Update `src/commands/agent-via-gateway.ts`, `src/cli/program/register.agent.ts`, focused tests in `src/commands/agent-via-gateway.test.ts` and CLI registration tests if needed, `docs/cli/agent.md`, `docs/tools/agent-send.md`, and add a one-line user-facing changelog entry. Validate with targeted tests and targeted formatting only; do not change Gateway dispatch semantics, plugin loading, model fallback, or default automatic fallback behavior.
