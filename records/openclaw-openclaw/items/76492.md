## Summary

Keep open. Current main still catches Gateway dispatch failures and runs the embedded agent after a single stderr line; the CLI only exposes `--local`, docs/tests assert automatic fallback, and related #71416 was already identified as the same unresolved fallback concern.

## What This Changes

Request to make `openclaw agent` Gateway fallback unmistakable and add a gateway-only no-fallback mode for diagnostics and CI.

## Best Possible Solution

Preserve the default fallback for compatibility, but add an explicit Gateway-only hard-fail mode and clearer fallback diagnostics, with docs, tests, and changelog coverage.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence reproduction path: make `callGateway` reject during `openclaw agent`, and current main logs one fallback line before running the embedded agent.

## Solution Assessment

Yes. The maintainable fix is narrow: keep existing default behavior, add an explicit fallback opt-out, and make the fallback warning explain the separate embedded process environment.

## Repair Work Prompt

Implement a narrow `openclaw agent` fallback diagnostic improvement for https://github.com/openclaw/openclaw/issues/76492, also covering related https://github.com/openclaw/openclaw/issues/71416. Preserve the current default Gateway-to-embedded fallback for compatibility. Add an explicit hard-fail mode, preferably `--no-fallback`, and consider `OPENCLAW_AGENT_NO_FALLBACK=1` only if it matches existing CLI env patterns. When fallback occurs, emit a louder structured stderr warning that states Gateway dispatch failed, the embedded run uses the current CLI shell/process environment rather than the Gateway environment, plugin behavior may differ, and the user can check `openclaw gateway status` or use no-fallback mode. Keep JSON stdout parseable and existing embedded fallback result metadata stable. Add or update tests for normal fallback, timeout fallback, JSON stderr behavior, hard-fail behavior, and CLI option registration. Update `docs/cli/agent.md` and add a user-facing `CHANGELOG.md` entry. Do not change Gateway protocol or embedded agent runtime semantics.
