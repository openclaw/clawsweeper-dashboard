## Summary

Keep open. Current `main` and the latest release still use embedded fallback for Gateway transport/timeout failures, and there is no hard-fail diagnostic option; the issue is the canonical thread with source-level reproduction and an open draft PR that needs maintainer product review.

## What This Changes

Request a louder or disable-able `openclaw agent` embedded fallback path so Gateway diagnostic probes do not report success through local embedded execution.

## Best Possible Solution

Review, land, or replace the focused diagnostic-mode PR while preserving the current interactive fallback default unless maintainers explicitly approve a broader behavior change.

## Reproduction Assessment

Yes, at source level: forcing a Gateway transport error or timeout makes `agentCliCommand` invoke the embedded agent and report fallback metadata/stderr diagnostics. The issue discussion also includes live before-fix auth-mismatch and timeout data, but I did not run a live Gateway repro in this read-only review.

## Solution Assessment

Unclear as a final product choice: an additive no-fallback diagnostic mode is a narrow plausible fix, but the flag/env contract and any exit-code or JSON-shape changes need maintainer approval. Changing the default fallback behavior would be riskier than preserving it and adding an explicit operator-selected hard-fail mode.
