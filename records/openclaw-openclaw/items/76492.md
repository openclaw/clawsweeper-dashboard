## Summary

Keep open: current main still falls back from Gateway transport/auth/timeout failures into embedded agent execution, and the requested hard-fail diagnostic mode is not implemented. Fresh live repro comments and the open draft PR show this remains the canonical tracker.

## What This Changes

The issue requests louder embedded-fallback diagnostics and a gateway-only `openclaw agent` mode such as `--no-fallback` or `OPENCLAW_AGENT_NO_FALLBACK=1`.

## Best Possible Solution

Land an additive gateway-only no-fallback mode with clear diagnostics, focused regression tests, and public CLI/docs coverage while preserving default fallback compatibility unless maintainers deliberately change it.

## Reproduction Assessment

Yes, at source level: current main catches Gateway timeout and transport/auth failures and invokes embedded fallback, with tests asserting that behavior. The issue comments also include fresh live macOS and WSL2 examples, though I did not run a live Gateway probe in this read-only review.

## Solution Assessment

Yes for the proposed additive hard-fail mode: a no-fallback flag/env var is the narrowest maintainable solution for diagnostic and CI probes. Changing default fallback behavior or prefixing every embedded stderr line should remain a separate compatibility decision.
