## Summary

Keep open: current main still falls back to embedded execution after Gateway transport or timeout failures, and there is no `--no-fallback` or env hard-fail mode. The linked PR that targets this behavior is still open, draft, and conflicting, so this issue should remain the canonical tracker.

## What This Changes

Request a louder and optionally fail-closed `openclaw agent` Gateway fallback mode so diagnostic probes do not report embedded success as Gateway success.

## Best Possible Solution

Land an additive gateway-only no-fallback diagnostic mode with focused tests and docs, while preserving default fallback compatibility unless maintainers explicitly change it.

## Reproduction Assessment

Yes, at source level: current `agentCliCommand` catches Gateway transport and timeout failures and invokes embedded fallback, and focused tests assert that behavior. I did not run a live Gateway probe in this read-only review.

## Solution Assessment

Mostly yes: an additive no-fallback flag/env mode is the narrowest maintainable direction for diagnostic probes. The broader warning format, stderr prefixing, and JSON shape should remain explicit maintainer decisions.
