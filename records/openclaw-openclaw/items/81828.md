## Summary

Keep open: current main and the latest release still lack the requested `skills.update` size-drop opt-in, and the only linked implementation was closed unmerged. The source path is clear, but exposing a caller-controlled size-drop bypass on an admin RPC needs maintainer product/security approval.

## What This Changes

Add a narrow `skills.update` API field that forwards the existing `allowConfigSizeDrop` writer option for intentional skill env credential cleanup.

## Best Possible Solution

Approve the safety policy, then restore or replace https://github.com/openclaw/openclaw/pull/81833 with a narrow admin-scoped protocol change that forwards the existing writer option only for intentional skill credential cleanup and includes docs plus regression tests.

## Reproduction Assessment

Yes, source-level. Current main rejects `allowConfigSizeDrop` as an extra `skills.update` property and has no path to pass writer options; if env cleanup shrinks the file below the writer threshold, the shared guard blocks unless that option is true.

## Solution Assessment

Mostly yes, pending maintainer policy approval. Forwarding the existing writer option through the admin-scoped schema, handler, docs, and focused tests is narrow, but maintainers should decide whether to constrain it to disabling a skill and removing only that skill's env/apiKey values.
