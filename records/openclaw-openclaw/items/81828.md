## Summary

Keep open because current main still lacks the requested `skills.update` size-drop opt-in, and the only linked implementation PR was closed unmerged. The code path is clear, but exposing a per-call config size-drop bypass on an admin RPC still needs maintainer policy approval before implementation.

## What This Changes

Request adding an optional `allowConfigSizeDrop` flag to config-mode `skills.update` for intentional large config shrink when disabling a skill and clearing env secrets.

## Best Possible Solution

Approve a narrow admin-scoped protocol change that forwards the existing writer option only for intentional config-mode skill cleanup, with schema, docs, and regression tests proving default behavior remains guarded.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main rejects `allowConfigSizeDrop` as an extra `skills.update` property, the handler cannot forward writer options, and the shared writer blocks large size drops unless that option is true.

## Solution Assessment

Mostly yes, pending maintainer policy approval. Forwarding the existing writer option through the admin-scoped schema and mutation path is narrow, but maintainers should define the accepted cleanup constraints before landing it.
