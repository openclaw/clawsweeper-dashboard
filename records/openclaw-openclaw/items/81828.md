## Summary

Keep open: current main still rejects the requested `allowConfigSizeDrop` payload and the `skills.update` config path cannot forward the writer option, even though the shared config writer already supports it. This is a narrow feature/policy request, not an already-implemented cleanup.

## What This Changes

Add a trusted `skills.update` option that forwards `allowConfigSizeDrop` for intentional skill env/API-key cleanup writes.

## Best Possible Solution

Define a narrow admin-only `skills.update` contract for intentional cleanup, limit it to the target skill’s `enabled`/`apiKey`/`env` removals, and add protocol, docs, and regression coverage for the size-drop path.

## Reproduction Assessment

Yes, from source inspection: current main rejects `allowConfigSizeDrop` as an extra `skills.update` property and the env cleanup path writes without `writeOptions`. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the policy constraints. The requested direction is plausible, but the maintainable fix should not be a global bypass; it should honor the flag only for tightly scoped intentional skill cleanup.
