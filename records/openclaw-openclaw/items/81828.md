## Summary

Keep open: current main and latest release still do not expose the requested `skills.update` size-drop opt-in, while the closed direct implementation PR was not merged. The source path is clear, but this is a security-sensitive protocol/API policy decision rather than a strict bug-fix cleanup.

## What This Changes

Allow trusted `skills.update` callers to request a narrow `allowConfigSizeDrop` opt-in when intentionally disabling a skill and removing large skill env secrets.

## Best Possible Solution

Land a narrow maintainer-approved protocol, handler, mutation-helper, docs, and regression-test change that forwards `allowConfigSizeDrop` only for trusted intentional skill cleanup while preserving every other destructive-write guard.

## Reproduction Assessment

Yes, source-level: current main rejects `allowConfigSizeDrop` as an extra `skills.update` property and the mutation helper has no write-options path. I did not run a live write because this review is read-only.

## Solution Assessment

Mostly yes, pending maintainer policy approval: using the existing writer option is the narrow technical path, but the accepted API must constrain which admin cleanup writes may bypass the size-drop guard.
