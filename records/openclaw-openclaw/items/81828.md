## Summary

Keep open. Current `main` has the shared writer option, and `skills.update` can delete env keys, but the gateway protocol and handler do not expose or forward a size-drop opt-in; adding one is a new admin RPC capability that should be constrained before implementation.

## What This Changes

The issue requests a `skills.update` API option that lets trusted callers intentionally allow large config size drops when disabling a skill and clearing env secrets.

## Best Possible Solution

Add a narrow admin-only `allowConfigSizeDrop` field for config-mode `skills.update`, constrain it to intentional cleanup semantics, forward it through `mutateConfigFileWithRetry` write options, and cover the schema plus rejected-write behavior with focused tests.

## Reproduction Assessment

Yes, source-level. A request containing `allowConfigSizeDrop` cannot pass the current schema, and the env-deletion path reaches the shared config writer without the option that would bypass only `size-drop:*`.

## Solution Assessment

Unclear until maintainer policy is chosen. The requested direction is plausible because the writer option already exists, but blindly forwarding a boolean for every `skills.update` mutation would be broader than the safest cleanup-only fix.
