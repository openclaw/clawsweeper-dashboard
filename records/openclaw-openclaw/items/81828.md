## Summary

Keep open because current main still lacks the requested `skills.update` size-drop opt-in, and an open closing PR already targets this issue. The source path is clear, but exposing a per-call config size-drop bypass is a policy/API change that should land through the linked PR or an equivalent maintainer-approved implementation.

## What This Changes

Allow trusted `skills.update` calls to pass an intentional `allowConfigSizeDrop` flag when disabling a skill and clearing large env values.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81833 or an equivalent narrow admin-only implementation that forwards the existing writer option only for intentional skill cleanup and keeps the rest of the config guard intact.

## Reproduction Assessment

Yes, from source inspection: current main rejects `allowConfigSizeDrop` as an extra `skills.update` property and the config mutation path cannot forward writer options. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the exact policy boundary. The requested direction is plausible, but the best solution should be a tightly scoped admin cleanup path rather than a broad config size-drop bypass.
