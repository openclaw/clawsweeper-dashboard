## Summary

Keep open: current main still lacks the requested `skills.update` size-drop opt-in, and an open linked implementation PR has closing syntax for this issue. That PR should be reviewed or closed before this issue is resolved; https://github.com/openclaw/openclaw/issues/71865 is adjacent size-drop guard work, but it covers a different auth-login formatting trigger.

## What This Changes

The issue requests a trusted `skills.update` API option that forwards `allowConfigSizeDrop` when intentionally disabling a skill and clearing large env secrets.

## Best Possible Solution

Add a narrow admin-only protocol contract that forwards the existing writer option only for intentional target-skill cleanup, with gateway docs and regression coverage for flagged and unflagged size-drop writes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the schema rejects `allowConfigSizeDrop`, the handler drops any such field, and the mutation helper writes without `writeOptions`; I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear until maintainers accept the exact safety constraints. The linked PR is the immediate implementation candidate, but the maintainable shape should keep the bypass admin-scoped and limited to intentional skill cleanup.
