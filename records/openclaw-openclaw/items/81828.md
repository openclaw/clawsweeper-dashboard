## Summary

Keep open because current main still lacks the requested `skills.update` size-drop opt-in, and the only closing PR was closed unmerged. The source path is clear, but exposing a per-call size-drop bypass is an admin RPC/API policy change that needs maintainer approval.

## What This Changes

Request to let trusted config-mode `skills.update` calls pass an `allowConfigSizeDrop` opt-in when intentionally clearing large skill environment values.

## Best Possible Solution

Approve the policy boundary, then land or recreate a narrow admin-scoped change that adds the config-mode `skills.update` flag, forwards it into `mutateConfigFileWithRetry.writeOptions`, and documents/tests that other destructive guards still apply.

## Reproduction Assessment

Yes. Source inspection on current `origin/main` shows the requested field is rejected by schema and cannot reach the writer, while the writer would reject a large env cleanup unless `allowConfigSizeDrop` is true.

## Solution Assessment

Unclear pending maintainer policy approval. The closed PR’s approach is the narrow technical shape, but the repository still needs an explicit decision on allowing this admin RPC override.
