## Summary

Keep open because current main still lacks the requested `skills.update` size-drop opt-in, and an open closing PR already targets the issue. The code path is clear, but exposing a per-call size-drop bypass is an admin RPC/API policy change that should land through the linked PR or an equivalent maintainer-approved implementation.

## What This Changes

Request to let trusted `skills.update` callers pass `allowConfigSizeDrop` for intentional skill env/API-key cleanup that can shrink `openclaw.json` by more than half.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81833 or an equivalent narrow admin-scoped implementation that forwards the existing writer option only for intentional skill cleanup and keeps other destructive config guards active.

## Reproduction Assessment

Yes. Source inspection shows current main rejects `allowConfigSizeDrop` as an extra `skills.update` property and the mutation path cannot forward writer options, while the shared writer blocks large size drops unless that option is true.

## Solution Assessment

Mostly yes, pending maintainer policy approval. Forwarding the existing writer option through the existing admin-scoped `skills.update` path is narrow, but maintainers should confirm the exact caller and mutation-shape constraints.
