## Summary

Keep open: the PR targets a valid source-proven status hang, but this branch is not the best merge path because it is weaker than the already-open canonical fix and still has blocking timeout/audit-severity issues.

## What This Changes

The PR bounds sandbox-browser Docker audit probes during status/security audit execution, forwards the status timeout budget into that path, and adds regression tests for non-returning Docker probes.

## Best Possible Solution

Land or cherry-pick the stronger `execDockerRaw` subprocess-timeout approach from https://github.com/openclaw/openclaw/pull/85046, then close this branch as superseded if it has no unique maintainer-approved remainder.

## Reproduction Assessment

Yes. Source inspection shows current main routes text `openclaw status --deep` through the security audit into unbounded sandbox-browser Docker probes; a non-settling Docker probe test double reproduces the hang deterministically.

## Solution Assessment

No. The timeout direction is right, but this branch is not the best implementation because it leaves subprocess cleanup in the caller wrapper, reports skipped security-audit work as info, and overlaps a stronger open PR for the same issue.

Full review comments:

- **[P2] Move the timeout into the Docker subprocess helper:** `src/security/audit-extra.async.ts:308`
  - body: This races the audit helper and aborts the existing Docker call, but `execDockerRaw` still only sends SIGTERM and waits for the child close path with stdio left attached. If the Docker CLI is the thing that wedges, Node can still keep the status process alive after the audit promise returns; put the timeout in `execDockerRaw` with prompt rejection, stdio cleanup, a SIGKILL fallback, and a sleeping-docker regression test.
  - confidence: 0.85
- **[P2] Warn when Docker audit coverage is skipped:** `src/security/audit-extra.async.ts:558`
  - body: A timeout here skips the sandbox-browser drift and non-loopback port checks, but the new finding is only `info`. That makes a security-audit coverage gap easy to miss; emit a warning so operators see that Docker-backed container checks did not actually run.
  - confidence: 0.82
- **[P2] Keep gateway and subprocess timeout budgets separate:** `src/security/audit.ts:1099`
  - body: `deepTimeoutMs` is documented as the deep gateway probe limit, but this change also uses it for Docker audit subprocesses. Direct security-audit callers that tuned gateway probing can now shorten sandbox-browser audit checks, so use an explicit subprocess timeout option or a status-only forwarding field instead.
  - confidence: 0.76
