## Summary

Keep open. The patch targets a real remote-skills probe churn problem, but the new probe state is connection-scoped and is deleted on disconnect, failure counts also carry across changed signatures, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds in-memory remote bin probe signatures with a 24-hour success TTL, exponential failure backoff, an optional force parameter, and focused skills-remote tests.

## Best Possible Solution

Keep the throttling direction only after probe state is scoped per node/signature across reconnects, failure history resets when the signature changes, regression tests cover those cases, and the contributor attaches redacted real remote-node proof.

## Reproduction Assessment

Yes for the PR-introduced defects by source inspection: disconnect deletes the record that stores the new cache/backoff fields, and failure counts are reused after the signature changes. I did not run a live gateway/node scenario because this review is read-only and must keep the checkout clean.

## Solution Assessment

No. The approach is useful, but the state needs to survive the reconnect pattern it is meant to throttle and retry history should be keyed to the active probe signature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep probe backoff outside the connection record:** `src/infra/skills-remote.ts:23-26`
  - body: The new TTL/backoff fields live on RemoteNodeRecord, but disconnects call removeRemoteNodeInfo(), which deletes that record. The churn reports this is trying to address include nodes disconnecting/reconnecting, so every reconnect starts with no probeSignature or failedProbeCount and immediately probes again. Store probe state in a node/signature cache that survives disconnects while keeping disconnected bins out of eligibility.
  - confidence: 0.9
- **[P2] Reset failure counts for new probe signatures:** `src/infra/skills-remote.ts:214`
  - body: markRemoteNodeProbeFailure() always increments existing.failedProbeCount even when existing.probeSignature belongs to different bins or capabilities. After a long failure run, a changed requirement that fails once inherits the old count and may be suppressed for up to 24h, so the new signature does not get a fresh retry schedule.
  - confidence: 0.86
