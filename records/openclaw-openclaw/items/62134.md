## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve streamable HTTP redirect scrubbing:** `src/agents/mcp-transport.ts:105-108`
  - body: Current main passes `fetchStreamableHttpWithRedirectScrub` to streamable HTTP transports so configured headers are not replayed to another origin during redirects. This branch's constructor omits that fetch hook, so rebasing or merging it would reintroduce cross-origin header leakage.
  - confidence: 0.92
- **[P1] Keep the callback session through token exchange:** `src/agents/mcp-oauth-provider.ts:124-126`
  - body: `waitForAuthorizationCode()` awaits a promise whose `finally` clears `pendingAuthorizationCode`, making `provider.redirectUrl` undefined before `finishAuth(code)` exchanges the authorization code. MCP SDK v1.29.0 requires `redirectUrl` for the authorization-code token request, so first-time OAuth fails before tokens are saved.
  - confidence: 0.95
- **[P1] Initialize callbacks when cached clients reauthorize:** `src/agents/mcp-oauth-provider.ts:157-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` undefined. The SDK treats that as a non-interactive flow before it can start browser reauthorization, so expired or missing tokens for a registered server cannot recover.
  - confidence: 0.9
- **[P1] Reconnect with a fresh MCP client and transport:** `src/agents/pi-bundle-mcp-runtime.ts:224-227`
  - body: After `finishAuth`, this retries `connectWithTimeout` on the same client and transport. The SDK attaches the transport before initialization finishes; streamable failures leave the client connected, and closing SSE still leaves the transport instance started, so the retry can fail even after valid tokens are saved.
  - confidence: 0.88
- **[P1] Scope OAuth token files to endpoint identity:** `src/agents/pi-bundle-mcp-runtime.ts:173-176`
  - body: The provider loads and saves OAuth state using only the display `serverName`. Different workspaces or remote MCP URLs using the same key can share one token file, which can send credentials minted for one MCP server to another before auth has a chance to correct it.
  - confidence: 0.88
- **[P2] Write OAuth token files with private permissions:** `src/agents/mcp-oauth-provider.ts:80-82`
  - body: This stores access and refresh tokens using default directory and file modes, which depend on the process umask. Create the directory and token files with restrictive modes, and repair loose existing modes before saving credential state.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces OAuth bearer-token persistence and touches MCP transport security paths with concrete credential-isolation and filesystem-mode issues.

Concerns:

- **[high] Streamable redirects can lose current-main header scrubbing:** `src/agents/mcp-transport.ts:105`
  - body: The PR head's streamable HTTP constructor omits the current-main redirect-scrubbing fetch hook, risking configured auth headers being replayed to a cross-origin redirect target if this branch is merged without preserving that hardening.
  - confidence: 0.92
- **[high] OAuth tokens are keyed only by display name:** `src/agents/pi-bundle-mcp-runtime.ts:175`
  - body: A global token file derived only from `serverName` can mix credentials across workspaces or different MCP URLs that reuse a key like `github`, risking bearer tokens being sent to the wrong remote server.
  - confidence: 0.88
- **[medium] OAuth token files can be too broadly readable:** `src/agents/mcp-oauth-provider.ts:82`
  - body: The new token persistence path writes access and refresh tokens using default filesystem modes, so local readability depends on umask instead of an explicit credential-storage policy.
  - confidence: 0.9
