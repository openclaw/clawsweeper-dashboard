The branch adds opt-in `auth: "oauth"` for remote MCP HTTP/SSE servers, an OAuth provider with local callback and token persistence, transport `authProvider` wiring, UnauthorizedError retry handling, and focused tests.

## Best Possible Solution

Keep the OAuth direction, but merge only after rebasing onto current MCP transport/runtime code, fixing lifecycle and credential-isolation issues, adding docs/tests, and providing redacted real OAuth MCP proof.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve streamable HTTP redirect scrubbing:** `src/agents/mcp-transport.ts:105-108`
  - body: Current main passes `fetchStreamableHttpWithRedirectScrub` to streamable HTTP transports so configured headers are not replayed to a different origin during redirects. This branch's constructor omits that current-main fetch hook, so rebasing/merging it as written would reintroduce credential/header leakage on cross-origin redirects.
  - confidence: 0.91
- **[P1] Keep the callback session through token exchange:** `src/agents/mcp-oauth-provider.ts:124-126`
  - body: `waitForAuthorizationCode()` awaits a promise whose `finally` clears `pendingAuthorizationCode`, so `provider.redirectUrl` is undefined by the time `finishAuth(code)` asks the SDK to exchange the authorization code. MCP SDK 1.29.0 requires `redirectUrl` for the authorization-code token request, so first-time OAuth fails before tokens are saved.
  - confidence: 0.95
- **[P1] Initialize a callback session for cached client reauth:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` undefined. The SDK treats that as non-interactive before it can start a browser authorization flow, so reauthorization after missing or expired tokens cannot work.
  - confidence: 0.9
- **[P1] Reconnect with a fresh MCP client and transport:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After `finishAuth`, this retries with the same `Client` and transport. The SDK attaches the transport before `start()` completes, so a failed auth start can leave the protocol already connected and make the retry fail even after valid tokens are saved.
  - confidence: 0.9
- **[P1] Scope OAuth token files to endpoint identity:** `src/agents/pi-bundle-mcp-runtime.ts:175-176`
  - body: The provider loads and saves OAuth state using only the display `serverName`, so different workspaces or remote MCP URLs using the same key can share one token file. That can send credentials minted for one MCP server to another before auth has a chance to correct it.
  - confidence: 0.88
- **[P2] Write OAuth token files with private permissions:** `src/agents/mcp-oauth-provider.ts:80-82`
  - body: This stores access and refresh tokens using default directory and file modes, which depend on the process umask. Create the directory and token files with restrictive modes and repair loose existing modes before saving credential state.
  - confidence: 0.9
