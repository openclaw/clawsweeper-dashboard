## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve streamable redirect scrubbing:** `src/agents/mcp-transport.ts:105-108`
  - body: Current main routes Streamable HTTP through `fetchStreamableHttpWithRedirectScrub` so sensitive headers are stripped on cross-origin redirects. This constructor drops that fetch hook while adding OAuth, so a redirected MCP endpoint could receive configured headers or bearer tokens that current main would scrub.
  - confidence: 0.91
- **[P1] Keep the callback session through finishAuth:** `src/agents/mcp-oauth-provider.ts:122-126`
  - body: `pendingAuthorizationCode.promise.finally()` clears `pendingAuthorizationCode` as soon as the callback resolves, before `finishAuth(code)` runs. The MCP SDK's authorization-code exchange requires `provider.redirectUrl`, so this cleanup can make the token exchange fail immediately after a successful browser callback.
  - confidence: 0.9
- **[P1] Create a callback session for cached clients:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` undefined. The SDK treats that as a non-interactive flow before refresh/new authorization, so expired or missing tokens for an already-registered client can fail without opening the browser.
  - confidence: 0.87
- **[P1] Namespace OAuth state by endpoint identity:** `src/agents/mcp-oauth-provider.ts:61-62`
  - body: The token store path is derived only from `serverName`, so different workspaces or endpoints using the same display key can share and overwrite one OAuth state file. Key the store by stable server identity such as normalized URL plus any approved scope/workspace component so tokens cannot be sent to the wrong server.
  - confidence: 0.88
- **[P2] Write OAuth state with private permissions:** `src/agents/mcp-oauth-provider.ts:77-82`
  - body: `mkdirSync` and `writeFileSync` use default filesystem permissions, which can leave access and refresh tokens readable by other local users on permissive umasks. Create the directory as `0700` and files as `0600`, and handle existing files/directories consistently.
  - confidence: 0.9
- **[P2] Align the public MCP auth config contract:** `src/config/types.mcp.ts:18-19`
  - body: The PR adds a public `auth: "oauth"` type but does not add it to config schema/help/docs, so users and validation tooling do not get a complete supported contract for the new mode. Add schema, docs, and focused config tests with the same change.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The PR introduces OAuth bearer-token persistence and callback handling with concrete redirect, token-isolation, and filesystem-permission concerns.

Concerns:

- **[high] Redirect scrubbing can be lost:** `src/agents/mcp-transport.ts:105`
  - body: The PR head omits current main's streamable HTTP redirect-scrubbing fetch hook, which can leak configured headers or OAuth bearer tokens to a cross-origin redirect target.
  - confidence: 0.91
- **[medium] OAuth tokens are not endpoint-isolated:** `src/agents/mcp-oauth-provider.ts:61`
  - body: Persisting state by display server name alone can reuse or overwrite tokens across different endpoints/workspaces that share a key, risking credential mis-association.
  - confidence: 0.88
- **[medium] Token files rely on default permissions:** `src/agents/mcp-oauth-provider.ts:82`
  - body: Access and refresh tokens are written with default directory/file modes, so local readability depends on umask instead of an explicit private credential-store policy.
  - confidence: 0.9
