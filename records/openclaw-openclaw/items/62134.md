## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve streamable redirect scrubbing:** `src/agents/mcp-transport.ts:105-108`
  - body: Current main passes `fetchStreamableHttpWithRedirectScrub` so configured headers are not replayed to a different origin during redirects. This constructor drops that fetch hook while adding OAuth, so merging the branch would reintroduce cross-origin credential/header leakage for streamable HTTP redirects.
  - confidence: 0.91
- **[P1] Keep the callback session through token exchange:** `src/agents/mcp-oauth-provider.ts:124-126`
  - body: `waitForAuthorizationCode()` awaits a promise whose `finally` clears `pendingAuthorizationCode`, so `provider.redirectUrl` is undefined by the time `finishAuth(code)` asks the SDK to exchange the code. MCP SDK 1.29.0 requires `redirectUrl` for the authorization-code token request, so first-time OAuth fails before tokens are saved.
  - confidence: 0.95
- **[P1] Initialize a callback session for cached reauth:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` undefined. The SDK treats that as non-interactive before it can start a browser authorization flow, so reauthorization after missing or expired tokens cannot work.
  - confidence: 0.9
- **[P1] Reconnect with a fresh client and transport:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After `finishAuth`, this retries `connectWithTimeout` on the same instances. The SDK attaches the transport before `start()` completes, and SSE close also leaves the transport's started state behind, so a failed auth start can make the retry fail even after valid tokens are saved.
  - confidence: 0.9
- **[P1] Scope OAuth token files to endpoint identity:** `src/agents/pi-bundle-mcp-runtime.ts:175-176`
  - body: The OAuth provider loads and saves state using only the display `serverName`, so different workspaces or remote MCP URLs using the same key can share one token file. That can send credentials minted for one MCP server to another before auth has a chance to correct it.
  - confidence: 0.88
- **[P2] Write OAuth token files with private permissions:** `src/agents/mcp-oauth-provider.ts:80-82`
  - body: This stores access and refresh tokens using default directory and file modes, which depend on the process umask. Create the directory and token files with restrictive modes and repair loose existing modes before saving credential state.
  - confidence: 0.9
- **[P2] Align the new auth config with schema and docs:** `src/config/types.mcp.ts:18-19`
  - body: Adding `auth: "oauth"` to the exported config type without updating the Zod schema, schema tests/help, and public MCP docs leaves the new user-facing option out of OpenClaw's config contract and discovery surfaces.
  - confidence: 0.84
