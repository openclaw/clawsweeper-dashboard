## Summary

Keep open: current main and the latest release still lack generic remote MCP OAuth support, but this branch is not ready because it is conflicting, lacks real behavior proof, and has source-proven OAuth lifecycle/security blockers.

## What This Changes

The PR adds opt-in `auth: "oauth"` support for remote MCP HTTP/SSE servers with SDK auth-provider wiring, a local callback/token persistence layer, UnauthorizedError retry handling, and focused tests.

## Best Possible Solution

Land a rebased implementation that preserves current transport hardening, scopes private OAuth state by endpoint/workspace identity, uses a fresh client/transport OAuth reconnect path, updates the config/docs surface, and includes redacted live OAuth MCP proof.

## Reproduction Assessment

Yes for the source-level feature gap and the blocking PR defects: current main/release have no MCP OAuth path, and the PR head failures follow from the SDK v1.29.0 auth and reconnect contracts. No live end-to-end OAuth run was established for the branch.

## Solution Assessment

No. Passing the SDK `authProvider` is the right direction, but this PR needs lifecycle, redirect-hardening, credential-isolation, filesystem-permission, docs/config, and live-proof work before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve streamable redirect scrubbing:** `src/agents/mcp-transport.ts:105-108`
  - body: Current main passes a custom streamable HTTP fetch that manually follows redirects and strips unsafe headers on cross-origin redirects. This constructor drops that hook while adding OAuth, which can replay configured headers or bearer tokens to a redirected origin.
  - confidence: 0.93
- **[P1] Keep the callback session through token exchange:** `src/agents/mcp-oauth-provider.ts:122-126`
  - body: `waitForAuthorizationCode()` awaits the same promise whose `finally` clears `pendingAuthorizationCode`, so `provider.redirectUrl` is undefined by the time `finishAuth(code)` exchanges the code. The MCP SDK requires `redirectUrl` for authorization-code token exchange, so first-time OAuth fails after the callback arrives.
  - confidence: 0.95
- **[P1] Initialize callback sessions with cached client info:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` unset. Reauthorization after missing or invalid tokens then enters the SDK's non-interactive path instead of opening the browser flow.
  - confidence: 0.91
- **[P1] Reconnect with a fresh MCP transport after OAuth:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After `UnauthorizedError`, the retry reuses the same `Client` and transport. The SDK attaches and starts the transport before initialization completes, and the transports are not reset for a second `start()`, so the post-`finishAuth` retry can fail as already connected/started.
  - confidence: 0.9
- **[P1] Scope OAuth state by endpoint identity:** `src/agents/pi-bundle-mcp-runtime.ts:173-176`
  - body: OAuth state is persisted using only the display server name, so different workspaces or URLs that reuse a key like `github` share the same token/client-state file. That can send bearer credentials minted for one endpoint to another before auth can correct it.
  - confidence: 0.89
- **[P2] Write OAuth state files with private permissions:** `src/agents/mcp-oauth-provider.ts:79-82`
  - body: Access and refresh tokens are written with default directory and file permissions, which depend on the process umask. Create or repair the directory and token files with restrictive modes before saving credential state.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces OAuth bearer-token persistence and redirect/callback handling with concrete token-leakage and local-file permission risks.

Concerns:

- **[high] Streamable redirects can lose header scrubbing:** `src/agents/mcp-transport.ts:105`
  - body: The PR head omits current-main redirect-scrubbing fetch behavior for streamable HTTP, risking configured headers or bearer tokens being replayed to a cross-origin redirect target.
  - confidence: 0.93
- **[high] OAuth tokens are keyed only by display name:** `src/agents/pi-bundle-mcp-runtime.ts:173`
  - body: A global token file derived only from the configured server key can mix credentials across workspaces or different MCP URLs that reuse the same display name.
  - confidence: 0.89
- **[medium] OAuth token files can be too broadly readable:** `src/agents/mcp-oauth-provider.ts:82`
  - body: Access and refresh tokens are written with default filesystem modes, so local readability depends on umask instead of explicit private permissions.
  - confidence: 0.9
