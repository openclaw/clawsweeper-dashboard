## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep the callback session through finishAuth:** `src/agents/mcp-oauth-provider.ts:124-126`
  - body: `waitForAuthorizationCode()` awaits the promise whose `finally` clears `pendingAuthorizationCode`, so by the time the runtime calls `finishAuth(code)`, `provider.redirectUrl` is undefined. The MCP SDK requires that redirect URL for the authorization-code token exchange, so first-time OAuth fails before tokens are saved.
  - confidence: 0.95
- **[P1] Create callback state for cached client reauth:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` undefined. SDK v1.29.0 treats that as a non-interactive flow before it can start a new browser redirect, so reauthorization after token expiry cannot complete.
  - confidence: 0.9
- **[P1] Reconnect with fresh MCP client and transport:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After `finishAuth`, this retries `connectWithTimeout` on the same client and transport. The SDK attaches the transport before initialization completes, and HTTP/SSE transports retain started state after failed auth, so the retry can fail as already connected or already started even with valid tokens.
  - confidence: 0.9
- **[P1] Namespace OAuth state by endpoint identity:** `src/agents/mcp-oauth-provider.ts:61-62`
  - body: The persisted OAuth state path is based only on `serverName`, so different workspaces or remote MCP URLs using the same display key can share one token file. That can send credentials minted for one MCP server to another before auth has a chance to correct it.
  - confidence: 0.88
- **[P2] Write OAuth state with private permissions:** `src/agents/mcp-oauth-provider.ts:80-82`
  - body: This writes access and refresh tokens with default directory and file modes, which depend on the process umask. Create the directory and token file with restrictive permissions and repair loose existing modes before saving credential state.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces bearer-token persistence with credential isolation and filesystem-permission weaknesses.

Concerns:

- **[high] OAuth tokens are keyed only by display name:** `src/agents/mcp-oauth-provider.ts:61`
  - body: A global token file derived only from `serverName` can mix credentials across workspaces or different MCP URLs that reuse a key like `github`, risking bearer tokens being sent to the wrong remote server.
  - confidence: 0.88
- **[medium] OAuth token files can be too broadly readable:** `src/agents/mcp-oauth-provider.ts:82`
  - body: The new token persistence path writes access and refresh tokens with default mode bits; on permissive umasks, other local users may be able to read the credential file.
  - confidence: 0.9
