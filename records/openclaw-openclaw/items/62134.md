## Summary

Keep open: current main still lacks remote MCP OAuth and the recent Figma repro supports the gap, but this PR is not mergeable because it lacks live OAuth proof and still has source-reproducible OAuth lifecycle and credential-storage/security blockers.

## What This Changes

The branch adds opt-in `auth: "oauth"` for remote MCP HTTP/SSE servers, a new OAuth provider/callback/token persistence layer, transport `authProvider` wiring, UnauthorizedError retry handling, and focused tests.

## Best Possible Solution

Rebase onto current main, preserve transport hardening, fix OAuth callback/reconnect lifecycle and endpoint-scoped private token storage, then provide redacted live OAuth MCP proof.

## Reproduction Assessment

Yes for the blocking PR defects: they are source-reproducible from the PR head plus the MCP SDK v1.29.0 contract. The underlying feature gap is also supported by current-main source and a real Figma Unauthorized report, but the branch itself has no live OAuth success proof.

## Solution Assessment

No. Passing the SDK `authProvider` is the right direction, but this branch needs lifecycle, redirect-hardening, credential-isolation, and proof work before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve streamable redirect scrubbing:** `src/agents/mcp-transport.ts:105-108`
  - body: Current main passes `fetchStreamableHttpWithRedirectScrub` so configured headers are not replayed to a cross-origin redirect target. This constructor drops that fetch hook while adding OAuth, which would reintroduce credential/header leakage on streamable HTTP redirects.
  - confidence: 0.92
- **[P1] Keep the callback session through token exchange:** `src/agents/mcp-oauth-provider.ts:124-126`
  - body: `waitForAuthorizationCode()` awaits the same promise whose `finally` clears `pendingAuthorizationCode`, so `provider.redirectUrl` is undefined before `finishAuth(code)` exchanges the authorization code. The MCP SDK requires `redirectUrl` for that token request, so first-time OAuth fails before tokens are saved.
  - confidence: 0.95
- **[P1] Initialize callback sessions with cached client info:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` unset. The SDK treats that as non-interactive and cannot start the browser flow when cached registration needs reauthorization.
  - confidence: 0.91
- **[P1] Reconnect with a fresh MCP client after OAuth:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After an UnauthorizedError this retries `connectWithTimeout` on the same client and transport. The SDK attaches the transport before initialization completes, so the failed first connection can leave the protocol or transport already started; recreate the client/transport for the post-`finishAuth` connection.
  - confidence: 0.9
- **[P1] Scope OAuth state by endpoint identity:** `src/agents/pi-bundle-mcp-runtime.ts:173-176`
  - body: OAuth state is loaded and saved using only the display `serverName`, so different workspaces or URLs that reuse a key like `github` can share one token file. That can send bearer credentials minted for one MCP endpoint to another before auth can correct it.
  - confidence: 0.89
- **[P2] Write OAuth state files with private permissions:** `src/agents/mcp-oauth-provider.ts:80-82`
  - body: This stores access and refresh tokens using default directory and file permissions, which depend on the process umask. Create or repair the directory and token files with restrictive modes before saving credential state.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces bearer-token persistence and OAuth transport wiring with concrete redirect, token-isolation, and filesystem-permission risks.

Concerns:

- **[high] Streamable redirects can lose header scrubbing:** `src/agents/mcp-transport.ts:105`
  - body: The PR head's streamable HTTP constructor omits current-main redirect-scrubbing fetch behavior, risking configured auth headers being replayed to a cross-origin redirect target.
  - confidence: 0.92
- **[high] OAuth tokens are keyed only by display name:** `src/agents/pi-bundle-mcp-runtime.ts:175`
  - body: A global token file derived only from the configured server key can mix credentials across workspaces or different MCP URLs that reuse the same display name.
  - confidence: 0.89
- **[medium] OAuth token files can be too broadly readable:** `src/agents/mcp-oauth-provider.ts:82`
  - body: Access and refresh tokens are written with default filesystem modes, so local readability depends on umask rather than explicit private permissions.
  - confidence: 0.9
