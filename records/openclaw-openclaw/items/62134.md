## Summary

Keep open. Current main still lacks native OAuth for remote MCP clients, so the PR is not obsolete, but the branch has blocking OAuth lifecycle and token-storage security defects that should be fixed before merge.

## What This Changes

The PR adds an opt-in `auth: "oauth"` mode for remote MCP servers, a local callback/token persistence OAuth provider, and transport/runtime wiring for SSE and streamable HTTP MCP clients.

## Best Possible Solution

Keep the PR open but require changes or a replacement implementation that preserves the callback redirect URL through token exchange, recreates client/transports for post-auth reconnects, namespaces token state by server identity, writes credentials with private permissions, and adds focused OAuth lifecycle tests.

## Reproduction Assessment

Yes. A first-time `auth: "oauth"` connection that reaches the callback will await `waitForAuthorizationCode()`, clear the provider callback session, then call `finishAuth(code)` while the SDK requires `provider.redirectUrl` for the authorization-code exchange.

## Solution Assessment

No. The direction is appropriate for the feature, but this branch is not the best mergeable solution until the OAuth lifecycle and credential-storage defects are fixed and validated against the SDK contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep redirect URL until token exchange completes:** `src/agents/mcp-oauth-provider.ts:124-126`
  - body: `waitForAuthorizationCode()` awaits the promise whose `finally` clears `pendingAuthorizationCode`, so by the time the runtime calls `finishAuth(code)`, `provider.redirectUrl` is undefined. The MCP SDK requires that redirect URL for authorization-code token exchange, so the normal first-time OAuth flow fails instead of saving tokens.
  - confidence: 0.94
- **[P1] Recreate the transport before retrying OAuth:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After an OAuth `UnauthorizedError`, this retries `connectWithTimeout` on the same `Client` and transport instance. SDK initialization failures close already-started transports, and both SSE/streamable transports retain started state, so the retry can fail as already connected/started even after a valid callback. Build a fresh client/transport using the saved tokens before reconnecting.
  - confidence: 0.88
- **[P1] Namespace OAuth state by server identity:** `src/agents/mcp-oauth-provider.ts:61-62`
  - body: The token file is keyed only by the display server name under a global home-directory store. Reusing a name like `github` in another workspace or for a different endpoint will share access/refresh tokens and client state, which can send credentials minted for one MCP server to another. Include a stable server identity such as normalized URL/resource metadata in the storage key.
  - confidence: 0.86
- **[P2] Write token state with private permissions:** `src/agents/mcp-oauth-provider.ts:80-82`
  - body: `mkdirSync` and `writeFileSync` use default permissions for a file containing OAuth access and refresh tokens. On common umasks this can leave the file readable by other local users, so create the directory/file with restrictive modes and repair existing loose permissions before writing.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR introduces bearer-token persistence and has concrete credential isolation and file-permission issues.

Concerns:

- **[medium] OAuth token files can be too broadly readable:** `src/agents/mcp-oauth-provider.ts:82`
  - body: The new persistence path writes access and refresh tokens with default mode bits, which depend on process umask; use explicit private modes for the directory and state file.
  - confidence: 0.9
- **[high] OAuth tokens are keyed only by display name:** `src/agents/mcp-oauth-provider.ts:61`
  - body: A global token file based only on `serverName` can mix credentials across workspaces or different MCP URLs that use the same display key, risking credential disclosure to the wrong server.
  - confidence: 0.86
