## Summary

Keep open: current main still lacks remote MCP OAuth support, but this external feature PR is not ready to merge because live proof is missing and source inspection against MCP SDK 1.29.0 shows OAuth lifecycle and token-storage blockers.

## What This Changes

The branch adds opt-in `auth: "oauth"` for remote MCP HTTP/SSE servers, an MCP OAuth provider with callback/token persistence, transport `authProvider` wiring, UnauthorizedError retry handling, and focused tests.

## Best Possible Solution

Keep the OAuth support direction, but merge only after rebasing on current MCP transport code, fixing SDK lifecycle and credential-storage issues, adding docs/tests, and providing redacted real OAuth MCP proof.

## Reproduction Assessment

Yes for the blocking PR defects: the failure paths are source-reproducible from the PR head and the pinned MCP SDK 1.29.0 contract. No live end-to-end OAuth MCP reproduction or after-fix proof was provided by the contributor.

## Solution Assessment

No. Passing the SDK `authProvider` is the right broad direction, but this branch is not the best mergeable solution until the OAuth callback lifecycle, reconnect strategy, credential isolation, file permissions, and proof gaps are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep the callback session until finishAuth completes:** `src/agents/mcp-oauth-provider.ts:124-126`
  - body: `waitForAuthorizationCode()` awaits a promise whose `finally` clears `pendingAuthorizationCode`, so `provider.redirectUrl` is undefined by the time `finishAuth(code)` asks the SDK to exchange the authorization code. MCP SDK 1.29.0 requires `redirectUrl` for the authorization-code token request, so first-time OAuth fails before tokens are saved.
  - confidence: 0.95
- **[P1] Initialize a callback session for cached client reauth:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` undefined. The SDK treats that as non-interactive and tries token fetch without an authorization code, so reauthorization after missing or expired tokens cannot start the browser flow.
  - confidence: 0.9
- **[P1] Reconnect with a fresh MCP client and transport:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After `finishAuth`, this retries with the same `Client` and transport. The SDK keeps HTTP/SSE transports in a started state after failed auth, and `Protocol.connect` also attaches the transport before initialization, so the retry can fail as already connected or already started even with valid tokens.
  - confidence: 0.9
- **[P1] Scope OAuth token files to endpoint identity:** `src/agents/pi-bundle-mcp-runtime.ts:175-176`
  - body: The provider loads and saves OAuth state using only the display `serverName`, so different workspaces or remote MCP URLs using the same key can share one token file. That can send credentials minted for one MCP server to another before auth has a chance to correct it.
  - confidence: 0.88
- **[P2] Write OAuth token files with private permissions:** `src/agents/mcp-oauth-provider.ts:80-82`
  - body: This stores access and refresh tokens using default directory and file modes, which depend on the process umask. Create the directory and token files with restrictive modes and repair loose existing modes before saving credential state.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch introduces OAuth bearer-token persistence without endpoint-scoped identity or explicit private filesystem modes.

Concerns:

- **[high] OAuth tokens are keyed only by display name:** `src/agents/pi-bundle-mcp-runtime.ts:175`
  - body: A global token file derived only from `serverName` can mix credentials across workspaces or different MCP URLs that reuse a key like `github`, risking bearer tokens being sent to the wrong remote server.
  - confidence: 0.88
- **[medium] OAuth token files can be too broadly readable:** `src/agents/mcp-oauth-provider.ts:82`
  - body: The new token persistence path writes access and refresh tokens with default mode bits; on permissive umasks, other local users may be able to read the credential file.
  - confidence: 0.9
