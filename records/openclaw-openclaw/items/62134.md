Full review comments:

- **[P1] Keep the callback session through token exchange:** `src/agents/mcp-oauth-provider.ts:122-126`
  - body: `waitForAuthorizationCode()` awaits the promise whose `finally` clears `pendingAuthorizationCode`, so `provider.redirectUrl` is undefined by the time `finishAuth(code)` asks the SDK to exchange the authorization code. The SDK requires `redirectUrl` for its authorization-code token request, so first-time OAuth fails after the callback arrives.
  - confidence: 0.96
- **[P1] Create a callback session when cached client info exists:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When `clientInfo` is already persisted, `clientInformation()` returns without preparing a redirect URL. In SDK v1.29.0, `auth()` checks `!provider.redirectUrl` before loading tokens/refreshing and then calls `fetchToken()` without an authorization code, so previously registered servers can fail instead of refreshing or reauthorizing.
  - confidence: 0.93
- **[P1] Reconnect with a fresh MCP transport after OAuth:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After `UnauthorizedError`, this retries `connectWithTimeout` on the same transport instance. The MCP SDK starts and attaches transports during the first failed initialization, and HTTP/SSE transports keep started/closed state, so the post-`finishAuth` retry can fail as already started/connected instead of completing the first OAuth login.
  - confidence: 0.9
- **[P1] Preserve streamable redirect header scrubbing:** `src/agents/mcp-transport.ts:105-108`
  - body: Current main passes a custom streamable HTTP fetch that strips unsafe headers across cross-origin redirects. This constructor drops that hook while adding OAuth, so configured headers or OAuth bearer tokens can be replayed to a redirected origin.
  - confidence: 0.94
- **[P1] Scope OAuth state by endpoint identity:** `src/agents/pi-bundle-mcp-runtime.ts:173-176`
  - body: The OAuth store is keyed only by the display server name, so two workspaces or URLs using a common key like `github` can share and overwrite one token/client-state file. That can send bearer credentials minted for one MCP endpoint to another before auth has a chance to correct it.
  - confidence: 0.9
- **[P2] Write OAuth state with private permissions:** `src/agents/mcp-oauth-provider.ts:77-82`
  - body: Access and refresh tokens are written with default directory and file modes, which depend on the process umask. Create or repair the MCP auth directory and token file with restrictive permissions before saving bearer credentials.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The PR introduces OAuth bearer-token persistence and callback handling with concrete redirect, token-isolation, and filesystem-permission risks.

Concerns:

- **[high] Streamable redirects can lose header scrubbing:** `src/agents/mcp-transport.ts:105`
  - body: The PR head omits current-main redirect-scrubbing fetch behavior for streamable HTTP, risking configured headers or bearer tokens being replayed to a cross-origin redirect target.
  - confidence: 0.94
- **[high] OAuth tokens are keyed only by display name:** `src/agents/pi-bundle-mcp-runtime.ts:173`
  - body: A global token file derived only from the configured server key can mix credentials across workspaces or different MCP URLs that reuse the same display name.
  - confidence: 0.9
- **[medium] OAuth token files can be too broadly readable:** `src/agents/mcp-oauth-provider.ts:82`
  - body: Access and refresh tokens are written with default filesystem modes, so local readability depends on umask instead of explicit private permissions.
  - confidence: 0.89
