## Summary

Keep open. Current main still does not implement remote MCP OAuth, and this PR is useful directionally, but the exact head is not merge-safe because the OAuth SDK lifecycle still fails in source-reviewable paths and the new token persistence has concrete credential-isolation and file-permission problems.

## What This Changes

This PR adds opt-in `auth: "oauth"` support for remote SSE and streamable HTTP MCP servers, including an OAuth provider, loopback callback handling, token persistence, transport `authProvider` wiring, UnauthorizedError retry logic, and focused tests.

## Best Possible Solution

Keep the OAuth direction, but merge only after the implementation preserves the callback redirect URL through token exchange, initializes callback state for cached-client reauthorization, reconnects with fresh SDK client/transport instances, scopes persisted credentials by stable server identity, writes state with private permissions, and adds docs/changelog plus focused lifecycle coverage.

## Reproduction Assessment

Yes. A source-level reproduction is enough: on first-time `auth: "oauth"`, the callback promise clears `pendingAuthorizationCode` before runtime calls `finishAuth(code)`, and the pinned MCP SDK then requires `provider.redirectUrl` for authorization-code token exchange. Cached client-info reauthorization also reproduces from source because `redirectUrl` remains undefined and the SDK selects the non-interactive token path.

## Solution Assessment

No. Passing the MCP SDK `authProvider` is the right architectural direction, but this branch is not the best mergeable solution until the OAuth lifecycle, credential-storage, current-main rebase, docs, and changelog defects are fixed against the SDK contract and current runtime shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep the callback session through finishAuth:** `src/agents/mcp-oauth-provider.ts:124-126`
  - body: `waitForAuthorizationCode()` awaits the promise whose `finally` clears `pendingAuthorizationCode`, so by the time the runtime calls `finishAuth(code)`, `provider.redirectUrl` is undefined. The MCP SDK requires that redirect URL for the authorization-code token exchange, so the normal first-time OAuth flow fails before tokens are saved.
  - confidence: 0.95
- **[P1] Create a callback session for cached client info:** `src/agents/mcp-oauth-provider.ts:155-159`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `provider.redirectUrl` undefined. In SDK v1.29.0 that makes `auth()` choose the non-interactive token path before refresh/new redirect handling, so reauthorization after token expiry fails instead of opening the browser.
  - confidence: 0.9
- **[P1] Reconnect with a fresh MCP client and transport:** `src/agents/pi-bundle-mcp-runtime.ts:227`
  - body: After `finishAuth`, this retries `connectWithTimeout` on the same `Client` and transport. SDK `connect()` records the transport before start/initialize completes, and the HTTP/SSE transports retain started state after the failed auth path, so the retry can fail as already connected or already started even with valid tokens.
  - confidence: 0.9
- **[P1] Namespace token files by server identity:** `src/agents/mcp-oauth-provider.ts:61-62`
  - body: The persisted OAuth state path is based only on `serverName`, so two workspaces or MCP URLs using the same display key can share one access/refresh token file. That can send credentials minted for one remote MCP server to another before auth has a chance to correct it.
  - confidence: 0.88
- **[P2] Write OAuth state with private permissions:** `src/agents/mcp-oauth-provider.ts:80-82`
  - body: This writes access and refresh tokens using default directory and file modes, which depend on the process umask. Create the directory and file with restrictive permissions and repair loose existing modes before saving token state.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/config/types.mcp.ts:18-19`
  - body: This adds a user-facing `auth: "oauth"` MCP feature and new credential behavior, but the PR does not include a `CHANGELOG.md` entry. Repo policy requires user-facing feature changes to be recorded with appropriate contributor credit.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR introduces OAuth bearer-token persistence with credential isolation and filesystem-permission weaknesses.

Concerns:

- **[high] OAuth tokens are keyed only by display name:** `src/agents/mcp-oauth-provider.ts:61`
  - body: A global token file derived only from `serverName` can mix credentials across workspaces or different MCP URLs that reuse a key like `github`, risking bearer tokens being sent to the wrong remote server.
  - confidence: 0.88
- **[medium] OAuth token files can be too broadly readable:** `src/agents/mcp-oauth-provider.ts:82`
  - body: The new token persistence path writes access and refresh tokens with default mode bits; on permissive umasks, other local users may be able to read the credential file.
  - confidence: 0.9
