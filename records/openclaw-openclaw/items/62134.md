Full review comments:

- **[P1] Preserve streamable redirect scrubbing:** `src/agents/mcp-transport.ts:105-108`
  - body: Current main routes streamable HTTP MCP through `fetchStreamableHttpWithRedirectScrub`, but the PR-head constructor omits that fetch option. If this branch is merged as resolved here, configured headers or bearer tokens can again be forwarded across cross-origin redirects.
  - confidence: 0.94
- **[P1] Keep the callback session through finishAuth:** `src/agents/mcp-oauth-provider.ts:122-126`
  - body: `pendingAuthorizationCode` is cleared as soon as the callback promise settles, but `redirectUrl` is derived from that pending session. The SDK needs `provider.redirectUrl` during authorization-code exchange, so `finishAuth(code)` can fail immediately after a successful browser callback.
  - confidence: 0.93
- **[P1] Create a callback session for cached clients:** `src/agents/mcp-oauth-provider.ts:155-160`
  - body: When persisted `clientInfo` exists, `clientInformation()` returns without creating a callback session, leaving `redirectUrl` undefined. The MCP SDK treats that as a non-interactive flow, so expired or missing tokens for an already registered client can fail instead of opening the browser again.
  - confidence: 0.92
- **[P1] Namespace OAuth state by server identity:** `src/agents/mcp-oauth-provider.ts:61-62`
  - body: The persisted state path is derived only from `serverName`, so two workspaces or configs that reuse a name for different MCP endpoints can share and overwrite OAuth tokens. Include a stable endpoint or config identity in the key before saving bearer credentials.
  - confidence: 0.88
- **[P2] Write OAuth state with restrictive permissions:** `src/agents/mcp-oauth-provider.ts:77-82`
  - body: Access and refresh tokens are written with default directory and file modes, which can be world-readable under a permissive umask. Create the directory and token file with explicit private permissions, and repair existing files if needed.
  - confidence: 0.9
