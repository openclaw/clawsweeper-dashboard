## Summary

Keep open: this is a protected maintainer-labeled bug-fix PR, and the source path behind the reported OAuth failure is plausible, but the patch needs real behavior proof and should not pass the full persisted auth profile store to every plugin harness.

## What This Changes

The PR loads the persisted auth profile store for plugin-owned harness attempts, passes that store into run attempts, adds OpenAI/Codex auth metadata regression assertions, and updates the changelog.

## Best Possible Solution

Forward only the selected/forwarded openai-codex profile metadata or a scoped auth store to the Codex app-server while keeping unrelated credentials out of generic plugin-harness attempt params.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection shows current main passes an empty authProfileStore to plugin-owned harness attempts while the Codex app-server uses that store to resolve native auth metadata, and the linked beta report provides the observed OAuth-only failure context.

## Solution Assessment

No. Forwarding enough auth metadata to the Codex app-server is the right direction, but this patch uses a broader-than-needed full auth store instead of a scoped selected-profile handoff.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Limit the harness auth store to the forwarded profile:** `src/agents/pi-embedded-runner/run.ts:1213`
  - body: This changes every non-PI harness attempt from receiving the empty auth store to receiving the full persisted agent/default auth profile store. That store can contain inline API keys plus OAuth access and refresh tokens, while the Codex path only needs to verify/use the selected `openai-codex:*` profile. Please pass a scoped store or metadata containing only the forwarded harness profile so unrelated provider credentials are not placed on generic plugin-harness attempt params.
