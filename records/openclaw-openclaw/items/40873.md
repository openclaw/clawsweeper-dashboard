## Summary

Keep open: current main still has the TUI assistant-string path that can render inline reasoning tags, and this PR is useful but not mergeable as-is because it lacks real behavior proof and introduces two focused correctness issues.

## What This Changes

The PR adds inline reasoning-tag extraction/stripping for TUI string message content and formatter regression coverage for assistant string payloads.

## Best Possible Solution

Land a rebased, role-aware TUI formatter fix that parses inline reasoning tags only for assistant string content, matches the shared depth-aware strip semantics, and keeps focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source path is `TuiStreamAssembler` receiving `{ role: "assistant", content: "<think>hidden</think>Visible" }` with thinking display off; current main passes the string through visible content while docs say closed thinking blocks stay hidden.

## Solution Assessment

No. The direction is right, but the current patch should be rebased and narrowed so tagged parsing applies to assistant strings only and the extractor matches existing nested-depth tag semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Restrict tagged parsing to assistant strings:** `src/tui/tui-formatters.ts:304-310`
  - body: `extractTextBlocks` is also used for non-assistant transcript records. Routing every string through reasoning-tag splitting can remove literal `<think>` examples from user or command text, even though only assistant inline reasoning should be hidden.
  - confidence: 0.94
- **[P3] Track nested thinking depth:** `src/shared/text/reasoning-tags.ts:103-114`
  - body: The new extractor flips a boolean while the existing stripper tracks nested depth. For nested `<think>` blocks, thinking display can omit the rest of the outer reasoning even though the content stripper removes it.
  - confidence: 0.82
