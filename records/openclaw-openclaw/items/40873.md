## Summary

Keep this PR open: the underlying TUI formatter bug is still source-reproducible on current main, but this branch still has a blocking role-scoping bug and lacks required after-fix real behavior proof.

## What This Changes

The PR adds inline reasoning-tag extraction for TUI string message content and formatter regression tests so `<think>...</think>` text can be hidden or surfaced separately.

## Best Possible Solution

Land a rebased, role-aware, depth-aware TUI formatter fix for assistant string payloads, with focused regression coverage and real TUI/session proof, while preserving literal `<think>` text in user and command messages.

## Reproduction Assessment

Yes, by source inspection: feed an assistant message with string content like `<think>hidden</think>Visible` through `TuiStreamAssembler` with thinking disabled, and current main routes it through `extractContentFromMessage()` without tag stripping. I did not run a live TUI session because this review is read-only.

## Solution Assessment

No. The PR targets the right bug, but the current patch should restrict tagged parsing to assistant strings, mirror the existing depth-aware tag semantics, and include after-fix real behavior proof before merge.
