## Summary

Keep open: current main still has a source-reproducible TUI string-content leak path, but this PR is not ready because it strips tagged text from non-assistant strings, the new extractor diverges from nested-depth parser semantics, and no after-fix real TUI proof is attached.

## What This Changes

This PR adds reasoning-tag extraction, routes TUI string content through it, and adds formatter regression tests for inline `<think>...</think>` assistant payloads.

## Best Possible Solution

Rebase a narrow TUI fix that applies inline reasoning-tag splitting only to assistant string payloads, mirrors `stripReasoningTagsFromText` semantics, adds regressions for user literals and nested tags, and attaches redacted TUI output proof.

## Reproduction Assessment

Yes, at source level. Current main’s streamed TUI path displays `extractContentFromMessage`, and that helper returns assistant string content after only render sanitization, so `<think>hidden</think>Visible` can remain visible without a live TUI run.

## Solution Assessment

No. The PR targets the right bug, but the implementation should be constrained to assistant-visible string payloads and should share the nested/depth semantics of the existing reasoning-tag stripper.
