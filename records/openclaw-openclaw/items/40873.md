## Summary

Keep open: the TUI string-content leak still has a source-reproducible path on current main, but this branch should not merge as-is because it rewrites non-assistant string transcripts, uses extractor semantics that diverge from current depth-aware stripping, and lacks required real behavior proof.

## What This Changes

The PR adds shared reasoning-tag extraction plus TUI formatter and regression-test changes to split inline `<think>...</think>` blocks out of string message content.

## Best Possible Solution

Land a rebased, role-aware TUI formatter fix that strips and optionally surfaces inline reasoning tags only for assistant string payloads, preserves current phase-aware extraction, reuses depth-aware shared tag semantics, and adds focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source path is `TuiStreamAssembler` receiving `{ role: "assistant", content: "<think>hidden</think>Visible" }` with thinking display off; current main routes that string through TUI helpers that do not strip reasoning tags.

## Solution Assessment

No. The PR addresses the right bug, but the implementation should be rebased and narrowed so tagged parsing applies only to assistant strings and matches the existing depth-aware stripping behavior.
