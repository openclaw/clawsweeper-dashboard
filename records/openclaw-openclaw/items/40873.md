## Summary

Keep this PR open: the underlying TUI reasoning-tag leak is still source-reproducible on current main, but the submitted patch is not merge-ready because it applies tagged parsing too broadly, diverges from the existing nested-tag parser semantics, and lacks real behavior proof.

## What This Changes

The PR adds inline reasoning-tag extraction for TUI string content and regression tests for hiding or showing tagged reasoning.

## Best Possible Solution

Rebase this as an assistant-only TUI/live-chat fix that reuses canonical depth-aware reasoning-tag semantics, preserves literal user and command examples, adds nested/non-assistant regressions, and includes redacted real TUI proof.

## Reproduction Assessment

Yes, at source level. On current main, assistant string content containing <think> tags reaches the TUI helpers without tagged reasoning extraction, and streamed TUI display uses those helpers.

## Solution Assessment

No. The PR targets the right bug, but the implementation should be constrained to assistant-visible strings and should share the existing nested/depth-aware reasoning-tag semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Restrict tagged parsing to assistant strings:** `src/tui/tui-formatters.ts:304-310`
  - body: `extractTextBlocks` is also used for non-assistant records through `extractTextFromMessage`, so routing every string through `resolveTaggedStringContentParts` can remove literal `<think>...</think>` examples from user or command transcript text. The hiding rule should apply only to assistant output.
  - confidence: 0.94

  - body: The new extractor tracks a boolean while the existing stripper tracks nested `thinkingDepth`. For nested tags, thinking display can omit part of the outer reasoning even though the visible-content stripper removes the entire block, creating mismatched output.
  - confidence: 0.82
