## Summary

Keep open: current main and the latest release partially improve the missing local-provider guidance and doctor repair path, but the reported upgrade break is not fully fixed because memory CLI still resolves through the memory-core manager path while configured provider-plugin activation is gateway/startup-oriented, and the local model identity mismatch remains only covered by open, unmerged follow-up work.

## What This Changes

The issue reports a release upgrade regression where `memorySearch.provider: "local"` can lose the llama.cpp provider runtime, produce misleading provider errors, and reject or dirty existing local embedding indexes after model identity changes.

## Best Possible Solution

Keep llama.cpp embeddings plugin-owned, but make the upgrade path coherent by aligning memory CLI provider-plugin activation with gateway behavior and adding a deliberate migration, normalization, or reindex path for old local model identities.

## Reproduction Assessment

Yes for the source-level split, but not for a full live upgrade replay: current code shows memory CLI direct manager use plus gateway-only configured provider-plugin startup, and exact model identity checks explain the reported path mismatch. I did not run the Ubuntu/npm two-release upgrade scenario.

## Solution Assessment

No, current main is only a partial solution: it improves errors and doctor repair, but the direct CLI provider activation and old local model identity compatibility still need a maintainer-chosen fix path. The safest end state is a coordinated compatibility repair rather than another one-off fallback.
