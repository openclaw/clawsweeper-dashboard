## Summary

Keep open: current main fixes several reported slices, including the local-provider install guidance and default-model identity mismatch, but the CLI reindex path still appears unproven and likely incomplete because lazy CLI plugin loading scopes the memory command to its command owner rather than the configured embedding-provider plugin.

## What This Changes

The issue reports that local memory embeddings broke across upgrades because the llama.cpp provider extraction, model identity changes, CLI indexing behavior, and error messages left users without a clear migration path.

## Best Possible Solution

Keep the current targeted fixes, then add a maintainer-approved upgrade path for local embeddings that either loads the configured provider plugin for CLI reindexing or documents and validates the supported gateway-routed reindex flow with live upgrade proof.

## Reproduction Assessment

Source-reproducible, not live-reproduced: the issue body gives concrete upgrade symptoms, and current source still shows a plausible CLI-only gap where memory command loading is scoped to the memory command owner while local embeddings depend on a separate provider plugin. A live packaged upgrade remains missing.

## Solution Assessment

No: the merged model-identity repair is a good partial fix, but it is not the complete solution because the CLI indexing/plugin-loading and upgrade-proof pieces remain unresolved. The safer complete path is a maintainer-approved CLI/gateway migration behavior plus focused live upgrade validation.
