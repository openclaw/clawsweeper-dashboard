## Summary

Keep open: current main and the latest release have improved the missing-provider guidance and doctor repair path, but the reported upgrade-compatibility problem around local model identity still needs a maintainer decision and live upgrade proof.

## What This Changes

The issue reports that upgrading local memory embeddings from built-in llama.cpp support to the provider plugin can leave memory search/indexing broken, with misleading diagnostics and no clear migration for existing local model/index identity.

## Best Possible Solution

Maintainers should decide the upgrade contract for old local embedding indexes, then either add a doctor migration/repair that safely normalizes compatible local model identities or keep the current explicit-rebuild behavior with clearer release/doctor guidance.

## Reproduction Assessment

Yes, source inspection gives a medium-confidence reproduction path for the remaining identity issue: a stored index model identity that differs from the current local provider model identity is still treated as mismatched. I did not run the full release-upgrade/local GGUF flow here.

## Solution Assessment

Unclear. Current main has the right direction for missing-plugin guidance and plugin repair, but the best remaining fix depends on whether maintainers want a compatibility migration for old local model identities or an intentional rebuild-only contract.
