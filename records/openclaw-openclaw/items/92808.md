## Summary

The issue should stay open because current main has improved the missing-provider guidance and doctor repair path, but source inspection still supports the reported split where memory CLI commands do not load configured embedding provider plugins the same way gateway startup does, and the local model identity mismatch remains an upgrade-compatibility question.

## What This Changes

The issue asks OpenClaw to restore a smooth upgrade path for local memory embeddings after the llama.cpp runtime moved into the external provider plugin, including CLI behavior, doctor repair, and old index/model identity handling.

## Best Possible Solution

Keep local GGUF embeddings plugin-owned, but make the upgrade path consistent by ensuring doctor and memory CLI use the same configured provider-plugin activation model as gateway and by adding a deliberate migration or reindex path for old local model identities.

## Reproduction Assessment

Source-reproducible: current code shows memory CLI manager access through memory-core while command policy defaults to no plugin loading, matching the reported CLI/gateway behavior split, but no live upgrade repro was run.

## Solution Assessment

No: current main improves errors and doctor repair, but the remaining CLI activation and old-index identity handling need maintainer-directed compatibility work rather than closure.
