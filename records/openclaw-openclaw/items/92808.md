## Summary

Current main now fixes the central local-memory upgrade breakage: the llama.cpp provider supplies stable identity aliases for the official default GGUF model, memory-core accepts provider-declared identity aliases, and the remaining package/install guidance is already handled by current docs, doctor, and prior closed subreports.

## What This Changes

The issue reports local memory search breaking after OpenClaw upgrades when `memorySearch.provider` is `local`, including missing llama.cpp plugin repair, misleading provider errors, and HF URI versus filesystem-path index identity mismatches.

## Best Possible Solution

Keep the current-main provider-owned identity compatibility and explicit plugin repair path, then ship it in the next release rather than keeping this umbrella issue open.

## Reproduction Assessment

No live v2026.6.5-to-v2026.6.6 package upgrade was run in this read-only pass. Source inspection and the merged PR proof show the former mismatch path: the same official default local model could be stored as an HF URI and later reported as a downloaded GGUF path.

## Solution Assessment

Yes. The current-main implementation is the maintainable layer because the llama.cpp provider owns its model identity aliases and memory-core only accepts provider-declared identities instead of broad path equivalence or a new fallback.
