## Summary

Keep open: current main still has the reported Bedrock AWS SDK auth mismatch, with documented no-key Bedrock auth resolving correctly but the image/media-understanding path still requiring a static API key before runtime execution.

## What This Changes

Allow the image tool/media-understanding path to use Amazon Bedrock `auth: "aws-sdk"` without a static API key, matching text chat behavior.

## Best Possible Solution

Allow `aws-sdk` no-static-key auth through the image/media-understanding runtime boundary, keep strict API-key enforcement for key-based providers, and cover Bedrock image calls with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to make `getApiKeyForModel` return `{ mode: "aws-sdk", apiKey: undefined }` for an image-capable `amazon-bedrock` model and call the image tool path; current main reaches `requireApiKey` and throws before the AWS SDK chain can run.

## Solution Assessment

Yes. The narrow maintainable fix is to share the existing AWS SDK missing-key allowance with the image/media runtime paths, while auditing callers instead of globally changing a string-returning helper to return `undefined` unchecked.

## Repair Work Prompt

Fix openclaw/openclaw#77397. Implement a narrow fix that allows `aws-sdk` no-static-key auth through the direct image runtime and any affected media-understanding execution boundary, without weakening API-key enforcement for `api-key`, `token`, or `oauth` providers.
