## Summary

Keep open: current main still has the reported Bedrock aws-sdk auth mismatch in image/media paths, and an open PR already references this issue with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Allow Bedrock image/media execution to use `auth.mode: "aws-sdk"` without requiring a pasted static `apiKey`.

## Best Possible Solution

Land or revise the linked media-runtime fix so image understanding and media runner execution accept Bedrock aws-sdk credential-chain auth without weakening API-key enforcement for other modes.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure `amazon-bedrock` with `auth: "aws-sdk"` and no `apiKey`, then invoke image/media understanding so `getApiKeyForModel` or `resolveApiKeyForProvider` returns `mode: "aws-sdk"` with no key and current main reaches `requireApiKey`.

## Solution Assessment

Yes. The narrow maintainable solution is to mirror the existing text/runtime aws-sdk no-static-key allowance in the image/media runtime boundary, with focused regression tests for `describeImageWithModel` and runner auth setup.
