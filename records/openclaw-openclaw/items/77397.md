## Summary

Keep open: current `main` still has the reported Bedrock AWS SDK auth mismatch. The text path has an explicit `aws-sdk` no-key carve-out, but image/media-understanding paths still call `requireApiKey` before the AWS SDK credential chain can run.

## What This Changes

The issue requests that Bedrock image/media-understanding runs support `auth: "aws-sdk"` without a configured `apiKey`, matching documented Bedrock auth and the text runtime path.

## Best Possible Solution

Allow `aws-sdk` no-static-key auth through the image/media-understanding runtime boundary, preserve strict API-key enforcement for normal key-based providers, and add focused Bedrock regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to mock `getApiKeyForModel`/`resolveApiKeyForProvider` returning `{ mode: "aws-sdk", apiKey: undefined }` for an image-capable `amazon-bedrock` model and call the image/media-understanding path; current `main` then reaches `requireApiKey` and throws.

## Solution Assessment

Yes. The narrow maintainable fix is to make the media runtime use the same AWS SDK no-key allowance as the text path, preferably through a small shared helper rather than changing unrelated provider auth behavior.
