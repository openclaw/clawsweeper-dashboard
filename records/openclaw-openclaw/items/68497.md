## Summary

Keep open. Current main still has no browser-originated canvas action transport: the injected helper only posts to native iOS/Android handlers, the browser WebSocket is reload-only, and the canvas host rejects non-GET action POSTs. The related web canvas-node issue remains open and should be coordinated, but it does not supersede this narrower action-bridge request.

## What This Changes

The issue requests exposing a web-browser fallback for window.openclawSendUserAction so canvas A2UI actions can be routed back to Gateway without an iOS or Android native bridge.

## Best Possible Solution

Define a reviewed browser canvas action transport at the Canvas plugin/Gateway boundary with schema validation, existing canvas auth/capability checks, origin/CSRF handling, action scoping, rate limiting, status feedback, and coordination with https://github.com/openclaw/openclaw/issues/66983.

## Reproduction Assessment

Yes, at source level. Opening the canvas page in a normal browser without WebKit or Android handlers leaves sendUserAction returning false, POST requests are rejected with 405, and the existing WebSocket only handles reload messages.

## Solution Assessment

Unclear. The requested capability is valid, but the suggested same-origin POST shape is not enough by itself until maintainers choose the browser action transport and security boundary.
