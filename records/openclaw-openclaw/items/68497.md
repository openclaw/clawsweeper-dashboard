## Summary

Keep open. Current main still has no browser-originated canvas action transport: the injected helper only posts to native iOS/Android handlers, the canvas WebSocket is reload-only, and non-GET canvas host requests are rejected. The related web canvas-node issue should be coordinated, but it is still open and does not make this action-bridge request implemented.

## What This Changes

The issue requests a browser-side `window.openclawSendUserAction` fallback that routes A2UI canvas actions from desktop web browsers back to Gateway.

## Best Possible Solution

Define a reviewed browser canvas action transport in the canvas plugin/Gateway boundary, with schema validation, existing canvas auth/capability checks, origin/CSRF handling, action scoping, rate limiting, action-status feedback, and coordination with https://github.com/openclaw/openclaw/issues/66983.

## Reproduction Assessment

Yes, at source level. Open the canvas page in a normal browser without WebKit/Android handlers and the helper returns false; POST is rejected with 405 and the existing WebSocket only handles reload messages.

## Solution Assessment

Unclear. The requested capability is valid, but the suggested same-origin POST shape is not enough by itself until maintainers choose the browser action transport and security boundary.
