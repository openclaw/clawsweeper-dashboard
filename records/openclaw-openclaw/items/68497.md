## Summary

Keep open. Current main still has no browser-originated Canvas/A2UI action transport: the injected helper only posts to native iOS/Android handlers, the Canvas WebSocket is reload-only, and the host still rejects non-GET/HEAD action POSTs. The broader web canvas-node issue at https://github.com/openclaw/openclaw/issues/66983 is related and should be coordinated, but it does not supersede this narrower action-bridge request.

## What This Changes

Request to expose a web-browser fallback for `window.openclawSendUserAction` that routes Canvas/A2UI user actions back to the Gateway over WebSocket or HTTP.

## Best Possible Solution

Define a reviewed browser Canvas action transport at the Canvas plugin/Gateway boundary with strict payload schema validation, explicit bearer or capability auth, origin/CSRF handling, action scoping, rate limiting, status feedback, tests, and coordination with the web canvas-node design.

## Reproduction Assessment

Yes, at source level. In current main, `sendUserAction` exists but only posts to native iOS/Android handlers, the Canvas WebSocket only handles `reload`, and a browser action POST under the Canvas host returns 405.

## Solution Assessment

No, not exactly as suggested. The requested capability is valid, but same-origin POST or WebSocket fallback should only be added after maintainers define the browser-to-Gateway action trust boundary and capability semantics.
