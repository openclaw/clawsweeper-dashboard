## Summary

Keep this issue open. Current main still defines `window.openclawSendUserAction`, but in a normal browser it only tries native WebKit/Android handlers and there is still no Gateway POST or WebSocket action path for browser-originated canvas actions; related #66983 is broader web canvas-node support and does not fully replace this narrower fallback request.

## What This Changes

The issue requests a browser fallback for canvas A2UI user actions so `window.openclawSendUserAction` can route actions to the Gateway outside native iOS/Android bridges.

## Best Possible Solution

Add a typed browser canvas action transport that reuses existing canvas auth/capability checks, validates and scopes A2UI user-action payloads, dispatches through the same agent-request path used by native nodes, and emits `openclaw:a2ui-action-status` for success and failure.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: open the served canvas page in a normal browser without native WebKit/Android handlers, click a button, and the helper returns false because the only browser WebSocket behavior is reload and there is no POST action route.

## Solution Assessment

Unclear. The requested capability is valid, but the best implementation needs maintainer security/product direction on whether the browser fallback should be a same-origin POST endpoint or a typed extension of the existing canvas WebSocket protocol.
