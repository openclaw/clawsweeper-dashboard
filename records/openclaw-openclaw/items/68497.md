## Summary

Keep open. Current main and v2026.6.8 still expose only a native iOS/Android-style Canvas action helper, with no Gateway-routed browser action ingress; the remaining work is a product and security-boundary decision, not conservative cleanup.

## What This Changes

Request to add browser-routed Canvas/A2UI user-action ingress so `window.openclawSendUserAction` works outside native iOS/Android nodes.

## Best Possible Solution

Define a plugin-owned, security-reviewed Canvas browser action ingress with explicit Gateway auth, origin, capability, action-scope, replay, revocation, tests, docs, and coordination with web Canvas-node support.

## Reproduction Assessment

Yes, at source level. Current main injects the helper but routes it only to native bridge handlers, and Canvas HTTP/WebSocket handling has no browser-origin action ingress.

## Solution Assessment

No, not as the suggested same-origin POST or WebSocket fallback alone. The narrow endpoint shape is plausible, but the best fix needs an explicit security/product contract for action authorization, origin scope, capability tokens, replay, and revocation.
