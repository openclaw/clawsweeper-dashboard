## Summary

Keep open. Current main still lacks a browser-originated Canvas/A2UI action transport, and the related web Canvas-node issue is adjacent rather than a replacement for this narrower action bridge request.

## What This Changes

The issue requests a browser-accessible Canvas A2UI action bridge so `window.openclawSendUserAction` can route web page actions back to the Gateway instead of only native iOS/Android handlers.

## Best Possible Solution

Define a reviewed browser Canvas action transport at the Canvas plugin/Gateway boundary with strict payload schema validation, explicit bearer or scoped capability auth, origin/CSRF handling, action scoping, rate limiting, status feedback, tests, and coordination with https://github.com/openclaw/openclaw/issues/66983.

## Reproduction Assessment

Yes, at source level. Current main still injects a helper that only posts to native handlers, the Canvas WebSocket only handles reload, and browser action POSTs under the Canvas host would hit the non-GET/HEAD 405 path.

## Solution Assessment

No, not exactly as suggested. The requested capability is valid, but an unconditional same-origin POST or WebSocket fallback should wait for a maintainer-reviewed browser-to-Gateway action trust boundary.
