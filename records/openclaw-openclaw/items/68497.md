## Summary

Keep open. Current main still lacks a browser-originated canvas action transport: the injected helper posts only to native iOS/Android handlers, the canvas WebSocket remains reload-only, and the canvas host still rejects non-GET/HEAD requests. The related web canvas-node request at https://github.com/openclaw/openclaw/issues/66983 should be coordinated, but it is still open and does not supersede this browser action-bridge request.

## What This Changes

Expose web browser canvas A2UI user actions through a Gateway-backed WebSocket or POST fallback for `window.openclawSendUserAction`.

## Best Possible Solution

Define a reviewed browser canvas action transport in the canvas plugin/Gateway boundary, with schema validation, existing canvas auth/capability checks, origin/CSRF handling, action scoping, rate limiting, action-status feedback, and coordination with https://github.com/openclaw/openclaw/issues/66983.

## Reproduction Assessment

Yes, at source level. In a normal browser without native WebKit/Android handlers, the current helper has no Gateway action path, `/__openclaw__/canvas/action` would hit the non-GET 405 path, and the existing WebSocket only handles reload messages.

## Solution Assessment

Unclear. The requested capability is valid, but the suggested same-origin POST/WebSocket fallback is not sufficient by itself until maintainers choose the browser action transport and security boundary.
