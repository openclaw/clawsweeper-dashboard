## Summary

Keep this issue open. Current main exposes `window.openclawSendUserAction`, but the helper still only posts to native iOS/Android bridges; browser-originated A2UI actions are not routed to the Gateway, and there is no authenticated canvas action POST or WebSocket action protocol. Related #66983 is broader web canvas-node support and does not fully supersede this narrower fallback request.

## What This Changes

Request to add a browser fallback for the canvas A2UI action bridge so `window.openclawSendUserAction` can route regular web-browser actions to the Gateway via WebSocket or an authenticated POST endpoint.

## Best Possible Solution

Keep the issue open and add a core canvas/gateway transport for browser A2UI user actions: define a typed payload, authenticate and rate-limit it through the existing canvas auth boundary, dispatch it to the same action path native canvas nodes use, emit `openclaw:a2ui-action-status`, update the A2UI host/bootstrap, and cover normal-browser plus auth-failure cases in focused tests.

## Reproduction Assessment

Yes. A regular browser lacks both native bridge objects, and current main’s helper returns `false` in that case; the default canvas page then logs `Failed to send action: <name>`. Source inspection also confirms there is no POST action route and the canvas WebSocket only handles reloads.

## Solution Assessment

Unclear. The requested browser fallback is the right capability, but the maintainable shape should be a typed authenticated Gateway action transport rather than only making the helper return success; maintainers need to choose the narrow fallback design versus folding this into #66983.
