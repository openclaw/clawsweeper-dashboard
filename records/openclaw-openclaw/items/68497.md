## Label Justifications

## Summary

Keep open. Current main partially exposes `window.openclawSendUserAction` in hosted Canvas HTML, but it still posts only to native iOS/Android bridges and there is no Gateway-routed browser action ingress; this remains security-sensitive product work, not conservative cleanup.

## What This Changes

Requests a browser fallback for Canvas/A2UI user actions so `window.openclawSendUserAction` can route actions from Gateway-hosted web canvases back to the Gateway.

## Best Possible Solution

Define a plugin-owned, security-reviewed Canvas web action ingress with explicit Gateway auth, origin/capability/action-scope semantics, tests, docs, and coordination with broader web Canvas node support.

## Reproduction Assessment

Yes, at source level. Current main exposes the helper but routes it only to native bridge handlers, and Canvas HTTP handling has no POST or WebSocket action path for browser-origin user actions.

## Solution Assessment

No, not enough to accept the suggested endpoint as-is. A same-origin POST/WebSocket path is plausible, but the best solution needs an explicit security/product contract for action authorization, origin scope, capability tokens, and coordination with web Canvas nodes.
