## Summary

Keep open. Current main and the latest release still expose only a native iOS/Android-style Canvas action helper, with no Gateway-routed browser action ingress; this remains product and security-boundary work rather than conservative cleanup.

## What This Changes

The issue requests a web browser fallback for `window.openclawSendUserAction` so hosted Canvas/A2UI pages can send user actions back to the Gateway without an iOS or Android native bridge.

## Best Possible Solution

Define a plugin-owned, security-reviewed Canvas browser action ingress with explicit Gateway auth, origin, capability, action-scope, replay, revocation, tests, docs, and coordination with web Canvas-node support.

## Reproduction Assessment

Yes, at source level. Current main exposes the helper but routes it only to native bridge handlers, and Canvas HTTP/WebSocket handling has no browser-origin action ingress.

## Solution Assessment

No, not as the suggested endpoint alone. A same-origin POST or WebSocket fallback is plausible, but the best fix needs an explicit security/product contract for action authorization, origin scope, capability tokens, replay, and revocation.
