## Summary

Keep open. Current main and v2026.6.8 still expose Canvas A2UI actions only through native iOS/Android-style bridge handlers, with no Gateway-routed browser action ingress; this remains product and security-boundary work rather than conservative cleanup.

## What This Changes

Request to expose a browser-capable Canvas A2UI action bridge so `window.openclawSendUserAction` can send user actions back to the Gateway from regular web browsers.

## Best Possible Solution

Define a security-reviewed browser Canvas action ingress contract with explicit Gateway auth, origin, capability/action scope, replay and revocation semantics, focused tests, docs, and coordination with web Canvas-node support.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 expose the helper but route it only to native bridge handlers, and Canvas HTTP/WebSocket handling has no browser-origin action ingress.

## Solution Assessment

No, not as the suggested endpoint alone. A same-origin POST or WebSocket fallback is plausible, but the best fix needs an explicit security/product contract for authorization, origin scope, capability tokens, replay, and revocation.
