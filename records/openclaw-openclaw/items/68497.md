## Label Justifications

## Summary

Keep open. Current main and v2026.6.10 still expose the Canvas action helper only as a native iOS/Android bridge wrapper, with no browser-origin Gateway action ingress; the remaining work is a product and security-boundary decision, not conservative cleanup.

## What This Changes

Request to make browser-loaded Canvas/A2UI pages send user actions back to the Gateway through `window.openclawSendUserAction`, WebSocket, or a same-origin action endpoint.

## Best Possible Solution

Define a security-reviewed browser Canvas action ingress contract, then implement the narrow helper/route with scoped auth, origin/capability checks, replay protections, tests, and public docs.

## Reproduction Assessment

Yes, at source level. Current main injects the helper but routes it only to native bridge handlers, and Canvas HTTP/WebSocket handling has no browser-origin action ingress.

## Solution Assessment

No, not as the suggested endpoint alone. A POST or WebSocket fallback is plausible, but the best fix needs an explicit security/product contract for authorization, origin scope, capability tokens, replay, and revocation.
