## Summary

Keep this issue open. Current main still exposes `window.openclawSendUserAction` only as a native iOS/Android bridge helper; browser-originated actions are not routed to the Gateway over the canvas WebSocket or an authenticated HTTP action endpoint. Related #66983 is broader web canvas-node support and does not fully supersede this narrower fallback request.

## What This Changes

The issue requests a browser fallback for `window.openclawSendUserAction` so served canvas/A2UI button actions can reach the Gateway via WebSocket or an authenticated HTTP endpoint outside native iOS/Android WebViews.

## Best Possible Solution

Keep the issue open and design a core canvas/gateway action transport: authenticate it through existing canvas bearer/capability checks, validate a typed A2UI user-action payload, dispatch it through the same action handling path used by native canvas nodes, emit `openclaw:a2ui-action-status`, and cover normal browser, unauthorized, malformed, and native-bridge fallback cases in focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: open the served canvas page in a normal browser with no native WebKit/Android bridge, click a button, and `openclawSendUserAction` returns false because there is no Gateway action route or WebSocket action protocol.

## Solution Assessment

Unclear. The requested browser fallback is the right capability, but the best implementation needs maintainer/security direction on whether the transport should be a typed POST endpoint or an extension of the canvas WebSocket protocol.
