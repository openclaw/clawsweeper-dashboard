## Summary

Keep this issue open. Current main still lacks a browser-originated canvas action transport: the injected helper only posts to native WebKit/Android handlers, the canvas WebSocket is live-reload only, and Gateway has no POST/WebSocket dispatch stage for browser A2UI actions; related #66983 is broader web canvas-node support and does not fully replace this narrower fallback request.

## What This Changes

The issue requests a browser fallback for canvas A2UI user actions so desktop/web canvas pages can route button clicks back to Gateway without native iOS or Android bridges.

## Best Possible Solution

Add a typed browser canvas action transport that reuses canvas bearer/capability auth, validates and scopes A2UI action payloads, dispatches through the same agent-request boundary as native nodes, and returns `openclaw:a2ui-action-status` results.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: open the served canvas/A2UI page in a normal browser without native WebKit or Android handlers, click an action, and the helper returns false or the A2UI host reports `missing native bridge` because no browser POST/WebSocket action route exists.

## Solution Assessment

Unclear. The requested capability is valid, but the suggested POST/WS fallback needs maintainer security/product direction so browser JavaScript cannot become an under-scoped agent-request ingress.
