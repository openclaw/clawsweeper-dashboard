## Summary

Keep open. Current main and v2026.6.10 still only route Canvas A2UI user actions through native iOS/Android-style bridge handlers; there is no browser-origin Gateway action ingress, so the remaining work is a security-sensitive product/API decision rather than cleanup.

## What This Changes

The issue requests a browser-accessible Canvas A2UI action bridge so web canvas pages can send user actions back to the Gateway without native iOS or Android bridges.

## Best Possible Solution

Define a security-reviewed browser Canvas action ingress contract in the Canvas plugin/Gateway boundary, then implement the narrow helper and route with auth, origin, capability, replay, revocation, tests, and public docs.

## Reproduction Assessment

Yes, at source level. Current main injects `openclawSendUserAction`, but it only posts to native bridge handlers, while Canvas HTTP/WebSocket handling has no browser-origin action ingress.

## Solution Assessment

No, not as the suggested POST or WebSocket fallback alone. The maintainable fix needs an explicit security and product contract for authorization, origin scope, capability tokens, replay/idempotency, revocation, and action routing.
