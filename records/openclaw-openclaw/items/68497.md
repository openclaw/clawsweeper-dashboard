## Summary

Keep open. Current main still exposes the helper only as a native WebKit/Android bridge wrapper, with no browser-originated POST or WebSocket action transport back into Gateway.

## What This Changes

Request to let desktop/web canvas pages submit A2UI user actions to Gateway through a browser-safe fallback for `window.openclawSendUserAction`.

## Best Possible Solution

Add a typed browser canvas action transport in the canvas plugin/Gateway boundary, with schema validation, existing canvas auth/capability checks, origin/CSRF handling, action scoping, rate limiting, `agent.request` dispatch, and `openclaw:a2ui-action-status` feedback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: open the canvas page in a normal browser without WebKit/Android handlers and call a button or `window.openclawSendUserAction`; the helper returns false, POST is rejected with 405, and the WebSocket only supports reload.

## Solution Assessment

Unclear. The requested capability is valid, but the suggested same-origin POST shape is not sufficient by itself until maintainers choose the browser action transport and security boundary.
