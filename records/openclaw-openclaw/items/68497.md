## Summary

Keep open. Current main still has no browser-originated canvas action transport: the injected helper only posts to native iOS/Android handlers, the canvas WebSocket remains reload-only, and non-GET canvas host requests are rejected before any action dispatch.

## What This Changes

Request to route A2UI canvas user actions from ordinary web browsers back to the Gateway via a WebSocket or HTTP fallback for `window.openclawSendUserAction`.

## Best Possible Solution

Define a reviewed browser canvas action transport in the Canvas plugin/Gateway boundary with schema validation, canvas capability checks, origin/CSRF handling, action scoping, rate limiting, status feedback, and coordination with the web canvas-node design.

## Reproduction Assessment

Yes, at source level. A normal browser can load the canvas helper, but action submission still returns false without WebKit/Android handlers, `/__openclaw__/ws` only reloads pages, and non-GET canvas requests return 405.

## Solution Assessment

Unclear. The requested capability is valid, but the proposed same-origin POST or WebSocket fallback is not enough until maintainers choose the browser action transport and security boundary.
