## Summary

Keep this issue open. Current main still has a native-handler-only A2UI action helper and no browser POST or WebSocket action dispatch path, so the requested browser fallback remains unresolved and needs maintainer product/security direction.

## What This Changes

Expose a browser fallback for canvas A2UI user actions so `window.openclawSendUserAction` can route actions from regular web pages back to the Gateway.

## Best Possible Solution

Add a typed browser canvas action transport behind existing canvas auth/capability checks, validate and scope A2UI payloads, dispatch through the same agent-request path as native nodes, and emit the existing action-status event.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: in a normal browser with no native bridge, `window.openclawSendUserAction` calls `postToNode`, which returns `false`, and current main has no POST or outbound WebSocket action route.

## Solution Assessment

Unclear. The requested capability is reasonable, but the best implementation needs maintainer direction on POST versus WebSocket plus the required auth, origin, schema, rate-limit, and session-scoping boundaries.
