## Summary

Keep open. Current main and latest release still expose only a native iOS/Android Canvas action helper, with no Gateway-routed browser action ingress; the remaining work is a security-sensitive product/API decision, not conservative cleanup.

## What This Changes

Request to make web-hosted Canvas/A2UI actions call `window.openclawSendUserAction` and route back to the Gateway through WebSocket or HTTP POST.

## Best Possible Solution

Define a plugin-owned, security-reviewed Canvas web action ingress with explicit Gateway auth, origin/capability/action-scope semantics, tests, docs, and coordination with broader web Canvas node support.

## Reproduction Assessment

Yes, at source level. Current main exposes the helper but routes it only to native bridge handlers, and Canvas HTTP/WebSocket handling has no browser-origin user-action ingress.

## Solution Assessment

No, not as the suggested endpoint alone. A same-origin POST or WebSocket path is plausible, but the maintainable solution needs an explicit security/product contract for authorization, origin scope, capability tokens, and related web Canvas-node behavior.
