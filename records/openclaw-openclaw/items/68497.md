## Summary

Keep open. Current main still lacks a browser-originated canvas action transport: the injected helper posts only to native iOS/Android handlers, the WebSocket path is reload-only, and the host still rejects non-GET/HEAD action POSTs. The broader web canvas-node issue at https://github.com/openclaw/openclaw/issues/66983 remains open and should be coordinated, but it does not supersede this narrower action-bridge request.

## What This Changes

Requests a browser fallback for `window.openclawSendUserAction` so web canvas pages can send A2UI user actions back to the Gateway via WebSocket or HTTP.

## Best Possible Solution

Define a reviewed browser canvas action transport at the Canvas plugin/Gateway boundary with schema validation, existing canvas bearer or capability checks, origin/CSRF handling, action scoping, rate limiting, status feedback, tests, and coordination with the open web canvas-node design.

## Reproduction Assessment

Yes, at source level. In a normal browser without WebKit or Android handlers, `sendUserAction` returns false, non-GET/HEAD action POSTs are rejected with 405, and the existing WebSocket only handles reload messages.

## Solution Assessment

No, not exactly as suggested. The requested capability is valid, but treating same-origin POST auth as enough is too narrow; maintainers should choose a transport and explicit browser action trust boundary first.
