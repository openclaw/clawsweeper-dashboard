## Summary

Keep this issue open. Current main and the latest release still expose `window.openclawSendUserAction` as a native WebKit/Android bridge only, with no browser POST or WebSocket action-dispatch path; the remaining work needs maintainer security/product direction and coordination with #66983.

## What This Changes

Expose a browser fallback for `window.openclawSendUserAction` that routes canvas A2UI user actions back to Gateway over an authenticated WebSocket or same-origin POST.

## Best Possible Solution

Add a typed authenticated browser action transport that reuses canvas bearer/capability auth, validates and scopes A2UI action payloads, dispatches through the same agent-request boundary as native clients, and aligns with the web canvas-node direction in #66983.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: open the served canvas page in a normal browser with no WebKit/Android handler and click a button; the helper returns false, and the suggested POST endpoint would currently hit the canvas host's 405 path.

## Solution Assessment

Unclear. The requested capability is valid, but the best implementation needs maintainer direction on POST versus WebSocket, payload validation, capability scope, origin/CSRF handling, and how it should relate to #66983.
