## Summary

Keep open. Current main and v2026.6.10 still lack browser-origin Canvas/A2UI action ingress; the remaining work is a security and product contract, not conservative cleanup.

## What This Changes

The issue requests a browser fallback for `window.openclawSendUserAction` that routes Canvas A2UI user actions back to the Gateway over WebSocket or a same-origin POST endpoint.

## Best Possible Solution

Define a security-reviewed browser Canvas action ingress contract, then implement the narrow helper and route with scoped auth, capability/origin checks, replay protections, tests, and public docs.

## Reproduction Assessment

Yes, as source-level absence proof. Current main injects `openclawSendUserAction`, but it only posts to native handlers and the Canvas host has no browser-origin action POST or WebSocket ingress.

## Solution Assessment

No, not as a bare same-origin POST or WebSocket fallback. The maintainable fix needs an explicit security/product contract for auth, origin scope, capabilities, replay, revocation, and action authorization.
