## Summary

Keep this open: the initial `google-gemini-cli` 404 is expected for a Claude model, but the reporter's follow-up points at a real current mismatch where `google-antigravity` is still referenced by tests/live docs while the shipped Google plugin does not expose it as an auth-capable provider.

## What This Changes

The issue asks how to configure OpenClaw so Claude models available through Antigravity/Gemini Code Assist work in the web UI instead of failing through `google-gemini-cli` or missing `google-antigravity` auth discovery.

## Best Possible Solution

Maintain one canonical Antigravity decision: either restore a documented auth-capable provider/plugin path for `google-antigravity`, or remove/update the stale Antigravity docs/tests and point users to supported providers or a ClawHub/external plugin.

## Reproduction Assessment

Yes from source for the remaining auth/setup mismatch: the command has the exact `No provider plugins found` path, and `google-antigravity` is hook-only rather than auth-discoverable in the bundled Google provider. No for a full linux_arm64 packaged reproduction, which still needs live proof.

## Solution Assessment

Unclear. Routing Claude away from `google-gemini-cli` is correct, but the durable solution requires maintainer direction on whether Antigravity should be restored as a public provider or treated as an external/plugin-only path with docs cleanup.
