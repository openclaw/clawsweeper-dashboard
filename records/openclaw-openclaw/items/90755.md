## Label Justifications

- `P2`: The issue is a normal-priority Control UI session continuity bug with visible but bounded impact.
- `impact:session-state`: The reported failure is session/run state drift after returning to a running chat session.
- `impact:message-loss`: The UI hides in-progress assistant output and can make an active response appear lost or stalled.

## Summary

Keep open: current main and the latest release still leave the Control UI history loader without an `inFlightRun` adoption path, and prior direct-adoption PRs were closed because the gateway buffer is cumulative and can duplicate already persisted active-turn text.

## What This Changes

The issue asks for Control UI session reconnect or switch-back to show the active assistant response, tool progress, or at least a running indicator instead of only the last committed user message.

## Best Possible Solution

Define a canonical active-run history snapshot that separates already persisted progress from the unpersisted portion, then have Control UI and TUI adopt that contract with focused unit and browser proof.

## Reproduction Assessment

Yes, at source level: current main emits `chat.history.inFlightRun` from the gateway and the TUI consumes it, while the Control UI history loader neither types nor adopts it. I did not run a live browser reconnect replay in this read-only review.

## Solution Assessment

No merged solution exists yet, and the direct UI adoption approach from the closed PRs is not the best fix. The maintainable path is the maintainer-requested canonical gateway/history contract for the unpersisted active-run portion.
