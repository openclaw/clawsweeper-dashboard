## Summary

Keep open: current main and the latest release still leave Control UI without a safe active-run restoration path, and the prior direct `inFlightRun.text` adoption PRs were closed because the existing buffer can duplicate already-persisted assistant progress.

## What This Changes

The issue reports that returning to a running Control UI chat session can show only the last committed user message while the agent's in-progress response remains active but invisible.

## Best Possible Solution

Design a canonical gateway/history contract that exposes or reconciles only the unpersisted active-run portion, then update Control UI and keep TUI aligned against that single contract with real reconnect proof.

## Reproduction Assessment

Yes by source, but not by a live run in this read-only review. Current gateway/TUI paths expose and consume `inFlightRun`, while Control UI has no `inFlightRun` references and can clear local stream state on history reload.

## Solution Assessment

No current implementation on main solves it. Direct Control UI adoption of cumulative `inFlightRun.text` is not the best fix after maintainer review; the safer path is a canonical unpersisted active-run contract shared by gateway, Control UI, and TUI.
