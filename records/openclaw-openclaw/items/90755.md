## Summary

Keep open: current main and the latest release still leave the Control UI history loader unaware of `chat.history.inFlightRun`, and the open linked PR now owns the narrow fix path but has not merged yet.

## What This Changes

The issue reports that returning to a running Control UI chat shows only committed history and hides the in-progress assistant response, tool activity, or running state.

## Best Possible Solution

Land a narrow Control UI adoption of the existing `chat.history.inFlightRun` payload, then track active tool-call replay separately only if the landed fix still leaves visible running state missing.

## Reproduction Assessment

Yes. The source path is high-confidence: `chat.history` can return `inFlightRun` and TUI consumes it, but Control UI's history result and loader do not, and rendering depends on `chatStream` being non-null.

## Solution Assessment

Yes. Reusing the existing gateway/TUI `inFlightRun` contract in Control UI is the narrow maintainable fix; a new backend protocol or reconnect subsystem would be broader than needed unless tool snapshots remain missing after that lands.
