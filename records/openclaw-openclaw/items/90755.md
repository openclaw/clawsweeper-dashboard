## Summary

Keep open: current main has the gateway-side in-flight run snapshot, but the Control UI history loader does not consume it, so switching back to a running chat can still lose the visible partial assistant response in the web UI.

## What This Changes

The issue asks for reconnecting Control UI chats to show an in-progress agent response, tool activity, or at least a running-session indicator after navigating away and back.

## Best Possible Solution

Adopt the existing chat.history inFlightRun contract in the Control UI, rehydrate chatRunId/chatStream on session reload, and add focused UI tests for buffered text and empty-text running-state recovery; extend the same contract for active tool-call snapshots only if existing session.tool delivery cannot cover them.

## Reproduction Assessment

Yes, we have a high-confidence source reproduction path: chat.history can return inFlightRun, the TUI consumes it, but the Control UI history result type and loader ignore it. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

Yes, the best narrow fix is Control UI parity with the existing gateway/TUI inFlightRun contract, not a new reconnect subsystem. If active tool-call replay remains missing after that, extend the same history payload rather than adding a parallel path.
