## Summary

Keep open: current main and the latest release still expose `inFlightRun` from gateway history and consume it in the TUI, but the Control UI history loader still does not adopt that payload; the exact Control UI candidate PRs are closed unmerged, so the reported behavior remains actionable.

## What This Changes

The issue reports that returning to a running Control UI chat session hides the in-progress assistant response and asks for restored partial output or a running indicator.

## Best Possible Solution

Land a narrow Control UI adoption of the existing gateway `chat.history`/`chat.startup` `inFlightRun` payload, mirroring TUI behavior and leaving broader active-run hello recovery or tool-call replay to separate tracked work.

## Reproduction Assessment

Yes, at source level: gateway history can return `inFlightRun` and TUI consumes it, but Control UI `loadChatHistory` currently ignores that field and can clear stream state. I did not run a live browser reproduction because this review is read-only.

## Solution Assessment

Yes. Reusing the existing gateway/TUI `inFlightRun` contract in the Control UI loader is the narrow maintainable fix; adding a new backend protocol or broad reconnect subsystem would be larger than needed for the reported assistant-output gap.
