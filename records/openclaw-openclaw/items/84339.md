## Summary

Keep open: this is a protected maintainer PR, and the diff has a blocking cache-key issue. The proposed memo can reuse an installed plugin index across different plugin discovery inputs, which conflicts with current plugin metadata compatibility rules.

## What This Changes

The PR adds a process-scoped memo around `loadInstalledPluginIndex`, bypasses it for explicit rebuild inputs, and invalidates it from installed-index refresh/write paths.

## Best Possible Solution

Keep the perf win only after the memo is keyed by the same resolved discovery/control-plane inputs used by plugin metadata snapshots, with regression coverage for load paths, env roots, and freshness invalidation.

## Reproduction Assessment

Yes for the review finding from source inspection: current main uses `plugins.load.paths` and env-resolved roots as discovery inputs, while the PR memo key can collide without them. I did not run tests because this review is read-only.

## Solution Assessment

No as written: memoizing the installed-index layer can be acceptable, but the key must match the existing discovery/control-plane compatibility boundary or it will return stale/wrong plugin indexes.

- **[P1] Include plugin load paths in the memo key:** `src/plugins/installed-plugin-index.ts:124-128`

  - confidence: 0.9

A manifest edit or external install-record update in the same process can keep serving stale plugin metadata until one of this process's write paths invalidates the memo.
  - confidence: 0.82
