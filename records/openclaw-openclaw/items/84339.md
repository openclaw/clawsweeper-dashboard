## Summary

Keep open: the startup perf target is real, but this patch can return an installed plugin index for the wrong discovery scope because the memo key omits the canonical discovery/control-plane fingerprint. This is a maintainer-labeled member PR, so it should be repaired or explicitly owned by maintainers, not closed by cleanup.

## What This Changes

The PR adds a process-scoped single-entry memo to `loadInstalledPluginIndex`, invalidates it on refresh/write paths, and adds unit tests for memo reuse and explicit invalidation.

## Best Possible Solution

Keep the startup optimization only if cache eligibility/keying is scoped by the canonical plugin control-plane or discovery fingerprint, with regression coverage for load paths, env-resolved roots, and source freshness; otherwise fix the higher-layer `PluginMetadataSnapshot` reuse path instead.

## Reproduction Assessment

Yes for the review finding: source inspection shows current discovery depends on `plugins.load.paths` and env-derived roots, while the PR cache key does not include those canonical inputs. I did not run tests because this review is read-only.

## Solution Assessment

No: a lower-level process memo keyed this narrowly is not the best maintainable fix. The safer path is to key it by the existing control-plane/discovery fingerprint with freshness tests, or keep reuse at the explicit `PluginMetadataSnapshot` layer.
