## Summary

Keep open. Current main does not implement `neverReply`, and the PR still has blocking correctness gaps where several channel branches suppress replies before writing the pending history that the feature promises; the plugin-based listen-only discussion is useful but not a complete replacement for this PR as written.

## What This Changes

The PR adds a channel/account/default `neverReply` config and wires group-message suppression plus pending-history docs/tests across group-capable channels.

## Best Possible Solution

Decide the architecture first; if the core config path is accepted, repair it with one tested shared pending-history pattern per channel, otherwise land the generic plugin hook seam and keep listen-only behavior in an external ClawHub/plugin package.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: enable `neverReply` for Discord, LINE, Matrix, or Mattermost group traffic and the new branch returns before the documented pending-history write path is reached.

## Solution Assessment

No. The requested behavior is useful, but this PR currently repeats per-channel gates and diverges from the promised retention semantics; a shared helper pattern or a maintainer-approved plugin seam would be safer.
