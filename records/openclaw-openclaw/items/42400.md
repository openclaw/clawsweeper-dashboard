## Summary

Keep open: current main does not implement `neverReply`, and the current PR diff does not consistently preserve the promised pending-history context, so it needs repair and maintainer review rather than cleanup closure.

## What This Changes

The PR adds a `neverReply` channel config and SDK resolver intended to let group messages pass access control without invoking the LLM or sending replies.

## Best Possible Solution

Keep the PR open and either repair the core implementation so silence and pending-history semantics are consistent across channels, or make the plugin hook broadcast path sufficient and move listen-only behavior out of core.

## Reproduction Assessment

Yes. Source inspection of the PR diff shows several `neverReply` branches returning before existing pending-history helpers run, while current main has no `neverReply` implementation to make the PR obsolete.

## Solution Assessment

No. The feature direction is useful, but this patch is not yet the best implementation because the advertised context accumulation is inconsistent and the schema/docs expose semantics the channel handlers do not reliably honor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Record history before neverReply returns:** `extensions/discord/src/monitor/message-handler.preflight.ts:394-401`
  - body: This new branch returns before `historyEntry` is recorded into `params.guildHistories`, so Discord suppresses the reply but drops the group message instead of storing it as pending context. Move the check after the existing history entry is built and record it like the no-mention path; the same audit should cover the other channel early returns that log “stored for context.”.
  - confidence: 0.88
- **[P2] Store LINE messages before denying neverReply:** `extensions/line/src/bot-handlers.ts:320-323`
  - body: Returning `denied` here exits before the existing LINE group-history record path, so `channels.line.neverReply` loses the passive context the feature promises. Record the pending history entry with the same key/body fallback as the mention-skip path before returning.
  - confidence: 0.86
- **[P2] Use the Mattermost pending-history path:** `extensions/mattermost/src/mattermost/monitor.ts:1359-1366`
  - body: This check runs before Mattermost computes the route `historyKey` and before the local `recordPendingHistory` helper exists, so no suppressed group post reaches `channelHistories` despite the log claiming it was stored. Compute the route/history key first and call the existing helper before returning.
  - confidence: 0.87
- **[P2] Do not expose inert per-group neverReply config:** `src/config/zod-schema.providers-core.ts:121-136`
  - body: The schema accepts `neverReply` on Telegram group/topic entries, but `resolveNeverReply` only receives channel and account id, so group-specific values validate but cannot take effect. Either thread group scope through the resolver or remove these schema fields.
  - confidence: 0.83
