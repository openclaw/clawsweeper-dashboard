## Summary

Keep open. Current main does not implement the proposed `neverReply` surface, so the PR is not obsolete, but the latest head still has correctness bugs where several documented history-retaining channels return before writing pending history, and it still needs maintainer direction on whether this should be a cross-channel core config or a plugin/hook-based feature.

## What This Changes

The PR adds a `neverReply` config option, schema/docs/SDK metadata, tests, and per-channel group-message gates intended to suppress group replies while retaining passive pending history where supported.

## Best Possible Solution

Maintainers should choose the architecture first. If core `neverReply` is accepted, move the suppression into a shared, testable admission/history pattern, fix the channel-specific early returns, remove or implement ignored config fields, and add representative channel tests proving no LLM call, no outbound reply, DMs unaffected, and pending-history write/read/clear behavior. If the plugin path is preferred, land the missing generic hook seam and keep listen-only behavior in a plugin such as the linked `flowolforg/openclaw-listen-only` implementation.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction for the PR regressions: configure `neverReply: true` with `historyLimit > 0` on Discord, LINE, Matrix, Mattermost, or file-only Slack group traffic and the new branches suppress dispatch before the pending-history write path can run.

## Solution Assessment

No, not as currently written. The requested user outcome is valid, but the repeated per-channel early returns already drift across history semantics; a shared channel-turn admission or maintainer-approved plugin seam is a narrower long-term solution.
