## Summary

Keep open: the PR is useful and not implemented on current main, but the latest head introduces a functional regression for token-authenticated local document attachment links and is currently dirty/unmergeable.

## What This Changes

The PR hardens Control UI bootstrap/media auth, adds build provenance and checksum artifacts, surfaces Slack/channel health warnings, and updates docs, i18n, workflows, and tests.

## Best Possible Solution

Keep the PR open, preserve the hardening direction, restore an authenticated document-download path, rebase onto current main, and require exact-head changed-gate/CI proof before merge or split.

## Reproduction Assessment

Yes. In a token-authenticated Control UI, the availability/blob fetch path sends Authorization, but the rendered local document anchor navigates to the bare assistant-media URL, which the Gateway will reject without bearer, query-token, trusted-proxy, or Tailscale identity.

## Solution Assessment

No. The overall direction is maintainable, but the document attachment handoff is not the best implementation because it removes the only auth material available to normal anchor navigation in token-auth deployments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve auth for local document downloads:** `ui/src/ui/chat/grouped-render.ts:1224-1225`
  - body: Local document attachments now render a bare `/__openclaw__/assistant-media?source=...` href after the authenticated availability check. In token/password Control UI sessions, clicking that link performs a normal navigation without the Authorization header, and the Gateway only accepts bearer/query-token/trusted identity for this route, so the download 401s even though the UI just proved the file is available. Keep a safe authenticated handoff for document links before removing query-token URLs there.
  - confidence: 0.9
