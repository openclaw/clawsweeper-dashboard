## Summary

Keep open: latest PR head still has a security-sensitive regression in the local assistant document attachment path. The hardening direction is useful, but document links are converted to client-created blob URLs and no longer navigate through the Gateway assistant-media response that applies CSP, nosniff, and referrer headers.

## What This Changes

The PR changes Control UI bootstrap/media auth, build provenance and checksum CI artifacts, Slack/channel health diagnostics, docs, i18n, workflow wiring, and regression tests.

## Best Possible Solution

Keep the hardening direction, but leave non-media documents on the authenticated Gateway media route or force safe downloads, then rerun exact-head CI.

## Reproduction Assessment

Not applicable as a user bug report. The blocking finding has a high-confidence static reproduction path: inspect the latest PR head document-link rendering and compare it with current main's Gateway assistant-media security headers.

## Solution Assessment

No, not as currently implemented. Authorization-backed blob fetches are reasonable for validated media previews, but document links should preserve server-side response protections or become forced downloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep document links on the Gateway media route:** `ui/src/ui/chat/grouped-render.ts:1213`
  - body: Local document attachments now link to a client-created `blob:` URL. That skips the Gateway assistant-media response path that applies CSP, `X-Content-Type-Options: nosniff`, and `Referrer-Policy`, so local HTML/SVG/PDF attachments under an allowed preview root can be opened without those response protections. Keep non-media links on the authenticated route or force a safe download, and reserve blob URLs for validated media previews.
  - confidence: 0.88
