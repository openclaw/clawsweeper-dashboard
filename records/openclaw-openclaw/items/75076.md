## Summary

Keep open: the PR still has a blocking security regression because local document attachments are rendered as client-created blob links, bypassing the Gateway assistant-media response protections that current main applies.

## What This Changes

This PR hardens Control UI bootstrap/media auth and CSP, adds Control UI build provenance/checksum artifacts, surfaces Slack/channel health readback warnings, updates docs/i18n, and adds browser/unit coverage.

## Best Possible Solution

Keep the hardening direction, but route non-media documents through authenticated Gateway media responses or force downloads before merge.

## Reproduction Assessment

Yes for the review finding: render or inspect a local non-image assistant attachment on PR head and the link href becomes blob: instead of /__openclaw__/assistant-media. That statically demonstrates the Gateway response headers are bypassed.

## Solution Assessment

No, not as currently implemented. Authorization-backed blob fetches are reasonable for validated media previews, but document links should preserve server-side response protections or become forced downloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep document links on the Gateway media route:** `ui/src/ui/chat/grouped-render.ts:1213`
  - body: Local document attachments now link to a client-created `blob:` URL. That skips the Gateway assistant-media response path that applies CSP, `X-Content-Type-Options: nosniff`, and `Referrer-Policy`, so local HTML/SVG/PDF attachments under an allowed preview root can be opened without those response protections. Keep non-media links on the authenticated route or force a safe download.
