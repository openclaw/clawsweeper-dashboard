## Summary

Keep open. Current main still treats `/steer` from Telegram native commands as the existing subagent-control command that requires a target, while current docs define main-session active-run steering through queue modes rather than a bare `/steer` shorthand; adding that shorthand is a command-contract/product change because subagent targets can be labels as well as numeric ids.

## What This Changes

The issue requests that `/steer <message>` from Telegram steer the active main run instead of being parsed as `/steer <subagent-target> <message>`.

## Best Possible Solution

Make an explicit maintainer decision for the `/steer` grammar, then either document the current subagent-only behavior more clearly or implement an unambiguous main-session shorthand with focused parser, Telegram native-command, queue, and docs coverage.

## Reproduction Assessment

Yes for the reported parser behavior: current source shows Telegram native `/steer check progress` is rebuilt as `/steer check progress`, then the shared command dispatcher treats `check` as a subagent target. It is not a strict bug reproduction against a documented contract because current docs define `/steer` as subagent-only.

## Solution Assessment

Unclear. A shorthand may be useful, but the proposed direction is not automatically the best fix because current subagent steering accepts non-numeric labels and prefixes; maintainers should choose the grammar before implementation.
