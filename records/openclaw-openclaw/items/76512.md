## Summary

Keep open. Current main still routes Telegram native `/steer ...` through the shared subagent command path that requires a target, while two open PRs already reference this issue with closing syntax, so cleanup should wait for a reviewed/merged fix or an explicit maintainer decision on the command grammar.

## What This Changes

The issue requests that Telegram `/steer <message>` deliver guidance to the active agent run instead of treating the first word as a required subagent id.

## Best Possible Solution

Align text/native `/steer` with the Control UI optional-target behavior using explicit ambiguity rules, then land one linked PR with parser, Telegram/text-channel, docs, changelog, and regression-test coverage.

## Reproduction Assessment

Yes for the reported parser behavior from source inspection: Telegram native `/steer check progress` is dispatched as `/steer check progress`, and the shared handler treats `check` as the subagent target. I did not run a live Telegram session, and the remaining question is the intended cross-surface command contract.

## Solution Assessment

Unclear until maintainers choose the grammar. Optional-id `/steer` matches the Control UI, but the implementation should preserve label/run-id subagent targeting and avoid silently changing valid current commands.
