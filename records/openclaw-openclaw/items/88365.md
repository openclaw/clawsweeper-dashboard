## Summary

Keep open: the PR is a narrow fix for the WSL clipboard path, but it is not merge-ready because the new `bash -lc` bridge invokes login-shell startup files while the tokenized dashboard URL is on stdin. I found no broader churn, but this shell choice needs a small fix before maintainer landing.

## What This Changes

The PR adds a WSL-only clipboard backend that pipes clipboard text through bash to Windows `clip.exe` and adds a regression test that keeps the token URL out of argv.

## Best Possible Solution

Use a non-login, non-profile shell bridge for WSL clipboard copies, keep the token URL on stdin, preserve existing non-WSL fallbacks, and add live WSL2 readback proof when a runner is available.

## Reproduction Assessment

Yes, by source inspection and the linked report: current main uses argv-based process spawning for `clip.exe`, while the dashboard token URL is supplied on stdin. I did not get live WSL2 readback proof because the PR body reports the WSL2 runner allocation failed.

## Solution Assessment

No, not as written: the WSL shell bridge is the right shape, but it should not use a login shell for a token-bearing stdin path. A plain non-login shell command is the safer narrow fix.

- **[P2] Use a non-login shell for WSL clipboard copies:** `src/infra/clipboard.ts:4`
  - body: The new `bash -lc` wrapper makes bash a login shell; bash reads `/etc/profile` and the user's profile before running the command.
