## Summary

Keep open. The PR addresses a real restart-continuation problem, but it is not merge-ready because the per-request deferral timeout can still be lost when an explicit restart coalesces with an existing pending or preparing restart, and the branch still needs the shared timeout constant and required changelog entry.

## What This Changes

The PR forces restart-sentinel continuation turns to use automatic source reply delivery, adds a 30-second explicit restart deferral timeout to gateway-tool and chat restart paths, and extends restart continuation and deferral tests.

## Best Possible Solution

Revise the branch so restart-sentinel continuations explicitly request automatic source delivery, explicit restart timeout policy is exported from the restart module, pending/preparing restart state stores and updates the effective timeout across coalescing, and focused tests cover both per-request timeout and coalesced restart ordering before adding the changelog entry.

## Reproduction Assessment

Yes. Current main can be checked by dispatching a restart continuation from a Telegram-style group/topic session and observing that no automatic source delivery mode is requested; the PR-specific timeout gap is reproduced by scheduling a no-timeout restart under gateway.reload.deferralTimeoutMs=0, then coalescing an explicit restart with deferralTimeoutMs and watching the old pending wait remain indefinite.

## Solution Assessment

No as submitted. The automatic source-delivery override is the right narrow fix, but the timeout must be stored in restart scheduler state and updated when restart requests coalesce instead of being captured only from the first scheduled call.
