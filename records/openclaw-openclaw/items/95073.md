## Summary

Keep open: current main and the latest release still build an invalid Silent quick-create cron job, but two open closing PRs already target this exact workflow, so the issue should wait for one fix PR to land or be rejected.

## What This Changes

The issue asks for the Control UI Cron quick-create dialog to create and close successfully when the Silent delivery preset is selected.

## Best Possible Solution

Land one focused PR that maps the Silent quick-create preset to a backend-accepted cron job shape with regression coverage, then close this issue.

## Reproduction Assessment

Yes, source-reproducible: the Silent quick-create path produces main plus agentTurn, addCronJob sends that to cron.add, and the cron service rejects that combination. I did not run a live Control UI gateway flow in this read-only pass.

## Solution Assessment

Yes, the clean fix is in the quick-create mapping, not in loosening backend validation. Mapping Silent to an isolated agentTurn with delivery none preserves the user’s agent prompt while matching the existing accepted cron shape.
