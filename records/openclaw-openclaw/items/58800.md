## Summary

Keep open: the linked Slack identity regression is still present on current main, but this PR head is not ready because it passes the wrong value through one edit path, lacks missing-scope fallback parity, and has no real Slack proof for the claimed visible behavior.

## What This Changes

The PR threads Slack sender identity through message edit, draft-stream, preview-finalize, and status-final update paths and adds focused regression tests.

## Best Possible Solution

Land or replace this with a Slack-plugin-only fix that preserves visible agent identity across edit helper, draft edit, preview finalization, and status-final updates, with missing-scope fallback parity and real Slack proof.

## Reproduction Assessment

Yes, at source level. Configure a Slack route with a custom agent identity and trigger a draft/preview/status edit path; current main resolves identity for sends but omits it from the edit helper and key edit callers.

## Solution Assessment

No. Threading identity through the Slack plugin is the right boundary, but this revision passes the wrong value in one path, lacks send-path fallback parity, and relies on `chat.update` authorship fields without Slack-visible proof.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Pass the resolved Slack identity to preview finalization:** `extensions/slack/src/monitor/message-handler/dispatch.ts:582`
  - body: This passes `finalThreadTs` as `identity`, so `editSlackMessage` receives a timestamp string instead of the resolved Slack identity object. The normal preview-finalize edit still lacks `username` and `icon_*`, leaving this path with the same visible identity regression the PR is meant to fix.
  - confidence: 0.94
- **[P2] Retry edits without custom identity on missing scope:** `extensions/slack/src/actions.ts:223-239`
  - body: When custom identity is present, these new `chat.update` calls can fail for workspaces missing `chat:write.customize`, while current sends catch that condition and retry without identity. Without the same fallback, previously successful edits can become failed edits instead of degrading to the bot default identity.
  - confidence: 0.89
- **[P2] Use a Slack-supported authorship strategy for edits:** `extensions/slack/src/actions.ts:223-239`
  - body: The Slack SDK contract exposes `username` and `icon_*` for `chat.postMessage`, but not for `chat.update`, and the linked bug says update calls drop those fields. Payload-only unit tests do not prove the edited Slack message keeps the agent identity; add credible live/API proof for this method or switch to a supported strategy.
  - confidence: 0.82
