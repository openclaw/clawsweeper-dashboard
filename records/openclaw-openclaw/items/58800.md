## Summary

Keep open. The linked Slack identity regression is still source-reproducible, but this PR revision is not mergeable: it drops identity on the preview-finalize path by passing a thread timestamp, lacks send/edit missing-scope fallback parity, and relies on `chat.update` authorship fields that the Slack SDK contract does not expose.

## What This Changes

The PR threads Slack custom sender identity through Slack message edit, draft-stream, preview-finalize, and status-final update paths and adds focused regression tests.

## Best Possible Solution

Land a Slack-plugin-only fix that uses a Slack-supported strategy to keep visible agent identity consistent across draft edits, preview finalization, and status-final updates, while preserving missing-scope fallback behavior and current edit-text safeguards.

## Reproduction Assessment

Yes, at source level. Configure a Slack custom agent identity and trigger a draft/preview edit path: current main resolves identity for sends but the edit helper, draft edit branch, and preview finalizer do not forward it; I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

No. Threading identity through the Slack plugin is the right boundary, but this revision passes the wrong value in one path, misses the established missing-scope fallback, and uses `chat.update` authorship fields without Slack SDK/API proof that they fix the visible behavior.

- **[P1] Pass the resolved Slack identity to preview finalization:** `extensions/slack/src/monitor/message-handler/dispatch.ts:582`
  - body: This passes `finalThreadTs` as `identity`, so `editSlackMessage` receives a thread timestamp string instead of the resolved Slack identity object.   - confidence: 0.94

  - body: When custom identity is present, these new `chat.update` calls can fail for workspaces missing `chat:write.customize`, while current sends catch that exact condition and retry without identity. Without the same fallback, previously successful edits can become failed edits instead of degrading to the bot default identity.
  - confidence: 0.89
- **[P2] Use a Slack-supported authorship strategy for edits:** `extensions/slack/src/actions.ts:217-239`
  - body: The current Slack SDK contract exposes `username` and `icon_*` for `chat.postMessage`, but not for `chat.update`, and the linked bug says update calls drop those fields. Payload-only unit tests do not prove the edited Slack message keeps the agent identity; either add credible live/API proof for this method or switch to a supported strategy.
  - confidence: 0.82
