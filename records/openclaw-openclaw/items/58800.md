## Summary

Keep open: the linked Slack identity regression is still real on current main, but this PR is not merge-ready because the normal preview-finalization path passes a thread timestamp as the identity and the core approach relies on unproven Slack `chat.update` identity fields with only mocked test proof.

## What This Changes

The PR adds optional Slack identity forwarding to Slack edit helpers, draft-stream and preview-finalization call sites, plus focused Slack regression tests.

## Best Possible Solution

Repair or replace this with a current-main Slack fix that uses a supported Slack behavior, passes the resolved Slack identity through every edited-message path, preserves graceful no-identity fallback, and includes real Slack proof.

## Reproduction Assessment

Yes, at source level: current main resolves Slack agent identity for sends but edit helpers and preview finalization do not carry that identity into `chat.update`. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

No. The fix boundary is plausible, but the submitted implementation is not the best solution until it fixes the wrong `identity` value, proves Slack supports the edit identity behavior, and preserves the existing missing-scope fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Pass the resolved Slack identity into final preview edits:** `extensions/slack/src/monitor/message-handler/dispatch.ts:582`
  - body: The final preview path sets `identity: finalThreadTs`, but `finalThreadTs` is a Slack thread timestamp, not the `{ username, iconUrl, iconEmoji }` object. This means the normal draft-final edit still cannot forward the configured agent identity, so the main reported path remains broken.
  - confidence: 0.96
- **[P1] Do not ship chat.update identity fields without Slack proof:** `extensions/slack/src/actions.ts:221-223`
  - body: `editSlackMessage` now sends `username`, `icon_url`, and `icon_emoji` to `chat.update`, but Slack's current `chat.update` reference does not list those identity arguments, while customized authorship is documented for `chat.postMessage`. Without live Slack proof or a supported alternate delivery strategy, this can merge a payload-only no-op for the reported bug. ([docs.slack.dev](https://docs.slack.dev/reference/methods/chat.update))
  - confidence: 0.8
- **[P2] Retry edits without identity when customize scope is missing:** `extensions/slack/src/actions.ts:223`
  - body: The send path already retries without custom identity on missing `chat:write.customize`, but this edit path calls `chat.update` with identity fields and returns or throws without that fallback. Workspaces that can currently edit successfully without the optional scope could start seeing draft or status edits fail.
  - confidence: 0.84
