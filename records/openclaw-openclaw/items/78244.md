## Summary

Keep open: the PR targets a real `doctor --fix` warning loop, but the current patch leaves `messages.groupChat.visibleReplies` unset when the message tool is unavailable, and current warning code still treats that unset configured-channel policy as default `message_tool`. The external PR also still lacks after-fix real `doctor --fix` then `doctor` output.

## What This Changes

The PR adds message-tool policy checks to the doctor legacy config normalizer so it skips writing `messages.groupChat.visibleReplies: "message_tool"` when agent tool policies do not allow the message tool.

## Best Possible Solution

Share the message-tool availability logic with the warning path, persist `messages.groupChat.visibleReplies: "automatic"` when tool-only delivery is unavailable, keep `"message_tool"` when available, add focused normalizer regression coverage, and include redacted real `doctor --fix` followed by `doctor` output.

## Reproduction Assessment

Yes, source-level. Current main writes `message_tool` during doctor repair, and current warning code warns for an unset configured-channel default when the message tool is unavailable; I did not run mutating doctor commands in this read-only review.

## Solution Assessment

No. Reusing the policy check is the right direction, but leaving the value unset does not change the default/warning contract; the narrower maintainable fix is to persist `automatic` or update repair and warning semantics together.
