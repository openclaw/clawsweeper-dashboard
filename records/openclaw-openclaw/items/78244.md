## Summary

Keep open: this PR targets a real doctor/config repair bug, but the current patch still leaves the visible-reply value unset when the message tool is unavailable, so the warning path can continue to warn on the default `message_tool` policy. It also lacks after-fix real behavior proof from the contributor's setup.

## What This Changes

The PR adds message-tool policy checks to `normalizeMissingGroupVisibleRepliesDefault` so doctor repair skips writing `messages.groupChat.visibleReplies: "message_tool"` when any configured agent policy lacks the message tool.

## Best Possible Solution

Share or mirror the warning path’s message-tool availability logic, persist `messages.groupChat.visibleReplies: "automatic"` when tool-only delivery is unavailable, keep `"message_tool"` when available, add focused normalizer regression coverage, and include redacted real doctor output.

Yes, source-level: current main writes `message_tool` during doctor repair, while the warning helper treats an unset configured-channel policy as default `message_tool` and warns when `message` is unavailable. I did not run the mutating CLI workflow in this read-only review.

## Solution Assessment

No. Policy-aware gating is the right direction, but returning without writing anything leaves the same default/warning mismatch; persisting `automatic` is the narrower maintainable repair.
