## Summary

Keep open: this is an active external PR for a still-open linked bug, but the patch does not fully resolve the doctor repair/warning loop on current main and lacks after-fix real behavior proof.

## What This Changes

The PR adds a message-tool availability check to the doctor legacy config normalizer before writing `messages.groupChat.visibleReplies: "message_tool"`.

## Best Possible Solution

Share one message-tool availability helper between doctor repair and warnings, persist `automatic` when `message_tool` would remain unavailable, and cover the repair path with focused regression tests plus real command proof.

## Reproduction Assessment

Yes, at source level. Current main repairs a configured-channel config with no visible-reply setting to `message_tool`, while the warning path treats default or explicit `message_tool` as warnable when policy blocks the message tool.

## Solution Assessment

No. The direction is right, but this patch leaves the value unset in unavailable cases and duplicates an older availability check that no longer matches current warning semantics.
