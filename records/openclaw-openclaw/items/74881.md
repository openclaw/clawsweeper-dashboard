## Summary

Keep open. Current main still has the native Windows Gmail helper spawn gap, and this PR targets a real bug from the linked report, but the latest diff is not merge-ready because it misses the CLI `gog` path, can still direct-spawn a resolved `.cmd`, executes a bare `where.exe` during lookup, expands deprecated SDK surface without baseline/docs follow-through, and lacks the required changelog entry.

## What This Changes

The PR adds a Windows-only command resolver for `gcloud`, `gog`, and `tailscale`, applies it to parts of Gmail setup and the gateway Gmail watcher, adds executable-path tests, and re-exports `resolveSystemBin` from the deprecated plugin SDK `infra-runtime` barrel.

Land a focused Windows Gmail helper fix that resolves `gcloud`, `gog`, and `tailscale` with a non-executing PATH/PATHEXT resolver, covers both `gmail-setup-utils` and `gmail-ops` plus the gateway watcher, routes `.cmd`/`.bat` launches through a Windows-safe process seam, avoids unrelated SDK exports, and records the user-facing fix in the changelog.

Yes. The linked bug has concrete native PowerShell reproduction, current main still shows the implicated bare `gcloud`, `gog`, and `tailscale` invocations, and the remaining PR gaps are statically reproducible from the documented CLI routing and PR diff.

## Solution Assessment

No. The resolver direction is reasonable, but this implementation is not the best fix until it covers the CLI `gog` path, avoids direct `.cmd` spawn, replaces bare `where.exe`, drops or fully baselines the SDK export, and adds changelog coverage.

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete Windows command-resolution concern by executing `where.exe` by bare name during helper lookup.

Concerns:
