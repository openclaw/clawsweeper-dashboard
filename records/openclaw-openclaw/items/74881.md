## Summary

Keep open. The PR targets a valid native-Windows Gmail setup blocker, but the latest head is not merge-ready: it misses the documented CLI `gog` path, can still directly spawn a resolved `.cmd`, executes `where.exe` by bare name during lookup, adds an unrelated SDK export without baseline follow-through, and lacks the required user-facing changelog entry.

## What This Changes

The branch adds a Windows-only `resolveExecutable` helper, applies it to `gcloud` and `tailscale` in Gmail setup utilities and `gog` in the gateway watcher, adds resolver tests, and exports `resolveSystemBin` from the deprecated plugin SDK `infra-runtime` barrel.

## Best Possible Solution

Land a focused Windows Gmail helper fix that resolves `gcloud`, `gog`, and `tailscale` through a non-executing PATH/PATHEXT resolver, routes any `.cmd`/`.bat` launch through the existing Windows-safe process seam, covers setup/run/gateway watcher paths with tests, avoids unrelated SDK surface expansion, and adds the required changelog entry.

## Reproduction Assessment

Yes. The linked bug has concrete native PowerShell reproduction, current main still shows matching bare helper invocations, and the PR-specific gaps are statically reproducible from the latest diff plus the documented CLI routing.

## Solution Assessment

No. The resolver direction is reasonable, but this implementation is not the best fix because it misses the CLI `gog` path, bypasses the Windows-safe spawn wrapper for long-running `gog`, executes bare `where.exe`, changes public SDK surface without follow-through, and omits changelog coverage.
