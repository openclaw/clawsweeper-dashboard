## Summary

Keep open: the PR targets a valid native Windows Gmail setup crash, but the latest diff is not merge-ready because it leaves the CLI `gog` path unresolved, can directly spawn a resolved `.cmd`, adds a bare `where.exe` lookup security concern, and includes an unrelated invalid SDK re-export plus no changelog entry.

## What This Changes

The PR adds a Windows-only executable resolver for `gcloud`, `tailscale`, and gateway `gog` Gmail helper calls, adds resolver tests, and re-exports `resolveSystemBin` from the deprecated plugin SDK infra-runtime barrel.

## Best Possible Solution

Land a focused Windows Gmail helper fix that uses a non-executing PATH/PATHEXT resolver, covers setup/run/gateway `gcloud`, `gog`, and `tailscale` paths, preserves the shared Windows-safe process seam, removes the SDK exposure, and adds the changelog entry.

## Reproduction Assessment

Yes. The linked issue gives a concrete native Windows PowerShell reproduction, and current main still has the source path that spawns bare `gcloud`, `tailscale`, and `gog`; I did not run a live Windows repro in this read-only review.

## Solution Assessment

No. The resolver direction is reasonable, but this implementation is not the best fix until it covers the CLI `gmail-ops` path, avoids direct `.cmd` spawn and bare `where.exe`, removes the unrelated SDK export, and records the user-facing fix in the changelog.

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete Windows command-resolution concern by executing bare `where.exe` during helper lookup.

Concerns:

- **[medium] Bare where.exe lookup adds executable trust:** `src/infra/executable-path.ts:143`
  - body: `resolveExecutable` calls `execFileSync("where.exe", ...)`, so the resolver executes a separately PATH-resolved binary before the intended helper command. Use a pinned System32 path or a non-executing filesystem PATH/PATHEXT lookup to avoid a shadowed `where.exe` influencing setup.
