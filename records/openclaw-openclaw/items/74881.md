## Summary

Keep open: the linked Windows Gmail setup bug is valid and the PR is aimed at the right area, but the latest diff is not merge-ready because it leaves the documented CLI `gog` path uncovered, can still directly spawn a resolved `.cmd`, adds a bare `where.exe` lookup trust issue, and includes an invalid SDK re-export.

## What This Changes

The PR adds a Windows-only executable resolver for `gcloud`, `gog`, and `tailscale` in Gmail setup/watcher code, adds resolver tests, and re-exports `resolveSystemBin` from the deprecated plugin SDK infra-runtime barrel.

## Best Possible Solution

Land a focused Windows Gmail helper fix that uses a non-executing PATH/PATHEXT resolver, covers setup/run/gateway `gcloud`, `gog`, and `tailscale` paths, uses the shared Windows-safe process seam for `.cmd`/`.bat`, removes unrelated SDK exposure, and adds the required changelog entry.

## Reproduction Assessment

Yes. The linked bug gives a concrete native Windows PowerShell reproduction, and current main source still has bare helper invocations in the documented Gmail setup/run path; I did not run a Windows live repro in this read-only review.

## Solution Assessment

No. The resolver direction is reasonable, but this implementation is not the best fix until it covers the CLI `gmail-ops` path, avoids direct `.cmd` spawn and bare `where.exe`, removes the invalid SDK export, and records the user-facing fix in the changelog.
