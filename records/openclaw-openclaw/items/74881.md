## Summary

Keep open. The linked Windows Gmail setup crash is real and this PR targets the right area, but the latest diff is not merge-ready because it only partially covers the Gmail paths, can still direct-spawn resolved `.cmd` shims, executes bare `where.exe` during lookup, adds an invalid/unscoped SDK export, and lacks the required changelog entry.

## What This Changes

This PR adds a Windows-only executable resolver, applies it to some `gcloud`, `gog`, and `tailscale` Gmail helper calls, adds resolver tests, and re-exports `resolveSystemBin` from the deprecated plugin SDK `infra-runtime` barrel.

## Best Possible Solution

Land a focused Windows Gmail helper fix that resolves `gcloud`, `gog`, and `tailscale` with a non-executing PATH/PATHEXT resolver, covers setup/run/gateway watcher paths, routes `.cmd`/`.bat` launches through Windows-safe process seams, avoids unrelated SDK exports, and records the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. The linked bug gives a concrete native PowerShell reproduction, and current main still has the implicated bare helper invocations; the PR-specific gaps are statically reproducible from the diff and documented CLI routing.

## Solution Assessment

No. The resolver direction is reasonable, but this implementation is not the best fix until it covers the CLI `gog` path, avoids direct `.cmd` spawn and bare `where.exe`, removes or fully baselines the SDK export, and adds changelog coverage.
