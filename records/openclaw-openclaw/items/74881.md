## Summary

Keep open. The PR targets a real native-Windows Gmail setup failure, but the latest provided diff is not merge-ready because it only resolves part of the helper surface, leaves a direct `.cmd` spawn path, adds a bare `where.exe` execution step, expands a deprecated SDK barrel without the required API baseline/docs follow-through, and lacks the required changelog entry.

## What This Changes

The PR adds a Windows-only executable resolver for `gcloud`, `tailscale`, and `gog`, wires it into Gmail setup utilities and the gateway Gmail watcher, adds resolver tests, and exports `resolveSystemBin` from the deprecated plugin SDK `infra-runtime` barrel.

## Best Possible Solution

Land a focused Windows Gmail helper fix that uses a non-executing PATH/PATHEXT resolver or the existing executable-path helpers, sends any resolved `.cmd`/`.bat` through the Windows-safe process wrapper, covers both CLI setup/run and gateway watcher paths, avoids unrelated SDK surface expansion unless the baseline/docs are intentionally updated, and adds the user-facing changelog entry.

## Reproduction Assessment

Yes. The linked issue has concrete native PowerShell reproduction, current main still has bare `gcloud`, `tailscale`, and `gog` helper invocations, and the PR-specific gaps are statically reproducible from the latest provided diff plus the Node child_process contract.

## Solution Assessment

No. The resolver direction is reasonable, but this implementation is not the best fix because it misses the CLI `gog` path, bypasses Windows-safe spawn for long-running `gog`, executes bare `where.exe`, changes public SDK surface without follow-through, and omits changelog coverage.
