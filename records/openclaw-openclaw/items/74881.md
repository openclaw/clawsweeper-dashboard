## Summary

Keep open: the linked Windows Gmail setup bug is valid on current main, but this PR is not merge-ready because it leaves the documented CLI `gog` path unresolved, can direct-spawn a resolved `.cmd`, executes bare `where.exe` during lookup, adds an invalid unrelated SDK export, and lacks the required changelog entry.

## What This Changes

This PR adds a Windows-only `gcloud`/`gog`/`tailscale` executable resolver, applies it to some Gmail setup and gateway watcher calls, adds resolver tests, and re-exports `resolveSystemBin` from the deprecated SDK infra-runtime barrel.

## Best Possible Solution

Land a focused Gmail Windows helper fix that covers setup/run/gateway paths, uses the existing non-executing PATH/PATHEXT resolver or shared Windows spawn seam, keeps the SDK surface unchanged, and records the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. The linked bug gives concrete native PowerShell steps, and current main still routes the documented Gmail setup/run flow through bare helper command spawns that match the reported `ENOENT`.

## Solution Assessment

No. Resolving helper commands is the right direction, but this implementation needs to cover `gmail-ops`, avoid bare `where.exe`, route resolved `.cmd` files through the Windows-safe wrapper, remove the SDK export, and add changelog coverage.
