## Summary

The issue is valid on current main and already has two active closing PRs, so it should remain open until maintainers choose and land one complete integration-boundary fix; current evidence favors keeping evaluation pure and rejecting fixes that either log in the evaluator or expose an unwired callback.

## What This Changes

Surface empty `allOf` and `anyOf` tool-availability groups as actionable malformed-descriptor diagnostics instead of only hiding the affected tool.

## Best Possible Solution

Keep `evaluateToolAvailability` side-effect free, validate descriptors at the real registration or planner-integration owner boundary, warn once only for malformed `unsupported-signal` diagnostics, and retain normal hidden-plan behavior for runtime unavailability.

## Reproduction Assessment

Yes. On current main, a descriptor with `{ allOf: [] }` or `{ anyOf: [] }` deterministically receives an `unsupported-signal` diagnostic and is placed in `ToolPlan.hidden`, with no observable notification channel.

## Solution Assessment

No, neither open PR currently demonstrates the complete best fix: evaluator logging violates the pure boundary, while an unwired planner callback does not produce the requested startup diagnostic. The fix should be completed at the actual descriptor registration or planner integration boundary.
