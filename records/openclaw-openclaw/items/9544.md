## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` for the bundled WhatsApp plugin, and the merged hardening only covers `creds.json` durability and auth-read barriers, not a production database-backed signal-key/auth store.

## What This Changes

Request replacing the WhatsApp plugin's Baileys multi-file auth persistence with a production auth-state store, preferably SQL-backed with caching for signal keys.

## Best Possible Solution

Implement a WhatsApp-plugin-owned production auth state with safe migration from existing Baileys files, strict credential/key permissions, backup or rollback handling, logout cleanup, docs/config alignment, focused regression tests, and security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high confidence because source, docs, tests, upstream source, and related PR state all show Baileys multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, and caching semantics before implementation.
