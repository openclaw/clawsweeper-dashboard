- `P2`: This is a normal-priority WhatsApp auth-storage improvement with meaningful durability and security impact, but it is not a confirmed urgent regression blocking all users today.

- `impact:security`: The affected surface stores WhatsApp credentials and signal keys, and the proposed change alters sensitive auth persistence.
- `impact:session-state`: The issue concerns WhatsApp linked-session state, key material, migration, and reconnect stability.

## Summary

Keep open: current main and the latest release still route WhatsApp auth through Baileys `useMultiFileAuthState`, while the merged hardening only made `creds.json` writes safer and the consolidation PR was closed unmerged.

## What This Changes

Request to replace the bundled WhatsApp plugin's Baileys multi-file JSON auth state with a production SQL or consolidated signal-key auth store with caching.

## Best Possible Solution

Keep this open for a maintainer-owned WhatsApp plugin auth-store design that replaces Baileys multi-file key persistence, safely migrates existing auth dirs, preserves status/logout/doctor behavior, documents the storage contract, and adds fresh-install, migration, corruption-recovery, and key-persistence coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/design request. Source inspection gives high confidence that current main still uses the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

The direction is sound, but the exact storage design needs maintainer judgment. The prior closed PR is useful prior art, while the safer solution needs a migration-aware plugin-owned auth store instead of treating the existing `creds.json` atomic-write hardening as complete.
