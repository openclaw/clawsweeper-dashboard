## Summary

Keep open: current main and release v2026.6.5 still use Baileys’ production-discouraged multi-file authentication store. This MEMBER-authored, security-sensitive migration needs explicit maintainer agreement on SQLite ownership, schema, migration, and recovery behavior.

## What This Changes

Replace Baileys’ multi-file WhatsApp authentication state with a production database-backed store and safely migrate existing credentials and signal keys.

## Best Possible Solution

Define one plugin-owned SQLite auth schema in the canonical OpenClaw state database, add a transactional and idempotent migration from every shipped Baileys auth file, verify fresh pairing, restart, reconnect, logout, and corruption recovery, then remove steady-state file fallbacks.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.5 directly use `useMultiFileAuthState`, whose upstream implementation stores signal keys as individual JSON files. The claimed corruption and CPU/I/O severity still needs representative live measurement.

## Solution Assessment

The database-backed direction is appropriate, but the proposed generic key/value shape is not yet the complete best fix because OpenClaw must define the canonical SQLite owner, typed schema, transactional migration, recovery semantics, and security posture.
