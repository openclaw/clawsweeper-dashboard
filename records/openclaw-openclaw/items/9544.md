## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` for WhatsApp auth, while the merged hardening only stabilizes `creds.json` writes and the prior consolidated-auth PR was closed unmerged.

## What This Changes

Request replacing WhatsApp Baileys multi-file JSON auth persistence with a database-backed, cached production auth state.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved logout/status/docs behavior, focused fresh-install and migration tests, and explicit security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high-confidence because source, docs, dependency source, and related PR history all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, caching behavior, and credential security semantics before implementation.
