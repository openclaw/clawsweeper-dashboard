## Summary

Keep open: current main still calls Baileys `useMultiFileAuthState` for the bundled WhatsApp plugin, while the merged hardening at https://github.com/openclaw/openclaw/pull/67815 explicitly kept the existing auth model and the related consolidated auth-state PR at https://github.com/openclaw/openclaw/pull/66863 was closed unmerged.

## What This Changes

Request to replace the bundled WhatsApp plugin's Baileys multi-file auth state with a durable SQL-backed, cached auth store for credentials and signal keys.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state implementation with safe migration from existing Baileys files, preserved logout/status/docs behavior, focused fresh-install and migration tests, and explicit security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high-confidence because source, docs, upstream Baileys source, and related PR history show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, caching behavior, and credential security semantics before implementation.
