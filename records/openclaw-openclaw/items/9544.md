## Summary

Keep open: current `main` still uses Baileys `useMultiFileAuthState` for the bundled WhatsApp plugin, and the merged auth hardening only stabilizes the existing file-based model rather than replacing it with a durable consolidated auth store.

## What This Changes

Request replacing the bundled WhatsApp plugin's Baileys multi-file auth state with a production-oriented database-backed auth store with signal-key caching and per-account isolation.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state implementation with safe migration from existing Baileys files, preserved setup/login/logout/status/docs behavior, focused fresh-install and migration tests, and explicit security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high-confidence because source, docs, dependency guidance, and related PR history show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right ownership boundary, but maintainers should choose the backend, migration/recovery model, caching behavior, and credential security semantics before implementation.
