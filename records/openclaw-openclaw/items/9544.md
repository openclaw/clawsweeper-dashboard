## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` in the bundled WhatsApp plugin, while the shipped hardening only makes `creds.json` writes safer and the related replacement attempt was closed unmerged with review concerns.

## What This Changes

Request replacing the WhatsApp plugin's Baileys multi-file auth state with a production database-backed or consolidated credential and signal-key store.

## Best Possible Solution

A WhatsApp-owned auth-state implementation persists credentials and signal keys in a durable store, safely migrates existing Baileys files, preserves logout/status/docs behavior, and has focused fresh-install, migration, recovery, and key-persistence tests plus security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high-confidence because source, docs, upstream Baileys source, and the closed related PR show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, and caching semantics before implementation.
