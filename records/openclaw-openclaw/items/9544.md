## Summary

Keep open: current main and the latest release still use Baileys `useMultiFileAuthState` for WhatsApp auth, and the merged hardening only stabilizes `creds.json` writes and auth-state reads. The unmerged prior attempt at https://github.com/openclaw/openclaw/pull/66863 is useful context, but the remaining work is a security-sensitive durable credential and signal-key store design.

## What This Changes

The issue asks the WhatsApp plugin to replace Baileys `useMultiFileAuthState` with a durable SQL-style auth store and cached signal-key layer.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, documented storage semantics, focused tests, live migration proof, and explicit security review.

## Reproduction Assessment

Not applicable: this is a feature and security-hardening request rather than a broken existing OpenClaw contract. Current-state verification is high-confidence because current main, `v2026.5.12`, docs, config, tests, related PR history, and upstream Baileys source all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, cache behavior, and credential security semantics before implementation.
