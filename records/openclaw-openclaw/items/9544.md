## Summary

Keep open: current main and the latest release still route WhatsApp through Baileys `useMultiFileAuthState`, and the earlier replacement PR was closed unmerged after review concerns. The issue is also member-authored and already marked as needing maintainer/product/security review, so this should not be auto-closed.

## What This Changes

The issue requests replacing Baileys multi-file WhatsApp auth state with a production database-backed auth store for credentials and signal keys.

## Best Possible Solution

Design a WhatsApp plugin-owned durable SQLite auth store, migrate existing Baileys files through doctor/setup-safe migration, keep runtime on one canonical path, and update login/status/logout/docs/tests for fresh install, upgrade, corruption recovery, and signal-key persistence.

## Reproduction Assessment

Yes for the implementation gap: current main still imports and calls `useMultiFileAuthState(authDir)` and uses Baileys `state.keys`. No live corruption, IO-growth, or security-risk reproduction was attempted in this read-only review.

## Solution Assessment

Unclear as written. Replacing the store in the WhatsApp plugin is the right boundary, but OpenClaw policy points toward a SQLite/doctor-migration design rather than a broad SQL/Postgres option without maintainer approval.
