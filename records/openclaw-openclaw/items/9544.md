## Label Justifications

- `P2`: This is a normal-priority WhatsApp auth-storage improvement with durability and security impact, but it is not a confirmed urgent regression blocking all users today.
- `impact:data-loss`: The request is about avoiding corrupted or lost WhatsApp credential and signal-key state currently stored across Baileys JSON files.
- `impact:security`: The affected surface stores WhatsApp credentials and signal keys, and the proposed change alters sensitive auth persistence.
- `impact:session-state`: The issue concerns WhatsApp linked-session state, key material, migration, and reconnect stability.

## Summary

Keep open: current main and the latest release still route WhatsApp auth through Baileys `useMultiFileAuthState`; the shipped work hardens `creds.json` writes but does not replace multi-file signal-key persistence, and the only consolidation PR was closed unmerged after review found regressions.

## What This Changes

The issue requests replacing Baileys multi-file WhatsApp auth persistence with a production-grade database or consolidated auth store.

## Best Possible Solution

Keep this open for a maintainer-owned WhatsApp plugin auth-store design that replaces Baileys multi-file key persistence, safely migrates existing auth dirs, preserves status/logout/doctor behavior, documents the storage contract, and adds fresh-install, migration, corruption-recovery, and key-persistence coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/design request. Source inspection gives high confidence that current main still uses the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

The direction is sound, but the exact storage design needs maintainer judgment. The safer path is a migration-aware plugin-owned auth store, not treating the existing `creds.json` atomic-write hardening as completion of this request.
