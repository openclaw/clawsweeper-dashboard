## Summary

Keep open: current main and the latest release still use Baileys multi-file auth for WhatsApp, and the shipped hardening only makes `creds.json` writes safer rather than replacing credential and signal-key persistence.

## What This Changes

Request to replace WhatsApp's Baileys `useMultiFileAuthState` multi-file JSON auth persistence with a durable SQL/database-backed auth store and signal-key cache.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, documented storage semantics, focused tests, live migration proof, and explicit security review.

## Reproduction Assessment

Not applicable: this is a feature and security-hardening request rather than a broken existing OpenClaw contract. Current-state verification is high-confidence because current main, `v2026.5.12`, related PR history, docs/config, and upstream Baileys source all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, cache behavior, and credential security semantics before implementation.
