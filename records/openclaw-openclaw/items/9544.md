## Summary

Current main and the latest release still use Baileys’ multi-file authentication state; existing hardening only makes `creds.json` writes and recovery safer, so the requested database-backed signal-key store and migration remain necessary. Because this security-sensitive item was opened by a repository member and has protected maintainer signals, it needs explicit maintainer design review rather than automated implementation.

## What This Changes

Replace Baileys multi-file WhatsApp authentication persistence with a production database-backed auth state and safe migration path.

## Best Possible Solution

Implement one canonical SQLite-backed auth store inside the bundled WhatsApp plugin, perform a verified one-time migration from existing Baileys files, and then make runtime, logout, identity lookup, status checks, and cleanup use only the database without a steady-state file fallback.

## Reproduction Assessment

Yes at source level: current main directly initializes WhatsApp with Baileys `useMultiFileAuthState`, and the upstream helper documents its production limitation. This read-only review did not independently reproduce a particular corruption or CPU/IO incident.

## Solution Assessment

Yes on the database-backed direction, but not yet on a complete implementation design. The narrow maintainable solution is plugin-owned SQLite with a single verified migration owner, not another consolidated JSON file or a permanent fallback stack.
