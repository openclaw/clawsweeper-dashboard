## Summary

Keep open. Current main still uses Baileys `useMultiFileAuthState` for the bundled WhatsApp plugin, while the related implementation PR #66863 was closed unmerged and the shipped hardening remains scoped to `creds.json` atomic writes and backup recovery.

## What This Changes

Request replacing Baileys multi-file WhatsApp auth persistence with a production database-backed and cached auth store for credentials and signal keys.

## Best Possible Solution

Implement a WhatsApp-plugin-owned production auth state with safe migration from existing Baileys files, credential and signal-key permissions, backup/rollback handling, logout cleanup, docs/config alignment, focused regression tests, and security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high confidence because source, docs, tests, upstream source, and related PR state all show Baileys multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, and caching semantics before implementation.
