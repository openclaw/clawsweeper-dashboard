## Summary

Keep open: current main and the latest release still build the WhatsApp socket with Baileys `useMultiFileAuthState`, so the requested production auth-store replacement has not landed. The issue is valid source-level work, but it needs maintainer design and security review before implementation.

## What This Changes

The issue requests replacing the bundled WhatsApp plugin’s Baileys multi-file auth-state storage with a production database-backed or consolidated signal-key auth store.

## Best Possible Solution

Build a WhatsApp-plugin-owned production auth-store seam, migrate existing Baileys files safely, preserve status/login/logout semantics, update public docs/config contracts, and prove fresh install, upgrade, corruption recovery, and signal-key persistence before closing.

## Reproduction Assessment

Yes, at source level: current main still calls `useMultiFileAuthState(authDir)` in the WhatsApp socket creation path, and the latest release tag has the same behavior. I did not run a live WhatsApp account because the missing implementation is directly visible in source and tests.

## Solution Assessment

Unclear whether the exact SQL/Postgres shape is the best solution; the maintainable direction is a reviewed WhatsApp-plugin auth-store abstraction with a safe migration path. Maintainers still need to choose the storage backend, compatibility contract, and security posture.
