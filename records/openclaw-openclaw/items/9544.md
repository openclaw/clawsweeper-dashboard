## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` for the bundled WhatsApp plugin, and the merged hardening only stabilizes `creds.json` rather than replacing the multi-file signal-key store. The closed replacement attempt at https://github.com/openclaw/openclaw/pull/66863 is useful prior art but was not merged and had correctness concerns.

## What This Changes

Request replacing WhatsApp's Baileys multi-file auth state with a durable SQL or consolidated signal-key store plus caching.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, docs updates, focused fresh-install and migration tests, and explicit security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high-confidence because current source, docs, dependency source, and related PR history all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, caching behavior, and credential security semantics before implementation.
