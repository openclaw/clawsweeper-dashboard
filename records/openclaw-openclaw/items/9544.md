## Label Justifications

- `P2`: This is a normal-priority WhatsApp auth-storage improvement with durability and security impact, but it is not a confirmed urgent regression blocking all users today.
- `impact:data-loss`: The request targets corrupted or lost WhatsApp credential and Signal-key state currently stored across Baileys JSON files.
- `impact:security`: The affected surface stores WhatsApp credentials and Signal keys, and changing it alters sensitive auth persistence.
- `impact:session-state`: The issue concerns linked WhatsApp session state, key material, migration, and reconnect stability.

## Summary

Keep open: current main and v2026.5.18 still build the WhatsApp socket on Baileys useMultiFileAuthState, with only creds.json atomic-write hardening around part of the auth state. The prior consolidated auth-state PR was closed unmerged after review concerns, and the newer discussion adds concrete durability/security failure modes rather than making the request obsolete.

## What This Changes

Replace the bundled WhatsApp plugin's Baileys multi-file auth state with a production-grade database-backed auth store for credentials and signal keys.

## Best Possible Solution

Keep this open for a maintainer-owned WhatsApp plugin auth-store design that safely migrates existing Baileys credentials and signal keys, preserves status/login/logout/doctor behavior, updates docs, and proves fresh install, migration, corruption recovery, key persistence, and large-store performance.

## Reproduction Assessment

No, not as a strict bug reproduction; this is a feature/design request. Source inspection gives high confidence that current main and v2026.5.18 still use the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

Yes on direction, but unclear on exact design. The maintainable path is a migration-aware WhatsApp plugin-owned auth store; treating creds.json atomic-write hardening or the closed prior PR as completion would leave signal-key persistence and migration risks unresolved.
