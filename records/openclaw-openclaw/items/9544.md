## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` for WhatsApp auth, and the merged work so far hardens `creds.json` reads/writes without replacing the multi-file signal-key store. The closed implementation attempt at https://github.com/openclaw/openclaw/pull/66863 is useful prior art, but it was not merged.

## What This Changes

The issue requests replacing Baileys multi-file WhatsApp auth persistence with a durable SQL or consolidated auth-state store plus signal-key caching.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, documented storage semantics, focused tests, and explicit security review.

## Reproduction Assessment

Not applicable: this is a feature and security-hardening request rather than a broken existing contract. Current-state verification is high-confidence because source, docs, dependency source, and related PR history all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, caching behavior, and credential security semantics before implementation.
