## Summary

Keep open: current main and v2026.5.12 still route WhatsApp auth through Baileys `useMultiFileAuthState`; the merged hardening covers `creds.json` durability but does not replace multi-file signal-key persistence. The related consolidation PR was closed unmerged, and the merged auth-stabilization PR explicitly scoped itself away from replacing the Baileys auth store.

## What This Changes

The issue requests replacing the bundled WhatsApp plugin's Baileys `useMultiFileAuthState` persistence with a production database-backed auth state for credentials and signal keys.

## Best Possible Solution

Keep this open for a maintainer-owned WhatsApp plugin auth-store design that replaces Baileys multi-file key persistence, safely migrates existing auth directories, preserves status/logout/doctor behavior, documents the storage contract, and covers fresh install, migration, corruption recovery, and key persistence.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/design request. Source inspection gives high confidence that current main still uses the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

The direction is sound, but the exact storage design needs maintainer judgment. The safer path is a migration-aware plugin-owned auth store, not treating the existing `creds.json` atomic-write hardening as completion of this request.
