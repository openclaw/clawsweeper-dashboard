## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` for WhatsApp auth, and the merged hardening so far only stabilizes `creds.json` and auth-read behavior rather than replacing the multi-file signal-key store.

## What This Changes

The issue requests replacing WhatsApp's Baileys multi-file JSON auth state with a durable SQL-style auth store, caching, and safe per-WhatsApp-ID key retrieval.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, docs/config updates, focused tests, and explicit security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request rather than a broken existing contract. Current-state verification is high-confidence because source, docs, dependency source, and related PR history all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, caching behavior, and credential security semantics before implementation.
