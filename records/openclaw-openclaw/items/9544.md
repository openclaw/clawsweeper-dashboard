- `P2`: This is a normal-priority WhatsApp auth-storage improvement with durability and security impact, but it is not a confirmed urgent regression blocking the whole runtime today.
- `impact:data-loss`: The issue targets corrupted or lost WhatsApp credential and Signal-key state currently persisted across Baileys JSON files.

## Summary

Keep open: the current WhatsApp plugin still uses Baileys multi-file auth state for Signal-key persistence, and the issue discussion identifies durability, performance, and sensitive-key risks that are only partly mitigated by the merged creds.json hardening. The request remains a design/security-sensitive auth-store migration rather than a cleanup item ClawSweeper should close.

## What This Changes

Replace the bundled WhatsApp plugin's Baileys useMultiFileAuthState persistence with a production-ready database or consolidated auth store for credentials and Signal keys.

## Best Possible Solution

Keep this open for a maintainer-owned WhatsApp plugin auth-store design that safely migrates existing Baileys credentials and Signal keys, preserves login/logout/status/doctor behavior, updates docs, and proves fresh install, migration, crash recovery, key persistence, and large-store performance.

## Reproduction Assessment

No, not as a strict bug reproduction; this is a feature/design request. Source inspection gives high confidence that current main still uses the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

Yes on direction, but unclear on exact design. The maintainable path is a migration-aware WhatsApp plugin-owned auth store; treating creds.json atomic-write hardening or the closed prior PR as completion would leave Signal-key persistence and migration risks unresolved.
