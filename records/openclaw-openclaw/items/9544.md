## Label Justifications

- `P2`: This is a normal-priority WhatsApp auth-storage improvement with durability and security impact, but it is not a confirmed urgent regression blocking all users today.
- `impact:data-loss`: The request targets corrupted or lost WhatsApp credentials and Signal-key state currently persisted across Baileys JSON files.
- `impact:security`: The affected store contains WhatsApp credentials and Signal keys, so replacing it changes sensitive auth persistence.
- `impact:session-state`: The issue concerns linked WhatsApp session state, key material, migration, and reconnect stability.

## Summary

Keep open: current main still creates the WhatsApp Baileys socket with `useMultiFileAuthState`, and the merged hardening since this report protects the existing credential-file model rather than replacing the multi-file Signal-key store with a transactional auth state.

## What This Changes

The issue requests replacing the bundled WhatsApp plugin's Baileys `useMultiFileAuthState` persistence with a SQL-backed, cacheable auth state for credentials and Signal keys.

## Best Possible Solution

Build a WhatsApp plugin-owned transactional auth store, likely with a local SQLite default, that safely migrates existing Baileys multi-file credentials and Signal keys while preserving account directories, login/status/logout behavior, docs, and recovery paths.

## Reproduction Assessment

No, not as a strict runtime bug reproduction; this is a feature/design request. Source inspection is high-confidence that current main and v2026.5.20 still use the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

Yes on direction, unclear on exact design. The maintainable path is a WhatsApp plugin-owned transactional auth store with safe migration and durability proof, not treating current `creds.json` hardening or the closed prior PR as complete.
