## Summary

Keep open: current main still calls Baileys `useMultiFileAuthState` for WhatsApp auth, and the shipped hardening only stabilizes the existing file-backed `creds.json` path rather than replacing credential and signal-key persistence.

## What This Changes

The issue requests replacing Baileys `useMultiFileAuthState` in the WhatsApp plugin with a durable SQL-style credential and signal-key store.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, documented storage semantics, focused tests, and explicit security review.

## Reproduction Assessment

Not applicable: this is a feature and security-hardening request rather than a broken existing OpenClaw contract. Current-state verification is high-confidence because source, docs, upstream Baileys source, latest-release runtime surface, and related PR history all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, caching behavior, and credential security semantics before implementation.

## Risks / Open Questions

- The backend choice, migration and rollback behavior, credential permissions, backup recovery model, and cache semantics are not settled.
- Replacing credential and signal-key persistence can regress fresh login, logout cleanup, status/link detection, existing `authDir` compatibility, and corrupted-auth recovery.
- The open startup-stall report at https://github.com/openclaw/openclaw/issues/78165 may share the same large Baileys auth-store cost, so the storage design should account for both corruption and bootstrap latency.
