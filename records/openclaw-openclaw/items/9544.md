## Summary

Keep open. Current main and the latest release still use Baileys `useMultiFileAuthState` for the WhatsApp plugin, while the shipped hardening only makes `creds.json` writes safer; the requested production credential and signal-key auth store has not landed.

## What This Changes

Request to replace the WhatsApp plugin's Baileys multi-file JSON credential and signal-key store with a production SQL-backed auth store plus caching and migration behavior.

## Best Possible Solution

Implement a plugin-owned production auth store for WhatsApp credentials and signal keys, with safe migration from existing Baileys files, updated docs/config semantics, and regression coverage for fresh install, migration, corruption recovery, and key persistence.

## Reproduction Assessment

Yes for the reported source condition: current main and `v2026.5.12` still import and call `useMultiFileAuthState` in the WhatsApp socket path. I did not establish a live corruption or performance failure path in this read-only review.

## Solution Assessment

Unclear as written because the requested SQL/Postgres shape needs maintainer storage and migration decisions. The maintainable direction is a plugin-owned production auth store with explicit compatibility and security review, not a small local shim.
