## Summary

Keep open. Current main and the latest release still use Baileys `useMultiFileAuthState` for WhatsApp credentials and signal keys, and this maintainer-authored, security-sensitive auth-state migration needs explicit maintainer direction rather than cleanup closure.

## What This Changes

The issue requests replacing the bundled WhatsApp plugin's Baileys multi-file JSON auth state with a production database-backed auth store for credentials and signal keys.

## Best Possible Solution

Land a plugin-owned SQLite-backed Baileys auth-state replacement with a doctor migration from existing multi-file credentials and signal keys, preserving account/status/logout behavior and adding security-reviewed migration tests and live WhatsApp proof.

## Reproduction Assessment

Yes for the implementation gap: current main still imports and calls `useMultiFileAuthState(authDir)`, and docs/config still model Baileys file auth. No live corruption, CPU, or security failure was reproduced in this read-only review.

## Solution Assessment

Yes, the requested direction is the right durable end state, but it needs maintainer-owned design because it changes sensitive persisted auth state. The closed consolidated-store PR is useful prior art but not the best solution as-is because review found fresh-install and key persistence regressions.
