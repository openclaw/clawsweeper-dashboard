## Summary

Keep open: current main and v2026.6.8 still use Baileys multi-file auth for WhatsApp credentials and signal keys, and this MEMBER-authored, security-sensitive persisted-auth migration needs maintainer direction rather than cleanup closure.

## What This Changes

Replace the bundled WhatsApp plugin's Baileys multi-file auth state with a production SQL/SQLite-backed auth store for credentials and signal keys.

## Best Possible Solution

Land a plugin-owned SQLite-backed Baileys auth-state replacement with a doctor migration from existing multi-file credentials and signal keys, preserving account/status/logout behavior and adding security-reviewed migration tests plus live WhatsApp proof.

## Reproduction Assessment

Yes for the source-level implementation gap: current main still imports and calls `useMultiFileAuthState(authDir)`, and docs/config still model Baileys file auth. No live corruption, CPU/IO, or security failure was reproduced in this read-only review.

## Solution Assessment

Yes for the requested direction, but the implementation needs maintainer-owned design because it changes sensitive persisted auth state. The closed consolidated-store PR is useful prior art but not the best solution as-is because review found fresh-install and key persistence regressions.
