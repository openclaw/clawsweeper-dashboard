## Label Justifications

- `P2`: This is a normal-priority WhatsApp auth-storage improvement with durability and security impact, but it is not a confirmed urgent regression blocking all users today.
- `impact:data-loss`: The request targets corrupted or lost WhatsApp credentials and Signal-key state currently persisted across Baileys JSON files.
- `impact:security`: The affected store contains WhatsApp credentials and Signal keys, so replacing it changes sensitive auth persistence.
- `impact:session-state`: The issue concerns linked WhatsApp session state, key material, migration, and reconnect stability.

## Summary

Keep open: current main and v2026.5.20 still build the WhatsApp socket on Baileys `useMultiFileAuthState`, while shipped hardening only protects `creds.json` writes and auth reads around the existing multi-file store. The earlier consolidated auth-state attempt at https://github.com/openclaw/openclaw/pull/66863 was closed unmerged, and https://github.com/openclaw/openclaw/pull/67815 explicitly scoped itself to hardening the current model rather than replacing it.

## What This Changes

Replace the WhatsApp plugin's Baileys multi-file auth persistence with a production-grade consolidated or database-backed auth store for credentials and Signal keys.

## Best Possible Solution

Build a migration-aware WhatsApp plugin auth store, likely with a transactional local SQLite default, that replaces Baileys credential and Signal-key file persistence while preserving existing account directories, login/status/logout behavior, docs, and recovery paths.

## Reproduction Assessment

No, not as a strict runtime bug reproduction; this is a feature/design request. Source inspection is high-confidence that current main and v2026.5.20 still use the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

Yes on direction, unclear on exact design. The maintainable path is a WhatsApp plugin-owned transactional auth store with safe migration and live durability proof, not treating current `creds.json` hardening or the closed prior PR as complete.
