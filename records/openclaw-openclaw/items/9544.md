## Label Justifications

- `P2`: This is a normal-priority WhatsApp auth-storage improvement with durability and security impact, but it is not a confirmed urgent regression blocking all users today.
- `impact:data-loss`: The request targets corrupted or lost WhatsApp credentials and Signal-key state currently persisted across Baileys JSON files.
- `impact:security`: The affected store contains WhatsApp credentials and Signal keys, so replacing it changes sensitive auth persistence.
- `impact:session-state`: The issue concerns linked WhatsApp session state, key material, migration, and reconnect stability.

## Summary

Keep open: current main and the latest release still create the WhatsApp socket through Baileys `useMultiFileAuthState`, so the requested production auth-store replacement remains unimplemented and needs maintainer design/security review.

## What This Changes

The issue requests replacing WhatsApp's Baileys multi-file auth state with a production database-backed or consolidated auth store for credentials and Signal keys.

## Best Possible Solution

Build a WhatsApp plugin-owned transactional auth store that safely migrates existing Baileys multi-file credentials and Signal keys while preserving login, status, logout, docs, and recovery behavior.

## Reproduction Assessment

Yes for the current-state claim: source inspection shows `createWaSocket` still calls `useMultiFileAuthState` on current main and the latest release. This is not a strict runtime bug reproduction; it is a feature/design request backed by source and upstream contract evidence.

## Solution Assessment

Yes on direction, unclear on exact design. The maintainable path is a WhatsApp plugin-owned transactional auth store with safe migration and durability proof, not treating current `creds.json` hardening or the closed prior PR as complete.
