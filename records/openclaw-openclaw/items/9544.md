## Summary

Keep open: current main and v2026.5.18 still create the WhatsApp socket with Baileys `useMultiFileAuthState`, and the shipped hardening only makes `creds.json` writes safer rather than replacing multi-file signal-key persistence. The prior replacement PR was closed unmerged after review concerns, so the storage-design request remains valid.

## What This Changes

Replace the bundled WhatsApp plugin's Baileys multi-file auth state with a production auth store for credentials and Signal keys.

## Best Possible Solution

Keep this open for a maintainer-owned WhatsApp plugin auth-store design that migrates existing Baileys credentials and Signal keys, preserves operational behavior, updates docs, and proves fresh install, migration, corruption recovery, and key persistence.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/design request. Source inspection gives high confidence that current main and v2026.5.18 still use the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

Unclear as an implementation choice: moving off Baileys multi-file auth is the right direction, but the backend, schema, migration, and rollback plan need maintainer product and security decisions. The prior closed PR is useful prior art but should not be copied without fixing the reviewed fresh-install and key-write durability gaps.
