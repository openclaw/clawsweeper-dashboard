- `impact:session-state`: The issue concerns linked WhatsApp session state, key material, migration, and reconnect stability.

## Summary

Keep open: current main and the latest release still build the WhatsApp socket through Baileys `useMultiFileAuthState`, and the only related replacement PR was closed unmerged with fresh-install and key-loss concerns.

## What This Changes

The issue requests replacing WhatsApp's Baileys multi-file JSON auth state with a production database-backed or transactional cached auth store for credentials and Signal keys.

## Best Possible Solution

Build a WhatsApp plugin-owned transactional auth store, likely with a local SQLite default, that safely migrates existing Baileys credential and Signal-key files while preserving login, status, logout, docs, and recovery behavior.

## Reproduction Assessment

No, not as a strict runtime bug reproduction; this is a feature/design request. Source inspection is high-confidence that current main and `v2026.5.22` still use the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

Yes on direction, unclear on exact design. The maintainable path is a WhatsApp plugin-owned transactional auth store with safe migration and durability proof, not treating current `creds.json` hardening or the closed prior PR as complete.
