## Summary

Keep open: current main and the latest release still use Baileys multi-file auth for WhatsApp, and the merged hardening only makes `creds.json` persistence safer rather than replacing credential and signal-key storage.

## What This Changes

The issue requests replacing the WhatsApp plugin's Baileys multi-file auth-state persistence with a production durable auth store.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, documented storage semantics, focused tests, live migration proof, and explicit security review.

## Reproduction Assessment

Not applicable: this is a feature and security-hardening request rather than a broken existing OpenClaw contract. Current-state verification is high-confidence because current main, `v2026.5.12`, related PR history, docs/config, and upstream Baileys guidance all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, cache behavior, and credential security semantics before implementation.
