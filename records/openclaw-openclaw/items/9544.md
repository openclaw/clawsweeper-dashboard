## Summary

Keep open. Current main and v2026.6.10 still use Baileys `useMultiFileAuthState` for WhatsApp auth, the related implementation PR closed unmerged, and this MEMBER-authored security/session-state migration needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests replacing the bundled WhatsApp plugin’s Baileys multi-file auth state with a production database-backed auth store for credentials and signal keys.

## Best Possible Solution

Land a plugin-owned SQLite-backed Baileys auth-state replacement with a security-reviewed doctor migration from existing multi-file credentials and signal keys, preserving account/status/logout behavior and adding focused migration plus live WhatsApp proof.

## Reproduction Assessment

Yes for the implementation gap: current main still imports and calls `useMultiFileAuthState(authDir)`, and docs/config still model Baileys file auth. No live corruption, CPU, or security failure was reproduced in this read-only review.

## Solution Assessment

Yes directionally, but not as a ready implementation. Replacing the store is the durable solution, while the exact SQLite schema, migration behavior, backup semantics, and security model need maintainer-owned design.
