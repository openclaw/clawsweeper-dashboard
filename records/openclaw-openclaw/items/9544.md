## Summary

Keep open: current main and the latest release still create the WhatsApp Baileys socket through `useMultiFileAuthState`, so the requested production auth-store replacement has not landed.

## What This Changes

The issue requests replacing the WhatsApp Baileys `useMultiFileAuthState` JSON-file auth store with a production-oriented database-backed auth state for credentials and Signal keys.

## Best Possible Solution

Build a WhatsApp plugin-owned transactional auth store that safely migrates existing Baileys multi-file credentials and Signal keys while preserving login, status, logout, docs, and recovery behavior.

## Reproduction Assessment

Yes for the current-state claim: source inspection shows `createWaSocket` still calls `useMultiFileAuthState` on current main and the latest release. This is not a strict runtime bug reproduction; it is a feature/design request backed by source and upstream contract evidence.

## Solution Assessment

Yes on direction, unclear on exact design. The maintainable path is a WhatsApp plugin-owned transactional auth store with safe migration and durability proof, not treating current `creds.json` hardening or the closed prior PR as complete.
