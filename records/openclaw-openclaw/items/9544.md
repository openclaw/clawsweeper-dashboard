## Summary

Keep open: current main and v2026.5.22 still create the WhatsApp socket through Baileys `useMultiFileAuthState`, so the requested production auth-store replacement has not landed.

## What This Changes

Request replacing the bundled WhatsApp plugin's Baileys multi-file auth state with a production database-backed or consolidated auth store.

## Best Possible Solution

Build a WhatsApp plugin-owned transactional auth store that safely migrates existing Baileys multi-file credentials and Signal keys while preserving login, status, logout, docs, and recovery behavior.

## Reproduction Assessment

Yes for the current-state claim: source inspection shows `createWaSocket` still calls `useMultiFileAuthState` on current main and the latest release. This is a feature/design request rather than a strict runtime bug reproduction.

## Solution Assessment

Yes on direction, unclear on exact design. The maintainable path is a WhatsApp plugin-owned transactional auth store with safe migration and durability proof, not treating current `creds.json` hardening or the closed prior PR as complete.
