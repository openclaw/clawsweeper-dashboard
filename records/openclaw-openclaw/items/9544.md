## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` for the bundled WhatsApp plugin, while shipped hardening only stabilizes `creds.json` writes and auth-read barriers. The related replacement attempt at https://github.com/openclaw/openclaw/pull/66863 was closed unmerged, and the newer https://github.com/openclaw/openclaw/issues/78165 tracks a narrower event-loop symptom rather than this durable auth-store request.

## What This Changes

Request to replace WhatsApp's Baileys multi-file auth state with a durable SQL-style auth store for credentials and signal keys.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, docs updates, focused fresh-install and migration tests, and explicit security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high-confidence because source, docs, dependency source, and related pull request history all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration and recovery model, caching behavior, and credential security semantics before implementation.
