## Summary

Keep open: current main and the latest release still build the bundled WhatsApp socket with Baileys `useMultiFileAuthState`, and the open issue is maintainer-authored with security/session-state labels, so it is not a conservative cleanup close candidate.

## What This Changes

Replace the bundled WhatsApp plugin's Baileys multi-file auth-state persistence with a production database-backed auth store for credentials and signal keys.

## Best Possible Solution

Build a WhatsApp-plugin-owned production auth-store seam, use the repository's database-first storage direction where practical, migrate existing Baileys files safely, and prove fresh install, upgrade, corruption recovery, logout, status, and signal-key persistence before closing.

## Reproduction Assessment

Yes, at source level: current main still calls `useMultiFileAuthState(authDir)` in the WhatsApp socket creation path, and the latest release tag has the same behavior. I did not run a live WhatsApp account because the missing implementation is directly visible in source and tests.

## Solution Assessment

Unclear whether the exact SQL/Postgres shape proposed in the issue is the best final design. The maintainable direction is a reviewed WhatsApp-plugin auth-store abstraction with a safe migration path, likely aligned with OpenClaw's database-first storage rules.
