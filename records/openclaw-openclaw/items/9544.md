## Summary

Keep open. Current main and the latest release still use Baileys `useMultiFileAuthState` for the bundled WhatsApp plugin; the existing hardening only makes `creds.json` writes more atomic and recoverable, not a SQL/database-backed signal-key auth store. The related implementation PR #66863 was closed unmerged, so there is no merged replacement to close this against.

## What This Changes

Request to replace the bundled WhatsApp plugin's Baileys multi-file JSON auth state with a production database-backed credential and signal-key store, with caching for the signal layer.

## Best Possible Solution

Keep this issue open until the bundled WhatsApp plugin replaces Baileys multi-file auth with a production-grade credential and signal-key store, safely migrates existing auth files, preserves logout/status/config behavior, updates WhatsApp docs/config help, and adds focused coverage for fresh install, migration, corruption recovery, and key persistence.

## Reproduction Assessment

Not applicable as a feature/security-hardening request. The current-state verification is high confidence: source, docs, schema comments, tests, and the latest release tag still show `useMultiFileAuthState` and Baileys multi-file auth files in use.

## Solution Assessment

Unclear for implementation details. Moving this into the WhatsApp plugin is the right boundary, but the database choice, migration format, credential permissions, caching, and recovery semantics need maintainer design before a replacement PR should be attempted.
