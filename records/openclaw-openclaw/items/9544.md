## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` for the bundled WhatsApp plugin, while the merged hardening only makes `creds.json` writes safer and does not add a production database-backed auth/signal-key store. The related implementation attempt at https://github.com/openclaw/openclaw/pull/66863 was closed unmerged after review concerns, so it does not supersede this issue.

## What This Changes

Request replacing the bundled WhatsApp plugin's Baileys multi-file auth persistence with a production database-backed auth and signal-key store with caching.

## Best Possible Solution

Implement the replacement inside the WhatsApp plugin with safe migration from existing Baileys files, strict credential/key permissions, rollback or recovery handling, logout/status/docs alignment, focused tests, and security review.

## Reproduction Assessment

Not applicable as a feature and security-hardening request. Current-state verification is high confidence because source, docs, upstream source, and related PR state all show Baileys multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, and caching semantics before implementation.
