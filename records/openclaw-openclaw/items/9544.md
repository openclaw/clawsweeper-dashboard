## Summary

Keep open. Current main and the latest release still use Baileys `useMultiFileAuthState` for WhatsApp auth and signal keys, the prior replacement PR is closed unmerged, and this member-authored security/session-state migration needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests replacing WhatsApp's Baileys multi-file JSON auth state with a SQL/SQLite-backed auth store and cached signal-key layer.

## Best Possible Solution

Land a plugin-owned SQLite-backed Baileys auth-state replacement with a security-reviewed doctor migration from existing multi-file credentials and signal keys, preserving account/status/logout behavior and adding live WhatsApp proof.

## Reproduction Assessment

Yes for the implementation gap: current main still imports and calls `useMultiFileAuthState(authDir)`, and docs/config still model Baileys file auth. No live corruption, CPU, or security failure was reproduced in this read-only review.

## Solution Assessment

Yes, the requested direction is the durable end state, but it is a sensitive persisted-auth migration rather than a narrow bug patch. The closed consolidated-store PR is useful prior art but not the best solution as-is because review identified fresh-install and key-persistence regressions.
