## Summary

Keep open. Current main and the latest release still use Baileys `useMultiFileAuthState` for WhatsApp auth, the only related implementation PR was closed unmerged, and this MEMBER-authored security/session migration needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests replacing Baileys multi-file WhatsApp auth state with a production SQL-backed/cached auth store for WhatsApp credentials and signal keys.

## Best Possible Solution

Land a plugin-owned SQLite-backed Baileys auth-state replacement with a security-reviewed doctor migration from existing multi-file credentials and signal keys, preserving account/status/logout behavior and adding focused migration and live WhatsApp proof.

## Reproduction Assessment

Yes for the implementation gap: current main still imports and calls `useMultiFileAuthState(authDir)`, and docs/config still model Baileys file auth. No live corruption, CPU, or security failure was reproduced in this read-only review.

## Solution Assessment

Yes directionally, but not as a ready implementation: replacing the store is the durable solution, while the exact SQLite schema, migration behavior, backup semantics, and security review need maintainer-owned design. The closed consolidated-store PR is useful prior art but not best as-is because review found fresh-install and key-persistence regressions.
