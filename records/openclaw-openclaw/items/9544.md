## Summary

Keep open: current main still uses Baileys `useMultiFileAuthState` for WhatsApp auth, and the merged hardening so far only stabilizes the existing file-backed `creds.json` path. The prior replacement attempt at https://github.com/openclaw/openclaw/pull/66863 was closed unmerged and had review concerns, so the requested durable signal-key auth store still needs maintainer design work.

## What This Changes

Replace the bundled WhatsApp plugin's Baileys multi-file auth state with a production-grade durable auth store, preferably database-backed, with signal-key caching.

## Best Possible Solution

Add a WhatsApp-owned durable auth-state store with safe migration from existing Baileys files, preserved login/status/logout behavior, documented storage semantics, focused tests, and explicit security review.

## Reproduction Assessment

Not applicable: this is a feature and security-hardening request rather than a broken existing contract. Current-state verification is high-confidence because source, docs, upstream source, and related PR history all show multi-file auth remains in use.

## Solution Assessment

Unclear for the exact SQL/Postgres shape. The WhatsApp plugin is the right boundary, but maintainers should choose the backend, migration/recovery model, caching behavior, and credential security semantics before implementation.
