## Summary

Keep open: current main and v2026.5.18 still build the WhatsApp socket on Baileys useMultiFileAuthState, and the shipped hardening only makes creds.json writes safer rather than replacing multi-file signal-key persistence. The prior replacement PR was closed unmerged after review concerns, so the storage-design request remains valid.

## What This Changes

Request to replace WhatsApp/Baileys multi-file auth persistence with a production database or consolidated auth store for credentials and Signal keys.

## Best Possible Solution

Keep this open for a maintainer-owned WhatsApp plugin auth-store design that migrates existing Baileys credentials and signal keys, preserves status/logout/doctor behavior, updates docs, and proves fresh install, migration, corruption recovery, and key persistence.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/design request. Source inspection gives high confidence that current main and v2026.5.18 still use the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

The direction is sound, but the exact storage design still needs maintainer judgment. The safer path is a migration-aware WhatsApp plugin-owned auth store, not treating creds.json atomic-write hardening as completion of this request.
