## Summary

Keep open: current main and release v2026.6.5 still use Baileys’ production-discouraged multi-file authentication store, while this maintainer-authored issue requires an explicit security and migration design decision rather than conservative cleanup.

## What This Changes

Replace the bundled WhatsApp plugin’s Baileys multi-file authentication state with a production database-backed store and safely migrate existing credentials and signal keys.

## Best Possible Solution

Implement one plugin-owned SQLite authentication store using the appropriate shared/plugin state database, transactionally persist credentials and signal keys, migrate and verify the existing Baileys files through the WhatsApp plugin’s doctor contract, then remove steady-state file fallback after successful migration.

## Reproduction Assessment

Yes at source level: current main unambiguously initializes WhatsApp through Baileys `useMultiFileAuthState`, whose upstream implementation stores signal keys in individual JSON files. This read-only review did not execute a live corruption or performance reproduction.

## Solution Assessment

Yes in direction, but the maintainable solution is a single plugin-owned SQLite path with transactional persistence and doctor migration, not selectable SQL backends, a consolidated JSON sidecar, or permanent file/database fallback logic.

## Risks / Open Questions

- Migrating existing credentials and signal keys incorrectly could log users out, corrupt linked-device state, or leave partially migrated sessions that cannot reconnect.
- The database schema, transaction boundaries, encryption and file-permission posture, multi-account isolation, and crash-recovery behavior need explicit security review before implementation.
- The shipped `authDir` setting and logout, identity-resolution, status, backup, and doctor behavior currently assume a directory of Baileys files, so the upgrade contract must be designed rather than silently replaced.
- The related event-loop report at https://github.com/openclaw/openclaw/issues/78165 may share this storage path, but a database migration should not claim to solve that separate performance symptom without live proof.
