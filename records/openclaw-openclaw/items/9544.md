- `P2`: This is a normal-priority WhatsApp auth-storage improvement with durability and security impact, but it is not a confirmed urgent regression blocking all users today.
- `impact:data-loss`: The request is about avoiding corrupted or lost WhatsApp credential and Signal-key state currently stored across Baileys JSON files.
- `impact:security`: The affected surface stores WhatsApp credentials and Signal keys, and the proposed change alters sensitive auth persistence.
- `impact:session-state`: The issue concerns WhatsApp linked-session state, key material, migration, and reconnect stability.

## Summary

Keep open: current main and the latest release still instantiate the WhatsApp socket with Baileys useMultiFileAuthState, while shipped hardening only makes creds.json safer and does not replace multi-file Signal-key persistence. The closed replacement PR and newer symlink-hardening PR are useful context, but neither supersedes the requested production auth-store design.

## What This Changes

Request to replace the bundled WhatsApp plugin's Baileys useMultiFileAuthState persistence with a production-grade SQL or consolidated auth store for credentials and Signal keys.

## Best Possible Solution

Adopt a WhatsApp plugin-owned, migration-aware auth store that persists credentials and Signal keys transactionally, preserves existing status/logout/doctor behavior, updates docs, and proves fresh install, migration, corruption recovery, and key persistence.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/design request. Source inspection gives high confidence that current main and v2026.5.18 still use the Baileys multi-file auth path the issue asks to replace.

## Solution Assessment

The direction is sound, but the exact storage design still needs maintainer security and migration judgment. The safer path is a migration-aware WhatsApp plugin-owned auth store, not treating creds.json atomic-write hardening as completion of this request.
