## Summary

Keep this PR open. The sandbox `env.vars` gap is real, but the patch is not ready because it bypasses runtime guards from `applyConfigEnvVars` and changes the sandbox exec environment contract without docs/security approval.

## What This Changes

The PR adds a `configEnvVars` exec default, threads config `env.vars` into sandbox env construction, and adds regression tests for config env propagation and precedence.

## Best Possible Solution

Define an approved effective config-env map for sandbox exec that preserves runtime guards, layers below explicit sandbox/backend and per-call env, and updates sandbox docs/tests with the admitted sources and precedence.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `env.vars` and run sandbox exec; current main builds sandbox env from defaults, sandbox env, and per-call env only, while host/gateway exec inherits config-applied `process.env`.

## Solution Assessment

No. Threading config env into sandbox exec is plausible, but this patch uses the collector directly instead of the guarded effective runtime env and does not document the changed sandbox contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards `collectConfigRuntimeEnvVars(options?.config)` directly into sandbox exec defaults. That helper only normalizes and filters names, while `applyConfigEnvVars` also skips keys already present in the runtime environment and values containing unresolved `${VAR}` references, so sandbox exec can receive stale config values or literal placeholders that host/gateway exec intentionally omits.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff expands config-level environment variables into sandbox command environments and needs explicit sandbox/security approval before merge.

Concerns:

- **[medium] Config secrets newly bypass sandbox env contract:** `src/agents/pi-tools.ts:408`
  - body: The patch forwards config `env.vars` into every sandbox exec environment, while current docs say sandbox exec does not inherit host `process.env` and Docker container creation separately filters configured sandbox env. Because exec-time env is serialized into Docker/backend exec specs, this changes which configured secrets reach sandboxed commands and should be approved and documented deliberately.
  - confidence: 0.9
