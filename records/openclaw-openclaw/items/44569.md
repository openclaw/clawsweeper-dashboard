## Summary

Keep open. Current main still lacks the proposed sandbox config-env propagation, but the PR remains blocked because it forwards raw collected config env into sandbox exec without the runtime non-override and unresolved-placeholder guards, and it changes the sandbox secret/env contract without maintainer security approval or real sandbox proof.

## What This Changes

The PR adds a configEnvVars exec default, passes collectConfigRuntimeEnvVars into createExecTool, layers those vars into buildSandboxEnv below sandbox/per-call env, and adds config-env propagation tests.

## Best Possible Solution

Define an approved sandbox config-env contract, derive the effective env with the same non-override and unresolved-placeholder guards used by runtime config env, document the precedence/security model, and prove it with focused regression coverage plus a real sandbox exec run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure env.vars and run sandbox exec; current main builds sandbox exec env from PATH/HOME, sandbox env, and per-call env only, while non-sandbox exec inherits the runtime process env after config loading.

## Solution Assessment

No. Threading collectConfigRuntimeEnvVars directly into sandbox exec is not the best fix because it bypasses applyConfigEnvVars guard parity and leaves the sandbox env policy undocumented; an approved effective-runtime-env helper or explicit sandbox contract would be safer.

## Review Findings
