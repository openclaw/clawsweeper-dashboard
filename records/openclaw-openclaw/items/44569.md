## Summary

Keep open. Current main still omits config `env.vars` from sandbox exec environments, but the PR changes a security-sensitive sandbox contract and its current wiring bypasses runtime env guard parity.

## What This Changes

The branch adds `configEnvVars` to exec defaults, layers it into `buildSandboxEnv`, wires it from `createOpenClawCodingTools`, and adds tests for config `env.vars` propagation into sandbox exec.

## Best Possible Solution

Approve and document the sandbox config-env contract first, then implement it through a shared effective-runtime-env helper that preserves non-override and unresolved-placeholder guards while keeping sandbox and per-call precedence explicit.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main's sandbox exec path builds env from defaults, sandbox env, and per-call env only, while the gateway path uses the process env after config env loading.

## Solution Assessment

No. Forwarding config env into sandbox exec is plausible, but this patch uses the raw collector instead of the guarded effective runtime env and leaves the sandbox env policy undocumented.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards `collectConfigRuntimeEnvVars(options?.config)` directly into sandbox exec defaults. That collector filters names only; it does not skip keys already present in the runtime env or unresolved `${VAR}` placeholders like `applyConfigEnvVars` does, so sandbox exec can now see stale config values or literal placeholders that gateway exec intentionally omits.
  - confidence: 0.93

## Security Review
