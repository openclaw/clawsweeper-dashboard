## Summary

Keep open. Current main still has the sandbox env gap, but this branch needs changes because it forwards collected config env entries without runtime guard parity and changes the sandbox secret contract without real behavior proof.

## What This Changes

The PR threads config env.vars into sandbox exec environment construction and adds propagation and precedence tests.

## Best Possible Solution

Define an approved sandbox config-env contract, derive the effective env with the same runtime guards, document the precedence/security model, and prove it with focused tests plus a redacted real sandbox exec run.

## Reproduction Assessment

Yes, at source level: configure env.vars and use sandbox exec; current main builds sandbox env from PATH/HOME, sandboxEnv, and per-call params only, while gateway exec inherits process env after config load. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. Passing collectConfigRuntimeEnvVars directly into sandbox exec is not the best fix because it loses applyConfigEnvVars guard parity and leaves the sandbox secret-forwarding contract undocumented.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards collectConfigRuntimeEnvVars(options?.config) directly into exec defaults. That collector filters key names only; unlike applyConfigEnvVars/createConfigRuntimeEnv, it does not skip existing runtime env values or unresolved ${VAR} placeholders, so sandbox exec can receive literal placeholders or stale config values that gateway exec intentionally omits.
  - confidence: 0.93
