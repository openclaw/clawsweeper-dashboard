## Summary

Keep this PR open, but it is not mergeable as-is. Source inspection confirms current main does not forward config env.vars into sandbox exec, while the PR forwards raw collected config values in a way that skips existing runtime guards, changes the documented sandbox env boundary, and still lacks real Docker sandbox proof.

## What This Changes

The PR threads config env.vars through exec defaults into buildSandboxEnv and adds focused env propagation tests for gateway and sandbox exec environments.

## Best Possible Solution

Define an approved sandbox config-env contract, derive forwarded values from the same guarded runtime semantics as gateway mode, update docs/tests for precedence and security behavior, and prove it with redacted Docker sandbox output.

## Reproduction Assessment

Yes at source level: current main's sandbox env builder only layers PATH/HOME, sandbox env, and per-call params env, so config env.vars are absent from sandbox exec. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. Forwarding collectConfigRuntimeEnvVars directly is not the narrowest maintainable solution because it skips applyConfigEnvVars guard parity and changes the documented sandbox env contract without proof or approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reuse the guarded runtime env contract:** `src/agents/pi-tools.ts:408`
  - body: This forwards collectConfigRuntimeEnvVars(options?.config) into sandbox exec defaults, but gateway mode applies env.vars through applyConfigEnvVars, which skips existing env keys and unresolved ${VAR} placeholders. Sandbox exec can now receive stale config values or literal placeholders that gateway exec intentionally avoids; derive the forwarded map from the same guarded runtime contract.
  - confidence: 0.92
