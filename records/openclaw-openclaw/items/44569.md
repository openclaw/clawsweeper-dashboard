## Summary

Keep open. The PR targets a real sandbox exec env propagation gap, but the patch is not merge-ready because it forwards raw collected config env vars instead of the runtime-effective env semantics and changes the sandbox secret exposure contract without matching docs or security approval.

## What This Changes

The PR adds a `configEnvVars` exec default, threads config `env.vars` into sandbox environment construction, and adds regression tests for sandbox and host env precedence.

## Best Possible Solution

Add an approved effective-config-env path for sandbox exec that mirrors `applyConfigEnvVars` precedence and placeholder guards, layers below sandbox/backend and per-call env, and updates sandbox documentation/tests to describe the intended secret propagation boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `env.vars` and run sandbox exec; current main builds sandbox env without config or process env, while gateway exec uses inherited process env after config loading.

## Solution Assessment

No. The intended fix direction is plausible, but directly forwarding `collectConfigRuntimeEnvVars()` is not the narrowest maintainable solution because it bypasses runtime env guards and changes sandbox secret exposure without a documented contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards `collectConfigRuntimeEnvVars(options?.config)` directly into sandbox exec defaults, but that helper only filters names. `applyConfigEnvVars` also preserves existing runtime env values and drops unresolved `${VAR}` placeholders, so sandbox exec can now receive stale config values or literal placeholders that host/gateway exec intentionally omits.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff expands config-level env values into sandbox exec environments and needs sandbox/security approval before merge.

Concerns:

- **[medium] Config secrets newly bypass sandbox env contract:** `src/agents/pi-tools.ts:408`
  - body: The patch forwards config `env.vars` into every sandbox exec environment, while current docs say sandbox exec does not inherit host `process.env` and Docker container creation separately filters configured sandbox env. Because exec-time env is serialized into docker/backend exec specs, this changes which configured secrets reach sandboxed commands.
  - confidence: 0.9
