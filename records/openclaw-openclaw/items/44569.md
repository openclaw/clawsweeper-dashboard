## Summary

Keep open: current main and the latest release still do not forward config `env.vars` into sandbox exec, but this PR changes the documented sandbox env isolation contract and still has a correctness finding, a security-contract concern, and missing real Docker proof.

## What This Changes

The PR adds a `configEnvVars` exec default, layers config `env.vars` into `buildSandboxEnv`, passes `collectConfigRuntimeEnvVars()` from `pi-tools`, and adds env propagation tests.

## Best Possible Solution

Define an explicit approved sandbox config-env contract, preserve gateway env guard parity, document precedence and security behavior, and prove it with focused tests plus redacted Docker sandbox output.

## Reproduction Assessment

Yes, at source level: current main builds sandbox exec env without config `env.vars`, while host/gateway exec can inherit config-applied process env. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. Directly forwarding `collectConfigRuntimeEnvVars()` is not the narrowest maintainable solution because it skips `applyConfigEnvVars` guard parity and changes the documented sandbox secret-exposure contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reuse runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards `collectConfigRuntimeEnvVars(options?.config)` into sandbox exec defaults, but that collector does not apply the runtime guards used by `applyConfigEnvVars` and `createConfigRuntimeEnv`: existing env keys win, and unresolved `${VAR}` placeholders are skipped. Sandbox exec can therefore receive stale config values or literal placeholders that gateway exec intentionally avoids, causing sandbox-only auth/config failures.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The PR expands config-level env values into sandbox command environments and needs explicit sandbox/security contract approval before merge.

Concerns:

- **[medium] Approve sandbox config env forwarding first:** `src/agents/pi-tools.ts:408`
  - body: The patch sends global config `env.vars` through sandbox exec environments even though current docs keep host/global env out of sandboxed runs. `collectConfigRuntimeEnvVars` allows credential-shaped keys, and this Docker exec path is separate from container-create env sanitization, so the change expands sandbox secret exposure unless the contract is approved, filtered, documented, and tested.
  - confidence: 0.9
