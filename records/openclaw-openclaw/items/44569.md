## Summary

Keep open: the source-level gap is real, but this patch currently bypasses runtime env guard parity, lacks contributor real Docker proof, and changes the sandbox/global-env security contract that current docs keep separate.

## What This Changes

The PR threads config `env.vars` into sandbox exec environments via `configEnvVars` and adds unit tests for propagation and precedence.

## Best Possible Solution

Keep explicit `agents.defaults.sandbox.docker.env` as the supported sandbox credential path unless maintainers intentionally extend the sandbox contract; if they do, derive values through guarded runtime env semantics, document precedence/exposure, and prove it with Docker.

## Reproduction Assessment

Yes at source level: current main's sandbox env builder only layers default PATH/HOME, configured sandbox env, and per-call params env, so config `env.vars` do not reach sandbox exec. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. Directly forwarding `collectConfigRuntimeEnvVars` is not the narrowest maintainable solution because it skips runtime guard parity and changes sandbox secret exposure without an approved contract or real proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reuse runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards `collectConfigRuntimeEnvVars(options?.config)` directly into sandbox exec defaults, but that collector does not apply the runtime guards used by `applyConfigEnvVars`: existing env keys win, and unresolved `${VAR}` placeholders are skipped. Sandbox exec can therefore receive stale config values or literal placeholders that gateway exec intentionally avoids; derive the sandbox config env from the same guarded runtime contract.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff expands global config env values into sandbox command environments and needs explicit sandbox/security contract approval before merge.

Concerns:

- **[medium] Approve sandbox config env forwarding first:** `src/agents/pi-tools.ts:408`
  - body: The patch sends global config `env.vars` through sandbox exec environments even though current docs keep host/global env out of sandboxed runs. `collectConfigRuntimeEnvVars` allows credential-shaped keys, so this expands sandbox secret exposure unless the contract is approved, filtered, documented, and tested.
  - confidence: 0.9
- **[medium] Do not rely on container-create env sanitization:** `src/agents/bash-tools.shared.ts:30`
  - body: The new config env layer reaches per-command Docker exec environments; the existing explicit Docker env handling applies during container creation and does not sanitize this new per-exec path. Credential-shaped config vars would bypass the PR body's claimed container-create sanitizer unless a separate per-exec policy is added.
  - confidence: 0.88
