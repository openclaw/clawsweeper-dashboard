## Summary

Keep this PR open. Current main still has the sandbox env propagation gap, but the proposed patch should not merge as-is because it forwards collected config values without the runtime guards used by host/gateway env loading and changes the sandbox secret-exposure contract that current docs explicitly describe differently.

## What This Changes

The PR adds a `configEnvVars` exec default, threads config `env.vars` into sandbox env construction, and adds regression tests for config env propagation and env-layer precedence.

## Best Possible Solution

Introduce an approved sandbox config-env path that computes an effective env map with the same precedence and unresolved-placeholder guards as `applyConfigEnvVars`, layers it below explicit sandbox/backend env and per-call env, and updates sandbox docs/tests to state exactly which configured env sources are admitted into sandbox exec.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `env.vars` and run sandbox exec; current main’s sandbox branch builds an env from defaults, sandbox env, and per-call env only, while host/gateway exec inherits `process.env` after config loading.

## Solution Assessment

No. Threading config env vars into sandbox exec is a plausible direction, but this patch forwards the raw collected config map instead of the effective guarded runtime env, so sandbox exec can receive stale config values or literal unresolved placeholders that host/gateway exec intentionally skips.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards `collectConfigRuntimeEnvVars(options?.config)` directly into sandbox exec defaults. That helper only normalizes and filters names, while `applyConfigEnvVars` also skips keys already present in the runtime env and values that still contain unresolved `${VAR}` references, so sandbox exec can receive stale config values or literal placeholders that host/gateway exec intentionally omits.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff expands config-level environment variables into sandbox command environments and needs explicit sandbox/security approval before merge.

Because exec-time env is serialized into docker/backend exec specs, this changes which configured secrets reach sandboxed commands and should be approved and documented deliberately.
