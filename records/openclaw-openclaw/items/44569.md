## Summary

Keep open: current main still omits config env.vars from sandbox exec environments, but this PR changes the sandbox env contract without guard parity, documentation, security approval, or real Docker sandbox proof.

## What This Changes

The PR adds a configEnvVars exec default, threads config env.vars into sandbox environment construction, and adds env propagation/precedence tests.

## Best Possible Solution

Define an explicit sandbox config-env contract, derive the effective env with applyConfigEnvVars/createConfigRuntimeEnv guard parity, document precedence and security behavior, and prove it with focused tests plus redacted real sandbox exec output.

## Reproduction Assessment

Yes, at source level: current main builds sandbox exec env without config env.vars, while host exec can inherit process.env after config load. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. Passing collectConfigRuntimeEnvVars directly into sandbox exec is not the narrowest maintainable fix because it misses runtime guard parity and changes sandbox secret exposure without an approved documented contract.
