## Summary

Keep open. Current main still has the underlying sandbox env gap, and this PR targets the right path, but the patch is not merge-ready because it forwards raw collected config env vars without the runtime guards used by host/gateway env handling and changes the sandbox secret-exposure contract.

## What This Changes

The PR adds a `configEnvVars` exec default, threads `collectConfigRuntimeEnvVars()` into sandbox exec environment construction, layers it between defaults and sandbox/per-call env, and adds regression tests for config env propagation.

## Best Possible Solution

Revise the change around an effective config-env helper that mirrors `applyConfigEnvVars` semantics, preserves existing runtime-env precedence, rejects unresolved placeholders, layers below sandbox/backend env and per-call env, and documents the approved sandbox env contract alongside the related admin-configured env work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `env.vars` and run sandbox exec; current mainâ€™s sandbox path builds a clean env without config/process env, while host/gateway exec inherits `process.env` after config loading.

## Solution Assessment

No. The direction is plausible, but directly forwarding `collectConfigRuntimeEnvVars()` is not the best fix because it skips runtime guards and changes sandbox secret exposure; a guarded effective-env path plus explicit sandbox docs/security approval is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards `collectConfigRuntimeEnvVars(options?.config)` directly into sandbox exec defaults. That helper only normalizes and filters names, while `applyConfigEnvVars` also skips keys already present in the runtime env and values that still contain unresolved `${VAR}` references, so sandbox exec can receive stale config values or literal placeholders that host/gateway exec intentionally omits.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff expands config-level environment variables into sandbox command environments and needs explicit sandbox/security approval before merge.

Concerns:

- **[medium] Config secrets newly bypass sandbox env contract:** `src/agents/pi-tools.ts:408`
  - body: The patch forwards config `env.vars` into every sandbox exec environment, while current docs say sandbox exec does not inherit host `process.env` and Docker container creation separately filters configured sandbox env. Because exec-time env is serialized into `docker exec`/backend exec specs, this changes which configured secrets reach sandboxed commands and should be approved and documented deliberately.
  - confidence: 0.9
