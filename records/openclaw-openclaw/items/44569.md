## Summary

Keep open. The PR targets a real source-level sandbox exec env gap, but the branch is conflicting, lacks contributor-owned Docker sandbox proof, and changes the sandbox env/secret contract while bypassing the runtime guards used by gateway env.vars handling.

## What This Changes

The branch adds configEnvVars to ExecToolDefaults/buildSandboxEnv, passes collectConfigRuntimeEnvVars from createOpenClawCodingTools, and adds env propagation tests.

## Best Possible Solution

Define an approved sandbox config-env contract, preserve gateway env guard parity, update docs/tests for precedence and security behavior, and prove it with redacted Docker sandbox output.

## Reproduction Assessment

Yes at source level: current main's sandbox env builder only layers PATH/HOME, sandbox env, and per-call params env, while gateway mode can inherit process.env after applyConfigEnvVars. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. Directly forwarding collectConfigRuntimeEnvVars is not the narrowest maintainable solution because it skips applyConfigEnvVars guard parity and changes the sandbox secret-exposure contract without maintainer/security approval or real proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reuse runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards collectConfigRuntimeEnvVars(options?.config) directly into sandbox exec defaults, but that collector does not apply the runtime guards used by applyConfigEnvVars: existing env keys win, and unresolved ${VAR} placeholders are skipped. Sandbox exec can therefore receive stale config values or literal placeholders that gateway exec intentionally avoids; derive the sandbox config env from the same guarded runtime contract.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff expands config-level env values into sandbox command environments and needs an explicit sandbox/security contract before merge.

Concerns:

- **[medium] Approve sandbox config env forwarding first:** `src/agents/pi-tools.ts:408`
  - body: The patch sends global config env.vars through sandbox exec environments even though current docs keep host/global env out of sandboxed runs. collectConfigRuntimeEnvVars allows credential-shaped keys, so this expands sandbox secret exposure unless the contract is approved, filtered, documented, and tested.
  - confidence: 0.9
- **[medium] Do not rely on container-create env sanitization:** `src/agents/bash-tools.shared.ts:30`
  - body: The PR's new config env layer reaches the per-command docker exec environment, while the existing sanitizeEnvVars call only filters configured Docker env during container creation. Credential-shaped config vars would bypass that claimed sanitizer unless a separate per-exec policy is added.
  - confidence: 0.88
