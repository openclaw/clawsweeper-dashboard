## Summary

Keep open: the PR targets a real source-visible sandbox env gap, but the proposed fix changes the documented sandbox env isolation contract, bypasses the runtime env.vars guards used by gateway mode, and still lacks contributor-owned real Docker sandbox proof.

## What This Changes

This PR threads config env.vars into sandbox exec environment construction through ExecToolDefaults/buildSandboxEnv and adds focused env propagation tests.

## Best Possible Solution

Define an approved sandbox config-env contract, preserve gateway env guard parity, update docs/tests for precedence and security behavior, and prove it with redacted Docker sandbox output.

## Reproduction Assessment

Yes at source level: current main's sandbox env builder only layers default PATH/HOME, sandbox env, and per-call params env, while gateway mode can inherit process.env after applyConfigEnvVars. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. Directly forwarding collectConfigRuntimeEnvVars is not the narrowest maintainable solution because it skips applyConfigEnvVars guard parity and changes the sandbox secret-exposure contract without approval or proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reuse runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards collectConfigRuntimeEnvVars(options?.config) into sandbox exec defaults, but that collector does not apply the runtime guards used by applyConfigEnvVars: existing env keys win, and unresolved ${VAR} placeholders are skipped. Sandbox exec can therefore receive stale config values or literal placeholders that gateway exec intentionally avoids; derive the sandbox config env from the same guarded runtime contract.
  - confidence: 0.92

## Security Review

Status: needs_attention
