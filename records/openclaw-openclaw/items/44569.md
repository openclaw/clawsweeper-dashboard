## Summary

Keep this PR open, but not mergeable as-is: it addresses a real source-level sandbox env gap, yet the patch bypasses runtime env.vars guards, changes a security-sensitive sandbox secret-forwarding contract, and still lacks contributor real behavior proof.

## What This Changes

The PR adds a configEnvVars exec default, threads config env.vars into sandbox exec environment construction, and adds propagation/precedence tests.

## Best Possible Solution

Define an approved sandbox config-env contract, derive the effective env with the same runtime guards, document precedence/security behavior, and prove it with focused tests plus redacted real sandbox exec output.

## Reproduction Assessment

Yes, at source level: current main builds sandbox exec env from PATH/HOME, sandboxEnv, and paramsEnv only, while gateway exec can inherit config env.vars after config load. I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. Passing collectConfigRuntimeEnvVars directly into sandbox exec is not the narrowest maintainable fix because it misses runtime guard parity and leaves the sandbox secret-forwarding contract undocumented.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve runtime env.vars guards:** `src/agents/pi-tools.ts:408`
  - body: This forwards collectConfigRuntimeEnvVars(options?.config) directly into exec defaults, but that collector does not apply the runtime guards used by applyConfigEnvVars/createConfigRuntimeEnv: skip already-present env keys and skip unresolved ${VAR} placeholders. Sandbox exec can therefore receive literal placeholders or stale config values that gateway exec intentionally omits.
  - confidence: 0.93
