## Summary

Keep this PR open, but do not merge it as-is. Current main still has the sandbox env propagation gap, but the patch forwards raw collected config env vars into sandbox exec without preserving runtime guards and changes the sandbox secret-exposure contract that docs currently describe.

## What This Changes

The PR threads collected config env.vars through exec defaults into buildSandboxEnv so sandbox exec commands receive config-level environment variables below sandbox env and per-call env overrides.

## Best Possible Solution

Revise the patch into an explicit sandbox env propagation contract: build an effective config env map that preserves applyConfigEnvVars semantics, layer it below sandbox/backend env and per-call params, coordinate with the related sandbox env reports, and update sandbox/env docs if maintainers approve forwarding config env vars into sandbox exec.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure env.vars, enable sandbox exec, and reference the key inside exec; current main’s sandbox path constructs the command env without process.env or config env vars.

## Solution Assessment

No. The proposed direction addresses the right gap, but direct collectConfigRuntimeEnvVars forwarding is not the safest fix because it skips runtime precedence and unresolved-placeholder guards; a guarded effective-env helper is the narrower maintainable path.

## Review Findings

Overall correctness: patch is incorrect
