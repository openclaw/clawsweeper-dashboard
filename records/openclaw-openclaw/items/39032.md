## Summary

Keep open: current main and the latest release still allow raw `tool`/`toolResult` text to be selected as subagent completion output, and the existing focused fix path is the open maintainer-labeled PR at https://github.com/openclaw/openclaw/pull/80110. The discussion adds useful context from the earlier unmerged fix PR and the later `childCompletionFindings` variant, so maintainers should resolve those paths before closing this issue.

## What This Changes

The issue asks OpenClaw to prevent subagent completion announcements from forwarding internal tool-failure or raw tool-output reasoning into the requester session.

## Best Possible Solution

Land or rebase https://github.com/openclaw/openclaw/pull/80110, or an equivalent selector/docs/test fix, and add coverage for the `childCompletionFindings` variant if it still reproduces.

## Reproduction Assessment

Yes, at source level. A child history with empty assistant text followed by `toolResult` or `tool` text populates `latestRawText`, the selector returns it, and current e2e coverage still asserts the raw tool text is announced.

## Solution Assessment

Yes for the central leak path: the narrow maintainable fix is to stop selecting raw tool/toolResult output as completion text and fall back to post-compaction assistant text or `(no output)`. Channel-specific `message` validation would only block one trigger, and the nested `childCompletionFindings` path should be verified before closing.
