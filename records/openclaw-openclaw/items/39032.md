## Label Justifications

- `P1`: The issue describes a live agent/session workflow leaking internal subagent/tool output into requester sessions.
- `impact:session-state`: The affected path is subagent completion state being selected and injected into the requester session.
- `impact:security`: The leaked content can include internal tool names, session keys, and error/debug details that should not be user-facing.

## Summary

Keep open: current main and the latest shipped release still let subagent output selection treat `toolResult`/`tool` text as raw completion fallback without checking `is_error`/`isError`, while the exact proposed fix in https://github.com/openclaw/openclaw/pull/39367 was closed unmerged. Later internal-context hardening covered adjacent surfaces, but not this selector path.

## What This Changes

The issue asks OpenClaw to prevent subagent completion announcements from forwarding internal tool-failure reasoning or tool error text into the requester session.

## Best Possible Solution

Add a narrow selector hardening that skips `is_error`/`isError` tool results before they can become subagent completion output, preserve successful tool-result behavior, add focused regression coverage, and coordinate it with https://github.com/openclaw/openclaw/pull/81360.

## Reproduction Assessment

Yes, at source level: a child history containing an error `toolResult` with no safe assistant final can be selected via `latestRawText`, and failed announce flow still calls `readSubagentOutput()`. I did not run a live subagent reproduction because this review is read-only and must not create runtime artifacts.

## Solution Assessment

No, current main is not the best solution because it still lacks the error-tool-result guard. The closed unmerged fix direction was narrow, but it needs to be reapplied to the current split helper and reviewed alongside the open lost-context selector fix.
