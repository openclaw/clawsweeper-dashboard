## Summary

Keep open. The feature is not already on current main, and the PR contains real iOS work, but the current diff still introduces Live Activity working-state bugs that should block merge until repaired.

## What This Changes

The PR adds iOS Liquid Glass styling, an App Intent/shortcut path into Talk Mode, and Live Activity/Dynamic Island connection-state UI.

## Best Possible Solution

Keep the feature in the iOS app only if maintainers want this UX, but make Live Activity task updates owned or tokenized so camera errors and overlapping tasks restore the correct state before rebase and device/simulator validation.

## Reproduction Assessment

Yes for the blocking findings from source inspection: camera snap/clip starts a Live Activity working task before awaiting capture, and the shared error path calls showCameraHUD with kind .error, which now skips both replacement and cleanup. I did not run the iOS app.

## Solution Assessment

No. The overall feature direction is plausible, but the current HUD-to-Live-Activity coupling needs exact cleanup ownership before it is the safest maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear the Live Activity when camera capture fails:** `apps/ios/Sources/Model/NodeAppModel.swift:1736-1737`
  - body: Camera snap/clip now publishes "Taking photo..." or "Recording..." before awaiting the capture. If that await throws, the shared catch calls `showCameraHUD(..., kind: .error)`, but this new guard skips Live Activity updates for errors, so the Dynamic Island can keep showing the in-progress task indefinitely. Clear the working task on error without publishing the raw error text.
  - confidence: 0.9
- **[P2] Keep auto-hide timers from clearing newer work:** `apps/ios/Sources/Model/NodeAppModel.swift:1752-1753`
  - body: A camera success HUD starts this auto-hide timer, but screen recording and direct LiveActivityManager working updates do not cancel or identify that HUD task. If a screen recording starts before the camera success timer fires, this unconditional `handleWorking(task: nil)` returns the Dynamic Island to idle while recording is still active. Track the active task before clearing it.
  - confidence: 0.87
