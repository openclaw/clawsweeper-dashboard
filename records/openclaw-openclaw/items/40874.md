## Summary

Keep this PR open: the iOS feature is not already implemented on current main, and the current PR head still has Live Activity working-state lifecycle bugs around camera errors and stale HUD timers.

## What This Changes

The PR adds iOS 26-gated Liquid Glass styling, an Open Talk Mode App Intent/shortcut, and expanded Live Activity / Dynamic Island status for connection and transient work.

## Best Possible Solution

Merge the iOS feature only after the Live Activity working-task lifecycle is tokenized or otherwise scoped so sanitized errors and stale HUD timers cannot leave or clear incorrect Dynamic Island state.

## Reproduction Assessment

Yes, source-level: the camera snap/clip and screen-recording paths directly show working-state transitions that can be left uncleared or cleared by an older timer. No device repro is needed to see the control-flow bug, though device validation should confirm the visible Dynamic Island behavior after repair.

## Solution Assessment

No, not as currently patched: the privacy direction of not publishing raw error strings is right, but the PR also needs sanitized cleanup on error and a way to prevent old camera HUD timers from clearing newer working tasks. The feature scope otherwise belongs in the iOS app and widget surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Clear live-activity state on camera failures:** `apps/ios/Sources/Model/NodeAppModel.swift:1736-1738`
  - body: Camera snap/clip publish a working task before awaiting the operation. If the camera call throws, the catch path shows an error HUD, but this guard skips both raw-error publication and clearing the prior task, leaving the Dynamic Island or lock screen stuck on the previous camera task. Clear the working state for errors without sending raw error text to the Live Activity.
  - confidence: 0.88
- **[P2] Avoid clearing newer live-activity tasks from HUD timers:** `apps/ios/Sources/Model/NodeAppModel.swift:1752-1753`
  - body: The camera HUD auto-hide task always calls handleWorking(task: nil) when its sleep finishes. Screen recording sets its own long-running working task without cancelling that camera timer, so a prior camera success HUD can fire during recording and revert the Dynamic Island to the connection state while recording is still active. Clear only the task this timer owns, or cancel/namespace the camera timer when a newer working task starts.
  - confidence: 0.84
