## Summary

Keep open: current main only contains the earlier minimal Live Activity connection-status slice, while this branch still has unique Action Button and Liquid Glass feature work plus two source-reproducible Live Activity lifecycle defects that should be fixed before merge.

## What This Changes

The PR adds iOS 26-gated Liquid Glass styling, an Open Talk Mode App Intent/Action Button shortcut, and expanded Dynamic Island/Live Activity task/status UI for the iOS app.

## Best Possible Solution

Keep the branch on the PR path, but make Live Activity working-state cleanup owner-scoped and clear failed camera tasks immediately before an iOS owner decides whether to merge the platform UX feature.

## Reproduction Assessment

Yes for the PR blockers: source inspection of PR head gives a high-confidence sequence where an older camera HUD timer can clear a newer screen-recording working state, and camera failures leave the prior task visible until the error HUD timer fires. I did not run a live device reproduction in this read-only review.

## Solution Assessment

No, not as currently patched. The feature belongs in the iOS app/widget surfaces, but the working-state cleanup needs an owner/task match instead of a global nil reset, and camera failure handling should restore Live Activity state immediately without exposing raw error text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope HUD timer cleanup to its own task:** `apps/ios/Sources/Model/NodeAppModel.swift:1753`
  - body: The camera HUD auto-hide task clears the Live Activity with `handleWorking(task: nil)` after its delay regardless of which operation currently owns the working state. If a camera success HUD starts, then screen recording starts before the HUD delay fires, this old timer restores connection state while recording is still active. Track an owner/task token or clear only when the visible task still matches this HUD.
  - confidence: 0.86
- **[P3] Clear Live Activity state when camera work fails:** `apps/ios/Sources/Model/NodeAppModel.swift:1736-1738`
  - body: `snap` and `clip` set a camera working state before awaiting camera work, but the catch path shows an error HUD with `kind == .error`, which skips `handleWorking` and leaves the prior task visible until the error HUD auto-hide timer fires. Clear the task immediately on failure while keeping raw error text off the lock screen/Dynamic Island.
  - confidence: 0.78
