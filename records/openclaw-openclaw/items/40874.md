## Summary

Keep open: current main still only has the minimal Live Activity connection baseline, while this PR contains unique iOS App Intent, Talk Mode, Liquid Glass, and expanded Dynamic Island work. The branch still has two source-reproducible Live Activity lifecycle issues that should be fixed before merge.

## What This Changes

The PR adds iOS 26-gated Liquid Glass styling, an Open Talk Mode App Intent/Action Button shortcut, and expanded Dynamic Island/Live Activity task UI for the iOS app.

## Best Possible Solution

Repair Live Activity working-state ownership and failure cleanup on the PR branch, then have an iOS owner review the user-facing platform feature before merge.

## Reproduction Assessment

Yes for the PR blockers: PR head source gives a high-confidence sequence where an older camera HUD timer can clear a newer screen-recording state, and camera failures leave stale working state until the HUD timer fires. I did not run a live iOS device reproduction in this read-only review.

## Solution Assessment

No, not as currently patched. The feature belongs in the iOS app/widget surfaces, but Live Activity cleanup needs to be owner/task-scoped and camera failures should restore the prior state immediately without exposing raw error text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope HUD cleanup to the task it created:** `apps/ios/Sources/Model/NodeAppModel.swift:1753`
  - body: The camera HUD auto-hide task always calls `handleWorking(task: nil)` after its delay. If a camera success HUD starts and then screen recording begins before that delay fires, the old HUD timer restores connection state while recording is still active; clear only when the visible Live Activity task still belongs to that HUD.
  - confidence: 0.86
- **[P3] Clear working state on camera failure:** `apps/ios/Sources/Model/NodeAppModel.swift:1736-1738`
  - body: `snap` and `clip` set a camera working state before awaiting camera work, but camera errors go through the error HUD path, which skips `handleWorking` cleanup. That leaves the previous camera task on the Dynamic Island/lock screen until the error HUD timer fires; restore the prior state immediately while keeping raw errors private.
  - confidence: 0.78
