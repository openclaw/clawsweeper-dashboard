## Summary

Keep this PR open, but not merge-ready: the feature is not already on main and source review found a PR-introduced Dynamic Island state bug that can clear an active screen-recording task from a stale camera HUD timer.

## What This Changes

The PR adds iOS 26-gated Liquid Glass styling, an App Intent/App Shortcuts path into Talk Mode, and Live Activity/Dynamic Island connection and task-status UI for the iOS app.

## Best Possible Solution

Fix the Live Activity cleanup race, rebase the branch, then have the iOS owners decide whether the Action Button, Dynamic Island, and Liquid Glass UX should land as core app behavior.

## Reproduction Assessment

Yes for the PR finding by source inspection: start a camera HUD flow that schedules auto-hide, then start screen recording before that timer fires; the old timer reaches handleWorking(nil) and clears the newer screen-recording Live Activity state. This is PR-only feature code, not a current-main regression I reproduced at runtime.

## Solution Assessment

No, not as-is. The PR should make Live Activity task cleanup token-aware or cancel/replace the HUD timer when a new non-HUD task starts, so an older camera HUD cannot clear unrelated active work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard HUD cleanup before clearing Live Activity:** `apps/ios/Sources/Model/NodeAppModel.swift:1753`
  - body: This auto-hide task can outlive the HUD state that created it. If a camera success HUD schedules this timer and the user starts screen recording before it fires, `handleScreenRecordInvoke` sets the Dynamic Island to `Recording screen…`, but the older HUD timer still reaches this line and calls `handleWorking(nil)`, reverting the Live Activity while recording is still running. Track the active task/token or cancel this timer when screen recording starts before clearing shared Live Activity state.
  - confidence: 0.86
