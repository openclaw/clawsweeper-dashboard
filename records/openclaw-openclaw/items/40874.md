## Summary

Keep open: current main does not include the Liquid Glass/App Intent/expanded working-state Live Activity feature, and PR head still has two source-reproducible Live Activity lifecycle defects before this can be considered merge-ready.

## What This Changes

The PR adds iOS 26-gated Liquid Glass styling, an App Intent/Action Button entry into Talk Mode, and expanded Live Activity/Dynamic Island connection and working-state UI.

## Best Possible Solution

Keep the branch open, scope Live Activity working-state cleanup to the owning task, clear failed camera work immediately without exposing raw error text, then have an iOS owner decide whether to land the platform UX feature.

## Reproduction Assessment

Yes for the PR blockers: PR-head source inspection gives a high-confidence sequence where an older camera HUD timer can clear a newer screen-recording working state, and camera failures leave the prior task visible until the error HUD timer fires. I did not run a live iOS device or simulator reproduction in this read-only review.

## Solution Assessment

No, not as currently patched. The feature belongs in the iOS app/widget surfaces, but the Live Activity working-state cleanup should be tied to the owning task and camera failure handling should restore state immediately without publishing raw error text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope HUD cleanup to its own task:** `apps/ios/Sources/Model/NodeAppModel.swift:1753`
  - body: The camera HUD auto-hide task always calls `handleWorking(task: nil)` after its delay. If a camera success HUD is visible and screen recording starts before that delay fires, the old timer restores connection state while recording is still active; track an owner/task token or clear only when the current task still matches this HUD.
  - confidence: 0.86
- **[P3] Clear Live Activity on camera failure:** `apps/ios/Sources/Model/NodeAppModel.swift:1736-1738`
  - body: `snap` and `clip` set a camera working state before awaiting camera work, but the catch path shows an error HUD with `kind == .error`, which skips `handleWorking`. A failed camera operation leaves the previous task on the Dynamic Island/lock screen until the error HUD timer fires; clear the task immediately while still keeping raw error text off Live Activity.
  - confidence: 0.78
