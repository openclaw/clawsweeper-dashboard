## Summary

## What This Changes

The PR adds a QQBot GatewayConnection flag that skips close handling after opcode RECONNECT so sessionId and lastSeq survive for the next RESUME attempt.

## Best Possible Solution

Handle 4009 resume eligibility in ReconnectState for every 4009 close, keep 4006/4007 clearing behavior covered by tests, and require redacted after-fix QQBot proof showing RESUME/RESUMED.

## Reproduction Assessment

Yes. The source path is clear on current main: RECONNECT closes the socket, the close handler processes 4009 through ReconnectState, sessionId/lastSeq are cleared, and the next HELLO cannot use RESUME.

## Solution Assessment

No. Preserving session state for 4009 is the right direction, but the maintainable fix belongs in the close-code policy rather than a connection-global close-handler bypass.
