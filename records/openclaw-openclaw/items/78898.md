## Summary

Keep open: current main and v2026.6.10 still route QQBot RECONNECT-triggered closes through 4009 session clearing, and this PR is the active candidate fix but still lacks after-fix live proof and focused regression coverage.

## What This Changes

The PR adds an `isReconnectInitiated` flag in the QQBot gateway connection so close events after `GatewayOp.RECONNECT` skip `handleClose` and preserve session state for `RESUME`.

## Best Possible Solution

Land a narrow QQBot reconnect-policy fix only after adding focused close-code tests and redacted live QQBot proof that RECONNECT followed by 4009 reconnects with `RESUME` and reaches `RESUMED`.

## Reproduction Assessment

Yes, source-level: current main handles RECONNECT by closing the socket, then maps 4009 through `handleClose` to clear `sessionId` and `lastSeq`, so the next HELLO cannot choose `RESUME`. I did not run a live QQBot connection in this read-only review.

## Solution Assessment

Unclear as a merge-ready implementation: preserving session state for RECONNECT/4009 is the right direction, but the close-handler bypass needs focused tests and live proof before calling it the best fix. A `ReconnectState` policy-helper approach may be cleaner if maintainers want close-code policy centralized.
