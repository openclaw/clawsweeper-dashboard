## Summary

Keep open: current main and v2026.6.9 still clear QQBot session state after `RECONNECT` leads into a `4009` close, so the PR addresses a real active bug. The patch is plausible, but it is not merge-ready without after-fix live QQBot proof and focused regression coverage for the session-state path.

## What This Changes

The PR adds a reconnect-initiated close flag in QQBot `GatewayConnection` so a close after `GatewayOp.RECONNECT` skips `handleClose` and preserves `sessionId`/`lastSeq` for `RESUME`.

## Best Possible Solution

Land a narrow QQBot reconnect-policy fix only after adding focused tests and redacted live QQBot proof that a server `RECONNECT` followed by `4009` reconnects with `RESUME` and reaches `RESUMED`.

## Reproduction Assessment

Yes, source-reproducible: current main handles `RECONNECT` by closing the socket, then maps `4009` through `handleClose` to clear `sessionId` and `lastSeq`, so the next `HELLO` cannot choose `RESUME`. I did not run a live QQBot connection in this read-only review.

## Solution Assessment

Unclear as a merge-ready implementation: preserving state for `RECONNECT`/`4009` is the right direction, but the current bypass needs focused tests and live proof before calling it the best fix. A policy-helper approach in `ReconnectState` may be cleaner if maintainers want the close-code decision centralized.
