## Summary

Keep open: current main and v2026.6.9 still clear QQBot session state after the RECONNECT/4009 path, and this PR is the active candidate for the canonical linked issue, but it still lacks after-fix live QQBot proof and focused regression coverage for a session-state change.

## What This Changes

Adds an `isReconnectInitiated` flag to QQBot `GatewayConnection` so close events after `GatewayOp.RECONNECT` skip `handleClose` and preserve `sessionId`/`lastSeq` for `RESUME`.

## Best Possible Solution

Land a narrow QQBot reconnect-policy fix only after adding focused tests and redacted live QQBot proof that a server `RECONNECT` followed by `4009` reconnects with `RESUME` and reaches `RESUMED`.

## Reproduction Assessment

Yes, source-level: current main handles `RECONNECT` by closing the socket, then maps `4009` through `handleClose` to clear `sessionId` and `lastSeq`, so the next `HELLO` cannot choose `RESUME`. I did not run a live QQBot connection in this read-only review.

## Solution Assessment

Unclear as a merge-ready implementation: preserving state for `RECONNECT`/`4009` is the right direction, but the current bypass needs focused tests and live proof before calling it the best fix. A policy-helper approach in `ReconnectState` may be cleaner if maintainers want close-code policy centralized.
