## Best Possible Solution

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: current main handles RECONNECT by closing the socket, then the close handler passes 4009 to ReconnectState, which clears sessionId/lastSeq before the next HELLO can choose RESUME. I did not run a live QQBot connection in this read-only review.

## Solution Assessment

No, this is a plausible mitigation but not the best fix shape. The maintainable fix belongs in the close-code policy for SESSION_TIMEOUT so every 4009 follows the official RESUME contract, with tests and real after-fix proof.
