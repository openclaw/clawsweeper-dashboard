## Summary

Keep open: current main and the latest release still schedule a synchronous 30-minute SQLite TRUNCATE checkpoint on the Node event loop for shared SQLite-backed stores, and the related open symptom report has matching idle 8-12s stall logs. The report is credible, but the repair should preserve the earlier bounded-WAL fix instead of blindly switching checkpoint modes.

## What This Changes

Request to prevent periodic SQLite WAL TRUNCATE checkpoints from blocking the gateway event loop for SQLite-backed task, plugin-state, proxy, and memory stores.

## Best Possible Solution

Adopt a maintainer-approved WAL maintenance strategy that keeps WAL files bounded without running long checkpoint I/O on the gateway event loop, such as off-thread checkpointing or a documented PASSIVE periodic checkpoint plus safe shutdown/size-limit truncation.

## Reproduction Assessment

No for the full production stall: I did not run a 30-minute current-main gateway with a large WAL and capture the delay. Yes at source level: current main and v2026.5.7 schedule a synchronous TRUNCATE checkpoint on the event loop, and https://github.com/openclaw/openclaw/issues/80820 has matching idle 30-minute stall logs.

## Solution Assessment

Unclear. The proposed PASSIVE or hybrid direction is plausible, but the current TRUNCATE timer was intentionally added to fix bounded WAL growth, so maintainers should choose the checkpoint strategy before a narrow repair PR is queued.
