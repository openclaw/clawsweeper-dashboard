## Summary

Keep open: current main and the latest release still run a 30-minute synchronous SQLite TRUNCATE checkpoint on the gateway event loop, and the related open symptom report shows matching idle 8-12s stalls. The report is credible, but the exact fix should account for the prior intentional WAL-growth fix before changing checkpoint behavior.

## What This Changes

The issue reports that periodic SQLite WAL maintenance runs a synchronous TRUNCATE checkpoint on the Node event loop every 30 minutes, potentially blocking all SQLite-backed gateway stores.

## Best Possible Solution

Make periodic WAL maintenance unable to stall the gateway event loop while preserving bounded WAL files through an owner-approved strategy such as off-thread checkpointing, shutdown-only truncation, journal size limits, or a carefully documented PASSIVE/TRUNCATE split.

## Reproduction Assessment

No for the full production stall: I did not run a 30-minute current-main gateway with a large WAL and capture an 8-12s delay. Yes at source level: current main and v2026.5.7 schedule a synchronous TRUNCATE checkpoint on the event loop, and the related open report has matching 30-minute idle stall logs.

## Solution Assessment

Unclear. The proposed hybrid direction is plausible, but the current TRUNCATE timer was intentionally added to fix unbounded WAL growth, so the best repair needs maintainer choice on the checkpoint strategy rather than a blind one-line mode change.
