## Label Justifications

- `P1`: The issue describes a recurring gateway event-loop stall that can block real channel, HTTP, and timer work for deployed users.
- `impact:crash-loop`: The affected behavior is process-level availability degradation from long synchronous SQLite I/O on the gateway event loop.

## Summary

Keep open: current `main` and the latest release still run the shared SQLite WAL checkpoint timer with the default `TRUNCATE` mode through synchronous `DatabaseSync.exec()`, and https://github.com/openclaw/openclaw/pull/82366 is the active open implementation candidate for this issue. The report is source-reproducible, but the fix should preserve the earlier WAL sidecar growth protection while removing the periodic event-loop stall.

## What This Changes

This issue asks OpenClaw to stop running periodic SQLite WAL `TRUNCATE` checkpoints synchronously on the gateway event loop for SQLite-backed stores.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/82366 with a WAL maintenance strategy that keeps periodic work lightweight on the event loop while preserving controlled WAL truncation.

## Reproduction Assessment

Yes from source inspection: current `main` schedules a 30-minute interval that calls synchronous `DatabaseSync.exec()` with `PRAGMA wal_checkpoint(TRUNCATE)`. I did not run a live 30-minute stall reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the checkpoint tradeoff. The linked PASSIVE-periodic/TRUNCATE-on-close approach is narrow and matches the credible root cause, but it should be reviewed against the earlier WAL-growth fix before merge.
