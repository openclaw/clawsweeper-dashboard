## Summary

Keep open: current main and v2026.5.12 still schedule the 30-minute synchronous SQLite TRUNCATE checkpoint, the related open symptom report has matching idle 8-12s stall logs, and an open PR now appears to address the WAL timer path but has not merged.

## What This Changes

The issue reports that periodic SQLite WAL TRUNCATE checkpoints run synchronously on the Node event loop every 30 minutes and can stall gateway traffic for SQLite-backed stores.

## Best Possible Solution

Review and land a focused fix, likely the PASSIVE-periodic/TRUNCATE-on-close approach in https://github.com/openclaw/openclaw/pull/81897 if maintainers accept the WAL-growth tradeoff and the branch gets real behavior proof.

## Reproduction Assessment

No for a full live current-main reproduction of the multi-second stall; this review did not run a gateway for 30 minutes with a large WAL. Yes at source level: current main and v2026.5.12 schedule synchronous DatabaseSync.exec TRUNCATE checkpoints on the event loop, and the related report supplies matching production timing logs.

## Solution Assessment

Unclear. The hybrid PASSIVE timer with TRUNCATE on close is plausible and already proposed in an open PR, but maintainers should confirm it preserves the bounded-WAL contract or choose off-thread checkpointing instead.
