## Summary

Keep open: current main and the latest release still schedule a synchronous 30-minute SQLite `TRUNCATE` checkpoint on the gateway event loop, and an open PR attempts the WAL part but is unmerged and blocked by broader review/proof issues.

## What This Changes

The issue requests eliminating the 30-minute synchronous SQLite WAL `TRUNCATE` checkpoint on the gateway event loop for SQLite-backed stores.

## Best Possible Solution

Make periodic WAL maintenance unable to stall the gateway event loop while preserving bounded WAL behavior, likely by landing a narrow `PASSIVE` timer plus `TRUNCATE`-on-close or off-thread checkpoint fix with regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level, but not as a live 8-12 second run: current main schedules a 30-minute `setInterval` that calls synchronous `DatabaseSync.exec()` with `PRAGMA wal_checkpoint(TRUNCATE)`, and https://github.com/openclaw/openclaw/issues/80820 contains matching idle stall logs.

## Solution Assessment

Is this the best way to solve the issue? Unclear as proposed; the hybrid direction in https://github.com/openclaw/openclaw/pull/81897 is plausible, but the final repair should preserve the prior WAL-growth fix and prove the runtime stall path.

## Risks / Open Questions

- `PASSIVE` avoids waiting for readers or writers, but it is still invoked via synchronous `DatabaseSync.exec()` and may still perform disk I/O on the caller thread.
