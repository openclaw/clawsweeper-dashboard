## Label Justifications

- `P1`: A latest-release regression can silently drop `/compact` command feedback in real channel workflows.
- `impact:message-loss`: The primary symptom is a generated command reply being accepted as delivered but not reaching the user.
- `impact:session-state`: The affected command manages context compaction, so missing feedback can cause repeated compaction attempts and session-state confusion.

## Summary

Keep open: current main and v2026.6.1 contain the earlier core fix for explicit command replies and internal compaction callbacks, but the fresh release logs point to an uncovered Feishu/Lark system-command dispatch path, and the hook-message `notifyUser` request conflicts with current tested de-duplication behavior.

## What This Changes

The issue reports that `/compact` still loses visible replies on OpenClaw 2026.6.1 with Feishu/Lark and WebChat, and asks to revisit compaction `notifyUser` delivery when hook messages are present.

## Best Possible Solution

Reproduce the release/current-main behavior through the Lark plugin path, then either route plugin system-command replies through the canonical command-turn dispatch contract or strengthen the SDK so `delivered=true` cannot be returned without an actual channel send; leave hook-message de-duplication unchanged unless maintainers intentionally change that contract.

## Reproduction Assessment

No high-confidence local reproduction yet. The reporter provides concrete v2026.6.1 logs, but source inspection only verifies the canonical core/bundled paths, not the external plugin `dispatchSystemCommand` path described in the report.

## Solution Assessment

Unclear. The prior core fix remains the right shape for bundled command replies, but this report needs a plugin-path or SDK-contract fix rather than a broad change to current hook-message de-duplication without maintainer confirmation.
