## Summary

Keep open: the PR addresses a valid compaction bug, but the current head still has the reserve/budget threshold bug from the prior review and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a threshold-aware no-real-conversation skip helper, forwards manual `/compact` token and context-budget data into compaction, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the token-pressure bypass direction, but compute the threshold with a reserve capped against the caller budget and add a regression for a reserve larger than the active session window.

## Reproduction Assessment

Yes, by source inspection. Manual `/compact` on a limited heartbeat/tool-result slice such as the linked 236k/200k session reaches the current outer no-real-conversation return before token pressure can affect the decision.

## Solution Assessment

No. The approach is narrow and maintainable, but the current PR head still subtracts a compaction-model reserve from the caller context window instead of keeping both values on the same budget.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Cap the reserve to the caller budget:** `src/agents/pi-embedded-runner/compact.ts:339-340`
  - body: The helper now compares observed tokens against `callerContextWindowTokens`, but it still subtracts the raw compaction reserve from the settings manager. With a 16k active session and the default 20k reserve, the threshold clamps to 1, so below-threshold tool-only slices bypass the skip and trigger unnecessary compaction. Cap or resolve the reserve against the same caller budget before subtracting it.
