## Summary

Keep open: the PR targets the right bug, but the latest head still mixes the caller context window with a reserve prepared for the compaction target window, which can re-enable unnecessary compaction for under-threshold idle/tool-only small active sessions.

## What This Changes

The PR changes manual `/compact` and direct compaction so fresh token usage and caller context budget can let over-threshold tool-heavy sessions bypass the idle skip.

## Best Possible Solution

Land a narrow compaction fix that compares observed token pressure against one consistent active-session budget, with the reserve resolved or capped for that same caller window while preserving the below-threshold idle skip.

## Reproduction Assessment

Yes. Current main can be reproduced with a limited heartbeat/tool-heavy slice that has no meaningful user message inside the 20-message tool-result lookback plus persisted usage such as 236k/200k; the outer guard skips before token pressure can affect the decision.

## Solution Assessment

No. The PR is the right shape, but the threshold calculation must keep the context window and reserve on the same caller budget rather than subtracting a compaction-target reserve from the active-session window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Cap the reserve to the caller budget:** `src/agents/pi-embedded-runner/compact.ts:339-340`
  - body: The helper now switches the threshold window to `callerContextWindowTokens`, but it still subtracts the reserve from the settings manager prepared with `ctxInfo.tokens` for the compaction model. With `agents.defaults.compaction.model` set to a large summarizer and a 16k active session, the default 20k reserve clamps the threshold to 1, so idle/tool-only sessions below the active threshold bypass the skip and make an unnecessary compaction call. Resolve or cap the reserve against the same caller budget used here.
