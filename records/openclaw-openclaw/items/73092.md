## Summary

Keep open. The PR targets a real current-main compaction bug and has the right token-forwarding shape, but the latest head still mixes the caller context window with a reserve resolved for the compaction model, and it lacks real behavior proof.

## What This Changes

The branch forwards persisted token usage and caller context budget into manual `/compact`, then changes the no-real-conversation guard to skip only when observed tokens are below threshold, with tests and a changelog entry.

## Best Possible Solution

Keep the narrow compaction fix, but resolve or cap the reserve against the same caller context budget used for the threshold, add a regression where reserve exceeds the caller window, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a limited heartbeat/tool-result slice with observed usage such as 236k/200k reaches the current outer no-real-conversation return before compaction can use token pressure.

## Solution Assessment

No. The proposed direction is maintainable, but the current PR head still subtracts a compaction-model reserve from the caller context window instead of keeping both values on the same budget.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Cap the reserve to the caller budget:** `src/agents/pi-embedded-runner/compact.ts:339-340`
  - body: The helper now uses `callerContextWindowTokens` for the threshold window, but it still subtracts `reserveTokens` from the settings manager prepared for the compaction model. With a large summarizer and a 16k active session, the default 20k reserve makes the threshold clamp to 1, so below-threshold idle/tool-only slices bypass the skip and trigger unnecessary compaction. Resolve or cap the reserve against the same caller budget before subtracting it.
  - confidence: 0.87
