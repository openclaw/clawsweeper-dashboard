## Summary

Keep open. The fix direction is narrow and the PR has real-behavior proof, but the diff still has one budget-threshold bug: it treats the queued wrapper's compaction-target `contextTokenBudget` as a caller budget, so some over-threshold tool-heavy sessions can still hit the idle skip before compaction.

## What This Changes

The PR adds a threshold-aware compaction idle-skip helper, forwards manual `/compact` token and caller-budget data into compaction, adds focused tests, and updates the changelog.

## Best Possible Solution

Land the caller-budget idle-skip bypass after separating explicit caller budgets from wrapper-resolved compaction budgets, then close https://github.com/openclaw/openclaw/issues/73056 with the merge proof.

## Reproduction Assessment

Yes, by source inspection. Current main can still hit the outer no-real-conversation return on a tool-heavy limited slice before token pressure is considered, matching the linked manual `/compact` failure shape.

## Solution Assessment

No, not as written. The manual caller-budget forwarding is the right approach, but the threshold helper should only ignore the compaction reserve when an explicit caller budget is present, not when the wrapper has filled `contextTokenBudget` from the compaction target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use only explicit caller budgets for the idle threshold:** `src/agents/pi-embedded-runner/compact.ts:1204-1205`
  - body: `compactEmbeddedPiSession()` fills `contextTokenBudget` from the compaction target when callers omit it, so this fallback makes `callerContextWindowTokens` defined for normal queued compaction too. That disables reserve subtraction and means a tool-heavy session at the compaction threshold, for example 180k/200k with a 20k reserve, can still be skipped until it reaches the full window. Please only use `callerContextTokenBudget` here, or carry an explicit flag proving the budget came from the active caller.
