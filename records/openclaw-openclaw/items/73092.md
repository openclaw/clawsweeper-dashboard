## Summary

Keep open. The linked bug is valid and current main still has the outer no-real-conversation skip, but the latest PR remains incorrect because it compares caller-window token pressure against a reserve prepared for the compaction target model, which can collapse the skip threshold.

## What This Changes

The branch adds a threshold-aware no-real-conversation compaction guard, forwards fresh manual `/compact` token usage and caller budget, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow compaction fix that uses one active-session budget for both the threshold window and the reserve in the idle-skip bypass, with regression coverage for a small caller model and larger compaction summarizer.

## Reproduction Assessment

Yes. Current main can be reproduced with a limited heartbeat/tool-result slice that lacks a meaningful user message in the 20-message lookback while fresh persisted usage is over the active context threshold.

## Solution Assessment

No, not yet. The PR is the right shape, but the reserve used in the new threshold must be resolved or capped for the same caller budget as the window comparison.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Cap the reserve against the caller budget:** `src/agents/pi-embedded-runner/compact.ts:337-338`
  - body: The new guard switches the threshold window to `callerContextWindowTokens`, but still subtracts the reserve from the settings manager prepared with the compaction target window. With a large configured summarizer and a small active model, the default reserve can exceed the caller window, clamp the threshold to 1 token, and let idle/tool-only sessions below the active threshold bypass the skip. Resolve or cap the reserve for the same caller budget used by this guard.
