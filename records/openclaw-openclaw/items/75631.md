## Summary

Keep this open. Current main still has macOS app paths that can submit or locally write a full config snapshot while gateway RPC writes have stronger base-hash and destructive-write protections, so the reported `gateway.auth` loss and restart/disconnect behavior is not proven fixed.

## What This Changes

The issue reports that macOS app/dashboard settings writes can rewrite persisted config, drop `gateway.auth`, and restart the Gateway, disconnecting dashboard WebSocket clients with service-restart close codes.

## Best Possible Solution

Harden the macOS config layer so app/dashboard settings changes use conflict-aware partial mutation or merge against a fresh gateway snapshot, preserve unknown and auth fields by default, and reject local fallback writes that remove `gateway.auth` unless the user is explicitly editing auth. Add regression coverage for Talk/voicewake/dashboard settings writes preserving auth and unrelated root fields while avoiding unnecessary restarts.

## Reproduction Assessment

Yes. The issue provides concrete macOS setup, RPC/log signatures, and expected file drift, and current-main source still contains the full-snapshot plus local fallback paths needed for a stale app snapshot to clobber auth; this review did not run the macOS app live.

## Solution Assessment

No current implementation is not the best solution. The gateway RPC path has base-hash and destructive-write protections, but the macOS app still needs an equivalent patch/conflict-aware contract or a hardened local writer before this can close.
