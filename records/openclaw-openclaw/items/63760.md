## Summary

Keep open. Current main still has the cost-control gap for completion announce relays: direct and queued announce delivery can invoke the requester session through the normal `agent` path without any announce-specific model override or lightweight/no-history context policy.

## What This Changes

The issue requests configurable cheaper-model and lighter-context policy for subagent and ACP completion announce runs.

## Best Possible Solution

Add a schema-backed announce relay policy under the existing subagent defaults/per-agent inheritance surface, with an optional configured model resolved through normal model selection/auth checks and an optional lightweight/no-history announce context that still preserves delivery routing, internal task-completion event synthesis, idempotency, and fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: follow a subagent or ACP completion into `deliverSubagentAnnouncement`, then observe the direct or queued `agent` call uses the requester session without model/context override fields and therefore falls back to normal session model/history resolution.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented as policy on the existing config and agent-call path; a hardcoded cheaper model or raw direct message send would be less correct because announce delivery still needs routing, auth, retries, and assistant-voice synthesis.
