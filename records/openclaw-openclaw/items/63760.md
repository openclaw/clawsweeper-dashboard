## Summary

Keep open. Current main still lacks announce-specific model or light-context controls: completion announces are routed through the requester session's normal gateway `agent` path, while config/docs only expose spawned-subagent model selection and announce timeout.

## What This Changes

The issue requests cheaper, lighter announcement delivery for subagent or ACP completions instead of using the parent model with full cold-cache context.

## Best Possible Solution

Define an explicit announce-cost policy, then implement a documented config/API contract that preserves requester delivery semantics while allowing a cheaper model or bounded context where safe.

## Reproduction Assessment

Yes for the source-level behavior: direct announces call the gateway `agent` method on the requester session without announce-specific model or light-context overrides. No live token-cost run was performed, so the exact dollar impact is not reproduced here.

## Solution Assessment

Unclear. The proposed `announce.model` and `announce.lightContext` knobs are plausible, but maintainers should first decide how much announce runs may diverge from the requester agent's model, session context, and user-facing voice.
