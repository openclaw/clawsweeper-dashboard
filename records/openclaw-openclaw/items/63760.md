## Summary

Keep open: current main still lacks an announce-specific model or lightweight-context policy, although active requester sessions now have a steering path that partially reduces cold follow-up runs.

## What This Changes

The issue requests cheaper, lighter announce delivery for subagent and ACP completion handoffs to avoid parent-model cold-cache costs.

## Best Possible Solution

Define an explicit announce delivery policy for model and context selection, then wire it through the existing requester-session handoff without weakening active-session steering or delivery routing semantics.

## Reproduction Assessment

Yes, at source level: an inactive requester completion follows the direct announce `agent` call and the current config contract has no announce-specific model or light-context option. I did not run a live ACP/subagent completion because this review is read-only and live billing proof is outside the cleanup pass.

## Solution Assessment

Unclear. The proposed `announce.model` and `announce.lightContext` direction is plausible, but the exact config shape and whether to prefer a cheaper agent run, a non-agent relay, or context-mode reuse needs maintainer product/API judgment.
