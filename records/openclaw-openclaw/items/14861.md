## Summary

Keep open: current main still has compact startup/readiness logs and plugin-init failure warnings, but no operator-facing started/skipped/failed Gateway boot outcome collector. The concrete Gmail watcher missing-account path still produces no normal startup outcome, and live issue data shows no linked closing PR.

## What This Changes

Request adding Gateway boot observability that reports started, skipped, and failed outcomes for sidecars/hooks such as gmail-watcher.

## Best Possible Solution

Add a redacted Gateway startup outcome collector around existing sidecar and hook start points, then emit one concise post-sidecar summary while preserving current readiness and failure behavior.

## Reproduction Assessment

Yes, at source level. With hooks enabled but no hooks.gmail.account, startGatewaySidecars skips the Gmail watcher path, and the lifecycle wrapper still suppresses the no-account reason when it is called.

## Solution Assessment

Yes. The narrow maintainable fix is to collect outcomes where each subsystem already starts or skips and log a redacted summary after sidecars settle; opt-in traces or fatal optional skips would not solve the operator observability gap.
