## Summary

Keep open because current main still lacks a structured Gateway boot outcome summary, and the concrete Gmail watcher skipped-start case remains silent in normal startup.

## What This Changes

The issue requests structured Gateway boot observability for subsystem startup successes, skips, and failures such as gmail-watcher and hooks.

## Best Possible Solution

Add a redacted Gateway startup outcome collector around existing sidecar start points and emit one concise post-sidecar summary while preserving current readiness and failure behavior.

## Reproduction Assessment

Yes, at source level. With hooks enabled but no hooks.gmail.account, startGatewaySidecars never calls the Gmail watcher lifecycle wrapper, and the wrapper still suppresses the no-account skipped reason when it does run.

## Solution Assessment

Yes. The narrow maintainable path is to collect outcomes where each sidecar already starts and log them once after sidecars settle; making optional skips fatal or only restoring formatted startup output would not solve this observability gap.
