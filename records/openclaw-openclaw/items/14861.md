## Summary

Keep open because current main still lacks a structured Gateway boot outcome summary, and the concrete Gmail watcher missing-account path remains silent during normal startup.

## What This Changes

The issue requests Gateway startup logging that reports subsystem, hook, and sidecar startup outcomes, including skipped Gmail watcher reasons.

## Best Possible Solution

Add a redacted Gateway startup outcome collector around existing sidecar start points and emit one concise post-sidecar summary while preserving current readiness and failure behavior.

## Reproduction Assessment

Yes, at source level. With hooks enabled but no hooks.gmail.account, current startup never schedules the Gmail watcher wrapper, and the wrapper still suppresses the no-account skipped reason when called.

## Solution Assessment

Yes. The narrow maintainable path is to collect outcomes where each sidecar already starts and log them once after sidecars settle; making optional skips fatal or only changing formatting would not solve this observability gap.
