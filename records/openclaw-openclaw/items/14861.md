## Summary

Keep open: current main still has compact startup logs and trace-only sidecar metrics, while the Gmail watcher skip remains source-reproducible and no linked PR closes it.

## What This Changes

Request adding Gateway startup observability that logs started/skipped/failed outcomes for subsystems and hooks such as Gmail watcher after boot.

## Best Possible Solution

Add a redacted Gateway startup outcome collector around existing sidecar and hook start points, then emit one concise post-sidecar summary while preserving readiness and failure behavior.

## Reproduction Assessment

Yes, at source level. With hooks enabled but no hooks.gmail.account, current startup skips the Gmail watcher path, and the lifecycle wrapper still suppresses the no-account reason when called directly.

## Solution Assessment

Yes for the direction. The maintainable fix is to collect redacted started/skipped/failed outcomes where subsystems already start or skip and log one summary after sidecars settle, rather than making optional skips fatal or relying on trace-only diagnostics.
