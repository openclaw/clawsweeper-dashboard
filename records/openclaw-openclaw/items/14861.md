## Summary

Keep open: current main has opt-in timing traces and compact startup logs, but no operator-facing started/skipped/failed outcome collector, and the Gmail missing-account startup path remains silent during normal startup.

## What This Changes

The issue requests Gateway startup observability that reports which subsystems/hooks started, skipped, or failed, with Gmail watcher startup as the concrete silent-skip example.

## Best Possible Solution

Add a redacted Gateway startup outcome collector around existing sidecar start points and emit one concise post-sidecar summary while preserving current readiness and failure behavior.

## Reproduction Assessment

Yes, at source level. With hooks enabled but no hooks.gmail.account, startGatewaySidecars never calls the Gmail watcher lifecycle wrapper, and the wrapper still suppresses the no-account skipped reason when it does run.

## Solution Assessment

Yes. The narrow maintainable path is to collect outcomes where each sidecar already starts or skips and log them once after sidecars settle; opt-in timing traces or fatal optional skips would not solve the operator observability gap.
