## Summary

## What This Changes

## Best Possible Solution

Return a canonical post-write config for both root-file writes and include fast-path writes, then add after-fix real behavior proof and merge through normal review and CI.

## Reproduction Assessment

Yes. Current main source shows `config.set`, `config.patch`, and `config.apply` respond with parsed/validated in-memory config while the writer computes a distinct persisted config after write-side transforms; I did not run a live RPC in this read-only review.

## Solution Assessment

No. The core approach is right for normal root writes, but the fallback leaves include fast-path writes returning pre-write config; the fix should carry a refreshed persisted/source config through that path before merge.
