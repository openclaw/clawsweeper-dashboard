## Summary

Keep open: current main still has only partial guardrails for this class of bad config. Codex-specific doctor warnings can catch disabled Codex runtime routes, and generic config validation rejects only statically suppressed model refs, but there is still no general analyzer tying active agent model refs, stale `agents.*.models` runtime-policy keys, enabled plugins/providers, and save/reload warnings together.

## What This Changes

The issue requests config and doctor validation that catches inconsistent agent model refs, runtime policy declarations, and plugin availability before later runtime model-resolution failure.

## Best Possible Solution

Add a shared model-route consistency analyzer surfaced through `doctor --lint` and config validation warnings, covering active refs, disabled provider/runtime plugins, stale runtime-policy keys, and missing explicit runtime policy without changing runtime fallback semantics by surprise.

## Reproduction Assessment

No for a live reproduction: this read-only review did not run the gateway or config writer. Source inspection does show a medium-confidence path where current validation checks static model suppressions and specialized Codex disabled-plugin warnings, but not the broader plugin/runtime consistency described here.

## Solution Assessment

Yes for the warning/lint direction: a shared analyzer reused by doctor and config validation is the narrow maintainable fix. Hard-failing config saves should remain a maintainer policy choice because aliases, custom providers, and forward-compatible model names can make strict validation noisy.
