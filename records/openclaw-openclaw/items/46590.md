## Summary

Keep open. Current main and v2026.6.10 still do not support declarative `agents.list[].cron`; current cron ownership is SQLite-backed and managed through CLI/Gateway APIs, while the related cron-isolation issue overlaps but does not own this distinct config-source request.

## What This Changes

The issue requests a declarative `cron` array under `agents.list[]` so scheduled tasks can be owned by agent configuration instead of only by the central cron job store.

## Best Possible Solution

Define one cron ownership and reconciliation model first; if declarative agent schedules belong in core, update the config schema, scheduler loading, doctor/docs, and focused tests together.

## Reproduction Assessment

Not applicable. This is a feature/config request, and source inspection confirms the requested `agents.list[].cron` field is absent rather than broken existing behavior.

## Solution Assessment

Unclear. `agents.list[].cron` is plausible, but it is not proven to be the best solution until maintainers choose the scheduler source of truth, reconciliation behavior, legacy-job handling, and operator/admin model.
