## Label Justifications

- `P2`: The issue reports a focused shipped/runtime bug where documented OpenAI Codex model params are ignored in a common configuration path.
- `impact:auth-provider`: The failure is caused by OpenAI model refs being resolved through an OpenAI-Codex runtime/auth provider route for parameter lookup.

## Summary

Keep open: current main still resolves model params and `/status` through the active `openai-codex` provider path, so canonical `openai/gpt-5.5` `textVerbosity` can be missed; the linked fix PR is open and not merged.

## What This Changes

The issue asks for canonical `openai/gpt-5.5` model params, especially `textVerbosity`, to be honored when the Codex runtime internally routes through `openai-codex`.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/84259 with a fix that uses the selected canonical `openai/*` config as the lookup source while preserving explicit legacy `openai-codex/*` compatibility.

## Reproduction Assessment

Yes. Source inspection shows current main can resolve params/status by `openai-codex/gpt-5.5`, miss the configured `openai/gpt-5.5` params, and then apply the GPT-5 `low` verbosity default.

## Solution Assessment

Yes. The maintainable fix is to separate runtime/auth provider routing from the selected model ref used for user-configured params, with a legacy `openai-codex/*` fallback for existing configs.
