## Summary

Keep open: current main still supports the reported source path where canonical `openai/gpt-5.5` model params are missed after Codex runtime routing switches the active provider to `openai-codex`, causing the GPT-5 default `low` verbosity to win.

## What This Changes

The issue asks OpenClaw to honor `textVerbosity` configured under canonical `openai/gpt-5.5` when the Codex runtime/auth path uses `openai-codex` internally.

## Best Possible Solution

Carry the selected canonical model ref as the config-param source for runtime/status lookup, while keeping `openai-codex/*` as an explicit legacy fallback for existing configs.

## Reproduction Assessment

Yes, from source: the documented `openai/gpt-5.5` Codex route can rewrite the active provider to `openai-codex`, and the resolver/status path then looks up `openai-codex/gpt-5.5` before defaulting GPT-5 text verbosity to `low`. I did not run a live OAuth session in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to separate the runtime/auth provider from the config-param source, using the selected canonical `openai/*` ref for user params and status with a legacy `openai-codex/*` fallback.
