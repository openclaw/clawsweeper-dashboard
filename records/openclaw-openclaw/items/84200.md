## Summary

Keep open: source review supports the reported bug. Current runtime routing can turn the documented `openai/gpt-5.5` selection into an `openai-codex/gpt-5.5` lookup for params/status, so the configured canonical model override is missed and the GPT-5 default `low` verbosity wins.

## What This Changes

Report that `agents.defaults.models["openai/gpt-5.5"].params.textVerbosity` is ignored when the canonical OpenAI model is executed through the native Codex runtime with Codex OAuth.

## Best Possible Solution

Keep the issue open and fix runtime/status parameter lookup to carry the selected canonical model ref as the config source, with a legacy `openai-codex/*` fallback for existing configs.

## Reproduction Assessment

Yes, from source: the documented `openai/gpt-5.5` Codex route can rewrite the active provider to `openai-codex`, and the resolver then looks up `openai-codex/gpt-5.5` before defaulting GPT-5 text verbosity to `low`. I did not run a live OAuth session in this read-only review.

## Solution Assessment

Yes, the reported fix direction is the right boundary: keep using the runtime provider for auth/transport, but use the selected canonical model ref for user-configured params and status, with explicit legacy fallback.
