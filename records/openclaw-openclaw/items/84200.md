## Summary

Keep open: current main still uses the active `openai-codex` provider for parameter/status lookup, so canonical `openai/gpt-5.5` `textVerbosity` can still be missed; the linked open fix PR should be reviewed and landed before closing this issue.

## What This Changes

The issue reports that `agents.defaults.models["openai/gpt-5.5"].params.textVerbosity` is ignored when the canonical OpenAI model is routed through Codex runtime auth.

## Best Possible Solution

Land a compatibility-preserving fix that uses the selected canonical `openai/*` model ref for configured params/status while keeping legacy `openai-codex/*` configs working, then close this issue through the linked PR.

## Reproduction Assessment

Yes, source-level: the documented `openai/gpt-5.5` Codex route can become an active `openai-codex` lookup, and `resolveExtraParams` then misses the canonical model params before defaulting GPT-5 verbosity to `low`. I did not run a live OAuth session in this read-only review.

## Solution Assessment

Yes, the issue’s suggested boundary is the maintainable fix: runtime/auth can use `openai-codex`, but config params and status need the selected canonical model ref with legacy fallback preserved. The linked open PR is the right implementation path to review rather than closing the issue early.
