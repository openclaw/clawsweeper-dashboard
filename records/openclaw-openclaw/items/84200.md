## Summary

Keep open: current `main` still resolves model params and `/status` verbosity from the active `openai-codex` runtime provider key, while the documented user config remains `openai/gpt-5.5`; the linked fix PR is open and should own the implementation path until it lands or is replaced.

## What This Changes

The issue reports that `agents.defaults.models["openai/gpt-5.5"].params.textVerbosity` is ignored when the canonical OpenAI model is routed through the Codex runtime/auth provider.

## Best Possible Solution

Review and land or replace the linked fix PR so config lookup uses the selected canonical OpenAI model key while transport/auth still use the Codex provider route, with regression coverage for `/status` and runtime params.

## Reproduction Assessment

Yes. Source inspection shows the documented `openai/gpt-5.5` Codex route can rewrite the active provider to `openai-codex`, after which param and status lookup use `openai-codex/gpt-5.5` and fall back to GPT-5 `low` verbosity.

## Solution Assessment

Yes. The reported fix direction is the maintainable boundary: keep the runtime/auth provider route separate from the model-config lookup key, and preserve legacy `openai-codex/*` fallback for old configs.
