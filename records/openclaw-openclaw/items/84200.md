The issue reports that canonical `openai/gpt-5.5` `textVerbosity` config is ignored when the model is routed through the native Codex runtime using `openai-codex` auth.

## Best Possible Solution

Review and land a compatibility-preserving fix that keeps `openai-codex` for auth/transport while resolving user-configured params and status from the selected canonical `openai/*` model ref, with legacy `openai-codex/*` behavior covered by tests.

## Reproduction Assessment

Yes, from source: canonical `openai/gpt-5.5` can be routed through the `openai-codex` runtime provider, and the extra-param/status lookup currently keys off that active provider before defaulting GPT-5 text verbosity to `low`. I did not run a live OAuth session in this read-only review.

## Solution Assessment

Yes, the maintainable solution is to fix the resolver/status boundary rather than changing docs or auth routing: use the selected canonical model ref for configured params while preserving legacy `openai-codex/*` compatibility.
