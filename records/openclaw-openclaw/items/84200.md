## Summary

Keep open: current main still supports the reported source-level mismatch where a documented canonical OpenAI model ref can be routed through the internal Codex provider before extra-param and status lookup, so the canonical `textVerbosity` override can be missed.

## What This Changes

The issue requests that `textVerbosity` set on `agents.defaults.models["openai/gpt-5.5"]` be honored for native Codex runtime sessions that authenticate through `openai-codex`.

## Best Possible Solution

Carry the selected canonical model ref as the config-source for extra params and status when runtime/auth routing changes the active provider to `openai-codex`, with an explicit legacy `openai-codex/*` fallback.

## Reproduction Assessment

Yes, from source: the documented `openai/gpt-5.5` Codex route can rewrite the active provider to `openai-codex`, and the resolver then looks up `openai-codex/gpt-5.5` before defaulting GPT-5 text verbosity to `low`. I did not run a live OAuth session in this read-only review.

## Solution Assessment

Yes, the right fix boundary is to keep runtime/auth transport on `openai-codex` while resolving user-configured params and status from the selected canonical model ref, with a legacy fallback for existing configs.
