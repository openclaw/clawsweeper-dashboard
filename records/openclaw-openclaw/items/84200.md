## Summary

This should stay open: current main still has a plausible provider/runtime alias gap where `openai-codex/gpt-5.5` parameter lookup does not fall back to the documented canonical `openai/gpt-5.5` model entry, and the native Codex app-server turn builder does not read or forward resolved text verbosity.

## What This Changes

The issue reports that `textVerbosity` configured under canonical `openai/gpt-5.5` is ignored when that model runs through Codex OAuth/native Codex routing.

## Best Possible Solution

Keep canonical `openai/*` model params as the source of truth across OpenAI Codex auth/runtime aliases, and add focused coverage for both status display and Codex/native or PI transport behavior.

## Reproduction Assessment

Yes, from source inspection: calling the shared resolver as `provider=openai-codex, model=gpt-5.5` cannot read params configured under `openai/gpt-5.5` and then falls back to `text_verbosity: low`. I did not perform a live Codex OAuth turn in this read-only review.

## Solution Assessment

Unclear at the transport edge until the Codex app-server protocol is checked, but the maintainable fix direction is clear: resolve params from the configured canonical model ref and only forward native verbosity through a verified app-server contract.
