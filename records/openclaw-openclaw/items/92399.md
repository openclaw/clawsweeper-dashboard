## Summary

The strict-extension heuristic is a reasonable generic fix for the captured provider corruption, and the updated head preserves equal, shrinking, phase-separated, reasoning-separated, and tool-separated messages. However, the shared LLM consumer emits `text_start` for each provisional snapshot block, removes that block when collapsing, and emits `text_end` for the earlier index, leaving an unmatched started content index in the event stream; this needs a focused repair before merge.

## What This Changes

The PR adds shared handling that collapses immediately adjacent same-phase OpenAI Responses message items when each later text strictly extends the previous text, applying it to both stream consumers and completed-response backfill with regression tests.

## Best Possible Solution

Keep the centralized immediate strict-extension collapse and newest exact-content item signature, but make the shared consumer's provisional block lifecycle balanced—either defer its public start until the item is known to be distinct or provide a supported replacement/cancellation sequence—and assert the complete emitted event order.

## Reproduction Assessment

Yes. The linked report supplies a concrete current-release command and raw cumulative SSE evidence, current main appends each completed item independently, and the PR's real local-agent replay demonstrates four copies before the patch and one after.

## Solution Assessment

No, not in its current form. The shared strict-extension compatibility seam is the narrowest maintainable direction because upstream exposes no snapshot marker, but the shared consumer must preserve a balanced public stream-event lifecycle.
