- `impact:data-loss`: A backup command that stalls or produces no completed archive can leave users without a usable restore artifact for their local OpenClaw state.

## Summary

Keep open. Current main and v2026.6.6 partially mitigate live log races with volatile-file filtering and retry logic, but the broader large-state backup problem is still not resolved: the CLI has no merged exclude/smart-exclude/media/browser/session-index controls, the docs still describe local size limits, and the linked closing PR remains open, conflicting, and proof/risk-blocked.

## What This Changes

Make `openclaw backup create` complete or fail clearly on multi-GB `.openclaw` state, with possible state-dir exclusion, progress, size-warning, or streaming improvements.

## Best Possible Solution

Core backup should get a proof-backed large-state strategy, either by landing a repaired exclusion/smart-exclude path or replacing it with a narrower maintainer-approved design that preserves restore-critical data; timeout temp cleanup should remain tracked by https://github.com/openclaw/openclaw/issues/50442.

## Reproduction Assessment

No high-confidence live reproduction was run for a 4 GB current-main install. Source inspection confirms the full backup still includes the whole state directory and lacks the requested exclude controls, while related merged work proves only a narrower live-log race mitigation.

## Solution Assessment

Unclear; the open linked PR is a plausible route but not yet the best proven fix because it is conflicting and marked for proof/merge-risk review. The safest path is maintainer review of that PR's public CLI/defaults/security shape or a narrower replacement that preserves restore-critical data.
