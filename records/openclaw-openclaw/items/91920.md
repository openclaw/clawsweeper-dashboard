## What This Changes

The issue reports weekly and daily cron jobs failing on OpenClaw 2026.6.5 after all configured MiniMax and BlockRun fallback candidates fail with overload or timeout errors.

## Best Possible Solution

Confirm that the jobs exhaust only genuinely unavailable providers and record that failure clearly; if recurring jobs should automatically retry transient provider outages, define that scheduling and duplicate-execution policy explicitly rather than treating normal fallback exhaustion as a scheduler bug.

## Reproduction Assessment

No, we do not have a high-confidence reproduction of a cron regression. We can reproduce the reported terminal state from source whenever every configured fallback candidate fails, but the report lacks the job and routing configuration needed to show behavior changed in OpenClaw 2026.6.5.

## Solution Assessment

No code change is currently justified. The narrow next step is to collect effective job/routing diagnostics; expanding automatic retries to recurring schedules would require a product decision because retries can duplicate side effects.
