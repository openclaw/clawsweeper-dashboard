## Summary

Keep this PR open, but it is not merge-ready: the object-format parser change is useful, while the URL-safe base64 fix still rejects normal unpadded base64url payloads and the external PR only supplies unit-test output rather than real proxy/runtime proof.

## What This Changes

The PR normalizes single-object OpenAI-compatible image response `data` into an array and broadens base64 canonicalization to accept URL-safe `-` and `_` characters, with parser and helper tests updated.

## Best Possible Solution

Complete padded and unpadded base64url normalization with focused regression tests, keep object-data normalization only if maintainers accept the SDK parser compatibility broadening, and leave URL-only or mixed-entry proxy behavior on the adjacent PRs.

## Reproduction Assessment

Yes at source level for the patch defect: the PR head still rejects unpadded base64url because `canonicalizeBase64` keeps the length-mod-4 check, while Node emits and decodes unpadded base64url such as `-w`. The original live NewAPI proxy failure was not reproduced here; the PR body supplies unit-test output only.

## Solution Assessment

No. The parser and helper locations are plausible, but padded-only URL-safe support is not the complete fix for base64url proxy responses; the helper should normalize unpadded base64url consistently or the PR should narrow its claim.
