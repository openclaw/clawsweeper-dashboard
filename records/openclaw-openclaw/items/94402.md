## Summary

Keep open: the PR is a focused and useful parser compatibility fix, but it is not merge-ready because the URL-safe base64 support still rejects common unpadded base64url payloads and the contributor proof is Vitest-only.

## What This Changes

The PR normalizes single-object OpenAI-compatible image response data into an array and allows `-`/`_` characters in shared base64 canonicalization.

## Best Possible Solution

Complete padded and unpadded base64url normalization, keep the object-data compatibility if maintainers accept the SDK behavior change, and leave URL-backed response downloads to https://github.com/openclaw/openclaw/pull/81443.

## Reproduction Assessment

Yes at source level: current main rejects strict non-array `data` and standard-only base64, and a Node probe shows the PR still rejects unpadded base64url. A live NewAPI payload was not provided or reproduced here.

## Solution Assessment

No. Object-data normalization is a reasonable narrow fix, but the URL-safe base64 implementation should also handle unpadded base64url before this can be considered the best repair.
