## Summary

Keep open: the PR targets a real OpenAI-compatible image parser gap, but its URL-safe base64 support is incomplete because common unpadded base64url payloads still fail the canonicalizer, and the external proof is only unit-test output rather than real behavior proof.

## What This Changes

The PR normalizes single-object OpenAI-compatible image `data` payloads into an array and expands the shared base64 canonicalizer to accept URL-safe `-` and `_` characters, with parser and base64 tests updated.

## Best Possible Solution

Complete base64url normalization for padded and unpadded inputs with focused regression coverage, keep object-data normalization only if maintainers accept the SDK parser broadening, and leave URL-only response downloading on https://github.com/openclaw/openclaw/pull/81443.

## Reproduction Assessment

Yes at source level for the patch defect: Node v24 emits unpadded base64url like `-w`, while the PR still rejects cleaned base64 lengths not divisible by 4. The original NewAPI proxy failure was not reproduced live here; the PR body provides unit-test output only.

## Solution Assessment

No. The parser/helper locations are plausible, but padded-only URL-safe support is not the narrow complete fix for base64url responses; the helper should normalize unpadded base64url or delegate to an equivalent decoder contract consistently.
