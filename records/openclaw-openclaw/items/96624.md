## Summary

Keep open: the status-scoped fallback is the right boundary, but the persisted lookup still returns the first matching stored snapshot, so completed calls with multiple persisted records can report stale state; the PR also still has only unit-test proof.

## What This Changes

The PR adds persisted voice-call lookup helpers and uses them as fallbacks for `voicecall.status`, the `voice_call` `get_status` action, and legacy status mode.

## Best Possible Solution

Keep the status-only persisted fallback, but make it return the newest/canonical stored snapshot by call id or provider call id, add a multi-snapshot regression test, and require redacted runtime proof for the completed-call status flow.
