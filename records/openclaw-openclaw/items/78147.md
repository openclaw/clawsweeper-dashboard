## Summary

Keep open: the lineage guard is a useful direction, but the patch currently mutates active streaming state while ignoring stale terminal events, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds OpenAI WebSocket response metadata typing, stale terminal-event lineage checks, and a regression test for stale `response.completed` handling across incremental turns.

## Best Possible Solution

Keep the PR open, preserve active streaming buffers when discarding stale terminal events, add coverage for stale terminals arriving after newer-turn deltas, then require real runtime proof before merge.

## Reproduction Assessment

No high-confidence live reproduction is present here. Source inspection makes the stale-terminal failure path reproducible in principle because current `main` accepts any terminal WebSocket response event as the active turn.

## Solution Assessment

No. The lineage check is the right narrow area, but ignored stale terminal events should leave active-turn streaming state untouched, and this PR needs real behavior proof before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve active output buffers when ignoring stale terminals:** `src/agents/openai-ws-stream.ts:1200-1202`
  - body: The stale-terminal branch returns without finalizing, but it first clears the output phase/text/emitted maps for the active WebSocket run. In the reported ordering, the newer turn may already have emitted `response.output_item.*` or `response.output_text.delta` before the stale `response.completed`; clearing these maps can drop phase state and cause later current-turn deltas or done events to be missed or re-emitted incorrectly. Ignoring a stale terminal event should not mutate the active turn's streaming buffers.
