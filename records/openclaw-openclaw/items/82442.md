## What This Changes

The issue asks OpenClaw to handle abrupt debug-proxy CONNECT client disconnects without crashing the process.

## Best Possible Solution

Land the paired PR or an equivalent narrow fix that handles errors from both tunnel sockets and verifies the disconnect path before closing the issue.

## Reproduction Assessment

Yes, at source level: current main has no clientSocket error listener after the CONNECT tunnel is piped, and Node treats unhandled error events as thrown process errors. A live proof would start the debug proxy, open a CONNECT tunnel, then abruptly reset the client.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: add a client socket error handler that records the failure and destroys the paired upstream socket, with symmetric cleanup on upstream errors.
