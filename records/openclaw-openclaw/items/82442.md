## What This Changes

The issue reports that abrupt client disconnects in debug proxy CONNECT tunnels can emit an unhandled client socket error and crash the process.

## Best Possible Solution

Review the paired PR as the fix path and require focused regression coverage or real debug-proxy disconnect proof before closing this issue.

## Reproduction Assessment

Yes from source inspection, but not from a live run: current main shows the CONNECT tunnel has no clientSocket error handler after piping sockets. A high-confidence live repro would start the debug proxy, open a CONNECT tunnel, abruptly reset the client, and observe the unhandled socket error path.

## Solution Assessment

Yes, the narrow maintainable fix is to add a client socket error handler that records the event and destroys the upstream socket, with symmetric cleanup for upstream errors. The linked PR appears to target that exact code path and should be reviewed directly.
