## What This Changes

The issue asks OpenClaw to handle abrupt client disconnect errors in the debug proxy CONNECT tunnel by adding a client socket error handler and destroying the paired upstream socket.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/82444 as the fix path, require real disconnect proof or focused regression coverage, and close this issue only after that PR lands or is replaced.

## Reproduction Assessment

Yes at source level: current main has no clientSocket error listener in the CONNECT tunnel, and Node throws unhandled 'error' events without a listener. A live repro would start the debug proxy, open a CONNECT tunnel, abruptly reset the client, and observe the unhandled socket error path.

## Solution Assessment

Yes, the narrow maintainable fix is to add a regular error handler for the accepted client socket and destroy the paired upstream socket on failure. The linked PR targets that exact path, so a separate ClawSweeper repair would duplicate active work.
