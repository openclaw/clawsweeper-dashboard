## Summary

Keep open: current main and v2026.6.6 contain the original CONNECT clientSocket fix from https://github.com/openclaw/openclaw/pull/82444, but this issue is now explicitly paired with open PR https://github.com/openclaw/openclaw/pull/88052 for a remaining same-class upstream response crash, so maintainers should resolve that PR or split the scope before closing.

## What This Changes

The issue reports that the debug proxy CONNECT tunnel lacked a clientSocket error handler and used clientSocket.end() on upstream socket errors, allowing abrupt disconnects to crash the proxy.

## Best Possible Solution

Keep the shipped CONNECT socket fix, then either land or refresh the narrow upstreamRes error-handler PR, or move that remaining scope to a separate canonical issue before closing this one.

## Reproduction Assessment

Yes for the remaining same-class upstream response path: current main still consumes upstreamRes without an error listener, and a local Node probe confirms unhandled error events throw. The original CONNECT clientSocket path itself is already fixed on current main and in v2026.6.6.

## Solution Assessment

Yes for the remaining crash class: add the error handler at the stream consumption point and keep the CONNECT socket fix already on main. The only process question is whether maintainers want https://github.com/openclaw/openclaw/pull/88052 to close this issue or to track that follow-up separately.
