## Summary

Keep open: current main still hardcodes the DM pairing pending TTL and max count, and this PR is not safe to merge as-is because the new parser accepts malformed integer prefixes while the change also alters security-sensitive pairing limits that docs still describe as fixed.

## What This Changes

The PR adds module-load environment overrides for DM pairing pending request TTL and maximum count in `src/pairing/pairing-store.ts`, plus a new Vitest resolver test file.

## Best Possible Solution

If OpenClaw wants operator-tunable DM pairing limits, land a stricter version that validates whole integer strings, updates pairing/security docs and the changelog, and adds store-level tests proving overridden TTL/max behavior; otherwise keep the fixed defaults and close this contributor feature after maintainer review.

## Reproduction Assessment

Yes. The parse defect is reproducible from the PR diff: setting an env value such as `3.7` or `3abc` reaches `Number.parseInt` and is treated as `3` instead of falling back to the documented default.

## Solution Assessment

No. The proposed direction may be acceptable, but this patch is not the best implementation until parsing is strict, docs/security posture are aligned, and maintainers approve the new bounds and env surface.
