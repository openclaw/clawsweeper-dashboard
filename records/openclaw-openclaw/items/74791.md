## Summary

Keep this PR open: current main still uses the fixed 1-hour DM pairing TTL and 3-request cap, while the proposed env override patch has a blocking parser mismatch and changes a security-sensitive pairing control without docs/changelog alignment or maintainer signoff.

## What This Changes

The PR adds module-load `OPENCLAW_PAIRING_PENDING_TTL_MINUTES` and `OPENCLAW_PAIRING_PENDING_MAX` overrides in `src/pairing/pairing-store.ts` plus a new resolver unit test file.

## Best Possible Solution

If maintainers want operator-tunable DM pairing limits, land a stricter version that validates full-string integers only, documents env names/defaults/bounds in pairing and security docs, adds the required changelog entry, and includes resolver plus store-level coverage for overridden TTL and cap behavior.

## Reproduction Assessment

Yes for the concrete review finding: the PR helper uses `Number.parseInt`, and the new test explicitly expects `3.7` to resolve to `3`. I did not run tests because this review was read-only, but the diff is sufficient to reproduce the parser behavior.

## Solution Assessment

No: this is not yet the best way to solve the feature request because the parser contradicts the stated fallback contract and the security-sensitive env surface is not documented. The safer path is strict integer parsing plus maintainer-approved bounds and docs.
