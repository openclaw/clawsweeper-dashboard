## Summary

Keep this PR open: the feature is not implemented on current main, but the proposed patch has a concrete parser bug and changes DM pairing security controls without maintainer/security signoff or matching docs.

## What This Changes

The PR adds module-load environment overrides for DM pairing pending-request TTL and max count, plus resolver unit tests.

## Best Possible Solution

Land this only if maintainers want operator-tunable DM pairing limits, with strict full-string integer parsing, approved bounds, pairing/security docs, a changelog entry, and tests for both resolver and store behavior under overridden values.

## Reproduction Assessment

Yes for the introduced parser bug: the PR diff and its new test show that `3.7` resolves to `3` through `Number.parseInt` instead of falling back. Current-main verification also shows the env override feature itself is not already present.

## Solution Assessment

No: this is not yet the best way to solve the requested tuning need because it contradicts its own integer-only contract and leaves security-facing docs stale. A safer version needs strict validation plus explicit maintainer/security approval of the env names and bounds.
