## Summary

Keep open: current main does not implement the requested operator tuning knobs, and the proposed patch has a concrete parser defect plus security/docs alignment work before it should merge.

## What This Changes

The PR adds module-load environment overrides for DM pairing pending request TTL and max count in `src/pairing/pairing-store.ts`, plus a new resolver unit test file.

## Best Possible Solution

If maintainers want operator-tunable DM pairing limits, land a stricter version that validates full-string integers only, documents env names/defaults/bounds in pairing and security docs, adds the required changelog entry, and covers both resolver behavior and store behavior under overridden TTL/max values.

## Reproduction Assessment

Yes for the concrete parser defect: the proposed helper accepts `OPENCLAW_PAIRING_PENDING_TTL_MINUTES=3.7` or similar numeric prefixes through `Number.parseInt` instead of falling back. No local tests were run because this was a read-only review, but the PR diff and its own test expectation show the behavior.

## Solution Assessment

No: this is not yet the best way to solve the feature request because the parser contradicts the stated contract and the security-sensitive env surface is not documented. The safer path is strict integer parsing plus explicit maintainer approval of the bounds and docs.
