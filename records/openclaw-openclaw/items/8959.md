## Summary

Keep open because current main and the latest release still expose only a single-select shared message presentation/message-tool contract. The closed dialog-tool PR and merged `before_agent_reply` hook are useful adjacent context, but neither implements the requested portable multi-select API, min/max semantics, or one structured selected-values callback across channels.

## What This Changes

Request to add a portable multi-select component to the message tool so users can select multiple options and submit one callback across supported channels.

## Best Possible Solution

Add an additive shared batch-selection contract to message presentation/message tool, export and document it through the public SDK, then implement channel render/callback handling plus a text fallback.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives a high-confidence current-main gap because the public message-tool and presentation contracts still define only single-select semantics.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable path is an explicit shared presentation/message-tool contract, not treating the closed dialog PR, Telegram-only callback helpers, or generic plugin interception as completion.
