## Summary

Keep open: current main and v2026.6.6 still expose single-select message presentation semantics only. The merged `before_agent_reply` hook and channel-native select primitives are useful adjacent infrastructure, but they do not provide the requested portable message-tool multi-select contract or selected-values callback shape.

## What This Changes

The issue requests an additive cross-platform multi-select contract for the message tool so agents can receive several selected values in one interaction.

## Best Possible Solution

Design one additive portable batch-select presentation/message-tool contract with typed selected-value aggregation, explicit per-channel limits, deterministic fallback text, and channel adapters for Slack, Discord, Telegram, and plain transports.

## Reproduction Assessment

Not applicable: this is a new-capability request, not a broken established behavior. Source and docs inspection give high confidence that the requested portable batch-select contract is absent on current main and v2026.6.6.

## Solution Assessment

No: closing this as implemented would confuse adjacent primitives with the requested shared contract. A one-channel selector or plugin-hook-only workaround is too narrow because the issue asks for portable message-tool UI, callback aggregation, and fallbacks.
