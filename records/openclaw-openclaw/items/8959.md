## Summary

Keep open because current main still has only a single-select shared message presentation/message-tool contract, while the requested portable batch multi-select API and callback semantics remain missing. The closed dialog-tool PR and newer plugin hooks are useful design context, but they do not implement the requested message-tool surface.

## What This Changes

The issue requests a cross-platform message-tool multi-select component so users can visually select multiple options and submit one callback across channels.

## Best Possible Solution

Add an additive shared batch-selection contract to message presentation/message tool, export and document it through the public SDK, then implement channel render/callback handling with a text fallback.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives a high-confidence current-main gap because the public message-tool and presentation contracts still define only single-select semantics.

## Solution Assessment

No current implementation or close path is sufficient. The maintainable path is an explicit shared API/product decision rather than treating single-channel callbacks, polls, or the closed dialog PR as completion.
