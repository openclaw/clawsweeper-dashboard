## Summary

Current main and v2026.6.5 still expose only a portable single-select message-presentation contract. The requested one-submit batch selection remains meaningful work, but its public SDK result shape, channel capability limits, and fallback behavior require maintainer product judgment before implementation.

## What This Changes

Add a portable message-tool multi-select control that submits all selected values in one interaction across supported channels.

## Best Possible Solution

Keep the existing single-select behavior unchanged and design a distinct additive batch-selection presentation type with an explicit selected-values result, capability-aware native channel mappings, and a documented fallback before implementation begins.

## Reproduction Assessment

Not applicable; this requests a new portable interaction capability, and current main behaves according to its documented single-select contract.

## Solution Assessment

Yes at the architectural level: the shared message-presentation SDK is the narrowest maintainable boundary, but the proposed channel table is not implementation-ready until maintainers define the result envelope, capability limits, and fallback semantics.
