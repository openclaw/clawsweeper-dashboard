## Summary

Keep open; current main still lacks a portable message-tool multi-select contract, and the related dialog PR plus merged plugin hook do not implement the requested shared SDK schema, renderer behavior, or selected-values callback.

## What This Changes

The issue requests a cross-platform multi-select option for the message tool so agents can collect several selected values in one user interaction.

## Best Possible Solution

Define an additive shared message presentation/message-tool batch-selection contract, export and document it, then implement native channel rendering, callback parsing, and text fallback together.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives a high-confidence gap check because current main still models only single-select message presentation in the tool schema, SDK export, docs, and portable renderers.

## Solution Assessment

Unclear until maintainers choose the public API shape and core/plugin boundary. If OpenClaw owns this UX in the message tool, the maintainable path is contract-first shared presentation work rather than a one-channel workaround.
