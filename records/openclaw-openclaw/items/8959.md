## Summary

Keep open; current main still does not provide the requested portable message-tool multi-select contract. The existing surface is single-select plus buttons, and the related dialog proposal at https://github.com/openclaw/openclaw/pull/18286 is closed unmerged rather than a canonical implementation path.

## What This Changes

Add a portable message-tool multi-select component that lets users select multiple options visually and submit one callback containing all selected values.

## Best Possible Solution

Define an additive batch-selection contract in the shared message presentation/message-tool boundary, export and document it, then implement channel render and callback handling for Slack, Discord, Telegram, and fallback text.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives a high-confidence gap check because current main still models the shared message presentation surface as single-select plus buttons.

## Solution Assessment

Unclear until maintainers choose the public API and callback semantics. The maintainable direction is contract-first shared presentation work rather than a channel-only native component patch.
