## Summary

Keep open: current main and v2026.6.8 still expose only a single-select portable message-presentation/message-tool contract. The merged before_agent_reply hook and native channel primitives are useful adjacent work, but they do not provide the requested cross-platform batch-select UI, min/max selection semantics, or structured selected-values callback.

## What This Changes

The issue requests a portable message-tool multi-select UI so users can choose multiple options visually and submit them in one callback.

## Best Possible Solution

Design one additive portable batch-select presentation/message-tool contract with typed selected-value aggregation, explicit channel limits, deterministic fallback text, docs, SDK exports, and channel adapter coverage.

## Reproduction Assessment

Not applicable: this is a new-capability request, not a broken established behavior. Source, docs, channel-renderer, related-PR, and v2026.6.8 inspection give high confidence that the requested portable batch-select contract is absent.

## Solution Assessment

No, not as an immediate automated fix: a one-channel selector, Telegram poll, or plugin-hook workaround is too narrow for the requested shared message-tool UI and callback aggregation. The maintainable path is an additive public contract with per-channel rendering and fallback semantics decided first.
