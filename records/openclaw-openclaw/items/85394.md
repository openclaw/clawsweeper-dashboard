## Summary

Keep open. The per-action cron split is the right direction and the supplied schema proof is useful, but the new `cron_*` tools still bypass existing side-effect/replay accounting and leave required job ids optional in model-facing schemas.

## What This Changes

This PR adds nine per-action cron tools alongside the legacy `cron` tool, wires them into agent and MCP tool creation, and updates catalog, policy, and regression tests for the new names.

## Best Possible Solution

Keep the per-action cron split and legacy alias, but land it only after the new cron names are fully integrated with mutation/replay accounting and their schemas require canonical job ids where execution requires them.

## Reproduction Assessment

Yes for the source-level schema problem: current `main` exposes the single flat `CronToolSchema`, and PR head narrows the action schemas. I did not reproduce the full live 33-call model retry loop against a real gateway/model.

## Solution Assessment

Not yet. The per-action split with a retained legacy alias is maintainable, but it is incomplete until `cron_*` side effects are wired into retry/replay accounting and job-specific schemas require the ids their executors need.

Full review comments:

- **[P1] Account for cron_* side effects before exposing them:** `src/agents/tools/cron-tool.ts:1170-1180`
  - body: By returning these new first-class `cron_*` tools, the PR moves scheduler mutations onto names that current side-effect accounting does not recognize. `tool-mutation.ts` only knows the legacy `cron` action path, and `successfulCronAdds` only increments for `toolName === "cron"` with `action: "add"`; a successful `cron_add` or `cron_update` can therefore look replay-safe after an incomplete turn. Please update mutation classification and cron-add tracking for the new family before merge.
  - confidence: 0.91
- **[P2] Require jobId in the per-action job schemas:** `src/agents/tools/cron-tool.ts:365-375`
  - body: The job-specific schemas still advertise `jobId` and compatibility `id` as optional even though the executors immediately throw when neither is present. For example, `cron_update` only requires `patch`, so models can emit a schema-valid call that the tool rejects and retries. Please require canonical `jobId` for get/update/remove/run/runs while keeping `id` as runtime compatibility if needed, and assert those required keys.
  - confidence: 0.84
