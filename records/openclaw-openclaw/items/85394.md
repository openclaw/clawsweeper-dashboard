## Summary

Keep open: the per-action cron decomposition is the right direction for the retry-loop bug, but this patch is not merge-ready because the new `cron_*` tool names bypass existing cron deny policies and the add/update schemas still advertise required payloads as optional.

## What This Changes

The branch refactors the cron agent tool into shared action handlers, adds nine `cron_*` per-action schemas/tools while keeping the legacy `cron` tool, wires them into agent and MCP tool lists, and adds schema/wiring tests.

## Best Possible Solution

Land a bounded version that keeps the narrow per-action cron schemas, preserves all existing `cron` policy/deny semantics across the new tool family, marks required per-action payloads correctly, and includes real schema-output proof before merge.

## Reproduction Assessment

Yes for the source-level schema problem: current main has one flat `CronToolSchema`, and the PR narrows the action schemas. I did not reproduce the live 33-call frontier-model retry loop against current main.

## Solution Assessment

No, not as currently patched. The decomposition is a maintainable fix direction, but it must preserve cron policy boundaries and correct required fields before it is the best solution.

- **[P1] Preserve cron denials for the new tool names:** `src/agents/openclaw-tools.ts:378`
  - body: Spreading `createCronTools()` exposes `cron_status`, `cron_update`, and the other per-action scheduler tools as separate core tools, but existing policy paths deny only the legacy `cron` name. `SUBAGENT_TOOL_DENY_ALWAYS` currently contains `cron`, and `isToolAllowedByPolicyName` matches the actual name/glob, so subagents or user configs with `tools.deny: ["cron"]` would still receive the new cron tools.   - confidence: 0.94
- **[P2] Require job and patch in per-action schemas:** `src/agents/tools/cron-tool.ts:351-364`
  - body: `CronAddSchema` and `CronUpdateSchema` reuse `CronJobObjectSchema`/`CronPatchObjectSchema`, but those are `Type.Optional(...)` for the legacy flat union. Split required inner schemas or assert the required keys for the per-action tools.
  - confidence: 0.88
