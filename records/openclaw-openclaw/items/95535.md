## Summary

Keep open: current main and v2026.6.9 still expose `sessions_yield` as a subagent/follow-up-event yield primitive, and the linked docs-only and runtime lifecycle PRs are open rather than merged. The remaining choice is a maintainer product contract decision, not a cleanup close.

## What This Changes

The issue reports that `sessions_yield` does not resume a main session after isolated cron completion and asks for either clearer tool/docs guidance or an explicit cron-completion wait/wake contract.

## Best Possible Solution

Keep the issue open until maintainers decide whether the docs-only clarification closes the report, the broader yielded-parent lifecycle PR should land, or a separate explicit cron wait/wake contract is needed; avoid implicit main-session wakes for every isolated cron completion without that decision.

## Reproduction Assessment

Yes, source-reproducible. Current main keeps `sessions_yield` as a subagent/follow-up-event yield primitive and persists yield context without triggering a turn, while isolated cron delivery does not provide a general parent-session wake path.

## Solution Assessment

Unclear. Docs/tool-text clarification is the lowest-risk mitigation, but a runtime wait/wake path would add or change a session/cron product contract that maintainers need to choose explicitly.
