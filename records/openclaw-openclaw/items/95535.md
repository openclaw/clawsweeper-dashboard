## Label Justifications

- `P2`: The issue affects a real cron/session workflow but is bounded to isolated cron completion semantics and tool guidance rather than a core outage.
- `impact:session-state`: The affected behavior is whether a yielded session receives follow-up state or remains idle until another inbound event arrives.

## Summary

Keep open: current main and v2026.6.9 still have the original `sessions_yield` wording and do not make isolated cron completion a parent-session wake event. The open wording PR is a candidate mitigation, but maintainers still need to decide whether docs-only closure is enough or whether a narrower cron wait/wake contract remains.

## What This Changes

The issue asks OpenClaw to clarify or extend `sessions_yield`/cron semantics so a yielded session does not wait indefinitely for isolated cron completion that only delivers through cron delivery channels.

## Best Possible Solution

Maintainers should decide the contract explicitly: land https://github.com/openclaw/openclaw/pull/95569 only if docs/tool-text clarification is accepted as the closure path, or keep runtime wait/wake semantics tracked under a narrower RFC or https://github.com/openclaw/openclaw/issues/92369.

## Reproduction Assessment

Yes by source inspection. Current main records `sessions_yield` with `triggerTurn: false`, main-target cron enqueues and wakes differently from detached isolated cron, and the isolated delivery path guards default delivery from main-session awareness.

## Solution Assessment

Unclear until maintainers choose the product contract. A docs/tool-description clarification is the narrowest low-risk mitigation, while a runtime wait selector or cron completion wake would define new session/cron semantics.
