## Label Justifications

- `P2`: The issue affects a real cron/session workflow but is bounded to isolated cron completion semantics and documentation rather than a core outage.
- `impact:session-state`: The affected behavior is whether a yielded main session receives or remains stuck waiting for isolated cron completion state.

## Summary

Keep open: current main and v2026.6.9 still have the reported yield/isolated-cron behavior, and the direct docs mitigation is an open draft PR rather than a merged fix. The remaining choice is maintainer-owned: land wording only, keep a runtime wait/wake API follow-up open, or explicitly decide that isolated cron completion should not resume yielded parent sessions.

## What This Changes

The issue requests clarification or a product/API path for `sessions_yield` after an isolated cron run completes without resuming the parent session.

## Best Possible Solution

Land the docs/tool-text clarification only if maintainers choose it as the intended mitigation, and track any explicit cron wait or parent-session wake API as a separate product contract rather than waking every isolated cron by default.

## Reproduction Assessment

Yes. Source inspection shows `sessions_yield` persists a hidden context with `triggerTurn: false`, while isolated cron completion follows a detached delivery path that does not generally wake the yielded parent session.

## Solution Assessment

Unclear. A docs/tool-description clarification is the lowest-risk mitigation, but an explicit cron wait or wake API would be a new product contract that maintainers need to choose deliberately.
