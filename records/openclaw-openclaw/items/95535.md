## Summary

Keep open: current main and v2026.6.9 still leave `sessions_yield` as a subagent/follow-up-event primitive, and isolated cron completion is not a general parent-session wake. The open docs-only PR is the right candidate to review, but it has not merged and maintainers still need to decide whether docs-only closure is enough or a runtime wait/wake API remains.

## What This Changes

The issue asks OpenClaw to clarify or change `sessions_yield` behavior so agents do not wait indefinitely for isolated cron completion events that never resume the yielded session.

## Best Possible Solution

Review and either land or revise the linked docs-only PR, then have maintainers explicitly decide whether a separate runtime wait/wake API remains necessary.

## Reproduction Assessment

Yes, source-reproducible. Current main persists the yield context without triggering a turn, while isolated cron completion does not generally enqueue a parent-session wake.

## Solution Assessment

Unclear. A docs/tool-description clarification is the lowest-risk mitigation, but a runtime wait selector or cron completion wake API would define new session/cron behavior and needs maintainer selection.
