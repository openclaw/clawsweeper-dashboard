## Summary

Keep open: current main and v2026.6.9 still have the subagent-oriented `sessions_yield` wording and the isolated cron path does not provide a first-class parent-session wait/resume contract. The linked docs PR is an open candidate mitigation, but it should not close the broader product-contract question until maintainers decide whether docs-only is enough.

## What This Changes

The issue asks OpenClaw to clarify that `sessions_yield` does not wait for isolated cron completion, or to add an explicit cron completion wait/resume contract.

## Best Possible Solution

Land a clear docs/tool-text mitigation if maintainers accept it, and track any first-class cron wait or completion notification API as an explicit product decision rather than an implicit main-session wake.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `sessions_yield` records a yield without a cron wait selector, and detached isolated cron completion does not follow the same immediate heartbeat path as main-session cron jobs. The issue also includes a concrete live run timeline, though this review did not rerun that channel scenario.

## Solution Assessment

Unclear. A docs/tool-description clarification is the narrowest low-risk mitigation, while a runtime wake or wait selector would create a new cron/session contract that maintainers need to choose explicitly.
