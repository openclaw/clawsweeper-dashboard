## Label Justifications

- `P2`: The issue affects a real cron/session workflow but is bounded to isolated cron completion semantics and documentation rather than a core outage.
- `impact:session-state`: The issue is about whether a yielded main session receives or remains stuck waiting for isolated cron completion state.

## Summary

Keep open: current main and the latest release still have the subagent-oriented `sessions_yield` contract, isolated cron completion is not a general parent-session wake, and the linked docs-only PR is still open with closing syntax, so maintainers need to decide whether that mitigation is enough or a runtime wait contract remains.

## What This Changes

The issue reports that `sessions_yield` can leave a main session waiting indefinitely after an isolated cron run completes, and asks for clearer tool/docs guidance or an explicit cron completion wake/wait contract.

## Best Possible Solution

Land a maintainer-approved tool/docs clarification if that is the chosen mitigation, and track any explicit cron completion wait/wake API separately; avoid implicit parent wakes for all isolated cron completions.

## Reproduction Assessment

Yes, source-reproducible. Current main records `sessions_yield` with `triggerTurn: false`, tests show the yielded parent is idle, and isolated cron completion follows delivery paths that do not generally wake the parent session.

## Solution Assessment

Unclear. A docs/tool-description clarification is the lowest-risk mitigation and already has a candidate PR, while an explicit cron completion waiter or wake selector is a new product/API contract that maintainers need to choose deliberately.
