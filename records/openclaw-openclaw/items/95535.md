## Label Justifications

- `impact:session-state`: The affected behavior is whether a yielded main session receives or remains stuck waiting for the isolated cron completion event.

## Summary

Keep open: current main still makes `sessions_yield` a cooperative end-turn primitive and does not provide a cron-completion wait or immediate main-session wake contract for isolated cron runs. The issue is not obsolete; the remaining maintainer choice is whether to clarify the model-facing docs/tool text or add an explicit cron completion notification API.

## What This Changes

The issue asks OpenClaw to clarify or change how `sessions_yield` behaves when a main session tries to wait for an isolated cron run to complete.

## Best Possible Solution

Clarify `sessions_yield` as a subagent/runtime-event turn-ending primitive and document cron completion waiting through `cron.runs` or CLI `--wait`; add main-session cron completion wake behavior only as an explicit designed contract.

## Reproduction Assessment

Yes, source-reproducible: `sessions_yield` only records yield intent and relies on a later event, while isolated cron completion does not request an immediate parent-session wake. No live cron run was executed in this read-only review.

## Solution Assessment

No, using bare `sessions_yield` to wait for isolated cron completion is not currently the supported mechanism. The narrow maintainable path is docs/tool-description clarity unless maintainers intentionally add a new cron completion wake option.
