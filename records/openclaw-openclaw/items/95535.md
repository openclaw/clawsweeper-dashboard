## Label Justifications

- `P2`: The issue affects a real cron/session workflow but is bounded to isolated cron completion semantics and a product contract decision rather than a core outage.
- `impact:session-state`: The affected behavior is whether a yielded parent session receives state needed to resume after an isolated cron run completes.

## Summary

Keep open: current `main` still has the original `sessions_yield` description and no isolated-cron completion wake selector, while the docs mitigation is still an open PR and the runtime API question is tracked separately in https://github.com/openclaw/openclaw/issues/96190.

## What This Changes

The issue asks OpenClaw to clarify that `sessions_yield` does not resume on isolated cron completion, or to add an explicit cron-completion wake/wait contract.

## Best Possible Solution

Review and land or reject the narrow docs clarification in https://github.com/openclaw/openclaw/pull/95569, and keep the opt-in runtime wake/API decision in https://github.com/openclaw/openclaw/issues/96190 instead of adding implicit isolated-cron fan-out.

## Reproduction Assessment

Yes, at source level: `sessions_yield` records a yielded turn and needs a later event, while the isolated cron path still does not generally enqueue a parent-session completion wake. I did not run a live channel cron reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the contract. The docs-only PR is the narrowest mitigation, while a runtime wake/wait selector or `notifyOnCompletion` option is a new API/config decision better tracked by the open RFC.
