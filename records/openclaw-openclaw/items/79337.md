## Summary

Keep open: this is an active maintainer-labeled feature PR, and the latest head adds a blocking approval-routing bug for shared ACP session keys.

## What This Changes

The PR adds ACP exec approval relay support, Gateway tool-event subscription for `openclaw acp`, focused relay tests, ACP docs, and a changelog entry.

## Best Possible Solution

Keep the relay design, but bind early approval requests only when they are run-scoped or uniquely attributable; otherwise wait for the run-scoped agent approval event before resolving exec approval.

## Reproduction Assessment

Not applicable for the PR as a feature. The review finding is source-reproducible: raw `exec.approval.requested` events carry only `sessionKey`, and the new handler can select the first pending prompt when no run id is present.

## Solution Assessment

No; the main permission relay is narrow, but the early approval-request path should not resolve an exec approval through an arbitrary pending prompt for the same shared Gateway session key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Guard unscoped approval events for shared sessions:** `src/acp/translator.ts:1236`
  - body: The new early `exec.approval.requested` path only has a `sessionKey`, so `startApprovalRelay` calls `findPendingBySessionKey` without a run id. If two ACP sessions are attached to the same Gateway session, this binds the approval to the first pending prompt and records the approval id, so the later run-scoped agent approval event cannot correct it; one client can approve or deny another client's exec. Please wait for the run-scoped event or require a unique matching pending prompt before relaying.
  - confidence: 0.86

## Security Review
