## Summary

Keep open. The PR addresses a real, still-unimplemented silent-reply leak and runner retry/warning path, but the proposed runner change references an `AssistantMessage` type that is not imported or locally aliased, so the patch is not merge-ready.

## What This Changes

The PR adds reasoning-prefaced `NO_REPLY` classification in auto-reply token handling and updates embedded-runner incomplete-turn guards/tests so stripped silent replies do not trigger retries or user-visible failure warnings.

## Best Possible Solution

Resolve the missing runner type import or replace it with an existing local indexed type, then land the narrow end-to-end suppression with token and runner regression coverage while preserving the #19537 substantive-reply contract.

## Reproduction Assessment

Yes. The PR body provides the exact production payload, and current main statically reproduces the failure because the auto-reply gate does not classify `think\n...NO_REPLY` as silent and the runner silent check ignores `lastAssistant.content` when `assistantTexts` is empty.

## Solution Assessment

No, not as written. The behavioral direction is narrow and maintainable, but the unresolved `AssistantMessage` type makes the patch fail typechecking; importing the type from `@mariozechner/pi-ai` or using `EmbeddedRunAttemptResult["lastAssistant"]` is the safer merge-ready form.
