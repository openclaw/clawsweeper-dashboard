## Summary

Keep open. The PR targets a valid heartbeat persistence bug, but the current head still writes heartbeat runtime state for empty prior entries and can pair a model-only session entry with the heartbeat provider; the linked issue should remain open until one consolidated branch fixes those edge cases.

## What This Changes

The PR adds a heartbeat-only `preserveRuntimeModel` path in session persistence, wires it from `agent-command`, adds session-store regression tests, and adds a changelog entry.

## Best Possible Solution

Land one heartbeat persistence fix that treats heartbeat model selection as run-scoped, preserves only an existing coherent runtime selection, leaves empty and model-only session state intact, and consolidates the duplicate implementation branch after merge.

## Reproduction Assessment

Yes. Configure `agents.defaults.heartbeat.model` differently from the primary model, run a heartbeat, and inspect `session_status` or the session store; current main writes the heartbeat run provider/model into shared runtime fields.

## Solution Assessment

No. A heartbeat-specific preservation hook is the right boundary, but this implementation must not seed empty sessions with heartbeat run metadata or combine a preserved model with the heartbeat provider.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Leave empty heartbeat entries without a runtime model:** `src/agents/command/session-store.ts:128-135`
  - body: When `preserveRuntimeModel` is true and the stored entry has no prior runtime model, this branch records the heartbeat run's provider/model/context as shared session state.   - confidence: 0.9
