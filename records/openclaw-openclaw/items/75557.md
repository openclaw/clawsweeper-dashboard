## Summary

Keep open. The PR is pointed at a valid, narrow bug and the chosen direction is close, but the patch still falls back to heartbeat-run metadata when the session has incomplete prior runtime metadata, which can leave the smaller heartbeat model/window visible after the heartbeat. It also lacks the required user-facing changelog entry for this fix.

## What This Changes

The PR adds a heartbeat-only `preserveRuntimeModel` flag from `agent-command` into `updateSessionStoreAfterAgentRun` and adds session-store unit tests for preserving, falling back, and overwriting runtime model/context metadata.

## Best Possible Solution

Keep heartbeat model selection as a per-run override, but make heartbeat persistence leave shared-session runtime model/context state on the configured primary or previously known non-heartbeat runtime state. Cover the shared-session heartbeat path through status/list behavior, update the stale heartbeat overflow docs if the mitigation is no longer needed, and add the required changelog entry before merge.

## Reproduction Assessment

Yes. Configure shared-session `heartbeat.model` differently from the primary, run a heartbeat, and inspect the session store/status path; current main writes the heartbeat runtime model/context and status reads that runtime identity when no explicit override exists.

## Solution Assessment

No. Preserving prior runtime metadata is the right direction, but falling back to the heartbeat run's model/context when prior metadata is absent or incomplete leaves part of the reported bleed in place; the safer fix is to recompute or omit context for the preserved primary/runtime model instead of using heartbeat metadata.
