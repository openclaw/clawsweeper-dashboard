## Summary

Keep open. The PR targets a real heartbeat model-bleed path on current main, but the latest diff still persists heartbeat runtime metadata for empty sessions and can cross-pair a preserved model with the heartbeat provider for model-only session entries.

## What This Changes

The branch adds a heartbeat-only `preserveRuntimeModel` option to session persistence, wires it from agent heartbeat runs, adds session-store regression tests, and adds a changelog fix entry.

## Best Possible Solution

Land one consolidated heartbeat persistence fix that treats heartbeat model selection as run-scoped, preserves only a valid prior runtime tuple, leaves empty prior runtime fields unset, and then closes the duplicate branch after merge.

## Reproduction Assessment

Yes. Configure `agents.defaults.heartbeat.model` differently from the primary model, run a heartbeat, and inspect `session_status` or the session store; current main writes the heartbeat run provider/model into the shared session runtime fields.

## Solution Assessment

No. A heartbeat-specific preservation hook is the right boundary, but this implementation must avoid writing the heartbeat model when there is no prior runtime selection and must not combine a preserved model with the heartbeat provider.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not seed empty sessions with the heartbeat model:** `src/agents/command/session-store.ts:128-134`
  - body: When `preserveRuntimeModel` is true and the stored entry has no prior runtime model, this branch records the heartbeat run's provider/model/context as the session runtime state. A first heartbeat on an otherwise default session would still make status prefer the heartbeat model instead of falling back to the configured primary, leaving the bleed unresolved for empty entries.
  - confidence: 0.87
- **[P2] Do not borrow the heartbeat provider for model-only entries:** `src/agents/command/session-store.ts:132-134`
  - body: A stored runtime model is valid without `modelProvider`; status can infer or preserve that form. Here the old model is kept while `provider` falls back to the heartbeat run, so a model-only entry can be persisted as an invalid cross-provider pair such as `ollama/claude-opus-4-6`.
  - confidence: 0.9
