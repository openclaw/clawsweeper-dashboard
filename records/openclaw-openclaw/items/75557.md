## Summary

Keep open. The linked heartbeat bleed bug is valid on current main, but this PR is not mergeable as-is because it can persist an invalid preserved-model/heartbeat-provider pair and it removes unrelated changelog history; the related #75567 branch is also open and dirty, so it is source material rather than a clean superseding fix.

## What This Changes

This PR adds a heartbeat-only `preserveRuntimeModel` path to session persistence, wires it from heartbeat runs, adds session-store regression tests, and changes `CHANGELOG.md` while deleting unrelated release-note history.

## Best Possible Solution

Land one rebased heartbeat persistence fix that preserves existing runtime model/provider/context metadata as a coherent state, covers model-only entries, keeps unrelated changelog history intact, and then resolves the linked bug report.

## Reproduction Assessment

Yes. Configure a shared session with `heartbeat.model` different from the primary model, run a heartbeat, then inspect `session_status` or the session store; current main writes the heartbeat run's provider/model/context into the shared entry. The PR-specific regression is reproduced by a stored entry with `model` but no `modelProvider` and a heartbeat result from another provider.

## Solution Assessment

No. A heartbeat-specific preserve path is the right direction, but this implementation still uses the heartbeat provider when preserving a model-only entry and carries a destructive changelog rewrite; preserving or resolving provider/model together is the safer fix.
