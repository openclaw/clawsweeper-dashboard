## Summary

Keep open. Current main still does not surface a `chat.send` idempotency key through the hydrated `chat.history` `__openclaw` metadata, and the request is a narrow additive gateway protocol/history improvement rather than obsolete cleanup.

## What This Changes

The issue requests that hydrated `chat.history` user messages include the original `chat.send` `idempotencyKey` for client-side optimistic-message correlation.

## Best Possible Solution

Add an additive gateway history contract that persists the chat-send key for user transcript rows and hydrates it as `__openclaw.idempotencyKey`, with focused regression coverage and docs for client correlation semantics.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the current `chat.history` path returns projected transcript messages, while the chat-send user transcript helper and metadata attachment path do not place `idempotencyKey` in `__openclaw`. I did not run a live gateway scenario because this review is read-only and `pnpm` is unavailable in the environment.

## Solution Assessment

Yes, the narrow maintainable path is an additive persistence/projection change for chat-send user turns, plus protocol/docs/tests. It should not be folded into the broader author-attribution/trust-model proposal.
