## Summary

Keep open: current main still omits the persisted key from the history metadata envelope, and the linked closing PR is still open, conflicting, and previously reviewed as needing real behavior proof plus a focused SSE-path fix.

## What This Changes

The issue asks the Gateway to include persisted `chat.send` `idempotencyKey` values in `chat.history` user-message `__openclaw` metadata for client correlation.

## Best Possible Solution

Land a corrected gateway change that exposes the existing per-message key as `__openclaw.idempotencyKey` for history hydration and live session-history/broadcast paths, with focused regression and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. A transcript record whose `message` contains `idempotencyKey` reaches `parsedSessionEntryToMessage`, which only copies `id` and `seq`, and `chat.history` uses that hydrated output.

## Solution Assessment

Yes in direction: echoing the existing persisted key through the history metadata envelope is the narrow maintainable solution. The current linked PR still needs its SSE path aligned with the real `chat.send` update shape and real after-fix proof before merge.
