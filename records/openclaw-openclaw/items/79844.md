Keep open: current `main` still omits the persisted `idempotencyKey` from the `chat.history` `__openclaw` envelope, and an open closing PR already targets the issue, so it should remain open until that PR or a replacement merges.

## What This Changes

Expose a persisted `chat.send` `idempotencyKey` as `__openclaw.idempotencyKey` on user messages returned by `chat.history` for client-side correlation.

## Best Possible Solution

Land a focused additive gateway fix that hydrates the existing per-message key into `__openclaw.idempotencyKey` for history and live history paths, with regression coverage and real behavior proof.

Yes. Source inspection gives a high-confidence path: a transcript row with `message.idempotencyKey` reaches `parsedSessionEntryToMessage`, but current `main` only attaches `id` and `seq` before `chat.history` returns the projected messages.
