## Summary

Keep open: current main still does not hydrate `idempotencyKey` into the `chat.history` `__openclaw` metadata envelope, and this is a narrow additive Gateway protocol enhancement rather than a stale, duplicate, or out-of-repo cleanup item. The related companion-SQLite issue is broader and does not supersede this specific correlation gap.

## What This Changes

The issue requests that `chat.history` include the persisted `idempotencyKey` in hydrated user-message `__openclaw` metadata for client-side correlation.

## Best Possible Solution

Persist the `chat.send` key on the canonical user transcript record when needed, then expose it additively as `__openclaw.idempotencyKey` in `chat.history` with focused Gateway tests and protocol docs.

## Solution Assessment
