Keep open: current main still does not place the chat-send idempotency key into the `__openclaw` metadata returned by `chat.history`, and the request is a narrow additive gateway protocol enhancement rather than a cleanup close.

## What This Changes

Request to echo `chat.send` `idempotencyKey` on hydrated user messages returned by `chat.history` for client-side optimistic-message correlation.

## Best Possible Solution

Add an additive gateway history change that preserves the chat-send idempotency key on stored user rows when needed and exposes it as `__openclaw.idempotencyKey` during history hydration, with focused RPC/hydration tests and protocol docs.

Yes. Source inspection gives a high-confidence path: hydrate a transcript user message that has `message.idempotencyKey`, and the current `__openclaw` envelope receives only transcript id/seq before `chat.history` returns it.

## Solution Assessment

Yes. Echoing the existing per-message correlation key through the history hydration layer is the narrowest maintainable direction; if the write path no longer stores it, the same focused change should preserve it at the canonical user-transcript append/update point first.
