## Summary

Keep open: current main still hydrates transcript messages without copying a persisted `idempotencyKey` into the `__openclaw` history envelope, and the related open SQLite companion issue is broader rather than a canonical fix for this narrower Gateway metadata gap.

## What This Changes

Expose a persisted `chat.send` idempotency key as `__openclaw.idempotencyKey` on user messages returned by `chat.history`.

## Best Possible Solution

Add the additive Gateway history metadata field during transcript hydration, preserve backward compatibility, and cover it with focused hydration/RPC tests plus protocol/WebChat docs.

## Reproduction Assessment

Yes. Source inspection on current main shows that a transcript message with `message.idempotencyKey` would be hydrated with only `id` and `seq` in `__openclaw`, and `chat.history` uses that hydration path.

## Solution Assessment

Yes. Copying the existing per-message key into `__openclaw.idempotencyKey` at the Gateway hydration boundary, after verifying persistence, is the narrowest maintainable additive fix.
