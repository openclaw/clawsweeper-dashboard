## Summary

Keep open: current main still lacks a history metadata path that returns the client-supplied idempotency key, and duplicate search did not find another item covering this exact chat.history correlation field.

## What This Changes

The issue requests exposing chat.send idempotencyKey values on hydrated chat.history user-message metadata for client-side optimistic-message correlation.

## Best Possible Solution

Add a narrow additive Gateway change that persists the chat.send key on the durable user transcript row if needed and exposes it as __openclaw.idempotencyKey during history hydration, with protocol/docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: hydrate a transcript user message that has message.idempotencyKey and chat.history will attach only id/seq metadata under __openclaw, with no idempotencyKey copy.

## Solution Assessment

Yes. Echoing the existing client-supplied key through the transcript append/hydration path is the narrowest maintainable solution; it should not change dedupe semantics or add a config option.
