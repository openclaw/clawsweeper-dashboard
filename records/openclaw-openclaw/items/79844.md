## Summary

Keep open: current main still hydrates transcript messages without carrying a stored `idempotencyKey` into `__openclaw`, and the broader companion-seam issues do not replace this narrow Gateway history contract.

## What This Changes

Expose `chat.send` idempotency keys in `chat.history` user-message `__openclaw` metadata so clients can correlate optimistic bubbles with persisted messages.

## Best Possible Solution

Add a small additive Gateway history contract that copies a stored string `message.idempotencyKey` into `__openclaw.idempotencyKey` for hydrated user messages, with focused hydration/RPC tests and protocol docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a transcript message containing `message.idempotencyKey` will be hydrated by `parsedSessionEntryToMessage` with only `id` and `seq` metadata before `chat.history` returns it.

## Solution Assessment

Yes. Echoing the existing per-message key during history hydration is the narrowest maintainable solution; the broader SQLite companion seams should not be required for current WebSocket clients to correlate sends.
