## Summary

Keep open: current main still drops persisted user-message idempotency metadata during chat.history hydration, so clients cannot correlate optimistic bubbles to confirmed transcript rows from history.

## What This Changes

Surface a persisted chat.send idempotencyKey on hydrated user messages returned by chat.history for client-side correlation.

## Best Possible Solution

Add additive Gateway history metadata propagation that carries persisted user-message idempotency keys into __openclaw.idempotencyKey, with focused hydration/RPC coverage and protocol docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: hydrate a transcript record with message.idempotencyKey and parsedSessionEntryToMessage will attach only id and seq before chat.history returns the projected message.

## Solution Assessment

Yes. Echoing the existing per-message key through the history hydration layer is the narrowest maintainable solution; while fixing, preserve or add the key at the canonical chat.send transcript append/update point if current main no longer writes it.
