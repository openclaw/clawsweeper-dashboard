## Summary

Keep open: current main and the latest release still hydrate transcript metadata with `id`/`seq` only, so the requested `idempotencyKey` is still missing from `chat.history` `__openclaw`. An open linked PR already targets this exact gap, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue requests that `chat.history` include the persisted `idempotencyKey` inside hydrated user-message `__openclaw` metadata for optimistic-message correlation.

## Best Possible Solution

Review and either land or close the linked PR; the desired implementation should surface `idempotencyKey` in the `__openclaw` envelope for full history, recent-tail history, and live session-message paths without removing existing top-level fields.

## Reproduction Assessment

Yes, at source level: a transcript message can contain a top-level `idempotencyKey`, but the current hydrate and inline append paths add only `id`/`seq` to `__openclaw`. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Yes, the proposed direction is narrow: copy the already-persisted key into the existing metadata envelope at hydration/broadcast boundaries. The open linked PR should be the first review target rather than starting a second fix lane.
