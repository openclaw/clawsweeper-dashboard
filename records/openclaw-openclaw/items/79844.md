## Summary

Keep open: current main and the latest release still persist the chat-send user `idempotencyKey` but do not copy it into hydrated or live `__openclaw` history metadata, and the open linked PR is the active fix candidate rather than a merged fix.

## What This Changes

The issue requests exposing persisted `idempotencyKey` inside hydrated `chat.history` user-message `__openclaw` metadata for client correlation.

## Best Possible Solution

Merge a narrow Gateway projection fix that carries the persisted user-message `idempotencyKey` into `__openclaw` for full history reads, bounded/recent reads, live history appends, and `session.message` broadcasts.

## Reproduction Assessment

Yes from source: `chat.send` persists the user-turn idempotency key, while current history hydration, live-history append, and session-message metadata omit it from `__openclaw`. I did not run a live repro because this review is read-only.

## Solution Assessment

Yes, the narrow hydrate/projection-side fix is the right owner boundary: the data is already persisted, so Gateway history should surface it without adding config, storage migration, or client-side matching heuristics.
