## Summary

Keep open: current `main` still does not put a persisted user-message `idempotencyKey` into the hydrated `__openclaw` envelope returned by `chat.history`, and the related companion-seam thread treats this as a narrower prerequisite rather than a duplicate or completed fix.

## What This Changes

The issue requests that `chat.history` include a persisted user-message `idempotencyKey` in the hydrated `__openclaw` metadata envelope for client-side correlation.

## Best Possible Solution

Add an additive Gateway history change that exposes persisted user-message idempotency keys as `__openclaw.idempotencyKey`, preserving the key at the canonical chat.send transcript write path if needed, with focused tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: hydrate a transcript message containing `message.idempotencyKey` and `parsedSessionEntryToMessage` will only add `id` and `seq` under `__openclaw` before `chat.history` returns it.

## Solution Assessment

Yes. Echoing the existing per-message key through the history hydration layer is the narrowest maintainable solution; if the current write path drops the key, preserve it at the user-transcript append point first.
