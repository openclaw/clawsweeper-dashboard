## Summary

Keep open: current main still omits a persisted user-message `idempotencyKey` from the `chat.history` `__openclaw` envelope, and the linked fix PR at https://github.com/openclaw/openclaw/pull/80044 is still open and unmerged.

## What This Changes

Expose `chat.send` idempotency keys inside hydrated `chat.history` user-message `__openclaw` metadata for client-side optimistic message correlation.

## Best Possible Solution

Land a focused Gateway fix, likely through the linked PR after review fixes, that hoists persisted user-message keys into `__openclaw.idempotencyKey` for history reads and live session-history events.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a transcript row with `message.idempotencyKey` reaches `parsedSessionEntryToMessage`, which currently attaches only `id` and `seq` before `chat.history` returns the projected messages.

## Solution Assessment

Yes. Echoing the existing per-message key at the Gateway hydration and inline event boundary is the narrow maintainable direction; the issue should stay open until https://github.com/openclaw/openclaw/pull/80044 or an equivalent fix merges.
