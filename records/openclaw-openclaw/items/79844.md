## Summary

Keep open: current main still omits `__openclaw.idempotencyKey` from hydrated `chat.history` messages, and an open closing PR specifically targets this issue, so the issue should remain open until that PR is landed, replaced, or closed.

## What This Changes

Expose persisted `chat.send` idempotency keys on hydrated `chat.history` user-message metadata for client-side correlation.

## Best Possible Solution

Land or replace the targeted Gateway fix so the persisted key is exposed consistently in history hydration and any live history event path, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a transcript row containing `message.idempotencyKey` flows through `parsedSessionEntryToMessage`, which only adds `id` and `seq` to `__openclaw`, and `chat.history` returns those hydrated messages.

## Solution Assessment

Yes. Echoing the existing per-message idempotency key through the Gateway history metadata envelope is the narrow additive solution; the open closing PR should be reviewed or superseded rather than closing this issue early.
