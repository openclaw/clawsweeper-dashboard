## Summary

Keep open: current main's history hydrator reads transcript messages but only attaches transcript id/seq metadata, so a persisted user-message idempotencyKey cannot reach chat.history. This is a narrow additive protocol enhancement, not a stale or duplicate cleanup close.

## What This Changes

Expose chat.send idempotencyKey on hydrated chat.history user-message __openclaw metadata for client-side optimistic-message correlation.

## Best Possible Solution

Add an additive Gateway history change that carries persisted chat.send idempotency keys into __openclaw.idempotencyKey on user rows, with focused hydration/RPC tests and protocol docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: hydrate a transcript user message with message.idempotencyKey and chat.history will return only the existing __openclaw id/seq metadata.

## Solution Assessment

Yes. Echoing the existing per-message key through the history hydration layer is the narrowest maintainable solution; if current chat.send no longer persists it, the same focused fix should preserve it at the canonical user-transcript append point first.
