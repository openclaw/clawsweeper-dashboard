## Summary

Keep open: current main still lacks a `chat.history` response path that places a persisted user-message idempotency key in `__openclaw`, and the related broader companion-transcript issue does not replace this narrow Gateway API gap.

## What This Changes

Expose `chat.send` idempotency keys on hydrated `chat.history` user-message metadata so clients can correlate optimistic and persisted messages.

## Best Possible Solution

Add an additive Gateway history change that preserves the key at the user transcript append point when needed and returns it as `__openclaw.idempotencyKey` on user rows, with focused hydration/RPC tests and protocol docs or changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a transcript user message with `message.idempotencyKey` reaches `parsedSessionEntryToMessage`, which only adds `id` and `seq` to `__openclaw`, and `chat.history` returns that projected message.

## Solution Assessment

Yes. The narrowest maintainable solution is to echo the existing per-message key through the Gateway transcript hydration path, while also ensuring the current chat.send user transcript append path actually preserves that key.
