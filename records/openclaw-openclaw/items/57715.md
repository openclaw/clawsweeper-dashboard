## Summary

Keep open. Current main has the lower-risk early filter/search/active/limit path before row construction, but the remaining child-session lookup still scans the store inside each returned row and linked PR #59436 is still open rather than merged.

## What This Changes

The issue asks OpenClaw to reduce `sessions.list` latency by applying cheap filters and limits before expensive row hydration, avoiding unnecessary transcript fallback reads, and replacing per-row child-session scans with a request-scoped index or cache.

## Best Possible Solution

Keep this issue open until #59436 or an equivalent maintainer-reviewed patch lands with a request-scoped child-session index and focused regression coverage. Preserve the current early filter/limit behavior, keep transcript hydration bounded to returned rows, and decide separately whether a TTL or metadata index is warranted for repeated full-list reads.

## Reproduction Assessment

Yes. The discussion contains multiple concrete reproductions, and current source still shows a deterministic remaining reproduction path: many returned sessions with child links cause `buildGatewaySessionRow()` to call a store-scanning `resolveChildSessionKeys()` per row.

## Solution Assessment

Yes, the linked fix direction is the narrowest maintainable path for the remaining root cause: keep the current early row filtering and add a request-scoped child index. A broader TTL cache or persisted summary index should be treated as follow-up unless benchmarks show the returned-row transcript fallback still dominates.
