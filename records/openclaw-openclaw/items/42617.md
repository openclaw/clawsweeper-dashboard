## Summary

Keep this PR open. Current main still lacks the requested `pairingMessage` config surface, and the linked feature request remains valid, but the branch should not merge as-is because it changes the established fenced pairing-code reply format, reaches from bundled plugins into core internals instead of a public SDK seam, and includes an unrelated lockfile package snapshot.

## What This Changes

The PR adds optional `pairingMessage` config/types/schema support for Telegram and iMessage pairing challenges, threads that config through their DM pairing paths, changes shared pairing reply formatting, adds focused tests, touches a few Telegram typing annotations, and modifies `pnpm-lock.yaml`.

## Best Possible Solution

Implement this as an additive public pairing/config seam that preserves the existing fenced code and CLI formatting by default, exposes any new helper/type through an appropriate `openclaw/plugin-sdk/*` entrypoint, wires Telegram and iMessage without extension-to-core imports, updates strict config/schema/docs metadata, and keeps unrelated type or lockfile churn out of the feature branch.

## Reproduction Assessment

Yes. Current-main inspection is enough to reproduce the missing feature surface: the formatter and strict Telegram/iMessage config types have no `pairingMessage` field, and the PR formatter regression is visible by comparing its changed `buildPairingReply` output with the existing `expectPairingReplyText` helper contract.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best merge shape until it preserves the existing reply contract, routes shared pairing helpers through the SDK, and drops the unrelated lockfile change.
