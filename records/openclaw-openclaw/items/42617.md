## Summary

Keep open. Current main still lacks the requested pairing-message configuration, and the paired feature issue remains open; the PR contains relevant implementation work, but it is stale against current main and needs cleanup before it is mergeable.

## What This Changes

The PR adds a configurable `pairingMessage` object for Telegram and iMessage pairing challenges, threads it through pairing reply generation and DM access paths, and adds tests for custom text, `showCliHint`, `senderIdLabel`, and default compatibility.

## Best Possible Solution

Implement the feature as an additive public pairing/config seam that preserves current default fenced-code and CLI formatting, exposes any needed helper or config type through `openclaw/plugin-sdk/channel-pairing`, wires Telegram and iMessage without extension-to-core imports, updates strict schemas and generated config/docs metadata, and adds focused tests for defaults, empty-string overrides, sender-label overrides, and CLI-hint suppression.

Yes. ## Solution Assessment

No, not as currently proposed. The feature direction is valid, but this branch is not the best merge shape because it regresses current fenced pairing-code formatting, reaches from bundled plugins into core internals instead of an SDK seam, and carries an unrelated lockfile change.
