## Summary

This PR is a focused fix for a real CLI diagnostic bug: current main can still turn an arbitrary unknown command into a misleading `plugins.allow` suggestion when a restrictive allowlist is configured. The proposed command-surface check is in the right owner boundary and I did not find a blocking correctness issue in the supplied diff.

## What This Changes

The PR adds a manifest-backed known-plugin-CLI-surface check to the missing plugin command diagnostic, wires it into CLI startup, updates tests, and adds a changelog entry.

## Best Possible Solution

Merge the narrow diagnostic fix after normal CI/touched-surface validation, keeping plugin ownership classification manifest-backed rather than expanding ad hoc reserved command lists.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `resolveMissingPluginCommandMessage` for an arbitrary token with a non-empty `plugins.allow` and no alias/tool owner, and current main reaches the final allowlist message instead of the unknown-command fallback.

## Solution Assessment

Yes. The PR fixes the misleading fallback at the existing CLI policy boundary by asking the manifest command-surface classifier whether the token is a real plugin CLI surface, which is narrower and more maintainable than adding more reserved non-plugin names.
