## Summary

Current main already resolves the PR's user-facing bug by removing provider-usage loading from `channels list`; `channels list --json` now emits channel-only JSON with no `usage` field, so usage-loader failures cannot break the output.

## What This Changes

The PR wraps `channels list --json` provider-usage loading in a catch block and adds a regression test so JSON output omits `usage` when usage loading rejects.

## Best Possible Solution

Keep the current channel-only `channels list` contract and leave provider usage/quota reporting in `openclaw status` and `openclaw models list`.

## Reproduction Assessment

No. On current main, `channels list` no longer has a provider-usage loading path, so a `loadProviderUsageSummary()` failure cannot prevent `channels list --json` from writing JSON.

## Solution Assessment

Yes. Removing provider-usage fetching from a channel-management command is now the narrower maintainable solution than catching one usage-loader failure mode in JSON output.
