## Summary

Keep open; current main still has the JSON usage-load failure path, but this PR needs a few narrow fixes before merge and overlaps the broader open #58685 warning PR.

## What This Changes

The PR wraps JSON usage loading in `channelsListCommand` and adds a Vitest mock/regression test so `channels list --json` omits `usage` when usage loading rejects.

## Best Possible Solution

Land one canonical channels-list fix that preserves JSON stdout, reports usage-fetch failures on stderr, keeps current usage-auth behavior and successful usage output unchanged, includes focused tests plus changelog coverage, and supersedes the other overlapping PR if equivalent work lands first.

## Reproduction Assessment

Yes. Mock `loadProviderUsageSummary` to reject and call `channelsListCommand({ json: true })`; current main reaches the usage await before `writeRuntimeJson`, while the PR test targets the desired JSON payload.

## Solution Assessment

No. The try/catch direction is the right narrow fix, but the patch should also emit a stderr diagnostic, preserve the current `skipPluginAuthWithoutCredentialSource` option, reset the new mock between tests, and add the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Surface usage failures on stderr in JSON mode:** `src/commands/channels/list.ts:127-130`
  - body: When `loadProviderUsageSummary()` throws, this catch emits the same JSON shape as `--no-usage` and gives callers no diagnostic. Text mode already reports the failure through `runtime.error`, and JSON mode can do the same on stderr without polluting machine-readable stdout.
  - confidence: 0.9
- **[P2] Preserve the current usage auth skip option:** `src/commands/channels/list.ts:126`
  - body: Current main calls `loadProviderUsageSummary({ skipPluginAuthWithoutCredentialSource: true })` from `channels list` so the command does not invoke plugin usage auth when no credential source exists. The PR's bare JSON-path call needs to keep that option when rebased, or it can reintroduce unwanted usage-auth work on this path.
  - confidence: 0.82
- **[P3] Reset the new usage mock between tests:** `src/commands/channels.adds-non-default-telegram-account.test.ts:13-15`
  - body: The new `usageMocks.loadProviderUsageSummary` mock is set to reject in the regression test but is not reset in `beforeEach`, so later JSON-list tests can inherit that rejected implementation and fail for the wrong reason.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/commands/channels/list.ts:127-130`
  - body: This is a user-facing CLI fix, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires active-version changelog coverage for user-facing fixes before merge.
  - confidence: 0.9
