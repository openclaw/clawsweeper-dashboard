## Summary

## What This Changes

The branch adds heuristic leaked-reasoning preamble and structural-contamination stripping to the shared assistant visible-text delivery sanitizer, expands sanitizer tests, and removes an obsolete command-analysis barrel.

## Best Possible Solution

Land a narrower sanitizer that runs on the canonical outbound user-facing path, strips only high-confidence leak markers, preserves ordinary prose and fenced examples, and is proved with targeted tests plus redacted Telegram or equivalent runtime evidence.

## Reproduction Assessment

Yes for the PR-introduced blockers by source inspection: the scoring and envelope regex paths deterministically over-strip, and the generic ReplyPayload sanitizer path bypass is visible in current main. I did not establish a live Gemini plus Telegram reproduction for the original provider leak.

## Solution Assessment

No. The current approach is useful in direction, but the best fix should target the canonical outbound sanitizer path and use narrower leak markers with explicit false-positive tests before broad delivery-profile stripping is enabled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate envelope stripping on explicit leak markers:** `src/shared/text/assistant-visible-text.ts:795`
  - body: `CONTAM_ENVELOPE_RE` removes any fenced `json` block that contains a `schema` key before the code-region guard runs. A normal answer that shows a plugin manifest or schema example in a fenced block will lose that whole code block during delivery, so this needs an explicit leakage marker or metadata-key contract plus a fenced-schema pass-through test.
  - confidence: 0.9
- **[P1] Remove conversational cues from preamble scoring:** `src/shared/text/assistant-visible-text.ts:634`
  - body: The scoring regex still treats normal openers such as `for the record`, `now i`, `i need to`, and `i should` as leak evidence, and two matching lines cross the strip threshold because of the suspicious-line-count bonus. That means ordinary multi-line explanations can have their first paragraph silently dropped before send, so these cues should not score without a stronger leak marker.
  - confidence: 0.88
- **[P1] Wire the cleaner through the generic reply sanitizer:** `src/shared/text/assistant-visible-text.ts:853-854`
  - body: This enables the new cleaner only in the assistant-visible delivery profile, but the generic final and streaming auto-reply paths sanitize `ReplyPayload.text` through `sanitizeUserFacingText` before directive normalization. A Telegram/Gemini payload with an untagged preamble can therefore still bypass this PR's cleaner, so the fix needs to cover that canonical delivery path with a focused seam test.
  - confidence: 0.82
