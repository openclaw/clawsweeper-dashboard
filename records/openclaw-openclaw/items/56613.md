## Summary

Keep open. The per-agent TTS half is implemented and present in the latest release through `agents.list[].tts`, but current main still has no Voice/Talk target picker and Android Voice/Talk continues routing through the resolved main session key; the related Android PR only covers Chat.

## What This Changes

The issue asks for mobile/macOS Voice/Talk mode to support agent/session switching and distinct per-agent TTS voices.

## Best Possible Solution

Keep `agents.list[].tts` as the voice identity surface, then add a maintainer-approved shared Voice/Talk target-selection contract with Android proof and follow-through for iOS/macOS if those surfaces are changed.

## Reproduction Assessment

Yes. Source inspection on current main shows the Android Voice surfaces lack selector state, manual mic uses `resolveMainSessionKey()` for `chat.send`, and Talk Mode sends through `mainSessionKey`.

## Solution Assessment

Partly. `agents.list[].tts` is the right current solution for per-agent voice identity, but the remaining routing work should reuse or extend a shared target-selection contract rather than add `talk.voiceId` or another ad hoc session source.
