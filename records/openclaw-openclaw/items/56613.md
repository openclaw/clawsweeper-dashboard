## Summary

Keep open. Current main has implemented the per-agent TTS half as `agents.list[].tts`, but the remaining Voice/Talk path still has no selector and still routes Android mic and Talk Mode sends through the resolved main session; the related open PR only covers Android Chat selection.

## What This Changes

The issue requests Voice/Talk tab agent/session selection across app surfaces plus per-agent TTS voice identity for multi-agent setups.

## Best Possible Solution

Keep `agents.list[].tts` as the voice identity surface, then add selected session/agent routing to Voice/Talk using the existing session or agent-selector contract and validate Android plus any desired iOS/macOS parity separately.

## Reproduction Assessment

Yes. Source inspection on current main shows Android Chat selection is isolated in `ChatController`/`ChatSheetContent`, while manual mic and Talk Mode still build gateway requests around `resolveMainSessionKey()` or `mainSessionKey`.

## Solution Assessment

Yes for the remaining direction, but it needs maintainer UX scoping. Reusing the existing session or agent-selector contract is the maintainable path; adding the requested `talk.voiceId` shape would duplicate the canonical `agents.list[].tts` surface.
