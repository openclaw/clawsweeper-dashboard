## Summary

Keep open. Current main implements the per-agent TTS half through the canonical `agents.list[].tts` surface, but the Android Voice/Talk path still lacks session selection and still routes manual mic and Talk Mode sends through the resolved main session key. The maintainer discussion explicitly kept this open for that remaining UI/routing work.

## What This Changes

Request Talk/Voice mode to support agent/session switching and per-agent voice identity across Android and related app surfaces.

## Best Possible Solution

Keep `agents.list[].tts` as the voice identity surface, then add Android Voice/Talk session selection using the existing Chat session state or selector pattern. Both manual mic sends and Talk Mode sends should route through the selected session key, with focused Android tests covering selection changes and main-session fallback.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A source-level trace on current main shows Android Chat selection is isolated to `chat.switchSession`, while Voice manual mic and Talk Mode continue to build `chat.send` around `resolveMainSessionKey()` or `mainSessionKey`.

## Solution Assessment

Is this the best way to solve the issue? Yes. Reusing the existing session-selection contract and routing Voice/Talk sends through that key is narrower and more maintainable than adding a new `talk.voiceId` config shape or duplicating agent identity logic.
