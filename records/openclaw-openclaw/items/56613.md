## Summary

Keep open. Current main implements the per-agent TTS half through the canonical `agents.list[].tts` surface, but the remaining Talk/Voice session-selection behavior is still incomplete, especially on Android where the Voice tab has no session picker and both manual mic and Talk Mode route through the resolved main session key. The issue discussion also includes a maintainer comment explicitly keeping this open for that remaining UI/routing work.

## What This Changes

Requests Talk/Voice mode agent/session switching across mobile/macOS voice surfaces plus per-agent TTS voice identity.

## Best Possible Solution

Keep the canonical per-agent voice path as `agents.list[].tts`, then add explicit Voice/Talk session or agent selection where the voice UI lacks it. Android should get a Voice-tab picker or shared selected session state and route both `MicCaptureManager` and `TalkModeManager` through that selected key; iOS/macOS should either expose the same selection affordance or clearly bind Talk to the existing active agent/session UI with focused tests and docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A source-level trace on current main shows Android Chat selection is isolated to `chat.switchSession`, while Voice manual mic and Talk Mode continue to build `chat.send` around `resolveMainSessionKey()`/`mainSessionKey`; no live device run was needed to confirm that routing gap.

## Solution Assessment

Is this the best way to solve the issue? Yes for the remaining direction: reuse or share the existing session-selection contract and route Voice/Talk sends through the selected key, instead of adding a new `talk.voiceId` config shape or duplicating agent identity logic.
