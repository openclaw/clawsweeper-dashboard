## Summary

Keep open. Current main and the latest release cover the per-agent TTS half through `agents.list[].tts`, but Android Voice/Talk still has no session or agent selector and still sends manual mic and Talk Mode traffic through the resolved main session; the related open Android PR addresses Chat selection only.

## What This Changes

The issue requests Voice/Talk agent/session switching and per-agent TTS voice identity for multi-agent mobile and desktop voice workflows.

## Best Possible Solution

Keep `agents.list[].tts` as the voice identity surface, then add Voice/Talk session or agent routing through the shared selected-session contract with explicit Android, iOS, and macOS UX parity decisions.

## Reproduction Assessment

Yes. Source inspection on current main shows Android Voice/Talk still does not collect selector state, while manual mic and Talk Mode sends use `resolveMainSessionKey()` or `mainSessionKey`; I did not run a live Android device repro.

## Solution Assessment

Yes for the direction, but it needs product/UX coordination. Reusing the existing selected-session or agent-selector contract is the maintainable path, while adding the proposed `talk.voiceId` shape would duplicate the now-canonical `agents.list[].tts` surface.
