## Summary

Keep open: the issue has a protected `maintainer` label and current source inspection supports the reported TUI state split between lifecycle errors and terminal chat errors.

## What This Changes

The issue asks for active-run lifecycle/provider failures in the TUI to render a visible terminal error, clear stale run state, and unblock later prompts.

## Best Possible Solution

Handle active-run lifecycle terminal errors in `src/tui/tui-event-handlers.ts` with the same visible terminal cleanup semantics as chat `state: "error"`, while preserving retryable/fallback behavior covered by the embedded backend.

## Reproduction Assessment

Yes from source inspection: send a lifecycle `phase: "error"` event for the active run without a terminal chat error, and the handler sets footer status but leaves the transcript/error-clearing path untouched. I did not run a live TUI repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is to route terminal lifecycle errors for the active TUI run into the existing visible terminal error cleanup path, with a focused regression test that covers active-run clearing, stale pending text dismissal, and input unblocking.
