- `impact:session-state`: The affected state is stale TUI run/session state: `activeChatRunId` can remain set after an error.

## Summary

Keep open: this is protected maintainer-owned work, and the current tree still has a TUI lifecycle-error path that can update footer state separately from the visible chat-error cleanup path, even though current main also added a gateway-side chat-error bridge for linked chat.send lifecycle failures.

## What This Changes

The issue requests that terminal TUI lifecycle/provider failures for the active chat run render a visible run error, dismiss stale pending text, clear the active run, and unblock input.

## Best Possible Solution

Keep the issue open for explicit maintainer handling, then either confirm the current gateway bridge is sufficient or add a narrow TUI lifecycle-error cleanup path with focused regression coverage for visible error rendering, watchdog dismissal, active-run clearing, and retryable fallback preservation.

## Reproduction Assessment

Yes from source inspection: send a lifecycle `phase: "error"` event for the active run without a terminal chat error, and the TUI handler updates activity status without rendering a transcript error or clearing the active run. I did not run a live provider/TUI failure in this read-only review.

## Solution Assessment

Yes for the reported TUI state split: the narrow maintainable fix is to route terminal lifecycle errors for the active run into the existing visible error cleanup semantics, while preserving retryable fallback behavior. Maintainers should first decide whether the current gateway-side bridge already covers the product path well enough.
