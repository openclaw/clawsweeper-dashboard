## Summary

Keep this issue open: current main and the latest release still lack a global agent dry-run mode, and the related issues cover broader workflow modes or narrower dry-run scopes rather than superseding this concrete tool-execution request.

## What This Changes

Add a global `openclaw agent --dry-run` mode that keeps tools visible to the model while returning structured previews instead of executing tool calls.

## Best Possible Solution

Define an explicit request-scoped agent execution-mode contract, then implement dry-run at the canonical tool invocation boundaries with coverage for built-in, plugin/channel, client-hosted, gateway, and hook-order behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but source inspection gives high-confidence absence proof: `openclaw agent` has no `--dry-run` option and current tool invocation paths still execute real handlers.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A shared tool-boundary interception point is plausible, but the best implementation needs maintainer-approved semantics for hook order, read-only tools, client/gateway paths, concurrency, and state persistence.
