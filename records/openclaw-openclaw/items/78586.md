## Summary

Keep this PR open: it is a plausible operator feature, but the current patch has a remote shared-secret auth bug and the external contributor has not provided real behavior proof beyond unit-test claims.

## What This Changes

The PR adds an admin-scoped `openclaw sessions abort <session-key> [--force]` CLI command, a `POST /api/sessions/:key/abort` HTTP handler, session abort metadata fields, diagnostics, docs, changelog, and unit tests.

## Best Possible Solution

Keep the PR open, fix the remote shared-secret operator auth path or route the CLI through the scoped Gateway RPC, add targeted server/CLI auth coverage, and require live proof that an orphaned session is aborted without restarting the Gateway.

## Reproduction Assessment

Yes for the review finding by source inspection: shared-secret HTTP auth drops declared scopes on current main, and the new handler uses that resolver before the `sessions.delete` scope check. I did not run tests because this was a read-only review.

## Solution Assessment

No. The feature direction may be useful, but this patch does not yet handle the remote token/password CLI path it exposes and needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor shared-secret admin auth on remote aborts:** `src/gateway/sessions-abort-http.ts:141-142`
  - body: For remote gateways using `gateway.remote.token` or `--token`, shared-secret HTTP auth is marked as not trusted for declared scopes, so `resolveTrustedHttpOperatorScopes` returns `[]` here and the `sessions.delete` check rejects with `missing scope: operator.admin` before the session is touched. That makes the new remote CLI path fail even with the configured operator secret; use an explicitly trusted operator-scope resolver for this endpoint or route the CLI through the existing scoped Gateway RPC.
  - confidence: 0.89
