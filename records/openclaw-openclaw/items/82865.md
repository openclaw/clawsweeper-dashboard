## Summary

Keep open: current main still has the reported Dockerfile startup flag, and an open linked contributor PR is the right place to evaluate the fix before resolving this issue. The unauthenticated-access severity still needs live Docker validation because the runtime also has token generation and non-loopback auth guards.

## What This Changes

The issue asks OpenClaw to stop the official Docker gateway image from starting with `--allow-unconfigured`, or otherwise fail/document unconfigured non-loopback Docker deployments.

## Best Possible Solution

Make Docker startup fail closed unless onboarding or config supplies a valid gateway auth path, while preserving the documented setup flow and adding focused Docker/gateway startup coverage.

## Reproduction Assessment

No, not from this review. Source inspection clearly reproduces the Dockerfile flag and config-guard bypass, but the claimed unauthenticated network access should be confirmed in a built Docker image because runtime token generation and non-loopback auth guards also run.

## Solution Assessment

Unclear pending PR review. Removing or constraining the Docker `--allow-unconfigured` default looks like the likely direction, but it must be checked against Docker onboarding and the existing runtime token/non-loopback guard behavior.
