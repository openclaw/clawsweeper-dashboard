## Summary

Keep open: the Teams proactive-threading problem is real, but this PR is not mergeable as written because it bypasses current reply-style policy, duplicates the current `threadId` contract, and lacks inspectable after-fix Teams proof.

## What This Changes

The PR adds `threadRootMessageId` storage and uses it to route Microsoft Teams proactive fallback text, media, and raw activity sends back into channel threads, with messenger tests.

## Best Possible Solution

Rework or replace the branch so raw file/card proactive sends use the existing `threadId ?? activityId` contract only when resolved `replyStyle` is `thread`, keep configured top-level behavior intact, add focused tests, and attach redacted Teams proof.

## Reproduction Assessment

Yes for source-level reproduction: current main's raw proactive activity helper still clears `activityId`, and the discussion includes a v2026.4.21 production repro where a long Teams channel-thread turn posted top-level after proactive fallback. I did not run a live Teams tenant in this read-only review.

## Solution Assessment

No as written. The useful remaining fix should reuse current main's `threadId` contract and resolved reply-style policy instead of adding a second thread-root field and forcing channel sends into threads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the resolved replyStyle for text sends:** `extensions/msteams/src/send.ts:334`
  - body: This forces `replyStyle: "thread"` whenever a channel ref has `threadRootMessageId`, ignoring the resolved `replyStyle` from channel/team/global config. A channel configured for `top-level` would start replying inside a thread, contradicting the current Teams reply-style contract.
  - confidence: 0.87
- **[P2] Gate raw thread routing on replyStyle:** `extensions/msteams/src/send.ts:401-411`
  - body: The raw file/card helper appends `;messageid=` for any channel ref with `threadRootMessageId`, but it has no `replyStyle` input. File cards and OneDrive links in channels configured `top-level` would be forced into threads unlike the text path contract.
  - confidence: 0.84
- **[P2] Reuse the stored threadId field:** `extensions/msteams/src/conversation-store.ts:36-43`
  - body: Current main already stores the channel thread root as `threadId` from `;messageid=` or `activity.replyToId`. Adding `threadRootMessageId` creates a second thread-root source and leaves refs that only have `threadId` unthreaded in the new raw proactive path.
  - confidence: 0.82
