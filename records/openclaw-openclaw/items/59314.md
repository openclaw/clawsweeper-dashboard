## Summary

This PR should stay open but should not merge as written: it targets a real Microsoft Teams proactive-threading bug, yet the branch duplicates current `threadId` state, bypasses the resolved `replyStyle` policy, and lacks attached after-fix real Teams proof.

## What This Changes

The branch adds `threadRootMessageId` to Microsoft Teams stored conversation references and uses it in text, messenger, and raw proactive send paths to keep channel replies in threads.

## Best Possible Solution

Reuse the existing `threadId` field and resolved `replyStyle` contract for all Teams proactive send paths, including raw file/card sends, with focused regression tests and redacted real Teams proof.

## Reproduction Assessment

Yes for source-level reproduction: current main still has a raw proactive activity helper that clears `activityId`, and the discussion includes a v2026.4.21 production repro where a long Teams channel-thread turn posted top-level after proactive fallback. I did not run a live Teams tenant in this read-only review.

## Solution Assessment

No as written. The branch should be reworked to use current main's canonical `threadId` and resolved `replyStyle` rather than adding `threadRootMessageId` and forcing channel sends into threads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

  - body: This derives `replyStyle: "thread"` whenever a channel ref has `threadRootMessageId`, ignoring the policy-resolved style already carried by the send context.   - confidence: 0.87

  - confidence: 0.84
- **[P2] Reuse the stored threadId field:** `extensions/msteams/src/conversation-store.ts:36-43`

  - confidence: 0.82
