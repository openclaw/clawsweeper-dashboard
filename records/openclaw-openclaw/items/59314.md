## Summary

Keep open: the Teams threading bug is real and partly corroborated by source and production comments, but this PR should not merge as written because it bypasses current `threadId` and `replyStyle` contracts and still lacks attached after-fix real behavior proof.

## What This Changes

The branch adds a new Teams `threadRootMessageId` field and uses it to route proactive Microsoft Teams text, media, and raw activity sends back into channel threads, with messenger tests.

## Best Possible Solution

Keep the existing policy-aware `threadId` model and extend raw proactive activity sends to apply it only when the resolved reply style is threaded, with focused tests.

## Reproduction Assessment

Yes for source-level reproduction: current main's raw proactive activity helper still clears `activityId`, and the discussion includes a v2026.4.21 production repro where a long Teams channel-thread turn posted top-level after proactive fallback. I did not run a live Teams tenant in this read-only review.

## Solution Assessment

No as written. Current main already has `threadId` and resolved `replyStyle` contracts, so the fix should reuse those rather than add `threadRootMessageId` and force channel sends into threads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the resolved replyStyle for text sends:** `extensions/msteams/src/send.ts:334`
  - body: This forces `replyStyle: "thread"` whenever a channel ref has `threadRootMessageId`, ignoring the resolved channel/team/global config. A channel configured for `top-level` would start replying inside a thread, contradicting the current Teams reply-style contract.
  - confidence: 0.87
- **[P2] Gate raw thread routing on replyStyle:** `extensions/msteams/src/send.ts:401-411`
  - body: The raw file/card helper appends `;messageid=` for any channel ref with `threadRootMessageId`, but it has no `replyStyle` input. File cards and OneDrive links in channels configured `top-level` would be forced into threads unlike the text path contract.
  - confidence: 0.84
- **[P2] Reuse the stored threadId field:** `extensions/msteams/src/conversation-store.ts:36-43`
  - body: Current main already stores the channel thread root as `threadId` from `;messageid=` or `activity.replyToId`. Adding `threadRootMessageId` creates a second thread-root source and leaves refs that only have `threadId` unthreaded in the new raw proactive path.
  - confidence: 0.82
