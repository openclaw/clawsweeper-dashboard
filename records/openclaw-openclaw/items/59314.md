## Summary

Keep this PR open, but do not merge it as written. Current main and v2026.4.29 already cover the ordinary revoked-TurnContext reply dispatcher path with stored `threadId`, but `send.ts` still has thread-unaware proactive text/media and raw activity paths, so the PR is not obsolete; the proposed patch needs to preserve documented `top-level` behavior before it is mergeable.

## What This Changes

The PR stores a Teams channel thread root on conversation references and uses it to route proactive fallback text, media/file, and no-context thread sends back into the originating channel thread.

## Best Possible Solution

Keep the existing current-main `threadId` model, add a narrow MS Teams fix for thread-scoped proactive send-tool/raw activity paths, and preserve the documented/configured `top-level` route with regression tests for both behaviors.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The PR discussion gives a production repro with a Teams channel thread, a >15s tool-using turn, and proactive fallback posting top-level; locally, current code inspection shows the remaining `send.ts` proactive paths do not consume the stored thread id.

## Solution Assessment

Is this the best way to solve the issue? No as written. The maintainable fix should reuse current main's `StoredConversationReference.threadId` and gate thread routing to explicit/current-thread sends, rather than adding a parallel `threadRootMessageId` fallback that also changes top-level sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Gate stored thread fallback to thread replies:** `extensions/msteams/src/messenger.ts:503-508`

That breaks the documented Threads-style channel mode and the current guard that top-level sends must not append `;messageid=...`; only the thread fallback/current-thread path should opt into this routing.
