## Summary

Keep open: the Teams proactive-thread bug is real and current main still has a raw file/card proactive-send gap, but this PR is not mergeable as written because it bypasses current reply-style and thread-id contracts and still lacks after-fix real Teams proof.

## What This Changes

The PR adds a stored Teams thread-root field and uses it to route proactive fallback text, media, and raw activity sends back into channel threads, with messenger tests.

## Best Possible Solution

Replace or rebase this into a narrow MS Teams fix that uses `threadId ?? activityId`, only supplies a thread id when resolved `replyStyle` is `thread`, covers raw file/card proactive sends, and includes focused tests plus redacted real Teams proof.

## Reproduction Assessment

Yes for source-level reproduction: current main's raw proactive activity helper still clears `activityId`, and the discussion includes a v2026.4.21 production repro where a long Teams channel-thread turn fell back to proactive send and posted top-level. I did not run a live Teams tenant in this read-only review.

## Solution Assessment

No as written. The intended raw proactive routing fix is useful, but the patch should reuse current main's `threadId` contract and preserve configured `replyStyle: "top-level"` behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the resolved replyStyle for text sends:** `extensions/msteams/src/send.ts:334`
  - body: This forces `replyStyle: "thread"` whenever a channel ref has `threadRootMessageId`, ignoring the resolved `replyStyle` that current main computes from channel/team/global config. A channel configured for `top-level` will start replying inside a thread, contradicting the documented Teams Threads-style mode.
  - confidence: 0.87
- **[P2] Gate raw thread routing on replyStyle:** `extensions/msteams/src/send.ts:401-411`
  - body: The raw file/card helper now appends `;messageid=` for any channel ref with `threadRootMessageId`, but `sendProactiveActivityRaw` has no `replyStyle` input. File cards and OneDrive links in channels configured `top-level` will be forced into threads, unlike the current text path contract.
  - confidence: 0.84
- **[P2] Reuse the stored threadId field:** `extensions/msteams/src/conversation-store.ts:36-43`
  - body: Current main already stores the channel thread root as `threadId` from `;messageid=` or `activity.replyToId`. Adding `threadRootMessageId` and wiring new send paths to it creates a second thread-root source and leaves refs that only have `threadId` or replyToId-derived roots unthreaded in raw proactive sends.
  - confidence: 0.8
