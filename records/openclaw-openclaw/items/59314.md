## Summary

Keep open: the Microsoft Teams threading bug is real and the text/no-context portion is now partly covered by https://github.com/openclaw/openclaw/pull/78387, but this branch still has blocking correctness issues against current `threadId` and `replyStyle` contracts and has no attached after-fix real behavior proof.

## What This Changes

The PR stores Teams channel thread-root metadata and uses it to route proactive Microsoft Teams text, media, and raw activity sends back into channel threads, with messenger tests.

## Best Possible Solution

Rework or replace the branch so raw file/card proactive sends use the existing `threadId ?? activityId` contract only when resolved `replyStyle` is `thread`, preserve configured top-level behavior, add focused tests, and attach redacted Teams proof.

## Reproduction Assessment

Yes for source-level reproduction: current main's raw proactive activity helper still clears `activityId`, and the discussion includes a v2026.4.21 production repro where a long Teams channel-thread turn posted top-level after proactive fallback. I did not run a live Teams tenant in this read-only review.

## Solution Assessment

No as written. Current main already has `threadId` and resolved `replyStyle` contracts, so the fix should reuse those rather than add `threadRootMessageId` and force channel sends into threads.

## Review Findings

Overall correctness: patch is incorrect

  - body: This forces `replyStyle: "thread"` whenever a channel ref has `threadRootMessageId`, ignoring the resolved channel/team/global config. A channel configured for `top-level` would start replying inside a thread, contradicting the current Teams reply-style contract.
  - confidence: 0.87

  - confidence: 0.84

  - confidence: 0.82
