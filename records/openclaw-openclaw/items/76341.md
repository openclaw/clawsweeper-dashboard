## Summary

Keep open. The PR targets a real Slack preview cleanup problem, but the patch is not merge-ready because it calls `draftStream.clear()` during mid-turn transitions even though Slack draft-stream `clear()` permanently stops the stream, so later narrative or tool-preview updates in the same turn become no-ops.

## What This Changes

The PR changes Slack draft preview handling so tool-progress `Working...` bullet drafts are deleted while agent text segments are preserved, and it enables preview streaming for non-off Slack modes including top-level DMs.

## Best Possible Solution

Keep the ephemeral bullet behavior, but implement it with a non-stopping delete/reset operation or a fresh draft-stream instance for later same-turn previews, then add lifecycle regression coverage and update Slack docs/changelog.

## Reproduction Assessment

Yes, by source inspection: on the PR branch, a tool-progress bullet followed by narrative text takes the `await draftStream.clear()` path, and current main's `clear()` permanently stops later `draftStream.update()` calls. I did not run live Slack because this was a read-only review.

## Solution Assessment

No. The requested direction is reasonable, but the patch should not reuse `clear()` for mid-turn deletion unless `clear()` is split from stop semantics or the Slack dispatch creates a new draft stream afterward.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid stopping the draft stream mid-turn:** `extensions/slack/src/monitor/message-handler/dispatch.ts:996`
  - body: `draftStream.clear()` is not a reset-only operation: in the Slack draft stream it calls `discardPending()`, which calls `stop()` and sets the stream's stopped flag. After this bullets-to-narrative path runs, the following `draftStream.update(...)` calls for narrative text or later tool progress in the same turn are ignored, so the live preview disappears after the first deleted bullet draft. Use a delete/reset operation that keeps the stream usable, or create a new draft stream before sending more previews.
  - confidence: 0.92
- **[P3] Update the Slack streaming docs for DM previews:** `extensions/slack/src/monitor/message-handler/dispatch.ts:130`
  - body: This change turns preview streaming on for top-level DMs, but the Slack docs still tell users that top-level DMs do not show the thread-style preview and recommend thread replies or `typingReaction` instead. If the behavior change is intentional, update the docs in the same PR so users do not get contradictory guidance.
  - confidence: 0.82
