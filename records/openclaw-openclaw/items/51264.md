## Summary

Keep open. Current main still loads and merges every discovered agent session store before applying `agentId`, label, active, search, or `limit` filters, so the central eager-load OOM risk remains even though recent cache cloning and cleanup maintenance reduce part of the memory pressure.

## What This Changes

The issue requests lazy or paginated gateway session-store loading so hook dispatch and session lookup do not materialize every agent's full `sessions.json` at once.

## Best Possible Solution

Add a targeted gateway session lookup/listing path that can load only the relevant agent store when `agentId`, canonical key, or a resolved target is known, and make bounded listings honor `agentId` and `limit` before reading unrelated stores. Keep the all-agent aggregation path only for true all-agent listings, preserve disk-only store visibility, and add regression tests proving scoped requests do not read unrelated `sessions.json` files.

## Reproduction Assessment

Yes for the root cause: with a templated per-agent session store and multiple agent directories, any `sessions.list` call or label/sessionId resolve on current main enters `loadCombinedSessionStoreForGateway` and reads every discovered store before filtering. Full heap exhaustion is workload-dependent and was not live-reproduced in this read-only pass.

## Solution Assessment

Yes, keeping this open for a lazy/indexed per-agent lookup is the best direction. Existing JSON-clone and cleanup maintenance mitigations reduce memory growth, but they do not remove the all-store materialization that the report identifies.
