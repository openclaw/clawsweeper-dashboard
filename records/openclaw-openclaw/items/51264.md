## Summary

Keep open: current main has meaningful scoped-store and hook-routing mitigations, but `sessions.resolve` still materializes the combined all-agent store before applying `agentId` for sessionId and label lookups, leaving a concrete residual OOM path.

## What This Changes

The issue requests lazy per-agent gateway session-store loading so scoped hook/session operations do not eagerly read every agent's `sessions.json`.

## Best Possible Solution

Finish the lazy scoped lookup path by loading only the relevant agent store whenever `agentId`, a canonical key, or another resolved target is known, while keeping all-agent aggregation only for true cross-agent listings/searches.

## Reproduction Assessment

Yes at source level: `sessions.resolve` with `sessionId` or `label` plus `agentId` still calls the all-agent combined loader before filtering. I did not live-reproduce the heap OOM because this review is read-only and workload-dependent.

## Solution Assessment

Yes, the narrow maintainable fix is to route scoped resolve/list callers through targeted store loading rather than adding another cache or heap workaround. The existing `agentId` schema and scoped loader seam are already present.
