## Best Possible Solution

Define cancellation as an `AbortError` or explicit aborted outcome before exposing the API, then lock the chosen result shape with tests and real proof.

## Reproduction Assessment

Yes for the PR finding: the diff shows workers returning on abort before recording an error or aborted state, and the added tests accept undefined result slots with `hasError === false`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The worker checkpoint is a reasonable insertion point, but an SDK-facing helper should not expose skipped work as a successful complete result unless maintainers deliberately choose and document that contract.
