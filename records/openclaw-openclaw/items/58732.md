## Summary

Keep this PR open, but it is not merge-ready: the AbortSignal API is a useful small addition, but the current diff turns cancellation into a success-shaped partial result and still lacks external real behavior proof.

## What This Changes

The PR adds an optional `signal?: AbortSignal` parameter to `runTasksWithConcurrency` and tests for pre-aborted and mid-run cancellation.

## Best Possible Solution

Expose cancellation as an `AbortError` or explicit aborted outcome before extending the public helper API, then lock that result shape with tests and real proof.

## Reproduction Assessment

Yes for the PR finding: the diff shows workers returning on abort before setting `hasError`, `firstError`, or an aborted marker, and the added tests accept undefined skipped result slots. I did not run tests because this is a read-only review.

## Solution Assessment

No. The worker checkpoint is a reasonable insertion point, but a plugin-facing helper should not expose skipped work as a successful complete result unless maintainers deliberately choose and document that contract.
