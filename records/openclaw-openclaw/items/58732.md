## Summary

This PR is still a live, focused improvement because current `main` does not support cancellation in `runTasksWithConcurrency`, but the proposed contract is not merge-ready: aborted scheduling returns a success-shaped result with skipped slots, and the external PR still lacks real behavior proof.

## What This Changes

Adds optional `signal?: AbortSignal` support to `runTasksWithConcurrency` and unit tests for pre-aborted and mid-run cancellation.

## Best Possible Solution

Expose cancellation as an `AbortError` or explicit aborted outcome before extending the public helper API, then lock that result shape with focused tests and real proof.

## Reproduction Assessment

Yes for the PR finding: the diff shows workers returning on abort before setting `hasError`, `firstError`, or any aborted marker, and the added tests accept undefined skipped result slots. I did not run tests because this is a read-only review.

## Solution Assessment

No. The worker checkpoint is a reasonable insertion point, but a plugin-facing helper should not expose skipped work as a successful completed result unless maintainers deliberately choose and document that contract.
