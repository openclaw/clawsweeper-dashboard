## Summary

Keep open: current main still has no AbortSignal support for the helper, but this PR is not merge-ready because the abort path returns a success-shaped partial result and the external real-behavior proof gate is still missing.

## What This Changes

Adds an optional `signal?: AbortSignal` parameter to `runTasksWithConcurrency` plus unit tests for pre-aborted and mid-run abort cases.

## Best Possible Solution

Land a revised additive cancellation API that stops scheduling after abort while surfacing cancellation as a non-success or explicit aborted outcome, with focused tests and redacted terminal/log proof.

## Reproduction Assessment

Yes for the review finding: source inspection of the PR diff shows the abort branch returns before recording any error or aborted state, while the added tests accept `hasError === false`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The scheduling checkpoint is reasonable, but exposing partial skipped work as a successful result is not the best public helper contract.
