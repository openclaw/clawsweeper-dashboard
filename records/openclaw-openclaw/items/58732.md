## Summary

Keep open: current main still lacks AbortSignal support, but this PR is not mergeable as written because cancellation returns a success-shaped partial result and the external PR lacks real behavior proof.

## What This Changes

The PR adds an optional `signal?: AbortSignal` parameter to `runTasksWithConcurrency` and two unit tests for pre-aborted and mid-run cancellation.

## Best Possible Solution

Land a revised additive API that stops claiming new tasks after abort while surfacing cancellation as a non-success outcome, preserving no-signal behavior and SDK compatibility.

## Reproduction Assessment

Yes. By source inspection of the PR diff, a pre-aborted or mid-run signal makes workers return before later tasks are claimed while `firstError` and `hasError` remain unset.

## Solution Assessment

No. Checking the signal before task pickup is the right scheduling point, but returning `hasError: false` with skipped result slots is not the narrowest maintainable contract for current consumers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Surface aborted runs as non-success:** `src/utils/run-with-concurrency.ts:25-27`
  - body: When `signal.aborted` is hit, this branch returns without setting `hasError` or `firstError`, so skipped tasks produce undefined result slots while the helper reports success. Existing consumers gate on `hasError` before trusting `results`, so cancellation should surface as an AbortError or explicit aborted outcome instead of a successful partial result.
