## Summary

Keep open: the feature is not on current main, but the PR should not merge as-is because aborted runs return a success-shaped partial result, and the PR body only provides unit-test proof rather than real behavior proof.

## What This Changes

The PR adds an optional `signal?: AbortSignal` parameter to `runTasksWithConcurrency` and tests pre-aborted and mid-run cancellation behavior.

## Best Possible Solution

Land a revised implementation that stops new task pickup on abort while surfacing cancellation as a non-success outcome, preserving no-signal behavior and allowing in-flight tasks to finish.

## Reproduction Assessment

Yes. By source inspection, the PR's deterministic `limit: 1` abort scenario skips later tasks while the patched worker returns without setting `hasError` or `firstError`.

## Solution Assessment

No. Checking the signal before claiming the next task is the right scheduling point, but returning `hasError: false` with holes in `results` is not the narrowest maintainable contract.
