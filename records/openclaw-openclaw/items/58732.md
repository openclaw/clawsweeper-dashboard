## Summary

Keep open. The cancellation feature is useful and narrow, but this PR should not merge as-is because it can return a success-shaped result after skipping pending tasks, which conflicts with current callers that treat `hasError: false` results as usable, and the public helper/API change still needs a changelog entry.

## What This Changes

The PR adds an optional `signal?: AbortSignal` parameter to `runTasksWithConcurrency`, stops workers from taking new tasks once the signal is aborted, and adds unit tests for pre-aborted and mid-run abort scenarios.

## Best Possible Solution

Land the cancellation feature after the helper preserves its result contract: no-signal behavior remains unchanged, already-running tasks may finish, and an abort that skips pending work is surfaced through `hasError`/`firstError` or another explicit cancellation state, with tests and changelog updated together.

## Reproduction Assessment

Yes. The PR's deterministic `limit: 1` abort scenario reproduces the bug in the proposed patch: later tasks are skipped and result slots stay undefined while the patch can still report `hasError: false`.

## Solution Assessment

No. Checking the signal before picking up the next task is the right scheduling point, but silently returning a successful partial result is not the narrowest maintainable API behavior; cancellation should be reported as non-success or with an explicit result state.
