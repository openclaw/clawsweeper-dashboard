## Summary

Keep this PR open, but it is not ready to merge: the proposed cancellation path can return `hasError === false` with partial undefined results from a public SDK-reexported helper, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds an optional `signal?: AbortSignal` parameter to `runTasksWithConcurrency`, checks `signal.aborted` before workers pick up new tasks, and adds two unit tests for pre-aborted and mid-run abort behavior.

## Best Possible Solution

Land a revised additive cancellation API that stops scheduling after abort while returning a non-success or explicit aborted outcome, with focused tests and redacted real terminal/log proof.

## Reproduction Assessment

Yes for the review finding. By source inspection of the PR diff, the `signal.aborted` branch returns before error or aborted state is recorded, and the added pre-aborted test expects `hasError` to remain false.

## Solution Assessment

No. Checking cancellation before task pickup is a reasonable scheduling point, but a public SDK-reexported helper should surface abort as an AbortError or explicit aborted outcome instead of successful partial results.
