## Best Possible Solution

Keep the feature as a focused candidate, but rebase it onto current main and define a stable SDK cancellation result contract before merge.

## Reproduction Assessment

Yes for the patch defect: source inspection of the PR head shows the abort branch returns before scheduling the remaining tasks while leaving hasError false. Current main itself has no signal option to reproduce as runtime behavior.

## Solution Assessment

No. AbortSignal support is plausible, but this patch is not the best merge shape because cancellation is indistinguishable from successful completion for an SDK-exported helper.
