## Summary

Keep open. Current main still makes the HTTPS_PROXY-only native OpenAI completions failure plausible: the OpenAI SDK path now uses OpenClaw's guarded provider fetch, and strict guarded fetch intentionally ignores ambient proxy env unless an explicit provider proxy or managed proxy lifecycle is active.

## What This Changes

The issue reports that native OpenAI Chat Completions requests fail in HTTPS_PROXY-only environments while other provider and channel traffic from the same gateway reaches the proxy.

## Best Possible Solution

Preserve strict guarded-fetch defaults while adding a maintainer-approved proxy path or diagnostics for native model providers, likely through explicit provider `request.proxy.mode: "env-proxy"`, managed `proxy.enabled`, or a narrowly trusted public-provider env-proxy policy.

## Reproduction Assessment

Yes. The issue gives concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current main independently explains the failure mode through strict guarded fetch without ambient env-proxy trust.

## Solution Assessment

Unclear. Ambient HTTPS_PROXY auto-promotion would address the report, but current tests intentionally lock strict direct/DNS-pinned behavior, so the safest solution needs maintainer security and product judgment first.
