## Summary

Keep open. Current main still makes the reported ambient HTTPS_PROXY-only OpenAI completions failure plausible: native OpenAI completions uses OpenClaw's guarded provider fetch, and strict guarded fetch intentionally ignores inherited proxy env unless managed proxy mode or an explicit provider proxy override is active. Changing that default crosses the SSRF/proxy trust boundary, so this needs maintainer/security review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make native OpenAI Chat Completions requests honor HTTPS_PROXY-only sandbox routing, or emit a clear diagnostic when that provider path bypasses proxy routing.

## Best Possible Solution

Preserve the strict SSRF default while making the model-provider proxy contract explicit. Maintainers should either add a reviewed trusted env-proxy path or clearer diagnostic for native public OpenAI endpoints in proxy-only deployments, or direct users through the existing models.providers.openai.request.proxy.mode: "env-proxy" / managed proxy.enabled configuration paths.

## Reproduction Assessment

Yes. The issue provides concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current source/tests independently show that strict provider fetch ignores ambient proxy env without managed or explicit trusted proxy configuration.

## Solution Assessment

Unclear. Blindly honoring inherited HTTPS_PROXY would solve the symptom, but the current test-locked strict behavior protects SSRF destination pinning; the safest fix needs maintainer security/product judgment around trusted provider endpoints and diagnostics.
