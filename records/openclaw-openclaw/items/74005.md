## Summary

Keep open. Current main still makes the reported HTTPS_PROXY-only OpenAI failure plausible: native OpenAI Chat Completions now goes through OpenClaw's guarded provider fetch, and strict guarded fetch intentionally ignores ambient proxy env unless the operator enables a provider proxy override or managed proxy mode. The report is recent, concrete, and the likely fix touches SSRF/proxy trust policy, so it needs maintainer judgment rather than cleanup closure.

## What This Changes

The issue asks OpenAI `openai-completions` requests to `api.openai.com` in an HTTPS_PROXY-only sandbox to route through the configured proxy or emit an actionable diagnostic when that route is bypassed.

## Best Possible Solution

Preserve strict SSRF defaults while making the model-provider proxy contract explicit. Maintainers should either add a reviewed trusted env-proxy path for native public model endpoints in proxy-only deployments, or improve diagnostics/docs so users are directed to provider `request.proxy.mode: "env-proxy"` or managed `proxy.enabled` when ambient HTTPS_PROXY alone is insufficient.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path using an HTTPS_PROXY-only sandbox with direct egress blocked, and current source independently explains why native OpenAI completions without provider proxy or managed proxy mode would stay on strict guarded fetch instead of ambient env-proxy routing.

## Solution Assessment

Unclear. Automatically honoring inherited HTTPS_PROXY would address the reported failure, but the current SSRF tests document deliberate strict-mode behavior, so the safest fix needs maintainer security and product judgment first.
