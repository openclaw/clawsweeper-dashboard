## Summary

Keep open. Current main still leaves the reported HTTPS_PROXY-only OpenAI completions failure class plausible, and the remaining decision is not cleanup-safe because it intersects the provider request proxy contract and SSRF/DNS-pinning policy.

## What This Changes

The issue asks OpenAI `openai-completions` requests to `api.openai.com` to honor an HTTPS_PROXY-only deployment or emit a clear diagnostic when proxy routing is not active.

## Best Possible Solution

Preserve strict SSRF defaults while making the OpenAI provider proxy behavior explicit. Maintainers should either approve a narrowly trusted env-proxy path for public native model endpoints, or improve diagnostics/docs so HTTPS_PROXY-only operators are directed to `models.providers.openai.request.proxy.mode: "env-proxy"` or managed `proxy.enabled` before model traffic falls back or fails.

## Reproduction Assessment

Yes. The issue gives concrete HTTPS_PROXY-only steps and proxy-log evidence, and current source shows an ordinary OpenAI completions request without provider `request.proxy` or managed proxy active remains on strict guarded fetch rather than ambient env-proxy routing.

## Solution Assessment

Unclear. Automatically honoring inherited HTTPS_PROXY would address this deployment, but current tests document a deliberate strict-mode invariant, so the safer solution needs maintainer security/product judgment before implementation.
