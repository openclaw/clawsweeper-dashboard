## Summary

Keep open. Current main still routes native OpenAI Chat Completions through OpenClaw's guarded provider fetch path, and that strict path intentionally does not honor ambient HTTPS_PROXY unless a provider proxy override or managed proxy lifecycle is active. The report is therefore plausible and actionable, but the remaining fix needs maintainer security judgment around SSRF pinning and proxy trust.

## What This Changes

The issue requests that native OpenAI `openai-completions` traffic to `api.openai.com` work in HTTPS_PROXY-only gateway environments or emit a clear proxy-bypass diagnostic.

## Best Possible Solution

Keep the strict guarded-fetch default unless maintainers explicitly approve a broader proxy contract. The maintainable end state is either a reviewed trusted env-proxy path for public model-provider endpoints with SSRF tests, or clearer diagnostics/docs that direct proxy-only users to `models.providers.openai.request.proxy.mode: "env-proxy"` or managed `proxy.enabled`.

## Reproduction Assessment

Yes. The issue provides a concrete HTTPS_PROXY-only gateway setup and proxy-log evidence, and current main explains the failure mode because native OpenAI completions stay in strict guarded fetch without ambient env-proxy trust.

## Solution Assessment

Unclear. Automatically honoring ambient HTTPS_PROXY would address the reported setup, but it is not clearly the safest fix because current strict-mode tests intentionally preserve DNS-pinned behavior unless an explicit provider or managed proxy mode is active.
