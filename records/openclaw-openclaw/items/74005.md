## Summary

Keep open. Current main still makes the reported HTTPS_PROXY-only OpenAI failure plausible, and the remaining decision is security-sensitive: native OpenAI Chat Completions now goes through OpenClaw's guarded provider fetch path, whose strict mode intentionally ignores ambient proxy env unless an explicit trusted provider proxy or managed proxy lifecycle is active.

## What This Changes

The issue asks OpenClaw to make `openai-completions` calls to `api.openai.com` honor an inherited HTTPS forward proxy in proxy-only sandboxes, or to emit clearer diagnostics when that path bypasses proxy routing.

## Best Possible Solution

Preserve the strict guarded-fetch default while making the proxy contract explicit for native model providers: either add a reviewed trusted env-proxy path for public provider endpoints, or improve diagnostics and docs so proxy-only users are directed to provider `request.proxy.mode: "env-proxy"` or managed `proxy.enabled`.

## Reproduction Assessment

Yes. The issue provides concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current main independently explains the failure mode by routing native OpenAI completions through strict guarded fetch without ambient env-proxy trust.

## Solution Assessment

Unclear. Ambient HTTPS_PROXY auto-promotion would address the reported setup, but current tests intentionally lock strict-mode direct/DNS-pinned behavior, so the safest fix needs maintainer security and product judgment first.
