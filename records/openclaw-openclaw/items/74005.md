## Summary

Keep open. Current main still makes the reported HTTPS_PROXY-only OpenAI failure plausible: native OpenAI Chat Completions is routed through OpenClaw's guarded provider fetch, and strict guarded fetch intentionally ignores ambient HTTP_PROXY/HTTPS_PROXY unless the operator enables a trusted provider proxy override or managed proxy mode. The report is recent, concrete, and the fix boundary affects SSRF/proxy trust policy, so this should stay with maintainers rather than be swept closed.

## What This Changes

The issue requests that OpenAI Chat Completions traffic to api.openai.com honor an HTTPS_PROXY-only deployment, or at least produce a clear diagnostic when the OpenAI path bypasses the configured proxy route.

## Best Possible Solution

Preserve strict SSRF defaults while making the model-provider proxy contract explicit. Maintainers should either add a reviewed trusted env-proxy path for native public model endpoints in proxy-only deployments, or improve diagnostics/docs so users are directed to provider `request.proxy.mode: "env-proxy"` or managed `proxy.enabled` when ambient HTTPS_PROXY alone is insufficient.

## Reproduction Assessment

Yes. The issue gives concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current source independently explains why native OpenAI requests without explicit provider proxy or managed proxy mode stay on strict guarded fetch rather than ambient env-proxy routing.

## Solution Assessment

Unclear. Automatically honoring inherited HTTPS_PROXY would address the observed deployment, but the current test suite documents a deliberate strict-mode SSRF/DNS-pinning invariant, so the safest solution needs maintainer security and product judgment first.
