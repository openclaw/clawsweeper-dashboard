## Summary

Keep open. Current main still leaves the reported HTTPS_PROXY-only OpenAI provider failure class plausible: native OpenAI Chat Completions uses OpenClaw's guarded model fetch, and that strict guard intentionally ignores ambient HTTP_PROXY/HTTPS_PROXY unless the operator has explicitly enabled provider env-proxy mode or managed OpenClaw proxy mode. The issue is recent, detailed, and its likely fix intersects SSRF/proxy trust policy, so this needs maintainer judgment rather than cleanup closure.

## What This Changes

The issue asks for OpenAI `openai-completions` traffic to `api.openai.com` to honor HTTPS_PROXY-only gateway environments, or to emit a clear diagnostic when the OpenAI-specific request path bypasses the configured proxy.

## Best Possible Solution

Preserve strict SSRF defaults while making the model-provider proxy contract explicit. Maintainers should either add a reviewed trusted-env-proxy path for public native model endpoints in proxy-only deployments, or improve diagnostics/docs so users are directed to `models.providers.openai.request.proxy.mode: "env-proxy"` or managed `proxy.enabled` instead of assuming ambient HTTPS_PROXY alone is sufficient.

## Reproduction Assessment

Yes. The issue provides concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current source independently shows why OpenAI provider requests without explicit `request.proxy` or managed proxy mode remain on strict guarded fetch instead of ambient env-proxy routing.

## Solution Assessment

Unclear. Automatically honoring inherited HTTPS_PROXY would address the reported deployment, but current tests document a deliberate SSRF/DNS-pinning invariant, so the safest fix needs maintainer security/product judgment first.
