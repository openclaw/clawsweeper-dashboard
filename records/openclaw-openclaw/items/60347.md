## Summary

Keep open: the feature is valid and dependency-backed, but the current PR is not mergeable because the accepted `gpu` setting would be dropped before the local embedding runtime and the public config surfaces are incomplete.

## What This Changes

The PR adds a `memorySearch.local.gpu` option for local node-llama-cpp embeddings, validates it, passes it to `getLlama()`, and updates a generated config baseline hash.

## Best Possible Solution

Land a refreshed implementation that adds `memorySearch.local.gpu` to the public config contract, resolves default and per-agent values through `resolveMemorySearchConfig()`, forwards the option to the existing local embedding runtime only when explicitly configured, and updates generated config artifacts, docs, tests, and changelog.

## Reproduction Assessment

Yes. A focused unit path can configure `agents.defaults.memorySearch.local.gpu: "vulkan"`, resolve memory search config, create the local embedding provider with a mocked node-llama-cpp import, and assert that `getLlama()` receives `gpu: "vulkan"`; the current PR does not thread the value through the resolver.

## Solution Assessment

No, not as proposed. The direction is the right narrow solution, but the branch needs config-resolution propagation plus generated schema/help/docs, changelog, and focused tests before it is maintainable.
