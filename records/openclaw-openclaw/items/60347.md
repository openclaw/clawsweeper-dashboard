## Summary

Keep this PR open: the requested GPU backend selector is still missing on current main and the node-llama-cpp API supports the shape, but the proposed branch is incomplete against the memory/config path and needs a refresh before merge.

## What This Changes

The PR adds a `memorySearch.local.gpu` config option for local node-llama-cpp embeddings, validates the setting, passes it into `getLlama()`, and updates generated config baseline hashes/types.

## Best Possible Solution

Land a refreshed narrow implementation that adds `memorySearch.local.gpu` to the public config contract, merges default and per-agent values through `resolveMemorySearchConfig`, passes the option to `getLlama()` only when explicitly configured, updates generated schema/help/docs, and preserves the existing optional `node-llama-cpp` runtime dependency policy.

## Reproduction Assessment

Yes. A focused unit test can configure `agents.defaults.memorySearch.local.gpu: "vulkan"`, call `resolveMemorySearchConfig()` plus `createLocalEmbeddingProvider()` with a mocked node-llama-cpp import, and assert that `getLlama()` receives `gpu: "vulkan"`; current main lacks that propagation.

## Solution Assessment

No, not as currently proposed. The direction is sound and dependency-backed, but the PR needs the resolved config merge, generated config surfaces, docs, and tests before it is the narrow maintainable solution.
