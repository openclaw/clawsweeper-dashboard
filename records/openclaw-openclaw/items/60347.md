Full review comments:

- **[P1] Thread gpu through resolved memory settings:** `src/config/types.tools.ts:385`
  - body: Adding `memorySearch.local.gpu` to the accepted config and reading `options.local?.gpu` in the provider is not enough because gateway-created providers receive `resolveMemorySearchConfig().local`. That resolver currently builds `local` from only `modelPath`, `modelCacheDir`, and `contextSize`, so `gpu: "vulkan"` or `gpu: false` would be accepted by validation and then silently dropped before `getLlama()` is called.
  - confidence: 0.93
- **[P2] Update the active node-llama wrapper type:** `src/types/node-llama-cpp.d.ts:20-24`
  - body: This PR updates the legacy ambient `node-llama-cpp` declaration, but current main imports `getLlama` through `packages/memory-host-sdk/src/host/node-llama.ts`. After rebasing, the package wrapper still types `getLlama` as accepting only `logLevel`, so the new `{ gpu }` argument needs to be added to the active wrapper contract instead of only this declaration.
  - confidence: 0.88
- **[P2] Document the public gpu config field:** `src/config/zod-schema.agent-runtime.ts:657-665`
  - body: The new runtime schema field is a user-facing config contract, but the PR does not add matching config help, labels, or memory-config documentation for `memorySearch.local.gpu`. Schema/UI/docs consumers and users would still see only the existing local model/cache/context fields, so the branch should update those public surfaces and regenerate the config artifacts from current main.
  - confidence: 0.86
