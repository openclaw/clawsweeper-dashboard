## Summary

Keep open: the GPU backend selector is a plausible feature, but this PR is not mergeable against current main because the proposed `gpu` setting is not carried through the current resolver/active wrapper path, lacks aligned public config docs/metadata/tests, and still needs contributor-supplied real behavior proof.

## What This Changes

The PR adds an optional `memorySearch.local.gpu` config field, Zod validation, a `getLlama` GPU passthrough for local embeddings, ambient node-llama-cpp type updates, and a generated checksum change.

## Best Possible Solution

Land a rebased implementation that exposes only upstream-supported local llama backend selection through schema, help, labels, docs, resolver, provider, active wrapper types, and focused tests while preserving current `contextSize` behavior and adding redacted runtime proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the current resolver and active wrapper path show where `gpu` must be threaded before it can affect local embeddings. No live Orange Pi or GPU runtime proof was available in the PR artifacts.

## Solution Assessment

No, not as currently proposed. The backend selector direction is reasonable, but this branch needs a current-main rebase across resolver, active wrapper, docs/help/labels, schema tests, provider tests, and real behavior proof.
