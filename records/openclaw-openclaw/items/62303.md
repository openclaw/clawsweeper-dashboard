## Summary

Keep open. Current main still gates Telegram DM topic auto-labeling on the session-store `systemSent` flag, so a DM-topic session that already has `systemSent: true` skips the generated-label and `editForumTopic` rename path; I found no linked open PR or current-main fix superseding this report.

## What This Changes

Fix Telegram DM topics so default-on `autoTopicLabel` renames newly created topics even when core session prompt state already has `systemSent: true`.

## Best Possible Solution

Keep the repair inside the Telegram plugin by tracking an idempotent per-topic label attempt/state independent of core `systemSent`, while preserving default-on config, direct-over-account precedence, `RawBody` input, and fire-and-forget rename delivery.

## Reproduction Assessment

Yes. A dispatch-level unit test can use a DM-topic context with `SessionKey`, `RawBody`, a visible final response, and a loaded session entry where `systemSent: true`; current main will skip `generateTopicLabel` and `editForumTopic` because `isFirstTurnInSession` is false.

## Solution Assessment

No patch is proposed here. The narrow maintainable fix is to decouple Telegram topic-label attempts from core prompt/session state rather than changing global first-turn or `systemSent` semantics.
