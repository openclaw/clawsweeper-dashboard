## Summary

Keep open. Current main still gates Telegram DM-topic auto-renaming on `!entry?.systemSent`, so a newly-created DM topic whose session entry is already marked `systemSent: true` can skip `generateTopicLabel` and `editForumTopic`; the later Codex label-generator fix did not remove this dispatcher gate.

## What This Changes

The issue reports that Telegram DM topics keep their provisional fallback title instead of being renamed by the default-on `autoTopicLabel` flow after the first user message.

## Best Possible Solution

Keep the fix inside the Telegram plugin by tracking an idempotent per-DM-topic label attempt independently of core `systemSent`, while preserving default-on config, direct-over-account precedence, `RawBody` input, and fire-and-forget topic rename delivery.

## Reproduction Assessment

Yes, at source level. A dispatch-level check can use a DM-topic `SessionKey`, `RawBody`, visible final response, and a loaded session entry with `systemSent: true`; current main then skips `generateTopicLabel` and `editForumTopic` because `isFirstTurnInSession` is false.

## Solution Assessment

No patch is proposed here. The narrow maintainable fix is to decouple Telegram topic-label attempts from core prompt/session setup state instead of changing global first-turn or `systemSent` semantics.
