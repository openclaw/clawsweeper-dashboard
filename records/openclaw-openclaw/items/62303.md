## Summary

Keep open. Current main and v2026.5.18 still gate Telegram DM-topic auto-renaming on `!entry?.systemSent`, so a topic whose session entry is already bootstrapped can skip `generateTopicLabel` and `editForumTopic` and keep the fallback title; the newer related report was closed back to this issue as the canonical bug.

## What This Changes

Telegram DM-topic auto-labeling should rename new topics after the first user message instead of leaving the fallback title.

## Best Possible Solution

Repair the Telegram dispatcher so DM-topic label attempts are tracked independently of core `systemSent`, with focused dispatch regression coverage and unchanged config semantics.

## Reproduction Assessment

Yes, at source level. A dispatch-level reproduction can preload a Telegram DM-topic turn with `SessionKey`, `RawBody`, a visible final response, and a session-store entry where `systemSent: true`; current main then bypasses the label block.

## Solution Assessment

No proposed patch is under review. The narrow maintainable fix is to keep the change inside the Telegram plugin and decouple topic-label attempts from global session bootstrap semantics.

## Repair Work Prompt

Fix the Telegram DM-topic `autoTopicLabel` regression tracked by https://github.com/openclaw/openclaw/issues/62303. Current `extensions/telegram/src/bot-message-dispatch.ts` gates fire-and-forget label generation on `isFirstTurnInSession`, which is derived from `sessionStore[SessionKey].systemSent`; when the session is already bootstrapped before the post-reply hook, new topics never call `generateTopicLabel` or `bot.api.editForumTopic`. Keep the repair inside the Telegram plugin: decouple the label attempt from core `systemSent` semantics using a pre-turn snapshot or an idempotent per-DM-topic label-attempt marker. Preserve default-on `autoTopicLabel`, direct-over-account precedence, `RawBody` input, the 500-character label prompt input cap, fire-and-forget delivery, and the merged Codex label-generator behavior from https://github.com/openclaw/openclaw/pull/78450. Add a dispatch regression test where a DM-topic turn has `SessionKey`, `RawBody`, a visible final response, and a loaded session entry with `systemSent: true`, and assert `generateTopicLabel` and `editForumTopic` are called once; keep disabled-config coverage. Add a changelog entry because this is user-visible Telegram behavior. Do not change global `systemSent` semantics, Telegram action permissions, or unrelated Telegram topic routing.
