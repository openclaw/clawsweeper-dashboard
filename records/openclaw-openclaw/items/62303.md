## Summary

Keep open. Current main still uses the session-store `systemSent` flag as the Telegram DM-topic first-turn gate before `generateTopicLabel` and `editForumTopic`, so a newly-created topic whose session entry is already marked `systemSent: true` can skip the rename path and keep the fallback title; the later Codex label-generator fix does not remove this dispatcher gate.

## What This Changes

The issue asks OpenClaw to restore default Telegram DM topic auto-renaming so new DM topics receive the generated `autoTopicLabel` instead of keeping the fallback title.

## Best Possible Solution

Keep the fix inside the Telegram plugin by tracking an idempotent per-DM-topic label attempt independently of core `systemSent`, while preserving default-on config, direct-over-account precedence, `RawBody` input, and fire-and-forget topic rename delivery.

## Reproduction Assessment

Yes, at source level. A dispatch-level test can use the existing DM-topic harness with `SessionKey`, `RawBody`, a visible final response, and a loaded session entry where `systemSent: true`; current main will skip `generateTopicLabel` and `editForumTopic` because `isFirstTurnInSession` is false.

## Solution Assessment

No patch is proposed here. The narrow maintainable solution is to decouple Telegram topic-label attempts from core prompt/session setup state instead of changing global first-turn or `systemSent` semantics.
