## Label Justifications

- `P2`: This is a valid Telegram plugin regression with a clear source-level path, but it affects topic naming rather than core runtime availability or message delivery.
- `impact:session-state`: The failure is caused by Telegram topic labeling depending on persisted session bootstrap state via `systemSent`.

## Summary

Keep open. Current main and the latest stable release still gate Telegram DM-topic auto-renaming on `!entry?.systemSent`, so a topic whose session entry is already marked bootstrapped can skip the label generation and `editForumTopic` path.

## What This Changes

The issue reports that Telegram DM topics keep their provisional fallback title instead of being renamed by the default-on `autoTopicLabel` flow after the first user message.

## Best Possible Solution

Keep the repair inside the Telegram plugin by recording an idempotent per-DM-topic label attempt independently of core `systemSent`, while preserving default-on config, direct-over-account precedence, `RawBody` input, and fire-and-forget delivery.

## Reproduction Assessment

Yes, at source level. A dispatch-level test can preload the Telegram DM-topic harness with `SessionKey`, `RawBody`, a visible final response, and a session-store entry where `systemSent: true`; current main will skip `generateTopicLabel` and `editForumTopic`.

## Solution Assessment

Not applicable to a proposed patch because this is an issue. The maintainable fix is to decouple Telegram topic-label attempts from core prompt/session bootstrap state instead of changing global first-turn or `systemSent` semantics.
