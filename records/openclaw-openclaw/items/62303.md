## Label Justifications

- `P2`: This is a real Telegram DM-topic regression with a clear source path, but it affects topic naming rather than core runtime availability or message delivery.
- `impact:session-state`: The bug is caused by using session-store `systemSent` state as the eligibility signal for a per-topic Telegram rename.

## Summary

Keep open. Current main and the latest stable release still gate Telegram DM-topic auto-renaming on `!entry?.systemSent`, so a bootstrapped topic can skip `generateTopicLabel` and `editForumTopic`; the newer report at https://github.com/openclaw/openclaw/issues/82320 was closed as a duplicate of this canonical issue.

## What This Changes

The issue asks for Telegram DM topics to auto-rename after the first real message using `autoTopicLabel` instead of keeping the provisional topic title.

## Best Possible Solution

Repair this inside the Telegram plugin by tracking an idempotent per-DM-topic label attempt independently of core `systemSent`, while preserving default-on config, direct-over-account precedence, `RawBody` input, Codex label generation, and fire-and-forget delivery.

## Reproduction Assessment

Yes, at source level. A dispatch-level test can preload the DM-topic harness with `SessionKey`, `RawBody`, a visible final response, and `loadSessionStore` returning `systemSent: true`; current main will skip `generateTopicLabel` and `editForumTopic`.

## Solution Assessment

Not applicable to a proposed patch because this is an issue. The safest implementation direction is a Telegram-local idempotent topic-label marker rather than changing global `systemSent` semantics without broader session-state proof.
