## Summary

Keep open. Current main still collapses tool-policy evaluation to filtered tool lists and generic unavailable-tool results, so embedded and gateway audit surfaces cannot log the matched policy layer or rule; the issue is also paired with open PR #55826, which explicitly targets this work and remains unmerged.

## What This Changes

The issue requests audit logging for tool-policy allow/deny decisions that includes the decision, matched configuration path, and matched rule when applicable.

## Best Possible Solution

Retain structured policy explanation data through matcher and pipeline evaluation, then emit sanitized audit lines for embedded tool calls and gateway `/tools/invoke` allow/deny outcomes with focused tests for allow, deny, allowlist miss, gateway deny, and sandbox policy cases.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a policy that removes a tool, then trace the pipeline and gateway/embedded call sites to see only filtered tool lists and generic logs remain, with no matched rule metadata available.

## Solution Assessment

Unclear for the open PR. The direction is right, but maintainers should verify whether #55826 covers both embedded and HTTP per-call audit requirements; otherwise a narrow replacement or follow-up should extend the same policy explanation seam.
