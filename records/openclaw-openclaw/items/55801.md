## Summary

Keep open. Current main still filters tool policy down to a surviving tool list without retaining matched policy layer or rule attribution, and the issue is paired with open PR #55826, which explicitly closes this issue but has not merged.

## What This Changes

The issue requests audit logging for tool-policy allow/deny decisions that includes the policy decision, matched configuration path, and matched rule when one applies.

## Best Possible Solution

The end state should retain a structured policy explanation through the matcher/pipeline and then emit sanitized audit lines for embedded tool calls and HTTP `/tools/invoke`, with tests covering allow, deny, allowlist miss, gateway deny, and sandbox policy cases.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a tool policy that filters a tool, then follow the pipeline and call sites to see that only the filtered tool list reaches embedded and HTTP consumers, so no matched rule can be logged.

## Solution Assessment

Unclear for the open PR. The right direction is narrow and maintainable, but PR #55826 appears to focus on pipeline-removal and sandbox messages, so maintainers should verify whether it also satisfies the issue's per-call embedded and HTTP audit requirements or replace it with a fuller targeted patch.
