## Summary

Keep open; current main and the latest release still apply the active embedding model as a predicate on lexical FTS and LIKE keyword rows before hybrid merge, so exact keyword rows indexed under a different model can still disappear. The prior target-repo repair at https://github.com/openclaw/openclaw/pull/48328 is closed and unmerged, while the remaining open fork PR is not a target-repo merge candidate and still targets obsolete `src/memory` paths.

## What This Changes

The issue reports that hybrid `memory_search` can return no exact FTS keyword matches even though the SQLite FTS index contains matching chunks.

## Best Possible Solution

Remove provider-model filtering from lexical FTS MATCH and LIKE fallback paths, keep vector search model-scoped, and add regression coverage for chunks indexed under a different embedding model.

## Reproduction Assessment

Yes for source-level reproduction: seed `chunks_fts` with a matching row whose stored `model` differs from the active provider model, then call the active keyword helper with `providerModel` set. Current main appends `AND model = ?`, filtering that lexical row before hybrid merge.

## Solution Assessment

Yes in direction: lexical FTS should ignore embedding model while vector search stays model-scoped. The old target-repo PR is not the right implementation vehicle because it is closed, unmerged, and targeted obsolete `src/memory` files.
