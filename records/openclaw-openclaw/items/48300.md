## Summary

Keep open: current main still has a source-backed path where active memory-core FTS keyword search is filtered by the current embedding model, and the open closing PR is still unmerged and needs to be retargeted to the active implementation.

## What This Changes

The issue reports that hybrid `memory_search` can return no FTS keyword matches even when the SQLite FTS index contains matching chunks.

## Best Possible Solution

Retarget or replace the open FTS model-filter fix in `extensions/memory-core`, keep vector search model filtering, cover MATCH and LIKE fallback paths, and require redacted real `memory_search` proof before closing the issue.

## Reproduction Assessment

Yes for the source-level provider/model mismatch path: seed an FTS row under one model and search with a different active provider model; current main appends `AND model = ?` to active keyword queries. I did not establish the reporter's full gateway/Ollama stale-index path live.

## Solution Assessment

Yes, removing the model predicate only from lexical FTS search is the narrow maintainable fix, while keeping vector search model-specific. The existing open PR has the right intent but must be applied to active `extensions/memory-core` code and backed by real behavior proof.
