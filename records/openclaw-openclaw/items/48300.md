## Summary

Keep this open: current main still has a source-visible path where FTS keyword hits are filtered by the active embedding model, and the open linked fix PR explicitly references this issue with closing syntax. The issue should stay open until that PR, or a rebased replacement against the current memory-core plugin path, is resolved.

## What This Changes

The issue asks for hybrid memory_search to return SQLite FTS keyword matches instead of an empty result set when exact keyword matches exist.

## Best Possible Solution

Rebase or replace the linked fix in `extensions/memory-core` so FTS keyword search is not gated by embedding model, while vector search remains model-specific and regression coverage proves mixed-model FTS recall.

## Reproduction Assessment

Yes for the provider/model-mismatch variant: index a chunk under one embedding model, switch the active memory embedding model, then search an exact keyword in hybrid mode. Source inspection shows current main filters FTS rows by the active model, but I did not run a live macOS/Ollama gateway reproduction of the reporter's stale-process hypothesis.

## Solution Assessment

Yes, the narrow fix direction is to remove the model predicate from lexical FTS search while keeping vector search model-scoped. The existing linked PR has that shape, but it needs to be reconciled with the current plugin path before it can close the issue.
