## Summary

Keep open; current main still passes the active embedding model into lexical keyword search and applies `AND model = ?` to FTS MATCH and LIKE fallback rows before hybrid merge, so exact FTS rows indexed under a different model can still disappear. The target-repo PR that attempted this repair is now closed and unmerged, and it touched obsolete `src/memory` paths rather than the active `extensions/memory-core` implementation.

## What This Changes

The issue reports that hybrid `memory_search` can return no lexical FTS results for exact keyword matches even though the SQLite FTS index contains matching chunks.

## Best Possible Solution

Remove the provider-model predicate from memory-core lexical FTS MATCH and LIKE fallback paths, keep vector search model-scoped, and add regression coverage for chunks indexed under a different embedding model.

## Reproduction Assessment

Yes for source-level reproduction: seed `chunks_fts` with a row whose stored `model` differs from the active provider model, then call the active keyword helper with `providerModel` set. Current main appends `AND model = ?`, filtering that lexical row before hybrid merge.

## Solution Assessment

Yes in direction: lexical FTS should ignore embedding model while vector search stays model-scoped. The old linked PR is not the best implementation vehicle because it is closed, conflicting, and targeted obsolete `src/memory` files.
