## Summary

Keep open: current main and the latest release still lack a durable sender identity contract for multi-agent chat messages, and the related closed items only cover adjacent UI styling, sender labels, prompt metadata, or broader multi-agent UX rather than this focused human/main-agent/sub-agent attribution path.

## What This Changes

Request durable multi-role sender identity in the Chat UI so human messages, main-agent steer messages, and sub-agent replies can be distinguished in multi-agent workflows.

## Best Possible Solution

Define an additive per-message sender metadata contract, persist it in transcripts, expose it through history and live events, carry it through chat.send, sessions.send/sessions.steer, and subagent completion delivery, then render it in Control UI with explicit trust boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main and v2026.5.12 lack durable sender identity fields across Gateway schemas, transcript persistence, subagent delivery, and Control UI rendering.

## Solution Assessment

Yes in direction, but avatars alone are not enough. The maintainable solution is an additive durable sender metadata contract with protocol, transcript, event, UI, and trust-model follow-through rather than reusing browser-local identity or senderLabel hints.
