## Summary

Keep open. Current main and v2026.6.10 have useful partial attribution for `sessions_send` forwarded display, browser-local identity, and trace provenance, but they still do not provide the requested durable human/main-agent/sub-agent sender identity contract across protocol input, transcript/model history, subagent announcements, live/history payloads, and per-message Control UI identity rendering.

## What This Changes

The issue asks for durable multi-role sender identity in the Chat UI so human, main-agent, and sub-agent messages are distinguishable in multi-agent workflows.

## Best Possible Solution

Define one additive trusted sender metadata contract, persist it where appropriate, propagate it through `chat.send`, `sessions.send`/`sessions.steer`, subagent announce delivery, history/live events, and render stable names, avatars, and colors without changing provider-facing roles.

## Reproduction Assessment

Yes, at source level. Current schemas, provenance types, transcript input, subagent delivery, history projection, and Control UI rendering provide trace fields or derived labels rather than a durable human/main-agent/sub-agent sender identity contract.

## Solution Assessment

Yes in direction, but not as new provider message roles or a UI-only avatar patch. The maintainable solution is additive trusted sender metadata that preserves canonical transcript roles while giving history and UI consumers durable renderable identity.
