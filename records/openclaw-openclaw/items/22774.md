## Summary

Keep open: current main still lacks the durable human/main-agent/sub-agent sender identity contract requested here, and the related sender metadata issues/PRs are adjacent rather than a replacement for this focused Control UI/protocol attribution path.

## What This Changes

Request durable multi-role sender attribution in the Chat UI so human, main-agent, and sub-agent messages can be distinguished in multi-agent workflows.

## Best Possible Solution

Define one additive per-message sender metadata contract with explicit trust semantics, persist it in transcripts, expose it through history/live events, carry it through inter-session delivery, and render it in Control UI grouping/avatars.

## Reproduction Assessment

Yes as a source-level capability check rather than a live five-agent UI reproduction: current main lacks durable sender fields across chat.send, sessions.send, transcript persistence, subagent announce delivery, and Control UI rendering.

## Solution Assessment

Yes in direction, but avatars alone are not enough. The maintainable solution is a durable sender metadata contract with protocol, transcript, event, UI, and trust-model follow-through rather than reusing browser-local identity or senderLabel hints.
