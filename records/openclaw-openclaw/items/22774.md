## Summary

Keep this open because current main still lacks a durable per-message sender identity contract across Gateway send APIs, transcript/history/live-event payloads, subagent announce delivery, and Control UI avatar rendering; adjacent provenance, `senderLabel`, and `spawnedBy` fields do not satisfy the multi-agent auditability request.

## What This Changes

Request support for distinct human, main-agent, and sub-agent message attribution in Chat UI for multi-agent workflows.

## Best Possible Solution

Add an additive per-message sender metadata contract with stable sender id/kind/display/source fields and renderable visual hints, persist it in transcripts, expose it through history and live events, carry it through Gateway send and subagent paths, and render it in Control UI without relying on role-only grouping.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main capability gap: sender identity metadata is not accepted by the relevant Gateway send schemas or rendered by Control UI beyond user-role `senderLabel`.

## Solution Assessment

No current implementation is the best solution yet. A role-only or UI-only patch would be too shallow; the maintainable path is an additive durable sender metadata contract across protocol, persistence, events, and rendering.
