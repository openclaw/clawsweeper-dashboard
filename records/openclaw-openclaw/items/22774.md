## Summary

Keep open: current main still has only machine-oriented provenance, browser-local identity, and role/senderLabel-based rendering, not the durable human/main-agent/sub-agent sender identity contract requested for multi-agent auditability. Related open issues cover narrower UI styling or sessions_send attribution, and older PRs were closed unmerged, so none replaces this focused tracker.

## What This Changes

Request durable multi-role sender identity in Chat UI so human, main-agent, and sub-agent messages are distinguishable in multi-agent workflows.

## Best Possible Solution

Define an additive per-message sender metadata contract, persist it in transcripts, expose it through history/live events, carry it through send and subagent paths, and render it in Control UI with clear trust boundaries.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Source inspection on current main shows the relevant schemas, transcript writer, subagent announce path, and Control UI renderer do not carry durable sender identity metadata.

## Solution Assessment

Yes on direction: a durable per-message sender identity contract is the maintainable way to solve the reported auditability gap. A UI-only role/color patch or relying on existing senderLabel recovery would be too shallow because it would not preserve identity across protocol, transcripts, history, live events, and subagent delivery.
