## Summary

Keep open. Current main and v2026.6.10 have partial provenance and `sessions_send` display labels, but they still lack the requested durable human/main-agent/sub-agent sender identity contract across protocol input, transcripts, subagent announcements, history/live payloads, and per-message Control UI avatars.

## What This Changes

Request durable multi-role sender identity in Chat UI so human, main-agent, and sub-agent handoffs render with distinct identity instead of looking like human messages.

## Best Possible Solution

Define one additive trusted per-message sender metadata contract, persist it where appropriate, propagate it through chat/session/subagent delivery and history/live events, and render stable names, avatars, and colors without changing provider-facing roles.

## Reproduction Assessment

Yes, at source level. Current schemas, provenance types, subagent delivery, history projection, and Control UI rendering provide trace fields or derived labels rather than a durable human/main-agent/sub-agent sender identity contract.

## Solution Assessment

Yes in direction, but not as a UI-only avatar patch or new provider message roles. The maintainable solution is additive trusted sender metadata that preserves canonical transcript roles while giving history and UI consumers durable renderable identity.
