## Summary

Keep this open. Current main has useful adjacent provenance, `senderLabel`, and `spawnedBy` plumbing, but it still lacks the requested durable per-message human/main-agent/sub-agent sender identity contract across Gateway send APIs, transcript history, live events, subagent announce delivery, and Control UI avatar rendering.

## What This Changes

Request multi-role sender attribution in Chat UI so human, main-agent, and sub-agent messages render with distinct identity in multi-agent workflows.

## Best Possible Solution

Add an additive per-message sender metadata contract that is persisted in transcripts, exposed through history and live events, carried by `chat.send`/`sessions.send`/`sessions.steer`/subagent announce paths, and rendered by Control UI with stable labels and visual identity.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main capability gap: sender identity metadata is not accepted by the relevant Gateway send schemas or rendered by Control UI beyond user-role `senderLabel`.

## Solution Assessment

No current implementation is the best solution yet. The existing `senderLabel`, provenance, and `spawnedBy` pieces are useful foundations, but the maintainable fix is an additive durable sender metadata contract rather than another role-only or UI-only special case.
