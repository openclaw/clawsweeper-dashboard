## Summary

Keep open. Current main and v2026.6.8 only provide derived `senderLabel` display for some `sessions_send` handoffs; they still lack the requested durable human/main-agent/sub-agent sender identity contract across protocol inputs, transcript persistence, subagent delivery, history/live events, and per-message Control UI avatars.

## What This Changes

Request durable multi-role sender identity in the Chat UI so human, main-agent, and sub-agent messages remain distinguishable in multi-agent workflows.

## Best Possible Solution

Define one additive trusted sender metadata contract, persist it with applicable messages, propagate it through handoffs and subagent announcements, expose it in history/live events, and render stable names, avatars, and colors without changing provider-facing roles.

## Reproduction Assessment

Yes, as a source-level capability check. Current schemas, provenance types, subagent delivery, transcript/history projection, and Control UI rendering provide derived labels and role-based avatars rather than a durable human/main-agent/sub-agent sender identity contract.

## Solution Assessment

Yes in direction, but not as new provider message roles or a UI-only avatar patch. The maintainable solution is additive trusted sender metadata that preserves canonical transcript roles while giving UI and history consumers durable renderable identity.
