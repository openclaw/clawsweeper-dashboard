## Summary

Keep open. Current main has a narrow `sessions_send` forwarding label path, but it still lacks the requested durable human/main-agent/sub-agent sender identity contract across protocol inputs, persisted provenance, subagent delivery, history/live events, and per-message Control UI avatars.

## What This Changes

Request durable multi-role sender attribution in Chat UI so human, main-agent, and sub-agent messages are visually and audibly distinguishable in multi-agent workflows.

## Best Possible Solution

Define one additive trusted sender metadata shape, persist it with each applicable message, propagate it through session handoffs and subagent announcements, expose it in history and live events, and render stable names/avatars/colors without changing provider-facing message roles.

## Reproduction Assessment

Yes, as a source-level capability check. Current schemas, provenance types, subagent delivery, and Control UI rendering provide only derived labels and role-based avatars, not durable human/main-agent/sub-agent sender identity.

## Solution Assessment

Yes in direction, but not by adding provider message roles. The maintainable path is additive trusted sender metadata that preserves canonical transcript roles while giving UI/history consumers durable renderable identity.
