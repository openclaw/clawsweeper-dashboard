## Summary

Keep this open: current main still has trace provenance, browser-local identity, and user-role `senderLabel` rendering, but not the durable human/main-agent/sub-agent sender identity contract requested for multi-agent auditability. The related items remain narrower or broader trackers, so they should be coordinated with rather than treated as replacements.

## What This Changes

The issue requests durable multi-role sender identity in Chat UI so human, main-agent, and sub-agent messages render with distinct, auditable identities.

## Best Possible Solution

Add an additive durable per-message sender metadata contract, persist it in transcripts, expose it through history/live events, carry it through send and subagent paths, and render it in Control UI with explicit trust boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source and test inspection give a high-confidence current-main capability gap: the relevant Gateway schemas, transcript writer, subagent announce path, and Control UI renderer do not carry durable sender identity metadata.

## Solution Assessment

Yes, the durable sender-identity direction is the maintainable solution. A UI-only role/color patch or existing `senderLabel` recovery would be too shallow because it would not preserve audit identity across protocol, transcripts, history, live events, and subagent delivery.
