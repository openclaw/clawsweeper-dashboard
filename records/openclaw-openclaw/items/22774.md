## Summary

Keep open: current main and v2026.5.12 still lack the durable human/main-agent/sub-agent sender identity contract requested here; the related issues and PRs cover adjacent sender labels, trusted provenance, or Codex-specific history preservation rather than this focused Control UI/protocol attribution path.

## What This Changes

Request durable multi-role sender attribution in the Chat UI so human, main-agent, and sub-agent messages remain distinguishable in multi-agent workflows.

## Best Possible Solution

Define an additive per-message sender metadata contract with explicit trust semantics, persist it in transcripts, expose it through history/live events, carry it through chat.send, sessions.send/sessions.steer, and subagent completion delivery, then render it in Control UI avatars/grouping.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main and v2026.5.12 lack durable sender identity fields across Gateway schemas, transcript persistence, subagent delivery, and Control UI rendering.

## Solution Assessment

Yes in direction, but avatars alone are not enough. The maintainable solution is a durable sender metadata contract with protocol, transcript, event, UI, and trust-model follow-through rather than reusing browser-local identity or senderLabel hints.
