## Label Justifications

- `P2`: Missing sender attribution materially impairs multi-agent auditability and debugging but does not make the core runtime unusable.

## Summary

Keep open. Current `main` and v2026.6.6 provide a narrow derived label for `sessions_send` messages, but they still lack the requested durable sender identity across protocol inputs, persisted provenance, subagent completion delivery, history/live events, and per-message Control UI rendering; the open regression report only tracks breakage in that narrower forwarding display.

## What This Changes

Requests durable human, main-agent, and sub-agent sender identity in multi-agent Chat UI messages.

## Best Possible Solution

Define one additive trusted sender metadata shape, persist it with each applicable message, propagate it through session handoffs and subagent announcements, expose it in history and live events, and render stable names and avatars without changing provider-facing message roles.

## Reproduction Assessment

Yes, from source inspection. Current schemas, provenance, and UI models provide a derived forwarding label but no durable human/main-agent/sub-agent identity record across all requested paths.

## Solution Assessment

Yes in direction, but not as additional provider message roles. The narrowest maintainable solution is additive trusted sender metadata while preserving canonical `user` and `assistant` transcript roles.
