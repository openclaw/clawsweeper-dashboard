## Summary

Keep open: current main still has trace provenance, browser-local identity, and user-role `senderLabel` rendering, but not the durable human/main-agent/sub-agent sender identity contract requested for multi-agent auditability. The related issues and PRs are narrower, broader, still open, or closed unmerged rather than replacements.

## What This Changes

Add durable multi-role sender identity in Chat UI so human, main-agent, and sub-agent messages render distinctly and remain auditable across multi-agent workflows.

## Best Possible Solution

Define an additive durable per-message sender metadata contract, persist it in transcripts, expose it through history/live events, carry it through send and subagent paths, and render it in Control UI with explicit trust boundaries.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives a high-confidence capability gap: the relevant Gateway schemas, transcript writer, subagent announce path, and Control UI renderer do not carry durable sender identity metadata.

## Solution Assessment

Yes in direction, with a stronger contract: durable per-message sender metadata is the maintainable solution. A UI-only role/color patch or existing `senderLabel` recovery would be too shallow because it would not preserve audit identity across protocol, transcripts, history, live events, and subagent delivery.
