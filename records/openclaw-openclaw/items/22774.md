## Summary

Keep this open: current main still has provenance, `senderLabel`, `spawnedBy`, and browser-local identity pieces, but not the durable per-message human/main-agent/sub-agent sender identity contract requested for multi-agent chat auditability.

## What This Changes

The issue requests Chat UI support for distinct human, main-agent, and sub-agent message identity in multi-agent workflows.

## Best Possible Solution

Define an additive per-message sender metadata contract with stable sender id/kind/display/source fields and renderable visual hints, persist it in transcripts, expose it through history and live events, and render it in Control UI without relying on role-only grouping.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main capability gap: sender identity metadata is not accepted by the relevant Gateway send schemas or rendered by Control UI beyond user-role senderLabel.

## Solution Assessment

No current implementation is the best solution yet. A role-only or UI-only patch would be too shallow; the maintainable path is an additive durable sender metadata contract across protocol, persistence, events, and rendering.
