## Summary

Keep open. Current main partially improves final payload selection, but it still lacks the requested opt-in coalescing/final-only mode and the block-streaming path still flushes buffered text before tool execution; https://github.com/openclaw/openclaw/issues/25592 overlaps but tracks a broader leak/explicit-send policy question rather than replacing this focused coalescing request.

## What This Changes

Request an opt-in mode to merge or defer assistant text blocks emitted between tool calls so one agent turn produces one outbound chat message.

## Best Possible Solution

Add a maintainer-approved opt-in delivery mode that covers both final payload assembly and block-streamed tool boundaries while preserving media, reply targets, reasoning/status metadata, and channel limits.

## Reproduction Assessment

Yes, at source level: current code and tests still flush buffered block replies before tool execution, which preserves split delivery in the block-streaming path. I did not run a live Telegram or iMessage reproduction in this read-only review.

## Solution Assessment

Unclear. The requested config option is plausible, but maintainers still need to choose the durable semantics among coalesce-all, final-only suppression, or explicit message-tool delivery.
