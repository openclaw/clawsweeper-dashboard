## Summary

Keep open. Current main still lacks the requested opt-in coalescing/final-only delivery: final assistant text blocks remain separate reply payloads, generic dispatch sends each payload separately, block streaming still flushes at tool-start boundaries, and no matching config surface exists.

## What This Changes

Request an opt-in agent or channel setting that merges tool-interleaved assistant text blocks from one agent turn into one outbound message.

## Best Possible Solution

Define one explicit core reply-delivery mode that covers both final payload assembly and block-streamed text around tool execution, then add schema/docs and regression coverage for text-to-tool-to-text turns.

## Reproduction Assessment

Not applicable as a feature request rather than a documented-behavior bug. Source inspection does give a high-confidence current behavior path: multiple assistantTexts become multiple reply payloads, and block streaming flushes before tool execution.

## Solution Assessment

Unclear. The reply pipeline is the right area, but maintainers still need to choose the delivery semantics and config surface before a narrow implementation would be safe.
