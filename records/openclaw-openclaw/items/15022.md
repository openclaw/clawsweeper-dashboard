## Summary

Keep open. Current main and the latest shipped tag still preserve multiple final reply items and flush block replies at tool-start boundaries, while the related broader channel-leak issue remains open for maintainer policy decisions rather than providing a merged fix.

## What This Changes

The issue requests an opt-in reply-delivery mode that coalesces tool-interleaved assistant text blocks into one outbound channel message.

## Best Possible Solution

Define one explicit opt-in reply-delivery policy covering final payload assembly and block-streamed tool-boundary text, with schema/docs and regression tests for text-to-tool-to-text turns.

## Reproduction Assessment

Yes, at source level: current main still flushes block replies before tool execution and dispatches each final reply item separately when multiple payloads remain. The recent BlueBubbles/iMessage comment also provides a real-channel reproduction on OpenClaw 2026.5.3.

## Solution Assessment

Unclear. The reply pipeline is the right surface and the closed hold/resume PR is useful prior art, but maintainers still need to choose coalesce-all, final-only, or suppress-intermediate semantics and decide the config scope.
