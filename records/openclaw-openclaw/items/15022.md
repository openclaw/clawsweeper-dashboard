## Summary

Keep open. Current main still lacks the requested configurable coalescing, final-only, or suppression mode for tool-interleaved assistant text; the existing code only partially canonicalizes final payloads and still preserves a block-reply flush boundary before tool execution.

## What This Changes

The issue requests a configurable way to merge, defer, or suppress assistant text blocks emitted between tool calls so one agent turn does not produce many outbound channel messages.

## Best Possible Solution

Define one explicit core reply-delivery policy for tool-interleaved turns, then wire config schema, docs, and regression coverage across final payload assembly and block replies.

## Reproduction Assessment

Yes for the current behavior path, though this is a feature request rather than a broken defined contract. Current main still flushes block replies at tool_execution_start, has no requested coalescing config, and the May 5 BlueBubbles comment reports the same behavior on OpenClaw 2026.5.3.

## Solution Assessment

Unclear. The reply pipeline is the right surface and https://github.com/openclaw/openclaw/pull/16275 is useful prior art, but maintainers need to choose delivery semantics and metadata boundaries before a narrow implementation is safe.
