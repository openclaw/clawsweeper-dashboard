## Summary

Keep open. Current main has partial protections, including canonical final-answer selection and enforced final-tag handling for selected provider paths, but it still lacks the requested opt-in coalesce/final-only delivery policy and still preserves block-streaming flushes at tool boundaries.

## What This Changes

The issue asks for an opt-in policy to merge or defer text blocks emitted around tool calls into one outbound channel message.

## Best Possible Solution

Define one maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, while coordinating broader leak/security policy under https://github.com/openclaw/openclaw/issues/25592.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main has no requested coalescing policy or hold/resume API, and tests still assert pre-tool buffered text is delivered at the tool boundary. I did not run a live channel scenario in this read-only review.

## Solution Assessment

Unclear. The reply pipeline is the right surface, but the requested config shape is a product decision because OpenClaw must choose merge-all, final-only, suppress-intermediate, or per-channel behavior before implementation.
