## Summary

Keep open. Current main partially reduces final-payload fragmentation when a structured final answer is available, but it still lacks the requested configurable coalesce/final-only/suppress mode and the block-streamed path still flushes at tool-start boundaries; the recent BlueBubbles/iMessage comment also keeps this from being stale.

## What This Changes

Request an opt-in reply delivery mode that coalesces or suppresses text blocks interleaved with tool calls into one outbound channel message per agent turn.

## Best Possible Solution

Define one explicit opt-in core reply-delivery policy for tool-interleaved turns, then wire config schema, docs, and regression coverage across final payload assembly and block replies.

## Reproduction Assessment

Yes, at source level: current main still flushes block replies before tool execution and still sends each final reply item separately when multiple payloads remain. I did not run a live channel repro in this read-only review, but the May 5 BlueBubbles/iMessage comment provides a recent real-channel reproduction on OpenClaw 2026.5.3.

## Solution Assessment

Unclear. The reply pipeline is the right surface and the closed hold/resume pull request is useful prior art, but maintainers still need to choose coalesce-all, final-only, or suppress-intermediate semantics and decide whether the scope is global, agent-level, channel-level, or both.
