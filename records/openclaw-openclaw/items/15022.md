## Summary

Keep open. Current main has some final-answer suppression improvements, but it still lacks the requested opt-in coalescing/final-only delivery mode and still preserves separate block replies around tool-interleaved text in current source/tests.

## What This Changes

The issue requests an opt-in way to coalesce tool-interleaved assistant text blocks from one agent turn into one outbound channel message.

## Best Possible Solution

Keep this as the focused core reply-pipeline product decision, then implement one opt-in delivery mode covering final payloads and block-streamed tool-interleaved text with regression coverage.

## Reproduction Assessment

Yes, at source level. Current tests still expect separate tool-interleaved text_end block replies, current tool-start handling flushes before execution, and the requested config/hook surface is absent; I did not run live channel proof in this read-only review.

## Solution Assessment

Unclear until maintainers choose the desired product semantics. The maintainable path should be a single opt-in reply-delivery policy rather than a narrow coalescer-only patch that ignores final payloads or metadata boundaries.
