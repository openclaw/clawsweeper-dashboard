## Summary

Keep open. Current main still has source-reproducible paths that split tool-interleaved assistant text into multiple outbound messages, while partial final-answer/commentary safeguards do not add the requested opt-in coalescing or final-only delivery policy.

## What This Changes

The issue requests a configurable way to merge tool-interleaved assistant text blocks from one agent turn into a single outbound channel message.

## Best Possible Solution

Define a maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, coordinate with https://github.com/openclaw/openclaw/issues/25592 and related PRs, then land schema/docs, regression tests, and live channel proof.

## Reproduction Assessment

Yes, at source level. Current main still preserves multiple unphased assistant text entries as separate final payloads and still flushes buffered block-streamed text at tool-start boundaries; I did not run live Telegram or iMessage proof in this read-only review.

## Solution Assessment

Unclear. Coalescing is a plausible option, but the safest solution depends on a maintainer policy choice between merge-all, final-only, suppression, explicit-send, or channel-specific delivery semantics.
