## Summary

Keep open. Current main has partial final-answer safeguards, but it still lacks the requested opt-in coalescing/final-only policy for unphased final payloads and block-streamed text flushed around tool execution.

## What This Changes

The issue requests an opt-in policy to merge or defer tool-interleaved assistant text blocks so a single agent turn does not produce many outbound channel messages.

## Best Possible Solution

Define a maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, coordinate it with https://github.com/openclaw/openclaw/issues/25592 and https://github.com/openclaw/openclaw/pull/58445, then land schema/docs, regression tests, and live channel proof.

## Reproduction Assessment

Yes, at source level. Current main can still preserve multiple unphased assistant text entries as separate final payloads and still flush buffered block-streamed text at tool-start boundaries; I did not run live Telegram or BlueBubbles proof in this read-only review.

## Solution Assessment

Unclear. Coalescing is a plausible option, but the safest maintainable solution depends on a maintainer choice between merge-all, final-only, suppression, explicit-send, or channel-specific delivery semantics.
