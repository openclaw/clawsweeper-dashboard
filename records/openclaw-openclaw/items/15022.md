## Summary

Keep open. Current main still does not provide the requested opt-in coalescing or final-only delivery mode for tool-interleaved assistant text; the block-streamed path still flushes at tool boundaries, and the open related PR only addresses ordering/direct-send behavior rather than this delivery policy.

## What This Changes

The issue requests a config-controlled mode that merges text blocks interleaved with tool calls into one outbound channel message.

## Best Possible Solution

Keep this as the canonical coalescing/final-only delivery policy issue, then implement an opt-in core reply-pipeline mode with schema, docs, and regression coverage once maintainers choose the semantics.

## Reproduction Assessment

Yes at source level: current main still flushes block replies before tool execution and still lacks the requested coalescing/final-only config surface. I did not run a live Telegram or BlueBubbles reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the delivery contract. A hold/defer or final-only mode is plausible, but it should be opt-in and must cover both final payload assembly and block-streamed tool boundaries without breaking existing message-boundary semantics.
