## Summary

Keep open. Current main still lacks the requested opt-in coalescing or final-only delivery policy for tool-interleaved assistant text; the related open PRs are adjacent hooks or partial mitigations rather than a merged core delivery fix.

## What This Changes

The issue requests a configurable policy that merges multiple assistant text blocks from one tool-using agent turn into a single outbound channel message.

## Best Possible Solution

Define a maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, coordinate it with https://github.com/openclaw/openclaw/issues/25592 and https://github.com/openclaw/openclaw/pull/58445, then land schema/docs plus regression and live channel proof.

Yes, at source level. Current main still preserves multiple assistant text entries as separate final reply payloads and still flushes buffered block-streamed text at tool-start boundaries; I did not run a live Telegram or BlueBubbles turn in this read-only review.

## Solution Assessment

Unclear. The reply pipeline is the right area, but the exact merge-versus-final-only-versus-suppress policy and config shape need maintainer product judgment before implementation is safe.
