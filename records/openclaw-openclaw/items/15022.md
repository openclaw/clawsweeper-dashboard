## Summary

Keep open. Current main has partial final-payload canonicalization, but it still lacks a configurable coalescing/final-only mode that covers text emitted around tool calls; the block-reply path still flushes before tool execution, and a recent iMessage/BlueBubbles repro confirms the gap remains user-visible.

## What This Changes

Add a configurable reply-delivery mode that merges or defers assistant text blocks emitted around tool calls into one outbound channel message.

## Best Possible Solution

Define one explicit core reply-delivery mode covering final payload assembly and block replies around tool execution, then add schema/docs and regression coverage for text-tool-text turns.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests new opt-in delivery behavior. Source and tests give a high-confidence current behavior path: tool execution starts still flush block replies, and no requested coalescing config exists.

## Solution Assessment

Unclear. The reply pipeline is the right surface, but maintainers need to choose coalesce versus final-only versus suppression semantics before a narrow implementation is safe.
