## Summary

Keep this issue open: current main reduces some final-payload replay by preferring canonical final-answer text, but the block-streamed path still deliberately flushes at tool start and preserves separate logical assistant blocks. The broader open suppression request at https://github.com/openclaw/openclaw/issues/25592 is related but asks for final-only or explicit-send semantics, so it does not supersede this distinct request to retain and coalesce the text.

## What This Changes

Add an opt-in core delivery policy that combines assistant text emitted around tool calls into one outbound channel message.

## Best Possible Solution

Define one opt-in turn-delivery policy in the shared reply pipeline, with explicit `all`, `coalesce`, and possibly `final-only` semantics, while preserving rich-payload and reply-target boundaries and applying the policy consistently to streamed and final replies.

## Reproduction Assessment

Yes, at source level with high confidence. Current main still flushes block replies at tool start and explicitly preserves logical assistant-message boundaries, matching the reported multi-bubble path; Telegram and BlueBubbles reporters also supplied concrete user-level reproduction steps.

## Solution Assessment

Unclear as a standalone boolean. Coalescing is a maintainable and distinct desired behavior, but the best solution is a canonical delivery-policy model shared with final-only suppression rather than another isolated setting competing with existing streaming controls.
