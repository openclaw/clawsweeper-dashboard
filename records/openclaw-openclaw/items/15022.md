## Summary

Keep open. Current main still lacks the requested opt-in coalesce/final-only delivery policy for tool-interleaved assistant text: final payload assembly only collapses some canonical final-answer cases, while block replies still preserve boundaries at tool start and across assistant message indexes. The related closed issues point back to this issue as the canonical thread, and https://github.com/openclaw/openclaw/pull/16275 is useful unmerged prior art rather than a shipped fix.

## What This Changes

Add a configurable reply-delivery mode that coalesces text blocks interleaved with tool calls into one outbound channel message.

## Best Possible Solution

Keep this as the canonical core issue and define one opt-in reply-delivery policy with config schema, docs, and regression coverage across final payload assembly and block-streamed tool turns.

## Reproduction Assessment

Yes, at source level for the current behavior path, though this is a feature request rather than a broken defined contract. Current main still flushes block replies at tool start, preserves logical assistant block boundaries, and has no requested config surface.

## Solution Assessment

Unclear. The reply pipeline is the right surface and the closed hold/resume PR is useful prior art, but maintainers still need to choose the delivery semantics and config scope before implementation.
