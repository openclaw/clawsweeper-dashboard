## Summary

Keep open. Current main and the latest shipped release still lack the requested opt-in policy for coalescing or final-only delivery across tool-interleaved assistant text; current code has partial final-answer selection, but the block-streamed path still preserves a pre-tool flush boundary and the fallback final payload can still deliver multiple assistant text entries. The broader leak tracker at https://github.com/openclaw/openclaw/issues/25592 and related PRs are useful context, but they do not make this narrower coalescing request implemented.

## What This Changes

The issue requests an opt-in configuration policy that merges or defers tool-interleaved assistant text blocks so one agent turn produces one outbound channel message.

## Best Possible Solution

Define one maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, coordinate it with https://github.com/openclaw/openclaw/issues/25592, and land schema/docs plus regression and live transport proof for text-to-tool-to-text turns.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main has no requested coalescing policy or hold/resume API, final payloads can still preserve multiple text entries, and tests still assert pre-tool buffered text is delivered at the tool boundary.

## Solution Assessment

Unclear. The reply pipeline is the right surface, but the exact config and delivery semantics need maintainer product judgment before a narrow implementation is safe.
