## Summary

Keep open. Current main has useful partial mitigations for phased final-answer text, but it still lacks the requested opt-in coalescing/final-only delivery policy for tool-interleaved assistant text, and the remaining behavior intersects broader delivery-policy work and open adjacent PRs.

## What This Changes

Request an opt-in config/policy that merges interleaved assistant text blocks from one agent turn into one outbound channel message.

## Best Possible Solution

Define a maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, coordinate it with https://github.com/openclaw/openclaw/issues/25592 and the open related PRs, then land schema/docs, regression tests, and live channel proof.

## Reproduction Assessment

Yes, at source level. Current main still flushes block-streamed text at tool-start boundaries and can preserve multiple unphased assistant text entries as separate final payloads; I did not run live Telegram or BlueBubbles proof in this read-only review.

## Solution Assessment

Unclear. Coalescing is a plausible direction, but current partial final-answer mitigations and the broader leakage/security discussion mean maintainers need to choose merge-all, final-only, suppress, or explicit-send semantics before implementation.
