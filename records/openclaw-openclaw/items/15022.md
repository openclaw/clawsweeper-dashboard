## Summary

Keep open: current main and the latest release still preserve the split behavior, while the requested coalescing/final-only turn delivery remains a product/config decision rather than implemented behavior.

## What This Changes

The issue requests an option to merge or defer tool-interleaved assistant text blocks so one agent turn does not produce many outbound channel messages.

## Best Possible Solution

Keep this as the focused core reply-pipeline product decision: define an opt-in coalescing or final-only delivery contract, then implement it across final payload assembly and block-streamed tool-boundary delivery with regression coverage.

## Reproduction Assessment

Yes. Current-main source and tests show a source-level reproduction path: final payload assembly can keep multiple assistant text blocks as separate reply payloads, and block streaming flushes buffered text before `tool_execution_start`; I did not run a live Telegram or iMessage scenario in this read-only review.

## Solution Assessment

Unclear. The requested coalescing/final-only mode is a plausible solution, but the best exact behavior needs maintainer direction because existing `message_tool` visible-reply mode solves a different explicit-send policy and the new config must preserve reply/media/reasoning boundaries.
