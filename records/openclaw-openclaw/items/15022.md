## Summary

Keep open. Current main and v2026.5.22 still lack the requested opt-in coalescing or final-only delivery policy; the block-streamed path still flushes buffered text before tool execution, and final payload assembly only partially collapses text when final-answer metadata is available.

## What This Changes

Request an opt-in configuration or delivery policy that merges tool-interleaved assistant text blocks from one agent turn into a single outbound channel message.

## Best Possible Solution

Define a maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, then land schema/docs, regression tests, and live channel proof in coordination with https://github.com/openclaw/openclaw/issues/25592 and https://github.com/openclaw/openclaw/pull/58445.

## Reproduction Assessment

Yes, at source level. Current main still flushes buffered block-streamed text before tool execution, and current tests explicitly expect that pre-tool delivery boundary; this read-only review did not run live Telegram or iMessage proof.

## Solution Assessment

Unclear. Coalescing is plausible, but the maintainable solution depends on a product choice between merge-all, final-only, suppression, explicit-send, or plugin-hook delivery semantics.
