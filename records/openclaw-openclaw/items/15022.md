## Summary

Keep open. Current main partially improves final payload assembly when structured final-answer metadata exists, but it still lacks the requested opt-in coalesce/final-only delivery policy and still preserves the block-streamed pre-tool flush path; related PRs are closed, unmerged, or only partial.

## What This Changes

The issue requests an opt-in delivery mode that coalesces or suppresses tool-interleaved assistant text blocks so one agent turn does not create many outbound channel messages.

## Best Possible Solution

Define a maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, then land schema/docs, regression tests, and live channel proof while coordinating with https://github.com/openclaw/openclaw/issues/25592.

## Reproduction Assessment

Yes, at source level. Current main still flushes buffered block-streamed text before tool execution, and current tests explicitly expect that pre-tool delivery boundary; I did not run live Telegram or iMessage proof in this read-only review.

## Solution Assessment

Unclear. Coalescing is plausible, but the maintainable fix depends on a maintainer choice between merge-all, final-only, suppression, explicit-send, or plugin-hook semantics and the upgrade-safe default.
