## Summary

Keep open. Current main and the latest shipped release still lack the requested opt-in coalesce/final-only delivery policy; the broader channel-leak thread and open PRs are relevant, but they do not fully replace this narrower reply-coalescing request.

## What This Changes

The issue requests an opt-in reply-delivery mode that coalesces assistant text blocks interleaved with tool calls into one outbound channel message.

## Best Possible Solution

Define one explicit opt-in reply-delivery policy covering final payload assembly and block-streamed tool-boundary text, coordinate it with https://github.com/openclaw/openclaw/issues/25592, and add schema, docs, and regression tests for text-to-tool-to-text turns.

## Reproduction Assessment

Yes, at source level: current main still has no requested config or hold/resume API, flushes block replies before tool execution, and tests assert buffered pre-tool text is delivered at that boundary. I did not run a live channel scenario in this read-only review.

## Solution Assessment

Unclear. The reply pipeline is the right surface and the closed hold/resume PR is useful prior art, but maintainers still need to choose the delivery semantics and config scope before a safe implementation can land.
