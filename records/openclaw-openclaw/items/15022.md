## Summary

Keep open. Current main still preserves multiple assistant text blocks as separate outbound reply payloads and still flushes streamed text before tool execution, while the directly related implementation PR was closed unmerged. The requested opt-in coalescing/final-only delivery remains a real core reply-pipeline feature request.

## What This Changes

Request an opt-in setting that merges text blocks emitted across one tool-interleaved agent turn into a single outbound channel message.

## Best Possible Solution

Keep this open as the focused core reply-pipeline feature. The maintainable end state is an opt-in delivery mode that covers both final embedded-run assistantTexts and block-streamed text around sequential and parallel tool execution, with config schema/docs and regression tests for text -> tool -> text -> tool -> text turns.

## Reproduction Assessment

Yes. A source-level reproduction is clear: simulate a turn with assistant text before tool_execution_start and the existing tests show the block reply flush at that boundary; final payload assembly also maps multiple assistantTexts to multiple reply items.

## Solution Assessment

Unclear. The requested opt-in coalescing/final-only behavior is a plausible direction, and PR #16275 is useful prior art, but the exact public config and metadata-boundary behavior need maintainer design before a safe implementation.
