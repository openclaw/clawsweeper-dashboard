## Summary

Keep open. Current main still lacks the requested opt-in coalescing or final-only delivery policy for tool-interleaved assistant text, and the latest related PR is an open plugin-hook proposal rather than a merged solution for this core delivery behavior.

## What This Changes

The issue asks for a configurable way to merge or suppress multiple assistant text blocks from one tool-heavy agent turn before sending outbound channel messages.

## Best Possible Solution

Define a maintainer-approved opt-in reply-delivery policy that covers both final payload assembly and block-streamed tool-boundary text, coordinate it with https://github.com/openclaw/openclaw/issues/25592 and https://github.com/openclaw/openclaw/pull/58445, and land schema/docs plus regression and live channel proof.

## Reproduction Assessment

Yes, at source level. Current main lacks the requested coalescing/final-only policy, still preserves multiple final reply items in fallback assembly, and still flushes block-streamed text at tool-start boundaries.

## Solution Assessment

Unclear. A reply-pipeline policy is the right area, but the exact config shape and merge-versus-suppress semantics need maintainer product judgment before a maintainable implementation is safe.
