## Summary

Keep open. Current main has partial mitigations such as canonical final-answer selection, but it still lacks the requested opt-in coalescing/final-only delivery policy and still preserves the block-streaming flush boundary before tool execution.

## What This Changes

The issue requests a configurable policy to merge or otherwise coalesce assistant text blocks from one tool-interleaved agent turn into a single outbound channel message.

## Best Possible Solution

Define one maintainer-approved opt-in reply-delivery policy that covers final payload assembly and block-streamed tool-boundary text, with docs/config schema and regression coverage for text -> tool -> text turns.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main has no requested coalescing policy or hold/resume API, final payloads can still preserve multiple text blocks, and tests still assert pre-tool buffered text is delivered at the tool boundary.

## Solution Assessment

Unclear. The reply pipeline is the right surface, but the requested config shape needs maintainer product judgment because OpenClaw must choose merge-all, final-only, suppression, or per-channel behavior before implementation.
