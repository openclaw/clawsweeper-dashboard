## Summary

Keep open. Current main improves non-streaming final delivery by preferring the canonical final answer, but block streaming still deliberately flushes before tool execution, so tool-interleaved narration can remain split across outbound messages. The requested semantics and configuration shape require a maintainer product decision rather than an automatic repair.

## What This Changes

Requests an opt-in delivery mode that combines assistant text emitted around tool calls into one outbound channel message.

## Best Possible Solution

Define one opt-in turn-level delivery policy in the shared reply pipeline, apply it consistently to streamed and non-streamed runs, and preserve explicit boundaries for metadata-bearing, media, notice, reply-target, and size-limited payloads with sequential and parallel tool coverage.

## Reproduction Assessment

Yes, at source level: current main explicitly flushes block replies before tool-start processing, and tests preserve that boundary. No fresh live transport reproduction was performed during this read-only review.

## Solution Assessment

No, the proposed config concept is reasonable but an idle-timer-only or coalescer-only change is not the best complete fix. The maintainable solution must define shared turn-level semantics across block-streamed and final delivery while protecting payload metadata boundaries.
