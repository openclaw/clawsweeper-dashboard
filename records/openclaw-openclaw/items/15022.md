## Summary

Keep open. Current main and the latest release still only partially canonicalize final payloads and do not provide the requested configurable coalesce/final-only/suppress mode for tool-interleaved assistant text, while the block-reply path still preserves flush boundaries around tool execution.

## What This Changes

The issue requests a config-backed reply-delivery mode that merges or suppresses tool-interleaved assistant text blocks so one agent turn does not produce many outbound channel messages.

## Best Possible Solution

Define one explicit opt-in core reply-delivery policy for tool-interleaved turns, then wire config schema, docs, and regression coverage across final payload assembly and block replies.

## Reproduction Assessment

Yes, for the current behavior path at source level, though this is a feature request rather than a broken defined contract. Current main still flushes block replies at tool start, lacks the requested config surface, and the May 5 BlueBubbles comment reports the same behavior on OpenClaw 2026.5.3.

## Solution Assessment

Unclear. The reply pipeline is the right surface and https://github.com/openclaw/openclaw/pull/16275 is useful prior art, but maintainers need to choose coalesce-all, final-only, or suppress-intermediate semantics and decide whether scope is global, agent-level, channel-level, or both.
