## Summary

Keep open. Current main still lacks the requested configurable coalescing/final-only delivery policy: final payload assembly only prefers a canonical final answer when one is available, the fallback can still preserve multiple assistant text entries, and block-streamed text still flushes at tool-start boundaries. The broader leak tracker at https://github.com/openclaw/openclaw/issues/25592 and the closed unmerged hold/resume PR are useful context, but they do not make this narrower coalescing request implemented.

## What This Changes

Request to add a configurable policy that merges or defers tool-interleaved assistant text blocks so one agent turn produces one outbound channel message.

## Best Possible Solution

Define one maintainer-approved opt-in reply-delivery policy covering both final payload assembly and block-streamed tool-boundary text, coordinate it with https://github.com/openclaw/openclaw/issues/25592, then land schema/docs plus regression and live transport proof for text-to-tool-to-text turns.

## Reproduction Assessment

Yes, at source level. Current main has no requested coalescing policy or hold/resume API, final payload fallback can still preserve multiple text entries, and tests still assert pre-tool buffered text is delivered at the tool boundary.

## Solution Assessment

Unclear. The reply pipeline is the right surface, but the exact config and delivery semantics need maintainer product judgment before a narrow implementation is safe.
