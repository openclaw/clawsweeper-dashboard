## Summary

Keep open: current main still lacks an opt-in reply-pipeline mode that coalesces or defers tool-interleaved assistant text into a single outbound message, and the existing code still preserves the split boundaries that this feature asks to change.

## What This Changes

Request an opt-in delivery setting that merges assistant text blocks around tool calls into one outbound channel message.

## Best Possible Solution

Add a documented opt-in core reply-pipeline delivery mode that covers both final embedded-run payload assembly and block-streamed text around sequential and parallel tool execution while preserving non-text delivery metadata.

## Reproduction Assessment

Yes, from source inspection: current main still preserves multiple assistant text blocks as separate final reply items and flushes block replies at tool_execution_start. I did not run a live Telegram reproduction, but the current code path is clear.

## Solution Assessment

Unclear until maintainers choose the delivery contract. A pipeline-level opt-in mode with hold/defer semantics is likely safer than a Telegram-only workaround, but merge-all versus final-only behavior needs an owner decision.
