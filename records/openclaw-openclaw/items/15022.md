## Summary

Keep open. Current main has a partial final-answer mitigation, but it still lacks the requested configurable coalesce/final-only/suppress delivery policy, and the block-streamed path still flushes at tool-start boundaries; the recent BlueBubbles/iMessage reproduction keeps the report current.

## What This Changes

Requests an opt-in delivery policy/config to merge or suppress assistant text blocks interleaved with tool calls so one agent turn does not create many chat messages.

## Best Possible Solution

Define one explicit opt-in reply-delivery policy covering final payload assembly and block-streamed tool-boundary text, with config schema/docs and regression tests for text -> tool -> text turns.

## Reproduction Assessment

Yes, at source level: current main still flushes block replies before tool execution and dispatches each final reply item separately when multiple payloads remain. I did not run a live channel repro, but the May 5 BlueBubbles/iMessage comment provides a recent real-channel reproduction on OpenClaw 2026.5.3.

## Solution Assessment

Unclear. The reply pipeline is the right surface and the closed hold/resume PR is useful prior art, but maintainers still need to choose coalesce-all, final-only, or suppress-intermediate semantics and decide the config scope.
