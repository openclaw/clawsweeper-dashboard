## Summary

Keep open. Current main has partial final-payload canonicalization, but the requested configurable coalescing/final-only delivery mode is still missing; block replies still flush before tool execution, tests assert that boundary, and the recent iMessage/BlueBubbles report shows the delivery-policy gap remains active.

## What This Changes

The issue requests configurable coalescing of assistant text blocks emitted around tool calls into one outbound channel message.

## Best Possible Solution

Define one explicit core reply-delivery mode covering final payload assembly and block replies around tool execution, then add config schema, docs, and regression coverage for text-tool-text turns.

## Reproduction Assessment

Yes for the current behavior path, but not as a strict bug in an already-defined contract. Source inspection and the BlueBubbles repro show that block replies still flush before tool execution and no requested coalescing config exists.

## Solution Assessment

Unclear. The reply pipeline is the right surface and the closed hold/resume PR is useful prior art, but maintainers still need to choose the delivery semantics and metadata boundaries before a narrow implementation is safe.
