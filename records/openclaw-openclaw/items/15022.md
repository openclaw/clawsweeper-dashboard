## Summary

Keep open. Current main and v2026.5.12 include a partial final-payload collapse, but they still do not provide the requested opt-in coalesce/final-only delivery policy for tool-interleaved text, and block-streamed replies still preserve tool-boundary splits.

## What This Changes

The issue asks for configurable merging or final-only delivery of multiple assistant text blocks from one tool-heavy agent turn into a single outbound channel message.

## Best Possible Solution

Define one explicit opt-in reply-delivery policy covering both final embedded-run payload assembly and block-streamed tool-boundary text, coordinated with https://github.com/openclaw/openclaw/issues/25592 and backed by schema, docs, and regression tests for text-to-tool-to-text turns.

## Reproduction Assessment

Yes, at source level: current main still has no requested config or hold/resume API, block-streamed assistant blocks are deliberately kept separate, and tests assert buffered pre-tool text is delivered at the tool boundary. I did not run a live channel scenario in this read-only review.

## Solution Assessment

Unclear. The reply pipeline is the right surface and the closed hold/resume PR is useful prior art, but the safe fix needs a maintainer decision on delivery semantics and config scope before implementation.
