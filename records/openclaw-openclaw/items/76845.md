## Summary

Keep open. Current origin/main still drops trusted elevated defaults before exec approval follow-up agent continuations reach the embedded runner, while this PR threads that state through an internal, session-scoped token and I found no blocking functional or security issue in the diff.

## What This Changes

The PR adds a short-lived exec-approval follow-up token to carry trusted elevated defaults into gateway agent continuations, forwards `bashElevated` into embedded runs, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land the internal token-based propagation after CI confirms it, keeping elevated-default transfer session-scoped and out of the public gateway schema.

## Reproduction Assessment

Yes, source-reproducible. The reported flow maps directly to current origin/main because the approval follow-up `agent` RPC and gateway dispatch preserve route context but have no path for `bashElevated`, so a second elevated exec can hit the existing unavailable gate.

## Solution Assessment

Yes. The PR is the narrow maintainable fix: it propagates only trusted elevated defaults through a short-lived session-bound internal token, avoids a public schema field, and does not reuse an `allow-once` approval decision.
