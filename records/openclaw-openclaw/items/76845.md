## Summary

Keep open. The PR targets a valid current-main bug and uses the right internal elevated-defaults seam, but the patch currently makes approval follow-up agent idempotency unstable by appending a fresh token to the Gateway dedupe key, and the external real-behavior proof still does not show the full follow-up flow.

## What This Changes

The branch preserves elevated exec defaults across approval follow-up agent runs with a session-scoped one-use token, forwards `bashElevated` into embedded runs, and adds regression tests plus a changelog entry.

## Best Possible Solution

Preserve trusted elevated defaults only on the internal approval-follow-up path while keeping the Gateway agent idempotency key stable per approval, then require redacted runtime proof of the full second elevated-command approval flow before merge.

## Reproduction Assessment

Yes by source inspection: current main builds exec approval follow-up agent calls without `bashElevated`, while `createExecTool` throws the reported unavailable error when elevated defaults are absent. I did not run a live Telegram flow in this read-only review.

## Solution Assessment

No for the current patch as submitted. The internal token propagation seam is a good direction, but the dedupe key must remain stable and the contributor still needs real full-flow proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the follow-up idempotency key stable:** `src/agents/bash-tools.exec-approval-followup-state.ts:45`
  - body: This appends a fresh UUID-backed token to the Gateway agent `idempotencyKey`. The agent handler dedupes completions by `agent:${idempotencyKey}`, and the existing follow-up key was stable per approval ID, so duplicate completion dispatches for the same approved command no longer hit dedupe and can start duplicate follow-up runs. Keep the dedupe key stable and carry or look up the elevated state without changing that public idempotency value.
  - confidence: 0.86
