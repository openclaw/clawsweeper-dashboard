## Summary

Keep open. Current source has installer-side hardening for noninteractive apt and gum ioctl fallback, but the public hosted installer still serves the stale Node 22/raw apt path that matches the report, and open PR #73855 uses closing syntax for this issue.

## What This Changes

The issue reports that the public Linux installer fails Node.js auto-install under WSL2, prints a false success message, then exits because Node.js is still unavailable.

## Best Possible Solution

Deploy or sync the hosted installer from current `scripts/install.sh`, then land or replace the hosted-drift guard so future source/hosted installer mismatches fail before users receive a stale artifact.

## Reproduction Assessment

Yes. Fetching the live hosted installer currently reproduces the stale markers from the report: NodeSource 22 setup, raw apt installs, false `Node.js v22 installed` success text, and the Node v22+ PATH error; a WSL2 smoke is still the final runtime confirmation after deployment.

## Solution Assessment

No, not yet. The source hardening is the right direction, but the user-facing path remains broken while `openclaw.ai/install.sh` is stale; the narrow maintainable solution is hosted installer sync plus a drift check like the linked PR.
