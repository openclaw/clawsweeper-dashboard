## Summary

Keep open. Current source and the latest release tag contain the installer hardening for noninteractive apt, Node 24 setup, and gum ioctl fallback, but the user-facing hosted installer still serves the stale Node 22/raw apt script that matches the report. Open PR #73855 also uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent hosted-sync/deploy fix lands.

## What This Changes

The report asks the Linux/WSL installer to either install Node.js successfully or detect and stop accurately when NodeSource/apt installation fails, instead of reporting success and later crashing on the Node version check.

## Best Possible Solution

Deploy or sync the hosted installer from current scripts/install.sh, keep the source-side hardening, and land or replace the hosted-drift guard so future source/hosted mismatches fail before users hit a stale installer.

## Reproduction Assessment

Yes. Fetching the live hosted installer currently reproduces the relevant preconditions at the artifact level: stale Node 22 setup, raw apt installs, narrow ioctl detection, the false Node.js v22 success message, and the final Node v22+ PATH error from the report.

## Solution Assessment

No. The current source hardening is the right direction, but the observable user problem remains while openclaw.ai serves the stale artifact; #73855 is complementary drift detection, not a hosted deploy by itself.
