## Summary

Keep open. Current source and the latest release tag contain the installer hardening for noninteractive apt, Node 24 NodeSource setup, and gum ioctl fallback, but the public hosted installer still serves the stale Node 22/raw apt script that matches the report. Because open PR #73855 uses closing syntax for this issue and the user-facing hosted artifact is still out of sync, this should not be closed before that PR or an equivalent hosted-sync/deploy fix lands.

## What This Changes

The issue reports that the WSL2 Linux installer automatic Node.js install path prints ioctl errors, reports Node.js v22 installed, then exits because Node is still missing from PATH.

## Best Possible Solution

Deploy or sync the hosted installer from current scripts/install.sh, keep the source-side hardening, and land or replace the hosted-drift guard so future installer-source mismatches fail before users hit the stale artifact.

## Reproduction Assessment

Yes. Fetching the live hosted installer reproduces the relevant artifact-level preconditions: stale Node 22 setup, raw apt Node.js install calls, narrow ioctl detection, false Node.js v22 success text, and the final Node v22+ PATH error text from the report.

## Solution Assessment

No. The source hardening is the right direction, but it is incomplete while the documented hosted installer remains stale; the maintainable solution is hosted sync/deploy plus a drift check like the linked PR.
