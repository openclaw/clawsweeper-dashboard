## Summary

Keep open: current source and the latest tag contain installer hardening, but the live hosted installer still serves the stale Node 22/raw apt script that matches the report, and open PR #73855 still uses closing syntax for this issue.

## What This Changes

The issue asks the Linux/WSL installer to detect Node.js auto-install failures accurately instead of printing success and then crashing when Node is missing from PATH.

## Best Possible Solution

Deploy or sync the hosted installer from current scripts/install.sh and land a hosted-drift guard so future source/hosted mismatches fail visibly.

## Reproduction Assessment

Yes. A high-confidence artifact-level reproduction exists because fetching the live hosted installer still shows the stale setup_22.x/raw apt path, false Node.js v22 success message, and final Node.js v22+ PATH error described in the report.

## Solution Assessment

No. The source hardening is the right code fix, but the issue is not solved for users while the public hosted installer remains stale; PR #73855 is useful guard work but should be paired with an actual hosted sync/deploy.
