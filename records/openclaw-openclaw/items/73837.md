## Summary

Keep open. Current main and the latest release have source-side installer hardening, but the live hosted installer still differs from `scripts/install.sh` and contains the stale Node 22/raw apt markers that match the report. PR #73855 is also still open and uses closing syntax for this issue, so the issue should remain open until the hosted installer is redeployed and the drift guard is landed or superseded.

## What This Changes

The issue reports that the Linux installer can claim Node.js was installed after ioctl-related apt failures, then immediately fail because Node is absent.

## Best Possible Solution

Redeploy the hosted `install.sh` from the hardened `scripts/install.sh`, verify the WSL/Linux no-preinstalled-Node path cannot print Node success unless `node` is usable, then land or replace PR #73855 with a drift guard so future hosted/source divergence fails visibly.

## Reproduction Assessment

Yes. Fetching the live hosted installer still exposes the stale Node 22/raw apt markers that match the reported failure path; a final WSL smoke should confirm the runtime behavior after redeploy.

## Solution Assessment

No, the current state is not complete. Source-side hardening is useful, but the maintainable fix is hosted installer sync plus a guard such as PR #73855 so users stop receiving the stale script.
