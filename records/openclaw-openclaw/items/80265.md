## Summary

Keep open: the PR targets a valid auth-epoch reset bug, but the current patch is broader than the existing backend auth-policy contract, fails test typechecking, and still lacks real OpenClaw session proof.

## What This Changes

The PR makes `resolveCliAuthEpoch` ignore the local CLI credential fingerprint whenever an auth-profile credential exists, bumps the epoch version, adds a regression test, and updates the changelog.

## Best Possible Solution

Stabilize the Claude CLI storage-presence flip through the existing `authEpochMode`/`skipLocalCredential` seam or a narrow Claude identity fallback while preserving combined-mode host credential invalidation, fixing the type error, and adding redacted real-session proof.

## Reproduction Assessment

Yes. Source inspection shows current main hashes a local credential part plus a profile credential part, so mocking the local reader from `null` to a credential while the profile stays unchanged changes the epoch and then maps to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

No. The proposed resolver-level shortcut is broader than the backend policy contract; the safer fix should preserve `combined` semantics and only skip or stabilize the local part for the intended Claude/profile-owned case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve combined auth-epoch mode:** `src/agents/cli-auth-epoch.ts:191`
  - body: This condition makes any existing auth profile suppress the local CLI credential part, even when the backend did not opt into `profile-only`. The documented `combined` mode and prepare-time `skipLocalCredential` seam exist so ambient host credential changes can still invalidate sessions; with this patch, a profile-backed CLI using host auth can reuse a session after the local account or token changes.
  - confidence: 0.87
- **[P2] Use the Claude credential provider literal:** `src/agents/cli-auth-epoch.test.ts:493`
  - body: The new test stub returns `localCredential` from `readClaudeCliCredentialsCached`, but its type allows `provider: string`; CI's `check:test-types` rejects that because Claude OAuth credentials require the literal `provider: "anthropic"`. Narrow the stub type or reuse the exported credential type so the test compiles.
  - confidence: 0.92
