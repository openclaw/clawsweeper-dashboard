## Summary

Keep this PR open: it targets a valid CLI auth-epoch bug, but the current diff bypasses the existing profile-owned auth seam and does not cover the identity-less Claude CLI case reported in https://github.com/openclaw/openclaw/issues/80178.

## What This Changes

The PR bumps the CLI auth epoch version, changes `resolveCliAuthEpoch` to skip local CLI credential fingerprints when an OAuth auth-profile credential has email/accountId, updates auth-epoch tests, and adds a changelog entry.

## Best Possible Solution

Keep host credential fingerprinting in combined mode unless a backend explicitly owns execution auth, then add a narrowly reviewed fix and regression coverage for the identity-less Claude storage-presence flip without reusing sessions across real account changes.

## Reproduction Assessment

Yes. Current main is source-reproducible: mock the local Claude credential reader from `null` to an OAuth credential while the auth-profile credential stays unchanged, and the resolver hashes different part lists that downstream session reuse treats as `auth-epoch` invalidation.

## Solution Assessment

No. The PR is not the best fix as written because `profileOwnsIdentity` bypasses the documented combined/profile-only epoch policy; a safer solution should use the backend ownership seam or a specifically reviewed Claude identity-less rule.

## Review Findings

Full review comments:

- **[P1] Keep host auth in the epoch until the backend owns execution auth:** `src/agents/cli-auth-epoch.ts:193`

Bundled Claude/Codex/Gemini backends still run against ambient CLI auth, so changing the real local CLI account while a stale profile remains can keep the old epoch and reuse a session across an auth change.
  - confidence: 0.87

In the reported case `profileOwnsIdentity` is false, so file-missing versus file-present still hashes different part lists while the changelog says the issue is fixed.
  - confidence: 0.82
