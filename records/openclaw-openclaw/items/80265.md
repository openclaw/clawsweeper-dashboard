## Summary

Keep this PR open: the underlying bug is valid and source-reproducible, but the proposed resolver-level shortcut bypasses the documented combined host+profile auth-epoch contract, lacks real OpenClaw session proof, and overlaps with https://github.com/openclaw/openclaw/pull/74493.

## What This Changes

The branch bumps the CLI auth epoch version, skips local CLI credential fingerprints whenever an auth profile credential exists, updates auth-epoch tests, and adds changelog entries.

## Best Possible Solution

Stabilize the Claude CLI storage-presence flip through the existing `authEpochMode`/`skipLocalCredential` seam or a narrow Claude identity fallback, while preserving real account/profile invalidation and adding redacted real-session proof.

## Reproduction Assessment

Yes. Source inspection on current main shows that mocking the local Claude credential reader from `null` to an OAuth credential while the profile credential stays unchanged changes the resolver part list, and the session path maps that epoch mismatch to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

No. The PR’s resolver-level shortcut is broader than the backend policy contract; the safer fix should preserve `combined` semantics and only skip or stabilize the local part for the intended Claude/profile-owned case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve combined auth-epoch mode:** `src/agents/cli-auth-epoch.ts:191`
  - body: This condition treats the mere presence of an auth profile as `profile-only`, bypassing the prepare-time `skipLocalCredential` decision. For backends that rely on documented `combined` mode, changing the local host credential would no longer flip the epoch, so a profile-backed session can be reused under the wrong ambient CLI account.
  - confidence: 0.89
