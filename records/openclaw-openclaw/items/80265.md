## Summary

Keep this PR open, but do not merge it as-is: the reported bug is valid, yet the patch makes every profile-backed epoch behave as profile-only and also needs real behavior proof beyond mocked tests.

## What This Changes

The PR bumps the CLI auth epoch version, skips local CLI credential fingerprints when an auth profile credential exists, updates unit coverage, and edits the changelog.

## Best Possible Solution

Stabilize the Claude CLI storage-presence flip through the existing authEpochMode/skipLocalCredential seam or a narrow identity fallback, while preserving invalidation for real account/profile changes and adding unit plus redacted real-session proof.

## Reproduction Assessment

Yes. Source inspection shows current main hashes a local CLI part plus a profile part, so mocking the local credential reader from null to a credential while the profile stays unchanged changes the epoch and then maps to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

No. The proposed resolver-level shortcut is broader than the existing backend policy contract; the safer fix should preserve `combined` semantics and only skip or stabilize the local part for the intended Claude/profile-owned case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve combined auth-epoch mode:** `src/agents/cli-auth-epoch.ts:191`
  - body: This condition makes any existing auth profile suppress the local CLI credential part, even when the backend did not opt into `profile-only`. The documented `combined` mode and the prepare-time `skipLocalCredential` seam exist so ambient host credential changes can still invalidate sessions; with this patch, a profile-backed CLI using host auth can reuse a session after the local account/token changes. Keep the local part unless the caller actually requested profile-only, or scope the storage-presence fix to the Claude identity fallback.
  - confidence: 0.87

## Security Review
