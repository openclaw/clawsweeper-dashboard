## Summary

Keep open: the reported auth-epoch reset bug is real, but this PR's resolver-level shortcut is too broad and still lacks real behavior proof. It suppresses local host credential epochs whenever any auth profile exists, which weakens the documented combined-mode session invalidation boundary.

## What This Changes

The PR bumps `CLI_AUTH_EPOCH_VERSION`, changes `resolveCliAuthEpoch` to prefer auth-profile credentials over local CLI fingerprints whenever a profile credential exists, adjusts auth-epoch tests, and adds a changelog entry.

## Best Possible Solution

Stabilize the Claude CLI local-storage presence flip through a narrow backend-owned or Claude/profile-owned auth-epoch path while preserving combined host-plus-profile invalidation semantics.

## Reproduction Assessment

Yes, by source inspection and the linked issue logs: current main hashes local plus profile parts, so local credential presence changing from absent to present while the profile stays stable changes the epoch and maps to `invalidatedReason: "auth-epoch"`. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The resolver-level `!profileCredential` shortcut is broader than the backend contract; the safer fix should preserve `combined` semantics and only drop local credentials through the existing `authEpochMode`/`skipLocalCredential` seam or an equally narrow Claude-owned policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve combined auth-epoch mode:** `src/agents/cli-auth-epoch.ts:191`
  - body: The new `!profileCredential` gate makes any selected auth profile suppress the local CLI fingerprint even when the backend is still in documented `combined` mode. That can reuse a profile-backed CLI session after the ambient host credential/account changes, bypassing the `skipLocalCredential` policy seam that should only drop local auth for profile-owned prepared execution.
