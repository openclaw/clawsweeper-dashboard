## Summary

Keep open: the reported bug is valid and source-reproducible, but this PR broadens the auth-epoch contract by suppressing local credential hashing for any profile-backed run and only provides mock/unit proof.

## What This Changes

The PR bumps `CLI_AUTH_EPOCH_VERSION`, changes `resolveCliAuthEpoch` to prefer auth profile credentials over local CLI fingerprints, updates auth-epoch unit expectations, and adds a changelog entry.

## Best Possible Solution

Stabilize the Claude CLI local-storage-presence flip through the existing backend-owned auth-epoch policy seam or a narrow Claude/profile-owned path, while preserving combined-mode host-auth invalidation.

## Reproduction Assessment

Yes. Source inspection on current main shows the resolver hashes local plus profile parts, so a local credential changing from absent to present while the profile stays stable changes the epoch and maps to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

No. The resolver-level `!profileCredential` shortcut is broader than the backend contract; the safer fix should preserve `combined` semantics and use `authEpochMode`/`skipLocalCredential` or an equivalent narrow Claude-owned policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

- **[P2] Preserve combined auth-epoch mode:** `src/agents/cli-auth-epoch.ts:191`
