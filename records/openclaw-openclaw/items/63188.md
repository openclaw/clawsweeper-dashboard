## Summary

Keep open. Current main still preserves provider length stop reasons but does not implement the requested embedded `mode:"run"` fail/continue policy, and the related maintainer discussion explicitly keeps this subagent truncation track separate from the main-session truncation tracker.

## What This Changes

The issue requests configurable embedded-run handling for provider `stopReason:"length"`, including hard failure signaling and optional bounded continuation.

## Best Possible Solution

Add a documented fail-mode first for embedded-run length stops, then evaluate opt-in continuation only after replay and side-effect rules are explicit.

## Reproduction Assessment

Yes, at source level: mock an embedded attempt whose last assistant has `stopReason:"length"` and partial visible text, and current main reaches the normal terminal path instead of a fail/continue branch. I did not run a live provider truncation replay because this review is read-only.

## Solution Assessment

Unclear for the full catch-and-continue design, because replay and side-effect policy need maintainer judgment. The narrowest maintainable first step is the requested fail-mode subset that surfaces length-truncated terminal runs as explicit errors without replaying work.
