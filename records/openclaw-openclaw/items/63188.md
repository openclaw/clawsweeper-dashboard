## Label Justifications

- `P2`: This is a real runtime resilience improvement with user-visible partial-output risk, but it needs policy design rather than emergency regression handling.
- `impact:data-loss`: The report centers on incomplete or empty generated files being left on disk without a hard failure signal.
- `impact:session-state`: The affected path is embedded subagent run/session completion state and whether a truncated turn is marked completed, failed, or recoverable.

## Summary

Keep open. Current main has adjacent empty/reasoning-only retry mitigations, but it still lacks the requested configurable embedded `mode:"run"` policy for provider `stopReason:"length"`; partial visible length-terminated runs still fall through to the normal terminal path.

## What This Changes

Request configurable embedded-run `onLengthTruncation` behavior so `mode:"run"` subagents can fail or continue instead of silently completing after provider output truncation.

## Best Possible Solution

Add a documented embedded-run length-truncation policy that first supports explicit failure signaling, then only adds opt-in continuation once replay and side-effect rules are defined.

## Reproduction Assessment

Yes, at source level: a mocked embedded attempt with `lastAssistant.stopReason="length"` and partial visible text reaches the normal terminal path because the incomplete-turn guard returns `null`. I did not run a live provider truncation replay because this review is read-only.

## Solution Assessment

Unclear for the full catch-and-continue proposal; the narrow fail-mode subset is the safest first step, while continuation needs explicit replay and side-effect policy before implementation.
