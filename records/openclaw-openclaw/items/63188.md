## Summary

Keep open. Current main still preserves provider `stopReason:"length"` as metadata, but run-mode subagent terminal lifecycle and announce paths can still convert that terminal event into a successful child outcome, and there is no configurable fail/continue truncation policy in the agent config schema or docs. The issue has fresh production and local-model reproductions plus complementary related reports, so it is actionable rather than stale or implemented.

## What This Changes

Request configurable run-mode subagent handling for provider `stopReason:"length"`, including fail/continue options so truncated child runs do not silently appear successful.

## Best Possible Solution

Add a narrow opt-in native-subagent truncation policy first: when a run-mode child reaches terminal `stopReason:"length"`, configured fail mode should surface an error or partial-result signal to the parent instead of a successful child outcome. Treat automatic continuation as a follow-up design because it needs replay and side-effect safety.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to emit a lifecycle `end` with `stopReason:"length"` for a child run and observe `agent.wait` plus `applySubagentWaitOutcome` still produce `status:"ok"`; the issue discussion also provides production and local-model reproductions.

## Solution Assessment

Yes for the first-stage fail policy, not yet for automatic continuation. Hard-failing or marking partial output on configured `length` truncation is the narrowest maintainable fix because it removes silent success without replaying potentially side-effecting work.
