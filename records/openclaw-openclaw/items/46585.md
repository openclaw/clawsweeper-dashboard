## Summary

Keep open: current main still forwards `facing` and `deviceId` together, and the macOS picker resolves the explicit AVFoundation device id before applying front/back selection. The report is recent, concrete, and the related allowlist issues do not cover this Continuity Camera facing behavior.

## What This Changes

Request that macOS iPhone Continuity Camera honor `--facing front` when snapping from an explicit Continuity `--device-id`, instead of always capturing the rear camera.

## Best Possible Solution

Resolve a Continuity device plus requested facing to the actual front/back AVFoundation capture device when the platform exposes one; otherwise return a clear warning or failure and document the limitation.

## Reproduction Assessment

Yes: a maintainer with macOS plus an iPhone Continuity Camera can run `openclaw nodes camera list --node <mac-node>`, then compare `snap --device-id <continuity-id> --facing front` and `--facing back`. I did not execute the hardware path here, but current-main source inspection shows the exact selection path reported.

## Solution Assessment

Unclear until live AVFoundation verification. The safest fix is either API-backed Continuity front/back resolution or explicit unsupported behavior with docs, not an unverified source-only guess.
