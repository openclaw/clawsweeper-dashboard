## Summary

Keep open: current `main` and the latest release still accept the reported `--facing front --device-id <ContinuityCameraId>` path, while the macOS camera picker returns the matching device ID before `facing` can affect selection. The report is concrete, no canonical fix or duplicate now owns the remaining Continuity Camera facing behavior, and this needs real macOS+iPhone validation plus a maintainer product/API decision.

## What This Changes

The issue requests that iPhone Continuity Camera honor `--facing front` versus `--facing back` when `openclaw nodes camera snap` is invoked with a specific camera `--device-id`.

## Best Possible Solution

Define the macOS Continuity Camera contract, then either implement tested front/rear selection for iPhone Continuity devices or return a clear diagnostic and docs note when `--facing` cannot switch a single unspecified external camera device.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction path in this review because it requires a real macOS+iPhone Continuity Camera setup. Source inspection does show the current-main mechanism: with `deviceId` set, the macOS picker returns that device before applying `facing`.

## Solution Assessment

Unclear. A direct code fix is best only if macOS exposes a selectable front Continuity camera through AVFoundation; otherwise the maintainable solution is a clear unsupported-combination diagnostic and public contract.
