## Summary

Current main still has the same unresolved macOS Continuity Camera boundary: `deviceId` selection wins before `facing`, and the right fix needs maintainer direction plus real macOS+iPhone validation.

## What This Changes

The issue asks for iPhone Continuity Camera captures with `--facing front` to use the front camera instead of always using the rear camera.

## Best Possible Solution

Define the macOS Continuity Camera contract, then either implement tested front-camera selection for iPhone Continuity devices or make the CLI/docs report that `--facing` cannot switch a single external Continuity camera device.

## Reproduction Assessment

No high-confidence live reproduction is available from this review because it requires real macOS+iPhone Continuity Camera hardware. Source inspection does reproduce the likely mechanism: with a matching `deviceId`, the macOS picker returns that device before using `facing`.

## Solution Assessment

Unclear. The requested behavior may need a different AVFoundation Continuity API path, while the safer near-term solution may be an explicit unsupported-device diagnostic and docs update if Apple only exposes the rear camera here.
