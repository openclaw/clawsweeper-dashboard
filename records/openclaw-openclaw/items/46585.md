## Summary

Keep open: current main and the latest release still select a matching macOS AVFoundation device id before applying the requested facing, so a matched Continuity Camera can ignore `--facing front`/`--facing back`. The related allowlist reports only covered whether `camera.snap` was advertised, not this Continuity Camera selection behavior.

## What This Changes

The issue requests that macOS node camera capture honor `--facing front` for an iPhone Continuity Camera when a Continuity `--device-id` is provided.

## Best Possible Solution

Resolve a Continuity device plus requested facing to the actual front/back AVFoundation capture device when the platform exposes one; otherwise return a clear warning or failure and document the limitation.

## Reproduction Assessment

Yes, with hardware: a maintainer can list the macOS node cameras, then run `snap --device-id <continuity-id> --facing front` and `--facing back` and compare photos. I did not execute that hardware path, but source inspection shows both commands select the explicit device id before facing is considered.

## Solution Assessment

Unclear until live AVFoundation verification. The narrow maintainable solution is either API-backed Continuity front/back resolution or explicit unsupported behavior with docs, not an unverified picker guess.
