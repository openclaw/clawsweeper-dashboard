## Summary

Keep open. Current main still has the reported behavior shape: the CLI forwards `facing` and `deviceId` together, while the macOS camera picker returns an explicit device-id match before applying front/back selection. The latest stable release has the same relevant camera files, and there is no current doc or test coverage that documents or validates a Continuity Camera front/back limitation.

## What This Changes

The issue requests that macOS iPhone Continuity Camera honor `--facing front` and `--facing back` when snapping with an explicit Continuity `--device-id`.

## Best Possible Solution

The macOS camera path should either resolve a Continuity device plus requested facing to the correct AVFoundation front/back capture device, if the platform exposes one, or fail/warn explicitly and document the limitation instead of silently returning rear-camera output for a front-camera request.

## Reproduction Assessment

Yes, for a maintainer with the required hardware: pair an iPhone as a Continuity Camera, run `openclaw nodes camera list --node <mac-node>`, then compare `snap --device-id <continuity-id> --facing front` with `--facing back`. Source inspection of current main matches the reported selection path, but I could not execute the hardware capture locally.

## Solution Assessment

Unclear until live AVFoundation verification. The current implementation is not the best end state because explicit device-id selection can bypass facing; the maintainable solution is API-backed Continuity resolution when possible, otherwise explicit unsupported behavior plus docs.
