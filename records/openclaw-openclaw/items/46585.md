Keep open: current main and the latest release still let an explicit macOS camera `deviceId` match return before `facing` can affect selection, so the reported Continuity Camera behavior remains plausible and needs real hardware validation plus a supported-contract decision.

## What This Changes

The issue asks for `openclaw nodes camera snap --facing front` to select the iPhone Continuity Camera front camera instead of always capturing from the rear camera when a Continuity device ID is supplied.

## Best Possible Solution

Define the macOS Continuity Camera facing contract, then either select a compatible front/back Continuity device when AVFoundation exposes one or return a clear unsupported-facing diagnostic with docs for single external Continuity devices.

## Reproduction Assessment

No high-confidence live reproduction is available from this review because it requires a real macOS plus iPhone Continuity Camera setup. Source inspection does show a current-main path where a matching `deviceId` returns before `facing` can affect device selection.

## Solution Assessment
