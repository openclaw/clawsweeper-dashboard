## Summary

Keep open: current main still has the reported behavior shape, because the CLI forwards both `facing` and `deviceId` and the macOS camera picker returns the explicit device-id match before applying front/back selection. The issue is recent, concrete, and not covered by an implemented fix or a documented Continuity Camera limitation.

## What This Changes

Make iPhone Continuity Camera `snap --device-id <id> --facing front` select the requested front camera or fail with explicit unsupported behavior instead of silently capturing the rear camera.

## Best Possible Solution

The macOS camera path should resolve a Continuity device plus requested facing to the actual AVFoundation front/back capture device when available; otherwise it should fail or warn explicitly and document the limitation.

## Reproduction Assessment

Yes, for a maintainer with the required macOS+iPhone Continuity Camera hardware: run `openclaw nodes camera list --node <mac-node>`, then compare `snap --device-id <continuity-id> --facing front` and `--facing back`. I did not execute that hardware path here, but current-main source inspection matches the reported selection path.

## Solution Assessment

Unclear until live AVFoundation verification. A source-only fix should not guess whether Continuity front/back mapping is possible; the maintainable outcome is either API-backed selection or explicit unsupported behavior with docs.
