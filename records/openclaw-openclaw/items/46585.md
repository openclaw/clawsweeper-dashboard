## Summary

Keep open: current main and the latest v2026.6.6 release still accept `facing` with an explicit Continuity Camera `deviceId`, but macOS resolves the device id before consulting the requested facing, so the reported path remains plausibly broken and needs live hardware proof plus a maintainer behavior decision.

## What This Changes

The issue asks for iPhone Continuity Camera captures with `--device-id` and `--facing front` to use the front camera instead of always returning the rear camera.

## Best Possible Solution

Live-test the signed macOS companion with an iPhone Continuity Camera, then either implement a proven front/back selection path or make the explicit-device limitation visible in CLI/tool output and docs.

## Reproduction Assessment

No live high-confidence reproduction is available from this review because it needs signed macOS hardware plus an iPhone Continuity Camera. Source inspection does reproduce the decision path: the CLI/tool forwards `facing` with `deviceId`, and macOS returns the matching device before using facing.

## Solution Assessment

Unclear. A blind source-only fix is not the best path until live AVFoundation proof shows whether OpenClaw can select the requested iPhone lens or should surface a platform limitation.
