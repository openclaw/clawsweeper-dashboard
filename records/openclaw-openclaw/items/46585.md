## Summary

Current main still prioritizes an explicit camera `deviceId` over the requested `facing`, so the reported single Continuity Camera device would be captured the same way for `front` and `back`. I would keep this open because the right outcome needs maintainer judgment and likely real macOS+iPhone validation: either add supported Continuity-device camera selection, or document/diagnose that iPhone Continuity Camera cannot provide a front-facing capture through the current AVFoundation path.

## What This Changes

The issue requests that `openclaw nodes camera snap --device-id <iPhone Continuity Camera> --facing front` capture from the iPhone front camera instead of always using the rear camera.

## Best Possible Solution

Decide the macOS Continuity Camera contract explicitly, then either implement tested front-camera selection for iPhone Continuity devices or make the CLI/docs report that `--facing` cannot switch a single external/Continuity camera device.

## Reproduction Assessment

No high-confidence live reproduction against current main is available. Source inspection does reproduce the likely mechanism: with a supplied `deviceId`, macOS selects that exact AVFoundation device before applying `facing`.

## Solution Assessment

Unclear. The requested front-camera behavior may need a new AVFoundation/Continuity-device implementation, while a safer near-term solution may be clearer docs or an explicit unsupported-device diagnostic.
