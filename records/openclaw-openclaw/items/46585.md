## Summary

Keep open: current main and the latest release still accept the reported `--facing front --device-id <ContinuityCameraId>` path, while the macOS picker returns the matching device ID before `facing` can influence selection. The issue is detailed and still needs maintainer direction plus real macOS+iPhone Continuity Camera validation.

## What This Changes

Request that iPhone Continuity Camera snaps honor `--facing front` and `--facing back` when a Continuity camera device ID is supplied.

## Best Possible Solution

Define the macOS Continuity Camera contract, then either implement tested front-camera selection for iPhone Continuity devices or report and document that `--facing` cannot switch a single external Continuity camera device.

## Reproduction Assessment

No high-confidence live reproduction is available from this review because it requires real macOS+iPhone Continuity Camera hardware. Source inspection does show the likely current-main mechanism: a matching `deviceId` is returned before `facing` is considered.

## Solution Assessment

Unclear. A direct code fix is best only if macOS exposes a selectable front Continuity camera; otherwise the maintainable solution is a diagnostic and docs contract for unsupported external Continuity facing selection.
