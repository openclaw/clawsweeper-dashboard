The issue asks OpenClaw to make `--facing front` select the iPhone Continuity front camera when `camera.snap` is run with a Continuity Camera device id.

## Best Possible Solution

Define the macOS Continuity Camera `deviceId` plus `facing` contract, then either select distinct front/back Continuity devices when exposed or return a clear unsupported-facing diagnostic with docs for single external Continuity devices.

No. A high-confidence live reproduction needs a real macOS plus iPhone Continuity Camera setup, which was not available in this read-only review. Source inspection does show a plausible current-main path where an exact `deviceId` match returns before `facing` can affect camera selection.
