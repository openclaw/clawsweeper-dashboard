## Summary

Keeping this open: current main and the latest release still return an exact macOS camera `deviceId` match before applying `facing`, while the CLI and docs expose `front|back` as part of the camera snap contract. I did not find a duplicate or merged fix; the remaining question is whether AVFoundation can expose the iPhone front camera through Continuity Camera or whether OpenClaw should fail clearly instead of silently taking the rear camera.

## What This Changes

The issue asks OpenClaw's macOS node camera snap to honor `--facing front` for iPhone Continuity Camera when a camera `--device-id` is supplied.

## Best Possible Solution

Verify the Continuity Camera AVFoundation behavior on real macOS/iPhone hardware, then make macOS camera selection either honor `front|back` for a selected device when the platform exposes it or return a clear unsupported diagnostic and document the limitation.

## Reproduction Assessment

No high-confidence live reproduction was established in this review; the source path is clear enough to explain the report because `--facing front --device-id ...` reaches macOS and the selector returns the exact device before consulting `facing`. A real Continuity Camera setup is still needed to confirm whether the iPhone front camera is exposed to macOS at all.

## Solution Assessment

Unclear. The current behavior should not silently ignore `facing`, but the best implementation depends on whether AVFoundation exposes separate front/back Continuity Camera devices or only one external device.
