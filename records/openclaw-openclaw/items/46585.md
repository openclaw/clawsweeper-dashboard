## Summary

Keep open: current main and latest release still accept `--facing` with an explicit Continuity Camera `deviceId`, but the macOS camera picker returns the matched device before consulting the requested facing, and no canonical issue or PR fixes front-camera selection for this path.

## What This Changes

The issue requests that `openclaw nodes camera snap --facing front --device-id <Continuity Camera>` capture the iPhone front camera instead of always returning the rear camera.

## Best Possible Solution

Live-test the macOS companion with a real iPhone Continuity Camera, then either implement a proven AVFoundation lens-selection path or make OpenClaw warn/document that explicit external Continuity Camera device ids cannot honor `--facing`.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection does reproduce the decision path: the CLI and agent tool pass `facing` plus `deviceId`, while macOS returns the matched device id before consulting facing.

## Solution Assessment

Unclear. The current implementation silently accepts a facing it may not be able to honor for explicit external device ids; the best fix depends on live AVFoundation proof and a maintainer choice about warning, failing, or supporting lens selection.
