## Label Justifications

- `P2`: The issue is a real user-visible delay in external-channel Codex sessions, but it is scoped to transcript projection rather than runtime availability.
- `impact:message-loss`: The accepted inbound channel message is missing from the durable transcript and UI until the Codex turn completes.
- `impact:session-state`: The reported failure is session transcript state lagging behind accepted external-channel input.

## Summary

Keep open: current main still writes the Codex app-server prompt to the durable session transcript through the end-of-turn mirror, while inbound channel/session recording only updates routing/session metadata, so the reported WebChat/Control UI delay remains source-reproducible and is corroborated by the Telegram follow-up.

## What This Changes

The issue asks for accepted external-channel Codex user messages to be appended to the session transcript immediately instead of appearing only when the Codex turn finalizes.

## Best Possible Solution

Add a narrow Codex app-server early prompt mirror after a stable turn id exists, using the existing `${turnId}:prompt` identity and idempotency contract so the final mirror remains the canonical dedupe pass.

## Reproduction Assessment

Yes from source inspection, with reporter-supplied Feishu and Telegram timing evidence. A Codex app-server turn currently records channel/session metadata first, but the durable `role=user` transcript row is produced only by the final mirror after completion.

## Solution Assessment

Yes. The narrowest maintainable fix is to mirror only the accepted prompt once the Codex turn has a stable id, reusing the existing mirror identity and dedupe contract instead of moving channel-specific transcript writes into the generic channel record path.
