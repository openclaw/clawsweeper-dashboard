## Label Justifications

- `P2`: The issue affects external-channel visibility in WebChat/Control UI for Codex sessions but has a bounded runtime/transcript surface and no evidence of data loss or crash-loop severity.
- `impact:message-loss`: Inbound messages are accepted by the channel but suppressed from the durable UI transcript until the turn ends, creating an operator-visible message delivery gap.
- `impact:session-state`: The durable session transcript state lags behind the accepted external message, causing WebChat/Control UI history to drift from live channel state.

## Summary

Keep open: current main still records Codex app-server inbound prompts into the durable transcript through the end-of-turn mirror, while the inbound channel record path only updates session metadata, so the WebChat/Control UI delay remains source-reproducible and is supported by the linked Telegram timing evidence.

## What This Changes

The issue asks for Codex runtime inbound user prompts from external channels to be written to the session transcript as soon as the turn is accepted, rather than only after turn completion.

## Best Possible Solution

Add a narrow Codex app-server early prompt mirror after the accepted turn id is known, reusing the existing `${turnId}:prompt` identity and idempotency scope so the final mirror remains the dedupe authority.

## Reproduction Assessment

Yes. Source inspection shows the Codex `role=user` transcript row is built and mirrored through the final result path, and the issue discussion includes Feishu plus Telegram timing evidence showing the UI stays stale until that final mirror runs.

## Solution Assessment

Yes. The narrowest maintainable fix is to mirror only the accepted Codex prompt once the stable turn id exists, instead of moving channel-specific transcript writes into the generic inbound session metadata path.
