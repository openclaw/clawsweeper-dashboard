## Label Justifications

- `P2`: The issue causes a real user-visible delay in external-channel Codex sessions, but the blast radius is scoped to transcript projection and UI visibility rather than core runtime availability.
- `impact:message-loss`: The accepted inbound channel message is missing from the durable transcript/UI until the Codex turn completes.
- `impact:session-state`: The affected state is the session transcript projection lagging behind an accepted external-channel turn.

## Summary

Keep open: current main still waits for Codex turn completion before mirroring the prompt into the session transcript, so channel-accepted inbound user messages remain absent from transcript-backed WebChat/Control UI until finalization; the linked Telegram evidence in https://github.com/openclaw/openclaw/issues/84020 confirms this is not Feishu-specific.

## What This Changes

Persist the Codex app-server inbound user prompt to the session transcript as soon as a channel turn is accepted so WebChat/Control UI can show it before the assistant reply.

## Best Possible Solution

Add a narrow Codex app-server prompt mirror immediately after `turn/start` returns a stable turn id, using the existing mirror identity and idempotency scope, while leaving external assistant/tool observer streaming to the separate canonical issue.

## Reproduction Assessment

Yes at source level, though not by a live current-main Feishu/Telegram run here. Start a Codex app-server turn and inspect the session transcript before completion: the source records metadata first and only mirrors the prompt after `completion` resolves.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix for the inbound-user half. Mirroring the prompt only after `turn/start` has a stable id reuses the existing Codex mirror/dedupe contract and avoids moving channel-specific transcript policy into the generic record path.
