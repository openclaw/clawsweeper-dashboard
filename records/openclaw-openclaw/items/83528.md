- `P2`: This is a user-visible external-channel Codex transcript delay with a bounded runtime/UI projection surface, not a core availability failure.
- `impact:message-loss`: The accepted inbound channel message is temporarily absent from the durable transcript and UI until Codex turn completion.

## Summary

Keep open: current main still mirrors the Codex app-server user prompt only after the Codex turn completes, while the channel inbound record path only updates metadata/route state, so WebChat/Control UI has no durable inbound transcript row to display immediately.

## What This Changes

The issue requests immediate durable transcript persistence for Codex external-channel inbound user messages once a turn is accepted.

## Best Possible Solution

Add a focused Codex app-server prompt mirror immediately after `turn/start` returns a stable turn id, using the existing `${turnId}:prompt` identity and thread-stable idempotency scope so the final mirror remains a deduping completion pass.

## Reproduction Assessment

Yes from source inspection, though not from a live channel run. On current main, a Codex app-server turn has a stable `turnId` before completion, but the user prompt is only built into `messagesSnapshot` and mirrored after `completion` resolves.

## Solution Assessment

Yes. The suggested prompt-only early mirror is the narrow maintainable fix because it stays inside the Codex app-server mirror boundary and reuses existing mirror identity/idempotency instead of moving channel record logic into core.
