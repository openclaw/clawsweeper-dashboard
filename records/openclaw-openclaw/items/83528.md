## Summary

Keep open: current main still mirrors the Codex app-server user prompt only after the Codex turn completes, and the related Telegram reproduction in https://github.com/openclaw/openclaw/issues/84020 confirms the same shared external-channel symptom beyond Feishu.

## What This Changes

The issue requests immediate durable transcript writes for accepted inbound external-channel user messages in Codex sessions so WebChat/Control UI can show them before the turn finishes.

## Best Possible Solution

Add a narrow Codex app-server prompt mirror immediately after `turn/start` returns a stable turn id, using the existing `${turnId}:prompt` identity and thread-stable idempotency scope so the final mirror remains the canonical dedupe pass.

## Reproduction Assessment

Yes, at source level: start a Codex app-server turn and inspect the session transcript before completion; current main records channel metadata first but only mirrors the user prompt after `completion` resolves. A related shipped-version Telegram timeline also matches this exact timing path, though I did not run a live current-main channel repro.

## Solution Assessment

Yes, the suggested runtime-side fix is the narrow maintainable path: mirror only the accepted prompt after a stable Codex turn id exists and rely on the existing mirror identity/idempotency contract to make the final snapshot a no-op for that prompt.
