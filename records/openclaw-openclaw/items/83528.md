## Summary

Keep open: current main still waits for the Codex app-server turn to complete before mirroring the prompt into the durable transcript, and the related Telegram timeline confirms the same symptom is not Feishu-specific.

## What This Changes

The issue asks for Codex-backed external-channel turns to persist the inbound user transcript entry immediately instead of waiting until turn finalization.

## Best Possible Solution

Add a prompt-only Codex app-server mirror immediately after `turn/start` returns a stable turn id, using the existing `${turnId}:prompt` identity and thread-stable idempotency scope, while keeping the final snapshot mirror as the canonical dedupe pass.

## Reproduction Assessment

Yes at source level, though not from a live current-main Feishu run. Start a Codex app-server external-channel turn and inspect the transcript before completion: current main records metadata/route state first, but the user prompt is only mirrored after `completion` resolves.

## Solution Assessment

Yes, the suggested direction is the narrow maintainable fix. Mirroring only the prompt after `turn/start` has a stable id uses the existing mirror identity/dedupe contract and avoids moving channel-owned record logic into a generic transcript writer.
