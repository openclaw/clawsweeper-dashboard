## Summary

Keep open: current main and v2026.5.12 still construct and mirror the Codex app-server user prompt only after the Codex turn completes, while the channel record path only updates session metadata/route state. That leaves WebChat/Control UI without an immediate durable inbound transcript row for external-channel Codex turns.

## What This Changes

The issue asks OpenClaw to append Codex app-server inbound user prompts to the durable session transcript as soon as an external-channel turn is accepted, instead of waiting for turn completion.

## Best Possible Solution

Add a narrow Codex app-server prompt mirror immediately after `turn/start` returns a stable turn id, using the existing `${turnId}:prompt` identity and thread-stable idempotency scope while keeping the final snapshot mirror as the dedupe-backed canonical pass.

## Reproduction Assessment

Yes at source level, though not from a live Feishu run. Start a Codex app-server turn with delayed completion and inspect the session transcript before completion: current code records route metadata first but only builds and mirrors the prompt after `completion` resolves.

## Solution Assessment

Yes, the suggested direction is the narrow maintainable fix. Reusing the existing `${turnId}:prompt` identity and mirror dedupe contract avoids moving channel-owned record logic into core and keeps the final mirror flow intact.
