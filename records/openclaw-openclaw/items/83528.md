- `impact:message-loss`: The accepted inbound channel message is absent from the durable transcript/UI until the Codex turn completes.

## Summary

Keep open: current main still mirrors the Codex app-server user prompt only after the Codex turn completes, while the channel-turn record path updates session metadata/route state without writing a transcript row, so WebChat/Control UI has no durable inbound message to display immediately.

## What This Changes

The issue asks for Codex app-server external-channel inbound prompts to be appended to the session transcript as soon as a stable turn id exists, while keeping final transcript mirroring idempotent.

## Best Possible Solution

Add a narrow Codex app-server prompt mirror immediately after `turn/start` returns a stable turn id, using the existing `${turnId}:prompt` identity and thread-stable idempotency scope so the final mirror remains the canonical no-op/dedupe pass.

## Reproduction Assessment

Yes from source inspection, though not from a live Feishu run. Start a Codex app-server turn and inspect the session transcript before completion: the current code records metadata/lastRoute first, but the user prompt is only mirrored after `completion` resolves.

## Solution Assessment

Yes, the suggested direction is the narrow maintainable fix. Mirroring only the prompt after `turn/start` has a stable id uses the existing mirror identity/dedupe contract and avoids moving channel-owned record logic into core.
