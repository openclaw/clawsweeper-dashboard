## Label Justifications

- `P2`: This is a real user-visible external-channel Codex transcript delay, but it is scoped to session projection rather than runtime availability.
- `impact:message-loss`: The accepted inbound channel message is absent from the durable transcript and WebChat/Control UI until the Codex turn completes.
- `impact:session-state`: The bug is about session transcript state lagging behind the accepted external-channel turn.

## Summary

Keep open: current main still mirrors the Codex app-server prompt only after turn completion, while inbound session recording only updates session metadata and route state. The linked Telegram reproduction at https://github.com/openclaw/openclaw/issues/84020 supports this as shared external-channel Codex behavior, not a Feishu-only report.

## What This Changes

Append the inbound Codex external-channel user prompt to the durable session transcript as soon as the turn has a stable id, so WebChat/Control UI can show it before the assistant reply completes.

## Best Possible Solution

Add an early Codex app-server prompt mirror immediately after successful turn/start using the existing `${turnId}:prompt` identity and thread-stable idempotency scope, keeping the final mirror as the dedupe/canonicalization pass.

## Reproduction Assessment

Yes at source level, though not through a live current-main Feishu run in this read-only review. Start a Codex app-server turn and inspect the transcript before completion: run-attempt waits for completion before mirrorTranscriptBestEffort, and buildResult creates the user prompt snapshot only then.

## Solution Assessment

Yes, the narrow runtime-side fix is the maintainable direction. Mirroring only the prompt once turn/start returns a stable id uses the existing mirror identity/dedupe contract and avoids changing external-channel observer streaming policy.
