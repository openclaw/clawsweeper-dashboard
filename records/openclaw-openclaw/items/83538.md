Keep this open: current main and the latest release still route manual cron completion through the same `deleteAfterRun` success gate, and the linked closing PR is open but unmerged. The issue should remain attached to that PR rather than be closed or duplicated.

## What This Changes

The issue reports that manual `cron run` can consume and delete a future one-shot `deleteAfterRun` cron job, losing the scheduled fire and associated state.

## Best Possible Solution

Review and either land the linked fix PR after accepting its cron state compatibility risk, or close that PR with a maintainer policy decision and keep this issue as the canonical manual-run deletion bug.

## Reproduction Assessment

Yes, source-level: manual completion calls `applyJobResult` without a manual discriminator, and an ok `at` job with `deleteAfterRun: true` is removed from the store. I did not run a live gateway reproduction in this read-only review.

Unclear until maintainers accept the policy, but the linked PR’s owner-boundary shape is plausible because it adds the discriminator where both manual and scheduled outcomes converge. A separate new fix PR would be worse while that closing PR is open.
