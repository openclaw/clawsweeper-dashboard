## Summary

Keep this open: current main still has the destructive source path, and an open linked PR with closing syntax is already the implementation path, so the issue should close only after that PR lands or is replaced.

## What This Changes

Manual `cron run` should not silently consume or remove a future `deleteAfterRun` one-shot cron job when the scheduled run has not completed.

## Best Possible Solution

Land a narrow cron-service fix that carries manual-run context into `applyJobResult`, preserves scheduled one-shot jobs and retry state for manual outcomes, keeps scheduled `deleteAfterRun` behavior unchanged, and closes this issue through the linked PR after proof.

## Reproduction Assessment

Yes, source-level: current main manual run completion calls `applyJobResult`, and successful `at` jobs with `deleteAfterRun` return `shouldDelete=true` without knowing the run was manual. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Mostly yes: the narrow maintainable solution is to mark manual outcomes at the shared result handler and leave scheduled one-shot behavior intact. Maintainers still need to confirm the semantic choice in the linked PR before the issue closes.
