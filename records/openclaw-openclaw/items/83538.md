## Summary

Keep open. Current main still lets a manual force run reach the same successful one-shot deletion path as scheduled execution, while the no-execution/empty-history part needs a focused repro before a safe fix shape is chosen.

## What This Changes

The issue requests that manual `cron run` not silently consume and delete future one-shot `deleteAfterRun` cron jobs without durable evidence that work executed.

## Best Possible Solution

Define manual one-shot semantics explicitly, then make the scheduler preserve future `deleteAfterRun` jobs by default or require an explicit destructive override, with a durable run-history/audit record before any job removal.

## Reproduction Assessment

Partially. Source inspection gives a clear path for a successful manual force run to delete a one-shot `deleteAfterRun` job, but the stronger claim that the job can disappear with no executed run and empty history still needs a focused current-main repro.

## Solution Assessment

Unclear. The safest direction appears to be preserving scheduled one-shot jobs on ordinary manual runs or requiring an explicit destructive override, but maintainers should choose the policy before implementation.
