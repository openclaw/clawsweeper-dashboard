## Summary

Keep open. Current main still lets a queued/manual force run reach the normal successful-run `deleteAfterRun` removal path for one-shot `at` jobs, and the related closed/open cron items do not settle whether manual verification should consume a future scheduled fire or how to guarantee durable run evidence before deletion.

## What This Changes

The issue reports that manual `cron run` can consume a `deleteAfterRun` one-shot cron and remove the future scheduled fire without a visible terminal run record.

## Best Possible Solution

Make `cron run` safe for `deleteAfterRun` jobs by either not consuming the scheduled one-shot on manual force runs or requiring an explicit destructive override, and ensure a terminal run/audit record is durable before any auto-delete is persisted.

## Reproduction Assessment

Yes for the central destructive path by source inspection: a manual force run can finish `ok`, hit the `deleteAfterRun` predicate, and remove the stored one-shot job. No for the exact no-execution/no-history timing, which was not live-reproduced in this read-only review.

## Solution Assessment

No, current main is not the best solution: it applies the scheduled-run auto-delete rule to manual force runs without an explicit destructive confirmation and does not synchronously gate deletion on durable run-history persistence. Maintainers should choose the manual-run policy before a narrow fix lands.
