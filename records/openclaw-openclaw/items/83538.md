## Label Justifications

- `P1`: A manual verification path can silently remove scheduled one-shot cron jobs and destroy future automation.
- `impact:data-loss`: The report concerns persisted cron job definitions being deleted without an intended scheduled completion.
- `impact:message-loss`: Deleting the job also removes the future scheduled fire that would have delivered the cron output.

## Summary

Keep open: current main still applies scheduled one-shot delete-after-run handling to manual cron completions, and an open linked PR with closing syntax is the active fix path.

## What This Changes

The issue reports that manually running a one-shot cron job with `deleteAfterRun: true` can silently delete the stored job and future scheduled fire.

## Best Possible Solution

Land or replace the linked cron fix so manual verification runs preserve the stored one-shot job and future scheduled fire, while timer-driven successful one-shot runs keep the existing delete-after-run behavior.

## Reproduction Assessment

Yes. Source inspection shows the default forced manual `cron.run` path enqueues a run that later calls `applyJobResult`, whose current successful one-shot `deleteAfterRun` branch deletes the job without checking whether the run was manual.

## Solution Assessment

Yes. The narrow maintainable solution is to distinguish manual cron result application from scheduled timer result application, preserving scheduled one-shot deletion while preventing manual verification runs from consuming the job.
