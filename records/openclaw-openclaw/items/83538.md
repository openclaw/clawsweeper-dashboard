## Summary

Keep open: current main still conflates manual and scheduled cron result handling, and the open linked fix PR explicitly closes this issue but has not merged yet.

## What This Changes

The issue reports that manual `openclaw cron run` can silently remove `deleteAfterRun` one-shot cron jobs and lose future scheduled fires without usable run-history proof.

## Best Possible Solution

Land or replace the linked cron service fix so manual runs are explicitly distinguished from scheduled runs, preserve pending one-shot schedules for manual outcomes, and keep scheduled `deleteAfterRun` behavior unchanged.

## Reproduction Assessment

Yes for the destructive core path: current main manual completion calls `applyJobResult` without a manual discriminator, and successful `deleteAfterRun` at-jobs return `shouldDelete=true`. I did not live-reproduce the exact no-history/no-execution timing variant in this read-only review.

## Solution Assessment

Yes, distinguishing manual from scheduled runs at `applyJobResult` is the maintainable boundary, and the open linked PR is the active candidate. This issue should remain open until that PR merges or maintainers choose a replacement fix.
