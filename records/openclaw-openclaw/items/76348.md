## Summary

Keep open: the performance direction is useful, but the PR has blocking cache-correctness issues around hook-mutated bootstrap context and auth-bound PI model discovery, plus the required changelog entry is missing.

## What This Changes

The PR adds in-memory LRU caches for resolved bootstrap contexts and PI model discovery, adjusts auth-store cache keys, passes channel id into bootstrap resolution, and adds plugin tool-cache kill switches.

## Best Possible Solution

Keep the perf work, but merge only after cache keys cover hook-produced bootstrap inputs and auth-bound registries, with focused regression tests and a single-line changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable bootstrap-extra-files, resolve context once, edit a matched extra file without touching root bootstrap files, and the PR cache can return stale context; the model cache issue is likewise source-reproducible by calling discoverModels with a different authStorage but the same agentDir/options.

## Solution Assessment

No. Caching final hook-mutated context and auth-bound registries is too broad; the safer fix is to cache only fully fingerprinted inputs, bypass final-context caching when mutable bootstrap hooks can affect files, and scope model discovery to the credential storage/source.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Do not reuse hook-adjusted bootstrap snapshots blindly:** `src/agents/bootstrap-files.ts:374-376`
  - body: The new cache returns before `applyBootstrapHookOverrides` runs. The key fingerprints raw bootstrap files and the hook registry version, but `bootstrap-extra-files` reads hook config and matched file contents during `agent:bootstrap`, so edits to injected files can keep serving stale context.
  - confidence: 0.92
- **[P2] Include auth storage in model discovery cache:** `src/agents/pi-model-discovery.ts:318-321`
  - body: `discoverModels` still accepts caller-provided `authStorage`, and the registry is bound to that object. This key fingerprints disk auth files/config but not the storage instance or runtime credential source, so a later call can reuse a registry built from stale credentials.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/agents/bootstrap-cache.ts:14-18`
  - body: This is a user-facing `perf` runtime change, and repo policy requires non-test/internal PRs to update `CHANGELOG.md`. The PR file list has no changelog entry.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI supply-chain change, but the PI model discovery cache crosses an auth-sensitive boundary until credential storage/source identity is part of the cache contract.

Concerns:

- **[medium] Auth-bound registry cache can reuse stale credentials:** `src/agents/pi-model-discovery.ts:318`
  - body: The cached PI model registry is constructed with a specific `authStorage`, but the cache lookup does not include that storage or runtime credential source. In long-lived processes or tests using in-memory credentials, model discovery can run against an older credential set.
  - confidence: 0.82
