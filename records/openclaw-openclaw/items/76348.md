## Summary

Keep this PR open, but it still needs changes before merge: the bootstrap-context cache can serve stale hook-mutated files, the model-discovery cache crosses an auth-storage boundary, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds in-memory runtime caches for bootstrap context resolution and PI model discovery, adjusts auth profile cache keys, passes channel id into bootstrap resolution, and adds a tool descriptor cache kill switch.

## Best Possible Solution

Keep the performance goal, but only merge a refreshed branch whose cache keys or bypass rules fully cover hook-produced bootstrap inputs and auth-bound model registries, with focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable bootstrap-extra-files, resolve bootstrap context once, edit a matched extra file without changing root bootstrap files, then resolve again and the PR can return the old cached context before hooks rerun.

## Solution Assessment

No. Caching final hook-mutated context and caller-auth-bound registries is too broad; a safer approach is to cache only fully fingerprinted inputs, bypass final-context caching when hooks can add files, and scope model discovery to the actual credential source.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Fingerprint hook-added files before reusing context:** `src/agents/bootstrap-files.ts:374-376`
  - body: The cache hit returns before `applyBootstrapHookOverrides` runs. Because `bootstrap-extra-files` reads configured globs and file contents during `agent:bootstrap`, editing a matched extra file can keep serving the old context until an unrelated raw bootstrap input changes.
  - confidence: 0.92
- **[P2] Include auth storage in the model cache boundary:** `src/agents/pi-model-discovery.ts:317-323`
  - body: `discoverModels` still accepts a caller-provided `authStorage`, and the returned registry is constructed from that object. The new key only fingerprints files/config, so a later call with a different in-memory or runtime auth store can receive a registry bound to stale credentials.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI supply-chain change, but the model-discovery cache introduces an auth-sensitive credential reuse risk.

Concerns:

- **[medium] Auth-bound registry cache can reuse stale credentials:** `src/agents/pi-model-discovery.ts:317`
  - body: The cached PI model registry is constructed with a specific authStorage, but the cache lookup does not include that storage or runtime credential source. In a long-lived process or test path using in-memory credentials, model discovery can run against an older credential set.
  - confidence: 0.82
