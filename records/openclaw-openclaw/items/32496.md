## Summary

Keep open. Current main still has the OpenAI-compatible penalty passthrough gap: config accepts the keys and preserves them, but the stream options and OpenAI-compatible request payload only map temperature, top_p, token caps, and response format; the direct penalty fields still require the documented extra_body workaround.

## What This Changes

The issue requests direct `frequency_penalty` and `presence_penalty` support for OpenAI-compatible provider model params.

## Best Possible Solution

Add finite-number passthrough for `frequency_penalty` and `presence_penalty` on OpenAI-compatible completions payloads, keep `extra_body` for vendor-specific fields, and treat top_k/repetition_penalty as separate scope.

## Reproduction Assessment

Yes, source-level. Configure `frequency_penalty` or `presence_penalty` under accepted model params; current main preserves the keys but never emits them into stream options or the OpenAI-compatible payload unless the user nests them under `params.extra_body`.

## Solution Assessment

Yes for the narrow request. A finite numeric mapping for the two documented OpenAI-compatible penalty fields is more maintainable than a broad unvalidated generation-parameter passthrough.
