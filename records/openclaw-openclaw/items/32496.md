## Summary

Keep open. Current main still accepts and preserves arbitrary model params, but the Pi/OpenAI-compatible path only forwards temperature, topP, token caps, and response format; direct frequency/presence penalties still require the documented extra_body workaround. The related implementation PRs are closed unmerged, and the broader generation/thinking tracker does not replace this narrow passthrough request.

## What This Changes

Add direct frequency_penalty and presence_penalty passthrough for OpenAI-compatible provider requests configured via model params.

## Best Possible Solution

Add finite-number passthrough for frequency_penalty and presence_penalty on OpenAI-compatible completions payloads, keep extra_body for vendor-specific fields, and leave top_k/repetition_penalty to provider-specific follow-up unless maintainers expand scope.

## Reproduction Assessment

Yes, source-level. Configure frequency_penalty or presence_penalty under accepted model params; current main preserves the keys but never emits them into stream options or the OpenAI-compatible payload unless the user nests them under params.extra_body.

## Solution Assessment

Yes for the narrow request. A finite numeric mapping for the two documented OpenAI-compatible penalty fields is more maintainable than a broad unvalidated generation-parameter passthrough.
