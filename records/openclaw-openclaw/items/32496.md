## Summary

Keep open. Current main has an `extra_body` escape hatch for OpenAI-compatible Completions proxies, but the direct `params.frequency_penalty` and `params.presence_penalty` keys requested here are still accepted as arbitrary params and then never normalized or forwarded; the linked fix PRs were closed unmerged.

## What This Changes

The issue requests forwarding `frequency_penalty` and `presence_penalty` from configured model params to OpenAI-compatible provider request payloads.

## Best Possible Solution

Add a focused finite-number passthrough for the common OpenAI-compatible penalty fields on supported routes, while leaving broader generation knobs to the open umbrella issues and `extra_body` for vendor-specific payloads.

## Reproduction Assessment

Yes, source-level. Configure the requested keys under the accepted `params` map, trace them through `resolveExtraParams`, and the current stream wrappers and OpenAI payload builders never emit them.

## Solution Assessment

Yes. A narrow transport-level mapping for the two documented penalty fields is the most maintainable path; requiring `extra_body` remains a lower-level workaround and should not be the only way to use common OpenAI-compatible sampling controls.
