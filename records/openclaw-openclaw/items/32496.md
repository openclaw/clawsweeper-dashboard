## Summary

Keep open. Current main still accepts arbitrary model params and offers `params.extra_body` as an advanced workaround, but it does not directly normalize or forward `frequency_penalty` or `presence_penalty`; the related implementation PRs were closed unmerged, and the broader generation-parameter issues do not replace this narrow request.

## What This Changes

Request direct support for `frequency_penalty` and `presence_penalty` in model params so OpenAI-compatible provider payloads receive those fields.

## Best Possible Solution

Add focused finite-number passthrough for `frequency_penalty` and `presence_penalty` on OpenAI-compatible payload routes, with regression coverage and docs that keep `extra_body` as the vendor-specific escape hatch.

## Reproduction Assessment

Yes, source-level. Configure `frequency_penalty` or `presence_penalty` under accepted model `params`; current main preserves the keys but never emits them into stream options or OpenAI-compatible payloads unless they are manually nested under `params.extra_body`.

## Solution Assessment

Yes for the narrow title. A finite numeric mapping for the two documented OpenAI-compatible penalty fields is narrower than adding a new `generation` block or broad vendor-parameter passthrough here.
