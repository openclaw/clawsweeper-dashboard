## Summary

Keep open. Current main still accepts arbitrary model params, but direct `frequency_penalty`/`presence_penalty` values are not normalized or forwarded by the shared extra-param wrapper or the OpenAI HTTP/WebSocket payload builders; the linked fix PRs were closed unmerged, and broader generation-parameter issues do not supersede this narrow request.

## What This Changes

The issue requests passthrough support for OpenAI-compatible `frequency_penalty` and `presence_penalty` model parameters from `openclaw.json` into provider API calls.

## Best Possible Solution

Add focused OpenAI-family transport passthrough for finite numeric frequency and presence penalties, preserving existing provider-specific wrappers and `extra_body` behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure those params, trace them through schema and `resolveExtraParams`, then observe that the shared stream wrapper and OpenAI payload builders never consume or emit them.

## Solution Assessment

Yes. A narrow transport-level mapping is the best path because the config surface already accepts params, while `extra_body` is only a lower-level workaround and the broader #47852/#71350 requests can remain separate.
