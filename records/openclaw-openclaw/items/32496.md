## Summary

Keep open. Current main still has the direct OpenAI-compatible penalty passthrough gap: config accepts these params and preserves them, but the stream wrapper and OpenAI-compatible payload only map temperature, top_p, token caps, and response format; the prior implementation PRs are closed unmerged.

## What This Changes

Request direct support for `frequency_penalty` and `presence_penalty` model params so OpenAI-compatible provider requests can receive configured repetition penalties.

## Best Possible Solution

Add finite-number passthrough for `frequency_penalty` and `presence_penalty` on OpenAI-compatible completions payloads, keep `extra_body` as the vendor-specific escape hatch, and leave `top_k`/`repetition_penalty` to separate scope.

## Reproduction Assessment

Yes, source-level. Configure `frequency_penalty` or `presence_penalty` under accepted model params; current main preserves those keys but never maps them into stream options or the OpenAI-compatible payload unless the user nests them under `params.extra_body`.

## Solution Assessment

Yes for the narrow request. A finite numeric mapping for the two documented OpenAI-compatible penalty fields is more maintainable than broad unvalidated generation-param passthrough.
