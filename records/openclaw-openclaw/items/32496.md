## Summary

Keep open. Current main still accepts and preserves arbitrary model params, but the OpenAI-compatible streaming path only forwards temperature, top_p, and token caps directly; frequency_penalty and presence_penalty still require the documented extra_body workaround, and the narrow related PRs were closed unmerged.

## What This Changes

Support direct frequency_penalty and presence_penalty model params for OpenAI-compatible providers.

## Best Possible Solution

Add finite-number passthrough for frequency_penalty and presence_penalty on OpenAI-compatible completions payloads, while keeping extra_body for vendor-specific fields and treating top_k or repetition_penalty as separate policy.

## Reproduction Assessment

Yes, source-level. Configure frequency_penalty or presence_penalty under accepted model params; current main preserves the keys but never emits them into stream options or OpenAI-compatible payloads unless the user manually nests them under params.extra_body.

## Solution Assessment

Yes for the narrow title. A finite numeric mapping for the two documented OpenAI-compatible penalty fields is more maintainable than a broad unvalidated generation-parameter passthrough.
