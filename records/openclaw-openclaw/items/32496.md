## Summary

Keep open. Current main still accepts arbitrary model `params` but does not directly normalize or forward `frequency_penalty` or `presence_penalty` on the OpenAI-compatible transport path; users can only work around this today with `params.extra_body`, and the linked implementation PRs were closed unmerged.

## What This Changes

The issue requests direct support for forwarding `frequency_penalty` and `presence_penalty` from model `params` to OpenAI-compatible provider requests.

## Best Possible Solution

Add a focused finite-number passthrough for `frequency_penalty` and `presence_penalty` on supported OpenAI-compatible payload routes, with regression coverage and docs that keep `extra_body` as the vendor-specific escape hatch.

## Reproduction Assessment

Yes, source-level. Configure `frequency_penalty` or `presence_penalty` under accepted model `params`; `resolveExtraParams` preserves them, but the current stream wrapper and OpenAI-compatible payload builder never emit those fields unless they are manually nested under `params.extra_body`.

## Solution Assessment

Yes for the narrow title. A finite-number mapping for the two documented OpenAI-compatible penalty fields is narrower than adding a new public `generation` block, while broader sampling aliases can remain with https://github.com/openclaw/openclaw/issues/47852 and https://github.com/openclaw/openclaw/issues/71350.
