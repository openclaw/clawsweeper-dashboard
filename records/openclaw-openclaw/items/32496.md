## Summary

Keep open. Current main partially improved nearby sampling support (`topP` and native Ollama params), but direct `frequency_penalty` and `presence_penalty` support for OpenAI-compatible model params is still absent; the narrow implementation attempts were closed unmerged and the broader related issues do not replace this request.

## What This Changes

The issue requests direct `frequency_penalty` and `presence_penalty` passthrough from configured model params to OpenAI-compatible provider requests.

## Best Possible Solution

Add finite-number passthrough for `frequency_penalty` and `presence_penalty` on OpenAI-compatible completions payloads, with focused regression coverage and docs that keep `extra_body` for vendor-specific fields.

## Reproduction Assessment

Yes, source-level. Configure `frequency_penalty` or `presence_penalty` under accepted model `params`; current main preserves the keys but never emits them into stream options or OpenAI-compatible payloads unless the user manually nests them under `params.extra_body`.

## Solution Assessment

Yes for the narrow title. A finite numeric mapping for the two documented OpenAI-compatible penalty fields is more maintainable than a broad unvalidated generation-parameter passthrough.
