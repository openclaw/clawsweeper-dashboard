## Summary

Keep open. Current main still accepts and merges arbitrary model `params`, and OpenAI-compatible proxy users can work around this with `extra_body`, but the direct `frequency_penalty` and `presence_penalty` keys requested here are not normalized or forwarded on the OpenAI-compatible transport path; the linked implementation PRs were closed unmerged. Native Ollama already has its own whitelist for these keys, which helps that plugin but does not resolve the requested OpenAI-compatible provider behavior.

## What This Changes

Request support for passing `frequency_penalty` and `presence_penalty` from model `params` to OpenAI-compatible provider payloads.

## Best Possible Solution

Add a focused finite-number passthrough for common OpenAI-compatible penalty fields on supported OpenAI-style payload routes, while keeping `extra_body` as the lower-level escape hatch for vendor-specific fields.

## Reproduction Assessment

Yes, source-level. Configure `frequency_penalty` or `presence_penalty` under accepted model `params`, trace through `resolveExtraParams`, and current OpenAI-compatible stream/payload builders never emit those fields unless the user manually nests them in `extra_body`.

## Solution Assessment

Yes. A narrow, finite-number transport-level mapping for the two documented penalty fields is more maintainable than requiring every user to know the `extra_body` escape hatch, while broader sampling knobs can remain with the umbrella generation-parameter issues.
