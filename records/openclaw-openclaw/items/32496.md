## Summary

Keep open. Current main accepts arbitrary model params, but direct `frequency_penalty` and `presence_penalty` are still absent from the Pi/OpenAI-compatible stream options and payload builder; only `extra_body` or native Ollama covers adjacent cases.

## What This Changes

Support top-level `frequency_penalty` and `presence_penalty` model params for OpenAI-compatible provider requests.

## Best Possible Solution

Add finite-number passthrough for `frequency_penalty` and `presence_penalty` on OpenAI-compatible completions payloads while keeping vendor-specific extras under `extra_body`.

## Reproduction Assessment

Yes, source-level. Configure `frequency_penalty` or `presence_penalty` under accepted model params; current main preserves the keys but never maps them into stream options or the OpenAI-compatible payload unless nested under `params.extra_body`.

## Solution Assessment

Yes for the narrow request. A finite numeric mapping for the two OpenAI-compatible penalty fields is more maintainable than a broad unvalidated generation-parameter passthrough.
