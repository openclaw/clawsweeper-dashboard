## Summary

Keep open. Current main still preserves arbitrary model params but does not directly forward `frequency_penalty` or `presence_penalty` through the Pi/OpenAI-compatible request path, and the related implementation PRs were closed unmerged.

## What This Changes

The issue requests direct support for `frequency_penalty` and `presence_penalty` in OpenAI-compatible model parameters.

## Best Possible Solution

Add finite-number passthrough for `frequency_penalty` and `presence_penalty` on OpenAI-compatible completions payloads while keeping `extra_body` for vendor-specific fields.

## Reproduction Assessment

Yes, source-level. Configure `frequency_penalty` or `presence_penalty` under accepted model params; current main preserves the keys but never maps them into stream options or the OpenAI-compatible payload unless the user nests them under `params.extra_body`.

## Solution Assessment

Yes for the narrow request. A finite numeric mapping for the two documented OpenAI-compatible penalty fields is more maintainable than a broad unvalidated generation-parameter passthrough.
