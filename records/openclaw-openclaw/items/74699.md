## Summary

Keep open. Current main still has the source-level path where default Codex app-server steering is accepted synchronously but a later delivery failure cannot reach the followup queue, and this PR is the narrow repair direction. The branch is not merge-ready because it is dirty against current main and the durable fallback needs to be reconciled with the newer completion-flush behavior.

## What This Changes

The PR makes Codex app-server queued steering report delivery success, adds a durable embedded-run queue helper, awaits that result in reply handling so rejected steering can fall back to a followup, and updates regression tests.

## Best Possible Solution

Land a rebased narrow fix that makes each default Codex queued user message either reach live `turn/steer` while the turn is still active or fall through to the existing followup queue when the turn is terminal or the app-server rejects steering, with regression coverage for the completion race.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a default `steer` message queued during an active Codex app-server run shortly before completion: current main acknowledges the message synchronously, then any delayed or rejected `turn/steer` has no result path back to `runReplyAgent()` for followup queueing.

## Solution Assessment

No, not as submitted against current main. Durable steering result propagation is the right narrow design, but the branch must be rebased and the completion-flush order adjusted so terminal turns produce a false delivery result instead of another internal `turn/steer` attempt.
