## Summary

Keep open. Current main still has a source-level gap where `runReplyAgent()` treats Codex steering as accepted before the debounced `turn/steer` request has actually succeeded, and failures are logged inside the steering queue without being routed back to the followup queue. The PR is a narrow implementation of the durable-delivery direction, but its head is dirty against current main and must be reconciled with the newer flush-on-completion behavior before merge.

## What This Changes

The branch makes Codex app-server queued steering report delivery success, adds a durable embedded-run queue helper, awaits that helper in `runReplyAgent()` so failed steering can fall back to followup queueing, updates queue handle types, and adds regression tests.

## Best Possible Solution

Land a rebased narrow fix that makes each default Codex queued user message either successfully reach live `turn/steer` or fall back through the existing followup queue when the turn is already terminal or the app-server rejects steering, with tests that reconcile the current completion-flush behavior to that contract.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a default `steer` message queued during an active Codex app-server run just before completion: current main acknowledges the message synchronously, then a delayed or failed `turn/steer` has no result path back to `runReplyAgent()` for followup queueing.

## Solution Assessment

Yes, the proposed durable-delivery direction is the narrow maintainable fix because it preserves live steering while making failure observable to the existing followup path. The current branch is not merge-ready until it is rebased and reconciled with current main’s later completion-flush change.
