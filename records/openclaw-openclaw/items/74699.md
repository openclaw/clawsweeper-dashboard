## Summary

Keep open. Current main still has a credible Codex steering durability gap: the caller treats a queued message as handled before the debounced `turn/steer` is accepted, and a later rejection is only logged. The PR is aimed at the right contract, but it is currently dirty against main and needs maintainer reconciliation with the later partial flush-on-completion behavior before it can be considered merge-ready.

## What This Changes

The branch makes Codex app-server queued steering report delivery success, adds a durable embedded-message helper, awaits it in `runReplyAgent` so rejected steering can fall back to followup queueing, updates queue-handle types, and adds regression tests.

## Best Possible Solution

Land a reconciled narrow fix that makes active-run steering delivery observable: each queued Codex message should either be accepted by live `turn/steer` before the turn is terminal or be queued through the existing followup path, with tests covering the current-main completion race and the caller fallback contract.

## Reproduction Assessment

Yes. The high-confidence source reproduction is a default `steer` message queued while a Codex app-server run is active and then completed before the debounce send is accepted: current main returns from `runReplyAgent()` after a synchronous true, while a later `turn/steer` rejection cannot schedule the documented followup.

## Solution Assessment

Yes, the proposed direction is the right maintainable shape because it makes steering delivery observable and reuses the existing followup queue. It is not merge-ready as-is because the branch is dirty against current main and must be reconciled with the later flush-on-completion change.
