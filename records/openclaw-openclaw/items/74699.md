## Summary

Keep open. Current main still has the source-level Codex queued-steering drop path, but this PR is dirty against current main and needs a focused repair for the newer completion-flush ordering before it can merge.

## What This Changes

The PR makes Codex app-server steering return accepted/rejected delivery, adds a durable embedded-run queue helper consumed by `runReplyAgent()`, adds focused regression tests, and aligns one changed-target routing test expectation.

## Best Possible Solution

Land a rebased narrow fix that makes each default Codex queued message either reach an accepted live `turn/steer` or enqueue a followup when the turn is terminal or steering rejects.

## Reproduction Assessment

Yes, by source-level control flow. A default `steer` message can be accepted while a Codex app-server run is active, then `turn/steer` rejection is swallowed after `runReplyAgent()` has already returned without scheduling a followup.

## Solution Assessment

No as-is. Durable delivery is the right fix direction, but the PR must be reconciled with current main's `turn/completed` flush so terminal batches resolve as rejected/followup instead of attempting `turn/steer` after completion.
