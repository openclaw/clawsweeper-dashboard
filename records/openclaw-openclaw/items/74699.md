## Summary

Keep open. Current main has a partial completion-time flush for pending Codex app-server steering, but it still acknowledges queued input before `turn/steer` delivery is known and still logs late/rejected steering internally without routing the message to the followup queue. The PR is a narrow implementation direction, but its head is dirty against current main and needs maintainer reconciliation with the newer completion-flush behavior before merge.

## What This Changes

The PR makes Codex app-server queued steering return delivery success, adds a durable embedded-run queue helper awaited by `runReplyAgent()`, widens queue handle return types, adds regression coverage, and updates a changed-target routing test expectation.

## Best Possible Solution

Land a rebased narrow fix that makes every default Codex app-server queued user message either reach a live `turn/steer` successfully or fall back through the existing followup queue when the active turn is terminal or the app-server rejects steering, with tests aligned to the documented queue contract.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a default `steer` message queued during an active Codex app-server run just before the turn becomes terminal: current main reports the message accepted synchronously, but a stale or rejected `turn/steer` has no result path back to `runReplyAgent()` for followup queueing.

## Solution Assessment

Yes in direction, but not merge-ready as-is. Returning durable steering success and awaiting it in the reply runner is the narrow maintainable fix; the branch needs a maintainer rebase to reconcile that with current main’s completion-time flush.
