## Summary

Keep open. Current main only partially fixes the original drop by flushing pending Codex steering at completion, but it still treats queued steering as delivered before `turn/steer` acceptance and has no rejected-steering path back to followup queueing; the PR is useful but needs maintainer reconciliation with current main before merge.

## What This Changes

The PR makes Codex app-server queued steering report accepted/rejected delivery, adds a durable embedded-run queue helper, awaits it from `runReplyAgent()` for followup fallback, and updates focused regression coverage.

## Best Possible Solution

Land a rebased narrow fix that makes each default Codex queued user message either reach an accepted live `turn/steer` or fall back through the existing followup queue when the turn is terminal or steering rejects.

## Reproduction Assessment

Yes, via source-level reproduction. A default `steer` message can be accepted while a Codex app-server run is active, then the later `turn/steer` rejection is swallowed while `runReplyAgent()` has already returned without scheduling a followup.

## Solution Assessment

Unclear as-is. Durable delivery is the right direction, but this branch should be reconciled with current main’s completion-flush implementation so accepted live steering is preserved and terminal or rejected steering falls back cleanly.
