## Summary

Keep open. Current main has only a partial source-level fix for the original drop path: it flushes pending Codex steering during completion, but `runReplyAgent()` still treats queued steering as synchronously delivered and cannot fall back if the later `turn/steer` is rejected. The PR’s durable-delivery direction is useful, but it is stale against current main’s completion-flush change and needs maintainer review before it is merge-ready.

## What This Changes

The PR makes Codex app-server queued steering report delivery success or failure, adds a durable embedded-run queue helper, awaits that helper in `runReplyAgent()` so failed steering can fall back to followup queueing, updates queue handle types, and adds regression tests.

## Best Possible Solution

Land a rebased narrow fix that preserves live Codex steering when it is still valid, but makes each default queued user message either reach `turn/steer` successfully or fall back through the existing followup queue when the turn is terminal or steering is rejected.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a default `steer` message queued during an active Codex app-server run near terminal completion, with the app-server rejecting a delayed `turn/steer`; current main acknowledges the message synchronously before that later failure can trigger followup queueing.

## Solution Assessment

Unclear as currently branched. Durable steering success/failure propagation is the right direction, but this PR must be rebased and adjusted so current main’s `turn/completed` flush does not bypass the terminal-state fallback contract.

## Review Findings

- **[P2] Block steering once completion is observed:** `extensions/codex/src/app-server/run-attempt.ts:841`
