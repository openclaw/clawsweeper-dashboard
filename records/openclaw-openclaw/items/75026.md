## Summary

Keep open: current main still has the MiniMax Anthropic-compatible usage propagation gap, and the linked PR #75136 is an open implementation candidate with closing syntax for this issue. The evidence points to a transport-boundary bug, not a Control UI display bug: MiniMax defaults to `anthropic-messages`, the Anthropic transport only copies native Anthropic usage keys, and downstream usage normalization can only preserve fields it receives.

## What This Changes

MiniMax Anthropic-compatible runs should persist nonzero token usage so the Control UI Usage page shows token counts, not only message counts.

## Best Possible Solution

Review, rebase if needed, and land PR #75136 or an equivalent narrow fix that normalizes compatible-provider usage at the Anthropic stream transport boundary before assistant snapshots and session persistence. The Control UI should continue to consume persisted session/transcript usage rather than special-case MiniMax or infer token counts from message totals.

## Reproduction Assessment

Yes. The issue gives exact setup details and raw nonzero provider usage, and current main can be checked with a deterministic Anthropic SSE fixture containing MiniMax/OpenAI-style `prompt_tokens`/`completion_tokens` or `message_stop` usage that the transport currently ignores.

## Solution Assessment

Yes. A transport-boundary fix is the narrow maintainable solution because it repairs session logs, status summaries, cost totals, and the Usage page together; a Control UI-only workaround would leave the persisted usage data wrong.
