## Summary

Keep open. The report is recent, externally reported, and matches the documented `current`/`session:<id>` cron behavior boundary; current `main` still routes session-bound cron `agentTurn`s through the same cron session persistence path and normal inbound freshness check without a regression test proving the next user turn keeps the same session.

## What This Changes

The issue reports that a cron `agentTurn` bound with `sessionTarget: "current"` runs in the active chat session but causes the next user message to start a new empty session.

## Best Possible Solution

Preserve the bound chat session entry, transcript identity, and lifecycle freshness after a `current` or explicit `session:<id>` cron `agentTurn`, so the following user message reuses the same session and history.

## Reproduction Assessment

No high-confidence live reproduction was run. The issue provides concrete logs, and source inspection shows the documented `current` path flows through bound-session cron persistence and then normal inbound freshness checks, so this is a source-reproducible bug candidate.

## Solution Assessment
