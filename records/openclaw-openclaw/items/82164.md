## Summary

Keep open. Current main still documents `current` and `session:<id>` cron jobs as context-preserving paths, and the source path still routes session-bound cron turns through the same session entry and inbound freshness machinery without a regression test proving the following user turn keeps the same session.

## What This Changes

The issue reports that a cron `agentTurn` bound with `sessionTarget: "current"` runs in the active chat session, but the next user message silently creates a new session and loses prior context.

## Best Possible Solution

Preserve the bound chat session entry, transcript identity, and lifecycle freshness after a `current` or explicit `session:<id>` cron `agentTurn`, so the following user message reuses the same session and history.

## Reproduction Assessment

No high-confidence live reproduction was run. The report has concrete logs, and current source shows the relevant path from `current` normalization through session-bound cron persistence into normal inbound freshness checks, but the exact reset still needs a failing regression or live proof on current main.

## Solution Assessment

Unclear because there is no proposed patch yet. The best fix should be a narrow cron/session persistence regression fix, not a new cron mode, reset setting, or broad routing redesign.
