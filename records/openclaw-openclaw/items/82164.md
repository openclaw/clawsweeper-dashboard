## Summary

Keep open. The report matches a documented cron/session behavior boundary: `current` is supposed to bind to the active conversation context, and current `main` still routes session-bound cron turns through the same session persistence/freshness machinery without a regression test proving the next inbound user turn keeps the same session.

## What This Changes

Reports that a `sessionTarget: "current"` cron `agentTurn` runs in the bound chat but causes the next user message to start a new session and lose conversation history.

## Best Possible Solution

Preserve the bound chat session entry, transcript identity, and lifecycle freshness after a `current` or explicit `session:<id>` cron `agentTurn`, so the following user message reuses the same session and history.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The issue provides concrete logs, and source inspection shows the documented `current` path flows through bound-session cron persistence and then normal inbound freshness checks, which is enough for a source-reproducible bug candidate.

## Solution Assessment

Unclear because there is no proposed patch yet. The best fix should be a narrow cron/session persistence regression fix, not a new cron mode or reset configuration change.
