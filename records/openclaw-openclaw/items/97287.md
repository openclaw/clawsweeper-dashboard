## Summary

Keep open: current main and v2026.6.10 still retry only Feishu rate-limit errors, so 99991663/99991664 token-invalid failures can still leave outgoing Feishu sends stuck until manual recovery. The issue is now paired with two open candidate PRs, so it should stay open until one is reviewed, proven, and merged.

## What This Changes

The issue asks for Feishu token-invalid errors 99991663/99991664 to invalidate cached token state and retry once so outgoing messages recover without manual gateway restart.

## Best Possible Solution

Land one Feishu-local fix that clears the SDK and CardKit token caches, re-resolves fresh token/client state for a single retry, preserves Feishu diagnostics, and includes focused tests plus redacted real Feishu proof before closing this issue.

## Reproduction Assessment

No live high-confidence reproduction was established in this review. Source inspection does reproduce the failure path: current main treats 99991663/99991664 as non-retryable while Feishu token/client caches can remain in use.

## Solution Assessment

Yes, the best direction is a Feishu-plugin-local one-shot invalidation and retry. The implementation must clear the token cache actually used by SDK-backed sends and CardKit fetches rather than adding a core fallback or config switch.
