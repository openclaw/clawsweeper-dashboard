## Summary

Keep open. Current main still does not expose the originating Telegram callback message id on generic agent callback turns, and the linked implementation PR at https://github.com/openclaw/openclaw/pull/55548 is still open, conflicting, and previously reviewed as incomplete rather than merged.

## What This Changes

The issue requests passing the originating Telegram inline-button `message_id` and related chat context through callback turns so agents can call `editMessage` on the source message.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/55548 with a rebased fix that carries source message id and chat id through every generic Telegram callback turn while preserving callback query identity for answering and dedupe.

## Reproduction Assessment

Yes, source-level. A Telegram `callback_query` with distinct `id` and `message.message_id` values reaches current generic callback paths that still use `callback.id` as `messageIdOverride`, and main has no separate source-message-id carrier.

## Solution Assessment

Yes, keeping this issue open is the right handling. The narrow maintainable fix is to review, rebase, land, or replace the linked PR rather than closing the issue before the callback metadata reaches main.
