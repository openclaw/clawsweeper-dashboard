## Summary

Keep open: current main and the latest release still expose generic Telegram callback clicks as `callback_data` text while keeping `callback.id` as the synthetic inbound message id, so the original Telegram `message.message_id` is still not available to ordinary agent turns. The prior target-repo fix PR was closed unmerged, and the remaining fix needs a maintainer decision on the callback metadata/current-message contract.

## What This Changes

Expose the originating Telegram callback message id to agents so inline-button handlers can edit the source message.

## Best Possible Solution

Add a Telegram-owned callback metadata path that preserves `callback.id` for answer/dedupe while exposing the source chat id and message id to prompt/tool context, with focused Telegram tests and docs for the new field.

## Reproduction Assessment

Yes, source-level: a Telegram callback query with distinct `callback.id` and `callback.message.message_id` reaches current callback paths that pass `callback.id` into `messageIdOverride`, and no separate source-message field exists on current main.

## Solution Assessment

Unclear until maintainers choose the permanent context shape. The direct id swap is not the best fix; the safer path is separate callback source-message metadata while preserving callback query identity for answer/dedupe.
