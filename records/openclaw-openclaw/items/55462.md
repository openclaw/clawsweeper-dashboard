- `P2`: This is a focused Telegram inline-button callback workflow gap with limited blast radius but real agent usability impact.
- `impact:session-state`: The issue is about missing or mis-associated callback source message identity in agent turn context.

## Summary

Keep open: current main and the latest release still route generic Telegram callback turns with the callback query id as the turn message id and do not expose a separate originating message id; the linked implementation PR is open, conflicting, and not merged.

## What This Changes

Expose the source Telegram inline-button callback message id so agents can edit the originating message after receiving callback data.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/55548 with a rebased fix that carries callback source message id and chat id through every generic Telegram callback turn while preserving callback query identity for answering and dedupe.

## Reproduction Assessment

Yes, source-level. A callback query with distinct `callback.id` and `message.message_id` follows current generic callback paths that set `messageIdOverride` to the callback id and have no separate callback source-message-id carrier.

## Solution Assessment

Yes for triage: keeping this open is the right outcome while the linked PR remains unmerged. The durable implementation should add explicit callback source metadata instead of replacing callback query identity in paths that need it for answering and dedupe.
