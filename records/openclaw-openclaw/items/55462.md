## Summary

Keep open. Current main and the latest release still do not expose the source Telegram callback message id on the generic agent callback path, and the linked implementation PR remains open and unmerged.

## What This Changes

The issue requests that Telegram inline-button callback events expose the originating message id so agents can edit the same message after a button click.

## Best Possible Solution

Land a rebased equivalent of the linked PR that carries `callbackMessage.message_id` and chat id through every generic Telegram callback turn, preserves `callback.id` for answering/dedupe, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-level. A `callback_query` with `id: "cbq-1"` and `message.message_id: 10` reaches generic callback routing that sets `messageIdOverride` from the callback id, and the context builder then exposes that override as the message id.

## Solution Assessment

Yes, keeping this open is the best current handling. The narrow maintainable path is to review, rebase, land, or replace https://github.com/openclaw/openclaw/pull/55548 rather than closing this issue before the fix reaches main.
