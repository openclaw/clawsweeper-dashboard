## Summary

Keep open: current main and latest release still route generic Telegram callback clicks as `callback_data` text and keep `callback.id` as the synthetic inbound message id, with no separate agent-visible source `message_id` carrier. The target-repo fix PR is closed unmerged and the remaining work needs maintainer agreement on the callback metadata shape.

## What This Changes

Request to expose the Telegram `callback_query` source message id and chat id to agent turns so inline-button handlers can edit the originating Telegram message.

## Best Possible Solution

Add a Telegram-owned callback metadata path that preserves `callback.id` for answer/dedupe while exposing the source message id and chat id to prompt/tool context, with focused tests and docs for the new field.

## Reproduction Assessment

Yes, source-level: a Telegram callback query with distinct `callback.id` and `callback.message.message_id` reaches current callback paths that pass `callback.id` into `messageIdOverride`, and no separate source-message field exists on current main.

## Solution Assessment

Unclear until maintainers choose the permanent context shape. Exposing the source message id is the right direction, but the best fix should keep callback query identity for answer/dedupe and carry source-message metadata separately.
