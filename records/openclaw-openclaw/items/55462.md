## Summary

Keep this open. Current main and the latest release still do not expose the Telegram callback source message id to the agent-visible inbound context, and the remaining work is already paired with the open implementation PR at https://github.com/openclaw/openclaw/pull/55548 rather than being ready to close.

## What This Changes

Request to include the Telegram callback query source message id, and likely chat id, in agent-visible context so agents can edit the originating inline-button message.

## Best Possible Solution

Land or replace the linked PR with a bounded Telegram callback metadata fix that exposes source message id and chat id to agent prompt/tool context while preserving callback query identity for answering and dedupe.

## Reproduction Assessment

Yes, source-level. A Telegram `callback_query` with distinct `callback.id` and `callback.message.message_id` reaches current callback paths that pass `callback.id` into `messageIdOverride`, and no separate callback source-message field exists on main.

## Solution Assessment

Yes to keeping the issue open for the linked implementation path, but no to a one-line `MessageSid` swap. The safer fix is a separate source-message metadata/current-message carrier that does not break callback dedupe identity.
