## Summary

Keep open: current main and the latest release still route generic Telegram callback clicks as agent-visible `callback_data` text while preserving `callback.id` as the inbound `MessageSid`, so agents still lack a safe source-message id for editing the clicked inline-button message. The related target-repo PR is closed unmerged, and the remaining work needs a deliberate callback metadata/current-message contract rather than cleanup closure.

## What This Changes

The issue requests that Telegram inline-button callback turns expose the source message id so agents can call `editMessage` on the clicked message.

## Best Possible Solution

Add a Telegram-owned callback metadata/current-message path that preserves `callback.id` for answering and dedupe while exposing the source chat/message ids to generic agent turns and message actions across fallback and managed callback paths.

## Reproduction Assessment

Yes, source-level: a Telegram `callback_query` with distinct `callback.id` and `callback.message.message_id` reaches current callback paths that pass `callback.id` into `MessageSid`, while the generic agent payload only contains `callback_data`.

## Solution Assessment

No final solution has landed. The best fix is a deliberate callback source-message metadata/current-message contract, not a simple id replacement that could break callback dedupe or callback answering.
