## Summary

This issue is still necessary: current main and the latest release still route generic Telegram callback clicks as `callback_data` text while using the callback query id as the inbound message id, so agents still lack the source Telegram message id needed for `editMessage`. The related target-repo PR is closed unmerged and the remaining open work is a maintainer decision about the callback metadata contract, not cleanup closure evidence.

## What This Changes

The issue asks for Telegram callback-query turns to expose the source message id, and likely chat id, so agents can edit the original inline-button message.

## Best Possible Solution

Add a Telegram-owned callback metadata path that preserves `callback.id` for answering and dedupe while exposing the source chat/message ids to agent turns and message actions across generic and managed callback paths.

## Reproduction Assessment

Yes, source-level. A Telegram callback query with distinct `callback.id` and `callback.message.message_id` reaches current callback paths that pass `callback.id` into `messageIdOverride`, while the generic agent payload only contains `callback_data`.

## Solution Assessment

No final solution has landed. The safest fix is a deliberate callback source-message metadata/current-message contract that keeps callback query identity separate, not a simple id swap.
