## Summary

Keep open: current main and v2026.5.18 still route generic Telegram callback turns with the callback query id as `MessageSid` and do not expose a separate originating Telegram message id to the agent. The linked implementation PR is open, conflicting, and unmerged, so this issue should remain the canonical tracker for the callback source-message-id gap.

## What This Changes

Request to expose the originating Telegram inline-button callback message id, alongside callback data, so agents can edit the source message after a callback.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/55548 with a rebased Telegram fix that carries callback source `message_id` and `chat_id` through every generic callback turn while preserving callback query identity for answering and dedupe.

## Reproduction Assessment

Yes, source-level. A Telegram callback query with distinct `callback.id` and `message.message_id` still follows paths that set `messageIdOverride` to the callback id and expose no separate source-message-id carrier.

## Solution Assessment

Yes for triage: keeping this issue open is the right outcome while the linked PR remains unmerged. The best implementation should add explicit callback source metadata rather than repurposing the callback query id field that answering and dedupe may still need.
