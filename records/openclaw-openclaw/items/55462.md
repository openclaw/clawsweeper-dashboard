- `impact:session-state`: The issue is about missing or mis-associated callback source message identity in the agent turn context.

## Summary

Keep open: current main and v2026.5.18 still route generic Telegram callback turns with the callback query id as the inbound message id and do not expose a separate originating Telegram message id to agents. The open linked implementation PR at https://github.com/openclaw/openclaw/pull/55548 remains the active fix path, but it is currently conflicting and unmerged.

## What This Changes

Telegram callback query turns should expose the originating Telegram message id to agents so editMessage can target the source message after an inline-button click.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/55548 with a rebased Telegram fix that carries callback source message id and chat id through every generic callback turn while preserving callback query identity for answering and dedupe.

## Reproduction Assessment

Yes, source-level. A Telegram callback_query with distinct callback.id and message.message_id still reaches current generic callback paths that set messageIdOverride to callback.id and expose no separate source-message-id carrier.

## Solution Assessment

Yes for triage: keeping this issue open is the right outcome while the linked implementation PR remains unmerged. The implementation should add explicit callback source metadata rather than repurposing the callback query id field that answering and dedupe still need.
