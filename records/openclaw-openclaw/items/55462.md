## Summary

This issue should stay open: current main still turns ordinary Telegram callback clicks into `callback_data` text while using the callback query id as the synthetic inbound message id, so agents still do not get the source Telegram message id needed for `editMessage`. A linked fork PR is open but conflicting and unmerged, so it is an implementation candidate rather than cleanup closure evidence.

## What This Changes

The issue asks for Telegram inline button callback turns to expose the source Telegram message id so agents can edit the original message after a callback click.

## Best Possible Solution

Land or replace the linked implementation with a Telegram-owned callback metadata path that preserves callback query identity for answer/dedupe while exposing source chat/message ids for agent edit actions.

## Reproduction Assessment

Yes. Source inspection shows a Telegram callback query with distinct `callback.id` and `callback.message.message_id` reaches current callback paths that pass `callback.id` into `messageIdOverride`, while no separate source-message field exists on current main.

## Solution Assessment

No final solution has landed. The safest direction is separate callback source-message metadata or a deliberate current-message contract change, not an unreviewed one-line replacement of callback query identity.
