## Summary

This issue is still necessary: current main and the latest release still route ordinary Telegram inline-button callbacks as `callback_data` text while using the callback query id as the synthetic inbound message id, so agents still do not get the source Telegram message id needed for edit actions. Plugin-owned callbacks now have structured source-message metadata, but that does not solve the generic agent-turn path reported here.

## What This Changes

Request to expose the source Telegram callback message id, and likely chat id, to agent-visible callback turns so agents can edit the original inline-button message.

## Best Possible Solution

Land or replace the implementation with a Telegram-owned callback metadata path that preserves callback query identity for answer/dedupe while exposing source chat and message ids to agent edit/current-message tooling.

## Reproduction Assessment

Yes, source-level. Current main has callback fixtures with distinct `callback.id` and `message.message_id`, and the runtime paths still pass `callback.id` into the synthetic agent turn without a generic source-message metadata carrier.

## Solution Assessment

No final solution has landed. The maintainable direction is to carry source callback message metadata separately, or deliberately feed it into current-message tooling, while retaining callback query identity for answer and dedupe behavior.
