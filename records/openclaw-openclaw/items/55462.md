## Summary

Keep open: current main and the latest release still route generic Telegram callback-query agent turns with `messageIdOverride: callback.id` and no separate source `message_id`, while the linked implementation PR remains open and should be reviewed rather than bypassed with a new cleanup close.

## What This Changes

The issue requests forwarding the Telegram callback query source message ID to the agent so inline-button handlers can call `editMessage` on the original message.

## Best Possible Solution

Review and either land or replace the linked Telegram PR with an owner-approved inbound metadata shape that exposes the source callback message id to generic agent turns and validates the `editMessage` flow.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: trigger a generic Telegram `callback_query` with `data` and `message.message_id`, and the current handler passes `callback.id` as the inbound `MessageSid` without exposing the source message id.

## Solution Assessment

Unclear until maintainers choose the public metadata shape; the fix belongs in the Telegram inbound callback path, but the active PR still needs review because it changes agent-visible context rather than only a private implementation detail.
