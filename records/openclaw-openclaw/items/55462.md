- `P2`: This is a focused Telegram inline-button callback workflow gap with real usability impact but limited channel-specific blast radius.
- `impact:session-state`: The issue is about missing callback source message identity in the agent turn context, which can mis-associate callback turns and edit targets.

## Summary

Keep open: current main and v2026.5.19 still do not expose a separate Telegram callback source message id to agent turns, and the open linked implementation PR remains the active fix path.

## What This Changes

The issue asks for Telegram callback turns to expose the originating message_id and chat id so agents can call editMessage after inline-button clicks.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/55548 with a rebased fix that carries callback source message id and chat id as explicit metadata while preserving callback.id for answering and dedupe.

## Reproduction Assessment

Yes, source-level. A Telegram callback_query with distinct callback.id and message.message_id reaches current generic callback paths that set messageIdOverride to callback.id and expose no separate source-message-id carrier.

## Solution Assessment

Yes for triage: keeping this issue open is the right path while the linked implementation PR remains unmerged. The implementation should add explicit callback source metadata instead of repurposing the callback query id field.
