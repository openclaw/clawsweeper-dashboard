## Summary

Keep open. Current main and v2026.5.12 still route generic Telegram callback turns without a separate source message-id carrier, and an open implementation PR explicitly closes this issue, so it should stay paired until that PR lands, is replaced, or is closed.

## What This Changes

The issue asks for Telegram inline-button callback turns to expose the originating message id so agents can edit the original message.

## Best Possible Solution

Land a rebased replacement for https://github.com/openclaw/openclaw/pull/55548 that carries the callback source message id and chat id through every generic Telegram callback turn while preserving `callback.id` for answering and dedupe.

## Reproduction Assessment

Yes. A Telegram `callback_query` with a distinct `id` and `message.message_id` reaches current generic callback paths that pass `callback.id` into `messageIdOverride`, which becomes the agent-visible `MessageSid`.

## Solution Assessment

Yes, keeping this open is the best current handling. The narrow fix should split callback event identity from source message identity and move through the already-linked PR or a replacement, not close this issue before merge.
