- `impact:session-state`: The missing source message id affects agent-visible callback context and downstream threading/edit-message state for the originating Telegram message.

## Summary

Keep open: current main and v2026.6.6 still route generic Telegram callback clicks as `callback_data` text with `callback.id` as the inbound message id, and there is no separate agent-visible source `message_id` carrier yet. The older target PR was closed unmerged, while the remaining fix needs maintainer agreement on the context shape because a one-line `MessageSid` swap would break callback dedupe.

## What This Changes

Expose the Telegram callback_query source message id, and likely chat id, to agent context so `editMessage` can update the original inline-button message.

## Best Possible Solution

Add a Telegram-owned callback metadata path that preserves `callback.id` for answering and dedupe while exposing the source message id and chat id to agent prompt/tool context, with focused callback tests and Telegram docs updated for the new field.

## Reproduction Assessment

Yes, source-level. A Telegram callback_query with distinct `callback.id` and `callback.message.message_id` reaches callback paths that pass `callback.id` into `messageIdOverride`, and current main has no separate callback source-message field.

## Solution Assessment

Unclear until maintainers choose the permanent context shape. Keeping callback query identity for dedupe while adding source-message metadata is safer than replacing `MessageSid` outright.
