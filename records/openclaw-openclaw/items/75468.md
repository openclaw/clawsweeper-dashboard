## Summary

Keep open. Current main has opt-in same-params/no-progress tool-loop detection, but it still lacks the requested content-aware write/edit dedupe and a standard-run tool-call cap; the linked related issues cover Ralph-specific and bot-to-bot loops, not this file-write loop.

## What This Changes

Request to add agent safety guardrails that skip duplicate write/edit content and cap standard-run tool executions to prevent runaway file-write loops.

## Best Possible Solution

Keep this open as the canonical design item for a small extension of the existing file-tool and tool-loop surfaces: track recent write/edit content fingerprints per run, return a clear duplicate/unchanged result, add a bounded run-level tool-call limit, and document/test the config semantics.

## Reproduction Assessment

Yes for the missing safeguards: current source shows no content hash dedupe in write/edit operations, loop detection is opt-in and same-params based, and max_tool_calls is documented as ignored. The exact reported runaway session is only partially reproducible without the original trajectory.

## Solution Assessment

Unclear. The requested direction is sound, but the proposed same-target hash check would not catch identical content written to versioned filenames; the safer solution should fingerprint recent write/edit content and integrate limits through the existing loop-detection/config surface.
