## Summary

Keep open. Current main has opt-in same-params/no-progress loop detection, but it still lacks the requested content-aware write/edit dedupe and a standard run-level tool-call cap; the remaining design needs maintainer decisions about scope, defaults, and false positives.

## What This Changes

Add safeguards that detect duplicate-content file write/edit loops and enforce session- or run-level limits on repeated tool execution.

## Best Possible Solution

Extend the existing loop-detection/file-tool wrapper surface with explicit config, per-run recent write/edit content fingerprints that cover versioned filenames, same-path unchanged no-op results, a bounded tool-call counter, and docs/tests for the intended stop signals.

## Reproduction Assessment

Yes for the missing safeguards: current source shows no content-hash dedupe around write/edit execution, and `max_tool_calls` is documented as accepted but ignored. The exact 382-file session is not fully reproducible from the report because no runnable trajectory was provided.

## Solution Assessment

Unclear. The requested direction is sound, but a same-target hash check alone would not catch identical content written to new versioned filenames; the safer design is recent per-run content fingerprinting plus a configurable run cap integrated with existing loop detection.
