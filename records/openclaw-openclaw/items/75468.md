## Summary

Keep open. Current main still lacks content-aware write/edit dedupe and a general run/session tool-call cap; the existing loop detector is opt-in and pattern-based, while the linked #75924 work is narrower consecutive-error detection rather than this design.

## What This Changes

Add agent safeguards for runaway loops, including content-hash dedupe for file writes/edits and configurable run-level tool-call or identical-operation limits.

## Best Possible Solution

Keep this as the canonical design item for a core agent safeguard that adds run-scoped content fingerprinting for write/edit and a documented tool-call budget aligned with the existing loop-detection surface.

## Reproduction Assessment

Yes for the missing safeguards: source and docs show no content-hash write/edit dedupe, no requested safeguard config fields, and max_tool_calls is still ignored. The original runaway model trajectory is only partially reproducible without the full transcript.

## Solution Assessment

Unclear. The requested direction is sound, but the proposed same-target hash check would miss versioned filenames, so the maintainable fix should define cross-path fingerprint scope and explicit run-budget semantics first.
