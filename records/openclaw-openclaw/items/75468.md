## Summary

Keep open. Current main has opt-in tool-loop detection, but it is disabled by default, largely keys off identical tool arguments, and does not add content-aware write/edit dedupe or a supported per-turn tool-call cap for normal agent runs.

## What This Changes

Request adds agent safeguards for duplicate file write/edit content detection plus run-level tool-call or identical-operation limits to stop runaway autonomous loops.

## Best Possible Solution

Keep the issue open and design this as an extension of the existing `tools.loopDetection` and file-tool wrappers: track recent write/edit content fingerprints per run, return a clear unchanged/duplicate result to the model, add a bounded run-level tool-call cap, and document/test the versioned-filename duplicate case.

## Reproduction Assessment

Yes for the missing safeguards: current main shows write/edit paths do not hash or skip duplicate content, and loop detection is disabled by default with same-params matching. The exact historical 382-file runaway was not replayed because the report does not include a runnable session trace.

## Solution Assessment

Unclear. The requested direction is sound, but a same-existing-file hash check alone would not stop identical content under new versioned filenames; a safer solution should combine recent content fingerprints with the existing loop-detection config surface and a documented run cap.
