## Summary

Keep open. Current main has opt-in same-params/no-progress tool-loop detection, but it still lacks the requested content-aware write/edit dedupe and a standard run-level tool-call cap, so the reported agent-safety gap remains actionable.

## What This Changes

Request to add agent safeguards that skip duplicate write/edit content and cap standard agent tool-call iterations to stop runaway loops.

## Best Possible Solution

Extend the existing tool-loop safety surface with run-scoped write/edit content fingerprints, a clear unchanged/duplicate tool result, and a documented configurable tool-call cap, with tests that cover identical content written to new filenames as well as same-path no-op writes.

## Reproduction Assessment

Yes for the missing safeguards: current source and docs show no content-hash skip path and no enforced max_tool_calls cap. The exact historical 382-file loop is not fully reproducible without the original session trace and config.

## Solution Assessment

Unclear. The requested direction is sound, but the same-path content hash proposal is not sufficient for the reported versioned-filename loop; the safer solution is a run-scoped content fingerprint plus a general cap integrated with existing loop detection.
