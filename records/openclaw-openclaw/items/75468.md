## Summary

Keep open. Current main still has only opt-in, same-params/no-progress tool-loop detection and does not implement content-aware write/edit dedupe or a standard-run max tool-call cap; the related Ralph and bot-to-bot issues cover different loop surfaces.

## What This Changes

Request agent safeguards that prevent repeated identical file writes and add configurable run-level tool execution limits for ordinary agent turns.

## Best Possible Solution

Keep this open as the canonical design item for extending the existing agent tool-loop surface: add content-aware write/edit fingerprinting that can catch versioned duplicate outputs, wire a bounded run-level tool-call limit through config/docs/tests, and preserve the related Ralph and bot-to-bot reports as separate loop classes.

## Reproduction Assessment

Yes for the missing safeguards: source and docs show no content hash dedupe, no requested safeguard config fields, and `max_tool_calls` is still ignored. The exact runaway session is only partially reproducible without the original trajectory.

## Solution Assessment

Unclear. The requested direction is sound, but the proposed same-target hash comparison would not catch identical content written to versioned filenames; a better fix should integrate recent content fingerprints and run limits into the existing loop-detection/config surface.
