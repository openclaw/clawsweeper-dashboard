## Summary

Keep open. Current main has useful opt-in repetitive tool-call detection, but it does not implement the requested same-content file-write safeguard or a general session/run-level max tool-call limit, and the reported versioned-filename loop would evade the existing same-tool/same-arguments detector.

## What This Changes

The issue requests agent safeguards for duplicate file writes, repeated identical operations, and session/run-level tool-call iteration caps.

## Best Possible Solution

Define one core agent-safeguard design around the existing `tools.loopDetection` path: add an explicit per-run tool-call cap, add clear no-op handling for same-path unchanged writes/edits, and decide whether a configurable repeated-content detector should catch cross-path duplicate writes without breaking normal multi-file generation.

## Reproduction Assessment

No for the exact 382-file incident: the issue includes file evidence but not a runnable prompt/model/tool transcript. The current source does give high-confidence static proof of the missing guard because fresh versioned paths go straight to write operations and evade same-argument loop detection.

## Solution Assessment

Unclear as proposed. Same-path hash dedup is safe but insufficient for the described versioned-file loop; the stronger solution should combine per-run caps with carefully scoped repeated write-content detection in the tool runtime.

- The max-tool-call portion overlaps open #9912, and edit safety overlaps #72555/#58726, so maintainers should choose one canonical implementation scope before a broad fix PR.
