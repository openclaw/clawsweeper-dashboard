## Summary

Keep open. Current main still only has opt-in same-args/no-progress/ping-pong tool-loop detection; write/edit paths do not content-dedupe identical file bodies, and `max_tool_calls` is still accepted but ignored. The related #75924 work is narrower error-cascade detection, not a replacement for this design item.

## What This Changes

Request to add agent safeguards for duplicate write/edit content, run or turn-level tool-call limits, and optional repeated-output detection.

## Best Possible Solution

Keep this as the canonical core safeguard design item, with a run-scoped content-fingerprint signal for write/edit and a documented tool-call budget aligned with existing loop detection.

## Reproduction Assessment

Yes for the missing safeguards: current source and docs show no content-hash write/edit dedupe, no requested safeguard config fields, and ignored `max_tool_calls`. The original runaway model trajectory is not fully reproducible from the report alone.

## Solution Assessment

Unclear. The requested direction is sound, but the proposed same-target hash check would miss the versioned-filename loop, so the maintainable fix should first define cross-path fingerprint scope and explicit run-budget semantics.
