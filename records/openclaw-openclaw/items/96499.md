## Summary

Keep open: the PR addresses a real unbounded success-body read, but the `/embeddings` path now uses the generic 16 MiB provider JSON cap instead of the existing 64 MiB remote-embedding response budget, which can break large valid memory embedding batches before merge.

## What This Changes

The PR changes GitHub Copilot model discovery and embedding success JSON parsing to use the bounded provider JSON reader and updates the embedding tests to use real `Response` objects and the new malformed-JSON message.

## Best Possible Solution

Keep the discovery hardening as-is, but make the `/embeddings` success-reader preserve the established remote embedding response budget or add a proven split path before merge.

## Reproduction Assessment

Yes, source-reproducible: current main uses `await response.json()` on both successful Copilot JSON paths, and the PR body includes node:http proof that the old path reads a full oversized body while the bounded reader cancels near the cap. I did not rerun the proof locally in this read-only review.

## Solution Assessment

No, not yet: using a shared bounded reader is the right direction, but applying the generic 16 MiB cap to `/embeddings` is narrower than the existing remote embedding helper behavior. The better fix is to preserve the larger embedding response budget while still bounding the stream.
