## Summary

Keep open: this is a valid Copilot embedding hardening PR, but it is not merge-ready because the `/embeddings` success path adopts the generic 16 MiB provider JSON cap, while the existing remote embedding JSON helper budget is 64 MiB and memory indexing can send batched embedding requests.

## What This Changes

The PR changes GitHub Copilot model discovery and embedding success-response parsing to use `readProviderJsonResponse`, updates setup-error matching, and adjusts embedding tests to use real `Response` bodies.

## Best Possible Solution

Keep model discovery on the generic bounded reader, but make the `/embeddings` success reader preserve the established remote embedding response budget and cover both a large valid batch and an oversized stream in tests.

## Reproduction Assessment

Yes. Current main still uses unbounded `response.json()` on both successful Copilot JSON paths, and the PR body includes node:http proof that the old path reads the whole oversized body while the bounded reader cancels near the cap.

## Solution Assessment

No, not as submitted. The shared bounded reader is the right direction, but `/embeddings` should preserve the established remote embedding response budget instead of inheriting the generic 16 MiB provider JSON cap.
