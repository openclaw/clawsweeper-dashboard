## Summary

Keep open: current main still omits `messages` from the direct `before_compaction` hook payload, but this PR should not merge as written because it exposes the live session array and only weakly asserts the payload contents.

## What This Changes

The PR adds `messages: session.messages` to the direct compaction `before_compaction` hook payload and adds focused assertions for the new field.

## Best Possible Solution

Thread a defensive validated/limited pre-compaction message snapshot through `runBeforeCompactionHooks`, then assert non-empty, empty, and post-compaction-stability behavior.

## Reproduction Assessment

Yes, by source inspection: current main sends only counts in the direct helper, while the hook type already allows `messages` and the wired compaction path has a message payload test.

## Solution Assessment

No, not in the current PR form. Passing the field is the right direction, but the maintainable fix is to pass a snapshot through the current helper boundary and strengthen the assertions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass a snapshot to before_compaction hooks:** `src/agents/pi-embedded-runner/compact.ts:762`
  - body: The new payload exposes `session.messages` by reference. Hooks run before the real-content check and `session.compact(...)`, and the harness mutates that same array, so a plugin that mutates or retains `event.messages` can alter or observe the wrong compaction input. Pass a shallow snapshot of the validated pre-compaction messages instead.
  - confidence: 0.92
- **[P3] Assert the message payload contents:** `src/agents/pi-embedded-runner/compact.hooks.test.ts:376`
  - body: The added non-empty assertion accepts any array, so an empty or stale payload would still pass. Assert the expected pre-compaction length/content, and preferably that the captured payload remains stable after compaction mutates session state.
  - confidence: 0.84
