## Summary

Keep open. Current main still lets a successful metadata-only async exec completion become a heartbeat agent turn; if the model replies with missing-context prose instead of the expected quiet sentinel/tool response, that prose can still be delivered to Telegram or another chat surface.

## What This Changes

Suppress successful async/background exec completions with no captured output so chat surfaces do not show missing-context agent replies after session resolution fails.

## Best Possible Solution

Consume successful structured exec completions with no captured output at runtime before any agent-visible relay turn, while keeping output-bearing successes and failures relayable with concise user-facing text.

## Reproduction Assessment

Yes at source level: enqueue `Exec completed (..., code 0)` with no output, have the mocked heartbeat agent return visible missing-context text instead of `HEARTBEAT_OK` or `heartbeat_respond notify=false`, and current delivery code will treat it as outbound text. I did not establish a live Telegram reproduction from logs.

## Solution Assessment

No. The current prompt-based suppression is not the best fix because it depends on model compliance; the narrower maintainable fix is a runtime-level consume guard for successful metadata-only completions before the model turn.

## Repair Work Prompt

Fix the async exec completion suppression bug described in https://github.com/openclaw/openclaw/issues/76523. Treat successful structured exec completions with no captured stdout/stderr as runtime-consumed before any heartbeat/agent relay turn, so a model-generated missing-context reply cannot reach Telegram or other chat surfaces. Preserve delivery for output-bearing successful completions and failed/signal completions, and keep #75322's `deliverToUser=false` behavior in mind without broadening the scope. Add regression coverage where `Exec completed (..., code 0)` with no output and a mocked visible missing-context agent reply does not call Telegram/outbound delivery. Also review the exec approval followup direct path that emits `Background command finished.` and only change it if it participates in the same no-output leak. Do not add a config option, do not change provider auth/session reset policy, and do not suppress failed or output-bearing completion notices.
