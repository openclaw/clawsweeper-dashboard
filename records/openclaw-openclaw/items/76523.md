## Summary

Keep open. Current main still handles successful metadata-only async exec completions through a model prompt and only suppresses delivery if the agent returns a recognized silent reply, so a missing-context apology like the reported Telegram reply can still be delivered.

## What This Changes

Prevent successful async/background exec completions with no captured output or lost session context from producing user-visible agent replies asking for the original command or logs.

## Best Possible Solution

Consume successful metadata-only exec completions at runtime before any agent-visible relay turn, while preserving relay behavior for output-bearing successes and failures.

## Reproduction Assessment

No high-confidence live Telegram reproduction is available from the report alone. The source path is reproducible: enqueue `Exec completed (..., code 0)` with no output and have the mocked agent return visible missing-context text; current skip logic would deliver it.

## Solution Assessment

No. The current prompt-based suppression depends on model compliance, while the narrow maintainable fix is to consume non-relayable successful completions before the model can generate user-visible text.

## Repair Work Prompt

Fix the async exec completion leak tracked by https://github.com/openclaw/openclaw/issues/76523. Successful metadata-only completions such as `Exec completed (id, code 0)` with no stdout/stderr must be consumed before any agent-visible relay turn, even if a model would return visible missing-context text. Keep failed completions and output-bearing completions relayable. Start in `src/infra/heartbeat-events-filter.ts` and `src/infra/heartbeat-runner.ts`; check `src/agents/bash-tools.exec-approval-followup.ts` only to confirm whether its direct `Background command finished.` status remains intentional or should be suppressed for the same no-output success case. Add regression coverage where a mocked heartbeat reply asks for missing context/logs and Telegram is not called. Preserve #75322 as a distinct `deliverToUser=false` issue, add a single-line CHANGELOG Fixes entry for this issue, and do not add config flags or provider-specific handling.
