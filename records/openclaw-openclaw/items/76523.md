## Summary

Keep open. Current main still routes a successful metadata-only async exec completion through an agent prompt and only suppresses delivery if the model returns the expected silent token, so a model-generated missing-context apology can still reach Telegram; the open #75322 path is related but covers `deliverToUser=false`, not this user-visible relay path.

## What This Changes

The issue reports that async exec completions with lost session context can still produce user-visible “no context for session” replies after the #74595 metadata-only completion fix.

## Best Possible Solution

Make successful metadata-only exec completions a runtime-level silent consume, with regression coverage proving that even a model-style missing-context reply is not delivered to Telegram or other chat surfaces.

## Reproduction Assessment

No high-confidence live reproduction is available from the report alone, but the source path is reproducible: enqueue `Exec completed (..., code 0)` with no output, have the mocked agent return visible missing-context text instead of `HEARTBEAT_OK`, and current delivery logic would not treat it as skippable.

## Solution Assessment

No. The current prompt-based suppression is not the narrowest reliable fix because it depends on model compliance; the safer fix is to consume successful metadata-only completions before any agent-visible relay turn, while leaving failed or output-bearing completions relayable.
