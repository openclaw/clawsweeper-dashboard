## Summary

Keep open: current main already covers the model-picker part, but the no-argument `/model` display still prints raw current and available model ids while catalog entries carry providers. This PR remains a plausible fix path, but it needs refresh, real behavior proof, and a small default-provider correction before merge.

## What This Changes

The PR qualifies Control UI model picker values and `/model` slash-command current/available output with provider/model refs, plus focused regression tests.

## Best Possible Solution

Refresh and narrow the patch to provider-qualified `/model` status output through the existing chat-model-ref helpers, with coverage for session, inherited-default, duplicate-id, and provider-native slash-id cases.

## Reproduction Assessment

Yes. Source inspection shows current main renders no-argument `/model` output from raw `session.model` or defaults and bare `m.id` values even though catalog entries carry providers.

## Solution Assessment

No, not as submitted. Qualifying `/model` output is the right direction, but the branch omits `sessions.defaults.modelProvider` when the displayed model is inherited and also needs refresh plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the defaults provider for inherited model display:** `ui/src/ui/chat/slash-command-executor.ts:193-194`
  - body: When there is no session model, `rawModel` falls back to `sessions.defaults.model` but this line still passes only `session?.modelProvider`. That makes `/model` print a bare inherited default even when `sessions.defaults.modelProvider` is available, so the displayed value may not round-trip. Track whether the model came from the session or defaults and pass the defaults provider only for the defaults branch.
