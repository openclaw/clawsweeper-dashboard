## Summary

Keep open. Current main already preserves provider-qualified values through the Control UI model picker, but it still prints bare values for the no-argument `/model` slash-command output, which is the remaining useful behavior in this PR and is also tracked by open issue #46935. The PR head is currently dirty/unmergeable against main, so it needs maintainer rebase or a narrow cherry-pick rather than cleanup closure.

## What This Changes

This PR qualifies Control UI model picker and `/model` slash-command model refs with provider metadata and adds regression coverage for non-Anthropic providers, slash-qualified ids, and legacy no-provider fallbacks.

## Best Possible Solution

Keep the shipped picker implementation on main and land a rebased, minimal slash-command fix that formats the current and available `/model` values as canonical provider-qualified refs where provider metadata is available, with regression tests covering provider-qualified output and legacy no-provider fallback behavior.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: the no-argument `/model` path maps available models to `m.id`, and the existing test expects bare output; #46935 provides a concrete user-facing reproduction with `/model claude-opus-4-6` being routed to the wrong provider.

## Solution Assessment

Mostly yes. The PR’s remaining slash-command helper usage is the narrow maintainable direction, but the whole branch is stale and overlaps with picker behavior already on main, so the best maintainer path is to rebase or cherry-pick the slash-command change and tests.
