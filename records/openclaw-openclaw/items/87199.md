## Summary

Keep open: current main and the latest release still generate a wrapper-first macOS LaunchAgent when service env entries exist, and an open closing PR already owns the narrow shell-first fix path.

## What This Changes

The issue reports that macOS LaunchAgents generated for external-home setups spawn the env-wrapper directly, causing launchd Permission denied or EX_CONFIG exits before the gateway starts.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/89967, or an equivalent shell-first LaunchAgent serialization/readback fix, then close this issue after merge and release tracking.

## Reproduction Assessment

Yes. Current main source still emits wrapper-first ProgramArguments when service env exists, and v2026.6.9 has the same shape; I did not run a fresh live external-home macOS launchd reproduction in this read-only review.

## Solution Assessment

Yes. The best fix path is the linked shell-first LaunchAgent PR, which keeps env values in the owner-only env file while making launchd spawn a stable shell executable; changing only status parsing or forcing a literal gateway run token would not fix the raw spawn failure.
