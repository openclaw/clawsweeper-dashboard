## Summary

Keep open: current main and latest release still generate wrapper-first macOS LaunchAgent ProgramArguments when service env entries exist, while the linked open PR with closing syntax already owns the fix path and includes macOS launchd proof.

## What This Changes

Fix macOS LaunchAgent generation so service-env installs do not ask launchd to spawn the generated env-wrapper directly and keep gateway startup working on external-home volumes.

## Best Possible Solution

Review and land or intentionally reject https://github.com/openclaw/openclaw/pull/89967, preserving owner-only env-file loading while making launchd spawn a stable shell before the wrapper.

## Reproduction Assessment

Yes for the source-level failure surface: current main and v2026.6.8 still serialize the generated env wrapper as ProgramArguments[0] when service env entries exist. No exact external-home macOS run was performed in this checkout.

## Solution Assessment

Yes: the narrow fix is to make launchd start a stable executable such as /bin/sh before the generated env wrapper while preserving owner-only env-file loading. Changing only status parsing or forcing a literal gateway run token would not address the raw launchd spawn failure.
