## Summary

Keep open: current main and v2026.6.8 still generate wrapper-first macOS LaunchAgent ProgramArguments when service env entries exist, and https://github.com/openclaw/openclaw/pull/89967 is the open linked fix candidate.

## What This Changes

The issue requests fixing macOS LaunchAgent generation so service-env installs do not launch the generated wrapper directly and fail under launchd on external-home macOS setups.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/89967, preserving owner-only env-file loading while making launchd start a stable shell before the generated wrapper.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still serialize the generated env wrapper as ProgramArguments[0] when service env entries exist. I did not run the exact external-home macOS setup in this read-only review.

## Solution Assessment

Yes: the narrow fix is to make launchd start a stable shell before the generated env wrapper while preserving owner-only env-file loading. Changing only status parsing or forcing a literal gateway run token would not address the raw launchd spawn failure.
