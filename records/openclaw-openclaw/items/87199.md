## Summary

Keep open: current main and the latest release still generate wrapper-first macOS LaunchAgent ProgramArguments when service env entries exist, and https://github.com/openclaw/openclaw/pull/89967 is the open closing PR that owns the fix path.

## What This Changes

The issue reports a macOS LaunchAgent gateway startup crash where generated service-env wrapper arguments make launchd spawn the wrapper directly on external-home setups.

## Best Possible Solution

Review and land or intentionally reject https://github.com/openclaw/openclaw/pull/89967, while keeping the external APFS plist-location work tracked separately.

## Reproduction Assessment

Yes for the source-level failure surface: current main and v2026.6.8 still serialize the generated env wrapper as ProgramArguments[0] when service env entries exist. I did not run the exact external-home macOS setup in this read-only checkout.

## Solution Assessment

Yes: the narrow fix is to make launchd start a stable executable such as /bin/sh before the generated env wrapper while preserving owner-only env-file loading. Changing only status parsing or forcing a literal gateway run token would not address the raw launchd spawn failure.
