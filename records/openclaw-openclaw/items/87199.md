## Summary

Keep open: current `main` still generates a macOS LaunchAgent that executes the generated environment wrapper directly before the gateway runtime, matching the central reported service-generation problem. The bare `gateway --port` part is less clearly a runtime bug because current CLI code intentionally treats bare `gateway` as the run path, but the wrapper-as-executable launchd surface remains actionable.

## What This Changes

The issue reports that the macOS gateway LaunchAgent generated by OpenClaw 2026.5.22 fails under launchd on an external-home setup because it executes the generated env-wrapper directly and uses the bare gateway command shape.

## Best Possible Solution

Update macOS LaunchAgent generation so launchd starts a stable system executable or direct runtime path while still loading the generated owner-only service environment file and avoiding secrets in the plist.

## Reproduction Assessment

Yes for the generated service command shape: current source still prepends the env-wrapper as the first launchd argument and generates `gateway --port`. No for the full external-home launchd failure, which was not live-reproduced in this checkout.

## Solution Assessment

Unclear whether the reporter's hand-edited shell command is the exact best permanent shape, but closing is not appropriate. The maintainable fix should stay in the launchd generator and preserve the existing service-env privacy boundary.
