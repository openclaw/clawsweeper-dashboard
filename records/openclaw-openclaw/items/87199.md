Update macOS LaunchAgent generation so launchd starts a stable system executable or runtime path instead of the generated home-volume wrapper, while preserving owner-only env-file loading and adding focused tests plus real macOS external-home proof before landing.

## Repair Work Prompt

Fix the macOS gateway LaunchAgent env-wrapper startup bug reported in https://github.com/openclaw/openclaw/issues/87199. Current main and v2026.5.22 make `prepareLaunchAgentProgramArguments` prepend `ai.openclaw.gateway-env-wrapper.sh` when service env entries exist, so launchd executes the generated home-volume wrapper directly; on external-home setups this can fail with Permission denied and leave the gateway stopped. Change the macOS LaunchAgent generation so `ProgramArguments[0]` is a stable system executable or runtime path, not the generated wrapper, while still loading the owner-only env file and never embedding API keys or managed env values directly in the plist. Keep the fix scoped to launchd service generation/reading/tests; do not add a global LaunchDaemon mode or new config option. The bare `gateway --port` form is currently supported by the CLI parent action, so only change it to `gateway run` if the generated service command and status/readiness tests are updated deliberately without removing bare compatibility. Add focused regression coverage in the launchd plist/generation tests and update docs only if the operator-visible command shape changes. Before landing, request real macOS launchd proof on an external-home or equivalent permission-constrained setup.
