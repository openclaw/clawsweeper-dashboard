## Summary

Keep open: current main and the latest release still serialize macOS service-env LaunchAgents with the generated env-wrapper as the executable, and the open closing PR remains the active fix candidate rather than merged behavior.

## What This Changes

The issue reports that macOS LaunchAgents generated with service env wrappers can fail under launchd on external-home setups because the raw plist starts at the generated env-wrapper and the gateway does not come up.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/89967 or an equivalent shell-first LaunchAgent serialization/readback fix, while keeping the external-APFS plist-location work in its separate canonical path.

## Reproduction Assessment

Yes. Source inspection on current main shows service-env LaunchAgents still put the generated wrapper path at ProgramArguments[0], and the existing test asserts the raw plist contains that wrapper while readback unwraps it.

## Solution Assessment

Yes. The best fix is to change the LaunchAgent env-wrapper serialization/readback boundary so launchd starts a stable shell executable; changing status parsing or the supported gateway --port CLI command would not fix the raw spawn failure.
