## Summary

Keep open: current main and v2026.6.6 still generate wrapper-first macOS LaunchAgent ProgramArguments when service env entries exist, and an open mergeable PR with closing syntax already owns the fix path.

## What This Changes

The issue reports that macOS LaunchAgent generation can spawn the generated env-wrapper directly and fail under launchd on external-home setups.

## Best Possible Solution

Review and land or intentionally close https://github.com/openclaw/openclaw/pull/89967, preserving owner-only env-file loading and leaving the supported gateway command shape unchanged.

## Reproduction Assessment

Yes for the source-level failure surface: current main and v2026.6.6 still generate wrapper-first LaunchAgent ProgramArguments when service env entries exist. No live external-home launchd run was performed in this Linux checkout.

## Solution Assessment

Yes: the narrow fix is to make launchd start a stable executable such as `/bin/sh` before the generated env wrapper while preserving owner-only env-file loading. Changing only status parsing or forcing `gateway run` would not address the raw launchd spawn failure.
