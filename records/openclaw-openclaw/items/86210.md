## Summary

Keep open; the latest branch appears to address the earlier unsupported per-agent `contextEngine` finding, but it still exposes per-agent recall selection while the memory runtime/capability path remains process-global, so agents can be routed to the wrong recall backend.

## What This Changes

The PR adds purpose-specific memory role slots, legacy memory-slot migration, per-agent memory slot overrides, and wires those through plugin activation, config validation, runtime, diagnostics, docs, and tests.

## Best Possible Solution

Land the memory role contract only after maintainer sign-off and either implement plugin-id/agent-keyed recall runtime dispatch or keep per-agent recall overrides out of this PR until that dispatch exists.

## Reproduction Assessment

Yes for the review finding: source inspection shows per-agent `memory.recall` can resolve to a different plugin while `ensureMemoryRuntime` returns the first singleton memory runtime already registered. The broader PR is a feature/API change, not a bug reproduction.

## Solution Assessment

No, not as written: the role-slot direction is plausible, but per-agent recall must not validate and activate without a runtime dispatcher that can choose the matching recall provider per agent. The safer alternatives are keyed runtime dispatch or removing per-agent recall from this PR.
