## Summary

Keep open: the Discord ambiguity being fixed is real, and the branch is now narrow, but the current implementation still applies a Discord-shaped `channel:` prefix to every non-prefixed legacy `channelId` fallback and can break existing non-Discord channel targets. The external-PR real behavior proof is also still insufficient because the PR body only shows pre-fix logs and source/unit reasoning for this branch.

## What This Changes

The branch changes message action normalization so legacy `channelId` fallback values for to-mode actions become `target`/`to` values with a `channel:` prefix unless already prefixed, and adds table tests for Discord and channel-info cases.

## Best Possible Solution

Land a channel-aware normalization fix that preserves legacy `channelId` semantics for non-Discord providers while disambiguating bare Discord numeric channel IDs, with regression coverage and redacted after-fix Discord proof.

## Reproduction Assessment

Yes, at source level: current main loses the `channelId` channel-kind hint when migrating to `to`, and Discord bare numeric targets require a default kind to avoid the ambiguity error. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No, not as currently implemented: the Discord bare-ID fix is the right direction, but the branch should scope the prefix to Discord bare numerics or preserve a channel-kind hint without rewriting other providers.
