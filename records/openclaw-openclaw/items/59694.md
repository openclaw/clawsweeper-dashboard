## Summary

Keep open for maintainer review: current main and v2026.6.10 still read raw config path segments and inherited defaults, while the PR head uses the shared blocked-key guard with focused tests and sufficient live-output proof.

## What This Changes

The branch imports the shared blocked-key guard into `config-eval`, returns `undefined` for blocked dotted path segments, hardens defaults fallback to own non-blocked keys, and adds regression tests.

## Best Possible Solution

Land the narrow shared-helper hardening after maintainer review and exact-head merge gates, since current main and the latest release still lack the guard.

## Reproduction Assessment

Yes. Source inspection shows current main indexes raw dotted path segments and uses `pathStr in defaults`, and the contributor supplied Node v22.22.1 output comparing main vs PR behavior for blocked paths and defaults fallback.

## Solution Assessment

Yes. Reusing `isBlockedObjectKey` in the shared helper is the right boundary because hook and skill eligibility both call `isConfigPathTruthyWithDefaults`, and sibling config writers already use the same blocked-key policy.
