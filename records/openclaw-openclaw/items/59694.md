## Summary

This PR remains necessary: current main can resolve inherited prototype members as truthy skill or hook configuration requirements, while the proposed guard consistently rejects the same dangerous path segments already blocked elsewhere in configuration handling. The patch is narrow, correct, and supported by convincing direct runtime proof.

## What This Changes

The PR makes `resolveConfigPath` return `undefined` when a dotted configuration path contains `__proto__`, `prototype`, or `constructor`.

## Best Possible Solution

Land the shared blocked-key check at the raw traversal boundary; focused regression coverage for the three blocked segments would be useful but is not required to establish patch correctness.

## Reproduction Assessment

Yes. Current source and the contributor’s direct Node comparison establish that inherited `constructor` and `__proto__` values resolve on main and become `undefined` after the patch.

## Solution Assessment

Yes. Reusing the repository’s canonical blocked-key utility at the raw traversal boundary is the narrowest solution and stays consistent with existing configuration path handling.
