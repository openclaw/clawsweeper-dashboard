## Summary

Keep open: the PR adds a useful direct guard, but it is not merge-ready because the shared defaults wrapper used by skills and hooks can still satisfy unsafe prototype-key config paths through inherited defaults.

## What This Changes

The PR imports the shared prototype-key guard into `src/shared/config-eval.ts`, returns `undefined` for blocked path segments in `resolveConfigPath`, and adds direct resolver regression coverage.

## Best Possible Solution

Guard unsafe path segments at the shared config-requirement wrapper boundary as well as the raw resolver, with regression coverage for both direct resolution and the skills/hooks defaults path.

## Reproduction Assessment

Yes. Source inspection and a read-only Node probe show current `resolveConfigPath` can reach prototype-chain values, and the patched direct guard still leaves `isConfigPathTruthyWithDefaults` able to return inherited defaults for blocked paths.

## Solution Assessment

No. The direct resolver guard is the right direction, but the best fix must also close the wrapper/defaults fallback that production skills and hooks call.

- **[P1] Block fallback defaults for unsafe config paths:** `src/shared/config-eval.ts:31-32`
  - body: Returning `undefined` here is not enough for the production path: `isConfigPathTruthyWithDefaults` treats that as unresolved and then checks `pathStr in defaults`.
