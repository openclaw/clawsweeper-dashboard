## Summary

Keep open. Current main still traverses raw config path segments in `resolveConfigPath` without blocking prototype keys, and the PR is a narrow hardening patch that matches established guard patterns elsewhere in config handling. The remaining issue is merge readiness: this security-facing fix should get the required changelog entry, and maintainer security review should decide whether to ask for focused regression coverage before landing.

## What This Changes

This PR imports `isBlockedObjectKey` and makes `resolveConfigPath` return `undefined` when a dot-path segment is `__proto__`, `prototype`, or `constructor`.

## Best Possible Solution

Land the same inline blocked-key check in `resolveConfigPath`, record it under `CHANGELOG.md` Unreleased Fixes with contributor attribution, and add focused `config-eval` regression coverage if maintainers want test proof before merge.

## Reproduction Assessment

Yes. A focused check can call `resolveConfigPath` with `__proto__`, `prototype`, or `constructor` path segments; current main reaches raw property access, while the PR would return `undefined` before traversal.

## Solution Assessment

Yes. Because `resolveConfigPath` accepts raw string paths directly, the local guard is the narrowest maintainable fix; callers that use `getConfigValueAtPath` are already protected by `parseConfigPath`.
