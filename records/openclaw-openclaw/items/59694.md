## Summary

Keep open: current main and the latest release still use raw config-path traversal, and the PR matches the existing blocked-key contract, but this external PR still needs real behavior proof plus refreshed regression/CI validation before merge.

## What This Changes

The PR adds an `isBlockedObjectKey` check to `resolveConfigPath` so blocked prototype-key path segments return `undefined` before property access.

## Best Possible Solution

Merge the guard with focused regression assertions once the contributor supplies redacted real behavior proof and the relevant checks are refreshed.

## Reproduction Assessment

Yes, via source inspection rather than an executed test. Current main splits arbitrary path strings and indexes `current[part]`, so a segment such as `constructor` can traverse inherited prototype state instead of being rejected.

## Solution Assessment

Yes for the runtime code. Because `resolveConfigPath` accepts raw dot-separated strings, checking each segment with the existing blocked-key helper before property access is the narrowest maintainable fix; proof and regression coverage remain to be added.
