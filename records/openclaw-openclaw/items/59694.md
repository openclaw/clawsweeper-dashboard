## Summary

This PR identifies a real prototype-chain read in the shared config resolver, but the resolver-only guard is not merge-ready because the adjacent production defaults fallback still accepts inherited `constructor`; current `main` has no equivalent fix, so the PR remains necessary but needs a narrow repair.

## What This Changes

The PR imports the shared blocked-object-key utility and makes `resolveConfigPath` return `undefined` when a dotted path contains `__proto__`, `prototype`, or `constructor`.

## Best Possible Solution

Apply blocked-path rejection to the complete resolver/defaults decision, consult defaults only through safe own properties, and add wrapper-level regressions for all three blocked keys while preserving normal dotted paths and explicit defaults.

## Reproduction Assessment

Yes, from current source with high confidence: after the proposed resolver returns `undefined` for `constructor`, `pathStr in defaults` succeeds on a normal object and the wrapper reads the inherited constructor value.

## Solution Assessment

No. The direct traversal guard is appropriate, but the narrow maintainable fix must protect the adjacent defaults fallback that interprets the same untrusted path.
