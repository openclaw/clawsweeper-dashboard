## Summary

The central hardening is still needed on current main, but the PR protects only the direct resolver; the production truthiness wrapper can still accept blocked names through inherited properties on its defaults object. One focused repair and regression test should make this landable.

## What This Changes

The PR adds `isBlockedObjectKey` checks to `resolveConfigPath` so dotted config lookups return `undefined` for `__proto__`, `prototype`, and `constructor` segments.

## Best Possible Solution

Reject blocked segments across the complete `isConfigPathTruthyWithDefaults` boundary—using the shared guard and own-property fallback semantics—and add focused tests proving blocked paths are false while normal explicit defaults remain unchanged.

## Reproduction Assessment

Yes. On current main, a plain object exposes inherited `constructor` and `__proto__` values through `resolveConfigPath`, and after this patch the wrapper still reaches inherited `constructor` through `pathStr in defaults`.

## Solution Assessment

No, not as submitted. The resolver guard is the right primitive, but the narrowest complete fix must also prevent the wrapper’s inherited defaults lookup from re-enabling blocked path names.
