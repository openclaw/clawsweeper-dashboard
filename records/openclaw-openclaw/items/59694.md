## Summary

Keep open. Current main still reads raw `resolveConfigPath` segments without a blocked-key check, and this PR’s one-file guard matches the existing prototype-key utility pattern; the remaining cleanup is changelog/test coverage before merge.

## What This Changes

The PR imports `isBlockedObjectKey` and makes `resolveConfigPath` return `undefined` for `__proto__`, `prototype`, or `constructor` path segments.

## Best Possible Solution

Land the inline blocked-key check with a single-line `CHANGELOG.md` Fixes entry and focused `config-eval` regression assertions for the three blocked segments.

## Reproduction Assessment

Yes. A focused call such as `resolveConfigPath({}, "constructor")` reaches inherited property access on current main, while this PR returns `undefined` before traversal.

## Solution Assessment

Yes for the runtime code. Because `resolveConfigPath` accepts raw string paths, checking each segment with the existing helper before property access is the narrowest maintainable fix; changelog and test coverage should be added before merge.
