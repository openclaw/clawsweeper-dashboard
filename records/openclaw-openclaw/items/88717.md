## Label Justifications

- `P2`: The report describes a real but bounded status/session-state correctness bug in Codex-backed sessions rather than a runtime outage.
- `impact:session-state`: The affected behavior is persisted session compaction/token state drifting from the native Codex thread state shown to users.

## Summary

Keep open: current main still treats Codex native `/compact` as a started-but-not-completed OpenClaw compaction, and status is still driven by session-store `compactionCount` and token freshness rather than a bridged native completion signal.

## What This Changes

The issue asks OpenClaw to make Codex native `/compact` completion and token usage visible in `/status` and session state instead of showing `Compactions: 0` with stale counts.

## Best Possible Solution

Add a native Codex compaction status and usage bridge that keeps OpenClaw's completed-compaction count honest, marks token freshness correctly, and shows native pending/completed/unknown state in status.

## Reproduction Assessment

Yes, from source inspection: current main starts native Codex compaction, returns pending without a completion watcher, and `/status` reads persisted `compactionCount`/token fields. I did not run the live Telegram/Codex flow in this read-only review.

## Solution Assessment

Unclear until maintainers choose the status contract: counting native completion as an OpenClaw-completed compaction would conflict with current docs, while native-specific status plus stale-token handling looks safer.
