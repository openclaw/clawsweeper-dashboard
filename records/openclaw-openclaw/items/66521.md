## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested early Signal group typing cue, and the latest PR revision fixes the original DM scope issue by using `entry.isGroup`, but the new callback still fires before dispatch resolves `sendPolicy` and typing suppression.

## What This Changes

The PR adds a Signal group-only `typingCallbacks.onReplyStart()` call before inbound dispatch so group chats show typing feedback during pre-dispatch work.

## Best Possible Solution

Keep the UX improvement, but start early Signal group typing only after the same source-reply and typing policy used by dispatch has been resolved. Add focused Signal tests for allowed group feedback plus denied or typing-disabled sessions emitting no typing signal.

## Reproduction Assessment

Yes. A code-level reproduction is to apply the PR and process a Signal group turn whose session has `sendPolicy: "deny"` or typing disabled; the added pre-dispatch `onReplyStart` fires before the dispatch path can pass `suppressTyping: true`.

## Solution Assessment

No. The group-only guard and callback reuse are good pieces, but the maintainable fix needs to share or defer to the resolved dispatch typing policy instead of starting a channel side effect before policy resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Defer early typing until dispatch policy is known:** `extensions/signal/src/monitor/event-handler.ts:329`
  - body: This starts Signal typing before `dispatchInboundMessage` resolves session `sendPolicy` and typing suppression. A group session with `sendPolicy: "deny"` is supposed to suppress outbound indicators, but this early callback has already called the channel typing path, leaking presence despite the policy. Move the early start behind the same resolved typing policy, or add an equivalent guard before firing it.
  - confidence: 0.9
