## Summary

Keep open. The revised PR is closer to the right shape because it is group-only and routes through `typingCallbacks`, but it still starts Signal typing before dispatch resolves the existing source-reply and typing-suppression policy, and it lacks attached real behavior proof.

## What This Changes

The PR adds a group-only pre-dispatch Signal typing start via `typingCallbacks?.onReplyStart()` before `dispatchInboundMessage`.

## Best Possible Solution

Start early Signal group typing only after resolving the same source-reply visibility and typing-suppression policy used by normal reply typing, add focused regression coverage, and attach redacted live Signal proof.

Yes for the blocking review finding: source inspection shows a Signal group turn with `sendPolicy: "deny"` or suppressed typing can reach the proposed pre-dispatch `typingCallbacks?.onReplyStart()` before dispatch computes `sourceReplyPolicy.suppressTyping`. The claimed user-visible timing improvement still lacks attached live proof.

## Solution Assessment

No. The group-only `typingCallbacks` route is an improvement over the original direct send, but the early start must be behind the existing suppression policy before it is merge-safe.
