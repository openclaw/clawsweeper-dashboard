## Summary

Keep open. The `sessions_send` selector conflict is fixed on current main and in `v2026.6.6`, but the `message(action="send")` poll-field conflict remains source-reproducible and overlaps active message-tool schema/validation issues that still need a maintainer contract decision.

## What This Changes

The issue reports that schema-visible optional parameters cause `sessions_send` and `message(action="send")` calls to fail when models include conflicting selector or poll fields.

## Best Possible Solution

Keep the shipped `sessions_send` precedence behavior, then land one maintainer-approved shared message-tool fix that makes plain sends robust against schema-padded poll/component fields while preserving strict validation for explicit poll and advanced channel actions.

## Reproduction Assessment

Yes, for the remaining message-tool half at source level: current main still throws for `action="send"` when poll content or channel-extra poll params make `hasPollCreationParams()` return true. The `sessions_send` half no longer reproduces on current main or `v2026.6.6`.

## Solution Assessment

No, the current implementation is only a partial fix. The best path is a maintainer-approved message-tool contract that scopes or normalizes poll/component fields for plain sends without weakening deliberate poll actions.
