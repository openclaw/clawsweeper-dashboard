## Summary

Keep this open. Current `main` still contains both reported rejection paths: `sessions_send` rejects calls that include both `sessionKey` and `label`, and `message` still exposes poll fields while `runMessageAction()` rejects `action: "send"` calls that include poll-creation parameters. The latest release tag inspected, `v2026.4.29`, contains the same behavior, while the related implementation candidates are still not landed as a complete fix.

## What This Changes

The issue asks OpenClaw to prevent LLMs from tripping optional-parameter conflicts in `sessions_send` (`sessionKey` plus `label`) and `message` (`send` plus poll fields).

## Best Possible Solution

Review and land or replace the two narrow fixes: make `sessions_send` treat an authoritative `sessionKey`/session id as precedence over redundant label fields, and make `message` avoid failing `send` calls merely because stray poll-creation fields were supplied. Close only after both halves are on `main`, or after maintainers split any remaining half into a canonical follow-up.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `sessions_send` with both `sessionKey` and non-empty `label`, or call `message` with `action: "send"` plus non-empty poll-creation params such as `pollQuestion`/`pollOption`; current code reaches the reported errors, and the send-validation test asserts the second path.

## Solution Assessment

Yes for keeping this open; current `main` is not the fix. The narrowest maintainable path is to review the existing focused implementation candidates, keeping the two behavior changes separate unless maintainers choose a combined PR.
