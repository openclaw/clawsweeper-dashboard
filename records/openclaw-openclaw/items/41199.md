## Summary

Keep open. Both reported rejection paths are still present on current main and in the latest release, and the related fixes are still open, conflicting, partial, or awaiting maintainer consolidation.

## What This Changes

The issue reports that model-filled optional parameters make `sessions_send` reject mixed `sessionKey`/`label` calls and make `message` reject `send` calls that include poll fields.

## Best Possible Solution

Land or replace the linked fixes with one consolidated contract: explicit session identifiers take safe precedence without denied-key disclosure, and message send normalization tolerates model-filled no-op poll params while preserving real poll validation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `sessions_send` with both `sessionKey` and `label`, or call `message` with `action: "send"` plus non-empty poll creation params, and current main still takes the reported rejection branches.

## Solution Assessment

Unclear as a complete fix today. Preferring explicit session identifiers is likely the narrow sessions fix, but the message send/poll path still needs maintainers to choose and land one safe normalization contract across the open PRs.
