## Summary

Keep open. Current main and v2026.6.6 fixed the `sessions_send` selector conflict, but `message(action="send")` still rejects content-bearing or channel-specific poll params before delivery, and the only direct poll-stripping PR was closed unmerged.

## What This Changes

The issue reports that LLM-generated tool calls add conflicting optional fields to `sessions_send` and `message(action="send")`, causing inter-agent or outbound message delivery failures.

## Best Possible Solution

Keep the shipped `sessions_send` precedence behavior, then choose one message-tool contract that lets plain sends survive model-filled optional poll fields while preserving real poll validation, with focused regression tests for both cases.

## Reproduction Assessment

Yes, at source level. `runMessageAction()` still throws before dispatch for `action="send"` plus detected poll-creation params, and current tests assert that rejection path; I did not run live or local tests in this read-only review.

## Solution Assessment

No, closing this issue is not the best outcome yet. The sessions half is solved, but the message send/poll schema and normalization contract still needs one maintainer-approved direction rather than another stale duplicate PR.
