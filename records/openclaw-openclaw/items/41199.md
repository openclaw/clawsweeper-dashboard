## Summary

Keep open. The `sessions_send` selector-conflict half is fixed on current main and in `v2026.6.6`, but current source and tests still preserve the `message(action="send")` poll-parameter rejection path for content-bearing or channel-specific poll fields, with related open work still needing maintainer direction.

## What This Changes

The issue reports that model-filled optional parameters make `sessions_send` and `message` reject otherwise intended agent-to-agent or channel messages.

## Best Possible Solution

Keep the shipped `sessions_send` precedence fix, then land a maintainer-approved message-tool contract that tolerates model-filled send noise without breaking real poll creation, with focused regression and real delivery proof.

## Reproduction Assessment

Yes, at source level. The sessions selector conflict no longer reproduces in current main or `v2026.6.6`, but `runMessageAction()` still throws for `action="send"` with detected poll-creation params and tests assert that behavior.

## Solution Assessment

Unclear for the remaining half. The sessions fix is the right narrow repair, but the message poll path needs a maintainer-approved contract before replacing the current fail-closed validation.
