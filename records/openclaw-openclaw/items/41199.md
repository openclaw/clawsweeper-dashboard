## Label Justifications

- `P1`: The issue blocks real agent-to-agent and outbound message workflows when models include conflicting optional tool parameters.
- `impact:message-loss`: Both reported failures prevent intended inter-agent or channel messages from being delivered.

## Summary

Keep open: current main and the latest shipped tag still contain both reported rejection paths, while the active linked fixes are partial or waiting on maintainer consolidation/proof.

## What This Changes

The issue reports that model-generated calls to `sessions_send` and `message` include conflicting optional selector or poll parameters, causing agent-to-agent or outbound sends to fail.

## Best Possible Solution

Land or replace the linked fixes with one consolidated contract: explicit session identifiers take precedence without unauthorized key disclosure, and send actions tolerate model-filled no-op/default poll params while preserving intentional poll validation.

## Reproduction Assessment

Yes, at source level. Current main still rejects `sessions_send` calls with both `sessionKey` and `label`, and `runMessageAction()` still throws before dispatch for `action: "send"` plus detected poll-creation params.

## Solution Assessment

Unclear as a complete fix today. Preferring explicit session identifiers is the narrow sessions fix, but the send/poll normalization needs one maintainer-approved contract across the overlapping PRs.
