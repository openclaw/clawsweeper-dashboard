## Summary

Keep open. Current main and v2026.6.6 fix the `sessions_send` selector conflict, but the `message(action="send")` poll-param rejection path remains source-reproducible and is still a maintainer contract decision shared with related message-tool schema issues.

## What This Changes

The issue reports that agent-to-agent and shared message tools fail when models add conflicting optional parameters to `sessions_send` and `message(action="send")` calls.

## Best Possible Solution

Keep the shipped `sessions_send` precedence behavior, then consolidate the remaining shared `message` tool fix so plain sends survive schema-padded poll/component fields while explicit poll and advanced channel actions remain validated.

## Reproduction Assessment

Yes, at source level. Current main still throws in `runMessageAction()` for `action="send"` plus detected poll-creation params, and focused tests still assert that rejection path.

## Solution Assessment

No, closing as fully implemented is not the best outcome yet. The sessions half is fixed, but the remaining message-tool schema/normalization contract needs a maintainer-approved direction shared with the related open message-tool issues.
