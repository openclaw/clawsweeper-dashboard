## Summary

Current main and the latest release already provide the requested provider-plugin onboarding path: installed manifests can expose provider ids, API-key choices, model catalog rows, picker entries, and ClawHub publishing/install guidance without adding AIMLAPI as a bundled core provider.

## What This Changes

The issue asks OpenClaw to define and document how third-party model provider plugins installed through ClawHub can register provider metadata and appear in normal onboarding/provider selection.

## Best Possible Solution

Keep the existing plugin-first path: provider authors should publish/install through ClawHub and use the documented manifest, auth-choice, catalog, and model-picker SDK surfaces instead of adding another bundled core provider.

## Reproduction Assessment

Not applicable: this is a feature/contract clarification request, not a bug report. Source, docs, tests, and release-tag inspection show the requested provider-plugin onboarding path already exists.

## Solution Assessment

Yes. The current solution is the maintainable boundary: manifests expose cheap onboarding/auth metadata, provider runtime owns vendor behavior, model catalogs feed picker/list surfaces, and ClawHub remains the distribution surface.
