## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Forward inherited tool policy to delegate spawns:** `src/agents/tools/sessions-delegate-tool.ts:160-171`
  - body: `sessions_spawn` forwards `inheritedToolAllowlist` and `inheritedToolDenylist` into `spawnSubagentDirect`, where they are written onto the child session. The new delegate tools omit those fields, so a parent running under a restricted tool policy can delegate to a child that no longer carries the same allow/deny restrictions.
  - confidence: 0.88
- **[P1] Do not bypass active-child limits in batch fan-out:** `src/agents/tools/sessions-delegate-tool.ts:284-288`
  - body: This loop starts all delegate promises immediately. `spawnSubagentDirect` checks `countActiveRunsForSession` before any of these children are registered, so a batch can have every child pass the same stale active count and exceed `maxChildrenPerAgent`; spawn sequentially or reserve capacity before launching children.
  - confidence: 0.84
- **[P1] Gate only delegate-owned cleanup:** `src/agents/subagent-registry-run-manager.ts:677-682`
  - body: Registering an output-capture gate for every `expectsCompletionMessage === false` run also catches existing non-delegate paths such as direct thread-bound `sessions_spawn(mode="session")` delivery. Those runs have no delegate tool to call `signalOutputCaptured`, so cleanup/finalization waits for the 30s fallback before continuing.
  - confidence: 0.9
- **[P2] Return durable state for timed-out batch children:** `src/agents/tools/sessions-delegate-tool.ts:301-318`
  - body: When the overall batch timeout wins, unresolved delegate promises are replaced with synthetic timeout results after 500 ms while the original child promises keep running. The parent receives no `runId` or `childSessionKey` for those children, so it cannot inspect or clean up still-running delegated work deterministically.
  - confidence: 0.78
