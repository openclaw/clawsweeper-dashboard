## Summary

Keep open. Current main still lacks the requested config/schema/runtime deny-path gate, and PR #75101 explicitly targets this issue but remains open and unmerged, so the issue should stay open until that implementation or a narrow replacement lands.

## What This Changes

The issue requests an operator-configurable `tools.exec.denyPathPatterns` exec policy that blocks commands whose path-like arguments match sensitive path globs before execution.

## Best Possible Solution

Review and land PR #75101 or a narrow replacement that adds a shared pre-approval hard-deny gate for literal path-like argv and shell-wrapper payload tokens, updates config schema/types/help/docs, covers global plus per-agent behavior, and documents known bypass limits.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` config key and no runtime deny-path phase in either the node-host or gateway exec policy paths. The private production leak itself is not publicly reproducible from the available data.

## Solution Assessment

Unclear until PR #75101 is reviewed. A shared deny gate before allowlist/approval is the right maintenance boundary for the reported class, but the exact matching semantics and documented bypass scope need security/product review.
