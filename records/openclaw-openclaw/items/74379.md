## Summary

Keep open. Current main still has no accepted exec deny-path config or runtime deny-path gate, and the linked closing PR remains open and unmerged, so this security-sensitive request should stay open for maintainer review.

## What This Changes

Request adds an opt-in `tools.exec.denyPathPatterns` policy that blocks exec commands whose literal path-like arguments match sensitive file globs before secrets can enter transcripts or provider context.

## Best Possible Solution

Land PR #75101 or a narrow replacement that adds a shared hard-deny gate before allowlist/approval execution, wires the config type/schema/help/docs, covers global plus per-agent behavior, and documents v1 bypass limits such as indirection and symlinks.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` key and no runtime deny-path check in the node-host or gateway exec policy paths. The private production leak itself is not publicly reproducible from the available issue data.

## Solution Assessment

Unclear until PR #75101 is reviewed. A shared pre-exec deny gate is the right maintainable boundary for this class of leak, but the exact tokenization and matching behavior should be reviewed as security-sensitive.
