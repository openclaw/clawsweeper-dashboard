## Summary

Keep open. Current main still lacks the requested config-level exec path denylist, and #75101 is an open, unmerged closing PR, so this issue remains the correct tracker for maintainer security/config review.

## What This Changes

Request a config-level `tools.exec.denyPathPatterns` denylist that blocks exec commands whose path-like arguments match sensitive-file globs before command execution.

## Best Possible Solution

Review and land #75101 or a focused replacement that adds a shared documented hard-deny gate before exec approval/allowlist decisions, with schema/help/docs coverage, regression tests, and documented bypass limits.

## Reproduction Assessment

Yes for the missing control: current-main source inspection shows no accepted `tools.exec.denyPathPatterns` schema key and no pre-exec argument path-deny phase. The private token leak itself is not publicly reproducible from the available artifacts.

## Solution Assessment

Unclear until #75101 is reviewed. A config-backed hard-deny gate before allowlist/approval is the narrow maintainable boundary; optional warn or approval modes should wait for explicit maintainer security design.
