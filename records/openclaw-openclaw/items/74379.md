## Summary

Keep open. Current main still lacks the requested config/schema/runtime `tools.exec.denyPathPatterns` gate, and the active closing PR is still open and unmerged in the provided context, so this issue should remain the tracker for maintainer security review.

## What This Changes

Request a `tools.exec.denyPathPatterns` config/runtime denylist that blocks exec commands whose literal path-like arguments match sensitive glob patterns before allowlist or approval handling.

## Best Possible Solution

Review and land PR #75101 or a focused replacement that adds a shared pre-exec hard-deny gate for literal path-like argv and shell-wrapper payload tokens, updates config types/schema/help/docs, adds targeted runtime tests, documents bypass limits, and keeps this issue open until that fix merges.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` schema key and no pre-exec deny-path phase in the inspected runtime paths. The private production token leak itself is not publicly reproducible from the available artifacts.

## Solution Assessment

Yes for the direction: a config-backed hard-deny gate before exec allowlist/approval is the narrow maintainable boundary for the reported leak class. Optional warn or require-approval behavior should wait for explicit maintainer security design.
