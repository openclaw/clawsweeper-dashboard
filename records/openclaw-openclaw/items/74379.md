## Summary

Keep open. Current main still lacks the requested `tools.exec.denyPathPatterns` config/schema/runtime gate, and the open closing PR #75101 is still the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Request a config-level exec path denylist that blocks commands whose literal path-like arguments match sensitive file globs before execution.

## Best Possible Solution

Add a shared, default-off exec hard-deny layer that checks literal path-like argv and shell-wrapper payload tokens before allowlist/approval, with config types/schema/help/docs and focused runtime tests; keep broader action modes or deeper sandbox semantics as explicit follow-up design.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` schema key and no deny-path phase in the inspected exec policy paths. The reported private token leak itself is not publicly reproducible from the available artifacts.

## Solution Assessment

Unclear for the full proposed API, but the hard-block gate is the narrow maintainable direction for this class of leak. Optional `warn` or `require_approval` behavior should wait for maintainer security design rather than blocking the core defense-in-depth fix.
