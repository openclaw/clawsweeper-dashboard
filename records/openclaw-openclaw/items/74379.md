## Summary

Keep open. Current main still lacks the requested config/schema/runtime/docs deny-path gate, and the linked closing PR #75101 remains open and unmerged, so this issue should stay open until that implementation or a narrow replacement lands.

## What This Changes

Add a config-level `tools.exec.denyPathPatterns` denylist that blocks exec commands whose literal path-like arguments match sensitive file globs before secrets can enter transcripts or provider context.

## Best Possible Solution

Review and land PR #75101 or a narrow replacement that adds a shared hard-deny gate before allowlist/approval for literal path-like exec argv and shell-wrapper payload tokens, updates config schema/types/help/docs, adds focused runtime tests, and documents v1 bypass limits.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` key and no runtime deny-path phase in the inspected exec policy paths. The private production leak itself is not publicly reproducible from the available data.

## Solution Assessment

Unclear until the linked PR is reviewed. A shared pre-approval deny gate is the right maintainable boundary for this class of secret-exposure issue, but the exact glob/tokenization semantics and documented bypass scope need maintainer security review.
