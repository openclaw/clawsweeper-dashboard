## Summary

Keep open. Current main still lacks the requested `tools.exec.denyPathPatterns` config/schema/runtime gate, and the linked closing PR #75101 is open and unmerged, so this issue should remain the tracker until a reviewed fix lands.

## What This Changes

Request to add an opt-in config-level denylist that blocks exec commands when literal argv or shell-payload path tokens match sensitive path globs.

## Best Possible Solution

Review and land PR #75101 or a narrow replacement that adds a shared pre-approval hard-deny gate for literal exec path arguments and shell-wrapper payload tokens, updates config schema/help/docs, documents bypass limits, and includes focused runtime tests before closing this issue.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` key and no deny-path phase in the inspected exec policy paths. The private production leak itself is not publicly reproducible from the available data.

## Solution Assessment

Unclear until the linked implementation is reviewed. A config-backed hard-deny gate before allowlist/approval is the right maintainable boundary, while optional warn or approval modes should wait for explicit maintainer security design.
