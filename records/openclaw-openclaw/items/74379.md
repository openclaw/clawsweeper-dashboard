## Summary

Keep open. Current main still lacks the requested config/schema/runtime `tools.exec.denyPathPatterns` gate, docs only describe the narrower `safeBins` path-token restriction, and the linked closing PR #75101 is still open and unmerged in the provided context, so this issue should remain the tracker.

## What This Changes

Add an opt-in config-level exec path denylist that blocks commands whose literal path-like arguments match sensitive glob patterns before execution.

## Best Possible Solution

Review and land PR #75101 or a focused replacement that adds a shared pre-exec hard-deny gate for literal path-like argv and shell-wrapper payload tokens, updates config types/schema/help/docs, adds targeted runtime tests, documents bypass limits, and keeps this issue open until that fix merges.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` schema key and no runtime deny-path phase in the inspected exec policy paths. The reported private token leak itself is not publicly reproducible from the available data.

## Solution Assessment

Unclear until the linked implementation is reviewed. A config-backed hard-deny gate before allowlist/approval is the narrow maintainable boundary for this class; optional warn or require-approval behavior should wait for explicit maintainer security design.
