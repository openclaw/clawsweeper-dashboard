## Summary

Keep open. Current main still lacks the requested config/schema/runtime deny-path gate, and the open closing PR #75101 targets the same security-sensitive behavior but has not merged, so this issue should remain open until that PR or a narrow replacement lands.

## What This Changes

Request a config-level `tools.exec.denyPathPatterns` denylist that blocks exec commands when literal path-like arguments match sensitive-file glob patterns before execution.

## Best Possible Solution

Review and land PR #75101 or a narrow replacement that adds a shared pre-exec hard-deny gate for literal path-like argv and shell-wrapper payload tokens, updates config schema/types/help/docs, covers global plus per-agent behavior, and documents known bypass limits.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` config key and no runtime deny-path phase in the node-host or gateway exec policy paths. The private production leak itself is not publicly reproducible from the available data.

## Solution Assessment

Unclear until the open implementation is reviewed. A shared deny gate before allowlist/approval is the right maintenance boundary for the reported class, but the specific matching semantics and documented bypass scope need security/product review.
