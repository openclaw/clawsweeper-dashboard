## Summary

Keep open. Current main still lacks the requested config/schema/runtime deny-path gate, and the linked closing PR #75101 is still described in the provided GitHub context as open and unmerged, so this issue should remain the tracker until that implementation or a narrow replacement lands.

## What This Changes

Request an opt-in `tools.exec.denyPathPatterns` config-level denylist that blocks literal path-like exec arguments before command execution so sensitive files cannot enter transcripts or provider context.

## Best Possible Solution

Review and land PR #75101 or a narrow replacement that adds a shared pre-approval hard-deny gate for literal path-like exec argv and shell-wrapper payload tokens, updates config schema/types/help/docs, includes focused runtime tests, documents bypass limits, and keeps this issue open until the fix merges.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` config key and no runtime deny-path phase in the inspected exec policy paths. The private production token leak itself is not publicly reproducible from the available data.

## Solution Assessment

Unclear until the linked implementation is reviewed. A config-backed hard-deny gate before allowlist/approval is the right maintainable boundary for this class, while optional `warn` or `require_approval` modes should wait for explicit maintainer security design.
