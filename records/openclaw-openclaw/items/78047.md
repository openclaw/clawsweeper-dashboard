## Summary

Keep open: this draft, maintainer-labeled PR targets a real Nix-mode immutability gap. Current main persists plugin auto-enable changes during gateway startup even though Nix mode is documented as externally managed/read-only; the patch looks directionally correct, but it needs maintainer scope review and real behavior proof before merge.

## What This Changes

The PR makes gateway startup plugin auto-enable runtime-only when `OPENCLAW_NIX_MODE=1` and adds a focused regression test.

## Best Possible Solution

Land a maintainer-approved startup path that keeps Nix-managed config immutable while preserving effective runtime plugin activation; expand the scope beyond Nix only if maintainers choose that broader policy.

## Reproduction Assessment

Yes, from source inspection: on current main, a Nix-mode startup with plugin auto-enable changes reaches `replaceConfigFile`, so the documented read-only Nix config contract is violated. I did not run a live Nix gateway, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear. The Nix-only runtime snapshot approach matches the documented Nix contract, but the PR discussion asks whether all startup config mutation should move out of gateway startup, which needs maintainer scope judgment.
