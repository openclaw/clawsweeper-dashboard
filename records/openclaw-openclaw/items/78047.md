## Summary

Keep open: the policy direction and real behavior proof are strong, but the current head still makes legacy config-only plugin install records disappear from runtime plugin loading. That compatibility blocker should be fixed before maintainer/CODEOWNERS review proceeds.

## What This Changes

The PR makes startup-derived config changes runtime-only and adds Nix-mode refusal guards for config, plugin lifecycle, update, onboarding, and doctor repair mutators, with docs and changelog updates.

## Best Possible Solution

Keep the no-automatic-config-write and Nix immutability policy, but preserve legacy `plugins.installs` records in memory for passive reads while reserving durable cleanup for explicit writes or doctor repair.

## Reproduction Assessment

Yes. The remaining regression is source-reproducible: the PR strips `plugins.installs` on passive reads, while plugin loading still depends on the persisted index or `cfg.plugins.installs`.

## Solution Assessment

No. The central Nix guard and runtime-only startup derivations are the right shape, but legacy install records must remain runtime-visible before read-time persistence is removed.
