## Summary

Keep open: this is a plausible bug fix with sufficient real-behavior proof, but the latest head still selects the newest reset archive globally across all transcript candidates. That can violate existing candidate priority and make async history/count/title reads return a stale/custom/legacy archive instead of the preferred current-store archive when active files are missing.

## What This Changes

The PR adds async fallback from missing active session transcript files to reset archive siblings for chat history, sessions-history HTTP counts/pagination, and session title fields.

## Best Possible Solution

Preserve active-candidate behavior, then fall back by candidate order and choose the newest reset archive only within the first candidate base that has valid reset archives.

## Reproduction Assessment

Yes. Source inspection on current main shows reset archives are created, while the affected async readers/count/title helpers only resolve active transcript candidates, so an archive-only session falls through to empty, null, or zero results.

## Solution Assessment

No as currently written. The narrow fallback is the right shape, but archive selection must preserve resolveSessionTranscriptCandidates priority instead of picking the newest archive globally across all candidate bases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Preserve candidate priority when selecting archives:** `src/gateway/session-transcript-files.fs.ts:265-266`
