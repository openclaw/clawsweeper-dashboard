## Summary

Keep open: the PR is a focused and useful bug fix with supplied real behavior proof, but the current implementation can choose a lower-priority reset archive when multiple candidate transcript bases have archives.

## What This Changes

The PR adds a latest reset-archive fallback for async session transcript readers/count/title helpers, plus regression and HTTP pagination tests and a changelog entry.

## Best Possible Solution

Keep the narrow missing-active fallback, but make archive lookup preserve `resolveSessionTranscriptCandidates` priority before selecting the newest reset archive within that candidate base.

## Reproduction Assessment

Yes from source inspection: current main creates reset archives but the affected async readers only resolve active transcript paths, so archive-only sessions fall through to empty, null, or zero defaults. I did not execute a local repro because this review is read-only.

## Solution Assessment

No as currently written: the fallback should preserve candidate priority instead of choosing the newest archive globally across all candidate bases. Once that is fixed, the overall approach is the narrowest maintainable solution for the missing-active case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve candidate priority when selecting archives:** `src/gateway/session-transcript-files.fs.ts:265-266`
  - body: `findLatestResetArchiveAsync` keeps one `latest` while scanning every candidate base, but `resolveSessionTranscriptCandidates` deliberately orders explicit/current candidates before stale generated `sessionFile` fallbacks. If active files are missing and both a preferred candidate and a lower-priority stale/custom candidate have reset archives, the newer lower-priority archive wins, so `chat.history`, counts, and title fields can return the wrong session. Pick archives according to candidate order, then choose the newest archive only within the selected candidate base, and add a stale/generated-sessionFile regression.
  - confidence: 0.87
