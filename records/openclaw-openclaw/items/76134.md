## Summary

Keep open: the PR addresses a real archive-only async read gap, but latest head still routes the shared message-count helper through the archive fallback, which can make new write/event sequence numbers come from an archived transcript while active-wins history later reads the newly created active transcript.

## What This Changes

The PR adds a latest reset-archive fallback for missing-active async transcript history, title, visit, and count paths, with gateway tests and a changelog entry.

## Best Possible Solution

Keep the archive fallback for read/history surfaces, but split archive-aware history totals from active-transcript counts used to assign new live message sequences, then add a regression for sending after an archive-only state.

## Reproduction Assessment

Yes. The original issue is reproducible with a missing '<id>.jsonl' plus a sibling '<id>.jsonl.reset.<timestamp>'; the remaining PR regression is reproducible by sending or creating a message in that same archive-only state and comparing the returned messageSeq with the active-only history after the write.

## Solution Assessment

No as written. The fallback resolver is the right narrow approach for async history/title reads, but reusing it inside the shared count helper leaks archive totals into write-side sequence assignment.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Keep write-side counts on the active transcript:** `src/gateway/session-utils.fs.ts:526`
  - body: Routing `readSessionMessageCountAsync` through the archive fallback also changes callers that use this helper to assign the next live `messageSeq` before writing, such as `sessions.send`, `sessions.create`, and session event broadcasts. Keep archive-aware totals on history/read paths or add a separate helper for write-side counts.
