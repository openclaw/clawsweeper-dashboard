## Summary

Keep open. Current main still exposes `tools.sessions.visibility` as one string-valued scope and applies that same scope across send, history, list, and status access; the requested per-capability object model is not implemented. The related open A2A-send, allowlist, and per-agent scoping items are adjacent, but they do not supersede this permission-model request.

## What This Changes

The issue requests a backward-compatible object form for `tools.sessions.visibility` with separate send, history, list, and metadata controls while preserving the existing string shorthands.

## Best Possible Solution

Define one coherent session-tool capability model that preserves the existing string shorthands, adds a validated fail-closed object form for separate send/history/list/metadata permissions, and updates schema, docs, generated config metadata, and focused access-control tests together.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that current main still has a string-only schema and one shared visibility guard for send, history, list, and status.

## Solution Assessment

Unclear. The requested object model is plausible, but maintainers should choose the permission model alongside the active A2A-send, allowlist, and per-agent visibility proposals before implementation.
