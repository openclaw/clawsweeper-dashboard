## Summary

Keep open: current main and the latest release still model `tools.sessions.visibility` as one string-valued scope shared by send, history, list, and status, so the requested per-capability authorization object remains unimplemented and needs maintainer product/security review.

## What This Changes

The issue requests replacing the single `tools.sessions.visibility` scope enum with per-capability authorization knobs for send, history, list, and metadata access.

## Best Possible Solution

Define one backward-compatible session-tool authorization model that preserves existing string shorthands, adds any approved capability granularity, keeps sandbox and A2A restrictions fail-closed, and updates schema, docs, and isolation tests together.

## Reproduction Assessment

Yes at the source level: current schema, docs, and guards still model one `tools.sessions.visibility` enum rather than per-capability knobs. This is a feature/config gap, not a failing bug reproduction.

## Solution Assessment

Unclear until maintainers choose the permission model. The proposed object form is plausible, but a narrower A2A-send exception, allowlist model, or per-agent scoping model may better satisfy OpenClaw's config-surface bar.
