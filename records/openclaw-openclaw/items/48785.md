## Summary

Keep open. Current main still has one string-valued session visibility scope and applies that same scope across send, history, list, status, and session-backed memory filtering; the adjacent allowlist issue and A2A-send PR do not implement the requested per-capability object model.

## What This Changes

Request to replace the single `tools.sessions.visibility` string scope with backward-compatible per-capability controls for send, history, list, and metadata.

## Best Possible Solution

Define one coherent session-tool capability model: preserve existing string shorthands, add a validated object form for separate send/history/list/metadata permissions, keep sandbox and A2A restrictions fail-closed, update generated schema/help/docs, and add tests proving cross-agent send can be allowed without transcript history or broad listing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/config request; the source check is high-confidence because current main still has a string-only schema and one shared visibility guard for the affected actions.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The requested direction plausibly addresses the privacy gap, but maintainers should settle the exact permission model alongside A2A policy and allowlist work before implementation.
