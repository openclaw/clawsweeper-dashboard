## Summary

Keep open. Current main and the latest release still use one string-valued session visibility scope across list, history, send, and status, so the requested per-capability authorization model remains unimplemented and needs maintainer product/security review.

## What This Changes

The issue requests splitting `tools.sessions.visibility` into separate send, history, list, and metadata permission knobs so cross-agent sends can be allowed without transcript read access.

## Best Possible Solution

Define one backward-compatible, maintainer-approved session authorization model that preserves current string shorthands, adds only the approved per-capability controls, and updates schema, docs, migrations or doctor handling, and isolation tests together.

## Reproduction Assessment

Yes for source verification, but not as a bug reproduction. Current main and `v2026.6.10` still expose only the single string visibility setting and shared guard, so the requested configuration cannot be expressed today.

## Solution Assessment

Unclear until maintainers choose the durable permission model. The proposed object form is plausible, but per-agent scoping, target allowlists, sandbox visibility, and an A2A-send-specific exception may need to be consolidated with it.
