## Summary

Keep open. Current main and v2026.6.8 still model `tools.sessions.visibility` as one string-valued scope shared by send, history, list, and status; the requested per-capability authorization object remains unimplemented and needs maintainer product/security review.

## What This Changes

The issue requests replacing the single `tools.sessions.visibility` enum with opt-in per-capability knobs for send, history, list, and metadata access.

## Best Possible Solution

Define one backward-compatible session-tool authorization model that preserves existing string shorthands, adds any approved capability granularity, keeps sandbox and A2A restrictions fail-closed, and updates schema, docs, and isolation tests together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main and v2026.6.8 still lack the requested object form and still enforce one shared visibility scope.

## Solution Assessment

Unclear until maintainers choose the permission model. The proposed object form is plausible, but a narrower A2A-send exception, allowlist model, or per-agent scoping model may better satisfy OpenClaw's config-surface bar.
