## Summary

Keep open. Current main still exposes session-tool visibility as one string scope and applies it across send, history, list, status, and session-backed memory filtering; the adjacent allowlist, A2A-send, and per-agent proposals do not implement the requested per-capability object model.

## What This Changes

Request structured per-capability `tools.sessions.visibility` flags so send, history, list, and metadata/status access can be scoped independently.

## Best Possible Solution

Define one backward-compatible session-tool capability model that preserves string shorthands, validates object-form capabilities, keeps sandbox and A2A restrictions fail-closed, updates schema/help/docs, and proves cross-agent send can be allowed without transcript read/list/status access.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/config request; the source check is high-confidence because current main is still string-only and routes all affected actions through one shared visibility guard.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The per-capability object model plausibly addresses the privacy gap, but maintainers should settle how it composes with A2A policy, sandbox clamping, per-agent overrides, and allowlist work before implementation.
