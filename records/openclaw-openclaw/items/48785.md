## Summary

Keep open. Current main still exposes `tools.sessions.visibility` as one string-valued scope and applies that shared scope to send, history, list, and status paths; the requested per-capability object model is not implemented, and the related open allowlist/A2A/per-agent proposals do not supersede it.

## What This Changes

Requests a backward-compatible structured `tools.sessions.visibility` object with separate send, history, list, and metadata permissions.

## Best Possible Solution

Define one coherent session-tool capability model: preserve the existing string shorthands, add a validated object form for separate send/history/list/metadata permissions, keep sandbox and A2A restrictions fail-closed, update schema/help/docs, and add focused access-control tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/config request; source inspection gives high confidence that current main still has a string-only schema and one shared visibility guard.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The requested direction plausibly addresses the privacy gap, but maintainers should settle the exact permission model alongside A2A policy, per-agent overrides, and allowlist work before implementation.
