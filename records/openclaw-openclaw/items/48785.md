## Summary

Keep open. Current main and the latest shipped tag still expose `tools.sessions.visibility` as one string-valued scope shared by send, history, list, and status access, while the requested per-capability object form remains a security-sensitive product decision.

## What This Changes

Request a backward-compatible object form for `tools.sessions.visibility` so send, history, list, and metadata access can be controlled separately.

## Best Possible Solution

Define one backward-compatible, fail-closed session-tool capability model that keeps string shorthands, validates an object form, preserves sandbox and A2A restrictions, updates schema/types/help/docs, and adds focused access-control tests.

## Reproduction Assessment

Not applicable as a feature/config request. Source, docs, tests, and the latest shipped tag give high confidence that the requested object form does not exist today.

## Solution Assessment

Unclear until maintainers choose the public permission model. The object form is plausible, but it should be designed together with A2A send, allowlists, per-agent scoping, sandbox clamps, and memory/status semantics.
