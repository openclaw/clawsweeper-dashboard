## Summary

Keep open. Current main still exposes `tools.sessions.visibility` as one string-valued scope and uses that same scope for send, history, list, and status access; the requested per-capability object model is not implemented. The adjacent open A2A-send, allowlist, private-delivery, and per-agent proposals do not supersede this permission-model request.

## What This Changes

Request to add a backward-compatible structured `tools.sessions.visibility` object with separate send, history, list, and metadata permissions.

## Best Possible Solution

Define one coherent session-tool capability model that preserves the existing string shorthands, adds a validated object form for separate send/history/list/metadata permissions, keeps read surfaces fail-closed, and updates schema, docs, generated config metadata, and focused access-control tests.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that current main still has a string-only schema and one shared visibility guard.

## Solution Assessment

Unclear. The requested object model is plausible, but maintainers should settle the permission model alongside the active A2A-send, allowlist, and per-agent visibility proposals before implementation.
