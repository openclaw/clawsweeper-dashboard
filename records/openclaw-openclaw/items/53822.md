## Summary

Keep open: current main does not have this Docker volume config surface, and the PR is still a plausible feature, but the latest head still has a security-sensitive bind-source validation mismatch and lacks required real Docker behavior proof. The earlier reserved-target review thread appears addressed in the current diff, so the remaining blocker is narrower.

## What This Changes

The PR adds optional `sandbox.docker.volumes` entries for ephemeral, named, and bind Docker mounts across config/schema/runtime/docs, and rejects that config on SSH/OpenShell backends.

## Best Possible Solution

Keep the branch open, patch bind volume source validation to align with Docker `--mount` args, add regression coverage, and require redacted Docker runtime proof before merge.

## Reproduction Assessment

Yes for the blocker. Source inspection of PR head shows that a bind volume source such as `/tmp/ws:/outside` passes the new schema, is validated through a colon-delimited synthetic bind string, and is then sent to Docker as the original structured `source`.

## Solution Assessment

No. The feature direction is plausible, but the implementation should validate structured bind volume sources directly or reject colon-bearing POSIX bind sources before building both validation input and Docker `--mount` args.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: `volumeSettingToBindSpec` validates bind-strategy volumes by converting the structured source into `source:target:mode`, but `appendVolumeMounts` later sends the original `source` to Docker. A source like `/tmp/ws:/outside` can make the sandbox policy check `/tmp/ws` while Docker receives `/tmp/ws:/outside`, so reject unsafe `:` in POSIX bind volume sources or validate the structured source directly.
  - confidence: 0.9
