## Summary

Keep this PR open: the feature is not on current main, and the current PR head still has a security-sensitive validation mismatch for bind-strategy volume sources plus missing real Docker behavior proof.

## What This Changes

The branch adds optional `sandbox.docker.volumes` support for ephemeral, named, and bind Docker mounts, with config/schema validation, Docker `--mount` wiring, docs, changelog, and SSH/OpenShell rejection coverage.

## Best Possible Solution

Keep the feature branch open, validate structured bind volume sources directly or reject colon-bearing POSIX bind sources before both policy validation and Docker argument construction, add regression coverage, and require redacted real Docker proof before merge.

## Reproduction Assessment

Yes. Source inspection of PR head shows a bind volume source like `/tmp/ws:/outside` is converted to a legacy colon-delimited bind string for validation while Docker receives the original structured source.

## Solution Assessment

No, not as currently implemented. The feature direction is plausible, but the safer implementation is to validate the structured bind source directly or reject `:` in POSIX bind volume sources before both validation and Docker `--mount` construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: `volumeSettingToBindSpec` builds a legacy `source:target:mode` string for validation, but `appendVolumeMounts` sends the original structured `source` to Docker. A source such as `/tmp/ws:/outside` makes the bind parser validate `/tmp/ws` while Docker is asked to mount `/tmp/ws:/outside`, so reject `:` in POSIX bind volume sources or validate the structured source directly before both paths.
  - confidence: 0.9
