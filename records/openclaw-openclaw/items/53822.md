## Summary

Keep open: current main still lacks the requested config surface, but the latest PR head remains unsafe because structured bind volumes are validated through a colon-delimited adapter that can check a different source path than Docker mounts, and the external PR still lacks real Docker behavior proof.

## What This Changes

The PR adds `sandbox.docker.volumes` for ephemeral, named, and bind Docker mounts, threads it through sandbox config/schema/Docker and browser create args, rejects unsupported SSH/OpenShell backends, and updates docs/tests/changelog.

## Best Possible Solution

Keep the feature branch open, validate bind-strategy volumes structurally or reject colon-bearing bind sources, add regression coverage, and require redacted real Docker output proving each volume strategy before merge.

## Reproduction Assessment

Yes. Source inspection shows a bind volume source like `/tmp/workspace:/outside` passes the PR schema, is validated as `/tmp/workspace` after colon splitting, but is emitted to Docker as `source=/tmp/workspace:/outside`.

## Solution Assessment

No. The feature direction is reasonable, but the current implementation should validate the structured bind source directly or reject colon-bearing bind sources before building Docker `--mount` arguments, then merge only with real runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: Bind-strategy volumes are converted to a `source:target:mode` string only for `validateSandboxSecurity`, but Docker receives the original `source` in `--mount`. Because `splitSandboxBindSpec` splits on the first colon, `/tmp/workspace:/outside` is validated as `/tmp/workspace` while Docker is asked to mount `/tmp/workspace:/outside`, letting a colon-bearing source evade the allowed-root check. Reject `:` in bind volume sources or validate the structured source directly.
  - confidence: 0.9
