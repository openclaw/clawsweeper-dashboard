## Summary

Keep open. The feature is useful and the earlier reserved-target review blockers appear addressed, but the current PR head still builds Docker `--mount` specs by direct string interpolation, leaving a P1 sandbox mount option-injection issue before merge.

## What This Changes

The PR adds optional `sandbox.docker.volumes` config for ephemeral, named, and bind Docker mounts, with config/schema plumbing, Docker/browser create-arg wiring, SSH/OpenShell rejection, docs, tests, generated schema, and a changelog entry.

## Best Possible Solution

Keep the declarative volumes feature, but make `--mount` construction CSV-safe or reject mount-control characters in volume fields, then add focused regression coverage before merge.

## Reproduction Assessment

Yes for the review finding from source inspection: a volume `source` such as `cache,type=bind,source=/etc` or a `target` such as `/cache,target=/workspace` is accepted by PR validation and becomes extra Docker `--mount` fields. I did not run a live Docker create command in this read-only review.

## Solution Assessment

No, not as currently patched. The feature direction is reasonable, but the narrow maintainable fix is to encode or reject Docker mount CSV-control input before emitting `--mount`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build --mount specs without option injection:** `src/agents/sandbox/docker.ts:406-416`
  - body: `appendVolumeMounts` interpolates `source` and `target` directly into Docker's comma-separated `--mount` spec. Docker parses this as CSV key/value fields, so a named volume source like `cache,type=bind,source=/etc` or a target like `/cache,target=/workspace` can add later `type`, `source`, or `target` keys and bypass the PR's bind-source and reserved-target validation. Quote/escape according to Docker's parser or reject comma-bearing mount fields before emitting args.
  - confidence: 0.9

## Security Review

Status: needs_attention
