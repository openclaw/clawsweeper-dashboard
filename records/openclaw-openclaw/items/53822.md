## Summary

Keep this PR open: current main still lacks the requested Docker volume config surface, but PR head has a security-relevant bind-source validation mismatch and does not yet include real Docker runtime proof.

## What This Changes

The PR adds optional `sandbox.docker.volumes` entries for ephemeral, named, and bind Docker mounts across config, schema, runtime, tests, docs, and non-Docker backend rejection paths.

## Best Possible Solution

Keep the PR open, align structured bind-source validation with Docker `--mount` construction, add a colon-source regression test, and require redacted real Docker runtime proof before merge.

## Reproduction Assessment

Yes for the blocker: source inspection of PR head shows a bind volume source like `/tmp/ws:/outside` is adapted into a colon-delimited legacy bind string for validation while Docker receives the original structured source. I did not run runtime tests because this review is read-only.

## Solution Assessment

No, not as implemented. The feature direction is plausible, but the PR should validate structured bind volume sources directly or reject colon-bearing POSIX bind sources before both policy validation and Docker argument construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: `volumeSettingToBindSpec` validates bind-strategy volumes by converting the structured source into `source:target:mode`, but `appendVolumeMounts` later passes the original source to Docker. A source like `/tmp/ws:/outside` can make the sandbox policy check `/tmp/ws` while Docker receives `/tmp/ws:/outside`, so reject unsafe `:` in POSIX bind volume sources or validate the structured source directly.
  - confidence: 0.9
