## Summary

Keep this PR open: the earlier reserved-target concerns appear addressed, but the current head still builds Docker `--mount` strings with unescaped config values, leaving a concrete mount-option injection blocker before merge.

## What This Changes

The branch adds a declarative `sandbox.docker.volumes` config surface for Docker sandbox mounts, wires it into Docker/browser create args, rejects it on SSH/OpenShell, and updates docs, schema, changelog, and tests.

## Best Possible Solution

Keep the feature direction, but make mount argument serialization CSV-safe or reject mount-control characters, then add focused regression tests for injected `source`, `target`, and `type` fields.

## Reproduction Assessment

Yes for the review finding from source inspection: comma-bearing `source` or `target` values pass the PR validation and become extra Docker `--mount` CSV fields. I did not run Docker locally in this read-only review.

## Solution Assessment

No, not as currently patched. The feature shape is reasonable, but the mount builder must encode or reject CSV control characters before this is a maintainable sandbox config surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build --mount args without CSV key injection:** `src/agents/sandbox/docker.ts:396-416`
  - body: `appendVolumeMounts` interpolates config `source` and `target` directly into Docker's comma-separated `--mount` spec. Docker parses that string as CSV key/value fields, so values containing commas can add later `type`, `source`, or `target` keys after OpenClaw's validation and change what is mounted. Quote/escape the CSV fields or reject mount-control characters before emitting `--mount`.
  - confidence: 0.9
