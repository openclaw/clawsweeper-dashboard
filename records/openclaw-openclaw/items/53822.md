## Summary

Keep open. Current main does not include this feature, and PR head still has a security-sensitive bind-volume validation mismatch plus missing real Docker behavior proof.

## What This Changes

The PR adds optional `sandbox.docker.volumes` config for ephemeral, named, and bind Docker mounts, with validation, Docker `--mount` wiring, docs, changelog, and SSH/OpenShell rejection coverage.

## Best Possible Solution

Keep the feature branch open only after bind volume sources are validated as structured values or colon-bearing POSIX sources are rejected before both policy checks and Docker args, with regression coverage and redacted real Docker proof.

## Reproduction Assessment

Yes for the PR-head defect: source inspection shows a bind volume source like `/tmp/ws:/outside` is converted into a colon-delimited validation string while Docker receives the original structured source. Current main cannot reproduce the new feature behavior because `sandbox.docker.volumes` is not present there.

## Solution Assessment

No, not as currently implemented. The maintainable fix is to validate structured bind volume sources directly or reject `:` in POSIX bind volume sources before both policy validation and Docker `--mount` construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: `volumeSettingToBindSpec` builds a legacy `source:target:mode` string from structured bind volumes, but `appendVolumeMounts` later passes the original source to Docker. A POSIX path containing `:` makes validation inspect the segment before the first colon while Docker receives the full source, so reject colon-bearing bind sources or validate the structured source directly.
  - confidence: 0.9

## Security Review

Status: needs_attention
