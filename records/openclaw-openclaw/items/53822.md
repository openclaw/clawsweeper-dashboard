## Summary

This PR should stay open: the feature is not on current main, but the exact PR head still has a security-sensitive bind-volume validation mismatch and lacks after-fix real Docker behavior proof.

## What This Changes

The PR adds optional `sandbox.docker.volumes` config for ephemeral, named, and bind Docker mounts, wiring it through schema/config merge, Docker and browser create args, SSH/OpenShell rejection paths, docs, changelog, and tests.

## Best Possible Solution

Keep the branch open, validate bind-strategy volume sources directly or reject colon-bearing POSIX sources before building Docker `--mount` args, add regression coverage, and require redacted real Docker output or logs before merge.

## Reproduction Assessment

Yes. Source inspection of the exact PR head gives a high-confidence path: configure a bind volume source containing a colon, such as `/tmp/ws:/outside`, and the synthetic validation string can be parsed as host `/tmp/ws` while Docker receives the original structured source.

## Solution Assessment

No. The feature direction is plausible, but the current implementation should validate the structured bind source itself, or explicitly reject unsafe colon-bearing POSIX bind sources, instead of adapting it through the legacy colon-delimited bind parser.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: `volumeSettingToBindSpec` validates bind-strategy volumes by converting the structured source into a colon-delimited bind string, but `appendVolumeMounts` later sends the original `source` to Docker. A source like `/tmp/ws:/outside` can be validated as `/tmp/ws` while Docker receives `/tmp/ws:/outside`, so the sandbox allowed-root and blocked-path checks can inspect the wrong host path. Reject unsafe `:` in POSIX bind volume sources or validate the structured source directly.
  - confidence: 0.9
