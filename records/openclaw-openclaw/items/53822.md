## Summary

Keep open: the feature is not implemented on current main, and the latest PR head still has a source-reproducible bind-volume validation bypass plus missing real Docker runtime proof.

## What This Changes

The PR adds optional `sandbox.docker.volumes` entries for ephemeral, named, and bind-style Docker mounts, with config/schema plumbing, Docker/browser create-arg wiring, SSH/OpenShell rejection, docs, tests, and a changelog entry.

## Best Possible Solution

Keep the PR open, validate bind-strategy volumes from structured fields or reject colon-bearing bind sources, add a regression test for that case, and require redacted real Docker output for the new mount strategies.

## Reproduction Assessment

Yes. From source inspection, a bind volume source such as `/tmp/ws:/outside` passes the PR schema, is validated as `/tmp/ws` after colon splitting, and is emitted to Docker as `/tmp/ws:/outside`.

## Solution Assessment

No. The feature direction is reasonable, but the current implementation should validate the structured bind source directly or reject colon-bearing bind sources before building Docker `--mount` arguments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: `volumeSettingToBindSpec` builds a colon-delimited bind spec only for validation, but `appendVolumeMounts` later emits the original `source` in Docker `--mount`. A source like `/tmp/ws:/outside` is validated as `/tmp/ws` by `splitSandboxBindSpec` while Docker receives `/tmp/ws:/outside`, so the allowed-root policy can check a different path than the one mounted. Reject `:` in bind volume sources or validate the structured source directly.
  - confidence: 0.88
