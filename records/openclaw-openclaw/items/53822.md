## Summary

Keep open: current main does not include this new Docker volume config surface, and the exact PR head still has a high-confidence bind-source validation bypass plus no real Docker behavior proof from the external contributor. The earlier reserved-target review comments appear addressed in the current head.

## What This Changes

The PR adds `sandbox.docker.volumes` for ephemeral, named, and bind-style Docker mounts, wires it through config/schema/Docker/browser paths, rejects it on SSH/OpenShell, and updates docs/tests/changelog.

## Best Possible Solution

Keep the PR open, validate bind-strategy volume sources directly or reject unsafe colon-bearing POSIX sources, add a regression test, and require redacted real Docker proof before merge.

## Reproduction Assessment

Yes. Source inspection of the exact PR head shows a bind volume source such as `/tmp/ws:/outside` is converted to a colon-delimited validation string that the existing parser reads as host `/tmp/ws`, while Docker receives the original structured source.

## Solution Assessment

No. The feature direction is plausible, but the implementation should validate the structured bind source directly or reject unsafe colon-bearing POSIX bind sources before building Docker `--mount` arguments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: `volumeSettingToBindSpec` validates bind-strategy volumes by converting the structured source into a colon-delimited bind string, but `appendVolumeMounts` later sends the original `source` to Docker. A source like `/tmp/ws:/outside` can be validated as `/tmp/ws` while Docker mounts `/tmp/ws:/outside`, so allowed-root and blocked-path policy can check the wrong host path. Reject unsafe `:` in POSIX bind volume sources or validate the structured source directly.
  - confidence: 0.9
