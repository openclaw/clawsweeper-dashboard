## Summary

Keep this PR open: current main does not have the requested `sandbox.docker.volumes` surface, but the PR still has a source-provable sandbox bind validation mismatch and lacks required after-fix real Docker behavior proof.

## What This Changes

The PR adds optional declarative `sandbox.docker.volumes` config for ephemeral, named, and bind Docker mounts, with validation, Docker `--mount` wiring, docs, changelog, and SSH/OpenShell rejection coverage.

## Best Possible Solution

Keep the PR open, validate structured bind volume sources directly or reject colon-bearing POSIX bind sources before both policy validation and Docker argument construction, add regression coverage, and require redacted real Docker runtime proof before merge.

## Reproduction Assessment

Yes. Source inspection of PR head shows a bind volume source like `/tmp/ws:/outside` is converted into a colon-delimited legacy bind string for validation while Docker receives the original structured source.

## Solution Assessment

No, not as currently implemented. The feature direction is plausible, but the maintainable fix is to validate the structured bind source directly or reject `:` in POSIX bind volume sources before both policy validation and Docker `--mount` construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject colon-bearing bind volume sources:** `src/agents/sandbox/docker.ts:386`
  - body: `volumeSettingToBindSpec` builds a legacy `source:target:mode` string for validation, but `appendVolumeMounts` sends the original structured `source` to Docker. A source such as `/tmp/ws:/outside` makes the bind parser validate `/tmp/ws` while Docker is asked to mount `/tmp/ws:/outside`, so reject `:` in POSIX bind volume sources or validate the structured source directly.
  - confidence: 0.9
