## Summary

Keep open: the PR fixes the main structured-body path and supplies real behavior proof, but the latest head still treats a mention entry's display name as an identity candidate, leaving a `requireMention` bypass on display-name collisions.

## What This Changes

The PR parses Nextcloud Talk structured webhook content, feeds the effective text into mention and command dispatch, forces automatic group reply delivery, adds mention parsing tests, and updates the changelog.

## Best Possible Solution

Keep the structured-body parsing and automatic Nextcloud Talk source delivery, but restrict explicit mention matching to stable structured identifiers before merge.

## Reproduction Assessment

Yes. Current main is source-reproducible: the Talk webhook body is structured JSON while the current inbound handler applies command and mention gating to `rawBody`.

## Solution Assessment

No. The overall direction is right, but the implementation should stop using mutable mention display names as authenticated bot identity before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Remove display names from mention identity checks:** `extensions/nextcloud-talk/src/inbound.ts:117`
  - body: The follow-up removed `account.name` from expected IDs, but this line still compares `entry.name`. In Talk rich objects, `name` is display text, so an allowed room message mentioning another user whose display name equals the bot account id or API-user local part can satisfy `requireMention`; compare only stable `id`/`mentionId` fields.
  - confidence: 0.88
