## Summary

Keep open: current main and v2026.6.9 still pass Nextcloud Talk structured `object.content` through as raw text for mention gating, while this PR has plausible live proof and no discrete code-review blocker; the remaining question is maintainer ownership of the structured-body parser boundary before merge.

## What This Changes

The PR adds Nextcloud Talk structured `object.content` parsing for mention detection and command text, and adds focused tests for mention parsing plus shared visible-reply delivery policy.

## Best Possible Solution

Land this or a coordinated replacement as the canonical Nextcloud Talk structured-body parser for mention/text handling, while explicitly deciding how it should coexist with the rich-object attachment parser work.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.9 copy Nextcloud Talk `object.content` into `message.text` and match mentions against `rawBody`, while the upstream contract says that field is structured JSON with `message` and `parameters`. I did not run a live Nextcloud Talk server in this read-only review.

## Solution Assessment

Not fully settled: plugin-local structured parsing is the right layer for the bug, but the best merge shape should coordinate with the open rich-object attachment parser so Nextcloud Talk has one maintainable parser boundary.
