## Summary

Keep this PR open: the reported Nextcloud Talk structured-body bug is real and the PR is useful, but the current head still has two merge-blocking parser/mention-gate issues.

## What This Changes

The PR parses Nextcloud Talk structured webhook JSON, resolves explicit mentions, feeds effective text into inbound dispatch, forces automatic room reply delivery, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the structured-body support, but render rich-object placeholders accurately and gate explicit mentions on stable bot identifiers before merging.

## Reproduction Assessment

Yes. Current main is source-reproducible: Nextcloud Talk object.content is structured JSON, while the inbound handler uses rawBody for command gating, mention matching, and agent context.

## Solution Assessment

No. Parsing structured bodies and using automatic source delivery are reasonable directions, but this implementation should preserve non-mention rich-object text and avoid treating the account display label as an authenticated mention identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve non-mention rich-object text:** `extensions/nextcloud-talk/src/inbound.ts:88`
  - body: The parser removes every `{key}` listed in `parameters`, but Talk uses the same placeholder mechanism for non-user rich objects such as calls, files, and links whose `name` is rendered in the chat view. A message like `look at {mention-call1}` reaches the agent as `look at`, losing content the user saw; strip only the bot mention and substitute other placeholders with display text.
  - confidence: 0.92
- **[P2] Stop treating account labels as mentions:** `extensions/nextcloud-talk/src/inbound.ts:111`
  - body: `account.name` is documented as a CLI/UI display label, but adding it to `expectedIds` lets an allowed room message that mentions another user with the same display name satisfy `requireMention`. Please match explicit mentions only against stable Nextcloud identifiers unless a separate bot display-name setting is introduced.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain or secrets-handling concern, but it weakens mention gating by accepting a mutable display label as a bot identity.

Concerns:

- **[low] Mutable display label can bypass mention gating:** `extensions/nextcloud-talk/src/inbound.ts:111`
  - body: In allowlisted rooms with `requireMention`, treating `channels.nextcloud-talk.*.name` as an authenticated bot mention identity can trigger the agent when a different user with that display name is mentioned.
  - confidence: 0.86
