## Summary

Keep open. Current main still routes anonymous OpenAI-compatible chat-completion requests through generated agent sessions, while the public docs do not yet tell uptime monitors to use the lightweight probe endpoints; #55785 was closed unmerged and #62436 tracks broader passthrough runtime work rather than this narrow monitoring guidance gap.

## What This Changes

Request public gateway monitoring guidance that directs uptime probes to `/health` or `/healthz` instead of `/v1/chat/completions`, with runtime warning/rate-limit ideas left as follow-ups.

## Best Possible Solution

Add public gateway monitoring guidance that names the contract: use `GET /health` or `/healthz` for liveness, use `/ready` or `/readyz` when readiness matters, and avoid `/v1/chat/completions` for uptime checks because it is a full agent endpoint that can create sessions and call a model.

## Reproduction Assessment

Yes. Source inspection shows an anonymous `POST /v1/chat/completions` request without `x-openclaw-session-key` or `user` gets an `openai:<uuid>` session key and runs through the agent path, while `/healthz` is a shallow probe covered by tests.

## Solution Assessment

Partly. Docs-first monitoring guidance is the narrowest maintainable fix for this issue; runtime warnings, metrics, rate limits, or passthrough mode should remain separate follow-up decisions.
