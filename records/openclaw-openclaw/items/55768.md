## Summary

Keep open. Current `main` and the latest shipped release still route anonymous OpenAI-compatible Chat Completions through the normal agent/session path, and the public health/OpenAI HTTP docs still do not clearly warn uptime monitors away from that endpoint; the closed related docs PR attempted the right idea but was not merged.

## What This Changes

The issue asks OpenClaw to document and enforce that uptime monitors use lightweight health/readiness probes instead of `/v1/chat/completions`, which creates normal agent sessions.

## Best Possible Solution

Add concise public docs that identify `/health` or `/healthz` as liveness probes, `/ready` or `/readyz` as readiness probes, and explicitly say `/v1/chat/completions` is for real chat clients because it executes and persists a normal agent run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an anonymous `POST /v1/chat/completions` without `x-openclaw-session-key` or `user` gets an `agent:<id>:openai:<uuid>` session key before entering the agent command path; I did not run it live to avoid creating review artifacts.

## Solution Assessment

Yes for the immediate repair. A docs-only update is the narrow maintainable fix for misconfigured uptime monitoring; warning, metrics, passthrough, or rate-limit behavior should be separate product work.
