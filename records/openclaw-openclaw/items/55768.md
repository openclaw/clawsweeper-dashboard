## Summary

Keep open. Current main and the latest release still route anonymous `/v1/chat/completions` calls through generated `openai:<uuid>` agent session keys and the normal persisted agent path, while the public health docs still do not clearly warn uptime monitors to use `/health` or `/healthz` instead.

## What This Changes

The issue asks OpenClaw to prevent or document uptime monitors using `/v1/chat/completions` as a health check because anonymous requests create persisted agent sessions.

## Best Possible Solution

Publish focused public gateway health guidance that tells uptime monitors to use `/health` or `/healthz` for liveness and `/ready` or `/readyz` for readiness, warns that Chat Completions performs a normal persisted agent run, and leaves no-persistence relay behavior to https://github.com/openclaw/openclaw/issues/62436.

## Reproduction Assessment

Yes, at source level. An anonymous `POST /v1/chat/completions` without `user` or `x-openclaw-session-key` still gets an `agent:<id>:openai:<uuid>` session key before dispatching through the normal agent command path; I did not run it live to avoid creating session artifacts.

## Solution Assessment

Yes for the immediate repair. A docs-only update is the narrow maintainable fix for misconfigured uptime monitoring; runtime warnings, metrics, passthrough, or rate-limit behavior should stay as separate product work.
