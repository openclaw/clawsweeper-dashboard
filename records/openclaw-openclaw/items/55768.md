## Summary

Keep open. Current main still documents and implements `/v1/chat/completions` as a normal agent run with generated session keys, while the gateway health docs still do not explicitly tell uptime monitors to use the lightweight liveness/readiness probes instead.

## What This Changes

The issue requests monitoring guidance and possible safeguards so uptime probes use `/health` or related probe endpoints instead of creating OpenAI-compatible chat sessions.

## Best Possible Solution

Add focused public docs that identify `/health` or `/healthz` as liveness probes, `/ready` or `/readyz` as readiness probes, and explicitly say `/v1/chat/completions` is for real chat clients because it executes and persists a normal agent run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an anonymous `POST /v1/chat/completions` without `user` or `x-openclaw-session-key` gets an `agent:<id>:openai:<uuid>` session key before entering `agentCommandFromIngress`; I did not run it live to avoid creating review artifacts.

## Solution Assessment

Yes for the immediate repair. A docs-only update is the narrow maintainable fix for misconfigured uptime monitoring; warning, metrics, passthrough, or rate-limit behavior should stay separate product work.
