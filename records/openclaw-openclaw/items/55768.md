## Summary

Keep open. Current main still routes anonymous `/v1/chat/completions` requests through generated OpenAI session keys and the normal persisted agent path, while the public health docs still do not clearly tell uptime monitors to use `/health`/`/healthz` instead; the earlier health-doc PR was closed unmerged and the broader stateless passthrough request is separate.

## What This Changes

The issue asks for monitoring guidance, and possibly later runtime safeguards, so uptime checks use lightweight probe endpoints instead of creating Chat Completions agent sessions.

## Best Possible Solution

Publish focused public gateway health guidance that names `/health`/`/healthz` for liveness and `/ready`/`/readyz` for readiness, warns that Chat Completions performs and persists a normal agent run, and leaves no-persistence relay behavior to the separate passthrough design issue.

## Reproduction Assessment

Yes, at source level. An anonymous `POST /v1/chat/completions` without `user` or `x-openclaw-session-key` still gets an `agent:<id>:openai:<uuid>` session key before dispatching through the normal agent command path; I did not run it live to avoid creating session artifacts.

## Solution Assessment

Yes for the immediate repair. A docs-only update is the narrow maintainable fix for misconfigured uptime monitoring; runtime warnings, metrics, passthrough, or rate-limit behavior should stay as separate product work.
