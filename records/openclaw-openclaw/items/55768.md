## Summary

Keep open. Current main and v2026.5.19 still route anonymous Chat Completions through generated agent session keys, while the public health page does not explicitly tell external uptime monitors to use the lightweight probe endpoints; the earlier docs PR was closed unmerged and the broader no-persistence relay request remains separate.

## What This Changes

The issue requests public guidance and guardrails so uptime monitors use `/health` or `/healthz` instead of `/v1/chat/completions`.

## Best Possible Solution

Publish focused public gateway health guidance that names the correct liveness/readiness endpoints and warns that Chat Completions performs and persists a normal agent run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an anonymous `POST /v1/chat/completions` without `user` or `x-openclaw-session-key` gets an `agent:<id>:openai:<uuid>` session key before entering `agentCommandFromIngress`; I did not run it live to avoid creating session artifacts.

## Solution Assessment

Yes for the immediate repair. A docs-only update is the narrow maintainable fix for misconfigured uptime monitoring; warning, metrics, passthrough, or rate-limit behavior should stay separate product work.
