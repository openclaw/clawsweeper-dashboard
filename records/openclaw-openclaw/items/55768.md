## Summary

Keep open. Current main still creates generated OpenAI-compatible agent sessions for anonymous chat-completion calls, the public health docs still focus on CLI/channel health rather than external HTTP uptime probes, #55785 closed unmerged, and #62436 tracks a broader passthrough-mode feature rather than this narrow monitoring guidance gap.

## What This Changes

Request guidance and safeguards so uptime monitors use lightweight gateway probe endpoints instead of `/v1/chat/completions`.

## Best Possible Solution

Add public gateway monitoring guidance that names `/health` and `/healthz` for liveness, `/ready` and `/readyz` for readiness, and warns that `/v1/chat/completions` is a full agent endpoint that can create sessions and call a model.

## Reproduction Assessment

Yes. Source inspection shows an anonymous `POST /v1/chat/completions` request without `x-openclaw-session-key` or `user` gets an `openai:<uuid>` session key and runs through the agent path, while `/healthz` is a shallow probe covered by tests.

## Solution Assessment

Partly. Docs-first monitoring guidance is the narrowest maintainable fix for this issue; runtime warning, metrics, rate-limit, or passthrough changes should stay separate follow-up decisions.
