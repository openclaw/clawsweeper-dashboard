## Summary

Keep open. Current main still creates generated OpenAI session keys for anonymous Chat Completions requests and sends them through the agent path, while public gateway docs still lack explicit uptime-monitor guidance; the earlier docs PR is closed unmerged and the related passthrough issue tracks broader product work.

## What This Changes

Request guidance and optional safeguards so uptime monitors use lightweight health probes instead of the session-creating OpenAI Chat Completions endpoint.

## Best Possible Solution

Document the operational contract in public gateway docs: use `/health` or `/healthz` for liveness, `/ready` or `/readyz` for readiness, and avoid `/v1/chat/completions` for uptime checks because it is a full agent endpoint that can create sessions and call a model.

## Reproduction Assessment

Yes. Source inspection shows an anonymous `POST /v1/chat/completions` without `x-openclaw-session-key` or `user` gets an `agent:<id>:openai:<uuid>` session key and runs through `agentCommandFromIngress`; I did not run a live request because this read-only review should not create session artifacts.

## Solution Assessment

Yes for the narrow repair. Docs-first monitoring guidance is the smallest maintainable fix here; warning, metrics, passthrough, or rate-limit behavior should stay separate product work.
