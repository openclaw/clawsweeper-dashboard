## Summary

Keep open. Current main still makes anonymous OpenAI-compatible Chat Completions a normal agent run with a generated session key, while public docs still do not clearly tell uptime monitors to use the lightweight health/readiness probes instead.

## What This Changes

The issue asks for operational guidance and optional safeguards so uptime monitors use `/health` or related probe endpoints instead of `/v1/chat/completions`.

## Best Possible Solution

Document the operational contract in public gateway docs: use `/health` or `/healthz` for liveness, `/ready` or `/readyz` for readiness, and avoid `/v1/chat/completions` for uptime checks because it creates a normal agent run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an anonymous `POST /v1/chat/completions` without `x-openclaw-session-key` or `user` gets `agent:<id>:openai:<uuid>` and enters `agentCommandFromIngress`; I did not run it live to avoid creating review artifacts.

## Solution Assessment

Yes for the narrow repair. A docs update is the smallest maintainable fix for misconfigured uptime monitoring; warning, metrics, passthrough, or rate limits should be separate product work.
