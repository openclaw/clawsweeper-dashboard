## Summary

Keep open. Current main and the latest release still route anonymous Chat Completions through generated agent session keys, while the gateway health docs do not explicitly steer external uptime monitors to the lightweight liveness/readiness probes; the earlier docs PR at https://github.com/openclaw/openclaw/pull/55785 was closed unmerged, and the broader no-persistence relay request remains separate at https://github.com/openclaw/openclaw/issues/62436.

## What This Changes

Request to document external HTTP monitoring against `/health`/`/healthz` and `/ready`/`/readyz` instead of Chat Completions, with runtime warning/rate-limit ideas left as optional follow-up work.

## Best Possible Solution

Publish focused public gateway health guidance that names the correct liveness/readiness endpoints and warns that Chat Completions performs and persists a normal agent run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an anonymous `POST /v1/chat/completions` without `user` or `x-openclaw-session-key` gets an `agent:<id>:openai:<uuid>` session key before entering `agentCommandFromIngress`; I did not run it live to avoid creating session artifacts.

## Solution Assessment

Yes for the immediate repair. A docs-only update is the narrow maintainable fix for misconfigured uptime monitoring; warning, metrics, passthrough, or rate-limit behavior should stay separate product work.
