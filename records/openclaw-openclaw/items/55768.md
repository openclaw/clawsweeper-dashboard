## Summary

Keep open. Current main still routes anonymous OpenAI-compatible chat-completion requests through the full agent/session path, public docs still lack explicit uptime-monitor guidance, the prior docs PR is closed unmerged, and the related passthrough issue is a broader product feature rather than a replacement for this monitoring docs gap.

## What This Changes

Request guidance that uptime monitors use lightweight gateway probe endpoints instead of `/v1/chat/completions`, with optional follow-ups for warnings or rate limits on high anonymous session creation.

## Best Possible Solution

Document the operational contract in public gateway docs: use `/health` or `/healthz` for liveness, `/ready` or `/readyz` for readiness, and avoid `/v1/chat/completions` for uptime checks because it is a full agent endpoint that can create sessions and call a model.

## Reproduction Assessment

Yes. Source inspection shows an anonymous `POST /v1/chat/completions` without `x-openclaw-session-key` or `user` gets an `agent:<id>:openai:<uuid>` session key and proceeds through `agentCommandFromIngress`; I did not run a live request because this read-only review should not create session artifacts.

## Solution Assessment

Yes for the narrow repair. Docs-first monitoring guidance is the smallest maintainable fix; gateway warning, metrics, passthrough, or rate-limit behavior should remain separate follow-up decisions.
