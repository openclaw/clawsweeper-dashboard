## Summary

Keep open. Current main still sends anonymous OpenAI-compatible chat-completion requests through the persisted agent-session path, and the public gateway docs still do not tell external uptime monitors to use the lightweight probe endpoints instead. The closed docs PR #55785 was not merged, and #62436 tracks broader stateless passthrough behavior rather than this narrow monitoring guidance gap.

## What This Changes

Request documentation and deployment guidance that uptime monitors use lightweight Gateway probe endpoints such as `GET /health` or `/healthz` instead of `POST /v1/chat/completions`, with optional future warning or rate-limit work for excessive session creation.

## Best Possible Solution

Add public Gateway monitoring guidance that names the intended probe contract: use `GET /health` or `/healthz` for liveness, use `/readyz` when readiness matters, and avoid `/v1/chat/completions` for uptime checks because it is a full agent endpoint that can create persisted sessions and call a model. Cross-link that guidance from the OpenAI HTTP API docs, while leaving passthrough mode, warning counters, and rate limiting as separate runtime/product follow-ups.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: `POST /v1/chat/completions` without `x-openclaw-session-key` or `user` resolves to an `openai:<uuid>` session key and enters the agent command path, while `/healthz` returns a shallow liveness response.

## Solution Assessment

Yes for the narrow repair lane. A docs-first fix is the smallest maintainable solution because lightweight probe endpoints already exist and are covered by tests; changing session persistence, warning heuristics, or rate limits should be separate runtime work.
