## Summary

Keep open. Current main still sends anonymous `/v1/chat/completions` calls through generated OpenAI session keys and the normal agent execution path, while the public docs still lack explicit uptime-monitor guidance warning users to use the lightweight HTTP probes instead; the related docs PR was closed unmerged and the broader no-persistence relay request remains separate.

## What This Changes

The issue asks for deployment/docs guidance that external uptime checks use `/health` or readiness probes instead of `/v1/chat/completions`, with optional runtime warning and rate-limit follow-up.

## Best Possible Solution

Publish focused public gateway health guidance that names `/health`/`/healthz` for liveness and `/ready`/`/readyz` for readiness, warns that Chat Completions performs and persists a normal agent run, and leaves no-persistence relay behavior to the separate passthrough design issue.

## Reproduction Assessment

Yes, at source level. An anonymous `POST /v1/chat/completions` without `user` or `x-openclaw-session-key` still gets an `agent:<id>:openai:<uuid>` session key before dispatching through `agentCommandFromIngress`; I did not run it live to avoid creating session artifacts.

## Solution Assessment

Yes for the immediate fix. A docs-only update is the narrow maintainable repair for misconfigured uptime monitoring; warning, metrics, passthrough, or rate-limit behavior should stay as separate product work.
