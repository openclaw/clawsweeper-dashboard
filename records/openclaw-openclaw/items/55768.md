## Summary

Keep open. Current main still routes anonymous OpenAI-compatible Chat Completions through generated `agent:<id>:openai:<uuid>` sessions, while public gateway docs still do not explicitly tell uptime monitors to use `/health` or `/healthz` instead.

## What This Changes

The issue requests operational guidance that uptime monitors use `/health` rather than `/v1/chat/completions`, plus optional warning/rate-limit ideas for repeated anonymous session creation.

## Best Possible Solution

Document the operational contract in public gateway docs: use `/health` or `/healthz` for liveness, `/ready` or `/readyz` for readiness, and avoid `/v1/chat/completions` for uptime checks because it creates a normal agent run.

## Reproduction Assessment

Yes. Source inspection shows an anonymous `POST /v1/chat/completions` without `x-openclaw-session-key` or `user` gets an `agent:<id>:openai:<uuid>` session key and runs through `agentCommandFromIngress`; I did not run a live request because that would create session artifacts.

## Solution Assessment

Yes for the narrow repair. Documentation is the smallest maintainable fix for misconfigured uptime monitors; warning, metrics, passthrough, or rate-limit behavior should stay separate product work.
