## Summary

Keep open. Current main still sends anonymous OpenAI-compatible chat-completion requests through the generated agent-session path, while the lightweight health/readiness probe endpoints exist but the public docs do not tell uptime monitors to use them or warn against `/v1/chat/completions`. The direct docs PR #55785 was closed unmerged, and #62436 tracks broader stateless/passthrough runtime work rather than superseding this narrow monitoring guidance gap.

## What This Changes

Request monitoring guidance and optional safeguards so uptime checks use lightweight health/readiness endpoints instead of `/v1/chat/completions`, which creates OpenAI-compatible agent sessions.

## Best Possible Solution

Add public gateway monitoring guidance that names the operational contract: use `GET /health` or `/healthz` for liveness, use `/readyz` when readiness matters, and avoid `/v1/chat/completions` for uptime checks because it is a full agent endpoint that can create persisted sessions and call a model. Cross-link that guidance from the OpenAI-compatible HTTP docs; keep passthrough mode, warning heuristics, and rate limits as separate product/runtime follow-ups.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: a `POST /v1/chat/completions` request without `x-openclaw-session-key` or `user` resolves to an `openai:<uuid>` session key and enters the agent command path, while `/healthz` returns a shallow liveness response covered by tests.

## Solution Assessment

Yes for the immediate repair lane. A docs-first fix is the narrowest maintainable step because the lightweight probe endpoints already exist and are tested; changing session persistence, adding warning heuristics, or adding rate limits should be separately reviewed runtime work.
