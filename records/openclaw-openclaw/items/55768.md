## Summary

Keep open. Current main still creates generated OpenAI session keys for anonymous chat-completion requests and sends them through the agent path, while public docs do not yet warn uptime monitors to use the lightweight probe endpoints; the earlier docs PR at https://github.com/openclaw/openclaw/pull/55785 is closed unmerged, and https://github.com/openclaw/openclaw/issues/62436 tracks a broader passthrough feature rather than this monitoring guidance gap.

## What This Changes

The issue asks for monitoring guidance to use `/health` or `/healthz` instead of `/v1/chat/completions`, with optional runtime warning/rate-limit follow-ups for high-frequency session creation.

## Best Possible Solution

Document the operational contract in public gateway docs: use `/health` or `/healthz` for liveness, `/ready` or `/readyz` for readiness, and avoid `/v1/chat/completions` for uptime checks because it is a full agent endpoint that may create sessions and call a model.

## Reproduction Assessment

Yes. Source inspection shows an anonymous `POST /v1/chat/completions` without `x-openclaw-session-key` or `user` gets an `agent:<id>:openai:<uuid>` session key and runs through `agentCommandFromIngress`; I did not run a live request because this read-only review should not create session artifacts.

## Solution Assessment

Yes for the narrow repair. Docs-first monitoring guidance is the smallest maintainable fix here; warning, metrics, passthrough, or rate-limit behavior should remain separate product decisions.
