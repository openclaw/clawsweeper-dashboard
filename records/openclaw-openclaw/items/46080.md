## Summary

Keep open. Current `origin/main` still supports custom `api: "anthropic-messages"` providers and replays `toolResult` messages into Anthropic `tool_result` blocks, but the non-visible empty-turn retry guard only covers strict/Gemini, `openai-completions`, zero-usage stops, and Ollama-style providers, so the reported custom Anthropic-compatible post-tool empty `stop` turn remains a focused runtime gap.

## What This Changes

The issue reports that successful custom `anthropic-messages` tool execution can be followed by an empty final assistant turn, yielding `No reply from agent` instead of a visible answer or safe recovery.

## Best Possible Solution

Add an `anthropic-messages` empty/non-visible final-turn recovery path with focused regression coverage, while preserving side-effect replay safety instead of adding a broad always-reply policy.

## Reproduction Assessment

Yes, at source level. A custom `modelApi: "anthropic-messages"` run with `payloadCount: 0`, a successful prior `toolResult`, and a final assistant `stop` with `content: []` and nonzero usage skips the current retry guard and proceeds toward the incomplete-turn/no-payload path.

## Solution Assessment

Yes, the narrow maintainable fix is to extend the provider-family-aware empty-turn recovery to `anthropic-messages` and add a regression test. A broad fallback that turns successful tool results into user replies would be riskier because it can duplicate or over-disclose tool output.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/46080 on current `main`. The observable bug is that custom providers using `api: "anthropic-messages"` can complete a tool call successfully, then return a final assistant `stop` with `content: []`, leaving the runner with no visible payload and a `No reply from agent` outcome. Keep the fix focused on the embedded runner empty/non-visible final-turn recovery path. Add regression coverage in `src/agents/pi-embedded-runner/run.incomplete-turn.test.ts` for a replay-safe `modelApi: "anthropic-messages"` empty assistant `stop` with nonzero output usage and no payloads, parallel to the existing `openai-completions` case. Update `src/agents/pi-embedded-runner/run/incomplete-turn.ts` so `anthropic-messages` is included in the provider-family retry guard, while preserving existing guards that avoid retries after side effects, tool errors, aborts, timeouts, committed messaging delivery, or unsafe replay metadata. Do not add a new config option, do not implement a global always-reply policy, and do not blindly inline successful tool results as final text. Add a changelog entry if runtime behavior changes.
