## Summary

Keep open. Current main still supports custom Anthropic Messages providers but does not route nonzero empty `stop` final turns through the existing non-visible retry guard, and the issue discussion has multiple matching reports with no open or merged PR owning this exact gap.

## What This Changes

The issue reports that custom `anthropic-messages` providers can complete a tool call but persist an empty final assistant message, producing no user-visible reply.

## Best Possible Solution

Add `anthropic-messages` aware recovery for empty post-tool final turns, with replay-safe retry coverage and a sanitized completed-action fallback when retrying would duplicate side effects.

## Reproduction Assessment

Yes, source-level. A custom `modelApi: "anthropic-messages"` run with `payloadCount: 0`, final assistant `content: []`, `stopReason: "stop"`, and nonzero usage reaches the empty-response resolver but misses both the non-visible retry allowlist and zero-usage fallback.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing incomplete-turn recovery policy for Anthropic Messages and cover unsafe retry fallback behavior, not add a broad channel-level require-reply option.
