## Summary

Keep open. Current main partially improved the area with zero-usage empty-stop handling and a Kimi-specific `anthropic-messages` branch, but the reported non-Kimi custom/proxied `anthropic-messages` path with empty final content and nonzero output usage is still not covered by the non-visible retry guard.

## What This Changes

The issue asks OpenClaw to avoid surfacing `No reply from agent` when an `anthropic-messages` provider records a successful tool result but the final assistant turn is empty.

## Best Possible Solution

Extend the existing replay-safe non-visible retry guard to generic `modelApi === "anthropic-messages"`, add focused non-Kimi regression coverage, and leave broad fallback-reply policy to https://github.com/openclaw/openclaw/issues/60607.

## Reproduction Assessment

Yes, from source inspection rather than live execution. A non-Kimi custom/proxied `anthropic-messages` empty assistant turn with nonzero output usage reaches `resolveEmptyResponseRetryInstruction`, but current main does not return the retry instruction because the guard only covers generic `openai-completions`, Kimi-on-Anthropic, Ollama, and strict/Gemini paths.

## Solution Assessment

Yes. The narrow maintainable fix is the same model-API-based guard shape already used for `openai-completions`, plus regression tests, not a provider-name regex or a global synthesized tool-result reply.
