## Summary

Keep open. Current main still supports custom `api: "anthropic-messages"` providers and the Anthropic tool-result replay path, but the empty final assistant turn is only retried for strict-agentic/Gemini, `openai-completions`, zero-usage stops, or Ollama-style cases; the reported nonzero custom Anthropic-compatible post-tool empty `stop` turn can still fall through to a generic incomplete-turn error instead of a recovered final reply or completed-action acknowledgement.

## What This Changes

The issue requests that OpenClaw recover or synthesize a user-visible reply when a successful Anthropic-compatible tool_result round trip is followed by an empty assistant content array.

## Best Possible Solution

Add a provider-family-aware post-tool empty-turn recovery path for custom `anthropic-messages` runs, with regression coverage that retries replay-safe cases and emits a sanitized completed-action acknowledgement when retrying is unsafe.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction can mock a custom `api: "anthropic-messages"` run where a successful `toolResult` is followed by an assistant `stop` with `content: []`, output usage, and zero payloads; current main reaches the existing incomplete-turn path instead of recovered success output.

## Solution Assessment

No. Current main is safer than a fully silent response because it can surface an error, but it is not the best fix for a successful post-tool round because it loses the distinction between a failed answer and a completed tool action.
