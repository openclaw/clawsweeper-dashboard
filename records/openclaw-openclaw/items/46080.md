## Summary

Keep open. Current main still leaves custom `modelApi: "anthropic-messages"` empty final turns outside the non-visible retry allowlist, while the issue discussion now has multiple matching reports and a recent source-level confirmation of the same gap.

## What This Changes

The issue asks OpenClaw to recover or produce a user-visible reply when a custom `anthropic-messages` provider completes a tool call but returns a final assistant turn with empty content.

## Best Possible Solution

Add `anthropic-messages` aware recovery for empty post-tool final turns, with replay-safe retry coverage and a sanitized completed-action fallback when retrying is unsafe.

## Reproduction Assessment

Yes, source-level. A custom `modelApi: "anthropic-messages"` attempt with `payloadCount: 0`, final assistant `content: []`, `stopReason: "stop"`, and nonzero usage reaches the empty-response resolver but misses both the non-visible retry guard and zero-usage fallback.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing incomplete-turn recovery policy for `anthropic-messages` and cover the unsafe-retry fallback, not add a broad channel-level require-reply option.
