## Summary

Keep open. Current main still supports custom `anthropic-messages` providers, but the empty non-visible retry guard only covers strict-agentic/Gemini, `openai-completions`, zero-usage empty stops, and Ollama-style cases, so the reported custom Anthropic-compatible post-tool empty `stop` turn remains actionable.

## What This Changes

The issue asks OpenClaw to recover or surface a reply when a custom `anthropic-messages` provider produces a successful tool result followed by an empty final assistant turn.

## Best Possible Solution

Extend the existing non-visible empty-turn recovery to custom `anthropic-messages` runs, with replay-safe retry behavior and a sanitized completion acknowledgement when retrying is unsafe.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: mock a custom `api: "anthropic-messages"` run where a successful `toolResult` is followed by assistant `stop`, `content: []`, nonzero usage, and zero terminal payloads; current main does not qualify that shape for the empty-response retry guard.

## Solution Assessment

No. Current main can surface a generic incomplete-response error, but it is not the best fix for a successful post-tool round because it loses the distinction between failed answer generation and completed tool work.
