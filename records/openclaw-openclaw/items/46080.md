## Summary

Keep open. Current main has adjacent empty-turn safeguards, but the reported custom `anthropic-messages` post-tool success case still lacks a success recovery path: a nonzero empty final `stop` turn is not retried for custom Anthropic-compatible providers, and successful tool results are not used to synthesize a visible completion.

## What This Changes

The issue requests a fix for custom `anthropic-messages` tool rounds that execute successfully but persist an empty final assistant turn, leaving the user with no visible reply.

## Best Possible Solution

Guarantee a visible terminal outcome for successful Anthropic-compatible tool rounds: preserve provider final text when present, retry only when replay-safe, and otherwise return a sanitized completion acknowledgement from the latest successful tool result without rerunning side effects or exposing raw tool payloads.

## Reproduction Assessment

Yes. A mocked embedded-runner attempt can record a successful tool result, then return a custom `anthropic-messages` assistant `stop` with `content: []`, `payloadCount: 0`, and nonzero usage; current source flow skips the empty-response retry and reaches the incomplete-turn error path.

## Solution Assessment

No. Current main's generic empty-turn guard prevents some silent completions, but the narrow maintainable fix is a post-tool success recovery path for Anthropic-compatible tool rounds rather than treating successful tool execution as a generation error.
