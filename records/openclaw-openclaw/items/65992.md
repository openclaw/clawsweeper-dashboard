## Summary

Keep open. Current main still uses slug-only local source page IDs and paths, so same-slug source ingests can overwrite the same `sources/<slug>.md` page; this PR is a focused implementation that addresses that remaining behavior with no blocking review finding found.

## What This Changes

The PR changes memory-wiki source ingest to generate new source page identities from a readable slug plus a 12-hex canonical-path hash, reuse matching legacy source pages, and add regression coverage for collisions, symlink aliases, fallback titles, and lookup scope.

## Best Possible Solution

Land the focused memory-wiki plugin fix after maintainer review, rebase, and green CI, keeping the collision, legacy-reuse, and symlink regression tests intact; handle any broader source identity index or performance redesign as a separate follow-up.

## Reproduction Assessment

Yes. The current-main code path is enough to reproduce: ingesting two local files whose titles normalize to the same slug targets the same `source.<slug>` ID and `sources/<slug>.md` path, and the second write overwrites the first.

## Solution Assessment

Yes. A path-derived suffix plus legacy source-page reuse is the narrowest maintainable fix for same-slug overwrites; Unicode-aware slugging alone would not protect identical titles or different titles that still normalize to the same slug.
