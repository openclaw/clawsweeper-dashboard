## Summary

Keep open: current main still uses slug-only source IDs/paths, while this PR provides a focused collision-resistant memory-wiki ingest fix but still needs a changelog entry and fresh validation before merge.

## What This Changes

The PR changes memory-wiki source ingest to derive new source page IDs/paths from a readable slug plus canonical-path hash, preserve matching legacy source pages, and add regression coverage for collisions, symlinks, fallback titles, and lookup scope.

## Best Possible Solution

Land the focused memory-wiki plugin hardening after adding the changelog entry, refreshing against current main, and rerunning memory-wiki plus changed validation; leave broader source identity indexing or performance redesign as a separate follow-up.

## Reproduction Assessment

Yes. From current-main source, ingesting two local files whose titles normalize to the same slug computes the same `source.<slug>` ID and `sources/<slug>.md` path, then writes the second file to the same target.

## Solution Assessment

Yes for the behavior. A canonical-path hash suffix with legacy source-page reuse is a narrow maintainable fix for same-slug overwrites; the remaining blocker is release-note and refresh work, not a different design.
