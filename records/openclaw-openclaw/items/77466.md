## Summary

Keep this PR open, but it is not mergeable as-is: the proposed coordination command path is source-reproducibly broken, embeds developer-local approval constants in a security-sensitive execution surface, lacks real behavior proof, and carries stale release metadata plus broad unrelated churn.

## What This Changes

The branch adds an `agent-exec` command, coordination-only plugin/provider suppression plumbing, proof/watchdog harnesses and tests, plus broad release/docs/plugin/runtime updates.

## Best Possible Solution

Split and rebase this into a narrow portable `agent-exec` design that registers every generated option, derives policy and approval from validated job/config data, removes local constants, and includes explicit maintainer security review plus real runtime proof.

## Reproduction Assessment

Yes, for a PR-introduced defect by source inspection: the renderer emits `agent-exec --tool-policy coordination_only`, the command does not register that option, and Commander 14 errors on unknown options. I did not execute the command because this was a read-only review.

## Solution Assessment

No. The direction may be useful, but this patch is not the best current solution because it mixes a new execution/trust boundary with developer-local constants, stale release metadata, and an unusable generated command path.
