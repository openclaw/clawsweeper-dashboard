## Summary

Keep this PR open: the proposed coordination path is useful source work, but the patch currently introduces an unusable generated `agent-exec` invocation, a developer-local authorization boundary, and stale release metadata relative to current main.

## What This Changes

The PR adds an `agent-exec` coordination command, coordination-only plugin/provider suppression plumbing, proof harness scripts, runtime forwarding, and broad regression coverage.

## Best Possible Solution

Rebase and split this into a narrow, portable `agent-exec` design with registered CLI options, validated job/config-derived coordination policy, no developer-local constants, and no unrelated release metadata changes.

## Reproduction Assessment

Yes for the main PR defect: source inspection shows the generated command passes `--tool-policy`, but the proposed `agent-exec` command does not register that option, so Commander would reject the generated invocation before the action runs.

## Solution Assessment

No. The direction may be useful, but this patch is not the best current solution because it adds a new execution surface with a non-portable trust boundary and stale release metadata mixed into the branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Register the generated tool-policy flag:** `src/cli/program/register.agent.ts:97`
  - body: The coordination renderer generates `agent-exec ... --tool-policy coordination_only`, and preaction reads that flag, but this command never declares `--tool-policy`. Commander rejects unknown options before the action runs, so the generated proof command cannot reach `agentExecCommand`.
  - confidence: 0.9
- **[P1] Remove developer-local coordination constants:** `src/agents/coordination/job-contract.ts:4-15`
  - body: The new job contract hard-codes one developer's repo root, job root, entrypoint path, allowed agent, and approver. That makes validation fail for normal OpenClaw checkouts and bakes a personal approval model into core; these values need to come from validated job/config inputs or test fixtures.
  - confidence: 0.88
- **[P2] Drop the stale version downgrade:** `package.json:3`
  - body: This branch changes the root package version to `2026.4.21` while current main is already `2026.5.4`. Merging this release metadata would roll the package/app/schema version line backward; rebase and remove unrelated release prep from the PR.
  - confidence: 0.86
