## Summary

Keep open: current main does not implement the proposed `agent-exec` surface, and the PR branch has blocking source-proven defects in its generated CLI path, coordination trust boundary, and stale release metadata.

## What This Changes

Adds a new `agent-exec` coordination execution path, coordination-only plugin/provider suppression plumbing, a proof harness, and related regression coverage.

## Best Possible Solution

Rebase and split this into a portable `agent-exec` design that registers every generated CLI option, derives coordination trust from validated job/config inputs, and keeps unrelated release/backport drift out of the branch.

## Reproduction Assessment

Yes for the main PR defect: source inspection shows the generated command passes `--tool-policy`, while the proposed `agent-exec` command does not register that option, so Commander would reject the generated invocation before the action runs.

## Solution Assessment

No. The direction may be useful, but this patch is not the best current solution because it mixes stale release changes with a non-portable coordination trust boundary and an unusable generated command path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Register the tool-policy flag used by agent-exec:** `src/cli/program/register.agent.ts:97`
  - body: The coordination renderer builds `agent-exec ... --tool-policy coordination_only`, but this command never declares `--tool-policy`. Commander rejects unknown options before the action runs, so the generated proof command cannot reach `agentExecCommand`.
  - confidence: 0.9
- **[P1] Remove developer-local coordination constants:** `src/agents/coordination/job-contract.ts:4-15`
  - body: The job contract hard-codes one developer's repo root, job root, entrypoint path, allowed agent, approver, and approval statement. That makes validation fail for normal OpenClaw checkouts and bakes a personal approval model into core.
  - confidence: 0.88
- **[P2] Drop the stale version downgrade:** `package.json:3`
  - body: The branch changes the root package version to `2026.4.21` while current main is already `2026.5.4`. Merging this release metadata would roll the package/app/schema version line backward and corrupt release artifacts.
  - confidence: 0.86
