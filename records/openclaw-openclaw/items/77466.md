## Summary

Keep this PR open for maintainer review, but it should not merge as-is: the branch adds a broad coordination execution surface, the generated command path is unusable because it passes an unregistered `--tool-policy` flag, the job contract hard-codes developer-local trust constants, release metadata is stale against current main, and external real behavior proof is missing.

## What This Changes

The branch adds a structured `agent-exec` coordination execution path, coordination-only plugin/provider suppression plumbing, proof harness scripts, and broad adjacent runtime/test/release changes.

## Best Possible Solution

Rebase and split this into a narrow, portable `agent-exec` design that registers every generated CLI option, derives coordination policy from validated job/config data, removes developer-local trust constants, and keeps unrelated release/backport churn out of the branch.

## Reproduction Assessment

Yes for the main PR defect: source inspection shows the generated proof command passes `--tool-policy coordination_only`, while the proposed `agent-exec` command does not register that option. Commander 14 reports unknown options as errors unless explicitly configured otherwise.

## Solution Assessment

No. The direction may be useful, but this patch is not the best current solution because it mixes a new execution/trust boundary with stale release metadata and an unusable generated command path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Register the tool-policy flag used by agent-exec:** `src/cli/program/register.agent.ts:92-97`
  - body: The coordination renderer generates `agent-exec ... --tool-policy coordination_only`, and the preaction plumbing relies on that policy to suppress plugin/provider loading, but this command never declares `--tool-policy`. Commander rejects unknown options before the action runs, so the generated proof command cannot reach `agentExecCommand`.
  - confidence: 0.91
- **[P1] Remove developer-local coordination constants:** `src/agents/coordination/job-contract.ts:4-15`
  - body: The new job contract hard-codes one developer's repo root, job root, entrypoint path, allowed agent, approval mode, and approver into core. That makes validation fail for normal OpenClaw checkouts and bakes a personal approval model into a safety-sensitive execution path; these values need to come from validated job/config inputs or test fixtures.
  - confidence: 0.9
- **[P2] Drop the stale release version change:** `package.json:3`
  - body: This branch sets the root package version to `2026.4.21`, while current main is already on the `2026.5.12-beta.1` line. Merging or resolving this branch by taking its release metadata would roll package/app/schema versioning backward, so the coordination work needs to be rebased and separated from release backport churn.
  - confidence: 0.86
