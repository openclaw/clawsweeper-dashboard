## Summary

Keep this PR open for maintainer review, but the current patch is not mergeable as-is: the central generated command path passes an unregistered flag, the coordination contract is pinned to one developer-local environment, and the branch carries stale release metadata that conflicts with current main.

## What This Changes

The PR adds a structured `agent-exec` coordination path, coordination-only plugin/provider suppression plumbing, proof harness scripts, and related regression coverage.

## Best Possible Solution

Rebase and split the work into a narrow, portable `agent-exec` design that registers every generated CLI option, derives coordination policy from validated job/config data, removes developer-local constants, and keeps unrelated release/backport changes out of the branch.

## Reproduction Assessment

Yes for the main PR defect: source inspection shows the generated command passes `--tool-policy` while the proposed `agent-exec` command does not register that option, so Commander would reject the generated invocation before the action runs.

## Solution Assessment

No. The direction may be useful, but this patch is not the best current solution because it mixes stale release changes with a non-portable coordination trust boundary and an unusable generated command path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Register the tool-policy flag used by agent-exec:** `src/cli/program/register.agent.ts:97`
  - body: The coordination renderer generates `agent-exec ... --tool-policy coordination_only`, and preaction reads that flag to decide whether plugin loading should be skipped, but this command never declares `--tool-policy`. Commander rejects unknown options before the action runs, so the generated proof command cannot reach `agentExecCommand`.
  - confidence: 0.9
- **[P1] Remove developer-local coordination constants:** `src/agents/coordination/job-contract.ts:5-13`
  - body: The new job contract hard-codes one developer's repo root, job root, entrypoint path, allowed agent, and approver. That makes validation fail for normal OpenClaw checkouts and bakes a personal approval model into core; these values need to come from validated job/config inputs or test fixtures.
  - confidence: 0.88
- **[P2] Drop the stale version downgrade:** `package.json:3`
  - body: This branch changes the root package version to `2026.4.21` while current main is already `2026.5.4`. Merging this release metadata would roll the package/app/schema version line backward and corrupt release artifacts; rebase and remove unrelated release prep from the PR.
  - confidence: 0.86
