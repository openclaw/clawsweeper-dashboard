## Summary

Keep open: current main does not implement `agent-exec`, and this PR is not mergeable as-is because the proposed command path is unusable, the coordination contract hard-codes developer-local trust data, and the branch carries stale release metadata.

## What This Changes

The PR adds a new `agent-exec` coordination command, threads command/tool-policy context through agent, plugin, and provider runtime paths, and adds proof harness scripts/tests alongside release/backport changes.

## Best Possible Solution

Rebase and split this into a portable `agent-exec` design that declares every generated CLI option, derives authorization and paths from validated job/config data, keeps plugin/provider suppression behind a narrow reviewed seam, and removes unrelated release metadata.

## Reproduction Assessment

Yes for the main PR defect: source/diff inspection shows the generated command passes `--tool-policy coordination_only` while the proposed `agent-exec` command does not register that option. Current main has no `agent-exec` surface, so the requested behavior is not an existing-main bug reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the best solution because it mixes a new execution surface with hard-coded local authorization data and stale release metadata.
