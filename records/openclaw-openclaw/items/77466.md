## Summary

Keep this PR open for maintainer handling, but do not merge as-is: current main does not have the proposed `agent-exec`/`coordination_only` surface, and the branch still has source-proven correctness, portability, security, and proof blockers.

## What This Changes

The branch adds an `agent-exec` coordination command, coordination-only plugin/provider suppression plumbing, proof/watchdog harnesses, regression tests, and unrelated release/docs/plugin/runtime churn.

## Best Possible Solution

Rebase and split this into a narrow portable `agent-exec` design that registers every generated option, derives policy and approval from validated job/config data, removes local constants, and gets explicit maintainer security review plus real runtime proof.

## Reproduction Assessment

Yes for a PR-introduced defect: source inspection shows the rendered command passes `agent-exec --tool-policy coordination_only`, while the proposed command does not register that option. Commander 14 treats unknown options as errors unless unknown options are explicitly allowed.

## Solution Assessment

No. The direction may be useful, but this patch is not the best current solution because it mixes a new execution/trust boundary with developer-local constants, stale release metadata, and an unusable generated command path.
