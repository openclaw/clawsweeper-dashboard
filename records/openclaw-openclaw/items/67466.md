## Summary

Keep open: the PR contains useful Feishu timeout/status work, but the proposed monitor lifecycle change breaks the gateway startAccount contract and the remaining latency work overlaps with open Feishu timeout/serialization reports.

## What This Changes

The PR parallelizes Feishu startup bot-identity probes, caps startup and setup/status probes at 3000ms, starts account monitors in the background, and updates Feishu startup/status tests.

## Best Possible Solution

Keep the Feishu startAccount promise tied to monitorSingleAccount, add bounded/degraded status probing, and address startup bot-info latency with a lifecycle-safe timeout or serialization queue.

## Reproduction Assessment

Yes for the PR regression: current gateway code and tests show an immediately resolved startAccount is treated as a channel exit and restart loop. The original live Feishu timing was not rerun, but current source confirms the long startup/status probe paths are plausible.

## Solution Assessment

No. The 3000ms status cap is useful, but detaching monitorFeishuProvider({ accountId }) from the monitor lifetime is not maintainable; a safer fix belongs in the probe timeout/queue layer while preserving gateway lifecycle ownership.

## Review Findings

Overall correctness: patch is incorrect

  - body: The accountId path is what Feishu's gateway startAccount returns.   - confidence: 0.86
