## Summary

Keep open: the Feishu startup/status latency problem is still meaningful, but this PR is not merge-ready because it detaches the gateway-managed Feishu monitor lifetime, drops task-scoped channelRuntime plumbing, logs failed probe accounts as unknown, and still lacks after-fix real Feishu proof.

## What This Changes

The PR parallelizes Feishu multi-account startup probes, adds 3,000ms startup/status probe caps, backgrounds monitor startup, and rewrites Feishu startup/status tests.

## Best Possible Solution

Preserve the long-lived single-account startAccount contract and task-scoped channelRuntime, make the timeout cap an explicitly proven Feishu compatibility choice, and require redacted real Feishu startup/status proof before merge.

## Reproduction Assessment

Yes for the PR-introduced lifecycle bug: source inspection shows Feishu startAccount returns monitorFeishuProvider with accountId, while the PR makes that branch return immediately and gateway code treats returned-promise resolution as channel exit. No high-confidence after-fix reproduction is available for the claimed latency improvement because the PR provides before-only numbers.

## Solution Assessment

No. Bounding startup/status probes is plausible, but this implementation is not the best fix because it detaches monitor lifetime and changes timeout policy without real Feishu proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep the Feishu account task tied to startAccount:** `extensions/feishu/src/monitor.ts:87-93`
  - body: `channel.ts` returns `monitorFeishuProvider({ accountId })` from the gateway `startAccount` hook, and `server-channels.ts` treats that returned promise as the account task: when it resolves, the gateway records a channel exit and schedules restart attempts. This branch now starts the real monitor in the background and returns immediately, so every gateway-managed Feishu account can enter restart loops while the real monitor is detached from tracked abort/status state.
  - confidence: 0.94
- **[P2] Pass channelRuntime through the background monitor helper:** `extensions/feishu/src/monitor.ts:55-60`
  - body: The helper calls `monitorSingleAccount` without `channelRuntime`, even though current gateway startup provides a task-scoped channel runtime and current main forwards it into `monitorSingleAccount`. Without this, Feishu handlers fall back to the global runtime instead of the task-scoped surface used for gateway lifecycle and runtime context cleanup.
  - confidence: 0.8
- **[P2] Preserve the account id when a startup probe rejects:** `extensions/feishu/src/monitor.ts:122-124`
  - body: The rejected branch of `Promise.allSettled` only has the thrown error, not the account captured by the map callback, so the warning logs `feishu[unknown]` and gives operators no account to fix. Catch inside the per-account async callback and return a degraded result with that account instead.
  - confidence: 0.82
