## Label Justifications

- `P1`: The PR targets a real Feishu startup regression but currently introduces a high-impact gateway lifecycle failure for Feishu users.
- `merge-risk: 🚨 compatibility`: The branch changes the gateway startAccount lifetime contract and is dirty against current main.

- `merge-risk: 🚨 availability`: Returning immediately from startAccount can make the gateway restart Feishu repeatedly and eventually stop monitoring.

## Summary

Keep open: the Feishu startup latency goal is valid, but this patch breaks the normal gateway-managed Feishu account lifecycle, reverses intentional probe sequencing, is dirty against main, and still lacks after-fix real behavior proof.

## What This Changes

The branch caps Feishu startup/status probes at 3 seconds, parallelizes multi-account bot-identity probes, detaches Feishu monitors into background tasks, and updates related tests.

## Best Possible Solution

Keep the gateway-owned monitor promise long-lived, use a Feishu-owned bounded or queued startup probe strategy, retain the status timeout only after rebase, and prove the result with redacted live Feishu startup logs or a recording.

## Reproduction Assessment

Yes for the PR regression at source level: current Feishu gateway startAccount passes accountId to monitorFeishuProvider, and server-channels treats an immediately resolved startAccount promise as a channel exit. The original live 100s Feishu startup delay was not rerun in this read-only review.

## Solution Assessment

No. The setup/status timeout cap is useful, but detaching account monitors and unbounded probe fan-out are not the narrowest maintainable fix; the safer direction keeps monitor ownership tracked and uses bounded or queued probes.
