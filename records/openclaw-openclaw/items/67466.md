## Label Justifications

- `P1`: This PR affects a real Feishu channel startup workflow and currently introduces a gateway monitor lifecycle regression that can stop monitoring.
- `merge-risk: 🚨 availability`: The changed accountId path can resolve immediately and trigger repeated gateway restart handling for Feishu accounts.
- `merge-risk: 🚨 message-delivery`: Detached or exhausted Feishu monitors can leave incoming Feishu messages unobserved even while background tasks are no longer tracked correctly.

## Summary

Keep open: the PR still has a blocking Feishu gateway lifecycle regression, reverses the current sequential startup-probe safeguard, lacks after-fix real Feishu proof, and overlaps the still-open startup serialization work.

## What This Changes

The PR caps Feishu startup/status probes at 3 seconds, parallelizes multi-account identity probes, detaches Feishu monitors into background tasks, and updates Feishu startup/status tests.

## Best Possible Solution

Keep the gateway-owned monitor promise long-lived, use a Feishu-owned bounded or queued startup-probe policy, keep the useful status timeout cap if maintainers accept it, and reconcile with https://github.com/openclaw/openclaw/pull/63481 before merge.

## Reproduction Assessment

Yes for the PR regression by source inspection: Feishu gateway.startAccount uses monitorFeishuProvider with accountId, and server-channels treats a resolved startAccount promise as a channel exit that restarts. The original live 100-second Feishu startup delay was not rerun in this read-only review.

## Solution Assessment

No. The status timeout cap is useful, but fire-and-forget account monitors and unbounded parallel startup probes are not the narrowest maintainable fix; the safer path preserves gateway lifecycle ownership and uses bounded or queued Feishu probes.
