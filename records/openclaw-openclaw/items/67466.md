## Summary

Keep this PR open, but it still needs changes before merge: it targets a real Feishu startup/status latency problem, yet the current diff detaches the gateway-managed account monitor promise, reintroduces parallel probe bursts, and lacks after-fix real behavior proof.

## What This Changes

The PR parallelizes Feishu startup identity probes, caps startup/status probe timeouts at 3000ms, fire-and-forgets account monitors, and updates Feishu startup/status tests.

## Best Possible Solution

Keep Feishu gateway account startup tied to the long-lived monitor, bound or degrade startup/status probes without endpoint bursts, and reconcile the final startup-probe strategy with https://github.com/openclaw/openclaw/pull/63481.

## Reproduction Assessment

Yes for the PR regression: source inspection shows Feishu gateway startAccount uses the accountId branch, and server-channels treats an immediately resolved startAccount as an exited channel that restarts. The original live 100s Feishu startup latency was not rerun.

## Solution Assessment

No. The status timeout cap is useful, but fire-and-forget account monitors and parallel startup probes violate current lifecycle and rate-limit protections; the safer path is bounded/degraded probes plus a shared queue while preserving monitor ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep startAccount tied to the Feishu monitor lifetime:** `extensions/feishu/src/monitor.ts:87-93`
  - body: Feishu gateway startAccount selects this accountId branch. Returning immediately after launching monitorSingleAccount makes server-channels treat the account as exited, set running false, and schedule restarts while the real monitor is detached from tracked task and abort state.
  - confidence: 0.95
- **[P2] Preserve serialized startup probes:** `extensions/feishu/src/monitor.ts:104-111`
  - body: This Promise.allSettled starts all account identity probes at once, reversing the current sequential preflight that was added to avoid bursting Feishu's bot-info endpoint. That can reintroduce the multi-account timeout behavior described by related Feishu startup reports.
  - confidence: 0.86
- **[P2] Log failed probe warnings with the account id:** `extensions/feishu/src/monitor.ts:122-131`
  - body: A rejected allSettled result only carries the reason, so this branch logs feishu[unknown] for unexpected probe failures. Catch inside the per-account callback while account.accountId is still in scope and return a degraded identity for that account.
  - confidence: 0.84
