## Summary

Keep open: the branch has a useful Feishu status-probe timeout cap, but it is not mergeable as-is because it detaches the gateway account monitor lifetime, reverses current startup probe serialization, lacks after-fix real behavior proof, and overlaps with the open serialization fix path.

## What This Changes

The PR caps Feishu startup/status probes at 3 seconds, parallelizes multi-account identity probes, detaches Feishu monitors into background tasks, and updates Feishu startup/status tests.

## Best Possible Solution

Preserve the gateway-owned monitor promise, keep Feishu startup probes bounded without unbounded fan-out, salvage the setup-status timeout cap, and reconcile with the queue-based serialization work before merge.

## Reproduction Assessment

Yes for the PR regression: current source shows Feishu gateway startAccount uses the accountId branch, and server-channels treats an immediately resolved startAccount promise as a channel exit that restarts. The original live 100s Feishu startup delay was not rerun in this read-only review.

## Solution Assessment

No. The status timeout cap is useful, but detached monitors and parallel startup probes are not the narrowest maintainable fix; a safer solution keeps monitor ownership tracked and uses bounded or queued probes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep startAccount tied to monitor lifetime:** `extensions/feishu/src/monitor.ts:87-93`
  - body: Feishu gateway startAccount calls monitorFeishuProvider with accountId. Returning after a fire-and-forget monitorSingleAccount makes server-channels treat the account as exited, mark running false, and schedule restarts while the real monitor is detached from task and abort tracking.
  - confidence: 0.95
- **[P2] Preserve Feishu startup probe serialization:** `extensions/feishu/src/monitor.ts:104-111`
  - body: This Promise.allSettled starts all bot-info probes concurrently, removing the current sequential preflight that avoids bursting Feishu's endpoint and conflicting with the related multi-account timeout reports. Use a shared queue or bounded/degraded approach instead.
  - confidence: 0.88
- **[P2] Keep account context when startup probes fail:** `extensions/feishu/src/monitor.ts:122-131`
  - body: A rejected allSettled result only carries a reason, so this branch logs feishu[unknown] and skips that account rather than degrading it with its account id in scope. Catch inside the per-account callback and return a degraded identity for that account.
  - confidence: 0.86
