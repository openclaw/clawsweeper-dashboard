## Summary

This PR should stay open because it targets a real Feishu startup/status latency problem, but the branch still breaks the Feishu gateway lifecycle by detaching the account monitor promise, conflicts with the linked multi-account serialization work, and lacks after-fix real behavior proof.

## What This Changes

The PR caps Feishu status/startup probes at 3 seconds, parallelizes multi-account startup identity probes, detaches Feishu account monitors into background tasks, and updates Feishu startup/status tests.

## Best Possible Solution

Keep the useful status probe timeout direction, preserve gateway-owned monitor lifetimes, and solve slow Feishu startup with a bounded or serialized startup-probe strategy that aligns with the linked Feishu timeout reports.

## Reproduction Assessment

Yes for the PR regression: current source shows Feishu startAccount uses monitorFeishuProvider({ accountId }), and gateway task handling restarts accounts when that promise resolves. The original live Feishu 100 second startup latency was not rerun here.

## Solution Assessment

No. Capping status probes is directionally useful, but detaching account monitors and parallelizing startup probes are not the narrowest maintainable fix because they break gateway lifecycle tracking and conflict with the linked serialization reports.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep startAccount tied to the Feishu monitor lifetime:** `extensions/feishu/src/monitor.ts:87-93`
  - body: Feishu gateway startAccount calls monitorFeishuProvider with accountId, and server-channels treats a resolved startAccount promise as channel exit. Returning immediately after a background monitor call makes the gateway mark the account stopped and schedule restarts while the real monitor is detached from task and abort tracking.
  - confidence: 0.96
- **[P2] Do not parallelize the startup identity probes:** `extensions/feishu/src/monitor.ts:104-111`
  - body: This Promise.allSettled starts all Feishu bot identity probes concurrently, which conflicts with the linked multi-account timeout reports where concurrent bot-info and tenant-token requests are the suspected failure mode. A bounded or serialized queue is safer than increasing startup burstiness.
  - confidence: 0.88
- **[P2] Log failed startup probes with the account id:** `extensions/feishu/src/monitor.ts:122-131`
  - body: A rejected allSettled result only exposes the reason, so this branch can log feishu[unknown] for unexpected probe failures. Catch inside the per-account callback while account.accountId is still in scope, then return a degraded identity for that account.
  - confidence: 0.86
