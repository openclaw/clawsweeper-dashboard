## Label Justifications

- `P1`: The PR targets a real Feishu startup bottleneck but currently introduces a high-impact gateway lifecycle failure for Feishu users.
- `merge-risk: 🚨 compatibility`: The branch changes the implicit startAccount lifetime contract that current gateway lifecycle tracking depends on.
- `merge-risk: 🚨 message-delivery`: Detached or exhausted Feishu monitors can stop receiving channel events and lose user-visible Feishu message handling.
- `merge-risk: 🚨 availability`: Returning immediately from startAccount can make the gateway repeatedly restart Feishu accounts and eventually stop monitoring.

## Summary

Keep open: the Feishu startup latency target is valid, but this branch is not merge-ready because it breaks the gateway-owned monitor lifetime, reverses current Feishu probe sequencing, lacks after-fix real Feishu proof, and conflicts with overlapping Feishu startup work.

## What This Changes

The PR caps Feishu startup/status probes at 3 seconds, parallelizes multi-account bot-identity probes, detaches Feishu monitors into background tasks, and updates Feishu startup/status tests.

## Best Possible Solution

Keep the gateway-owned monitor promise long-lived, use a Feishu-owned bounded or queued startup probe strategy, retain the status timeout only after rebase, and require redacted live Feishu startup proof.

## Reproduction Assessment

Yes for the PR regression at source level: current Feishu gateway startAccount passes accountId to monitorFeishuProvider, and server-channels treats an immediately resolved startAccount promise as channel exit. The original live 100s Feishu startup delay was not rerun in this read-only review.

## Solution Assessment

No. The status timeout cap may be useful, but detaching account monitors and unbounded probe fan-out are not the narrowest maintainable fix; the safer direction keeps monitor ownership tracked and uses bounded or queued probes.
