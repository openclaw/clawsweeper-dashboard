## Summary

Keep open. The PR targets a real Feishu startup-latency problem, but the current patch breaks the gateway startAccount lifecycle, reverses existing sequential startup-probe protection, and still lacks after-fix real Feishu proof.

## What This Changes

The PR caps Feishu startup/status probes at 3 seconds, parallelizes multi-account identity probes, starts Feishu monitors in background tasks, and updates Feishu startup/status tests.

## Best Possible Solution

Keep the Feishu monitor promise tied to the gateway task, bound slow status/startup probes in Feishu-owned code, and reconcile the final startup-probe strategy with the open serialization work before merge.

## Reproduction Assessment

Yes for the PR regression: current source shows Feishu gateway startAccount uses the accountId branch, and server-channels treats an immediately resolved startAccount as channel exit that restarts. The original live 100s Feishu startup latency was not rerun.

## Solution Assessment

No. The status timeout cap is useful, but detached monitors and parallel startup probes break lifecycle and rate-limit protections; the safer fix preserves monitor ownership and uses bounded or queued probes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep startAccount tied to the Feishu monitor lifetime:** `extensions/feishu/src/monitor.ts:87-93`
  - body: Feishu gateway startAccount calls monitorFeishuProvider with accountId. Returning after fire-and-forget monitorSingleAccount makes server-channels treat the account as exited, mark running false, and schedule restarts while the real monitor is detached from task and abort tracking.
  - confidence: 0.95
- **[P2] Preserve startup probe serialization:** `extensions/feishu/src/monitor.ts:104-111`
  - body: This Promise.allSettled starts all bot-info probes concurrently, undoing the current sequential preflight that avoids bursting Feishu's endpoint and conflicting with the related multi-account timeout reports. Keep a shared queue or bounded/degraded approach instead.
  - confidence: 0.88
- **[P2] Log failed probes with the account id:** `extensions/feishu/src/monitor.ts:122-131`
  - body: A rejected allSettled result only has a reason, so this branch logs feishu[unknown] for unexpected probe failures. Catch inside the per-account callback and return a degraded identity while account.accountId is still in scope.
  - confidence: 0.86
