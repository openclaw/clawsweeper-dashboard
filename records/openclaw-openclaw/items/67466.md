## Summary

Keep open: the Feishu startup/status latency problem is real and current main still lacks part of the proposed timeout cap, but this PR is not merge-ready because the account-specific gateway path now detaches the monitor lifetime and returns immediately. Source review confirms a P1 lifecycle regression, the branch is conflicting with current main, and the PR still lacks after-fix real Feishu behavior proof.

## What This Changes

The branch caps Feishu startup/status probes at 3000 ms, parallelizes multi-account probes, and changes Feishu monitor startup to launch account monitors in background tasks.

## Best Possible Solution

Preserve the gateway-managed single-account monitor promise, bound only the Feishu probe/status work that can safely degrade without detaching channel lifetime, and land it with focused lifecycle tests plus real Feishu startup proof.

## Reproduction Assessment

Yes for the PR-introduced lifecycle bug: source inspection shows Feishu `startAccount` returns `monitorFeishuProvider({ accountId })`, the PR returns from that branch immediately, and the gateway treats task settlement as channel exit/restart. No live proof establishes the claimed after-fix latency improvement.

## Solution Assessment

No. Bounding Feishu startup/status probes is a plausible direction, but this implementation is not the best fix because it detaches the gateway-managed monitor lifetime and changes timeout policy without real upgrade proof.
