## Summary

Keep open; this is a plausible Feishu startup-latency fix, but the current diff breaks the gateway monitor lifecycle, reverses intentional sequential probe behavior, and lacks after-fix real behavior proof.

## What This Changes

The PR caps Feishu startup/status probes at 3 seconds, parallelizes multi-account identity probes, detaches Feishu monitors into background tasks, and updates Feishu startup/status tests.

## Best Possible Solution

Keep gateway-owned monitor promises long-lived, bound Feishu status/startup probes without unbounded API fan-out, and reconcile the timeout work with the queue-based serialization PR before merge.

## Reproduction Assessment

Yes, at source level for the PR regression: Feishu gateway startAccount uses monitorFeishuProvider with accountId, and server-channels treats an immediately resolved startAccount promise as a channel exit that restarts. The original live 100s Feishu startup delay was not rerun in this read-only review.

## Solution Assessment

No. The status timeout cap may be useful, but detaching monitors and starting unbounded parallel startup probes is not the narrowest maintainable fix; a safer path keeps lifecycle tracking intact and uses bounded or queued probe behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep startAccount tied to monitor lifetime:** `extensions/feishu/src/monitor.ts:87-93`
  - body: Feishu gateway startAccount calls monitorFeishuProvider with accountId. Returning after a fire-and-forget monitorSingleAccount makes server-channels treat the account as exited, mark it not running, and schedule restarts while the real monitor is detached from task and abort tracking.
  - confidence: 0.95
- **[P2] Preserve Feishu startup probe serialization:** `extensions/feishu/src/monitor.ts:104-111`
  - body: This Promise.allSettled starts all bot-info probes concurrently, removing the current sequential preflight that avoids bursting Feishu's endpoint and conflicting with the related multi-account timeout reports. Use a shared queue or bounded/degraded approach instead.
  - confidence: 0.88
- **[P2] Keep account context when startup probes fail:** `extensions/feishu/src/monitor.ts:122-131`
  - body: A rejected allSettled result only carries a reason, so this branch logs feishu[unknown] and skips that account rather than degrading it with its account id in scope. Catch inside the per-account callback and return a degraded identity for that account.
  - confidence: 0.86
