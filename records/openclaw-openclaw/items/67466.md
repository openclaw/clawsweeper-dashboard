## Label Justifications

- `P1`: The PR targets a real Feishu startup regression but currently introduces a high-impact gateway lifecycle failure for Feishu users.
- `merge-risk: 🚨 compatibility`: The branch changes the gateway startAccount lifetime contract and is conflicting against current main.
- `merge-risk: 🚨 message-delivery`: Detached or exhausted Feishu monitors can stop receiving channel events and break user-visible Feishu message handling.
- `merge-risk: 🚨 availability`: Returning immediately from startAccount can make the gateway restart Feishu repeatedly and eventually stop monitoring.

## Summary

Keep this PR open, but the current patch is not merge-ready: it detaches the Feishu account monitor from the gateway lifecycle, removes the current sequential startup preflight, and still lacks after-fix real Feishu proof.

## What This Changes

The branch caps Feishu startup/status probes at 3 seconds, parallelizes multi-account bot-identity probes, detaches Feishu monitors into background tasks, and updates Feishu startup/status tests.

## Best Possible Solution

Keep the gateway-owned monitor promise long-lived, use a Feishu-owned bounded or queued startup probe strategy, preserve any useful status timeout cap after rebase, and prove the result with redacted live Feishu startup evidence.

## Reproduction Assessment

Yes for the PR regression at source level: current Feishu gateway startAccount passes accountId to monitorFeishuProvider, and server-channels treats an immediately resolved startAccount promise as a channel exit. The original live 100-second Feishu startup delay was not rerun in this read-only review.

## Solution Assessment

No. The setup/status timeout cap may be useful, but detaching account monitors and unbounded probe fan-out are not the narrowest maintainable fix; the safer direction keeps monitor ownership tracked and uses bounded or queued probes.
