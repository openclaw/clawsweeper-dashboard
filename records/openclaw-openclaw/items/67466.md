## Summary

Keep open for human review: the Feishu latency problem is real, but this PR introduces a gateway lifecycle regression in the account-specific path, is merge-conflicting, and lacks after-fix real Feishu proof.

## What This Changes

The PR caps Feishu startup/status probes at 3000ms, parallelizes multi-account bot identity probes, detaches monitor startup into background tasks, and updates Feishu tests.

## Best Possible Solution

Land a Feishu-local timeout/status mitigation only after preserving the gateway monitor lifetime, keeping account-scoped diagnostics, rebasing on current main, and proving real Feishu startup/status behavior; keep https://github.com/openclaw/openclaw/issues/72721 as the canonical tracker for the broader timeout problem.

## Reproduction Assessment

Yes for the PR-introduced lifecycle bug: source inspection shows Feishu `startAccount` returns `monitorFeishuProvider({ accountId })`, while the PR returns from that branch immediately and the gateway treats task settlement as channel exit/restart. No live proof establishes the claimed after-fix latency improvement.

## Solution Assessment

No. Bounding Feishu startup/status probes is a plausible direction, but this implementation is not the best fix because it detaches the gateway-managed monitor lifetime and changes timeout policy without real upgrade proof.
