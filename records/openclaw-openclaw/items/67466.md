## Summary

Keep open: the PR targets a real Feishu startup-latency problem, but the current branch is unsafe to merge because it detaches gateway-managed monitor promises, reverses existing probe serialization despite related timeout reports, lacks after-fix real behavior proof, and is conflicting with current main.

## What This Changes

This PR caps Feishu startup/status probes at 3 seconds, parallelizes multi-account bot-identity probing, detaches Feishu account monitors into background tasks, and updates Feishu startup/status tests.

## Best Possible Solution

Preserve the gateway-owned monitor promise, bound or queue Feishu startup probes instead of unbounded fan-out, keep the status timeout cap if still applicable after rebasing, reconcile with https://github.com/openclaw/openclaw/pull/63481, and add redacted live Feishu startup proof.

## Reproduction Assessment

Yes for the PR regression: source inspection shows current Feishu gateway startAccount passes accountId to monitorFeishuProvider, and server-channels treats an immediately resolved startAccount promise as a channel exit. The original live 100s Feishu startup delay was not rerun in this read-only review.

## Solution Assessment

No. The setup/status timeout cap is useful, but detaching account monitors and unbounded parallel probe fan-out are not the narrowest maintainable fix; the safer direction keeps monitor ownership tracked and uses bounded or queued probes.
