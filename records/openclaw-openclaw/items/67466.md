## Summary

Keep this PR open. Current main still does not include the PR's focused Feishu setup/status 3000ms probe cap, but the PR is not safe to merge as-is because it makes the gateway-managed Feishu account monitor resolve immediately and therefore violates the channel lifecycle contract.

## What This Changes

The PR parallelizes Feishu startup bot-identity probes, caps startup and setup/status probes at 3000ms, starts Feishu account monitors in the background, and updates Feishu startup/status tests.

## Best Possible Solution

Preserve the long-lived gateway startAccount contract, extract the useful setup/status timeout cap, and handle startup bot-info latency through a lifecycle-safe timeout or serialization queue that does not burst Feishu APIs or detach monitors from abort/restart ownership.

## Reproduction Assessment

Yes for the PR regression: current gateway code and tests reproduce the contract by treating an immediately resolved startAccount as a clean channel exit followed by restart attempts. The original latency report is source-confirmed as plausible because current main still has 10000ms setup/status probe and 30000ms startup bot-info defaults, but I did not run a live Feishu timing test.

## Solution Assessment

No. The status timeout cap is useful, but making monitorFeishuProvider({ accountId }) fire-and-forget is not the narrowest maintainable fix because it breaks gateway lifecycle tracking; a safer fix keeps that promise tied to monitorSingleAccount and addresses probe latency inside the startup/status probe layers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep startAccount tied to the Feishu monitor lifetime:** `extensions/feishu/src/monitor.ts:87-93`
  - body: The accountId path is what the gateway returns from startAccount. Returning immediately after starting monitorSingleAccount in the background makes server-channels treat the account as exited, mark it not running, and schedule restart attempts while the real monitor is detached from tracked abort/task state.
  - confidence: 0.95
- **[P2] Preserve the account id when probe preflight rejects:** `extensions/feishu/src/monitor.ts:122-131`
  - body: The rejected Promise.allSettled result only carries the error, so this branch logs feishu[unknown] for unexpected probe failures. Catch inside the per-account map callback instead, where account.accountId is still available, and return a degraded identity for that account.
  - confidence: 0.86
