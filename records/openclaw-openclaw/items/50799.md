## Summary

Keep open. Current main still enforces the reported image-only path in the shared Swift macOS chat composer and view model, while the web Control UI and Gateway already support non-video document attachments, so this remains a narrow app-side bug rather than a duplicate or ClawHub/core-boundary issue.

## What This Changes

The issue asks OpenClaw's macOS GUI paperclip attachment picker to allow common document/file uploads such as PDF, TXT, and DOCX instead of only images.

## Best Possible Solution

Broaden the shared Swift macOS attachment intake to match the existing Control UI and Gateway contract: allow supported non-video file types, keep image previews for images, render generic file previews for documents, preserve unsupported-type and size validation, and cover PDF/TXT/DOCX plus image parity in focused Swift tests.

## Reproduction Assessment

Yes. The deterministic reproduction path is the macOS paperclip flow: NSOpenPanel is limited to [.image], and the view model rejects non-image UTTypes even if documents are passed in programmatically.

## Solution Assessment

Yes, fixing the shared Swift app attachment picker/loading path is the narrowest maintainable solution. The Gateway chat.send contract and Control UI already support non-image file attachments, so a core protocol rewrite is not needed unless Swift tests expose a missing generic seam.
