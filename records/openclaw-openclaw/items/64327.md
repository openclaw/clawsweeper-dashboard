## Summary

Keep open. Current main still resolves and uses the effective context-token cap internally, but the public plugin hook contract and the llm_output/model_call hook emitters do not expose that resolved limit to hook consumers. The related #64260 is adjacent native-context-window telemetry work, not a replacement for this narrower effective-limit request.

## What This Changes

Expose the resolved effective context-token limit from agent contextTokens/model context resolution to llm_output and model-call hook consumers.

## Best Possible Solution

Add a backward-compatible hook field for the resolved effective context-token limit, preferably with native/reference/source metadata, then thread it through embedded Pi llm_output, sanitized model-call hooks, and shared harness/CLI hook paths with docs and focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure a 200k model window with agents.defaults.contextTokens at 20k; current tests prove the resolver returns 20k, while the hook types and emitters expose no field carrying that value.

## Solution Assessment

Yes. The narrow maintainable fix is an additive plugin hook contract update fed from existing ContextWindowInfo/ctxInfo data; plugin-side workarounds cannot reliably recover the agent-level cap after runtime resolution.
