## Summary

Keep open. Current main still resolves and uses the effective context-token cap internally, but the plugin hook contract and the `llm_output` / `model_call_*` emitters do not expose that resolved cap to hook consumers. The related #64260 is adjacent native context-window metadata work, not a replacement for this narrower effective-limit request.

## What This Changes

Expose the resolved effective context-token limit from agent `contextTokens` configuration in hook event context or payloads alongside the model's native context window.

## Best Possible Solution

Add backward-compatible hook metadata for the resolved effective context-token limit, ideally with reference/native-window and source fields, then thread it through the embedded, model-call, and CLI harness hook paths with plugin docs and focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure `agents.defaults.contextTokens` below a model's native window, observe the existing resolver test proving the effective cap, then inspect the hook types and emitters to see that no hook-visible field carries that value.

## Solution Assessment

Yes. The narrow maintainable fix is an additive plugin hook contract update fed from existing `ctxInfo.tokens`; plugin-side workarounds cannot reliably recover the internal agent cap after model resolution.
