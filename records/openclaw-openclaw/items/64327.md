## Summary

Keep open. Current main still resolves and uses the effective context-token cap internally, but the plugin hook contract and llm_output/model_call hook emitters do not expose that resolved cap to hook consumers; #64260 is adjacent native context-window telemetry, not a replacement for this narrower effective-limit request.

## What This Changes

Expose the resolved effective context-token cap alongside model metadata in llm_output and model_call hook contexts.

## Best Possible Solution

Add a backward-compatible hook metadata field for the resolved effective context-token limit, with clear native-vs-effective naming and source/reference metadata, then thread it through embedded, model-call, and harness hook paths with docs and focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure a 200k model with agents.defaults.contextTokens at 20k; current tests prove the resolver returns 20k, while the public hook types and emitters expose no field carrying that value.

## Solution Assessment

Yes. The narrow maintainable fix is an additive plugin hook contract update fed from existing ContextWindowInfo/contextTokenBudget data; plugin-side workarounds cannot reliably recover the agent-level cap after runtime resolution.
