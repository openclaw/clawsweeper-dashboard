## Summary

Keep this PR open for maintainer handling, but do not merge it as-is. The launchd plist change addresses a real debugging pain point, but current main still relies on clean `exit(0)` plus boolean `KeepAlive=true` for supervised restart/update relaunches, and the branch also includes unrelated sandbox Docker image behavior with a concrete supply-chain/runtime regression.

## What This Changes

The PR changes the macOS LaunchAgent plist, launchd audit, and launchd test from boolean `KeepAlive=true` to `SuccessfulExit=false`, while also modifying sandbox Docker/browser/doctor handling for daemon and image availability.

## Best Possible Solution

Use this branch as context, but replace it with a focused daemon lifecycle fix that covers crash-loop debuggability, clean stop, `openclaw gateway restart`, update restarts, and launchd handoff behavior together. Remove the sandbox Docker/browser/doctor changes from this PR unless they are split into a separate review that preserves the no-silent-Debian-substitution contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main returns supervised restart mode under launchd and exits with status 0 for restart/update handoff, while `SuccessfulExit=false` tells launchd not to relaunch after a successful exit. The Docker regression is reproducible by missing the default OpenClaw sandbox image while Docker is available.

## Solution Assessment

No. `SuccessfulExit=false` is a plausible launchd primitive for crash debugging, but using it alone is not the best fix because current restart/update code still relies on clean-exit relaunch; the unrelated sandbox image changes should be split or removed.
