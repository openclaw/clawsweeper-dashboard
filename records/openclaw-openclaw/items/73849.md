## Summary

Keep this PR open, but do not merge it as-is. The linked launchd crash-loop problem is plausible, yet the proposed `SuccessfulExit=false` plist change conflicts with current restart and update handoff paths that intentionally exit 0 under launchd, and the branch also carries unrelated sandbox image provenance changes that should not be folded into a daemon lifecycle fix.

## What This Changes

The PR changes the macOS LaunchAgent KeepAlive policy to `SuccessfulExit=false`, updates launchd audit/test expectations, and modifies sandbox Docker/browser/doctor handling for missing images and Docker daemon-unavailable errors.

## Best Possible Solution

Replace this with a focused daemon lifecycle change that preserves intentional launchd-supervised relaunches while improving crash-loop debuggability, and drop or split the sandbox Docker/browser/doctor changes so the existing no-silent-Debian-substitution contract remains intact.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: under launchd, current main returns supervised restart mode and the run loop exits with status 0 for update and full-process restart handoff; `SuccessfulExit=false` would prevent launchd from relaunching after those successful exits.

## Solution Assessment

No. `SuccessfulExit=false` is a plausible launchd primitive for debugging crashes, but using it alone is not the best fix because current restart/update behavior depends on successful-exit relaunch, and the unrelated sandbox image changes add avoidable security and correctness risk.
