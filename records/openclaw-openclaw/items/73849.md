## Summary

Keep this PR open, but do not merge it as-is. The daemon idea addresses a real open launchd debugging problem, but the proposed plist change breaks current clean-exit relaunch paths for explicit restart/update handoff, and the branch also carries unrelated sandbox image provenance changes that need to be split or removed.

## What This Changes

The PR changes the macOS LaunchAgent `KeepAlive` policy and launchd audit/test expectations while also modifying sandbox Docker, browser, and doctor handling for Docker daemon and image availability.

## Best Possible Solution

Land a focused daemon lifecycle fix that preserves explicit `openclaw gateway restart` and update restarts while improving crash-loop debuggability, with sandbox Docker/browser/doctor changes removed or split into a separate security-owned PR that preserves the no-silent-Debian-substitution contract.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current launchd-supervised restart/update paths return `mode: "supervised"` and then exit with status 0, while the PR's `SuccessfulExit=false` policy would not relaunch after that successful exit.

## Solution Assessment

No. `SuccessfulExit=false` may be one useful launchd primitive for debugging crash loops, but applying it alone is not the best fix because current clean restart and update paths depend on boolean `KeepAlive`; the unrelated sandbox changes are also outside the daemon fix boundary.
