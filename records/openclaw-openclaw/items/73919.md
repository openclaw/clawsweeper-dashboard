## Summary

Keep open. The PR addresses a real bundled MCP runtime leak that is still visible on current main, but the proposed runtime-manager hardening still has a different-fingerprint in-flight replacement race that should block merge.

## What This Changes

The PR propagates one-shot bundled MCP cleanup through isolated cron, isolated heartbeat, queued follow-up, and reply-runner paths, hardens session runtime disposal, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the narrow one-shot cleanup direction, but add a post-await session recheck or per-session single-flight guard before replacement runtime creation and lock that interleaving in a focused runtime-manager test.

## Reproduction Assessment

Yes. #68450 gives a high-confidence live A/B reproduction for the leak, current-main source inspection shows the missing cleanup paths, and the PR race can be reproduced by interleaving another same-session getOrCreate while the stale in-flight replacement path awaits.

## Solution Assessment

No as-is. The one-shot cleanup boundary is the right maintainable direction, but the runtime manager needs to re-read or serialize session state after awaiting stale in-flight creation before creating a replacement.
