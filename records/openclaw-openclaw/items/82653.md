## Summary

Keep open: current main and the latest shipped release still parse the reported voicecall numeric options with `Number(...)` and no finite-value guard. The issue now has an open closing PR at https://github.com/openclaw/openclaw/pull/82668, so it should remain open until that PR lands, is replaced, or is closed.

## What This Changes

Validate voicecall numeric CLI options so invalid strings fail fast instead of flowing as `NaN` into tail, latency, and expose commands.

## Best Possible Solution

Review and land a focused voice-call plugin fix that adds finite integer parsing for these flags, preserves the current bounds, and adds invalid-value regression coverage; keep the issue open until the linked PR merges or is replaced.

## Reproduction Assessment

Yes, source-level current-main inspection gives a high-confidence reproduction path: the reported flags flow through `Number(...)`, and invalid strings become `NaN` before reaching slices, sleeps, or generated URLs. I did not run the full live CLI because `tail` can loop and `expose` may call Tailscale, but the failure path is directly visible in source.

## Solution Assessment

Yes: fail-fast finite integer parsing inside the voice-call plugin CLI is the narrow maintainable fix. A core-wide CLI parsing policy is unnecessary for this localized plugin command bug.
