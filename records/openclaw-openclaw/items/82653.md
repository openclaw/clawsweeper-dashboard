## Summary

Keep open: current main and the latest release still parse the reported Voice Call CLI numeric options with `Number(...)` and clamps that allow `NaN` to flow into tailing, latency slicing, and webhook exposure URLs.

## What This Changes

The issue asks the `voicecall` CLI to reject invalid numeric values for `tail`, `latency`, and `expose` options instead of letting `NaN` reach runtime behavior.

## Best Possible Solution

Add strict finite integer parsing for these Voice Call CLI options, fail fast with option-specific messages, preserve documented minimums, and cover invalid inputs with focused CLI tests.

## Reproduction Assessment

Yes, by source inspection: the current parser sends `Number("nope")` through `Math.max` or URL formatting, so the reported `NaN` paths are reachable without a product decision.

## Solution Assessment

Yes, a narrow CLI parser/test fix is the best path; it should not change Voice Call runtime/provider behavior or add new configuration.
