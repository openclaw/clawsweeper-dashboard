## Summary

Keep open: current main and the latest shipped release still parse the reported voicecall numeric options with `Number(...)` and no finite-value check, so invalid strings can flow through as `NaN`. This is a narrow CLI validation bug, and a GitHub search did not find a better canonical issue.

## What This Changes

The issue asks for `voicecall` numeric CLI flags to reject invalid values instead of accepting `NaN`.

## Best Possible Solution

Add a plugin-local finite integer option parser for these voicecall flags, preserve the existing documented bounds, fail with an error naming the bad option, and cover invalid-value cases in voice-call CLI tests.

## Reproduction Assessment

Yes from source inspection: current main has the exact reported `Number(...)` plus `Math.max(...)` paths, and a small Node probe confirms invalid strings become `NaN`. I did not run the full CLI because this read-only review avoids commands that may create runtime artifacts or call Tailscale.

## Solution Assessment

Yes: the issue's requested fail-fast numeric parsing is the narrow maintainable fix. This should stay in the voice-call plugin CLI rather than adding a core-wide CLI parsing policy.
