## Summary

Keep open: current main and the latest shipped release still parse the reported voicecall numeric options with `Number(...)` and no finite-value check, and the only related open PR is for a different logging helper rather than this CLI path.

## What This Changes

The issue requests that voicecall CLI numeric flags reject non-numeric values instead of allowing `NaN` to flow into tail, latency, or expose behavior.

## Best Possible Solution

Add a voice-call plugin-local finite integer parser for these flags, preserve existing bounds, fail with clear option-specific errors, and cover invalid values in focused CLI tests.

## Reproduction Assessment

Yes, from source inspection: current main has the exact `Number(...)` plus `Math.max(...)` paths that allow non-numeric option strings to become `NaN`. I did not execute the full `expose` command because it can call Tailscale, but the failing value flow is direct in the source.

## Solution Assessment

Yes: fail-fast finite integer parsing in the voice-call CLI is the narrow maintainable fix. This should stay in the plugin command implementation rather than adding a core-wide CLI parsing policy.
