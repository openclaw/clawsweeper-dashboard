## Summary

Current main still lacks native command-content deny rules for exec approvals, and this issue remains the canonical tracker after the narrower related report was closed as a duplicate. Closed implementation attempts provide useful prior art, but none merged and at least one exposed fail-open and cross-path consistency concerns, so the remaining work requires explicit security and product design rather than cleanup closure.

## What This Changes

Add an exec-approvals denylist mode with command-argument matching and approval-on-match behavior.

## Best Possible Solution

Adopt one additive, fail-closed command-policy model with deterministic precedence and parity across gateway, node, and system.run, then expose the same canonical schema through CLI, Control UI, protocol, focused tests, and public documentation.

## Reproduction Assessment

Yes at source level: current main still has no native exec-approval denylist collection, denylist mode, approval-on-match mode, or command-content deny evaluation path. This establishes the missing capability without needing to execute a dangerous command.

## Solution Assessment

No, the original array-and-mode sketch is not yet the complete best solution. A maintainable implementation must first settle precedence, argument and shell normalization, fail-closed behavior, and parity across every execution host.
