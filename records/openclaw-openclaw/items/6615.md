## Summary

Keep open: current main and v2026.6.6 still lack native exec-approval denylist mode, denylist entries, `ask: "on-match"`, or deny-before-allow evaluation; related plugins and the open path-deny PR are useful prior art but do not replace this core security-policy decision.

## What This Changes

The issue requests native exec-approval denylist support so broad exec can be allowed while specific dangerous command or argument patterns are blocked or sent for approval.

## Best Possible Solution

Keep this as the canonical native exec-approval denylist tracker until maintainers land or explicitly reject an additive, version-safe core policy that supports deny-before-allow matching and documents its security tradeoffs.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested policy values, config collection, protocol field, UI editor, and docs are absent, so the feature gap is high-confidence without running a live command.

## Solution Assessment

Unclear. The requested direction is valid, but the best implementation shape needs maintainer security/product design; the open path-deny PR is a plausible narrower hardening layer, not a complete replacement for this issue.
