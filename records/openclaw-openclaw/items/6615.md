## Summary

Keep open. Current main still lacks native exec-approvals denylist storage, `security: "denylist"`, `ask: "on-match"`, and deny-before-allow evaluation; the external SafeClaw/CRE references and closed PR attempts are useful prior art, but this remains native security-policy design work.

## What This Changes

The issue requests native exec-approvals denylist support with argument matching and on-match prompting so users can allow most commands except dangerous patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node paths, exposes the config in CLI and Control UI, documents fail-open tradeoffs, and coordinates with https://github.com/openclaw/openclaw/issues/71097.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high-confidence proof that native denylist storage, `security: "denylist"`, and `ask: "on-match"` behavior are absent.

## Solution Assessment

Unclear as proposed. Native core support is the right surface for host exec authorization, but maintainers still need to decide policy shape, deny precedence, matching semantics, audit behavior, and prompt versus hard-deny behavior before implementation.
