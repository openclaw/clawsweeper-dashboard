## Summary

Keep this open. Current main still does not implement native exec-approvals denylist support: the policy model, config sanitizer, runtime schema, gateway protocol schema, Control UI, docs, and tests remain centered on `deny | allowlist | full` and allowlist entries. The linked SafeClaw and CRE projects plus closed PRs #43222, #60684, and #72466 are useful prior art, but none landed on `main`, and maintainer discussion has already kept this security-sensitive design open.

## What This Changes

The issue requests native exec-approvals denylist support so users can allow most commands while prompting or blocking commands that match dangerous binary or argument patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent binary and argument matching across gateway and node `system.run` paths, exposes the configuration in CLI and Control UI, documents the security model, and coordinates the overlap with #41140 before implementation.

## Reproduction Assessment

Yes. The missing feature is reproducible by read-only source inspection on current main: the requested `denylist`, `security: "denylist"`, `ask: "on-match"`, and `argsMatch` concepts are absent from the policy types, normalization, schemas, UI, docs, and execution evaluation path.

## Solution Assessment

Unclear that the exact proposed API is the best final design. The maintainable path is a maintainer-reviewed additive security policy with explicit deny-before-allow ordering and compatibility across config, protocol, node/gateway execution, UI, and docs.
