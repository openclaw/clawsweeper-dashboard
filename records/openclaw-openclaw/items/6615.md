## Summary

Keep this canonical feature issue open: current main still lacks an allow-all-except policy, the narrower command-content report was closed into this tracker, and multiple closed implementation attempts exposed unresolved security and precedence choices that require maintainer product review.

## What This Changes

Add native exec-approval rules that can match command paths and arguments to require approval or deny execution while allowing nonmatching commands.

## Best Possible Solution

Define one canonical ordered rule evaluator with explicit allow, require-approval, and deny outcomes; specify scope precedence and shell-analysis semantics; enforce it identically across gateway and node execution; and document negative rules as defense-in-depth rather than complete containment.

## Reproduction Assessment

Yes, at source level. Current main has no denylist mode, `on-match` ask mode, or negative-rule collection across policy normalization, the Control UI, and the documented contract.

## Solution Assessment

No, the exact `security: denylist` plus `ask: on-match` proposal is not yet the best-proven permanent interface. An ordered outcome-based rule model would avoid multiplying overlapping modes and would make deny-versus-approval precedence auditable.
