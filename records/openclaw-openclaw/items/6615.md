## Summary

Keep open. Current main and v2026.5.12 still lack native exec-approvals denylist, `ask: "on-match"`, denylist storage, or deny-match evaluation, and the related implementation attempts are either closed or still open/unmerged. This is host-exec security policy, so it needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue requests native exec-approvals denylist support with argument-aware matching and prompt-on-match behavior.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes the policy in CLI and Control UI, documents the security model, and resolves overlap with related exec risk/path-deny trackers.

## Reproduction Assessment

Yes for the missing capability: source inspection on current main shows no accepted denylist mode, no `on-match` ask mode, no denylist collection, and no deny-match branch. This is not a broken existing contract reproduction because the issue requests a new policy feature.

## Solution Assessment

Unclear until maintainers choose the policy shape. A native deny-before-allow policy is plausible, but the exact implementation needs security/product decisions around defaults, precedence, argument matching, auditability, and UI/docs exposure.
