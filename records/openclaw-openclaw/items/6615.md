## Summary

Current main still lacks negative exec-approval rules, and the closed implementation attempts do not supersede this issue. This remains a security-sensitive product and configuration decision requiring explicit maintainer ownership.

## What This Changes

Add native exec-approval rules that match command paths and arguments, then require approval or deny matching commands while allowing nonmatches.

## Best Possible Solution

Define one additive rule model with explicit allow, require-approval, and deny outcomes, evaluate negative rules before broad allow decisions, and use identical normalized-command semantics across local, gateway, and node execution paths.

## Reproduction Assessment

Yes for the capability gap: current-main types, persisted policy, and approval evaluation directly show that deny-pattern and on-match behavior is absent.

## Solution Assessment

Unclear: argument-aware negative rules address the user need, but a tri-state rule overlay with explicit precedence is safer and more maintainable than independently adding `denylist` and `on-match` modes.
