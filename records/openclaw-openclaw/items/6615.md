## Summary

Keep open. Current main still lacks native exec-approvals denylist storage, `security: "denylist"`, `ask: "on-match"`, and deny-before-allow evaluation; CRE/SafeClaw and prior closed PRs are useful prior art, but they do not replace a native security-policy decision.

## What This Changes

Request adds native exec-approval denylist mode/config with argument-aware matching and prompt-on-match behavior.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes configuration in CLI and Control UI, and documents the security model and fail-closed behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high-confidence proof that native `denylist`, `security: "denylist"`, and `ask: "on-match"` behavior are absent.

## Solution Assessment

Unclear as proposed. Native core support is the right venue for host exec authorization, but maintainers should settle policy shape, precedence, matching semantics, and prompt versus hard-deny behavior before implementation.
