## Summary

Current main and latest release still lack native exec-approvals denylist mode, a denylist collection, `ask: "on-match"`, or a deny-before-allow decision path, so this remains valid open security-policy work rather than cleanup-close material.

## What This Changes

The issue requests native exec-approvals support for an allow-everything-except policy with denylist entries, argument matching, and prompting on denylist matches.

## Best Possible Solution

Define an additive native exec approvals deny policy or deny-pattern overlay with explicit precedence, gateway/node/system.run parity, CLI and Control UI support, docs, audit behavior, upgrade-safe tests, and a clear relationship to the narrower path-deny work.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.5 shows only the existing exec security/ask modes, an allowlist collection, and no denylist/on-match decision branch.

## Solution Assessment

Unclear until maintainer/security acceptance. A native deny-before-allow policy is plausible and aligned with the user need, but maintainers still need to choose the config shape, matching semantics, prompt versus hard-deny behavior, audit behavior, and overlap with narrower path-deny work.
