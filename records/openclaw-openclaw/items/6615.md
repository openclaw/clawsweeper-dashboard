## Summary

Keep open. Current main and the latest release still lack native exec-approval denylist/on-match support, and the discussion shows this remains a security-sensitive feature/config design rather than cleanup. SafeClaw/CRE and the closed unmerged implementation attempts are useful prior art, but they do not replace a merged OpenClaw policy change; https://github.com/openclaw/openclaw/issues/41140 was closed back to this issue as the canonical tracker.

## What This Changes

Request native exec-approval denylist support with command/argument matching and an on-match approval mode so operators can allow most commands while prompting or blocking dangerous patterns.

## Best Possible Solution

Define an additive native exec-approval policy that evaluates operator deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node system.run paths, and updates schema, CLI/UI, docs, tests, and security guidance together.

## Reproduction Assessment

Yes for the missing capability: current-main source, protocol schema, Control UI, docs, and focused tests still accept only the existing allowlist-centered policy values and have no denylist/on-match evaluation path. This is a feature-gap reproduction, not a failing existing bug.

## Solution Assessment

Unclear. The requested denylist/on-match direction is plausible, but the maintainable fix needs a maintainer security/product decision on exact semantics, evaluation order, and overlap with related deny-pattern/path-deny trackers.
