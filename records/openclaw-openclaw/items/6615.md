## Summary

Current main still cannot express the requested allow-all-except policy, and all native implementation attempts in the issue timeline remain closed unmerged. The request should stay open for an explicit maintainer security/product decision rather than be treated as implemented, duplicated, or a safe automated fix.

## What This Changes

Add native exec-approval rules that broadly allow execution while prompting or blocking selected command and argument patterns.

## Best Possible Solution

Adopt one ordered, parsed-command policy-rule model with explicit allow, deny, and require-approval outcomes, deterministic precedence before broad allow decisions, and identical config, protocol, gateway/node, CLI, UI, test, and documentation behavior.

## Reproduction Assessment

Yes. Current main’s accepted policy modes, normalized configuration, protocol schema, UI, and approval evaluator have no representation or evaluation path for allow-all-except command or argument patterns.

## Solution Assessment

Unclear. The user problem is well established, but a separate `denylist` mode is not yet proven better than a general ordered rule layer that avoids parallel evaluators and makes prompt-versus-block outcomes explicit.
