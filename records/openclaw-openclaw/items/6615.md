## Summary

Keep open. Current main still lacks native exec-approvals denylist storage, `security: "denylist"`, `ask: "on-match"`, and deny-before-allow evaluation; the SafeClaw/CRE links plus closed implementation PRs are useful prior art, but the latest maintainer review and the newer security-labeled design thread point to explicit policy follow-up rather than cleanup closure.

## What This Changes

Add native exec-approvals denylist support so operators can allow most commands while blocking or prompting on configured dangerous command and argument patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports command and argument matching consistently across gateway and node paths, exposes the config in CLI and Control UI, documents fail-open tradeoffs, and coordinates with https://github.com/openclaw/openclaw/issues/71097.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high-confidence proof that native denylist storage, `security: "denylist"`, and `ask: "on-match"` behavior are absent.

## Solution Assessment

Unclear as proposed. Native core support is the right surface for host exec authorization, but maintainers still need to decide policy shape, deny precedence, matching semantics, audit behavior, feature gating, and prompt versus hard-deny behavior before implementation.
