## Summary

Keep open. Current main and the latest release still lack native exec-approvals denylist, deny-pattern, or `on-match` semantics, and the remaining work is a security-sensitive policy/config decision rather than conservative cleanup.

## What This Changes

The issue requests native exec-approvals denylist support so users can allow most commands while prompting or blocking specific command and argument patterns.

## Best Possible Solution

Define an additive native exec approvals deny policy or deny-pattern overlay with explicit precedence, audit and fallback semantics, gateway/node/system.run parity, CLI and Control UI support, and public docs; keep this issue as the broad tracker unless maintainers intentionally split the work into a protected canonical security design item.

Yes for source-level absence, not as a failing bug reproduction. Current main accepts only existing exec security/ask modes, stores allowlists only, and lacks a denylist collection or deny-match decision branch.

## Solution Assessment

Unclear until maintainer/security acceptance. A native denylist is plausible and aligned with the user need, but maintainers still need to choose the public config shape, denial precedence, prompt versus hard-deny behavior, audit behavior, and relationship to narrower denied-path work.
