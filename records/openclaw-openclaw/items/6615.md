## Summary

Keep open. Current main and the latest release still lack native exec-approvals denylist storage, `security: "denylist"`, `ask: "on-match"`, or deny-before-allow evaluation; SafeClaw/CRE and earlier closed PRs are useful prior art, but the active implementation still needs maintainer/security policy acceptance before this issue can close.

## What This Changes

Add native exec approvals denylist support so operators can allow most commands while prompting or blocking dangerous command and argument patterns.

## Best Possible Solution

Land one additive native exec approvals denylist contract through https://github.com/openclaw/openclaw/pull/82596 or a successor after maintainer/security acceptance of rule shape, precedence, prompt behavior, audit semantics, host parity, UI, docs, and regression coverage.

## Reproduction Assessment

Yes for source-level absence, not as a failing bug reproduction. Current main and v2026.5.22 only accept existing exec security/ask modes, store allowlists, and have no deny-match decision branch.

## Solution Assessment

Unclear until maintainer/security acceptance. Native deny rules are plausible, but maintainers still need to settle config shape, denial precedence, prompt versus hard-deny behavior, default rules, audit behavior, and gateway/node/macOS parity.
