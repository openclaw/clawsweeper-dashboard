## Summary

Keep open. Current main and the latest release still expose exec approvals as allowlist-centered policy only, and the related discussion plus open implementation PR show a real security/product gap rather than cleanup-ready stale work.

## What This Changes

Add native exec-approval denylist support so broad host exec can still prompt or block specific command and argument patterns.

## Best Possible Solution

Define an additive native exec deny policy that evaluates deny rules before broad allow decisions, supports command and argument matching consistently across gateway and node `system.run` paths, exposes the config in CLI and Control UI, documents the security tradeoffs, and lands through the open PR or a focused replacement after security review.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for source-level proof of the missing capability. Current main still accepts only the existing exec security/ask modes, stores allowlists only, and has no deny-match decision branch in the approval path.

## Solution Assessment

Unclear. A native deny policy is plausible, but the exact implementation needs maintainer security/product decisions around precedence, matching depth, fail-closed behavior, UI/config shape, and audit events before this should merge.
