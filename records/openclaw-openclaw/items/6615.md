## Summary

Current main still cannot express “allow everything except these commands”: argument-aware matching now narrows allowlist grants, but there is no deny-pattern collection, prompt-on-match outcome, or deny-before-broad-allow evaluation. This remains the canonical tracker after the narrower command-content report was closed into it, while every linked implementation PR closed unmerged; the remaining work needs an explicit security-policy and compatibility decision rather than cleanup closure.

## What This Changes

Add native exec-approval rules that allow broad execution while prompting for or blocking configured command and argument patterns.

## Best Possible Solution

Define one additive tri-state rule model with explicit allow, require-approval, and deny outcomes; evaluate normalized executable and parsed argv through one shared gateway/node policy path before broad allow decisions, then expose the same contract through protocol, CLI, Control UI, docs, and focused security tests.

## Reproduction Assessment

Yes. Current-main types, storage shape, decision logic, UI/docs, and the latest release all show that allow-everything-except command rules remain unavailable; this is source-reproducible absence of a requested capability, not a runtime bug.

## Solution Assessment

No, the literal `security: denylist` plus `ask: on-match` proposal is not yet proven to be the best permanent API. A unified rule/outcome model is safer because it makes precedence explicit and avoids competing allowlist and denylist evaluators.
