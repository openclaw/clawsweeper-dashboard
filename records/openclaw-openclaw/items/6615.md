## Summary

Keep open. Current main still lacks native exec-approvals denylist storage, `security: "denylist"`, `ask: "on-match"`, and deny-before-allow evaluation; the SafeClaw/CRE links and closed implementation attempts are useful prior art, but the latest maintainer review explicitly kept this as native security-policy work.

## What This Changes

The issue requests native exec-approvals support for allowing commands by default while prompting or blocking commands that match deny patterns, including argument-level matching.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, reuses the existing command/argument matching seams where appropriate, exposes the config in CLI and Control UI, documents fail-open tradeoffs, and coordinates overlap with https://github.com/openclaw/openclaw/issues/71097.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high-confidence proof that native denylist storage, `security: "denylist"`, and `ask: "on-match"` behavior are absent.

## Solution Assessment

Unclear as proposed. Native core support is the right venue for host exec authorization, but maintainers still need to decide policy shape, deny precedence, matching semantics, audit behavior, and prompt versus hard-deny behavior before implementation.
