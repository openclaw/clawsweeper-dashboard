## Label Justifications

- `P2`: This is a normal-priority security-policy feature with clear user value but no current-main emergency regression.
- `impact:data-loss`: The requested policy targets destructive deletion, disk, email, and messaging commands that can cause irreversible user-visible data loss.
- `impact:security`: Exec approvals are a host command authorization boundary, and denylist semantics directly affect which commands can run without human review.

## Summary

Keep open. Current main and the latest release still lack native exec-approvals denylist mode, `ask: "on-match"`, denylist storage, or a deny-before-allow evaluation path; SafeClaw/CRE links are useful prior art, but the native OpenClaw work is still pending and https://github.com/openclaw/openclaw/pull/82596 is still open for implementation review.

## What This Changes

Add exec-approvals denylist support so operators can allow most commands while blocking or prompting on configured dangerous command and argument patterns.

## Best Possible Solution

Land one additive native exec approvals contract through https://github.com/openclaw/openclaw/pull/82596 or a successor after maintainer/security acceptance of rule shape, precedence, prompt behavior, audit semantics, host parity, UI, docs, and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; yes for source-level absence. Current main and v2026.5.20 only accept existing exec security/ask modes, store allowlists, and have no deny-match decision branch.

## Solution Assessment

Unclear. Native deny rules are plausible, but maintainers still need to settle config shape, deny-before-allow precedence, prompt versus hard-deny behavior, default rules, audit behavior, and gateway/node/macOS parity.
