## Summary

Keep open. Current main and latest release v2026.6.5 still lack native exec-approvals denylist mode, a denylist collection, `ask: "on-match"`, or a deny-before-allow decision path; related open work is narrower path-deny hardening and does not supersede this broader policy request.

## What This Changes

Request native exec-approvals denylist support so users can allow most host exec commands while blocking or prompting for configured command and argument patterns.

## Best Possible Solution

Define an additive native exec approvals deny policy or deny-pattern overlay with explicit precedence, gateway/node/system.run parity, CLI and Control UI support, docs, audit behavior, upgrade-safe tests, and a clear relationship to the narrower path-deny work.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.5 only accept the existing exec security/ask modes, resolve allowlists, and lack a denylist collection or deny-match decision branch.

## Solution Assessment

Unclear until maintainer/security acceptance. A native denylist is plausible and aligned with the user need, but maintainers still need to choose the config shape, precedence, prompt versus hard-deny behavior, and overlap with narrower path-deny work.
