## Summary

Keep open. Current main still has allowlist-focused exec approvals with no native denylist collection, no `security: "denylist"`, no `ask: "on-match"`, and no deny-before-allow decision path; the linked SafeClaw/CRE comments and closed PRs are useful prior art, but the latest maintainer review and the newer security-labeled design thread point to explicit policy design rather than cleanup closure.

## What This Changes

The issue requests native exec-approvals denylist support so users can allow most host exec commands while prompting or blocking specific command and argument patterns.

## Best Possible Solution

Define one additive native exec approvals design that evaluates deny or HIL rules before broad allow/full decisions, supports command and argv matching across gateway and node paths, exposes it in CLI and Control UI, documents the security tradeoffs, and coordinates with https://github.com/openclaw/openclaw/issues/71097.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high-confidence proof that native denylist storage, `security: "denylist"`, `ask: "on-match"`, and deny-before-allow evaluation are absent.

## Solution Assessment

Unclear as proposed. Native core support is the right surface for host exec authorization, but maintainers still need to decide policy shape, deny precedence, matching semantics, audit outcomes, feature gating, timeout behavior, and prompt versus hard-deny defaults.
