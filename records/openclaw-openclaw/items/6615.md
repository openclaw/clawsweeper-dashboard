## Summary

Keep open. Current main still lacks native exec-approvals denylist or `on-match` semantics, and the remaining work is a security-sensitive policy/config decision rather than a cleanup close candidate.

## What This Changes

Request to add exec-approvals denylist support so operators can allow most commands while blocking or prompting on specific dangerous command and argument patterns.

## Best Possible Solution

Define an additive native exec approvals deny policy or deny-pattern overlay with explicit precedence, audit, fallback, gateway/node/system.run parity, CLI and Control UI support, and docs; coordinate overlap with https://github.com/openclaw/openclaw/issues/71097 and the denied-path PRs without closing this canonical tracker prematurely.

Yes for source-level absence, not as a failing bug reproduction. Current main only accepts existing exec security/ask modes, resolves allowlists, and lacks a denylist collection or deny-match decision branch.

## Solution Assessment

Unclear until maintainer/security acceptance. A native denylist is plausible and aligned with the user need, but maintainers still need to choose the config shape, denial precedence, prompt versus hard-deny behavior, audit behavior, and gateway/node parity.
