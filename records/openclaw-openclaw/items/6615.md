## Summary

Keep open. Current main still lacks native exec-approvals denylist storage, `security: "denylist"`, `ask: "on-match"`, and deny-before-allow evaluation; CRE and SafeClaw are useful prior art, but the maintainer discussion explicitly keeps native core policy design open.

## What This Changes

The issue requests native exec-approvals denylist support, including denylist rules, argument matching, and prompting on deny matches.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes configuration in CLI and Control UI, and documents the security model.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high-confidence proof that native `denylist`, `security: "denylist"`, and `ask: "on-match"` behavior are absent.

## Solution Assessment

Unclear as proposed. Native core support is the right venue for host exec authorization, but maintainers should settle policy shape, precedence, matching semantics, and prompt versus hard-deny behavior before implementation.
