## Summary

Current main still cannot express “allow broadly except these command or argument patterns.” Recent exec-policy work added named modes and argument-scoped reusable allow entries, but enforcement remains deny-all, allowlist, or full, and all four native denylist PRs cited in the timeline closed unmerged. The issue should remain open for explicit maintainer security/product design rather than be closed as implemented or duplicated.

## What This Changes

Add native exec-approval deny or require-approval rules that match selected executable and argument patterns while allowing other commands.

## Best Possible Solution

Extend the canonical exec-policy evaluator with parsed-command rules carrying explicit deny or require-approval actions, evaluate them before broad allow decisions, and reuse the same normalized result across gateway and node hosts, protocol, CLI, Control UI, tests, and docs.

## Reproduction Assessment

Yes. Current-main policy types, evaluator inputs, tests, protocol, and docs provide a high-confidence source reproduction: argument matching exists only for reusable allow entries, with no deny-pattern or on-match decision path.

## Solution Assessment

No, the literal proposed denylist mode is not yet proven to be the best design. A single parsed-command rule layer with explicit actions and defined precedence is safer than parallel substring or regex checks added independently to each execution path.
