## Summary

Keep open: current main still has no native exec-approvals denylist mode, denylist collection, `ask: "on-match"`, or deny-pattern evaluation path. The SafeClaw and CRE links in the discussion are useful prior art, but the latest maintainer context treats this as security-policy design rather than cleanup already solved in core.

## What This Changes

The issue requests native exec-approvals denylist support for allowing broad host exec while prompting or blocking matched dangerous command and argument patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes the configuration in CLI and Control UI, and documents the security model alongside regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main confirms the requested modes, fields, UI surface, protocol payload, docs, and deny evaluation path are absent.

## Solution Assessment

Unclear until maintainers settle the security design. The maintainable path is a version-safe native policy change coordinated across exec approval surfaces, using the linked plugins and closed PRs as prior art rather than treating them as the core fix.
