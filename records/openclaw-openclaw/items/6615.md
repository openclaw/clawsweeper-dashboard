## Summary

Keep open: current main and the latest release still lack native exec-approvals denylist support, while the closest implementation work is open and unmerged. The remaining decision is security/product review of the policy shape, not conservative cleanup.

## What This Changes

The issue requests native exec-approval denylist support so users can allow most commands while prompting or blocking specific dangerous command and argument patterns.

## Best Possible Solution

Review, narrow, and land a maintainer-approved native exec-approvals denylist design that evaluates deny rules before broad allow decisions and documents the security tradeoff across gateway and node exec paths.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 can be checked directly: policy types, normalizers, protocol schema, UI options, and docs have no native denylist collection or on-match prompt mode.

## Solution Assessment

Unclear as a final implementation shape. The use case is valid, but maintainers still need to choose the permanent security semantics and decide how the open denylist and path-deny pull requests should converge.
