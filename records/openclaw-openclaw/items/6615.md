## Summary

Keep open. Current main and v2026.5.20 still lack native exec-approvals denylist mode, denylist storage, `ask: "on-match"`, or deny-before-allow evaluation; the active implementation PR remains open and needs maintainer/security policy review before this issue can close.

## What This Changes

Request native exec-approvals denylist support so operators can allow broad exec while prompting or blocking specific command and argument patterns.

## Best Possible Solution

Land one additive native exec approvals denylist contract through https://github.com/openclaw/openclaw/pull/82596 or a successor after maintainer/security acceptance of rule shape, precedence, prompt behavior, audit semantics, host parity, UI, docs, and regression coverage.

## Reproduction Assessment

Yes for source-level absence, not as a failing bug reproduction. Current main and v2026.5.20 only accept existing exec security/ask modes, store allowlists, and have no deny-match decision branch.

## Solution Assessment

Unclear until maintainer/security acceptance. Native deny rules are plausible, but maintainers still need to settle config shape, deny-before-allow precedence, prompt versus hard-deny behavior, default rules, audit behavior, and gateway/node/macOS parity.
