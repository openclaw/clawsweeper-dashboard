## Summary

Keep this open for maintainer product and security review. Current main still lacks native deny-pattern rules, while the discussion shows sustained demand and useful plugin/prior-PR examples; no merged implementation or narrower canonical item now owns the remaining work.

## What This Changes

Add native exec-approval deny patterns with argument matching and configurable require-approval or hard-deny behavior.

## Best Possible Solution

Adopt one core-owned rule model with explicit `deny` or `require-approval` actions, shared full-command analysis, deterministic precedence before broad allows, and parity across gateway, node, CLI, Control UI, protocol, tests, and security documentation.

## Reproduction Assessment

Yes at source level. Current-main policy types, normalization, Control UI, protocol surfaces, and public documentation directly establish that native deny rules are absent; this is a feature gap rather than a failure of an existing contract.

## Solution Assessment

No, the exact proposed shape is not yet established as the best solution. Rule-level hard-deny and require-approval actions using shared command analysis are safer and more expressive than a standalone denylist mode paired with a global `ask: "on-match"` value.
