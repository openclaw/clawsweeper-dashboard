## Summary

Keep open. Current main and the latest shipped release still expose exec approvals as `deny | allowlist | full` with allowlist-only storage and prompt evaluation, while an active implementation PR is still under review and the remaining work needs security/product decisions before landing.

## What This Changes

The issue requests native exec-approvals denylist support so broad command execution can stay allowed while matched dangerous commands prompt or block.

## Best Possible Solution

Land one additive native exec approvals contract through https://github.com/openclaw/openclaw/pull/82596 or a successor, with deny-rule precedence, argument/content matching, gateway/node/macOS parity, UI/CLI/schema/docs coverage, audit behavior, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The missing feature is source-reproducible with high confidence because current main and v2026.5.12 only accept the existing exec security/ask modes, store allowlists, and have no deny-match decision branch.

## Solution Assessment

Unclear. Native deny rules are a plausible solution and an active PR is exploring them, but maintainers still need to settle config shape, precedence, prompt versus hard-deny behavior, defaults, audit semantics, and host parity.
