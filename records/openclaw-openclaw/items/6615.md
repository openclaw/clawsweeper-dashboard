## Summary

Keep this open: current `main` and the latest release still lack a native exec-approvals denylist mode, denylist collection, `ask: "on-match"`, or deny-before-allow evaluation path. The SafeClaw and CRE links in the discussion are useful prior art, and several PRs explore related designs, but none has merged a native OpenClaw policy that resolves the request.

## What This Changes

The issue requests native exec-approvals denylist support so most commands can run freely while specific dangerous command or argument patterns are blocked or prompted.

## Best Possible Solution

Keep this as the canonical native exec-approvals denylist tracker until maintainers merge or explicitly reject one additive policy that evaluates deny rules before broad allow decisions, supports command and argument matching consistently, and updates protocol, CLI, Control UI, tests, and docs together.

## Reproduction Assessment

Yes at the source level: current main exposes only the existing exec approval modes and no denylist/on-match evaluation path, so the requested configuration cannot be represented. This is a feature gap rather than a broken current contract.

## Solution Assessment

Unclear for the exact proposed shape: a denylist/on-match mode is plausible, but the best solution needs maintainer security-policy direction on whether this should be a fourth mode, a deny-rule overlay, or a narrower path/content hard-deny primitive.
