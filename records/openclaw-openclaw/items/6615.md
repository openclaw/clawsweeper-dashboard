## Summary

Keep open: current main and the latest release still lack native exec-approval denylist mode, prompt-on-match semantics, and a deny-pattern collection across runtime, protocol, CLI/UI, and docs. The external SafeClaw/CRE links and open implementation PRs are useful prior art, but no merged OpenClaw core change has closed the gap.

## What This Changes

Request native exec-approval denylist or deny-pattern support, including argument matching and prompt-on-match behavior, so broad exec can stay usable while interrupting dangerous commands.

## Best Possible Solution

Define an additive native exec-approvals deny policy that is accepted by runtime config, gateway protocol validators, CLI, Control UI, docs, and tests, and evaluate deny rules consistently before broad allow decisions in gateway and node system.run paths.

## Reproduction Assessment

Yes, for the absence of the requested capability: source inspection on current main shows only deny/allowlist/full and off/on-miss/always, with allowlist-only storage, protocol, UI, and docs. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear as a final product shape: a native deny-pattern layer belongs in core exec approvals, but maintainers still need to choose the supported config mode, prompt semantics, and gateway/node behavior before implementation lands.
