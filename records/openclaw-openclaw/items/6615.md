## Summary

Keep this open. Current main still has no native exec-approvals denylist mode, denylist collection, `ask: "on-match"`, or macOS argument-aware deny-pattern evaluation; the SafeClaw and CRE links plus the closed PRs are useful prior art, but they do not replace a merged native OpenClaw policy change.

## What This Changes

The issue requests native exec-approval denylist support so operators can allow most commands while prompting or blocking specific command and argument patterns.

## Best Possible Solution

Define an additive native exec-approval policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node exec paths, exposes the policy in CLI and Control UI, documents the security model, and coordinates overlap with #41140.

## Reproduction Assessment

Not applicable as a bug reproduction. This is a feature request, and source inspection gives high confidence that current main lacks the proposed denylist and `on-match` surfaces.

## Solution Assessment

Unclear. Native deny rules are a plausible direction, but the exact semantics need maintainer security/UX judgment before a narrow implementation is safe.
