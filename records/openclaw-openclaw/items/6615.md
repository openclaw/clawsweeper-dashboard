## Summary

Current main still cannot express native command deny patterns or prompt-on-match behavior, and the related implementation PRs closed without merging. SafeClaw and the CRE plugin are useful prior art, but the existing maintainer discussion correctly leaves the native security-policy and core-versus-plugin boundary for explicit product review.

## What This Changes

Add native exec-approval deny patterns that can prompt or block selected commands while permitting other execution.

## Best Possible Solution

Choose one additive authorization model with explicit deny precedence, argument-aware matching, and identical enforcement across every exec host, or first add a generic plugin authorization hook if optional policy engines should remain outside core.

## Reproduction Assessment

Yes for the capability gap: current-main policy types, protocol schema, UI options, and evaluator interfaces directly show that native deny-pattern or prompt-on-match policy is not expressible. This is not a failure of an existing documented contract.

## Solution Assessment

Unclear as proposed: denylist mode is plausible, but precedence, prompt versus hard-deny semantics, command parsing, upgrade behavior, and core-versus-plugin ownership require an explicit maintainer decision.
