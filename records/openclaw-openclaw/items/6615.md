## Summary

Keep open. Current main still has native exec approvals modeled around allowlist/full/deny policy and does not provide a denylist mode, denylist collection, `ask: "on-match"`, or cross-host deny-pattern evaluation; the SafeClaw and CRE links are useful prior art, but the latest maintainer discussion correctly treats this as native security-policy follow-up for https://github.com/openclaw/openclaw/issues/6615.

## What This Changes

The issue requests native exec-approvals denylist support, including argument-aware deny rules and prompting when a denylist rule matches.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes the policy in CLI and Control UI, and documents the security model.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main lacks the requested denylist mode, denylist collection, and `ask: "on-match"` handling.

## Solution Assessment

Unclear as proposed. Native core support is the right venue for exec-approval enforcement, but maintainers still need to choose the exact policy shape, precedence, and prompt-versus-hard-deny behavior before a narrow implementation is safe.
