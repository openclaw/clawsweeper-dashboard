- `impact:data-loss`: The requested policy targets dangerous deletion, disk, email, and messaging commands that can cause irreversible user-visible data loss.

## Summary

Keep open. Current main and v2026.5.12 still expose exec approvals as allowlist-focused `deny | allowlist | full` policy, while native denylist/on-match behavior remains under security/product review with an active implementation PR.

## What This Changes

Add native exec-approvals denylist support so operators can allow most commands while blocking or prompting for matching dangerous command and argument patterns.

## Best Possible Solution

Land one additive native exec approvals contract through https://github.com/openclaw/openclaw/pull/82596 or a successor, with deny-rule precedence, command/argument matching, gateway/node/macOS parity, CLI/UI/schema/docs coverage, audit behavior, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The missing feature is source-reproducible with high confidence because current main and v2026.5.12 only accept the existing exec security/ask modes, store allowlists, and have no deny-match decision branch.

## Solution Assessment

Unclear. Native deny rules are a plausible solution, but maintainers still need to settle config shape, precedence, prompt versus hard-deny behavior, defaults, audit semantics, and host parity before this is the best implementation path.
