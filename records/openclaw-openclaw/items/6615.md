## Summary

Keep open. Current main and the latest release still expose exec approvals as allowlist-centered policy only, while the discussion and related threads show a real security/product gap that needs native design rather than cleanup closure.

## What This Changes

The issue requests native exec-approvals denylist support so broad command execution can remain allowed while selected command and argument patterns are prompted or blocked.

## Best Possible Solution

Define an additive native exec-approval deny policy that evaluates deny rules before broad allow decisions, supports command and argument matching consistently across gateway and node `system.run` paths, exposes the configuration in CLI and Control UI, documents the security model, and coordinates with the open denylist and deny-path threads.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for source-level proof of the missing capability. Current main still accepts only the existing exec security/ask modes, stores allowlists only, and has no deny-match decision branch in the approval path.

## Solution Assessment

Unclear. A native deny policy is a plausible direction, but the best implementation still requires maintainer security/product decisions around precedence, parser depth, default behavior, UI/config shape, and audit events.
