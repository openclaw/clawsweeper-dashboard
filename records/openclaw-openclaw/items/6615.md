## Summary

Keep this issue open. Current main and the latest release still expose exec approvals as allowlist-focused policy only, and the open path-deny PR is useful but narrower than the requested native denylist plus prompt-on-match mode.

## What This Changes

Request native exec-approvals denylist support that can allow most commands while prompting or blocking configured dangerous command and argument patterns.

## Best Possible Solution

Define an additive native exec-approvals policy that evaluates deny rules before broad allow decisions, supports command and argument matching consistently across gateway and node paths, exposes the policy in CLI and Control UI, documents the security tradeoff, and coordinates the narrower path-deny work as a subset rather than a replacement.

## Reproduction Assessment

Yes, at source level for the capability gap. Current main and `v2026.6.6` can be checked directly: policy types, persisted schema, protocol, UI, and docs have no native denylist or prompt-on-match path.

## Solution Assessment

Unclear as an implementation direction. The requested use case is valid and vision-aligned, but maintainers still need to choose the policy shape and reconcile it with the narrower open path-deny PR and the related security-labeled denylist-mode issue.
