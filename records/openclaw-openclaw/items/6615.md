## Summary

Keep open. Current main still lacks native exec-approvals denylist mode, denylist storage, `ask: "on-match"`, and deny-pattern evaluation; the linked external plugins and closed PR attempts are useful prior art but do not replace a merged core policy change.

## What This Changes

The issue requests native exec-approvals denylist support with argument/content matching and prompt-on-match behavior.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes configuration in CLI and Control UI, and documents the security model.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main lacks the requested denylist mode, denylist collection, and `ask: "on-match"` handling.

## Solution Assessment

Unclear as proposed. Native core support is the right venue for host exec authorization, but maintainers still need to choose policy shape, precedence, prompt versus hard-deny behavior, and compatibility before a narrow implementation is safe.
