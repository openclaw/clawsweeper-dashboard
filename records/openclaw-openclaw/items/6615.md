## Summary

Keep open. Current main and v2026.6.5 still lack native exec-approvals denylist, `on-match`, or full command/argument deny-pattern semantics, and the newer open path-deny PR does not supersede this broader security-policy request.

## What This Changes

Request adds native exec-approvals denylist/on-match support so operators can allow most commands while blocking or prompting for configured dangerous command and argument patterns.

## Best Possible Solution

Define an additive native exec approvals deny policy or deny-pattern overlay with explicit precedence, gateway/node/system.run parity, CLI and Control UI support, audit behavior, fallback behavior, docs, and upgrade-safe tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.5 only accept the existing exec security/ask modes, resolve allowlists, and lack a denylist collection or deny-match decision branch. This is a feature absence rather than a failing bug reproduction.

## Solution Assessment

Unclear until maintainer/security acceptance. A native denylist is plausible and aligned with the user need, but maintainers still need to choose the config shape, denial precedence, prompt versus hard-deny behavior, audit behavior, and gateway/node parity.
