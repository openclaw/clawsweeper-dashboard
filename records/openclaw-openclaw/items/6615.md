## Summary

The requested capability is still necessary: current main has hardened argument-aware allowlist matching, but it still cannot allow commands broadly while denying or prompting on selected command-content patterns. All linked implementation PRs are closed unmerged, and the broader security-by-intent issue overlaps without replacing this concrete policy decision.

## What This Changes

Add exec-approval rules that allow commands by default while matching selected binaries or arguments for denial or explicit approval.

## Best Possible Solution

Use one canonical action-based exec-policy evaluator where argument-aware `deny` and `require-approval` rules run before broad allow decisions, reuse existing command-analysis hardening, and expose the same contract through storage, gateway/node protocol, CLI, Control UI, tests, and docs; explicitly retain this as the concrete tracker or fold it into https://github.com/openclaw/openclaw/issues/48532.

## Reproduction Assessment

Yes at the source level: current main’s policy types, sanitizer, evaluator, tests, UI, protocol surface, and docs conclusively show no allow-by-default deny or require-approval pattern path. A failing runtime reproduction is not applicable to this feature request.

## Solution Assessment

No, the exact `security: "denylist"` plus `ask: "on-match"` proposal is not yet the best proven contract. An action-based rule layer with explicit precedence in the existing evaluator is safer and avoids multiplying coupled security and prompt modes.
