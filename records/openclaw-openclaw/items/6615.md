## Summary

Current main still lacks the requested allow-everything-except command policy, and this issue remains the canonical focused tracker after the narrower duplicate and all implementation PRs closed unmerged. The request fits OpenClaw’s security direction, but it needs maintainer product and security judgment because rule actions, precedence, matching semantics, and configuration compatibility affect a core authorization boundary.

## What This Changes

Add native exec-approval rules that allow commands broadly while prompting or blocking configured command and argument patterns.

## Best Possible Solution

Introduce a core-owned typed rule model with explicit allow, require-approval, and deny actions, evaluated by one canonical command-and-argument policy engine before broad allow decisions and shared across local and node execution; align its schema, protocol, CLI, Control UI, docs, migration handling, and security regression coverage.

## Reproduction Assessment

Not applicable. Current main behaves according to the existing documented security and ask modes; the issue requests an additional authorization capability rather than reporting broken existing behavior.

## Solution Assessment

No, the literal `security: "denylist"` and `ask: "on-match"` additions are not yet established as the best design. Explicit typed rule actions and a shared evaluator more safely express prompt versus hard deny, precedence, and future policy evolution.

## Risks / Open Questions

- Command matching must use the same parsed command facts across local and node hosts; independent string or regex implementations would risk shell-wrapper and quoting inconsistencies.
