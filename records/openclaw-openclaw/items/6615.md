## Summary

The feature remains necessary: current main and v2026.6.5 still provide only deny, allowlist, and full exec security with approval on misses, while the narrower command-content report was closed into this issue as the canonical tracker and all linked implementation PRs closed unmerged. Existing plugin examples are useful prior art, but there is no demonstrated plugin contract that can replace a single authoritative deny-before-allow decision across gateway and node host execution.

## What This Changes

Add exec-approval deny rules that can match executables and command arguments, either blocking matching commands or requiring approval while allowing other commands.

## Best Possible Solution

Define a small additive deny-before-allow policy in the canonical core exec-approval engine, with closed allow/approve/deny outcomes, consistent executable and argument matching across gateway and node hosts, and aligned schema, UI, focused tests, and security documentation.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.5 conclusively lack a denylist collection, `denylist` security mode, `on-match` ask mode, and deny-before-allow evaluation path; a destructive live command is unnecessary to establish that absence.

## Solution Assessment

No, the issue’s exact proposed JSON should not be accepted without design review. The capability is needed, but the narrowest maintainable solution is one canonical core decision model rather than reviving any broad closed implementation or relying on an external plugin without proven enforcement coverage.
