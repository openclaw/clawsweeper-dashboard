## Summary

Keep this issue open: current main and v2026.6.5 still cannot express “allow everything except these command/argument patterns,” and closed implementation attempts did not land. The request fits OpenClaw’s operator-controlled security direction, but its fail-open security semantics and permanent configuration contract require maintainer product/security judgment. Likely related people: @vincentkoc (high confidence, recent exec-approval refactor), @steipete (medium confidence, policy reviewer and prior area work), and mvanhorn (medium confidence, prior implementation work).

## What This Changes

Add native exec-approval exception rules that can match executable paths and arguments, then prompt or deny on matches while allowing other commands.

## Best Possible Solution

Add one additive, version-safe rule evaluator to the existing parsed exec-approval pipeline, with explicit allow, require-approval, and deny outcomes, deterministic precedence, fail-safe error behavior, and shared gateway/node semantics exposed through protocol, CLI, Control UI, and docs.

## Reproduction Assessment

Not applicable: this is a new policy capability, and current source plus shipped docs conclusively show that it is absent rather than broken.

## Solution Assessment

No, the exact proposed shape is not yet proven best. The user need is valid, but integrating explicit rule outcomes into the existing parsed-command evaluator is likely safer than maintaining separate allowlist and denylist decision stacks.
