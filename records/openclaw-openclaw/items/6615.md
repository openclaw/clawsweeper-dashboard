## Summary

Keep open. Current main and the latest release still expose exec approvals as allowlist-centered policy only, while a native denylist implementation remains open and unmerged; the linked SafeClaw/CRE/plugin prior art is useful but does not replace a core security-policy decision.

## What This Changes

The issue requests native exec-approvals denylist support with command and argument matching plus prompt-on-match behavior.

## Best Possible Solution

Define and land an additive native exec deny policy after maintainer security/product review, with consistent gateway/node enforcement, CLI/UI/schema/docs coverage, tests for precedence and command/argument parsing, and coordination with https://github.com/openclaw/openclaw/pull/82596 plus related exec-policy trackers.

## Reproduction Assessment

Not applicable as a bug reproduction, but source-level proof is high-confidence: current main accepts only the existing security/ask modes, stores allowlists only, and has no deny-match decision branch.

## Solution Assessment

Unclear. A native deny policy is plausible and supported by prior art, but the exact config shape, precedence, fallback, audit, and UI behavior need maintainer security/product decisions before this is a narrow repair.
