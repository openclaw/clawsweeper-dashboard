## Summary

Current main and the latest release still lack native deny-pattern or prompt-on-match exec approvals, while the narrower related report was closed into this issue and all implementation PRs closed unmerged. Keep this canonical issue open for explicit security and product design because the need is real, but matching precedence, bypass resistance, and core-versus-plugin ownership are not safe automation decisions.

## What This Changes

Add exec-approval deny rules that match command paths and arguments, then prompt or block matches while allowing other commands.

## Best Possible Solution

Keep one canonical tracker and define an additive parsed-command policy contract with deny-before-allow precedence, explicit prompt-versus-hard-deny outcomes, and identical enforcement across gateway and node hosts; if a generic plugin policy hook can provide those guarantees, prefer that lean-core seam over a one-off native mode.

## Reproduction Assessment

Yes for the capability gap. Current main and v2026.6.6 can be checked directly: policy types, persisted schema, protocol, UI, and docs have no native deny-pattern or prompt-on-match path.

## Solution Assessment

No, the proposed fields are not yet a complete best solution. The use case is valid, but maintainers must first choose core versus plugin ownership and define canonical command matching, precedence, bypass handling, and prompt versus hard-deny semantics.
