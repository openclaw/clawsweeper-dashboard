## Summary

Keep this PR open for owner/secops review: it is a useful security-audit enhancement, but it changes suppression visibility, has a protected security label, and the current diff lets suppression config hide the audit's own suppression-disclosure marker.

## What This Changes

Adds typed `security.audit.suppressions` config, filters matched audit findings into `suppressedFindings`, updates CLI/docs/changelog output, and approval-gates agent shell edits to suppressions outside yolo mode.

## Best Possible Solution

Land a secops-approved suppression design that keeps accepted findings auditable and keeps suppression usage visible even when suppressions are configured.

## Reproduction Assessment

Yes for the review finding by source inspection: a suppression targeting the dangerous-flag finding for `security.audit.suppressions configured (<count>)` would pass through the generic filter and remove the disclosure from active output. This is a feature PR rather than a runtime bug report, so I did not run a live audit command in the read-only checkout.

## Solution Assessment

No, not as-is. The implementation is narrow, but it should keep the suppression-disclosure marker unsuppressible and still needs secops approval for the active-output filtering policy.
