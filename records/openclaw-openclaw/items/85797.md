## Summary

Keep open: the report is recent and credible, and current main still has a CLI image-generation path that can select the Codex OAuth route but surface the generic `No API key found for provider "openai-codex"` resolver error when the profile cannot materialize credentials; adjacent auth issues and PRs do not clearly own this exact CLI/tool-path divergence.

## What This Changes

The issue asks for `openclaw capability image generate --model openai/gpt-image-2` to use the configured `openai-codex` OAuth profile like the working `image_generate` tool route, or to emit an OAuth-specific failure.

## Best Possible Solution

Align CLI image generation with the same resolvable `openai-codex` OAuth credential path used by the working tool route, add a focused CLI regression plus live OAuth proof, and make unresolved OAuth profiles produce an OAuth-specific diagnostic instead of a static API-key message.

## Reproduction Assessment

No, not as a high-confidence live current-main repro. The issue provides a concrete `v2026.5.20` command/output, and source inspection shows the CLI can reach the Codex OAuth route before throwing the shared missing-key resolver error, but I did not prove it with live OAuth credentials.

## Solution Assessment

Unclear. The maintainable direction is not another API-key fallback; it should either share the tool route's working OAuth auth state or emit a precise OAuth-profile-unresolvable error with regression coverage.
