Keep open: the report describes a documented OAuth-supported image-generation path that still lacks a current-main CLI regression test, and the related open Codex image/auth issues are adjacent rather than a canonical replacement for this narrower CLI auth-resolution failure.

## What This Changes

The issue requests that `openclaw capability image generate` resolve configured `openai-codex` OAuth credentials for `openai/gpt-image-2` instead of reporting a missing API key when OAuth profiles exist.

## Best Possible Solution

Make the CLI image command use the same OAuth-capable auth resolution as the working `image_generate` tool path, with a regression for unset `OPENAI_API_KEY` plus an `openai-codex` OAuth profile.

## Solution Assessment

Yes. A narrow compatibility fix in the CLI/shared image runtime/OpenAI provider auth path is preferable to a new config option or broad fallback policy because docs already define Codex OAuth support for this model.
