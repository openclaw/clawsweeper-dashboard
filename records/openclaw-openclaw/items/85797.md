## Summary

Keep open: OpenClaw documents Codex OAuth as supported for `openai/gpt-image-2`, the report gives a concrete release failure for the CLI-only path, and the related open reports cover adjacent Codex image/auth failures but do not supersede this narrower CLI auth-resolution bug.

## What This Changes

The issue reports that `openclaw capability image generate --model openai/gpt-image-2` resolves `openai-codex` but requires a static API key instead of using the configured Codex OAuth profile.

## Best Possible Solution

Make the CLI image-generation command use the same OAuth-capable auth resolution as the working agent tool path, and add a regression proving `OPENAI_API_KEY` can be unset when an `openai-codex` OAuth profile is available for `openai/gpt-image-2`.

## Reproduction Assessment

No high-confidence current-main live reproduction is available here because the review environment does not have the reporter's Codex OAuth profile. The provided release command and source path are strong enough to treat this as source-reproducible and worth keeping open.

## Solution Assessment

Yes, a narrow fix should stay inside the existing CLI/shared image-runtime/OpenAI provider auth path rather than adding a new config option or fallback policy. The current docs already define Codex OAuth support, so this should be a compatibility repair with regression coverage.
