## Summary

Keep open. Current main and latest v2026.5.22 still route non-owning Codex app-server compaction through native `thread/compact/start`, ignore OpenClaw compaction model/provider overrides, and expose no force-local config; the timeout claim still needs live proof and the fix shape needs maintainer product/API direction.

## What This Changes

Request a clean OpenClaw config or environment control to force local or context-engine-owned Codex compaction for ChatGPT-OAuth sessions whose native remote compaction stalls on large contexts.

## Best Possible Solution

Define a maintainer-approved Codex compaction policy in the Codex plugin: either a documented force-local or force-context-engine option with tests, or a clear documented context-engine-owned escape path that avoids provider-name spoofing.

## Reproduction Assessment

No high-confidence live repro was run. Source inspection confirms the no-knob/native-routing part on current main, but not the reported 300s Cloudflare failure timing in a real ChatGPT-OAuth large-context session.

## Solution Assessment

Unclear. A force-local knob is plausible, but maintainers should first choose whether the supported fix is a Codex-specific config/env override, a context-engine-owned route, or an upstream Codex setting.
