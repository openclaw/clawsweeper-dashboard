## Summary

Keep open. Current main and the latest release still route non-owning Codex app-server compaction through native Codex server-side compaction and have no first-class force-local compaction setting, so the reported ChatGPT-OAuth timeout case still needs maintainer policy direction.

## What This Changes

The issue asks for a clean OpenClaw config or environment knob to force local Codex compaction for large ChatGPT-OAuth contexts that fail during remote compaction.

## Best Possible Solution

Define a narrow Codex compaction policy: either expose and document a supported force-local/force-context-engine option with schema, docs, and tests, or route the request upstream and document the supported OpenClaw workaround without relying on provider-name spoofing.

## Reproduction Assessment

No high-confidence live reproduction was run. Source and upstream Codex inspection support the native remote-compaction route and the missing OpenClaw force-local knob, but the 1 MB ChatGPT-OAuth timeout itself still needs live proof.

## Solution Assessment

Unclear. A force-local knob is plausible, but maintainers should first choose whether the durable fix belongs in OpenClaw config/env, an owning context-engine workflow, or upstream Codex provider settings.
