## Summary

Keep open. Current main and the latest release still route non-owning Codex app-server compaction through Codex native/server-side compaction, and the strict OpenClaw compaction schema still has no supported force-local switch for this ChatGPT-OAuth timeout case.

## What This Changes

The issue requests a supported OpenClaw config or environment knob to force local Codex compaction for large ChatGPT-OAuth sessions that fail through the remote compact path.

## Best Possible Solution

Define a maintainer-approved Codex compaction policy that avoids provider-name spoofing: either add a narrow documented force-local/force-context-engine option with coverage, or document the owning context-engine route as the supported escape hatch and pursue any needed upstream Codex setting.

## Reproduction Assessment

No high-confidence live reproduction was run. Source and upstream Codex inspection do reproduce the key routing shape: non-owning Codex compaction stays on the native remote/server-side path, and current config has no supported force-local switch.

## Solution Assessment

Unclear. A force-local knob is plausible, but maintainers should first decide whether the durable contract belongs in OpenClaw config, the context-engine ownership path, or upstream Codex configuration.
