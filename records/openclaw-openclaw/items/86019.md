## Summary

Keep open. Current main and the latest release still route Codex app-server compaction through native `thread/compact/start` and explicitly ignore local OpenClaw compaction overrides; the merged OAuth fallback fixes address direct-API auth routing, not the requested force-local policy for ChatGPT-OAuth remote compaction timeouts.

## What This Changes

The issue asks for a supported way to force local compaction for ChatGPT-OAuth Codex sessions when remote Codex compaction stalls or disconnects on large contexts.

## Best Possible Solution

Maintainers should choose the Codex compaction contract: either add a narrow supported force-local or local-summarizer escape hatch with schema/docs/tests, or explicitly keep native Codex compaction as the only supported path and document recovery expectations.

## Reproduction Assessment

No current-main live repro was run. The issue has reporter-provided live evidence on v2026.5.22 and source-level proof that current Codex-backed compaction still follows the native remote-capable provider path rather than a supported local override.

## Solution Assessment

Unclear. A force-local knob is plausible, but current docs and policy make Codex native compaction the owner boundary, and config/env additions require maintainer approval before this can be called the best fix.
