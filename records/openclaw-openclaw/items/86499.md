## Summary

Keep open: current main still collapses DeepSeek const-literal unions to the first variant, and this PR is a plausible narrow fix, but it should not merge yet because the new enum branch drops nullable-union semantics and the external PR lacks real behavior proof.

## What This Changes

The PR changes `normalizeDeepSeekSchema` so DeepSeek `anyOf`/`oneOf` const string literal unions are converted into `enum` schemas instead of selecting only the first variant.

## Best Possible Solution

Preserve all const-string options while carrying forward the existing nullable marker, add focused provider-tools regression coverage, then land with redacted real DeepSeek/tool-schema output showing the fixed schema.

## Reproduction Assessment

Yes. Current main is source-reproducible because `normalizeDeepSeekSchema` strips the union key and selects only the first non-null variant, so a multi-const union becomes one const value.

## Solution Assessment

No, not as submitted. The enum conversion is the right narrow shape for pure const-string unions, but it should preserve nullable unions and include focused regression coverage.
