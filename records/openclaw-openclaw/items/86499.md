## Summary

Keep open: the PR is aimed at a real current-main DeepSeek schema bug, but the submitted enum fast path skips the existing nullable handling and can turn nullable const-literal unions into non-nullable schemas.

## What This Changes

The PR adds an enum conversion branch to DeepSeek tool-schema normalization for `anyOf`/`oneOf` unions whose non-null variants are string `const` schemas.

## Best Possible Solution

Preserve all string-const enum values while carrying forward `nullable: true` when a null variant was present, then add focused SDK and DeepSeek provider regression coverage before merge.

## Reproduction Assessment

Yes. Current main is source-reproducible because `normalizeDeepSeekSchema` strips the active `anyOf`/`oneOf` key and selects only the first non-null variant, so a multi-value string `const` union collapses to the first value.

## Solution Assessment

No, not as submitted. The enum conversion is the right narrow shape for pure const-string unions, but it must preserve nullable unions and include focused regression coverage.
