## Summary

Keep open: current main has structured cron failure handling, but the reported gap is not fully covered because prose-only blocked/incomplete cron output can still remain an `ok` run by design, and the issue asks for new capability/preflight semantics rather than a cleanup close.

## What This Changes

Requests stronger cron/runtime capability checks and failure status handling when scheduled jobs lose expected file, shell, or mail tools.

## Best Possible Solution

Add a structured capability/failure signal path for cron jobs, likely job-declared requirements plus preflight or runner metadata, so missing file/shell/mail capabilities produce `error` with diagnostics without relying on prose heuristics.

Source inspection does show the core status concern is plausible because prose-only blocked output is intentionally not promoted to an error without structured metadata.
