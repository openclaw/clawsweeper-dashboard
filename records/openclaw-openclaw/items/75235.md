## Summary

Keep open. Current main still has the reported failure class: failed assistant turns are removed before Anthropic-compatible replay, the converted Anthropic `messages` array can still be empty, and the existing auto-reset logic only recognizes Gemini function-call corruption and role-ordering conflicts. The related reports and PRs cover adjacent empty-payload, blank-user, and fresh-session UX cases, but they do not supersede this leading-assistant/no-user transcript recovery bug.

## What This Changes

Request to recover sessions whose transcript has no replayable user turn or starts with failed assistant-only entries, and to provide a clearer post-reset readiness signal.

## Best Possible Solution

Add a focused corruption recovery path that detects assistant-only or zero-replayable-user session history, archives the bad transcript through the existing reset mechanism, clears the active session, and returns the existing friendly corruption-reset copy. Keep the broader fresh-session greeting/readiness behavior aligned with the reset command contract and #75314 rather than burying it in the Anthropic transport layer.

## Reproduction Assessment

Yes. The issue provides a concrete JSONL shape, and current source inspection shows a source-level reproduction path: failed assistant turns are filtered from replay, no user turn remains, and the Anthropic-compatible payload can still be sent with an empty `messages` array.

## Solution Assessment

Yes for the corruption loop: using the existing session reset/archive path is the narrowest maintainable fix. A transport-only fallback is not sufficient because it can mask the 400 while leaving the corrupt transcript and ambiguous reset UX in place.
