## Summary

Keep open. Current main still has the reported failure class: assistant-only failed turns are filtered out before Anthropic replay, the resulting empty Anthropic `messages` payload is still passed through, and the runner reset logic does not recognize the `messages: at least one message is required` schema error as recoverable session corruption.

## What This Changes

Request to recover from leading-assistant/no-user corrupted transcripts by resetting the active session and to improve the post-reset readiness signal.

## Best Possible Solution

Add a focused recovery path that detects assistant-only or zero-replayable-user session history either before provider dispatch or when the Anthropic schema error is returned, archives or resets through the existing session-reset mechanism, clears the active session store entry, returns the existing friendly corruption-reset copy, and coordinates any broader greeting/readiness UX with #75314 rather than mixing in unrelated startup behavior.

## Reproduction Assessment

Yes. The issue provides a concrete transcript shape, and current source inspection shows the source-level reproduction path: failed assistant turns are removed from replay, no replayable user turn remains, Anthropic receives an empty `messages` array, and the current reset detectors do not catch that schema failure.

## Solution Assessment

Yes. A narrow session-corruption detector or read-time repair/reset guard is the best fix boundary; adding only an Anthropic empty-message fallback would be less maintainable because it could leave the corrupted transcript active.
