## Summary

Keep open. Current main still has the reported failure class: failed assistant turns are removed from replay before the Anthropic-compatible payload is built, the payload builder still allows an empty `messages` array, and the auto-reset logic does not recognize the Anthropic `messages: at least one message is required` schema failure. Related open items cover adjacent empty-payload and fresh-session UX cases, but they do not supersede the leading-assistant/no-user transcript reset requested here.

## What This Changes

The issue asks OpenClaw to detect leading-assistant or no-user-message corrupted sessions, archive and reset them automatically, return the existing friendly corruption-reset message, and improve post-reset readiness feedback.

## Best Possible Solution

Add a focused recovery path that treats assistant-only or zero-replayable-user transcripts as terminal session corruption, archives the bad transcript through the existing reset mechanism, clears the active session, and returns the existing friendly corruption-reset copy. Keep any reset-readiness UX change aligned with the fresh-session greeting work in #75314 rather than masking the poisoned transcript with a transport-only fallback.

## Reproduction Assessment

Yes. The issue gives a concrete transcript shape, and current source inspection shows the same source-level path: failed assistant turns are filtered from replay, no user turn remains for Anthropic, and the auto-reset detector does not catch the resulting schema error.

## Solution Assessment

Yes. A narrow runtime reset or read-time repair guard is the best fix boundary; a transport-only empty-message fallback would avoid one provider 400 but could leave the corrupted session and ambiguous reset UX in place.
