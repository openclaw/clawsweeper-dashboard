## Summary

Keep open. Current main still has the reported failure class: assistant-only error turns can be filtered out before Anthropic-compatible transport, yielding an empty provider message list, and the auto-reset path does not recognize the Anthropic `messages: at least one message is required` schema failure. The existing ClawSweeper comment and related reports are useful context, but the leading-assistant/no-user reset and reset-readiness UX remain distinct from the adjacent empty-payload and fresh-session issues.

## What This Changes

The issue requests automatic recovery for assistant-only/no-user session transcripts that cause Anthropic-compatible providers to reject replay with an empty messages payload, plus clearer reset readiness feedback.

## Best Possible Solution

Add a focused recovery path that treats zero-replayable-user or assistant-only failed transcripts as terminal session corruption, archives the bad transcript through the existing reset mechanism, clears the active session, returns the existing friendly corruption-reset copy, and keeps any reset-readiness UX aligned with the existing startup/reset command contract.

## Reproduction Assessment

Yes. The issue gives a concrete transcript shape, and current source inspection shows a source-level reproduction path: failed assistant turns are filtered before Anthropic transport, no replayable user turn remains, and the existing reset detector does not catch the resulting Anthropic schema error.

## Solution Assessment

Yes. A narrow reset or read-time repair guard is the best fix boundary; a transport-only empty-message fallback would avoid one 400 but could leave the poisoned session and ambiguous reset UX in place.
