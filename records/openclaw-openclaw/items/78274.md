## Summary

Keep open: current main still has the Feishu group dispatch path and a documented group/channel visible-reply default that can explain `replies=0`, but the reported wrong-session routing for a second bound agent has not been disproven or covered by a focused regression test.

## What This Changes

Feishu group messages bound to a non-main agent should run in that agent's Feishu group session and send visible replies back to Feishu when the operator expects legacy automatic replies.

## Best Possible Solution

Add a focused Feishu group multi-agent diagnostic or regression test that distinguishes automatic-reply suppression from lost `SessionKey`/agentId propagation, then either fix the routing path or point the operator to the existing `messages.groupChat.visibleReplies: "automatic"` behavior.

## Reproduction Assessment

No, not at high confidence. Source and docs show a current-main path where group messages can process successfully but produce `replies=0` under the `message_tool` default; a live Feishu run or targeted regression is still needed to prove the reported wrong-session dispatch.

## Solution Assessment

Unclear. Changing the group default back would be a product policy decision, so the narrow maintainable path is first to prove whether Feishu loses `SessionKey`/agentId or whether this is documented reply-visibility behavior needing clearer diagnostics/config guidance.
