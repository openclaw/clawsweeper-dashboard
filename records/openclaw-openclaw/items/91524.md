## Label Justifications

- `P1`: An established automated channel workflow is unusable for attachments and affects real scheduled deliveries.
- `impact:message-loss`: Requested non-interactive Zalo media messages never reach their target.

## Summary

Current main and the latest v2026.6.5 release retain the reported failure shape: non-interactive sends use direct process-local delivery, and attachment uploads can await zca-js callbacks indefinitely. The issue remains necessary and needs channel-owner review plus live Zalo validation.

## What This Changes

Fix Zalo personal-channel media sends that hang in non-interactive CLI, agent, and cron contexts while interactive sends succeed.

## Best Possible Solution

Keep the repair within the zalouser plugin: route attachments through a proven listener-capable authenticated session, bound and clean up the complete upload/send operation, and verify successful CLI, cron, and interactive delivery against a real Zalo account.

## Reproduction Assessment

Yes, at high confidence from source and the detailed v2026.6.1 reproduction: current main still combines direct process-local delivery with an unbounded callback-dependent attachment upload. Live current-main confirmation requires an authenticated Zalo account.

## Solution Assessment

No final implementation has been proposed, and a timeout-only repair would be incomplete. The narrow maintainable solution is to establish the channel-owned session route needed by zca-js uploads while retaining bounded failure and cleanup.
