## Summary

Keep open; current main and the latest release still route non-interactive zalouser media sends through the direct adapter into zca-js attachment upload, and the only related open PR is inbound-photo work rather than this outbound upload hang.

## What This Changes

The issue reports that zalouser media/file sends hang and can exit 13 in non-interactive CLI, agent-message, and cron contexts while text sends and interactive media replies still work.

## Best Possible Solution

Make non-interactive zalouser media sends use a valid live upload path and return a platform receipt, with a bounded actionable error when the required upload/listener callback path is unavailable.

## Reproduction Assessment

Yes, at source level rather than from an independent live run: current main still routes non-interactive zalouser media through the direct adapter into zca-js attachment upload, and zca-js waits on upload callbacks for non-image attachments. A live Zalo account is still needed to prove the exact hang and any chosen fix.

## Solution Assessment

Unclear until maintainers choose the session ownership policy. The current direct upload path without bounded failure is not sufficient; the repair should address the zalouser upload/listener boundary rather than local media validation or dry-run behavior.
