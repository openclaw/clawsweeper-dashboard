## Summary

Keep this PR open. The human-day direction still addresses real current-main inconsistencies, but the current PR head is not merge-ready because it regresses current main's session freshness lifecycle contract and stale-safe timestamp handling.

## What This Changes

The PR adds shared human-day helpers and applies them to daily session reset freshness, session-memory artifacts, memory-flush paths, usage daily buckets, WhatsApp session snapshots, and related tests.

## Best Possible Solution

Keep the shared human-day abstraction, but rebase it onto current main while preserving the existing freshness contract: daily reset should compare the user-local reset cycle for sessionStartedAt, idle expiry should use lastInteractionAt, invalid or future timestamps should remain stale-safe, and usage bucket modes should stay aligned with caller date interpretation.

## Reproduction Assessment

Yes. Current main has focused unit coverage for lifecycle-based freshness and future timestamp rejection, and the PR head's reset-policy implementation removes the inputs and guards those tests exercise.

## Solution Assessment

No as proposed. The shared helper is the right maintainable direction, but the reset-policy integration must be layered onto current main's lifecycle inputs instead of replacing them with updatedAt-only freshness.
