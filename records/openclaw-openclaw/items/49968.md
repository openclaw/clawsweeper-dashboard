## Summary

Keep this PR open. Current main has not absorbed the human-day behavior, and the direction addresses a real daily-boundary mismatch, but the current head is not merge-ready because one freshness path can still fall back to host timezone, the session-memory docs still describe the old boundary, and the external real-behavior proof gate is not satisfied.

## What This Changes

The PR adds shared human-day time helpers and applies configured user-timezone day keys to session reset freshness, session-memory artifacts, memory flush paths, usage summaries, tests, and changelog text.

## Best Possible Solution

Land one coordinated human-day implementation after daily freshness cannot silently omit config, user-facing docs match the new timezone semantics, and redacted after-fix runtime proof is added.

## Reproduction Assessment

Yes. Source inspection of PR head shows evaluateSessionFreshness uses params.cfg for human-day reset keys while the gateway requested-session caller still omits cfg; with host TZ UTC, userTimezone Asia/Shanghai, and daily reset at 4, that entry point can still compare host-day cycles.

## Solution Assessment

No. The shared helper direction is reasonable, but the current patch needs the cfg contract/caller gap fixed, session-memory docs aligned, and real behavior proof added before it is the maintainable solution.
