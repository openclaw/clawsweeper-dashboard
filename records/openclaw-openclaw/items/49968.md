## Summary

Keep open. The requested human-day cleanup is still useful because current main still has mixed host/local/UTC daily-boundary behavior, but this PR should not merge as-is because its reset-policy changes drop current-main freshness safeguards for session lifecycle timestamps and future timestamps.

## What This Changes

The PR adds a shared human-day time helper and applies it to session reset freshness, session-memory artifacts, memory-flush paths, usage daily buckets, and related WhatsApp/gateway/session tests.

## Best Possible Solution

Keep the shared human-day direction, but rebase onto current main and preserve the existing freshness contract: daily staleness should use `sessionStartedAt`, idle staleness should use `lastInteractionAt`, invalid/future timestamps should be stale-safe, and the human-day cycle comparison should be layered onto those inputs rather than replacing them.

## Reproduction Assessment

Yes. Current main has focused unit tests for lifecycle and future-timestamp freshness behavior, and comparing PR head against those tests shows the PR removes the inputs and guards those tests exercise.

## Solution Assessment

No as proposed. A shared helper for user-local day boundaries is the right shape, but the reset-policy part must be adapted to current main's lifecycle semantics before this is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve lifecycle timestamps in freshness checks:** `src/config/sessions/reset-policy.ts:95-103`
  - body: This change drops `sessionStartedAt` and `lastInteractionAt` from `evaluateSessionFreshness` and computes daily and idle staleness from `updatedAt`. Current main intentionally keeps those timestamps separate so non-interactive/system updates do not extend reset windows; after this change, a session that started before the daily boundary or has been idle can remain fresh if `updatedAt` was bumped. Keep the lifecycle inputs and apply the human-day cycle comparison to `sessionStartedAt` while idle expiry uses `lastInteractionAt`.
  - confidence: 0.91
- **[P2] Reject future activity timestamps before comparing cycles:** `src/config/sessions/reset-policy.ts:112-116`
  - body: `resolveSessionActivityTimestamp` now accepts any finite number, so a corrupt or future `updatedAt` can produce a reset cycle after `now` and keep the session fresh. Current main treats future timestamps as invalid/stale-safe; retain the `now` and non-negative guard when adapting this helper.
  - confidence: 0.87
