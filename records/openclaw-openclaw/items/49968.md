## Summary

Keep open. The PR still has a blocking consistency bug because one gateway requested-session path omits cfg from the new human-day freshness calculation, and the external after-fix real behavior proof gate is still unmet; current main also does not already implement this change.

## What This Changes

The PR adds shared human-day helpers and applies configured user-timezone day semantics to session reset, session-memory, memory-flush, and usage daily buckets.

## Best Possible Solution

Land the shared human-day approach only after every daily freshness caller passes cfg or the API prevents silent host fallback, the timezone docs are aligned, and the contributor adds redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection of PR head shows the gateway requested-session path loads cfg but omits it from evaluateSessionFreshness, so with host TZ UTC, userTimezone Asia/Shanghai, and daily reset at 4, that entry point can still compare host-day cycles.

## Solution Assessment

No. The shared helper is a reasonable direction, but the implementation is not the narrowest maintainable fix while cfg remains optional and one gateway caller silently falls back to host timezone; docs also need to describe the expanded userTimezone semantics.
