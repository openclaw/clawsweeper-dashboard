## Summary

Keep open: the human-day direction is useful and not already on current main, but the current PR head still leaves the gateway requested-session freshness path on host-timezone fallback, does not update the published session-memory date docs, and only provides test/CI claims rather than after-fix real behavior proof.

## What This Changes

The PR adds shared human-day helpers and rewires session reset, session-memory, memory-flush, usage buckets, tests, and changelog text toward configured user-timezone daily boundaries.

## Best Possible Solution

Land one coordinated human-day implementation after every daily freshness caller passes cfg or the API prevents silent fallback, the timezone/session-memory docs are updated, and redacted real behavior proof is added.

## Reproduction Assessment

Yes. Source inspection of PR head shows the gateway requested-session path loads cfg but omits it from evaluateSessionFreshness; with host TZ UTC, userTimezone Asia/Shanghai, and daily reset at 4, that entry point can still compare host-day cycles.

## Solution Assessment

No. The shared human-day helper is a reasonable direction, but the implementation is not merge-ready while cfg remains optional and one caller silently falls back to host timezone; docs also need to describe the expanded userTimezone behavior.
