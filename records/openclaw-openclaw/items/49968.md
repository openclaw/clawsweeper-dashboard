## Summary

Keep open. Current main still lacks the central human-day behavior, and PR head remains not merge-ready because one requested-session freshness path can still fall back to host timezone, the published session-memory docs still describe host-local dates, and the external real behavior proof gate is unmet.

## What This Changes

The PR adds shared human-day time helpers and applies them to daily session reset, session-memory artifacts, memory flush paths, usage day bucketing, tests, and changelog text.

## Best Possible Solution

Land one coordinated human-day implementation after every daily freshness entry point receives cfg or the API prevents silent fallback, session-memory/timezone docs are aligned, and redacted real behavior proof is added.

## Reproduction Assessment

Yes. Source inspection of PR head shows evaluateSessionFreshness now uses params.cfg for human-day reset keys while the gateway requested-session caller still omits cfg; with host TZ UTC, userTimezone Asia/Shanghai, and daily reset at 4, that entry point can still compare host-day cycles.

## Solution Assessment

No. The shared human-day helper direction is reasonable, but the implementation is not merge-ready while cfg remains optional for daily freshness, one caller silently falls back to host timezone, docs still describe the old boundary, and real behavior proof is absent.
