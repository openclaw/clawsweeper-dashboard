## Summary

Keep open: the reported failure matches current source. Voice-call consults run with `messageProvider: "voice"`, exec approval followups preserve that as the followup `channel`, and the gateway `agent` handler still rejects `voice` as an unknown channel before the approved continuation can resume.

## What This Changes

The issue requests that voice-originated exec approval continuations resume through a supported internal/session path instead of failing with `unknown channel: voice`.

## Best Possible Solution

Treat voice as a supported internal non-delivery agent source, or map voice approval followups to an internal continuation before validation, while preserving `messageProvider: "voice"` for tool policy and avoiding chat delivery.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: voice consults set `messageProvider` to `voice`, approval followup passes that as `channel`, and current gateway validation rejects it with the reported error before the session continuation can run.

## Solution Assessment

Yes. The narrow maintainable fix is to support `voice` as an internal non-delivery source or translate it to an existing internal continuation, not to add a user config option or make `voice` a deliverable chat channel.
