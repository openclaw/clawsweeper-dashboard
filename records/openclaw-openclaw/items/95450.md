## Summary

Keep open: this PR targets a real Tavily SecretRef fallback bug, but the branch is not merge-ready because it passes an invalid resolver mode and then treats every unavailable configured SecretRef as missing before the ambient `TAVILY_API_KEY` fallback. The stronger open fix path at https://github.com/openclaw/openclaw/pull/95110 covers the same issue, but it is not merged or proof-positive enough to safely close this PR as superseded.

## What This Changes

The PR changes Tavily API-key normalization so unresolved configured SecretRefs return `undefined` and the existing `process.env.TAVILY_API_KEY` fallback can run, with one Tavily tools test for an env-backed SecretRef fallback.

## Best Possible Solution

Use the guarded inspect-mode resolver shape: preserve unavailable ref metadata, allow fallback only for an allowed env-backed `TAVILY_API_KEY` ref, block other refs, add negative regression tests, and require redacted real command proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main calls strict Tavily SecretRef normalization before the documented `TAVILY_API_KEY` fallback, and both Tavily search and extract use that resolver; I did not run the live Tavily command in this read-only review.

## Solution Assessment

No. Inspect mode is the right direction, but this PR uses an invalid mode value and loses the unavailable-ref metadata needed to block unsafe ambient fallback.
