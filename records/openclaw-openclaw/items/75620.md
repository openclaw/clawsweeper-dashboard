## Summary

Keep this PR open. The performance direction matches the cold plugin inspect boundary, but the latest head introduces a concrete setup auto-enable regression and exact-head checks are red, so it is not ready to merge or automerge.

## What This Changes

The PR threads a reusable cold manifest registry/runtime context through plugin status, inspect, metadata snapshot, provider compat, setup auto-enable, plugin auto-enable, tests, and the changelog.

## Best Possible Solution

Keep the cold status/inspect registry reuse, but do not reuse an allowlist-scoped manifest registry for setup auto-enable unless it is guaranteed to include the setup plugin IDs that may need to be newly enabled.

## Reproduction Assessment

Yes. The exact-head GitHub check annotation reproduces the failure in `src/config/plugin-auto-enable.core.test.ts:233`, where browser is no longer added to a restrictive allowlist from a non-disabled plugin entry.

## Solution Assessment

No. The overall reuse strategy is narrow and maintainable, but this specific setup auto-enable reuse path is unsafe because it can hide plugins that are not already present in the current allowlist-scoped registry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve setup auto-enable discovery outside the allowlist:** `src/config/plugin-auto-enable.shared.ts:666`
  - body: Passing the already-resolved manifest registry into setup auto-enable reuses a registry built from the current restrictive `plugins.allow`. That registry can omit the very plugin setup probes we need to discover, so a config with `plugins.allow: ["telegram"]` and `plugins.entries.browser: {}` no longer auto-enables `browser`; exact-head CI fails the existing test at `src/config/plugin-auto-enable.core.test.ts:233`. Keep this lookup scoped to `setupPluginIds` or ensure the reused registry is a superset that includes those IDs.
  - confidence: 0.93
