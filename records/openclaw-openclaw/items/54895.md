## Summary

Keep open. Current main and v2026.5.22 appear to have improved the wait-timeout result path for active runs, but the central request still remains: there is no agent-callable `sessions_send` progress stream, `sessions_subscribe` tool, or buffered `agent.events` read API for a caller session to observe a peer run.

## What This Changes

The issue requests inter-session progress streaming for long-running `sessions_send` calls, especially ACP/Codex sessions where the target run continues after the caller's wait window expires.

## Best Possible Solution

Add an additive, permission-checked, sanitized, bounded run-event observation surface for delegated session runs, then expose it through either `sessions_subscribe` or an explicit `sessions_send` progress option.

## Reproduction Assessment

Yes, at source level. Current main has no `sessions_subscribe`, no `streamProgress` parameter, no `agent.events` read method, and the A2A path remains completion-only after `sessions_send`.

## Solution Assessment

Unclear pending maintainer design. A separate `sessions_subscribe` or buffered `agent.events` path is plausible, but the best solution must settle authorization, redaction, buffering, retention, throttling, and overlap with the related progress-event issues first.

## Risks / Open Questions

- A run-event observation API needs explicit authorization so one session cannot inspect unrelated runs, sibling sessions, or arbitrary run IDs.
- Forwarding tool arguments, tool results, assistant deltas, or partial replies can expose sensitive intermediate data unless payloads are sanitized and redacted.
- The design overlaps related progress-event work in https://github.com/openclaw/openclaw/issues/52826 and https://github.com/openclaw/openclaw/issues/74845, so maintainers should avoid creating competing event streams.
- Channel-visible progress forwarding needs retention, throttling, and delivery-policy decisions, especially for Telegram topics and other rate-limited transports.
