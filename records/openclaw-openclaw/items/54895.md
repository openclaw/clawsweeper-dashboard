## Summary

Keep open. Current main and v2026.5.19 still lack an agent-callable inter-session progress stream for long-running `sessions_send`; the wait-timeout path is partly softened to `accepted`, but callers still cannot observe the target run's tool, assistant, or lifecycle progress before final A2A handling.

## What This Changes

The issue requests inter-session progress streaming or subscription support so `sessions_send` callers can observe long-running target-session work before the final announce.

## Best Possible Solution

Add an additive, permission-checked core run-event observation surface with bounded sanitized buffering, then expose it through either `sessions_subscribe` or an explicit `sessions_send` progress option while preserving accepted delivery and final announce behavior.

## Reproduction Assessment

Yes, at source level rather than through a live ACP run. Current main has no `sessions_subscribe`, no `streamProgress` parameter, no `agent.events` method, and the A2A path remains final-only after `sessions_send`.

## Solution Assessment

Unclear pending maintainer design. A separate `sessions_subscribe` or buffered `agent.events` path is plausible, but the best implementation must settle authorization, redaction, buffering, retention, throttling, and channel-delivery policy first.
