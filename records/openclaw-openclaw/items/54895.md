## Summary

Keep open. Current main still lacks a caller-visible inter-session progress stream for long-running `sessions_send` runs; adjacent ACP parent-streaming and session-subscriber paths do not give the calling agent a permission-checked way to observe a peer run it started.

## What This Changes

Request an inter-session progress streaming or subscription capability for `sessions_send` so callers can observe long-running ACP/Codex session work before final completion.

## Best Possible Solution

Define an additive, permission-checked run-progress observation contract for inter-session sends, then expose it through a focused `sessions_subscribe` tool or explicit `sessions_send` progress option while keeping delivery acceptance separate from wait timeout.

## Reproduction Assessment

Yes, at source level. Current main shows `sessions_send` has only fire-and-forget or wait semantics, no event subscription option, and no gateway `agent.events` API for the caller to poll or stream progress.

## Solution Assessment

Unclear. `sessions_subscribe` is a plausible narrow API, but maintainers still need to decide authorization, redaction, buffering, retention, and whether this should align with the broader structured process-event work.
