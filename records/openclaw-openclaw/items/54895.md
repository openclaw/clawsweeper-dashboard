## Summary

Keep open. Current main partially fixes the soft wait-timeout symptom by returning `accepted` for non-terminal `sessions_send` waits, but it still lacks inter-session progress streaming or an agent-callable run-event read/subscription API.

## What This Changes

Request an inter-session progress stream for long-running `sessions_send` runs so the caller can observe target-run tool, assistant, and lifecycle events before final completion.

## Best Possible Solution

Add an additive, permission-checked run-event observation surface with bounded buffering and explicit delivery policy, then expose it through `sessions_subscribe` or an opt-in `sessions_send` progress mode while preserving the existing accepted/final-announce behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live ACP run. A long-running `sessions_send` run can be accepted and kept pending after the wait window, but current main has no caller-visible progress channel before completion.

## Solution Assessment

Unclear. The requested direction is valid, but the final API shape needs maintainer design; raw forwarding of target-session streams would not be safe without authorization, redaction, buffering, retention, and throttling policy.
