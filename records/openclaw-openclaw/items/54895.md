## Summary

Keep open. Current main and v2026.5.18 still lack an agent-callable inter-session progress stream for `sessions_send`; the delivery-timeout part is now softened to `accepted` for active runs, but subscription/progress visibility, buffering, authorization, redaction, and channel policy remain unresolved.

## What This Changes

The issue requests live progress visibility for long-running `sessions_send` delegations through `sessions_subscribe`, a `streamProgress` option, or a gateway run-event read API.

## Best Possible Solution

Design an additive, permission-checked run-event observation surface with bounded buffering and sanitized payloads, then expose it through `sessions_subscribe` or an explicit `sessions_send` progress option while preserving accepted delivery and final announce behavior.

## Reproduction Assessment

Yes, at source level. Current main has no `sessions_subscribe`, no `streamProgress` parameter, no `agent.events` method, and the A2A flow remains final-reply oriented rather than streaming peer-run progress.

## Solution Assessment

Unclear pending maintainer design. The requested direction is valid, but the final shape needs explicit authorization, redaction, buffering, retention, throttling, and channel-delivery policy.
