## Summary

Keep open. Current main and v2026.5.18 still provide `sessions_send` as send plus optional wait plus final A2A announce, but there is no agent-callable progress stream, subscription tool, or buffered run-event read API for the caller session.

## What This Changes

The issue requests caller-visible progress streaming or subscription support for long-running `sessions_send` runs, especially ACP/Codex sessions that continue working after the caller's wait window expires.

## Best Possible Solution

Design an additive, permission-checked run-event observation surface with bounded buffering and redacted event payloads, then expose it through `sessions_subscribe` or an explicit `sessions_send` progress option while preserving accepted delivery and final announce behavior.

## Reproduction Assessment

Yes, at source level rather than with a live ACP run. Current main has no `sessions_subscribe`, no `streamProgress` parameter, no `agent.events` method, and A2A delivery remains final-only after `sessions_send`.

## Solution Assessment

Unclear pending maintainer design. The requested direction is valid, but the final API needs explicit authorization, redaction, buffering, retention, throttling, and channel-delivery policy before implementation.
