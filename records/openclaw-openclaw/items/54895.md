## Summary

Keep open. Current main still lacks inter-session progress streaming for long-running `sessions_send`: the tool exposes only fire-and-forget or synchronous wait behavior, wait expiry is still returned as `status: "timeout"`, A2A announce remains completion-only, and the gateway has no agent-callable run-event read or subscription API. The related Task Flow and ACP parent-streaming work are adjacent, but they do not let a caller observe a peer run started through `sessions_send`.

## What This Changes

Request inter-session progress streaming or subscription for `sessions_send` so callers can observe long-running ACP/Codex target-session progress instead of only receiving a wait timeout or final announce.

## Best Possible Solution

Add an additive, permission-checked core run-event observation surface, likely backed by bounded per-run event buffering, then expose it through either a `sessions_subscribe` tool or an explicit `sessions_send` progress option. The implementation should also separate accepted delivery from wait timeout and add regression coverage for long-running sends, event isolation, ACP/Codex behavior, and topic-bound delivery.
