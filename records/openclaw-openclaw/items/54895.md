## Summary

Keep open. Current main still lacks inter-session progress streaming for long-running `sessions_send` runs: the tool exposes only fire-and-forget or wait modes, wait expiry still returns `status: "timeout"`, A2A announce remains completion-only, and the gateway has no agent-callable run-event read/subscription method. The adjacent ACP parent-streaming path applies to `sessions_spawn`, not observing a peer run started through `sessions_send`.

## What This Changes

Request live progress visibility for long-running `sessions_send` runs so caller sessions can distinguish accepted delivery from wait timeout and observe target-session progress before final announce.

## Best Possible Solution

Add an additive, permission-checked core run-event observation surface, likely backed by bounded per-run event buffering, then expose it through either a `sessions_subscribe` tool or an explicit `sessions_send` progress option. The final design should also separate accepted delivery from wait timeout and cover event isolation, ACP/Codex behavior, and topic-bound delivery.

## Reproduction Assessment

Yes. The code path is enough to reproduce with high confidence: a `sessions_send` run longer than the wait timeout reaches `agent.wait`, returns `timeout`, and has no caller-side event subscription/read API to surface progress.

## Solution Assessment

Unclear. The requested direction is valid, but the exact API shape needs maintainer design; a bounded, authorized event-read/subscription seam is plausible, while directly forwarding raw tool and assistant streams without authorization/redaction would not be the best fix.
