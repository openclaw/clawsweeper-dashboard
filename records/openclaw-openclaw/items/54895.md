## Summary

Keep open. Current main and v2026.6.8 still lack a caller-visible progress stream or subscription/read API for long-running peer `sessions_send` runs; existing event routing serves target sessions, WebSocket recipients, or session subscribers rather than the internal caller that started the run.

## What This Changes

Request an inter-session progress stream or subscription API so `sessions_send` callers can observe long-running target-session runs instead of only fire-and-forget or wait-for-final behavior.

## Best Possible Solution

Define an additive, permission-checked run-progress observation contract with bounded buffering/replay/redaction, then expose it through `sessions_subscribe` or an explicit `sessions_send` progress option while preserving delivery acceptance semantics.

## Reproduction Assessment

Yes, at source level. Current main shows `sessions_send` only supports fire-and-forget or synchronous `agent.wait`, and the gateway has no caller-accessible run-event read/subscription method; I did not run a live Telegram/ACP scenario in this read-only review.

## Solution Assessment

Unclear final API. `sessions_subscribe` is plausible, but maintainers need to decide authorization, redaction, buffering/retention, replay, and alignment with broader process-event work.
