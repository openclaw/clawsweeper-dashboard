## Summary

Keep open. Current main still lacks an agent-callable inter-session progress stream for long-running `sessions_send`; adjacent channel streaming, ACP spawn parent streaming, and session UI events do not let the calling session observe a peer run started through `sessions_send`.

## What This Changes

The issue requests inter-session progress streaming for long-running `sessions_send` runs so callers can observe delegated session progress before final completion.

## Best Possible Solution

Add an additive, permission-checked run-event observation surface with bounded sanitized buffering, then expose it through either `sessions_subscribe` or an explicit `sessions_send` progress option while preserving accepted delivery and final announce behavior.

## Reproduction Assessment

Yes, at source level. Current main has no `sessions_subscribe`, no `streamProgress` parameter, no `agent.events` method, and the A2A path remains final-only after `sessions_send`.

## Solution Assessment

Unclear pending maintainer design. A separate `sessions_subscribe` or buffered `agent.events` path is plausible, but the best solution must settle authorization, redaction, buffering, retention, throttling, and overlap with the related progress-event issues first.

## Risks / Open Questions

- A run-event observation API needs explicit authorization so one session cannot inspect unrelated runs, sibling sessions, or arbitrary run IDs.

- Channel-visible forwarding needs retention, throttling, and delivery-policy decisions, especially for Telegram topics and other rate-limited transports.
