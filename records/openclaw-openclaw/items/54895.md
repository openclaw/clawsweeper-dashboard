## Summary

Keep open. Current main partially fixed the misleading soft-timeout result by returning `accepted` for non-terminal waits, but the central request remains missing: there is still no caller-authorized inter-session progress stream, `sessions_subscribe` tool, or `agent.events`/buffered run-event read API for a `sessions_send` originator to observe a delegated long-running run.

## What This Changes

Request an inter-session progress streaming or subscription capability for long-running `sessions_send` delegated runs, especially ACP/Codex-backed sessions.

## Best Possible Solution

Define one additive, permission-checked core run-event observation surface with bounded buffering, redaction, and retention, then expose it through `sessions_subscribe` or an explicit `sessions_send` progress mode with focused ACP/Codex and channel proof.

## Reproduction Assessment

Yes, at source level: current main has no `sessions_subscribe`, no `agent.events` read API, and no `sessions_send` progress option, while tests show only the soft-timeout result behavior is partially improved. I did not run a live Telegram/ACP reproduction in this read-only review.

## Solution Assessment

Unclear final API shape. The poll-based `sessions_subscribe` proposal is plausible, but maintainers need to decide authorization, redaction, buffering, retention, and channel delivery before this becomes a safe implementation task.
