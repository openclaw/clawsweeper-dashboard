## Summary

Keep open. Current main partially fixes the misleading wait-timeout outcome by returning `accepted` for non-terminal waits, but it still does not provide inter-session progress streaming or a run-event subscription/read API for agents using `sessions_send`.

## What This Changes

Add real-time progress visibility for long-running `sessions_send` runs, likely through a subscription/event stream or explicit streaming option.

## Best Possible Solution

Design an additive, permission-checked core run-event observation surface with bounded retention and redaction, then expose it through `sessions_subscribe` or a clearly scoped `sessions_send` progress option with regression coverage for long-running peer sends.

## Reproduction Assessment

Yes, by source inspection rather than a live ACP/Telegram run. Current `sessions_send` has no streaming or subscription input, A2A is final-only, and the gateway has no agent-callable run-event read API.

## Solution Assessment

Unclear because there is no patch and the proposed options are design alternatives. The safest direction is a bounded, authorized core event observation API first, then a tool-level surface on top of it.
