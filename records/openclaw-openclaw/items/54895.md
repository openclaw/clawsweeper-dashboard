## Summary

Keep open. Current main and v2026.6.10 partly improve the timeout wording by returning `accepted` for nonterminal wait expiry, but they still lack a caller-visible progress stream, `sessions_subscribe`, `agent.events`, or another buffered run-event read API for long-running peer `sessions_send` runs.

## What This Changes

The issue requests caller-visible progress streaming or subscription for long-running inter-session `sessions_send` runs.

## Best Possible Solution

Add an additive, permission-checked run-progress observation contract with bounded replay and redaction, then expose it through `sessions_subscribe` or a maintainer-approved `sessions_send` progress option.

## Reproduction Assessment

Yes, at source level. Current main lacks `sessions_subscribe`, `agent.events`, or another caller-readable buffered run-event API for a `sessions_send` caller; I did not run a live Telegram/ACP scenario in this read-only review.

## Solution Assessment

Unclear final API shape. A separate `sessions_subscribe` tool is plausible, but maintainers still need to decide the event API, authorization, redaction, retention, replay, and channel delivery semantics.
