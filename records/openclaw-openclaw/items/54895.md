## Summary

Keep open. Current main partially improves the timeout-reporting side by returning `accepted` for non-terminal wait timeouts, but it still has no inter-session run-progress stream, no `sessions_subscribe` tool, and no `agent.events`/buffered event-read API for a caller session to observe a delegated run.

## What This Changes

The issue requests inter-session progress visibility for long-running `sessions_send` runs, especially ACP/Codex sessions that continue working after the caller wait expires.

## Best Possible Solution

Define an additive, permission-checked core run-event observation surface with bounded buffering and redaction, then expose it through `sessions_subscribe` or an explicit `sessions_send` progress mode with focused regression and live Telegram/ACP proof.

## Reproduction Assessment

Yes, at source level: current main has no `sessions_subscribe`, no `agent.events` read API, and no progress option on `sessions_send`. I did not run a live Telegram/ACP reproduction in this read-only cleanup pass.

## Solution Assessment

Unclear final API shape. The proposed poll-based `sessions_subscribe` direction is plausible, but maintainers should decide authorization, redaction, buffering, retention, and channel delivery before implementation.
