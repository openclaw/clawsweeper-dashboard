## Summary

Keep open. Current main has the soft-timeout improvement for `sessions_send`, but it still lacks the requested inter-session progress streaming or an agent-callable run-event read/subscription API; related timeout and node-host streaming items do not own this exact peer-run observation gap.

## What This Changes

Request an inter-session progress streaming or subscription capability for `sessions_send` so callers can observe long-running ACP/Codex session progress before final delivery.

## Best Possible Solution

Add an additive, permission-checked run-event observation surface with bounded buffering, then expose it through `sessions_subscribe` or an explicit `sessions_send` progress option while preserving accepted/final-announce behavior.

## Reproduction Assessment

Yes, at source level rather than with a live ACP run. Current main has no `sessions_subscribe`, `streamProgress`, `agent.events`, or buffered caller-visible run-event path, while `sessions_send` and A2A announce still wait for completion/final delivery.

## Solution Assessment

Unclear. The requested direction is valid, but the final surface needs maintainer design for authorization, redaction, buffering, retention, and channel delivery policy before implementation.
