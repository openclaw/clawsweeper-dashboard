## Summary

Keep open. Current main and v2026.5.18 softened the misleading wait-timeout path by returning `accepted` for non-terminal waits, but they still do not provide a model-facing `sessions_subscribe`, `streamProgress`, or `agent.events` style progress surface for the caller session.

## What This Changes

The issue requests inter-session progress streaming or subscription support so callers can observe long-running `sessions_send` target runs before final completion.

## Best Possible Solution

Design an additive, permission-checked run-event observation surface with bounded redacted events, then expose it through `sessions_subscribe` or an explicit `sessions_send` progress option while preserving current accepted-delivery and final A2A announce behavior.

## Reproduction Assessment

Yes, at source level rather than with a live ACP run. Current main has no model-callable `sessions_subscribe`, `streamProgress`, or `agent.events` path, and the existing A2A flow waits for completion before visible delivery.

## Solution Assessment

Unclear pending maintainer design. The requested direction is valid, but the final API needs explicit authorization, redaction, buffering, retention, throttling, and channel-delivery policy before implementation.
