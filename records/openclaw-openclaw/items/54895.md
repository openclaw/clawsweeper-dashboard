## Summary

Keep open. Current main and v2026.5.18 still lack an agent-callable inter-session progress stream for long-running `sessions_send`; the delivery-timeout path is partly softened to `accepted`, but callers still cannot subscribe to target-run tool, assistant, or lifecycle progress before the final A2A announce.

## What This Changes

Request an inter-session progress streaming or subscription capability for long-running `sessions_send` runs, especially ACP/Codex-backed sessions.

## Best Possible Solution

Design an additive, permission-checked run-event observation surface with bounded sanitized buffering, then expose it through `sessions_subscribe` or an explicit `sessions_send` progress option while preserving accepted delivery and final announce behavior.

## Reproduction Assessment

Yes, at source level rather than through a live ACP run. Current main has no `sessions_subscribe`, no `streamProgress` parameter, no `agent.events` method, and the A2A path remains final-only after `sessions_send`.

## Solution Assessment

Unclear pending maintainer design. The requested direction is valid, but the final API needs explicit authorization, redaction, buffering, retention, throttling, and channel-delivery policy before implementation.
