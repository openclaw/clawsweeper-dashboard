## Summary

Keep this PR open, but do not merge it as-is. It fixes a real Feishu duplicate text/media path, yet the new extension-based voice detection is broader than the media layer's actual voice-note behavior and would drop captions for ordinary audio file sends.

## What This Changes

The PR changes Feishu `sendMedia` so text is not sent separately when the media is detected as voice audio by `audioAsVoice` or file extension.

## Best Possible Solution

Keep the duplicate suppression, but gate it on the same conditions that actually produce a Feishu voice-note delivery and preserve captions for regular audio/file media.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call Feishu `sendMedia` with non-empty `text` and voice media, and current main sends text before media; the PR also has a source-reproducible caption-loss case for non-voice `.mp3` media without `audioAsVoice`.

## Solution Assessment

No. The direction is narrow, but the implementation should align suppression with actual voice-note delivery and add focused regression tests plus the required changelog entry.
