## Summary

This should close as plugin-surface work, not an OpenClaw core bug: Lossless Claw is an external plugin, its current docs explicitly support `/lcm` slash/native commands rather than `openclaw lcm`, and OpenClaw already has generic plugin CLI/metadata APIs for a plugin to expose a root CLI if it wants one.

## What This Changes

The issue asks for the Lossless Claw `lcm` diagnostics surface to be reachable despite a restrictive `plugins.allow` configuration.

## Best Possible Solution

Keep OpenClaw core unchanged; use the existing Lossless Claw plugin’s `/lcm` slash/native commands, allowlist the plugin id `lossless-claw`, and have the plugin add root CLI descriptors if it later wants `openclaw lcm`.

## Reproduction Assessment

Yes for the unsupported root command: current source rejects unowned CLI roots and the external Lossless Claw manifest does not advertise `lcm` as a root CLI command. The claimed protected `config.patch` path is not reproduced in current source because `plugins.allow` is not in the protected replacement sets.

## Solution Assessment

Yes: the maintainable boundary is the existing plugin surface, not a core special case. OpenClaw already exposes `api.registerCli` and manifest command metadata for plugins that choose to own root CLI commands.
