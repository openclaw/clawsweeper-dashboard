## Summary

Keep open: the PR targets a real current-main MCP cleanup gap, but it is not merge-ready because it retires the session MCP runtime before proving the old embedded run has drained, and the supplied proof does not exercise an active-MCP timeout path.

## What This Changes

The branch adds an unconditional `retireSessionMcpRuntime` call in gateway session cleanup immediately after aborting the old embedded run and before the 15-second drain wait.

## Best Possible Solution

Retire stale MCP runtimes only after the embedded run drains, is explicitly force-cleared, or can defer disposal until active MCP leases release, with reset/delete timeout coverage and active-MCP real behavior proof.

## Reproduction Assessment

Yes at source level: current main can return from the session reset/delete timeout path without retiring the MCP runtime, and the PR moves retirement before the drain wait. I did not run the live WSL2/LadybugDB file-locking MCP repro in this read-only review.

## Solution Assessment

No: reaping in the gateway lifecycle owner is the right direction, but immediate pre-wait runtime disposal is not the best fix because active MCP tool work can still hold leases after abort. The safer path is drained, explicit force-clear, or deferred lease-release retirement semantics.
