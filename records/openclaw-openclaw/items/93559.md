## Summary

Keep open: the stale-MCP cleanup bug is real, but this branch is not merge-ready because it retires the session MCP runtime before the old embedded run has drained or been force-cleared, and it still lacks active-MCP reset/delete proof.

## What This Changes

The PR adds an unconditional `retireSessionMcpRuntime` call immediately after aborting a gateway session's embedded run and before waiting for that run to drain.

## Best Possible Solution

Retire session MCP runtimes only after the embedded run drains, is explicitly force-cleared, or disposal is deferred until active leases release, with reset/delete timeout coverage and active-MCP proof.

## Reproduction Assessment

Yes at source level: current main can return from the session reset timeout path without retiring the MCP runtime, and the PR moves retirement before the wait. I did not run a live WSL2/LadybugDB file-locking reset in this read-only review.

## Solution Assessment

No; this fixes a real cleanup gap in a plausible place, but pre-wait retirement can dispose MCP transports while the old run is still active. The safer solution is a drained, force-cleared, or deferred-lease retirement contract.
