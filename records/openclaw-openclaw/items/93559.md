## Summary

Keep open: the PR targets a real current-main stale MCP cleanup gap, but retiring the session MCP runtime before the embedded run has drained can close MCP transports while old in-flight tool work still owns the runtime.

## What This Changes

The PR adds an unconditional `retireSessionMcpRuntime` call before `waitForEmbeddedAgentRunEnd` in gateway session runtime cleanup, plus verbose logging for cleanup failures.

## Best Possible Solution

Retire MCP runtimes only after the embedded run drains, is explicitly force-cleared, or can defer disposal until active MCP leases release, with reset/delete timeout coverage and active-MCP proof.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 can return from the reset/delete timeout path without retiring the MCP runtime, while the PR moves retirement before proving the run ended. I did not run the live file-locking MCP scenario in this read-only review.

## Solution Assessment

No. The PR fixes a real cleanup gap in a plausible place, but immediate pre-wait disposal is not the best lifecycle-safe fix because active MCP work may still hold runtime leases.
