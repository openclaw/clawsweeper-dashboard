## Summary

Keep open: the linked stale-MCP-process issue is still open, and this PR repeats an ordering-only cleanup that can retire the MCP runtime before the embedded run has actually drained. Current source shows the timeout path is real, but the patch needs a lifecycle-safe retirement contract before merge.

## What This Changes

The PR adds a pre-wait `retireSessionMcpRuntime` call and verbose logging in `src/gateway/session-reset-service.ts` so gateway session cleanup tries to retire MCP processes before waiting for the embedded run to end.

## Best Possible Solution

Adopt a lifecycle-safe cleanup contract that retires MCP runtimes only after the run has drained, has been explicitly force-cleared, or can defer disposal until the final MCP lease releases, with reset and delete timeout coverage.

## Reproduction Assessment

Yes at source level: current main returns `UNAVAILABLE` without MCP retirement when the embedded-run drain wait times out. I did not run a live file-locking MCP child-process reset in this read-only review.

## Solution Assessment

No, not as submitted. The PR fixes a real cleanup gap in a plausible place, but retiring the MCP runtime before a drained, force-cleared, or deferred-lease contract can close transports under active tool work.
