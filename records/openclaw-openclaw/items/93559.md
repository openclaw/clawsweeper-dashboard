## Summary

Keep open: the branch targets a real gateway/MCP cleanup gap, but it is not merge-ready because it retires the MCP runtime before proving the old embedded run has drained and only supplies test/build/code-order proof rather than active-MCP runtime proof.

## What This Changes

The PR adds unconditional MCP runtime retirement in gateway session cleanup before waiting for the old embedded agent run to end.

## Best Possible Solution

Retire stale MCP runtimes only after the old run drains, is explicitly force-cleared, or disposal is deferred until active MCP leases release, with reset/delete timeout coverage and active-MCP real behavior proof.

## Reproduction Assessment

Yes at source level: current main can return from the reset/delete timeout path without retiring the MCP runtime, and the PR moves retirement before the drain wait. I did not run the live file-locking MCP repro in this read-only review.

## Solution Assessment

No: reaping in the gateway lifecycle owner is the right direction, but immediate pre-wait runtime disposal is not the best fix because active MCP tool work can still hold leases after abort. The safer path is drained, explicit force-clear, or deferred lease-release retirement semantics.
