## Summary

This PR still addresses a valid `skills info` lookup bug on current `main`, but the proposed resolver is not merge-ready because one fallback can silently choose the wrong case-only collision and the user-facing CLI fix lacks the required changelog entry.

## What This Changes

The PR updates the skills CLI formatter/tests to trim `skills info` input, add exact/case-insensitive/separator-normalized lookup, sanitize not-found output, and add lookup regression coverage.

## Best Possible Solution

Keep exact matches first, require unique matches for every fallback resolver path, add collision regression coverage, and add a single-line changelog fix entry before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a `SkillStatusReport` with two entries whose `name` or `skillKey` differ only by case, then call `formatSkillInfo` with a lower-cased token; the PR head's case-insensitive `.find()` branch returns the first match.

## Solution Assessment

No. The PR targets the right surface, but the narrow maintainable fix needs the case-insensitive fallback to use the same uniqueness rule as normalized lookup and needs the changelog entry.
