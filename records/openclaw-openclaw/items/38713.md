## Summary

Keep open: current main still uses exact-only `skills info` lookup, so the PR is not obsolete, but the PR head still has a blocking ambiguity bug in the case-insensitive fallback and lacks the required changelog entry.

## What This Changes

The PR adds normalized and case-insensitive `openclaw skills info` lookup, trims/sanitizes not-found output, and updates skills formatter tests.

## Best Possible Solution

Keep exact matches first, require every fallback resolver path to resolve exactly one candidate, add the case-collision regression, and add a single-line changelog fix entry before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a `SkillStatusReport` with two entries whose `name` or `skillKey` differ only by case, then call `formatSkillInfo` with a lowercase token; the PR head's case-insensitive `.find()` branch returns the first match.

## Solution Assessment

No. The PR targets the right formatter surface, but the fallback resolver is not maintainable until case-insensitive matches use the same uniqueness rule as normalized matches and the required changelog entry is added.
