## Summary

Keep open. Current main still has the exact-only lookup, so this PR is not obsolete, but the PR head still has a case-insensitive fallback that can select the first colliding skill and show the wrong setup/API-key guidance.

## What This Changes

The branch changes `openclaw skills info <name>` formatting and tests to trim requested names, add case-insensitive and separator-normalized skill lookup, sanitize not-found output, and cover the new lookup paths.

## Best Possible Solution

Update the resolver so exact matches still win, then both fallback paths collect matches and resolve only when the match is unique, with a regression for case-insensitive collisions and the required changelog entry before normal validation.

## Reproduction Assessment

Yes. Build a `SkillStatusReport` with two entries whose `name` or `skillKey` differ only by case, then call `formatSkillInfo` with a lower-cased token; the PR head's `.find()` branch returns the first match instead of treating it as ambiguous.

## Solution Assessment

No. The PR addresses the right CLI surface, but the case-insensitive fallback needs the same uniqueness handling already used by the normalized fallback before this is the narrow maintainable fix.
