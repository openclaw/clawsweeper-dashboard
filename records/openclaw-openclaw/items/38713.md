## Summary

Keep open: current main still has exact-only `skills info` lookup, and this PR targets the right CLI surface, but the PR head still silently selects the first case-insensitive collision and lacks the required changelog entry.

## What This Changes

The PR changes `openclaw skills info <name>` to trim requested names, add case-insensitive and separator-normalized skill lookup, sanitize not-found output, and update formatter tests.

## Best Possible Solution

Keep exact matches first, require unique matches for every fallback resolver path, add the collision regression, and add a single-line changelog fix entry before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: build a `SkillStatusReport` with two entries whose `name` or `skillKey` differ only by case, then call `formatSkillInfo` with a lower-cased token. The PR head's case-insensitive `.find()` branch returns the first match instead of treating the lookup as ambiguous.

## Solution Assessment

No. The PR addresses the right surface, but it is not the narrow maintainable fix until the case-insensitive fallback uses the same uniqueness rule as the normalized fallback and the changelog is added.
