## Summary

Keep open. Current main still has the original exact-only `skills info` lookup, so the PR is not obsolete, but the PR head still has a blocking ambiguity bug in its new case-insensitive fallback: it returns the first lowercase match instead of requiring a unique match, which can show details and config/API-key guidance for the wrong skill.

## What This Changes

This PR changes `openclaw skills info <name>` formatting and tests to add trimmed, case-insensitive and separator-normalized skill-name resolution plus sanitized not-found output.

## Best Possible Solution

Keep the PR open and update the resolver so exact matches still win, then both case-insensitive and separator-normalized fallback paths collect matches and resolve only when exactly one skill matches. Preserve the sanitized not-found output, add an ambiguous case-insensitive regression test, and run the targeted skills CLI test plus the normal changed gate before landing.

## Reproduction Assessment

Yes. A focused unit-style reproduction is a `SkillStatusReport` with two skills whose `name` or `skillKey` differ only by case, then calling `formatSkillInfo` with the lowercase token; the PR head's `.find()` branch returns the first match instead of reporting ambiguity.

## Solution Assessment

No. The PR is close and addresses the right CLI surface, but the case-insensitive fallback should mirror the normalized fallback by filtering all matches and requiring a unique result before it is safe to merge.
