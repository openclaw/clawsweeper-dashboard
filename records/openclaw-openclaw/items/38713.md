## Summary

Keep open: current main still has exact-only `skills info` lookup, and this PR addresses the right CLI bug, but the public PR head still has an ambiguous case-insensitive fallback and is missing the required user-facing changelog entry.

## What This Changes

The PR adds trimmed, case-insensitive, and separator-normalized lookup for `openclaw skills info <name>`, sanitizes not-found output, and updates formatter tests.

## Best Possible Solution

Keep exact matches first, require unique matches for all fallback lookup modes, add the collision regression, and add a single-line changelog fix entry before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with two skill entries whose names or keys differ only by case, `formatSkillInfo(report, "lowercase-token", {})` on the PR head returns the first `.find()` match instead of treating the lookup as ambiguous.

## Solution Assessment

No. The PR targets the correct formatter, but the current branch is not the best mergeable fix until the case-insensitive fallback uses the same uniqueness rule as normalized lookup and the changelog is added.
