## Summary

Keep open: current main still has exact-only `skills info` lookup, while this PR provides the focused formatter fix, regression coverage, changelog entry, and supplied real CLI proof; I found no blocking correctness or security issue.

## What This Changes

The PR makes `openclaw skills info <name>` resolve exact, case-insensitive, and separator-normalized skill names only when fallback matches are unique, sanitizes not-found output, adds formatter tests, and records the fix in the changelog.

## Best Possible Solution

Land the focused formatter, test, and changelog change after maintainer review, preserving exact-match priority and unique-only fallback semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main exact-matches `skillName`, so querying `excel-xlsx` against a report entry keyed as `Excel-XLSX` or `excel_xlsx` misses despite the documented `skills info <name>` command; I did not execute tests because this review is read-only.

## Solution Assessment

Yes. The PR is the narrow maintainable fix: resolve only in the formatter path, keep exact matches first, and return not-found rather than choosing an arbitrary skill when fallback matching is ambiguous.
