## Summary

Keep open: current main still has the exact-only `skills info` lookup, and this PR addresses the right CLI bug, but the latest resolver can still return a case-insensitive match before checking a conflicting normalized match. That can show setup guidance for the wrong skill in a mixed case/separator collision, so one small resolver/test repair should land before merge.

## What This Changes

The PR changes `openclaw skills info` lookup to trim input, resolve exact/case-insensitive/separator-normalized names, reject ambiguous fallback matches, sanitize not-found output, and add tests plus a changelog entry.

## Best Possible Solution

Keep exact matches first, then collect all non-exact case-insensitive and normalized candidates into one unique skill set and return not-found unless that set has exactly one entry.

## Reproduction Assessment

Yes. Current main's `skills info` path is source-reproducible as exact-only lookup, and the PR-head edge case is source-reproducible with two status entries whose key/name collide only after fallback matching.

## Solution Assessment

No. The PR is focused and mostly in the right place, but the fallback resolver should reject ambiguity across all non-exact matching modes before returning a skill.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject cross-mode ambiguous skill matches:** `src/cli/skills-cli.format.ts:129-133`
  - body: This returns a unique case-insensitive match before checking normalized matches. If one skill has `skillKey: "Excel-XLSX"` and another visible skill is named `Excel XLSX`, `openclaw skills info excel-xlsx` resolves to the first skill even though the normalized lookup is ambiguous, and the CLI can print setup/API-key guidance for the wrong skill. Collect non-exact candidates across both fallback modes and require one unique skill before returning.
  - confidence: 0.84

## Security Review
