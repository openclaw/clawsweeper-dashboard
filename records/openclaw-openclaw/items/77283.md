## Summary

Keep open: current main still waits on route resolution, model catalog loading, and recent-model I/O before acknowledging owner Discord picker component interactions, so this remains a valid narrow Discord plugin bug rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to make Discord `/model` picker button/select interactions acknowledge quickly after 2026.5.3 so model selection does not expire under load.

## Best Possible Solution

Have Discord model picker owner interactions defer immediately and then edit the original picker message through the existing deferred-update semantics, with regression coverage for slow model-data and recent-model reads.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: owner button/select interactions perform async route/model/recent-model work before the first Discord response. I did not establish a live failing Discord run in this read-only environment.

## Solution Assessment

Yes for the repair boundary, but the literal suggested `deferUpdate()` plus later `update()` needs adapting to this codebase: use component `acknowledge()` for `DeferredMessageUpdate`, then edit the original message via the deferred reply/edit path rather than sending a second initial update callback.
