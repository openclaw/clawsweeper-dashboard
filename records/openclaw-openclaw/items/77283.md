## Summary

Keep open: current main still leaves the Discord model picker fallback owner path without an early component acknowledgement while it performs async routing, model-data, and preference work before the first update, and I found no exact duplicate or open PR covering this failure.

## What This Changes

The issue reports that selecting a model from the Discord model picker fails after upgrading to OpenClaw 2026.5.3 because the component interaction is not acknowledged before async work.

## Best Possible Solution

Add an early component acknowledgement for model picker owner interactions, route subsequent picker refreshes through edit/original-response-safe semantics, and add regression coverage for delayed model-data and preference reads.

## Reproduction Assessment

Yes, source-reproducible: a registered fallback button/select owner interaction inherits defer=false, then the handler awaits route resolution, model-data loading, and recent-preference reads before its first update. I did not live-run Discord, so this is not marked as reproduced.

## Solution Assessment

Yes, the requested direction is the right narrow fix, but it should be implemented through the internal interaction wrapper so post-ack picker rendering edits the original response instead of calling the initial update callback again.
