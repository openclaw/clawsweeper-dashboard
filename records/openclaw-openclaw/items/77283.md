## Summary

Keep open: current main and v2026.5.3 still leave the Discord model picker component handler without an early acknowledgement before route, model, and preference I/O, so the reported interaction-timeout failure remains source-reproducible.

## What This Changes

The issue requests prompt acknowledgement for Discord model picker button/select interactions so model selection does not time out under load.

## Best Possible Solution

Acknowledge owner model-picker component interactions before route/model/preference reads, then use edit/reply/follow-up semantics after the deferred update with regression coverage for delayed picker data.

## Reproduction Assessment

Yes, source-reproducible: current main registers these model picker fallback components without automatic defer and the handler awaits route, model-data, and recent-preference work before its first component response. I did not live-run Discord, so this is not marked as live reproduced.

## Solution Assessment

Yes in direction, but not as a raw one-line defer followed by the existing `update()` calls. The narrow maintainable fix is to acknowledge early using the Discord wrapper's deferred-update path and route later picker refreshes through edit/reply semantics.
