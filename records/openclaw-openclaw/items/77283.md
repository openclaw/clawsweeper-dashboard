## Summary

Keep open: current main still leaves the Discord model picker component path non-deferred while it performs async route, model, and preference work before the first component response, and I found no merged or open duplicate that already covers this exact failure.

## What This Changes

The issue asks for Discord model picker select/button interactions to acknowledge before async work so picker-based model changes do not hit Discord's component interaction deadline.

## Best Possible Solution

Add an early deferred component acknowledgement for Discord model picker fallback interactions, then route subsequent picker refreshes through edit/follow-up semantics with regression coverage for delayed model-data and preference reads.

## Reproduction Assessment

Yes, source-reproducible: current main registers the model picker button/select with defer=false and the handler performs route resolution, model-data loading, and recent-preference reads before the first update. I did not live-run Discord, so this is not marked as live reproduced.

## Solution Assessment

The requested direction is right, but the exact one-line deferUpdate suggestion needs adaptation to the internal Discord wrapper. The narrow maintainable fix is to acknowledge early and make post-ack picker updates edit the original response or use existing reply/edit semantics.
