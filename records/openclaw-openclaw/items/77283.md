## Summary

Keep open: current main still leaves the Discord model picker component path non-deferred while it does async route, model-data, and preference work before the first component response. The report is a narrow Discord plugin bug and the existing discussion already points to the same source path.

## What This Changes

The issue asks for Discord model picker component selections to be acknowledged before async route/model/preference work so model switches do not hit Discord's interaction timeout.

## Best Possible Solution

Acknowledge the model picker component immediately, then route later picker refresh/notice/apply responses through edit-original or follow-up behavior with regression coverage for slow model-data and preference reads.

## Reproduction Assessment

Yes, source-reproducible: the current fallback button/select path has defer=false and reaches async route, provider-runtime, and preference-store work before its first update. I did not live-run Discord, so this is not marked as reproduced.

## Solution Assessment

No, not as the exact one-line deferUpdate patch. The right direction is early component acknowledgement, but the implementation must also make subsequent picker updates use edit/follow-up semantics that are valid after the deferred acknowledgement.
