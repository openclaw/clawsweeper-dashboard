## Summary

Keep open: the reported bug is source-backed and the fix direction is right, but the proposed helper introduces a catalog-plugin fallback regression that should be corrected before merge.

## What This Changes

This PR extracts target-kind inference into outbound target normalization, forwards the inferred kind into outbound and cron id-like resolution, adds focused tests, and updates the changelog.

## Best Possible Solution

Update the shared inference helper to use the same loaded-or-catalog plugin lookup as the neighboring normalization paths, add a catalog-fallback regression test, then land the narrow resolver fix.

## Reproduction Assessment

Yes. Source inspection on current main shows outbound computes an inferred kind but passes only the explicit preferredKind into id-like resolution, while cron delivery passes no preferredKind at all.

## Solution Assessment

No, not as written. Forwarding inferred preferredKind is the right fix, but the new shared helper should preserve catalog-plugin inference for unloaded channels before this is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the catalog fallback for inferred target kind:** `src/infra/outbound/target-normalization.ts:80-82`
  - body: The new helper reads only `getLoadedChannelPluginForRead`, but the current `detectTargetKind` path and the neighboring normalization helpers can use catalog plugins through the loaded-or-catalog fallback. When a channel plugin is not loaded yet but exposes `inferTargetChatType`, this defaults direct-shaped plugin targets to `group`, so directory and id-like resolution can route them through the wrong kind. Reuse the existing target-read fallback here and add a regression test for an unloaded catalog plugin.
  - confidence: 0.86
