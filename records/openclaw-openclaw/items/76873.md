## Summary

The Discord bug is valid and the PR is close, with the author adding useful targeted and live Discord validation, but the patch should stay open because the extracted helper drops the existing bundled/catalog plugin fallback for chat-type inference.

## What This Changes

The PR extracts target-kind inference into target normalization, passes inferred preferredKind into outbound and cron id-like target resolution, adds resolver/cron regression tests, and updates CHANGELOG.md.

## Best Possible Solution

Keep the PR direction, but have inferTargetResolveKind use the existing loaded-or-catalog plugin lookup and add an unloaded-plugin regression test before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the outbound path computes inferred kind but does not pass it into id-like resolution, and the cron path calls id-like resolution without any preferredKind.

## Solution Assessment

No, not as currently patched. Reusing shared inference is the right shape, but the helper should preserve the existing catalog fallback so unloaded plugins do not lose chat-type inference.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve catalog chat-type inference:** `src/infra/outbound/target-normalization.ts:79-81`
  - body: This helper queries only `getLoadedChannelPluginForRead` for `inferTargetChatType`. The code it replaces used `getChannelPlugin`, and the rest of this module deliberately falls back to bundled/catalog plugins through `resolveChannelPluginForTargetRead`; when a channel is not loaded, static plugin inference will now default to `group` and can pass the wrong `preferredKind` into id-like resolution. Please use the existing loaded-or-catalog resolver and cover that unloaded-plugin case.
  - confidence: 0.86
