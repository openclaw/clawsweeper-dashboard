## Summary

Keep open: current main still builds Fal image edits with the Flux-style `/image-to-image` route and singular `image_url`, while Fal GPT Image 2 edit requires `/edit` with `image_urls`. The reporter's hotfix verification and Fal's published schema both support a narrow provider-local fix.

## What This Changes

The issue requests fixing Fal GPT Image 2 reference-image edits so `image_generate` uses Fal's `/edit` endpoint and request schema while preserving Flux edit behavior.

## Best Possible Solution

Patch the Fal plugin with model-family-specific edit route/body handling, add focused regression tests for GPT Image 2 and Flux, and update docs/changelog if capability copy changes.

## Reproduction Assessment

Yes, from source inspection: `fal/openai/gpt-image-2` splits to provider `fal` and model `openai/gpt-image-2`, then the Fal provider posts edits to `/openai/gpt-image-2/image-to-image` with `image_url`. I did not run a live Fal request.

## Solution Assessment

Yes: a provider-local, model-family-specific fix is the narrowest maintainable solution because Fal's GPT Image 2 and Flux edit endpoints intentionally use different paths and request fields.
