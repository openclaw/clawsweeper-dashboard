## Summary

Keep open: current main still contains the Fal provider edit-routing bug, and the linked open PR uses closing syntax for this issue, so cleanup should wait for that PR to be reviewed, merged, or closed.

## What This Changes

The issue reports that Fal GPT Image 2 reference-image generation is sent to `/image-to-image` with a singular `image_url` instead of the model's `/edit` route and plural `image_urls` edit payload.

## Best Possible Solution

Review and either land or close the linked Fal-provider PR; the desired end state is a provider-local fix with model-family-specific routing, payload fields, capability metadata, docs, and regression coverage.

## Reproduction Assessment

Yes from source inspection: `fal/openai/gpt-image-2` reaches the Fal provider as `openai/gpt-image-2`, and current main would build an edit request for `openai/gpt-image-2/image-to-image` with `image_url`. I did not run a live Fal request.

## Solution Assessment

Yes: a provider-local, model-family-specific Fal fix is the narrowest maintainable solution because the wrong route and payload are isolated to `extensions/fal` and do not require a core API change.
