## Summary

Keep open: current main still routes Fal GPT Image 2 edit requests through the Flux-style `/image-to-image` path with a singular `image_url` body, while Fal documents GPT Image 2 edits at `/edit` with `image_urls`. The linked #77388 PR uses closing syntax for this issue and is still open, so the issue should remain paired until that PR or an equivalent fix lands.

## What This Changes

Fix Fal image edit routing and request bodies so `fal/openai/gpt-image-2` reference-image requests use Fal's `/edit` contract while preserving Flux image-to-image behavior.

## Best Possible Solution

Land or revise #77388, or an equivalent provider-local patch, with regression coverage and docs/changelog so GPT Image 2 and Nano Banana edit models use Fal's `/edit` contract while Flux remains unchanged.

## Reproduction Assessment

Yes from source inspection: `fal/openai/gpt-image-2` parses to provider `fal` and model `openai/gpt-image-2`, then current main would POST edits to `/openai/gpt-image-2/image-to-image` with `image_url`. I did not run a live Fal generation request.

## Solution Assessment

Yes: a Fal provider-local model-family fix is the narrowest maintainable solution because the mismatch is in Fal endpoint/schema selection, not the core image-generation API. The issue should stay open while the linked closing PR is reviewed or replaced.
