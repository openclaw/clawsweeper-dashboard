## Summary

Keep open. Current main still has a credible stale-config path: managed web_search execution can keep a caller-supplied config when it differs from the active source snapshot, and Gemini then reads plugin-scoped webSearch credentials from that config during provider execution. The existing runtime snapshot code resolves plugin-scoped credentials correctly, but only execution paths that actually select the active resolved snapshot benefit from it.

## What This Changes

The issue asks OpenClaw to ensure Gemini managed web_search execution uses the resolved active secrets runtime snapshot instead of allowing stale raw SecretRefs from plugin-scoped config to reach provider runtime code.

## Best Possible Solution

Managed web_search should late-bind the active resolved runtime config for gateway/session-owned tool calls when runtime metadata comes from the active secrets snapshot, while preserving intentionally supplied custom configs for SDK/runtime callers. Add a regression test that models an existing Gemini web_search wrapper retaining a stale raw plugin SecretRef after a resolved runtime snapshot is active.

## Reproduction Assessment

Yes. The issue gives concrete config and error output, and current source gives a static reproduction path: pass a stale raw config that does not match the active source snapshot into managed web_search after a resolved runtime snapshot is active, then Gemini reads the raw plugin-scoped apiKey.

## Solution Assessment

Unclear. The requested direction is correct, but the best implementation needs owner judgment at the managed tool/runtime boundary so resolved snapshots win for session tools without silently discarding deliberate custom configs.
