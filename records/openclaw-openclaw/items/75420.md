## Summary

Keep open. Current main still has the stale managed web_search wrapper path described in the report: resolved runtime snapshots work for matching source-config callers, but a long-lived wrapper can keep passing a caller-supplied raw config into Gemini provider creation, where the plugin-scoped SecretRef is read strictly and can still throw.

## What This Changes

The issue asks OpenClaw to ensure Gemini managed web_search execution uses the resolved active secrets runtime snapshot instead of passing unresolved plugin-scoped SecretRefs to provider runtime code after upgrade/reload.

## Best Possible Solution

Managed session-owned web_search execution should late-bind the active resolved runtime config and current runtime web-tool metadata at execute time, while explicit SDK/plugin custom config paths keep their existing custom-config semantics. Add regression coverage for a stale existing Gemini wrapper holding a raw plugin SecretRef after an active resolved snapshot is installed or refreshed.

## Reproduction Assessment

Yes. The issue gives concrete config and error output, and current source provides a high-confidence static reproduction path: stale managed web_search wrapper config is preserved, passed into Gemini createTool, merged from plugin config, then rejected by the strict SecretRef reader.

## Solution Assessment

Unclear. The requested resolved-snapshot direction is correct, but the safest implementation should be scoped to the managed web_search wrapper/runtime boundary rather than globally changing custom config selection semantics.
