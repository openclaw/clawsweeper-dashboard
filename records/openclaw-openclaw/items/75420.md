## Summary

Keep open. Current main still has the source-config leak shape described in the report: managed web_search tools capture config/runtime metadata when the tool is created, runWebSearch preserves caller-supplied configs that do not match the active source snapshot, and the Gemini provider later reads plugin-scoped webSearch credentials strictly from ctx.config.

## What This Changes

The issue requests that managed Gemini web_search use the resolved active runtime snapshot instead of a stale or raw config when resolving plugin-scoped SecretRefs.

## Best Possible Solution

Managed gateway/session-owned web_search execution should late-bind the active resolved runtime config and current runtime web-tool metadata, while preserving explicit custom config behavior for SDK and plugin runtime callers. Add a regression test for an existing managed wrapper holding a raw Gemini plugin SecretRef after a resolved runtime snapshot is active or refreshed.

## Reproduction Assessment

Yes. The issue gives concrete config and error output, and current source gives a static reproduction path: execute managed web_search from a wrapper holding a stale raw config that does not match the active source snapshot, then Gemini reads the raw plugin-scoped apiKey and throws from the strict SecretRef normalizer.

## Solution Assessment

Unclear. The requested resolved-snapshot direction is correct, but the safest implementation should distinguish managed agent wrappers from explicit SDK/custom config callers instead of globally replacing every nonmatching caller config with the active runtime snapshot.
