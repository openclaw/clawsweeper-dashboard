## Summary

Keep open. The report still matches current main: resolved runtime snapshots exist and are used for matching source-config callers, but managed web_search execution can still preserve a stale caller-supplied raw config and pass it into Gemini provider execution, where plugin-scoped SecretRefs are read strictly.

## What This Changes

The issue asks OpenClaw to make managed Gemini web_search execution use the resolved active secrets runtime snapshot instead of stale raw plugin SecretRef config in existing sessions.

## Best Possible Solution

Managed web_search execution should late-bind the active resolved runtime config for gateway/session-owned tool calls, while preserving explicit custom config behavior for SDK and plugin runtime callers. Add a regression test for an existing wrapper holding a stale raw Gemini plugin SecretRef after an active resolved snapshot is installed or refreshed.

## Reproduction Assessment

Yes. The issue gives concrete config and error output, and current source gives a high-confidence static reproduction path: call managed web_search with a stale raw config that no longer matches the active source snapshot after a resolved runtime snapshot is active, then Gemini reads the raw plugin-scoped apiKey.

## Solution Assessment

Unclear. The requested resolved-snapshot direction is right, but applying it globally in selectApplicableRuntimeConfig conflicts with existing custom-config behavior; the safer fix likely belongs at the managed web_search wrapper/runtime boundary.
