## Summary

Keep open. Current main still preserves a path where managed web_search tools can reuse captured caller config/runtime metadata, and Gemini still reads plugin-scoped credentials from that config through the strict SecretRef reader. The report is concrete and not superseded by a merged fix.

## What This Changes

Gemini managed web_search should use the resolved active runtime snapshot so plugin-scoped SecretRef API keys do not reach provider runtime after upgrade or secrets reload.

## Best Possible Solution

Managed gateway/session-owned web_search execution should late-bind the active resolved runtime config and current runtime web-tool metadata, while explicit SDK/plugin custom config continues to use the caller-supplied config. Add a regression test for an already-created managed wrapper holding a raw Gemini plugin SecretRef after the active runtime snapshot changes.

## Reproduction Assessment

Yes. The issue gives concrete config and error output, and current main gives a static reproduction path: execute managed web_search from a wrapper holding stale raw config that no longer matches the active source snapshot, then Gemini reads the raw plugin-scoped apiKey and the strict SecretRef normalizer throws.

## Solution Assessment

Unclear. The resolved-snapshot direction is correct, but the maintainable fix needs to distinguish managed agent wrappers from explicit SDK/custom config callers instead of globally overriding every nonmatching caller config.
