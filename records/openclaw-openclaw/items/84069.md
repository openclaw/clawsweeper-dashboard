## Summary

Keep open: current main still renders an inherited `Off` choice plus an explicit `off` override for off-only thinking profiles, matching the reporter's screenshots/video and making this a narrow Control UI bug.

## What This Changes

The issue asks OpenClaw Control to avoid duplicate `off` entries in the chat thinking selector and make off-only thinking selection unambiguous.

## Best Possible Solution

Normalize the Control UI thinking selector so off-only/default-off models show one clear disabled or reset-safe `Off` state, while keeping explicit `off` available when it changes behavior from the inherited default.

## Reproduction Assessment

Yes. The current source path is enough to reproduce logically: off-only thinking metadata reaches `renderChatThinkingSelect`, which renders both `value=""` default `Off` and `value="off"` override `Off`; the reporter also supplied visual proof from v2026.5.18.

## Solution Assessment

Yes. The best fix is a small UI option-state change in the Control chat selector, with regression coverage for off-only and multi-level thinking profiles, rather than changing gateway thinking policy.

In the Control UI chat thinking selector, avoid showing duplicate visible `Off` entries when the inherited/default thinking level is off and the option list also contains `off`, especially for off-only/non-reasoning model profiles such as Ollama mistral. Consider whether the sessions table thinking selector shares the same helper/bug and keep any fix bounded to UI option rendering/state.
