## Summary

Keep open: the stale-`SUDO_USER` bug is real, but this PR still changes the shared systemctl user-scope resolver in a way that can regress fresh sudo-to-root systemd installs, and the supplied proof is unit-test/build output rather than live Linux systemd proof.

## What This Changes

The PR makes systemd user-scope resolution async, checks a non-root `SUDO_USER` unit file before preferring machine scope, and updates daemon systemd tests for stale sudo-user cases.

## Best Possible Solution

Keep the stale-`SUDO_USER` fix, but make the scope decision install-safe: choose root `--user` only when the root/current unit is known to own the lifecycle, preserve existing sudo-to-root machine-scope install behavior, and prove both paths on Linux.

## Reproduction Assessment

Yes, source-reproducible. Current main routes effective-root plus non-root `SUDO_USER` through the sudo caller's machine-scoped user manager, and the PR regression is source-reproducible because `writeSystemdUnit` probes systemd before writing a fresh unit.

## Solution Assessment

No, not yet. The current patch is a plausible stale-env heuristic, but placing it in the shared resolver is too broad; the safer fix needs installed-unit/current-owner context or an install-safe branch.
