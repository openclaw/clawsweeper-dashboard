## Summary

Keep this PR open, but it is not ready to merge: the branch implements proxy retry suppression by falling back to direct network access, which conflicts with the current fail-closed managed proxy contract and still has correctness issues in the Discord and circuit-breaker paths.

## What This Changes

The branch adds a shared proxy circuit breaker with direct fallback, wires it into proxy fetch and old Discord proxy paths, lowers Discord gateway reconnect attempts, broadens token rate-limit matching, and adds unit/integration tests.

## Best Possible Solution

Design a fail-closed proxy retry/circuit breaker that reduces dead-proxy churn without direct egress fallback, port it to current shared proxy and Discord extension paths, and validate it with regression tests plus redacted real dead-proxy runtime proof.

## Reproduction Assessment

No. Source inspection shows proxy fetches still go through configured dispatchers without circuit state, so repeated dead-proxy attempts are plausible, but there is no high-confidence live multi-channel CPU reproduction or after-fix proof.

## Solution Assessment

No. A circuit breaker may be the right direction, but this implementation is fail-open, misaccounts some gateway failures, allows concurrent half-open bursts, and needs a current-main Discord extension port.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured proxy routing on failures:** `src/infra/net/proxy-fetch.ts:18-28`
  - body: When the circuit is open or a proxy connection error occurs, this wrapper calls global `fetch` directly. That bypasses operator-managed proxy egress even though the current proxy contract is fail-closed for supported clients, so retry suppression needs to keep traffic on the configured proxy path unless maintainers approve a policy change.
  - confidence: 0.9
- **[P1] Guard proxy failure accounting by proxy use:** `src/discord/monitor/gateway-plugin.ts:79-80`
  - body: In `registerClient`, a circuit-open request runs without the proxy dispatcher, but the catch block still records a proxy failure for any connect-style error. A direct Discord or network failure can therefore extend the proxy cooldown even though the proxy was not attempted.
  - confidence: 0.93
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:72-73`
  - body: After cooldown expiry, `isProxyCircuitOpen` switches to `half_open` and returns false without reserving the probe window. Concurrent callers can all pass through, hit the proxy together, and inflate `consecutiveFailures`, recreating the burst the circuit breaker is meant to avoid.
  - confidence: 0.88
- **[P2] Keep request-token overflow out of rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:17`
  - body: Adding the broad string `too many tokens per` also matches `too many tokens per request`. Current tests keep that message as context overflow with no failover reason, so this change would misclassify per-request context errors as rate limits.
  - confidence: 0.86
