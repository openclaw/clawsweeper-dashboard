## Summary

Keep open: current main still lacks the shared proxy circuit breaker, but this PR is not mergeable as-is because it silently fails open around proxy routing and is stale against current proxy/Discord code paths.

## What This Changes

The PR adds proxy circuit-breaker state with direct fetch/WebSocket fallback, wires it into shared proxy fetch and Discord REST/gateway paths, lowers Discord gateway reconnect attempts, and broadens several rate-limit matchers.

## Best Possible Solution

Implement shared proxy backoff that throttles failing proxy attempts while preserving fail-closed managed proxy routing unless maintainers explicitly approve an opt-in direct-fallback policy.

## Reproduction Assessment

Yes, from source inspection: configure an unavailable proxy and exercise makeProxyFetch plus Discord/Telegram proxy call sites. Current main still dispatches through ProxyAgent/HttpsProxyAgent without shared cooldown state, so the retry-pressure problem is source-reproducible even though I did not run a live proxy test.

## Solution Assessment

No. A circuit breaker/backoff is a reasonable direction, but this patch fails open without an explicit policy, misattributes some direct-connection errors, allows concurrent half-open probes, and needs a current-main rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve fail-closed proxy routing before direct fallback:** `src/infra/net/proxy-fetch.ts:17-28`
  - body: The wrapper routes through global direct fetch whenever the circuit is open or a proxy connect error is caught. For managed or explicit proxy deployments, current docs treat proxy routing as an egress guardrail, so this silently changes protected traffic from fail-closed to fail-open unless an explicit policy opts in.
  - confidence: 0.9
- **[P2] Only record gateway failures for proxied requests:** `src/discord/monitor/gateway-plugin.ts:77-80`
  - body: When useProxy is false, the metadata request already went direct, but this catch block still records connection-style errors against the proxy circuit. A transient direct Discord or network failure can extend the proxy cooldown even though the proxy was not used.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:70-73`
  - body: After cooldown expiry, isProxyCircuitOpen switches to half_open but leaves the expired timestamp in place, so concurrent callers can all pass through and probe the proxy. That recreates a burst of failed connects and inflates consecutiveFailures faster than intended.
  - confidence: 0.86
- **[P2] Rebase env proxy wrapping onto current resolver semantics:** `src/infra/net/proxy-fetch.ts:69-76`
  - body: This branch constructs EnvHttpProxyAgent directly and wraps an older resolveProxyFetchFromEnv shape. Current main accepts an injected env and uses resolveEnvHttpProxyAgentOptions for lowercase precedence and ALL_PROXY fallback, so the circuit breaker should wrap that existing resolved-agent path instead of replacing it.
  - confidence: 0.82
