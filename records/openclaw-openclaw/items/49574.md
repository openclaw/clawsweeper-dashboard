## Summary

Keep open: current main has not implemented the proxy circuit breaker, but this PR is not merge-ready because it conflicts with current main, lacks real unavailable-proxy proof, and introduces fail-open proxy bypass plus circuit bugs.

## What This Changes

The PR adds a proxy circuit breaker around shared proxy fetch and Discord proxy paths, lowers Discord gateway reconnect attempts, adds tests, and adjusts several rate-limit classifiers.

## Best Possible Solution

Rebase a fail-closed or explicitly policy-gated circuit breaker onto current src/infra/net/proxy-fetch.ts and extensions/discord, bound half-open probes, add focused regression tests, and include redacted unavailable-proxy proof.

## Reproduction Assessment

No. Source inspection shows current main has no proxy circuit state, so a dead configured proxy would still be retried by callers, but I did not run a real unavailable-proxy CPU exhaustion scenario.

## Solution Assessment

No. A circuit breaker is a reasonable direction, but this implementation bypasses configured proxies, misattributes direct-connection failures, allows concurrent half-open probes, and needs to be rebased onto the current Discord plugin paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not bypass configured proxies after failures:** `src/infra/net/proxy-fetch.ts:18-26`
  - body: When the circuit is open, and again after a connection-style proxy error, this wrapper sends the request through global `fetch` with no proxy dispatcher. That conflicts with the documented fail-closed operator proxy model and can leak runtime traffic outside the configured egress/audit path; prefer throttled fail-closed behavior or an explicit policy-gated fallback.
  - confidence: 0.83
- **[P2] Gate proxy failure recording on actual proxy use:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: When `useProxy` is false, the gateway metadata request is direct, but the catch block still records connection errors against the proxy circuit. A transient direct-path failure can extend the proxy blackout and prevent recovery from being probed correctly.
  - confidence: 0.9
- **[P2] Reserve a single half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-68`
  - body: After cooldown expiry, `lastFailAt` remains expired while the state is `half_open`, so concurrent callers can all pass through and probe the proxy at once. That recreates a burst of failed proxy connects and inflates the exponential backoff counter.
  - confidence: 0.86
