## Summary

Keep open; this is not a cleanup close. The unavailable-proxy churn is still plausible on current main, but this branch should not merge as-is because it lacks real behavior proof and the proposed circuit breaker has correctness and proxy-policy problems.

## What This Changes

The branch adds a shared proxy-probe circuit breaker around core proxy fetch and Discord proxy paths, reduces Discord gateway reconnect attempts, adds tests, and touches rate-limit classifier strings.

## Best Possible Solution

Rebase a proxy circuit breaker onto current src/infra/net/proxy-fetch.ts and extensions/discord, keep proxy fallback fail-closed or explicitly policy-gated, add focused regression tests, and include real unavailable-proxy proof.

## Reproduction Assessment

Source-reproducible, not live-reproduced. Current main has no circuit state in the proxy fetch helper, so a dead configured proxy would still be retried by each caller, but I did not run a real CPU exhaustion scenario.

## Solution Assessment

No. A circuit breaker is a reasonable direction, but this implementation needs rebase work, corrected failure attribution, bounded half-open probing, and a maintainer decision on direct proxy fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not bypass configured proxies on circuit-open fetches:** `src/infra/net/proxy-fetch.ts:18`
  - body: When the circuit is open, the wrapper silently retries the same request through global fetch. For an operator-configured proxy, that can send traffic outside the required proxy path; prefer fail-fast/throttled behavior or an explicit policy-gated fallback.
  - confidence: 0.78
- **[P2] Gate proxy failure recording on actual proxy use:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: When `useProxy` is false, the gateway metadata request is direct, but the catch block still records connection errors against the proxy circuit. A transient direct-path failure can extend the proxy blackout even after the proxy has recovered.
  - confidence: 0.9
- **[P2] Reserve a single half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-68`
  - body: After cooldown expiry, `lastFailAt` remains expired while the state is `half_open`, so concurrent callers can all pass through and probe the proxy at once. That recreates the failed-connect burst and inflates the backoff counter.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive fail-open path that can bypass an explicitly configured proxy after a connection error.

Concerns:

- **[medium] Configured proxy can be bypassed after failure:** `src/infra/net/proxy-fetch.ts:18`
  - body: The new circuit-open path falls back to global fetch for requests that were configured to use a proxy. In deployments where the proxy is an egress or audit control, this can leak traffic outside the intended route.
  - confidence: 0.78
