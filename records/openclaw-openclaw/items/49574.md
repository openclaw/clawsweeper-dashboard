## Summary

Keep open, but the PR is not mergeable as-is: the backoff idea may be useful, but this branch silently falls back to direct network access against the documented fail-closed proxy boundary, targets stale Discord/source paths, and still lacks after-fix real behavior proof.

## What This Changes

The branch adds proxy circuit-breaker state with direct-fetch fallback to shared proxy fetch and old Discord monitor paths, lowers Discord gateway reconnect attempts, and expands several rate-limit matchers.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer: keep managed and explicit proxy modes fail-closed by default, add only a documented opt-in fallback/circuit breaker if maintainers approve it, port any Discord changes to current plugin files, and prove the result with a real closed-proxy run.

## Reproduction Assessment

No high-confidence end-to-end reproduction of CPU exhaustion or retry-rate reduction is attached. Source inspection does show current main selects proxy dispatchers and rethrows request-time proxy failures, so the unavailable-proxy path is source-reproducible but not behavior-proven.

## Solution Assessment

No. A shared backoff mechanism may be useful, but this implementation fails open around configured proxy routing, needs the gateway attribution and half-open fixes, and must be rebased onto the current plugin layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep configured proxy failures fail-closed:** `src/infra/net/proxy-fetch.ts:17-28`
  - body: When the circuit opens or a proxy connect fails, this wrapper retries with global fetch. For managed proxy deployments, that sends traffic outside the operator egress boundary instead of failing closed, so direct egress needs an explicit policy before this path can merge.
  - confidence: 0.86
- **[P1] Record gateway failures only for proxied attempts:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: The circuit-open path sets useProxy false and performs a direct fetch, but the catch block still records any connect-style error as a proxy failure. A direct Discord reachability failure can therefore extend the proxy cooldown and prevent recovery probes.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-67`
  - body: After cooldown expiry, every caller sees the same expired lastFailAt and probes the proxy. Under concurrent traffic this recreates the burst the circuit is meant to avoid and inflates the backoff counter.
  - confidence: 0.84
