## Summary

Keep open. The PR addresses a plausible proxy retry problem, but the proposed direct fallback conflicts with the current fail-closed proxy contract, lacks real behavior proof, and still has concrete correctness issues in the circuit-breaker implementation.

## What This Changes

Adds proxy circuit-breaker state that falls back to direct fetch after proxy connection errors, wires it into shared proxy fetch and old Discord monitor paths, lowers Discord gateway reconnect attempts, and tweaks rate-limit matching.

## Best Possible Solution

Keep proxy routing fail-closed by default; if direct fallback is approved, add an explicit documented opt-in in the shared proxy layer and port it to active Discord plugin paths with real proof.

## Reproduction Assessment

No high-confidence end-to-end reproduction is attached. Source inspection shows current main selects proxy dispatchers and propagates request-time proxy failures, but CPU exhaustion and retry-rate reduction are not behavior-proven here.

## Solution Assessment

No. A shared circuit breaker may be useful, but this patch fails open around configured proxy routing, needs gateway attribution and half-open concurrency fixes, and must be rebased onto the active extensions/discord layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep configured proxy failures fail closed:** `src/infra/net/proxy-fetch.ts:17-28`
  - body: When the circuit opens or a proxy connect fails, this wrapper retries with global fetch. For managed proxy deployments, that sends traffic outside the operator egress boundary instead of failing closed, so direct egress needs explicit product/security policy before this path can merge.
  - confidence: 0.86
- **[P1] Record gateway failures only for proxied attempts:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: The circuit-open path sets useProxy false and performs a direct fetch, but the catch block still records any connect-style error as a proxy failure. A direct Discord reachability failure can therefore extend the proxy cooldown and prevent recovery probes.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-76`
  - body: After cooldown expiry, every caller sees the same expired lastFailAt and probes the proxy. Under concurrent traffic this recreates the burst the circuit is meant to avoid and inflates the backoff counter.
  - confidence: 0.84
