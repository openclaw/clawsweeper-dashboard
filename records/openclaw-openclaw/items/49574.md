## Summary

Keep this PR open, but it should not merge as-is. The CPU-exhaustion problem is still plausible on current main because there is no shared proxy circuit state, but the branch fails open to direct egress, targets stale pre-extension Discord paths, and would overwrite newer env-proxy resolution semantics.

## What This Changes

The PR adds proxy circuit-breaker state with direct-fetch fallback for proxy connection errors, wires it into shared proxy fetch and Discord REST/gateway paths, lowers Discord gateway reconnect attempts, and broadens several rate-limit matchers for “too many tokens per” errors.

## Best Possible Solution

Land a rebased, policy-aware implementation that adds shared proxy backoff without silently bypassing managed or explicit proxy routing, wraps the existing env-proxy resolver instead of replacing it, updates current extension-owned Discord and Telegram paths, permits only one half-open probe, and includes focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to configure an unavailable proxy and exercise makeProxyFetch plus Telegram/Discord proxy call sites; current main shows those paths still dispatch through ProxyAgent or EnvHttpProxyAgent without shared cooldown state.

## Solution Assessment

No. A circuit breaker is a reasonable direction, but this implementation is not the best fix because it fails open without an explicit policy, misattributes some direct-connection errors to the proxy circuit, allows concurrent half-open probes, and is stale against current main’s proxy and Discord plugin structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve fail-closed proxy routing before direct fallback:** `src/infra/net/proxy-fetch.ts:17-28`
  - body: The wrapper routes through global fetch whenever the circuit is open or a proxy connect error is caught. For managed or explicit proxy deployments, current docs treat proxy routing as an egress guardrail, so this can silently change protected traffic from fail-closed to fail-open unless an explicit policy opts into that behavior.
  - confidence: 0.9
- **[P2] Only record gateway failures for proxied requests:** `src/discord/monitor/gateway-plugin.ts:77-80`
  - body: When useProxy is false, the metadata request already went direct, but this catch block still records connection-style errors against the proxy circuit. A transient direct Discord or network failure can extend the proxy cooldown even though the proxy was not used.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:70-73`
  - body: After cooldown expiry, isProxyCircuitOpen switches to half_open but leaves the expired timestamp in place, so concurrent callers can all pass through and probe the proxy. That recreates a burst of failed connects and inflates consecutiveFailures faster than intended.
  - confidence: 0.86
- **[P2] Rebase env proxy wrapping onto current resolver semantics:** `src/infra/net/proxy-fetch.ts:69-76`
  - body: This branch constructs EnvHttpProxyAgent directly and wraps an older resolveProxyFetchFromEnv shape. Current main accepts an injected env and uses resolveEnvHttpProxyAgentOptions for lowercase precedence and ALL_PROXY fallback, so the circuit breaker should wrap that existing resolved-agent path instead of replacing it.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete proxy trust-boundary concern by falling back to direct HTTP/WebSocket egress after proxy failures.

Concerns:

- **[high] Direct fallback can bypass proxy egress policy:** `src/infra/net/proxy-fetch.ts:17`
  - body: The proposed circuit breaker uses global direct fetch or an unproxied WebSocket when proxy use is suppressed. Without an explicit fail-open policy, this can bypass the operator or channel proxy boundary that current docs describe as a fail-closed egress guardrail.
  - confidence: 0.9
