## Summary

Keep open: the dead-proxy retry concern is still plausible on current main, but this branch needs maintainer direction because it bypasses documented proxy routing, has blocking logic regressions, lacks real behavior proof, and targets Discord code that has since moved.

## What This Changes

The branch adds a shared proxy circuit breaker/fallback around proxy fetch and Discord REST/gateway traffic, lowers Discord gateway reconnect attempts, and broadens some rate-limit error matching.

## Best Possible Solution

Design a fail-closed proxy retry/circuit breaker that reduces repeated dead-proxy attempts without bypassing operator-managed proxy routing, port it to current core and Discord plugin paths, and back it with regression tests plus real dead-proxy runtime proof.

## Reproduction Assessment

No. The dead-proxy CPU exhaustion claim is source-plausible because current `makeProxyFetch` sends each call through the proxy dispatcher, but there is no high-confidence live multi-channel dead-proxy reproduction or after-fix proof attached.

## Solution Assessment

No. A circuit breaker may be the right direction, but this implementation falls back to direct network access despite the fail-closed proxy contract and needs state-machine, Discord-port, and error-classification fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured proxy routing on failures:** `src/infra/net/proxy-fetch.ts:17-18`
  - body: The wrapper returns global `fetch` when the circuit is open or a proxy connect error occurs, so an operator-configured proxy is silently bypassed after the first failure. Current network-proxy docs define this routing as fail-closed egress control, so the retry suppression needs to happen without sending protected traffic direct unless maintainers explicitly add that policy.
  - confidence: 0.9
- **[P1] Guard proxy failure accounting by the proxy attempt:** `src/discord/monitor/gateway-plugin.ts:78-79`
  - body: In `registerClient`, the circuit-open path sets `useProxy` false and performs a direct fetch, but the catch block still records a proxy failure for any connect-style direct-fetch error. That can extend the proxy cooldown because Discord or direct egress failed, not because the proxy was probed.
  - confidence: 0.93
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:70-73`
  - body: After cooldown expiry, `isProxyCircuitOpen` switches to `half_open` and returns false without reserving the probe window. Concurrent callers can all observe the expired cooldown, all hit the dead proxy, and inflate `consecutiveFailures`, recreating the burst the circuit breaker is meant to avoid.
  - confidence: 0.88
- **[P2] Keep per-request token errors out of rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:17`
  - body: Adding the broad string `too many tokens per` also matches `too many tokens per request`, which current main tests keep in the context-overflow lane with no failover. This would regress model error classification outside the proxy fix.
  - confidence: 0.86
