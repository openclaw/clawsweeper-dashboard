## Summary

Keep open. The useful backoff idea is not on current main, but this branch is not mergeable as-is because it silently bypasses configured proxies, has source-readable circuit bugs, targets removed paths, and lacks required real behavior proof.

## What This Changes

The PR adds a proxy circuit breaker with direct-fetch fallback, wires it into shared proxy fetch and old Discord monitor paths, lowers Discord gateway reconnect attempts, expands rate-limit matchers, and adds proxy tests.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer, keep managed proxy routing fail-closed by default, make any direct fallback explicit and documented, port Discord changes to current plugin files, and prove the result with focused tests plus a redacted real-runtime run.

## Reproduction Assessment

No high-confidence end-to-end reproduction is attached. Source inspection shows current main lacks a circuit/backoff path, but the PR does not include a real unavailable-proxy CPU or retry-rate measurement.

## Solution Assessment

No. A backoff/circuit mechanism is a plausible direction, but this implementation fails open around configured proxy routing, needs the gateway attribution and half-open fixes, and must be rebased onto the current plugin layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep configured proxy failures fail-closed:** `src/infra/net/proxy-fetch.ts:17-28`
  - body: When the circuit opens or a proxy connect fails, this wrapper retries with global fetch. For managed proxy deployments, that sends traffic outside the operator egress boundary instead of failing closed, so the circuit should back off without direct egress unless an explicit policy opts in.
  - confidence: 0.86
- **[P1] Only record gateway failures for proxied attempts:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: The direct gateway path sets useProxy false, but the catch block still records any connection-style error as a proxy failure. If direct Discord reachability fails while the circuit is open, this extends the proxy cooldown and can prevent recovery probes.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-67`
  - body: After cooldown expiry, every caller sees the same expired lastFailAt and probes the proxy. Under concurrent traffic this recreates the burst the circuit is meant to avoid and inflates the backoff counter.
  - confidence: 0.84
