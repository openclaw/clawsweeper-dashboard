## Summary

Keep this PR open. The retry-storm problem is plausible and current main still lacks a shared proxy circuit breaker, but this branch is not mergeable as-is because it fails open to direct network egress across the documented managed-proxy boundary and is stale against current proxy/env and Discord plugin paths.

## What This Changes

The branch adds a proxy circuit breaker with exponential backoff and direct-fetch fallback, wires it into shared proxy fetch and Discord REST/gateway paths, lowers Discord gateway reconnect attempts, and broadens several token-rate-limit matchers.

## Best Possible Solution

Land a rebased, policy-aware implementation that reduces dead-proxy retry storms while preserving fail-closed managed proxy routing, wraps the current env proxy resolver instead of replacing it, allows only one half-open probe, records failures only for requests that actually used the proxy, and updates the current plugin paths with focused regression tests and a changelog entry if user-facing behavior changes.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure an unavailable HTTP proxy and exercise makeProxyFetch plus Discord REST/gateway proxy call sites; current-main source shows those paths use ProxyAgent or EnvHttpProxyAgent without shared cooldown state, while the PR tests model fallback and cooldown behavior.

## Solution Assessment

No. A circuit breaker is a reasonable direction, but this implementation is not the best fix because it silently fails open across a documented proxy trust boundary and does not preserve current-main proxy/env and Discord plugin contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve fail-closed proxy mode before direct fallback:** `src/infra/net/proxy-fetch.ts:17-28`
  - body: The wrapper routes requests through global fetch whenever the circuit is open or a proxy connect error occurs. For managed proxy deployments, the documented proxy is the runtime egress guardrail, so this lets protected traffic leave directly when the proxy is unavailable. Keep managed proxy paths fail-closed unless a narrow explicit fail-open policy is active.
  - confidence: 0.9
- **[P2] Only record gateway failures for proxied requests:** `src/discord/monitor/gateway-plugin.ts:77-80`
  - body: When useProxy is false, the metadata request already went direct, but the catch block still records connect-style errors against the proxy circuit. A transient direct Discord or network failure can extend the proxy cooldown even though the proxy was not used.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:70-73`
  - body: After cooldown expiry, isProxyCircuitOpen transitions to half_open but leaves the expired timestamp in place, so concurrent callers can all pass through and probe the proxy. That can recreate the burst of failed connects and inflate consecutiveFailures faster than intended.
  - confidence: 0.86
- **[P2] Wrap the existing env proxy resolver:** `src/infra/net/proxy-fetch.ts:60-76`
  - body: This branch rewrites resolveProxyFetchFromEnv instead of wrapping the current resolveEnvHttpProxyAgentOptions path. A rebase would drop current injectable-env behavior and lowercase/ALL_PROXY precedence coverage unless the circuit composes with the existing resolver.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete proxy trust-boundary concern by falling back to direct HTTP/WebSocket egress after proxy failures.

Concerns:

- **[high] Direct fallback can bypass managed proxy egress control:** `src/infra/net/proxy-fetch.ts:17`
  - body: The proposed circuit breaker uses global direct fetch while the proxy circuit is open or after proxy connect errors. For proxy.enabled deployments, current docs describe the operator proxy as the runtime egress guardrail, so protected traffic could leave the process directly when the proxy is unavailable.
