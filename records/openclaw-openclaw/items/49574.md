## Summary

Keep open. Current main still lacks a proxy circuit breaker for the shared proxy fetch/Discord/Telegram paths, so the PR is not obsolete; however, the proposed patch is not ready because it fails open around a documented managed-proxy egress boundary, is stale against the current Discord plugin layout, and has concrete circuit-state bugs already identified in review comments.

## What This Changes

The PR adds proxy circuit state with direct-fetch fallback for proxy connection errors, wires it into shared proxy fetch and Discord REST/gateway paths, lowers Discord gateway reconnect attempts, and broadens several rate-limit matchers for “too many tokens per” errors.

## Best Possible Solution

Land a rebased, policy-aware implementation that adds shared proxy backoff without bypassing managed proxy egress controls, preserves current env proxy resolution semantics, limits half-open probing to one sentinel request, records failures only for requests that actually used a proxy, and updates the current Discord/Telegram plugin paths with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure an unavailable proxy and exercise makeProxyFetch plus Discord/Telegram proxy call sites; current main source shows those paths use ProxyAgent or EnvHttpProxyAgent without shared cooldown state, while the PR tests model fallback and cooldown behavior.

## Solution Assessment

No. A circuit breaker is a reasonable direction, but this implementation is not the best fix because it silently fails open across a documented proxy trust boundary and does not preserve newer current-main proxy/env contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve fail-closed proxy mode before direct fallback:** `src/infra/net/proxy-fetch.ts:17-28`
  - body: The wrapper routes requests through global `fetch` whenever the circuit is open or a proxy connect error occurs. In managed proxy deployments, current docs describe the operator proxy as the runtime egress guardrail for HTTP and WebSocket traffic, so this silently bypasses that control when the proxy is unavailable. Keep managed proxy paths fail-closed unless a narrow explicit fail-open/debug policy is active.
  - confidence: 0.9
- **[P2] Only record gateway failures for proxied requests:** `src/discord/monitor/gateway-plugin.ts:77-80`
  - body: When `useProxy` is false, the metadata request already went direct, but the catch block still records connect-style errors against the proxy circuit. A transient direct Discord/network failure can extend the proxy cooldown and delay recovery even though the proxy was not used.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:70-73`
  - body: After cooldown expiry, `isProxyCircuitOpen` transitions to `half_open` but leaves the expired timestamp in place, so concurrent callers can all pass through and probe the proxy. That can inflate `consecutiveFailures` and recreate the burst of failed connects the breaker is meant to prevent.
  - confidence: 0.86
- **[P2] Preserve current env proxy resolution semantics:** `src/infra/net/proxy-fetch.ts:60-76`
  - body: This branch rewrites `resolveProxyFetchFromEnv` to read `process.env` directly and constructs `EnvHttpProxyAgent` without the current `resolveEnvHttpProxyAgentOptions` path. A rebased merge would drop the current injectable env contract and ALL_PROXY/lowercase precedence coverage unless the circuit wraps the existing resolved-agent behavior instead of replacing it.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete proxy trust-boundary concern by falling back to direct HTTP/WebSocket egress after proxy failures.

Concerns:

- **[high] Direct fallback can bypass managed proxy egress control:** `src/infra/net/proxy-fetch.ts:17`
  - body: The proposed circuit breaker uses global direct fetch while the proxy circuit is open or after proxy connect errors. For `proxy.enabled=true` deployments, current docs describe the operator proxy as the runtime egress guardrail, so protected traffic could leave the process directly when the proxy is unavailable.
