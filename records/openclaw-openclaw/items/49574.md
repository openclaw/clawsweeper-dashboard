## Summary

Keep open: the proxy-backoff idea is plausible, but this branch changes the documented fail-closed proxy boundary, has concrete correctness bugs, lacks real unavailable-proxy proof, and is stale against current Discord plugin paths.

## What This Changes

The branch adds proxy circuit-breaker state and direct-fetch fallback to shared proxy fetch plus old Discord monitor paths, lowers Discord gateway reconnect attempts, and tweaks rate-limit regexes.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer, keep managed proxy routing fail-closed by default, make any direct fallback explicit and documented, port Discord changes to current plugin files, and prove the result with focused tests plus a redacted real-runtime run.

## Reproduction Assessment

No high-confidence end-to-end reproduction of CPU exhaustion or retry-rate reduction is attached. Source inspection does show current main selects proxy dispatchers and rethrows request-time failures instead of using a circuit/backoff path.

## Solution Assessment

No. A shared backoff mechanism may be useful, but this implementation fails open around configured proxy routing, needs the gateway attribution and half-open fixes, and must be rebased onto the current plugin layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep configured proxy failures fail-closed:** `src/infra/net/proxy-fetch.ts:17-28`
  - body: When the circuit opens or a proxy connect fails, this wrapper retries with global fetch. For managed proxy deployments, that sends traffic outside the operator egress boundary instead of failing closed, so the circuit should back off without direct egress unless an explicit policy opts in.
  - confidence: 0.86
- **[P1] Record gateway failures only for proxied attempts:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: The direct gateway path sets useProxy false, but the catch block still records any connection-style error as a proxy failure. If direct Discord reachability fails while the circuit is open, this extends the proxy cooldown and can prevent recovery probes.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-67`
  - body: After cooldown expiry, every caller sees the same expired lastFailAt and probes the proxy. Under concurrent traffic this recreates the burst the circuit is meant to avoid and inflates the backoff counter.
  - confidence: 0.84
