## Summary

Keep open: the circuit-breaker idea is not on current main and addresses a plausible proxy reliability bug, but this PR should not merge as-is because it can fail open around configured proxy routing, has two concrete circuit-state bugs, targets now-moved Discord paths, and lacks real behavior proof.

## What This Changes

The PR adds a proxy circuit breaker and direct-fetch fallback, wires it into shared proxy fetch and Discord REST/gateway paths, lowers Discord gateway reconnect attempts, and broadens several rate-limit matchers.

## Best Possible Solution

Implement a current-main backoff/circuit mechanism in the shared proxy runtime and Discord extension paths that suppresses repeated dead-proxy attempts without silently bypassing configured proxy policy, then prove it with focused tests and a redacted real-runtime run.

## Reproduction Assessment

No high-confidence end-to-end reproduction is present. Source inspection shows current main has proxy fetch and Discord proxy paths without a circuit breaker, but the reported CPU exhaustion was not measured against a real unavailable-proxy setup.

## Solution Assessment

No. A circuit breaker/backoff is a reasonable direction, but this implementation fails open around configured proxy routing, needs the gateway and half-open fixes, and must be rebased onto the current Discord extension layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep configured proxy failures fail-closed:** `src/infra/net/proxy-fetch.ts:20-30`
  - body: When the circuit is open or a proxy connect fails, this wrapper sends the same request through global `fetch`. For deployments using the configured proxy as egress control, a temporary proxy outage would bypass that boundary and send traffic directly, contrary to the documented fail-closed proxy contract. The circuit should back off without direct egress unless an explicit policy says otherwise.
  - confidence: 0.86
- **[P1] Only record gateway failures for proxied attempts:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: In `registerClient`, the direct-fetch path sets `useProxy` false, but the `catch` still records any connection-style error as a proxy failure. If direct Discord reachability fails while the proxy circuit is open, this extends the proxy cooldown and can prevent recovery attempts.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-67`
  - body: After cooldown expiry, every caller in the half-open window sees the same expired `lastFailAt` and probes the proxy. Under concurrent traffic that recreates the connection burst the circuit is meant to avoid and inflates the backoff counter.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch changes proxy failure handling in a way that can bypass an operator-configured egress proxy.

Concerns:

- **[high] Direct fallback can bypass proxy egress policy:** `src/infra/net/proxy-fetch.ts:20`
  - body: The new circuit-open and connect-error paths call global `fetch` instead of failing closed through the configured proxy. OpenClaw documents managed proxy routing as central egress control and a fail-closed path, so this needs a different design or explicit policy before merge.
