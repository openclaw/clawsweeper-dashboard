## Summary

Keep open: the reported proxy retry/CPU problem is plausible and not implemented on current main, but this PR is not mergeable as-is because it introduces a fail-open proxy bypass, has circuit-state bugs, lacks real dead-proxy proof, and is stale against the current Discord plugin paths.

## What This Changes

The PR adds a shared proxy circuit breaker with direct-fetch fallback, wires it into proxy fetch and Discord proxy paths, lowers Discord gateway reconnect attempts, and expands a few rate-limit matchers.

## Best Possible Solution

Rework the idea as a fail-closed or explicitly policy-gated proxy circuit breaker on current `src/infra/net/proxy-fetch.ts` and `extensions/discord`, with bounded half-open probing, focused regression tests, and redacted real dead-proxy proof.

## Reproduction Assessment

No. Source inspection shows current main has no shared proxy circuit state, but this review did not run a real unavailable-proxy scenario that demonstrates CPU exhaustion.

## Solution Assessment

No. A circuit breaker may be a reasonable direction, but this implementation bypasses configured proxies, has circuit-state bugs, and needs a current-main rebase plus a maintainer decision on proxy failure policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not bypass configured proxies after failures:** `src/infra/net/proxy-fetch.ts:18-26`
  - body: When the circuit is open, and again after a connection-style proxy error, this wrapper sends the request through global `fetch` with no proxy dispatcher. That conflicts with the documented fail-closed operator proxy model and can route runtime traffic outside the configured egress/audit path.
  - confidence: 0.86
- **[P2] Gate proxy failure recording on actual proxy use:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: When `useProxy` is false, the gateway metadata request is direct, but the catch block still records connection errors against the proxy circuit. A transient direct-path failure can extend the proxy blackout and prevent recovery from being probed correctly.
  - confidence: 0.9
- **[P2] Reserve a single half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-68`
  - body: After cooldown expiry, `lastFailAt` remains expired while the state is `half_open`, so concurrent callers can all pass through and probe the proxy at once. That recreates a burst of failed proxy connects and inflates the exponential backoff counter.
  - confidence: 0.86
- **[P1] Only record gateway failures for proxied attempts:** `src/discord/monitor/gateway-plugin.ts:77-79`
  - body: In `registerClient`, the direct-fetch path sets `useProxy` false, but the `catch` still records any connection-style error as a proxy failure. If direct Discord reachability fails while the proxy circuit is open, this extends the proxy cooldown and can prevent recovery attempts.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-67`
  - body: After cooldown expiry, every caller in the half-open window sees the same expired `lastFailAt` and probes the proxy. Under concurrent traffic that recreates the connection burst the circuit is meant to avoid and inflates the backoff counter.
  - confidence: 0.84
