## Summary

Keep this PR open, but the current branch should not merge: it changes proxy failure handling to fail open, lacks real dead-proxy runtime proof, and still has concrete correctness issues already visible in the diff.

## What This Changes

The branch adds a shared proxy circuit breaker with direct-fetch fallback, applies it to shared proxy fetch and legacy Discord proxy fetch/gateway files, lowers Discord gateway reconnect attempts, broadens rate-limit regexes, and adds proxy circuit tests.

## Best Possible Solution

Design a fail-closed retry-suppression mechanism for the current shared proxy and active Discord plugin paths, preserving operator-managed egress and proving dead-proxy behavior with focused tests plus redacted runtime evidence.

## Reproduction Assessment

No high-confidence live reproduction is attached. Source inspection shows repeated proxy dispatcher attempts without circuit state are plausible on current main, but the multi-channel CPU exhaustion claim and after-fix behavior have not been proven in a real run.

## Solution Assessment

No. A circuit breaker may be useful, but this implementation fails open to direct egress, misaccounts some gateway failures, allows concurrent half-open bursts, and needs a current-main Discord plugin port.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve configured proxy routing on failures:** `src/infra/net/proxy-fetch.ts:18-28`
  - body: When the circuit is open or a proxy connection error occurs, this wrapper calls global fetch directly. That bypasses operator-managed proxy egress even though the current proxy contract is fail-closed for supported clients, so retry suppression needs to keep traffic on the configured proxy path unless maintainers approve a policy change.
  - confidence: 0.91
- **[P1] Guard proxy failure accounting by proxy use:** `src/discord/monitor/gateway-plugin.ts:78-79`
  - body: In registerClient, a circuit-open request runs without the proxy dispatcher, but the catch block still records a proxy failure for any connect-style error. A direct Discord or network failure can therefore extend the proxy cooldown even though the proxy was not attempted.
  - confidence: 0.93
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:70-73`
  - body: After cooldown expiry, isProxyCircuitOpen switches to half_open and returns false without reserving the probe window. Concurrent callers can all pass through, hit the proxy together, and inflate consecutiveFailures, recreating the burst the circuit breaker is meant to avoid.
  - confidence: 0.88
- **[P2] Keep request-token overflow out of rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:17`
  - body: Adding the broad string 'too many tokens per' also matches 'too many tokens per request'. Current tests keep that message as context overflow with no failover reason, so this change would misclassify per-request context errors as rate limits.
  - confidence: 0.86
