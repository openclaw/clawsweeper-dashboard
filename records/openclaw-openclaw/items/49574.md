## Summary

Keep this PR open. Current main still lacks the shared proxy-fetch circuit breaker, and the PR remains useful, but it is not mergeable as-is because it is stale against current plugin paths and its direct fallback behavior conflicts with managed proxy fail-closed semantics.

## What This Changes

The PR adds proxy circuit state with direct fallback for shared proxy fetch and Discord REST/gateway paths, reduces Discord gateway reconnect attempts, and includes unrelated rate-limit matcher updates.

## Best Possible Solution

Land a rebased, policy-aware implementation that adds proxy backoff at the shared helper/plugin-facing surface and current Discord/Telegram call sites, preserves managed-proxy fail-closed behavior, permits only one half-open probe per cooldown, records failures only for requests that actually used the proxy, and covers fallback, cooldown, recovery, and managed-proxy cases with focused tests.

## Reproduction Assessment

Yes. Configure an unavailable loopback proxy and exercise makeProxyFetch plus Discord/Telegram proxy call sites; current main source shows each request uses the proxy dispatcher without shared cooldown state, while the PR tests model connection-error fallback and cooldown behavior.

## Solution Assessment

No, not as proposed. A circuit breaker is a reasonable direction, but this branch needs a rebase and policy-aware fallback so protected managed-proxy deployments do not silently fail open.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve fail-closed proxy mode before direct fallback:** `src/infra/net/proxy-fetch.ts:17-27`
  - body: The wrapper falls back to global fetch when the circuit is open or a proxy connect error occurs. In managed proxy deployments, protected traffic is documented to route through the operator proxy and fail closed; this bypasses that egress control when the proxy is unavailable. Gate direct fallback on an explicit fail-open/debug policy and keep managed proxy paths on the proxy/failure path.
  - confidence: 0.88
- **[P2] Only record gateway failures for proxied requests:** `src/discord/monitor/gateway-plugin.ts:77-80`
  - body: When useProxy is false, the gateway metadata request is already direct, but the catch block still records connect-style errors against the proxy circuit. A transient direct Discord/network failure can extend the proxy cooldown and delay recovery even though the proxy was not used.
  - confidence: 0.9
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:64-76`
  - body: After cooldown expiry, the circuit transitions to half_open but leaves the expired timestamp in place, so concurrent callers can all pass through and probe the proxy. That can inflate consecutiveFailures and recreate the connection burst the breaker is meant to prevent.
  - confidence: 0.84
