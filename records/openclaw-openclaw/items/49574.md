## Summary

Keep open: the PR targets a plausible proxy retry-storm bug, but the current patch should not merge because it adds direct egress fallback, has circuit/accounting bugs, touches stale Discord paths, and lacks real dead-proxy runtime proof.

## What This Changes

The PR adds shared proxy circuit-breaker state, direct fallback after proxy connection failures, Discord REST/gateway wiring, a lower Discord gateway retry cap, and broader rate-limit text matching.

## Best Possible Solution

Design a fail-closed proxy retry/circuit breaker that suppresses repeated dead-proxy attempts without direct egress fallback, port it to current shared proxy and extensions/discord paths, and validate it with regression tests plus redacted runtime proof.

## Reproduction Assessment

No high-confidence runtime reproduction is attached. Current source makes repeated proxy attempts through configured dispatchers plausible, but the CPU-exhaustion claim still needs live dead-proxy, multi-channel proof.

## Solution Assessment

No. A circuit breaker may be a good direction, but this implementation bypasses configured proxy routing, has branch-local logic bugs, and needs a current-main Discord port.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured proxy routing on failures:** `src/infra/net/proxy-fetch.ts:18-28`
  - body: When the circuit is open or a proxy connection error occurs, this wrapper calls global fetch directly. That bypasses the operator-managed proxy even though current docs define managed proxy routing as a fail-closed egress guardrail, so retry suppression needs to keep traffic on the configured proxy path unless maintainers approve a policy change.
  - confidence: 0.9
- **[P1] Guard proxy failure accounting by proxy use:** `src/discord/monitor/gateway-plugin.ts:79`
  - body: In registerClient, a circuit-open request runs without the proxy dispatcher, but the catch block still records a proxy failure for any connect-style error. A direct Discord or network failure can therefore extend the proxy cooldown even though the proxy was not attempted.
  - confidence: 0.93
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:72`
  - body: After cooldown expiry, isProxyCircuitOpen switches to half_open and returns false without reserving the probe window. Concurrent callers can all observe the expired cooldown, all hit the proxy, and inflate consecutiveFailures, recreating the burst the circuit breaker is meant to avoid.
  - confidence: 0.88
- **[P2] Keep request-token overflow out of rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:17`
  - body: Adding the broad string 'too many tokens per' also matches 'too many tokens per request'. Current tests require that wording to stay in the context-overflow lane with no failover reason, so this change would misclassify per-request context errors as rate limits.
  - confidence: 0.86
