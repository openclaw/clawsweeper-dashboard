## Summary

Keep this PR open, but it is not merge-ready: the proposed direct fallback conflicts with the current fail-closed proxy contract, the branch still has concrete review bugs, and the contributor has not attached real dead-proxy runtime proof.

## What This Changes

The branch adds shared proxy circuit-breaker state, direct-fallback behavior, Discord REST/gateway wiring, a lower Discord gateway retry limit, and broader rate-limit matching.

## Best Possible Solution

Design a fail-closed proxy retry/circuit breaker that suppresses repeated dead-proxy attempts without direct egress fallback, port it to the current shared proxy and extensions/discord paths, and validate it with regression tests plus real dead-proxy runtime proof.

## Reproduction Assessment

No. Current main source makes repeated proxy fetches through the configured dispatcher and the retry-storm claim is plausible, but there is no high-confidence live multi-channel dead-proxy CPU reproduction or after-fix runtime proof.

## Solution Assessment

No. A circuit breaker may be the right direction, but this PR bypasses configured proxy routing and needs a fail-closed design plus a current-main Discord port.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured proxy routing on failures:** `src/infra/net/proxy-fetch.ts:18-28`
  - body: When the circuit is open or a proxy connect error occurs, this wrapper calls global fetch directly. That silently bypasses an operator-managed proxy even though the current proxy docs define this path as fail-closed egress control, so retry suppression needs to keep traffic on the configured proxy path unless maintainers explicitly approve a new policy.
  - confidence: 0.9
- **[P1] Guard proxy failure accounting by proxy use:** `src/discord/monitor/gateway-plugin.ts:79`
  - body: In registerClient, a circuit-open request runs without the proxy dispatcher, but the catch block still records a proxy failure for any connect-style error. A direct Discord/network failure can therefore extend the proxy cooldown even though the proxy was not attempted.
  - confidence: 0.93
- **[P2] Allow only one half-open proxy probe:** `src/infra/net/proxy-probe.ts:72`
  - body: After cooldown expiry, isProxyCircuitOpen switches to half_open and returns false without reserving the probe window. Concurrent callers can all observe the expired cooldown, all hit the proxy, and inflate consecutiveFailures, recreating the burst the circuit breaker is meant to avoid.
  - confidence: 0.88
- **[P2] Keep request-token overflow out of rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:17`
  - body: Adding the broad string 'too many tokens per' also matches 'too many tokens per request'. Current tests keep that message as context overflow with no failover reason, so this change would misclassify per-request context errors as rate limits.
  - confidence: 0.86
