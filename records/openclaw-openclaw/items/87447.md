## Summary

Keep this open: current `main` and the latest release still append Dream Diary entries without a retention cap, and the only built-in cleanup I found dedupes exact duplicates rather than bounding file growth. The severity is a little lower than the issue body suggests because current bootstrap sources do not list `DREAMS.md` as a default every-turn bootstrap file, but the unbounded generated artifact is still a real memory/dreaming maintenance problem.

## What This Changes

The issue asks OpenClaw to bound, rotate, or archive Dream Diary entries so `DREAMS.md` does not grow indefinitely.

## Best Possible Solution

Define a maintainer-approved internal retention policy for the Dream Diary, then implement it in `memory-core` with regression coverage and docs that explain what is kept and what is archived or discarded.

## Reproduction Assessment

Yes. Source inspection shows each narrative run appends to `DREAMS.md` with no size or entry-count bound, although I did not run a live multi-week dreaming scenario.

## Solution Assessment

Unclear. A bounded internal retention policy is the right direction, but maintainers should choose the retention/archive semantics before implementation.
