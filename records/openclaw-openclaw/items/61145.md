## Summary

Keep open: current main still lacks `gateway.webchat.defaultPeerId`, and the PR is useful but incomplete because it promises WebChat/TUI routing while only wiring the TUI path and missing generated config metadata plus changelog coverage. The related browser Control UI behavior is still tracked by open #67735, so the work is not obsolete.

## What This Changes

The PR adds an opt-in `gateway.webchat.defaultPeerId` config, docs, schema/type entries, gateway snapshot `dmScope`, and TUI session-key tests/logic for peer-based default session routing.

## Best Possible Solution

Keep the PR open and align the contract: either fully wire the opt-in peer default through gateway snapshot/browser Control UI plus TUI with generated metadata, docs, tests, and changelog, or narrow this change to TUI and leave browser behavior to #67735.

## Reproduction Assessment

Yes. The high-confidence path is source-level: with `gateway.webchat.defaultPeerId` configured, the PR's TUI helper can select a peer key, but browser Control UI still maps empty/default sessions to `sessionDefaults.mainSessionKey` because `ui/src/ui/app-gateway.ts` is unchanged.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is the right narrow mechanism, but the proposed public config is incomplete until browser default selection, generated config metadata, changelog, and tests are aligned or the scope is reduced.
