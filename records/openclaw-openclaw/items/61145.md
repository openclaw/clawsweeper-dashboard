## Summary

Keep open: this PR is still a focused implementation candidate for the WebChat/Control UI identityLinks gap, but the current diff is incomplete and should not merge yet because it only wires TUI behavior while documenting WebChat behavior, leaves generated config metadata stale, and lacks after-fix real behavior proof.

## What This Changes

Adds optional `gateway.webchat.defaultPeerId` config/schema/docs/snapshot plumbing plus TUI session-key logic/tests to route empty defaults through peer-based DM session construction.

## Best Possible Solution

Align the public contract before merge: either fully wire the opt-in peer default through browser Control UI, TUI, config metadata, protocol/docs, and account variants, or narrow this branch to TUI-only while leaving full WebChat continuity to the canonical issue.

## Reproduction Assessment

Yes for the PR defects by source inspection. The branch promises WebChat/TUI peer-default routing, but current browser Control UI still normalizes default selections to `sessionDefaults.mainSessionKey` and sends `state.sessionKey` directly.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is a reasonable mechanism, but the proposed public config is incomplete until browser WebChat behavior, generated config metadata, and product semantics are aligned.
