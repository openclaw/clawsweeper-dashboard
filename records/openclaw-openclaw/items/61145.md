## Summary

This PR should stay open: the opt-in peer-default idea addresses a real session-continuity gap, but the branch still promises browser WebChat behavior that the current Control UI path does not honor, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds optional `gateway.webchat.defaultPeerId` config, schema/docs/protocol snapshot fields, and TUI session-key logic/tests for resolving empty/default selections through peer-based DM routing.

## Best Possible Solution

Land a consistent opt-in session-default contract that either wires both browser Control UI and TUI through the canonical peer session, with generated metadata and proof, or deliberately narrows the feature and docs to TUI-only while the broader browser behavior remains in the canonical issue.

## Reproduction Assessment

Yes for the remaining PR defects by source inspection: the browser Control UI initializes from `sessionDefaults.mainSessionKey` and sends `state.sessionKey` directly, while the PR only wires the new default into TUI logic. The contributor has not supplied real after-fix WebChat/TUI runtime proof.

## Solution Assessment

Unclear as proposed. The TUI helper change is plausible, but the PR is not the best complete solution while it documents browser WebChat behavior without wiring the browser Control UI contract or generated config metadata.
