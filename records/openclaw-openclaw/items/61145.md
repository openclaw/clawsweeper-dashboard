## Summary

Keep open: this is a focused feature PR, but the current branch is incomplete for its stated WebChat/TUI contract. Browser Control UI still defaults empty selections to the main session, generated config help/labels are not updated for the new key, the related canonical issue is still open, and the external PR has only source/unit proof rather than after-fix real behavior proof.

## What This Changes

Adds optional `gateway.webchat.defaultPeerId` config/schema/docs/snapshot plumbing plus TUI session-key logic/tests to route default sessions through peer-based DM session construction.

## Best Possible Solution

Align the public contract before merge: either wire the opt-in peer default through browser Control UI, TUI, Gateway snapshot/protocol, generated config metadata, docs, and tests, or narrow this branch to TUI-only while leaving full WebChat continuity to https://github.com/openclaw/openclaw/issues/67735.

Yes for the PR defects by source inspection. The branch promises WebChat/TUI peer-default routing, but the diff only changes TUI resolution while current browser Control UI still normalizes default selections to `sessionDefaults.mainSessionKey`.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is a reasonable mechanism, but the proposed public config is incomplete until browser WebChat behavior, generated config metadata, and maintainer product semantics are aligned.
