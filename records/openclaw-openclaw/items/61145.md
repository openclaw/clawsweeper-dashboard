## Summary

Keep open. Current main still lacks the requested peer-based WebChat/TUI default, and the proposed branch is useful but incomplete: it wires TUI-only behavior while the browser WebChat/Control UI path still defaults through the main-session snapshot and the generated config surfaces are stale.

## What This Changes

The PR adds an opt-in `gateway.webchat.defaultPeerId` config, schema/docs entries, gateway snapshot `dmScope`, and TUI session-key logic/tests for peer-based default session routing.

## Best Possible Solution

Keep the PR open, rebase/update it against current main, and make the runtime/docs/config contract consistent. The maintainable endpoint is either a fully wired opt-in peer default for both TUI and browser WebChat/Control UI, or a narrower TUI-only config/docs change with the broader browser behavior left to #67735.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: with `gateway.webchat.defaultPeerId` configured, the PR’s TUI helper would select a peer session, but browser WebChat/Control UI still applies `snapshot.sessionDefaults.mainSessionKey` for empty/default sessions.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is the right narrow mechanism, but the patch is incomplete until the browser default-session path, generated config metadata, and changelog are updated or the documented scope is reduced.
