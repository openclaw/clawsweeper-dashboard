## Summary

Keep open. The earlier undeclared-variable review issue appears fixed in the current PR head, but the branch still promises WebChat/TUI behavior while only wiring the TUI helper, leaves generated config metadata out of sync, and lacks after-fix real behavior proof.

## What This Changes

The PR adds an opt-in `gateway.webchat.defaultPeerId` config, docs, snapshot `dmScope`, and TUI session-key tests/logic for peer-based default session routing.

## Best Possible Solution

Align the contract before merge: either fully wire the opt-in peer default through gateway snapshot/browser Control UI plus TUI with generated metadata and focused tests, or narrow the change to TUI and leave browser WebChat behavior to https://github.com/openclaw/openclaw/issues/67735.

## Reproduction Assessment

Yes. The source-level path is high confidence: with `gateway.webchat.defaultPeerId` configured, the PR's TUI helper can select a peer key, but browser Control UI still maps empty/default sessions to `sessionDefaults.mainSessionKey` because `ui/src/ui/app-gateway.ts` is unchanged.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is a reasonable TUI mechanism, but the proposed public WebChat/TUI config is incomplete without browser default selection and generated config metadata, and the WebChat session-model boundary needs maintainer judgment.
