## Summary

Keep open. Current main still defaults TUI and browser WebChat/Control UI to the agent main session, and this PR is useful but incomplete because it only wires the peer-default behavior into the TUI path while documenting WebChat behavior and adding a config key without the generated config surfaces.

## What This Changes

The PR adds an opt-in `gateway.webchat.defaultPeerId` config, schema/docs entries, gateway snapshot `dmScope`, and TUI session-key logic/tests for peer-based default session routing.

## Best Possible Solution

Keep the PR open, rebase it onto current main, and make the runtime/docs/config contract consistent: either implement the configured peer default for both TUI and browser WebChat/Control UI, or narrow the public config/docs to TUI only. The finished change should regenerate config schema/help metadata and add focused coverage for the browser default-session path as well as TUI fallback behavior.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: with `gateway.webchat.defaultPeerId` configured, browser WebChat still applies `snapshot.sessionDefaults.mainSessionKey` for empty/default sessions, while current main has no peer-default config surface at all.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is the right narrow direction, but the proposed patch is incomplete until the browser WebChat path and generated config metadata are updated or the feature scope is reduced.
