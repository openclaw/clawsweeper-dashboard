## Summary

Keep open: current main does not implement peer-default WebChat/TUI routing, and this PR is a plausible but incomplete feature branch. It still overpromises browser WebChat behavior, misses generated config metadata, and lacks real after-fix behavior proof.

## What This Changes

Adds optional `gateway.webchat.defaultPeerId` config/schema/docs/snapshot plumbing plus TUI session-key logic/tests to route default WebChat/TUI sessions through peer-based DM session construction.

## Best Possible Solution

Align the contract before merge: either wire the opt-in peer default through browser Control UI, TUI, Gateway snapshot/protocol, generated config metadata, docs, and tests, or narrow this branch to TUI-only and leave full WebChat continuity to https://github.com/openclaw/openclaw/issues/67735.

## Reproduction Assessment

Yes for the PR defects by source inspection. The branch promises WebChat/TUI peer-default routing, but the diff only changes TUI resolution while current browser Control UI still normalizes default selections to `sessionDefaults.mainSessionKey`.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is a reasonable mechanism, but the proposed public config is incomplete until browser WebChat behavior and generated config metadata are aligned or the scope is reduced.
