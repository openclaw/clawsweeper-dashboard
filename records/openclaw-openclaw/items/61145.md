## Summary

Keep open: current main still lacks the new WebChat/TUI default-peer config, and the PR is useful but incomplete because the browser Control UI path still defaults empty sessions to the main session while the added docs promise WebChat/TUI peer routing. Generated config metadata and changelog coverage are also missing, and open #67735 tracks the broader browser WebChat behavior.

## What This Changes

The PR adds an opt-in `gateway.webchat.defaultPeerId` config with docs/schema/type updates, exposes `dmScope` in the gateway snapshot, and updates TUI session resolution tests/logic for peer-based default routing.

## Best Possible Solution

Align the contract before merge: either wire the opt-in peer default through gateway snapshot/browser Control UI and TUI with generated metadata, tests, docs, and changelog, or narrow the docs/config to TUI-only and leave browser WebChat continuity to #67735.

## Reproduction Assessment

Yes. The high-confidence path is source-level: with `gateway.webchat.defaultPeerId` configured, the PR's TUI helper can select a peer key, but browser Control UI still maps empty/default sessions to `sessionDefaults.mainSessionKey` because `ui/src/ui/app-gateway.ts` is unchanged.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is the right mechanism, but the proposed public config is incomplete until browser default selection, generated config metadata, changelog, and tests are aligned or the scope is reduced.
