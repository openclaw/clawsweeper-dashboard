## Summary

Keep open: the PR is useful but incomplete. It addresses the TUI path, but the promised WebChat behavior is not wired through the browser Control UI, generated config metadata is missing, and the related WebChat identity-link behavior is still tracked in open #67735.

## What This Changes

The PR adds optional `gateway.webchat.defaultPeerId` config, schema/docs/snapshot updates, and TUI resolver tests/logic to default empty WebChat/TUI sessions through peer DM routing.

## Best Possible Solution

Align the public contract by either wiring the opt-in peer default through gateway snapshot/browser Control UI with generated config metadata, docs, tests, and changelog, or narrowing this PR to TUI-only while leaving browser behavior to #67735.

## Reproduction Assessment

Yes. The source-level path is high-confidence: with `gateway.webchat.defaultPeerId`, the PR's TUI helper can choose a peer key, but browser Control UI still normalizes empty/default sessions to `snapshot.sessionDefaults.mainSessionKey` because `ui/src/ui/app-gateway.ts` is unchanged.

## Solution Assessment

No, not as-is. Reusing `buildAgentPeerSessionKey` is the right narrow mechanism for TUI, but the public WebChat config is incomplete until browser default selection, generated metadata, changelog, and tests are aligned or the scope is reduced.
