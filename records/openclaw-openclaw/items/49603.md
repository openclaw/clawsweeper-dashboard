## Summary

Close as implemented. The merged session-lock fix now reclaims untracked current-process locks with matching starttime during acquisition and startup cleanup, with regression coverage and release notes shipped; the later sidecar and heartbeat comments are distinct follow-up concerns.

## What This Changes

The issue reports same-PID orphan session lock files surviving gateway restart or state reset and blocking later session writes.

## Best Possible Solution

Keep the shipped session-aware orphan-self lock cleanup, and open focused follow-up issues only for any remaining heartbeat or transcript-sidecar recovery behavior still reproducible on current releases.

## Reproduction Assessment

Yes for the historical bug: create a `.jsonl.lock` with `pid: process.pid` and matching `starttime` while no in-memory lock entry exists, then run acquisition or cleanup on the pre-fix code. Current main is not expected to reproduce it because the fixed classifier and regression tests now remove that lock class.

## Solution Assessment

Yes. The shipped fix is the maintainable path because it reuses shared session-aware inspection, checks in-memory ownership before deleting, and avoids the unsafe blanket rule of deleting every current-PID lock.
