## Summary

Close as implemented: current main reclaims untracked current-process session locks during acquisition and startup cleanup, covers that path with regression tests, and the merged fix shipped in v2026.5.2.

## What This Changes

The issue asks for orphaned self-owned session lock files to be cleared when the gateway restarts or loses in-memory lock ownership while the PID remains alive.

## Best Possible Solution

Keep the shipped shared lock classifier and use separate focused issues for broader runtime heartbeat or transcript-sidecar cleanup requests if they still reproduce after v2026.5.2.

## Reproduction Assessment

Yes for the original bug: create a fresh `.jsonl.lock` with `pid: process.pid` and matching current `starttime` while no HELD_LOCKS entry exists, then run cleanup or acquisition. I did not run tests in this read-only review, but current main now includes focused regression coverage for that path.

## Solution Assessment

Yes. The merged implementation centralizes orphan-self classification, preserves active in-process locks, and applies the same decision to acquisition and startup cleanup without adding a new config surface.
