## Summary

Close as implemented: current main now reclaims untracked self-owned session locks for the original restart/current-PID lock wedge, and the fix is included in v2026.5.12. The newer runtime heartbeat/stuck-session concerns are broader than this boot-time lock cleanup and remain tracked in https://github.com/openclaw/openclaw/issues/71127.

## What This Changes

The issue requests recovery for orphaned `agents/*/sessions/*.jsonl.lock` files owned by the gateway's current PID so restarts do not leave sessions permanently blocked.

## Best Possible Solution

Keep the shipped session-lock reclaim behavior and use https://github.com/openclaw/openclaw/issues/71127 for the broader runtime stall, heartbeat, and model/tool abort policy work.

## Reproduction Assessment

Yes for the original pre-fix failure: a focused source reproduction is a `.jsonl.lock` owned by `process.pid` with no in-memory held entry. Current main no longer reproduces the restart/current-format case because cleanup and acquisition classify that lock as `orphan-self-pid` and regression tests cover it.

## Solution Assessment

Yes. The shipped fix is narrower and safer than deleting all current-PID locks because it uses session-aware held-lock state plus process starttime, while preserving active in-process locks.
