## Summary

Current main now reclaims the reported untracked same-PID session locks, and the merged implementation is present in the verified v2026.5.18 release. Later comments describe broader lock-liveness and transcript-sidecar cleanup concerns, but those are separate from the fixed current-PID startup/acquisition deadlock path.

## What This Changes

The issue requests recovery for session lock files owned by the current gateway PID after in-memory lock ownership is lost.

## Best Possible Solution

Keep the merged session-lock reclaim behavior and route any fresh reports about lock heartbeats, long-held live locks, or transcript sidecar cleanup to narrower follow-up issues with current-version logs.

## Reproduction Assessment

No against current main. The former high-confidence source repro, a fresh current-PID lock with matching starttime and no held in-memory entry, is now classified as orphan-self-pid and removed by cleanup.

## Solution Assessment

Yes. The merged fix uses the existing lock metadata and held-lock registry rather than adding a new startup-only sweeper, so it covers both acquisition and startup cleanup while preserving active in-process locks.
