## Summary

Close as implemented: current main and v2026.5.22 now reclaim untracked current-process session locks with matching starttime during cleanup, and the remaining no-restart live-lock reports are tracked separately.

## What This Changes

The issue asks Gateway startup cleanup to remove orphan `agents/*/sessions/*.jsonl.lock` files owned by the current PID when in-memory lock state has been lost.

## Best Possible Solution

Keep the merged session-lock cleanup implementation and use the newer runtime-lock issue for heartbeat or no-restart lock-liveness work.

## Reproduction Assessment

No current-main failing reproduction remains. The historical source-level fixture is now covered by a regression test that creates a current-PID lock with matching starttime and expects cleanup to remove it.

## Solution Assessment

Yes. The merged fix reuses shared lock inspection for acquisition, startup cleanup, and doctor cleanup instead of adding a one-off boot sweeper; broader live-lock heartbeat behavior should stay in the separate runtime-lock track.
