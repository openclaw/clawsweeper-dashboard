## Summary

Keep open. Current main still preserves fresh current-PID session lock files during startup cleanup, while two open PRs already target the remaining self-owned lock recovery path.

## What This Changes

The issue requests boot-time cleanup of self-owned orphan session lock files when the gateway loses in-memory lock ownership but the PID still matches the running process.

## Best Possible Solution

Startup/shared cleanup should reclaim untracked current-process .jsonl.lock files before session work, including current-format starttime locks when HELD_LOCKS lacks ownership, while preserving active in-process locks.

## Reproduction Assessment

Yes. A focused source-level reproduction can create a fresh .jsonl.lock owned by process.pid with empty HELD_LOCKS, call cleanStaleLockFiles({ removeStale: true }), and observe current main preserving it.

## Solution Assessment

No. Current main only recovers some self-owned orphan cases during later acquisition contention; the maintainable fix belongs in the startup/shared cleanup contract that the open PRs are already trying to adjust.
