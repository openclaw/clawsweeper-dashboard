## Label Justifications

- `P1`: The reported failure can wedge active session processing until manual lock deletion or gateway recovery.
- `impact:session-state`: The issue is about persistent `.jsonl.lock` files disagreeing with in-memory session lock ownership.
- `impact:crash-loop`: The reported lock state can make gateway/session work appear stuck across restarts and require operator intervention.

## Summary

Current main and the shipped v2026.5.3 release already implement the central startup recovery: untracked self-owned session locks with matching process start time are classified as `orphan-self-pid` and removed during session-lock cleanup, which gateway startup invokes for agent session directories.

## What This Changes

The issue asks for gateway startup/session-lock cleanup to reclaim orphaned `.jsonl.lock` files owned by the current gateway PID when in-memory lock state has been lost.

## Best Possible Solution

Keep the shipped session-lock cleanup implementation and track any remaining runtime heartbeat or transcript-sidecar recovery gaps in separate, current-main issues.

## Reproduction Assessment

Yes for the pre-fix bug: a focused source-level reproduction is an untracked current-PID `.jsonl.lock` with matching `starttime`, then running `cleanStaleLockFiles({ removeStale: true })`. Current main has the regression test and source path showing that case is now removed.

## Solution Assessment

Yes. The shipped implementation centralizes orphan-self classification, covers startup cleanup and acquisition, and preserves actively held locks through the shared inspection path.
