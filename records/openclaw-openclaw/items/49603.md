## Summary

Keep open. Current main and v2026.6.6 partially handle same-PID session-lock cleanup, but source still preserves fresh OpenClaw-owned current-PID locks without `starttime` during startup cleanup, and an open closing PR remains conflicting and needs maintainer handling.

## What This Changes

The issue asks gateway startup cleanup to reclaim orphaned session `.jsonl.lock` files owned by the current process so later session writes do not time out.

## Best Possible Solution

Review or replace the linked PR with a narrow, maintainer-approved startup cleanup policy for untracked self-owned locks, while keeping broader heartbeat/status runtime-lock work in separate canonical issues if needed.

## Reproduction Assessment

Yes, at source level. A fresh current-PID `.jsonl.lock` without `starttime` and with an OpenClaw owner is preserved by startup cleanup because `reclaimLockWithoutStarttime` is false there; I did not run a live gateway repro in this read-only pass.

## Solution Assessment

Unclear. A boot-time sweep is the right layer for this specific recovery, but maintainers need to decide the no-starttime/live self-lock policy and either repair or replace the conflicting linked PR.
