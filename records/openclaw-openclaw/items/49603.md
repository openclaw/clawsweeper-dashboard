## Label Justifications

- `P1`: The original bug could wedge session writes behind a live gateway PID until manual lock deletion, breaking active agent/channel workflows.
- `impact:session-state`: The report is about disk lock files disagreeing with in-memory session lock state after gateway recovery.
- `impact:crash-loop`: The failure mode can leave the gateway effectively unavailable for affected sessions through repeated lock timeouts.

## Summary

Current main now fixes the same-process orphan session-lock cleanup path: untracked current-PID locks with matching process start time are classified as `orphan-self-pid`, removed by startup cleanup, and covered by regression tests. The merged fix is present in release `v2026.5.2`.

## What This Changes

The issue requests gateway startup recovery for orphaned `agents/*/sessions/*.jsonl.lock` files whose PID matches the live gateway process after in-memory lock state is lost.

## Best Possible Solution

Keep the shipped session-lock reclaim implementation from https://github.com/openclaw/openclaw/pull/75822 and split any remaining heartbeat or transcript-sidecar cleanup behavior into narrower follow-up reports with current-release evidence.

## Reproduction Assessment

No current-main failure remains for this exact bug. The pre-fix path was source-reproducible by writing a current-PID `.jsonl.lock` with matching starttime and no held in-memory entry, and current tests now cover cleanup and acquisition recovery for that case.

## Solution Assessment

Yes. The merged fix is the narrow maintainable solution: reuse session-aware lock inspection to reclaim only untracked self-owned locks, preserve active in-process locks, and keep legacy no-starttime cleanup conservative while acquisition still recovers that shape.
