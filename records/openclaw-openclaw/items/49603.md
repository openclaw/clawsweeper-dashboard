## Summary

Keep open. Current main still runs the startup session-lock sweep through `cleanStaleLockFiles()`, but that helper only removes locks already marked stale by PID/age/starttime inspection; the self-owned orphan path remains acquisition-contended only, and open PR #49660 still references this issue with closing syntax.

## What This Changes

The issue requests boot-time cleanup of session `.jsonl.lock` files owned by the current gateway PID when in-memory session lock state has been lost, preventing later session attempts from timing out on a lock that only appears live.

## Best Possible Solution

Keep the issue open and review PR #49660 against current main. The end state should put self-owned startup-orphan classification in the startup/shared cleanup path, preserve truly active in-process locks, log a clear `orphan-self-pid` reason, and add focused regression coverage for both `cleanStaleLockFiles()` and the gateway startup sidecar path.

## Reproduction Assessment

Yes. A focused unit reproduction can create a fresh `.jsonl.lock` owned by `process.pid` with empty in-memory `HELD_LOCKS`, call `cleanStaleLockFiles({ removeStale: true })`, and observe current main preserving it; I verified the behavior from source and existing tests in this read-only review.

## Solution Assessment

No, current main is not the best solution because the only self-owned orphan recovery is in the later acquisition contention path. The maintainable fix belongs in startup cleanup or a shared cleanup contract so operators get recovery before session work begins.
