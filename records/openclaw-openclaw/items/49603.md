## Summary

Close as implemented: current main now classifies untracked self-owned `.jsonl.lock` files as `orphan-self-pid`, startup cleanup removes them, and the fix shipped in `v2026.5.2`. The later transcript-sidecar accumulation report is related operationally but distinct from the original self-owned lock-file bug and remains better tracked separately.

## What This Changes

The issue asks for gateway restart cleanup of session lock files owned by the current process after in-memory lock state is lost.

## Best Possible Solution

Keep the shipped `v2026.5.2` session-lock implementation and handle broader lock/cleanup accumulation through the separate canonical thread if it is still reproducible.

## Reproduction Assessment

Yes for the original failure mode: a focused source-level setup writes a fresh `.jsonl.lock` with `pid: process.pid` and matching starttime while no `HELD_LOCKS` entry exists. Current main now has regression coverage for that setup and removes the lock instead of preserving it, so the current-main failure is not reproduced.

## Solution Assessment

Yes. The merged fix is the narrow maintainable solution because it shares session-aware orphan-self classification across acquisition and startup cleanup while preserving active in-process locks.
