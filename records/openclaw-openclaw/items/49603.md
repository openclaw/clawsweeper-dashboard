## Summary

Keep open. Current main still runs startup cleanup through `cleanStaleLockFiles()`, which only removes locks classified stale by PID/age/starttime inspection; untracked self-owned locks with a live current PID are not swept before session work, and open PR #49660 still references this issue with closing syntax.

## What This Changes

The request is to clear current-process-owned session `.jsonl.lock` files on gateway startup when the in-memory lock map has been lost.

## Best Possible Solution

Shared session-lock cleanup should classify untracked self-owned locks before startup work, preserve locks still present in `HELD_LOCKS`, report `orphan-self-pid`, and cover startup, acquisition, and doctor cleanup behavior with focused tests.

## Reproduction Assessment

Yes. A focused unit reproduction can write a `.jsonl.lock` owned by `process.pid` with a valid current starttime and no matching `HELD_LOCKS` entry, then call `cleanStaleLockFiles({ removeStale: true })` and observe current main preserving it.

## Solution Assessment

No. Current main is not sufficient, and the supplied PR #49660 body appears to target the legacy missing-starttime subset; PR #75822’s stated boundary is closer because it covers untracked self-owned current-format locks.
