## Summary

Close as implemented: the startup current-PID orphan `.jsonl.lock` cleanup is now on current `main`, covered by regression tests, and shipped in v2026.5.3 via the merged session-lock fix.

## What This Changes

The issue asks for gateway startup/session-lock cleanup to reclaim orphaned self-owned session lock files when the live PID matches the current gateway but in-memory lock ownership was lost.

## Best Possible Solution

Keep the shipped v2026.5.3 session-lock implementation and use a new narrow issue for any remaining current-release runtime heartbeat or transcript-sidecar cleanup behavior.

## Reproduction Assessment

Yes for the pre-fix bug: create a current-PID `.jsonl.lock` with matching `starttime` and no in-memory held-lock entry, then run startup/acquisition cleanup. Current main no longer reproduces that failure because the cleanup path classifies it as `orphan-self-pid` and tests cover removal.

## Solution Assessment

Yes. The merged implementation is the maintainable fix for this issue because it centralizes orphan-self classification for startup cleanup and acquisition while keeping active in-process acquisition locks protected.
