## Label Justifications

- `P1`: A stale session write lock can block live gateway/channel turns until manual recovery.
- `impact:session-state`: The issue is about persisted transcript lock state disagreeing with in-memory session-lock ownership.
- `impact:message-loss`: Subsequent messages in the affected session can time out or fail while the lock remains.
- `impact:crash-loop`: The failure mode is tied to gateway restart/recovery and can leave the runtime effectively wedged.

## Summary

Current main and the latest release already cover the central same-PID orphan session-lock case: untracked current-process `.jsonl.lock` files with matching process start time are classified as `orphan-self-pid` and reclaimed during both startup cleanup and acquisition retry.

## What This Changes

The issue asks OpenClaw to recover from stale session JSONL lock files owned by the live gateway PID after in-memory lock ownership has been lost.

## Best Possible Solution

Keep the merged session-lock implementation from OpenClaw pull request https://github.com/openclaw/openclaw/pull/75822 and track broader lock-liveness or transcript-sidecar cleanup problems in narrower follow-up issues.

## Reproduction Assessment

Yes. Source and regression tests show the repro shape: create a `.jsonl.lock` with `pid === process.pid`, no in-process held lock, and matching `starttime`; current main now classifies it as `orphan-self-pid` and removes or reclaims it.

## Solution Assessment

Yes. The implemented fix lives in the session-lock owner module and is exercised from both gateway startup cleanup and acquisition retry, which is narrower and more maintainable than a gateway-only special-case sweep.
