## Label Justifications

- `P1`: The original report described session lock deadlocks that could block gateway/channel workflows until manual cleanup.
- `impact:session-state`: The problem was persistent session lock files disagreeing with in-memory session lock state after restart.
- `impact:crash-loop`: The reported timeout loop could leave the session workflow repeatedly unusable until the lock was removed.

## Summary

Close as implemented: current main and release `v2026.5.22` reclaim untracked current-process session locks during startup cleanup, with regression coverage for the exact self-PID lock shape; later comments describe broader runtime lock leaks and cleanup issues that are already better tracked separately.

## What This Changes

Issue requests clearing orphaned session lock files owned by the current gateway PID during gateway restart so same-process stale locks do not block future session writes.

## Best Possible Solution

Keep the merged self-PID startup cleanup from https://github.com/openclaw/openclaw/pull/75822, close this boot-time lock issue, and route no-restart lock leaks to https://github.com/openclaw/openclaw/issues/84193 or narrower cleanup reports.

## Reproduction Assessment

Yes. A focused source-level reproduction is to write a `.jsonl.lock` with `pid: process.pid`, a matching `starttime`, and no held lock entry, then run startup cleanup; the current regression test now proves that path is removed as `orphan-self-pid`.

## Solution Assessment

Yes. The merged solution is the narrow maintainable fix for this issue: it shares orphan-self classification with cleanup/acquisition, preserves tracked active locks, and avoids turning broader runtime heartbeat/compaction policy into this startup cleanup issue.
