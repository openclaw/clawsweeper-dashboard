## Summary

Current main and the v2026.5.3 release already implement the restart-time self-owned session-lock cleanup requested here; newer comments about runtime lock heartbeats/stuck-session aborts describe a broader recovery-policy problem that is already tracked separately.

## What This Changes

The issue asks for startup recovery of orphaned session lock files owned by the current gateway PID when in-memory lock state has been lost.

## Best Possible Solution

Keep the shipped v2026.5.3 session-lock reclaim implementation, and route runtime liveness or stuck-session abort policy to https://github.com/openclaw/openclaw/issues/71127 instead of keeping this fixed restart-cleanup issue open.

## Reproduction Assessment

Yes for the pre-fix bug: a current-PID `.jsonl.lock` with matching `starttime` and no `HELD_LOCKS` entry was the focused source-level reproduction. Current main no longer reproduces that path because cleanup now marks it `orphan-self-pid` and tests cover removal.

## Solution Assessment

Yes. Centralizing orphan-self classification in the session-lock inspection path is the narrow maintainable fix for startup cleanup and acquisition, while heartbeat/status abort policy is a separate runtime recovery decision.
