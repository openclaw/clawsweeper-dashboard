## Summary

Keep open. Current main still does not perform a startup/shared cleanup of fresh self-owned session lock files whose PID is the current process and whose in-memory lock entry was lost; recovery is limited to stale classification and a later acquisition-time path. Open PR #49660 still references this issue with closing syntax, and newer related PR #75822 appears to target the current-format matching-starttime variant, so the issue should remain open until maintainers choose and land the canonical fix.

## What This Changes

The issue asks OpenClaw to sweep or reclaim orphaned `agents/*/sessions/*.jsonl.lock` files owned by the current gateway PID after in-memory `HELD_LOCKS` state is lost, preventing subsequent session writes from timing out.

## Best Possible Solution

Keep this issue open and converge on a single canonical fix, likely by reviewing #49660 alongside the newer #75822 behavior. The end state should share self-owned orphan classification between startup cleanup, doctor inspection, and acquisition-time recovery, preserve truly active `HELD_LOCKS`, log an `orphan-self-pid` style reason, and add focused regression coverage for startup cleanup with current-format lock payloads.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a fresh `.jsonl.lock` payload with `pid: process.pid`, current `createdAt`, and a valid current `starttime`, leave `HELD_LOCKS` empty, then call `cleanStaleLockFiles({ removeStale: true })`; current main preserves it because it is not classified stale or self-orphaned.

## Solution Assessment

No. Current main is not the best solution because it recovers some contended legacy self-locks only during acquisition, while the reported failure needs startup/shared cleanup before session work begins; a shared classification helper used by cleanup and acquisition is the safer direction.
