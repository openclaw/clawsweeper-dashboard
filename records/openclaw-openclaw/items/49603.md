## Summary

Current main now reclaims untracked self-owned session lock files with matching process start time during both startup cleanup and acquisition, and the fix shipped in v2026.5.2.

## What This Changes

The issue reports that gateway/session cleanup can leave a self-owned `.jsonl.lock` file in place when its PID is the current process but the in-memory held-lock state was lost.

## Best Possible Solution

Keep the shipped shared classifier that handles self-owned orphan locks in both cleanup and acquisition; track transcript sidecar accumulation separately if it still reproduces after the lock fix.

## Reproduction Assessment

Yes. A high-confidence regression path is to create an untracked `.jsonl.lock` with `pid: process.pid` and matching `starttime`; current main has source and tests showing that cleanup and acquisition reclaim it instead of timing out.

## Solution Assessment

Yes. The merged fix is the narrow maintainable solution because it reuses one session-aware orphan-self classifier, preserves active `HELD_LOCKS` entries, and covers both startup cleanup and lock acquisition.
