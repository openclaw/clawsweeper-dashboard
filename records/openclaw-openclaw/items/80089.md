## Summary

Keep open. Current `main` still matches the reported failure mode: model registration is resolved from `models.providers` or runtime discovery, while `agents.defaults.models` is treated as allowlist/metadata, and the runtime still emits the generic `Unknown model` text without naming the missing provider model list.

## What This Changes

Request a clearer `Unknown model` diagnostic and docs for the case where a model is present only in `agents.defaults.models` but missing from `models.providers.<provider>.models[]`.

## Best Possible Solution

Add a narrow runtime diagnostic and a short docs example that tell operators to add the matching `models.providers.<provider>.models[]` entry when a requested full ref exists only in `agents.defaults.models`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with only `agents.defaults.models` set and no matching `models.providers` entry, the resolver has no registered model source and returns the generic unknown-model error.

## Solution Assessment

Yes. The narrowest maintainable fix is a diagnostic/docs improvement in the model-resolution path; automatic config mirroring would change the config contract and is not needed for this report.
