## Summary

The report is valid on current main: the model resolver only treats `models.providers` and runtime/provider discovery as model registration sources, while `agents.defaults.models` is only an allowlist/metadata surface, so the current runtime can still fall through to a generic `Unknown model` without naming the missing provider model entry.

## What This Changes

Improve the `Unknown model` diagnostic when a model appears in `agents.defaults.models` but is absent from `models.providers[provider].models[]`, with documentation for the two-block custom-provider setup.

## Best Possible Solution

Add a narrow runtime diagnostic, plus a short docs example, that tells operators to add the matching `models.providers.<provider>.models[]` entry when a requested full ref exists only in `agents.defaults.models`.

## Reproduction Assessment

Yes. Source inspection matches the reporter's command sequence: without a `models.providers` provider/model entry, the resolver has no registered model source and the error builder returns the generic `Unknown model` text.

## Solution Assessment

Yes. The narrowest maintainable fix is a diagnostic/docs improvement in the model-resolution path, not automatic config mirroring, because mirroring would change the config contract and overlap with broader open PRs.
