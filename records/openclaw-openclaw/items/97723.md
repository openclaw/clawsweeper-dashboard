## Summary

Keep open: this is a focused candidate fix for the open memory dreaming outcome-normalization bug, and the updated PR body now supplies real failed-run event-log proof. It still needs maintainer review because it changes a public plugin SDK event contract and the persistent memory dreaming event shape, and the PR has a protected `maintainer` label.

## What This Changes

The PR adds a `MemoryDreamOutcome` type with optional `outcome` and `error` fields on memory dreaming completion events, writes `completed` on successful dream events, and appends best-effort `failed` outcome events for light, REM, and deep dreaming failures.

## Best Possible Solution

Land one maintainer-approved additive `completed`/`failed` dreaming outcome contract that keeps old events compatible, uses the provided failed-run proof, and leaves interrupted/partial crash recovery to the separate lifecycle design issue.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 lack `outcome`/`error` fields on `memory.dream.completed`, and the deep failure catch only logs. I did not execute a live failed dreaming run in this read-only review, but the PR body now includes redacted real-run output.

## Solution Assessment

Yes, as a bounded implementation shape pending maintainer acceptance: `completed`/`failed` only is more conservative than the overlapping broader taxonomy in https://github.com/openclaw/openclaw/pull/97705. The safer product path is to approve that narrow contract explicitly and keep interrupted/partial recovery in the separate lifecycle issue.
