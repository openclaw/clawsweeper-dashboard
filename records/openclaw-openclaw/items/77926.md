## Summary

Keep open: current main still rejects the startup plugin metadata snapshot when a reader supplies a different concrete agent workspace, so the follow-up describes a real remaining hot-path miss; the safe fix needs maintainer design around workspace-scoped plugin discovery.

## What This Changes

The issue asks OpenClaw to reuse plugin metadata snapshots on dispatch/read paths when the active agent workspace differs from the startup/default workspace.

## Best Possible Solution

Add a workspace-aware metadata reuse path that keeps plugin discovery correctness for per-agent workspaces while avoiding repeated main-thread manifest/stat scans on dispatch and model/catalog hot paths.

## Reproduction Assessment

Yes, source-level: configure a non-default agent whose resolved workspace differs from the default, then call a model/catalog reader with that workspace; the current snapshot lookup returns undefined and the cold metadata scan path is selected. I did not establish a live strace reproduction in this read-only review.

## Solution Assessment

Unclear. A simple default-workspace setter fix is not sufficient for this follow-up because distinct agent workspaces may have distinct workspace plugins; the narrowest maintainable fix likely needs an owner-approved workspace-scoped snapshot strategy.
