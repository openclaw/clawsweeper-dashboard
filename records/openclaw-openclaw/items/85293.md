## Summary

This PR should stay open, but not merge yet: it targets a plausible Codex app-server timeout cleanup problem, lacks external real-behavior proof, and the new request timeout signal still races the outer timeout wrapper so isolated cleanup may run after the caller has already resumed.

## What This Changes

The PR adds AbortSignal plumbing and focused tests for Codex app-server request timeouts and isolated-client initialization cleanup.

## Best Possible Solution

Coordinate with the broader Codex app-server timeout recovery work, then land a single lifecycle-safe fix that aborts timed-out requests, awaits isolated cleanup before returning, preserves expected timeout errors, and includes redacted real runtime proof.

## Reproduction Assessment

No high-confidence live reproduction is provided. Source inspection supports the concern because the request wrapper uses a Promise.race timeout around an inner cleanup path, but the PR only proves behavior with mocks.

## Solution Assessment

No. Passing an AbortSignal is a reasonable part of the fix, but the proposed request wrapper still lets the outer timeout race cleanup, so the process-lifecycle guarantee should be tightened before merge.

- **[P2] Await isolated cleanup before returning timeout:** `extensions/codex/src/app-server/request.ts:63-68`
