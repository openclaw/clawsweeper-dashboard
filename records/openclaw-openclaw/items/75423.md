## Summary

Keep this PR open. The stale health-cache bug is real and the branch moves in the right direction, but the current comparator can mark fresh multi-account health snapshots stale on every request when the channel top-level summary represents a preferred account while `runtime.channels` represents the default account.

## What This Changes

The PR updates gateway health cache invalidation and tests, adds a changelog entry, and includes adjacent fixes for agent fast-mode/model metadata/SSE transport, guarded extension fetches, and CLI plugin runtime dependency startup.

## Best Possible Solution

Keep the runtime-snapshot freshness strategy, but compare each runtime lifecycle snapshot against the cached account with the same `accountId` so fresh multi-account health summaries still use the cache TTL. After that narrow repair, rerun exact-head changed-surface checks and let the automerge path proceed if clean.

## Reproduction Assessment

Yes. A source-level reproduction is a channel with a default account and a different preferred/bound account whose lifecycle differs: `getHealthSnapshot` stores the preferred account at `channels[channelId]`, while `getRuntimeSnapshot().channels[channelId]` is the default account, causing the new comparator to treat the cache as stale every time.

## Solution Assessment

No, not as-is. The proposed cache-refresh direction is maintainable, but the comparator needs to select cached account summaries by `runtimeSnapshot.accountId` or otherwise avoid comparing the default runtime account to a preferred top-level health summary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Compare runtime channels to their cached account:** `src/gateway/server-methods/health.ts:50-52`
  - body: `cached.channels[channelId]` is not always the same account as `runtime.channels[channelId]`: health summaries put the preferred/bound account at the channel top level, while runtime snapshots put the default account there. In a multi-account setup where those accounts have different lifecycle values, this makes every fresh cache entry look stale and bypasses the TTL on every health RPC. Select `cachedChannel.accounts[runtimeSnapshot.accountId]` before falling back to the top-level summary, or rely on the per-account loop for account-aware comparisons.
  - confidence: 0.91
