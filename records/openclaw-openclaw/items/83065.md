## Summary

Keep open: current main still surfaces managed inbound attachments as `media://inbound/...` prompt notes, but the actual `read` execution path delegates that string to the upstream read tool without media-reference normalization. The linked fixing PR was closed unmerged after review found it patched the wrong layer.

## What This Changes

The issue asks OpenClaw to make Signal inbound `media://inbound/<file>` attachment handles resolvable by a built-in mechanism such as `read`, a resolver tool, or channel runtime context.

## Best Possible Solution

Resolve supported `media://inbound/...` references inside the actual OpenClaw read execution path with the existing inbound-media policy, then document the supported handle for Signal/media attachments.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Signal media can become a `media://inbound/...` prompt note, and current `read` passes that string to an upstream cwd-relative resolver without OpenClaw media-reference normalization.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing media-reference resolver and inbound-media access policy in the real read execution path; the closed guard-only PR is not the right layer.
