## Summary

Keep open: current main still generates `media://inbound/...` prompt references for managed inbound media, but the ordinary structured `read` execution path does not resolve those references before handing the path to the base read tool. The explicitly linked fix PR was closed unmerged after review found it patched the wrong layer, and the adjacent open sandbox-media PR addresses image/PDF sandbox resolution rather than this generic read-tool gap.

## What This Changes

The issue requests a built-in way for agents to resolve Signal inbound `media://inbound/<file>` prompt references without skill-specific path rewriting.

## Best Possible Solution

Resolve supported `media://inbound/...` handles inside the actual OpenClaw read execution path while preserving managed inbound-media containment, then document the supported Signal/media attachment handle.

## Reproduction Assessment

Yes. Source inspection shows Signal can populate media attachment paths, prompt notes rewrite managed inbound media to `media://inbound/...`, and the current structured read path has no media-reference resolver before base read execution.

## Solution Assessment

Yes. The narrowest maintainable repair is to reuse the existing media-reference/local-media policy in the actual `read` path; adding a Signal-only env var or shell-wide URI rewriting would be broader than the reported failure.
