## Summary

Keep open: current main still has a source-backed gap where managed inbound media can be shown to the agent as `media://inbound/...`, but the built-in `read` path passes that string to the upstream filesystem resolver without using OpenClaw's media-reference normalization.

## What This Changes

The issue requests a documented built-in way for Signal inbound attachments surfaced as `media://inbound/<file>` to be accessible to agents without per-skill prompt workarounds.

## Best Possible Solution

Reuse the existing media-reference resolver in the structured `read` path, add focused regression coverage, and document the supported Signal/media attachment handle while leaving shell command rewriting for a separate explicit decision.

## Reproduction Assessment

Yes, at source level: current main can emit `media://inbound` prompt notes, and the current `read` path sends that string to the upstream cwd-relative resolver. I did not run a live Signal attachment flow in this read-only review.

## Solution Assessment

Yes for the narrow fix: wiring `resolveMediaReferenceLocalPath()` into `read` is smaller and better-owned than adding a Signal-only env var or rewriting arbitrary shell commands.
