## Summary

Keep open: current main still has the reported source-level gap. Signal/media-store attachments can be surfaced as `media://inbound/...`, but the structured `read` path does not resolve those refs, and the only PR that referenced this issue was closed unmerged after review found it patched the wrong layer.

## What This Changes

The issue asks for Signal inbound media refs exposed to agents as `media://inbound/<file>` to be resolvable by a built-in mechanism without per-skill workarounds.

## Best Possible Solution

Resolve supported `media://inbound/...` refs inside the actual OpenClaw read execution path with the existing media-reference policy, add focused regression coverage and docs, and leave shell command URI rewriting for a separate explicit decision.

## Reproduction Assessment

Yes. Source inspection shows Signal can populate media paths, the prompt note converts managed inbound media to `media://inbound/...`, and the read tool does not resolve that scheme before calling the base read implementation.

## Solution Assessment

Yes for a narrow repair. The maintainable path is to reuse the existing media-reference resolver and media access policy in the actual read path; adding a Signal-only env var or rewriting arbitrary shell commands would be broader than needed for this bug.
