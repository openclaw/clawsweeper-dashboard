## Summary

Keep open: current main still surfaces managed inbound media as `media://inbound/...` prompt refs, but the built-in `read` path sends that string through the upstream cwd-relative resolver without OpenClaw media-reference normalization. The linked fix PR was closed unmerged after review found it patched the wrong layer, so the original bug still needs a focused replacement fix.

## What This Changes

Signal inbound attachments should be accessible from built-in agent tooling when surfaced as `media://inbound/<file>` prompt references.

## Best Possible Solution

Add policy-aware `media://inbound/...` support to the actual OpenClaw read flow, preserve the managed inbound-media allowance for workspace-only/sandboxed reads, and document the supported Signal/media handle without adding a Signal-only config knob.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Signal can put managed media into prompt notes as `media://inbound/...`, and current `read` delegates that string to the upstream cwd-relative resolver, producing the reported path shape. I did not run a live Signal attachment flow.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing inbound media resolver and media access policy in the read execution path; the closed guard-only PR is not the best fix because default reads bypass that guard and workspace-only reads need an explicit media-store allowance.
