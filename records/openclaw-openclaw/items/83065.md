## Summary

Keep open: current main still emits managed inbound media handles and has a resolver, but the OpenClaw read wrapper and upstream read tool still treat `media://inbound/...` as a cwd-relative filesystem path. The only linked fix PR was closed unmerged after review found it touched the wrong layer.

## What This Changes

The issue requests a supported way for agents to resolve Signal inbound `media://inbound/<file>` attachment handles through built-in tooling or channel context.

## Best Possible Solution

Resolve supported `media://inbound/<id>` references in the actual read execution path using the existing media-reference/local-media policy, add focused read regressions, and document the supported attachment handle.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Signal stores fetched attachments as managed inbound media, prompt notes expose them as `media://inbound/...`, and the current read wrapper sends that string to a cwd-relative upstream resolver.

## Solution Assessment

Yes for a narrow read-tool repair. Reusing the existing media-reference resolver/policy in `createOpenClawReadTool` is more maintainable than adding a Signal-only environment variable or rewriting arbitrary shell command strings.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83065. Current main can prompt Signal/media attachments as `media://inbound/<id>`, but `createOpenClawReadTool` passes `read.path` to upstream `@earendil-works/pi-coding-agent` without resolving the media reference, so upstream treats it as a cwd-relative path like `media:/inbound/...`. Implement a narrow structured-read fix: resolve recognized `media://inbound/<id>` references with the existing media-reference/local-media policy before invoking the base read path, and preserve managed inbound-media allowance for workspace-only reads. Add focused regression coverage for default host read and workspace-only/sandboxed read behavior as applicable, and update the relevant Signal or shared media docs with the supported mechanism. Do not rewrite arbitrary `exec` shell strings, add config flags, support unsupported media hosts or nested IDs, or weaken sandbox/path validation. Treat https://github.com/openclaw/openclaw/pull/83092 as context only; it was closed because it resolved in the workspace guard instead of the actual read layer.
