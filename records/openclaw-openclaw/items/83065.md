## Summary

Keep open: current main can render Signal/media-store attachments as `media://inbound/...` prompt notes, but the built-in `read` path still sends that string through the upstream cwd-relative resolver without OpenClaw media-reference normalization, matching the reported `media:/inbound` failure shape.

## What This Changes

The issue requests a built-in, documented way for agents handling Signal inbound attachments to resolve `media://inbound/<file>` references without per-skill path-convention prompts.

## Best Possible Solution

Wire the existing media-reference resolver into the OpenClaw `read` path and document that supported handle for Signal/media attachments, while leaving arbitrary shell URI rewriting for a separate explicit decision.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Signal media can become a `media://inbound` prompt note, and the current read tool resolves that string as a cwd-relative path. I did not run a live Signal attachment flow in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to reuse `src/media/media-reference.ts` in the structured `read` path and add focused tests/docs. Adding a new Signal-only env var or rewriting arbitrary shell commands would be broader than needed for the reported failure.
