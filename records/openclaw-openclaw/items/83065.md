The issue requests a built-in way for agents to access Signal inbound media surfaced as `media://inbound/<file>` without per-skill path conventions.

## Best Possible Solution

Resolve supported `media://inbound/...` references in the OpenClaw read execution path using the existing media resolver and managed-media policy, with focused regression tests and Signal/media docs.

Yes. Source inspection gives a high-confidence path: Signal can provide managed inbound media paths, prompt rendering turns those into `media://inbound/...`, and the current read tool passes that string to a cwd-relative resolver.

## Solution Assessment

Yes for the issue boundary: the narrow maintainable fix is to wire the existing media-reference resolver and managed-media allowance into the structured read path. The closed guard-only PR is not the best solution because default reads bypass that layer.

Fix https://github.com/openclaw/openclaw/issues/83065. Signal-managed inbound media can be rendered to agents as `media://inbound/<id>`, but the built-in `read` tool currently sends that string through the upstream cwd-relative path resolver and fails. Treat https://github.com/openclaw/openclaw/pull/83092 as a useful but rejected wrong-layer attempt: do not fix this only inside `wrapToolWorkspaceRootGuardWithOptions`. Resolve supported `media://inbound/...` references in the actual OpenClaw read execution path, reuse `src/media/media-reference.ts` and existing managed-media access policy, preserve default and workspace-only behavior for non-media paths, and do not rewrite arbitrary `exec` shell strings. Add focused regression coverage for default host read and workspace-only read where feasible, update Signal/media documentation for the supported handle, and add a changelog entry because this fixes user-visible attachment access.
