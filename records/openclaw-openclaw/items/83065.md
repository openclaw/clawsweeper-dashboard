## Summary

Keep open: current main and the latest release still surface managed inbound media as `media://inbound/...` prompt notes, but the structured `read` tool does not resolve those handles before passing them to the upstream read path. The only PR that referenced this issue was closed unmerged, and the adjacent open sandbox-media PR targets a different tool path rather than the default read-tool gap.

## What This Changes

The issue asks for Signal inbound attachments surfaced as `media://inbound/<file>` to be accessible through a documented built-in mechanism such as `read`, a resolver tool, or a channel-provided media path handle.

## Best Possible Solution

Wire the existing inbound-only media-reference resolver into the actual structured `read` execution path, preserve the managed inbound-media access boundary, and document the supported Signal/media attachment handle; leave arbitrary shell URI rewriting for a separate explicit decision.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Signal can place media paths in context, prompt-note code renders managed media as `media://inbound/...`, and the current read tool passes that string through without media-reference resolution. I did not run a live Signal attachment flow.

## Solution Assessment

Yes, the narrowest maintainable fix is to reuse the existing inbound media resolver in the structured read path with policy-aware tests. A new Signal-only env var or generic shell URI rewriting would be broader than needed for the reported failure.
