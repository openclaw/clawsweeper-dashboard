## Summary

Keep open: the PR targets a valid diagnostics startup-noise bug, but the current patch skips liveness sampling during grace, which also skips monitor reset and baseline refresh; it also lacks the required changelog entry for a user-visible WARN-level fix.

## What This Changes

The PR adds an internal diagnostic heartbeat startup-grace option, wires Gateway startup to pass 60 seconds, and adds a regression test for suppressed liveness warnings during that window.

## Best Possible Solution

Keep the 60-second Gateway startup-grace behavior, but continue sampling and resetting liveness metrics during grace while suppressing only liveness event and WARN emission, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main samples diagnostic liveness immediately while channel health checks have a separate 60-second startup grace, and the PR defect is source-reproducible because the sampler owns event-loop-delay reset and baseline refresh.

## Solution Assessment

No. The PR is aimed at the right source path, but skipping the sampler during grace is not the narrowest maintainable fix; gate emitted liveness reports while preserving sampler reset semantics.
