## Summary

Keep open. The PR targets a valid current-main diagnostics gap, but the patch skips the liveness sampler during the grace window, which also skips the sampler reset and can carry a startup delay spike into the first post-grace warning; it also needs the required changelog entry.

## What This Changes

This PR adds a `startupGraceMs` option to diagnostic heartbeat startup, wires the gateway to pass 60 seconds, and adds diagnostic liveness grace-window coverage.

## Best Possible Solution

Keep the 60s gateway startup grace direction, but sample and reset liveness metrics during grace while suppressing only liveness event/warning emission, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes from source inspection. Current main has no diagnostic liveness startup grace, and the PR’s proposed implementation is source-reproducible as a reset carryover bug because `sampleDiagnosticLiveness` owns the event-loop-delay reset.

## Solution Assessment

No. The PR is solving the right problem, but the safer implementation is to keep sampling/resetting during grace and suppress only the report emission until grace expires.
