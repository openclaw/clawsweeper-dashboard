## Summary

Keep open. The PR targets a valid source-reproducible diagnostics warning problem, but the patch suppresses liveness sampling during grace rather than only suppressing report emission, so startup-delay metrics can carry into the first post-grace sample; it also misses the required changelog entry.

## What This Changes

The PR adds a diagnostic heartbeat `startupGraceMs` option, wires gateway startup to pass 60 seconds, and adds a regression test for the linked startup liveness warning report.

## Best Possible Solution

Keep the 60s diagnostic startup grace, but continue sampling/resetting liveness metrics during grace and suppress only liveness event/warning emission, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main samples diagnostic liveness immediately while channel health checks have their own 60s grace, and active work can turn a startup event-loop delay into a WARN-level liveness message. I did not run a live cold-start replay under this read-only review.

## Solution Assessment

No. The PR is aimed at the right source path, but skipping the sampler during grace is not the narrowest maintainable fix; the safer repair is to keep sampler reset semantics and gate only report emission.
