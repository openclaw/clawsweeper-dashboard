## Summary

Keep open. Current main still lacks diagnostic liveness startup grace, and this PR targets the right bug, but the patch skips the sampler during grace and can carry startup event-loop-delay metrics into the first post-grace warning; it also lacks the required changelog entry.

## What This Changes

The PR adds an internal diagnostics heartbeat `startupGraceMs` option, wires gateway startup to pass 60 seconds, and adds a regression test for suppressed liveness warnings during that window.

## Best Possible Solution

Keep the 60s gateway startup-grace behavior, but continue sampling and resetting liveness metrics during grace while suppressing only liveness event/warning emission, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main samples diagnostic liveness immediately while channel health checks have a separate 60s startup grace, and the PR defect is source-reproducible because `sampleDiagnosticLiveness` owns event-loop-delay reset and baseline refresh.

## Solution Assessment

No. The PR is aimed at the right source path, but skipping `sampleLiveness` during grace is not the safest fix; the narrower maintainable solution is to keep sampler reset semantics and gate only report emission.
