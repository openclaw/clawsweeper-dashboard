## Summary

Keep open. Current main has transcript-backed token/cost summaries and auth-profile selection/rotation, but completed call usage is still parsed and aggregated by provider/model/session rather than by the resolved auth profile; the burn0/#55379 discussion remains a useful workaround, not a native replacement for OpenClaw auth-profile attribution.

## What This Changes

Request per-call model cost visibility with attribution to the resolved auth profile, including fallback/profile-rotation events or session_status breakdowns.

## Best Possible Solution

Record safe resolved authProfileId metadata when each model call completes, propagate it through transcript usage/status diagnostics, aggregate costs by profile, and emit a clear profile-rotation/fallback event without exposing secrets.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source inspection: current usage parsing, Gateway aggregates, status output, and docs expose provider/model/session cost data but not resolved auth-profile attribution.

## Solution Assessment

Unclear for the exact UI/API shape, but native runtime/transcript metadata is the right product boundary. A network-layer workaround like burn0/#55379 can help with request cost, but it cannot reliably attribute OpenClaw's selected auth profile.
