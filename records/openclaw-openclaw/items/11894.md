## Summary

Keep open. Current main has transcript-backed usage/cost summaries and auth-profile rotation/fallback routing, but completed model calls are still parsed and aggregated by provider/model/session surfaces rather than by the resolved auth profile that paid for the call. The burn0 discussion is useful as an interim cost workaround, but it does not replace native OpenClaw auth-profile attribution.

## What This Changes

Request native per-call cost tracking and resolved auth-profile attribution, especially when auth profile fallback switches from a free-tier profile to a pay-as-you-go profile.

## Best Possible Solution

Record the resolved authProfileId at model-call completion as safe metadata, carry it through transcript/usage/status or diagnostics APIs, add cost aggregation by auth profile, and emit a clear profile rotation/fallback event for user-visible fallback cases.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source inspection: current transcript parsing, usage aggregation, gateway usage types, and status output expose cost by provider/model/session but not by resolved auth profile.

## Solution Assessment

Unclear for the exact UI/API shape, but the native direction is the right product boundary. A network-layer workaround like burn0 can help with per-request cost, but the maintainable OpenClaw fix needs resolved auth-profile metadata from the model runtime itself.
