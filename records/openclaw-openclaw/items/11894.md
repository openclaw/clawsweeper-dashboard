## Summary

Current main provides per-call token/cost accounting and auth-profile rotation, but it still does not attribute completed calls or cost aggregates to the resolved auth profile. The burn0 discussion is a useful interim cost workaround, while the native profile-attribution and safe fallback-notice contract remain real core work requiring maintainer product and security judgment.

## What This Changes

Add per-call cost visibility, resolved auth-profile attribution, per-profile cost aggregation, and a visible signal when auth-profile fallback occurs.

## Best Possible Solution

Capture a safe stable profile identity from the resolved runtime auth state for each completed model call, persist it in the canonical usage record, add per-profile cost aggregates, and expose a distinct profile-rotation/fallback signal without revealing secret material or inferring attribution from mutable session state.

## Reproduction Assessment

Yes at the source level: current main knows the resolved profile during auth selection, while its completed-call diagnostic, transcript usage, gateway aggregation, and status contracts omit that identity. This is a feature gap rather than a failing existing contract requiring a live repro.

## Solution Assessment

The requested outcome is valid, but the best implementation is first-class safe metadata propagated from the runtime auth decision, not a network interceptor or later inference from the session's current profile. Maintainers still need to decide the public identifier and fallback-notice policy.

## Risks / Open Questions

- A fallback event needs clear semantics for profile rotation versus model/provider failover; conflating them could produce misleading billing attribution or noisy user notices.
- Transcript and diagnostic retention/access differ, so maintainers must choose which surfaces may persist or expose the attribution before implementation.
