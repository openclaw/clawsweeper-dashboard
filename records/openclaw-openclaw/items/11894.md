## Summary

Keep open: current main has transcript-backed usage/cost summaries, live per-turn usageState, provider-usage auth context, and auth-profile fallback routing, but completed model-call accounting still does not persist or aggregate the resolved auth profile that paid for each call. The burn0 workaround and adjacent cost-tracking issues are useful context, not a native replacement for OpenClaw auth-profile attribution.

## What This Changes

The issue requests native per-call cost tracking with attribution to the auth profile actually used for each model call, especially when fallback credentials are selected.

## Best Possible Solution

Add one additive completed-call attribution contract that records a safe resolved authProfileId and feeds transcript, status, gateway/API, UI, diagnostics, and plugin views without credential material.

## Reproduction Assessment

Yes, at source level: current main keeps completed-call usage/cost data and resolved auth-profile routing as separate facts, and the accounting contracts do not join them. No live provider fallback run was needed for this read-only feature-gap review.

## Solution Assessment

No current implementation fully solves it. The maintainable path is one canonical completed-call attribution contract before deriving transcript, status, gateway/API, UI, hook, or event views.

## Risks / Open Questions

- Auth-profile identifiers are credential-adjacent metadata, so persistence and exposure need explicit redaction, authorization, retention, and diagnostic-export rules.
- Fallback retries, cooldown probes, user-pinned profiles, provider aliases, plugin-owned transports, and CLI harnesses can each select credentials differently; a single fallback flag would be too imprecise for billing attribution.
- Adding profile attribution only to a hook, status line, or ad hoc transcript field would leave durable history, gateway usage, UI, diagnostics, and plugin consumers with competing usage sources.
