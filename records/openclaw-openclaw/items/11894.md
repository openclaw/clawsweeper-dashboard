## Summary

Keep open: current main still has cost/usage summaries, per-turn usage footer state, and auth-profile fallback routing, but completed-call accounting still does not persist or aggregate the resolved auth profile that paid for each call. The burn0 workaround and related cost-tracking issues are useful context, not a native replacement for OpenClaw auth-profile attribution.

## What This Changes

Request native per-call cost tracking that attributes each completed provider call to the resolved auth profile, especially after fallback.

## Best Possible Solution

Add one additive completed-call attribution contract that records a safe resolved authProfileId and feeds transcript, status, gateway/API, UI, diagnostics, and plugin views without credential material.

## Reproduction Assessment

Yes. At source level, current main keeps completed-call usage/cost data and resolved auth-profile routing as separate facts, and the accounting contracts do not join them; no live provider fallback run was needed for this feature-gap review.

## Solution Assessment

No current implementation fully solves it. The maintainable path is one canonical completed-call attribution contract before deriving transcript, status, gateway/API, UI, hook, or event views; burn0 remains only an interim external cost workaround.

## Risks / Open Questions

- Auth-profile identifiers are credential-adjacent metadata, so persistence and exposure need explicit redaction, authorization, retention, and diagnostic-export rules.
- Fallback retries, cooldown probes, user-pinned profiles, provider aliases, plugin-owned transports, and CLI harnesses can each select credentials differently; a single fallback flag would not be enough for billing attribution.
- Adding profile attribution only to a hook, status line, or ad hoc transcript field would leave durable history, gateway usage, UI, diagnostics, and plugin consumers with competing usage sources.
