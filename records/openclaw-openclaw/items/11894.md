## Summary

Current main and v2026.6.5 provide transcript-backed per-call cost data and auth-profile rotation, but completed-call usage still does not retain or aggregate the resolved profile that paid for each call. The burn0 workaround and https://github.com/openclaw/openclaw/issues/13219 help with cost reporting, while open https://github.com/openclaw/openclaw/pull/89629 proposes per-turn usage metadata, but none provides native resolved-profile attribution; this issue remains necessary pending a maintainer decision on safe identifier exposure and event semantics.

## What This Changes

Requests per-model-call cost reporting tied to the resolved authentication profile, including fallback visibility and per-profile aggregation.

## Best Possible Solution

Record a safe resolved-profile identifier and typed selection reason with each completed model call, propagate them through the canonical transcript and usage contracts, aggregate costs by profile, and expose only authorized, redacted identifiers without credential material.

## Reproduction Assessment

Yes at source level: current main has per-call transcript usage/cost and separate auth-profile routing, but completed-call usage types and aggregates do not join those facts. This is a feature gap rather than a failure of an existing documented contract.

## Solution Assessment

The requested outcome is appropriate, but the three suggested presentation options do not settle the safest implementation. The narrowest maintainable path is to establish one canonical completed-call attribution contract first, then derive transcript, status, API, and event views from it.

## Risks / Open Questions

- Auth-profile identifiers are credential-adjacent metadata; persistence and exposure need explicit redaction, authorization, retention, and diagnostic-export rules.
- Fallback, overload rotation, rate-limit rotation, retry, and explicit profile override are distinct events; collapsing them into one fallback flag could produce misleading billing attribution.
- Adding this only to a live hook would not satisfy durable transcript, historical usage, or session-status aggregation and could create competing observability sources.
