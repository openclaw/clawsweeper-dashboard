## Summary

Keep open. Current main has transcript-backed token/cost summaries and auth-profile routing, but completed-call usage is still parsed and aggregated by provider/model/session rather than by the resolved auth profile that paid for each call; the burn0 thread is useful as an external workaround, not a native attribution fix.

## What This Changes

Request native per-call cost visibility, resolved auth-profile attribution, per-profile cost summaries, and a fallback/profile-use event.

## Best Possible Solution

Record safe resolved authProfileId metadata for completed model calls, aggregate cost by auth profile, and expose additive transcript, Gateway, status/UI, and fallback-event signals without storing credentials or secret material.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The high-confidence verification path is source inspection: current main exposes provider/model/session usage and auth routing, but not completed-call resolved auth-profile attribution.

## Solution Assessment

Unclear for the exact API and UI shape, but native metadata at the normalized model-call usage seam is the maintainable boundary. Network-layer burn0-style tracking can help with per-request cost, but it cannot reliably attribute OpenClaw's selected auth profile.
