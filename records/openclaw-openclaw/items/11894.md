- `impact:security`: The issue asks OpenClaw to persist or expose credential-adjacent billing/profile attribution metadata.

## Summary

Keep open: current main and v2026.6.8 still keep completed-call usage/cost data, fallback routing, and auth-profile selection as separate facts, so OpenClaw cannot attribute a paid call or cost bucket to the resolved auth profile that was actually used.

## What This Changes

Request native per-call cost visibility with resolved auth-profile attribution for fallback/profile-rotation scenarios.

## Best Possible Solution

Define one additive completed-call attribution contract that records a safe resolved authProfileId and feeds transcript, status, gateway/API, UI, diagnostics, and plugin views without credential material.

## Reproduction Assessment

Yes. At source level, current main and v2026.6.8 keep completed-call usage/cost data and resolved auth-profile routing as separate facts, and the accounting contracts do not join them; no live provider fallback run was needed for this feature-gap review.

## Solution Assessment

No current implementation fully solves it. The maintainable path is one canonical completed-call attribution contract before deriving transcript, status, gateway/API, UI, hook, or event views; burn0 remains only an interim external cost workaround.
