## Label Justifications

- `P2`: The missing attribution limits billing and fallback observability but does not prevent current provider calls or sessions from operating.
- `impact:security`: Persisting and displaying auth-profile identity involves credential-adjacent metadata and requires a deliberate exposure boundary.
- `impact:auth-provider`: The issue asks which provider auth profile was selected for each completed model call.

## Summary

Current main provides per-call token and cost data and carries the selected auth profile through model execution, but completed-call usage parsing and gateway aggregates still discard that profile identity. The burn0 workaround is useful for request-level cost visibility but cannot supply native OpenClaw auth-profile attribution, and the broader related issues do not replace this narrower remaining requirement.

## What This Changes

Add native per-call cost reporting, resolved auth-profile attribution, per-profile aggregation, and visible fallback or rotation events.

## Best Possible Solution

Persist one sanitized resolved-profile identifier with canonical completed model-call usage, enforce appropriate session and local access boundaries, and derive per-profile cost aggregates and fallback notices from that single record.

## Reproduction Assessment

Yes for the capability gap: current-main source carries `authProfileId` during execution but omits it from completed transcript usage, session summaries, and gateway usage aggregates.

## Solution Assessment

The requested outcome is valid, but the best implementation is a single security-reviewed completed-call attribution field with derived views rather than separate transcript, status, and fallback-event implementations.
