## Summary

This PR remains necessary because current main still ends the active Windows Scheduled Task without suppressing its repeating trigger. The update-scoped design is coherent, but this maintainer-authored draft is currently conflicted with newer update-lifecycle work and changes persistent autostart state, so it should remain open for rebase and Windows lifecycle review rather than cleanup or automated repair.

## What This Changes

The branch temporarily disables an enabled Windows gateway Scheduled Task during managed package replacement, conditionally restores it across update exits, and adds regression coverage for scheduler state and lifecycle ordering.

## Best Possible Solution

Rebase the update-scoped conditional suspension onto current main, preserve all newer recovery and handoff behavior, restore only tasks suspended by the same update invocation, and validate the final lifecycle with a native Windows package-update run.

## Reproduction Assessment

Yes. Current main ends the running task without disabling its registered PT3M trigger, and the linked report provides a concrete package-update timing path in which Task Scheduler relaunches the gateway during replacement.

## Solution Assessment

Yes in design: update-specific suspension is narrower and more maintainable than changing general service stop/restart semantics. The final rebased implementation still needs lifecycle review and native Windows package-update validation before merge.
