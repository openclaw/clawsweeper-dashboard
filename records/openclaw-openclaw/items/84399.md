## Summary

Keep open: the PR is a useful fix for the hardcoded overflow hint and the follow-up commit addresses agent context-token caps, but the latest patch still computes the dynamic hint from the original primary route even when the run has already fallen back to another provider/model.

## What This Changes

The PR replaces the static `20000` context-overflow `reserveTokensFloor` hint with tiered context-window guidance and adds focused unit tests for tiers, session fallback, and agent context-token caps.

## Best Possible Solution

Compute the hint from the same effective runtime provider/model and capped context budget that produced the overflow, then keep the tiered guidance with regression coverage for primary, capped, and fallback-model cases.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.5.18 hardcode 20000, and the PR head can still miscompute the dynamic value on the fallback path because it stores `fallbackProvider`/`fallbackModel` but passes the original run provider/model into the hint. I did not run tests because this was a read-only checkout review.

## Solution Assessment

No, not yet: the tiered helper plus agent-cap handling is the right direction, but the hint must use the effective runtime/fallback model before it is safe to merge.
