## Summary

Keep open. This is a focused fix for https://github.com/openclaw/openclaw/issues/65839, but the patch is not ready because the new hint can over-recommend when an agent’s effective context cap is lower than the model’s native context window.

## What This Changes

The PR replaces the static `20000` context-overflow `reserveTokensFloor` hint with tiered values derived from model/session context metadata and adds focused unit tests.

## Best Possible Solution

Use the effective context window for the active agent/session, including configured `contextTokens` caps, before selecting the reserve-floor tier, while preserving the existing heartbeat-bleed hint path.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original bug via the static `20000` hint, and source inspection also reproduces the PR gap because runtime compaction uses an effective `contextTokens` cap that the new helper ignores.

## Solution Assessment

No. The context-aware direction is right, but the helper should resolve the same effective context budget used by runtime compaction instead of preferring native model metadata over a valid agent/session cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Use the effective context cap for the hint tier:** `src/auto-reply/reply/agent-runner-execution.ts:717-723`
  - body: This lookup ignores `agents.defaults.contextTokens` or per-agent `contextTokens` caps and returns the native model window before considering the active session entry. Runtime compaction already applies that cap through `contextTokensOverride`, so a 1M model capped to a smaller effective window can now tell the user to set `reserveTokensFloor` to `100000`, which may exceed the actual budget and recreate the overflow guidance bug. Resolve the effective agent/session context window before choosing the tier, and cover the capped-model case in the tests.
  - confidence: 0.84
