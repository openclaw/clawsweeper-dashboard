## Summary

Keep open: current main still routes ACP parent completion through queued system events plus heartbeat wake, and that path can still skip when heartbeat is disabled, no wake handler is registered, or no heartbeat agents are scheduled. The open same-author PR at https://github.com/openclaw/openclaw/pull/60402 remains the narrow implementation candidate, while later comments broaden the desired endpoint to shared yielded-parent delivery.

## What This Changes

Ensure ACP child completion wakes a yielded parent session without manual refresh, especially when heartbeat delivery is disabled.

## Best Possible Solution

Land or replace the open PR at https://github.com/openclaw/openclaw/pull/60402 for the narrow ACP heartbeat-disabled gap, then converge ACP and native child completion on a maintained yielded-parent delivery primitive with focused regression coverage.

## Reproduction Assessment

Yes, source-level: ACP completion enqueues a parent-session event and schedules a heartbeat wake, while the wake layer can return with no handler and the runner can skip before targeted handling when heartbeat is disabled. I did not run the live browser or Matrix flow in this read-only review.

## Solution Assessment

No for current main: heartbeat wake is not a self-sufficient delivery path when heartbeat is disabled or unavailable. The safer endpoint is a dedicated yielded-parent completion delivery path, with the open PR covering the narrow ACP heartbeat-disabled gap first.
