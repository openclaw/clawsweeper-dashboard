## Label Justifications

- `P1`: Yielded parent sessions can remain unresponsive after child completion, breaking active agent/channel workflows until manual intervention.
- `impact:message-loss`: Completion messages can be queued or delayed without producing the expected visible parent response.
- `impact:session-state`: The parent session can remain yielded/running after the child has completed, creating requester state drift.

## Summary

Keep open: current main still has a source-level yielded-parent completion gap. The original direct-resume explanation is stale, but ACP parent streaming still queues session-scoped events behind the heartbeat wake path, and recent comments plus open related PRs show the broader ACP/native resumption behavior is still active.

## What This Changes

The issue asks for yielded parent sessions to wake and resume automatically when ACP or child sessions complete, without manual refresh or a follow-up user message.

## Best Possible Solution

Converge ACP and native child completion onto one maintained yielded-parent resumption contract that does not depend on optional heartbeat scheduling and only treats delivery as complete once the parent turn or visible channel result has actually succeeded.

## Reproduction Assessment

Yes, source-level. ACP `streamTo:"parent"` enqueues parent system events and requests a heartbeat wake, while the current heartbeat path can skip before targeted delivery when heartbeat is disabled, unregistered, or unscheduled; recent comments add matching ACP and native production traces.

## Solution Assessment

No for current main. The heartbeat-coupled relay is not a complete yielded-parent completion contract; the safer direction is a dedicated, maintained wake/resumption path that covers both ACP and native completion without duplicating delivery semantics.
