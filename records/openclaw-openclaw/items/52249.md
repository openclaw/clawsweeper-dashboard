## Label Justifications

- `P1`: Yielded parent sessions can remain unresponsive after child completion, breaking active agent/channel workflows until manual intervention.
- `impact:message-loss`: Completion messages can be queued, delayed, or fail to produce the expected visible parent response.
- `impact:session-state`: The parent session can remain yielded or running after the child has completed, creating requester state drift.

## Summary

Keep open: current main still has a source-level yielded-parent completion gap. The old direct-resume root cause is stale, but ACP completion still depends on heartbeat wake delivery, and recent discussion plus open related PRs show the broader ACP/native resumption behavior is still active.

## What This Changes

A yielded parent session should automatically resume when an ACP or native child completes, without requiring UI refresh or a new user message.

## Best Possible Solution

Converge ACP and native child completion onto one maintained yielded-parent resumption contract that does not depend on optional heartbeat scheduling and only marks delivery complete once the parent turn or visible channel result has actually succeeded.

## Reproduction Assessment

Yes, source-level. ACP `streamTo:"parent"` enqueues parent system events and requests a heartbeat wake, while the current heartbeat path can skip before targeted delivery when heartbeat is disabled, unregistered, or unscheduled; recent comments add matching ACP and native production traces.

## Solution Assessment

No for current main. The heartbeat-coupled relay is not a complete yielded-parent completion contract; the safer direction is a dedicated, maintained wake/resumption path that covers both ACP and native completion without duplicating delivery semantics.
