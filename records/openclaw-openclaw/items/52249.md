## Label Justifications

- `P1`: Yielded parent sessions can remain unresponsive after child completion, breaking active agent/channel workflows until manual user intervention.
- `impact:message-loss`: Completion messages can be queued or delayed without producing the expected visible parent response.
- `impact:session-state`: The parent session can stay in a yielded/running state after the child has completed, causing requester state drift.

## Summary

Keep open: current main still routes ACP parent completion through session-scoped system events plus heartbeat wake, and the heartbeat path can skip delivery when heartbeat is disabled or no targeted heartbeat agent is scheduled. Recent discussion and the open paired PR show this remains an active yielded-parent resumption bug rather than a cleanup-close candidate.

## What This Changes

Fix yielded ACP parent sessions so child completion wakes and resumes the parent without a UI refresh or manual inbound message, including heartbeat-disabled deployments.

## Best Possible Solution

Make yielded-parent completion delivery self-sufficient: ACP child completion should wake the parent without depending on optional heartbeat infrastructure, and the broader ACP/native completion contract should converge on one maintained resumption path.

## Reproduction Assessment

Yes, source-level. ACP streamTo:"parent" enqueues session-scoped system events and requests a heartbeat wake, while current heartbeat code can skip before targeted delivery when heartbeat is disabled or unscheduled; issue comments add matching production traces.

## Solution Assessment

No for current main. The current heartbeat-coupled relay is not a complete fix for yielded-parent completion delivery; the safer direction is a maintained wake/resumption path that works for ACP without optional heartbeat and fits the broader native-subagent contract.
