## Summary

Keep open: current main still sends ACP parent-stream completion through session-scoped system events plus heartbeat wake, and the wake path can still no-op or return disabled before targeted delivery when heartbeat infrastructure is absent or disabled. The open same-author PR at https://github.com/openclaw/openclaw/pull/60402 remains the narrow implementation candidate for the heartbeat-disabled seam, while https://github.com/openclaw/openclaw/pull/56442 covers a broader parent-completion notify contract that needs maintainer review.

## What This Changes

Ensure ACP child completion wakes a yielded parent session without requiring a manual UI refresh, including when heartbeat delivery is disabled or unavailable.

## Best Possible Solution

Land or replace the narrow heartbeat-disabled targeted wake fix, then converge ACP and native yielded-parent completion on a maintained delivery primitive with regression coverage for heartbeat-disabled, heartbeat-enabled, and push-channel flows.

## Reproduction Assessment

Yes, source-level: ACP completion enqueues a session-scoped event and schedules a heartbeat wake, while the wake layer can return without a handler and the runner can return disabled before targeted handling. I did not run the live browser or Matrix flow.

## Solution Assessment

No for current main: heartbeat wake is not a self-sufficient parent-completion delivery path when heartbeat is disabled or unavailable. The maintainable endpoint is a dedicated yielded-parent completion delivery path, with the open narrow PR covering the heartbeat-disabled ACP gap first.
