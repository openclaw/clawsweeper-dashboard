## Summary

Keep open: current main still routes ACP parent-stream completion through queued system events plus heartbeat wake, and the heartbeat wake path can still no-op or return disabled before targeted delivery. The same-author PR at https://github.com/openclaw/openclaw/pull/60402 remains the narrow implementation candidate, while https://github.com/openclaw/openclaw/pull/56442 covers a broader ACP parent-completion notify contract that needs maintainer judgment.

## What This Changes

ACP child completion should automatically wake a yielded parent session without requiring a UI refresh or manual follow-up message, including heartbeat-disabled ACP parent-stream flows.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/60402 for the narrow heartbeat-disabled ACP wake gap, then converge ACP and native child completion on a maintained yielded-parent delivery primitive with regression coverage.

## Reproduction Assessment

Yes, source-level: ACP completion enqueues a parent-session event and requests a heartbeat wake, while the wake layer can no-op without a handler and the runner still returns before targeted handling when heartbeats are disabled. I did not run the live UI or Matrix flow in this read-only review.

## Solution Assessment

No for current main: heartbeat wake is not a self-sufficient delivery path when heartbeat is disabled or unavailable. The narrow paired PR is a reasonable first fix, but the durable endpoint should be a dedicated yielded-parent completion delivery path rather than relying on periodic heartbeat semantics.
