## Label Justifications

- `P1`: Yielded parent sessions can remain unresponsive after child completion, breaking active agent/channel workflows until manual intervention.
- `impact:message-loss`: Completion messages can stay queued, arrive late, or fail to produce the expected visible parent response.
- `impact:session-state`: The parent session can remain yielded, running, or stale after the child has completed, creating requester state drift.

## Summary

Keep open: the old direct-resume explanation is stale, but current main still has a source-backed yielded-parent completion gap because ACP parent completion rides system-event plus heartbeat wake delivery, while disabled or unscheduled heartbeat paths can skip before targeted delivery; newer duplicate reports now point back to this issue as the canonical tracker.

## What This Changes

The issue asks OpenClaw to resume yielded parent sessions automatically when ACP or native child sessions complete, without requiring a manual UI refresh or relying on optional heartbeat scheduling.

## Best Possible Solution

Converge ACP and native child completion onto one maintained yielded-parent resumption contract that does not depend on optional heartbeat scheduling and only marks delivery complete after the parent turn or visible channel result has actually succeeded.

## Reproduction Assessment

Yes, source-level. Current ACP `streamTo:"parent"` enqueues parent-session events and requests a heartbeat wake, while disabled or unregistered heartbeat paths can skip before targeted delivery; recent comments add matching ACP and native production traces.

## Solution Assessment

No for current main. The heartbeat-coupled relay is not a complete yielded-parent completion contract; the safer direction is a dedicated maintained resumption path that covers ACP and native completion without duplicating delivery semantics.
