## Summary

Keep open. Current main no longer matches the original direct-resume root-cause theory, but the narrowed heartbeat-disabled failure still follows from source: ACP completion queues a parent-session system event and requests a heartbeat wake, while the wake layer and runner can no-op when no heartbeat handler or enabled heartbeat agent is available. The same-author PR #60402 is already the narrow implementation candidate, and the latest comments broaden the desired endpoint toward shared yielded-parent completion delivery.

## What This Changes

The issue asks for yielded parent sessions to resume automatically after ACP or child-task completion without depending on optional heartbeat delivery.

## Best Possible Solution

Land or replace the narrow ACP heartbeat-disabled fix with a maintained yielded-parent completion delivery path that works without optional heartbeat scheduling, then cover ACP streamTo:"parent" and native child completion with focused heartbeat-disabled regression tests.

## Reproduction Assessment

Yes for the narrowed ACP path: enqueue an ACP streamTo:"parent" completion with no heartbeat handler or no enabled heartbeat agent, and current main leaves the parent event dependent on a wake path that can skip disabled. The native-child extension in the comments is credible but still needs its own focused regression.

## Solution Assessment

No for current main: the relay has been moved onto system-event plus heartbeat wake, but that is still not self-sufficient when heartbeat delivery is disabled. The maintainable direction is a shared completion wake/delivery primitive rather than another transcript-sniffing or relay-specific direct re-entry path.
