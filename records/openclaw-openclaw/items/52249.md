## Label Justifications

- `P1`: Yielded parent sessions can remain unresponsive after child completion, breaking active agent/channel workflows until manual intervention.
- `impact:message-loss`: Completion messages can remain queued, delayed, or fail to produce the expected visible parent response.
- `impact:session-state`: The parent session can remain yielded, running, or stale after the child has completed, creating requester state drift.

## Summary

Keep open: current main still leaves ACP parent completion coupled to heartbeat wake delivery, and disabled or unscheduled heartbeat paths can skip targeted delivery before the parent drains the completion event. A recent merged native Codex fix improves one adjacent path, but the ACP/yielded-parent contract is still represented by open PRs and needs maintainer direction.

## What This Changes

Yielded parent sessions should resume automatically when ACP or native child completion arrives, instead of staying stuck until refresh or a manual follow-up message.

## Best Possible Solution

Converge ACP and native child completion onto one maintained yielded-parent resumption contract that does not depend on optional heartbeat scheduling and only marks delivery complete after the parent turn or visible channel result succeeds.

## Reproduction Assessment

Yes, source-level. Current ACP streamTo parent delivery enqueues system events and requests a heartbeat wake, while disabled or unscheduled heartbeat paths can return before targeted delivery; recent Matrix, Telegram, ACP, and native-subagent comments add matching production traces.

## Solution Assessment

No for current main. The heartbeat-coupled relay is not a complete yielded-parent completion contract; the safer direction is a maintained resumption path that covers ACP and native completion without duplicating delivery semantics.
