## Label Justifications

- `impact:session-state`: The parent session can remain yielded, running, or stale after the child has completed, creating requester state drift.

## Summary

Keep open: current main no longer matches the original direct-resume root-cause text, but the remaining yielded-parent wake problem is still source-backed because ACP and task terminal events are queued as session system events and still depend on heartbeat wake delivery that can be disabled, unregistered, or skipped before the parent turn runs.

## What This Changes

Ensure yielded parent sessions resume automatically after ACP or native child completion without requiring a manual UI refresh or later user message.

## Best Possible Solution

Converge ACP and native child completion onto one maintained yielded-parent resumption contract that does not depend on optional heartbeat scheduling and only marks delivery complete after the parent turn or visible channel result has actually succeeded.

## Reproduction Assessment

Yes, source-level: current ACP and task terminal completion paths enqueue parent-session events and depend on heartbeat wake delivery, while disabled or unscheduled heartbeat paths can skip before targeted delivery. The issue discussion also contains recent ACP and native production traces matching that source path.

## Solution Assessment

No for current main. The heartbeat-coupled relay and task-event path are not a complete yielded-parent completion contract; the safer direction is a dedicated maintained resumption path that covers ACP and native completion without duplicating delivery semantics.
