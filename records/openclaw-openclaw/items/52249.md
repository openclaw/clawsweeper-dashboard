## Summary

Keep open: current main no longer has the old direct-resume path, but ACP parent stream events still depend on the heartbeat wake machinery, and disabled heartbeat handling can leave queued parent events undrained. The same-author PR #60402 targets this gap but is still open and unmerged.

## What This Changes

The issue asks for ACP child completion to wake a yielded parent session without requiring a UI refresh or heartbeat-enabled polling.

## Best Possible Solution

Review, rebase, and land #60402 or a narrow replacement that lets targeted ACP parent-session wake/delivery drain queued completion events even when periodic heartbeat is disabled, with regression coverage for streamTo parent plus sessions_yield without heartbeat.

## Reproduction Assessment

Yes, at source level: run ACP streamTo parent plus sessions_yield with heartbeat disabled or no heartbeat handler, then ACP completion queues a parent event but the wake path does not drain it. I did not run a live E2E in this read-only review.

## Solution Assessment

No for current main; it still relies on optional heartbeat machinery for ACP parent wake delivery. The narrow maintainable fix is a targeted session wake/delivery path that works without periodic heartbeat, while a larger yield-wait lifecycle can remain a follow-up if needed.
