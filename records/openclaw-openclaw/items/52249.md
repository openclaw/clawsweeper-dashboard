## Summary

Keep open. Current main no longer matches the original direct-resume root-cause theory, but the narrowed heartbeat-disabled failure still follows from source: ACP completion queues parent-session events and asks heartbeat to wake, while the wake path can be a no-op when no handler or enabled heartbeat agent is available. The same-author PR #60402 is already the narrow implementation candidate, and the later native-child comment broadens the desired endpoint beyond an ACP-only scheduler tweak.

## What This Changes

The issue requests that yielded parent sessions automatically resume when ACP or child-session completion arrives, without requiring UI refresh or optional heartbeat delivery.

## Best Possible Solution

Review and either land or replace #60402 for the narrow ACP heartbeat-disabled wake gap, then follow through with a shared yielded-parent completion delivery path that works without optional heartbeat scheduling and covers both ACP streamTo:"parent" and native child completion regressions.

## Reproduction Assessment

Yes for the narrowed ACP path: enqueue an ACP streamTo:"parent" completion while the parent agent has no registered/enabled heartbeat delivery, and current main leaves the event dependent on a wake path that can skip disabled. The native-child report is credible but still needs its own focused current-main regression.

## Solution Assessment

No for current main: moving the relay onto system-event plus heartbeat wake removed the stale direct-resume path but still couples completion delivery to optional heartbeat infrastructure. The safer long-term solution is a dedicated yielded-parent completion wake/delivery primitive rather than transcript inspection or another relay-specific re-entry path.
