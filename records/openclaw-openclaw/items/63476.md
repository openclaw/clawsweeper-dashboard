## Summary

Keep open. Current main still has a source-level path where normal agent sessions pass `workspaceDir` into the image tool, and `media://inbound/...` can be rewritten into a malformed workspace path before the shared media loader sees the canonical managed-media URI; the open linked PR remains the right implementation candidate to review or supersede.

## What This Changes

The issue asks OpenClaw to preserve inbound image data through the agent image-tool pipeline so configured vision models receive the image instead of reporting that no image was uploaded.

## Best Possible Solution

Land #63497 or an equivalent narrow patch that preserves canonical `media://inbound` references through image-tool workspace normalization, lets `loadWebMedia` or the shared media-reference helper hydrate them, and adds regression coverage for `createOpenClawTools`/`createImageTool` with `workspaceDir` before closing with fix provenance.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level: create the normal agent image tool with `workspaceDir`, pass `media://inbound/<id>`, and the current branch resolves it to a malformed workspace `media:/inbound` path before provider/image analysis can run.

## Solution Assessment

Yes for the current triage direction. Keeping this issue open while reviewing or superseding #63497 is the narrowest maintainable path; broad Telegram, provider, or gateway changes are not needed for the observed URI-preservation bug.
