## Summary

Keep open. Current main still has a source-level path where the normal agent image tool receives `workspaceDir` and can rewrite `media://inbound/...` into a malformed workspace path before managed media hydration runs, and an open closing PR already targets this issue.

## What This Changes

The issue reports that agent-session image-tool calls lose inbound image data, causing the vision model to say no image was received instead of describing the uploaded Telegram image.

## Best Possible Solution

Land a rebased or consolidated fix that treats `media://inbound` as managed media before workspace or sandbox path normalization, covers the agent-session `workspaceDir` path, and then close this issue with the related reports.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create the normal image tool with `workspaceDir` and pass `media://inbound/<id>`; current main can resolve it into a workspace `media:/inbound/<id>` path before managed-media hydration can run.

## Solution Assessment

Yes for triage. Keeping this issue open while maintainers review or consolidate the open fixing PRs is the narrowest maintainable path; broad Telegram or provider changes are not needed for the observed URI-preservation bug.
