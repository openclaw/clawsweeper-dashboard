## Summary

Keep open. Current main still has a source-level path where `media://inbound/...` can be rewritten into a malformed workspace path before managed-media hydration, and an open closing PR already targets this issue.

## What This Changes

The issue reports that inbound image data is lost when the agent-session image tool receives Telegram media, causing the vision model to answer as if no image was provided.

## Best Possible Solution

Land a rebased or consolidated fix that preserves managed `media://inbound` references until the shared media resolver handles them, covers the agent-session `workspaceDir` path, and then close this issue with the related malformed-path report.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create the normal image tool with `workspaceDir` and pass `media://inbound/<id>`; current main can route that value into `resolve(options.workspaceDir, normalizedRef)` before managed-media hydration runs.

## Solution Assessment

Yes for triage. Keeping this issue open while maintainers review or consolidate the open fixing PR is the narrowest maintainable path; the fix belongs in media URI preservation/resolution, not in Telegram or provider-specific code.
