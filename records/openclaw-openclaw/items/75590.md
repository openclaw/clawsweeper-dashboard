## Summary

Keep open. The PR is a useful narrow fix for the Cancel/Delete dismissal leak, but it is not merge-ready because the same-button toggle dismissal still removes the popover without unregistering the document click listener.

## What This Changes

The PR updates `ui/src/ui/chat/grouped-render.ts` so the delete confirmation popover uses a shared `dismissPopover` helper for Cancel, Delete, and outside-click dismissal.

## Best Possible Solution

Land one canonical narrow patch that routes every delete-confirm dismissal path through listener cleanup, including Cancel, Delete, outside-click, and clicking the delete button again while the popover is already open. Add a focused regression test that proves `document.addEventListener` and `document.removeEventListener` stay balanced.

## Reproduction Assessment

Yes. Current main reproduces from source by opening a delete confirmation, waiting for the requestAnimationFrame listener registration, then dismissing with Cancel/Delete; the PR still reproduces by clicking the same delete button again to close the existing popover.

## Solution Assessment

No, not as written. The shared helper is the right direction, but it is scoped only to the newly-created popover branch and cannot clean up the existing-popover toggle branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clean up the toggle dismissal path:** `ui/src/ui/chat/grouped-render.ts:656-659`
  - body: The new cleanup helper only runs after a fresh popover is created. If the user opens the delete confirmation, waits for the `requestAnimationFrame` listener registration, and clicks the same delete button again, the earlier `existing.remove(); return;` branch still removes the popover without unregistering `closeOnOutside`; the capture handler also ignores that button click. Hoist or track the cleanup so this dismissal path removes the document listener too.
  - confidence: 0.9
