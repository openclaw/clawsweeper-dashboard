## Summary

Keep open. The PR targets a real Control UI listener leak and moves in the right direction, but the patch is not merge-ready because the same-button toggle path still removes an already-open delete confirmation popover without unregistering the document click listener. The older #69982 is open for the same bug, so maintainers should consolidate around one corrected branch.

## What This Changes

The PR updates `ui/src/ui/chat/grouped-render.ts` so Cancel, Delete, and outside-click delete-confirm dismissals use a shared helper that removes the popover and unregisters the document click listener.

## Best Possible Solution

Land one canonical narrow patch that routes every delete-confirm dismissal path through listener cleanup, including Cancel, Delete, outside click, and clicking the delete button again while the popover is already open. Add focused Vitest coverage for listener balance and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main reproduces from source by opening a delete confirmation, waiting for the `requestAnimationFrame` listener registration, then dismissing with Cancel or Delete; on the PR head, clicking the same delete button again still follows `existing.remove(); return;` without cleanup.

## Solution Assessment

No, not as written. A shared cleanup helper is the right narrow approach, but it needs to be reachable from the existing-popover toggle path and backed by a regression test before merge.
