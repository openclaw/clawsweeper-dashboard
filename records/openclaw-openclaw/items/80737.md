## Summary

Keep open: the heartbeat-boundary fix is directionally right and the PR includes useful live Telegram proof, but the current parser still misses a common terminal-newline form, so the marker can still leak and delivery can remain non-silent until that narrow case is fixed.

## What This Changes

The PR strips standalone trailing `notify=false` from heartbeat fallback replies, forwards it as silent delivery metadata through durable sends, adjusts exec-completion fallback handling, adds regression coverage, and includes unrelated formatting churn across several files.

## Best Possible Solution

Keep the heartbeat-owned parser and silent forwarding path, but accept final CR/LF after a standalone marker and add LF/CRLF terminal-newline regression coverage before landing the canonical fix for the linked Telegram issue.

## Reproduction Assessment

Yes. Current main lacks `notify=false` consumption in heartbeat fallback delivery, and a focused check of the PR regexp shows the remaining `notify=false\n` and `notify=false\r\n` variants still fail to match.

## Solution Assessment

No, not yet. Parsing in heartbeat normalization and forwarding `silent` through the existing delivery contract is the right boundary, but the suffix matcher needs to tolerate terminal line breaks so the full reported class is fixed.
