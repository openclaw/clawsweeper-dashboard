## Summary

Keep open: current main still lacks the heartbeat fallback `notify=false` fix, and this PR is the canonical branch for the linked Telegram leak, but it should not merge until the parser handles a marker followed by terminal newlines.

## What This Changes

This PR strips standalone trailing `notify=false` markers from heartbeat fallback replies, forwards the result as silent delivery metadata, and adds heartbeat/exec regression coverage.

## Best Possible Solution

Update this canonical PR so the heartbeat fallback parser consumes a standalone final marker even when trailing line endings follow it, then land the linked Telegram bug fix with focused regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main has no `notify=false` fallback parser or silent forwarding in the heartbeat delivery call, and a focused regex check shows the PR still misses `notify=false\n`. I did not run a live current-main Telegram reproduction.

## Solution Assessment

Not yet. The normalization boundary is the right narrow fix, but the matcher must tolerate terminal line endings after the marker before merge so common model output cannot still leak the control text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Allow final newlines after notify=false:** `src/infra/heartbeat-runner.ts:738`
  - body: The new matcher only accepts horizontal whitespace after `false`, so a reply like `No interruption needed.\n\nnotify=false\n` will not be stripped and can still send visible control text without `silent`. Terminal newlines are common in model output, so the parser and regression coverage should allow them.
  - confidence: 0.86
