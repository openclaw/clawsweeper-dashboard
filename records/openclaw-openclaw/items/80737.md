## Summary

Keep open: the PR targets a real current-main heartbeat/Telegram bug and has sufficient live proof for the basic marker case, but the proposed parser still misses `notify=false` when the marker is followed by a terminal newline.

## What This Changes

The PR strips standalone trailing `notify=false` heartbeat fallback markers, forwards `silent: true` into durable Telegram delivery, and adds focused heartbeat/test-helper regression coverage.

## Best Possible Solution

Update the heartbeat fallback marker parser to consume a final standalone `notify=false` even when trailing line endings follow it, add regression coverage for that case, and land one canonical fix for the linked Telegram issue.

## Reproduction Assessment

Yes. Source inspection shows current main lacks `notify=false` fallback handling, and the PR head regex at `src/infra/heartbeat-runner.ts:738` cannot match `...\nnotify=false\n`; I did not run a live current-main Telegram reproduction.

## Solution Assessment

No, not yet. The owner-boundary direction is right, but the parser should tolerate trailing line endings after the final standalone marker and tests should lock that in before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Allow final newlines after notify=false:** `src/infra/heartbeat-runner.ts:738`

Model replies commonly include a terminal newline, and in that case the fallback still sends visible `notify=false` without `silent`; please accept trailing line endings and cover that regression.
