## Summary

Keep open: this PR targets a real current-main Telegram heartbeat bug and includes sufficient live Telegram proof, but the parser still misses a trailing marker followed by a final newline or CRLF, so a common model-output shape can still leak visible `notify=false`.

## What This Changes

The PR strips a standalone trailing `notify=false` marker from heartbeat fallback replies, forwards silent delivery metadata through durable Telegram delivery, and adds heartbeat/exec-completion regression coverage.

## Best Possible Solution

Tighten the heartbeat fallback normalizer so a final standalone `notify=false` line is consumed even when line endings follow it, preserve inline text, add regression coverage, and land this as the canonical fix for the linked Telegram bug.

## Reproduction Assessment

Yes, at source level: current main normalizes heartbeat fallback text without parsing `notify=false` and sends the durable batch without `silent`. A focused regex check also shows the PR still misses a final newline after the marker; I did not run live current-main Telegram.

## Solution Assessment

Not yet. The normalization boundary and structured silent forwarding are the right approach, but the marker parser needs to tolerate terminal line endings before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Allow final newlines after notify=false:** `src/infra/heartbeat-runner.ts:738`

  - confidence: 0.88
