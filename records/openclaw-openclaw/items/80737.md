## Summary

Keep open: this is the canonical PR for the linked Telegram heartbeat leak and it has sufficient live proof, but the current matcher still lets a standalone `notify=false` marker leak when the model leaves a final line break.

## What This Changes

The PR strips trailing standalone `notify=false` markers from heartbeat fallback replies, forwards `silent: true` into delivery, and adds heartbeat/exec regression coverage.

## Best Possible Solution

Tighten this PR's parser to consume final line endings after the standalone marker, add regression coverage for that case, and then land it as the canonical fix for https://github.com/openclaw/openclaw/issues/80569.

## Reproduction Assessment

Yes for source-level reproduction: current main and the latest release strip heartbeat tokens but not a trailing `notify=false`, and they do not pass `silent` on fallback sends. I did not run a live current-main Telegram reproduction in this read-only review.

## Solution Assessment

Not yet. The ownership boundary is right, but the marker parser should tolerate final line endings before merge so common model output cannot leak the control marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Allow final line breaks after notify=false:** `src/infra/heartbeat-runner.ts:738`
  - body: The new trailing marker regex only permits horizontal whitespace after `false`, so model output ending with `notify=false\n` is not stripped or marked silent. Heartbeat/model text commonly carries a final newline; without accepting final line endings, the visible control marker can still be sent through Telegram.
  - confidence: 0.9
