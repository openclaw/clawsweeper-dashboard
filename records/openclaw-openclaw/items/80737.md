## Summary

Keep open: the PR is a scoped fix for the linked Telegram heartbeat bug, but the new trailing-marker matcher still misses a standalone `notify=false` followed by a terminal newline, so the marker can still leak in a common model-output shape.

## What This Changes

This PR updates heartbeat fallback normalization to strip standalone trailing `notify=false` markers, forward silent delivery metadata, and add heartbeat/exec regression coverage.

## Best Possible Solution

Update the PR so the heartbeat fallback parser consumes a standalone final marker even when trailing line endings follow it, then land one canonical fix for the linked Telegram bug.

## Reproduction Assessment

Yes, from source inspection and a focused matcher check: current main sends heartbeat fallback text without consuming `notify=false`, and the PR matcher still misses the terminal-newline variant. I did not run a live current-main Telegram reproduction.

## Solution Assessment

Not yet. The owner-boundary approach is right, but the matcher should tolerate final line whitespace/newlines before merge so the visible marker cannot leak in common model output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Allow final newlines after notify=false:** `src/infra/heartbeat-runner.ts:738`
  - body: The new matcher only accepts horizontal whitespace after `false`, so `...\nnotify=false\n` is not stripped; the fallback then passes visible `notify=false` through without `silent`. Terminal newlines are common in model text, so the leak can survive this fix unless the parser allows trailing line breaks and tests cover it.
  - confidence: 0.86
