## Summary

Keep open: the PR targets a valid heartbeat/Telegram bug and has useful live proof, but the current matcher still misses `notify=false` when normal trailing line endings follow it, so a common model-output shape can still leak the marker before merge.

## What This Changes

The PR strips standalone trailing `notify=false` markers from heartbeat fallback replies, converts them to silent delivery metadata, and adds regression coverage for Telegram heartbeat and exec-completion sends.

## Best Possible Solution

Land one canonical fix that keeps structured `heartbeat_respond` as the primary contract, strips only standalone trailing fallback markers including final line endings, and preserves inline `notify=false` text.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the PR regex leaves `notify=false` visible for replies that end with a normal terminal newline. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not yet. The normalization boundary is right, but the parser must tolerate final line endings before this is the narrowest complete fix.
