## Summary

Keep open: this PR addresses a real Telegram heartbeat delivery bug and has sufficient live Telegram proof, but the proposed matcher still fails when a standalone trailing `notify=false` marker is followed by a line ending, so the visible control-text leak can survive common model output.

## What This Changes

The PR strips standalone trailing `notify=false` markers from heartbeat fallback replies, forwards silent durable-delivery metadata, and adds heartbeat plus exec-completion regression coverage.

## Best Possible Solution

Tighten the heartbeat-only marker parser to accept final line endings and blank lines, preserve inline `notify=false` text, add heartbeat and exec-completion regression coverage, then land this as the canonical fix for https://github.com/openclaw/openclaw/issues/80569.

## Reproduction Assessment

Yes for source-level reproduction: current main normalizes fallback text without `notify=false` handling and sends the durable batch without `silent`, while a focused regex probe shows this PR still misses newline-suffixed markers. I did not run a live current-main Telegram reproduction.

## Solution Assessment

No, not yet. The owner-boundary approach is right, but the matcher must tolerate terminal line endings while keeping inline `notify=false` text untouched.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Accept trailing line endings after notify=false:** `src/infra/heartbeat-runner.ts:738`
