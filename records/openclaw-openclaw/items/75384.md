## Summary

Keep open: the updated PR still has a correctness blocker because U+00A0 is treated as whitespace by JavaScript trimming, and OpenClaw's Anthropic transport drops user string content whose trimmed length is zero before building the API payload. The patch tests the value passed to `activeSession.prompt`, but not the provider payload that needs a non-empty user turn.

## What This Changes

The PR adds a changelog entry, changes runtime-only embedded-runner blank transcript prompts to submit a U+00A0 placeholder to `activeSession.prompt`, and updates the context-engine regression test expectation.

## Best Possible Solution

Keep the memory-flush `transcriptPrompt` blank, but make the runtime-only provider submission use a sentinel or transport path that actually survives Anthropic payload filtering while trajectory and transcript-facing prompt values remain empty. Add a payload-level regression test so the fix proves the Anthropic `messages` entry is non-empty, not only that `activeSession.prompt` received a placeholder.

## Reproduction Assessment

Yes. Static tracing on current main reproduces the path from memory flush `transcriptPrompt: ""` to a runtime-only empty prompt submission, and the PR's U+00A0 replacement can be checked against the Anthropic transport's `.trim()` filters without a live provider call.

## Solution Assessment

No. The runner-level boundary is the right place to avoid exposing `NO_REPLY` and flush instructions, but U+00A0 is not the best placeholder because existing Anthropic payload conversion treats it as blank; use a non-trimmed invisible sentinel or explicit transport handling and cover the final payload.
