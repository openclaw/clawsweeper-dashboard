## Summary

Keep open. The bug is valid, but the patch is not yet correct: its U+00A0 provider placeholder is still whitespace under JavaScript trimming, and the current Anthropic transport drops whitespace-only user messages before building the request.

## What This Changes

The branch adds a changelog entry, changes runtime-only embedded-runner blank prompt submission to send a U+00A0 placeholder, and updates the context-engine regression expectation.

## Best Possible Solution

Keep the memory-flush transcript prompt blank and the flush instructions in runtime/system context, but use a named non-whitespace provider-only sentinel or payload-boundary normalization that survives the Anthropic transport filter, with regression coverage that inspects the provider payload while preserving blank transcript and trajectory fields.

## Reproduction Assessment

Yes. Static tracing reproduces the path on current main, and the PR candidate can be checked by applying the current Anthropic transport rule: U+00A0 trims to an empty string and is therefore filtered as a blank user message.

## Solution Assessment

No. Fixing the runner/provider-facing blank turn is the right boundary, but a whitespace-only NBSP placeholder is not robust with the current Anthropic payload builder; a non-whitespace provider-only sentinel plus payload-level test is safer.
