## Summary

Keep open: current `main` and shipped v2026.5.18 still reject per-session Talk realtime instructions, and the requested change is a valid additive protocol capability that needs maintainer direction on prompt-policy boundaries.

## What This Changes

Add a bounded per-session realtime context/instructions field to `talk.client.create` and `talk.session.create` so channel-selected context can be merged into OpenClaw-owned realtime sessions.

## Best Possible Solution

Add an additive, bounded per-session context field after maintainer approval, append it after OpenClaw's built-in and configured realtime prompt without replacing policy text, and update protocol docs plus focused schema/handler tests for browser and gateway relay paths.

## Reproduction Assessment

Yes, from source inspection: the current schemas disallow extra properties and tests pin request-time instruction rejection, so a request containing `instructions` will fail validation. I did not run a live Gateway RPC because this read-only review already has direct validator, handler, docs, and shipped-release evidence.

## Solution Assessment

Unclear until maintainers decide the prompt-policy boundary. The requested direction is plausible, but the safer implementation is a bounded context field appended to OpenClaw-owned base/configured realtime instructions rather than allowing request data to replace configured policy text.
