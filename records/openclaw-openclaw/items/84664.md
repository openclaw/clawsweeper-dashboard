## Summary

Keep open: current main and latest v2026.6.8 still reject request-time Talk `instructions`, and the handlers still assemble realtime provider instructions only from configured gateway state. This remains a focused additive Talk API request that needs maintainer product/security judgment, not cleanup closure.

## What This Changes

Request optional per-session realtime context or instructions on `talk.client.create` and gateway-relay `talk.session.create` so thin channels can pass session context into Talk.

## Best Possible Solution

Add an additive, bounded per-session context field at the Talk protocol boundary after maintainers decide caller trust, merge order, validation, and redaction, while preserving gateway-owned base and configured realtime instructions.

## Reproduction Assessment

Yes. Source inspection shows both Talk create schemas reject `instructions` and both handlers use configured realtime instructions only; the reporter also provided live installed-package Gateway failures on 2026-05-26.

## Solution Assessment

Unclear pending maintainer decision. The requested direction is plausible, but the best fix needs a bounded context contract and explicit merge/trust rules rather than an unreviewed raw instruction override.
