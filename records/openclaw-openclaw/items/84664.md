## Summary

Keep open: current `main` and the shipped v2026.5.18 Talk API still reject per-session realtime instructions, and the request is a real additive protocol capability rather than an already-fixed bug.

## What This Changes

The issue requests a bounded per-session realtime instructions/context field for `talk.session.create` gateway-relay sessions and `talk.client.create` browser sessions.

## Best Possible Solution

Add an additive, bounded per-session context/instructions field if maintainers approve the prompt-policy boundary, preserve the built-in realtime prompt and configured instructions, and update protocol docs plus focused schema/handler tests.

## Reproduction Assessment

Yes from source inspection: the current schemas disallow extra properties and tests pin instruction override rejection, so a request with `instructions` will fail validation. I did not run a live Gateway RPC because the read-only review already has direct validator and shipped-release evidence.

## Solution Assessment

Unclear until maintainers decide the prompt-policy boundary. The requested direction is plausible, but the safer implementation is a bounded context field appended after OpenClaw's base/configured realtime prompt rather than replacing configured instructions.
