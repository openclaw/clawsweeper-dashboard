## Summary

Keep open: current main and v2026.6.11 still reject request-time Talk `instructions`, and the requested additive API needs maintainer product and security design rather than cleanup closure.

## What This Changes

Allow Talk client/session creation to accept bounded per-session realtime context for browser and gateway-relay sessions and merge it with configured base instructions.

## Best Possible Solution

Define an additive, bounded per-session Talk context contract at the protocol boundary that preserves Gateway-owned base instructions, validates caller data, and documents the security model.

## Reproduction Assessment

Yes. Source inspection shows both create schemas still reject the requested field, and the reporter supplied live Gateway failures from installed OpenClaw versions showing the same unexpected-property error.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested direction is plausible, but a bounded per-session context merge contract is safer than accepting a raw instruction override.
