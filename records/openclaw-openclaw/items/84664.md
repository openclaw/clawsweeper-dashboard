## Summary

Keep open: current `main` and the latest release still reject per-session Talk realtime instructions, and this is an additive protocol/product decision rather than an already-fixed bug.

## What This Changes

The issue requests an additive Talk API field for per-session realtime context/instructions during gateway relay and browser realtime session creation.

## Best Possible Solution

Add an additive, bounded per-session context field only after the trusted-caller semantics are approved, compose it after the built-in realtime prompt and configured instructions, and update protocol docs plus focused schema/handler tests.

## Reproduction Assessment

Yes from source inspection: both create schemas are strict and omit `instructions`, while tests pin rejection of request-time instruction overrides. I did not run a live Gateway RPC because the validators and shipped release source already show the failure path.

## Solution Assessment

Unclear until maintainers decide the prompt-policy boundary. The requested direction is plausible, but the safer implementation is a bounded context field appended after OpenClaw's base/configured realtime prompt rather than replacing configured instructions.
