## Summary

Keep open: current main still passes raw `opts.replyToId` to Mattermost `root_id` at the send sink, so the central send-time root-resolution fix is not fully implemented, but this PR is not merge-ready because lookup failures can silently move intended thread replies to top-level posts and real Mattermost proof is missing.

## What This Changes

The PR adds a Mattermost send-time lookup that maps `replyToId` posts to their thread root before passing `root_id`, with mocked send tests for child, root, cross-channel, and lookup-failure cases.

## Best Possible Solution

Land a current-main send-sink repair that resolves child post IDs to the real Mattermost root, preserves or visibly fails intended threading on lookup failure, stays aligned with merged message-tool threading, and includes redacted live Mattermost proof.

## Reproduction Assessment

Yes at source level: Mattermost rejects child posts as `root_id`, current main can still pass raw `opts.replyToId` into `rootId`, and the PR's mocked tests exercise the intended resolution path. I did not establish a live Mattermost reproduction in this read-only review.

## Solution Assessment

No as submitted: resolving at the send sink is a plausible repair boundary, but silently dropping `root_id` on lookup failure is not the safest message-delivery behavior. The safer path is to preserve or visibly fail intended threading and prove it in a real Mattermost workspace.
