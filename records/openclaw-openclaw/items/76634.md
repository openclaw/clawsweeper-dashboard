## Summary

Keep open: the Mattermost root-resolution fix is still useful and not on current main, but the merge result has a fallback receipt-metadata bug and the external contributor still has not supplied real Mattermost proof.

## What This Changes

The PR changes Mattermost outbound sends to fetch a referenced post before setting `root_id`, then adds send tests for child replies, top-level roots, cross-channel references, and lookup failures.

## Best Possible Solution

Resolve the reply target at the Mattermost send sink, align receipt metadata with the actual delivered root or top-level fallback, then require redacted live Mattermost proof before merge.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` can pass a child post ID straight through as Mattermost `root_id`, and Mattermost upstream rejects non-root or wrong-channel roots. I did not run a live Mattermost instance in this read-only review.

## Solution Assessment

Partially. Resolving the referenced post at the Mattermost send sink is the right owner-boundary fix, but the merge result should also align receipt metadata with the delivered root/top-level fallback and needs real Mattermost proof before merge.
