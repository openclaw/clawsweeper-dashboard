## Summary

Keep open: current main and the latest release still pass raw Mattermost `replyToId` values as `root_id`, so the PR is a useful fix candidate, but it is not merge-ready because the new fallback can return stale threaded receipt metadata and the external contributor has not supplied real Mattermost proof.

## What This Changes

The PR adds a Mattermost post lookup before outbound sends, maps child reply targets to their thread root, drops missing or cross-channel reply targets from `root_id`, and adds mocked send tests.

## Best Possible Solution

Resolve reply roots at the Mattermost send sink, keep returned receipt/tool metadata consistent with the delivered root or top-level fallback, and merge only with redacted real Mattermost proof.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` can pass a child post ID straight through as Mattermost `root_id`, and Mattermost upstream rejects non-root or wrong-channel roots. I did not run a live Mattermost instance in this read-only review.

## Solution Assessment

Partially. Resolving the referenced post at the Mattermost send sink is the right owner-boundary fix, but the PR should also align receipt metadata with the delivered root/top-level fallback and provide real Mattermost proof before merge.
