## Summary

Keep open. The PR addresses a real Mattermost threading bug: current main sends `replyToId` directly as `root_id`, while Mattermost rejects child reply posts as roots. The implementation and tests are narrow, but the user-facing fix still needs the required changelog entry before merge.

## What This Changes

The PR resolves Mattermost reply targets via `/posts/<id>` before setting `root_id` and adds send tests for child, top-level, cross-channel, and lookup-failure cases.

## Best Possible Solution

Land the Mattermost root-resolution change after adding a single-line Unreleased/Fixes changelog entry and keeping the focused send regression tests green.

## Reproduction Assessment

Yes. Current main passes a child `replyToId` directly as Mattermost `root_id`, and Mattermost `CreatePost` rejects a root whose referenced post itself has `RootId`; I did not run a live Mattermost instance in this read-only pass.

## Solution Assessment

Yes. Resolving the referenced post through the existing Mattermost client in the plugin send path is the narrow owner-local fix; the remaining issue is the missing required changelog entry.
