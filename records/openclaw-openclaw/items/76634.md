## Summary

Keep open: the Mattermost send-sink problem is still useful to fix, but this branch is conflicting, has only mocked/unit verification, and currently converts lookup failures into top-level posts for replies that were meant to stay threaded.

## What This Changes

The PR adds a Mattermost send-time lookup that maps `replyToId` child posts to their thread root before setting `root_id`, plus mocked send tests for root, child, cross-channel, and lookup-failure cases.

## Best Possible Solution

Land a current-main send-sink repair that resolves child IDs to the real Mattermost root, never silently posts top-level on lookup failure, keeps receipt metadata aligned with the delivered root, and includes live Mattermost proof.

## Reproduction Assessment

Yes at source level: current main passes raw `opts.replyToId` into Mattermost `rootId`, and the PR targets the child-post-as-root failure path. I did not establish a live failing Mattermost reproduction in this read-only review.

## Solution Assessment

No as submitted: resolving at the Mattermost send sink is the right boundary, but the patch must not silently drop threading on lookup failure and needs a current-main rebase plus live proof.
