## Summary

Keep open: current main and the latest release still pass raw Mattermost `replyToId` values as `root_id`, so the PR addresses a real send-sink gap, but the merge result needs receipt metadata aligned with the resolved root and redacted real Mattermost proof before merge.

## What This Changes

The PR changes Mattermost outbound sends to resolve a referenced post into the actual thread root before calling the post API and adds focused send tests for root, child, cross-channel, and lookup-failure cases.

## Best Possible Solution

Resolve the Mattermost reply target at the send sink, return receipt metadata that matches the resolved root or top-level fallback, and require redacted real Mattermost proof before merge.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` pass `opts.replyToId` directly as Mattermost `rootId`, while Mattermost server validation requires `root_id` to reference a root post in the same channel. I did not run a live Mattermost server in this read-only review.

## Solution Assessment

Partially. Resolving the referenced post at the Mattermost send sink is the right owner-boundary fix, but the merge result should also align receipt metadata with the delivered root or top-level fallback and needs real Mattermost proof before merge.
