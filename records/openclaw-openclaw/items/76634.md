## Summary

Keep open: current main and the latest release still pass raw Mattermost reply targets as `root_id`, but this PR is not merge-ready because it lacks real Mattermost proof, is currently conflicting, and introduces message-delivery edge cases around lookup failure and receipt metadata.

## What This Changes

The branch changes Mattermost outbound sends to resolve a `replyToId` post through `GET /posts/{id}`, post to the resolved thread root, drop unresolved or cross-channel roots, and add focused send tests.

## Best Possible Solution

Refresh the branch, keep the send-sink root resolution, limit top-level fallback to deliberate non-resolvable cases, align receipt metadata with the delivered root, and require redacted live Mattermost proof before merge.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` pass raw `opts.replyToId` into Mattermost `root_id`, while Mattermost server code rejects a nested reply as a root. I did not run a live Mattermost workspace in this read-only review.

## Solution Assessment

No, not yet: resolving at the Mattermost send sink is the right boundary, but the patch should not silently top-level fallback on transient lookup failures and should record the resolved root in receipt metadata.

Full review comments:

- **[P1] Do not drop threading on transient lookup failures:** `extensions/mattermost/src/mattermost/send.ts:417-418`
  - body: The helper catches every `GET /posts/{id}` failure and returns `undefined`, so a valid threaded reply can be sent as a top-level post whenever the lookup hits a transient 5xx/network/permission failure. Limit fallback to deliberate non-resolvable cases such as 404 or cross-channel roots, or fail/retry instead of silently changing the delivery target.
  - confidence: 0.82
