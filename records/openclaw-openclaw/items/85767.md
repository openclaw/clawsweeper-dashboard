## Summary

Keep open: the latency direction is useful, but this PR is not merge-ready because the deferred flush still reuses a completed ReplyOperation, and current main cancels backends attached to completed operations.

## What This Changes

This PR defers auto-reply memory flush until after reply completion, registers pending flushes by session key, adds a next-turn preflight barrier/rebind path, and updates focused tests.

## Best Possible Solution

Keep the latency improvement, but run post-reply memory flush through a flush-safe maintenance context that does not attach to a completed reply operation and decide the deferred visible-error behavior before merge.

## Reproduction Assessment

Yes. Source inspection shows the PR passes a completed ReplyOperation into the post-reply flush, while current main cancels backends attached to operations that already have a result.

## Solution Assessment

No. Deferring the flush is a plausible direction, but reusing the completed reply operation is not the right contract for work that must run after reply completion.
