## Summary

Close as duplicate/superseded: current source, docs, and tests explain the reported `replies=0` shape as the documented group visible-reply policy, and the remaining Feishu upgrade/diagnostic decision is already tracked in https://github.com/openclaw/openclaw/issues/77666 with a verified `messages.groupChat.visibleReplies: "automatic"` workaround.

## What This Changes

The issue requests that Feishu group @mention replies from the main bot be delivered visibly instead of ending with `replies=0` after the model produces text.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/77666 as the canonical place for the Feishu upgrade/diagnostic decision, while operators who want legacy visible group replies use the message tool or set `messages.groupChat.visibleReplies: "automatic"`.

## Reproduction Assessment

Yes, from source inspection rather than a live Feishu account: a group turn with no automatic visible-reply config resolves to `message_tool_only`, the model can produce final text, and shared dispatch returns `queuedFinal=false` with no final send. That matches the reported `replies=0` logs.

## Solution Assessment

Yes for cleanup: closing this duplicate is the narrowest maintainable path because the remaining compatibility and diagnostic question is already tracked in the canonical Feishu issue. A code change here would be a product-policy decision, not a standalone Feishu card-dispatcher fix.
