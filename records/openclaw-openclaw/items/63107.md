## Summary

Keep open: current main and v2026.6.8 include the supported `--command` / `--command-argv` cron path, but they still do not warn when shell-looking text is passed as a main-session `systemEvent`; the open linked PR https://github.com/openclaw/openclaw/pull/63112 remains the implementation candidate.

## What This Changes

Issue requests a non-blocking cron add/edit warning when main-session `systemEvent` payload text looks like a shell command because system events are dispatched as notifications, not executed.

## Best Possible Solution

Keep this issue open until maintainers either land/refocus https://github.com/openclaw/openclaw/pull/63112 as a narrow warning that points to command cron, or explicitly decide that the shipped command-payload docs are sufficient.

## Reproduction Assessment

Yes, at source level: current main routes main-session `systemEvent` cron text into `enqueueSystemEvent`, while only `command` payloads execute through the command runner. I did not run a live gateway reproduction in this read-only pass.

## Solution Assessment

No final solution is on main yet. A warning is a plausible mitigation, but it should be maintainer-reviewed and aligned with the shipped `--command` / `--command-argv` path.
