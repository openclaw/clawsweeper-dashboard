## Summary

Keep open: current main and v2026.6.10 support first-class command cron jobs, but `--system-event` on the main session still queues text as a system event with no shell-command warning; the open linked PR remains the candidate fix.

## What This Changes

The issue requests a non-blocking CLI warning when main-session cron `systemEvent` payload text looks like a shell command.

## Best Possible Solution

Keep this issue open until maintainers land or refocus the linked warning PR, or explicitly decide that command-cron docs are sufficient without a heuristic warning.

## Reproduction Assessment

Yes, at source level: main-session cron `systemEvent` text is queued through `enqueueSystemEvent`, while only `command` payloads reach the command runner. I did not run a live gateway reproduction in this read-only pass.

## Solution Assessment

Unclear as final product direction: a CLI warning is a plausible narrow mitigation, but it should point to `--command`/`--command-argv` and needs maintainer acceptance of the heuristic.
