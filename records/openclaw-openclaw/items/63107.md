## Summary

Keep open: current main and latest release have the supported `--command` cron path, but `cron add` and `cron edit` still do not warn when shell-looking text is submitted as a main-session `systemEvent`; the open linked PR remains the candidate fix.

## What This Changes

The issue requests a non-blocking cron CLI warning when main-session `systemEvent` payload text looks like a shell command because that text is queued as a system event, not executed.

## Best Possible Solution

Keep this issue open until maintainers either land or refocus the linked warning PR to point at `--command` / `--command-argv`, or explicitly decide that current command-cron docs are sufficient.

## Reproduction Assessment

Yes, at source level: current main queues main-session `systemEvent` text through `enqueueSystemEvent`, while command payloads execute through the command runner. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainer review: a CLI warning is the narrowest mitigation for the reported footgun, but the linked PR needs conflict resolution and command-cron wording before it is the best final fix.
