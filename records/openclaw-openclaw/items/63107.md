## Summary

Keep open: current main and v2026.6.8 have the supported `--command` / `--command-argv` cron path, but they still do not warn when shell-looking text is passed as a main-session `systemEvent`; the open linked PR https://github.com/openclaw/openclaw/pull/63112 is the implementation candidate and needs maintainer review rather than a new cleanup close.

## What This Changes

The issue requests a non-blocking cron CLI warning when main-session `systemEvent` payload text looks like a shell command that will not be executed.

## Best Possible Solution

Keep the issue open until maintainers either land/refocus https://github.com/openclaw/openclaw/pull/63112 as a narrow warning that points at command cron, or explicitly decide that the shipped command-payload docs are sufficient.

## Reproduction Assessment

Yes at source level: current main routes main-session systemEvent cron text into `enqueueSystemEvent` and only `command` payloads execute through the command runner. I did not run a live gateway reproduction in this read-only pass.

## Solution Assessment

No final solution is on main yet. The best path is a maintainer-reviewed warning or an explicit decision that the current command-payload contract is enough, with the linked PR as the current candidate.
