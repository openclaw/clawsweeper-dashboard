## Label Justifications

- `P2`: The misleading payload choice can silently defeat scheduled work, but a supported explicit command-job path exists in the current release.
- `impact:other`: The issue concerns scheduled actions appearing successful without producing their intended operational effect.

## Summary

Current main and release v2026.6.5 provide a first-class command cron payload, but they do not implement the requested warning for shell-looking system-event text. The paired closing PR remains open, so this issue should stay open until maintainers either land a refreshed diagnostic or explicitly close both items in favor of the command payload contract.

## What This Changes

Request a non-blocking CLI warning when shell-looking text is supplied as a main-session system event instead of an executable command payload.

## Best Possible Solution

Keep shell execution explicit through `--command` and `--command-argv`, then add a narrow payload-kind diagnostic pointing users to that supported path if maintainers decide the existing help and documentation are insufficient.

## Reproduction Assessment

Yes at source level: current CLI and runtime treat `systemEvent` and `command` as distinct payload kinds, so shell syntax embedded in system-event text never enters the command runner. A live failing run was not repeated during this read-only review.

## Solution Assessment

Unclear. Warning users is reasonable, but heuristic shell detection is less robust than an explicit payload-kind diagnostic tied directly to the supported `--command` path.
