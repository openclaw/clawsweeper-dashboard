The paired closing PR remains open and conflicting, so this issue should stay open until maintainers either land/refresh that diagnostic or explicitly close both items in favor of the command-payload contract.

## What This Changes

The issue requests a non-blocking CLI warning when a main-session cron `systemEvent` payload looks like a shell command because system events are notifications and do not execute shell commands.

## Best Possible Solution

Keep shell execution explicit through `--command`/`--command-argv`, then either land a refreshed narrow warning PR or close both items with a maintainer decision that the documented command payload is sufficient.

## Reproduction Assessment

Yes at source level: current main accepts main-session `systemEvent` cron payloads and enqueues their text as a system event rather than executing it as a command. A live failing run was not repeated during this read-only review.

## Solution Assessment

Unclear. A non-blocking warning is a narrow mitigation, but maintainers still need to decide whether heuristic shell-looking text should warn now that `--command` is the explicit supported execution path.
