## Summary

Keep open: current main and the latest release include first-class command cron jobs, but the old main-session `systemEvent` path still accepts shell-looking text without a warning; the open linked PR remains the candidate fix rather than a merged implementation.

## What This Changes

The issue requests a non-blocking CLI warning when a main-session cron `systemEvent` payload looks like a shell command, because that text is queued as a system event and is not executed as a command.

## Best Possible Solution

Keep this issue open until maintainers land or refocus https://github.com/openclaw/openclaw/pull/63112 as a narrow warning that points script users at current command cron, or explicitly decide the shipped command-cron docs are enough.

## Reproduction Assessment

Yes, at source level: current main queues main-session `systemEvent` text and can return ok, while command execution is only on `payload.kind: "command"`. I did not run a live gateway in this read-only review.

## Solution Assessment

Yes, a CLI warning is the narrowest maintainable mitigation because it does not change main-session runtime semantics. The warning should reference current command cron rather than older model-backed isolated-message guidance.
