## Summary

Keep this PR open: the in-process cleanup is useful, but it does not fix the linked crash-created stale-temp path and the PR body only provides test output, not real behavior proof of cleanup after a failed write/rename scenario.

## What This Changes

Wraps the memory-core short-term store atomic writes in `try...finally` blocks that best-effort unlink the generated `.tmp` path.

## Best Possible Solution

Keep the useful in-process unlink cleanup, add stale matching `.tmp` cleanup in the short-term repair or dreaming sweep path, and cover both failure modes with focused regression tests plus real run output.

## Reproduction Assessment

Yes by source inspection. Current main writes `tmpPath` and then renames it with no cleanup, so a failed rename or process kill between those awaits leaves a matching `.tmp` file under `memory/.dreams`.

## Solution Assessment

No. The `try...finally` cleanup is a good partial fix for surviving write/rename exceptions, but it cannot run after a force-kill; the maintainable fix also needs stale-temp cleanup in repair or sweep.
