## Summary

Keep open: the current-main temp-file leak is real, but this PR only handles in-process write/rename failures and does not address crash-created stale `.tmp` files from the linked report; it also needs real behavior proof before merge.

## What This Changes

The PR wraps `writePhaseSignalStore` and `writeStore` atomic writes in `try...finally` blocks that unlink the generated `.tmp` path.

## Best Possible Solution

Keep the useful in-process cleanup, add stale matching `.tmp` cleanup in the short-term repair or dreaming sweep path, and cover both failure modes with focused regression tests plus real run output.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main writes `tmpPath` and then renames it with no cleanup, so a mocked rename failure or a force-kill between those awaits leaves a matching `.tmp` file under `memory/.dreams`.

## Solution Assessment

No. The `try...finally` cleanup is a good partial fix for write/rename exceptions, but it cannot execute after a process crash; the maintainable fix also needs repair or sweep cleanup for stale temp files.
