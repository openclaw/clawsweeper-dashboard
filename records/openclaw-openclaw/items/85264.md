## Summary

Keep open: the Windows performance problem is plausible, but this patch makes security-sensitive path checks depend on stale process-global realpath data and the PR still lacks inspectable real Windows behavior proof.

## What This Changes

The branch wraps OpenClaw path-safety re-exports with process-wide caches for realpath and existing-ancestor resolution to reduce repeated Windows `lstat` calls.

## Best Possible Solution

Keep the performance fix, but move caching to bounded per-operation/request scope or an upstream fs-safe design with explicit invalidation, then add stale symlink/ancestor regression coverage and Windows profile proof.

## Reproduction Assessment

Yes for the PR defect: source inspection shows module-level caches return realpath and ancestor results without invalidation while current main routes security-sensitive containment through those helpers. The reported Windows slowdown itself was not independently reproduced from the PR body.

## Solution Assessment

No. A process-global cache is not the best fix for security-boundary path checks; a scoped cache or upstream cache contract with invalidation would preserve current semantics while addressing repeated `lstat` work.

Full review comments:

- **[P1] Keep realpath checks tied to current filesystem state:** `src/infra/path-safety.ts:13-16`
  - body: This process-wide cache returns old realpath values without rechecking disk.   - body: The sandbox host path resolver depends on the deepest existing ancestor at the time of the check. Caching by input path forever means a later created, removed, or symlinked ancestor can still resolve through the old canonical parent, making sandbox path policy stale.
  - confidence: 0.84
