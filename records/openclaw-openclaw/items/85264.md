## Summary

Keep open: the Windows `lstat` performance target is useful, but this patch is not merge-ready because it adds process-wide caches to security-sensitive realpath and ancestor-resolution checks without invalidation, and the Windows proof is not independently inspectable.

## What This Changes

The branch adds process-wide realpath and existing-ancestor caches in `src/infra/path-safety.ts`, `src/infra/path-guards.ts`, and `src/infra/boundary-path.ts` to reduce repeated Windows `lstat` work.

## Best Possible Solution

Keep the performance goal, but move caching to bounded per-operation/request scope or an upstream fs-safe design with explicit invalidation, then add stale symlink/ancestor regression coverage and inspectable Windows profile proof.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection shows module-level caches return realpath and ancestor results without invalidation while current main uses these helpers for plugin hook, plugin skill, and sandbox bind security checks. The Windows slowdown itself was not independently reproduced beyond the PR-body profile numbers.

## Solution Assessment

No. A process-global cache is not the safest fix for security-sensitive paths; scoped caching or an upstream invalidation-aware contract would preserve freshness while still reducing repeated `lstat` work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep realpath checks tied to current filesystem state:** `src/infra/path-safety.ts:6`
  - body: `globalRealpathCache` stores successful realpath results for the life of the process, and `isPathInsideWithRealpath` is used for plugin hook and skill containment with `requireRealpath: true`. If a symlink or plugin path is retargeted after the first check, later boundary checks can reuse the stale inside result instead of reading the current filesystem. Keep caching scoped to one operation or add invalidation before sharing it globally.
  - confidence: 0.88
- **[P1] Do not cache ancestor resolution without invalidation:** `src/infra/boundary-path.ts:5`
  - body: `ancestorPathCache` memoizes `resolvePathViaExistingAncestorSync` forever by input string, but sandbox bind validation relies on that helper to catch symlink escapes through existing ancestors. After an ancestor directory or symlink changes, the stale canonical path can drive the allow/deny decision. Use a request-local cache or invalidate on filesystem changes instead.
  - confidence: 0.86
