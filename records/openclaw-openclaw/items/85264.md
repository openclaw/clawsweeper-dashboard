## Summary

Keep open: the Windows performance problem is plausible and has a paired open issue, but this PR is not merge-ready because process-wide path caches can reuse stale filesystem answers across security-boundary checks, and the proof is still a summarized profile rather than an inspectable artifact.

## What This Changes

This PR adds process-wide caches around realpath and existing-ancestor path resolution, reroutes path guard exports through them, and adds cache invalidation tests.

## Best Possible Solution

Move the optimization into a scoped or invalidation-aware path boundary, preserve live fs-safe freshness for security checks, and add regression tests that fail when symlink or ancestor changes are reused without invalidation.

## Reproduction Assessment

Yes for the PR-introduced cache defect: source inspection shows global Maps return before fresh fs-safe realpath and existing-ancestor reads, while runtime call sites do not invoke the added invalidators. The original Windows slowdown was not independently reproduced because no inspectable profile artifact is attached.

## Solution Assessment

No. The current approach caches above fs-safe with process-wide lifetime; a safer fix is scoped or invalidation-aware caching at the path-safety boundary plus stale symlink/ancestor regression tests and inspectable Windows proof.
