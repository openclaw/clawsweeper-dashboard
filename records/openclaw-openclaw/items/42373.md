## Summary

Keep open: current main still has no currency metadata for model pricing and still treats cost values as USD/dollar-formatted across config, runtime, UI, macOS, diagnostics, and docs. This is a valid feature request, but it needs maintainer design before implementation because mixed-currency aggregation and existing USD-shaped public fields are compatibility-sensitive.

## What This Changes

Add a costCurrency configuration surface so model cost values can display as CNY/RMB/EUR/USD symbols instead of always using dollar-formatted USD.

## Best Possible Solution

Keep this open for a maintainer-led pricing-currency design: either explicitly retain USD-only pricing or add backward-compatible currency metadata through config/schema/types, pricing lookup and aggregation, session/API payloads, CLI/chat/status output, Control UI/macOS formatting, diagnostics metrics, docs, and mixed-currency tests.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence verification path for the current gap: source inspection shows strict schemas would not accept a currency key, and all cost display/metrics paths remain USD-shaped.

## Solution Assessment

Unclear that the proposed per-provider display symbol alone is the best solution. Maintainers should first decide per-provider versus per-model/global currency, conversion behavior, mixed-currency totals, and compatibility for USD-named public fields.
