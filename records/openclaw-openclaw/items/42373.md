Keep open: current main and latest release still define model pricing as USD and render dollar-prefixed cost labels, so the request remains a valid low-priority product/config decision rather than a cleanup-close candidate.

## What This Changes

Request adds a `costCurrency` config option so configured model costs can render with local currency symbols instead of hardcoded dollars.

## Best Possible Solution

Choose one canonical currency contract first; if accepted, add schema, docs, tests, and shared formatting across status, `/usage`, CLI, Gateway usage summaries, and Control UI with USD as the default.

## Reproduction Assessment

Yes, at source level: `costCurrency` has no schema/source hit, docs define configured pricing as USD, and shared renderers emit dollar-prefixed labels. A live UI run was not necessary because the current source path is explicit.

## Solution Assessment

Unclear. A per-provider `costCurrency` field is plausible, but maintainers should first decide whether this is display-only metadata, a pricing-unit contract, conversion, or a global user preference.
