This remains a low-priority product/config feature that needs maintainer direction before implementation.

## What This Changes

Request to add a `costCurrency` config option so model cost estimates can display non-USD currency symbols.

## Best Possible Solution

Decide one canonical currency contract first; if accepted, add schema, docs, tests, and a shared formatter across status, `/usage`, CLI, and Control UI while preserving USD as the default.

## Reproduction Assessment

Yes. Source inspection reproduces the current limitation: the config schema has no `costCurrency` key, pricing is documented as USD, and the relevant UI/status formatters hardcode `$`.

## Solution Assessment

Unclear. A per-provider `costCurrency` field is plausible, but maintainers need to decide whether currency is only a display symbol or changes the meaning of configured pricing values.
