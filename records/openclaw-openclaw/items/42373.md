## Summary

Keep open: current main still lacks currency metadata for model pricing and still treats costs as USD/dollar-formatted across config, runtime, UI, macOS, diagnostics, and docs. The request is a valid feature, but mixed-currency aggregation and existing USD-shaped public fields need maintainer product/API design before implementation.

## What This Changes

Add a costCurrency configuration option so model cost displays can use currencies such as CNY, EUR, or USD instead of always showing dollars.

## Best Possible Solution

Keep this open for a maintainer-led pricing-currency design that either documents USD-only pricing or adds backward-compatible currency metadata through config/schema/types, pricing lookup, aggregation, display surfaces, telemetry, docs, and mixed-currency tests.

## Reproduction Assessment

Not applicable as a feature request. Current-main source inspection gives a high-confidence verification path: strict schemas do not define costCurrency, and user-facing formatters, docs, and telemetry remain USD-shaped.

## Solution Assessment

Unclear that a per-provider display symbol alone is the best solution. Maintainers should first decide currency scope, per-model versus per-provider metadata, conversion or non-aggregation behavior, and compatibility for USD-named public fields.
