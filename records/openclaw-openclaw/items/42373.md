## Summary

Keep open: current main still treats model costs as USD and renders dollar-prefixed cost labels, so the requested configurable currency behavior is not implemented. This is a low-priority product/config feature rather than a strict bug or closable duplicate.

## What This Changes

The issue requests a `costCurrency` config option so provider cost displays can use non-USD currency symbols such as `¥` or `€`.

## Best Possible Solution

Decide one canonical currency contract first; if accepted, add schema, docs, tests, and a shared formatter across status, `/usage`, CLI, and Control UI while preserving USD as the default.

## Reproduction Assessment

Yes, source inspection reproduces the current limitation: the cost contract is documented as USD per 1M tokens, `formatUsd()` and Control UI `formatCost()` hardcode `$`, and no `costCurrency` key exists.

## Solution Assessment

Unclear; the proposed per-provider `costCurrency` is plausible but not proven best because current pricing fields are semantically USD, so maintainers need to choose the currency/unit/display contract before implementation.
