## Summary

Keep open: current main still has no costCurrency config surface and the cost contracts remain USD/dollar-shaped across schema, runtime, UI, telemetry, and docs. This is a valid feature request, but the mixed-currency and compatibility details need maintainer API/product design before implementation.

## What This Changes

Add a configurable currency setting for model cost display so non-USD provider pricing can show symbols such as CNY or EUR instead of dollars.

## Best Possible Solution

Keep this open for a maintainer-led pricing-currency design that either documents USD-only pricing or adds backward-compatible currency metadata through config/schema/types, cost lookup, aggregation, display surfaces, telemetry, docs, and mixed-currency tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path: costCurrency is absent, strict schemas reject unknown cost keys, and display/telemetry/docs remain USD-shaped.

## Solution Assessment

Unclear that a per-provider display symbol alone is the best solution. Maintainers should first decide currency scope, mixed-currency aggregation behavior, and compatibility for USD-named public fields.
