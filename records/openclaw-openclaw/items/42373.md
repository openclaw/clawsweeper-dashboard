## Summary

Keep open: current main and v2026.6.6 still define model pricing as USD and render dollar-specific cost labels. This remains a low-priority product/config decision because maintainers need to choose the currency semantics before implementation.

## What This Changes

The issue requests a `costCurrency` configuration option so model-cost displays can use non-USD currency symbols.

## Best Possible Solution

Decide one canonical currency contract first; if accepted, add schema, docs, tests, and shared formatting across status, `/usage`, CLI, Gateway usage summaries, and Control UI while preserving USD as the default.

## Reproduction Assessment

Yes, at source level: the schema has no `costCurrency`, docs define configured pricing as USD, and multiple renderers use dollar-specific formatters. I did not run a live UI or CLI scenario because the current source path is explicit.

## Solution Assessment

Unclear. A per-provider `costCurrency` field is plausible, but maintainers should first choose whether this is display-only metadata or a canonical pricing-unit contract that flows through all cost aggregation surfaces.
