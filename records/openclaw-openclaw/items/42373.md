## Summary

Keep open: current main still has no currency-aware pricing config and the request would change public model pricing, usage aggregation, SDK, CLI, and UI display contracts, so it needs maintainer product/API direction before implementation.

## What This Changes

Add a `costCurrency` option so model/provider cost displays can use non-USD currency symbols such as CNY or EUR.

## Best Possible Solution

Define a backward-compatible currency contract across model config, provider/plugin metadata, transcript usage, aggregation, SDK/API fields, CLI/status output, UI formatting, and docs, or explicitly keep model pricing USD-only.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a bug. Source inspection on current main shows no currency config key and multiple USD/dollar-only display, SDK, and usage paths.

## Solution Assessment

Unclear: a per-provider display symbol is not enough until maintainers decide currency scope, mixed-currency aggregation behavior, and compatibility for existing `costUsd` surfaces.
