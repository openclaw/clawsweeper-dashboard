## Summary

Keep open: current main still has a USD-only pricing contract and dollar-only display in status, CLI, SDK, docs, and Control UI paths. The requested currency option is useful, but it changes public config/API semantics and needs maintainer product direction before implementation.

## What This Changes

The issue requests a `costCurrency` config option so model-cost values can display as currencies such as CNY or EUR instead of always using dollars.

## Best Possible Solution

Define a backward-compatible currency contract across model config, provider/plugin metadata, usage aggregation, SDK/API fields, CLI/status output, UI formatting, and docs, or explicitly keep model pricing USD-only.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a broken existing behavior report. Source inspection verifies current main has no currency config key and still documents and formats costs as USD.

## Solution Assessment

No: the proposed per-provider display symbol is not enough by itself until maintainers decide currency scope, mixed-currency aggregation behavior, and `costUsd` compatibility. The maintainable fix needs a typed currency contract or an explicit decision to keep costs USD-only.
