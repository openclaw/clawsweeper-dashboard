## What This Changes

The issue requests a new `costCurrency` configuration option so model cost values can be displayed in currencies such as CNY or EUR instead of always using `$`.

## Best Possible Solution

Define a backward-compatible typed currency contract across model config, provider/plugin metadata, pricing cache, usage aggregation, SDK/API fields, CLI/status output, UI formatting, diagnostics, and docs, or explicitly keep model pricing USD-only.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a broken existing behavior report. Source inspection verifies current main has no currency config key and still documents and formats costs as USD.

## Solution Assessment

No: the proposed per-provider symbol-only option is not enough until maintainers decide currency scope, mixed-currency aggregation behavior, and `costUsd` compatibility. The maintainable path is a typed currency contract or an explicit decision to keep costs USD-only.
