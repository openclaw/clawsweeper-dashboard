## Summary

Keep open: current main and `v2026.6.8` still define model pricing as USD and render dollar-prefixed cost labels, while the requested currency support would add new config/product surface that needs maintainer direction.

## What This Changes

The issue asks for a new `costCurrency` configuration option so configured model costs can render with local currency symbols instead of always using dollars.

## Best Possible Solution

Maintainers should choose one canonical currency contract first; if accepted, update schema, docs, tests, status, `/usage`, CLI summaries, Gateway usage surfaces, and Control UI formatting while preserving USD as the default.

## Reproduction Assessment

Yes. Current main and `v2026.6.8` are source-reproducible: there is no `costCurrency` config key, and the schema, docs, status/CLI formatter, and Control UI formatters still use USD/dollar semantics.

## Solution Assessment

Unclear. A per-provider `costCurrency` field is plausible, but maintainers need to decide whether the permanent contract is provider pricing metadata, a user display preference, or conversion-aware accounting before implementation.
