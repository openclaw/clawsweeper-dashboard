## Summary

Keep open: current main still has USD-only pricing contracts and dollar-only cost display, while accepting non-USD configured costs would need an API/product decision for currency scope and mixed-currency aggregation.

## What This Changes

The issue requests a new `costCurrency` config option so model cost displays can use symbols such as CNY/RMB or EUR instead of always showing dollars.

## Best Possible Solution

Design a backward-compatible currency model for pricing metadata, totals, SDK/diagnostics fields, and display surfaces, or explicitly document configured model pricing as USD-only.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a bug. Source inspection verifies current main has no `costCurrency` key and still formats costs as USD.

## Solution Assessment

Unclear: a per-provider display symbol alone is likely not enough because cost totals aggregate across providers and existing API fields are USD-shaped. Maintainers should decide currency scope and mixed-currency semantics before implementation.
