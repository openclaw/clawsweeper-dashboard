## Summary

Keep open: current main still has USD-only model pricing metadata and dollar-only cost display, and adding non-USD costs needs maintainer API/product design for currency scope and mixed-currency aggregation.

## What This Changes

Request adding a `costCurrency` config option so model pricing and cost displays can use non-USD currency symbols such as `¥` or `€`.

## Best Possible Solution

Keep this open for maintainer-led pricing-currency design that either documents USD-only pricing or adds backward-compatible currency metadata across config, SDK/manifest docs, cost estimation, aggregation, diagnostics, UI surfaces, and tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path: current schemas and docs have no currency metadata, and display formatters still emit dollar strings.

## Solution Assessment

Unclear that a per-provider display symbol alone is the best solution. Maintainers should first decide provider versus model currency scope, mixed-currency aggregation behavior, and compatibility for USD-named fields.
