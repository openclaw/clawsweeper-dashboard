## Summary

Keep this PR open, but do not merge it as-is: the hardening direction is useful, yet the diff introduces source-proven regressions in existing session targeting, provider plain-text tool-call repair, and bundled provider tool support.

## What This Changes

The PR hardens tool-call handling by tightening `sessions_send` target validation, adding repeated tool-error loop termination, disabling default plain-text tool-call promotion, and changing tool-capability defaults for unknown/OpenRouter models.

## Best Possible Solution

Keep the loop-hardening and terminal-error intent, but preserve existing provider opt-in wrappers, valid `label + agentId` targeting, and bundled-provider tool support before merge.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the introduced regressions: existing provider wrapper call sites become pass-through, `sessions_send({ label, agentId })` now fails validation, and providers without `compat.supportsTools` lose tool payloads.

## Solution Assessment

No. The PR is a plausible hardening patch, but the best fix is narrower: keep existing valid contracts and make fail-closed behavior explicit only for providers or selector combinations proven unsafe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep the wrapper opt-in behavior:** `src/plugin-sdk/provider-stream-shared.ts:212`
  - body: This makes `createPlainTextToolCallCompatWrapper` return the underlying stream unchanged, but current providers use that exact helper as their opt-in to plain-text tool-call repair: Ollama returns it from `createOllamaStreamFn`, XAI adds it before tool streaming, and LM Studio stores it as `streamWithPlainTextToolCalls`. Since those call sites are not migrated to `wrapPlainTextToolCallStream`, this PR silently removes the repair path for providers that already opted in. Either leave this helper wrapping or update all existing callers to the new explicit wrapper in the same change.
  - confidence: 0.93
- **[P1] Keep label-plus-agent targeting valid:** `src/agents/tools/sessions-send-tool.ts:338-342`
  - body: This treats `agentId` as mutually exclusive with `label`, but current `sessions_send` uses `agentId` as a label qualifier: when `label` is present it passes `{ label, agentId }` to `sessions.resolve` after A2A policy checks. The gateway also has coverage for resolving sessions by both label and agent id. With this change, an existing cross-agent or disambiguated label send becomes a terminal validation error before resolution, so valid message delivery is blocked. Reject `sessionKey + agentId` if needed, but keep `label + agentId` working or replace it with an equivalent supported target form.
  - confidence: 0.91
- **[P1] Preserve tool support for existing providers:** `src/agents/model-tool-support.ts:17-20`
  - body: This provider allowlist flips any model without `compat.supportsTools` from tool-enabled to tool-disabled unless the provider is one of four ids. Existing bundled OpenAI-compatible providers such as Fireworks resolve dynamic models with `provider: "fireworks"`, `api: "openai-completions"`, and no compat override, while `buildOpenAICompletionsParams` only strips tools when `supportsModelTools(model)` is false. After this change those existing models stop receiving `params.tools`, so coding agents lose tool use even though the provider path previously supported it. Please either keep the old default for known bundled providers or add explicit `compat.supportsTools: true/false` coverage before making the global default fail-closed.
  - confidence: 0.89
