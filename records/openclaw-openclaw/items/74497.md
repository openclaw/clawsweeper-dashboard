## Summary

Keep open. Current main explains one likely symptom through the group/channel visible-reply default and a commenter reports the `messages.groupChat.visibleReplies: "automatic"` workaround on 2026.4.29+, but the requested Telegram group discovery/migration behavior is still not implemented: directory groups are config-backed, wildcard entries are intentionally omitted from directory output, and observed group-to-supergroup migrations only rewrite existing old-ID config.

## What This Changes

The issue requests Telegram group chats to be discovered, ingested, and visibly answered like direct messages, including `openclaw directory groups list --channel telegram` showing the group.

## Best Possible Solution

Define Telegram semantics for newly observed and migrated groups, then implement the smallest safe path in the Telegram plugin: expose the observed `-100...` ID through diagnostics or a read-only observed-groups surface, clarify `visibleReplies` in Telegram troubleshooting, and require explicit group authorization before normal replies are enabled.

## Reproduction Assessment

Yes. The issue provides concrete live steps and migration logs, and current main has matching code paths for private-by-default group replies, config-backed directory listing, and migration-only-when-old-config-exists behavior.

## Solution Assessment

Unclear. Setting `messages.groupChat.visibleReplies` to `automatic` is a valid workaround for visible final replies, but it is not a complete fix for group discovery or fresh migrated-group config semantics; a safer solution should separate observation from authorization.
