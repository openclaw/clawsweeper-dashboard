## Summary

Keep open. Current main still matches the reported Telegram group failure path: migration events are logged but do not create or authorize a new group entry when the old ID was never configured, directory group listing remains config-backed, and the discussion has multiple live confirmations that direct messages work while group chats fail.

## What This Changes

The issue asks OpenClaw to discover and ingest Telegram group chats, including newly created or migrated groups, so the bot can respond there while direct Telegram messages continue working.

## Best Possible Solution

Define the intended Telegram semantics for newly observed and migrated groups, then implement that policy in the Telegram plugin without granting reply authority from observation alone. A maintainable end state would either keep directories config-backed with clearer diagnostics for the new supergroup ID, or add a read-only observed-groups/doctor surface plus explicit config steps before replies are enabled.

## Reproduction Assessment

Yes. The issue gives concrete live setup steps and the exact migration/no-config log, and that log maps directly to the current migration handler and helper behavior on main.

## Solution Assessment

Unclear. Automatically discovering and enabling Telegram groups from observed traffic is not clearly the safest fix; the narrow maintainable solution needs an explicit policy for observed/migrated groups, directory visibility, and sender authorization.
