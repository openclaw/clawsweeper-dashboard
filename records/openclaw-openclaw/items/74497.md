## Summary

Keep open. Current main still matches the reported path: Telegram migration events are logged, but only existing configured group IDs are migrated; a fresh migrated group with no old-ID config remains unconfigured, the Telegram group directory is config-backed, and group ingress is still authorization-gated. The comments add multiple live confirmations that direct messages work while group behavior fails, so this needs maintainer policy and implementation rather than cleanup closure.

## What This Changes

The issue asks for Telegram group chats, including newly created or migrated groups, to be discovered by `openclaw directory groups list --channel telegram` and ingested/responded to like direct messages after Telegram privacy/admin setup.

## Best Possible Solution

Define explicit Telegram semantics for newly observed and migrated groups, then implement them in the Telegram plugin without granting reply authority from observation alone. A maintainable end state would expose the observed `-100...` ID through diagnostics or a read-only observed-groups surface, guide the user to add explicit `channels.telegram.groups` config, and preserve the existing group/sender authorization gates before replies are enabled.

## Reproduction Assessment

Yes. The issue gives concrete live setup steps and migration/no-config logs, and those logs map directly to the current migration handler and helper behavior on main.

## Solution Assessment

Unclear. The report identifies the right failure path, but automatically discovering and enabling Telegram groups is not clearly the safest solution; maintainers should choose between read-only discovery/diagnostics, guided config creation, and any explicit opt-in auto-repair behavior.
