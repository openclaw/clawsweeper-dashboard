## Summary

Keep open: current main still lacks the requested authoritative, capability-level approval resolver seam, and the closest related work only covers a narrower external plugin verification path. ## What This Changes

The issue requests a registerable, exclusive, cross-harness approval resolver API that gates side-effecting capabilities and binds approval decisions to exact tool requests.

## Best Possible Solution

Design a narrow, security-reviewed core/plugin SDK resolver contract that covers side-effect capabilities, exclusive routing, request digest/proof binding, and fail-closed behavior while leaving wallet or policy provider implementations outside core.

## Reproduction Assessment

Yes for the API gap: source inspection of current main shows `before_tool_call.requireApproval` and `plugin.approval.*` exist, but no registerable exclusive resolver, capability descriptor, params digest, or proof-binding seam. This is not a strict runtime bug reproduction because the requested behavior is new core/plugin API surface.

## Solution Assessment

Unclear pending maintainer security and API design. A core/plugin SDK seam is the right layer if external approvers must be authoritative across harnesses, but the capability taxonomy, resolver identity, proof storage, and upgrade contract need explicit design before implementation.
