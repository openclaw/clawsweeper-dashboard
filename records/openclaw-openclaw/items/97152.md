## Summary

Keep open: current main and v2026.6.10 still lack the requested authoritative, capability-level, exclusive approval-resolver seam, and related work only covers narrower plugin/Codex approval routes.

## What This Changes

Request a core/plugin SDK approval-resolver API that can exclusively gate side-effect capabilities across harnesses and bind external proof to exact approval requests.

## Best Possible Solution

Design a narrow, security-reviewed core/plugin SDK resolver contract for capability classification, exclusive resolver identity, request digest/proof binding, and fail-closed behavior while keeping wallet or policy providers external.

## Reproduction Assessment

Yes for the source-level API gap: current main has `before_tool_call.requireApproval` and `plugin.approval.*`, but no registerable exclusive resolver, capability descriptor, params digest, or proof-binding seam. This is not a strict runtime bug reproduction because the requested behavior is new core/plugin API surface and some bypass evidence is private.

## Solution Assessment

Unclear pending maintainer security and API design. A core/plugin SDK seam is the right layer if external approvers must be authoritative across harnesses, but the capability taxonomy, resolver identity, proof storage, and upgrade contract need explicit design before implementation.
