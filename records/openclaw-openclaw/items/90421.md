## Summary

Keep open: the latest revision fixes the earlier read-scope mutation concern, but the patch is still not merge-ready because stale status is only derived for list results and the Control UI immediately loads inspect detail, which can replace the stale list status with the persisted pending record.

## What This Changes

The PR adds read-only Skill Workshop list-time stale detection for create/update proposals, refactors stale marking into a helper, and adds focused service tests plus a standalone proof script.

## Best Possible Solution

Keep list read-only, but make the stale read projection consistent across list and inspect/detail or preserve the derived status in the UI, then add focused coverage for the list-to-detail path and provide visible UI or gateway proof.

## Reproduction Assessment

Yes. Source inspection of current main shows listSkillProposals returns persisted manifest status without target-state reconciliation, and the linked issue describes a create proposal whose target skill was manually installed later.

## Solution Assessment

No. The read-only direction is good, but the current implementation is too narrow because inspect/detail still returns pending and the UI consumes inspect immediately after list.
