## Summary

Keep open with a state-convergence blocker: the PR fixes the visible Skill Workshop waiting-state symptom for list/detail reads, but it can return `stale` while the stored proposal remains `pending`, so write-side quota and direct lifecycle paths can still treat the same proposal as pending.

## What This Changes

The branch adds read-only stale status derivation for Skill Workshop list/inspect when create targets already exist or update targets are missing, plus service tests for those scenarios.

## Best Possible Solution

Use one canonical Skill Workshop stale-state path so list, inspect, lifecycle actions, and pending-limit checks all classify the same proposal the same way, then choose whether this PR or the overlapping PR should carry the final fix.

## Reproduction Assessment

Yes. Current main's scoped list/inspect paths do not recheck target skill existence, and the UI renders actions for returned `pending` proposals, so the linked waiting-state behavior is source-reproducible.

## Solution Assessment

No, not yet. Deriving stale in the shared read paths is a good narrow part of the fix, but the PR should not leave write-side quota and direct lifecycle paths using a different pending/stale truth unless maintainers explicitly accept that split.
