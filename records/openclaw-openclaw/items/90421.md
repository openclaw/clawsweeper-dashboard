## Summary

Keep this PR open, but do not merge it as-is. The fix targets the right Skill Workshop service path and covers the reported stale waiting state, but the new list-time reconciliation can overwrite a concurrent terminal proposal status and the external PR still has only test-output proof.

## What This Changes

The PR adds list-time reconciliation for Skill Workshop proposals so pending create/update proposals become stale when the target skill already exists or is missing, with focused service tests.

## Best Possible Solution

Keep the list-time reconciliation, but serialize it with the existing proposal target lock and re-read that the record is still pending immediately before marking stale, then add real gateway/UI proof.

## Reproduction Assessment

Yes. Current main’s `listSkillProposals({ workspaceDir })` returns pending manifest entries without checking target skill presence, while the gateway/UI consume that status directly; I did not execute tests because this was a read-only review.

## Solution Assessment

No as written. List-time reconciliation is the right narrow layer for the reported stale waiting state, but the write must be locked and conditional so it does not replace a concurrent terminal status.

## Review Findings

Overall confidence: 0.84

Full review comments:

- **[P2] Serialize reconciliation before marking stale:** `src/skills/workshop/service.ts:1000`
  - body: `listSkillProposals` now mutates proposal records from a read path, but this stale write is outside the target lock used by apply/revise/reject and it does not re-read the record before overwriting it. If a create proposal is being applied while the UI lists proposals, the list can read the old pending record, observe the just-created skill file, and write `stale` after `applySkillProposal` has written `applied`, leaving a successfully installed proposal misclassified. Take the existing target lock and re-read that the record is still pending before calling `markProposalStale`.
