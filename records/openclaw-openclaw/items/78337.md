## Summary

Keep open: current main confirms that remote browser CDP URLs are documented as credential-bearing secret material, but `browser.cdpUrl` and `browser.profiles.*.cdpUrl` are still string-only raw config fields and are absent from the SecretRef target surface.

## What This Changes

The issue requests SecretRef support for browser CDP URLs so remote-browser credentials do not need to be stored inline in `openclaw.json`.

## Best Possible Solution

Add CDP URL support through the existing SecretInput/SecretRef runtime contract, preferably resolving the whole CDP URL before browser config normalization, and document the supported shape instead of introducing a new embedded interpolation grammar unless maintainers explicitly want that API.

## Reproduction Assessment

Yes, at source level. Current main rejects structured SecretRefs for the CDP URL schema, omits these paths from the SecretRef target surface, and passes literal placeholder strings through URL parsing unchanged.

## Solution Assessment

No for literal `${secret:...}` substring interpolation as the first implementation path. The narrower maintainable solution is to reuse OpenClaw's existing whole-value SecretInput contract for `browser.cdpUrl` and `browser.profiles.*.cdpUrl`, with docs/tests covering full CDP URL secrets.
