## Summary

Keep open: current main still treats `browser.cdpUrl` and `browser.profiles.*.cdpUrl` as raw URL strings, while the SecretRef surface neither supports nor explicitly excludes those credential-bearing paths. The report is valid, but the fix needs an owner-approved security/config contract rather than a cleanup close.

## What This Changes

The issue requests SecretRef support or explicit documentation for credential-bearing browser CDP URLs.

## Best Possible Solution

Define an owner-approved CDP URL credential contract, then update browser config resolution, schema and SecretRef registry metadata, docs, redaction expectations, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `browser.profiles.<name>.cdpUrl` with a SecretRef-like placeholder and `resolveProfile` will parse that literal string, while structured SecretRef objects are rejected by schema.

## Solution Assessment

Unclear. The gap is real, but adding ad hoc URL-substring interpolation is not obviously the best contract; maintainers should choose between structured SecretInput support, explicit interpolation semantics, or documented unsupported behavior.
