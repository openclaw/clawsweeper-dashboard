## Summary

Keep open: current main still treats `cdpUrl` as raw string browser config, while the SecretRef surface excludes that path. The report is valid, but the requested inline interpolation expands secret/config semantics and needs browser plus secops owner judgment before implementation.

## What This Changes

Request support for SecretRef-backed remote CDP URLs so Browserbase or Browserless API keys do not have to be stored in plaintext `openclaw.json`.

## Best Possible Solution

Define an approved CDP URL SecretRef contract, then update schema, runtime resolution, SecretRef audit/docs, redaction coverage, and regression tests together; if rejected, document `cdpUrl` as unsupported explicitly.

## Reproduction Assessment

Yes, at source level: configure `browser.profiles.<name>.cdpUrl` with a SecretRef-like placeholder and `resolveProfile` will parse the raw string rather than resolve it. I did not run a live Browserbase or Browserless connection in this read-only review.

## Solution Assessment

Unclear: adding substring interpolation exactly as proposed is not obviously the best contract because current SecretRefs are structured or whole-value env templates. A safer path is an owner-approved CDP URL SecretRef surface, or explicit documentation if maintainers reject support.
