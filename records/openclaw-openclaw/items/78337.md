## Summary

Keep open: current main and the latest release still treat browser CDP URLs as string-only raw config outside SecretRef resolution, so the reported credential-at-rest gap remains and no merged canonical PR owns it.

## What This Changes

Request support or explicit documentation for SecretRef-backed remote CDP URLs in browser profile config.

## Best Possible Solution

Add a maintainer-approved SecretRef contract for `browser.cdpUrl` and `browser.profiles.*.cdpUrl`, likely structured whole-field `SecretInput`, with schema/types/runtime collectors/docs/tests; if rejected, document the unsupported status explicitly.

## Reproduction Assessment

Yes, by source inspection: current main accepts CDP URLs only as strings, resolves them directly in the browser profile resolver, and does not collect browser config for SecretRef resolution. I did not run a live remote CDP service repro in this read-only review.

## Solution Assessment

Unclear as requested: partial placeholder interpolation is not a current OpenClaw SecretRef contract. A structured whole-field SecretInput expansion is the narrower maintainable path unless maintainers explicitly approve URL-substring interpolation.
