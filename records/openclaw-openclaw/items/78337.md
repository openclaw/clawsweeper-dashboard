## Summary

Keep open: current main still treats browser CDP URLs as raw string config, the SecretRef target surface omits `browser.cdpUrl` and `browser.profiles.*.cdpUrl`, and adding support would change a security-sensitive SecretRef contract that needs owner review.

## What This Changes

The issue requests SecretRef support or explicit documentation for `browser.profiles.*.cdpUrl` so remote CDP tokens do not have to be stored inline in config.

## Best Possible Solution

Define an owner-approved CDP URL secret contract, then update the browser schema/runtime resolution, SecretRef registry/docs, and focused regression coverage together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a profile `cdpUrl` with an inline SecretRef-like placeholder and `resolveProfile` will parse that placeholder as part of the raw URL string rather than resolving it.

## Solution Assessment

Unclear. Adding substring interpolation exactly as proposed is not obviously the best contract; a safer direction is an owner-reviewed SecretInput-compatible CDP URL design, or explicit unsupported-surface docs if maintainers reject that expansion.
