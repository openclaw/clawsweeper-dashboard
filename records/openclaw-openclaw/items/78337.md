## Summary

Keep open: current main still treats CDP URLs as raw string config, and the SecretRef surface does not include these paths. The requested inline interpolation would expand current SecretRef semantics, so this needs browser/secret-surface owner review rather than an automatic close.

## What This Changes

The issue requests SecretRef-backed secret handling for `browser.cdpUrl` and `browser.profiles.*.cdpUrl` so remote CDP tokens do not have to be stored inline in `openclaw.json`.

## Best Possible Solution

Define an owner-approved CDP URL SecretRef contract, then update the browser config schema/runtime resolution, SecretRef audit/docs, and regression coverage together.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure a profile `cdpUrl` with an inline SecretRef-like placeholder and `resolveProfile` will parse the raw string rather than resolving it. I did not run a live Browserbase/Browserless connection because this review is read-only.

## Solution Assessment

Unclear: adding interpolation exactly as proposed is not obviously the best fix because current SecretRefs are structured or whole-value env templates. A safer direction is to extend the existing SecretInput contract for CDP URL credentials, or explicitly document the field as unsupported if maintainers reject that expansion.
