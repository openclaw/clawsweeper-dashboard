## Summary

Keep open: current main still treats `browser.cdpUrl` and `browser.profiles.*.cdpUrl` as raw string URL config, while the SecretRef surface does not define these paths. The report is valid, but choosing URL substring interpolation, structured whole-value SecretRefs, or explicit unsupported documentation is a security-sensitive contract decision.

## What This Changes

Support SecretRef-managed credentials in browser CDP URL config instead of requiring API keys to be embedded in `openclaw.json`.

## Best Possible Solution

Define an owner-approved CDP URL credential contract, then update browser config resolution, schema and SecretRef registry metadata, docs, redaction expectations, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `browser.profiles.<name>.cdpUrl` with a SecretRef-like placeholder and `resolveProfile` will parse that literal string, while structured SecretRef objects are rejected by schema.

## Solution Assessment

Unclear. The gap is real, but adding ad hoc URL-substring interpolation is not obviously the best contract; a browser/secops decision should choose between structured SecretInput support, explicit interpolation semantics, or documented unsupported behavior.
