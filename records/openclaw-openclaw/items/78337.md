## Label Justifications

- `P2`: This is a concrete credential-handling gap with limited surface area and no evidence of an active outage or exploit.
- `impact:security`: The report concerns avoiding long-lived remote CDP credentials being stored as plaintext URL material in `openclaw.json`.
- `impact:auth-provider`: The affected behavior is SecretRef credential resolution for configured remote CDP authentication URLs.

## Summary

Keep open: current main and the latest release still treat browser profile CDP URLs as raw string config outside SecretRef resolution, and the open Browserbase driver PR is only partial overlap because it explicitly leaves SecretRef surface expansion out of scope.

## What This Changes

The issue requests SecretRef support or explicit unsupported-surface documentation for browser profile CDP URLs so remote CDP tokens do not have to be stored as plaintext URL material in config.

## Best Possible Solution

Maintainers should either approve a canonical SecretRef-supported contract for browser CDP credentials with schema/runtime collector/docs/tests, or explicitly document `browser.profiles.*.cdpUrl` as unsupported and point users to a safe approved alternative.

## Reproduction Assessment

Yes. Source inspection of current main shows `browser.profiles.*.cdpUrl` is a string-only config field and `resolveProfile()` consumes it as raw URL text without SecretRef collection or resolution.

## Solution Assessment

Unclear. SecretRef support is likely the right end state, but maintainers need to choose the credential contract before implementation because partial string interpolation, structured SecretInput, and provider-specific fields have different security semantics.
