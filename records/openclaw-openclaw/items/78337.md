## Summary

Keep open: current main still treats `browser.cdpUrl` and `browser.profiles.*.cdpUrl` as raw URL strings, and the canonical SecretRef surface does not define either path. The report is valid, but the fix needs an owner decision on the CDP URL credential contract rather than an automatic one-off interpolation patch.

## What This Changes

Request SecretRef support, or explicit unsupported documentation, for credential-bearing browser CDP URL config fields so remote CDP tokens do not need plaintext in `openclaw.json`.

## Best Possible Solution

Define an owner-approved CDP URL credential contract, then update browser config resolution, schema/registry metadata, docs, redaction expectations, and focused regression coverage together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `browser.profiles.<name>.cdpUrl` with a SecretRef-like placeholder and `resolveProfile` will parse that literal string, while structured SecretRef objects are rejected by schema.

## Solution Assessment

Unclear. The gap is real, but ad hoc URL-substring interpolation is not clearly the best contract; a browser/SecretRef owner should choose between structured SecretInput support, explicit interpolation semantics, or documented unsupported behavior.
