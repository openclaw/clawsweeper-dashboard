## Label Justifications

- `P2`: This is a bounded but real credential-handling gap in browser CDP configuration, not a core runtime outage.
- `impact:security`: The issue concerns avoiding long-lived remote CDP credentials being stored as plaintext URL material in config.
- `impact:auth-provider`: The affected path is a credential-bearing provider/browser attachment URL that currently bypasses SecretRef resolution.

## Summary

Keep open: current main and the latest release still treat browser CDP URLs as raw string config outside SecretRef resolution, so the credential-at-rest gap remains and no merged canonical item owns it.

## What This Changes

The issue asks for `browser.profiles.*.cdpUrl` to support SecretRef-backed remote CDP credentials instead of requiring plaintext URL tokens in `openclaw.json`.

## Best Possible Solution

Maintainers should choose a structured SecretInput or explicitly approved URL-template contract for browser CDP credentials, then update schema/types, runtime SecretRef collection, docs, and regression coverage; if declined, document the unsupported status clearly.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `browser.profiles.*.cdpUrl` is schema-typed as a string and `resolveProfile` consumes the raw string without SecretRef collection or resolution.

## Solution Assessment

Unclear. SecretRef support is likely the right end state, but maintainers need to choose whether this should be a structured whole-field SecretInput, an explicit URL-template contract, or a Browserbase-specific credential path.
