- `impact:security`: The issue is about avoiding long-lived remote CDP credentials being stored or handled as plaintext config values.
- `impact:auth-provider`: The requested fix depends on SecretRef resolution for a credential-bearing browser/provider URL path.

## Summary

Keep open: current main and v2026.6.6 still treat browser CDP URLs as raw strings, omit them from the SecretRef surface, and the related Browserbase PR explicitly leaves SecretRef expansion out of scope.

## What This Changes

The issue requests SecretRef support for browser CDP URLs so remote browser API keys do not have to live in plaintext inside `openclaw.json`.

## Best Possible Solution

Add a maintainer-approved SecretRef contract for CDP URLs, preferably by extending the existing structured SecretInput path for `browser.cdpUrl` and `browser.profiles.*.cdpUrl`, then update schema/types, runtime collection, docs, and focused tests without inventing ad hoc placeholder interpolation unless that broader syntax is explicitly approved.

## Reproduction Assessment

Yes, by source inspection: current main accepts CDP URLs only as strings, resolves them directly in the browser profile resolver, and does not collect those paths for SecretRef resolution. I did not run a live remote CDP service repro in this read-only review.

## Solution Assessment

Not as written: the requested `${secret:...}` / `{{SECRET:...}}` interpolation syntax is not an existing OpenClaw SecretRef contract. A structured whole-field SecretRef expansion is the narrower maintainable path unless maintainers intentionally approve partial-string interpolation.
