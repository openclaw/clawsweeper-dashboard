## Summary

Keep open: current main still treats browser CDP URLs as raw string config, while the SecretRef supported/unsupported surface still does not define a contract for `browser.cdpUrl` or `browser.profiles.*.cdpUrl`. The report is valid, but the fix expands security-sensitive SecretRef semantics and needs owner review rather than automatic cleanup.

## What This Changes

The issue requests SecretRef support for secret-bearing browser CDP URLs so remote browser API keys do not need to be stored in cleartext config.

## Best Possible Solution

Define an owner-approved CDP URL credential contract, then update browser config resolution, schema/SecretRef registry metadata, docs, redaction expectations, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure a profile `cdpUrl` with an inline SecretRef-like placeholder and `resolveProfile` will parse and use that literal string rather than resolving it first.

## Solution Assessment

Unclear. The reported gap is real, but adding substring interpolation exactly as suggested is not obviously the narrowest maintainable fix because current SecretRefs are structured or whole-value env templates.
