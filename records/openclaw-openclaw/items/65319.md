## Summary

Keep open. Current main improves diagnostics for known manifest command aliases, but still has an ownerless fallback that recommends adding command tokens to `plugins.allow`, and alias lookup does not normalize leading slash inputs like `/lossless`; the maintainer discussion also explicitly left this open as adjacent to #64732.

## What This Changes

The issue requests non-contradictory CLI/config guidance for lossless-claw command aliases under plugin-id-only `plugins.allow`.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize slash-prefixed alias lookup, and harden the ownerless missing-command fallback so command names are never recommended as allowlist entries.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `resolveMissingPluginCommandMessage` with `lcm` or `/lossless` under a restrictive `plugins.allow` and no matching alias owner, and current main reaches the fallback that recommends the command token while config validation treats it as a non-plugin.

## Solution Assessment

No. The current manifest-backed alias handling is the right direction for known aliases, but it is incomplete for slash-prefixed or unresolved command tokens; a narrow diagnostics fix is safer than allowing command ids in `plugins.allow`.
