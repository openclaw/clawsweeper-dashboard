## Summary

Current main already fixes the contradictory `plugins.allow` guidance: unowned `/lossless`/`lcm`-style roots now fall through to an unknown-command diagnostic, while real plugin-owned CLI metadata still points users at the owning plugin ID. The fix is on main via the merged CLI diagnostic PR, but it is not in the latest tagged release I could verify.

## What This Changes

The issue asks for `/lossless`/`lcm` CLI allowlist diagnostics to stop telling users to add command tokens to `plugins.allow` when validation only accepts plugin IDs.

## Best Possible Solution

Keep `plugins.allow` plugin-ID-only and keep the current command ownership diagnostics; any future Lossless Claw root CLI should be advertised through plugin CLI metadata rather than command tokens in the allowlist.

## Reproduction Assessment

No, not on current main. Source inspection shows unowned roots now fall through to the no-owner unknown-command diagnostic, and regression tests assert unknown roots under restrictive allowlists do not mention `plugins.allow`; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: keep command tokens out of `plugins.allow`, resolve real plugin-owned CLI roots to their owning plugin IDs, and let unknown roots use normal unknown-command diagnostics.
