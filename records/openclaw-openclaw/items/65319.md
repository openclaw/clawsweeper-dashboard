## Summary

Current main fixes the contradictory allowlist guidance: unowned command roots now fall through to an unknown-command diagnostic, while plugin-owned CLI metadata still points at the owning plugin ID.

## What This Changes

The issue reports contradictory `plugins.allow` guidance for plugin command aliases such as `/lossless` and `lcm`, where runtime diagnostics ask for command tokens while config validation accepts only plugin IDs.

## Best Possible Solution

Keep `plugins.allow` plugin-ID-only and keep the current command ownership diagnostics; any future Lossless Claw root CLI should be advertised through plugin CLI metadata rather than command tokens in the allowlist.

## Reproduction Assessment

No current-main failing path remains by source inspection: an unowned `/lossless`-style root now falls through to unknown-command handling, and the regression test asserts the restrictive-allowlist diagnostic does not mention `plugins.allow`. I did not run the CLI because this review is read-only, but the source and tests directly cover the reported contradiction.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: keep command tokens out of `plugins.allow`, resolve real plugin-owned CLI roots to their owning plugin IDs, and let unknown roots use normal unknown-command diagnostics.
