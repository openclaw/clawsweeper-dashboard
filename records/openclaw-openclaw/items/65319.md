## Summary

Keep open. Current main has manifest-backed command-alias diagnostics, but the reported contradiction still exists for unresolved or slash-prefixed command tokens: CLI fallback can recommend adding the command token to `plugins.allow`, while config validation still treats the same token as a stale plugin id. Maintainer discussion also explicitly left this issue open as adjacent to #64732 rather than resolved.

## What This Changes

The issue asks OpenClaw to stop giving contradictory `plugins.allow` guidance for `lossless-claw` command surfaces such as `/lossless` and `lcm`.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize command-alias lookup for slash-prefixed tokens, and make the ownerless missing-command fallback avoid telling users to add command names to `plugins.allow`. If the current lossless-claw package does not declare the needed aliases, coordinate that manifest update separately after the core guard is fixed.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `resolveMissingPluginCommandMessage` with `/lossless` or `lcm`, a restrictive `plugins.allow` that includes `lossless-claw`, and no matching manifest alias; current main reaches the fallback that recommends the command token, while validation would warn that the token is not a plugin.

## Solution Assessment

No. The current manifest-backed handling is the right direction for known aliases, but it is incomplete for slash-prefixed or unresolved command tokens; the safer fix is a narrow core diagnostic hardening plus targeted tests, not changing `plugins.allow` to accept command ids.
