## Summary

Keep open. Current main has partial manifest-backed command-alias diagnostics, but the reported contradiction can still occur for unresolved or slash-prefixed command tokens: the CLI fallback can still recommend adding the command token to `plugins.allow`, while validation treats unknown command tokens as stale plugin ids. The maintainer discussion also explicitly left this issue open as adjacent to #64732 rather than closing it as a duplicate.

## What This Changes

The issue asks OpenClaw to stop giving contradictory `plugins.allow` guidance for `lossless-claw` command surfaces such as `/lossless` and `lcm`.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize command-alias lookup for slash-prefixed runtime command tokens, and make the ownerless missing-command fallback avoid recommending command names as allowlist entries. If `lossless-claw` lacks the expected manifest aliases, handle that as a plugin manifest follow-up after the generic core diagnostic guard is fixed.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: with a restrictive `plugins.allow`, a command token such as `/lossless` that does not resolve to a manifest alias reaches the fallback in `src/cli/run-main-policy.ts` and can be recommended as a `plugins.allow` entry even though validation will reject it as a plugin id.

## Solution Assessment

No current solution on main is complete. The manifest-backed alias handling is the right direction, but it needs slash-token normalization and a non-contradictory ownerless fallback to cover the reported path.
