## Summary

Keep open. Current main has manifest-backed improvements for known command aliases, but the reported contradiction still exists for slash-prefixed or ownerless command tokens because the CLI fallback can still recommend adding the command token to `plugins.allow` while config validation remains plugin-id-only.

## What This Changes

The issue asks OpenClaw to stop giving contradictory `plugins.allow` guidance for `openclaw /lossless` or `openclaw lcm` when the owning plugin is `lossless-claw`.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize slash-prefixed alias lookup, and make the ownerless missing-command fallback avoid recommending command names as allowlist entries.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `resolveMissingPluginCommandMessage` with `lcm` or `/lossless` under a restrictive `plugins.allow` and no matching alias owner, and current main reaches the fallback that recommends the command token while config validation treats it as a non-plugin.

## Solution Assessment

No. The current manifest-backed alias handling is the right direction for known aliases, but it is incomplete for slash-prefixed or unresolved command tokens; a narrow diagnostics fix is safer than allowing command ids in `plugins.allow`.
