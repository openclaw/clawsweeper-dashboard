## Summary

Current main already removes the contradictory allowlist guidance: unowned command roots fall through to an unknown-command diagnostic, while plugin-owned CLI metadata still points at the owning plugin ID.

## What This Changes

The issue asks OpenClaw to stop telling users to add `/lossless` or `lcm` command tokens to `plugins.allow` when that config accepts plugin IDs such as `lossless-claw`.

## Best Possible Solution

Keep `plugins.allow` plugin-ID-only and keep command ownership diagnostics metadata-driven; future Lossless Claw root CLI support should be declared through plugin CLI metadata rather than command tokens in the allowlist.

## Reproduction Assessment

No current-main failing path remains by source inspection: unowned command roots now fall through to the unknown-command path, and regression tests assert the restrictive-allowlist diagnostic does not mention `plugins.allow`. This read-only review did not execute the CLI or tests.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: keep command tokens out of `plugins.allow`, resolve real plugin-owned CLI roots to their owning plugin IDs, and let unknown roots use normal unknown-command diagnostics.
