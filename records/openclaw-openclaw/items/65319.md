## Summary

Keep open. Current main improved manifest-backed command alias diagnostics, but the source still has the contradictory fallback for unresolved command tokens and slash-prefixed aliases; the maintainer discussion also explicitly left this issue open as adjacent to #64732 rather than resolved.

## What This Changes

The issue asks OpenClaw to stop telling users to add `/lossless` or `lcm` command tokens to `plugins.allow` when that allowlist is documented and validated as plugin-id-only.

## Best Possible Solution

Keep plugins.allow plugin-id-only, normalize slash-prefixed alias lookup, and make the ownerless missing-command fallback avoid telling users to add command names to plugins.allow, with targeted regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call resolveMissingPluginCommandMessage with /lossless or lcm under a restrictive plugins.allow and no matching alias owner, and current main reaches the fallback that recommends the command token while config validation remains plugin-id-only.

## Solution Assessment

No. The current manifest-backed alias handling is the right direction for known aliases, but it is incomplete for slash-prefixed or unresolved command tokens; a narrow diagnostics hardening is safer than changing plugins.allow to accept command ids.
