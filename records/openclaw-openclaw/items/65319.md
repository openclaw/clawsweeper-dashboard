## Summary

Keep open. Current main still has a source-level path where slash-prefixed or ownerless plugin command tokens fall through to guidance that tells users to add the command token to `plugins.allow`, while the config contract remains plugin-id-only.

## What This Changes

The issue asks OpenClaw to stop giving contradictory `plugins.allow` guidance for `lossless-claw` command surfaces such as `/lossless` and `lcm`.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize slash-prefixed command aliases, and avoid recommending unknown command tokens as allowlist entries; add lossless-claw manifest metadata separately if needed.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `/lossless` is normalized without stripping the slash, alias lookup misses a plain alias, and the fallback recommends adding the command token even though config validation treats command tokens as non-plugins.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostics and alias normalization while preserving the plugin-id-only allowlist contract; accepting command ids in `plugins.allow` would conflict with current schema/help and validation behavior.
