## Summary

Keep open. Current main still has a source-level path where an unresolved command token under a restrictive `plugins.allow` falls back to telling users to add that token to `plugins.allow`, while config/schema guidance keeps `plugins.allow` plugin-id-only; the maintainer discussion also left the lossless-claw variant open after the narrower #64732 fix.

## What This Changes

The issue asks OpenClaw to stop giving contradictory `plugins.allow` guidance for `openclaw /lossless` and related lossless-claw command aliases.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize slash-prefixed command aliases before owner lookup, and harden missing-command diagnostics so command names are never recommended as allowlist entries.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `resolveMissingPluginCommandMessage` for `/lossless` or `lcm` with a restrictive `plugins.allow` when no owner resolves, and current main reaches the fallback that recommends the command token while config validation treats command tokens as non-plugins.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostics and alias normalization; accepting command ids in `plugins.allow` would conflict with the documented plugin-id contract.
