## Summary

Keep open. Current main still has the contradictory ownerless CLI fallback for command tokens and no leading-slash alias normalization, while the maintainer discussion explicitly left this issue open as adjacent to #64732 rather than resolved.

## What This Changes

The issue reports contradictory diagnostics where plugin command tokens like `/lossless` or `lcm` are recommended for `plugins.allow` even though validation treats that allowlist as plugin-id-only.

## Best Possible Solution

Keep plugins.allow plugin-id-only, normalize slash-prefixed alias lookup, and make the ownerless missing-command fallback avoid recommending command names for plugins.allow, with targeted regression tests and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call resolveMissingPluginCommandMessage with /lossless or lcm under restrictive plugins.allow when no alias owner resolves, and current main reaches the fallback that recommends the command token while config validation remains plugin-id-only.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostics and alias normalization; accepting command ids in plugins.allow would weaken the current plugin-id-only contract and broaden the config surface unnecessarily.
