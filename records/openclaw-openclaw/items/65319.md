## Summary

Keep open. Current main has better manifest-declared command-alias handling, but the reported contradiction still remains for slash-prefixed or ownerless command tokens: the CLI fallback can tell users to add the command token to `plugins.allow`, while config validation keeps `plugins.allow` plugin-id-only and warns on unknown command tokens. The maintainer discussion also explicitly left this open as adjacent to #64732 rather than resolved.

## What This Changes

Fix contradictory CLI and config guidance so `/lossless` and `lcm` command surfaces under restrictive `plugins.allow` point to the owning plugin ID or the correct runtime surface instead of recommending command IDs.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize command-alias lookup for slash-prefixed tokens, and harden the ownerless missing-command fallback so command-shaped tokens do not tell users to add themselves to `plugins.allow`. If the current lossless-claw package lacks the needed `commandAliases`, handle that as a separate plugin manifest follow-up after the core diagnostics path is safe.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `resolveMissingPluginCommandMessage` with `/lossless` or unresolved `lcm` plus `plugins.allow: ["lossless-claw"]` reaches the fallback that recommends the command token, while validation treats that same token as a non-plugin unless alias ownership resolves.

## Solution Assessment

No for current main. Manifest-backed alias diagnostics are the right direction for known aliases, but they are incomplete for slash-prefixed or unresolved command tokens; the safer fix is narrow core diagnostic hardening plus targeted tests, not changing `plugins.allow` to accept command IDs.
