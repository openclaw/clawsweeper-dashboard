## Summary

Keep open. Current main improves known manifest command aliases, but `/lossless` can still miss alias ownership because lookup does not strip a leading slash, and the ownerless fallback still recommends adding the command token to `plugins.allow`; the maintainer discussion also explicitly left this issue open adjacent to #64732.

## What This Changes

The issue asks for non-contradictory CLI and config guidance for lossless-claw command aliases while keeping `plugins.allow` plugin-id-only.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize slash-prefixed command tokens before alias ownership checks, and harden missing-command diagnostics so command names are never recommended as allowlist entries.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `resolveMissingPluginCommandMessage` with `/lossless` or `lcm` under a restrictive `plugins.allow` when no owner resolves, and current main reaches the fallback that recommends the command token while config validation treats command tokens as non-plugins.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostics and alias normalization; accepting command ids in `plugins.allow` would conflict with the documented plugin-id contract.
