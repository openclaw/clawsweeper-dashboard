## Summary

Keep open: this draft feature PR is plausible, but it lacks after-fix real behavior proof and has a blocking correctness issue where accepted retry settings can exceed cron's setup watchdog before the provider wake window completes.

## What This Changes

The PR adds `cron.modelPreflight` timeout/retry config, applies it to isolated cron local-provider preflight, documents the settings, and adds config plus runtime unit coverage.

## Best Possible Solution

Make preflight retry semantics fit cron's setup timeout and abort model, prove the behavior with a real local-provider wake-up run, and keep fallback-chain handling tracked separately in https://github.com/openclaw/openclaw/issues/79329.

## Reproduction Assessment

Yes for the review finding: a valid config whose total preflight window exceeds 60s can be traced through source to the setup watchdog firing before `onExecutionStarted`. I did not run a live Ollama cold-start reproduction.

## Solution Assessment

No as written. A retry config can be useful, but the patch should first bound or coordinate the retry window with cron's watchdog/abort behavior so accepted settings reliably do what the docs promise.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep preflight retry windows under the setup watchdog:** `src/config/zod-schema.ts:713-715`
  - body: The schema accepts `timeoutMs: 60000`, `maxAttempts: 10`, and `retryDelayMs: 60000`, but isolated cron's setup watchdog fires after 60s until `onExecutionStarted` is called, and model preflight runs before that callback. A valid config such as two 60s attempts, or three attempts with 30s delays, can time out the cron setup before the sleeping provider gets the advertised wake window. Bound the total preflight window or make the watchdog/preflight abort path understand this pre-execution wait.
  - confidence: 0.86
