## Summary

Keep open: the PR adds a plausible retry knob for local-provider cron preflight, but it is draft, lacks after-fix real behavior proof, and currently accepts retry windows that can exceed cron's setup watchdog before the run is marked started.

## What This Changes

The PR adds `cron.modelPreflight` timeout, max-attempt, and retry-delay config, wires it into isolated cron local-provider preflight retries, updates docs/help labels, and adds config plus runtime tests.

## Best Possible Solution

Make preflight retry semantics fit cron's setup timeout and abort model, prove the behavior with a real local-provider wake-up run, and keep fallback-chain handling tracked in https://github.com/openclaw/openclaw/issues/79329.

## Reproduction Assessment

Yes for the review finding: a valid config with a total retry window over 60s can be traced through source to preflight running before `onExecutionStarted` while the setup watchdog is active. I did not run a live Ollama cold-start reproduction.

## Solution Assessment

No as written. A retry config may be useful, but the patch should first bound or coordinate the retry window with cron's setup watchdog and abort behavior so accepted settings match the documented wake-up window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep preflight retry windows under the setup watchdog:** `src/config/zod-schema.ts:712-714`
  - body: The schema accepts `timeoutMs: 60000`, `maxAttempts: 10`, and `retryDelayMs: 60000`, but isolated cron keeps a 60s setup watchdog active until `onExecutionStarted`, and this preflight runs before that callback. A valid config such as two 60s attempts can time out setup before the provider gets the advertised wake window, so bound the total retry window or make the watchdog/preflight abort path understand this wait.
  - confidence: 0.87
