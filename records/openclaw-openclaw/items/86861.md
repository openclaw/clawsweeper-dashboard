## Label Justifications

- `P2`: The report describes a scheduled-task reliability regression under provider rate limits, now addressed on current main.
- `impact:auth-provider`: The affected behavior is provider credential cooldown tracking and auth-profile failure policy for cron runs.

## Summary

Current main fixes the central cron-specific false cooldown path by keeping cron-originated overloaded and transient rate-limit failures local to the run while preserving shared handling for real credential failures; the fix is not in the latest stable release yet.

## What This Changes

The issue reports that cron-triggered tasks in OpenClaw 5.20+ can fail when credential cooldown waiting consumes the task timeout.

## Best Possible Solution

Keep the current main cron-local transient auth failure policy, and leave broader reason-aware cron guardrails to https://github.com/openclaw/openclaw/issues/14376.

## Reproduction Assessment

Yes at source level: pre-fix cron embedded runs used the shared/default auth failure policy, and current main now proves the cron call site forwards the local-transient policy. A live MiniMax cron reproduction was not run in this read-only review.

## Solution Assessment

Yes for the cron-specific failure: reusing the existing auth-profile failure-policy seam is the narrow owner-boundary fix and avoids adding a new timeout or config surface. Broader recurring guardrail policy remains separate.
