## Summary

Keep open. Current main has memory recall/read tooling plus read and bootstrap injection caps, but it still lacks the requested tool-managed bounded memory write/CRUD surface with hard per-store character budgets; draft #70540 is an unmerged partial memory_add step.

## What This Changes

Add a built-in bounded memory write tool with add/replace/remove semantics, hard configurable per-store character limits, duplicate handling, frozen startup snapshots, and optional credential screening.

## Best Possible Solution

Define a memory-core-owned CRUD/write API with config-backed store budgets, atomic add/list/replace/remove behavior, duplicate and credential handling, prompt/docs updates, and regression coverage coordinated with #70540.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and docs inspection show current main exposes memory_search/memory_get plus read and bootstrap caps, not a bounded memory write path.

## Solution Assessment

Unclear pending maintainer design. A built-in memory-core tool is plausible because the request depends on hard enforcement, but the store model, budget semantics, duplicate handling, and credential-screening contract should be settled before implementation.
