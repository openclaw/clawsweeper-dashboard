## Summary

Keep open. Current main and the latest release still expose recall-only memory tools; the bounded code on main is limited to dreaming promotion compaction and does not provide the requested hard-limited add/replace/remove write contract or credential-safety policy.

## What This Changes

The issue requests a built-in bounded memory write tool with hard per-store character limits, add/replace/remove operations, duplicate handling, snapshot injection, and optional credential-pattern blocking.

## Best Possible Solution

Design a memory-core or plugin-SDK owned bounded CRUD contract, including hard-limit semantics, duplicate/replace/remove identity, docs, tests, and secret-safety behavior, before implementation.

## Reproduction Assessment

Yes for the source-level capability gap: current main exposes only `memory_search` and `memory_get`, while the runtime fixture marks `memory_add` unavailable. No live reproduction was run for the exact long-running memory-bloat workload.

## Solution Assessment

No, not as a direct automation fix. The proposal is plausible, but the best path needs maintainer decisions on the public write API, hard-limit configuration, memory block identity, duplicate policy, and security scanning behavior.
