## Summary

Keep open. Current main has memory recall/read tooling plus read and injection caps, but it still lacks the requested bounded memory CRUD/write surface with hard per-store character budgets; the related draft PR is only a partial add-only step.

## What This Changes

The issue requests a built-in bounded memory write tool with hard configurable per-store character limits, add/replace/remove semantics, duplicate handling, frozen snapshots, and optional credential screening.

## Best Possible Solution

Define a memory-core-owned write API with config-backed budgets, atomic add/list/replace/remove semantics, duplicate handling, optional credential screening, prompt/docs updates, and regression coverage coordinated with #70540.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and docs inspection on current main show `memory_search`/`memory_get` plus retrieval/bootstrap caps, not a bounded write path.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the maintainable solution should settle CRUD shape, budget enforcement, duplicate policy, and credential-screening contract before implementation.
