## Summary

Keep open. Current main still exposes read-only `memory_search`/`memory_get` plus a narrow Dreaming promotion compaction guard; the requested bounded CRUD write surface, hard per-store limits, replace/remove behavior, duplicate policy, frozen snapshot semantics, and secret-handling contract remain unresolved, and https://github.com/openclaw/openclaw/pull/70540 only covers the first `memory_add` slice.

## What This Changes

Add a built-in memory write/CRUD tool with configurable hard per-store character limits, over-budget rejection and usage reporting, substring replace/remove, duplicate detection, frozen snapshots, and optional credential scanning.

## Best Possible Solution

Define and approve a memory-core write contract covering add/list/replace/remove, per-store budgets, over-budget policy, duplicate handling, prompt guidance, permission grants, and secret policy; then land https://github.com/openclaw/openclaw/pull/70540 or a successor as the first implementation slice within that design.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence current main lacks bounded memory CRUD writes: memory-core registers only `memory_search`/`memory_get`, and the docs still describe only two memory tools.

## Solution Assessment

Unclear. A built-in memory write API is plausible, but hard rejection versus eviction/archive/consolidation, write grants, budgets, and secret handling need maintainer product and security decisions before implementation is safe.
