## Summary

Keep open. Current main and v2026.6.6 still expose recall-only memory tools; the bounded character-budget code is limited to dreaming promotion writes and does not implement the requested add/replace/remove memory contract with hard per-store limits or credential-safety policy.

## What This Changes

Request a built-in bounded memory write tool for add/replace/remove with hard per-store character limits, duplicate handling, snapshot injection, and optional credential scanning.

## Best Possible Solution

Define a memory-core or plugin-SDK owned bounded CRUD contract, including hard-limit semantics, duplicate/replace/remove identity, docs, tests, and secret-safety behavior, before implementation.

## Reproduction Assessment

Yes for the source-level capability gap: current main exposes only `memory_search` and `memory_get`, while the QA fixture marks `memory_add` absent. No live reproduction was run for the exact long-running memory-bloat workload.

## Solution Assessment

No, not as a direct automation fix. The proposal is plausible, but the best path needs maintainer decisions on the public write API, hard-limit configuration, memory block identity, and security scanning policy.
