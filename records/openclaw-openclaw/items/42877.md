## Summary

Keep open. Current main and v2026.6.10 still expose memory recall tools plus narrower bounded promotion/flush guards, but they do not provide the requested hard-limited add/replace/remove memory write contract with duplicate, usage, and secret-safety semantics.

## What This Changes

Request a built-in bounded memory write tool with hard per-store character limits, add/replace/remove operations, duplicate handling, frozen snapshot injection, and optional credential-pattern blocking.

## Best Possible Solution

Design a memory-core owned bounded CRUD contract with explicit write authority, hard-limit and usage semantics, replacement identity, duplicate/consolidation policy, secret handling, docs, and focused tests before implementation.

## Reproduction Assessment

Yes for the source-level capability gap: current main and v2026.6.10 expose only `memory_search` and `memory_get`, while the requested bounded write operations are absent. I did not live-reproduce the multi-week memory-bloat workload.

## Solution Assessment

No as an immediate automated fix. The proposal is plausible, but the best path needs maintainer decisions on the public memory write API, config shape, replacement identity, duplicate handling, and secret-safety policy.
