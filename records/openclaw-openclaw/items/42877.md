## Summary

Keep open. Current main and the latest release still expose read-only `memory_search`/`memory_get` plus a Dreaming-only `MEMORY.md` compaction cap; the requested bounded CRUD write surface, hard per-store limits, duplicate handling, frozen snapshot semantics, and secret-handling contract remain unresolved, and https://github.com/openclaw/openclaw/pull/70540 is only an open draft `memory_add` slice.

## What This Changes

The issue requests a built-in bounded memory write tool with hard per-store character limits and CRUD-style add, replace, and remove behavior.

## Best Possible Solution

Define and approve a memory-core write contract covering add/list/replace/remove, per-store budgets, over-budget behavior, duplicate handling, prompt guidance, permission grants, snapshot semantics, and secret policy; then land https://github.com/openclaw/openclaw/pull/70540 or a successor as the first implementation slice.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence current main lacks bounded memory CRUD writes: memory-core registers only `memory_search`/`memory_get`, and docs/tests still describe only read tools plus a Dreaming-specific compaction guard.

## Solution Assessment

Unclear. A built-in memory write API is plausible, but hard rejection versus eviction/archive/consolidation, write grants, budgets, snapshot semantics, and secret handling need maintainer product and security decisions before implementation is safe.
