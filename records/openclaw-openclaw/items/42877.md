## Summary

Keep open. Current main still has read-only memory tools plus a narrow Dreaming promotion compaction guard; the requested bounded write/CRUD tool, hard per-store limits, replace/remove, duplicate policy, and secret-handling behavior remain unresolved, and the related open draft PR only covers the first `memory_add` slice.

## What This Changes

Request a built-in bounded memory write/CRUD tool with hard per-store character limits, rejection/usage feedback, substring replace/remove, duplicate detection, frozen session snapshots, and optional credential scanning.

## Best Possible Solution

Define and approve a memory-core write contract covering add/list/replace/remove, per-store budgets, over-budget policy, duplicate handling, prompt guidance, and secret policy; then land the open `memory_add` slice or a successor inside that design.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence current main lacks bounded memory CRUD writes: memory-core registers only `memory_search`/`memory_get`, and the docs still describe only two memory tools.

## Solution Assessment

Unclear. A built-in memory write API is plausible, but hard rejection versus eviction/archive/consolidation, write grants, budgets, and secret handling need maintainer product/security decisions before implementation is safe.
