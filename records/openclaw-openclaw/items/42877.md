## Summary

Keep open. Current main and v2026.6.6 still expose recall-only memory tools, while the existing bounded MEMORY.md logic is limited to dreaming promotion cleanup and does not implement the requested hard-limit add/replace/remove memory contract.

## What This Changes

The issue requests a built-in bounded memory write tool with hard character limits, add/replace/remove operations, duplicate handling, frozen session snapshots, and optional credential-pattern blocking.

## Best Possible Solution

Design one memory-core-owned durable memory mutation contract that covers add/list/replace/remove, per-store hard limits, block identity, duplicate handling, secret-scan behavior, docs, and regression coverage before implementation.

## Reproduction Assessment

Yes at source level, not as a runtime bug reproduction: current main and v2026.6.6 expose only memory_search and memory_get, and exact searches found no memory add/list/replace/remove implementation.

## Solution Assessment

Unclear as proposed. memory-core is the plausible owner, but maintainers still need to decide the permission model, hard-limit semantics, block identity for replace/remove, config surface, and credential-scanning behavior before a fix PR is safe.
