## Summary

Keep open. Current main and the latest release still expose recall-only memory tools, while the existing bounded MEMORY.md code is limited to dreaming promotion cleanup and does not implement the requested hard-limit add/replace/remove memory contract.

## What This Changes

Request to add a built-in bounded durable-memory CRUD/write tool with hard per-store character limits, duplicate handling, session-start snapshots, and optional credential scanning.

## Best Possible Solution

Design one memory-core-owned durable memory mutation contract that covers add/list/replace/remove, per-store hard limits, block identity, duplicate handling, secret-scan behavior, docs, and regression coverage before implementation.

## Reproduction Assessment

Yes at source level, not as a runtime bug reproduction: current main and v2026.6.6 expose only memory_search and memory_get, and targeted searches found no memory_add/list/replace/remove implementation. No live scenario was run because this is a requested new durable-memory write capability.

## Solution Assessment

Unclear as proposed. memory-core is the plausible owner, but maintainers still need to decide the permission model, hard-limit semantics, block identity for replace/remove, config surface, and credential-scanning behavior before a fix PR is safe.
