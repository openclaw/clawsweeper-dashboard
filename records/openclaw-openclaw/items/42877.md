## Summary

Keep open. Current main has bounded read/search behavior and bootstrap truncation guidance, but it still lacks the requested bounded memory write/CRUD tool with hard per-store limits; the linked draft PR is only a partial `memory_add` slice and remains unmerged with product/security concerns.

## What This Changes

The issue requests a built-in bounded memory write tool with configurable hard character limits, add/replace/remove operations, duplicate handling, frozen session snapshots, and optional credential-pattern blocking.

## Best Possible Solution

Define a staged memory-core CRUD contract with explicit write authority, configurable per-store budgets, safe block identity, duplicate/secret handling, and tests/docs; keep the open `memory_add` draft as only the first possible implementation slice.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main exposes only `memory_search` and `memory_get`, while user reports provide supporting evidence that heavy memory setups still want hard write-time bounds.

## Solution Assessment

Unclear. A built-in memory-core write path is plausible, but the safest solution should be staged and permissioned rather than landing the whole proposed tool shape without decisions on limits, eviction, duplicate handling, and secret scanning.
