## Summary

Keep this issue open. Current main still provides memory recall/read tooling and bounded read/injection controls, but it does not implement the requested bounded memory CRUD/write surface with hard per-store character budgets, over-limit rejection, substring replace/remove, duplicate handling, or credential screening; the related open draft PR is only a partial add-only step.

## What This Changes

The issue requests a built-in bounded memory write tool with hard per-store character limits, add/replace/remove operations, duplicate handling, session-start snapshots, and optional credential-pattern screening.

## Best Possible Solution

Keep this open for a memory-core-owned API design: config-backed per-store character budgets, atomic add/list/replace/remove behavior, duplicate handling, over-limit responses with current usage and entries for consolidation, credential-pattern screening, docs, and focused tests coordinated with the open memory_add draft.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Current-main verification is high confidence because the registered tools, schemas, prompt guidance, docs, gateway allowlist, and exact symbol search all show only `memory_search`/`memory_get` plus read/injection limits.

## Solution Assessment

No patch is proposed on this issue, and the related draft PR is only a partial add-only implementation. The best path is a deliberate memory-core CRUD/budget design rather than closing this as stale, implemented, or a ClawHub-only plugin request.
