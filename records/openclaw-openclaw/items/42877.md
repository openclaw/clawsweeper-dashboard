## Summary

Keep open. Current main and v2026.6.10 still provide memory recall plus narrower promotion/flush guardrails, but they do not implement the requested hard-limited add/replace/remove memory write contract with usage feedback, duplicate handling, snapshot semantics, and secret-safety policy.

## What This Changes

The issue requests a built-in bounded memory write tool with hard per-store character limits, CRUD-style updates, duplicate handling, frozen session snapshots, and optional credential-pattern blocking.

## Best Possible Solution

Design a memory-core-owned bounded CRUD contract with explicit write authority, hard-limit and usage semantics, replacement identity, duplicate/consolidation policy, secret handling, docs, and focused tests before implementation.

## Reproduction Assessment

Yes for the source-level capability gap: current main and v2026.6.10 expose only memory_search and memory_get, while the requested bounded write operations are absent. I did not live-reproduce the multi-week memory-bloat workload.

## Solution Assessment

No as an immediate automated fix. The proposal is plausible, but the best path needs maintainer decisions on the public memory write API, per-store config shape, replacement identity, duplicate handling, and secret-safety policy.
