## Summary

Keep open. Current main and v2026.6.6 still expose recall-only memory tools plus a narrow dreaming-promotion budget; the requested bounded durable-memory write/CRUD contract remains a product and security design item.

## What This Changes

The issue requests a built-in bounded memory write tool with hard per-store character limits, add/replace/remove semantics, duplicate handling, frozen session snapshots, and optional credential-pattern blocking.

## Best Possible Solution

Keep this open for a memory-core/plugin-owned design that defines bounded durable writes, hard-limit rejection behavior, permission and security boundaries, storage identity, duplicate handling, replace/remove semantics, docs, and focused tests before implementation.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/design request. Source inspection gives high confidence that current main and v2026.6.6 expose recall tools and narrow promotion compaction, not a bounded CRUD memory write tool.

## Solution Assessment

Unclear as proposed. A memory-core write tool is plausible, but the exact built-in tool shape, permission model, hard-limit behavior, duplicate semantics, and secret-scanning contract need maintainer product/security decisions first.
