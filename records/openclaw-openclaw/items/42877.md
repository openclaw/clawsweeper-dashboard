## Summary

Keep open. Current main still has read-only `memory_search`/`memory_get` plus optional Dreaming promotion into `MEMORY.md`, but it does not provide a bounded memory CRUD tool with hard per-store character budgets, over-budget rejection, replace/remove, duplicate handling, or credential guardrails; the related draft at https://github.com/openclaw/openclaw/pull/70540 is only a first `memory_add` slice and already has security/product review blockers.

## What This Changes

Request a built-in bounded memory write tool that enforces configurable hard character limits and supports add, replace, remove, duplicate detection, frozen session snapshots, and optional credential-pattern blocking.

## Best Possible Solution

Define a memory-core bounded CRUD contract with explicit write-tool authority, per-store budget configuration, over-budget behavior, duplicate/replace/remove semantics, secret-handling policy, docs, and tests; treat https://github.com/openclaw/openclaw/pull/70540 as a possible first implementation slice only after that direction is approved.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested bounded memory write CRUD surface and only exposes read tools plus optional promotion into `MEMORY.md`.

## Solution Assessment

Unclear. A memory-core write API is a plausible direction, but the requested hard budgets, replacement/removal model, write permissions, and secret scanning need maintainer product and security decisions before a narrow code fix is safe.
