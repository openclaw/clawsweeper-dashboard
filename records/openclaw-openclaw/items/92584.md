## Summary

Keep open for maintainer review: the patch is a narrow security hardening with sufficient real behavior proof, but it removes a shipped assistant-media query-token fallback, so maintainers should explicitly accept the compatibility break or request a transition path.

## What This Changes

This PR removes the Control UI assistant-media `?token=` auth fallback in the Gateway, updates media tests to use bearer headers, and adds an e2e regression for rejecting query-token auth while bearer and mediaTicket access still work.

## Best Possible Solution

Land the narrow fallback removal after maintainer acceptance, with release-note context naming bearer auth and scoped mediaTicket URLs as the supported assistant-media paths.

## Reproduction Assessment

Yes at source/proof level. Current main and `v2026.6.9` both parse `token` from assistant-media query params and pass `allowQueryToken: true`; the PR body also supplies before/after live loopback gateway output.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. With scoped mediaTicket already covering first-party browser media and bearer auth covering programmatic clients, deleting the leftover query-token fallback is the narrowest hardening path.
