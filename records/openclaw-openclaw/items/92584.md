Keep open for maintainer review: the patch is narrow, proof-backed, and has no blocking correctness finding, but merging intentionally removes a shipped assistant-media query-token auth fallback.

## What This Changes

The PR removes assistant-media query-string token auth, updates media tests to use bearer auth, and adds an e2e regression proving query tokens are rejected while bearer and mediaTicket access still work.

## Best Possible Solution

Land the narrow security hardening if maintainers accept the compatibility break, with merge or release-note context that assistant-media query-token auth is removed.

## Reproduction Assessment

Yes. Current main and v2026.6.6 still parse `token` from assistant-media query strings, and the PR body includes before/after live gateway output showing a correct `?token=` request changing from 200 to 401 while bearer and mediaTicket still work.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. Removing route-level query-token auth is the clean final step now that scoped mediaTicket URLs are the documented browser path; a transition window would preserve the URL-secret exposure longer.
