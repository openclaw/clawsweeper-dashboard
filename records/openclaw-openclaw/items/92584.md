## Summary

Keep open: this is a focused, well-proven security hardening PR, but current main and the latest release still accept assistant-media `?token=` auth, so the remaining decision is maintainer acceptance of the shipped compatibility break.

## What This Changes

The PR removes query-string token authentication from the Control UI assistant-media gateway route and updates tests to use bearer or scoped mediaTicket auth instead.

## Best Possible Solution

Land the narrow fallback removal after maintainer acceptance, with release-note context naming bearer auth and scoped mediaTicket URLs as the supported assistant-media paths.

## Reproduction Assessment

Yes at source/proof level: current main parses `token` from assistant-media query params and passes `allowQueryToken: true`, and the PR body supplies before/after live loopback gateway output. I did not run the e2e test in this read-only review.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. With scoped mediaTicket already covering first-party browser media and bearer auth covering programmatic clients, deleting the leftover query-token fallback is the narrowest hardening path.
