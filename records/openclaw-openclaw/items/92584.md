## Summary

Keep open for maintainer review: the patch looks correct and well-proven, but it intentionally removes a shipped assistant-media `?token=` compatibility path that existing clients or saved URLs may still rely on.

## What This Changes

This PR removes the assistant-media query-token auth fallback in `src/gateway/control-ui.ts`, updates gateway tests to use bearer headers, and adds an e2e regression that rejects `?token=` while preserving bearer and mediaTicket access.

## Best Possible Solution

Land the narrow fallback removal after maintainer acceptance, with release-note context naming bearer auth and scoped mediaTicket URLs as the supported assistant-media paths.

## Reproduction Assessment

Yes at source/proof level. Current main and `v2026.6.10` both parse `token` from assistant-media query params and pass `allowQueryToken: true`; the PR body supplies before/after live loopback gateway output.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. With scoped mediaTicket already covering first-party browser media and bearer auth covering programmatic clients, deleting the leftover query-token fallback is the narrowest hardening path.
