## Summary

Keep open for maintainer review: this PR still removes a shipped server-side assistant-media query-token fallback that current main has not removed. The patch looks narrow and proof-backed; the remaining decision is whether maintainers accept the compatibility break for clients or saved URLs that still send credentials in `?token=`.

## What This Changes

The PR removes assistant-media query-string token auth from the Control UI gateway route, updates media tests to use bearer auth, and adds an e2e regression proving query-token rejection with bearer/mediaTicket success.

## Best Possible Solution

Land this narrow hardening after maintainer acceptance of the compatibility break, with release-note or merge context that bearer auth and scoped mediaTicket URLs are the supported assistant-media paths.

## Reproduction Assessment

Yes. Current main and v2026.6.6 source both parse `token` from assistant-media query params and pass `allowQueryToken: true`; the PR body also includes before/after live loopback gateway output.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. With scoped mediaTicket already covering browser media and bearer auth covering programmatic clients, removing the leftover query-token fallback is the narrowest hardening path; a transition window would preserve the credential-in-URL exposure longer.
