## Summary

Keep open for maintainer acceptance: the patch is focused and well proven, but it intentionally removes a shipped assistant-media query-token fallback that existing custom clients or saved URLs may still rely on.

## What This Changes

The PR removes `?token=` authentication for the Control UI assistant-media route, updates existing media tests to use bearer headers, and adds an e2e regression proving query-token rejection while bearer and media-ticket access still work.

## Best Possible Solution

Land the narrow hardening only after maintainers accept the shipped fallback removal, with release context naming bearer auth and scoped mediaTicket URLs as the supported assistant-media paths.

## Reproduction Assessment

Yes. Current main and `v2026.6.8` both parse `token` from assistant-media query params and pass `allowQueryToken: true`; the PR body also supplies before/after live loopback gateway output.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. With scoped mediaTicket already covering browser media and bearer auth covering programmatic clients, deleting the leftover query-token fallback is the narrowest hardening path.
