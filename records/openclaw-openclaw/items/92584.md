## Summary

Keep open: this is a focused, correct security-hardening PR with sufficient real loopback gateway proof, but it removes a query-token assistant-media auth fallback that is still present in current main and v2026.6.9, so maintainers need to accept the compatibility/security tradeoff before merge.

## What This Changes

The PR removes query-string token authentication from the Control UI assistant-media route while keeping bearer-header and signed media-ticket access.

## Best Possible Solution

Land the focused hardening only after maintainers accept the shipped fallback removal, with release context naming bearer auth and scoped mediaTicket URLs as the supported assistant-media paths.

## Reproduction Assessment

Yes. Current main and v2026.6.9 both parse `token` from assistant-media query parameters and enable `allowQueryToken: true`; the PR body also supplies before/after live loopback gateway output.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. With scoped mediaTicket already covering browser media and bearer auth covering programmatic clients, deleting the leftover query-token fallback is the narrowest hardening path.
