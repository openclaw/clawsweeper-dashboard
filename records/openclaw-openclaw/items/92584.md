## Summary

Keep open for maintainer review: the patch is a narrow credential-handling hardening with sufficient live proof, but it intentionally removes a shipped assistant-media `?token=` auth fallback, so the compatibility/security tradeoff needs maintainer acceptance before merge.

## What This Changes

The branch removes assistant-media query-string token extraction, updates media tests to use bearer auth, and adds an e2e regression proving `?token=` is rejected while bearer and `mediaTicket` access still work.

## Best Possible Solution

Land the hardening after maintainers accept the shipped fallback removal, with merge or release-note context that bearer auth and scoped `mediaTicket` URLs are the supported assistant-media paths.

## Reproduction Assessment

Yes. Current main and v2026.6.6 both parse `token` from assistant-media query params and pass `allowQueryToken: true`; the PR body also includes before/after live loopback gateway output for the failing and fixed cases.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. With scoped `mediaTicket` covering browser media and bearer auth covering programmatic clients, removing the leftover query-token fallback is the narrowest hardening path; a transition window is the safer alternative only if compatibility must temporarily win.
