## Summary

Keep open: this is a focused security hardening PR with sufficient live proof, but it intentionally removes a shipped assistant-media query-token fallback, so maintainers still need to accept the compatibility break before merge.

## What This Changes

The PR removes Control UI assistant-media query-string token authentication, updates media tests to use bearer auth, and adds a gateway e2e regression for query-token rejection with bearer/mediaTicket success.

## Best Possible Solution

Land the narrow hardening after maintainer acceptance of the shipped fallback removal, with merge or release context naming bearer auth and scoped mediaTicket URLs as the supported assistant-media paths.

## Reproduction Assessment

Yes. Current main and v2026.6.8 both parse `token` from assistant-media query params and pass `allowQueryToken: true`; the PR body also includes before/after live loopback gateway output.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. With scoped mediaTicket already covering browser media and bearer auth covering programmatic clients, deleting the leftover query-token fallback is the narrowest hardening path.
