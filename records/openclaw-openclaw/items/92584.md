## Summary

Keep open: this PR is still useful because current main and v2026.6.6 still accept the Control UI assistant-media token via `?token=` server-side, even though the first-party UI now uses scoped media tickets. The patch looks correct and proof is sufficient; maintainers should explicitly accept the fail-closed compatibility break for any custom query-token clients.

## What This Changes

The PR removes assistant-media query-token auth from Control UI read authentication and updates gateway tests to require Authorization bearer or scoped mediaTicket access.

## Best Possible Solution

Land the header-or-ticket-only assistant-media server behavior while keeping existing page-bootstrap `?token=` import compatibility untouched.

## Reproduction Assessment

Yes. Current main accepts a correct assistant-media `?token=` through `allowQueryToken`, and the PR body supplies loopback HTTP before/after output showing that only the query-token case changes from 200 to 401 while header and mediaTicket still work.

## Solution Assessment

Yes. The prior ticket-flow fix already preserved browser-native media rendering, so removing the leftover server-side query-token fallback is now the narrowest maintainable hardening path.
