## Summary

Keep open: current main and the latest release still accept assistant-media `?token=` auth, and the PR is a narrow, proof-backed hardening change. The remaining decision is whether maintainers accept the intentional fail-closed compatibility break for clients or saved URLs that still rely on that fallback.

## What This Changes

The PR removes assistant-media `?token=` auth from `src/gateway/control-ui.ts`, updates gateway media tests to use `Authorization: Bearer`, and adds an e2e regression for rejected query-token auth plus accepted bearer and mediaTicket access.

## Best Possible Solution

Land this narrow security hardening if maintainers accept the compatibility break, with release-note context that query-token assistant-media auth is removed and bearer or mediaTicket auth are the supported paths.

## Reproduction Assessment

Yes for the PR behavior: current main and the latest release still wire assistant-media auth with `allowQueryToken: true`, and the PR's e2e test/proof exercises a live gateway before and after the change. I did not run tests in this read-only review.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff: removing query-token auth from this route aligns it with bearer auth, the existing signed mediaTicket browser flow, and the documented route contract. The safer compatibility alternative would be a documented transition window, but that would preserve the secret-in-URL exposure longer.
