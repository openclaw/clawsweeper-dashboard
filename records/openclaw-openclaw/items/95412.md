## Summary

Keep open: this is a focused Discord REST security hardening PR with sufficient real behavior proof and no blocking code findings; the remaining maintainer decision is accepting the new fail-closed 8 MiB response cap.

## What This Changes

The PR replaces Discord RequestClient's unbounded response.text() read with an 8 MiB readResponseWithLimit wrapper and adds overflow, stall, and normal-payload regression tests.

## Best Possible Solution

Land the focused RequestClient cap after maintainers accept or adjust the 8 MiB ceiling; track requestDiscord success-response hardening separately if owners want a broader sweep.

## Reproduction Assessment

Yes. Current main and v2026.6.8 have an unbounded RequestClient response.text() read, and the PR body gives a concrete before/after RequestClient flood and stall proof path, though I did not execute it locally in this read-only review.

## Solution Assessment

Yes for the RequestClient path: the PR reuses the existing plugin SDK response-limit helper at the correct extension boundary and covers overflow, idle timeout, and normal payloads. It is a scoped mitigation rather than complete Discord-wide hardening because requestDiscord still has an unbounded success read.
