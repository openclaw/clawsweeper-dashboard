## What This Changes

The PR replaces Discord RequestClient's unbounded response.text() read with an 8 MiB readResponseWithLimit wrapper and adds overflow, stall, and normal-payload regression tests.

## Best Possible Solution

Land the focused RequestClient cap after the PR body/check gate is fixed and maintainers accept the 8 MiB ceiling; track requestDiscord success reads as a separate follow-up.

## Reproduction Assessment

Yes for the RequestClient path: it reuses the existing plugin SDK response-limit helper at the right plugin boundary and covers overflow, idle timeout, and normal payloads. It is not a complete Discord-wide hardening because requestDiscord still has an unbounded success-body read.
