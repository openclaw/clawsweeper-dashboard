## Summary

Keep open: this is a focused, still-useful Discord RequestClient hardening PR, and current `main` plus the latest release still have the unbounded body read it targets. I found no patch correctness finding, but the new 8 MiB fail-closed ceiling and same-line overlap with the open gzip-response PR need maintainer acceptance before merge.

## What This Changes

The PR adds an 8 MiB Discord RequestClient response-body cap with idle-timeout handling and regression tests for overflow, stall, and normal JSON responses.

## Best Possible Solution

Land the focused RequestClient hardening after maintainers accept or adjust the 8 MiB ceiling, and coordinate the final reader with the open gzip-response PR if both protections are desired.

## Reproduction Assessment

Yes. Current `main` and `v2026.6.8` reach an unbounded `response.text()` in Discord `RequestClient.executeRequest`, and the PR body includes a concrete before/after flood and stall proof path; I did not execute it in this read-only review.

## Solution Assessment

Yes for the targeted RequestClient memory-boundary bug. The patch reuses the existing plugin SDK response-limit helper at the extension boundary; the best final Discord reader may need to combine this cap with the open raw-gzip decoding PR and leave `requestDiscord` as follow-up.
