## Summary

## What This Changes

The PR replaces Discord RequestClient's unbounded REST success-body read with the shared response-limit helper and adds regression tests for overflow, idle timeout, and normal payload parsing.

## Best Possible Solution

Land the focused RequestClient hardening after maintainer acceptance of the cap, and track the remaining `requestDiscord` success-body read as a separate Discord hardening follow-up if desired.

## Reproduction Assessment

Yes. Current main has an unbounded `response.text()` in `RequestClient.executeRequest`, and the PR body provides before/after terminal proof against that path; I did not run tests in this read-only review.

## Solution Assessment

Yes for the RequestClient path: the PR reuses the existing plugin SDK response-limit helper and preserves the parse/status flow. It is intentionally not a complete Discord-wide policy because `requestDiscord` still has an unbounded success read.
