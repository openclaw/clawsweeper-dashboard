## Summary

Keep open: current main and v2026.6.9 still read Discord RequestClient response bodies with unbounded `response.text()`, and this PR is a focused, useful hardening patch. The patch looks correct for that path, but the new default 8 MiB fail-closed ceiling needs maintainer acceptance before merge.

## What This Changes

The PR replaces Discord RequestClient's unbounded response read with an 8 MiB `readResponseWithLimit` wrapper and adds overflow, idle-timeout, and normal-payload regression tests.

## Best Possible Solution

Land the focused RequestClient cap after Discord/security owners accept the response-size ceiling, and track the remaining `requestDiscord` success read as separate hardening if desired.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 still call `response.text()` on the full RequestClient body. The PR body also supplies after-fix live output for a flood/stall proof, though I did not execute that proof locally.

## Solution Assessment

Yes for the focused RequestClient path: it reuses the existing plugin SDK response-limit helper at the right extension boundary and preserves the existing parse/status flow. It is not complete Discord-wide hardening because `requestDiscord` still has an unbounded success read.
