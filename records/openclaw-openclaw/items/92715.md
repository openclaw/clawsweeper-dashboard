## Summary

Keep open. Current source wires Discord status reactions, but the real tool-start callback is still filtered by verbose/tool-summary visibility, which matches this report’s default-config symptom: queued/stall/final reactions can appear while tool-phase reactions never reach Discord.

## What This Changes

The issue asks OpenClaw to show Discord status-reaction transitions during active tool and skill execution instead of leaving the message on the queued/stall state until final synthesis.

## Best Possible Solution

Decouple Discord status-reaction lifecycle updates from verbose/tool-summary delivery gates while keeping text progress summaries suppressed unless verbose or streaming-preview settings explicitly allow them.

## Reproduction Assessment

Yes at source level, but not via a live Discord run in this read-only pass. With default verbose off, tool-start progress callbacks are filtered before Discord's statusReactions.setTool path, matching the reported missing intermediate reactions.

## Solution Assessment

No, current main is not the best complete solution. The maintainable fix is to give channel status reactions a lifecycle signal that is independent from visible verbose/tool-summary delivery, rather than treating reaction-only status as text progress.
