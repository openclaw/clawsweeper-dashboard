## Summary

Keep open: this PR addresses a real current-main OpenAI TTS directive gap and includes sufficient after-fix behavior proof, but the patch still makes out-of-range `[[tts:speed=...]]` directives throw out of directive parsing instead of returning a warning.

## What This Changes

The PR adds OpenAI TTS speed directive parsing and tests, extends telephony provider-override coverage, and adds a changelog entry.

## Best Possible Solution

Land provider-owned OpenAI speed directive support after invalid and out-of-range speed handling uses non-fatal warnings while valid overrides still serialize through normal and telephony OpenAI speech requests.

## Reproduction Assessment

Yes, source-reproducible. Current main's OpenAI parser does not claim `speed`, while the directive router only stores provider overrides after `handled: true`; the PR finding is also source-reproducible because the new branch calls `requireInRange` without a catch.

## Solution Assessment

No, not as-is. Provider-local OpenAI parsing with the 0.25..4 range is the right direction, but model-emitted directive validation failures should return warnings instead of throwing through `parseTtsDirectives`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Return warnings for out-of-range speed directives:** `extensions/openai/speech-provider.ts:234`
  - body: The new speed branch calls `requireInRange` directly. `parseTtsDirectives` does not catch provider parser exceptions, while adjacent provider parsers convert invalid numeric settings into warnings, so a model-emitted `[[tts:speed=99]]` for OpenAI would throw instead of stripping the directive and continuing. Return `handled: true` with a warning for out-of-range values and update the focused test expectation.
  - confidence: 0.92
