## Summary

Keep this PR open: it targets a real current-main OpenAI TTS directive gap, but the latest patch is not merge-ready because invalid model-supplied speed directives can throw out of directive parsing and the new telephony override parameter is not exposed through the public plugin runtime type.

## What This Changes

The PR adds OpenAI TTS speed directive parsing, forwards provider overrides through OpenAI telephony synthesis and speech-core telephony requests, adds focused tests, and records the #12163 behavior in the changelog.

## Best Possible Solution

Land the provider-local OpenAI speed parser and regression coverage after making invalid speed directives nonfatal warnings, exposing telephony overrides in the public TTS runtime params type, and placing the changelog entry under fixes.

## Reproduction Assessment

Yes. On current main, a parser-level reproduction with OpenAI as the preferred provider and `[[tts:speed=1.5]]` drops the speed override because the OpenAI provider returns `handled: false` for `speed`.

## Solution Assessment

No, not quite as submitted. The provider-local parser and override plumbing are the right scope, but the patch should preserve nonfatal directive-warning behavior and keep the public runtime type aligned with the implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Return warnings for invalid OpenAI speed directives:** `extensions/openai/speech-provider.ts:222`
  - body: Out-of-range speed currently calls `requireInRange` without a catch or manual warning path. `parseTtsDirectives` does not catch provider parser exceptions, so a model-emitted `[[tts:speed=5]]` can throw out of reply TTS processing instead of being stripped and logged like ElevenLabs/MiniMax invalid voice settings. Return a handled warning for invalid range values and update the test away from `toThrow`.
  - confidence: 0.89
- **[P2] Expose telephony overrides in the runtime params type:** `extensions/speech-core/src/tts.ts:1321`
  - body: The implementation now accepts `overrides` on `textToSpeechTelephony` and threads them to provider telephony synthesis, but public plugin callers use `PluginRuntime.tts.textToSpeechTelephony` through `TtsTelephonyRequestParams`, which still only allows `text`, `cfg`, and `prefsPath`. Typed callers cannot pass the new overrides without casts, so add `overrides?: TtsDirectiveOverrides` to the public params type and update the API check if it reports drift.
  - confidence: 0.87
- **[P3] Move the changelog entry under fixes:** `CHANGELOG.md:17`
  - body: The added changelog line is under `### Changes`, but this is a user-facing bug fix and the entry itself says it fixes #12163. Move the same single-line entry under `### Fixes` so the release notes classify it with the other fixes.
  - confidence: 0.78
