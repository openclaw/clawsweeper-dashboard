## Summary

Keep open. Current main still drops OpenAI `[[tts:speed=N]]` directives before they become provider overrides, and this PR is a focused provider-local fix that matches the existing directive routing contract and OpenAI's documented speech speed range. I found no blocking functional or security findings; the remaining action is maintainer review plus normal CI/Testbox validation before merge.

## What This Changes

The PR adds an OpenAI-owned `speed`/`openai_speed`/`openaispeed` TTS directive parser case with range validation, focused OpenAI speech-provider tests, and a changelog entry.

## Best Possible Solution

Land a rebased version of this provider-owned OpenAI parser fix with the focused regression tests, keep core directive routing unchanged, and track telephony directive support separately if maintainers want that surface to honor provider overrides too.

## Reproduction Assessment

Yes. On current main, `[[tts:speed=1.5]]` with OpenAI selected reaches the OpenAI parser as `key: "speed"`, but that parser returns unhandled, and `parseTtsDirectives` only forwards handled provider overrides; no live OpenAI call is needed to reproduce the drop.

## Solution Assessment

Yes for regular OpenAI speech synthesis. Provider-local parsing is the narrowest maintainable fix because TTS speed ranges differ by provider; extending telephony should be a separate contract-aware change rather than folded into this parser PR.
