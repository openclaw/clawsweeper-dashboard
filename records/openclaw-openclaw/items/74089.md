## Summary

Keep open: this is the right provider-owned fix path for the OpenAI TTS speed directive gap, but the current patch still has a blocking parser-contract bug because out-of-range speed directives throw instead of becoming non-fatal directive warnings.

## What This Changes

The PR adds OpenAI `speed`/`openai_speed`/`openaispeed` directive parsing, focused tests, telephony override coverage, and a changelog entry.

## Best Possible Solution

Merge after converting OpenAI out-of-range speed values into directive warnings while preserving the provider-owned parser, OpenAI `0.25..4` range, aliases, and telephony request serialization.

## Reproduction Assessment

Yes. Source inspection shows current main routes `[[tts:speed=...]]` to the selected provider but OpenAI returns unhandled for `speed`; the review finding is also source-reproducible because `parseTtsDirectives` does not catch provider parser exceptions.

## Solution Assessment

No, not as-is. Provider-owned OpenAI parsing and existing synthesis plumbing are the narrow maintainable direction, but direct `requireInRange` throwing should be replaced with a warning-return path before merge.
