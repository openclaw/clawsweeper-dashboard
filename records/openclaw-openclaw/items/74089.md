## Summary

Keep open: the provider-owned OpenAI speed directive fix is the right direction and the PR proof is useful, but the patch still lets an out-of-range model directive throw out of directive parsing instead of returning a warning.

## What This Changes

The PR adds OpenAI TTS speed directive parsing, focused regression tests, telephony override coverage, and a changelog entry for the linked OpenAI TTS speed request.

## Best Possible Solution

Keep the OpenAI-owned directive parser approach, but make invalid and out-of-range speed directives return warnings instead of throwing before merge.

## Reproduction Assessment

Yes. Source inspection shows current main's OpenAI parser does not claim `speed`, while `parseTtsDirectives` only records provider overrides after a handled provider parse result; the PR body supplies after-fix request serialization proof for the valid path.

## Solution Assessment

No, not as-is. The owner-local parser and existing request-body plumbing are the narrow maintainable fix, but range validation should follow the directive warning contract so malformed model output does not abort TTS parsing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Return a warning for out-of-range speed directives:** `extensions/openai/speech-provider.ts:233`
  - body: The new `speed` branch calls `requireInRange` directly. `parseTtsDirectives` does not catch provider parser exceptions, so a model-emitted `[[tts:speed=99]]` for OpenAI would throw out of directive parsing instead of stripping the directive and returning a warning like adjacent provider parsers. Wrap range validation or return an invalid-speed warning for out-of-range values.
  - confidence: 0.88
