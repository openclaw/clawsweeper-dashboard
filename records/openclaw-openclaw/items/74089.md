## Summary

Keep open: current main still drops OpenAI `[[tts:speed=N]]` directives, and this PR is the right narrow area. The later commits address the Greptile telephony gap, but the patch is not merge-ready because the public TTS runtime params type still does not expose the new telephony `overrides` parameter.

## What This Changes

The PR adds OpenAI TTS speed directive parsing and range validation, forwards provider overrides through OpenAI telephony synthesis and speech-core telephony requests, adds focused tests, and records the fix in the changelog.

## Best Possible Solution

Land the provider-local OpenAI speed parser and telephony override forwarding after adding the matching public runtime params type and refreshing any generated SDK API baseline hash, keeping the fix additive and avoiding a generic core parser change.

## Reproduction Assessment

Yes. On current main, configuring OpenAI TTS with `allowVoiceSettings: true` and parsing `[[tts:speed=1.5]]` hits the OpenAI parser's default unhandled path, so `parseTtsDirectives` never records an OpenAI speed override.

## Solution Assessment

Not quite as submitted. The provider-local parser, OpenAI range validation, and telephony forwarding are the narrow maintainable fix, but the public `TtsTelephonyRequestParams` contract must also accept `overrides?: TtsDirectiveOverrides` so typed plugin callers can use the new seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Expose telephony overrides in the runtime params type:** `extensions/speech-core/src/tts.ts:1321`

Typed callers using `api.runtime.tts.textToSpeechTelephony({ overrides })` still get an excess-property error, so add `overrides?: TtsDirectiveOverrides` to that params type and refresh the SDK API baseline if it reports drift.
