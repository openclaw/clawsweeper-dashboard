## Summary

Keep this PR open: the OpenAI parser fix follows the provider-owned directive pattern, but the latest head still leaves the public telephony runtime parameter type stale, so typed plugin/runtime callers cannot use the new override path without casts.

## What This Changes

The branch adds OpenAI TTS speed directive parsing, forwards speech overrides through OpenAI telephony synthesis and speech-core telephony requests, adds focused tests, and records the behavior in the changelog.

## Best Possible Solution

Land the provider-local OpenAI speed parser and regression coverage after making the telephony override parameter part of the public TTS runtime contract, so bundled and third-party plugin callers see the same typed capability as the implementation.

## Reproduction Assessment

Yes. Current main drops `[[tts:speed=1.5]]` for OpenAI because the OpenAI parser returns `handled: false` for speed and the directive router only records provider overrides for handled provider tokens.

## Solution Assessment

Not quite as submitted. The parser and provider override plumbing are the narrow maintainable fix, but the new telephony override seam should also update `TtsTelephonyRequestParams` so the supported runtime facade matches the implementation.

## Review Findings

Overall correctness: patch is incorrect
