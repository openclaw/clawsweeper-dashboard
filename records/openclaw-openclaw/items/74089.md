## Summary

Keep open: the root-cause fix is on the right surface, but the PR still leaves the supported plugin runtime type narrower than the implementation, so typed callers cannot use the new telephony override path without casts.

## What This Changes

The PR adds OpenAI TTS speed directive parsing, threads provider overrides through OpenAI telephony synthesis and speech-core telephony requests, adds focused tests, and records the behavior in the changelog.

## Best Possible Solution

Land the provider-local OpenAI speed parser and regression coverage after adding the same `overrides?: TtsDirectiveOverrides` parameter to the public telephony runtime params type, keeping the fix additive and avoiding a generic core parser change.

## Reproduction Assessment

Yes. Current main can be verified by selecting OpenAI TTS with `allowVoiceSettings: true` and parsing `[[tts:speed=1.5]]`; the OpenAI provider returns unhandled for `speed`, so `parseTtsDirectives` never records an OpenAI speed override.

## Solution Assessment

Not quite as submitted. The parser, range validation, and provider override forwarding are the narrow maintainable fix, but the PR should also update the public `TtsTelephonyRequestParams` contract so typed plugin callers can use the new telephony override seam.
