## Summary

Keep open: the fix is in the right OpenAI provider boundary and the happy-path proof is supplied, but the new range check can throw from directive parsing for out-of-range model directives instead of returning a non-fatal warning.

## What This Changes

The PR adds OpenAI `[[tts:speed]]` directive parsing and tests, updates telephony provider override coverage, and adds a changelog entry.

## Best Possible Solution

Merge a scoped OpenAI provider fix once invalid and out-of-range speed directives are non-fatal warnings while valid speed overrides still serialize into normal and telephony OpenAI speech requests.

## Reproduction Assessment

Yes. Source inspection shows current main's OpenAI parser does not claim `speed`, while the request builder already supports `speed`; the PR body supplies after-fix request-body proof for the happy path.

## Solution Assessment

No, not as-is. The owner-local parser and telephony forwarding are the right direction, but range validation should follow the directive-parser warning contract instead of throwing out of `parseTtsDirectives`.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Return a warning for out-of-range speed directives:** `extensions/openai/speech-provider.ts:234`
