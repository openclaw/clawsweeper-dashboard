## Summary

Keep open. The provider-owned OpenAI speed directive parser is the right shape for the reported bug, but the latest PR head adds telephony override reads that currently fail typecheck because `SpeechTelephonySynthesisRequest` has no `providerOverrides` field and the core telephony caller does not pass one.

## What This Changes

The PR adds OpenAI TTS `speed` directive parsing with tests and a changelog entry, then updates OpenAI telephony synthesis to read model, voice, and speed provider overrides.

## Best Possible Solution

Land the provider-local OpenAI speed directive parser with focused tests, and resolve the telephony commit by either removing it from this PR or completing an additive telephony override contract and core pass-through with matching tests. Keep generic directive routing in core provider-owned rather than adding a global speed parser.

## Reproduction Assessment

Yes. The original bug has a source-level reproduction on current main: selected OpenAI plus `[[tts:speed=1.5]]` returns no OpenAI provider override because the OpenAI parser does not handle `speed`. The PR regression is also reproduced by PR-head CI annotations showing the missing `providerOverrides` telephony request type.

## Solution Assessment

Partly. The OpenAI-owned parser case is the narrowest maintainable fix for normal speech synthesis because speed ranges are provider-specific; the telephony add-on is not currently the best implementation because it reads a field that is neither in the SDK request type nor passed by the core telephony caller.
