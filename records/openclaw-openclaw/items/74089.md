## Summary

Keep open: the PR targets a real current-main OpenAI TTS directive gap, but it should not merge as-is because out-of-range `[[tts:speed=...]]` values would throw out of directive parsing instead of becoming directive warnings.

## What This Changes

The PR adds OpenAI speech-provider parsing for `speed`/`openai_speed`/`openaispeed`, focused parser and telephony override tests, and a changelog entry.

## Best Possible Solution

Land a provider-owned OpenAI speed directive parser that warns and drops invalid or out-of-range values while preserving valid speed overrides in normal and telephony OpenAI speech requests.

## Reproduction Assessment

Yes. Source inspection shows current main sends `[[tts:speed=...]]` to the selected provider parser, but OpenAI returns unhandled for `speed`, so no OpenAI override is stored.

## Solution Assessment

No, not as-is. The owner-local parser and telephony forwarding are the right shape, but out-of-range OpenAI speed directives should follow the directive warning contract instead of aborting parsing.
