## Summary

Keep this PR open: current main still drops OpenAI `[[tts:speed=N]]` directives, and the provider-local parser fix is the right direction, but the latest diff adds telephony override plumbing without updating the public runtime facade parameter type that plugin callers use.

## What This Changes

The PR adds OpenAI TTS speed directive parsing and range validation, forwards provider overrides through OpenAI telephony synthesis and speech-core telephony requests, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land the provider-local OpenAI speed parser and regression coverage after aligning the new telephony override parameter with the public plugin runtime contract, so both core code and third-party plugin callers see the same typed capability.

## Reproduction Assessment

Yes. On current main, `parseTtsDirectives` with preferred provider OpenAI and `[[tts:speed=1.5]]` reaches OpenAI's parser, but the parser returns `handled: false`, so no OpenAI provider override is recorded.

## Solution Assessment

Not quite as submitted. The OpenAI-owned parser case is the narrow maintainable fix, but the added telephony override seam should also update `TtsTelephonyRequestParams` so the public runtime API matches the implementation.
