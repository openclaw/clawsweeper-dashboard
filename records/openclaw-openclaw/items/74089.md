## Summary

Keep open: the PR fixes the documented OpenAI TTS speed directive gap, but it still makes out-of-range model-emitted speed directives throw out of directive parsing instead of returning a warning.

## What This Changes

Adds OpenAI TTS speed directive parsing and tests, extends telephony provider-override coverage, and updates the changelog.

## Best Possible Solution

Merge the provider-owned OpenAI speed directive fix after out-of-range values use the non-fatal warning path while valid overrides still serialize through normal and telephony OpenAI speech requests.

## Reproduction Assessment

Yes. Source inspection shows current main's OpenAI parser does not claim speed, while parseTtsDirectives only stores provider overrides after handled:true; the PR finding is also source-reproducible because the new branch calls requireInRange without a catch.

## Solution Assessment

No, not as-is. Provider-local OpenAI parsing and the 0.25..4 range are the right direction, but validation failures from model-emitted directives should be returned as warnings rather than thrown through parseTtsDirectives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Return warnings for out-of-range speed directives:** `extensions/openai/speech-provider.ts:234`
  - body: The new speed branch calls requireInRange directly. parseTtsDirectives does not catch provider parser exceptions, while adjacent provider directive parsers convert invalid numeric settings into warnings, so a model-emitted [[tts:speed=99]] for OpenAI would throw out of directive parsing instead of stripping the directive and continuing. Return handled:true with a warning for out-of-range values and update the focused test expectation.
  - confidence: 0.92
