## Summary

Keep this issue open: current main and v2026.6.5 still decode the originating session without showing it in the macOS exec approval prompt, while richer channel/sender context remains a legitimate but security-sensitive product decision.

## What This Changes

Show the originating session and safe requester/channel context in the macOS exec approval popup.

## Best Possible Solution

Add a focused, tested non-empty Session row to the existing native presenter, then separately extend the canonical approval protocol with privacy-conscious turn-source labels only after macOS and security owners approve their trust semantics.

## Reproduction Assessment

Yes at source level: current main definitively decodes `sessionKey` but omits it from `buildAccessoryView`; no live NSAlert run was needed to establish that code-path gap.

## Solution Assessment

Partly: rendering the existing Session value is the narrowest maintainable parity fix, but adding sender/channel strings directly is not the best full solution without a canonical provenance, privacy, and labeling contract.
