## Label Justifications

- `impact:session-state`: The native popup cannot distinguish which session triggered a pending exec approval when multiple sessions are active.

## Summary

Keep open: current main still decodes `sessionKey` for macOS exec approvals but the native prompt does not render it, and the Swift prompt model still does not expose safer requester/channel context.

## What This Changes

Request to show the session key and safe requester/channel context in the macOS exec approval dialog.

## Best Possible Solution

Add a trimmed non-empty Session row to the native presenter, then add Channel/Sender/Thread display only after a privacy-reviewed metadata contract maps raw turn-source fields to safe user-facing labels.

## Reproduction Assessment

Yes, at source level: current main decodes `sessionKey` in the Swift request but `buildAccessoryView` does not add a Session row. The richer requester part is also source-visible because TypeScript payload fields exist while the Swift prompt model ignores them.

## Solution Assessment

Partly: rendering the existing session key is the narrowest maintainable parity fix. The requester/channel/sender portion needs a privacy and security contract before adding new native UI fields.
