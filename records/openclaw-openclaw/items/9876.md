## Summary

Keep this open: current main still passes session and turn-source metadata through the exec approval path, but the native macOS approval prompt does not render the existing Session value or decode/display requester context.

## What This Changes

The issue requests adding session and requester/channel context to the native macOS exec approval popup.

## Best Possible Solution

Add the trimmed Session row first, then decode and display only maintainer-approved, sanitized requester context from existing turn-source metadata with focused native prompt coverage.

## Reproduction Assessment

Yes, at source level: current main passes sessionKey into the native prompt path and the web UI renders it, while buildAccessoryView never renders a Session row. I did not live-run the macOS alert in this read-only review.

## Solution Assessment

Partly. The Session row is the narrow maintainable first fix, but adding channel, sender, and thread context should be a reviewed product/security decision rather than ad hoc new fields.
