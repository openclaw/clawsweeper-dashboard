## Label Justifications

- `impact:session-state`: The popup cannot distinguish which session triggered a pending exec approval when multiple sessions are active.
- `impact:security`: Requester attribution affects whether a user can make an informed decision at the privileged exec approval boundary.

## Summary

Keep open: current main and v2026.6.6 still decode `sessionKey` for macOS exec approvals, but the native prompt does not render it and the Swift prompt model does not decode higher-level requester metadata.

## What This Changes

Request to show the exec approval session key and safe requester/channel context in the macOS native approval popup.

## Best Possible Solution

Add a trimmed non-empty Session row to the native presenter, then add Channel/Sender/Thread display only after a privacy-reviewed metadata contract maps raw turn-source fields to safe user-facing labels.

## Reproduction Assessment

Yes, at source level: current main passes and decodes `sessionKey`, but `buildAccessoryView` never renders it, and the Swift model lacks the richer turn-source fields. A live macOS dialog run was not needed to see the missing rows in the presenter construction path.

## Solution Assessment

Partly: rendering the existing session key is the narrowest maintainable parity fix. The requester/sender/channel portion needs a separate privacy and security decision before adding new native UI fields.
