## Summary

Keep this open: current main and the latest release still pass session and turn-source metadata through exec approval paths, but the native macOS prompt does not render the existing Session value or decode/display requester context. The Session row is source-reproducible; richer channel/sender/thread display needs a maintainer-approved privacy and security contract.

## What This Changes

The issue asks the macOS exec approval popup to show the triggering session and eventually richer requester context such as channel, sender, or tool reason.

## Best Possible Solution

Add the trimmed Session row to the native prompt first, then define a privacy-aware requester-context display contract from existing turn-source metadata with focused Swift and gateway coverage.

## Reproduction Assessment

Yes, at source level: current main passes sessionKey into the native prompt path and the web UI renders it, while macOS buildAccessoryView lacks a Session row and requester fields. I did not live-run the macOS alert in this read-only review.

## Solution Assessment

Partly. The Session row is the narrow maintainable first fix, but channel, sender, and thread context should be a reviewed product/security display contract rather than raw ad hoc fields.
