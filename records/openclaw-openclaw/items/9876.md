## Label Justifications

- `P3`: This is a bounded macOS approval-context enhancement with limited blast radius and no current runtime outage.
- `impact:session-state`: The native popup cannot distinguish which session triggered an exec approval when multiple sessions are active.

## Summary

Keep open: current main and the latest release still decode session/turn-source metadata for exec approvals, but the macOS native popup does not render the existing session key or any requester context. The narrow Session row is clear; exposing channel/sender/thread context needs maintainer privacy and security direction.

## What This Changes

The issue requests showing session identity and requester/channel context in the macOS exec approval popup.

## Best Possible Solution

Render a trimmed non-empty Session row from the existing sessionKey, then expose requester metadata only through a maintainer-approved safe display/redaction contract with focused macOS prompt coverage.

## Reproduction Assessment

Yes, at source level: current main decodes and passes sessionKey into macOS exec approval prompts, but buildAccessoryView does not add a Session row. The richer requester-context portion is also source-visible because TypeScript/protocol payload fields exist while Swift has no native fields for them.

## Solution Assessment

Partly: rendering the existing session key is the narrowest maintainable parity fix. The requester/channel/sender portion needs a privacy and security contract before new native UI fields should be added.
