## Summary

Keep this open. Current main still carries `sessionKey` and turn-source metadata through the exec approval path, but the macOS native prompt does not render the session and the Swift request model does not decode requester context; the closest related issues are broader or different approval-routing problems, not a canonical replacement.

## What This Changes

The request asks to show session and safe requester context in the macOS exec approval dialog.

## Best Possible Solution

Add the trimmed Session row first, then decode and display only maintainer-approved, sanitized requester context from existing turn-source metadata with focused macOS prompt coverage.

## Reproduction Assessment

Yes, at source level: current main passes `sessionKey` into the native prompt path and the web UI renders it, while `buildAccessoryView` never renders a Session row. I did not live-run the macOS alert in this read-only review.

## Solution Assessment

Partly. The Session row is the narrow maintainable first fix, but adding channel, sender, and thread context should be a reviewed product/security decision rather than ad hoc new fields.
