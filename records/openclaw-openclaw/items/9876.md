## Summary

Keep open. Current main still carries session metadata into the macOS exec approval request path, and the shared gateway payload already has turn-source fields, but the native macOS approval prompt does not render the session or decode/display requester context. The prior maintainer-facing comment in the discussion reached the same conclusion, and the current source still matches that gap.

## What This Changes

Request that the macOS exec approval dialog show the triggering session and requester/channel context so users can identify which session or chat surface caused an approval prompt.

## Best Possible Solution

Add native macOS prompt support for non-empty Session plus safe origin rows from the existing turnSource metadata, with a small testable formatter or debug hook for the rendered context rows. Defer human-friendly sender names or tool reasoning until the shared approval payload defines those fields across channels.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: construct an ExecApprovalPromptRequest with a non-empty sessionKey and the current buildAccessoryView path never reads or displays that field, while gateway and node prompt paths pass through the same presenter.

## Solution Assessment

Yes for the narrow direction. Rendering the existing sessionKey and decoding/displaying existing turnSource fields in the Swift prompt is the smallest maintainable fix; adding new senderName/toolReason fields should wait for a shared payload/API contract.
