- `impact:session-state`: The native approval popup cannot identify which session triggered a command when multiple sessions or channels are active.

## Summary

Keep open: current main and v2026.6.10 still carry session and turn-source metadata through exec approval paths, but the macOS native popup does not render the existing session key or requester context. The Session row is source-reproducible; richer channel/sender/thread display needs maintainer privacy and security direction.

## What This Changes

The issue requests adding session and requester context to the macOS exec approval popup so users can identify which session or channel triggered a command approval.

## Best Possible Solution

Render a trimmed non-empty Session row from the existing sessionKey, then add channel/sender/thread context only through a maintainer-approved safe display and redaction contract with focused macOS prompt coverage.

## Reproduction Assessment

Yes, at source level: current main decodes and passes sessionKey into macOS exec approval prompts, but buildAccessoryView does not add a Session row. The richer requester-context portion is also source-visible because TypeScript/protocol payload fields exist while Swift has no native fields for them.

## Solution Assessment

Partly: rendering the existing session key is the narrowest maintainable parity fix. The requester/channel/sender portion needs a privacy and security contract before new native UI fields should be added.
