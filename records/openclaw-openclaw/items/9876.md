## Label Justifications

- `P3`: This is a bounded macOS approval UX and security-context enhancement with limited blast radius and no current runtime outage.
- `impact:session-state`: The native popup cannot distinguish which session triggered a pending exec approval when multiple sessions are active.
- `impact:security`: Requester attribution affects whether a user can make an informed decision at the privileged exec approval boundary.

## Summary

Keep open: current main and v2026.6.8 still carry session and turn-source approval metadata, but the macOS native exec approval prompt does not render the existing session key and the Swift prompt model does not expose requester/channel fields.

## What This Changes

Request to show the triggering session and safe requester/channel context in macOS exec approval popups.

## Best Possible Solution

Add a trimmed non-empty Session row to the native presenter, then add Channel/Sender/Thread display only through a privacy-reviewed requester metadata contract with focused macOS prompt coverage.

## Reproduction Assessment

Yes, at source level: current main decodes sessionKey in the Swift request but buildAccessoryView does not add a Session row. The richer requester-context gap is also source-visible because TypeScript payload fields exist while the Swift prompt model ignores them.

## Solution Assessment

Partly: rendering the existing session key is the narrowest maintainable parity fix. The requester/channel/sender portion needs a privacy and security contract before adding new native UI fields.
