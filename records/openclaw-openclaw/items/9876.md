## Summary

Keep open: current main still decodes and forwards macOS exec approval session keys but the native popup does not render them, and the broader requester/channel/sender display remains a privacy and security product decision.

## What This Changes

The issue requests that the macOS exec approval popup show the triggering session plus safer requester/channel context.

## Best Possible Solution

Add the narrow trimmed Session row from the existing sessionKey, then design and test a safe requester metadata display contract before exposing channel/sender/thread context in the native prompt.

## Reproduction Assessment

Yes, at source level: current main decodes and passes sessionKey into macOS exec approval prompts, but buildAccessoryView does not add a Session row. The richer requester-context portion is also source-visible because TypeScript payload fields exist while Swift ignores them.

## Solution Assessment

Partly: rendering the existing session key is the narrowest maintainable parity fix. The requester/channel/sender portion needs a privacy and security contract before adding new native UI fields.
