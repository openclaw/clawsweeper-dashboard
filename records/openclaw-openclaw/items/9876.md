## Label Justifications

- `P3`: This is a bounded macOS approval-context enhancement with limited blast radius and no current runtime outage.
- `impact:session-state`: The native popup cannot distinguish which session triggered a pending exec approval when multiple sessions are active.
- `impact:security`: Requester attribution affects whether a user can make an informed decision at the privileged exec approval boundary.

## Summary

Keep open: current main still decodes macOS exec approval session keys but does not render them in the native popup, and the broader requester/channel display remains a privacy and security product decision.

## What This Changes

The issue requests that macOS exec approval popups display the triggering session and, where safe, requester/channel context.

## Best Possible Solution

Add the narrow trimmed Session row from the existing sessionKey first, then design safe requester metadata display from the gateway payload with privacy/security review and focused native prompt coverage.

## Reproduction Assessment

Yes, at source level: current main decodes and passes sessionKey into macOS exec approval prompts, but buildAccessoryView does not add a Session row. The richer requester-context portion is also source-visible because TypeScript payload fields exist while Swift ignores them, but no live NSAlert run was performed.

## Solution Assessment

Partly: rendering the existing session key is the narrowest maintainable parity fix. The requester/channel/sender portion needs a privacy and security contract before adding new native UI fields.
