## Label Justifications

- `P3`: This is a low-blast-radius macOS approval prompt enhancement rather than an urgent regression or crash.
- `impact:session-state`: The missing UI context makes it hard to distinguish which active session triggered an approval.
- `impact:security`: The affected surface is an exec approval security prompt where requester identity affects operator trust decisions.

## Summary

Keep this open. Current `main` still carries session and turn-source metadata through the exec approval path, but the macOS native prompt does not render the existing `sessionKey` or decode/display higher-level requester context; the earlier maintainer-facing review reached the same keep-open result and the related crash issue is a separate closed problem.

## What This Changes

Request that macOS exec approval popups show the triggering session and safe requester/channel context like the web approval UI.

## Best Possible Solution

Add the narrow trimmed Session row first, then extend the native request model to display only safe, sanitized requester context from existing turn-source metadata with focused macOS prompt coverage.

## Reproduction Assessment

Yes, at source level: current `main` passes `sessionKey` into the native prompt path and the web UI renders it, while `buildAccessoryView` never renders a Session row. I did not live-run the macOS alert in this read-only review.

## Solution Assessment

Partly. The Session row is a narrow maintainable first fix, but adding channel/sender/thread context should be a reviewed product and privacy decision rather than ad hoc new fields.
