## Label Justifications

- `P3`: This is a bounded macOS approval-UX enhancement with limited blast radius and no current runtime outage.
- `impact:session-state`: The popup cannot distinguish which session triggered a pending exec approval when multiple sessions are active.
- `impact:security`: Requester attribution affects whether a user can make an informed decision at the privileged exec approval boundary.

## Summary

Keep open: current main and v2026.6.6 still decode `sessionKey` but the macOS native prompt does not render it, and the richer requester metadata carried by the TypeScript approval payload is not decoded or displayed by the Swift prompt model.

## What This Changes

The issue requests that macOS exec approval dialogs show the session key and optional safe requester/channel context.

## Best Possible Solution

Add a trimmed, non-empty Session row to the native presenter first, then add Channel/Sender/Thread display only after a privacy-reviewed metadata contract maps raw turn-source fields to safe user-facing labels.

## Reproduction Assessment

Yes, at source level: current main passes and decodes `sessionKey`, but `buildAccessoryView` never renders it, and the Swift model lacks the richer turn-source fields. A live macOS dialog run was not needed to see the missing rows in the presenter construction path.

## Solution Assessment

Partly: rendering the existing session key is the narrowest maintainable parity fix. The requester/sender/channel portion needs a separate privacy and security decision before adding new native UI fields.
