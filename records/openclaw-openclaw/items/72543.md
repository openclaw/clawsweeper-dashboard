## Summary

Keep open. Current main still has the reported Control UI gap: the shared chat session option builder can omit the canonical main session when it is not present in the returned session rows and is not the active session, even though the Gateway snapshot already exposes the canonical key.

## What This Changes

The issue requests that the Control UI chat session dropdown always include and normalize the canonical main assistant session, keyed as agent:main:main or the configured main-session equivalent.

## Best Possible Solution

Add a shared Control UI session-option path that derives the canonical main key from Gateway sessionDefaults, pins that option into the grouped selector for desktop and mobile even when sessions.list omits it, and keeps alias normalization aligned for selected and persisted settings without changing the Gateway protocol.

## Reproduction Assessment

Yes. A focused unit repro can call resolveSessionOptionGroups with active sessionKey set to a subagent session, a sessionsResult that lacks agent:main:main, and state.hello containing sessionDefaults.mainSessionKey; current code will return only discovered rows plus the active subagent key.

## Solution Assessment

Yes. The narrow source-level fix belongs in the shared Control UI option builder plus existing default-session normalization path; adding a new Gateway API or hard-coding a persona label would be broader than needed.
