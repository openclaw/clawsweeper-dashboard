## Summary

Keep open. Current main still has the Control UI gap: the shared chat session option builder is used by both desktop and mobile selectors, but it only adds discovered session rows plus the currently selected key, so it cannot surface the canonical main session when `sessions.list` omits it while the active key is a subagent or another non-main session.

## What This Changes

The issue requests that the Control UI chat dropdown always include a stable canonical main-session option for `agent:main:main` and normalize plain `main` aliases consistently across selected and persisted session state.

## Best Possible Solution

Add shared Control UI option-building logic that derives the canonical main key from Gateway session defaults, injects one stable main-session option before or alongside discovered rows, and keeps desktop/mobile selectors plus existing alias normalization aligned without adding a new Gateway API.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resolveSessionOptionGroups` with active session `agent:main:subagent:...`, a `sessionsResult` that lacks `agent:main:main`, and a `state.hello.snapshot.sessionDefaults.mainSessionKey`; current main returns discovered rows plus the active key, leaving the canonical main option absent.

## Solution Assessment

Yes. The narrow maintainable repair belongs in the shared Control UI session option builder and should reuse existing Gateway default-session data; the safer alternative to the issue's example is to use generic agent/session metadata for the label instead of hard-coding `Chelmsworth`.
