## Summary

This should stay open: current main and the latest release still leave Qwen/DashScope image requests outside the prompt-in-user-content path, so the image tool can send a system prompt plus an image-only user message. The linked older Qwen issues were endpoint/model-resolution fixes, not this DashScope multimodal payload-shape failure.

## What This Changes

The issue requests a fix for Qwen/DashScope vision requests failing with a 400 schema error when the image tool sends multimodal content.

## Best Possible Solution

Teach the model-backed image runtime to send prompt text in user content for Model Studio/DashScope vision requests, add focused regression coverage, and verify with a redacted live DashScope image request when credentials are available.

## Reproduction Assessment

Yes for a source-level reproduction: configure qwen/dashscope as an OpenAI-compatible text+image model, invoke the image tool, and current main can build a system prompt plus image-only user content. Live reproduction of the provider 400 still needs DashScope credentials.

## Solution Assessment

Yes, the best fix appears to be a narrow provider-capability payload rule in the shared model-backed image runtime. It should not add a config option, change Coding Plan catalog suppression, or hardcode unverified qwen3.7 catalog rows.
