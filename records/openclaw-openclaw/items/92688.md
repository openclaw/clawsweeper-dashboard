## Summary

This should stay open: current main and the latest release still build Qwen/DashScope image-tool requests as a system prompt plus image-only user content, while the provider contract and reporter logs point to a rejected multimodal message schema. The prior Qwen fixes addressed model/catalog routing, not this request payload shape.

## What This Changes

The issue reports that Qwen/DashScope OpenAI-compatible vision requests from the image tool fail with a 400 schema error because DashScope rejects the request content shape.

## Best Possible Solution

Keep this issue open and repair the Qwen/DashScope image path so the prompt text is sent beside image_url content for OpenAI-compatible Qwen vision requests, with focused regression coverage and live DashScope proof when credentials are available.

## Reproduction Assessment

## Solution Assessment

Yes, the best fix is a narrow provider-capability payload repair in the model-backed image runtime or Qwen provider wrapper. It should not add a config option, change catalog suppression, or hardcode new qwen3.7 catalog rows without separate vendor-backed catalog evidence.
