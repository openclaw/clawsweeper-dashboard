## Summary

This should stay open: current main still sends Qwen/DashScope image-tool prompts as a separate system prompt while the user message can contain only image content, and the reported DashScope 400 is a credible provider-schema failure on that path. The prior linked Qwen issues covered catalog/endpoint routing, not this image request payload shape.

## What This Changes

The issue asks OpenClaw to make Qwen/DashScope vision models work with the image tool when configured as OpenAI-compatible text+image models.

## Best Possible Solution

Teach the Qwen/DashScope image path to send the prompt text in the user content alongside the image_url payload, add focused regression coverage, and verify with a DashScope live image request when credentials are available.

## Reproduction Assessment

Yes, source-level reproduction is clear: configure qwen/dashscope as an OpenAI-compatible text+image model, invoke the image tool, and current main builds a system prompt plus image-only user content. I did not run a live DashScope request because provider credentials are required.

## Solution Assessment

Yes, a narrow provider-capability fix in the model-backed image runtime looks like the best path; it should not add a config option, change catalog suppression, or hardcode qwen3.7 catalog entries without vendor-backed model data.
