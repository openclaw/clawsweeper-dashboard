## Summary

Keep this PR open for a focused repair: config-owned executable helper commands are a reasonable hardening boundary, but the patch also silently replaces the unrelated `browserProfile` request value, and the supplied proof does not exercise a real paired Chrome node.

## What This Changes

The PR marks `googlemeet.chrome` as dangerous and adds a Gateway-side plugin policy that replaces Chrome-node start request fields with Google Meet plugin configuration, plus tests and documentation.

## Best Possible Solution

Make plugin config authoritative only for the executable audio helper command fields, preserve supported non-command start parameters such as `browserProfile`, add direct-invocation regression coverage, and verify a redacted paired-node Chrome/audio start before merge.

## Reproduction Assessment

Yes. On the PR head, a `googlemeet.chrome` start request with a caller-selected `browserProfile` deterministically reaches `buildGoogleMeetChromeStartParams`, where line 377 replaces that value with plugin configuration before node forwarding.

## Solution Assessment

No, not as currently scoped. Config ownership is appropriate for executable helper commands, but replacing `browserProfile` is unrelated to that boundary and should be removed unless maintainers explicitly choose a broader breaking contract.
