## Summary

Keep open for maintainer landing: current main still has the Google Chat auth-token transport path implicated by the linked regression, while this PR addresses that path with focused plugin-local code, regression coverage, green checks, and sufficient live Google Chat proof.

## What This Changes

The PR changes Google Chat auth transport to use `google-auth-library`'s exported `Gaxios`, preserves native `Request` data through the SSRF fetch wrapper, updates dependency contracts/tests, and adds a changelog entry.

## Best Possible Solution

Land the plugin-local auth transport fix after routine maintainer review, preserving current-main package metadata during landing, then let the merge close https://github.com/openclaw/openclaw/issues/77307.

## Reproduction Assessment

Yes. The linked issue provides the live `unsupported_grant_type` symptom, and current main has a source-visible token-request path where auth request data can be lost before this PR's transport changes; I did not run an independent live Google Chat setup.

## Solution Assessment

Yes. Reusing `google-auth-library`'s exported `gaxios` and preserving `Request` data inside the existing SSRF-guarded Google Chat transport is the narrow plugin-owned fix; no core API or config change is needed.
