## Summary

This should stay open: current source still routes package updates through the detected package manager's global root, and the Windows pnpm custom-prefix path described by the report is not covered by existing target-resolution logic.

## What This Changes

The issue asks for `openclaw update` to update the OpenClaw installation under `OPENCLAW_HOME` or the current executable location instead of npm's global prefix.

## Best Possible Solution

Teach the package update target resolver to prefer the running package root for valid custom global-install layouts, especially Windows pnpm prefixes, while preserving existing npm staged-install verification and state-home semantics.

## Reproduction Assessment

Yes for source-level reproduction: mock a Windows package root like `D:\openclaw\node_modules\openclaw`, make `pnpm root -g` return a different global root, and the current path will still build a plain `pnpm add -g` update against the manager default. I did not run a live Windows pnpm install in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is in update target resolution: use the current running package root or a derived package-manager prefix when it is a valid global install, instead of adding a new config knob or changing general `OPENCLAW_HOME` state behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78377. On current main, `openclaw update` package mode finds the running OpenClaw package root, then resolves the update target through `resolveGlobalManager` and `resolveGlobalInstallTarget`; for a Windows pnpm custom install rooted at `D:\openclaw\node_modules\openclaw`, if `pnpm root -g` points elsewhere, the updater can install to the default npm/pnpm global root instead of the running package root. Implement a narrow resolver change so package updates prefer the current running package root when it is a valid global-install layout, especially Windows pnpm custom prefixes, and ensure the pnpm command/env targets that root. Do not redefine `OPENCLAW_HOME` as a general install prefix, do not add a new config option, and preserve npm staged install verification/swap behavior. Add regression coverage for the win32 custom pnpm-prefix mismatch and keep existing npm and Windows pnpm tests green. Add a single-line changelog fix entry because this is user-facing.
