## Summary

Keep open. The prior ClawSweeper comment is still source-backed on current main: bundled runtime-deps staging still prefers pnpm without `CI=true` or a `confirmModulesPurge=false` override, while both install paths run with non-interactive stdin, which matches pnpm's abort condition.

## What This Changes

The issue reports that bundled plugin runtime dependency staging fails under non-TTY launchd/systemd service starts because pnpm aborts module purges without CI or a purge-confirmation override.

## Best Possible Solution

Land a narrow runtime-deps package-manager fix that makes OpenClaw-owned bundled dependency staging explicitly non-interactive for pnpm, updates the focused package-manager tests, and records the user-facing service-startup fix in the changelog and dependency-resolution docs if the documented command changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run the bundled runtime-deps pnpm install path with an existing modules directory under non-TTY stdin and without `CI` or `confirmModulesPurge=false`; current main supplies that shape and pnpm source throws the reported error.

## Solution Assessment

Yes. The narrowest maintainable fix is to add a pnpm-specific `--config.confirm-modules-purge=false` or equivalent shared install environment override for these internal non-interactive installs, without changing broader plugin activation, repair timing, or service installation behavior.
