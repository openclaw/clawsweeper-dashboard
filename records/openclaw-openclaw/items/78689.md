Keep open: current main still has a plausible Feishu direct-session native Codex delivery gap, and I did not find a merged PR, shipped fix, or canonical open replacement that proves Feishu direct replies now deliver or surface a clear delivery error.

## What This Changes

The issue reports that Feishu direct-session replies disappear when a direct-session agent is switched from the PI/OpenAI-Codex route to the native Codex harness, even though explicit native Codex smokes succeed.

## Best Possible Solution

Keep the Codex message-tool default for ordinary native Codex turns, but make Feishu direct text commands mark the explicit text-command source and surface message-tool send failures with visible diagnostics.

## Reproduction Assessment

No. There is not a high-confidence live reproduction in this review; source inspection on current main still shows the Feishu direct inbound context lacking `CommandSource` while Codex direct replies can be `message_tool_only`, and the issue body supplies logs plus rollback proof.

## Solution Assessment

Unclear until a live Feishu/Codex run separates automatic-reply suppression from the reported HTTP 400 target/account failure. The narrow maintainable direction is Feishu command metadata and delivery-error surfacing, not disabling the Codex harness default globally.
