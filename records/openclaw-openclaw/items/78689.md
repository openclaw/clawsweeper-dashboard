Keep open: current main has partial Codex-side improvement, but the Feishu direct-session failure is not proven fixed and still has a plausible source path involving Codex tool-only delivery plus Feishu command/message-target handling.

## What This Changes

The issue reports that Feishu direct-session replies and `/status` stopped reaching the user when the session was switched from PI/OpenAI-Codex to the native Codex harness.

## Best Possible Solution

Keep the Codex message-tool default, but make Feishu direct text commands and direct message-tool sends either deliver visibly or return a clear Feishu delivery error.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. Source inspection still shows a plausible path where Feishu direct text turns lack `CommandSource: "text"` and inherit native Codex `message_tool_only` delivery, while the reported HTTP 400 needs live Feishu validation.

## Solution Assessment

Unclear until a live Feishu run separates command-source suppression from account/target send failure. The narrow maintainable direction is Feishu-specific metadata, target resolution, or delivery-error surfacing rather than rolling back to PI or changing the global Codex default.
