## Summary

Keep open: current main still has a plausible Feishu direct-session failure path for native Codex, with Codex direct replies defaulting to message-tool-only delivery and Feishu text inbound contexts not marking `/status` as an explicit text command. I did not find a shipped or current-main fix that proves Feishu direct replies now deliver or surface a user-visible delivery error.

## What This Changes

Report a Feishu direct-session delivery failure after switching an agent from the PI/OpenAI-Codex route to the native Codex harness.

## Best Possible Solution

Keep the Codex harness default intact, but make authorized Feishu direct text commands visibly deliver and make Feishu message-tool/direct delivery failures surface clearly instead of looking like a successful private Codex run.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a medium-confidence current-main path where Feishu direct text turns lack `CommandSource: "text"`, so native Codex direct sessions can suppress automatic `/status` replies under the harness `message_tool` default; the reported HTTP 400 still needs live Feishu proof.

## Solution Assessment

Unclear until a live Feishu run separates command-source suppression from message-tool target/account failure. The narrow maintainable direction is to fix Feishu source-delivery metadata/error surfacing, not to roll users back to PI or disable the Codex harness default globally.
