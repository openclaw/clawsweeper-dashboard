## Summary

Keep open: current main still returns HTML/code-formatted Telegram progress preview text for non-rich drafts, the latest release still used code-styled progress rows, and no related open PR safely owns this exact plain-readable fallback bug.

## What This Changes

The issue asks for Telegram progress draft previews to use plain readable status text instead of HTML/code-span formatting.

## Best Possible Solution

Keep one plain readable progress line model as the canonical draft text, and derive Telegram rich/HTML presentation from that model only where the transport can use it safely.

## Reproduction Assessment

Yes from source inspection: current Telegram progress draft tests still expect `telegramHtmlPreview(...)` payloads containing `<b>`, `<code>`, and `<br>` for ordinary progress rows. I did not run the Vitest command because this review must stay read-only and avoid artifact-writing commands.

## Solution Assessment

Yes, the suggested direction is the likely best fix: keep `TelegramDraftPreview.text` plain and derive rich HTML separately. Changing the shared progress-line builder or adding a new config knob would be broader than needed.
