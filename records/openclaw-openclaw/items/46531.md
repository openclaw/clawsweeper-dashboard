## Summary

Keep open. Current main still permits an env-backed model provider SecretRef to be persisted from the invoking shell while gateway startup resolves active provider refs against the daemon runtime and fails hard when the daemon cannot see the env var.

## What This Changes

The issue asks OpenClaw to prevent model/provider setup from persisting daemon-unresolvable Moonshot/Kimi SecretRefs that can crash-loop gateway startup.

## Best Possible Solution

Add a daemon-aware provider credential activation boundary: validate against the service runtime or persist through managed daemon-readable storage before commit, then reject, disable, or roll back failed secondary provider additions without weakening required-secret fail-fast semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY absent from the daemon runtime; current collection marks it active and startup activation throws the required-secrets failure.

## Solution Assessment

Not applicable as a patch. Current main has useful install-time and gateway-RPC guardrails, but the narrow maintainable fix still needs a daemon-aware mutation/activation policy rather than a blanket relaxation of SecretRef startup checks.
