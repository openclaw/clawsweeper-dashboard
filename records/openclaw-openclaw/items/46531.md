## Summary

Keep open. Current main still has a source-reproducible setup/runtime mismatch: shell-guided config writes can persist an env-backed provider SecretRef from the interactive environment, while gateway startup resolves active provider refs in the daemon environment and fails hard when that env var is absent.

## What This Changes

The issue asks OpenClaw to prevent model-assisted Moonshot/Kimi setup from persisting a daemon-invisible `${MOONSHOT_API_KEY}` SecretRef that can crash-loop the gateway.

## Best Possible Solution

Add a daemon-aware provider credential activation boundary: validate against the service runtime or safely persist through managed daemon-readable storage before commit, then reject, disable, or roll back failed secondary provider additions without weakening fail-closed required-secret semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an enabled `models.providers.moonshot.apiKey` env SecretRef with `MOONSHOT_API_KEY` present in the setup shell but absent from the daemon runtime; current collection marks it active and startup activation throws the required-secrets failure.

## Solution Assessment

Not applicable as a patch. The narrow maintainable direction is daemon-aware validation or rollback around provider credential activation, not a blanket relaxation of startup SecretRef failures.
