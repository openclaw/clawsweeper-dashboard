## Summary

Keep open: current main still has the underlying SecretRef setup/runtime mismatch outside the guarded gateway RPC write path, and the linked fixing PR is open rather than merged. The issue discussion correctly identifies that env refs can be validated in one process while daemon startup resolves them in another, and https://github.com/openclaw/openclaw/pull/84454 is the active closing path.

## What This Changes

The issue reports that model-assisted Moonshot/Kimi setup can persist an env-backed provider SecretRef that the gateway daemon cannot resolve, causing gateway startup to fail in a restart loop.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84454 or an equivalent maintainer-approved provider SecretRef activation policy that preflights daemon-visible refs before persistence and preserves rollback/recovery for failed secondary-provider setup.

## Reproduction Assessment

Yes for the deterministic lower-level path: an active models.providers.<id>.apiKey env SecretRef with the env var absent from the daemon runtime is collected, resolution throws, and startup wraps it as a fatal secret failure. The exact model-assisted shell recommendation was not live-reproduced.

## Solution Assessment

Unclear until maintainers accept the policy. The linked PR is a focused implementation path for pre-commit SecretRef preflight and rollback, but the fail-closed behavior for active provider refs needs maintainer approval before the issue can close.
