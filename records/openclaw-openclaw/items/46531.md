## Summary

Keep this open: current main still has a source-reproducible path where an enabled model provider env SecretRef that is not visible to the gateway runtime makes startup fail, and the remaining fix needs a maintainer decision about daemon-aware validation versus optional-provider degradation.

## What This Changes

The issue asks OpenClaw to prevent model-assisted Moonshot/Kimi setup from persisting an env-backed provider secret that the systemd gateway service cannot resolve and then crash-looping the gateway.

## Best Possible Solution

Define the provider-secret activation policy first, then implement daemon-aware preflight or rollback so persisted model/provider changes cannot leave enabled env SecretRefs unresolved at the next gateway startup.

## Reproduction Assessment

Yes for the lower-level failure path: an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY missing from the gateway runtime will be collected, fail env resolution, and make startup throw. The model-assisted conversation itself is nondeterministic, but the persisted config failure path is clear from source and tests.

## Solution Assessment

Unclear. The report lists several possible fixes, and the safest path needs a maintainer decision between daemon-environment preflight, service-env persistence, write rollback, or treating unused provider refs as inactive rather than startup-fatal.
