## Summary

Keep open: current main partially blocks unresolved SecretRefs on gateway RPC config writes and can copy config env refs into service install environments, but active provider SecretRefs are still startup-fatal when unresolved, so the daemon setup/runtime mismatch remains unresolved.

## What This Changes

The issue asks OpenClaw to prevent model/provider setup from persisting daemon-unresolvable Moonshot/Kimi SecretRefs that crash-loop gateway startup.

## Best Possible Solution

Define a provider SecretRef activation policy that validates against the daemon runtime or makes secondary provider refs degradable/rollback-safe while preserving strict failures for required credentials.

## Reproduction Assessment

Yes for the deterministic lower-level path: an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY absent from the gateway runtime is collected, fails env resolution, and is wrapped as a startup failure. The model-assisted bash path itself was not live-reproduced.

## Solution Assessment

Unclear. The existing RPC preflight and service-env preservation are partial mitigations, but maintainers still need to choose the startup/setup policy before a narrow repair is safe.
