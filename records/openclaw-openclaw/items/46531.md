## Summary

Keep open. Current main has useful partial guardrails, but the reported daemon/runtime mismatch remains: provider setup can still persist env-backed credentials based on the invoking shell, ordinary non-dry-run CLI config writes can still save them without active SecretRef preflight, and enabled model provider refs remain startup-required when the daemon cannot resolve them.

## What This Changes

This issue asks OpenClaw to prevent assistant/TUI/CLI model-provider setup from saving daemon-unresolvable Moonshot/Kimi env SecretRefs that can crash-loop the gateway.

## Best Possible Solution

Add a daemon-aware activation boundary for provider credential mutations: model-provider setup should either write secrets through an OpenClaw-managed daemon-visible store or validate against the gateway service runtime before commit, and failed secondary provider additions should be rejected, disabled, or rolled back before startup can break.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist an enabled models.providers.moonshot.apiKey env SecretRef while MOONSHOT_API_KEY is absent from the daemon runtime; current collection treats it as active and startup activation throws the required-secrets failure.

## Solution Assessment

Not applicable as a patch. Current partial safeguards are insufficient because RPC config writes and install-time env propagation do not cover interactive/provider setup, normal non-dry-run CLI writes, or post-install shell-guided mutations.
