## Summary

Keep open: current main still has the daemon/runtime mismatch at the lower-level SecretRef path. RPC config writes and daemon install planning add partial guardrails, but an active provider env SecretRef that is absent from the gateway runtime still makes startup fail, so the remaining fix needs maintainer policy on validation, rollback, or degradable secondary providers.

## What This Changes

The issue asks OpenClaw to prevent model-assisted Moonshot/Kimi setup from persisting a daemon-unresolvable provider SecretRef that can crash-loop the local gateway.

## Best Possible Solution

Define a provider SecretRef activation policy that either validates against the daemon runtime before persistence or keeps unresolved secondary provider refs from taking down an otherwise valid gateway, with rollback or clear recovery behavior.

## Reproduction Assessment

Yes for the deterministic lower-level path: an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY absent from the gateway runtime is collected, fails env resolution, and is wrapped as a startup failure. The model-assisted bash path itself was not live-reproduced.

## Solution Assessment

Unclear. Current main has useful partial mitigations for RPC config writes and daemon install env capture, but the best remaining fix requires a maintainer policy choice for unresolved active provider refs at startup.
