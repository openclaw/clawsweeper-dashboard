## Summary

Keep open. Current main has useful install-time and gateway-RPC guardrails, but the reported post-install daemon mismatch remains possible: env-backed model provider refs can still be persisted from the invoking shell, and enabled provider refs remain startup-fatal when the daemon cannot resolve them.

## What This Changes

Prevent model/provider setup from saving daemon-unresolvable env SecretRefs or crash-looping the gateway when a secondary provider credential is missing.

## Best Possible Solution

Add a daemon-aware activation boundary for provider credential mutations: validate against the service runtime or write through managed daemon-readable storage before commit, then reject, disable, or roll back failed secondary provider additions before startup can break.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY absent from the daemon runtime; current collection marks it active and startup activation throws the required-secrets failure.

## Solution Assessment

Not applicable as a patch. Current main's install-time env carry-forward and gateway-RPC preflight are useful but do not cover post-install shell-guided mutations, ordinary non-dry-run CLI config writes, or the enabled-provider startup-fatal path.
