## Summary

Keep open: current main has helpful install-time mitigation for env-backed SecretRefs, but the reported unsafe post-install/config-mutation path is still plausible because config writes can store env refs and gateway startup still treats unresolved active provider secrets as fatal.

## What This Changes

The issue reports that assistant-guided Moonshot/Kimi setup can save an env-backed provider secret that resolves in an interactive shell but not in the gateway daemon, causing gateway startup failure.

## Best Possible Solution

Align provider setup with daemon runtime by either storing env-backed provider secrets in daemon-readable state, blocking unsafe activation before restart, or narrowly degrading unavailable non-required providers without taking down the gateway.

## Reproduction Assessment

Yes at source level, but not as a live systemd run. Current code can write env SecretRefs through config mutation, active model provider apiKey refs are collected for runtime secrets, and startup still fails fast when required secret resolution fails.

## Solution Assessment

No for closing this as already solved. The current install-time service-env bridge is useful, but it does not fully address model-assisted or config-set changes made after a daemon is already installed; maintainers still need to choose the safest boundary for blocking, service-env refresh, or scoped degradation.
