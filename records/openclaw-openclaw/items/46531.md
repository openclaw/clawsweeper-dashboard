## Label Justifications

- `P1`: A daemon-invisible provider SecretRef can make the local gateway unavailable in a restart loop for real users.
- `impact:auth-provider`: The issue centers on provider credential SecretRef resolution across setup and gateway runtime environments.
- `impact:crash-loop`: The reported failure mode is a systemd gateway restart loop caused by startup secret resolution failure.

## Summary

Keep open: current main still has the setup/runtime SecretRef mismatch. Setup can validate an env ref against the current session, while gateway startup resolves active provider refs against the daemon runtime and still fails fast when that env var is unavailable.

## What This Changes

Prevent model-assisted Moonshot/Kimi setup from persisting daemon-unresolvable provider SecretRefs that can crash-loop the gateway.

## Best Possible Solution

Define a provider SecretRef activation policy that prevents daemon-invisible refs from being persisted or activated, with clear rollback/recovery behavior for secondary providers.

## Reproduction Assessment

Yes for the deterministic lower-level path: an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY absent from the gateway runtime is collected, fails env resolution, and is wrapped as a startup failure. The exact model-assisted bash suggestion was not live-reproduced.

## Solution Assessment

Unclear. Current main has partial RPC-write guardrails, but the best remaining fix requires maintainers to choose the intended policy for unresolved active provider refs at startup and setup time.
