## Summary

Keep open: current main has added gateway config-write SecretRef preflight, but the underlying startup contract still fails fast when an enabled provider SecretRef is unresolved, and the remaining question is a product boundary decision rather than conservative cleanup.

## What This Changes

The issue reports that model-assisted Moonshot/Kimi setup can persist a daemon-unresolvable env SecretRef and leave the gateway crash-looping.

## Best Possible Solution

Define the provider-secret activation policy, then route model-assisted provider setup through daemon-runtime preflight or add daemon-visible storage/rollback/degradation for enabled provider refs.

## Reproduction Assessment

Yes for the deterministic lower-level path: an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY absent from the gateway runtime is collected, fails env resolution, and startup throws. The model-assisted chat path itself is nondeterministic and was not live-reproduced.

## Solution Assessment

Unclear. The current RPC preflight is a good partial fix, but the remaining issue requires maintainers to choose between daemon-visible setup validation, service-env persistence, rollback, or changing fail-fast behavior for enabled secondary providers.
