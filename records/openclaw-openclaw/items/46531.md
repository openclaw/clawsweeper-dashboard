## Summary

Keep open: current main still has the source-reproducible lower-level failure where an enabled model provider env SecretRef that is missing from the gateway runtime is collected, resolution fails, and startup throws the fatal required-secrets error. The remaining fix is not a simple cleanup close because maintainers need to choose the product boundary between daemon-aware validation, service-env persistence, rollback, or optional-provider degradation.

## What This Changes

The issue reports that model-assisted Moonshot/Kimi setup can persist an env-backed provider secret ref that is invisible to the gateway daemon and causes a startup crash loop.

## Best Possible Solution

Define the provider-secret activation policy, then implement daemon-aware validation or rollback so model/provider setup cannot persist an enabled env SecretRef that the managed gateway cannot resolve on next startup.

## Reproduction Assessment

Yes for the deterministic lower-level path: an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY absent from the gateway runtime is collected, fails env resolution, and makes startup throw. The model-assisted chat path itself is nondeterministic and was not live-reproduced.

## Solution Assessment

Unclear. The report lists several plausible fixes, and the safest maintainable answer needs a maintainer decision between daemon-environment preflight, service-env persistence, write rollback, or degrading unused provider refs instead of failing startup.
