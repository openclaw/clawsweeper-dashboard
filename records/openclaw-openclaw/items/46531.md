## Summary

Keep open: current main now rejects unresolved SecretRefs on gateway RPC config writes and can preserve some config env refs during service install, but an already-persisted active provider SecretRef is still startup-fatal, so the setup/runtime policy gap is not fully resolved.

## What This Changes

The issue reports that model-assisted Moonshot/Kimi setup can persist a `${MOONSHOT_API_KEY}` provider SecretRef that the systemd gateway daemon cannot resolve, causing a gateway crash loop.

## Best Possible Solution

Define and implement the provider SecretRef activation policy so model/provider setup cannot persist daemon-unresolvable refs that brick startup, while preserving fail-closed behavior for truly required secrets.

## Reproduction Assessment

Yes for the deterministic lower-level path: an enabled models.providers.moonshot.apiKey env SecretRef with MOONSHOT_API_KEY absent from the gateway runtime is collected, fails env resolution, and is wrapped as a startup failure. The model-assisted bash path itself was not live-reproduced.

## Solution Assessment

Unclear. The current RPC preflight is a good partial fix, but the remaining daemon/startup behavior requires maintainers to choose between service-env persistence, daemon-runtime preflight, rollback, or optional-provider degradation.
