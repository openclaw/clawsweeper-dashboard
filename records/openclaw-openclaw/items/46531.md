## Summary

Keep open. Current main still has the reported model-provider SecretRef failure shape: enabled model providers are treated as active secret surfaces, unresolved refs are resolved during startup activation, and startup activation still aborts the gateway on resolution failure; adjacent open work covers channels, auth profiles, or retry/backoff, but not this model-provider daemon-environment mismatch.

## What This Changes

The issue requests preventing model-assisted Moonshot/Kimi setup from saving a daemon-unresolvable `${MOONSHOT_API_KEY}` provider secret that can crash-loop the gateway.

## Best Possible Solution

Define a model-provider SecretRef activation policy that validates against the managed gateway environment before persisting setup changes, or degrades only demonstrably unused provider credentials while preserving fail-fast behavior for selected/required credentials.

## Reproduction Assessment

Yes, at source level. Configure an enabled `models.providers.moonshot.apiKey` env SecretRef with `MOONSHOT_API_KEY` absent from the gateway process env, then run startup secret activation; current main collects that ref as active and wraps the missing env error as a fatal startup failure.

## Solution Assessment

Unclear. The current behavior is internally consistent with the documented active-SecretRef fail-fast contract, so maintainers need to choose the narrow policy: daemon-env preflight/storage during setup, model-provider degradation, or a selected-model-only requiredness rule.
