The issue asks for `SESSIONS_HISTORY_TEXT_MAX_CHARS` from config/env to control `sessions_history` per-entry text truncation instead of the fixed 4000-character cap.

## Best Possible Solution

If maintainers accept the knob, add a documented bounded sessions-history text-cap setting that preserves the 4000 default and 80 KiB hard cap, clamps invalid values, and covers valid and invalid overrides in regression tests.
