## Summary

Keep open. Current main still hardcodes the sessions_history per-text-block limit at 4000 characters, so the reported env value cannot affect truncation; because the env var is not currently documented as a supported option, this needs maintainer decision before implementation.

## What This Changes

Allow `SESSIONS_HISTORY_TEXT_MAX_CHARS` from runtime env/config env to control the `sessions_history` text truncation cap instead of the hardcoded 4000-character limit.

## Best Possible Solution

Add a documented, bounded sessions-history text cap setting or env reader, preserve the 4000 default and 80 KiB hard response cap, and add regression coverage for an override above 4000.

## Reproduction Assessment

Yes, by source inspection. With `SESSIONS_HISTORY_TEXT_MAX_CHARS=16000`, any recalled text longer than 4000 characters still reaches the constant-only `truncateHistoryText` path on current main.

## Solution Assessment

Partly. Reading an env value is the likely implementation direction if maintainers accept the new env contract, but the safe form should use the existing clamped pattern rather than a bare `readEnvInt(...) ?? 4000`.
