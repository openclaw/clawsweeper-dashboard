## Summary

Keep open. Current main and latest release still use a fixed 4000-character `sessions_history` text cap, while accepting `SESSIONS_HISTORY_TEXT_MAX_CHARS` would create a new supported env/config contract that needs maintainer approval before an automated fix.

## What This Changes

The issue requests making `SESSIONS_HISTORY_TEXT_MAX_CHARS` control the `sessions_history` per-message truncation limit instead of the current fixed 4000-character cap.

## Best Possible Solution

Add a documented, bounded sessions-history text-cap setting only if maintainers accept the config contract; otherwise explicitly decline the env knob while keeping the bounded-history behavior documented.

## Reproduction Assessment

Yes, by source inspection: setting `SESSIONS_HISTORY_TEXT_MAX_CHARS` cannot affect `sessions_history` on current main because the tool never reads it and truncates through a literal 4000-character constant. This proves the reported symptom, while the expected env behavior still needs contract acceptance.

## Solution Assessment

Unclear as submitted. A bare `readEnvInt` would not be the safest fix; if maintainers accept the knob, the maintainable path is a clamped env/config reader with docs and regression tests while preserving the existing hard byte cap.
