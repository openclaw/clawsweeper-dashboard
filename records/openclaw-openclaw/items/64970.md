## Summary

Keep open. Current main and the latest release still apply a fixed 4000-character sessions_history text cap, and the linked fix PR was closed unmerged; because `SESSIONS_HISTORY_TEXT_MAX_CHARS` is not currently a documented supported knob, the remaining action needs maintainer config/product acceptance before implementation.

## What This Changes

Requests an env/config-backed override for sessions_history per-text truncation so values such as `SESSIONS_HISTORY_TEXT_MAX_CHARS=16000` are honored.

## Best Possible Solution

Add a documented, bounded sessions-history text-cap setting or explicitly decline the knob; if accepted, preserve the 4000 default and 80 KiB hard cap, use the clamped env-reader pattern, and add regression coverage for valid and invalid overrides.

## Reproduction Assessment

Yes, by source inspection. On current main, setting `SESSIONS_HISTORY_TEXT_MAX_CHARS` cannot affect `sessions_history` output because the tool never reads that env var and always truncates through the literal 4000-character constant.

## Solution Assessment

Unclear. A clamped env reader is the narrow implementation if maintainers accept the setting, but exposing `SESSIONS_HISTORY_TEXT_MAX_CHARS` as a supported config/env contract is a product decision first.
