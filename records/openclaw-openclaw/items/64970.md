## Summary

Keep open. Current main still ignores `SESSIONS_HISTORY_TEXT_MAX_CHARS`: the sessions-history tool hardcodes the 4000-character text cap, while the comparable bash output caps already use env-backed parsing and clamping. The related fix PR was closed unmerged, and an exact open-PR search did not find a replacement.

## What This Changes

The issue requests that `SESSIONS_HISTORY_TEXT_MAX_CHARS` configured in OpenClaw env be honored for `sessions_history` text truncation instead of always using the hardcoded 4000-character cap.

## Best Possible Solution

Implement a narrow agent-tools fix: make the per-entry `sessions_history` text cap read `SESSIONS_HISTORY_TEXT_MAX_CHARS` through the existing env-int and clamp helpers, preserve the current 4000 default and 80 KiB overall hard cap, keep redaction behavior unchanged, and add regression coverage for an override above 4000.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with `SESSIONS_HISTORY_TEXT_MAX_CHARS=16000`, any `sessions_history` response containing a text block longer than 4000 characters still follows the constant-only path and truncates around 4000 characters.

## Solution Assessment

Yes, if implemented with clamping. The narrowest maintainable fix is to mirror the existing bash output limit pattern rather than using a bare `readEnvInt(...) ?? 4000`, so zero or negative values cannot silently empty history output.
