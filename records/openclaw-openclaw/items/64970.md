## Summary

Keep open. Current main still ignores `SESSIONS_HISTORY_TEXT_MAX_CHARS`: the sessions-history tool uses a hardcoded 4000-character per-text cap, while the comparable bash output limit path reads and clamps env overrides. The related fix PR was closed unmerged, so there is no shipped or pending implementation to close against.

## What This Changes

The issue requests making the `sessions_history` per-text truncation limit honor `SESSIONS_HISTORY_TEXT_MAX_CHARS` instead of always using the hardcoded 4000-character cap.

## Best Possible Solution

Implement a narrow agent-tools fix: read `SESSIONS_HISTORY_TEXT_MAX_CHARS` through the existing env-int helper with a safe clamp and the current 4000 default, keep the 80 KiB overall history cap unchanged, add focused regression coverage for an override above 4000, and document or changelog the env knob if maintainers want it treated as user-facing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with `SESSIONS_HISTORY_TEXT_MAX_CHARS=16000`, any `sessions_history` response containing a text block longer than 4000 characters still follows the constant-only path and truncates around 4000 characters.

## Solution Assessment

Yes, with clamping. The narrowest maintainable fix is to mirror the existing `readEnvInt` plus `clampWithDefault` pattern used by bash output limits while preserving sessions-history redaction and hard byte caps.
