## Summary

Keep open. Current main still hardcodes the `sessions_history` per-entry text cap at 4000 characters, while config env vars are applied to the runtime environment and comparable bash output caps already use env-backed clamping. The earlier fix PR #65029 was closed unmerged, and exact open-PR searches found no replacement.

## What This Changes

Make `sessions_history` honor `SESSIONS_HISTORY_TEXT_MAX_CHARS` instead of always truncating per-entry text at 4000 characters.

## Best Possible Solution

Implement the narrow agent-tools fix with env-backed clamping, keep the 4000 default and 80 KiB hard cap, preserve redaction behavior, and add regression coverage for an override above 4000.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: with `SESSIONS_HISTORY_TEXT_MAX_CHARS=16000`, any recalled text block longer than 4000 characters still follows the constant-only truncation path.

## Solution Assessment

Yes. The best fix is to mirror the existing bash cap pattern with `clampWithDefault(readEnvInt(...), 4000, 1, ...)`, rather than a bare `readEnvInt(...) ?? 4000`, so invalid zero or negative values cannot silently empty history output.
