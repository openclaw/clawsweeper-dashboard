## Summary

Keep this PR open. Current main still does not provide `agents.defaults.systemPromptFile`, so the proposed feature is not obsolete, but the patch is not ready to merge because it leaves public config metadata/docs stale, has an unresolved prompt-file mutability boundary, and misses CLI session invalidation for file changes. The broader #50103 prompt-rules issue is relevant API context, not a merged replacement.

## What This Changes

The PR adds an optional `agents.defaults.systemPromptFile` config key, reads the configured local file, prepends its contents to CLI and embedded runner system prompts, and adds helper tests for file-reading behavior.

## Best Possible Solution

Land a maintainer-reviewed prompt-default design that either tightens this PR or folds it into #50103: define explicit path/trust semantics, include prompt-file identity or content in CLI session invalidation, update generated config schema/help/docs and docs, cover both runner paths with focused tests, and add a user-facing changelog entry if the feature ships.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no `agents.defaults.systemPromptFile` schema/runtime path, and the CLI reuse bug follows from hashing `params.extraSystemPrompt` before the PR's later file read while resumed CLI sessions do not resend first-session system prompts.

## Solution Assessment

No. The hook point is plausible, but this is not the best merge shape until public config metadata/docs, prompt-file trust semantics, and CLI session invalidation are handled; #50103 may also be the right place to settle the final API contract.

  - confidence: 0.86
