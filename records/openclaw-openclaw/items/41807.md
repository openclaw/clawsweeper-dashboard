## Summary

Keep this PR open. Current main still does not implement `agents.defaults.systemPromptFile`, so the contribution is not obsolete, but the patch is not ready to merge because it leaves public config metadata/docs stale, lacks a clear immutable prompt-file boundary, and misses CLI session invalidation for prompt-file changes.

## What This Changes

This PR adds an optional `agents.defaults.systemPromptFile` config key, reads the configured local file into CLI and embedded runner system prompts, and adds helper tests for file-reading behavior.

## Best Possible Solution

Land a maintainer-reviewed prompt-default design that either tightens this branch or folds the work into #50103: define explicit path and mutability semantics, include prompt-file identity or content in CLI session invalidation, update generated config schema/help/docs and user docs, cover both runner paths with focused tests, and add a user-facing changelog entry if the feature ships.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no `agents.defaults.systemPromptFile` schema/runtime path, and the CLI reuse issue follows from hashing `params.extraSystemPrompt` before the PR's later file read while resumed CLI sessions skip first-session system prompts.

## Solution Assessment

No. The hook point is plausible, but this is not the best merge shape until public config metadata/docs, prompt-file trust semantics, CLI session invalidation, and changelog coverage are handled; #50103 may also be the right place to settle the final API contract.
