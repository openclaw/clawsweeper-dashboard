## Summary

Keep open: this maintainer-labeled MEMBER PR needs owner review, and the current diff still risks using QQ C2C passive reply capacity with 5s input-notify calls before final answer delivery.

## What This Changes

The PR changes QQBot C2C typing keepalive from a 60s window renewed every 50s to a 10s window renewed every 5s, adds a fake-timer test, and adds a changelog entry.

## Best Possible Solution

Budget or cap C2C typing renewals against the same passive reply envelope, or land the shorter cadence only with contract/live proof that `input_notify` is exempt, plus a long-response regression test.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection confirms the PR would send repeated 5s `input_notify` requests on the same C2C `msg_id` until stop, but the exact quota/UI effect needs live or contract proof.

## Solution Assessment

No. Shortening the cadence alone is not the safest fix because it does not coordinate with the QQ passive-reply budget already modeled in the plugin; capping/budgeting renewals or proving exemption is safer.
