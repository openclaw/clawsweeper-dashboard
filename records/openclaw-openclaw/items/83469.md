## Label Justifications

- `P2`: This is a focused QQBot behavior fix with limited blast radius, but it needs owner review before merge.
- `merge-risk: 🚨 compatibility`: Existing long C2C responses that currently fit the passive-reply budget could start failing after the renewal cadence increases tenfold.
- `merge-risk: 🚨 message-delivery`: The diff can consume repeated message-endpoint sends before the real QQBot reply, which may cause the final message to be rejected or lost.

## Summary

Keep open: the PR is focused, but the new 5s QQBot C2C typing renewal can repeatedly post `input_notify` for the same inbound `msg_id` before the final reply, while the current QQBot code documents and guards a four-passive-reply-per-hour budget. That delivery-risk needs a cap, shared budgeting, or live/API proof that `input_notify` is exempt before merge.

## What This Changes

The PR changes QQBot C2C typing keepalive from a 60s window renewed every 50s to a 10s window renewed every 5s, and adds a fake-timer unit test for one renewal and stop behavior.

## Best Possible Solution

Budget or cap C2C typing renewals so the final reply remains deliverable, or provide live/API proof that QQ `input_notify` renewals do not consume passive reply quota, then keep the 5s/10s behavior under a long-response regression test.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection shows the PR would send unbounded 5s `input_notify` renewals on the same C2C `msg_id` until final delivery, but the exact QQ API quota effect needs live or contract proof.

## Solution Assessment

No. Shortening the cadence alone is not the safest fix because it does not account for the QQ passive-reply budget already modeled in the plugin; the safer path is to cap/budget renewals or prove the notify calls are exempt.
