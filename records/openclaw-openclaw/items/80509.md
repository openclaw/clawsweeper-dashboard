## Summary

Keep open: current main documents and implements queued steering at the next model boundary, not user-message preemption during a long tool loop, so the requested mid-turn interrupt behavior remains a real feature gap that needs runtime/product design.

## What This Changes

Request mid-turn interruption for incoming user messages during long active agent tool loops, with a visible notice and resumable/superseding behavior.

## Best Possible Solution

Design an OpenClaw-owned mid-turn interrupt policy that extends the existing queue/steer/interrupt contract while preserving tool-result ordering, transcript consistency, and Pi/Codex harness boundaries.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request. Current source and docs do confirm the existing behavior that queued user input is handled at model boundaries or by coarse abort, not by mid-tool user-message preemption.

## Solution Assessment

Unclear: the requested direction is valid, but polling active channels between tool calls plus heuristic supersede/resume behavior needs maintainer design. The safer path is to define the semantics in the existing queue/steering contract before implementation.
