## Summary

Keep this PR open. The auth-bridge direction matches the linked regression, but the current head still has a concrete credential-handling blocker: when symlinks fail, it copies Codex `auth.json` into the OpenClaw state tree, and current backup creation archives that state tree with no exclusion for the bridged credential.

## What This Changes

The PR bridges host Codex `auth.json` into the ACPX isolated Codex home, adds ACPX regression coverage, updates ACP docs and the changelog, and adjusts one Codex plugin test type.

## Best Possible Solution

Keep the narrow `auth.json` bridge direction, but make the credential location backup-safe before merge: prefer symlink-only behavior or add an explicit backup exclusion/redaction with regression coverage proving copied Codex auth cannot enter default archives. After that, rebase and rerun the focused ACPX tests plus the changed gate.

## Reproduction Assessment

Yes. The linked issue and duplicate context give a high-confidence product repro for the original bug, and current main confirms the managed Codex wrapper forces an isolated `CODEX_HOME` that only receives generated config. The PR-introduced security finding is also reproducible from the diff: force symlink failure, observe the fallback copy under state, then follow the backup archive path that includes state recursively.

## Solution Assessment

No. Bridging only Codex `auth.json` while keeping config hooks isolated is the right narrow product fix, but this implementation is not the best mergeable form while the copy fallback can persist credential material inside the default backup source tree.
