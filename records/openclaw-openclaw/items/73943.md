## Summary

Keep open: the PR addresses a real current-main ACPX/Codex auth gap, but the proposed fallback copy path is not ready to merge because it can persist Codex credentials under the OpenClaw state tree that default backups archive.

## What This Changes

The PR bridges host Codex `auth.json` into ACPX's isolated Codex home, keeps generated Codex config isolated, updates ACP docs/changelog, adds ACPX regression tests, and adjusts one Codex plugin test type.

## Best Possible Solution

Keep the host-auth reuse direction, but make the credential bridge backup-safe before merge: use symlink-only behavior, store copied credentials in a backup-aware credential location, or add a precise backup exclusion with regression coverage while preserving Codex config-hook isolation.

## Reproduction Assessment

Yes. Current main can be verified by inspection for the original bug, and the PR's backup exposure is reproducible by forcing the symlink path to fail so `auth.json` is copied under `<stateDir>/acpx/codex-home` before running backup archive creation.

## Solution Assessment

No. Bridging only `auth.json` is the narrow right product direction, but copying it into the OpenClaw state tree without a backup exclusion is not the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep copied Codex auth out of backups:** `extensions/acpx/src/codex-auth-bridge.ts:281`
  - body: When symlink creation fails, this fallback writes the host Codex auth file into `<stateDir>/acpx/codex-home/auth.json`. `openclaw backup create` includes the state directory and only filters `state/extensions/**/node_modules`, so routine backups can contain Codex tokens. Use symlink-only behavior, a backup-aware credential location, or an explicit backup exclusion with coverage before merging.
  - confidence: 0.91
