## Summary

Current main treats rich cards as the generic `presentation` contract, while raw provider-native `card` is intentionally outside the shared message tool surface. The reported duplicate path appears tied to an external/plugin-owned raw-card implementation, and promoting raw `card` to first-class core send content would need maintainer API/product judgment rather than an automated cleanup fix.

## What This Changes

The issue asks OpenClaw to prevent duplicate plain text when `message(action="send")` includes both text and a provider-native `card` parameter.

## Best Possible Solution

Keep provider-native cards plugin-owned and steer shared sends to `presentation`, unless maintainers intentionally add a documented, versioned raw-card contract.

## Reproduction Assessment

No. The duplicate text-plus-card behavior depends on an external raw-card plugin path, while current bundled Feishu evidence shows `presentation` renders as one card and raw `card` is not a shared send content type.

## Solution Assessment

No. The proposed core normalization is not the best fix because deleting `message` when only raw `card` remains breaks current send validation; plugin-owned handling or a deliberate `presentation`/raw-card contract decision is safer.
