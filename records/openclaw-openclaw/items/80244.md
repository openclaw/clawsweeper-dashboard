## Summary

Close as plugin-owned/ClawHub work: current OpenClaw core intentionally keeps provider-native card fields out of the shared message tool, already provides plugin seams for native cards, and the linked PR discussion converged on a plugin-level fix rather than core normalization.

## What This Changes

The issue asks OpenClaw core to prevent duplicate text before provider-native cards when `message` or `text` and `card` are supplied to the message tool.

## Best Possible Solution

Keep OpenClaw core on the generic `presentation` and `channelData` contract, and have the Lark/Feishu plugin suppress standalone text when sending native cards; a new raw-card core contract should only proceed through maintainer design review.

## Reproduction Assessment

Yes for the external plugin source path, but not as a current OpenClaw core defect. The target repo source shows raw `card` is plugin-owned, while the reported duplicate behavior depends on the external Lark plugin sending text before its native card.

## Solution Assessment

No. Core normalization is not the best fix because it would delete the required message text before raw `card` is recognized as content; the maintainable fix is plugin-owned card delivery or a separate product decision to add a new first-class card contract.
