## Summary

Keep open. The report is plausible from the shared message-action source path, but the exact duplicate text-plus-card behavior was not demonstrated in bundled Feishu on current main and the requested blanket core normalization conflicts with the current presentation/plugin-boundary docs.

## What This Changes

The issue requests that `message(action="send")` treat `message`/`text` as card notification metadata, not a separate text send, when a native `card` payload is also present.

## Best Possible Solution

Define the compatibility contract for native `card` params, then implement any Feishu/Lark-specific suppression or card routing inside the channel/plugin boundary, reusing `presentation` or `prepareSendPayload` rather than adding provider-specific parsing to shared core.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection shows shared core can forward both `message` and provider-native `card` data to plugin-owned send paths, but the exact duplicate behavior depends on the Lark/Feishu plugin implementation and was not demonstrated in bundled Feishu current main.

## Solution Assessment

No. The proposed blanket core deletion of `message`/`text` when `card` is present is too broad for the current contract; a safer fix should be Feishu/Lark-owned or contract-guided so provider-specific fallback text semantics are preserved.
