## Summary

Current main addresses both operational failures: status/read-only channel discovery no longer needs Feishu's full runtime SDK in core, and Discord stale-socket health now uses provider transport activity instead of quiet app-event freshness.

## What This Changes

The issue reports that `openclaw status` crashes in `openclaw@2026.4.21` when Feishu's missing package loads, and that Discord providers repeatedly restart with `stale-socket` while reconnecting.

## Best Possible Solution

Keep the current main behavior: Feishu remains a separately installed plugin dependency surface, and Discord health remains based on real transport activity rather than quiet application traffic.

## Reproduction Assessment

Yes, by source and package metadata: the historical crash path was a packaged Feishu runtime dependency missing from core, and the Discord loop matched stale-socket health evaluation without provider transport activity. I did not live-run the old macOS package in this read-only review.

## Solution Assessment

Yes, the current solution is the narrow maintainable fix: move Feishu SDK ownership to the Feishu plugin package and feed Discord transport heartbeat/message activity into the generic health policy.
