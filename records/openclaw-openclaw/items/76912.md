## Summary

Current main addresses both operator-facing failures: status no longer depends on eagerly loading the Feishu runtime, and Discord health now uses explicit gateway transport activity instead of ordinary app activity for stale-socket decisions.

## What This Changes

The issue reports that `openclaw status` crashes on `2026.4.21` when Feishu’s missing package dependency is loaded, and that Discord channels repeatedly restart for `stale-socket` while the gateway remains reachable.

## Best Possible Solution

Keep the current main behavior and ship it; operators on `2026.4.21` should move to a build after the proof timestamp and use the managed `@openclaw/feishu` plugin install path for Feishu.

## Reproduction Assessment

Yes for the reported release from the issue evidence and source path: an old root-packaged Feishu runtime could require a missing dependency during diagnostics. I did not establish a current-main crash; current main’s package, CLI routing, read-only setup loading, and Discord transport-liveness code no longer match the reported failure mode.

## Solution Assessment

Yes. The maintainable fix is the current split: Feishu’s heavy SDK belongs to the Feishu plugin package, read-only status avoids full runtime preload and degrades on setup failures, and Discord stale-socket evaluation uses explicit websocket transport activity.
