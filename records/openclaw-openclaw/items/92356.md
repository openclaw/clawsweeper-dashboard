## Summary

Keep open: the PR fixes the flagged `isReasoning` leak and the earlier runner early-return regression, but it still leaves existing legacy/blockquote reasoning shapes eligible as the main heartbeat reply when `includeReasoning` is false.

## What This Changes

The PR updates heartbeat reply selection to ignore `isReasoning` payloads, preserves opt-in reasoning-only delivery in the runner, and adds selector/runner regression tests.

## Best Possible Solution

Align the heartbeat selector with the existing SDK/runner reasoning classifier, then land one canonical PR with regressions for flagged and prefixed reasoning plus opt-in includeReasoning delivery.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main selects trailing outbound reasoning because it only checks sendable content, and the PR body includes before/after live selector output for the flagged shape.

## Solution Assessment

Is this the best way to solve the issue? Not yet: selector-level filtering is the right layer and the runner early-return fix is good, but the best fix should reuse the existing reasoning classifier so legacy and blockquoted reasoning are not still eligible as main replies.
