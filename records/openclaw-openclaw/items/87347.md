## Summary

Keep open: current main still contains the agent-side provider gate that can route manifest-owned web_search providers away from the live runtime registry, and the linked fix with closing syntax is still an open PR. The issue should close only after that fix, or a superseding one, lands.

## What This Changes

Issue requests a fix so agent-side web_search in long-running gateway sessions resolves the same loaded Brave provider that CLI infer and plugin inspection see.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/88684 with focused regression coverage and live or redacted gateway-agent proof, then close this issue from the merged fix.

## Reproduction Assessment

Yes, at source level: current main can pass `preferRuntimeProviders: false` for a manifest-owned configured search provider before `runWebSearch` reaches the loaded runtime registry. I did not establish the full 3-6 hour gateway reproduction live in this read-only sweep.

## Solution Assessment

Likely yes, but the fix is still in an open PR. Keeping agent web_search runtime-first while leaving web_fetch’s bundled-owner gate intact is narrower than duplicating provider ownership policy at the agent boundary.
