## Label Justifications

- `P1`: The report describes a production regression where channel-driven agent web_search can stop resolving the configured provider until a disruptive gateway restart.

## Summary

Keep open: current main and v2026.6.8 still contain the agent-side manifest-owner gate that can route a configured Brave web_search provider away from the live runtime registry, and the linked open PR with closing syntax is the active fix candidate.

## What This Changes

Agent-side web_search should keep using the loaded runtime Brave provider during long-lived gateway sessions instead of reporting no provider while CLI web search and plugin inspection still work.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/88684 with focused regression coverage and redacted live provider-path proof, then close this issue from the merged fix.

## Reproduction Assessment

Yes at source level: current main can pass preferRuntimeProviders: false for a manifest-owned configured Brave search provider before runWebSearch uses the live runtime registry. I did not establish the full 3-6 hour gateway reproduction live in this read-only sweep.

## Solution Assessment

Likely yes: keeping agent web_search runtime-first while leaving web_fetch's bundled-owner gate intact is the narrowest current fix shape. The actual solution should come through the existing linked PR or a superseding PR rather than a duplicate repair branch.
