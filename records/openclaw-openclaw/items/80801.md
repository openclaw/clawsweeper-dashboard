## Summary

Keep this PR open, but it needs a narrow fix before merge: the new `--force` path can log that cached profiles were removed even when the production store helper reports lock/write failure by returning `null`. The rest of the change is plausible and has supplied terminal proof, so this is a repairable PR review finding rather than a cleanup close.

## What This Changes

The PR adds `openclaw models auth login --force`, auth-class fallback remediation hints, a session-scoped fallback skip cache with tests, a probe script, and a changelog entry.

## Best Possible Solution

Keep the PR open, make the `--force` purge failure handling honor the helper's `null` result, and then proceed through normal maintainer review for the new flag and fallback policy.

## Reproduction Assessment

Yes for source-level review: current main lacks the new flag/cache/hint behavior, and the PR bug is traceable because the store updater returns `null` on failure while the PR logs success after a non-throwing purge call. I did not run live OAuth in this read-only checkout.

## Solution Assessment

No, not yet: the design is mostly maintainable, but `modelsAuthLoginCommand` must treat a `null` purge result as the same non-blocking failure path before the `--force` behavior is reliable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle null purge failures before logging success:** `src/commands/models/auth.ts:777-783`
  - body: `removeProviderAuthProfilesWithLock` is backed by `updateAuthProfileStoreWithLock`, which catches lock/write failures and returns `null` instead of throwing. This await will fall through to the success log on that failure path, so `--force` can tell the user profiles were removed while stale credentials remain and the auth flow may still short-circuit; check for `null` and emit the warning path before continuing.
