## Summary

Keep this PR open, but it should not merge yet: the current diff has two source-reproducible correctness issues and the real behavior proof is still incomplete for the destructive CLI path.

## What This Changes

The PR adds a `models auth login --force` CLI flag, auth failover remediation hints, and a session-scoped fallback skip cache with tests and a changelog entry.

## Best Possible Solution

Revise the PR so profile deletion happens only after a valid auth method is selected, expired skip buckets are globally bounded or cleared on session teardown, and the PR body includes redacted real CLI proof for `--force`.

## Reproduction Assessment

Yes for the PR-introduced bugs: source inspection shows `--force` purges before auth-method resolution can fail, and the new skip cache only prunes the queried session bucket. The original operator rough edges have helper/unit proof, but the PR still lacks live CLI proof for the `--force` purge path.

## Solution Assessment

No; the approach is close but not safe enough because it can delete credentials before later validation fails and can retain expired one-off session buckets indefinitely. Moving the purge immediately before `runProviderAuthMethod` and bounding or clearing the skip cache is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve the auth method before purging profiles:** `src/commands/models/auth.ts:689-693`
  - body: `--force` deletes cached profiles immediately after provider selection, but `pickProviderAuthMethod` can still return `null` for an unknown `--method` or canceled method prompt. That path removes the user's working credentials and then throws before any auth flow runs, so purge only after a valid method is selected and immediately before `runProviderAuthMethod`.
  - confidence: 0.92
- **[P2] Prune expired skip buckets across sessions:** `src/agents/fallback-skip-cache.ts:105-112`
  - body: The TTL only removes expired entries when the same session bucket is queried. A gateway that sees many one-off sessions with auth fallback failures will keep stale buckets in the global map past their TTL, so prune globally on writes/checks or clear the cache from session teardown.
  - confidence: 0.86
