## Summary

Keep open: the patch has blocking correctness issues around destructive auth-profile purging and skip-cache lifetime, and the external PR proof does not demonstrate the real `--force` CLI auth path.

## What This Changes

Adds a `models auth login --force` flag, auth-failover remediation hints, and an in-memory session-scoped fallback skip cache with tests and a changelog entry.

## Best Possible Solution

Revise the PR so destructive purging happens only after a valid auth method is selected, stale skip markers are globally bounded or cleared on session teardown, and real redacted proof covers the CLI `--force` path.

## Reproduction Assessment

Yes for the PR-introduced purge-order bug: source inspection shows the new `--force` deletion block runs before auth-method resolution, and that resolution can return null and throw. The original auth rough edges have helper/unit proof, but no live `--force` CLI reproduction in the PR body.

## Solution Assessment

No; the current approach is close but not safe enough because it deletes credentials before later validation can fail and leaves TTL cache buckets retained across one-off sessions. Moving the purge immediately before `runProviderAuthMethod` and bounding or clearing the cache is the narrower maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve the auth method before purging profiles:** `src/commands/models/auth.ts:689-692`
  - body: `--force` deletes cached profiles immediately after provider selection, but `pickProviderAuthMethod` can still return `null` for an unknown `--method` or a canceled method prompt. That path removes the user's working credentials and then throws before any auth flow runs; resolve the method first and purge only immediately before `runProviderAuthMethod`.
  - confidence: 0.92
- **[P2] Prune expired skip buckets across sessions:** `src/agents/fallback-skip-cache.ts:49-56`
  - body: The TTL only removes expired entries when the same session bucket is queried. A gateway that sees many one-off sessions with auth fallback failures will keep those stale buckets in the global map indefinitely, so the cache can grow beyond its 60s window; prune globally on writes/checks or clear it from session teardown.
  - confidence: 0.86
