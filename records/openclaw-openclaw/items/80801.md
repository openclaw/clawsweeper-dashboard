## Summary

Keep open: the PR addresses a plausible auth/fallback rough edge, but the current patch has two merge-blocking correctness issues and still lacks real proof for the destructive `--force` CLI path.

## What This Changes

Adds `openclaw models auth login --force`, appends auth remediation hints to fallback summaries, and introduces a session-scoped fallback skip cache with tests and a changelog entry.

## Best Possible Solution

Revise the PR so profile purging happens only after a valid auth method is selected, stale skip entries are globally bounded or cleared on session teardown, and redacted real CLI proof covers the `--force` path.

## Reproduction Assessment

Yes for the PR-introduced bugs: source inspection shows the purge runs before method validation, and the new cache only prunes expired entries inside a queried session bucket. The original auth rough edges are plausible but only partially proven by helper probes and unit tests in the PR body.

## Solution Assessment

No; the current patch is close, but deleting credentials before auth-method validation and retaining stale session buckets are not the narrowest maintainable implementation. Moving the purge immediately before `runProviderAuthMethod` and adding global pruning or session teardown cleanup would address the concrete risks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve the auth method before purging profiles:** `src/commands/models/auth.ts:689-692`
  - body: `--force` deletes cached profiles immediately after provider selection, but `pickProviderAuthMethod` can still return `null` for an unknown `--method` or a canceled method prompt. That path removes the user's credentials and then throws before any auth flow runs, so move the purge until after a valid method is selected and immediately before `runProviderAuthMethod`.
  - confidence: 0.92
- **[P2] Prune expired skip buckets across sessions:** `src/agents/fallback-skip-cache.ts:65-70`
  - body: The cache only prunes expired entries when the same session bucket is queried. A gateway that sees many one-off sessions with auth fallback failures will keep those stale buckets in the global map after the TTL, so prune globally on writes/checks or wire `clearFallbackSkipCacheForSession` into session teardown.
  - confidence: 0.86
