## Summary

Keep open: the implementation has two blocking source-level defects, and the real behavior proof does not cover the new destructive `--force` CLI path against a real profile store.

## What This Changes

The PR adds `models auth login --force`, auth-class fallback remediation hints, and a session-scoped fallback skip cache for repeated auth failures.

## Best Possible Solution

Keep the auth UX direction, but validate the auth method before any destructive purge, wire session attribution through real fallback callers, and prove the fixed CLI path with redacted live or terminal output.

## Reproduction Assessment

Yes for the PR defects from source inspection: the purge runs before method validation, and the new skip cache requires `params.sessionId` while the direct CLI fallback caller does not pass it. I did not run the branch because this review is read-only.

## Solution Assessment

No; the proposed direction is useful, but this patch should not merge until the destructive purge is reordered and the skip cache is wired into the actual session path it is meant to improve.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate the auth method before purging profiles:** `src/commands/models/auth.ts:683-690`
  - body: With this ordering, `openclaw models auth login --provider openai --method typo --force` removes the provider's cached profiles and only then fails when `pickProviderAuthMethod` returns no match. That turns a typo into credential loss without running a replacement auth flow, so select and validate the method before calling `removeProviderAuthProfilesWithLock`.
  - confidence: 0.91
- **[P2] Pass the session id into fallback before relying on the cache:** `src/agents/model-fallback.ts:901-903`
  - body: The new cache only checks and records skips when `params.sessionId` is set, but the direct CLI agent path calls `runWithModelFallback` with `runId` and `agentDir` while passing `sessionId` only to the inner attempt. Normal CLI turns therefore keep retrying the same broken fallback candidate on later turns; thread the session id through that caller and cover it with a regression test.
  - confidence: 0.84
