## Summary

Keep open: this is a valid Codex runtime bug-fix PR, but the patch still has a blocking compatibility gap because newly canonicalized fingerprints are compared byte-for-byte against order-sensitive fingerprints already written by current main.

## What This Changes

The PR updates the Codex app-server runtime, tests, and changelog to stabilize dynamic-tool fingerprints, preserve bindings during no-tool maintenance turns, and project mirrored history when starting without a reusable native binding.

## Best Possible Solution

Land the Codex-plugin fix after making fingerprint compatibility tolerate pre-existing order-sensitive stored bindings while still invalidating real schema changes.

## Reproduction Assessment

Yes. By source inspection, current main clears Codex bindings on order-sensitive dynamic-tool fingerprint changes and only projects mirrored history when the context engine is active; I did not run a live or test reproduction in this read-only pass.

## Solution Assessment

No, not as written. The fix direction is narrow and owner-local, but it should normalize legacy stored fingerprint arrays before comparison so existing sessions do not lose native binding on the first upgraded turn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize legacy fingerprints before comparing:** `extensions/codex/src/app-server/thread-lifecycle.ts:306`

Parse legacy array fingerprints and compare their sorted entries before invalidating.
