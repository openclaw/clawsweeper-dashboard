## Summary

Keep open. Current main still has a source-level regression path: Anthropic Claude CLI setup writes canonical Anthropic model refs plus the legacy whole-agent runtime key, while runtime selection ignores that legacy key and only routes CLI execution from model/provider runtime policy. A second user confirmed the symptom, and targeted GitHub searches did not find a canonical duplicate or open fix PR.

## What This Changes

The issue reports a regression after upgrading from 2026.5.7 to 2026.5.12 where gateway-mediated Telegram/Dashboard Claude CLI turns fail or use Anthropic embedded billing while direct CLI still works.

## Best Possible Solution

Update Anthropic Claude CLI auth/migration/defaulting so canonical Anthropic model refs keep Claude CLI execution through model-scoped or provider-wildcard runtime policy, without re-enabling legacy whole-agent runtime selection.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current Claude CLI migration can emit canonical Anthropic model entries without model-scoped claude-cli runtime policy, and current runtime selection ignores the legacy agents.defaults.agentRuntime key. A live gateway/Telegram repro was not run.

## Solution Assessment

Yes, the narrow repair is to make the Anthropic CLI setup/migration write the runtime policy shape that current routing already consumes. Re-enabling whole-agent runtime keys would conflict with the documented runtime contract.
