## Summary

Keep open. Current main still has a plausible source-level regression path for Claude CLI auth setup: the Anthropic CLI migration writes canonical Anthropic model refs and a legacy whole-agent runtime key, while runtime selection now intentionally ignores whole-agent runtime keys and only routes to CLI backends from provider/model-scoped runtime policy.

## What This Changes

The issue reports a regression after upgrading to 2026.5.12 where gateway-mediated Telegram and UI turns stop using the expected Claude CLI path while direct CLI usage still works.

## Best Possible Solution

Make Claude CLI auth setup and legacy migration write provider/model-scoped or provider-wildcard runtime policy for canonical Anthropic refs, while keeping whole-agent runtime keys ignored.

## Reproduction Assessment

Yes for a source-level reproduction path: Claude CLI auth setup can produce canonical Anthropic defaults without model-scoped `claude-cli` runtime policy, and current runtime routing then falls through to the non-CLI provider path. A live gateway/Telegram repro was not run in this read-only review.

## Solution Assessment

No current fix is present. The narrow fix is to update Anthropic CLI auth/migration output and regression tests so canonical Anthropic model refs retain Claude CLI execution through model/provider runtime policy, not through ignored whole-agent runtime fields.
