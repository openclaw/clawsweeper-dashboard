## Summary

Keep open: current main still reads and snapshots HEARTBEAT.md before run-kind filtering can omit it, and changing that path is a prompt/bootstrap contract decision rather than conservative cleanup.

## What This Changes

The issue requests loading HEARTBEAT.md only for heartbeat runs instead of reading it in every workspace bootstrap load.

## Best Possible Solution

Define run-kind-aware bootstrap load/cache semantics, then skip HEARTBEAT.md before disk reads and cache snapshots for ordinary non-heartbeat runs while preserving real heartbeat behavior and documenting the harness-specific prompt contract.

## Reproduction Assessment

Yes for source-level behavior: current main still loads HEARTBEAT.md in the raw workspace bootstrap loader/cache before any normal-run filter can omit it. I did not run a live heartbeat/session path because this is a feature/performance contract request rather than a failing runtime bug.

## Solution Assessment

Unclear until maintainers settle the prompt contract. Passing run-kind state into the loader/cache is the likely narrow technical shape, but it changes documented HEARTBEAT.md bootstrap semantics and overlaps broader selective/lazy context work.
