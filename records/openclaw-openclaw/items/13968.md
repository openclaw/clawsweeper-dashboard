## Summary

This issue still appears valid on current main: HEARTBEAT.md is loaded and cached before run-kind filtering, and the current default-agent prompt contract can still include heartbeat guidance on ordinary runs. The related heartbeat light-context and broader selective/lazy context issues are adjacent but do not supersede this narrower request.

## What This Changes

Request to skip HEARTBEAT.md during workspace bootstrap loading for non-heartbeat runs so it is loaded only for actual heartbeat polls.

## Best Possible Solution

Add a run-kind-aware bootstrap load/cache path that omits HEARTBEAT.md before disk reads, snapshot caching, hooks, and prompt injection for ordinary non-heartbeat default runs, while preserving it for real heartbeat runs and updating focused tests plus docs.

## Reproduction Assessment

Yes from source inspection: current main still reads and caches the fixed bootstrap file set, including HEARTBEAT.md, before run-kind filtering. I did not run a live heartbeat/default-message scenario in this read-only review.

## Solution Assessment

Unclear without maintainer product confirmation: the run-kind-aware loader/cache split is the narrow technical shape, but it intentionally changes whether normal default-agent runs receive HEARTBEAT.md when heartbeat guidance is enabled.
