## Summary

Keep open. Current main still reads and snapshots the fixed workspace bootstrap file set, including `HEARTBEAT.md`, before later run-kind and heartbeat-guidance filters; the request is narrower than the broader lazy-context trackers and needs maintainer product judgment because it changes the documented normal-run contract.

## What This Changes

The issue requests run-kind-aware bootstrap loading so `HEARTBEAT.md` is skipped for normal non-heartbeat runs and preserved for heartbeat runs.

## Best Possible Solution

Keep this as the narrow `HEARTBEAT.md` tracker until maintainers decide the normal-run contract, then implement pre-load/cache filtering keyed by run kind with focused tests and heartbeat/system-prompt docs updates.

## Reproduction Assessment

No high-confidence bug reproduction is applicable because this is an enhancement, not broken current behavior. The eager-load behavior is high-confidence from source: the resolver gets raw bootstrap files from the cache/loader before filters, and the loader still enumerates `HEARTBEAT.md`.

## Solution Assessment

Yes, if maintainers accept the behavior change: the narrow maintainable solution is run-kind-aware raw loading and cache snapshots before disk reads. It should not be automated until the documented normal-run `HEARTBEAT.md` contract is decided.
