## Summary

Current main still has the reported behavior: HEARTBEAT.md is loaded in the fixed bootstrap file set before run-kind filtering, cached without run-kind input, and still documented as present in normal non-Codex runs when heartbeat guidance is enabled. The broader selective/lazy context issue and the open tiered-bootstrap PR are related, but neither is a merged canonical fix for this narrower on-demand HEARTBEAT.md loading request.

## What This Changes

Request to skip loading HEARTBEAT.md for ordinary non-heartbeat runs and load it only for real heartbeat runs.

## Best Possible Solution

Define run-kind-aware bootstrap loader/cache semantics that skip HEARTBEAT.md before disk reads, session snapshots, hooks, and prompt injection for ordinary non-heartbeat runs, while preserving real heartbeat behavior and updating docs/tests for the new prompt contract.

## Reproduction Assessment

Yes for source-level behavior: current main still reads and caches HEARTBEAT.md in the raw bootstrap path before any normal-run filter can omit it. I did not run a live heartbeat/session path because this is a feature/performance contract request rather than a failing runtime bug.

## Solution Assessment

Unclear until maintainers settle the prompt contract. Passing run-kind state into the loader/cache is the likely narrow technical shape, but it changes documented HEARTBEAT.md bootstrap semantics and overlaps broader selective/lazy context work.
