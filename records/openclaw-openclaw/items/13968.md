## Summary

Keep open: current main and the latest release still load and cache `HEARTBEAT.md` before any run-kind filter can omit it, and the remaining change needs a maintainer product/API decision rather than conservative cleanup.

## What This Changes

The issue requests loading `HEARTBEAT.md` only for heartbeat runs so ordinary non-heartbeat messages avoid unnecessary bootstrap context.

## Best Possible Solution

Define and document run-kind-aware bootstrap load/cache semantics that skip `HEARTBEAT.md` before disk reads and cache snapshots for ordinary non-heartbeat runs, or explicitly fold that narrow requirement into the canonical selective/lazy context design.

## Reproduction Assessment

Not applicable as a feature/performance request. Source inspection on current main and v2026.5.12 shows `HEARTBEAT.md` is still in the raw loader/cache path before any run-kind filter can omit it.

## Solution Assessment

Unclear until maintainer product/API confirmation. Passing run-kind state into the loader/cache is the likely narrow technical shape, but it changes documented prompt semantics and overlaps the broader selective/lazy context design.
