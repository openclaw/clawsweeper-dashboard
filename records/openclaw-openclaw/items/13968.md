## Summary

Current main and v2026.6.6 still eagerly read and snapshot HEARTBEAT.md before run filtering, and ordinary default-agent turns retain it when heartbeat guidance is enabled. The optimization remains useful, but it changes a documented prompt contract and needs a maintainer product decision across loading, caching, hooks, and prompt injection; the broader tiered-loading work preserves current main-session behavior and does not supersede this request.

## What This Changes

Make HEARTBEAT.md load only for heartbeat runs instead of ordinary messages.

## Best Possible Solution

Define one run-kind-aware bootstrap policy before disk reads and snapshot caching, carry it consistently through hooks and prompt injection, and document and test normal, heartbeat, cron, subagent, disabled-guidance, and configurable-tier behavior.

## Reproduction Assessment

Yes at source level: current main eagerly reads and snapshots HEARTBEAT.md before filtering, and ordinary full-context default-agent runs keep it when guidance is enabled. This confirms the reported behavior, but it is currently an intentional documented contract rather than a bug.

## Solution Assessment

No, passing only an isHeartbeat flag through the two proposed functions is not the best complete solution. The loader, per-session cache, bootstrap hooks, documented prompt contract, and sibling run modes need one shared run-kind-aware policy.
