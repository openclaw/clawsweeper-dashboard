## Summary

Keep open. Current main still reads and caches HEARTBEAT.md before run-kind filtering, and the documented behavior still permits default non-heartbeat runs to receive heartbeat guidance when enabled; the broader selective-context issue is adjacent but does not replace this narrow product/API decision.

## What This Changes

Request on-demand HEARTBEAT.md loading so non-heartbeat runs skip reading/injecting it while heartbeat polls still receive it.

## Best Possible Solution

Define a run-kind-aware bootstrap load/cache contract that omits HEARTBEAT.md before disk reads, session snapshot caching, hooks, and prompt injection for ordinary non-heartbeat runs, while preserving real heartbeat behavior and updating docs/tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high-confidence evidence that current main still reads and caches HEARTBEAT.md before any run-kind filter can omit it.

## Solution Assessment

Unclear until maintainer product confirmation. Passing run-kind or heartbeat state into the loader/cache is the narrow technical shape, but it changes the documented normal-run prompt-context contract and may overlap the broader selective-context design.
