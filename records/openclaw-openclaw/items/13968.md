## Summary

Keep this issue open: current main and the latest release still intentionally include HEARTBEAT.md in ordinary default-agent runs when heartbeat guidance is enabled, and the complete bootstrap set is read and cached before later run filtering. The requested optimization therefore remains unimplemented, while changing it requires an explicit decision about the normal-run prompt contract.

## What This Changes

Load HEARTBEAT.md only for actual heartbeat runs instead of repeatedly adding it to ordinary message context.

## Best Possible Solution

Adopt a single run-kind-aware bootstrap pipeline that decides the eligible built-in files before disk reads and snapshot caching, preserves HEARTBEAT.md for actual heartbeat runs, and aligns hooks, prompt construction, focused tests, and heartbeat documentation with the chosen normal-run contract.

## Reproduction Assessment

Yes. With a non-empty HEARTBEAT.md and default heartbeat guidance enabled, the documented and source-defined path loads the complete bootstrap set and retains that file for ordinary full-context runs; this read-only review did not execute a runtime token benchmark.

## Solution Assessment

Yes, conditionally. If maintainers choose heartbeat-only visibility, deciding file eligibility before the loader/cache boundary is the narrowest maintainable solution; merely adding a final filter would not satisfy the on-demand loading claim.
