## Summary

Keep open. Current main and the latest release still load and cache `HEARTBEAT.md` before run-kind filtering, while docs still allow normal default-agent runs to include it when heartbeat guidance is enabled; related work covers adjacent lightweight heartbeat and broader selective-context behavior, not this exact loader/cache contract.

## What This Changes

Request to skip loading `HEARTBEAT.md` for non-heartbeat runs so it is read, cached, and injected only when a heartbeat run actually needs it.

## Best Possible Solution

Define a run-kind-aware bootstrap load/cache contract that skips `HEARTBEAT.md` before disk reads, session snapshots, hooks, and prompt injection for ordinary non-heartbeat runs, while preserving real heartbeat behavior and updating docs/tests.

## Reproduction Assessment

Not applicable as a feature/performance request. Source inspection gives high-confidence evidence that current main still reads and caches `HEARTBEAT.md` before any run-kind filter can omit it.

## Solution Assessment

Unclear until maintainer product/API confirmation. Passing run-kind or heartbeat state into the loader/cache is the narrow technical shape, but a post-load filter would not satisfy the requested load/cache savings and the docs contract must change.
