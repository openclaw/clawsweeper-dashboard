## Summary

Keep open: current main still eagerly reads and snapshots HEARTBEAT.md before run-kind filtering, and the related selective/lazy context issue is a broader design umbrella rather than a settled replacement for this narrower loader/cache contract.

## What This Changes

The issue requests loading HEARTBEAT.md only for heartbeat runs instead of including it in every workspace bootstrap load.

## Best Possible Solution

Define a documented run-kind-aware bootstrap load/cache contract that skips HEARTBEAT.md before reads and cache snapshots for ordinary non-heartbeat runs, or explicitly fold that narrower requirement into https://github.com/openclaw/openclaw/issues/61347.

## Reproduction Assessment

Not applicable as a feature/performance request. Source inspection on current main shows HEARTBEAT.md is still read and cached before any run-kind filter can omit it.

## Solution Assessment

Unclear until maintainer product/API confirmation. Passing heartbeat or run-kind state into the loader/cache is the likely narrow technical shape, but it changes documented prompt semantics and overlaps the broader selective/lazy workspace-context design.
