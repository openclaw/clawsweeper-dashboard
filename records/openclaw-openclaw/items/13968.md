## Summary

Keep open: current main and v2026.6.8 still read and cache HEARTBEAT.md before run-kind filtering, and current docs still allow HEARTBEAT.md in normal default-agent runs when heartbeat guidance is enabled. The related lightweight-heartbeat fix and tiered-bootstrap work are useful adjacent efforts, but they do not implement this narrow on-demand HEARTBEAT.md loading contract.

## What This Changes

Request to load HEARTBEAT.md only for heartbeat runs instead of including it in ordinary non-heartbeat bootstrap context.

## Best Possible Solution

Define the desired normal-run heartbeat guidance contract, then make the loader and per-session cache run-kind aware before disk reads, snapshot caching, hooks, and prompt injection, with docs and focused tests updated together.

## Reproduction Assessment

Yes, at source level: current main eagerly reads and snapshots HEARTBEAT.md before run-kind filtering, and ordinary full-context default-agent runs keep it when heartbeat guidance is enabled. I did not run a live gateway session in this read-only review.

## Solution Assessment

No, not as only passing an isHeartbeat flag through two functions. The maintainable fix needs one shared run-kind-aware policy across the loader, cache, hooks, prompt injection, docs, and tests.
