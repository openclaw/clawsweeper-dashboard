## Summary

The requested behavior remains missing on current main: HEARTBEAT.md is read into the shared bootstrap snapshot before run-kind filtering, while ordinary default-agent runs still retain it when heartbeat guidance is enabled. The change remains useful, but it alters the documented prompt contract and therefore needs the existing maintainer product decision rather than automatic implementation.

## What This Changes

Requests loading HEARTBEAT.md only for heartbeat runs instead of reading and injecting it during ordinary messages.

## Best Possible Solution

Adopt one explicit run-kind-aware load/cache contract that omits HEARTBEAT.md before disk reads, hooks, snapshot caching, and prompt injection for ordinary runs while preserving it for actual heartbeat runs, then align focused tests and public docs.

## Reproduction Assessment

Yes, at source level: current main always reads HEARTBEAT.md into the raw bootstrap set before later filtering. This read-only review did not run a token benchmark, but the relevant load/cache path is deterministic.

## Solution Assessment

Yes for the implementation boundary, but not yet as an unconditional product decision. Passing run kind into the loader/cache is preferable to another config option or a post-load-only filter, provided maintainers intentionally change the normal-run guidance contract.
