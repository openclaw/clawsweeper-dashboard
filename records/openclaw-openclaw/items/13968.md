Keep open: current main and v2026.6.10 still read and cache HEARTBEAT.md before run-kind filtering, and current docs still allow normal default-agent runs to receive HEARTBEAT.md when heartbeat guidance is enabled. Related tiered-bootstrap work overlaps, but it preserves current defaults and does not replace this narrower prompt/context contract decision.

## What This Changes

The issue requests loading HEARTBEAT.md only for heartbeat runs instead of injecting it into ordinary non-heartbeat message context.

## Best Possible Solution

Define the desired normal-run heartbeat guidance contract, then make loader/cache, hooks, prompt injection, docs, and focused tests share one run-kind-aware bootstrap policy.

## Reproduction Assessment

Yes, at source level: current main eagerly reads and snapshots HEARTBEAT.md before run-kind filtering, and ordinary full-context default-agent runs keep it when heartbeat guidance is enabled. I did not run a live gateway session in this read-only review.

## Solution Assessment

No, not as only passing an isHeartbeat flag through the two named functions. The maintainable fix needs one shared run-kind-aware policy across loader, cache, hooks, prompt injection, docs, and tests.
