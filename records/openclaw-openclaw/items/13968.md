## Summary

This issue still appears valid on current main: HEARTBEAT.md is filtered only after the workspace bootstrap set is loaded and cached, and normal default-agent runs can still include heartbeat guidance when cadence is enabled. The merged heartbeat light-context and bootstrap-cache work are adjacent but do not implement the requested on-demand loader/cache behavior.

## What This Changes

Request on-demand HEARTBEAT.md loading so normal non-heartbeat messages skip heartbeat guidance while real heartbeat runs keep it.

## Best Possible Solution

Add a run-kind-aware bootstrap load/cache path that omits HEARTBEAT.md before read/cache/prompt injection for ordinary non-heartbeat default runs, preserves it for actual heartbeat runs, and updates focused tests plus heartbeat prompt docs for the new contract.

## Reproduction Assessment

Yes from source inspection: current main still reads and caches the fixed bootstrap file set, including HEARTBEAT.md, before run-kind filtering. I did not run a live heartbeat/default-message scenario in this read-only review.

## Solution Assessment

Unclear without maintainer product confirmation: the requested run-kind-aware loader/cache split is the narrow technical shape, but it intentionally changes whether normal default-agent runs receive heartbeat guidance when cadence is enabled.
