## Summary

Keep open. Current main still loads and snapshots `HEARTBEAT.md` before run-kind filtering, while the documented normal-run contract still allows it when heartbeat guidance is enabled; the related selective-context issue is broader and the prior conditional-loading PRs are closed, so this is not safely closeable.

## What This Changes

Request to load or inject `HEARTBEAT.md` only for heartbeat runs so ordinary messages avoid its token cost.

## Best Possible Solution

Define a maintainer-approved run-kind-aware bootstrap load/cache contract that omits `HEARTBEAT.md` before disk reads, session snapshot caching, hooks, and prompt injection for ordinary non-heartbeat runs while preserving real heartbeat behavior and updating docs/tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence evidence that current main still reads and caches `HEARTBEAT.md` before any run-kind filter can omit it.

## Solution Assessment

Unclear until maintainer product/API confirmation. Passing run-kind or heartbeat state into the loader/cache is the narrow technical shape, but it changes the current normal-run prompt-context contract and a post-load filter would not satisfy the requested savings.
