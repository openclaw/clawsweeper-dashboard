## Summary

Keep open: current main still reads and caches HEARTBEAT.md before run-kind filtering, and current docs still allow HEARTBEAT.md in ordinary default-agent runs when heartbeat guidance is enabled. The source gap is clear, but changing it needs a maintainer-approved prompt/context contract rather than a narrow mechanical cleanup.

## What This Changes

The issue requests loading HEARTBEAT.md only for heartbeat runs instead of including it in normal non-heartbeat workspace bootstrap context.

## Best Possible Solution

Define the desired normal-run heartbeat guidance contract, then make the loader, per-session cache, hooks, prompt injection, docs, and focused tests share one run-kind-aware bootstrap policy.

## Reproduction Assessment

Yes, at source level: current main eagerly reads and snapshots HEARTBEAT.md before run-kind filtering, and ordinary full-context default-agent runs keep it when heartbeat guidance is enabled. I did not run a live gateway session in this read-only review.

## Solution Assessment

No, not as only passing an isHeartbeat flag through the two named functions. The maintainable fix needs one shared run-kind-aware policy across loader, cache, hooks, prompt injection, docs, and tests.
