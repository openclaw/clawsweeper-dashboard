## Summary

Keep open: the crash report is credible and high impact, current main still routes session `.jsonl.lock` acquisition through the `@openclaw/fs-safe` FileHandle-backed sidecar lock, and no current-main macOS/Node 25 heartbeat-with-exec reproduction or exact leak fix has landed.

## What This Changes

The issue reports Node 24+ gateway crashes from leaked session transcript `.jsonl.lock` FileHandles during sustained session-store and heartbeat load.

## Best Possible Solution

Build a macOS arm64 Node 25 heartbeat-with-exec stress reproduction with forced GC/open-handle observation, then fix the exact session lifecycle or `@openclaw/fs-safe` sidecar-lock release path and validate on Node 22 plus Node 24/25.

## Reproduction Assessment

No high-confidence current-main reproduction is available yet. The reporter's production evidence is strong and specific, but current-main Linux Node 25 probes did not reproduce and the remaining lead is macOS Node 25 heartbeat sessions that run `exec`.

## Solution Assessment

Unclear. Explicit `release()`/`finally` cleanup is the right contract, but current code and `fs-safe` already have centralized close paths, so the safest fix is to isolate the actual leaked holder before changing lock lifecycle broadly.
