## Summary

Keep open: the bug fix is still relevant, but the PR head still overmatches unsupported dotted gateway actions whose leaf looks read-only, and it lacks after-fix real gateway or Telegram behavior proof.

## What This Changes

The PR changes `isMutatingToolCall` and its tests so dotted gateway actions ending in read-only verbs, including `config.schema.lookup`, are classified as non-mutating while cron/canvas dotted or lookup actions remain mutating.

## Best Possible Solution

Classify only declared or read-scoped gateway methods as non-mutating, keep unsupported gateway and flat cron/canvas actions fail-closed, and land with focused regression coverage plus redacted gateway or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main classifies `gateway` `config.schema.lookup` as mutating even though the gateway scope table marks it read-only, and that mutation state reaches the warning-rendering path.

## Solution Assessment

No, not as written. The classifier-level fix is the right boundary, but the PR should match declared or read-scoped gateway methods instead of trusting any read-like dotted leaf.
