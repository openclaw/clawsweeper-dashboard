## Summary

Keep open: current main still has the reported classifier bug, but this PR should not merge as-is because the gateway leaf fallback marks unsupported read-like gateway actions as non-mutating, and contributor after-fix real behavior proof is still missing.

## What This Changes

The PR updates `src/agents/tool-mutation.ts` and tests so dotted gateway actions with read-only leaf verbs, especially `config.schema.lookup`, are treated as non-mutating while cron/canvas stay fail-closed.

## Best Possible Solution

Classify declared/read-scoped gateway actions such as `config.get` and `config.schema.lookup` as non-mutating, keep unknown and write-like gateway actions fail-closed, then land with focused regression coverage and redacted real gateway or Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main checks the full `gateway` action string against a flat read-only set, while `config.schema.lookup` is a declared `operator.read` gateway method and the warning path consumes the resulting `mutatingAction` bit.

## Solution Assessment

No, not as written. The fix boundary is right, but the PR should match declared or read-scoped gateway actions instead of trusting any read-like leaf segment.
