## Summary

Keep open. The underlying bug is still present on current main and this PR is on the right surface, but the current patch still overmatches unsupported gateway actions and lacks required after-fix real behavior proof for the visible warning flow.

## What This Changes

The PR updates `src/agents/tool-mutation.ts` and its tests so dotted gateway actions with read-only leaf verbs, including `config.schema.lookup`, are classified as non-mutating while cron/canvas unsupported actions stay fail-closed.

## Best Possible Solution

Classify only declared or read-scoped gateway actions exposed through the gateway tool as non-mutating, keep unsupported gateway actions fail-closed, and land with focused regression coverage plus redacted real gateway or Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main classifies `gateway` `config.schema.lookup` as mutating even though the gateway descriptor marks it read-only, and that mutation state reaches the warning-rendering path.

## Solution Assessment

No, not as written. The classifier boundary is right, but the PR should match declared or read-scoped gateway actions instead of trusting any read-like leaf segment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Restrict gateway read-only matches to declared actions:** `src/agents/tool-mutation.ts:136`
  - body: The gateway leaf fallback makes unsupported actions such as `agents.files.list` or `foo.get` non-mutating because their final segment is read-only, but `createGatewayTool` only accepts the fixed `GATEWAY_ACTIONS` enum and throws `Unknown action` for other strings. Match declared or read-scoped gateway actions instead of any read-like leaf so invalid calls stay fail-closed.
  - confidence: 0.88
