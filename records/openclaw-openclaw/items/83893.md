## Summary

Keep open as a narrow P3 cleanup: current main still reaches into Commander's runtime-only `rawArgs` field through a cast, and Commander 14.0.3 typings do not expose that property. The report overstates the gap because current main already has focused fallback tests, so this is maintainability work rather than a reproduced CLI bug.

## What This Changes

The issue asks OpenClaw to stop or guard access to Commander's non-public `rawArgs` property in CLI action reparsing.

## Best Possible Solution

Encapsulate or eliminate the private Commander `rawArgs` access while preserving the existing lazy-command fallback semantics and keeping JSON-producing CLI paths quiet.

## Reproduction Assessment

Yes for the source-level concern: current main still casts `Command` to read `rawArgs`, and Commander 14.0.3 typings do not expose that member. No high-confidence user-visible CLI failure has been reproduced.

## Solution Assessment

Yes, a narrow cleanup is appropriate, but it should not add noisy runtime warnings that can disturb JSON output. The existing fallback behavior is intentional and tested, so the repair should make the dependency contract explicit or move argv ownership into OpenClaw without changing parse behavior.
