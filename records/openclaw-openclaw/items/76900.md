## Summary

Keep open: current main still has the strict Discord token path and status/runtime/health paths can reach the Discord account resolver, while open PR #76911 already targets this issue with closing syntax.

## What This Changes

The issue requests that Discord SecretRef-backed tokens be resolved or degraded on status/runtime/health paths instead of crashing the gateway.

## Best Possible Solution

Land #76911 or an equivalent Discord-owned fix, reconciling it with #76958 if needed, so SecretRef-backed Discord tokens use the active runtime snapshot or report configured-unavailable on read-only surfaces.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current tests assert the unresolved SecretRef throw, and status plus health paths still reach the same Discord account/token resolver. I did not run a live gateway because this was a read-only review and `pnpm` is unavailable in this shell.

## Solution Assessment

Yes, the best direction is a narrow Discord-owned fix rather than a core SecretRef contract change. The open PRs should be reviewed/landed or superseded with an equivalent patch that hardens read-only surfaces without weakening action/startup token requirements.
