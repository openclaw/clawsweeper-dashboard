## Summary

Keep open: current main still has the strict Discord token path reachable from channel runtime/status/health surfaces, and the linked fix PR #76911 is open but not merged.

## What This Changes

The issue asks for Discord channel status, runtime snapshots, and health monitoring to tolerate unresolved SecretRef-backed tokens without crashing the gateway.

## Best Possible Solution

Review and land a narrow fix, likely through #76911 if it passes review, that reports unresolved Discord SecretRefs as configured-unavailable on status/runtime/health surfaces without exposing token values.

## Reproduction Assessment

Yes, at source level: an unresolved SecretRef object in channels.discord.token reaches resolveDiscordToken -> normalizeDiscordToken -> normalizeResolvedSecretInputString, and status/runtime/health paths still call that resolver. I did not run a live gateway or tests because this review was read-only.

## Solution Assessment

Not applicable to this issue alone because no patch is attached here; #76911 is the active implementation candidate. The maintainable fix should use non-strict inspection or resolver-error isolation for status/runtime/health while keeping actual channel startup credential use strict.
