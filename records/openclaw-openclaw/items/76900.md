## Summary

Keep open: current main still has the strict Discord token path reachable from runtime/status/health snapshots, and the issue is already paired with an open closing PR that should be reviewed before this issue closes.

## What This Changes

Discord SecretRef-backed tokens should not crash gateway channel status, runtime snapshot, or health monitoring when still unresolved.

## Best Possible Solution

Make Discord status/runtime/health inspection tolerate unresolved configured SecretRefs without exposing token values, while keeping actual token use strict where a real Discord client is started or sends messages.

## Reproduction Assessment

Yes at source level: put an unresolved SecretRef object in channels.discord.token and call the Discord resolver through getRuntimeSnapshot or channels.status; the strict token helper throws. I did not execute a live gateway in this read-only review.

## Solution Assessment

Yes: a narrow Discord-owned resolver or inspection fix is the maintainable path, with generic snapshot isolation only if other plugins share the same failure mode. The already-open #76911 should be reviewed or adjusted against that boundary.
