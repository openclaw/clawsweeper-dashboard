## Summary

Keep open: the report matches current source because Discord status/runtime/health paths can still reach strict token normalization with an unresolved SecretRef, while the docs say Discord token SecretRefs are supported and the non-strict inspection helper is not used on all affected paths.

## What This Changes

Discord SecretRef-backed tokens should not crash gateway channel status, runtime snapshot, or health-monitor paths; they should resolve or report configured-unavailable.

## Best Possible Solution

Make Discord account/status inspection and generic channel runtime snapshots tolerate unresolved configured SecretRefs by reporting configured-unavailable/unavailable state without exposing token values or rejecting the health/status task.

## Reproduction Assessment

Yes at source level: put an unresolved SecretRef object in `channels.discord.token` and call the Discord resolver through `channels.status` or `channelManager.getRuntimeSnapshot`; the strict token path throws. I did not run a live gateway because this review was read-only.

## Solution Assessment

No proposed patch is attached. The narrow maintainable solution is to use non-strict credential inspection or generic resolver-error isolation for status/runtime/health surfaces, while keeping actual channel startup token use strict.
