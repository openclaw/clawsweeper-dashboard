## Summary

Keep open. Current main and the shipped v2026.4.10 heartbeat fix already spread heartbeat interval runs across stable per-agent phases, but cron `every` still uses fixed anchors and the requested prime/phase public controls are not present; the remaining work needs a maintainer product and compatibility decision.

## What This Changes

The issue requests prime-derived cadence and deterministic phase jitter controls for heartbeat and cron interval schedules to reduce synchronized provider traffic spikes.

## Best Possible Solution

Keep the shipped heartbeat phase spreading, choose one durable rule for cron interval phase spreading that preserves upgrade safety, and add public exact-phase controls only if maintainers decide cron expressions are not enough for fixed wall-clock needs.

## Reproduction Assessment

Partly yes from source: heartbeat phase spreading is directly verifiable, and cron `every` schedules with equal anchors and intervals remain aligned. The report does not include a measured current-main multi-agent load reproduction showing provider spikes or failure frequency.

## Solution Assessment

No, not as proposed. Deterministic phase spreading addresses the anti-herding invariant more directly than prime intervals, while fixed `:55`-style wall-clock timing is better handled by cron expressions unless maintainers want a new exact-phase API.

## Risks / Open Questions

- Adding `everyMode`, `phaseJitter`, or exact phase fields expands a public configuration and CLI surface that must stay aligned across schema, persistence, status output, docs, and doctor behavior.
- Automatically changing cron `every` anchors could shift existing users' persisted schedules on upgrade; preserving old anchors while changing new-job defaults would create separate fresh-install and upgrade semantics.
- Prime interval transforms add another scheduling policy but do not solve same-phase alignment by themselves, so maintainers should require data or a clearer product reason before adding that mode.
