## Summary

Keep open. Current main only partially addresses the thundering-herd concern: heartbeat has an internal stable per-agent phase and cron expressions have top-of-hour staggering, but the requested public scheduling contract is still absent for prime cadence, configurable heartbeat jitter, cron `every` jitter/prime semantics, and resolved phase/status exposure.

## What This Changes

The issue requests additive scheduling controls that use prime-derived heartbeat/cron `every` intervals and deterministic per-agent phase jitter to reduce synchronized load spikes.

## Best Possible Solution

Define a backward-compatible scheduler contract that preserves the existing internal heartbeat phase, then add explicit public controls for heartbeat phase/jitter, optional prime cadence, cron `every` jitter/prime semantics, and resolved schedule/status visibility across config schemas, Gateway protocol, CLI, Control UI, docs, and focused scheduler tests.
