## Summary

Keep open. Current main already implements and ships stable heartbeat phase spreading, but cron `every` remains a fixed `everyMs`/`anchorMs` schedule with no prime, phase, or stagger contract, so the remaining request still needs maintainer API/product direction.

## What This Changes

The issue requests prime-derived heartbeat and cron interval scheduling plus deterministic per-agent phase jitter to reduce synchronized traffic bursts.

## Best Possible Solution

Keep the shipped heartbeat phase spreading, then have maintainers choose one canonical cron `every` contract for phase/stagger/prime behavior with schema, CLI, docs, persistence, and upgrade tests before implementation.

## Reproduction Assessment

Yes for the missing capability: current source shows cron `every` only supports `everyMs`/`anchorMs`, the CLI rejects stagger/exact with `--every`, and deterministic stagger applies only to cron-expression schedules. The heartbeat phase portion is already implemented on main.

## Solution Assessment

Unclear until maintainers choose the product contract. Prime cadence and new phase knobs are plausible, but the maintainable solution may be a smaller cron `every` stagger/phase contract, a docs-supported anchor pattern, or no new public surface.
