## Summary

Keep open. Current main and the latest release already cover deterministic heartbeat phase spreading, but cron `every` schedules still use fixed `everyMs` plus optional `anchorMs` semantics and have no retained stagger, phase, or prime-cadence controls, so the remaining work needs a maintainer product/API decision.

## What This Changes

The issue requests prime-derived cadence and deterministic per-agent phase jitter for heartbeat and cron `every` schedules to reduce synchronized scheduled work.

## Best Possible Solution

Keep the shipped heartbeat phase spreading, then have maintainers choose one explicit cron `every` phase/stagger/anchor contract before adding prime cadence or new public fields.

## Reproduction Assessment

Yes. The remaining source-level gap is reproducible from current main: cron `every` only stores `everyMs`/`anchorMs`, normalization prunes `staggerMs`, and the CLI rejects `--stagger` or `--exact` with `--every`.

## Solution Assessment

No for the full proposal as written. Heartbeat phase jitter is already implemented, while prime cadence is less clearly maintainable than a deliberate cron `every` phase/stagger/anchor contract selected by maintainers.
