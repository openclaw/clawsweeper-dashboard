## Summary

Keep open. Current main and the latest release have stable per-agent heartbeat phase scheduling, but cron `every` still has fixed `everyMs`/`anchorMs` semantics and no prime, phase, or stagger contract; the remaining work is a scheduling API/product decision, not conservative cleanup.

## What This Changes

Request to add prime-derived cadence and deterministic per-agent phase jitter for heartbeat and cron interval schedules to reduce synchronized provider-load spikes.

## Best Possible Solution

Keep the shipped heartbeat phase spreading and have maintainers choose one explicit cron `every` phase/stagger/anchor contract before adding prime cadence or new public fields.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof of the remaining gap: heartbeat phase exists, while cron `every` lacks phase, stagger, and prime controls.

## Solution Assessment

No for the full proposal as written. Heartbeat phase jitter is already implemented, and the maintainable next step is a maintainer-selected cron `every` phase/stagger contract rather than adding prime defaults first.
