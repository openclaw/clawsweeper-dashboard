## Summary

Keep open. Current main already implements stable heartbeat phase spreading, but cron `every` remains a fixed-interval `everyMs`/`anchorMs` schedule with no prime, phase, or stagger contract, so the remaining request still needs maintainer API/product direction.

## What This Changes

Request prime-derived cadence and deterministic phase jitter for heartbeat and cron `every` schedules to reduce synchronized load spikes.

## Best Possible Solution

Keep the shipped heartbeat phase spreading and have maintainers choose one canonical cron `every` phase/stagger/prime contract with schema, CLI, docs, and tests before implementation.

## Reproduction Assessment

Yes for the missing capability: current source shows cron `every` only supports `everyMs`/`anchorMs`, the CLI rejects stagger/exact with `--every`, and normalization removes `staggerMs` from `every`. A live load-spike reproduction was not run because the remaining issue is source-proven API/product surface.

## Solution Assessment

Unclear until maintainers choose the product contract. Heartbeat phase spreading is already solved on main, but cron `every` needs a deliberate choice among phase jitter, stagger windows, prime cadence, an anchor reinterpretation, or no new public surface.
