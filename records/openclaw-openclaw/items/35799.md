## Summary

Keep open. Current main partially addresses the load-spike concern with stable heartbeat phase scheduling and cron top-of-hour staggering, but the requested prime cadence mode, configurable heartbeat jitter surface, user-selectable phase behavior, and cron `every` jitter/prime semantics are still absent and require maintainer API/product judgment.

## What This Changes

The issue requests prime-derived heartbeat and cron interval cadence plus deterministic per-agent phase jitter to reduce synchronized provider traffic.

## Best Possible Solution

Decide the public scheduler policy first; if accepted, add backward-compatible schema, protocol, CLI/UI, docs, and tests for opt-in prime cadence and interval jitter while preserving exact schedules unless maintainers intentionally change defaults.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature/config-policy request. Source inspection on current main verifies the requested prime mode and cron `every` jitter fields are absent while related scheduler mitigations exist.

## Solution Assessment

Unclear. Stable heartbeat phase scheduling is already a narrow mitigation for clustering, but prime cadence and cron `every` jitter alter public timing/config semantics and need maintainer scoping before implementation.
