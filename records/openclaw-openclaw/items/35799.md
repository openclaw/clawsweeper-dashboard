## Summary

Keep open. Current main and v2026.6.10 already cover stable heartbeat phase spreading, but cron `every` schedules still use fixed `everyMs`/`anchorMs` semantics and do not retain stagger, phase, or prime-cadence controls, so the remaining work needs a maintainer product/API decision.

## What This Changes

Request to add prime-derived cadence and deterministic phase jitter for heartbeat and cron `every` schedules to avoid synchronized provider-load spikes.

## Best Possible Solution

Keep the shipped heartbeat phase spreading and have maintainers define one backwards-compatible cron `every` phase/stagger/anchor contract before adding prime cadence or new public fields.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: cron `every` types/schema only allow `everyMs`/`anchorMs`, normalization prunes `staggerMs`, and next-run computation advances from last run or anchor without phase or prime controls.

## Solution Assessment

No for the full proposal as written. Heartbeat phase spreading is already shipped, and the best remaining path is a maintainer-chosen cron `every` phase/stagger/anchor contract rather than adding prime cadence as the default answer.
