## Summary

Keep open. Current main partially mitigates the load-spike problem with internal per-agent heartbeat phasing and cron-expression staggering, but the requested public scheduling contract remains absent: no prime cadence mode, no configurable heartbeat phase jitter/max window, no cron every jitter or prime fields, and no status exposure for resolved heartbeat phase.

## What This Changes

Request to add prime-derived cadence and deterministic configurable phase jitter for heartbeat and cron every schedules, plus CLI/status observability, to reduce synchronized request bursts.

## Best Possible Solution

Keep this as a scheduler/API design item. The end state should preserve the existing internal heartbeat phase, then add an explicit backward-compatible public contract for heartbeat phase/jitter and any accepted prime cadence or cron every jitter semantics across config types/schema, Gateway protocol, CLI/Control UI, status output, docs, and focused scheduler tests.

## Reproduction Assessment

Not applicable as a feature request. The missing behavior is high-confidence from static source, schema, CLI, protocol, docs, and test inspection against current main.

## Solution Assessment

Unclear. The requested direction is plausible, but the best defaults and public field names need maintainer API/product judgment; the safest implementation path is additive opt-in controls first, with default changes considered separately.
