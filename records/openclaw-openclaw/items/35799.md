## Summary

Keep open. Current main and the latest release cover stable heartbeat phase spreading, but cron `every` schedules still use fixed `everyMs`/`anchorMs` semantics and do not retain stagger, phase, or prime-cadence controls, so the remaining work needs a maintainer product/API decision.

## What This Changes

The issue requests prime-derived heartbeat and cron `every` cadence plus deterministic per-agent phase jitter to reduce synchronized scheduling spikes.

## Best Possible Solution

Keep the shipped heartbeat phase spreading and have maintainers choose one explicit cron `every` phase/stagger/anchor contract before adding prime cadence or new public fields.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main's cron `every` type only has `everyMs`/`anchorMs`, the normalizer deletes `staggerMs` from `every`, and the CLI rejects `--stagger`/`--exact` with `--every`. I did not run a live multi-gateway load test because the remaining missing capability is provable from source and docs.

## Solution Assessment

No for the full proposal as written: heartbeat phase jitter is already shipped, and prime cadence is less precise than a clear cron `every` phase/stagger contract. The best next step is a maintainer decision on the public cron interval semantics and upgrade behavior.
