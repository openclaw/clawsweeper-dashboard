## Summary

Keep open. Heartbeat phase spreading is already implemented, but current main and v2026.6.10 still model cron `every` schedules as only `everyMs`/`anchorMs` and reject stagger/exact metadata for `--every`, so the remaining work needs a maintainer product/API decision.

## What This Changes

The issue asks to add prime-derived heartbeat and cron `every` cadences plus deterministic phase jitter to reduce synchronized scheduled load.

## Best Possible Solution

Keep the shipped heartbeat phase spreading, then choose one explicit cron `every` phase/stagger/anchor contract before adding prime cadence, public fields, docs, and compatibility tests.

## Reproduction Assessment

Yes. Source inspection shows current cron `every` schedules have only `everyMs`/`anchorMs`, normalization removes `staggerMs`, and the CLI rejects `--stagger`/`--exact` with `--every`; a live load test was not needed to prove the missing surface.

## Solution Assessment

No for the full proposal as written. Heartbeat phase jitter already exists, and the remaining cron interval behavior should be solved by a maintainer-approved phase/stagger/anchor contract rather than unilaterally adding prime cadence fields.
