## Summary

Keep open. Current main already includes default deterministic per-agent heartbeat phase alignment and cron top-of-hour staggering, but the requested prime cadence controls and cron `every` phase/jitter behavior are still absent and would change public scheduling/config semantics.

## What This Changes

Request prime-derived heartbeat and cron `every` intervals plus deterministic phase jitter to reduce synchronized polling spikes.

## Best Possible Solution

Make a maintainer product decision on the public scheduling contract: either document the existing heartbeat phase behavior as the supported mitigation and narrow the cron `every` follow-up, or add a versioned config/API design for `every` jitter/prime behavior across heartbeat, CLI, protocol, UI, and docs.

## Reproduction Assessment

Not applicable. This is an enhancement request, and current-main source inspection is enough to show which requested scheduling controls are absent.

## Solution Assessment

No. Current main partially solves the anti-spike problem for heartbeat and top-of-hour cron expressions, but it does not implement the requested prime cadence mode or cron `every` phase jitter, so closing as implemented would overstate the current behavior.
