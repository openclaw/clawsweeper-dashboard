## Summary

Keep open. Current main and v2026.6.10 already cover deterministic heartbeat phase spreading, but cron `every` schedules still only have fixed interval plus optional anchor semantics, so the remaining cron/API/config surface needs maintainer product judgment.

## What This Changes

Request prime-derived cadence and deterministic phase-jitter controls for heartbeat and cron `every` schedules to reduce synchronized load spikes.

## Best Possible Solution

Keep the shipped heartbeat phase spreading and have maintainers choose one explicit cron `every` phase/stagger/anchor contract before adding prime cadence or new public fields.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: cron `every` schedules still accept only `everyMs`/`anchorMs`, normalization strips `staggerMs`, and the CLI rejects `--stagger`/`--exact` with `--every`. I did not run a live multi-gateway load test because the remaining gap is a missing public scheduling capability.

## Solution Assessment

No for the full proposal as written: heartbeat phase jitter is already shipped, and prime cadence is less precise than a clear cron `every` phase/stagger contract. The best next step is a maintainer decision on interval scheduling semantics and upgrade behavior.
