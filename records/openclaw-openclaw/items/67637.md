## Summary

Keep open. Current main still lacks a bus-level tool-call trace surface on `QaBusMessage`; mock-provider `/debug/requests` and internal `onToolStart` data are adjacent but do not satisfy the requested QA bus contract, and the API/redaction boundary needs maintainer judgment.

## What This Changes

The issue requests optional tool-call trace data on QA bus messages so qa-lab scenarios can assert real tool use instead of only response text.

## Best Possible Solution

Define a backwards-compatible, sanitized QA trace surface sourced from actual tool-start events, then carry it through bus create, clone, snapshot, read, search, poll, wait, and qa-channel delivery paths with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the agent can emit sanitized tool-start data, but `QaBusMessage`, QA bus persistence/query paths, and qa-channel delivery do not expose tool-call traces.

## Solution Assessment

Unclear. The proposed optional `QaBusMessage.toolCalls` field is plausible, but maintainers should first choose the public SDK versus qa-lab-only API shape and define argument redaction/retention.
