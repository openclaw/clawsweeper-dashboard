## Summary

Keep open. Current main still lacks a bus-level tool-call trace surface on `QaBusMessage`; the existing mock-provider `/debug/requests` assertions are useful adjacent coverage, but they do not expose actual QA bus message traces and no canonical open or merged item supersedes this request.

## What This Changes

Add an optional tool-call trace field or equivalent assertion surface to QA bus messages so qa-lab scenarios can assert that real tools were called.

## Best Possible Solution

Define a backwards-compatible, sanitized QA trace surface sourced from actual tool-start events, then carry it through QA bus create, clone, snapshot, read, search, poll, wait, and qa-channel delivery paths with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the agent can emit sanitized tool-start data, but `QaBusMessage`, QA bus persistence/query paths, and qa-channel delivery do not expose tool-call traces.

## Solution Assessment

Unclear. The proposed optional `QaBusMessage.toolCalls` field is plausible, but maintainers should first choose the public SDK versus qa-lab-only API shape and define argument redaction and retention.
