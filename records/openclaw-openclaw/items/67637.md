## Summary

Keep open. Current main still lacks a QA bus message-level trace of actual agent tool calls; the existing mock-provider `plannedToolName` path is useful but not equivalent to `QaBusMessage` carrying real execution evidence.

## What This Changes

Request adds optional tool-call trace data to `QaBusMessage` so qa-lab scenarios can assert actual tool execution instead of only response text.

## Best Possible Solution

Design a backwards-compatible, sanitized QA trace surface sourced from actual tool-start execution events and carry it through QA bus create, clone, snapshot, read, search, poll, wait, and qa-channel delivery paths.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the QA bus contract, state creation, clone/query paths, and qa-channel outbound delivery have no field or API that can return actual tool-call traces.

## Solution Assessment

Unclear. The proposed optional `QaBusMessage.toolCalls` field is plausible, but maintainers should first choose the public SDK versus qa-lab-only API shape and define argument redaction.
