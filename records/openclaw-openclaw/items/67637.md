## Summary

Keep open. Current main still does not expose actual agent/tool execution traces on `QaBusMessage`; the QA bus contract, create/clone/read/search/poll paths, and qa-channel delivery path still carry message text, metadata, attachments, and reactions only. The existing mock-provider `/debug/requests` support is useful context for #67540, but it remains a mock-scoped workaround rather than the requested bus-level trace surface.

## What This Changes

Add optional tool-call trace data to QA bus messages so qa-lab scenarios can assert actual tool execution instead of relying on response text.

## Best Possible Solution

Design a backwards-compatible QA trace surface sourced from actual agent tool-start execution events, with sanitized tool names and arguments. Carry it through QA bus create, clone, snapshot, read, search, poll, and wait paths, and add focused qa-lab/qa-channel tests plus at least one scenario assertion that fails when an agent fabricates success without calling the expected tool.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing surface: `QaBusMessage` and the QA bus create/read/search/poll/delivery paths have no field or API that can return actual tool-call traces.

## Solution Assessment

Unclear. The requested optional `QaBusMessage.toolCalls` field is plausible, but maintainers should first decide whether this belongs in the public plugin SDK contract or in a narrower qa-lab scenario trace/assertion API with argument sanitization.
