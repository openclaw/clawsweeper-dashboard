## Summary

Keep open. Current main still lacks a QA bus message-level tool-call trace; scenarios can inspect mock-provider planned tool calls, but that is not actual agent/tool execution data on `QaBusMessage`.

## What This Changes

Add an optional tool-call trace surface to `QaBusMessage` so QA scenarios can assert which tools the agent actually invoked.

## Best Possible Solution

Design a backwards-compatible, sanitized QA trace surface sourced from actual agent tool-start events and carry it through bus create, clone, snapshot, read, search, poll, wait, and qa-channel delivery paths.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `QaBusMessage`, bus persistence/query paths, and qa-channel delivery have no field or API that can return actual tool-call traces.

## Solution Assessment

Unclear. The proposed optional `QaBusMessage.toolCalls` field is plausible, but maintainers should first decide whether the public SDK contract should grow or whether qa-lab should expose a narrower trace/assertion helper.
