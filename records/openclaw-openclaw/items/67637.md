## Summary

Keep open. Current main still lacks an actual agent/tool execution trace on `QaBusMessage`; the only nearby support is mock-provider planned-tool debug data, which is not the requested bus-level assertion surface.

## What This Changes

Request an optional tool-call trace on `QaBusMessage` so QA scenarios can assert which tools were actually invoked instead of relying on response text.

## Best Possible Solution

Add a backwards-compatible, sanitized QA trace surface sourced from actual agent tool-start events and expose it through the shared qa-lab transport path chosen by maintainers.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the QA bus contract, persistence, clone/query, wait, poll, and qa-channel delivery paths have no way to return actual tool-call traces.

## Solution Assessment

Unclear. The proposed optional `QaBusMessage.toolCalls` field is plausible, but maintainers should first choose between a public SDK field and a narrower qa-lab assertion/trace helper with a redaction contract.
