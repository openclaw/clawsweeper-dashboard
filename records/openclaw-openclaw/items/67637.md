## Summary

Keep open. Current main still does not expose actual tool execution traces on `QaBusMessage`; the existing mock-provider `/debug/requests` path is useful adjacent evidence but not the requested bus-level trace surface, and the public API plus argument-redaction boundary needs maintainer judgment.

## What This Changes

The issue requests adding tool-call trace data to `QaBusMessage` so qa-lab scenarios can assert on actual tool invocations rather than response text only.

## Best Possible Solution

Design a backwards-compatible, sanitized QA trace surface sourced from actual tool-start events, then carry it through QA bus create, clone, snapshot, read, search, poll, wait, and qa-channel delivery paths with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the agent can emit sanitized tool-start data, but the QA bus contract, persistence/query paths, and qa-channel delivery path do not attach tool-call traces to `QaBusMessage`.

## Solution Assessment

Unclear. The proposed optional `QaBusMessage.toolCalls` field is plausible, but maintainers should choose the public SDK versus qa-lab-only API shape and define argument redaction before implementation.
