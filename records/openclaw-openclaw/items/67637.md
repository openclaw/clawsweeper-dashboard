## Summary

Keep open. Current main still does not expose actual agent/tool execution traces on `QaBusMessage`; the current mock-provider `/debug/requests` `plannedToolName` path is useful scenario evidence, but it is not the requested bus-level trace surface and the API/redaction boundary needs maintainer judgment.

## What This Changes

Add tool-call trace data to QA bus messages so qa-lab scenarios can assert actual tool invocations instead of only response text.

## Best Possible Solution

Design a backwards-compatible, sanitized QA trace surface sourced from actual tool-start events, then carry it through QA bus create, clone, snapshot, read, search, poll, wait, and qa-channel delivery paths with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the QA bus contract, creation path, clone/query paths, and qa-channel delivery path do not carry tool-call traces, while existing scenario assertions only inspect mock-provider `/debug/requests` logs.

## Solution Assessment

Unclear. The optional `QaBusMessage.toolCalls` field is plausible, but maintainers should first choose the public SDK versus qa-lab-only API shape and define argument redaction before implementation.
