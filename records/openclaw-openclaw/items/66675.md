## Summary

Keep open. Current main still has the Linux/systemd restart and followup paths described in the report: systemd restarts are synchronous `systemctl --user restart` calls, post-restart health failures still turn the command into a failure, and expired exec approval followup warnings are not suppressed. Open PR #66685 is explicitly only warning-noise cleanup, and open PR #66735 appears closer to the systemd self-restart root cause, but neither is merged into main.

## What This Changes

The issue requests that `openclaw gateway restart` on Linux/systemd exit successfully when the service restart completes and status/probe are healthy instead of reporting a false failure from post-restart or followup handling.

## Best Possible Solution

A fix should make Linux/systemd restarts report success when the restart is safely handed off or the post-restart status/probe is healthy, with focused regression coverage around the systemd adapter and daemon lifecycle; benign expired approval followup noise can be cleaned up separately or alongside that root-cause fix.

## Reproduction Assessment

Unclear. The report gives a concrete Linux/systemd setup and current source shows plausible failure paths, but this read-only review did not run a live systemd restart or find an existing regression test that reproduces the false nonzero result end to end.

## Solution Assessment

No current-main solution is sufficient yet. PR #66735 looks closer to the maintainable root-cause fix for in-service systemd restarts, while PR #66685 is useful but only addresses warning noise and should not close this issue by itself.
