## Summary

Keep open: current main still runs Linux/systemd gateway restarts through the synchronous `systemctl restart` path, still treats post-restart health-check throws as a failed restart result, and still logs expired approval followup dispatch failures. The related repair PRs remain open and unmerged, so the issue is not ready for cleanup.

## What This Changes

Report that `openclaw gateway restart` on Linux/systemd can return a non-zero failure even when `status`, `probe`, and the dashboard show the restarted gateway is healthy.

## Best Possible Solution

Land a Linux/systemd self-restart handoff that reports a scheduled restart when invoked from inside the managed unit, and land the expired approval followup suppression as cleanup if it still applies.

## Reproduction Assessment

Unclear. The issue gives a concrete Linux/systemd setup and current source shows a plausible failure path, but this review did not run a live systemd user service or find a failing regression test on main.

## Solution Assessment

No, current main is not sufficient. A #66735-style detached systemd handoff is the narrowest root fix; suppressing approval-not-found followup warnings alone would improve logs but not guarantee truthful restart exit status.
