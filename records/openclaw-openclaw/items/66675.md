## Summary

Keep open: current main still restarts Linux/systemd services synchronously, still runs the hard post-restart health failure path, and still logs expired exec approval followup failures. The issue is already tied to focused open PRs for the systemd handoff and approval-warning pieces, so cleanup should wait for those branches to be reviewed, hardened, and validated.

## What This Changes

The issue reports that Linux/systemd `openclaw gateway restart` can return failure even though the gateway is healthy immediately afterward.

## Best Possible Solution

Land a hardened Linux/systemd self-restart handoff that returns the existing scheduled restart contract for in-unit restarts, keep external restarts synchronous, and land the expired approval followup warning cleanup separately with focused proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection plus the systemd contract show a plausible current-main path: the same `KillMode=control-group` user unit is restarted synchronously, and a failed post-restart health snapshot still makes the command fail.

## Solution Assessment

No, current main is not the best solution because it still uses the synchronous systemd restart path and hard post-check failure. The detached handoff direction in the open systemd PR is the right shape, with the approval-warning PR handled as related log-noise cleanup rather than the whole root cause.
