## Summary

Keep open: current main still has a source-supported Linux/systemd false-negative restart path, and the related systemd handoff and approval-warning fixes are still open rather than merged or shipped.

## What This Changes

The issue reports that Linux/systemd `openclaw gateway restart` or update restart health checks can return failure even though the gateway service is running and probes healthy afterward.

## Best Possible Solution

Land or replace a focused Linux/systemd self-restart handoff that returns the existing scheduled restart contract for in-unit restarts, keep external restarts synchronous, and land the expired approval followup warning cleanup separately.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection shows the same current-main path: systemd restart returns `completed`, the scheduled-restart bypass is not used, and the CLI still fails on an unhealthy post-restart snapshot.

## Solution Assessment

No, current main is not the best way to solve the issue because it still treats systemd restarts as synchronous completed restarts. The maintainable direction is a narrow scheduled handoff for in-unit Linux/systemd self-restarts, with expired approval followup warning cleanup kept separate.
