## Summary

Keep open: current main still lacks native outbound WhatsApp mention metadata, and Baileys supports the `mentions` payload field, but this PR is not merge-ready because it regresses the current send-result/cache behavior and leaves LID resolver edge cases unresolved. The branch usefully carries forward #56863 for #39879, so the right path is a focused repair rather than cleanup closure.

## What This Changes

The PR adds WhatsApp outbound native mention resolution for numeric @ tokens in group text and media captions, with docs, changelog, and focused tests.

## Best Possible Solution

Land a repaired WhatsApp plugin change that adds Baileys `mentions` for group text and captions while preserving send-result delivery tracking, reconnect metadata fallback, and LID-priority/collision behavior with tests.

## Reproduction Assessment

Yes. Source inspection on current main shows outbound group text and caption sends are plain Baileys payloads without `mentions`, and PR-head raw files reproduce the send-result, metadata-cache, and LID-priority blockers at the cited lines.

## Solution Assessment

No, not as currently proposed. Baileys `mentions` is the right mechanism, but the branch needs a current-main repair before it is the narrow maintainable fix.

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the current send-result contract:** `extensions/whatsapp/src/inbound/send-api.ts:59`
  - body: Current main returns `WhatsAppSendResult` with `providerAccepted`, `messageIds`, and `keys`, including combined audio text sends and reactions. This branch returns only `{ messageId }` for sends and `void` for reactions, which would regress delivery tracking if merged as-is.
  - confidence: 0.92
- **[P2] Preserve the shared group metadata cache:** `extensions/whatsapp/src/inbound/monitor.ts:318-324`
  - body: The new `getGroupMeta` path only uses a local cache and returns an empty entry after `groupMetadata` fails. Current main preserves `options.groupMetadataCache` and has a reconnect-fallback test, so mention participant hydration needs to be added without dropping that recovery path.
  - confidence: 0.88
- **[P2] Prefer explicit LID participants when selecting mention JIDs:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:132-133`
  - body: `chooseMentionJid` normalizes `id` before `lid`, so a participant with a phone JID in `id` and a LID JID in `lid` records the LID digits with the phone target. Prefer a valid explicit LID JID before falling back to the phone id.
  - confidence: 0.86
- **[P3] Disambiguate phone and LID digit collisions:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:213`
  - body: The resolver checks `byPhone` before `byLid` for every numeric token, so a phone-number participant can shadow another participant whose LID digits match the same token. Add a deterministic collision or token-form rule and cover it in tests.
  - confidence: 0.74
