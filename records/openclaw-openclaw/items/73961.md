## Summary

Keep open: current main still lacks native outbound WhatsApp mention metadata, and this PR is useful but not merge-ready because it is stale against current send-result and group-metadata cache behavior and still has resolver edge-case bugs.

## What This Changes

The PR adds WhatsApp outbound @mention metadata for group text and media captions by resolving numeric mention tokens against participant metadata.

## Best Possible Solution

Port the mention resolver onto current main's WhatsApp send-result and shared group-metadata cache implementation, add focused LID-priority and collision tests, and keep the scope limited to native outbound group mention metadata.

## Reproduction Assessment

Yes. Current main can be checked directly: outbound WhatsApp sends build text and caption payloads without `mentions`, and the PR blockers reproduce by comparing PR head against current `WhatsAppSendResult` and group metadata cache code plus pure resolver inputs.

## Solution Assessment

No, not as currently proposed. Baileys `mentions` is the right mechanism, but the branch needs a current-main repair before it is the narrow maintainable fix.

  - body: Current main normalizes sends into `WhatsAppSendResult` with `providerAccepted`, `messageIds`, and `keys`, including combined audio text results and reactions.   - body: The new `getGroupMeta` path only uses a local cache and returns an empty entry after `groupMetadata` fails.   - confidence: 0.86
- **[P2] Prefer explicit LID participants when selecting mention JIDs:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:132-133`
  - body: `chooseMentionJid` normalizes `id` before `lid`, so a participant with a phone JID in `id` and a LID JID in `lid` records the LID digits with the phone target. Prefer a valid LID JID when the explicit `lid` field is present.
  - confidence: 0.86
- **[P3] Disambiguate phone and LID digit collisions:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:213`
  - body: The resolver always checks `byPhone` before `byLid`, so a phone-number participant can shadow another participant whose LID digits match the same token. Add an explicit collision rule or token-form rule so sender intent is preserved.
  - confidence: 0.74
