Full review comments:

- **[P2] Preserve the current send-result contract:** `extensions/whatsapp/src/inbound/send-api.ts:59`
  - body: This branch is based before the current `WhatsAppSendResult` flow: `sendMessage` returns only `{ messageId }`, and reactions return `void`, while current main normalizes `providerAccepted`, `messageIds`, `keys`, and combined audio text results. Apply the mention wiring to the current `normalizeWhatsAppSendResult`/`combineWhatsAppSendResults` implementation so delivery tracking does not regress.
  - confidence: 0.92
- **[P2] Preserve the shared group metadata cache:** `extensions/whatsapp/src/inbound/monitor.ts:318-324`
  - body: The new `getGroupMeta` path uses only a local cache and returns an empty entry when `groupMetadata` fails. Current main keeps a shared reconnect cache and falls back to hydrated group metadata after transient fetch failures; preserve that cache path while adding `mentionParticipants`, or group inbound recovery can regress after reconnect churn.
  - confidence: 0.86
- **[P2] Prefer explicit LID participants when selecting mention JIDs:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:131-133`
  - body: `chooseMentionJid` normalizes `id` before `lid`, so a participant with `id: "15551234567@s.whatsapp.net"` and `lid: "277038292303944@lid"` records the LID digits with a phone JID target. That makes visible LID tokens resolve to a phone JID in LID-addressed groups; prefer a valid `lid` JID over a phone `id` when both are present.
  - confidence: 0.87
- **[P3] Disambiguate phone and LID digit collisions:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:213`
  - body: The resolver always does `byPhone.get(digits) ?? byLid.get(digits)`, so if one participant's phone number equals another participant's LID digits, a visible LID token resolves to the phone user. Use an explicit rule based on the token/participant metadata so sender intent is not silently shadowed.
  - confidence: 0.74
