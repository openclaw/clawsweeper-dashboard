## Summary

Keep this PR open: current main still lacks WhatsApp outbound mention metadata, and the branch is a useful scoped implementation, but PR head still has resolver bugs around LID priority and phone/LID digit collisions before merge.

## What This Changes

This PR adds WhatsApp outbound group mention resolution for text and media captions, with docs, changelog, and focused WhatsApp plugin tests.

## Best Possible Solution

Keep the implementation scoped to native outbound WhatsApp group mention metadata, but repair the resolver to prefer explicit LID JIDs and define collision behavior before merge.

## Reproduction Assessment

Yes. Source inspection shows current main sends WhatsApp text and captions without `mentions`, and PR-head resolver inputs with `{ id: phoneJid, lid: lidJid }` or equal phone/LID digits demonstrate the remaining misrouting paths.

## Solution Assessment

No, not yet. Baileys `mentions` is the right mechanism and the branch now preserves the send-result and cache contracts, but the LID selection and collision rules need a narrow resolver repair.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Prefer explicit LID JIDs for mention targets:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:132-133`
  - body: Baileys participants can carry both a phone JID in `id` and an explicit LID in `lid`, and `monitor.ts` passes both fields into this resolver. Choosing `id` first records the LID digits with the phone target, so native mentions in LID-backed groups can be ignored. Prefer a valid `lid` JID when present before falling back to phone identifiers.
  - confidence: 0.88
- **[P3] Define phone and LID digit collision handling:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:213`
  - body: For `@<digits>` tokens, `byPhone` always wins over `byLid`. If a phone-number participant has the same digits as another participant's LID, the LID participant cannot be mentioned. Add an explicit precedence or token-form rule and cover it with a resolver test.
  - confidence: 0.75
