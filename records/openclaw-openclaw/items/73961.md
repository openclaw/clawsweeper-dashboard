Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the current send-result contract:** `extensions/whatsapp/src/inbound/send-api.ts:59`
  - body: This branch returns only `{ messageId }` from `sendMessage`, but current main returns `WhatsAppSendResult` with `providerAccepted`, `messageIds`, and combined audio-text results. Merging this shape would regress WhatsApp auto-reply delivery tracking, so port the mention wiring onto the current `normalizeWhatsAppSendResult`/`combineWhatsAppSendResults` flow.
  - confidence: 0.92
- **[P2] Prefer explicit LID participants when selecting mention JIDs:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:129-136`
  - body: `chooseMentionJid` normalizes `id` before `lid`, so a participant shaped with a phone JID in `id` and a LID JID in `lid` records the LID digits with the phone target. That makes bare LID tokens resolve to a phone JID in LID-backed groups; prefer a valid `lid` JID before falling back to `id`.
  - confidence: 0.86
- **[P3] Disambiguate phone and LID digit collisions:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:213`
  - body: The resolver always does `byPhone.get(digits) ?? byLid.get(digits)`, so if a phone number equals another participant's LID digits, a visible LID token resolves to the phone user. Since this feature supports LID group tokens, use token intent, such as `+` for phone lookup and bare tokens for LID when both maps contain the key.
  - confidence: 0.74
