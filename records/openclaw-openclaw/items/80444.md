## Summary

Keep open; the PR targets a real persistence redaction gap, but the new recursion cutoff can leave nested small `toolResult.details` subtrees unredacted, so it should not merge as-is.

## What This Changes

The PR redacts secret-like strings in persisted `toolResult.details`, including small payloads and oversized summaries, and adds guard regression coverage.

## Best Possible Solution

Keep the persistence-boundary redaction approach, but make the deep details walk cycle-safe and ensure over-depth subtrees cannot preserve raw secret strings.

## Reproduction Assessment

Yes at source level: current main persists small details unchanged, and the PR body shows a focused `guardSessionManager` proof for shallow fake-token details. The remaining blocker is a source-reproducible nested-details bypass introduced by the proposed depth cutoff.

## Solution Assessment

No as written. Reusing the shared redactor at the persistence boundary is the right direction, but returning over-depth objects unchanged is not a safe solution for secret-bearing metadata.
