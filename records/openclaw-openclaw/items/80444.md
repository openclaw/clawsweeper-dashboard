## Summary

Keep this PR open: the requested security fix is valid, the body now includes usable after-fix behavior proof, but the patch still has a blocking redaction gap for long structured secret strings that cross the fixed lookahead window.

## What This Changes

The PR redacts persisted `toolResult.details` metadata through the session tool-result guard and adds focused redaction regression tests.

## Best Possible Solution

Keep the PR open and make long-string persisted-detail redaction truncation-aware so structured secrets cannot leak when their closing delimiter falls past the scan window.

## Reproduction Assessment

Yes, at source level. Current main preserves small `toolResult.details` unchanged, and the PR finding is reproducible from source with a long `{"token":"..."}` detail string whose closing quote falls beyond the new fixed lookahead.

## Solution Assessment

No, not yet. The intended persistence-boundary redaction is the right direction, but the fixed-prefix scan should be replaced with a truncation-aware redaction strategy or a safe fallback when the persisted prefix cannot be proven redacted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact long structured secrets before truncating:** `src/agents/session-tool-result-guard.ts:91-93`
  - body: This only runs the redactor on `value.slice(0, scanChars)` before persisting the first 2,000 chars. For structured patterns that require a closing delimiter, such as `{"token":"<long secret>"}` or a PEM block, the delimiter can fall past the 6,096-char scan window, so the regex never matches and the persisted prefix still contains raw secret bytes. Please make this truncation-aware or fall back to a safe summary when a long secret-like field starts inside the persisted prefix.
  - confidence: 0.88
