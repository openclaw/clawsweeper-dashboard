## Summary

Keep open: the PR is a useful persistence-boundary redaction fix, but latest head still leaves a concrete secret-persistence leak in the oversized-details summary/fallback path because non-string retained fields are copied unchanged.

## What This Changes

The PR redacts secret-like strings in persisted `toolResult.details`, passes logging redaction config into the guard, broadens field-key redaction, and adds focused regression tests.

## Best Possible Solution

Keep the persistence-boundary redaction approach, but recursively redact or omit every retained oversized-summary, fallback, and child-session structured field before session JSONL writes.

## Reproduction Assessment

Yes, at source level. Current main preserves small details unchanged, and latest PR head still copies non-string allowlisted fields from oversized details, so `details.status = { token: ... }` plus an oversized field can persist a raw nested token.

## Solution Assessment

No, not yet. The persistence boundary is the right place to solve this, but the summary and fallback branches need the same recursive redaction guarantee before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact retained summary objects before persisting:** `src/agents/session-tool-result-guard.ts:362`
  - body: In the oversized-details path, retained allowlisted fields are redacted only when they are strings; objects are copied as-is. A payload like `details.status = { token: ... }` plus oversized `aggregated` bypasses the small-details recursive redaction and writes the raw token under `status`, and the fallback has the same shape. Recursively redact retained structured values or reduce them to safe scalar metadata.
  - confidence: 0.91
