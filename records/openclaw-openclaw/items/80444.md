## Summary

Keep open. The PR is pointed at the right persistence boundary and the body proof covers a small fake-token details case, but the current head still has a security-significant truncation bug for long structured secrets.

## What This Changes

The PR redacts persisted `toolResult.details`, applies redaction to summarized oversized detail fields, broadens env-style key masking, and adds focused regression tests.

## Best Possible Solution

Keep persistence-boundary redaction, but make long-string handling truncation-aware or fail closed to a compact summary when a structured secret begins before the persisted prefix and cannot be fully matched.

## Reproduction Assessment

Yes, at source level. Current main preserves complete small `toolResult.details` unchanged, and the PR finding follows from redacting only a truncated prefix before persisting oversized string details.

## Solution Assessment

No, not yet. The persistence-boundary approach is maintainable, but the long-string branch needs a safe truncation-aware strategy before this fixes the secret persistence problem reliably.
