## Summary

Keep open: the UI-freeze report is plausible and not fixed on current main, especially because scoped config pages still normalize the full config schema before section scoping. The Telegram event-loop-blocking explanation is not proven by source inspection, and the older related #45698 remains open for the broader Control UI sluggishness thread.

## What This Changes

The issue asks for Communications, AI & Agents, and Automation menus to stay responsive despite slow Telegram health probes and large config schema rendering.

## Best Possible Solution

Profile this together with #45698, then land a narrow fix that reduces synchronous full-schema work in scoped config tabs and preserves bounded, non-blocking channel probes using the existing Telegram timeout/network controls unless a real config gap is proven.

## Reproduction Assessment

No high-confidence end-to-end reproduction yet. Source inspection does show a plausible current-main frontend path where scoped config tabs synchronously analyze the full schema, but it does not prove the reported Telegram getMe event-loop stall.

## Solution Assessment

Unclear. The likely first fix is frontend profiling and scoped/memoized schema rendering; adding a new Telegram httpTimeoutMs option is probably not the best first step because timeoutSeconds and DNS controls already exist.
