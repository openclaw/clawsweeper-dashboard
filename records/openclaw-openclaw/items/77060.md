Keep open: current main still has source-backed latency paths for config-heavy settings pages and live channel probes, but the Telegram health-monitor explanation is only partly consistent with the code and should be routed with the broader #45698 Control UI sluggishness tracker.

## What This Changes

The issue requests that dashboard-v2 settings menus and channel status probes remain responsive when Telegram Bot API health checks or large config schemas are slow.

## Best Possible Solution

Land one focused responsiveness fix for the #45698/#77060 cluster: use route-scoped or lazy config schema rendering for settings sections, and keep channel status visible while live probes complete in the background or return partial results.

The narrower maintainable path is to decouple visible UI rendering from full config-schema work and live channel probes while using the existing Telegram timeout, proxy, and DNS controls for network tuning.

## Repair Work Prompt

Fix the Control UI responsiveness bug reported in https://github.com/openclaw/openclaw/issues/77060 and related https://github.com/openclaw/openclaw/issues/45698. Current main shows two source-backed latency paths: settings tabs such as Communications, Automation, and AI & Agents call `config.schema`/`config.get` and analyze the full schema before scoping to the selected section, and the Channels tab awaits `channels.status` with `probe:true` and an 8s timeout while Telegram `probeAccount` can spend that budget in `getMe`. Implement a narrow fix that renders the selected settings surface without full-schema analysis when scoped data is enough, preferably using the existing `config.schema.lookup` API or an equivalent route-scoped cache, and make live channel probes nonblocking/partial for the visible Channels UI. Preserve existing Telegram `timeoutSeconds`, proxy, and DNS behavior; do not add a new `httpTimeoutMs` config option unless the code proves the existing contract cannot cover it. Add focused regression coverage for scoped settings rendering and channel status loading behavior. Add a user-facing changelog entry if runtime/UI behavior changes.
