## Summary

Keep this PR open: it addresses the legacy shared OpenAI Responses path and adds focused tests, but it is not merge-ready because the transport-aware OpenAI/Azure Responses send paths can still forward `content: null` after `onPayload`, and the real behavior proof is still a strict-provider mock rather than live provider output.

## What This Changes

The PR adds `sanitizeResponsesInput` to normalize null OpenAI Responses message content in the shared converter and shared lifecycle, with helper and lifecycle regression tests.

## Best Possible Solution

Share the null-content sanitizer across every OpenAI Responses SDK send boundary, especially `src/agents/openai-transport-stream.ts`, and pair it with focused transport-aware regression coverage plus live strict-provider proof or explicit maintainer acceptance of mock proof.

## Reproduction Assessment

Yes, source-reproducible: inject `content: null` into a Responses `input` array through `onPayload` and the current SDK-bound payload can carry an invalid shape. The linked report supplies a strict-provider 400 for that same schema violation, though I did not rerun a live provider.

## Solution Assessment

No. Sanitizing before SDK serialization is the right shape, but the current PR applies it to only one lifecycle and misses the current transport-aware OpenAI/Azure Responses send paths.
