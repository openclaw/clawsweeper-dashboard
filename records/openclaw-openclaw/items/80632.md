## Summary

Keep open: the PR addresses a real Slack thread-routing problem and supplies live proof, but it still misses the exported durable Slack send path and can override the documented `topLevel` opt-out for mission-tagged sends.

## What This Changes

The PR adds Slack mission-thread detection, a file-backed mission-thread registry, dispatcher/tool/outbound routing changes, and tests to keep mission-scoped Slack replies in canonical threads.

## Best Possible Solution

Put the mission-thread invariant in one Slack-owned send-context helper used by action, lazy outbound, and exported channel message sends, while preserving documented explicit top-level opt-outs and proving fresh and upgrade behavior.

## Reproduction Assessment

Yes. Source inspection at the PR head shows the explicit Slack top-level opt-out is followed by mission-thread lookup, and current main’s durable Slack message bridge reaches an exported sendText path that still lacks mission-thread resolution.

## Solution Assessment

No. The registry and mission-thread direction are plausible, but the implementation needs to preserve the existing Slack top-level contract and apply the invariant at the shared exported send surface rather than only the lazy outbound helper.
