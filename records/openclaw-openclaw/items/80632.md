## Summary

Keep open: the reported Slack mission-thread problem is plausible and the PR includes useful live smoke evidence, but the patch is not mergeable against current main because it targets removed legacy Slack paths, adds broad mission detection, and puts Slack-specific routing policy into core reply routing.

## What This Changes

The PR adds a file-backed Slack mission-thread registry and routes selected mission-scoped Slack replies, status updates, and tool sends to a canonical thread or fails closed.

## Best Possible Solution

Port the useful mission-thread routing idea into `extensions/slack` through the existing channel threading/message seams, with an explicit mission identity contract and focused Slack plugin tests plus live Slack proof.

## Reproduction Assessment

Yes for the PR defects: current source and the PR diff show the patch targets obsolete Slack paths, bypasses the channel threading seam, and classifies branch-like text as mission scope. The original live Slack misrouting claim is plausible from current `MessageThreadId` handling, but I did not reproduce it live against current main.

## Solution Assessment

No. The maintainable direction is to rework this through the current Slack plugin and a narrow mission contract rather than adding a legacy `src/slack` registry and Slack-specific policy in core `routeReply`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move the routing change to the Slack plugin:** `src/slack/monitor/message-handler/prepare.ts:533`
  - body: Current main no longer uses `src/slack` for Slack runtime code; the active handlers live under `extensions/slack/src`. This PR changes obsolete paths, so the intended inbound/threading behavior would not be wired into the current Slack plugin after a rebase.
  - confidence: 0.94
- **[P2] Narrow mission detection before failing closed:** `src/slack/mission-threads.ts:20`
  - body: `detectSlackMissionId` treats any text containing `feat/...`, `fix/...`, `chore/...`, or `codex/...` as mission-bound. Because the new send paths fail closed without a registry entry, ordinary Slack messages that mention a branch name can throw instead of sending.
  - confidence: 0.88
- **[P2] Keep Slack routing policy out of core routeReply:** `src/auto-reply/reply/route-reply.ts:13`
  - body: Importing `../../slack/mission-threads.js` from the provider-agnostic reply router bypasses the current plugin threading seam. Slack-specific reply transport should stay in the Slack plugin or a generic channel contract so core does not grow bundled channel policy.
  - confidence: 0.9
