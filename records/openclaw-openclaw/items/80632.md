## Summary

Keep open: the current PR head still broadens mission detection to ordinary branch-like Slack text and writes durable thread routes before Slack ingress accepts the message, so it can misroute or block normal Slack sends.

## What This Changes

The branch adds a Slack mission-thread registry and wires inbound preparation, dispatcher reply planning, Slack sendMessage actions, outbound sends, and tests to resolve mission-bound messages to canonical threads.

## Best Possible Solution

Keep the Slack-owned routing fix, but derive mission identity from an explicit trusted routing contract, persist canonical thread state only after accepted Slack ingress, and preserve ordinary non-mission Slack sends.

## Reproduction Assessment

Yes for the PR defects: source inspection of the current PR diff shows branch-like text is classified as mission scope and prepareSlackMessage writes the registry before later return-null ingress gates. I did not independently live-reproduce the original Slack misrouting beyond the PR's supplied live output.

## Solution Assessment

No. The Slack plugin is the right boundary, but the current mission identity and persistence ordering are too broad to be the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist mission threads only after ingress accepts the message:** `extensions/slack/src/monitor/message-handler/prepare.ts:544-553`
  - body: The registry write runs before the ambiguous-thread, mention, sender, bot-message, and control-command gates below it. A room message that matches mission text can be saved even when this handler later returns null, so later sends may route into a Slack thread OpenClaw intentionally ignored.
  - confidence: 0.88
- **[P2] Require explicit mission identity before fail-closed routing:** `extensions/slack/src/mission-threads.ts:20-23`
  - body: The detector treats ordinary branch names such as fix/foo or feat/bar, plus the hard-coded THREADING TEST phrase, as mission scope. Because sendMessage and outbound sends now throw when a detected mission has no registry entry, normal Slack messages mentioning branch names can fail instead of sending top-level.
  - confidence: 0.9
