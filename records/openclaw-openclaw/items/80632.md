## Summary

Keep open: the latest branch addressed the earlier mission-detector and pre-ingress persistence findings, but the new durable registry is still scoped only by mission id/channel id, which can overwrite or misresolve canonical mission threads in supported multi-account Slack setups.

## What This Changes

The PR adds Slack mission-thread detection, persistence, routing, tool-send/outbound resolution, and focused tests so mission-bound Slack replies stay in canonical threads.

## Best Possible Solution

Keep the Slack plugin routing fix, but make durable mission-thread state account/workspace scoped and cover that contract with multi-account regression tests.

## Reproduction Assessment

Yes for the blocking PR defect by source inspection: OpenClaw supports multiple Slack accounts, while the PR persists and resolves mission threads without account or workspace identity.

## Solution Assessment

No. The Slack plugin boundary and explicit mission marker are appropriate, but the registry must include Slack account/workspace scope before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope mission threads by Slack account:** `extensions/slack/src/mission-threads.ts:100`
  - body: The registry writes every route under `store.missions[missionId]` and resolution only checks `channelId`. OpenClaw supports multiple Slack accounts/workspaces, and the action path already resolves an `accountId`, so two accounts using the same mission name can overwrite each other or make the first account fail closed after the second write. Include account/workspace identity in the store key and in all persist/resolve calls.
  - confidence: 0.88
