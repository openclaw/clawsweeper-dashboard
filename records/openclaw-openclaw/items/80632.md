## Summary

The PR adds Slack mission-thread detection, a file-backed registry, dispatcher/tool/outbound routing changes, and regression tests so mission-scoped Slack replies stay in canonical threads.

## Best Possible Solution

Keep the PR open and repair the Slack send boundary so dispatcher, tool, payload, exported text, and exported media sends share one mission-thread resolution and fail-closed contract.

No. The registry and mission-thread direction are appropriate, but the invariant needs to live in the shared Slack send context or exported text/media adapter paths, not only the lazy outbound helper.
