## Summary

Keep this PR open for rework. The useful Slack mission-thread routing idea is now in the active Slack plugin surface and the PR body includes real Slack smoke output, but the patch still has blocking correctness issues around broad mission detection and persisting durable thread state before Slack ingress has accepted the message.

## What This Changes

Adds a Slack mission-thread detector and file-backed registry, then wires inbound preparation, dispatcher replies, sendMessage actions, outbound sends, and threading tests to resolve or fail closed on canonical mission threads.

## Best Possible Solution

Keep the Slack-owned routing fix, but derive mission identity from an explicit trusted context or documented contract, persist canonical thread state only after an inbound Slack message is accepted, and cover fail-closed behavior with focused Slack plugin regressions.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the registry write is inserted before later ingress skips, and the detector matches ordinary branch-like text used by the new fail-closed paths. I did not live-reproduce the original Slack misrouting claim in this read-only review.

## Solution Assessment

No. The revised patch is in the right Slack plugin area, but the mission detection and durable persistence rules are too broad to be the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist mission threads only after ingress accepts the message:** `extensions/slack/src/monitor/message-handler/prepare.ts:547-548`
  - body: The new registry write runs before the no-mention, sender, bot, and command ingress gates below it. A channel message that matches `Mission:` or a branch-like token can be recorded as the canonical mission thread even when this handler later returns `null`, so later sends may route into a thread OpenClaw intentionally ignored.
  - confidence: 0.88
- **[P2] Require an explicit mission marker before fail-closed routing:** `extensions/slack/src/mission-threads.ts:20-23`
  - body: `detectSlackMissionId` treats ordinary branch text such as `fix/foo` or `feat/bar` and the phrase `THREADING TEST` as mission scope. Because the new Slack action and outbound paths throw when a matched mission lacks a registry entry, normal Slack messages that mention branch names can start failing instead of sending top-level.
  - confidence: 0.9
