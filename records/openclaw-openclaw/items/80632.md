## Summary

Keep open: the force-push addressed the earlier broad detector and persistence-order findings, but the latest detector still misses common Slack app-mention mission prompts that include the bot mention before the Mission marker, so the central Slack mission-thread fix remains incomplete.

## What This Changes

The PR adds a Slack mission-thread JSON registry and routes inbound preparation, dispatcher replies, Slack sendMessage actions, outbound sends, and tests through canonical mission threads or fail-closed mission sends.

## Best Possible Solution

Keep the Slack-owned routing fix, but make mission identity extraction accept explicit Mission markers after leading Slack mention tokens and cover that path in threading/preparation tests.

## Reproduction Assessment

Yes for the PR defect by source inspection: Slack app-mention text can include mention tokens, while the PR's anchored `Mission:` detector is fed raw `message.text`, so `<@bot> Mission: ...` is not detected. I did not run a live Slack repro in this read-only review.

## Solution Assessment

No. The Slack plugin boundary is the right place, but the current detector needs to normalize Slack mention prefixes before it can reliably guarantee canonical mission threads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle leading Slack mentions in mission detection:** `extensions/slack/src/mission-threads.ts:21-22`
  - body: Slack app-mention events can include the bot mention in raw `message.text`, for example `<@U...> Mission: foo`. Because this detector only matches `Mission:` at the start of a line and the PR feeds it raw Slack text from threading, routing, and persistence, those mission prompts are not marked mission-bound under `replyToMode=off`; no `MessageThreadId` or registry entry is set, so replies can still escape to a top-level channel post. Strip or tolerate leading Slack mention tokens before checking the explicit mission marker.
