## Summary

Keep this PR open, but the current patch should not merge: it is based on a March merge base, changes obsolete `src/slack` runtime paths, and bypasses the current Slack plugin/threading architecture on `main`. The live Slack smoke text in the PR body is useful evidence, but the implementation needs to be ported and the mission-detection policy needs maintainer review.

## What This Changes

The PR adds a file-backed Slack mission-thread registry and wires selected mission replies/tool sends to resolve a canonical Slack thread or fail closed.

## Best Possible Solution

Port the useful Slack mission-thread idea through `extensions/slack` and the existing channel threading/message seams, with an explicit mission identity contract that does not classify ordinary branch names as mission-bound messages.

## Reproduction Assessment

Yes for the PR defects: source and compare inspection show the patch targets obsolete Slack paths and imports Slack-specific logic into core. The original live Slack misrouting claim is plausible from current `MessageThreadId` handling, but it was not reproduced live in this read-only review.

## Solution Assessment

No. The proposed direction needs to be reworked for the current Slack plugin architecture and should not rely on broad branch-name text detection as the mission boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move the routing change to the Slack plugin:** `src/slack/monitor/message-handler/prepare.ts:533`
  - body: Current main no longer uses `src/slack` for Slack runtime code; the active handlers live under `extensions/slack/src`. This PR changes obsolete paths, so the intended inbound/threading behavior would not be wired into the current Slack plugin after a rebase.
  - confidence: 0.94
- **[P2] Avoid treating branch names as mission scope:** `src/slack/mission-threads.ts:20`
  - body: `detectSlackMissionId` classifies any text containing `feat/...`, `fix/...`, `chore/...`, or `codex/...` as mission-bound. Because the new send paths fail closed without a registry entry, ordinary top-level Slack messages that mention a branch name can start throwing instead of sending.
  - confidence: 0.88
- **[P2] Keep Slack-specific routing out of core routeReply:** `src/auto-reply/reply/route-reply.ts:13`
  - body: Importing `../../slack/mission-threads.js` from the provider-agnostic reply router bypasses the current plugin threading seam. Slack-specific reply transport should stay in the Slack plugin or a generic channel contract so core does not grow bundled channel policy.
  - confidence: 0.9
