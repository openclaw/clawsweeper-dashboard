## Summary

Keep this PR open: the Slack mission-thread direction is useful and the supplied live proof is sufficient, but the latest head still misses the exported Slack channel outbound text/media path, so mission-bound durable sends can bypass the new registry and fail-closed guard.

## What This Changes

The PR adds Slack mission detection, a file-backed mission-thread registry, dispatcher/tool/outbound thread routing, and regression tests for mission-scoped Slack replies.

## Best Possible Solution

Centralize mission-thread resolution and the fail-closed invariant in the shared Slack send context or exported Slack outbound text/media paths so payload, text, media, tool, dispatcher, and routed durable sends all share the same canonical thread contract.

## Reproduction Assessment

Yes by source inspection for the blocking PR defect: mission text delivered through the channel message adapter can still reach slackChannelOutbound.sendText, which only forwards an explicit threadTs value. I did not run a live failing Slack repro because this review was read-only.

## Solution Assessment

No. The registry and dispatcher/tool direction are appropriate, but the implementation needs to cover the exported Slack outbound text/media adapter rather than only the lazy payload helper.
