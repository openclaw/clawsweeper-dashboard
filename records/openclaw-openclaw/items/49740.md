## Summary

Keep open. Current main still lacks per-job recurring cron retry controls, and open PR #51001 is an unmerged implementation candidate that links this issue for closure, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request configurable recurring cron retry options so failed jobs can retry after a delay before their next natural scheduled run.

## Best Possible Solution

Review and land PR #51001 or an equivalent implementation that adds durable per-job recurring retry configuration, CLI add/edit flags, protocol/schema support, scheduler state that retries before the natural schedule without cadence drift, docs, and focused regression tests.

## Reproduction Assessment

Yes. The current behavior is reproducible from the source path: a recurring job error schedules the later of the natural next run and the error backoff, and current CLI/protocol surfaces provide no way to configure per-job recurring retries.
