## Summary

Keep open. Current main provides global cron retry/backoff behavior for transient failures, but it still does not implement the requested per-job `openclaw cron add` / `openclaw cron edit` retry count, delay, or backoff controls; an open implementation PR also still specifically targets this issue.

## What This Changes

The issue requests per-cron retry controls so failed scheduled jobs can retry after transient provider failures instead of waiting for the next natural schedule.

## Best Possible Solution

Keep this issue open as the canonical product/API decision, then either refresh and review the linked fork PR or replace it with a maintainer-approved per-job retry design that layers cleanly on the existing global retry policy.

## Reproduction Assessment

## Solution Assessment
