## Summary

Keep open: current main and the latest release already have global transient retry behavior for recurring cron jobs, but the requested per-job CLI/API retry knobs are still absent and the issue is linked to an open, conflicting fork PR that needs maintainer product direction rather than a new ClawSweeper fix PR.

## What This Changes

The issue requests configurable cron auto-retry options such as `--retry-count`, `--retry-delay`, and optional backoff so transient failures do not defer daily jobs until their next natural schedule.

## Best Possible Solution

Keep the shipped global recurring retry behavior, then have maintainers either close this as satisfied by `cron.retry` or sponsor a narrow per-job retry design with CLI, protocol, persistence, docs, and upgrade proof.

## Reproduction Assessment

Yes for the remaining feature gap from source: `cron add` and `cron edit` do not define the requested per-job retry flags, while timer source and tests show current main already retries transient recurring failures globally. I did not run a live cron because the shipped source/test path answers the central behavior and the remaining request is API/product surface.

## Solution Assessment

Unclear as requested: global `cron.retry` is already the narrow shipped solution for the no-retry failure mode, but per-job CLI flags would be a new configuration/API surface. The best next step is maintainer direction on whether that extra per-job control should exist.
