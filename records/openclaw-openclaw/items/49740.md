## Summary

Current main and v2026.6.6 now implement the central cron retry behavior: recurring transient failures retry before the next natural scheduled slot using the existing cron retry policy, with regression coverage and docs. The exact per-job CLI flags did not land, but the daily-cron missed-output problem is addressed well enough; job-specific controls can be a narrower follow-up if maintainers still want that UX.

## What This Changes

Request configurable cron auto-retry behavior so failed recurring jobs retry after a delay instead of waiting for the next scheduled run.

## Best Possible Solution

Keep the shipped scheduler-level cron.retry behavior; track job-specific CLI controls or retry-list status only as a narrower enhancement if maintainers still want that UX.
