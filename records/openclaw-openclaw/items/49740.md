## Summary

Keep open. Current main and v2026.6.8 now ship global transient retry behavior for recurring cron jobs, but the requested per-job `cron add` / `cron edit` retry controls and persisted job/API contract are still missing, and the linked implementation branch remains unmerged and conflicting.

## What This Changes

Request per-job retry options for cron add/edit so failed recurring jobs retry after a delay or backoff before the next natural schedule.

## Best Possible Solution

Keep the shipped global retry behavior, then make an explicit maintainer decision on additive per-job cron retry controls before landing a fresh or repaired CLI/Gateway/job-schema/docs/tests implementation.

## Reproduction Assessment

Yes, at source level for the remaining request. Current main has global recurring retry behavior, but CLI add/edit, CronJob types, and Gateway schemas still do not expose per-job retry count/delay/backoff controls.

## Solution Assessment

Not yet. The shipped global `cron.retry` behavior is a good partial fix for the no-retry symptom, but the requested per-job API should only land after maintainers settle the durable job/protocol semantics and side-effect safeguards.
