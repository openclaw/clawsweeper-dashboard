Full review comments:

- **[P1] Preserve durable direct cron delivery:** `src/cron/isolated-agent/delivery-dispatch.ts:220-232`
  - body: This branch sends direct cron payloads through plain `deliverOutboundPayloads`. Current main uses the durable batch path with idempotency, retry, required durability, and partial-failure handling, so rebasing this as-is would regress scheduled delivery reliability.
  - confidence: 0.91
- **[P1] Fail cron runs when announce delivery fails:** `src/cron/isolated-agent/delivery-dispatch.ts:409-418`
  - body: When `runSubagentAnnounceFlow` returns false or throws, the non-best-effort path returns `status: "ok"` with `delivered: false`. One-shot jobs can then be recorded successful or deleted even though no message was delivered.
  - confidence: 0.86
- **[P2] Keep structured cron payloads on direct delivery:** `src/cron/isolated-agent/delivery-dispatch.ts:499-503`
  - body: The direct-delivery gate sends structured media/channel payloads through the announce/text path whenever cleanup, follow-up, or silent-token conditions apply. That path only forwards synthesized text, so media or channelData cron outputs can be dropped.
  - confidence: 0.84
- **[P2] Require explicit main announce delivery input:** `src/cli/cron-cli/register.cron-add.ts:221-247`
  - body: `--channel` defaults to `last`, so `hasDeliveryTarget` is truthy even when the user did not request delivery. Main system-event jobs therefore become announce jobs by default, and `--no-deliver` can still be overridden by the default channel.
  - confidence: 0.9
- **[P2] Preserve heartbeat defaults when fields are omitted:** `src/cron/service/timer.ts:1079-1083`
  - body: The override includes `to` and `accountId` even when they are undefined, then later spreads over configured heartbeat defaults. Main cron announce jobs that rely on configured recipients or accounts can lose those defaults and route incorrectly or be skipped.
  - confidence: 0.83
