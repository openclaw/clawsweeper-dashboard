## Best Possible Solution

Split or replace this with focused current-main PRs for main-session cron announce routing and Feishu `union_id` handling, remove unrelated Docker/secret-scan/test-skip churn, and require redacted live Telegram/Feishu proof before merge.

No. The goals are useful, but this branch is too broad and introduces delivery, security, and validation regressions; focused replacement PRs are the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Gate main announce delivery on explicit user input:** `src/cli/cron-cli/register.cron-add.ts:221-247`
  - body: `--channel` defaults to `last`, so `hasDeliveryTarget` is truthy even when the user did not request delivery. Main system-event jobs therefore get `delivery.mode="announce"` by default, and `--no-deliver` can still be overridden by the default channel.
  - confidence: 0.9
- **[P2] Omit undefined heartbeat override fields:** `src/cron/service/timer.ts:1077-1083`
  - body: The override includes `to` and `accountId` even when they are undefined, and the heartbeat runner spreads it over configured heartbeat defaults. Main cron announce jobs that rely on configured default recipients or accounts can lose those defaults and route incorrectly or be skipped.
  - confidence: 0.84
- **[P2] Keep structured cron payloads on the direct path:** `src/cron/isolated-agent/delivery-dispatch.ts:499-503`
  - body: The new gate no longer forces structured payloads through direct delivery. If `deleteAfterRun`, follow-up, or silent-token conditions are true, media/channel payloads fall into the announce path, which only forwards synthesized text and can drop the structured output.
  - confidence: 0.86
- **[P2] Return errors for required announce delivery failures:** `src/cron/isolated-agent/delivery-dispatch.ts:409-418`
  - body: When `runSubagentAnnounceFlow` reports no delivery, the non-best-effort path still returns `status: "ok"` with an error string. That makes failed required delivery look successful to cron scheduling and can delete one-shot jobs without delivering the message.
  - confidence: 0.86
