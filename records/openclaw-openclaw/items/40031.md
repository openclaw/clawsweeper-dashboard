- **[P1] Preserve durable direct cron delivery:** `src/cron/isolated-agent/delivery-dispatch.ts:220-232`
  - body: This branch sends direct cron payloads through plain `deliverOutboundPayloads`. Current main uses the durable batch path with retry, idempotency, partial-failure, stale-delivery, and queue protections, so rebasing this as-is would regress direct cron delivery reliability.
  - confidence: 0.9
- **[P2] Require explicit main announce delivery input:** `src/cli/cron-cli/register.cron-add.ts:221-247`
  - body: `--channel` defaults to `last`, so `hasDeliveryTarget` is truthy even when the user did not request delivery. Main system-event jobs therefore become announce jobs by default, and `--no-deliver` can still be overridden by the default channel.
  - confidence: 0.9
- **[P2] Preserve heartbeat defaults when fields are omitted:** `src/cron/service/timer.ts:1079-1083`
  - body: The override includes `to` and `accountId` even when they are undefined, and later heartbeat paths spread it over configured heartbeat defaults. Main cron announce jobs that rely on configured recipients or accounts can lose those defaults and route incorrectly or be skipped.
  - confidence: 0.84
- **[P2] Keep structured cron payloads on direct delivery:** `src/cron/isolated-agent/delivery-dispatch.ts:499-503`
  - body: The direct-delivery gate no longer checks `deliveryPayloadHasStructuredContent`, so structured payload jobs can fall through to the announce/text path when cleanup, follow-up, or silent-token conditions are true. That path only forwards synthesized text, so media or channel-data cron outputs can be dropped.
  - confidence: 0.86
- **[P1] Fail non-best-effort announce delivery errors:** `src/cron/isolated-agent/delivery-dispatch.ts:409-418`
  - body: When `runSubagentAnnounceFlow` returns false or throws, the non-best-effort path returns `status: "ok"` with an error string. One-shot jobs can then be recorded successful or deleted even though no message was delivered.
  - confidence: 0.82
