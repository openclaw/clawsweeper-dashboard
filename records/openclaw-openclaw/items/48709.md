- `impact:message-loss`: The reported failures include replies being suppressed or dropped after Gemini bloat, tagged reasoning, or mixed text/tool turns.

## Summary

Keep open. Current main still stores Gemini visible text signatures, replays them as Gemini thought signatures on same-route history, and keeps Google/Gemini wired to tagged reasoning while native thinking is also enabled; the mixed text-before-tool path has partial flush coverage, but the central session-state/message-loss problem remains.

## What This Changes

The issue reports Gemini 2.5 Pro session failures from replayed textSignature metadata, embedded think/final tag handling, and mixed text plus tool-call delivery.

## Best Possible Solution

Fix this at the Google provider and replay boundary: preserve required Gemini function-call/native-thinking signatures, bound or avoid replaying stale visible-text signatures, and coordinate the tagged-versus-native reasoning decision with the open Gemini reasoning issue.

## Reproduction Assessment

Yes, at source level: current main deterministically stores and replays same-route Gemini visible-text signatures, keeps Google/Gemini on tagged reasoning, and enforces final-tag suppression. I did not run the paid live Gemini plus Telegram loop.

## Solution Assessment

Unclear for automation. The issue direction is valid, but maintainers need to decide which Gemini signatures are required to replay and whether Google/Gemini should continue using tagged reasoning when native thinking payloads are enabled.
