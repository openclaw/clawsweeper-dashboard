## Summary

Keep open. Current main still has the central Gemini signature-bloat path: visible Google text blocks can retain `textSignature`, same-provider replay sends that back as Gemini `thoughtSignature`, and the sanitizer only strips `thought_signature` / `thoughtSignature`; the mixed text-before-tool path has partial flush protection, while the tagged/native reasoning conflict is still separately open at https://github.com/openclaw/openclaw/issues/69220.

## What This Changes

The issue requests fixing Gemini session failures caused by persisted/replayed text signatures, tagged-thinking/final-tag suppression, and text dropped around tool calls.

## Best Possible Solution

Handle this at the Google provider and replay boundary: preserve only required Gemini continuation signatures, bound or avoid replaying stale visible-text signatures, and align Gemini reasoning delivery with native thinking so final-less tagged output cannot silently suppress replies.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: current Google streaming can store visible text signatures, same-provider replay sends them back as `thoughtSignature`, and final-tag enforcement suppresses untagged Gemini text; the live Gemini/Telegram loop was not run.

## Solution Assessment

No current-main solution is complete. The text-before-tool flush path is partly protected, but signature retention/replay and Gemini tagged-versus-native reasoning need a provider-owner decision before a narrow implementation is safe.
