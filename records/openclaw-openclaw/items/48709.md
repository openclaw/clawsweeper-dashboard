## Summary

Keep open. Current main still has the unresolved Gemini signature bloat and tagged-reasoning delivery paths: visible Google text parts can retain `thoughtSignature` as OpenClaw `textSignature`, same-provider replay sends that value back as Gemini `thoughtSignature`, and the generic sanitizer does not remove `textSignature`. The mixed text-before-tool path now has flush coverage, so that portion appears partially addressed, but the signature retention/replay and Gemini tagged-versus-native reasoning behavior still need maintainer/provider-owner judgment.

## What This Changes

The issue requests fixes for Gemini 2.5 Pro session failures caused by large `textSignature` replay, tagged `<think>`/`<final>` handling, and mixed text-plus-tool turns dropping user-visible text.

## Best Possible Solution

Handle this at the Google provider and replay boundary: preserve only signatures required for valid Gemini continuation, avoid replaying stale visible-text signatures indefinitely, and move Gemini reasoning delivery away from strict tagged final output or add a provider-specific fallback that cannot silently suppress final-less visible replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a same-provider Google assistant text block with `textSignature` is serialized back to Gemini as `thoughtSignature`, while final-tag enforcement suppresses visible text that never enters a `<final>` block. The live Gemini/Telegram loop was not run in this read-only review.

## Solution Assessment

No current-main solution is complete. The text-before-tool flush path is partially covered, but the remaining signature persistence/replay and Gemini tagged-versus-native reasoning decisions need a provider-owner design before implementation.
