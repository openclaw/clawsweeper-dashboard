## Summary

Keep open. Current main and the latest v2026.5.22 release still retain and replay valid same-route Gemini visible-text signatures, still combine Google/Gemini tagged reasoning with native Gemini thinking payloads, and only partially mitigate the mixed text-before-tool path through block flushing.

## What This Changes

The issue asks OpenClaw to stop Gemini textSignature session bloat, avoid suppressing Gemini `<think>` text, and preserve user-visible text when Gemini returns text plus tool calls.

## Best Possible Solution

Fix the Google provider and replay boundary to bound or avoid replaying stale visible-text signatures while preserving required function-call/native-thinking signatures, then coordinate the Google native-thinking reasoning-mode fix with the open Gemini reasoning issue.

## Reproduction Assessment

Yes, at source level: current main deterministically stores and replays same-route Gemini text signatures, enables native Gemini thinking payloads, keeps Google/Gemini on tagged reasoning, and suppresses final-less tagged output. I did not run the paid live Gemini plus Telegram loop.

## Solution Assessment

Unclear for automation. The requested direction is valid, but maintainers need to decide which Gemini signatures must be retained or replayed and how Google/Gemini should handle tagged reasoning when native thinking is enabled.
