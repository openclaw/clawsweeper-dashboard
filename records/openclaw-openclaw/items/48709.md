## Summary

Keep open. Current main and the latest release still preserve/replay Gemini `textSignature` data and still combine Google tagged reasoning with native Gemini thinking; https://github.com/openclaw/openclaw/issues/69220 covers part of the tagged/native failure, but no canonical item replaces the separate signature-bloat path.

## What This Changes

The issue reports Gemini 2.5 Pro session failures from persisted/replayed text signatures, embedded `<think>` text, and mixed text plus tool-call delivery behavior.

## Best Possible Solution

Handle this at the Google provider and replay boundary: preserve required Gemini function-call/native-thinking signatures, bound or avoid stale visible-text signature replay, and align Gemini reasoning delivery with native thinking while coordinating the tagged/native and inter-tool-delivery trackers.

## Reproduction Assessment

Yes, at source level: current main deterministically stores and replays Gemini visible text signatures and still combines Google tagged reasoning with native Gemini thinking payloads. No fresh live 5-10 turn Telegram/Gemini run was performed, so the full timeout/silent-delivery symptom remains user-reported rather than locally reproduced.

## Solution Assessment

Unclear for automation. The right fix is provider-owned replay and reasoning-mode policy; closing as duplicate or stripping all signatures would be risky because Google requires some thought signatures for function-call replay.
