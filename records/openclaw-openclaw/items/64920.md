## Summary

Keep open. The Feishu topic-lane bug is valid and the PR contains useful direction and tests, but the current patch is not mergeable as-is because it still excludes native `topic_group` events from the new lane path and diverges from current main's topic-group thread-id session semantics; the unrelated cleanup changes also make a narrow replacement/update safer.

## What This Changes

The PR adds Feishu topic-aware sequential queue keys and tests, extracts group-session scope resolution, passes the scope into message dispatch, updates the changelog, and includes unrelated CI/type/lint cleanup outside Feishu.

## Best Possible Solution

Land a narrow Feishu-only implementation on current main that derives queue lanes from the same group-chat classification, effective group-session scope, and topic-id semantics as `resolveFeishuGroupSession`, while preserving chat-wide `/stop` and `/btw` escape lanes and splitting unrelated cleanup into separate PRs.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: two current-main Feishu topic events in the same chat still resolve to the same base sequential key, and `createSequentialQueue()` serializes work by that key.

## Solution Assessment

No. The PR direction is right, but the current patch is not the best merge candidate because it misses `topic_group` and does not mirror current session routing; sharing or exactly mirroring the existing group/session resolver is the safer fix.

## Review Findings

Full review comments:

- **[P2] Route topic_group events through topic lanes:** `extensions/feishu/src/monitor.account.ts:421`
  - body: `topic_group` is a Feishu group chat type on current main, but this gate only resolves topic scope for `chat_type === "group"`. Native topic-group messages therefore pass no topic scope into `getFeishuSequentialKey()` and keep the old chat-wide queue, so the central bug remains. Include `topic_group`, preferably through the existing group-chat helper.
  - confidence: 0.93
- **[P2] Match topic_group thread-id precedence:** `extensions/feishu/src/sequential-key.ts:74`
  - body: Current session routing treats `topic_group` specially: when both fields exist, `thread_id` is canonical so the starter and replies share one topic peer. This helper uses `root_id ?? thread_id`, so once `topic_group` is admitted, replies with both fields can queue on a different lane from the session key and interleave shared-session history. Share or mirror the session resolver's topic-id logic.
  - confidence: 0.88
