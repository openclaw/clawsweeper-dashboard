## Summary

Keep open. The underlying Feishu topic-group queueing bug is still present on current main, but this PR should not merge as-is because the proposed branch misses native `topic_group` routing and uses topic-id precedence that diverges from current session routing; it also carries unrelated cleanup outside the Feishu surface.

## What This Changes

The PR adds Feishu topic-aware sequential queue keys and tests, extracts group-session scope resolution, passes the scope into message dispatch, updates the changelog, and includes several unrelated CI/type/lint cleanup changes outside Feishu.

## Best Possible Solution

Land a narrow Feishu-only implementation that derives the sequential queue lane from the same group-chat classification, effective group-session scope, and topic-id semantics as `resolveFeishuGroupSession`, while preserving chat-wide `/stop` and `/btw` escape lanes and keeping unrelated cleanup in separate PRs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: construct two Feishu `topic_group` events with the same `chat_id` and different topic metadata, and current main still returns the same base sequential key while `createSequentialQueue()` serializes by that key.

## Solution Assessment

No. The PR direction is right, but the current patch is not the best merge candidate because it excludes `topic_group` from the new path and does not mirror current session-routing precedence; sharing or matching the existing Feishu group/session resolver is the safer boundary.

- **[P2] Route topic_group through the topic-lane path:** `extensions/feishu/src/monitor.account.ts:421`
  - body: `topic_group` is a Feishu group chat type on current main, but this gate only resolves topic scope for `chat_type === "group"`.   - confidence: 0.93

  - body: Current session routing uses `thread_id ?? root_id` for `topic_group` events so a topic starter and its replies share one topic peer. This helper uses root-first precedence; after `topic_group` is admitted, replies with both fields can queue on a different lane from the session key and interleave shared-session history. Match the session resolver or share a helper.
  - confidence: 0.88
