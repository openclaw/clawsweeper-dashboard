## Summary

Keep open: source inspection confirms the Feishu queue key still collapses topic metadata on current main, but this PR head is not merge-ready because it misses native `topic_group` routing, can diverge from thread-first session keys, and lacks real Feishu behavior proof.

## What This Changes

The PR adds Feishu topic-scoped sequential queue keys and regression tests, and also carries unrelated CI-unblock changes across channel/provider/gateway/SDK files.

## Best Possible Solution

Land a narrowed Feishu-only current-main fix that routes native `topic_group` messages onto topic-aware queue lanes, matches thread-first session routing, preserves chat-wide `/stop` and `/btw`, and includes focused tests plus real Feishu proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `getFeishuSequentialKey()` returns the chat-wide base key after `/stop` and `/btw`, and the receive handler enqueues work by that key, so topic messages sharing a chat can serialize.

## Solution Assessment

No. The direction is right, but this PR gates topic lanes on `chat_type === "group"` and uses root-first lane ids while current main treats `topic_group` as a group and uses `thread_id` first for native topic sessions.
