## Summary

Keep open: current main still has the reported Feishu topic-lane queue problem, but this PR is not a clean or correct merge candidate yet because its topic-lane gate misses native `topic_group` events and its lane key precedence diverges from current topic-session routing.

## What This Changes

The PR adds Feishu topic-aware sequential-key routing and tests, plus unrelated cleanup across Mattermost, OpenAI, Telegram, channel registry, gateway tests, and plugin SDK typing.

## Best Possible Solution

Land a narrow Feishu-only current-main fix that routes native `topic_group` messages onto topic-aware queue lanes, uses the same topic key precedence as session routing, preserves chat-wide `/stop` and `/btw`, and adds focused regression coverage plus real Feishu proof.

## Reproduction Assessment

Yes. Source inspection on current main shows topic metadata is ignored by `getFeishuSequentialKey()`, and the receive handler serializes messages that share that chat-wide key through `createSequentialQueue()`.

## Solution Assessment

No. The direction is right, but this PR is not the best current fix until it is narrowed/rebased to the current receive-handler path, admits native `topic_group`, matches thread-first native topic semantics, and includes real Feishu behavior proof.
