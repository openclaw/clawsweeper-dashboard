## Summary

Keep open: the reported Feishu topic-queue bug is valid and the PR direction is useful, but the current head still misses native `topic_group` routing, diverges from current thread-id session semantics, and carries unrelated cleanup across non-Feishu surfaces.

## What This Changes

The PR adds Feishu topic-aware sequential queue keys, tests, and a changelog entry, plus unrelated CI/type/lint cleanup in Mattermost, OpenAI, Telegram, channels, gateway, and plugin-sdk files.

## Best Possible Solution

Land a narrow Feishu-only current-main fix that passes effective group scope and chat type into sequential-key resolution, treats `topic_group` as a group, and uses `thread_id` as the native topic-group lane key while preserving chat-wide `/stop` and `/btw`.

## Reproduction Assessment

Yes. Source inspection on current main shows Feishu topic events still resolve to the chat-wide base key, and `createSequentialQueue()` serializes messages that share that key.

## Solution Assessment

No. The direction is right, but this PR is not the best merge candidate until it is narrowed to Feishu, rebased onto the current receive-handler path, and aligned with native `topic_group` thread-id semantics.
