## Summary

Keep this PR open: it is maintainer-labeled, currently dirty against main, and the descriptor-cache path introduces a concrete regression for context-gated plugin tool factories.

## What This Changes

The PR replaces per-context plugin tool factory-result caching with cached descriptors captured from `api.registerTool(...)`, lazy runtime execution, updated plugin/tool docs, tests, and a changelog entry.

## Best Possible Solution

Keep the descriptor-cache optimization only where cached descriptors cannot bypass context-dependent factory visibility, then rebase onto current main and add regression coverage for sandboxed/context-changed plugin factories.

## Reproduction Assessment

Yes. A source-level reproduction is deterministic: populate the descriptor cache from a non-sandboxed allowlisted Lobster planning pass, then resolve tools for a sandboxed context; the cached path returns a wrapper even though the live Lobster factory returns null for `ctx.sandboxed`.

## Solution Assessment

No as-is. The direction is promising for hot-path planning, but the cache must either preserve context-sensitive visibility semantics or fall back to runtime discovery for factories that are not proven descriptor-static.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep context-gated factories on the runtime path:** `src/plugins/tools.ts:568-574`
  - body: After a non-sandboxed planning pass caches a descriptor, this path can expose that cached tool in later sandboxed contexts without rerunning the factory first. The bundled Lobster plugin intentionally returns null when `ctx.sandboxed`, and current main has regression coverage for that behavior, so sandboxed runs can now see a tool that execution will reject as missing. Keep context-sensitive factories on the runtime path or make the descriptor cache account for the same context gates.
  - confidence: 0.92
