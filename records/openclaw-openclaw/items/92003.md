## Summary

Keep open: current main and v2026.6.6 still lack the WebChat Stop recovery behavior, and this active linked PR is the implementation path. The latest head addresses the maintainer-reported stale-run race with a shared startedAt generation guard, so I found no blocking line-level defect, but this remains a session-state merge-risk decision for maintainers.

## What This Changes

The PR persists successful WebChat `chat.abort` as terminal `killed` lifecycle state, locally reconciles confirmed Stop responses, guards stale same-session lifecycle writes, and adds UI/Gateway regressions.

## Best Possible Solution

Land this guarded client/Gateway lifecycle fix after session-state maintainer review accepts the shared startedAt generation guard, or replace it with a single-owner lifecycle persistence design that preserves the same Stop recovery behavior.

## Reproduction Assessment

Yes at source level: current main sends `chat.abort` without confirmed local run/stream reconciliation, and canonical lifecycle persistence lacks the PR's same-session run-generation stale guard. I did not run a fresh browser reproduction in this read-only review.

## Solution Assessment

Yes, this is the best current fix shape I found: reconcile the WebChat client only for confirmed run ids and guard stale same-session writes in canonical lifecycle persistence as well as direct abort persistence. A larger single-owner lifecycle refactor is possible later but not required for this bug fix.
