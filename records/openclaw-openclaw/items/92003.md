## Summary

This PR addresses a real P1 WebChat recovery bug, but it is not merge-ready: both the gateway persistence and client reconciliation can terminalize a newer run after an older abort races with it, and the supplied evidence is test-only rather than real WebChat behavior proof.

## What This Changes

This PR persists successful WebChat aborts as killed session lifecycle state and immediately reconciles local WebChat run, stream, tool, and session-row state after the abort RPC returns.

## Best Possible Solution

Keep gateway session state authoritative, derive abort terminal fields through the canonical lifecycle projection, reject writes for a newer same-session run, and reconcile WebChat locally only when the RPC confirms the captured run was aborted and it is still the active local run.

## Reproduction Assessment

Yes. The linked Stop-then-send path is concrete, and source inspection provides deterministic current-patch races by advancing the same session or local `chatRunId` before the old abort operation completes.

## Solution Assessment

No. Persisting terminal state and locally reconciling WebChat are the right boundaries, but both operations need run-identity/generation guards and the client must honor the RPC result.

## Review Findings

Full review comments:

- **[P1] Do not overwrite a newer run with the old abort snapshot:** `src/gateway/server-methods/chat.ts:1852-1854`
  - body: `sessionId` is reused across turns, so it is not a sufficient generation guard. If a new turn updates this row while the async abort persistence is waiting, this callback still replaces that newer active state with the old run's `killed` status and timestamps. Reject rows with a newer `startedAt` (or otherwise match the run generation) and add a deterministic race test.
  - confidence: 0.96
- **[P1] Only reconcile the run the server actually aborted:** `ui/src/ui/controllers/chat.ts:1250`
  - body: This discards the `chat.abort` response and always clears local state after any successful RPC. The gateway legitimately returns `aborted: false`, and `state.chatRunId` can also change while awaiting the request, so a stale Stop response can clear a newer run or terminalize a run that was never aborted. Check that the captured run appears in `runIds`/`aborted` and is still current before reconciling.
  - confidence: 0.98
