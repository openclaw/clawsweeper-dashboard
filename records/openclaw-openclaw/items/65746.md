## Summary

Keep this PR open: the PDF custom-provider bug still exists on current main, but the submitted helper drops existing registry lookup semantics and needs normalized inline-provider matching plus the required changelog entry before merge.

## What This Changes

The PR adds a PDF-only fallback from runtime registry lookup to `cfg.models.providers` inline custom provider models and adds a regression test for config-defined custom providers.

## Best Possible Solution

Land a config-aware PDF resolver that preserves the existing normalized registry lookup first, then falls back to `models.providers` with the same exact, normalized-provider, and provider-scoped model matching semantics used by the embedded runner, with focused coverage and release notes.

## Reproduction Assessment

Yes. Configure a `models.providers.local-openai` model, set `agents.defaults.pdfModel.primary` or pass `model: local-openai/gpt-5.4-mini`, and invoke the `pdf` tool; current main still reaches the registry-only resolver, and #65742 includes the matching 2026.4.10 runtime `Unknown model` failure.

## Solution Assessment

No, not as written. The PDF-only direction is narrow enough, but the helper should reuse current registry resolver semantics and mirror embedded-runner inline matching before it is the maintainable fix.
