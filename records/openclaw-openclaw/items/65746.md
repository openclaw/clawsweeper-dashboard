## Summary

Keep this PR open, but it still needs repair before merge. Current main has not implemented config-aware PDF model execution, and the PR addresses a real custom-provider gap, but the proposed helper drops existing normalized registry lookup behavior and only matches inline providers by exact ids.

## What This Changes

The PR adds a PDF-only fallback from runtime registry lookup to `models.providers` inline custom provider models and adds a regression test for config-defined custom provider model resolution.

## Best Possible Solution

Land a config-aware PDF resolver that preserves the existing normalized registry lookup first, then falls back to `models.providers` using the same normalized provider and provider-scoped model matching semantics as the embedded runner, with focused coverage for local custom providers, provider aliases, and provider-prefixed model ids.

## Reproduction Assessment

Yes. The source-level reproduction is a PDF model override/default for a custom provider in `models.providers` with a registry miss; current main still reaches `resolveModelFromRegistry`, and the issue/PR discussion includes real runtime verification of the `Unknown model` failure on 2026.4.10.

## Solution Assessment

No, not as written. The narrow PDF-only direction is reasonable, but the helper should reuse the current resolver semantics for registry lookup and mirror embedded-runner inline matching before it is merge-ready.
