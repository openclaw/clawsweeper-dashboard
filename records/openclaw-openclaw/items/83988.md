## Summary

Keep open: the direction is plausible, but this head is not merge-ready because it still does not prove or fully prevent live Telegram text-before-voice delivery and it puts caption capability policy in core; the supplied proof is unit/mock only.

## What This Changes

This PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram, attach accumulated text to the synthesized voice payload, preserve audio-only supplements for non-caption channels, and add dispatch tests.

## Best Possible Solution

Move captioned-voice support into a channel-owned capability/helper, hold or suppress the live text queue for that capability until TTS succeeds or fails, preserve non-caption channel behavior, and verify the Telegram path with redacted live proof.

## Reproduction Assessment

Yes at source level: current main settles visible text before final TTS, and the live ACP projector queues block text before finalization. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The proposed direction is plausible, but this diff does not hold already queued live text and puts caption capability policy in core; a channel-owned capability with queue suppression/fallback proof is safer.
