## Summary

This PR is a plausible fix direction, but it is not merge-ready: the live Telegram path can still queue visible block text before the captioned voice final, and the caption-capable decision is hard-coded in core instead of coming from the channel contract. The supplied proof is unit-test output only, while the matching Telegram review note calls for real Telegram proof for transport-visible behavior.

## What This Changes

The PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram-style captioned voice sends, include text on the synthesized voice payload, add failure fallback settlement, and extend dispatch tests.

## Best Possible Solution

Hold or suppress visible block delivery before captioned final-mode TTS, expose captioned-voice support through a channel-owned capability or helper, and land only after redacted Telegram proof covers success and fallback paths.

## Reproduction Assessment

Yes at source level: current main flushes ACP blocks before finalization and settles visible text before TTS synthesis. I did not run a live Telegram TTS session in this read-only review.

## Solution Assessment

No. Deferring `settleVisibleText()` inside finalization is not enough because the block payload can already be queued, and the caption decision should come from the channel contract rather than a core allowlist.

- **[P1] Hold queued block text before captioned TTS:** `src/auto-reply/reply/dispatch-acp.ts:244-245`

  - confidence: 0.87
