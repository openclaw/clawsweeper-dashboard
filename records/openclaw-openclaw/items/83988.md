## Summary

Keep open: this PR targets a real, source-reproducible Telegram final-mode TTS churn bug, but the patch still does not stop live ACP block previews from being sent before final TTS and it lacks required real Telegram proof.

## What This Changes

The PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram, send accumulated text with the synthesized voice payload, and add dispatcher unit tests.

## Best Possible Solution

Coordinate ACP projector/delivery with a channel-owned caption-capable voice contract so Telegram final-mode TTS withholds or replaces live text previews, sends one captioned voice note on success, and flushes text on failure.

## Reproduction Assessment

Yes at source level. Current main settles visible ACP text before final TTS, and live ACP projection can send block replies before finalization; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Deferring `settleVisibleText()` is directionally useful, but it is not sufficient because live block replies can already be sent; the fix needs to control that preview path and use channel-owned capability metadata instead of a core allowlist.
