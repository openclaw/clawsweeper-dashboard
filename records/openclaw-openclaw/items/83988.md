## Summary

Keep open. The PR targets a valid linked TTS/Telegram churn bug, but the current patch still leaves already queued visible ACP text ahead of the captioned voice payload, keeps caption support as a core Telegram allowlist, and has only mocked/unit proof rather than live Telegram behavior proof.

## What This Changes

The PR defers visible-text settlement for final-mode ACP TTS on Telegram, sends accumulated text on the TTS media payload, preserves audio-only supplements for non-caption channels, and adds dispatch tests.

## Best Possible Solution

Hold or replace the queued visible text until captioned TTS succeeds, fall back to the normal text path on failure, and source captioned-voice support from a channel-owned capability/helper rather than a core allowlist.

## Reproduction Assessment

Yes at source level: current main settles visible text before final TTS synthesis, and the dispatcher/coordinator code shows queued visible text is not held by skipping settlement. I did not run a live Telegram TTS session in this read-only review.

## Solution Assessment

No. Deferring settlement alone is not the best fix because it does not prevent already queued text from preceding the voice payload, and captioned voice support should be channel-owned rather than hard-coded in core.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Hold queued text before captioned TTS:** `src/auto-reply/reply/dispatch-acp.ts:244-245`
  - body: This skips `settleVisibleText()` for Telegram, but by the time finalization runs the projector has already queued visible block/final text through `delivery.deliver()`. Since settlement only waits for that queue, the captioned voice final is still queued after the text, so the live/final-only path can keep the preceding text bubble this PR is meant to remove. The fix needs to hold or replace that queued visible text before queueing the voice payload, with a regression test that asserts no prior block/final text send in the deferred path.
  - confidence: 0.89
- **[P2] Move captioned voice support to channel capabilities:** `src/auto-reply/reply/dispatch-acp.ts:241-243`
  - body: `CAPTIONED_VOICE_CHANNELS` puts a Telegram-specific transport capability in core ACP dispatch even though channel TTS/media behavior already belongs to channel/plugin contracts. This will drift as channels gain or lose captioned voice support; expose a channel-owned flag/helper and consume that instead of a core allowlist.
  - confidence: 0.78
