## Summary

This PR should stay open: it addresses a real source-reproducible Telegram final-mode TTS bug, but the current patch still changes non-deferred final TTS payload semantics and lacks required real Telegram proof.

## What This Changes

The PR defers ACP visible-text settlement for Telegram final-mode TTS, sends accumulated text with the synthesized voice payload, and adds dispatch-acp tests for ordering and fallback behavior.

## Best Possible Solution

Keep Telegram settlement deferral, but build the final TTS payload conditionally: include caption text only when text settlement was deferred, and keep audio-only `visibleTextAlreadyDelivered` supplements for non-deferred channels.

## Reproduction Assessment

Yes, from source inspection: current main settles ACP visible text before final-mode TTS synthesis, so Telegram can display text before a later voice payload. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No: the PR fixes the broad settlement deferral, but still attaches visible text to final TTS media for non-deferred channels. The safer solution is conditional payload construction plus Telegram live proof.
