## Summary

Keep open: the PR targets the right TTS churn bug, but the fix still acts after ACP visible text has already been flushed/queued, so Telegram can still see text before the captioned voice note. The supplied proof is unit-test output only and does not satisfy the live Telegram proof gate for this visible transport change.

## What This Changes

The PR defers ACP final-mode TTS visible-text settlement for caption-capable voice sends, includes accumulated text as the TTS caption, tracks final delivery failures, and adds dispatch tests.

## Best Possible Solution

Hold caption-eligible ACP visible output until the final TTS decision, deliver one Telegram voice note with caption on success, preserve text fallback on synthesis or send failure, and prove the real Telegram flow before merge.

## Reproduction Assessment

Yes. Source inspection shows `projector.flush(true)` can send or queue visible text before `finalizeAcpTurnOutput()` runs, while the PR only defers settlement inside finalization; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Deferring settlement alone is too late; the maintainable fix needs to hold or redirect visible projector output before it is enqueued, then make caption/fallback behavior explicit and proven.
