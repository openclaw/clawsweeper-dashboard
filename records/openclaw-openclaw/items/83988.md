## Summary

Keep open: the PR targets the right Telegram final-mode TTS churn path, but current head still cannot detect queued captioned `sendVoice` failures and the supplied proof is mocked unit-test output rather than real Telegram behavior.

## What This Changes

The branch defers ACP visible-text settlement for final-mode TTS, adds a captioned-final-text channel capability used by Telegram, suppresses live block delivery for that path, and adds focused tests.

## Best Possible Solution

Keep the caption-capability direction, but track queued final media failures independently of visible-text settlement, cover it with a Telegram-final visibility regression test, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the patch defect: a queued Telegram captioned final voice send can fail after `sendFinalReply()` returns true, but settlement returns before reading `getFailedCounts().final`. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet; the proposed settlement deferral is the right shape for the original churn, but final delivery failure tracking must be independent of Telegram's final-text visibility override and the PR still needs real Telegram proof.
