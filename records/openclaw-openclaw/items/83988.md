## Summary

Keep this PR open but not merge-ready: it addresses a real Telegram final-mode TTS churn report, but the patch still lets live ACP block text queue before the captioned voice payload and the supplied proof is mocked unit output rather than real Telegram behavior.

## What This Changes

The PR changes ACP final-mode TTS ordering to defer visible-text settlement for Telegram, attach visible text to the synthesized voice payload, preserve audio-only supplements for other channels, and add dispatch tests.

## Best Possible Solution

Make captioned-voice support channel-owned, hold or suppress live visible blocks for that final-mode path until the TTS outcome is known, and preserve text-only fallback when synthesis fails.

## Reproduction Assessment

Yes, at source level: current main settles visible text before final TTS, and the live ACP path queues block replies before finalization. I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. Deferring `settleVisibleText()` alone is not enough; the maintainable fix needs channel-owned captioned-voice support plus buffering or suppression of live blocks until TTS success or fallback is known.
