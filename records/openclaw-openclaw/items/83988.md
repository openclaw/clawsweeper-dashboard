## Summary

Keep this PR open: the central Telegram final-mode TTS fix is still useful and not on current main, but the branch still suppresses partial/live streaming for plain `auto: "tagged"` replies before it knows whether a TTS directive exists.

## What This Changes

The PR adds a captioned final-TTS channel capability, enables it for Telegram, changes ACP and generic reply dispatch to defer/suppress pre-audio text, adds directive-caption handling, tests, and docs.

## Best Possible Solution

Land this after preserving live/partial text for plain tagged replies, keeping directive-bearing tagged replies caption-safe, and making the captioned final-TTS compatibility tradeoff an explicit maintainer decision.

## Reproduction Assessment

Yes. From source, a Telegram caption-capable session with `messages.tts.auto: "tagged"` and `mode: "final"` sets `canCaptionFinalTts` before reply content is known, so plain replies lose `onPartialReply`/live ACP flushing even without directives.

## Solution Assessment

No, not yet. The capability-based captioned TTS direction is plausible, but the implementation needs a narrower tagged-mode gate so only directive-bearing replies suppress live text while plain tagged replies keep the existing stream path.
