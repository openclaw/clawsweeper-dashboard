## Summary

Keep open: the PR addresses part of the current text-before-final-TTS ordering, but it still does not stop live ACP block text from being queued before the final Telegram voice note, and it lacks real Telegram proof.

## What This Changes

The PR defers final-mode ACP TTS text settlement for Telegram, attaches accumulated text to the synthesized voice payload, and adds dispatcher unit tests.

## Best Possible Solution

Implement channel-owned captioned-voice behavior that suppresses or replaces the Telegram live text preview for final-mode TTS, preserves text fallback on synthesis or send failure, and proves the result in Telegram.

## Reproduction Assessment

Yes at source level: current main settles visible ACP text before final TTS, and live ACP projection can enqueue block text before finalization. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The patch captions the final media payload, but it only skips settlement and does not prevent the already-queued live Telegram block preview; the maintainable fix should handle the preview path and use channel-owned capability data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Defer the live text preview, not only settlement:** `src/auto-reply/reply/dispatch-acp.ts:245-247`
  - body: In live ACP mode, `projector.flush(true)` has already delivered accumulated text through `delivery.deliver("block")` before this final TTS branch runs. Skipping `settleVisibleText()` only stops waiting for those sends; because the dispatcher serializes block work before later final work, Telegram can still show the text preview followed by the voice note. The fix needs to suppress/defer/finalize that preview when final TTS will send a captioned voice payload.
  - confidence: 0.92
- **[P2] Move caption support out of ACP core:** `src/auto-reply/reply/dispatch-acp.ts:241-244`
  - body: This adds a Telegram allowlist in core ACP dispatch even though channel-specific delivery policy belongs in channel plugins or capability metadata. Keeping the decision here means future caption-capable channels will drift from core, and core now owns Telegram-specific behavior; expose or reuse a channel-owned captioned-voice capability instead.
  - confidence: 0.83
