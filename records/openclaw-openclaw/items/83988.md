## Summary

## What This Changes

The PR defers ACP final-mode TTS visible-text settlement for caption-capable voice delivery, sends accumulated text with the final TTS media payload, adds fallback handling, and expands dispatch-acp tests.

## Best Possible Solution

Land a narrow fix that derives captioned-voice support from a channel/plugin capability, tracks the final captioned voice send independently from earlier visible block failures, and proves the Telegram path live.

## Reproduction Assessment

Yes, at source level: current main settles visible text before final-mode TTS synthesis, and the PR’s ordering test captures that path. I did not run a live Telegram/TTS scenario in this read-only review.

## Solution Assessment

No. Deferring settlement is the right direction, but this implementation still needs captioned-final delivery accounting, a channel-owned capability boundary, and live Telegram proof before it is the maintainable fix.
