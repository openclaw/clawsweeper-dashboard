## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve tagged-mode streaming until a directive is present:** `src/auto-reply/reply/dispatch-from-config.ts:1431-1432`
  - body: This predicate treats every non-off caption-capable final TTS setup as something that will synthesize audio, but `auto: "tagged"` only runs TTS when the reply text contains a directive. For ordinary tagged-mode replies without `[[tts:...]]`, `maybeApplyTtsToPayload` returns the text payload unchanged, while this branch has already suppressed `onPartialReply` and block text, so Telegram users lose normal streaming and only see delayed fallback text. Please keep draft/block delivery enabled for no-directive tagged mode, and apply the same guard to the ACP predicate.
  - confidence: 0.9
- **[P1] Do not suppress ACP blocks for untagged tagged-mode replies:** `src/auto-reply/reply/dispatch-acp.ts:471-475`
  - body: The ACP gate has the same issue: `autoMode !== "off"` includes `tagged`, but the TTS runtime later skips audio when the accumulated text has no directive. That means caption-capable Telegram ACP turns in tagged mode suppress live block delivery even when no TTS should happen, changing normal replies into a silent wait plus final fallback. Gate this to modes that can be known upfront, or defer the decision until directives are known.
  - confidence: 0.88
- **[P2] Document the captioned-final TTS capability:** `src/channels/plugins/types.core.ts:314-322`
  - body: `captionedFinalText` is a new plugin-facing channel capability with real delivery consequences: core suppresses live text and relies on fallback when final voice delivery fails. The public TTS/channel SDK docs still describe streamed final-mode TTS as audio-only after text streaming and do not tell plugin authors when to opt into this field, so third-party channels could copy the type without understanding the compatibility risk.
  - confidence: 0.86
