## Summary

Keep open: the latest PR still does not stop the already-enqueued live ACP block text from being delivered before the final captioned voice payload, and contributor real-behavior proof is still missing.

## What This Changes

The PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram-style captioned voice delivery and adds dispatcher unit coverage for success, fallback, and non-deferred channels.

## Best Possible Solution

Use a channel-owned captioned-voice capability or helper to suppress or replace the live block preview for final-mode TTS, then deliver one captioned voice payload on success and flush text on synthesis or send failure.

## Reproduction Assessment

Yes, at source level: current main settles visible ACP text before final TTS, and both current main and this PR let live ACP projection enqueue block text before finalization. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The patch captions the final media payload, but it only skips settlement and does not prevent the already-queued live Telegram block preview; the maintainable fix should handle that preview path and use channel-owned capability data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Defer the live block payload too:** `src/auto-reply/reply/dispatch-acp.ts:247-249`
  - body: In live ACP mode, `projector.flush(true)` has already delivered accumulated text through `delivery.deliver("block")` before this final TTS branch runs. Skipping `settleVisibleText()` only stops waiting for those sends; because the dispatcher serializes block work before later final work, Telegram can still show the text preview followed by the voice note.
  - confidence: 0.92

  - confidence: 0.83
