## Summary

Keep open. The PR targets a real Telegram final-mode TTS churn bug, but it is not merge-ready because source inspection shows live ACP block text is still queued before the captioned voice payload, and no real Telegram proof has been provided.

## What This Changes

The PR defers visible-text settlement for caption-capable final-mode TTS, sends accumulated text with the final TTS media payload for Telegram, preserves audio-only supplements for other channels, and adds dispatcher tests.

## Best Possible Solution

Hold or suppress Telegram live block delivery when final-mode captioned TTS will be attempted, deliver one captioned voice payload on success, flush text only on TTS failure, and expose captioned-voice support through channel-owned capability rather than a core allowlist.

## Reproduction Assessment

Yes, at source level. ACP live output is flushed as block replies before finalization, and the dispatcher serializes those block replies ahead of the final voice payload; I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. Deferring waitForIdle alone is not the best fix because it does not stop already queued live text; the fix should gate or replace live block delivery for the captioned final TTS path and keep caption capability in the Telegram/channel layer.
