## Summary

Keep this PR open for maintainer review: current main still has the reported final-mode TTS text-before-audio ordering, and the patch looks like a plausible dispatch-layer fix, but it adds a public channel capability and changes Telegram final-mode TTS delivery defaults while tagged-mode live proof is still incomplete.

## What This Changes

Adds a `captionedFinalText` channel TTS capability, opts Telegram into it, and changes ACP/non-ACP final-mode TTS dispatch to defer or suppress live text so final voice notes can carry text as captions with fallback handling.

## Best Possible Solution

Land this only after maintainers accept the `captionedFinalText` SDK/default direction and tagged Telegram live proof confirms the directive-preview path, while keeping separate text-plus-voice delivery tracked in https://github.com/openclaw/openclaw/issues/42539.

## Reproduction Assessment

Yes, source-level: current main settles ACP visible text before accumulated final TTS synthesis and sends non-ACP block text before the accumulated final TTS branch. The PR also includes live before/after Telegram screenshots for always-mode behavior, but not for tagged mode.

## Solution Assessment

Yes for the core bug shape: capability-gated dispatch-layer deferral is cleaner than a Telegram-only delete/repost workaround. Merge readiness still depends on maintainer acceptance of the new public capability and tagged-mode live proof.
