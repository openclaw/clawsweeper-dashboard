## Summary

Keep this PR open: it targets a real ACP final-mode TTS message-ordering bug, but the current patch still lets live ACP block text queue before the captioned final voice note and the supplied proof is mocked unit-test output rather than a redacted live Telegram run.

## What This Changes

The branch defers ACP visible-text settlement for caption-capable final-mode TTS, sends accumulated text as a voice caption, tracks final delivery failures, and adds focused dispatch tests.

## Best Possible Solution

Land only after the patch prevents live block text from being queued before captioned final TTS, preserves text fallback on TTS/send failure, and includes redacted live Telegram proof; the channel-capability boundary should be accepted explicitly or moved out of ACP core.

## Reproduction Assessment

Yes for the source-level ordering problem, but not as a live Telegram reproduction. Current main settles visible text before final TTS, and source inspection shows live block replies are already queued before finalization.

## Solution Assessment

No. Deferring `settleVisibleText()` alone is not the best fix because live block delivery and dispatcher serialization can still put text before the final captioned voice note; the repair needs to gate that live block path or otherwise prove it cannot fire.
