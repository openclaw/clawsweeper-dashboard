## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate live block delivery before captioned final TTS:** `src/auto-reply/reply/dispatch-acp.ts:280-282`
  - body: The new deferral only skips `settleVisibleText()`, but live ACP output is already flushed by `projector.flush(true)` before `finalizeAcpTurnOutput()` runs. Since Telegram marks block text visible, a final-mode Telegram turn can still send a text bubble before the captioned voice note, leaving the reported churn or duplicate text in place. Please hold or reconcile live block delivery for the captioned-TTS path and test that no `sendBlockReply` precedes the captioned final.
  - confidence: 0.89
- **[P2] Move captioned voice eligibility to the channel capability:** `src/auto-reply/reply/dispatch-acp.ts:647`
  - body: This wires captioned voice support as `params.ttsChannel === "telegram"` in ACP core. The channel layer already owns outbound capabilities and Telegram-specific visibility decisions, so this creates a second source of truth and blocks other caption-capable transports from opting in without another core edit. Please expose a channel-owned capability or get explicit maintainer approval for this core exception.
  - confidence: 0.8
