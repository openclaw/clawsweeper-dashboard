## Summary

Keep this PR open, but the patch is not currently merge-ready: the capability-gated TTS deferral is the right layer for the reported Telegram churn, but the ACP deferred path can report `queuedFinal: false` after a successful captioned voice send, which can trigger downstream no-final-response handling.

## What This Changes

The PR adds a `captionedFinalText` channel capability and changes ACP/non-ACP final-mode TTS dispatch, Telegram capability metadata, docs, and regression tests so caption-capable channels can defer text into final voice-note captions.

## Best Possible Solution

Land this or an equivalent capability-gated final-mode TTS fix after preserving the successful final-delivery signal and getting maintainer approval for the public `captionedFinalText` contract, while keeping separate-delivery policy tracked in the canonical feature issue.

## Reproduction Assessment

Yes. Current main and v2026.6.9 still settle ACP visible text before final TTS, and the inspected Telegram screenshots show the before/after churn behavior; I did not run a fresh live Telegram session in this read-only review.

## Solution Assessment

No as submitted. The capability-gated dispatch-layer fix is the right shape, but the ACP `queuedFinal` regression must be fixed before maintainers decide whether to accept the public channel capability and Telegram delivery policy.
