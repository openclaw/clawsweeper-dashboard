## Summary

Keep this PR open: it addresses a real Telegram final-mode TTS churn bug, but the current branch still has a non-ACP delivery-accounting bug that can suppress the text fallback after failed final media delivery, and the live Telegram proof is not current-head complete for tagged mode or the later media/tool-result text-stripping changes.

## What This Changes

The PR adds a `captionedFinalText` TTS voice capability, enables it for Telegram, and changes ACP/non-ACP final TTS dispatch plus docs/tests to defer or suppress visible text until a captioned voice note can be sent.

## Best Possible Solution

Land only after final-media delivery accounting tracks actual successful delivery, current-head Telegram proof covers always/tagged/media/tool cases, and maintainers accept the `captionedFinalText` plugin capability contract.

## Reproduction Assessment

Yes, from source inspection: current main settles ACP visible text before final TTS and the non-ACP block path can send visible block text before final audio. I did not run live Telegram in this read-only review, but the PR screenshots also show the reported before-fix churn.

## Solution Assessment

No as-is. Capability-based captioning is a plausible owner-boundary fix, but the PR must track actual final media delivery before suppressing fallback text and needs current-head Telegram proof for the visible behavior.
