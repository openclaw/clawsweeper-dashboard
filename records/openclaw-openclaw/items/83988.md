## Summary

Keep open: the reported TTS churn is a real source-backed bug, but this PR still does not guarantee the visible Telegram behavior because live ACP block text is queued before final TTS, and the supplied proof is unit-test-only.

## What This Changes

The PR changes ACP final-mode TTS dispatch to defer visible-text settlement for caption-capable voice delivery, adds final-delivery failure tracking, and expands dispatch tests around settlement and fallback behavior.

## Best Possible Solution

Hold or suppress live ACP text projection when caption-capable final TTS owns the visible reply, move caption support to a channel/plugin-owned capability or explicitly approve the temporary core seam, then attach redacted live Telegram proof.

## Reproduction Assessment

Yes, from source inspection: live ACP text is flushed through the projector before final TTS dispatch, and current main also settles visible text before TTS synthesis. A live Telegram reproduction was not established in this read-only review.

## Solution Assessment

No. Deferring settlement inside finalizeAcpTurnOutput is too late for live ACP block delivery; the safer fix decides visible output ownership before the projector queues text and routes caption support through a channel-owned capability or an explicit maintainer-approved temporary seam.

Full review comments:

- **[P1] Gate live block delivery before captioned final TTS:** `src/auto-reply/reply/dispatch-acp.ts:280-282`
  - body: `shouldDeferTextForTts` is computed only after `projector.flush(true)` has already emitted live text through `deliver("block")`. In real dispatch, block and final replies are serialized, so Telegram can still receive the text bubble before the captioned voice note, and the new test does not assert that `sendBlockReply` is absent or reordered. Move the deferral decision earlier so final-mode captioned TTS owns the visible reply before live block text is queued.
  - confidence: 0.9
- **[P2] Move caption eligibility out of the core channel string check:** `src/auto-reply/reply/dispatch-acp.ts:647`
  - body: This wires captioned voice support with `params.ttsChannel === "telegram"` in ACP core. Voice-caption behavior is transport-specific, and keeping the policy here means other caption-capable channels cannot opt in without another core change while Telegram policy cannot live with the channel owner. Use a channel/plugin capability or get explicit maintainer approval for a documented temporary seam.
  - confidence: 0.82
