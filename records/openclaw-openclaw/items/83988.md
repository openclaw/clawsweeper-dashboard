## Summary

Keep open: the reported final-mode TTS churn is source-reproducible and the branch mostly targets the right dispatch path, but it still hardcodes Telegram caption support in ACP core and only supplies mock/unit-test proof for a visible Telegram delivery change.

## What This Changes

The branch defers ACP final-mode TTS visible-text settlement for caption-capable voice sends, includes accumulated text as the TTS caption, tracks final delivery failures, and adds dispatch tests.

## Best Possible Solution

Move captioned voice support into the channel TTS capability contract, keep non-caption channels on the existing eager-text/audio-only supplement path, and land only with redacted live Telegram proof for the captioned voice note and fallback path.

## Reproduction Assessment

Yes. Current main shows `settleVisibleText()` running before final-mode TTS synthesis in `finalizeAcpTurnOutput()`, and the PR's added tests encode that ordering failure, though no live Telegram reproduction was supplied.

## Solution Assessment

No. Deferring settlement for caption-capable final TTS is the right bug-fix direction, but the implementation should use a channel-owned caption capability instead of a hardcoded Telegram check in ACP core.
