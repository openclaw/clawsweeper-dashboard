## Summary

Keep this PR open: the deferral is aimed at a real source-reproducible Telegram TTS churn bug, but the branch still needs live Telegram proof and has merge-blocking delivery-accounting and channel-ownership concerns.

## What This Changes

The branch defers ACP final-mode TTS visible-text settlement for caption-capable Telegram delivery, sends accumulated text as the voice caption, and adds dispatch-acp tests for ordering and fallback paths.

## Best Possible Solution

Keep the PR open until caption support is channel-owned, final caption delivery accounting is separated from prior block failures, and redacted live Telegram proof shows one captioned final-mode voice note plus fallback behavior.

## Reproduction Assessment

Yes, at source level: current main settles visible text before final-mode TTS synthesis, and the PR’s mocked ordering test captures that path. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Deferring settlement is the right direction, but the implementation should track captioned-final delivery independently, move caption support to the channel/plugin contract, and include live Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.79

Full review comments:

- **[P2] Track captioned voice failure separately:** `src/auto-reply/reply/dispatch-acp.ts:286`
  - body: This check treats any visible delivery failure as proof that the captioned voice failed. Telegram’s visibility hook does not treat final caption text as visible, so an earlier block-send failure can flip `hasFailedVisibleTextDelivery()` even when the final `sendVoice` caption succeeded; that resets `finalMediaDelivered` and can send a redundant standalone final text. Track the captioned final attempt separately or compare final delivery failure state directly.
  - confidence: 0.81
- **[P2] Move caption capability out of ACP core:** `src/auto-reply/reply/dispatch-acp.ts:593`
  - body: The new `supportsCaptionedVoice` argument is still set with `params.ttsChannel === "telegram"` inside ACP core. Channel-specific caption policy belongs on the channel/plugin capability surface; otherwise future caption-capable channels or aliases need ACP-core edits and Telegram behavior can drift from the owner-owned transport contract.
  - confidence: 0.75
