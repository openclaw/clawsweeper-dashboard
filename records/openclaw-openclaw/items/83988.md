## Summary

Keep open: the PR targets a real Telegram final-mode TTS churn bug and uses a reasonable channel-capability seam, but the current patch suppresses all block payloads on captioned final TTS paths, which can drop media/non-text block replies, and the tagged-mode Telegram behavior still lacks live proof.

## What This Changes

The PR adds a `captionedFinalText` channel capability, opts Telegram into it, and changes ACP plus non-ACP final-mode TTS delivery to defer text and send voice notes with captions.

## Best Possible Solution

Keep the capability seam, but narrow suppression to deferred visible text only, preserve media/non-text block payload delivery on both ACP and non-ACP paths, document the channel plugin contract, and refresh tagged-mode Telegram proof before merge.

## Reproduction Assessment

Yes for the review finding: source inspection at PR head shows both dispatch paths returning before block media delivery when captioned final TTS is active. The original Telegram churn is also supported by the linked issue and before/after screenshots, though I did not run a live bot locally.

## Solution Assessment

No: the capability-based direction is reasonable, but this patch suppresses whole block payloads instead of only deferring visible text. The best fix is to preserve media/non-text block delivery and document the plugin contract before relying on the new capability.

Full review comments:

- **[P1] Preserve media blocks while deferring final TTS text:** `src/auto-reply/reply/dispatch-from-config.ts:2771-2772`
  - body: This early return suppresses every block payload when captioned final TTS is active. `hasOutboundReplyContent` includes media, and the existing dispatch path supports media-bearing block replies, so a Telegram reply that streams/generated a block with `mediaUrl` or `mediaUrls` would be dropped instead of delivered. Gate the suppression to text-only visible block delivery, or strip/defer the text while still routing the media/non-text payload.
  - confidence: 0.9
- **[P1] Do not drop ACP block media for captioned TTS:** `src/auto-reply/reply/dispatch-acp-delivery.ts:396-399`
  - body: `suppressBlockUserDelivery` returns before ACP block payloads reach the route/dispatcher path, regardless of whether the block contains media or other non-text content. That fixes duplicate text by discarding too much; preserve non-text block content and only defer the caption text that will be bundled into final TTS.
  - confidence: 0.88
- **[P2] Document the new channel capability in the plugin guide:** `src/channels/plugins/types.core.ts:323`
  - body: `captionedFinalText` is a public channel capability, but the PR only documents it in the TTS tool page. Channel plugin authors read `docs/plugins/sdk-channel-plugins.md` for capability contracts, so add the opt-in criteria, suppression behavior, and fallback expectations there before merging this plugin-facing API change.
