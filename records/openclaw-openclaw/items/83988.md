## Summary

Keep open: the patch starts final-mode TTS earlier and preserves audio-only supplements for non-caption channels, but source review still shows live Telegram ACP block text can be queued before the captioned voice payload, and no real Telegram proof was provided.

## What This Changes

This PR changes ACP final-mode TTS dispatch to defer settlement for Telegram, include accumulated text on the final voice payload, preserve audio-only supplements elsewhere, and add dispatcher tests.

## Best Possible Solution

Make the Telegram channel or adapter expose the captioned-voice capability, hold or suppress live block text for that final-mode path until the TTS outcome is known, and prove both one-captioned-voice success and text fallback in a live Telegram run.

## Reproduction Assessment

Yes at source level: current main settles visible text before final TTS, and the patched live path still flushes block replies before finalization. I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. Deferring settleVisibleText inside finalization is not enough because live block delivery already happened; the safer fix is a channel-owned captioned-voice capability plus suppression/holding of live text until TTS success or fallback is known.
