## Summary

Keep open: this is a focused fix for the linked Telegram final-mode TTS issue, but it is not merge-ready because live ACP block text can still be queued before the captioned voice note, caption support is hardcoded in core, and no real Telegram proof is present.

## What This Changes

This PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram, include accumulated text on the final voice-media payload, keep non-caption channels audio-only, and add dispatcher tests.

## Best Possible Solution

Hold or suppress Telegram live block delivery when captioned final TTS will be attempted, deliver one captioned voice payload on success, flush text on synthesis or delivery failure, and expose captioned-voice support through a channel-owned capability.

## Reproduction Assessment

Yes, at source level. Live ACP output can be flushed as block replies before finalization, and the PR defers waitForIdle without suppressing those queued block replies before the captioned final voice payload.

## Solution Assessment

No, not as-is. Deferring settlement alone does not stop already queued live text, and caption support should come from a channel-owned capability rather than a core Telegram allowlist.
