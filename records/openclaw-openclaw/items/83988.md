## Label Justifications

- `P2`: This is a normal-priority Telegram/TTS delivery bug fix with limited blast radius but real user-visible message-order impact.
- `merge-risk: 🚨 compatibility`: The PR changes final-mode TTS delivery semantics for existing Telegram setups and adds a core channel allowlist instead of a channel-owned contract.
- `merge-risk: 🚨 message-delivery`: The current patch can still misorder or duplicate Telegram text and voice delivery in live ACP mode.

## Summary

Keep this PR open but not merge-ready: it targets the right Telegram/TTS churn, but the patch only skips the settlement wait and does not stop live block text from being queued before final TTS, hard-codes Telegram caption policy in core, and still has only mocked/unit proof.

## What This Changes

The PR defers ACP visible-text settlement for Telegram final-mode TTS, includes visible text on the synthesized voice payload, preserves audio-only supplements elsewhere, and adds dispatch tests.

## Best Possible Solution

Make the captioned-voice capability channel-owned, hold or suppress live visible block delivery for that final-mode path until the TTS outcome is known, and prove both the single captioned voice success path and text fallback in Telegram.

## Reproduction Assessment

Yes at source level: current main settles visible text before final TTS, and the patched path still lets live block delivery queue text before finalization. I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. Deferring settleVisibleText alone is not enough; the maintainable fix needs a channel-owned captioned-voice signal and must buffer or suppress live block delivery until TTS success or fallback is known.
