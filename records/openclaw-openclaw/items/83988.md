## Summary

Keep open: the PR is a plausible boundary-aware fix for the Telegram final-mode TTS churn, but it currently has a non-ACP fallback bug that can lose deferred text on failed media delivery, and the current-head Telegram proof is still incomplete for the tagged/media/tool-result paths changed by the branch.

## What This Changes

The PR adds a `captionedFinalText` channel TTS capability, enables it for Telegram, and changes ACP plus non-ACP dispatch so final-mode TTS can send one captioned voice note instead of visible text followed by audio.

## Best Possible Solution

Fix the failed-media delivery tracking, then merge only after maintainers accept the plugin capability contract and current-head Telegram proof covers always-mode, tagged directive, and media/tool-result replies.

## Reproduction Assessment

Yes from source: current main settles visible ACP text before final TTS synthesis, and Telegram's voice send path can carry captions. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not as currently written. The capability-gated caption path is the right general boundary, but `deliveredFinalTtsMedia` must only be set after successful final media delivery and the current head still needs live Telegram proof for the visible paths changed.
