## Summary

Keep this PR open for maintainer review: the patch appears to fix the current final-mode Telegram TTS churn without a concrete line-level correctness blocker, but it adds public channel/plugin capability surface and changes message-delivery policy, so maintainer approval is still needed before merge.

## What This Changes

The PR defers final-mode TTS text on caption-capable channels, adds a `captionedFinalText` channel capability, marks Telegram as supporting it, and expands dispatch tests/docs around captioned final TTS fallback behavior.

## Best Possible Solution

Merge only after maintainers confirm `captionedFinalText` is the intended public channel API and keep configurable separate text-plus-voice delivery tracked separately.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.6.9 shows ACP visible text is settled before final TTS synthesis, and the generic dispatch path sends visible block text before accumulated final TTS audio.

## Solution Assessment

Likely yes for the reported bug: capability-gated deferral at the dispatch layer is cleaner than the older Telegram-side delete/combine approach, but it is still a public API and delivery-policy choice that needs maintainer approval.
