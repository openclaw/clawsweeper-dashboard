## Summary

Keep open: the PR addresses a real Telegram final-mode TTS churn bug, but the latest patch is not merge-ready because both dispatch paths can still count ordinary final media as delivered TTS media and skip the text fallback after suppressing live text.

## What This Changes

The PR adds a `captionedFinalText` channel TTS capability, enables it for Telegram, defers/suppresses final-mode TTS text delivery on ACP and non-ACP paths, and updates docs plus regression tests.

## Best Possible Solution

Gate captioned-final fallback suppression on actual TTS supplement media in both ACP and non-ACP paths, keep the capability opt-in documented, and require current-head Telegram proof for always, tagged directive, media preservation, and fallback behavior before merge.

## Reproduction Assessment

Yes. The original churn is source-reproducible from current main’s final-mode TTS ordering, and the new PR regression is source-reproducible because the code counts ordinary final media as `deliveredFinalTtsMedia` while the TTS runtime explicitly skips TTS for replies that already have media.

## Solution Assessment

No, not yet. The capability direction is reasonable, but the implementation needs to distinguish TTS supplement media from ordinary final media and needs current-head Telegram proof before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate delivered media on TTS supplement media:** `src/auto-reply/reply/dispatch-from-config.ts:2150`
  - body: In captioned-final mode the block text has already been withheld, but this line treats any delivered final media as delivered TTS media. `maybeApplyTtsToPayload` returns the original payload when the reply already has media, so a final image/file with no text can set `deliveredFinalTtsMedia` and skip the accumulated text fallback, dropping the text. Please count only real TTS supplement media here and mirror the same distinction in the ACP path.
  - confidence: 0.88
- **[P1] Only mark ACP TTS media for TTS supplements:** `src/auto-reply/reply/dispatch-acp-delivery.ts:437-438`
  - body: `hasFinalMedia` is true for any final payload media, but `finalizeAcpTurnOutput` uses `hasDeliveredFinalTtsMedia()` to decide whether suppressed block text still needs a fallback. If ACP emits ordinary final media after text blocks, this can mark the captioned voice path as satisfied even though no TTS caption carried the text. Gate this flag on the TTS supplement marker, not on generic media presence.
  - confidence: 0.84
