## Summary

Keep open: this targets a valid source-reproducible Telegram final-mode TTS delivery bug, but the branch still queues live block text before adding the same text as a voice caption, keeps Telegram caption policy in core, and supplies only mocked/unit proof.

## What This Changes

The PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram-style captioned voice delivery and adds dispatch tests for success, fallback, and non-final paths.

## Best Possible Solution

Make final-mode captioned voice an explicit channel-owned capability, hold or suppress pre-TTS visible text for that capability until the TTS outcome is known, and prove success plus fallback in a redacted Telegram run.

## Reproduction Assessment

Yes at source level: current main settles visible text before final TTS, and the live projector can queue block text before finalization. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Deferring only `settleVisibleText()` does not stop already queued live block delivery, and the Telegram allowlist belongs behind a channel-owned capability or helper.
