## Summary

Keep open: the PR targets a valid Telegram final-mode TTS churn bug, but the latest branch still has a blocking fallback gap for queued voice-send failures and only supplies mocked/unit proof for transport-visible Telegram behavior.

## What This Changes

The PR defers ACP final-mode TTS visible-text settlement for caption-capable voice delivery and adds dispatch-acp unit tests for settlement ordering, fallback, and non-final behavior.

## Best Possible Solution

Land a fallback-safe final-mode TTS fix that detects queued voice failures, sends the deferred text with `skipTts` when needed, and keys caption support off a channel-owned capability or helper with live Telegram proof.

Yes at source level: current main calls `settleVisibleText()` before final-mode TTS and the related issue plus PR tests exercise that ordering. I did not establish a live Telegram/TTS repro in this read-only review.

## Solution Assessment

No: the defer-and-caption direction is right, but the latest patch is not the best mergeable form until queued voice failures can fall back to text and caption support is owned by the channel contract.
