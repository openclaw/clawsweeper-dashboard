## Summary

Keep open: the PR targets a real Telegram final-mode TTS bug, but the current patch still misses the actual Telegram streamed-preview delivery path and lacks after-fix real behavior proof.

## What This Changes

The PR changes ACP final-mode TTS ordering to skip visible-text settlement for Telegram, attach accumulated text to the synthetic media payload, and add dispatcher unit tests.

## Best Possible Solution

Make deferred final-mode TTS an end-to-end Telegram delivery path: no prior finalized text preview, one captioned voice payload, and a reliable text fallback when synthesis or delivery fails.

## Reproduction Assessment

Yes, at source level: current main settles visible ACP text before final TTS, and the dispatcher/Telegram lane can send or finalize streamed text before late voice media. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Deferring only `settleVisibleText()` does not hold the already queued Telegram block delivery or preserve caption text through the TTS supplement media path; the fix needs to cover the channel delivery path as well.

## Review Findings

The branch can still produce a finalized text preview followed by an audio-only voice note, leaving the churn unresolved.
