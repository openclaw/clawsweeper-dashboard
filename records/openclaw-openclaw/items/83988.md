## Summary

Keep open: the Telegram direction is plausible, but the current head still changes non-deferred final-mode TTS payloads in a way that can duplicate visible text, and it lacks required live Telegram proof.

## What This Changes

The PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram, send accumulated reply text with the generated voice payload, and add dispatch-acp ordering/fallback tests.

## Best Possible Solution

Keep the Telegram deferral, but construct the final TTS payload conditionally: caption text only when settlement was deferred, and the existing audio-only `visibleTextAlreadyDelivered` supplement for all non-deferred channels.

## Reproduction Assessment

Yes, from source inspection: current main settles ACP visible text before final-mode TTS synthesis, so Telegram can display text before a later voice payload. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No: the latest branch fixes the earlier broad settlement deferral, but it still attaches visible text to final TTS media for non-deferred channels. The safer solution is conditional payload construction plus Telegram live proof.
