## Summary

Keep this PR open for maintainer review: it addresses the source-reproducible final-mode Telegram TTS churn and includes useful live screenshots, but enabling captioned final text by default changes Telegram delivery semantics while the separate text-plus-voice delivery request remains open.

## What This Changes

The PR adds a `captionedFinalText` channel TTS capability, enables it for Telegram, defers or suppresses final-mode TTS text delivery across ACP and generic dispatch, and updates tests, docs, and the plugin SDK API baseline.

## Best Possible Solution

Land a maintainer-approved TTS delivery policy: preserve existing separate/full-width text as the default until an explicit captioned mode is accepted, or intentionally accept the captioned default while reconciling the open separate-delivery request and keeping the plugin capability narrowly documented and tested.

## Solution Assessment

No, not as-is. The implementation is a plausible fix for the linked churn bug, but the best mergeable solution needs maintainer-approved delivery policy and SDK compatibility direction before changing Telegram's default to captioned voice notes.
