## Summary

Keep this PR open: the linked Telegram final-mode TTS churn is still present on current main, and the branch is a plausible fix path, but the new plugin-facing `captionedFinalText` capability remains too broad and current-head Telegram proof is still insufficient for a transport-visible delivery change.

## What This Changes

The branch adds a `captionedFinalText` channel TTS capability, enables it for Telegram, and rewires ACP and non-ACP final-mode TTS delivery to defer text until a captioned voice note or fallback text can be sent.

## Best Possible Solution

Constrain `captionedFinalText` to genuine caption-capable voice-note delivery in the SDK type or resolver, add an invalid opt-in regression test, and refresh redacted current-head Telegram proof for always mode, tagged directives, and media preservation before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows ACP visible text is settled before final TTS synthesis, matching the linked Telegram churn report; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not yet. The owner-boundary direction is plausible, but the new SDK capability still needs a type or resolver guard for caption-capable voice-note delivery plus current-head Telegram proof before this is the best mergeable solution.
