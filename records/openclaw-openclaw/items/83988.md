## Summary

Keep open: this is still a useful fix path for the linked Telegram final-mode TTS churn bug, but the current branch does not fully prevent live ACP text from being sent before the captioned voice note and still encodes Telegram caption policy in core.

## What This Changes

This PR changes ACP final-mode TTS dispatch to skip early visible-text settlement for Telegram, include accumulated text in the final TTS voice payload, and track final delivery failures with regression tests.

## Best Possible Solution

Suppress live block/draft visible text for caption-capable final-mode TTS through a channel-owned capability, preserve text fallback paths, and require exact-head Telegram proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: ACP live `text_delta` delivery calls `deliver("block")` and `sendBlockReply` before `finalizeAcpTurnOutput()` can synthesize the captioned voice note. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Deferring `settleVisibleText()` is part of the right fix, but the branch also needs to suppress live block text before final TTS and move caption eligibility behind a channel capability instead of a core Telegram check.
