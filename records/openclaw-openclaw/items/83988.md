## Summary

This PR is still useful and has strong Telegram proof for the main final-mode caption behavior, but it should not merge yet because the current suppression gate also catches `messages.tts.auto: "tagged"` turns before the final payload proves TTS is actually requested, and the new public channel capability is not documented.

## What This Changes

The PR adds a `captionedFinalText` channel voice capability and changes ACP final-mode TTS delivery so Telegram can suppress live block text, send it as a voice-note caption, and fall back to text on failures.

## Best Possible Solution

Keep the capability-based caption path, but gate suppression so untagged tagged-mode replies keep live block streaming, and document the plugin opt-in and fallback contract before merge.

## Reproduction Assessment

Yes. The source path on current main settles ACP visible text before final TTS synthesis, and the supplied Telegram Desktop proof shows the baseline separate text-plus-voice behavior versus the candidate captioned voice message.

## Solution Assessment

No, not yet. The capability-based owner boundary is a reasonable fix shape, but the suppression gate must respect tagged-mode semantics and the public capability needs docs before this is the best mergeable solution.

- **[P1] Preserve tagged-mode streaming before suppressing blocks:** `src/auto-reply/reply/dispatch-acp.ts:475`
  - body: This gate treats `messages.tts.auto: "tagged"` as a planned captioned TTS turn before the final text is known. Please keep live blocks enabled for untagged tagged-mode turns or delay suppression until the directive/audio decision is known, and add a regression test for that path.
  - confidence: 0.91
