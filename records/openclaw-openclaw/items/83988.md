## Summary

Keep open: the linked TTS churn problem is valid and the PR is focused, but it is not merge-ready because the patch only skips settlement after live block text can already have been queued, keeps caption capability as a core Telegram allowlist, and still has only mock/unit proof rather than redacted Telegram behavior proof.

## What This Changes

The PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram captioned voice payloads, preserve audio-only supplements elsewhere, add fallback settlement on TTS failure, and add dispatch tests.

## Best Possible Solution

Make captioned final-mode TTS a channel-owned capability, hold or suppress pre-TTS visible text for that capability until the TTS outcome is known, and prove success plus fallback in a redacted Telegram run.

## Reproduction Assessment

Yes, at source level: current main settles visible text before final TTS and the live projector can queue block text before finalization. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Deferring only settleVisibleText does not stop already queued live block delivery, and the caption-capability decision belongs behind a channel-owned capability or helper.
