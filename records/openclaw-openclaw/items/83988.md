## Summary

Keep open: the PR addresses the central Telegram final-mode TTS churn and has useful live Telegram proof, but the current head still suppresses live ACP block text for `messages.tts.auto: "tagged"` even when no directive will trigger audio, and it adds a public channel capability without matching public docs.

## What This Changes

The PR adds a captioned-final TTS channel capability and changes ACP final-mode delivery so caption-capable channels can hold block text and send it as a voice-note caption with fallback text delivery on failures.

## Best Possible Solution

Land a capability-driven fix that suppresses live ACP blocks only when captioned final TTS will actually synthesize, preserves tagged/no-directive streaming, documents the new public channel capability, and keeps the existing failure/abort text fallback.

## Solution Assessment

No, not quite: capability-driven captioned final delivery is the right direction, but the current implementation suppresses live blocks before it knows whether tagged mode will synthesize audio and does not document the new public capability.
