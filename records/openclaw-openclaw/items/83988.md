## Best Possible Solution

Maintainers should explicitly approve the `captionedFinalText` plugin capability contract and Telegram delivery semantics, wait for exact-head checks, then land this PR or narrow the behavior to Telegram-owned code if the public SDK surface is not accepted.

## Reproduction Assessment

## Solution Assessment

Mostly yes, with a maintainer-owned API decision still open. Deferring text until a caption-capable final voice note is delivered is the right bug-fix shape, and it is cleaner than the older Telegram-only delete/recombine approach because it prevents the premature text send instead of repairing it after delivery.
