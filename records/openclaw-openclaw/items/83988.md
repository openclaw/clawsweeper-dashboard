## Summary

Keep this PR open: it is a plausible active fix for Telegram final-mode TTS churn, but it is not merge-ready because the public channel capability/default still needs maintainer ownership, current-head tagged Telegram proof is incomplete, and the new SDK docs overstate tagged-mode block suppression.

## What This Changes

The PR adds a `captionedFinalText` channel TTS capability, opts Telegram into it, and changes ACP/non-ACP final-mode TTS dispatch to defer or suppress text so final voice notes can carry captions with fallback text paths.

## Best Possible Solution

Land a maintainer-approved capability-gated fix after the tagged-mode docs contract is corrected and current-head Telegram proof covers tagged/directive behavior; keep separate text-plus-voice delivery policy tracked in https://github.com/openclaw/openclaw/issues/42539.

## Reproduction Assessment

Yes. Current-main and v2026.6.10 source inspection show ACP settles visible text before final TTS synthesis and the generic dispatcher sends block text before the accumulated TTS supplement; I did not rerun live Telegram locally.

## Solution Assessment

Unclear until maintainer approval. Capability-gated deferral is a maintainable fix shape for the bug, but the new public SDK capability/default plus tagged-mode docs and proof gaps need resolution before this is the best merge-ready solution.
