## Summary

This PR should stay open: it has strong Telegram proof for the intended captioned voice-note behavior, but the current head still suppresses live ACP blocks before knowing whether tagged-mode TTS will actually synthesize audio, and it adds a public channel capability without public docs.

## What This Changes

This PR adds a `captionedFinalText` channel capability and changes ACP final-mode TTS delivery so caption-capable voice channels can hold live block text, send it as a voice-note caption, and fall back to text on failure.

## Best Possible Solution

Keep the capability-based Telegram final-mode caption path, but preserve tagged-mode live streaming until an actual TTS directive or synthesizeable final TTS plan is known, and document the public opt-in contract before merge.

## Reproduction Assessment

Yes. Current main shows `settleVisibleText()` before final TTS synthesis, and the prepared Mantis Telegram proof demonstrates the before/after message behavior; I did not rerun a live Telegram bot locally.

## Solution Assessment

No. The capability-based deferral is the right shape for actual final TTS, but suppressing blocks before tagged-mode directives are known is too broad; the safer fix preserves live streaming until TTS is actually requested.
