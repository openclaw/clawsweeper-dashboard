## Summary

Keep open: this targets a real final-mode ACP TTS ordering bug, but the patch still does not prove or preserve the intended Telegram one-message caption path and adds channel-specific policy in ACP core.

## What This Changes

The PR defers ACP visible-text settlement for final-mode TTS, includes accumulated text on deferred TTS media, preserves audio-only supplements for non-deferred channels, and adds dispatch tests.

## Best Possible Solution

Keep the final-mode TTS deferral, but make captioned voice delivery a channel-owned capability/path and prove the live Telegram success and fallback behavior end to end.

## Reproduction Assessment

Yes, at source level. Current main settles visible text before final TTS synthesis, and the Telegram lane source shows why the patched live path still needs a real transport proof; I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. The deferral is the right bug boundary, but this implementation still lets Telegram strip deferred TTS text from supplement media and hardcodes caption policy in core instead of a channel-owned capability.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve deferred TTS text through Telegram delivery:** `src/auto-reply/reply/dispatch-acp.ts:267-276`
  - body: This final payload is still marked as a TTS supplement. In Telegram live delivery, final media after a streamed preview goes through `mediaOnlyPayload()`, and that path calls `buildTtsSupplementMediaPayload()`, which removes `text` from every TTS supplement. The result is still a finalized text preview plus a captionless voice note, not the one captioned voice message this PR is meant to produce. Please update the Telegram/media path to keep text when the supplement is the primary visible delivery, and cover that lane path.
  - confidence: 0.91
- **[P2] Move caption support out of the ACP core allowlist:** `src/auto-reply/reply/dispatch-acp.ts:241-243`
  - body: Adding `CAPTIONED_VOICE_CHANNELS` bakes Telegram-specific delivery policy into ACP core, while channel plugins already own TTS voice delivery capabilities. This should come from a channel-owned captioned-voice capability or helper so new caption-capable channels do not require editing ACP dispatch and non-caption channels keep their own policy.
  - confidence: 0.84
