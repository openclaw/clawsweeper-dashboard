## What This Changes

Adds a `captionedFinalText` channel TTS capability, opts Telegram into it, and changes ACP/non-ACP final-mode TTS dispatch so caption-capable channels defer or suppress text and fall back to text when TTS delivery fails.

## Best Possible Solution

Land the capability-gated captioned-final fix only after maintainers confirm the `captionedFinalText`/Telegram default direction and the contributor supplies redacted current-head tagged-mode Telegram proof; otherwise pause the branch until the separate-delivery policy in the related product issue is reconciled.

## Reproduction Assessment

Yes. Current-main and v2026.6.10 source show ACP visible text is settled before final TTS, and the non-ACP dispatcher delivers block text before the accumulated final TTS supplement.

## Solution Assessment

Yes for the code boundary, with a product/API caveat. A channel capability keeps the caption decision out of Telegram-specific core branching and the branch adds fallback coverage, but maintainers still need to approve the public capability/default direction.
