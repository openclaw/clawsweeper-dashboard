## Summary

This PR targets a real Telegram final-mode TTS churn issue, but the current patch is not merge-ready: the live ACP path can still deliver or finalize text before the voice payload, the caption capability is hard-coded in core, and the supplied proof is mocked/unit-only rather than live Telegram behavior.

## What This Changes

This PR changes ACP final-mode TTS dispatch to defer visible-text settlement for Telegram, attach accumulated text to a synthesized voice payload, preserve audio-only supplements for non-deferred channels, and add dispatch tests.

## Best Possible Solution

Coordinate final-mode TTS with channel-owned captioned-voice capability and Telegram live-preview handling so success produces one visible captioned voice path, while TTS or voice-send failure still flushes text once.

## Reproduction Assessment

Yes, at source level: current main settles queued visible text before accumulated final TTS, and the live ACP/Telegram path can queue visible text before final media. I did not run a live Telegram/TTS reproduction in this read-only review.

## Solution Assessment

No. The defer-and-fallback shape is directionally right, but this implementation does not coordinate existing live preview delivery and puts Telegram caption policy in core instead of a channel-owned capability.
