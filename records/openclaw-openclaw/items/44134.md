## Summary

Keep this issue open. Current main still rebuilds the embedded-attempt tool catalog and runs provider-owned schema normalization for each attempt, while the discussion contains concrete Google/Gemini symptoms plus a later heap-pressure report tied to the same normalization path.

## What This Changes

The issue asks OpenClaw to avoid repeatedly reloading or normalizing provider tool schemas during Google Antigravity/Gemini sessions after logs showed repeated tool-schema snapshots.

## Best Possible Solution

Add a conservative normalized-tool-schema reuse layer at the runtime/provider boundary, reusing unchanged normalized parameter schemas while rebuilding executable tools and invalidating on tool schema, provider/model/API, plugin, config, or runtime-plan context changes.

## Reproduction Assessment

Yes for the root source behavior: current main builds attempt-local tools in `runEmbeddedAttempt` and calls `normalizeAgentRuntimeTools`/provider normalization every attempt. No live reproduction was established for the external Google ban itself because that depends on provider-side enforcement.

## Solution Assessment

Yes. A generic runtime/provider-boundary schema reuse layer is narrower and more maintainable than Google-only backoff, disabling tools, or adding session-persistence configuration, because the repeated work is shared across provider-owned normalization paths.
