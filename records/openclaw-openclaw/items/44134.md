## Summary

Keep this issue open. Current main still rebuilds attempt-local tools and sends them through provider-owned schema normalization on every embedded attempt, while the discussion has concrete Google/Gemini symptoms and a later heap-pressure report tied to the same normalization path.

## What This Changes

The issue asks OpenClaw to stop repeatedly reloading and normalizing Google/Gemini tool schemas across short-lived attempts because it can trigger provider anti-abuse controls and memory pressure.

## Best Possible Solution

Add a conservative normalized-tool-schema reuse layer at the runtime/provider boundary: reuse unchanged normalized parameter schemas across attempts, rebuild executable AgentTool objects each attempt, and invalidate on tool schema, provider, model/API, runtime-plan, plugin, or config context changes.

## Reproduction Assessment

Yes for the root behavior: current main creates toolsRaw per runEmbeddedAttempt and calls normalizeAgentRuntimeTools/normalizeProviderToolSchemas every attempt. No for the external Google ban itself, which depends on provider-side enforcement.

## Solution Assessment

Yes, a generic runtime/provider-boundary cache is the narrowest maintainable direction. Google-only backoff, disabled tools, or a session-persistence config knob would not address the shared Gemini/OpenAI allocation and repeated-normalization path.
