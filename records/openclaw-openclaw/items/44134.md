## Summary

Keep this open: current main still normalizes provider tool schemas during each embedded attempt, and the active cache implementation remains open at https://github.com/openclaw/openclaw/pull/78664. The heap-profile comment and the related PR body both support the same root behavior, but the fix has not landed on main.

## What This Changes

The issue reports repeated provider tool-schema loading/normalization during Google Antigravity use and asks OpenClaw to avoid repeated schema reload work.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78664 after review/rebase, or replace it with an equivalent conservative provider-owned cache that stores only cloned normalized schema parameters while preserving fresh tool objects, execute closures, approval state, and guards.

## Reproduction Assessment

Yes for the source-level root behavior: current main still builds attempt-local tools and passes them through provider normalization on each embedded attempt. The external Google ban itself was not reproduced because it depends on provider-side enforcement.

## Solution Assessment

Yes: a provider-boundary cache for normalized schema parameters is narrower than session persistence or retry backoff. The active PR should be reviewed, rebased, and landed or replaced before this issue closes.
