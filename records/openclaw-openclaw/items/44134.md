## Summary

Keep this issue open. Current main still rebuilds attempt-local tools and delegates provider-owned tool-schema normalization on every embedded attempt, while the issue discussion includes concrete Google/Gemini account-ban symptoms plus a later heap-pressure report tied to the same normalization path.

## What This Changes

The issue requests reducing repeated Google/Gemini tool-schema reloads by caching or persisting unchanged normalized tool schemas and avoiding unnecessary session recreation churn.

## Best Possible Solution

Add a conservative normalized-tool-schema reuse layer at the runtime/provider-schema boundary: reuse unchanged normalized parameter schemas across attempts, rebuild executable AgentTool objects each attempt, and invalidate on tool schema or provider/model context changes.

## Reproduction Assessment

Yes for the root behavior, not for the external Google ban. Current main shows each embedded attempt builds toolsRaw and calls normalizeAgentRuntimeTools, and a focused test can count repeated provider normalization for unchanged tools.

## Solution Assessment

Yes, a generic runtime/provider-boundary cache is the narrowest maintainable fix. A Google-only workaround, disabled tools, or session backoff would not address the OpenAI/Codex allocation report or the shared normalization cost.
