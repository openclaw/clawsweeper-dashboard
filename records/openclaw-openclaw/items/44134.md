## Summary

Keep open: current main and latest release still run Google/Gemini provider tool-schema normalization inside each embedded attempt, and the provider cleaners still clone schema objects without a catalog/session-level cache. The adjacent open Gemini schema PR is useful but only partially overlaps this issue, while the reported Google account-ban mechanism still needs live maintainer-led provider proof before changing session policy.

## What This Changes

The issue asks OpenClaw to stop repeatedly reloading or re-normalizing Google/Gemini tool schemas during provider sessions because reporters associate the hot path with Google anti-abuse bans and gateway OOM pressure.

## Best Possible Solution

Keep this as the canonical provider hot-path issue and land a maintainer-reviewed fix that prepares or caches normalized tool catalogs at a stable runtime/session boundary, with focused allocation tests plus live Google/Gemini proof or an explicit maintainer decision on the safe cache scope.
