## Summary

Keep open: current main and the latest release still normalize provider tool schemas on each embedded attempt, and the targeted cache fix is still an open conflicting PR rather than merged behavior.

## What This Changes

The issue asks OpenClaw to stop repeatedly reloading or renormalizing Google/Gemini provider tool schemas because it can trigger Google anti-abuse bans and gateway OOM pressure.

## Best Possible Solution

Rebase and review https://github.com/openclaw/openclaw/pull/78664, or replace it with an equivalent bounded provider-normalized schema cache at the same owner boundary, then close this issue after the fix merges.

## Reproduction Assessment

No for the full Google ban outcome; yes at source level for the repeated normalization path. Current main calls normalizeAgentRuntimeTools on each embedded attempt and the Google/Gemini hook maps provider tool schemas through the shared Gemini normalizer each time.

## Solution Assessment

Yes in direction: an internal bounded cache at the provider runtime/tool-schema boundary is the narrowest maintainable fix. Session-persistence config or backoff should stay secondary unless the cache path proves insufficient.
