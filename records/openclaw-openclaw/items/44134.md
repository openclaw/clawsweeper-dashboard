Investigate and prevent repeated Google provider tool-schema processing that may trigger anti-abuse bans, long response delays, and gateway memory pressure.

## Best Possible Solution

Prepare and retain provider-normalized schemas with the stable runtime plan or tool-catalog owner, while normalizing only genuinely dynamic MCP/LSP additions per attempt; validate Google request counts and OpenAI allocation changes as separate scenarios.

## Reproduction Assessment

Yes for the repeated-processing mechanism: current main invokes provider schema normalization during each embedded attempt, including isolated runs. No high-confidence current-main reproduction yet establishes the claimed Google ban or the reported multi-gigabyte allocation rate.

## Solution Assessment

No, the proposed generic cache, exponential backoff, and configurable session persistence are broader than necessary. The narrower fix is lifecycle-owned preparation of stable normalized tools with explicit invalidation for dynamic tool changes.
