## Summary

Keep open: the trust-hardening target is valid, but this patch is not mergeable because it adds trust flags that the current system-event queue neither types nor persists.

## What This Changes

The PR adds non-owner/untrusted flags to Matrix generic reaction `enqueueSystemEvent` calls and updates the focused reaction test to expect those flags for a non-emoji reaction key.

## Best Possible Solution

Choose the intended trust model first, then either carry untrusted system-event metadata through the queue and prompt builder with regression coverage or sanitize Matrix reaction text through the existing inbound system-tag guard.

## Reproduction Assessment

Yes from source inspection: Matrix reaction keys are arbitrary strings and current main enqueues them into generic `System:` event text. I did not run a live Matrix homeserver, but the queue and prompt path are clear in source.

## Solution Assessment

No: this PR adds fields that the current `enqueueSystemEvent` contract does not support, so it is not the best fix. The maintainable path is an end-to-end trust metadata contract or the existing sanitization pattern, with tests at the real queue/drain boundary.
