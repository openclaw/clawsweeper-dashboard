## Summary

Keep open. Current main still drops OpenRouter's `supported_parameters` tool-support signal and defaults missing `compat.supportsTools` to tool-capable, so the reported model can still receive tool payloads; the related open PRs are complementary but do not close this metadata propagation gap.

## What This Changes

Handle OpenRouter models that lack tool support, especially `perplexity/sonar-deep-research`, without routing them through tool-capable agent/runtime payloads.

## Best Possible Solution

Preserve OpenRouter `supported_parameters.includes("tools")` in runtime capabilities and map known toolless dynamic models to `compat.supportsTools: false`, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is clear: live OpenRouter metadata for `perplexity/sonar-deep-research` lacks `tools`, current main discards that signal, and normal embedded runs default absent compat to tool-capable.

## Solution Assessment

Yes. The narrowest maintainable fix is to feed OpenRouter metadata into the existing `compat.supportsTools` and `supportsModelTools()` gate; adding a new non-tool agent mode would be broader than needed for this bug.
