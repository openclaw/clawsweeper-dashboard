## Summary

Keep open. Current main still preserves OpenRouter `supported_parameters` only for reasoning/input metadata, so a metadata-known model without `tools` support can still be resolved as tool-capable for normal agent runs; the related open PRs cover fallback or transport guard pieces but do not add OpenRouter capability propagation.

## What This Changes

Request that OpenClaw detect OpenRouter models such as `perplexity/sonar-deep-research` that do not support tools and avoid routing them through tool-capable agent/runtime payloads without a clear compatibility decision.

## Best Possible Solution

Add provider-local OpenRouter tool-capability plumbing: preserve `supported_parameters.includes("tools")` as optional capability metadata, emit `compat.supportsTools: false` for known toolless dynamic OpenRouter models, and keep default-true behavior only when capability is unknown.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: live OpenRouter metadata lacks `tools` for the reported model, current main discards that signal, and the normal agent path defaults absent compat to tool-capable.

## Solution Assessment

Yes. The narrowest maintainable fix is to feed OpenRouter metadata into the existing generic `supportsModelTools()` gate; a new non-tool agent execution mode would be broader than needed for this bug.

Fix this issue by adding OpenRouter tool-capability propagation, not by adding a broad new agent mode. In `src/agents/pi-embedded-runner/openrouter-model-capabilities.ts`, preserve whether live OpenRouter `supported_parameters` explicitly includes `tools`; keep the value optional so old disk cache entries and unknown metadata preserve current default behavior. In `extensions/openrouter/index.ts`, set `compat.supportsTools: false` on dynamic OpenRouter models only when capabilities explicitly say tools are unsupported. Add focused tests proving `perplexity/sonar-deep-research`-style metadata without `tools` resolves to `compat.supportsTools: false`, metadata with `tools` does not regress, old/unknown cache data remains accepted, and the embedded runner/tool gate suppresses tools through the existing `supportsModelTools()` path. Coordinate with #74664/#74738 as complementary transport work; do not change package dependencies, provider auth, fallback policy, or introduce a new non-tool execution mode. Add a changelog entry because this is user-facing provider/runtime behavior.
