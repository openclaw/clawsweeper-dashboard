## Summary

Keep open. Current main still drops OpenRouter tool-support metadata before the generic runtime tool gate, so `perplexity/sonar-deep-research` can still be treated as tool-capable in normal agent execution even though OpenRouter’s public catalog does not advertise `tools` for that model. The raw model-run path suppresses tools, but that does not fix the reported `openclaw agent` runtime path.

## What This Changes

The issue requests cleaner handling for OpenRouter models whose metadata says they do not support tools, especially `openrouter/perplexity/sonar-deep-research`, so normal agent/runtime paths do not send tool-capable payloads or silently fall back after provider errors.

## Best Possible Solution

Add provider-local OpenRouter tool-capability plumbing: preserve whether `supported_parameters` includes `tools`, expose it as `compat.supportsTools` on dynamic OpenRouter models, keep default-true behavior only when capability is unknown, and cover both parsing and normal embedded-agent tool gating with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: live OpenRouter metadata lacks `tools` for the reported model, current OpenClaw does not retain that metadata, and ordinary agent execution defaults absent compat to tool-capable.

## Solution Assessment

Yes. The narrowest maintainable fix is to feed OpenRouter’s existing metadata into the existing generic `compat.supportsTools` gate; adding a separate non-tool agent architecture would be broader than needed for this bug.
