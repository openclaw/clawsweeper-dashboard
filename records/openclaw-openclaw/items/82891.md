## Summary

Keep open: this is a protected draft MEMBER PR with a useful fix direction, but the patch still holds the session write lock while context-engine finalizers can run plugin/provider work, preserving part of the contention window it is meant to remove.

## What This Changes

The PR narrows embedded PI session write-lock handling, adds stale/max-hold lock config and docs/baselines, and updates related transcript, doctor, gateway, SDK, Codex mirror, and regression tests.

## Best Possible Solution

Keep the lock-controller direction, but move context-engine afterTurn/ingest/maintenance outside the transcript lock and acquire/fence only the actual SessionManager or transcript rewrite/persistence operations, then rerun the focused Testbox and baseline checks.

## Reproduction Assessment

Yes. Current main source shows the embedded attempt lock spans the model turn, and the PR body gives a concrete Testbox reproduction with lock spans and a same-process contention timeout.

## Solution Assessment

No. The approach is close, but wrapping context-engine finalization in `withSessionWriteLock` is still too broad because those hooks can perform runtime LLM/provider work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move context-engine finalizers out of the session lock:** `src/agents/pi-embedded-runner/run/attempt.ts:3920`
  - body: This critical section wraps `finalizeAttemptContextEngineTurn`, which can call context-engine `afterTurn`, `ingestBatch`/`ingest`, and maintenance; the runtime context also exposes `llm.complete`. A context engine can therefore still do slow provider/model I/O while the transcript lock is held, preserving the contention window this PR is trying to remove. Move the lock down to the actual transcript persistence/rewrite calls.
  - confidence: 0.84
