## Summary

Keep open: this draft, maintainer-labeled PR is addressing a real embedded-session lock contention problem, but the proposed lock split still appears to hold the transcript lock around context-engine post-turn hooks that can perform LLM/provider work.

## What This Changes

The PR narrows embedded PI session write-lock handling, adds stale/max-hold lock config, propagates lock options through related paths, and adds focused regression coverage.

## Best Possible Solution

Keep the lock controller approach, but run context-engine afterTurn/ingest/maintenance outside the transcript lock and acquire/fence the lock only inside actual SessionManager or transcript rewrite operations.

## Reproduction Assessment

Yes. Current main clearly holds the embedded attempt write lock across the model turn, and the PR body gives a concrete Testbox reproduction with lock spans and a same-process contention timeout on unmodified main.

## Solution Assessment

No. The direction is right, but the proposed critical section is still too broad because it wraps context-engine post-turn hooks that can call the runtime LLM capability; the lock should be narrowed to actual transcript persistence and rewrite calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not hold the session lock across context-engine finalizers:** `src/agents/pi-embedded-runner/run/attempt.ts:3920`
  - body: This critical section wraps `finalizeAttemptContextEngineTurn`, which can call context-engine `afterTurn`, `ingestBatch`, and `maintain`; the runtime context also exposes `llm.complete`. A context engine can therefore perform slow model/provider I/O while the transcript lock is held, preserving the contention window this PR is trying to remove. Move the lock down into the transcript rewrite/persistence helpers instead of wrapping the whole finalizer.
  - confidence: 0.84
