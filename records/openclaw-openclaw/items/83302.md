## Summary

Keep open: current main still has Telegram inbound paths that preserve a thread id while storing a flat `OriginatingTo`, and the latest shipped tag shows the same pattern. The report is concrete, source-reproducible, and no canonical duplicate or open fix PR turned up in the targeted search.

## What This Changes

Preserve topic-qualified Telegram routing targets in inbound context and add regression coverage for non-General forum-topic follow-up delivery.

## Best Possible Solution

Compute the topic-aware Telegram route once in the plugin and feed it consistently into inbound `OriginatingTo`/`originatingTo` contexts, with focused tests for non-General forum topics and no core routing shim.

## Reproduction Assessment

Yes, source-reproducible: build a Telegram forum message with `message_thread_id` on current main and inspect the inbound context; `MessageThreadId` is preserved while `OriginatingTo` remains `telegram:<chatId>`. I did not run a live Telegram forum reproduction.

## Solution Assessment

Yes: the narrow maintainable fix is to use the existing Telegram topic-aware routing helper in the Telegram plugin’s inbound context builders and add regression coverage, rather than adding a generic core fallback.
