## Summary

Keep open: current main and the latest release still flatten Telegram forum-topic origin targets in normal inbound context, audio preflight context, and mention-skip hook context even when thread metadata is present. Existing Telegram helper/test coverage already defines the topic-qualified target form, so this looks like a narrow real bug rather than a new feature request.

## What This Changes

Preserve topic-qualified Telegram `OriginatingTo` values for non-General forum-topic inbound and follow-up delivery paths whenever a thread id is known.

## Best Possible Solution

Reuse the existing Telegram routing-target helper for inbound origin metadata, cover audio preflight and mention-skip hook contexts, and add focused non-General forum-topic regression tests for follow-up delivery.

## Reproduction Assessment

Yes, at source level: build a Telegram forum-group inbound context with a non-General `message_thread_id`, and current main carries `MessageThreadId` while setting `OriginatingTo` to the flat chat target. I did not run a live Telegram repro in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing `buildTelegramRoutingTarget` contract on the affected inbound-context paths instead of adding a new config or routing policy.
