## Summary

Keep open: the single-session ACP no-output class has been improved, but current main and the latest shipped tag still intentionally allow distinct ACP child sessions to run in parallel, and I found no merged fix, release note, test, or live current-main proof that the reported rapid two-spawn first-child launch failure is gone.

## What This Changes

The issue asks OpenClaw to stop rapid `sessions_spawn(runtime:"acp")` calls from accepting two child sessions while the first child never launches or emits assistant output.

## Best Possible Solution

Run a current-main and latest-release live ACPX/Claude two-spawn smoke; if it reproduces, fix the confirmed ACPX launch/readiness or add a narrow OpenClaw startup gate that preserves intentional different-session parallelism.

## Reproduction Assessment

No. The original v2026.3.22/acpx 0.1.16 stream logs are concrete, but this read-only pass did not run a current-main/latest-release ACPX/Claude two-spawn smoke and current source only proves the parallel path remains allowed.

## Solution Assessment

Unclear. Fixing ACPX launch/readiness is best if the backend drops the first child, while serializing all distinct ACP child sessions in OpenClaw needs maintainer approval because current tests intentionally allow that parallelism.
