## Summary

Keep open: current main still accepts ACP spawns before observed child output, intentionally runs different ACP child sessions in parallel, and I found no merged fix, release note, or live current-main proof that the reported rapid two-spawn first-child launch failure is gone.

## What This Changes

## Best Possible Solution

Run a current-main and latest-release ACPX/Claude two-spawn smoke; if it reproduces, fix the confirmed ACPX launch/readiness fault or add a narrow startup guard that preserves intentional different-session parallelism.

## Reproduction Assessment

## Solution Assessment

Unclear. Fixing ACPX launch/readiness is best if the backend drops the first child, while serializing distinct ACP child sessions needs maintainer approval because current tests intentionally preserve that parallelism.
