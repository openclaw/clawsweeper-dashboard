## Summary

Keep open: current main has newer ACPX integration, but source inspection still shows rapid ACP spawns create distinct child session keys and are only serialized per child key, with no source, test, release note, or related merged PR proving the first-child no-process failure is fixed.

## What This Changes

The issue reports that two rapid ACP `sessions_spawn(runtime="acp")` calls can leave the first child accepted but with no Claude Code process or assistant output while the second eventually runs.

## Best Possible Solution

Keep this open until maintainers either get live current-main ACPX proof that rapid Claude ACP spawns are now reliable or choose a bounded launch-coordination fix at the ACPX/OpenClaw boundary.

## Reproduction Assessment

No high-confidence live reproduction: I did not run ACPX/Claude, but source inspection confirms current main can still initialize and dispatch distinct ACP child session keys concurrently. The issue’s original stream-log evidence is credible but not enough to prove current-main failure after ACPX upgrades.

## Solution Assessment

Unclear: serializing all ACP launches may be a plausible mitigation, but it is not obviously the best fix because it trades off intended parallel ACP work and may belong in ACPX rather than OpenClaw core.
