## Summary

Keep open: current main and v2026.6.8 still allow distinct ACP child sessions to run concurrently, and I found no live, source, release, or related-item proof that the reported rapid two-spawn Claude ACP startup loss is fixed.

## What This Changes

The issue reports that two rapid `sessions_spawn(runtime="acp")` calls can leave the first accepted Claude ACP child with no launched process or assistant output while the second eventually runs.

## Best Possible Solution

Run a current v2026.6.8/main live ACPX two-spawn smoke; if it still reproduces, fix ACPX launch/readiness or add a narrow OpenClaw startup gate while preserving intentional different-session parallelism.

## Reproduction Assessment

No. The original report has concrete stream-log evidence on OpenClaw 2026.3.22/acpx 0.1.16, and current source still allows separate ACP child sessions to run in parallel, but I did not establish a failing current-main or v2026.6.8 live ACPX path.

## Solution Assessment

Unclear. The next step should be current ACPX validation; the fix belongs in backend launch/readiness handling if the child is dropped there, or in OpenClaw's ACP manager only if a bounded startup gate is proven necessary.
