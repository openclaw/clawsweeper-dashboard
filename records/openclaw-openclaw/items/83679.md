## Summary

Keep open: current main and the latest release have last-known-good config snapshots and manual doctor repair, but they explicitly do not auto-restore on gateway startup or hot reload failure. The requested safe-startup rollback remains a core gateway resilience feature that needs product and safety design.

## What This Changes

Request a gateway safe-startup mode that rolls back to a known-good or minimal safe config after repeated startup failures.

## Best Possible Solution

Design an opt-in gateway-owned safe-startup policy that reuses the existing last-known-good and doctor recovery primitives, defines health and exit-code semantics, and avoids overloading the existing `--safe` restart meaning.

## Risks / Open Questions

- Automatic rollback would mutate a user's active config during startup, so the design must avoid overwriting intentional edits or secret-redacted config snapshots.
- The requested `restart --safe` wording conflicts with the existing `--safe` restart flag, which currently means active-work deferral rather than config rollback.
- Health probation and exit-code policy are not defined in current code, so the feature needs maintainer agreement before implementation.
