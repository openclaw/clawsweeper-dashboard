## Summary

This looks like a valid current-main regression in the display/session-list layer: stale-but-preserved finite `totalTokens` are still filtered out for `/context` and Control UI consumers after the merged preservation fix. The report is narrow and actionable, but the exact suggested helper-only change should be applied carefully because several compaction and memory paths still rely on fresh-only semantics.

## What This Changes

The issue requests that stale-but-preserved session total token counts remain visible to `/context` and UI consumers instead of being treated as unavailable.

## Best Possible Solution

Expose finite preserved totals to display and session-list consumers while keeping a clearly named fresh-only path for compaction, memory, fork, and compact logic.

## Reproduction Assessment

Yes, by source inspection on current `main`: a row with finite `totalTokens` and `totalTokensFresh: false` is filtered to `undefined` by the helper and by the gateway/UI tests. I did not run an end-to-end UI session because this review is read-only.

## Solution Assessment

No, not as a blanket helper-only change: several runtime paths still use `resolveFreshSessionTotalTokens` as a fresh-only guard. The safer fix is to split or rename the helper semantics, then update only `/context`, session-list, and Control UI display consumers to show stale preserved totals with the freshness flag intact.
