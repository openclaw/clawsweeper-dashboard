## Summary

Keep open. Current main still suppresses Feishu text whenever native voice media is delivered, and there is no implemented opt-in config for preserving text; the adjacent generic TTS delivery-policy issue informs the design but does not fully supersede this Feishu accessibility request.

## What This Changes

Add a backward-compatible option for Feishu TTS replies to send both visible text and a voice bubble.

## Best Possible Solution

Add an explicit, backward-compatible TTS text-delivery policy that Feishu can honor at the chosen channel/account or generic scope while preserving current defaults and fallback behavior.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a failing bug reproduction because the requested behavior is a new opt-in policy.

## Solution Assessment

Unclear. A Feishu-only boolean would address the report, but a generic TTS text-delivery policy that Feishu honors may be the cleaner maintainable direction given the related auto-TTS delivery-policy discussion.
