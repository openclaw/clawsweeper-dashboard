## Summary

Keep open. Current main still suppresses Feishu text when native voice media is present, and the requested opt-in text preservation is not implemented; the open implementation PR at https://github.com/openclaw/openclaw/pull/81258 is the right place to resolve the remaining code and product-surface questions before this issue closes.

## What This Changes

Add a Feishu TTS configuration option that sends visible text alongside native voice-message replies while preserving the current voice-only default.

## Best Possible Solution

Land a backward-compatible TTS text-delivery policy that keeps Feishu voice-only as the default, optionally preserves text per the chosen global/channel/account scope, and documents fallback behavior.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a strict bug reproduction because the request is for a new opt-in delivery policy.

## Solution Assessment

Unclear. A Feishu-only boolean may be acceptable, but maintainers should confirm whether the durable solution is Feishu-specific or a shared TTS text-delivery policy, and the linked PR still needs fallback behavior fixed.
