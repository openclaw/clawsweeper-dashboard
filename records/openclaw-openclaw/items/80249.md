## Summary

Keep open. Current main still intentionally suppresses Feishu text when a reply carries native or voice-intent media, and preserving visible text would require a new opt-in config/product policy rather than a narrow bug fix.

## What This Changes

Request an opt-in Feishu/TTS setting that sends visible text as well as the native voice bubble for voice replies.

## Best Possible Solution

Add a backward-compatible TTS text-delivery policy that Feishu can honor per the chosen scope, preserving the current default suppression and fallback behavior unless the opt-in is enabled.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses visible text whenever a reply delivers native or voice-intent media. This is not a failing bug reproduction because the requested behavior is a new opt-in policy.

## Solution Assessment

Unclear. A Feishu-only boolean is plausible, but a generic `messages.tts` text-delivery policy shared with https://github.com/openclaw/openclaw/issues/73210 may be the cleaner maintainable direction.
