## Summary

Keep this issue open: current main and the latest release still suppress Feishu text whenever voice media is detected, and the open implementation candidate is blocked on maintainer product/config review rather than being merged.

## What This Changes

Requested an opt-in Feishu TTS setting that sends readable text alongside voice-message media instead of always suppressing the text.

## Best Possible Solution

Land a maintainer-approved opt-in Feishu TTS text-preservation mode, or decide on a broader TTS delivery-mode design, then close this issue when the chosen implementation is merged.

## Reproduction Assessment

Yes, from source inspection: Feishu final delivery sets `shouldDeliverText` false whenever `hasVoiceMedia` is true, and `audioAsVoice: true` makes media count as voice media. I did not run a live Feishu/Lark account because this is a read-only review.

## Solution Assessment

Unclear pending maintainer decision. A backward-compatible Feishu opt-in is plausible, but maintainers should decide whether the permanent surface is Feishu-specific `tts.preserveText` or a broader TTS delivery-mode model.
