## Summary

Keep open: current main still suppresses Feishu text when native voice media is present, and the linked implementation PR remains open with a review blocker plus an unsettled config-surface decision.

## What This Changes

Add an opt-in Feishu TTS setting to keep text visible alongside delivered voice replies.

## Best Possible Solution

Settle a backward-compatible TTS text-delivery policy, then land the linked Feishu implementation after preserving the current default suppression and fallback-text behavior.

## Reproduction Assessment

Yes, at source level: current main and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a failing bug repro because the requested behavior is a new opt-in delivery policy.

## Solution Assessment

Unclear. A Feishu-only `preserveText` option could solve this user's case, but maintainers should decide whether to align it with the broader generic TTS `textDelivery` policy before merge.
