## Summary

Keep open. Current main still intentionally suppresses Feishu text when native voice media is delivered, and an open related PR is already attempting the requested opt-in config surface, so the issue should remain open until that implementation is reviewed and either merged or closed.

## What This Changes

Request an opt-in Feishu TTS setting that sends the reply text alongside native voice media instead of voice-only delivery.

## Best Possible Solution

Review the open Feishu implementation PR and settle the config scope, preserving the current default while adding tests and docs for the chosen opt-in policy.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a failing bug repro because the requested behavior is a new opt-in delivery policy.

## Solution Assessment

Unclear. A Feishu-only boolean may be adequate, but maintainers should choose whether this belongs in Feishu account config or a generic TTS text-delivery policy before landing the related implementation.
