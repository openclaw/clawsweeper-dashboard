## Summary

Keep open. Current main and v2026.5.12 still suppress Feishu text when native voice media is present, and the requested opt-in config is not implemented; the linked open PR targets this issue but has not landed.

## What This Changes

Request an opt-in Feishu TTS setting that sends the text reply alongside the voice bubble instead of suppressing text for voice media.

## Best Possible Solution

Review the linked implementation or a narrower replacement, then land a backward-compatible TTS text-delivery policy that preserves current default suppression and fallback behavior.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a strict bug reproduction because the requested preserve-text behavior is a new opt-in policy.

## Solution Assessment

Unclear. The linked Feishu boolean is concrete, but maintainers should decide whether the durable API is Feishu-specific or a shared `messages.tts` text-delivery policy before merge.
