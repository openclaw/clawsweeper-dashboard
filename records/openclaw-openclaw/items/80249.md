## Summary

Keep open. Current main intentionally suppresses Feishu voice-reply text, and the requested opt-in would add a new config/product policy that is not implemented or cleanly covered by another exact canonical item.

## What This Changes

The issue requests a Feishu configuration option to send text alongside TTS voice replies instead of suppressing text whenever voice media is present.

## Best Possible Solution

Add an explicit, backward-compatible TTS text-delivery policy that Feishu can honor per channel/account if maintainers choose that scope, preserving the current default and failure fallback behavior.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a failing bug repro because the requested behavior is a new opt-in policy.

## Solution Assessment

Unclear. A plain Feishu-only boolean may work, but a generic `messages.tts.textDelivery` policy that Feishu honors could better align this with the related auto-TTS delivery-policy issue.
