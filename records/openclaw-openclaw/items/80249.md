## Summary

Keep open. Current main still intentionally suppresses Feishu text when a reply is delivered as native voice media, and the requested opt-in text-preservation behavior is a new delivery policy/config decision rather than an already-implemented fix.

## What This Changes

Add an opt-in Feishu/TTS configuration path that sends visible text alongside native voice-message replies.

## Best Possible Solution

Add a backward-compatible explicit TTS text-delivery policy that Feishu can honor at the maintainer-chosen scope while keeping voice-only as the default.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a failing bug reproduction because preserving text would be a new opt-in policy.

## Solution Assessment

Unclear. A Feishu-only boolean is plausible, but maintainers should first choose whether this belongs in generic `messages.tts` delivery policy or Feishu channel/account config.
