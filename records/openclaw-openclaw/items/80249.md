## Summary

Keep open. Current main still intentionally suppresses Feishu text when native voice media is present, and no config/docs surface implements the requested opt-in text-plus-voice policy; this is a new delivery-policy choice rather than a cleanup close.

## What This Changes

Add an opt-in Feishu/TTS configuration that sends visible text alongside native Feishu voice bubbles while preserving the current default behavior.

## Best Possible Solution

Add a backward-compatible text-delivery policy that Feishu can honor at the maintainer-chosen scope, defaulting to current suppression and preserving fallback text when native voice delivery fails.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu skips text whenever a reply has native voice media. This is not a failing bug reproduction because the requested behavior is a new opt-in delivery policy.

## Solution Assessment

Unclear. A Feishu-only boolean would satisfy the reported use case, but maintainers should decide whether this belongs in Feishu account config or a generic TTS text-delivery policy alongside https://github.com/openclaw/openclaw/issues/73210.
