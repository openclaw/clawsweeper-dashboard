## Summary

Keep this issue open: current main and v2026.6.10 still suppress Feishu text whenever voice media is detected, and the open implementation PR remains unmerged and needs maintainer product/config review.

## What This Changes

The issue requests an opt-in Feishu TTS mode that sends readable text alongside voice media instead of suppressing the text.

## Best Possible Solution

Review and repair the linked Feishu PR, then either land a maintainer-approved opt-in Feishu text-plus-voice mode or replace it with a broader TTS delivery-mode design.

## Reproduction Assessment

Yes. Source inspection shows current Feishu final delivery sets `shouldDeliverText` false whenever voice media is detected, and `audioAsVoice: true` makes media count as voice media; no live Feishu/Lark account was exercised in this read-only review.

## Solution Assessment

Unclear pending maintainer decision. A backward-compatible Feishu opt-in is a plausible bounded solution, but maintainers should decide whether the permanent surface is Feishu-specific `tts.preserveText` or a broader TTS delivery-mode model.
