## Summary

Keep open. Current `main` still suppresses Feishu text when native voice media is delivered, and the requested opt-in is a new Feishu/TTS delivery-policy setting that is being pursued by an open linked PR rather than already implemented.

## What This Changes

The issue requests an opt-in Feishu TTS setting that sends both the text reply and the voice bubble instead of suppressing text when voice media is present.

## Best Possible Solution

Review and resolve the linked Feishu PR or replace it with a maintainer-approved generic TTS delivery policy, preserving the current default and fallback-on-voice-failure behavior.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever native voice media is delivered successfully. This is not a bug repro because the requested opt-in is a new delivery policy.

## Solution Assessment

Unclear. A Feishu-only `preserveText` option may solve this use case, but maintainers should decide whether the policy belongs in Feishu config or a generic TTS text-delivery surface before merge.
