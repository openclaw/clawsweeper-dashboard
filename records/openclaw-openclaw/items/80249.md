## Summary

Keep open. Current main still suppresses Feishu text when native voice media is present, no preserve-text config exists, and an open linked PR is the active implementation path rather than a reason to close this issue early.

## What This Changes

Request an opt-in Feishu TTS setting that sends readable text alongside native voice media while preserving the current voice-only default.

## Best Possible Solution

Review and refine the linked Feishu PR, or replace it with a shared TTS delivery-policy design if maintainers want one config contract across channels.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses visible text whenever the reply delivers native voice media. This is not a failing bug repro because the requested outcome is a new opt-in delivery policy.

## Solution Assessment

Unclear. A Feishu-only boolean may be a narrow fit, but maintainers should confirm the config scope and the linked PR must avoid duplicating fallback text before merge.
