## Summary

Keep open. Current main still intentionally suppresses Feishu text when native voice media is present, and the requested opt-in text-preservation behavior would add a new public delivery/config policy rather than fix a broken existing contract.

## What This Changes

Request an opt-in Feishu/TTS setting that sends visible text alongside a native voice bubble instead of suppressing the text for voice media.

## Best Possible Solution

Add a backward-compatible TTS text-delivery policy that Feishu can honor per the chosen scope, keeping duplicate-text suppression as the default and preserving text fallback on voice upload degradation or failure.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever final replies deliver native voice media. This is not a failing bug reproduction because the requested behavior is a new opt-in policy.

## Solution Assessment

Unclear. A Feishu-only boolean would solve the immediate use case, but a shared messages.tts text-delivery policy that Feishu can honor may be cleaner given the related Telegram and auto-TTS delivery-policy requests.
