## Summary

Current main and the latest release still suppress Feishu text whenever voice media is present, so the requested behavior is not implemented. The direct implementation candidate is still open at https://github.com/openclaw/openclaw/pull/81258, but it is conflicting/waiting on author and carries a maintainer product decision around a new Feishu TTS config surface.

## What This Changes

The issue requests an opt-in Feishu TTS setting that sends readable text alongside voice media instead of dropping text for voice bubbles.

## Best Possible Solution

Make an explicit maintainer decision on Feishu-only preserve-text behavior versus a shared TTS delivery mode, then repair and land https://github.com/openclaw/openclaw/pull/81258 or replace it with the chosen canonical design.

## Reproduction Assessment

Yes, from source: current main gates Feishu text delivery on `!hasVoiceMedia`, and voice media is detected for `audioAsVoice` or native Ogg/Opus media. I did not run a live Feishu/Lark account in this read-only review.

## Solution Assessment

No close is appropriate yet. A Feishu opt-in config is a plausible solution, but the best final shape needs maintainer acceptance because it adds a config surface and changes visible message delivery volume/order when enabled.
