## Summary

Keep open. Current main still suppresses Feishu text whenever a reply is delivered as native voice media, and neither Feishu nor shared TTS config exposes an opt-in text-preservation policy; related TTS delivery issues are broader or channel-specific, not an exact canonical replacement.

## What This Changes

Request an opt-in Feishu/TTS setting that sends visible text alongside native Feishu voice bubbles for TTS replies.

## Best Possible Solution

Define a backward-compatible TTS text-delivery policy that Feishu honors at the chosen scope, defaulting to current suppression and preserving fallback text on voice upload failure or degradation.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses visible text when a final reply delivers `audioAsVoice` or native `.ogg`/`.opus` voice media. This is not a failing bug repro because the requested behavior is a new opt-in feature/config policy.

## Solution Assessment

Unclear. A Feishu-only boolean would solve the report, but a shared `messages.tts` text-delivery policy aligned with the related Telegram and auto-TTS issues may be the more maintainable contract.
