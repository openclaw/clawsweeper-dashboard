## Summary

Keep open. Current main still intentionally suppresses Feishu text when native voice media is delivered, and there is no implemented config/docs surface for the requested opt-in; the related generic auto-TTS delivery-policy issue is adjacent but not an exact canonical owner.

## What This Changes

Request to add an opt-in Feishu/TTS configuration that sends visible text alongside the native voice bubble instead of suppressing text for voice media.

## Best Possible Solution

Add a backward-compatible TTS text-delivery policy that Feishu can honor per channel/account if maintainers choose that scope, while preserving current default suppression and fallback-text behavior.

## Reproduction Assessment

Yes for the current behavior: source and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a failing bug repro because the requested opt-in is a new feature/config policy.

## Solution Assessment

Unclear. A Feishu-only boolean is plausible, but maintainers should decide whether this belongs in Feishu account config or a generic `messages.tts.textDelivery` policy alongside https://github.com/openclaw/openclaw/issues/73210.
