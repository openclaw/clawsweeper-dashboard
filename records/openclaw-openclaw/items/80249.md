## Summary

Keep open. Current main still intentionally suppresses Feishu text when a reply is delivered as native voice media, and the requested opt-in text-plus-voice behavior would add a new config/product policy rather than fix a broken existing contract.

## What This Changes

Request adding an opt-in Feishu/TTS setting to send visible text in addition to native voice bubbles instead of the current voice-only suppression.

## Best Possible Solution

Define a backward-compatible TTS text-delivery policy that Feishu can honor at channel/account scope, preserving current default suppression and fallback behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main suppresses Feishu text whenever native voice media is delivered.

## Solution Assessment

Unclear. A Feishu-only boolean could satisfy this report, but a shared `messages.tts` delivery policy coordinated with the related open TTS delivery issues is likely cleaner.
