## Summary

Keep open. Current main still suppresses Feishu text whenever native voice media is delivered, no accepted preserve-text or generic TTS text-delivery config exists, and the open implementation at https://github.com/openclaw/openclaw/pull/81258 is a linked fix candidate that still needs PR/product review rather than an issue close.

## What This Changes

Add an opt-in Feishu TTS delivery option so text can be sent alongside voice replies instead of being suppressed when voice media is present.

## Best Possible Solution

Review or replace the linked Feishu implementation so it preserves text without fallback duplicates, then document the maintainer-approved Feishu-specific or shared TTS delivery policy.

## Reproduction Assessment

Yes for the current behavior: source and regression tests show final Feishu replies with text plus voice media skip text and send it only as fallback on voice degradation. This is a feature request rather than a broken-contract bug.

## Solution Assessment

Unclear. A Feishu opt-in is plausible, but because it adds public config and overlaps the generic TTS text-delivery policy, maintainers should choose the accepted surface before landing.
