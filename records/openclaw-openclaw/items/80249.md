## Summary

Keep open. Current main still suppresses Feishu text when native voice media is present, the requested behavior needs a new config/product policy, and an open Feishu PR already tracks the concrete implementation path but is not merged.

## What This Changes

Add an opt-in Feishu TTS setting that sends visible text alongside voice-message media instead of suppressing the text.

## Best Possible Solution

Choose the Feishu-specific or generic TTS delivery-policy surface, fix the open PR's fallback behavior, add focused docs/tests, then close this issue after the implementation merges.

Yes for the current behavior: source and tests show Feishu suppresses text whenever a reply delivers native voice media. This is not a strict bug repro because the requested outcome is a new opt-in policy.

## Solution Assessment

No, not yet. The requested direction is plausible, but maintainers still need to choose the policy surface and the open implementation PR must avoid duplicate fallback text before merge.
