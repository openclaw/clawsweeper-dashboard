## Summary

Keep open. Current main still intentionally suppresses Feishu text when native voice media is delivered, tests encode that behavior, and the requested opt-in is a new config/product policy not yet implemented or exactly owned by the related generic TTS delivery-policy issue.

## What This Changes

Request adds a backward-compatible Feishu/TTS option to send visible text alongside native Feishu voice replies instead of suppressing text whenever voice media is present.

## Best Possible Solution

Add an explicit backward-compatible text-delivery policy, likely generic TTS with Feishu/account override support if maintainers choose that scope, while keeping the current default suppression and failure fallback behavior.

## Reproduction Assessment

Yes for the current behavior: current-main source and tests show Feishu suppresses visible text whenever a reply delivers native voice media. This is a source-level reproduction of an existing policy, not a failing bug path.

## Solution Assessment

Unclear. A Feishu-only boolean would solve the reporter's use case, but the related generic TTS delivery-policy issue means maintainers should choose the config scope before implementation.
