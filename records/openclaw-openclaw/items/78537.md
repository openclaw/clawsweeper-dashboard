## Summary

This issue should stay open because current `main` still documents `allowInsecurePath` mainly as a Windows ACL bypass and does not describe the POSIX uid ownership check or the root-owned system-package binary case; an open docs PR already targets the gap with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests documentation that `allowInsecurePath` also bypasses the POSIX uid ownership check for SecretRef file and exec provider paths, including trusted root-owned system-package binaries such as `/usr/bin/op`.

## Best Possible Solution

Land or replace the open docs PR so the gateway, CLI, and configuration-reference docs describe the POSIX uid ownership check and the trusted root-owned binary use case without changing runtime behavior.

## Reproduction Assessment

Yes for source behavior: current `main` still has the early `allowInsecurePath` bypass and the later POSIX uid mismatch error path in `src/secretref/resolve.ts`. I did not run a live Linux `/usr/bin/op` setup because this read-only review did not execute runtime scenarios.

## Solution Assessment

Yes, the best current solution is docs-only unless maintainers want a separate product decision for a new safer option name. The open docs PR is the narrowest current path because it updates the documented behavior without changing the compatibility-sensitive SecretRef security checks.
