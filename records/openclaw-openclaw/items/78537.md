## Summary

Keep open: current main and the latest release still omit the POSIX uid/root-owned package-binary guidance, and an open linked docs PR uses closing syntax for this issue, so the issue should remain open until that PR lands or is closed.

## What This Changes

The issue requests documenting that `allowInsecurePath` bypasses POSIX uid ownership checks for SecretRef file and exec provider paths, including root-owned system-package binaries like `/usr/bin/op`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79273, or an equivalent docs-only update, so the public CLI, gateway secrets, and configuration reference pages explain the POSIX uid check without changing the runtime/config contract.

## Reproduction Assessment

Yes. Current main source shows the POSIX uid rejection path and the docs still mention only the Windows ACL escape hatch, so the docs gap is source-reproducible without a live Linux 1Password setup.

## Solution Assessment

Yes. A focused docs-only update is the narrowest maintainable solution for this issue; splitting `allowInsecurePath` into new config knobs would be a separate API/product decision.
