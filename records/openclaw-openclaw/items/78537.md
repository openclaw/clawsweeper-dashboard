## Summary

Keep open: current main still has the reported docs gap, and the open linked docs PR uses closing syntax for this issue, so the issue should stay open until that PR is reviewed and landed or closed.

## What This Changes

The issue requests documentation that SecretRef `allowInsecurePath` also bypasses POSIX uid ownership checks for trusted root-owned system-package binaries such as `/usr/bin/op`.

## Best Possible Solution

Review and land the linked docs PR, or an equivalent docs-only update, so the public CLI, gateway secrets, and configuration reference pages explain the POSIX uid check without changing the runtime/config contract.

## Reproduction Assessment

Yes. Current main source shows the POSIX uid rejection path and current docs still mention only the Windows ACL escape hatch, so the docs gap is source-reproducible without a live Linux 1Password setup.

## Solution Assessment

Yes. A focused docs-only update is the narrowest maintainable fix for this issue; splitting `allowInsecurePath` into a new config option would be a separate API/product decision.
