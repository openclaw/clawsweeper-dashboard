## Summary

Keep open: current main and the latest release still document `allowInsecurePath` as a Windows ACL escape hatch without the POSIX uid/root-owned binary case, and an open contributor PR already uses closing syntax for this issue.

## What This Changes

Document that `allowInsecurePath` also bypasses POSIX uid ownership checks for SecretRef provider paths, including trusted root-owned system package binaries.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79273, or an equivalent docs-only update, so the public CLI, gateway secrets, and configuration reference pages explain the POSIX uid check without changing the runtime/config contract.

## Reproduction Assessment

Yes. Source inspection on current main shows the POSIX uid rejection path and the docs still describe only the Windows ACL escape hatch, so the docs gap is reproducible without a live Linux 1Password setup.

## Solution Assessment

Yes. A focused docs-only update is the narrowest maintainable solution for this issue; splitting `allowInsecurePath` into new config knobs would be a separate product/API decision.
