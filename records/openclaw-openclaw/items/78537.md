## Summary

Keep open: current main still has a source-confirmed SecretRef path-security behavior that the user-facing docs only describe as a Windows ACL escape hatch, leaving the POSIX uid/system-package path case undocumented.

## What This Changes

Document that `allowInsecurePath` also bypasses POSIX uid ownership checks for SecretRef file and exec provider paths, including Linux system-package exec binaries.

## Best Possible Solution

Update the user-facing secrets docs to describe POSIX uid ownership checks and package-manager/root-owned path guidance while preserving the existing security warning around broad `allowInsecurePath` bypass behavior.

## Reproduction Assessment

Yes for the docs gap: current main source has the POSIX uid check and current docs mention only Windows ACL fallback. I did not run a live Linux 1Password repro, but the reported error path is source-confirmed.

## Solution Assessment

Yes: the docs-only path is the narrowest maintainable fix for this issue. Splitting `allowInsecurePath` into new config options would be an API/product change and should not be bundled into this cleanup.
