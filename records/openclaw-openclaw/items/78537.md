## Summary

This should stay open because current main and `v2026.6.10` still document `allowInsecurePath` as Windows ACL-oriented guidance while the POSIX uid ownership behavior remains present in runtime source. The open linked docs PR is the active fix candidate, so this issue should close only after that PR or an equivalent narrow docs patch lands.

## What This Changes

Document that `allowInsecurePath` also bypasses POSIX/Linux uid ownership checks for SecretRef file and exec provider paths, including trusted root-owned system-package binaries.

## Best Possible Solution

Refresh and land the linked docs PR, or an equivalent narrow patch, across the gateway secrets guide, CLI secrets guide, and configuration reference without changing SecretRef runtime semantics.

## Reproduction Assessment

Yes. Source inspection shows the early `allowInsecurePath` bypass and later POSIX uid mismatch rejection, while current docs still describe only Windows ACL behavior; I did not run a live Linux `/usr/bin/op` setup in this read-only review.

## Solution Assessment

Yes. A docs-only update is the best current fix because it documents existing behavior without adding a compatibility-sensitive new SecretRef security option; splitting the option would be separate product/security work.
