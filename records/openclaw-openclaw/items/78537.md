## Summary

This issue should stay open because current main and the latest release still document `allowInsecurePath` mainly as a Windows ACL bypass while the POSIX uid ownership behavior remains real in source. The open linked docs PR is the active fix candidate and should be reviewed, updated, landed, or replaced before this issue closes.

## What This Changes

Document that SecretRef `allowInsecurePath` also covers POSIX uid ownership failures for trusted root-owned system-package file and exec provider paths.

## Best Possible Solution

Land or replace the open docs PR so the gateway, CLI, and configuration-reference docs explain POSIX uid ownership and trusted root-owned binaries while preserving the existing runtime behavior.

## Reproduction Assessment

Yes. Source inspection shows the early `allowInsecurePath` bypass and later POSIX uid mismatch rejection, while the affected docs still describe only Windows ACL behavior; I did not run a live Linux `/usr/bin/op` setup in this read-only review.

## Solution Assessment

Yes. A docs-only update is the best current fix because it documents existing behavior without adding a compatibility-sensitive new SecretRef security option; splitting the option would be a separate product/security decision.
