## Summary

Keep open: the report still matches current main. The initial heartbeat path passes the heartbeat model override, but exec/session follow-up paths can re-enter the generic gateway agent path without heartbeat origin or heartbeat model metadata, so this is distinct from the already-closed heartbeat model bleed issues.

## What This Changes

The issue reports that `agents.defaults.heartbeat.model` is honored for the first heartbeat turn but lost for follow-up or retry runs in the same heartbeat session.

## Best Possible Solution

Add a focused regression for heartbeat-originated follow-up and exec-denial retry paths, then carry heartbeat origin and the resolved heartbeat model through those follow-up contracts without making the heartbeat model sticky for later user turns.

## Repair Work Prompt

Fix the heartbeat follow-up model leak reported in https://github.com/openclaw/openclaw/issues/84825. Current main applies `agents.defaults.heartbeat.model` only when `GetReplyOptions.isHeartbeat` is true, but exec approval/session follow-ups can call the gateway `agent` path without heartbeat origin or heartbeat model metadata and then run as `trigger: "user"` on the normal default model. Add focused regression coverage for a heartbeat-originated follow-up/retry path, especially exec-denial or gateway-agent follow-up if feasible. Keep the fix bounded to heartbeat-origin propagation and model selection; do not add a new config option, do not make heartbeat.model persist into later normal user turns, and preserve the existing shared-session runtime model protection. Update docs only if the public behavior contract changes, and add a changelog entry only if maintainers require it during landing.
