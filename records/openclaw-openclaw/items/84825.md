## Label Justifications

- `P2`: The bug can route background heartbeat continuations to an unexpectedly expensive primary model, but the blast radius is limited to heartbeat sessions with follow-up runs.
- `impact:session-state`: The root problem is heartbeat session origin state not being preserved across follow-up execution boundaries.

## Summary

Keep open: current main still has a source-level heartbeat-origin gap for follow-up agent runs. The initial heartbeat reply path resolves `heartbeat.model`, and the direct reply runner now marks embedded heartbeat runs correctly, but gateway `agent` follow-ups still enter `agentCommand` as normal user-triggered runs and do not select the heartbeat model.

## What This Changes

The issue reports that heartbeat follow-up runs in an existing heartbeat session re-resolve to the default primary model instead of preserving `agents.defaults.heartbeat.model`.

## Best Possible Solution

Add a narrow heartbeat-origin contract for follow-up agent runs so heartbeat-triggered continuations select the configured heartbeat model while normal user turns and stored session model restoration remain unchanged.

## Reproduction Assessment

Yes at source level, but not as a live current-main run. The code shows heartbeat.model only resolves under `opts.isHeartbeat`, while gateway `agent` follow-ups can enter `agentCommand` without that flag or an explicit heartbeat model override.

## Solution Assessment

Yes, the best fix is a narrow contract repair: carry heartbeat origin/model selection into follow-up agent runs and add regression coverage, rather than adding a new setting or changing normal default model resolution.
