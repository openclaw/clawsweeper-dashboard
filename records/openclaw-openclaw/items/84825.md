## Summary

Keep open: current main still has a source-level path where exec-approval follow-up runs spawned from a heartbeat session can re-enter the generic agent RPC without heartbeat run-kind or heartbeat model override metadata, so the prior adjacent heartbeat-model fixes do not cover this reported follow-up leak.

## What This Changes

The issue requests that every follow-up run within a heartbeat-originated session continue to honor `agents.defaults.heartbeat.model` instead of re-resolving to the default primary model.

## Best Possible Solution

Add one heartbeat-origin contract for follow-up runs, then prove exec-approval, queued/retry, and fallback follow-ups keep the heartbeat model override without reintroducing prior heartbeat model bleed into normal user turns.

## Reproduction Assessment

No live current-main run was performed, so there is not an end-to-end runtime reproduction from this review. Source inspection gives a high-confidence path: exec-approval follow-ups re-enter the generic `agent` RPC without heartbeat run-kind or heartbeat model metadata, while the override is only applied when `opts.isHeartbeat` is true.

## Solution Assessment

Yes, the best fix is a narrow contract repair: carry heartbeat origin/model selection through follow-up construction and gateway agent ingress, with regressions for exec-approval and queued embedded follow-ups. A new config option is not needed.
