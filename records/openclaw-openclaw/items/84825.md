## Summary

Keep open: current main still depends on `opts.isHeartbeat` for the heartbeat model override, and the queued embedded follow-up path still emits `trigger: "user"`; the related merged fixes cover initial heartbeat model selection and stale fallback state, not this follow-up leak.

## What This Changes

The issue asks OpenClaw to preserve `agents.defaults.heartbeat.model` for follow-up runs spawned inside a heartbeat session.

## Best Possible Solution

Make heartbeat origin durable run metadata for every sub-run and queued follow-up, then assert that heartbeat trigger, model override, usage persistence, and fallback handling stay heartbeat-scoped for the whole heartbeat session.

## Reproduction Assessment

Yes at source level: the reporter supplied a concrete live trajectory, and current main still has a heartbeat override gated on `opts.isHeartbeat` plus a queued embedded follow-up path that hard-codes `trigger: "user"`. I did not execute a live heartbeat run in this read-only review.

## Solution Assessment

Yes, the narrow fix is to thread heartbeat context through follow-up run construction/execution or persist it as run/session metadata; adding a new config knob would not address the broken existing `heartbeat.model` contract.
