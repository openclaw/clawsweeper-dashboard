## Summary

Keep open: current `main` still applies `heartbeat.model` only when the run is explicitly marked as heartbeat, while gateway-created exec-approval follow-up agent runs do not carry a heartbeat-origin/model contract. The related timeout amplifier at https://github.com/openclaw/openclaw/issues/85841 is now closed as implemented, but that does not prove this model-selection leak is fixed.

## What This Changes

The issue reports that follow-up turns inside a heartbeat-originated session can re-resolve to `agents.defaults.model.primary` instead of the configured `agents.defaults.heartbeat.model`.

## Best Possible Solution

Define a single heartbeat-origin/model-selection contract for heartbeat-created follow-up runs, then add regression coverage proving those follow-ups use `heartbeat.model` without making the override sticky for later user turns.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. The source path is reproducible enough to keep open: heartbeat model selection still depends on `opts.isHeartbeat`, while gateway-created follow-up agent runs do not carry that heartbeat model contract.

## Solution Assessment

Yes, the best fix direction is a narrow contract repair, not a new config option: carry heartbeat origin and resolved heartbeat model through follow-up entry points and cover both the heartbeat-follow-up and later-user-turn cases.
