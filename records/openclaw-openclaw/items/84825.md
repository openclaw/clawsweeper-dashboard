## Summary

Keep open: the heartbeat model override contract is documented and wired for the initial run, but current main still lacks proof that heartbeat origin and model selection survive queued/retry follow-up runs. The related closed heartbeat-model issues cover adjacent initial-turn, opposite-bleed, and fallback cases, not this follow-up-origin loss.

## What This Changes

The issue reports that heartbeat follow-up or retry turns in the same heartbeat session resolve to `agents.defaults.model.primary` instead of `agents.defaults.heartbeat.model` after the initial heartbeat turn.

## Best Possible Solution

Add a focused regression for heartbeat-originated follow-up or retry turns, then carry heartbeat origin and resolved heartbeat model through that follow-up contract without persisting it into normal user turns.

## Reproduction Assessment

No, not at high confidence from this read-only review. The issue provides detailed live traces and current source shows the heartbeat model depends on `isHeartbeat` being retained, but I did not run a live heartbeat session or create a failing current-main test.

## Solution Assessment

Yes, the best direction is a narrow contract repair: preserve heartbeat origin and resolved heartbeat model through queued/retry/fallback follow-up execution and test the earlier opposite-direction heartbeat model bleed cases at the same time.
