## Summary

Keep open: current main still has a source-level gap where heartbeat-originated follow-ups can enter the gateway `agent` path without `isHeartbeat` or a heartbeat model override, so they re-resolve from normal session/default model state. The closed related approval-timeout issue at https://github.com/openclaw/openclaw/issues/85841 reduces one amplifier, but it does not close this model-routing bug.

## What This Changes

The issue requests that heartbeat-originated follow-up/retry runs continue to honor `agents.defaults.heartbeat.model` instead of re-resolving from the agent primary model.

## Best Possible Solution

Add a heartbeat-origin/model-selection contract that gateway and queued follow-ups can carry, with regressions proving heartbeat follow-ups use `heartbeat.model` while later user turns keep the primary model.

## Reproduction Assessment

Yes at source level: dispatch a gateway `agent` follow-up for a heartbeat session/channel without explicit provider/model override, and current main reaches `agentCommandFromIngress` without heartbeat model selection. I did not run a live heartbeat gateway scenario in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow contract repair that carries heartbeat origin and model selection through follow-up dispatch. A new config option or startup migration is not needed.

Current main honors `agents.defaults.heartbeat.model` for the initial heartbeat reply via `isHeartbeat` and `heartbeatModelOverride`, but gateway-dispatched follow-up runs such as exec-approval follow-ups can call the `agent` path with a heartbeat session/channel and no heartbeat origin/model metadata, causing `agentCommandFromIngress` to resolve the normal primary model.
