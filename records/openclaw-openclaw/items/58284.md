## Summary

Keep open: current main and the latest release still only support TTL-gated cache pruning, so active sessions that continuously refresh the cache marker still have no rolling or turn-age pruning trigger. The request remains valid but needs maintainer API direction because it adds core context-pruning behavior and public config surface.

## What This Changes

Requested behavior is rolling or turn-age-based context pruning so active sessions can trim stale tool results without waiting for cache TTL inactivity.

## Best Possible Solution

Choose one additive, opt-in active-session pruning contract, then implement it in the current `agent-hooks` context-pruning path with aligned schema, docs, focused tests, and unchanged defaults for existing `cache-ttl` users.

## Reproduction Assessment

Yes, at source level: cache-TTL mode reads a cache touch timestamp and returns while the TTL window is fresh, and current settings/schema accept no rolling or turn-age pruning option. I did not run a live Opus cost reproduction because the missing trigger is clear in source.

## Solution Assessment

Unclear until maintainers choose the public trigger API. The narrowest maintainable direction is one additive active-session pruning policy in the current context-pruning runtime, not reviving the closed stale PR as-is.
