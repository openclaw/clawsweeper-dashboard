## Summary

Keep open: current main and v2026.6.8 still expose only TTL-gated `cache-ttl` tool-result pruning, while the requested turn-based or rolling pruning trigger remains missing and needs maintainer product/config direction.

## What This Changes

Request to add turn-based or rolling context pruning so stale tool results can be trimmed every turn even when the cache TTL has not expired.

## Best Possible Solution

Choose one canonical active-session pruning trigger contract, then implement it in the active context-pruning runtime with aligned config schema, docs, and focused tests while preserving existing `cache-ttl` defaults.

## Reproduction Assessment

Yes, at source level. Current main accepts only `off` and `cache-ttl`, installs pruning only for `cache-ttl`, and returns inside the fresh TTL window before pruning, so the active-session rolling trigger is absent.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers should choose one public trigger/config contract before adding a new mode or nested option.
