## Summary

Keep open. Current main still implements only TTL-gated cache-ttl pruning, active turns inside the TTL window still skip the pruning pass, and the related rolling-mode PR remains open but stale against the active `pi-hooks` path and needs maintainer API direction.

## What This Changes

The issue requests turn-count or rolling context pruning so active sessions can prune old tool results before the cache TTL expires.

## Best Possible Solution

Choose one active-session pruning trigger API, then implement it in the active `pi-hooks` context-pruning path with aligned config types/schema, generated and public docs, changelog, and focused tests for consecutive active turns inside the TTL window.

## Reproduction Assessment

Yes, at source level: configure `contextPruning.mode: "cache-ttl"`, continue sending turns inside `ttlMs`, and the context hook returns before `pruneContextMessages` while cache-touch timestamps continue to refresh. I did not run a live session because this review is read-only.

## Solution Assessment

Unclear. The requested direction belongs in core because pruning runs during context assembly, but maintainers still need to choose whether the public contract should be turn-age, top-level rolling mode, or a threshold bypass before implementation.
