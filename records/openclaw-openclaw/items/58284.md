## Summary

Keep open. Current main and v2026.5.18 still only accept and wire `cache-ttl` pruning, successful active turns refresh the cache-TTL timestamp, and the hook returns inside `ttlMs` before pruning; the earlier rolling-mode PR was closed unmerged, so the remaining work is a real core config/API decision.

## What This Changes

The issue requests turn-based or rolling active-session context pruning so old tool results can be trimmed during continuous conversations instead of only after cache-TTL inactivity.

## Best Possible Solution

Choose one active-session pruning trigger contract, then implement it as an opt-in core pruning mode/knob in the active `pi-hooks` path with aligned config types, strict schema, docs, changelog, and regression coverage for continuous turns inside the TTL window.

## Reproduction Assessment

Yes, at source level. With `contextPruning.mode: "cache-ttl"`, successful active turns append fresh cache-TTL timestamps and the current hook returns before pruning while that timestamp is inside `ttlMs`.

## Solution Assessment

Unclear. The problem belongs in core because pruning runs during context assembly, but maintainers still need to choose the public trigger API and avoid silently changing existing `cache-ttl` semantics.
