## Summary

Keep open. Current main still only supports TTL-gated `cache-ttl` pruning, fresh TTL windows return before pruning, and the related rolling-mode PR is open/unmerged with stale-path review concerns, so this needs maintainer API direction rather than cleanup closure.

## What This Changes

Request adds turn-count or rolling pruning so old tool results can be trimmed during active sessions instead of only after cache-TTL inactivity.

## Best Possible Solution

Choose one active-session pruning trigger API, then implement it in the active `pi-hooks` context-pruning path with aligned config types, strict schema, generated/public docs, changelog, and regression tests for continuous turns inside the TTL window.

## Reproduction Assessment

Yes, at source level. Configure `contextPruning.mode: "cache-ttl"` and continue sending eligible turns inside `ttlMs`; current main returns before pruning while the fresh cache-touch timestamp keeps the TTL window alive.

## Solution Assessment

Unclear. The request belongs in core because pruning runs inside context assembly, but maintainers still need to choose the public trigger API across rolling, turn-age, and threshold-bypass designs.
