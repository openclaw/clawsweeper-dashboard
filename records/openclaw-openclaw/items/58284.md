## Summary

Keep open. Current main and v2026.5.22 still only support TTL-gated `cache-ttl` pruning, active successful turns can refresh the cache-touch timestamp, and the earlier rolling-mode PRs were closed unmerged or stale, so the remaining work is a real public config/API decision.

## What This Changes

Request an additive rolling or turn-age pruning trigger so old tool results can be trimmed during continuously active sessions instead of only after cache TTL expiry.

## Best Possible Solution

Choose one additive active-session pruning trigger contract, then implement it in the active `pi-hooks` path with aligned config types, strict schema, docs, changelog, and regression coverage for continuous turns inside the TTL window.

## Reproduction Assessment

Yes, at source level for the feature gap. With `contextPruning.mode: "cache-ttl"`, active successful turns can append fresh cache-TTL timestamps, and the current hook returns before pruning while that timestamp is inside `ttlMs`.

## Solution Assessment

Unclear until maintainers choose the API. The problem belongs in core because pruning runs during context assembly, but a safe fix needs an additive public trigger contract rather than silently changing existing `cache-ttl` semantics.
