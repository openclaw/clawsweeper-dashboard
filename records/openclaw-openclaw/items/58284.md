## Summary

Keep open. Current main and the latest tagged release still expose only TTL-gated `cache-ttl` pruning, so active sessions inside the TTL window still skip the pruning pass; the related rolling-mode PR remains open and needs current-path/API review rather than cleanup closure.

## What This Changes

Request turn-based or rolling context pruning so old tool results can be trimmed during continuously active sessions instead of only after cache TTL inactivity.

## Best Possible Solution

Choose one active-session pruning trigger API, then implement it in the active `pi-hooks` context-pruning path with aligned config types/schema, generated and public docs, changelog, and focused tests for consecutive active turns inside the TTL window.

## Reproduction Assessment

Yes, at source level. Configure `contextPruning.mode: "cache-ttl"`, keep sending turns inside `ttlMs`, and the context hook returns before `pruneContextMessages()` while cache-touch timestamps continue to refresh.

## Solution Assessment

Unclear. The request belongs in core because pruning runs during context assembly, but maintainers still need to choose the public contract between turn-age pruning, a rolling mode, or a threshold-based TTL bypass.
