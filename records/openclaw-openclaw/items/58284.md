## Summary

Keep open. Current main and v2026.6.6 still only accept and wire TTL-gated `cache-ttl` pruning, while this issue is now the canonical active-session rolling/turn-age pruning request that needs a maintainer product/config decision.

## What This Changes

Request an opt-in rolling or turn-age context-pruning trigger so old tool results can be trimmed during continuously active sessions, not only after cache TTL inactivity.

## Best Possible Solution

Choose one additive, opt-in active-session pruning trigger contract, then implement it in the active context-pruning runtime with aligned schema, docs, and focused tests while preserving `cache-ttl` as the default behavior.

## Reproduction Assessment

Yes, at source level. Current main only accepts and wires `cache-ttl`, appends fresh cache-touch markers after eligible attempts, and exits the pruning hook while the TTL window is fresh; this proves the requested active-session trigger is absent.

## Solution Assessment

Unclear until maintainers choose the API. A rolling or turn-age trigger is plausible, but the maintainable fix should be additive, preserve current defaults, and avoid splitting the same pruning decision across competing config knobs.
