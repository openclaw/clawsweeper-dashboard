## Summary

Keep open: current main and v2026.6.8 still only support TTL-gated cache pruning, so active sessions that keep refreshing the cache marker still lack rolling or turn-age pruning. The request is valid, but it needs maintainer API direction because it adds core context-pruning behavior and public config surface.

## What This Changes

Request an additive rolling or turn-age pruning trigger so old tool results can be trimmed during continuously active sessions, not only after `contextPruning.ttl` expires.

## Best Possible Solution

Choose one additive, opt-in active-session pruning contract and implement it in the existing context-pruning runtime with aligned schema, docs, and focused regression tests while preserving current defaults.

## Reproduction Assessment

Yes, at source level: current main only accepts and wires `cache-ttl`, and the hook returns during a fresh TTL window before pruning. I did not run a live Opus cost reproduction because the missing active-session trigger is clear in source.

## Solution Assessment

Unclear until maintainers choose the public trigger API. The narrow maintainable direction is one additive active-session pruning policy in the current context-pruning runtime, not reviving a closed stale PR as-is.
