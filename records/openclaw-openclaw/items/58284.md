## Summary

Keep open. Current main and the latest release still only support TTL-gated `cache-ttl` pruning, while this issue asks for an active-session rolling or turn-age trigger; the prior rolling-mode PRs were closed unmerged, so the remaining work is a real product/config decision.

## What This Changes

Request an active-session context-pruning policy that trims old tool results by turn age or a rolling window instead of only after cache-TTL inactivity.

## Best Possible Solution

Ship one additive, opt-in active-session pruning policy with schema/types/docs and regression tests proving existing `cache-ttl` behavior remains the default.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `cache-ttl` pruning exits while the last cache-touch marker is within TTL, and eligible successful attempts append fresh markers that keep active sessions from crossing the idle threshold.

## Solution Assessment

Unclear until maintainers choose the API. The issue identifies a real gap, but the best fix could be a rolling mode, a nested `rollingPrune` option, or a pressure-based TTL bypass, and it should preserve existing defaults.
