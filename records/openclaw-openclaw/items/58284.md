## Summary

Keep open. Current main still exposes only `off` and `cache-ttl`, the active pruning hook still skips pruning while the cache TTL window is fresh, and the linked rolling-pruning PR is open but unmerged and may need API-shape review before it satisfies the exact turn-age request.

## What This Changes

The issue requests turn-based or rolling context pruning so active sessions can prune old tool results even when cache-TTL inactivity never expires.

## Best Possible Solution

Keep this issue open until a current-main implementation lands that adds a documented rolling or turn-age pruning trigger in the active `src/agents/pi-hooks/context-pruning` path, updates config types/schema/generated docs/changelog, and includes focused tests for pruning on active consecutive turns without waiting for TTL expiry.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure `agents.defaults.contextPruning.mode: "cache-ttl"`, keep appending eligible attempts inside the TTL window, and the context hook returns before `pruneContextMessages()` because `Date.now() - lastTouch < ttlMs`; current schema also rejects any rolling mode.

## Solution Assessment

Unclear. The requested direction is valid and belongs in core because pruning happens during prompt/context assembly, but maintainers still need to choose between the issue's nested `rollingPrune.keepLastTurns` API and the linked PR's simpler `mode: "rolling"` approach.
