## Summary

Keep open. Current main and v2026.6.5 still only support TTL-gated `cache-ttl` pruning; source inspection shows active successful turns can refresh the cache-touch marker, and the linked rolling-mode PR was closed unmerged, so this remains a real product/config decision.

## What This Changes

The issue requests an opt-in rolling or turn-age context-pruning trigger so active sessions prune older tool results without waiting for cache TTL inactivity.

## Best Possible Solution

Add a single additive, opt-in context-pruning policy for active sessions, with schema/types/docs and focused tests proving default `cache-ttl` compatibility plus the new rolling or turn-age behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `cache-ttl` pruning exits while the last cache-touch marker is within TTL, and successful eligible attempts append fresh markers. I did not run a live cost-heavy session, but the current code path matches the reported limitation.

## Solution Assessment

Unclear until maintainers choose the API. The issue identifies a real gap, but the best fix could be a rolling/turn-age mode, a pressure-based TTL bypass, or a context-pruning API extension; it should be additive and preserve existing `cache-ttl` behavior by default.
