## Summary

Keep open: current main and v2026.6.10 still only support TTL-gated cache pruning, so continuous active sessions still lack a rolling or turn-age trigger. The issue is valid, but it needs maintainer API direction because it adds public context-pruning config and changes cost-versus-context-retention behavior.

## What This Changes

The issue requests a turn-count or rolling pruning trigger so active sessions can trim old tool results without waiting for `contextPruning.ttl` inactivity.

## Best Possible Solution

Choose one additive active-session pruning contract and implement it in the current context-pruning runtime with aligned schema, docs, and regression proof while preserving existing cache-ttl defaults.

## Reproduction Assessment

Yes, at source level: current main only accepts and wires `cache-ttl`, and the context hook returns during a fresh TTL window before pruning. A live Opus cost reproduction is not needed to prove the missing active-session trigger.

## Solution Assessment

Unclear until maintainers choose the public trigger API. The narrow maintainable direction is one additive active-session pruning policy in the current context-pruning runtime, not reviving a closed stale PR as-is.
