## Summary

Keep open. Current main and the latest shipped release still only accept and wire `cache-ttl` pruning, fresh TTL windows still return before pruning, and the prior rolling-mode PR was closed unmerged, so the remaining work is a real core feature/API decision rather than a cleanup close.

## What This Changes

The issue requests active-session context pruning that can trim old tool results by turn age or rolling cadence instead of waiting for a cache-TTL idle window.

## Best Possible Solution

Choose one active-session pruning trigger contract, then implement it in the active `pi-hooks` context-pruning path with aligned config types, strict schema, generated/public docs, changelog, and regression tests for continuous turns inside the TTL window.

## Reproduction Assessment

Yes, at source level. Configure `contextPruning.mode: "cache-ttl"` and keep sending turns inside `ttlMs`; current main returns before pruning while the fresh cache-touch timestamp keeps the TTL window alive.

## Solution Assessment

Unclear. The request belongs in core because pruning runs inside context assembly, but maintainers still need to choose the public trigger API across rolling mode, turn age, and high-usage TTL bypass designs.
