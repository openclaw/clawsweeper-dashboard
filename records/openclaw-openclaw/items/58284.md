## Summary

Keep open. Current main and v2026.5.19 still only accept and wire TTL-gated `cache-ttl` pruning, active successful turns can keep the cache timestamp fresh, and the earlier rolling-mode PR was closed unmerged, so the remaining work is a real public config/API decision.

## What This Changes

Request an opt-in turn-based or rolling pruning trigger so active sessions can trim older tool results without waiting for a cache-TTL idle gap.

## Best Possible Solution

Choose one additive active-session pruning trigger contract, then implement it in the active `pi-hooks` path with aligned config types, strict schema, docs, changelog, and regression coverage for continuous turns inside the TTL window.

## Reproduction Assessment

Yes, at source level. With `contextPruning.mode: "cache-ttl"`, active successful turns can append fresh cache-TTL timestamps, and the current hook returns before pruning while that timestamp is inside `ttlMs`.

## Solution Assessment

Unclear. The problem belongs in core because pruning runs during context assembly, but maintainers still need to choose the public trigger API and avoid silently changing existing `cache-ttl` semantics.
