## Summary

Keep open. Current main and v2026.5.22 still expose only TTL-gated `cache-ttl` pruning, and the closed rolling-mode PR did not land, so the active-session pruning request remains a real public config/API decision.

## What This Changes

The issue requests turn-based or rolling context pruning so active sessions can trim old tool results without waiting for cache-TTL inactivity.

## Best Possible Solution

Choose one additive active-session pruning trigger contract, then implement it in the active `pi-hooks` path with aligned config types, strict schema, docs, changelog, and regression coverage for continuous turns inside the TTL window.

## Reproduction Assessment

Yes, at source level. With `contextPruning.mode: "cache-ttl"`, completed active turns can append fresh cache-TTL timestamps, and the current hook returns before pruning while that timestamp is inside `ttlMs`.

## Solution Assessment

Unclear. The problem belongs in core because pruning runs during context assembly, but maintainers still need to choose the public trigger API and avoid silently changing existing `cache-ttl` semantics.
