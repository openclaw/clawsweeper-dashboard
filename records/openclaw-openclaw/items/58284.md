## Summary

Keep open. Current main and v2026.6.6 still only support TTL-gated `cache-ttl` context pruning, while this issue asks for an active-session turn-age or rolling trigger; the related rolling-mode PRs were closed unmerged, so the remaining work is a real product/config decision.

## What This Changes

Request an additive active-session context-pruning trigger that soft-trims or hard-clears older tool results by turn age instead of only after cache TTL inactivity.

## Best Possible Solution

Choose one additive, opt-in active-session pruning policy, then implement it in the core context-pruning path with aligned schema/types/docs and regression tests that preserve existing `cache-ttl` defaults.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `cache-ttl` pruning exits while the last cache-touch marker is within TTL, and eligible active attempts append fresh markers that keep sessions inside that gate.

## Solution Assessment

Unclear until maintainers choose the API. The issue identifies a real gap, but the best fix could be a rolling mode, a nested `rollingPrune` option, or a pressure-based TTL bypass, and it should preserve existing defaults.
