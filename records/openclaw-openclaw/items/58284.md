## Summary

Keep open: current main and the latest release still expose only TTL-gated context pruning, while this request asks for a new rolling or turn-age pruning trigger. The related open PR is relevant, but prior review found it stale against the active `pi-hooks` path, so the remaining work needs maintainer API direction rather than cleanup closure.

## What This Changes

The issue requests a rolling or turn-age context-pruning option so active sessions can trim old tool results every turn instead of waiting for cache TTL expiry.

## Best Possible Solution

Choose one additive active-session pruning API, then implement it in the active `pi-hooks` runtime with aligned config types, strict schema, generated/public docs, changelog, and focused tests; if `rolling` is the chosen API, repair or replace https://github.com/openclaw/openclaw/pull/43238 against current main.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request for a new pruning trigger/config surface. Source inspection does give high-confidence current behavior proof: current main and `v2026.5.7` only support `off | cache-ttl`, and the runtime returns before pruning during a fresh TTL window.

## Solution Assessment

Unclear. The turn-age/rolling direction is plausible and the existing pruner already has a recent-assistant cutoff, but maintainers should choose between the related rolling, nested turn-age, and `forcePruneRatio` APIs before implementation.
