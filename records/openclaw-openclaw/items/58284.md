## Summary

Keep open. Current main still only exposes TTL-gated cache pruning, the fresh-TTL path still returns before pruning, and the related rolling-mode PR remains open but stale against the active implementation path.

## What This Changes

Request adds turn-age or rolling pruning so active sessions can trim older tool results every turn instead of waiting for cache-ttl inactivity.

## Best Possible Solution

Choose one active-session pruning trigger API, then implement it in the active pi-hooks context-pruning path and embedded factory with aligned config types, strict schema, generated/public docs, changelog, and regression tests for continuous turns inside the TTL window.

## Reproduction Assessment

Yes, at source level. Configure cache-ttl pruning, keep sending eligible turns inside ttlMs, and current main returns before pruneContextMessages while fresh cache-touch timestamps continue to be appended.

## Solution Assessment

Unclear. The direction belongs in core because pruning runs inside context assembly, but maintainers still need to choose between turn-age pruning, rolling mode, and threshold bypass before implementation.
