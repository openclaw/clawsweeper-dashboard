## Summary

Keep open: current main and v2026.6.8 still only support TTL-gated cache pruning, so active sessions that keep refreshing the cache marker still have no rolling or turn-age pruning trigger. The request remains valid, but it needs maintainer API direction because it adds core context-pruning behavior and public config surface.

## What This Changes

The issue requests turn-based or rolling context pruning so active sessions can trim old tool results without waiting for cache TTL inactivity.

## Best Possible Solution

Choose one additive, opt-in active-session pruning contract and implement it in the existing context-pruning runtime with aligned schema, docs, and focused regression tests while preserving current defaults.

## Reproduction Assessment

Yes, at source level: cache-TTL mode reads a cache touch timestamp and returns while the TTL window is fresh, and current settings/schema accept no rolling or turn-age pruning option. I did not run a live Opus cost reproduction because the missing trigger is clear in source.

## Solution Assessment

Unclear until maintainers choose the public trigger API. The narrow maintainable direction is one additive active-session pruning policy in the current context-pruning runtime, not reviving a closed stale PR as-is.
