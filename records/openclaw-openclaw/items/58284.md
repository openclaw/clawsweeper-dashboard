## Summary

Keep open: current main and v2026.6.8 still only wire `cache-ttl` pruning, with no rolling, turn-age, or `rollingPrune.keepLastTurns` trigger in settings, schema, factory wiring, docs, or tests. The request is a useful core context-pruning feature, but it adds public config and trigger semantics, so it needs maintainer API direction rather than cleanup closure.

## What This Changes

Request to add turn-based or rolling context pruning so old tool results are trimmed during continuously active sessions instead of only after the cache TTL expires.

## Best Possible Solution

Choose one canonical active-session pruning trigger, keep `cache-ttl` behavior compatible by default, then implement the approved config/runtime/docs/tests in the active `agent-hooks` and embedded-runner paths.

## Reproduction Assessment

Yes, at source level: current main rejects non-`cache-ttl` pruning modes and skips pruning while the cache TTL window is fresh. I did not run a live cost reproduction because the requested trigger surface is absent in source.

## Solution Assessment

Unclear until maintainers choose the public API. A rolling trigger is plausible, but the closed implementation attempt is not enough because the active path, docs, tests, provider/cache semantics, and config contract still need design.
