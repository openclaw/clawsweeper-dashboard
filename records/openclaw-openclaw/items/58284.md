## Summary

Keep open: current main and the latest release still only support TTL-gated cache pruning, so active sessions that keep refreshing the cache marker have no turn-age or rolling pruning trigger. The remaining work is valid but needs maintainer API direction because it adds core context-pruning behavior and configuration surface.

## What This Changes

Request an opt-in rolling or turn-age pruning mode that trims older tool results every turn instead of waiting for cache-TTL inactivity.

## Best Possible Solution

Choose one additive, opt-in active-session pruning contract, then implement it in the active `agent-hooks` context-pruning path with aligned config schema, docs, and focused tests while preserving current `cache-ttl` defaults.

## Reproduction Assessment

Yes, at source level: successful cache-TTL turns append fresh cache markers, the hook returns while the TTL window is fresh, and current settings/schema accept no rolling or turn-age pruning option. I did not run a live Opus cost reproduction because the missing trigger is clear in source.

## Solution Assessment

Unclear as an implementation until maintainers choose the public trigger API. The narrowest maintainable direction is one additive active-session pruning policy in the existing context-pruning runtime, not reviving the closed stale PR as-is.
