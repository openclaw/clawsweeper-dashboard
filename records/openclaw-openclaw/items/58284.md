## Summary

Keep open. Current main and v2026.6.6 still only support TTL-gated `cache-ttl` pruning, while this issue asks for an active-session rolling or turn-age trigger; the prior rolling-mode PRs were closed unmerged, so the remaining work is a real product/config decision.

## What This Changes

Request an opt-in turn-age or rolling context-pruning policy that can trim old tool results during continuously active sessions instead of waiting for cache TTL idle time.

## Best Possible Solution

Ship one additive, opt-in active-session pruning policy with canonical schema/types/docs and focused tests proving existing `cache-ttl` behavior remains the default.

## Reproduction Assessment

Yes, at source level. Current main only accepts and installs `cache-ttl`, refreshes cache-touch markers after eligible attempts, and exits the context-pruning hook while the TTL window is fresh.

## Solution Assessment

Unclear until maintainers choose the API. The issue identifies a real gap, but the best fix could be a rolling/turn-age mode, a pressure-based TTL bypass, or a context-pruning API extension; it should be additive and preserve existing defaults.
