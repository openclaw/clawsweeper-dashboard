## Summary

Keep open: current main and v2026.6.10 still only support TTL-gated cache pruning, so continuous active sessions still lack a rolling or turn-age trigger. The request is valid, but it needs maintainer API direction because it adds public context-pruning config and trigger behavior.

## What This Changes

Request add an opt-in rolling or turn-based context-pruning trigger so old tool results can be trimmed during active sessions instead of only after cache-TTL inactivity.

## Best Possible Solution

Choose one additive active-session pruning contract and implement it in the current context-pruning runtime with aligned schema, docs, and regression proof while preserving current cache-ttl defaults.

## Reproduction Assessment

Yes, at source level: current main only accepts and wires cache-ttl, and the hook returns during a fresh TTL window before pruning. A live Opus cost reproduction is not needed to prove the missing trigger.

## Solution Assessment

Unclear until maintainers choose the public trigger API. The narrow maintainable direction is one additive active-session pruning policy in the current context-pruning runtime, not reviving a closed stale PR as-is.
