## Summary

Keep this PR open for review, but the current patch is not safe to merge: the path-only cache is shared between incompatible JSON reader contracts and can also bypass per-call fs-safe validation. The `maintainer` label and MEMBER author association also make this ineligible for conservative cleanup close.

## What This Changes

This PR adds a process-scoped sync JSON read cache in `src/infra/json-files.ts` and routes `json-file` and package-update sync reads through it.

## Best Possible Solution

Keep the performance goal, but move caching to an explicit plugin metadata snapshot or make the fs-layer cache reader-contract-aware, option-aware, and file-identity-revalidated before it can affect root-safe plugin reads.

## Reproduction Assessment

Yes. From source inspection, reading a file first through `tryReadJsonSync(path)` and then through `readRootJsonObjectSync({ rootDir, relativePath })` on the same resolved path returns the cached raw JSON to the structured-result API; I did not execute tests because this review was read-only.

## Solution Assessment

No. A process-wide path-only cache below fs-safe is not the narrowest maintainable fix because it conflates reader contracts and bypasses per-call validation; a snapshot-level cache or validated reader-specific cache is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep root-safe reads out of a path-only cache:** `src/infra/json-files.ts:75-77`
  - body: A cache entry written by `tryReadJsonSync` is raw JSON, while `readRootJsonObjectSync` returns a structured `{ ok, value, stat, path, rootRealPath }` result. Because both APIs share the same `path.resolve` key, whichever reader touches a file first controls the other reader's return shape, and root-safe cache hits also skip `rejectHardlinks`, `rootRealPath`, and `maxBytes` validation. Split the cache by reader contract and revalidate root reads, or leave `readRootJsonObjectSync` uncached.
  - confidence: 0.94
- **[P2] Clear symlink aliases after resolved writes:** `src/infra/json-files.ts:91-99`
  - body: `loadJsonFile(linkPath)` caches under the link path, but `saveJsonFile(linkPath)` resolves the symlink and calls `writeJsonSync(targetPath)`, so this invalidates only the target key. A later load through the link can return the pre-write JSON; add alias invalidation or avoid caching paths that may be symlink aliases.
  - confidence: 0.88
