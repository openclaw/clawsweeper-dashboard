## Summary

Keep open. The MS Teams outbound hook gap is real and the plugin-code direction is useful, but the PR still includes an unrelated fork-only ECR publishing workflow, lacks after-fix real MS Teams proof, and drops the available session key before plugin `message_sent` observers see the new event.

## What This Changes

Adds MS Teams outbound `message:sent`/`message_sent` hook emission after reply delivery, dispatcher tests for success/failure cases, and an EVA fork Docker/ECR publishing workflow.

## Best Possible Solution

Keep the scoped MS Teams hook wiring, remove the fork-only workflow, pass the available session key into the canonical sent context, and require redacted live Teams delivery/log proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main documents `message:sent` and Telegram emits it, while the MS Teams reply dispatcher sends or retries replies without any sent-hook path. I did not run a live Teams delivery repro in this read-only review.

## Solution Assessment

No, not as submitted. The hook wiring is the right direction, but the fork-only workflow must be removed, the available session key should be passed through the plugin sent context, and after-fix live Teams proof is needed.
