## Summary

Keep open. Current main still routes opencode-go through the passthrough Gemini replay family without a `dropReasoningFromHistory` policy, and the linked open PR https://github.com/openclaw/openclaw/pull/82052 uses closing syntax for this issue, so the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue asks OpenClaw to stop opencode-go passthrough OpenAI-compatible replay from resending assistant `reasoning` fields that strict completions endpoints reject.

## Best Possible Solution

Land or reject https://github.com/openclaw/openclaw/pull/82052 after maintainer review and proof, keeping the replay-policy fix in the shared passthrough helper rather than adding an opencode-go-only workaround.

## Reproduction Assessment

Yes at source level: opencode-go resolves through the passthrough replay helper, and that helper cannot set `dropReasoningFromHistory` for OpenAI-compatible completions. A fresh live 400 reproduction still needs opencode-go credentials and was not run.

## Solution Assessment

Yes. The linked PR's direction is the narrow maintainable fix because it reuses the existing replay-policy flag and passes existing `modelApi` context through the shared provider-family helper.
