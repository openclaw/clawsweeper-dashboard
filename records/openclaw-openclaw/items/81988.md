## Summary

Keep open: current main still routes OpenCode Go through a passthrough replay policy that does not drop OpenAI-completions reasoning history, and pi-ai serializes a `thinkingSignature` value such as `reasoning` back into a top-level assistant-message field that strict upstreams can reject.

## What This Changes

The issue asks OpenClaw to stop `opencode-go` OpenAI-completions replay from sending stored assistant `reasoning` fields that upstream rejects during multi-turn tool runs.

## Best Possible Solution

Add a focused OpenCode Go/OpenAI-completions replay sanitizer that removes unsupported `reasoning` history before pi-ai payload conversion while preserving Gemini passthrough signature sanitation and direct Moonshot `reasoning_content` behavior.

## Reproduction Assessment

Yes, source-level. The current `opencode-go` provider policy omits reasoning stripping, and pi-ai converts a retained `thinkingSignature: "reasoning"` block into an outgoing assistant `reasoning` field; no live credentialed run was performed.

## Solution Assessment

Unclear whether the exact one-line suggested fix is sufficient, because the existing `dropReasoningFromHistory` helper preserves the active tool-call continuation. The best fix should be verified at the outgoing payload boundary with a regression for the reported tool-use replay shape.
