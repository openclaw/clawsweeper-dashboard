## Summary

Keep open. The PR targets real gateway status/restart gaps that current main has not fully closed, but the branch is stale against current probe semantics and still has a blocking `--require-rpc` regression plus security-sensitive auth/transport concerns.

## What This Changes

The PR changes daemon gateway status and restart-health probing to use a health-only probe, enforce explicit auth for daemon URL overrides, bound local restart probe timeouts, and extract explicit-auth helpers.

## Best Possible Solution

Rebase and narrow the branch around current main: preserve `--require-rpc` status-read proof and current non-mutating device identity policy while adding only the missing strict override-auth boundary, health-only daemon probe behavior, and bounded restart timeout logic with focused tests.

## Reproduction Assessment

Yes. Source and test inspection gives a high-confidence path: current main still has the fixed 3s restart probe timeout and non-strict daemon status URL overrides, while current docs/tests also prove the `--require-rpc` read-probe behavior that this PR would regress.

## Solution Assessment

No, not in the current shape. The direction is plausible, but the PR needs a rebase and security reconciliation before it is the narrowest maintainable fix.
