## Summary

Keep open. The PR is still useful but not mergeable as-is: it would regress `--require-rpc` read-probe semantics, lacks the required changelog entry, and needs credential/transport probe security reconciliation against newer main changes.

## What This Changes

The PR changes daemon gateway status and restart-health probes to use a health-only `probeGateway` path, enforce explicit auth for URL overrides, bound local restart probe timeouts, and extract explicit-auth helpers with tests.

## Best Possible Solution

Rebase and narrow the work so `--require-rpc` keeps real `status` RPC proof, current non-mutating probe identity and strict auth boundaries are preserved, and only the missing health-only daemon probe plus bounded restart timeout behavior lands with tests and a changelog entry.

## Reproduction Assessment

Yes. Source and test inspection gives a high-confidence path: a gateway that accepts health/connect but breaks `status` would pass the PR's daemon status path even though current docs/tests require `--require-rpc` to prove read-scope RPC health.

## Solution Assessment

No. The direction is plausible, but the current patch is not the narrowest maintainable fix because it regresses an existing CLI contract and needs security reconciliation with newer main behavior.
