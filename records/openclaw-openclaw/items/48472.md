## Summary

Keep open. Current main has absorbed some adjacent gateway probe work, including preserved `--require-rpc` read semantics and merged restart-probe auth fixes, but this PR still has unmerged timeout/auth-boundary pieces and is not merge-ready because it regresses `--require-rpc` and needs security reconciliation.

## What This Changes

The PR changes daemon gateway status/restart probing to use lighter health/connect probes, enforce explicit auth for URL overrides, increase and bound local restart probe timeouts, and extract explicit-auth helpers.

## Best Possible Solution

Rebase and narrow the branch: preserve current `--require-rpc` read proof and probe identity policy, then add strict CLI override-auth enforcement plus a bounded local restart handshake budget with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Source/docs inspection gives a high-confidence path: current main documents and tests `--require-rpc` as a real read probe, while the PR diff removes that branch and current main still shows the fixed 3s restart probe timeout.

## Solution Assessment

No. The direction is plausible, but the current patch is not the narrowest maintainable solution because it mixes stale status-probe behavior with security-sensitive identity/auth changes that need maintainer review.

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff changes gateway credential, transport, and device-identity behavior on probe paths.

Concerns:

- **[high] Reconcile probe device identity fallback:** `src/gateway/probe.ts:88`
  - body: The PR disables device identity only for unauthenticated loopback probes, leaving non-loopback probes without explicit shared auth on the default `GatewayClient` identity path. `GatewayClient` treats undefined identity as load/create and can send a stored device token in connect auth, so this needs explicit trust-boundary review before merge.
  - confidence: 0.78
- **[medium] Decide private-ws hostname policy for the new probe guard:** `src/gateway/net.ts:475`
  - body: The new daemon probe guard relies on `isSecureWebSocketUrl(... allowPrivateWs)`, while the shared helper accepts hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`. Maintainers should confirm or tighten that transport policy before using it as the hardening boundary for this path.
  - confidence: 0.72
