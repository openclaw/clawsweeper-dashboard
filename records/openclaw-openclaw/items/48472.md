## Summary

Keep open. The PR is useful source work, but current main has partially superseded its auth/probe base while the PR still introduces a blocking `--require-rpc` regression and touches security-sensitive gateway auth, transport, and device-identity behavior that needs maintainer reconciliation before merge.

## What This Changes

The PR changes daemon gateway status and restart-health probing to use a health-only probe, adds explicit-auth handling for CLI URL overrides, extends bounded local restart probe timeouts, and extracts explicit gateway auth helpers.

## Best Possible Solution

Rebase or replace this with a narrowed maintainer-reviewed change on current main that adds the missing health-only daemon probe, strict explicit-auth boundary for CLI URL overrides, and deadline-bounded restart probe timeout while preserving `--require-rpc` status RPC semantics and current non-mutating device-identity policy.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: compare current `--require-rpc` docs/tests and `probeGatewayStatus` status-RPC path against the PR diff, which unconditionally switches daemon status probing to `detailLevel: "health"`.

## Solution Assessment

No. The direction is plausible, but this patch is not the best current solution because it regresses `--require-rpc` and must be reconciled with newer shipped probe-auth and device-identity hardening before merge.
