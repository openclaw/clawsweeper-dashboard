## Summary

Keep open. The branch still contains a merge-blocking regression in documented `--require-rpc` status semantics, lacks the required user-facing changelog entry, and changes gateway credential/device-identity policy in ways that need maintainer security review before merge.

## What This Changes

The PR changes daemon gateway status/restart probes to use lighter health/connect probing, extracts explicit-auth helpers, tightens URL-override auth handling, adjusts restart probe timeout budgeting, and adds focused tests.

## Best Possible Solution

Rebase and narrow the branch so it keeps documented `--require-rpc` read proof, adds strict CLI override-auth enforcement, uses the current probe identity policy, and implements a bounded local restart handshake budget with focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Source/docs inspection gives a high-confidence path: current main documents and tests `--require-rpc` as a real read probe, while the PR diff removes that branch; current main also still shows the fixed 3s restart probe timeout.

## Solution Assessment

No. The direction is useful, but the proposed patch is not the best current solution because it mixes stale status-probe behavior with security-sensitive identity/auth changes that need maintainer review and rebase against current main.
