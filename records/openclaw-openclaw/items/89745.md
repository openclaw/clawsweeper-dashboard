## Summary

Keep open: this PR targets the right backend usage-summary boundary for the selected-day aggregation bug, but as written it also drops untimestamped legacy rows from all-time/range=all queries that still pass finite start/end dates. That compatibility regression should be fixed before merge.

## What This Changes

The branch skips untimestamped transcript rows when deriving finite-range session usage summaries from cache or direct transcript scans, and adds cache/direct regression coverage.

## Best Possible Solution

Keep the selected-day timestamp filtering, but treat all-time/range=all ranges as unbounded for untimestamped rows and cover both cache-derived and direct summary paths.

## Reproduction Assessment

Yes, by source inspection: current main and the latest release allow untimestamped rows into bounded summaries, and the PR's all-time regression follows from UI All/protocol range=all mapping to startMs 0 plus a finite endMs. I did not run a live dashboard session in this read-only review.

## Solution Assessment

No as written. The backend cache/direct summary boundary is the right layer, but the timestamp-required predicate is too broad and should distinguish bounded selected-day ranges from all-time/range=all queries.
