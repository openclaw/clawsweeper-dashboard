## Summary

Keep open. Current main has the useful reverse mapping from observed Telegram topic IDs to names, including the merged work from https://github.com/openclaw/openclaw/pull/65973, but it still does not expose a deterministic topic-name-to-topic-ID resolver, topic catalog, or topic-list action for agent routing.

## What This Changes

The issue asks for first-class Telegram forum topic title-to-topic-ID routing support through a `forum_topics` context map, topic-list action, or channel metadata catalog.

## Best Possible Solution

Add a Telegram-owned observed topic catalog or title resolver with explicit freshness, ambiguity, and fail-safe behavior, exposed through the maintainer-chosen tool, context, or channel metadata surface.

## Reproduction Assessment

Yes, as a source-level capability gap. Current main exposes current-topic `topic_id`/`topic_name` plus numeric thread targeting, while the code has no topic catalog, list action, or name-to-ID resolver.

## Solution Assessment

Unclear until maintainers choose the product/API surface. The current reverse topic-name cache is useful, but it is not the requested routing-by-title resolver and should not be stretched into silent best-effort sends without ambiguity rules.
