## Summary

Keep open: current main and v2026.6.8 expose Telegram forum topic names only in the reverse direction, while routing still requires a numeric thread id. The merged topic-name PR is useful context, but it does not provide the requested name-to-topic_id catalog or resolution surface.

## What This Changes

The issue requests first-class Telegram forum topic name-to-topic_id support so agents can route messages by human-readable topic name.

## Best Possible Solution

Add a Telegram-owned topic catalog or resolve action backed by the existing persisted topic-name cache and service-message learning, with explicit ambiguity and staleness rules, then expose it to agent tools/context without moving Telegram-specific policy into core.

## Reproduction Assessment

Yes, at source level: the current message action path only accepts numeric messageThreadId/threadId values, and the topic-name cache only reads by chatId plus threadId. I did not run a live Telegram group, but the missing resolver is clear from current main and v2026.6.8 source.

## Solution Assessment

Unclear until maintainers choose the product surface. The merged topic-name context work is the right data source, but it is not the complete solution because it does not expose a name-to-id directory or resolver.
