## Summary

Keep open: current main and v2026.6.10 still expose Telegram forum topic names only when a numeric topic/thread id is already known; they do not provide the requested name-to-topic_id resolver, topic list action, or catalog surface for agent routing.

## What This Changes

The issue requests a durable Telegram forum topic-name to topic_id resolver or catalog surface so agents can route messages by human-readable topic name.

## Best Possible Solution

Add a Telegram-owned topic catalog or resolver backed by the persisted topic-name cache and service-message learning, with explicit ambiguity/staleness rules and an approved agent-facing seam.

## Reproduction Assessment

Yes, at source level: current main learns and exposes topic names only after a numeric forum thread id is known, while routing actions still require positive integer messageThreadId/threadId. I did not run a live Telegram forum group in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product surface. The existing persisted topic-name cache is the right data source, but the remaining fix needs a deliberate resolver/list/catalog contract rather than treating reverse current-topic metadata as complete.
