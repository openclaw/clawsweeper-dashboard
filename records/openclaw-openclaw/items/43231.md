## Summary

This still needs maintainer follow-up: current main and the latest release preserve numeric Telegram topic routing and observed ID-to-name metadata, but they do not expose a reliable topic-name-to-topic-ID resolver or agent-callable topic catalog. The merged work in https://github.com/openclaw/openclaw/pull/65973 is useful partial progress, while duplicate names, renames, closed topics, and observed-only Bot API data still need a product decision before implementation.

## What This Changes

The issue requests first-class Telegram forum topic title-to-ID resolution or a topic catalog/listing surface for agent routing.

## Best Possible Solution

Add an on-demand Telegram-owned observed-topic catalog and resolver action over the existing persistent cache, returning numeric IDs plus lifecycle state and rejecting unknown or ambiguous title matches.

## Reproduction Assessment

Yes. Source inspection shows numeric topic targeting and ID-to-name storage exist, but no public action or API accepts a human-readable topic title and returns the matching topic ID.

## Solution Assessment

No to injecting a potentially large and incomplete catalog into every session by default. The narrower maintainable path is an on-demand Telegram action using the observed-topic store, with explicit unknown and ambiguous results.
