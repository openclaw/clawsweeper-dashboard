## Summary

Keep open: current main and the latest release still expose Telegram forum topic names only after a numeric thread id is already known, while routing and documented targets still require numeric `threadId`/`:topic:<id>` values. The merged topic-name cache work is useful adjacent context, but it does not provide the requested name-to-topic_id catalog, list action, or resolver surface.

## What This Changes

The issue requests first-class Telegram forum topic-name to topic_id resolution for agent routing, such as a topic map in context, a topic-list action, or channel metadata endpoint.

## Best Possible Solution

Add a Telegram-owned topic catalog or resolver backed by the persisted topic-name cache and service-message learning, with explicit ambiguity and staleness rules, then expose it through the approved agent tool/context/metadata seam without moving Telegram policy into core.

## Reproduction Assessment

Yes, at source level: current main learns `TopicName` only after a numeric forum thread id is known, and Telegram message actions still require positive integer `messageThreadId`/`threadId`. I did not run a live Telegram forum group in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product surface. The existing persisted topic-name cache is the right data source, but the remaining fix needs a deliberate resolver/list/catalog contract rather than treating reverse current-topic metadata as complete.
