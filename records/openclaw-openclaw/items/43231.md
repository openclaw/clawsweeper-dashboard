## Summary

Keep open. Current main has useful partial support from #65973/#66107, but it only exposes the current forum topic name and persists a learned `chatId:threadId -> name` cache; it still does not provide an agent-visible topic directory, list action, metadata endpoint, or name-to-topic-ID resolver for routing by human-readable name.

## What This Changes

Request to let Telegram forum agents resolve human-readable topic names to topic IDs for routing instead of relying on manual numeric mappings.

## Best Possible Solution

Add a Telegram-owned, agent-facing topic directory or lookup surface backed by the learned topic-name cache, with explicit partial-cache and ambiguity semantics, while keeping routing inputs numeric once resolved.

## Reproduction Assessment

Yes, as a source-level capability gap: current main exposes only the active `topic_id`/`topic_name` pair and numeric topic targets, so asking an agent to route to a different forum topic by name still has no deterministic built-in resolver. I did not need live Telegram access to confirm that missing surface in the current code.

## Solution Assessment

No, the existing merged mitigation is not the full solution for this issue. It is useful reverse/current-topic context, but the best fix needs a deliberate Telegram channel API/product choice for listing or resolving known topics before routing.
