## Summary

Keep open. Current main partially mitigates the Telegram forum-topic context gap with the merged topic-name cache from https://github.com/openclaw/openclaw/pull/65973, but it still does not provide the requested name-to-topic-id catalog or list action needed for deterministic routing by human-readable topic name.

## What This Changes

Request first-class Telegram forum topic-name to topic_id resolution for agent routing, such as a forum_topics map, topic-list action, or metadata endpoint.

## Best Possible Solution

Add a Telegram-owned observed topic catalog surface with explicit partial-cache semantics so agents can resolve known topic titles to IDs without pretending Telegram can enumerate every topic on demand.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a missing feature/API surface. Source inspection confirms current main exposes the current TopicName/topic_id and numeric thread targeting, but no topic catalog or name-to-ID resolution path.

## Solution Assessment

No, the current partial implementation is not the full solution for this issue. The narrowest maintainable path is a deliberate Telegram channel/API design around the existing persisted observed-topic cache, including how agents should handle unknown or stale topic names.
