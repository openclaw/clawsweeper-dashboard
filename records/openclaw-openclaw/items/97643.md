## Summary

Keep open: the PR targets the right Control UI wrong-gateway bug, but the patch is not merge-ready because it removes the shipped legacy settings migration path that current tests still cover, and it has only unit-test output rather than real two-base-path browser proof.

## What This Changes

This PR removes Control UI reads and writes of the unscoped legacy settings key so settings stay scoped by gateway path.

## Best Possible Solution

Keep the PR open and revise the storage loader so mismatched legacy data cannot override the page-derived gateway, while preserving or explicitly retiring the existing same-profile legacy migration with focused tests.

## Reproduction Assessment

Yes by source inspection: current main reads the unscoped legacy settings key after scoped/default keys, so a legacy value from Gateway A can supply Gateway B's `gatewayUrl` before B has a scoped record. I did not run the live reverse-proxy scenario in this read-only review.

## Solution Assessment

No. Removing every legacy settings read fixes the cross-path poisoning, but it is broader than the safest fix because it also breaks the existing legacy migration behavior covered by current tests.
