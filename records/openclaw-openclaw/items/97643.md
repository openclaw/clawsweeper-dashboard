## Summary

Keep open: the updated branch is still not merge-ready because it now resets loaded custom Gateway URLs to the page-derived default and removes the legacy write path without adding another discoverable current-settings pointer.

## What This Changes

This PR changes Control UI settings loading to keep the legacy fallback but force the loaded `gatewayUrl` to the page default, and stops writing the unscoped legacy settings key.

## Best Possible Solution

Keep the storage-boundary fix, but distinguish stale sibling legacy records from intentional custom endpoints, add a page/default-scoped current-settings pointer or equivalent migration path, and cover both upgrade and reload behavior with tests plus real proof.

## Reproduction Assessment

Yes. By source inspection, save a custom Gateway URL through the existing UI field and reload: PR head either overwrites the loaded value with `defaultUrl` or cannot rediscover the custom-scoped record after removing the legacy write.

## Solution Assessment

No. The storage module is the right layer, but this implementation is too broad because it collapses all loaded Gateway URLs to the page default and removes the current-settings bridge custom endpoints need.
