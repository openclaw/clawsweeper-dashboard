## What This Changes

The branch removes the Control UI unscoped legacy settings key from `loadSettings()` fallback and stops writing saved settings to that legacy key.

## Best Possible Solution

Land a storage-boundary fix that blocks stale cross-base-path legacy `gatewayUrl` values while preserving safe legacy-only settings migration, with focused regression tests and real browser or network proof.

## Reproduction Assessment

Yes by source inspection: seed the legacy global settings key with Gateway A's `gatewayUrl`, load Gateway B with no B-scoped settings key, and `loadSettings()` can return Gateway A's URL. I did not run a live reverse-proxy setup in this read-only review.

## Solution Assessment

No. The owner boundary is right, but the diff is too broad because it drops legacy-only settings migration instead of limiting legacy data so it cannot override the current path's `gatewayUrl`.
