## Summary

Keep this PR open because it targets a plausible ACP lifecycle bug, but the current patch would regress valid long-running ACP turns by imposing a hard 30 second stream timeout outside the manager’s configured timeout and cleanup path.

## What This Changes

The PR adds ACP turn-stream unit tests and changes `consumeAcpTurnStream()` to throw when stream consumption times out after 30 seconds or completes without a `done` event.

## Best Possible Solution

Revise the fix to surface premature child exit through the existing ACP manager timeout/cancellation lifecycle, preserving configured turn durations and adding regression coverage for both missing terminal events and hung streams.

## Reproduction Assessment

Yes, from source inspection and the linked issue evidence: current code has clear stream-completion and hung-stream paths to inspect, and the report gives concrete orphan-session artifacts. I did not establish a live Docker/acpx reproduction in this read-only review.

## Solution Assessment

No. The proposed direction is reasonable, but this patch is not the best implementation because it bypasses the manager’s configured timeout and cleanup behavior with an unconditional 30 second cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve configured ACP turn timeouts:** `src/acp/control-plane/manager.turn-stream.ts:71`
  - body: Wrapping all stream consumption in a hard-coded 30s timeout makes valid long-running ACP turns fail after 30 seconds, even though `runTurn` already resolves a configured timeout from runtime options or the 48h agent default and performs cleanup through `awaitTurnWithTimeout`. Please keep this detection on the manager timeout/cancel path, or make it a narrow child-exit/no-event detector that cannot override configured turn duration.
