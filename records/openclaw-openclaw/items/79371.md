## Summary

Current main already implements and ships the ACP no-terminal-event failure path targeted by this PR; the branch's remaining fixed 30s stream timeout would regress configured long-running ACP turns.

## What This Changes

The PR adds a fixed 30s timeout and terminal-event check inside `consumeAcpTurnStream()` plus a new unit test file for early-ended ACP streams.

## Best Possible Solution

Keep the shipped manager-level timeout and terminal-event handling; consider a separate idle-timeout design only if a future issue defines that contract.

## Reproduction Assessment

Yes. The bug shape is reproducible at source level with a runtime whose `runTurn()` yields output and finishes without `done`; current main has a focused regression test for that path, and this review stayed read-only.

## Solution Assessment

No for this branch. Current main already uses the configured manager turn timeout plus a terminal-event check; adding a hard-coded 30s race inside stream consumption is not the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Respect configured ACP turn timeouts:** `src/acp/control-plane/manager.turn-stream.ts:72`
  - body: `withTimeout(runStream(), TURN_STREAM_TIMEOUT_MS)` races the entire ACP stream against a fixed 30s timer, while the manager already resolves turn timeouts from runtime options or `agents.defaults.timeoutSeconds`. Any legitimate ACP turn that runs longer than 30s will now fail with `ACP_TURN_FAILED` even though its configured run timeout has not expired. Keep timeout enforcement in `awaitTurnWithTimeout` or pass the resolved timeout through instead.
  - confidence: 0.91
