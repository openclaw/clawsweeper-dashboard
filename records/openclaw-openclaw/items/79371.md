## Summary

Close this PR because the requested ACP runTurn failure behavior is already implemented on current main and shipped in v2026.5.7; the PR’s remaining unique change adds a fixed 30s stream-wide timeout that would be a regression for valid long-running ACP turns.

## What This Changes

The PR changes ACP turn-stream handling so premature stream completion or no events reject with ACP_TURN_FAILED, and adds focused tests for terminal, error, and timeout cases.

## Best Possible Solution

Keep the shipped manager-level terminal-event guard and avoid merging a helper-level fixed timeout unless it is redesigned around the configured turn timeout and proven against a real ACP child process.

## Reproduction Assessment

Yes for the reported completed-stream shape: current main has a focused source test where runTurn yields output and then completes without done, and the manager rejects with ACP_TURN_FAILED. The PR proof remains mock-only for the real child-process crash path.

## Solution Assessment

No. The central fix is already in the manager on current main, while the PR’s added 30s helper timeout wraps the whole stream and can fail valid long-running ACP turns before the configured turn timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Do not cap the whole ACP stream at 30 seconds:** `src/acp/control-plane/manager.turn-stream.ts:71`
  - body: This races the entire `runStream()` promise against a fixed 30s timer. A legitimate ACP turn with streaming output, tool work, or model latency that lasts more than 30s will reject even though the manager's configured turn timeout has not expired, so long-running turns can be marked failed. Keep the existing manager timeout or implement a true per-event idle-gap guard tied to the configured timeout.
  - confidence: 0.93
