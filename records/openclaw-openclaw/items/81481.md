## Summary

Keep open. Current main already gives each agent a self-contained QMD home and per-agent QMD/XDG environment, but the proposed per-agent `qmd.command` override is genuinely absent from the schema, types, resolver, and docs; adding it would be a new config and command-execution surface that needs maintainer judgment.

## What This Changes

The issue asks for `agents.list[].memorySearch.qmd.command` so each agent can route QMD memory tools through its own wrapper or binary for isolated indexes.

## Best Possible Solution

Document the existing per-agent QMD home if that satisfies the privacy case, or intentionally add a versioned `memorySearch.qmd.command` override with schema, resolver, docs, and QMD manager tests if maintainers want per-agent wrapper support.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms `memorySearch.qmd` is strict and accepts only `extraCollections`, while `qmd.command` is resolved globally from `memory.qmd.command`.

## Solution Assessment

Unclear. The exact override is not clearly the best first solution because current main already isolates QMD indexes per agent; maintainers should decide whether the remaining need is documentation or a new per-agent binary/wrapper contract.
