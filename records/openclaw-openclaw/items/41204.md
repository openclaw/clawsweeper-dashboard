## Summary

Keep open: current main already has safer installer-specific validation and guarded subagent sweep behavior, but the active session-store writer queue is still uncapped, while this PR regresses current H2/H3 behavior and is not mergeable as-is.

## What This Changes

The branch adds a lock queue cap in session store locking, rewrites subagent orphan cleanup, adds shared skill install validation, and adds three related tests.

## Best Possible Solution

Keep current main's installer and subagent cleanup implementations, and replace only the queue-cap portion in `src/config/sessions/store-writer.ts` with deterministic coverage if maintainers want that hardening.

## Reproduction Assessment

Yes for the PR regressions: source inspection shows the PR would remove current archive guards, lose delete retry behavior, and reject valid installer specs. The remaining queue concern is source-plausible through the uncapped writer queue, but not live-reproduced here.

## Solution Assessment

No. The mixed patch is not the best fix because current main already has safer H2/H3 behavior, and the only remaining useful work is a narrow queue cap on the current writer path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore the archive guard before cleanup:** `src/agents/subagent-registry.ts:784-785`
  - body: After the orphan branch, runs that are not due for archive can fall into cleanup because the previous future-archive guard is gone. That can delete active or recently ended runs and call `sessions.delete` early.
  - confidence: 0.95
- **[P2] Keep archive cleanup on the retrying path:** `src/agents/subagent-registry.ts:728-746`
  - body: The due-archive branch removes the run and attachments before `sessions.delete` succeeds and then swallows delete failures. A transient gateway error can lose the retry and context-engine notification path current main preserves.
  - confidence: 0.9
- **[P2] Use installer-specific safe specs:** `src/agents/skills-install.ts:128`
  - body: The shared package-name regex is applied to node, go, and uv installs, but it rejects valid metadata current main accepts, including uv extras/version pins, Go `+incompatible`, and npm build metadata or tilde names.
  - confidence: 0.89
- **[P2] Avoid the serial 100-second lock queue test:** `src/config/sessions/store.lock-queue.test.ts:21-23`
  - body: The test queues up to 1001 accepted lock tasks that each sleep for 100 ms, then waits for the serial drain. Under normal Vitest timeouts this is expected to fail or add a very long test, so the proof needs a deterministic blocked-lock setup.
  - confidence: 0.78
