## Summary

This PR should stay open only for maintainer disposition: it is a security-sensitive mixed hardening branch with concrete regressions in subagent cleanup and installer validation, while the remaining useful queue-cap idea is narrower than the branch.

## What This Changes

The PR adds a session-store lock queue cap, subagent orphan cleanup during sweep, shared skill installer validation, and three related tests.

## Best Possible Solution

Keep current main's installer-specific validation and guarded subagent sweep behavior, and consider a separate narrow writer-queue cap in src/config/sessions/store-writer.ts if owners want that hardening.

## Reproduction Assessment

Yes for the PR regressions: the diff and current-main source show the lost archive guard, lost delete retry behavior, and installer regex contract regression. The original queue-exhaustion claim remains source-plausible from the uncapped writer queue, but I did not run a live stress reproduction.

## Solution Assessment

No. The mixed branch is not the best solution because H2/H3 are safer on current main and the remaining C1 concern should be handled as a focused store-writer queue-cap change with deterministic coverage.
