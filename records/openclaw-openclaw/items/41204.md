## Summary

Keep open: this PR is not safe to merge as written because it mixes a still-plausible session-store queue cap with subagent and skill-install changes that current main now handles through narrower contracts, while the PR diff introduces runtime and installer regressions.

## What This Changes

The PR adds a session-store lock queue cap, changes subagent sweep cleanup for orphaned runs, adds shared skill installer package/formula validation, and adds three related test files.

## Best Possible Solution

Keep current main's installer-specific validation and guarded subagent sweep behavior, and land any queue hardening as a focused change in the current session writer with deterministic tests.

## Reproduction Assessment

Yes for the PR regressions: the PR diff and current-main source show the missing archive guard, lost delete retry behavior, invalid shared installer regex, and slow queue test path. The queue exhaustion concern is code-provable through the unbounded pending array, but no live stress reproduction was run in this read-only pass.

## Solution Assessment

No. The proposed patch is not the narrowest maintainable solution because it replaces existing installer and subagent contracts with broader logic that regresses valid installs and cleanup behavior; a focused queue-cap PR would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore the archive guard before standard cleanup:** `src/agents/subagent-registry.ts:784-785`
  - body: After the orphan branch, every non-orphan entry falls into the Standard archive cleanup block even when it has no archiveAtMs or has a future archive time. That deletes active or recently ended runs and calls sessions.delete before they are eligible for archive cleanup.
  - confidence: 0.95
- **[P2] Keep archive cleanup on the retrying path:** `src/agents/subagent-registry.ts:728-746`
  - body: The new due-archive branch deletes the run and removes attachments before sessions.delete, swallows delete failures, and continues before the context-engine notification path. A transient delete failure can therefore lose the registry entry without retrying session cleanup or notifying the context engine.
  - confidence: 0.9
- **[P2] Use installer-specific safe specs:** `src/agents/skills-install.ts:128`
  - body: The shared package-name regex is applied to node, go, and uv installs, but it rejects valid metadata that current main accepts through installer-specific parsing, such as uv extras/version pins and npm registry names with supported selector syntax.
  - confidence: 0.89
- **[P2] Avoid the serial 100-second lock queue test:** `src/config/sessions/store.lock-queue.test.ts:21-23`
  - body: This test queues up to 1001 accepted lock tasks that each sleep for 100 ms, while the lock queue drains serially and Promise.allSettled waits for all of them. Under normal Vitest timeouts this is expected to fail or add a very long test, so the queue-limit proof needs a deterministic blocked-lock setup.
  - confidence: 0.78
