## Summary

Keep open. Current main still has a source-visible path where bare reset/new commands can be acknowledged before the reset startup prompt and startupContext path runs, while Control UI typed `/new` now creates a dashboard session without an initial agent turn; the related open fix PR covers only part of that surface.

## What This Changes

The issue requests that WebChat/Control UI bare `/new` and `/reset` start a reset startup agent turn with startupContext instead of only returning or creating a static reset acknowledgement.

## Best Possible Solution

Define the intended reset-greeting behavior once, then align Control UI typed `/new`, `chat.send` bare resets, `agent.send`, startupContext docs, and regression coverage to that contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: direct `chat.send` bare reset/new can stop in `maybeHandleResetCommand`, and Control UI typed `/new` takes the `sessions.create` path without an initial agent turn. I did not run live tests because this review is read-only.

## Solution Assessment

Unclear. Restoring fallthrough in `commands-reset.ts` is likely necessary for bare `chat.send` resets, but the best fix also needs a maintainer decision for Control UI typed `/new` because that path intentionally moved to dashboard session creation.
