## Summary

Keep open. This is plausible security hardening with sufficient live proof, but the branch needs maintainer direction: current main already covers part of the /exec and phone-control behavior, while this PR still weakens least-privilege command access and misses current runtime config mutation APIs.

## What This Changes

The PR adds internal `operator.admin` checks for gateway `chat.send` command authorization, `/exec` default persistence, and plugin runtime config writes, with gateway scope propagation into plugin command execution.

## Best Possible Solution

Model internal command capabilities centrally so read-only internal commands remain available to `operator.write`, mutating command and plugin config writes require `operator.admin`, and all runtime config mutation APIs share one sink-level guard if that direction is approved.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: this PR makes write-scoped internal `chat.send` callers unauthorized before command-specific gates can allow read-only commands, and current main exposes runtime config mutation APIs that the patch does not guard.

## Solution Assessment

No. The hardening goal is valid, but this patch is not the best solution because it needs a maintainer-approved internal command capability model and complete coverage of current runtime config write APIs.
