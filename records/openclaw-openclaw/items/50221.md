## Summary

Keep open. The PR targets a real internal-scope hardening area, but the proposed branch has a blocking authorization regression for `operator.write` chat clients and the remaining plugin runtime config sink behavior still needs maintainer scope-model direction. Current `main` already implements safer pieces of this area, so this should not be closed as implemented or merged as-is.

## What This Changes

The PR changes internal `chat.send` and plugin command authorization so `operator.admin` is required for command authorization, `/exec` default persistence, and plugin runtime config writes, with regression tests for `operator.write` and `operator.admin` paths.

## Best Possible Solution

Keep the current direction of forwarding gateway scopes through request and plugin command context, then add a maintainer-approved command capability or required-scope model that gates mutating actions without blocking read-only internal commands. If sink-level plugin config write enforcement is desired, design it as an explicit plugin runtime contract with tests for external plugins and deprecated config APIs.

## Reproduction Assessment

Yes. The PR diff makes `chat.send` set `CommandAuthorized` from `operator.admin`, and current command authorization rejects commands before action-level gates; a focused reproduction is an internal `operator.write` `chat.send` caller running a read-only command such as `/config show` or `/status`.

## Solution Assessment

No. The branch's blanket `CommandAuthorized=false` for non-admin internal callers is not the narrowest maintainable fix; command/action-level capability metadata or `requiredScopes` preserves least privilege while keeping read-only commands usable.
