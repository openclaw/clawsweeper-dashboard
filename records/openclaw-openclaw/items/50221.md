## Summary

Keep this PR open: the hardening goal is valid, but the current patch blocks read-only internal slash commands for `operator.write` clients and misses current runtime config mutation APIs, while the author has asked for owner guidance on the command-capability policy.

## What This Changes

The PR requires `operator.admin` for internal `chat.send` command authorization, `/exec` default persistence, and plugin runtime config writes, with scope propagation through plugin command execution and regression tests.

## Best Possible Solution

Model internal command capabilities centrally so read-only commands remain available to `operator.write`, mutating command and plugin config writes require `operator.admin`, and every runtime config mutation API uses one shared internal-scope gate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR makes write-scoped internal `chat.send` callers unauthorized before command-specific gates run, and current main exposes primary `mutateConfigFile`/`replaceConfigFile` APIs that this patch does not guard.

## Solution Assessment

No. The hardening direction is sound, but the patch should be reworked around command capability policy and complete runtime config mutation coverage instead of a blanket admin gate on internal chat commands.
