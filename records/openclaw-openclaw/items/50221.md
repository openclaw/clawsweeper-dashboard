## Summary

Keep open: this PR targets a real internal-scope hardening area, but the branch still introduces blocking least-privilege regressions and current main has already moved part of the solution toward narrower action-level scope checks.

## What This Changes

The PR gates internal `chat.send` command authorization, `/exec` default persistence, and plugin runtime config writes behind `operator.admin`, propagates gateway scopes into plugin command execution, and adds regression tests.

## Best Possible Solution

Keep `chat.send` on the current action-level authorization path, then make plugin config mutation policy explicit through command `requiredScopes`, targeted owner gates, or a versioned runtime write contract.

## Reproduction Assessment

Yes. Source inspection is enough: the PR sets `CommandAuthorized=false` for `operator.write` internal `chat.send` callers, and current command authorization rejects before command-specific handlers can apply narrower gates.

## Solution Assessment

No. The PR is not the best fix as written because it uses a blanket admin gate at `chat.send`; current main's command/action-level scope model is safer, while the generic plugin config-write sink still needs an explicit maintainer decision.
