## Summary

Keep open: this security-sensitive PR still has blocking correctness and policy issues against current main, and the author has asked for maintainer direction before reworking the command capability model.

## What This Changes

The PR changes internal gateway scope handling for chat command authorization, /exec persistence, and plugin runtime config writes.

## Best Possible Solution

Model internal command capabilities centrally, preserve read-only operations for operator.write clients, require operator.admin for mutating operations, and apply any approved runtime config write guard to every supported write API.

## Reproduction Assessment

Yes. Source inspection of the PR head and current main gives a high-confidence path: operator.write chat.send clients are denied before command-specific policy, mixed directive-only messages can lose allowed updates, and active runtime config write APIs are not covered by the PR guard.

## Solution Assessment

No. The PR uses a coarse admin gate and incomplete sink coverage; the maintainable path is command-level capability policy plus complete runtime config write enforcement if maintainers approve that security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve read-only internal commands for write-scoped clients:** `src/gateway/server-methods/chat.ts:1308`
  - body: This makes every internal chat.send caller without operator.admin set CommandAuthorized=false. The current command auth path treats that as a top-level gate, so read-only commands intentionally available to operator.write clients, such as /config show, are rejected before command-specific policy can allow them.
  - confidence: 0.93
- **[P2] Let allowed directives survive rejected exec persistence:** `src/auto-reply/reply/directive-handling.impl.ts:279-283`
  - body: This early return fires for directive-only messages with rejected /exec options before later code persists other allowed directive fields. A message like /think high /exec host=gateway would drop the allowed thinking update along with the rejected exec persistence.
  - confidence: 0.88
- **[P2] Guard all runtime config write APIs:** `src/plugins/runtime/runtime-config.ts:9-20`
  - body: The guard wraps only deprecated writeConfigFile, while current plugin runtime writes primarily use mutateConfigFile and replaceConfigFile. If sink-level hardening is approved, those active mutation APIs need equivalent enforcement.
  - confidence: 0.86
