## Summary

Keep this PR open, but it should not merge in its current shape: the security goal is valid, while the branch still blocks least-privilege read-only command flows and leaves the current plugin runtime config mutation APIs uncovered.

## What This Changes

The PR adds internal `operator.admin` checks for `chat.send` command authorization, `/exec` default persistence, and plugin runtime config writes, plus scope propagation into plugin command execution.

## Best Possible Solution

Model internal command capabilities centrally so read-only commands remain available to `operator.write`, mutating command and plugin config writes require `operator.admin`, and every current runtime config mutation API is covered by the same policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR makes `operator.write` `chat.send` callers unauthorized before command-specific gates run, and current main shows `mutateConfigFile` and `replaceConfigFile` remain outside the proposed guard.

## Solution Assessment

No. The security goal is sound, but the branch should be reworked around command capability policy plus complete runtime config mutation coverage rather than a blanket admin gate on internal chat commands.

  - body: The scope check wraps only deprecated `writeConfigFile`, but current plugin runtime exposes `mutateConfigFile` and `replaceConfigFile` as the primary mutation APIs. Plugin commands using those APIs can still mutate config without this admin check.
