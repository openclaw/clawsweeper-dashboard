## Summary

Keep open; this is a plausible security-hardening direction with sufficient live proof, but the current patch should not merge because it breaks least-privilege read-only command access and does not cover the current runtime config mutation API surface.

## What This Changes

The PR adds admin-scope enforcement for internal chat command authorization, /exec default persistence, and plugin runtime config writes, plus propagation of gateway client scopes into plugin command execution.

## Best Possible Solution

Model internal command capabilities centrally so read-only commands remain available to operator.write, mutating commands require operator.admin, and all plugin runtime config mutation sinks share one approved guard.

## Reproduction Assessment

Yes, by source inspection. The PR diff changes chat.send to a blanket admin gate, current command authorization uses that flag as a top-level gate, and current main exposes runtime config write methods the PR does not guard.

## Solution Assessment

No. The hardening goal is valid, but this patch should be revised around a maintainer-approved command capability model and complete runtime config sink coverage.

## Review Findings

  - confidence: 0.88

  - body: The guard wraps only `writeConfigFile`. Current main's plugin runtime contract has `mutateConfigFile` and `replaceConfigFile` as primary write APIs, so the proposed sink guard leaves active mutation APIs uncovered.
  - confidence: 0.84
