## Summary

Keep open: this is a plausible security-hardening PR with real behavior proof, but the current diff has blocking authorization regressions and needs maintainer direction for the long-term command scope model before merge.

## What This Changes

The branch tightens internal gateway scope handling by requiring `operator.admin` for `chat.send` command authorization, `/exec` default persistence, and plugin runtime config writes, with scope propagation and regression tests.

## Best Possible Solution

Adopt a central command capability or required-scope policy that keeps read-only internal commands available to `operator.write` while requiring `operator.admin` at mutating command and plugin config-write sinks.

## Reproduction Assessment

Yes. The PR body gives real gateway proof for the security gap, and source inspection shows how the proposed blanket `CommandAuthorized=false` reproduces the review regression without running a local gateway.

## Solution Assessment

No. The security goal is right, but the current patch is not the best fix because it uses a blanket command denial and misses current runtime config write APIs; command-level scopes plus sink guards are safer.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve read-only commands for write-scoped clients:** `src/gateway/server-methods/chat.ts:1308`
  - body: Changing `chat.send` to set `CommandAuthorized=false` unless the client has `operator.admin` blocks command handling before command-specific gates run. That means internal `operator.write` clients lose read-only commands like `/config show`, so this needs command-level scope checks instead of blanket denial.
  - confidence: 0.94
- **[P2] Let allowed directives survive denied exec updates:** `src/auto-reply/reply/directive-handling.impl.ts:285`
  - body: This early return fires for directive-only messages whenever `/exec` has options and `cleaned` is empty. Mixed directive messages like `/model ... /exec host=gateway` would drop the allowed model or thinking change instead of only skipping the denied `/exec` persistence.
  - confidence: 0.9
- **[P1] Guard all runtime config write APIs:** `src/plugins/runtime/runtime-config.ts:29-35`
  - body: Current main exposes `mutateConfigFile` and `replaceConfigFile` as the supported plugin runtime write APIs, while this guard only wraps deprecated `writeConfigFile`. After rebase, plugin commands using the supported APIs could still mutate config unless every runtime config write path is scoped.
  - confidence: 0.88
