## Summary

Keep open: this security-sensitive PR still has blocking correctness and security issues, and the author has already asked for maintainer direction on the command capability model. Current main covers some related scope gates, but the branch's blanket chat command gate and incomplete runtime config sink guard are not ready to merge.

## What This Changes

This PR changes internal gateway scope handling for chat.send command authorization, /exec default persistence, and plugin runtime config writes, with tests and plugin command scope propagation.

## Best Possible Solution

Model internal command capabilities centrally, preserve read-only operator.write operations, require operator.admin for mutations, and if sink-level config hardening is approved, cover mutateConfigFile, replaceConfigFile, and deprecated writeConfigFile consistently.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: operator.write chat.send clients are denied before command-specific policy, mixed directive-only messages can drop allowed updates, and current runtime config write APIs are not covered by the PR guard.

## Solution Assessment

No. The PR uses a coarse admin gate and incomplete sink coverage; the narrower maintainable solution is command-level capability policy plus complete runtime config write enforcement only if maintainers approve that boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve read-only internal commands for write-scoped clients:** `src/gateway/server-methods/chat.ts:1308`
  - body: This sets CommandAuthorized=false for every internal chat.send caller lacking operator.admin. resolveCommandAuthorization treats that as a top-level gate, so read-only commands that current main intentionally lets operator.write use, such as /config show, are rejected before command-specific policy can allow them.
  - confidence: 0.93
- **[P2] Let allowed directives survive rejected exec persistence:** `src/auto-reply/reply/directive-handling.impl.ts:279-283`
  - body: This early return fires for directive-only messages with rejected /exec options before later code persists other allowed directive fields. A message like /think high /exec host=gateway would drop the allowed thinking update along with the rejected exec persistence.
  - confidence: 0.88
- **[P2] Guard all runtime config write APIs:** `src/plugins/runtime/runtime-config.ts:9-20`
  - body: The sink guard wraps only deprecated writeConfigFile, but PluginRuntime config exposes mutateConfigFile and replaceConfigFile as current write APIs. A plugin using either current API can still mutate config from an internal operator.write command, so the advertised sink-level hardening is incomplete.
  - confidence: 0.86
