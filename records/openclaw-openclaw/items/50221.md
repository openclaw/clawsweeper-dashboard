## Summary

Keep this PR open, but it is not mergeable as-is: the hardening is useful and the author provided real gateway proof, but the branch makes write-scoped internal clients unauthorized for read-only commands and the author already asked for maintainer guidance on the command-scope model.

## What This Changes

The PR changes internal gateway authorization so operator.write clients lose broad command authorization, /exec persistence and plugin runtime config writes require operator.admin, and plugin command scope is propagated into runtime config writes.

## Best Possible Solution

Use a central command capability or command metadata policy so operator.write clients keep read-only internal commands, mutating commands require operator.admin, and plugin runtime config mutation gates cover the current config API surface.

## Reproduction Assessment

Yes from source inspection for the blockers. The PR makes operator.write chat.send callers unauthorized before command-specific read-only allowances run, and the added /exec early return can skip other valid directive changes when cleaned text is empty.

## Solution Assessment

No. The narrow maintainable direction is command-capability policy plus sink-level mutation guards, not making every internal chat.send slash command require operator.admin.
