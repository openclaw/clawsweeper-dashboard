## Summary

This PR should stay open, but not merge as-is. Current main already implements part of the intended internal-scope cleanup, while the branch still over-denies read-only internal commands and misses the current plugin runtime config mutation APIs.

## What This Changes

The PR changes internal gateway scope handling so `operator.write` callers lose broad command authorization, `/exec` persistence and plugin runtime config writes require `operator.admin`, and plugin command scopes are propagated through tests and runtime context.

## Best Possible Solution

Model internal command capabilities centrally so read-only commands remain available to `operator.write`, mutating command and plugin config writes require `operator.admin`, and any sink-level runtime config guard covers all current mutation APIs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the blockers: the branch makes `operator.write` `chat.send` callers unauthorized before command-specific gates run, and current parser/runtime config code shows the stale directive and config mutation assumptions.

## Solution Assessment

No. The PR has the right security goal, but the maintainable fix is command capability policy plus complete runtime config mutation coverage, not a blanket admin gate on all internal slash commands.
