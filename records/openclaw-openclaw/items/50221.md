## Summary

Keep open. Current main has already moved toward the safer shape by forwarding gateway scopes, enforcing admin on specific mutating directive and command paths, and supporting plugin command `requiredScopes`, but this PR branch still contains a blanket `chat.send` command-authorization change that would block read-only internal slash commands and leaves the plugin config-write sink model as a security-sensitive contract decision.

## What This Changes

This branch makes internal `chat.send` derive command authorization from `operator.admin`, gates `/exec` persistence and plugin runtime config writes on admin scope, propagates gateway scopes into plugin command execution, and adds regression tests.

## Best Possible Solution

Keep gateway client scopes in request and plugin command context, then use a maintainer-approved command capability or `requiredScopes` model to gate mutating actions while preserving read-only commands for `operator.write`. If runtime config write sink enforcement is desired, make it an explicit plugin runtime contract with compatibility notes and tests for both current and deprecated config APIs.

## Reproduction Assessment

Yes. The PR diff sets `CommandAuthorized=false` for internal `operator.write` `chat.send` callers, and current `resolveCommandSenderAuthorization` rejects commands when that flag is false; a focused check is an `operator.write` internal `chat.send` caller running a read-only command such as `/config show`.

## Solution Assessment

No. The PR's blanket admin gate at `chat.send` is not the narrowest maintainable fix; current main's action-level gates and plugin command `requiredScopes` preserve least privilege while still preventing admin-only mutations.

- **[P1] Preserve read-only commands for write-scoped callers:** `src/gateway/server-methods/chat.ts:1308`
  - body: Setting `CommandAuthorized` from `operator.admin` here makes `operator.write` `chat.send` callers fail generic command authorization before command-specific gates run. Read-only commands such as `/config show`, `/status`, and `/whoami` should remain available without granting full admin scope; keep chat authorization separate and enforce admin at mutating command actions.
  - confidence: 0.91
- **[P2] Apply other directives when exec persistence is denied:** `src/auto-reply/reply/directive-handling.impl.ts:280-284`
  - body: This early return fires for directive-only messages once `/exec` has options and the cleaned body is empty. Mixed directive-only messages such as `/model ... /exec host=gateway` or `/think high /exec node=...` then drop the allowed model/thinking updates along with the denied exec write. Continue processing allowed directives and include the exec-denied acknowledgement instead.
  - confidence: 0.86
- **[P3] Surface denied inline exec persistence:** `src/auto-reply/reply/directive-handling.persist.ts:91-92`
  - body: When inline `/exec` persistence is blocked, this path silently skips the write with no reply or trace signal. Operators embedding `/exec host=...` in a normal message cannot tell that persistence was rejected, so return an acknowledgement/event or at least log a visible warning for the denied update.
  - confidence: 0.75
