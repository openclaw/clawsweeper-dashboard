## Summary

Keep open: the security goal is valid, but the current branch should not merge as-is because it blanket-denies internal `operator.write` slash commands before command-specific gates can preserve read-only access, and its plugin config sink guard misses the current primary mutation APIs.

## What This Changes

The PR adds internal `operator.admin` enforcement for `chat.send` command authorization, `/exec` default persistence, and plugin runtime config writes, plus gateway-scope propagation through plugin command execution and regression tests.

## Best Possible Solution

Model internal command capabilities centrally so read-only commands remain available to `operator.write`, mutating command and plugin config writes require `operator.admin`, and every runtime config mutation API is covered by one policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR makes `operator.write` `chat.send` callers unauthorized before command-specific gates run, and current main shows `mutateConfigFile` and `replaceConfigFile` remain outside the proposed guard.

## Solution Assessment

No. The hardening goal is sound, but the branch should be reworked around command capability policy plus complete runtime config mutation coverage rather than a blanket admin gate on internal chat commands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve read-only commands for write-scoped clients:** `src/gateway/server-methods/chat.ts:1308`
  - body: This changes internal `chat.send` to set `CommandAuthorized` only for `operator.admin`. `operator.write` slash commands are then rejected before command-specific gates such as `/config show` or `/acp sessions` can allow read-only access, breaking current least-privilege behavior.
  - confidence: 0.93
- **[P2] Let allowed directives survive rejected exec persistence:** `src/auto-reply/reply/directive-handling.impl.ts:279-283`
  - body: The early return for empty cleaned directive-only messages runs before the later per-field persistence logic. A mixed message such as `/think high /exec host=gateway` from an internal `operator.write` client reports the exec denial and skips the allowed thinking update.
  - confidence: 0.88
- **[P2] Guard every runtime config mutation API:** `src/plugins/runtime/runtime-config.ts:10-21`
  - body: The scope check wraps only deprecated `writeConfigFile`, but current plugin runtime exposes `mutateConfigFile` and `replaceConfigFile` as the primary mutation APIs. Plugin commands using those APIs can still mutate config without this admin check.
  - confidence: 0.84
