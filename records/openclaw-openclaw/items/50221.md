## Summary

Keep open: the PR is a plausible security-hardening direction with real behavior proof, but the current patch is not safe to merge because it uses a coarse internal command gate, can drop unrelated directive updates, and leaves the active runtime-config write APIs uncovered.

## What This Changes

The PR changes internal gateway scope handling by deriving chat.send command authorization from operator.admin, blocking /exec default persistence for non-admin internal clients, and propagating plugin command scopes around runtime config writes.

## Best Possible Solution

Model internal command capabilities centrally or per command, preserve read-only operator.write paths, require operator.admin for mutating operations, and apply any approved runtime-config write guard consistently across active APIs.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main gives a high-confidence path: operator.write chat.send clients are broadly denied, mixed directive-only messages can lose allowed updates, and active runtime config write APIs are not covered by the PR guard.

## Solution Assessment

No. The PR's admin-default chat gate is too coarse and the runtime-config sink guard is incomplete; the safer solution is maintainer-approved command-level capability policy plus complete active API enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve read-only commands for write-scoped clients:** `src/gateway/server-methods/chat.ts:1308`
  - body: This sets CommandAuthorized=false for every internal chat.send caller without operator.admin. resolveCommandSenderAuthorization treats that as a top-level gate, so read-only commands intentionally available to operator.write clients, such as /config show or /acp sessions, are rejected before command-specific policy can allow them.
  - confidence: 0.93
- **[P2] Keep allowed directives when rejecting exec persistence:** `src/auto-reply/reply/directive-handling.impl.ts:279-283`
  - body: This early return fires before later directive persistence for directive-only messages once /exec has options and cleaned is empty. Because parsing strips all directives from cleaned, a mixed message like /think high /exec host=gateway drops the allowed thinking update along with the rejected exec default change.
  - confidence: 0.88
- **[P2] Guard active runtime config write APIs:** `src/plugins/runtime/runtime-config.ts:9-20`
  - body: The guard wraps only writeConfigFile, which current main marks deprecated, while runtime config writes primarily use mutateConfigFile and replaceConfigFile. If sink-level hardening is approved, those active write APIs need equivalent enforcement or plugin commands can bypass this guard.
  - confidence: 0.86
