## Summary

The PR should stay open because it changes security-sensitive authorization policy and still has correctness blockers: the `chat.send` gate denies read-only internal slash commands for write-scoped clients, and the runtime-config hardening does not cover the current mutation APIs on main.

## What This Changes

The PR requires `operator.admin` for internal `chat.send` command authorization, `/exec` default persistence, and plugin runtime config writes, and propagates gateway client scopes into plugin command execution.

## Best Possible Solution

Extend the command-scope capability model so read-only commands remain available to `operator.write`, durable mutations require `operator.admin`, and plugin config mutation enforcement covers the current runtime APIs consistently.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR makes write-scoped internal `chat.send` callers unauthorized before command-specific gates run, and current main exposes runtime config mutation APIs that the patch does not guard.

## Solution Assessment

No. The hardening direction is valid, but this patch should be reworked around explicit command capability policy and complete runtime config mutation coverage rather than a blanket admin gate on internal chat commands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

  - body: This makes internal `chat.send` set `CommandAuthorized=false` unless the client has `operator.admin`, so command authorization rejects every slash command before command-specific gates can allow read-only paths such as `/config show` or `/acp sessions`.   - confidence: 0.88

  - body: The scope check wraps only deprecated `writeConfigFile`, but current plugin runtime mutations use `mutateConfigFile` and `replaceConfigFile`. Plugin commands using those APIs can still mutate config without this admin check after the branch is rebased.
  - confidence: 0.84
