## Summary

Keep this PR open: the auth hardening is relevant, but the current patch is not ready because it applies a blanket internal `chat.send` admin gate and misses current runtime config mutation APIs. The author’s follow-up also asks for maintainer direction on the command capability model before continuing.

## What This Changes

The PR requires `operator.admin` for internal `chat.send` command authorization, `/exec` default persistence, and plugin runtime config writes, with plugin command scope propagation and focused tests.

## Best Possible Solution

Model internal command capabilities centrally so read-only commands remain available to `operator.write`, mutating commands require `operator.admin`, and all approved runtime config write sinks share one guard.

## Reproduction Assessment

Yes. Source inspection of the PR head and current main shows the blanket `chat.send` gate runs before command-specific read/write gates, and current main exposes runtime config write APIs the PR does not guard.

## Solution Assessment

No. The patch is not the best fix because it uses a coarse admin gate and incomplete sink coverage where the current code needs command-level capability policy and complete runtime config write enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve read-only internal commands for write-scoped clients:** `src/gateway/server-methods/chat.ts:1311`
  - body: This sets `CommandAuthorized=false` for every internal `chat.send` caller without `operator.admin`. `resolveCommandSenderAuthorization` treats that as a top-level gate unless allow-from bypasses it, so read-only commands explicitly allowed for `operator.write` clients, such as `/config show`, are rejected before command-specific scope policy can run.
  - confidence: 0.93
- **[P2] Let allowed directives survive rejected exec persistence:** `src/auto-reply/reply/directive-handling.impl.ts:282-285`
  - body: This early return triggers for directive-only messages with any `/exec` options before the later per-field persistence block. Mixed messages such as `/think high /exec host=gateway` would drop the allowed thinking update along with the rejected `/exec` persistence.
  - confidence: 0.88
- **[P2] Guard all runtime config write APIs:** `src/plugins/runtime/runtime-config.ts:9-20`
  - body: The guard wraps only `writeConfigFile`. Current main's plugin runtime contract has `mutateConfigFile` and `replaceConfigFile` as primary write APIs, so the proposed sink guard leaves active mutation APIs uncovered.
  - confidence: 0.86
