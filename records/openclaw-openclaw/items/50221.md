## Summary

Keep this PR open for maintainer judgment: current main shipped a narrower action-level fix for the central `chat.send`, `/exec`, and phone-control paths, but the PR still raises a broader plugin runtime config-write policy question and its current diff has blocking authorization regressions.

## What This Changes

The PR gates internal gateway command authorization, `/exec` persistence, and plugin runtime config writes on `operator.admin`, while forwarding gateway scopes into plugin command execution and adding regression tests.

## Best Possible Solution

Keep the current action-level authorization model for chat commands, then make a maintainer-approved plugin config mutation policy explicit through command `requiredScopes`, targeted owner gates, or a versioned runtime write contract that covers current config mutation APIs.

## Reproduction Assessment

Yes. The PR regression is source-reproducible: an internal `operator.write` `chat.send` caller would get `CommandAuthorized=false`, and current command authorization rejects commands before read-only handlers such as `/config show` can apply their narrower gates.

## Solution Assessment

No. This PR is not the best fix as written because it uses a blanket admin gate at `chat.send`; the maintainable direction is command/action-level scope enforcement plus an explicit decision for plugin runtime config writes.
