## Summary

Keep this PR open: it is a plausible security-hardening direction, but the current branch still has blocking correctness and security concerns around least-privilege command access and incomplete runtime-config write coverage. Current main already has narrower scope propagation/directive work, so the remaining path needs maintainer policy direction rather than cleanup closure.

## What This Changes

This PR changes internal gateway scope handling for chat.send command authorization, /exec directive persistence, and plugin runtime config writes, with tests and an AsyncLocalStorage-backed plugin command scope context.

## Best Possible Solution

Model internal command capabilities centrally, preserve read-only operations for operator.write clients, require operator.admin for mutating operations, and apply any approved runtime-config write guard consistently across active APIs.

## Reproduction Assessment

Yes. Source inspection of the PR head diff and current main gives a high-confidence path: operator.write chat.send clients are broadly denied, mixed directive-only messages can lose allowed updates, and active runtime config write APIs are not covered by the PR guard.

## Solution Assessment

No. The PR uses a coarse admin gate and incomplete sink coverage; the maintainable solution is command-level capability policy plus complete runtime-config write enforcement if maintainers approve that boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve read-only internal commands for write-scoped clients:** `src/gateway/server-methods/chat.ts:1308`
  - body: This makes every internal chat.send caller without operator.admin set CommandAuthorized=false. The current command auth path treats that as a top-level gate, so read-only commands intentionally available to operator.write clients, such as /config show, are rejected before command-specific policy can allow them.
  - confidence: 0.93
- **[P2] Let allowed directives survive rejected exec persistence:** `src/auto-reply/reply/directive-handling.impl.ts:279-283`
  - body: This early return fires for directive-only messages with rejected /exec options before later code persists other allowed directive fields. A message like /think high /exec host=gateway would drop the allowed thinking update along with the rejected exec persistence.
  - confidence: 0.88
- **[P2] Guard all runtime config write APIs:** `src/plugins/runtime/runtime-config.ts:9-20`
  - body: The guard wraps only deprecated writeConfigFile, while current plugin runtime writes primarily use mutateConfigFile and replaceConfigFile. If sink-level hardening is approved, those active mutation APIs need equivalent enforcement.
  - confidence: 0.86
