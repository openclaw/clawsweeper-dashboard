## Summary

Keep open: the PR addresses a real internal-scope hardening area, but the current patch is not mergeable as-is because it over-gates `chat.send`, drops allowed mixed directive updates, and leaves active runtime config write APIs unguarded.

## What This Changes

The PR changes gateway `chat.send`, directive persistence, plugin command execution, runtime config writes, and tests to require `operator.admin` for selected internal-channel mutations.

## Best Possible Solution

Model internal command capabilities centrally or per command, preserve read-only `operator.write` paths, require `operator.admin` for mutating operations, and apply any approved runtime-config write guard consistently across active APIs.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main gives a high-confidence path: `operator.write` `chat.send` clients are broadly denied, mixed directive-only messages can lose allowed updates, and active runtime config write APIs are not covered by the PR guard.

## Solution Assessment

No. The PR hardens real surfaces, but the admin-default chat gate is too coarse and the runtime-config sink guard is incomplete; the safer direction is maintainer-approved command capability policy plus complete active API enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve read-only commands for write-scoped clients:** `src/gateway/server-methods/chat.ts:1308`
  - body: This makes every non-admin internal `chat.send` slash command fail before command-specific policy runs. Current main intentionally lets write-scoped internal clients use read-only paths such as `/config show` and `/acp sessions`, so the patch would force admin credentials for read-only inspection.
  - confidence: 0.93
- **[P2] Keep allowed directives when rejecting exec persistence:** `src/auto-reply/reply/directive-handling.impl.ts:279-283`
  - body: This early return runs before persistence of other directive-only updates once `/exec` has options and `cleaned` is empty. Since parsing removes all directives from `cleaned`, input like `/think high /exec host=gateway` drops the allowed thinking update instead of only rejecting the exec persistence.
  - confidence: 0.88
- **[P2] Guard active runtime config write APIs:** `src/plugins/runtime/runtime-config.ts:9-20`
  - body: The new guard wraps only `writeConfigFile`, but current runtime config mutations primarily use `mutateConfigFile` and `replaceConfigFile`, and `writeConfigFile` is deprecated. If sink-level hardening is intended, the active APIs need equivalent enforcement.
  - confidence: 0.86
