## Summary

Close as superseded: the remaining LLM retry feature is already tracked by #49376 and the active provider-level retry PR #61443, while this branch retries at the prompt wrapper in a way that conflicts with current failover/auth-profile rotation and still has concrete correctness issues.

## What This Changes

The PR adds a pi-embedded-runner LLM retry helper and tests, then wraps `activeSession.prompt` with fixed exponential backoff for retryable HTTP, network, and timeout-shaped failures.

## Best Possible Solution

Keep #49376/#61443 as the canonical path for configurable provider-call retry, and fold any useful retry-after parsing coverage into that design without adding a prompt-level wrapper that delays auth-profile rotation or model fallback.

## Reproduction Assessment

Yes for the proposed patch defects: static review shows the PR passes a signal that the retry primitive cannot honor, compares status values without string normalization, and retries same-session prompt failures before the current failover path can rotate profiles or models.

## Solution Assessment

No, this is not the best way to solve the issue as-is. The retry policy should be coordinated with the existing provider/failover classifier and configurable retry surface being tracked in #49376/#61443.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve failover before retrying provider failures:** `src/agents/pi-embedded-runner/run/attempt.ts:1386-1393`
  - body: Wrapping `activeSession.prompt` retries 429/5xx failures against the same session/profile before the existing prompt-error path can mark the profile failed, rotate credentials, or fall back. That can hammer a rate-limited profile and delay recovery, so retry needs to be coordinated with the current failover policy.
  - confidence: 0.86
- **[P2] Wire abort signals into retry sleeps:** `src/agents/pi-embedded-runner/llm-retry.ts:28`
  - body: The new config exposes `signal` and the call site passes `runAbortController.signal`, but `withLlmRetry` delegates to `retryAsync`, whose backoff sleep is not abortable. A canceled or timed-out run can keep sleeping and issue another prompt call after abort.
  - confidence: 0.9
- **[P2] Parse string HTTP status codes:** `src/agents/pi-embedded-runner/llm-retry.ts:59-60`
  - body: `status` and `statusCode` are compared directly to numeric literals, so errors shaped like `{ status: "429" }` or `{ statusCode: "503" }` bypass the intended retry classification. Current failover code already normalizes digit-string statuses before classification.
  - confidence: 0.82
