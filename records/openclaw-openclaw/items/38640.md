## Summary

Keep this PR open, but do not merge it as-is. Current main still has no prompt-level LLM retry wrapper, while the branch would retry failover-worthy provider errors before the existing auth-profile rotation/model-fallback path can run and still has concrete abort/status parsing defects.

## What This Changes

The PR adds a pi-embedded-runner LLM retry helper and tests, then wraps `activeSession.prompt` in `runEmbeddedAttempt` with fixed exponential backoff for 429, 5xx, network, and timeout-shaped failures.

## Best Possible Solution

Fold the useful retry-after parsing and retry tests into the chosen provider/failover retry design, reusing the existing failover classifier and making any backoff abortable, configurable, and compatible with auth-profile rotation and model fallback.

## Reproduction Assessment

Yes for the proposed patch defects: static checks show the PR passes a signal that `withLlmRetry`/`retryAsync` never use, compares string HTTP statuses as non-retryable, and wraps prompt errors before the current failover logic can mark profile failures or rotate. Current main verification shows the requested prompt-level retry behavior itself is still absent.

## Solution Assessment

No, this is not the best way to solve it as-is. A maintainable fix should live at the runner/provider retry boundary selected by maintainers, share current failover status parsing, and avoid delaying existing auth-profile rotation or model fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve failover before retrying provider failures:** `src/agents/pi-embedded-runner/run/attempt.ts:1386-1393`
  - body: Wrapping `activeSession.prompt` means a 429/5xx is retried against the same session/profile up to eight times before the existing prompt-error path can mark the profile failed, rotate, or fall back. That can hammer a rate-limited profile and delay recovery; retry should be coordinated with the current failover policy instead of sitting in front of it.
  - confidence: 0.86
- **[P2] Wire abort signals into retry sleeps:** `src/agents/pi-embedded-runner/llm-retry.ts:28`
  - body: The new config exposes `signal`, and the call site passes `runAbortController.signal`, but `withLlmRetry` never checks it and `retryAsync` sleeps without an abort race. After a cancel or timeout during backoff, the retry loop can keep sleeping and issue another prompt call after the run has already been aborted.
  - confidence: 0.9
- **[P2] Parse string HTTP status codes:** `src/agents/pi-embedded-runner/llm-retry.ts:59-60`
  - body: `status` and `statusCode` are compared directly to numeric literals, so errors shaped like `{ status: "429" }` or `{ statusCode: "503" }` bypass the intended retry classification. Current failover code already parses digit-string statuses, so this helper should normalize them before checking retryable status codes.
  - confidence: 0.82
