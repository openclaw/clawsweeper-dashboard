This should stay open. Current main still runs the Gemini CLI backend with `--skip-trust` and JSON prompt flags, without mapping OpenClaw's YOLO exec policy to Gemini CLI's tool-approval mode; the reporter's follow-up that `--yolo` fixes tools and memory is still actionable, while the Matrix CPU symptom still needs logs or a focused reproduction before it can be called fixed.

The issue reports that a Matrix setup using `google-gemini-cli` on Node v25.9 hits 100% CPU, fails VDS tool use, leaves long-term memory files empty, and returns generic processing errors after recent updates.

## Best Possible Solution

Add a provider-local Gemini CLI approval-mode normalizer that follows OpenClaw's existing exec-policy boundary, preserves explicit user backend args, and then ask for a Matrix retest or logs if CPU saturation remains after tools and memory work again.
