## Summary

Keep this open. Current main still runs the Gemini CLI backend with `--skip-trust` but no Gemini approval-mode/Yolo mapping, while the reporter's follow-up and the related Gemini CLI write-tools report show the tools and memory part is still actionable; the Matrix CPU symptom still lacks logs or a focused reproduction path, so it should not be called fixed yet.

## What This Changes

The issue reports a regression where Matrix plus `google-gemini-cli` on a Linux VPS pegs CPU, cannot use host tools, and leaves long-term memory empty unless Gemini CLI is launched with an approval-bypass flag.

## Best Possible Solution

Add a Google-plugin-owned Gemini CLI approval normalizer that mirrors the Claude CLI pattern: when OpenClaw's effective exec policy is YOLO, add Gemini's preferred `--approval-mode yolo` unless the user explicitly configured `--approval-mode`, `--approval-mode=...`, or `--yolo`; under non-YOLO policy leave Gemini approval mode unset. Update tests, docs, and changelog for the Gemini tools/memory part, then ask for Matrix logs or a retest if CPU saturation remains.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the tools/memory failure has a high-confidence source-level reproduction path because current main still omits Gemini's approval-mode flag and related #68216 describes the same observable no-write-tools behavior. The Matrix CPU symptom does not yet have high-confidence reproduction data because no logs, profiler output, or disposable homeserver setup were provided.

## Solution Assessment

Is this the best way to solve the issue? Yes for the concrete Gemini tools/memory failure if it is policy-gated and preserves explicit user backend args; an unconditional `--yolo` default is less safe. It is not enough to solve or close the Matrix CPU part without post-fix retest data or focused Matrix diagnostics.
