## Summary

Keep open. Current main and the latest stable release still drive the Gemini CLI backend without an explicit approval mode while also passing `--skip-trust`, the reporter's `--yolo` workaround matches the still-open Gemini write-tools report, and the Matrix CPU symptom remains unique enough that this should not be closed as a duplicate without logs or profiling.

## What This Changes

The issue reports Matrix + Gemini CLI failures: sustained CPU saturation, failed VDS file/tool access, and empty long-term memory files on a Node v25.9 Ubuntu install.

## Best Possible Solution

Resolve the Gemini tools/memory part with a maintainer-approved headless approval-mode policy, regression tests, docs, and live OAuth edit proof, while collecting separate Matrix CPU diagnostics before claiming a channel fix.

## Reproduction Assessment

Partially. The Gemini tools/memory slice is source-reproducible from current backend argv plus the upstream Gemini CLI approval-mode contract, but the Matrix CPU symptom is not high-confidence reproducible without logs, profiling, or a disposable homeserver scenario.

## Solution Assessment

Unclear. The reporter's unconditional `--yolo` workaround likely restores native Gemini write tools, but the safer maintainable path is a maintainer-approved approval-mode choice, possibly narrower than all-action auto-approval, plus separate Matrix CPU investigation.
