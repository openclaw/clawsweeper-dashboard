## What This Changes

The PR adds zero-filled daily buckets to bounded cached and uncached cost-usage summaries, caps all-time zero-fill at 366 days, and adds normal, sparse-all-time, and DST regression coverage.

## Best Possible Solution

Land the backend calendar-day zero-fill if maintainers accept dense bounded `usage.cost.daily` as the contract, with current-head terminal proof and release-note context in the PR body.

## Reproduction Assessment

Yes. Current main source shows both cost-summary paths only add days that have usage records, while the UI requests `usage.cost` for explicit date ranges, so quiet days in a bounded range are omitted from `daily`.

## Solution Assessment

Yes, with the compatibility caveat. Filling the backend summary before sorting in both cached and uncached paths is the cleanest fix because it keeps all `usage.cost` consumers consistent; a UI-only fill would leave gateway/CLI consumers sparse.
