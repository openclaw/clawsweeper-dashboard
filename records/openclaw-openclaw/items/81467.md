## Summary

Keep open: the sparse 30-day usage problem is real and the PR has useful behavior proof, but the patch introduces a concrete All-range payload/rendering regression and leaves a DST edge case in the day filler.

## What This Changes

The PR adds backend daily zero-fill logic and regression tests so usage cost summaries include days with no recorded activity.

## Best Possible Solution

Keep dense daily buckets for bounded calendar ranges, while preserving unbounded All behavior and iterating real calendar dates so DST cannot drop a day.

## Reproduction Assessment

Yes. Current main source only creates daily buckets from usage entries, and existing tests still expect one daily entry for a 30-day range with one populated day; I did not run tests because this review is read-only.

## Solution Assessment

No. Backend zero-fill is the right direction for the 30-day chart, but the implementation should not zero-fill the 1970-to-today All range and should walk calendar dates rather than fixed 24-hour ticks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid zero-filling the All range from 1970:** `src/infra/session-cost-usage.ts:487`
  - body: The usage page's All preset sends `startDate=1970-01-01`, and `usage.cost` passes that parsed range into `loadCostUsageSummaryCached`. Calling `fillMissingDays` unconditionally here will make the All response include one zero bucket for every day since 1970, even when there are only a few active days, bloating the API payload, daily CSV, and chart rendering. Please gate dense filling to bounded ranges or have the All path start filling at the first activity day.
  - confidence: 0.91
- **[P3] Walk calendar days instead of 24-hour ticks:** `src/infra/session-cost-usage.ts:655`
  - body: This helper advances by fixed 24-hour increments and stops when the formatted key reaches the end day. Across spring DST with a late-day start, for example Mar 7 23:30 to Mar 9 23:30 in `America/New_York`, that produces Mar 7 then Mar 9 and skips Mar 8, so days-based callers can still return fewer buckets than `summary.days`. Use local calendar-date increments, preferably normalized away from midnight, or actually probe adjacent hours before stopping.
  - confidence: 0.82
