The PR adds bounded zero-filled calendar-day rows to cached and uncached cost-usage daily summaries and adds regression tests for empty, sparse, all-time, and DST-spanning ranges.

## Best Possible Solution

Land the bounded zero-fill if maintainers accept dense daily rows as the canonical bounded-range `usage.cost` contract; otherwise keep the RPC sparse and add UI-specific or explicit opt-in dense output.

## Reproduction Assessment

Yes at source level: current main only emits days present in usage records, so empty days in a bounded range are omitted. The PR body also includes before/after terminal output from the production loader path against an on-disk OpenClaw home.

## Solution Assessment

Yes if maintainers accept the API-shape change: the backend summary is the right owner because the UI renders directly from `daily` and both cached and uncached loaders share the sparse source. The safer alternative for compatibility-sensitive consumers is an explicit dense mode or UI-only zero-fill.
