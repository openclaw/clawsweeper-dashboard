## Summary

Keep open: the patch addresses the bounded sparse-chart problem, but it also zero-fills all-time ranges, which would synthesize about 20,588 daily entries when the usage page's All range starts at 1970-01-01.

## What This Changes

The PR adds zero-valued missing calendar days to cost usage daily summaries and updates session cost usage tests to expect dense daily output.

## Best Possible Solution

Keep dense daily output for bounded ranges such as 7d and 30d, while preserving sparse or capped behavior for all-time and extremely wide usage ranges.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main only adds daily buckets for usage entries, and the PR body includes a before/after loader run showing sparse output before and dense output after.

## Solution Assessment

Is this the best way to solve the issue? No as written; backend zero-fill is the right boundary for bounded chart ranges, but it needs a guard or cap so all-time ranges do not generate a synthetic day for every date since 1970.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Avoid zero-filling all-time usage ranges:** `src/infra/session-cost-usage.ts:487`
  - body: The usage page's All range sends `startDate: "1970-01-01"` through the same `usage.cost` path, and `range=all` also maps to `startMs: 0`. Calling `fillMissingDays` unconditionally here will synthesize roughly 20k zero buckets for an all-time chart, causing a huge daily payload/DOM instead of preserving sparse all-history behavior. Gate the fill to bounded windows or cap the range before sorting.
