## Summary

Keep this active bug-fix PR open: it addresses a real sparse daily-series problem, has sufficient terminal proof for the central behavior, but the new day-fill loop still has a DST edge case that can skip a calendar day.

## What This Changes

The PR adds bounded zero-filling for missing daily usage buckets in cost summaries and updates tests for dense bounded ranges plus sparse all-time fallback.

## Best Possible Solution

Keep the backend dense-series fix, but make the fill helper calendar-day based and add a DST regression so bounded ranges always contain each local day exactly once.

## Reproduction Assessment

Yes, source inspection on current main shows both cached and uncached cost summaries only emit days that have usage entries. The PR body also supplies terminal before/after proof for the production loader against an on-disk state layout.

## Solution Assessment

No, not yet: backend densification is the right boundary, but the helper should iterate calendar dates rather than 24h millisecond steps so DST transitions cannot skip a day.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Iterate over calendar days across DST:** `src/infra/session-cost-usage.ts:665-674`
  - body: The new helper advances by exactly 24h and only formats that cursor. For a range whose start time is late evening before a spring-forward transition, the cursor can jump from one local date to the date after next, leaving the skipped day absent from `daily` even though bounded ranges are supposed to show every calendar day. Build the sequence from calendar dates, or otherwise probe enough around each step, and add a DST regression.
  - confidence: 0.87
