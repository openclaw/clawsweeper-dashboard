## Summary

Keep open for maintainer/API review: the updated branch appears to fix the sparse bounded daily chart data and addresses the prior DST bug, with no new blocking code findings found in this pass. ## What This Changes

The PR adds bounded zero-filling of missing calendar days to cached and uncached cost-usage daily summaries, plus regression tests for empty, sparse, all-time, and DST-spanning ranges.

## Best Possible Solution

Land the bounded zero-fill if maintainers accept dense daily rows as the canonical bounded-range `usage.cost` contract; otherwise preserve sparse RPC semantics and add a UI-specific or explicit opt-in dense mode.

## Reproduction Assessment

Yes at source level: current main only emits days that appear in usage entries, so empty calendar days in a bounded range are omitted. The PR body also provides before/after terminal output from the production loader path against an on-disk OpenClaw home.

## Solution Assessment

Yes, if maintainers accept the API-shape change: the backend summary is the right owner because the UI already renders based on `daily.length` and both cached and uncached loaders share the sparse source. The safer alternative for compatibility-sensitive consumers is an explicit dense mode or UI-only zero-fill.
