## Summary

Keep this PR open: the patch appears to fix the sparse daily usage chart behavior with no concrete code defect found, but it intentionally changes the bounded `usage.cost.daily` response from activity-only rows to dense zero-valued calendar rows, so it still needs maintainer/API-owner acceptance before merge.

## What This Changes

The PR adds bounded missing-day zero-fill to cached and uncached cost-usage summaries and adds regression coverage for empty ranges, sparse activity, all-time fallback, and DST-spanning ranges.

## Best Possible Solution

Land the bounded zero-fill if maintainers accept dense daily rows as the canonical bounded-range `usage.cost` contract; otherwise keep the public RPC sparse and add an explicit dense mode or UI-side fill.

## Reproduction Assessment

Yes at source level: current main only emits days that appear in usage entries, so empty calendar days in a bounded range are omitted. The PR body also provides before/after terminal output from the production loader path against an on-disk OpenClaw home.

## Solution Assessment

Yes, if maintainers accept the API-shape change: the backend summary is the right owner because the UI renders directly from `daily.length` and both cached and uncached loaders share the sparse source. The safer alternative for compatibility-sensitive consumers is an explicit dense mode or UI-only zero-fill.
