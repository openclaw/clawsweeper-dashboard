## Summary

## What This Changes

Adds capped zero-filled calendar-day entries to bounded cached and uncached cost-usage daily summaries, with regression tests for empty, sparse, all-time, and DST ranges.

## Best Possible Solution

Bounded `usage.cost.daily` should either be explicitly accepted as a dense calendar series by usage/API owners, or the public RPC should stay sparse and densification should move to the chart/export layer.

## Reproduction Assessment

Yes, source-reproducible. Current main only inserts daily rows while processing usage entries, and existing tests assert one daily row for a 30-day range with one active day; I did not run tests because this review is read-only.

## Solution Assessment

Yes, with an API-contract caveat. Backend zero-fill is the clean consistent fix across cached and uncached summaries, but usage/API owner acceptance is required because the shared response row semantics change.
