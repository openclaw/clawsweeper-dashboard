## Summary

Keep open: the PR addresses a real sparse daily usage-series bug and its terminal proof is sufficient, but the current patch still has a P2 DST correctness hole because the zero-fill helper advances fixed 24-hour millisecond steps and can skip a local calendar day.

## What This Changes

The PR adds bounded zero-filling of missing daily cost-usage buckets in cached and uncached summaries and updates `session-cost-usage` tests.

## Best Possible Solution

Keep backend densification for bounded usage ranges, but generate missing buckets by local calendar dates and preserve the sparse all-time fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows both cached and uncached usage-cost summaries only emit days with usage entries, and the PR body supplies real loader before/after terminal proof.

## Solution Assessment

No. Backend densification is the right boundary, but the current helper should iterate calendar dates rather than fixed 24-hour millisecond steps so DST transitions cannot drop a day.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Iterate missing days by calendar date:** `src/infra/session-cost-usage.ts:664-674`
  - body: The helper still advances `cursor` by exactly 24 hours and formats only that instant. In `America/New_York`, a bounded range from 2026-03-07 23:30 to 2026-03-10 23:30 produces `2026-03-07`, `2026-03-09`, and `2026-03-10`, so the spring-forward date is missing even though bounded ranges should include every calendar day. Build the series from local calendar dates and add a DST regression.
