## What This Changes

The PR adds bounded zero-filling of missing calendar days in `CostUsageSummary.daily` for cached and uncached usage-cost summaries, with regression tests for empty, sparse, all-time, and DST-spanning ranges.

## Best Possible Solution

Land the backend densification after usage/API owners accept the bounded dense daily contract; if that contract is not acceptable, keep the RPC sparse and densify only the usage UI.

## Reproduction Assessment

## Solution Assessment

Yes, if owners accept the response-shape change. Filling at the backend summary layer covers both cached and uncached paths and keeps UI/CLI/RPC consumers consistent; the safer alternative is UI-only fill if the public `usage.cost.daily` contract must remain sparse.
