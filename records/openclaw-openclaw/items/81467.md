## Summary

Keep open: the reported sparse-day behavior is real and the patch looks technically sound, but it intentionally changes bounded `usage.cost.daily` semantics, so it needs maintainer/API-owner acceptance before merge.

## What This Changes

Adds zero-valued daily buckets for missing days in cached and uncached cost-usage summaries for bounded ranges, with tests for empty, sparse, all-time, and DST-spanning windows.

## Best Possible Solution

Land the bounded zero-fill only if maintainers accept dense daily rows as the canonical bounded-range `usage.cost` contract; otherwise keep the gateway response sparse and add an explicit dense/UI-facing mode.

## Reproduction Assessment

Yes at source level: current main only emits days that appear in usage entries, so empty calendar days in a bounded range are omitted. The PR body also provides before/after terminal output from the production loader path against an on-disk OpenClaw home.

## Solution Assessment

Yes if maintainers accept the API-shape change: the backend summary is the right shared owner because both cached and uncached loaders produce the sparse series and the existing UI renderer already adapts to 30-day data. The safer compatibility alternative is an explicit dense mode or UI-only zero-fill.
