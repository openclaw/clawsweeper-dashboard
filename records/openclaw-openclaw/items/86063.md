## Label Justifications

- `P2`: The issue can repeatedly increase Anthropic cache-write cost and latency in long sessions, but it does not make the core runtime unavailable.
- `impact:session-state`: The same conversation turn can change model-facing serialization when it moves from active turn state into historical replay.

## Summary

Keep open: current main includes the merged tool-loop and timestamp/content-form repairs, but the remaining active-turn inbound-context versus bare persisted-history asymmetry is still present in source. No merged fix or open canonical PR fully owns that remaining policy decision.

## What This Changes

The issue requests stable Anthropic prompt-cache conversation-prefix bytes across turns so long sessions reuse cached history instead of rewriting it every turn.

## Best Possible Solution

Choose an owner-approved representation that keeps current-turn inbound metadata useful without placing volatile bytes inside the reusable prompt-cache prefix, then add active-to-historical byte-stability coverage and real Anthropic cache-read/write proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current source builds live prompts with currentInboundContext, stores the prepared user turn as bare transcript text, and strips inbound metadata only after a turn becomes historical. I did not run a fresh paid/live Anthropic current-main session in this read-only review.

## Solution Assessment

Unclear: the two merged repairs are good partial fixes, but they do not settle the inbound-metadata policy. The best fix requires a maintainer choice about whether metadata should be replayed deterministically, stored differently, or kept outside the cacheable prefix.
