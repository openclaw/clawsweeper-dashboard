## Label Justifications

## Summary

Keep open: two merged repairs address tool-loop marker placement and timestamp/content-form stability, but current main still deliberately keeps inbound metadata on the active user turn and strips it after the turn becomes historical. That position-dependent byte change matches the remaining reported cache-prefix failure, and no other open canonical item owns it.

## What This Changes

Fix Anthropic prompt-cache prefix instability caused by moving cache markers and user turns changing serialization between active and historical positions.

## Best Possible Solution

Adopt one owner-approved model-facing representation that keeps ephemeral inbound metadata out of stale historical context without changing bytes inside the reusable cache prefix, add focused active-to-historical byte-stability coverage, and verify on a real Anthropic session that cache reads grow with conversation history while writes remain near the new-turn delta.

## Reproduction Assessment

No complete current-main live reproduction is attached, but the failure remains source-reproducible with high confidence: current main intentionally preserves metadata on the active turn and strips it from history, while several recent live traces show the matching conversation-suffix cache-write behavior.

## Solution Assessment

No. The two merged fixes are appropriate for tool-loop marker movement and timestamp/content-form drift, but they do not resolve the inbound-metadata transition; the narrowest maintainable solution must preserve metadata policy without mutating the reusable prefix.
