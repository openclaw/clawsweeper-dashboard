## Label Justifications

- `P2`: This is a real prompt-cache/session-state stability bug with meaningful Anthropic cost impact, but it is not a crash, security bypass, or unusable core runtime.
- `impact:session-state`: The issue is about the same conversation turn having different model-facing bytes as current prompt versus historical replay, which is session/context state drift.

## Summary

Keep open: current main has partial fixes for tool-loop cache marker movement and timestamp/content-form serialization, but the remaining active-turn inbound context versus bare historical replay asymmetry is still present in source and is explicitly covered by current tests as intentional behavior. The remaining fix is a maintainer product/context-policy decision, not conservative cleanup.

## What This Changes

The issue requests stable Anthropic prompt-cache conversation-history prefixes across turns by preventing current user/tool-result payloads and cache-control markers from changing the cached prefix when replayed as history.

## Best Possible Solution

Select one stable current/historical inbound-context representation, implement it at the reply/LLM-boundary owner seam, and prove with regression tests plus real Anthropic cache-read/cache-write behavior.

## Reproduction Assessment

Yes, at source level: `currentInboundContext` is composed into the active model prompt while persisted transcript text remains bare, and the LLM boundary strips metadata from historical user messages while preserving it on the active turn. I did not run a live paid Anthropic billing reproduction in this read-only review.

## Solution Assessment

No complete current-main solution exists yet; the merged fixes are good partial repairs, but the remaining best fix depends on the maintainer-selected metadata/context policy. A narrow marker-only patch would not settle the underlying active-versus-history byte stability issue.
