## What This Changes

The issue reports Anthropic conversation-history cache misses caused by historical inbound metadata stripping and movement of the latest-message cache_control breakpoint.

## Best Possible Solution

Define a stable conversation-cache contract that preserves current-turn metadata safety while ensuring the prefix before the conversation-history cache breakpoint remains byte-identical across turns, with regression tests and redacted live Anthropic proof.

## Reproduction Assessment

Yes, at source level: current main preserves current-turn inbound metadata, strips the same metadata once that turn becomes historical, and moves cache_control to the latest user/tool_result block. I did not establish a live Anthropic console reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice: the report's one-line alternatives would each leave part of the problem or remove the conversation cache feature. The best fix needs maintainer selection between a stable anchor, metadata sidecar/storage change, or a documented fallback mode.
