## Label Justifications

- `P2`: The CLI gives misleading readiness information for local models, but the report confirms the underlying Ollama sessions remain usable.
- `impact:auth-provider`: The false status is caused by API-key/provider-auth availability logic being applied to a local provider that operates without that authentication.

## Summary

Current `main` still produces the reported mismatch, and the clean open PR https://github.com/openclaw/openclaw/pull/92363 explicitly owns this issue, so it should remain open until that implementation lands or is replaced. The bug is narrow and well-supported, but the proposed fix needs review because a loopback URL alone does not prove that its service or model is currently reachable.

## What This Changes

Make `openclaw models list --json` stop reporting usable local Ollama models as unavailable solely because they lack provider API-key authentication.

## Best Possible Solution

Preserve model-level discovery as the strongest signal, but add a narrowly defined no-auth local-provider fallback only for cases where discovery cannot represent the configured model, with regression coverage for both a working local model and a local model known to be unavailable.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction: pass a loopback Ollama model without provider auth and without its key in `availableKeys`; `toModelRow` returns `local: true` and `available: false`, matching the reporter's live output.

## Solution Assessment

No, the current linked implementation is not yet clearly the best fix because it equates every local-looking URL with availability. The narrower maintainable solution should preserve real discovery failures while avoiding auth-only false negatives for local providers.
