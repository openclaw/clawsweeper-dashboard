## Summary

Keep open. The underlying Copilot baseUrl precedence bug still appears valid on current main, but this PR is not merge-ready: the diff targets the old inline runner flow, introduces an eager Copilot token import, conflates user-configured and token-derived baseUrls during profile rotation, and lacks required real behavior proof.

## What This Changes

The PR adds conditional Copilot token-derived baseUrl assignment in embedded runner initial auth and refresh paths so a configured custom baseUrl is not overwritten.

## Best Possible Solution

Implement configured-vs-token-derived baseUrl precedence in the current auth-controller/provider-runtime seam, preserving explicit provider config while still updating token-derived endpoints on refresh and profile rotation.

## Reproduction Assessment

Yes, by source inspection: configure `models.providers.github-copilot.baseUrl`, then current main's Copilot `prepareRuntimeAuth` returns `token.baseUrl` and the auth controller overwrites runtime and effective model baseUrl unconditionally. I did not run a live Business account reproduction.

## Solution Assessment

No. The precedence rule is right, but the PR edits stale inline runner auth code and uses mutable `model.baseUrl` as the user-config marker; the safer fix belongs in the current auth-controller/provider-runtime boundary with explicit origin tracking.
