## Summary

Keep open: the reported Copilot baseUrl precedence bug is still source-reproducible on current main, but this PR targets stale inline runner auth logic, has blocking correctness findings, and still lacks required real behavior proof.

## What This Changes

The branch adds guarded Copilot token-derived baseUrl assignment in the embedded runner so user-configured provider baseUrls are not overwritten.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary, tracking explicit configured baseUrls separately from token-derived endpoints across first auth, refresh, and profile rotation.

## Reproduction Assessment

Yes, by source inspection. Configure models.providers.github-copilot.baseUrl: current main's Copilot prepareRuntimeAuth returns token.baseUrl, and the auth controller applies that prepared baseUrl to runtime/effective models without an explicit-config guard.

## Solution Assessment

No. The precedence rule is right, but this diff edits stale run.ts inline auth flow and uses mutable model.baseUrl as the user-config marker; the safer fix belongs in auth-controller/provider runtime with explicit-origin tracking.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

  - body: This top-level import pulls Copilot token code into embedded-runner startup even though Copilot runtime behavior is otherwise loaded lazily through provider runtime. Keep the default-host comparison inside the lazy provider/auth-controller path so non-Copilot runs do not eagerly evaluate Copilot runtime code.
  - confidence: 0.9

  - body: The guard treats any non-default model.baseUrl as user-configured, but this field is also mutated by the first token exchange. Switching between mixed Copilot profiles can retain the old token-derived host and route the new token to the wrong endpoint.
  - confidence: 0.88
