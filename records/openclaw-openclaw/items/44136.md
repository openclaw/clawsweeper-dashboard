## Summary

Keep this PR open: it targets a real Copilot baseUrl precedence bug, but the branch is stale/conflicting and the proposed code is not safe to merge against current main.

## What This Changes

The PR adds guarded Copilot token-derived baseUrl assignment in `src/agents/pi-embedded-runner/run.ts` during initial auth and refresh so explicit user baseUrl values are not overwritten.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary, preserving explicit user baseUrls while allowing token-derived endpoints to refresh across auth refresh and profile changes.

## Reproduction Assessment

Yes, by source inspection. Current main lets the Copilot provider return a token-derived `baseUrl`, and the auth controller applies it to runtime and effective models without checking for an explicit configured provider baseUrl.

## Solution Assessment

No. The precedence rule is the right direction, but this PR mutates stale `run.ts` logic and uses `model.baseUrl` as the user-config marker, which is not robust after token-derived endpoints have already been written or profiles rotate.
