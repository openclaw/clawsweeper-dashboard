## Summary

Keep open. The PR targets a real GitHub Copilot auth-provider bug, but the branch is not safe to merge as-is because it is conflicted against current main, patches the old inline runner flow, conflates configured and token-derived baseUrl state, and still lacks real after-fix proof.

## What This Changes

The branch adds conditional Copilot token-derived baseUrl assignment in the embedded runner auth and refresh paths so non-default user-configured baseUrl values are not overwritten.

## Best Possible Solution

Implement configured-vs-token-derived Copilot baseUrl precedence in the current auth-controller/provider-runtime path with explicit origin tracking, preserving explicit provider config while still refreshing and rotating token-derived endpoints correctly.

## Reproduction Assessment

Yes by source inspection, with medium confidence: current main accepts a Copilot token-derived baseUrl from prepareRuntimeAuth and applies it to runtime and effective models without distinguishing explicit provider config. I did not run a live Copilot Business-account reproduction.

## Solution Assessment

No, not as currently patched. The precedence rule is valid, but it should be ported to auth-controller/provider-runtime and avoid using model.baseUrl as both the config marker and mutable runtime state.
