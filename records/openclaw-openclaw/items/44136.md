## Summary

Keep open: current main still has the Copilot baseUrl precedence gap, but this branch is not ready to merge because it edits the old inline runner auth flow, eagerly imports Copilot token helpers, loses baseUrl origin across profile rotation, and still lacks real behavior proof.

## What This Changes

The PR adds guarded Copilot token-derived baseUrl assignment in embedded-runner auth paths so explicit provider baseUrls are not overwritten.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary, carrying explicit configured baseUrl origin separately from token-derived endpoints across first auth, refresh, and profile rotation.

## Reproduction Assessment

Yes, by source inspection. Configure models.providers.github-copilot.baseUrl: current main's Copilot prepareRuntimeAuth returns token.baseUrl, and the auth controller applies that prepared baseUrl to runtime/effective models without an explicit-config guard.

## Solution Assessment

No. The precedence rule is right, but this diff edits stale run.ts inline auth flow and uses mutable model.baseUrl as the user-config marker; the safer fix belongs in auth-controller/provider runtime with explicit-origin tracking.
