## Summary

Keep open: current main still has the Copilot baseUrl precedence bug, but this PR is not merge-ready because it edits the old inline runner auth path, eagerly imports Copilot token code, can keep a stale token-derived host after profile rotation, and lacks external real-behavior proof.

## What This Changes

The PR adds guarded Copilot token-derived baseUrl assignment in `src/agents/pi-embedded-runner/run.ts` for initial auth and token refresh, updating both `model` and `effectiveModel`.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary with explicit configured-vs-token-derived baseUrl origin tracking across first auth, refresh, and profile rotation.

## Reproduction Assessment

Yes, by source inspection: configure `models.providers.github-copilot.baseUrl`; current main resolves that into the model, then Copilot `prepareRuntimeAuth` returns `token.baseUrl` and the auth controller applies it unconditionally to runtime and effective models. I did not run a live Business or Enterprise account reproduction.

## Solution Assessment

No. The precedence rule is right, but this diff edits stale inline `run.ts` auth flow and uses mutable `model.baseUrl` as the user-config marker; the safer fix belongs in the current auth-controller/provider-runtime seam with explicit origin tracking.
