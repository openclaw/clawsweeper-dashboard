## Summary

Keep open: the requested Copilot baseUrl precedence is still a valid auth-provider bug on current main, but this branch is not merge-ready because it patches the old inline runner flow, conflicts with main, lacks real behavior proof, and can keep stale token-derived endpoints during profile rotation.

## What This Changes

The PR conditionally applies Copilot token-derived baseUrl values in the embedded runner auth and refresh paths, preserving non-default user-configured baseUrl values and propagating the endpoint to effectiveModel.

## Best Possible Solution

Implement configured-vs-token-derived baseUrl precedence in the current auth-controller/provider-runtime seam with explicit origin tracking, preserving explicit provider config while still refreshing and rotating token-derived endpoints correctly.

## Reproduction Assessment

Yes by source inspection: current main receives a Copilot token-derived baseUrl from prepareRuntimeAuth and applies it to runtime and effective models without distinguishing explicit provider config. I did not run a live Copilot Business-account reproduction.

## Solution Assessment

No, not as currently patched. The precedence rule is right, but the implementation should be ported to auth-controller/provider-runtime and avoid using model.baseUrl as both the config marker and mutable runtime state.
