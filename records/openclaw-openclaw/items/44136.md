## Summary

Keep open: the underlying Copilot baseUrl precedence bug is still source-reproducible on current main, but this PR is not merge-ready because it edits the old inline runner path, keeps a stale mutable-baseUrl guard, and lacks after-fix real behavior proof.

## What This Changes

The PR adds guarded Copilot token-derived baseUrl assignments in the old `src/agents/pi-embedded-runner/run.ts` initial-auth and refresh paths so token endpoints only replace missing or default model baseUrls.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary, preserving explicit provider baseUrls while allowing token-derived Copilot endpoints to refresh across auth refresh and profile changes.

## Reproduction Assessment

Yes, by source inspection. Configure a GitHub Copilot provider `baseUrl`, then current main's Copilot runtime auth returns a token-derived `baseUrl` and the auth controller applies it to the runtime and effective models before request use.

## Solution Assessment

No. The precedence rule is the right direction, but this branch mutates stale `run.ts` logic and uses `model.baseUrl` as the user-config marker instead of fixing the current auth-controller seam with a reliable explicit-config signal.
