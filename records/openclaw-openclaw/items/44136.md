## Summary

Keep open: the underlying precedence bug is still source-reproducible on current main, but this PR is not merge-ready because it patches stale inline runner code, has correctness/import-boundary issues, and lacks external real behavior proof.

## What This Changes

The PR adds guarded Copilot token baseUrl assignments in `src/agents/pi-embedded-runner/run.ts`, updating both `model` and `effectiveModel` only when the current base URL is empty or the default Copilot host.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary with explicit configured-versus-token-derived base URL state, preserving custom endpoints while updating token-derived endpoints on initial auth, refresh, effective models, and profile rotation.

## Reproduction Assessment

Yes. By source inspection, current main’s Copilot provider returns `token.baseUrl` from `prepareRuntimeAuth`, and the auth controller unconditionally writes `preparedAuth.baseUrl` onto runtime and effective models, so an explicit `models.providers.github-copilot.baseUrl` can be overwritten.

## Solution Assessment

No. The precedence rule is right, but this branch edits stale inline runner code and uses `model.baseUrl` as the state marker, which is not robust once a token-derived endpoint has already been written.

## Review Findings

  - body: This top-level import pulls the Copilot token helper into embedded-runner startup even though the current Copilot runtime path is loaded lazily through the provider runtime.   - confidence: 0.9

  - confidence: 0.88
