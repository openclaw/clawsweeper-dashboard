## Summary

Keep open: the reported Copilot baseUrl precedence bug is still valid on current main, but this PR is not mergeable as-is because it patches the old inline runner flow, violates the lazy import boundary, and still mishandles profile rotation.

## What This Changes

The PR adds conditional GitHub Copilot token-derived baseUrl assignment in `src/agents/pi-embedded-runner/run.ts` during initial auth and token refresh so explicit user-configured endpoints are not overwritten.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/GitHub Copilot runtime-auth boundary, preserving explicit custom endpoints while keeping token-derived endpoints synchronized for initial auth, refresh, effective models, and profile rotation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: have GitHub Copilot `prepareRuntimeAuth` return a token-derived `baseUrl` while the runtime model starts with a custom `baseUrl`, then `initializeAuthProfile()` applies that prepared `baseUrl` unconditionally to runtime and effective models.

## Solution Assessment

No. The intended precedence rule is right, but this branch is not the best current implementation because it patches stale inline runner code, adds a static import across a lazy boundary, and cannot safely distinguish user-configured endpoints from previous token-derived endpoints.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep the Copilot token helper lazy:** `src/agents/pi-embedded-runner/run.ts:40`
  - body: This adds a top-level import from `../../providers/github-copilot-token.js` while the same file still dynamically imports that module at the Copilot-only call sites. That makes the lazy boundary ineffective for every embedded-runner start; keep the default endpoint lookup behind the existing lazy/runtime-auth path.
  - confidence: 0.9
- **[P2] Track token-derived base URLs across profile rotation:** `src/agents/pi-embedded-runner/run.ts:637-641`
  - body: After the first token exchange this writes `model.baseUrl` to a token-derived endpoint, so the next profile switch sees a non-default `model.baseUrl` and treats it as user-configured. Mixed individual/business profiles can keep the prior endpoint and send the new token to the wrong Copilot host.
  - confidence: 0.88
