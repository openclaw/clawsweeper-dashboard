## Summary

Keep open: the precedence bug is still visible in current main, but this PR is not merge-ready because it patches stale inline runner code, has concrete correctness issues, and lacks after-fix real behavior proof.

## What This Changes

The branch conditionally applies Copilot token-derived base URLs in `src/agents/pi-embedded-runner/run.ts` only when the model has no base URL or the default Copilot URL, updating both `model` and `effectiveModel`.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary with explicit configured-versus-token-derived base URL state, preserving custom endpoints while updating token-derived endpoints on initial auth, refresh, effective models, and profile rotation.

## Reproduction Assessment

Yes, by source inspection rather than live account testing. Current main returns a token-derived Copilot `baseUrl` from `prepareRuntimeAuth` and unconditionally writes it over runtime/effective models despite the documented explicit provider `baseUrl` override path.

## Solution Assessment

No. The intended precedence rule is right, but this branch patches stale inline runner code and uses `model.baseUrl` as the state marker, which is not robust for profile rotation after a token-derived endpoint has already been written.
