## Summary

Keep this PR open for maintainer review, but do not merge it as-is: the intended Copilot baseUrl precedence is still relevant, while the branch targets an old runner path, conflicts with current main, lacks inspectable real behavior proof, and can misroute mixed-profile Copilot auth by treating prior runtime baseUrl state as user configuration.

## What This Changes

The branch conditionally applies a token-derived GitHub Copilot API baseUrl in the embedded runner only when the model has no custom baseUrl or still uses the default.

## Best Possible Solution

Port the intended precedence rule into the current provider-runtime/auth-controller path so explicit configured proxy/custom baseUrl is distinguishable from runtime token baseUrl, add focused tests for default, explicit, and mixed-profile rotation cases, and require redacted real Copilot proof before merge.

## Reproduction Assessment

Yes from source inspection: current main's Copilot provider returns token-derived `baseUrl` and the auth controller applies it unconditionally, so a focused auth-controller test can show explicit configured baseUrl being overwritten. I did not run a live Copilot Business account.

## Solution Assessment

No. The precedence rule is valid, but this branch implements it in the stale embedded-runner flow and needs a current-main port that tracks explicit config separately from mutable runtime auth state.
