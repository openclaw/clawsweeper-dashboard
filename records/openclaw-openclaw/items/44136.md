## Summary

Keep this PR open because it targets a real GitHub Copilot auth-routing gap, but it is not merge-ready: the branch conflicts with current main, lacks inspectable real behavior proof, and still conflates user-configured baseUrl with mutable token-derived runtime baseUrl.

## What This Changes

The PR adds conditional GitHub Copilot token baseUrl assignment in the embedded runner during initial auth and token refresh, updating both model and effectiveModel only when the current baseUrl is absent or default.

## Best Possible Solution

Implement configured-vs-token-derived Copilot baseUrl precedence in the current auth-controller/provider-runtime path with explicit origin tracking, focused regression coverage, and redacted Business or Enterprise proof.

## Reproduction Assessment

Yes from source inspection: a focused test can mock two Copilot auth profiles returning different token-derived baseUrls and show the PR guard preserving the first runtime host as if it were user configuration. I did not run a live Copilot Business account.

## Solution Assessment

No. The precedence rule is valid, but the implementation should be ported to the current auth-controller/provider-runtime path and should avoid using model.baseUrl as both explicit config and mutable runtime state.
