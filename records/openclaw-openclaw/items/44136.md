## Summary

Keep open: the requested precedence rule is still useful, but this branch edits a stale inline runner path while current main owns runtime auth in the auth controller, and the PR still lacks attached real-environment proof.

## What This Changes

The PR conditionally applies Copilot token-derived baseUrl values in the old embedded runner auth and refresh paths so explicit provider baseUrl config is not overwritten.

## Best Possible Solution

Implement configured-vs-token-derived baseUrl precedence in the current auth-controller/provider-runtime seam with explicit origin tracking, preserving explicit provider config while still updating token-derived endpoints on refresh and profile rotation.

## Reproduction Assessment

Yes by source inspection: configure models.providers.github-copilot.baseUrl, then current main's Copilot prepareRuntimeAuth returns token.baseUrl and the auth controller overwrites runtime and effective model baseUrl unconditionally. I did not run a live Business-account reproduction.

## Solution Assessment

No, not as currently patched. The precedence rule is right, but the implementation should be ported to the current auth-controller/provider-runtime seam and avoid using model.baseUrl as both config marker and mutable runtime state.
