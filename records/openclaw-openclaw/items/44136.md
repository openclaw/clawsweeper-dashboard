## Summary

Keep open: the requested Copilot baseUrl precedence fix is still valid on current main, but this patch targets stale inline runner auth code, introduces a lazy-import regression, mishandles profile rotation, and lacks real behavior proof.

## What This Changes

The PR adds conditional Copilot token-derived baseUrl assignment in the embedded runner's initial auth and token refresh paths, including updates to both runtime and effective models.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary, preserving explicit user endpoints while refreshing token-derived endpoints on initial auth, refresh, effective model updates, and profile rotation.

## Reproduction Assessment

Yes, by source inspection. Current main passes Copilot token.baseUrl through prepareRuntimeAuth and the auth controller unconditionally copies preparedAuth.baseUrl onto the runtime and effective models, so an explicit provider baseUrl can be replaced.

## Solution Assessment

No. The precedence rule is right, but this PR edits stale run.ts logic and uses model.baseUrl as the state marker, which is not robust after a token-derived endpoint has already been written or profiles rotate.

## Review Findings

Overall correctness: patch is incorrect

  - body: This top-level import pulls the Copilot token helper into embedded-runner startup even though the Copilot runtime path is loaded lazily through provider runtime. Keep the default-host comparison inside the lazy provider/auth-controller path so non-Copilot runs do not eagerly evaluate Copilot runtime code.
  - confidence: 0.9

  - confidence: 0.88
