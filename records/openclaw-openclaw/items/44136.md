## Summary

The Copilot baseUrl precedence bug still appears valid on current main, but this branch should not merge as-is because it patches stale inline runner code, has correctness findings, and lacks external real behavior proof.

## What This Changes

The PR conditionally applies Copilot token-derived base URLs in `src/agents/pi-embedded-runner/run.ts` only when the model has no base URL or the default Copilot URL, updating both `model` and `effectiveModel`.

## Best Possible Solution

Implement the precedence rule in the current auth-controller/provider-runtime boundary with explicit configured-versus-token-derived base URL state, preserving custom endpoints while updating token-derived endpoints on initial auth, refresh, effective models, and profile rotation.

## Reproduction Assessment

Yes, by source inspection rather than live account testing. A configured `models.providers.github-copilot.baseUrl` can be overwritten because Copilot `prepareRuntimeAuth` returns `token.baseUrl` and `applyPreparedRuntimeRequestOverrides` unconditionally writes it to runtime and effective models.

## Solution Assessment

No. The intended precedence rule is right, but this branch patches stale inline runner code and uses `model.baseUrl` as the state marker, which is not robust for profile rotation after a token-derived endpoint has already been written.
