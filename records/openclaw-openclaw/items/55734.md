## Summary

Keep this open. Current main still lets Chrome internal CDP UI targets enter the browser tab list and default selection path, and the directly related fix PR was closed unmerged, so the reported navigation bug remains actionable.

## What This Changes

The issue requests that CDP browser tab selection skip Chrome internal UI targets such as `chrome://omnibox-popup.top-chrome/`, `chrome://tab-search.top-chrome/`, and `chrome-untrusted://` so navigation lands in real web-content tabs.

## Best Possible Solution

Add a shared browser-plugin predicate for selectable web-content CDP targets, exclude Chrome internal schemes/top-chrome surfaces consistently before default and explicit target resolution, and add focused regression tests for raw `/json/list`, Playwright-backed listing, and fallback to a real `about:blank` tab.

## Reproduction Assessment

Yes. The issue provides a concrete CDP target list, and current main would map a `chrome://omnibox-popup.top-chrome/` entry as a `page` candidate before navigation; a unit test using that fixture can reproduce without launching live Chrome.

## Solution Assessment

Yes. A browser-plugin-local target predicate is the narrow maintainable direction because the bug is in browser CDP target selection, not core routing or a missing plugin API; the prior unmerged PR is useful source context but still needs a current-main repair.
