## Summary

Keep this open. Current main still lets browser CDP target selection treat Chrome internal UI targets as selectable page tabs, and the directly related fix PR #55740 was closed unmerged, so the reported navigation bug remains valid and narrow enough for a repair PR.

## What This Changes

The issue requests filtering Chrome internal CDP targets such as top-chrome omnibox and tab-search surfaces so browser navigation selects real web-content tabs.

## Best Possible Solution

Add a shared browser-plugin predicate for selectable web-content CDP targets, apply it consistently to raw CDP listing, Playwright-backed listing, default and explicit target resolution, and managed tab cleanup, then cover omnibox popup, tab search, chrome-untrusted, and fallback-to-real-tab behavior with focused tests.

## Reproduction Assessment

Yes. The report includes a concrete CDP target list where an internal omnibox target is type page with a WebSocket URL, and current main's selection code would choose such a target when it appears first among page candidates.

## Solution Assessment

Yes. The narrow maintainable fix is to centralize the selectable-target rule in the browser plugin and reuse it across both raw CDP and Playwright target discovery instead of adding one-off checks only at navigation time.

## Repair Work Prompt

Fix the browser plugin CDP target-selection bug reported in https://github.com/openclaw/openclaw/issues/55734. Current main can select Chrome internal UI targets such as chrome://omnibox-popup.top-chrome/, chrome://tab-search.top-chrome/, and chrome-untrusted:// targets because raw CDP and Playwright page listing preserve them and ensureTabAvailable only prefers type=page. Add one shared predicate/helper for selectable web-content browser tabs, use it in raw /json/list tab mapping or selection, Playwright-backed listing, implicit/default selection, explicit target resolution where appropriate, and managed tab cleanup so internal Chrome UI targets are not navigated or closed as normal web tabs. Preserve third-party/browser profile behavior and do not move this into core. Add focused regression tests for omnibox popup, tab-search, chrome-untrusted/internal scheme filtering, and fallback/opening of a real about:blank tab when only internal targets are present. Use closed PR #55740 only as source context; do not depend on that branch. Add a user-facing changelog fix entry in the active version if required, crediting @Demine4 and @ShionEria when appropriate.
