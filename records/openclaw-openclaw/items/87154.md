## Summary

Keep open: the hardening direction is useful, but the current branch still has a P1 authorization-parity blocker. The fallback button recheck bypasses QQBot's bridge command authorization path, so valid command users can be rejected, and real QQ callback proof is still missing.

## What This Changes

The PR marks QQBot no-`execApprovals` fallback approval authorization as implicit, gates native fallback button clicks behind command-style actor authorization, exports implicit marker helpers in the Plugin SDK, and updates docs plus regression tests.

## Best Possible Solution

Route fallback button authorization through the same QQBot bridge command-authorization path used by slash commands, keep the intended fail-closed hardening, add `accessGroup:` and `accounts.default` regression coverage, and attach redacted real QQ callback proof.

## Reproduction Assessment

Yes. Source inspection of current main shows no-`execApprovals` QQBot fallback authorization returns authorized and the interaction handler resolves approvals after that result; I did not run live QQ proof in this read-only review.

## Solution Assessment

No, not yet. The intended hardening belongs in the QQBot fallback button path, but the current implementation should reuse the bridge command-authorization adapter so native buttons match slash-command authorization for access groups and `accounts.default`.
