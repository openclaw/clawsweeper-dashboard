## Summary

Keep open. Current main distinguishes Gateway WebSocket clients internally, but it does not provide a stable trusted WebChat client-type field for agent/runtime context, and related prompt-metadata work is still active in #47216.

## What This Changes

Request to expose WebChat client type, such as native Mac app, browser, or mobile web, to agents/runtime context.

## Best Possible Solution

Add an explicit, sanitized client metadata field only after deciding whether it belongs in trusted inbound metadata, command context, presence, or a UI-only/debug surface.

## Reproduction Assessment

Not applicable. This is a feature request; source inspection shows current main has client metadata at Gateway connect time but no dedicated trusted agent/runtime client-type field.

## Solution Assessment

Unclear. Reusing existing Gateway connect metadata is preferable to adding a custom header, but maintainers should first decide how much transport client identity belongs in model-visible prompt context.
