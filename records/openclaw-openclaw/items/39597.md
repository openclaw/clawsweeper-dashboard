## Summary

Keep open. Current main already distinguishes browser and native clients in Gateway connect metadata, presence, and logs, but it still does not expose a reviewed client-type field to agent-visible WebChat context; the remaining work needs product and security contract approval rather than cleanup.

## What This Changes

The issue requests agent-visible WebChat client-type metadata so agents can distinguish native Mac app, browser, and mobile web clients.

## Best Possible Solution

Define a stable allowlisted Gateway client metadata contract sourced from connect.client, document which fields are client-declared, expose only approved low-entropy fields through MsgContext or inbound metadata, and test browser plus native WebChat behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that Gateway clients already send structured metadata while agent-visible WebChat context still lacks a stable client-type field.

## Solution Assessment

No for the suggested ad hoc header or config switch. The maintainable path is to reuse existing Gateway connect metadata through a reviewed allowlist and stable agent-facing exposure contract.
