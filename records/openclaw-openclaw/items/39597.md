## Summary

Keep open. Current main and v2026.6.10 have Gateway connection-level client IDs, but WebChat/native turns still do not expose a trusted client-type surface to the agent runtime; the related Gateway logging issue only partially overlaps.

## What This Changes

Request to distinguish WebChat client types such as Mac app, browser, and mobile web in agent-facing runtime context instead of only exposing `channel=webchat`.

## Best Possible Solution

Define one additive Gateway-owned client metadata contract with explicit trust semantics, then propagate it through WebChat/native send paths into agent runtime metadata and docs/tests.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main has connection client metadata, but `chat.send`, WebChat `MsgContext`, and the runtime prompt still lack an agent-facing client-type field.

## Solution Assessment

Unclear. The suggested custom header or config flag is plausible, but the best fix should be a single Gateway/client metadata contract with explicit trust semantics rather than a WebChat-only side channel.
