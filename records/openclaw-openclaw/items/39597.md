## Summary

Keep open. Current main already distinguishes Gateway clients during the WebSocket connect handshake, but there is still no stable agent-visible client-type field; exposing that metadata is an API and trust-model decision rather than stale cleanup.

## What This Changes

Expose enough WebChat client metadata for agents to distinguish native Mac app, browser, and mobile web clients.

## Best Possible Solution

Define a small allowlisted Gateway-client metadata contract from connect.client, expose only approved stable fields through MsgContext/inbound metadata, and document/test Mac and browser WebChat behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main path: clients identify themselves during Gateway connect, but the agent metadata contract does not expose a dedicated stable client type.

## Solution Assessment

No. The suggested custom header or config flag is not the best starting point because the Gateway WebSocket connect protocol already carries structured client metadata; the missing piece is a maintained exposure contract.
