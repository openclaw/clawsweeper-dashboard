## Summary

Keep open. Current main already distinguishes Gateway clients during connection setup, but the agent-facing metadata path still lacks a stable WebChat client-type field, so this remains a feature/API contract decision rather than cleanup.

## What This Changes

The issue requests runtime or agent-context metadata that distinguishes WebChat clients such as the native Mac app, browser, and mobile web.

## Best Possible Solution

Define an allowlisted client metadata contract from Gateway connect metadata, expose only approved stable fields through MsgContext/inbound metadata, and cover browser and Mac WebChat behavior in tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main path: clients identify themselves during Gateway connect, but agents do not receive a stable client-type field.

## Solution Assessment

No. The suggested custom header or config flag is not the best starting point because the existing Gateway WebSocket connect protocol already carries structured client metadata; the missing piece is a maintained exposure contract.
