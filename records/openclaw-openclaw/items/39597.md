## Summary

Keep open. Current main already has structured Gateway client metadata for browser and native clients, but agents still do not have a stable client-type field; exposing that self-declared metadata needs maintainer API and trust-model judgment.

## What This Changes

The issue requests a stable way for agents to distinguish WebChat client types such as native Mac app, browser, and mobile web in runtime context.

## Best Possible Solution

Define an allowlisted Gateway-client metadata contract from connect.client, expose only approved stable fields through MsgContext/inbound metadata, and document/test browser and native WebChat behavior.

Not applicable as a feature request. Source inspection gives a high-confidence current-main path: Gateway clients identify themselves during connect, but agent-visible metadata has no dedicated stable client-type field.

## Solution Assessment

No for the suggested ad hoc header or config switch. The Gateway connect protocol already carries structured client metadata, so the maintainable path is a reviewed allowlist and exposure contract.
