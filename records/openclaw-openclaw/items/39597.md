## Summary

Keep open. Current main and v2026.6.10 already distinguish clients at WebSocket connect time, but WebChat/native turns still do not expose a trusted client-type field in `chat.send` or the agent runtime line; the remaining work needs product/security review for trust semantics.

## What This Changes

The issue requests an agent-visible way to distinguish WebChat client types such as native Mac app, browser Control UI, and mobile web.

## Best Possible Solution

Define one additive Gateway-owned client metadata contract with explicit trusted versus untrusted semantics, then propagate it through WebChat/native send paths into runtime metadata with docs and focused tests.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main carries client metadata in the WebSocket connect frame, but `chat.send`, WebChat `MsgContext`, and the runtime prompt still lack an explicit agent-facing client-type field.

## Solution Assessment

Unclear. The suggested custom header or config flag is plausible, but the maintainable fix should reuse one Gateway/client metadata contract with explicit trust semantics rather than adding a WebChat-only side channel.
