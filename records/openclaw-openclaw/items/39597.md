## Summary

Keep open. Current main already distinguishes Gateway clients during the WebSocket connect handshake, but the agent-facing context and trusted inbound metadata still lack a stable client-type field; deciding which client fields to expose, and whether they are trusted or untrusted, needs maintainer API judgment.

## What This Changes

Expose a stable way for agents to distinguish WebChat client types such as the native Mac app, browser WebChat, and mobile web clients.

## Best Possible Solution

Define a small allowlisted Gateway-client metadata contract from connect.client, expose only approved stable fields through MsgContext/inbound metadata, and document/test browser and native WebChat behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main path: clients identify themselves during Gateway connect, but agent-visible metadata still has no dedicated stable client-type field.

## Solution Assessment

No. The suggested custom header or config flag is not the best starting point because the Gateway connect protocol already carries structured client metadata; the missing piece is a maintained exposure contract.
