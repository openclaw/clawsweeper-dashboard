## Summary

Keep open. Current main already carries structured Gateway client metadata for browser/native clients and uses it for logs/presence, but agent-visible WebChat context still lacks a stable reviewed client-type field; exposing it needs an API, privacy, and trust-model decision.

## What This Changes

Request to let agents distinguish WebChat client types such as native Mac app, browser, and mobile web instead of only seeing WebChat channel context.

## Best Possible Solution

Define a stable allowlisted Gateway client metadata contract sourced from connect.client, expose only approved fields through MsgContext or inbound metadata, and document/test both browser and native WebChat behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that clients already send structured connect.client metadata while agent-visible context still lacks a stable client-type field.

## Solution Assessment

No for the suggested ad hoc header or config switch. The maintainable path is to reuse existing Gateway connect metadata through a reviewed allowlist and a stable agent-facing exposure contract.
