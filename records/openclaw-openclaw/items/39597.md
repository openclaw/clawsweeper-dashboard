## Summary

Keep open: current main records client identity during Gateway WebSocket connect, but the chat send and prompt metadata path does not expose a stable agent-visible client type for WebChat turns.

## What This Changes

Request to expose enough WebChat client metadata for agents to distinguish native Mac app, browser, and mobile web turns.

## Best Possible Solution

Add a small documented client-kind signal derived from existing Gateway connect.client metadata, and expose only stable sanitized fields to agent context unless maintainers explicitly approve more detail.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main can distinguish clients at the Gateway handshake, but that distinction is not exposed as stable agent runtime or inbound metadata.

## Solution Assessment

No, the suggested custom header is not the best first path because the Gateway already has typed connect.client metadata. The narrower maintainable path is to reuse that contract and decide a minimal prompt-facing client taxonomy.
