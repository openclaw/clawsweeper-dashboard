## Summary

Keep open. Current main and the latest release have connection-level Gateway client identity, but WebChat `chat.send` and agent-facing inbound metadata still lack a settled, trusted client-type field; this remains a product/security-sensitive feature request rather than implemented behavior.

## What This Changes

The issue requests agent/runtime context that distinguishes WebChat client types such as native macOS app, desktop browser, and mobile web.

## Best Possible Solution

Define one additive gateway-owned client-surface metadata contract with explicit trust semantics, then propagate it through WebChat/native send paths into agent metadata with docs and focused tests.

## Reproduction Assessment

Yes, as a source-level capability check: current main has connection-level client identity metadata, but `chat.send` and inbound agent metadata do not carry a trusted client-type field. This is a missing feature rather than a broken existing behavior reproduction.

## Solution Assessment

Unclear. A custom header or config flag is plausible, but maintainers should first decide the trust model and whether this belongs as connection-attested metadata, per-message metadata, or diagnostics-only context.
