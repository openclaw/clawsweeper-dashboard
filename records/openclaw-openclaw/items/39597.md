## Summary

Keep open. Current main has Gateway handshake metadata that can identify client classes, but WebChat agent context still has no settled, trusted client-type field; this is a product/security-sensitive feature request rather than a strict bug or an already-implemented item.

## What This Changes

Expose WebChat client type, such as native Mac app versus browser or mobile web, to agent runtime context.

## Best Possible Solution

Define one additive gateway-owned client-surface metadata contract that distinguishes install/client class from human identity, includes explicit trust semantics, and updates protocol schema, WebChat/native/browser send paths, inbound prompt metadata, docs, and focused tests.

## Reproduction Assessment

Yes, as a source-level capability check: current main carries client identity in the Gateway handshake but does not expose a trusted client-type field in chat.send, MsgContext, or inbound prompt metadata. This is a missing capability rather than a broken existing behavior repro.

## Solution Assessment

Unclear. A custom header or config flag is plausible, but maintainers should first decide the trust model and whether this belongs as connection-attested metadata, per-message metadata, or display-only diagnostics.
