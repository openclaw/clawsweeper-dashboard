## Summary

Keep open. Current main already distinguishes browser/native clients at the Gateway handshake and uses that metadata for internal logging/presence, but WebChat agent-visible context still has no reviewed client-type field; the remaining work needs product and security agreement on the public metadata contract.

## What This Changes

Expose WebChat client type, such as Mac app versus browser or mobile web, to agents instead of only surfacing `channel=webchat`.

## Best Possible Solution

Define a stable allowlisted Gateway client metadata contract sourced from `connect.client`, document it as client-declared unless server-owned, expose only approved low-entropy fields through MsgContext or inbound metadata, and test browser plus native WebChat behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that Gateway clients already send structured connect metadata while agent-visible WebChat context still lacks a stable client-type field.

## Solution Assessment

No for the suggested ad hoc header or config switch. The maintainable path is to reuse existing Gateway connect metadata through a reviewed allowlist and stable agent-facing exposure contract.
