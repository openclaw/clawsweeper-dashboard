## Summary

Keep open. Current main already has structured gateway client IDs and Mac/browser clients send different IDs, but agent-visible trusted inbound metadata still only exposes channel/provider/surface; a stable client-type contract remains product/API work.

## What This Changes

Request to expose WebChat client type, such as native Mac app versus browser or mobile web, in agent/runtime context.

## Best Possible Solution

Define a small gateway-client metadata contract from connect.client, expose only approved stable fields to agent context, and document/test Mac and browser WebChat behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection shows Mac and browser clients identify themselves during connect, but current agent metadata lacks a dedicated client-type field.

## Solution Assessment

No, the suggested custom header/config flag is not the best starting point. The WebSocket connect protocol already carries client.id/mode/platform, so the maintainable path is deciding which of those fields should be exposed through MsgContext/inbound metadata.
