## Summary

Keep open. Current main has gateway handshake metadata that can distinguish product/client classes, but WebChat agent context still lacks a settled, trusted client-type field; this remains a product/security-sensitive feature request rather than an implemented capability.

## What This Changes

The issue requests agent-visible WebChat client-type context so native Mac app, browser, and mobile web clients can be distinguished.

## Best Possible Solution

Define one additive gateway-owned client-surface metadata contract with explicit trust semantics, then propagate it through browser/native send paths into agent metadata with docs and focused tests.

## Reproduction Assessment

Yes, as a source-level capability check: current main has connection-level GatewayClientInfo, but WebChat chat.send and inbound agent metadata do not carry a client-type field. This is a missing feature rather than a broken existing behavior reproduction.

## Solution Assessment

Unclear. A custom header or config flag is plausible, but maintainers should first decide the trust model and whether this belongs as connection-attested metadata, per-message metadata, or diagnostics-only context.
