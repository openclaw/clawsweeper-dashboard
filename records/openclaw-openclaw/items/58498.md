## Summary

Keep open: current main still has a source-visible split between the auth profile shown in status and the generic provider-usage auth resolver, and the only open linked fix candidate is an unmerged conflicting fork PR.

## What This Changes

The issue asks for `/status` and session status usage/quota to use the same active OAuth profile shown for the current session.

## Best Possible Solution

Land or replace the linked candidate with a narrow fix that threads the session-selected OAuth profile into provider usage auth resolution, validates provider compatibility, and preserves provider-scoped fallback for stale or mismatched profile ids.

## Reproduction Assessment

Yes. Source inspection shows the status auth label can follow `sessionEntry.authProfileOverride`, while the generic provider usage auth resolver still orders profiles at provider scope without a preferred session profile.

## Solution Assessment

No for current main. The maintainable fix is to carry the session-selected profile into provider usage auth resolution and accept it only when it belongs to the requested provider, instead of relying on provider-level order alone.
