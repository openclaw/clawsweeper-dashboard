## Summary

Keep open: current main still has the reported mismatch path because the status label uses the session auth override while provider usage resolution still chooses credentials only by provider-level order. The linked #59208 PR is an open implementation candidate with closing syntax, so this issue should remain open until that PR, or an equivalent replacement, lands.

## What This Changes

The issue requests that status/session usage quota lines use the same active `openai-codex` OAuth profile shown in the status card.

## Best Possible Solution

Thread the session-selected OAuth profile into provider usage resolution so the status/session usage line and auth label resolve from the same validated profile, while preserving provider-scoped fallback when the preferred profile is absent, stale, or belongs to another provider.

## Reproduction Assessment

Yes: the source path gives a high-confidence reproduction with two `openai-codex` OAuth profiles and a session `authProfileOverride`. Current main passes that override to auth-label rendering but not to provider usage auth resolution.

## Solution Assessment

Yes: a narrow preferred-profile parameter through `buildStatusText` → `loadProviderUsageSummary` → `resolveProviderAuths`/`resolveOAuthToken` is the maintainable fix, with a mismatched-provider fallback guard. The open PR direction matches that, but the issue should close only after a reviewed merge or equivalent landed fix.
