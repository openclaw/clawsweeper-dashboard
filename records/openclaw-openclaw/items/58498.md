## Summary

## What This Changes

The issue asks for the status usage/quota line to use the same active OpenAI/Codex OAuth profile shown for the current session.

## Best Possible Solution

Close this issue as fixed in v2026.6.6 and ask for a fresh report only if the mismatch reproduces on current releases.

Yes by source and history: the pre-fix path could show `sessionEntry.authProfileOverride` in status while usage auth fell back to provider-level selection; current main now has direct regression coverage for the selected-profile and stale-profile cases.

## Solution Assessment

Yes: the implemented fix is in the right owner boundary, validating the selected OpenAI/Codex profile in status and passing it through the Codex provider hook to the app-server rate-limit read without adding new config.
