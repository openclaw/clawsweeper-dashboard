Keep this open because current main still has the selected-profile/usage-auth split, and the issue is already paired with the open closing PR at https://github.com/openclaw/openclaw/pull/59208.

## What This Changes

The report asks status/session usage to resolve quota windows from the OAuth profile active in the session instead of a different profile under the same provider.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/59208 or an equivalent patch that carries a validated session-selected OAuth profile into provider usage auth resolution while preserving provider-order fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists with two openai-codex OAuth profiles: the status label path uses sessionEntry.authProfileOverride, while usage auth still resolves by provider order only.

## Solution Assessment

Yes. Threading a validated preferred profile through loadProviderUsageSummary and resolveProviderAuths is the narrow maintainable fix because it aligns usage credentials with the already-rendered auth label without changing normal fallback behavior.
