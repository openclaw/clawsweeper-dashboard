## Summary

Keep open: current main still has a source-visible split where the status auth label can use the session auth override but normal provider usage auth resolves by provider-level profile order. A linked open fork PR claims this fix, but it is conflicting and unmerged, so the issue remains necessary.

## What This Changes

The issue reports that status output can display one active OAuth profile while the usage/quota line appears to be resolved from another credential.

## Best Possible Solution

Land or replace the linked fix candidate with a narrow change that passes and validates the active session OAuth profile into provider usage auth resolution while preserving provider-scoped fallback for stale or cross-provider profile ids.

## Reproduction Assessment

Yes, source-reproducible: current main still resolves normal provider usage OAuth from provider-level profile order, while the status auth label can come from sessionEntry.authProfileOverride.

## Solution Assessment

No for current main: the narrow maintainable solution is to thread the session-selected profile into usage auth resolution and validate it before falling back to the existing provider order.
