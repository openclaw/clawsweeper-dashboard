## Summary

Keep open: this PR targets a valid Feishu message-loss bug that is still present on current main and v2026.6.8, but the patch still loses mention-forward metadata on independently sent blocks and only supplies mocked unit-test output, so it needs a small code/test repair plus real Feishu proof before merge.

## What This Changes

The PR changes the Feishu reply dispatcher and tests so `blockStreaming: true` can send non-card block payloads as independent Feishu messages and suppress a matching duplicate final reply.

## Best Possible Solution

Merge a narrow Feishu dispatcher repair after preserving first-block mention metadata and adding redacted live Feishu proof that independent blocks arrive in order without a duplicate final reply.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 return from Feishu block delivery when `useStreamingCard` is false, so `streaming: false` plus `blockStreaming: true` loses block text. I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

No, not yet. The dispatcher is the right layer for the narrow repair, but the PR should preserve mention-forward metadata on the first independent block and prove the real Feishu delivery path; the broader segment-mode PR is not a safer replacement because it adds config surface and is conflicting.
