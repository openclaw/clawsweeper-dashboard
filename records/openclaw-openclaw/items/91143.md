## Summary

Keep open: the PR is a useful shared-helper fix with real behavior proof, but the patched regex still classifies ssh-named non-tunnel commands containing `:18789` as SSH tunnels, so it does not fully satisfy the stated invariant.

## What This Changes

The PR changes `classifyPortListener` to return `unknown` for ssh-named listeners that do not match the port-aware tunnel regex and adds focused port-format regression tests.

## Best Possible Solution

Keep the shared helper fix, but tighten tunnel detection so only `-L` or `-R` forwarding specs for the queried local port return `ssh`, with a regression for an ssh-named non-tunnel command containing `127.0.0.1:18789`.

## Reproduction Assessment

Yes. Current main clearly returns `ssh` for any normalized command containing `ssh`, and the PR body provides terminal output from the exported diagnostic path showing the before/after behavior for `sshd`.

## Solution Assessment

No, not yet. The fix is in the right shared helper, but the final regex must require `-L` or `-R` context before accepting a colon-prefixed queried port as an SSH tunnel.
