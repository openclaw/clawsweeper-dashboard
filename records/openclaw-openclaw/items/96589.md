## Summary

Keep open; current main and the latest release still drop the defaults-only cron failure-alert state on SQLite reload, and no linked PR currently owns the fix.

## What This Changes

The issue asks for per-job cron failure alerts enabled with default settings to survive SQLite/gateway reloads instead of becoming disabled.

## Best Possible Solution

Teach `failureAlertFromRow` to preserve `failure_alert_disabled === 0` as `{}` and add regression coverage for disabled, populated, and defaults-only alert configs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: create or patch a job with `failureAlert: {}`, persist it through the SQLite row codec, reload it, and observe that `failureAlertFromRow` returns `undefined`; I did not run a live gateway restart in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow fix belongs in the SQLite failure-alert codec, preserving the existing `false` and populated-config semantics without adding config surface or runtime fallback logic.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/96589. On current main, `openclaw cron edit <job> --failure-alert` can store `failureAlert: {}` as `failure_alert_disabled = 0` with all other failure-alert columns null, but `failureAlertFromRow` returns `undefined` after SQLite reload, disabling alerts when no global failure-alert default is enabled. Keep the repair scoped to the cron failure-alert SQLite codec and focused tests: treat `failure_alert_disabled === 0` as the explicit enabled-with-defaults sentinel returning `{}`; preserve `false` -> disabled and populated configs; do not add config flags, broad runtime fallback readers, or CHANGELOG.md edits. Add regression coverage in `src/cron/store.test.ts` for `false`, populated, and `{}` round trips, and include resolver proof that `{}` stays alert-active with no global config.
