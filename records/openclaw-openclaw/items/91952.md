## Summary

Keep this PR open for repair: the SQLite migration resolves the earlier sidecar concern and the live proof is strong, but both persisted quarantine paths can report failures after they are no longer active. The protected maintainer label also leaves the permanent operator-visible status scope for explicit maintainer approval.

## What This Changes

Adds compact plugin health to `/status` and detailed `/status plugins` reporting for plugin load errors, diagnostics, channel setup failures, context-engine quarantines, compatibility notices, and runtime tool-schema quarantines.

## Best Possible Solution

Retain the shared SQLite design, but make both persisted signals represent current health: reconcile owner/tool records after successful schema validation, identify context-engine processes by PID plus start time or another lifecycle-safe identity, add recovery tests, document `/status plugins`, and obtain explicit approval for the permanent default status line.

## Reproduction Assessment

Yes for both patch defects from source: fail and then successfully validate the same loaded tool to retain the TTL-backed record, or leave a context-engine record whose PID is later recycled to keep it falsely active. The PR's live proof does not exercise either recovery case.

## Solution Assessment

No. SQLite is the correct ownership boundary, but a current-health surface needs success-path reconciliation and lifecycle-safe process identity rather than TTL-only recovery and PID-only liveness.
