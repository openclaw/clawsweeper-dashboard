## Summary

Keep open. Current main still routes Gemini CLI OAuth through the bundled Google plugin's `google-gemini-cli` CLI backend, OAuth setup still selects that runtime, and Gemini turns still spawn the external `gemini` process. I found no current-main source, test, docs, changelog, or related-item evidence that the reported fixed per-turn startup/auth latency has been fixed.

## What This Changes

The report asks OpenClaw to fix a regression where `google-gemini-cli` OAuth turns on macOS became consistently slow after upgrading to OpenClaw 2026.3.23-2, apparently from fixed per-turn auth or credential retrieval latency.

## Best Possible Solution

Keep the issue open for maintainer triage on an affected macOS OAuth setup. The best end state is measured standalone `gemini --prompt` latency versus OpenClaw wrapper timing, followed by either a Gemini CLI Keychain doctor/docs remediation, clearer slow-start diagnostics, or a plugin-owned runtime mitigation that avoids repeated external credential initialization only if the Gemini CLI contract supports it.

## Reproduction Assessment

Unclear. The report provides a concrete macOS OAuth reproduction path and direct `gemini` comparison logs, but high-confidence reproduction still requires live Gemini CLI OAuth credentials and Keychain state on an affected macOS machine; repository inspection can only verify that the path remains present.

## Solution Assessment

Unclear. Current main is not the best solution because it still runs the external Gemini CLI child process per turn; the maintainable fix depends on measuring whether the delay is external CLI/Keychain behavior, OpenClaw wrapper overhead, or a supported opportunity for a plugin-owned execution bridge.
