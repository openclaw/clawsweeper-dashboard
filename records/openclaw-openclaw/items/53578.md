## Summary

Current main now stages the selected google-gemini-cli OAuth profile into a profile-scoped Gemini CLI home, writes `oauth_creds.json`, forces file storage, and clears inherited keychain/auth env, which addresses the repeated keychain/FileKeychain credential path described in the report. The proof commit is on current main only and is not included in the latest release tag.

## What This Changes

The issue reports a regression where google-gemini-cli OAuth turns became consistently slow after upgrading, likely due to fixed per-turn credential or keychain retrieval latency.

## Best Possible Solution

Keep the current main credential-staging bridge and let the next release carry it to users; users on releases through `v2026.6.8` need a newer main build or later release for this fix.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live current-main failure; the report gives concrete macOS keychain/FileKeychain diagnostics, and source inspection shows current main now bypasses that repeated keychain path by staging Gemini CLI file credentials.

## Solution Assessment

Is this the best way to solve the issue? Yes: the current implementation keeps the fix inside the Google plugin’s CLI prepare-execution boundary, avoids a new public SDK/config surface, and uses Gemini CLI’s own documented env/file contracts.
