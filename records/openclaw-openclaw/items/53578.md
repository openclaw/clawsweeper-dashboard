## Summary

Current main now routes google-gemini-cli OAuth execution through a provider-owned auth bridge that stages the selected OpenClaw OAuth profile into an isolated Gemini CLI home and forces file-backed credential storage, which addresses the reported repeated Keychain/FileKeychain credential path. The fix is not in the latest v2026.6.8 release tag, so this should close as current-main-only implemented.

## What This Changes

The issue reports a per-turn latency regression on the google-gemini-cli OAuth provider after upgrading to OpenClaw 2026.3.23-2.

## Best Possible Solution

Keep the current provider-owned Gemini CLI auth bridge and wait for the next release that contains it; no separate core change is needed for this report.

## Reproduction Assessment

Yes, by source and dependency-contract inspection rather than a fresh live run: the report’s Keychain/FileKeychain logs match the Gemini CLI credential-storage path, and current main now forces file storage in an isolated Gemini CLI home. I did not rerun the reporter’s macOS timing test in this read-only review.

## Solution Assessment

Yes. A Google-provider-owned `prepareExecution` bridge is the narrowest maintainable fix because it keeps credential staging local to the bundled google-gemini-cli runtime and avoids a new public Plugin SDK credential-forwarding contract.
