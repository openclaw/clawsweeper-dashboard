## Summary

Close as implemented on main: current main now stages selected google-gemini-cli OAuth credentials into an isolated Gemini CLI home and bypasses the repeated native keychain/FileKeychain lookup path described in the report; this is not yet in the latest release.

## What This Changes

The issue reports a fixed per-turn google-gemini-cli OAuth delay after upgrading to OpenClaw 2026.3.23-2, likely caused by repeated macOS keychain/FileKeychain credential lookup.

## Best Possible Solution

Keep the current Google-plugin credential-staging bridge and let the next release carry it to users; users on `v2026.6.8` need a current-main build or later release for this fix.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main failure was reproduced; the report gives concrete macOS keychain/FileKeychain diagnostics, and source plus release comparison shows the released path lacked the isolated Gemini CLI auth bridge while current main has it.

## Solution Assessment

Is this the best way to solve the issue? Yes: the current implementation stays inside the Google plugin’s CLI `prepareExecution` boundary, avoids a new public SDK/config surface, uses Gemini CLI’s own env/file contracts, and clears conflicting ambient auth state.
