## Summary

Keep open. Current main has meaningful timeout mitigations for Active Memory, but the direct-chat path still runs recall in the foreground prompt-build hook and can still spend the configured timeout budget before the main reply is built. The recent discussion and open related work point to remaining latency/reliability design work, not an already-complete fix.

## What This Changes

Active Memory should complete quickly or fail open without delaying direct-chat turns for roughly the configured 20-30 second timeout across model configurations.

## Best Possible Solution

Keep this as the broad Active Memory direct-chat latency/reliability tracker until maintainers choose and land the durable product direction: background/non-blocking recall, an opt-in single-shot low-latency mode, stricter foreground fail-open budgeting, or an explicit prewarm/readiness gate for cold memory work.

## Reproduction Assessment

Yes for source-level reproduction: an eligible direct-chat turn reaches `before_prompt_build`, the prompt helper awaits `runBeforePromptBuild`, and Active Memory awaits recall until success, timeout, or circuit-breaker skip. Live-provider reproduction was not attempted in this read-only review.

## Solution Assessment

No, current main is not the complete solution. The hard deadline, partial-result recovery, and circuit breaker are useful mitigations, but the requested behavior needs Active Memory to stop putting normal direct-chat replies behind the blocking recall path or to provide a clearly documented low-latency mode.
