## Summary

Keep open. Current main has useful mitigations for Active Memory timeouts, including a hard wait boundary and a consecutive-timeout circuit breaker, but the direct-chat recall still runs in the foreground prompt-build hook and can still spend the configured timeout budget before the main reply is built. The fresh issue discussion and related open reports point to remaining latency/reliability work rather than an already-shipped complete fix.

## What This Changes

Active Memory should stop delaying direct-chat turns for the full recall timeout and should complete quickly or fail open without blocking the user-visible reply path.

## Best Possible Solution

Keep this issue open as the broad Active Memory direct-chat latency/reliability tracker until maintainers choose and land the durable product direction: non-blocking/background recall, an opt-in single-shot low-latency mode, a stricter foreground fail-open budget, or an explicit prewarm/readiness gate that prevents cold memory work from being paid inside the first user turn.

## Reproduction Assessment

Yes for source-level reproduction: an eligible direct-chat turn reaches before_prompt_build, the prompt helper awaits runBeforePromptBuild, and the Active Memory hook awaits recall until success, timeout, or circuit-breaker skip. No for live-provider reproduction in this review because the checkout was inspected read-only without running a gateway.

## Solution Assessment

No, current main is not the complete solution. The hard deadline and circuit breaker are good mitigations, but the requested behavior needs Active Memory to stop putting normal direct-chat replies behind a blocking recall path or to provide a clearly documented low-latency mode.
