## Summary

Keep open. Current main still validates every stored CLI session binding against auth, system-prompt, and MCP hashes, and there is no explicit force-reuse path for a manually attached binding. The source, caller, and tests all support the reported behavior rather than showing it already fixed.

## What This Changes

The issue asks OpenClaw to preserve explicitly attached CLI session bindings on the first turn instead of invalidating them on hash drift.

## Best Possible Solution

Add a narrow explicit-reuse contract for manually attached CLI session bindings, keep normal automatic binding invalidation unchanged, and cover both the resolver and the CLI-runner path with focused tests.

## Reproduction Assessment

Yes for the current resolver path: a binding with a session id and mismatched `extraSystemPromptHash` or MCP/auth metadata deterministically returns an invalidation reason, and the CLI runner calls that resolver for stored bindings on each run. The live manual attach tool was not exercised in this read-only review.

## Solution Assessment

The suggested force-reuse short-circuit is directionally right only if it is scoped to a trusted explicit attach marker or caller. Applying it to all stored bindings would weaken the existing auth/prompt/MCP safety invalidation.
