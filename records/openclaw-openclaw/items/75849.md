## Summary

Keep open. Current main still treats stored CLI session binding hash drift as an invalidation before returning a reusable session id, and there is no explicit manual-attach or force-reuse contract in the binding type, read/write helpers, runner call site, tests, or source/docs search. The adjacent open auth-epoch/auth-profile PRs address credential-related resets, but they do not cover the manual attach case over system-prompt or MCP hash drift.

## What This Changes

Manual attach to an existing CLI session should keep using the bound CLI session even when stored binding hashes drift on the first turn.

## Best Possible Solution

Add an explicit manual-attach reuse contract to CLI session bindings, persist and read that flag through the session binding helpers, and short-circuit `resolveCliSessionReuse` only for that trusted flag while keeping normal auth, prompt, and MCP drift checks intact.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call `resolveCliSessionReuse` with a binding containing a `sessionId` plus stored prompt or MCP hash values that differ from the current values; current main returns only an `invalidatedReason`, so the runner proceeds without the reusable session id. I did not run tests because this review was required to be read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is an explicit manual-attach `forceReuse`/equivalent flag that is only set by trusted attach flows, not removing hash validation globally or broadening the adjacent auth-epoch fixes.
