## Summary

Keep open. Current main still validates stored CLI session bindings against auth, prompt, and MCP hashes before returning a reusable session id, and there is no `forceReuse` or manual-attach override. The report is narrow, source-backed, and not fully covered by the adjacent open auth-epoch PRs.

## What This Changes

Manual attach to an existing CLI session should keep using the explicitly attached binding on the first turn instead of treating hash drift as a fresh-session reset.

## Best Possible Solution

Add an explicit manual-attach reuse contract to CLI session bindings, preserve that flag through binding read/write, and short-circuit `resolveCliSessionReuse` only when the flag is explicitly set so normal safety checks remain intact.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call `resolveCliSessionReuse` with a binding containing a `sessionId` plus stored prompt or MCP hash values that differ from current values; current main returns only `invalidatedReason`, so the runner starts without the reusable session. I did not run tests because this review was required to be read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is the requested explicit `forceReuse` path for manual attach, not removing hash validation globally; it should be limited to trusted/manual binding creation and covered by helper tests.
