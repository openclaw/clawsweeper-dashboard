## Summary

Keep open. Current main still drops captured termination metadata at the foreground exec and process.poll result boundary, so the diagnostics gap remains source-reproducible even though the intermittent macOS SIGKILL itself is not deterministic.

## What This Changes

The issue asks OpenClaw to surface structured termination diagnostics for intermittent macOS exec SIGKILLs and consider safer handling for broad scans over large OpenClaw state/log directories.

## Best Possible Solution

Add backward-compatible termination diagnostic fields that propagate existing supervisor/runtime metadata to foreground exec and process.poll results, with signal policy and scan guardrails handled in separate focused follow-ups.

## Reproduction Assessment

No for the intermittent macOS SIGKILL itself: the report still lacks a deterministic current-main reproduction. Yes source-level for the diagnostics gap: current code captures exitSignal, failureKind, timedOut, noOutputTimedOut, and termination reason internally, then omits key fields from user-facing structured results.

## Solution Assessment

Yes: additive metadata propagation is the narrowest maintainable first fix because current main already has most requested termination context. Changing SIGTERM/SIGKILL behavior or adding broad-scan guardrails should stay separate from this repair.
