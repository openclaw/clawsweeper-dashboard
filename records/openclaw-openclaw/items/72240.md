## Summary

Keep open. Current main has richer signal, timeout, and failure-kind data internally, but the foreground exec result and process polling details still drop enough of that metadata that the reported macOS SIGKILL diagnostic gap remains actionable.

## What This Changes

The issue asks OpenClaw to expose structured cause metadata and bounded output/log context when exec/process commands terminate with SIGKILL, especially in macOS sessions with large local state or log scans.

## Best Possible Solution

Add an additive diagnostics contract that carries existing supervisor/runtime metadata through foreground exec and background process results: exitSignal, failureKind, exitReason or terminationReason, timedOut, noOutputTimedOut, relevant timeout limits, and bounded output tail/log/run/session correlation where already available. Keep signal-escalation policy changes and broad recursive-scan guardrails as separate follow-ups unless maintainers expand scope.

## Reproduction Assessment

Unclear for the exact macOS incident: the report describes a recurring but intermittent SIGKILL rather than a deterministic end-to-end repro. The user-facing diagnostics gap is source-reproducible on current main by tracing simulated signal, timeout, cancellation, foreground-result, and process.poll paths.

## Solution Assessment

Yes for the first repair step: enriching exec/process result metadata is the narrowest maintainable fix for the observable diagnostic gap. Broader supervisor signal policy, OS memory diagnosis, and recursive-scan guardrails should remain separate unless maintainers deliberately combine them.
