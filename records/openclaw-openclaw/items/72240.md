## Summary

Keep open. Current main still records termination cause internally but does not consistently expose signal/failure metadata through foreground exec, process.poll structured results, or the node-host exit result path, so the SIGKILL diagnostics gap remains source-reproducible.

## What This Changes

The issue asks OpenClaw to expose actionable diagnostics when exec-managed commands are killed with SIGKILL, especially on macOS under large state/log scans.

## Best Possible Solution

Propagate existing supervisor/runtime termination metadata into backward-compatible exec and process result details, then cover signal, timeout, and no-output-timeout paths with focused regression tests while leaving signal escalation and broad-scan guardrails to separate work.

## Reproduction Assessment

No for the intermittent macOS SIGKILL trigger itself; the issue does not provide a deterministic current-main reproduction. Yes at source level for the diagnostics gap: current main carries exitSignal, failureKind, timeout, and supervisor reason internally, then drops key fields from user-facing structured details.

## Solution Assessment

Yes for a narrow first repair. Additive metadata propagation is the maintainable fix for this issue; SIGTERM escalation, OOM/RSS attribution, and broad recursive-scan guardrails should remain separate focused work.
