## Summary

Keep open. Current main stores useful signal, timeout, and failure-kind metadata internally, and some human-readable paths mention signal exits, but foreground exec results and process polling still do not expose enough structured cause metadata to resolve the reported macOS SIGKILL diagnostic gap.

## What This Changes

The issue asks OpenClaw to expose structured cause metadata and bounded output/log context when exec/process commands terminate with SIGKILL, especially in macOS sessions with large local state or log scans.

## Best Possible Solution

Add an additive diagnostics contract that carries existing supervisor/runtime metadata through foreground exec and background process results: exitSignal, failureKind, exitReason or terminationReason, timedOut, noOutputTimedOut, relevant timeout limits, run/session/log correlation, and bounded output tail context. Keep signal-escalation policy changes and broad-scan guardrails separate unless maintainers deliberately expand scope.

## Reproduction Assessment

Unclear: the reported macOS SIGKILL incident is recurring but not deterministic from the issue body. The user-facing diagnostics gap is source-reproducible on current main by tracing simulated signal, timeout, cancellation, foreground-result, and process.poll paths.

## Solution Assessment

Yes for the first repair step: enriching exec/process result metadata is the narrowest maintainable fix for the observable diagnostic gap. Broader supervisor signal policy, OS memory diagnosis, and recursive-scan guardrails should remain separate follow-ups unless maintainers choose to combine them.
