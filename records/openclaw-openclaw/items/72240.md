## Summary

Keep open. Current main still has a user-facing diagnostics gap: exec/supervisor internals track signal and timeout metadata, but foreground exec results and process polling details do not expose enough structured termination cause data to diagnose the reported macOS SIGKILL failures.

## What This Changes

The issue requests clearer diagnostics for intermittent macOS exec SIGKILLs, including termination cause metadata, timeout/cancellation distinction, output tail context, and run/log correlation.

## Best Possible Solution

Add an additive exec/process diagnostics contract that carries existing supervisor/runtime metadata through foreground exec and process polling results: exitSignal, failureKind, exitReason or terminationReason, timedOut, noOutputTimedOut, timeout limits, duration, bounded output tail context, and useful run/session/log correlation. Keep signal-escalation policy and recursive-scan guardrails separate unless maintainers deliberately expand scope.

## Reproduction Assessment

Unclear: there is no deterministic macOS reproduction for the intermittent SIGKILL itself. The diagnostics gap is source-reproducible on current main by tracing simulated SIGKILL, timeout, and cancellation metadata through buildExecForegroundResult and process.poll details.

## Solution Assessment

Yes for the first repair step: additive metadata propagation is the narrowest maintainable fix for the observable diagnostic gap. Supervisor signal policy, memory tracking, and broad-scan guardrails are broader follow-ups.
