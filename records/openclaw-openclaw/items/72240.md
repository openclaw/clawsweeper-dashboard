## Summary

Keep open. Current main has richer exec/supervisor metadata internally, and text paths now mention timeout or signal exits in some cases, but the foreground exec result and process polling details still do not carry enough structured termination metadata to resolve the reported macOS SIGKILL diagnostic gap.

## What This Changes

The issue requests clearer user-facing diagnostics for intermittent macOS exec SIGKILLs, including termination cause metadata, output tail context, timeout/cancellation distinction, and log/run correlation.

## Best Possible Solution

Add an additive exec/process diagnostics contract that propagates existing supervisor/runtime metadata through foreground exec and process polling results: exitSignal, failureKind, exitReason or terminationReason, timedOut, noOutputTimedOut, timeout limits, duration, bounded output tail context, and run/session/log correlation. Keep signal-escalation policy and broad recursive-scan guardrails as separate follow-ups unless maintainers deliberately expand scope.

## Reproduction Assessment

Unclear: there is no deterministic macOS reproduction for the intermittent SIGKILL itself. The user-facing diagnostics gap is source-reproducible on current main by tracing a simulated RunExit with SIGKILL, timeout, or cancellation metadata through buildExecForegroundResult and process.poll details.

## Solution Assessment

Yes: additive metadata propagation is the narrowest maintainable first fix for the reported diagnostic gap. Changing supervisor kill policy, memory tracking, or filesystem-scan guardrails would be broader follow-up work and should not be required for the first repair.
