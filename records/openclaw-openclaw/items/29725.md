## Summary

Keep open: current main and the latest release still expose Discord presence only through static config, auto-presence, and the gated `setPresence` message action; they do not provide the requested plugin-runtime `api.runtime.channel.discord.updatePresence` API. The related self-profile issue overlaps on status/activity, but it is broader and does not clearly own this plugin SDK contract.

## What This Changes

The issue requests a plugin-runtime Discord API for dynamic bot presence updates without restarting OpenClaw.

## Best Possible Solution

Keep this open until maintainers choose a secure Discord-owned plugin/runtime API or action contract for dynamic presence updates, reusing the existing default-off presence gate and gateway registry where appropriate.

## Reproduction Assessment

Not applicable as a bug repro: this is a feature/API request, not broken existing behavior. Source and release inspection give a high-confidence current gap for the requested plugin-runtime Discord presence API.

## Solution Assessment

Unclear as proposed: dynamic presence control is plausible, but a direct `api.runtime.channel.discord` slot does not match the current generic runtime boundary. Maintainers should choose whether this belongs in a Discord plugin facade/action, a generic SDK seam, or the broader self-profile design.
