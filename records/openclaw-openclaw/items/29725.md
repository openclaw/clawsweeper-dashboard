## Summary

Keep open: current main has static Discord presence config, auto-presence, and a gated Discord `set-presence` message action, but it still does not expose a supported plugin-runtime API such as `api.runtime.channel.discord.updatePresence`.

## What This Changes

Request to add a plugin-facing Discord runtime API for dynamically updating the bot's activity and status without exposing the raw Discord gateway/client.

## Best Possible Solution

Add a narrow Discord-owned presence controller with stable product-level types, explicit account selection, operator authorization, documented SDK entrypoint, and precedence rules for static, automatic, and plugin-requested presence.

## Reproduction Assessment

Not applicable: this is a missing capability request, not a broken existing behavior. Source inspection confirms current main has internal/message-action presence paths but no plugin-runtime presence API.

## Solution Assessment

No, not exactly as proposed: a raw `gateway.updatePresence` wrapper would expose too much transport detail and leave authorization, account targeting, and auto-presence precedence undefined. The safer path is a Discord-owned controller exposed through a documented SDK contract.

## Risks / Open Questions

- A plugin-callable presence API would let third-party code mutate the bot's externally visible Discord identity state, so authorization and operator gating need an explicit decision.
- Multi-account Discord configs need clear account targeting; an implicit default account could update the wrong bot presence.
- Manual/plugin presence writes can conflict with static startup presence and the existing auto-presence controller unless precedence and restoration rules are defined.
