## Summary

This remains a valid missing Plugin SDK capability: current main supports static presence and internal runtime updates, but no supported plugin-facing controller. Keep it open for maintainer product/security design because a raw gateway wrapper would leave authorization, multi-account targeting, lifecycle, and precedence with auto-presence undefined.

## What This Changes

The issue requests a supported plugin runtime API for changing Discord bot status and activity without restarting OpenClaw.

## Best Possible Solution

Add a narrow Discord-owned presence controller with product-level types, explicit account selection, an operator gate, and defined precedence and restoration rules for static, automatic, and plugin-requested presence; expose it through the documented Plugin SDK without exposing Carbon or the raw client.

## Reproduction Assessment

Not applicable: this is a clearly evidenced missing capability rather than a failure of documented behavior; source inspection confirms only static configuration and internal dynamic writers exist.

## Solution Assessment

No, not exactly as proposed: directly forwarding `gateway.updatePresence` is too low-level and leaves writer precedence, authorization, lifecycle, and multi-account behavior undefined; a Discord-owned controller is the safer maintainable direction.

- A plugin-callable presence API needs explicit operator authorization and account selection rather than implicit access to whichever gateway is active.

- Publishing this through the Plugin SDK creates a compatibility-sensitive third-party contract, so it should not expose Carbon-specific payload types.
