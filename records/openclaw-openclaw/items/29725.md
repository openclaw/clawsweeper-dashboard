## Summary

Keep open: current main and v2026.6.10 still have static Discord presence, auto-presence, and a gated `setPresence` message action, but not the requested `api.runtime.channel.discord.updatePresence` plugin-runtime API. The request remains a product/security decision because it would expose bot-credential-backed Discord presence mutation to plugins.

## What This Changes

Request to expose a Discord plugin runtime API for dynamic bot presence updates.

## Best Possible Solution

Approve a Discord-owned, default-off, account-scoped plugin action/API contract that reuses the existing gateway presence path without exposing provider internals, then add focused tests and docs for gating and auto-presence precedence.

## Reproduction Assessment

Not applicable: this is a feature/API request, not a reported broken existing behavior. Source inspection gives a high-confidence current-main gap for the requested plugin-runtime Discord presence API.

## Solution Assessment

Unclear as proposed: dynamic presence control is plausible, but a direct `api.runtime.channel.discord` slot conflicts with the current generic runtime boundary unless maintainers intentionally add a Discord-owned seam.
