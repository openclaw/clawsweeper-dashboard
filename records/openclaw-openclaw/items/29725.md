## Summary

Keep this open. Current main has static Discord presence, auto-presence health signaling, and a gated `setPresence` message action, but it still does not expose the requested code-plugin runtime helper under `api.runtime.channel.discord`; this is a Discord/plugin-runtime API decision rather than a cleanup close.

## What This Changes

The issue requests a Discord plugin runtime helper that lets code plugins dynamically update bot presence without restarting OpenClaw.

## Best Possible Solution

Add a Discord-owned, documented runtime helper only after settling the permission gate, account selection, payload shape, and auto-presence precedence; otherwise document the existing gated `setPresence` message action as the supported path for plugins that can use channel actions.

## Reproduction Assessment

Yes. A source-level reproduction is enough: inspect `extensions/discord/src/runtime.ts` and `docs/plugins/sdk-runtime.md` on current main and the requested `api.runtime.channel.discord.updatePresence` helper is not present, while the adjacent `setPresence` action path is present under Discord message actions.

## Solution Assessment

Unclear. The requested API shape is plausible, but the best implementation should reuse the existing gateway registry/action validation and must first answer the gating and auto-presence precedence questions rather than exposing a second uncontrolled writer.
