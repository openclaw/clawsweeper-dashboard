## Summary

## What This Changes

The PR adds a Zalo personal-account outbound text normalizer, wires it into sendMessageZalouser, and adds unit/integration coverage for HR stripping, list blank-line collapse, fenced code preservation, and textStyles safety.

## Best Possible Solution

Keep the Zalo-specific normalizer on the agent markdown delivery path while preserving default/plain literal behavior, and align fence handling with the existing matching-marker parser before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows PR head normalizes omitted textMode sends and toggles fence state on any marker, while current main treats omitted textMode as plain and agent adapters opt into markdown explicitly. The real Zalo screenshots demonstrate the after-fix list rendering, but not the remaining compatibility cases.

## Solution Assessment

No, not as written. The send boundary is a reasonable place for Zalo-specific rendering normalization, but the patch should preserve the plain/default literal path and use matching fence-opener semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve default plain sends unless markdown mode opts in:** `extensions/zalouser/src/send.ts:45`
  - body: `sendMessageZalouser` currently treats omitted `textMode` as plain text, and current tests pin that default as literal. This change still normalizes the default path whenever manual `textStyles` are absent, so the direct tool, pairing replies, or callers sending `---` can have literal content removed before delivery; limit normalization to the explicit markdown/agent-rendering path or get maintainer approval for changing the plain contract.
  - confidence: 0.91
- **[P2] Match fenced-block closers to the opener:** `extensions/zalouser/src/outbound-format.ts:77-82`
  - body: The normalizer flips fence state on any backtick or tilde fence-looking line. A backtick code block containing a `~~~` line exits fence mode early, so a later `---` line in that same code block is treated as prose and stripped even though fenced content should pass through verbatim; track the opener marker and length like the existing markdown parser does.
  - confidence: 0.9
