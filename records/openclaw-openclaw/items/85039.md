## Summary

Keep this PR open: the Zalo rendering goal is useful and the supplied screenshots are valid after-fix proof, but the current head still introduces message-content loss in plain/default sends and fenced code blocks.

## What This Changes

The PR adds `normalizeZalouserOutboundText`, wires it into `sendMessageZalouser`, and adds tests for Zalo HR stripping, list blank-line collapse, fenced code preservation, and plain-mode text style safety.

## Best Possible Solution

Keep the Zalo-specific normalizer near the send boundary, but preserve the existing plain/default literal contract unless maintainers explicitly opt that path in, and reuse the current markdown parser’s matching-fence rules before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows plain/default sends are normalized and fence state toggles on any backtick or tilde fence line. The claimed Zalo visual improvement has after-fix real-client screenshot proof, but no same-conversation before/after pair.

## Solution Assessment

No, not as written: the Zalo send path is a reasonable place for channel rendering normalization, but the implementation should preserve the plain/default literal contract or make the opt-in boundary explicit, and it should match fenced-code closers to the opener.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve plain-mode sends unless normalization is explicit:** `extensions/zalouser/src/send.ts:43-45`
  - body: `sendMessageZalouser` currently treats omitted `textMode` as plain text, and current tests pin that default as literal. This change normalizes that default path unless manual `textStyles` are present, so a caller or the Zalo tool can send `---` and deliver an empty/rewritten message instead of the literal content; restrict normalization to the markdown/agent-rendering path or add an explicit internal opt-in.
  - confidence: 0.9
- **[P2] Match fenced-block closers to the opener:** `extensions/zalouser/src/outbound-format.ts:77-82`
  - body: The new normalizer treats any fence-looking line as a boundary, so a backtick code block that contains a `~~~` line flips back to prose mode. A later `---` line in that same code block is then stripped before sending, even though fenced content is meant to pass through verbatim; track the opener marker and only close on a matching marker/length.
  - confidence: 0.92
