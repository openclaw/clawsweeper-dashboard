## Summary

Keep this PR open: the proof and security posture look good, and the markdown-only send boundary is the right shape, but current head still misses parser-supported CRLF input and `+` unordered lists, so part of the Zalo rendering problem remains unfixed.

## What This Changes

The PR adds a zalouser outbound markdown normalizer, wires it into `sendMessageZalouser` for `textMode: "markdown"`, and adds tests for HR stripping, list spacing, fenced code, and plain-send passthrough.

## Best Possible Solution

Keep the markdown-only normalizer, but make its newline and list-marker grammar match the existing zalouser markdown parser before merge while leaving default/plain sends literal.

## Reproduction Assessment

Yes, at source level: the PR normalizer leaves `\r` and `+` markers outside its matching rules while the existing parser accepts CRLF and `+` unordered lists. I did not run a live before/after Zalo client test in this review.

## Solution Assessment

No, not quite: the markdown-only scope is the maintainable direction, but the helper should match the existing parser's newline and list-marker contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Normalize CRLF before scanning outbound markdown:** `extensions/zalouser/src/outbound-format.ts:69`
  - body: `parseZalouserTextStyles` already normalizes `\r\n?` before parsing, but this new preprocessor splits only on `\n` and the HR/list regexes do not accept the trailing `\r`. A markdown reply such as `hello\r\n---\r\nworld` or a CRLF blank-lined list will bypass the normalizer and still render with the Zalo artifact this PR is meant to remove.
  - confidence: 0.93
- **[P3] Include plus list markers in collapse rules:** `extensions/zalouser/src/outbound-format.ts:142-145`
  - body: The existing markdown parser treats `+ item` and `+ [ ] task` as unordered list syntax, but the new collapse regexes only recognize `-`, `*`, and numbered items. Blank-lined `+` lists will keep the extra Zalo spacing, creating a second list grammar that behaves differently from the parser.
  - confidence: 0.86
