## Summary

Keep this PR open: the real Zalo client proof is strong, but the current head still has a blocking markdown-mode regression because the new prose normalizer can rewrite indented code before the existing parser preserves it as literal content.

## What This Changes

The PR adds a Zalo outbound markdown normalizer in `sendMessageZalouser` plus tests to strip horizontal-rule lines and collapse list blank lines on explicit markdown sends.

## Best Possible Solution

Normalize only Zalo markdown prose while preserving every literal/code form the current parser already preserves, then have maintainers explicitly accept the channel-specific markdown compatibility change.

## Reproduction Assessment

Yes for the patch finding from source inspection: markdown-mode sends call the normalizer before the parser, and the normalizer skips only fenced code while its regexes match indented lines. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not yet: the owner boundary is reasonable, but the normalizer must preserve all code forms already supported by `parseZalouserTextStyles` before it is the best fix. After that, the remaining question is maintainer acceptance of the intentional markdown-mode compatibility change.
