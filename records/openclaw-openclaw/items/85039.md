## Summary

Keep open: the Zalo formatting goal is plausible, but this patch rewrites message text before the existing markdown parser can protect fenced code blocks, can leave supplied plain-mode style ranges misaligned, and still lacks attached real Zalo rendering proof.

## What This Changes

The PR adds a Zalo outbound text normalizer and wires it into sendMessageZalouser with unit coverage for horizontal-rule stripping and list blank-line collapse.

## Best Possible Solution

Make the Zalo normalizer block-aware so it only rewrites prose/list markdown outside fenced or preformatted content, preserve or rebase supplied style ranges, and attach redacted Zalo client proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows a markdown send containing a fenced code block with a `---` line reaches the new normalizer before the parser's fence handling and loses that line. I did not run tests because this review kept the checkout read-only.

## Solution Assessment

No: a Zalo-specific preprocessor may be the right direction, but this implementation needs markdown block awareness and style-range safety before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Skip normalization inside fenced code blocks:** `extensions/zalouser/src/outbound-format.ts:46-77`
  - body: Because `sendMessageZalouser` runs this normalizer before `parseZalouserTextStyles`, these global replacements also apply to fenced code blocks. A markdown reply such as a YAML/code block containing a line with `---` will lose that line, and long blank runs inside code can also be collapsed before the existing parser has a chance to preserve fenced content. Make the normalizer block-aware or apply it only to prose/list blocks.
  - confidence: 0.91
- **[P2] Keep plain textStyles aligned with normalized text:** `extensions/zalouser/src/send.ts:43`
  - body: In non-markdown mode, `options.textStyles` still describes the original `text`, but the new normalizer can remove HR/list/newline characters before those unchanged ranges are sent to Zalo. The transport clamp only truncates ranges; it does not rebase them, so styles after removed text can land on the wrong characters. Skip normalization when caller-supplied styles are present or rebase the styles with the text rewrite.
  - confidence: 0.82
