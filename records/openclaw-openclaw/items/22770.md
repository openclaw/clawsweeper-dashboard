## Summary

Keep open: current main still lacks the requested TUI external-editor compose flow. The central TUI input code has no Ctrl+E or equivalent editor handoff, the only suspend-and-spawn path is local auth, slash commands and public TUI docs do not expose this workflow, and the related vi/vim keybinding issue is adjacent rather than a substitute.

## What This Changes

Add a TUI compose mode that opens the current prompt in `$VISUAL` or `$EDITOR`, then loads the edited text back into the composer without auto-submitting it.

## Best Possible Solution

Implement this as core TUI composer behavior after maintainer signoff: reuse the existing suspend/resume pattern, write the current composer text to a temporary markdown file, launch `$VISUAL` then `$EDITOR`, read the saved content back into the composer without auto-send, surface clear status/errors, cover it with focused TUI tests, and document it in the public TUI guide while keeping #55836 scoped to in-place vi/vim bindings.

## Reproduction Assessment

Not applicable as a feature request; there is no runtime bug to reproduce. Source and docs inspection against current main gives a high-confidence verification path for the missing behavior: no key hook, no editor launch/reload code, no slash-command substitute, and no public TUI docs entry.

## Solution Assessment

Unclear until maintainers choose the user-facing trigger and local-process boundary. The narrow maintainable direction is a TUI-owned compose helper built around the existing suspension seam, not a broad plugin or Gateway change.
