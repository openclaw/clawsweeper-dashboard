## Summary

Keep this open: current main and the latest release still do not implement an external-editor compose flow, while the linked multiline and vi-mode issues are adjacent but not substitutes. The remaining blocker is maintainer approval of the public trigger and the local editor/temp-file security boundary.

## What This Changes

Add a TUI external-editor compose mode that opens the current draft in `$VISUAL` or `$EDITOR`, reloads the edited text into the composer, and does not auto-submit it.

## Best Possible Solution

Implement a dedicated TUI composer action after maintainer signoff, using the existing suspension lifecycle plus a narrowly tested external-editor helper with safe temp-file handling, direct process spawning, clear errors, buffer replacement, docs, and no auto-submit.

## Reproduction Assessment

Not applicable: this is a new TUI capability request, not a broken behavior with an existing contract. Source inspection does confirm the requested compose flow is absent on current main and in `v2026.6.5`.

Yes in broad direction, but not ready for autonomous implementation. The maintainable path is a core TUI helper using the existing suspension seam after maintainers choose the trigger and security contract.
