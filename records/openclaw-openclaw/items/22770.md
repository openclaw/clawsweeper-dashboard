## Summary

Keep this issue open: current main and v2026.6.6 still lack the requested TUI external-editor compose flow, and the related multiline, vi-mode, and expanded-composer issues cover adjacent editing problems rather than this `$VISUAL`/`$EDITOR` handoff. The request fits TUI ergonomics work, but it needs maintainer approval for the public trigger and local process/temp-file security boundary before implementation.

## What This Changes

Add a TUI compose mode that opens the current draft in `$VISUAL` or `$EDITOR`, then loads the edited text back without auto-submitting.

## Best Possible Solution

Add one core TUI handoff after maintainer signoff: reuse the existing suspension lifecycle, parse `$VISUAL`/`$EDITOR` into a direct argv spawn, manage a restrictive Markdown tempfile, load the edited draft back without sending, and document the workflow.

## Reproduction Assessment

Yes for the capability gap: current main and v2026.6.6 have no external-editor trigger, draft tempfile handoff, or `$VISUAL`/`$EDITOR` compose path. This is source-verifiable missing functionality, not a broken-behavior bug reproduction.

## Solution Assessment

Yes in broad shape, but not implementation-ready without maintainer approval. The existing TUI suspension seam plus Codex-style direct spawning and secure tempfile handling looks like the narrowest maintainable approach, while the public trigger and active-state policy still need a decision.
