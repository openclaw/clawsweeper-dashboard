## What This Changes

This PR changes the gateway config reloader to switch chokidar to polling after native watcher recreate retries are exhausted and updates the watcher recovery test for the two-phase lifecycle.

## Best Possible Solution

Merge the narrow gateway polling fallback after maintainer review and normal gates, with no new config knob and with final disabled status retained when polling recovery also fails.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main and v2026.6.6 retry native chokidar watchers outside Vitest and then disable hot-reload after the retry budget; the contributor also supplied production logs and standalone real chokidar output.

## Solution Assessment

Is this the best way to solve the issue? Yes; a bounded switch to chokidar polling after native watcher retry exhaustion is narrower than adding a config knob or requiring sysctl/operator restarts, and it keeps the final failure state if polling also fails.
