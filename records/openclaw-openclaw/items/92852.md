## Summary

Keep open for normal maintainer review: current main and v2026.6.6 still disable config hot-reload after native watcher retry exhaustion, while this clean PR adds a narrow fallback with source, dependency, CI, and after-fix proof support.

## What This Changes

The PR modifies the gateway config reloader to switch chokidar to polling after native watcher recreate retries are exhausted and updates the watcher recovery test for the two-phase failure path.

## Best Possible Solution

Merge the narrow gateway fallback after maintainer review and normal gates, preserving bounded native retries and disabling hot-reload only if polling-mode recovery also fails.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main and v2026.6.6 retry native chokidar watchers with usePolling false outside Vitest, then set hotReloadStatus to disabled after the retry budget; the reporter also supplied production logs, but I did not run a live ENOSPC gateway.

## Solution Assessment

Is this the best way to solve the issue? Yes; a bounded switch to chokidar polling after native watcher retry exhaustion is narrower than adding a config knob and avoids pushing operators to global CHOKIDAR_USEPOLLING or sysctl workarounds.
