## Summary

Keep open: current main and latest release still forward the SSRF guard dispatcher through the Slack media fetch wrapper, matching the reported placeholder-only Slack attachment failure path. The linked fixing PR is open in a fork, so this issue should stay open until maintainers review or replace that fix.

## What This Changes

The issue reports Slack DM file attachments in container sandboxes being reduced to placeholders because the Slack media fetch wrapper double-forwards an Undici dispatcher and fails before materializing media.

## Best Possible Solution

Land or replace the linked Slack media fix after maintainer/security approval that removes the double-dispatcher conflict while preserving Slack URL validation, auth and redirect behavior, bounded downloads, and the intended SSRF enforcement contract.

## Reproduction Assessment

Yes. The issue includes live Slack container logs for `invalid onRequestStart method`, and source inspection shows current main still passes the guard-created dispatcher through `createSlackMediaFetch`; I did not run a live Slack workspace.

## Solution Assessment

No, not as an unreviewed direct dispatcher strip. The proposed strip matches the observed failure, but the best fix must preserve equivalent SSRF enforcement or carry an explicit maintainer-approved Slack media transport contract.
