## Summary

Keep open: current main still maps `agents add --bind feishu:oc_xxx` into `match.accountId`, while Feishu docs and the router's most-specific routing path require `match.peer` for group conversations; four open PRs already reference this issue as candidate fixes.

## What This Changes

The issue asks for Feishu group/chat bind values generated by `openclaw agents add --bind` to produce routeable peer-shaped bindings instead of accountId-shaped bindings.

## Best Possible Solution

Land one focused linked PR that generates documented peer-shaped Feishu conversation bindings while preserving accountId semantics through an explicit or otherwise safe compatibility path.

## Reproduction Assessment

Yes, source-level: the bind parser stores `feishu:oc_xxx` as `match.accountId`, while the router and Feishu docs require `match.peer` for conversation-level group routing. I did not run a live Feishu gateway flow in this read-only review.

## Solution Assessment

Unclear until maintainers pick the compatibility shape. The stored `match.peer` target is the right routing model, but the best CLI behavior is a product choice between explicit peer syntax and safe Feishu conversation-id detection.
