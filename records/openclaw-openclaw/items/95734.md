## Summary

Keep open: current main and the latest release still do not generate a peer-shaped binding for the reported Feishu group flow, and three open PRs already reference closing this issue. The right cleanup is to review and land one focused fix, then let the issue close from the merged PR.

## What This Changes

Report that `openclaw agents add --bind 'feishu:<group>'` writes a Feishu group id into `match.accountId`, so inbound group messages continue routing to the default agent instead of the requested agent.

## Best Possible Solution

Land one focused PR that makes the agent bind CLI generate a documented peer-shaped binding for Feishu conversation ids while preserving existing accountId semantics or adding a clear explicit account form.

## Reproduction Assessment

Yes, source-level: current `parseBindingSpecs` writes the Feishu group id as `accountId`, while current routing only treats `match.peer` as the conversation-specific match tier. I did not run a live Feishu message route in this read-only review.

## Solution Assessment

Unclear until maintainers pick the compatibility shape: the workaround's `match.peer` target is correct, but the CLI should either add explicit peer syntax or safely recognize Feishu conversation ids without breaking account bindings.
