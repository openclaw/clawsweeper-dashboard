## Summary

Keep open: current main still parses `--bind feishu:oc_test` as an account-scoped binding while Feishu group routing supplies the chat id as `match.peer`, and the linked open fix PR should remain the implementation path until it merges or is rejected.

## What This Changes

This issue requests a CLI binding path that can create Feishu group/chat peer route bindings instead of treating the chat id as a channel account id.

## Best Possible Solution

Land a backward-compatible explicit peer-binding CLI syntax with parser tests and docs, or intentionally document peer bindings as config-only if maintainers do not want the CLI surface.

## Reproduction Assessment

Yes for the config-shape part by source inspection: `agents add` calls `parseBindingSpecs`, which stores `feishu:oc_test` as `match.accountId`, while Feishu inbound routing passes the chat id through `match.peer`. I did not live-send a Feishu group message.

## Solution Assessment

Unclear until maintainer review of the CLI contract: an additive explicit peer syntax like the linked PR is safer than reinterpreting existing `channel:account` specs. The final fix should include parser coverage and user-facing docs for the chosen syntax.
