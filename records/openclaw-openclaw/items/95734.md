## Summary

Keep open: current main still generates an account-scoped binding for the reported Feishu bind form while the router’s conversation-specific path matches `peer`, and three open closing PRs are still candidate fixes rather than merged provenance.

## What This Changes

The issue asks for `openclaw agents add --bind 'feishu:<group-id>'` to generate a routeable peer-shaped binding instead of an accountId-shaped binding that falls through to the default agent.

## Best Possible Solution

Land one focused PR that generates documented peer-shaped Feishu conversation bindings while preserving accountId semantics, then let the merged PR close this issue.

## Reproduction Assessment

Yes, source-level: current main parses `feishu:oc_test` as `match.accountId`, while route resolution only treats `match.peer` as the conversation-specific match tier. I did not run a live Feishu message route in this read-only review.

## Solution Assessment

Unclear until maintainers choose the bind grammar: `match.peer` is the right routing target, but the fix should preserve accountId semantics through plugin-owned recognition or an explicit peer/account syntax.
