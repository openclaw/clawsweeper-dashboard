## Summary

Keep open. Current `main` and the latest release still treat `agents add --bind feishu:oc_test` as an account-scoped binding, while Feishu group routing matches the incoming chat id through `match.peer`; the remaining work needs a maintainer decision on the CLI contract rather than cleanup closure.

## What This Changes

The issue requests a way for `openclaw agents add --bind feishu:<chat_id>` to create a Feishu group peer binding instead of an account binding so group messages route to the new agent.

## Best Possible Solution

Add or document one explicit per-conversation binding path, preferably a peer-specific CLI syntax or flag that creates `match.peer` without changing existing account-binding semantics.

## Reproduction Assessment

Yes: source inspection shows `--bind feishu:oc_test` is serialized as `match.accountId`, while Feishu group inbound routing supplies `peer: { kind: "group", id: chat_id }`. I did not run a live Feishu message flow.

## Solution Assessment

No: the literal shorthand would conflict with the documented account-binding contract. A safer solution is an explicit peer-binding CLI path or clearer docs for manual `match.peer` configuration.
