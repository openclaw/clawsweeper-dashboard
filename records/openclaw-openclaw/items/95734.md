## Summary

Keep open: current main and the latest release still generate the reported Feishu bind as an account-scoped route while Feishu inbound routing supplies a peer-shaped conversation identity, and several open PRs are competing candidate fixes rather than merged superseding work.

## What This Changes

The issue reports that `openclaw agents add --bind 'feishu:<chat_id>'` writes `match.accountId` even though Feishu conversation routing needs a `match.peer` binding.

## Best Possible Solution

Land one focused compatibility-safe bind grammar or channel-owned resolution path that generates `match.peer` for Feishu conversation ids while preserving account binding semantics, with docs, regression tests, and real CLI or Feishu routing proof.

## Reproduction Assessment

Yes, source-level: current `parseBindingSpecs` writes the Feishu group id as `match.accountId`, while Feishu inbound routing supplies `peer` and `resolveAgentRoute` prioritizes `binding.peer`. I did not run a live Feishu route in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public compatibility shape: `match.peer` is the right routing target, but the CLI needs either explicit peer syntax or safe Feishu conversation-id recognition without breaking account binds.
