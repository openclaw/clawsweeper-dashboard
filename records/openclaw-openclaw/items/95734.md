## Summary

Keep open: current main and the latest release still generate the reported Feishu `--bind` value as `match.accountId`, while Feishu inbound routing supplies a conversation `peer` and the router treats peer as the specific routing tier. Several open PRs already reference this issue, so the useful maintainer action is to choose one compatibility-safe fix rather than close the issue or queue another branch.

## What This Changes

The issue asks for `openclaw agents add --bind 'feishu:<group-id>'` to create a routeable Feishu conversation binding instead of an account-scoped binding that falls through to the default agent.

## Best Possible Solution

Land one focused PR that makes Feishu conversation binds produce a documented peer-shaped route while preserving existing account semantics or adding a clear explicit account form, with parser/generator tests and redacted CLI or Feishu route proof.

## Reproduction Assessment

Yes, source-level: current main parses `feishu:oc_test` into `match.accountId`, while Feishu inbound dispatch supplies `peer` and the route resolver's conversation tier looks at `match.peer`. I did not execute a live Feishu group message route in this read-only review.

## Solution Assessment

Unclear until maintainers pick the compatibility shape: the target `match.peer` route is correct, but the CLI must avoid silently breaking the documented `--bind <channel>:<account>` account form.
