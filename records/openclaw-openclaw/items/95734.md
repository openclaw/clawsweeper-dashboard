## Summary

Keep open: current main still writes the reported Feishu bind value as `match.accountId`, while Feishu inbound routing and docs use `match.peer` for conversation routing. Several open PRs are candidate fixes, so this issue should remain open until one compatibility-safe fix lands.

## What This Changes

The issue reports that `openclaw agents add --bind feishu:<groupId>` persists the group id as an account binding, causing Feishu group messages to route to the default agent instead of the bound agent.

## Best Possible Solution

Land one focused compatibility-safe PR that generates peer-shaped Feishu conversation bindings while preserving existing account binding semantics, then let this issue close from the merged fix.

## Reproduction Assessment

Yes, source-level: current main writes `feishu:oc_test` as `match.accountId`, while Feishu inbound routing supplies a `peer` and the router prioritizes `binding.peer`. I did not run a live Feishu route in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility shape: `match.peer` is the right target, but the fix needs either safe Feishu id inference or explicit peer/account syntax without breaking account binds.
