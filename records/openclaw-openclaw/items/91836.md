- `P1`: The report describes parent orchestration continuing to route work to saturated unresponsive child agents for hours, which can break active agent workflows.
- `impact:session-state`: The issue is about parent and child session state drifting because child context pressure is not surfaced to the controlling session.

## Summary

Keep open. Current main has subagent status/listing, parent session linkage, context-budget reporting, and compaction, but it still lacks the requested automatic child context-pressure notification to the parent orchestrator, operator alerting, or recovery routing; that is a product/API decision rather than conservative cleanup.

## What This Changes

Request automatic parent/operator notification and recovery routing when child or sub-agent sessions approach or exceed a configurable context threshold.

## Best Possible Solution

Define one opt-in context-pressure escalation contract that reuses existing parentSessionKey/subagent lineage, context-budget status, diagnostics, and compaction surfaces, with maintainer-approved defaults, docs, and focused tests.

## Reproduction Assessment

No high-confidence live reproduction was performed for the exact 524% context fleet failure. Source inspection does confirm the requested automatic parent context-threshold notification and recovery path is absent on current main.

## Solution Assessment

Unclear: the user’s desired parent alert/recovery behavior is plausible, but adding parent_agent_id and ad hoc thresholds is not clearly the best design. A maintainable solution should reuse existing parentSessionKey lineage, context-budget status, diagnostics, and compaction contracts after maintainer product review.
