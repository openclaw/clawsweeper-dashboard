## Summary

Keep open: current main still lacks a durable wake-inbox drain, and the remaining work depends on the unresolved live-session injection/security contract rather than conservative cleanup.

## What This Changes

The issue requests a durability layer that drains unconsumed wake-inbox records into a matching live session on activation, with ordering, dedupe, consume-after-write, and partial-failure retry semantics.

## Best Possible Solution

First settle the supported live user-turn injection contract in https://github.com/openclaw/openclaw/issues/83809, then implement a generic durable replay path with ordering, dedupe, consume-after-write, partial-failure handling, and audit states that distinguish stdin write from live-session application.

No as a strict live race repro: there are no runnable steps or logs for the accepted-but-undelivered session activation race. Yes at source level: current main has no wake-inbox terms and the live-session path writes only the active prompt.

## Solution Assessment

Unclear: the requested replay rules are plausible, but the best fix depends on the maintainer decision for the live injection contract, storage/audit model, and security boundary.
