## Summary

Keep open: the report is a valid current packaged-install mismatch, not stale cleanup. Auto-enable can still add a configured channel id to `plugins.allow` when no manifest-backed plugin claims it, while config validation still treats `plugins.allow` as discoverable plugin ids only.

## What This Changes

The issue asks OpenClaw to stop producing an unfixable `plugins.allow` doctor/config warning when an enabled Discord channel causes the gateway to re-add `discord` to the plugin allowlist.

## Best Possible Solution

Align config validation with the channel auto-enable contract so built-in/configured channel ids that the gateway materializes into `plugins.allow` are not reported as stale plugin ids, with packaged-mode regression coverage.

## Reproduction Assessment

Yes, by source inspection: packaged OpenClaw can lack `dist/extensions/discord`, auto-enable can still add `discord` from `channels.discord`, and validation then warns because `discord` is absent from the manifest registry. I did not run a live npm-package restart because this review is read-only.

## Solution Assessment

Yes, a narrow validation/contract fix is the best path: make the warning path recognize the same configured built-in channel ids that auto-enable preserves, or stop preserving only those ids when no plugin manifest can exist. Adding a config option would be broader than needed.
