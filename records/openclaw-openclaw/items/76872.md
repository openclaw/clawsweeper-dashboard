## Summary

Keep open: the report describes a real doctor/config consistency bug. In packaged layouts where the Discord plugin manifest is not present, gateway auto-enable can keep materializing `discord` in `plugins.allow`, while config validation still treats allowlist entries as plugin manifest ids only and emits the stale-plugin warning.

## What This Changes

The issue asks doctor to stop warning that channel-derived `discord` in `plugins.allow` is stale when the gateway re-adds it for an enabled Discord channel.

## Best Possible Solution

Align config validation with the channel-aware doctor/auto-enable model so recognized channel ids that may be managed in `plugins.allow` are not reported as stale plugin ids, while true unknown plugin ids remain warned or rejected as they are today.

## Reproduction Assessment

Yes from source inspection: simulate an installed-package registry without a Discord plugin manifest but with `channels.discord.enabled` and `plugins.allow: ["discord"]`; validation warns because `discord` is not in plugin `knownIds`, while auto-enable can re-add the channel id.

## Solution Assessment

Yes: the narrowest maintainable fix is to make the stale allowlist validation channel-aware, matching the existing stale-plugin scan behavior, or otherwise stop auto-enable from persisting channel ids that validation will reject.
