## Summary

Keep open: the PR is aimed at a real source-reproducible TUI session-state bug, but the current patch still misses a pending-run reset case and lacks after-fix real multi-client behavior proof.

## What This Changes

The PR adds a TUI `sessions.subscribe` hook, handles `sessions.changed` reset/new events by reloading the active session and clearing run state, and adds TUI handler tests plus a changelog entry.

## Best Possible Solution

Land a rebased TUI consumer for the existing `sessions.subscribe` and `sessions.changed` reset/new contract that clears active, pending, local, watchdog, session, and finalized run state, backed by redacted real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` events, while the TUI neither subscribes to nor dispatches that event family, so an external reset can leave TUI history and run state stale.

## Solution Assessment

No, not as-is. Reusing `sessions.subscribe` and `sessions.changed` is the narrow maintainable direction, but the branch needs pending-run reset/suppression, rebase verification, and real multi-client proof before merge.
