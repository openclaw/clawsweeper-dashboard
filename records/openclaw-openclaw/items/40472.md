## Summary

Keep open: current main still has the source-proven TUI reset-refresh gap, and this PR is the canonical closing path, but it is not merge-ready because it lacks real behavior proof and still needs current-main pending-run cleanup.

## What This Changes

This PR adds a TUI `sessions.subscribe` client hook, dispatches `sessions.changed` reset/new events to refresh the active session, clears stale TUI run state, adds handler tests, and updates the changelog.

## Best Possible Solution

Land or replace this PR with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that clears active, pending, local, watchdog, session, and finalized run state, backed by real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor dispatches that event family, so resetting the active session from another client can leave stale TUI state visible.

## Solution Assessment

No, not as-is. Reusing `sessions.subscribe` and `sessions.changed` is the right narrow seam, but the branch needs `pendingChatRunId` reset/suppression and real behavior proof before merge.
