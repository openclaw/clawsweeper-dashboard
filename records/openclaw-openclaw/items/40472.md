## Summary

Keep open: current main still lacks TUI subscription/dispatch for the existing session lifecycle event, and the linked issue remains open with this PR as the implementation path. The branch is useful but not merge-ready because the available proof is test/code-review only and the reset path still needs to account for current-main pending-run state when rebased.

## What This Changes

The PR adds a TUI `sessions.subscribe` wrapper, dispatches `sessions.changed` reset/new payloads to reload the active session and clear stale run state, and adds focused handler tests plus a changelog entry.

## Best Possible Solution

Land or replace this PR with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that clears active, pending, local, watchdog, session, and finalized run state, with real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor dispatches that event family, so resetting the active session from another client can leave stale TUI history/token state visible.

## Solution Assessment

No, not yet. Reusing the existing `sessions.subscribe` / `sessions.changed` contract is the right narrow direction, but this branch must also suppress and clear current-main `pendingChatRunId` and provide real behavior proof before merge.
