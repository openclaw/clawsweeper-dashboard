## Summary

Keep this PR open: current main has the gateway `sessions.changed` reset signal, but the TUI still does not subscribe to or dispatch it, so the PR remains useful; the runtime approach looks sound, with one narrow changelog wording issue before normal rebase and validation.

## What This Changes

The PR adds a TUI `sessions.changed` subscription and reset/new handler that clears tracked run/stream state, reloads the active session history, plus regression tests and a changelog entry.

## Best Possible Solution

Land this PR or an equivalent rebased patch after narrowing the changelog wording, preserving the existing `sessions.subscribe` / `sessions.changed` contract, and close #38966 once the fix ships.

## Reproduction Assessment

Yes. The source-level path is high-confidence: current main emits `sessions.changed` on `sessions.reset`, but the TUI neither subscribes to nor dispatches that event, so an externally reset active session cannot refresh the TUI view.

## Solution Assessment

Yes, with a small changelog caveat. Consuming the existing `sessions.subscribe` / `sessions.changed` lifecycle contract in the TUI is the narrowest maintainable fix; a new gateway event family is not needed.
