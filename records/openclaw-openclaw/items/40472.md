## Summary

Keep open: the bug still exists on current main and in the latest release, and this PR remains the canonical fix path, but the branch is merge-conflicting, misses current-main pending-run state in reset cleanup, and lacks real behavior proof.

## What This Changes

The PR adds TUI subscription and handling for `sessions.changed` reset/new events, clears tracked run state, reloads history, and adds focused handler tests plus a changelog entry.

## Best Possible Solution

Land or replace this PR with a rebase-aware TUI consumer for `sessions.subscribe`/`sessions.changed` reset events that also clears pending run state and includes real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits `sessions.changed` only to subscribers, while the TUI neither subscribes to nor dispatches that event family, so resetting the active TUI session from another client leaves stale state visible.

## Solution Assessment

No, not yet. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the right narrow direction, but the branch needs the current-main pending-run cleanup and real behavior proof before merge.
