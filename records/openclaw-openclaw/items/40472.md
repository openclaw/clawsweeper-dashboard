## Summary

Keep this PR open. Current main has the gateway `sessions.changed` publisher and subscription RPC, but the TUI still does not subscribe to or dispatch that event, so the linked external-reset bug remains unresolved; this PR is the active contributor path with a recent maintainer repair, though it needs normal review/rebase attention.

## What This Changes

The PR adds a TUI backend wrapper for `sessions.subscribe`, dispatches `sessions.changed` events in the TUI, clears active stale run/stream state on current-session `new`/`reset`, adds regression coverage, and documents the fix in the changelog.

## Best Possible Solution

Land this PR or an equivalent rebased patch that keeps the existing gateway lifecycle event contract, subscribes the TUI on connect/reconnect, handles active-session `new` and `reset` by clearing stale run/stream/local state and reloading history/session info, preserves the current streaming watchdog reconnect behavior, and closes #38966 after the fix ships.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: run the TUI on a session, reset the same session from another client, and current main cannot refresh because the gateway emits `sessions.changed` only to subscribers while the TUI neither subscribes nor handles that event.

## Solution Assessment

Yes, with a rebase and wording caveat. Consuming the existing `sessions.subscribe` / `sessions.changed` contract in the TUI is the narrow maintainable fix; a broader protocol change is only needed for fully disambiguating every theoretical late same-session event.
