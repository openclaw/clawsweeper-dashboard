## Summary

Keep open: the fix direction matches the current gateway session-change contract, but this branch is conflicting with main, misses current pending-run state during reset conflict resolution, and still lacks real multi-client TUI proof.

## What This Changes

The PR adds a TUI backend subscription method, subscribes on connect to `sessions.changed`, routes reset/new session events to a TUI handler that refreshes history and clears stale run state, and adds focused TUI handler tests plus a changelog entry.

## Best Possible Solution

Land or replace this with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that clears active, pending, local, watchdog, session, finalized, and suppressed run state, backed by redacted multi-client proof.

## Reproduction Assessment

Yes, by source inspection: current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor handles that event family. I did not run a live multi-client flow in this read-only review.

## Solution Assessment

No, not merge-ready as-is. Reusing `sessions.subscribe` and `sessions.changed` is the right contract, but the PR must be rebased and extended to clear/suppress current-main `pendingChatRunId` before merge.
