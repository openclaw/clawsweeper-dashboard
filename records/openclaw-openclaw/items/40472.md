## Summary

Keep open: this PR still targets a valid current-main TUI reset bug and remains the canonical closing path for https://github.com/openclaw/openclaw/issues/38966, but it is not merge-ready because real behavior proof is missing and the branch must account for current-main pending-run state.

## What This Changes

The PR adds TUI subscription and dispatch for `sessions.changed` reset/new events, clears stale run state, updates handler tests, and adds a changelog entry.

## Best Possible Solution

Land or replace this PR with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that clears active, pending, local, watchdog, session, and finalized run state.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor dispatches that event family, so resetting the active session from another client can leave stale TUI state visible.

## Solution Assessment

No, not merge-ready. Reusing `sessions.subscribe` and `sessions.changed` is the right narrow seam, but the branch must suppress and clear `pendingChatRunId` after rebase and provide real behavior proof before merge.

## Review Findings

Full review comments:

- **[P2] Clear pending run IDs on reset:** `src/tui/tui-event-handlers.ts:566-568`
  - body: Current main records `state.pendingChatRunId` after `chat.send` until the first chat event arrives, and agent events treat that id as a pending run.
