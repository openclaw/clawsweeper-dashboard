## What This Changes

The PR adds TUI subscription and dispatch for `sessions.changed` reset/new events, clears run/stream state on matching active-session resets, and adds handler tests plus a changelog entry.

## Best Possible Solution

Land a rebased TUI consumer for the existing `sessions.subscribe` and `sessions.changed` reset/new contract that clears active, pending, local, watchdog, session, and finalized run state, backed by redacted multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset events, while the TUI neither subscribes to `sessions.subscribe` nor dispatches `sessions.changed`, so an external reset can leave stale TUI state.

## Solution Assessment

No, not as-is. Reusing `sessions.subscribe` and `sessions.changed` is the right narrow direction, but this PR must clear pending-run state and be rebased/proven before merge.
