## Best Possible Solution

Land a rebased TUI consumer for the existing `sessions.subscribe` and `sessions.changed` reset/new contract that clears active, pending, local, watchdog, session, and finalized run state, backed by redacted real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor dispatches that event family, so an external reset can leave TUI history and run state stale.

## Solution Assessment

No, not as-is. Reusing `sessions.subscribe` and `sessions.changed` is the narrow maintainable direction, but the branch needs conflict resolution, pending-run reset/suppression, and real multi-client proof before merge.
