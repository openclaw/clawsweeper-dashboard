## Summary

Keep open: current main still has the TUI subscription/dispatch gap for external session resets, but this branch still misses newer pending-run state and does not include real multi-client behavior proof.

## What This Changes

The branch adds TUI subscription/dispatch for existing `sessions.changed` reset/new events, clears tracked run state before reloading active-session history, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land or replace this PR with a current-main TUI consumer for `sessions.subscribe`/`sessions.changed` that suppresses and clears active, finalized, local, watchdog, and pending run state, then prove the multi-client reset flow.

## Reproduction Assessment

Yes by source inspection. Current main emits `sessions.changed` for `sessions.reset` only to subscribers, while the TUI neither subscribes nor dispatches that event family, so resetting the active TUI session from another client leaves the TUI stale.

## Solution Assessment

No, not yet. Reusing the existing `sessions.subscribe` / `sessions.changed` contract is the narrow maintainable direction, but the branch must suppress and clear `pendingChatRunId` and include real behavior proof before merge.

## Review Findings

  - confidence: 0.82
