## What This Changes

The PR adds TUI subscription and handling for `sessions.changed` reset/new events, refreshes active-session history, clears stale run/stream state, and updates backend types, tests, and the changelog.

## Best Possible Solution

Land a rebased TUI consumer for the existing `sessions.subscribe` and `sessions.changed` reset/new contract that clears active, pending, local, watchdog, session, and finalized run state, backed by redacted multi-client proof.

## Reproduction Assessment

Yes. Source inspection shows current main emits subscription-gated `sessions.changed` reset events while the TUI neither subscribes to nor dispatches that event family, and the PR's pending-run race is traceable from the diff.

## Solution Assessment

Yes for the direction, but no for this exact patch yet. Reusing `sessions.subscribe` and `sessions.changed` is the narrow maintainable fix, but the patch must clear/suppress `pendingChatRunId`, rebase cleanly, and include real behavior proof.

Full review comments:

- **[P2] Clear pending runs during reset:** `src/tui/tui-event-handlers.ts:563-565`
