## Summary

Keep open: this is a valid TUI session-state bug fix using the existing `sessions.changed` seam, but the branch is not merge-ready because reset suppression still misses current-main pending run state and after-fix real behavior proof is absent.

## What This Changes

The PR adds a TUI `sessions.subscribe` hook, handles `sessions.changed` reset/new events by reloading the active session and clearing run state, and adds TUI handler tests plus a changelog entry.

## Best Possible Solution

Land a rebased TUI consumer for existing `sessions.subscribe` and `sessions.changed` reset/new events that clears active, pending, local, watchdog, session, and finalized run state, backed by real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits and advertises `sessions.changed`, while the TUI neither subscribes to nor dispatches that event family, so an external reset can leave TUI history and run state stale.

## Solution Assessment

No, not as-is. Reusing `sessions.subscribe` and `sessions.changed` is the right narrow fix path, but the branch needs pending-run reset/suppression and real multi-client proof before merge.
