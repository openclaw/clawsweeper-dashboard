## Summary

Keep open: the linked bug is still valid on current main because the gateway emits subscription-gated `sessions.changed` reset events while the TUI does not subscribe to or handle them. The branch is useful but not merge-ready because it still misses the pending-run reset case, lacks real multi-client proof, and is currently conflicting.

## What This Changes

The PR adds TUI subscription and handling for gateway `sessions.changed` reset/new events, clears stale stream/run state on matching current-session resets, and adds focused handler tests plus a changelog entry.

## Best Possible Solution

Land a rebased TUI consumer for the existing `sessions.subscribe` and `sessions.changed` reset/new contract that clears and suppresses active, pending, local, watchdog, session, and finalized run state, backed by redacted multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor handles that event family, so a multi-client reset can leave stale TUI state.

## Solution Assessment

No, not yet. The direction is the right maintainable fix, but the PR needs pending-run clearing/suppression, rebase against current main, and real behavior proof before merge.
