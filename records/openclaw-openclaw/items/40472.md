## Summary

Keep open: current main and the latest shipped tag still lack the TUI subscription/dispatch path, and this PR is the canonical fix path, but the branch still misses current-main pending run state and has only test/code-review proof.

## What This Changes

This PR adds TUI subscription and handling for `sessions.changed` reset/new events, clears tracked TUI run state, adds focused handler tests, and updates the changelog.

## Best Possible Solution

Land or replace this PR with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` that suppresses and clears active, pending, finalized, local, watchdog, and tracked run state, then prove the multi-client reset flow.

## Reproduction Assessment

Yes, by source inspection. Current main emits `sessions.changed` for resets only to subscribers, while the TUI neither subscribes to nor dispatches that event family, so resetting the active TUI session from another client leaves stale state visible.

## Solution Assessment

No, not yet. Reusing the existing `sessions.subscribe` and `sessions.changed` contract is the narrow maintainable direction, but this branch still needs `pendingChatRunId` suppression/clearing plus real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Suppress pending runs during reset:** `src/tui/tui-event-handlers.ts:198-213`
  - body: Current main records the generated `chat.send` run id in `state.pendingChatRunId` before the first chat event arrives. This reset suppression only captures active, watchdog, session, and finalized runs, so an external reset in that waiting gap can leave the pending old run unsuppressed and let its first late event bind after the reset. Add `pendingChatRunId` to the suppressed set and clear it in the reset path before reloading history.
  - confidence: 0.88
