## Summary

Keep open: the PR targets a real current-main TUI bug, but the branch is not merge-ready because it still misses current-main pending-run cleanup and has no after-fix real behavior proof for the multi-client reset flow.

## What This Changes

The PR adds a TUI `sessions.subscribe` wrapper, dispatches `sessions.changed` reset/new events to reload the active session and clear stale run state, and updates TUI handler tests plus the changelog.

## Best Possible Solution

Land or replace this PR with a current-main TUI consumer for `sessions.subscribe` and `sessions.changed` reset/new events that clears active, pending, local, watchdog, session, and finalized run state, backed by real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits subscription-gated `sessions.changed` reset/new events, while the TUI neither subscribes to nor dispatches that event family, so resetting the active session from another client can leave stale TUI state visible.

## Solution Assessment

No, not yet. Reusing `sessions.subscribe` / `sessions.changed` is the right narrow seam, but the branch must suppress and clear `pendingChatRunId` and provide real behavior proof before merge.

## Review Findings
