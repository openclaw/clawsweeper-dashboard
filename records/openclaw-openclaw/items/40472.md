## Summary

Keep open: the PR addresses a valid current-main TUI reset-refresh bug and the code path is mostly correct, but it still needs after-fix real multi-client behavior proof before merge and one P3 changelog wording fix.

## What This Changes

The PR adds a TUI `sessions.changed` subscription and handler, clears stale run state after current-session reset/new events, updates focused TUI handler tests, and adds a changelog entry.

## Best Possible Solution

Land or replace this with a rebased TUI consumer for the existing `sessions.subscribe` / `sessions.changed` reset lifecycle, with tracked stale run cleanup, narrow changelog wording, and redacted real multi-client proof.

## Reproduction Assessment

Yes. Source inspection shows current main emits `sessions.changed` after `sessions.reset` and only delivers it to subscribers, while the TUI neither subscribes to nor handles that event family.

## Solution Assessment

No, not yet. Reusing the existing session lifecycle contract is the right direction, but the branch needs to carry current-main pending-run cleanup into the reset path and narrow the changelog claim before it is the best implementation.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P2] Suppress pending runs when resetting:** `src/tui/tui-event-handlers.ts:566-568`
  - body: The reset handler suppresses active/session/finalized/watchdog run IDs, but current main also tracks a pre-event `pendingChatRunId` returned by `chat.send`. If an external reset arrives before the first chat event, that pending run is not suppressed or cleared, so a late delta/final for the old run can still be accepted and repaint the reset session. Add `state.pendingChatRunId` to the suppressed set before clearing it with the rest of the reset state.
  - confidence: 0.84
- **[P3] Narrow the changelog reset guarantee:** `CHANGELOG.md:16`
  - body: The changelog says late old-run events cannot repaint reset history, but the handler only suppresses run IDs known when the reset event arrives, and the PR body leaves broader same-session late-event disambiguation out of scope. Please narrow this to the active-session refresh and tracked stale run cleanup the patch guarantees.
  - confidence: 0.86
