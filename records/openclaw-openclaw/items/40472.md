## Summary

Keep open: the PR targets a real current-main TUI session-state bug, but it is not merge-ready because the branch still misses pending-run cleanup, lacks required real multi-client proof, and is currently conflicting.

## What This Changes

The PR makes the TUI subscribe to session lifecycle changes, refresh the active session after external reset/new events, clear stale run state, and adds targeted TUI handler tests.

## Best Possible Solution

Rebase onto current main, consume the existing `sessions.subscribe` and `sessions.changed` contract, clear active/pending/reconnect/watchdog/local/session/finalized run state on matching reset/new events, and back it with real multi-client proof.

## Reproduction Assessment

Yes, by source inspection. Current main emits and advertises `sessions.changed`, while the TUI does not subscribe to or dispatch that event family, so an external reset can leave TUI history and run state stale.

## Solution Assessment

No, not as-is. Reusing `sessions.subscribe` and `sessions.changed` is the right narrow fix path, but this branch still needs pending-run cleanup, conflict resolution, and real multi-client proof before merge.
