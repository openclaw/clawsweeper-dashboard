## Summary

Keep this PR open: current main still has the Feishu ambient-proxy behavior, but the branch needs proxy-policy review, test cleanup, docs, conflict resolution, and inspectable after-fix proof before merge.

## What This Changes

The PR changes Feishu WebSocket proxy handling so ambient proxy env is ignored by default and only used when `OPENCLAW_FEISHU_WS_USE_PROXY=1` is set.

## Best Possible Solution

Keep the Feishu-owned direct-by-default fix, but preserve managed proxy routing, restore test env state, document the opt-in and upgrade behavior, and require real patched-branch proof before merge.

## Reproduction Assessment

Yes. On current main, setting proxy env before `createFeishuWSClient()` follows `getWsProxyAgent()` to `resolveAmbientNodeProxyAgent()` and passes an agent into Lark `WSClient`; related comments also include real failure logs.

## Solution Assessment

No, not as submitted. The narrow Feishu fix is plausible, but it needs managed-proxy handling, env cleanup, docs, conflict resolution, and after-fix proof before it is the maintainable solution.

- **[P1] Preserve managed proxy routing:** `extensions/feishu/src/client.ts:81-83`
  - body: The new guard disables every explicit Feishu WS proxy agent unless `OPENCLAW_FEISHU_WS_USE_PROXY=1` is set. Since the shared Vitest config uses `isolate: false`, later tests in the worker can unexpectedly see Feishu WS proxy opt-in enabled.
  - confidence: 0.92
