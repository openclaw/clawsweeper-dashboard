## Summary

Keep open. Current main and v2026.5.7 still use ambient HTTP(S) proxy env for Feishu WebSocket, and this PR is a plausible focused fix, but it still needs test env cleanup, operator docs, inspectable patched-branch proof, and a maintainer decision on env-only versus config-file proxy control.

## What This Changes

The PR makes Feishu WebSocket proxy-agent creation opt in via `OPENCLAW_FEISHU_WS_USE_PROXY=1` and updates Feishu proxy tests for the new default.

## Best Possible Solution

Land a Feishu-owned default-direct WebSocket proxy fix after restoring test env state, documenting the opt-in, and deciding whether the escape hatch belongs only in env or also in Feishu config.

## Reproduction Assessment

Yes, source-reproducible. On current main, setting HTTP(S) proxy env and calling `createFeishuWSClient()` follows `getWsProxyAgent()` to `resolveAmbientNodeProxyAgent()` and passes the resulting agent to Lark `WSClient`; I did not run a live Feishu gateway.

## Solution Assessment

No, not as submitted. The Feishu-local opt-in gate is the narrow code path, but the PR still needs env cleanup, docs, inspectable after-fix proof, and a proxy-policy decision before it is the best mergeable solution.
