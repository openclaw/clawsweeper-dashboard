## Summary

Keep open: current main can already forward `providerOptions.deepgram.utterances`, but it still drops the returned Deepgram utterance payload and has no OpenClaw output/API surface for timestamps, words, or confidence metadata.

## What This Changes

The issue requests native Deepgram `utterances` support alongside existing Deepgram smart-format handling.

## Best Possible Solution

Document `providerOptions.deepgram.utterances` if pass-through is the intended support level; otherwise add an additive audio transcription metadata contract and update the Deepgram plugin, tests, and docs together.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high confidence that the query flag can already be sent while structured utterance metadata has no current OpenClaw result surface.

## Solution Assessment

No, the proposed two-line compat change is not the best complete solution because generic provider option pass-through already handles the request flag. Native support needs a maintainer decision between docs-only clarification and an additive metadata API.
