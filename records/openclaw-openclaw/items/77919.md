## Summary

Keep open: current main already forwards `providerOptions.deepgram.utterances`, but it does not document that option or expose Deepgram's returned utterance/word metadata downstream, so the remaining work needs a maintainer API/product decision.

## What This Changes

The issue requests native Deepgram `utterances` support for audio transcription alongside `smartFormat`, including downstream access to per-utterance and per-word metadata.

## Best Possible Solution

Document the existing pass-through if parameter-only support is intended; otherwise add an additive transcription metadata contract and preserve Deepgram `results.utterances` through the provider/SDK output.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that the query flag can already be sent, while structured utterance metadata has no current OpenClaw result surface.

## Solution Assessment

No, the proposed two-line query change is not the best complete solution because generic pass-through already handles `utterances`; native support needs docs and likely an additive media-output metadata design.
