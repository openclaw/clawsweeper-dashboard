## Summary

Keep open: current main can already forward `providerOptions.deepgram.utterances`, but the requested native value is structured utterance/word metadata and OpenClaw's current audio transcription result only exposes transcript text and model. This needs a maintainer product/API decision, not a two-line compat fix.

## What This Changes

The issue asks OpenClaw to support Deepgram `utterances` alongside `smartFormat` for segmented audio transcripts with timestamps and confidence data.

## Best Possible Solution

Keep the existing generic `providerOptions.deepgram.utterances` pass-through, document it if that is the intended support level, and add an additive audio transcription metadata contract only if OpenClaw should expose provider-returned utterances downstream.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that the query flag can already be sent while structured utterance metadata has no current OpenClaw result surface.

## Solution Assessment

No, the proposed two-line query change is not the best complete solution because the generic pass-through already handles `utterances`; native support would require docs and likely an additive media-output metadata design.
