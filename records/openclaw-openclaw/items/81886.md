## Summary

Keep open: the PR targets a real gateway sessions bug, but it is not merge-ready because the new compatibility check still clears alias-compatible auth profiles and the PR body only provides test/check output as real behavior proof.

## What This Changes

The PR adds sessions.patch auth-profile preservation logic and regression tests for same-provider model switches and cross-provider cleanup.

## Best Possible Solution

Preserve an existing session authProfileOverride during sessions.patch when it remains compatible under the same alias-aware auth-profile contract used at runtime, while still clearing truly incompatible provider-scoped profiles.

## Reproduction Assessment

Yes. Source inspection shows the current Dashboard path sends only key/model to sessions.patch, and current main then calls the model override helper without any auth-profile preservation signal, which takes the delete branch.

## Solution Assessment

No. The PR handles exact same-provider prefixes, but the compatibility check should account for existing provider aliases such as z.ai -> zai and plugin auth aliases before deciding to clear the override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle provider aliases before clearing auth profiles:** `src/gateway/sessions-patch.ts:92`
  - body: `shouldPreserveSessionAuthProfileOverride` compares the profile-id prefix directly with `resolved.ref.provider`. Existing auth-profile resolution treats aliases like `z.ai` -> `zai` and plugin auth aliases as compatible, so a session pinned to `zai:work` can still lose the override when Dashboard switches to a `z.ai/*` model. Normalize through the existing provider-auth alias contract or defer compatibility to the auth-profile resolver before returning false.
  - confidence: 0.88
