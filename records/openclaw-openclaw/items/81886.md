## Summary

Keep open: this PR addresses a real `sessions.patch` auth-profile regression and now includes acceptable post-patch terminal proof, but the patch still drops auth profiles that are compatible through provider-auth aliases.

## What This Changes

The PR changes gateway `sessions.patch` to preserve same-provider `authProfileOverride` during model switches and adds regression tests for same-provider preservation and cross-provider cleanup.

## Best Possible Solution

Use the existing alias-aware auth-profile compatibility contract for preservation, keep cross-provider cleanup, and add a regression for a provider-auth alias case.

## Reproduction Assessment

Yes. Source inspection shows the Dashboard sends only `key` and `model`, and current main then calls the model override helper without a preservation signal, which clears the stored auth override.

## Solution Assessment

No. The PR is on the right path, but it should normalize through `resolveProviderIdForAuth` or reuse the auth-profile resolver's compatibility rules before deciding to clear an existing profile.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Normalize both sides through the existing alias-aware auth contract, or delegate compatibility to that resolver, before returning false.
