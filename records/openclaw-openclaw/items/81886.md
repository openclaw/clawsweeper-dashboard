## Summary

Keep this PR open: it addresses a real source-reproducible Gateway/session bug, but the patch has a small correctness hole around custom auth profile ids and the contributor proof is still test-only.

## What This Changes

The PR changes `sessions.patch` to preserve auth profile override fields for same-provider model patches and adds gateway regression tests for preservation and provider-prefixed cleanup.

## Best Possible Solution

Preserve auth overrides only when the old and new provider are the same or when compatibility is proven through the existing auth-profile metadata path, with regression coverage for unprefixed cross-provider profile ids.

## Reproduction Assessment

Yes. The source path is high-confidence: Dashboard sends a model-only `sessions.patch`, current main passes no auth-profile data into `applyModelOverrideToSessionEntry`, and the existing gateway test asserts the auth override is cleared.

## Solution Assessment

No, not quite. The server-side `sessions.patch` boundary is the right place to fix this, but the PR should not infer compatibility solely from the profile id string because custom ids and provider aliases exist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clear unknown-profile pins on provider switches:** `src/gateway/sessions-patch.ts:87`
  - body: This preserves any `authProfileOverride` without a `:` regardless of the provider being selected. Auth profile ids are user-chosen independently from their provider metadata, so a valid custom id like `work` for one provider would survive a Dashboard switch to another provider even though `sessions.patch` has no store metadata proving compatibility. Please preserve colon-less ids only when the previous and next provider match, or resolve compatibility through the existing auth-profile metadata path, and add a cross-provider regression test for a custom id.
  - confidence: 0.86
