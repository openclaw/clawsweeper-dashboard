## Summary

Keep open. Current main still stringifies raw parsed config values in both full and path-scoped `/config show` chat replies, and the same-author closing PR at https://github.com/openclaw/openclaw/pull/65637 is still open, conflicting, and not merged.

## What This Changes

The issue reports that `/config show` chat replies expose unredacted config secrets such as tokens, passwords, and provider API keys.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/65637 or an equivalent narrow maintainer-approved fix that redacts before both full and path-scoped `/config show` rendering, with focused regression coverage and security release handling.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a config snapshot containing a sensitive value is cloned from `snapshot.parsed` and directly stringified by `/config show` or `/config show gateway.auth.token`. I did not run a live chat-channel repro in this read-only review.

## Solution Assessment

Yes. Reusing the existing schema-aware config redaction helper before lookup and rendering is the narrowest maintainable fix and matches the gateway `config.get` behavior.
