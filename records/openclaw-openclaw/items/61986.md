## Summary

Keep open. Current main still turns each queued Android `notifications.changed` event into a near-immediate heartbeat wake, the gateway tests assert that behavior, and an open fork PR references this issue but is not merged and appears to change wake scoping rather than wake timing.

## What This Changes

Android companion notification events should be buffered as context for the next scheduled heartbeat instead of triggering an immediate agent run per event.

## Best Possible Solution

Preserve queued notification context, defer ordinary Android notification changes to the next scheduled heartbeat, and cover any explicit immediate-wake exception with gateway and Android policy tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass a valid `notifications.changed` payload into `handleNodeEvent`; current main enqueues the event and calls `requestHeartbeat`, whose default delay is 250 ms.

## Solution Assessment

No for current main. The narrow maintainable fix is to keep notification context queued while removing ordinary immediate wakes; the linked PR's stated helper swap changes target scoping, not wake timing.
