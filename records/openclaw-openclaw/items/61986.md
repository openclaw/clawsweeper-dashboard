## Summary

Keep open. Current main still converts each queued Android `notifications.changed` event into a near-immediate heartbeat wake, and the gateway tests assert that behavior; the linked fork PR is open, unmerged, and its stated helper change appears insufficient under current helper semantics.

## What This Changes

Android companion notification changes should be buffered until the next configured heartbeat instead of causing one agent run per event.

## Best Possible Solution

Keep notification context enqueued, defer ordinary Android notification changes to the next scheduled heartbeat, and make any immediate-wake exception explicit and covered by gateway and Android policy tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: pass a valid `notifications.changed` payload into `handleNodeEvent`; current tests and source show it enqueues the notification and calls `requestHeartbeatNow`.

## Solution Assessment

No for current main. The narrow maintainable fix is to preserve queued notification context while removing ordinary immediate wakes; the linked PR's stated helper swap changes wake scoping, not wake timing.
