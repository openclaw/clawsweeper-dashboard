## Summary

Close as implemented. Current main routes Android notification wakes through the typed event-wake path and the shared heartbeat cooldown/flood gate, so noisy notification streams no longer bypass the configured heartbeat interval; the fix shipped in v2026.5.2.

## What This Changes

Android companion notification changes should be buffered into heartbeat cadence instead of triggering an agent run for each notification event.

## Best Possible Solution

Keep the shipped centralized event-wake cooldown; any stricter policy that defers the first-ever event wake would be a separate product decision.

## Reproduction Assessment

Yes for the historical bug, at source level: the Android listener emits per-notification events and the gateway queues them as heartbeat event wakes. On current main, that same path no longer reproduces the runaway because repeated event wakes are deferred by the scheduler until the next due heartbeat.

## Solution Assessment

Yes for the reported runaway API consumption. The scheduler-level fix is narrower and more maintainable than an Android-only debounce because it covers notification, exec, spawn, and retry event wakes while preserving documented immediate wake paths.
