## Summary

Close as implemented: current main and shipped releases route Android notification heartbeat wakes through the typed event-wake cooldown path, so repeated notification events no longer produce independent agent runs every few seconds.

## What This Changes

The issue reports Android companion notification events starting separate heartbeat agent runs instead of batching to heartbeat cadence, causing provider quota exhaustion.

## Best Possible Solution

Keep the shipped scheduler-level event-wake cooldown; any stricter policy such as never waking on the first notification should be raised as a new narrower product decision.

## Reproduction Assessment

No high-confidence failing current-main reproduction remains. The original 2026.4.5 source path is explained, but current main routes notification wakes through event-intent cooldown and defers subsequent wakes until the scheduled heartbeat window.

## Solution Assessment

Yes. The shipped scheduler-level cooldown is the maintainable fix because it covers notification, exec, spawn, and retry event wakes while preserving documented manual and immediate wake-now contracts.
