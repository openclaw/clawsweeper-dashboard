## Label Justifications

- `P1`: The issue reports a real Android companion workflow that can drive repeated agent runs and exhaust provider quota for an otherwise scheduled heartbeat agent.
- `impact:session-state`: Accepted notification events currently advance agent session/run state outside the configured heartbeat cadence instead of staying queued for the scheduled heartbeat.

## Summary

Keep open. Current main and the latest shipped tag still turn each accepted Android `notifications.changed` event into a near-immediate heartbeat wake, while the only linked closing PR is an open fork PR rather than a merged OpenClaw fix.

## What This Changes

This issue asks OpenClaw to batch Android companion notification events into the next scheduled heartbeat instead of starting an agent run for each notification change.

## Best Possible Solution

Keep notification changes queued as context, defer ordinary Android notification updates until the next scheduled heartbeat, and define any explicit immediate-wake exception with gateway and Android policy coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass a valid `notifications.changed` payload into `handleNodeEvent`; current main enqueues it and calls `requestHeartbeat`, which defaults to a 250 ms wake delay.

## Solution Assessment

No for current main, and unclear for the linked fork PR. The narrow maintainable direction is to preserve queued notification context while removing ordinary immediate wakes, with any urgent exception made explicit and tested.
