## Summary

Keep open. Current main still turns each queued Android `notifications.changed` event into an immediate heartbeat wake, and the existing gateway tests explicitly assert that behavior; the linked open fork PR is an implementation candidate, but it is not merged and its stated `scopedHeartbeatWakeOptions` approach does not appear sufficient by itself under current helper semantics.

## What This Changes

Android companion notification events should be buffered as system context for the next scheduled heartbeat instead of triggering an immediate agent run per notification.

## Best Possible Solution

Keep this bug open until the gateway notification path preserves the queued system-event context but stops issuing immediate heartbeat wakes for ordinary Android notification changes, with tests updated to assert next-heartbeat behavior and any urgent wake behavior made explicit and configurable.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: pass a valid `notifications.changed` payload into `handleNodeEvent`; current tests and source show it enqueues the notification and calls `requestHeartbeatNow`, which schedules a near-immediate heartbeat wake.

## Solution Assessment

No for the current main behavior. The requested next-heartbeat direction is the narrow maintainable fix, but the linked PR body’s stated `scopedHeartbeatWakeOptions` change appears insufficient by itself because that helper only changes target scoping and still leaves an immediate `requestHeartbeatNow` call.
