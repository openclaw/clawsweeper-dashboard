## Summary

Keep open. Current main still leaves the OpenClaw-owned docs and diagnostic gap for intermittent `gog` Gmail auth failures in non-interactive macOS scheduled runs: the Gmail/gog docs cover setup and watcher startup, but not durable cron auth checks or keyring/client troubleshooting, and the watcher still only classifies address-in-use errors.

## What This Changes

The issue asks for guidance and diagnostics for intermittent `gog` Gmail OAuth failures during macOS cron or scheduled runs where manual commands later succeed.

## Best Possible Solution

Add focused non-interactive Gmail/gog auth troubleshooting to the bundled gog skill and public Gmail scheduling/webhook docs, and optionally add a small watcher/CLI diagnostic hint for `No auth for gmail`; keep OpenClaw credential storage defaults and global cron environment behavior unchanged unless a concrete runtime bug is proven.

## Reproduction Assessment

Unclear. The intermittent auth failure itself requires a macOS cron environment with real gog OAuth/keyring state, but the missing OpenClaw docs and diagnostic coverage are reproducible by static inspection of current main.

## Solution Assessment

Yes. The narrow docs plus optional error-classifier path is the most maintainable next step; changing token storage or broad scheduled-run environment handling would overreach without a confirmed OpenClaw runtime defect.
