## Summary

Keep open. Current main has post-connect Debug and Logs surfaces and improved single connection error text, but it still lacks the requested session-only pre-auth browser diagnostics buffer, handshake/close/error timeline, and copyable sanitized log drawer; the only linked implementation candidate is closed unmerged.

## What This Changes

Add a Control UI debug mode that captures WebSocket lifecycle events, handshake frames, close/error details, and copyable ephemeral diagnostics without persistent browser storage.

## Best Possible Solution

Keep this as the active tracker and implement a Control UI-owned ephemeral diagnostics surface integrated with GatewayBrowserClient and the disconnected/login UI. It should capture sanitized WebSocket open/error/close/retry events, connect.challenge/connect response metadata, structured connect errors, close codes/reasons, and optional post-auth logs.tail snippets when available, then expose them through a copyable drawer backed only by memory or session-scoped storage.

## Reproduction Assessment

Not applicable as a conventional bug reproduction because this is a feature request. Source review gives a high-confidence verification path: disconnected rendering gates the dashboard, connect.challenge is consumed internally, and Debug/Logs controllers require a connected client.

## Solution Assessment

Yes, the requested Control UI-owned diagnostics mode is the right direction if it starts from a strict redaction policy. Relying only on logs.tail would not solve the issue because logs.tail requires a successful authenticated connection.
