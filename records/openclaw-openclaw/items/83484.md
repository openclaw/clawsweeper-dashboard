## Summary

Keep open: current `main` still has a source-backed split where Telegram main-DM routing records the canonical main session while runtime-policy maintenance can derive a peer-keyed Telegram direct key, and the active-session CLI only lists persisted rows by `updatedAt`. This is not covered by the older stale-DM cleanup issue because that cleanup retires old rows rather than exposing a live collapsed runtime-policy identity.

## What This Changes

The issue reports that Telegram direct main-DM routing can split active session state between `agent:main:main` and a peer-keyed direct session, hiding live work from session/status surfaces.

## Best Possible Solution

Preserve the intended runtime-policy isolation, but make session metadata/status surfaces expose the collapsed peer-key relationship or active runtime-policy key so operators can see the live Telegram direct turn.

## Reproduction Assessment

No live high-confidence repro was run, but source inspection gives a medium-confidence reproduction path: main-scoped Telegram DMs store metadata under `agent:main:main` while runtime policy can derive `agent:main:telegram:default:direct:<peer>`. The report’s sanitized logs match that current source shape.

## Solution Assessment

Unclear until maintainers choose the canonical product behavior. The safest direction is not a Telegram-only shim, but a central session/status representation that keeps runtime-policy isolation while making the collapsed identity visible.
