## Label Justifications

- `P2`: This is a normal-priority session observability/state bug affecting Telegram operators during incident debugging without a proven delivery loss or crash.
- `impact:session-state`: The issue is about inconsistent active session identity between canonical main and peer-keyed Telegram direct runtime state.

## Summary

Keep open: the report matches current-main source behavior where a Telegram DM can have a canonical run key and a separate peer-keyed runtime-policy key, while `sessions --active` only lists persisted rows by key and `updatedAt` without exposing that collapsed relationship.

## What This Changes

The issue asks OpenClaw to make Telegram direct main-DM session identity consistent or visibly collapsed across runtime, LCM/context maintenance, `sessions --active`, and status surfaces.

## Best Possible Solution

Preserve the intentional runtime-policy isolation, but make the run key, policy key, LCM/context maintenance, and session/status rows share an explicit alias/collapse contract so operators can see the active Telegram direct turn.

## Reproduction Assessment

Yes, from source inspection plus the reporter's live logs: main-scoped Telegram direct routing can use `agent:main:main` while runtime policy deliberately derives `agent:main:telegram:default:direct:<peer>`, and `sessions --active` lists only persisted entries by `updatedAt`. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Unclear until an owner chooses the contract: the fix should not remove the peer-keyed policy identity blindly because tests and comments use it for sandbox/current-session behavior. The safer direction is an explicit collapse/alias representation for sessions/status plus regression coverage for Telegram main-DM routing.
