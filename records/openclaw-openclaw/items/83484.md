## Summary

Keep open: current main still has a source-backed split where default Telegram direct routing can persist the turn under `agent:main:main` while runtime-policy and Codex app-server context paths can operate under a peer-keyed direct identity. The related closed reports cover stale cleanup or tool visibility, not the live collapsed-identity/status observability gap reported here.

## What This Changes

The issue asks OpenClaw to make Telegram direct main-DM active-session identity consistent or visibly collapsed across runtime policy, LCM/context maintenance, `sessions --active`, and status surfaces.

## Best Possible Solution

Keep runtime-policy isolation for external direct chats, but add a central session/status representation that exposes the collapsed Telegram peer identity or its alias relationship to `agent:main:main`.

## Reproduction Assessment

No high-confidence live repro was run. Source inspection shows default Telegram DMs can persist under `agent:main:main` while runtime policy and Codex context paths use `agent:main:telegram:default:direct:<peer>`, matching the reporter's sanitized logs.

## Solution Assessment

Unclear until maintainers choose the canonical visibility contract. The safest repair is a central session/status alias model rather than a Telegram-only shim that hides the intended runtime-policy isolation.
