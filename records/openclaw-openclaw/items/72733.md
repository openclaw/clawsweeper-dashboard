## Summary

Keep this PR open, but do not merge it as-is. Current main does not implement this Security Guardrail surface, and the latest PR head still leaves model-visible prompt bypasses, regex fallback gaps, and reversible secret-storage lifecycle concerns.

## What This Changes

The PR adds an opt-in reversible Security Guardrail that scans user prompts with a local OpenAI-compatible model or regex fallback, replaces sensitive values with `[VAULT_N]` tokens, restores them for replies and tools, and adds config, UI metadata, docs, and tests.

## Best Possible Solution

Land a reviewed guardrail design that redacts every model-visible prompt surface, handles block-level secrets before cloud transit, defines a protected reversible-vault lifecycle, and aligns with the broader guardrail/plugin API discussion in #29132 and #72741.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enable the guardrail with a distinct `transcriptPrompt` containing a secret and the attempt submits that unchanged; pass a normal multi-line private key through regex-only scanning and the line-by-line scanner misses the full block.

## Solution Assessment

No. The current PR is not the best way to solve the issue yet because it adds security policy and durable secret storage directly in the runner while leaving prompt-boundary and vault-lifecycle gaps; a reviewed pre-model guardrail contract or hardened runner implementation is safer.
