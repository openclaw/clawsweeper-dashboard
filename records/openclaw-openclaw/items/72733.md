## Summary

Keep open. The direction is useful and the author fixed earlier log/counter/dead-output review items, but the latest branch still has source-level redaction bypasses, durable raw-secret storage concerns, missing real behavior proof, and unresolved security/API boundary questions.

## What This Changes

The PR adds an opt-in reversible sensitive-data guardrail that scans outbound user prompts with a local model or regex fallback, replaces secrets with `[VAULT_N]` tokens, restores values for tools and replies, and adds config/UI/docs/tests.

## Best Possible Solution

Land only after maintainers choose a reviewed pre-model guardrail boundary that redacts every model-visible prompt surface, handles block secrets, defines reversible vault storage and cleanup, and aligns config, docs, UI, and tests.

## Reproduction Assessment

Yes for the blocking review findings at source level: current main can submit a distinct `transcriptPrompt`, and the PR still forwards it unchanged while applying multiline secret regexes one line at a time. I did not run a live model/tool repro in this read-only review.

## Solution Assessment

No. The proposed direction is useful, but the current patch is not the best merge shape because it hard-codes security policy into the embedded runner while leaving model-visible prompt and secret-storage lifecycle gaps.
