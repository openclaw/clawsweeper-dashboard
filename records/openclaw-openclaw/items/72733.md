## Summary

Keep this PR open. The feature is useful and the author addressed the initial raw-log and vault-counter review comments, but the latest head still leaves model-visible prompt and regex fallback gaps in a security-sensitive redaction path, and the durable vault lifecycle needs maintainer/security design review before merge.

## What This Changes

The PR adds an opt-in reversible Security Guardrail that scans prompts with a local OpenAI-compatible model or regex fallback, replaces sensitive values with `[VAULT_N]` tokens before model transit, restores them for tool paths, and adds config, UI metadata, docs, and a vault test.

## Best Possible Solution

Land a reviewed guardrail design that redacts every model-visible prompt surface, handles block-level secrets before cloud transit, defines a protected reversible-vault lifecycle, and documents/tests the public config and API boundary.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enable the guardrail with a distinct `transcriptPrompt` containing a secret and the attempt submits that unchanged; pass a normal multi-line private key through regex-only scanning and the full-block regex never sees the complete block because scanning is line-by-line.

## Solution Assessment

No. The current PR is not the best way to solve the issue yet because it hard-codes security policy into the embedded runner while leaving prompt-boundary and vault-lifecycle gaps; a reviewed pre-model guardrail contract or hardened runner implementation would be safer.
