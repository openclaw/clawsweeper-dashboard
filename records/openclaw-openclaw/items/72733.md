## Summary

Keep this PR open. The latest head addresses some earlier bot feedback, but the proposed guardrail still has blocking security issues in prompt coverage, regex fallback behavior, and reversible secret persistence; current main also does not already implement this feature, so the PR is not obsolete.

## What This Changes

The PR adds an opt-in reversible Security Guardrail that scans user prompts with a local OpenAI-compatible model or regex fallback, replaces sensitive values with `[VAULT_N]` placeholders before model transit, restores them for assistant/tool paths, and adds config, UI metadata, docs, and a vault test.

## Best Possible Solution

Keep the PR open and require a hardened design before merge: redact every model-visible prompt surface including `transcriptPrompt` and runtime context, make regex fallback block-aware, define a reviewed lifecycle or protected storage model for reversible vault data, add focused tests and a changelog entry, and let maintainers decide whether the durable API should be a generic pre-model guardrail contract.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass a secret through a distinct `transcriptPrompt` path and the attempt submits that value unchanged, and pass a normal multi-line PEM/OpenSSH/PGP private key through `scanSensitive` in regex-only mode and the full-block regex cannot match because scanning is line-by-line.

## Solution Assessment

No. The current PR is not the best way to solve the issue yet because it adds a security policy directly to the embedded runner while leaving model-visible prompt paths and vault lifecycle gaps; a reviewed pre-model guardrail contract with a hardened implementation would be safer.
