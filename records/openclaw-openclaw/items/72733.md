## Summary

Keep open: current main still lacks the proposed Security Guardrail surface, and the PR head remains security-sensitive with source-reproducible redaction and vault-lifecycle gaps that should block merge.

## What This Changes

The PR adds an opt-in reversible sensitive-data guardrail with local-LLM/regex scanning, `[VAULT_N]` replacement and restoration, runner/tool integration, config schema/UI metadata, docs, and a vault test.

## Best Possible Solution

Land only after maintainers choose a reviewed pre-model guardrail boundary that redacts every model-visible prompt surface, handles block secrets, defines reversible vault storage/cleanup, and aligns config, docs, UI, and tests.

## Reproduction Assessment

Yes for the blocking review findings at source level: current main submits distinct `transcriptPrompt` text as the model-visible prompt, and the PR still forwards that field unchanged while scanning multiline private-key patterns one line at a time. I did not run a live model/tool repro in this read-only review.

## Solution Assessment

No. The proposed direction is useful, but the current patch is not the best merge shape because it hard-codes security policy into the embedded runner while leaving model-visible prompt and secret-storage lifecycle gaps.
