## Summary

Keep open: current main does not implement this guardrail, and the PR head still has security-sensitive redaction gaps on model-visible prompt surfaces, regex fallback coverage, and reversible secret storage. The related guardrail/output-sanitization reports show the product need is real, but this needs maintainer/secops design review before merge.

## What This Changes

The PR adds an opt-in Security Guardrail that redacts prompts with local-LLM or regex detection, persists reversible vault tokens, restores tool/display paths, and adds config, UI metadata, docs, and a vault test.

## Best Possible Solution

Land a maintainer-reviewed guardrail design that redacts every model-visible prompt surface, handles block-level secrets before model transit, defines a protected reversible-vault lifecycle, and aligns with the plugin/security hook boundary tracked by #72741.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enable the guardrail with a distinct `transcriptPrompt` containing a secret and the attempt submits that unchanged; pass a normal multi-line private key through regex-only scanning and the full-block regex never sees the complete block because scanning is line-by-line.

## Solution Assessment

No. The current PR is not the best solution yet because it hard-codes sensitive prompt and vault behavior into the embedded runner while leaving prompt-boundary and secret-lifecycle gaps; a reviewed pre-model guardrail contract or hardened runner implementation would be safer.
