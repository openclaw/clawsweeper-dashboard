## Summary

Keep this PR open. Current main does not implement the proposed guardrail, and the PR still has merge-blocking security/correctness issues: the regex fallback still misses normal multi-line private keys, and the reversible vault persists raw secrets as a durable session sidecar without the reviewed lifecycle used by existing secret-scrub paths.

## What This Changes

The PR adds an opt-in reversible sensitive-data guardrail that scans prompts via a local OpenAI-compatible model or regex fallback, replaces secrets with `[VAULT_N]` tokens before model transit, restores them for assistant/tool paths, and adds config, UI metadata, docs, and a vault test.

## Best Possible Solution

Keep the PR open for maintainer/security review, then either harden it into a reviewed pre-model guardrail contract with protected or ephemeral vault storage and block-aware scanning, or split the implementation so core exposes a narrow contract and policy while provider-specific detection can live outside the runner.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass a normal multi-line PEM/OpenSSH/PGP private key through `scanSensitive`; because the latest PR head still splits the prompt by newline before applying the full-block regexes, regex-only mode cannot emit a complete private-key finding.

## Solution Assessment

No. The current PR is not the best mergeable solution yet because it embeds broad security policy and durable reversible secret storage directly into the embedded runner while leaving a concrete fallback bypass and an unreviewed raw-secret lifecycle.

Full review comments:

- **[P1] Scan full private-key blocks before splitting lines:** `src/security/patterns.ts:97-109`
  - body: The fallback scanner splits the prompt by newline and then applies every regex to each individual line. The new PEM/OpenSSH/PGP patterns require both BEGIN and END markers, so a normal multi-line private key is not detected and can still be sent to the cloud model in regex-only mode. Run block patterns against the full text before line-level patterns, or add equivalent block-aware scanning.
  - confidence: 0.94

File mode `0600` does not address retention after session end, backups, support bundles, or the existing SecretRef/audit/scrub lifecycle, so enabling the guardrail creates durable plaintext secret residue. Keep the vault ephemeral or integrate it with protected storage, cleanup, and audit handling before persisting it.
  - confidence: 0.86
