## Summary

Keep open: the PR is useful security hardening work, but the current head still has source-reproducible disclosure and secret-lifecycle blockers before it can merge.

## What This Changes

The PR adds an opt-in Security Guardrail that redacts prompt secrets into `[VAULT_N]` tokens, restores them for assistant/tool paths, and adds config, UI metadata, docs, and vault/pattern coverage.

## Best Possible Solution

Land a maintainer-approved pre-model guardrail boundary that redacts every model-visible prompt surface, handles block secrets before cloud transit, uses a reviewed reversible-vault lifecycle, and ships aligned docs, config, changelog, and tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable the guardrail with a distinct `transcriptPrompt` containing a secret, or use regex-only scanning on a normal multiline private-key block, and those values can remain model-visible.

## Solution Assessment

No. The current PR is not the best solution yet because it hard-codes broad security policy into the embedded runner while leaving prompt-surface and vault-lifecycle gaps; a reviewed pre-model guardrail contract or hardened runner implementation would be safer.
