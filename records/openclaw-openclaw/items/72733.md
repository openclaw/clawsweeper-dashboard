## Summary

Keep open. The earlier Greptile findings for raw secret logging, vault counter collisions, and dead `sanitizedPrompt` output are addressed in the latest head, but this security-sensitive PR still has blocking gaps around model-visible prompt redaction, regex-only private-key coverage, and durable plaintext vault storage.

## What This Changes

The PR branch adds an opt-in Security Guardrail that redacts prompt secrets into `[VAULT_N]` tokens, restores them for responses and tool parameters, and adds config, UI metadata, docs, and a vault test.

## Best Possible Solution

Land a maintainer-approved pre-model guardrail boundary that redacts every model-visible prompt surface, handles block secrets before cloud transit, uses a reviewed reversible-vault lifecycle, and aligns docs, config, changelog, and tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable the guardrail with a distinct `transcriptPrompt` containing a secret, or run regex-only scanning on a normal multi-line private key block, and those values can remain model-visible.

## Solution Assessment

No. The current PR is not the best solution yet because it hard-codes security policy into the embedded runner while leaving prompt-surface and vault-lifecycle gaps; a reviewed pre-model guardrail contract or hardened runner implementation would be safer.

  - body: `finalPrompt` is redacted, but this call still forwards `params.transcriptPrompt` unchanged. Current runner logic uses `transcriptPrompt` as `promptSubmission.prompt` when it differs from `effectivePrompt`, so a secret there can bypass the guardrail and reach the model.

  - confidence: 0.94

  - body: This writes the reversible token-to-secret map as plaintext JSON next to the session file. `0600` limits local users, but raw credentials still persist outside a reviewed SecretRef or cleanup lifecycle, which is too durable for a secret guardrail.
  - confidence: 0.82
