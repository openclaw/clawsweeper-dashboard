## Summary

Keep open. Current main does not already implement this guardrail, and the latest PR head still has security-sensitive merge blockers in the prompt boundary, regex fallback, and reversible-vault storage path.

## What This Changes

The PR adds an opt-in reversible Security Guardrail that redacts sensitive prompt values into `[VAULT_N]` tokens before cloud model transit, restores them for responses and tools, and adds config, UI metadata, docs, and a vault test.

## Best Possible Solution

Land a maintainer-approved guardrail design that redacts every model-visible prompt surface, handles block-level secrets before cloud transit, defines a protected reversible-vault lifecycle, and documents/tests the public config and API boundary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: enable the guardrail with a distinct `transcriptPrompt` containing a secret and the attempt can submit that unchanged; pass a normal multi-line private key through regex-only scanning and the full-block regex is only applied per line.

## Solution Assessment

No. The current PR is not the best way to solve this yet because it hard-codes security policy into the embedded runner while leaving model-visible prompt and vault-lifecycle gaps; a reviewed pre-model guardrail contract or hardened runner implementation would be safer.

  - body: `finalPrompt` is redacted, but the original `params.transcriptPrompt` is still passed into the attempt. When runtime context is split out, that transcript prompt becomes the actual model-visible prompt, so a user secret in that field bypasses the guardrail.

  - body: This writes the reversible token-to-secret map as plaintext JSON beside the session. `0600` limits cross-user reads, but the raw secrets still persist beyond the turn and outside a reviewed cleanup or SecretRef lifecycle.
