## Label Justifications

- `P1`: The reported failure can lose subagent completion visibility and stall agent workflows for real users.
- `impact:session-state`: The issue concerns requester/subagent state reconciliation after completion delivery failures.
- `impact:message-loss`: The reported symptom is a completed subagent result not reaching the requester-visible message path.

## Summary

Current main and the latest release already preserve successful subagent completion payloads after announce retry exhaustion and feed pending or suspended completions into the requester’s next turn, so the central lost-announcement problem is implemented.

## What This Changes

The issue requests durable persistence of subagent completion announcements so requester sessions can consume them later instead of relying on direct in-process dispatch.

## Best Possible Solution

Keep the existing registry-backed pending/suspended delivery plus requester-turn steering path; narrower wake/deadlock behavior can stay with the related open issues rather than reopening this duplicate design request.

## Reproduction Assessment

Yes at source level, but I did not run a live failing current-main repro. The issue logs match explicit retry-limit/no-visible-payload paths, and current tests now cover those paths preserving suspended delivery payloads.

## Solution Assessment

Yes. The current solution uses the existing subagent registry SQLite state and requester-turn steering queue instead of adding a separate system-event queue, which is a narrower fit for OpenClaw’s state-storage policy.
