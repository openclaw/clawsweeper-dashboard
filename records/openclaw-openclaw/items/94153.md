## Label Justifications

- `P1`: The issue concerns completed subagent work failing to reach requester sessions, which can break real agent workflows.
- `impact:session-state`: The remaining design is about durable requester/session completion state across busy sessions and restarts.
- `impact:message-loss`: The user-visible failure is lost or delayed subagent completion announcements.

## Summary

Keep open: current main fixed the narrow non-terminal direct-handoff loss, but the requested durable/persistent completion queue is still not implemented and remains a real core delivery-design question.

## What This Changes

Request to persist subagent completion announcements so the requester can consume them on a later turn instead of relying only on direct session dispatch.

## Best Possible Solution

Keep the current narrow direct-pending fix, then make a maintainer decision on a registry-backed or SQLite-backed durable completion delivery design that coordinates with https://github.com/openclaw/openclaw/issues/67777 and https://github.com/openclaw/openclaw/issues/86538.

## Reproduction Assessment

Partial. The old completion_handoff_pending give-up sequence is fixed on current main, but source and docs still show no persistent completion queue: system events are in-memory and subagent docs still call announce-back best-effort across restarts.

## Solution Assessment

Unclear as an implementation. The merged narrow fix is best for the closed pending-handoff bug, but the requested durable queue/spool needs maintainer choice between registry-backed persistence, SQLite queueing, or keeping best-effort semantics.
