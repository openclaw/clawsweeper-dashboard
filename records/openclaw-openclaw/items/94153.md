## Label Justifications

- `P1`: The reported failure could permanently hide completed subagent output from the requester and stall real agent workflows.
- `impact:session-state`: The issue concerns requester/subagent lifecycle state becoming unreconciled after completion announce failure.

## Summary

Close: current main now preserves subagent completion payloads through retry-limit/expiry and feeds pending or suspended completions into the requester’s next turn, so the reported direct-dispatch loss is no longer the current behavior.

## What This Changes

The issue asks OpenClaw to persist subagent completion announcements for requester-side consumption instead of relying only on direct in-process dispatch to a busy requester session.

## Best Possible Solution

Keep the current durable subagent delivery state plus requester steering queue path; broader yielded-parent lifecycle and accounting follow-ups can remain on their narrower open items.

## Reproduction Assessment

Yes. Source inspection shows the old failure family through direct handoff pending, no visible completion payload, or message-tool-required failure, and current tests cover preserving and steering the retained payload instead of losing it.

## Solution Assessment

Yes. The current implementation uses the existing subagent registry delivery state and requester steering queue, which is narrower and more durable than adding a separate transient system-event queue.
