## Summary

Close as a duplicate: the durable subagent completion queue/inbox request is already tracked by the older canonical issue at https://github.com/openclaw/openclaw/issues/67777, which remains open for the broader restart, drain, and orphan-prune completion-loss design.

## What This Changes

The issue requests persisting subagent completion announcements so a busy requester session can consume them later instead of relying on direct in-process dispatch.

## Best Possible Solution

Track the durable completion spool/inbox, requester-turn replay, and orphan-preservation policy on https://github.com/openclaw/openclaw/issues/67777 while keeping the broader write-lock symptom linked through https://github.com/openclaw/openclaw/issues/86538.

## Reproduction Assessment

Yes, at source level: current main can persist pending or suspended completion payloads, then resume can prune an orphaned ended run before announce replay. I did not run a live channel reproduction in this read-only cleanup review.

## Solution Assessment

No, a separate issue is not the best way to solve it. The maintainable path is to keep the durable spool/inbox and orphan-preservation policy on the older canonical issue, which already covers this failure mode and remains open.
