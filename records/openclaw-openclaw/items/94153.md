## Summary

Current main already fixes the observable pending direct-announce loss described here: merged PR https://github.com/openclaw/openclaw/pull/94349 removed the branch that treated a non-terminal completion handoff as failed, and added regression coverage for accepted plain completion delivery. Broader durable spool/restart/orphan-prune work remains tracked in https://github.com/openclaw/openclaw/issues/67777, so this newer duplicate symptom report can close against the landed narrow fix.

## What This Changes

The issue requests durable queueing of completed subagent announcements so a busy requester session does not lose the completion after direct announce timeout.

## Best Possible Solution

Keep the merged pending-handoff behavior from https://github.com/openclaw/openclaw/pull/94349 and route any remaining durable completion-spool semantics to https://github.com/openclaw/openclaw/issues/67777.

## Reproduction Assessment

Yes for the historical/release path: v2026.6.8 still has the `completion_handoff_pending` branch, while current main no longer reproduces that branch because pending direct completion handoffs are credited as delivered.

## Solution Assessment

Yes for the narrow issue: treating accepted or in-flight gateway handoffs as delivered is the smallest landed fix for the reported pending-handoff loss, while the larger durable-spool design remains a separate canonical discussion.
