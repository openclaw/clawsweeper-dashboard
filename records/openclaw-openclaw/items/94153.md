## Summary

Keep open: current main has a durable requester-turn steering queue and #94349 fixed the non-terminal direct-pending subset, but current source and the June 22 reports on 2026.6.9 still show visible-reply/message-tool-only give-up paths that can end without requester wake.

## What This Changes

The issue asks OpenClaw to persist subagent completion announcements for next-turn requester consumption instead of relying only on direct in-process dispatch.

## Best Possible Solution

Keep the SQLite-backed delivery payload and steering queue, then land the missing requester-visible blocked/wake behavior for retry-limit and expiry give-ups with group/message-tool-only regression coverage.

## Reproduction Assessment

Yes at source level: current main still has visible/message-tool delivery failures and retry-limit/expiry give-up paths matching the latest 2026.6.9 reports; I did not run a live timing-sensitive reproduction.

## Solution Assessment

Partly: the existing durable steering queue is the right persistence shape, but the issue is not fully solved until retry-limit and expiry give-ups surface a bounded requester-visible completion or failure event.
