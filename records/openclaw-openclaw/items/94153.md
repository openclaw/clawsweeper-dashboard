## Label Justifications

- `P1`: The issue describes completed subagent output being lost or leaving requester workflows stuck in real multi-subagent usage.
- `impact:session-state`: The failure is about requester/subagent completion state not being durably reconciled after handoff failure.
- `impact:message-loss`: A child can finish with output that never becomes visible to the requester.

## Summary

Keep open: current main and v2026.6.9 contain a narrow pending-handoff fix, but they do not implement the requested durable subagent completion queue; the existing system-event queue is explicitly ephemeral, and related open PRs cover narrower wake/fallback slices rather than this storage contract.

## What This Changes

The issue asks for subagent completion announcements to be persisted for later requester-session consumption instead of relying only on direct in-process dispatch.

## Best Possible Solution

Define a maintainer-approved durable completion outbox/inbox using canonical OpenClaw state storage, land or close the narrower related PRs on their own merits, and keep this issue as the tracker for persisted requester-side reconciliation.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 still have no durable queue path, the system-event queue is explicitly ephemeral, and no-visible/message-tool-only completion failures can still become failed direct deliveries. I did not run a live timing reproduction.

## Solution Assessment

Unclear as an implementation shape: persistence is the right class of fix, but the safest solution should use a maintainer-approved durable state owner instead of the current in-memory system-event queue.
