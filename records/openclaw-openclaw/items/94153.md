## Label Justifications

- `P1`: The report describes completed subagent output being lost in an active agent workflow.
- `impact:message-loss`: The reported failure drops subagent completion announcements after retries.
- `impact:session-state`: The implemented path persists completion handoff state and injects it into requester session turns.

## Summary

Current main implements the requested durable subagent-completion handoff well enough: completed subagent payloads are preserved in the SQLite-backed subagent registry, leased into the requester agent's next turn through the steering queue, and a still-pending completion handoff is no longer treated as a failed announce that triggers retry-limit loss.

## What This Changes

The issue requests persisting subagent completion announcements into requester-session state or a queue instead of relying on direct requester-session dispatch.

## Best Possible Solution

Keep the current-main persisted steering-queue implementation for this narrow completion handoff, and continue the broader session write-lock contention work in https://github.com/openclaw/openclaw/issues/86538.

## Reproduction Assessment

Yes. The issue logs match the old source path where pending completion handoff could be classified as `completion_handoff_pending`, and current main now avoids that failure while preserving queued payloads for requester-turn injection.

## Solution Assessment

Yes. The implemented path uses the existing subagent registry and requester-turn steering queue instead of adding a new config surface or separate queue system.
