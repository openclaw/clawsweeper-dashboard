## Summary

Keep open. Current main still treats a user-typed bare image URL in Control UI chat as text, and the existing remote-image policy makes the requested client-only regex-to-img approach unsafe without a deliberate managed-media path.

## What This Changes

The issue requests automatic inline rendering of user-sent bare image URLs in the chat window instead of showing them as plain text links.

## Best Possible Solution

Add an explicit Control UI/Gateway media flow for eligible bare image URLs: validate policy, ingest or proxy through the existing managed-image path, render same-origin managed image blocks inline, and preserve text/link fallback when validation or policy blocks the URL.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit `https://example.com/image.png` with no attachments; the UI creates a text block, the Gateway skips attachment parsing, and the renderer never receives a structured image block.

## Solution Assessment

No. The issue’s suggested client-side regex-to-img conversion is not the best solution because current CSP and docs intentionally prevent arbitrary remote browser image fetches; a Gateway-managed or policy-controlled media path is the safer design.

## Security Review

Status: needs_attention

Summary: The request would affect untrusted remote media loading and needs explicit policy before implementation.

Concerns:

- **[medium] Avoid arbitrary browser image fetches:** `docs/web/control-ui.md:348`
  - body: Rendering typed remote URLs directly as img tags would let chat text trigger requests from the operator browser, conflicting with the documented Control UI remote-image policy.
  - confidence: 0.94
- **[medium] Do not widen CSP without a media policy:** `src/gateway/control-ui-csp.ts:47`
  - body: The current Control UI CSP omits remote http(s) image sources; a client-only renderer change would either produce broken images or require broadening img-src without reviewed privacy controls.
  - confidence: 0.93
- **[medium] Reuse SSRF-guarded media ingestion:** `src/media/store.ts:239`
  - body: Any Gateway-side remote fetch should reuse the existing pinned-hostname, redirect, MIME, and byte-limit style protections rather than adding an ad hoc downloader for chat text.
  - confidence: 0.88
