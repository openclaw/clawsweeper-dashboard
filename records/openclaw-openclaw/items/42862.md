## Summary

Keep open. Current main still leaves a user-typed bare image URL in Control UI chat as text/link content, and the existing no-remote-fetch image policy means the suggested regex-to-img approach is not safe without maintainer policy direction.

## What This Changes

The issue requests automatic inline rendering of bare image URLs in the chat window instead of leaving them as plain text links.

## Best Possible Solution

Implement a deliberate Control UI/Gateway media path for eligible bare image URLs: validate policy, ingest or proxy through managed media, render same-origin image blocks inline, and preserve text/link fallback when validation blocks the URL.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit `https://example.com/image.png` with no attachments; current UI and Gateway code keep it as text and never produce a structured image block.

## Solution Assessment

No for the issue's suggested client-side regex-to-img approach. The maintainable solution needs a Gateway-managed or explicitly policy-controlled media path because current CSP and docs intentionally prevent arbitrary remote browser image fetches.

## Security Review

Status: needs_attention

Summary: The request would change untrusted remote media loading behavior in chat and needs explicit policy before implementation.

Concerns:

- **[medium] Avoid arbitrary browser image fetches:** `docs/web/control-ui.md:365`
  - body: Rendering typed remote URLs directly as img tags would let chat text trigger requests from the operator browser, conflicting with the documented no-remote-fetch policy and enabling tracking or read-timing leakage.
  - confidence: 0.94
- **[medium] Align renderer changes with CSP:** `src/gateway/control-ui-csp.ts:47`
  - body: The current Control UI CSP omits remote http(s) image sources, so a client-only renderer change would either produce broken images or require broadening img-src without reviewed privacy controls.
  - confidence: 0.93
- **[medium] Reuse SSRF-guarded media ingestion:** `src/media/store.ts:239`
  - body: Any Gateway-side remote fetch should reuse the existing pinned-hostname, redirect, MIME, and byte-limit media path instead of adding an ad hoc downloader for chat text.
  - confidence: 0.88
