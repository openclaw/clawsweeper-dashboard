## Summary

Keep open. Current main still treats a user-typed bare image URL as chat text, while existing Control UI policy intentionally blocks direct remote image loading, so this needs maintainer product/security direction rather than cleanup closure.

## What This Changes

Request auto-rendering image URLs typed in the chat window as inline images instead of plain text links.

## Best Possible Solution

Add a deliberate Control UI/Gateway media path for eligible bare image URLs: validate policy, ingest or proxy through managed media, render same-origin image blocks inline, and preserve text/link fallback when validation blocks the URL.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send `https://example.com/image.png` with no attachments; current main keeps it as a text block and never creates a structured image block.

## Solution Assessment

No for the suggested regex-to-img approach. The maintainable solution needs a policy-controlled Gateway/managed-media path or an explicit decision to keep remote typed URLs as links/text.

## Security Review

Status: needs_attention

Summary: The feature would change untrusted remote media loading behavior in Control UI chat and needs explicit policy before implementation.

Concerns:

- **[medium] Avoid arbitrary browser image fetches:** `docs/web/control-ui.md:365`
  - body: Rendering typed remote URLs directly as img tags would let chat text trigger operator-browser requests if CSP were loosened, conflicting with the documented no-remote-fetch policy.
  - confidence: 0.94
- **[medium] Keep img-src aligned with media policy:** `src/gateway/control-ui-csp.ts:47`
  - body: The current Control UI CSP omits remote http(s) image sources, so a client-only renderer change would produce broken images or require a reviewed CSP policy change.
  - confidence: 0.93
- **[medium] Use managed media ingestion for remote fetches:** `src/media/store.ts:419`
  - body: If Gateway fetches typed image URLs, it should reuse existing guarded media ingestion with redirect, MIME, and byte-limit handling instead of adding ad hoc chat URL downloading.
  - confidence: 0.86
