## Summary

Keep open. The user-facing problem is still worth solving, but this PR is stale against current main and the proposed patch has blocking correctness issues around stderr-backed exits and session-init recovery.

## What This Changes

The PR reclassifies silent non-zero `acpx` turn exits as `ACP_BACKEND_UNAVAILABLE` and adds regression coverage for runtime, manager, and error-text handling.

## Best Possible Solution

Move the classification to the current ACPX runtime boundary so only truly silent backend process exits become `ACP_BACKEND_UNAVAILABLE`, while preserving stderr-backed turn/control failures, permission-denied guidance, and session-init recovery.

## Reproduction Assessment

Yes, source inspection of the PR head gives a high-confidence reproduction path: a non-zero `acpx` prompt or control exit with stderr but no structured JSON error is classified as `ACP_BACKEND_UNAVAILABLE`. I did not execute it because the review is read-only and local test tooling is unavailable.

## Solution Assessment

No. The current PR is not the best fix because it treats missing structured ACP errors as silence, ignores stderr, and skips `recoverEnsureFailure`; current main likely needs a rebased fix in the `acpx` dependency or a narrow adapter-layer replacement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate prompt exits on empty stderr:** `extensions/acpx/src/runtime.ts:679`
  - body: This classifies every non-zero prompt exit without a structured ACP `error` event as `ACP_BACKEND_UNAVAILABLE`, even when stderr contains a concrete agent or validation failure. That routes stderr-backed turn failures to `/acp doctor`; only empty-stderr exits should get the backend-unavailable code.
  - confidence: 0.9
- **[P2] Preserve stderr-backed control failures:** `extensions/acpx/src/runtime.ts:1041-1044`
  - body: `runControlCommand` applies the backend-unavailable fallback to every non-zero control-command exit after JSON parsing. Control paths such as status, set-mode, cancel, or close can fail with stderr and no JSON error, so this can mislabel ordinary control failures as backend outages.
  - confidence: 0.88
- **[P2] Keep ensure-session recovery active:** `extensions/acpx/src/runtime.ts:483-484`
  - body: This early rethrow bypasses `recoverEnsureFailure` for silent `sessions ensure` exits, so the status probe and fallback reuse/recreate flow no longer run for a class of transient session-init failures. Preserving `ACP_SESSION_INIT_FAILED` should not remove the existing recovery attempt.
  - confidence: 0.82
