## Summary

Keep open: current main does not implement the requested silent-exit reclassification, and the PR branch is stale against the ACPX package-wrapper boundary while still introducing source-level classification and recovery regressions.

## What This Changes

The PR changes ACPX/ACP error classification so selected non-zero `acpx` process exits surface as `ACP_BACKEND_UNAVAILABLE`, with runtime, manager, fixture, and error-text test updates.

## Best Possible Solution

Move the exit classification to the ACPX runtime package so only genuinely silent prompt/backend crashes become `ACP_BACKEND_UNAVAILABLE`, then consume the published or explicitly pinned ACPX change in OpenClaw with focused wrapper, manager, and error-text coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR findings: the diff deterministically classifies by exit code/fallback without checking stderr/silence, and current-main tests cover the existing turn and session-init error-code behavior. I did not execute tests because this was a read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. The proposed branch edits stale in-tree ACPX runtime logic; the safer path is an ACPX package-level fix followed by a deliberate OpenClaw package update and narrow compatibility tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard backend-unavailable code behind truly silent prompt exits:** `extensions/acpx/src/runtime.ts:676`
  - body: `runTurn` now tags any non-zero prompt exit without a structured error event as `ACP_BACKEND_UNAVAILABLE`, but this path can also include concrete stderr output. Stderr-backed prompt/auth/model/validation failures should keep turn-failure recovery instead of routing users to `/acp doctor`.
  - confidence: 0.86
- **[P2] Preserve turn-failure codes for non-silent control exits:** `extensions/acpx/src/runtime.ts:1038-1044`
  - body: `resolveSilentAcpxControlExitErrorCode` only receives the exit code and fallback code, so every non-permission-denied control-command exit with an `ACP_TURN_FAILED` fallback becomes `ACP_BACKEND_UNAVAILABLE` even when the command wrote stderr. Status, cancel, close, mode, or config failures can therefore be mislabeled as backend outages.
  - confidence: 0.84
- **[P2] Keep ensure-session recovery path for silent ensure exits:** `extensions/acpx/src/runtime.ts:480-482`
  - body: The early rethrow skips `recoverEnsureFailure` for silent `sessions ensure` exits, so OpenClaw no longer runs the status probe and fallback reuse/recreate flow that can recover transient session-init failures.
  - confidence: 0.83
