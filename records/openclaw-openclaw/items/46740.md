## Summary

This PR should close as superseded: current main replaced the old OpenClaw-owned ACPX subprocess runtime with the shared `acpx/runtime` consumer path, and the branch now conflicts with that architecture while still carrying unsafe over-classification behavior.

## What This Changes

The branch changes ACPX runtime and manager error handling so silent non-zero acpx exits map to `ACP_BACKEND_UNAVAILABLE`, with runtime, manager, and error-text regression tests.

## Best Possible Solution

Keep the current `acpx/runtime` consumer and early-exit recovery path; any remaining backend-unavailable taxonomy work should be redesigned against current ACPX runtime result/error metadata in a fresh PR.

## Reproduction Assessment

Current main no longer has the old subprocess path, so the original branch target is not reproducible there. The PR's remaining regression is source-reproducible from the branch diff: stderr-backed prompt/control failures still get routed to `ACP_BACKEND_UNAVAILABLE`.

## Solution Assessment

No. The branch is no longer the best way to solve the issue because the affected runtime was replaced by `acpx/runtime`, and the remaining diff would over-classify non-silent failures instead of using current runtime error metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require empty stderr before backend-unavailable:** `extensions/acpx/src/runtime.ts:679`
  - body: `runTurn` reaches this branch whenever acpx exits nonzero without a structured error, but `stderr` can still contain a concrete harness or input failure. Setting `ACP_BACKEND_UNAVAILABLE` here sends those non-silent failures to `/acp doctor`; gate this on empty stderr or keep `ACP_TURN_FAILED` for stderr-backed exits.
  - confidence: 0.9
- **[P2] Do not up-classify non-silent control exits:** `extensions/acpx/src/runtime.ts:129-131`
  - body: `resolveSilentAcpxControlExitErrorCode` only receives the exit code and fallback, so every non-permission control command with fallback `ACP_TURN_FAILED` becomes backend unavailable even when acpx returned stderr. Control/input failures from mode, config, status, cancel, or close should preserve the fallback unless the exit was truly silent.
  - confidence: 0.88
