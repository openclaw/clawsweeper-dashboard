## Summary

Keep open: the PR addresses a plausible ACP/acpx error-classification bug, but the current patch is still incorrect against current main, lacks after-fix real behavior proof, and targets logic that now belongs at the `acpx/runtime` boundary.

## What This Changes

The PR changes acpx runtime exit classification, ACP manager fallback behavior, fixtures, and tests so some `acpx exited with code 1` paths report `ACP_BACKEND_UNAVAILABLE`.

## Best Possible Solution

Move the silent backend-exit classification into the acpx runtime contract, consume the appropriate pinned acpx release from OpenClaw, and keep any manager fallback narrow enough to preserve stderr/control/init recovery behavior.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection of acpx 0.7.0 and current manager tests shows the relevant current-main exit/error path still normalizes to `ACP_TURN_FAILED`.

## Solution Assessment

No. The current PR tries to fix the bug inside a stale OpenClaw wrapper and broad manager fallback, but current main has moved the reusable runtime boundary into `acpx/runtime`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Guard backend-unavailable behind truly silent prompt exits:** `extensions/acpx/src/runtime.ts:679-680`
  - body: This upgrades any non-zero prompt exit without a structured ACP error event to `ACP_BACKEND_UNAVAILABLE`, even when stderr contains a concrete adapter or validation failure. That routes normal turn failures to `/acp doctor`; only genuinely silent exits should get the backend-unavailable code.
  - confidence: 0.89
- **[P2] Preserve control fallback for stderr failures:** `extensions/acpx/src/runtime.ts:1041-1044`
  - body: `runControlCommand` now maps every non-permission non-zero control exit with fallback `ACP_TURN_FAILED` to backend-unavailable. Control commands can fail with plain stderr for input or adapter reasons, so this should keep `params.fallbackCode` unless the result is actually silent.
  - confidence: 0.87
- **[P2] Keep ensure-session recovery after silent ensure exits:** `extensions/acpx/src/runtime.ts:483-484`
  - body: The new early rethrow prevents `recoverEnsureFailure` from running for silent `sessions ensure` exits. That drops the existing status probe and reuse/recreate recovery path for transient ensure failures that could still be recoverable.
  - confidence: 0.84
