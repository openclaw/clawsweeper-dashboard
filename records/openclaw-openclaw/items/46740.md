## What This Changes

The PR changes ACPX runtime, ACP manager, fixtures, and tests so selected `acpx exited with code 1` failures report `ACP_BACKEND_UNAVAILABLE` and `/acp doctor` guidance.

## Best Possible Solution

Rebase onto the current `acpx@0.8.0` runtime surface and classify only genuinely silent pre-output backend exits while preserving stderr-backed failures, control-command fallback codes, and ensure/retry/failover recovery.

## Reproduction Assessment

## Solution Assessment

Full review comments:

- **[P2] Gate backend-unavailable on empty stderr:** `extensions/acpx/src/runtime.ts:676`
  - body: `runTurn` now tags any non-zero prompt exit without a structured ACP error as `ACP_BACKEND_UNAVAILABLE`, but that path can still include concrete stderr failure text. Those user/config/runtime failures would be routed to `/acp doctor`; only truly silent exits should be upgraded.
  - confidence: 0.84
- **[P2] Preserve control errors when stderr is present:** `extensions/acpx/src/runtime.ts:129-131`
  - body: `resolveSilentAcpxControlExitErrorCode` remaps every non-permission control exit from `ACP_TURN_FAILED` to `ACP_BACKEND_UNAVAILABLE` without knowing whether the failure was silent. Control paths like setMode, setConfigOption, status, cancel, and close can legitimately fail without meaning the backend is unavailable.
  - confidence: 0.82
- **[P2] Keep ensure recovery for silent init exits:** `extensions/acpx/src/runtime.ts:480-482`
  - body: This early rethrow skips `recoverEnsureFailure` for silent `sessions ensure` exits, so the status probe and fallback reuse/recreate flow no longer run after a transient ensure failure. That weakens the session-init recovery behavior the manager currently depends on.
  - confidence: 0.8
