## Summary

Keep open: this remains a plausible ACPX error-classification bug, but the PR is not merge-ready because it misclassifies non-silent/control failures, skips existing ensure recovery, lacks real behavior proof, and now conflicts with current main’s newer `acpx/runtime` boundary.

## What This Changes

The PR changes ACPX runtime and ACP manager fallback classification so selected nonzero `acpx exited with code 1` failures surface as `ACP_BACKEND_UNAVAILABLE`, with regression tests for runtime, manager, and error text.

## Best Possible Solution

Move silent backend-exit classification into the published `acpx` runtime contract, consume that release in the ACPX plugin, and preserve stderr, control-command, and ensure-recovery semantics with focused tests plus a real ACPX smoke proof.

## Reproduction Assessment

No live reproduction was run, but the source path is clear: current `acpx@0.7.0` can emit `RUNTIME`, and current OpenClaw normalizes unknown runtime codes to `ACP_TURN_FAILED`. Current tests also still expect a plain `acpx exited with code 1` turn error to surface as `ACP_TURN_FAILED`.

## Solution Assessment

No. This PR patches stale wrapper and manager fallback paths; the narrower maintainable fix is an `acpx` runtime-level classification contract consumed by OpenClaw, while preserving stderr, control-command, and session-recovery behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Guard backend-unavailable behind truly silent prompt exits:** `extensions/acpx/src/runtime.ts:679`
  - body: This tags any nonzero prompt exit without a structured error event as `ACP_BACKEND_UNAVAILABLE`, even when stderr contains a concrete adapter or validation failure. That routes ordinary turn failures to `/acp doctor`; only exits with no structured error and no stderr should get backend-unavailable guidance.
  - confidence: 0.89
- **[P2] Preserve turn-failure codes for non-silent control exits:** `extensions/acpx/src/runtime.ts:129-131`
  - body: `resolveSilentAcpxControlExitErrorCode` remaps every non-permission nonzero control exit from `ACP_TURN_FAILED` to backend-unavailable without checking whether the result was actually silent. Control commands can fail with plain stderr for input or adapter reasons, so this should keep the fallback unless the control result is truly silent.
  - confidence: 0.87
- **[P2] Keep ensure-session recovery after silent ensure exits:** `extensions/acpx/src/runtime.ts:480-482`
  - body: The early rethrow prevents `recoverEnsureFailure` from running for silent `sessions ensure` exits. That drops the existing status probe and reuse/recreate recovery path for transient ensure failures that could still be recoverable.
  - confidence: 0.84
