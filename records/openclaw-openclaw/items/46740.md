## Summary

Keep open. Current main does not implement the requested ACPX exit reclassification, and this branch is not merge-ready because it edits stale pre-#61495 in-tree runtime paths and broadens backend-unavailable classification beyond truly silent prompt/backend exits.

## What This Changes

The branch remaps selected non-zero `acpx` process exits to `ACP_BACKEND_UNAVAILABLE` and updates ACPX runtime, manager, fixture, and error-text tests.

## Best Possible Solution

Move the classification to the current ACPX boundary: only genuinely silent prompt/backend exits should surface as backend-unavailable, while stderr-backed failures, permission-denied exits, control commands, session initialization, and recovery paths preserve their existing semantics. OpenClaw should then consume the published or explicitly pinned ACPX change with focused wrapper, manager, and error-text coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review findings: the PR diff deterministically remaps prompt/control exits and early session-init errors, while current-main tests still expect plain `acpx exited with code 1` after a done event to surface as `ACP_TURN_FAILED`.

## Solution Assessment

Is this the best way to solve the issue? No. The PR targets stale in-tree ACPX runtime code and broadens classification beyond truly silent exits; the safer route is package-level ACPX classification followed by a narrow OpenClaw consumption/update layer.

Status, cancel, close, mode, or config failures can therefore be mislabeled as backend outages.
  - confidence: 0.84
