## Summary

Keep open: the ACPX error-classification problem is not already solved on current main, but this branch is not merge-ready because it is conflicting, lacks real behavior proof, and still broadens backend-unavailable routing in ways that can hide stderr/control/init recovery failures.

## What This Changes

The PR changes ACPX runtime and ACP manager error handling/tests so selected silent `acpx exited with code 1` failures report `ACP_BACKEND_UNAVAILABLE` and `/acp doctor` guidance.

## Best Possible Solution

Rebase against the current acpx 0.7 runtime/manager and classify only truly silent pre-output ACPX backend exits, while preserving stderr-backed failures, control-command fallback codes, and ensure/retry/failover recovery.

## Reproduction Assessment

No, not as a high-confidence real ACPX reproduction. Source inspection and PR tests make the failure path clear, but the PR body/comments only provide unit-test commands and CI notes, not a live after-fix ACPX run.

## Solution Assessment

No. The target bug is valid, but this implementation should be rebased and narrowed so backend-unavailable is gated on genuinely silent prompt exits while existing control/init recovery behavior stays intact.
