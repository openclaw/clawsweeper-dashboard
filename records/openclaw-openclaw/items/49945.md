## Summary

Keep this PR open: it targets a real TUI `/new` hook regression, but the current patch still routes reset/lifecycle semantics through the fresh `tui-*` target instead of the session being ended, so bundled hooks can archive/log the wrong key and empty sessions can be persisted.

## What This Changes

The PR adds a `hookSourceKey` field to `sessions.reset`, calls it from TUI `/new` before switching to a generated `tui-<uuid>` session, updates tests, and adds a changelog entry.

## Best Possible Solution

Restore TUI `/new` so it emits hook and reset-lifecycle context for the session being left, then switches to an isolated `tui-*` key without creating an empty stored target session or changing `/reset` behavior.

## Reproduction Assessment

Yes by source inspection: current main routes TUI `/new` through `setSession(uniqueKey)` only, while docs require `command:new` for `/new` and `/reset` already uses `sessions.reset`. I did not execute a failing test in this read-only pass.

## Solution Assessment

No. The PR partly restores `previousSessionEntry`, but it still resets and emits under the fresh `tui-*` target; the narrower maintainable fix should keep hook/lifecycle context on the session being ended and only then switch the TUI to its isolated key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Route hooks through the session being left:** `src/tui/tui-command-handlers.ts:475`
  - body: Calling `sessions.reset` with `nextSessionKey` makes reset lifecycle operate on a brand-new `tui-*` row. `performGatewaySessionReset` writes a new entry and emits reset/session hooks for the requested key, so `/new` no longer gives `before_reset`/`session_end` the old transcript and repeated `/new` can leave empty persisted sessions behind.
  - confidence: 0.9
- **[P2] Keep command:new keyed to the archived session:** `src/gateway/session-reset-service.ts:509-510`
  - body: `hookSourceKey` makes `previousSessionEntry` come from the session being left, but the event still uses the fresh target as `sessionKey`. Bundled hooks consume `event.sessionKey` directly, so session-memory and command-logger can label/log the old conversation under the new `tui-*` key.
  - confidence: 0.86
