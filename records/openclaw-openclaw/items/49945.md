## Summary

Keep open: the PR targets a real TUI hook regression, but the current diff still resets and attributes lifecycle hooks to the future empty `tui-*` session, and the contributor proof is only test-level.

## What This Changes

The branch adds `hookSourceKey` plumbing to `sessions.reset`, calls TUI `/new` through `resetSession(nextSessionKey, "new", currentSessionKey)`, and updates tests plus the changelog.

## Best Possible Solution

Preserve isolated TUI `tui-*` sessions while emitting `command:new`, `before_reset`, and session lifecycle hooks against the session being left, without persisting a blank future session row.

## Reproduction Assessment

Yes, by source inspection. Current main’s TUI `/new` only switches local session state, while docs and gateway reset tests establish `command:new` as the hook-visible reset path.

## Solution Assessment

No. Resetting the future `tui-*` key with a separate `hookSourceKey` restores one command event but leaves event identity, lifecycle hooks, and persistence tied to the wrong session; target the session being left or reuse a parent-aware create/hook path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: Calling `resetSession(nextSessionKey, "new", state.currentSessionKey)` still emits `command:new`, `before_reset`, `session_end`, and `session_start` around the fresh `agent:main:tui-*` target. Hooks see `event.sessionKey` and lifecycle context for the new empty session instead of the conversation that ended, so this should fire the hook/lifecycle path against `state.currentSessionKey` before switching keys.
  - confidence: 0.9
- **[P3] Avoid persisting blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: `sessions.reset` always writes a new store entry and transcript header for its target. Since this call targets a freshly generated `tui-*` key before any message is sent, every `/new` can leave an empty persisted session in history; keep the unique key lazy or use a hook-only path that does not create the row.
  - confidence: 0.86
