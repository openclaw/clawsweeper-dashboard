## Summary

Keep this PR open, but do not merge this patch as-is: the underlying TUI hook regression is real, while the proposed reset-path API change creates wrong hook attribution, possible empty session rows, protocol drift, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes TUI `/new` to call gateway `sessions.reset` for a fresh `tui-*` key with `hookSourceKey`, updates reset protocol plumbing/tests, and adds a changelog entry.

## Best Possible Solution

Preserve isolated TUI `tui-*` switching, emit `command:new`, `before_reset`, and lifecycle hooks against the session being left before switching, leave the new session lazy until first use, and avoid a public reset `hookSourceKey` contract.

## Reproduction Assessment

Yes, source-level with high confidence: current main’s TUI `/new` only switches local session state, while docs and reset-service code show the hook-visible path is gateway reset. I did not run a live TUI hook session in this read-only review.

## Solution Assessment

No. The PR restores a hook call, but it targets and persists the future `tui-*` session and exposes cross-session hook-source selection on the public reset RPC; the narrower maintainable fix should attribute hooks to the previous session before switching keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep hook source selection off the public reset RPC:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: Adding `hookSourceKey` to `sessions.reset` lets a gateway client reset one session while supplying another session's hook context. Bundled hooks can read `previousSessionEntry`, so keep this TUI-only attribution internal or validate a strict session relationship before exposing it.
  - confidence: 0.78
- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: The PR calls `resetSession(nextSessionKey, "new", state.currentSessionKey)`, so the hook event's `sessionKey` remains the new blank `agent:<agent>:tui-*` key. Hooks such as `command-logger` and `session-memory` use `event.sessionKey`, which can archive or log the previous transcript under the future session instead of the conversation that ended.
  - confidence: 0.91
- **[P3] Avoid persisting blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: `sessions.reset` materializes the reset target in the session store and writes a new transcript header. Running it on a freshly generated `tui-*` key before any user message means repeated `/new` can add empty sessions to history; keep the new session lazy or use a hook-only path for the old session.
  - confidence: 0.86
- **[P2] Regenerate protocol mirrors if reset params change:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: The schema change adds a reset parameter, but the Swift gateway models still define `SessionsResetParams` with only `key` and `reason`. If this public field remains, regenerate or update the protocol mirrors so native clients and protocol checks stay in sync.
  - confidence: 0.82
