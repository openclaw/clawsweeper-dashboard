## Summary

Keep open. Current main still has the reported TUI `/new` hook-observability regression, but this PR should not merge as-is because it routes the hook through a future `tui-*` reset target, which misattributes bundled hook output and persists empty sessions; it also adds a new reset protocol field that needs boundary review.

## What This Changes

This PR changes TUI `/new` to call Gateway `sessions.reset` on a generated per-client `agent:<agent>:tui-<uuid>` key before `setSession`, adds `hookSourceKey` to reset handling, updates reset-hook tests, and adds a changelog entry.

## Best Possible Solution

Land a narrower implementation that emits `command:new` for the session being left, provides that source session as `previousSessionEntry`, keeps the next `tui-*` key isolated for future messages, and does not create a persisted empty future session. If a protocol field remains necessary, constrain it to same-agent intended transitions, document it, and test bundled `command-logger` plus `session-memory` behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: on current main `/new` only calls `setSession(uniqueKey)`, while `/reset` calls `client.resetSession`, and the TUI command-handler test still asserts only one reset call across `/new` plus `/reset`.

## Solution Assessment

No. The PR restores an event path, but targeting `sessions.reset` at the fresh `tui-*` key is not the narrowest maintainable fix because it changes hook attribution and persists blank sessions; a source-session hook emission path is safer.

## Security Review

Status: needs_attention

Summary: The diff introduces a cross-session `hookSourceKey` input to a Gateway session mutation path without visible source/target authorization or protocol documentation.

Concerns:

- **[medium] Constrain hookSourceKey before exposing it:** `src/gateway/session-reset-service.ts:268`
  - body: `hookSourceKey` lets a caller make reset hooks receive a different session entry from the reset target.
