## Summary

Keep this PR open for now: it targets a real TUI `/new` hook regression, but the current branch is not merge-ready because live proof is missing and the implementation exposes hook-source selection in `sessions.reset` while still attributing the hook event to a fresh `tui-*` key.

## What This Changes

The branch routes TUI `/new` through `sessions.reset` for a generated `tui-*` session, adds a `hookSourceKey` reset parameter, updates tests, and adds a changelog entry.

## Best Possible Solution

Rebase and implement TUI `/new` hook emission against the session being left before switching to the isolated `tui-*` key, following the existing parent-session/internal hook pattern without adding a public `hookSourceKey`.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main’s TUI `/new` only calls `setSession(uniqueKey)`, while docs and gateway reset code show `command:new` hooks are emitted through the reset/hook path.

## Solution Assessment

No. The PR targets the right regression, but resetting the future `tui-*` key plus public `hookSourceKey` is not the safest fix; the hook should be attributed to the session being left before the TUI switches sessions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep hook source selection off the reset RPC:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: This adds `hookSourceKey` to public `sessions.reset`, so a caller can reset one key while causing hooks to read another session's context. Current main's `sessions.create emitCommandHooks` path emits parent-session hooks without exposing arbitrary source selection; remove this field or add a strict, documented contract if maintainers choose it.
  - confidence: 0.86
- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: The reset target is the fresh `tui-*` key while `hookSourceKey` points at the old session, so `event.sessionKey` remains the new key. Bundled hooks such as `command-logger` and `session-memory` use that key when recording the ended conversation, which mis-associates hook output with a blank future session.
  - confidence: 0.88
- **[P3] Avoid creating empty TUI session rows on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: Calling `sessions.reset` on a never-used `tui-*` key creates a persisted entry before any message is sent. Repeated `/new` commands can clutter session lists with blank TUI sessions; emit the hook/lifecycle for the current session before switching instead of resetting the future key.
  - confidence: 0.78
