## Summary

Keep this PR open. Current main still has the TUI `/new` regression, but the proposed patch is not ready to merge because it emits hook events under the new `tui-*` target session, creates blank persisted `tui-*` session rows, and adds a new cross-session `hookSourceKey` protocol field that needs tighter semantics and review.

## What This Changes

This PR changes TUI `/new` to call `sessions.reset` on a generated per-client `agent:<agent>:tui-<uuid>` key before `setSession`, extends the Gateway `sessions.reset` request with `hookSourceKey`, adds reset-hook tests, and adds a changelog entry.

## Best Possible Solution

Revise this PR or replace it with a narrower fix that makes TUI `/new` emit `command:new` for the session being left, supplies `previousSessionEntry` from that source session, preserves per-client future `tui-*` isolation, and avoids persisting empty future sessions. If a protocol field is still needed, keep it additive, validate its source/target semantics, document it, and cover bundled `command-logger` and `session-memory` behavior in tests.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: on current main, `/new` calls only `setSession(uniqueKey)` while `/reset` calls `client.resetSession`, and the linked issue's hook-observability steps match that code path.

## Solution Assessment

No. The PR restores a hook emission path, but it does so by resetting the future session key; the safer solution is to emit hook-visible `command:new` with source-session attribution and previous-session context without creating an empty persisted `tui-*` session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Emit hooks for the session being left:** `src/tui/tui-command-handlers.ts:469`
  - body: The new `/new` path resets `nextSessionKey` and passes the current key only as `hookSourceKey`. `performGatewaySessionReset` still emits `event.sessionKey` as the reset target, and bundled hooks use `event.sessionKey` to label or log the conversation, so this records the old transcript under the brand-new `tui-*` key.
  - confidence: 0.88
- **[P2] Avoid persisting blank TUI sessions:** `src/tui/tui-command-handlers.ts:469`
  - body: `sessions.reset` always writes a store entry and transcript header for the requested key. Calling it on a fresh `tui-*` key means repeated `/new` commands create empty sessions before any message is sent, cluttering session history compared with the current local-only session switch.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a new cross-session `hookSourceKey` input to a Gateway session mutation path without visible source/target authorization or public protocol documentation.

Concerns:

- **[medium] Constrain hookSourceKey before exposing it:** `src/gateway/session-reset-service.ts:268`
  - body: `hookSourceKey` lets a caller make reset hooks receive a different session entry from the reset target. Because bundled and custom hooks may read transcript metadata or write derived memory/log artifacts, the field should be limited to the intended TUI same-agent transition or replaced with an internal-only hook emission seam.
  - confidence: 0.72
