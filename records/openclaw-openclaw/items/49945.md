## Summary

Keep open: the TUI `/new` regression is real, but this PR is not mergeable as-is because it exposes cross-session hook-source selection on the public reset RPC, attributes hook-visible data to the future `tui-*` session, can persist blank sessions, and still lacks real behavior proof.

## What This Changes

The PR changes TUI `/new` to call `sessions.reset` on a newly allocated `tui-*` session with the current session as `hookSourceKey`, updates gateway reset schema/plumbing/tests, and adds a changelog entry.

## Best Possible Solution

Preserve isolated TUI `tui-*` switching while emitting `command:new`, `before_reset`, and lifecycle hooks against the previous session before switching, without a public cross-session reset parameter and with live TUI hook proof.

## Reproduction Assessment

Yes, source-level: current main’s TUI `/new` only switches local session state, while docs and reset coverage define `/new` as hook-visible through the gateway reset hook path. I did not run a live TUI hook session in this read-only review.

## Solution Assessment

No. The PR restores a hook call, but it targets/persists the future `tui-*` session and exposes cross-session hook-source selection on the public reset RPC; the narrower maintainable fix should attribute hooks to the previous session before switching keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep hook source selection off the public reset RPC:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: Adding `hookSourceKey` to `sessions.reset` lets a gateway client reset one session while supplying another session's hook context. Bundled hooks can read `previousSessionEntry`, so keep this TUI-only attribution internal or validate a strict session relationship before exposing it.
  - confidence: 0.78
- **[P2] Emit /new hooks against the session being left:** `src/tui/tui-command-handlers.ts:468`
  - body: The PR calls `resetSession(nextSessionKey, "new", state.currentSessionKey)`, but the hook event still uses the reset target as `event.sessionKey`. Hooks such as `command-logger` and `session-memory` use that key as the conversation identity, so the old transcript can be logged or archived under the future `tui-*` session.
  - confidence: 0.91
- **[P3] Avoid persisting blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:468`
  - body: Calling `sessions.reset` on a brand-new `tui-*` key materializes a session entry and transcript header before any user message exists. Repeated `/new` commands can therefore leave blank sessions in history; keep the new session lazy or emit hooks for the old session without resetting the new key.
  - confidence: 0.86
- **[P2] Regenerate protocol mirrors if reset params stay public:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: The schema adds a public reset parameter, but the Swift gateway model still defines `SessionsResetParams` with only `key` and `reason`. If this field remains public, update the generated mirrors and protocol checks so native clients stay in sync.
  - confidence: 0.82
