## Label Justifications

- `P2`: This is a focused TUI session-hook regression with a linked issue and limited blast radius, not a core runtime emergency.
- `merge-risk: 🚨 compatibility`: The branch changes the public `sessions.reset` protocol shape and is currently not mergeable against main.
- `merge-risk: 🚨 session-state`: The proposed reset path can associate old-session hook output with a new blank `tui-*` session and create empty session rows.
- `merge-risk: 🚨 security-boundary`: `hookSourceKey` broadens caller control over which session context hook code can read during reset.

## Summary

Keep open: the linked TUI hook regression is real, but this branch is not merge-ready because it exposes cross-session hook-source selection on the public reset RPC, can misattribute old-session hook output to a fresh TUI session, creates empty TUI session rows, lacks live proof, and is currently not mergeable.

## What This Changes

This branch makes TUI `/new` call `sessions.reset` on a new `agent:<agent>:tui-<uuid>` key, adds `hookSourceKey` to session reset plumbing, updates tests, and adds a changelog entry.

## Best Possible Solution

Rework the fix to emit `/new` hook and reset lifecycle events for the session being left before switching the TUI to an isolated `tui-*` key, following the existing parent-session hook pattern without adding `hookSourceKey` to the public reset RPC.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main’s TUI `/new` only calls `setSession(uniqueKey)`, while docs and gateway reset code show `command:new` hooks are emitted through the reset/hook path.

## Solution Assessment

No. The PR targets the right regression, but resetting the future `tui-*` key plus public `hookSourceKey` is not the safest fix; the hook should be attributed to the session being left before the TUI switches sessions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep hook source selection off the reset RPC:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: This adds `hookSourceKey` to public `sessions.reset`, so a caller can reset one key while causing hooks to read another session's context. Current main's parent-session hook path handles this internally without exposing arbitrary source selection; remove the public field or add a strict documented contract if maintainers choose it.
  - confidence: 0.86
- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: The reset target is the fresh `tui-*` key while `hookSourceKey` points at the old session, so `event.sessionKey` remains the new key. Bundled hooks use that key when recording the ended conversation, which can mis-associate hook output with a blank future session.
  - confidence: 0.88
- **[P3] Avoid creating empty TUI session rows on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: Calling `sessions.reset` on a never-used `tui-*` key creates a persisted entry before any message is sent. Repeated `/new` commands can clutter session lists with blank TUI sessions; emit the hook/lifecycle for the current session before switching instead.
  - confidence: 0.78
