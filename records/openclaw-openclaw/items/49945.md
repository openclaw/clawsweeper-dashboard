## Summary

Keep this PR open, but not merge-ready: it targets a real TUI `/new` hook regression, yet the branch still lacks real behavior proof and introduces session attribution, public reset-protocol, security-boundary, and conflict blockers.

## What This Changes

The PR changes TUI `/new` to call `sessions.reset` on a generated `agent:<agent>:tui-*` key with `hookSourceKey` support, adds gateway/TUI tests, and updates the changelog.

## Best Possible Solution

Preserve isolated TUI `tui-*` switching, emit `command:new` and reset lifecycle hooks against the session being left before switching, keep the new session lazy until first use, and avoid a public reset `hookSourceKey` contract.

## Reproduction Assessment

Yes, source-level with high confidence. Current main’s TUI `/new` only switches local session state through `setSession(uniqueKey)`, while the hooks docs and gateway reset path define `command:new` as hook-visible.

## Solution Assessment

No. The PR restores a hook call, but it targets and persists the future `tui-*` session and exposes cross-session hook-source selection on the public reset RPC; the narrower fix should attribute hooks to the previous session before switching keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep hook source selection off the public reset RPC:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: Adding `hookSourceKey` to `sessions.reset` lets a gateway client reset one session while supplying another session's hook context. Bundled hooks can read `previousSessionEntry`, so keep this TUI-only attribution internal or validate a strict session relationship before exposing it.
  - confidence: 0.82
- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: The PR calls `resetSession(nextSessionKey, "new", state.currentSessionKey)`, so the hook event's `sessionKey` remains the new blank `agent:<agent>:tui-*` key. Hooks such as `command-logger` and `session-memory` use `event.sessionKey`, which can archive or log the previous transcript under the future session instead of the conversation that ended.
  - confidence: 0.91
- **[P3] Avoid materializing blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: `sessions.reset` materializes the reset target in the session store and writes a new transcript header. Running it on a freshly generated `tui-*` key before any user message means repeated `/new` can add empty sessions to history; keep the new session lazy or use a hook-only path for the old session.
  - confidence: 0.86
- **[P2] Regenerate protocol mirrors if the reset parameter remains public:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: The schema change adds a reset parameter, but the Swift gateway model still defines `SessionsResetParams` with only `key` and `reason`. If this public field remains, regenerate or update protocol mirrors so native clients and protocol checks stay in sync.
  - confidence: 0.8
