## Summary

Keep this PR open: the current main regression is source-reproducible, but the patch is not mergeable as-is because it leaves the TUI backend reset contract incomplete and still emits/persists reset state around the fresh `tui-*` key rather than a coherent ended-session hook context.

## What This Changes

The PR adds a `hookSourceKey` parameter to `sessions.reset`, calls it from TUI `/new` before switching to a new `tui-<uuid>` key, updates gateway/TUI tests, and adds a changelog entry.

## Best Possible Solution

Repair the PR so TUI `/new` emits the documented `command:new` behavior for the session being left while preserving per-client `tui-<uuid>` isolation and avoiding blank persisted sessions.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: `/new` calls only `setSession(uniqueKey)`, while the hook-emitting path is `sessions.reset`; I did not execute Vitest in this read-only checkout.

## Solution Assessment

No. The proposed direction partially restores a `command:new` event, but the implementation should use a coherent previous/next session contract across TUI backends and hooks instead of resetting a fresh key with a mismatched source entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Update the TUI reset contract before passing hook source:** `src/tui/tui-command-handlers.ts:470`
  - body: The new `/new` call passes three arguments to `client.resetSession`, but `createCommandHandlers` receives `client` as `TuiBackend`, whose `resetSession` signature still accepts only `(key, reason?)`; the embedded backend also ignores this third value. This will fail the TUI typecheck and leaves one TUI backend unable to provide the hook source.
  - confidence: 0.92
- **[P2] Keep command:new keyed to the archived session:** `src/gateway/session-reset-service.ts:276-277`
  - body: `hookSourceKey` swaps `previousSessionEntry`, but the emitted hook still uses the reset target as `event.sessionKey`. Since TUI passes the fresh `tui-*` key as that target, bundled hooks such as `command-logger` and `session-memory` record the old transcript under the new empty key instead of the conversation that ended.
  - confidence: 0.9
- **[P2] Avoid persisting blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:470`
  - body: Calling `sessions.reset` on the fresh `tui-*` key writes a new session row and transcript header before the user sends anything. The existing isolation behavior only switched the client key, so repeatedly pressing `/new` will now clutter session history with empty TUI sessions.
  - confidence: 0.87
