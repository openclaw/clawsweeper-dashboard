## Summary

Keep open. The linked TUI hook regression is valid and still present on current main, but this PR is not mergeable as written because it exposes a public cross-session hook-source parameter, attributes hooks to the future blank session, can persist empty TUI sessions, misses protocol mirrors, and lacks real TUI hook proof.

## What This Changes

The PR changes TUI `/new` to call `sessions.reset` on a newly generated canonical `tui-*` key, adds a public `hookSourceKey` reset parameter, updates gateway reset context plumbing/tests, and adds a changelog entry.

## Best Possible Solution

Preserve isolated TUI `tui-*` session switching, but emit `command:new`, `before_reset`, and lifecycle hooks against the previous session before switching, leaving the new target lazy until first use and avoiding a public `hookSourceKey` reset contract.

## Reproduction Assessment

Yes, source-level. Current main’s TUI `/new` only calls `setSession(uniqueKey)`, while docs and reset coverage define `/new` as hook-visible through the gateway reset hook path.

## Solution Assessment

No. This PR restores a hook call, but it targets and persists the future `tui-*` session and exposes cross-session hook-source selection on the public reset RPC; the narrower fix should attribute hooks to the previous session before switching keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep hook source selection off the public reset RPC:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: Adding `hookSourceKey` to `sessions.reset` lets any gateway client reset one session while supplying another session's hook context. Bundled hooks can read `previousSessionEntry`, so keep this TUI-only source selection internal or validate a strict parent/access relationship before exposing it.
  - confidence: 0.78
- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: The PR calls `resetSession(nextSessionKey, "new", state.currentSessionKey)`, so the hook event's `sessionKey` remains the new blank `agent:<agent>:tui-*` key. Hooks such as `command-logger` and `session-memory` use `event.sessionKey`, which can archive the previous transcript under the future session instead of the conversation that ended.
  - confidence: 0.91
- **[P3] Avoid persisting blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: `sessions.reset` materializes the reset target in the session store and writes a new transcript header. Running it on a freshly generated `tui-*` key before any user message means repeated `/new` can add empty sessions to history; keep the new session lazy or use a hook-only path for the old session.
  - confidence: 0.86
- **[P2] Regenerate protocol mirrors if reset params change:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: The schema change adds a reset parameter, but the Swift gateway models still define `SessionsResetParams` with only `key` and `reason`. If this public field remains, regenerate/update the protocol mirrors so native clients and protocol checks stay in sync.
  - confidence: 0.82
