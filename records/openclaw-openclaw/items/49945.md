## Summary

Keep this PR open, but do not merge it as-is: the regression is real, and the branch restores part of the hook context while still emitting and persisting the reset under the fresh `tui-*` key that bundled hooks treat as the ended session.

## What This Changes

The PR updates TUI `/new`, gateway `sessions.reset` plumbing, reset-hook tests, TUI command-handler tests, and the changelog so `/new` resets a newly allocated `tui-<uuid>` session with previous-session hook context before switching the client.

## Best Possible Solution

Emit `command:new` and reset lifecycle hooks for the current session being left, then switch the TUI to an unpersisted unique `tui-*` key; add regression coverage for hook `sessionKey`, `previousSessionEntry`, lifecycle context, and absence of blank `tui-*` rows.

## Reproduction Assessment

Yes, by source inspection. Current main's TUI `/new` path only switches local session state, while docs and reset-hook tests establish `command:new` as the hook-visible reset path.

## Solution Assessment

No. The PR restores a `command:new` call, but routing it through `sessions.reset` on the future `tui-*` key misattributes hooks/lifecycle events and persists blank sessions; a narrower hook emission path should target the session being left.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: Passing `nextSessionKey` into `resetSession` makes `sessions.reset` emit command and lifecycle hooks with `sessionKey=agent:main:tui-*`. Bundled `command-logger` and `session-memory` treat `event.sessionKey` as the conversation that ended, so the old transcript is attributed to the brand-new TUI key. Fire the hook/lifecycle events against `state.currentSessionKey` before switching to the unique key.
  - confidence: 0.9
- **[P3] Avoid persisting blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: `sessions.reset` always writes a new session row and transcript header for its target key. Calling it with a fresh `tui-*` key before any message is sent means repeatedly pressing `/new` creates empty persisted sessions that will appear in session list/history, unlike the current lazy local switch behavior.
  - confidence: 0.86
