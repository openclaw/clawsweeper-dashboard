## Summary

Keep this PR open, but it needs changes before merge: the bug is valid and the PR is focused, yet the current diff still attributes the old conversation to the fresh `tui-*` hook target, creates empty persisted TUI sessions, and only has mock/unit-style proof.

## What This Changes

The PR routes TUI `/new` through `sessions.reset(reason="new")` for a new per-client `tui-*` key, adds `hookSourceKey` to `sessions.reset`, updates tests, and adds a changelog entry.

## Best Possible Solution

Fire the existing `/new` hook/lifecycle contract for the session being left, then switch the TUI to an isolated `tui-*` key without persisting an empty session row, with regression coverage and real hook proof.

## Reproduction Assessment

Yes. Source inspection on current main shows TUI `/new` only switches local state via `setSession(uniqueKey)`, while the documented and tested hook-visible path is behind reset/create lifecycle code.

## Solution Assessment

No. The direction is narrow, but this implementation targets the fresh `tui-*` session for reset/hook identity and persists it immediately; the fix should attribute hooks to the session being left and avoid blank session rows.

## Review Findings
