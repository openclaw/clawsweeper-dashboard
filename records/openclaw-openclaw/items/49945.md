## Summary

Keep open. Current main still has the TUI `/new` hook-observability regression, but this PR should not merge as-is because it routes hook attribution through a future `tui-*` reset target, persists empty sessions, and exposes a new reset protocol field without enough boundary control.

## What This Changes

This PR changes TUI `/new` to call Gateway `sessions.reset` on a generated per-client `agent:<agent>:tui-<uuid>` key before `setSession`, adds a `hookSourceKey` reset parameter, updates reset tests, and adds a changelog entry.

## Best Possible Solution

Emit `command:new` for the session being left, pass that source entry as `previousSessionEntry`, keep the next `tui-*` key isolated for future messages, and avoid creating a blank persisted future session.

## Reproduction Assessment

Yes. The source-level path is high-confidence: current main `/new` only calls local `setSession(uniqueKey)`, while `command:new` hook delivery is tied to `sessions.reset(reason="new")`.

## Solution Assessment

No. The PR restores an event path, but resetting the fresh future `tui-*` key changes hook attribution and creates blank session history; a source-session hook emission path is narrower and safer.

## Security Review

Status: needs_attention

Summary: The diff introduces a cross-session `hookSourceKey` input to a Gateway session mutation path without visible source/target authorization or protocol documentation.

Concerns:

- **[medium] Constrain hookSourceKey before exposing it:** `src/gateway/session-reset-service.ts:268`
  - body: `hookSourceKey` lets a caller make reset hooks receive a different session entry from the reset target.
