## Summary

Keep open: current main still has the TUI `/new` hook regression, but this PR is not merge-ready because it routes hook attribution through a public reset RPC escape hatch, can misattribute session state to a fresh `tui-*` key, and lacks real after-fix TUI hook proof.

## What This Changes

The PR routes TUI `/new` through `sessions.reset` for a new `tui-*` key, adds `hookSourceKey` to reset plumbing, updates tests, and adds a changelog entry.

## Best Possible Solution

Preserve isolated `tui-*` session switching while firing `command:new`, `before_reset`, and lifecycle hooks against the previous session through an internal or `sessions.create`-style parent hook path, without expanding the public reset contract.

## Reproduction Assessment

Yes, source-level: current main’s TUI `/new` only calls `setSession(uniqueKey)`, while docs and gateway reset coverage define `command:new` as hook-visible through the reset hook path. I did not run a live TUI hook session in this read-only review.

## Solution Assessment

No. The PR restores a hook call, but it targets/persists the future `tui-*` session and exposes cross-session hook-source selection on the public reset RPC; the safer fix should attribute hooks to the previous session before switching keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep hook source selection off the public reset RPC:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: Adding `hookSourceKey` to `sessions.reset` lets a gateway client reset one session while supplying another session's hook context. Bundled hooks can read `previousSessionEntry`, so keep this TUI-only attribution internal or validate a strict session relationship before exposing it.
  - confidence: 0.78
- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: The PR calls `resetSession(nextSessionKey, "new", state.currentSessionKey)`, so the hook event's `sessionKey` remains the new blank `agent:<agent>:tui-*` key. Hooks such as `command-logger` and `session-memory` can archive or log the previous transcript under the future session instead of the conversation that ended.
  - confidence: 0.91
- **[P3] Avoid persisting blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: `sessions.reset` materializes the reset target in the session store and writes a new transcript header. Running it on a freshly generated `tui-*` key before any user message means repeated `/new` can add empty sessions to history; keep the new session lazy or use a hook-only path for the old session.
  - confidence: 0.86
- **[P2] Regenerate protocol mirrors if reset params change:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: The schema change adds a reset parameter, but the Swift gateway models still define `SessionsResetParams` with only `key` and `reason`. If this public field remains, regenerate or update the protocol mirrors so native clients and protocol checks stay in sync.
  - confidence: 0.82
