## Summary

Keep this PR open: the linked TUI hook regression is still present on current main, but the proposed fix is not mergeable as-is because it attributes hooks to the new `tui-*` target, persists blank sessions, changes the public reset protocol without generated mirrors, and lacks real TUI+hook proof.

## What This Changes

The PR routes TUI `/new` through `sessions.reset` with a new `hookSourceKey`, updates gateway reset plumbing, tests, and the changelog.

## Best Possible Solution

Keep isolated TUI `tui-*` session switching, but emit `command:new`, `before_reset`, and lifecycle context against the session being left while keeping the new target lazy until first use.

## Reproduction Assessment

Yes, source-level. Current main’s TUI `/new` only switches local session state with `setSession(uniqueKey)`, while docs and bundled hooks define `/new` as a hook-visible `command:new` event and `/reset` still goes through gateway reset.

## Solution Assessment

No. The PR restores a hook call, but it targets and persists the future `tui-*` session and exposes source-session selection on the public reset RPC; the narrower fix should attribute hooks to the previous session before switching keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep hook source selection off the public reset RPC:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: Adding `hookSourceKey` to `sessions.reset` lets any gateway client pair one reset target with a different hook context. Because bundled hooks consume `previousSessionEntry` and can read the referenced transcript, keep this source selection internal to the TUI flow or validate a concrete parent/access relationship before exposing it.
  - confidence: 0.78
- **[P2] Emit /new hooks for the session being left:** `src/tui/tui-command-handlers.ts:467`
  - body: Calling `resetSession(nextSessionKey, "new", state.currentSessionKey)` still makes the fresh `agent:<agent>:tui-*` key the reset target and emitted `event.sessionKey`. Hooks such as `command-logger` and `session-memory` can therefore record the previous transcript under the new blank session instead of the conversation that ended.
  - confidence: 0.91
- **[P3] Avoid persisting blank TUI sessions on /new:** `src/tui/tui-command-handlers.ts:467`
  - body: `sessions.reset` writes a store entry and transcript for its target. Since this call targets a newly generated `tui-*` key before any user message exists there, repeated `/new` commands can leave empty sessions in history; use a hook-only path or keep the new TUI session lazy.
  - confidence: 0.86
- **[P2] Regenerate protocol mirrors for reset params:** `src/gateway/protocol/schema/sessions.ts:97`
  - body: Changing `SessionsResetParamsSchema` is a gateway wire-contract change, but the generated Swift protocol models are not updated in this PR. If this parameter remains, protocol checks and native clients will be out of sync with the TypeScript schema.
  - confidence: 0.82
