## Summary

This is a focused bug-fix PR with sufficient runtime proof and no blocking code findings from this review, but it should stay open for maintainer compatibility judgment because the new guard intentionally changes all OpenAI-compatible chat streams from outbound provider errors to a local fail-closed error when no user or assistant turn remains.

## What This Changes

The PR adds a pre-send OpenAI-compatible chat payload guard, narrows blank-prompt replay classification to user/assistant turns, and adds regression coverage for the guard and compaction recovery path.

## Best Possible Solution

Land this shape if maintainers accept user/assistant-free OpenAI-compatible chat payloads as globally invalid OpenClaw turns; otherwise scope the guard through provider compatibility while preserving the clear local error for affected Qwen-style routes.

## Reproduction Assessment

Yes for the transport invariant: current main source has no final user/assistant role guard, and the PR body supplies a local endpoint proof showing the new guard blocks system-only sends before HTTP. I did not live-reproduce the full long Qwen compaction sequence from the linked issue.

## Solution Assessment

Unclear until maintainers make the compatibility call. The pre-send guard is a narrow safety net for the reported invalid payload, but applying it globally across OpenAI-compatible providers is the part that needs maintainer judgment.
