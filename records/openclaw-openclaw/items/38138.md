## Summary

Keep open. Current main still chooses the Discord agent from static route inputs before mention detection, so a thread that inherits a parentPeer binding can still resolve to the parent-channel agent while explicit mentions and per-agent mentionPatterns only gate the selected route. The closed unmerged related PR attempted this behavior, and the broader content-routing issue is not a canonical replacement for this Discord-specific override.

## What This Changes

The issue requests that an explicit Discord @agent mention, or a matching agent mentionPatterns entry, route the message to that agent even when normal binding tiers would choose another agent.

## Best Possible Solution

Add a documented, cache-safe explicit target-agent mention route step before Discord mention gating, with deterministic precedence rules and regression coverage for a thread inheriting one bound agent while mentioning another configured agent. Update channel-routing and Discord mention docs after the contract is settled.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord thread with parentConversationId bound to one agent and message content/mentions targeting another; current route resolution cannot receive that mention signal, so effectiveRoute remains the static parentPeer-selected route.

## Solution Assessment

Unclear. The requested behavior is plausible, but the best implementation depends on maintainer decisions about whether this belongs as a resolver tier, a Discord preflight override, or an additive generic routing seam with explicit precedence rules.
