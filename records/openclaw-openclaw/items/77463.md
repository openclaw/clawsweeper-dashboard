## Summary

Keep this PR open: current main still has OpenRouter media understanding as image-only, and the PR is a useful plugin-local feature, but it is not merge-ready because the new STT format resolver misses common/parameterized audio MIME values and the required changelog entry is absent.

## What This Changes

The PR adds OpenRouter inbound audio transcription through the media-understanding provider, updates OpenRouter metadata/defaults, adds focused tests, and updates media/OpenRouter docs.

## Best Possible Solution

Keep the feature plugin-local, normalize OpenRouter STT format resolution through repo-consistent MIME handling, add focused coverage and the single-line changelog entry, then run the targeted provider/defaults/channel checks plus the normal changed gate.

## Reproduction Assessment

Not applicable for the central request because this is a feature PR, not a broken-current-main bug. The MIME regression in the proposed patch is source-reproducible: an extensionless audio attachment with `audio/ogg; codecs=opus`, `audio/opus`, or `audio/m4a` reaches the exact-match resolver and throws before sending the STT request.

## Solution Assessment

Mostly yes: implementing this in the OpenRouter plugin and manifest metadata is the right boundary. The patch should first reuse repo-style MIME normalization, cover the extensionless MIME cases, and add the required changelog entry.

  - body: The new resolver compares the raw MIME string after only trim/lowercase, so values already treated as audio elsewhere, such as `audio/ogg; codecs=opus`, `audio/opus`, and `audio/m4a`, fall through unless the filename has a usable extension. `MediaAttachmentCache` can pass an extensionless fallback like `media-1`, which makes OpenRouter STT throw before making the request even though the MIME is sufficient.
