## Summary

Keep open: current main still has a source-reproducible reset-path bug where a fresh `sessionId` can keep the previous `sessionFile`, and the ACP orphan portion is only partially covered by startup reaping rather than immediate failed-launch cleanup or doctor/task surfacing.

## What This Changes

The issue reports stale transcript mapping after gateway session reset plus orphaned Codex ACP processes after failed pre-initialize launches.

## Best Possible Solution

Fix gateway reset to rewrite UUID-derived transcript filenames to the new session id while preserving intentional custom `sessionFile` overrides, and make ACPX failed pre-initialize launches clean up or surface orphaned OpenClaw-owned processes without waiting for manual host inspection.

## Reproduction Assessment

Yes for the session reset half: seed a store row with `sessionId=old` and `sessionFile=old-topic-123.jsonl`, then invoke `sessions.reset`; current source will pass the stale file to a resolver that trusts it. The ACP orphan half has strong incident evidence but still needs a live failed-initialize harness to reproduce current-main process state end to end.

## Solution Assessment

Yes for the reset direction: reuse or extract the existing session-file rewrite behavior at the gateway reset boundary, with tests that preserve custom overrides. For ACP, the maintainable path is plugin-local lease/reaper cleanup plus doctor/task visibility, unless validation shows the missing cleanup must be fixed in the upstream `acpx` package.
