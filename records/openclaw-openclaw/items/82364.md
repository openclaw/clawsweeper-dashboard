## Summary

Keep open: current main still has the reported reset-store bug, because the gateway reset path can rotate `sessionId` while persisting the previous generated `sessionFile`. Current ACPX code has some lease and orphan reaping support, so the ACP half needs validation before expanding cleanup policy, but that does not make the direct session reset defect obsolete.

## What This Changes

Reports OpenClaw 2026.5.12 session resets persisting stale transcript paths and Codex ACP launch failures leaving orphaned processes.

## Best Possible Solution

Rewrite generated reset transcript paths to the new session id, including topic suffixes, while preserving explicit custom paths and validating the remaining ACP orphan behavior against the current lease/reaper implementation.

## Reproduction Assessment

Yes, source-level: `performGatewaySessionReset` rotates `sessionId` but still resolves against the old generated `sessionFile`, and the read path can fall back to that stale file if no new transcript exists. I did not run a mutating live repro because this review is read-only.

## Solution Assessment

Yes for the reset half: the narrow maintainable fix is to reuse or share the existing generated-path rewrite logic in the gateway reset service and add regression coverage. The ACP orphan half should be validated separately against current ACPX lease and startup reaper behavior before adding doctor/task policy.
