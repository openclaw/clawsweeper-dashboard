## Summary

Keep open. Current main still consumes the initial Mattermost WebSocket `posted` payload's `file_ids` directly, so an empty array becomes no downloaded media; the related fix PR #59591 was closed unmerged and does not make this obsolete.

## What This Changes

The issue asks the Mattermost plugin to recover file attachments when the initial `posted` WebSocket event carries empty `file_ids` by re-fetching the post before media resolution.

## Best Possible Solution

Implement a Mattermost-owned bounded post re-fetch before inbound media resolution when a WebSocket post has missing or empty `file_ids`, then pass recovered IDs into the existing SSRF-guarded media resolver and cover file-only plus text-with-file race cases in focused tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a `posted` event whose parsed post has `file_ids: []`; current main forwards that empty array to `resolveMattermostMedia`, which returns no media.

## Solution Assessment

Yes. The narrow maintainable fix is to add a Mattermost client `GET /api/v4/posts/{post_id}` helper and use it with a short bounded retry before media resolution; no core channel API change is needed.

## Repair Work Prompt

Fix the Mattermost inbound attachment race reported in openclaw/openclaw#59576. Current main reads `post.file_ids` from the initial WebSocket `posted` event and passes it directly to `resolveMattermostMedia`; when Mattermost sends `file_ids: []`, attachments are invisible to the agent. Implement this inside the Mattermost plugin only: add a typed `fetchMattermostPost(client, postId)` helper in `extensions/mattermost/src/mattermost/client.ts` using `GET /api/v4/posts/{post_id}`, then in the inbound monitor re-fetch the same post after a short bounded delay/retry when the non-debounced inbound post has missing/empty file IDs before calling `resolveMattermostMedia`. Keep the existing authenticated client and existing SSRF-guarded media download path; do not add core channel APIs, global migrations, new credentials, or broad retry infrastructure. Be careful around the inbound debounce path, which intentionally builds a merged post with `file_ids: []`; do not use a synthetic merged post ID to recover attachments incorrectly. Add focused tests for the client helper and monitor behavior for file-only and text-plus-file race cases. Because this is user-facing, add a single-line `CHANGELOG.md` fix entry with appropriate public reporter credit such as `Thanks @alexanderkreidich` and `Thanks @SeniorXie` if both are used as source context, and do not credit bots or maintainers.
