## Label Justifications

- `P2`: This is a valid channel-specific bug with user-visible Mattermost attachment loss but limited blast radius outside that plugin.
- `impact:message-loss`: Mattermost file attachments can be omitted from the agent's inbound media context when the WebSocket post carries empty or missing `file_ids`.

## Summary

Keep open. Current main and the latest release still pass the WebSocket post's `file_ids` directly into Mattermost media resolution, so a valid `posted` event with empty or missing attachment IDs loses inbound media; the direct fix PR is closed unmerged and the related open PR has not landed.

## What This Changes

Fix Mattermost inbound handling so attachments whose WebSocket `posted` event arrives with empty or missing `file_ids` are hydrated from REST before media resolution.

## Best Possible Solution

Land or repair the plugin-local Mattermost hydration work so incomplete posts are re-fetched with bounded retries before media resolution, preserving debounce/thread behavior and proving the file-size edge cases with regression coverage and redacted live Mattermost evidence.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Mattermost `posted` payload whose parsed post has `file_ids: []` or missing `file_ids`; current main accepts that payload, passes it to `resolveMattermostMedia`, and receives an empty media list.

## Solution Assessment

Yes in direction. A bounded Mattermost-plugin `GET /api/v4/posts/{post_id}` hydration before media resolution is the narrow maintainable fix, but it should be coordinated with https://github.com/openclaw/openclaw/pull/77619 and validated for latency and file-size behavior.
