## Summary

Keep open. Current main still routes the initial Mattermost WebSocket post's `file_ids` directly into inbound media resolution, so empty or missing IDs produce no downloaded attachments; the related fix PR was closed unmerged.

## What This Changes

Recover Mattermost inbound attachment IDs when a `posted` WebSocket event arrives before file linkage is populated.

## Best Possible Solution

Implement a Mattermost-plugin-local bounded post re-fetch before inbound media resolution when WebSocket `file_ids` are missing or empty, then reuse the existing guarded media download path and add focused regression tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Mattermost `posted` payload whose parsed post has `file_ids: []` or no `file_ids`; current main passes that value to `resolveMattermostMedia`, which returns no media.

## Solution Assessment

Yes. The narrow maintainable fix is a Mattermost client `GET /api/v4/posts/{post_id}` helper plus a short bounded retry before calling the existing media resolver; no core channel API change is needed.
