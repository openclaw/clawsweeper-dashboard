## Label Justifications

- `P2`: This is a valid channel-specific bug with user-visible attachment loss but limited blast radius outside Mattermost.
- `impact:message-loss`: Mattermost file attachments are silently absent from the agent's inbound media context when `file_ids` is empty.

## Summary

Keep open. Current main and the latest release still resolve Mattermost inbound media directly from the WebSocket post's `file_ids`, so the empty-array race still drops attachments; the direct fix PR is closed unmerged and the related hydration/thread-file PR is still open.

## What This Changes

The issue reports that Mattermost inbound messages can lose file attachments because the WebSocket `posted` event may contain an empty `file_ids` array before Mattermost finishes linking uploaded files.

## Best Possible Solution

Land or repair the open Mattermost PR so the plugin hydrates incomplete posts before inbound media resolution with bounded retry behavior, regression coverage, and redacted live Mattermost proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Mattermost `posted` payload whose parsed post has `file_ids: []` or missing `file_ids`; current main passes that value to `resolveMattermostMedia`, which returns no media.

## Solution Assessment

Yes in direction. A bounded plugin-local `GET /api/v4/posts/{post_id}` hydration before media resolution is the narrow maintainable fix, but it should be coordinated with https://github.com/openclaw/openclaw/pull/77619 and validated for latency.
