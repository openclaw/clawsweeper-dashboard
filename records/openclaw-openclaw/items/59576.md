## Label Justifications

- `P2`: The report is a valid Mattermost plugin bug with user-visible attachment loss but limited channel-specific blast radius.
- `impact:message-loss`: Mattermost file attachments are silently absent from the agent's inbound media context when `file_ids` is empty.

## Summary

Keep open. Current main and the latest release still route Mattermost inbound media only from the WebSocket post's `file_ids`, so the reported empty-array race still drops attachments; the direct fix PR is closed unmerged and the newer related PR is still open.

## What This Changes

The issue requests Mattermost post re-hydration so attachments are detected when the WebSocket `posted` event arrives before `file_ids` is populated.

## Best Possible Solution

Land a Mattermost-plugin-local post hydration path before inbound media resolution, with bounded retry behavior, regression coverage for empty `file_ids`, and redacted live Mattermost proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Mattermost `posted` payload whose parsed post has `file_ids: []` or missing `file_ids`; current main passes that value to `resolveMattermostMedia`, which returns no media.

## Solution Assessment

Yes in direction. A bounded plugin-local `GET /api/v4/posts/{post_id}` hydration before media resolution is the narrow maintainable fix, but it should be coordinated with the open related Mattermost PR and validated for latency.
