## Label Justifications

- `P2`: This is a valid Mattermost-specific bug with visible attachment loss, but the blast radius is limited to that plugin/channel path.
- `impact:message-loss`: Mattermost file attachments can be omitted from the agent's inbound media context when the WebSocket post carries empty or missing `file_ids`.

## Summary

Keep open. Current main and the latest release still resolve inbound Mattermost media directly from the WebSocket post's `file_ids`, so a valid `posted` payload with empty or missing IDs reaches the monitor but produces no inbound media; the direct fix PR is closed unmerged and the overlapping open PR still needs review/proof.

## What This Changes

The issue reports inbound Mattermost attachments being missed when WebSocket `posted` events arrive with empty `file_ids` and requests REST post hydration before media resolution.

## Best Possible Solution

Land or repair the plugin-local Mattermost hydration path so incomplete posts are re-fetched with bounded retries before media resolution, while preserving debounce/thread behavior and proving attachment-size edge cases.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Mattermost `posted` WebSocket payload whose parsed post has `file_ids: []` or no `file_ids`; current main accepts it, calls `resolveMattermostMedia(post.file_ids)`, and receives an empty media list.

## Solution Assessment

Yes in direction. A bounded Mattermost-plugin REST hydration step before media resolution is the narrow maintainable fix, but it should be coordinated with https://github.com/openclaw/openclaw/pull/77619 and validated for debounce, latency, and file-size behavior.
