## Summary

Keep open. Current main and the latest release still feed the WebSocket post's `file_ids` directly into Mattermost media resolution, so an empty or missing array produces no inbound attachment; the direct fix PR was closed unmerged and the newer related PR is still open.

## What This Changes

The issue asks the Mattermost plugin to recover attached file IDs when incoming WebSocket post events arrive with empty or missing `file_ids`.

## Best Possible Solution

Implement a Mattermost-plugin-local bounded post hydration/re-fetch before inbound media resolution when WebSocket `file_ids` are missing or empty, reusing the existing authenticated client and guarded media downloader, and coordinate with https://github.com/openclaw/openclaw/pull/77619 if that remains the carrier.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Mattermost `posted` payload whose parsed post has `file_ids: []` or no `file_ids`; current main passes that value to `resolveMattermostMedia`, which returns no media.

## Solution Assessment

Yes. The narrow maintainable fix is to hydrate or re-fetch the same Mattermost post through the existing REST client before media resolution, with bounded retry/latency and focused regression coverage.
