## Summary

Keep open. Current main still passes the WebSocket post's empty or missing `file_ids` straight into Mattermost media resolution, and the related replacement work is still an open PR rather than a shipped fix.

## What This Changes

Handle Mattermost WebSocket `posted` events whose `file_ids` are empty before attachment processing finishes so inbound file attachments are still downloaded.

## Best Possible Solution

Land a bounded Mattermost-plugin fix that hydrates the current post before inbound media resolution when WebSocket file IDs are missing or empty, with focused regression coverage and bounded latency.

## Reproduction Assessment

Yes, at source level: a parsed Mattermost `posted` payload with `file_ids: []` reaches `resolveMattermostMedia(post.file_ids)`, and that resolver returns no media for empty IDs. I did not run a live Mattermost instance in this read-only review.

## Solution Assessment

Yes. The narrow maintainable direction is Mattermost-plugin-local post hydration or bounded REST re-fetch before the existing media resolver; the open related PR may be the right vehicle if it repairs its blockers and keeps retry behavior bounded.
