## Label Justifications

- `P2`: This is a valid channel-specific bug with user-visible Mattermost attachment loss but limited blast radius outside the plugin.
- `impact:message-loss`: Mattermost file attachments can be omitted from the agent's inbound media context when the WebSocket post carries empty or missing `file_ids`.

## Summary

Keep open. Current main still resolves inbound Mattermost media only from the WebSocket post's `file_ids`, so an accepted `posted` event with empty or missing IDs loses the attachment; the direct fix PR is closed unmerged and the newer overlapping PR has not landed.

## What This Changes

Mattermost should hydrate or re-fetch inbound posts before media resolution when WebSocket `posted` events arrive before `file_ids` are populated.

## Best Possible Solution

Land or repair the plugin-local Mattermost hydration work so incomplete posts are re-fetched with bounded retries before media resolution, preserving debounce/thread behavior and proving attachment-size edge cases with regression coverage and redacted live Mattermost evidence.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Mattermost `posted` payload whose parsed post has `file_ids: []` or missing `file_ids`; current main accepts that payload, passes it to `resolveMattermostMedia`, and receives an empty media list.

## Solution Assessment

Yes in direction. A bounded Mattermost-plugin `GET /api/v4/posts/{post_id}` hydration before media resolution is the narrow maintainable fix, but it should be coordinated with https://github.com/openclaw/openclaw/pull/77619 and validated for latency and file-size behavior.
