## Summary

Keep open. Current main still returns early for a literal non-template `session.store`, so sibling per-agent session stores are not loaded for `sessions.list`; the linked closing PR remains open and should be reviewed, landed, or replaced before this issue is closed.

## What This Changes

The issue asks for `sessions.list` and the Control UI Sessions dashboard to include all active per-agent sessions instead of only the main/default session when multiple session stores exist.

## Best Possible Solution

Review and land #54716, or an equivalent current-main fix, so literal paths that structurally point at `agents/<id>/sessions/sessions.json` use all-agent discovery while genuinely shared literal stores keep their current single-store behavior. Include regression coverage for a literal `agents/main/sessions/sessions.json` configuration.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Configure `session.store` to a literal per-agent path such as `.../agents/main/sessions/sessions.json` while a sibling agent store exists, then call `sessions.list`; current main reads only the literal default-agent store and returns before sibling discovery.

## Solution Assessment

Is this the best way to solve the issue? Yes. The linked direction is the narrowest maintainable fix because it reuses existing all-agent discovery only for literal paths that match the per-agent store layout and preserves true shared literal-store behavior.
