## Summary

Keep open. Current main still has the literal `session.store` branch that reads one default-agent store and returns before all-agent discovery, and the issue is paired with open closing PR #54716 rather than a merged fix.

## What This Changes

The issue reports that `sessions_list` and the Sessions dashboard only show the main session when additional sessions, such as Telegram or newly created sessions, exist.

## Best Possible Solution

Review and land #54716, or an equivalent current-main fix in `src/config/sessions/combined-store-gateway.ts`, so literal paths that match `agents/<id>/sessions/sessions.json` use all-agent discovery while true shared literal stores keep single-store semantics. The fix should include regression coverage for a literal `agents/main/sessions/sessions.json` configuration.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The issue discussion and #54716 describe an API-backed reproduction with a literal per-agent `cfg.session.store`, and current main still shows the same early-return control flow that would return only the default-agent store.

## Solution Assessment

Is this the best way to solve the issue? Yes, the linked direction is the narrowest maintainable fix: route literal per-agent store paths through existing all-agent discovery without changing semantics for genuinely shared literal store files.
