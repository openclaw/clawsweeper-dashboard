## Summary

Keep open. Current main still has the literal `session.store` path that reads one default-agent store and returns before all-agent discovery, while the linked closing PR #54716 remains open and unmerged.

## What This Changes

The report asks for `sessions.list` and the Control UI Sessions view to include non-main agent and channel sessions when multiple session stores are present.

## Best Possible Solution

Review and land #54716, or an equivalent current-main fix, so literal per-agent paths under `agents/<id>/sessions/sessions.json` use all-agent discovery while true shared literal stores keep single-store semantics. The fix should include regression coverage for a literal `agents/main/sessions/sessions.json` configuration.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Configure `session.store` to a literal per-agent path such as `.../agents/main/sessions/sessions.json` while a sibling agent store exists, then call `sessions.list`; current main's control flow reads only the literal default-agent store and returns before sibling discovery.

## Solution Assessment

Is this the best way to solve the issue? Yes. The linked direction is the narrowest maintainable fix: route literal paths that structurally point at an agent store through existing all-agent discovery, while preserving current behavior for genuinely shared literal store files.
