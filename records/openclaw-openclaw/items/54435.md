## Summary

Keep open. Current main still routes literal `session.store` configs through one default-agent store before sibling discovery, and the closing PR at https://github.com/openclaw/openclaw/pull/54716 remains open and conflicting rather than merged.

## What This Changes

The issue reports that `sessions.list` and the Control UI sessions dashboard only return the main session when multiple agent/channel session stores exist, especially with a literal per-agent `session.store` path.

## Best Possible Solution

Repair and land https://github.com/openclaw/openclaw/pull/54716, or an equivalent current-main fix, so literal agent-layout stores discover only same-root sibling stores while true shared literal stores remain single-store, with regression coverage for `agents/main/sessions/sessions.json`.

## Reproduction Assessment

Yes. Configure `session.store` to a literal per-agent path such as `.../agents/main/sessions/sessions.json` with a sibling agent store, then call `sessions.list`; current-main source inspection shows the loader reads only the literal default-agent store and returns before sibling discovery.

## Solution Assessment

Yes for the issue direction, but not by closing this issue now. The narrow maintainable fix is to port literal agent-layout handling into the active config/sessions loader and target resolver while preserving single-store behavior for non-agent-layout literal paths.
