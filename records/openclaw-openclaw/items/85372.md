## Summary

Keep this PR open for maintainer handling: the status-notice plumbing is useful, but the diff intentionally makes plan notices bypass `/verbose off`, which changes an existing user preference and needs either a compatibility fix or explicit product approval. The PR also has the protected `maintainer` label, so it should not be cleanup-closed.

## What This Changes

This PR adds a generic `isStatusNotice` reply payload marker and uses it to show the first agent plan update as a transient status notice while excluding that notice from TTS, commitment extraction, streaming final-content accounting, and reply-threading slot consumption.

## Best Possible Solution

Keep the status-notice marker and final-content/TTS exclusions, but preserve `/verbose off` suppression by default unless maintainers intentionally approve a documented opt-in or policy change for plan notices.

## Reproduction Assessment

## Solution Assessment

No as currently written: the status-notice abstraction is a reasonable implementation direction, but bypassing the existing verbose gate is not the narrowest compatible fix without maintainer approval and documentation.
