## Summary

Keep open: this is a focused gateway-status bug-fix PR, but the fallback is keyed to the CLI process environment and the default managed unit name, so it does not cover the linked report's normal shell-driven system-level unit workflow. It also needs after-fix real systemd proof before merge.

## What This Changes

The PR adds an `INVOCATION_ID`-gated fallback in `readSystemdServiceRuntime` that runs system-level `systemctl show` when user-systemd availability fails.

## Best Possible Solution

Make `status --deep` reconcile service summaries with the existing system-level gateway service scan, or probe both scopes using the discovered system unit name, without relying on the CLI process environment.

## Reproduction Assessment

Yes, at source level: the linked issue gives a concrete system-level unit plus `openclaw status --deep` workflow, and current main returns `unknown` when `systemctl --user status` is unavailable. I did not run a live systemd host in this read-only review.

## Solution Assessment

No. The PR should probe the actual system-scope OpenClaw unit or reuse the existing deep service-discovery path; checking the diagnostic process's `INVOCATION_ID` is not the narrow durable fix.
