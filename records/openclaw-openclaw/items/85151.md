## Summary

Keep open: the linked Linux status bug is valid, but this PR does not fix the normal `openclaw status --deep` path because it keys off the status process `INVOCATION_ID` and only falls back when user-systemd availability throws.

## What This Changes

The PR adds a `readSystemdServiceRuntime` fallback that runs system-level `systemctl show` when user-systemd availability fails and `INVOCATION_ID` is set.

## Best Possible Solution

Implement status-only system-scope detection that probes the documented system unit when the user unit is unavailable or missing, labels it as externally managed, and leaves install/restart ownership with the existing user-systemd lifecycle unless an explicit system-management path is added.

## Reproduction Assessment

Yes. The linked issue gives a concrete Ubuntu system-unit setup, and source inspection confirms current status uses the CLI environment and user-systemd probes before formatting missing user units as not installed/unavailable.

## Solution Assessment

No. The PR's `INVOCATION_ID` gate is not the narrow maintainable fix because it checks the status process environment and only runs after `assertSystemdAvailable` throws, rather than probing system scope from the actual status failure modes.
