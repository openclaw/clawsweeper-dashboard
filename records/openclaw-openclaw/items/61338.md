## Summary

Keep open. Current main and the latest release still exact-match skill OS metadata against raw Node platform strings, so a skill declaring `os: macos` remains ineligible on Darwin; the issue is source-reproducible and has no merged canonical fix.

## What This Changes

The issue asks OpenClaw skills eligibility to treat `metadata.openclaw.os: ["macos"]` as satisfied on macOS/Darwin hosts.

## Best Possible Solution

Add a shared, narrow OS normalizer that maps supported aliases such as `macos` to `darwin` before skill requirement evaluation, runtime eligibility, and installer filtering, then document and test the accepted values.

## Reproduction Assessment

Yes, at source level: `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })` falls through to `macos`, and `buildWorkspaceSkillStatus` feeds that into skills status. I did not run a live macOS CLI reproduction in this read-only Linux review.

## Solution Assessment

Yes. The best fix is shared OS canonicalization plus docs and regression tests, not a CLI-only display change or telling users to delete the metadata line.
