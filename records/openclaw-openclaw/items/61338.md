## Summary

Keep open. Current main and `v2026.5.18` still exact-match skill OS metadata against raw platform strings, so a skill declaring `os: macos` remains ineligible when the runtime platform is `darwin`; no canonical fix PR or duplicate issue was found. ([nodejs.org](https://nodejs.org/api/process.html?utm_source=openai))

## What This Changes

Fix `openclaw skills check` so skills declaring `metadata.openclaw.os: ["macos"]` are treated as eligible on macOS/Darwin hosts.

## Best Possible Solution

Add shared OS alias canonicalization for skill requirements, runtime eligibility, and installer filtering, then document accepted values and cover `macos` on `darwin` with focused regression tests.

## Reproduction Assessment

Yes, at source level: `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })` falls through to `macos`, and `buildWorkspaceSkillStatus` feeds that missing OS into skills status. I did not run live macOS CLI proof in this read-only Linux review.

## Solution Assessment

Yes. The maintainable fix is a shared platform normalizer used by requirement evaluation, runtime eligibility, and install filtering, not a CLI-only display change or a docs-only workaround.
