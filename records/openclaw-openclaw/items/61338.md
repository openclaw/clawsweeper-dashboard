## Summary

Keep open. Current main still compares skill OS requirements and installer/runtime platform lists as raw Node platform strings, so `macos` does not match `darwin`; the documented and tested path only covers canonical `darwin`, `linux`, and `win32`.

## What This Changes

The issue requests that `openclaw skills check` treat a skill declaring `metadata.openclaw.os: macos` as eligible when running on macOS/Darwin.

## Best Possible Solution

Add a shared platform-name canonicalization helper for skill OS requirements and installer filters, preserving canonical `darwin`, `linux`, and `win32` while accepting common aliases such as `macos`/`osx` and `windows`; then document those accepted spellings and cover the alias path with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })`, which follows current main and returns `missing.os: ["macos"]`; `openclaw skills check` reaches that evaluator through `buildWorkspaceSkillStatus`.

## Solution Assessment

Yes. A narrow shared normalizer is the best maintainable fix because the same OS comparison appears in requirement evaluation, runtime eligibility, install-option filtering, and docs; changing only the CLI output would leave behavior inconsistent.
