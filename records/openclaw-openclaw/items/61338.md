## Summary

Keep open. Current main still compares skill OS requirements against raw Node platform strings, so `macos` does not satisfy `darwin` in the skills status/check path; nearby runtime and installer OS filters use the same exact-match pattern.

## What This Changes

The issue asks OpenClaw to treat a skill with `openclaw.os: macos` as eligible on macOS/Darwin when running `openclaw skills check`.

## Best Possible Solution

Add shared platform-name canonicalization for skill OS requirements, runtime eligibility, and installer filters, then document accepted aliases while preserving canonical Node platform values.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })`, which current main evaluates as missing; `buildWorkspaceSkillStatus` reaches that evaluator for `openclaw skills check`.

## Solution Assessment

Yes. The narrow maintainable fix is a shared normalizer used by the requirement evaluator, runtime eligibility, and installer filtering rather than a formatter-only or skills-check-only special case.
