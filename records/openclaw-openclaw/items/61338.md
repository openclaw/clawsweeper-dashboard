## Summary

Keep open. Current main and v2026.5.19 still exact-match skill OS metadata against raw `process.platform`, so `macos` remains ineligible on macOS where Node reports `darwin`; related searches did not find a canonical fix PR or duplicate that owns the remaining work.

## What This Changes

Make `openclaw skills check` treat a skill declaring `metadata.openclaw.os: macos` as eligible on macOS/Darwin instead of reporting a missing OS requirement.

## Best Possible Solution

Add one shared OS canonicalizer for skill requirements, runtime eligibility, and installer filtering that treats `macos` as `darwin`, then update focused tests and public skills docs.

## Reproduction Assessment

Yes, at source level: `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })` falls through to `macos`, and skills status feeds that into the missing requirements display. I did not run live macOS CLI proof in this read-only Linux review.

## Solution Assessment

Yes. The maintainable fix is shared OS alias canonicalization across requirement evaluation, runtime eligibility, and install filtering, not a CLI-only display change or a docs-only workaround.
