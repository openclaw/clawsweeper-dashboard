## Summary

Keep open. Current main and the latest release still exact-match skill OS metadata against raw Node platform strings, so a skill declaring `os: macos` remains ineligible on macOS/Darwin in the skills check path.

## What This Changes

The issue reports that `openclaw skills check` treats a `macos-calendar` skill declaring `openclaw.os: macos` as missing requirements on a macOS/Darwin machine.

## Best Possible Solution

Add shared OS alias canonicalization for skill requirements, runtime eligibility, and installer filtering, then document accepted values and cover `macos` on `darwin` with focused regression tests.

## Reproduction Assessment

Yes, at source level: `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })` falls through to return `macos`, and `buildWorkspaceSkillStatus` feeds that missing requirement into `openclaw skills check`. I did not run live macOS verification in this read-only Linux review.

## Solution Assessment

Yes. The narrow maintainable fix is a shared platform normalizer used by requirement evaluation, runtime eligibility, and install filtering, not a CLI-only display change or a docs-only workaround.
