## Summary

Keep open. Current main still treats skill OS requirements as exact Node platform strings, so `macos` does not match `darwin` in the `openclaw skills check` path, and related runtime/installer filters have the same raw comparison shape.

## What This Changes

The issue asks for `metadata.openclaw.os: macos` skills to be considered eligible when OpenClaw is running on macOS/Darwin.

## Best Possible Solution

Add shared platform-name canonicalization for skill OS requirements, runtime eligibility, and installer filters, then document accepted aliases and preserve canonical `darwin`, `linux`, and `win32` behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })`, which current main evaluates as missing; `openclaw skills check` reaches that evaluator through `buildWorkspaceSkillStatus`.

## Solution Assessment

Yes. A narrow shared normalizer is the best maintainable fix because the same OS comparison appears in requirement evaluation, runtime eligibility, and install-option filtering.
