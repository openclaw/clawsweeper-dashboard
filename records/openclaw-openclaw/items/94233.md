## Summary

Keep open: this is a focused fix for a source-reproducible auth-provider fallback log-spam bug, and current main plus v2026.6.8 still emit each matching fallback decision directly. I found no blocking patch defect; the main remaining concern is maintainer acceptance of the operator-visible diagnostic coalescing tradeoff.

## What This Changes

The PR adds bounded in-memory coalescing for auth/auth_permanent `candidate_failed` and `skip_candidate` fallback decision warnings, plus Vitest coverage for suppression and scoping behavior.

## Best Possible Solution

Land this logger-layer coalescer after normal maintainer review accepts the diagnostic tradeoff and required CI passes, while keeping broader provider quarantine/status policy in https://github.com/openclaw/openclaw/issues/47910.

## Reproduction Assessment

Yes. Source inspection shows current main classifies auth failures into the fallback loop, calls the warn-level fallback decision logger for failed/skipped candidates, and the default skip cache remains opt-in rather than solving repeated logs.

## Solution Assessment

Yes. For the narrow log-spam bug, candidate-aware auth-class coalescing at the fallback decision logger is the smallest maintainable fix because it reduces duplicate diagnostics without changing fallback routing.
