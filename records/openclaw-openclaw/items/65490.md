## Summary

Keep open. Current main has useful fallback visibility, status, cron delivery preview, and local cron provider preflight work, but it still does not satisfy the core production-operations requests: primary model startup reachability is non-blocking and warning-only, the agent-visible diagnostics surface is not a structured selected-model/active-model/fallback/last-provider-error contract, and cron add does not dry-run resolve announce delivery before persistence.

## What This Changes

The issue requests production execution-discipline improvements for startup primary-model health checks, structured agent-readable runtime diagnostics, cron add validation/dry-run behavior, and clearer pre-context error propagation.

## Best Possible Solution

Keep this open as the execution-discipline tracker or split it into canonical follow-ups: a configurable primary-provider startup preflight/degraded-mode policy, a safe structured runtime diagnostics RPC or agent tool exposing selected model, active model, fallback state, and last provider error category, and cron add/create preflight that resolves announce delivery before persisting or offers an explicit dry-run path.

## Reproduction Assessment

Partially yes. I did not live-reproduce the original disabled-provider production incident, but current source and tests give a high-confidence reproduction of the missing gates: startup prewarm is non-blocking, diagnostics lacks the requested structured fields, and cron add does not dry-run resolve announce delivery before persistence.

## Solution Assessment

No, current main is not the complete solution. The existing fallback notices, /status text, health snapshots, and cron delivery previews are useful partial work, but the narrow maintainable path still needs maintainer-scoped startup policy, a structured diagnostics contract, and create-time cron delivery preflight semantics.
