## Summary

Keep open. Current main has useful fallback/status and cron-delivery visibility work, but the central production-operations gaps remain: startup provider reachability is still non-blocking and warning-only, diagnostics is not a compact structured model/fallback/last-error surface, and cron creation still lacks a destination-resolving preflight before persistence.

## What This Changes

The issue requests stronger production agent discipline across startup provider health checks, agent-visible runtime diagnostics, cron add validation/dry-run behavior, clearer pre-context errors, and fallback/model-state visibility.

## Best Possible Solution

Keep this open as the production execution-discipline tracker, or split it into focused follow-ups for startup provider reachability/degraded mode, a safe structured runtime diagnostics RPC or tool, and cron add delivery-target preflight/dry-run before persistence.

## Reproduction Assessment

Partially yes. I did not run the live disabled-provider production scenario, but current source and tests give a high-confidence reproduction of the missing gates: startup prewarm is explicitly non-blocking, diagnostics lacks the requested structured fields, and cron.add accepts announce delivery without resolving a target first.

## Solution Assessment

No. Current main is a useful partial solution for fallback visibility and cron delivery previews, but it is not the narrowest complete fix for this issue because the remaining work crosses startup policy, diagnostics API design, and cron create semantics.
