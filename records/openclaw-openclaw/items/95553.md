## Label Justifications

- `P1`: The report describes a real large-session workflow where every near-limit turn can spend about a minute in doomed preflight compaction before proceeding un-compacted.
- `impact:session-state`: The affected path is transcript/context compaction before the next agent turn and can leave session context near overflow.
- `impact:crash-loop`: Repeated preflight timeouts and session write-lock contention can keep the session effectively unavailable or stuck in a retry-like failure loop.

## Summary

Keep open: current main and the latest release still pass the reply-operation abort signal into required budget/preflight compaction, so an earlier reply lifecycle abort can still beat `agents.defaults.compaction.timeoutSeconds`; the open closing PRs are candidate fixes, not merged fixes.

## What This Changes

The issue asks OpenClaw to make budget/preflight compaction honor `agents.defaults.compaction.timeoutSeconds` or provide a preflight disable gate for slow-backend deployments.

## Best Possible Solution

Keep this issue as the canonical tracker, then land one narrow fix that excludes the normal reply lifecycle timeout from preflight compaction while preserving explicit cancellation and proving the slow-backend path.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper races that abort before the configured timeout. I did not live-run the reporter's slow vLLM/Qwen setup.

## Solution Assessment

Unclear until maintainers choose the cancellation contract. Honoring the existing compaction timeout is the right end state, but a new preflight config gate should be separate unless maintainers explicitly want that additional surface.
