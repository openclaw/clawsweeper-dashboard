## Summary

Keep open: current main and v2026.6.9 still pass the reply-operation abort signal into required budget/preflight compaction, and the open linked fix PRs have not merged yet.

## What This Changes

The issue reports that budget/preflight compaction is cancelled around 60 seconds by `replyOperation.abortSignal` instead of respecting `agents.defaults.compaction.timeoutSeconds`.

## Best Possible Solution

Keep this issue as the canonical tracker until a narrow preflight cancellation-contract fix merges and proves slow budget compaction can use the configured compaction timeout without losing deliberate cancellation behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 pass `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper lets that external abort race the configured timeout. I did not live-reproduce the reporter's exact slow-provider timing.

## Solution Assessment

Yes on the reported direction, but unclear on the final cancellation policy: using the existing compaction timeout for slow preflight work is narrower than adding a new config toggle, while explicit abort/restart/drain behavior needs maintainer approval.
