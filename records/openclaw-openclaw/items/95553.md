## Label Justifications

- `P1`: The issue reports an urgent agent workflow failure where large-session compaction can repeatedly stall for about a minute, hold session write locks, and still proceed un-compacted.
- `impact:session-state`: The affected path is session transcript/context compaction before the next agent turn and can leave persisted session state near overflow.
- `impact:crash-loop`: The report describes repeated timeout/failure behavior and write-lock contention that can make the affected agent workflow effectively unavailable.

## Summary

Keep open: current main and v2026.6.10 still pass the reply operation abort signal into required budget/preflight compaction, so a reply lifecycle timeout can still beat `agents.defaults.compaction.timeoutSeconds`; the open closing PRs are candidate fixes, not merged fixes.

## What This Changes

The issue requests that budget-triggered preflight compaction stop being capped by the reply-operation abort signal and either honor the configured compaction timeout or provide an opt-out gate.

## Best Possible Solution

Land one narrow preflight-cancellation fix that lets required compaction run under `compaction.timeoutSeconds` while preserving explicit cancellation, then close this issue from the merged PR with focused regression and runtime proof.

## Reproduction Assessment

Yes at source level: current main still forwards `params.replyOperation.abortSignal` into budget-triggered compaction, and the compaction safety wrapper races that external signal against the configured timeout. I did not run the reporter's slow vLLM/Qwen deployment in this read-only review.

## Solution Assessment

Unclear until maintainers choose the cancellation contract. The narrow end state is clear, but the open PRs make different tradeoffs around lifecycle timeout filtering and explicit cancellation, so this issue should stay open until one path lands or is rejected.
