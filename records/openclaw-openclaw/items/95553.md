## Label Justifications

- `P1`: The report describes repeated pre-turn compaction delays and lock contention in a real long-running backend workflow.
- `impact:crash-loop`: The affected path can repeatedly time out before every large turn and keep the session effectively unavailable.
- `impact:session-state`: The issue concerns transcript/context compaction state before the next agent turn and during overflow recovery.

## Summary

Keep open: current main and v2026.6.9 still pass the reply operation abort signal into required budget/preflight compaction, and the timeout wrapper lets that external abort beat `agents.defaults.compaction.timeoutSeconds`; adjacent timeout issues do not clearly own this exact preflight cancellation-scope bug.

## What This Changes

The issue asks OpenClaw to make budget/preflight compaction honor the configured compaction timeout, or provide a separate way to disable that preflight path.

## Best Possible Solution

Define one preflight compaction cancellation contract: normal slow budget compaction should get the configured compaction timeout, while explicit user abort, restart, shutdown, or gateway-drain cancellation still stops it.

## Reproduction Assessment

Yes at source level: if the reply operation aborts before `resolveCompactionTimeoutMs` expires, required budget/preflight compaction is canceled through the external abort race. I did not live-reproduce the reporter's exact 60s timeout in the vLLM/Qwen environment.

## Solution Assessment

Unclear pending maintainer policy. Honoring the existing compaction timeout for preflight looks like the right end state, but maintainers should choose which abort sources remain allowed to interrupt compaction before a repair PR changes cancellation semantics.
