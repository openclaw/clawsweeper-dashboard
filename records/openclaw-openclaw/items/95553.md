## Label Justifications

- `P1`: Affected large-session users can spend about a minute per near-limit turn in doomed preflight compaction before proceeding un-compacted.
- `impact:session-state`: The affected behavior is pre-turn transcript/context compaction for persisted agent sessions.
- `impact:crash-loop`: Repeated compaction timeouts and session write-lock contention can keep affected sessions stuck or effectively unavailable.

## Summary

Keep open: current main and v2026.6.9 still route required budget/preflight compaction through `replyOperation.abortSignal`, and the open closing PRs are candidate fixes rather than merged fixes.

## What This Changes

The issue reports that budget-triggered preflight compaction is prematurely canceled by the reply-operation abort path, so `agents.defaults.compaction.timeoutSeconds` cannot govern slow required compaction.

## Best Possible Solution

Keep this issue as the canonical tracker until a narrow linked PR lands that lets required preflight compaction use the configured compaction timeout while preserving explicit user stop and restart cancellation.

## Reproduction Assessment

Yes at source level: current main still forwards the reply operation abort signal into required preflight compaction, and the compaction safety wrapper allows that external abort to win before the configured timeout. I did not live-run the reporter's slow-backend setup.

## Solution Assessment

Unclear as an implementation choice: honoring the existing compaction timeout is the right end state, but maintainers still need to choose the cancellation contract and whether the optional config gate belongs in the same fix.
