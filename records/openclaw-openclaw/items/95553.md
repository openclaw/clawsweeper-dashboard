## Label Justifications

- `P1`: The report describes repeated pre-turn compaction delays and lock contention in a real long-running backend workflow.
- `impact:session-state`: The affected path is transcript/context compaction before the next agent turn and can leave session state un-compacted near overflow.
- `impact:crash-loop`: The issue can repeatedly consume the pre-turn timeout window and leave large sessions effectively stuck in recurring compaction failures.

## Summary

Keep open: current main and v2026.6.9 still pass the reply-operation abort signal into required budget/preflight compaction, and the timeout wrapper can let that external abort win before `agents.defaults.compaction.timeoutSeconds`; the related timeout-progress PR does not directly own this cancellation-scope bug.

## What This Changes

The issue asks OpenClaw to stop hard-capping budget/preflight compaction around the reply-operation abort window and make that path honor the configured compaction timeout or provide a safe opt-out.

## Best Possible Solution

Define one preflight compaction cancellation contract: slow budget compaction should get the configured compaction timeout, while explicit user abort, restart, shutdown, and gateway-drain paths still have a deliberate cancellation route.

## Reproduction Assessment

Yes at source level: current main still passes `replyOperation.abortSignal` into required budget/preflight compaction, and the timeout wrapper races that signal against the configured compaction timeout. I did not live-reproduce the reporter's exact vLLM/Qwen 60s timing in this read-only review.

## Solution Assessment

Unclear pending maintainer policy. Honoring the existing compaction timeout for normal slow preflight compaction looks narrower than adding a new config toggle, but maintainers should decide which abort sources remain allowed to interrupt compaction before code changes.
