## Summary

Keep open: current main and v2026.6.9 still wire required budget/preflight compaction to the reply-operation abort signal, and two open closing PRs already own candidate fixes that need maintainer review rather than a new cleanup close.

## What This Changes

The issue asks OpenClaw to stop hard-capping budget-triggered preflight compaction around the reply lifecycle timeout and make it honor existing compaction timeout behavior or provide a preflight opt-out.

## Best Possible Solution

Review and land one canonical fix that gives normal slow preflight compaction the configured compaction timeout while preserving explicit user stop, restart, shutdown, and drain cancellation.

## Reproduction Assessment

Yes at source level: current main passes the reply-operation abort signal into required budget/preflight compaction, and the safety wrapper lets that external abort race the configured timeout. I did not live-reproduce the reporter's exact slow-provider setup.

## Solution Assessment

Unclear pending maintainer policy. The best direction is a dedicated preflight cancellation contract, but maintainers still need to choose how explicit stop, restart, drain, shutdown, and lifecycle timeout should interact.
