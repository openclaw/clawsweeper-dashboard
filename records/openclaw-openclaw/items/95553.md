## Label Justifications

- `P1`: The report describes a real large-session workflow where every near-limit turn can spend about a minute in doomed preflight compaction before proceeding un-compacted.
- `impact:session-state`: The affected path is transcript/context compaction before the next agent turn and can leave session context near overflow.
- `impact:crash-loop`: Repeated preflight timeouts and session write-lock contention can make the session effectively unavailable or stuck in a retry-like failure loop.

## Summary

Keep open: current main and the latest release still pass the reply-operation abort signal into required budget/preflight compaction, so the configured compaction timeout can still be beaten by the reply lifecycle abort. The linked open PRs are useful candidate fixes, not merged implementation.

## What This Changes

The issue asks OpenClaw to stop budget-triggered preflight compaction from being hard-capped by the reply operation abort signal so `compaction.timeoutSeconds` can govern, with an optional preflight-disable config gate as an escape hatch.

## Best Possible Solution

Keep this issue as the canonical tracker and land one narrow PR that prevents reply lifecycle timeouts from capping required preflight compaction while preserving explicit stop/restart cancellation; treat a preflight-disable config gate as a separate maintainer-approved surface.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper races that caller abort before the configured compaction timeout. I did not live-run the reporter's slow vLLM/Qwen setup.

## Solution Assessment

Unclear until maintainers choose the cancellation contract. Honoring the existing compaction timeout at the preflight call boundary looks like the narrowest fix, while adding a new preflight config gate should remain a separate product decision unless maintainers sponsor that surface.
