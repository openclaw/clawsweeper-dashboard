## Summary

Keep open: current main and the latest release still route required budget/preflight compaction through `replyOperation.abortSignal` before the compaction timeout can govern it, and the open PRs that close this issue are still unmerged.

## What This Changes

The issue asks for budget-triggered preflight compaction to honor `agents.defaults.compaction.timeoutSeconds` instead of being capped by the reply-operation abort lifecycle.

## Best Possible Solution

Land one narrow fix that keeps required preflight compaction governed by the existing compaction timeout while preserving explicit cancellation, then close this issue and supersede duplicate candidate PRs.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction, and `compactWithSafetyTimeout` races that external abort before the configured compaction timeout. I did not perform a live slow-backend reproduction of the reporter's exact 60-second timing.

## Solution Assessment

Unclear which open PR is the best merge target, but the best fix shape is clear: change the preflight cancellation boundary rather than adding a new config toggle first. Maintainers should choose the exact explicit-cancel semantics before landing one candidate.
