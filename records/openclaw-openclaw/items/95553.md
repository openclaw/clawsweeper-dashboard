## Summary

Keep open: current main and v2026.6.9 still pass the reply-operation abort signal into required budget/preflight compaction, and there are multiple open candidate PRs that should be reviewed rather than closing the issue before a fix lands.

## What This Changes

The issue reports that required preflight compaction is canceled by the reply-operation abort path around 60 seconds instead of being governed by `agents.defaults.compaction.timeoutSeconds`.

## Best Possible Solution

Land or replace one canonical fix that lets required preflight compaction honor the existing compaction timeout while preserving explicit stop, restart, shutdown, and drain cancellation semantics.

## Reproduction Assessment

Yes, at source level: current main passes the reply-operation abort signal into required budget/preflight compaction, and the safety wrapper races that external abort before the configured timeout. I did not run the reporter's exact slow-provider setup.

## Solution Assessment

Yes for keeping this issue open: the narrow fix belongs at the preflight compaction cancellation contract, not as a new config toggle while the existing timeout is still bypassed. The best patch should exclude the normal reply lifecycle timeout but keep explicit cancellation semantics.
