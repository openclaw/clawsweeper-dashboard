## Label Justifications

- `P1`: Affected large-session turns can repeatedly spend about a minute in doomed required preflight compaction before proceeding near overflow.
- `impact:session-state`: The issue is about transcript/context compaction before the next agent turn and can leave session state near overflow.
- `impact:crash-loop`: Repeated timeout and write-lock contention can make an affected slow-backend session effectively stuck or unavailable.

## Summary

Keep open: current main and v2026.6.10 still pass the reply-operation abort signal into required budget/preflight compaction, so the configured compaction timeout can still be preempted; open linked PRs are candidate fixes, not merged fixes.

## What This Changes

The issue requests that budget/preflight compaction stop being capped by the reply-operation lifecycle timeout and either honor `agents.defaults.compaction.timeoutSeconds` or provide a preflight opt-out.

## Best Possible Solution

Land one narrow fix that excludes the normal reply lifecycle timeout from required preflight compaction while preserving explicit stop/restart cancellation, then decide separately whether the `preflight.enabled` config gate should be added.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction, and `compactWithSafetyTimeout` races that external abort before the configured timeout. I did not live-run the reporter's slow vLLM/Qwen setup.

## Solution Assessment

Unclear until maintainers choose the cancellation contract. The narrowest maintainable fix appears to be filtering or scoping the preflight abort signal so lifecycle `TimeoutError` does not own compaction timing, while preserving explicit cancellation; the config gate is a separate product choice.
