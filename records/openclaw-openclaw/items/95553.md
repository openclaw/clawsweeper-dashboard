## Label Justifications

- `P1`: Large-session users can spend about a minute per near-limit turn in doomed preflight compaction before proceeding un-compacted or hitting overflow recovery.
- `impact:session-state`: The affected path is transcript/context compaction before the next agent turn and can leave session context near overflow.
- `impact:crash-loop`: Repeated preflight timeouts and session write-lock contention can make affected sessions effectively unavailable or stuck in a retry-like failure loop.

## Summary

Keep open: current main and the latest release still pass the reply-operation abort signal into required budget/preflight compaction, so `agents.defaults.compaction.timeoutSeconds` can still be beaten by the reply lifecycle timeout. The open closing PRs are implementation candidates, not merged fixes.

## What This Changes

Fix budget/preflight compaction so slow compactions are not hard-capped by the reply-operation abort signal and either honor the existing compaction timeout or get an explicit opt-out.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land one narrow cancellation-contract fix such as https://github.com/openclaw/openclaw/pull/95590, then decide separately whether a preflight opt-out config is still needed.

## Reproduction Assessment

Yes at source level: current main passes the reply operation abort signal into required budget/preflight compaction, and the safety wrapper races that external abort before the configured compaction timeout. I did not live-run the reporter's slow vLLM/Qwen setup.

## Solution Assessment

Unclear until maintainers choose the cancellation contract. The best direction is to let `compaction.timeoutSeconds` own slow required preflight compaction while preserving explicit user/restart cancellation; a new config gate should remain separate unless maintainers want that added surface.
