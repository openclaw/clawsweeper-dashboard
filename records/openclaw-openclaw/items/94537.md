## Summary

Keep this PR open, but it is not merge-ready: the hardening direction is useful for the memory-core data-loss cluster, yet the patch can delete valid recall rows on non-missing filesystem errors, can loosen private memory file permissions, and still lacks real OpenClaw runtime proof.

## What This Changes

The branch changes memory-core dreaming markdown writes to temp-file rename and adds repair-time removal/logging for recall entries whose source daily memory file is missing, with targeted tests.

## Best Possible Solution

Keep the focused hardening direction, but route the write paths through the existing mode-preserving atomic replacement helper, prune dangling refs only after confirmed missing regular files, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: the changed repair loop treats all fs.access failures as missing, and the changed temp-write paths rename fresh files over existing memory files. The linked mass-deletion issue still lacks a deterministic current-main reproduction path.

## Solution Assessment

No. Atomic writes and dangling-ref cleanup are plausible pieces, but the best implementation should reuse the existing mode-preserving atomic helper and existing regular-file/ENOENT source-check semantics instead of adding a broader deletion path.
