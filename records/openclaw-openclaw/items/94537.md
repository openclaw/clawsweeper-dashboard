## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve memory artifact permissions:** `extensions/memory-core/src/dreaming-markdown.ts:89-97`
  - body: These `replaceFileAtomic` calls omit `dirMode`, and the `@openclaw/fs-safe@0.3.0` helper chmods the destination parent directory to `0700` by default while new outputs use `0600`. Current `fs.mkdir`/`fs.writeFile` follows umask, so an upgrade can break group-readable memory workspaces or external readers of `memory/` and `memory/dreaming/*`.
  - confidence: 0.9
- **[P1] Keep the SDK facade additive and complete:** `src/plugin-sdk/memory-core-engine-runtime.ts:84-100`
  - body: The implementation can now emit `recall-store-dangling-ref`, but the public `ShortTermAuditIssue.code` union still omits that literal, and the facade also turns existing/new result fields into required properties. Keep added fields optional where needed and expose the new issue literal so typed plugin and doctor consumers do not get a partial or breaking SDK contract.
  - confidence: 0.92
- **[P2] Use the same dangling-entry predicate in audit and repair:** `extensions/memory-core/src/short-term-promotion.ts:2608-2619`
  - body: The new audit path counts every normalized `source: "memory"` entry, but repair only prunes entries whose path passes `isShortTermMemoryPath`. A legacy/raw store entry like `MEMORY.md` can therefore produce a fixable dangling-ref issue that `openclaw memory status --fix` will not clear; use the same predicate in both paths and cover the re-audit case.
  - confidence: 0.86
