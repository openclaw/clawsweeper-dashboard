## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve memory artifact permissions:** `extensions/memory-core/src/dreaming-markdown.ts:94-100`
  - body: These `replaceFileAtomic` calls omit `dirMode`, and only the inline path preserves an existing file mode. The fs-safe helper defaults the parent directory to `0700` and chmods the destination to the resolved mode after rename, while current `fs.mkdir`/`fs.writeFile` follows the process umask and preserves existing files. Existing shared `memory/` or `memory/dreaming/` dirs and reports can become private on upgrade, so preserve parent and existing file modes or make the private-mode change explicit with maintainer-approved tests/docs.
  - confidence: 0.9
- **[P2] Expose the new dangling audit code in the SDK type:** `src/plugin-sdk/memory-core-engine-runtime.ts:87`
  - body: The runtime now emits `recall-store-dangling-ref` and the facade adds `danglingRefCount`, but `ShortTermAuditIssue.code` in the public SDK facade still omits that literal. TypeScript consumers of `auditShortTermPromotionArtifacts` cannot handle the returned issue code without casts, so add the literal and a focused facade type test.
  - confidence: 0.88
- **[P2] Use the same file check for dangling repair:** `extensions/memory-core/src/short-term-promotion.ts:2802`
  - body: Audit checks referenced sources with `stat().isFile()`, but repair uses `fs.access`, so a directory or other non-file at `memory/YYYY-MM-DD.md` is reported as a fixable dangling ref and then left in place by repair. Reuse the helper or match its file-only semantics in repair, with a regression for a non-file source path.
  - confidence: 0.82
