## Summary

Keep open for maintainer product review: the report asks for a new provider- or agent-level retry knob, while current main already has cron retry and model failover machinery and the concrete MiniMax overload symptom is being tracked separately.

## What This Changes

The issue requests a provider- or agent-level `api_max_retries` setting for LLM API transient failures, motivated by MiniMax overloads during concurrent cron jobs.

## Best Possible Solution

Decide whether this should remain covered by `cron.retry` plus model failover, land the narrower transient-retry PR, or add a documented provider/agent retry policy limited to classified transient errors.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested provider/agent `api_max_retries` config is absent, while the concrete MiniMax overload recovery symptom is already represented by the open canonical issue and PR.

## Solution Assessment

Unclear. A raw `api_max_retries` knob may not be the best fit because OpenClaw already has classified cron retry and model failover semantics; maintainers should decide whether a new public retry budget belongs at provider, agent, or cron scope.
