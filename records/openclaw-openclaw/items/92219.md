## Summary

This PR is still necessary because current `main` retains the reported raw-config doctor path, but the unchanged PR head does not fix the consuming adapter: it resolves config for adapter discovery while invoking the adapter with the original unresolved config. The fallback can also repeat the same failing read, and the supplied evidence contains no real after-fix doctor invocation.

## What This Changes

The PR resolves channel-scoped SecretRefs before doctor preview adapter discovery and falls back to the original config when resolution fails.

## Best Possible Solution

Resolve configured channel secrets once at the generic preview dispatcher, pass the effective configuration to both adapter discovery and every preview adapter invocation, and convert resolution failure into a focused non-fatal diagnostic without exposing unresolved values to secret-reading adapters.

## Reproduction Assessment

Yes at source level. The linked issue supplies a concrete pre-fix command and error, while the adapter contract and unchanged PR data flow show that the raw config still reaches the failing consumer.

## Solution Assessment

No. The generic dispatcher is the correct ownership boundary, but this implementation resolves only for discovery and retains a fallback that can reproduce the whole-doctor abort.
