## Label Justifications

- `P2`: This is a bounded but reproducible diagnostic-command failure affecting users with SecretRef-backed channel configuration.

## Summary

This PR targets the correct generic doctor boundary, but it does not fix the reported failure: the resolved config is used for adapter discovery while each preview adapter still receives the original unresolved config. The resolution-error fallback also returns to the same unsafe raw-config path, and the supplied validation is test-harness output rather than real after-fix behavior.

## What This Changes

The PR resolves configured channel SecretRefs before doctor preview adapter discovery and falls back to the original config when resolution fails.

## Best Possible Solution

Resolve channel secrets once at the generic preview dispatcher, pass the resulting effective config into every preview adapter, and convert resolution failure into a focused non-crashing doctor warning rather than invoking secret-reading adapters with raw config.

## Reproduction Assessment

Yes at source level. The linked issue provides a concrete real pre-fix command and error, while the branch data flow proves that the same raw config still reaches the failing adapter; a real current-main or after-fix run was not performed in this review.

## Solution Assessment

No. Resolving secrets at the generic dispatcher is the narrowest maintainable direction, but this implementation wires the resolved value to adapter discovery instead of the adapter consumer and retains an unsafe fallback.
