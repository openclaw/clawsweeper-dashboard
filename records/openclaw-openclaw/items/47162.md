## Summary

Keep open: the PR addresses a real scanner blind spot, but the current patch is still not merge-ready because it leaves shell entrypoint bypasses, unscoped language rules, and a Python command-exec bypass, and it lacks real behavior proof for an external security-sensitive change.

## What This Changes

The PR extends the skill scanner to scan `.py`, `.sh`, and `.bash` files, adds Python and shell detection rules, and adds focused scanner tests.

## Best Possible Solution

Land one consolidated scanner-hardening patch that covers Python and shell skill scripts, handles extensionless shebang entrypoints, scopes rules by language, catches variable command payloads, and includes regression coverage for bypasses and false positives.

## Reproduction Assessment

Yes. Source inspection shows current main scans only JS/TS-family extensions, and the PR head still leaves source/path patterns that reproduce the bypass and false-positive concerns without needing a full runtime setup.

## Solution Assessment

No. The scanner is the right surface, but this branch should be revised or consolidated with the overlapping scanner-hardening PRs before merge.

  - body: Only `dynamic-code-execution-python` is extension-gated.   - confidence: 0.9

  - body: Adding `.sh` and `.bash` leaves `isScannable()` dependent on `path.extname()`, so common shebang files such as `bin/install` or `postinstall` remain skipped by directory and forced-file scans. Detect shell shebangs or declared entrypoints so the new shell rules cover real package scripts.
  - confidence: 0.91

  - body: The `os.system`/`os.popen`/`os.exec*` branch only matches quoted first arguments, so `cmd = ...; os.system(cmd)` bypasses a critical command-exec finding. Match non-literal payloads or add a broader source rule for these calls.
  - confidence: 0.88
