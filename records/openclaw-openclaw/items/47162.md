## Summary

Keep open: current main still lacks Python/shell skill scanning, but this external security PR is conflicting, lacks real behavior proof, and has blocking scanner accuracy findings that need contributor and secops revision before merge.

## What This Changes

The PR extends the skill scanner to `.py`, `.sh`, and `.bash` files, adds Python and shell detection rules, and expands scanner tests.

## Best Possible Solution

Rebase and revise this into a secops-reviewed scanner update that scopes rules by language, covers extensionless shebang entrypoints or declared scripts, detects non-literal Python command execution, and includes focused tests plus real scan proof.

## Reproduction Assessment

Yes, for source-level review findings: current main is JS/TS-extension-only, and the PR head still keeps shell discovery extension-based while missing variable Python command payloads. I did not run tests because this was a read-only review.

## Solution Assessment

No. Extending the scanner is the right surface, but this branch should be revised so the new rules are language-scoped, cover shebang entrypoints, catch non-literal Python execution, and land after security-owner review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope language rules to matching files:** `src/security/skill-scanner.ts:182-186`
  - body: Most new Python and shell rules run on any scanned JS/TS file because only `dynamic-code-execution-python` is extension-gated, and the new source rules have no file-extension guard. Policy strings or comments in existing scanned files can therefore emit Python/shell findings and critical install blocks; add language guards and false-positive coverage.
  - confidence: 0.9
- **[P2] Scan extensionless shell entrypoints:** `src/security/skill-scanner.ts:48-50`
  - body: Adding only `.sh` and `.bash` keeps discovery dependent on `path.extname()`, so common shebang entrypoints such as `bin/install` or `postinstall` remain skipped by directory and forced-file scans. Detect shell shebangs or declared entrypoints so the new shell rules cover real scripts.
  - confidence: 0.91
- **[P2] Match non-literal Python command payloads:** `src/security/skill-scanner.ts:186`
  - body: The `os.system`/`os.popen`/`os.exec*` branch only matches quoted first arguments, so `cmd = ...; os.system(cmd)` bypasses the critical command-exec finding. Match non-literal payloads or add a broader source rule for these calls.
  - confidence: 0.88
