## Summary

Keep open. Current main still has the Python/shell coverage gap this PR targets, so the work is not obsolete, but the PR head is not merge-ready because the new scanner coverage still leaves extensionless shell entrypoints and variable-based Python command execution unscanned. Earlier review feedback was partly addressed, but the later blocker findings still apply to security-sensitive install/audit behavior.

## What This Changes

The PR extends the skill scanner to `.py`, `.sh`, and `.bash` files, adds Python and shell detection rules plus line-rule extension guards, and adds focused scanner tests.

## Best Possible Solution

Revise this PR, or replace it with a focused scanner-hardening branch, so Python and shell coverage includes extension-based files plus extensionless shebang or declared entrypoints, Python command-exec matching covers non-literal `os.*` payloads without reintroducing noisy subprocess false positives, related scanner PRs are reconciled, and release notes/tests cover the final behavior.

## Reproduction Assessment

Yes. Source and diff inspection give a high-confidence reproduction path: current main excludes `.py`, `.sh`, and `.bash`, while the PR head still skips extensionless files through `path.extname()` and does not match `cmd = ...; os.system(cmd)`.

## Solution Assessment

No. The direction is right, but this is not the best complete fix until it handles extensionless shell entrypoints and non-literal Python command payloads with regression tests and a changelog entry.
