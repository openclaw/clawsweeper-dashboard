## Summary

Keep open: current main still has the Python/shell scanner blind spot, but this PR head has blocking scanner-policy correctness gaps, is currently conflicting, and lacks required real behavior proof.

## What This Changes

The PR extends `src/security/skill-scanner.ts` to scan `.py`, `.sh`, and `.bash` files, adds Python and shell regex rules, and adds focused scanner tests.

## Best Possible Solution

Rebase and consolidate the scanner hardening so Python and shell rules are language-scoped, extensionless shell entrypoints are covered, variable Python command execution is detected, and secops can review the final policy with focused regression tests.

## Reproduction Assessment

Yes, for source-level review findings: current main still scans only JS/TS-family extensions, and the PR head shows extensionless shell and variable-payload Python gaps. I did not run tests because this was a read-only review.

## Solution Assessment

No. Extending the scanner is the right surface, but this branch should be revised so the new rules are language-scoped, cover shebang entrypoints, catch non-literal Python execution, and land after security-owner review.
