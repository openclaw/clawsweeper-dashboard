## Summary

Keep open: current main still has the Python/shell scanner blind spot, and this PR is a useful direction, but the proposed scanner rules still have blocking security-quality issues before merge.

## What This Changes

The PR adds `.py`, `.sh`, and `.bash` scanning, Python and shell detection rules, line-rule extension filters, and focused skill-scanner tests.

## Best Possible Solution

Land one consolidated scanner-hardening patch that covers Python and shell files, extensionless shebang entrypoints, language-scoped rules, non-literal Python command payloads, regression tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main scans only JS/TS-family extensions, and the PR diff shows the proposed coverage plus remaining extensionless-shell and language-scoping gaps.

## Solution Assessment

No. The direction is right, but this patch should be revised or replaced by a canonical scanner-hardening patch that avoids new false positives and closes the remaining bypasses.

  - body: Most new Python and shell rules have no extension guard, and the Python source rules cannot be scoped by the current `SourceRule` type. Detect shell shebangs or declared entrypoints so the new shell rules cover real package scripts.

  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff improves scanner coverage but still introduces scanner policy gaps that can miss malicious scripts or block benign installs.

Concerns:

- **[high] Extensionless shell files bypass new rules:** `src/security/skill-scanner.ts:48`
  - body: The scanner remains extension-gated, so shebang shell entrypoints without `.sh` or `.bash` are skipped by install-time and deep-audit scans after this patch.
  - confidence: 0.91
- **[high] Python variable command payloads bypass detection:** `src/security/skill-scanner.ts:186`
  - body: The new Python execution rule only matches quoted `os.*` payloads, allowing variable-based command execution such as `os.system(cmd)` to avoid the critical finding.
  - confidence: 0.88
- **[medium] Unscoped language rules can block unrelated code:** `src/security/skill-scanner.ts:183`
  - body: Most new Python and shell rules run against JS/TS files too, so policy strings or comments in existing scanned files can produce language-specific findings and fail closed during installs.
  - confidence: 0.9
