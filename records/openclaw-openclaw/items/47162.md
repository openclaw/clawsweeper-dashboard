## Summary

Keep open: current main still has the Python/shell scanner blind spot, but the latest PR patch needs revision because it leaves security bypasses, can create language-rule false positives on existing JS/TS inputs, and overlaps other open scanner-hardening work.

## What This Changes

The PR extends `src/security/skill-scanner.ts` to scan `.py`, `.sh`, and `.bash` files, adds Python/shell detection rules, and adds focused scanner tests.

## Best Possible Solution

Land one consolidated scanner-hardening patch that covers Python and shell scripts, handles extensionless shebang entrypoints, scopes rules by language, matches non-literal command payloads, and includes regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main scans only JS/TS-family extensions, and the PR source shows the proposed shell coverage remains extension-gated while several new language rules are not scoped to matching file types.

## Solution Assessment

No. The direction is right, but this patch should be revised or replaced by a canonical scanner-hardening patch that closes the remaining bypasses and avoids new critical false positives before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope language-specific rules to matching files:** `src/security/skill-scanner.ts:182-186`
  - body: Most new Python and shell rules have no `includeExtensions` guard, and the new Python source rules have no language guard at all. Existing `.js`/`.ts` scan inputs that mention `os.system("...")`, `import pickle`, or `printenv | curl` in code, strings, or comments can now emit Python/shell findings and block installs, so scope these rules to their languages and add regression coverage.
  - confidence: 0.9
- **[P2] Scan extensionless shell entrypoints:** `src/security/skill-scanner.ts:48-50`
  - body: Adding `.sh` and `.bash` still leaves `isScannable()` dependent on `path.extname()`, so common shebang entrypoints such as `bin/install` or `postinstall` remain unscanned even if they contain `curl | bash` or env exfiltration. Detect shell shebangs or declared entrypoints so the new shell rules cover real package scripts.
  - confidence: 0.91
- **[P2] Match variable Python command payloads:** `src/security/skill-scanner.ts:186`
  - body: The `os.system`/`os.popen`/`os.exec*` branch only matches quoted first arguments, so `cmd = ...; os.system(cmd)` avoids the critical finding while executing the same dangerous path. Match non-literal payloads or add a broader source rule for these command-exec calls.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `CHANGELOG.md:42`
  - body: This changes user-visible install/audit security behavior, but `CHANGELOG.md` is untouched. Add a single active-version fix/security bullet so users can see that Python and shell skill scanning behavior changed.
  - confidence: 0.86

## Security Review

Summary: The diff improves coverage but still leaves scanner bypasses and can introduce blocking false positives on existing scanned files.

Concerns:

- **[high] Extensionless shell files bypass new rules:** `src/security/skill-scanner.ts:48`
  - body: The scanner remains extension-gated, so shebang shell entrypoints without `.sh` or `.bash` are skipped by install-time and deep-audit scans after this patch.
  - confidence: 0.91

  - body: Most new Python and shell rules run against JS/TS files too, so policy strings or comments in existing scanned files can produce new language-specific findings and fail closed during installs.
