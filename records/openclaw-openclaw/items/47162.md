## What This Changes

The PR extends `src/security/skill-scanner.ts` to scan `.py`, `.sh`, and `.bash` files, adds Python and shell detection heuristics, and expands scanner tests.

## Best Possible Solution

Land a security-owner-reviewed scanner update that scopes rules per language, covers extensionless shell entrypoints, catches variable Python command execution, and includes focused tests plus redacted real scan proof.

## Reproduction Assessment

Yes for the PR-review findings: source inspection of the PR head shows extension-only shell discovery, unscoped language rules, and literal-only `os.*` matching. I did not run tests because this was a read-only review.

## Solution Assessment

No. Extending the scanner is the right surface, but this branch should be revised to handle extensionless entrypoints, scope language rules, catch non-literal Python command payloads, and land after security-owner review.

Overall confidence: 0.9

Full review comments:

- **[P2] Scope language rules to matching files:** `src/security/skill-scanner.ts:182-186`
  - body: Most new Python and shell rules run on any scanned JS/TS file because only `dynamic-code-execution-python` is extension-gated.   - confidence: 0.9
- **[P2] Scan extensionless shell entrypoints:** `src/security/skill-scanner.ts:48-50`
  - body: Adding only `.sh` and `.bash` keeps discovery dependent on `path.extname()`, so common shebang entrypoints such as `bin/install` or `postinstall` remain skipped by directory and forced-file scans. Detect shell shebangs or declared entrypoints so the new shell rules cover real scripts.
  - confidence: 0.91
- **[P2] Match non-literal Python command payloads:** `src/security/skill-scanner.ts:186`
  - body: The `os.system`/`os.popen`/`os.exec*` branch only matches quoted first arguments, so `cmd = ...; os.system(cmd)` bypasses the critical command-exec finding. Match non-literal payloads or add a broader source rule for these calls.
  - confidence: 0.88
