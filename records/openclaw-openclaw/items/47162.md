## Summary

Keep open: current main still has the Python/shell scanner blind spot, but this PR is not mergeable yet because the new language-specific rules are incompletely scoped and still leave extensionless shell plus variable Python command-exec bypasses.

## What This Changes

The PR expands the dangerous-code skill scanner to scan `.py`, `.sh`, and `.bash` files, adds Python and shell detection rules, and adds focused scanner tests.

## Best Possible Solution

Land a consolidated scanner-hardening patch that expands script coverage, handles shebang or declared entrypoint files, scopes rules by language, and documents release-visible behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main skips `.py`, `.sh`, and `.bash` through `isScannable()`, while the PR still skips extensionless shebang files and misses `cmd = ...; os.system(cmd)` through the changed regex.

## Solution Assessment

No. The direction is right, but the patch should be revised before merge to scope language-specific rules, close the remaining bypasses, add regression tests, and include a changelog entry.

  - body: Most new Python and shell rules have no `includeExtensions` guard, so they also run on the existing JS/TS scanner surface. A TypeScript plugin containing a comment or string such as `os.system("...")` or `printenv | curl` can now emit critical language-specific findings and block install; scope these rules to `.py`, `.sh`, and `.bash` or add an equivalent language check with regression tests.
  - confidence: 0.86

The scanner still relies on `path.extname()` for directory walking and forced includes, so common shebang entrypoints such as `bin/install` or `postinstall` remain unscanned even if they contain the new shell attack patterns.
  - confidence: 0.93

  - body: The `os.system`/`os.popen`/`os.exec*` arm only matches quoted first arguments. A script can build `cmd` separately and call `os.system(cmd)`, which is the same dangerous behavior but avoids the new critical finding.
  - confidence: 0.88

  - body: This is a user-facing security scanner fix, but the PR only changes scanner code and tests. Repo policy requires an active-version changelog entry so users can see the new install/audit hardening.
