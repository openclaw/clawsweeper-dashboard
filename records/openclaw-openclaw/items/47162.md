## Summary

Keep open. Current main still lacks Python and shell skill scanning, and this PR addresses a real security blind spot with tests and author follow-up on earlier review comments, but the branch still has install-blocking scanner accuracy and bypass issues plus missing real behavior proof.

## What This Changes

The PR adds `.py`, `.sh`, and `.bash` skill-scanner coverage with Python and shell detection rules plus focused tests in `src/security/skill-scanner*`.

## Best Possible Solution

Land a consolidated scanner-hardening patch that covers Python and shell skill scripts, extensionless shebang entrypoints, language-scoped rules, non-literal Python command execution, and regression tests for bypasses and false positives.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main only scans JS/TS-family extensions, while the PR head still shows extension-gated shell coverage and unscoped language rules that can be reasoned about directly without writing files.

## Solution Assessment

No. Extending the scanner is the right surface, but this branch should be revised or consolidated so the new blocking rules are language-scoped, cover extensionless shell entrypoints, and catch variable Python command payloads.

  - body: Only `dynamic-code-execution-python` is extension-gated.   - confidence: 0.9

  - confidence: 0.91

  - body: The `os.system`/`os.popen`/`os.exec*` branch only matches quoted first arguments, so `cmd = ...; os.system(cmd)` bypasses the critical command-exec finding. Match non-literal payloads or add a broader source rule for these calls.
  - confidence: 0.88
