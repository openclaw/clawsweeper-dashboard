## Summary

Keep open: current main still has the JS/TS-only scanner blind spot, but this external security PR needs revision before merge because the PR head leaves bypass and false-positive risks and has no real behavior proof.

## What This Changes

The branch expands `src/security/skill-scanner.ts` and its tests to scan `.py`, `.sh`, and `.bash` files and add Python/shell security heuristics.

## Best Possible Solution

Land a security-owner-reviewed scanner update that scopes language rules by file type or shebang, covers extensionless shell entrypoints and variable Python command execution, and includes focused tests plus redacted real scan proof.

## Reproduction Assessment

Yes. Source inspection gives high-confidence focused `scanSource` cases for the PR-review findings: JS/TS files can hit unscoped language rules, extensionless shebang scripts remain skipped, and `os.system(cmd)` bypasses the new Python command rule.

## Solution Assessment

No. Extending the scanner is the right surface, but this branch should be revised for language scoping, extensionless shell coverage, variable Python command payloads, and security-owner review before merge.

Full review comments:

- **[P2] Scope language rules to matching files:** `src/security/skill-scanner.ts:182-186`
  - body: Most new Python and shell rules run on every scanned JS/TS file because only `dynamic-code-execution-python` uses `includeExtensions`, and Python source rules have no file-type guard.   - confidence: 0.9
- **[P2] Scan extensionless shell entrypoints:** `src/security/skill-scanner.ts:48-50`
  - body: Adding only `.sh` and `.bash` keeps discovery tied to `path.extname()`, so shebang entrypoints such as `bin/install` or `postinstall` remain skipped by directory and forced-file scans. Detect shell shebangs or declared entrypoints before relying on the new shell rules.
  - confidence: 0.91
- **[P2] Match non-literal Python command payloads:** `src/security/skill-scanner.ts:186`
  - body: `dangerous-exec-python` only reports `os.system`/`os.popen`/`os.exec*` when the first argument starts with a quote, so `cmd = ...; os.system(cmd)` bypasses the critical command-exec finding. Match variable payloads or add a broader source rule for these APIs.
  - confidence: 0.88
