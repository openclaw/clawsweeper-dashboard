## Summary

Keep open: the real repro is useful, but the patch applies the Codex OAuth fallback to every generic `openai` auth lookup, which conflicts with the current direct OpenAI API-key boundary and needs a narrower auth decision before merge.

## What This Changes

The branch broadens auth-profile ordering so `openai` lookups can select `openai-codex` OAuth/token profiles, intended to let OpenAI memory embeddings reuse Codex OAuth credentials.

## Best Possible Solution

Add an explicit, tested embedding-specific Codex OAuth bridge only for the approved native OpenAI embedding route, while preserving generic direct `openai` API-key auth behavior and documenting the supported boundary.

## Reproduction Assessment

Yes, source inspection shows current main only expands profile compatibility for `openai-codex` lookups, while memory embeddings request auth for `openai`; the PR body also includes before/after terminal proof from a real setup.

## Solution Assessment

No. The proposed implementation fixes the observed embedding lookup by changing the shared OpenAI auth contract; a narrower embedding-specific path is the safer maintainable solution.
