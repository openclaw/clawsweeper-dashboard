## Summary

Keep open: the auth fallback is plausibly useful, but this PR is not merge-ready because the new tests are not isolated, current docs still define embeddings as API-key-only, and the real behavior proof does not match the current head.

## What This Changes

The branch adds a host-gated memory embedding auth fallback from `openai` to `openai-codex` OAuth for native `api.openai.com` requests and adds regression tests.

## Best Possible Solution

Land this only if maintainers accept the embedding auth contract change, with the host-gated fallback kept narrow, docs updated, mocks reset, and current-head real proof showing success without `OPENAI_API_KEY` plus no fallback for custom endpoints.

## Reproduction Assessment

Yes for the current limitation: source inspection shows current main asks auth for `openai` only in the memory embedding client, while the auth-profile ordering code does not expand `openai` lookups back to `openai-codex`. I did not run a live credential flow in this read-only review.

## Solution Assessment

Mostly yes if maintainers approve the contract change: an embedding-client fallback guarded by the native OpenAI host is safer than broad generic auth-order expansion. It is not best as-is until docs, test isolation, and current-head proof are fixed.
