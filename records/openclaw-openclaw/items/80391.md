## Summary

Keep open: source inspection confirms the Crabbox wrapper sanity probes can still hang on current main, but this PR is not ready because it lacks after-fix real behavior proof and mixes the wrapper fix with release-verifier behavior that needs to be split or explicitly reviewed.

## What This Changes

The branch adds a 5-second SIGKILL timeout to Crabbox wrapper sanity probes and also changes release/postpublish verification timeout, import allowlist, and size-limit test expectations.

## Best Possible Solution

Land a narrow Crabbox wrapper timeout patch with redacted terminal or log proof, and handle any remaining release-verifier allowlist or timeout changes in a separate release-owned review if still needed.

## Reproduction Assessment

Yes, at source level: select a Crabbox binary that never exits for `--version` or `run --help`; current main calls those probes with unbounded `spawnSync`, so the wrapper blocks before dispatching the user command. I did not run a full fake-binary wrapper repro in this read-only review.

## Solution Assessment

No, not as submitted: the wrapper timeout itself is a narrow maintainable fix, but changing shared release/postpublish verifier behavior in the same PR is not the safest way to solve the Crabbox hang. The safer path is to split or justify those release changes and add real timeout proof.
