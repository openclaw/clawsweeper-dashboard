## Summary

Keep this PR open, but it is not merge-ready. Current main still keeps Slack thread participation in process-local memory only, so the restart-survival bug remains valid; the branch also overlaps the open Slack persistence PR #33845 and the merged #74190 plugin state-store direction, and the proposed patch has concrete P2 correctness issues.

## What This Changes

The PR adds a JSON-backed Slack thread-participation store, reads and writes it from Slack prepare/dispatch, adds thread_ts fallback and ambiguity handling, expands mention-gating types, adds Slack tests, and includes a Dockerfile runtime CLI assertion test.

## Best Possible Solution

Land one owner-approved Slack restart-survival implementation that preserves the in-memory hot path, stores account/channel/thread participation durably through the merged plugin state-store API or another selected store, keeps `thread.requireExplicitMention` semantics intact, and removes or live-proves any Slack `conversations.replies` fallback before shipping.

## Reproduction Assessment

Yes. Record Slack participation in a thread, restart the gateway or clear the process-global cache, then send a non-mention reply in a `requireMention` Slack channel; current main has no durable participation state for prepare to classify it as `bot_thread_participant`.

## Solution Assessment

No. The PR targets a real bug, but it is not the best merge shape because it adds a custom JSON persistence layer after the plugin state-store foundation landed, exposes a `threadContinuation` parameter that does not affect decisions, and adds an unverified Slack fallback that likely cannot recover the unknown parent timestamp.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Wire threadContinuation into mention decisions:** `src/channels/mention-gating.ts:14`
  - body: The new `threadContinuation` parameter is documented as contributing to `effectiveWasMentioned`, but both compatibility resolver entry points ignore it and the core decision still uses only explicit mentions, implicit mention kinds, and command bypass. A caller passing `threadContinuation: true` will still be skipped under `requireMention`, so either wire it into the decision or remove the field.
  - confidence: 0.89
- **[P2] Remove the reply-ts conversations.replies fallback:** `extensions/slack/src/monitor/thread-resolution.ts:57-60`
  - body: This fallback calls `conversations.replies` with the child reply's own timestamp, but that method retrieves a thread by its root/thread `ts`. In the exact case this code handles, the parent timestamp is unknown, so the fallback normally returns `thread_not_found` and spends an extra Slack API call without recovering `thread_ts`.
  - confidence: 0.87
- **[P2] Make ambiguous replies affect routing:** `extensions/slack/src/monitor/thread-resolution.ts:181-184`
  - body: The resolver now returns `_ambiguousThreadReply: true` for unresolved replies, but downstream prepare/routing still keys off `thread_ts` and does not consult this marker. That lets an unresolved Slack reply continue through root-message behavior despite the new type comment saying downstream should not treat it as a genuine root message.
  - confidence: 0.8
