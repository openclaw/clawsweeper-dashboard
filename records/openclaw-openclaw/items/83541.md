## Summary

Keep open: the PR is a focused status bug fix, but the proposed freshness gate does not distinguish raw current-prompt totals from totals that current main already derives from `input + cacheRead + cacheWrite`, so the added fixture does not prove the reported cumulative-cache failure is fixed. The external PR also still needs real after-fix behavior proof from a long cache-heavy CLI/TUI status path.

## What This Changes

The PR adds a `promptTokensFresh` flag to transcript status usage, gates `/status` Context promotion on it, and adds a regression test for legacy cache-heavy transcript usage.

## Best Possible Solution

Separate the status formatter's current Context numerator from aggregate transcript cache/token totals, using a latest/current prompt snapshot for Context and a regression fixture that fails on current main, then prove it against a redacted long cache-heavy Claude CLI/TUI status run.

## Reproduction Assessment

No. The PR's added fixture is not a high-confidence current-main reproduction because current parser code derives a fresh per-line total from `input/cacheRead/cacheWrite`; the original live Claude CLI transcript artifact is not attached.

## Solution Assessment

No. The current patch is close to the right boundary, but the freshness bit it uses is already set for derived prompt totals, so the safer solution is to compute Context from a latest/current snapshot while keeping aggregate cache totals separate.
