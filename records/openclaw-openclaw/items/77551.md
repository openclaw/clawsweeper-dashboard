## Summary

Keep open. Current main still has the Bedrock shared-credentials refresh gap: OpenClaw delegates `aws-sdk` auth to the AWS SDK default chain, but the sentinel path has no runtime refresh state and the Bedrock plugin does not invalidate the SDK shared-ini cache on auth failures.

## What This Changes

The issue requests that Bedrock `aws-sdk` auth re-read refreshed shared AWS credentials during gateway lifetime instead of failing with `ExpiredTokenException` until restart.

## Best Possible Solution

Keep Bedrock auth provider-owned and preserve the AWS SDK default-chain contract by forcing shared-credentials re-resolution on auth failures or each Bedrock request, with regression coverage for rotated ini credentials that have no expiration field.

## Reproduction Assessment

Yes, from source inspection: cached shared-ini file reads plus non-expiring AWS credentials explain why refreshed `~/.aws/credentials` can be ignored, and OpenClaw's sentinel path cannot trigger runtime-auth refresh. A live AWS rotation run was not performed.

## Solution Assessment

Yes, the narrowest fix is in the Bedrock provider/runtime auth boundary: invalidate or bypass the AWS shared-ini cache for `aws-sdk` Bedrock calls after auth-class failures without changing user config.
