## Summary

Keep open: current main still only supports a global `agents.defaults.userTimezone`, and this feature PR is useful but not merge-ready because the branch leaves several `"user"` timezone paths without the active agent context.

## What This Changes

The PR adds optional `agents.list[].userTimezone`, a shared agent-aware timezone resolver, validation/docs updates, and selected runtime/channel/SDK plumbing for per-agent timezone behavior.

## Best Possible Solution

Land the feature only after the public config contract is accepted and the branch consistently resolves the active config-aware agent for every prompt, heartbeat, memory, system-event, gateway timestamp, envelope, SDK, and bundled-plugin path that uses `"user"` timezone semantics.

## Reproduction Assessment

Yes from source inspection. Current main has a strict `agents.list[]` schema without `userTimezone`, and the relevant current-time, heartbeat, envelope, and system-event paths read only `agents.defaults.userTimezone`; I did not run tests because this review is read-only and `pnpm` is not available.

## Solution Assessment

No, not yet. The PR is the right maintainable direction, but it needs a rebase plus focused fixes for config-aware session-agent resolution and complete `agentId` propagation before merge.

- **[P2] Pass the heartbeat agent into current-time injection:** `src/infra/heartbeat-runner.ts:550`
  - body: This change makes quiet-hours use `agentId`, but the same heartbeat run still builds the prompt with `appendCronStyleCurrentTimeLine(prompt, cfg, startedAt)` and no agent id.   - confidence: 0.91
- **[P2] Resolve memory flushes with the config-aware agent:** `src/auto-reply/reply/agent-runner-memory.ts:469-471`
  - body: `agentIdForFlush` uses `resolveAgentIdFromSessionKey`, which maps legacy or alias sessions to `main`. If the configured default agent is not `main`, memory flush paths and `Current time` can use the wrong per-agent timezone; use `resolveSessionAgentId({ sessionKey, config: params.cfg })` or the already-resolved run agent instead.
  - confidence: 0.9
- **[P2] Thread agent ids through every envelope caller:** `src/auto-reply/envelope.ts:75`
  - body: `resolveEnvelopeFormatOptions` only applies the per-agent timezone when callers pass `opts.agentId`. Several callers still resolve envelope options with only config, so `envelopeTimezone: "user"` continues falling back to the default timezone on those routes.
  - confidence: 0.86
