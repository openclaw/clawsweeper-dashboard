## Summary

Keep open: current main still lacks the requested per-agent timezone support, and this PR remains the active implementation candidate, but the branch is not merge-ready because several `"user"` timezone paths can still fall back to the global/default timezone.

## What This Changes

The PR adds `agents.list[].userTimezone` and threads per-agent user-local timezone resolution through config schema, prompts, heartbeat, envelopes, system events, memory/post-compaction paths, tests, and docs.

## Best Possible Solution

Land the feature only after maintainer approval of the public config contract, a clean rebase onto current main, complete active-agent propagation for every `"user"` timezone path, regression coverage, and a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main rejects the per-agent config key and resolves prompt, heartbeat, envelope, system-event, and memory timezones from `agents.defaults.userTimezone`; PR head still leaves identifiable paths without config-aware agent resolution.

## Solution Assessment

No, not yet. The helper/schema direction is maintainable, but the PR must consistently pass the active config-aware agent through heartbeat prompt injection, memory flushes, and envelope callers before it is the right fix.
