## Summary

Keep this PR open. Current main still lacks `agents.list[].userTimezone`, and the linked feature remains valid, but the proposed patch is not merge-ready because it still leaves per-agent timezone gaps in memory-flush, heartbeat current-time injection, and direct envelope callers that already know the routed agent.

## What This Changes

The PR adds optional `agents.list[].userTimezone` config/schema/docs support, a shared per-agent timezone resolver, and threads agent-aware timezone selection through selected prompt, gateway, heartbeat, memory, system-event, and channel envelope paths with tests.

## Best Possible Solution

Rebase the PR onto current main and keep the feature as a single coherent config contract: accept and document `agents.list[].userTimezone`, validate IANA values consistently, centralize per-agent override resolution, and thread a config-aware active agent id through every `"user"` timezone consumer, including direct plugin envelope callers and heartbeat/memory helper paths, with focused regression tests for each boundary.

## Reproduction Assessment

Yes. Configure `agents.defaults.userTimezone` to one IANA zone and a different `agents.list[].userTimezone` for a non-default agent, then inspect prompt/time, heartbeat, envelope, system-event, memory flush, or post-compaction paths; current main source shows those paths still read the global default.

## Solution Assessment

No, not yet. The PR is the right implementation direction, but the latest diff still misses concrete call sites, so the best solution is to finish agent-id propagation and rebase before normal maintainer merge review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve memory flushes with the config-aware agent:** `src/auto-reply/reply/agent-runner-memory.ts:466-470`
  - body: This derives the flush agent from `resolveAgentIdFromSessionKey`, which maps legacy or alias session keys to `main`. If the configured default agent is not `main`, memory flush paths and `Current time` can use the wrong per-agent timezone; use `resolveSessionAgentId({ sessionKey, config: params.cfg })` or the already-resolved run agent instead.
  - confidence: 0.9
- **[P2] Pass the heartbeat agent into time-line injection:** `src/infra/heartbeat-runner.ts:1209`
  - body: The PR makes quiet-hours use `agentId`, but the heartbeat body still calls `appendCronStyleCurrentTimeLine(prompt, cfg, startedAt)` without that agent. A heartbeat can be scheduled in the agent's timezone while injecting `Current time` in the default timezone, which can point date-sensitive prompts at the wrong day.
  - confidence: 0.91
- **[P2] Thread agent ids through remaining envelope callers:** `src/auto-reply/envelope.ts:69-75`
  - body: The new resolver only uses a per-agent timezone when callers pass `opts.agentId`, but several direct inbound plugin paths already have `route.agentId` and still call `resolveEnvelopeFormatOptions(cfg)` without it. Those channels will keep using `agents.defaults.userTimezone` for `envelopeTimezone: "user"`, so the feature remains inconsistent outside the channels updated in this PR.
  - confidence: 0.84
