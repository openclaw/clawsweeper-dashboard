## Summary

Keep open. The feature is still not on current main and the related issues were routed to this PR, but this branch cannot merge as-is because heartbeat time injection still falls back to the default timezone and the external PR lacks real after-fix behavior proof.

## What This Changes

The PR adds optional `agents.list[].userTimezone` support and threads agent-aware timezone resolution through prompts, timestamps, heartbeat, memory, docs, and tests.

## Best Possible Solution

Rebase the PR on current main, keep the shared resolver approach, update every current `user` timezone surface with the active agent id, and add redacted real behavior proof plus focused regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main has a strict `agents.list[]` schema without `userTimezone`, and prompt/heartbeat timezone paths still read `agents.defaults.userTimezone`. I did not run a live gateway because this review is read-only and tests may write artifacts.

## Solution Assessment

No, not yet. The centralized resolver and schema direction are maintainable, but the PR misses the heartbeat current-time injection path and needs a current-main rebase plus real behavior proof before it is the best fix.

## Review Findings

Full review comments:

- **[P2] Pass the agent id into heartbeat time injection:** `src/infra/heartbeat-runner.ts:547`
  - body: `isWithinActiveHours` now evaluates quiet-hours with the heartbeat agent's timezone, but the heartbeat body is still built later with `appendCronStyleCurrentTimeLine(prompt, cfg, startedAt)` and no `agentId`. Thread the same `agentId` into the prompt injection call and cover it with a heartbeat test.
