## Summary

Keep open. Current main still has only global `agents.defaults.userTimezone`, while this PR is a valid implementation direction but is not merge-ready because the branch still leaves per-agent timezone propagation gaps in heartbeat, memory flush, and envelope paths.

## What This Changes

The PR adds optional `agents.list[].userTimezone` schema/docs support, a shared per-agent timezone resolver, and threads agent-aware timezone selection through selected prompt, gateway, heartbeat, envelope, memory, system-event, SDK, and channel paths with tests.

## Best Possible Solution

Rebase the branch onto current main and land the feature as one coherent config contract: accept and document `agents.list[].userTimezone`, validate IANA values consistently, centralize per-agent resolution, and pass a config-resolved active agent id to every `"user"` timezone consumer across prompt, heartbeat, memory, system events, gateway timestamps, SDK envelope helpers, and bundled plugin callers.

## Reproduction Assessment

Yes. Configure `agents.defaults.userTimezone` to one IANA zone and `agents.list[].userTimezone` to a different zone for the active agent, then inspect current-time, heartbeat, envelope/system-event, memory flush, or post-compaction paths; main still uses the global default, and the PR head leaves specific propagation gaps.

## Solution Assessment

No, not yet. The PR is the right maintainable direction, but it needs a rebase plus focused fixes for config-aware session-agent resolution and complete `agentId` threading before merge.
