## Summary

Keep open: current main does not implement per-agent `userTimezone`, and the latest PR head still has two source-visible propagation bugs plus missing inspectable real behavior proof.

## What This Changes

The PR adds `agents.list[].userTimezone`, central agent-aware timezone resolution, IANA validation, docs, and tests across prompt, status, envelope, heartbeat, cron, memory, and channel paths.

## Best Possible Solution

Keep the per-agent timezone direction, finish propagation through every `user` timezone path, add regressions for legacy/default-agent sessions and heartbeat prompt time, rebase against current main, and require redacted real runtime proof before merge.

## Reproduction Assessment

Not applicable as a feature request, but the remaining PR defects are source-reproducible: memory flush uses the raw session-key resolver, and heartbeat prompt time-line injection omits the resolved agent id. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as currently patched. The centralized resolver is the right shape, but the implementation needs the memory flush and heartbeat propagation fixes before it is the narrow maintainable solution.

## Review Findings

Full review comments:

- **[P2] Resolve flush timezone with the config-aware agent:** `src/auto-reply/reply/agent-runner-memory.ts:470`
  - body: `resolveAgentIdFromSessionKey` maps legacy/global keys to `main`, so memory flushes for a non-`main` default agent can still render `YYYY-MM-DD` and `Current time` in the wrong timezone.   - confidence: 0.89

Per-agent heartbeat windows can be evaluated in one timezone while the prompt receives another; pass the same agent id into the helper and add coverage.
  - confidence: 0.87
