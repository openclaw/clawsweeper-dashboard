## Summary

Keep open. Current main still has only global `agents.defaults.userTimezone`, while this PR is the relevant implementation path but is not merge-ready: the public PR API reports `mergeable_state: dirty`, and the latest diff still leaves concrete per-agent timezone propagation gaps.

## What This Changes

The PR adds optional `agents.list[].userTimezone` config/schema/docs support, a shared per-agent timezone resolver, and threads agent-aware timezone selection through selected prompt, gateway, heartbeat, memory, system-event, SDK, and channel envelope paths with tests.

## Best Possible Solution

Rebase the PR onto current main and land the feature as one coherent config contract: accept and document `agents.list[].userTimezone`, validate IANA values consistently, centralize per-agent timezone resolution, and pass a config-resolved active agent id to every `"user"` timezone consumer, including newly added plugin envelope callers and heartbeat/memory helper paths.

Yes. ## Solution Assessment

No, not yet. The PR is the right implementation direction, but it needs a rebase plus focused fixes for config-aware agent resolution and complete `agentId` threading before it is the narrow maintainable solution.

- **[P2] Resolve memory flushes with the config-aware agent:** `src/auto-reply/reply/agent-runner-memory.ts:466-470`
  - body: The new `agentIdForFlush` uses `resolveAgentIdFromSessionKey`, which maps legacy or alias sessions to `main`. If the configured default agent is not `main`, memory flush paths and `Current time` can use the wrong per-agent timezone; use `resolveSessionAgentId({ sessionKey, config: params.cfg })` or the already-resolved run agent instead.
  - confidence: 0.9
- **[P2] Pass the heartbeat agent into time-line injection:** `src/infra/heartbeat-runner.ts:547`
  - body: The PR makes quiet-hours use `agentId`, but the same heartbeat run still builds the prompt with `appendCronStyleCurrentTimeLine(prompt, cfg, startedAt)` later in this function. A heartbeat can be scheduled in the agent timezone while injecting `Current time` in the default timezone, which can point date-sensitive logic at the wrong day.
  - confidence: 0.91

  - body: `resolveEnvelopeFormatOptions` only applies a per-agent timezone when callers pass `opts.agentId`. Current main has additional inbound plugin/core callers that still call it with only `cfg`, so `envelopeTimezone: "user"` will keep using the default timezone after rebase unless those routes pass their active agent id too.
  - confidence: 0.86
