- **[P1] Keep existing userTimezone configs from failing startup:** `src/config/zod-schema.agent-defaults.ts:48`
  - body: Current main accepts any `agents.defaults.userTimezone` string and `resolveUserTimezone` falls back to the host timezone when it is invalid. This line changes the existing default key to `IanaTimezoneSchema`, so a user who already has a typo or unsupported timezone can hit a config parse failure after upgrading. Keep the existing key permissive with resolver/doctor diagnostics, or add an explicit migration/strict-mode decision before failing startup.
  - confidence: 0.9
- **[P2] Pass the heartbeat agent into current-time injection:** `src/infra/heartbeat-runner.ts:641`
  - body: `runHeartbeatOnce` now evaluates active hours with `agentId`, but the heartbeat body is still built with `appendCronStyleCurrentTimeLine(prompt, cfg, startedAt)` and therefore resolves the default timezone. In a multi-agent config, quiet-hours can be checked in the routed agent's timezone while the prompt tells the model a different local time.
  - confidence: 0.93
- **[P2] Resolve memory-flush agent ids through config:** `src/auto-reply/reply/agent-runner-memory.ts:469-471`
  - body: This uses `resolveAgentIdFromSessionKey`, which maps legacy/global session keys to `main`. If the configured default agent is not `main`, memory flush can choose the wrong `userTimezone` for the daily path and injected `Current time`; use the config-aware session-agent resolver instead.
  - confidence: 0.86
- **[P2] Pass agent context into inbound envelope options:** `src/auto-reply/reply/get-reply-run.ts:318`
  - body: `runPreparedReply` has the resolved `agentId` in scope, but this call still builds envelope options without it. Structured inbound context and history timestamps formatted here will keep using the default/host user timezone, so the PR does not consistently apply per-agent `userTimezone` to the inbound prompt context it claims to cover.
  - confidence: 0.78
