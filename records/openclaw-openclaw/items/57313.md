## Summary

Keep open: current main partially exposes run and current-candidate model context to hooks, but it still does not expose the persisted session provider/model override or override-source state requested here. This is a valid SDK/runtime improvement, but the public field names and user-pinned versus auto-fallback semantics need maintainer choice before a fix PR.

## What This Changes

The issue requests exposing per-session provider/model override state to `before_model_resolve` plugin hook context so routing plugins can yield to user-pinned model selections.

## Best Possible Solution

Add a backward-compatible SDK/runtime contract that exposes session override values plus source or an explicit user-pinned signal to `before_model_resolve`, then document and test the precedence contract alongside the fallback-iteration hook behavior.

## Reproduction Assessment

Yes, source-level. Current main persists session override and source state, uses that state internally for fallback resolution, and still omits it from the public `before_model_resolve` hook context.

## Solution Assessment

Unclear as proposed. Additive hook-context fields are the right direction, but raw override strings alone are weaker than exposing source or explicit user-pinned semantics and should be coordinated with the related fallback-contract issue.
