## Label Justifications

- `P2`: This is a valid public Plugin SDK/runtime improvement for routing plugins with limited blast radius but it needs a contract decision.
- `impact:session-state`: The missing hook context forces plugins to inspect persisted session override state directly.
- `impact:auth-provider`: The issue concerns provider/model routing precedence when a user has pinned a session model.

## Summary

Keep open: current main and the latest release still do not expose persisted session provider/model override values or their source to `before_model_resolve` hooks, so routing plugins cannot reliably yield to a user-pinned model without reading internal session state. The remaining work is valid but changes the public Plugin SDK contract and should be decided with the related fallback-hook contract issue at https://github.com/openclaw/openclaw/issues/63139.

## What This Changes

Expose current session provider/model override information to `before_model_resolve` hook context so routing plugins can yield to user-pinned `/model` selections.

## Best Possible Solution

Add a backward-compatible hook context contract that exposes session override values plus their source or an explicit user-pinned signal, then document and test precedence together with the fallback-iteration hook contract.

## Reproduction Assessment

Yes, source-reproducible: current main persists `providerOverride`, `modelOverride`, and `modelOverrideSource`, but the public hook context and runtime `hookCtx` passed to `runBeforeModelResolve` omit those persisted session fields.

## Solution Assessment

Unclear as requested: additive hook-context fields are the right direction, but raw override strings alone are weaker than exposing source or explicit user-pinned semantics and should be coordinated with https://github.com/openclaw/openclaw/issues/63139.
