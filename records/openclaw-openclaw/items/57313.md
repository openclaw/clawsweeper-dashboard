## Label Justifications

- `P2`: This is a valid Plugin SDK/runtime improvement for routing plugins with limited blast radius but a public contract decision.
- `impact:session-state`: The missing hook field forces plugins to inspect persisted session override state directly.
- `impact:auth-provider`: The issue concerns provider/model routing precedence when a user has pinned a session model.

## Summary

Keep open: current main and v2026.5.20 still keep persisted session provider/model override values and their source internal, while `before_model_resolve` plugins only receive run/session/current-candidate context. The request is valid, but it changes the public Plugin SDK contract and should be decided alongside the related fallback-hook contract discussion at https://github.com/openclaw/openclaw/issues/63139.

## What This Changes

Expose session-scoped provider/model override state to `before_model_resolve` plugin hook context so routing plugins can yield to explicit user model pins.

## Best Possible Solution

Add a backward-compatible hook context contract that exposes session override values plus their source or an explicit user-pinned signal, then document and test the precedence behavior alongside the fallback-iteration hook contract.

## Reproduction Assessment

Yes, source-level. Current main persists `providerOverride`, `modelOverride`, and `modelOverrideSource`, but the public hook context and runtime `hookCtx` still omit those persisted session fields before `runBeforeModelResolve`.

## Solution Assessment

Unclear as proposed. Additive hook-context fields are the right direction, but raw override strings alone are weaker than exposing source or explicit user-pinned semantics and should be coordinated with the related fallback-contract issue.
