## Label Justifications

- `P2`: This is a valid SDK/runtime improvement for model-routing plugins with limited blast radius but a public contract decision.
- `impact:auth-provider`: The issue concerns provider/model routing precedence when a user has pinned a session model.
- `impact:session-state`: The missing hook field forces plugins to inspect persisted session override state directly.

## Summary

Keep open: current main still does not expose session provider/model override or override-source state in the plugin hook context, even though session state and docs already distinguish user-pinned selections from automatic fallback selections.

## What This Changes

The issue requests adding session provider/model override fields to `PluginHookAgentContext` so `before_model_resolve` plugins can yield when a user has pinned a session model.

## Best Possible Solution

Add a backward-compatible SDK/runtime contract that exposes session override values plus source or an explicit user-pinned signal to `before_model_resolve`, then document and test the precedence contract for routing plugins.

## Reproduction Assessment

Yes, source-level. Current main persists session override/source state and uses it internally, while the public hook context passed to `before_model_resolve` still omits those fields.

## Solution Assessment

Unclear as proposed. Additive hook-context fields are the right direction, but raw override fields alone are weaker than exposing source or user-pinned semantics and should be coordinated with the related fallback-contract issue.
