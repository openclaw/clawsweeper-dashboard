## Label Justifications

- `P2`: This is a valid SDK/runtime improvement for model-routing plugins with limited blast radius but a public contract decision.
- `impact:session-state`: The missing hook field forces plugins to inspect persisted session override state directly.
- `impact:auth-provider`: The issue concerns provider/model routing precedence when a user has pinned a session model.

## Summary

Keep open: current main still exposes only run/current-candidate model context to plugin hooks, while the persisted session provider/model override and override source remain internal. The request is valid, but it changes the public SDK contract and overlaps the fallback-hook contract still being discussed at https://github.com/openclaw/openclaw/issues/63139.

## What This Changes

Expose per-session provider/model override state to `before_model_resolve` hook context so routing plugins can yield to explicit user model pins.

## Best Possible Solution

Add a backward-compatible hook context contract that exposes session override values plus their source or an explicit user-pinned signal, then document and test the precedence behavior alongside the fallback-iteration hook contract.

## Reproduction Assessment

Yes, source-level. Current main persists session override and source state, uses that state internally for fallback behavior, and still omits it from the public `before_model_resolve` hook context.

## Solution Assessment

Unclear as proposed. Additive hook-context fields are the right direction, but raw override strings alone are weaker than exposing source or explicit user-pinned semantics and should be coordinated with the related fallback-contract issue.
