## Summary

Keep open: current main still lacks session override/source fields in the plugin hook context, while session state already records the provider/model override and whether it came from a user selection or automatic fallback.

## What This Changes

Expose session provider/model override state to `before_model_resolve` hook context so routing plugins can yield to explicit user model selections.

## Best Possible Solution

Add a backward-compatible SDK/runtime contract that exposes session override values plus source or an explicit user-pinned signal to `before_model_resolve`, then document and test the precedence contract for routing plugins.

## Reproduction Assessment

Yes, source-level. Current main shows session override/source state exists in persisted session data and internal run preparation, but the public hook context and runtime `hookCtx` omit those fields.

## Solution Assessment

Unclear as proposed. The additive context field direction is maintainable, but raw override fields alone are weaker than exposing source/user-pinned semantics, and retry metadata from the adjacent fallback issue should not be folded in without maintainer choice.
