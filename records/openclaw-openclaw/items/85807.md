## Summary

Keep open: current main still emits a generation-mismatched Telegram preview as superseded without `retain:true`, while dispatch deletes non-retained superseded previews. A focused open fix PR already references this issue, so the issue should close only after that or an equivalent narrow fix lands.

## What This Changes

The issue reports that Telegram streaming can delete a successfully sent preview message when a late `sendMessage` response races a generation change.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85825 or an equivalent focused Telegram-only patch, then close this issue; avoid folding in unrelated changes from https://github.com/openclaw/openclaw/pull/85808.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: the existing race test models `forceNewMessage()` racing an in-flight `sendMessage`, and current dispatch deletes the resulting non-retained superseded preview. I did not run tests or live Telegram proof because this review is read-only.

## Solution Assessment

Yes; adding `retain:true` to the generation-mismatch superseded-preview callback and updating the existing race test is the narrow maintainable fix. The issue should remain open until the focused fix PR or equivalent patch lands.
