## Label Justifications

- `P1`: The report describes a user-visible Telegram channel workflow where a successfully sent answer preview can disappear during streaming.
- `impact:message-loss`: The affected behavior is deletion of a landed Telegram preview message, leaving expected chat content missing.

## Summary

Keep open: current main and the latest release still have the reported Telegram preview retention gap. The source path is narrow and source-reproducible: a generation-mismatched `sendMessage` result is reported as superseded without `retain:true`, and the dispatch callback deletes non-retained superseded previews.

## What This Changes

The issue reports that Telegram preview streaming can delete a successfully sent first message when a generation mismatch path calls `onSupersededPreview` without `retain:true`.

## Best Possible Solution

Set `retain:true` on the generation-mismatch superseded-preview path, update the existing race test and dispatch retention coverage, and validate Telegram partial/progress streaming behavior.

## Reproduction Assessment

Yes for a focused source-level reproduction: the existing race test models `forceNewMessage()` racing an in-flight `sendMessage`, and current dispatch deletes the resulting non-retained superseded preview. I did not run a live Telegram repro or tests because this sweep is read-only.

## Solution Assessment

Yes; adding `retain:true` to the generation-mismatch callback is the narrow maintainable fix and matches the existing overflow-preview retention contract. The dispatch callback should still delete genuinely stale previews that do not opt into retention.
