## Label Justifications

- `P1`: The report describes a user-visible Telegram streaming workflow where a successfully sent answer preview can disappear.
- `impact:message-loss`: The affected behavior is deletion of a landed Telegram preview message, leaving expected chat content missing.

## Summary

Keep open: current main and the latest release still have the Telegram preview retention gap, and a same-author companion PR now exists but has not landed. The issue remains valid and should close only after the fix is reviewed and merged or replaced by a narrowed repair.

## What This Changes

Telegram partial/progress preview streaming can delete a successfully sent preview when an in-flight `sendMessage` resolves after a generation mismatch without `retain:true`.

## Best Possible Solution

Land a narrow Telegram fix that adds `retain:true` on the generation-mismatch superseded-preview path, updates the race/dispatch retention tests, and leaves genuinely stale non-retained previews deletable.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: the current race test already models `forceNewMessage()` racing an in-flight send, and current dispatch deletes non-retained superseded previews. I did not run live Telegram or tests because this sweep is read-only.

## Solution Assessment

Yes; adding `retain:true` to the generation-mismatch callback is the narrow maintainable fix and matches the existing overflow-preview retention contract. The linked PR still needs review because its branch includes unrelated changes.
