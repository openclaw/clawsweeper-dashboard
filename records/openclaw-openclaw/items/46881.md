## Summary

Keep this PR open, but the current patch is not mergeable as-is. Source inspection shows current main still lacks WhatsApp reaction ingress, and the PR is useful source work, but its handler does not match Baileys' `messages.reaction` event shape and would drop group reactions while misattributing some DM self-reactions.

## What This Changes

The PR adds WhatsApp reaction fields and a `messages.reaction` handler that forwards non-empty reaction emojis to the agent as inbound messages.

## Best Possible Solution

Implement WhatsApp reaction ingress on current main using Baileys `reaction.key` for actor identity, preserve access-control and dedupe semantics, and add focused tests plus a changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main has no `messages.reaction` listener or reaction fields, while the PR handler can be checked against Baileys 7.0.0-rc.9 event types. I did not run a live WhatsApp account test.

## Solution Assessment

No, this PR is not the best mergeable solution yet. The feature should be carried forward, but the patch needs to derive reactor identity from Baileys `reaction.key`, handle removal/dedupe correctly, and be adapted to the current plugin SDK monitor code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Read the reactor from `reaction.key`:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` entries as `{ key, reaction }`, where the top-level `key` is the reacted-to message and the reaction event's sender key is `reaction.key`; there is no top-level `userJid`. With this code, group reactions hit `group && !reactorJid` and are skipped, while DM self-reactions are attributed to the peer and can reach `onMessage` as user input.
  - confidence: 0.94
- **[P2] Evict reaction dedupe on removal:** `extensions/whatsapp/src/inbound/monitor.ts:443`
  - body: Baileys marks reaction removals with falsey `reaction.text`, but this early `continue` runs before the dedupe key is touched. If a user removes and re-adds the same emoji to the same message within the dedupe TTL, the second add is still treated as a duplicate and never reaches the agent.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `extensions/whatsapp/src/inbound/types.ts:43-44`
  - body: This is a user-facing WhatsApp feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires user-facing `feat`, `fix`, and `perf` changes to include one.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: No supply-chain change was found, but the new reaction ingress path has an actor-misattribution access-control bug.

Concerns:

- **[medium] Self reactions can be accepted as peer input:** `extensions/whatsapp/src/inbound/monitor.ts:453`
