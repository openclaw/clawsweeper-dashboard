## Summary

Keep open for maintainer review rather than cleanup close: current main still lacks WhatsApp inbound reaction forwarding, but this branch is not ready to merge because it misreads the Baileys reaction actor contract, mishandles removal/reapply dedupe, lacks real WhatsApp proof, and overlaps open reaction-surface design work.

## What This Changes

The PR adds `reactionEmoji` and `reactionMessageId` to WhatsApp inbound messages and registers a Baileys `messages.reaction` handler that forwards non-empty reactions through `onMessage`.

## Best Possible Solution

Define one WhatsApp-owned reaction ingress design that uses Baileys `reaction.key`, handles add/remove dedupe correctly, preserves access control, aligns with the open hook/query trackers, and ships with focused tests plus redacted live WhatsApp proof.

## Reproduction Assessment

Not applicable as a feature PR. Current-main source inspection gives high confidence that WhatsApp reactions are not currently exposed through this inbound agent-message path, but that is an unimplemented capability rather than a broken documented contract.

## Solution Assessment

No. The direction is useful, but this patch reads the wrong Baileys field for reactor identity, mishandles removal/reapply dedupe, lacks real WhatsApp proof, and needs maintainer API/product alignment with the related reaction trackers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Read reactor identity from Baileys reaction.key:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`; there is no top-level `userJid`, and the reaction actor is carried by the reaction message key. Deriving `reactorJid` from `userJid` skips group reactions and can treat a DM owner's own reaction as the peer's inbound emoji before access control and `fromMe` metadata are computed.
  - confidence: 0.94
- **[P2] Handle reaction removals before dedupe suppresses reapply:** `extensions/whatsapp/src/inbound/monitor.ts:442-459`
  - body: Empty-text removal events return before the reaction state or dedupe entry can be reconciled, while add events cache the same account/chat/reactor/message reaction within the 20-minute TTL. Removing and then reapplying the same emoji can therefore drop the second real reaction as a duplicate.
  - confidence: 0.88
