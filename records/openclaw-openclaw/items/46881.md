## Summary

Keep open. Current main still does not forward WhatsApp `messages.reaction` events, but this PR is not merge-ready because the proposed dedupe path drops valid re-reactions after a removal and the branch overlaps a broader open WhatsApp message-type PR.

## What This Changes

The PR adds WhatsApp Web reaction-event forwarding to `onMessage` with `reactionEmoji`/`reactionMessageId`, registers and removes a `messages.reaction` listener, and moves inbound activity recording later in the normal upsert path.

## Best Possible Solution

Choose the canonical real-time reaction event contract, then land a rebased implementation with removal-aware dedupe, WhatsApp monitor regression tests, and a changelog entry; keep historical/queryable reaction support tracked separately in #11460.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection is enough to show current main lacks a `messages.reaction` listener and the PR adds one.

## Solution Assessment

No. Forwarding real-time reactions through the WhatsApp monitor is plausible, but this branch should not merge until the removal/dedupe bug is fixed, the implementation is ported to current main patterns, and maintainers choose its scope versus #48570.
