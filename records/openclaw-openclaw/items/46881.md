## Summary

Keep open: current main still lacks WhatsApp inbound reaction forwarding, and this focused PR is useful implementation input, but the patch should not merge as-is because it misreads the Baileys reaction event identity, suppresses valid re-reactions, and lacks real WhatsApp behavior proof.

## What This Changes

The branch adds WhatsApp Web reaction-event forwarding by extending inbound message fields, adding a `messages.reaction` handler, and registering cleanup for that listener.

## Best Possible Solution

Ship a WhatsApp-owned reaction ingress design that uses Baileys' actual payload, handles add/remove identity and dedupe correctly, preserves access control, aligns with the open hook/query trackers, and includes targeted tests plus redacted live WhatsApp proof.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection confirms current main has no WhatsApp `messages.reaction` listener or reaction metadata fields, but that is a missing new capability rather than a broken documented contract.

## Solution Assessment

No. The feature direction is useful, but this branch reads the wrong Baileys field for reactor identity, can drop valid repeated reactions after a removal, lacks real WhatsApp proof, and overlaps broader reaction hook/query decisions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use reaction.key for reactor identity:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }` and puts the actual reaction event key on `reaction.key`, not in a `userJid` field. With `reactorJid = userJid ?? (group ? undefined : remoteJid)`, group reactions are skipped and DM self-reactions are classified as peer reactions, so access control and sender metadata are wrong.
  - confidence: 0.93
- **[P2] Avoid suppressing valid repeated reactions:** `extensions/whatsapp/src/inbound/monitor.ts:458-459`
  - body: The dedupe key ignores reaction event identity and removal state, while empty reaction removals are skipped before any cache update. Removing and reapplying the same emoji to the same message within the TTL produces the same key and drops the second real reaction.
  - confidence: 0.88
