## Summary

Keep open: current main still lacks WhatsApp reaction ingress, but this PR should not merge yet because the current Baileys reaction event shape does not include the `userJid` field the branch relies on, reaction removals still break dedupe, real behavior proof is missing, and overlapping reaction-event work needs a scope decision.

## What This Changes

The PR adds WhatsApp Web `messages.reaction` forwarding to the agent by extending `WebInboundMessage` with reaction fields, adding a reaction handler, and registering/deregistering the listener.

## Best Possible Solution

Land a current-main WhatsApp-owned reaction ingress implementation after choosing the agent-message versus hook/event scope; use the current Baileys event shape, handle reaction removals, preserve access control and cleanup, add regression tests, and include real WhatsApp proof.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection of current main shows no `messages.reaction` listener or reaction fields, and the PR diff adds them.

## Solution Assessment

No. Forwarding real-time reactions is plausible, but this branch relies on the wrong Baileys event field for reactor identity, mishandles reaction removals, lacks real behavior proof, and overlaps broader open reaction-event design work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Derive the reactor from the Baileys reaction event:** `extensions/whatsapp/src/inbound/monitor.ts:453-455`
  - body: Current Baileys emits `messages.reaction` items with `{ key, reaction }` and puts the reaction event key on `reaction.key`; it does not provide `userJid`. With `const reactorJid = userJid ?? (group ? undefined : remoteJid)`, every group reaction is skipped by the next guard, and DM self-reactions are treated as peer reactions. Use `reaction.key` for reactor identity/fromMe and keep the reacted-message `key` separate.
  - confidence: 0.93
- **[P2] Clear dedupe state when reactions are removed:** `extensions/whatsapp/src/inbound/monitor.ts:443`
  - body: Removal events have falsy reaction text and return here before the handler computes or clears the dedupe key used for added reactions. If the same user removes and reapplies the same emoji to the same message within the dedupe TTL, the valid second reaction can be dropped as a duplicate.
  - confidence: 0.9
