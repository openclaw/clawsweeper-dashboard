## Summary

Keep open: current main still lacks inbound WhatsApp reaction delivery, and this PR is useful but not merge-ready because it relies on a non-contract `userJid` field, mishandles removal/reapply dedupe, lacks real WhatsApp proof, and overlaps unresolved hook/query surface decisions.

## What This Changes

The PR adds WhatsApp inbound reaction fields and registers a Baileys `messages.reaction` listener that forwards non-empty reactions to `onMessage`.

## Best Possible Solution

Define one WhatsApp-owned reaction ingress design that uses Baileys `reaction.key`, handles add/remove semantics correctly, preserves access control and privacy gates, and ships with focused tests, docs if the public surface changes, and redacted live WhatsApp proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a broken existing-behavior bug because this is a feature PR, but source inspection gives high confidence that current main lacks inbound WhatsApp reaction delivery and that the PR defects follow from the pinned Baileys event contract.

## Solution Assessment

Is this the best way to solve the issue? No. The direction is useful, but the patch should not merge until it derives actor identity from `reaction.key`, handles removal/reapply semantics, rebases onto current main, and aligns the public reaction surface with the open hook/query trackers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Read the reactor from the reaction event key:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` without a top-level `userJid`; the reaction event key is on `reaction.key`. Relying on `userJid` skips group reactions when that non-contract field is absent and can compute actor identity/access control from the wrong data.
  - confidence: 0.95
- **[P2] Reconcile removals before dedupe suppresses reapply:** `extensions/whatsapp/src/inbound/monitor.ts:442-459`
  - body: Empty reaction-removal events return before a dedupe key is computed or cleared, while add events cache the same account/chat/reactor/message/emoji key for 20 minutes. Removing and reapplying the same emoji within that TTL can drop the second real reaction as a duplicate.
  - confidence: 0.88
