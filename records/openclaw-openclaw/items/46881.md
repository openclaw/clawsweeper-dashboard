## Summary

Keep open. Current main still does not forward WhatsApp reaction events to agents, and this PR is useful source work, but it is not mergeable as-is because the reaction actor is derived from a non-contract field, removal events leave stale dedupe state, and the branch overlaps the broader open #48570 scope.

## What This Changes

The PR adds a WhatsApp Web `messages.reaction` listener, forwards reactions to `onMessage` with `reactionEmoji` and `reactionMessageId`, adjusts inbound activity recording, and extends `WebInboundMessage`.

## Best Possible Solution

Land one WhatsApp-owned implementation on current main that follows Baileys' `{ key, reaction }` contract, derives reactor identity from `reaction.key`, defines the agent-visible reaction contract deliberately, handles removals and re-reactions, and includes focused tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Baileys-shaped group reaction whose actor is only on `reaction.key`; this handler reads top-level `userJid`, and the remove/reapply dedupe case follows deterministically from the early falsey-emoji branch.

## Solution Assessment

No. Forwarding WhatsApp reactions is valid product work, but this PR is not the best merge path as-is because it relies on a non-contract actor field, mishandles removal dedupe lifecycle, and needs current-main integration and scope alignment with #48570.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read the reactor from reaction.key:** `extensions/whatsapp/src/inbound/monitor.ts:450`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`, with the actor-sensitive message key on `reaction.key`; there is no typed top-level `userJid`. This handler reads `userJid`, so Baileys-shaped group reactions can be skipped or attributed through the wrong identity before access control.
  - confidence: 0.9
- **[P2] Clear reaction dedupe state on removals:** `extensions/whatsapp/src/inbound/monitor.ts:443`
  - body: Removal events have falsey `reaction.text`, so this early `continue` leaves prior reaction dedupe state intact. Removing and then re-applying the same emoji to the same message within the dedupe window can make the second reaction invisible to the agent.
  - confidence: 0.84
