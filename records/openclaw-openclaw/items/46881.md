## Summary

Keep this PR open for maintainer review. Current main still does not forward WhatsApp `messages.reaction` events to agents, so the feature is not implemented, but this branch is stale against the current inbound monitor and has correctness issues in actor attribution and reaction-removal dedupe that should block merge as-is.

## What This Changes

The PR adds a WhatsApp `messages.reaction` handler, exposes `reactionEmoji` and `reactionMessageId` on inbound WhatsApp messages, registers listener cleanup, and moves normal inbound activity recording later in the upsert path.

## Best Possible Solution

Land one WhatsApp-owned implementation on current main that follows Baileys' `{ key, reaction }` contract, derives the reactor from `reaction.key`, routes accepted reactions through the current normalize/enrich/enqueue pipeline, deliberately handles removals and re-reactions with the claimable dedupe lifecycle, uses the current listener cleanup helper, and adds focused inbound tests plus docs/changelog coverage for the agent-visible reaction contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to invoke the proposed handler with a Baileys-shaped group reaction where the actor is on `reaction.key` and no top-level `userJid` exists; the handler treats it as unattributable and skips it. The remove/reapply case is also deterministic because the falsey emoji branch runs before any dedupe invalidation.

## Solution Assessment

No. Forwarding WhatsApp reactions is valid product work, but this PR is not the best merge path as-is because it relies on a non-contract actor field, leaves a dedupe lifecycle bug for removals, and needs rebasing onto current main's inbound monitor and dedupe structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read the reactor from `reaction.key`:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`, with the actor-sensitive message key on `reaction.key`; there is no typed top-level `userJid`. This handler reads `userJid`, so Baileys-shaped group reactions can be skipped or attributed through the wrong identity before access control.
  - confidence: 0.9
- **[P2] Clear reaction dedupe state on removals:** `extensions/whatsapp/src/inbound/monitor.ts:443`
  - body: Removal events have falsey `reaction.text`, so this early `continue` leaves prior reaction dedupe state intact. Removing and then reapplying the same emoji to the same message within the dedupe window can make the second reaction invisible to the agent.
  - confidence: 0.83
