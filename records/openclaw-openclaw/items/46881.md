## Summary

Keep open. Current main still lacks WhatsApp inbound reaction forwarding, so the product request is valid, but this PR is not ready to merge because the proposed handler derives actor identity from a non-contract field, leaves removal/re-reaction dedupe behavior broken, and overlaps the broader open WhatsApp message-type PR #48570.

## What This Changes

The PR adds a WhatsApp Web `messages.reaction` listener that forwards reactions to `onMessage` with `reactionEmoji` and `reactionMessageId`, adjusts inbound activity recording, and unregisters the new listener on close.

## Best Possible Solution

Land one WhatsApp-owned implementation on current main that follows Baileys' `{ key, reaction }` contract, derives reactor identity from `reaction.key`, uses the current normalize/enrich/enqueue and claimable-dedupe lifecycle, deliberately handles removals and re-reactions, cleans up listeners through the current lifecycle helper, and adds focused inbound tests plus docs/changelog coverage for the agent-visible reaction contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass a Baileys-shaped group reaction with the actor on `reaction.key` and no top-level `userJid`; this handler skips it as unattributable. The remove/reapply case is also deterministic because the falsey emoji branch runs before dedupe state can be cleared.

## Solution Assessment

No. Forwarding WhatsApp reactions is valid, but this PR is not the best merge path as-is because it relies on a non-contract actor field, leaves a dedupe lifecycle bug for removals, and needs rebasing onto current main's inbound access-control, listener, and claimable-dedupe structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read the reactor from reaction.key:** `extensions/whatsapp/src/inbound/monitor.ts:450`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`, with the actor-sensitive message key on `reaction.key`; there is no typed top-level `userJid`. This handler reads `userJid`, so Baileys-shaped group reactions can be skipped or attributed through the wrong identity before access control.
  - confidence: 0.9
- **[P2] Clear reaction dedupe state on removals:** `extensions/whatsapp/src/inbound/monitor.ts:440`
  - body: Removal events have falsey `reaction.text`, so this early `continue` leaves prior reaction dedupe state intact. Removing and then re-applying the same emoji to the same message within the dedupe window can make the second reaction invisible to the agent.
  - confidence: 0.83
