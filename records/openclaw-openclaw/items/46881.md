## Summary

Keep open: this is a plausible WhatsApp feature PR, but the branch still misreads the Baileys reaction event shape, can suppress valid re-reactions, lacks required real WhatsApp proof, and overlaps newer reaction-surface design work.

## What This Changes

The PR adds WhatsApp Web reaction-event forwarding by extending inbound message fields, adding a `messages.reaction` handler, and registering cleanup for that listener.

## Best Possible Solution

Ship a current-main WhatsApp-owned reaction ingress design that uses Baileys' actual event payload, handles add/remove identity and dedupe correctly, preserves access control, includes targeted tests, and is backed by redacted live WhatsApp proof.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection of current main shows no `messages.reaction` listener or reaction metadata fields, and the PR diff adds that new surface.

## Solution Assessment

No. Forwarding reactions may be useful, but this branch uses the wrong Baileys field for reactor identity, has an over-broad dedupe key for repeated real reactions, lacks real behavior proof, and overlaps broader open reaction-hook/query design work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use reaction.key for reactor identity:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }` and stores the actual reaction event key on `reaction.key`, not in a `userJid` field. With `reactorJid = userJid ?? (group ? undefined : remoteJid)`, group reactions are skipped and DM self-reactions are classified as peer reactions, so access control and sender metadata are wrong.
  - confidence: 0.93
- **[P2] Avoid suppressing valid repeated reactions:** `extensions/whatsapp/src/inbound/monitor.ts:458-459`
  - body: The dedupe key only uses account, chat, reactor, emoji, and reacted message id, so removing and reapplying the same emoji to the same message within the TTL produces the same key and drops the second real reaction. Include event identity such as `reaction.senderTimestampMs` or explicitly model removal state before deduping.
  - confidence: 0.88
