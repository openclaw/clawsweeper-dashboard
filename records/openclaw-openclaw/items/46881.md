## Summary

Keep open: the capability is still missing on current main, but this branch is not merge-ready because it misreads the Baileys reaction event shape, mishandles removal/reapply dedupe, lacks real WhatsApp proof, and overlaps broader open WhatsApp reaction API work.

## What This Changes

The PR adds WhatsApp inbound reaction fields and a Baileys `messages.reaction` listener that forwards non-empty reactions through the agent message path.

## Best Possible Solution

Define one WhatsApp-owned reaction ingress design that uses Baileys `reaction.key`, handles add/remove semantics correctly, preserves access control and privacy gates, and ships with focused tests, docs when the public surface changes, and redacted live WhatsApp proof.

## Reproduction Assessment

Not applicable as a broken existing-behavior repro; this is a feature PR. Source inspection gives high confidence that current main lacks inbound WhatsApp reaction delivery, and source review reproduces the PR defects against the pinned Baileys event contract.

## Solution Assessment

No. The direction is useful, but the patch should not merge until it derives actor identity from Baileys `reaction.key`, handles removal/reapply dedupe, rebases onto current main, and aligns the public reaction surface with the open hook/query trackers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Read reactor identity from the reaction key:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`; the top-level key is the reacted-to message and there is no top-level `userJid`. Using `userJid` here skips group reactions entirely when that non-upstream field is absent and can misclassify actor identity before access control and `fromMe` metadata are computed.
  - confidence: 0.94
- **[P2] Reconcile removals before dedupe suppresses reapply:** `extensions/whatsapp/src/inbound/monitor.ts:442-459`
  - body: Empty reaction-removal events return before any dedupe key is computed or cleared, while add events cache the same account/chat/reactor/message/emoji key for 20 minutes. Removing and then reapplying the same emoji within that TTL can drop the second real reaction as a duplicate.
  - confidence: 0.88
