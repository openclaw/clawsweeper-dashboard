## Summary

## What This Changes

This PR adds a WhatsApp `messages.reaction` listener that forwards non-empty reactions as `WebInboundMessage` objects with reaction metadata and adjusts accepted-inbound activity recording.

## Best Possible Solution

Implement reaction ingress on current main through an explicit WhatsApp reaction event or hook path, with add/remove semantics, correct actor identity, opt-in/default policy decided by maintainers, and real WhatsApp proof for DM and group flows.

## Reproduction Assessment

Yes, at source level. Current main has no generic WhatsApp reaction listener or reaction fields, and the PR head’s group-drop and removal/reapply paths are traceable from the proposed handler.

## Solution Assessment

No. The useful capability should be ported to current main with supported reaction payload handling and an explicit hook/agent-turn policy rather than forwarding every reaction as a normal message by default.
