## What This Changes

## Best Possible Solution

Define one WhatsApp-owned reaction ingress design that uses Baileys `reaction.key`, handles add/remove dedupe correctly, preserves access control, aligns with hook/query trackers, and ships with focused tests plus redacted live WhatsApp proof.

## Solution Assessment

No. The direction is useful, but this patch reads the wrong Baileys field for reactor identity, mishandles removal/reapply dedupe, lacks real WhatsApp proof, and needs maintainer API/product alignment with the related reaction trackers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use Baileys reaction.key for reactor identity:** `extensions/whatsapp/src/inbound/monitor.ts:437-453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`, with the actor identity carried on `reaction.key`; the event item has no `userJid`. This handler discards `reaction.key` and derives `reactorJid` from `userJid`, so group reactions without that non-upstream field are skipped and DM reactions can be misattributed before access control.
  - confidence: 0.94
- **[P2] Handle removals before reaction dedupe:** `extensions/whatsapp/src/inbound/monitor.ts:441-459`
  - body: Empty-text removal events return before the dedupe key is computed or cleared, while add events cache the same account/chat/reactor/emoji/message key. Removing and then reapplying the same emoji within the TTL will silently drop the second real reaction.
  - confidence: 0.88
