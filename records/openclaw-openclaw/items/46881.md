## Summary

Keep open: the WhatsApp reaction direction is useful, but this patch is not ready because it derives reactor identity from a non-Baileys field, drops remove/reapply flows through dedupe, and has no after-fix proof from a real WhatsApp setup.

## What This Changes

The PR adds WhatsApp Web reaction metadata to `WebInboundMessage` and registers a Baileys `messages.reaction` handler that forwards non-empty reactions to `onMessage`.

## Best Possible Solution

Define one WhatsApp-owned reaction ingress design that uses Baileys `reaction.key`, handles add/remove dedupe correctly, preserves access control, aligns with the hook/query trackers, and ships with focused tests plus redacted live WhatsApp proof.

## Reproduction Assessment

Not applicable as a feature PR. Current-main source inspection gives high confidence that WhatsApp reactions are not currently exposed through this inbound agent-message path, but that is an unimplemented capability rather than a broken documented contract.

## Solution Assessment

No. The direction is useful, but this patch reads the wrong Baileys field for reactor identity, mishandles removal/reapply dedupe, lacks real WhatsApp proof, and needs maintainer API/product alignment with the related reaction trackers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use Baileys reaction.key for reactor identity:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }` and stores the actual reaction event key on `reaction.key`; there is no `userJid` field. Deriving `reactorJid` from `userJid` means group reactions are skipped and DM self-reactions can be misattributed, so sender metadata and access control are wrong.
  - confidence: 0.94
- **[P2] Handle removals before reaction dedupe:** `extensions/whatsapp/src/inbound/monitor.ts:443-459`
  - body: Empty-text removal events return before the dedupe key is computed or cleared, while add events cache the same account/chat/reactor/emoji/message key. Removing and then reapplying the same emoji within the TTL will silently drop the second real reaction.
  - confidence: 0.88
