## Summary

## What This Changes

The PR adds a WhatsApp `messages.reaction` listener, `WebInboundMessage` reaction metadata fields, and forwards non-empty reactions through the existing `onMessage` path.

## Best Possible Solution

Implement reaction ingress on current main through an explicit WhatsApp reaction event/hook path, with add/remove semantics, correct actor identity, no default agent turn unless chosen, and real WhatsApp proof for DM and group flows.

## Reproduction Assessment

Yes, at source level. Current main has no reaction listener, and the PR head’s group-drop and removal/reapply paths are traceable from the pinned Baileys event type plus the proposed handler.

## Solution Assessment

No. The useful capability should be ported to current main with a supported reaction payload path and an explicit hook/agent-turn policy instead of forwarding every reaction as a normal message by default.

  - confidence: 0.82
