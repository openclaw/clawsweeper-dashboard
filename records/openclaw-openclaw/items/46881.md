## Summary

Keep open for maintainer review: current main still lacks WhatsApp reaction forwarding, and this PR is not merge-ready because the reaction actor handling conflicts with the Baileys event contract, removal/dedupe behavior is still wrong, and #48570 overlaps the same surface.

## What This Changes

The PR adds WhatsApp Web `messages.reaction` handling, forwards reactions through `onMessage` with `reactionEmoji` and `reactionMessageId`, registers listener cleanup, and adjusts inbound activity recording.

## Best Possible Solution

Land one canonical WhatsApp real-time reaction-event implementation that uses Baileys `reaction.key` for actor identity, current monitor/dedupe APIs, regression tests, and a changelog; keep historical/queryable reaction support tracked separately in #11460.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection shows current main lacks a `messages.reaction` listener and the PR branch adds one, but this is not a broken-existing-behavior reproduction path.

## Solution Assessment

No. Forwarding real-time reactions through the WhatsApp monitor is plausible, but this branch should derive actor identity from Baileys `reaction.key`, fix removal-aware dedupe, port to current main APIs, add tests/changelog, and resolve the overlap with #48570 before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Read the reactor from Baileys reaction.key:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }` with no `userJid`, and the actor is carried on `reaction.key`. This code ignores that field and falls back to `undefined` for group chats, so every group reaction is skipped; in DMs it also misattributes self reactions as peer input because `reaction.key.fromMe` is never considered.
  - confidence: 0.94
- **[P2] Handle reaction removals before dedupe:** `extensions/whatsapp/src/inbound/monitor.ts:443`
  - body: Removal events have an empty emoji and hit this early `continue` before the key for the original reaction can be cleared. If the same user removes and re-applies the same emoji to the same message within the dedupe TTL, the valid second reaction is still treated as a duplicate and never reaches the agent.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `extensions/whatsapp/src/inbound/types.ts:43-44`
  - body: This is a user-facing WhatsApp feature, but the branch only changes the monitor and inbound type files. Repo policy requires a `CHANGELOG.md` entry for `feat` changes before merge.
  - confidence: 0.88
