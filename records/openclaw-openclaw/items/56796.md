## Summary

Keep open: current `main` shipped the timeout lock-race mitigation, but this PR still proposes a distinct fail-closed policy for gateway-close errors and the current branch is not the best implementation because it treats pre-request closes as active remote runs.

## What This Changes

The PR changes `openclaw agent` to fail closed instead of using embedded fallback for gateway timeout, gateway request-timeout, and gateway-closed errors, with focused command tests.

## Best Possible Solution

Use a phase-aware gateway-agent error policy: keep fresh-session fallback for accepted-run timeouts, keep recoverable fallback for pre-request closes, and only fail closed or isolate fallback when the remote agent request may actually still be active.

## Reproduction Assessment

Yes, at the source level. A focused unit path can mock `callGateway` to reject with timeout, request-timeout, pre-request closed, and pending-request closed errors, then assert whether `agentCommand` uses fresh fallback or is not called; I did not run it because this review is read-only.

## Solution Assessment

No. The PR’s broad message-substring fail-closed policy is not the best current solution because main now has structured transport errors and shipped timeout isolation; the safer fix is explicit request-phase classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Distinguish pre-request gateway closes:** `src/commands/agent-via-gateway.ts:189`
  - body: The new classifier treats every `gateway closed` message as a possibly active remote run and throws before embedded fallback. `callGateway` can also receive `onClose` before `onHelloOk` sends `client.request`, so startup or handshake closes would lose recoverable fallback even though no remote turn exists. Classify the request phase instead of matching all close messages.
  - confidence: 0.84
