## Summary

Keep open: current main has shipped the timeout lock-race mitigation via fresh fallback sessions, but this PR still changes all gateway-closed errors into fail-closed errors without distinguishing pre-request closes from possibly active remote agent work.

## What This Changes

This PR changes `openclaw agent` to throw instead of entering embedded fallback when gateway errors mention timeout, request-timeout, or closed, and adds focused command tests.

## Best Possible Solution

Use a phase-aware gateway-agent error policy: preserve embedded fallback for pre-request/recoverable failures, and fail closed or isolate fallback only when the agent request was accepted or may still be active remotely.

## Reproduction Assessment

Yes. Source inspection plus a focused unit path can reproduce the PR regression by making `callGateway` reject with a pre-request gateway-closed transport error and observing that the new classifier throws instead of taking the existing fallback.

## Solution Assessment

No. The broad substring fail-closed policy is not the narrowest maintainable fix; the gateway call layer should expose request phase or accepted state so the CLI can preserve fallback before request and fail closed or isolate only after remote work may be active.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Distinguish pre-request gateway closes:** `src/commands/agent-via-gateway.ts:191`
  - body: The new classifier throws on any message containing `gateway closed`, but current `callGateway` can emit a `closed` transport error from `onClose` before `onHelloOk` sends the agent request. Startup or handshake closes would lose the existing embedded fallback even though no remote turn can hold the session lock, so this needs request-phase classification before failing closed.
