## Summary

Keep open: the PR addresses a real Gateway-agent fallback risk, but the branch is not mergeable as-is because current main now preserves timeout fallback with a fresh session while this PR would turn those cases into hard failures and still lacks phase-aware close handling.

## What This Changes

The PR adds a string-based classifier that throws instead of embedded fallback for Gateway timeout, request-timeout, and closed errors, plus focused unit tests.

## Best Possible Solution

Preserve the fresh-session timeout fallback already on main, and only add fail-closed behavior for close errors if Gateway errors carry request-sent or accepted-run phase metadata with focused tests and redacted real CLI/log proof.

## Reproduction Assessment

Yes, at source level. A focused unit path can mock timeout, request-timeout, pre-request close, and post-request close errors and assert fresh fallback, normal fallback, or fail-closed behavior; no tests were run during this read-only review.

## Solution Assessment

No. The proposed branch is broader than the maintainable fix because it hard-fails current timeout fallback and uses message matching without request-phase semantics for Gateway closes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve fresh timeout fallback:** `src/commands/agent-via-gateway.ts:190-191`
  - body: Current main treats typed Gateway timeouts and `gateway request timeout for agent` as lock-isolated fallback cases. This PR matches those strings first and throws before that path, converting the documented fresh-session fallback into a hard failure.
  - confidence: 0.88

  - confidence: 0.86
