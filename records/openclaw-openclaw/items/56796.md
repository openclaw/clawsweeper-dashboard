## Summary

Keep open for maintainer review: the PR targets a real gateway timeout/session-lock problem, but current main has already shipped a compatibility-preserving fresh-session timeout fallback, while this branch would hard-fail existing timeout and close fallback paths and still lacks inspectable real behavior proof.

## What This Changes

The PR adds string-matched fail-closed handling for `openclaw agent` gateway timeout, request-timeout, and closed errors, plus tests asserting embedded fallback is skipped.

## Best Possible Solution

Preserve the shipped fresh-session timeout fallback, and add fail-closed behavior only once Gateway errors carry request-phase or accepted-run metadata, with focused tests/docs and redacted real CLI proof.

## Reproduction Assessment

Yes, at source level. Mocked `callGateway` timeout, request-timeout, and close paths can exercise the PR behavior, and current-main tests show the shipped timeout path falls back under a fresh `gateway-fallback-*` session.

## Solution Assessment

No. Hard fail-closed timeout handling is not the best fit now that main preserves compatibility with a fresh-session fallback; close handling needs accepted/request-phase metadata before it can safely fail closed.
