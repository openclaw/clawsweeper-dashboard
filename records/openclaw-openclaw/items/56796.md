## Summary

Keep this PR open for maintainer review rather than close it as implemented. Current main has shipped the timeout lock-contention mitigation via fresh `gateway-fallback-*` embedded sessions, but this PR still proposes broad message-substring fail-closed behavior for `gateway closed` errors and would block recoverable fallback for closes that happen before an agent request is sent.

## What This Changes

The PR branch makes `openclaw agent` throw without embedded fallback for gateway timeout, gateway request-timeout, and gateway-closed error messages, with focused tests for those fail-closed cases.

## Best Possible Solution

Rebase or replace the branch with a phase-aware gateway-agent error policy: keep the shipped fresh-session timeout fallback, keep embedded fallback for recoverable pre-request closes, and only fail closed or isolate fallback when the gateway-side agent request may actually still be active. Add focused CLI and gateway tests for each phase.

## Reproduction Assessment

Yes. A focused unit path can mock `callGateway` to reject with a timeout `GatewayTransportError`, `gateway request timeout for agent`, a pre-request closed transport error, and a pending-request close, then assert whether `agentCommand` receives a fresh fallback session or is not called.

## Solution Assessment

No. The PR’s message-substring classifier is not the best current solution because main now has structured transport errors and shipped fresh-session timeout fallback; the safer direction is explicit request-phase classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Distinguish pre-request gateway closes:** `src/commands/agent-via-gateway.ts:189`
  - body: The new classifier treats every `gateway closed` message as an in-flight remote agent and throws before embedded fallback. `callGateway` can also receive `onClose` before `onHelloOk` sends the agent request, so startup or handshake closes would lose recoverable fallback even though no remote turn is active. Please classify the request phase instead of matching all close messages.
