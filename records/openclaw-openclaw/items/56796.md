## Summary

Keep open: the PR still needs rework before merge because current main already shipped the timeout lock-contention mitigation with fresh fallback sessions, while this branch broadly fails closed on every `gateway closed` message and would drop recoverable pre-request fallback.

## What This Changes

The PR branch changes `openclaw agent` to fail closed instead of embedded fallback on gateway timeout, gateway request-timeout, and gateway-closed errors, with focused command tests.

## Best Possible Solution

Use a phase-aware gateway-agent error policy: keep fresh-session fallback for accepted-run timeouts, keep recoverable fallback for pre-request closes, and only fail closed or isolate fallback when the remote agent request may actually still be active.

## Reproduction Assessment

Yes. A focused unit path can mock `callGateway` to reject with a timeout transport error, `gateway request timeout for agent`, a pre-request `closed` transport error, and a pending-request close, then assert whether `agentCommand` gets a fresh fallback session or is not called.

## Solution Assessment

No. The PR’s broad message-substring fail-closed policy is not the best current solution because main now has structured transport errors and shipped timeout isolation; the safer fix is explicit request-phase classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Distinguish pre-request gateway closes:** `src/commands/agent-via-gateway.ts:189`
  - body: The new classifier treats every `gateway closed` message as an in-flight remote agent and throws before embedded fallback. `callGateway` can also receive `onClose` before `onHelloOk` sends the agent request, so startup or handshake closes would lose recoverable fallback even though no remote turn is active. Please classify the request phase instead of matching all close messages.
  - confidence: 0.84
