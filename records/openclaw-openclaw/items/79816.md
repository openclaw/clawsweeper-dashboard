## Summary

Keep open: the patch addresses a plausible Telegram/subagent behavior gap, but it still needs after-fix real behavior proof and the new multi-select parser drops delimiter-containing option values.

## What This Changes

This PR adds Telegram `OC_MULTI`/`OC_SELECT` callback handling, regression tests, and treats `subagent_announce` completions like agent-mediated delivery.

## Best Possible Solution

Keep the owner-boundary fix in the Telegram plugin and subagent delivery layer, but preserve full callback values and require redacted runtime proof before merge.

## Reproduction Assessment

Yes, source-level: current main routes unknown Telegram callback data through the generic synthetic-message path, and current subagent completion gating does not include `subagent_announce`. I did not run live Telegram or gateway proof in this read-only review.

## Solution Assessment

No, not yet: dedicated callback handling and agent-mediated `subagent_announce` delivery are the right direction, but the multi-select parser should preserve the full option value and the PR needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.79

Full review comments:

- **[P2] Preserve full multi-select callback values:** `extensions/telegram/src/bot-handlers.runtime.ts:1611`
  - body: The parser uses `data.split("|")` and keeps only the first segment after `toggle`. Existing interactive option values are arbitrary strings, so `OC_MULTI|toggle|env|prod` compares `env` against the full button value `env|prod` and the tapped option never toggles. Parse the known action prefix and slice the value after it instead.
  - confidence: 0.84
