## Summary

Keep open. Current main and the latest shipped tag still route `sessions_send` through the session visibility guard, while the reported `message` behavior needs a deliberate out-of-session channel authorization contract rather than cleanup closure.

## What This Changes

Allow isolated, cron, or delegated sessions to notify the operator through the agent’s configured DM channel, or return a clearer policy error when that is not allowed.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely an explicit per-agent or per-channel opt-in with clearer preflight errors, while preserving default session isolation and existing announce-route delivery.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows the `sessions_send` rejection path on current main and the latest release, but the `message` tool behavior still needs a live Telegram or equivalent runtime repro.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a default fallback; an explicit opt-in plus clearer preflight errors is safer than silently letting isolated sessions inherit persistent DM bindings.
