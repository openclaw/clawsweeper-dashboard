## Summary

Keep open. Current main still treats `sessions_send` and channel sends as session or route scoped, while the requested persistent-DM fallback would change a session and messaging authorization boundary that needs maintainer product/security review.

## What This Changes

The issue requests a supported way for isolated cron runs and delegated sessions to notify an agent’s configured DM channel through `message` or `sessions_send`.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely an explicit agent/channel opt-in or route-binding rule, preserve default session isolation, and improve the preflight error to name the missing binding or policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live; source inspection shows `sessions_send` denies same-agent targets outside the allowed tree by default and `message` does not expose an obvious persistent-DM fallback, but the Telegram runtime path was not exercised.

## Solution Assessment

Is this the best way to solve the issue? No for an unconditional DM-pairing fallback; an explicit opt-in or route-binding rule plus clearer preflight errors is safer than widening isolated-session send authority by default.
