## Summary

Keep open. Current main still treats this as a session-visibility and explicit-delivery-target boundary: `sessions_send` is guarded by session-tree visibility, isolated cron does not inherit ambient routing, and docs/tests explicitly avoid using DM pairings as implicit automation recipients.

## What This Changes

The issue asks for isolated, cron, or delegated sessions to send `message` and `sessions_send` notifications to an agent's configured DM channel, likely through an explicit opt-in plus clearer preflight errors.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely an explicit agent/channel opt-in or route-binding rule, while preserving default session isolation and improving blocked-send preflight errors.

## Reproduction Assessment

Partly: source inspection gives a clear reproduction path for `sessions_send` tree-visibility rejection and the lack of implicit DM pairing fallback, but no live Telegram setup was exercised in this read-only review.

## Solution Assessment

No as an implicit default fallback: the safer direction is an explicit operator-controlled opt-in plus clearer errors because the requested behavior crosses a session and DM authorization boundary.
