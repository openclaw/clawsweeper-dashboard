## Summary

Keep open. Current main still preserves session/tool visibility and configured-allowlist boundaries for isolated lanes, and the requested persistent DM fallback or opt-in changes background-session authorization rather than being cleanup-closeable.

## What This Changes

Allow isolated cron or delegated sessions to notify a configured DM channel through `message` or `sessions_send`, or return a clearer opt-in/preflight error when that send is blocked.

## Best Possible Solution

Add a least-privilege out-of-session notification contract, keep default session isolation, and return a preflight error that names the blocked binding plus the required opt-in or configured route.

## Reproduction Assessment

No live high-confidence end-to-end reproduction yet. Source inspection gives a strong `sessions_send` guard reproduction on current main, but the exact Telegram `message` adapter failure still needs a real configured DM run.

## Solution Assessment

No for an implicit persistent-DM fallback as the default. The safer solution is an explicit opt-in or route-binding rule plus clearer preflight errors, with tests for both denied and allowed paths.
