## Summary

Keep open: current main has better cron/message-tool routing than the reported 2026.5.19 build, but `sessions_send` still enforces session-tree visibility and there is no clear persistent-DM-pairing fallback contract for isolated/delegated sends.

## What This Changes

The issue requests that isolated cron and delegated sessions be able to notify an agent's configured DM/channel binding through `message` or `sessions_send`.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely via explicit route bindings or an opt-in policy, preserve safe default session isolation, and return clearer preflight errors when a channel or binding is unavailable.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live; source inspection shows `sessions_send` still denies same-agent targets outside the visible tree by default, but the reported Telegram pairing-store path was not exercised in a real runtime.

## Solution Assessment

Is this the best way to solve the issue? No for an unconditional persistent-DM fallback; an explicit route-binding or opt-in policy with clear errors is safer than making isolated lanes inherit all DM pairing authority by default.
