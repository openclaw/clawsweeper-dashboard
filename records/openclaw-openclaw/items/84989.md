## Summary

Keep open: current main still treats `sessions_send` as session-visibility scoped and the `message` tool as channel-context scoped, so letting isolated or delegated lanes reuse persistent DM bindings needs an explicit product/security contract rather than cleanup closure.

## What This Changes

The issue requests allowing isolated, cron, and delegated sessions to send `message` or `sessions_send` notifications to the agent's configured DM channel binding.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely explicit agent/channel opt-in plus clearer preflight errors, while preserving default session isolation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the `sessions_send` visibility denial and missing persistent-DM fallback, but no live Telegram reproduction was run for the exact runtime error text.

## Solution Assessment

Is this the best way to solve the issue? No as a default fallback: resolving persistent DM bindings first would blur the current session/security boundary, so an explicit opt-in plus clearer preflight errors is safer.
