## Summary

Keep open. Current main and v2026.6.8 still enforce session-scoped visibility and do not treat persistent DM pairings as implicit automation recipients, so the central request remains a product/security decision rather than fixed cleanup.

## What This Changes

The issue requests that isolated, cron, or delegated sessions be able to notify a configured DM through `message` or `sessions_send`, or at least receive a clearer preflight error.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely explicit agent/channel opt-in plus clearer blocked-send errors, while preserving default session isolation and proving both `message` and `sessions_send` behavior.

## Reproduction Assessment

Yes from source inspection, but not from a live run: `sessions_send` checks session visibility before delivery, and current docs/tests keep isolated cron from inheriting ambient routing or pairing-store recipients. No live Telegram or Docker repro was run.

## Solution Assessment

No for implicit default fallback. The safer solution is an explicit operator-controlled out-of-session notification contract plus clearer preflight errors, not silent persistent DM inheritance.
