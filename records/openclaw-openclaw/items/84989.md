## Summary

Keep open. Current `main` still enforces session-tree visibility for `sessions_send`, and the documented cron contract says isolated jobs do not inherit ambient channel routing or DM pairing; the requested notification path needs an explicit product/security decision rather than cleanup closure.

## What This Changes

The issue requests a safe way for isolated, cron, or delegated sessions to notify through an agent's configured DM channel, or at least receive a clearer preflight error.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely an explicit agent/channel opt-in or route-binding rule, keep default session isolation safe, and improve preflight errors so agents can see which policy blocked the send.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live: source inspection clearly shows `sessions_send` denies same-agent targets outside the current tree by default, but the exact Telegram `message` path still needs a real Docker or live-channel repro.

## Solution Assessment

Is this the best way to solve the issue? No for implicit persistent-DM fallback as the default; an explicit opt-in plus clearer errors is the safer maintainable direction because this changes a session/security boundary.
