## Label Justifications

- `P2`: The report describes a real background-notification gap with workarounds and limited blast radius, but the expected behavior is not an established current contract.
- `impact:message-loss`: The core symptom is rejected outbound notification messages from cron or delegated sessions.
- `impact:session-state`: The rejection depends on session-tree visibility and delegated/isolated session identity rather than the channel adapter alone.
- `impact:security`: The requested fix would change which background sessions may send to approved DM destinations, an authorization boundary.

## Summary

Keep open. Current main still enforces session-tree visibility for `sessions_send`, and the cron docs say isolated jobs do not inherit ambient routing or DM pairing as fallback recipients; allowing background lanes to use persistent DM bindings needs a deliberate session/security contract, not cleanup closure.

## What This Changes

Requests that isolated cron and delegated sessions can notify through the agent's configured DM channel via `message` or `sessions_send`, or receive a clear missing-binding preflight error.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely an explicit agent/channel opt-in or route-binding rule, keep default session isolation safe, and improve errors so agents know which binding or policy blocked the send.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live: source inspection shows `sessions_send` denies same-agent targets outside the current tree by default, and the `message` path lacks an obvious persistent-DM fallback, but no Telegram runtime repro was run.

## Solution Assessment

Is this the best way to solve the issue? No as a default fallback: using persistent DM pairing as the primary source would blur a session/security boundary, so an explicit opt-in plus clearer preflight errors is the safer path.
