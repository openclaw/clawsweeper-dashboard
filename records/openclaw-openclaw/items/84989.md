## Summary

Keep open. Current main still preserves the session-visibility boundary for `sessions_send` and explicitly avoids treating DM pairing-store entries as implicit automation recipients; the remaining work needs a product/security decision for an out-of-session DM send contract.

## What This Changes

The issue requests that isolated, cron, and delegated sessions be able to notify an agent’s configured DM channel via `message` or `sessions_send`, or receive a clearer authorization error.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely an explicit per-agent/channel opt-in plus clearer preflight errors, while preserving default session isolation and avoiding implicit DM pairing-store fallback.

## Reproduction Assessment

Yes for the source-level rejection path: current main still checks `tools.sessions.visibility=tree` before `sessions_send` and rejects non-owned target sessions. No live Telegram setup was exercised, so the full adapter symptom remains medium-confidence.

## Solution Assessment

No for an implicit default fallback. The safer fix is an explicit operator-controlled out-of-session send contract plus clearer errors because persistent DM binding reuse crosses a session authorization boundary.
