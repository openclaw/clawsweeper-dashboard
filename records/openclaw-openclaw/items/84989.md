## Summary

Keep open. Current main still scopes `sessions_send` through session-tree visibility and does not show a persistent DM-pairing fallback for isolated `message` sends, but changing that default would alter background-session send authority and needs maintainer product/security review.

## What This Changes

The issue requests that cron, isolated, and delegated sessions be able to use `message` and `sessions_send` to notify an agent's configured DM channel, or receive clearer binding errors.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely through explicit route bindings or an opt-in policy, while preserving default session isolation and returning precise preflight errors for missing bindings.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live: source inspection shows `sessions_send` denies same-agent targets outside the current tree by default, and the `message` path lacks an obvious persistent-DM fallback, but no Telegram runtime repro was run.

## Solution Assessment

Is this the best way to solve the issue? No as a default persistent-DM fallback; the safer path is an explicit route-binding or opt-in contract with clearer preflight errors, because the requested behavior crosses a session/security boundary.
