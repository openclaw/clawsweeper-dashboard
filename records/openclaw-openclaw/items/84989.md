## Summary

Keep open. Current main and the latest release still keep outbound session sends scoped by requester session visibility and do not provide an explicit out-of-session DM authority path for isolated, cron, or delegated sessions.

## What This Changes

The issue requests a safe way for isolated, cron, or delegated sessions to send through an agent’s configured DM channel instead of being rejected by session visibility checks.

## Best Possible Solution

Define an explicit, least-privilege out-of-session notification contract with clear blocked-send errors, while keeping default session isolation and non-implicit DM pairing behavior intact.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible for the guard path: current main still gates `sessions_send` through requester session visibility and has no out-of-session DM opt-in, but I did not run a live Telegram/Docker repro.

## Solution Assessment

Is this the best way to solve the issue? No for silent default fallback; the maintainable direction is an explicit operator-controlled authority path plus clearer preflight errors before background sessions can send through persistent DM bindings.
