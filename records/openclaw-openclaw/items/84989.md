## Summary

Keep open. Current main and the latest release still leave the central authorization question unresolved: `sessions_send` is visibility-scoped, implicit cron delivery does not treat DM pairing-store entries as automation recipients, and allowing background/delegated sessions to use persistent DM bindings needs maintainer product/security review.

## What This Changes

The issue requests a safe way for isolated, delegated, cron, or sandboxed agent sessions to notify the operator through the agent's configured DM channel instead of failing session visibility or binding checks.

## Best Possible Solution

Define a least-privilege out-of-session notification contract, likely explicit per-agent or per-channel opt-in plus clearer denied-send errors, while preserving default session isolation and proving both `message` and `sessions_send` paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection supports the reported limitation for `sessions_send` through the visibility guard and for implicit cron delivery through the pairing-store recipient guard, but no live Telegram or Docker repro was run.

## Solution Assessment

Is this the best way to solve the issue? No for silent agent-wide DM inheritance; the safer solution is an explicit least-privilege opt-in with clear preflight errors and tests for both denied/default and opted-in behavior.
