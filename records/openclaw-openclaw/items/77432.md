## Summary

Keep open: the PR addresses a real source-reproducible diagnostic gap, but the current patch leaks the new internal catalog provenance field through the public `models.list` model-catalog contract.

## What This Changes

The PR marks provider-supplemental catalog rows, adds a warn-only doctor check for explicit model allowlists hiding those rows, adds focused tests, and records the change in the changelog.

## Best Possible Solution

Keep the warn-only doctor diagnostic, but keep provenance internal or strip `catalogSource` from public catalog responses and add regression coverage for that boundary.

## Reproduction Assessment

Yes. Source inspection shows current main can append provider supplemental rows, `getModelRefStatus` can mark them unavailable under a non-empty `agents.defaults.models` allowlist, and doctor has no general warning path today.

## Solution Assessment

No. The warn-only doctor approach is the right narrow direction, but the current implementation should not expose `catalogSource` through the public `models.list` schema.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep catalog provenance out of models.list responses:** `src/agents/model-catalog.ts:395`
  - body: Adding `catalogSource` directly to catalog entries means `loadModelCatalog({ readOnly: false })` can return provider-supplemental rows with a property that `ModelChoiceSchema` does not allow. `models.list` returns that full catalog directly for `view: "all"`, so strict protocol clients can receive an out-of-schema field. Keep this provenance internal or strip it before Gateway model-catalog responses.
  - confidence: 0.86
