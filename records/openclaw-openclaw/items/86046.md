## Summary

Keep this PR open: current main and the latest release still compile plugin manifest `modelPatterns` with raw `new RegExp`, while this focused patch uses the existing safe-regex helper and now documents the public manifest behavior. I did not find a blocking correctness or supply-chain issue after the docs update, but the fail-closed change to plugin model matching is a real compatibility decision for maintainers.

## What This Changes

The PR routes plugin manifest `modelPatterns` through `compileSafeRegex`, adds a ReDoS regression test, and documents the safe-regex manifest contract.

## Best Possible Solution

Merge the hardening with the manifest docs and regression coverage, while explicitly accepting that unsafe `modelPatterns` are treated like invalid regex sources and skipped.

## Reproduction Assessment

Yes. Current main plainly sends manifest `modelPatterns` through raw `new RegExp`, and the PR body supplies terminal before/after proof showing catastrophic backtracking with `(a+)+$` and rejection through `compileSafeRegex`.

## Solution Assessment

Yes, with a maintainer compatibility decision. Reusing the existing `compileSafeRegex` helper at the resolver boundary is the narrowest maintainable fix, and the docs/test update now covers the public plugin contract.
