## Summary

Keep open: current main still compiles plugin manifest `modelPatterns` with raw `new RegExp`, and this PR targets the right security boundary, but it changes a public plugin manifest contract without updating the public docs for unsafe patterns failing closed.

## What This Changes

This PR replaces raw plugin manifest `modelPatterns` RegExp compilation in provider model ownership resolution with `compileSafeRegex` and adds a provider resolver regression test for ReDoS-style patterns.

## Best Possible Solution

Land the focused ReDoS hardening in the provider resolver, document the safe-regex behavior for plugin `modelPatterns`, and make the compatibility tradeoff explicit before merge.

## Reproduction Assessment

Yes for source reproduction: current main directly runs `new RegExp(patternSource, "u").test(modelId)` for manifest `modelPatterns`, and the PR body supplies terminal proof that `(a+)+$` catastrophically backtracks on a non-matching model-like string. I did not run a live installed malicious plugin in this read-only review.

## Solution Assessment

Mostly yes: guarding at `resolveModelSupportMatchKind` is the right boundary and reuses the existing safe-regex helper. The missing piece is documenting the new fail-closed plugin manifest contract and clearing the compatibility risk before merge.
