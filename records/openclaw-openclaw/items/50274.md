## Summary

Keep open. Current main still treats model fallback as an explicit ordered chain and only validates `models.providers.*.models[].input` as known modality names, while the requested benchmark-backed ordering policy and capability-aware modality guardrail remain unimplemented design work.

## What This Changes

The issue requests benchmark-backed fallback ordering plus a config guardrail that rejects unsupported model input modalities before deploy.

## Best Possible Solution

Define a repeatable benchmark artifact for the active fallback chain, document the selected fallback policy, then add capability-aware validation for model inputs using trusted provider or catalog metadata while preserving valid custom providers.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path to verify that capability-aware modality validation and benchmark-backed fallback policy are missing, but the ordering request itself requires live benchmark data rather than a deterministic local reproduction.

## Solution Assessment

Unclear. The requested outcome is reasonable, but the best implementation needs maintainer-defined benchmark criteria and a trusted capability-source design; blanket schema tightening around `input` is not the narrowest maintainable fix.
