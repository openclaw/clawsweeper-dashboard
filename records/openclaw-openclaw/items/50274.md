## Summary

Keep open. Current main and v2026.6.10 still lack a benchmark-backed fallback ordering policy for the named active chain, and the model-input guardrail remains a product/API decision because current config accepts audio/video while the session registry and media-understanding paths still disagree on capability authority.

## What This Changes

The issue requests benchmark-backed fallback ordering for named OpenRouter fallback models plus a fail-fast guardrail for unsupported `models.providers.*.models[].input` declarations.

## Best Possible Solution

Define a benchmark-backed fallback policy for the active chain, then align schema, registry, catalog normalization, media routing, docs, tests, and any needed migration or override around one model-capability contract.

## Reproduction Assessment

Yes, at source level. Current main preserves configured fallback order and has conflicting model-input contracts: config accepts audio/video, the session registry still rejects those modalities in generated catalogs, and media-understanding still trusts user-declared image input without a catalog-capability cross-check.

## Solution Assessment

Unclear as a concrete implementation. The requested hardening is useful, but the best fix is not blanket schema tightening; maintainers first need to choose fallback ranking criteria and a compatibility-safe capability authority.
