## Summary

Keep open. Current main still uses explicitly configured fallback order, documents ordered fallback chains rather than latency/quality/cost ranking, and validates `models.providers.*.models[].input` only as allowed modality names. The requested benchmark-backed ordering policy and capability-aware invalid-modality guardrail are still real follow-up work, and the existing custom-provider catalog behavior means this needs maintainer design rather than cleanup closure.

## What This Changes

The issue requests benchmark-backed fallback ordering for the active fallback models plus a config guardrail that rejects unsupported `models.providers.*.models[].input` modalities before deployment.

## Best Possible Solution

Define a repeatable benchmark artifact for the active fallback chain, document the selected fallback-ordering policy in the model failover/models docs, then add capability-aware validation for configured model inputs using trusted provider or catalog metadata while preserving valid custom multimodal providers.

## Reproduction Assessment

Partially. Static inspection gives a high-confidence path to verify the missing capability-aware guardrail and policy artifact, but ranking the named fallback chain requires live benchmark data rather than a deterministic local reproduction.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer-defined benchmark criteria and a trusted capability-source design; blanket schema tightening around `input` is not the narrowest maintainable fix.
