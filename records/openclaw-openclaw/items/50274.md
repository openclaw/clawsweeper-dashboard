## Summary

Keep open. Current main still has ordered, operator-configured fallback chains and enum-only configured-model input validation; the requested benchmark-backed fallback policy and capability-aware modality guardrail are not implemented, and the fallback-policy half is not safely covered by the separate open guardrail issue.

## What This Changes

The issue requests benchmarking and ranking the active fallback models, defining a latency/quality/cost fallback ordering policy, and adding validation/tests/docs for unsupported `models.providers.*.models[].input` modalities.

## Best Possible Solution

End state should be a documented cost-first fallback policy backed by repeatable benchmark evidence, plus a capability-aware configured-model input contract that preserves valid custom provider metadata or provides an explicit migration path.

## Reproduction Assessment

Partially. Source inspection on current main verifies that capability-aware configured-model input validation and benchmark-backed fallback policy are absent, but fallback ranking itself needs live benchmark data and policy criteria rather than a deterministic local repro.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable fix should first define the benchmark criteria and trusted capability source; tightening `input` schema mechanically would be risky for custom provider catalogs.
