## Summary

Keep open. Current main still preserves configured fallback order, documents ordered candidate-chain behavior rather than latency/quality/cost fallback policies, and validates `models.providers.*.models[].input` as allowed modality names without checking those modalities against trusted provider/model capability metadata. The prior ClawSweeper reviews reached the same conclusion, and the issue still needs maintainer policy/design input before a narrow automated fix is safe.

## What This Changes

The issue asks to benchmark and rank the active fallback models, define fallback ordering policies for latency/quality/cost use cases, and add a config guardrail for unsupported model input modalities.

## Best Possible Solution

Define a repeatable benchmark artifact and documented fallback-order policy for the active model chain, then add capability-aware validation for configured model inputs using trusted provider/catalog metadata while preserving valid custom multimodal provider configs.

## Reproduction Assessment

Partially. Static inspection gives a high-confidence path for the missing capability-aware guardrail and policy artifact, but ranking the named models requires live benchmark data rather than a deterministic local reproduction.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer-defined benchmark criteria and a capability-source design; blanket schema tightening around `input` is not the narrowest maintainable fix.
