## Summary

Keep open. Current main still uses configured fallback order, treats OpenRouter scan ranking as a discovery helper rather than a runtime latency/quality/cost policy, and validates model input only as allowed modality names while preserving custom audio/video catalog metadata.

## What This Changes

The issue asks for benchmark-backed fallback ordering policies and capability-aware validation for `models.providers.*.models[*].input`.

## Best Possible Solution

Define a repeatable benchmark artifact for the active fallback chain, document the selected fallback-order policy, then validate configured inputs against trusted provider or catalog capability metadata while preserving valid custom multimodal providers.

## Reproduction Assessment

Partially. Static inspection gives a high-confidence way to verify the missing capability-aware guardrail and policy artifact, but ranking the named fallback models requires live benchmark data rather than a deterministic local reproduction.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer-defined benchmark criteria and a trusted capability-source design; blanket schema tightening around `input` is not the narrowest maintainable fix.
