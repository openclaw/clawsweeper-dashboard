## Summary

Keep open. Current main still lacks MS Teams webhook host binding, and this PR remains the direct implementation candidate, but the branch still has narrow merge blockers: inaccurate omitted-host bind logging, a quick-start doc change that would break direct ingress, stale generated channel config metadata, and a missing user-facing changelog entry.

## What This Changes

The PR adds optional `channels.msteams.webhook.host` support to the Microsoft Teams config schema/types, forwards the host to the Express webhook listener, updates startup logging/docs, and adds lifecycle coverage for default and host-configured listen behavior.

## Best Possible Solution

Land the host-binding feature after preserving omitted-host backward compatibility, logging the actual or explicitly unspecified bind address, keeping the baseline docs externally reachable, documenting loopback as an optional hardened tunnel/proxy setup, regenerating bundled channel config metadata, and adding the required changelog entry.

## Reproduction Assessment

Yes. Current-main inspection reproduces the missing host support in runtime, types, strict schema, generated metadata, and docs; the PR blockers are visible in the provided diff and were reinforced by a local Node omitted-host bind check.

## Solution Assessment

No as-is. The narrow `webhook.host` approach is the right maintainable fix for the requested behavior, but the current patch needs the logging, docs, generated metadata, and changelog corrections before merge.
