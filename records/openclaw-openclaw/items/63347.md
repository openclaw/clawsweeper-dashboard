## Summary

Keep this PR open. Current main still lacks `channels.msteams.webhook.host`, and the PR is a useful scoped implementation for the linked #62766 request, but it should not merge as-is because the diff documents loopback as the baseline quick-start bind, logs an assumed IPv4 wildcard address when the host is omitted, and appears to leave generated channel config metadata stale.

## What This Changes

The PR adds optional `channels.msteams.webhook.host` support to MS Teams config schema/types, forwards a configured host to the Express webhook listener, updates bind logging/docs, and adds lifecycle coverage.

## Best Possible Solution

Keep this PR as the active implementation candidate, but update it so omitted host preserves current listen behavior, configured nonblank host binds explicitly, logs reflect the actual or explicitly unspecified bind address, loopback is documented as an optional hardened tunnel/proxy setting, generated config metadata is refreshed, and release notes are handled per policy.

## Reproduction Assessment

Yes. Current-main source inspection verifies the missing host field across runtime/types/schema/docs, and the PR blockers are reproducible from the provided diff plus a live Node omitted-host listen check that returned an IPv6 unspecified bind address.

## Solution Assessment

No as-is. The narrow host-binding approach is the right maintainable direction, but the current diff needs the logging, docs, generated metadata, and release-note follow-through before it is the best fix.
