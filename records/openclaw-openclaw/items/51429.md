## Summary

Keep this open. Current main and the latest release still persist resolved default workspace paths as host-specific strings in the setup/onboarding/configure write paths, and the open closing PR is the active implementation path rather than a reason to close before merge.

## What This Changes

The issue reports that default workspace setup can save a machine-specific absolute home path into config, making another user's workspace path appear when config is reused or synced.

## Best Possible Solution

Land or replace the linked focused fix so recognized default workspace paths are written as portable `~/.openclaw/...` values while preserving explicit custom absolute paths and env-backed workspace mounts.

## Reproduction Assessment

Yes, at source level. Current main resolves the default workspace to a host-specific absolute path and setup/onboarding/configure persist that value without canonicalizing it back to the documented portable default.

## Solution Assessment

Yes. The narrow maintainable fix is write-time canonicalization for recognized default workspace values, while preserving explicit custom paths and `OPENCLAW_WORKSPACE_DIR` behavior.
