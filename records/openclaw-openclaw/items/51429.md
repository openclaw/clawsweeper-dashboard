## Summary

Keep this open. Current main and v2026.5.20 still write host-specific resolved default workspace values into config; the open closing PR is the active fix path, so this issue should close only after that or an equivalent fix merges.

## What This Changes

The issue asks OpenClaw to stop persisting a host-specific default workspace path such as `/Users/<name>/.openclaw/workspace` when setup/onboarding/configure uses the default workspace.

## Best Possible Solution

Land a focused write-time canonicalization that stores recognized home/profile default workspaces as `~/.openclaw/workspace`, preserves explicit custom and env-backed absolute paths, and covers setup, onboarding, and configure persistence with regression tests.

## Reproduction Assessment

Yes, at source level. Current main resolves the default workspace to a host-specific path and setup/onboarding/configure persist that value without canonicalizing it back to the documented portable default.

## Solution Assessment

Yes. The narrow maintainable solution is write-time canonicalization for recognized default workspace values, while preserving explicit custom paths and env-backed defaults.
