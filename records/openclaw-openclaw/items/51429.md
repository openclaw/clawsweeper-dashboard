## Summary

Keep this open. Current main and the latest release still serialize the resolved host-specific workspace path into config, and an open closing PR already targets the narrow portability fix, so this issue should wait for that PR to land rather than be closed early.

## What This Changes

The issue reports OpenClaw creating or using a host-specific `/Users/wangtao/.openclaw` workspace and asks for the default workspace configuration to stay portable as `~/.openclaw`.

## Best Possible Solution

Land a focused write-time canonicalization that stores recognized default workspaces as `~/.openclaw/workspace`, preserves custom absolute paths, and leaves runtime workspace resolution unchanged.

## Reproduction Assessment

Yes, at source level. Current main resolves the documented default workspace to a host-specific absolute path and setup/onboarding/configure write paths persist that value into config without canonicalizing it back to the portable default.

## Solution Assessment

Yes. The linked fix direction is the narrow maintainable path: canonicalize only recognized default workspace values during config writes, keep explicit custom absolute paths intact, and let the open closing PR proceed through proof and review.
