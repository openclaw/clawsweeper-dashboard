## Summary

Keep this issue open: current main and v2026.6.10 still persist resolved default workspace paths into config, and the open linked PR is the direct implementation candidate.

## What This Changes

The issue reports that onboarding/setup/configure can bake a host-specific absolute default workspace path into `agents.defaults.workspace`, making it look like OpenClaw has a hardcoded user path.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/79872 with a narrow repair that canonicalizes only recognized home/profile default workspace paths at config-write boundaries while preserving explicit custom absolute paths and `OPENCLAW_WORKSPACE_DIR` behavior.

Yes, at source level. Current main resolves default workspace values and persists them into config across setup/onboard/configure paths; this read-only review did not run the interactive CLI flow.

## Solution Assessment

Yes, the linked repair direction is the right layer. Canonicalizing recognized default workspace values at config-write boundaries is narrower than changing runtime workspace resolution.
