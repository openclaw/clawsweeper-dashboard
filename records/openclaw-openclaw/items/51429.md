## Summary

Current main and the latest release still persist resolved workspace values unchanged in setup, onboarding, and configure paths. The issue should stay open because https://github.com/openclaw/openclaw/pull/79872 is an open, unmerged closing PR for the same bug.

## What This Changes

Prevent setup/onboarding/configure from saving a host-specific absolute default workspace path into `openclaw.json`; the default workspace should remain portable.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79872 or an equivalent narrow repair that canonicalizes only recognized home/profile default workspaces to portable `~/.openclaw/...` values at config-write boundaries while preserving explicit custom absolute paths and `OPENCLAW_WORKSPACE_DIR` behavior.

## Reproduction Assessment

Yes at source level. Current main resolves default workspace input to an absolute path and persists that value through setup/onboard/configure config writes; I did not run the interactive flow because this review is read-only.

## Solution Assessment

Yes, with the existing linked repair direction: canonicalize recognized default workspace paths at config-write boundaries and leave explicit custom absolute workspaces unchanged. The open PR attempts that shape but still needs conflict resolution and maintainer review before this issue can close.
