## Summary

Keep this open. Current main still persists the resolved host-specific default workspace path into config, and an open closing PR already targets the narrow serialization fix, so the issue should wait for that PR to land rather than be closed early.

## What This Changes

The issue asks OpenClaw to stop writing machine-specific absolute default workspace paths into config when setup/onboarding/configure use the default workspace.

## Best Possible Solution

Land a focused write-time canonicalization that stores recognized default workspaces as `~/.openclaw/workspace`, preserves custom absolute paths, and leaves runtime path resolution unchanged.

## Reproduction Assessment

Yes, at source level. Current main resolves the documented default workspace to a host-specific absolute path and setup/onboarding/configure write that value into config; I did not run a live onboarding flow because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is write-time canonicalization for recognized default workspace paths, and the open closing PR follows that direction pending proof, conflict resolution, and normal review.
