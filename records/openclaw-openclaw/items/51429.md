## Summary

Keep this open. Current main and the latest release still serialize the resolved default workspace path directly into config, and the open closing PR is the active fix path rather than a reason to close before merge.

## What This Changes

The issue reports that setup/onboarding can persist a host-specific default workspace path, making another machine/user path appear as the agent workspace.

## Best Possible Solution

Land or replace the linked narrow fix so recognized default workspace values are stored portably while explicit custom absolute workspaces remain unchanged.

## Reproduction Assessment

Yes. Source inspection shows the default resolver produces a host-specific path and setup/onboard/configure write paths persist that value directly; I did not run the interactive flow because this sweep is read-only.

## Solution Assessment

Yes. Canonicalizing only recognized default workspace paths at config-write boundaries is the narrow maintainable solution, with explicit custom absolute paths preserved.
