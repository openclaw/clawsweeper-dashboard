The discussion correctly narrowed this from a literal username hardcode to config baking; a source search found no `wangtao` path in the checkout.

## What This Changes

The issue reports that setup/onboarding can persist a machine-specific absolute default workspace path instead of the documented portable OpenClaw workspace default.

## Best Possible Solution

Land or replace the linked PR with focused write-time canonicalization that stores recognized home/profile default workspaces as `~/.openclaw/...`, preserves explicit custom absolute paths and env-backed defaults, and keeps runtime workspace resolution unchanged.

## Reproduction Assessment

Yes, at source level. Current main resolves the default workspace to a host-specific path and setup/onboarding/configure persist that value without canonicalizing it back to the documented portable default.

## Solution Assessment

Yes, the maintainable direction is narrow write-time canonicalization. The current linked PR still needs to be made compatible with the newer `OPENCLAW_WORKSPACE_DIR` contract and merged before this issue can close.
