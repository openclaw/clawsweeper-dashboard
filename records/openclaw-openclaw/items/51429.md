## Summary

Keep this open. Current main still writes resolved default workspace paths into config, and the open closing PR is the active fix path rather than evidence that the issue is already fixed.

## What This Changes

The issue asks OpenClaw to stop baking machine-specific absolute default workspace paths into setup/onboarding/configure config writes.

## Best Possible Solution

Land a narrow write-boundary fix that preserves explicit custom absolute workspaces but serializes recognized default workspaces as portable `~/.openclaw/...` values across setup, onboarding, and configure.

Yes for source reproduction: current main resolves default workspace inputs and persists the resolved string into `agents.defaults.workspace`. I did not run an interactive setup flow because this review is read-only.

## Solution Assessment

Yes, the linked repair direction is the narrow maintainable fix: canonicalize only recognized default workspace paths at config-write boundaries while preserving explicit custom absolute workspaces.
