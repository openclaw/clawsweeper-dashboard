## Label Justifications

- `P1`: The PR targets and currently risks a user-facing gateway/channel workflow where final replies can be lost or duplicated.
- `merge-risk: 🚨 message-delivery`: The diff changes when stored final replies are sent and cleared, which can duplicate or suppress channel-visible messages if the crash window is not closed.
- `merge-risk: 🚨 session-state`: The diff reads and clears persisted `pendingFinalDelivery*` fields from a new background loop, so stale or uncleared session state can drive later recovery behavior.

## Summary

Keep open: the PR targets a real gateway message-loss bug, but it is not merge-ready because the new reaper sends stored pending-final text before clearing the session state without a durable idempotency or claim primitive tied to `pendingFinalDeliveryIntentId`. A crash or restart in that window can leave `pendingFinalDelivery` looking unsent and trigger another recovery delivery.

## What This Changes

The branch adds a gateway scheduled reaper that scans terminal sessions with `pendingFinalDelivery`, sends the saved reply through durable message delivery, and clears the pending fields with focused unit coverage.

## Best Possible Solution

Add a durable-delivery idempotency or session-claim contract tied to `pendingFinalDeliveryIntentId`, prove crash/restart and concurrent-replacement behavior, then land the reaper as the pending-final subfix for the canonical delivery-loss tracker.

## Reproduction Assessment

Yes at source level: the PR sends the pending-final reply before clearing the session entry, while current startup recovery still treats uncleared `pendingFinalDelivery` as unsent. I did not run a live gateway/channel crash reproduction in this read-only review.

## Solution Assessment

No as submitted. A steady-state reaper is a plausible mitigation, but the best fix must first make the send/clear boundary restart-idempotent through a durable idempotency or claim primitive.
