## Label Justifications

- `P1`: The PR targets and currently risks a user-facing agent/channel workflow where generated replies can be lost or duplicated.
- `merge-risk: 🚨 message-delivery`: The new reaper sends saved replies outside the original dispatch path and can duplicate delivery without a stable send claim.

## Summary

Keep open: the branch targets a real P1 pending-final delivery bug, but the proposed reaper can still lose or duplicate user replies because it clears the current session entry after an async send without proving it is clearing the same pending reply, and it does not actually use a stable persisted delivery intent before sending.

## What This Changes

Adds a gateway scheduled reaper that scans terminal sessions with `pendingFinalDelivery`, sends the saved reply through durable message delivery, clears the pending fields, and adds focused reaper/CI-shard coverage.

## Best Possible Solution

Land the reaper only after it claims the exact pending-final snapshot with a stable persisted delivery intent before sending, clears only when the current entry still matches that snapshot, and covers concurrent replacement plus restart-before-clear regressions.

## Reproduction Assessment

Yes at source level. The linked issue provides concrete stuck `pendingFinalDelivery` state, and current main shows pending finals can be written without a durable intent while the PR's new reaper clears by key after async send without matching the current entry.

## Solution Assessment

No. A steady-state reaper is a plausible mitigation, but this implementation is not the best landable form until it has a real stable claim/intent and guarded clearing semantics.
