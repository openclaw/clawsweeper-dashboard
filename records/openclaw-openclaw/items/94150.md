## Label Justifications

- `P1`: The PR targets a real user-facing gateway/channel workflow where final replies can be lost, and the current patch can also duplicate those replies in a restart window.

## Summary

Keep open: this is a focused repair for a real gateway/session-state message-loss bug, but the current patch is not merge-ready because its new reaper can still duplicate the final reply across a crash/restart window and the PR has no live gateway/channel proof.

## What This Changes

The PR adds a gateway scheduled reaper that scans terminal sessions with `pendingFinalDelivery`, sends the saved reply through durable message delivery, and clears the pending fields.

## Best Possible Solution

Add a durable idempotency or session-claim contract for the pending-final send, prove crash/restart and live channel behavior, then land this as the pendingFinalDelivery subfix under the canonical same-session delivery tracker.

## Reproduction Assessment

Yes at source level: the PR sends the pending-final reply before clearing the session entry, while current startup recovery still treats uncleared `pendingFinalDelivery` as unsent. I did not run a live gateway/channel crash reproduction in this read-only review.

## Solution Assessment

No as submitted. A steady-state reaper is a plausible mitigation, but the best fix must make the send/clear boundary restart-idempotent through a durable idempotency or claim primitive before merge.
