## Summary

Keep this PR open: the trust-surface feature is useful and the real behavior proof is sufficient, but the `--card` path still has a P1 security-boundary blocker because it can follow an off-registry Skill Card URL supplied by a verification response.

## Best Possible Solution

Keep the feature direction, but constrain Skill Card downloads to the verified ClawHub registry or a slug/version card endpoint, add rejection coverage for off-registry card URLs, and have maintainers explicitly accept the passive verdict lookup behavior.

## Reproduction Assessment

## Solution Assessment

No: the overall trust-surface design is strong, but the current `--card` implementation should not follow off-registry URLs before merge; same-registry validation or slug/version card fetch is the safer solution.
