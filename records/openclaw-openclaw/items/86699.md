## Summary

Keep open: the feature direction and proof are useful, but the `--card` path still trusts and fetches an absolute Skill Card URL returned by the verification envelope, and the PR tests assert that cross-origin fetch behavior. That is a security-boundary blocker before this can merge.

## What This Changes

## Best Possible Solution

Keep the trust feature, but constrain Skill Card downloads to the verified ClawHub registry-owned endpoint or an explicit same-registry allowlist, then refresh the branch and dependency review before merge.

## Reproduction Assessment

Yes. Source inspection and added tests show `--card` can fetch `https://cards.example.test/...` directly from a verification response, so the security-boundary issue is reproducible from the PR branch without a live service.

## Solution Assessment

No, not yet. The OpenClaw-side trust surface is the right shape, but card downloads should be registry-bound or endpoint-derived before this is the safest implementation.
