## Summary

Keep open. The PR is useful and has real behavior proof, but the current head still lets `openclaw skills verify --card` fetch a Skill Card from an arbitrary off-registry URL supplied by the verification envelope, which is a trust-boundary blocker before merge.

## What This Changes

The PR adds ClawHub skill verification, Skill Card fetching, gateway trust methods, Control UI verdict/card rendering, protocol models, docs, and regression coverage.

## Best Possible Solution

Keep the feature direction, but reject Skill Card URLs whose normalized origin and registry base do not match the resolved ClawHub registry before fetching or rendering them, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `verify --card` passes the verification envelope's `card.url` into `fetchClawHubSkillCard`, and the new test explicitly accepts an off-registry `cards.example.test` URL.

## Solution Assessment

No, not yet. The product direction is reasonable, but the safest implementation must bind Skill Card fetches to the resolved ClawHub registry or make an explicit maintainer-approved external-hosting contract.
