## Summary

Keep open. Gemini web_search is implemented on current main, but the remaining ZAI/GLM side is still absent from the bundled ZAI plugin contract, runtime registration, tests, and docs; the issue discussion and closed related PRs also point to unresolved ZAI follow-up rather than an implemented fix.

## What This Changes

The issue requests provider-native web_search support for ZAI/GLM and Google/Gemini so users can use those providers' built-in search without configuring a separate search API key.

## Best Possible Solution

Add a bundled ZAI-owned web_search provider through the existing webSearchProviders plugin contract, with manifest metadata, lazy runtime execution, ZAI credential/config reuse, docs, changelog, and registration/runtime tests. Keep the shipped Gemini provider intact and leave Gemini freshness behavior to #66498.

## Reproduction Assessment

Yes. The high-confidence reproduction is structural on current main: ZAI has no webSearchProviders manifest entry, no registerWebSearchProvider call, and no ZAI search runtime, while Gemini has all three pieces.

## Solution Assessment

Yes, if implemented as a provider-plugin feature rather than a core special case. The existing Gemini/Kimi/Grok pattern keeps provider-owned search behavior behind the shared webSearchProviders contract and avoids hardcoding ZAI details into core model traffic.
