## Summary

Keep this open: current main implements the Gemini side through the bundled Google plugin, but the ZAI/GLM side is still absent from the ZAI manifest, runtime registration, shared contract expectations, and docs. Related ZAI implementation PRs were closed as ClawHub/community-plugin work, so the remaining question is a maintainer product/API boundary decision rather than conservative cleanup.

## What This Changes

The issue requests provider-native web_search support for ZAI/GLM and Google/Gemini without requiring a separate third-party search API key.

## Best Possible Solution

Maintainers should choose the ZAI ownership/default boundary; if official bundled support is desired, implement it as a ZAI-owned webSearchProviders integration with manifest metadata, lazy runtime execution, credential/config handling, docs, and registration/runtime tests.

## Reproduction Assessment

Yes for the remaining source-level gap: current main registers and documents Gemini web_search, while ZAI has no webSearchProviders manifest contract, no registerWebSearchProvider call, no docs entry, and no shared contract-test expectation. I did not run a live GLM request because the missing registration is directly visible in source.

## Solution Assessment

Unclear until maintainers choose the product boundary. The maintainable implementation path is provider-owned webSearchProviders integration or external ClawHub distribution, not ad hoc core request passthrough.
