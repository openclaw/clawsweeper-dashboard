## Summary

Keep this open: current main implements the Gemini side, but ZAI/GLM still has no bundled webSearchProviders contract, registration path, runtime provider, or docs entry. The remaining question is a product boundary decision: official bundled ZAI search and any auto-fallback behavior versus external ClawHub/plugin distribution.

## What This Changes

The issue requests provider-native web_search support for ZAI/GLM and Google/Gemini, preferably reusing the model provider credentials when no separate search provider is configured.

## Best Possible Solution

Maintainers should choose the ZAI ownership/default boundary; if bundled support is desired, implement it as a ZAI-owned webSearchProviders provider with manifest metadata, lazy runtime execution, credential/config handling, docs, and registration/runtime tests.

## Reproduction Assessment

Yes. The remaining gap is source-reproducible: current main registers and documents Gemini web_search, while ZAI has no webSearchProviders manifest contract, no registerWebSearchProvider call, no docs entry, and no shared contract-test expectation.

## Solution Assessment

Unclear until maintainers choose the product boundary. The maintainable implementation path is a provider-owned webSearchProviders integration or external ClawHub distribution, not ad hoc core request passthrough.
