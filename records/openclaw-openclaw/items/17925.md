## Summary

Keep this open: current main and the latest release cover the Gemini side, but the ZAI/GLM side still has no bundled web-search provider contract, registration path, runtime provider, or docs entry. The remaining question is a maintainer product decision about official bundled ZAI support versus an external ClawHub/plugin path.

## What This Changes

The issue requests native provider-backed web_search support for ZAI/GLM and Google/Gemini, reusing model-provider credentials instead of requiring a separate search API key.

## Best Possible Solution

Choose the ZAI ownership/default boundary; if bundled support is desired, implement a ZAI-owned webSearchProviders provider with manifest metadata, lazy runtime execution, credential/config handling, docs, and registration/runtime tests instead of ad hoc core passthrough.

## Reproduction Assessment

Yes. The remaining gap is source-reproducible: current main registers and documents Gemini web_search, while ZAI has no webSearchProviders manifest contract, no registerWebSearchProvider call, no docs entry, and no shared contract-test expectation.

## Solution Assessment

Unclear until maintainers choose the product boundary. The maintainable implementation path is a provider-owned webSearchProviders integration or external ClawHub distribution, not request-time core passthrough.
