## Summary

Keep this open: current main has the Gemini web_search path, including freshness/date filter support, but the ZAI/GLM side still has no bundled webSearchProviders contract, registration, docs entry, or runtime provider. The remaining choice is product ownership: bundled official ZAI support versus the existing ClawHub/plugin path.

## What This Changes

Request native or provider-owned web_search support for ZAI/GLM and Gemini so users can use provider-supplied search without a separate third-party search API key.

## Best Possible Solution

Maintainers should choose the ZAI ownership/default boundary; if bundled support is desired, implement it as a ZAI-owned webSearchProviders provider with manifest metadata, lazy runtime execution, credential/config handling, docs, and registration/runtime tests.

## Reproduction Assessment

Yes. The remaining source-level gap is high confidence: current main registers and documents Gemini web_search, while ZAI has no webSearchProviders manifest contract, no registerWebSearchProvider call, no docs entry, and no shared contract-test expectation.

## Solution Assessment

Unclear until maintainers choose the product boundary. The maintainable implementation path is provider-owned webSearchProviders integration or external ClawHub distribution, not ad hoc core request passthrough.
