## Summary

Keep this open: current main implements the Gemini side through the bundled Gemini web_search provider, but the ZAI/GLM side is still missing from the manifest contract, registration path, docs, and shared registration expectations. Existing labels and prior review comments also route the remaining work through a maintainer product/API ownership decision before automation should create a bundled provider.

## What This Changes

The issue requests native web_search support for ZAI/GLM and Gemini provider APIs without requiring a separate third-party search key.

## Best Possible Solution

Choose the ZAI ownership boundary, then add a ZAI-owned WebSearchProvider through the existing plugin contract with manifest metadata, runtime execution, credential handling, docs, and registration/runtime tests if maintainers want it bundled; keep the current Gemini provider intact.

## Reproduction Assessment

Yes for the remaining source-level gap: current main registers and documents Gemini web_search, while ZAI has no webSearchProviders manifest contract, no registerWebSearchProvider call, no docs entry, and no shared contract-test expectation. I did not run a live GLM request because the missing registration is directly visible in source.

## Solution Assessment

Unclear until maintainers choose the product boundary; the maintainable implementation path is a provider-owned webSearchProviders integration, not ad hoc core request passthrough. If bundled support is approved, the existing Gemini/Kimi/Grok provider pattern is the narrowest direction.
