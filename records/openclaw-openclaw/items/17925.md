## Summary

Keep open: current main and the latest release cover Gemini web_search, but the ZAI/GLM side still has no bundled web-search provider contract, registration path, runtime, or docs entry. The remaining question is a maintainer product decision about official bundled ZAI search versus the external plugin/ClawHub path.

## What This Changes

The issue requests provider-native web_search support for ZAI/GLM and Google/Gemini using existing provider credentials instead of requiring a separate third-party search key.

## Best Possible Solution

Maintainers should decide bundled versus ClawHub ownership; if bundled ownership is approved, add a ZAI-owned web_search provider through the existing plugin webSearchProviders contract with manifest metadata, lazy runtime execution, credential/config handling, docs, and provider registration tests.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the remaining gap: current main and v2026.6.5 register Gemini web_search but the ZAI plugin has no webSearchProviders contract or registerWebSearchProvider path. I did not run a live GLM call because no ZAI credential was available and this is a feature request rather than a strict bug lane.

## Solution Assessment

Unclear as a product direction: the maintainable code shape is a provider-owned ZAI webSearchProviders implementation, but maintainers still need to choose whether it should be bundled or shipped externally through ClawHub. Direct model-runtime passthrough would be the weaker layer because OpenClaw now resolves web_search through provider plugins.

- The open product question is whether official bundled ZAI search belongs in OpenClaw core or should remain an external ClawHub/plugin implementation using the existing webSearchProviders API.
