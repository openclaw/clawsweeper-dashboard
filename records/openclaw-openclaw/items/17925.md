## Summary

Keep this open: current main covers the Gemini half through the bundled Gemini web_search provider, but the ZAI/GLM half is still missing from the bundled ZAI plugin manifest, runtime registration, docs, and shared registration contract expectations. Prior maintainer comments and live labels also route the remaining ZAI work through maintainer product/API ownership before automation should add it.

## What This Changes

The issue requests native provider web_search passthrough for ZAI/GLM and Google/Gemini without requiring a separate third-party search API key.

## Best Possible Solution

Choose the ZAI ownership/default boundary, then add a ZAI-owned WebSearchProvider through the existing plugin contract with manifest metadata, lazy runtime execution, credential/config handling, docs, and registration/runtime tests if maintainers want it bundled.

## Reproduction Assessment

Yes for the remaining source-level gap: current main registers and documents Gemini web_search, while ZAI has no webSearchProviders manifest contract, no registerWebSearchProvider call, no docs entry, and no shared contract-test expectation. I did not run a live GLM request because the missing registration is directly visible in source.

## Solution Assessment

Unclear until maintainers choose the product boundary; the maintainable implementation path is a provider-owned webSearchProviders integration, not ad hoc core request passthrough. If bundled support is approved, the existing Gemini/Kimi/Grok provider pattern is the narrowest direction.
