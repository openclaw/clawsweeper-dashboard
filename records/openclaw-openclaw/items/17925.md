## Summary

Keep this open: current main covers the Gemini half through the bundled Gemini web_search provider, but the ZAI/GLM half is still missing from the ZAI plugin manifest, runtime registration, docs, and registration contract expectations. Prior issue comments and live labels route the remaining ZAI/default-boundary decision to maintainer product/API ownership rather than conservative cleanup.

## What This Changes

The issue requests provider-native web_search support for ZAI/GLM and Google/Gemini, preferably reusing the model provider API key when no separate web search provider is configured.

## Best Possible Solution

Choose the ZAI ownership/default boundary, then add a ZAI-owned WebSearchProvider through the existing plugin contract with manifest metadata, lazy runtime execution, credential handling, docs, and registration/runtime tests if maintainers want it bundled.

## Reproduction Assessment

Yes for the remaining source-level gap: current main registers and documents Gemini web_search, while ZAI has no webSearchProviders manifest contract, no registerWebSearchProvider call, no docs entry, and no shared contract-test expectation. I did not run a live GLM request because the missing registration is directly visible in source.

## Solution Assessment

Unclear until maintainers choose the product boundary; the maintainable implementation path is a provider-owned webSearchProviders integration, not ad hoc core request passthrough. If bundled support is approved, the existing Gemini/Kimi/Grok provider pattern is the narrowest direction.
