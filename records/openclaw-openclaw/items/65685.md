## Summary

Keep this issue open. Current main still has the reported live-active `enqueue-followup` path: it enqueues the followup, only drains immediately when the active snapshot is stale, and then unconditionally calls `typing.cleanup()`, which removes the queued message liveness signal. The related items in the provided context are adjacent typing/queue reports, not a canonical duplicate or shipped fix for this branch.

## What This Changes

The issue requests preserving a visible typing/liveness signal for followup messages queued behind an active run instead of immediately clearing the typing controller.

## Best Possible Solution

Update the active `enqueue-followup` path to compute post-enqueue liveness once, drain and clean up only for stale snapshots, and keep or refresh typing for a followup truly queued behind a live run until that queued turn runs or resolves.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `runReplyAgent` with `isActive: true`, `shouldFollowup: true`, and `isRunActive: () => true`; current main reaches `enqueue-followup` and still calls `typing.cleanup()` before returning.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `isRunActive` seam and shared typing signaler in `agent-runner.ts`, rather than adding Discord-specific behavior or changing queue policy semantics.
