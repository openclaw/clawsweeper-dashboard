## Summary

Keep this issue open. Current `main` still has the reported live-active `enqueue-followup` path: it enqueues the followup, only drains immediately when the active snapshot is stale, and then unconditionally calls `typing.cleanup()` before returning, which removes the queued message liveness signal.

## What This Changes

The issue asks OpenClaw to preserve a visible typing/liveness signal for followup messages queued behind a still-active run instead of immediately cleaning up the queued followup typing controller.

## Best Possible Solution

Update the active `enqueue-followup` path to compute post-enqueue liveness once, drain and clean up only for stale snapshots, and keep or refresh typing for a followup that is truly queued behind a live run until the queued turn runs or is otherwise resolved.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `runReplyAgent` with `isActive: true`, `shouldFollowup: true`, and `isRunActive: () => true`; current `main` reaches `enqueue-followup` and still calls `typing.cleanup()` before returning.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing `isRunActive` seam and shared typing signaler in `agent-runner.ts`, rather than adding Discord-specific behavior or changing queue policy semantics.
