## Summary

Current main and shipped releases already route Kimi/Anthropic-compatible streaming through OpenClaw's own per-request SSE parser instead of the Anthropic SDK stream state machine that produced the reported `message_start`/`message_stop` error.

## What This Changes

The issue reports Kimi/Moonshot models failing on v2026.3.28 with an Anthropic stream event-order error after sending a message.

## Best Possible Solution

Keep the shipped boundary-aware Anthropic Messages transport as the canonical fix for this event-order failure; handle newer Kimi-specific cache, thinking, or model-id failures in their narrower open issues.

## Reproduction Assessment

Yes for the historical v2026.3.28 failure: this issue and related reports include repeated Kimi logs with the exact event-order error. Current main was not live-smoked, but the current source and regression test remove the old SDK state-machine reproduction path.

## Solution Assessment

Yes. Replacing the shared Anthropic-compatible path with a per-request OpenClaw SSE parser is the maintainable fix because it avoids provider-specific workarounds while preserving Kimi's Anthropic-compatible route.
