## Summary

Keep this open. Current main still routes Kimi Coding through the Anthropic Messages stream path and only sanitizes the raw stream-order error; I did not find source, tests, or docs proving the reported Kimi run now completes successfully.

## What This Changes

The issue reports that Kimi/Moonshot models fail after sending a message because the Anthropic-compatible stream raises `Unexpected event order, got message_start before receiving "message_stop"`.

## Best Possible Solution

Add a provider-scoped regression test for the reported Kimi Anthropic-compatible stream-order failure and implement Kimi-specific recovery, fallback, or transport routing without weakening strict Anthropic handling globally.

## Reproduction Assessment

Yes for a high-confidence source-level repro path: the issue logs show repeated provider `kimi` failures, and `src/agents/anthropic-transport-stream.test.ts` already has a mocked SSE harness suitable for simulating the invalid event sequence. Live-provider reproduction was not run because no Kimi/Moonshot key was available.

## Solution Assessment

No current solution is proven. Sanitizing the raw error is useful UX hardening, but the maintainable fix is a Kimi-owned stream compatibility path or transport fallback that lets successful provider replies complete.
