## Summary

Keep open: current main still has no `gateway.auth.tokenScopes` or equivalent config contract, and the remaining work is a security-sensitive gateway auth feature rather than a cleanup close. The open implementation PR at https://github.com/openclaw/openclaw/pull/52059 covers much of the same direction, but it is still open with real-proof and auth-boundary blockers, so this issue should stay available as product/security context.

## What This Changes

The issue requests a gateway auth config field that lets trusted headless shared-token deployments assign effective operator scopes without repeated interactive scope-upgrade approval.

## Best Possible Solution

Agree on one bounded shared-token/password scope contract, then land it through the existing gateway auth implementation path with token/password-only gating, generated config schema/docs updates, focused WebSocket/pairing tests, and redacted live gateway proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source inspection gives a high-confidence current-main check: the config field is absent, and compatibility tests lock in the underlying device-less shared-token missing-scope behavior.

## Solution Assessment

Unclear whether the exact `tokenScopes` shape is the best API. A bounded config field is plausible, and https://github.com/openclaw/openclaw/pull/52059 is the closest implementation path, but maintainers need to decide the shared-token, trusted-proxy, loopback, and admin-scope policy before merge.
