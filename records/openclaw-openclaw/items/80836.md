## Summary

Keep open: current main still lacks a gateway auth config surface for token-derived scopes, and the existing WebSocket path intentionally clears device-less self-declared scopes or routes broader paired-device scopes through approval. The related open implementation attempt at https://github.com/openclaw/openclaw/pull/52059 should be reconciled with this request, but the remaining work needs maintainer/security policy for a new privileged auth config knob.

## What This Changes

The issue requests a `gateway.auth.tokenScopes`-style config field so trusted token-mode headless deployments can statically grant operator scopes without interactive scope-upgrade approval.

## Best Possible Solution

Design one bounded gateway auth config contract for trusted token/password deployments, define admin-scope and bind-mode rules, then land it with schema/help/docs, compatibility tests, and live proof through the related PR or a replacement implementation.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main confirms the requested config field is absent and device-less shared-token/password sessions do not derive effective scopes from config.

## Solution Assessment

Unclear whether the exact `tokenScopes` shape is the best final API. A first-class bounded config field is plausible, but maintainers should decide the shared-token trust boundary, admin allowance, and relationship to https://github.com/openclaw/openclaw/pull/52059 before coding or landing.
