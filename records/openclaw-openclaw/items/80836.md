## Summary

Keep open: current main and the latest release do not provide `gateway.auth.tokenScopes` or an equivalent config surface, and the request would add a security-sensitive shared-token scope grant. The related open work at https://github.com/openclaw/openclaw/pull/52059 and https://github.com/openclaw/openclaw/issues/60225 overlaps, but it has not settled the maintainer/security policy or superseded this detailed headless-deployment report.

## What This Changes

Request adds a `gateway.auth.tokenScopes` config field so token-authenticated headless or single-tenant gateway sessions can receive configured operator scopes without per-device approval.

## Best Possible Solution

Define a bounded headless shared-auth scope contract, reconcile it with https://github.com/openclaw/openclaw/pull/52059 and https://github.com/openclaw/openclaw/issues/60225, then update config types/schema/help/docs and add focused auth, pairing, and config regression coverage.

## Reproduction Assessment

Not applicable as a feature request. Source and checked-in tests give high-confidence evidence that current main lacks the requested config field and clears device-less shared-token/password scopes by default.

## Solution Assessment

Unclear whether the exact `tokenScopes` shape is the best final API. A first-class configured scope grant is plausible, but maintainers should decide the admin-scope, loopback/public-bind, token-vs-password, and trusted-proxy boundaries before implementation.
