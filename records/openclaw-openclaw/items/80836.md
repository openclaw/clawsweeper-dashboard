## Summary

Keep open: current main and the latest release still do not expose `gateway.auth.tokenScopes`, and current tests/source preserve the reported device-less shared-token behavior. The right path is the existing open implementation PR, which links this issue as a closing reference and needs maintainer security/product review before this issue can close.

## What This Changes

Request to add a `gateway.auth.tokenScopes` config field so trusted headless token-auth deployments can statically grant operator scopes without repeated interactive device approval.

## Best Possible Solution

Resolve the open token-scopes PR with explicit security bounds and upgrade documentation, or replace it with another maintainer-approved noninteractive scope-grant path before closing this issue.

## Reproduction Assessment

Yes. Source and existing tests show that a device-less shared-token WebSocket connect requesting `operator.read` currently reports `auth.scopes=[]`, while the reported subscribe path requires `operator.read`.

## Solution Assessment

Unclear until maintainers resolve the linked PR. A bounded `tokenScopes` config path is plausible, but because it widens shared-token authority it needs explicit security/product approval rather than an automatic fix PR.
