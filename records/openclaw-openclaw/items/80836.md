## Summary

Keep open; current main lacks the requested token-scopes config surface and deliberately preserves explicit approval for broader device scopes, so this is a security-sensitive feature/config design request rather than implemented cleanup.

## What This Changes

Request adds a `gateway.auth.tokenScopes` config option so shared-token gateway sessions can receive configured operator scopes in headless or single-tenant deployments.

## Best Possible Solution

Design a bounded headless trust mechanism in gateway auth/device pairing with explicit loopback or single-tenant safeguards, schema/docs/tests, and no ad hoc relaxation of the existing scope-upgrade guard.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an RFE for a new config authority. Source and docs do confirm the current designed behavior that broader scope requests create pending upgrade approvals.

## Solution Assessment

Unclear; the requested `tokenScopes` field is plausible, but prior gateway history deliberately blocks silent scope upgrades, so the maintainable solution needs explicit auth/security design review and bounded semantics.
