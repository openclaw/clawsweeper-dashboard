## Summary

Keep open: current main does not have a `gateway.auth.tokenScopes` config surface, and the existing gateway path deliberately clears or revalidates unbound shared-token scopes through device pairing. This is a plausible core feature, but it needs maintainer product/security policy before implementation because it would let config grant operator scopes from a shared token.

## What This Changes

Request adding a `gateway.auth.tokenScopes` config field so token-mode headless deployments can grant static operator scopes without interactive device scope-upgrade approvals.

## Best Possible Solution

Design a bounded gateway auth config surface that can resolve token-mode scopes only under approved trust boundaries, then align schema/help/docs and add focused handshake, pairing, and config regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested config field is absent and current shared-token sessions do not derive effective scopes from config.

## Solution Assessment

Unclear whether the exact `tokenScopes` shape is the final best API. A first-class bounded config field is plausible, but maintainers should decide the admin-scope and loopback/public-bind policy before implementation.
