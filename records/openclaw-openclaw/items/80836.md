## Summary

## What This Changes

Requests a new `gateway.auth.tokenScopes` config field so trusted headless deployments can grant static operator scopes to shared-token Gateway sessions.

## Best Possible Solution

Review and land a bounded, documented token-scope or device-token policy through https://github.com/openclaw/openclaw/pull/81792 or a successor, with Gateway/security approval and upgrade proof, then close this issue from the merged implementation.

## Reproduction Assessment

Yes, at source level. Current main lacks `gateway.auth.tokenScopes`, checked-in tests expect device-less shared-token auth to report empty scopes, and read-scoped methods such as `sessions.messages.subscribe` require `operator.read`.

## Solution Assessment

Unclear until maintainers resolve the linked implementation. A bounded static token-scope config is plausible and safer than trusting client-declared scopes, but privileged grants need explicit security bounds and approval.
