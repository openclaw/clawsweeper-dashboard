## Summary

Keep open: current main still has no `gateway.auth.tokenScopes` or equivalent bounded config surface, and the existing gateway auth path deliberately clears or revalidates shared-token scopes through device identity/pairing. The linked open PR overlaps with the device-less part of this request, but it does not settle the broader paired-device scope-upgrade policy and has its own proof/security blockers.

## What This Changes

The issue requests a new gateway auth config field that can grant effective operator scopes to shared-token sessions in trusted headless or single-tenant deployments.

## Best Possible Solution

Design one bounded gateway auth scope contract, reconcile it with the open `gateway.auth.scopes` PR and related shared-auth reports, then land schema/help/docs plus focused WebSocket pairing and config regression tests under secops review.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested config surface and intentionally fails closed for unbound shared-token scopes.

## Solution Assessment

Unclear whether the exact `tokenScopes` API is the best final shape. A bounded config-granted scope mechanism is plausible, but maintainers should decide token/password versus trusted-proxy coverage, admin-scope gating, and paired-device upgrade semantics before implementation.
