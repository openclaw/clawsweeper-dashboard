## Summary

Keep open: current main and the latest shipped release still have no configured token-scope auth surface, and the related open implementation PR remains unmerged with real-behavior-proof and security-policy blockers. This is a plausible core gateway feature, but it changes the shared-token authorization boundary and needs maintainer/security policy before implementation.

## What This Changes

The issue requests a `gateway.auth.tokenScopes` config field so token-authenticated headless deployments can grant operator scopes without interactive device-pairing approval.

## Best Possible Solution

Define one bounded gateway auth scope contract, reconcile it with the open `gateway.auth.scopes` PR and adjacent shared-auth issue, then implement schema/help/docs and focused handshake, pairing, and config regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and release checks confirm the requested config surface is absent, while existing tests define the current scope-clearing behavior for device-less shared-token/password clients.

## Solution Assessment

Unclear whether the exact `tokenScopes` shape is the best final API. A first-class bounded config field is plausible, but maintainers need to decide admin-scope gating, empty-list semantics, auth-method limits, and bind/locality policy before coding.
