Keep open: current main and v2026.6.8 still do not expose `gateway.auth.tokenScopes`, and current source/tests preserve the reported device-less shared-token WebSocket scope-clearing behavior. ## What This Changes

The issue requests a `gateway.auth.tokenScopes` config field so trusted headless or single-tenant token-auth deployments can receive static operator scopes without interactive approval.

## Best Possible Solution

Resolve and land a bounded, documented token-scope or device-token policy with security-owner approval, then close this issue from the merged implementation.

## Reproduction Assessment

Yes at source level. Current main lacks `gateway.auth.tokenScopes`, checked-in tests show device-less shared-token WebSocket auth reports `auth.scopes=[]`, and `sessions.messages.subscribe` requires `operator.read`; I did not run a live ECS/Fargate repro.

## Solution Assessment

Unclear until maintainers resolve the linked PR. A bounded static token-scope config is plausible and safer than trusting client-declared scopes, but privileged grants need explicit security bounds and approval.
