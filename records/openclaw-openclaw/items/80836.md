## Summary

Keep open: current main still lacks a gateway auth config surface for token-derived scopes, and the existing gateway path deliberately clears or pairing-validates unbound shared-auth scopes. The related implementation PR at https://github.com/openclaw/openclaw/pull/52059 is open but not merged and still needs proof/security policy review, so this should not be closed as cleanup.

## What This Changes

The issue requests a bounded `gateway.auth.tokenScopes`-style config field for headless token-mode deployments to grant operator scopes without repeated interactive scope-upgrade approval.

## Best Possible Solution

Define a bounded core gateway auth-scope contract, then land https://github.com/openclaw/openclaw/pull/52059 or a replacement with explicit token/password gating, locality/admin policy, schema/help/docs alignment, regression tests, and redacted real gateway proof.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested config field is absent, while current tests and gateway code intentionally clear or pairing-validate shared-token scopes.

## Solution Assessment

Unclear whether the exact `tokenScopes` shape is the best final API. A first-class bounded config field is plausible, but maintainers should decide the auth boundary and reconcile it with the open `gateway.auth.scopes` PR before implementation.
