## Summary

Keep open: current main still exposes `sessionKey` and `agentId` on hook context but not on the `before_agent_start` or `agent_end` event payloads. The PR is not mergeable as-is because it targets the old hook contract layout, includes a type error in new coverage, and still lacks required real behavior proof from a plugin or memory run.

## What This Changes

The PR adds optional `sessionKey` and `agentId` fields to `before_agent_start` and `agent_end` hook events, forwards them from embedded runner paths, and adds hook regression tests.

## Best Possible Solution

Land a rebased additive hook contract in the current split hook files, mirror `sessionKey` and `agentId` into the two event payloads only when present, keep absent-field compatibility, add focused tests on current runner/helper surfaces, and include redacted real plugin-run proof.

## Reproduction Assessment

Yes, at source level: current main has `sessionKey` and `agentId` on `PluginHookAgentContext`, but the `before_agent_start` and `agent_end` event types and emitted event objects omit those fields. I did not run a live memory plugin in this read-only review.

## Solution Assessment

No, not as-is: adding optional event fields is a narrow compatible direction, but this branch must be rebased onto the split hook contracts, the test typing issue must be fixed, and real after-fix plugin-run proof must be provided before merge.
