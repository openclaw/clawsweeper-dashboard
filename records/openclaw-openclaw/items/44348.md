## Summary

Keep open: current main still exposes `sessionKey` and `agentId` only through `PluginHookAgentContext`, not through the `before_agent_start` or `agent_end` event payloads, so the central request is not implemented. The PR is useful but stale against the current hook type split and current hook emitter set.

## What This Changes

The PR adds optional `sessionKey` and `agentId` to `before_agent_start` and `agent_end` hook event payloads, threads them through embedded-runner hook calls, and adds regression coverage.

## Best Possible Solution

The end state should be an additive plugin hook contract that mirrors `sessionKey` and `agentId` from the existing hook context into event payloads wherever those events are emitted, while preserving legacy absent-field behavior.

## Reproduction Assessment

Yes, by source inspection. Current main's event types and embedded-runner event objects omit `sessionKey` and `agentId`, while the second hook context argument already carries those values.

## Solution Assessment

No, not as-is. The proposed direction is narrow and maintainable, but the PR must be rebased onto the current split hook type modules and updated for all current hook emitters before it is the best fix.
