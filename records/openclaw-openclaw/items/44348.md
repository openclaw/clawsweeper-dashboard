## Summary

Keep this PR open for maintainer review. Current main still exposes `agentId` and `sessionKey` through the hook context, not the legacy `before_agent_start` and `agent_end` event payloads targeted here, so the requested behavior is not implemented on main; the PR is also dirty against current main and needs to be rebased onto the split hook type files before it can be merged safely.

## What This Changes

The PR adds optional `sessionKey` and `agentId` fields to legacy `before_agent_start` and `agent_end` hook event payloads, threads them through embedded-runner hook calls when present, and adds regression coverage for populated and omitted legacy fields.

## Best Possible Solution

If maintainers accept the API shape, rebase onto current main by adding optional identity fields to `src/plugins/hook-before-agent-start.types.ts` and `src/plugins/hook-types.ts`, populate them from existing hook context only when present, keep backward-compatible omission behavior, add focused regression coverage, and update `docs/plugins/hooks.md`. If maintainers prefer the existing second-argument context contract, close the PR manually with that rationale.

## Reproduction Assessment

Yes. A source inspection against current main is enough to reproduce the API gap: `agentId` and `sessionKey` are present on `PluginHookAgentContext`, while `before_agent_start` and `agent_end` event objects do not expose those fields.

## Solution Assessment

Unclear until maintainer API review. The proposed optional fields are a narrow backward-compatible way to help event-first plugin handlers, but the current PR needs a rebase onto the split hook type files and docs, and maintainers may instead decide that plugins should read identity from the existing `ctx` argument.
