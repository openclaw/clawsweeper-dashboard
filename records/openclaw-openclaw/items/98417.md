## Summary

Keep open: this PR still appears necessary and now has sufficient real behavior proof. Current main keeps long-running MCP sessions alive and relies on the SDK refresh path, while this patch adds a bounded OAuth-only session-recreate retry for stale streamable HTTP MCP sessions with focused regressions and a redacted runtime transcript.

## What This Changes

The PR marks resolved MCP transports as OAuth-backed, recreates stale streamable HTTP MCP sessions after OAuth 401 failures, and adds regression coverage for catalog refresh, tool-call retry, and backoff preservation.

## Best Possible Solution

Land this reactive session-recreate fix after auth/runtime maintainer review and required checks; if it lands, close https://github.com/openclaw/openclaw/issues/98377 and redirect or close the alternate https://github.com/openclaw/openclaw/pull/98407 path.

## Reproduction Assessment

Yes for source-level reproduction: the linked report gives concrete token-expiry steps, current main reuses persistent MCP sessions, and the pinned SDK can throw StreamableHTTPError(401) after successful auth on the old streamable HTTP session. I did not run a live one-hour current-main expiry test in this read-only review.

## Solution Assessment

Yes, this appears to be the best bounded fix: it recreates only OAuth streamable HTTP MCP sessions after the SDK refresh path still receives a 401, without adding timers, config, or a parallel auth path. The main alternative is the open proactive-refresh PR, which is broader and already has separate review concerns.
