## Summary

Keep open: current main and v2026.6.9 still let `cacheRetention: "standard"` fall through to `undefined` before Bedrock Claude receives it, and the two linked closing pull requests are open rather than merged.

## What This Changes

The issue asks OpenClaw to normalize, stop emitting, or reject Bedrock Claude `cacheRetention: "standard"` instead of silently dropping the explicit setting.

## Best Possible Solution

Land one canonical resolver or validation/doctor fix with focused regression coverage for Bedrock Claude and sibling cache-retention families, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, source-level. Current main classifies explicit Bedrock Claude cache config into the Anthropic Bedrock family, then accepts only `none`, `short`, and `long`, so `standard` resolves to `undefined` before stream options are forwarded.

## Solution Assessment

Unclear as a final policy choice. The central resolver is the right narrow boundary for a runtime fix, but maintainers still need to choose between documenting an alias, validating invalid input, or migrating it with doctor before one linked PR lands.
