## Summary

Keep open: current main and the latest release still accept only `none`, `short`, and `long`, so Bedrock Claude `cacheRetention: "standard"` falls through to `undefined`; the two linked pull requests remain open implementation candidates rather than merged fixes.

## What This Changes

The issue reports that Bedrock Claude model params using `cacheRetention: "standard"` are accepted in config but silently dropped by the runtime cache-retention resolver.

## Best Possible Solution

Land one narrow resolver or config-validation fix that handles `standard` deterministically for Bedrock Claude, with regression coverage for Bedrock and sibling Anthropic cache-retention paths, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. Calling the current resolver with `cacheRetention: "standard"`, provider `amazon-bedrock`, model API `openai-completions`, and a Claude Bedrock model id follows the current-main path to `undefined`; I did not run a live Bedrock request in this read-only review.

## Solution Assessment

Yes, the best fix boundary is the central cache-retention resolver or the adjacent config validation/default path. The issue should stay open while maintainers review and land one of the linked pull requests or a narrow replacement.
