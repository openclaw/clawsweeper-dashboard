## Summary

Keep open: current main still rejects `cacheRetention: "standard"` before Bedrock Claude request options receive it, and the two closing pull requests are open candidates rather than merged fixes.

## What This Changes

The issue asks OpenClaw to stop silently discarding `cacheRetention: "standard"` for Bedrock Claude models, either by normalizing it, emitting canonical defaults, or warning on invalid config.

## Best Possible Solution

Land one narrow resolver or config-validation fix that handles `standard` deterministically for Bedrock Claude, with regression coverage for Bedrock and sibling Anthropic cache-retention paths.

## Reproduction Assessment

Yes, at source level: `resolveCacheRetention({ cacheRetention: "standard" }, "amazon-bedrock", "openai-completions", "us.anthropic.claude-opus-4-7")` reaches the `anthropic-bedrock` family path and returns `undefined`. I did not execute a live Bedrock request in this read-only review.

## Solution Assessment

Yes in boundary, but not yet on main: the maintainable fix belongs in the central cache-retention resolver or config validation/default path, not in the Bedrock provider fallback alone. One of the open linked PRs should land or be replaced before this issue closes.
