## Summary

Keep open: current main still allows same-provider cross-context message sends by default, but this PR is not ready to merge because the Telegram topic handling can escape the bound topic, the branch reaches into stale Telegram internals, and the external proof gate is still only unit-level.

## What This Changes

The PR flips `tools.message.crossContext.allowWithinProvider` to opt-in, updates config help/changelog and outbound tests, and also includes unrelated failover retry and UI scroll changes.

## Best Possible Solution

Land a refreshed, narrow outbound-policy hardening patch that keeps provider-specific topic identity behind channel/plugin contracts, adds regression coverage for same-provider and Telegram topic cases, and includes redacted real bound-chat proof.

## Reproduction Assessment

Yes, by source inspection. Current main returns early for same-provider contexts unless `allowWithinProvider` is explicitly `false`, and the focused tests only cover that opt-out case.

## Solution Assessment

No. The desired default-deny end state is sound, but this branch should be narrowed, refreshed against current main, and changed to use a plugin-owned Telegram same-context contract before merge.
