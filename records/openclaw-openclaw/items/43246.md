## Summary

Keep open. The intended outbound-scope hardening is still needed on current main, but this PR has blocking correctness and security findings around Telegram topic handling, generated config drift, and unrelated bundled changes.

## What This Changes

The PR changes message-tool same-provider cross-context sends to opt-in, updates related config help/tests/changelog, adds Telegram topic handling, and also touches failover retry classification plus Control UI scroll fallback behavior.

## Best Possible Solution

Land a narrow maintainer-reviewed outbound policy hardening patch: flip same-provider forwarding to explicit opt-in, keep the legacy override, compare Telegram topics through a plugin-owned seam, require topic identity for topic-bound contexts, regenerate config artifacts, and split unrelated edits.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists on current main by calling `enforceCrossContextPolicy` or `runMessageAction` with default config, the same `currentChannelProvider`, a bound `currentChannelId`, and a different explicit same-provider target; source and tests show that path is currently allowed unless `allowWithinProvider` is set to false.

## Solution Assessment

No. Default-deny is the right direction, but this PR is not the best implementation because the Telegram helper uses an invalid/core-to-plugin path, topic-bound Telegram contexts can still escape to the base chat, generated config output is stale, and unrelated changes are bundled in.
