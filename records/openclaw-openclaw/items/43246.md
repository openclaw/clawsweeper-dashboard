## Summary

Keep open: current main still permits same-provider target hops by default, so the central security hardening remains relevant, but this PR is not mergeable as-is because it lacks real bound-chat proof and introduces Telegram topic/core-boundary issues.

## What This Changes

The PR flips same-provider `message` cross-context sends to default-deny, updates config help/tests/changelog, adds Telegram topic handling, and includes unrelated failover retry and UI scroll changes.

## Best Possible Solution

Land a narrowed outbound-policy fix on current main that defaults same-provider sends to deny, uses plugin-owned same-context metadata for Telegram topics, updates focused config docs/tests, and leaves unrelated changes out.

## Reproduction Assessment

Yes, by source inspection on current main: a bound same-provider send with default config returns when `allowWithinProvider !== false`. I did not run a live chat setup.

## Solution Assessment

No. Default-deny same-provider forwarding is the right end state, but this branch should be rebased and narrowed, use a plugin-owned Telegram context contract, require topic IDs for topic-bound contexts, and add real behavior proof.
