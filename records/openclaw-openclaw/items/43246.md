## Summary

This PR addresses a real source-reproducible outbound message isolation gap, but it should stay open because the branch is conflicting with current main, still has Telegram topic-safety findings, includes unrelated failover/UI edits, and lacks after-fix real bound-chat proof.

## What This Changes

The PR changes message-tool same-provider cross-context sends from default-allow to default-deny, updates config help/tests/changelog, adds Telegram topic handling, and also includes unrelated failover and UI scroll changes.

## Best Possible Solution

Land a narrow current-main patch that flips the same-provider default to opt-in, preserves the legacy override, uses plugin-owned Telegram threading contracts, rejects topic-less targets from topic-bound contexts, and documents the migration.

## Reproduction Assessment

Yes, by source inspection: current main returns early for same-provider sends unless `allowWithinProvider` is explicitly false, and the focused tests only cover that opt-out case. I did not run tests in this read-only review.

## Solution Assessment

No. Default-deny is the right direction, but the branch should be rebased to current main, move Telegram equivalence through the plugin/threading seam, block topic-less topic-bound targets, and drop unrelated edits before merge.
