## Summary

Keep open: current main still has the permissive same-provider default, but this PR is not merge-ready because it lacks real behavior proof and its Telegram topic exception can let topic-bound actions leave the bound topic.

## What This Changes

The PR flips message-tool same-provider cross-context sends to default-deny, updates config help/tests/changelog, adds a Telegram topic exception, and includes unrelated failover and UI scroll edits.

## Best Possible Solution

Refresh this into a narrow outbound-policy hardening patch that flips the default, routes provider-specific topic equivalence through the channel/plugin threading contract, blocks topic-less topic-bound Telegram escapes, drops unrelated edits, and includes redacted real chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `allowWithinProvider !== false` returns before target comparison, and current tests only assert same-provider blocking for explicit opt-out config.

## Solution Assessment

No. Default-deny is the right direction, but this branch should use the plugin/channel threading seam for Telegram and must block topic-less topic-bound targets before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect
