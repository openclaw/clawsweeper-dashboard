## Summary

Keep this PR open because current main still has the reported same-provider routing gap, but the branch is not merge-ready: it lacks real behavior proof, crosses the Telegram plugin boundary, and introduces a topic-bound Telegram escape.

## What This Changes

The PR changes outbound message policy so same-provider cross-context sends are denied by default unless `tools.message.crossContext.allowWithinProvider=true` is set, with config/help/test/changelog updates and some unrelated failover/UI edits.

## Best Possible Solution

Refresh this into a narrow outbound-policy hardening patch that flips the default, keeps provider-specific topic matching behind channel/plugin threading contracts, blocks topic-less topic-bound escapes, drops unrelated edits, and includes redacted real chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows same-provider bound sends return early unless `allowWithinProvider` is explicitly `false`, and current tests only block that explicit opt-out case; I did not run tests because this is a read-only review.

## Solution Assessment

No. Default-deny is the right direction, but this branch is not the best implementation because it pulls Telegram parsing into core and permits topic-bound Telegram actions to target the base chat without a topic id.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route Telegram matching through plugin threading:** `src/infra/outbound/outbound-policy.ts:7`
  - body: Core outbound policy should not import Telegram target parsing directly. Telegram already exposes topic identity through its plugin threading contract, so same-context matching should use that owner seam before changing the global default.
  - confidence: 0.9
- **[P2] Require topic IDs in topic-bound Telegram contexts:** `src/infra/outbound/outbound-policy.ts:90-92`
  - body: When `currentThreadTs` is set, treating a topic-less Telegram target as same-context allows guarded plugin actions such as `sticker` to deliver to the base chat because that path only forwards `messageThreadId` when `threadId` is explicitly present.
  - confidence: 0.91
