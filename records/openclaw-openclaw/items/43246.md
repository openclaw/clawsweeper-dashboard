## Summary

Keep open: current main still permits default same-provider target hops, so the central security hardening remains useful, but this PR cannot merge as-is because it lacks real behavior proof, is merge-conflicting, includes unrelated changes, and introduces Telegram plugin-boundary/topic-scope problems.

## What This Changes

The PR flips `tools.message.crossContext.allowWithinProvider` to opt-in, updates config help/changelog/outbound tests, adds Telegram topic special casing, and includes unrelated failover retry and UI scroll changes.

## Best Possible Solution

Land a narrowed current-main outbound policy patch that flips same-provider sends to opt-in through plugin-owned same-context/thread contracts, adds focused regression tests, includes redacted real bound-chat proof, and splits unrelated failover/UI edits.

## Reproduction Assessment

Yes, by source inspection. Current main returns early when `allowWithinProvider !== false` before comparing the target, and current tests cover only the explicit `false` opt-out case.

## Solution Assessment

No. Default-deny same-provider forwarding is the right end state, but this branch is not the best implementation until Telegram topic identity uses a plugin-owned contract, topic-less targets are denied from topic-bound contexts, unrelated edits are split, and real behavior proof is added.
