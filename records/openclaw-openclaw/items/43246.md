## Summary

Keep open: current main still has the same-provider cross-context default bug, and the linked closed issue explicitly delegates the remaining work to this PR. The branch is not merge-ready because it lacks real behavior proof, is merge-conflicting, crosses the Telegram plugin boundary, and still permits a topic-bound Telegram action to escape to the base chat.

## What This Changes

The PR default-denies same-provider cross-context message sends, updates config help/tests/changelog, and also changes unrelated failover retry and UI scroll behavior.

## Best Possible Solution

Land a narrow, security-reviewed outbound policy patch on current main that flips same-provider sends to opt-in, preserves existing overrides, handles Telegram topic identity through a plugin-owned contract, updates focused docs/tests, and removes unrelated edits.

## Reproduction Assessment

Yes, by source inspection. On current main, a bound same-provider send with default config returns at `allowWithinProvider !== false` before the target comparison, and current tests only cover the explicit false case.

## Solution Assessment

No. Default-deny same-provider forwarding is the right end state, but this branch is not the best implementation until the Telegram plugin-boundary and topic-escape issues are fixed, unrelated edits are split, and real behavior proof is added.
