## Summary

Keep this PR open because current main still has the reported permissive same-provider default, but the branch should not merge as-is: it lacks real behavior proof, is conflicting, includes unrelated changes, and its Telegram topic exception creates a same-chat escape from topic-bound contexts.

## What This Changes

The PR flips same-provider message-tool cross-context sends to opt-in by default, updates config help/types/tests/changelog, and also carries unrelated failover retry and UI scroll changes.

## Best Possible Solution

Refresh this into a narrow outbound-policy hardening patch that flips the default through the effective message config, keeps provider-specific matching behind plugin threading contracts, blocks topic-less sends from topic-bound Telegram sessions, drops unrelated changes, and includes redacted real chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows same-provider bound sends return early unless `allowWithinProvider` is explicitly false, and current tests only cover the explicit opt-out path; I did not run tests in this read-only review.

## Solution Assessment

No. The default-deny direction is sound, but this branch is not the best mergeable solution because Telegram matching belongs behind a plugin seam, topic-bound topic-less targets remain allowed, unrelated changes are mixed in, and real behavior proof is missing.
