## Summary

Keep open: current main still allows same-provider bound message sends by default, so the PR addresses a real issue, but the branch is not mergeable as-is because it lacks real behavior proof and introduces Telegram topic-boundary/security problems.

## What This Changes

The PR flips same-provider message-tool cross-context sends to explicit opt-in, updates config help/types/tests/changelog, and also includes unrelated failover retry and UI scroll changes.

## Best Possible Solution

Refresh this into a narrow outbound-policy hardening patch that flips the default, keeps provider-specific chat/topic matching behind plugin threading contracts, blocks topic-bound topic-less escapes, drops unrelated changes, and includes redacted real chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows same-provider bound sends return early unless `allowWithinProvider` is explicitly false; I did not run tests in this read-only review.

## Solution Assessment

No. The default-deny direction is sound, but this branch is not the best mergeable solution because it crosses the Telegram plugin boundary, permits a topic-bound escape, includes unrelated changes, and lacks real behavior proof.

  - body: Core outbound policy should not import Telegram target parsing directly. Telegram already owns chat/topic threading identity through its plugin contract, so same-context matching should flow through that seam before changing the default policy.
