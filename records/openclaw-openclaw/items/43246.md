## Summary

Keep open: current main still allows same-provider cross-context message sends by default, so the hardening target is real, but this PR should not merge as-is because the head is dirty and has concrete correctness/security blockers.

## What This Changes

The PR flips message-tool same-provider cross-context sends to opt-in, updates related tests/config help/changelog, adds Telegram topic special-casing, and also includes unrelated failover retry and Control UI scroll edits.

## Best Possible Solution

Land a narrowed maintainer-reviewed hardening patch that flips same-provider forwarding to explicit opt-in, preserves the legacy override, handles provider-specific target matching through plugin-owned seams, requires strict Telegram topic identity, regenerates config artifacts, credits the changelog entry, and drops unrelated edits.

Yes. ## Solution Assessment

No. Default-deny is the right hardening direction, but this PR is not the best implementation as-is because it is dirty, crosses the plugin boundary for Telegram parsing, leaves a topic-bound escape, misses generated config help, and includes unrelated edits.
