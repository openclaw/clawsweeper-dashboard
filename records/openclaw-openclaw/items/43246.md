## Summary

Keep open: current main still allows the reported same-provider hop, and the PR is useful hardening, but the latest branch still has build-boundary, Telegram topic-scope, generated schema, and changelog blockers before merge.

## What This Changes

The PR changes message outbound policy/config help/tests/changelog to default-deny same-provider cross-context sends, adds Telegram topic handling, and also touches failover retry classification and Control UI scroll fallback behavior.

## Best Possible Solution

Land a narrow maintainer-reviewed outbound policy hardening patch that flips the same-provider default, preserves explicit opt-ins, handles Telegram topics through a plugin-owned API/seam, regenerates config artifacts, credits the contributor, and splits unrelated edits.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction: call `enforceCrossContextPolicy` or `runMessageAction` with a bound same-provider context, default config, and a different explicit target; the policy returns early because `allowWithinProvider` defaults to true.

## Solution Assessment

No. Default-deny same-provider forwarding is the right end state, but this PR is not the best implementation until the Telegram boundary/topic bugs, generated schema drift, changelog attribution, and unrelated edits are fixed.
