## Summary

Keep open. Current main still has the permissive same-provider default, so this PR addresses a real bug, but it is not merge-ready because after-fix real behavior proof is missing and the branch still has Telegram topic/boundary correctness issues plus unrelated edits.

## What This Changes

The PR changes message cross-context policy to deny same-provider target hops by default, updates config help/tests/changelog, adds Telegram topic handling, and includes unrelated failover retry and UI scroll changes.

## Best Possible Solution

Land a narrow, security-reviewed outbound policy change on current main that flips same-provider sends to opt-in, preserves legacy overrides, handles channel-specific context through plugin contracts, updates focused docs/tests, and excludes unrelated edits.

## Reproduction Assessment

Yes, by source inspection on current main. A bound same-provider send with default config returns when `allowWithinProvider !== false`; I did not run a live chat setup.

## Solution Assessment

No. Default-deny same-provider forwarding is the right end state, but this branch should be rebased and narrowed, use a plugin-owned Telegram same-context contract, require topic IDs for topic-bound contexts, and add real behavior proof.
