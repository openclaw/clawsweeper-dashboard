## Summary

Keep this PR open because current main still has the reported permissive same-provider default, but the branch is not merge-ready: it adds Telegram-specific parsing in core, leaves a topic-bound Telegram escape, lacks real behavior proof, and includes unrelated failover/UI edits.

## What This Changes

The PR changes message-tool same-provider cross-context sends to default-deny, updates config help/tests/changelog, adds a Telegram topic exception, and includes unrelated failover and UI scroll changes.

## Best Possible Solution

Refresh this into a narrow outbound-policy hardening patch that flips the default, uses the channel/plugin threading contract for provider-specific topic equivalence, blocks topic-less topic-bound Telegram escapes, drops unrelated edits, and includes redacted real chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows same-provider bound sends return early unless `allowWithinProvider` is explicitly false, and current tests only block that explicit opt-out path.

## Solution Assessment

No. Default-deny is the right direction, but this branch is not the best implementation because it pulls Telegram parsing into core and permits topic-bound Telegram actions to target the base chat without a topic id.
