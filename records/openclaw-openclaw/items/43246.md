## Summary

Keep open: current main still allows same-provider cross-context message sends by default, but this PR is not merge-ready because its Telegram topic handling has a remaining escape, the Telegram import crosses the plugin boundary, generated config help is stale, and the branch bundles unrelated changes.

## What This Changes

This PR flips message-tool same-provider cross-context sends to opt-in, updates related config help/tests/changelog, adds Telegram topic handling, and includes unrelated failover retry and Control UI scroll fallback changes.

## Best Possible Solution

Land a narrow maintainer-reviewed outbound policy hardening patch: default-deny same-provider forwarding, preserve explicit opt-ins, compare Telegram topics through a plugin-owned seam, require topic identity for topic-bound contexts, regenerate config artifacts, credit the contributor, and split unrelated edits.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main by calling `enforceCrossContextPolicy` or `runMessageAction` with a bound same-provider context, default config, and a different explicit target; the code returns early because `allowWithinProvider` defaults to true.

## Solution Assessment

No. Default-deny is the right direction, but this PR is not the best implementation until the Telegram boundary/topic bugs, generated schema drift, changelog attribution, and unrelated edits are fixed.
