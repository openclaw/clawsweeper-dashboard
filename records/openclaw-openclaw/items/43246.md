## Summary

Keep this PR open for maintainer follow-up: the underlying same-provider cross-context message bug is still present on current main, but the proposed branch is not merge-ready because it has blocking correctness/security findings and carries unrelated failover/UI changes.

## What This Changes

The PR changes outbound message policy to default-deny same-provider cross-context sends, updates config help/types/tests/changelog, adds Telegram topic handling, and also changes failover retry classification plus Control UI scroll fallback behavior.

## Best Possible Solution

Refresh this as a narrow outbound policy hardening patch: flip same-provider forwarding to explicit opt-in, preserve the legacy override, compare channel/thread identity through a supported plugin-owned seam, require topic identity for topic-bound Telegram contexts, regenerate config schema artifacts, and split unrelated failover/UI work into separate PRs.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is `enforceCrossContextPolicy` or `runMessageAction` on current main with default config, matching `currentChannelProvider`, a bound `currentChannelId`, and a different explicit same-provider target; the policy returns early at the permissive default.

## Solution Assessment

No. The default-deny direction is correct, but this PR is not the best implementation as written because it imports through an invalid Telegram path, leaves a topic-bound Telegram escape, misses generated config output, and mixes unrelated changes.
