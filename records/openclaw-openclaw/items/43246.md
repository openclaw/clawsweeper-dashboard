## Summary

Keep open. Current main still allows the reported same-provider bound-send hop, and this PR addresses the central hardening direction, but it is not mergeable as-is because it lacks real behavior proof and introduces Telegram topic-routing/security defects.

## What This Changes

The PR flips outbound `allowWithinProvider` to opt-in, updates config help/types/tests/changelog, adds Telegram same-context handling, and includes unrelated failover/UI scroll changes.

## Best Possible Solution

Refresh this into a narrow outbound-policy hardening patch that flips the default, keeps provider-specific topic matching behind channel/plugin threading contracts, blocks topic-less topic-bound escapes, drops unrelated edits, and includes redacted real chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows same-provider bound sends return early unless `allowWithinProvider` is explicitly `false`; I did not run tests because this is a read-only review.

## Solution Assessment

No. The default-deny direction is sound, but this branch crosses the Telegram plugin boundary, permits a topic-bound escape, carries unrelated changes, and lacks real behavior proof.

  - body: Core outbound policy should not import Telegram target parsing directly. Telegram already exposes chat/topic identity through its plugin threading contract, so same-context matching should use a channel-owned seam before changing the global default.
