## Summary

Keep open: current main still has the permissive same-provider default, but this PR is not mergeable as-is because it lacks real behavior proof and introduces Telegram topic/security issues plus unrelated changes.

## What This Changes

The branch flips same-provider message sends from opt-out to opt-in, updates config docs/tests/changelog, and includes unrelated failover retry and UI scroll changes.

## Best Possible Solution

Land a refreshed, narrow outbound-policy hardening patch that defaults bound same-provider sends to denied, keeps provider-specific topic identity behind plugin/threading contracts, removes unrelated changes, and includes redacted real bound-chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows same-provider bound sends return early unless `allowWithinProvider` is explicitly `false`, and focused tests only cover that opt-out path; I did not run tests in this read-only review.

## Solution Assessment

No. The default-deny direction is sound, but this branch is not the best mergeable form because it crosses the Telegram plugin boundary, permits a topic-bound escape, includes unrelated changes, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route Telegram matching through a plugin-owned seam:** `src/infra/outbound/outbound-policy.ts:7`
  - body: Core outbound policy should not import Telegram target parsing directly. Telegram now lives under `extensions/telegram` and exposes threading/target behavior through plugin seams, so same-context matching should be provided by that contract before this default is flipped.
  - confidence: 0.9
- **[P2] Require topic IDs for topic-bound Telegram targets:** `src/infra/outbound/outbound-policy.ts:90-92`
  - body: When `currentThreadTs` is set, this helper treats `telegram:<chat>` as same-context because the target has no `messageThreadId`.   - confidence: 0.91
