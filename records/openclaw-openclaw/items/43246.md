## Summary

Keep this PR open: current main still has the reported permissive same-provider default, but the proposed patch is not merge-ready because it introduces a bad Telegram dependency path, leaves a topic-bound Telegram escape, misses generated config schema updates, and mixes unrelated failover/UI changes into a security hardening PR.

## What This Changes

The PR changes outbound message policy to default-deny same-provider cross-context sends, updates config help/types/tests/changelog, adds Telegram topic handling, and also changes failover retry classification plus Control UI scroll fallback behavior.

## Best Possible Solution

Land a narrow maintainer-reviewed outbound policy hardening patch: flip same-provider forwarding to explicit opt-in, preserve the legacy override, compare Telegram topic identity through a plugin-owned seam, require topic identity for topic-bound contexts, regenerate config artifacts, and split unrelated failover/UI edits into separate work.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists on current main by calling `enforceCrossContextPolicy` or `runMessageAction` with default config, matching `currentChannelProvider`, a bound `currentChannelId`, and a different explicit same-provider target; the policy returns early at the permissive default.

## Solution Assessment

No. The default-deny direction is right, but this PR is not the best implementation as written because the Telegram helper crosses an invalid boundary/path, topic-bound Telegram sessions can still escape, generated config output is stale, and unrelated changes are bundled in.
