## Summary

Keep this PR open: current main still has the reported permissive default, but this branch needs real behavior proof and fixes for Telegram topic isolation/core-plugin boundary issues before merge.

## What This Changes

The PR flips same-provider message cross-context sends to default-deny, updates config docs/tests/changelog, and also carries unrelated failover retry and UI scroll changes.

## Best Possible Solution

Land a refreshed, narrow outbound-policy hardening patch that defaults bound same-provider sends to denied, keeps provider-specific topic identity behind plugin/threading contracts, removes unrelated changes, adds focused regression tests, and includes redacted real bound-chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows same-provider bound sends return early unless `allowWithinProvider` is explicitly `false`, and focused tests only cover that opt-out case.

## Solution Assessment

No. The default-deny direction is sound, but this branch is not the best mergeable form because it pulls Telegram-specific parsing into core, permits topic-less sends from topic-bound contexts, and includes unrelated changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route Telegram matching through a plugin-owned seam:** `src/infra/outbound/outbound-policy.ts:7`
  - body: Core outbound policy should not import Telegram target parsing directly. Telegram now lives under `extensions/telegram` and exposes behavior through plugin/SDK seams, so same-context matching should be provided by a channel/threading contract before this default is flipped.
  - confidence: 0.9
- **[P2] Require topic IDs for topic-bound Telegram targets:** `src/infra/outbound/outbound-policy.ts:90-92`
  - body: When `currentThreadTs` is set, this helper treats `telegram:<chat>` as same-context because the target has no `messageThreadId`.   - confidence: 0.91
