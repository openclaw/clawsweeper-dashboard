## Summary

Keep open. Current main still has the permissive same-provider default and this PR targets a real outbound data-scope hardening gap, but the branch is not merge-ready because it crosses the Telegram plugin boundary, leaves a topic-bound escape, mixes unrelated edits, and only shows unit-test proof.

## What This Changes

The branch flips same-provider message cross-context sends to default-deny, updates config help/tests/changelog, adds Telegram topic handling, and also changes failover retry classification plus Control UI scroll fallback behavior.

## Best Possible Solution

Land a narrow security-reviewed outbound policy patch that flips the same-provider default, preserves explicit opt-ins, handles Telegram topics through a plugin-owned seam with exact topic identity, updates config help/tests, and splits unrelated edits.

Yes, by source inspection. On current main, call `enforceCrossContextPolicy` or `runMessageAction` with a bound same-provider context, default config, and a different explicit target; `allowWithinProvider !== false` returns before the target comparison.

## Solution Assessment

No. Default-deny same-provider forwarding is the right end state, but this branch is not the best implementation until the Telegram plugin-boundary and topic-escape bugs are fixed, unrelated edits are split, and real behavior proof is added.
