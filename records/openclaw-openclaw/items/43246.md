## Summary

Keep open: current main still has the same-provider default-allow bug, but this PR cannot merge as-is because it is conflicting with main, lacks real behavior proof, and its Telegram topic exception can allow topic-bound actions to target the base chat.

## What This Changes

The PR flips same-provider message-tool cross-context sends from default-allow to default-deny, updates config help/changelog/tests, adds a Telegram topic exception, and includes unrelated failover/UI scroll changes.

## Best Possible Solution

Refresh this into a narrow outbound-policy hardening patch that keeps the default-deny behavior, uses the channel/plugin threading seam for provider-specific equivalence, rejects topic-less targets from topic-bound Telegram contexts, drops unrelated edits, rebases cleanly, and includes redacted real chat proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the same-provider guard returns early unless `allowWithinProvider` is explicitly false, and current tests only cover the explicit opt-out case.

## Solution Assessment

No. Default-deny is the right direction, but the PR's Telegram topic handling should go through the plugin/channel threading contract and must not treat topic-less targets as same-context when a topic is bound.
