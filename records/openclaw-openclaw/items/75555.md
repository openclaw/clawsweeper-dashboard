## Summary

Keep open. Current main still matches the report: Feishu docs advertise `channels.feishu.blockStreaming`, the Feishu source/generated channel schema does not accept that key, and the reply dispatcher still forces core block streaming off; the linked open PR #75573 uses closing syntax, so this issue should stay open until that implementation is reviewed and merged or replaced.

## What This Changes

The issue requests accepting the documented `channels.feishu.blockStreaming` boolean for the Feishu channel so block-level streaming can be enabled.

## Best Possible Solution

Keep this issue open until PR #75573 or a replacement fix adds source-level Feishu support for explicit block-streaming opt-in, updates generated metadata and tests, and reconciles the dispatcher so the setting controls block replies without regressing the silent-drop guard.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: `channels.feishu.blockStreaming` is absent from the strict Feishu channel schema that config validation applies, so `config set` or `config patch` rejects the key before runtime.

## Solution Assessment

No, not as a generated-output-only or schema-only fix. The maintainable fix belongs in `extensions/feishu/src/config-schema.ts`, generated metadata, Feishu config tests, and likely `reply-dispatcher.ts` so explicit opt-in can override the current hard-disable safely.
