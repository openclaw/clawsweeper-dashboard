## Summary

Keep open. Current `main` still closes the long-lived WhatsApp inbound listener through raw `ws.close`, the shared WhatsApp close helper has the same policy, the pinned Baileys version exposes `end(error)` with broader teardown behavior, and the same-author fix PR is closed unmerged.

## What This Changes

The issue requests changing WhatsApp socket teardown from raw `sock.ws?.close()` to Baileys `sock.end(error)` with a fallback to prevent zombie sockets after reconnects.

## Best Possible Solution

Fix this in the WhatsApp plugin by teaching the inbound listener and shared WhatsApp socket close helper to prefer Baileys `end(error)` when present, keep raw `ws.close` as a compatibility fallback, and update teardown tests so the current raw-close behavior is no longer locked in.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current `main`: `listener.close()` calls `closeInboundMonitorSocket(sock)`, which calls only `sock.ws?.close?.()`, while the pinned Baileys source exposes `end(error)` with the broader cleanup path described in the report.

## Solution Assessment

Yes. The narrow maintainable fix is to change the WhatsApp plugin-owned teardown helpers and tests; no core change or new plugin API is needed.
