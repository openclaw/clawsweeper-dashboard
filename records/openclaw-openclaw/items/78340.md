## Summary

Keep open: the PR targets the right Control UI bug, but the fallback implementation can still let stale legacy config override an explicit `tools.exec.security: "allowlist"`, and the external PR needs observed after-fix real behavior proof before merge. Terminal screenshots, console output, copied live output, linked artifacts, recordings, or redacted logs count; updating the PR body should trigger a fresh ClawSweeper review, or a maintainer can comment `@clawsweeper re-review`.

## What This Changes

The PR changes `extractQuickSettingsSecurity` in the Control UI to read `tools.exec.security` before falling back to the legacy `agents.defaults.exec.security` path.

## Best Possible Solution

Keep the Quick Settings badge aligned with the runtime config contract by treating `tools.exec.security` presence separately from its value, falling back only when that path is absent, and covering the precedence case in a focused UI test.

## Reproduction Assessment

Yes. Source inspection reproduces the current-main bug: render Quick Settings with `configForm` or `configSnapshot.config` containing `tools.exec.security` and no `agents.defaults.exec.security`, and current main still passes the default `allowlist` badge value.

## Solution Assessment

No. The direction is right, but the proposed patch uses `allowlist` as an absence sentinel; the safer fix is to track whether `tools.exec.security` was present and only then decide whether legacy fallback applies.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve explicit allowlist before falling back:** `ui/src/ui/app-render.ts:592`
