## Summary

Keep open. This protected maintainer PR still needs review, and the diff appears to introduce a Control UI ordering regression by skipping the authoritative reload after message-phase updates without preserving the `sessions.list` newest-first order locally.

## What This Changes

The PR skips full Control UI session-list reloads for locally applied message updates, carries known session keys through transcript update events, and avoids eager media provider discovery for explicit generation model config.

## Best Possible Solution

Keep the latency reduction, but make the local session patch preserve the same newest-first ordering and reload semantics as `sessions.list` before skipping the server refresh.

## Reproduction Assessment

Yes. A source-level reproduction is to start with two session rows, apply a message-phase update that gives the lower row a newer `updatedAt`, and observe that the PR path returns without `loadSessions` while the local patch keeps the row at its old index.

## Solution Assessment

No. Skipping the reload is only safe if local state maintains the authoritative list contract; the safer fix is to reorder/filter the patched rows locally or keep the reload when a message update can change ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve session ordering before skipping reloads:** `ui/src/ui/app-gateway.ts:751-756`
  - body: `sessions.list` returns rows sorted by newest `updatedAt`, but `applySessionsChangedEvent` updates an existing row in place. With this early return, a message update that makes an older row newest no longer triggers the authoritative reload, so the Control UI can show stale session ordering until some later refresh. Please either resort the local rows to match `sessions.list` or keep the reload when the update can affect ordering.
  - confidence: 0.86
