## Summary

Keep open. This is a protected maintainer PR, it is currently reported unmergeable, and the latest reviewed diff still has a correctness issue in the Control UI session update path that should block merge until fixed.

## What This Changes

The PR reduces Control UI/Gateway session-list churn by skipping some `sessions.list` reloads, carrying session keys through transcript update events, and avoiding eager media provider discovery for explicit generation model config.

## Best Possible Solution

Keep the latency reduction, but make the local session patch preserve the same ordering/filtering contract as `sessions.list` before skipping the reload.

## Reproduction Assessment

Yes. A source-level reproduction is to start with two session rows, apply a message-phase update that gives the lower row a newer `updatedAt`, and observe that the PR path returns without `loadSessions` while the local patch keeps the row at its old index.

## Solution Assessment

No. Skipping the reload is only safe if local state maintains the authoritative newest-first list contract; the safer fix is to resort/filter patched rows locally or keep the reload when a message update can change ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve session order before skipping reloads:** `ui/src/ui/app-gateway.ts:751-756`
  - body: `sessions.list` returns rows sorted by newest `updatedAt`, but `applySessionsChangedEvent` updates an existing row in place. With this early return, a message update that makes an older row newest no longer triggers the authoritative reload, so the Control UI can show stale session ordering until a later refresh. Please either resort the local rows to match `sessions.list` or keep the reload when the update can affect ordering.
  - confidence: 0.86
