## Summary

Keep open: this PR targets the right guarded model-fetch layer for the linked Anthropic socket/fallback bug, but the current head is not merge-ready because the retry is narrower than the linked transient fetch-failure scope, can replay a consumed Request body, and still lacks after-fix gateway proof.

## Best Possible Solution

## Solution Assessment

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Cover the linked transient fetch failure shape:** `src/agents/provider-transport-fetch.ts:838`
  - body: The linked issue includes a cold-start Anthropic case where a bare `TypeError: fetch failed` falls into model fallback, but this branch retries only when the error or its immediate cause has `code === "UND_ERR_SOCKET"`. That path can still rethrow into cross-provider fallback while this PR closes the linked issue, so either cover the agreed transient pre-response shape here or narrow the closing claim and keep a follow-up open.
  - confidence: 0.84
- **[P2] Do not replay a consumed Request body:** `src/agents/provider-transport-fetch.ts:851-855`
  - body: On retry, this reuses the same `guardedFetchOptions`, including `baseInit.body`; when callers pass a `Request` with a body, the first `fetchWithSsrFGuard` attempt can disturb or lock that stream. The retry can then fail before making the fresh provider attempt, so rebuild or buffer retry-safe request bodies, or restrict this retry to reusable body types.
  - confidence: 0.78
