## Summary

Keep open. This PR is an active candidate for the open auth/session bug, but it is not merge-ready because the patch can select an agent-primary auth profile while the run provider/model still use the fallback, and it has no real behavior proof.

## What This Changes

The PR adds an auto-source branch in `src/agents/auth-profiles/session-override.ts` that prepends the agent default model provider's auth profile order when resolving session auth overrides.

## Best Possible Solution

Fix stale auto fallback provider, model, and auth state together in the session/model selection and inheritance paths, keep the auth resolver provider-scoped, and require focused regression coverage plus redacted runtime proof.

## Reproduction Assessment

Yes, source-reproducible. The PR defect follows from the diff plus the current caller: the resolver can return an agent-primary auth profile while the run provider/model remain on the fallback; I did not run a live MiniMax/DeepSeek setup in this read-only pass.

## Solution Assessment

No. Reordering provider-specific auth profiles inside an auth-only helper is not the best fix; the selected provider/model and auth profile need to recover together in session/model selection.
