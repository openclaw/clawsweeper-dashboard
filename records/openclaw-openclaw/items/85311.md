## Summary

Keep open: the patch changes auth-profile ordering inside a helper that returns only an auth profile, so it can select a primary-provider profile while the run is still on the fallback provider, and it does not cover the normal TUI/WebChat session path described in the linked report.

## What This Changes

This PR prepends the agent default model provider's auth profile order when `resolveSessionAuthProfileOverride` handles an auto-selected session auth profile override.

## Best Possible Solution

Fix the normal agent run/session-selection path so auto fallback model and auth overrides recover together, with focused regression coverage for TUI/WebChat new and continuing sessions plus redacted real runtime proof.

## Reproduction Assessment

Yes for the introduced patch defect by source inspection: with a DeepSeek current provider/profile and an auto source, the inserted order can make `pickNextAvailable` return a MiniMax profile while the run provider remains DeepSeek. The original TUI/WebChat report still needs real after-fix proof in this PR.

## Solution Assessment

No. Reordering provider-specific auth profiles inside a helper that returns only an auth profile is not the safest fix; recovery belongs where the model/provider session selection and auth profile selection can be updated together.

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep auth profile orders provider-specific:** `src/agents/auth-profiles/session-override.ts:187`
  - body: `resolveSessionAuthProfileOverride` only returns an auth profile id; it does not switch `provider` or `model`. With a fallback run still on `deepseek` and an auto `deepseek:default` current profile, this splice makes the order `[minimax:global, deepseek:default]`, so the later new-session `pickNextAvailable(current)` can persist `minimax:global` while the caller is still running DeepSeek. That can fail auth or forward a provider-specific credential to the wrong runtime; the fix needs to clear/re-resolve the auto fallback at the model/session selection layer instead of mixing provider orders here.
  - confidence: 0.92
- **[P1] Patch the path that creates TUI/WebChat runs:** `src/agents/auth-profiles/session-override.ts:170`
  - body: The changed resolver is only called from the `/btw` side-question path in production, while the linked Control UI/WebChat symptom is created through the normal agent run path (`agent-command.ts` -> attempt execution -> embedded runner). A fresh TUI session that starts with the wrong auth profile will not be fixed by this branch unless that path calls the new logic, so please move the repair or add an integration test proving the reported path exercises it.
  - confidence: 0.86
