## Summary

Keep open: current main still uses fixed deep Dreaming promotion thresholds and records short-term recall signals from `memory_search`, while `memory_get` follows a read-only path without the same tracking call. The linked open fix PR at https://github.com/openclaw/openclaw/pull/94783 is only a partial threshold proposal and has not merged, so this issue still owns the broader threshold plus signal-source problem.

## What This Changes

The issue asks memory-core Dreaming promotion to work with Gemini-like embedding score ranges and to account for more recall evidence so short-term memory can actually promote into durable memory.

## Best Possible Solution

Define an upgrade-safe memory-core Dreaming signal and threshold policy, then land or rework the linked PR only if it fits that policy and keep this issue open until the threshold and signal-source gaps are both intentionally handled.

## Reproduction Assessment

Yes for the source-level signal gap: `memory_search` queues `recordShortTermRecalls`, while `memory_get` only reads and returns memory content. The exact Gemini score distribution still needs live workspace proof, so the full combined issue is source-reproducible rather than fully reproduced here.

## Solution Assessment

No for a threshold-only fix. Lowering global defaults is a plausible partial mitigation, but the best solution needs maintainer-approved threshold calibration, upgrade behavior, and privacy-safe signal semantics for non-search access.
