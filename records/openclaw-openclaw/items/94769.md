## Summary

Keep open: current main still has fixed deep Dreaming promotion thresholds and records live short-term recall signals from memory_search, while memory_get/read/exec access does not feed the same signal path; the store-growth portion is partly mitigated, but the combined Gemini-threshold plus signal-source problem remains.

## What This Changes

The issue requests that memory-core Dreaming promotion work reliably for Gemini-like embedding score distributions and account for important memory access signals beyond explicit memory_search calls.

## Best Possible Solution

Define the memory-core Dreaming signal contract for explicit recall, daily/grounded evidence, memory_get access, and provider score calibration, then align defaults, ranking, tests, and docs around that contract.

## Reproduction Assessment

Yes for the memory_get signal gap at source level: memory_search queues recordShortTermRecalls while memory_get only reads and returns memory content. The exact Gemini score distribution was not live-reproduced here and remains reporter-provided evidence.

## Solution Assessment

Unclear. Lowering minScore alone is too narrow because current tests intentionally allow daily-only promotion, while recording memory_get/read/exec access needs a maintainer-owned signal and privacy policy.
