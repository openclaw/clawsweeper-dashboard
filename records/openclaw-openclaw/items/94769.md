## Summary

Keep open: current main still uses static high Dreaming promotion thresholds and `memory_get`/read/exec do not feed the `memory_search` short-term recall tracking path. The growth portion is partly mitigated by the current 512-entry cap, and daily/session ingestion already counts toward `signalCount`, but the Gemini threshold policy and non-search signal semantics remain unresolved.

## What This Changes

The issue asks memory-core Dreaming to promote short-term recall reliably with Gemini-style embedding scores and to decide whether non-search memory access should contribute promotion signals.

## Best Possible Solution

Define an upgrade-safe Dreaming promotion policy, preferably provider-aware or explicitly configurable, then align defaults, signal eligibility, docs, tests, and live Gemini-backed proof before closing this issue.

## Reproduction Assessment

Partly, but not end-to-end. Source inspection shows the static threshold gates and the missing `memory_get` recall hook on current main; the exact Gemini score distribution and live after-fix promotion behavior still need redacted runtime proof.

## Solution Assessment

Unclear. Lowering global defaults alone is only a partial mitigation because the best fix needs maintainer direction on provider calibration, upgrade behavior, and which access paths should become promotion signals.
