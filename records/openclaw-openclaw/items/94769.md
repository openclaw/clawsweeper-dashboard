## Summary

Keep open: current main still uses fixed deep-promotion thresholds and records live recall signals from memory_search, while memory_get/read/exec access does not enter the short-term recall path; overlapping open issues cover pieces but not this full Gemini-threshold plus signal-source gap.

## What This Changes

The issue requests that memory-core Dreaming promotion work with lower-scoring embedding providers and collect or clean short-term recall signals beyond explicit memory_search calls.

## Best Possible Solution

Define the Dreaming signal contract for organic recalls, daily/grounded evidence, and explicit memory reads, then adjust scoring/defaults and source-specific recording with focused tests and docs.

## Reproduction Assessment

Yes, at source level for the memory_get signal gap: memory_search queues recordShortTermRecalls while memory_get only reads and returns memory content. The exact Gemini score distribution was not live-reproduced here and remains reporter-provided evidence.

## Solution Assessment

Unclear. Lowering minScore alone is too narrow because current tests intentionally allow daily-only promotion, while recording memory_get/read/exec access needs a maintainer-owned signal and privacy policy.
