## Summary

Keep open: current main still uses fixed 0.75/0.8 promotion score defaults, memory_search remains the only live tool path that records short-term recall signals, and the open linked PR only addresses the threshold portion while leaving signal semantics and upgrade-safe default policy unresolved.

## What This Changes

The issue asks memory-core Dreaming promotion to become reachable for Gemini-like embedding scores and to decide whether memory_get, read, exec, or daily evidence should feed short-term promotion signals.

## Best Possible Solution

Define the Dreaming signal contract and threshold policy first, then land a tested, documented memory-core change that handles provider score ranges without silently changing unrelated users' durable-memory behavior.

## Reproduction Assessment

Yes for the source-level signal gap: memory_search queues recordShortTermRecalls while memory_get only reads and returns memory content, and read/exec paths do not call the same writer. The exact Gemini score distribution still needs live provider proof.

## Solution Assessment

Unclear. Lowering minScore is a plausible partial mitigation, but the best fix needs maintainer-approved threshold policy plus explicit semantics for memory_get/read/exec/daily evidence.
