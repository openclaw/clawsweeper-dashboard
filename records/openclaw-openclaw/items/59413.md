## Summary

Keep open: current main still lacks the requested per-candidate or provider/model retry-before-fallback budget for opaque proxy pools. Existing controls cover auth-profile rotation and cooldown caps, while the linked retry PR was closed unmerged and related open issues do not settle the config-policy choice.

## What This Changes

Request configurable retry-before-fallback budgets per fallback candidate or provider/model so pool-based proxy model providers can retry before falling through.

## Best Possible Solution

Have maintainers choose one canonical provider/model or candidate-scoped retry policy that composes with auth.cooldowns, timeout classification, and replay safety, then add schema, docs, logs, and focused fallback tests.

## Reproduction Assessment

Yes for the missing core capability: current source shows one outer fallback attempt per candidate and the config schema has no retry budget fields. No live TicketPro reproduction was run.

## Solution Assessment

Unclear as proposed: per-candidate retries are plausible for opaque proxy pools, but a provider/model or reason-scoped policy may be safer than ad hoc fallback-list fields.
