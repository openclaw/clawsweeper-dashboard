## Summary

Keep open: current main and the latest release still lack a public retry-before-fallback budget scoped to a fallback candidate or provider/model route, and the related open retry PRs do not yet settle the requested config contract.

## What This Changes

Add configurable per-candidate or provider/model retry-before-fallback budgets so pool/proxy providers can retry the same route before model fallback.

## Best Possible Solution

Have maintainers choose one canonical retry policy that composes provider/model route budgets with idempotent same-model retry, SDK retry caps, auth.cooldowns, timeout classification, docs, and focused fallback tests.

## Reproduction Assessment

Yes, at source level for the missing capability: current main still has one runFallbackAttempt per candidate and strict schemas reject per-candidate or provider retry-budget fields. No live TicketPro reproduction was run in this read-only review.

## Solution Assessment

Unclear as proposed: per-candidate retries are plausible for opaque proxy pools, but the best implementation needs a maintainer-chosen provider/model or candidate-scoped policy that remains idempotent and composes with cooldowns and adjacent retry work.
