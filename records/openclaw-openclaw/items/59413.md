## Summary

Keep open. Current main has model fallback plus cooldown/profile-rotation controls, but it still does not expose or apply a per-candidate or provider/model retry-before-fallback budget for pooled proxy routes. The linked retry/cooldown PR was closed unmerged, and the issue discussion identifies this remaining config granularity gap rather than a resolved duplicate.

## What This Changes

The issue requests configurable retry behavior per fallback candidate or provider/model so pooled proxy providers can retry the same route before OpenClaw falls through to the next model.

## Best Possible Solution

Add a backward-compatible retry policy at the provider/model or fallback-candidate boundary, applied inside the model fallback path before a candidate is marked failed. Preserve existing auth cooldown/profile-rotation semantics, keep permanent auth and billing failures fail-fast, and document the new operator-facing config with focused tests for timeout/502 retry success and retry exhaustion before cross-provider fallback.

## Reproduction Assessment

Yes. A focused unit test can configure a primary pooled-proxy model plus one fallback, make the primary throw timeout-classified failures, and assert current main calls that candidate only once before trying the fallback.

## Solution Assessment

No, current main is not the best solution for this use case because the existing knobs tune auth-profile rotation and cooldowns, not retries for a specific provider/model route. A narrow provider/model retry override is the safer direction than increasing global retries for every fallback candidate.
