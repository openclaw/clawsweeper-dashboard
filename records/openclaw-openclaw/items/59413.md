## Summary

Keep open: current main still lacks a public per-candidate or provider/model retry-before-fallback budget for pool-style proxy providers, and the active same-model retry replacement work does not yet resolve the per-route config decision.

## What This Changes

Request a configurable retry-before-fallback budget per fallback candidate or provider/model route for pool-based proxy providers.

## Best Possible Solution

Have maintainers choose a canonical retry policy that composes with the idempotent same-model retry work, auth.cooldowns, timeout classification, and replay safety, then add schema, docs, logs, and focused fallback tests.

## Reproduction Assessment

Yes, at source level for the missing capability: current main still performs one outer runFallbackAttempt per candidate and strict schemas reject per-candidate or provider retry-budget fields. No live TicketPro reproduction was run.

## Solution Assessment

Unclear as proposed: per-candidate retries are plausible for opaque proxy pools, but the safest solution likely needs a maintainer-chosen provider/model or candidate-scoped policy that stays idempotent and composes with existing cooldowns.
