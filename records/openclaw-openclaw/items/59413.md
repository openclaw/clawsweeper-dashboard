## Summary

Keep this open. Current main has the fast-fallback/profile-rotation work that addressed the opposite retry-too-much problem, but it still does not provide the per-candidate or provider/model retry budget requested here; the discussion around https://github.com/openclaw/openclaw/pull/55118 also narrowed the remaining ask to that missing config shape.

## What This Changes

The issue requests configurable retry attempts before model fallback for pool-based or proxy providers whose repeated attempts may hit different backend keys.

## Best Possible Solution

Add a deliberate provider/model or fallback-candidate retry policy that keeps existing defaults unchanged, documents merge precedence, and covers timeout/502 pool-provider behavior with focused regression tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this asks for a new retry configuration mode. Source inspection does confirm the current schema and fallback loop lack the requested per-candidate/provider-model retry budget.

## Solution Assessment

Unclear until maintainers choose the config model. A provider/model-level policy with optional agent override looks more maintainable than embedding ad hoc retry fields directly in every fallback string list, but the final shape should be a product/API decision.
