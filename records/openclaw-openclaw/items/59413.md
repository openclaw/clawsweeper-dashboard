## Summary

Keep open: current main still has no public provider/model or fallback-candidate retry-before-fallback budget for pool-style proxy providers, and the related same-model retry work does not settle this per-route config policy. The related https://github.com/openclaw/openclaw/issues/49185 was closed in favor of this issue, so this remains the canonical tracker for the proxy/provider retry-budget decision.

## What This Changes

Request to add a provider/model or fallback-candidate retry-before-fallback budget so proxy/key-pool providers can retry before OpenClaw advances to the next model.

## Best Possible Solution

Maintainers should choose the canonical retry policy scope, likely provider/model route first with optional candidate or agent overrides only if needed, then implement schema, docs, logging, replay-safety gates, and focused fallback tests.

## Reproduction Assessment

Yes, at source level for the missing capability: current main attempts each fallback candidate once and strict schemas reject per-candidate or provider retry-budget fields. No live TicketPro reproduction was run.

## Solution Assessment

Unclear as proposed: per-candidate retries are plausible for opaque proxy pools, but the safest design needs maintainer choice on provider/model versus candidate scope and must compose with cooldowns and idempotent retry work.
