## Summary

Keep open. Current main still has model fallback, cooldown, and auth-profile rotation controls, but it does not expose or apply a retry-before-fallback budget for a specific fallback candidate or provider/model route; the related retry PR was closed unmerged and the issue discussion identifies this exact remaining gap.

## What This Changes

Request a configuration surface that lets pooled proxy routes retry a fallback candidate or provider/model multiple times before advancing to the next model.

## Best Possible Solution

Add a backward-compatible retry policy at the provider/model or fallback-candidate boundary, apply it before a candidate is marked failed, and document the new operator-facing config with focused tests for timeout/502 retry success and retry exhaustion before cross-provider fallback.

## Reproduction Assessment

Yes. A focused unit test can configure a pooled-proxy primary plus one fallback, make the primary throw timeout-classified failures, and assert current main calls that primary only once before trying the fallback.

## Solution Assessment

Yes, the requested provider/model or per-candidate retry budget is the narrowest maintainable direction; increasing global retries would not fit heterogeneous chains where pooled proxies and direct providers need different budgets.
