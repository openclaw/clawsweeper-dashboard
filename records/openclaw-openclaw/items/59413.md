## Summary

Keep open. Current main still lacks a retry-before-fallback budget scoped to a fallback candidate or provider/model route, while the related retry PR was closed unmerged and the issue discussion narrowed the remaining request to that route-specific config surface.

## What This Changes

The issue requests configurable retry-before-fallback behavior per fallback candidate or provider/model route for pooled proxy providers.

## Best Possible Solution

Add a backward-compatible retry policy at the provider/model or fallback-candidate boundary, apply it before marking that candidate failed, and document the operator-facing config with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence focused repro: configure a primary plus one fallback, make the primary throw timeout-classified 502 failures, and assert current main calls that primary once before trying the fallback.

## Solution Assessment

Yes. A route-scoped retry budget is the narrowest maintainable direction; increasing global retries or profile rotations would not fit fallback chains where pooled proxies and direct providers need different retry behavior.
