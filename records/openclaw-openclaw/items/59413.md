## Summary

Keep open: current main has auth-profile rotation knobs for rate-limit and overload cases, but it still does not provide the requested per-candidate or provider/model retry-before-fallback budget for opaque proxy pools. The linked https://github.com/openclaw/openclaw/pull/55118 was closed unmerged, and https://github.com/openclaw/openclaw/issues/79604 is related but narrower to OpenClaw auth-profile rotation rather than repeated attempts against one proxy route.

## What This Changes

Feature request for per-fallback-candidate or provider/model retry-before-fallback settings so pool/proxy providers can retry before moving to the next model.

## Best Possible Solution

Have maintainers choose a retry policy that composes with `auth.cooldowns` and timeout classification, then add schema/types/docs plus focused fallback tests for opaque proxy pools.

## Reproduction Assessment

Yes for the missing core capability: current source shows one outer fallback attempt per candidate and no retry fields on fallback config. No live TicketPro reproduction was run, so the external proxy success-rate claim remains based on the sanitized issue logs.

## Solution Assessment

Unclear as submitted: per-candidate retries are plausible for opaque proxy pools, but a provider/model or reason-scoped policy integrated with existing failover controls may be safer than adding ad hoc fields to fallback list entries.
