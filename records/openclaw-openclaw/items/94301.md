## Summary

Keep open: the PR is aimed at the right Telegram owner boundary, but the new immediate-drain helper still drops wakeups requested while a drain is already active, so worker-spooled updates can still wait for the interval timer in that timing window.

## What This Changes

The PR adds immediate Telegram isolated-ingress drain requests after worker-spooled `update` and `spooled` messages, adds a long-interval regression test, and still carries a gateway update hunk that is already present on current main.

## Best Possible Solution

Keep the Telegram-owned wakeup approach, but record wakeups requested during active drains, run a follow-up drain as soon as the active pass exits, add a long-interval regression for that timing window, and refresh away the already-main gateway hunk.

## Reproduction Assessment

Yes. Current main lacks an immediate wakeup after worker-spooled writes, and the remaining PR race is source-reproducible because `drainOnce()` returns while `drainActive` is true after the drain has already listed pending rows.

## Solution Assessment

No, not yet. Waking the existing Telegram isolated-ingress drain is the right owner boundary, but the implementation should remember wakeups requested during an active drain and run a follow-up pass instead of dropping them.
