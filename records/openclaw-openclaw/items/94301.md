Keep open: the owner-boundary fix is plausible, but the PR still drops immediate-drain wakeups requested while a drain is already active, leaving part of the Telegram message-delay failure mode unresolved.

## What This Changes

## Best Possible Solution

Keep the Telegram-owned drain wakeup, but preserve wakeups requested during active drains, add a focused regression for that timing window, and refresh away already-main branch drift before merge.

## Reproduction Assessment

Yes. Current main and v2026.6.8 lack an immediate wakeup after worker-spooled writes, and the remaining PR race is source-reproducible because `drainOnce()` returns while `drainActive` is true after the drain has already listed pending rows.

## Solution Assessment

No, not yet. Waking the existing Telegram isolated-ingress drain is the right owner boundary, but the implementation should remember wakeups requested during an active drain and run a follow-up pass instead of dropping them.
