## Summary

Keep this PR open: current main and v2026.6.8 still lack an immediate Telegram isolated-ingress drain wakeup after worker-spooled writes, and the linked Telegram ingress routing issue remains open. The branch is useful but not merge-ready because the direct `requestImmediateDrain()` call is dropped while `drainOnce()` is already active, leaving a real timing window where a worker-spooled update can still wait for the interval timer.

## What This Changes

The branch adds immediate drain requests after Telegram isolated-ingress worker `update` and `spooled` messages, adds a long-interval regression test, and carries a gateway update-finalizer hunk that is already present on current main.

## Best Possible Solution

Keep the Telegram-owned wakeup approach, but preserve wakeups requested during active drains, add a focused long-interval regression for that timing window, and refresh away the already-main gateway hunk before merge.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 write worker-spooled Telegram updates without an immediate drain wakeup, and the PR's remaining active-drain race follows from `drainActive` returning before recording a pending request.

## Solution Assessment

No, not yet. Waking the existing Telegram isolated-ingress drain is the right owner boundary, but the implementation should remember wakeups requested during an active drain and run a follow-up pass instead of dropping them.
