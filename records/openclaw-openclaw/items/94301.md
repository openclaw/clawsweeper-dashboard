## Summary

Keep open: the PR is aimed at the right Telegram-owned drain path, but the new immediate wakeup is still dropped when it is requested during an active drain, so a worker-spooled update can remain delayed until the interval timer.

## What This Changes

The PR adds immediate Telegram isolated-ingress drain requests after worker update/spooled messages, adds a long-interval regression test, and carries a gateway update hunk that is already present on current main.

## Best Possible Solution

Keep the Telegram-owned drain wakeup approach, but preserve wakeups requested during active drains, add a focused regression for that timing window, and refresh away already-main branch drift before merge.

## Reproduction Assessment

Yes. Current main and v2026.6.9 lack an immediate wakeup after worker-spooled writes, and the remaining PR race is source-reproducible because `drainOnce()` returns while `drainActive` is true after the drain has already listed pending rows.

## Solution Assessment

No, not yet. Waking the existing Telegram isolated-ingress drain is the right owner boundary, but the implementation should remember wakeups requested during an active drain and run a follow-up pass instead of dropping them.
