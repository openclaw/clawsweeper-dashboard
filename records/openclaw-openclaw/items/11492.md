## Summary

Current main and v2026.6.5 still omit both requested Slack agent-facing actions. The feature remains necessary, and the only open linked fork PR is conflicting and implements `channel-list` plus a separate search feature without the requested `channel-info`, so it is not a safe canonical replacement.

## What This Changes

Add Slack agent-facing `channel-info` and `channel-list` actions so agents can resolve channel IDs and discover readable workspace metadata.

## Best Possible Solution

Implement both actions within the Slack plugin’s existing discovery, dispatcher, lazy runtime, API-helper, test, and documentation surfaces, with cursor-based pagination and the existing `channelInfo` gate unless maintainers explicitly approve a separate gate.

## Reproduction Assessment

Yes at the source-contract level: current main definitively omits both actions from Slack discovery and dispatch. This is a missing capability rather than a failing existing behavior.

## Solution Assessment

Yes on ownership and general shape: the narrowest maintainable solution is a plugin-local Slack implementation using the existing action infrastructure. Maintainers still need to decide the permanent gate and whether any focused portion of the conflicting fork PR should be retained.
