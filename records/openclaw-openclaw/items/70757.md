## Summary

Keep this PR open. Current main still authenticates BlueBubbles webhooks against the API password and documents that password in the public webhook URL, while the PR’s dedicated `webhookSecret` direction is the right fix for the linked security bug. The current PR head still has a source-proven validation test/message mismatch and lacks the required user-facing changelog entry, so it should not merge as submitted.

## What This Changes

This PR adds a dedicated BlueBubbles `webhookSecret` across runtime webhook authentication, setup and config validation, SecretRef metadata, docs, generated config metadata, and regression tests.

## Best Possible Solution

Land this PR or an equivalent maintainer-approved patch after aligning the validation messages, adding the changelog entry, rebasing against current main if needed, and running the BlueBubbles runtime/setup/config tests plus the normal changed gate; then close #70755 as implemented.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: configure `channels.bluebubbles.password`, publish the documented BlueBubbles webhook URL containing that password, and a forged POST with an allowlisted sender authenticates because the webhook handler compares the caller token to the same password before sender allowlist, pairing, and command gating.

## Solution Assessment

No, not as submitted. The dedicated `webhookSecret` design is the narrow maintainable fix, but this head still has a validation-message/test mismatch and a missing changelog entry before it is merge-ready.
