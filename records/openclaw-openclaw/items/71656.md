## Summary

Keep open. Current main has narrowed the corpus-noise problem by filtering generated heartbeat/cron/silent transcript text before session export, but the issue's remaining core concerns are still real in source: session and daily ingestion scores are fixed constants, identical session snippets can still become separate candidates when timestamp/line metadata differs, and session scanning still uses deterministic sorted order under a global cap without a persisted fair-rotation cursor.

## What This Changes

The issue asks memory-core dreaming to pre-filter and dedupe noisy session corpus entries, score daily/session ingestion by content quality and recency, and rotate scan budget so session corpus entries continue receiving light-phase hits as corpus size grows.

## Best Possible Solution

Finish a narrow memory-core and memory-host repair that keeps the existing generated-noise filters, adds deterministic content-level dedupe for repeated session snippets, derives bounded ingestion scores from source/content/recency signals, and proves many-session-file progress under a constrained sweep cap without expanding plugin APIs.

## Reproduction Assessment

Yes for the remaining source-level defects: current main directly shows fixed ingestion scores, timestamp/line-scoped session dedupe, and capped sorted session scanning. The exact April 9 stall is unclear without the reporter's local `memory/.dreams/phase-signals.json` and corpus files.

## Solution Assessment

Yes. A focused repair in memory-core/session export is the narrowest maintainable path; it should avoid broad core/plugin API changes and keep the separate missing-artifact issue as a separate root-cause investigation unless tests prove overlap.
