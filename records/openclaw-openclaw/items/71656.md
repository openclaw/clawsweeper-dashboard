## Summary

Keep open. Current main still only partially addresses the issue: generated heartbeat/cron/silent-reply noise is filtered, but ingestion scores remain source-constant, duplicate session snippets are still allowed when timestamp or line metadata differs, and the session sweep still lacks a many-file fair-rotation guarantee under the global cap.

## What This Changes

The issue requests memory-core dreaming improvements for corpus noise filtering/deduplication, content/recency-aware scoring, and fair session-corpus scan rotation.

## Best Possible Solution

Finish a narrow memory-core and memory-host repair: keep the existing generated-noise filters, add deterministic content-level dedupe for repeated session snippets, compute bounded content/source/recency-aware ingestion scores before recall recording, and add regression coverage for many-session-file progress under a tight sweep cap.

## Reproduction Assessment

Yes for the remaining source-level defects: current main directly shows fixed ingestion scores, position-scoped session dedupe, and capped sorted session scanning without a many-file fairness test. The exact April 9 production stall is unclear without the reporter's local phase-signals and corpus files.

## Solution Assessment

Yes, a focused memory-core/session export repair is the best path. It should avoid core plugin API changes, avoid changing Dream Diary artifact semantics, and treat missing-artifact issue #72546 as separate unless tests prove a shared root cause.
