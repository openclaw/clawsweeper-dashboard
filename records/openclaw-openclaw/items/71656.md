## Summary

Keep open. Current main still only partially addresses the reported memory dreaming quality and scaling problems: session export filters some generated heartbeat, cron, and silent-reply noise, but ingestion scoring is still source-constant, session dedupe is still tied to positional metadata, and the session sweep can still spend a tight global cap on earlier sorted files instead of guaranteeing fair progress across a large corpus.

## What This Changes

The issue requests memory-core corpus pre-filtering and deduplication, content/recency-aware scoring, and fair session-corpus scan rotation so dreaming promotion keeps progressing as the corpus grows.

## Best Possible Solution

Finish the repair in memory-core and session export: retain the existing generated-noise filters, add deterministic content-level dedupe for repeated session snippets, compute bounded content/source/recency-aware ingestion scores before recall recording, and add a many-session-file regression proving fair progress under a tight sweep cap.

## Reproduction Assessment

Yes for the remaining source-level defects: current main directly shows fixed ingestion scores, position-scoped session dedupe, and only partial scan-budget coverage. The exact April 9 stall is unclear without the reporter's local phase-signals and session-corpus files.

## Solution Assessment

Yes, a narrow memory-core/session export repair is the best path. It should avoid core plugin API changes, avoid changing Dream Diary artifact semantics, and treat related artifact-write issue #72546 as separate unless tests prove a shared root cause.
