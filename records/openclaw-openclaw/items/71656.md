## Summary

Keep open. Current main partially addresses corpus noise by filtering generated heartbeat/cron/silent session text, but the remaining source-level defects are still present: ingestion scores are fixed by source, repeated content can be re-ingested when timestamp/line metadata changes, and many-file session scanning still lacks a persisted fair-rotation guarantee under the shared cap.

## What This Changes

Improve memory-core dreaming by filtering and deduplicating noisy session corpus entries, using content- and recency-aware scoring, and rotating scan budget across corpus sources.

## Best Possible Solution

Finish a narrow memory-core and memory-host repair: preserve the existing generated-noise filters, add deterministic content-level dedupe for repeated session snippets, compute bounded content/source/recency-aware ingestion scores before recall recording, and add regression coverage for many-session-file progress under a tight sweep cap.

## Reproduction Assessment

Yes for the remaining source-level defects: current main directly shows fixed ingestion scores, timestamp/line-scoped session dedupe, and capped sorted session scanning. The exact April 9 stall is unclear without the reporter's local phase-signals and corpus files.

## Solution Assessment

Yes. A focused memory-core/session export repair is the narrowest maintainable path; it should avoid plugin API changes and avoid folding in the separate missing-artifact report unless tests prove a shared root cause.
