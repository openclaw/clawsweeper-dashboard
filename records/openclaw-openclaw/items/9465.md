## Summary

Keep open: current main still lacks a persisted cron hook lifecycle, and the remaining implementation path is active, user-visible, and security-sensitive rather than cleanup-ready.

## What This Changes

The issue requests configurable cron job hooks before, after, and on error so jobs can gate, transform, notify, retry, or escalate around scheduled execution.

## Best Possible Solution

Settle the core-versus-plugin contract, then land or explicitly supersede a hardened additive lifecycle hook design with protocol/config/UI/CLI/docs/tests and a documented trust boundary.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. Source inspection on current main verifies the requested persisted cron hook contract is absent from storage/runtime types, Gateway schemas, config validation, and scheduled/startup/manual execution paths.

## Solution Assessment

Unclear for the exact proposed schema. The maintainable path is maintainer product/security review of the lifecycle contract, likely by landing or superseding https://github.com/openclaw/openclaw/pull/45465 while considering which preflight-only pieces belong in plugins/ClawHub.

## Risks / Open Questions

- Shell and webhook lifecycle hooks would add scheduled local code execution and outbound network calls, so permissions, path rules, timeouts, logging, failure handling, skip semantics, and secret exposure need explicit approval.
- The durable public contract is unsettled between this issue's pre/post/onError shape, the open implementation candidate's beforeRun/afterComplete/onFailure/afterRun shape, and existing plugin hooks such as before_agent_reply and cron_changed.
- Starting an automated replacement branch would likely duplicate or conflict with the open implementation candidate at https://github.com/openclaw/openclaw/pull/45465 and the related plugin-hook discussion at https://github.com/openclaw/openclaw/pull/72179.
