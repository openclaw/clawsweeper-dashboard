## Label Justifications

- `P2`: The feature offers meaningful cron cost, reliability, and ergonomics improvements but is not an urgent regression.
- `impact:session-state`: Hook decisions would determine whether scheduled agent sessions run, skip, retry, or escalate and how those outcomes are persisted.
- `impact:security`: The proposed shell, script, and webhook hooks add local execution and outbound-network authority around unattended jobs.

## Summary

Keep this issue open as the canonical product-design umbrella: current main and v2026.6.5 still lack configurable cron lifecycle hooks, while the two overlapping implementation PRs remain conflicting, proof-incomplete, and split between a narrow pre-run gate and a much broader script lifecycle API.

## What This Changes

Add configurable pre/post/error cron lifecycle hooks for conditional execution, retries, escalation, notifications, and external logic.

## Best Possible Solution

Keep one canonical issue and have maintainers first choose a narrow typed contract—likely a deterministic pre-model gate plus observable completion/failure events or a decision-capable plugin seam—then require consistent persistence, Gateway, scheduled/startup/manual execution, CLI/UI, docs, migration, and security behavior.

## Reproduction Assessment

Yes at source level: current main has no persisted hook field, the strict Gateway schemas cannot accept one, and all run modes enter the shared executor without the requested configurable gate. This is a missing capability rather than a broken documented behavior.

## Solution Assessment

No, the original four-kind hook design is not yet established as the best solution; the strongest demonstrated need is a deterministic pre-model gate, while post/error orchestration should reuse a typed plugin/event boundary rather than introduce several competing execution mechanisms without a security decision.
