## Label Justifications

- `P2`: This is a meaningful cron automation feature with real production-cost and noise reports, but it is not an urgent regression or core outage.
- `impact:security`: The requested shell and webhook hooks would let persisted cron configuration run host commands or network calls around scheduled execution.
- `impact:session-state`: The requested lifecycle can skip, trigger, or follow cron agent turns and therefore affects cron run/session state and escalation behavior.

## Summary

Keep open: current main and the latest release still lack the persisted cron pre/post/onError hook lifecycle requested here. Shipped command cron jobs solve part of the zero-LLM direct-exec pain, but they do not provide a conditional pre-run gate that can decide whether to run an agent turn, nor post/error hook phases for escalation or cleanup.

## What This Changes

The issue requests a cron hook lifecycle for pre, post, and error custom logic, including shell/webhook gates around scheduled agent execution.

## Best Possible Solution

Design and land a maintainer-approved cron hook lifecycle, or an equivalent scheduler-owned plugin/API seam, that covers persisted job schemas, Gateway validation, scheduled/startup/manual execution, CLI/UI/docs, tests, and a clear security model for shell/webhook execution.

## Reproduction Assessment

Yes, at source level: current main has no cron hooks field in the persisted type or Gateway schemas, and scheduled/manual execution paths enter the core executor without pre/post/error phases. I did not run a live cron add command because this review is read-only.

## Solution Assessment

No, closing as implemented would not be the best resolution. The shipped command payload is a good partial solution for pure deterministic jobs, but the maintainable fix for this issue is a security-reviewed cron lifecycle or plugin/API hook seam that can gate and follow agentTurn runs.
