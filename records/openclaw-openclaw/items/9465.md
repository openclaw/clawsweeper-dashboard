## Label Justifications

- `P2`: The issue requests a valuable cron improvement with active implementations, but it is not an urgent regression or core outage.
- `impact:security`: The proposed shell and webhook hooks cross scheduled code-execution, filesystem, secrets, and outbound-network boundaries.
- `impact:session-state`: Hook aborts, retries, escalation, and cleanup can change cron execution, delivery, and persisted run-state transitions.

## Summary

Keep open. Current main partially addresses the reported cost and reliability problem through direct command cron jobs, but it still does not provide the requested persisted pre-, post-, and error-hook lifecycle, and two open implementation PRs remain under active product and security review.

## What This Changes

Add configurable pre-, post-, and error lifecycle hooks to cron jobs for gating, processing, notifications, and escalation workflows.

## Best Possible Solution

Keep command payloads as the canonical path for deterministic scheduled scripts, then land or explicitly supersede the open hook PRs with one additive, security-reviewed lifecycle contract covering only the remaining gating and notification needs across persisted types, protocol, execution paths, CLI/UI, docs, and tests.

## Reproduction Assessment

Yes, at source level. Current main clearly supports direct command jobs but its persisted cron types, Gateway schemas, and execution contract still lack general pre-, post-, and error-hook phases.

## Solution Assessment

No, the original all-purpose hook union is no longer the narrowest maintainable solution because command payloads now cover deterministic execution; the remaining lifecycle behavior should use one security-reviewed extension boundary without creating a second command-execution path.
