## Label Justifications

- `impact:security`: The requested lifecycle includes user-configured script and webhook execution around scheduled jobs.

## Summary

Keep open: current main partially addresses the reported no-LLM cost problem with command-backed cron jobs, but it still lacks decision-capable pre/post/error lifecycle hooks, and the two related implementation PRs are conflicting, proof-incomplete, and propose divergent contracts that require maintainer product and security judgment.

## What This Changes

Requests configurable pre-, post-, and error hooks around cron job execution for gating, escalation, retries, notifications, and external logic.

## Best Possible Solution

Retain command jobs as the canonical no-LLM execution path, then define one typed, additive lifecycle-decision seam—preferably usable by plugins—that consistently covers scheduled, startup catch-up, and manual runs with explicit abort, timeout, failure, logging, security, and state-transition semantics.

## Reproduction Assessment

Yes, at source level: current main clearly exposes command jobs and observation-only cron events while the persisted job shape and executor lack decision-capable pre/post/error lifecycle phases. This is a missing capability rather than a failing existing contract.

## Solution Assessment

No, the original broad four-kind hook union is no longer the narrowest maintainable solution because command jobs now own standalone deterministic execution. Maintainers should narrow the remaining work to one lifecycle-decision contract and select or replace one implementation PR accordingly.

## Risks / Open Questions

- A broad hook system would overlap with the command payload and create competing ways to perform deterministic cron work.
- Shell or webhook lifecycle hooks introduce operator-controlled code execution and network access requiring explicit permission, trust, timeout, cancellation, output-redaction, and failure semantics.
- Abort, retry, and escalation decisions can affect cron state transitions, backoff, delivery, run history, and session continuity across scheduled, startup catch-up, and manual executions.
- The two open PRs propose different configuration and lifecycle models, so landing either without first choosing the permanent contract could create a difficult compatibility surface.
