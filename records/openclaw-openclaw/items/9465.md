## Summary

Current main and v2026.6.5 still lack a persisted cron gate or lifecycle-hook contract. Two conflicting open PRs propose materially different, security-sensitive APIs, so this remains active maintainer product design rather than cleanup-ready work.

## What This Changes

Add configurable pre-, post-, and error hooks to cron jobs for deterministic gating, escalation, retries, notifications, and external processing.

## Best Possible Solution

Adopt one additive, security-reviewed cron execution contract that first solves deterministic pre-model gating, defines consistent terminal outcomes across every execution path, and expands to post/error phases only when ownership and data-exposure rules are explicit.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.5 expose no persisted hook field or deterministic pre-model gate, so the capability gap is directly verifiable. This is a new feature request rather than a failure of an existing documented contract.

## Solution Assessment

Unclear. A deterministic pre-model gate addresses the strongest demonstrated use case, but the broad pre/post/error lifecycle design is not yet proven to be the safest permanent API, and the two active PRs make incompatible scope choices.

## Risks / Open Questions

- Shell or executable hooks run unattended operator-configured code and need explicit trust, path-resolution, timeout, cancellation, output-redaction, and failure semantics.
- Webhook hooks can expose job or session information and need SSRF, credential, retry, and logging boundaries.
- The two active PRs define competing configuration shapes and scopes; landing either without a product decision could establish an incompatible permanent cron API.
- Skip, failure, retry, and post-run behavior must remain consistent across scheduled execution, startup catch-up, and manual runs to avoid run or session-state drift.
