- `impact:security`: The requested shell and webhook hooks would execute code or network calls around scheduled jobs and therefore affect security boundaries.

## Summary

Keep open: current main and the latest release still lack a persisted cron hook lifecycle, and the existing command-payload path only covers deterministic no-LLM jobs, not pre/post/error hooks or hook-based gating around agent runs.

## What This Changes

The issue requests a cron job hook lifecycle for running pre, post, and error hooks such as agent turns, system events, shell commands, and webhooks around scheduled executions.

## Best Possible Solution

Keep this as the canonical maintainer/product design item until cron lifecycle hooks are either explicitly scoped into a permissioned cron/plugin API or narrowed into follow-ups that reuse command payloads and existing typed plugin hooks.

## Reproduction Assessment

Yes, as a source-level feature gap: the current cron job type and Gateway schema do not accept hooks/preHook, and the execution paths do not run a pre/post/error hook phase. A live repro is not necessary to show that the requested persisted lifecycle is absent.

## Solution Assessment

No, the broad proposed hook union is not yet the best implementation plan by itself. The maintainable solution needs maintainer-scoped semantics that account for existing command payloads, observe-only plugin cron_changed hooks, and the security model for user scripts/webhooks.
