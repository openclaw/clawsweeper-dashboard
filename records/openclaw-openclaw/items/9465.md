## Summary

Keep this issue open: current main and v2026.6.5 still lack a persisted, execution-controlling cron hook contract, while two open implementation PRs propose conflicting scopes and security semantics that require maintainer product selection rather than another automated fix.

## What This Changes

Requests configurable cron lifecycle hooks for pre-run gating, post-processing, error handling, escalation, and external actions.

## Best Possible Solution

Choose one additive, security-reviewed cron extension contract—preferably a narrow plugin/runtime decision seam if it can cover the use cases—then implement it consistently across persisted types, Gateway schemas, every execution entry point, CLI/UI/docs, and upgrade-safe tests while closing or superseding the competing PRs.

## Reproduction Assessment

Yes for the capability gap: current main has no persisted execution hook field and its existing cron_changed plugin hook is observe-only. This is not a failing existing contract, so there is no bug reproduction beyond source verification.

## Solution Assessment

No, the original four-kind hook union is not yet established as the best solution because it combines gating, orchestration, arbitrary code execution, webhooks, and retries. Maintainers should first select one canonical owner boundary and reconcile the narrower preHook and direct-exec alternatives.
