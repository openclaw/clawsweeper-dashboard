## Summary

Keep open. Current main still shows the reported source-level drift: direct inbound metadata and inter-session context enter the provider-facing system prompt path, while owner-only tool policy can still change the exposed tool set by sender or runtime grant; the related prompt-cache issues are adjacent, not a canonical duplicate for this same-session OpenAI direct-vs-inter-session case.

## What This Changes

The issue requests cache-stable OpenAI provider payloads across direct user turns and inter-session or cron-triggered turns in the same live session.

## Best Possible Solution

Define a provider-facing cache-stable session contract: move direct and A2A source metadata into current-turn input/provenance or an OpenAI-respected non-prefix surface, keep the exposed tool schema stable where safe, and enforce owner-only denial at execution time with regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still routes direct inbound metadata and `sessions_send` A2A context through `extraSystemPrompt` into OpenAI Codex `instructions`, and owner-only policy can still change the exposed tool list; I did not establish a live `cacheRead=0` trace.

## Solution Assessment

Partly. Moving source metadata out of provider-facing instructions is the narrow maintainable direction, but the tool-schema portion needs a maintainer policy decision so cache stability does not weaken owner-only authorization.
