## Summary

Keep open. Current main still has source-dependent provider-facing prompt and tool drift: direct inbound metadata and inter-session context still flow through `extraSystemPrompt` into `instructions`, and owner-only tool policy can still change the exposed tool set by sender or runtime grant.

## What This Changes

The issue requests stable OpenAI prompt-cache prefixes for the same session by keeping dispatch-source metadata and tool schemas from drifting between direct and inter-session or cron turns.

## Best Possible Solution

Move direct and inter-session source metadata into current-turn context or another non-instruction payload, then define a stable owner-only tool-schema policy backed by runtime authorization tests and prompt-cache observability coverage.

## Reproduction Assessment

Yes for a source-level structural reproduction: compare a direct auto-reply turn with a `sessions_send` or cron-relayed turn to the same session and current main shows different `extraSystemPrompt` text plus possible owner-only tool filtering. I did not rerun a live OpenAI cache trace.

## Solution Assessment

No complete current-main solution exists. The metadata placement fix is narrow, but stabilizing owner-only tool schemas requires an explicit authorization policy decision before implementation.
