## Summary

Keep open. Current main still has the reported source-dependent prompt drift: direct inbound metadata and `sessions_send` agent-to-agent context enter `extraSystemPrompt` and become OpenAI Codex `instructions`, while owner-only tool policy can still change the exposed tool set by sender or runtime grant. The related prompt-cache issues cover adjacent cross-run or chat-continuation cases, not this same-session direct-vs-inter-session drift.

## What This Changes

The issue asks OpenClaw to keep a session's OpenAI prompt-cache prefix stable by moving source-dependent metadata out of instructions, avoiding source-dependent tool-schema drift, and adding cache-drift diagnostics.

## Best Possible Solution

Move source-specific direct and inter-session context into current-turn message context or structured provenance, then define a cache-stable owner-only tool exposure policy that preserves runtime authorization and add OpenAI Responses payload-stability regression coverage.

## Reproduction Assessment

Yes, at source level. Current main shows direct inbound metadata and `sessions_send` A2A metadata taking different `extraSystemPrompt` paths into OpenAI Codex `instructions`, and owner-only policy can still change the tool list; I did not run a live OpenAI cache trace.

## Solution Assessment

Partly. Moving source metadata out of provider-facing instructions is the narrow maintainable fix, but the tool-schema portion needs an explicit maintainer policy decision so cache stability does not weaken owner-only authorization.
