## Label Justifications

- `impact:session-state`: The reported failure is same-session prompt/context drift caused by dispatch-source differences.

## Summary

Keep open. Current main has partial cache hygiene work, but the central source-level drift remains: direct inbound metadata and inter-session context still flow through `extraSystemPrompt` into the provider-facing system prompt that OpenAI Codex Responses sends as `instructions` under the same `prompt_cache_key`.

## What This Changes

The issue requests a cache-stable provider-facing prompt prefix and tool surface for the same live session across direct and inter-session or cron-dispatched turns.

## Best Possible Solution

Define a cache-stable session contract: move source-specific direct and A2A metadata into current-turn input or structured provenance, keep tool schemas stable only where authorization remains enforceable, and cover both cache stability and restricted-tool denial in regression tests.

## Reproduction Assessment

Yes, at source level. Current main still routes direct inbound metadata and `sessions_send` A2A context through `extraSystemPrompt` into OpenAI Codex `instructions`; I did not establish a live provider `cacheRead=0` trace.

## Solution Assessment

Partly. Moving source metadata out of provider-facing instructions is the narrow maintainable direction, but the tool-schema portion needs maintainer policy and security review so cache stability does not weaken authorization.
