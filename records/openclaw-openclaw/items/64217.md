## Summary

Keep open. Current main still has a source-reproducible prompt-cache stability bug: direct inbound turns and inter-session turns can send different provider-facing OpenAI Codex `instructions`, and owner-only tool filtering can still change the exposed tool set for the same target session.

## What This Changes

The issue requests stable provider-facing OpenAI prompt instructions and tool schemas for the same live session and `prompt_cache_key` across direct and inter-session or cron-triggered turns.

## Best Possible Solution

Move direct inbound and inter-session source metadata out of Codex Responses `instructions`, then choose a stable owner-only tool-schema policy that preserves non-owner runtime denial and adds prompt-cache drift observability.

## Reproduction Assessment

Yes for a source-level structural reproduction: compare a direct auto-reply turn with a `sessions_send` or cron-relayed turn to the same target session and current main shows different `extraSystemPrompt` text plus sender/grant-dependent owner-only tool filtering. I did not rerun a live OpenAI cache trace.

## Solution Assessment

No complete current-main solution exists. The metadata placement fix is narrow, but the tool-schema part needs an explicit security/product decision so schema stability does not weaken owner-only authorization.
