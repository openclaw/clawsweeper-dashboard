## Summary

Keep open: current main still has the core same-session drift path because inter-session sends add source-specific context to the provider-facing system prompt and owner-only tool exposure can still vary by sender or runtime grant.

## What This Changes

The issue reports that source-dependent `instructions` and tool-set changes can invalidate OpenAI/Codex prompt-cache reuse for the same live session and `prompt_cache_key`.

## Best Possible Solution

Move A2A requester/target metadata out of `extraSystemPrompt` into triggering-turn provenance, then make an explicit owner-only tool policy: stable schema with runtime denial, or a documented and tested cache-impact exception.

## Reproduction Assessment

Yes. A high-confidence structural reproduction is to compare a direct auto-reply turn with a `sessions_send` turn for the same target session and inspect the assembled Codex Responses payload; source shows both `instructions` and tool names can still differ before provider submission.

## Solution Assessment

No current-main solution is complete. The narrow maintainable path is to stabilize A2A prompt placement first, then resolve the owner-only tool-schema tradeoff with maintainer security/product review.
