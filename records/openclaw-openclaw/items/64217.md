## Summary

Keep open. Current main still has the reported same-session drift: source-specific metadata flows into the provider-facing system prompt/`instructions`, and owner-only tool policy can still change the exposed tool set by sender or cron runtime grant; #62809 is adjacent but does not supersede this narrower same-session issue.

## What This Changes

The issue requests stable provider-facing instructions and tool schemas for the same live OpenAI/Codex session across direct and inter-session/cron-triggered turns.

## Best Possible Solution

Move direct and inter-session source metadata out of provider-facing `instructions` into the triggering turn or structured provenance, then make an explicit owner-only tool-schema policy with regression coverage for cache stability and authorization.

## Reproduction Assessment

Yes for the structural reproduction: direct auto-reply, `sessions_send`, and isolated cron paths can be compared on current main and source inspection shows different `instructions`/tool inputs for the same target session. I did not rerun a live OpenAI cache trace, but the provider-facing payload path is clear.

## Solution Assessment

No current-main solution is complete. The A2A/source-metadata placement fix is narrow, but the owner-only tool schema part needs a maintainer policy decision before changing exposed tool schemas.
