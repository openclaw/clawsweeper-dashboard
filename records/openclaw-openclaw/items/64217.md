## Label Justifications

- `P2`: The issue describes a real cache/cost regression in a specific OpenAI prompt-caching workflow rather than a core outage.
- `impact:session-state`: The reported failure is same-session prompt/context drift caused by dispatch-source differences.
- `impact:security`: The tool-schema portion crosses owner-only authorization for tools such as `cron`, `gateway`, and `nodes`.

## Summary

Keep open. Current main has partial cache hygiene work, including volatile-ID stripping and prompt-cache diagnostics, but the central same-session drift remains: direct inbound metadata and inter-session context still enter the provider-facing system prompt, and owner-only tool policy can still change the exposed tool set by sender or runtime grant.

## What This Changes

The issue asks OpenClaw to keep OpenAI prompt-cache-sensitive instructions and tool schemas stable within the same target session across direct and inter-session or cron-triggered turns.

## Best Possible Solution

Define a cache-stable provider-facing session contract: move source-specific direct and A2A metadata into current-turn input or structured provenance, keep tool schemas stable only where authorization remains enforceable, and cover both cache stability and owner-only denial in regression tests.

## Reproduction Assessment

Yes, at source level. Current main still routes direct inbound metadata and `sessions_send` A2A context through `extraSystemPrompt` into OpenAI Codex `instructions`, while owner-only policy can still change the tool list; I did not establish a live `cacheRead=0` trace.

## Solution Assessment

Partly. Moving source metadata out of provider-facing instructions is the narrow maintainable direction, but the tool-schema part needs a maintainer policy decision so cache stability does not weaken owner-only authorization.
