## Summary

Keep open. Current main still has the reported same-session drift: direct auto-reply turns and inter-session sends can produce different provider-facing system prompt suffixes, Codex/OpenAI Responses still forwards the assembled system prompt as `instructions`, and owner-only tool availability can still change the exposed tool set by sender or runtime grant.

## What This Changes

The issue requests stable provider-facing instructions and tool schemas for the same live session and `prompt_cache_key` across direct user turns and inter-session or cron-triggered turns.

## Best Possible Solution

Move inter-session requester/target context into the triggering user turn or structured provenance instead of the system prompt, then make an explicit maintainer decision for owner-only tool schemas: either expose a stable schema with runtime authorization denial, or document and test an intentional cache-impact exception.

## Reproduction Assessment

Yes for the structural reproduction: compare a direct auto-reply turn with a `sessions_send` turn for the same target session and inspect the assembled OpenAI Responses payload. Source inspection shows `instructions` and owner-only tool exposure can still differ without needing a live provider cache trace.

## Solution Assessment

No current-main solution is complete. Prompt-cache observability and inbound metadata stabilization help diagnosis, but the maintainable fix still needs A2A metadata moved out of `extraSystemPrompt` plus an explicit owner-only tool-schema policy.
