## Summary

Keep open. Current main has partial cache-stability work, but the same-session drift described here still exists: inter-session sends still add requester/target-specific A2A text through the system prompt path, OpenAI Responses still sends that assembled system prompt as `instructions`, and owner-only tools are still removed from the exposed schema unless the sender or runtime grant authorizes them.

## What This Changes

The issue requests stable provider-facing instructions and tool schemas for the same session and `prompt_cache_key` across direct user turns and inter-session or cron-triggered turns.

## Best Possible Solution

Move inter-session requester/target context into the triggering user turn or provenance data instead of the system prompt, then make an explicit maintainer decision for owner-only tool schemas: either expose a stable schema with runtime authorization denial or document and test an intentional cache-impact exception.

## Reproduction Assessment

Yes for the structural reproduction: compare a direct auto-reply turn with a `sessions_send` turn for the same target session and inspect the assembled OpenAI Responses payload. The source path shows `instructions` and owner-only tool exposure can still differ without needing a live provider cache trace.

## Solution Assessment

No current-main solution is complete. Prompt-cache observability and inbound metadata stabilization help diagnosis, but the narrow maintainable fix still needs A2A metadata moved out of `extraSystemPrompt` plus a deliberate owner-only tool-schema policy.
