## Summary

Keep open. Current main still has the reported structural drift: direct chat metadata and inter-session metadata are still routed through `extraSystemPrompt` into provider-facing system instructions, and owner-only tool exposure can still vary by sender or runtime grant; the adjacent fixes at https://github.com/openclaw/openclaw/issues/77431 and https://github.com/openclaw/openclaw/issues/62809 do not supersede this same-session OpenAI prompt-cache issue.

## What This Changes

The issue requests stable provider-facing instructions and tool schemas for a live session so source metadata and owner-only tool policy do not invalidate OpenAI prompt caching.

## Best Possible Solution

Move source-specific direct and inter-session context into current-turn message context or structured provenance, then define a cache-stable owner-only tool exposure policy that preserves runtime authorization and adds regression coverage for OpenAI Responses payload stability.

## Reproduction Assessment

Yes for the structural reproduction: current source shows direct inbound metadata and `sessions_send` A2A metadata taking different `extraSystemPrompt` paths into OpenAI Codex `instructions`, and owner-only policy can change the tool list. I did not run a live OpenAI cache trace in this read-only review.

## Solution Assessment

Partly. Moving source metadata out of provider-facing instructions is the narrow maintainable fix, but the tool-schema portion needs an explicit maintainer policy decision so cache stability does not weaken owner-only authorization.
