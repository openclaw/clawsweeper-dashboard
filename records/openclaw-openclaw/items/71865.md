## Summary

Keep open. Current main and the latest v2026.5.7 release still compare canonical two-space JSON output against the previous raw config byte length, while auth login still reaches that shared writer without a semantic or normalized-size check; the closed related report points back to this issue as the canonical unresolved tracker.

## What This Changes

Fix auth login config persistence when a PowerShell-created BOM/verbose `openclaw.json` triggers the shared size-drop write guard.

## Best Possible Solution

Teach the shared config writer to treat formatting-only shrink semantically or against a canonicalized previous payload, then cover the PowerShell/BOM auth-login path and the related models-set path while preserving fail-closed clobber protection.

## Reproduction Assessment

Yes, at source level. A valid BOM/verbose `openclaw.json` can be read as the base config, then auth login flows through `persistProviderAuthResult` to `updateConfig` and `writeConfigFile`, where canonical JSON can be less than half the previous raw bytes and trigger `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix; an auth-only `allowConfigSizeDrop` bypass would leave the same false positive in other config mutations.
