## Summary

Keep open. Current `main` still has the shared config-writer false positive: models auth login reaches `updateConfig`, which writes through the shared writer without `allowConfigSizeDrop`, and the writer still compares canonical two-space output bytes against the previous raw file bytes.

## What This Changes

The issue reports that `openclaw models auth login` can appear successful but fail to persist OAuth config data when an existing PowerShell/BOM-formatted `openclaw.json` shrinks by more than the size-drop guard threshold during canonical rewrite.

## Best Possible Solution

Fix the shared config writer to treat formatting-only shrink as safe through a semantic or canonicalized comparison, while preserving fail-closed rejection for real destructive writes.

## Reproduction Assessment

Yes, source-level. A semantically unchanged but raw-large `openclaw.json` can flow from models auth login through `updateConfig` into the shared writer, where current `main` still compares canonical next bytes against raw previous bytes and can reject the write as `size-drop:*`.

## Solution Assessment

Yes. A writer-level semantic or canonicalized size-drop check is the narrow maintainable fix; an auth-only bypass would leave the same false positive in `models set` and other config mutations.
