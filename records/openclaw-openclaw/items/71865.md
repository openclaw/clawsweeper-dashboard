## Summary

Keep open. Current main and the latest release still block this formatting-only shrink through the shared config writer, and the related models-set report was closed in favor of this issue as the canonical tracker.

## What This Changes

The issue reports that auth login cannot persist provider auth when an existing PowerShell-formatted openclaw.json shrinks enough to trigger the config writer size-drop guard.

## Best Possible Solution

Fix the shared config writer to compare formatting-only shrink semantically or against a canonicalized previous payload, with regressions for the PowerShell/BOM auth-login case and the related models-set path.

## Reproduction Assessment

Yes, at source level. A valid existing config whose raw PowerShell/BOM formatting is more than twice the canonical JSON size flows from auth login through updateConfig into writeConfigFile, where current main rejects the write as size-drop.

## Solution Assessment

Yes. A writer-level normalized or semantic comparison is the narrowest maintainable fix; an auth-only allowConfigSizeDrop bypass would leave the same false positive in other config mutations and weaken safety unevenly.
