## Summary

Keep open. Current main still compares the canonical two-space JSON payload against the previous raw config byte size, and the models auth persistence path does not opt out or perform a semantic formatting-only comparison, so the PowerShell/BOM false-positive remains possible. Related context reinforces this as the canonical unresolved report: #70941 was closed in favor of this stronger reproduction, while #74163 lists it as remaining Windows P1 work.

## What This Changes

The issue requests that provider auth config writes persist across PowerShell/BOM-formatted `openclaw.json` files instead of being rejected by the size-drop guard.

## Best Possible Solution

Fix the shared config writer to treat formatting-only shrinkage semantically or canonically, then add a regression that drives a BOM/verbose config through the auth persistence path.

## Reproduction Assessment

Yes. Start with a valid BOM-prefixed, deeply indented `openclaw.json`, apply a small provider auth config mutation through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile`, and current main can still reach `CONFIG_WRITE_REJECTED` before replacing the active file.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix because other config mutations can hit the same raw-byte false positive; an auth-only `allowConfigSizeDrop` bypass would leave the shared heuristic wrong elsewhere.
