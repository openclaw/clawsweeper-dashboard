## Summary

Keep open. Current main still compares the canonical two-space config rewrite against the raw previous file bytes, so a valid BOM/verbose-formatted config can still trip the blocking size-drop guard during models auth persistence; the related Windows tracker keeps this unresolved rather than superseding it.

## What This Changes

Fix `models auth login` persistence so a PowerShell-created, BOM/verbose-formatted `openclaw.json` is not rejected as a destructive size drop when only formatting changes shrink the file.

## Best Possible Solution

Update the shared config writer to distinguish formatting-only shrink from semantic data loss, likely by comparing against a canonicalized previous config or a structured diff before applying the size-drop block, then add targeted config-writer and models-auth regression tests for BOM/verbose-formatted input.

## Reproduction Assessment

Yes. A fixture can create a valid BOM-prefixed, heavily indented `openclaw.json` whose raw byte size is more than double the canonical two-space rewrite, then exercise the `models auth login` persistence path through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile` and observe `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. Fixing the shared writer is the narrowest maintainable solution because other config mutations can hit the same false positive; an auth-only override would leave the bug elsewhere and weaken safety on one path.
