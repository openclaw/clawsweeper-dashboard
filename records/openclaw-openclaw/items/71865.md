## Summary

Keep open: current main and v2026.6.8 still compare canonical next JSON against the raw prior file size, so the PowerShell/BOM formatting false positive remains. The linked fix is still an open draft PR, so this issue should remain the tracker until that writer-level repair lands or is replaced.

## What This Changes

The issue reports that auth/model config writes can fail to persist when a PowerShell/BOM-formatted `openclaw.json` trips the shared size-drop guard.

## Best Possible Solution

Land or replace the writer-level canonical-size comparison fix with regression coverage for PowerShell/BOM formatting while preserving destructive-shrink rejection for real clobbers.

## Reproduction Assessment

Yes for a source-level reproduction: current main still compares raw prior file bytes to canonical next JSON bytes, and auth config patches reach that writer without `allowConfigSizeDrop`. I did not run a live Windows/PowerShell repro in this read-only review.

## Solution Assessment

Yes: the best fix is in the shared config writer, not an auth-only bypass, because the false positive comes from serialization format changing the size baseline. The open candidate at https://github.com/openclaw/openclaw/pull/88382 has that shape but still needs to land or be replaced.
