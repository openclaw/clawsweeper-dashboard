## Summary

Keep open. Current main still has the raw-byte size-drop guard on the shared config writer path used by models auth, so the reported PowerShell/BOM verbose-format false positive remains plausible and actionable; the related #70941 was closed in favor of this stronger reproduction, while #74163 only tracks the Windows backlog and lists this item as unresolved.

## What This Changes

The issue requests that `models auth login` persist provider auth config when `openclaw.json` was authored by PowerShell with a UTF-8 BOM and verbose indentation instead of being rejected by the size-drop guard.

## Best Possible Solution

Fix the shared TypeScript config writer to treat formatting-only shrink from a semantically equivalent prior config as non-destructive, while keeping real data-loss size drops blocked, then add focused regression coverage for BOM/verbose prior files and the models auth persistence path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: make a valid `openclaw.json` whose raw BOM/verbose formatting is more than twice the canonical two-space rewrite, then follow `models auth login` through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, where current main rejects `size-drop`.

## Solution Assessment

Yes. A shared config-writer fix is the narrowest maintainable solution because the false positive can affect any OpenClaw-owned config mutation; an auth-only `allowDestructiveWrite` bypass would leave other commands broken and weaken safety inconsistently.
