## Summary

Keep open. Current main still compares the canonical two-space JSON payload against the previous raw config byte size, so a formatting-only shrink from a BOM/verbose PowerShell file can still trip the shared size-drop guard before auth-login config changes persist.

## What This Changes

Fix auth login persistence when an existing PowerShell-formatted or BOM-prefixed `openclaw.json` shrinks substantially during OpenClaw's canonical JSON rewrite.

## Best Possible Solution

Make the shared config writer evaluate formatting-only shrink semantically, then cover the PowerShell/BOM input through the auth persistence flow while preserving the existing fail-closed safety model.

## Reproduction Assessment

Yes. A focused regression can start from a valid BOM-prefixed, deeply indented `openclaw.json`, apply a small auth-profile mutation through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, and observe the current `CONFIG_WRITE_REJECTED` path before the active config changes.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix because other config mutations can hit the same raw-byte false positive; an auth-only `allowConfigSizeDrop` bypass would leave the underlying heuristic wrong elsewhere.
