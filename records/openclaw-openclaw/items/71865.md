## Summary

Keep open. Current main still compares the canonical two-space rewrite against the raw prior file size, so a valid but verbose/BOM-formatted config can still hit the blocking size-drop guard during models auth persistence. The related #70941 was closed in favor of this stronger shared-writer report, while #74163 is only an open Windows tracker listing this as unresolved.

## What This Changes

The issue reports that `openclaw models auth login` can fail to persist credentials when a PowerShell-created verbose/BOM `openclaw.json` triggers the shared config writer size-drop guard.

## Best Possible Solution

Fix the shared TypeScript config writer so formatting-only shrink is evaluated against a canonicalized previous config or semantic diff, while preserving the size-drop protection for real data loss and keeping the fix available to every config mutation path.

## Reproduction Assessment

Yes. A source-level reproduction can create a valid config whose raw BOM/verbose formatting is more than double the canonical two-space JSON; `models auth login` then follows `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, where current main rejects `size-drop`. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. A shared config-writer fix is the narrowest maintainable solution because the false positive can affect any config mutation; an auth-only destructive-write override would leave other writers broken and weaken the guard inconsistently.

## Repair Work Prompt

Fix the shared config writer false positive tracked by https://github.com/openclaw/openclaw/issues/71865. Current main compares the canonical two-space rewrite of `openclaw.json` against the raw previous file byte size, so a valid PowerShell-created verbose/BOM config can shrink by more than 50% and be rejected as `size-drop` during `openclaw models auth login`. Keep the fix in the shared TypeScript config writer, not as an auth-only bypass. Preserve destructive clobber protection for real data loss, but allow formatting-only shrink when the previous parsed/source config is semantically equivalent apart from the intended mutation. Add focused regression coverage for a valid verbose/BOM previous config that canonicalizes below the size threshold and still persists an auth/config mutation, plus coverage that true destructive shrink remains rejected. Update `CHANGELOG.md` because this is user-facing. Do not weaken `gateway-mode-removed` blocking, do not add startup/load-time migration, and do not change unrelated config schema or provider auth behavior.
