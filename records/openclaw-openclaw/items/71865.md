## Summary

Keep open. Current main still applies the destructive-write size-drop guard to raw previous config bytes before accounting for formatting-only shrink, and the models auth login persistence path reaches that shared writer without a size-drop override or canonical previous-size comparison.

## What This Changes

Auth login should persist provider auth config when an existing `openclaw.json` was written by PowerShell with BOM/verbose formatting instead of being blocked by a false size-drop rejection.

## Best Possible Solution

Update the shared config writer so formatting-only shrink is evaluated semantically, likely by canonicalizing the previous parsed config or using a structured diff before applying the byte-size guard, then cover the BOM/verbose input path through the auth persistence flow.

## Reproduction Assessment

Yes. A focused regression can create a valid BOM-prefixed, deeply indented `openclaw.json`, apply a small auth-profile mutation through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, and observe the current `CONFIG_WRITE_REJECTED` path before the active config changes.

## Solution Assessment

Yes. Fixing the shared writer is the narrowest maintainable solution because other config mutations can hit the same raw-byte false positive; an auth-only override would leave the underlying safety heuristic wrong elsewhere.
