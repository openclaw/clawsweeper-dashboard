## Summary

Keep open: current main and v2026.6.10 still compare canonical next JSON against the raw previous file size, and the linked writer-level fix PR remains open, draft, and conflicting, so this issue remains the canonical tracker.

## What This Changes

The issue reports that PowerShell/BOM-formatted `openclaw.json` can make auth login's normal config rewrite look like a destructive size drop, so OAuth/config changes do not persist.

## Best Possible Solution

Land or replace the linked writer-level canonical-size comparison fix with regression coverage for verbose/BOM formatting while preserving destructive-shrink rejection for real clobbers.

## Reproduction Assessment

Yes, source-level: current main serializes the next config as canonical two-space JSON but compares it to `snapshot.raw` byte size, and auth config patches reach that writer without `allowConfigSizeDrop`. I did not run a live Windows/PowerShell repro in this read-only review.

## Solution Assessment

Yes, the best fix is in the shared config writer rather than an auth-only bypass because the false positive comes from formatting-only shrink in the common writer baseline. The existing linked PR has that shape but must land or be replaced before this issue can close.
