## Summary

Keep open. Current main still applies the config size-drop guard to raw byte length before distinguishing formatting-only shrink, and the models auth persistence path reaches that shared writer without an override; the related tracker keeps this Windows case unresolved rather than superseding it.

## What This Changes

The issue reports that `openclaw models auth login` can report success without persisting credentials when a PowerShell/BOM-heavy `openclaw.json` shrinks under the shared config writer's size-drop guard.

## Best Possible Solution

Update the shared config writer to treat formatting-only shrink differently from semantic data loss, likely by canonicalizing the previous parsed config or using a structured diff before applying the size-drop block, then add targeted writer and models-auth regression tests for BOM/verbose-formatted input.

## Reproduction Assessment

Yes. A focused test can create a valid BOM-prefixed, heavily indented `openclaw.json`, apply a small auth-profile mutation through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, and observe `CONFIG_WRITE_REJECTED` before the active config changes.

## Solution Assessment

Yes. Fixing the shared writer is the narrowest maintainable solution because other config mutations can hit the same false positive; an auth-only override would leave the underlying write-safety heuristic wrong elsewhere.
