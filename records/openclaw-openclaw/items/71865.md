## Summary

Keep open. Current main still compares the canonical two-space rewrite against the raw prior file byte size, so a valid but verbose/BOM-formatted `openclaw.json` can still be rejected as a destructive `size-drop` during models auth persistence.

## What This Changes

The issue requests that `models auth login` persist auth updates when the existing `openclaw.json` was created by PowerShell with verbose formatting or a UTF-8 BOM, without weakening destructive-write protection.

## Best Possible Solution

Fix the shared TypeScript config writer to evaluate formatting-only shrink against a canonicalized previous config or semantic diff while preserving size-drop protection for real data loss, then add focused writer and models-auth regression coverage for the BOM/verbose-format case.

## Reproduction Assessment

Yes. A source-level fixture can create a valid config whose raw BOM/verbose formatting is more than double the canonical two-space JSON; `models auth login` then follows `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, where current main rejects `size-drop`.

## Solution Assessment

Yes. A shared config-writer fix is the narrowest maintainable solution because the false positive can affect any config mutation; an auth-only destructive-write override would leave other commands broken and weaken protection inconsistently.
