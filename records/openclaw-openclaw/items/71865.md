## Summary

Keep open. Current main still sends models auth login through the shared config writer, which compares canonical two-space output bytes against the previous raw file bytes and can still reject a PowerShell/BOM-formatted config as a destructive size drop.

## What This Changes

The issue requests that auth-login config persistence stop falsely rejecting PowerShell/BOM-formatted `openclaw.json` files as destructive size drops.

## Best Possible Solution

Fix the shared config writer to treat formatting-only shrink as safe through semantic or canonicalized comparison while preserving fail-closed rejection for real destructive writes.

## Reproduction Assessment

Yes, source-level. A BOM-prefixed or heavily formatted but semantically valid `openclaw.json` can flow through models auth login to `updateConfig` and `writeConfigFile`, where current main compares canonical next bytes against previous raw bytes and can throw `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. A writer-level semantic or canonicalized size-drop check is the narrow maintainable fix; an auth-only bypass would leave the same false positive in other config mutations.
