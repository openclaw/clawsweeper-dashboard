## Summary

Keep open. Current main still blocks this path because the shared config writer compares canonical two-space JSON output against the previous raw file byte size, and `models auth login` reaches that writer without an opt-out or semantic formatting check.

## What This Changes

The issue asks OpenClaw to let OAuth login persist config changes when `openclaw.json` was authored by PowerShell with BOM/verbose formatting that shrinks substantially when canonicalized.

## Best Possible Solution

Fix the shared config writer to distinguish semantic data loss from formatting-only shrink, then cover the BOM/PowerShell auth-login path while preserving destructive-write rejection.

## Reproduction Assessment

Yes. A focused regression can start from a valid BOM-prefixed, deeply indented `openclaw.json`, run an auth-profile config mutation through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, and observe `CONFIG_WRITE_REJECTED` before persistence.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix because other config mutations can hit the same raw-byte false positive; an auth-only bypass would leave the shared heuristic wrong.
