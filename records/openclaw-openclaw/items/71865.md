## Summary

Keep open. Current main still evaluates the destructive-write size-drop guard against raw previous config bytes before accounting for formatting-only shrink, and the models auth login persistence path reaches that shared writer without an override or semantic comparison.

## What This Changes

The issue asks OpenClaw to let `models auth login` persist OAuth config changes when `openclaw.json` was written by PowerShell with BOM/verbose formatting.

## Best Possible Solution

Teach the shared config writer to distinguish semantic data loss from formatting-only shrink, then add BOM/verbose auth-persistence regression coverage.

## Reproduction Assessment

Yes. A focused regression can create a valid BOM-prefixed, deeply indented `openclaw.json`, apply a small auth-profile mutation through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, and observe the current `CONFIG_WRITE_REJECTED` path.

## Solution Assessment

Yes. Fixing the shared writer is the narrowest maintainable solution because other config mutations can hit the same raw-byte false positive; an auth-only size-drop override would leave the safety heuristic wrong elsewhere.
