## Summary

Keep open. Current main still compares the canonical two-space rewrite against the raw previous file size, so a PowerShell/BOM or otherwise verbose `openclaw.json` can still trip the blocking size-drop guard during `models auth login`; related #70941 was closed in favor of this canonical tracker, and #74163 lists it as unresolved Windows work.

## What This Changes

The issue requests that `models auth login` persist credentials when `openclaw.json` was written in verbose PowerShell/BOM-style formatting instead of being blocked by the config writer's size-drop guard.

## Best Possible Solution

Fix the shared config writer to evaluate formatting-only shrink against a canonicalized previous config or semantic diff while preserving fail-closed protection for real destructive changes, then add focused writer and auth-login regression coverage for the BOM/verbose-format case.

## Reproduction Assessment

Yes. A valid config whose raw BOM/verbose formatting is more than twice the canonical two-space JSON size still flows through `models auth login` to `updateConfig` and `replaceConfigFile`, where current main can raise `CONFIG_WRITE_REJECTED` for `size-drop`.

## Solution Assessment

Yes. A shared config-writer fix is the narrowest maintainable solution because it resolves the false positive for all config mutations; an auth-only destructive-write bypass would leave the same bug in other commands and weaken protection inconsistently.
