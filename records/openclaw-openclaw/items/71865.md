## Summary

Keep open. Current main still serializes the next config to canonical two-space JSON, compares that size to the previous raw file bytes, and blocks the auth-login mutation when a BOM/verbose PowerShell config shrinks by formatting alone.

## What This Changes

The issue asks OpenClaw to let `models auth login` persist auth updates when `openclaw.json` was authored by PowerShell with a UTF-8 BOM and verbose formatting.

## Best Possible Solution

Teach the shared config writer to distinguish formatting-only shrink from semantic data loss, then cover the PowerShell/BOM auth-login path while preserving fail-closed clobber protection.

## Reproduction Assessment

Yes, source-level. A valid BOM-prefixed, deeply formatted `openclaw.json` can flow through `persistProviderAuthResult` to `updateConfig` and `writeConfigFile`, where current main still compares canonical JSON bytes against the previous raw bytes and can throw `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix because command-specific auth bypasses would leave the same false positive in other config mutations.

## Repair Work Prompt

Fix the config writer false positive tracked by https://github.com/openclaw/openclaw/issues/71865. A BOM-prefixed or verbose PowerShell-authored `openclaw.json` can be much larger than OpenClaw's canonical two-space JSON even when the semantic config is unchanged; `models auth login` then persists through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile` and can hit `CONFIG_WRITE_REJECTED` from a formatting-only size drop. Implement the fix in the shared writer, not as an auth-only bypass: compare against a canonicalized/semantic previous payload or otherwise exempt formatting-only shrink while preserving true destructive shrink rejection, `gateway-mode-removed`, validation failures, rejected payload behavior, audit logging, and fail-closed safety. Add focused regression coverage for BOM/verbose PowerShell input through the auth/config persistence path plus coverage that a real destructive shrink is still rejected. Include a user-facing changelog entry unless maintainers decide to fold it into release notes.
