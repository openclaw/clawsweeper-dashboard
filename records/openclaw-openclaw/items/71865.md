## Summary

Keep open. Current main and latest release `v2026.5.7` still compare canonical next config bytes against the previous raw file bytes, so a BOM-prefixed or heavily formatted PowerShell-created `openclaw.json` can still be rejected during auth-login persistence as a formatting-only `size-drop`.

## What This Changes

The issue asks for auth login/config writes to avoid rejecting semantically safe updates when a PowerShell-created `openclaw.json` shrinks mostly because of BOM or formatting normalization.

## Best Possible Solution

Teach the shared config writer to compare semantic or canonicalized content before treating size shrink as data loss, while preserving fail-closed rejection for true clobbers.

## Reproduction Assessment

Yes, source-level. A BOM-prefixed or deeply formatted `openclaw.json` can flow through auth login to the shared writer, where current main still compares canonical next JSON bytes against previous raw bytes and can reject the write as `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix because command-specific auth exceptions would leave the same false positive in other config mutations.
