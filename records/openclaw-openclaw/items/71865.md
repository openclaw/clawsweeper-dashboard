## Summary

Keep open: current main and latest release still compare the next canonical JSON byte count against the raw prior file bytes, so PowerShell/BOM formatting can still trip the shared size-drop guard. ## What This Changes

The issue reports that PowerShell-formatted UTF-8 BOM `openclaw.json` files can make auth login appear successful while the config write is rejected by the size-drop guard.

## Best Possible Solution

Land the writer-level canonical-size fix from the linked PR, or an equivalent shared config-writer repair, then close this issue with the merged commit as proof.

## Reproduction Assessment

Yes. Source inspection shows current main still compares raw prior file bytes with canonical next JSON bytes, and auth config patches reach that writer without a size-drop opt-out.

Yes. A writer-level canonical-size comparison is narrower than an auth-only bypass because the false positive comes from shared config serialization.
