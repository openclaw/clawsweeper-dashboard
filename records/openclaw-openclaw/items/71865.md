## Summary

Keep open. Current main still compares the canonical two-space config rewrite against raw previous file bytes, so a valid BOM/verbose-formatted config can still trip the blocking size-drop guard during models auth persistence; the related Windows tracker also keeps this as remaining work rather than a merged fix.

## What This Changes

Fix models auth login persistence when a PowerShell/BOM formatted openclaw.json shrinks substantially during OpenClaw's canonical JSON rewrite.

## Best Possible Solution

Update the shared config writer to distinguish formatting-only shrink from semantic data loss, likely by comparing against a canonicalized previous config or structured diff before applying the size-drop block, while preserving rejection for true clobbers such as gateway-mode removal or missing large sections.

## Reproduction Assessment

Yes. A targeted fixture can create a valid BOM-prefixed, heavily indented `openclaw.json`, run the models auth persistence path through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, and observe `CONFIG_WRITE_REJECTED` when the canonical rewrite falls below half the raw byte size.

## Solution Assessment

Yes. Fixing the shared writer is the narrowest maintainable solution because models auth is only one caller of the same config write guard; an auth-only override would leave other valid formatting-only rewrites vulnerable and weaken one path's safety semantics.
