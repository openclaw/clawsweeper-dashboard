## Summary

Keep open. Current main and latest release still compare the next canonical two-space JSON byte count against the previous raw file byte count, so a valid BOM/verbose PowerShell config can still trip the shared size-drop guard during auth-login persistence.

## What This Changes

The issue requests that `openclaw models auth login` persist provider OAuth config when the existing `openclaw.json` was written by PowerShell with BOM/verbose formatting.

## Best Possible Solution

Fix the shared config writer to treat formatting-only shrink as safe by using semantic or canonicalized comparison, with focused auth-login and writer regression coverage while preserving fail-closed clobber checks.

## Reproduction Assessment

Yes, source-level. A valid BOM-prefixed, deeply formatted `openclaw.json` can flow through `persistProviderAuthResult` to `updateConfig` and `writeConfigFile`, where current main still compares canonical JSON bytes against the previous raw bytes and can throw `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix because command-specific auth bypasses would leave the same false positive in other config mutations.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/71865. The shared config writer currently serializes the next config as canonical two-space JSON but compares that byte size against the previous raw file bytes, so a valid PowerShell/BOM/verbose `openclaw.json` can shrink by formatting alone and get rejected during `openclaw models auth login`. Implement the fix in the shared writer, not as an auth-only `allowConfigSizeDrop` bypass. Preserve fail-closed protection for real destructive size drops and `gateway-mode-removed`. Add focused regression coverage for a BOM-prefixed, verbose-format config whose semantic auth-login/config mutation should persist, plus coverage showing true clobbers still reject. Include a user-facing changelog entry if the fix changes shipped behavior. Do not weaken config validation, include handling, or destructive-write audit behavior.
