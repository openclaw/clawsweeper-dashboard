## Label Justifications

- `P1`: The issue can block provider auth login persistence for Windows users whose config was created or edited through PowerShell.
- `impact:data-loss`: The reported successful login drops the intended config update instead of persisting it to `openclaw.json`.
- `impact:auth-provider`: The affected workflow is provider OAuth/auth profile persistence through `openclaw models auth login`.

## Summary

Keep open. Current main and the latest release still route models auth login through the shared config writer that compares canonical next JSON bytes against the previous raw file bytes, so a PowerShell/BOM-formatted config can still false-positive the size-drop guard.

## What This Changes

The issue reports that `openclaw models auth login` can report success without persisting auth config when a PowerShell/BOM-formatted `openclaw.json` trips the config writer size-drop guard.

## Best Possible Solution

Fix the shared config writer to treat formatting-only shrink as safe using semantic or canonicalized comparison while preserving fail-closed protection for real destructive writes.

## Reproduction Assessment

Yes, source-level. A semantically valid but raw-large `openclaw.json` can flow through models auth login to `updateConfig`, then `writeConfigFile` compares canonical next bytes against raw previous bytes and can throw `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. The best fix belongs in the shared writer’s size-drop safety check; an auth-only bypass would leave the same formatting false positive in other config mutations.
