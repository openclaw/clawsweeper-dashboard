## Summary

Keep open. Current main still applies the size-drop guard to canonical two-space JSON output versus the previous raw config bytes, and `models auth login` reaches that writer without semantic shrink handling or an explicit size-drop override.

## What This Changes

The issue reports that `models auth login` can appear successful while failing to persist credentials when a PowerShell-formatted BOM/verbose `openclaw.json` trips the config size-drop guard.

## Best Possible Solution

Teach the shared config writer to distinguish formatting-only raw-byte shrink from destructive data loss, then cover the PowerShell/BOM input through the auth persistence flow while keeping true shrink rejection fail-closed.

## Reproduction Assessment

Yes. A focused regression can start from a valid BOM-prefixed, deeply indented `openclaw.json`, apply a small provider auth mutation through `persistProviderAuthResult` -> `updateConfig` -> `replaceConfigFile` -> `writeConfigFile`, and assert the current `CONFIG_WRITE_REJECTED` path leaves the active config unchanged.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix because other config mutations can hit the same raw-byte false positive; an auth-only bypass would leave the shared heuristic wrong elsewhere.
