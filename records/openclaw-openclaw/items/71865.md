## Summary

Keep open. Current main still compares the canonical two-space JSON output size against the previous raw file size, so a valid BOM/verbose PowerShell `openclaw.json` can still be rejected as a destructive size drop during auth persistence.

## What This Changes

The issue requests that model auth login persist config changes when the existing `openclaw.json` is BOM-prefixed or heavily formatted by PowerShell.

## Best Possible Solution

Fix the shared config writer to treat formatting-only shrink as safe through semantic or canonicalized comparison while preserving fail-closed rejection for real destructive writes, then add focused writer and auth-login regression coverage.

## Reproduction Assessment

Yes, source-level. A valid BOM-prefixed or heavily formatted `openclaw.json` can flow through `persistProviderAuthResult` to `updateConfig` and `writeConfigFile`, where current main still compares canonical next bytes against previous raw bytes and can throw `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. A writer-level semantic or canonicalized comparison is the narrowest maintainable fix because an auth-only bypass would leave the same formatting-only false positive in other config mutations.
