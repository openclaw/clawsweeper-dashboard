## Summary

Keep open. Current main still has the shared config writer compare canonical two-space output against the raw previous file size, and `models auth login` still reaches that writer without a normalized-size policy or destructive-write override, so the reported PowerShell/BOM verbose-format false positive remains actionable.

## What This Changes

The issue reports that `openclaw models auth login` can appear successful but fail to persist auth config when a PowerShell-created, BOM/verbose-formatted `openclaw.json` trips the config writer size-drop guard.

## Best Possible Solution

Fix the shared config writer to evaluate size-drop against a canonicalized previous config or semantic diff for valid formatting-only rewrites, while keeping true destructive-write rejection intact, then add writer-level and auth-login regression coverage for the PowerShell/BOM verbose-format case.

## Reproduction Assessment

Yes. A valid config whose raw BOM/verbose formatting is more than twice the canonical two-space JSON size still follows `models auth login` -> `updateConfig` -> `replaceConfigFile` and can hit the blocking `size-drop` guard before persistence.

## Solution Assessment

Yes. A shared writer fix is the narrowest maintainable solution; an auth-only bypass would leave the same false positive in other OpenClaw-owned config mutations and weaken protection unevenly.
