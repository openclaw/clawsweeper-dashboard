## Summary

Keep open. Current main still contains the reported reply-critical surfaces: embedded runs synchronously wait for runtime plugin loading before model resolution, that helper still performs an unscoped runtime registry ensure, and bundled runtime dependency fallback installs still use synchronous process execution. Current main has adjacent #75360 and #75283 latency/runtime-deps mitigations, but those do not prove this broader Docker first-reply runtime-plugins/system-prompt event-loop starvation report is fixed.

## What This Changes

The issue reports v2026.4.29 first post-boot embedded LLM calls blocking the Node event loop for 20-79 seconds around runtime plugin loading or system-prompt assembly, causing Telegram and WhatsApp channel drops.

## Best Possible Solution

Keep this open until maintainers capture a focused first post-boot Docker/Linux profile on current main that separates runtime plugin resolution, bundled runtime dependency staging, provider/catalog hooks, channel capability lookup, prompt construction, and token estimation. The durable end state should remove the confirmed broad synchronous work from the reply critical path or prewarm it outside channel timeout windows, with regression coverage for the chosen boundary.

## Reproduction Assessment

Unclear. The reporter gives concrete v2026.4.29 Docker timings and current main still has matching synchronous/broad hot paths, but this review did not find a current-main profile, fixture, or test reproducing the 20-79 second event-loop stall end to end.

## Solution Assessment

Unclear. Eager or worker-thread tokenizer initialization is not proven to be the best fix; the safer route is to profile and then scope or defer the confirmed runtime plugin, dependency staging, provider/catalog, channel-capability, or token-estimation work.
