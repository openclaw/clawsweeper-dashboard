## Summary

Keep open. Current main still lacks both requested surfaces: Android has no `sms.read` invoke command, and the gateway/channel stack has no Android-node SMS ingress channel. The linked implementation PR is closed unmerged, and this crosses SMS privacy, Android permissions, inbound agent-trigger authority, outbound SMS replies, and allowlist semantics, so it needs maintainer product/security review before implementation.

## What This Changes

The issue requests an Android `sms.read` command for fetching SMS by message id or thread id, plus SMS as a text-only allowlist-gated messaging channel through a connected Android node.

## Best Possible Solution

Keep this issue open and split the work into approved design pieces: first define a narrow Android `sms.read` command contract with dangerous-command policy and tests, then separately decide whether Android-node SMS ingress belongs as a bundled channel, a plugin-backed channel with a generic node-event seam, or outside OpenClaw core.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence absence check. Current-main source inspection and exact searches show no `sms.read` enum/spec/handler, no `RECEIVE_SMS`/`SMS_RECEIVED` receiver, no `sms.allowFrom`, and no gateway `sms.received` case.

## Solution Assessment

No, the combined request is not yet the best automated implementation path. The maintainable path is to split command-read behavior from inbound SMS channel routing and get maintainer security/product approval before code changes.
