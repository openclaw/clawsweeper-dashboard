## Summary

Keep open. Current main still lacks both requested surfaces: Android exposes `sms.send` and `sms.search`, but not `sms.read`, and the gateway has no Android-node SMS ingress channel. The linked implementation PR is closed unmerged, and the remaining work needs maintainer product/security decisions before implementation.

## What This Changes

Request to add an Android `sms.read` invoke command plus a node-mediated SMS messaging channel with inbound receiver, outbound replies, and phone allowlist config.

## Best Possible Solution

Keep the issue open and split the design: define a narrow, security-reviewed Android `sms.read` contract first, then decide whether SMS ingress should be a bundled channel, an external plugin using a generic node-event seam, or outside core.

## Reproduction Assessment

Not applicable. This is a feature request, not a broken existing contract; source inspection verifies current main lacks `sms.read`, an inbound SMS receiver, and a gateway `sms.received` channel path.

## Solution Assessment

Unclear. The requested direction is plausible, but the linked all-in-one approach is not the best maintainable path until maintainers split and approve the Android SMS permission model, privacy policy, and channel/plugin boundary.

## Risks / Open Questions

- Inbound SMS would expose private text content to agents, so maintainers need to decide permission prompts, logging/redaction, retention, and allowlist semantics before implementation.
- Outbound replies through `sms.send` create real phone/SMS side effects and should remain aligned with dangerous node-command policy and node-declared capability gates.
- The right boundary is still unsettled: Android app command support must be built into the node, while the messaging channel may belong in a plugin if the needed node-event seam is made generic.
