## Summary

Keep open. Current main still only supports Android `sms.send`/`sms.search`; it has no `sms.read` command, no Android `RECEIVE_SMS`/receiver bridge, no `sms.received` gateway ingress, and no `channels.sms`/`sms.allowFrom` config surface. The linked implementation PR is closed unmerged, and the adjacent open SMS/MMS Play-flavor PR does not resolve this issue.

## What This Changes

The issue requests an Android `sms.read` invoke command plus a full Android-node-mediated SMS messaging channel with allowlisted inbound texts and outbound `sms.send` replies.

## Best Possible Solution

Keep this open for maintainer design: first define a narrow `sms.read` command contract, then separately decide whether SMS ingress belongs in a bundled channel, a plugin plus new node-event API, or outside core.

## Reproduction Assessment

Not applicable; this is a feature request, not a bug. Source inspection on current main shows the requested command, event, receiver, and channel config surfaces are absent.

## Solution Assessment

Unclear. The requested direction may be useful, but a broad built-in SMS channel is not the best maintainable path until maintainers decide the Android permission model, privacy policy, and plugin/core boundary.
