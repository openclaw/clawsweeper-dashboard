## Summary

Keep open. Current main still lacks both requested surfaces: Android exposes `sms.send` and `sms.search` but not `sms.read`, and the gateway/channel stack has no Android-node SMS ingress channel or `sms.allowFrom` configuration. The linked implementation PR is closed unmerged, while the remaining work crosses Android SMS permissions, private-message reads, inbound agent-trigger authority, outbound SMS replies, and sender allowlist semantics, so it needs maintainer product/security review before implementation.

## What This Changes

Request to add an Android `sms.read` invoke command for message/thread reads and an allowlist-gated SMS messaging channel mediated by a connected Android node.

## Best Possible Solution

Keep the issue open and split the work into approved design pieces: first define a narrow Android `sms.read` command contract with dangerous-command policy and tests, then separately decide whether Android-node SMS ingress belongs as a bundled channel, a plugin-backed channel with any needed generic node-event seam, or outside OpenClaw core.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence absence check. Current-main source, docs, and exact searches show no `sms.read` enum/spec/handler, no `SMS_RECEIVED`/`RECEIVE_SMS` receiver path, no `sms.allowFrom`, and no gateway `sms.received` case.

## Solution Assessment

No, the combined request is not yet the best automated implementation path. The safer maintainable path is to separate direct SMS read support from inbound SMS channel routing and require maintainer security/product approval before code changes.
