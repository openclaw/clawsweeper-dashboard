## Summary

Keep open. Current main and v2026.6.8 still lack the Android-specific `sms.read` command and Android-node SMS ingress path; the merged Twilio-backed `sms` plugin covers provider-hosted SMS only, so the remaining Android-device work needs maintainer product/security direction.

## What This Changes

Request to add an Android `sms.read` invoke command plus an Android-node-mediated SMS channel with inbound routing, SMS replies, and allowlist-style sender control.

## Best Possible Solution

Keep this as the canonical Android-device SMS tracker and split any implementation into a narrow, security-reviewed `sms.read` contract followed by a separate Android SMS ingress/channel ownership decision.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source and release inspection give a high-confidence current-main gap: Android exposes `sms.send` and `sms.search`, while `sms.read`, `sms.received`, and Android SMS receiver/ingress code are absent.

## Solution Assessment

No, not as one automated bundled-core change. The safer solution is to split a security-reviewed Android read command from the channel/plugin/config decision, while treating the existing Twilio SMS plugin as provider-hosted SMS rather than Android-device SMS.
