## Summary

Keep open. Current main still lacks the requested Android `sms.read` command and an Android-node SMS ingress channel, while the linked implementation PR is closed unmerged and the remaining work needs privacy/security and product boundary decisions before implementation.

## What This Changes

The issue requests an Android `sms.read` invoke command plus an allowlist-gated SMS messaging channel backed by connected Android nodes.

## Best Possible Solution

Keep the issue open and split the design into an approved Android `sms.read` command contract first, then a separately reviewed SMS ingress channel design with explicit privacy, allowlist, and outbound-reply controls.

## Reproduction Assessment

Not applicable as a feature request, but the absence check is high-confidence. Current-main source, docs, and exact searches show no `sms.read` enum/spec/handler, no `RECEIVE_SMS`/`sms.received` ingress path, and no `sms.allowFrom` or `channels.sms` configuration.

## Solution Assessment

No, this is not yet a safe automated implementation path. The maintainable path is to separate direct SMS reads from inbound SMS channel routing and require maintainer product/security approval before code changes.

## Risks / Open Questions

- The request would read private SMS content and let external phone numbers trigger agent runs, so sender normalization, allowlist semantics, logging, and retention need explicit approval.
- Outbound agent replies through `sms.send` create a real-world messaging side effect and should be reviewed with dangerous-command policy and rate/authority controls.
- It is not yet settled whether Android-node SMS ingress belongs in bundled core, an external channel plugin with a new generic node-event seam, or outside OpenClaw.
