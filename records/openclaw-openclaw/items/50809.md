## Summary

Keep open. Current main and v2026.6.10 still lack the requested Android-node `sms.read` command and Android-device SMS ingress/reply channel; the remaining work is a product and security decision, not conservative cleanup.

## What This Changes

Request to add an Android-node `sms.read` command plus an allowlist-gated SMS channel that routes inbound device SMS to agents and sends replies through the connected Android node.

## Best Possible Solution

Keep this as the canonical Android-device SMS tracker and split the work into a security-reviewed `sms.read` command contract plus a separate Android-node SMS channel ownership decision that coexists cleanly with the Twilio SMS plugin.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source and release inspection give a high-confidence current gap: Android exposes `sms.send` and `sms.search`, while `sms.read`, `sms.received`, and Android SMS receiver ingress are absent.

## Solution Assessment

No, not as one combined automated change. The safer path is to split the Android read command from the Android-device channel/config/security decision and keep the existing Twilio SMS plugin as a separate provider-hosted channel unless maintainers choose a shared contract.
