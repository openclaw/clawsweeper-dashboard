## Summary

Keep open. Current main still lacks both requested surfaces: no Android `sms.read` invoke command and no Android-node-mediated SMS channel ingress path. The linked implementation PR is closed unmerged, and the remaining work is security-sensitive enough to need maintainer product/security review before implementation.

## What This Changes

Request to add an Android node `sms.read` invoke command for message/thread reads and to make SMS a DM-only, allowlist-gated messaging channel through connected Android nodes.

## Best Possible Solution

Keep the issue open and split the work into approved pieces: first define a narrow, dangerous-command-aware Android `sms.read` contract, then separately decide whether Android-node SMS ingress belongs as a bundled channel, a plugin-backed channel with any needed generic node-event seam, or outside OpenClaw core.

## Reproduction Assessment

Not applicable as a runtime bug report, but yes for an absence check. Current-main source inspection and exact searches show no `sms.read` enum/spec/handler, no `RECEIVE_SMS` receiver or `sms.received` gateway case, no `sms.allowFrom` config path, and no SMS chat channel docs.

## Solution Assessment

No, not as one combined automated change. The maintainable path is to separate SMS read access from inbound SMS channel routing and get maintainer security/product approval before code changes.
