## Summary

Keep open. Current main and v2026.6.8 still lack the requested Android `sms.read` command and Android-node SMS ingress path; Android `sms.search` and the Twilio-backed `sms` plugin only partially overlap, so the remaining work needs maintainer product/security direction.

## What This Changes

Add Android-node `sms.read` by id/thread and route allowlisted device SMS conversations to agents with outbound replies through the connected Android node.

## Best Possible Solution

Keep this as the canonical Android-device SMS tracker and split any implementation into maintainer-approved pieces: a narrow `sms.read` contract first, then a separate Android SMS ingress/channel ownership decision.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source inspection gives a high-confidence current-main gap: Android exposes `sms.send` and `sms.search`, while `sms.read`, `sms.received`, and an Android SMS receiver/ingress path are absent.

## Solution Assessment

No, not as a single automated bundled-core change. The safer path is a maintainer-approved split between a security-reviewed Android read command and a channel/plugin ownership decision, while reusing the existing Twilio SMS channel only where provider-hosted SMS applies.

## Risks / Open Questions

- Android-node SMS ingress would let incoming private texts wake agents and let agents send carrier SMS, so maintainers need a fail-closed permission, pairing/allowlist, logging, redaction, and audit model before implementation.
- Current main already uses channel id `sms` for the Twilio-backed plugin, so Android-device SMS needs an explicit channel naming, config ownership, or plugin/API decision to avoid competing SMS sources of truth.
- The Play build intentionally removes SMS send/search because READ_SMS and SEND_SMS are restricted permissions, so any Android-device SMS expansion may need third-party-build-only scope or an explicit Play policy position.
