## Summary

Keep open. Current main and v2026.6.8 still lack the requested Android `sms.read` command and Android-node SMS ingress path; the shipped SMS channel is Twilio-backed and does not replace device SMS, so the remaining work needs maintainer product and security direction.

## What This Changes

The issue requests an Android-node `sms.read` invoke command plus an Android-device SMS channel that routes allowlisted inbound texts to agents and replies through `sms.send`.

## Best Possible Solution

Keep this as the canonical Android-device SMS tracker and split it into maintainer-approved pieces: a narrow `sms.read` contract, then a separate channel/API decision for Android SMS ingress that coexists cleanly with the Twilio SMS plugin.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source inspection gives a high-confidence current-main gap: Android exposes `sms.send` and `sms.search`, while `sms.read`, `sms.received`, and an Android SMS receiver/ingress path are absent.

## Solution Assessment

No, not as a single automated bundled-core change. The safer path is a maintainer-approved split between a security-reviewed Android read command and a channel/plugin ownership decision, while reusing the existing Twilio SMS channel only where provider-hosted SMS applies.
