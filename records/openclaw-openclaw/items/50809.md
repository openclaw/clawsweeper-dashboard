## Summary

Keep open. Current main still lacks both requested surfaces: Android exposes `sms.send` and `sms.search` but not `sms.read`, and the gateway/channel stack has no Android-node SMS ingress channel or `sms.allowFrom` configuration; the linked implementation PR is closed unmerged and the remaining work needs maintainer product/security approval.

## What This Changes

Request adding an Android `sms.read` invoke command and treating device SMS as a DM-only, allowlist-gated messaging channel through connected Android nodes.

## Best Possible Solution

Keep the issue open and split it into approved design pieces: first a narrow Android `sms.read` command contract, then a separate decision on whether Android-node SMS ingress belongs in core, a plugin-backed channel with new mobile/node seams, or outside OpenClaw.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence absence check. Current-main source, docs, tests, and exact searches show no `sms.read` command, no Android SMS receiver, no `sms.received` gateway path, and no SMS channel config/docs.

## Solution Assessment

No, the combined request is not yet the best automated implementation path. The safer maintainable path is to separate direct SMS read support from inbound SMS channel routing and require maintainer security/product approval before code changes.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it would read private SMS content and let inbound SMS messages trigger agents and outbound replies.
