## Summary

Keep open. Current main still lacks the requested Android `sms.read` invoke command and lacks an Android-node-mediated SMS channel ingress path; the linked implementation PR is closed and unmerged, and the remaining work needs maintainer product and security review before implementation.

## What This Changes

The issue requests an Android `sms.read` command for fetching messages by SMS id or thread id, plus SMS as a text-only allowlist-gated messaging channel through a connected Android node.

## Best Possible Solution

Keep this issue open and split the work into approved design pieces: first define a narrow Android `sms.read` command contract with dangerous-command policy and tests, then separately decide whether Android-node SMS ingress belongs as a bundled channel, a plugin-backed channel with a generic node-event seam, or outside OpenClaw core.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence absence check. Current-main source inspection and exact searches show no `sms.read` enum/spec/handler, no `RECEIVE_SMS`/`SMS_RECEIVED` receiver, no `sms.allowFrom`, and no gateway `sms.received` case.

## Solution Assessment

No, the combined request is not the best automated implementation path yet. The maintainable path is to split command-read behavior from inbound SMS channel routing and get maintainer security/product approval before code changes.
