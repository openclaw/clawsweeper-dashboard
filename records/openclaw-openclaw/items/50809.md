## Summary

Keep open. Current main still exposes Android `sms.send` and `sms.search` only, has no `sms.read` invoke command, and has no Android-node SMS ingress channel; the linked implementation PR is closed unmerged, and the remaining work needs maintainer product/security decisions before implementation.

## What This Changes

The issue requests an Android `sms.read` invoke command plus an allowlist-gated SMS messaging channel that routes inbound Android texts to agents and replies through `sms.send`.

## Best Possible Solution

Split the work into maintainer-approved pieces: define a narrow, security-reviewed Android `sms.read` contract first, then decide whether Android-node SMS ingress belongs in core, a plugin-backed channel, or outside OpenClaw.

## Reproduction Assessment

Not applicable. This is a feature request, not a broken existing contract; source inspection confirms current main lacks `sms.read`, an inbound SMS receiver, and a gateway `sms.received` channel path.

## Solution Assessment

Unclear. The requested direction is plausible, but an all-in-one core implementation is not the best maintainable path until maintainers approve the Android SMS permission model, privacy policy, allowlist semantics, and channel/plugin boundary.

## Risks / Open Questions

- Inbound SMS would expose private text content to agents, so permission prompts, allowlist semantics, logging, redaction, and retention need maintainer decisions.

- The correct boundary is unsettled: Android node support must be in the app, while channel routing may belong in a plugin or a new generic node-event seam.
