## Summary

Keep open: the PR targets a real cron delivered/error mismatch, but the timer-level normalizer currently treats channel delivery as proof that an Apply Patch/Edit failure was nonfatal, which can clear real failed work and suppress retry/backoff.

## What This Changes

The PR normalizes delivered cron Apply Patch/Edit warning results to `ok`, applies that normalized status to task and finished events, and lets direct text delivery use completed descendant summaries when the parent run produced no text.

## Best Possible Solution

Keep delivery status as channel evidence, add an explicit cron isolated-run success/nonfatal-warning classification before fatal payload outcome selection, then add the required changelog entry and focused regression coverage.

## Reproduction Assessment

Yes. A focused unit path can pass `status: "error"`, `delivered: true`, and an Apply Patch/Edit error into the new normalizer and show it clears the run error even though current delivery code only proves the message reached the channel.

## Solution Assessment

No. The current PR is too late in the pipeline: normalizing in `timer.ts` by error text conflates delivery with task success; the safer fix belongs in cron isolated payload/result classification where success evidence is still available.
