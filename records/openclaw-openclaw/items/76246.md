## Summary

Keep open: the PR addresses a real cron status problem, but the timer-level normalization can erase real tool failures after channel delivery succeeds, and the user-facing fix still needs a changelog entry.

## What This Changes

The PR normalizes delivered Apply Patch/Edit warning results to `ok` in cron status persistence and lets completed descendant summaries be delivered when the parent cron turn produced no text.

## Best Possible Solution

Classify Apply Patch/Edit warnings as nonfatal inside isolated cron payload/result handling only when explicit success evidence exists, keep timer persistence as a mechanical consumer of that classification, and add the required changelog entry.

## Reproduction Assessment

Yes. A focused unit path can pass `status: "error"`, `delivered: true`, and `⚠️ 🩹 Apply Patch failed` into the PR's new normalization path and observe it clearing `lastRunStatus`, `lastError`, and backoff state without proving the tool failure was nonfatal.

## Solution Assessment

No. Normalizing in `timer.ts` based on channel-delivery metadata is too late in the pipeline; the safer fix belongs in isolated payload/result classification where successful output after a warning is still visible.
