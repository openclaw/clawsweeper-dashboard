## Summary

Keep open: the PR addresses a real cron delivery/status problem, but the current patch can mask real Apply Patch/Edit failures because it normalizes status in timer persistence from `delivered` plus error text alone, and it also lacks the required changelog entry.

## What This Changes

The branch updates cron timer persistence and isolated delivery dispatch so delivered Apply Patch/Edit warning results are normalized to `ok` and completed descendant summaries can be delivered when the parent cron run has no text.

## Best Possible Solution

Classify Apply Patch/Edit warning payloads in the isolated cron result layer only when explicit success evidence exists, keep timer persistence as a consumer of that classification, retain the descendant-summary fallback, and add a release-note entry.

## Reproduction Assessment

Yes. The PR's new unit path passes `status: "error"`, `delivered: true`, and `⚠️ 🩹 Apply Patch failed` into `applyJobResult` and observes it clearing the error/backoff state; the same path demonstrates how a real delivered error notification would be masked.

## Solution Assessment

No. Timer-layer normalization based on channel delivery metadata is not the narrowest maintainable fix; the safer solution belongs in isolated payload/result classification where successful output and warning context are still available.
