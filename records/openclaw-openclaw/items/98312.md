## What This Changes

The PR changes configured fallback model resolution so maxTokens stays undefined when config, provider, sibling model, and static catalog sources do not provide it, with a regression test for a custom Xiaomi OpenAI-compatible provider.

## Best Possible Solution

Land the narrow resolver change after contributor-supplied real behavior proof, with maintainers explicitly accepting provider-default output budgets for unconfigured custom maxTokens.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.11 can synthesize DEFAULT_CONTEXT_TOKENS into model.maxTokens for a configured fallback model, and the OpenAI-compatible transport forwards that value as max_completion_tokens; I did not run a live Xiaomi request in this read-only review.

## Solution Assessment

Yes. Removing the resolver's wire-facing synthetic maxTokens fallback is the narrowest maintainable fix; catalog-only updates would not cover arbitrary custom providers, and a new config option would add surface for a broken omitted-metadata case.
