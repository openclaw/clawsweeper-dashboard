## Summary

Keep open: current main still has the reported sticky auto-fallback behavior, and the docs/tests describe it as intentional policy rather than an obsolete bug. The remaining decision is what recovery semantics OpenClaw should use after a primary model cooldown expires.

## What This Changes

Request recovery from an auto-selected fallback model after the primary model cooldown expires instead of persisting the fallback across later turns or inherited sessions.

## Best Possible Solution

Define a cooldown-aware recovery policy for auto-sourced model overrides, then align candidate ordering, inherited-session behavior, diagnostics, docs, and focused regression tests around that policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a direct or parent session with an auto fallback override, let the primary cooldown expire, and the next selection keeps the fallback while the fallback runner tries that model before the configured primary.

## Solution Assessment

Unclear. Retrying the primary first after cooldown would address the report, but it would reverse documented behavior; a maintainer should choose the recovery policy before code changes.
