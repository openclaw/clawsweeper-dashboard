## Summary

Keep open: current main still has the sticky auto-fallback behavior described here, and the related cooldown-cascade fixes do not cover recovery of an auto-pinned fallback after the primary cooldown expires. The behavior is also documented as intentional cache-friendly policy, so this needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests that model fallback stop remaining pinned to the last successful fallback after the primary model cooldown expires, including across new Feishu sessions.

## Best Possible Solution

Define the intended recovery policy for `modelOverrideSource: "auto"` after cooldown expiry, then update fallback ordering, session inheritance/reset behavior, diagnostics, and docs/tests together.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create a session or parent session with an auto fallback override and an expired primary auth cooldown; current selection preserves or inherits the fallback, and `runWithModelFallback` starts from that fallback before considering the configured primary.

## Solution Assessment

Unclear. Retrying the primary first after cooldown would address the report, but it conflicts with the current documented sticky-fallback behavior; a configurable or cooldown-aware recovery policy may be safer than an unconditional behavior flip.
