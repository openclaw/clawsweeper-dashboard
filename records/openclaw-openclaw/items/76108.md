## Summary

Keep this open: current main still matches the reported sticky auto-fallback behavior. The code and docs intentionally preserve auto fallback overrides until reset, so the requested post-cooldown recovery is a distinct remaining product/runtime policy issue rather than something already implemented by the related cooldown fixes.

## What This Changes

The issue requests that OpenClaw retry the configured primary or rebuild the fallback order after a primary model cooldown expires instead of staying pinned to the last successful fallback.

## Best Possible Solution

Add an explicit, documented recovery rule for auto-sourced fallback overrides that can re-evaluate or clear the fallback pin after the matching primary cooldown expires, while preserving strict user-selected `/model` behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: create a session entry with `providerOverride`, `modelOverride`, and `modelOverrideSource: "auto"`; current model selection keeps that fallback across turns, and tests assert the same behavior.

## Solution Assessment

Unclear. The requested direction fixes the user-visible recovery gap, but the current sticky behavior is documented and tested as intentional, so maintainers should choose the recovery policy before implementation.
