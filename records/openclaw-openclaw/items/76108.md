## Summary

Keep open: current main still implements the sticky auto-fallback behavior described here, including parent-session inheritance, and the behavior is documented/tested as intentional. This needs maintainer policy direction rather than cleanup closure.

## What This Changes

The issue requests that automatically selected fallback models recover to the configured primary after the primary model's cooldown expires, instead of remaining sticky across sessions.

## Best Possible Solution

Define a cooldown-aware recovery policy for `modelOverrideSource: "auto"`, then update session selection, fallback ordering, diagnostics, docs, and focused tests together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist an auto fallback override, let the primary auth cooldown expire, and observe that selection still starts from the stored fallback and may inherit it from a parent session.

## Solution Assessment

Unclear. Retrying the primary first after cooldown would address the report, but a configurable or cooldown-aware reset/probe policy may be safer than unconditionally replacing the documented sticky fallback behavior.
