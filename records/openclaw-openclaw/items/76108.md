## Summary

Keep open: current main still implements and documents sticky auto-fallback selection after a cooldowned primary, so this is not cleanup-closeable as implemented or duplicate cleanup.

## What This Changes

The issue asks OpenClaw to retry the configured primary model after its cooldown expires instead of continuing to pin an auto-selected fallback across turns or derived sessions.

## Best Possible Solution

Define the intended cooldown-recovery policy for auto fallback overrides, then update candidate ordering, session inheritance/reset behavior, diagnostics, docs, and focused tests together.

## Reproduction Assessment

Yes. A source-level reproduction is clear: persist a direct or parent `modelOverrideSource: "auto"` fallback, expire the primary auth cooldown, and observe that selection starts from the fallback before the configured primary.

## Solution Assessment

Unclear. Retrying the primary first after cooldown would address the report, but a cooldown-aware policy or configurable recovery behavior is safer than silently reversing the documented sticky fallback semantics.
