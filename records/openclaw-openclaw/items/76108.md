## Summary

Close as duplicate/superseded: #44935 is the open canonical recovery-policy issue, and current main still shows this report is the same sticky auto-fallback/session-inheritance behavior rather than a separate MiniMax/Feishu-specific root cause.

## What This Changes

Request recovery from an auto-selected fallback model after primary cooldown expiry instead of persisting the fallback across later turns and inherited or new channel sessions.

## Best Possible Solution

Use #44935 as the canonical design thread for cooldown-aware primary recovery, preserving this report's MiniMax/Feishu reproduction as supporting context.

## Reproduction Assessment

Yes. Source inspection gives high confidence: a session or parent session with `modelOverrideSource: "auto"` keeps selecting the fallback after `clearExpiredCooldowns` clears only auth-state fields, and the fallback runner starts from that selected fallback.

## Solution Assessment

Yes for triage. Keeping one canonical recovery-policy issue is safer than implementing a one-off primary-first change here, because current docs/tests intentionally protect sticky fallback until maintainers define the recovery behavior.
