## Summary

Keep this PR open: it addresses a real Telegram message-tool-only preview problem that is not implemented on current main, but it should not merge while internal harness/default and tool-policy helpers are promoted through public `plugin-sdk/reply-runtime` without maintainer API direction.

## What This Changes

The PR changes Telegram dispatch to suppress transient previews when source replies resolve to message-tool-only delivery, extracting shared reply-runtime helpers for source-visible defaults and message-tool availability.

## Best Possible Solution

Keep the Telegram fix open, but resolve whether these helpers are a public Plugin SDK contract or replace them with a narrower maintainer-approved bundled/internal seam before merge.

## Reproduction Assessment

Source-reproducible: current main creates Telegram draft streams before core dispatch can report `sourceReplyDeliveryMode: "message_tool_only"`, while the PR moves that policy check before draft callbacks are installed. The linked issue and PR body include real Telegram screenshots/log proof, but this read-only review did not run a live Telegram repro.

## Solution Assessment

No, not fully. Suppressing Telegram previews from the shared source-reply resolver is plausible, but exporting internal harness/tool-policy helpers through public `reply-runtime` needs maintainer API direction or a narrower seam.
