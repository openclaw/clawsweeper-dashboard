## Summary

Keep open for maintainer review: the patch appears correct and well proved, with no blocking code findings, but it intentionally promotes new helpers through the public plugin SDK `reply-runtime` surface and needs explicit API-contract acceptance before merge.

## What This Changes

The PR extracts shared source-visible-reply/default and message-tool availability helpers, exports them via `plugin-sdk/reply-runtime`, and uses them in Telegram dispatch to suppress draft/reasoning previews when source replies resolve to message-tool-only.

## Best Possible Solution

Land the Telegram behavior fix once maintainers accept the additive `reply-runtime` SDK contract; otherwise expose the policy through a narrower approved runtime seam without changing the user-visible behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main creates Telegram draft/reasoning preview callbacks before core later resolves non-room source delivery to `message_tool_only`. I did not run a live current-main reproduction in this read-only review.

## Solution Assessment

Yes, with an API-contract caveat. Pre-resolving the same shared source-reply policy before Telegram draft lanes is the right fix shape, but maintainers still need to accept the public `reply-runtime` helper exports or choose a narrower seam.
