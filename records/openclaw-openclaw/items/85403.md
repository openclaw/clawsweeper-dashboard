## Summary

Keep open: current main still creates Telegram draft/reasoning previews before core dispatch resolves message-tool-only source delivery, so the proposed behavior is still useful. The merge blocker is not staleness; it is that the PR exposes new public `reply-runtime` Plugin SDK API before maintainers choose and document the intended contract shape.

## What This Changes

This PR extracts source-visible-reply default and message-tool availability logic from core dispatch, exports it through `openclaw/plugin-sdk/reply-runtime`, and uses it in Telegram dispatch to suppress draft/reasoning previews when source delivery is message-tool-only.

## Best Possible Solution

Land the Telegram preview suppression only after maintainers choose the SDK contract shape, either by documenting the public `reply-runtime` helpers or replacing them with a narrower bundled/internal seam that keeps source-reply policy single-source.

Yes, source/proof-backed: the linked Telegram issue gives a concrete prompt and screenshots, and current main shows previews are created before source-reply suppression is known. I did not run a fresh live current-main Telegram reproduction in this read-only review.

## Solution Assessment

No, not yet: the Telegram suppression path is plausible and well-proven, but promoting core harness/model/tool-policy helpers through public `reply-runtime` is not the best merge shape until maintainers choose the API contract.
