## Summary

Keep this PR open: the code change is focused and appears to address a real ordinary-timeout/auth-profile ordering gap on current main, but it is not merge-ready because the external PR only supplies synthetic/unit proof and the default fallback/auth-profile timing change needs maintainer-visible acceptance.

## What This Changes

The PR broadens the embedded agent timeout retry gate so the first ordinary assistant-stage timeout retries the same model/profile before auth-profile rotation, with focused failover tests.

## Best Possible Solution

Land a bounded same-profile ordinary-timeout retry only after real runtime proof shows the first timeout stays local, the second timeout still rotates or falls back, and non-timeout auth/rate-limit/quota failures remain fast-failover paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: current main's retry gate is idle-only and the rotate_profile branch advances auth profiles before any same-profile ordinary-timeout retry; I did not establish a live gateway/provider reproduction.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes: the fix sits at the existing assistant failover helper and run-loop gate rather than duplicating provider or auth-profile policy, but merge should wait for real runtime proof and maintainer acceptance of the bounded fallback delay.
