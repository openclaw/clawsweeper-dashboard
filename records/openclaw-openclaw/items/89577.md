## Summary

Keep this PR open for maintainer review: the patch is a focused security hardening and appears correct, with sufficient browser proof, but it intentionally replaces the old `Math.random()` fallback with a fail-closed Web Crypto requirement that maintainers should accept explicitly before merge.

## What This Changes

The PR changes the Canvas A2UI host action ID generator to keep `crypto.randomUUID()` first, use `crypto.getRandomValues()` for the fallback suffix, and throw when Web Crypto randomness is unavailable.

## Best Possible Solution

Land the Web Crypto-only A2UI action ID path if maintainers accept the no-Web-Crypto fail-closed behavior; otherwise revise the compatibility policy explicitly instead of keeping silent predictable IDs.

## Reproduction Assessment

Yes by source inspection: current main's `#makeActionId()` falls back to `Math.random()` when `crypto.randomUUID()` is unavailable. The PR body also supplies browser output exercising the intended fallback after removing `randomUUID`.

## Solution Assessment

Yes, for the A2UI host path this is the narrow maintainable fix: keep `randomUUID()` first, use Web Crypto random bytes for the fallback, and avoid silently predictable IDs. The remaining question is the maintainer policy choice around fail-closed behavior in no-Web-Crypto WebViews.
