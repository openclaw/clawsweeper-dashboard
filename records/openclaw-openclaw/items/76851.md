## Summary

Keep open: this targets a valid latency regression, but the PR still performs one clone-heavy secrets snapshot read per plugin-tool construction, which is the same construction-time clone current main already performs.

## What This Changes

The PR adds a changelog entry and changes plugin tool setup to cache the active secrets runtime snapshot for the one-shot runtimeConfig while keeping getRuntimeConfig as a live getter.

## Best Possible Solution

Use config-only runtime/source snapshot access, or an equivalent lightweight accessor, for the construction-time runtimeConfig while preserving the live ctx.getRuntimeConfig() contract for long-lived callbacks.

## Reproduction Assessment

Yes for the PR defect via source inspection: current main already supplies runtimeConfig, so resolvePluginTools does not invoke getRuntimeConfig during construction, and the PR still performs one getActiveSecretsRuntimeSnapshot() call to build that value. I did not establish live timing for the original 8.3s report in this read-only pass.

## Solution Assessment

No. The current diff is not the best fix because it preserves the expensive construction-time secrets snapshot clone; a narrower maintainable fix should use the existing config-only runtime snapshot helpers and keep the getter live.
