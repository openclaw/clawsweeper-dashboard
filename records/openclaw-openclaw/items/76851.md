## Summary

Keep open: this PR targets a real-looking latency regression, but the diff changes the documented live `ctx.getRuntimeConfig()` contract for plugin tools and would make existing regression coverage fail after a runtime config refresh.

## What This Changes

The PR caches the active secrets runtime snapshot in `resolveOpenClawPluginToolsForOptions` and adds a changelog entry for the `core-plugin-tools` latency regression.

## Best Possible Solution

Keep the eager `runtimeConfig` value cheap for tool construction while preserving a live `getRuntimeConfig` callback, then validate the existing live-refresh test and the targeted plugin-tool tests with before/after timing evidence.

## Reproduction Assessment

Yes for the PR-introduced regression: the current integration test captures `ctx.getRuntimeConfig`, refreshes the active secrets runtime snapshot, and expects the getter to return the new runtime config. The proposed captured snapshot would keep returning the pre-refresh value.

## Solution Assessment

No. Caching a single snapshot for the eager construction-time config is a good direction, but making the runtime getter return that captured snapshot changes documented behavior and is not the narrowest safe fix.
