## Summary

Keep this PR open: the force-pushed version preserves live `getRuntimeConfig()` semantics, but the new cache is still ineffective for the linked `core-plugin-tools` latency regression because it performs the same one clone-heavy secrets snapshot read during plugin tool construction as current main.

## What This Changes

The PR adds a changelog entry and changes `src/agents/openclaw-plugin-tools.ts` to precompute the one-shot plugin tool `runtimeConfig` while leaving `getRuntimeConfig` as a live getter.

## Best Possible Solution

Use the existing config-only runtime/source snapshots or an equivalent lightweight accessor to avoid the secrets snapshot clone on plugin-tool prep, keep `ctx.getRuntimeConfig()` live, and add regression coverage plus timing proof for the linked latency issue.

## Reproduction Assessment

Yes for the PR defect via source inspection: current main already performs one clone-heavy snapshot read to create `runtimeConfig`, and this PR still performs one. I did not run live timing in this read-only review.

## Solution Assessment

No. The updated approach fixes the earlier stale-getter concern, but it is not the narrowest maintainable latency fix because it leaves the same construction-time secrets snapshot clone in place.
