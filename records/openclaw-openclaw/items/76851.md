## Summary

Keep open: the PR is aimed at the right regression, but the branch still leaves the expensive full secrets snapshot clone reachable through the live plugin-tool config getter during tool registration.

## What This Changes

The PR adds a changelog entry and changes plugin tool resolution to seed the initial `runtimeConfig` from config-only snapshots while leaving `getRuntimeConfig` backed by the active secrets snapshot.

## Best Possible Solution

Route both the initial `runtimeConfig` and the live `getRuntimeConfig` callback through config-only runtime snapshots or an equivalent non-cloning live config helper, preserving refresh semantics and adding focused regression coverage.

## Reproduction Assessment

Yes. The related issue gives concrete gateway trace steps, and source inspection shows the PR still leaves the full secrets snapshot clone reachable from memory tool registration during plugin-tool prep.

## Solution Assessment

No. The initial config-only snapshot direction is right, but the live getter must also avoid `getActiveSecretsRuntimeSnapshot()` or the hot path can still pay the deep clone.
