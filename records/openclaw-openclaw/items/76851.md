## Summary

This PR should stay open because the optimization target is valid, but the current patch changes the live runtime-config getter into a stale closure; the existing ClawSweeper comment correctly identified the same blocker after the force-push.

## What This Changes

The PR caches the active secrets runtime snapshot in `src/agents/openclaw-plugin-tools.ts` and adds a changelog entry for the linked `core-plugin-tools` latency regression.

## Best Possible Solution

Avoid the expensive secrets snapshot clone on the hot construction path while keeping `ctx.getRuntimeConfig()` live against the current runtime snapshot, then validate the linked tests and timing before merge.

## Reproduction Assessment

Yes for the PR-introduced regression: the current integration test captures `ctx.getRuntimeConfig`, refreshes the active secrets runtime snapshot, and expects the getter to observe the refreshed config. The original 8.3s latency is source-plausible from clone-heavy snapshot reads, but I did not run live timing in this read-only review.

## Solution Assessment

No. Caching one snapshot for both eager construction-time config and the long-lived getter is not the narrowest maintainable fix because it breaks the documented refresh contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P2] Keep getRuntimeConfig live after runtime refresh:** `src/agents/openclaw-plugin-tools.ts:49`
  - body: Moving `getActiveSecretsRuntimeSnapshot()` out of `resolveCurrentRuntimeConfig` makes `ctx.getRuntimeConfig()` close over the snapshot from tool construction. Long-lived plugin tools currently rely on that getter to observe later `activateSecretsRuntimeSnapshot()` refreshes, so this patch can leave config or credential updates invisible until tools are rebuilt.
  - confidence: 0.96
