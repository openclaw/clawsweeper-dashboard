## Summary

Keep open: this PR addresses a real current-main Google Chat thread-routing bug and has sufficient live proof, but the patch still has a cross-target thread substitution bug that should block merge until fixed.

## What This Changes

The PR carries Google Chat inbound thread resource names through webhook, reply delivery, message-tool threading, outbound send validation, tests, and the changelog.

## Best Possible Solution

Keep the plugin-boundary resource normalization, but require same-target validation before substituting the inbound Google Chat thread in both the action handler and threading adapter.

## Reproduction Assessment

Yes. Source inspection shows current main loses the generic MessageThreadId and sends raw reply ids as Google Chat thread names; the supplied live before/after screenshots show the observable thread-placement failure and improvement.

## Solution Assessment

No. The approach is close and correctly keeps Google Chat resource handling inside the plugin boundary, but it is not merge-ready until current-message-to-thread substitution is also gated by the outbound target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Guard current-message substitution by target:** `extensions/googlechat/src/actions.ts:86-89`
  - body: When `replyTo` is the current Google Chat message resource, this returns `inboundThreadId` without checking that `to` still matches the inbound `currentChannelId`. A tool call that sends to `spaces/BBB` while replying to the current `spaces/AAA/messages/...` will pass the `spaces/AAA/threads/...` resource into a `spaces/BBB/messages` create request, which can fail or land on the wrong thread instead of sending unthreaded. Add the same same-target check used for omitted reply targets before substituting the inbound thread.
  - confidence: 0.88
- **[P2] Check target before adapter auto-thread substitution:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` has the same gap for explicit current-message `replyToId` values: it returns the inbound thread without verifying that the outbound `to` is the current Google Chat target. The shared message-action runner can then inject a cross-space `threadId` before dispatch, bypassing the different-target protection that only runs when `replyToId` is absent.
  - confidence: 0.86
