## Summary

Keep open. This PR addresses a real current-main Google Chat threading bug and has sufficient live proof, but the latest head still has two cross-target current-message substitution defects that should block merge.

## What This Changes

The PR preserves Google Chat inbound thread resource names through reply delivery and message-tool sends, validates thread.name inputs, adjusts Google auth transport selection, and adds Google Chat/core regression coverage plus a changelog entry.

## Best Possible Solution

Keep the plugin-boundary resource normalization, add same-target guards plus focused regression tests for explicit current-message reply targets, then land this PR.

## Reproduction Assessment

Yes. Current-main source inspection shows the raw reply/thread resource mismatch, and the inspected before/after Google Chat screenshots show the live failure and after-fix improvement.

## Solution Assessment

No. The overall plugin-boundary approach is maintainable, but the latest head is not the best complete fix until current-message-to-thread substitution is gated to the same Google Chat target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate current-message thread substitution by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: When `replyTo` is the current Google Chat message resource, this returns the inbound thread without checking that `to` still matches the inbound `currentChannelId`. A tool call sending to another space can pass the original space's thread into the new space's create request, causing a failed send or wrong-thread delivery instead of an unthreaded send.
  - confidence: 0.88
- **[P2] Check target before adapter thread fallback:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` applies the different-target guard only when `replyToId` is absent. If the shared runner passes an explicit current-message reply target while sending to another Google Chat space, this still returns the inbound thread and bypasses the intended different-target protection.
  - confidence: 0.87
