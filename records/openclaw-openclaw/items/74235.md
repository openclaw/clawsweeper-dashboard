## Summary

Keep open; this PR addresses a real current-main Google Chat threading gap, but current-message remaps still need a same-target guard before merge.

## What This Changes

The PR preserves Google Chat thread resource names through ingress, reply delivery, message-tool threading, API validation, auth transport selection, tests, and a changelog entry.

## Best Possible Solution

Keep the plugin-boundary thread-resource fix, add normalized same-target guards before remapping current message resources, and land with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main lets Google Chat message-shaped reply targets flow into thread.name, and the inspected before/after screenshots show the observable top-level escape and after-fix in-thread behavior.

## Solution Assessment

No, not as currently written. The plugin-boundary design is the right direction, but current-message remaps need a normalized same-target check in both message-tool and direct action paths.
