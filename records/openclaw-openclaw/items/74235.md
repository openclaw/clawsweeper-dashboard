## Summary

Keep this PR open: it addresses a real Google Chat threading gap, but the latest head still has two target-guard bugs where an explicit current-message reply can carry the inbound space's thread into a different outbound space.

## What This Changes

The PR preserves Google Chat inbound thread resource names through reply delivery and message-tool paths, validates outbound thread resources, updates Google auth gaxios selection, adds regression coverage, and adds a changelog entry.

## Best Possible Solution

Add same-target guards for explicit current-message remaps in both Google Chat resolver paths, with focused regression tests, while keeping resource-name conversion inside the Google Chat plugin boundary.

## Reproduction Assessment

Yes. Current main's Google Chat ingress lacks MessageThreadId and current reply delivery passes raw replyToId into thread.name; the PR body also supplies live before/after Google Chat screenshots for the user-visible thread placement.

## Solution Assessment

No, not yet. The ownership boundary and resource-normalization direction are sound, but explicit current-message remaps need same-target guards before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Check the target before remapping current replyTo:** `extensions/googlechat/src/actions.ts:85-89`
  - body: When explicit `threadId` or `replyTo` is a Google Chat message resource equal to the current inbound message, this branch returns the inbound thread without confirming `to` is still the current space. A tool send to `spaces/BBB` during a `spaces/AAA` turn can pass the AAA thread into a BBB request instead of sending unthreaded or rejecting the unsupported cross-target reply.
  - confidence: 0.91
- **[P2] Gate explicit auto-thread remaps by target:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` only checks the target when `replyToId` is absent. If the message tool supplies the current inbound message as an explicit `replyToId` while targeting another space, this still returns the source space's inbound thread and can misroute or fail the Google Chat send.
  - confidence: 0.9
