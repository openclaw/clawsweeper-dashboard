## Summary

Keep this draft PR open. Current main still has the Google Chat threading gaps the PR targets, but the latest branch drops the shared `hasRepliedRef` from Google Chat tool context, so `replyToMode: "first"` can thread every same-target `message` tool send in a turn; it also still needs the required changelog entry and the author’s pending live-validation hold resolved.

## What This Changes

The PR preserves Google Chat thread resource metadata through webhook context, reply delivery, outbound adapters, and message-tool sends, adds thread-resource validation, and expands core plus Google Chat regression tests.

## Best Possible Solution

Keep the PR open and fix the Google Chat tool-context regression by carrying `hasRepliedRef` through the plugin `buildToolContext`, add a regression for two same-turn `message` tool sends under `replyToMode: "first"`, add the single-line changelog entry, then complete the isolated Google Chat runtime validation before maintainer review and landing. After merge, reconcile #69422, #42510, #67055, and the already-closed #39554 against the shipped behavior.

## Reproduction Assessment

Yes. The source-level reproduction for the new PR bug is clear: with `replyToMode: "first"`, Google Chat `buildToolContext` returns no `hasRepliedRef`, so `resolveAndApplyOutboundReplyToId` cannot mark the first message-tool reply as consumed and later same-target sends can still be mapped to the inbound thread.

## Solution Assessment

No, not as currently written. The overall direction is the right boundary for the original Google Chat bug, but the plugin-owned context needs to preserve the shared first-reply state and the PR needs the changelog plus draft validation before it is merge-ready.
