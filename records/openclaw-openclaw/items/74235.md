## Summary

Keep this PR open: it fixes a real current-main Google Chat threading gap and includes credible real-behavior proof, but the latest head still has a same-target guard bug that can route a current-message remap to the inbound thread while sending to a different Google Chat space.

## What This Changes

The PR preserves Google Chat inbound thread resources through reply delivery and message-tool paths, validates outbound thread resource names, adjusts auth transport selection, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the plugin-boundary resource-name fix, add same-target checks before mapping current-message reply targets to the inbound thread in both Google Chat action paths, then land after focused regression tests and CI.

## Reproduction Assessment

Yes. Current-main source shows message-shaped Google Chat reply targets can reach thread.name, and the inspected before/after Google Chat screenshots show the observed top-level escape and after-fix in-thread behavior.

## Solution Assessment

No, not yet. The overall plugin-boundary approach is the right maintainable direction, but same-target checks are still missing for current-message remaps in the shared message-tool resolver and direct action resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate current-message auto threads by target:** `extensions/googlechat/src/channel.adapters.ts:264`
  - body: When `replyToId` equals the current inbound Google Chat message, this returns `currentThreadTs` without checking that `to` is the current space. A tool send from `spaces/AAA` to `spaces/BBB` with `replyTo` set to the current `spaces/AAA/messages/...` value will pass the `spaces/AAA/threads/...` thread into the `spaces/BBB` request; require the normalized target to match before returning the inbound thread.
  - confidence: 0.88
- **[P2] Gate direct action remaps by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: The direct Google Chat action path has the same issue: an explicit message-shaped `replyTo` equal to `currentMessageId` returns the inbound thread before confirming the send target is the current space. Add the same normalized target guard here so direct plugin dispatch cannot inject an inbound thread into a different `to` space.
  - confidence: 0.86
