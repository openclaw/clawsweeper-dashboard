## Summary

Keep this draft PR open. It targets a real current-main Google Chat threading gap, but the latest patch still appears to regress explicit media sends that carry a valid thread resource through `threadId`, and the author has explicitly paused the PR for live gateway validation.

## What This Changes

The PR preserves Google Chat thread resource names through webhook context, reply delivery, routed outbound sends, and message-tool sends, adds Google Chat thread resource validation, and expands core plus Google Chat regression tests.

## Best Possible Solution

Revise this PR so Google Chat resource-name normalization stays inside the plugin boundary, valid explicit thread resources are preserved consistently for text and media paths, and the draft live-validation hold is resolved with isolated gateway proof before landing. After merge, reconcile #39554, #42510, #67055, and #69422 against the shipped behavior.

Yes. The current-main bug is source-reproducible from the missing `MessageThreadId` plus raw `payload.replyToId` delivery, and the PR finding is reproducible by calling direct Google Chat outbound media delivery with `threadId: "spaces/.../threads/..."` and no `replyToId`.

## Solution Assessment

No, not as currently patched. The overall direction is maintainable, but the media path should preserve valid explicit thread targets the same way the text path does, and the PR should leave draft only after the reported live-runtime concern is isolated or cleared.
