## Best Possible Solution

Keep the plugin-boundary resource-name normalization, add explicit top-level/thread-null suppressor handling to the Google Chat action fallback, and land after focused message-tool threading coverage passes.

Yes. Current main is source-reproducible through Google Chat ingress and delivery passing message/thread values without a separate MessageThreadId, and the PR regression is source-reproducible with same-target message.send using topLevel:true or threadId:null under replyToMode first/all.
