## Summary

Keep this PR open: current main still has the Google Chat thread-routing gap and this branch is a focused fix, but the latest head still has a blocking same-target bug in explicit current-message remaps for message-tool sends.

## What This Changes

The PR adds Google Chat thread-resource validation and carries inbound `message.thread.name` through reply, block-streaming, and message-tool delivery with regression tests and a changelog entry.

## Best Possible Solution

Keep the Google Chat resource-name conversion at the plugin boundary, but require the outbound target to match the inbound Google Chat space before converting a current-message resource into the inbound thread.

## Reproduction Assessment

Yes. Source inspection gives a concrete path: a Google Chat message-tool send to another space with `replyTo` equal to the current inbound message can still receive the original inbound thread before delivery.

## Solution Assessment

No, not yet. The PR's overall boundary is maintainable, but both resolver paths need the same-target guard before this is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate explicit current-message remaps by target:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: When `replyToId` equals the current inbound Google Chat message, this resolver returns `currentThreadTs` without checking that `to` still matches the current space. A tool send to `spaces/BBB` can therefore inherit `spaces/AAA`'s inbound thread and pass the wrong `thread.name` to Google Chat. Add the normalized target check to this remap and cover the different-target case.
  - confidence: 0.91
- **[P2] Gate direct action remaps by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: The direct Google Chat action resolver has the same gap: an explicit message-shaped `replyTo` equal to `currentMessageId` maps to the inbound thread before checking `to`. A direct send to another space can carry the source space's thread, so this branch should require the normalized target to match the current channel before returning `inboundThreadId`.
  - confidence: 0.89
