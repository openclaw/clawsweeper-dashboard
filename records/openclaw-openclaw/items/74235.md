## Summary

Keep open: this PR addresses a real current-main Google Chat threading bug and has supplied live screenshot proof, but the latest head still lets explicit current-message reply targets inherit the inbound thread when sending to a different Google Chat space.

## What This Changes

The PR preserves Google Chat thread resource names through inbound, reply, and message-tool delivery, validates outbound thread names, adjusts Google Chat auth gaxios transport selection, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land this PR after adding same-target guards and regression tests for both Google Chat current-message-to-thread substitution paths while keeping resource normalization inside the Google Chat plugin boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows Google Chat can pass message-shaped reply ids as thread.name, and the PR body’s inspected before/after screenshots show the live thread-placement failure and improvement.

## Solution Assessment

No, not yet. The plugin-boundary approach is the right fix shape, but the current PR head must gate current-message-to-thread substitution by outbound target in both direct and shared message-tool paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate current-message substitution by target:** `extensions/googlechat/src/actions.ts:85-88`
  - body: When `replyTo` is the current Google Chat message resource, this branch returns the inbound thread without checking that `to` still matches the inbound `currentChannelId`. A tool call sending to another Google Chat space can therefore pass the original space's thread into the new space's send request, which can fail or land in the wrong thread; require the same-target check before returning `inboundThreadId`.
  - confidence: 0.9
- **[P2] Check target before adapter thread substitution:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` only checks same-target when `replyToId` is absent. If the shared runner supplies an explicit current-message `replyToId` while sending to a different Google Chat space, this returns the inbound thread and injects it into the cross-space send; gate the explicit current-message branch on the outbound target too.
  - confidence: 0.89
