## Summary

Keep this PR open: it addresses a real current-main Google Chat threading gap and has sufficient live screenshot proof, but the current head still remaps explicit current-message reply targets to the inbound thread without checking that the outbound `to` target is the same Google Chat space.

## What This Changes

The PR preserves Google Chat thread resource names across webhook ingress, reply delivery, and message-tool sends so assistant replies stay in the inbound Chat thread.

## Best Possible Solution

Add same-target guards and focused regression tests for explicit current-message remaps, while keeping Google Chat resource-name normalization inside the Google Chat plugin boundary.

## Reproduction Assessment

Yes. Current main has a source-reproducible path where Google Chat reply metadata can flow into `thread.name` without preserving the inbound thread resource, and the PR body supplies live before/after Google Chat screenshots for the user-visible thread placement.

## Solution Assessment

No, not yet. The ownership boundary and thread-resource direction are right, but the current patch still needs same-target checks before remapping a current message resource to the inbound thread.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Check the target before remapping current replies:** `extensions/googlechat/src/actions.ts:85-88`
  - body: When `replyTo` is the current Google Chat message resource, this branch returns the inbound thread without confirming `to` is still the current space. A tool send to another space during the same turn can pass the source space's thread into the other request instead of sending unthreaded or rejecting the unsupported cross-target reply.
  - confidence: 0.9
- **[P2] Require same target for explicit auto-thread remaps:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` checks the target only when `replyToId` is absent. If the message tool supplies the current inbound message as an explicit `replyToId` while targeting a different space, this still returns the source space's inbound thread and can misroute or fail the Google Chat send.
  - confidence: 0.9
