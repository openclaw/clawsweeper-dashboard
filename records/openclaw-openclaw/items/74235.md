## Summary

Keep open: the PR addresses a real Google Chat threading bug and has useful proof, but the current head can still remap an inbound thread into a different outbound Google Chat space when the tool supplies the current inbound message as an explicit reply target.

## What This Changes

The PR preserves Google Chat thread resource names through inbound context, reply delivery, message-tool threading, outbound validation, auth transport selection, tests, and changelog.

## Best Possible Solution

Land the PR after adding normalized same-target guards for explicit current-message remaps in both Google Chat thread resolvers, with focused regression coverage for different-target sends.

## Reproduction Assessment

Yes. Current main's missing thread-resource plumbing is source-reproducible, and PR-head source inspection gives a concrete same-turn, different-target message-tool path that still misroutes the inbound thread.

## Solution Assessment

No, not yet. Keeping Google Chat resource conversion inside the plugin boundary is the right direction, but explicit current-message remaps need the same normalized target guard already used for omitted reply targets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Gate current-message remaps by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: When `replyTo` equals the current inbound Google Chat message, this branch returns the inbound thread without checking that `to` still targets the current space. A send to `spaces/BBB` during an inbound `spaces/AAA` turn can pass the `spaces/AAA` thread into the BBB request instead of sending unthreaded or rejecting the unsupported cross-target reply.
  - confidence: 0.91
- **[P2] Guard auto-thread remaps for explicit replies:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` only checks the current target when `replyToId` is absent. The shared message-action path can supply the current inbound message as an explicit `replyToId`, so a different-space send can still receive the source space's inbound thread; add the same normalized target guard before returning `inboundThread`.
  - confidence: 0.9
