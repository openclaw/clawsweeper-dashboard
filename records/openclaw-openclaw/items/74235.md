## Summary

Keep this PR open: it addresses a real current-main Google Chat threading bug and has usable live screenshot proof, but one P2 correctness issue remains in the current-message-to-thread remap for cross-target sends.

## What This Changes

The PR adds Google Chat thread-resource validation and inbound-thread plumbing across webhook context, reply delivery, message-tool routing, auth transport selection, regression tests, and the changelog.

## Best Possible Solution

Keep the Google Chat boundary-based approach, add same-target validation before mapping current message resources to inbound threads, cover that case in Google Chat action/threading tests, then land after exact-head checks pass.

## Reproduction Assessment

Yes. Current-main source shows message-shaped Google Chat reply targets can flow into thread.name, and the inspected before/after screenshots show the observable top-level escape and after-fix in-thread behavior; I did not rerun a live current-main Google Chat environment.

## Solution Assessment

No, not as currently written. The owner-boundary design is the right direction, but the current-message remap needs the same same-target guard used for implicit tool threading before this is the narrow maintainable fix.
