## Summary

Keep open: this is a valid Google Chat threading fix with sufficient live proof, but the current head still has a blocking edge case where a current-message remap can inject the inbound thread into a different outbound space.

## What This Changes

The PR preserves Google Chat inbound thread resource names through reply delivery, message-tool routing, outbound thread validation, auth transport selection, tests, and a changelog entry.

## Best Possible Solution

Keep the plugin-boundary resource-name fix, add a normalized same-target guard for current-message remaps in both message-tool and direct action paths, then land after focused tests and CI.

## Reproduction Assessment

Yes. Current-main source shows message-shaped Google Chat reply targets can flow into thread.name, and the inspected before/after screenshots show the observable top-level escape and after-fix in-thread behavior.

## Solution Assessment

No, not as currently written. The plugin-boundary design is the right direction, but current-message remaps need a normalized same-target check before reusing the inbound thread.
