## Summary

Keep open: this targets a real current-main Google Chat threading bug and includes live proof, but the current PR head still has two P2 correctness blockers in thread-target normalization and first-reply state handling.

## What This Changes

The PR carries Google Chat thread resource names through inbound context, reply delivery, message-tool routing, shared reply plumbing, tests, and the changelog.

## Best Possible Solution

Revise this PR so only the known current-message fallback maps to the inbound thread, Google Chat tool context preserves hasRepliedRef, and the focused regression tests cover both edge cases before merge.

## Reproduction Assessment

Yes, source-level: current main forwards raw Google Chat reply targets, and the PR head has clear paths for an explicit message-resource reply target and repeated same-turn message-tool sends with replyToMode="first". I did not run a live Google Chat webhook in this read-only review.

## Solution Assessment

No, not yet. Keeping Google Chat resource-name handling inside the plugin boundary is the right direction, but the current implementation must distinguish current-message fallback from arbitrary message resources and preserve first-reply state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Only translate the current Google Chat message fallback:** `extensions/googlechat/src/monitor-reply-delivery.ts:24-28`
  - body: This branch rewrites any Google Chat message-shaped replyToId to the inbound thread. Explicit reply tags can name a different message resource, so this can silently post into the current thread instead of rejecting the unsupported target or translating only the known current-message fallback.

  - confidence: 0.9
