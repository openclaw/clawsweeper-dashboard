## Summary

Keep open: this is an active, focused Google Chat bug-fix PR for a real current-main threading gap, with supplied live screenshot proof. The patch is close, but it still silently drops an explicit non-current Google Chat message-resource reply target into a top-level send, which should be fixed before merge.

## What This Changes

The PR carries Google Chat thread resource names through webhook context, reply delivery, message-tool sends, outbound adapter normalization, API validation, Google auth transport selection, tests, and changelog.

## Best Possible Solution

Land this PR after the explicit unsupported message-resource reply target path fails visibly instead of silently sending a top-level message, keeping Google Chat resource normalization inside the plugin boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows core can hand Google Chat delivery a message-shaped currentMessageId and the plugin passes payload.replyToId directly to thread.name; the PR also supplies inspected before/after Google Chat screenshots.

## Solution Assessment

Mostly yes. The overall boundary is right because Google Chat resource conversion stays in the plugin/API layer, but the PR should reject or fail explicit non-current message-resource reply targets rather than dropping them into top-level sends.
