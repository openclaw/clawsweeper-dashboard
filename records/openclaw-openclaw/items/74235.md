## Summary

Keep open: this is a valid Google Chat threading fix, but the current PR head still has two blocking same-target guard gaps that can route an inbound thread from one Google Chat space into a send targeting another space.

## What This Changes

The PR preserves Google Chat thread resource names through inbound context, reply delivery, and message-tool paths, validates outbound thread names, adjusts Google auth transport selection, adds regression tests, and updates the changelog.

## Best Possible Solution

Land this PR after adding normalized same-target guards for explicit current-message remaps in both Google Chat thread resolvers, with focused regression tests for different-target sends.

## Reproduction Assessment

Yes. Source inspection of PR head shows that a Google Chat tool send to spaces/BBB with replyTo equal to the current inbound message from spaces/AAA can still inherit the spaces/AAA inbound thread.

## Solution Assessment

No, not yet. Keeping Google Chat resource conversion inside the plugin boundary is the right direction, but explicit current-message remaps need the same normalized same-target guard used for omitted reply targets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Gate current-message remaps by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: When `replyTo` equals the current inbound Google Chat message, this branch returns the inbound thread before checking that `to` is still the current space. A send to `spaces/BBB` during an inbound `spaces/AAA` turn can pass the `spaces/AAA` thread into the BBB request instead of sending unthreaded or rejecting the unsupported cross-target reply.
  - confidence: 0.91
- **[P2] Guard auto-thread remaps for explicit replies:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` only checks the current target when `replyToId` is absent. The shared message-action path can supply the current inbound message as an explicit `replyToId`, so a different-space send can still receive the source space's inbound thread; add the same normalized target guard before returning `inboundThread`.
  - confidence: 0.9
