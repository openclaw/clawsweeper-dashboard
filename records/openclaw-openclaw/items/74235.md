## Summary

Keep this active PR open: it addresses a real current-main Google Chat threading bug and has sufficient live proof, but one target-scoping bug remains before merge.

## What This Changes

The PR preserves Google Chat inbound thread resource names through reply delivery and message-tool sends, adds thread-resource validation, adjusts Google auth transport selection, and expands Google Chat plus reply plumbing tests.

## Best Possible Solution

Land the PR after tightening the current-message-to-thread translation to the same Google Chat target/space and keeping the resource-shape logic inside the Google Chat plugin boundary.

## Reproduction Assessment

Yes, source-level: current main omits MessageThreadId for Google Chat and forwards payload.replyToId directly to thread.name. The PR discussion also supplies live Google Chat output showing the intended in-thread behavior after the fix.

## Solution Assessment

No, not yet: the overall approach is the right narrow fix, but the current-message reply target should only be converted to the inbound thread when the outbound Google Chat target matches the inbound space.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Gate current-message thread reuse by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: When `replyTo` is the current Google Chat message resource, this branch returns the inbound thread without checking that `to` is the same current Chat target. A tool call can send to another space while carrying the current message as `replyTo`, causing the send to use `spaces/AAA/threads/...` under a `spaces/BBB/messages` create call instead of treating that explicit reply target as unsupported. Please gate this mapping on the same Google Chat target/space in both this action path and the channel threading resolver.
