## Summary

Keep open. This is a real Google Chat threading bug with sufficient proof and green CI, but the latest PR head still lets an explicit current-message reply target carry the inbound thread into a different Google Chat space, so it needs a narrow fix before merge.

## What This Changes

The PR preserves Google Chat inbound thread resources through reply delivery and message-tool paths, validates thread resources before API calls, adjusts auth transport selection, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Keep the plugin-boundary thread normalization, add same-target guards for current-message-to-thread substitution, and then land after targeted Google Chat/message-action checks stay green.

## Reproduction Assessment

Yes, source-reproducible: current main forwards raw Google Chat reply IDs to thread.name, and the inspected before/after screenshots show the real Chat behavior. The new blocker is also source-reproducible by sending to another space with replyTo equal to the current inbound message resource.

## Solution Assessment

No, not yet. The plugin-boundary normalization is the right direction, but current-message-to-thread substitution must also verify the outbound Google Chat target before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate message-resource replies by target:** `extensions/googlechat/src/actions.ts:86-88`
  - body: When a tool send targets another Google Chat space during an inbound turn, an explicit `replyTo` equal to the current message resource takes this branch and returns the inbound thread before resolving the outbound space. That can send a `spaces/AAA/threads/...` thread to a `spaces/BBB/messages` request, causing a failed send or wrong conversation instead of an unthreaded send.
  - confidence: 0.9
- **[P2] Check target before auto-threading explicit replies:** `extensions/googlechat/src/channel.adapters.ts:252-264`
  - body: `resolveAutoThreadId` only checks the outbound target when `replyToId` is absent. The shared message-action runner can pass an explicit current-message `replyToId` for a send to another space, and this still returns the inbound thread, writing a foreign-space `threadId` into the action params before delivery.
  - confidence: 0.88
