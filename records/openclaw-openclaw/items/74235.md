## Summary

Keep this PR open. It addresses a real current-main Google Chat threading gap and has sufficient live screenshot proof, but one P2 correctness issue remains: current-message-to-inbound-thread remapping still lacks a same-target guard for explicit message-tool sends.

## What This Changes

The PR preserves Google Chat inbound thread metadata through reply/tool delivery, validates outbound thread resource names, adjusts Google Chat auth transport selection, and expands Google Chat/core reply regression coverage plus a changelog entry.

## Best Possible Solution

Keep the Google Chat boundary-based fix, add a same-target guard before any current-message-to-inbound-thread remap, cover that in action and adapter tests, then land after queued CI passes and reconcile https://github.com/openclaw/openclaw/issues/69422, https://github.com/openclaw/openclaw/issues/42510, and https://github.com/openclaw/openclaw/pull/67055.

## Reproduction Assessment

Yes. Current-main source shows message-shaped Google Chat reply targets can flow into thread.name, and the inspected before/after screenshots show the observable top-level escape and after-fix in-thread behavior.

## Solution Assessment

No, not as currently written. The plugin-boundary design is the right solution, but current-message remaps need a normalized same-target check before the inbound thread is reused.
