## Summary

Keep open: this is an active Google Chat bug-fix PR with useful proof and green CI, but the latest head still lets the inbound thread from one Google Chat space be substituted when an explicit current-message reply is sent to a different space.

## What This Changes

The PR carries Google Chat inbound thread resource names through webhook context, reply delivery, message-tool threading, outbound API validation, tests, auth transport fixture alignment, and a changelog entry.

## Best Possible Solution

Keep the plugin-boundary resource normalization and land this PR after adding same-target guards plus focused regression tests for explicit current-message reply targets sent to a different Google Chat space.

## Reproduction Assessment

Yes. Source inspection on current main shows Google Chat loses a generic MessageThreadId and sends raw replyToId values as thread.name; the inspected before/after screenshots show the live thread-placement failure and improvement.

## Solution Assessment

No. The proposed direction is maintainable and correctly keeps Google Chat resource handling inside the plugin, but the latest head is not merge-ready until current-message-to-thread substitution is gated by the outbound target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate current-message substitution by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: When `replyTo` is the current Google Chat message resource, this returns `inboundThreadId` without checking that `to` still matches the inbound `currentChannelId`. A tool call that sends to `spaces/BBB` while replying to the current `spaces/AAA/messages/...` will pass the `spaces/AAA/threads/...` resource into a `spaces/BBB/messages` create request, which can fail or land in the wrong thread instead of sending unthreaded. Add the same same-target guard used for omitted reply targets before substituting the inbound thread.
  - confidence: 0.88
- **[P2] Check target before adapter thread substitution:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` has the same gap for explicit current-message `replyToId` values: it returns the inbound thread without verifying that the outbound `to` is the current Google Chat target. The shared message-action runner can then inject a cross-space `threadId` before dispatch, bypassing the different-target protection that only runs when `replyToId` is absent.
  - confidence: 0.87
