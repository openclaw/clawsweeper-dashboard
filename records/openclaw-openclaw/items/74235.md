## Summary

Keep open: this PR addresses a real Google Chat thread-routing gap that is still present on current main, but the current PR head still has a cross-target threading bug in the message-tool path that should block merge.

## What This Changes

The PR preserves Google Chat thread resource names through inbound context, reply delivery, message-tool threading, outbound validation, auth transport selection, regression tests, and the changelog.

## Best Possible Solution

Keep the Google Chat resource conversion inside the plugin boundary, but land only after both explicit current-message remap paths require the outbound target to match the inbound Google Chat space and have focused regression coverage.

## Reproduction Assessment

Yes. Current-main source shows the original message/thread resource conflation, and PR-head source shows a concrete cross-target path where an explicit current-message reply can map to the inbound thread without a same-target check.

## Solution Assessment

No, not yet. The overall direction is the narrow maintainable fix, but the current PR must add same-target guards before converting a current message resource into the inbound Google Chat thread.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Gate explicit current-message remaps by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: When `replyTo` is a Google Chat message resource equal to the current inbound message, this branch returns `inboundThreadId` without checking that `to` is still the current Google Chat space. A tool send to `spaces/BBB` during an inbound `spaces/AAA` turn can pass the `spaces/AAA` thread into the BBB request instead of sending unthreaded or rejecting the unsupported cross-target reply.
  - confidence: 0.91
- **[P2] Check the target before returning the inbound thread:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` only applies the same-target check when `replyToId` is absent. The shared message-action path can supply the current inbound message as an explicit `replyToId`, so a different-space send can still inherit the source space's inbound thread; apply the target guard before returning `inboundThread`.
  - confidence: 0.9
