## Summary

Keep open: this PR fixes a real Google Chat threading gap on current main, but the latest diff still has narrow P2 wrong-thread regressions and drops first-reply state for Google Chat message-tool sends.

## What This Changes

The PR carries Google Chat thread metadata through inbound context, shared reply plumbing, message-tool delivery, outbound validation, auth transport, tests, and the changelog.

## Best Possible Solution

Revise this PR so Google Chat converts only the known current-message fallback to the validated inbound thread, rejects or drops unsupported explicit message resources before delivery, preserves hasRepliedRef in tool context, and keeps the normalization inside the Google Chat plugin boundary.

## Reproduction Assessment

Yes, source-level: current main lacks a separate MessageThreadId at Google Chat ingress and forwards raw replyToId values into thread.name. I did not run a live webhook in this read-only review, but the failing path and PR regression targets are clear.

## Solution Assessment

No, not yet. The overall direction is the narrow maintainable fix, but the implementation must stop arbitrary message-resource rewrites and preserve first-reply state before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject non-current message resources in reply delivery:** `extensions/googlechat/src/monitor-reply-delivery.ts:28`
  - body: `resolveOutboundThread` converts any Google Chat message-shaped `replyToId` to the inbound thread. An explicit reply tag can name a different message resource, so this silently posts to the current thread instead of rejecting or only translating the known current-message fallback.
  - confidence: 0.9
- **[P2] Reject non-current message targets in actions:** `extensions/googlechat/src/actions.ts:75`
  - body: When the message tool supplies a message-shaped `replyTo`, this returns the inbound thread without checking the current inbound message or the target space. A send to another Google Chat space can receive the current space's thread and land in the wrong place or fail at the API.
  - confidence: 0.88
- **[P2] Preserve hasRepliedRef in Google Chat tool context:** `extensions/googlechat/src/channel.adapters.ts:208-211`
  - body: `replyToMode: "first"` relies on `toolContext.hasRepliedRef` to stop inherited replies after the first message-tool send. This `buildToolContext` drops that ref, so repeated Google Chat sends in one turn can all inherit the inbound thread.
  - confidence: 0.9
