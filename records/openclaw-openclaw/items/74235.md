## Summary

Keep open: current main still has the Google Chat thread-routing gap and this PR addresses the right area, but the patch needs a focused fix for `replyToMode: "first"` message-tool auto-threading before it is merge-ready.

## What This Changes

The branch preserves Google Chat inbound thread resource names, normalizes outbound thread targets in reply and message-action paths, adds shared resource-name validators, expands regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the Google Chat resource-name logic at the plugin boundary, fix the `replyToMode: "first"` auto-threading regression, refresh exact-head validation, and then land or request narrow revisions.

## Reproduction Assessment

Yes, source-level. Current main can pass Google Chat message resources or missing inbound-thread context into outbound sends, and the new first-mode issue is traceable through the shared message-action threading helper plus the PR's Google Chat resolver.

## Solution Assessment

No, not as currently written. The overall boundary-focused approach is maintainable, but `replyToMode: "first"` must stop auto-threading later same-target message-tool sends before this patch is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor first-mode consumption before auto-threading:** `extensions/googlechat/src/channel.adapters.ts:205-218`
  - body: The shared message-action runner calls `resolveAutoThreadId` after `replyToMode: "first"` has already consumed the first reply and left later sends without `replyToId`. This resolver still returns `currentThreadTs` for any same-target send with no `replyToId`, so every subsequent Google Chat `message` tool send in the turn stays threaded instead of only the first one. Carry/check `hasRepliedRef` or otherwise make the auto-thread fallback share the same first-slot decision before returning the inbound thread.
  - confidence: 0.87
