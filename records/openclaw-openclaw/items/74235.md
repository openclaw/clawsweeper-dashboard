## Summary

Keep open: this PR addresses a real current-main Google Chat threading gap and has sufficient live-output proof, but the current head still has two blocking threading defects around arbitrary message-resource remapping and `replyToMode: "first"` state.

## What This Changes

The PR carries Google Chat inbound thread resource names through reply and message-tool delivery, validates outbound thread names, adds regression coverage, and updates the changelog.

## Best Possible Solution

Revise this PR so Google Chat only maps the known current-message fallback to the captured thread, carries first-mode reply state through tool context, and then land it with the existing regression and live proof updated if needed.

## Reproduction Assessment

Yes, source-level. Current main shows Google Chat reply delivery forwarding raw `replyToId` as `thread`, while the PR head shows the two remaining edge cases in the changed code paths; I did not run a live failing Google Chat webhook in this read-only review.

## Solution Assessment

No, not yet. The overall boundary is right because Google Chat resource-name handling stays inside the plugin, but the current patch needs two narrow corrections before it is the safest maintainable fix.

Overall correctness: patch is incorrect

- **[P2] Avoid rewriting unrelated message resources to the inbound thread:** `extensions/googlechat/src/monitor-reply-delivery.ts:24-28`
  - body: This converts every Google Chat message-shaped `replyToId` to the inbound thread. Explicit tags can supply a different message resource, so the patch can silently post into the current inbound thread instead of rejecting the unsupported target or only translating the known current-message fallback.

  - confidence: 0.9
