## Summary

Keep open: this PR fixes a real current-main Google Chat threading gap, but the current head still has two P2 cross-target remap bugs where an explicit current-message reply target can inject the inbound thread into a different Google Chat space.

## What This Changes

The PR carries Google Chat inbound thread resources through reply delivery and message-tool paths, validates Google Chat thread resource names, adjusts Google Chat auth transport selection, and adds regression tests plus a changelog entry.

## Best Possible Solution

Keep the Google Chat resource-name normalization at the plugin boundary, add same-target guards before remapping current message resources, and land with regression tests for direct and shared message-tool cross-target sends.

## Reproduction Assessment

Yes, source-level. Current main lacks the separate Google Chat thread fact, and the PR-head blocker follows a concrete explicit-reply path through the shared message-action runner into the Google Chat remap logic.

## Solution Assessment

No, not yet. The overall plugin-boundary approach is maintainable, but both resolver paths need the same normalized-target guard before converting a current message resource into the inbound thread.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate shared remaps by target:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: When `replyToId` is explicit and equals `currentMessageId`, this returns `currentThreadTs` without checking that `to` still matches `currentChannelId`. The shared message-action path preserves explicit `replyTo`, so a send to `spaces/BBB` during a `spaces/AAA` turn can receive `spaces/AAA/threads/...` as `thread.name`. Require the normalized target match before this remap.
  - confidence: 0.9
- **[P2] Gate direct action remaps by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: The direct Google Chat action resolver has the same cross-target gap: an explicit current-message `replyTo` maps to `inboundThreadId` before the target check used for implicit fallback. Add the same normalized `to` versus current-channel check here so direct dispatch cannot attach an inbound thread to another space.
  - confidence: 0.88
