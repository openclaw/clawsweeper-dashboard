## Summary

Keep open: the current-main Google Chat threading gap is real and the PR is aimed at the right boundary, but the latest head still has source-level P2 blockers that can misroute explicit message-resource replies and break replyToMode="first" message-tool semantics.

## What This Changes

The PR carries Google Chat thread-resource metadata through webhook context, auto-reply/block delivery, message-tool routing, send validation, auth transport, regression tests, and the changelog.

## Best Possible Solution

Revise this PR so Google Chat only converts the known current-message fallback to the validated inbound thread, preserves hasRepliedRef in tool context, and adds focused regressions for explicit message-resource targets plus repeated first-mode message-tool sends.

## Reproduction Assessment

Yes, source-level: current main lacks MessageThreadId at Google Chat ingress and forwards raw replyToId values into Google Chat thread.name. I did not run a live Google Chat webhook in this read-only review, but the current-main path and PR regression targets are clear.

## Solution Assessment

No, not yet. Keeping Google Chat resource-name normalization inside the plugin boundary is the right direction, but the implementation must avoid arbitrary message-resource rewrites and must preserve shared first-reply state before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject arbitrary message-resource reply targets in delivery:** `extensions/googlechat/src/monitor-reply-delivery.ts:22-27`
  - body: This branch maps any Google Chat message-shaped replyToId to the inbound thread. An explicit reply tag can name a different or unsupported message resource, so this can silently post to the current inbound thread instead of rejecting it or translating only the known current-message fallback.
  - confidence: 0.88
- **[P2] Reject arbitrary message-resource reply targets in actions:** `extensions/googlechat/src/actions.ts:80-82`
  - body: The message-action path applies the same rewrite to explicit threadId/replyTo values. If an agent supplies a Google Chat message resource explicitly, the send is redirected to the inbound thread, which can put the message in the wrong conversation thread.
  - confidence: 0.86
- **[P2] Preserve first-mode reply state in tool context:** `extensions/googlechat/src/channel.adapters.ts:197-215`
  - body: replyToMode="first" relies on toolContext.hasRepliedRef to stop inherited replies after the first message-tool send. This buildToolContext returns Google Chat fields without carrying the hasRepliedRef argument, so repeated sends in one turn can keep inheriting the inbound thread.
  - confidence: 0.9
