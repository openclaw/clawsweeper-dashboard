## Summary

Keep this PR open: it addresses a real current-main Google Chat threading gap and has sufficient real behavior proof, but the current head still has a blocking target-safety bug in two resolver paths.

## What This Changes

The branch carries Google Chat inbound thread resources through reply and message-tool paths, validates outbound Google Chat thread resource names, adjusts Google auth gaxios selection, and expands tests plus changelog coverage.

## Best Possible Solution

Keep the Google Chat resource-name handling at the plugin boundary, add same-target checks for current-message remaps, then land after targeted Google Chat/outbound threading tests and normal CI.

## Reproduction Assessment

Yes. Source inspection shows the current-main bug path, and the current PR head still has a concrete same-target reproduction path where a send to spaces/BBB with replyTo set to the current spaces/AAA message can inherit spaces/AAA's inbound thread.

## Solution Assessment

No, not yet. The overall plugin-boundary normalization is the right shape, but the two current-message remap paths need the same target guard before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate current-message remaps by target:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: When `replyToId` equals the current inbound Google Chat message, this returns `inboundThread` without checking that `to` still matches the inbound space. A message-tool send to another space can therefore receive the original space's thread target instead of staying top-level or rejecting the unsupported reply target.
  - confidence: 0.9
- **[P2] Gate direct action remaps by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: The direct Google Chat action resolver has the same issue: an explicit message-shaped `replyTo` equal to `currentMessageId` maps to `inboundThreadId` before checking the outbound `to` target. Add the same normalized target check here so direct dispatch cannot inject the inbound thread into a different space.
  - confidence: 0.88
