## Summary

Keep open: this PR addresses a real Google Chat threading gap on current main, but the latest head still lets explicit current-message reply targets remap to the inbound thread without proving the outbound Google Chat space is the same target.

## What This Changes

The PR carries Google Chat inbound thread resource names through reply delivery and message-tool paths, validates outbound thread resource names, updates Google auth transport selection, adds regression tests, and records a changelog entry.

## Best Possible Solution

Keep the Google Chat resource-name conversion at the plugin boundary, but require the normalized outbound target to match the inbound Google Chat space before converting a current-message resource into the inbound thread.

## Reproduction Assessment

Yes. Source inspection gives a concrete path: a Google Chat message-tool send to `spaces/BBB` with `replyTo` equal to the current inbound `spaces/AAA/messages/...` can still receive the `spaces/AAA/threads/...` inbound thread before delivery.

## Solution Assessment

No, not yet. The PR's core direction is maintainable, but both explicit current-message remap paths need the same-target guard and regression coverage before this is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate explicit current-message remaps by target:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: When `replyToId` equals the current inbound Google Chat message, this resolver returns `currentThreadTs` without checking that `to` still matches the current space. A tool send to another space can therefore inherit the source space's thread and pass the wrong `thread.name` to Google Chat; require the normalized target check for this remap too.
  - confidence: 0.91

A direct send to another space can carry the source space's thread, so this branch should require the normalized target to match before returning `inboundThreadId`.
  - confidence: 0.89
