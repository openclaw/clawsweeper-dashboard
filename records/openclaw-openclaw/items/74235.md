## Summary

Keep open: this PR addresses a real Google Chat threading bug and has sufficient live screenshot proof, but the latest head still has two same-target guard gaps that can carry the source thread into a different Google Chat space.

## What This Changes

The PR preserves Google Chat inbound thread resource names through reply delivery and message-tool paths, validates thread resources, adjusts shared reply plumbing, and adds Google Chat/core regression coverage.

## Best Possible Solution

Add same-target guards and focused regression tests for explicit current-message remaps, keeping Google Chat resource-name normalization inside the Google Chat plugin boundary before landing this PR.

## Reproduction Assessment

Yes. Current main has a source-reproducible path where Google Chat message/thread resource names can flow through reply delivery without a preserved inbound thread, and the PR body supplies live before/after Google Chat screenshots for the visible thread-placement behavior.

## Solution Assessment

No, not yet. The owner boundary and thread-resource validation are the right direction, but explicit current-message remaps still need same-target checks in both the direct action handler and shared message-tool threading adapter.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate current-message remaps by target:** `extensions/googlechat/src/actions.ts:85-89`
  - body: When `replyTo` equals the current inbound Google Chat message resource, this branch returns the inbound thread without checking that `to` still matches the current space. A tool send to another space during the same turn can carry the source space's thread into the other request instead of sending unthreaded or rejecting the unsupported cross-target reply.
  - confidence: 0.9
- **[P2] Gate auto-thread remaps by target:** `extensions/googlechat/src/channel.adapters.ts:258-264`
  - body: `resolveAutoThreadId` only checks the target when `replyToId` is absent. If the message tool supplies the current inbound message as an explicit `replyToId` while targeting a different space, this still returns the source space's inbound thread and can misroute or fail the Google Chat send.
  - confidence: 0.9
