- `impact:message-loss`: Discord edit/delete gateway events are currently not delivered into the inbound agent workflow.
- `impact:session-state`: Corrected or deleted source messages can leave an active run proceeding from stale original content.

## Summary

Keep open. Current main still lacks Discord `MESSAGE_UPDATE`/`MESSAGE_DELETE` listener registration, gateway mapping/hydration, edit-aware replay semantics, and a source Discord message-id cancellation map; related trackers overlap but do not fully supersede this Discord edit-plus-delete request.

## What This Changes

Add Discord message edit handling to reprocess corrected commands and message delete handling to cancel the active run started by the deleted source message.

## Best Possible Solution

Define the maintainer-approved Discord edit/delete policy, then implement it in the Discord plugin with typed update/delete listeners, gateway hydration, edit-aware replay, precise source-message cancellation, docs/config alignment, and focused tests.

## Reproduction Assessment

Yes, at source level. Current main has no registered Discord `MESSAGE_UPDATE` or `MESSAGE_DELETE` listener path, same-message edits would collide with replay dedupe keyed by account/channel/message id, and delete events have no source-message cancellation map.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right boundary, but maintainers should settle opt-in/defaults, debounce, partial update hydration, edit-aware replay, and cancellation authorization first.
