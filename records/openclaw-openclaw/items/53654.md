## Summary

Keep open. Current main still lacks Discord `MESSAGE_UPDATE`/`MESSAGE_DELETE` listener registration, update/delete gateway mapping, edit-aware replay semantics, and a source message-id cancellation map; the related edit-only and cross-channel trackers overlap but do not fully supersede this Discord edit-plus-delete request.

## What This Changes

The issue requests Discord message edit handling for reprocessing and message delete handling for cancelling the run started by the deleted message.

## Best Possible Solution

Define the Discord edit/delete semantics, then implement them inside the bundled Discord plugin with typed update/delete listeners, monitor registration, hydration/filtering, edit-aware replay, precise source-message cancellation, config/docs alignment, and focused tests.

## Reproduction Assessment

Yes, at source level. Current main has no registered Discord `MESSAGE_UPDATE` or `MESSAGE_DELETE` listener path, and same-message edits would collide with replay dedupe keyed by account, channel, and message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right boundary, but maintainers should settle opt-in/defaults, debounce, partial update hydration, edit-aware replay, and cancellation authorization first.
