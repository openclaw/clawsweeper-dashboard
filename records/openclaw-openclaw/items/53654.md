## Summary

Keep open. Current main and the latest release still lack Discord messageUpdate/messageDelete listener registration, edit-aware replay semantics, and a source-message cancellation map; the related open trackers cover edit reprocessing and broad recall policy separately, but do not fully supersede this Discord-specific combined request.

## What This Changes

Request support for Discord message edits to reprocess corrected commands and message deletes to cancel the active run triggered by that message.

## Best Possible Solution

Define the Discord edit/delete semantics, then implement them in the bundled Discord plugin with typed update/delete dispatch support, edit-aware replay and debounce behavior, source-message cancellation mapping, config/docs alignment, and focused monitor tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source inspection: current main has no registered Discord MESSAGE_UPDATE or MESSAGE_DELETE monitor listener, and the same-message replay key would suppress a naive edit reprocess path.

## Solution Assessment

Is this the best way to solve the issue? Unclear as an automatic repair; the Discord plugin is the right owner boundary, but config defaults, debounce/rate-limit behavior, partial update hydration, and delete cancellation authorization need maintainer direction first.
