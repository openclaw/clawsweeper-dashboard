## Summary

Keep open. Current main and v2026.5.18 still lack Discord MESSAGE_UPDATE/MESSAGE_DELETE listener registration, edit-aware replay semantics, and a source-message delete-to-cancel map; the related edit-only and cross-channel recall trackers do not fully supersede this Discord-specific combined request.

## What This Changes

Request support for Discord messageUpdate and messageDelete events so edited commands can be reprocessed and deleted source messages can cancel active runs.

## Best Possible Solution

Define Discord edit/delete semantics first, then implement typed update/delete dispatch, listener registration, hydration/filtering, edit-aware replay, precise source-message cancellation, config/docs alignment, and focused Discord monitor tests in the bundled Discord plugin.

## Reproduction Assessment

Yes. Source inspection is enough: current main has no registered Discord MESSAGE_UPDATE or MESSAGE_DELETE listener path, and same-message edits would collide with replay dedupe keyed by account, channel, and message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right boundary, but maintainers should settle opt-in/defaults, debounce and hydration policy, edit replay semantics, and precise cancellation authorization first.
