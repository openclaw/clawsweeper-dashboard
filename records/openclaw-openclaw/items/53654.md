## Summary

Keep this open. Current main still has no Discord inbound path for message edits or deletes: the internal gateway layer only hydrates create/reaction dispatches, monitor startup registers no update/delete listeners, and the run queue has no source-message cancellation map. The open related Discord tracker covers only edit reprocessing, so it does not supersede this broader edit-and-delete request.

## What This Changes

Add Discord `messageUpdate` handling for edit-to-reprocess and `messageDelete` handling for delete-to-cancel active agent work triggered by the source message.

## Best Possible Solution

Implement this inside the bundled Discord plugin: add typed gateway update/delete dispatch support, register monitor listeners, route qualifying edits through the existing authorization/preflight/session path with edit-aware replay semantics, and cancel only the active run tied to the deleted source Discord message.

## Reproduction Assessment

Yes. A current-main static reproduction is high-confidence: dispatching `GatewayDispatchEvents.MessageUpdate` or `MessageDelete` would not match any registered listener, and a naive edit re-dispatch using the existing message handler would be blocked by the same account/channel/message replay key.

## Solution Assessment

Yes, the narrow maintainable direction is a Discord plugin repair rather than a core migration. The implementation should reuse the existing message preflight/run queue where possible, adding only the minimal generic lifecycle seam if precise per-source cancellation cannot be expressed locally.
