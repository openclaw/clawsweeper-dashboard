## Summary

Keep open. Current main still has no Discord inbound path for `MESSAGE_UPDATE` or `MESSAGE_DELETE`, and the related open trackers split the remaining work between Discord edit reprocessing and broader recall/delete cancellation policy rather than fully superseding this Discord-specific request.

## What This Changes

Request support for Discord message edits to reprocess corrected commands and message deletes to cancel the active run triggered by that message.

## Best Possible Solution

Define the Discord edit/delete semantics, then implement them in the bundled Discord plugin with typed update/delete dispatch support, edit-aware replay/debounce behavior, source-message cancellation mapping, docs/config alignment, and focused monitor tests.

## Reproduction Assessment

Yes for the current missing behavior by source inspection: a Discord MESSAGE_UPDATE or MESSAGE_DELETE dispatch has no matching registered monitor listener, and same-message edits have no edit-aware replay key. I did not run live Discord E2E because the current dispatch and listener wiring is exact enough to prove the gap.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right owner boundary, but maintainers should decide config defaults, debounce/rate-limit behavior, partial update hydration, and delete cancellation authorization before implementation.
