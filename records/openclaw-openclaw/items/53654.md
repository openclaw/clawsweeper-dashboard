## Summary

Keep open. Current main and v2026.5.18 still lack Discord MESSAGE_UPDATE/MESSAGE_DELETE listener registration, typed update/delete hydration, edit-aware replay semantics, and a source-message-to-run cancellation map; the related open trackers cover the edit half and broad recall policy but do not fully supersede this Discord-specific combined request.

## What This Changes

The issue asks for Discord messageUpdate/messageDelete support so edited commands reprocess and deleted source messages cancel their active agent run.

## Best Possible Solution

Define the Discord-specific edit/delete semantics, then implement them in the bundled Discord plugin with typed gateway dispatch support, listener registration, edit-aware replay, precise source-message cancellation, config/docs alignment, and focused monitor tests.

## Reproduction Assessment

Yes, for the reported current behavior by source inspection: current main has no registered Discord MESSAGE_UPDATE or MESSAGE_DELETE monitor listener, and same-message replay is keyed by account/channel/message id. I did not run a live Discord bot repro in this read-only review.

## Solution Assessment

Unclear as an automatic repair. The bundled Discord plugin is the right owner boundary, but defaults/config, debounce, partial update hydration, and source-message cancellation authorization need maintainer direction first.

## Risks / Open Questions

- Discord MESSAGE_UPDATE payloads can be partial or non-content-only, so implementation needs hydration, content-change filtering, and debounce instead of replaying every update.
- Delete-to-cancel needs strict account/channel/source-message/author mapping so a delete event cannot cancel unrelated newer work in the same session or channel.
- This overlaps https://github.com/openclaw/openclaw/issues/20173 and https://github.com/openclaw/openclaw/issues/55691, so maintainers should decide the split, defaults, and config surface before implementation.
