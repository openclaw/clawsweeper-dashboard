## Summary

Keep this open. Current main still has no Discord gateway path for message update or delete events, and the related open Discord trackers cover only the edit half rather than the broader edit-to-reprocess plus delete-to-cancel behavior requested here.

## What This Changes

Add Discord messageUpdate and messageDelete handling so edited mentioned messages can be reprocessed and deleting the source message can cancel the active run it started.

## Best Possible Solution

Implement this inside the bundled Discord plugin by adding typed MessageUpdate and MessageDelete gateway listener support, registering both monitor listeners, making edit reprocessing explicitly replay-aware, and adding a precise active-run cancellation path keyed to the original Discord message id.

## Repair Work Prompt

Implement Discord edit-to-reprocess and delete-to-cancel for the bundled Discord plugin. Add typed gateway support for MessageUpdate and MessageDelete in the current internal Discord layer, register monitor listeners during Discord startup, and route eligible edited messages through the existing preflight/mention/authorization path with explicit edit-aware replay or supersede semantics. Add a precise active-run cancellation map keyed by account/channel/source Discord message id so deleting the source message aborts only the run started by that message. Handle partial Discord update payloads carefully, hydrating or ignoring updates that cannot prove changed user content, and do not reintroduce @buape/carbon or move owner-specific behavior into core unless a small generic plugin SDK seam is truly required. Add focused tests for listener registration, gateway dispatch hydration, replay behavior for same-message edits, and delete cancellation. Add a user-facing changelog entry if behavior changes ship; credit the reporters with Thanks @sws-apps, @nickytonline, and @softengineware as appropriate.
