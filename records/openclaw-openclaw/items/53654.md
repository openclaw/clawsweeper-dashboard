## Summary

Keep open. Current main still has no Discord MESSAGE_UPDATE or MESSAGE_DELETE listener path, no update/delete gateway hydration, no edit-aware replay semantics, and no source Discord message-id cancellation map; the edit-only tracker at https://github.com/openclaw/openclaw/issues/20173 and cross-channel recall tracker at https://github.com/openclaw/openclaw/issues/55691 overlap but do not fully replace this Discord edit-plus-delete request.

## What This Changes

Add Discord message edit and delete event support so edited commands can be reprocessed and deleted source messages can cancel the active run they started.

## Best Possible Solution

Define the Discord edit/delete semantics, then implement them in the bundled Discord plugin with typed update/delete listeners, monitor registration, hydration/filtering, edit-aware replay, precise source-message cancellation, config/docs alignment, and focused tests.

## Reproduction Assessment

Yes, at source level. Current main has no registered Discord MESSAGE_UPDATE or MESSAGE_DELETE listener path, and same-message edits would collide with replay dedupe keyed by account, channel, and message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right boundary, but maintainers should settle opt-in/defaults, debounce, partial update hydration, edit-aware replay, and cancellation authorization first.
