## Summary

Keep open. Current main and the latest release still lack Discord MESSAGE_UPDATE/MESSAGE_DELETE listener registration, edit-aware replay, and a source-message cancel map; related open trackers cover only the edit half or broader cross-channel recall policy, so they do not supersede this Discord-specific combined request.

## What This Changes

Request support for Discord edit-to-reprocess via MESSAGE_UPDATE and delete-to-cancel via MESSAGE_DELETE for active runs triggered by the deleted source message.

## Best Possible Solution

Define the Discord edit/delete semantics, then implement them in the bundled Discord plugin with typed update/delete dispatch support, listener registration, hydration/filtering, edit-aware replay and debounce behavior, precise source-message cancellation, config/docs alignment, and focused monitor tests.

## Reproduction Assessment

Yes. Source inspection is enough: current main has no registered Discord MESSAGE_UPDATE or MESSAGE_DELETE listener path, and same-message edits would collide with replay dedupe keyed by account/channel/message id.

## Solution Assessment

Unclear as an automatic repair. The Discord plugin is the right boundary, but maintainers should settle opt-in/defaults, debounce, partial update hydration, and precise cancellation authorization first.
