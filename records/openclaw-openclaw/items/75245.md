## Summary

Keep open. Current main still has the reported plugin hook wrapper behavior: `registerHook()` gives plugin handlers a shallow-cloned context, while the bootstrap hook consumer reads `bootstrapFiles` back from the original event after dispatch. The linked fix PR was closed unmerged, and no open replacement PR was found, so the issue remains actionable.

## What This Changes

The issue asks OpenClaw to preserve mutable `event.context` changes made by plugin-registered hooks, especially `agent:bootstrap` updates to `context.bootstrapFiles`.

## Best Possible Solution

Land or replace the narrow PR #75249-style fix: pass the original hook context object to plugin handlers, expose `pluginConfig` only during that handler call, restore or delete it in `finally`, and add regression coverage for plugin-registered `agent:bootstrap` mutation plus `pluginConfig` cleanup.

## Reproduction Assessment

Yes. Register a plugin with `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles`, then call the bootstrap override path; by inspection, current main sends the assignment to the wrapper clone while `applyBootstrapHookOverrides()` reads the unchanged original event context.

## Solution Assessment

Yes. Preserving the original context with temporary `pluginConfig` restoration is the narrowest backward-compatible fix for the documented mutable hook contract; a bootstrap-only workaround or an immutable return-value API would either miss other hooks or require a larger SDK change.
