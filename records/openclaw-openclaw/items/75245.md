## Summary

Keep open. Current main still has the reported wrapper behavior: plugin handlers receive a cloned `event.context`, while the bootstrap consumer reads `bootstrapFiles` back from the original event context. The linked PR is the right implementation candidate and uses closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that plugin-registered hooks, especially `agent:bootstrap`, preserve mutable `event.context` changes made through `api.registerHook()`.

## Best Possible Solution

Preserve the original hook context object in the plugin wrapper while temporarily exposing and restoring `pluginConfig`, then add regression coverage proving plugin-registered `agent:bootstrap` mutations affect injected bootstrap context and that `pluginConfig` does not leak after the handler returns.

## Reproduction Assessment

Yes. Register a plugin with `api.registerHook('agent:bootstrap')` that replaces `event.context.bootstrapFiles` with a sentinel file; by inspection on current main, the assignment lands on the wrapper clone and `applyBootstrapHookOverrides()` reads the unchanged original context.

## Solution Assessment

Yes. The narrow maintainable fix is in `src/plugins/registry.ts`; a bootstrap-only workaround would leave the generic mutable hook contract broken.
