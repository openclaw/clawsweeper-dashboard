## Summary

Keep open. Current main still has the cloned plugin hook context wrapper, while the bootstrap hook consumer and public docs still rely on mutable `context.bootstrapFiles`; the linked open PR is the implementation candidate and the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue reports that plugin-registered hook handlers lose mutable `event.context` changes because `registerHook()` shallow-clones context before invoking the handler.

## Best Possible Solution

Preserve the original hook context object in `src/plugins/registry.ts` while temporarily exposing and restoring `pluginConfig`, then add regression coverage for plugin-registered `agent:bootstrap` mutation and pluginConfig cleanup; reviewing and landing #75249, or a narrow replacement, is the best path.

## Reproduction Assessment

Yes. Register a plugin via `api.registerHook('agent:bootstrap')` that replaces `event.context.bootstrapFiles` with a sentinel file, then resolve bootstrap context and observe current main still uses the original files because the assignment lands on the wrapper clone.

## Solution Assessment

Yes. The narrow maintainable fix is to repair the generic `registerHook()` wrapper; a bootstrap-only special case would leave the broader mutable hook contract broken.
