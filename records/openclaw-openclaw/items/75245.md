## Summary

Keep open. Current `main` still shallow-clones `event.context` in the plugin `registerHook` wrapper, while `agent:bootstrap` explicitly reads the original event context after hooks run and the public docs describe `context.bootstrapFiles` as mutable.

## What This Changes

Fix plugin `api.registerHook()` so mutable `event.context` changes, especially `agent:bootstrap` `bootstrapFiles`, survive handler execution.

## Best Possible Solution

Preserve the original hook context object through the plugin wrapper, temporarily attach and restore `pluginConfig`, and add regression coverage for plugin-loaded `agent:bootstrap` mutation propagation plus config cleanup.

## Reproduction Assessment

Yes. A plugin loaded through `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles` follows a direct source-proven failing path: the wrapper mutates a cloned context, then bootstrap resolution reads the unchanged original context.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/plugins/registry.ts`; special-casing `agent:bootstrap` would leave the generic mutable hook contract broken, and a return-value API would be a larger compatibility change.

Fix openclaw/openclaw#75245 by preserving mutable hook context changes made by plugins registered through `api.registerHook()`. Keep the fix in `src/plugins/registry.ts`: pass the original `evt.context` object to the handler, temporarily expose `pluginConfig` during that handler, and restore/delete `pluginConfig` afterward so it does not leak. Add regression coverage proving a plugin-loaded `agent:bootstrap` hook can replace `context.bootstrapFiles` and that the resulting bootstrap/context files use the sentinel replacement; also cover `pluginConfig` visibility and non-leakage. Use closed PR #75249 only as prior direction, not as landed code. Do not special-case bootstrap in agent code, do not make hook context immutable, and do not change bundled hook behavior outside the wrapper contract. Add a single-line `CHANGELOG.md` fix entry because this is user-facing plugin SDK behavior.
