## Summary

Keep open. Current main still shallow-clones `event.context` in the plugin `registerHook` wrapper, while the `agent:bootstrap` consumer reads `bootstrapFiles` back from the original event context; the related fix PR is closed and unmerged, so the reported bug remains valid and narrow.

## What This Changes

Make plugin hooks preserve `event.context` mutations, especially `agent:bootstrap` `bootstrapFiles` changes, while keeping `pluginConfig` scoped to the current handler.

## Best Possible Solution

Update the plugin registry wrapper to preserve mutable hook context changes while temporarily exposing `pluginConfig`, then add loader-backed bootstrap mutation and config non-leakage regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A plugin loaded through `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles` reproduces by source path: the wrapper writes to a cloned context while bootstrap resolution reads the unchanged original event context.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/plugins/registry.ts`; special-casing bootstrap would leave the generic mutable hook contract broken, and a return-value hook API would be a larger compatibility change.

## Repair Work Prompt

Fix the plugin hook mutable-context bug reported in https://github.com/openclaw/openclaw/issues/75245, using https://github.com/openclaw/openclaw/pull/75249 only as source context. In `src/plugins/registry.ts`, preserve the original `event.context` object passed to plugin hook handlers so mutations such as `event.context.bootstrapFiles = updatedFiles` survive `triggerInternalHook()`, while exposing `pluginConfig` only for the current handler and restoring/deleting it in a `finally` block so it does not leak to other handlers. Add regression coverage proving a plugin loaded through `api.registerHook('agent:bootstrap')` can replace bootstrap files and that resulting `resolveBootstrapContextForRun()` context files use the sentinel replacement. Also cover that `pluginConfig` remains visible to the handler but is not permanently left on the shared event context. Keep the fix generic; do not special-case subagents, bundled plugin ids, or bootstrap-only policy. Add a user-facing `CHANGELOG.md` fix entry with appropriate human attribution if known.
