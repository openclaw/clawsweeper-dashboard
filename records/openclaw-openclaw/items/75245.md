## Summary

Keep this issue open because current main and the latest release still use the shallow-cloned plugin hook context, while an open linked PR already owns the focused repair path.

## What This Changes

The issue reports that `api.registerHook()` handlers cannot persist `agent:bootstrap` mutations such as replacing `event.context.bootstrapFiles` because the registry wrapper passes a cloned context.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/89002 or an equivalent registry-wrapper repair that preserves documented `agent:bootstrap` context mutations while keeping `pluginConfig` scoped per handler.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main clones `evt.context` in the plugin registry wrapper, while `applyBootstrapHookOverrides` reads `bootstrapFiles` from the original event after hook dispatch.

## Solution Assessment

Yes, the linked registry-level repair is the best narrow direction. Fixing the generic `registerHook` wrapper is cleaner than a bootstrap-only workaround, provided `pluginConfig` cleanup remains per-handler.
