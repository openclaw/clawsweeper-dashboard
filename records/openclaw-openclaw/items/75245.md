## Summary

Keep open. Current main still wraps plugin hook events with a shallow-cloned `event.context`, while the bootstrap hook consumer reads `bootstrapFiles` back from the original event context; the provided GitHub context also shows PR #75249 is an open, unmerged implementation candidate using closing syntax for this issue.

## What This Changes

The issue requests preserving mutable `event.context` changes made by plugin-registered `agent:bootstrap` hooks so updated `bootstrapFiles` are used for injected context and prompt reports.

## Best Possible Solution

Preserve the original hook context object in the plugin `registerHook()` wrapper while temporarily exposing and restoring `pluginConfig`, then land regression coverage proving plugin-registered `agent:bootstrap` mutations affect injected bootstrap context and `pluginConfig` does not leak after the handler returns.

## Reproduction Assessment

Yes. Register a plugin with `api.registerHook('agent:bootstrap')` that replaces `event.context.bootstrapFiles` with a sentinel file; by source inspection on current main, that assignment lands on the wrapper clone and `applyBootstrapHookOverrides()` reads the unchanged original context.

## Solution Assessment

Yes. The narrow maintainable fix is in `src/plugins/registry.ts`; a bootstrap-only workaround would leave the generic mutable hook contract broken, while a return-value API would be a larger compatibility change.
