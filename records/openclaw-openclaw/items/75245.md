## Summary

Keep open. Current main still has the reported mismatch: plugin `registerHook()` handlers receive a shallow-cloned `context`, while the `agent:bootstrap` consumer reads `bootstrapFiles` back from the original event object after dispatch. The related fix PR is closed unmerged, so this remains a narrow actionable bug.

## What This Changes

The issue requests that plugin-registered mutable hook context changes, especially `agent:bootstrap` changes to `context.bootstrapFiles`, survive the `registerHook()` wrapper and feed the final injected bootstrap context.

## Best Possible Solution

Preserve the original hook context object for plugin handlers, attach `pluginConfig` only for the duration of that handler call with `finally` cleanup, and add regression coverage proving plugin-registered `agent:bootstrap` mutations affect returned bootstrap files and injected context while `pluginConfig` does not leak.

## Reproduction Assessment

Yes. Register a plugin hook with `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles`; by inspection, current main sends the assignment to the wrapper clone while `applyBootstrapHookOverrides()` reads the unchanged original event context.

## Solution Assessment

Yes. The reporter's Option A is the narrowest backward-compatible fix because it preserves the documented mutable hook contract without adding a bootstrap-only special case or a larger return-value API migration.
