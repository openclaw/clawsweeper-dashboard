## Summary

Keep open. Current main still has the reported plugin hook wrapper behavior: plugin handlers receive a shallow-cloned context, while the bootstrap hook consumer reads `bootstrapFiles` back from the original event after hook dispatch. The related fix PR #75249 was closed unmerged, so the issue remains valid and actionable.

## What This Changes

The issue requests preserving mutable `event.context` changes made by plugin-registered `agent:bootstrap` hooks so updated `bootstrapFiles` feed the injected prompt context.

## Best Possible Solution

Land a narrow registry-wrapper fix that preserves the original hook context object for plugin handlers, exposes `pluginConfig` only during that handler call, restores or deletes it in `finally`, and adds regression coverage for plugin-registered `agent:bootstrap` mutation plus `pluginConfig` cleanup.

## Reproduction Assessment

Yes. Register a plugin with `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles`; by source inspection, current main writes that assignment to the wrapper clone while `applyBootstrapHookOverrides()` reads the unchanged original event context.

## Solution Assessment

Yes. Preserving the original context with temporary `pluginConfig` restoration is the narrowest backward-compatible fix for the mutable hook contract; a bootstrap-only workaround would leave the generic `registerHook()` footgun in place.

## Repair Work Prompt

Fix openclaw/openclaw#75245. Current main clones `event.context` in `src/plugins/registry.ts` when wrapping plugin `api.registerHook()` handlers to inject `pluginConfig`, which drops mutations required by `agent:bootstrap`. Implement the narrow backward-compatible repair: pass the original context object to the handler, temporarily attach `pluginConfig`, and restore the previous value or delete it in a `finally` block after the handler returns. Do not special-case only `agent:bootstrap`, do not redesign the hook API, and do not leak `pluginConfig` onto the shared event context after a handler. Add focused regression coverage: a plugin registered with `api.registerHook('agent:bootstrap')` replaces `bootstrapFiles` with a sentinel file and that file survives through the bootstrap override/context path; a plugin still sees `pluginConfig`; and `pluginConfig` is cleaned up/restored after the handler. Include a user-facing `CHANGELOG.md` fix entry crediting the reporter, not bot/maintainer handles. Use PR #75249 only as prior context; create a clean narrow replacement if needed.
