## Summary

Keep open. Current main still has the reported mismatch: plugin `registerHook()` handlers receive a shallow-cloned `context`, while the bootstrap hook consumer reads `bootstrapFiles` from the original event context after dispatch. The docs still describe `agent:bootstrap` `context.bootstrapFiles` as mutable, and the linked implementation PR is closed unmerged, so the issue remains valid and actionable.

## What This Changes

The issue requests preserving mutations made by plugin-registered `agent:bootstrap` hooks so updated `context.bootstrapFiles` is used when building injected bootstrap context and prompt reports.

## Best Possible Solution

Preserve the existing mutable hook contract by passing the original hook context object to plugin handlers, exposing `pluginConfig` only for that handler call, restoring or deleting it in `finally`, and adding regression coverage for plugin-registered `agent:bootstrap` mutation plus `pluginConfig` cleanup.

## Reproduction Assessment

Yes. Register a plugin with `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles`, then resolve bootstrap context; by source inspection, the assignment lands on the wrapper clone while `applyBootstrapHookOverrides()` reads the unchanged original event context.

## Solution Assessment

Yes. The narrow backward-compatible fix is to preserve the original context object with temporary `pluginConfig` restoration; a bootstrap-only special case would miss the generic `registerHook()` contract bug, and an immutable return-value API would be a larger SDK change.
