## Label Justifications

- `P2`: The bug affects plugin hook behavior and agent bootstrap context, but the blast radius is bounded to plugins relying on mutable hook context.
- `impact:session-state`: Dropped bootstrap mutations can put the wrong files into an agent session's model-visible context.

## Summary

Keep open: current main and latest release still shallow-clone the hook context in the plugin `registerHook` wrapper, while the bootstrap consumer and docs still rely on `agent:bootstrap` mutating `context.bootstrapFiles`. The direct fix PRs were closed unmerged, and the open skipBootstrap PR is adjacent but does not repair the wrapper.

## What This Changes

The issue reports that plugin `registerHook` handlers lose mutable `agent:bootstrap` context changes, so `bootstrapFiles` replacements do not reach final prompt assembly.

## Best Possible Solution

Update the plugin `registerHook` wrapper to pass the original context object to handlers, restore or remove `pluginConfig` in `finally`, and add loader regression tests for `agent:bootstrap` mutation propagation plus config non-leakage.

## Reproduction Assessment

Yes, at source level: a plugin loaded through `api.registerHook("agent:bootstrap", ...)` receives a cloned `context`, while core reads the original `event.context.bootstrapFiles` after hooks return. A live packaged-plugin repro was not run because this review is read-only.

## Solution Assessment

Yes. The best narrow fix is in `src/plugins/registry.ts`: preserve the original mutable context object while temporarily scoping `pluginConfig`; changing bootstrap core would only special-case one affected hook.
