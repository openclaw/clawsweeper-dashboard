## Summary

Keep open. Current main still shallow-clones `event.context` inside the plugin `registerHook` wrapper while the bootstrap path reads `bootstrapFiles` back from the original event context, so the reported mutation loss remains present; the related fix PR was closed unmerged.

## What This Changes

The issue reports that plugin `registerHook()` handlers can mutate a cloned hook context, causing `agent:bootstrap` changes such as `bootstrapFiles` replacement to be lost before prompt injection.

## Best Possible Solution

Preserve the original hook context object in `registerHook`, expose `pluginConfig` only for the current handler with `finally` cleanup, and add plugin-loader regression coverage for bootstrap mutation propagation and config non-leakage.

## Reproduction Assessment

Yes. A plugin loaded through `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles` reproduces by source path: the wrapper mutates a cloned context, while bootstrap resolution reads the unchanged original event context.

## Solution Assessment

Yes. The narrow maintainable fix is in `src/plugins/registry.ts`; special-casing bootstrap would leave the generic mutable hook contract broken, and replacing mutation with a return-value API would be a larger compatibility change.
