## Summary

Keep open. Current main still clones the plugin hook context in `api.registerHook()`, while the bootstrap override path reads `bootstrapFiles` back from the original event context, so `agent:bootstrap` mutations made by plugins remain lost; the linked fix PR is closed unmerged.

## What This Changes

Preserve mutable `event.context` changes made by `api.registerHook()` handlers, especially `agent:bootstrap` `bootstrapFiles` replacements.

## Best Possible Solution

Update the plugin registry wrapper to preserve the original hook context while exposing per-handler `pluginConfig` with cleanup, then add loader-backed bootstrap mutation and no-leak regression coverage.

## Reproduction Assessment

Yes. Register a plugin with `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles`, then resolve bootstrap context; current main deterministically sends the assignment to the wrapper clone while bootstrap resolution reads the unchanged original event.

## Solution Assessment

Yes. Fixing `src/plugins/registry.ts` is the narrowest maintainable solution; a bootstrap-only special case would leave the generic `registerHook()` mutable-context contract broken, while a return-value API would be a larger compatibility change.

## Repair Work Prompt

Fix the `api.registerHook()` mutable context regression tracked by https://github.com/openclaw/openclaw/issues/75245. In `src/plugins/registry.ts`, preserve the original `evt.context` object for plugin hook handlers so assignments such as `event.context.bootstrapFiles = updatedFiles` survive, while still exposing `pluginConfig` only during each handler call and restoring/removing it in `finally` so it does not leak to the shared event object or other handlers. Add regression coverage proving a loader-registered plugin `agent:bootstrap` hook can replace `bootstrapFiles` and that resolved bootstrap `contextFiles` use the sentinel replacement. Also keep or add coverage that `pluginConfig` is visible inside the handler but not left on `event.context` after `triggerInternalHook`. Do not change typed `api.on(...)` hook semantics or add bootstrap-specific policy in agent code. Add a single-line `CHANGELOG.md` fix entry because this is user-facing plugin behavior. Related prior source work: https://github.com/openclaw/openclaw/pull/75249.
