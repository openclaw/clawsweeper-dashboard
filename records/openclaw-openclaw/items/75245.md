## Label Justifications

- `P2`: This is a source-reproducible plugin hook contract bug with meaningful agent bootstrap impact but a bounded plugin registry repair surface.
- `impact:session-state`: The issue is about bootstrap context mutations being lost before final agent prompt/session context construction.

## Summary

Keep this issue open because current main and the latest release still shallow-clone plugin hook context, so `agent:bootstrap` mutations made through `api.registerHook()` are still lost; the prior focused repair PRs are all closed unmerged and the only open related PR is adjacent skipBootstrap work, not this wrapper bug.

## What This Changes

The issue reports that `api.registerHook()` drops mutable hook context changes, especially `agent:bootstrap` updates to `event.context.bootstrapFiles`, before core builds injected context and prompt report metadata.

## Best Possible Solution

Land a narrow plugin-registry repair that passes the original mutable event context to hook handlers, temporarily injects `pluginConfig`, restores or removes it in `finally`, and adds regression tests for `agent:bootstrap` mutation propagation plus config cleanup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main clones `evt.context` in the plugin registry wrapper, while `applyBootstrapHookOverrides` reads `bootstrapFiles` from the original event after hook dispatch.

## Solution Assessment

Yes. Fixing the generic `registerHook` wrapper is the narrow maintainable solution; a bootstrap-only workaround would leave the documented mutable hook contract broken for other hooks.
