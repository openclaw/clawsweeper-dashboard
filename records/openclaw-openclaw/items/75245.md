## Summary

Keep open. Current main still shallow-clones `event.context` in the plugin hook wrapper while the bootstrap hook path reads `bootstrapFiles` back from the original event context, so plugin-loaded `agent:bootstrap` mutations are still dropped.

## What This Changes

The issue requests preserving mutable hook context changes made through `api.registerHook()`, especially `agent:bootstrap` updates to `context.bootstrapFiles`.

## Best Possible Solution

Preserve the original hook context object in the plugin `registerHook` wrapper, temporarily expose `pluginConfig` with cleanup, and add loader-backed regression coverage for bootstrap mutation propagation and config non-leakage.

## Reproduction Assessment

Yes. A plugin loaded through `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles` reproduces by source path: the wrapper mutates a cloned context while bootstrap resolution reads the unchanged original event context.

## Solution Assessment

Yes. The narrowest maintainable fix is in the plugin registry wrapper; special-casing bootstrap would leave the generic mutable hook contract broken, while a return-value hook API would be a larger compatibility change.

## Repair Work Prompt

Fix the mutable hook context bug reported in https://github.com/openclaw/openclaw/issues/75245. In `src/plugins/registry.ts`, update the `api.registerHook()` wrapper so plugin handlers receive the original event context object, while `pluginConfig` remains visible only during the current handler and is restored or removed in a `finally` block afterward. Do not special-case `agent:bootstrap`, do not change the hook API to a return-value model, and preserve the documented per-handler `pluginConfig` isolation. Add regression coverage showing a plugin loaded through `api.registerHook('agent:bootstrap')` can replace `event.context.bootstrapFiles` and that `resolveBootstrapContextForRun()` uses the sentinel file in both returned `bootstrapFiles` and injected `contextFiles`. Add or keep coverage that `pluginConfig` is visible to the handler but does not permanently leak onto the shared event context. Add a single-line `CHANGELOG.md` fix entry for the active version without bot attribution. Use #75249 only as source context; it is closed and unmerged.
