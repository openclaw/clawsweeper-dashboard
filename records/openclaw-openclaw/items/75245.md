## Summary

Keep open. Current main still shallow-clones `event.context` in the plugin `registerHook` wrapper while `agent:bootstrap` reads `bootstrapFiles` back from the original event context, so plugin mutations are still dropped before prompt injection.

## What This Changes

The issue requests that context mutations made by plugins through `api.registerHook('agent:bootstrap')` survive the wrapper and affect bootstrap file injection.

## Best Possible Solution

Preserve the original hook context object in `registerHook`, expose `pluginConfig` only for the current handler with `finally` cleanup, and add loader-backed regression coverage for bootstrap mutation propagation and config non-leakage.

## Reproduction Assessment

Yes. A plugin loaded through `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles` reproduces by source path: the wrapper writes to a cloned context while bootstrap resolution reads the unchanged original event context.

## Solution Assessment

Yes. The narrowest maintainable repair is the registry wrapper fix; special-casing bootstrap would leave the generic mutable hook contract broken, and a return-value hook API would be a larger compatibility change.

## Repair Work Prompt

Fix openclaw/openclaw#75245. In `src/plugins/registry.ts`, change the `api.registerHook` wrapper so plugin handlers receive the original mutable `event.context` object, while `pluginConfig` is attached only for that handler call and restored or deleted in a `finally` block afterward. Do not special-case `agent:bootstrap`, do not replace the public hook API with a return-value-only contract, and preserve the documented behavior that `pluginConfig` is visible during the handler but does not permanently leak onto the shared event context. Add regression coverage for a plugin loaded through `api.registerHook('agent:bootstrap')` replacing `event.context.bootstrapFiles` with a sentinel file and verify the resolved bootstrap/context files use the sentinel rather than the original files. Add or adjust loader coverage proving `pluginConfig` remains visible inside the handler and absent/restored after triggering. Add a single-line `CHANGELOG.md` fix entry because this is user-facing plugin behavior. Validate with the targeted tests and changed gate below; use Testbox for broad `pnpm check:changed` per repo policy.
