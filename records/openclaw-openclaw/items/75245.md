## Summary

Keep open. Current main still passes plugin hook handlers a shallow-cloned `event.context`, while the `agent:bootstrap` consumer reads `bootstrapFiles` back from the original event context; the related fix PR was closed unmerged, so the bug remains actionable.

## What This Changes

The issue asks OpenClaw to make plugin `api.registerHook()` preserve mutable `event.context` changes for `agent:bootstrap` and other mutable internal hooks while keeping `pluginConfig` scoped per handler.

## Best Possible Solution

Preserve the original hook context object in the plugin wrapper, temporarily expose `pluginConfig` with `finally` cleanup, and add loader-backed regression coverage for bootstrap mutation plus no persistent `pluginConfig` leak.

## Reproduction Assessment

Yes. Register a plugin with `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles`, then resolve bootstrap context; current main sends the assignment to the wrapper clone while the bootstrap code reads the unchanged original event.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/plugins/registry.ts`; a bootstrap-only special case would leave the generic wrapper contract broken, and an immutable return-value API would be a larger compatibility change.

## Repair Work Prompt

Fix issue #75245 by preserving mutable hook context changes made through plugin `api.registerHook()` handlers. In `src/plugins/registry.ts`, update the `registerHook` wrapper so it passes the original `evt.context` object to the handler and exposes `pluginConfig` only for that handler call, restoring the previous value or deleting it in a `finally` block. Do not special-case only `agent:bootstrap`, do not change the public API to a new immutable return-value contract, and do not let `pluginConfig` leak onto the shared event after the handler returns. Add regression coverage for a loader-backed plugin registering `api.registerHook('agent:bootstrap')` that replaces `event.context.bootstrapFiles`, then verify `applyBootstrapHookOverrides()`/bootstrap context uses the sentinel file. Also cover that `pluginConfig` remains visible during the handler and is restored/removed afterward. Add a user-facing `CHANGELOG.md` fix entry because this affects plugin authors. Related refs: #75245 and closed unmerged PR #75249.
