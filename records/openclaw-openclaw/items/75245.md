## Summary

Keep open. Current main still passes plugin hook handlers a shallow-cloned `event.context`, while `agent:bootstrap` reads `bootstrapFiles` back from the original event context; the related fix PR is closed unmerged and no replacement open PR was found.

## What This Changes

The issue asks OpenClaw to preserve mutable `event.context` changes made by plugin `api.registerHook()` handlers, especially `agent:bootstrap` `bootstrapFiles` replacement.

## Best Possible Solution

Preserve the original hook context object in the plugin wrapper, expose `pluginConfig` only during each handler call with `finally` cleanup, and add loader-backed regression coverage for bootstrap mutation plus no persistent config leak.

## Reproduction Assessment

Yes. Register a plugin with `api.registerHook('agent:bootstrap')` that assigns `event.context.bootstrapFiles`, then resolve bootstrap context; current main sends that assignment to the wrapper clone while the bootstrap consumer reads the unchanged original event.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/plugins/registry.ts`; a bootstrap-only special case would leave the generic plugin hook contract broken, and a return-value API would be a larger compatibility change.

## Repair Work Prompt

Fix issue #75245 by preserving mutable hook context changes made through plugin `api.registerHook()` handlers. In `src/plugins/registry.ts`, update the `registerHook` wrapper so it passes the original `evt.context` object to the handler and exposes `pluginConfig` only for that handler call, restoring the previous value or deleting it in a `finally` block. Do not special-case only `agent:bootstrap`, do not change the public API to a new immutable return-value contract, and do not let `pluginConfig` leak onto the shared event after the handler returns. Add regression coverage for a loader-backed plugin registering `api.registerHook('agent:bootstrap')` that replaces `event.context.bootstrapFiles`, then verify `applyBootstrapHookOverrides()` or bootstrap context uses the sentinel file. Also cover that `pluginConfig` remains visible during the handler and is restored/removed afterward. Add a user-facing `CHANGELOG.md` fix entry because this affects plugin authors. Related refs: https://github.com/openclaw/openclaw/issues/75245 and closed unmerged PR https://github.com/openclaw/openclaw/pull/75249.
