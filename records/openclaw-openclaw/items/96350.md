## Summary

Keep open: current main and v2026.6.10 still ignore Android `system.notify` deeplink/contentIntent-style parameters, but adding a notification tap target is security-sensitive and needs maintainer product/security direction before implementation.

## What This Changes

Add Android `system.notify` deeplink/contentIntent support so tapping a notification can open an intended action route.

## Best Possible Solution

Define a narrow Android notification-tap contract, preferably sharing the allowlisted deeplink policy from https://github.com/openclaw/openclaw/issues/88293 or restricting taps to OpenClaw-owned deep links, then implement it with Android, gateway policy, and docs coverage.

## Reproduction Assessment

Yes. Current main source shows Android `system.notify` parses only title/body/sound/priority and builds a notification without `setContentIntent`; I did not run a live Pixel tap repro.

## Solution Assessment

Unclear. Passing a raw deeplink directly into `system.notify` is plausible but not the best final shape until maintainers decide the allowlist/OpenClaw-only target policy and how it relates to the broader Android deeplink request.
