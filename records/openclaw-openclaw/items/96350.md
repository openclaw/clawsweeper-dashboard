## Summary

Keep open: current main and v2026.6.10 still parse and post Android `system.notify` without a `contentIntent`, and accepting caller-driven deeplinks needs maintainer product/security direction rather than automatic cleanup.

## What This Changes

Request to let Android `system.notify` notifications include a deeplink/contentIntent so tapping them can open a relevant action instead of doing nothing.

## Best Possible Solution

Define a narrow Android notification-tap contract, ideally sharing the broader allowlisted deeplink policy or restricting taps to OpenClaw-owned deep links, then implement it with Android unit coverage and docs.

## Reproduction Assessment

Yes, source-reproducible: current Android source parses only title/body/sound/priority and builds the notification without `setContentIntent`. I did not run a live Pixel/Android tap repro in this read-only review.

## Solution Assessment

Unclear: wiring a raw `deeplink` parameter directly into `system.notify` is plausible but not clearly the safest final shape. A maintainable solution needs maintainer agreement on allowed targets and whether this shares the broader allowlisted Android URL/deeplink policy.
