## Summary

Keep open: current main still builds Android `system.notify` notifications without a deeplink/contentIntent path, and the safe product shape needs maintainer security review before implementation.

## What This Changes

Add Android support for a deeplink/contentIntent parameter on `system.notify` so tapping a node notification can open the intended action.

## Best Possible Solution

Define a narrow Android notification-tap contract, preferably sharing the allowlisted deeplink policy from the broader Android URL-launch discussion or restricting taps to OpenClaw-owned deep links, then implement it with Android unit coverage and docs.

## Reproduction Assessment

Yes, source-reproducible: current Android source parses only title/body/sound/priority and builds the notification without `setContentIntent`. I did not run a Pixel/Android live repro in this read-only review.

## Solution Assessment

Unclear: wiring a raw `deeplink` parameter directly into `system.notify` is plausible but not clearly the safest final shape. The better solution needs maintainer agreement on allowed targets and whether this shares the broader allowlisted Android URL/deeplink command policy.
