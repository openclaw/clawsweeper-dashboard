## Summary

Keep open: current main and the latest release still build Android `system.notify` notifications without any deeplink/contentIntent path, and the remaining work is a security-sensitive Android product/API decision rather than a narrow bug fix.

## What This Changes

The issue asks for Android `system.notify` notifications to honor a deeplink/contentIntent so tapping the notification can open a specific local app route.

## Best Possible Solution

Define a narrow Android notification-tap contract, preferably reusing the allowlisted deeplink policy from the broader Android URL-launch discussion or restricting taps to OpenClaw-owned deep links, then implement it with Android tests and docs.

## Reproduction Assessment

Yes, source-reproducible: current Android code parses only title/body/sound/priority and builds the notification without a content intent. I did not run a live Pixel repro in this read-only review.

## Solution Assessment

Unclear: adding a raw `deeplink` parameter to `system.notify` is plausible but not yet the safest confirmed shape. Maintainers should first decide whether notification taps share the broader allowlisted Android URL-launch policy or stay limited to OpenClaw-owned deep links.
