## Summary

Keep open: current main and the latest release still build Android `system.notify` notifications without a deeplink/contentIntent path, and adding one is a new security-sensitive Android launch surface that needs maintainer product/security direction.

## What This Changes

Request Android `system.notify` notifications to open a deeplink/contentIntent when tapped.

## Best Possible Solution

Define a narrow allowlisted Android notification-tap contract, preferably sharing policy with the broader Android deeplink-launch discussion, then add Android parsing, PendingIntent construction, focused tests, and user-facing docs.

## Reproduction Assessment

Yes, source-reproducible: current Android source parses only title/body/sound/priority and builds the notification without `setContentIntent`. I did not run a Pixel/Android live repro in this read-only review.

## Solution Assessment

Unclear: wiring a raw `deeplink` parameter directly into `system.notify` is plausible but not clearly the safest final shape. The better solution needs maintainer agreement on allowlisted targets and whether this shares the broader Android URL/deeplink command policy.
