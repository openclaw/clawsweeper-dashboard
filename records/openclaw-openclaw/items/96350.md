## Summary

Keep open: current main and v2026.6.10 still build Android `system.notify` notifications without any deeplink/contentIntent path, and the remaining work needs security/product direction rather than cleanup closure.

## What This Changes

Request Android `system.notify` to honor a deeplink/contentIntent parameter so tapping posted notifications opens the intended Android or OpenClaw target.

## Best Possible Solution

Define a narrow Android notification-tap contract, sharing the allowlisted deeplink policy from the broader Android URL-launch discussion or limiting taps to OpenClaw-owned deep links, then add PendingIntent support with Android coverage.

## Reproduction Assessment

Yes, source-reproducible: current main carries only title/body/sound/priority through Android `system.notify`, and the notification builder never sets a content intent. I did not run a live Pixel repro in this read-only review.

## Solution Assessment

Unclear: directly wiring a raw `deeplink` parameter into `system.notify` is only a plausible implementation. The safer final shape needs maintainer agreement on whether taps use the broader allowlisted Android URL policy or an OpenClaw-only deep-link contract.
