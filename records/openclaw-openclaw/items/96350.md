## Summary

Keep open: current main and the latest release still build Android `system.notify` notifications without a deeplink/contentIntent path, and the safe implementation shape needs maintainer product/security direction.

## What This Changes

Requested Android `system.notify` notification taps should open a supplied deeplink/contentIntent instead of doing nothing.

## Best Possible Solution

Define a narrow Android notification-tap contract, ideally sharing the safe-target policy from https://github.com/openclaw/openclaw/issues/88293 or restricting taps to OpenClaw-owned deep links, then implement it with Android and gateway coverage plus docs.

## Reproduction Assessment

Yes. Source inspection on current main shows Android `system.notify` parses only title/body/sound/priority and builds the notification without `setContentIntent`; I did not run a live Pixel repro.

## Solution Assessment

Unclear. Wiring a raw `deeplink` parameter directly into `system.notify` is plausible but not the safest final shape until maintainers choose allowed targets, confirmation behavior, and whether this shares the broader Android URL-launch policy.
