## Summary

Keep open: current main and v2026.6.10 still build Android `system.notify` notifications without a deeplink/contentIntent path, and the safe fix requires maintainer security/product direction rather than a strict bug-fix lane.

## What This Changes

Request Android `system.notify` to attach a tap action/contentIntent from a deeplink parameter so Android notifications can open a target flow.

## Best Possible Solution

Define a narrow Android notification-tap contract, preferably sharing the allowlisted deeplink policy from the broader Android URL-launch discussion or restricting taps to OpenClaw-owned deep links, then implement it with Android unit coverage and docs.

## Reproduction Assessment

Yes, source-reproducible: current Android source parses only title/body/sound/priority and builds the notification without `setContentIntent`. I did not run a Pixel/Android live repro in this read-only review.

## Solution Assessment

Unclear: wiring a raw `deeplink` parameter directly into `system.notify` is plausible but not the safest final shape until maintainers choose the allowlist/OpenClaw-owned target policy and its overlap with the broader Android URL-launch issue.
