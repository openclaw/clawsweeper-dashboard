## Summary

Keep open: current main and the latest release still ignore any Android `system.notify` deeplink/contentIntent input, and a safe fix needs maintainer product/security direction rather than cleanup closure.

## What This Changes

The issue requests Android `system.notify` support for a deeplink/contentIntent so tapping an Android notification can open the intended local flow.

## Best Possible Solution

Keep this open until maintainers choose a narrow Android notification-tap contract, preferably reusing an allowlisted deeplink policy or limiting taps to OpenClaw-owned deep links, then implement it with Android handler coverage and docs.

## Reproduction Assessment

Yes, source-reproducible: current Android source parses only title/body/sound/priority and builds the notification without `setContentIntent`. I did not run a Pixel/Android live repro in this read-only review.

## Solution Assessment

Unclear: wiring a raw `deeplink` parameter directly into `system.notify` is plausible but not clearly the safest final shape. A shared allowlisted Android deeplink policy or OpenClaw-only deep-link target would be safer than accepting arbitrary app-launch intents.
