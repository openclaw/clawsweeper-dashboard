## Summary

Keep open: current main and the latest release still ignore any deeplink/contentIntent data on Android `system.notify`, and the related deeplink-launch issue is only a partial canonical overlap rather than a complete replacement for notification tap behavior.

## What This Changes

The issue requests Android notification tap handling for `system.notify` by wiring a deeplink/contentIntent parameter into an Android PendingIntent.

## Best Possible Solution

Fold the deeplink security policy into the canonical Android allowlisted-launch design, then decide whether `system.notify` should expose a constrained contentIntent delivery surface or whether a first-class open-url command is the only supported path.

## Reproduction Assessment

Yes for the missing behavior from source: Android `system.notify` parses only title/body/sound/priority and builds a notification without a content intent. I did not run a Pixel device reproduction in this read-only review.

## Solution Assessment

No, not yet as a standalone implementation request: setting a PendingIntent directly from a caller-supplied deeplink is too broad without the allowlist and Android launch policy tracked in the related issue.
