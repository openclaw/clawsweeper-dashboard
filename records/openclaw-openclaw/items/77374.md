## Summary

Keep open: this is a fresh Control UI/WebChat history-loss bug with a plausible current-main source path, and the newer provider-dependent `reasoning_content` clue needs maintainer review before a generic rendering change risks exposing hidden reasoning.

## What This Changes

The issue asks for persisted assistant messages to remain visible in Control UI/WebChat across subsequent sends and reloads when the backend transcript still contains those turns.

## Best Possible Solution

Add a regression from a real or minimal Xiaomi/opencode-go transcript shape, then fix the provider/transport or projection boundary so visible assistant answers persist while hidden reasoning stays hidden.

## Reproduction Assessment

Source-reproducible for the likely `reasoning_content` shape: current UI/Gateway code would keep such a transcript row but render it as an empty assistant message. The exact reporter JSONL is still needed for a high-confidence live reproduction of the full provider-dependent path.

## Solution Assessment

Unclear: broadly displaying `reasoning_content` is not the safest fix because existing docs and code treat reasoning as hidden. The narrowest maintainable solution should compare affected and unaffected JSONL, then normalize only provider-misclassified final-answer text or add an explicit safe projection rule.
