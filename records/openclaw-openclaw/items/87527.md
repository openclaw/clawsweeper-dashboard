## Label Justifications

- `merge-risk: 🚨 compatibility`: Changing omitted child-stdin defaulting in service-managed mode can alter behavior for existing service deployments and child-process callers.

## Summary

Keep open: the patch is a plausible narrow fix, but current main and the latest release still have the inherited-stdin default, and this PR still lacks changed-path real behavior proof for a service-managed child spawn with omitted `stdinMode`. The change is compatibility-sensitive because existing service-managed callers that omit `stdinMode` would stop inheriting stdin and receive EOF instead.

## What This Changes

The PR changes `createChildAdapter` so service-managed runtimes with omitted `stdinMode` default child stdin to `pipe-closed` and reuses a cached `serviceManaged` boolean for detached handling.

## Best Possible Solution

Land the adapter change only after focused service-managed omitted-stdin coverage and redacted launchd/Node after-fix proof, or fold the same semantics into the broader EBADF hardening tracked by https://github.com/openclaw/openclaw/issues/77750.

## Reproduction Assessment

No, not at live-runtime confidence. Current source clearly has the inherited-stdin default the PR changes, but the supplied bash-tools logs do not exercise the changed omitted-stdin branch.

## Solution Assessment

Likely yes as a narrow mitigation, but not fully proven. The adapter is the right owner for omitted `stdinMode` defaults and `pipe-closed` avoids the earlier open-pipe EOF problem, but changed-path coverage and real service proof are still needed.
