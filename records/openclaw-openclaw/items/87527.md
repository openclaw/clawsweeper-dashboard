## Label Justifications

- `P2`: The PR targets a bounded process-spawn availability failure in service-managed runtimes with limited blast radius but real user impact.
- `merge-risk: 🚨 compatibility`: Changing omitted child-stdin defaulting in service-managed mode can alter behavior for existing service deployments and child-process callers.

## Summary

Keep open: current main and v2026.6.8 still inherit stdin for service-managed child spawns when `stdinMode` is omitted, and this PR is a plausible small mitigation, but it still needs direct changed-path proof and maintainer comfort with the compatibility-sensitive default change before merge.

## What This Changes

The PR changes `createChildAdapter` so service-managed runtimes with omitted `stdinMode` default child stdin to `pipe-closed` and reuse one `serviceManaged` check for detached handling.

## Best Possible Solution

Land this adapter default only after focused service-managed omitted-stdin coverage and redacted launchd/Node after-fix proof, or fold the same semantics into the broader hardening tracked by https://github.com/openclaw/openclaw/issues/77750.

## Reproduction Assessment

No, not at high live-runtime confidence. Source shows the inherited-stdin default that the PR changes, but the supplied after-fix logs do not directly exercise a service-managed child spawn with omitted `stdinMode`.

## Solution Assessment

Likely yes as a narrow adapter mitigation, but not yet fully proven. The adapter owns omitted `stdinMode` defaults, while bash-tool proof is not enough because current bash-tool exec passes explicit stdin modes.
