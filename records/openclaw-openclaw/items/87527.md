## Summary

Keep open: current main and the latest release still leave the generic service-managed child-adapter no-input path on inherited stdin, so the branch is not obsolete. The remaining blocker is proof quality: the copied bash-tools logs do not convincingly exercise the changed default because the bash exec path already passes an explicit `stdinMode`.

## What This Changes

The PR changes `createChildAdapter` so service-managed runtimes default omitted child stdin to `pipe-closed` instead of inheriting fd 0.

## Best Possible Solution

Land a narrow child-adapter fix with focused service-managed omitted-stdin coverage and redacted launchd/Node after-fix proof, or fold the same `pipe-closed` semantics into the broader EBADF hardening tracked by https://github.com/openclaw/openclaw/issues/77750.

## Reproduction Assessment

No, not at live-runtime confidence. The current source clearly has the inherited-stdin default the PR changes, but I did not reproduce the launchd/Node EBADF failure, and the supplied bash-tools logs do not exercise the changed default branch.

## Solution Assessment

Likely yes as a narrow mitigation, but not fully proven. The adapter is the right owner for omitted `stdinMode` defaults and `pipe-closed` avoids the prior open-pipe EOF problem, but it still needs changed-path coverage and real service proof before merge.
