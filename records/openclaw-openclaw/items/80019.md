## Summary

Keep open: current main still does not provide the requested read-only cron inspection scope beyond scoped `status` and current-job-only `list`, and the linked open PR explicitly references this issue as its fix path. The remaining boundary is security-sensitive because broader `list` or `runs` access could expose cron inventory or diagnostics from an isolated job.

## What This Changes

The issue asks for isolated scheduled jobs to inspect cron status, job inventory, and recent run history while keeping mutation actions blocked except current-job self-removal.

## Best Possible Solution

Review and either land or adjust the linked PR for current-job `runs`, then separately define and test any broader isolated cron inventory policy before exposing it.

## Reproduction Assessment

Yes by source inspection: current main whitelists scoped `status` and `list`, but the guard and tests still reject `runs` from self-remove scope. I did not execute tests because this review is read-only and `pnpm` is unavailable here.

Unclear: the linked PR is a plausible narrow fix for current-job run history, but broader inventory or cross-job diagnostics need a maintainer permission-boundary decision.
