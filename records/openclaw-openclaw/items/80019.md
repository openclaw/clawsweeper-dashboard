## Summary

Keep open: current main still only permits scoped `status` and current-job-only `list`, while `runs` remains denied; the open implementation PR at https://github.com/openclaw/openclaw/pull/80022 references this issue with closing syntax and should be reviewed or closed before this issue is resolved.

## What This Changes

Request a read-only cron inspection scope for isolated scheduled jobs so they can inspect scheduler status, job/list data, and recent run history while mutation actions remain blocked.

## Best Possible Solution

Define the safe isolated-job read boundary, then land one canonical change through https://github.com/openclaw/openclaw/pull/80022 or a replacement with scoped tests and docs.

## Reproduction Assessment

Yes by source inspection: current main allows self-scoped `status` and self-filtered `list`, but `assertCronSelfRemoveScope` still throws for `runs` and the test table asserts that denial. I did not execute tests because this review is read-only.

## Solution Assessment

Unclear: current-job `runs` access may be a narrow maintainable extension, but broader inventory or all-run inspection needs maintainer product/security approval before implementation.
