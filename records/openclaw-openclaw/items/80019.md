## Summary

Keep open: current main still blocks `cron runs` from self-remove-scoped isolated cron jobs, the remaining work is paired with an open implementation PR, and the permission boundary needs maintainer review before this issue can close.

## What This Changes

The issue requests a read-only cron inspection scope so isolated scheduled-job turns can call `status`, `list`, and recent `runs` while mutation actions remain blocked.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/80022 and either land a current-job-only run-history scope with focused tests/docs/proof or explicitly decide that broader isolated cron inspection is out of bounds.

## Reproduction Assessment

Yes by source inspection: current main allows scoped `status` and current-job-only `list`, but the guard rejects `runs` before gateway dispatch and the test table locks in that denial.

## Solution Assessment

Unclear: the linked PR's current-job-only shape is plausible, but expanding isolated scheduled-job read access is a security/product boundary decision that maintainers should settle before merge.
