## Summary

Keep open: current main already covers the narrow self-introspection fix for `status` and current-job-only `list`, but this issue asks for a broader read-only cron health scope, especially inventory and run-history access, which is still missing and needs a permission-boundary decision.

## What This Changes

Add a read-only cron inspection scope so isolated scheduled-job turns can inspect scheduler status, job inventory, and recent runs while keeping mutation actions blocked except current-job self-removal.

## Best Possible Solution

Define the intended read boundary for isolated cron health monitors, then implement the selected `status`/`list`/`runs` behavior in the cron tool and gateway contracts with regression tests and docs.

## Reproduction Assessment

Yes for the remaining gap by source inspection: current main permits scoped `status` and current-job-only `list`, but the guard and tests still reject `runs` and do not expose job inventory. I did not execute tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear: broad read-only inventory and run-history inspection is useful, but current main deliberately narrowed the safety scope in https://github.com/openclaw/openclaw/pull/78403. Maintainers should choose the safe scope before implementation.
