## Summary

Keep open: current main still limits isolated cron self-cleanup scope to `status`, self-filtered `list`, and current-job `remove`; `runs` and broader inventory remain blocked, and the linked open PR is the active implementation path for the narrow current-job run-history part.

## What This Changes

Request isolated scheduled-job turns to get read-only cron inspection for scheduler status, job listing, and recent run history while keeping mutation actions blocked.

## Best Possible Solution

Define the isolated cron read boundary, land the narrow accepted scope with cron-tool regression tests and docs, and keep mutation paths blocked except current-job self-removal.

## Reproduction Assessment

Yes by source inspection: current main blocks self-scoped `runs` in `assertCronSelfRemoveScope`, and the regression table still expects that denial. The existing behavior for `status` and self-filtered `list` is also covered in source and tests.

## Solution Assessment

Unclear: current-job-only `runs` is a plausible narrow implementation if maintainers accept it, but broad job inventory or run diagnostics needs an explicit isolation boundary before code changes.
