## Summary

Keep open: current main still finalizes short Telegram partial-stream replies by editing the existing preview message, and there is no URL-aware fresh-send policy or config surface for link-preview reliability. This is a real Telegram UX gap, but it needs maintainer product judgment before changing delivery semantics.

## What This Changes

The issue requests Telegram URL-containing final replies to avoid unreliable edited-message link previews, likely by fresh-sending those finals or adding a URL-final delivery setting.

## Best Possible Solution

Maintainers should choose a Telegram URL-final delivery policy, implement it narrowly in the draft/finalization path while preserving linkPreview: false, and require real Telegram proof with redacted chat/account details.

## Reproduction Assessment

No. I did not establish a high-confidence live Telegram preview-card reproduction. Source inspection does confirm that partial-mode final replies can introduce or finalize URLs through editMessageText rather than a fresh sendMessage.

## Solution Assessment

Unclear. Fresh-sending URL-containing finals is plausible, but the maintainable fix needs a maintainer choice between changing partial-mode defaults, adding an opt-in setting, or recommending existing progress/off modes for final-only delivery.
