## Summary

Keep this PR open for maintainer review: the idea is useful, but the branch is stale against current main and implements Defuddle in the old utility-owned extraction path instead of the current plugin-owned web content extractor path. It also adds a new untrusted-HTML parser dependency at the root without the plugin dependency ownership metadata expected for this surface.

## What This Changes

The PR adds Defuddle-first HTML extraction for `web_fetch`, keeps Readability as fallback, updates related docs/config copy/tests/scripts, and adds the `defuddle` dependency.

## Best Possible Solution

Implement Defuddle through the `webContentExtractors` plugin contract or a reviewed evolution of `web-readability`, keep parser dependencies with the owning plugin and dependency metadata, then update tests/docs/changelog after maintainer approval of the default order.

## Reproduction Assessment

Not applicable as a feature PR. The blocking issue is high-confidence from static source inspection: current main routes readable HTML extraction through plugin extractors, while this PR adds Defuddle to the old utility-owned path.

## Solution Assessment

No. The maintainable path is to expose Defuddle as a web content extractor plugin or within the existing web-readability plugin, with dependency ownership and ordering tests on that contract.
