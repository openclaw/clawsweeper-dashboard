## Summary

Keep open for maintainer handling. Earlier review feedback appears partly addressed in the PR head, including extractor tagging/test assertions and `useAsync: false`, but current `main` has since moved readable HTML extraction behind the plugin `webContentExtractors` contract. This branch still adds Defuddle to the old agent utility path and root dependencies, so it needs rework plus maintainer approval for default extractor behavior and dependency scope.

## What This Changes

This PR adds Defuddle as the primary `web_fetch` HTML extractor with Readability fallback, updates extractor tagging/config/docs/tests/scripts, and adds the `defuddle` dependency.

## Best Possible Solution

Implement Defuddle through the existing `webContentExtractors` contract as a maintainer-approved extractor plugin, keep the `defuddle` dependency scoped to that plugin, set `autoDetectOrder` ahead of Readability only if maintainers approve it as the default, preserve `useAsync: false`, and align docs/config/help/tests with the plugin-owned extractor model.

## Reproduction Assessment

Not applicable as a feature PR, but the integration mismatch is high-confidence from static inspection: current `web_fetch` calls `src/web-fetch/content-extractors.runtime.ts`, while the PR adds Defuddle to `src/agents/tools/web-fetch-utils.ts`, which is no longer the active readable extractor entry point.

## Solution Assessment

No, this is not the best way to solve the issue as currently proposed. The maintainable path is a web-content extractor plugin using the current manifest/runtime contract, not a root dependency and old utility-path implementation.
