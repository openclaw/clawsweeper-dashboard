## Summary

Keep open. The Defuddle idea may be useful, and earlier review comments appear to have been addressed in the PR discussion, but current main now routes HTML extraction through plugin-owned web-content extractors while this patch still adds Defuddle to the old utility path and root dependency surface.

## What This Changes

This PR adds Defuddle as the primary `web_fetch` HTML extractor with Readability fallback, updating extractor tagging, config/help/docs/tests/scripts, and the dependency lockfile.

## Best Possible Solution

Implement Defuddle through the `webContentExtractors` plugin contract, scope the dependency to the extractor plugin package, and only put it ahead of Readability after explicit maintainer approval of the default behavior.

## Reproduction Assessment

Not applicable as a feature PR, but the integration mismatch is high-confidence from static inspection: current `web_fetch` calls the plugin-backed extractor runtime, while the PR adds Defuddle to the old utility path.

## Solution Assessment

No. The maintainable path is a web-content extractor plugin using the existing manifest/runtime contract, not a root dependency and legacy utility-path implementation.
