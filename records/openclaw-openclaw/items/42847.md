## Summary

Keep open: the token-choice concern is still source-reproducible on current main, but this PR is not mergeable as-is because it edits retired Slack action files, lacks real Slack proof, and leaves Slack read credential policy/security unresolved.

## What This Changes

The PR changes the old Slack action handler and tests so reads prefer `botToken` when `userTokenReadOnly` is false, with `userToken` fallback only when no bot token exists.

## Best Possible Solution

Decide Slack read credential policy explicitly, then update the active `extensions/slack` runtime, focused tests, and docs with either a denied-token fallback or an explicit opt-in/authorization boundary.

## Reproduction Assessment

Yes for the token-choice path by source inspection: current main's active Slack runtime returns `userToken ?? botToken` for reads and tests assert user-token precedence. No live Slack `missing_scope` reproduction was established, and the PR body says live Slack API calls were not verified.

## Solution Assessment

No. The diff edits removed `src/agents/tools` Slack files and switches to bot-token-first reads without an active-runtime port, denied-token fallback, explicit authorization or opt-in policy, or real Slack proof.
