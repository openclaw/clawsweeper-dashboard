## Summary

Keep open: the branch targets retired Slack action files while current main uses the Slack plugin runtime, and the proposed credential priority change needs explicit security/product judgment before it is ported.

## What This Changes

The PR changes Slack action token selection in `src/agents/tools/slack-actions.ts` so reads prefer `botToken` when `userTokenReadOnly` is false, and adds two selector tests.

## Best Possible Solution

Decide the Slack credential policy explicitly, then implement it in `extensions/slack/src/action-runtime.ts` with active tests and docs; bot-token-first reads should include an authorization boundary, explicit opt-in, or permission-denied user-token fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows the active runtime still picks `userToken ?? botToken` for reads and active tests assert that selector behavior; I did not run live Slack API calls.

## Solution Assessment

No. This branch changes a retired path and broadens Slack read authority; the maintainable fix is a policy-approved port to the active Slack plugin runtime with matching coverage and docs.

  - body: Current `main` no longer uses `src/agents/tools/slack-actions.ts`; Slack actions execute through `extensions/slack/src/action-runtime.ts`. As written, this branch will not affect the live handler after it is rebased.

- **[P2] Retry or preserve user-token reads after bot denials:** `src/agents/tools/slack-actions.ts:131`
  - body: When both tokens are configured, the new selector falls back to the user token only if the bot token is missing. If Slack denies the bot token for a private channel, file, or missing scope while the user token can read it, reads that previously succeeded will fail.
  - confidence: 0.84
- **[P2] Gate broader bot-token reads explicitly:** `src/agents/tools/slack-actions.ts:131`
  - body: Read actions accept caller-supplied channel, message, thread, and file identifiers. Preferring the bot token for those reads can fetch data under broader bot scopes unless the runtime adds an explicit opt-in or authorization boundary.
  - confidence: 0.82
