## Summary

Keep open as the canonical follow-up for the closed Slack token-selection issue, but this branch is not merge-ready: current main still has the source-level token path, while the PR is conflicting, edits retired files, lacks live Slack proof, and raises credential-policy/security questions.

## What This Changes

The branch modifies the retired Slack action handler so reads prefer the bot token when `userTokenReadOnly: false`, plus two token-selection unit tests in the retired test file.

## Best Possible Solution

Decide the Slack read credential policy explicitly; if bot-token-first reads are desired, implement it in `extensions/slack` with an opt-in or authorization boundary, denied-token fallback, docs, focused tests, and redacted live Slack proof.

## Reproduction Assessment

Yes for the source-level token-choice path: current main returns `userToken ?? botToken` for reads and active tests assert user-token precedence. No live Slack `missing_scope` reproduction or after-fix real Slack proof was established.

## Solution Assessment

No. This PR edits retired files and changes Slack credential policy without an active-runtime port, denied-token fallback, explicit authorization or opt-in boundary, or real Slack proof.
