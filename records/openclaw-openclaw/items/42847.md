## Summary

Keep this PR open, but do not merge it as-is. It tracks the remaining Slack token-selection request that #42752 was closed into, but the branch is stale: it changes the retired `src/agents/tools` Slack action path, while current `main` runs the Slack action handler from `extensions/slack`; the proposed policy also needs maintainer security/product judgment before bot-token-first reads become default.

## What This Changes

The PR changes the old Slack action handler so `userTokenReadOnly: false` makes read operations prefer `botToken`, and adds two selector tests for that old path.

## Best Possible Solution

Decide the Slack credential policy explicitly, then implement the chosen behavior in `extensions/slack/src/action-runtime.ts` with active tests and docs; bot-token-first reads should include an authorization boundary, explicit opt-in, or permission-denied user-token fallback before becoming default behavior.

## Reproduction Assessment

Yes. Source inspection on current `main` shows the active Slack action runtime still returns `userToken ?? botToken` for reads and active tests assert that selector behavior; I did not need live Slack tokens for that proof.

## Solution Assessment

No. The branch addresses the symptom in a retired path and changes Slack read authority; the maintainable path is a policy-approved port to the active Slack plugin runtime with matching tests and documentation.
