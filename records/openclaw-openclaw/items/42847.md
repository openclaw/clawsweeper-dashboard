## Summary

Keep this PR open, but do not merge it as-is. Current main has moved Slack action handling into the bundled Slack plugin runtime, while this branch still patches the retired core tool path, and the proposed bot-token-first read policy needs explicit maintainer security/product judgment.

## What This Changes

The PR changes Slack action read-token selection so `userTokenReadOnly: false` prefers `botToken` for reads and adds two selector tests.

## Best Possible Solution

Decide the Slack credential policy explicitly, then implement it in `extensions/slack/src/action-runtime.ts` with active regression tests and docs; if bot-token-first reads are desired, add opt-in or authorization/fallback semantics instead of silently broadening read authority.

## Reproduction Assessment

Yes. Source inspection on current main shows that with both tokens and `userTokenReadOnly: false`, the active Slack action runtime still returns the user token for reads and passes that read option into `readMessages` and related read actions.

## Solution Assessment

No. The branch addresses the reported symptom in a retired path and changes credential priority without an explicit broader-read authorization boundary or denied-bot fallback; the maintainable fix needs a policy-approved port to the active Slack plugin runtime.

- **[P2] Port the selector to the active Slack runtime:** `src/agents/tools/slack-actions.ts:131`
  - body: This PR still changes `src/agents/tools/slack-actions.ts`, but current main no longer uses that Slack action runtime; Slack actions now execute through `extensions/slack/src/action-runtime.ts`. As written, the branch will not fix current Slack reads after rebase.
  - confidence: 0.96
- **[P2] Preserve user-token fallback when bot reads are denied:** `src/agents/tools/slack-actions.ts:131`
  - body: When `userTokenReadOnly` is `false`, the new selector tries the bot token first whenever it exists, and the read paths do not retry with the user token on Slack permission errors. Private channels or files readable only by the user token can regress from succeeding to failing.
  - confidence: 0.84
- **[P2] Gate broader bot-token reads explicitly:** `src/agents/tools/slack-actions.ts:131`
  - body: Switching reads to the bot token can make tool-supplied channel and file reads run under broader bot scopes without checking that the requester could access the target. Keep least-privilege reads or add an explicit opt-in and authorization boundary before changing this default.
  - confidence: 0.8
