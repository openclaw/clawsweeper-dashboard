## Summary

Keep open. The PR targets retired Slack action files, while current main runs Slack actions through `extensions/slack`; the intended read-token policy is not implemented there and still needs maintainer judgment because it can broaden Slack read authority.

## What This Changes

The PR changes Slack read-token selection so `userTokenReadOnly: false` makes reads prefer the bot token and adds old token-selection unit tests.

## Best Possible Solution

Decide the Slack read credential policy explicitly; if bot-token-first reads are desired, implement it in the active Slack plugin runtime with an opt-in or authorization boundary, denied-token fallback behavior, docs, focused tests, and redacted live Slack proof.

## Reproduction Assessment

Yes for the source-level token-choice path: current main returns `userToken ?? botToken` for reads and active tests assert user-token precedence. No live Slack `missing_scope` reproduction or after-fix real Slack proof was established.

## Solution Assessment

No. This PR edits removed files and changes Slack credential policy without an active-runtime port, denied-token fallback, explicit authorization or opt-in boundary, or real Slack proof.
