- `P2`: This is a normal-priority Slack credential-routing improvement with limited surface but real compatibility and security implications.
- `merge-risk: 🚨 compatibility`: Changing read credential priority can break existing setups where the configured user token can read channels that the bot token cannot.

## Summary

Keep open, but the current patch is not merge-ready: it changes retired Slack action files, while current main runs Slack actions from `extensions/slack`, and the proposed bot-token-first read policy needs explicit security/compatibility review plus real Slack proof.

## What This Changes

The branch changes Slack action read-token selection so reads prefer `botToken` when `userTokenReadOnly` is false and adds two token-selection unit tests in the retired `src/agents/tools` Slack action test file.

## Best Possible Solution

Decide the Slack read credential policy explicitly; if bot-token-first reads are desired, implement it in `extensions/slack` with an opt-in or authorization boundary, denied-token fallback, docs, focused tests, and redacted live Slack proof.

## Reproduction Assessment

Yes for the source-level token-choice path: current main still returns `userToken ?? botToken` for reads and active tests assert user-token precedence. No live Slack `missing_scope` reproduction or after-fix Slack proof was established.

## Solution Assessment
