## Summary

Keep this PR open, but it is not merge-ready. Current main still has the reported read-token behavior in the active Slack plugin runtime, while this branch edits the retired `src/agents/tools/slack-actions.ts` path and introduces a bot-token-first read policy that needs explicit maintainer product/security approval before it is ported.

## What This Changes

The PR changes the old Slack action token selector so reads prefer the bot token when `userTokenReadOnly` is false, and adds two tests for that old selector path.

## Best Possible Solution

Decide the Slack token policy explicitly, then implement it in `extensions/slack/src/action-runtime.ts` with active regression tests and Slack docs. If bot-token-first reads are desired, add explicit opt-in or authorization/fallback semantics; otherwise keep least-privilege user-token reads and document the workaround from #42752.

## Reproduction Assessment

Yes. A focused current-main check is enough: configure Slack with both tokens and `userTokenReadOnly: false`, call a read action such as `readMessages`, and the active selector still chooses the user token override. I verified this through source and tests; no tests were run because the review was read-only.

## Solution Assessment

No. The proposed direction addresses the reported symptom, but this branch changes a retired path and changes credential priority without bot-denied fallback or an explicit broader-read authorization boundary. The safer route is a maintainer-approved port to the active Slack plugin runtime.
