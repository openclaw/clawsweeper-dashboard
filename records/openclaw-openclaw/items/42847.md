## Summary

Keep this PR open, but do not merge it as-is. Current main still has the reported user-token-first read behavior in the active Slack plugin runtime, while this branch targets the retired `src/agents/tools/slack-actions.*` path and changes Slack read authority without a resolved credential-policy/security decision.

## What This Changes

The PR changes Slack read-token selection to prefer `botToken` when `userTokenReadOnly` is false and adds two selector tests under the old `src/agents/tools` Slack action path.

## Best Possible Solution

Decide the Slack credential policy explicitly, then implement the chosen behavior in `extensions/slack/src/action-runtime.ts` with active tests and docs; bot-token-first reads should have an authorization boundary, explicit opt-in, or denied-bot fallback before becoming default behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the active Slack action runtime returns `userToken ?? botToken` for reads and the active tests assert that behavior when both tokens are configured; no live Slack API call was needed for that selector proof.

## Solution Assessment

No. The branch addresses the symptom in a retired path and silently changes read authority; the maintainable fix needs a policy-approved port to the active Slack plugin runtime with matching tests and docs.

  - body: When `userTokenReadOnly` is false, the new selector falls back to the user token only when no bot token exists. If Slack denies the bot token for a private channel, file, or scope while the user token can read it, reads that previously succeeded will fail without retry.

  - body: Read actions accept tool-supplied channel, thread, message, and file identifiers. Switching those reads to the bot token can fetch data under broader bot scopes unless the runtime adds an explicit opt-in or authorization boundary.
  - confidence: 0.8
