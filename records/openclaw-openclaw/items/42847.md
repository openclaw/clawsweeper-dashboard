## Summary

Keep open for human review rather than cleanup: current main still has the reported token-choice behavior, but this branch touches retired files and the proposed credential precedence has live-proof and security/policy blockers.

## What This Changes

The PR changes the old `src/agents/tools` Slack action handler and tests so read actions prefer `botToken` when `userTokenReadOnly` is false, falling back to `userToken` only when no bot token exists.

## Best Possible Solution

Decide the Slack read credential policy explicitly, then update the active `extensions/slack` runtime, focused tests, and docs with either a denied-token fallback or an explicit opt-in/authorization boundary.

## Reproduction Assessment

Yes for the token-choice path by source inspection: current main's active Slack runtime returns `userToken ?? botToken` for reads and active tests assert user-token precedence. No live Slack `missing_scope` reproduction was established; the PR body says real Slack API calls were not verified.

## Solution Assessment

No. This PR edits retired files and proposes bot-token-first reads without an active-runtime port, denied-token fallback, explicit authorization policy, or real Slack proof.
