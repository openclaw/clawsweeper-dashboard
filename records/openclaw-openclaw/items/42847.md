## Summary

Keep open, but the patch is not mergeable as-is: it changes retired Slack action files, while current main uses the Slack extension runtime and still asserts user-token-first reads. The proposed bot-token-first behavior also needs credential-policy/security judgment and real Slack proof before merge.

## What This Changes

The PR changes the old Slack action handler and tests so Slack reads prefer the bot token when `userTokenReadOnly` is false.

## Best Possible Solution

Decide the Slack credential policy explicitly, then update the active Slack extension runtime, tests, and docs with an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes for the token-choice path by source inspection: current main's active Slack runtime returns `userToken ?? botToken` for reads and active tests assert user-token precedence. The live Slack `missing_scope` scenario was not run in this read-only review.

## Solution Assessment

No. The PR edits retired files and changes Slack credential precedence without an active-runtime port, denied-token fallback, explicit authorization policy, or live Slack proof.

  - confidence: 0.83
