## Summary

Keep open for human review: current main has not implemented the PR’s intended token-choice change, but the patch is not correct as submitted because it edits the removed Slack action handler and raises unresolved Slack credential-policy/security questions.

## What This Changes

The PR changes Slack read-token selection and tests so read operations prefer `botToken` when `userTokenReadOnly` is false, falling back to `userToken` only when no bot token exists.

## Best Possible Solution

Decide the Slack read credential policy explicitly; if bot-token-first reads are desired, implement it in `extensions/slack` with an authorization or opt-in boundary, denied-token fallback behavior, docs, focused tests, and redacted live Slack proof.

## Reproduction Assessment

Yes for the source-level token-choice path: current main’s active Slack runtime returns `userToken ?? botToken` for reads and active tests assert that behavior. No live Slack `missing_scope` reproduction was established; the PR body says live Slack API calls were not verified.

## Solution Assessment

No. This PR edits the retired handler and changes Slack credential policy without an active-runtime port, denied-token fallback, explicit authorization or opt-in boundary, or real Slack proof.
