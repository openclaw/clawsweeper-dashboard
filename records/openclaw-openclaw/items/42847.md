The PR changes Slack read-token selection so `userTokenReadOnly: false` prefers `botToken` for read operations and adds two legacy Slack action unit tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve a denied-token fallback for reads:** `src/agents/tools/slack-actions.ts:131`
  - body: When `userTokenReadOnly` is `false`, this always tries `botToken` first and never retries `userToken` on Slack permission failures. Existing setups where a user token can read a private channel but the bot cannot would regress from a successful read to a failed read.
  - confidence: 0.84
- **[P1] Do not broaden Slack read authority by default:** `src/agents/tools/slack-actions.ts:131`
  - body: Read params can name channel, thread, message, and file identifiers, but this switches reads to the broader bot token whenever user writes are allowed. Without an explicit opt-in or authorization check, that can expose Slack data outside the configured user token's visibility.
  - confidence: 0.82
- **[P2] Port the change to the active Slack runtime:** `src/agents/tools/slack-actions.ts:131`
  - body: Current main no longer has `src/agents/tools/slack-actions.ts`; Slack action execution lives in `extensions/slack/src/action-runtime.ts`. This branch is conflicting and will not update live Slack behavior until the change is ported to the extension-owned runtime and tests.
  - confidence: 0.94
