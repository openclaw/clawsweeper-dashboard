## Summary

Keep open. The PR targets a plausible Slack credential-policy problem, but the patch is not mergeable as-is because it edits the retired Slack action handler, lacks real Slack behavior proof, and changes read authority in a way that needs maintainer policy/security judgment.

## What This Changes

The PR changes Slack read-token selection and old handler tests so reads prefer the bot token when `userTokenReadOnly` is false.

## Best Possible Solution

Decide the Slack credential policy explicitly, then update the active Slack extension runtime, tests, and docs with an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes, by source inspection for the token-choice path: current main’s active Slack runtime returns `userToken ?? botToken` for reads and active tests assert user-token precedence. The live Slack `missing_scope` scenario was not run.

## Solution Assessment

No. The PR edits retired files and changes credential precedence without an active-runtime port, denied-token fallback, explicit authorization policy, or live Slack proof.

  - confidence: 0.83
