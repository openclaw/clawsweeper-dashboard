## Summary

Keep open, but do not merge as-is: the PR targets the retired Slack action handler while current main runs Slack actions from the plugin runtime, and the proposed bot-token read policy still needs security/product review plus real Slack proof.

## What This Changes

The PR changes the retired Slack action handler and its tests so reads prefer `botToken` when `userTokenReadOnly` is false and fall back to `userToken` only when no bot token exists.

## Best Possible Solution

Decide the Slack read credential policy explicitly; if bot-token-first reads are desired, implement it in the active Slack plugin runtime with an opt-in or authorization boundary, denied-token fallback behavior, docs, focused tests, and redacted live Slack proof.

## Reproduction Assessment

Yes for the source-level token-choice path: current main returns `userToken ?? botToken` for reads and active tests assert user-token precedence. No live Slack `missing_scope` reproduction was established.

## Solution Assessment

No. This PR edits removed files and changes Slack credential policy without an active-runtime port, denied-token fallback, explicit authorization or opt-in boundary, or real Slack proof.
