## Summary

Keep open: current main still has the reported Slack read-token behavior, but this PR is not mergeable as-is because it edits retired Slack action files and makes a security-relevant bot-token-first policy change without an active-runtime port, fallback/opt-in, or real Slack proof.

## What This Changes

The branch changes Slack read token selection in the old `src/agents/tools/slack-actions.ts` handler to prefer `botToken` when `userTokenReadOnly` is false and adds two old-handler token-selection tests.

## Best Possible Solution

Decide the Slack credential policy explicitly, then implement approved behavior in `extensions/slack/src/action-runtime.ts` with active tests, docs, and an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes, by source inspection. Current main’s active Slack runtime still returns `userToken ?? botToken` for reads, and the active token-selection test expects the user token when both tokens are configured.

## Solution Assessment

No. The PR edits retired files and makes a security-relevant bot-token-first policy change without an active-runtime port, explicit authorization boundary, opt-in, or denied-token fallback.

  - confidence: 0.83
