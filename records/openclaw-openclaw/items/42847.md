## Summary

Keep open. Current main still has the reported Slack read-token behavior in the active plugin runtime, but this PR is not merge-ready because it patches retired `src/agents/tools` files and changes read operations to prefer a potentially broader bot token without a settled authorization/fallback design.

## What This Changes

The PR changes the old Slack action token selector so reads prefer the bot token when `userTokenReadOnly` is false, and adds two old-path tests for bot-token read preference and user-token fallback.

## Best Possible Solution

Decide the Slack token policy explicitly, then implement it in `extensions/slack/src/action-runtime.ts` with matching active tests and Slack docs. If bot-token-first reads are desired for `userTokenReadOnly: false`, add clear authorization/fallback semantics; if user-token-first reads are the intended least-privilege model, document the #42752 workaround and close the PR manually.

## Reproduction Assessment

Yes. A focused current-main check is enough: configure Slack with both tokens and `userTokenReadOnly: false`, call `readMessages`, and the active selector still sends the user token override. I verified the source and tests; no tests were run because this review is read-only.

## Solution Assessment

No. The proposed direction may address #42752, but this branch edits retired files and changes credential priority without resolving fallback or broader-read authorization. The safer solution is a maintainer-approved port to the active Slack plugin runtime with regression tests and docs.
