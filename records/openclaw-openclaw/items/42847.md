## Summary

Keep open. The PR is not mergeable as-is because it edits retired Slack action files while current main’s active Slack runtime still uses user-token-first reads, and the proposed bot-token-first read policy needs maintainer security/product judgment plus real Slack behavior proof.

## What This Changes

The PR changes Slack action read-token selection in `src/agents/tools/slack-actions.ts` to prefer `botToken` when `userTokenReadOnly` is false and adds two token-selection tests.

## Best Possible Solution

Decide Slack read credential policy explicitly, then implement it in the active `extensions/slack` runtime with tests, docs, live proof, and either authorization/opt-in or permission-denied fallback.

## Reproduction Assessment

Yes, by source inspection. Current main’s active Slack action runtime returns `userToken ?? botToken` for reads, and the active token-selection test expects the user token when both tokens are configured; no live Slack workspace probe was run in this read-only review.

## Solution Assessment

No. The branch edits retired `src/agents/tools` files and changes credential policy without updating the active extension runtime, adding an explicit opt-in or authorization boundary, or handling bot-token permission denials.

  - confidence: 0.83
