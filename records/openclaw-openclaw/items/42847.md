## Summary

Keep open: the underlying Slack token-selection problem is source-reproducible, but this PR is not currently mergeable or correct because it edits the retired Slack action handler while current `main` runs the Slack extension runtime, and the proposed bot-token-first read policy still needs security/product judgment plus real Slack proof.

## What This Changes

The PR changes the old `src/agents/tools` Slack action handler and tests so Slack reads prefer the bot token when `userTokenReadOnly` is false.

## Best Possible Solution

Decide the Slack read credential policy explicitly, then update the active `extensions/slack` runtime, focused tests, and docs with an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes for the token-choice path by source inspection: current `main` returns `userToken ?? botToken` for Slack reads and active tests assert user-token precedence. The live Slack `missing_scope` scenario was not run in this read-only review.

## Solution Assessment

No. This PR edits retired files and changes Slack credential precedence without an active-runtime port, denied-token fallback, explicit authorization policy, or live Slack proof.
