## Summary

Keep open, but the proposed patch is not mergeable as-is: it changes a removed legacy Slack handler, while current Slack actions run through `extensions/slack`, and the requested bot-token-first read policy still needs explicit security/product review plus real Slack proof.

## What This Changes

This PR changes the legacy Slack action handler and tests so Slack reads prefer `botToken` when `userTokenReadOnly` is false, falling back to `userToken` only when `botToken` is absent.

## Best Possible Solution

Decide Slack read credential policy explicitly, then update the active `extensions/slack` runtime, tests, and docs with an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes for the token-choice path by source inspection: current main still returns `userToken ?? botToken` for Slack reads and active tests assert user-token precedence. The live Slack `missing_scope` scenario was not run in this read-only review.

## Solution Assessment

No. This PR edits retired files and changes Slack credential precedence without an active-runtime port, denied-token fallback, explicit authorization policy, or live Slack proof.
