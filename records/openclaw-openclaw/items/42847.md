## Summary

Keep open for maintainer judgment. The linked Slack token-selection problem is still visible in current main by source inspection, but this PR edits a retired handler and its proposed bot-token-first policy has unresolved security and fallback implications.

## What This Changes

The PR changes the old Slack action handler so read operations prefer `botToken` when `userTokenReadOnly` is false, and adds two token-selection tests for that old handler.

## Best Possible Solution

Decide the Slack credential policy explicitly, then implement any approved change in `extensions/slack/src/action-runtime.ts` with active tests, docs, and either an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes, by source inspection. Current main’s active Slack action runtime still returns `userToken ?? botToken` for reads, and the active token-selection test expects the user token when both tokens are configured.

## Solution Assessment

No. The PR edits retired files and makes a security-relevant bot-token-first policy change without an active-runtime port, explicit authorization boundary, opt-in, or denied-token fallback.
