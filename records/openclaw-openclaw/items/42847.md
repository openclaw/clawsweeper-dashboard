## Summary

Keep open, but the patch is not merge-ready: it targets retired Slack handler files, while current main’s active Slack extension runtime still uses user-token read precedence and the proposed bot-token-first policy has unresolved security and proof gaps.

## What This Changes

This PR changes Slack read-token selection so `userTokenReadOnly: false` makes read operations prefer the bot token, and adds two token-selection tests in the old Slack handler test file.

## Best Possible Solution

Decide the Slack credential policy explicitly, then update the active Slack extension runtime, tests, and docs with an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes, by source inspection for the token-choice path: current main’s active Slack runtime returns `userToken ?? botToken` for reads and active tests assert user-token precedence. The live Slack `missing_scope` scenario was not run.

## Solution Assessment

No. The PR edits retired files and changes Slack credential precedence without an active-runtime port, denied-token fallback, explicit authorization policy, or live Slack proof.
