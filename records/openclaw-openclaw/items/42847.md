## Summary

Keep open; the patch is not mergeable as-is because it changes a retired Slack action path on current main and broadens Slack read authority without an explicit authorization boundary.

## What This Changes

The PR changes Slack read-token selection in `src/agents/tools/slack-actions.ts` so reads prefer `botToken` when `userTokenReadOnly` is false and adds two selector tests.

## Best Possible Solution

Decide the Slack read-authority policy explicitly, then implement it in `extensions/slack/src/action-runtime.ts` with active regression tests and docs; if bot-token-first reads are desired, add explicit opt-in or authorization/fallback semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows the active Slack runtime still returns `userToken ?? botToken` for read operations even when `userTokenReadOnly` is false, and those read opts are passed to `readMessages`, `downloadFile`, pins, reactions, member info, and emoji reads.

## Solution Assessment

No. The PR changes a retired path and silently changes credential priority; a maintainable fix needs a policy-approved port to the active Slack plugin runtime with an explicit authorization or opt-in boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Port the selector to the active Slack runtime:** `src/agents/tools/slack-actions.ts:131`
  - body: This PR changes `src/agents/tools/slack-actions.ts`, but current main routes Slack actions through `extensions/slack/src/action-runtime.ts`. As written, the branch will not fix Slack reads after rebase because the edited runtime is no longer tracked on current main.
  - confidence: 0.96
- **[P2] Preserve fallback when bot reads are denied:** `src/agents/tools/slack-actions.ts:131`
  - body: When `userTokenReadOnly` is false, the new selector tries the bot token first whenever it exists, and the read paths do not retry with the user token on Slack permission errors. Private channels or files readable only by the user token can regress from succeeding to failing.
  - confidence: 0.84
- **[P2] Gate broader bot-token reads explicitly:** `src/agents/tools/slack-actions.ts:131`
  - body: Switching reads to the bot token can make tool-supplied channel and file reads run under broader bot scopes without checking that the requester could access the target. Keep least-privilege reads or add an explicit opt-in and authorization boundary before changing this default.
  - confidence: 0.8
