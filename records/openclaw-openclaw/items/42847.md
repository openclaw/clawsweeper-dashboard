## Summary

Keep open, but do not merge as-is. The branch is now aimed at retired Slack handler files, while the active Slack runtime still lives under `extensions/slack`; the proposed bot-token-first read policy also needs explicit security/product judgment and real Slack behavior proof.

## What This Changes

The PR changes the old Slack action handler and tests so read operations prefer the bot token when `userTokenReadOnly` is false.

## Best Possible Solution

Decide the Slack credential policy explicitly, then update the active Slack extension runtime, tests, and docs with an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes, by source inspection for the token-choice path: current main's active Slack runtime returns `userToken ?? botToken` for reads and active tests assert user-token precedence. The live Slack `missing_scope` scenario was not run in this read-only review.

## Solution Assessment

No. The PR edits retired files and changes credential precedence without an active-runtime port, denied-token fallback, explicit authorization policy, or live Slack proof.
