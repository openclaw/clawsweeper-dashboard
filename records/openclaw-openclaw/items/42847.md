## Summary

Keep open for maintainer decision, but the patch is not mergeable as-is: it changes retired Slack action files, conflicts with current docs/tests, and broadens Slack read authority without an explicit policy boundary or live proof.

## What This Changes

The PR changes the old Slack action handler and tests so reads prefer the bot token when `userTokenReadOnly` is false.

## Best Possible Solution

Decide the Slack credential policy explicitly, then update the active Slack extension runtime, tests, and docs with either an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes, by source inspection for token choice: current main's active Slack runtime returns `userToken ?? botToken` for reads and active tests assert user-token precedence. The live Slack `missing_scope` scenario was not run.

## Solution Assessment

No. The PR edits retired files and changes credential precedence without an active-runtime port, denied-token fallback, explicit authorization policy, or live Slack proof.
