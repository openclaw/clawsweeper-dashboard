## Summary

Keep open. The underlying Slack read-token behavior is still unresolved on current main, but this PR is not mergeable as-is because it edits retired files and changes Slack read authority without an explicit security/policy boundary or real Slack proof.

## What This Changes

The PR changes the old Slack action handler and its tests so read operations prefer `botToken` when `userTokenReadOnly` is false.

## Best Possible Solution

Decide the Slack credential policy explicitly, then update the active Slack extension runtime, tests, and docs with an authorization boundary, explicit opt-in, or permission-denied fallback.

## Reproduction Assessment

Yes, by source inspection. Current main still chooses `userToken ?? botToken` for reads, and the active token-selection tests expect the user token when both tokens are configured.

## Solution Assessment

No. The PR edits retired files and makes a bot-token-first read policy change without an active-runtime port, explicit authorization boundary, opt-in, or denied-token fallback.
