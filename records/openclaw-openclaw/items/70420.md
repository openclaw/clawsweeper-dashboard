## Summary

Keep open. Current main still separates channel/provider runtime health from persisted session rows, and the docs still do not explicitly warn that `openclaw sessions`, Gateway `sessions.list`, or agent `sessions_list` are not Discord socket-liveness checks, so the reported false-alarm path remains actionable as a narrow docs fix.

## What This Changes

Clarify or change Discord reconnect/session behavior so session-based checks do not falsely report a healthy Discord reconnect as disconnected.

## Best Possible Solution

Land a narrow docs clarification that session listing surfaces persisted conversation/session-store entries, not provider socket or channel liveness. Keep connectivity guidance pointed at `openclaw channels status --probe`, `openclaw status --deep`, `openclaw health --verbose`, and readiness/channel-status surfaces unless maintainers intentionally choose a broader cross-channel synthetic-session contract.

## Reproduction Assessment

Yes. The issue supplies runtime logs, and current-main source gives a high-confidence path: after a quiet Discord reconnect, runtime status can be connected while `sessions.list` still returns only persisted session-store rows that already exist.

## Solution Assessment

Yes. A docs-first fix is the narrowest maintainable solution because current code intentionally separates runtime liveness from session persistence; materializing heartbeat sessions would be a broader cross-channel behavior change.
