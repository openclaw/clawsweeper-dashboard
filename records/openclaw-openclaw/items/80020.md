## Summary

Keep open: the fix is in the right area, but the patch is incomplete because the new `explicitSetPaths` option is dropped before the exported write path reaches the low-level persistence helper, and the supplied proof only exercises that helper directly.

## What This Changes

The PR adds explicit config write paths to preserve caller-set values that equal runtime defaults, wires them from config set/patch operations into write options, and adds helper-level regression coverage.

## Best Possible Solution

Forward `explicitSetPaths` through the exported `writeConfigFile` path and any runtime-snapshot pre-projection that can drop those values, then prove the actual CLI/config write persists default-valued fields with regression coverage.

## Reproduction Assessment

Yes. The linked issue gives a concrete `config set`/`config patch` reproduction, and current source shows the same failure path through `createMergePatch(runtimeConfig, nextConfig)`; I did not run live writes because this was a read-only review.

## Solution Assessment

No. The helper approach is a good direction, but this patch is not complete until `explicitSetPaths` survives `replaceConfigFile` through the exported writer into `createConfigIO().writeConfigFile` and is proven through an end-to-end write path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Forward explicit paths through the public writer:** `src/config/io.ts:205`
  - body: `runConfigOperations` now sends `explicitSetPaths` through `replaceConfigFile`, but `replaceConfigFile` calls the exported `writeConfigFile` wrapper. That wrapper rebuilds the options for `createConfigIO().writeConfigFile` and only forwards `unsetPaths`/flags, so the new option never reaches `resolvePersistCandidateForWrite` on the actual CLI path; setting `dmPolicy`/`groupPolicy` to their defaults will still be dropped. Forward the option there and cover the exported wrapper or CLI write path.
