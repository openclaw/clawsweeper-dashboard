## Summary

Keep open because the latest report claims the inverted MiniMax status label still occurs on v2026.5.5, while current source, docs, tests, and the prior upstream contract all say OpenClaw should be rendering remaining quota as `% left`; this needs live/raw MiniMax payload verification before closing or changing parser semantics.

## What This Changes

The issue asks for the MiniMax usage percentage in `openclaw status` and status cards to be labeled according to whether the value is used quota or remaining quota.

## Best Possible Solution

Verify a redacted live `minimax-portal` `token_plan/remains` response against the MiniMax dashboard; if the endpoint now returns consumed values, update the MiniMax parser, docs, and focused regression tests, otherwise clarify the status wording or close this as a duplicate of the prior fixed MiniMax usage issue.

## Reproduction Assessment

No. Source inspection explains how the displayed line is formed, but the failing condition depends on the current MiniMax payload semantics, and this environment had no MiniMax credentials for a live redacted check.

## Solution Assessment

Unclear. The narrow fix, if the report is confirmed, belongs in `src/infra/provider-usage.fetch.minimax.ts` or the shared formatter, but the current docs/tests/upstream issue argue against changing labels or parser semantics without fresh provider evidence.
