## Summary

Keep open: current main still lacks the device-signature limiter, and this PR is a focused security fix, but the patch still resets the new bucket before final device admission and needs stronger WS-handler proof before merge.

## What This Changes

The PR adds device-handshake field bounds, device-identity shape prechecks, a device-signature auth-rate-limit scope in the WebSocket connect path, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Move the device-signature bucket reset to a point after successful device admission/registration, keep the schema bounds and pre-crypto gate, and require redacted WS-handler proof showing lockout after the threshold.

## Reproduction Assessment

Yes. Current main source shows caller-controlled device public keys reach deriveDeviceIdFromPublicKey and v3/v2 verifyDeviceSignature without a device-signature limiter; the PR body supplies a concrete WS flood PoC, but I did not run it in this read-only pass.

## Solution Assessment

No for the current patch. The layered cap/precheck/rate gate is the right direction, but resetting at line 915 before pairing/admission lets a valid shared-auth but unadmitted device clear the bucket before the handshake is rejected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Defer signature-bucket reset until admission:** `src/gateway/server/ws-connection/message-handler.ts:915`
  - body: The new limiter records a failure before crypto, but resets as soon as `authOk` is true. Pairing, metadata, role, and scope checks below can still reject before `setClient`, so a client with valid shared auth and an unadmitted or upgrade-requesting device can clear the bucket on every attempt and continue reaching signature verification. Reset only after successful device admission/registration, or skip reset on those rejection paths.
  - confidence: 0.87
