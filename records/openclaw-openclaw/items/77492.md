## Summary

Keep open: the updated head fixes the earlier PEM-before-limiter and auth-failure reset findings, but it still clears the device-signature bucket before pairing, role, scope, and metadata approval gates can reject the connection.

## What This Changes

The PR hardens pre-auth WebSocket device-signature handling with bounded device fields, shape pre-checks, a new per-IP signature rate-limit scope, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the layered bounds/pre-check/rate-limit approach, but clear the device-signature scope only after the connect path is actually admitted, including pairing and role/scope/metadata gates, with a regression for an auth-ok but pairing-required valid self-signature.

## Reproduction Assessment

Yes. Source inspection on current main shows a device-bearing connect can reach public-key derivation and v3/v2 signature verification before later auth and pairing checks; the PR body also includes a concrete PoC, though I did not run it here.

## Solution Assessment

No. The mitigation direction is right and the first two bypasses were addressed, but the reset still happens too early for pairing-required or upgrade-required failures after `authOk`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reset the signature bucket after pairing completes:** `src/gateway/server/ws-connection/message-handler.ts:884-885`
  - body: This reset runs as soon as shared/open/device-token auth succeeds, before `requirePairing` can reject an unpaired device or a role/scope/metadata upgrade. A client that can make `authOk` true and generate its own valid device signature can still fail pairing repeatedly while clearing the bucket each time, so move the reset after the connection is fully admitted and add that regression case.
