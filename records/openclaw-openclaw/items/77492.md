## Summary

Keep open: the mitigation direction is good, but the current PR head still leaves source-level bypasses in the pre-auth device-signature DoS defense. The new bucket is cleared before final device admission, and direct non-browser remote WS clients without configured `gateway.auth.rateLimit` do not get the new limiter by default.

## What This Changes

Adds Gateway device-handshake field bounds, device identity shape pre-checks, a per-IP device-signature rate-limit scope, a PoC script, regression tests, browser/control-UI expectation updates, and a changelog entry.

## Best Possible Solution

Keep the layered bounds and shape checks, but ensure every non-loopback pre-auth device-signature path has a default limiter, reset the signature bucket only after full connect admission, add direct-remote/no-Origin and shared-auth-pairing rejection regressions, and update the changelog credit.

## Reproduction Assessment

Yes. Current main source shows the device-id derivation and v3/v2 signature verification before auth admission, and the PR body includes a concrete PoC plus regression tests; I did not live-run them in this read-only pass.

## Solution Assessment

No. The bounds and pre-checks are the right shape, but the proposed limiter placement and optional direct-client limiter leave maintainable bypasses that should be closed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Limit direct remote signature handshakes by default:** `src/gateway/server/ws-connection/message-handler.ts:758`
  - body: This new gate only runs when `authRateLimiter` is present. For WS clients without an Origin header, `resolveHandshakeBrowserSecurityContext` selects the regular limiter, and `server.impl.ts` leaves that limiter undefined unless `gateway.auth.rateLimit` is configured. A direct remote client on a token-auth non-loopback gateway can still send valid-shape device handshakes and reach `deriveDeviceIdFromPublicKey` plus v3/v2 signature verification without this bucket, so use an always-created signature limiter for non-loopback device signatures or route this path through a default limiter.
  - confidence: 0.82
- **[P2] Reset the signature bucket after device admission:** `src/gateway/server/ws-connection/message-handler.ts:885`
  - body: The bucket is reset as soon as `authOk` is true, but pairing, metadata-upgrade, role-upgrade, and scope-upgrade gates can still reject before the connection is admitted. A client that passes shared auth while failing device admission can keep clearing the pessimistic signature attempt and continue forcing the crypto path, so defer this reset until after those gates pass and cover that case.
  - confidence: 0.9
- **[P3] Credit the contributor in the changelog:** `CHANGELOG.md:50`
  - body: This user-facing security fix is authored by a known human contributor, but the changelog entry only has `[AI]`. Repo policy asks contributor-facing entries to include the human credit on the same line, so add `Thanks @fede-kamel.`.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch reduces the pre-auth DoS surface but still leaves concrete limiter bypasses that should be fixed before merge.

Concerns:

- **[high] Direct remote clients can bypass the new bucket:** `src/gateway/server/ws-connection/message-handler.ts:758`
  - body: The new device-signature scope is checked only when `authRateLimiter` exists, but direct no-Origin WS handshakes use the optional regular limiter, which is absent unless `gateway.auth.rateLimit` is configured. That leaves the described unauthenticated remote CPU path open for exposed token/password deployments without explicit rate-limit config.
  - confidence: 0.82
- **[medium] Bucket clears before pairing admission:** `src/gateway/server/ws-connection/message-handler.ts:885`
  - body: `AUTH_RATE_LIMIT_SCOPE_DEVICE_SIGNATURE` is reset immediately after `authOk`, while later device admission checks can still reject the connection. Authenticated-but-unadmitted device handshakes can therefore clear the pessimistic attempt and continue reaching signature verification.
  - confidence: 0.9
