## Summary

Keep open: the PR addresses a valid gateway pre-auth CPU-DoS path, but the patch still clears the new signature bucket before final device admission and the external contributor has not supplied after-fix real gateway proof.

## What This Changes

The PR adds gateway device-signature field bounds, pre-crypto device key/signature shape checks, a per-IP device-signature rate-limit bucket, a PoC script, regression tests, browser/control-UI expectation updates, and a changelog entry.

## Best Possible Solution

Keep the layered bounds, shape checks, and pre-crypto gate, but move the signature-bucket reset after actual device admission, add a regression for shared-auth admission rejection, and require after-fix PoC output before merge.

## Reproduction Assessment

Yes. Current main source shows device-id derivation and v3/v2 signature verification before auth/pairing admission, and the PR includes a concrete PoC and focused regression tests; I did not live-run the gateway in this read-only pass.

## Solution Assessment

No. The mitigation shape is maintainable, but the reset point is still too early and the external PR still needs after-fix real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Defer signature-bucket reset until device admission:** `src/gateway/server/ws-connection/message-handler.ts:885`
