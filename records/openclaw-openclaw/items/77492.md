## Summary

Keep open: the layered hardening is pointed at a real current-main pre-auth crypto path, but the latest PR head still resets the new device-signature bucket after credential auth and before device pairing/admission. The supplied terminal proof exercises helper modules, not the after-fix WS handler path that owns the security boundary.

## What This Changes

The PR adds device public-key/signature bounds, device-identity shape pre-checks, a per-IP device-signature auth-rate-limit scope in the Gateway WS connect handler, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the schema bounds, shape pre-checks, and pre-crypto gate, but reset the device-signature bucket only after successful device admission/registration, then add after-fix WS/gateway proof.

## Reproduction Assessment

Yes, source-reproducible. Current main derives the device id and runs v3/v2 signature verification before final auth and pairing/access admission; the PR body also includes pre-fix PoC timing from a running gateway, but I did not run the live gateway in this read-only review.

## Solution Assessment

No. The bounds, shape pre-checks, and pre-crypto gate are the right direction, but the reset must move past final device admission/registration and the proof should exercise the real WS handler path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Defer signature-bucket reset until device admission:** `src/gateway/server/ws-connection/message-handler.ts:915`
  - body: The new bucket is reset as soon as credential auth succeeds, but the not-paired, metadata-upgrade, role-upgrade, and scope-upgrade branches below can still reject before `setClient` registers the connection. A client with valid shared auth but an unadmitted device can repeatedly clear the bucket and keep reaching signature verification, so move the reset until successful admission/registration or avoid resetting on these rejection paths.
  - confidence: 0.87
