## Summary

Keep open: the branch addresses a real current-main gateway security gap, but the patch still resets the new signature bucket before final device admission and the supplied proof does not exercise the patched WebSocket handler path.

## What This Changes

The PR adds device handshake length bounds, device-identity plausibility checks, a per-IP device-signature auth-rate-limit scope in the WebSocket connect path, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the layered defense, but keep the signature bucket consumed until the device/socket is fully admitted, and add WS-level after-fix proof plus a regression for the authOk-then-pairing-rejection path before merge.

## Reproduction Assessment

Yes. Current main source shows caller-controlled device public keys and signatures reach device-id derivation and v3/v2 crypto verification before a device-signature limiter; the PR also supplies a concrete flood PoC with measured pre-fix latency impact, though I did not live-run it here.

## Solution Assessment

No for the current patch. The schema bounds, shape checks, and pre-crypto gate are the right direction, but the reset should move past final device admission/registration or avoid clearing the bucket on later pairing/access rejection paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Defer signature-bucket reset until admission succeeds:** `src/gateway/server/ws-connection/message-handler.ts:915`
  - body: The new bucket resets as soon as `authOk` is true, but the not-paired, metadata-upgrade, role-upgrade, and scope-upgrade paths below can still reject before the socket is admitted. A caller with valid shared auth and an unadmitted device can therefore clear the bucket on every attempt and keep reaching signature verification, so reset only after successful device admission/registration or avoid resetting on those rejection paths.
  - confidence: 0.88
