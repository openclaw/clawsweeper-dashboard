## Summary

Keep open: the PR targets a real gateway pre-auth CPU DoS path, but it is not merge-ready because the new device-signature bucket is still reset before final device admission and the external contributor has not provided after-fix real gateway proof.

## What This Changes

Adds device-handshake field bounds, device key/signature shape pre-checks, a per-IP device-signature rate-limit scope, a PoC script, regression tests, browser/control-UI expectation updates, and a changelog entry.

## Best Possible Solution

Keep the layered bounds, shape checks, and per-IP signature limiter, but reset the signature bucket only after the device connection is fully admitted and add a regression for shared-auth success followed by pairing or upgrade rejection before merge.

## Reproduction Assessment

Yes. Current main source shows device-id derivation and v3/v2 signature verification before auth/pairing admission, and the PR includes a concrete PoC plus regression tests; I did not live-run them in this read-only pass.

## Solution Assessment

No. The mitigation shape is right, but the PR still clears the new bucket before final device admission and does not include after-fix real behavior proof from a real gateway setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Defer signature-bucket reset until final admission:** `src/gateway/server/ws-connection/message-handler.ts:885`
  - body: The new device-signature bucket is reset as soon as `authOk` is true, but this is still before not-paired, metadata-upgrade, role-upgrade, and scope-upgrade checks can reject the connect. A client that can satisfy shared auth while failing device admission can keep clearing the bucket and continue reaching the signature crypto path, so move this reset until after the device connect is actually admitted and cover that case.
  - confidence: 0.88
- **[P3] Credit the contributor in the changelog:** `CHANGELOG.md:72`
  - body: This user-facing security fix is from a known human contributor, but the changelog entry only has `[AI]`. Repo policy asks contributor-facing entries to include the human credit on the same line, so add the contributor credit before merge.
  - confidence: 0.9
