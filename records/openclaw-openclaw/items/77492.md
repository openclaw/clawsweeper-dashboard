## Summary

Keep open: the PR addresses a real source-reproducible gateway CPU-DoS path, but the new signature bucket is still reset before device pairing/access admission completes, and the supplied after-fix proof does not drive the changed real WS message-handler path.

## What This Changes

The PR adds gateway device public-key/signature bounds, device-identity shape pre-checks, a new per-IP device-signature rate-limit scope in the WS connect handler, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the layered schema/precheck/rate-limit mitigation, but reset the signature bucket only after successful device admission/registration or otherwise ensure admission rejections cannot clear it, then add after-fix WS/gateway proof.

## Reproduction Assessment

Yes, source-reproducible. Current main derives the device id and runs v3/v2 signature verification before final auth and pairing/access admission, and the PR body includes PoC timing from a running gateway, but I did not run the live gateway in this read-only review.

## Solution Assessment

No. The bounds, shape pre-checks, and pre-crypto gate are the right direction, but the reset should move past final device admission/registration and the proof should exercise the real WS handler path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Defer signature-bucket reset until device admission:** `src/gateway/server/ws-connection/message-handler.ts:884`
  - body: The new bucket is reset immediately after authOk, but pairing, metadata-upgrade, role-upgrade, and scope-upgrade checks below can still reject before setClient registers the socket.
