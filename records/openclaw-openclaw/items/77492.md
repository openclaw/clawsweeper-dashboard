## Summary

Keep open: this is a plausible security fix for the Gateway pre-auth device-signature CPU path, but the latest diff still resets the new signature bucket before final device admission and the supplied proof does not exercise the patched WS handler.

## What This Changes

The PR adds device public-key/signature bounds, device-identity shape pre-checks, a per-IP device-signature rate-limit scope in the Gateway WS connect handler, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the layered bounds and pre-crypto gate, move the device-signature bucket reset to successful handshake registration/admission, and require after-fix WS-level proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main derives device ids and runs v3/v2 signature verification before the auth decision, and the PR body provides a concrete WS flood PoC with pre-fix latency numbers, though I did not run a live gateway in this read-only pass.

## Solution Assessment

No, not yet. The schema bounds, shape pre-checks, and pre-crypto bucket are the right direction, but the bucket reset should happen only after final device admission/registration and the proof should drive the real WS handler path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Defer signature-bucket reset until admission:** `src/gateway/server/ws-connection/message-handler.ts:915`
  - body: The new bucket is reset immediately after `resolveConnectAuthDecision` returns `authOk`, but the handshake can still be rejected below for stale gateway auth generation, missing pairing, metadata upgrade, role upgrade, or scope upgrade before `setClient` registers the socket. A client with valid shared auth but an unadmitted or upgrade-pending device can repeatedly clear the bucket and keep reaching signature verification; move this reset to the successful registration/admission path.
  - confidence: 0.88

## Security Review
