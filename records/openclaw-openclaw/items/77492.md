## Summary

Keep open: this PR is the active fix path for https://github.com/openclaw/openclaw/issues/77979, but the current head still resets the new device-signature bucket before final device admission and the external real-behavior proof does not drive the changed WS handler path after the fix.

## What This Changes

The PR adds device handshake key/signature bounds, pre-crypto shape checks, a per-IP device-signature auth rate-limit scope in the gateway WS handler, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the layered mitigation, move the signature-bucket reset after successful device admission or registration, add regression coverage for shared-auth admission rejection, and require after-fix real WS/gateway proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main derives the submitted device public key and runs v3/v2 signature verification before final auth and pairing admission; I did not run the live gateway in this read-only review.

## Solution Assessment

No. The schema bounds, shape pre-checks, and pre-crypto gate are the right direction, but the reset should wait until final device admission and the proof should cover the real WS path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Defer the signature-bucket reset until admission:** `src/gateway/server/ws-connection/message-handler.ts:885`
  - body: The new bucket is reset as soon as authOk is true, but not-paired, metadata-upgrade, role-upgrade, and scope-upgrade checks can still reject before the socket is registered. A shared-authenticated but unadmitted device can keep clearing the bucket and continue reaching signature verification, so move the reset after successful device admission or registration and cover that path.
