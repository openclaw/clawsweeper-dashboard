## Summary

Keep open: the PR fixes the main pre-auth crypto-amplification path directionally, but the latest head still clears the new device-signature bucket before device pairing, metadata, role, and scope admission can reject the handshake.

## What This Changes

Adds Gateway device-handshake field bounds, device key/signature shape pre-checks, a per-IP device-signature rate-limit scope, a PoC script, regression tests, browser/control-ui test expectation updates, and a changelog entry.

## Best Possible Solution

Keep the layered bounds, shape pre-checks, and per-IP signature limiter, but clear the signature bucket only after the connect is fully admitted and add a shared-auth-but-not-paired regression before landing.

## Reproduction Assessment

Yes. Current main source shows device-id derivation and v3/v2 signature verification before auth/pairing admission, and the PR includes a concrete PoC plus targeted regression tests; I did not live-run them in this read-only pass.

## Solution Assessment

No. The mitigation shape is right, but resetting the new bucket after authOk rather than after final connect admission leaves a source-level bypass for authenticated-but-unpaired device handshakes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Defer signature-bucket reset until connect admission:** `src/gateway/server/ws-connection/message-handler.ts:885`
  - body: The new bucket is reset as soon as `authOk` is true, but this is still before the not-paired, metadata-upgrade, role-upgrade, and scope-upgrade gates can reject the connect. A client that can satisfy the shared auth step while failing device admission can keep clearing the bucket and continue forcing the signature crypto path, so move this reset until after those gates pass and cover that case.
