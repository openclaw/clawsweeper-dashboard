## Summary

Keep open; the PR addresses the main pre-auth crypto-cost path, but the latest head still resets the new device-signature bucket after `authOk` and before later pairing/metadata/role/scope gates can reject the connect attempt. The changelog entry also still lacks the required known-contributor credit.

## What This Changes

Adds Gateway device-handshake field bounds, input shape pre-checks, a per-IP device-signature rate-limit scope, a PoC script, regression tests, and a changelog entry for pre-auth signature CPU hardening.

## Best Possible Solution

Keep the layered bounds/pre-check/rate-limit mitigation, but only clear the signature bucket after the connect path is fully admitted and add a regression for auth-ok pairing-required rejection before landing.

## Reproduction Assessment

Yes, from source inspection and the PR's concrete PoC. Current main reaches device-id derivation and v3/v2 signature verification before full authorization; I did not live-run the PoC in this read-only pass.

## Solution Assessment

No. The mitigation direction is right, but the reset still happens too early; moving the reset after successful admission is the narrower maintainable fix.
