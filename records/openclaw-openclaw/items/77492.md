## Summary

Keep this PR open: it targets a real gateway pre-auth CPU-amplification bug and the layered direction is useful, but the current patch still has security-sensitive limiter semantics that can weaken the new bucket and the supplied proof does not exercise the patched WS handler path.

## What This Changes

The PR adds device public-key/signature bounds, device-identity shape pre-checks, a per-IP device-signature auth-rate-limit gate in the Gateway WS connect handler, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the schema caps, shape pre-checks, and pre-crypto gate, but reset the device-signature bucket only after successful admission/registration and return an AUTH_RATE_LIMITED-style detail for lockout responses, then add redacted WS-path proof.

## Reproduction Assessment

Yes, source-reproducible. Current main derives the device id and reaches v3/v2 signature verification before final auth and pairing/access admission; the PR also includes a concrete running-gateway PoC command and timing data, though I did not run it in this read-only review.

## Solution Assessment

No. The core direction is maintainable, but the reset needs to move past final device admission/registration and the lockout response should use rate-limited semantics so clients do not treat it like an ordinary signature failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reset the signature bucket after admission:** `src/gateway/server/ws-connection/message-handler.ts:915`
  - body: The new bucket is cleared immediately after `authOk`, but the not-paired, metadata-upgrade, role-upgrade, and scope-upgrade branches below can still reject before `setClient` registers the socket. A client with valid shared auth but an unadmitted or upgrade-pending device can repeatedly clear the bucket and keep reaching signature verification, so move this reset until successful admission/registration or avoid resetting on those rejection paths.
  - confidence: 0.89
- **[P2] Return a rate-limited detail code for lockout:** `src/gateway/server/ws-connection/message-handler.ts:806-808`
  - body: This rate-limit branch reports `reason: "device-signature-rate-limited"` but maps the detail `code` through the ordinary invalid-signature path. Gateway reconnect suppression only treats `AUTH_RATE_LIMITED` as terminal, so clients can immediately reconnect through the locked-out path instead of honoring the lockout; use `AUTH_RATE_LIMITED` or add a dedicated mapping with reconnect coverage.
  - confidence: 0.82
