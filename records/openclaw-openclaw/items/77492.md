## Summary

Keep open: the PR is useful security hardening, but the proposed gate is still bypassable because it runs after PEM public-key parsing and clears the new bucket before the overall handshake is authenticated.

## What This Changes

The PR adds pre-auth device field bounds, device identity shape pre-checks, a new device-signature rate-limit scope in websocket auth, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the layered hardening, but move the rate gate ahead of all attacker-controlled PEM parsing and only clear the device-signature bucket after the whole connect handshake is authorized.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main reaches deriveDeviceIdFromPublicKey and the v3/v2 signature verifier on a pre-auth device handshake. The PR body also includes a live PoC, but I did not run it under the read-only review constraint.

## Solution Assessment

No. The direction is right, but the limiter is placed too late for PEM public keys and is reset before full authentication succeeds, so the security boundary is still incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move the signature limiter before PEM parsing:** `src/gateway/server/ws-connection/message-handler.ts:780-784`
  - body: This check runs after `deriveDeviceIdFromPublicKey(device.publicKey)`. For PEM inputs, including the PoC shape described in the PR body, that helper still calls `crypto.createPublicKey` before the limiter can reject a locked-out IP, so a locked-out attacker can continue forcing the key-parse part of the CPU work. Move the gate ahead of device-id derivation or make the pre-derivation path reject/normalize PEM without crypto, and add a PEM-key regression test.
  - confidence: 0.9
- **[P2] Keep the signature bucket locked until auth succeeds:** `src/gateway/server/ws-connection/message-handler.ts:824`
  - body: The new bucket is reset as soon as the device signature is valid, before `resolveConnectAuthDecision` can reject a missing token, bad token, or unpaired device. An unauthenticated client can generate its own keypair and valid signatures, repeatedly clear the bucket, and still force the pre-auth signature work. Reset this scope only after the full connect handshake is authorized, and cover valid-signature-but-unauthorized attempts.
  - confidence: 0.82
