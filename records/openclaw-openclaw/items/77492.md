## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Defer signature-bucket reset until admission:** `src/gateway/server/ws-connection/message-handler.ts:915`
  - body: The new bucket is reset as soon as `authOk` is true, but the handshake can still fail later in pairing, metadata, role, or scope checks before `setClient` registers the connection. A client with valid shared auth and its own valid device signature can repeatedly reset the pre-crypto bucket while still being rejected, so the defense does not cover all unadmitted device handshakes.
  - confidence: 0.88
- **[P2] Return the standard rate-limit error contract:** `src/gateway/server/ws-connection/message-handler.ts:927`
  - body: The lockout response reports `DEVICE_AUTH_SIGNATURE_INVALID` and puts `retryAfterMs` only inside `details`. Existing gateway clients pause reconnects for `AUTH_RATE_LIMITED` and read `retryAfterMs` from the top-level error, so a legitimate client that hits this bucket can immediately reconnect and churn the lockout instead of backing off.
  - confidence: 0.82
