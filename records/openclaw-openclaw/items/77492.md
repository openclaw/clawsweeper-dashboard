## Summary

Keep open: this is a plausible security fix, but the current patch still resets the new signature bucket before final device admission and the supplied proof does not exercise the real WebSocket handler path.

## What This Changes

The PR adds gateway device-handshake field bounds, device-identity plausibility checks, a device-signature auth-rate-limit scope in the WebSocket connect path, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the schema bounds, shape pre-checks, and pre-crypto rate gate, but reset the device-signature bucket only after successful device admission/registration or avoid resetting it on pairing and upgrade rejection paths, then add redacted WS-handler proof.

## Reproduction Assessment

Yes. Current main source shows device-id derivation and v3/v2 signature verification before the auth decision, and the PR body supplies a concrete WS flood PoC with observed pre-fix latency degradation; I did not live-run the PoC in this read-only pass.

## Solution Assessment

No for the current patch. The layered bounds and pre-crypto gate are the right direction, but the reset should move past final device admission/registration and the proof should exercise the real WS handler path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Defer signature-bucket reset until device admission:** `src/gateway/server/ws-connection/message-handler.ts:915`
  - body: The new bucket resets as soon as `authOk` is true, but the not-paired and access-upgrade branches below can still reject before `setClient` registers the socket. A client with valid shared auth but an unadmitted device can clear the bucket on every attempt and keep reaching signature verification, so reset after successful admission/registration or avoid resetting on those rejection paths.
  - confidence: 0.87
