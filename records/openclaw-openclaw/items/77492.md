Full review comments:

- **[P1] Serialize device-signature limiter attempts:** `src/gateway/server/ws-connection/message-handler.ts:975-1001`
  - body: `check()` and `recordFailure()` run as separate operations, so a burst of WebSocket handshakes from the same IP can all observe `allowed` before any failure is recorded. That lets more than `maxAttempts` requests enter `deriveDeviceIdFromPublicKey` and v3/v2 signature verification, leaving the CPU-DoS fix incomplete; wrap the full signature attempt in the existing per-IP/scope serializer and cover it with a concurrent forged-burst regression.
  - confidence: 0.9
- **[P1] Preserve bootstrap-token limiter behavior:** `src/gateway/server/ws-connection/message-handler.ts:974-1004`
  - body: This gate runs before `resolveConnectAuthDecision`, so device-bearing bad bootstrap-token attempts can be stopped by the new signature bucket before the merged bootstrap-token limiter reports `authReason: "rate_limited"`; the latest CI fails the bootstrap-token regression tests at lines 78 and 107. Either keep the existing bootstrap/device-token fallback semantics intact or get explicit security-owner approval and update focused tests for the new precedence.
  - confidence: 0.86
