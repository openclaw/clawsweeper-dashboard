## Summary

Keep open: current main still has a source-reproducible pre-auth device-signature CPU path, and this PR is the active fix vehicle, but the patch still resets the new signature bucket before final device admission and the supplied after-fix proof is helper-level rather than WS-handler proof.

## What This Changes

The PR adds gateway device-handshake field bounds, device-identity shape pre-checks, a device-signature auth-rate-limit scope in the WebSocket connect path, regression tests, a PoC script, and a changelog entry.

## Best Possible Solution

Keep the schema bounds, shape pre-checks, and pre-crypto rate gate, but reset the device-signature bucket only after successful device admission/registration or avoid resetting it on pairing and upgrade rejection paths, then add redacted WS-handler proof.

## Reproduction Assessment

Yes. Current main derives the device id and runs v3/v2 device-signature verification before the auth decision, and the PR body supplies a concrete WS flood PoC with observed pre-fix latency degradation.

## Solution Assessment

No for the current patch. The layered bounds and pre-crypto gate are the right direction, but the reset should move past final device admission/registration and the proof should exercise the real WS handler path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Defer signature-bucket reset until device admission:** `src/gateway/server/ws-connection/message-handler.ts:915`
  - body: The new bucket is reset as soon as credential auth succeeds, but the not-paired, metadata-upgrade, role-upgrade, and scope-upgrade branches below can still reject before `setClient` registers the socket. A client with valid shared auth but an unadmitted device can repeatedly clear the bucket and keep reaching signature verification, so move the reset until successful admission/registration or avoid resetting on these rejection paths.
  - confidence: 0.87
