## Summary

Keep open: current main still has the Feishu HTTP shutdown race, and this PR is a narrow plausible fix for the linked bug, but it lacks required real after-fix behavior proof from the external contributor before merge.

## What This Changes

The branch adds a Feishu-local HTTP server close helper and awaits it from monitor-state and webhook abort cleanup before deleting server state.

## Best Possible Solution

Land a narrow Feishu plugin fix that awaits HTTP server shutdown in both stop-state and webhook abort paths, backed by real restart proof and ideally focused regression coverage; keep the broader resource-manager idea as separate follow-up work.

## Reproduction Assessment

No live reproduction was run in this read-only review, but the issue is source-reproducible: current main calls asynchronous `server.close()` and immediately clears Feishu HTTP server state before the close callback. A concrete runtime repro would start a Feishu webhook monitor, stop or abort it, and immediately restart on the same port while observing port-release behavior.

## Solution Assessment

Yes, the proposed direction is a maintainable local fix: it keeps the shutdown logic inside the Feishu plugin and follows Node's documented `server.close()` then `closeAllConnections()` order. Before merge, the PR still needs contributor-supplied real behavior proof, and focused regression coverage would reduce future cleanup regressions.
