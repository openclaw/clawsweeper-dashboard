## Summary

Keep this PR open. Current main still clears Feishu HTTP server state immediately after `server.close()` in both stop-state and webhook abort paths, while the PR remains a focused implementation candidate for the linked open Feishu cleanup bug. The patch shape is consistent with Node's HTTP shutdown contract, but it still needs maintainer review on the public stop contract and fresh validation before landing.

## What This Changes

The PR adds a Feishu HTTP server close helper with a timeout and `closeAllConnections()`, then awaits it before clearing monitor state in `monitor.state.ts` and webhook abort cleanup in `monitor.transport.ts`.

## Best Possible Solution

Land this PR or a maintainer-owned equivalent in the Feishu plugin after settling the stop-wrapper contract and adding targeted HTTP close sequencing/restart coverage. Keep the broader cross-plugin ResourceManager discussion in #48793 separate from this narrow Feishu fix.

## Reproduction Assessment

Yes for the cleanup-ordering bug: current main directly shows `server.close()` followed by immediate map deletion/resolution, and a targeted test can stub or delay the close callback to reproduce the premature state cleanup. I did not run live restart tests because this review was read-only.

## Solution Assessment

Yes with caveats: awaiting a Feishu-owned HTTP close helper that calls `server.close()` before `closeAllConnections()` is the narrowest maintainable fix for this plugin. The safer landing path is to also decide whether `stopFeishuMonitor()` should expose the Promise and add focused regression coverage.
