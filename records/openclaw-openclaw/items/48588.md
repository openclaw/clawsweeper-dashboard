## Summary

Keep open. Current main still clears Feishu HTTP server state immediately after calling asynchronous `server.close()`, while this PR remains the active narrow implementation path for the linked Feishu cleanup bug and has addressed the earlier review comments about shared close logic, timer cleanup, rejection propagation, and `closeAllConnections()` ordering.

## What This Changes

The PR adds an awaitable Feishu HTTP server close helper with timeout/connection cleanup and uses it before clearing monitor state in `monitor.state.ts` and webhook abort cleanup in `monitor.transport.ts`.

## Best Possible Solution

Land this PR or a maintainer-owned equivalent in the Feishu plugin after rechecking the merge preview, deciding the `stopFeishuMonitor()` promise contract, and adding focused cleanup/restart coverage. Keep the broader cross-plugin ResourceManager proposal in #48793 separate from this narrow Feishu fix.

## Reproduction Assessment

Yes. Current main directly shows the cleanup-ordering bug: a delayed or stubbed `server.close()` callback would allow `httpServers` state to be deleted and the monitor promise to resolve before the port is actually closed; this can be covered with a targeted Feishu webhook cleanup test.

## Solution Assessment

Yes, with caveats. A Feishu-owned awaitable HTTP close helper used by both state cleanup and webhook abort cleanup is the narrowest maintainable fix; the remaining maintainer decision is whether to expose that async shutdown through `stopFeishuMonitor()` and how much regression coverage to require.
