## Summary

Keep open: current main and the latest release still have the Feishu HTTP cleanup race, and this PR remains the linked fix path, but it still needs real behavior proof plus a wrapper/test update before merge.

## What This Changes

The branch adds an awaited Feishu HTTP server close helper and uses it before clearing monitor state and webhook cleanup maps.

## Best Possible Solution

Land a refreshed Feishu-local async shutdown fix that propagates the cleanup Promise through stopFeishuMonitor(), updates affected cleanup hooks/tests to await it, keeps the bounded close helper, and leaves the broader ResourceManager design to https://github.com/openclaw/openclaw/issues/48793.

## Reproduction Assessment

Yes, at source level: current main still clears Feishu HTTP server state immediately after server.close() in both stop cleanup and webhook abort cleanup. A live Feishu restart loop would still be needed to measure the external EADDRINUSE or memory-growth symptom.

## Solution Assessment

No, not as-is: the awaited bounded close helper is the right narrow direction, but the public stop wrapper still drops the async cleanup Promise and the PR lacks after-fix real behavior proof.
