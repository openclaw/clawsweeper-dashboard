## Summary

Keep open: current main still has the Feishu close-before-clear cleanup race, but this PR remains incomplete because the public stop wrapper cannot await the new async cleanup and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The branch adds an awaited Feishu HTTP server close helper and uses it in monitor state cleanup and webhook abort cleanup before clearing server and bot state.

## Best Possible Solution

Land or replace this with a narrow Feishu cleanup fix that propagates the awaited stop Promise through `stopFeishuMonitor()` and focused cleanup tests, while keeping the broader ResourceManager design in its separate tracker.

## Reproduction Assessment

Yes, at source level: current main and the latest release close Feishu HTTP servers and clear state before the close callback in both stop and webhook-abort paths. I did not run a live Feishu restart loop in this read-only review.

## Solution Assessment

No, not yet. Awaiting HTTP close in state and transport cleanup is the right direction, but the public stop wrapper and cleanup tests must be able to await that shutdown before merge.
