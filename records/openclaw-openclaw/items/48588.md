## Summary

Keep open: current main and the latest release still have the Feishu HTTP shutdown race, and this PR is a focused fix path, but it still drops awaitability at the public stop wrapper/tests and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a bounded `closeHttpServer()` helper and awaits Feishu HTTP server close before clearing monitor state in stop and webhook-abort cleanup paths.

## Best Possible Solution

Land a refreshed Feishu-local async shutdown fix that returns the cleanup Promise through `stopFeishuMonitor()`, updates affected cleanup hooks/tests to await it, keeps the bounded close helper, and leaves the ResourceManager design to the separate follow-up.

## Reproduction Assessment

Yes, at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop cleanup and webhook-abort cleanup. A live Feishu restart loop would only be needed to measure the external `EADDRINUSE` or memory-growth symptom.

## Solution Assessment

No, not as-is: awaiting bounded HTTP shutdown is the right narrow direction, but the exported stop wrapper still drops the async cleanup Promise and the PR lacks after-fix real behavior proof.
