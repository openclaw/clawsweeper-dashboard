## Summary

Keep open: current main still has the Feishu HTTP server close-before-state-clear race, and this PR is a plausible narrow fix, but it is not merge-ready because the existing stop wrapper remains non-awaitable and no after-fix real behavior proof is present.

## What This Changes

The PR adds an awaitable Feishu HTTP server close helper and uses it before clearing monitor state in state cleanup and webhook abort cleanup.

## Best Possible Solution

Land a narrow Feishu-owned async HTTP close fix that returns the cleanup Promise through public stop helpers, shares the timeout close helper in both cleanup paths, and adds focused restart/cleanup regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main calls `server.close()` and immediately deletes or clears Feishu `httpServers` state in both stop and webhook-abort cleanup paths. A live restart loop would only be needed to measure the external EADDRINUSE or memory symptom.

## Solution Assessment

No, not as written. The shared close helper direction is right, but leaving `stopFeishuMonitor()` as a void wrapper prevents existing callers from awaiting shutdown through the public stop path.
