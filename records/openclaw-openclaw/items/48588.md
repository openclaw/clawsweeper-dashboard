## Summary

Keep open: current main and the latest release still clear Feishu HTTP server state immediately after `server.close()`, and this PR targets that real restart/port-release race, but the PR head still drops the async shutdown through the public stop wrapper and has no after-fix real behavior proof.

## What This Changes

The PR adds a Feishu `closeHttpServer()` helper and awaits HTTP server shutdown before clearing monitor state in state cleanup and webhook-abort cleanup paths.

## Best Possible Solution

Land a refreshed Feishu-local shutdown fix that returns and awaits cleanup through `stopFeishuMonitor()`, reuses the bounded close helper in both paths, adds focused regression coverage, and attaches redacted real Feishu restart or port-release proof.

## Reproduction Assessment

Yes, at source level: current main calls `server.close()` and then deletes or clears `httpServers` in state and webhook cleanup before the close callback. A live Feishu restart loop would only be needed to measure the external `EADDRINUSE` or memory-growth symptom.

## Solution Assessment

No, not as-is: awaiting bounded HTTP shutdown is the right narrow fix shape, but the public stop wrapper still drops the async cleanup promise and the PR lacks after-fix real behavior proof.
