## Summary

Keep open: current main and the latest release still have the Feishu close-then-clear race, and this PR is the canonical narrow fix path, but the branch still drops the async stop Promise through `stopFeishuMonitor()` and lacks real behavior proof.

## What This Changes

The PR adds a Feishu HTTP server close helper and awaits it in monitor state and webhook abort cleanup before clearing HTTP server state.

## Best Possible Solution

Land or replace this PR with a Feishu-owned awaited HTTP shutdown helper applied to state and webhook abort cleanup, propagate the Promise through `stopFeishuMonitor()` and affected tests, and keep the broader ResourceManager discussion separate.

## Reproduction Assessment

Yes, at source level: current main calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop and webhook-abort cleanup paths. I did not run a live Feishu restart loop in this read-only review.

## Solution Assessment

No, not yet. Awaiting server close in state and transport cleanup is the right direction, but the public stop wrapper and cleanup tests need to be able to await that shutdown.
