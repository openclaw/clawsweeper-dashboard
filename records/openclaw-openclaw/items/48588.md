## Summary

Keep open: current main and the latest release still clear Feishu HTTP server state before the close callback, and this PR is the canonical narrow fix path, but it still needs the public stop wrapper to return the async cleanup Promise plus after-fix real behavior proof.

## What This Changes

The PR adds a Feishu HTTP server close helper and awaits it from monitor state and webhook abort cleanup before clearing Feishu server maps.

## Best Possible Solution

Land a refreshed Feishu-local async shutdown fix that returns the stop Promise through the public wrapper, updates affected cleanup hooks/tests to await it, keeps the bounded close helper, and leaves the broader resource-manager design to https://github.com/openclaw/openclaw/issues/48793.

## Reproduction Assessment

Yes, at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop cleanup and webhook abort cleanup. A live restart loop would be needed to measure the external EADDRINUSE or memory-growth symptom.

## Solution Assessment

No, not as-is: the awaited close helper is the right narrow direction, but the public `stopFeishuMonitor()` wrapper still drops the cleanup Promise and after-fix real behavior proof is missing.
