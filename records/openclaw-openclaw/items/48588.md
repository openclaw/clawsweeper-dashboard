## Summary

This PR should stay open because it targets a real Feishu cleanup race, but the branch still leaves callers unable to await the public stop path and it lacks after-fix real behavior proof.

## What This Changes

The PR adds a bounded Feishu HTTP server close helper and awaits it before clearing Feishu monitor state and webhook cleanup maps.

## Best Possible Solution

Land a refreshed Feishu-local async shutdown fix that propagates the cleanup Promise through `stopFeishuMonitor()`, updates affected cleanup hooks/tests to await it, keeps the bounded close helper, and leaves the broader resource-manager design to https://github.com/openclaw/openclaw/issues/48793.

## Reproduction Assessment

Yes, at source level: current main still clears Feishu HTTP server state immediately after `server.close()` in stop cleanup, and webhook abort cleanup has the same pattern. A live Feishu restart loop would still be needed to prove the external EADDRINUSE or memory-growth symptom.

## Solution Assessment

No, not as-is: the bounded awaited close helper is the right narrow direction, but the public stop wrapper still drops the async cleanup Promise and the PR lacks after-fix real behavior proof.
