## Summary

Keep open: current main and the latest release still have the Feishu HTTP cleanup race, and this PR is a useful fix direction, but it should not merge until the shutdown Promise is propagated through the public wrapper/tests and after-fix real behavior proof is attached.

## What This Changes

The PR changes Feishu monitor shutdown paths to await bounded HTTP server close before clearing Feishu monitor state maps.

## Best Possible Solution

Land a refreshed Feishu-local async shutdown fix that returns the cleanup Promise through stopFeishuMonitor(), updates affected cleanup hooks/tests to await it, keeps the bounded close helper, and leaves the ResourceManager design to the separate follow-up.

## Reproduction Assessment

Yes, at source level: current main still calls server.close() and clears Feishu HTTP server state before the close callback in both stop and webhook-abort cleanup paths. A live Feishu restart loop would be needed only to measure the external EADDRINUSE or memory-growth symptom.

## Solution Assessment

No, not as-is: the awaited bounded close helper is the right narrow direction, but the exported stop wrapper still drops the async cleanup Promise and the PR lacks after-fix real behavior proof.
