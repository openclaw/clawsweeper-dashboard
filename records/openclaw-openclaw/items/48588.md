## Summary

Keep open: current main and the latest release still have the Feishu HTTP cleanup race, and this PR is a plausible narrow fix, but it is not merge-ready because real behavior proof is missing and the public stop wrapper still drops the async cleanup promise.

## What This Changes

The PR adds a Feishu HTTP server close helper and uses it in monitor state and webhook cleanup to await shutdown before clearing per-account state.

## Best Possible Solution

Land a refreshed Feishu-local cleanup fix that returns and awaits shutdown through `stopFeishuMonitor()`, adds focused HTTP close regression coverage, and attaches redacted real Feishu restart or port-release proof; keep broader resource lifecycle design in https://github.com/openclaw/openclaw/issues/48793.

## Reproduction Assessment

Yes, at source level: current main still calls `server.close()` and clears Feishu HTTP server state before the close callback in both stop cleanup and webhook abort cleanup. A live Feishu restart loop would only be needed to measure the external `EADDRINUSE` or memory-growth symptom.

## Solution Assessment

No, not as-is: awaiting bounded HTTP shutdown is the right narrow fix shape, but the public stop wrapper still drops the async cleanup promise and the PR lacks after-fix real behavior proof.
