- `merge-risk: 🚨 session-state`: The same-lane nested enqueue can prevent deferred transcript maintenance from committing, leaving session transcript state stale.

## Summary

Keep open for revision. The underlying livelock report is credible and still present in current main/latest release, but this PR re-enqueues the worker onto the same maintenance lane that already invoked it, which can deadlock deferred maintenance before it reaches the transcript work.

## What This Changes

The PR changes deferred context-engine turn maintenance to enqueue the worker body on the dedicated maintenance lane instead of waiting for the session lane to drain.

## Best Possible Solution

Keep the linked bug open and revise this PR around a tested ordering invariant that lets transcript-correctness maintenance complete without waiting forever on foreground turns or enqueuing behind itself.

## Reproduction Assessment

Yes at source level for the PR defect: current scheduling already runs the worker in `resolveDeferredTurnMaintenanceLane(sessionKey)`, and the PR awaits a second task on that same single-concurrency lane. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The PR targets a real bug, but the same-lane nested enqueue is not a safe fix; the maintainable path is a tested ordering change that avoids both the original session-lane livelock and the new self-deadlock.
