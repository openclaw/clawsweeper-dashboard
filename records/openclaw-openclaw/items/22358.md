## Label Justifications

- `P2`: This is a normal-priority plugin API feature with real production workflow impact, but it is not an emergency regression or core runtime outage.
- `impact:session-state`: The requested hook would let plugins inspect and act on subagent session completion state, trajectories, summaries, and parent-session announcements.
- `impact:security`: The requested hook may expose child output or session history and allow trusted plugins to alter delivery behavior.
- `impact:message-loss`: A key use case is detecting, suppressing, enriching, or recovering subagent completion announcements before they reach a parent or user-facing channel.

## Summary

Keep open: current main and the latest release still expose only adjacent subagent observation/routing hooks, not the requested post-completion/pre-announcement interception point. The open related implementation PR is useful context, but it is unmerged, partial, and still needs API/security review before it can resolve this issue.

## What This Changes

The issue requests a public plugin hook that runs after a spawned subagent finishes and before the completion announcement so plugins can inspect, enrich, suppress, or persist the result.

## Best Possible Solution

Approve a narrow typed plugin lifecycle hook or equivalent delivery lifecycle contract at the post-subagent-completion/pre-announcement seam, with bounded payloads, clear return semantics, docs, and ordering/regression tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 list only adjacent subagent hooks, and the announce flow has no public interception point before delivery. This is a missing feature/API surface, not a broken documented behavior.

## Solution Assessment

Unclear pending maintainer API and security design. A post-completion/pre-announcement plugin hook is plausible, but the permanent solution should align with existing hook contracts, the open guard-hook PR, and strict payload/return semantics.
