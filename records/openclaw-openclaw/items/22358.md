## Label Justifications

- `P2`: This is a useful public plugin/API improvement with production workflow impact, but it requires maintainer design rather than urgent regression repair.
- `impact:session-state`: The hook would observe and act on subagent completion state, child run metadata, and possibly session-derived output.
- `impact:security`: The requested API could expose conversation-derived child data and allow plugin code to modify or suppress user-visible completion delivery.
- `impact:message-loss`: One stated use case is detecting failed completion announcement delivery and triggering fallback notification for long-running subagent work.

## Summary

Keep open: current main and v2026.5.20 still provide adjacent subagent observation/routing hooks, but not the requested post-completion, pre-announcement interception point with announcement mutation or delivery-result visibility.

## What This Changes

Request a public extension hook that fires after a spawned subagent completes and before the parent completion announcement is delivered.

## Best Possible Solution

Design a typed public lifecycle/delivery contract for post-subagent completion that defines bounded payloads, delivery-result visibility, hook ordering, timeout/failure behavior, and return semantics for enrichment, suppression, or filtering.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and v2026.5.20 inspection show the requested public completion-interception hook is absent while adjacent subagent observation and routing hooks exist.

## Solution Assessment

Unclear as a final API shape. A post-completion/pre-announcement hook fits the plugin direction, but maintainers still need to choose payload limits, return semantics, ordering, delivery-failure visibility, and security boundaries.
