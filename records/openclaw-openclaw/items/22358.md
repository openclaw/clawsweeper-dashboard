## Summary

Keep open for maintainer API design. Current main and v2026.6.6 expose adjacent observation and routing hooks, but still lack a public completion-payload interception point; the broader related issues do not fully cover announcement enrichment, suppression, and delivery-result observation.

## What This Changes

Requests a public post-subagent-completion hook that can inspect completed child context, run deterministic processing, and influence the parent completion announcement.

## Best Possible Solution

Introduce a deliberate two-phase typed plugin contract: a bounded post-run/pre-announcement preparation hook for enrichment or suppression, followed by an immutable delivery-result observation event, with explicit permissions, ordering, timeout/error semantics, documentation, and focused regression coverage.

## Reproduction Assessment

Yes at source level: current main’s public hook types, runner, documentation, tests, and announce flow demonstrably lack the requested interception semantics. This is a missing capability rather than a regression in documented behavior.

## Solution Assessment

No, not exactly as proposed: pre-announcement mutation and final delivery-result observation occur at different phases, so a two-phase typed contract is safer than one callback attempting both jobs.

## Risks / Open Questions

- Exposing child transcript or completion output can disclose credentials, private conversation data, tool results, or other sensitive context to plugins unless access is explicitly bounded and permissioned.
- Allowing plugins to enrich or suppress announcements changes message-delivery and session-state behavior, so ordering, handler timeouts, failures, retries, and fallback semantics need an explicit product decision.
- Final delivery success is known only after the proposed pre-announcement phase, so one callback cannot reliably provide both mutable announcement preparation and final delivery outcome without ambiguous timing.
