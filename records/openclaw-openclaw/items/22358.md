- `P2`: This is a normal-priority plugin/agent lifecycle feature with concrete production workflow impact but no current emergency regression.
- `impact:session-state`: The requested hook is meant to inspect child run history/output and drive memory, trajectory, and parent-session completion behavior.

- `impact:message-loss`: One stated use case is detecting failed subagent completion announces and controlling whether completion messages are delivered, suppressed, or replaced.

## Summary

Keep open. Current main and v2026.6.6 still expose only adjacent subagent observation/routing hooks, while the requested post-completion/pre-announcement hook with announcement enrichment, suppression, and delivery-result semantics remains missing; the overlapping open implementation PR is conflicting and still needs maintainer API/security decisions.

## What This Changes

Request a plugin extension hook that fires after a spawned subagent completes and before its completion announcement is sent, allowing plugins to inspect results and optionally modify or filter the announcement.

## Best Possible Solution

Design and land one typed plugin lifecycle/delivery contract that fires after child completion and before announcement delivery, exposes minimized child metadata/result/delivery status, and defines explicit enrichment, suppression, timeout, and error semantics with docs and regression tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 lack a `post_subagent_complete`, `subagent_complete`, or `before_subagent_result_deliver` hook, and the announce flow sends the built internal completion event directly into delivery. This is a missing capability rather than a broken documented behavior.

## Solution Assessment

Unclear pending maintainer API design. A typed plugin hook at the post-completion/pre-announcement point is plausible, but the permanent payload, security boundary, return semantics, and overlap with broader lifecycle hooks need maintainer approval first.
