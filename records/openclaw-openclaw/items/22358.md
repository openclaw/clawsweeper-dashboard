## Label Justifications

- `P2`: This is a normal-priority feature/API gap with concrete multi-agent delivery and plugin-use impact, but it is not an emergency regression.
- `impact:message-loss`: The requested hook would let integrations detect failed completion announcement delivery, suppress noisy completions, or trigger fallback notifications.
- `impact:session-state`: The use case centers on deterministic post-run artifacts, trajectory files, memory/index updates, and child-session completion state.
- `impact:security`: The hook would expand plugin access to child output or history and could alter delivery, so sensitive-data and trust-boundary review is required.

## Summary

Keep open. Current main and v2026.6.6 still expose only adjacent observation/routing hooks; the requested post-subagent-completion/pre-announcement extension point remains missing and needs maintainer API/security design rather than cleanup closure.

## What This Changes

Request a public extension hook after subagent completion and before parent-session announcement delivery so plugins can inspect, persist, enrich, filter, or suppress completion output.

## Best Possible Solution

Define one bounded public lifecycle or guard hook at the post-completion/pre-announcement boundary, with minimal payload, explicit permissions, return semantics, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 expose only observation/routing subagent hooks, and no public hook can inspect or mutate the completion announcement before delivery. This is a missing capability rather than a broken documented behavior.

## Solution Assessment

Unclear pending maintainer API/security design. A hook at this boundary is plausible, but the best solution must first settle payload bounds, mutation/suppression semantics, delivery-result visibility, ordering, and permissions.
