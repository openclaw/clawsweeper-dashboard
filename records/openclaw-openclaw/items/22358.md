## Summary

Keep open. Current main still exposes only adjacent subagent spawn, routing, spawned, and ended hooks; none can inspect the completed child output before the parent announcement is built, modify or suppress that announcement, or observe the final delivery result through the public plugin API. The recent maintainer discussion also explicitly says related delivery fixes improved symptoms without covering this extension point.

## What This Changes

Requested a public extension hook that fires after a subagent completes and before the parent completion announcement is emitted, with completion metadata and optional announcement modification.

## Best Possible Solution

Design a typed public plugin hook, or a broader canonical lifecycle/delivery hook design, at the post-subagent-completion/pre-announcement boundary. It should define bounded child/run metadata, safe completion-output or history access, usage/timing where available, delivery status/error visibility where appropriate, return semantics for enrich/suppress/filter, and docs plus regression tests for ordering and privacy boundaries.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/API gap. There is a high-confidence source-level verification path: inspect the current hook catalog and announce flow and search for the proposed hook names, which confirms no public post-completion/pre-announcement interception exists.

## Solution Assessment

Unclear as a final API, but the requested direction belongs in the core plugin SDK rather than as a prompt workaround. The safest path is maintainer-led API design before implementation because the hook would affect delivery ordering, announcement mutation, and child-run data exposure.
