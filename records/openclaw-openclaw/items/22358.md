## Summary

Current main and v2026.6.5 still provide adjacent subagent observation and routing hooks, but no public plugin interceptor between completion finalization and the parent announcement. The issue remains distinct from the broader workspace-hook request and the separate notification-latency report, so it still needs maintainer API and security design.

## What This Changes

Requests a public extension hook after subagent completion and before parent announcement delivery, with access to completion metadata and controlled announcement modification.

## Best Possible Solution

Define a typed plugin lifecycle result contract with bounded child/run metadata, explicit sensitive-data access rules, deterministic ordering, and separate narrowly specified semantics for announcement transformation and final delivery-result observation.

## Reproduction Assessment

Yes, at source level. Enumerating the public plugin hooks and tracing registry completion through the announcement flow shows no post-completion/pre-announcement interceptor on current main or v2026.6.5.

## Solution Assessment

Unclear for the exact proposed callback. A plugin lifecycle extension is the appropriate boundary, but maintainers should separate pre-delivery announcement transformation from post-delivery outcome observation rather than overload one hook.

## Risks / Open Questions

- Exposing completion output or child history may disclose sensitive session content to plugins unless access, redaction, and trust boundaries are explicitly defined.
- Allowing plugins to suppress or rewrite announcements could hide successful or failed work and create message-loss behavior unless ordering, timeout, error, and fallback semantics are specified.
- Delivery status is known after the pre-announcement point, so one hook may not cleanly satisfy both announcement mutation and reliable fallback notification without a separate post-delivery result contract.
