## Label Justifications

- `impact:message-loss`: The requested hook is partly motivated by detecting or compensating for failed subagent completion announcements.

## Summary

Keep open. Current main and v2026.6.6 still expose only adjacent subagent observation/routing hooks; the requested post-completion/pre-announcement extension point is not implemented, and the open overlapping PR is not a merged or clean canonical replacement.

## What This Changes

Request a public extension hook after a subagent completes and before the parent completion announcement is delivered so plugins can inspect, persist, enrich, suppress, or react to completion data.

## Best Possible Solution

Keep this as the narrow design tracker for a typed post-subagent-completion/pre-announcement plugin hook or an explicitly equivalent lifecycle/delivery API with bounded payloads, mutation rules, security controls, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: current main exposes adjacent subagent observation/routing hooks, but no public post-completion/pre-announcement hook that can inspect, alter, suppress, or observe delivery results for the completion announcement. This is a missing capability rather than a broken documented behavior.

## Solution Assessment

Unclear pending maintainer API design. A hook at this boundary is plausible, but the best solution must settle payload bounds, mutation semantics, delivery-result visibility, ordering, and security behavior first; the open related PR is not yet a clean canonical fix.
