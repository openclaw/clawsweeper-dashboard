## Label Justifications

- `P2`: This is a useful plugin API improvement with production workflow impact, but it requires maintainer design rather than urgent regression repair.
- `impact:session-state`: The requested hook would observe and act on subagent completion state, child run metadata, and potentially session-derived output.
- `impact:security`: The requested API could expose conversation-derived child data to plugin code and allow modification or suppression of completion delivery.
- `impact:message-loss`: One stated use case is detecting failed completion announcement delivery and triggering fallback notification for long-running subagent work.

## Summary

Keep open. Current main and v2026.5.20 still expose adjacent subagent observation/routing hooks only; the requested post-completion, pre-announcement interception point is still missing and needs maintainer API/security design rather than cleanup closure.

## What This Changes

The issue requests a public post-subagent-completion hook that can inspect child results, observe delivery status, and modify, enrich, filter, or suppress the parent completion announcement.

## Best Possible Solution

Design a typed public lifecycle/delivery contract covering bounded child metadata or approved output access, delivery-result visibility, hook ordering, timeout/failure behavior, and return semantics for announcement enrichment, suppression, or filtering, with docs and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and v2026.5.20 inspection show the requested public completion-interception hook is absent while adjacent subagent observation and routing hooks exist.

## Solution Assessment

Unclear as a concrete API. A post-completion/pre-announcement hook fits the plugin direction, but maintainers still need to choose payload limits, return semantics, ordering, delivery-failure visibility, and security boundaries.
