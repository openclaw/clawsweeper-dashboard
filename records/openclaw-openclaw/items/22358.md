## Summary

Keep open. Current main still lacks a post-subagent-completion/pre-announcement plugin hook: the adjacent subagent hooks are either routing-only or observation-only, and the announce flow builds and delivers the completion event without a public interception point. The discussion also has fresh maintainer context saying related delivery fixes improved symptoms but did not cover this API gap, so this should stay open for API design rather than be closed as stale, duplicate, implemented, or ClawHub-only.

## What This Changes

Add a typed plugin hook that fires after a spawned subagent finishes and before the parent completion announcement, allowing plugins to inspect completion metadata or history, persist artifacts, and optionally enrich, suppress, or filter the announcement.

## Best Possible Solution

Define a public typed plugin lifecycle/delivery hook at the post-subagent-completion and pre-announcement point, with bounded metadata, explicit conversation-access gating, delivery-status/error visibility where appropriate, clear mutation/suppression semantics, docs, and regression tests covering hook ordering and announce delivery behavior.

## Risks / Open Questions

- The hook needs explicit trust and privacy rules before implementation because completion output, child history, usage, and delivery failures can expose cross-agent or cross-channel content to plugins.
- Return semantics for enrichment, suppression, and filtering need to be ordered against existing delivery-target, message_sending, before_dispatch, and reply_dispatch hooks to avoid duplicate, skipped, or misleading announcements.
