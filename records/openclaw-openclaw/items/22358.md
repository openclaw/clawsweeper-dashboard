## Label Justifications

- `P2`: This is a concrete plugin API improvement with production workflow impact, but it is not an urgent core regression and needs maintainer design.
- `impact:session-state`: The requested hook would observe subagent completion state, child run metadata, and possibly session-derived output.
- `impact:security`: The requested API could expose conversation-derived child data to plugin code and allow modification or suppression of completion delivery.
- `impact:message-loss`: A stated use case is detecting failed subagent completion announcement delivery and triggering fallback notification for long-running work.

## Summary

Keep open. Current main and v2026.5.20 still expose adjacent subagent observation/routing hooks only; the requested post-completion, pre-announcement interception point is still missing and needs public API and security design.

## What This Changes

Request a public extension hook that fires after subagent completion and before parent announcement delivery so plugins can inspect, persist, filter, enrich, or suppress completion output.

## Best Possible Solution

Design a typed public lifecycle/delivery contract for subagent completion that defines bounded child metadata or approved output access, hook ordering, delivery-result visibility, failure semantics, and safe return behavior for announcement enrichment, suppression, or filtering, with docs and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and release inspection show the requested public completion-interception hook is absent while adjacent subagent observation and routing hooks exist.

## Solution Assessment

Unclear as a final API shape. A post-completion/pre-announcement hook fits the plugin direction, but maintainers still need to choose payload limits, return semantics, ordering, delivery-failure visibility, and security boundaries.

## Risks / Open Questions

- The requested hook could expose child-session output or history to plugin code, so payload minimization, capability boundaries, redaction, and audit semantics need design before implementation.
- Allowing plugins to enrich, suppress, or filter completion announcements could create silent delivery or ordering regressions unless return, timeout, and failure behavior are explicit.
- The request overlaps broader lifecycle and delivery architecture work such as https://github.com/openclaw/openclaw/issues/43454 and https://github.com/openclaw/openclaw/issues/66638, so a narrow hook should not preclude the canonical event-delivery model.
