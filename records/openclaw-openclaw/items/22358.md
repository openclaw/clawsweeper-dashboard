## Label Justifications

- `P2`: This is a concrete plugin API improvement with production workflow impact, but it is not an urgent confirmed regression or core runtime outage.
- `impact:session-state`: The requested hook would observe subagent completion state, child run metadata, and possibly session-derived output before parent delivery.
- `impact:security`: The requested hook could expose child conversation data and allow plugins to modify or suppress completion delivery.
- `impact:message-loss`: A stated use case is detecting failed completion announcement delivery and triggering fallback notification for long-running subagent work.

## Summary

Keep open. Current main and v2026.5.22 still expose only adjacent subagent observation/routing hooks; the requested post-completion, pre-announcement interception point remains missing and needs maintainer API plus security design.

## What This Changes

Request a public hook after spawned subagent completion and before parent announcement delivery so plugins can inspect, persist, modify, filter, or recover completion output.

## Best Possible Solution

Design a typed public lifecycle/delivery hook contract with bounded metadata/output access, explicit ordering and timeout behavior, delivery-result visibility, safe return semantics for enrichment or suppression, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and release inspection show the requested public completion-interception hook is absent while adjacent subagent observation and routing hooks exist.

## Solution Assessment

Unclear as a final API shape. A post-completion/pre-announcement hook fits the plugin direction, but maintainers still need to choose payload limits, return semantics, ordering, delivery-failure visibility, and security boundaries.
