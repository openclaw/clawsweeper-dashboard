## Summary

Keep open: the PR still has blocking correctness regressions in ACP runtime config handling and the external contributor has not provided after-fix real ACP/Foreman behavior proof.

## What This Changes

The PR changes ACPX launch/runtime config handling for Claude/Gemini workers and relaxes `sessions_yield` when no `onYield` callback is available.

## Best Possible Solution

Keep launch-time model handling only where the adapter contract supports it, keep runtime config controls truthful, and require a real ACP/Foreman proof run before merge.

## Reproduction Assessment

No live Foreman/ACP reproduction is provided. The blocking patch regression is source-reproducible because the PR can return success from `setConfigOption` while current main persists the requested option afterward.

## Solution Assessment

No: broad no-op suppression is not the best fix because explicit model/thinking controls can look successful while the live ACP session remains unchanged. The safer path is to distinguish startup replay from direct runtime controls and either apply supported adapter controls or return an unsupported-control error.
