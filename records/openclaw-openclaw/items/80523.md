## Summary

Keep open: this is a plausible ACP dispatch fix direction, but the patch currently turns supported ACP model/effort updates into successful no-ops and still needs real Foreman/ACP behavior proof before merge.

## What This Changes

The PR adds Claude/Gemini ACP launch-time model handling, skips selected runtime config replays for those adapters, and relaxes `sessions_yield` when only gateway session context exists.

## Best Possible Solution

Keep the launch-time model path only where the adapter contract supports it, and make runtime controls truthful by applying supported model/effort updates, re-ensuring when needed, or returning an unsupported-control error instead of silently succeeding.

## Reproduction Assessment

No live Foreman/ACP reproduction is available from the PR, but the blocking regression is source-reproducible: `setSessionConfigOption` persists after `runtime.setConfigOption` succeeds, while the PR changes Claude/Gemini `setConfigOption` to return without touching the adapter.

## Solution Assessment

No: broad no-op suppression is not the best fix because it makes active model/thinking updates look successful. The safer path is to distinguish automatic replay from direct user/runtime controls and preserve supported adapter updates or fail clearly.
