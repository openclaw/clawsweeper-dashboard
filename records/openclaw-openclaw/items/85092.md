## Summary

Keep open: the current CLI source still has a credible scoped-auth-store bug. The logout path computes and writes against the configured default agent directory, while the store update helper can return success without saving when the selected profile IDs came from a merged/inherited store rather than the file being updated.

## What This Changes

The issue asks for `openclaw infer model auth logout --provider <id>` to honor `OPENCLAW_AGENT_DIR` for non-default agents or fail instead of reporting a successful profile removal without writing the scoped agent's auth store.

## Best Possible Solution

Make the infer CLI auth logout resolve the same scoped agent directory that operators targeted, and ensure `removedProfiles` reflects profiles actually removed from the intended owner store or a clear failure.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: set `OPENCLAW_AGENT_DIR` for a secondary agent and run `openclaw infer model auth logout --provider <id>`. I did not run a disk-writing repro because this review is read-only, but current main still routes logout through the default-agent path and can report profile IDs without saving.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it derives the write target from the configured default agent and treats a no-save update as success. The maintainable fix is to resolve the requested auth-store scope first and use one owner-aware removal path with regression coverage.
