## Summary

Keep open. Current main still reads document extractor metadata before the extraction try/catch, so the useful hardening is not obsolete, but this protected maintainer draft is currently conflicting and the patch tests a mocked resolver path rather than the production plugin-artifact boundary where malformed descriptors are materialized.

## What This Changes

This PR guards document-extractor id/mimeTypes/extract reads, preserves extractor receiver binding, and adds a regression test for unreadable metadata followed by a healthy PDF extractor.

## Best Possible Solution

Refresh against current main, then either move or add malformed descriptor isolation at the plugin artifact loader boundary or explicitly keep this as narrow dispatcher hardening alongside the loader-boundary PR.

## Reproduction Assessment

Yes for the dispatcher-level failure modeled by the PR: current main reads extractor.mimeTypes before the extraction try/catch. The production malformed-descriptor path is less certain because real entries are validated and materialized upstream before this dispatcher sees them.

## Solution Assessment

No as a complete production fix; the dispatcher guard is a plausible defensive layer, but malformed plugin descriptor isolation belongs at the public-artifact loader/resolver boundary. A refreshed branch should either add that boundary fix or narrowly document that this PR only hardens already-resolved rows.
