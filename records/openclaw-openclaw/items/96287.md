## Summary

Keep open: current main and v2026.6.10 still have the reported Control UI clear bug for saved model/thinking fields, and the backend still lacks thinking:null clear semantics even though model:null is now implemented.

## What This Changes

The issue requests explicit clear semantics when users blank saved Cron Control UI agentTurn model/thinking overrides.

## Best Possible Solution

Land a focused UI/backend patch that sends explicit null clears for stored override fields and makes thinking:null a patch-only clear signal matching model:null, while preserving omitted-field merge behavior and create-time strictness.

Yes. Source inspection on current main shows blank Control UI model/thinking fields are omitted from cron.update, and backend thinking:null is dropped or ignored before it can clear the stored value.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve omitted-field merge behavior and use explicit null only for stored override keys that the user intentionally cleared, then align thinking with the existing model/fallbacks patch-only clear contract.

## Repair Work Prompt

Fix the Cron Control UI and cron update backend so clearing saved isolated agentTurn model/thinking fields removes the stored overrides. Preserve omitted-field merge semantics. In ui/src/ui/controllers/cron.ts, when editing an existing agentTurn job, send payload.model = null or payload.thinking = null only when the current stored payload actually owns that key and the corresponding form field is blank. In src/cron/types.ts, src/cron/normalize.ts, src/cron/service/jobs.ts, and packages/gateway-protocol/src/schema/cron.ts, allow update-patch payload.thinking: null to survive normalization/validation and delete the persisted thinking override, matching the existing model:null contract. Do not add a new config option and do not change cron.add/create payload storage semantics. Add focused regression tests for UI patch construction, normalizeCronJobPatch, protocol validation, and service merge behavior; keep omitted model/thinking preserving existing values. Update existing cron docs only if they document the model null-clear API and need thinking parity. CHANGELOG.md is release-owned; do not edit it.
