## Summary

Keep open: current main, the deployed installer, and v2026.5.18 still run Homebrew setup before the Node check, so the reported non-admin macOS install block remains. Two linked fix PRs addressed the same issue, but both are closed unmerged and no open replacement PR was found.

## What This Changes

The issue asks for `install.sh` on macOS to skip Homebrew setup when an already-supported Node runtime is present on `PATH`.

## Best Possible Solution

Gate Homebrew setup behind the specific macOS branches that actually need brew, add installer-order regression tests, and align the installer flow docs with the conditional behavior.

## Reproduction Assessment

Yes, at source level: current main and the deployed installer call `install_homebrew` before `check_node`, and `install_homebrew` exits for non-admin macOS users without brew. I did not run the live non-admin macOS setup, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes: moving Homebrew setup behind the brew-dependent install branches is the narrow maintainable fix. It should preserve the Node-missing path and add regression coverage for both supported-Node and missing-Node ordering.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83232. On current main, `scripts/install.sh` calls `install_homebrew` before `load_nvm_for_node_detection` and `check_node`, so a non-admin macOS user without brew can fail before an adequate Node on `PATH` is accepted. Move Homebrew setup so macOS invokes it only when a brew-dependent install branch is actually entered; preserve `load_nvm_for_node_detection` before `check_node`, preserve the missing/outdated Node path so Homebrew runs before `install_node`, and avoid broad installer policy or config changes. Account for the macOS Git install branch if needed so existing Node plus existing Git skips Homebrew. Add focused regression coverage in `test/scripts/install-sh.test.ts`, update `docs/install/installer.md` if the flow wording changes, and add a single-line user-facing fix entry to the active changelog.
