## Summary

Keep open: current main and latest release still call `install_homebrew` before Node detection, so a macOS user without Homebrew/admin rights can be blocked before the installer checks an already-supported Node on PATH. This is a narrow, source-backed installer bug, not a stale or duplicate cleanup candidate.

## What This Changes

Skip macOS Homebrew bootstrap when the existing Node runtime already satisfies the installer minimum.

## Best Possible Solution

Move Homebrew setup behind the Node check so macOS only bootstraps brew when `install_node` actually needs it, then add a focused installer regression test and update the installer flow docs if they no longer describe the behavior.

## Reproduction Assessment

Yes at source level: current main calls `install_homebrew` before `check_node`, and `install_homebrew` can exit before a supported Node on PATH is considered. I did not run a live macOS non-admin environment, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes: the requested reordering is the narrow maintainable fix because Homebrew is only needed for the macOS Node installation branch. The fix should preserve Linux behavior and version-manager detection while making brew conditional on a failed Node check.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83232. In `scripts/install.sh`, macOS currently calls `install_homebrew` before Node detection, which blocks no-admin/no-brew users even when Node already satisfies `NODE_MIN_VERSION`. Reorder the flow so `load_nvm_for_node_detection` and `check_node` run first, and call `install_homebrew` only when macOS actually needs `install_node` for missing or outdated Node. Preserve Linux behavior, existing version-manager detection, and the Node minimum contract. Add or update `test/scripts/install-sh.test.ts` to simulate macOS with no brew/non-admin and a supported Node on PATH, asserting Homebrew is not invoked before the installer proceeds. Update `docs/install/installer.md` if the flow description changes, and add a changelog fix entry if landing as a user-facing installer fix.
