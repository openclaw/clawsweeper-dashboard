Full review comments:

- **[P1] Bound the pre-routing hook before awaiting it:** `src/plugins/hooks.ts:1001-1008`
  - body: `runBeforeRouteInboundMessage` delegates to the generic claiming hook, but this new inbound path hook is not in the default modifying timeout map. Because channel turns await it before record/dispatch, a handler without its own `timeoutMs` can stall inbound message delivery instead of failing open.
  - confidence: 0.88
- **[P2] Move compatibility redirects to the current dispatch owner:** `src/plugin-sdk/inbound-reply-dispatch.ts:197-200`
  - body: On current main this file is a deprecated re-export facade, and recording reads `ctxPayload.SessionKey` before `routeSessionKey`. This branch changes only `params.route.sessionKey` on the old path, so compatibility callers can still record/dispatch under the original session or lose the edit during rebase.
  - confidence: 0.9
- **[P2] Document the public routing hook contract:** `src/plugins/hook-types.ts:111`
  - body: Adding a hook name exposes a public plugin API, but docs still list only existing message hooks. The contract needs timing, redirect/suppress semantics, timeout/error behavior, trust boundaries, and the related reply/recency fields before plugin authors depend on it.
  - confidence: 0.84
- **[P3] Assert redirects reach downstream session use:** `src/channels/turn/kernel.test.ts:953-956`
  - body: The redirect test only proves the hook was called and the turn dispatched. Assert the recorded session, returned route key, or payload session key so the test fails if redirect is ignored downstream.
  - confidence: 0.82
