## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bound the pre-routing hook before awaiting it:** `src/plugins/hooks.ts:1001-1008`
  - body: `runBeforeRouteInboundMessage` delegates to the generic claiming-hook runner, but the new hook is not added to the default modifying timeout map. Because channel turns await this hook before record/dispatch, a handler without its own `timeoutMs` can stall inbound message delivery instead of failing open.
  - confidence: 0.9
- **[P2] Move compatibility redirects to the current dispatch owner:** `src/plugin-sdk/inbound-reply-dispatch.ts:197-200`
  - body: On current main, `src/plugin-sdk/inbound-reply-dispatch.ts` is a deprecated re-export facade and the implementation lives under `src/channels/message`. This branch edits the stale surface and updates only the route session key there, while recording prefers `ctxPayload.SessionKey`, so compatibility callers can keep recording or dispatching under the original session after a redirect.
  - confidence: 0.88
- **[P2] Document the public routing hook contract:** `src/plugins/hook-types.ts:111`
  - body: Adding this hook name creates a public plugin API, but the plugin hook docs still list only the existing message and dispatch hooks. The contract needs timing, redirect/suppress semantics, timeout/error behavior, trust boundaries, and session-state expectations before plugin authors depend on it.
  - confidence: 0.84
- **[P3] Assert redirects reach downstream session use:** `src/channels/turn/kernel.test.ts:953-956`
  - body: The new redirect test proves the hook was called and the turn dispatched, but it does not assert the recorded session, returned route key, or payload session key. That lets a future regression ignore the redirect downstream while the test still passes.
  - confidence: 0.82
