## Summary

Keep open: this is a plausible core plugin API feature tied to the open channel-bridging request, but the patch is not mergeable because it calls hook APIs and route fields that current main does not define, and the PR only provides unit/mock proof for an external-plugin flow.

## What This Changes

The PR adds a pre-routing inbound-message hook call in the channel turn kernel and tests mocked redirect/suppress hook results.

## Best Possible Solution

Define the hook as a complete additive SDK contract, wire it through hook types, runner, docs, and tests, and fire it from the resolved turn path using the actual resolved payload/session facts with clear redirect/suppression guardrails.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the blocker: the patch imports unexported hook types, calls a missing runner method, and references fields absent from the current turn contracts; I did not run a build because this review is read-only.

## Solution Assessment

No. The requested capability belongs in a complete, documented plugin hook/API change, but this patch only inserts the kernel call and tests, leaving the public contract, runner wiring, payload update path, and guardrails incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the hook contract before calling it:** `src/channels/turn/kernel.ts:3-9`
  - body: The production change calls a hook that current main does not define: `hook-message.types.ts` exports no `PluginHookBeforeRouteInboundMessage*` types, `PluginHookName`/`PluginHookHandlerMap` omit `before_route_inbound_message`, and `HookRunner` returns no `runBeforeRouteInboundMessage`. This will fail typechecking/build, and plugin authors still cannot register the hook.
  - confidence: 0.95
- **[P2] Update the resolved payload on redirect:** `src/channels/turn/kernel.ts:503`
  - body: Redirect writes `params.ctxPayload.SessionKey`, but `RunChannelTurnParams` has no `ctxPayload`, and record/dispatch consume `resolved.ctxPayload.SessionKey` when present. After the compile errors are fixed, redirects with an existing resolved payload would still record/dispatch to the original session.
  - confidence: 0.88
- **[P3] Make the redirect test assert the hook call:** `src/channels/turn/kernel.test.ts:953`
  - body: This line references the matcher property without invoking it, so the test passes even if the hook never ran. Use `toHaveBeenCalledOnce()` or assert the recorded session key so the redirect path is actually covered.
  - confidence: 0.93
