## Summary

Keep open: this is an active new plugin-routing API PR, not conservative cleanup. The current branch still has concrete type-contract failures, mock-only proof, and an unresolved redirect/suppression trust-boundary decision.

## What This Changes

The PR adds a public `before_route_inbound_message` hook with runner/type wiring plus channel-turn and compatibility-dispatch integration/tests for redirecting or suppressing inbound messages.

## Best Possible Solution

Define a narrow documented SDK contract or extend the existing binding surface with typed redirect/suppress semantics, guardrails, focused tests, docs, and real channel proof.

## Reproduction Assessment

Yes for the PR blockers: CI and source inspection show PR head reads `params.ctxPayload`/`resolved.sessionKey`, which do not exist in the current turn contracts, and the new tests omit required `cfg`. As the linked user problem, this is a feature/API request rather than broken documented current-main behavior.

## Solution Assessment

No. A pre-routing hook may be useful, but this patch should first compile against `resolved.ctxPayload`/`routeSessionKey`, document SDK semantics, and define redirect/suppression scope and audit rules before merge.
