## Summary

Keep this PR open: it is an active feature/API proposal, not cleanup-close material, and the latest revision still has a blocking turn-kernel type/contract error plus real-behavior-proof and trust-boundary gaps.

## What This Changes

The PR adds a `before_route_inbound_message` plugin hook with hook types, runner wiring, compatibility dispatch wiring, channel turn integration, and redirect/suppress tests.

## Best Possible Solution

Define the hook as a complete documented SDK contract, use the actual resolved turn payload/session facts, add guardrails for redirect/suppression scope, and land only with maintainer approval plus real channel-bridge proof.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows the PR head references `params.ctxPayload` and `resolved.sessionKey`, which are absent from the current turn contracts. As a user issue, this is a feature/API request rather than a broken documented behavior.

## Solution Assessment

No. A pre-routing hook may be the right direction, but this patch should first compile against the actual turn contracts and define redirect/suppression guardrails before it is the maintainable solution.
