## Summary

Keep open: the PR is still the active implementation candidate for a new plugin routing hook, but the current head has a compatibility-path redirect bug, only mock/unit proof, and a plugin-controlled redirect/suppression trust boundary that needs maintainer approval before merge.

## What This Changes

The PR adds `before_route_inbound_message` hook types and runner plumbing, wires the hook into channel turn and compatibility inbound reply dispatch paths, and adds redirect/suppression kernel tests.

## Best Possible Solution

Define a documented SDK hook contract with one consistent routing point, keep `routeSessionKey` and `ctxPayload.SessionKey` synchronized across modern and compatibility paths, cover both paths, and land only with real bridge proof plus maintainer-approved redirect/suppression scope.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows the compatibility redirect updates only `params.route.sessionKey` while recording uses `ctxPayload.SessionKey` first. For the linked request itself, reproduction is not applicable because this is a new plugin API capability.

## Solution Assessment

No. A pre-routing hook may be the right direction, but this patch is not yet the narrow maintainable solution until the compatibility payload is synchronized, coverage proves that path, and maintainers approve the redirect/suppression trust model.
