## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bound the pre-routing hook before awaiting it:** `src/plugins/hooks.ts:1001-1008`
  - body: `runBeforeRouteInboundMessage` delegates to the generic claiming hook, but this new hook is not in the default modifying-hook timeout map. Because both call sites await it before inbound routing/dispatch, any registered handler without its own `timeoutMs` can stall message delivery instead of failing open as the linked contract requires.
  - confidence: 0.87
- **[P2] Keep compatibility redirects in sync with the payload session:** `src/plugin-sdk/inbound-reply-dispatch.ts:197-200`
  - body: This redirect path only replaces `params.route.sessionKey`, but the downstream recording path prefers `ctxPayload.SessionKey` when it is present. Compatibility callers that already pass a finalized payload can still record and dispatch under the original session, so the redirect silently fails on that path.
  - confidence: 0.91
- **[P2] Document the public routing hook contract:** `src/plugins/hook-types.ts:111`
  - body: Adding this hook name exposes a public plugin API, but the hook docs still list only the existing message hooks. The PR should document timing, redirect/suppress semantics, timeout/error behavior, and trust boundaries before plugin authors depend on it.
  - confidence: 0.8
- **[P3] Assert redirect reaches downstream session use:** `src/channels/turn/kernel.test.ts:953-956`
  - body: The redirect test proves the hook was called and the turn dispatched, but it never asserts the recorded session, returned route key, or payload session key. Please assert the downstream session value so the test fails when a redirect is ignored after the hook returns.
  - confidence: 0.84
