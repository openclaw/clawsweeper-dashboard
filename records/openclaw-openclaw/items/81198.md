## Summary

Keep open: current main does not implement the requested public pre-routing hook, and this active PR is the implementation path for the linked feature request. The patch is not merge-ready because the compatibility redirect path still leaves the recorded payload on the original session, cross-agent redirects do not update the agent identity, and the PR supplies unit-test proof only for a security-sensitive routing surface.

## What This Changes

This PR adds a public `before_route_inbound_message` plugin hook, wires redirect/suppression into channel turn and compatibility dispatch paths, and adds kernel tests.

## Best Possible Solution

Define a documented SDK hook or binding extension that updates session key and agent identity consistently across modern and compatibility paths, with focused tests and real bridge proof before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the compatibility path updates only the route session key while recording prefers `ctxPayload.SessionKey`, and redirect logic does not realign `agentId`. For the linked request itself, reproduction is not applicable because this is a new public plugin API capability.

## Solution Assessment

No. A pre-routing hook may be the right direction, but this patch is not the maintainable solution until session payload, route key, and agent identity stay aligned, compatibility coverage is added, and maintainers approve the routing trust model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep the compatibility payload session key in sync:** `src/plugin-sdk/inbound-reply-dispatch.ts:197-200`
  - body: `recordChannelMessageReplyDispatch` records with `ctxPayload.SessionKey ?? routeSessionKey`, but this redirect block only replaces `params.route.sessionKey`. Compatibility callers normally pass a finalized context that already has `SessionKey`, so the hook still records and dispatches the original session instead of the redirected one.
  - confidence: 0.92
- **[P2] Recompute the agent identity when redirecting sessions:** `src/channels/turn/kernel.ts:508`
  - body: A `redirectSessionKey` can target a different agent, but this path only mutates the session key. The downstream reply pipeline still uses the `agentId` resolved before the hook, unlike existing binding routing which updates both values, so cross-agent redirects can record under one session while dispatching as the wrong agent.
  - confidence: 0.84
- **[P3] Assert the redirected session reaches recording:** `src/channels/turn/kernel.test.ts:956`
  - body: This test only checks that the hook ran and the turn dispatched. Please assert the redirected key is used by `recordInboundSession`, the result payload, or dispatch context so the test fails if downstream code keeps the original session key.
  - confidence: 0.86
