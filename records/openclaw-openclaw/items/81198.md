Full review comments:

- **[P1] Bound the pre-routing hook before awaiting it:** `src/plugins/hooks.ts:1001-1008`
  - body: `runBeforeRouteInboundMessage` delegates to the claiming hook, but the PR head's default modifying-hook timeout map does not include the new hook. Since `runChannelTurn` awaits this before record/dispatch, a handler without its own `timeoutMs` can stall inbound delivery instead of failing open.
  - confidence: 0.91
- **[P2] Move compatibility wiring to the current owner:** `src/plugin-sdk/inbound-reply-dispatch.ts:159-207`
  - body: This branch adds redirect/suppress logic to `src/plugin-sdk/inbound-reply-dispatch.ts`, but current main has turned that file into a deprecated re-export facade and moved the implementation to `src/channels/message/inbound-reply-dispatch.ts`. Rebase the hook into the current owner so compatibility callers do not drift or miss the behavior.
  - confidence: 0.88
- **[P2] Document the public routing hook contract:** `src/plugins/hook-types.ts:111`
  - body: Adding this hook name exposes a new public plugin API, but the plugin hook docs still list only the existing message hooks. The contract needs timing, redirect/suppress semantics, timeout/error behavior, trust boundaries, and event fields before external plugin authors depend on it.
  - confidence: 0.86
- **[P2] Use the resolved account in hook context:** `src/channels/turn/kernel.ts:474-492`
  - body: `runChannelTurn` builds the hook context from `params.accountId` even though `resolveTurn` can return the effective account id used downstream. Multi-account channel hooks could therefore route on an undefined or stale account value after the channel adapter has already resolved the real account.
  - confidence: 0.79
