## Summary

Keep open. The PR targets a real Control UI/gateway image-attachment bug, but the current branch is conflicting, lacks after-fix real behavior proof, and the proposed UI fix can still lose or suppress attachment sends because it updates pending state from render-time props and blocks unrelated send commands.

## What This Changes

The PR updates gateway image-capability lookup and Control UI chat attachment handling to keep image attachments from being sent before FileReader reads complete, with targeted regression tests.

## Best Possible Solution

Land a refreshed fix that centralizes provider-qualified catalog-id matching, applies FileReader pending deltas and attachment appends against live state, gates only composer sends that depend on pending attachments, and includes real Control UI or gateway proof.

## Reproduction Assessment

Yes, at source level. Current main can miss provider-qualified image-capable catalog rows and can submit while FileReader callbacks have not appended attachment state; I did not run a browser repro in this read-only review.

## Solution Assessment

No. The gateway direction is useful, but the UI patch computes pending counts from captured props, leaves async appends tied to stale state, and blocks more send paths than the attachment race requires.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending-read updates against live state:** `ui/src/ui/views/chat.ts:597-602`
  - body: `updateAttachmentReadPending` adds the delta to `props.attachmentReadPending`, but those props are captured when the FileReader handler is attached. Concurrent reads can each update from the same stale value, so send can become enabled early or remain blocked after reads finish.
  - confidence: 0.92
- **[P2] Let attachment-independent commands bypass pending reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: This early return runs before stop-command, /btw, messageOverride, and local slash-command handling. Controls that do not send attachments can silently no-op while an attachment read is active.
  - confidence: 0.88
- **[P3] Expose processing state in the accessible name:** `ui/src/ui/views/chat.ts:1693`
  - body: The send button becomes disabled and changes its title while attachments are processing, but the accessible name remains Send or Queue, so screen-reader users do not hear why the control is unavailable.
  - confidence: 0.78
