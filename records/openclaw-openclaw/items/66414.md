## Summary

Keep open: the reported image-attachment loss is still source-reproducible on current main, but this PR is not merge-ready because the patch has stale-state blockers, conflicts with main, and only provides test output rather than real behavior proof.

## What This Changes

The PR adds provider-qualified image-capability lookup and a Control UI pending FileReader counter that blocks chat sends while image attachments load.

## Best Possible Solution

Land a rebased fix that accepts provider-qualified catalog ids through the shared lookup, applies attachment appends and pending-read deltas against live state, gates only attachment-dependent sends, and includes real Control UI/gateway proof.

## Reproduction Assessment

Yes, source-reproducible. Current main can miss provider-qualified catalog ids and can send before async FileReader callbacks append attachments; no live browser repro was run in this read-only review.

## Solution Assessment

No, not as written. The gateway lookup direction is useful, but the PR needs live-state attachment updates, narrower send gating, a rebase, and real behavior proof before it is the maintainable fix.

  - body: The disabled send button changes the title to `Processing attachment...` but leaves the accessible name as Send or Queue, so screen-reader users do not get the reason the control is unavailable.
  - confidence: 0.78
