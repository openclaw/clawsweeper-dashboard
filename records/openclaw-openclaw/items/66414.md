## Summary

Keep open: the PR addresses real current-main gaps, but the proposed patch still has blocking correctness issues around stale pending-count state and over-broad send blocking, and it needs rebasing onto the current run-controls split.

## What This Changes

The PR adds provider-qualified gateway image-capability matching and Control UI pending-FileReader state to prevent image attachments from being sent before they finish loading.

## Best Possible Solution

Preserve the useful gateway regression and Control UI intent, but rebase onto current main and implement pending attachment reads as owner-applied deltas or another live-state updater, append completed FileReader results against latest attachment state, guard only ordinary attachment-dependent sends, wire pending state through run-controls accessibility text, and add the required changelog entry.

## Reproduction Assessment

Yes. Current main has the bare-model catalog lookup path and asynchronous FileReader-before-send path, and the PR patch has a high-confidence static reproduction for concurrent reads because callbacks apply pending deltas from captured render props.

## Solution Assessment

No. The direction is right, but this patch is not the best solution because it computes pending counts from stale snapshots and blocks override/local commands that do not depend on in-flight attachments; a narrower owner-applied state contract is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending deltas against live state:** `ui/src/ui/views/chat.ts:600-602`
  - body: `updateAttachmentReadPending` derives the next counter from render-time props. With multiple concurrent FileReader callbacks, increments and decrements can collapse onto the same stale base and re-enable send before all attachments finish loading.
  - confidence: 0.93
- **[P2] Let override commands bypass pending reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: The pending-read return runs before `messageOverride` and local command handling, so button-triggered `/new`, `/compact`, `/stop`, and other attachment-independent actions can silently no-op while a file read is active.
  - confidence: 0.88
- **[P3] Reflect attachment processing in the button label:** `ui/src/ui/views/chat.ts:1687-1693`
  - body: The disabled send button title changes for attachment processing, but the accessible name remains Send or Queue. Screen-reader users get a disabled control without the reason.
  - confidence: 0.78
- **[P3] Add the user-facing changelog entry:** `ui/src/ui/app-chat.ts:273-276`
  - body: This changes user-visible Control UI attachment submission and gateway image-capability behavior, but the PR does not update `CHANGELOG.md` as required for user-facing fixes.
  - confidence: 0.84
