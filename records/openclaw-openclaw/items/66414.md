## Summary

Keep open. The PR targets a real current-main gap in Control UI image attachment submission and gateway vision-capability lookup, but the proposed patch still has blocking correctness issues around stale FileReader state and over-broad send blocking, and it is stale against the current extracted chat run-controls path.

## What This Changes

The PR adds provider-qualified gateway image-support catalog matching and Control UI pending-FileReader state to block chat submission while image attachments are still loading, with focused regression tests.

## Best Possible Solution

Preserve the useful gateway regression and Control UI intent, but rebase onto current main and implement pending attachment reads as state deltas applied by the owner, append completed FileReader results against the latest attachment state, guard only ordinary attachment-dependent sends, update run-controls accessibility text, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main has the bare-model catalog lookup and FileReader-before-send paths, and the PR patch itself has a high-confidence static reproduction for concurrent reads because each FileReader callback applies deltas to the captured props.attachmentReadPending value.

## Solution Assessment

No. The direction is right, but this patch is not the best solution because pending counts are computed from render snapshots and the send guard blocks override/local commands that do not depend on in-flight attachments; the safer fix is an owner-applied delta/append contract with a narrower send guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending deltas against live state:** `ui/src/ui/views/chat.ts:600-602`
  - body: `updateAttachmentReadPending` derives the next counter from the render-time `props.attachmentReadPending`. With multiple concurrent FileReader callbacks, increments and decrements can collapse onto the same stale base and re-enable send before all attachments finish loading, so the dropped-attachment race can still occur.
  - confidence: 0.93
- **[P2] Let override commands bypass pending reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: This early return runs before `messageOverride` and local command handling, so button-triggered `/new`, `/compact`, `/stop`, and other actions that do not send the pending attachments can silently no-op while a file read is active. Restrict the guard to normal sends that depend on in-flight attachment state.
  - confidence: 0.88
- **[P3] Reflect attachment processing in the button label:** `ui/src/ui/views/chat.ts:1687-1693`
  - body: The disabled button title changes to "Processing attachment...", but the accessible name remains "Send message" or "Queue message". Screen-reader users get a disabled control without the reason, so include the pending state in the aria label too.
  - confidence: 0.78
- **[P3] Add the user-facing changelog entry:** `ui/src/ui/app-chat.ts:273-276`
  - body: This changes user-visible Control UI attachment submission and gateway image-capability behavior, but the PR does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing fixes before merge.
  - confidence: 0.84
