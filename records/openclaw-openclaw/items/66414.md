## Summary

Keep open: the bug still appears valid on current main, but this PR is not merge-ready because the pending-read counter uses captured render props, attachment-independent commands can be blocked, real behavior proof is missing, and the branch currently conflicts with main.

## What This Changes

The PR extends gateway image-capability lookup for provider-qualified catalog ids and adds Control UI pending-attachment state to block chat send/Enter while FileReader reads are in flight.

## Best Possible Solution

Land a rebased fix that updates the shared catalog lookup, applies attachment pending deltas and completed-read appends against live state, gates only attachment-dependent sends, and includes real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main can miss provider-qualified image-capable catalog rows, and Control UI can send while FileReader callbacks are still pending because attachment state is only updated after async reads complete.

## Solution Assessment

No. The gateway lookup direction is useful, but this PR computes pending counts from captured render props and blocks override/local commands; a live-state delta/update contract plus narrower send gating is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending deltas against live state:** `ui/src/ui/views/chat.ts:597-601`
  - body: `updateAttachmentReadPending` adds `delta` to `props.attachmentReadPending`, but that value is the render-time snapshot captured by FileReader handlers. Concurrent reads can each increment or decrement from the same stale base, so send can be re-enabled early or remain blocked.
  - confidence: 0.92
- **[P2] Let attachment-independent commands bypass pending reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: The new early return runs before `messageOverride`, `/stop`, `/new`, and local slash-command handling. Enabled controls can silently no-op while an attachment read is active even though those actions do not send attachments.
  - confidence: 0.87
- **[P3] Expose processing state in the accessible name:** `ui/src/ui/views/chat.ts:1687-1693`
  - body: The disabled send button changes `title` to `Processing attachment...` but leaves `aria-label` as Send or Queue, so screen-reader users do not get the reason the control is unavailable.
  - confidence: 0.78
