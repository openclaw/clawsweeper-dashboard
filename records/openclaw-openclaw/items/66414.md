## Summary

Keep open. Current main still lacks both the provider-qualified image-support lookup and a Control UI pending-attachment-read guard, so the PR addresses a real remaining bug, but the proposed patch is stale against current main and has blocking correctness issues in the new pending-read state handling.

## What This Changes

The PR adds provider-qualified gateway image-support catalog matching plus Control UI state intended to block chat send while FileReader attachment reads are pending, with focused regression tests.

## Best Possible Solution

Keep the gateway regression test, but implement provider-qualified matching in the current shared catalog lookup path. On the Control UI side, apply pending-read deltas in owning state, append completed reads against the latest attachment state, block only ordinary sends that depend on in-flight attachments, preserve new-session/compact/stop behavior, and update the pending-state accessible label with focused tests for concurrent paste/select/drop reads.

## Reproduction Assessment

Yes. Static inspection of current main shows the exact catalog lookup and FileReader snapshot paths, and the PR patch shows where the new pending-read guard would fail with concurrent reads or override commands.

## Solution Assessment

No. The PR’s direction is useful, but the current implementation is not the best fix because it tracks pending reads from render-time props and blocks commands that do not send attachments; the safer approach is to apply deltas in owning state and guard only attachment-dependent sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending deltas against live state:** `ui/src/ui/views/chat.ts:600-602`
  - body: `updateAttachmentReadPending` computes the next count from `props.attachmentReadPending`, which is the render-time props object captured by each FileReader callback. Multiple concurrent reads can collapse increments or decrements onto the same stale base, re-enabling send before every attachment has finished loading. Apply deltas in the owning state instead.
  - confidence: 0.93
- **[P2] Let override commands bypass pending attachment reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: This early return runs before `messageOverride` and local command handling, so New session, compact, stop, and other actions that do not send the pending attachments can silently no-op during a read. Restrict the guard to normal user sends that depend on the in-flight attachment state.
  - confidence: 0.88
- **[P3] Reflect attachment processing in the send button label:** `ui/src/ui/views/chat.ts:1687-1693`
  - body: When pending reads disable the send button, the title changes to "Processing attachment..." but the accessible name remains "Send message" or "Queue message". Screen-reader users get a disabled button without the reason, so include the pending state in the aria label too.
  - confidence: 0.78
