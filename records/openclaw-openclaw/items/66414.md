## Summary

Keep open: the attachment-loss problem is real, but this PR is not merge-ready because the new pending-read counter is based on render-time props, the branch is currently conflicting with main, and the external PR still has only unit-test proof.

## What This Changes

This PR changes gateway image-capability lookup and Control UI chat state so image attachments are not sent until FileReader work finishes.

## Best Possible Solution

Rebase onto current main, move provider-qualified matching into the shared model catalog lookup, and make attachment reads update pending state and attachment appends against live app state before adding real UI/Gateway proof.

## Reproduction Assessment

Yes at source level. Current main and the PR both use asynchronous FileReader callbacks around chat attachment state, and the gateway lookup path can be checked directly through resolveGatewayModelSupportsImages and findModelCatalogEntry.

## Solution Assessment

No, not as submitted. The gateway direction is plausible, but the UI fix adds a snapshot-based counter instead of a live state update contract and blocks override sends too broadly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Apply pending-read deltas to live state:** `ui/src/ui/views/chat.ts:600-601`
  - body: `props` is a render-time snapshot, so multiple pasted/selected/dropped files can all compute the next pending count from the same stale value. That can set the counter back to 0 while other FileReaders are still running, re-enabling send early and preserving the attachment-loss race this PR is trying to fix.
  - confidence: 0.92
- **[P2] Bypass the pending guard for override commands:** `ui/src/ui/app-chat.ts:273-275`
  - body: This guard returns before slash/local command handling even when `messageOverride` is supplied, so actions such as compact/reset-style sends that do not include attachment payloads can become enabled no-ops while an attachment is loading. Limit the guard to composer sends that would depend on pending attachments.
  - confidence: 0.78
- **[P3] Reflect attachment processing in the button label:** `ui/src/ui/views/chat.ts:1693`
  - body: The send button becomes disabled and changes its title while attachments are processing, but the accessible name remains `Send message` or `Queue message`. Screen reader users do not get the reason the control is unavailable, so the aria-label should follow the pending state too.
  - confidence: 0.76
