## Summary

Keep open: the underlying image-attachment bugs are still source-reproducible on current main, but this PR is not merge-ready because real behavior proof is missing and the UI pending-read implementation can still suppress or mis-time sends.

## What This Changes

The PR changes gateway image-capability lookup to recognize provider-qualified catalog ids and adds Control UI pending-read state that blocks chat sends while FileReader attachment reads are in flight.

## Best Possible Solution

Land a refreshed fix that centralizes provider-qualified catalog-id matching, applies FileReader pending deltas and attachment appends against live state, gates only attachment-dependent sends, and includes real browser or runtime proof.

## Reproduction Assessment

Yes, at source level. Current main can miss provider-qualified image-capable catalog rows and can send while FileReader callbacks have not yet appended attachment state; I did not run a browser repro in this read-only review.

## Solution Assessment

No. The gateway direction is useful, but the UI fix computes pending counts from captured props and blocks more send paths than the attachment race requires; live-state deltas/appends with narrower gating are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending-read deltas against current state:** `ui/src/ui/views/chat.ts:600-602`
  - body: `updateAttachmentReadPending` adds the delta to `props.attachmentReadPending`, but those props are captured when the FileReader handler is attached. Concurrent reads can each update from the same stale value, so send can become enabled early or remain blocked after reads finish.
  - confidence: 0.92
- **[P2] Let attachment-independent commands bypass pending reads:** `ui/src/ui/app-chat.ts:276-278`
  - body: This early return runs before stop-command, /btw, messageOverride, and local slash-command handling. Controls that do not send attachments can silently no-op while an attachment read is active.
  - confidence: 0.88
- **[P3] Expose processing state in the accessible name:** `ui/src/ui/views/chat.ts:1693`
  - body: The send button becomes disabled and changes its title while attachments are processing, but the accessible name remains Send or Queue, so screen-reader users do not hear why the control is unavailable.
  - confidence: 0.78
