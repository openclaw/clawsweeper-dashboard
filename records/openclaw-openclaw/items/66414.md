## Summary

Keep open: the underlying image-attachment bug is still present on current main and the PR has useful direction, but it should not merge as-is because the UI pending counter uses render-time state, the send guard blocks attachment-independent commands, the branch conflicts with current main, and external real behavior proof is still absent.

## What This Changes

The PR adds provider-qualified gateway image-capability lookup and tracks pending Control UI attachment reads so send/Enter are blocked while FileReader work is active.

## Best Possible Solution

Land a rebased fix that resolves the shared catalog lookup, applies attachment pending deltas and completed-read appends against live state, gates only attachment-dependent sends, and includes real Control UI proof.

## Reproduction Assessment

Yes, at source level. Current main can miss provider-qualified image-capable catalog rows, and the Control UI can send while FileReader callbacks are still pending because attachment state is only updated after the async read completes.

## Solution Assessment

No. The gateway direction is useful, but this PR computes pending counts from stale render props and blocks override/local commands; a live-state delta/update contract plus a narrower send guard is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending deltas against live state:** `ui/src/ui/views/chat.ts:597-601`
  - body: `updateAttachmentReadPending` adds `delta` to `props.attachmentReadPending`, which is the render-time value captured by FileReader handlers. Multiple concurrent reads can each increment or decrement from the same stale base, so send can be re-enabled before all attachments finish or remain blocked.
  - confidence: 0.92
- **[P2] Let attachment-independent commands bypass pending reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: The new early return runs before `messageOverride`, `/stop`, `/new`, and local slash-command handling, so enabled command controls can silently no-op while an attachment read is active even when they would not send attachments.
  - confidence: 0.87
- **[P3] Expose processing state in the accessible name:** `ui/src/ui/views/chat.ts:1687-1693`
  - body: The disabled send button changes `title` to `Processing attachment...` but leaves `aria-label` as Send or Queue, so screen-reader users do not get the reason the control is unavailable.
  - confidence: 0.78
