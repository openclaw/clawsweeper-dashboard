## Summary

Keep open. The PR targets a real source-visible image attachment problem, but the current head is conflicting, lacks real behavior proof, and still has blocking UI correctness issues in the pending-read counter and send gating that reviewers already called out.

## What This Changes

The PR changes gateway image-capability lookup and Control UI chat attachment loading so image attachments are less likely to be submitted before FileReader work completes.

## Best Possible Solution

Rebase and land a narrower fix that centralizes provider-qualified catalog id matching, applies FileReader pending/appends against live state, gates only attachment-dependent sends, and proves the Control UI flow in a real browser.

## Reproduction Assessment

Yes, at source level. Current main can miss provider-qualified image-capable catalog rows and can send while FileReader callbacks have not yet appended attachment state; I did not run a browser repro in this read-only review.

## Solution Assessment

No. The gateway direction is useful, but the UI fix computes pending counts from captured props and blocks more send paths than the attachment race requires; a live-state delta/append contract with narrower gating is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply attachment pending updates against live state:** `ui/src/ui/views/chat.ts:597-601`
  - body: `updateAttachmentReadPending` adds the delta to `props.attachmentReadPending`, but those props are captured when the FileReader handler is attached. Concurrent reads can each update from the same stale value, so send can become enabled early or remain blocked after reads finish.
  - confidence: 0.92
- **[P2] Let attachment-independent commands bypass pending reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: This early return runs before `messageOverride`, stop-command, and local slash-command handling. Controls that do not send attachments can silently no-op while an attachment read is active.
  - confidence: 0.87
- **[P3] Expose processing state in the accessible name:** `ui/src/ui/views/chat.ts:1693`
  - body: The send button becomes disabled and changes its title while attachments are processing, but the accessible name remains Send or Queue, so screen-reader users do not hear why the control is unavailable.
  - confidence: 0.78
