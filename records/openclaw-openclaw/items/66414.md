## Summary

Keep open: the PR targets a real image-attachment bug, but the submitted patch is not ready because it still has stale async state hazards, blocks attachment-independent commands, is now conflicting with current main, and lacks real after-fix behavior proof.

## What This Changes

The PR changes gateway image-capability lookup and Control UI chat attachment state so image reads are tracked before sending.

## Best Possible Solution

Rebase and keep the useful gateway lookup direction, but apply attachment pending deltas against live state, append completed reads against current attachments, gate only attachment-dependent sends, and wire the current run-controls accessibility state.

## Reproduction Assessment

Yes, at source level. Current main can miss provider-qualified image-capable catalog ids, and its async FileReader handlers can race handleSendChat because sends snapshot attachments without any pending-read guard.

## Solution Assessment

No. The direction is right, but this patch derives pending counts from stale render props and blocks override/local commands; a live-state delta/update contract plus a narrower send guard is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending deltas against live state:** `ui/src/ui/views/chat.ts:597-601`
  - body: `updateAttachmentReadPending` computes the next count from render-time props. With concurrent FileReader callbacks, multiple increments or decrements can apply against the same stale value, re-enabling send before all reads finish or leaving the composer blocked.
  - confidence: 0.92
- **[P2] Let attachment-independent commands bypass pending reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: The new pending-read return runs before messageOverride and local slash-command handling, so actions like `/compact`, `/stop`, or `/new` can silently no-op while a file read is active even though they do not send attachments.
  - confidence: 0.87
- **[P3] Expose processing state in the accessible name:** `ui/src/ui/views/chat.ts:1687-1693`
  - body: The PR disables the send button and changes its title while attachments are processing, but leaves the accessible name as Send or Queue, so screen-reader users do not get the reason for the disabled state.
  - confidence: 0.78
