## Summary

Keep open: current main still has the reported gateway/UI attachment gaps and the PR is a useful focused fix attempt, but this patch has blocking stale-state and command-routing regressions plus only test-based proof.

## What This Changes

The PR updates gateway image-capability lookup/tests and Control UI chat state/render/send/tests to track pending FileReader attachment reads before sending.

## Best Possible Solution

Rebase and preserve the useful gateway lookup fix, but implement attachment pending state through live owner-applied updates, append completed reads against latest attachment state, guard only attachment-dependent sends, and update run-controls accessibility before merge.

## Reproduction Assessment

Yes. Source inspection shows current main can miss provider-qualified image-capable catalog ids and can send before FileReader callbacks finish; the PR patch also has a high-confidence static reproduction for stale pending counts because callbacks apply deltas from captured props.

## Solution Assessment

No. The direction is right, but this patch is not the best fix because it derives pending counts from render snapshots and blocks attachment-independent commands; a live-state updater plus a narrower send guard is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending deltas against live state:** `ui/src/ui/views/chat.ts:597-601`
  - body: `updateAttachmentReadPending` derives the next count from render-time props. With concurrent FileReader callbacks, multiple increments or decrements can apply against the same stale value, re-enabling send before all reads finish or leaving the composer blocked.
  - confidence: 0.92
- **[P2] Let attachment-independent commands bypass pending reads:** `ui/src/ui/app-chat.ts:273-276`
  - body: The new pending-read return runs before messageOverride and local slash-command handling, so actions like `/compact`, `/stop`, or other local commands can silently no-op while a file read is active even though they do not send attachments.
  - confidence: 0.87
- **[P3] Expose processing state in the button accessible name:** `ui/src/ui/views/chat.ts:1687-1693`
  - body: The PR disables the send button and changes its title while attachments are processing, but leaves the accessible name as Send or Queue, so screen-reader users do not get the reason for the disabled state.
  - confidence: 0.78
