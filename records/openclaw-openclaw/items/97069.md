## Summary

Keep open: current main still has the misleading approval explanation path. The code can mark redirected or otherwise non-persistable exec approvals as one-shot, but the UI and forwarded approval messages collapse every missing Allow Always action into the ask-always policy wording.

## What This Changes

The issue reports that redirected exec commands hide reusable approval with a misleading ask-always policy explanation even when the effective gateway policy is ask=on-miss.

## Best Possible Solution

Preserve the one-shot security behavior, but carry a typed unavailable-reason through approval registration, gateway state, Control UI parsing, and forwarded/chat renderers so non-persistable commands get a specific explanation.

## Reproduction Assessment

Yes from source, with high confidence but not a live run: current main has a one-shot no-reusable-pattern path and the visible renderers still use ask-always wording whenever Allow Always is absent. The issue body also gives concrete steps and screenshots for `openclaw --version 2>&1` on the gateway host.

## Solution Assessment

Yes: the best fix is a narrow explanation/protocol rendering repair, not changing exec policy or allowing persistence for commands the planner cannot make reusable safely.
