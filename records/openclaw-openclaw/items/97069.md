## Summary

Keep open: current main and v2026.6.10 still collapse one-shot/non-persistable Allow Always into the ask-always policy wording, and open fix PRs now reference this issue but are not merged.

## What This Changes

The issue asks for approval prompts to explain non-persistable one-shot commands separately from ask=always policy when Allow Always is unavailable.

## Best Possible Solution

Land a single canonical fix that preserves one-shot safety, carries a typed Allow Always unavailable reason through the gateway approval protocol, and renders reason-specific copy in UI, forwarded, native, and reaction approval surfaces.

## Reproduction Assessment

Yes from source with high confidence. Current main creates one-shot non-persistable Allow Always decisions, then renders the ask-always policy explanation whenever allow-always is absent.

## Solution Assessment

Yes: the best fix is reason plumbing and renderer copy, not making non-persistable commands durable or adding a config option. The open PRs at https://github.com/openclaw/openclaw/pull/97077 and https://github.com/openclaw/openclaw/pull/97081 are implementation candidates to review before closing this issue.
