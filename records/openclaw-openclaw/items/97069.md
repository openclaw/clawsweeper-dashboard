## Summary

Keep open: current main still collapses one-shot/non-persistable exec approvals into the ask-always explanation, and the issue has open linked fix PRs that should be reviewed before closure.

## What This Changes

The issue reports that redirected exec commands hide Allow Always with the wrong ask-always explanation instead of a non-persistable command explanation.

## Best Possible Solution

Keep this issue open while maintainers review and land one canonical fix PR, preferably one that carries a typed one-shot/non-persistable reason through all approval surfaces instead of changing only one renderer.

## Reproduction Assessment

Yes from source: current main removes `allow-always` for both `ask=always` and one-shot persistence, then renders a policy-only explanation whenever `allow-always` is absent. The full gateway/UI flow was not run because this is a read-only review.

## Solution Assessment

No, current main is not the best way to solve it because it cannot distinguish policy from non-persistable commands in the user-facing prompt. The best fix should preserve the one-shot approval restriction while carrying an explicit unavailable reason to reply, forwarded, UI, and reaction renderers.
