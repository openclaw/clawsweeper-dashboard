## Summary

Keep open: current main and v2026.6.10 still collapse ask=always and one-shot/non-persistable approvals into the same missing Allow Always state, and the rendered copy still blames the effective policy. Several open PRs target this issue, but none has merged, so the issue should remain the canonical tracker.

## What This Changes

The issue requests accurate Allow Always unavailable copy for redirected or otherwise non-persistable exec commands when the effective approval policy is not ask=always.

## Best Possible Solution

Land one canonical fix that carries an explicit Allow Always unavailable reason, or an equivalent production-scoped fact, from gateway/node approval producers through tool details, view models, reply/forwarder/reaction renderers, and Control UI, with regression coverage and visible prompt proof.

## Reproduction Assessment

Yes. The issue body gives concrete screenshots, and current-main source shows one-shot persistence removes Allow Always while the renderers still emit the policy-only explanation; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes. A narrow production reason-plumbing fix is the best path; renderer-local inference from `allowedDecisions` alone is too weak because current main intentionally uses the same decision list for both causes.
