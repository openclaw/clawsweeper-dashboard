## Summary

Keep open: current main still collapses one-shot/non-persistable exec approvals into the ask-always explanation, and v2026.6.10 has the same visible wording. The open harmless-redirection PR overlaps the specific `2>&1` example but does not fix the general missing unavailable-reason path.

## What This Changes

The issue asks for exec approval prompts to explain that Allow Always is unavailable because a redirected command is non-persistable/one-shot, instead of claiming the effective policy is ask-always.

## Best Possible Solution

Preserve the one-shot security behavior, but carry a typed Allow Always unavailable reason through approval requests and render reason-specific copy in Control UI, forwarded replies, and channel/native approval surfaces.

## Reproduction Assessment

Yes from source, with high confidence but no live run in this read-only review. Current main has a one-shot `no-reusable-pattern` path and still renders the ask-always policy wording whenever Allow Always is absent.

## Solution Assessment

Yes: the best fix is a narrow reason-plumbing and renderer-copy repair, not changing the exec policy or making unsafe commands persistable. The harmless-redirection PR is only a partial overlap because it does not fix other one-shot unavailable causes.
