## Summary

Keep open: current main still collapses one-shot/non-persistable exec approvals into the ask-always policy explanation, and several open closing PRs now own candidate fixes. This issue should remain the canonical report until one focused fix lands.

## What This Changes

The issue asks OpenClaw to explain that Allow Always is unavailable because a redirected command is non-persistable, not because the effective approval policy is ask=always.

## Best Possible Solution

Land one focused fix that preserves one-shot safety while carrying a typed unavailable reason through gateway approval requests, agent details, Control UI, forwarded replies, reaction prompts, and channel/native approval surfaces.

Yes, from source with high confidence: current main creates one-shot/non-persistable approval state for unsafe reusable grants, then the renderers show the ask-always policy warning whenever Allow Always is absent. I did not run a live gateway prompt in this read-only review.

## Solution Assessment

Yes: the best fix is reason plumbing plus reason-specific renderer copy, not changing the approval policy or making non-persistable commands durable. The existing open PRs should be reviewed against that owner-boundary shape.
