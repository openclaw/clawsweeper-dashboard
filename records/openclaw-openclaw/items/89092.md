## Summary

Keep open. The PR targets a real macOS LaunchAgent path problem and includes useful live launchd proof, but the new resolver redirects non-matching custom HOME layouts to `/Users/<USER>` whenever USER/LOGNAME is set, which can break existing custom-home installs.

## What This Changes

This PR adds a shared TypeScript LaunchAgent plist path resolver, updates macOS gateway install/restart/status/cleanup call sites plus the Swift app resolver to use boot-volume `/Users/<shortName>` paths for external HOME values, and adds TS/Swift tests.

## Best Possible Solution

Narrow the resolver to the proven external `/.../Users/<shortName>` shape, preserve non-matching configured HOME paths, mirror the same rule in Swift, and then land with the supplied launchd proof plus focused tests.

## Reproduction Assessment

Yes. Current main source joins LaunchAgent plist paths from HOME, and the PR body supplies live after-fix macOS launchd proof for `HOME=/Volumes/.../Users/x`; the introduced custom-HOME fallback issue is also source-reproducible from the new resolver.

## Solution Assessment

No. Centralizing LaunchAgent plist path resolution is the right layer, but the current fallback is too broad; the maintainable fix is to preserve non-matching HOME paths and share that narrower rule across TS and Swift.
