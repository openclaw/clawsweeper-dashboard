## Summary

The reported hybrid-search defect is real, but this draft is not the best fix: it heuristically collapses overlapping chunks, while the linked alternative preserves keyword-only results at the filtering boundary. This PR also lacks real CJK/trigram behavior proof.

## What This Changes

The PR changes memory-core hybrid result merging so overlapping keyword and vector chunks with different IDs share scores, with regression tests.

## Best Possible Solution

Fix the post-merge filtering boundary so valid keyword-only hits survive `minScore` without pretending they are the same chunk as an overlapping vector result, and validate the canonical implementation with a real CJK/trigram search.

## Reproduction Assessment

Yes. The linked issue provides a concrete trigram/CJK configuration, and current source shows how an exact-ID mismatch plus `minScore` filtering loses the keyword contribution; no live reproduction was independently run during this read-only review.

## Solution Assessment

No. The narrower maintainable fix is to preserve keyword-only results at the filtering boundary, not infer chunk identity from line overlap and transfer the keyword score and snippet to one selected vector chunk.

Full review comments:

- **[P1] Preserve the keyword hit instead of assigning it to one overlap:** `extensions/memory-core/src/memory/hybrid.ts:91-99`
  - body: This fallback treats any same-file line overlap as chunk identity, chooses only one vector candidate, and then omits the keyword entry. With sliding or otherwise overlapping vector chunks, one keyword range can intersect multiple candidates, so the tie-break transfers its score/snippet to an arbitrary chunk and suppresses the actual keyword hit. Fix the downstream `minScore` filtering to retain keyword-only results, as the linked alternative does, or establish a canonical one-to-one identity rather than using overlap.
  - confidence: 0.9
- **[P2] Keep the shared FTS token normalizer:** `extensions/memory-core/src/memory/hybrid.ts:30-35`
  - body: The scoring fix does not require changing query token normalization. Replacing `normalizeStringEntries` with a local trim/filter implementation creates a second normalization path and can drift from the plugin SDK contract; leave `buildFtsQuery` unchanged in this PR.
  - confidence: 0.86
