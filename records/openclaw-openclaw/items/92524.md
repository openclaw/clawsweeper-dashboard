## Summary

This PR addresses a real hybrid-search loss, but it repairs the wrong boundary: current main already preserves differing IDs during merge and drops lower weighted keyword hits later during threshold selection. The proposed overlap heuristic can instead collapse distinct stored chunks, so maintainers should choose the canonical semantics before merging.

## What This Changes

The branch merges same-source/path keyword and vector results whose line ranges overlap despite differing chunk IDs, adds overlap and disjoint-range tests, and replaces the shared FTS token normalizer with equivalent local processing.

## Best Possible Solution

Preserve independently keyed FTS hits through post-merge threshold selection, retain exact stored-ID merge semantics, and verify the result with a real indexed CJK/trigram search.

## Reproduction Assessment

Yes, from source with high confidence: differing IDs survive the merge, then lower weighted keyword-only hits are discarded whenever another strict result survives. A real indexed reproduction was not independently run.

## Solution Assessment

No. The verified loss occurs during post-merge filtering, while this patch changes chunk identity and can collapse a keyword range into an arbitrary overlapping vector chunk.
