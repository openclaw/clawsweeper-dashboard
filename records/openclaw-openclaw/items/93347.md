## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Bind client restrictions to the run, not the session key:** `src/gateway/session-client-id-registry.ts:26`
  - body: The new registry stores only the last `client.id` for a `sessionKey`, while the MCP loopback resolver later reads that session key without knowing which run is making the tool call. Two clients or turns sharing a session can overwrite each other, and a later non-`chat.send` turn within the TTL can inherit stale restrictions, so tool availability can be widened or narrowed for the wrong run. Thread this through run/connection lifecycle state before using it for tool exposure.
  - confidence: 0.89
- **[P1] Do not treat client.id as an enforcement boundary:** `src/gateway/server-methods/chat.ts:3231`
  - body: `clientInfo?.id` comes from the gateway connect payload; current protocol code only normalizes it to a known product id, it does not prove the caller is that surface. Because this config applies restrictions only when the claimed id matches a configured key, a client with the same auth/scopes can claim another canonical id and avoid the narrowed policy. Use authenticated device/role/scope/connection metadata, or make the feature explicitly advisory instead of an enforceable restriction.
  - confidence: 0.86
- **[P2] Add the config docs and schema-help entries:** `src/config/zod-schema.ts:1072`
  - body: This introduces a user-facing `gateway.tools.byClientId` policy key, but the existing config reference and schema help/label surfaces still describe only `gateway.tools.allow` and `gateway.tools.deny`. Add the docs/help/label coverage and explain empty-allow fail-closed behavior plus the selector's security limits so operators can safely configure it.
  - confidence: 0.82
