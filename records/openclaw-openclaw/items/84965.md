## Summary

This PR is still necessary: current main still sends `audioAsVoice` Discord payloads through `discordVoice` before any visible text delivery, while the PR adds a scoped fallback for the known unavailable-adapter failure and supplied real Discord API readback proof. I found no blocking patch defects, but the message-delivery behavior change still needs maintainer acceptance before merge.

## What This Changes

The PR updates the Discord outbound plugin to fall back from an unavailable `discordVoice` send to normal Discord text for visible `audioAsVoice` payloads, preserves reply targeting, and adds regression coverage for that fallback.

## Best Possible Solution

Land the scoped Discord plugin fallback if maintainers accept the text-delivery failure mode, then let the linked bug close when the PR merges.

## Reproduction Assessment

Yes from source, though I did not run it: current main routes `audioAsVoice` media through `discordVoice` first, and default deps can produce the exact unavailable-adapter message before text delivery. The linked issue provides concrete cron steps and matching failure text.

## Solution Assessment

Yes: the PR keeps the voice path first and falls back only for the known unavailable-adapter error when visible text exists, which is a narrow Discord plugin-boundary fix. Maintainers may still choose to require full cron proof before merge.
