## Summary

Keep open: current main has Telegram polling stall and spooled-handler recovery, but it still does not implement the reported Telegram source-timestamp-to-dispatch latency watchdog or guarantee a visible DM failure response when a voice-message turn aborts without a final reply.

## What This Changes

The issue requests prompt Telegram voice-note dispatch from Telegram receipt to agent execution, plus a visible failure path when the resulting voice-message turn aborts before replying.

## Best Possible Solution

Add source-message latency measurement from Telegram `message.date` through dispatch start, expose diagnostics/status before enabling restart behavior, and add a bounded visible aborted-turn fallback for voice-message DM flows with focused Telegram tests and live proof.

## Reproduction Assessment

No high-confidence live reproduction was established. The issue provides concrete production timestamps, and current source inspection shows plausible gaps around Telegram `message.date` latency measurement and no-response fallback handling.

## Solution Assessment

Unclear as written: a watchdog is a reasonable direction, but automatic restart thresholds need maintainer approval. A safer first step is measurement/status plus a bounded visible fallback before changing restart policy.
