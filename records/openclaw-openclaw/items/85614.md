## Summary

Keep open: the report is fresh, has a matching v2026.5.20 Telegram confirmation, and current main has related heartbeat deferral/filtering safeguards but no direct proof that the first post-heartbeat user turn can no longer be steered or swallowed by heartbeat context.

## What This Changes

The issue asks OpenClaw to treat the first normal user message after a heartbeat poll as an independent user turn instead of answering with HEARTBEAT_OK or heartbeat-related content.

## Best Possible Solution

Add a regression for a heartbeat poll followed immediately by a normal channel user message, then fix the heartbeat transcript/session boundary so heartbeat-only state cannot steer that user turn while intentional visible heartbeat alerts remain available as history.
