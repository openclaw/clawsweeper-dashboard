## Label Justifications

- `P1`: The report describes a regression that effectively swallows the first user message after heartbeat for daily chat users.
- `impact:message-loss`: The first post-heartbeat user message receives unrelated heartbeat output instead of an answer to the message.
- `impact:session-state`: The suspected failure is heartbeat turn state bleeding into the next normal user turn in the same session.

## Summary

Keep open: current main has related heartbeat/session safeguards, but I did not find a merged fix that proves the completed-heartbeat to next-user-turn contamination is gone. Default heartbeats still run through the shared session unless `isolatedSession` is enabled, and adjacent open heartbeat reports show this area still has pending-delivery and preemption gaps.

## What This Changes

The issue asks for the first user turn after a heartbeat poll to be treated as a normal independent user message instead of continuing heartbeat mode.

## Best Possible Solution

Add focused regression coverage and a narrow runtime fix so a completed heartbeat turn cannot carry heartbeat trigger, prompt, pending delivery, or provider conversation state into the immediately following normal user turn.

## Reproduction Assessment

Yes, at source level with medium confidence: current main still runs default heartbeats through the shared session and labels only heartbeat turns as `isHeartbeat`, while the issue provides a concrete sequence where the next user turn behaves as if that label or prompt state persisted. I did not execute a live provider/channel repro.

## Solution Assessment

Unclear because there is no patch yet, but the likely best fix is a narrow heartbeat-to-user turn boundary repair with regression tests. Using `heartbeat.isolatedSession: true` is a workaround, not a complete fix for default behavior.
