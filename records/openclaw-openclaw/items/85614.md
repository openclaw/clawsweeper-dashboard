## Label Justifications

- `P2`: The issue describes a user-visible heartbeat/session regression with meaningful UX impact, now fixed on main but not yet proven shipped.
- `impact:message-loss`: The reported first post-heartbeat user message could be swallowed or answered with heartbeat output instead of a normal reply.
- `impact:session-state`: The failure was caused by heartbeat/background work colliding with the active reply/session lane.

## Summary

Current main now covers the reported heartbeat/user-turn collision through merged shared reply-turn admission: visible user turns wait for lane ownership while heartbeat turns skip instead of continuing in the same active reply context. The fix is main-only after the latest release, so users would need a build containing the merged fix commit rather than v2026.5.22.

## What This Changes

The issue reports that the first normal user message after a heartbeat poll can be answered as if it were still part of the heartbeat context.

## Best Possible Solution

Keep the merged shared reply-turn admission behavior and ask for a new focused report only if a build containing `62b51a6295eeeb74430580e37fd0cbf34803a0a8` still reproduces the first-message-after-heartbeat failure.

## Reproduction Assessment

Yes for the pre-fix failure: the merged PR discussion and regression tests model a heartbeat/background turn occupying the reply-operation lane while a visible user turn arrives. I did not run a live repro in this read-only review, and current main contains the fix.

## Solution Assessment

Yes. A shared reply-turn admission gate is the maintainable fix because it establishes priority isolation between heartbeat and visible user turns without adding a new config knob or making every heartbeat isolated by default.
