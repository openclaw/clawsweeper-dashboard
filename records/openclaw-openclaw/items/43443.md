## Summary

Keep open. The EIO-only bootstrap retry idea is still plausible and is not implemented across current main's shared launchd bootstrap paths, but this branch is not safe to merge as-is because it adds PID-only SIGKILL escalation and is stale relative to the current kickstart-first restart model.

## What This Changes

The PR adds launchd EIO bootstrap retry logic, replaces the old PID wait with a 15-second wait plus SIGKILL escalation, and expands launchd unit coverage for bootstrap retry and PID wait behavior.

## Best Possible Solution

Implement a shared bounded EIO-only `launchctl bootstrap` retry helper on current main and use it from `bootstrapLaunchAgentOrThrow`, `repairLaunchAgentBootstrap`, and detached handoff bootstrap fallbacks while preserving the kickstart-first restart model. Add force-kill escalation only with explicit maintainer approval and a launchd job identity check before signaling.

## Reproduction Assessment

Yes for the blocking PR finding: the PR branch shows `process.kill(pid, "SIGKILL")` after only a numeric-PID liveness probe. The original EIO race has a concrete manual reproduction in the PR body, but this review did not rerun it live on macOS.

## Solution Assessment

No. EIO-only retry is the narrower maintainable direction, but this PR mixes it with unsafe PID-based force killing and an older restart lifecycle; a current-main retry-focused patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify the saved PID before SIGKILL:** `src/daemon/launchd.ts:440`
  - body: The helper reads a numeric PID from `launchctl print`, waits, verifies only with `kill(pid, 0)`, then sends `SIGKILL`. If the LaunchAgent exits and the PID is reused before the final probe or between the probe and signal, this can kill an unrelated local process. Drop the force-kill path or verify the process still belongs to the expected launchd job before signaling.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete local process-control safety risk by sending SIGKILL to a saved PID without verifying launchd job identity.

Concerns:

- **[medium] SIGKILL can target a reused PID:** `src/daemon/launchd.ts:440`
  - body: The proposed escalation sends SIGKILL to a saved PID after `kill(pid, 0)`. PID reuse can satisfy that check for a different process, creating a local availability and safety risk in a service-management command.
