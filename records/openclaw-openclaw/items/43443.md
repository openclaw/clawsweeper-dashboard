## Summary

Keep open: the PR addresses a plausible launchd EIO race, but it is stale against current main's kickstart-based restart lifecycle and introduces PID-only SIGKILL escalation that should not merge as-is.

## What This Changes

The PR adds launchd bootstrap EIO retries, PID wait escalation with SIGKILL, install/restart/repair call-site changes, and launchd unit coverage.

## Best Possible Solution

Rework on current main as a bounded EIO-only retry for actual bootstrap fallback, install, repair, and detached handoff paths while preserving the kickstart restart lifecycle; add focused tests and a changelog entry.

## Reproduction Assessment

Yes for the blocking PR findings from source inspection: the PR still uses a bootout/bootstrap restart shape and sends SIGKILL after only PID liveness. The original launchctl EIO race is plausible from the PR's manual test but was not rerun live here.

## Solution Assessment

No. The EIO retry direction is useful, but the maintainable fix should be a current-main retry rework that preserves `kickstart -k` and removes or identity-verifies any force-kill behavior.
