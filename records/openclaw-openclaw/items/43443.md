## Summary

Keep open: current main still has raw bootstrap paths where EIO retry may be useful, but this PR is stale against the current launchd restart lifecycle and adds unsafe PID-only SIGKILL escalation, so it should not merge as-is.

## What This Changes

The PR adds launchd EIO bootstrap retry logic, PID wait escalation, install/restart/repair call-site changes, and launchd unit coverage.

## Best Possible Solution

Rework on current main as bounded EIO-only bootstrap retry for real bootstrap fallback, install, and doctor repair paths; add force-kill only with explicit maintainer approval and launchd job identity verification.

## Reproduction Assessment

Yes for the blocking PR findings from source inspection: the PR sends SIGKILL after only a numeric PID liveness probe and keeps a bootout/bootstrap restart shape that current main intentionally replaced. The original launchctl EIO race is manually described in the PR body but was not rerun live.

## Solution Assessment

No. The EIO retry direction is useful, but this PR mixes it with unsafe PID-based killing and an older restart lifecycle; a narrower current-main retry rework is safer.
