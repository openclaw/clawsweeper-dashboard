## Summary

Keep open: the EIO retry direction is still relevant, but this PR is not merge-ready because it adds PID-only SIGKILL escalation and is based on an older launchd restart lifecycle than current main.

## What This Changes

The PR adds EIO-specific launchctl bootstrap retries, SIGKILL escalation while waiting for launchd restart PIDs, retry coverage for install/repair paths, and launchd unit tests.

## Best Possible Solution

Port a bounded EIO-only bootstrap retry to current main across direct bootstrap, doctor repair, and detached handoff fallback paths; add force-kill escalation only if maintainers approve it and the code verifies launchd job identity before signaling.

## Reproduction Assessment

Yes for the blocking PR finding: the supplied PR diff shows numeric PID liveness followed by `SIGKILL` without a launchd job identity check. The original EIO race has a concrete manual macOS reproduction in the PR body, and current main still has raw bootstrap paths, but I did not rerun the live macOS scenario.

## Solution Assessment

No. EIO-only retry is a narrower maintainable fix, while this PR mixes that with PID-based force-kill escalation and does not cover the newer current-main handoff bootstrap path.

## Review Findings

## Security Review

Status: needs_attention
