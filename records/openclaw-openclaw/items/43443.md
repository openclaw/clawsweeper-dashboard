## Summary

Keep open: the EIO retry direction is useful and current main still has direct bootstrap paths, but this branch is not a safe merge candidate because it is based on the older bootout/bootstrap restart lifecycle and adds PID-only SIGKILL escalation.

## What This Changes

The PR adds EIO retry handling around `launchctl bootstrap`, PID wait escalation with SIGKILL, install/restart/repair call-site changes, and launchd unit coverage.

## Best Possible Solution

Rework against current main as a bounded EIO-only retry on actual bootstrap fallback, install, repair, and detached handoff paths while preserving the kickstart restart lifecycle and avoiding PID-only SIGKILL unless job identity is verified.

## Reproduction Assessment

Yes for the blocking PR findings from source inspection: the branch still uses a bootout/bootstrap restart shape and sends SIGKILL after only PID liveness. The original launchd EIO race is plausible from the PR's manual trap test, but I did not rerun a live macOS launchd reproduction in this read-only review.

## Solution Assessment

No. The EIO retry direction is useful, but the safer maintainable fix is a current-main retry rework that preserves `kickstart -k` restart behavior and removes or identity-verifies force-kill behavior.

## Review Findings

  - body: Current main restarts LaunchAgents with `launchctl kickstart -k` and tests assert normal restart does not call `bootout` or `bootstrap`.
