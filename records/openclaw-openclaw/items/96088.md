## Label Justifications

- `P2`: This is a shipped Gmail watcher reliability problem with limited blast radius and a clear affected surface, but it is not an emergency crash, data-loss, or security bypass.
- `impact:auth-provider`: The reported failure is an expired or revoked Gog OAuth credential that OpenClaw does not currently expose as distinct auth health.
- `impact:message-loss`: A stale Gmail watch can leave inbound Gmail events undelivered while the local listener still appears live.

## Summary

Keep open: current `main` and the latest release still match the reported behavior, with watch-start and renewal failures logged but not converted into a Gmail watcher health state, and no current status/doctor surface exposes stale watcher coverage distinctly from transient network failures.

## What This Changes

The issue requests that Gmail watcher OAuth invalidation and watch-renewal failure be surfaced as degraded or terminal health instead of appearing live while coverage is stale.

## Best Possible Solution

Add watcher-owned health state that distinguishes terminal OAuth invalidation from transient network failures, bounds renewal noise, and exposes that state through existing Gateway/status or doctor surfaces while keeping Gog responsible for the underlying credential repair.

## Reproduction Assessment

Yes at source level: current `main` still logs failed watch-start/renewal attempts and keeps or starts the watcher process, and status/doctor inspection did not reveal a health state for that condition. A live invalid_grant Gmail account was not available in this read-only review.

## Solution Assessment

No current solution is sufficient. The best fix belongs in the Gmail watcher lifecycle/ops path, aligned with Gog's existing auth diagnostics, rather than relying on operators to scrape logs or adding a fallback watcher.
