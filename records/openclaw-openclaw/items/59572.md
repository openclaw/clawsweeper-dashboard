## Summary

Keep open: this is a real TUI feature PR, but the current branch still regresses current-main `/context` UX, does not handle valid null context reports, omits current `/context map` from its new completions/help, and lacks after-fix real TUI proof.

## What This Changes

The PR adds a TUI `/context visualize`/`/context viz` overlay, slash-command completion/help changes, a `sessions.usage` context report fetcher, visualization helpers, and TUI test updates.

## Best Possible Solution

Integrate the visualizer into the existing `/context` selector alongside current list/detail/json/map/help behavior, handle unavailable context reports explicitly, keep focused tests, and require real TUI proof before merge.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main opens a local selector for bare `/context`, while the PR head sends bare `/context` to the gateway, and `sessions.usage` can validly return `contextWeight: null` while the overlay stays in loading.

## Solution Assessment

No. The feature direction is plausible, but the branch should be rebased onto current selector and `/context map` behavior, add an explicit unavailable state for null reports, and show real TUI proof before merge.

  - body: Current main opens a local mode selector for bare `/context`, with tests asserting it does not send immediately. This branch only opens the overlay for `visualize` or `viz`, so bare `/context` falls through to `sendMessage(raw)` and would regress the current TUI flow; add the visualizer to the selector while keeping direct `/context viz` support.

  - confidence: 0.91

  - body: Current main supports and documents `/context map`, but this new local `context` slash command shadows the gateway command entry and lists only `visualize`, `viz`, `list`, `detail`, and `json` in completions/help. Add `map` so the TUI does not hide a valid current mode.
  - confidence: 0.82
