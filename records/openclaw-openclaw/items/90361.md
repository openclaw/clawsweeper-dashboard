## Summary

This should stay open: current main still has the same memory-core search, status, and safe-reindex shape that can expose transient missing metadata through the live manager while the durable index may be valid.

## What This Changes

The issue requests a fix for intermittent first-class `memory_search` unavailable results reporting `index metadata is missing` during overlapping search/reindex activity despite a valid builtin memory index.

## Best Possible Solution

Fix the manager-level search/reindex concurrency so live searches and status reads use a stable durable index or an explicit rebuild-in-progress state, while preserving true provider/model/settings mismatch blocking.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection shows the reported race is plausible because search can start async sync, safe reindex temporarily swaps the manager DB, and the tool fails closed on post-search missing identity.

## Solution Assessment

No, the local dist hotfix is best treated as a mitigation rather than the upstream shape. The safer fix is to isolate or serialize safe reindex relative to live search/status, then add a narrow wrapper guard only if it preserves real mismatch failures.
