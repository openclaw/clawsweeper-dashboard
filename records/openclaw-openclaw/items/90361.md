## Summary

This should stay open: current main and v2026.6.5 still show the memory_search/safe-reindex surfaces that can expose missing index identity during live reads, and the closing repair PR is still open and conflicting.

## What This Changes

The issue asks OpenClaw to fix builtin memory_search returning index metadata is missing when the durable SQLite memory index is valid, likely from safe-reindex/search/status concurrency and stale live manager state.

## Best Possible Solution

Land one coordinated memory-core repair that keeps live search/status reads off half-built or stale SQLite index state, preserves true provider/model/settings mismatch blocking, and includes deterministic race coverage plus real memory_search proof.

## Reproduction Assessment

Yes, at source level: pause or force safe reindex after the live manager switches to the temp DB, then read status or run memory_search concurrently. The issue also has multiple live affected-install reports, but I did not run a live current-main repro in this read-only review.

## Solution Assessment

No landed solution yet. The best fix is manager-level synchronization or isolation for safe reindex plus stale-handle refresh, not a tool-wrapper-only mitigation that could hide real mismatches.
