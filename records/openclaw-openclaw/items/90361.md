## Summary

This should stay open: current main still contains the reported memory-core search/status/reindex shape, and the only relevant changes since v2026.6.1 are watch-pressure cleanup, not a fix for transient missing index metadata during overlapping search/reindex work.

## What This Changes

The issue reports intermittent first-class `memory_search` failures with `index metadata is missing` despite a valid builtin SQLite memory index, likely caused by a search/reindex race.

## Best Possible Solution

Fix the memory-core manager/tool boundary so live searches either use a stable index snapshot during safe reindex or report an explicit transient rebuild state without discarding valid hits or hiding true identity mismatches.

## Reproduction Assessment

No, not from a live current-main run in this read-only review. The report is source-reproducible because current main still starts async search sync before search completion, exposes the safe-reindex temp DB through the live manager, and hard-pauses `memory_search` on a post-search missing identity status.

## Solution Assessment

Yes for the direction, but the minimal wrapper hotfix is only an acceptable mitigation if it is narrowly limited. The best upstream solution is to keep search/status on a stable durable index during safe reindex or serialize that transition while preserving hard failures for real identity mismatches.
