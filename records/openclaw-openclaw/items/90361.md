## Label Justifications

- `P1`: Affected agents can intermittently or persistently lose first-class memory retrieval despite a healthy durable index.
- `impact:session-state`: The live agent process can expose unavailable or stale persisted-memory state while CLI and fresh managers see a valid index.

## Summary

Current main fixes one persistent missing-metadata cause, but still exposes the shared temporary database to concurrent search/status reads during safe reindex. Because a mergeable open PR explicitly closes this issue and addresses that remaining race, the issue is still necessary and should remain open pending maintainer review of the linked memory-core fixes.

## What This Changes

Prevent builtin memory search from reporting a valid durable index as missing during safe reindex or after an atomic index rebuild.

## Best Possible Solution

Land one coordinated memory-core lifecycle fix that keeps readers off the temporary rebuild database, always writes verified metadata into the rebuilt database, and refreshes live handles after external atomic swaps while retaining fail-closed behavior for genuine identity mismatches.

## Reproduction Assessment

Yes, at source level with high confidence. Current main retains the temp-DB exposure and search/sync overlap, while multiple affected installations and linked deterministic tests independently demonstrate the same failure class; this read-only review did not run a fresh live stress reproduction.

## Solution Assessment

Yes for the manager-level direction, but the final implementation should coordinate the two linked fixes. A tool-wrapper exception would mask transient state and risk weakening true provider/model mismatch handling, whereas lifecycle synchronization and handle refresh address the owning boundary.
