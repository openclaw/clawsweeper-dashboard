## What This Changes

## Best Possible Solution

Land a narrow gateway fix that marks affected clients before success responses, rechecks invalidation immediately before handleGatewayRequest, removes unrelated deprecated Canvas fields, and includes focused regression coverage plus redacted real WebSocket proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection shows current main responds before the queued disconnect and lacks a device-token invalidation recheck, so the race is source-reproducible but still needs real pipelined WebSocket proof.

## Solution Assessment

No. Synchronous invalidation before responding is the right direction, but the branch needs a final recheck immediately before handleGatewayRequest and should drop the unrelated deprecated Canvas type additions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Recheck invalidation immediately before dispatch:** `src/gateway/server/ws-connection/message-handler.ts:1799-1807`
  - body: The new guard runs before the `await import("../../server-methods.js")`. Because each message starts its own async task, a request that passed this check can pause at the import while a later buffered rotate/revoke marks the client invalidated, then resume and call `handleGatewayRequest`; add a second invalidation check after the import and before dispatch, with a regression for that interleaving.
  - confidence: 0.86
- **[P3] Drop the deprecated Canvas fields from the client type:** `src/gateway/server/ws-types.ts:17-19`
  - body: This PR re-adds `canvasHostUrl`, `canvasCapability`, and `canvasCapabilityExpiresAtMs` to `GatewayWsClient`, but the protocol docs say those Canvas aliases are intentionally unsupported in favor of `pluginSurfaceUrls.canvas`. Keeping this unrelated type path invites drift from the supported protocol.
  - confidence: 0.78
