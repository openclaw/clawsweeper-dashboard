## Summary

Keep open: the PR targets a real gateway token-revocation race, but the current head still checks `client.invalidated` only before the async dispatch task. A request that passed that early guard can still reach `handleGatewayRequest` after another pipelined request invalidates the same device-token client, and the supplied proof is a synthetic ordering script rather than real gateway WebSocket traffic.

## What This Changes

The PR adds a synchronous invalidation flag and request-context helper for device-token WebSocket clients, marks clients before pair-removal/token-rotation/token-revoke responses, adds an early dispatch guard, and includes focused plus unrelated test updates.

## Best Possible Solution

Land the narrow invalidation-helper design only after adding a final invalidation recheck immediately before `handleGatewayRequest`, covering that interleaving with a regression test, removing the deprecated canvas fields, and attaching redacted real WebSocket proof.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a high-confidence reproduction of the race window: current main responds before queued disconnect and lacks a device-token invalidation recheck on request dispatch.

## Solution Assessment

No, not yet. The invalidation flag is the right direction, but the best maintainable fix needs a final check immediately before handler dispatch plus real WebSocket proof.
