## Summary

Keep open: this is a focused and useful gateway security fix, but it is not merge-ready because the new invalidation guard runs before the async dispatch boundary and the supplied proof is synthetic rather than actual pipelined WebSocket traffic.

## What This Changes

The PR adds synchronous invalidation of device-token WebSocket clients during token rotation, token revocation, and pair removal, plus a dispatch guard and focused unit tests.

## Best Possible Solution

Land the narrow invalidation-helper design only after adding a final invalidation recheck immediately before `handleGatewayRequest`, covering that interleaving with a regression test, and attaching redacted real WebSocket proof.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection does show current main responds before the queued disconnect and lacks a device-token invalidation recheck on request dispatch.

## Solution Assessment

No, not yet. The invalidation flag is the right direction, but it needs a final check immediately before handler dispatch plus real WebSocket proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Recheck invalidation at final dispatch:** `src/gateway/server/ws-connection/message-handler.ts:1796-1804`
  - body: This guard runs before the async IIFE that imports and calls `handleGatewayRequest`. A pipelined request can pass this early check, then another request can rotate/revoke the token and mark the client invalidated before the first request resumes; because there is no second check inside the IIFE, that first request still reaches the handler with the revoked device-token client. Add the same invalidation close path immediately before `handleGatewayRequest` and cover that interleaving.
  - confidence: 0.84
