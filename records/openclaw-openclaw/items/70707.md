## Summary

Keep open: current main still has the device-token revocation/rotation dispatch window, and this PR adds the right general invalidation seam, but the patch leaves a smaller async dispatch race and still lacks real WebSocket behavior proof.

## What This Changes

The PR adds a GatewayWsClient invalidation flag, a request-context invalidation helper, pre-response invalidation in device removal/token handlers, a dispatch guard, and focused gateway tests.

## Best Possible Solution

Keep the invalidation-helper design, repeat the invalidation check immediately before handleGatewayRequest after the dynamic import, add a regression for that final-dispatch window, and require redacted real pipelined WebSocket proof before merge.

## Reproduction Assessment

Yes, source-level: current main responds before queueMicrotask-scheduled disconnect in the three affected handlers and the dispatcher reaches handleGatewayRequest without a device-token invalidation check. I did not run a live pipelined WebSocket repro in this read-only review.

## Solution Assessment

No, not yet. The invalidation flag/helper is the right narrow approach, but the guard needs to be repeated at the final dispatch point after the awaited dynamic import and the PR needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Recheck invalidation at the final dispatch point:** `src/gateway/server/ws-connection/message-handler.ts:1535`
  - body: This guard runs before the async block that awaits import("../../server-methods.js") and later calls handleGatewayRequest. A request can pass the check while the client is still valid, yield at that await, then execute after another RPC has marked the client invalidated, leaving a queued-RPC window in the path this PR is meant to close. Repeat the check inside the async block immediately before handleGatewayRequest.
  - confidence: 0.84
