## Summary

Keep open: current main still has the source-level device-token queued-RPC window, and this PR adds a narrow synchronous invalidation guard for it. The diff looks correct, but the external PR still lacks after-fix real behavior proof and is currently conflicting with one failing parity-gate check.

## What This Changes

Adds synchronous device-client invalidation before device removal, token rotation, and token revocation responses, rejects invalidated WebSocket clients before RPC dispatch, and adds gateway tests for the ordering and request-context behavior.

## Best Possible Solution

Land this PR, or an equivalent narrow mainline patch, after the branch is rebased, exact-head checks are handled, and the contributor adds real behavior proof for the pipelined WebSocket path.

## Reproduction Assessment

Yes, source-level. Current main responds before the queued disconnect in all three affected handlers and the dispatcher has no device-token invalidation guard before handler logic; I did not run a live pipelined WebSocket repro.

## Solution Assessment

Yes. The invalidated flag plus pre-dispatch close is a narrow fix that preserves response flushing while closing the queued-RPC window; the remaining blockers are proof, rebase, and normal gate handling rather than a different code design.
