## Summary

Keep open: the PR targets a real current-main gateway revocation race, but the patch still leaves a dispatch window because the new invalidation check runs before an awaited import and not immediately before handler dispatch; the external PR also still needs real WebSocket behavior proof before merge.

## What This Changes

The branch adds a GatewayWsClient invalidation flag, a request-context invalidation helper, pre-response invalidation in device removal/token handlers, a pre-dispatch close guard, and focused gateway tests.

## Best Possible Solution

Keep the invalidation-helper design, but re-check the client invalidation state immediately before handleGatewayRequest runs, add a focused regression for that dispatch window, rebase, and require redacted real WebSocket proof before landing.

## Reproduction Assessment

Yes, source-level. Current main responds before the queued disconnect in all three affected handlers and the dispatcher lacks a device-token invalidation guard before handler logic; I did not run a live pipelined WebSocket repro in this read-only review.

## Solution Assessment

No, not yet. The invalidation flag approach is a narrow maintainable fix, but the guard must be moved or repeated at the final dispatch point after the awaited import to close the remaining race.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Recheck invalidation at the final dispatch point:** `src/gateway/server/ws-connection/message-handler.ts:1535`
  - body: This guard runs before the async block that later awaits `import("../../server-methods.js")` and then calls `handleGatewayRequest`. A request can pass the new check while the client is still valid, yield at that await, then execute after another RPC has marked the client invalidated, which leaves a queued-RPC window in the path this PR is meant to close. Move or repeat the check inside the async block immediately before `handleGatewayRequest`.
  - confidence: 0.82
