## Summary

Keep open: this PR targets a real gateway device-token revocation race, but the current patch still leaves an async dispatch window and needs after-fix real WebSocket proof before merge.

## What This Changes

The branch adds GatewayWsClient invalidation state, synchronous device-client invalidation before token/pairing responses, a dispatch close guard, request-context support, and focused gateway unit tests.

## Best Possible Solution

Keep the invalidation-helper design, repeat the invalidation guard at the final dispatch point after the awaited import, add a regression for that window, rebase, and require redacted real pipelined WebSocket proof before landing.

## Reproduction Assessment

Yes, source-level. Current main responds before the queued disconnect in all three affected handlers and async-dispatches requests without a device-token invalidation guard; I did not run a live pipelined WebSocket repro in this read-only review.

## Solution Assessment

No, not yet. The invalidation flag/helper is a maintainable direction, but the guard must be repeated immediately before handleGatewayRequest after the await to close the remaining dispatch window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Recheck invalidation at final dispatch:** `src/gateway/server/ws-connection/message-handler.ts:1538`
  - body: This check happens before the async block that awaits `import("../../server-methods.js")` and later calls `handleGatewayRequest`. A queued request can pass while the client is still valid, yield at that await, then run after another RPC has marked the client invalidated, leaving the same pipelined-RPC window. Repeat the guard immediately before `handleGatewayRequest`.
  - confidence: 0.82
