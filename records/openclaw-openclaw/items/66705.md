## Summary

Keep open: current main and v2026.6.6 still do not inject the requested exec child-process env markers, and the linked closing PR is open rather than merged. The report remains actionable, but the env contract needs maintainer product/security review before landing or choosing a plugin-only path.

## What This Changes

The issue requests automatic injection of `OPENCLAW_SESSION_KEY` and `OPENCLAW_AGENT_ID` into exec child process environments so scripts can route callbacks to the current agent session.

## Best Possible Solution

Maintainers should decide whether this becomes a core runtime marker contract or an opt-in plugin using `resolve_exec_env`; if core-owned, land or replace the linked PR with explicit docs, override semantics, tests, and real proof for every exec host it changes.

## Reproduction Assessment

Yes, by source inspection: current main passes session context into exec defaults but `runExecProcess` only injects `OPENCLAW_SHELL=exec` into the spawned environment. I did not run a live exec session in this read-only review.

## Solution Assessment

Unclear: env injection is the natural core boundary if maintainers want ambient markers, but current `resolve_exec_env` support also makes an opt-in plugin path possible. The best final solution depends on maintainer decisions about core defaults, security language, override behavior, and `OPENCLAW_SESSION_ID` scope.
