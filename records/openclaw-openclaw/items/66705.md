## Summary

Keep open. Current main still does not inject `OPENCLAW_SESSION_KEY` or `OPENCLAW_AGENT_ID` into exec child-process environments, and open PR #73349 uses closing syntax for this issue, so this should stay open until that PR or a successor lands.

## What This Changes

Expose the active exec session key and agent ID as environment variables in child processes so callback/webhook workflows can route back to the originating agent session.

## Best Possible Solution

Review and land #73349 or a narrow successor that injects trusted, sanitized session and agent env values across all exec hosts, documents the contract, and adds regression tests for presence and spoofing behavior.

## Reproduction Assessment

Yes. A source-level reproduction path is high-confidence: an exec command that prints the relevant env values will see `OPENCLAW_SHELL=exec`, while current env construction never adds `OPENCLAW_SESSION_KEY` or `OPENCLAW_AGENT_ID`.

## Solution Assessment

Yes for the requested direction, provided the implementation injects trusted runtime values after env sanitization. The narrowest maintainable path is to finish the linked PR lane rather than create a parallel fix.
