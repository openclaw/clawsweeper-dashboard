## Summary

Keep open: current main and v2026.6.6 have softer `sessions_send` accepted/timeout handling plus a SQLite session-delivery queue for restart continuations, but they still do not provide the requested durable `sessions_send` inbox with at-least-once acknowledgement, retry, idempotency, and target-agent recovery.

## What This Changes

The issue requests a persistent agent-to-agent `sessions_send` message queue so failed or timed-out sends can be retried and recovered instead of being possibly lost.

## Best Possible Solution

Design a bounded SQLite-backed `sessions_send` handoff/inbox contract in the gateway/session-tool boundary, with explicit ack/retry/idempotency semantics and focused docs/tests, instead of adding a JSONL inbox plus BOOT.md polling.

## Reproduction Assessment

Yes, at source level: current main routes `sessions_send` through active/gateway run paths and the only persistent session queue is not wired as a target-agent inbox. I did not run a live timeout-loss scenario.

## Solution Assessment

No, not as proposed: a per-agent JSONL inbox plus BOOT.md polling conflicts with OpenClaw's current SQLite/state-storage direction. The maintainable solution is a first-class SQLite-backed handoff/recovery contract at the gateway/session-tool boundary.
