## Label Justifications

- `P2`: The request addresses reliability for agent-to-agent delivery, but it is a backlog product/stability item rather than an urgent core outage.
- `impact:message-loss`: The central problem is that a timed-out or failed `sessions_send` may leave the sender unable to know whether the target agent received the message.
- `impact:session-state`: A persistent inbox would change durable agent/session state and restart recovery semantics.

## Summary

Keep this open: current `main` and the latest release have `sessions_send` fire-and-forget, wait, A2A follow-up, and process-local active-run steering, but they do not implement a durable per-agent inbox with retry/ack recovery. The request is real product work, but the proposed JSONL queue conflicts with repository storage policy and needs maintainer design before implementation.

## What This Changes

The issue requests a persistent per-agent message queue for `sessions_send` so agent-to-agent messages can survive timeouts, retries, and restarts.

## Best Possible Solution

Define a narrow SQLite-backed durable A2A delivery contract for `sessions_send` that covers ack, retry, recovery, idempotency, and wake behavior, or explicitly fold this into the existing handoff/delivery-mode design; do not add a JSONL sidecar inbox.

## Reproduction Assessment

Source-level yes for the missing durable queue: current `sessions_send` starts an agent run and may return accepted/timeout, while active-run queuing is process-local. I did not run a live timeout-loss scenario, so this is not a high-confidence runtime reproduction of message loss frequency.

## Solution Assessment

No: the JSONL per-agent inbox sketch is not the best repository-native implementation because OpenClaw runtime queues should use SQLite. The better path is a maintainer-approved durable A2A contract that reuses existing queue/storage patterns where appropriate.
