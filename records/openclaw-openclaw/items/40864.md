## Summary

Keep open: current main and the latest release still do not provide a durable per-agent `sessions_send` inbox with at-least-once retry/ack semantics. Existing durable queue code is scoped to restart continuation recovery, and the related A2A handoff work remains a draft proposal rather than an implementation.

## What This Changes

The issue asks OpenClaw to add persistent agent-to-agent message queuing so `sessions_send` can recover from timeouts, retries, and missed delivery.

## Best Possible Solution

Define the first-party A2A handoff and acknowledgement contract first, then add durable `sessions_send` storage, retry, acknowledgement, docs, and focused tests while keeping higher-level meeting lifecycle features separate.

## Reproduction Assessment

Not applicable; this is a feature request for a new durable delivery contract, not a broken documented behavior. Source review shows current main dispatches directly and only has restart-continuation queueing, not a persistent `sessions_send` inbox.

## Solution Assessment

Unclear as a direct queue-first implementation. A durable queue is plausible, but the maintainable path is to settle the A2A handoff/ack protocol first and then wire persistence into that contract.
