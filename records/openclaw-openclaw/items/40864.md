## Summary

Keep open: current main has `sessions_send` A2A send/announce handling and durable delivery queues for other delivery paths, but it does not provide the requested persistent per-agent inbox/retry contract for ordinary `sessions_send` failures or timeouts.

## What This Changes

The issue requests a persistent per-agent message queue so `sessions_send` delivery can recover from timeouts or failed target-agent delivery.

## Best Possible Solution

If maintainers sponsor this direction, design a core-owned SQLite-backed A2A delivery/inbox contract that reuses existing queue primitives where appropriate and defines ack/retry/idempotency semantics before implementation.

## Reproduction Assessment

Yes from source inspection, not from a live run: current main has direct `sessions_send` run submission and timeout/error reporting, but no persistent per-agent inbox with retry/ack polling for ordinary A2A sends.

## Solution Assessment

No as proposed: the JSONL per-agent queue shape conflicts with the repository's SQLite-first storage policy, so the maintainable solution would be a designed core delivery contract rather than the issue's file-based sketch.
