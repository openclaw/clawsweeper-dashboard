## Summary

Current main reduces delivery ambiguity through fire-and-forget admission, active-run queueing, transcript-commit waits, fallback dispatch, and idempotency keys, but it does not provide a persistent inbox recovered after process restart. The remaining durable-delivery request is valid but requires a maintainer-owned product and architecture decision.

## What This Changes

The issue requests durable, retryable agent-to-agent message delivery with acknowledgements, deduplication, and restart recovery.

## Best Possible Solution

Define a lifecycle-owned delivery state machine for accepted, committed, acknowledged, retryable, expired, and deduplicated messages, backed by canonical SQLite/session storage while retaining the existing active-run fast path.

## Reproduction Assessment

No, there is not a high-confidence end-to-end reproduction proving current main loses a message after acceptance or restart. Source inspection does establish that the current queue path is active-run scoped and does not demonstrate restart-durable recovery.

## Solution Assessment

No, the proposed JSONL inbox and BOOT.md polling are not the best solution because they create a second state path outside canonical session and SQLite ownership. The reliable-delivery goal needs an explicit gateway/session lifecycle design.
