## Summary

Keep open: current main and v2026.5.19 still lack a durable per-agent `sessions_send` inbox with retry and acknowledgement. The related open handoff PRs are useful Phase 1/proposal work, but they explicitly stop short of the durable queue requested here.

## What This Changes

Requests a persistent A2A `sessions_send` queue or inbox with retry, acknowledgement, idempotency, and wake/heartbeat recovery for uncertain delivery after timeout or failure.

## Best Possible Solution

Define the first-party A2A handoff, acknowledgement, retry, idempotency, and storage contract first, then integrate durable `sessions_send` queueing with docs and regression coverage around the existing session-delivery infrastructure where it fits.

## Reproduction Assessment

Not applicable; this is a request for a new durable delivery contract, not a broken documented behavior. Source review confirms current main and v2026.5.19 use direct dispatch plus restart-continuation queueing, not a persistent `sessions_send` inbox.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but the maintainable path is to settle A2A acknowledgement, retry, idempotency, and storage semantics before wiring persistence into runtime behavior.
