## Label Justifications

- `P2`: This is a normal-priority reliability/product improvement for agent-to-agent delivery, not a current core outage.
- `impact:message-loss`: The reported problem is that a timed-out or failed `sessions_send` can leave the sender unable to know whether the target agent received the message.
- `impact:session-state`: A durable inbox would change persisted agent/session state and restart recovery semantics.

## Summary

Keep this open: current main and v2026.6.6 support immediate accepted returns, A2A follow-up, and process-local active-run steering, but they do not provide a durable per-agent inbox with retry/ack recovery for `sessions_send`. The request is real core product work, but it needs a maintainer-approved SQLite-backed delivery contract rather than the proposed JSONL sidecar queue.

## What This Changes

Request to add a persistent agent-to-agent inbox/queue so `sessions_send` messages can survive timeouts or failures and be retried or acknowledged.

## Best Possible Solution

Define a narrow SQLite-backed durable A2A delivery contract for `sessions_send` covering ack, retry, recovery, idempotency, and wake behavior, or fold this into the handoff design; do not add a JSONL sidecar inbox.

## Reproduction Assessment

Source-level yes for the missing durable queue: current `sessions_send` starts or waits on agent runs and uses only process-local active-run queuing. I did not establish a live timeout-loss reproduction, so the runtime failure frequency is not proven here.

## Solution Assessment

No, the proposed JSONL inbox is not the best repository-native solution. The better path is a maintainer-approved durable A2A contract backed by the existing SQLite storage direction.
