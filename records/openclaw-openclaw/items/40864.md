## Summary

Keep open: current main still has direct `sessions_send` execution plus adjacent durable recovery queues, but not the requested persistent agent inbox with at-least-once retry and acknowledgement semantics. The open handoff and teleconference proposals are relevant design context, not an implementation of this delivery contract.

## What This Changes

The issue requests a persistent `sessions_send` delivery queue/inbox with retry, acknowledgement, and idempotency so timed-out or failed sends are not lost.

## Best Possible Solution

Define the first-party A2A handoff and durable delivery contract, likely composing with https://github.com/openclaw/openclaw/pull/63789 and the lifecycle discussion in https://github.com/openclaw/openclaw/issues/65403, then add queue/retry/inbox behavior with focused tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main shows no public queue/ack/retry/inbox contract, while existing tests cover accepted, delayed, and terminal timeout behavior only.

## Solution Assessment

Unclear as a direct implementation path. A persistent queue is plausible, but the maintainable solution should first settle the transport acknowledgement and lifecycle boundaries before wiring durable retry behavior into `sessions_send`.
