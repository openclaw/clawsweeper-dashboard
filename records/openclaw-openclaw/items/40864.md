## Summary

Keep open: current main still has timeout-oriented `sessions_send` plus adjacent restart-continuation delivery recovery, but it does not provide the requested durable agent inbox with at-least-once retry and acknowledgement semantics. The closest related work is the open draft #63789 protocol proposal, which is useful context but not an implementation of this persistent queue.

## What This Changes

The issue requests persistent agent-to-agent message delivery for `sessions_send`, including durable inbox storage, retry, acknowledgement, and recovery semantics.

## Best Possible Solution

Define the first-party A2A handoff and delivery contract, likely aligned with #63789, then deliberately generalize or add durable queue/retry behavior with regression tests and docs.

## Reproduction Assessment

Not applicable as a feature request. Source inspection shows the requested persistent inbox and at-least-once delivery contract are not present on current main, while existing coverage is for timeout follow-up behavior.

## Solution Assessment

Unclear as a direct implementation path. A persistent queue is plausible, but the maintainable first step is to settle the A2A handoff/ack contract before wiring durable retry behavior into `sessions_send`.
