## Summary

Keep open. Current main still has immediate `accepted` and delayed A2A follow-up paths, plus a separate session delivery queue for restart/channel delivery, but it does not provide a durable per-target A2A inbox with ack, retry, and restart recovery semantics.

## What This Changes

Request to add a durable at-least-once agent-to-agent queue for `sessions_send` with retry, acknowledgment, idempotency, and restart recovery.

## Best Possible Solution

Define a maintainer-approved SQLite-backed `sessions_send` delivery contract covering durable intent write, target ack/read state, retry limits, restart recovery, visibility policy, docs, and live multi-agent proof; avoid a JSONL/BOOT.md polling design.

## Reproduction Assessment

Yes, at source level: current main has accepted/A2A paths and separate session-delivery recovery, but no durable per-agent `sessions_send` inbox or retry/ack state. I did not establish a live timeout/restart reproduction in this read-only review.

## Solution Assessment

Unclear as proposed: the reliability goal is valid, but the JSONL/BOOT.md-style queue should be replaced by a maintainer-approved SQLite-backed core delivery contract.
