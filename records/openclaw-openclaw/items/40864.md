## Summary

Keep open: current main and v2026.5.20 still provide direct `sessions_send` dispatch plus best-effort A2A follow-up, not a durable per-agent inbox with retry and acknowledgement. Related handoff PRs are useful adjacent work, but they intentionally stop short of the persistent queue requested here.

## What This Changes

Request durable persisted `sessions_send` queueing with retry, acknowledgement, and recovery so agents can handle missed messages after timeouts or failures.

## Best Possible Solution

Define the first-party A2A durability contract, then implement durable `sessions_send` queueing by composing with the accepted handoff/ack layer and reusing queue infrastructure without creating competing delivery state.

## Reproduction Assessment

Not applicable; this is a request for a new durable delivery contract, not a broken documented behavior. Source and release inspection confirm current `sessions_send` lacks a persistent inbox or retry/ack queue.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but maintainers should first settle acknowledgement, retry, idempotency, storage, and lifecycle semantics and align them with the open handoff work.
