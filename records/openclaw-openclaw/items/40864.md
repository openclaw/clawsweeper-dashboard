## Summary

Keep open: current main and the latest release still lack a durable `sessions_send` inbox/queue with acknowledgement and retry semantics. The related handoff PRs are relevant, but they intentionally cover lower-level acknowledgement/ledger work rather than the persistent delivery queue requested here.

## What This Changes

Request adding a persistent per-agent `sessions_send` message queue so agent-to-agent messages can survive timeout, failure, and restart with retry and acknowledgement.

## Best Possible Solution

Define the first-party A2A handoff, acknowledgement, retry, idempotency, storage, and cleanup contract, then implement durable `sessions_send` queueing against that accepted protocol.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this requests a new durable delivery contract rather than a broken documented guarantee. Source and release inspection confirm current behavior lacks a persistent `sessions_send` inbox, acknowledgement, and retry queue.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but the maintainable path is to settle A2A acknowledgement, retry, idempotency, and storage semantics before wiring persistence into runtime behavior.
