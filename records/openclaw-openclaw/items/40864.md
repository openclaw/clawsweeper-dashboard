## Label Justifications

- `P2`: This is a normal-priority reliability/product improvement for agent-to-agent delivery, not a current core outage.
- `impact:message-loss`: The reported problem is that a timed-out or failed `sessions_send` can leave the sender unable to know whether the target agent received the message.
- `impact:session-state`: A durable inbox would change persisted agent/session state and restart recovery semantics.

## Summary

Keep open: current main still uses direct run start/wait paths or process-local active-run steering for `sessions_send`; it has adjacent SQLite session delivery recovery, but no durable per-agent inbox with ack/retry recovery for this tool.

## What This Changes

The issue asks for `sessions_send` to persist messages in a reliable per-agent queue/inbox with retry, acknowledgement, idempotency, and recovery.

## Best Possible Solution

Define a narrow SQLite-backed `sessions_send` durable A2A delivery contract, or explicitly route this to plugin/ClawHub if maintainers decide core should not own it; do not add a JSONL sidecar inbox.

## Reproduction Assessment

Source-level yes for the missing durable queue: current `sessions_send` starts/waits on runs or process-local active-run queues, and no durable inbox enqueue/ack path exists. No live timeout-loss reproduction was run, so runtime frequency remains unproven.

## Solution Assessment

No, the proposed JSONL inbox plus BOOT polling is not the best OpenClaw-native solution. The maintainable path is a SQLite-backed delivery contract with ack, retry, idempotency, wake/recovery, docs, and tests, coordinated with the open handoff/timeout proposals.
