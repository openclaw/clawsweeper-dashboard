## Label Justifications

- `P2`: This is a meaningful multi-agent reliability improvement with limited blast radius, but it requires API and storage design before implementation.
- `impact:message-loss`: The reported failure is that inter-agent messages may not reliably reach the target after `sessions_send` timeout or failure.
- `impact:session-state`: A durable inbox and acknowledgement model would change in-flight inter-agent session state, recovery, and retry bookkeeping.

## Summary

Keep open. Current main and v2026.6.8 still do not provide a durable `sessions_send` inbox with at-least-once acknowledgement, retry, idempotency, and target-agent recovery; the existing persistent session-delivery queue is scoped to restart continuations, not ordinary inter-agent sends.

## What This Changes

The issue requests a persistent, recoverable message queue for `sessions_send` so target agents can receive missed messages after timeouts, failures, or restarts.

## Best Possible Solution

Design a first-class SQLite-backed `sessions_send` handoff/inbox contract at the gateway/session-tool boundary, with explicit ack, retry, idempotency, recovery, redaction, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: current main routes ordinary `sessions_send` through direct Gateway `agent`/`agent.wait` paths, and the only persistent session-delivery queue is not wired as a target-agent inbox. I did not run a live timeout-loss scenario.

## Solution Assessment

No, the proposed JSONL inbox plus BOOT.md polling is not the best repo-aligned implementation. The maintainable solution should be a SQLite-backed core handoff/recovery contract with explicit ack, retry, and idempotency semantics.
