- `impact:message-loss`: The reported failure is `sessions_send` messages that may not reliably reach the target agent after timeout or failure.

## Summary

Keep open. Current main and v2026.6.8 still lack a persistent `sessions_send` inbox with at-least-once enqueue, acknowledgement, retry, and target-agent recovery; existing queue/recovery code is scoped to restart continuations and related PRs only cover partial timeout or acknowledgement slices.

## What This Changes

The issue requests a persistent at-least-once `sessions_send` message queue/inbox with acknowledgement, retry, and recovery for missed inter-agent messages.

## Best Possible Solution

Design a first-class SQLite-backed `sessions_send` handoff/inbox contract at the gateway/session-tool boundary, reusing or extending the existing session-delivery queue with explicit ack, retry, idempotency, redaction, recovery, and docs/tests.

## Reproduction Assessment

Yes, at source level: current main routes ordinary `sessions_send` through direct Gateway agent/run paths, and the only persistent session-delivery queue is not wired in as a target-agent inbox. I did not run a live timeout-loss scenario.

## Solution Assessment

No, the suggested JSONL inbox plus BOOT.md polling is not the best repo-aligned shape. The maintainable solution should be a SQLite-backed core handoff/recovery contract with explicit ack/retry/idempotency semantics.
