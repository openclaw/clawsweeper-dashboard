## Summary

Keep open: current main and v2026.5.22 still lack a durable `sessions_send` inbox/queue with acknowledgement, retry, idempotency, and recovery semantics; related acknowledgement and timeout work is narrower.

## What This Changes

The issue requests a persistent agent-to-agent message queue for `sessions_send` so timed-out or failed sends can be retried and recovered by target agents.

## Best Possible Solution

Define a first-party A2A delivery contract for acknowledgement, retry, idempotency, durable storage, cleanup, and failure visibility, then implement queueing against that accepted protocol.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this requests a new durable delivery contract rather than a broken documented guarantee. Source, docs, release-tag inspection, and related PR review confirm current behavior lacks a persistent `sessions_send` inbox with ack and retry.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but the maintainable path is to settle A2A acknowledgement, retry, idempotency, storage, and failure semantics first; https://github.com/openclaw/openclaw/pull/83094 is only a Phase 1 acknowledgement proposal.
