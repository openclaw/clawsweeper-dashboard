## Summary

Keep open. Current main still treats ordinary `sessions_send` as a direct Gateway `agent` dispatch with timeout/error reporting, while the durable session-delivery queue in the tree is scoped to restart-continuation recovery rather than a general A2A inbox with accepted, acknowledged, retried delivery.

## What This Changes

Add a persistent A2A message queue/inbox for `sessions_send` so timed-out or failed sends have at-least-once delivery, acknowledgement, retry, idempotency, and restart recovery.

## Best Possible Solution

Define a core reliable A2A delivery contract: persist accepted `sessions_send` messages before dispatch, return durable acceptance/acknowledgement state, drain pending entries on startup or heartbeat, and cover timeout, restart, duplicate-delivery, permission, and transcript-provenance behavior with focused source and integration tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: ordinary `sessions_send` calls Gateway `agent` directly, and timeout/error results do not expose durable queue or acknowledgement state while current durable recovery is restart-continuation scoped.

## Solution Assessment

Unclear until maintainer design. A persistent queue is the right class of solution, but the proposed file-only BOOT polling inbox is weaker than a core contract covering acceptance, retry, dedupe, recovery, permissions, and provenance.
