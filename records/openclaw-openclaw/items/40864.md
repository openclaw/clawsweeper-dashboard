## Summary

Keep open: current main still lacks the requested persistent `sessions_send` inbox/queue with at-least-once retry and acknowledgement semantics. The existing durable queue is restart-continuation recovery only, while the related handoff and teleconference threads remain open design work rather than an implementation.

## What This Changes

Requested addition of a durable per-agent inbox/queue for `sessions_send` with retry, acknowledgement, idempotency, and recovery semantics.

## Best Possible Solution

Define the first-party A2A transport contract first, then add durable `sessions_send` handoff storage/retry/ack behavior with focused tests and docs without folding in the higher-level meeting lifecycle.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken documented behavior. Source and docs inspection show current `sessions_send` has direct delivery plus timeout-oriented A2A follow-up, not a persistent inbox contract.

## Solution Assessment

Unclear as a direct implementation request. A durable queue is plausible, but the maintainable path is to settle the A2A handoff and ack semantics before wiring persistent retry behavior into `sessions_send`.
