## Summary

Keep open. Current main implements immediate `accepted`/A2A follow-up behavior for `sessions_send`, but it still has no durable per-agent inbox, persisted retry/ack lifecycle, or restart recovery for messages whose send/start path times out or fails.

## What This Changes

The issue requests a persistent at-least-once delivery queue/inbox for `sessions_send` so target agents can recover missed messages after timeout, failure, or restart.

## Best Possible Solution

Define a maintainer-approved SQLite-backed durable A2A delivery contract for `sessions_send`, including idempotency, ack/read states, retry limits, recovery on restart, visibility policy, docs, and focused source plus live multi-agent proof.

## Reproduction Assessment

Yes, at source level: current main has immediate accepted/A2A paths and in-memory runtime queues, but no durable per-agent inbox or retry/ack state. I did not run a live multi-agent timeout/restart scenario in this read-only review.

## Solution Assessment

Unclear on exact API/storage shape: the persistent queue direction is plausible, but the proposed JSONL/BOOT.md-style inbox should instead become a maintainer-approved SQLite-backed core delivery contract with policy and migration decisions.
