## Summary

Keep open: current main and the latest release still do not provide a durable per-agent `sessions_send` inbox with at-least-once retry/ack semantics. The current path starts direct agent runs and best-effort A2A announce delivery; the existing durable queue is scoped to restart-continuation recovery, so this needs maintainer protocol/product sequencing before implementation.

## What This Changes

The issue requests a persistent per-agent message queue for `sessions_send` so agent-to-agent messages can survive timeouts, be retried, and be acknowledged.

## Best Possible Solution

Define the first-party A2A handoff and acknowledgement contract first, then add durable `sessions_send` storage, retry, acknowledgement, docs, and focused tests while reusing the generic queue infrastructure where it fits.

## Reproduction Assessment

Not applicable; this is a feature request for a new durable delivery contract, not a broken documented behavior. Source review shows current main dispatches directly and only has restart-continuation queueing, not a persistent `sessions_send` inbox.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but the maintainable path is to settle the A2A handoff/ack protocol first and then wire persistence into that contract.
