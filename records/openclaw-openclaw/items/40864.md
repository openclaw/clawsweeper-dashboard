## Summary

Keep open: current main and v2026.5.22 still do not provide the requested durable `sessions_send` inbox/queue with acknowledgement, retry, and recovery semantics; related A2A PRs are useful but only cover narrower acknowledgement or timeout-compensation layers.

## What This Changes

The issue requests a persistent agent-to-agent message queue so `sessions_send` can provide reliable delivery, retry, acknowledgement, and missed-message recovery.

## Best Possible Solution

Define a first-party A2A delivery contract for acknowledgement, retry, idempotency, durable storage, cleanup, and visible failure states, then implement `sessions_send` queueing against that accepted protocol.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this requests a new durable delivery contract rather than a broken documented guarantee. Source, docs, tests, and release inspection confirm current behavior lacks a persistent `sessions_send` inbox, acknowledgement, and retry queue.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but the maintainable path is to settle A2A acknowledgement, retry, idempotency, storage, and failure semantics before wiring persistence into runtime behavior.
