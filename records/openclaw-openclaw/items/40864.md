## Summary

Keep open. Current main and v2026.6.6 have softer accepted/timeout handling and a SQLite session-delivery queue for restart-continuation recovery, but they do not implement a persistent `sessions_send` inbox with at-least-once enqueue, acknowledgement, retry, and target-agent polling/recovery.

## What This Changes

Request to add persistent reliable delivery for agent-to-agent `sessions_send`, replacing ambiguous timeout/failure outcomes with durable queue, acknowledgement, and recovery semantics.

## Best Possible Solution

Design a narrow SQLite-backed `sessions_send` handoff/inbox contract integrated with the existing gateway agent path and session-delivery queue, with explicit ack/retry/idempotency semantics and docs/tests, instead of adding a JSONL inbox plus BOOT.md polling.

## Reproduction Assessment

Yes, at source level: current main routes `sessions_send` through direct gateway/active-run paths and the only persistent session queue is not wired into `sessions_send` as an inbox. I did not run a live timeout-loss scenario.

## Solution Assessment

No. The proposed per-agent JSONL inbox plus BOOT.md polling does not match OpenClaw's current SQLite/state-storage direction; the maintainable solution is a first-class SQLite-backed handoff and recovery contract owned by the gateway/session tool boundary.
