## Summary

Keep open. Current main still has no durable inbox or acknowledged retry contract for ordinary `sessions_send`; the existing persistent session-delivery queue is scoped to restart-continuation recovery, while `sessions_send` still starts a Gateway `agent` run directly and reports timeout/error state without a durable queue receipt.

## What This Changes

The issue asks for a persistent, acknowledged message queue so `sessions_send` can provide reliable delivery and recovery after timeouts or restarts.

## Best Possible Solution

Define a core reliable A2A delivery contract that persists accepted `sessions_send` work before dispatch and drains it with explicit acknowledgement, retry, dedupe, permission, and provenance behavior.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction: ordinary `sessions_send` dispatches directly to Gateway `agent`, and timeout/error paths do not expose a durable queue entry or acknowledgement state.

## Solution Assessment

Unclear until maintainer design. A persistent queue is the right class of solution, but the proposed per-agent file plus BOOT polling model should likely become a core Gateway/session contract rather than an agent-prompt workaround.
