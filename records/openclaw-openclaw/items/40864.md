## Summary

Keep open: current main and the latest release still use direct `sessions_send` dispatch plus best-effort A2A announce/restart-continuation queueing, not a durable per-agent inbox with retry and acknowledgement. The open handoff PRs are relevant adjacent work, but they intentionally stop short of the persistent queue requested here.

## What This Changes

The issue requests a durable per-agent `sessions_send` queue/inbox with retry, acknowledgement, and idempotency so cross-agent messages survive timeouts and failures.

## Best Possible Solution

Define the first-party A2A handoff, acknowledgement, retry, idempotency, and storage contract first, then implement durable `sessions_send` queueing or explicitly compose it with the accepted handoff/lifecycle layer.

## Reproduction Assessment

Not applicable; this is a request for a new durable delivery contract, not a broken documented behavior. Source review confirms current main and `v2026.5.20` use direct dispatch plus restart-continuation queueing, not a persistent `sessions_send` inbox.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but the maintainable path is to settle A2A acknowledgement, retry, idempotency, and storage semantics before wiring persistence into runtime behavior.
