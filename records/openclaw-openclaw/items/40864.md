## Summary

Keep open: current main still does not provide a durable per-agent `sessions_send` inbox with at-least-once retry and acknowledgement semantics. The existing durable queue is restart-continuation infrastructure, and the open acknowledgement PR explicitly stops short of adding a durable work queue or retry engine.

## What This Changes

The issue requests a persistent per-agent queue for `sessions_send` so timed-out or failed A2A messages can be retried and acknowledged instead of possibly being lost.

## Best Possible Solution

Define the first-party A2A handoff and acknowledgement contract first, then add durable `sessions_send` storage, retry, acknowledgement, docs, and focused tests while reusing the existing session-delivery queue only where it fits cleanly.

## Reproduction Assessment

Not applicable; this is a request for a new durable delivery contract, not a broken documented behavior. Source review confirms current main has direct dispatch plus restart-continuation queueing, not a persistent `sessions_send` inbox.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but the maintainable path is to settle the A2A handoff and acknowledgement protocol first, then wire persistence and retry into that contract.
