## Summary

Keep open: current main still does not provide the requested durable `sessions_send` inbox with at-least-once retry and acknowledgement semantics. The related handoff proposal at https://github.com/openclaw/openclaw/pull/63789 is still an open draft and deliberately narrower than persistent retry, while the external message-bus plugin discussed at https://github.com/openclaw/openclaw/issues/52290 is useful ClawHub/community work rather than a first-party `sessions_send` implementation.

## What This Changes

The issue requests a persistent per-agent message queue for reliable `sessions_send` delivery with retry, acknowledgement, idempotency, and recovery semantics.

## Best Possible Solution

Define the first-party A2A handoff and acknowledgement contract, then implement durable retry/recovery in the generic session delivery path with focused source, tests, and docs.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken documented behavior. Current source and docs show direct `sessions_send` delivery with optional wait and best-effort A2A announce, not a durable inbox contract.

## Solution Assessment

Unclear as a direct implementation request. A persistent queue is plausible, but the maintainable path is to settle the A2A handoff and ack semantics before wiring retryable storage into `sessions_send`.
