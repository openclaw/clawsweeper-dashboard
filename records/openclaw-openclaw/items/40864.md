## Summary

Keep open: current main and v2026.5.19 still use direct `sessions_send` dispatch plus best-effort A2A follow-up and restart-continuation recovery, not a durable per-agent inbox with retry and acknowledgement. The related open implementation proposal at https://github.com/openclaw/openclaw/pull/83094 is useful Phase 1 acknowledgement work, but it explicitly does not add the durable work queue or retry engine requested here.

## What This Changes

Request to add a durable per-agent `sessions_send` message queue with at-least-once delivery, retry, acknowledgement, and restart recovery for missed A2A messages.

## Best Possible Solution

Define the first-party A2A handoff and acknowledgement contract first, then add durable `sessions_send` storage, retry, acknowledgement, docs, and focused tests while reusing the existing session-delivery queue only where it cleanly fits.

## Reproduction Assessment

Not applicable; this is a request for a new durable delivery contract, not a broken documented behavior. Source review confirms current main and v2026.5.19 use direct dispatch plus restart-continuation queueing, not a persistent `sessions_send` inbox.

## Solution Assessment

Unclear as a direct per-agent inbox implementation. A durable queue is plausible, but the maintainable path is to settle A2A acknowledgement, retry, idempotency, and storage semantics before wiring persistence into the runtime.
