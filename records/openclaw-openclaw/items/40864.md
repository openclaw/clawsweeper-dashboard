## Summary

Keep open: current `main` has partial adjacent reliability work, but not the requested durable per-agent inbox for ambiguous or failed `sessions_send` delivery.

## What This Changes

The issue requests a persistent agent-to-agent message queue with acknowledgements, retries, heartbeat/startup polling, and idempotency for reliable `sessions_send` delivery.

## Best Possible Solution

Define the core A2A/session-delivery contract first, then extend the existing durable session-delivery primitives only if they can safely cover `sessions_send` failures without changing security or duplicate-processing expectations unexpectedly.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request for stronger delivery guarantees. Source inspection confirms current `sessions_send` has no documented persistent inbox/retry contract for ambiguous failures.

## Solution Assessment

Unclear. A durable queue is a plausible direction, but the best implementation needs product/API judgment around acknowledgement timing, retries, idempotency, storage, and cross-agent authorization.
