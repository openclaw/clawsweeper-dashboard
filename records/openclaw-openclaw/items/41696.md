## What This Changes

The PR changes Discord allowlist parsing so bare numeric entries try guild resolution before falling back to channel lookup, with regression coverage for that behavior.

## Best Possible Solution

Move the guild-first parser change and regression tests to `extensions/discord/src/resolve-channels.*`, preserve explicit prefix and bare-channel fallback behavior, and require redacted resolver or live Discord proof before merge.

## Reproduction Assessment

Yes, by source inspection: the active parser returns `channelId` for a bare numeric entry, and the active regression test expects that bare entry to remain unresolved while `guild:999` resolves. I did not run tests because this was a read-only review.

## Solution Assessment

No. The intended guild-first algorithm is plausible, but this branch applies it to removed legacy files; the maintainable fix is the same behavior in the active Discord extension resolver with proof and an explicit compatibility decision.
