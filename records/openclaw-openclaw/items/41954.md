## Summary

Keep open: current main has not implemented the module split, and the PR is stale against newer cron UI behavior that is now covered by source and tests.

## What This Changes

The PR extracts Control UI cron view helper code from `ui/src/ui/views/cron.ts` into `ui/src/ui/views/cron/filters.ts`, `helpers.ts`, `jobs.ts`, and `runs.ts` while intending to preserve behavior.

## Best Possible Solution

Carry the split forward only if maintainers still want the cleanup: rebase onto current main, extract the current cron view behavior into focused modules, and preserve the payload guard, sanitized markdown rendering, error-body behavior, and in-app run-chat navigation.

## Reproduction Assessment

Not applicable as a user bug report. For PR correctness, yes: static comparison of the PR patch against current main shows missing current-main behavior that is also covered by focused cron view tests.

## Solution Assessment

No. The PR as submitted is not the best mergeable solution anymore; the safer path is a current-main extraction that preserves the covered behavior, or closing the cleanup if the churn is not worth carrying forward.

  - confidence: 0.9
- **[P2] Render run summaries through the markdown body:** `ui/src/ui/views/cron/runs.ts:54`
  - body: Current main renders run summaries and fallback errors in `.cron-run-entry__body.chat-text` via sanitized markdown and avoids duplicating an error when no summary exists. This extraction keeps summary/error text in the list subtitle and still emits error metadata, regressing the current display behavior.
  - confidence: 0.88
- **[P2] Keep in-app run chat navigation:** `ui/src/ui/views/cron/runs.ts:33`
  - body: Current main passes `onNavigateToChat` into `renderRun` so normal left-clicks on run-chat links switch sessions inside the app. The extracted `renderRun` accepts only `entry` and `basePath`, leaving the anchor as ordinary URL navigation after rebase.
  - confidence: 0.86
