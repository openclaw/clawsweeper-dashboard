## Summary

Keep open. Current main still lacks the requested IRC fixes, and the PR is useful, but it cannot merge as-is because the new closed-send guard breaks graceful QUIT, delayed reclaim timers are not tied to client lifetime, duplicate reclaim attempts remain possible, and after-fix real behavior proof is missing.

## What This Changes

The PR changes bundled IRC plugin client/send/test code to throw on empty or closed sends, delay transient shutdown, identify fallback nicks with account-qualified NickServ, and schedule post-identify GHOST/NICK reclaim.

## Best Possible Solution

Merge a repaired IRC plugin change that preserves the useful reliability fixes while keeping graceful QUIT delivery, binding delayed reclaim work to client lifetime, preventing duplicate reclaim attempts, adding focused tests, and including inspectable after-fix real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows password-only NickServ IDENTIFY, sanitized-empty `sendPrivmsg` returning silently, and transient sends quitting immediately; the PR diff then introduces the QUIT regression through the new closed-send guard.

## Solution Assessment

No. The direction is sound, but this branch is not the best merge form until QUIT ordering, reclaim timer lifecycle, duplicate reclaim behavior, and real behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QUIT when guarding closed sends:** `extensions/irc/src/client.ts:175-177`
  - body: The new guard rejects whenever `closed` is true, but `quit()` sets `closed = true` before calling `sendRaw("QUIT ...")`. Every graceful shutdown now swallows that error and closes the socket without sending the QUIT frame, so move the closed transition until after QUIT is written or bypass this guard for shutdown.
  - confidence: 0.94
- **[P2] Tie reclaim timers to client shutdown:** `extensions/irc/src/client.ts:354-369`
  - body: The delayed NickServ GHOST/NICK timers are not stored, cleared, or unref'd. Once scheduled, they can keep one-shot IRC sends alive after `quit()` and can fire against a client that has already been closed, so track these timers and clear or unref them during shutdown.
  - confidence: 0.86
- **[P3] Avoid duplicate NickServ reclaim attempts:** `extensions/irc/src/client.ts:346-354`
  - body: `tryRecoverNickCollision()` can already send GHOST/NICK during collision recovery. If that path has already run and the fallback nick still reaches welcome, this delayed block sends a second GHOST/NICK sequence for the same desired nick; gate it on the earlier recovery state.
  - confidence: 0.74
