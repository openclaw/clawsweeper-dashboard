## Summary

Current main still has the IRC behavior this PR is trying to fix, but the branch is not mergeable as-is because the proposed closed-send guard breaks graceful QUIT and the delayed NickServ reclaim timers outlive client shutdown. Prior Greptile and Codex review comments are consistent with the live diff and current source, so this should stay open for a repaired PR plus real behavior proof rather than cleanup closure.

## What This Changes

The PR updates the bundled IRC plugin to surface empty/closed send failures, delay transient disconnects, use account-qualified NickServ IDENTIFY for fallback nicks, and add delayed GHOST/NICK reclaim coverage.

## Best Possible Solution

Merge a repaired IRC plugin change that keeps the useful send/NickServ fixes while preserving QUIT delivery, tying delayed reclaim timers to client shutdown, avoiding duplicate reclaim attempts, and adding focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main still uses password-only NickServ IDENTIFY, silently returns from sanitized-empty sendPrivmsg, and immediately quits transient sends. I did not perform a live IRC run in this read-only review.

## Solution Assessment

No. The repair direction is sound, but this branch should not merge until QUIT ordering, reclaim timer lifetime, duplicate reclaim behavior, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QUIT when guarding closed sends:** `extensions/irc/src/client.ts:175-177`
  - body: The new guard rejects whenever `closed` is true, but `quit()` sets `closed = true` before it sends `QUIT` through `sendRaw`. Every graceful shutdown now swallows that error and closes the socket without a QUIT frame, so move the closed transition until after QUIT is written or bypass this guard for shutdown.
  - confidence: 0.94
- **[P2] Tie reclaim timers to client shutdown:** `extensions/irc/src/client.ts:354-369`
  - body: The delayed NickServ GHOST/NICK timers are not stored, cleared, or `unref()`'d. Once scheduled, they can keep one-shot IRC sends alive after `quit()` and can fire against a client that has already been closed, so track these timers and clear or unref them as part of shutdown.
  - confidence: 0.86
- **[P3] Avoid duplicate NickServ reclaim attempts:** `extensions/irc/src/client.ts:346-350`
  - body: `tryRecoverNickCollision()` can already send GHOST/NICK during collision recovery. If fallback login still reaches welcome, this delayed block can send a second reclaim sequence for the same desired nick; gate it on whether recovery was already attempted.
  - confidence: 0.74
