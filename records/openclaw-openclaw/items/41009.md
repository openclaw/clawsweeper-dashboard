## Summary

Keep open. Current main still lacks the requested Serve/Funnel bind-to-loopback migration, so the PR is not obsolete, but the branch is not merge-ready because PR head weakens explicit `auto` validation and targets the pre-refactor legacy migration files instead of the current doctor/runtime migration flow.

## What This Changes

The PR branch adds legacy config rules, migration logic, helper reuse, and tests to normalize literal non-loopback `gateway.bind` values to `loopback` for Tailscale Serve/Funnel configs.

## Best Possible Solution

Port the migration into `src/commands/doctor/shared/legacy-config-migrations.runtime.gateway.ts` with matching current legacy-rule detection, keep Serve/Funnel validation strict for explicit `auto`, and add focused coverage for literal `lan`, `tailnet`, and non-loopback `custom` sources plus env-resolved non-literal cases.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main validates `gateway.bind="lan"` plus `gateway.tailscale.mode="serve"` as invalid and the current runtime gateway migration list has no bind-to-loopback normalization for that case.

## Solution Assessment

No. The migration direction is appropriate, but this branch is not the best implementation because it weakens PR-head validation and uses obsolete migration paths; the safer path is a current-main runtime/doctor migration that leaves the validation contract unchanged.
