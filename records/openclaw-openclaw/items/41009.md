## Summary

Keep open. The PR addresses a real crash-loop class, but it is not safe to merge as-is because the branch targets the old legacy-migration layout and the validation refactor appears to let `gateway.bind: "auto"` bypass the Serve/Funnel loopback guard.

## What This Changes

The PR adds a legacy migration for Tailscale Serve/Funnel configs with explicit non-loopback gateway binds, shared bind-resolution helpers, and regression tests.

## Best Possible Solution

Port the bind normalization into the current doctor/shared gateway migration rules and startup-safe legacy detection path, add focused regression coverage and a changelog entry, and preserve the existing Serve/Funnel loopback validation semantics.

## Reproduction Assessment

Yes, source-reproducible. Current main rejects `gateway.bind="lan"` with `gateway.tailscale.mode="serve"`, the current legacy gateway migration registry lacks a bind-to-loopback migration, and startup asserts invalid snapshots before the later Control UI origin seed can help.

## Solution Assessment

No. The migration direction is right, but this PR should first be ported to the current migration architecture and should keep validation-specific handling for `gateway.bind: "auto"` intact.
