## Summary

Keep this PR open, but it should not merge as-is: it targets the old migration files, changes validation in a way that can let explicit `gateway.bind="auto"` pass Serve/Funnel checks, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a legacy config rule/migration and tests to normalize literal non-loopback Tailscale Serve/Funnel gateway binds to `loopback`, while reusing a shared bind predicate in validation.

## Best Possible Solution

Port the explicit non-loopback Serve/Funnel bind normalization into `src/commands/doctor/shared/legacy-config-migrations.runtime.gateway.ts` with a matching legacy rule, keep validation handling for `auto` separate from migration matching, and add focused regression coverage plus real runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main rejects `gateway.bind="lan"` with `gateway.tailscale.mode="serve"`, has no matching legacy migration, and validates the startup snapshot before later origin seeding can run.

## Solution Assessment

No. The migration direction is right, but this stale branch should be ported to the current doctor/shared migration architecture and should not reuse a legacy-migration predicate as the runtime validation predicate.
