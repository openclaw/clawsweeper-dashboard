## Summary

Keep open. This is a real source-reproducible config migration bug and this PR is still the canonical implementation path, but the current branch is not merge-ready because it targets retired migration files, relaxes explicit `auto` validation, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a legacy config migration, detection rule, helper reuse, and tests to normalize explicit non-loopback Tailscale Serve/Funnel gateway binds to `loopback`.

## Best Possible Solution

Port the normalization into the active doctor/shared gateway migration and legacy-rule registry, preserve strict validation for explicit `auto`, add focused regression coverage, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind="lan"` with `gateway.tailscale.mode="serve"`, and the active doctor/shared migration registry has no repair for that combination; no live launchd crash-loop was run in this read-only review.

## Solution Assessment

No. The migration direction is right, but this branch targets the retired migration surface and treats explicit `auto` as safe in validation; the safer fix is to port the migration/rule into the active registry while keeping the current validation invariant.
