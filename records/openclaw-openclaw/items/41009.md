## Summary

Keep open: the PR targets a real upgrade crash-loop, but the branch is stale against current main and its migration is wired to the old legacy migration layout rather than the current doctor/runtime startup path.

## What This Changes

The PR adds a legacy config migration and rule to normalize explicit non-loopback `gateway.bind` values to `loopback` for Tailscale Serve/Funnel, reuses loopback bind helpers in validation, and adds regression tests.

## Best Possible Solution

Port the Serve/Funnel bind normalization into the current runtime/doctor migration and startup-read architecture so literal legacy configs are repaired before validation, while preserving the existing `auto` fail-safe and non-Tailscale behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current validation rejects `gateway.bind="lan"` with `gateway.tailscale.mode="serve"`, and startup throws on invalid snapshots before a general legacy repair can run. I did not execute the tests because this review is read-only and `pnpm` is unavailable.

## Solution Assessment

No. The direction is right, but this PR is no longer the best current-main fix because it edits the old legacy migration/rule files and needs to be rebased into the current doctor/runtime migration path without weakening `auto` validation.
