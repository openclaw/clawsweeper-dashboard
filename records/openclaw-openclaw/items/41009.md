## Summary

Keep open: the bug is valid and current main still lacks the requested migration, but this PR is not merge-ready because it targets retired migration files, weakens the PR-head Serve/Funnel `auto` validation path, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds legacy config detection, migration tests, a shared gateway bind helper, and validation reuse to normalize explicit non-loopback Tailscale Serve/Funnel binds to `loopback`.

## Best Possible Solution

Port the normalization into the active doctor/shared gateway migration registry with matching legacy detection and startup/doctor regression coverage, while preserving strict validation for explicit `auto`.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind="lan"` with `gateway.tailscale.mode="serve"`, and the active migration registry has no repair for that combination; no live launchd crash-loop was run in this read-only review.

## Solution Assessment

No. The migration direction is right, but this branch targets retired migration files and changes validation through a helper that excludes explicit `auto`; the safer fix is to port the migration/rule into the active doctor/shared registry and keep validation strict.
