## Summary

Keep open: this is a real user-facing config migration bug, but the PR is not correct on current main because it targets retired migration files, lets explicit `auto` skip the Serve/Funnel config guard, and still lacks inspectable after-fix real behavior proof.

## What This Changes

The PR adds legacy config detection, migration logic, helper reuse, and tests to normalize explicit non-loopback Tailscale Serve/Funnel `gateway.bind` values to `loopback`.

## Best Possible Solution

Port the normalization and legacy rule into the active doctor/shared gateway migration registry, preserve the loopback-only invariant including explicit `auto`, add focused regression coverage, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind="lan"` with `gateway.tailscale.mode="serve"`, and the active doctor/shared migration registry has no repair for that combination; no live launchd crash-loop was run in this read-only review.

## Solution Assessment

No. The migration direction is right, but this branch targets retired migration files and relaxes explicit `auto` validation; the maintainable fix belongs in the active doctor/shared registry while keeping the current loopback-only guard.
