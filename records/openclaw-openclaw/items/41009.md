## Summary

Keep this PR open, but it should not merge as-is: the underlying migration bug is still real on current main, while the PR branch weakens the Serve/Funnel loopback validation contract and is stale against the current migration layout.

## What This Changes

The PR adds config validation helpers, legacy-rule detection, migration coverage, and tests to auto-normalize literal non-loopback `gateway.bind` values to `loopback` for Tailscale Serve/Funnel configs.

## Best Possible Solution

Port the migration to the current `src/commands/doctor/shared/legacy-config-migrations.runtime.gateway.ts` flow, preserve the existing validation semantics for explicit `auto`, add source-literal legacy detection for `lan`, `tailnet`, and non-loopback `custom` under `serve`/`funnel`, and cover ordering before the existing allowed-origins migration with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main still rejects `gateway.bind="lan"` plus `gateway.tailscale.mode="serve"` and has no matching migration, while PR head accepts explicit `auto` because the new helper returns false before validation reports an issue.

## Solution Assessment

No. The migration direction is appropriate, but this patch is not the best implementation because it relaxes a security-sensitive validation check and targets obsolete migration files; the safer fix is a current-main runtime/doctor migration that leaves validation strict.
