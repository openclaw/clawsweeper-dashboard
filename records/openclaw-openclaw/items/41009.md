## Summary

Keep open: the underlying crash-loop migration bug is still present on current main, and this PR is the canonical implementation path, but the branch is stale/conflicting, targets retired migration files, relaxes explicit `auto` validation, and still lacks real behavior proof.

## What This Changes

The PR adds legacy config detection, migration logic, helper reuse, and tests to normalize explicit non-loopback Tailscale Serve/Funnel gateway binds to `loopback`.

## Best Possible Solution

Port the normalization rule and legacy detection to the active doctor/shared gateway migration registry before Control UI origin seeding, preserve explicit `auto` validation, add focused regression coverage, and merge only after redacted real behavior proof is attached.

## Reproduction Assessment

Yes, at source level. Current main rejects `gateway.bind="lan"` with `gateway.tailscale.mode="serve"`, and the active doctor/shared migration registry has no repair for that combination; no live launchd crash-loop was run in this read-only review.

## Solution Assessment

No. The migration direction is right, but this branch targets retired migration files and relaxes explicit `auto` validation; the maintainable fix belongs in the active doctor/shared registry while keeping the current loopback-only guard.
