## Summary

Keep open: the underlying upgrade crash-loop bug is still valid on current main, but this PR is not mergeable as-is because it targets an obsolete migration location and weakens the Serve/Funnel bind validation path for explicit `gateway.bind="auto"`.

## What This Changes

The PR adds a legacy config migration, legacy-rule detection, shared bind helpers, and regression tests to normalize explicit non-loopback Tailscale Serve/Funnel binds to `loopback`.

## Best Possible Solution

Port the explicit non-loopback Serve/Funnel bind normalization into `src/commands/doctor/shared/legacy-config-migrations.runtime.gateway.ts` with a matching legacy rule, keep validation-specific handling for `auto`, and cover startup recovery plus config validation with focused regression tests.

## Reproduction Assessment

Yes. Current main source-reproduces the failure path: `gateway.bind="lan"` with `gateway.tailscale.mode="serve"` is invalid, no matching legacy migration exists, and startup validates before the later origin-seeding path can help.

## Solution Assessment

No. The migration direction is right, but this PR should be ported to the current doctor/shared migration architecture and should keep migration matching separate from the Serve/Funnel runtime validation predicate.
