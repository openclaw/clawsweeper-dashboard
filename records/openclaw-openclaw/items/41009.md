## Summary

Keep open: the underlying migration gap still appears real on current main, but this PR is not merge-ready because its head weakens explicit `auto` bind validation and targets migration files that no longer exist on current main.

## What This Changes

The PR branch adds legacy config rules, migration logic, helper reuse, and tests to normalize literal non-loopback `gateway.bind` values to `loopback` for Tailscale Serve/Funnel configs.

## Best Possible Solution

Add the Serve/Funnel bind normalization to the current doctor/runtime legacy migration structure, keep explicit `auto` rejected for Serve/Funnel validation, and cover literal `lan`, `tailnet`, non-loopback `custom`, and env-resolved non-literal cases.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main rejects `gateway.bind="lan"` plus `gateway.tailscale.mode="serve"`, and the current gateway runtime migration file has no bind-to-loopback normalization for that legacy combination.

## Solution Assessment

No. The migration direction is appropriate, but this branch is not the best implementation because it weakens PR-head validation and uses obsolete migration paths; the safer path is a current-main migration that leaves the validation contract strict.
