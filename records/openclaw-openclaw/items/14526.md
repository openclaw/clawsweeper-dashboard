## Summary

Keep open. Current main has adjacent updater hardening, default-off auto-update, a config-only pre-update snapshot, manual backup/verify, gateway health/version restart checks, and manual rollback guidance, but it still does not implement the requested opt-in safe-update workflow tying full state backup, retention, channel/agent smoke checks, rollback/downgrade, and state restore into one supported mode.

## What This Changes

The issue proposes an opt-in safe-update workflow with pre-update backups, health checks, retention policy, rollback, and state restore.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement an opt-in orchestrator around existing update, backup, health/status, channel/agent smoke, retention, rollback, and restore primitives with explicit secret and downgrade guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. Source, docs, tests, and live issue inspection show adjacent primitives exist, but no supported mode ties full backup, retention, smoke checks, rollback, and state restore into the updater.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is either a designed core updater mode that reuses existing primitives or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.
