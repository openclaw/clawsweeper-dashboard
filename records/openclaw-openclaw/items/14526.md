## Summary

Keep open: current main has useful updater, auto-update, manual backup, plugin convergence, and gateway health/version checks, but it still lacks the requested opt-in safe-update workflow with update-coupled full snapshots, retention, channel/real-agent smoke checks, and controlled rollback plus state restore.

## What This Changes

Add an opt-in safe update workflow that snapshots OpenClaw state before updating, runs post-update health/channel/agent checks, and rolls back or restores state when checks fail.

## Best Possible Solution

Make a maintainer decision on whether safe update belongs in core; if accepted, design one documented opt-in updater mode that composes backup/verify, deep status, channel readiness, real-agent smoke, rollback/restore, retention, and secret-handling policy.

## Solution Assessment

Unclear until maintainers choose the product and security policy. The direction is reasonable, but automatic rollback, restore guarantees, retention limits, smoke criteria, and optional failure reporting need explicit design before implementation.

## Risks / Open Questions

- Safe-update backups would package credentials, auth profiles, sessions, config, logs, and possibly workspace data, so retention, permissions, encryption, and redaction policy need explicit security review.
- Automatic downgrade or restore can revive stale secrets, stale session state, or schema-incompatible config; maintainers need to choose prompted versus automatic behavior and prove upgrade paths.
- Real channel and agent smoke checks may require live credentials and can be slow or flaky, so pass/fail criteria and opt-in defaults need product judgment.
- The newer optional failure-reporting idea overlaps telemetry and privacy policy and should not be folded into a one-shot automated fix without maintainer approval.
