## Summary

Keep open. Current main has adjacent updater, backup, config-snapshot, plugin-convergence, and gateway restart-health primitives, but it still lacks the integrated opt-in safe-update workflow, and recent maintainer-facing discussion explicitly kept the broader proposal active.

## What This Changes

The issue requests an optional safer self-update workflow with pre-update backups, post-update health and smoke checks, retention policy, and rollback or restore behavior.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement it as an opt-in updater orchestrator with explicit backup, retention, smoke-check, rollback, restore, and secret-handling guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. Source, docs, tests, and live discussion show adjacent primitives exist, but no supported mode ties full backup, retention, smoke checks, rollback, and state restore into the updater.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is either a designed core updater mode that reuses existing primitives or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.
