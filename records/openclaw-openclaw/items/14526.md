## Summary

Keep open. Current main has updater, auto-update config, manual backup/verify, health/status, and manual rollback guidance, but it still lacks the requested opt-in safe-update workflow that ties snapshots, retention, channel/agent smoke checks, rollback, and state restore into one supported mode.

## What This Changes

Add an optional safe-update workflow that snapshots OpenClaw state before updating, runs post-update health and smoke checks, and rolls back or restores on failure.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement an opt-in orchestrator around existing update, backup, health/status, channel/agent smoke, retention, rollback, and restore primitives with explicit secret and downgrade guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. Source and docs inspection show adjacent primitives exist, but no supported mode ties backup, retention, smoke checks, rollback, and restore into the updater.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is either a designed core updater mode that reuses existing primitives or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Risks / Open Questions

- Safe-update backups would archive secrets-bearing local state, so retention, permissions, deletion, and disclosure behavior need explicit design.
- Automated downgrade plus state restore can break config compatibility unless maintainers define confirmation, compatibility checks, and failure boundaries.
- Real channel and agent smoke checks may require live credentials and can be slow or flaky, so opt-in defaults and pass/fail criteria need product judgment.
