## Summary

This should stay open: current main has update, auto-update, manual backup/verify, gateway health, channel status, and package-swap safety primitives, but it still lacks the requested integrated opt-in safe-update workflow with update-coupled backups, retention, real-agent smoke checks, and controlled rollback plus state restore.

## What This Changes

Request an optional safe-update workflow that snapshots OpenClaw state before update, runs post-update health and smoke checks, and rolls back or restores on failure.

## Best Possible Solution

Make a maintainer decision on whether safe update belongs in core; if accepted, design one documented opt-in updater mode that orchestrates backup/verify, status/deep health, channel readiness, real-agent smoke, rollback/restore, retention, and secret-handling policy.

## Solution Assessment

Unclear until maintainers choose the product and security policy. The direction is reasonable, but automatic rollback, restore guarantees, retention limits, smoke criteria, and optional failure reporting need explicit design before implementation.

## Risks / Open Questions

- Safe-update backups would package credentials, auth profiles, sessions, config, logs, and possibly workspace data, so retention, permissions, encryption, and redaction policy need explicit security review.
- Automatic rollback or downgrade can revive stale secrets, session state, or schema-incompatible config; maintainers need to choose prompted versus automatic behavior and prove both fresh-install and upgrade paths.
- Real channel and agent smoke checks may require live credentials and can be slow or flaky, so the pass/fail criteria and opt-in defaults need product judgment.
- The newer optional failure-reporting idea overlaps telemetry and privacy policy and should not be folded into a one-shot automated fix without maintainer approval.
