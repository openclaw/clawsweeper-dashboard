## Summary

Keep open. Current main has updater, backup, health/status, Control UI update, and limited package-swap safety primitives, but not the requested opt-in safe-update workflow that ties pre-update snapshots, retention, gateway/channel/real-agent checks, and rollback/restore together.

## What This Changes

Request to add an optional safe-update workflow with pre-update backup, post-update health checks, and rollback or restore behavior.

## Best Possible Solution

Make a maintainer scope decision for safe update as a first-class updater mode; if accepted, implement an explicit opt-in orchestrator around existing update, backup, doctor, status/health, channel-status, Control UI update, and package rollback primitives with secret-safe retention and compatibility semantics.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence verification path is source/docs inspection, and current main still lacks a safe-update CLI/API workflow tying backups, health checks, smoke checks, rollback, and restore together.

## Solution Assessment

No. The current adjacent primitives are not the complete solution; the narrowest maintainable direction is a designed core updater mode or a maintainer-approved split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes automated local-state archiving plus automated downgrade and restore behavior.

Concerns:

- **[medium] Define secret-safe backup retention and restore policy:** `docs/cli/backup.md:11`
  - body: Safe-update backups can include credentials, auth profiles, sessions, and workspace data, while current backup docs leave size and retention unmanaged. The design needs explicit permissions, storage, retention, deletion, and leakage controls before implementation.
  - confidence: 0.88
- **[medium] Constrain automatic downgrade and state restore:** `docs/install/updating.md:188`
  - body: Rollback is currently documented as a manual operator action. Automating downgrade plus state restore can run older code against state or config changed by a newer version, so compatibility checks and confirmation rules should be designed before automation.
  - confidence: 0.86
