## Summary

Current main and v2026.6.5 provide adjacent updater, auto-update, manual backup/verify, doctor, plugin payload smoke, restart-safety, repair, and manual rollback guidance, but they still do not implement an update-coupled snapshot with retention, channel or real-agent verification, automatic downgrade, and state restore. The production incident context and external openclaw-auto-update-kit are useful evidence, while the related open items cover narrower scheduling, scanning, and Control UI surfaces rather than superseding this broader recovery contract.

## What This Changes

Add an optional safe-update workflow that snapshots critical state, verifies the updated runtime, and rolls back or restores on failure.

## Best Possible Solution

Define a core-owned transactional update protocol with a version-aware restorable snapshot boundary, non-destructive preflight, layered gateway/plugin/channel/agent verification, and operator-controlled downgrade or restore semantics before splitting implementation into focused PRs.

## Reproduction Assessment

Not applicable: this issue proposes a new integrated update mode rather than one broken existing behavior. Multiple linked production incidents demonstrate the operational need, but they have distinct underlying failures.

## Solution Assessment

The integrated core orchestration direction is reasonable, but automatic package downgrade plus generic archive restore is not yet the best complete design; version-aware migration compatibility, intentionally excluded live state, smoke side effects, and secret retention need explicit contracts first.

## Risks / Open Questions

- Automatic downgrade can leave config, plugin metadata, or state written by the newer version incompatible with the restored older binary.
- The existing backup archive intentionally excludes active transcripts and queues, so treating it as a complete automatic rollback image could silently lose recent state.
- Channel and real-agent smoke checks may send messages, consume provider quota, mutate sessions, or disclose operational details unless their side effects and privacy boundaries are explicit.
- Snapshot retention and archives containing credentials or auth profiles need secure permissions, storage, deletion, and failure-handling policy.
