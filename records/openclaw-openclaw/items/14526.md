## Summary

Keep open. Current main has updater, auto-update, manual backup/verify, gateway health/version verification, and manual rollback guidance, but it still lacks the requested opt-in safe-update workflow tying pre-update snapshots, retention, channel/agent smoke checks, package downgrade, and state restore into one supported mode.

## What This Changes

The issue requests an optional safe-update mode with pre-update backups, post-update health/smoke checks, retention policy, and rollback or state restore on failure.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement an opt-in orchestrator around existing update, backup, health/status, channel/agent smoke, retention, rollback, and restore primitives with explicit secret and downgrade guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. Source and docs inspection show adjacent primitives exist, but no supported mode ties backup, retention, smoke checks, rollback, and restore into the updater.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is either a designed core updater mode that reuses existing primitives or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.
