## Summary

Keep open: current main and v2026.5.22 have useful update, backup, doctor, and gateway-health primitives, but not the integrated opt-in safe-update workflow with update-coupled backup, retention, real channel/agent smoke checks, automatic rollback, and state restore.

## What This Changes

Request an optional safe-update workflow that snapshots OpenClaw state before update, verifies the post-update system, and rolls back or restores state when checks fail.

## Best Possible Solution

Make a maintainer product/security decision on whether safe update belongs in core; if accepted, design one documented opt-in updater mode that composes backup/verify, deep status, channel readiness, real-agent smoke, rollback/restore, retention, and secret-handling policy.

## Solution Assessment

Unclear until maintainers choose the product and security policy. An opt-in core workflow is plausible, but automatic rollback, restore guarantees, retention limits, smoke criteria, and optional failure reporting need explicit design first.

## Risks / Open Questions

- Safe-update archives can include credentials, auth profiles, sessions, config, logs, and private workspace data, so retention, permissions, encryption, and redaction policy need explicit security review.
- Automatic downgrade or restore can revive stale secrets, stale session state, or schema-incompatible config unless maintainers define prompted versus automatic behavior and compatibility checks.
- Real channel and agent smoke checks may require live credentials and can be slow or flaky, so pass/fail criteria and opt-in defaults need product judgment.
- The optional failure-reporting idea added in discussion overlaps telemetry and privacy policy and should not be folded into an automated fix without maintainer approval.
