## Summary

Keep open. Current main has updater, auto-update, manual backup/verify, gateway health/version verification, and manual recovery guidance, but it still does not provide the requested opt-in safe-update workflow tying pre-update snapshots, retention, channel/agent smoke checks, rollback/downgrade, and state restore into one supported mode; the Bortlesboat operator kit and production evidence are useful prior art, not a core implementation.

## What This Changes

Request an opt-in safe-update workflow that creates a pre-update backup, runs post-update health and smoke checks, and rolls back or restores state when checks fail.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement an opt-in orchestrator around existing update, backup, health/status, channel/agent smoke, retention, rollback, and restore primitives with explicit secret and downgrade guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. Source, docs, tests, and live issue inspection show adjacent primitives exist, but no supported mode ties backup, retention, smoke checks, rollback, and restore into the updater.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is either a designed core updater mode that reuses existing primitives or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Risks / Open Questions

- Safe-update backups would archive secrets-bearing local state, so retention, permissions, deletion, and disclosure behavior need explicit design.

- Real channel and agent smoke checks may require live credentials and can be slow or flaky, so opt-in defaults and pass/fail criteria need maintainer product judgment.
