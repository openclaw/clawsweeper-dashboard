## Summary

Keep open. Current main has adjacent updater, backup create/verify, config snapshot, restart health, auto-update, and manual rollback guidance, but it still does not provide the proposed opt-in safe-update workflow with update-coupled full backup, retention, channel/agent smoke checks, automatic rollback/downgrade, and state restore; the live discussion and related items also keep this broader work alive rather than superseding it.

## What This Changes

The issue proposes an opt-in safe-update workflow that backs up OpenClaw state before updating, runs post-update health/channel/agent checks, and rolls back or restores on failure.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement an opt-in orchestrator around existing update, backup, health/status, channel/agent smoke, retention, rollback, and restore primitives with explicit secret and downgrade guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. Source, docs, and live related-item inspection show adjacent primitives exist, but no supported mode ties full backup, retention, smoke checks, rollback, and state restore into the updater.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is either a designed core updater mode that reuses existing primitives or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Risks / Open Questions

- Safe-update backups would archive secrets-bearing local state, so retention, permissions, deletion, and disclosure behavior need explicit design.
- Automatic downgrade plus state restore can worsen schema compatibility or lose state unless confirmation, compatibility checks, and partial-failure boundaries are defined.
- Channel and real-agent smoke checks may require live credentials and can be slow or flaky, so opt-in defaults and pass/fail criteria need maintainer judgment.
- The open rollback PR covers only one package-doctor subset and should be coordinated rather than treated as a full replacement.
