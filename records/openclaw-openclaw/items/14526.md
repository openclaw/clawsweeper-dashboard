## Summary

Keep open. Current main has manual update, backup, health/status, channel probe, and rollback-guidance primitives, but it still lacks the requested opt-in safe-update workflow that creates update-tied snapshots, enforces retention, runs channel/real-agent smoke checks, and automatically downgrades plus restores state on failure.

## What This Changes

Request an opt-in safe update workflow that backs up OpenClaw state before updating, runs post-update health and smoke checks, and offers or performs rollback/restore if checks fail.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement an opt-in orchestrator around existing update, backup, health/channel/agent checks, retention, rollback, and restore with explicit secrets and compatibility guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. The high-confidence verification path is source and docs inspection, which shows no updater mode tying backups, retention, smoke checks, rollback, and restore together.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The narrowest maintainable direction is either a designed core updater mode or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Security Review

Status: needs_attention

Summary: The proposal is security-sensitive because it would archive secrets-bearing local state and automate downgrade/restore behavior.

  - confidence: 0.9

  - body: Rollback is currently documented as a manual operator action, and the update docs warn that downgrades can break configuration. Automating downgrade plus restore needs compatibility checks and confirmation rules before implementation.
