## Summary

Keep open. Current main has adjacent updater, backup, health/status, Control UI update, and package-swap rollback primitives, and the discussion includes useful production evidence plus an external operator kit, but the requested opt-in safe-update workflow with update-tied snapshots, retention, channel/real-agent checks, and automatic downgrade plus state restore is still missing.

## What This Changes

Add an optional safe-update workflow that creates pre-update backups, runs post-update health and smoke checks, and can roll back or restore state on failure.

## Best Possible Solution

Make a maintainer scope decision for safe update as a first-class core updater mode; if accepted, implement an explicit opt-in orchestrator around existing update, backup, doctor, status/health, channel-status, Control UI update, and package rollback primitives with secret-safe retention and compatibility semantics.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence verification path is source and docs inspection, and current main still lacks a safe-update workflow tying backups, retention, smoke checks, rollback, and restore together.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The current adjacent primitives are not the complete solution; the narrowest maintainable direction is a designed core updater mode or a maintainer-approved split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Security Review

Status: needs_attention

Summary: This feature is security-sensitive because it would automate local-state archiving and downgrade/restore behavior.

Concerns:

- **[medium] Define secret-safe backup retention and restore policy:** `docs/cli/backup.md:11`
  - body: Backups include config, auth profiles, channel/provider credentials, sessions, and optionally workspaces, while current docs leave size and retention unmanaged. A safe-update design needs explicit storage, permissions, retention, deletion, and leakage controls.
  - confidence: 0.88
- **[medium] Constrain automatic downgrade and state restore:** `docs/install/updating.md:194`
  - body: Rollback is currently documented as a manual operator action. Automating downgrade plus state restore can run older code against state or config changed by a newer version, so compatibility checks and confirmation rules should be designed first.
  - confidence: 0.86
