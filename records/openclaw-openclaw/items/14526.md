## Summary

Keep open: current main and latest release still provide adjacent update, auto-update, backup, health, channel probe, and manual recovery primitives, but not the requested integrated opt-in safe-update workflow with update-coupled backups, retention, real-agent smoke checks, automatic rollback, and state restore.

## What This Changes

Request an optional safer update workflow that snapshots OpenClaw state before updating, runs post-update health/smoke checks, and rolls back or restores state when checks fail.

## Best Possible Solution

Make a maintainer product/security decision on whether safe update belongs in core; if accepted, design one documented opt-in updater mode that composes backup/verify, deep status, channel readiness, real-agent smoke, rollback/restore, retention, and secret-handling policy.

## Risks / Open Questions

- Safe-update backups would package credentials, auth profiles, sessions, config, logs, and workspace data, so storage location, permissions, encryption, redaction, and retention policy need explicit security review.
- Automatic downgrade or restore can revive stale secrets, stale session state, or schema-incompatible config; maintainers need to choose prompted versus automatic behavior and define rollback boundaries.
- Real channel and agent smoke checks may require live credentials and can be slow or flaky, so pass/fail criteria and opt-in defaults need product judgment.
- The optional update-break reporting idea overlaps telemetry and privacy policy and should not be folded into an automated fix without maintainer approval.
