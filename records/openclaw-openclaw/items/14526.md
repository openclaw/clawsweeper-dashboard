## Summary

Keep open: current main has adjacent updater, backup, auto-update, plugin convergence, and gateway health/version checks, but it still lacks the requested opt-in safe-update workflow with update-coupled full-state snapshots, retention, live channel/agent smoke checks, and rollback plus state restore.

## What This Changes

The issue proposes an optional safe-update workflow that backs up OpenClaw state before updating, runs post-update health and smoke checks, and rolls back or restores state on failure.

## Best Possible Solution

Make a maintainer product/security decision on whether safe update belongs in core; if accepted, design one documented opt-in updater mode that composes backup/verify, deep status, channel readiness, real-agent smoke, rollback/restore, retention, and secret-handling policy.

## Risks / Open Questions

- Safe-update backups would package credentials, auth profiles, sessions, config, logs, and workspace data, so retention, permissions, encryption, and redaction policy need explicit security review.
- Automatic downgrade or restore can revive stale secrets, stale session state, or schema-incompatible config; maintainers need to choose prompted versus automatic behavior and prove upgrade paths.
- Real channel and agent smoke checks may require live credentials and can be slow or flaky, so pass/fail criteria and opt-in defaults need product judgment.
- The optional failure-reporting idea overlaps telemetry and privacy policy and should not be folded into an automated fix without maintainer approval.
