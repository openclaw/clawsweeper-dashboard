## Summary

Keep open. Current main and v2026.5.12 have useful updater, backup, config-snapshot, plugin convergence, and gateway restart-health primitives, but they still do not provide the requested opt-in safe-update workflow that ties full backups, retention, smoke checks, rollback, and state restore together.

## What This Changes

The issue proposes an opt-in safe update workflow with pre-update state backup, post-update health and smoke checks, and rollback or restore behavior.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement it as an opt-in updater orchestrator with explicit backup, retention, smoke-check, rollback, restore, and secret-handling guarantees.

Unclear until maintainers choose the product and security boundary. The maintainable direction is either a designed core updater mode that reuses existing primitives or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Risks / Open Questions

- Channel and real-agent smoke checks may require live credentials and can be slow or flaky, so maintainers need to define opt-in defaults and pass/fail criteria.
