## Summary

Keep open. Current main still has adjacent updater, manual backup, health/status, and rollback-guidance primitives, but not the requested integrated safe-update workflow with update-tied backup, retention, channel/agent smoke checks, and automatic rollback plus state restore.

## What This Changes

The issue requests an opt-in safe-update workflow with pre-update backups, post-update health and smoke checks, retention controls, and rollback/restore behavior.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement an opt-in orchestrator around existing update, backup, health/status, channel/agent smoke, retention, rollback, and restore primitives with explicit secret and downgrade guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. Source and docs inspection provide high confidence that no current mode ties backups, retention, smoke checks, rollback, and restore into the updater.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The narrow maintainable direction is either a designed core updater mode or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Security Review

Status: needs_attention

Summary: The requested workflow is security-sensitive because it would archive secrets-bearing local state and automate downgrade/restore behavior.

Concerns:

- **[medium] Define secret-safe backup retention and storage policy:** `docs/cli/backup.md:11`
  - body: Backups include config, auth profiles, channel/provider credentials, sessions, and optionally workspaces, while current docs leave size and retention unmanaged. A safe-update design needs explicit permissions, retention, deletion, and leakage controls.
  - confidence: 0.9
- **[medium] Constrain automatic downgrade and state restore:** `docs/install/updating.md:200`
  - body: Rollback is currently documented as manual operator action, and update docs warn that downgrades can break configuration. Automating downgrade plus restore needs compatibility checks and confirmation rules.
