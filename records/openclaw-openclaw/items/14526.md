## Summary

Keep open. Current main has adjacent updater, Control UI update, backup/verify, doctor, health/status, and manual rollback primitives, but it still does not implement the requested opt-in safe-update workflow tying pre-update snapshots, retention, gateway/channel/real-agent checks, and controlled rollback/restore together.

## What This Changes

The issue requests an optional safer self-update mode with pre-update backup, post-update health and smoke checks, retention policy, and rollback or state restore on failure.

## Best Possible Solution

Make a maintainer scope decision for safe update as a first-class updater mode; if accepted, implement an explicit opt-in orchestrator around existing update, backup, doctor, status/health, channel-status, Control UI update, and package rollback primitives with clear secret-preservation and compatibility semantics.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence verification path is source/docs inspection, and current main still lacks a safe-update CLI/API workflow tying backups, health checks, smoke checks, rollback, and restore together.

## Solution Assessment

No. The current adjacent primitives are not the complete solution; the narrowest maintainable direction is a designed core updater mode or a maintainer-approved split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.
