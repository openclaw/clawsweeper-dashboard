## Summary

Keep open. Current main and v2026.5.12 still have useful updater, manual backup, config-snapshot, plugin convergence, and restart-health primitives, but they do not implement the requested opt-in safe-update workflow tying full backups, retention, channel or real-agent smoke checks, automatic downgrade, and state restore together. The live discussion also adds concrete rollout evidence and an external operator kit rather than making the core request obsolete.

## What This Changes

Add an opt-in safe-update workflow that creates pre-update backups, runs post-update health and smoke checks, and rolls back or restores state on failure.

## Best Possible Solution

Make a maintainer scope decision for a first-class safe-update mode; if accepted, implement it as an opt-in updater orchestrator with explicit backup, retention, smoke-check, rollback, restore, and secret-handling guarantees.

## Reproduction Assessment

Not applicable: this is a feature proposal rather than a current-main bug. Source, docs, latest-release inspection, and live issue context show adjacent primitives exist, but no supported mode ties full backup, retention, smoke checks, rollback, and state restore into the updater.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is either a designed core updater mode that reuses existing primitives or an explicit split across backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Risks / Open Questions

- Safe-update backups would archive credentials, auth profiles, sessions, workspaces, and possibly cron state, so retention, permissions, deletion, and disclosure behavior need explicit policy.
- Automatic downgrade plus state restore can create schema, plugin, and session incompatibilities unless confirmation, compatibility checks, and partial-failure boundaries are designed first.
- Channel and real-agent smoke checks may require live credentials and can be slow or flaky, so maintainers need to define opt-in defaults and pass/fail criteria.
