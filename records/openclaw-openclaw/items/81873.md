## Summary

Keep open: current main implements the low-level Chrome MCP `cdpUrl` argument mapping, but the WSL2/user-profile contract is still inconsistent enough that cleanup-closing this fresh installed-build report would be premature.

## What This Changes

The issue reports that an existing-session browser profile in WSL2 ignores configured `cdpUrl`/Chrome MCP arguments and still launches Chrome MCP with `--autoConnect`.

## Best Possible Solution

Align the browser profile contract: either officially support configured existing-session `cdpUrl` across manual config, profile creation, WSL2 docs, and diagnostics, or enforce/communicate raw `attachOnly` remote CDP as the supported WSL2 Windows Chrome path.

## Reproduction Assessment

Unclear. The reporter provides concrete WSL2 steps and logs from 2026.5.7, but I did not reproduce WSL2 plus Windows Chrome; current main forwards resolved existing-session profile objects while still showing source-visible contract contradictions.

## Solution Assessment

Unclear. Passing the resolved profile object is already implemented on current main, so the safer next step is a maintainer decision on the intended `existing-session` versus raw remote CDP contract before changing behavior again.
