## Summary

Keep open. Current source and the prior merged Chrome MCP launch work cover the basic `cdpUrl` pass-through, but this fresh report exposes unresolved contract conflicts across docs, profile creation, and WSL split-host guidance; LakshmanTurlapati's FSB bridge comment is useful adjacent context, not a reason to close the OpenClaw issue.

## What This Changes

The issue asks OpenClaw to honor `browser.profiles.user.cdpUrl` for Chrome MCP existing-session profiles in WSL2 by launching Chrome MCP with `--browserUrl` instead of `--autoConnect`.

## Best Possible Solution

Align the browser profile contract so either existing-session `cdpUrl` is supported consistently across manual config, profile creation, docs, and runtime, or WSL2 + Windows Chrome users are clearly routed to raw remote CDP or a browser-side bridge.

## Reproduction Assessment

Unclear. Current source should forward `cdpUrl` for an explicitly resolved existing-session profile, but I did not establish a WSL2 + Windows Chrome run and the repository still has contradictory config/docs paths.

## Solution Assessment

Unclear. A pass-through fix appears to already exist for the central Chrome MCP argument builder; the maintainable next step is to decide the supported WSL split-host contract before changing code or closing the report.
