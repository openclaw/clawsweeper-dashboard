## Summary

This should stay open: current main has pieces of the requested behavior, but the exact WSL2/user-profile contract is still unresolved and a related open PR only covers the top-level `browser.cdpUrl` default-profile gap.

## What This Changes

The issue reports that the `user` existing-session browser profile ignores configured `cdpUrl`/`mcpArgs` in WSL2 and launches Chrome MCP with `--autoConnect` instead of endpoint flags.

## Best Possible Solution

Settle the browser profile contract, then either land a focused resolver fix that keeps endpoint URLs redacted or update docs/diagnostics to steer WSL2 + Windows Chrome users to raw remote CDP instead of `user`/Chrome MCP.

## Reproduction Assessment

Unclear. Source inspection reproduces a related resolver gap for top-level `browser.cdpUrl` with `defaultProfile: "user"`, but current source should forward profile-local `cdpUrl` once a profile is actually resolved as existing-session, and I did not have the WSL2 + Windows Chrome environment to confirm the reported installed-build path.

## Solution Assessment

Unclear. The related resolver PR is a plausible narrow fix for one config shape, but this issue also needs a maintainer decision on whether WSL2 + Windows Chrome should be supported through existing-session endpoint forwarding or documented as raw remote CDP only.
