## Summary

The reported Chrome MCP launch mapping is already implemented on current main and shipped: existing-session profiles carry the resolved profile object into Chrome MCP, and cdpUrl is mapped to --browserUrl or --wsEndpoint instead of --autoConnect.

## What This Changes

The issue reports that existing-session browser profiles ignore cdpUrl or mcpArgs and still launch Chrome MCP with --autoConnect in a WSL2 plus Windows Chrome setup.

## Best Possible Solution

Keep the shipped full-profile Chrome MCP launch path, and direct WSL2 plus Windows Chrome users to a WSL-reachable raw remote CDP profile unless a new report shows the resolved existing-session profile still loses cdpUrl on a current release.

## Reproduction Assessment

No. I did not establish a failing current-main reproduction path; source and shipped history show the described cdpUrl/mcpArgs loss was fixed, and the WSL split-host setup should normally use raw remote CDP.

## Solution Assessment

Yes. Passing the resolved profile object into Chrome MCP is the narrow maintainable fix for the reported argument mapping; WSL-specific Windows Chrome path probing would be a separate feature rather than the core fix here.
