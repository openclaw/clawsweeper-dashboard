## Summary

Keep open. Current main has the earlier Chrome MCP cdpUrl forwarding fix, and that fix is in the reporter's stated build line, but this fresh WSL2 report exposes a remaining contract gap: code, profile creation, browser docs, and WSL guidance still disagree about whether existing-session cdpUrl is supported or whether WSL + Windows Chrome should use raw remote CDP.

## What This Changes

The issue reports that an existing-session browser profile configured with cdpUrl still launches Chrome MCP with --autoConnect under WSL2 instead of forwarding --browserUrl.

## Best Possible Solution

Choose and align one browser profile contract: either support existing-session cdpUrl consistently across manual config, profile creation, docs, and tests, or reject it clearly and steer WSL2 + Windows Chrome users to an attachOnly raw remote CDP profile with diagnostics.

## Reproduction Assessment

No. Source inspection shows the fully resolved existing-session path should forward cdpUrl, but the reported WSL2 installed flow was not live-reproduced and the documented config snippet may hit a different profile-resolution contract.

## Solution Assessment

Unclear. The prior cdpUrl forwarding fix is the narrow route-level solution and is shipped, but the remaining issue is better solved by a maintainer decision that aligns existing-session, raw remote CDP, docs, and profile creation behavior.
