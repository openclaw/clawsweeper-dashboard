## Summary

This fresh report should stay open: current `main` appears to carry resolved existing-session profile options through to Chrome MCP, but the WSL2 topology and profile/config contract are inconsistent enough that maintainers should decide the intended behavior before cleanup-closing it.

## What This Changes

The issue requests that existing-session Chrome MCP browser profiles honor configured `cdpUrl`/`mcpArgs` in WSL and launch Chrome MCP with endpoint flags instead of `--autoConnect`.

## Best Possible Solution

Align the browser profile contract: either make configured existing-session `cdpUrl` work consistently across manual config, profile creation, docs, and release packaging, or steer WSL2 + Windows Chrome users to a raw `attachOnly` remote CDP profile with diagnostics that explain why `user`/Chrome MCP is the wrong split-host mode.

## Reproduction Assessment

Unclear. Current source shows the main runtime paths should forward a resolved existing-session profile to Chrome MCP, but I did not have a WSL2 + Windows Chrome setup to confirm the reported installed-build failure.

## Solution Assessment

Unclear. Forwarding `cdpUrl` through Chrome MCP is already implemented in source, while the more maintainable WSL2 path may be raw remote CDP plus clearer diagnostics rather than encouraging `user`/existing-session across the host boundary.
