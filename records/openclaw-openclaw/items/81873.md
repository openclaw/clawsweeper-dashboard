## Summary

Keep open: current main forwards `cdpUrl` for resolved existing-session profiles, but this report still exposes an unresolved contract mismatch around the built-in `user` profile, profile creation, and WSL2 guidance.

## What This Changes

The issue asks for `browser.profiles.user.cdpUrl` in a WSL2 existing-session/Chrome MCP setup to be passed to Chrome MCP as `--browserUrl` instead of falling back to `--autoConnect`.

## Best Possible Solution

Maintain one clear browser profile contract: either support and test `cdpUrl` on the built-in `user` existing-session profile, including partial config merging, or document and diagnose WSL2 Windows Chrome as raw remote CDP only.

## Reproduction Assessment

Unclear. Source inspection shows a contract mismatch for the posted partial `browser.profiles.user` config, but current main should forward `cdpUrl` once the profile is resolved as `existing-session`, and I did not run WSL2 plus Windows Chrome.

## Solution Assessment

Unclear. The earlier Chrome MCP profile-object forwarding fix is already present; maintainers should first decide whether `cdpUrl` on built-in `user` is supported or whether WSL2 users should be steered to raw remote CDP with clearer diagnostics.
