## Summary

Keep open. Current main still has the Control UI ordering gap: the chat model picker persists selection with an optimistic `sessions.patch`, but app state, send controls, and `handleSendChat` do not expose or await a pending model patch before `chat.send` starts.

## What This Changes

The issue requests a Control UI guarantee that a model selected from the chat dropdown is persisted before the next chat send starts.

## Best Possible Solution

Add a Control UI-side per-session model-switch barrier: record the pending `sessions.patch`, disable model selection and send/queue entry points while it is pending, await it before dispatching sends, roll back optimistic selection and avoid sending on patch failure, and leave the Gateway `chat.send` protocol unchanged.

## Reproduction Assessment

Yes. A high-confidence regression can defer the dropdown `sessions.patch`, dispatch the select change, immediately trigger `handleSendChat`, and assert that `chat.send` is not called until the patch resolves successfully; current source has no state or await point to enforce that ordering.

## Solution Assessment

Yes. A UI-side barrier is the narrowest maintainable fix because the picker already owns session persistence and `chat.send` has no model override field; changing the protocol would be broader than needed for this race.

Fix the Control UI race from https://github.com/openclaw/openclaw/issues/54240. Add a per-session pending chat-model switch barrier in the UI so selecting a chat model stores the pending `sessions.patch`, disables the model picker and send/queue entry points while the patch is pending, and makes `handleSendChat` await the patch before any `chat.send`.
