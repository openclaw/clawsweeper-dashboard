## Summary

Keep open. Current `origin/main` still has the reported Control UI ordering gap: the model picker performs an optimistic async `sessions.patch`, but shared UI state, send controls, queued sends, and `handleSendChat` do not observe or await an in-flight model switch before `chat.send` starts.

## What This Changes

The issue asks for the Control UI to block or wait for an in-flight chat model `sessions.patch` before allowing the next chat send to run.

## Best Possible Solution

Add a Control UI-side per-session model-switch barrier: track pending `sessions.patch` operations, disable the model picker and send/queue controls while a switch is pending, await the pending patch before dispatching sends, roll back the optimistic selection and restore draft/attachments if the patch fails, and leave the Gateway `chat.send` protocol unchanged unless maintainers choose a broader API change.
