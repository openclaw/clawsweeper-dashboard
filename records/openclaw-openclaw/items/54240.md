## Summary

Keep open. Current main still has the Control UI ordering gap: the model picker performs an optimistic `sessions.patch`, but shared app state, send controls, and `handleSendChat` do not expose or await a pending model-switch operation before `chat.send` starts.

## What This Changes

The issue requests that Control UI model dropdown changes reliably persist before the next chat send, or that sending be blocked until persistence completes.

## Best Possible Solution

Add a Control UI-side per-session model-switch barrier: store the pending `sessions.patch`, disable the model picker and send/queue controls while it is pending, await it before dispatching sends, roll back optimistic selection on patch failure, and keep the Gateway `chat.send` protocol unchanged unless maintainers choose a broader protocol design later.

Add a per-session pending model-switch barrier in the Control UI: store the pending patch promise/status in app/view state, disable the model picker and send/queue controls while the active session has a pending model patch, await the pending patch before normal sends, queued sends, and any explicitly chosen related send paths, and roll back the optimistic selection without sending if the patch fails. Keep the Gateway `chat.send` protocol unchanged unless a maintainer explicitly asks for a broader API change. Add focused regression tests with a deferred `sessions.patch` proving `chat.send` waits and does not fire on patch failure.
