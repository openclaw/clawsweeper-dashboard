## Label Justifications

- `P1`: The report describes a real Google Chat channel workflow where Space messages are received but not processed, blocking group collaboration for affected users.
- `impact:message-loss`: The issue is about Google Chat Space messages being silently dropped after webhook receipt instead of reaching the agent pipeline.
- `impact:session-state`: The expected `googlechat:group` session is not created for affected Space messages, so conversation/session state never starts.

## Summary

Keep open because current main and the latest release still do not model Google Chat's modern Space metadata, and an open PR specifically uses closing syntax for this issue. The reporter also confirmed downstream operational relevance after the earlier ClawSweeper pass, so this should stay open until the Google Chat routing fix lands or maintainers explicitly consolidate it with the thread-routing work.

## What This Changes

Google Chat Space/group inbound messages should create and process `googlechat:group` sessions instead of returning HTTP 200 and silently dropping while DMs work.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/58993 or an equivalent classifier/type fix, then coordinate any remaining Space/thread metadata work through https://github.com/openclaw/openclaw/issues/69422 and https://github.com/openclaw/openclaw/pull/74235.

## Reproduction Assessment

Yes at source/reporter level: current main still lacks the modern Google Chat Space metadata fields and the linked PR/comment thread gives a concrete classifier gap plus live deployment confirmations. I did not run a live Google Workspace webhook replay for the exact HTTP 200/no-session symptom.

## Solution Assessment

Partially: the narrow classifier/type fix in https://github.com/openclaw/openclaw/pull/58993 is the right first repair for the modern metadata gap, but broader Space/thread preservation should stay coordinated with the separate thread-routing issue and PR.
