## Label Justifications

- `P1`: The report describes a real Google Chat channel workflow where Space messages are received but not processed, blocking group collaboration.
- `impact:message-loss`: The issue is about Google Chat Space messages being silently dropped after webhook receipt instead of reaching the agent pipeline.
- `impact:session-state`: The expected `googlechat:group` session is not created for affected Space messages, so conversation state never starts.

## Summary

Keep open: current main and the latest release still derive Google Chat direct/group routing only from legacy `space.type`, while the issue has an open closing PR for the classifier gap and separate open tracking for broader thread-routing metadata.

## What This Changes

Google Chat Space/group messages should create and process `googlechat:group` sessions instead of being acknowledged and then silently dropped while DMs work.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/58993 or an equivalent classifier/type fix, while keeping broader Google Chat thread-target preservation coordinated through https://github.com/openclaw/openclaw/issues/69422 and https://github.com/openclaw/openclaw/pull/74235.
