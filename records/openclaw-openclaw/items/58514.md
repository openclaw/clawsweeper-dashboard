## Summary

Keep open. Current main still relies on the deprecated Google Chat `space.type` field for DM-vs-group classification, the local `GoogleChatSpace` type still lacks the modern `spaceType` and `singleUserBotDm` fields, and an open PR with closing syntax is already attached to this issue. The reporter also reconfirmed on April 30, 2026 that the Google Chat Spaces behavior remains operationally relevant, so this should wait for PR review/landing or explicit consolidation with the related Google Chat routing work.

## What This Changes

The issue requests reliable Google Chat Space/group message handling so Space events create and process `googlechat:group` sessions instead of being silently dropped while DMs continue to work.

## Best Possible Solution

Google Chat inbound handling should use the current Google payload contract to distinguish DMs, group chats, and Spaces, add regression coverage for legacy `type`, modern `spaceType`, and `singleUserBotDm` payloads, and then either land the linked narrow fix or explicitly consolidate any broader thread metadata work under the related Google Chat routing issue.

## Reproduction Assessment

Yes. The issue has concrete end-to-end steps, and current main provides a source-level reproduction path: a Google Chat event whose `space` omits deprecated `type` and supplies modern `spaceType`/`singleUserBotDm` cannot be classified from those modern fields because the plugin never reads them.

## Solution Assessment

Unclear whether the already-open PR is the full best solution. Adding `spaceType` and `singleUserBotDm` support is the narrow maintainable first fix, but maintainers should decide whether thread/space metadata preservation belongs in that PR or remains separate in #69422.
