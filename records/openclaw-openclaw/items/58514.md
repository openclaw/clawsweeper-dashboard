## Label Justifications

- `P1`: The report describes a real Google Chat channel workflow where Space messages are received but not processed, blocking group collaboration for affected users.
- `impact:message-loss`: The issue is about Google Chat Space messages being silently dropped after webhook receipt instead of reaching the agent pipeline.
- `impact:session-state`: The expected `googlechat:group` session is not created for affected Space messages, so conversation state never starts.

## Summary

Keep open: current main and v2026.5.19 still use only legacy `space.type` for Google Chat direct/group classification, and an open pull request specifically closes this issue. The reporter also confirmed downstream relevance, so cleanup should wait for the classifier fix or an explicit maintainer consolidation with the related thread-routing work.

## What This Changes

Google Chat Space/group inbound messages should create and process `googlechat:group` sessions instead of returning HTTP 200 without agent routing while DMs continue to work.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/58993 or an equivalent classifier/type fix, then keep broader thread-target preservation coordinated through https://github.com/openclaw/openclaw/issues/69422 and https://github.com/openclaw/openclaw/pull/74235.

## Reproduction Assessment

Yes at source/reporter level: current main still lacks the modern Google Chat Space metadata fields and the linked PR/comment thread gives a concrete classifier gap plus live deployment confirmations. I did not run a live Google Workspace webhook replay for the exact HTTP 200/no-session symptom.

## Solution Assessment

Partially: the narrow classifier/type fix in https://github.com/openclaw/openclaw/pull/58993 is the right first repair for the modern metadata gap, while broader Space/thread preservation should stay coordinated with the separate thread-routing issue and PR.
