## Label Justifications

- `P1`: The issue reports a broken Google Chat Space workflow where accepted messages do not reach the agent for affected users.
- `impact:message-loss`: Space messages are reported as accepted by the webhook but silently produce no agent response.
- `impact:session-state`: The reported failure is specifically missing `googlechat:group` session creation while direct sessions are created.

## Summary

Keep open: current main has the merged modern `spaceType` classifier, but that PR explicitly scoped itself to DM/Space classification and left live Space/group session creation tracked here. Source inspection still shows a plausible current drop path for Space @mentions when `botUser` is not configured, matching older and current reports of HTTP 200 with no `googlechat:group` session.

## What This Changes

Google Chat Space/group messages should create and process `googlechat:group` sessions instead of returning HTTP 200 while silently dropping the message.

## Best Possible Solution

Keep this issue open as the canonical Space/group ingress bug until a live-verified Google Chat fix proves Space @mentions create and process `googlechat:group` sessions, or maintainers explicitly consolidate the remaining work into a narrower Google Chat routing issue.

## Reproduction Assessment

Yes at source level, but not by live run: current group activation can still drop Google Chat Space messages before session creation when mention detection does not recognize the bot user. The issue and related reports provide production evidence of HTTP 200 with no `googlechat:group` session.

## Solution Assessment

No, closing after the merged classifier PR is not the best resolution. That PR is the right narrow fix for modern DM/Space metadata classification, but the best remaining solution is a live-verified Space/group ingress fix or explicit consolidation with the broader Google Chat routing work.
