## Label Justifications

- `P1`: The report describes a real Google Chat channel workflow where Space messages are received but not processed, blocking group collaboration for affected users.
- `impact:message-loss`: Google Chat Space messages can be silently dropped after webhook receipt instead of reaching the agent pipeline.
- `impact:session-state`: The expected `googlechat:group` session is not created for affected Space messages, so conversation state never starts.

## Summary

Keep open: current main and the latest release still use only deprecated Google Chat `space.type` for direct/group classification, while an open PR specifically closes this issue. The reporter also confirmed downstream relevance after the first review, so cleanup should wait for that classifier fix or explicit consolidation with the related thread-routing work.

## What This Changes

Google Chat Space/group messages should reliably create `googlechat:group` sessions and be processed instead of returning HTTP 200 with no agent session or reply.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/58993 or an equivalent classifier/type fix, while keeping broader Google Chat thread-target preservation coordinated through https://github.com/openclaw/openclaw/issues/69422 and https://github.com/openclaw/openclaw/pull/74235.

Yes for the first repair: the narrow classifier/type fix in https://github.com/openclaw/openclaw/pull/58993 is the maintainable starting point for the modern metadata gap. Broader Space/thread preservation should stay coordinated with the separate thread-routing issue and PR.
