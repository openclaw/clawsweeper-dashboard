Keep open. Current main still has the reported MCP channel bridge retention pattern, and #71648 is an open unmerged PR that directly targets this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests bounding and clearing MCP channel bridge pending Claude permission and approval maps to prevent long-running `openclaw mcp serve` memory growth.

## Best Possible Solution

Review, rebase, land, or replace #71648 with an equivalent bridge lifecycle fix that clears both maps on close, prevents post-close writes, expires stale entries, and explicitly resolves the hard-cap question.

Yes. A high-confidence static reproduction exists on current main: add unique Claude permission requests without matching `yes/no` replies, or approval requested events without matching resolved frames, and the corresponding maps retain entries; `close()` also leaves both maps untouched.
