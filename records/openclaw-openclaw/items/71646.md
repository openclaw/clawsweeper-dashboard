## Summary

Keep open. Current main still has the reported MCP channel bridge retention pattern, and the same-author closing PR #71648 is still open and unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Bound MCP channel bridge pending Claude permission and approval state with TTL, close-clear, and cap behavior to avoid memory growth in long-running `openclaw mcp serve` processes.

## Best Possible Solution

Review, rebase, land, or replace #71648 with an equivalent MCP bridge fix that clears both pending maps on close, prevents post-close writes, expires stale Claude permission and approval entries, and either includes a cap or records the cap as an explicit accepted follow-up coordinated with #56420.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: add unique Claude permission requests without matching `yes/no` replies, or approval requested events without matching resolved frames, and the corresponding maps retain entries; `close()` also leaves both maps untouched.

## Solution Assessment

Yes for keeping this open. Because current main is still affected and #71648 is an open closing PR for the same report, the maintainable path is PR review, landing, or replacement rather than closing this issue before the fix reaches main.
