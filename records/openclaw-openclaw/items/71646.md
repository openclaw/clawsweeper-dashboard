## Summary

Keep open. Current main still stores both pending collections in unbounded maps and only removes entries on matching reply/resolved events; the same-author closing PR #71648 is open, so this should stay open until that lands or is replaced.

## What This Changes

Request bounds the `OpenClawChannelBridge` `pendingClaudePermissions` and `pendingApprovals` maps with TTL, close clearing, and caps to prevent long-running MCP serve memory growth.

## Best Possible Solution

Land or replace #71648 with an equivalent MCP bridge lifecycle fix that clears pending maps on close, prevents post-close writes, expires stale entries, and explicitly tracks any remaining cap follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: add unique Claude permission requests without matching replies, or approval requested events without matching resolved frames, and the corresponding maps retain entries; `close()` also leaves both maps untouched.

## Solution Assessment

Yes. Bounding the two pending maps inside `src/mcp/channel-bridge.ts` is the narrow maintainable fix, and #71648 is the current linked implementation candidate rather than a reason to close before merge.
