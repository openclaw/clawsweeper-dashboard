## Summary

Keep open. Current main still stores both MCP bridge pending collections in unbounded maps and removes them only on matching reply/resolved events, while the same-author closing PR #71648 remains open and unmerged.

## What This Changes

Add lifecycle bounds for `pendingClaudePermissions` and `pendingApprovals` in `src/mcp/channel-bridge.ts` so missed replies or dropped resolution events cannot grow memory without limit.

## Best Possible Solution

Land or replace #71648 with an equivalent MCP bridge lifecycle fix that clears the maps on close, prevents post-close writes, expires stale entries, includes focused regression tests, and adds the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: add unique Claude permission requests without matching `yes/no` replies, or approval requested events without matching resolved frames, and the corresponding maps retain entries; `close()` also leaves both maps untouched.

## Solution Assessment

Yes. Fixing the lifecycle inside `src/mcp/channel-bridge.ts` is the narrow maintainable path, and #71648 is the current implementation candidate rather than a reason to close before merge.
