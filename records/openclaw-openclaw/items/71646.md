## Summary

Keep open. Current main still has the reported MCP channel bridge retention pattern, and the same-author closing PR #71648 remains open and unmerged, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks to bound MCP channel bridge pending Claude permission and approval maps with TTL/cleanup, close-clear behavior, and a cap to prevent long-running `openclaw mcp serve` memory growth.

## Best Possible Solution

Land or replace #71648 with an equivalent narrow MCP bridge fix that clears both pending maps on close, prevents writes after close, expires stale Claude permission and approval entries, and explicitly resolves whether a separate cap is required.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: add unique Claude permission requests without matching replies, or approval requested events without matching resolved frames, and the corresponding maps retain entries; `close()` also leaves both maps untouched.

## Solution Assessment
