## Summary

Keep open. Current main still clears the TUI editor before any active-run/busy decision, production wiring does not provide a busy predicate, and the normal send path still reaches `client.sendChat` without rejecting an existing active run; the direct fix PR closed unmerged and the merged adjacent pending-send PR did not change this submit boundary.

## What This Changes

The issue requests that TUI input submitted while an agent run is active not be silently swallowed or replayed as stale context.

## Best Possible Solution

Add a TUI-local busy/queue-aware guard before clearing the editor, preserve the draft when normal chat is not accepted, show a short queued/steered or Esc-to-abort status, and add focused regression coverage.

## Reproduction Assessment

Yes, at source level. Submitting normal text through `createEditorSubmitHandler` while `state.activeChatRunId` is non-null still clears the editor first and can continue into the normal `sendMessage` path without an active-run rejection.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before `editor.setText("")`; a guard only inside `sendMessage` would still lose the visible draft, and changing gateway queue semantics would be broader than needed.
