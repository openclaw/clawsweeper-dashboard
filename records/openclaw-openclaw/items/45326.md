## Summary

Keep open. Current main still clears normal TUI input before any active-run guard, wires the submit helper without a busy predicate, and lets normal sends reach `chat.send` while another run may be active; the direct fix PR was closed unmerged and the later merged pending-send work did not cover this submit boundary.

## What This Changes

Prevent normal TUI input entered while a model run is active from being cleared, swallowed, or later submitted as stale context.

## Best Possible Solution

Add a TUI-local pre-clear guard that blocks normal chat submits while an active run or pending optimistic send exists, preserves the draft, shows a short busy/Esc-to-abort hint, and locks it with focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit non-command text through `createEditorSubmitHandler` while `state.activeChatRunId` is non-null: current production wiring supplies no busy predicate, the handler clears the editor, and `sendMessage` still sends normal chat.

## Solution Assessment

Yes. Guarding at the submit boundary before `editor.setText("")` is the narrowest maintainable fix; guarding only inside `sendMessage` would still lose the draft, while queuing or making Enter an interrupt would be broader product behavior.
