## Summary

Keep open. Current main still clears TUI submissions before any active-run guard, production wiring passes no busy predicate into the submit handler, and normal chat sends can still reach `chat.send` while a run is active; the directly linked fix PR was closed unmerged, and the merged adjacent PR covered pending-send reconciliation rather than this submit boundary.

## What This Changes

The issue asks for TUI input typed or pasted during active generation to be interrupted, rejected, or preserved rather than silently swallowed and later sent as stale prompt text.

## Best Possible Solution

Add a TUI-local submit guard before `editor.setText("")`: block ordinary chat submits while an active run or pending optimistic send exists, preserve the draft, and show a concise busy/Esc-to-abort hint while keeping intended command paths such as `/abort` available.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text through `createEditorSubmitHandler` while `state.activeChatRunId` is non-null: current production wiring cannot block before clearing, and the normal send path still calls `client.sendChat`.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before clearing the editor; guarding only inside `sendMessage` would still lose the draft, while gateway queueing or Enter-as-interrupt semantics would be broader product work.
