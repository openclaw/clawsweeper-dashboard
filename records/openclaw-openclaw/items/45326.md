## Summary

Keep open. Current main still clears normal TUI submissions before any active-run guard and the normal send path still reaches `client.sendChat` while `state.activeChatRunId` can already be set; the direct fix PR #45586 was closed unmerged, and merged #59800 covered adjacent pending-send reconciliation rather than this submit boundary.

## What This Changes

The issue requests that the TUI not swallow or replay input submitted while a model run is active.

## Best Possible Solution

Add a TUI-local submit guard before `editor.setText("")`: block normal chat while an active run or pending optimistic send exists, preserve the draft, show a short busy/Esc-to-abort hint, and add focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text through the production `createEditorSubmitHandler` path while `state.activeChatRunId` is non-null: the editor is cleared first and `sendMessage` still proceeds to `client.sendChat`.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before clearing the draft; guarding only inside `sendMessage` would still lose the typed text, and gateway queue semantics would be a broader product change.
