## Summary

Keep open. Current main still clears normal TUI submissions before any active-run guard and the normal send path still reaches `client.sendChat` while an active run can already exist; the direct fix PR was closed unmerged, and merged adjacent work only covered pending-send reconciliation and busy-state visibility.

## What This Changes

The issue requests that the TUI stop accepting and losing normal chat input submitted while a model run is active, either by interrupting or by rejecting/preserving the draft.

## Best Possible Solution

Add a TUI-local guard before clearing the editor: block normal chat when an active run or pending optimistic send exists, keep the draft visible, show a short busy/Esc-to-abort hint, and add focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text through the production `createEditorSubmitHandler` path while `state.activeChatRunId` is non-null: the editor is cleared first and `sendMessage` still proceeds to `client.sendChat`.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before `editor.setText("")`; guarding only inside `sendMessage` would still lose the draft, and changing gateway queue semantics would be broader than needed.
