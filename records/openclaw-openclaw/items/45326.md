## Summary

Keep open. Current main still clears the TUI editor before any active-run guard, wires the submit handler without a busy predicate, and lets normal sends reach `chat.send` while `state.activeChatRunId` may already be set; the direct fix PR was closed unmerged and the later merged pending-send work did not cover this submit boundary.

## What This Changes

The issue asks the TUI to handle normal text submitted during active generation without swallowing it or replaying it as stale input on a later turn.

## Best Possible Solution

Add a TUI-local normal-submit guard before `editor.setText("")`: when an active run or pending optimistic send exists, preserve the draft, skip `sendMessage`, and show a short busy/Esc-to-abort hint while keeping intended slash-command and bang-line behavior explicit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text through `createEditorSubmitHandler` while the TUI state has an active run: the handler clears the editor and calls `sendMessage`, whose current path has no non-`/btw` active-run rejection.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before editor clearing; guarding only inside `sendMessage` would still lose the draft, while gateway queueing or Enter-as-interrupt semantics would be broader product changes.
