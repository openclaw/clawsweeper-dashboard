## Summary

Keep open. Current main still clears normal TUI submissions before any active-run guard, production wiring passes no busy predicate into the submit handler, and `sendMessage` still sends normal chat without rejecting an existing `activeChatRunId`. The direct fix PR #45586 was closed unmerged, while merged adjacent PR #59800 improved pending-send reconciliation but did not cover this submit boundary.

## What This Changes

The issue requests TUI handling that interrupts, rejects, or preserves input typed during model generation so stale text is not swallowed and replayed on the next turn.

## Best Possible Solution

Add a TUI-local submit guard before the editor is cleared: block normal chat messages while an active run exists, preserve the typed draft, and show a short busy/Esc-to-abort hint. Keep slash commands and bang-lines available during active runs, and leave broader gateway queue or interrupt-and-replace semantics to a separate product decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text through `createEditorSubmitHandler` while `state.activeChatRunId` is non-null: the handler clears the editor and routes to `sendMessage`, and the send path has no active-run rejection. I did not run live TUI tests because this review is explicitly read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before `editor.setText("")`; guarding only inside `sendMessage` would still lose the draft, while changing gateway concurrency or overloading Enter as interrupt would be broader than the reported behavior bug.
