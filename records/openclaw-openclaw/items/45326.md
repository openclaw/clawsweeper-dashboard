## Summary

Keep open. Current main still clears the TUI editor before any busy-run guard and still routes normal chat submissions to `sendMessage`, whose normal path reaches `client.sendChat` without rejecting an existing active run; the direct fix proposal at https://github.com/openclaw/openclaw/pull/45586 closed unmerged, while https://github.com/openclaw/openclaw/pull/59800 covered adjacent pending-send reconciliation rather than this submit-time guard.

## What This Changes

The issue asks the TUI to avoid swallowing or replaying normal input typed or pasted while a model response is active.

## Best Possible Solution

Add a TUI-local busy guard before clearing the editor: block normal chat when an active run or pending optimistic send exists, keep the draft visible, show a short busy/Esc-to-abort hint, and add focused regression coverage.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: submit normal text through the production `createEditorSubmitHandler` while `state.activeChatRunId` is non-null. The editor clears first and the normal `sendMessage` path still reaches `client.sendChat` without an active-run rejection.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before `editor.setText("")`; a guard only inside `sendMessage` would still lose the visible draft, and changing gateway queue semantics would be broader than needed.
