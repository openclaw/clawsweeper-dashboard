## Summary

Keep open. Current main still clears TUI submissions before any active-run guard, production wiring passes no busy predicate into the submit handler, and normal sends still reach `chat.send`; #45586 was closed unmerged, while merged #59800 covered pending-send reconciliation rather than this submit boundary.

## What This Changes

The issue asks the TUI to prevent normal text typed or pasted during active model generation from being swallowed and later replayed as a stale prompt.

## Best Possible Solution

Add a TUI-local submit guard before `editor.setText("")`: block normal chat while an active run or pending optimistic send exists, preserve the draft, and show a short busy/Esc-to-abort hint while keeping intended command paths available.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text through `createEditorSubmitHandler` while `state.activeChatRunId` is non-null: current production wiring cannot block before clearing, and the normal send path still calls `client.sendChat`.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before clearing the editor; guarding only inside `sendMessage` would still lose the draft, while gateway queueing or Enter-as-interrupt semantics would be broader product changes.
