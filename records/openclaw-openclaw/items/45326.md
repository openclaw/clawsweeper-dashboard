## Summary

Keep open. Current main still clears the TUI draft before any active-run/busy decision, production wiring does not pass a busy predicate into the submit helper, and normal chat send still reaches `client.sendChat` without rejecting an existing active run; the direct fix PR was closed unmerged and the adjacent pending-send fix did not touch this submit boundary.

## What This Changes

The issue reports that normal TUI input submitted during active model generation is cleared/swallowed and can later be sent as stale input.

## Best Possible Solution

Add a TUI-local busy-aware guard before clearing the editor, preserve the draft when normal chat is rejected, keep explicit abort on Esc, and cover active/pending-run submits with focused regression tests.

## Reproduction Assessment

Yes, at source level. Submitting normal text through `createEditorSubmitHandler` while a run is active can clear the editor before rejection, and the current `sendMessage` path still lacks an active-run rejection for normal chat.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before `editor.setText("")`; a guard only inside `sendMessage` would still lose the visible draft, and changing gateway queue semantics would be broader than needed.
