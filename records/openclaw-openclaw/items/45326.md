## Summary

Keep open. Current main still clears normal TUI submissions before any active-run guard, production wiring passes no busy predicate into the submit handler, and normal chat sends still reach `chat.send` while another run can be active; the direct fix PR was closed unmerged and the later merged pending-send work did not cover this submit boundary.

## What This Changes

The issue reports that normal TUI text entered during active model generation is cleared or hidden and can later be sent as stale input on a following turn.

## Best Possible Solution

Add a TUI-local submit guard before the editor is cleared: block normal chat while an active run or pending optimistic send exists, preserve the draft, and show a short busy/Esc-to-abort hint while keeping slash commands and bang-lines available.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text through `createEditorSubmitHandler` while `state.activeChatRunId` is non-null: the handler clears the editor and routes to `sendMessage`, and the send path has no active-run rejection.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before `editor.setText("")`; guarding only inside `sendMessage` would still lose the draft, while gateway queueing or Enter-as-interrupt semantics would be broader product changes.
