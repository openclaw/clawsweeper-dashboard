## Summary

Keep open: current main still shows a source-level gap where `/usage` stores the footer mode but the queued followup delivery path used by channels does not append the footer, and open closing PRs are candidate fixes rather than proof that the issue is resolved.

## What This Changes

The issue reports that after upgrading to OpenClaw 2026.6.8, Telegram `/usage full` no longer causes usage footers to appear on messages.

## Best Possible Solution

Land or reject a narrowed followup-runner fix that mirrors `agent-runner` usage-footer behavior, adds focused regression coverage, and includes redacted Telegram proof before closing this issue.

## Reproduction Assessment

Yes, at source level: `/usage` persists `responseUsage`, `agent-runner` appends the footer from that setting, and current `followup-runner` sends channel payloads without reading that setting. I did not run a live Docker/Telegram/OpenAI Codex reproduction.

## Solution Assessment

Yes, keeping this issue open is the best cleanup outcome until a linked candidate PR lands or maintainers reject the fix shape. Queuing a new repair PR would duplicate active linked work.
