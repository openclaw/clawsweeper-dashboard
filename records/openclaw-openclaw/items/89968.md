## Summary

Keep open: the PR is aimed at the right shared config metadata collector, but the live head is draft/conflicting and the channel-row isolation can still leave partial metadata from a row that throws after earlier writes.

## What This Changes

The PR wraps plugin/channel schema metadata collection per plugin manifest record and adds focused regressions for poisoned plugin-level and channel-level metadata rows.

## Best Possible Solution

Keep the collector-level isolation, but make channel metadata commits atomic per record, resolve the current main conflict, and rerun focused plus changed-gate validation before merge.

## Reproduction Assessment

Yes by source inspection: current main reads plugin record fields directly inside the metadata collector loops, so a throwing/unreadable field can abort collection. I did not run the regression in this read-only review, but the PR body also reports red/green focused test proof.

## Solution Assessment

No, not yet: collector-level isolation is the right layer, but the channel collector should stage or roll back per-record writes so a late throw cannot leave partial metadata behind. The branch also needs conflict resolution and broad validation before it is merge-ready.
