## Best Possible Solution

Keep the feature in the Feishu plugin, rebase it onto current main, and revise it so status/last-activity lives in the card note, final replies are never silently filtered, heartbeat cleanup is idempotent on every dispatcher teardown path, completion pings use the pre-close duration, and tests/docs/changelog cover the new public behavior.

## Reproduction Assessment

Yes. The PR bugs have high-confidence source-level reproduction paths: deliver an openai-codex final English reply that matches the heuristic, clean up a dispatcher before idle/error after startStreaming, and follow the long-final path where closeStreaming resets streamingStartedAt before the completion-ping check.

## Solution Assessment

No. The feature direction is appropriate for the Feishu plugin, but this implementation is not yet the best solution because it relies on broad final-message content heuristics and has lifecycle ordering bugs; gating scratchpad suppression to non-final/internal chunks and making cleanup/duration handling explicit is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Do not filter final replies as scratchpad text:** `extensions/feishu/src/reply-dispatcher.ts:519-520`
  - body: `hasText` is cleared whenever the broad scratchpad heuristic matches, before checking `info.kind`. For openai-codex sessions this can match normal final English answers such as requests containing "could" or "need", causing the final reply to return without sending anything. Restrict the filter to non-final/internal chunks or require a stronger scratchpad signal.
  - confidence: 0.91
- **[P2] Stop the heartbeat from cleanup paths:** `extensions/feishu/src/reply-dispatcher.ts:434`
  - body: `startHeartbeat()` creates an interval, but `onCleanup` does not call `stopHeartbeat()` or `closeStreaming()`. If the dispatcher is torn down before `onIdle` or `onError`, the interval survives and keeps the dispatcher state reachable. Stop the heartbeat from cleanup as well as closeout.
  - confidence: 0.87
- **[P2] Check completion duration before closing the stream:** `extensions/feishu/src/reply-dispatcher.ts:563`
  - body: The final streaming path calls `closeStreaming()` before `shouldSendCompletionPing()`, and `closeStreaming()` resets `streamingStartedAt` to zero. That makes the completion ping condition false even for long tasks, so the advertised completion confirmation never fires. Capture the duration or ping decision before resetting stream state.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `extensions/feishu/src/config-schema.ts:183-184`
  - body: This PR adds user-facing Feishu streaming status and filtering behavior but does not update `CHANGELOG.md`. Repo policy requires user-facing feature/fix/perf changes to add an active-version changelog entry with appropriate contributor credit.
  - confidence: 0.78
