## Summary

Keep open: the PR targets a plausible provider timeout problem, but the current patch is not mergeable because its global 120s fallback bypasses the existing local-provider no-timeout exemption, and the contributor has not provided after-fix real behavior proof for the claimed DeepSeek/Gateway hang.

## What This Changes

The PR changes provider model resolution so missing or invalid `models.providers.<id>.timeoutSeconds` attaches a default 120000ms request timeout.

## Best Possible Solution

Preserve provider-scoped timeout semantics and local-provider no-timeout behavior, then add a targeted bounded cloud/request timeout fix with focused regression coverage and redacted real setup proof.

## Reproduction Assessment

Yes for the introduced regression: source inspection shows the PR fallback becomes `modelRequestTimeoutMs`, and the idle resolver consumes that before the local-provider exemption. No live reproduction proves the claimed DeepSeek/Gateway indefinite hang after the fix.

## Solution Assessment

No. A global fallback inside `resolveProviderRequestTimeoutMs` is too broad because it changes local-provider behavior; the safer fix should bound the unconfigured cloud/request path without overriding local opt-out semantics.
