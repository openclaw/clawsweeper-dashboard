## Summary

Keep open: the branch is a plausible timeout fix attempt, but its fallback is too broad and would override current local-provider no-default-timeout behavior; it also still lacks real after-fix runtime proof.

## What This Changes

The PR changes provider model timeout resolution so missing or invalid `models.providers.<id>.timeoutSeconds` becomes a default 120000 ms request timeout.

## Best Possible Solution

Keep remote/provider HTTP requests bounded without turning missing provider timeout config into explicit model timeout metadata for local providers, then add focused regression coverage and contributor-provided runtime proof.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the new fallback becomes `modelRequestTimeoutMs` and is consumed before the local-provider exemption. No for the claimed production hang fix: no live DeepSeek/Gateway proof was provided or run in this read-only review.

## Solution Assessment

No. A global fallback inside `resolveProviderRequestTimeoutMs` is too broad; the safer fix should bound the unconfigured remote/provider request path while preserving local-provider timeout semantics.

## Review Findings
