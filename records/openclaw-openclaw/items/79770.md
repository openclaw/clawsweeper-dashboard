## Summary

Keep open: the patch is not ready because the new fallback would regress the existing local-provider timeout exemption, and the external PR still lacks after-fix real behavior proof for the claimed Gateway/DeepSeek hang.

## What This Changes

The PR changes provider model resolution so missing or invalid `models.providers.<id>.timeoutSeconds` returns a new 120,000 ms default request timeout.

## Best Possible Solution

Preserve the provider-scoped timeout contract and local-provider no-watchdog behavior, add focused regression coverage, and require redacted real setup proof for the reported DeepSeek/Gateway hang.

## Reproduction Assessment

Yes for the blocking PR regression: source inspection shows the new fallback becomes `modelRequestTimeoutMs` and is consumed before local-provider baseUrl detection. The underlying DeepSeek production hang is not live-reproduced here.

## Solution Assessment

No. A global fallback inside `resolveProviderRequestTimeoutMs` is too broad because it changes local-provider semantics; the safer fix should bound the unbounded remote/provider request path without overriding local-provider opt-out behavior.
