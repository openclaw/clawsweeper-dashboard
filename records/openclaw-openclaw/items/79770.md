## What This Changes

Adds a default 120000 ms provider request timeout when `models.providers.<id>.timeoutSeconds` is absent, invalid, or non-positive.

## Best Possible Solution

Keep the released stream/transport timeout fix and leave provider `timeoutSeconds` as an explicit override so local and self-hosted providers can still opt out of the default watchdog.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the fallback would set `modelRequestTimeoutMs` before the local-provider exemption. The original DeepSeek hang was not live-replayed, but current main now bounds the same stream setup/request path through the shipped idle-timeout wrapper.

## Solution Assessment

No. The resolver-level fallback is broader than the maintained contract; the current main fix bounds cloud/provider setup through the stream timeout path while preserving local provider opt-out behavior.
