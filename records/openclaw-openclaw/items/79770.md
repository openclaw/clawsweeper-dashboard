## Summary

Keep open. The PR targets a plausible unbounded-provider-timeout problem, but the proposed blanket fallback would regress current local-provider timeout behavior and the contributor has not provided real after-fix Gateway/provider proof.

## What This Changes

The PR changes provider model resolution so missing or invalid `models.providers.<id>.timeoutSeconds` returns a hard-coded 120000 ms request timeout instead of leaving `requestTimeoutMs` unset.

## Best Possible Solution

Bound the slow cloud/provider HTTP request path without assigning an implicit provider timeout to local/self-hosted models that rely on the existing no-default-idle-watchdog policy, then add focused regression coverage and real setup proof.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows a default `requestTimeoutMs` would be consumed before the current local-provider exemption. The original DeepSeek hang is not reproduced from the PR evidence because only a typecheck is reported.

## Solution Assessment

No. A global fallback in `resolveProviderRequestTimeoutMs` is broader than the existing timeout contract; the safer fix should bound the cloud/provider request path while preserving local-provider opt-out behavior.
