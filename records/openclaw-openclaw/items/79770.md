## Summary

Keep open: this PR is still missing real behavior proof and the proposed global fallback changes established local-provider timeout semantics by assigning `requestTimeoutMs` before the local no-default-watchdog branch can apply.

## What This Changes

The PR changes `src/agents/pi-embedded-runner/model.ts` so missing, invalid, or non-positive provider `timeoutSeconds` resolves to a hard-coded 120000 ms request timeout.

## Best Possible Solution

Bound the slow cloud/provider HTTP path while preserving the existing local/self-hosted no-default-watchdog behavior, then add focused regression coverage and real setup proof.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the proposed default `requestTimeoutMs` would be consumed before the local-provider exemption. No for the original DeepSeek hang: the PR provides only a typecheck, and current normal stream wrapping already supplies an abort signal and bounded request timeout through the idle watchdog path.

## Solution Assessment

No. A global fallback in `resolveProviderRequestTimeoutMs` is broader than the current timeout contract; the safer solution should target cloud/provider hangs without imposing a default timeout on local providers that intentionally opt out unless explicitly configured.
