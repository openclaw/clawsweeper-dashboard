## Summary

Keep open: the PR targets a plausible timeout gap, but the patch changes configured local providers without `timeoutSeconds` from the current no-default-idle-watchdog behavior to a 120s cutoff, and it still lacks real after-fix Gateway/provider proof.

## What This Changes

The PR changes provider model resolution so missing or invalid `models.providers.<id>.timeoutSeconds` becomes a 120s `requestTimeoutMs` fallback.

## Best Possible Solution

Preserve the provider-scoped timeout contract, add a bounded default only where cloud/model HTTP requests are otherwise unbounded, keep local-provider opt-out behavior intact, and cover the split with focused tests plus redacted real setup proof.

## Reproduction Assessment

Yes for the introduced regression: source inspection of the PR diff and current main shows the fallback becomes `modelRequestTimeoutMs` and is consumed before the local-provider exemption. No live reproduction proves the claimed DeepSeek hang after the fix.

## Solution Assessment

No. A global fallback in `resolveProviderRequestTimeoutMs` is too broad because it overrides existing local-provider timeout semantics; the safer fix should target only the unbounded cloud/request path or carry implicit defaults separately from explicit provider timeouts.
