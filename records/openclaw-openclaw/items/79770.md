## Summary

Keep open: the PR targets a plausible provider timeout gap, but the broad fallback would make configured local providers without `timeoutSeconds` carry a 120s request timeout and bypass the current local-provider idle-watchdog exemption; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds a 120 second default request timeout fallback when provider `timeoutSeconds` is missing or invalid.

## Best Possible Solution

Preserve the provider-scoped timeout contract while keeping local-provider no-timeout behavior unchanged, add focused regression coverage, and require redacted real setup proof for the reported provider hang.

## Reproduction Assessment

Yes for the introduced regression: source inspection shows the new fallback becomes `modelRequestTimeoutMs` and is consumed before the current local-provider exemption. No live reproduction proves the claimed DeepSeek indefinite hang after the fix.

## Solution Assessment

No. A global 120s fallback inside `resolveProviderRequestTimeoutMs` is too broad because it changes local-provider semantics; the safer fix should target the unbounded cloud/request path without overriding local-provider opt-out behavior.
