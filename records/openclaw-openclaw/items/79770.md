## Summary

Keep open, but this PR should not merge as-is: the global 120s fallback would assign `requestTimeoutMs` before the current local/self-hosted provider exemption can apply, and the PR still lacks after-fix real behavior proof for the claimed DeepSeek/Gateway hang.

## What This Changes

The PR changes provider timeout resolution so absent, invalid, or non-positive `models.providers.<id>.timeoutSeconds` returns a default 120000 ms request timeout.

## Best Possible Solution

Bound the intended slow remote/provider HTTP path without assigning implicit `requestTimeoutMs` to local/self-hosted providers, then add focused regression tests and redacted real setup proof.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the new fallback would set `modelRequestTimeoutMs` before `resolveLlmIdleTimeoutMs` reaches the local-provider exemption. The original DeepSeek hang is not high-confidence reproduced because the PR body only reports a typecheck.

## Solution Assessment

No. A global fallback inside `resolveProviderRequestTimeoutMs` is broader than the current timeout contract; the safer fix should preserve local-provider opt-out behavior while bounding the intended cloud/provider HTTP path.
