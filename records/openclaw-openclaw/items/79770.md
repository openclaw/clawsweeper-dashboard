## Summary

Keep open: the PR targets a plausible unbounded provider request path, but the proposed global 120s fallback regresses current local-provider timeout behavior and still lacks after-fix real behavior proof.

## What This Changes

The PR changes provider model resolution to return a 120000 ms request timeout when `models.providers.<id>.timeoutSeconds` is missing or invalid.

## Best Possible Solution

Preserve provider-scoped timeout semantics and local-provider no-default-watchdog behavior while adding a bounded cloud/request-timeout fix in the narrow path that can hang, with focused regression coverage and redacted real setup proof.

## Reproduction Assessment

Yes for the introduced regression: source inspection shows the new fallback becomes `modelRequestTimeoutMs` and is consumed before the local-provider default-watchdog exemption. The claimed DeepSeek/Gateway hang after the fix has not been reproduced with live proof.

## Solution Assessment

No. A global fallback inside `resolveProviderRequestTimeoutMs` is too broad because it changes configured local-provider semantics; the safer solution should bound the problematic cloud/request path without overriding local-provider opt-out behavior.
