## Summary

Keep open: this PR is a plausible timeout bug-fix attempt, but the patch globally assigns a 120s provider request timeout when no timeout is configured, which regresses the current local-provider idle-watchdog policy, and it still lacks real behavior proof for the claimed DeepSeek/Gateway incident.

## What This Changes

The PR changes provider timeout resolution so missing, invalid, or non-positive `models.providers.<id>.timeoutSeconds` returns a default 120000 ms request timeout.

## Best Possible Solution

Bound slow remote provider HTTP calls without turning local/self-hosted provider requests into implicit 120s idle-timeout failures, then add focused regression coverage and real setup proof.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the new default would be treated as an explicit model timeout before the local-provider exemption. The original DeepSeek hang is not reproduced because the PR evidence only reports a typecheck.

## Solution Assessment

No. A global fallback in `resolveProviderRequestTimeoutMs` is broader than the current timeout contract; the safer solution should target remote HTTP provider hangs while preserving local-provider opt-out semantics.

## Review Findings
