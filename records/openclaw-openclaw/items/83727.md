## Label Justifications

- `P1`: The issue blocks the official Twitch channel connection path for users with otherwise valid credentials.
- `impact:auth-provider`: The failure is caused by Twurple auth-provider intent registration for chat token lookup.
- `impact:crash-loop`: The reported gateway repeatedly exits and retries the Twitch channel after the auth failure.
- `impact:message-loss`: Because the bot never establishes a stable Twitch chat connection, Twitch inbound and outbound messages cannot be delivered.

## Summary

Keep open: current main and latest shipped release still omit Twurple's chat intent registration for refreshing Twitch auth, and the linked closing fix PR is open but unmerged.

## What This Changes

The issue reports that @openclaw/twitch fails to establish a stable Twitch chat connection because Twurple cannot find a registered `chat` intent for the configured auth provider.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/83750; once the fix merges, close this issue as implemented by that PR.

## Reproduction Assessment

Yes, at source level: current main calls `addUserForToken` without intents, and Twurple v8.1.4 `ChatClient` always queries the `chat` intent before throwing the exact reported error. I did not run a live Twitch credential reproduction.

## Solution Assessment

Yes: the narrow maintainable fix is to register `['chat']` when adding the refreshing token and add a regression assertion for that call. The open linked PR appears to follow that shape, but it is not merged yet.
