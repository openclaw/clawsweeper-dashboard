## Summary

Keep open: current main and the latest stable release still omit Twurple chat intent registration in the Twitch refreshing-token path, and the open fixing PR that closes this issue has not merged yet. ## What This Changes

The issue reports that the Twitch plugin cannot establish stable chat because Twurple cannot fetch a token for the `chat` intent, then adds that replies to `twitch:channel:*` fail after the auth fix.

## Best Possible Solution

Land or close the focused Twitch chat-intent PR first, then split or fix the provider-prefixed Twitch target normalization path if the reply-routing symptom still reproduces.

## Reproduction Assessment

Yes, at source level: current main calls `addUserForToken` without Twurple's documented `chat` intent, and the reporter provided matching runtime logs. I did not run a live Twitch credential reproduction in this read-only review.

## Solution Assessment

Yes for the primary auth bug: passing `['chat']` at the existing refreshing-token registration call is the narrow owner-boundary fix. The later `twitch:channel:*` delivery symptom should be handled as a separate focused Twitch target-normalization fix if it remains after the auth PR lands.
