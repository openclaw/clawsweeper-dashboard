## Summary

Keep this PR open, but do not merge it as-is. Current main still has the hard `sessions_send` wait-timeout behavior, so the contribution targets a real gap, but the patch conflates soft wait-window expiry with terminal aborted timeouts and can route stale assistant history through delayed announce delivery.

## What This Changes

The branch changes `sessions_send` so an initial wait timeout returns accepted/pending delivery and starts A2A announce delivery, adds a bounded-follow-up comment in the A2A flow, and adds delayed-timeout regression tests.

## Best Possible Solution

Revise the fix so `sessions_send` preserves hard timeout/failure semantics for terminal aborted runs, treats only proven in-flight wait-window expiry as accepted pending delivery, passes baseline protection into delayed A2A reply reads, and covers delayed reply, no-new-reply, second-wait-timeout, and terminal-timeout cases in focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Current main is reproducible with a mocked `sessions_send` path where `agent` returns a run id and the first nonzero-timeout `agent.wait` returns `status: "timeout"`; the tool returns a hard timeout at `src/agents/tools/sessions-send-tool.ts:379`. The PR regressions are reproducible with mocks for a terminal aborted timeout and for a timed-out run that completes without a new post-baseline assistant reply.

## Solution Assessment

No. The direction is reasonable for soft wait-window expiry, but this implementation is not the safest fix because it converts every timeout to accepted pending delivery and drops the baseline used to avoid stale replies. A safer implementation should classify timeout source explicitly and thread a pre-run baseline through the delayed announce path.

Full review comments:

- **[P1] Do not mark terminal timeouts as accepted:** `src/agents/tools/sessions-send-tool.ts:380-383`
  - body: `agent.wait` can return `status: "timeout"` for an already-ended aborted run as well as for caller wait-window expiry. This branch converts both cases into `accepted` pending delivery, so a terminal timed-out run loses its failure signal and callers can wait for an announce that will never arrive.
  - confidence: 0.9
- **[P1] Pass the baseline into delayed A2A reads:** `src/agents/tools/sessions-send-tool.ts:380`
  - body: The normal wait path captures `baselineReply` so old assistant history is not reused as the new run's reply, but the timeout branch calls `startA2AFlow(undefined, runId)`. The delayed A2A flow then reads the latest assistant reply without that baseline, so a timed-out run that finishes without new assistant text can announce stale history.
  - confidence: 0.89
