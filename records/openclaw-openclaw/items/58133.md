## Summary

Keep open. The PR addresses a real current-main `sessions_send` timeout problem, but the proposed branch should not merge as-is because it treats all wait timeouts as accepted pending delivery, drops the baseline guard for delayed A2A reads, and still needs the required changelog entry; there is also an open competing fix in #68196 that maintainers should consolidate or choose between.

## What This Changes

The PR changes `sessions_send` to start delayed A2A announce delivery and return accepted/pending delivery after an initial wait timeout, adds a bounded-follow-up comment, and adds delayed-timeout regression tests.

## Best Possible Solution

Consolidate the timeout fix around a contract that distinguishes soft wait-window expiry from terminal run timeout, preserves baseline protection when a delayed A2A read is needed, keeps terminal failures visible to callers, and adds focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Current main can be reproduced with a mocked `sessions_send` path where `agent.wait` returns `status: "timeout"` after the target run is accepted, causing an immediate hard timeout; the PR regressions are reproducible with mocked terminal-aborted timeout and stale-history/no-new-reply cases.

## Solution Assessment

No. The direction is reasonable for soft wait-window expiry, but this implementation is not the best fix because it converts ambiguous timeout results into accepted delivery and bypasses the baseline guard; a safer solution should classify timeout source explicitly or use a compensation path that keeps stale replies out.

Full review comments:

- **[P1] Do not mark terminal timeouts as accepted:** `src/agents/tools/sessions-send-tool.ts:380-383`
  - body: `agent.wait` can return `status: "timeout"` both when the caller's wait window expires and when a run has already ended as aborted/timed out. This branch converts both cases into `accepted` pending delivery, so terminal failures lose their failure signal and callers can wait for an announce that will never arrive.
  - confidence: 0.9
- **[P1] Pass the baseline into delayed A2A reads:** `src/agents/tools/sessions-send-tool.ts:380`
  - body: The normal wait path captures `baselineReply` so old assistant history is not reused as the new run's reply, but the timeout branch calls `startA2AFlow(undefined, runId)`. The delayed A2A path then reads the latest assistant reply without that baseline, so a timed-out run that finishes without new assistant text can announce stale history.
  - confidence: 0.89

  - body: This changes the user-visible `sessions_send` status and delivery contract, but the PR does not update `CHANGELOG.md`. Add a single-line active-version entry with appropriate contributor credit before merge.
