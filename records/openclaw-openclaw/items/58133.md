## Summary

Keep open. This PR targets a real current-main `sessions_send` timeout-delivery gap, but it should not merge as-is because it turns ambiguous terminal timeouts into accepted delivery, drops baseline protection for delayed A2A reply reads, and overlaps the competing fix in #68196.

## What This Changes

The branch changes `sessions_send` to return accepted/pending delivery and start delayed A2A announce delivery after an initial wait timeout, with bounded-follow-up comments and regression tests.

## Best Possible Solution

Land one consolidated fix that distinguishes soft wait-window expiry from terminal run timeout, preserves the baseline reply guard, keeps delayed announce delivery alive only for proven in-flight runs, and adds the required changelog entry.

## Reproduction Assessment

Yes. Current main is reproducible with a mocked `sessions_send` path where `agent` returns a run id and the first nonzero-timeout `agent.wait` returns `status: "timeout"`; the tool returns a hard timeout before any delayed announce path. The PR regressions are reproducible with mocked terminal-aborted timeout and stale-history/no-new-reply cases.

## Solution Assessment

No. The high-level direction is valid for soft wait-window expiry, but this implementation is not the safest fix because it treats all timeout results as accepted and reads delayed replies without the pre-run baseline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not mark terminal timeouts as accepted:** `src/agents/tools/sessions-send-tool.ts:380-383`
  - body: `agent.wait` can return `status: "timeout"` for both caller wait-window expiry and an already-ended aborted run. This branch converts both cases into accepted pending delivery, so terminal failures lose their failure signal and callers can wait for an announce that will never arrive.
  - confidence: 0.9
- **[P1] Pass the baseline into delayed A2A reads:** `src/agents/tools/sessions-send-tool.ts:380`
  - body: The normal wait path captures `baselineReply` so old assistant history is not reused as the new run's reply, but the timeout branch calls `startA2AFlow(undefined, runId)`. The delayed A2A flow then reads the latest assistant reply without that baseline and can announce stale history.
  - confidence: 0.89
- **[P3] Add the required changelog entry:** `src/agents/tools/sessions-send-tool.ts:383`
  - body: This changes the user-visible `sessions_send` status and delivery contract, but the PR does not update `CHANGELOG.md`. Add a single-line active-version entry with appropriate human contributor credit before merge.
  - confidence: 0.86
