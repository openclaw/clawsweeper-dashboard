## Summary

Keep open: current main still appears to have the reported scoped-dispatch gap for embedded local agent runs. The `--local` CLI path enters `agentCommand` directly, subagent completion delivery now calls the in-process Gateway dispatcher, and that dispatcher still throws when neither a Gateway request scope nor fallback context is present.

## What This Changes

The issue reports that `openclaw agent --local` loses subagent completion announces after the in-process announce handoff change.

## Best Possible Solution

Install and clear a valid Gateway fallback or request context for embedded local agent runs, then add focused regression coverage for subagent completion announce delivery through `openclaw agent --local`.

## Reproduction Assessment

Yes at source level: the reported `--local` path calls `agentCommand` directly, and completion announce delivery calls `dispatchGatewayMethodInProcess("agent")`, which throws when no request scope or fallback context exists. I did not run a model-backed live CLI repro in this read-only review.

## Solution Assessment

Yes, a narrow fix at the embedded local-agent boundary is the most maintainable direction. Prefer installing a scoped or fallback Gateway context for `--local` runs with regression coverage over reverting the in-process dispatcher wholesale.
