## Summary

Keep open: current main still has a source-reproducible `--local` announce failure path. The in-process announce dispatcher requires a gateway request scope or fallback context, while `openclaw agent --local` calls the embedded agent directly and does not install either.

## What This Changes

The issue asks OpenClaw to restore subagent completion announces for agents launched with `openclaw agent --local` after the in-process announce handoff change.

## Best Possible Solution

Land a focused fix that makes local embedded subagent completions use a valid announce handoff, either by adapting the queue-first completion path in https://github.com/openclaw/openclaw/pull/76927 or by installing a safe local gateway context for the embedded run.

## Reproduction Assessment

Yes, source-reproducible. Current main’s `--local` path bypasses gateway request-scope setup, while completion direct announce calls the in-process `agent` dispatcher that throws without scope or fallback context.

## Solution Assessment

Unclear until maintainers choose the narrow route. A queue-first completion announce path is likely safer than making unscoped dispatch globally fall back to loopback, but the fix should include explicit `openclaw agent --local` subagent coverage.
