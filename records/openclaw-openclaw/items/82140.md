## Summary

Keep open: current `main` still has the reported scoped-dispatch gap for embedded local agent runs. The `--local` CLI path still enters `agentCommand` directly, completion announce delivery still calls the in-process Gateway dispatcher, and that dispatcher still throws when neither a Gateway request scope nor fallback context is present.

## What This Changes

The issue asks OpenClaw to restore `openclaw agent --local` subagent completion announce delivery so the parent receives the child result instead of yielding indefinitely.

## Best Possible Solution

Install and clear a valid Gateway fallback or request context for embedded local agent runs, then add regression coverage that exercises subagent completion announce delivery through the `openclaw agent --local` path while preserving the in-process dispatcher for scoped Gateway runs.

## Reproduction Assessment

Yes, at source level: the reported CLI path still bypasses Gateway request scope setup, the completion path still calls `dispatchGatewayMethodInProcess("agent")`, and the dispatcher still throws without scope or fallback. I did not run the model-backed live CLI command during this read-only review.

## Solution Assessment

Yes; a narrow fix at the embedded local-agent boundary is the best maintainable direction. Prefer installing a scoped or fallback Gateway context for `--local` runs with regression coverage over reverting the in-process dispatcher wholesale.
