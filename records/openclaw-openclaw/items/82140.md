## Summary

Keep open: current main still has the reported `openclaw agent --local` scoped-dispatch gap. The local CLI path bypasses Gateway request-scope setup, subagent completion delivery now uses the in-process Gateway dispatcher, and no canonical fix or duplicate repair PR was found.

## What This Changes

The issue requests restoring `openclaw agent --local` subagent completion announces so parent sessions receive child output instead of hitting in-process Gateway dispatch without a request scope.

## Best Possible Solution

Install and clear a valid Gateway request or fallback context for embedded local agent runs, then add regression coverage for subagent completion announce delivery through `openclaw agent --local`.

## Reproduction Assessment

Yes at source level: `--local` calls `agentCommand` directly, completion announce delivery calls `dispatchGatewayMethodInProcess("agent")`, and the dispatcher throws when neither request scope nor fallback context exists. I did not run the live CLI/model repro in this read-only review.

## Solution Assessment

Yes: a narrow fix at the embedded local-agent boundary is the best direction. Prefer installing a scoped or fallback Gateway context for local runs with regression coverage over reverting the in-process dispatcher change wholesale.
