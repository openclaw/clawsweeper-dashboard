## Summary

Keep open: current main still has the reported scoped-dispatch gap for embedded local agent runs. The `--local` path enters the agent command directly, while subagent completion announce still uses the in-process Gateway dispatcher that throws without a request scope or fallback context.

## What This Changes

The issue requests fixing a regression where `openclaw agent --local` loses subagent completion announces after the in-process handoff change.

## Best Possible Solution

Install a valid Gateway request or fallback context for embedded `openclaw agent --local` runs and add focused regression coverage for subagent completion announce delivery from that entry point.

## Reproduction Assessment

Yes at source level: the reported path calls `agentCommand` directly for `--local`, while completion announce uses `dispatchGatewayMethodInProcess("agent")`, whose current implementation throws without a request scope or fallback context. I did not run a live model-backed CLI repro in this read-only review.

## Solution Assessment

Yes: a narrow fix at the embedded local-agent boundary is the best direction. Prefer installing or scoping the needed Gateway context for local runs, with regression coverage, rather than reverting the in-process dispatcher change wholesale.
