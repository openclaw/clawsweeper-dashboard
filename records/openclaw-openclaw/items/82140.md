## Summary

Keep open: current main still has the reported scoped-dispatch gap for embedded local agent runs. The `--local` CLI path calls `agentCommand` directly, while subagent completion delivery now uses `dispatchGatewayMethodInProcess("agent")`, which still requires a Gateway request scope or fallback context.

## What This Changes

The issue requests restoring subagent completion announce delivery for agents launched with `openclaw agent --local`.

## Best Possible Solution

Install and reliably clear a valid Gateway request/fallback context for embedded local agent runs, then add regression coverage for subagent completion announce delivery through `openclaw agent --local`.

## Reproduction Assessment

Yes at source level: the reported `--local` path calls `agentCommand` directly, and completion announce delivery calls `dispatchGatewayMethodInProcess("agent")`, which throws when no request scope or fallback context exists. I did not run a model-backed live CLI repro in this read-only review.

## Solution Assessment

Yes: a narrow fix at the embedded local-agent boundary is the maintainable direction. Prefer installing a scoped or fallback Gateway context with cleanup and regression coverage over reverting the in-process dispatcher wholesale.
