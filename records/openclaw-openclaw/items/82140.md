## Summary

Keep open: current main still has the reported scoped-dispatch gap for embedded `openclaw agent --local` subagent completion announcements. The issue includes concrete 2026.5.12 logs, and source inspection shows the local CLI path bypasses the Gateway request scope while completion delivery now requires either that scope or a fallback Gateway context.

## What This Changes

Fix a regression where subagents spawned from `openclaw agent --local` complete but fail to announce their result back to the yielded parent session.

## Best Possible Solution

Install and clear a valid Gateway fallback or request context for embedded local agent runs, then add regression coverage for subagent completion announcements through the `--local` path while preserving the in-process dispatcher for normal Gateway requests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the reported local path calls `agentCommand` directly, completion announce delivery calls `dispatchGatewayMethodInProcess("agent")`, and the dispatcher throws when no request scope or fallback context exists.

## Solution Assessment

Is this the best way to solve the issue? Yes: a narrow fix at the embedded local-agent boundary is more maintainable than reverting the in-process dispatcher wholesale, provided regression coverage proves the local path.

Add focused regression coverage that proves the `--local` path can deliver a child completion announce to the parent, plus guard coverage that existing Gateway-scoped in-process dispatch still works. Update `CHANGELOG.md` under the active fixes section if the repair is user-visible.
