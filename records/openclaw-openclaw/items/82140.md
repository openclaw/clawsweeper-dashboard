## Summary

Keep open: current main still has the reported `--local` scoped-dispatch gap. The local CLI path bypasses Gateway request scope setup, while subagent completion announce now uses the in-process Gateway dispatcher that still throws when no scope or fallback context exists.

## What This Changes

The issue reports a regression where `openclaw agent --local` subagent completion announces fail with a missing Gateway request scope and leave the parent waiting.

## Best Possible Solution

Install and clear a valid Gateway request or fallback context for embedded local agent runs, then cover `openclaw agent --local` subagent completion announce delivery with focused regression tests.

## Reproduction Assessment

Yes at source level: the reported `--local` path calls `agentCommand` directly, completion announce calls `dispatchGatewayMethodInProcess("agent")`, and the dispatcher still throws when no request scope or fallback context exists. I did not run a live model-backed CLI repro in this read-only review.

## Solution Assessment

Yes: a narrow fix at the embedded local-agent boundary is the maintainable solution. Prefer scoped or fallback Gateway context setup for `--local` with regression coverage over reverting the in-process dispatcher back to loopback RPC.
