## Summary

Keep open: current main still has the reported local embedded-agent scope gap. The released `v2026.5.12` regression is still source-reproducible on current `main`: `--local` calls the embedded agent directly, completion announce delivery uses in-process gateway dispatch, and that dispatcher still throws when no request scope or fallback context exists.

## What This Changes

The issue reports a regression where `openclaw agent --local` fails to deliver subagent completion announces because in-process gateway dispatch has no request scope or fallback context.

## Best Possible Solution

Install and clear a valid gateway request or fallback context for embedded local agent runs, then add focused regression coverage for subagent completion announce delivery through `openclaw agent --local`.

## Reproduction Assessment

Yes at source level: the `--local` path calls `agentCommand` directly, completion announce delivery calls `dispatchGatewayMethodInProcess("agent")`, and the dispatcher throws without a request scope or fallback context. I did not run a model-backed live CLI repro in this read-only review.

## Solution Assessment

Yes, a narrow fix at the embedded local-agent boundary is the best direction. It should provide the missing gateway context for `--local` without reverting the in-process dispatcher behavior for scoped gateway runs.
