## Summary

Keep open: the PR addresses real runtime failure modes that are still not fully implemented on current main, but it is not mergeable as-is because it lacks real behavior proof, has active failing checks, and has source-backed blocking review findings across dispatcher, memory, subagent, diagnostics, and queue paths.

## What This Changes

The PR adds plan-stall fail-closed handling, session-delta retry after fetch failures, memory search degradation on embedding failures, subagent/diagnostic/queue fail-close changes, LLM idle-timeout and compaction tuning, and focused tests.

## Best Possible Solution

Split the useful work into focused owner-reviewed patches: dispatcher stall fail-close with refreshed timers and listener cleanup, memory session-delta retry with real backoff, memory search fallback, and separately reviewed diagnostic/subagent/queue/compaction policy changes.

## Reproduction Assessment

Yes for the PR-introduced defects by source inspection: the changed lines create deterministic false-abort, listener-leak, retry-backoff, fallback, orphan, diagnostic, and lane-attribution problems. The original Discord/main-session freeze and memory fetch outage were not live-reproduced here.

## Solution Assessment

No. The direction is useful, but this bundled patch is not the best merge shape because it crosses several runtime surfaces and introduces blocking regressions; smaller patches with real behavior proof are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Refresh the plan-stall timer on each plan update:** `src/auto-reply/reply/dispatch-from-config.ts:336`
  - body: `armControlPlaneStallTimer` returns while a timer is already set, so an active run that keeps emitting plan updates for more than 45 seconds before concrete output is aborted even though control-plane progress is still happening.
  - confidence: 0.91
- **[P2] Clean up forwarded abort listeners on early exits:** `src/auto-reply/reply/dispatch-from-config.ts:320`
  - body: The forwarded abort listener is installed before plugin-bound early-return paths, but cleanup only runs in the later resolver `finally`, so handled, declined, or errored plugin-bound dispatches can leak listeners on long-lived abort signals.
  - confidence: 0.87
- **[P2] Preserve completed cleanup state for orphaned runs:** `src/agents/subagent-registry-helpers.ts:236`
  - body: Clearing `cleanupCompletedAt` reopens runs that were already announced and cleaned up on a prior boot, allowing duplicate orphan retries and parent-facing failure messages.
  - confidence: 0.85
- **[P2] Limit stale reconciliation to tracked run sources:** `src/logging/diagnostic.ts:406`
  - body: The heartbeat flips any `processing` session to idle when it is missing from the embedded/reply registries, but plugin inbound dispatches can run outside those registries and be reclassified mid-flight.
  - confidence: 0.86
- **[P2] Resolve the true session before fail-closing the main lane:** `src/process/command-queue.ts:76`
  - body: Mapping the shared `main` lane to `agent:main:main` can mark the wrong session idle when a failover error belongs to another run using that lane, leaving the real session stuck.
  - confidence: 0.86
- **[P2] Fall back to keyword results when embedding lookup fails:** `extensions/memory-core/src/memory/manager.ts:403`
  - body: When embedding lookup fails, vector-first mode can still return vector-only results with an empty vector, silently producing empty memory searches instead of keyword fallback or a visible error.
  - confidence: 0.84
- **[P2] Enforce retry backoff when a debounce timer exists:** `extensions/memory-core/src/memory/manager-sync-ops.ts:490`
  - body: The fetch-failure retry path exits when `sessionWatchTimer` already exists, so active transcript traffic can keep the normal 5s debounce populated and skip the intended 60s retry backoff.
  - confidence: 0.88
