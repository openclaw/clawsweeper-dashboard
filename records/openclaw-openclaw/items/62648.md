## Summary

Keep open: the diagnostic gap is real, but this branch changes the shared AJV message string instead of rendering the structured `additionalProperty` field at config diagnostic consumers, which can duplicate property names on paths that already decorate the error. The external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR appends AJV `additionalProperties` property names to `formatAjvErrors` messages and adds one schema-validator unit test.

## Best Possible Solution

Render `error.additionalProperty` exactly once in config diagnostic consumers, especially plugin config validation, while keeping `JsonSchemaValidationError.message` as the base AJV message and adding focused regression coverage.

## Reproduction Assessment

Yes, via source inspection rather than execution. `validateJsonSchemaValue` exposes structured `additionalProperty`, but plugin config validation still formats only `error.message`.

## Solution Assessment

No. The safer maintainable fix is to render the structured `additionalProperty` field in the remaining config consumers instead of changing the shared message string that other consumers already decorate.
