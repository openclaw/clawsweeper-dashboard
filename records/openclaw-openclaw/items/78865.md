## Summary

Keep open: current main has stronger opt-in loop blocking and a default-on post-compaction guard, but it still does not provide the requested default hard stop, arbitrary-tool rate-limit cooldown, error-category policy, escalation budget, or user retry confirmation. Related open items cover parts of the problem, but no single canonical thread fully owns this broader platform policy request.

## What This Changes

The issue requests platform-level circuit breakers for repeated failing tool calls, rate-limit cooldowns, error classification, escalation counters, and user confirmation before further retries.

## Best Possible Solution

Define a maintainer-approved retry-safety contract that composes with the existing loop-detection and before-tool-call seams, including default policy, error categories, 429 cooldown state, escalation counters, override UX, config/docs, and focused regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature and policy request. Source inspection gives a high-confidence check that current main only covers narrower opt-in/default-on guard paths and lacks the requested cooldown, classification, and confirmation behavior.

## Solution Assessment

Unclear until maintainers choose the policy. Reusing the existing loop-detection, before-tool-call veto, and approval seams is plausible, but the default thresholds, cooldown model, category taxonomy, and user override flow are product/API decisions.
