## Summary

Keep open: current main has opt-in rolling tool-loop detection and a default-on post-compaction guard, but it still does not implement the requested default hard stop with rate-limit cooldowns, error-category policy, escalation counters, and user retry confirmation for arbitrary failing tools.

## What This Changes

The issue requests platform-level circuit breakers for repeated failing tool calls, rate-limit cooldowns, error classification, escalation counters, and user confirmation before retries.

## Best Possible Solution

Define a maintainer-approved retry safety policy on top of the existing loop-detection, tool-outcome, provider-error, and approval seams, including defaults, error classes, cooldown state, user override behavior, and focused regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a request for stronger default platform behavior. Source inspection does confirm current main lacks the requested default arbitrary-tool rate-limit cooldown and retry-confirmation policy.

## Solution Assessment

Unclear until maintainers choose the policy. Reusing the existing loop-detection, after-tool-call/before-tool-call, provider classification, and approval seams looks maintainable, but the default thresholds, cooldowns, and override UX need product judgment first.
