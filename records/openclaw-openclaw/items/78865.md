## Summary

Keep open: current main has partial tool-loop guardrails, but it still does not implement the requested default arbitrary-tool rate-limit/fatal-error cooldown and user retry-confirmation policy. The related https://github.com/openclaw/openclaw/issues/73781 remains open and narrower, so this issue is not safe to close as a duplicate.

## What This Changes

The issue requests platform-level tool call circuit breakers with repeated-failure detection, rate-limit cooldowns, error classification, escalation counters, and user confirmation before continued retries.

## Best Possible Solution

Define a maintainer-approved tool retry safety policy on top of the existing loop-detection and before-tool-call seams, then implement default behavior, error taxonomy, cooldown state, override UX, and focused regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for stronger default platform behavior. Source inspection confirms current main has partial loop guards but lacks arbitrary-tool rate-limit cooldowns and user retry confirmation.

## Solution Assessment

Unclear until maintainers choose the product policy. Reusing the existing loop-detection, tool-outcome, and before-tool-call seams looks maintainable, but the default thresholds, cooldown semantics, and confirmation UX need explicit design first.
