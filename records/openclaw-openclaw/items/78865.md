## Summary

Keep open: current main has partial tool-loop guardrails, including opt-in repeated-call blocking and a default-on post-compaction guard, but it still does not implement the requested default hard stop, arbitrary-tool rate-limit cooldown, error-category policy, or user retry confirmation flow.

## What This Changes

Request for platform-level tool-call circuit breakers covering repeated failures, rate-limit cooldowns, error-category handling, escalation, and user retry confirmation.

## Best Possible Solution

Define a maintainer-approved retry safety policy on top of the existing loop-detection, before-tool-call, and approval seams, then land the focused canonical fixes or one cohesive implementation with tests and docs.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature and product-policy request. Source inspection does confirm that current main lacks arbitrary-tool rate-limit cooldowns, user retry confirmation, and a consecutive cross-tool failure detector.

## Solution Assessment

Unclear until maintainers choose the policy. Reusing the existing before-tool-call and loop-detection seams looks maintainable, but default enforcement, cooldown semantics, thresholds, and override UX need design first.
