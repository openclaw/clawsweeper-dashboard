## Summary

Keep open: current main has opt-in rolling tool-loop detection and a default-on post-compaction guard, but it still lacks the requested default retry breaker, arbitrary-tool rate-limit cooldown, error-category policy, and user retry confirmation. Related https://github.com/openclaw/openclaw/issues/60767 overlaps, but this issue adds a narrower rate-limit/cooldown and user-override requirement that is not terminally covered there.

## What This Changes

Request platform-level circuit breakers that detect repeated failing tool calls, classify rate-limit and fatal errors, pause or freeze tools, and ask the user before allowing further retries.

## Best Possible Solution

Layer an owner-approved retry safety policy onto the existing before-tool-call and loop-detection seams, covering rate-limit/fatal classification, cooldown state, user override UX, and focused regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a request for stronger default platform behavior. Source inspection does confirm the current rolling-history guard is opt-in and does not provide arbitrary-tool rate-limit cooldown or retry confirmation.

## Solution Assessment

Unclear until maintainers choose the product policy. Reusing the existing before-tool-call, loop-detection, diagnostic, and approval surfaces looks plausible, but default thresholds, cooldown duration, and override UX need design before implementation is safe.
