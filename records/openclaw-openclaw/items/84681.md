## Summary

Keep open: the durable registered/current-turn split is the right direction, but current head introduces a blocking allowlist-boundary regression by ignoring `toolsAllow` for registered schemas, and the live proof still does not cover a current-head heartbeat turn.

## What This Changes

This PR changes Codex app-server dynamic tool handling to separate durable registered schemas from current-turn callable tools so `heartbeat_respond` can remain in the thread schema while normal turns deny execution, with regression coverage for heartbeat thread reuse.

## Best Possible Solution

Keep the registered/available split, but build the durable schema without widening configured tool allowlists, then land only after current-head live heartbeat proof confirms thread reuse and normal-turn denial.

## Reproduction Assessment

Yes, at source level: current main fingerprints per-turn dynamic tool specs for thread reuse, and heartbeat turns add `heartbeat_respond` while normal turns do not. I did not run a live harness because this review is read-only.

## Solution Assessment

No for the current patch: separating registered schemas from available tools is maintainable, but bypassing `toolsAllow` for the durable registry widens an existing policy boundary. The safer fix is an allowlist-bounded stable registry that adds only the heartbeat schema needed for reuse.
