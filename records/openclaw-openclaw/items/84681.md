## Summary

The patch looks directionally correct and I did not find a blocking code defect, but this PR should stay open for protected-label maintainer handling and better real behavior proof; the body currently proves the behavior through targeted tests/build rather than a real heartbeat run or runtime log.

## What This Changes

The PR splits Codex dynamic tools into durable thread-registered specs and current-turn available specs so `heartbeat_respond` can stay in the thread schema without being advertised or executable on normal turns.

## Best Possible Solution

Keep the durable-schema/current-turn-available split, then land after maintainer review once redacted runtime evidence or CI shows a real heartbeat sequence reusing the Codex thread with heartbeat execution still scoped to heartbeat turns.

## Reproduction Assessment

Yes, at source level: current main can be checked by running a normal Codex app-server turn, then a heartbeat turn, then a normal turn and observing the dynamic-tool fingerprint change because `heartbeat_respond` is only present on heartbeat turns. I did not execute the harness because this review is read-only.

## Solution Assessment

Yes, the proposed split is a narrow maintainable fix: register the durable schema for Codex thread compatibility while keeping prompts, context-engine available tools, reports, trajectory context, and execution authorization turn-scoped. The safer remaining step is proof, not a different implementation.
