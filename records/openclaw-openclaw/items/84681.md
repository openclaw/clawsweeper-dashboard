## Summary

Keep this PR open. The registered/available tool split is the right direction for the heartbeat thread-rotation bug, but the current head still builds the durable registry by bypassing both `toolsAllow` and runtime-plan narrowing, which widens an existing tool schema boundary and can change restricted-user behavior before execution denial ever runs.

## What This Changes

The PR changes Codex app-server dynamic-tool handling so thread registration uses a durable schema set while prompt, report, context-engine, and trajectory surfaces use only current-turn tools, with tests for heartbeat thread reuse.

## Best Possible Solution

Keep the registered/available split, but derive the durable registry from the allowed and runtime-normalized tool surface plus only the forced heartbeat schema needed for stable reuse, then add regression tests and current-head live log proof.

## Reproduction Assessment

Yes, source-reproducible: current main fingerprints per-turn dynamic tool specs, and heartbeat turns can add `heartbeat_respond` where normal turns do not, so the stored and current fingerprints can diverge. I did not run a live harness in this read-only review.

## Solution Assessment

No for the current patch: separating registered schemas from current-turn available tools is maintainable, but bypassing allowlist and runtime-plan narrowing is not the narrowest safe fix. The safer solution keeps the durable registry bounded and only forces the heartbeat schema needed for reuse.
