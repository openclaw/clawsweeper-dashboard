## Summary

Keep open: the registered/available split is a useful fix direction, but current head still widens narrow Codex tool policies by forcing `heartbeat_respond` into the durable schema registry, and the real behavior proof does not cover the current head or an actual heartbeat tool call.

## What This Changes

The PR splits Codex dynamic tool specs into durable registered schemas and current-turn available schemas, forces `heartbeat_respond` into the durable set, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the registered/available split, but register heartbeat durably only when heartbeat is actually allowed by the policy, then add empty/narrow allowlist coverage using the real factory path and provide current-head live heartbeat proof.

## Reproduction Assessment

Yes. Source inspection shows current main fingerprints per-turn dynamic tool specs, and heartbeat-triggered tool availability can change that fingerprint; source inspection of the PR head also shows the remaining narrow-allowlist widening.

## Solution Assessment

No. The registered/available split is the right shape, but forcing heartbeat into explicit deny or narrow allowlists is broader than necessary; the safer fix is policy-aware durable registration plus current-head live proof.
