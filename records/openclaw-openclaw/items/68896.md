## Summary

Keep open. Current main still does not implement the requested hold-for-continuous browser dictation UX; it exposes Realtime Talk instead, and the current test suite explicitly expects the old Voice input button to be absent. Because browser-native dictation was recently removed as stale/provider-bypassing behavior, this needs maintainer product and accessibility direction before a repair PR is safe.

## What This Changes

Add a smart Control UI voice input button where short clicks auto-send dictated text and a 3-second hold enters continuous dictation mode.

## Best Possible Solution

Decide whether browser-native dictation should return beside Talk; if accepted, implement it in Control UI source with accessible pointer/touch/keyboard behavior, no generated-dist edits or window globals, focused tests, and reconciled docs.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: source and tests show no STT button state or handlers, and the requested UX cannot currently be exercised.

## Solution Assessment

Unclear. The requested UX may be useful, but the provided implementation direction is not the best maintainable path; the safer solution is a maintainer-approved source-level design that either reintroduces dictation beside Talk or updates docs to make Talk the intended replacement.
