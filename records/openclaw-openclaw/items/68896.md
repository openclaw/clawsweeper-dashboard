## Summary

Keep open. Current main does not implement the requested hold-for-continuous dictation UX, and the remaining Control UI voice surface is the separate realtime Talk path; because browser dictation was intentionally removed and docs still partially mention it, this needs maintainer product/accessibility direction before an automated fix.

## What This Changes

Add a smarter Control UI microphone button where a short click starts one-shot dictation that auto-sends on pause and a 3-second hold starts continuous auto-send voice mode.

## Best Possible Solution

Keep this open for maintainer product/design review. If accepted, implement the behavior in Control UI source with explicit one-shot and continuous dictation modes, pointer/touch/keyboard accessibility, no generated-dist edits or window globals, focused tests for auto-send and Talk coexistence, and updated Control UI docs.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: source and tests show the requested Voice input button and STT state are absent, so the proposed UX cannot currently be exercised.

## Solution Assessment

Unclear. The requested behavior may be useful, but the issue’s dist-file patch and global restart approach are not the best maintainable fix; the safer path is a source-level design decision that either cleanly reintroduces dictation beside Talk or updates docs to make Talk the intended replacement.
