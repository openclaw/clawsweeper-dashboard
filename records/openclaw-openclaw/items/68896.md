## Summary

Keep open. Current main still lacks the requested browser dictation button and hold-for-continuous UX; the remaining voice surface is the separate realtime Talk path, and docs still disagree with source/tests about whether a microphone dictation button exists.

## What This Changes

Request a single Control UI mic button where short click auto-sends one dictation and a 3-second hold enters continuous auto-send mode.

## Best Possible Solution

Resolve the product direction first, then either reintroduce source-level browser dictation with accessible one-shot and continuous modes that coexist with Talk, or update docs/UI copy to make realtime Talk the intended replacement.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: source and tests show no Voice input button or dictation state, so the requested UX cannot currently be exercised.

## Solution Assessment

Unclear. The requested UX may be useful, but the issue’s generated-dist patch, window global, and manual recognition restart are not the best maintainable path; a source-level Control UI design is safer.
