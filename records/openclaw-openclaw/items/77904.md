## Summary

Keep open: the hang is valid and the approach is close, but the helper currently drains stdout only before forcing process exit, which can truncate documented stderr diagnostics on successful model inspection paths.

## What This Changes

The PR adds a default-runtime-gated one-shot exit helper, calls it after successful `models list` and `models status` output, adds focused tests, and updates the changelog.

## Best Possible Solution

Keep the narrow default-runtime lifecycle fix, but make the one-shot exit helper preserve all pending CLI output streams before terminating.

## Reproduction Assessment

Yes. The PR gives timeout-based before-fix commands on v2026.5.4, and current main source shows routed `models list` and `models status` use `defaultRuntime` and return after printing output without a one-shot exit; I did not run local commands in this read-only review.

## Solution Assessment

No. A defaultRuntime-only post-output exit is the right boundary, but the proposed helper should also drain pending stderr so documented diagnostics are not lost when it forces process termination.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Flush stderr before forcing one-shot exit:** `src/cli/one-shot-exit.ts:27-31`
  - body: The helper waits only for `process.stdout` before calling `runtime.exit(0)`. `models list` can still emit successful diagnostics on stderr before printing rows, and the models docs explicitly reserve stderr for JSON-mode diagnostics; Node documents that `process.exit()` can truncate pending stdout or stderr I/O. Please drain pending stderr too before forcing the exit.
  - confidence: 0.84
