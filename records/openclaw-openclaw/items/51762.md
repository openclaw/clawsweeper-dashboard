  - body: Changing the empty-list fallback to `resolveDefaultAgentId(cfg)` makes `listAgentIds({})` read `process.env.OPENCLAW_DEFAULT_AGENT_ID`.   - confidence: 0.88
- **[P1] Do not derive persisted session keys from ambient env:** `src/config/sessions/session-key.ts:52-54`
  - body: With this fallback, callers that omit `agentId` persist `agent:<env>:...` based on the current process environment. Gateway, CLI, cron, and service-manager processes can have different env, so the same configured session can split or appear lost; derive the default from config in the caller and pass an explicit agent id.
  - confidence: 0.86

  - body: `agents.defaultAgentId` duplicates the existing canonical `agents.list[].default` and first-entry default contract, and docs already say the old `routing.defaultAgentId` migrates into `agents.list`. Without maintainer-sponsored semantics, docs, doctor behavior, and upgrade proof, users get two competing sources of truth for default-agent state.
  - confidence: 0.84
