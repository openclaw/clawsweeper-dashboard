## Summary

Keep open: the requested feature is not on current main, and the latest PR head still has source-confirmed env-threading defects that can route an env-selected default agent to the wrong workspace during provider setup.

## What This Changes

The PR adds `agents.defaultAgentId` and `OPENCLAW_DEFAULT_AGENT_ID` fallback handling across default agent resolution, session keys, provider auth paths, config schema, generated config metadata, tests, and changelog.

## Best Possible Solution

Land one consistent default-agent precedence contract across IDs, dirs, workspaces, provider setup, and session keys, with schema/docs comments matching the implemented order.

## Reproduction Assessment

Yes. Source inspection of PR head shows a high-confidence path: use only `OPENCLAW_DEFAULT_AGENT_ID` in a supplied env bag, then follow `resolveAgentWorkspaceDir()` and provider-auth calls that compute the agent with that env but omit it when resolving the workspace.

## Solution Assessment

No. The feature direction is reasonable, but this patch should thread the supplied env consistently through workspace/provider setup resolution and align the precedence comment before merge.
