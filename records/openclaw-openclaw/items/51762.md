## Summary

Keep open. Current main still hardcodes the bootstrap fallback agent directory to `main`, and the PR is the active implementation for linked feature report #17477, but the latest PR head still leaves model-state paths on the bootstrap-only helper so a config-only default agent can route sessions/workspace to one agent while model config/probe state reads or writes under `agents/main/agent`. The latest force-push did positively address several earlier blockers, including schema acceptance, session-key fallback, provider-auth default dirs, and the auth-choice mock.

## What This Changes

The PR adds `agents.defaultAgentId` and `OPENCLAW_DEFAULT_AGENT_ID` support across default-agent resolution, bootstrap directory resolution, session-key helpers, provider auth setup, config schema/generated baseline, changelog, and focused tests.

## Best Possible Solution

Finish the default-agent contract as one coherent path: keep `resolveOpenClawAgentDir()` as the pre-config bootstrap/env helper, but route config-loaded default-agent model/auth/status/probe state through `resolveDefaultAgentDir(cfg, env)` or `resolveAgentDir(cfg, resolveDefaultAgentId(cfg, env), env)`, then align the precedence comment and generated docs surface with the resolver.

## Reproduction Assessment

Yes. Source inspection on the latest PR head gives a high-confidence reproduction: configure only `agents.defaultAgentId: "ops"` with no `OPENCLAW_DEFAULT_AGENT_ID` or explicit agent-dir override, then model config/probe paths still call `resolveOpenClawAgentDir()` and target `agents/main/agent` while routing/workspace resolve the default agent as `ops`.

## Solution Assessment

No. The PR direction is the right narrow product fix, but the current patch is incomplete because some config-loaded default-agent state paths still use the bootstrap fallback helper; the safer solution is to use the new config-aware default-agent dir helper at those call sites.
