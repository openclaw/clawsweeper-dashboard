## Summary

Keep open: current main still lacks the requested default-agent fallback feature, and the latest PR head is useful but still has env-threading gaps in workspace/provider-auth resolution plus one misleading config comment.

## What This Changes

The PR adds configurable default-agent fallback handling through `agents.defaultAgentId` and `OPENCLAW_DEFAULT_AGENT_ID`, plus schema, generated docs baseline, session/provider routing changes, tests, and changelog.

## Best Possible Solution

Apply one consistent default-agent precedence contract everywhere default agent IDs drive directories, workspaces, provider setup, and session keys, then land with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection of the latest PR head gives a high-confidence path: set only `OPENCLAW_DEFAULT_AGENT_ID` in a supplied env bag, then follow the provider-auth/default-workspace calls that compute the agent ID with that env but omit it when resolving the workspace.

## Solution Assessment

No. The feature direction is reasonable, but this patch should thread the supplied env through workspace and provider setup resolution and align the config comment before merge.

  - body: `resolveAgentWorkspaceDir()` accepts an `env` bag, but this comparison still calls `resolveDefaultAgentId(cfg)` without it.   - confidence: 0.87

  - body: These provider auth paths make `agentId` and `agentDir` env-aware, but `workspaceDir` still calls `resolveAgentWorkspaceDir()` without `params.env`. Provider hooks can therefore run in a different workspace than the env-selected default agent directory.
  - confidence: 0.85

  - body: The loaded-provider setup path still derives `agentId` and `workspaceDir` without `params.env`, then uses that workspace for manifest/install-catalog/provider setup. An env-only default can still search or install providers from the `main` workspace while later auth writes under the env-selected agent.
  - confidence: 0.84
- **[P3] Align the defaultAgentId precedence comment:** `src/config/types.agents.ts:148`
  - body: The comment says `defaultAgentId` applies when `agents.list` has no default-flagged entry, but the resolver and tests choose `agents.list[0]` before this key. Please make the comment and generated docs match the actual precedence.
  - confidence: 0.9
