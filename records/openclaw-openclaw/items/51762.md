## Summary

Keep open: current main does not implement the configurable default-agent contract, and this PR is still the active implementation path, but the latest head still has source-confirmed env-threading bugs that can split an env-selected default agent from its workspace/provider setup.

## What This Changes

Adds `agents.defaultAgentId` and `OPENCLAW_DEFAULT_AGENT_ID` fallback handling across config schema, agent/session resolution, provider auth paths, generated config metadata, tests, and changelog.

## Best Possible Solution

Land one consistent default-agent precedence contract that carries the same cfg/env through ID, dir, workspace, provider setup, session keys, schema, docs metadata, and targeted regressions.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: supply only `OPENCLAW_DEFAULT_AGENT_ID` through the env bag, then follow `resolveAgentWorkspaceDir()` or provider-auth setup paths that compute the agent with that env but resolve workspace without it.

## Solution Assessment

No. The feature direction is coherent, but this patch should thread the supplied env consistently through workspace/provider setup resolution and align the config comment before merge.

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread env into default workspace detection:** `src/agents/agent-scope-config.ts:180`
  - body: `resolveAgentWorkspaceDir()` accepts an env bag, but this comparison still calls `resolveDefaultAgentId(cfg)` without it. With only `OPENCLAW_DEFAULT_AGENT_ID` in the supplied env, the env-selected default agent is treated as non-default and gets a per-agent workspace instead of the default workspace.
  - confidence: 0.9
- **[P2] Pass env into provider auth workspace resolution:** `src/plugins/provider-auth-choice.ts:260`
  - body: This provider auth path now makes `agentId` and `agentDir` env-aware, but `workspaceDir` still calls `resolveAgentWorkspaceDir()` without `params.env`. Provider hooks can therefore run in a workspace that does not match the env-selected default agent directory.
  - confidence: 0.88
- **[P2] Use params.env in loaded provider setup:** `src/plugins/provider-auth-choice.ts:326-328`
  - body: The loaded-provider setup path still derives `agentId` and `workspaceDir` without `params.env`, then uses that workspace for manifest, install-catalog, and provider setup. An env-only default can still search or install providers from the `main` workspace while later auth writes under the env-selected agent.
  - confidence: 0.86
- **[P2] Pass env in direct plugin provider workspace:** `src/plugins/provider-auth-choice.ts:513`
  - body: The direct plugin-provider path computes env-aware `agentId` and `agentDir`, then resolves `workspaceDir` without `params.env`. A caller that supplies an env bag can still load setup providers from a workspace that does not match the selected default agent.
  - confidence: 0.85
- **[P3] Align the defaultAgentId precedence comment:** `src/config/types.agents.ts:148`
  - body: The comment says `defaultAgentId` applies when `agents.list` has no default-flagged entry, but the resolver and tests choose `agents.list[0]` before this key. Please make the comment and generated docs match the actual precedence.
  - confidence: 0.9
