## Summary

Keep open. Current main still does not provide a configurable default-agent ID, and this PR is a useful implementation candidate, but the latest provided diff still has merge-blocking behavior: config-only defaults can split default-agent state between the custom agent and `main`, the env-derived agent ID is joined into a filesystem path before normalization, and the user-facing feature lacks the required changelog entry.

## What This Changes

The PR adds `agents.defaultAgentId` and `OPENCLAW_DEFAULT_AGENT_ID` fallback handling across agent resolution, bootstrap agent-dir resolution, session key helpers, config schema/types, generated schema output, and focused tests.

## Best Possible Solution

Keep the PR open and finish it as one coherent default-agent contract: route loaded-config callers through a config-aware default-agent directory helper, normalize or reject the env fallback before any filesystem join, keep session-key and schema behavior aligned, add the required changelog entry, then run the targeted agent/session tests plus the changed gate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main rejects `agents.defaultAgentId` and hardcodes `main`; with the PR diff, config-only defaults route sessions to the custom ID while unmodified default-agent directory callers still use the env-or-`main` bootstrap helper.

## Solution Assessment

No. The PR direction is appropriate, but this is not yet the best maintainable solution because it introduces two default-agent resolution paths instead of a single normalized config-aware contract for routing and state directories.
