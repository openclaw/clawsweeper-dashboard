## Summary

Keep open. Current main still lacks configurable default-agent fallback support, and the latest PR head fixes several earlier concerns but is not merge-ready: default-agent model state paths still use the bootstrap-only agent-dir helper, so a config-only default can continue reading or writing model state under `main`, and GitHub reports the PR branch as dirty/unmergeable.

## What This Changes

The PR adds `agents.defaultAgentId` and `OPENCLAW_DEFAULT_AGENT_ID` default-agent fallback handling across agent resolution, bootstrap agent-dir resolution, session-key helpers, provider auth setup, config schema/generated baseline, changelog, and focused tests.

## Best Possible Solution

Finish this as one coherent default-agent contract: keep the normalized bootstrap env fallback, route every config-loaded default-agent state caller through `resolveDefaultAgentDir(cfg, env)` or `resolveAgentDir(cfg, resolveDefaultAgentId(cfg), env)`, align the public type/schema comment with the intended precedence, rebase the branch cleanly, and then run the focused agent/session/model tests plus the changed gate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with a PR-style config containing only `agents.defaultAgentId: "ops"` and no env override, model config/status/probe/list paths still call `resolveOpenClawAgentDir()`, which resolves `agents/main/agent` rather than the configured default-agent directory.

## Solution Assessment

No. The PR direction is appropriate and several earlier defects are fixed, but the current patch still leaves bootstrap-only and config-aware default directory resolution split across common model-state paths.
