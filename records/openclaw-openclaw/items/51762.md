## Summary

Keep open: current main does not implement the requested configurable default agent, and the PR is useful, but latest head still leaves env-selected defaults inconsistent in workspace and provider-auth paths plus a misleading precedence comment.

## What This Changes

The PR adds `agents.defaultAgentId` and `OPENCLAW_DEFAULT_AGENT_ID` fallback handling across config schema, default agent/session resolution, provider auth path selection, generated config baseline, tests, and changelog.

## Best Possible Solution

Land one consistent precedence contract across default agent id, agent dir, workspace, provider setup, and session keys, with schema/docs comments matching `agents.list` default, first list entry, `agents.defaultAgentId`, `OPENCLAW_DEFAULT_AGENT_ID`, then `main`.

## Reproduction Assessment

Yes. Source inspection at PR head gives a high-confidence path: pass only `OPENCLAW_DEFAULT_AGENT_ID` through `params.env`, then follow workspace/provider-auth calls that compute the agent with that env but resolve workspace without it.

## Solution Assessment

No. The feature direction is reasonable, but the patch should thread the supplied env consistently through workspace/provider setup resolution and align the precedence comment before merge.

Overall confidence: 0.87

Full review comments:

- **[P2] Thread env through default workspace detection:** `src/agents/agent-scope-config.ts:180`
  - body: `resolveAgentWorkspaceDir()` accepts an env bag, but this comparison still calls `resolveDefaultAgentId(cfg)` without it. With only `OPENCLAW_DEFAULT_AGENT_ID` in the supplied env, the env-selected default agent is treated as non-default and gets a per-agent workspace instead of the default workspace.
  - confidence: 0.88
- **[P2] Pass env into provider auth workspace resolution:** `src/plugins/provider-auth-choice.ts:260`
  - body: This provider auth path makes `agentId` and `agentDir` env-aware, but `workspaceDir` still calls `resolveAgentWorkspaceDir()` without `params.env`. Provider hooks can therefore run in a workspace that does not match the env-selected default agent directory.
  - confidence: 0.86
- **[P2] Use params.env in loaded provider setup:** `src/plugins/provider-auth-choice.ts:326-328`
  - body: The loaded-provider setup path still derives `agentId` and `workspaceDir` without `params.env`, then uses that workspace for manifest, install-catalog, and provider setup. An env-only default can still search or install providers from the wrong workspace.
  - confidence: 0.85
- **[P2] Pass env in direct plugin provider workspace:** `src/plugins/provider-auth-choice.ts:513`
  - body: The direct plugin-provider path computes env-aware `agentId` and `agentDir`, then resolves `workspaceDir` without `params.env`. A caller that supplies an env bag can still load setup providers from a workspace that does not match the selected default agent.
  - confidence: 0.84
- **[P3] Align defaultAgentId precedence comment:** `src/config/types.agents.ts:148`
  - body: The comment says `defaultAgentId` applies when `agents.list` has no default-flagged entry, but the resolver and tests choose `agents.list[0]` before this key. Please make the comment and generated docs match the actual precedence.
  - confidence: 0.9
