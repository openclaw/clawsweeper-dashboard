## Summary

Keep open. Current main still has a source-visible path where an over-budget session can ask compaction to run, compaction returns `ok: true, compacted: false` for an empty/header-only transcript, and the runner only treats actual compaction as recovery; `sessions cleanup --fix-missing` also only handles absent/malformed transcript paths, not present empty transcripts.

## What This Changes

The issue requests fixing a Discord embedded-session deadlock where stale session token state plus an empty transcript makes every inbound message bounce during preflight compaction.

## Best Possible Solution

Implement a core bug fix that treats `no real conversation messages` during required preflight compaction as stale session metadata, clears or refreshes the token snapshot, preserves successful fallback-turn transcript writes, and extends cleanup/doctor coverage for empty-present transcripts.

## Reproduction Assessment

Yes, from source inspection we have a credible current-main path: an empty/header-only transcript returns `ok && !compacted`, and the runner lacks a fresh-session recovery branch for that result. I did not run a live Discord fallback repro, so confidence is medium rather than high.

## Solution Assessment

No, current main is not the best end state because it still depends on actual compaction success and only cleans up missing transcript files. The narrow maintainable fix should repair the stale session metadata and transcript persistence boundary, then cover empty-present cleanup explicitly.
