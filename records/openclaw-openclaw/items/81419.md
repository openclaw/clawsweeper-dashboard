## Summary

Keep open: this is a source-backed Xiaomi MiMo compatibility bug on current main, not an already-implemented or duplicate cleanup item. The current Xiaomi OpenAI-compatible path advertises reasoning MiMo models but does not apply the existing reasoning-content replay backfill that only targets DeepSeek V4 model ids, and Xiaomi’s own contract says missing `reasoning_content` after tool-call history causes 400 responses.

## What This Changes

Ensure Xiaomi MiMo thinking-mode tool-call history replays `reasoning_content` on assistant messages so follow-up turns do not fail with `400 Param Incorrect`.

## Best Possible Solution

Add a Xiaomi-owned OpenAI-completions replay compatibility path that backfills `reasoning_content` for affected MiMo reasoning models during enabled-thinking tool-call replay, with focused regression coverage and coordination with the existing Xiaomi output-normalization work.

## Reproduction Assessment

Yes at source level, but not from a fresh live Xiaomi run. Xiaomi’s docs define the 400 condition, and current main’s Xiaomi OpenAI-compatible path has no Xiaomi-specific hook or accepted manifest compat field to set Pi’s replay backfill behavior.

## Solution Assessment

Yes, a Xiaomi/plugin-owned compat wrapper or a narrowly supported manifest compat field is the maintainable fix boundary. The existing open Xiaomi output-normalization PR is useful related work, but it does not solve the missing `reasoning_content` replay on request history.
