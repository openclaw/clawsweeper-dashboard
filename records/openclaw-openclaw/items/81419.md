## Summary

Keep open: current main still has a source-backed Xiaomi MiMo OpenAI-compatible replay bug. Xiaomi’s provider surface uses `openai-completions` for reasoning MiMo models, but the existing `reasoning_content` replay backfill is only wired for DeepSeek V4 model ids and OpenClaw’s accepted model compat fields do not currently expose Pi’s replay flag.

## What This Changes

The issue asks OpenClaw to pass `reasoning_content` back on Xiaomi MiMo assistant/tool-call history so multi-turn tool conversations do not fail with `400 Param Incorrect`.

## Best Possible Solution

Add a Xiaomi/OpenAI-compatible replay compatibility path, preferably through a reusable provider-family helper, that backfills `reasoning_content` for affected MiMo reasoning models during enabled-thinking assistant/tool-call replay and covers bundled plus custom Xiaomi Token Plan refs.

## Reproduction Assessment

Yes, at source level: current main routes Xiaomi MiMo through OpenAI completions without a Xiaomi replay wrapper, while Pi only adds `reasoning_content` when a compat flag is set and Xiaomi documents a 400 when that field is missing. I did not run a fresh live Xiaomi request.

## Solution Assessment

Yes: a narrow Xiaomi-owned wrapper or generic OpenAI-compatible replay helper is the maintainable fix boundary. The open Xiaomi output-normalization PR is useful adjacent work, but it does not solve request-history passback.
