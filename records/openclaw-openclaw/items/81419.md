## Summary

Keep open: this is a source-backed Xiaomi MiMo compatibility bug on current main. The Xiaomi OpenAI-compatible path exposes reasoning MiMo models, but neither the Xiaomi plugin nor the generic replay wrapper applies the existing `reasoning_content` assistant-history backfill outside DeepSeek V4.

## What This Changes

The issue requests MiMo OpenAI-compatible request replay to preserve or backfill `reasoning_content` on assistant messages with tool calls during multi-turn thinking conversations.

## Best Possible Solution

Add a narrow MiMo OpenAI-compatible replay compatibility path that applies DeepSeek-style thinking controls and backfills `reasoning_content` for affected Xiaomi/MiMo models, including custom Xiaomi-compatible base URLs, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: Xiaomi documents the 400 condition, Pi only backfills the field behind a DeepSeek-only compat flag, and current OpenClaw does not wire that behavior for MiMo. I did not perform a live Xiaomi API run.

## Solution Assessment

Yes, a narrow provider/family compatibility wrapper is the best fix direction. The existing Xiaomi response-normalization PR is useful adjacent work, but it does not solve request-history replay.
