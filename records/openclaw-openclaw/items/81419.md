## Summary

Keep open: current main still has a source-backed Xiaomi MiMo OpenAI-compatible replay bug. The bundled Xiaomi provider and custom Xiaomi-compatible provider configs use the existing OpenAI-completions path, but the only `reasoning_content` assistant-history backfill is still wired to DeepSeek V4 model ids, and a fresh issue comment reports the same 400 failure on v2026.5.12-beta.4+ Linux.

## What This Changes

The issue asks OpenClaw to preserve or backfill `reasoning_content` on Xiaomi MiMo assistant tool-call history so multi-turn thinking-mode requests do not fail with 400 errors.

## Best Possible Solution

Add a narrow Xiaomi/MiMo OpenAI-compatible replay compatibility path that applies DeepSeek-style thinking payloads and backfills `reasoning_content` for affected MiMo assistant tool-call history, while leaving the broader Anthropic-compatible design on https://github.com/openclaw/openclaw/issues/81261.

## Reproduction Assessment

Yes, at source level: the current Xiaomi OpenAI-compatible path has no Xiaomi replay wrapper, while the existing `reasoning_content` backfill is gated to DeepSeek V4 ids only. The later issue comment also provides a fresh real-run failure log, but I did not repeat it against a live Xiaomi key.

## Solution Assessment

Yes, the best fix direction is a provider/family compatibility wrapper or equivalent upstream compat detection, not a broad user config knob. The repair should cover bundled `xiaomi` and custom Xiaomi-compatible provider ids/base URLs without changing unrelated OpenAI-compatible providers.
