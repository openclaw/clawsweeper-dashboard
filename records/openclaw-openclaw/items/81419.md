## Summary

Keep open: current main still has a source-backed Xiaomi MiMo OpenAI-compatible replay bug. Xiaomi’s contract requires `reasoning_content` in multi-turn tool-call history, while OpenClaw only backfills that field for DeepSeek V4 and the Xiaomi provider has no equivalent request wrapper.

## What This Changes

The issue requests Xiaomi MiMo `reasoning_content` passback for assistant tool-call history in multi-turn OpenAI-compatible conversations.

## Best Possible Solution

Add a provider-aware OpenAI-compatible replay path that backfills `reasoning_content` for affected MiMo reasoning models during enabled-thinking tool-call history replay, preferably reusing the existing replay helper or existing `compat.thinkingFormat` surface instead of broad core hard-coding.

## Reproduction Assessment

Yes at source level, but not from a fresh live Xiaomi run. Current main routes Xiaomi through OpenAI completions and only applies the `reasoning_content` replay backfill to DeepSeek V4 model ids, while Xiaomi’s docs define the 400 condition for missing passback.

## Solution Assessment

Yes, the narrow maintainable fix is a Xiaomi/provider-aware replay wrapper or an existing model-compat-driven path that invokes the same backfill behavior. The open Xiaomi output-normalization PR is related but solves response rendering, not request replay.
