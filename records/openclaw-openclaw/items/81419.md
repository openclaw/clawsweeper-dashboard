## Summary

Keep open: current main still lacks Xiaomi MiMo request-history replay support for `reasoning_content` on the OpenAI-compatible path. This is a source-backed provider compatibility bug, not solved by the related Xiaomi output-rendering PR or the broader Anthropic-compatible tracking issue.

## What This Changes

Request to include `reasoning_content` on replayed Xiaomi MiMo assistant tool-call messages during multi-turn thinking conversations.

## Best Possible Solution

Add a Xiaomi-owned OpenAI-completions replay compatibility path, or a narrow documented compat seam consumed by Xiaomi, that preserves/backfills `reasoning_content` for affected MiMo reasoning models during tool-call replay without broad provider heuristics.

## Reproduction Assessment

Yes at source level: Xiaomi's current OpenAI-compatible path has no replay backfill, while Xiaomi's published contract requires `reasoning_content` in later tool-call history and Pi only auto-enables that behavior for DeepSeek. I did not run a fresh live Xiaomi API reproduction.

## Solution Assessment

Yes: a provider-scoped wrapper or narrow existing-compat integration is the best fix boundary. A broad core heuristic for all OpenAI-compatible providers would be riskier, and the related output-normalization PR does not solve request replay.
