## Summary

Keep this open. Current main still has JSON output and an adjacent optional `llm-task` schema validator, but it does not expose a caller-supplied JSON Schema contract, validation path, retry policy, or schema-specific failure envelope on the requested CLI/Gateway surfaces.

## What This Changes

Add first-class JSON Schema input and validation support to `openclaw infer model run` and/or `openclaw agent`, ideally with invalid-output repair and machine-readable failures.

## Best Possible Solution

Define an additive provider-neutral JSON Schema contract on `openclaw infer model run` first, with file/inline schema input, final-output validation, explicit retry semantics, structured errors, docs, and focused CLI/protocol tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Static inspection gives high-confidence evidence of the gap because current CLI metadata, parser, execution, docs, and Gateway protocol expose JSON output but no caller-supplied JSON Schema contract.

## Solution Assessment

Partly: first-class schema validation is the right direction, but the exact CLI/protocol surface and repair semantics need maintainer API/product decisions. A narrow provider-neutral `infer model run` implementation is the safest first step.
