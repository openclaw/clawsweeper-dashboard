## Summary

Keep this issue open. Current main has the related raw `infer model run` improvements, but it still does not expose a JSON Schema input contract, schema validation, repair/retry semantics, or machine-readable schema-failure reporting on the requested CLI/Gateway surfaces.

## What This Changes

Request first-class JSON Schema support on `openclaw agent` and `openclaw infer model run`, including output validation, defined repair/retry behavior, and machine-readable failure reporting.

## Best Possible Solution

Implement a provider-neutral JSON Schema contract on `openclaw infer model run` first: accept schema from a file or inline JSON, parse and validate final JSON output with a shared validator, define retry/repair behavior and exhausted-failure envelopes, keep local and Gateway behavior consistent, and update CLI docs/tests. Broader `openclaw agent` exposure should be additive after that contract is stable.

## Reproduction Assessment

Not applicable as a feature request, but the gap is reproducible by static inspection: the current command metadata, parser, execution path, Gateway schema, and docs expose JSON output only, not schema input or validation semantics.

## Solution Assessment

Yes for the narrow first step: implementing this on `infer model run` with a stable core validation contract is the maintainable path. Provider-native schema modes can be used later as optimizations, but they should not define inconsistent provider-specific CLI behavior.

## Repair Work Prompt

Implement JSON Schema support for `openclaw infer model run` as the first core surface for https://github.com/openclaw/openclaw/issues/67306. Add a provider-neutral `--json-schema <path-or-inline>` contract, parse the final model output as JSON when a schema is supplied, validate it with an existing shared JSON Schema validator where appropriate, and return structured machine-readable validation/parse/repair-exhausted errors in `--json` mode. Define repair behavior explicitly: either implement a small bounded retry/repair loop or document and encode no-repair semantics clearly. Keep provider-native schema modes out of scope unless they can be hidden behind the same stable contract. Preserve current raw `model run` behavior, keep local and Gateway transports consistent, and only add an additive Gateway protocol field if needed. Do not broaden this into a full `openclaw agent` UX unless the minimal protocol shape naturally supports it. Update CLI docs and focused tests; add a changelog entry only if repo policy treats the CLI feature as user-facing for the active version.
