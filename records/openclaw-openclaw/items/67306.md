## Summary

Keep this issue open. Current main has the raw/lean `infer model run` path, but it still has no caller-supplied JSON Schema option, final-output schema validation, schema-aware retry/repair policy, or machine-readable schema failure contract on the requested CLI/Gateway surfaces; the optional `llm-task` plugin is useful adjacent work but does not satisfy this core CLI request.

## What This Changes

Add first-class JSON Schema input and validation support to `openclaw agent` and/or `openclaw infer model run`, including explicit invalid-output repair behavior and machine-readable failures.

## Best Possible Solution

Define a narrow provider-neutral JSON Schema contract on `openclaw infer model run` first, with file/inline schema input, final JSON parsing and validation, structured validation errors, documented retry or no-retry semantics, local/Gateway parity, and docs/tests; treat broader `openclaw agent` exposure as a follow-up decision.

## Reproduction Assessment

Not applicable as a feature request. The gap is high-confidence by static inspection of command metadata, parser options, execution paths, Gateway schema, tests, and docs on current main.

## Solution Assessment

Yes, a provider-neutral `infer model run` contract is the best first slice. Provider-native schema modes can remain an implementation optimization behind the same validation envelope, while full `openclaw agent` exposure should wait for an explicit API decision.
