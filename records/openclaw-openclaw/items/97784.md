## Summary

This PR is still necessary: current main and the latest release still use seven unbounded Microsoft Teams graph-upload success-body JSON reads, while the PR narrowly replaces them with the existing bounded provider JSON reader and adds credible loopback proof. I found no blocking patch defect; the remaining decision is maintainer acceptance of the intended 16 MiB fail-closed behavior for unusually large successful Graph metadata responses.

## What This Changes

The PR changes Microsoft Teams graph-upload success-response parsing to use `readProviderJsonResponse` and adds loopback tests for normal, malformed, and oversized JSON responses.

## Best Possible Solution

Land the narrow helper reuse once maintainers accept the 16 MiB cap tradeoff and exact-head checks remain acceptable.

## Solution Assessment

Yes. Reusing the public `openclaw/plugin-sdk/provider-http` helper at each raw graph-upload success read is the narrow maintainable fix; a custom local reader or broad multi-plugin sweep would be higher risk for this PR.
