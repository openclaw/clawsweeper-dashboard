## Summary

Keep open for maintainer review: the patch is focused, uses the existing bounded JSON helper on all seven MSTeams graph-upload success reads, and includes sufficient loopback terminal proof; the remaining decision is accepting the 16 MiB fail-closed behavior for Graph metadata responses.

## What This Changes

The PR replaces seven Microsoft Teams graph-upload `res.json()` success-response reads with `readProviderJsonResponse` and adds loopback tests for normal, malformed, and oversized JSON responses.

## Best Possible Solution

Land the narrow helper reuse once maintainers accept the 16 MiB cap tradeoff and exact-head merge checks are acceptable, while keeping attachment, SSO, and broader raw-read sweeps separate.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.10 have seven raw `res.json()` success reads in the affected graph-upload paths; an oversized successful JSON stream would exercise the unbounded path, and the PR proof shows the patched path rejects at the cap.

## Solution Assessment

Yes. Reusing the public `openclaw/plugin-sdk/provider-http` helper at each raw graph-upload success read is the narrow maintainable fix; a custom local reader or broad multi-plugin sweep would add more risk for this PR.
