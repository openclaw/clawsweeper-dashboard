## Summary

Keep open: the linked TUI bug is real, but this PR does not yet preserve the draft in the actual TUI because pi-tui clears the editor before `onSubmit`; the PR tests and proof use a mocked editor and miss that contract.

## What This Changes

The PR adds busy-state guards to the TUI submit and send paths so normal chat text is blocked while a run is active while slash commands can still route.

## Best Possible Solution

Fix the actual editor-clearing boundary by restoring the blocked draft or gating submission before pi-tui clears it, keep the send-path busy guard, and add proof that exercises the real TUI editor path.

## Reproduction Assessment

Yes, from source inspection: current main clears the TUI editor before normal sends, and the pi-tui editor clears before `onSubmit`, so the linked busy-submit draft loss is reproducible at the submit boundary. I did not run an interactive terminal session in this read-only review.

## Solution Assessment

No; the PR blocks extra sends, but the proposed draft-preservation path runs after the real editor has already cleared the buffer. The maintainable fix should handle pi-tui's submit contract directly and test that contract.
