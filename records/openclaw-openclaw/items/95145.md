## What This Changes

The issue asks OpenClaw to report macOS 26/Tahoe from the macOS product version instead of deriving macOS 15.x from Darwin kernel 25.x.

## Best Possible Solution

Centralize or share the macOS product-version display logic for agent runtime metadata, avoid double-appending architecture in prompt rendering, and keep gateway/platform policy IDs unchanged.

## Reproduction Assessment

Yes from source inspection: on Darwin, the runtime prompt paths still use `os.release()`, so a Tahoe host with Darwin 25.5.0 will send that kernel release into agent context. I did not run a live macOS 26 host in this read-only review.

## Solution Assessment

Yes, using the macOS product version for Darwin runtime metadata is the narrowest maintainable fix. It should reuse or mirror the existing `sw_vers` behavior without adding a config knob or changing canonical gateway platform IDs.
