## Summary

Keep open: current main and the latest release still build agent runtime OS metadata from `os.type()` plus `os.release()` in the affected prompt paths, and five open PRs are linked as candidate fixes. The issue should stay open until one canonical fix lands and the competing PRs are deduplicated.

## What This Changes

The issue reports that macOS 26/Tahoe hosts expose Darwin 25.x in agent runtime OS metadata, causing agent context to show the wrong macOS generation.

## Best Possible Solution

Land one canonical runtime-OS helper fix that uses the macOS product version for Darwin agent runtime metadata, preserves non-Darwin prompt output, avoids double-appending architecture, and keeps status/trajectory summary shape stable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on Darwin, the three runtime prompt builders pass `os.release()` into `runtimeInfo.os`, and the issue provides the Tahoe `uname -r` versus `sw_vers` mismatch needed to trigger the wrong prompt value.

## Solution Assessment

Yes. The narrow maintainable fix is to share a cached macOS product-version runtime label helper for the agent prompt call sites while preserving existing non-Darwin output; the remaining question is which linked PR should become canonical.
