## Summary

Keep open: current main and the latest release still feed Darwin kernel release values into agent runtime OS metadata, and multiple open PRs with closing references are already candidate fixes for the same bug.

## What This Changes

The issue reports that macOS Tahoe hosts expose Darwin 25.x as the runtime OS version instead of the macOS 26 product version from `sw_vers`.

## Best Possible Solution

Land one canonical runtime OS helper fix that uses `sw_vers -productVersion` for Darwin agent runtime metadata, preserves non-Darwin prompt output, avoids double-appending architecture, and keeps status/trajectory summary shape stable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on Darwin, three runtime prompt builders still pass `os.release()` into `runtimeInfo.os`, and `buildRuntimeLine()` renders that value.

## Solution Assessment

Yes. The narrow maintainable direction is to use a cached Darwin product-version helper for runtime OS metadata; the remaining maintainer choice is which open candidate PR has the best helper, fallback, and proof shape.
