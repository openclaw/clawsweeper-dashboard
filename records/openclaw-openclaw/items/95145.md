## Summary

Keep open: current main and the latest release still feed `os.type()` plus `os.release()` into several agent runtime prompt paths, and multiple open PRs explicitly reference this issue with closing syntax, so the issue should stay open until one canonical fix lands.

## What This Changes

The issue reports that macOS Tahoe hosts can expose Darwin 25.x in OpenClaw runtime OS metadata, causing agent context to see the wrong macOS marketing version instead of macOS 26.x.

## Best Possible Solution

Land one canonical runtime-OS helper fix that uses the macOS product version for Darwin agent runtime metadata, preserves non-Darwin prompt output, avoids double-appending architecture, and keeps status/trajectory summary shape stable.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on Darwin, three runtime prompt builders still pass `os.release()` into `runtimeInfo.os`, and `buildRuntimeLine()` renders that value. I did not run a live macOS Tahoe host in this read-only review.

## Solution Assessment

Yes, using `sw_vers -productVersion` for Darwin runtime metadata is the narrow maintainable direction. The remaining maintainer choice is which open candidate PR has the best helper, cache, fallback, and proof shape.
