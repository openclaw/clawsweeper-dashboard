## Summary

Keep open: current main and the latest release still feed `os.type()` plus `os.release()` into agent runtime prompt metadata, and multiple linked fix PRs are still open implementation candidates rather than merged fixes.

## What This Changes

The issue reports that macOS 26 Tahoe hosts expose Darwin 25.x in OpenClaw runtime OS metadata instead of the macOS product version from `sw_vers`.

## Best Possible Solution

Land one canonical fix that centralizes the Darwin product-version runtime OS label, updates the CLI, embedded attempt, and compaction prompt paths, preserves non-Darwin output, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows all three runtime prompt paths still use `os.release()` on current main, so a Tahoe host with Darwin 25.x will expose the kernel release in agent context; I did not run a live macOS 26 host.

## Solution Assessment

Yes. Reusing a cached `sw_vers -productVersion` product-version helper for runtime OS metadata is the narrow maintainable fix; changing fallback mapping or only status diagnostics would leave the agent prompt bug partially unresolved.
