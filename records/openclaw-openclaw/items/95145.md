## Summary

Keep open: current main and the latest release still pass `os.type()` plus `os.release()` into agent runtime prompt metadata, so Tahoe hosts can still expose Darwin 25.x in agent context. Multiple open PRs already reference this issue with closing syntax, so the issue should remain the canonical tracker until maintainers land one fix branch.

## What This Changes

The issue reports that OpenClaw maps Darwin kernel 25.x to the wrong macOS version on Tahoe instead of using the macOS product version from `sw_vers`.

## Best Possible Solution

Land one focused fix that routes all agent runtime OS metadata through a shared cached Darwin product-version helper, preserves non-Darwin output, and adds regression coverage for the three prompt paths.

## Reproduction Assessment

Yes, from source inspection: current main still builds runtime prompt OS metadata from `os.release()`, and the reporter supplied Tahoe evidence where `uname -r` is `25.5.0` while `sw_vers` is `26.5.1`. I did not run a live macOS 26 host in this read-only review.

## Solution Assessment

Yes, the requested direction is the best fix shape: use the macOS product version for Darwin runtime metadata and keep the architecture appended only by `buildRuntimeLine()`. The exact branch still needs maintainer selection because several open PRs implement overlapping variants.
