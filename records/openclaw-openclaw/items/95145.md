## Summary

Keep open: current main and v2026.6.9 still feed Darwin `os.release()` into agent runtime prompt metadata, and several open PRs with closing syntax are candidate fixes rather than merged implementation provenance.

## What This Changes

The issue asks OpenClaw to report the macOS product version on Darwin 25/Tahoe hosts instead of exposing the Darwin kernel version in host and runtime metadata.

## Best Possible Solution

Land one canonical cached runtime OS label helper that uses `sw_vers -productVersion` for Darwin agent runtime metadata, preserves non-Darwin prompt output and existing status summary shape, then close or supersede the duplicate PRs.

## Reproduction Assessment

Yes. Source inspection shows all three runtime prompt builders still pass `os.release()` into `runtimeInfo.os`, and `buildRuntimeLine()` renders that value; I did not run a live Tahoe host in this read-only review.

## Solution Assessment

Yes. Using the macOS product version from `sw_vers` for Darwin runtime metadata is the narrow maintainable fix; the remaining choice is which open candidate PR has the right cache, fallback, and proof shape.
