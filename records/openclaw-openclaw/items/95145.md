## Summary

Keep open: current main and the latest release still render Darwin `os.release()` into agent runtime OS metadata, and multiple open pull requests still reference this issue with closing syntax. A commenter linked https://github.com/openclaw/openclaw/pull/95361, but that PR is closed unmerged, so it is not implementation provenance.

## What This Changes

The issue reports that macOS Tahoe hosts expose Darwin 25.x as the runtime OS version instead of the macOS 26 product version.

## Best Possible Solution

Land one canonical runtime-OS helper fix that uses the macOS product version for Darwin agent runtime metadata, preserves non-Darwin prompt output, avoids double-appending architecture, and keeps status/trajectory summary shape stable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on Darwin, three runtime prompt builders still pass `os.release()` into `runtimeInfo.os`, and `buildRuntimeLine()` renders that value; I did not run a live Tahoe host in this read-only review.

## Solution Assessment

Yes for the direction: using `sw_vers -productVersion` for Darwin runtime metadata is the narrow maintainable fix. The remaining maintainer choice is which open candidate PR has the best helper, cache, fallback, and proof shape.
