## Summary

Keep open: current main and the latest release still expose Darwin `os.release()` in agent runtime prompt metadata, and multiple open PRs with closing syntax already own candidate fixes.

## What This Changes

The issue asks OpenClaw to report the macOS product version on Tahoe instead of deriving agent/runtime OS metadata from Darwin kernel 25.x.

## Best Possible Solution

Land one focused canonical PR that routes all runtime prompt OS metadata through a cached Darwin `sw_vers -productVersion` helper while preserving non-Darwin prompt output and avoiding unrelated status/schema changes.

## Reproduction Assessment

Yes, source-reproducible: current main still feeds `os.type()` plus `os.release()` into the CLI, embedded-attempt, and compaction runtime prompt OS fields, while the report provides the Tahoe `sw_vers` versus `uname` mismatch. I did not run a live Tahoe host repro in this read-only review.

## Solution Assessment

Yes, the narrow shared-helper direction in the better linked PRs is the best fix shape; the issue itself should remain open until one candidate PR merges. A fallback-only Darwin mapping would be too narrow because current runtime prompt callers still bypass `sw_vers`.
