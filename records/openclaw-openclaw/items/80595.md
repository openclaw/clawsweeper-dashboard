## Summary

Keep open: current main and the latest release still support only the generated `gateway.systemd.env` path, while this issue asks for a new opt-in systemd environment-source mode. That is a real operator ergonomics request, but it changes service credential handling and needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests an explicit Linux systemd gateway mode that can use the state-dir `.env` directly, or otherwise disable generation of `gateway.systemd.env`.

## Best Possible Solution

If maintainers accept the tradeoff, add an explicit opt-in systemd env-source contract with schema/help/docs, install and doctor preservation behavior, validation equivalent to the generated-file path, and regression coverage while keeping `gateway.systemd.env` as the default.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main and `v2026.6.6` verifies the premise: OpenClaw still intentionally generates and references `gateway.systemd.env`.

## Solution Assessment

Unclear until maintainers choose the contract. Direct `.env` loading is plausible only as an explicit opt-in with validation and doctor/install support; switching the default or bypassing the current generated-file safety path would not be the best fix.
