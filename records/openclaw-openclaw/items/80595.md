## Summary

Keep open: current main and the latest release still implement only the generated `gateway.systemd.env` path, while this issue requests a new opt-in Linux systemd environment-source contract. That is a valid operator ergonomics request, but it changes service credential handling and needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests an explicit opt-in Linux systemd mode that lets the gateway service use the state-dir `.env` directly, or disables generation of `gateway.systemd.env`.

## Best Possible Solution

Keep generated `gateway.systemd.env` as the default; if maintainers accept the tradeoff, add an explicit validated opt-in systemd env-source contract with schema/help/docs, install and doctor preservation behavior, and Linux systemd proof.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main and `v2026.6.8` verifies the premise: OpenClaw still intentionally generates and references `gateway.systemd.env` and has no requested direct-`.env` option.

## Solution Assessment

Unclear until maintainers choose the contract. Direct `.env` loading is plausible only as an explicit opt-in with validation and doctor/install support; switching the default or bypassing the current generated-file safety path would not be the best fix.
