## Summary

Keep open: current main still implements the generated `gateway.systemd.env` model and does not provide a supported Linux systemd config mode for using the state-dir `.env` directly as `EnvironmentFile`. The linked closed issues and merged PRs improved preservation, install handoff, and update-time repair behavior, but they did not add this requested opt-in operator contract.

## What This Changes

The issue requests a supported Linux systemd option to use the state-dir `.env` directly as the gateway unit `EnvironmentFile`, or to disable generation of `gateway.systemd.env`.

## Best Possible Solution

Add a deliberate Linux systemd service-environment contract with schema/types/docs, validation for systemd-compatible `.env` values, and install/doctor preservation semantics; alternatively document that the generated env file is the only supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection and current tests show main still renders the generated `gateway.systemd.env` behavior described by the issue.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable implementation needs schema, validation, docs, and doctor/install behavior rather than an ad hoc unit override.
