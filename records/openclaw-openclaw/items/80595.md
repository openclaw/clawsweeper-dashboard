## Summary

Keep open: current `main` and latest release still support only the generated `gateway.systemd.env` model for Linux user-systemd gateway services. The related fixes improved generated-file safety and update/doctor preservation, but they do not add a supported config key or install/doctor behavior for using `%h/.openclaw/.env` directly; that remains a new service-environment contract needing product and security review.

## What This Changes

The issue requests an explicit Linux systemd gateway option to use the state-dir `.env` directly as the service `EnvironmentFile`, or to disable generation of `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate opt-in Linux systemd service-env contract with schema, systemd-compatible env validation, docs, install/doctor preservation semantics, and a default that keeps generated `gateway.systemd.env` for existing users.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current `main` still generates `gateway.systemd.env` and has no supported direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and install/doctor semantics rather than preserving an unsupported unit edit.
