## Summary

Keep open: current main and v2026.5.12 still use the generated `gateway.systemd.env` model, and there is no supported config surface for using the state-dir `.env` directly. The related fixes improved generated-file safety and update/doctor behavior, but this remains a new Linux systemd service-environment contract needing maintainer product and security judgment.

## What This Changes

The issue requests an opt-in Linux systemd mode or config option to use the state-dir `.env` or a custom `EnvironmentFile` directly instead of generating `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate Linux systemd service-env contract with schema, validation for systemd-compatible env files, docs, and install/doctor preservation behavior; if maintainers reject that contract, document generated `gateway.systemd.env` as the only supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and has no supported direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the supported service-env contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and install/doctor semantics rather than an unsupported unit edit.
