## Summary

Keep open: current main still supports only the generated `gateway.systemd.env` path for Linux systemd installs, with no supported config option for using the state-dir `.env` directly as the unit `EnvironmentFile`. The related closed reports hardened the generated-file approach, and the open provenance PR is adjacent but does not implement this requested operator mode.

## What This Changes

Request an opt-in Linux systemd mode or config option that uses the state-dir `.env` directly as the gateway unit `EnvironmentFile` or disables generation of `gateway.systemd.env`.

## Best Possible Solution

Add a deliberate Linux systemd config contract with systemd-compatible `.env` validation, docs, and clear install/doctor preservation behavior, or explicitly document that the generated env file remains the only supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection shows current main still stages state-dir `.env` into generated `gateway.systemd.env` rather than supporting a direct state-dir `EnvironmentFile` mode.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs explicit schema, validation, docs, and doctor/install behavior rather than an ad hoc unit override.
