## Summary

Keep open: current main still hard-codes the generated `gateway.systemd.env` flow and has no supported config surface for using the state-dir `.env` directly. The related closed issues improved generated-file safety and doctor/update behavior, but this request remains a new Linux systemd service-environment contract that needs maintainer product and security judgment.

## What This Changes

The issue requests an opt-in Linux systemd gateway mode that uses the state-dir `.env` directly, or otherwise disables generation of `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate Linux systemd service-env contract with schema, validation for systemd-compatible env files, docs, and install/doctor preservation behavior; otherwise document that `gateway.systemd.env` remains the supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and lacks a supported direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and doctor/install semantics rather than an unsupported unit edit.
