## Summary

Keep open: current main and the latest tagged release still generate and reference `gateway.systemd.env`, and there is no supported config surface for choosing the state-dir `.env` or a custom systemd `EnvironmentFile`. The related fixes improved generated-file safety and update/doctor behavior, but this remains a new Linux service-environment contract that needs product and security review.

## What This Changes

Request an opt-in Linux systemd gateway mode that uses the state-dir `.env` or a configured file directly as the service `EnvironmentFile` instead of generating `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate opt-in Linux systemd service-env contract with schema, validation for systemd-compatible env files, docs, and install/doctor preservation behavior; otherwise document generated `gateway.systemd.env` as the only supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and has no supported config surface for direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and doctor/install semantics rather than preserving unsupported unit edits.
