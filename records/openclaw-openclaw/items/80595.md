## Summary

Keep open: current main and v2026.6.6 still use the generated Linux systemd `gateway.systemd.env` path, and the requested direct state-dir `.env` or custom `EnvironmentFile` mode is not implemented. The remaining question is a product and security decision about whether OpenClaw should add a new opt-in daemon environment contract.

## What This Changes

The issue requests an opt-in Linux systemd gateway mode that uses the state directory `.env` directly, or a custom `EnvironmentFile`, instead of always generating `gateway.systemd.env`.

## Best Possible Solution

Keep `gateway.systemd.env` as the safe default; if maintainers accept the opt-in mode, add schema/help/docs, installer and doctor preservation, validation for systemd-compatible values, and upgrade tests for both default and opt-in behavior.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for the reported current behavior: source and release inspection show systemd installs still generate and reference `gateway.systemd.env`. The requested direct `.env`/custom EnvironmentFile mode is absent from current schema and docs.

## Solution Assessment

Unclear until maintainers choose the service-environment contract. Direct `.env` mode is plausible, but it is only the best fix if it preserves the current generated-file safety checks, docs, and upgrade/doctor behavior.
