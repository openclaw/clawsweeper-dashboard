## Summary

Keep open: current main still uses the generated `gateway.systemd.env` path and there is no supported Linux systemd config surface for selecting the state-dir `.env` directly. The related closed fixes made generated-file handling safer, but this remains a new service-environment contract that needs maintainer product and security judgment.

## What This Changes

Request a supported opt-in Linux systemd mode that uses the state-dir `.env` or a custom path directly as the gateway `EnvironmentFile` instead of generating `gateway.systemd.env`.

## Best Possible Solution

Define an explicit Linux systemd service-env contract with schema, docs, systemd-compatible validation, and install/doctor preservation semantics while keeping generated `gateway.systemd.env` as the default.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and has no supported direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and doctor/install semantics rather than an unsupported unit edit.

## Risks / Open Questions

- Direct state-dir `.env` mode could bypass the current generated-file filtering/provenance unless validation, dangerous-env filtering, and drift semantics are designed up front.
- Systemd `EnvironmentFile` syntax is not identical to common dotenv syntax, so multiline values, quoting, optional-file behavior, and precedence need an explicit user contract.
- The adjacent provenance work in https://github.com/openclaw/openclaw/pull/81019 may affect the implementation boundary, so this should be aligned rather than patched independently.
