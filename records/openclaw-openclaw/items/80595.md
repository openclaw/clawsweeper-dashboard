## Summary

Keep open: current main and the latest tagged release still use the generated `gateway.systemd.env` path, and the requested direct state-dir `.env` mode would add a new Linux systemd service-environment configuration contract rather than fix a broken existing contract.

## What This Changes

The issue requests an opt-in Linux systemd mode that lets operators use the state-dir `.env` directly as the gateway service `EnvironmentFile` instead of the generated `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate opt-in Linux systemd env-file contract, including config schema, systemd-compatible validation, docs, and install/doctor behavior; otherwise document that generated `gateway.systemd.env` remains the supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and has no supported direct state-dir `.env` selector.

## Solution Assessment

Unclear until maintainers choose the contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and doctor/install semantics rather than an unsupported unit edit.
