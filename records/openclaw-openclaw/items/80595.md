## Summary

Keep open: current main and the latest tagged release still use the generated `gateway.systemd.env` model, and there is no supported `gateway.systemd.environmentFile` or equivalent opt-out config. The related closed reports improved safety around the generated file, but they do not implement this explicit operator mode.

## What This Changes

The issue requests an opt-in Linux systemd mode to use the state-dir `.env` directly, or disable generation of `gateway.systemd.env`, for the gateway service `EnvironmentFile`.

## Best Possible Solution

Add a deliberate Linux systemd service-environment config contract with validation, docs, tests, and install/doctor preservation semantics, or explicitly document that the generated env file remains the only supported OpenClaw-managed model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still renders the generated `gateway.systemd.env` behavior and lacks the requested config surface.

## Solution Assessment

Unclear until maintainers choose the service-environment contract. A direct `EnvironmentFile` option is plausible, but the maintainable path needs schema/help/docs alignment plus systemd-compatible `.env` validation and doctor/install behavior.
