## Summary

Keep open: current main and v2026.5.12 still use the generated `gateway.systemd.env` path with no supported config for selecting the state-dir `.env` directly. The related closed items improved generated-file safety and update/doctor behavior, but this remains a new Linux service-environment configuration contract with security and product implications.

## What This Changes

Request explicit Linux systemd support for using the state-dir `.env` or a custom env file instead of generating `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate Linux systemd service-env contract with schema, validation, docs, and install/doctor preservation behavior, or explicitly document generated `gateway.systemd.env` as the only supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and lacks a supported direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and doctor/install semantics rather than an unsupported unit edit.
