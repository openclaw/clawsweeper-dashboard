## Summary

Keep open: current main and the latest release still use the managed `gateway.systemd.env` path, while this issue asks for a new opt-in Linux systemd environment mode that needs maintainer product and security approval.

## What This Changes

The issue requests an opt-in Linux systemd mode that uses the state-dir `.env` directly, or avoids generating `gateway.systemd.env`, while keeping the generated file as the default.

## Best Possible Solution

Maintainers should decide whether to add a narrow opt-in direct-dotenv mode; if accepted, keep `gateway.systemd.env` as the default and add schema/docs, install and doctor preservation, validation, and upgrade coverage.

## Reproduction Assessment

Yes for the current source behavior: current main still stages dotenv-backed values through `gateway.systemd.env`, and the schema has no opt-in direct-dotenv option. A live Linux systemd run was not executed in this read-only review.

## Solution Assessment

Unclear until maintainers choose the service-environment contract. The requested direct `.env` mode is plausible as an explicit operator opt-in, but the generated-file default is safer unless equivalent validation and repair behavior are designed.
