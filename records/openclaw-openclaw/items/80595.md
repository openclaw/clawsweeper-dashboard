## Summary

Keep open: current main still uses the generated `gateway.systemd.env` flow for Linux systemd installs and has no supported config surface to point the unit directly at the state-dir `.env`. The related closed reports improved the generated-file behavior and update-time preservation, but they do not implement this opt-in operator mode.

## What This Changes

Request support for Linux systemd installs to use the state-dir `.env` directly as the gateway service `EnvironmentFile`, or to disable generating `gateway.systemd.env`.

## Best Possible Solution

Add a deliberate Linux systemd config contract with validation for systemd-compatible `.env` values and clear install/doctor preservation semantics, or document that the generated env file remains the only supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection shows current main still renders the generated `gateway.systemd.env` path and has no supported option for direct state-dir `.env` use.

## Solution Assessment

Unclear until maintainers choose the service-environment contract. A direct `EnvironmentFile` mode is plausible, but the maintainable solution needs schema, validation, docs, and doctor/install behavior rather than an unsupported unit override.
