## Summary

Keep open: current main still uses the generated `gateway.systemd.env` path for Linux systemd installs, and there is no supported config surface for opting into the state-dir `.env` as the unit `EnvironmentFile`. The related closed reports made the generated-file path safer, but they do not implement this requested operator mode.

## What This Changes

Request an explicit Linux systemd gateway service mode that uses the state-dir `.env` directly as `EnvironmentFile` or disables generation of `gateway.systemd.env`.

## Best Possible Solution

Add a deliberate Linux systemd config contract with validation for systemd-compatible `.env` values and clear install/doctor preservation semantics, or document that the generated env file remains the supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection and existing tests show current main still renders the generated `gateway.systemd.env` behavior the issue describes.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs an explicit schema, validation, docs, and doctor/install behavior rather than an ad hoc unit override.
