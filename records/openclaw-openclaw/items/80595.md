## Summary

Keep open: current main and the latest tagged release still use the generated `gateway.systemd.env` path, and there is no supported config surface for selecting the state-dir `.env` directly. The related closed issues improved generated-file preservation and update-time unit rewriting, but this remains a new Linux service environment contract that needs maintainer product and security judgment.

## What This Changes

Request an opt-in Linux systemd gateway mode that renders the state-dir `.env` or a configured path directly as `EnvironmentFile` instead of generating `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate Linux systemd service-env contract with schema, validation, docs, install rendering, and doctor preservation semantics; otherwise explicitly document the generated `gateway.systemd.env` model as the supported path.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection on current main confirms OpenClaw still generates `gateway.systemd.env` and lacks a supported direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, systemd-compatible validation, docs, and doctor/install behavior rather than preserving an unsupported unit edit.
