## Summary

Keep open: current main and the latest release still use the generated `gateway.systemd.env` path, and there is no supported `gateway.systemd` config surface for using the state-dir `.env` directly. This remains a new Linux systemd service-environment contract that needs maintainer product and security judgment.

## What This Changes

The issue requests an opt-in Linux systemd mode that uses the state-dir `.env` directly, or otherwise disables generation of `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate Linux systemd service-env contract with schema, validation for systemd-compatible env files, docs, install/doctor preservation behavior, and tests; otherwise document generated `gateway.systemd.env` as the only supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and has no supported direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and install/doctor semantics rather than an unsupported unit edit.
