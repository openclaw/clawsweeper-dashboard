## Summary

Keep open: current main still generates and uses `gateway.systemd.env` for Linux systemd installs, and the requested opt-in config surface for using the state-dir `.env` directly is not implemented. The related closed reports improved preservation and repair behavior around the generated file, but this remains a new service-environment contract decision.

## What This Changes

Support an opt-in Linux systemd mode that uses the state-dir `.env` directly as `EnvironmentFile` or disables generation of `gateway.systemd.env`.

## Best Possible Solution

Maintainers should either add a deliberate Linux systemd config contract with systemd-compatible dotenv validation plus install/doctor preservation semantics, or explicitly document that the generated env file remains the only supported managed model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source and test inspection on current main show OpenClaw still renders the generated `gateway.systemd.env` behavior described in the issue.

## Solution Assessment

Unclear until maintainers choose the supported service-environment contract. A direct `EnvironmentFile` option is plausible, but the maintainable path needs schema, validation, docs, and doctor/install semantics rather than an unsupported unit override.
