## Summary

Keep open: current main and v2026.6.10 still support the generated `gateway.systemd.env` contract only, while this issue asks for a new opt-in Linux systemd environment-source mode. The request is valid operator ergonomics, but it changes credential loading and config surface, so it needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests an explicit opt-in Linux systemd mode to use the state-dir `.env` directly as the gateway service `EnvironmentFile`, or otherwise disable generation of `gateway.systemd.env`.

## Best Possible Solution

Keep the generated env file as the safe default, and only add a documented opt-in direct state-dir `.env` mode if maintainers approve the credential-safety contract and validation rules.

## Reproduction Assessment

Not applicable as a bug repro, but the current behavior is source-reproducible: systemd staging writes and references `gateway.systemd.env`, and the config schema has no supported direct `.env` mode.

## Solution Assessment

Unclear until maintainers decide the product/security contract. The requested opt-in is plausible, but the safest default remains the generated env file with explicit validation before any direct state-dir `.env` mode exists.
