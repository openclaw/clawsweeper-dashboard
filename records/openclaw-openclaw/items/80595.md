## Summary

Keep open: current main and the latest tagged release still use the generated `gateway.systemd.env` path, and there is no supported config surface for choosing state-dir `.env` directly. This remains a new Linux systemd service-environment contract that needs maintainer product and security judgment.

## What This Changes

The issue requests an explicit Linux systemd gateway option to use the state-dir `.env` or another configured file as the service `EnvironmentFile` without regenerating `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate Linux systemd service-environment contract with schema, validation for systemd-compatible env files, install/doctor preservation behavior, tests, and docs; otherwise document that generated `gateway.systemd.env` remains the supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and lacks a supported direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, tests, and doctor/install semantics rather than an unsupported unit edit.
