## Summary

Keep open: current main and the latest tagged release still use the generated `gateway.systemd.env` path, and there is no supported config surface for choosing the state-dir `.env` directly. The related closed issues improved generated-file safety and doctor/update behavior, but this remains a new service-environment configuration contract that needs maintainer product and security judgment.

## What This Changes

Requests an explicit Linux systemd gateway configuration mode that uses the state-dir `.env` or a custom env file directly as the service `EnvironmentFile` instead of generating `gateway.systemd.env`.

## Best Possible Solution

Define a deliberate Linux systemd service-env contract, including schema, validation for systemd-compatible env files, docs, and install/doctor preservation behavior; otherwise document that generated `gateway.systemd.env` remains the supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection confirms current main still generates `gateway.systemd.env` and has no supported config surface for direct state-dir `.env` mode.

## Solution Assessment

Unclear until maintainers choose the supported contract. A direct `EnvironmentFile` option is plausible, but the maintainable solution needs schema, validation, docs, and doctor/install semantics rather than an unsupported unit edit.
