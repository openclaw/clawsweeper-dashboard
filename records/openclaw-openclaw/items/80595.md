## Summary

Keep open: current main and the latest release still use the generated `gateway.systemd.env` path, and live related-item search did not find a canonical item that owns the requested opt-in direct `.env` mode. ## What This Changes

The issue requests a supported opt-in Linux systemd gateway mode that uses the state-dir `.env` directly as the unit `EnvironmentFile` and stops regenerating `gateway.systemd.env`.

## Best Possible Solution

Keep this open for maintainer product/security review; if accepted, add an explicit opt-in with schema/help/docs, install and doctor preservation behavior, validation for systemd-incompatible dotenv content, and upgrade tests while preserving `gateway.systemd.env` as the default.

## Reproduction Assessment

Yes for the current behavior from source and tests: current main still writes dotenv-backed values to `gateway.systemd.env` and references that generated file. This is not a bug reproduction; it is a request for a new supported opt-in mode.

## Solution Assessment

Unclear without maintainer product/security direction. The proposed direct `.env` mode is plausible, but the current generated-file default is intentionally safer unless an opt-in preserves equivalent validation and upgrade behavior.
