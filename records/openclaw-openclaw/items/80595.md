## Summary

Keep open: current main and the latest release still use the generated `gateway.systemd.env` path, and there is no supported config surface for selecting the state-dir `.env` as the Linux systemd `EnvironmentFile`. The related closed reports made the generated-file approach safer, but this asks for a new opt-in service-environment contract that needs maintainer product/security judgment.

## What This Changes

The issue requests an opt-in Linux systemd mode that uses the state directory `.env` directly as the gateway service `EnvironmentFile`, or disables generation of `gateway.systemd.env`.

## Best Possible Solution

Add a deliberate Linux systemd config contract with schema, docs, validation for systemd-compatible `.env` values, and install/doctor preservation semantics, or explicitly document that the generated env file remains the only supported model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection and tests show current main still renders a generated `gateway.systemd.env` path and lacks a config option for direct state-dotenv mode.

## Solution Assessment

Unclear until maintainers choose the supported service-environment contract. A direct `EnvironmentFile` option is plausible, but it needs validation, schema/docs, and doctor/install behavior rather than an ad hoc unit override.
