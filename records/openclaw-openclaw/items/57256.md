## Summary

Keep open. Current main partially avoids the old human `status --deep` wording by rendering skipped memory inspection as `not checked`, but it still does not expose live gateway memory runtime health separately from embedding readiness, and the open fork PR that uses closing syntax is still conflicting and unmerged.

## What This Changes

The issue asks for `openclaw status --deep` to stop reporting an active memory plugin as unavailable when the live gateway is already using that plugin successfully.

## Best Possible Solution

Define and land a maintainer-approved live-gateway memory status contract that distinguishes runtime health from embedding readiness, then coordinate any remaining passive local-status side-effect work through https://github.com/openclaw/openclaw/issues/62599.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection shows current main still lacks a live runtime-health field/fallback and still derives gateway memory probe readiness from embedding state, while the exact old human `unavailable` wording has been softened to `not checked` on the fast/deep status row.

## Solution Assessment

No, current main is not the complete solution. The maintainable fix is a clear gateway doctor/status contract for runtime health versus embedding readiness, plus focused human and JSON status coverage.
