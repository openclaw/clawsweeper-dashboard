## Summary

Keep open. Current main and latest release do not implement the live-gateway memory-health fallback requested here, and an open cross-repository PR still uses closing syntax for this issue, so the issue should stay open until that PR is reviewed, replaced, or closed.

## What This Changes

The issue requests that `openclaw status --deep` stop reporting a live, working memory plugin as unavailable and instead reflect the gateway-backed memory runtime health.

## Best Possible Solution

Review or replace the linked live-gateway fallback work so human and JSON deep status can distinguish healthy live memory runtime from embedding readiness, while coordinating the broader passive-status side-effect question in https://github.com/openclaw/openclaw/issues/62599.

## Reproduction Assessment

No, not as a high-confidence live reproduction in this read-only pass. Source inspection does show the remaining path: current main lacks the live-gateway memory status fallback and still has doctor/gateway readiness tied to embedding state rather than runtime health.

## Solution Assessment

No, current main is only a partial mitigation. The maintainable fix should make live gateway memory runtime health explicit for status output, or choose a clear passive local status contract and update the linked PR/related issue accordingly.
