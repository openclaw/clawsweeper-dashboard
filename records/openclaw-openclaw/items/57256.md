## Summary

Keep open. Current main still does not make `openclaw status --deep` trust live gateway memory runtime health; it only avoids the old fast-row `unavailable` wording by rendering skipped memory inspection as `not checked`, while the behavior-fix PRs in the target repo are closed unmerged.

## What This Changes

Make `openclaw status --deep` report an enabled memory-slot plugin as healthy when the live gateway memory runtime is working, instead of showing unavailable or not checked.

## Best Possible Solution

Land a focused status/doctor fix that surfaces live gateway memory provider/runtime health for deep human and JSON status, while keeping embedding readiness separate and avoiding new config or broad plugin-loader policy changes.

## Reproduction Assessment

Yes, source-reproducible: current `status --deep` does not resolve a memory snapshot unless `--all` is set, so an enabled memory slot cannot be reported healthy from live gateway evidence. I did not run a live external memory backend in this read-only review.

## Solution Assessment

No, current main is only a partial mitigation because it changes skipped output to `not checked` but still does not use live gateway memory runtime health. The narrow maintainable fix is to wire live gateway provider/runtime evidence into deep status and JSON while separating embedding readiness.
