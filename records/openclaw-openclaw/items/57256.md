## Summary

Keep open. Current main partly mitigates the old fast-row wording by rendering skipped memory inspection as `not checked`, but `openclaw status --deep` still does not surface live gateway memory runtime health for a working external memory plugin.

## What This Changes

The issue asks `openclaw status --deep` and related doctor/status probes to report live memory-plugin health instead of `unavailable` when Mem0, Honcho, or another memory plugin is working in the gateway.

## Best Possible Solution

Land a focused status/doctor fix that surfaces live gateway memory provider/runtime health for deep human and JSON status, keeps embedding readiness separate, and avoids broad plugin-loader policy changes.

## Reproduction Assessment

Yes, source-reproducible: current `status --deep` still returns no memory snapshot unless `--all` is set, so it cannot report healthy live gateway memory evidence. I did not run a live external memory backend in this read-only review.

## Solution Assessment

No, current main is only a partial mitigation because it changes skipped output to `not checked` but still does not use live gateway memory runtime health. The narrow maintainable fix is to wire gateway/provider runtime evidence into deep status and JSON while separating embedding readiness.
