## Summary

Keep this issue open. Current main partially mitigates the original misleading `unavailable` row by rendering skipped memory inspection as `not checked`, but `status --deep` still does not surface live gateway memory runtime health, and #59685 remains the broader open implementation candidate for the reported class of external memory-plugin false negatives.

## What This Changes

The issue requests that `openclaw status --deep` stop reporting active third-party memory plugins as unavailable when the live gateway runtime and direct memory operations are working.

## Best Possible Solution

Review #59685 against current main and either land a focused runtime-health fix for `status --deep` or explicitly document and close the PR if `not checked` plus `status --json --all` is the intended behavior.

## Reproduction Assessment

Yes, source-reproducible for the remaining gap: non-JSON `status --deep` still skips memory snapshot resolution and renders the active memory plugin as `not checked`, so it cannot report the live gateway plugin as healthy. I did not run a live Mem0/Honcho/Qdrant gateway reproduction.

## Solution Assessment

No, current main is not yet the complete fix requested here. The narrower maintainable path is to resolve the open #59685 review by landing or superseding its live runtime-health approach after accounting for the newer cold-registry fix.
