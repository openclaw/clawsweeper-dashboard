## Summary

Keep open. Current main has softened the old human status wording to `not checked`, but the gateway memory probe still derives readiness from embedding state instead of a separate live runtime-health signal, and the issue is still paired with an open conflicting fork PR that uses closing syntax. ## What This Changes

The issue asks for `openclaw status --deep` to stop reporting a working live gateway memory plugin as unavailable and to reflect live plugin health accurately.

## Best Possible Solution

Land or replace a maintainer-approved live-gateway memory status contract that separates runtime health from embedding readiness, while coordinating CLI-local status side-effect work through https://github.com/openclaw/openclaw/issues/62599.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. No, current main is only a partial mitigation. The maintainable fix needs a clear gateway doctor/status contract for runtime health versus embedding readiness, plus focused human and JSON status coverage.
