## What This Changes

The issue asks for `openclaw status --deep` to stop reporting a working external memory plugin as unavailable when the live Gateway runtime is healthy.

## Best Possible Solution

Review, port, or replace the linked status-memory PR with a focused Gateway-backed runtime-health fallback for deep human and JSON status, with embedding readiness kept separate and covered by regression tests.

## Reproduction Assessment

Yes, source-reproducible: current `status --deep` still resolves memory only when `opts.all` is set, and the typed status result has no Gateway memory-runtime fallback. I did not run a live external memory backend in this read-only review.

## Solution Assessment

No, current main is only a partial mitigation because it can render skipped memory inspection as `not checked` while still not surfacing healthy live Gateway memory state. The narrower maintainable fix is to wire Gateway memory provider/runtime evidence into deep status and JSON while keeping embedding readiness separate.
