## Summary

Keep this open. Current main has a shipped mitigation that avoids some misleading fast-status output, but it still does not implement the requested live-gateway memory health fallback for `openclaw status --deep`, and #59685 remains open in the target repo as the broader implementation candidate.

## What This Changes

The issue asks for `openclaw status --deep` to report an active external memory plugin as healthy when the live gateway is already using that plugin successfully.

## Best Possible Solution

Land or explicitly supersede #59685 with a maintained implementation that makes `status --deep` and JSON deep status use the live gateway memory runtime/provider when the command-local memory snapshot cannot materialize, while keeping the shipped fast-path `not checked` behavior for ordinary status output.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: configure an external memory slot, have the local status memory manager return null, and run deep status; current main does not call a live gateway memory fallback and therefore cannot show the active provider.

## Solution Assessment

Unclear. The current shipped mitigation is useful but narrower than the issue; the best fix should be either the reviewed #59685 runtime-health split or a smaller gateway-provider fallback that covers both human and JSON deep status without reintroducing heavyweight fast-status probes.
