## Summary

Keep this issue open: current main still resolves a fresh same-key session id before any per-key reservation step, and the direct implementation candidate is an open, unmerged PR.

## What This Changes

The issue reports that concurrent OpenAI-compatible HTTP requests with the same `x-openclaw-session-key` can fork into isolated session ids and lose memory scope.

## Best Possible Solution

Keep this issue open while maintainers review or land https://github.com/openclaw/openclaw/pull/85404; close it after that or an equivalent same-key reservation fix reaches main.

## Reproduction Assessment

Yes. Current main shows the session id is minted before any reservation step, and two first-use same-key callers can each hit the fresh `crypto.randomUUID()` path before a session entry exists.

## Solution Assessment

Yes. A narrow same-key reservation or re-resolution in the agent-command session identity path is the best fix because it covers both `/v1/chat/completions` and `/v1/responses` callers without pushing queuing into every client.
