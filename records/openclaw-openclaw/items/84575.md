## What This Changes

The issue asks `/v1/chat/completions` to serialize concurrent requests with the same `x-openclaw-session-key` so the second turn keeps the same session memory and context.

## Best Possible Solution

Add a current-main regression for two concurrent same-key Chat Completions requests, then move the serialization boundary so session resolution, session-store preparation, and embedded execution share the same per-session lane.

## Reproduction Assessment

No high-confidence live current-main reproduction yet. Source inspection and the reporter’s embedded-runtime logs show a plausible same-key race around pre-lane session preparation, but no failing current-main test proves the second run enters a different memory scope.

## Solution Assessment

Unclear. Per-session serialization matches the documented queue contract, but the safer fix is to prove whether session resolution/persistence before `runEmbeddedPiAgent` is the leaking boundary before adding another endpoint-level queue.
