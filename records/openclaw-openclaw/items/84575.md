## Summary

Keep this issue open: current main still resolves a fresh same-key session id before any reservation step, and the specific fix is owned by an open, unmerged linked PR rather than by shipped or current-main code.

## What This Changes

The issue requests that concurrent OpenAI-compatible HTTP requests with the same `x-openclaw-session-key` share one session/memory scope instead of forking an isolated turn.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85404, or an equivalent reservation fix, then close this issue from the merged implementation.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a new random session id before the per-session execution lane, so two fresh same-key requests can fork before either mapping is persisted. I did not run a live repro in this read-only pass, but the reporter's gateway logs and the linked PR's regression proof align with the source path.

## Solution Assessment

Yes, the linked reservation-helper approach is the narrowest maintainable fix found: it keeps `resolveSession` synchronous, reserves only brand-new same-key sessions in the async caller, and avoids endpoint-specific queuing. The issue should stay open until that fix or an equivalent lands.
