Keep open. The report is specific and plausible for older OpenClaw, but current main has downstream embedded-run session-key serialization while the OpenAI-compatible endpoint lacks a focused same-key in-flight regression or current-main live trace, so this is not safe cleanup-close material.

## What This Changes

Serialize or otherwise handle concurrent `/v1/chat/completions` requests that reuse the same `x-openclaw-session-key` so follow-up turns do not observe an isolated memory scope.

## Best Possible Solution

Add focused current-main proof for two concurrent same-key OpenAI HTTP requests, then either confirm the existing runtime lane covers the endpoint or add a bounded endpoint/runtime serialization path for the missing runtime.

## Solution Assessment

Unclear. Per-session queuing matches the embedded runner’s documented contract for human chat, but maintainers should decide how OpenAI HTTP should handle queueing, busy responses, CLI/ACP runtimes, and intentional same-key parallel callers.
