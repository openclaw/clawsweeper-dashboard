## What This Changes

The issue requests per-session-key serialization for OpenAI-compatible HTTP requests so concurrent same-key turns do not fork into isolated session memory scope.

## Best Possible Solution

Rebase and repair https://github.com/openclaw/openclaw/pull/85404, or land an equivalent agent-command session reservation fix, then close this issue from the merged implementation.

## Reproduction Assessment

Yes. Current main resolves a new random session id before any same-key reservation, so two fresh concurrent same-key requests can fork before either mapping is persisted; I did not run a live HTTP repro in this read-only pass.

## Solution Assessment

Yes, the linked reservation-at-agent-command approach is the best available shape because it preserves resolveSession as a synchronous helper and fixes identity before endpoint-specific queuing; it still needs to land or be replaced before this issue can close.
