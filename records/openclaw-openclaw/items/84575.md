## Summary

Keep this issue open: the reporter’s follow-up rules out the earlier CLI-runtime suspicion, and current source shows a plausible embedded-path race because `/v1/chat/completions` reaches session resolution and metadata preparation before the embedded per-session lane is entered. This still needs a focused current-main regression before choosing the exact serialization point.

## What This Changes

The issue reports that concurrent `/v1/chat/completions` requests with the same `x-openclaw-session-key` can lose shared session memory and asks for same-key request serialization.

## Best Possible Solution

Add a current-main regression for two in-flight same-key OpenAI-compatible requests, then move same-session serialization far enough up the call path that session resolution, metadata persistence, and context assembly happen after any active turn for that key has completed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: source inspection plus the reporter's embedded-runtime logs make the race plausible, but a focused current-main test holding one same-key HTTP request in flight has not been run.

## Solution Assessment

Is this the best way to solve the issue? The requested serialization direction is right, but the implementation should be pinned by a regression first so it covers pre-lane session preparation without adding a second broad queue around already-serialized embedded execution.
