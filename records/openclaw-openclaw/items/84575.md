## Summary

Keep this issue open: current main still resolves first-use same-key HTTP requests through `resolveSession` before any per-key reservation step, and the direct implementation candidate remains open and unmerged.

## What This Changes

The issue reports that concurrent OpenAI-compatible HTTP requests sharing an `x-openclaw-session-key` can fork into separate session identities and lose session memory.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85404 or an equivalent same-key session reservation fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Current main resolves the HTTP request's session key into `agentCommandFromIngress`, then mints a new session id in `resolveSession` before any per-key reservation step, so the first-use same-key race is source-reproducible.

## Solution Assessment

Yes. A narrow reservation or re-resolution in the shared agent-command session identity path is the best fix because it covers `/v1/chat/completions` and `/v1/responses` without pushing serialization into every client.
