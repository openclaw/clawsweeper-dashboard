## Summary

Keep open: the PR targets a valid Google Chat SecretRef inspection bug and the supplied live proof is sufficient, but the latest patch still has merge-blocking review findings around degraded-status detection and env-dependent test isolation.

## What This Changes

The PR adds a Google Chat read-only account inspector for SecretRef-backed serviceAccount/serviceAccountRef credentials, wires it into Google Chat config adapters, refactors credential inspection, adds tests, and updates the changelog.

## Best Possible Solution

Land the plugin-local Google Chat inspector after it reports unavailable SecretRefs through the status/audit fields the host already recognizes and after the new tests are isolated from host GOOGLE_CHAT_* env vars.

## Reproduction Assessment

Yes at source level. Current main lacks a Google Chat config.inspectAccount adapter, the read-only host boundary depends on plugin inspectors, and the current Google Chat runtime resolver throws on serviceAccount/serviceAccountRef SecretRefs.

## Solution Assessment

No as currently patched. A Google Chat plugin-owned inspect adapter is the right product direction, but it must expose degraded credential state through the existing status/audit detection path and keep the regression test hermetic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

  - body: `inspectGoogleChatAccount` reports SecretRefs only as `credentialSource: "configured_unavailable"`, but the host degraded-status checks only recognize `tokenStatus`/bot/app/signing/user status fields. Status and audit rows can therefore treat a SecretRef-backed Google Chat account as OK and omit the documented "secret unavailable" warning; add a recognized status field or extend the shared helper with coverage.
  - confidence: 0.86

On machines with those vars set, this assertion returns `env` instead of `none`, so stub or clear them for the test and restore afterward.
  - confidence: 0.9
