## Summary

Keep open: current main does not yet route Feishu SDK REST/token or WebSocket bootstrap HTTP through OpenClaw's proxy-aware agent path, and this PR remains the active candidate fix for the linked Feishu proxy failure. The patch looks focused and the supplied terminal/runtime proof is sufficient, but merging still needs explicit maintainer acceptance of the Feishu proxy-routing and managed-proxy security contract.

## What This Changes

The PR replaces Feishu's timeout-only Lark SDK HTTP wrapper with a proxy-aware wrapper, passes it to both Lark Client and WSClient bootstrap, and adds tests for ambient proxy, caller-agent preservation, and managed-proxy override behavior.

## Best Possible Solution

Land only after maintainers accept the Feishu policy that SDK HTTP/token and WS bootstrap follow ambient proxy routing, while managed proxy mode overrides request agents; keep per-channel direct/bypass policy as separate follow-up work.

## Reproduction Assessment

Yes, at source level. Current main uses a timeout-only Feishu SDK HTTP wrapper and no WS bootstrap `httpInstance`, while the Lark SDK uses `httpInstance` for ordinary SDK requests and WS `pullConnectConfig`.

## Solution Assessment

Yes, with maintainer policy acceptance. Reusing the Lark SDK `httpInstance` seam inside the Feishu plugin is the narrow owner-boundary fix; a channel-level direct/proxy mode is broader follow-up for mixed networks.
