## Summary

Keep open: current main still lacks the proxy-aware Feishu SDK HTTP path, and this branch remains the active focused fix candidate, but merging it intentionally changes credentialed Feishu egress behavior and needs maintainer acceptance rather than cleanup closure.

## What This Changes

This PR replaces Feishu's timeout-only Lark SDK HTTP wrapper with a proxy-aware wrapper, wires it into both Lark Client and WSClient bootstrap, and adds client tests for ambient proxy, caller-agent preservation, and managed-proxy override.

## Best Possible Solution

Accept or reject the Feishu proxy egress policy explicitly; if accepted, land this focused httpInstance fix and track broader per-channel direct/proxy controls separately.

## Reproduction Assessment

Yes, at source level. Current main uses a timeout-only Feishu SDK HTTP wrapper and no WS bootstrap httpInstance, while the pinned Lark SDK uses httpInstance for tenant-token requests and WS pullConnectConfig; the PR body also includes live terminal proof from an equivalent installed-package hotfix.

## Solution Assessment

Yes, with maintainer policy acceptance. Reusing the Lark SDK httpInstance seam inside the Feishu plugin is the narrow owner-boundary fix, while a channel-level direct/proxy mode is broader follow-up work for mixed-network setups.
