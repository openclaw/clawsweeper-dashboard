## Summary

Keep open: this PR is still a focused candidate for the linked Feishu proxy failure because current main and v2026.6.10 still lack proxy-aware Lark SDK HTTP/WS bootstrap wiring, but merge needs maintainer acceptance of the Feishu egress/security contract.

## What This Changes

The PR replaces Feishu's timeout-only Lark SDK HTTP wrapper with a proxy-aware wrapper and passes it to both Lark Client REST/token calls and WSClient bootstrap requests.

## Best Possible Solution

Land only after maintainers explicitly accept the Feishu egress policy for SDK REST/token HTTP and WS bootstrap, while tracking broader per-channel direct/bypass policy separately.

## Reproduction Assessment

Yes, at source level. Current main passes only a timeout wrapper to Lark.Client and no httpInstance to WSClient bootstrap, while the Lark SDK uses the injected httpInstance for tenant-token and pullConnectConfig requests.

## Solution Assessment

Yes as a narrow code seam, but only with maintainer policy acceptance. Reusing the Lark SDK httpInstance inside the Feishu plugin is the maintainable owner-boundary fix; the direct-versus-proxy default remains a product/security decision.
