## Summary

Keep open. Current main still has the reported provider-string gate, so an Anthropic-resolved session with an existing Claude CLI binding and nonempty local transcript returns local history before reading or merging the underlying Claude CLI JSONL.

## What This Changes

Allow chat history CLI-session imports to merge for Anthropic-routed sessions that have a Claude CLI session binding instead of only `claude-cli` provider sessions.

## Best Possible Solution

Update the helper contract so Claude CLI imports are merged when the session entry has a valid Claude CLI binding for Anthropic-selected CLI-runtime sessions, and add a focused regression test for `provider: "anthropic"` with nonempty local messages. Preserve the existing merge/dedupe behavior and avoid changing unrelated CLI backend formats.

## Reproduction Assessment

Yes. A deterministic unit reproduction is to call `augmentChatHistoryWithCliSessionImports` with `entry.cliSessionBindings["claude-cli"]`, `provider: "anthropic"`, a valid Claude JSONL home, and at least one local message; current main returns the local array before reading the JSONL.

## Solution Assessment

Mostly yes. Generalizing the safety check from provider string to the presence of a CLI binding is the right direction, but the patch should document or test the intended behavior for stale bindings when the current provider is unrelated to Anthropic/Claude CLI.
