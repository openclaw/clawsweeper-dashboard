## Summary

Keep this issue open: current main still re-applies Tailscale Serve during gateway startup without a startup cleanup step, and several open fixing PRs already target this issue. The issue should remain the canonical bug until a maintainer reviews and lands one fix path.

## What This Changes

This issue reports that `gateway.tailscale.mode: "serve"` accumulates redundant Tailscale Serve entries across gateway restarts when startup re-applies Serve without clearing stale entries.

## Best Possible Solution

Keep this issue as the canonical bug while maintainers choose or repair one linked PR that makes Serve startup idempotent without unnecessarily clearing operator-managed routes.

## Reproduction Assessment

Yes for a source-level reproduction path, but not a live local reproduction here. Current main re-applies Serve on startup and defaults `resetOnExit` to false, matching the reporter's restart steps; proving duplicate daemon entries still requires a real Tailscale setup.

## Solution Assessment

No, the suggested unconditional reset is only a plausible fix. The best fix should clear stale OpenClaw-owned Serve state before re-applying Serve while preserving `serviceName` and `preserveFunnel` semantics and addressing the risk to manually configured Serve routes.
