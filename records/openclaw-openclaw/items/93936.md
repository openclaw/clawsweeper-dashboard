## Summary

Keep open: current main and the latest release still re-apply Tailscale Serve at gateway startup without a startup reset or scoped clear, and several open closing PRs already target this issue.

## What This Changes

The issue reports that gateway startup with Tailscale Serve appends redundant Serve entries across restarts because startup does not clear prior OpenClaw-managed Serve state first.

## Best Possible Solution

Review and land one narrow linked PR that makes Serve startup idempotent, uses scoped cleanup when `serviceName` is configured, and documents or proves the unscoped reset behavior for existing users.

## Reproduction Assessment

Yes from source, with medium confidence: configure Serve mode, start and restart the gateway, then compare `tailscale serve status`; current main re-applies Serve without startup cleanup. I did not run the live Tailscale restart loop because this checkout has no `tailscale` binary available.

## Solution Assessment

No for current main: it still re-applies Serve without startup cleanup. The reset-before-serve direction is plausible, but the best fix should handle scoped cleanup and external Serve entries deliberately before merge.
