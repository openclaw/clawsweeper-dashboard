## Summary

Keep open: current main and the latest release still enable Tailscale Serve at gateway startup without first clearing existing Serve routes, and several open PRs are linked as candidate fixes. This issue should remain the canonical tracker until one reviewed fix lands.

## What This Changes

The issue asks OpenClaw to stop accumulating redundant Tailscale Serve entries when the gateway starts in `gateway.tailscale.mode: "serve"`.

## Best Possible Solution

Land one narrow fix that performs best-effort startup cleanup before OpenClaw re-applies Serve routes, preserves the `preserveFunnel` skip path, covers service-name and unscoped modes in tests, and records any reset compatibility tradeoff in the PR.

## Reproduction Assessment

Yes, source-reproducible with medium confidence: current main enters serve mode and calls `enableTailscaleServe` before any startup cleanup. A live Tailscale restart/status loop was not run in this read-only review.

## Solution Assessment

Yes directionally: startup cleanup in `startGatewayTailscaleExposure` is the narrow owner-boundary fix because the infra helper already owns the reset/clear commands. The exact PR still needs normal review for reset scope and live behavior proof.
