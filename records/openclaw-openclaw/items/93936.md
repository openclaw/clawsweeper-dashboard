## Summary

Keep open: current main and v2026.6.9 still re-apply Tailscale Serve on gateway startup without a startup pre-cleanup, and multiple open candidate PRs already target the bug. The fix shape needs maintainer review because an unscoped `tailscale serve reset` can remove unrelated operator-managed Serve entries.

## What This Changes

The issue reports that gateway Serve mode re-adds Tailscale Serve routes on each startup without clearing stale OpenClaw entries, causing duplicate or stale routes and failed Tailscale access.

## Best Possible Solution

Land one narrow fix that makes Serve startup idempotent while preserving operator-managed Serve and Funnel routes, with scoped cleanup where possible plus regression coverage and live Tailscale proof when feasible.

## Reproduction Assessment

Yes, there is a clear source-backed reproduction path: configure Serve mode, start and restart the gateway, then compare `tailscale serve status`. I did not execute a live Tailscale daemon here, so this remains source-reproducible rather than live-reproduced.

## Solution Assessment

No, current main is not the best solution because it still re-applies Serve without startup cleanup. The requested reset-before-serve direction is plausible, but an unqualified global reset is not the best fix because it can clear unrelated Serve entries.
