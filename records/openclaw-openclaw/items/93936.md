## Summary

Keep open: current main and the latest release still call Tailscale Serve setup on every gateway startup without a startup reset or scoped clear, while `resetOnExit` defaults to false and several linked fix PRs remain unmerged.

## What This Changes

The issue asks OpenClaw to prevent Tailscale Serve entries from accumulating when the gateway restarts in serve mode.

## Best Possible Solution

Review and land one narrow fix that makes Serve startup idempotent, preferably scoped to `serviceName` when configured and backed by regression coverage plus live Tailscale proof when feasible.

## Reproduction Assessment

Yes, there is a clear source-backed reproduction path: configure Serve mode, start and restart the gateway, then compare `tailscale serve status`. I did not execute a live Tailscale daemon here, so the review is source-reproducible rather than freshly live-reproduced.

## Solution Assessment

No, current main is not the best solution because it still re-applies Serve without startup cleanup. The requested reset-before-serve direction is plausible, but the final fix should handle scoped cleanup and external Serve entries deliberately.
