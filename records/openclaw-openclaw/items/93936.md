## Summary

Keep open. Current main and the latest release still start Tailscale Serve without a startup reset, and this issue has multiple open closing PRs that should be reviewed rather than closing the canonical user report ahead of a merge.

## What This Changes

Issue reports that gateway startup in Tailscale Serve mode accumulates redundant Serve entries across restarts and asks for cleanup before reconfiguring routes.

## Best Possible Solution

Review and land one narrow startup cleanup PR that reuses the existing Serve disable/reset helper while preserving serviceName, preserveFunnel, and resetOnExit behavior.

## Reproduction Assessment

Yes from source: current main reaches `enableTailscaleServe` on Serve startup without a preceding reset, while cleanup is gated behind `resetOnExit`. I did not run a live Tailscale daemon restart loop in this read-only review.

## Solution Assessment

Yes in shape: the maintainable fix is bounded startup cleanup using existing `disableTailscaleServe` semantics. The selected PR still needs compatibility review for unscoped `serve reset` versus service-scoped cleanup.
