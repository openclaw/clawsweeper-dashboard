## Summary

Keep open: current main and the latest release still re-apply Tailscale Serve on gateway startup without a startup reset or scoped clear, and four open linked PRs are already candidate fixes for this report.

## What This Changes

The issue reports duplicate Tailscale Serve entries accumulating after gateway restarts when `gateway.tailscale.mode` is `serve`.

## Best Possible Solution

Review and land one narrow idempotent-Serve startup fix, with scoped cleanup for `serviceName` and an explicit maintainer decision for the no-service reset behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure Serve mode, start the gateway, restart it, and compare `tailscale serve status`. I did not execute a live Tailscale daemon in this read-only review, so this is source-reproducible rather than freshly live-reproduced.

## Solution Assessment

No, current main is not the best solution because it still re-applies Serve without startup cleanup. The reset-before-serve direction is plausible, but maintainers need to choose the upgrade-safe cleanup scope before one linked PR lands.
