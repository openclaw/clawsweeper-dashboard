## Summary

Keep open: current main and the latest release still apply Tailscale Serve on gateway startup without first clearing stale Serve state, and several open PRs already target this issue with closing references. This should remain the canonical issue until maintainers choose and land one compatibility-safe fix rather than creating another repair lane.

## What This Changes

The issue reports duplicate Tailscale Serve entries accumulating across gateway restarts and requests cleanup before Serve is reapplied.

## Best Possible Solution

Keep this issue as the canonical bug, then land one linked PR that clears only OpenClaw-owned Serve state by default or explicitly documents a maintainer-approved broad-reset policy, with focused tests and redacted live Tailscale status proof across a restart.

## Reproduction Assessment

Yes, source-reproducible with high confidence: configure Serve mode, start and restart the gateway, then compare `tailscale serve status`; current code reapplies Serve without startup cleanup and Tailscale documents background Serve state as persistent. I did not run a live Tailscale daemon in this read-only review.

## Solution Assessment

Not yet. Reset-before-serve is a plausible mitigation, but the best fix must avoid deleting unrelated operator-owned Serve routes by default or make that ownership policy explicit and tested.
