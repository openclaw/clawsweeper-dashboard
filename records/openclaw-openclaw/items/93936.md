## Summary

Keep open: current main and the latest release still re-apply Tailscale Serve on gateway startup without a startup reset or scoped clear, and multiple open closing PRs now own candidate fixes.

## What This Changes

The issue asks OpenClaw to prevent redundant Tailscale Serve entries from accumulating across gateway restarts in `gateway.tailscale.mode: "serve"`.

## Best Possible Solution

Land one canonical gateway/Tailscale fix that makes Serve startup idempotent while preserving unrelated operator-managed Serve routes by default, with live Tailscale before/after proof when feasible.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure Serve mode, start and restart the gateway, then compare `tailscale serve status`. I did not execute a live Tailscale daemon in this review.

## Solution Assessment

No, current main is not the best solution because it still re-applies Serve without startup cleanup. The fix direction should be narrowed to OpenClaw-owned or service-scoped cleanup unless maintainers explicitly accept broad reset behavior.
