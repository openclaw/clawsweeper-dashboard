## Summary

Keep this issue open: current main and the latest release still configure Tailscale Serve on startup without clearing stale Serve state first, and several open PRs already reference this issue as candidate fixes. The issue should stay open until maintainers choose and land one canonical fix path.

## What This Changes

The issue reports that gateway startup in `gateway.tailscale.mode: "serve"` accumulates redundant Tailscale Serve entries across restarts because stale entries are not cleaned up before reconfiguration.

## Best Possible Solution

Land one canonical fix that removes stale OpenClaw-owned Serve state, or deliberately documents and proves a broad reset policy, then close this issue after the fix merges.

## Reproduction Assessment

Yes, at source and reporter-step level, but not live-reproduced in this review. Current main calls `tailscale serve --bg --yes` without startup cleanup while `resetOnExit` defaults to false, and the issue gives concrete restart plus `tailscale serve status` steps.

## Solution Assessment

No, the suggested unconditional `tailscale serve reset` is only a plausible repair direction. The best fix should preserve unrelated operator-managed Serve routes by default or make broad cleanup an explicit maintainer-approved behavior with live proof.
