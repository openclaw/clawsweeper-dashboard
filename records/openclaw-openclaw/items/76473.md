## Summary

Keep open. Current main still lacks the requested high-frequency cron advisory, and the closed predecessor #38731 is superseded by this PR, but this patch is not merge-ready because the CLI warning goes to stdout and the new Gateway tests do not match the current handler context.

## What This Changes

The PR adds a shared high-frequency `every` schedule warning helper, emits warnings in CLI cron add/edit and Gateway cron add/update, adds regression tests, and records a changelog entry.

## Best Possible Solution

Rebase or replace this PR with a current-main patch that keeps short intervals valid, warns non-blockingly for CLI add/edit and Gateway add/update, preserves JSON stdout, and extends the existing cron validation test harness.

## Reproduction Assessment

Yes. Source inspection shows current main accepts `--every 15m` and Gateway `{ kind: "every", everyMs: 900000 }` schedules without any high-frequency warning path.

## Solution Assessment

No. The shared helper and add/edit/update parity are the right direction, but the CLI advisory should go to stderr and the Gateway tests must reuse or extend the current handler context before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit CLI warnings on stderr:** `src/cli/cron-cli/register.cron-add.ts:135`
  - body: `defaultRuntime.log` writes to stdout, and these cron commands also emit their JSON response on stdout. With a sub-30-minute schedule, the warning text can precede the JSON payload and make `openclaw cron add/edit ... --json` output unparsable; emit this advisory through `defaultRuntime.error` like the existing missing-agent warning.
  - confidence: 0.9
- **[P2] Build the gateway tests with the real handler context:** `src/gateway/server-methods/cron.validation.test.ts:4-13`
  - body: The new test context only provides `cron.add`, `cron.update`, and logging, but the current handlers call `context.getRuntimeConfig()`, and `cron.update` also calls `context.cron.getDefaultAgentId()` and `context.cron.getJob()`. These tests will throw before reaching the warning assertions; extend the existing `cron.validation` test context instead.
  - confidence: 0.92
