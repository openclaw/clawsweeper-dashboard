## Best Possible Solution

Rebase or replace this PR with a current-main patch that warns non-blockingly for sub-30-minute `every` schedules in CLI add/edit and Gateway add/update, preserves JSON stdout, and fits the existing cron validation tests.

## Reproduction Assessment

Yes. Source inspection shows current main accepts short `every` cron schedules without an advisory, and the PR's stdout and test-context failures are clear from the proposed code.

## Solution Assessment

No. The shared helper and add/edit/update parity are the right direction, but the CLI advisory must go to stderr and the Gateway tests must be rebased onto the current handler context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit CLI warnings on stderr:** `src/cli/cron-cli/register.cron-add.ts:135`
  - body: `defaultRuntime.log` writes to stdout, and these cron commands also emit their JSON response on stdout via `writeJson`. With a sub-30-minute schedule, the warning text will precede the JSON payload and make `openclaw cron add/edit ... --json` output unparsable; use `defaultRuntime.error` for this advisory like the existing missing-agent warning.
  - confidence: 0.9
- **[P2] Build the gateway tests with the real handler context:** `src/gateway/server-methods/cron.validation.test.ts:4-13`
  - body: The new test context only provides `cron.add`, `cron.update`, and logging, but the current handlers unconditionally call `context.getRuntimeConfig()` and `cron.update` also calls `context.cron.getDefaultAgentId()` and `context.cron.getJob()`. These tests will throw before reaching the warning assertions; extend the existing cron validation test context instead.
  - confidence: 0.92
