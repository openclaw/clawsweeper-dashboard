## Summary

Keep open. Current main still lacks the high-frequency cron warning, but this replacement PR is dirty against current main and the patch still needs repair because CLI warnings go to stdout and the new gateway tests use an incomplete handler context.

## What This Changes

The PR adds a shared high-frequency `every` schedule helper, CLI and Gateway warning calls for cron add/edit/update paths, a changelog entry, and focused regression tests.

## Best Possible Solution

Rebase or replace this PR with a current-main patch that keeps short intervals valid, warns non-blockingly for CLI add/edit and Gateway add/update, preserves JSON stdout, fits the existing cron validation tests, and either includes or explicitly tracks broader UI/agent-tool coverage.

## Reproduction Assessment

Yes. Source inspection shows current main accepts `--every 15m` and Gateway `{kind:"every", everyMs:900000}` schedules without a warning, and the PR's own diff has source-reproducible stdout and test-context failures.

## Solution Assessment

No. The shared helper and add/edit/update parity are the right direction, but the implementation should emit CLI advisories on stderr and rebase the gateway tests onto the current cron validation context before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit CLI warnings on stderr:** `src/cli/cron-cli/register.cron-add.ts:135`
  - body: `defaultRuntime.log` writes to stdout, and these cron commands also emit their JSON response on stdout via `writeJson`. With a sub-30-minute schedule, the warning text will be printed before the JSON payload and make `openclaw cron add/edit ... --json` output unparsable; use `defaultRuntime.error` for this advisory like the existing missing-agent warning.
  - confidence: 0.9
- **[P2] Build the gateway tests with the real handler context:** `src/gateway/server-methods/cron.validation.test.ts:4-13`
  - body: The new test context only provides `cron.add`, `cron.update`, and logging, but the current handlers unconditionally call `context.getRuntimeConfig()` and `cron.update` also calls `context.cron.getDefaultAgentId()` and `context.cron.getJob()`. These tests will throw before reaching the warning assertions; extend the existing `cron.validation` test context instead.
  - confidence: 0.92
