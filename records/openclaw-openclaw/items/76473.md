## Summary

Keep this PR open. Current main still lacks the high-frequency cron advisory, and this PR is the active replacement for the earlier closed implementation, but the branch needs a narrow repair before merge because the CLI warning can corrupt JSON stdout and the new Gateway tests do not match the current handler context.

## What This Changes

The PR adds a shared high-frequency `every` schedule warning and wires it into CLI cron add/edit plus Gateway cron add/update with tests and a changelog entry.

## Best Possible Solution

Land a current-main repair that keeps short cron intervals valid, emits non-blocking CLI advisories on stderr for add/edit, logs Gateway warnings for add/update, preserves JSON stdout, and adds focused regression coverage using the existing cron validation helpers.

## Reproduction Assessment

Yes. Source inspection shows current main accepts CLI `--every 15m` and Gateway `{ kind: "every", everyMs: 900000 }` schedules without any high-frequency advisory path.

## Solution Assessment

No. The shared helper and add/edit/update parity are the right direction, but this implementation should move CLI warnings to stderr and rework the Gateway tests around the current handler context before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit CLI cron advisories on stderr:** `src/cli/cron-cli/register.cron-add.ts:135`
  - body: The new warning uses `defaultRuntime.log`, which writes to stdout, while cron add/edit also emit their response with `writeJson`. For a sub-30-minute schedule, the warning can be printed before the JSON payload and make machine-readable output unparsable; route this advisory through `defaultRuntime.error` for both add and edit, matching the existing cron warnings.
  - confidence: 0.9
- **[P2] Reuse the current cron validation context:** `src/gateway/server-methods/cron.validation.test.ts:4-13`
  - body: The added `createCronContext` only provides `cron.add`, `cron.update`, and logging, but current handlers call `context.getRuntimeConfig()` and update validation also reads `context.cron.getDefaultAgentId()` plus `context.cron.getJob(jobId)`. These tests will throw before reaching the warning assertions; extend the existing validation-test context instead.
  - confidence: 0.92
