## Summary

Keep open. Current main still lacks the requested sub-30-minute cron advisory and this PR is the active replacement for #38731, but the patch is not mergeable as-is because the CLI warning goes to stdout and the Gateway tests do not match the current handler context.

## What This Changes

The PR adds a shared high-frequency `every` schedule warning helper, wires it into CLI cron add/edit and Gateway cron.add/update, adds focused tests, and adds a changelog entry.

## Best Possible Solution

A narrow main-aligned change should warn non-blockingly for sub-30-minute `every` schedules in CLI add/edit and Gateway add/update, preserve machine-readable stdout, and cover the current handler/test contexts.

## Reproduction Assessment

Yes. Source inspection shows current main accepts `--every 15m` and Gateway `{ kind: "every", everyMs: 900000 }` schedules without an advisory, and the PR defects are also source-reproducible from the diff and current handler calls.

## Solution Assessment

No. The shared helper and add/edit/update parity are the right direction, but this patch should emit CLI advisories on stderr and reuse the current Gateway validation test context before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit CLI warnings on stderr:** `src/cli/cron-cli/register.cron-add.ts:135`
  - body: `defaultRuntime.log` writes to stdout, and cron add/edit also emit JSON through `writeJson`. With a sub-30-minute schedule, the warning text would appear before the JSON payload and make machine-readable output unparsable; emit the advisory with `defaultRuntime.error` in both CLI paths.
  - confidence: 0.9
- **[P2] Reuse the current cron validation context:** `src/gateway/server-methods/cron.validation.test.ts:4-13`
  - body: The new test context only provides `cron.add`, `cron.update`, and logging, but current `cron.add` calls `context.getRuntimeConfig()` and `cron.update` also calls `context.cron.getDefaultAgentId()` and `context.cron.getJob()`. These tests will throw before proving the warning; extend the existing cron validation helper instead.
  - confidence: 0.92
