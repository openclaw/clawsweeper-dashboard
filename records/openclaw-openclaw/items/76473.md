## Summary

Keep open. The PR addresses a real missing cron guardrail on current main, but it still has merge-blocking correctness issues: the CLI warning goes to stdout ahead of JSON output, and the new Gateway tests do not provide the handler context current main requires.

## What This Changes

The PR adds a shared high-frequency `every` schedule warning, wires it into CLI cron add/edit and Gateway cron add/update, and adds changelog plus focused regression coverage.

## Best Possible Solution

Land a current-main repair that keeps short intervals valid, emits non-blocking CLI advisories on stderr for add/edit, logs Gateway warnings for add/update, and extends the existing cron validation coverage.

## Reproduction Assessment

Yes, from source inspection. Current main accepts CLI `--every 15m` and Gateway `{ kind: "every", everyMs: 900000 }` schedules without any high-frequency advisory path.

## Solution Assessment

No. The shared helper and add/edit/update coverage are the right direction, but this implementation should route CLI warnings to stderr and reuse the current Gateway validation-test context before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit CLI cron advisories on stderr:** `src/cli/cron-cli/register.cron-add.ts:135`
  - body: The new warning uses `defaultRuntime.log`, which writes to stdout, while cron add/edit also emit their JSON response on stdout. For a sub-30-minute schedule, the warning can be printed before the JSON payload and make machine-readable output unparsable; route this advisory through `defaultRuntime.error` for both add and edit, matching the existing cron warnings.
  - confidence: 0.9
- **[P2] Reuse the current cron validation context:** `src/gateway/server-methods/cron.validation.test.ts:4-13`
  - body: The added `createCronContext` only provides `cron.add`, `cron.update`, and logging, but current handlers call `context.getRuntimeConfig()` and update validation also reads `context.cron.getDefaultAgentId()` plus `context.cron.getJob(jobId)`. These tests will throw before reaching the warning assertions; extend the existing validation-test context instead.
  - confidence: 0.92
