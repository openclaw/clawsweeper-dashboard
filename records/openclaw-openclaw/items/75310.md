## Summary

Keep open. The PR targets a valid runtime-deps recovery bug, but the current head is not ready because the new completion marker is only written after a real install path reaches finalization; already materialized installs can return early without ever writing the marker, so valid pre-marker trees stay flagged incomplete on every gateway startup or doctor repair pass. Open #75183 also explicitly supersedes this implementation and tracks the same linked bug, so maintainers should route the final fix between the two PRs.

## What This Changes

The PR adds a bundled runtime-deps completion marker, checks it during gateway startup and doctor repair to force full reinstall when prior staging looks incomplete, and adds regression tests plus a changelog entry.

## Best Possible Solution

Use a completion signal that is reconciled whenever the install root is proven fully materialized, including materialized fast paths and legacy complete trees, or adopt #75183's package-plan materialization approach. The final fix should cover both the interrupted no-main package sentinel case and the already-complete no-marker upgrade case.

## Reproduction Assessment

Yes. A source-level reproduction can create an install root with `node_modules`, generated manifest/package contents that satisfy `isRuntimeDepsPlanMaterialized`, and no `.install-complete`; after this PR, gateway or doctor repair still leaves the marker absent and the next pass reports the same tree incomplete again.

## Solution Assessment

No. The early-return boundary is the right place to repair this bug, but this marker implementation is not complete because it does not converge already materialized installs; a safer fix either writes/reconciles the marker on the fast path or relies on the stronger materialization plan being developed in #75183.
