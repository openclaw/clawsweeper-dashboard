## What This Changes

The branch adds `.vcf`/`text/vcard` MIME mapping, allows `text/vcard` in host-read media loading, and adds MIME plus web-media test coverage.

## Best Possible Solution

Support vCards in the shared media loader only after routing `text/vcard` through full-buffer text validation with binary-tail regression coverage, then land one canonical PR with real workflow proof.

## Reproduction Assessment

Yes, source-level: current main lacks `text/vcard` in the host-read allowlist, and `file-type@22.0.1` detects `BEGIN:VCARD` buffers as `text/vcard`, so the reported path is blocked today. I did not run a live Telegram or delivery repro in this read-only review.

## Solution Assessment

No as submitted: the shared media loader is the right layer, but vCards need the same full-buffer text validation as other host-read text documents before the allowlist widens. A safer fix is to add `text/vcard` to the validated text path or add an equivalent validation branch before the sniffed-document return.
