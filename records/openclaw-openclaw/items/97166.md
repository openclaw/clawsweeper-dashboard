## Best Possible Solution

## Reproduction Assessment

Yes, source-level: current main lacks `text/vcard` in the host-read allowlist, and `file-type@22.0.1` detects `BEGIN:VCARD` buffers as `text/vcard`, so the reported local `.vcf` path is blocked today. I did not run a live Telegram or delivery repro in this read-only review.

## Solution Assessment

No as submitted: the shared media loader is the right layer, but vCards need full-buffer text validation before the allowlist widens. A safer fix is to add `text/vcard` to the validated text path or add an equivalent validation branch before the sniffed-document return.
