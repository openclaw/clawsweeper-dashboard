## Summary

Keep open. Current main still has a plausible Slack room routing mismatch: unthreaded room messages under `replyToMode: "all"` can share the channel session while outbound delivery targets each message timestamp as a Slack thread, and the linked draft PR #75995 is a reproduction aid rather than a merged fix.

## What This Changes

The issue reports intermittent Slack message/session linkage errors where channel-root and thread replies are delivered into the wrong OpenClaw session or Slack thread.

## Best Possible Solution

Add a Slack-plugin routing fix and focused regression coverage so room-channel top-level messages under `replyToMode: "all"` use session keys aligned with their outgoing Slack `thread_ts`.

## Reproduction Assessment

Yes for the likely source-level root cause: two top-level room messages through `resolveSlackRoutingContext` with `replyToMode="all"` currently resolve to the same channel session while delivery targets each message timestamp as a Slack thread. A live Slack repro is still missing.

## Solution Assessment

Yes. A narrow fix in Slack prepare-routing with targeted tests is the maintainable repair path; broad session, streaming, or core reply rewrites would add unnecessary risk first.
