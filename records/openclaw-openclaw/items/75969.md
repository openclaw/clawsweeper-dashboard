## Summary

Keep open. Current main still has a credible Slack routing mismatch: unthreaded room/channel messages can share one session while `replyToMode="all"` delivery and context metadata target each message timestamp as a Slack thread; the related draft PR is test-only and not a merged fix.

## What This Changes

The issue asks for Slack messages and OpenClaw sessions to remain linked to the Slack thread where the triggering message was posted, especially under `replyToMode="all"`.

## Best Possible Solution

Add a narrow Slack-plugin fix and regression coverage that aligns room/channel session routing with the actual Slack reply thread target for `replyToMode="all"`, while preserving established shared-session behavior for other modes and chat types.

## Reproduction Assessment

Yes for the likely source-level root cause: two unthreaded room/channel messages through `resolveSlackRoutingContext` with `replyToMode="all"` currently share a session, while delivery metadata targets each message timestamp as a Slack thread. A live Slack repro is still missing.

## Solution Assessment

Yes: the best repair path is a focused Slack routing/test change, using the draft PR’s matrix as reproduction evidence, rather than a broad session, streaming, or core routing rewrite.
