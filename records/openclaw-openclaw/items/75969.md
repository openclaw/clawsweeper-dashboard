## Summary

Keep open. Current main still routes unthreaded Slack room/channel messages with `replyToMode="all"` through the base channel session while reply metadata points at the message timestamp, and open PRs #64080/#75995 already track the Slack fix path.

## What This Changes

Route Slack top-level channel messages under `replyToMode="all"` into the same thread-scoped session used for Slack reply delivery instead of the base channel session.

## Best Possible Solution

Land or replace the Slack-owned fix so one canonical thread id drives both inbound session routing and outbound reply delivery for top-level channel messages with `replyToMode="all"`, with focused regression coverage and no unrelated channel changes.

## Reproduction Assessment

Yes. A high-confidence source path exists: a top-level Slack room/channel message with `replyToMode="all"`, no `thread_ts`, and no seeded mention computes `autoThreadId` but resolves the routed session through the base channel while `MessageThreadId` remains the message timestamp.

## Solution Assessment

Yes. The narrow maintainable fix is to make Slack derive one canonical thread id for this case so `SessionKey` and `MessageThreadId` align; #64080/#75995 are the current review path rather than a second repair job.
