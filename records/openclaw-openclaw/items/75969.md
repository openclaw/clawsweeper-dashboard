## Summary

Keep open. Current main still has the Slack room/channel `replyToMode="all"` session-routing mismatch, and the same root cause is already tracked by open canonical work in #64078/#64080 plus the reporter's draft reproduction PR #75995.

## What This Changes

The issue reports Slack messages being associated with the wrong Slack thread or channel-root session under `replyToMode="all"`.

## Best Possible Solution

Land or replace #64080 with an equivalent Slack-owned patch that derives one conversation thread id for both inbound session routing and reply delivery, with regression coverage from #75995 and no changes to unrelated channels.

## Reproduction Assessment

Yes. A high-confidence source path exists: a top-level Slack room/channel message with `replyToMode="all"`, no `thread_ts`, and no seeded mention computes `autoThreadId` but resolves `SessionKey` to the base channel while `MessageThreadId` remains the message timestamp.

## Solution Assessment

Yes. Unifying the routed thread id with the reply-delivery thread id is the narrowest maintainable fix; #64080 and #75995 target that boundary, but the final patch should be reviewed against current `prepare-routing.ts`.
