## Summary

Keep open. Current main still skips the computed Slack auto-thread id for roomish top-level channel messages, while the reply path targets the message timestamp under `replyToMode="all"`; the linked #75995 is useful draft test evidence, not a merged production fix.

## What This Changes

Slack top-level room messages with `replyToMode="all"` should route to the session matching the Slack message thread instead of sharing the channel session.

## Best Possible Solution

Fix Slack plugin routing so roomish top-level `replyToMode="all"` messages get distinct thread-scoped sessions aligned with the outgoing Slack `thread_ts`, while preserving DM, MPIM, and `off`/`first`/`batched` behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear: two top-level Slack room messages through `resolveSlackRoutingContext` with `isRoomish=true` and `replyToMode="all"` resolve to the same channel session, and current tests explicitly assert that behavior.

## Solution Assessment

Yes. A narrow change in Slack prepare-routing with focused regression tests is the maintainable fix; generic channel binding changes or Telegram race fixes should wait for separate reproduction.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75969 in the Slack plugin. Current `resolveSlackRoutingContext` computes `autoThreadId` for top-level messages when `replyToMode="all"`, but roomish channels ignore it because `canonicalThreadId` uses only `roomThreadId`, causing fresh top-level channel posts to share the channel session while outbound replies target the message timestamp. Implement the narrow Slack routing fix so roomish top-level channel messages under `replyToMode="all"` use distinct `:thread:<messageTs>` session keys aligned with Slack `thread_ts`. Preserve top-level DM stability, MPIM/group-DM behavior, true thread replies by parent `thread_ts`, seeded app-mention thread roots, and `replyToMode` values `off`, `first`, and `batched`. Use #75995 as TDD/reference evidence but do not make a test-only PR. Check #75630 for overlap before finalizing. Add/update focused regression coverage and a single-line user-facing `CHANGELOG.md` fix entry with reporter credit if policy allows.
