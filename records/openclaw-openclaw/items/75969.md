## Summary

Keep open. Current main still has Slack room routing that can put multiple top-level channel messages into the same session even when `replyToMode` is `all`, and the linked draft PR #75995 provides a plausible focused reproduction rather than a merged fix.

## What This Changes

The issue reports Slack channel and thread messages being associated with the wrong OpenClaw session or delivered outside the expected Slack thread when `replyToMode` is `all`.

## Best Possible Solution

Add a Slack-plugin fix and regression coverage so room channel top-level messages under `replyToMode=all` use per-message thread-scoped sessions aligned with the outgoing Slack `thread_ts`, while preserving other reply modes and DM/MPIM behavior.

## Reproduction Assessment

Yes for the likely source-level root cause: two top-level room messages through `resolveSlackRoutingContext` with `replyToMode=all` currently resolve to the same channel session, while delivery targets each message timestamp as a thread. A live Slack repro is still missing.

## Solution Assessment

Yes, a narrow fix in Slack prepare-routing plus focused tests is the best repair path; broad session or streaming rewrites would add unnecessary risk before this routing mismatch is addressed.
