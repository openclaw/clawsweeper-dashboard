## Summary

Keep open. Current main still has the Slack `replyToMode="all"` room/channel routing mismatch described in the discussion, and the reporter has an open paired draft test PR while the broader canonical fix path is still unmerged.

## What This Changes

Slack channel messages can be associated with the wrong Slack thread/session when `replyToMode="all"` is enabled.

## Best Possible Solution

Land or replace #64080 with an equivalent Slack-owned fix that uses one canonical thread id for inbound session routing and reply delivery, and fold in the focused regression coverage from #75995 before closing this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a top-level Slack channel message with `replyToMode="all"`, no `thread_ts`, and no seeded mention gets `MessageThreadId=messageTs` while `SessionKey` remains the base channel session.

## Solution Assessment

Yes. Unifying the routed thread id with the reply-delivery thread id in the Slack routing layer is the narrowest maintainable fix, with tests preserving `off`, `first`, and existing true-thread behavior.
