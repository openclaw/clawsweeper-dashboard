## Summary

Keep open. Current main still has a Slack room/channel routing mismatch where top-level messages under `replyToMode="all"` can share the channel session while reply delivery targets per-message Slack threads, and the linked draft PR #75995 is reproduction coverage rather than a merged production fix.

## What This Changes

The issue requests fixing Slack inbound session/thread routing so replies from Slack channel roots and threads stay attached to the corresponding Slack thread/session.

## Best Possible Solution

Patch Slack prepare-routing so roomish top-level channel messages under `replyToMode="all"` use thread-scoped session keys aligned with the Slack message timestamp, while preserving real thread replies, seeded mention roots, DMs/MPIMs, and other reply modes.

## Reproduction Assessment

Yes for the likely source-level root cause: two roomish top-level Slack messages through `resolveSlackRoutingContext` with `replyToMode="all"` currently resolve to the same channel session while thread metadata targets each message timestamp. A live Slack repro is still not attached.

## Solution Assessment

Yes, a narrow Slack-plugin routing fix with focused regression tests is the best path; broad core binding changes or the separate Telegram race would add risk and should stay out of this patch.
