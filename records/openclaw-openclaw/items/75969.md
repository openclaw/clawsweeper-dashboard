## Summary

Keep open. Current main still has a Slack routing mismatch where ordinary top-level room messages under `replyToMode="all"` share the channel session while outbound delivery targets each message timestamp as a Slack thread, and the linked draft PR is a useful reproduction rather than a merged fix.

## What This Changes

Fix Slack so responses from channel roots and thread replies stay attached to the corresponding Slack thread/session instead of crossing into another thread or the channel root.

## Best Possible Solution

Add a Slack-plugin fix and regression coverage so `replyToMode="all"` room/channel roots that receive Slack thread replies use session keys aligned with the Slack `thread_ts`, while preserving `off`/`first`/`batched`, DM, MPIM, and mention-seeded behavior.

## Reproduction Assessment

Yes for the likely source-level root cause: two top-level Slack channel messages through `resolveSlackRoutingContext` with `replyToMode="all"` currently resolve to the same channel session, while the reply planner targets each message timestamp as the Slack thread. A deterministic live Slack repro is still missing.

## Solution Assessment

Yes for the Slack part: the narrowest maintainable fix is in Slack prepare-routing/session-key behavior with focused regression tests. The Telegram comment should stay as corroborating evidence until a separate log-backed reproduction proves the same core layer is involved.
