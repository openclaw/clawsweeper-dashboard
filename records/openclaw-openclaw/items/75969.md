## Summary

Keep open. Current main still has a Slack room/channel `replyToMode="all"` routing split: inbound processing uses the base channel session while reply metadata points at the message thread, and open related work already tracks the same fix path.

## What This Changes

The issue reports intermittent Slack messages being attached to earlier sessions or delivered outside the intended thread under `replyToMode="all"`.

## Best Possible Solution

Land or replace #64080 with a Slack-owned patch that derives one canonical thread id for both inbound session routing and reply delivery for room/channel `replyToMode="all"`, using #75995 as regression coverage.

## Reproduction Assessment

Yes. A high-confidence source path exists for an unmentioned top-level Slack room/channel message with `replyToMode="all"`: `routedThreadId` is undefined while `MessageThreadId` is the message timestamp.

## Solution Assessment

Yes. Unifying the Slack routing thread id with the reply-delivery thread id is the narrowest maintainable fix; #64080 and #75995 target that boundary and should be reviewed against current `prepare-routing.ts`.
