## Summary

Keep open. Current main still has the Slack room routing mismatch: ordinary top-level channel messages under `replyToMode: "all"` keep the shared channel session while reply metadata targets a per-message Slack thread, and the related implementation/test PRs remain unmerged.

## What This Changes

The issue reports intermittent Slack message/session linkage errors where channel-root messages and thread replies can be delivered into the wrong OpenClaw session or Slack thread.

## Best Possible Solution

Review and land #64080 or an equivalent Slack-owned replacement that aligns inbound room session keys with Slack reply thread targets for `replyToMode="all"`, while preserving other reply modes, DMs, MPIMs, seeded mention roots, runtime bindings, docs, and changelog coverage.

## Reproduction Assessment

Yes for the likely Slack source-level root cause: call `resolveSlackRoutingContext` for two unseeded top-level room messages with `replyToMode="all"`; current main resolves both to the base channel session while reply metadata uses each message timestamp. A live Slack reproduction is still missing from this issue.

## Solution Assessment

Yes, if maintainers confirm the `replyToMode="all"` room contract: unifying the routed thread id with `MessageThreadId` in Slack prepare-routing is the narrowest maintainable fix. #64080 still needs current-main review/rebase and any required changelog/test alignment before merge.
