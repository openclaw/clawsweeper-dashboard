## What This Changes

The PR imports `truncateUtf16Safe` into `src/shared/assistant-error-format.ts` and uses it for the long raw assistant-error fallback truncation.

## Best Possible Solution

Land the scoped helper fix after review/CI, ideally with a small formatter regression test; broaden to the sibling raw fallback only if maintainers want all assistant error truncation code-point safe in the same PR.

## Reproduction Assessment

Yes, from source inspection: current main slices a UTF-16 string at code-unit offset 600, so an astral character split at that boundary can leave a dangling surrogate before the ellipsis. I did not run repo tests because this review is read-only and local test commands may write artifacts.

## Solution Assessment

Yes for the named shared helper: using the existing dependency-free `truncateUtf16Safe` at the fallback truncation site is the narrowest maintainable fix. The only safer expansion would be adding a regression test and, if desired, applying the same helper to the adjacent raw `formatAssistantErrorText` fallback.
