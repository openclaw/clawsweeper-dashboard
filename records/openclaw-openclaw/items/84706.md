## Summary

Keep open: the issue has fresh v2026.5.19 live evidence and a plausible current source path where runtime catalog metadata can still force off-only thinking validation before provider-owned OpenAI/Codex thinking policy is applied.

## What This Changes

The issue asks for `sessions_spawn` on canonical `openai/*` and `openai-codex/*` GPT-5 models to accept supported non-off thinking levels and report the effective model consistently.

## Best Possible Solution

Make `/think` and `sessions_spawn` validate against one effective OpenAI/Codex capability source, merging provider-owned GPT-5 reasoning metadata over stale runtime catalog rows and adding regression coverage for both canonical and `openai-codex/*` spawn calls.

## Reproduction Assessment

Yes at source level, but not as a live local run: current code shows an existing runtime catalog row with `reasoning: false` will force off-only validation before OpenAI provider policy is consulted. The issue also supplies fresh v2026.5.19 live probes for the same error shape.

## Solution Assessment

No, the current behavior is not the best solution because runtime catalog rows can preserve stale capability metadata ahead of provider-owned OpenAI/Codex policy. The narrow maintainable fix is to align catalog precedence/effective-target resolution and cover `sessions_spawn` plus `/think` together.
