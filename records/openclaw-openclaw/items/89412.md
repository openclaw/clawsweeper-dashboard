## Label Justifications

- `P1`: The issue reports a documented multi-agent config that can fail startup and lose route bindings during doctor repair after upgrade.
- `impact:data-loss`: `doctor --fix` can silently remove valid route binding entries from user config.
- `impact:crash-loop`: The same config shape is rejected at load time, leaving the gateway or CLI unable to start with that config.
- `impact:message-loss`: Deleted or rejected route bindings can prevent channel traffic from reaching the intended implicit `main` agent.

## Summary

Keep open: current main and v2026.6.8 still build allowed binding targets only from `agents.list`, and the live closing PR is open rather than merged, so the documented implicit-`main` binding bug is not fixed on main yet.

## What This Changes

The issue reports that configs with a populated `agents.list` reject explicit `agentId: "main"` route bindings at load time and `doctor --fix` deletes those bindings.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/89419 or a replacement that preserves exact `main` across schema, doctor, route resolution, and configured-binding compilation with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: the current schema and doctor predicates both omit `DEFAULT_AGENT_ID` when `agents.list` is non-empty. I did not run tests because this review is read-only.

## Solution Assessment

Yes, if scoped to exact-`main` preservation across config validation, doctor pruning, runtime routing, and configured-binding compilation. A schema-only repair would be too narrow because sibling route consumers still resolve unlisted `main` through fallback default-agent selection.
