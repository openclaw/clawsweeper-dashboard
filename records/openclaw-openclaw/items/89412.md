## Summary

Keep open: current main still rejects and prunes explicit `main` route bindings when `agents.list` is populated, which contradicts current docs and CLI routing behavior.

## What This Changes

The issue asks OpenClaw to accept and preserve route bindings that explicitly target the implicit default agent `main` even when `agents.list` contains other agents.

## Best Possible Solution

Treat `DEFAULT_AGENT_ID` as an allowed explicit binding target in both config validation and doctor pruning, while continuing to reject or prune genuinely missing configured-agent ids and leaving broader fallback-routing policy to the existing related issues.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a non-empty `agents.list` plus a binding with `agentId: "main"` fails the schema set-membership check and is removed by the doctor filter. I did not run the Vitest snippet during this read-only review.

## Solution Assessment

Yes, the suggested fix is the narrow maintainable path: add `DEFAULT_AGENT_ID` to both allowed-id sets and add regression tests. It should not change default-agent selection, fallback routing, or the broader privacy/product questions in https://github.com/openclaw/openclaw/issues/44375 and https://github.com/openclaw/openclaw/issues/60712.
