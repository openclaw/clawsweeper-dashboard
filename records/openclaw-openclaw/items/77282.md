## Summary

Keep open. Current main supports the reported failure mode, and the PR is a focused fix, but the patch appears to broaden staged npm swaps to Windows non-npm roots that the existing layout helper cannot distinguish from npm roots.

## What This Changes

The PR stages pnpm-detected npm-layout updates through npm prefix swaps, adds legacy install-runtime aliases, expands upgrade-survivor coverage, and updates docs plus changelog.

## Best Possible Solution

Keep the npm-prefix swap for provably npm-shaped roots, but add a stricter non-npm staging predicate with Windows pnpm regression coverage while preserving the guarded install-runtime alias generation.

## Reproduction Assessment

Yes. Source inspection shows current main stages only npm targets and still emits pnpm add -g for pnpm targets, matching the overlay path described; I did not run a live global update in this read-only review.

## Solution Assessment

No, not quite. The direction is narrow and maintainable, but the branch should avoid coercing non-npm Windows roots through npm unless the root is proven to be npm layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid staging Windows pnpm roots through npm:** `src/infra/package-update-steps.ts:95`
  - body: Dropping the manager guard means every target accepted by resolveNpmGlobalPrefixLayoutFromGlobalRoot can enter the staged npm path. On Windows that helper returns a layout for any node_modules global root, so a real pnpm target can now run npm i -g --prefix and then swap npm-installed files and shims into pnpm's package tree. Keep real pnpm roots on the pnpm path unless the root is proven to be npm layout, and add a Windows pnpm regression case.
  - confidence: 0.82
