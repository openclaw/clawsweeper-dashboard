## Summary

Keep open: the PR addresses a real package-update overlay failure, but the latest head still appears to introduce a Windows pnpm regression by coercing any npm-layout-shaped non-npm target through npm staging.

## What This Changes

The PR changes package update staging, runtime postbuild aliases, upgrade-survivor coverage, docs, tests, and changelog to harden package updates with stale plugin install chunks.

## Best Possible Solution

Keep the clean npm-prefix swap for roots proven to be npm layout, but preserve real pnpm roots on the pnpm path and add Windows pnpm regression coverage before merge.

## Reproduction Assessment

Yes. Source inspection shows current main stages only npm targets and still uses pnpm add -g for pnpm targets, while the PR routes npm-layout-shaped pnpm targets through npm staging; I did not run a live global update.

## Solution Assessment

No. The staged-swap direction is narrow, but the implementation should use a stricter predicate so non-npm Windows roots are not treated as npm installs solely because their global root ends in node_modules.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid staging real Windows pnpm roots through npm:** `src/infra/package-update-steps.ts:97`
  - body: This coercion sends any non-npm target accepted by resolveNpmGlobalPrefixLayoutFromGlobalRoot through npm. On Windows that helper accepts any node_modules global root, so a real pnpm global install can be updated with npm i -g --prefix and then swapped back into pnpm's package tree. Keep real pnpm roots on the pnpm path unless the target is proven to be npm layout, and add a Windows pnpm regression case.
  - confidence: 0.82
