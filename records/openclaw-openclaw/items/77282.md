## Summary

Keep this PR open: the update-hardening direction is valid, but the current diff still coerces real Windows pnpm global roots onto an npm staged-swap path, which can update the wrong package-manager layout. It is also maintainer-authored and carries the protected `maintainer` label, so it should not be cleanup-closed.

## What This Changes

The PR stages pnpm-detected npm-layout global updates through npm prefix swaps, adds stable and legacy plugin install runtime aliases, expands upgrade-survivor coverage for stale source-only plugin shadows, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the clean npm-prefix swap for roots proven to be npm layout, preserve real pnpm roots on the pnpm path, and add Windows pnpm regression coverage before merge.

## Reproduction Assessment

Yes. Source inspection shows current main stages only npm targets and still uses `pnpm add -g` for pnpm targets, while the PR routes any npm-layout-shaped target through npm staging; the Windows pnpm regression follows from the helper accepting any win32 `node_modules` root. I did not run a live Windows global update.

## Solution Assessment

No. The staged-swap direction is the right narrow fix for npm-layout roots, but the implementation needs a stricter predicate so non-npm Windows roots are not treated as npm installs solely because their global root ends in `node_modules`.

## Review Findings
