## Summary

Keep open: current main still has the `agents add` config-write race, and the linked implementation candidates are still open rather than merged.

## What This Changes

The issue asks for concurrent `openclaw agents add` invocations to preserve all agent config mutations instead of losing entries or surfacing `ConfigMutationConflictError`.

## Best Possible Solution

Land one canonical `agents add` conflict-handling fix that rereads fresh config on conflict, preserves binding outcomes, rechecks duplicates, and adds focused regression coverage before closing this issue.

## Reproduction Assessment

Yes, at source level: current main reads a config hash once and commits with that stale hash, while the config writer throws `ConfigMutationConflictError` if another writer wins first. I did not run the live parallel CLI command because this review is read-only and that path mutates user config.

## Solution Assessment

Yes for the direction, but not as a cleanup close now. The maintainable fix belongs in the `agents add` config commit path, with the open implementation candidate repaired or superseded and validated before merge.
