## Summary

Keep open: current main still commits `agents add` with the initial config hash and no retry handling, while the same-author fix PR is still open with unresolved review/proof blockers.

## What This Changes

Request retry-safe concurrent `openclaw agents add` config writes so parallel invocations preserve all new agents instead of losing entries or surfacing raw conflicts.

## Best Possible Solution

Land a corrected `agents add` retry implementation that rebuilds from a fresh valid snapshot on conflict, preserves final binding outcomes, rechecks duplicate agents on retry, adds regression coverage and changelog, then close this issue.

## Reproduction Assessment

Yes, at source level: current main reads a config hash once, commits with that stale hash, and the config writer throws `ConfigMutationConflictError` when another writer wins the race. I did not run the live parallel CLI repro because this review is read-only.

## Solution Assessment

Yes for the direction, but not by closing this issue now. The narrow maintainable fix belongs in `agents add` retry handling around config commits, with the linked PR repaired and validated before merge.
