## Summary

Keep open: current main still commits `agents add` from a single pre-read config snapshot with no conflict retry, and the same-author fix PR is still open with review/proof blockers.

## What This Changes

Make concurrent `openclaw agents add` invocations preserve all agent config changes instead of losing entries or surfacing raw config conflict errors.

## Best Possible Solution

Land a corrected `agents add` conflict retry or locked transform that mutates a fresh valid config snapshot, rechecks duplicate and binding state, and adds regression coverage plus real concurrent CLI proof.

## Reproduction Assessment

Yes, at source level: current main reads one config hash, commits with that stale hash, and the config writer throws `ConfigMutationConflictError` when another writer wins the race. I did not run the live parallel CLI repro because this review is read-only.

## Solution Assessment

Yes for the direction, but not by closing this issue now. The narrow maintainable fix belongs in `agents add` retry or locked read-modify-write handling around config commits, with the linked PR repaired and validated before merge.
