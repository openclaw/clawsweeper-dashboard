## Summary

Keep this PR open, but it is not mergeable as-is. Static review confirms the previous ClawSweeper blockers remain: generated config and SDK artifacts are missing, Google/provider strictness is not fully threaded, live tool-name repair reporting is incomplete, and docs/changelog contracts drift from the proposed public API.

## What This Changes

The PR adds strict/warn/off tool strictness handling, structured strictness reports, gateway request/response fields, config/SDK type surfaces, docs, and tests across embedded runner and provider transports.

## Best Possible Solution

Amend this PR so generated config and SDK artifacts, provider strictness propagation, repair reporting, docs, changelog, and focused validation all agree before treating the public API as stable.

## Reproduction Assessment

Yes. Static tracing reproduces the blockers: generated schema consumers read files the PR does not update, provider-owned Google streams do not receive strictness, live tool-name normalization lacks the repair callback, and the documented Google test path does not exist.

## Solution Assessment

No. The feature direction is plausible, but this is not the best mergeable form until every advertised strict/warn/off contract is wired through the generated artifacts, runtime paths, report collection, and docs.

- **[P2] Regenerate generated config schema artifacts:** `src/config/zod-schema.ts:350-355`
  - body: Adding `toolStrictness` to `OpenClawSchema` is not enough because `openclaw config schema` is built from `GENERATED_BASE_CONFIG_SCHEMA`. Regenerate and commit `src/config/schema.base.generated.ts` plus the config docs hash so schema consumers expose the new key.
  - confidence: 0.92
- **[P2] Thread strictness into provider-owned Google streams:** `src/agents/provider-transport-stream.ts:34`
  - body: The new `ctx.strictness` only reaches core-owned transport factories. Google resolves through the provider `createStreamFn` hook, whose registration still constructs `createGoogleGenerativeAiTransportStreamFn()` without the strictness options, so embedded Google runs can ignore warn/strict mode.
  - confidence: 0.9
- **[P2] Pass repair callbacks through tool-name normalization:** `src/agents/pi-embedded-runner/run/attempt.tool-call-normalization.ts:1125`
  - body: Warn mode documents `toolNameNormalization` repairs, but the live trim wrapper carries only `mode`. Streamed tool names can still be normalized without calling `collectToolStrictnessRepair`, leaving `toolStrictnessReport.repairs` incomplete.
  - confidence: 0.86
- **[P2] Regenerate the Plugin SDK API baseline:** `src/plugin-sdk/provider-transport-runtime.ts:13-14`
  - body: This exports new public types from `openclaw/plugin-sdk/provider-transport-runtime`, but the PR does not update `docs/.generated/plugin-sdk-api-baseline.sha256`. The SDK API baseline check should drift until the public surface hash is regenerated.
  - confidence: 0.86
- **[P3] Fix the Google transport test path:** `docs/help/strict-tool-mode-testing.md:31`
  - body: The testing doc tells contributors to run `src/agents/google-transport-stream.test.ts`, but the Google transport tests live under `extensions/google/transport-stream.test.ts`. The documented focused command will fail before exercising strict mode.
  - confidence: 0.95
- **[P3] Add the user-facing changelog entry:** `src/gateway/openai-http.ts:65-66`
  - body: This adds public config and gateway API behavior, but the PR does not touch `CHANGELOG.md`. OpenClaw policy requires a single-line changelog entry for user-facing feature changes.
  - confidence: 0.83
- **[P3] Mention warn mode in config help:** `src/config/schema.help.ts:35-36`
  - body: The schema accepts `off`, `warn`, and `strict`, but the config help text describes only off and strict. Users inspecting config help would miss the report-only warn mode this PR adds.
  - confidence: 0.88
- **[P3] Translate the design doc to English:** `docs/design/strict-tool-mode.md:1`
  - body: The new design document is primarily Chinese, while the docs guide says English docs in this repo are the source of truth. Translate it or keep non-source design notes out of `docs/` so maintained docs remain consistent.
  - confidence: 0.82
