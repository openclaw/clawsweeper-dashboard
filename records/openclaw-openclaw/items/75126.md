## Summary

Keep this PR open. The feature is not implemented on current main, and the proposed patch is not mergeable yet because several public-contract paths still disagree with the claimed strict-tool-mode behavior.

## What This Changes

The branch adds strict/warn/off tool strictness plumbing, structured repair reports, config and gateway request/response fields, provider transport handling, tests, and strict-tool-mode documentation.

## Best Possible Solution

Amend the same PR branch so every public contract matches the feature: regenerate config schema and docs baselines, pass strictness through provider-owned Google streams using a generic provider contract, report live tool-name normalization repairs, fix the documented test path, keep source docs in English, add the required changelog entry, and rerun focused strictness plus config checks before broad validation.

## Reproduction Assessment

Yes. Static source and diff tracing is enough to reproduce the review findings: config schema consumers read the generated schema, provider-owned Google streams still do not receive strictness, live name trimming lacks a repair callback, and the documented Google test file is absent.

## Solution Assessment

No. The feature direction is reasonable, but this is not the best mergeable form until the generated public surfaces, provider-owned runtime path, report contract, docs, and release notes are aligned.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P2] Regenerate the base config schema:** `src/config/zod-schema.ts:350-355`
  - body: Adding `toolStrictness` to `OpenClawSchema` is not enough because `openclaw config schema` is served from `GENERATED_BASE_CONFIG_SCHEMA`. Without regenerating and committing the generated base schema and config docs baseline, config/schema consumers will not expose the new key even though parsing accepts it.
  - confidence: 0.91
- **[P2] Thread strictness into the Google provider stream:** `src/agents/provider-transport-stream.ts:87-90`
  - body: The Google transport factory now accepts strictness, but the provider-owned runtime path still constructs `createGoogleGenerativeAiTransportStreamFn()` without it. Embedded Google runs will keep repairing replayed JSON-string tool args even when the runtime resolved strict mode.
  - confidence: 0.89
- **[P2] Record live tool-name repairs in reports:** `src/agents/pi-embedded-runner/run/attempt.ts:1865-1874`
  - body: Warn mode is documented to record `toolNameNormalization`, and the normalizer can emit that event, but `runEmbeddedAttempt` passes only `mode` into `wrapStreamFnTrimToolCallNames`. Live streamed tool-name normalization still mutates names without calling `collectToolStrictnessRepair`, so `toolStrictnessReport` misses those repairs.
  - confidence: 0.84
- **[P3] Correct the Google transport test path:** `docs/help/strict-tool-mode-testing.md:31-35`
  - body: The new testing doc tells contributors to run `src/agents/google-transport-stream.test.ts`, but that file does not exist; the Google transport tests live at `extensions/google/transport-stream.test.ts`. The documented focused check will fail before it verifies strict mode.
  - confidence: 0.95
- **[P3] Keep the new design doc in English:** `docs/design/strict-tool-mode.md:1-7`
  - body: The added design doc is written in Chinese, but the docs guide says English docs in this repo are the source of truth and foreign-language docs are not maintained here. Convert this page to English or keep it out of the published docs source.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/config/zod-schema.ts:350-355`
  - body: This PR adds user-facing config and gateway API fields but does not update `CHANGELOG.md`. OpenClaw policy requires user-facing feature changes to have an active-version changelog entry with contributor credit, so release notes would miss this new surface.
  - confidence: 0.88
