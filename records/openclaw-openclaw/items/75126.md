## Summary

Keep open. Current main still lacks strict tool mode, so the PR is not obsolete, but the proposed patch has merge blockers: the generated config schema is stale, Google provider-owned transport does not receive strictness at runtime, live tool-name repair reporting is not wired into the outer report, and one new docs command points at a non-existent test file.

## What This Changes

The PR adds opt-in strict/warn/off tool strictness plumbing, structured diagnostics reports, config and gateway request/response fields, provider transport handling, tests, and strict-tool-mode documentation.

## Best Possible Solution

Keep this PR open and amend the same branch: regenerate the base config schema and docs baseline, pass strictness through the provider-owned Google stream contract, wire live tool-name normalization repairs into toolStrictnessReport, fix the docs test path, then rerun the focused strictness and config checks.

## Reproduction Assessment

Yes for the review findings. Static source tracing is enough to reproduce the gaps: the served schema comes from GENERATED_BASE_CONFIG_SCHEMA, the Google provider still builds its stream without strictness, live tool-name trimming has no report callback, and the documented Google test path is absent.

## Solution Assessment

No. The overall feature direction is reasonable, but this is not the best mergeable form until the generated config surface, provider-owned runtime path, report contract, and docs validation command all match the claimed behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Regenerate the base config schema:** `src/config/zod-schema.ts:350-355`
  - body: Adding `toolStrictness` to `OpenClawSchema` is not enough because `openclaw config schema` is served from `GENERATED_BASE_CONFIG_SCHEMA`. Without regenerating and committing the generated base schema and config docs baseline, config/schema consumers will not expose the new key even though parsing accepts it.
  - confidence: 0.91
- **[P2] Pass strictness through the Google provider runtime:** `src/agents/provider-transport-stream.ts:87-90`
  - body: The Google transport factory now accepts `strictness`, but the provider-owned runtime path still constructs `createGoogleGenerativeAiTransportStreamFn()` without it. Embedded Google runs will keep repairing replayed JSON-string tool args even when the runtime resolved strict mode.
  - confidence: 0.89
- **[P2] Record live tool-name repairs in the report:** `src/agents/pi-embedded-runner/run/attempt.ts:1865-1874`
  - body: Warn mode is documented to record `toolNameNormalization`, and the normalizer can emit that event, but `runEmbeddedAttempt` passes only `mode` into `wrapStreamFnTrimToolCallNames`. Live streamed tool-name normalization still mutates names without calling `collectToolStrictnessRepair`, so `toolStrictnessReport` misses those repairs.
  - confidence: 0.84
- **[P3] Correct the Google transport test path:** `docs/help/strict-tool-mode-testing.md:31-35`
  - body: The new testing doc tells contributors to run `src/agents/google-transport-stream.test.ts`, but that file does not exist; the Google transport tests live at `extensions/google/transport-stream.test.ts`. The documented focused check will fail before it verifies strict mode.
  - confidence: 0.95
