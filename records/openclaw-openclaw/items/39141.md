## Summary

Keep open: current main has diagnostic stuck-session logging/recovery and coarse session update signals, but not the requested opt-in session-activity watchdog, alert outputs, configurable alert classes, or optional model summaries. The prerequisite public activity API is still open, so this remains a product/API design item rather than conservative cleanup.

## What This Changes

Request an opt-in gateway-side watchdog that consumes future `sessions.activity` state and emits deduped, rate-limited operator alerts, with optional tokenized summaries and configuration.

## Best Possible Solution

Define the canonical `sessions.activity` contract first, then add a small opt-in alert observer that reuses that state, avoids message-content leakage, and has clear dedupe, cooldown, authorization, docs, and tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a request for new opt-in gateway monitoring and alerting behavior. Source inspection gives high confidence that current main lacks the requested `sessions.activity`-backed watchdog and alert outputs.

## Solution Assessment

Unclear until the prerequisite activity contract is accepted. The gateway-side observer direction is plausible, but the best maintainable solution should reuse the canonical session activity/event model rather than invent parallel polling, timer, or attention-state ownership.
