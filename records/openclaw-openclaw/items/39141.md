## Summary

Keep open: current main and v2026.6.10 still have coarse session metadata, Control UI-visible active-run flags, channel health/stuck-session diagnostics, and diagnostic recovery, but not the requested opt-in gateway watchdog/alerting layer built on a `sessions.activity.*` contract. The prerequisite activity-state design remains open, so this is still a maintainer product/API decision rather than cleanup closure.

## What This Changes

Request to add an opt-in gateway-side session activity watchdog that consumes session activity state and emits deduped operator alerts, with optional model-generated summaries.

## Best Possible Solution

First settle the canonical `sessions.activity` and attention-state contract, then build a minimal opt-in watchdog on top of that state with privacy-preserving templates, dedupe/cooldowns, docs, and focused tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this requests a new opt-in gateway watchdog and alerting capability. Source and release inspection give high-confidence evidence that the prerequisite activity API and requested alert layer are absent on current main and v2026.6.10.

## Solution Assessment

Unclear until maintainers settle the product/API boundary. The right direction is to consume the canonical session activity contract from https://github.com/openclaw/openclaw/issues/39127 rather than inventing independent polling or diagnostic-only state.
