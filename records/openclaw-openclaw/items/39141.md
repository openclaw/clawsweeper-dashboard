## Summary

Keep open. Current main still lacks the prerequisite `sessions.activity.get/list` API and activity state-change WebSocket contract, and the existing stuck-session diagnostics, task ledger, and channel health checks are adjacent operational signals rather than the requested opt-in per-session watchdog with configurable alert classes, dedupe/cooldowns, operator delivery, and optional tokenized summaries. The linked prerequisite #39127 is still open in the provided GitHub context, and the other related items are narrower inputs rather than superseding this request.

## What This Changes

Request an opt-in gateway watchdog that consumes per-session activity state and emits no-token, deduped operator alerts for stale or blocked sessions, with optional tokenized summaries.

## Best Possible Solution

Keep this open and sequence it behind #39127 or an equivalent gateway-owned per-session activity/attention API. Once that contract exists, add a disabled-by-default gateway watchdog with structured no-token alerts, configurable thresholds, dedupe/cooldowns, safe UI/log/webhook/plugin delivery seams, optional rate-limited model summaries, and focused protocol/runtime/docs tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is source and protocol inspection: the requested activity RPC/event contract and watchdog configuration/delivery surface are absent from the current gateway methods, schema, docs, and diagnostics implementation.

## Solution Assessment

Unclear whether every requested alert output should land exactly as proposed, but sequencing the watchdog after a gateway-owned activity API is the maintainable direction. Existing diagnostic stuck-session warnings and channel health checks are not the best substitute because they are not session-scoped operator alerting contracts.
