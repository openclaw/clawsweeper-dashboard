## Summary

Keep open. The report is detailed and current source shows a plausible message-only path: cron forwards `payload.toolsAllow`, but normal tool policy can still remove the requested runtime tools while forced delivery keeps `message` callable, so the run can avoid the empty-allowlist guard and finish with a misleading tool-unavailable reply.

## What This Changes

The issue reports that isolated cron `agentTurn` jobs with `payload.toolsAllow` for `exec`/`bash` expose only `message` and cannot run the requested local shell command.

## Best Possible Solution

Keep the issue open and make isolated cron fail explicitly when `payload.toolsAllow` requests tools that are removed by effective policy, or deliberately document and test any maintainer-approved behavior that lets cron `toolsAllow` grant those tools.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection gives a medium-confidence reproduction shape: a runtime `toolsAllow` can be forwarded, then effective tool policy can remove `exec` while forced delivery leaves `message` as the only callable tool.

## Solution Assessment

Unclear pending maintainer policy choice. The maintainable direction is to prevent silent message-only success, but maintainers should decide whether cron `toolsAllow` may widen restrictive profiles or should only produce a clear blocked-tool error.
