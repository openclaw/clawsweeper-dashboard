## Summary

This PR should stay open because the proposed guard is too broad: it treats known env-var auth references as non-secret markers and can suppress valid provider catalogs. It also does not cover the self-hosted vLLM/SGLang discovery helper described in the linked issue, and the external PR has no real behavior proof yet.

## What This Changes

The PR imports `isNonSecretApiKeyMarker` and makes the single/paired provider catalog helpers return `null` when the resolved `apiKey` is missing or classified as a non-secret marker.

## Best Possible Solution

Preserve env-var/SecretRef catalog entries, skip only true local non-runnable markers with a narrowed check or discovery-key-aware contract, and cover the self-hosted discovery helper if this PR is meant to close the linked report.

## Reproduction Assessment

Yes for the PR regression by source inspection: an env-backed provider whose resolver returns `apiKey: "LITELLM_API_KEY"` would hit the new `isNonSecretApiKeyMarker(apiKey)` guard and return `null`. The linked 401 behavior was not live-reproduced against a real LiteLLM/vLLM setup.

## Solution Assessment

No. The current patch uses the marker predicate in its default env-var-inclusive mode and does not cover the self-hosted discovery helper cited by the linked issue.
