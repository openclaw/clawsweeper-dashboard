## Summary

Keep open: the PR targets a real provider-catalog auth bug, but it is not merge-ready because after-fix real behavior proof is still missing and the catalog guard remains broad enough to hide valid marker-backed provider catalogs.

## What This Changes

The PR imports `isNonSecretApiKeyMarker` and adds marker guards in provider catalog and self-hosted discovery helpers to avoid `/models` discovery with placeholder API keys.

## Best Possible Solution

Narrow marker handling to skip only dynamic unauthenticated discovery for marker-only local auth while preserving env-var, SecretRef, OAuth/SDK catalog markers; add focused regression tests and redacted after-fix provider proof.

## Reproduction Assessment

Yes, source-reproducible: current main can pass a truthy non-secret marker as `apiKey` while `discoveryApiKey` is undefined, and the local model discovery fetch then sends no Authorization header. The reported LiteLLM scenario has before-fix logs but no after-fix live proof yet.

## Solution Assessment

No. The direction is right for dynamic discovery, but the current PR applies broad marker classification at the static catalog boundary; a safer fix is discovery-key-aware or local-marker-only gating with regression coverage.
