## Summary

Keep this PR open for changes: it targets a real provider-catalog auth problem, but the current guard can suppress valid env-var/SecretRef-backed provider catalogs, does not cover the self-hosted discovery path named by the linked issue, and lacks required real behavior proof.

## What This Changes

The PR changes shared provider catalog helpers to return null when the resolved apiKey is missing or recognized as a non-secret auth marker.

## Best Possible Solution

Preserve env-var and SecretRef catalog entries, skip only true marker-only discovery, and cover the self-hosted discovery helper if this PR is intended to close the linked issue.

## Reproduction Assessment

Yes for the PR regression by source inspection: an env-backed provider whose resolver returns a known env-var marker as apiKey would hit the new guard and return null. The linked 401 behavior was not live-reproduced in a real LiteLLM/vLLM setup.

## Solution Assessment

No. The patch uses the marker predicate in its env-var-inclusive default mode and does not cover the self-hosted discovery helper cited by the linked issue.

If this PR is meant to close that issue, update that helper or stop closing the issue.
