## Summary

Keep open: the SDK fetch hook is a plausible boundary, but the helper explicitly bypasses non-OK SSE responses even though the OpenAI SDK reads failed response bodies with `response.text()`, leaving a hostile 4xx/5xx SSE body unbounded; the external PR also still needs real runtime proof beyond Vitest streams.

## What This Changes

The PR adds a `createSdkBoundedFetch` helper and wires it into the `openai-completions` OpenAI SDK client to cap SSE response bodies at 16 MiB, with helper and provider tests.

## Best Possible Solution

Bound all OpenAI SDK SSE bodies for this provider, including non-OK error responses, add focused status-error coverage, and supply redacted real runtime proof while maintainers accept or tune the 16 MiB cap.

## Reproduction Assessment

Yes, as a source-reproducible path. Current main constructs the `openai-completions` OpenAI SDK client without a custom fetch, and the inspected OpenAI SDK accumulates SSE chunks until a delimiter and reads failed bodies with `response.text()`.

## Solution Assessment

No, not yet. The SDK fetch hook is a narrow supported fix point, but skipping non-OK SSE responses leaves a known unbounded SDK body-read path and makes the patch incomplete.
