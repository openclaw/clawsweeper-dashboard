## Summary

Keep open: the PR is aimed at a real source-reproducible Codex app-server timeout bug, but the current patch still lets mismatched request families count as attempt progress and the external proof remains fake-client only.

## What This Changes

The PR replaces the fixed post-turn/start Codex app-server timeout with a progress-aware idle watchdog and adds regressions for current-turn notifications plus non-turn request traffic.

## Best Possible Solution

Keep the progress-aware watchdog approach, but gate request-response progress on actual current-turn validation for every request family and require redacted live Codex app-server proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main accepts a turn and then enforces a fixed post-start timeout while current-turn notifications are tracked separately. I did not run a live Codex app-server session in this read-only review.

## Solution Assessment

No, not yet: the code direction is right, but selected request families are marked as attempt progress before their helpers confirm the request belongs to the active turn. The narrower fix is to move or derive attemptProgress from successful current-turn validation, then add the matching regression proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate attempt progress on current-turn request validation:** `extensions/codex/src/app-server/run-attempt.ts:1368`
  - body: This marks `mcpServer/elicitation/request`, `item/tool/requestUserInput`, and approval requests as turn activity before the handlers validate that the params match the active thread/turn. Those helpers return `undefined` for missing or mismatched turn ids, but the `finally` block still calls `touchTurnCompletionActivity(... attemptProgress: true)`, so an unrelated request with one of these method names can keep a dead accepted turn alive. Set the progress flag only after the request is confirmed to belong to the current turn, or share the same validation before marking it as attempt progress.
  - confidence: 0.86
