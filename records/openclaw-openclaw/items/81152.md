## Summary

Keep open: the PR targets a real source-reproducible Codex app-server timeout, but the patch still has a watchdog-scope correctness issue and the supplied proof is mock-only for an external contributor PR.

## What This Changes

The PR changes Codex app-server post-start attempt timeout handling from a fixed deadline to a progress-aware idle watchdog reset by current-turn activity.

## Best Possible Solution

Keep the progress-aware watchdog approach, but scope attempt-watch resets to current-turn notifications and turn-scoped request responses, add a non-turn request regression, and require redacted real app-server proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main accepts a turn and then enforces a fixed post-start timeout even though current-turn notification activity is tracked elsewhere. I did not run a live Codex app-server session in this read-only review.

## Solution Assessment

No, not yet: the general owner-boundary fix is appropriate, but the current patch lets non-turn request traffic reset the new attempt watchdog. The safer solution is to split or gate the reset path so only current-turn progress extends the attempt.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the attempt watchdog scoped to turn progress:** `extensions/codex/src/app-server/run-attempt.ts:1147`
  - body: Adding `scheduleTurnAttemptIdleWatch()` inside `touchTurnCompletionActivity()` makes every caller reset the new post-start watchdog. The request handler calls this helper before request method/turn filtering and again in `finally`, including for `account/chatgptAuthTokens/refresh`, pre-turn, and other non-turn requests, so unrelated app-server request traffic can keep a dead accepted turn alive after the absolute timeout was removed. Gate the attempt-watch reset to current-turn notifications and turn-scoped request responses, and cover a non-turn request case.
  - confidence: 0.84
