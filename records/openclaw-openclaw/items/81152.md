## Summary

Keep open: the focused Codex app-server bugfix direction is useful, but the patch is not merge-ready because the new watchdog can be reset by non-turn request traffic, and the supplied proof is still mock-only.

## What This Changes

The PR replaces the Codex app-server post-`turn/start` absolute timeout with a progress-aware idle watchdog and adds regression coverage for current-turn progress past the old deadline.

## Best Possible Solution

Keep the narrow progress-aware watchdog approach, but reset it only from current-turn notifications or validated turn-scoped request responses and require redacted real app-server proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main accepts `turn/start` and then enforces a fixed post-start timeout independent of current-turn progress. I did not run a live Codex app-server session in this read-only review.

## Solution Assessment

No, not as submitted: the progress-aware watchdog is the right direction, but the reset path is too broad. The safer fix is to gate attempt-watch resets to current-turn notifications and turn-scoped request responses, with a non-turn request regression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the attempt watchdog scoped to turn progress:** `extensions/codex/src/app-server/run-attempt.ts:1147`
  - body: Calling `scheduleTurnAttemptIdleWatch()` from `touchTurnCompletionActivity()` makes every caller reset the post-start watchdog. The request handler touches activity before method/turn filtering and again in `finally`, including auth refresh and non-turn requests, so unrelated app-server traffic can keep a dead accepted turn alive. Gate this reset to current-turn notifications and turn-scoped request responses, and cover a non-turn request case.
  - confidence: 0.84
