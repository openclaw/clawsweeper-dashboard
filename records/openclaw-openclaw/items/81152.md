## Summary

Keep open: the PR is aimed at a real, source-reproducible Codex app-server timeout bug, but the head still has a blocking watchdog-scoping bug and the external real-behavior proof is mock-only.

## What This Changes

The PR changes the Codex app-server attempt watchdog from a fixed post-start deadline to a progress-aware idle timeout, with regression coverage for progressing current turns and unrelated non-turn app-server requests.

## Best Possible Solution

Keep the progress-aware watchdog design, but only let validated current-turn notifications and validated turn-scoped request responses affect attempt progress, then require redacted real Codex app-server proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main accepts turn/start, arms a fixed params.timeoutMs timer, and can abort even if current-turn notifications arrive later. I did not run a live OAuth/frontier Codex app-server session in this read-only review.

## Solution Assessment

No, not yet: the progress-aware watchdog is the right narrow direction, but the PR still counts selected request methods as attempt progress before their helpers validate active thread/turn scope. The safer fix is to derive request progress from successful current-turn validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Gate request progress on current-turn validation:** `extensions/codex/src/app-server/run-attempt.ts:1368`
  - body: This sets `requestCountsAsTurnActivity` before the elicitation, user-input, and approval handlers prove the request belongs to the active turn. Those helpers can return `undefined` for stale or mismatched params, but the `finally` block still records `attemptProgress: true`, so an unrelated request with one of these supported method names can keep a dead accepted turn alive. Move the progress flag behind the same current-turn validation or return an explicit counted result from the helpers.
  - confidence: 0.86
