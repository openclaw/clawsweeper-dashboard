## Summary

Keep open: the fix direction matches the linked bug, but this patch makes the new attempt-idle watchdog reset from the shared request/activity helper, so non-turn app-server requests can extend a dead turn; the PR also only supplies fake app-server Vitest proof, not real after-fix behavior proof.

## What This Changes

The PR replaces Codex app-server's post-turn-start absolute attempt timeout with a progress-aware idle watchdog and adds regression coverage for progress past the old deadline.

## Best Possible Solution

Revise the Codex runner change so only current-turn notifications and validated turn-scoped request responses refresh the new attempt-idle watchdog, keep idle/dead-turn interruption intact, add a regression for non-turn request activity, and attach real after-fix behavior proof before merge.

## Reproduction Assessment

Yes. Current main source and tests show an accepted in-progress Codex app-server turn is still aborted by a fixed post-start params.timeoutMs timer even if progress would arrive later; I did not run a live Codex session in this read-only review.

## Solution Assessment

No. The intended progress-aware timeout is the right narrow direction, but this implementation should scope watchdog resets to real current-turn progress instead of all request activity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Scope attempt-idle resets to turn progress:** `extensions/codex/src/app-server/run-attempt.ts:1147`
  - body: The new attempt watchdog is scheduled from `touchTurnCompletionActivity()`, but that helper is also called for every app-server request before method or turn validation. That means non-turn activity such as `account/chatgptAuthTokens/refresh` can now refresh the post-start attempt watchdog and keep a dead current turn alive, reintroducing the keepalive-hides-stuck-turn class that the adjacent account-notification liveness fix avoided. Please only refresh this new watchdog from current-turn notifications and validated turn-scoped request responses.
  - confidence: 0.86
