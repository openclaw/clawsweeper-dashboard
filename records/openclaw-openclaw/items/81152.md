## Summary

Keep open: this targets a valid source-reproducible Codex app-server timeout bug, but the branch is still merge-blocked because selected request methods can record attempt progress before active-turn validation, and the supplied proof is fake-client only.

## What This Changes

The branch replaces the Codex app-server post-turn/start absolute timeout with a progress-aware idle watchdog and adds fake-client regressions for current-turn progress plus non-turn request traffic.

## Best Possible Solution

Keep the progress-aware watchdog direction, but derive request attempt-progress only after current-turn validation succeeds, add focused regressions for invalid/mismatched request families, and require redacted real app-server proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main accepts a turn and then enforces a fixed post-start timeout, while current-turn notifications are already scoped to the active thread and turn. I did not run a live Codex app-server session in this read-only review.

## Solution Assessment

No, not yet: the progress-aware watchdog is the right direction, but request-response progress must be gated on successful active-turn validation for every request family before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate request progress on current-turn validation:** `extensions/codex/src/app-server/run-attempt.ts:1368`
  - body: The new requestCountsAsTurnActivity flag is set before the elicitation, user-input, and approval handlers verify the request belongs to the active thread/turn. Those helpers can return undefined for invalid or mismatched scope, but the finally block still records attemptProgress, so a stale or unrelated request with one of these method names can keep a dead turn alive. Set the flag only after successful scope validation, or share a validated helper result before touching attempt progress.
