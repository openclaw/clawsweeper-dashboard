## Summary

Keep open: the progress-aware watchdog direction addresses a real Codex app-server timeout bug, but the latest PR head still marks some request families as attempt progress before current-turn validation and the external real-behavior proof remains mock-only.

## What This Changes

The PR replaces the Codex app-server post-start fixed timeout with a progress-aware idle watchdog and adds focused regression tests for progressing turns and unrelated non-turn request traffic.

## Best Possible Solution

Keep the progress-aware Codex watchdog, but only record attempt progress after each turn-scoped request is validated as belonging to the active turn, add regressions for mismatched request params, and require redacted live app-server proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main accepts turn/start and then enforces a fixed post-start timeout, while the PR defect is visible from source because selected request helpers can reject mismatched turn scope after the progress flag has already been set. I did not run tests or a live Codex app-server session in this read-only review.

## Solution Assessment

No, not yet: the watchdog design is the right narrow direction, but request-response progress needs to be derived from successful current-turn validation rather than method name alone. The safer alternative is to set attemptProgress only after the handler returns a validated current-turn response and to prove that with focused regressions plus live proof.
