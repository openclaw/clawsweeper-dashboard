## Summary

Keep this PR open, but do not merge it as-is: the durable Slack ingress state is written before dispatch, yet the patch does not consume pending duplicates correctly or replay pending records, so it can still duplicate or strand inbound Slack events.

## What This Changes

The PR adds a durable Slack inbound ingress journal around Slack message acceptance, dispatch completion, duplicate/drop outcomes, and retryable release handling.

## Best Possible Solution

Keep the Slack-owned durable ingress journal, but make the handler consume the full journal accept state and add a Slack replay path for pending records before merge.

## Reproduction Assessment

Yes from source inspection for the PR defects: the SDK returns pending/completed duplicate states, the Slack wrapper collapses them, and no Slack path reads pending records. I did not run a live Slack reproduction.

## Solution Assessment

No. The storage boundary is plausible, but the best fix must follow the durable receive contract by suppressing pending/completed duplicates and replaying pending records through an owned Slack recovery path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Handle pending ingress duplicates before dispatch:** `extensions/slack/src/monitor/inbound-delivery-state.ts:268`
  - body: `journal.accept()` can return `pending` when this same Slack event already has a durable pending record. This wrapper collapses that to `accepted: false`, and the caller then continues through the normal delivery/seen checks, so after a restart or stuck-session retry the event can dispatch again while the original pending record is still waiting. Return the full accept kind and stop or replay from that durable state before enqueueing.
  - confidence: 0.87
- **[P2] Add a consumer for pending ingress records:** `extensions/slack/src/monitor/inbound-delivery-state.ts:151-154`
  - body: The PR writes pending Slack ingress records and releases retryable failures back to the pending store, but no Slack monitor path calls `pending()` or re-enqueues those stored payloads. A crash or stall before delivery recording leaves the event durable but inert for the pending TTL, so the claimed recovery still depends on Slack sending another event. Add a startup/recovery replay path like the sibling WhatsApp durable receive drain.
  - confidence: 0.84
