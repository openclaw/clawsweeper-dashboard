## Summary

Keep open: the PR addresses a real Telegram approval cleanup problem, but its runtime change targets a removed legacy handler while current main resolves Telegram approvals through the native approval runtime; it also lacks required real Telegram proof and still has docs issues.

## What This Changes

The PR adds delayed deletion for resolved Telegram exec approval messages in a legacy handler and bundles several unrelated documentation cleanups.

## Best Possible Solution

Implement resolved-message deletion in the active Telegram native approval runtime or shared final-action plumbing with owned or unref'd delayed deletion, cover approve and deny flows, and keep docs cleanup separate and source-only.

## Reproduction Assessment

Yes, at source level: current main's Telegram native runtime returns `clear-actions` and the callback handler calls `clearCallbackButtons()` after resolving approvals, so the message remains while only buttons are removed. I did not run a live Telegram flow in this read-only review.

## Solution Assessment

No. The PR edits a removed legacy handler; the maintainable fix belongs in `extensions/telegram/src/approval-handler.runtime.ts` or the shared final-action path, with real Telegram proof after the change.
