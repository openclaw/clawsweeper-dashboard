## Summary

Keep open. Current main and the latest release still publish an unpinned LiteLLM PyPI install example without advisory/remediation text, while the bundled OpenClaw LiteLLM integration is only a network/proxy plugin and not a Python package dependency. The issue remains valid, but the public security wording should be reviewed against current upstream guidance before landing.

## What This Changes

Request to update the LiteLLM provider docs with safe install guidance and remediation advice for users exposed to compromised LiteLLM PyPI versions 1.82.7 and 1.82.8.

## Best Possible Solution

Land a docs-only update to `docs/providers/litellm.md` that keeps OpenClaw runtime code unchanged, replaces or qualifies the unpinned install example, links the upstream OSV/LiteLLM incident sources, and gives dated affected-version checks, credential rotation, host audit, cleanup, and safe-version guidance reviewed by a maintainer/security owner.

## Reproduction Assessment

Not applicable as a runtime bug; this is a docs/security guidance issue. The docs gap is high-confidence: `docs/providers/litellm.md:50` on current main and v2026.4.29 still contains the unpinned install example, and targeted search finds no advisory/remediation text.

## Solution Assessment

Unclear as originally specified: adding advisory/remediation guidance is the right direction, but a v1.82.6-only pin is no longer the narrowest public guidance because LiteLLM now documents v1.83.0 and verified safe versions. The best fix is date-qualified security guidance tied to upstream sources. ([docs.litellm.ai](https://docs.litellm.ai/blog/security-update-march-2026))
