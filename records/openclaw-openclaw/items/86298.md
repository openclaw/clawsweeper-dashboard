## Summary

Keep open: the intended fix is valid, but this branch puts the guard on a gateway notification helper rather than the isolated cron agent-turn announce path that the linked report exercises, so the user-visible token leak would remain. The PR also has no after-fix real behavior proof.

## What This Changes

The branch imports `isSuppressedControlReplyText` into `src/cron/delivery.ts` and skips `deliverCronAnnouncePayload` when the message is an exact suppressed control token.

## Best Possible Solution

Wire `isSuppressedControlReplyText` into the shared isolated cron announce dispatch path before outbound delivery, add focused regression coverage for exact control tokens, and include redacted real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an isolated `agentTurn` cron with announce delivery and a final `REPLY_SKIP`/`ANNOUNCE_SKIP` reply reaches `dispatchCronDelivery`, which currently only strips `NO_REPLY`. I did not run a live cron in this read-only review.

## Solution Assessment

No, this is not the best fix as written because the guard is placed in `src/cron/delivery.ts`, while the reported runtime flow is handled by `src/cron/isolated-agent/delivery-dispatch.ts`. The narrow maintainable fix is to apply the existing helper in that shared dispatch path with regression coverage.
