## Summary

Keep open. Current main has useful mitigations for context pressure, but the reported recovery shape is still present: generic reserveTokensFloor reset guidance remains, overflow meta errors still reset the session, the embedded runner still uses a bounded compaction/retry loop, bootstrap injection remains the default retry behavior unless explicitly disabled, and the empty-session tool-result truncation diagnostic is still reachable.

## What This Changes

The issue asks OpenClaw to investigate repeated context-overflow hard resets where high compaction reserve settings do not stop retry loops, bootstrap context appears to be re-injected on each retry, and recovery diagnostics are misleading or not actionable.

## Best Possible Solution

Define one overflow-recovery contract that separates model context-window/maxTokens misconfiguration from reserve-floor tuning, makes reset guidance budget-aware, and specifies when retry/post-compaction paths should suppress, shrink, dedupe, or mark bootstrap reinjection before a hard reset.

## Reproduction Assessment

Yes for the source-level recovery shape: force an embedded context-overflow meta error or oversized prompt path and current main still exercises compaction retry, optional truncation, and auto-reply reset behavior. The exact Basecamp/Voltti session is not reproducible from the repository alone.

## Solution Assessment

No complete current solution exists yet. The recent precheck, contextInjection opt-outs, and tool-result recovery work reduce pressure, but the remaining reset UX, generic reserve guidance, and default bootstrap reinjection policy need a maintainer-approved design before this is fixed.
