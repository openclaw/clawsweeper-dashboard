## Summary

Keep open. Current main still has the source-level overflow recovery shape reported here, while the later discussion adds independent Basecamp, custom-provider, and Telegram workload evidence that this is broader than one bad `reserveTokensFloor` value.

## What This Changes

The issue asks OpenClaw to stop repeated context-overflow retries from rebuilding bootstrap context, hard-resetting the same session, and giving reserve-floor-only recovery guidance.

## Best Possible Solution

Define one overflow-recovery contract that separates model context-window limits, reserve-token tuning, bootstrap footprint, tool-result pressure, and resumable turn state, then implement recovery paths that avoid reinflating stable context before a hard reset.

## Reproduction Assessment

Yes, at source level. Current main still shows the three-attempt overflow compaction loop, conditional transcript continuation, normal bootstrap rebuilding, static reserve-floor fallback hint, and hard-reset path; I did not replay the exact channel sessions.

## Solution Assessment

Unclear. The requested investigation is valid, but the best fix needs a shared overflow-recovery and turn-resume contract rather than a one-line reserve-floor, bootstrap, or retry-count change.
