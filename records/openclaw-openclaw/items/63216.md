## Summary

Keep this issue open: current main and v2026.6.10 include several mitigations, but overflow recovery can still exhaust the three-attempt path into a blocked context-overflow result, and default documented behavior still rebuilds bootstrap context on post-compaction retries.

## What This Changes

The issue asks OpenClaw to stop repeated context-overflow reset/retry loops when high reserve headroom, tool-result truncation, and bootstrap reinjection still leave a session over budget.

## Best Possible Solution

Keep this as the active maintainer tracker until OpenClaw has a proven recovery design that retries from a smaller canonical context, avoids re-inflating the same oversized bootstrap payload, and preserves or clearly reports session continuity loss.

## Reproduction Assessment

No, not at high confidence from this review alone. The source path and user logs give a medium-confidence reproduction shape through repeated overflow recovery and final context_overflow, but no current-main live Basecamp or Telegram repro was run.

## Solution Assessment

Unclear. Current mitigations are useful but not the best complete solution; maintainers still need to choose whether the right fix is minimal-context retry, bootstrap suppression, checkpoint/spillover behavior, Codex thread ownership changes, or a narrower combination.
