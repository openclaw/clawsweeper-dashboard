Keep open: current main still has the reported source-level recovery shape, and the recent Telegram/QA comment adds another corroborating workload rather than resolving the issue. Related open items cover narrower parts of the problem, but no merged fix or single canonical issue fully supersedes this combined reset-loop report.

## What This Changes

The issue asks OpenClaw to improve context-overflow recovery when compaction, truncation, and bootstrap reinjection repeatedly lead to hard session resets despite high reserve-token settings.

## Best Possible Solution

Define a shared overflow recovery contract that distinguishes model-window and reserve-budget misconfiguration from compaction failure, preserves stable turn state across retries, avoids unnecessary bootstrap reinflation, and provides bounded recovery handoff before hard reset.

## Reproduction Assessment

Yes, at source level: current main still has static reserve-floor guidance, bounded overflow compaction retries, conditional continuation-only retry handling, default bootstrap reinjection, empty-session truncation outcomes, and hard reset fallback. I did not replay the exact Basecamp/Voltti session because that transcript is not available here.

## Solution Assessment

No, the current behavior is not the best solution: tuning `reserveTokensFloor`, raising model context windows, or making the retry count configurable only addresses pieces of the failure mode. The maintainable fix needs a shared recovery contract before choosing a narrow patch.
