## Summary

Keep open. Current main has useful mitigations for context-window pressure and overflow diagnostics, but the reported recovery shape still exists in source: generic reserve-floor reset guidance remains, embedded overflow errors can still hard-reset a session, the run loop still retries compaction up to three times before giving up, bootstrap context is still rebuilt by default on attempts, post-compaction AGENTS reinjection remains default, and empty-session tool-result truncation is still a possible diagnostic result.

## What This Changes

The issue requests investigation of repeated context-overflow reset loops where compaction retries, bootstrap reinjection, tool-result truncation, and fallback guidance interact poorly despite high configured reserve headroom.

## Best Possible Solution

Define a maintainer-approved overflow-recovery contract that distinguishes model context-window misconfiguration from reserve-floor tuning, makes reset guidance budget-aware, and gives retry/post-compaction paths explicit rules for suppressing, shrinking, deduping, or marking bootstrap reinjection before falling back to a hard reset or minimal-context recovery.

## Reproduction Assessment

Yes for the source-level recovery shape: force an embedded context-overflow meta error or oversized prompt path and current main still exercises the bounded compaction retry, optional truncation, and hard-reset fallback paths. The exact Basecamp/Voltti session key cannot be reproduced from this checkout alone.

## Solution Assessment

No. Current main has useful partial mitigations, including reserve-floor capping and a heartbeat-specific hint, but the remaining behavior crosses retry semantics, injected context policy, model budget diagnosis, and reset UX, so the best fix needs a coherent maintainer-owned contract rather than a narrow mechanical patch.
