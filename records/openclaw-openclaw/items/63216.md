## Summary

Keep open. Current main still has the same source-level failure shape for this report: context overflow uses a bounded three-attempt compaction loop, the auto-reply reset path still hard-resets after overflow recovery failure, bootstrap context is normally rebuilt for attempts/retries, and the user-facing guidance still emphasizes reserveTokensFloor even though the discussion shows contextWindow/reserveTokens/tool-result growth can be the real pressure point.

## What This Changes

The issue asks OpenClaw to investigate repeated context-overflow reset loops where compaction retries, tool-result truncation, and bootstrap reinjection do not recover a long-running session even with high reserve token settings.

## Best Possible Solution

Define a shared overflow recovery contract that distinguishes model context-window, reserveTokens, reserveTokensFloor, bootstrap budget, and tool-result pressure; then add recovery paths that avoid reinflating stable context and preserve resumable task state before hard reset.

## Reproduction Assessment

Yes, at source level: current main still has the bounded overflow compaction retry loop, default bootstrap rebuild behavior, optional tool-result truncation, generic reserveTokensFloor guidance, and hard reset fallback. I did not replay the exact Basecamp/Voltti or Telegram sessions because their transcripts and credentials are not available in this checkout.

## Solution Assessment

No. The current implementation has useful mitigation pieces, but it still treats several distinct causes as one overflow recovery path and can keep retrying into reset with misleading operator guidance.
