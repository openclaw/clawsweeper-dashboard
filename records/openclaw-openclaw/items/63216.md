## Summary

Keep open. Current main has useful partial mitigations for aggregate tool-result truncation and bounded replay after silent reset, but the reported loop family still has source-level evidence: static reserve-floor guidance, three-attempt overflow recovery, default bootstrap reinjection on recovery paths, and hard reset fallback remain.

## What This Changes

Investigate and fix context-overflow recovery so retries do not repeatedly reset the same session, reinject bootstrap context, or give misleading reserveTokensFloor guidance.

## Best Possible Solution

Define a shared overflow recovery contract that distinguishes model context-window and reserve-budget misconfiguration from compaction failure, avoids reinflating stable bootstrap context during recovery when possible, and provides proactive tool-output pressure relief plus checkpointed or minimal-context recovery before hard reset.

## Reproduction Assessment

Yes, at source level: current main still has the bounded overflow retry loop, optional truncation, default bootstrap/post-compaction reinjection, and hard reset fallback. I did not replay the exact Basecamp/Voltti session because that transcript is not available here.

## Solution Assessment

No. Current main contains useful partial mitigations, but the best solution needs a maintainer-owned recovery contract rather than another one-off change to the hint, retry count, or truncation branch.
