## Summary

Keep open. Current main still has an Anthropic-compatible outbound path where replay repair can produce `assistant, toolResult, user`; the transport then converts the `toolResult` span into a `role: "user"` payload entry and returns the payload without a final same-role coalescing pass. The adjacent open MiniMax empty-payload PR is related to the same transport file but does not implement this same-role merge.

## What This Changes

The issue requests Anthropic-compatible outbound message normalization so consecutive same-role messages are merged before custom `anthropic-messages` proxies receive the payload.

## Best Possible Solution

Add a final Anthropic Messages payload normalization at the OpenClaw transport boundary, after transport conversion and before payload policy/send, so adjacent `user` and `assistant` payload entries are coalesced without changing core provider routing or patching `node_modules`. Document the outbound transcript rule and keep the fix scoped to Anthropic-compatible APIs.

## Reproduction Assessment

Yes. A source-level reproduction exists: an assistant tool call followed by a repaired or synthesized `toolResult` and then a user turn becomes `assistant, user(tool_result), user(real user)` in the Anthropic payload because conversion returns `params` without a coalescing pass.

## Solution Assessment

Yes. The narrowest maintainable fix is a final Anthropic-compatible payload normalizer in `src/agents/anthropic-transport-stream.ts`; replay-level validation alone is insufficient because the role adjacency is introduced later when `toolResult` messages are converted to Anthropic `user` payload entries.
