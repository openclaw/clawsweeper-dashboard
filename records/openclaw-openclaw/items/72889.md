## Summary

Keep open. Current main still only normalizes stored Anthropic user turns before provider conversion; after replay repair, `toolResult` entries are converted into outbound Anthropic `user` messages and can remain adjacent to a following user turn, so the reported custom-proxy 400 path remains a valid source-level bug.

## What This Changes

The issue requests Anthropic-compatible provider payload normalization so consecutive same-role messages are merged before sending to the Anthropic Messages API or strict custom proxies.

## Best Possible Solution

Add a final Anthropic-compatible outbound payload normalization step after replay repair and provider conversion, with regression coverage for `assistant`, `toolResult`, `user` becoming an alternating request without mutating stored transcript history.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a repaired `assistant`, `toolResult`, `user` transcript is currently expected by tests, and Anthropic conversion maps it to outbound `assistant`, `user`, `user`.

## Solution Assessment

Yes. The narrow maintainable fix is final payload coalescing at the Anthropic-compatible transport or shared wrapper boundary, rather than patching stored transcripts or editing bundled dependency artifacts.
