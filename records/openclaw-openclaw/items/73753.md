## Summary

Keep open. Current `main` still enables the bundled Discord voice path by default, includes `GuildVoiceStates` on that default path, installs the Carbon voice plugin, and loads `DiscordVoiceManager`; the shipped related work only covers the text-only `voice.enabled=false` intent-selection case from #73709, not this enabled-voice idle CPU report.

## What This Changes

The issue reports sustained `openclaw-gateway` CPU usage after embedded Discord bot accounts connect when `channels.discord.voice.enabled=true`, which is the documented default.

## Best Possible Solution

Profile the bundled Discord plugin on current `main` with voice enabled after real bot accounts connect, identify the actual hot loop or dependency behavior, then land the narrowest extension-owned fix with Discord voice regression coverage, updated docs/config text if behavior changes, and a user-facing changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear: the report gives a concrete A/B path on v2026.4.26 and current source still follows the implicated default voice path, but this review did not live-profile current `main` with real Discord credentials on Linux.

## Solution Assessment

Is this the best way to solve the issue? Not applicable because there is no patch. Closing via the #73709 text-only intent fix would be incorrect; the maintainable path is profiling the enabled-voice runtime first, then fixing that path or making a deliberate default change.
