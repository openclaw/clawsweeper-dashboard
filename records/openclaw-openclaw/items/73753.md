## Summary

Keep open. Current main still leaves the bundled Discord plugin's default voice-enabled path active, including `GuildVoiceStates`, `VoicePlugin`, and the voice runtime manager; the shipped related fix only covers text-only `voice.enabled=false` intent selection, not the enabled-voice idle CPU spin reported here.

## What This Changes

The issue requests investigation and repair of sustained gateway CPU usage when the bundled Discord plugin runs with `channels.discord.voice.enabled=true`.

## Best Possible Solution

Profile `openclaw-gateway` on current main with embedded Discord and `channels.discord.voice.enabled=true` after real bot connections, then land a focused Discord plugin runtime fix or an explicit maintainer-approved default/config change with regression coverage, docs, and a changelog entry.

## Reproduction Assessment

Unclear: the report gives concrete v2026.4.26 A/B observations and current main still follows the implicated enabled-voice path, but I did not live-profile current main with real Discord bot accounts.

## Solution Assessment

Not applicable because there is no patch. Closing via the #73709 text-only intent fix would not solve this enabled-voice report; the maintainable path is profiling first, then patching the actual loop or making a deliberate default change.
