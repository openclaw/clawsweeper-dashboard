## Summary

Keep open: current main still defaults direct sessions to daily reset at 4 and leaves the active-transcript byte guard disabled unless operators opt in. The maintenance-mode part is partly outdated in source, and the latest Codex/CLI compaction note is related but not a full superseding fix, so this remains a defaults/docs/product decision rather than conservative cleanup.

## What This Changes

The issue requests safer out-of-box defaults and documentation for long-running coding-agent sessions to avoid nightly direct-session resets, unbounded transcript/session growth, and unclear reset-policy schema handling.

## Best Possible Solution

Choose an explicit long-running-agent default or hardening profile, align runtime/docs/schema help, and add fresh-install plus upgrade tests for direct reset, byte-triggered compaction, and session maintenance.

## Reproduction Assessment

No full week-long live reproduction was run against current main. Source inspection does reproduce the key default predicates: direct resets fall back to daily at 4 and the active-transcript byte guard is disabled unless compaction rotation plus a byte threshold are explicitly configured.

## Solution Assessment

Unclear: the requested defaults are plausible, but changing them is not a narrow bug fix because it changes session lifecycle, compaction cost, and retention semantics for existing installs. A maintainer-approved default/profile plus docs/schema alignment is the safer path.
