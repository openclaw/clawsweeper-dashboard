## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent ownership for UUID rows:** `src/gateway/session-utils.ts:643-644`
  - body: Returning a bare UUID from `canonicalizeSessionKeyForAgent` erases the agent store that produced the row. With per-agent stores, `sessions.list` can expose an unscoped UUID that later `sessions.get`/patch/delete resolves against the current default agent, so changing defaults or having multiple stores can miss or mutate the wrong session.
  - confidence: 0.91
- **[P1] Use a WebSocket client that can send headers:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:809`
  - body: This path builds `Authorization` and `chatgpt-account-id` headers, but passes them to `globalThis.WebSocket` as `{ headers }`. In the Node runtime checked here, the standard constructor is `constructor(url, protocols = [])`, so the proxy auth headers are not a proven transport path and proxied requests can fail authentication or tenant routing.
  - confidence: 0.9
- **[P2] Resolve main-key aliases before UUID short-circuit:** `src/gateway/session-utils.ts:668-669`
  - body: The UUID fast path runs before the existing `session.mainKey` alias handling. If an operator configured a UUID-shaped main key, `resolveSessionStoreKey` now returns the bare UUID instead of the canonical agent main key, splitting history for clients that use that documented alias.
  - confidence: 0.86
- **[P2] Normalize UUID links for configured default agents:** `ui/src/ui/app-settings.ts:51-52`
  - body: Legacy UUID links could have been generated with the configured default agent, not only `agent:main`. Returning early for every other agent id leaves links such as `agent:ops:<uuid>` in the URL/settings path and prevents convergence on the bare Open WebUI UUID identity.
  - confidence: 0.82
- **[P2] Send the system prompt only once:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:722`
  - body: `buildRequestBody` sets `instructions: context.systemPrompt` while `convertResponsesMessages` also injects the same prompt into `input`. Proxied requests therefore receive duplicate policy text, which increases token use and can change model behavior on long prompts.
  - confidence: 0.8
