## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve agent ownership for UUID store rows:** `src/gateway/session-utils.ts:643-644`
  - body: Returning a bare UUID from `canonicalizeSessionKeyForAgent` also changes how per-agent stores are merged: a UUID row from an agent-specific store is surfaced as unscoped, so later get/patch/delete/preview calls for that bare key resolve against the current default agent rather than the store that owned the row. Keep enough agent ownership in the combined target, or add a tested alias contract that round-trips the owning store.
  - confidence: 0.9
- **[P2] Normalize UUID links for any default agent:** `ui/src/ui/app-settings.ts:51-52`
  - body: This strips the prefix only for `agent:main:<uuid>`, but legacy Control UI links were generated with the configured default agent. Installations whose default agent was `ops`, for example, can keep `agent:ops:<uuid>` links and continue alias drift instead of converging on the intended shared UUID session.
  - confidence: 0.86
- **[P2] Resolve main-key aliases before the UUID fast path:** `src/gateway/session-utils.ts:668-669`
  - body: The new UUID early return runs before `rawMainKey` alias handling, so a configured `session.mainKey` that is UUID-shaped now returns the bare UUID instead of the canonical main session key. That silently changes existing main-session routing for a valid stored config value.
  - confidence: 0.82
- **[P2] Canonicalize UUID aliases before message subscriptions:** `src/gateway/session-utils.ts:668-669`
  - body: The bare UUID and `agent:main:<uuid>` forms can still become different canonical keys, while `sessions.messages.subscribe` and transcript fan-out use exact string keys. A legacy client subscribed on the prefixed key can miss live events emitted under the bare UUID key unless both aliases normalize to one subscription target or both forms are handled.
  - confidence: 0.78
- **[P2] Avoid duplicating the system prompt in Codex payloads:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:723`
  - body: `buildRequestBody` sends `instructions: context.systemPrompt` while `convertResponsesMessages` already inserts the same prompt as a system/developer input message. That duplicates prompt tokens and can change model behavior or hit context limits sooner on long prompts.
  - confidence: 0.78
