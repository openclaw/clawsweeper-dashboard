## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Preserve agent ownership for UUID store rows:** `src/gateway/session-utils.ts:640-642`
  - body: Returning a bare UUID from `canonicalizeSessionKeyForAgent` also changes how per-agent stores are merged: a UUID row from an agent-specific store is surfaced as unscoped, so later get/patch/delete/preview calls for that bare key resolve against the current default agent rather than the store that owned the row. Keep enough agent ownership in the combined target, or add a tested alias contract that round-trips the winning store path.
  - confidence: 0.86
- **[P2] Normalize UUID links for non-main default agents:** `ui/src/ui/app-settings.ts:50-52`
  - body: The Control UI only strips `agent:main:<uuid>`, but current canonicalization used the configured default agent when it generated legacy links. A deployment whose default agent was `ops` can still carry `agent:ops:<uuid>` in URLs and settings, so this migration does not converge those chats back to the bare Open WebUI UUID form.
  - confidence: 0.8
- **[P2] Resolve main-key aliases before the UUID fast path:** `src/gateway/session-utils.ts:665-667`
  - body: The new early return runs before the existing `session.mainKey` handling, so a configured main key that happens to be UUID-shaped no longer maps to the canonical main session. That silently changes existing config behavior and can split the main session from clients that use the configured alias.
  - confidence: 0.74
- **[P2] Avoid sending the system prompt twice:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:723-724`
  - body: `buildRequestBody` sets `instructions: context.systemPrompt`, while `convertResponsesMessages` also injects the same system prompt as a system/developer input message. Proxy Codex calls will duplicate the prompt, increasing token usage and potentially changing model behavior for long or policy-heavy prompts.
  - confidence: 0.79
