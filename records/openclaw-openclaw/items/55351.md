## Summary

Keep open. The UUID session split is still source-reproducible on current main and this PR is the active implementation path, but the branch is not mergeable as-is because it lacks after-fix real behavior proof, introduces concrete session-alias regressions, and bundles unrelated credentialed Codex proxy/auth changes.

## What This Changes

The PR preserves Open WebUI UUID session keys across gateway/session lookup and Control UI URL handling, adds transcript lookup regressions, and also adds embedded Codex proxy/auth, compaction-auth, and account-routed send changes.

## Best Possible Solution

Land a narrow replacement or rebased fix that uses one shared, agent-aware session alias contract for UUID-backed Open WebUI chats, keeps transcript lookup and Control UI URL normalization consistent, and splits proxy/auth/routing work into separately reviewed PRs.

## Reproduction Assessment

Yes, at source level. Current main still prefixes bare UUID-like session keys through the default agent and applies Control UI URL session values unchanged, matching the split-session path from the linked Open WebUI report.

## Solution Assessment

No. UUID preservation is the right direction, but this branch is not the best mergeable form because it introduces alias/scope regressions and bundles unrelated credentialed runtime changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent scope for UUID store rows:** `src/gateway/session-utils.ts:643-644`
  - body: Returning bare UUIDs here also affects combined per-agent store merging: agent-store UUID rows can be listed without their owning agent scope, and later get/patch/delete can resolve against the current default agent instead of the row's store.
  - confidence: 0.87
- **[P2] Normalize UUID links for every default agent:** `ui/src/ui/app-settings.ts:51-52`
  - body: This only strips agent:main:<uuid>, but legacy links were generated with whichever agent was configured as default. Non-main defaults keep the prefixed key and can continue the split identity this PR is trying to converge.
  - confidence: 0.86
- **[P2] Resolve main-key aliases before UUID keys:** `src/gateway/session-utils.ts:668-669`
  - body: The UUID early return runs before main-key alias handling, so a configured session.mainKey that happens to be UUID-shaped no longer maps to the canonical main session key and can split history for that configuration.
  - confidence: 0.8
- **[P1] Use a WebSocket transport that forwards auth headers:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:809`
  - body: connectWebSocket passes auth headers through globalThis.WebSocket as the second constructor argument, but the standard global WebSocket API does not provide a headers option. Proxied Codex calls can therefore lose Authorization and account routing headers.
  - confidence: 0.78
- **[P2] Send the system prompt once:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:722-723`
  - body: buildRequestBody sets instructions from context.systemPrompt while convertResponsesMessages already injects the same prompt into the input messages. That duplicates system instructions on every Codex proxy request and can change model behavior or token use.
  - confidence: 0.88
