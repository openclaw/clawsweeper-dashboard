## Summary

Keep open. The Open WebUI UUID session split is still valid on current main, but this PR should not merge as-is because it mixes the narrow session fix with unrelated credentialed Codex proxy/auth/routing changes and still has correctness regressions in agent scoping, alias handling, and live subscription canonicalization.

## What This Changes

The PR preserves Open WebUI UUID session keys across gateway resolution and Control UI URLs, threads agent context into transcript lookup, adds regression tests, and also adds Codex proxy transport, compaction-auth, model-auth, and outbound routing changes.

## Best Possible Solution

Land a narrow Open WebUI UUID session fix that preserves bare UUIDs without losing agent ownership, normalizes legacy aliases for the configured default agent, keeps mainKey and message-subscription canonicalization consistent, and passes agent context through transcript lookup; split the Codex proxy/auth/routing work into separate owner-reviewed PRs.

## Reproduction Assessment

Yes. The linked bug report gives concrete Open WebUI and Control UI steps, and current main still has the two key source conditions: bare UUID keys are canonicalized to the default agent and Control UI URL session values are not normalized.

## Solution Assessment

No. The intended direction is right, but this PR’s implementation is not the narrowest maintainable fix because it drops agent scope for UUID rows, leaves alias forms divergent, and includes unrelated security-sensitive runtime transport changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent scope for UUID store rows:** `src/gateway/session-utils.ts:643-644`
  - body: Returning bare UUIDs here also affects per-agent store merging. A UUID row from another agent store can be listed as unscoped, so later get/patch/delete resolves through the current default agent and can miss or mutate the wrong history.
  - confidence: 0.87
- **[P2] Normalize UUID aliases for any default agent:** `ui/src/ui/app-settings.ts:51-52`
  - body: This only strips `agent:main:<uuid>`, but legacy UUID links were generated with whichever agent was default. Installs with a non-main default keep the prefixed key and can continue the split identity this PR is meant to converge.
  - confidence: 0.86
- **[P2] Resolve main aliases before the UUID fast path:** `src/gateway/session-utils.ts:668-670`
  - body: The UUID early return runs before `session.mainKey` alias handling. If an operator configured a UUID-shaped main key, it now resolves to the bare UUID instead of the canonical main session key, splitting existing clients.
  - confidence: 0.8
- **[P2] Canonicalize UUID aliases for message subscriptions:** `src/gateway/session-utils.ts:668-670`
  - body: Bare UUID and `agent:<id>:<uuid>` forms remain different canonical keys, while message subscriptions and transcript fan-out use exact keys. A legacy client subscribed on the prefixed form can miss live messages emitted under the bare form.
  - confidence: 0.78
- **[P2] Send the system prompt only once:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:722-723`
  - body: `buildRequestBody` sets `instructions` from `context.systemPrompt`, while `convertResponsesMessages` also injects the same prompt into the input messages. That duplicates system instructions on every Codex proxy call.
  - confidence: 0.88
