## Summary

Keep open. The session-identity bug remains real on main, but this branch is not a mergeable fix because it introduces regressions in canonical session ownership and bundles unrelated credentialed agent transport/auth/routing changes.

## What This Changes

The PR preserves Open WebUI UUID chat keys in gateway session resolution, normalizes Control UI URL aliases, threads agent context into transcript lookup, adds regression tests, and also adds Codex proxy/auth/outbound routing changes.

## Best Possible Solution

Land a narrow Open WebUI UUID session fix that preserves shared UUID identity without losing agent ownership or alias canonicalization, and split proxy/auth/routing work into separate reviewed PRs.

## Reproduction Assessment

Yes. #49202 gives concrete Open WebUI to Control UI UUID steps, and current main still has the two source conditions: bare UUID keys canonicalize through the default agent and Control UI URL sessions are passed through unchanged.

## Solution Assessment

No. The intended direction is valid, but this branch is not the best implementation because it introduces session-key regressions and unrelated credentialed runtime transport changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent scope for UUID store rows:** `src/gateway/session-utils.ts:643-644`
  - body: Returning bare UUIDs from `canonicalizeSessionKeyForAgent` also affects per-agent store merging. A UUID row from another agent store can be listed as unscoped, so later get/patch/delete resolves through the current default agent and can miss or mutate the wrong history.
  - confidence: 0.87
- **[P2] Normalize UUID aliases for any default agent:** `ui/src/ui/app-settings.ts:51-52`
  - body: This only strips `agent:main:<uuid>`, but legacy UUID links were generated with whichever agent was default. Installs with a non-main default keep the prefixed key and can continue the split identity this PR is meant to converge.
  - confidence: 0.86
- **[P2] Resolve main-key aliases before UUID normalization:** `src/gateway/session-utils.ts:668-670`
  - body: The UUID early return runs before `session.mainKey` alias handling. If an operator configured a UUID-shaped main key, it now resolves to the bare UUID instead of the canonical main session key, splitting existing clients.
  - confidence: 0.8
- **[P2] Canonicalize UUID aliases for message subscriptions:** `src/gateway/session-utils.ts:668-670`
  - body: Bare UUID and `agent:<id>:<uuid>` forms remain different canonical keys, while message subscriptions and transcript fan-out use exact keys. A legacy client subscribed on the prefixed form can miss live messages emitted under the bare form.
  - confidence: 0.78
- **[P1] Use a WebSocket client that supports headers:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:809`
  - body: `connectWebSocket` passes `{ headers }` to `globalThis.WebSocket`, but the standard Node/Bun WebSocket constructor treats the second argument as protocols, so the bearer/account headers are not transmitted for this transport.
  - confidence: 0.82
- **[P2] Send the system prompt only once:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:722-723`
  - body: `buildRequestBody` sets `instructions` from `context.systemPrompt`, while `convertResponsesMessages` also injects the same prompt into the input messages. That duplicates system instructions on every Codex proxy call.
  - confidence: 0.88
