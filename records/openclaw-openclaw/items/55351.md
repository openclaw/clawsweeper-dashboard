## Summary

Keep open: the Open WebUI UUID session bug is still valid on current main, but this PR is not merge-ready because it mixes the useful session fix with unrelated credentialed runtime transport/auth changes and still introduces session-key regressions.

## What This Changes

The PR changes gateway session canonicalization, transcript lookup, and Control UI URL normalization for Open WebUI UUID chats, adds regression tests, and also adds Codex proxy transport, compaction auth, and outbound routing changes.

## Best Possible Solution

Land a narrow replacement or revised PR that preserves bare Open WebUI UUID session identity without losing agent ownership, keeps bare and prefixed aliases canonical for lookup/subscriptions, threads agent context through transcript resolution, and leaves proxy/auth/routing work to separate owner-reviewed changes.

## Reproduction Assessment

Yes, from source inspection and the linked report. Current main still canonicalizes bare UUID-shaped session keys through the default agent and passes Control UI URL session keys through unchanged, matching the reported split-session path.

## Solution Assessment

No. The intended UUID-preservation direction is valid, but this branch is not the best implementation because it introduces alias/agent-scope regressions and unrelated credentialed proxy/auth/routing changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent scope for UUID store rows:** `src/gateway/session-utils.ts:640-642`
  - body: Returning bare UUIDs from `canonicalizeSessionKeyForAgent` also affects per-agent store merging. A UUID row from another agent store can be listed as unscoped, so later get/patch/delete resolves through the current default agent and can miss or mutate the wrong history.
  - confidence: 0.87
- **[P2] Normalize UUID aliases for any default agent:** `ui/src/ui/app-settings.ts:51-52`
  - body: This only strips `agent:main:<uuid>`, but legacy UUID links were generated with whichever agent was default. Installs with a non-main default keep the prefixed key and can continue the split identity this PR is meant to converge.
  - confidence: 0.86
- **[P2] Resolve main-key aliases before UUID fast path:** `src/gateway/session-utils.ts:665-667`
  - body: The UUID early return runs before `session.mainKey` alias handling. If an operator configured a UUID-shaped main key, it now resolves to the bare UUID instead of the canonical main session key, splitting existing clients.
  - confidence: 0.8
- **[P2] Canonicalize UUID aliases for subscriptions:** `src/gateway/session-utils.ts:665-667`
  - body: Bare UUID and `agent:<id>:<uuid>` forms remain different canonical keys, while message subscriptions and transcript fan-out use exact keys. A legacy client subscribed on the prefixed form can miss live messages emitted under the bare form.
  - confidence: 0.78
- **[P2] Send the system prompt only once:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:728-729`
  - body: `buildRequestBody` sets `instructions` from `context.systemPrompt`, while `convertResponsesMessages` also injects the same prompt into the input messages. That duplicates system instructions on every Codex proxy call.
  - confidence: 0.88
