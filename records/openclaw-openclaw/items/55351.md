## Summary

Keep this PR open: the Open WebUI UUID split is a real supported-surface bug and this PR is the canonical implementation path, but the branch still has blocking correctness issues, lacks after-fix real behavior proof, and bundles unrelated credentialed Codex proxy/auth changes.

## What This Changes

The PR preserves Open WebUI UUID session keys across gateway/session lookup and Control UI URL handling, adds regression tests, and also adds embedded Codex proxy/auth, compaction-auth, and account-routed send changes.

## Best Possible Solution

Land a narrow rebased fix that preserves Open WebUI UUID identity through one shared, agent-aware session alias contract, keeps transcript lookup consistent, and splits proxy/auth/routing work into separately reviewed PRs.

## Reproduction Assessment

Yes, at source level. Current main still prefixes bare UUID-like session keys through the default agent and applies Control UI URL session values unchanged, matching the split-session path from the linked Open WebUI report.

## Solution Assessment

No. UUID preservation is the right direction, but this branch is not the best mergeable form because it introduces alias/scope regressions and bundles unrelated credentialed runtime changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent scope for UUID store rows:** `src/gateway/session-utils.ts:643-644`
  - body: Returning bare UUIDs here also affects combined per-agent store merging: `loadCombinedSessionStoreForGateway` canonicalizes each agent-store row through this helper, so a UUID row from another agent store can be listed without its owning agent scope and later get/patch/delete can resolve against the current default agent instead of the row's store.
  - confidence: 0.87
- **[P2] Normalize UUID links for all default agents:** `ui/src/ui/app-settings.ts:51-52`
  - body: This only strips `agent:main:<uuid>`, but legacy links were generated with whichever agent was configured as default. Non-main defaults keep the prefixed key and can continue the split identity this PR is trying to converge.
  - confidence: 0.86
- **[P2] Resolve aliases before the UUID fast path:** `src/gateway/session-utils.ts:668-669`
  - body: The UUID early return runs before main-key alias handling, so a configured `session.mainKey` that happens to be UUID-shaped no longer maps to the canonical main session key. It also leaves bare and prefixed UUID aliases as different canonical keys for exact-key paths like live session subscriptions.
  - confidence: 0.8
- **[P2] Send the system prompt once:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:722-723`
  - body: `buildRequestBody` sets `instructions` from `context.systemPrompt` while `convertResponsesMessages` already injects the same prompt into the input messages. That duplicates system instructions on every Codex proxy request, increasing tokens and potentially changing model behavior.
  - confidence: 0.88
