## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent owner for UUID rows:** `src/gateway/session-utils.ts:643-645`
  - body: Returning bare UUIDs here drops the per-agent store owner while `loadCombinedSessionStoreForGateway` merges per-agent stores. A UUID row from a non-default or previously default agent can then list as an unscoped key and later `sessions.get`/patch/delete resolve against the current default agent store, showing empty history or touching the wrong row after an upgrade/default-agent change.
  - confidence: 0.88
- **[P2] Normalize UUID links beyond main:** `ui/src/ui/app-settings.ts:51-52`
  - body: This only strips `agent:main:<uuid>`, but legacy Control UI links were generated with whichever agent was configured as default. A setup whose default agent was `ops` keeps `agent:ops:<uuid>` in settings/URLs, so the migration does not converge that Open WebUI chat back to the bare UUID identity.
  - confidence: 0.82
- **[P2] Resolve main-key aliases before UUID fast path:** `src/gateway/session-utils.ts:668-670`
  - body: The UUID fast path runs before the existing `session.mainKey` alias check. If an operator configured a UUID-shaped main key, `resolveSessionStoreKey` now returns the bare UUID instead of the canonical agent main key, splitting the documented main-session alias behavior.
  - confidence: 0.8
- **[P1] Use a header-capable WebSocket client:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:809`
  - body: `globalThis.WebSocket` is constructed with `{ headers }`, but the standard Node WebSocket constructor does not provide a header init contract. The Authorization and `chatgpt-account-id` headers built above can be dropped or rejected, causing every proxied Codex WebSocket request to fail auth or tenant routing.
  - confidence: 0.86
- **[P2] Avoid duplicating the system prompt:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:722-723`
  - body: `buildRequestBody` sends `instructions: context.systemPrompt` while `convertResponsesMessages` already injects the same system/developer message into `input`. That duplicates prompt content on every proxy call, increasing token usage and potentially changing behavior for long prompts.
  - confidence: 0.78
