## Summary

Keep open. The reported Open WebUI UUID session split is still source-reproducible on current main and the PR targets the right area, but the branch currently has blocking correctness issues, unrelated credentialed Codex proxy/auth/routing changes, and no inspectable real behavior proof.

## What This Changes

The PR preserves bare Open WebUI UUID session keys, normalizes Control UI chat URLs, threads agent context into transcript reads, and also adds embedded Codex proxy/auth, compaction-auth, and account-routed send changes.

## Best Possible Solution

Land a narrow revised PR or replacement that fixes UUID session identity through one shared alias/canonicalization contract, keeps transcript lookup agent-aware, and splits Codex proxy/auth/routing into separately reviewed work.

## Reproduction Assessment

Yes, at source level. Current main still prefixes bare UUID-like session keys through the default agent and applies Control UI URL session keys unchanged, matching the split-session path described by the linked Open WebUI report.

## Solution Assessment

No. The UUID-preservation direction is valid, but this PR is not the best mergeable form because it loses agent/alias semantics in edge cases and bundles unrelated credentialed proxy/auth/routing changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use a WebSocket client that can send headers:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:809`
  - body: `globalThis.WebSocket` follows the standard URL/protocols constructor, so passing `{ headers }` here does not reliably transmit `Authorization` or `chatgpt-account-id`. Proxied Codex requests that require those headers can fail auth or tenant routing unless this uses a WebSocket implementation with explicit header support.
  - confidence: 0.86
- **[P1] Preserve agent scope for UUID store rows:** `src/gateway/session-utils.ts:643-645`
  - body: Returning bare UUIDs from `canonicalizeSessionKeyForAgent` also affects per-agent store merging. A UUID row from another agent store can be listed as unscoped, so later get/patch/delete resolves through the current default agent and can miss or mutate the wrong history.
  - confidence: 0.87
- **[P2] Normalize UUID links for any default agent:** `ui/src/ui/app-settings.ts:51-52`
  - body: This only strips `agent:main:<uuid>`, but legacy UUID links were generated with whichever agent was configured as default. Non-main defaults keep the prefixed key and can continue the split identity this PR is trying to converge.
  - confidence: 0.86
- **[P2] Resolve main-key aliases before UUID fast paths:** `src/gateway/session-utils.ts:668-669`
  - body: The UUID early return runs before `session.mainKey` alias handling. If an operator configured a UUID-shaped main key, it now resolves to the bare UUID instead of the canonical main session key, splitting existing clients.
  - confidence: 0.8
- **[P2] Use one canonical key for UUID subscriptions:** `src/gateway/session-utils.ts:668-669`
  - body: Bare UUID and `agent:<id>:<uuid>` forms remain different canonical keys, while `sessions.messages.subscribe` uses the exact canonical key. A legacy client subscribed on the prefixed form can miss live messages emitted under the bare form.
  - confidence: 0.78
- **[P2] Send the system prompt only once:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:722-723`
  - body: `buildRequestBody` sets `instructions` from `context.systemPrompt`, while `convertResponsesMessages` already injects the same prompt into the input messages. That duplicates system instructions on every Codex proxy call.
  - confidence: 0.88
