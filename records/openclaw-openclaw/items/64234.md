## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add privacy to the runtime config schema:** `src/config/types.openclaw.ts:138`
  - body: The PR adds privacy only to the TypeScript config type. OpenClawSchema is strict and still lacks a top-level privacy key, so normal config validation rejects the documented settings before these controls can be enabled.
  - confidence: 0.95
- **[P1] Wire at-rest encryption into session persistence:** `src/privacy/types.ts:98-99`
  - body: privacy.atRest.enabled is documented as encrypting session JSONL files, but the new encryption helpers are only exercised by their module and tests. Active sessions still open the normal session file path, so enabling this setting does not protect transcripts at rest.
  - confidence: 0.9
- **[P1] Thread privacy config through embedded prompts:** `src/agents/system-prompt.ts:475-476`
  - body: buildAgentSystemPrompt only masks runtime fields and context files when privacyConfig is supplied, but the embedded messaging wrapper has no matching parameter and the runner call does not pass params.config?.privacy. Messaging channels therefore keep sending unmasked system prompts even when these settings are enabled.
  - confidence: 0.9
- **[P2] Use async key derivation for session encryption:** `src/privacy/session-encrypt.ts:95-96`
  - body: deriveKey calls pbkdf2Sync with the 210k-iteration default from async encrypt/decrypt APIs. Once wired into gateway or session flows, each operation can block the Node event loop, and directory encryption repeats that CPU-bound pause per file.
  - confidence: 0.86
- **[P2] Close the encrypted-file probe descriptor in a finally:** `src/privacy/session-encrypt.ts:140-143`
  - body: If fd.read() throws after fs.open() succeeds, the catch returns false without closing the file descriptor. A long-running gateway that probes many session files can leak descriptors under transient read errors.
  - confidence: 0.82
- **[P2] Mask repo paths that contain spaces:** `src/privacy/payload-redact.ts:202`
  - body: The repo-path masking regex captures only up to whitespace, so paths such as C:/Users/Alice Smith/workspace are partially rewritten and still leak the rest of the identifying path when maskRepoPath is enabled.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention because advertised privacy controls can be rejected, inactive, or only helper-level on production paths.

Concerns:

- **[high] Advertised privacy controls can be no-op or rejected:** `src/config/types.openclaw.ts:138`
  - body: The new config is not accepted by the strict runtime schema, embedded messaging prompts do not receive privacyConfig, and at-rest encryption is not wired into production session persistence. Users could enable documented protections while data still leaves or remains stored in the old form.
  - confidence: 0.92
- **[medium] Synchronous KDF can stall gateway execution:** `src/privacy/session-encrypt.ts:96`
  - body: Using pbkdf2Sync for a high-iteration KDF inside gateway-facing async APIs creates a denial-of-service and responsiveness risk once session encryption is integrated, especially for batch directory encryption.
  - confidence: 0.86
- **[medium] Runtime path masking can leak paths with spaces:** `src/privacy/payload-redact.ts:202`
  - body: The repo= masking regex stops at whitespace, so common home-directory or Windows paths containing spaces can still disclose identifying path fragments despite the privacy setting being enabled.
  - confidence: 0.84
- **[low] Encrypted-file probe can leak descriptors:** `src/privacy/session-encrypt.ts:140`
  - body: isEncryptedSessionFile does not close its descriptor in a finally, so read errors after a successful open can accumulate file descriptors in a long-running process.
  - confidence: 0.82
