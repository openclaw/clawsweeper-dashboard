Overall confidence: 0.93

Full review comments:

- **[P1] Add the privacy section to runtime config validation:** `src/config/types.openclaw.ts:138`
  - body: The PR adds `privacy` only to the TypeScript `OpenClawConfig` type. `OpenClawSchema` remains strict and has no `privacy` key, so an `openclaw.yml` using the documented `privacy.*` settings will be rejected as an unknown top-level field, making the feature unusable through normal configuration.
  - confidence: 0.95
- **[P1] Wire at-rest encryption into session persistence:** `src/privacy/types.ts:98-99`
  - body: `privacy.atRest.enabled` is documented as making OpenClaw encrypt session JSONL files, but the new `encryptSessionFile` APIs are only referenced by tests and the helper module. No production session writer calls them, so users can enable the setting and still leave transcripts plaintext on disk.
  - confidence: 0.96
- **[P1] Pass privacyConfig through embedded system prompts:** `src/agents/system-prompt.ts:476`
  - body: Adding `privacyConfig` to `buildAgentSystemPrompt` only protects callers that pass it. The messaging-channel path still goes through `buildEmbeddedSystemPrompt`, which calls the prompt builder without `privacyConfig`, so hostname/path masking and context-file redaction silently do not apply for embedded Discord/WhatsApp/etc. runs.
  - confidence: 0.93
- **[P1] Preserve the resolved CLI execution provider:** `src/auto-reply/reply/agent-runner-execution.ts:996`
  - body: This change passes the fallback `provider` into `runCliAgent` instead of the resolved `cliExecutionProvider`. When config or session runtime selection maps a provider to a CLI harness, the run will be invoked with the wrong provider and can bypass the intended harness/auth profile path.
  - confidence: 0.87
- **[P2] Use async PBKDF2 for session encryption:** `src/privacy/session-encrypt.ts:95-96`
  - body: `deriveKey` uses `pbkdf2Sync` even though the public encrypt/decrypt APIs are async. With 210k iterations, every encryption/decryption blocks the event loop, and directory encryption multiplies that stall across files, freezing gateway work while privacy maintenance runs.
  - confidence: 0.9
- **[P3] Close the encrypted-file descriptor in a finally block:** `src/privacy/session-encrypt.ts:140-143`
  - body: If `fd.read()` throws after `fs.open()` succeeds, the current `isEncryptedSessionFile` path skips `fd.close()` and then swallows the error. A gateway that checks many files can leak descriptors under repeated read failures.
  - confidence: 0.82
