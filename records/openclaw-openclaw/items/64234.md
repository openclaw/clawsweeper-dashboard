## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add privacy to the runtime config schema:** `src/config/types.openclaw.ts:138`
  - body: The branch adds `privacy` only to the TypeScript config type. `OpenClawSchema` is strict and still has no top-level `privacy` key, so normal config validation rejects the documented settings before any redaction, media blocking, or encryption controls can be enabled.
  - confidence: 0.95
- **[P1] Wire at-rest encryption into session persistence:** `src/privacy/types.ts:98-99`
  - body: `privacy.atRest.enabled` is documented as encrypting session JSONL files, but the new encryption code is only a helper with tests. Active sessions still use the existing transcript paths, so enabling this setting would not protect newly written transcripts at rest.
  - confidence: 0.9
- **[P1] Thread privacy config through embedded prompts:** `src/agents/system-prompt.ts:475-476`
  - body: `buildAgentSystemPrompt` only masks runtime fields and context files when `privacyConfig` is supplied, but the embedded messaging prompt wrapper has no matching parameter and the runner call does not pass `params.config?.privacy`. Messaging channels therefore keep sending unmasked system prompts even when these settings are enabled.
  - confidence: 0.9
- **[P2] Use async key derivation for session encryption:** `src/privacy/session-encrypt.ts:95-96`
  - body: `deriveKey` calls `pbkdf2Sync` with a high iteration count from async encryption APIs. Once this is wired into gateway/session flows, each encrypt/decrypt operation can block the Node event loop, and directory encryption repeats that CPU-bound pause per file.
  - confidence: 0.86
- **[P2] Close the encrypted-file probe descriptor in a finally:** `src/privacy/session-encrypt.ts:140-143`
  - body: If `fd.read()` throws after `fs.open()` succeeds, the catch returns false without closing the file descriptor. A long-running gateway probing many session files can leak descriptors under transient read errors.
  - confidence: 0.82
- **[P2] Mask repo paths that contain spaces:** `src/privacy/payload-redact.ts:202`
  - body: The repo-path masking regex captures only up to whitespace, so paths such as `C:/Users/Alice Smith/workspace` are partially rewritten and still leak identifying path fragments when `maskRepoPath` is enabled.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention because advertised privacy controls can be rejected, inactive, or only helper-level on production paths.

Concerns:

- **[high] Advertised privacy controls can be rejected or no-op:** `src/config/types.openclaw.ts:138`
  - body: The runtime schema does not accept the documented top-level privacy key, embedded messaging prompts do not receive privacyConfig, and session encryption is not wired into transcript persistence. Users could enable documented protections while data still leaves or remains stored in the old form.
  - confidence: 0.92
- **[high] At-rest encryption is not production storage yet:** `src/privacy/types.ts:98`
  - body: The new encryption module can encrypt an existing file, but active session writers are not integrated with it, so `privacy.atRest.enabled` does not provide the promised session-at-rest protection.
  - confidence: 0.9
- **[medium] Synchronous KDF can stall gateway execution:** `src/privacy/session-encrypt.ts:96`
  - body: Using `pbkdf2Sync` for a high-iteration KDF inside gateway-facing async helpers creates responsiveness and denial-of-service risk once session encryption is integrated, especially for batch directory encryption.
  - confidence: 0.86
- **[medium] Runtime path masking can leak paths with spaces:** `src/privacy/payload-redact.ts:202`
  - body: The repo masking regex stops at whitespace, so common home-directory or Windows paths containing spaces can still disclose identifying path fragments despite the privacy setting being enabled.
  - confidence: 0.84
- **[low] Encrypted-file probe can leak descriptors:** `src/privacy/session-encrypt.ts:140`
  - body: `isEncryptedSessionFile` does not close its descriptor in a finally, so read errors after a successful open can accumulate file descriptors in a long-running process.
  - confidence: 0.82
