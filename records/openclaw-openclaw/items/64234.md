## Summary

Keep open: this is a security-sensitive XL privacy feature PR, and source inspection still shows blocking correctness gaps plus overlap with the broader open privacy-filter PR. The useful direction is real, but the current branch should not merge as-is.

## What This Changes

The PR adds a new privacy config surface, PII redaction hooks for prompts/tool outputs/user messages/TTS, media attachment blocking, AES-256-GCM session-encryption helpers, tests, and a new plugin SDK privacy runtime export.

## Best Possible Solution

Converge on one maintainer-approved privacy architecture, then split any remaining useful pieces here into narrower schema, prompt, media-policy, and real session-storage follow-ups with production-path proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of the PR head shows strict config schema rejection, missing embedded prompt wiring, helper-only at-rest encryption, synchronous KDF use, descriptor leakage, and incomplete repo-path masking. Current main inspection confirms the proposed privacy feature is not already implemented.

## Solution Assessment

No. The privacy direction is valid, but this branch is not the best merge unit until maintainers choose the canonical architecture and the implementation is wired through production config, prompt, provider, and session paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add privacy to the runtime config schema:** `src/config/types.openclaw.ts:138`
  - body: The branch adds `privacy` only to the TypeScript config type. `OpenClawSchema` is strict and still has no top-level `privacy` key, so normal config validation rejects the documented settings before any redaction, media blocking, or encryption controls can be enabled.
  - confidence: 0.95
- **[P1] Wire at-rest encryption into session persistence:** `src/privacy/types.ts:98-99`
  - body: `privacy.atRest.enabled` is documented as encrypting session JSONL files, but the new encryption code is only a helper with tests. Active sessions still use the existing transcript paths, so enabling this setting would not protect newly written transcripts at rest.
  - confidence: 0.9
- **[P1] Thread privacy config through embedded prompts:** `src/agents/system-prompt.ts:475-476`
  - body: `buildAgentSystemPrompt` only masks runtime fields and context files when `privacyConfig` is supplied, but the embedded messaging prompt wrapper has no matching parameter and the runner call does not pass `params.config?.privacy`. Messaging channels therefore keep sending unmasked system prompts even when these settings are enabled.
  - confidence: 0.9
- **[P2] Use async key derivation for session encryption:** `src/privacy/session-encrypt.ts:95-96`
  - body: `deriveKey` calls `pbkdf2Sync` with a high iteration count from async encryption APIs. Once this is wired into gateway/session flows, each encrypt/decrypt operation can block the Node event loop, and directory encryption repeats that CPU-bound pause per file.
  - confidence: 0.86
- **[P2] Close the encrypted-file probe descriptor in a finally:** `src/privacy/session-encrypt.ts:140-143`
  - body: If `fd.read()` throws after `fs.open()` succeeds, the catch returns false without closing the file descriptor. A long-running gateway probing many session files can leak descriptors under transient read errors.
  - confidence: 0.82
- **[P2] Mask repo paths that contain spaces:** `src/privacy/payload-redact.ts:200-202`
  - body: The repo-path masking regex captures only up to whitespace, so paths such as `C:/Users/Alice Smith/workspace` are partially rewritten and still leak identifying path fragments when `maskRepoPath` is enabled.
  - confidence: 0.84
