## Summary

Keep open. Current main still has the stale Control UI dropdown path: fresh `sessions.list` data replaces the cached rows, but the chat session option resolver then re-adds the active session key even when that deleted direct session is absent from the Gateway response. The related implementation PR #48190 was closed unmerged, so there is no merged fix to treat as implemented.

## What This Changes

The issue reports that the Control UI sessions dropdown continues to show a deleted Discord session after the Gateway sessions store no longer contains it.

## Best Possible Solution

Treat fresh `sessions.list` rows as authoritative for normal direct sessions, normalize a missing active direct key to a valid fallback such as the main/default session, preserve intentional missing subagent and cron fallbacks, and add focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: set the Control UI active `sessionKey` to a deleted direct session, return a fresh `sessions.list` response without that key, and `resolveSessionOptionGroups` still adds that missing key to the dropdown.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Control UI session selection/dropdown path. Fetching fresh data alone is insufficient while the renderer reinserts a missing active direct key after the refresh.
