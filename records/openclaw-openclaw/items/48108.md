## Summary

Keep open. Current main still has the stale Control UI dropdown path: the Gateway/UI refresh flow exists after session deletion, but the chat session option resolver still re-adds the currently selected session key even when the fresh Gateway session rows no longer contain that deleted direct session. The direct fix attempt in #48190 is closed and unmerged.

## What This Changes

The issue asks for the Control UI sessions dropdown to stop showing deleted sessions once the Gateway sessions store no longer contains them.

## Best Possible Solution

Treat fresh `sessions.list` rows as authoritative for normal direct sessions, normalize a missing active direct key to a valid fallback such as the Gateway main/default session, preserve intentional missing subagent and cron fallbacks, and add regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to set the Control UI active `sessionKey` to a deleted direct session, return a fresh `sessions.list` response without that key, and observe `resolveSessionOptionGroups` still adding that missing key to the dropdown.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Control UI session selection/dropdown path; fetching fresh data alone is insufficient while the renderer reinserts a missing active direct key.
