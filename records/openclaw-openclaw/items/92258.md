## Summary

Current main does not implement Telegram Bot API 10.1 Rich Messages. The request is directionally valid, but its full scope needs a maintainer product/API decision: OpenClaw has portable presentation blocks and Telegram text/HTML draft streaming, not a general rich-document model, so directly mirroring Telegram’s full schema would create a channel-specific parallel content system.

## What This Changes

Add Telegram Bot API 10.1 rich-message sending, streamed rich-message drafts, thinking blocks, and related structured formatting support.

## Best Possible Solution

After the upstream schema and library support stabilize, define a maintainer-approved portable subset of structured content, extend the existing presentation contract for those semantics, and encode and fall back within the Telegram plugin while reusing the current draft lifecycle.

## Reproduction Assessment

Not applicable: this issue requests support for a newly released Telegram API and does not report broken existing OpenClaw behavior.

## Solution Assessment

No, implementing every Telegram rich-message class directly is not yet the best solution; the safer design is a stable portable subset with Telegram-local encoding, fallback, and reuse of the existing draft stream.
