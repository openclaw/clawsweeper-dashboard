## Summary

Keep open: current main still has the reported wrapper shape, with `wiki_lint` returning the raw lint result through tool `details` while the CLI uses the same lint core through a separate summary path. The issue is narrow, current, and not covered by the related wikilink-resolution reports.

## What This Changes

The issue asks for `wiki_lint` to return a bounded, transport-safe result with a relative `reportPath` so direct tool invocation succeeds like `openclaw wiki lint`.

## Best Possible Solution

Keep lint core behavior intact, and make only the `wiki_lint` agent tool return relative, bounded, JSON-safe details with focused regression coverage and a real direct-tool smoke proof.

## Reproduction Assessment

No, not from this read-only review: I did not run a bridge vault plus direct `wiki_lint` invocation. Source inspection does confirm current main still returns the raw lint result through the tool wrapper, matching the reported failure surface.

## Solution Assessment

Yes, the wrapper-only direction is the best fit: normalize the tool-facing `reportPath` and bound `details` without changing lint core, CLI, or gateway semantics.
