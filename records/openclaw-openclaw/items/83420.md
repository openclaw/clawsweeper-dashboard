## Summary

Keep open: current main still returns the raw `lintMemoryWikiVault()` result through the `wiki_lint` agent tool while the CLI consumes the same lint core through a separate rendering path, so the reported tool-wrapper bug remains actionable and is not covered by the related wikilink-resolution issues.

## What This Changes

The issue requests that the memory-wiki `wiki_lint` agent tool return successfully with stable, bounded tool output when CLI lint succeeds on the same vault.

## Best Possible Solution

Keep lint core behavior intact, and make only `createWikiLintTool` return relative, bounded, JSON-safe details with focused regression coverage and real direct-tool smoke proof.

## Reproduction Assessment

No, not by execution in this read-only review. Source inspection plus the reporter's live validation show current main still matches the failure surface: `wiki_lint` returns raw lint details with an absolute `reportPath`, while CLI lint uses a separate rendering path.

## Solution Assessment

Yes, the wrapper-only fix direction is the best fit: normalize the tool-facing `reportPath` and bound `details` in `createWikiLintTool` without changing lint core, CLI, or gateway contracts.
