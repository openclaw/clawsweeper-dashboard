## Summary

Keep open: current main still has the abandoned guarded-stream cleanup gap in `buildManagedResponse`, and the reporter plus Linux confirmation give credible real-world FD/RSS leak evidence. The native Anthropic parser reduces the original SDK-specific path, but it does not fix the shared wrapper’s no-EOF/no-cancel case.

## What This Changes

The issue requests fixing gateway resource leaks when streamed Anthropic/provider responses are partially read and then abandoned without EOF or `body.cancel()`.

## Best Possible Solution

Add exact-once abandoned-stream cleanup in `buildManagedResponse` with focused regression coverage, without changing provider identity, auth, proxy policy, or adding a config knob.

## Reproduction Assessment

Yes, at source level: return a streamed response through `buildGuardedModelFetch`, read part of `response.body`, then abandon it without EOF or `body.cancel()`. Current main has no path from that state to `finalize()`, though I did not run the long-duration socket-count repro.

## Solution Assessment

Yes. The narrow maintainable fix is a generic exact-once cleanup safety net in `buildManagedResponse`, because that wrapper owns `result.release`; consumer-only Anthropic parsing and abort handling do not cover every guarded stream consumer.
