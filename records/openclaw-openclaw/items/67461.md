## Summary

Keep open: current `main` still has the source-level cleanup gap described in the report, and the macOS report plus Linux confirmation match the guarded streamed response lifecycle. No open or merged OpenClaw PR appears to own the remaining fix.

## What This Changes

The issue reports leaked gateway sockets when streamed Anthropic API responses are abandoned before EOF or cancellation.

## Best Possible Solution

Add exact-once cleanup for abandoned partially consumed guarded streamed responses in `buildManagedResponse`, with regression coverage for the no-EOF/no-cancel path and no new config knob.

## Reproduction Assessment

Yes, at source level: return a streamed response through `buildGuardedModelFetch`, read part of `response.body`, then abandon it without EOF or `body.cancel()`. Current `main` has no path from that state to `finalize()`, though I did not run the long-duration socket-count reproduction.

## Solution Assessment

Yes. A generic fix in `buildManagedResponse` is the narrowest maintainable direction because that wrapper owns `result.release` and covers Anthropic plus other guarded streamed provider consumers.

Fix the abandoned guarded streamed response cleanup bug reported in https://github.com/openclaw/openclaw/issues/67461. In `src/agents/provider-transport-fetch.ts`, make `buildManagedResponse` call the guarded fetch `release` callback exactly once when a wrapped response body is partially read and then abandoned without EOF or explicit `body.cancel()`. Keep the fix generic to provider guarded fetches; do not add a config option, provider-specific policy, or startup migration. Add focused regression coverage in `src/agents/provider-transport-fetch.test.ts` for a partial body read followed by abandonment/no EOF/no cancel, plus existing EOF/error/cancel exact-once behavior as needed. Update `CHANGELOG.md` because this is a user-visible gateway reliability fix.
