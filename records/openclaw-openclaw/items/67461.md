## Summary

Keep open: current main still has the abandoned streamed-response cleanup gap, and the issue has credible macOS and Linux long-running gateway evidence plus a matching external implementation reference. No OpenClaw duplicate or merged fix appears to own the remaining work.

## What This Changes

The issue reports leaked guarded-fetch sockets when streamed Anthropic responses are partially consumed and abandoned without reaching EOF or `cancel()`.

## Best Possible Solution

Add shared provider-transport abandonment cleanup so guarded fetch resources and local-service leases are released exactly once for partially consumed streamed responses, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: return a streamed response through `buildGuardedModelFetch`, read one chunk from `response.body`, then drop it without calling `body.cancel()`; current main has no path from that state to `finalize()`. I did not run the multi-hour socket-count reproduction.

## Solution Assessment

Yes. A generic fix in `buildManagedResponse` is the narrowest maintainable direction because that wrapper owns `result.release()` and covers Anthropic plus other guarded streamed provider consumers.

## Repair Work Prompt

Fix the streamed provider guarded-fetch leak reported in https://github.com/openclaw/openclaw/issues/67461. Keep the fix scoped to the shared provider transport lifecycle: `buildManagedResponse` must release guarded fetch resources and any local-service lease exactly once when a streamed response is partially consumed and then abandoned without EOF or `body.cancel()`. Prefer a deterministic cleanup path where possible; if using FinalizationRegistry as a safety net, keep it module-scoped and prove it does not double-release. Add regression coverage for partial-read/no-cancel behavior and preserve existing EOF, error, explicit cancel, timeout refresh, SSE sanitizer, and Anthropic abort behavior. Do not add a config flag or move provider-specific policy into core. Add a changelog entry if the fix is user-visible reliability work.
