## Summary

Keep open. Current main still loads configured channel plugins before `messageCommand`, so a Telegram CLI send can still initialize Telegram runtime locally before the gateway-owned `message.action` path can delegate; the paired fix PR is open and unmerged.

## What This Changes

Make `openclaw message send --channel telegram ...` delegate to the live gateway message-action path without initializing Telegram runtime in the local CLI process or hanging.

## Best Possible Solution

Review and land PR #75478 or an equivalent narrow fix that skips local Telegram plugin loading before gateway dispatch while preserving existing message-send normalization and media behavior.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction path: the CLI still loads configured channel plugins before `messageCommand`, while Telegram actions are gateway-owned and generic delegation depends on loaded plugin state.

## Solution Assessment

Yes in direction. The maintainable fix is to keep Telegram ownership in the plugin and route eligible live CLI sends to the gateway before local Telegram runtime loading, with regression coverage for the existing CLI send contract.
