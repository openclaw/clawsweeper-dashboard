## Summary

Keep open. Current main has fixed the Telegram wrapper execution-mode hook, but the CLI path still eagerly loads the configured channel plugin registry before dispatch and gateway message-action forwarding still goes through the least-privilege helper; the paired implementation PR remains open and unmerged.

## What This Changes

Make `openclaw message send --channel telegram ...` delegate to the live gateway message-action path without initializing Telegram runtime in the local CLI process or hanging.

## Best Possible Solution

Review and land PR #75478 or an equivalent narrow fix that skips local Telegram plugin loading before gateway dispatch while preserving the wrapper execution-mode fix already on main.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction path: the CLI still loads configured channel plugins before `messageCommand`, while gateway forwarding depends on plugin execution-mode resolution; I did not perform a live Telegram send in this read-only review.

## Solution Assessment

Yes in direction. The narrow maintainable solution is to keep Telegram ownership in the plugin, preserve gateway execution mode through the wrapper, and add the CLI gateway fast path or equivalent normal CLI gateway delegation before local plugin registry loading.
