## Summary

Keep open. Current main still matches the reported Telegram routing gap, and the linked open PR is an implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that Telegram CLI message sends can hang because `openclaw message send --channel telegram ...` may load/run Telegram locally instead of delegating to the live gateway message-action path.

## Best Possible Solution

Land PR #75478 or an equivalent narrow fix that preserves Telegram gateway execution mode through the bundled wrapper, avoids local Telegram runtime loading for gateway-owned CLI sends, uses the normal CLI gateway scope path, and keeps regression coverage in the Telegram action contract and CLI helper tests.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the inner Telegram adapter says gateway, the public wrapper drops that hook, the runner requires that hook to delegate, and the CLI helper loads configured channel plugins before dispatch.

## Solution Assessment

Yes. The maintainable direction is to keep Telegram-specific routing behavior in the Telegram plugin wrapper while adding a focused CLI gateway fast path for gateway-owned sends; the existing open PR should receive normal review before merge.
