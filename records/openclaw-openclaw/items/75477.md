## Summary

Keep open. Current main still matches the reported routing problem: Telegram's inner action adapter declares gateway execution, but the public bundled Telegram plugin wrapper does not forward that execution-mode hook, and the CLI message helper still loads the configured channel plugin registry before dispatch. The linked open PR #75478 is an implementation candidate with explicit closing linkage, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Fix `openclaw message send --channel telegram ...` so Telegram sends delegate to the live gateway `message.action` path and return without initializing the Telegram runtime in the local CLI process.

## Best Possible Solution

Review and land #75478, or an equivalent narrow fix, so the Telegram bundled wrapper preserves gateway execution mode, the CLI avoids eager Telegram runtime loading for gateway-owned sends, and message-action gateway calls use the normal CLI gateway scope path with regression coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: the runner requires a gateway execution-mode hook, the public Telegram plugin wrapper omits that hook, and the CLI helper eagerly loads channel plugins before dispatch.

## Solution Assessment

Yes. Delegating Telegram sends through the gateway-owned `message.action` path while keeping Telegram-specific behavior in the Telegram plugin is the narrow maintainable direction; the open PR still needs normal review before merge.
