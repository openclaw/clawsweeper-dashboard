## Summary

Keep open. Current main still matches the reported Telegram routing problem, and the linked open PR #75478 uses closing syntax for this issue, so this issue should remain open until that implementation lands or is replaced.

## What This Changes

The issue reports that `openclaw message send --channel telegram ...` can hang because the CLI can load Telegram runtime locally instead of delegating the send to the live Gateway message-action path.

## Best Possible Solution

Land #75478 or an equivalent narrow fix that keeps Telegram-specific execution-mode behavior in the Telegram plugin, avoids eager Telegram runtime loading for Gateway-owned CLI sends, and uses the normal CLI Gateway call path with regression coverage.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the inner Telegram adapter is gateway-owned, the public wrapper omits that hook, and the CLI helper loads channel plugins before dispatch.

## Solution Assessment

Yes. Delegating Telegram CLI sends through the Gateway-owned `message.action` path while preserving the plugin-owned execution-mode contract is the narrow maintainable direction; the open PR still needs normal review before merge.
