## Summary

Keep open. Current main still matches the reported source-level routing problem, and PR #75478 is open and explicitly linked as the implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks for Telegram CLI sends to delegate directly to the live gateway message-action path, return JSON instead of hanging, and avoid initializing Telegram runtime in the local CLI process.

## Best Possible Solution

Land PR #75478 or an equivalent narrow replacement that preserves Telegram gateway execution mode through the bundled wrapper, routes Telegram CLI sends through a gateway fast path before local plugin loading, uses the normal CLI gateway helper semantics, and keeps regression coverage on those contracts.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the inner Telegram adapter is gateway-owned, the public wrapper omits that hook, delegation requires the hook, and the CLI helper loads configured channel plugins before dispatch.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Telegram wrapper, CLI message dispatch path, and message-action gateway helper boundary; that preserves the generic gateway execution-mode contract while keeping Telegram-specific behavior plugin-owned.
