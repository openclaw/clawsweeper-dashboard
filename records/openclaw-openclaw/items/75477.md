## Summary

Keep open. Current main still matches the reported Telegram routing problem, and open PR #75478 is explicitly linked as the implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Fix the Telegram `openclaw message send` path so gateway-owned Telegram actions delegate to the live gateway without loading Telegram runtime locally and use the normal CLI gateway call path.

## Best Possible Solution

Review and land #75478, or an equivalent narrow fix, so the Telegram wrapper preserves gateway execution mode, the CLI avoids local Telegram runtime loading for gateway-owned sends, and gateway message actions use the normal CLI helper semantics with regression coverage.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the Telegram inner adapter declares gateway execution, the public wrapper drops that hook, and the CLI helper loads channel plugins before dispatching.

## Solution Assessment

Yes for the proposed direction. Keeping Telegram-specific behavior in the Telegram plugin while routing gateway-owned sends through `message.action` is the narrow maintainable fix; the linked PR still needs normal review before merge.
