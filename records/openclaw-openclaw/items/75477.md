## Summary

Keep open. Current main still matches the reported Telegram routing problem, and the provided context shows open, unmerged PR #75478 with explicit closing linkage, so this issue should remain open until that PR lands or is replaced.

## What This Changes

`openclaw message send --channel telegram ...` should delegate to the live gateway `message.action` path without initializing the local Telegram runtime and should return a confirmed result instead of hanging.

## Best Possible Solution

Review and land #75478, or an equivalent narrow replacement, so the Telegram wrapper preserves gateway execution mode, the CLI avoids local Telegram runtime loading for gateway-owned sends, and the message-action gateway path uses the normal CLI gateway semantics with regression coverage.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the inner Telegram adapter is gateway-owned, the public wrapper omits that hook, the runner only delegates when the hook is visible, and the CLI helper loads channel plugins before dispatch.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the Telegram wrapper, CLI message dispatch fast path, and message-action gateway helper boundary; that keeps Telegram-specific behavior plugin-owned while preserving the generic gateway execution-mode contract.
