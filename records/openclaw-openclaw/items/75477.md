## Summary

Keep open. Current main still matches the reported routing bug: the public Telegram plugin wrapper drops the gateway execution-mode hook, the CLI message helper still loads the configured channel plugin registry before dispatch, and the message-action gateway helper still calls the least-privilege path. PR #75478 is open and explicitly linked as the implementation candidate, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Route Telegram CLI message sends through the live gateway without local Telegram runtime initialization, while preserving gateway-owned message-action execution mode and normal CLI gateway scope behavior.

## Best Possible Solution

Land PR #75478 or an equivalent narrow replacement that forwards Telegram gateway execution mode through the wrapper, dispatches targeted Telegram CLI sends to message.action before local plugin registry loading, uses normal CLI gateway helper semantics, and keeps focused regression coverage for those contracts.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the inner Telegram adapter is gateway-owned, the public wrapper hides that hook, delegation requires the hook, and the CLI helper loads channel plugins before dispatch.

## Solution Assessment

Yes. The linked fix direction is the narrow maintainable solution because it preserves the generic gateway execution-mode contract while keeping Telegram-specific behavior in the Telegram plugin and only adding a CLI fast path for the reported dispatch case.
