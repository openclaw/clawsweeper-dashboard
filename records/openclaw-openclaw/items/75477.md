## Summary

Keep open. Current main has partially moved in the right direction by exposing Telegram's gateway execution-mode hook, but it still matches the remaining reported routing problem: the CLI helper loads the message plugin registry before dispatch, and gateway-owned message actions still call the least-privilege gateway helper. PR #75478 is open and explicitly linked as the implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Request that `openclaw message send --channel telegram ...` delegate to the live gateway `message.action` path without initializing Telegram runtime locally, while preserving gateway execution mode and using normal CLI gateway scopes.

## Best Possible Solution

Review and land PR #75478, or an equivalent narrow replacement, that keeps Telegram-owned behavior inside the Telegram plugin, dispatches targeted Telegram CLI sends through `message.action` before local registry/runtime loading, switches this CLI gateway path to normal CLI gateway helper semantics, and preserves focused regression coverage for the wrapper, CLI helper, and outbound runner contracts.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the CLI helper still loads the plugin registry before `messageCommand`, and the outbound runner still sends gateway-owned actions through `callGatewayLeastPrivilege`; the linked report adds production confirmation for the stuck Telegram send.

## Solution Assessment

Yes. The linked fix direction is the narrow maintainable path because it preserves the generic gateway execution-mode contract, keeps Telegram-specific ownership in the Telegram plugin, and limits the CLI change to the reported targeted Telegram dispatch case.
