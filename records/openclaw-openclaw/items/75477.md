## Summary

Keep open. Current main still matches the reported Telegram CLI routing problem, and the issue is explicitly paired with open, unmerged PR #75478, so the issue should remain open until that implementation lands or is replaced.

## What This Changes

The issue requests that `openclaw message send --channel telegram ...` delegate through the live gateway message-action path without initializing Telegram runtime locally or hanging.

## Best Possible Solution

Review and land PR #75478, or an equivalent narrow replacement, so the Telegram wrapper preserves gateway execution mode, the CLI avoids local Telegram runtime loading for gateway-owned sends, and the gateway call uses the normal CLI helper path with regression coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: the inner Telegram adapter is gateway-owned, the public wrapper omits that hook, the runner only delegates when the hook is visible, and the CLI helper loads channel plugins before dispatch.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Telegram wrapper, CLI message dispatch path, and message-action gateway helper boundary; keeping this issue open while PR #75478 is reviewed is the right workflow.
