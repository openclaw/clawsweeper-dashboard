## Summary

Keep open. Current main still matches the reported Telegram CLI routing problem, and open PR #75478 explicitly targets this issue, so the issue should remain open until that PR is reviewed, landed, or replaced.

## What This Changes

Ensure `openclaw message send --channel telegram` delegates to the live gateway message-action path without initializing Telegram runtime locally, while preserving the correct gateway execution-mode and CLI gateway-call behavior.

## Best Possible Solution

Land #75478 or an equivalent narrow fix that forwards Telegram gateway execution mode through the bundled wrapper, avoids eager local Telegram registry/runtime loading for gateway-owned CLI sends, uses the intended CLI gateway-call boundary, and keeps the regression tests close to those seams.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: the inner Telegram adapter is gateway-owned, the exported plugin wrapper drops that hook, the runner only delegates when the hook is visible, and the CLI helper loads configured channel plugins before dispatch.

## Solution Assessment

Yes. The narrow maintainable direction is to fix the Telegram wrapper, CLI message dispatch path, and message-action gateway helper boundary rather than moving Telegram-specific runtime behavior into core.
