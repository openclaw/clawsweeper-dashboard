## Summary

Keep open. Current main still matches the reported Telegram gateway-delegation bug, and the paired implementation PR is open/unmerged, so this issue should remain open until that fix or an equivalent lands.

## What This Changes

Ensure `openclaw message send --channel telegram ...` delegates to the running gateway and returns JSON without loading Telegram runtime in the local CLI process.

## Best Possible Solution

Review and land PR #75478, or an equivalent narrow fix that preserves Telegram's gateway execution mode through the wrapper and gives CLI sends a gateway-first path.

## Reproduction Assessment

Yes. Current main provides a high-confidence source-level reproduction path: `message send` always calls `ensurePluginRegistryLoaded(...)`, while the Telegram public wrapper drops the inner `resolveExecutionMode: () => "gateway"` hook that the runner requires before delegating.

## Solution Assessment

Yes in direction. The narrow maintainable path is plugin-owned execution-mode preservation plus CLI gateway dispatch via the existing gateway call contract, with regression coverage around the wrapper and CLI helper behavior.
