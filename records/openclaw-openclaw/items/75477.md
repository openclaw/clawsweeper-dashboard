## Summary

Keep open. Current main still matches the reported routing problem, and the provided GitHub context shows open, unmerged PR #75478 with explicit closing linkage for this issue, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks for Telegram CLI message sends to delegate through the live gateway without initializing Telegram runtime in the CLI process, and for message-action gateway calls to use normal CLI gateway behavior.

## Best Possible Solution

Review and land #75478, or an equivalent narrow replacement, so the Telegram wrapper preserves gateway execution mode, the CLI avoids eager local Telegram runtime loading for gateway-owned sends, and message-action gateway calls use the CLI gateway helper path with regression coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: the inner Telegram adapter is gateway-owned, the public wrapper omits that hook, the runner only delegates when the hook is visible, and the CLI helper loads channel plugins before dispatch.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Telegram wrapper, CLI message dispatch path, and message-action gateway helper boundary; that keeps Telegram-specific behavior plugin-owned while preserving the generic gateway execution-mode contract.
