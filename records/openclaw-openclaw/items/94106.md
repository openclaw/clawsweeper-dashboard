## Label Justifications

- `P1`: The PR changes a default secrets migration path where mistakes can silently remove or preserve provider/plugin credentials.

## Summary

Keep open: the provider same-value over-scrub fix is pointed at the right code path, but the latest patch appears to regress supported plugin/config migrations to file or exec SecretRefs by no longer scrubbing their known `.env` line.

## What This Changes

The PR narrows `secrets apply` `.env` scrubbing to keys derived from migrated provider/auth-profile targets or known env-ref target ids, and adds collision regression tests.

## Best Possible Solution

Keep the same-value provider collision protection, but preserve documented cleanup of each migrated target's own known credential env vars for file, exec, and env SecretRef destinations.

## Reproduction Assessment

Yes. Source inspection shows current main scrubs migrated values from all known secret env keys, while the PR skips plugin/config known env keys when the destination ref is `file` or `exec`; I did not run tests because this review is read-only.

## Solution Assessment

No. The call-site fix is the right shape for the provider collision, but the allow-list must also preserve known env cleanup for supported non-env SecretRef destinations.
