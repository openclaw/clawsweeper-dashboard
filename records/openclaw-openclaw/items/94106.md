## Summary

Keep this PR open: the central key/value scrub-map direction is good and well evidenced, but the latest guard is too narrow and now skips documented provider-indexed API-key targets that current main would scrub from `.env`.

## What This Changes

The PR changes `secrets apply` to scrub `.env` lines by env-key-to-migrated-value ownership instead of matching every migrated value against all known secret env keys, and adds collision regression coverage.

## Best Possible Solution

Keep the key/value ownership model, but expand env-ref ownership to provider-indexed API-key credential targets while continuing to exclude provider metadata such as headers, request auth, and TLS targets.

Yes. The original collision is demonstrated by the PR's real CLI before/after proof, and the remaining review finding is source-reproducible from the PR guard plus the current target registry.

## Solution Assessment

No. The env-key-to-value map is the right boundary, but the ownership predicate must also cover provider-indexed API-key credential targets before this is the best fix.
