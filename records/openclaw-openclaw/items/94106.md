## Summary

Keep open: the PR fixes the reported provider same-value over-scrub, but the current patch still misses a real cleanup path for known plugin/config secrets migrated to non-env SecretRefs.

## What This Changes

The PR changes `secrets apply` to derive `.env` scrub keys from migrated provider/auth-profile/plugin targets and adds regression coverage for same-value collisions.

## Best Possible Solution

Keep the same-value collision fix, but derive non-provider plugin/config scrub keys from target identity or registry metadata so env/file/exec migrations of known secret targets scrub their own known `.env` line without returning to global value-only deletion.

## Reproduction Assessment

Yes. Source inspection shows current `main` scrubs migrated plaintext values against all known secret env keys, while this PR only includes plugin/config keys when the new ref is env-backed; `secrets configure` can create file-backed refs for those same targets.

## Solution Assessment

No. The provider/auth-profile narrowing is the right fix shape for the reported collision, but the plugin/config branch is too narrow and should handle known non-provider targets independently of the new ref source.
