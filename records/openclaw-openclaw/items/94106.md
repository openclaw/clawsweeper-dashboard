## Label Justifications

- `P1`: The PR handles a default secrets migration path that can silently remove provider credentials and break auth on the next run.
- `merge-risk: 🚨 compatibility`: Changing `.env` deletion scope during migration can affect existing upgrade and credential cleanup behavior.
- `merge-risk: 🚨 auth-provider`: The affected path controls provider/plugin credential env vars and can leave provider auth unresolved if the allow-list is wrong.

## Summary

Keep open: the current PR head still has a blocking auth-compatibility defect because it expands provider env scrubbing for any provider-indexed target, including non-auth targets such as provider headers and request auth.

## What This Changes

The PR changes `src/secrets/apply.ts` to derive the `.env` scrub allow-list from migrated target provider/env-ref metadata instead of all known secret env vars, and adds regression coverage for several collision cases.

## Best Possible Solution

Gate provider env-var expansion to actual provider auth credential targets and auth-profile credential targets, while keeping the known env-ref branch for plugin/config targets and the non-provider env-ref preservation case.

## Reproduction Assessment

Yes. Source inspection shows a concrete PR-head path: a `models.providers.openai.headers.*` plan target resolves `providerId: openai`, the PR adds `OPENAI_API_KEY` to the allow-list, and `scrubEnvRaw` removes any matching migrated value from `.env`.

## Solution Assessment

No. The proposed boundary is close, but the current implementation lacks the credential-target gate needed to keep provider-owned non-auth migrations from deleting provider auth env vars.
