## Summary

Keep this PR open for maintainer/security review before merge. The patch looks like the right code-path fix and I found no blocking line-level correctness findings, but it intentionally changes the default one-way `.env` scrub boundary for SecretRef migration and the current head still has unrelated failing required Node lanes.

## What This Changes

The PR narrows `secrets apply` `.env` scrubbing to migrated provider/auth-profile/plugin env keys and adds same-value collision regression coverage.

## Best Possible Solution

Land the scoped scrub after maintainers accept the registered-key/provider-auth boundary and required checks are green; keep broader service-env cleanup in the separate discussion at https://github.com/openclaw/openclaw/issues/92522.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main deletes `.env` lines by migrated value against all known secret env keys. I did not rerun the CLI locally, but the PR body includes before/after terminal output from real on-disk apply scenarios.

## Solution Assessment

Yes, this is the best observed fix shape: derive the `.env` allow-list at the apply call site that already has resolved targets, and use existing registry/provider-env metadata rather than a broad value-only scrub. The only safer alternative would be a larger provenance model for custom env keys if maintainers want that policy.
