## Summary

Keep open. The current head fixes the earlier plugin/config scrub gap, but the new env-ref-id fallback is too broad and can delete the `.env` variable that a newly written non-provider SecretRef depends on.

## What This Changes

The PR scopes `secrets apply` `.env` scrubbing to migrated target env keys and adds shared-value collision, auth-profile, and Firecrawl plugin regression coverage.

## Best Possible Solution

Keep scoped provider/plugin scrubbing, but do not add arbitrary env SecretRef ids to the `.env` scrub allow-list; cover both Firecrawl-positive and gateway-auth-negative cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: migrate a non-provider target such as `gateway.auth.token` to an env SecretRef whose `.env` line has the same plaintext value, and the PR allow-list makes that env line eligible for deletion.

## Solution Assessment

No. Scoping the scrub at apply time is the right boundary for the shared-value provider collision, but the current env-ref-id fallback is too broad and should be constrained before merge.
