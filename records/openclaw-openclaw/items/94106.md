## Summary

Keep open: this fixes the reported provider credential over-scrub cases, but the new allow-list can now skip `.env` cleanup for supported plugin/config targets when they are migrated to file or exec SecretRefs instead of env refs.

## What This Changes

The PR narrows `openclaw secrets apply` `.env` scrubbing to migrated provider/plugin credential env keys and adds collision regression tests under `src/secrets/`.

## Best Possible Solution

Keep the scoped scrub direction, but add target-owned allow-list coverage and regression proof for file/exec-backed plugin/config credential migrations without returning to global value-only scrubbing.

## Reproduction Assessment

Yes, source-reproducible: on the PR head a file or exec SecretRef has no `envRefId`, and only provider-auth entries set `migratesProviderAuth`, so a Firecrawl/plugin config migration to a file ref leaves the scrub allow-list empty. I did not run a local repro because this review is read-only.

## Solution Assessment

No, not yet. The call-site/registry-backed direction is the right maintainable layer, but the current implementation is too narrow for non-env SecretRef backends that are already part of the supported credential surface.
