## Summary

Keep this PR open for maintainer review: current main still has the broad value-based `.env` scrub, while the PR provides a focused source fix, regression coverage, and sufficient real behavior proof. I found no concrete patch defect, but the credential cleanup boundary is compatibility/security-sensitive and should remain an explicit maintainer acceptance point before merge.

## What This Changes

The PR narrows SecretRef migration `.env` scrubbing to env keys associated with migrated provider/auth-profile/plugin credential targets and adds collision regression coverage.

## Best Possible Solution

Land the scoped scrub after maintainers accept the registered-key/provider-auth boundary and exact-head checks are green; keep broader service-env cleanup in the separate discussion at https://github.com/openclaw/openclaw/issues/92522.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main deletes `.env` lines by migrated value against all known secret env keys. I did not rerun the CLI locally in this read-only review, but the PR body includes before/after terminal output from real on-disk apply scenarios.

## Solution Assessment

Yes, this is the best observed fix shape: derive the `.env` allow-list at the apply call site that already has resolved targets, and use existing registry/provider-env metadata rather than broad value-only scrubbing. A larger provenance model for custom env keys is a future policy choice, not required for this bounded fix.
