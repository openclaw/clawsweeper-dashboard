## Summary

Keep open: the PR fixes the shared-value over-scrub path for the covered provider/auth-profile/env-ref cases, but the latest branch appears to regress current cleanup for non-provider known-secret targets migrated to file or exec SecretRefs, leaving stale plaintext `.env` credentials behind.

## What This Changes

The PR changes `secrets apply` `.env` scrubbing to pair env keys with migrated plaintext values by provider or target ownership, and adds collision regression coverage.

## Best Possible Solution

Keep the per-target key/value ownership model, but add a narrow way for non-provider known-secret targets to scrub their own stale env key when migrated to file or exec refs, with regression coverage that does not restore global value matching.

## Solution Assessment

No. The per-target value ownership is the right direction, but the latest implementation is still too narrow because non-provider known-secret targets only scrub their `.env` key when the new ref is also an env ref.
