## Label Justifications

- `P1`: The PR handles a default secrets migration path that can silently remove provider credentials and break auth on the next run.
- `merge-risk: 🚨 compatibility`: Changing `.env` deletion scope during migration can affect existing upgrade and credential cleanup behavior.
- `merge-risk: 🚨 auth-provider`: The affected path controls provider/plugin credential env vars and can leave provider auth either unresolved or over-preserved if the allow-list is wrong.

## Summary

Keep open: the reported cross-provider `.env` collision is well covered, but the provider-wide allow-list still applies to every provider-indexed secret target, so a same-value provider header or request-auth migration can delete a non-migrated provider auth env var.

## What This Changes

The PR narrows `openclaw secrets apply` `.env` scrubbing to env keys associated with migrated secret targets and adds regression coverage for provider collisions, auth-profile targets, plugin/config targets, and non-provider env refs.

## Best Possible Solution

Gate provider env-var scrubbing to provider credential/auth-profile targets and rely on known env-ref ids for plugin/config targets, with regression coverage preserving provider auth env during a same-value provider header or request-auth migration.

## Reproduction Assessment

Yes. Current-main source shows the reported global `.env` scrub path, and PR-head source shows the remaining same-provider non-auth target collision without needing a full local write run.

## Solution Assessment

No. The proposed direction is close, but the provider-wide allow-list should be narrowed to credential/auth-profile targets so unrelated provider-owned secret targets cannot delete provider auth env vars.
