## Summary

Keep open: the per-target/keyed scrub direction fixes the shared-value over-scrub class, but this branch still drops shipped cleanup for non-provider known-secret targets migrated to file or exec SecretRefs, leaving plaintext behind in `.env`.

## What This Changes

This PR changes `secrets apply` `.env` scrubbing from a global migrated-value match to provider/target-owned keyed scrub maps and adds collision regression tests.

## Best Possible Solution

Land after the keyed scrub map also preserves known non-provider/plugin `.env` cleanup for file and exec SecretRef migrations, with regression coverage for both over-scrub and under-scrub cases.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current `main` scrubs migrated values against all known secret env keys, while this branch only assigns non-provider env keys when the new ref is `source: "env"`. I did not run tests because this review is read-only.

## Solution Assessment

No. The keyed target/value direction is the right boundary, but this implementation narrows too far; the safer fix is to keep per-target ownership while adding registry-backed env-key ownership for known non-provider file/exec migrations.
