## Label Justifications

- `P1`: The PR affects default secrets migration behavior that can delete, preserve, or leave behind provider credentials in real auth workflows.
- `merge-risk: 🚨 auth-provider`: The patch changes how SecretRef migrations clean provider/plugin credential env vars during auth cleanup.

## Summary

Keep open: the PR fixes the shared-value collision for provider targets, auth-profile targets, and env-backed plugin/config targets, but it still regresses `.env` cleanup for plugin/config targets migrated to file or exec SecretRefs because those targets lack provider identity and the new allow-list only adds `target.ref.id` for env refs.

## What This Changes

The PR narrows `secrets apply` `.env` scrubbing to env keys derived from migrated targets and adds regression tests for shared-value provider, auth-profile, and env-backed Firecrawl plugin/config migrations.

## Best Possible Solution

Keep the scoped scrub, but derive scrub-eligible env keys for every migrated target from provider or target metadata independent of the new SecretRef source, and cover env/file/exec plugin/config cases.

## Reproduction Assessment

Yes. The PR body supplies real CLI before/after proof for the original shared-value collision, and source inspection shows the remaining file/exec plugin/config path can reach `scrubEnvFiles` with migrated values but an empty allow-list.

## Solution Assessment

No. Scoping `.env` scrub at apply time is the right layer, but using env-ref id as the only no-provider plugin/config source leaves file/exec SecretRef migrations under-scrubbed.
