- `P1`: The PR concerns default secrets migration behavior that can incorrectly remove or retain provider credentials and affect real auth-provider workflows.

## Summary

Keep open: this is a focused fix for a real secrets-apply over-scrub bug, but the current patch is not merge-ready because the new provider-scoped env allow-list misses auth-profile targets and can leave migrated plaintext credentials in `.env`. The submitted proof is useful regression evidence, but it is still Vitest/mock-harness proof rather than a real `openclaw secrets apply` run from an external contributor setup.

## What This Changes

The PR narrows `secrets apply` `.env` scrubbing from all known secret env vars to env vars for migrated providers and adds a shared-value collision regression test.

## Best Possible Solution

Carry provider identity through every target kind that contributes `scrubbedValues`, including auth-profile targets, then verify both model-provider collision protection and auth-profile `.env` cleanup with focused tests plus real apply/CLI proof.

No. Scoping `.env` scrub at the apply call site is the right boundary for model-provider migrations, but the current implementation is incomplete because auth-profile migrations also add scrubbed values and need provider-scoped keys.
