## Summary

Keep this PR open: it fixes the primary same-value provider credential collision, but the narrowed `.env` allow-list introduces a plugin/config under-scrub path for file or exec SecretRef migrations.

## What This Changes

The branch changes `secrets apply` to derive `.env` scrub keys from migrated targets instead of the global known-key set and adds collision regression tests.

## Best Possible Solution

Keep the provider/auth-profile scoped cleanup, but derive known plugin/config env keys for migrated targets independently of the new SecretRef source so env, file, and exec migrations scrub their own known `.env` plaintext without deleting unrelated same-value credentials.

## Reproduction Assessment

Yes. Source inspection shows current main removes migrated values from all known `.env` secret keys, while the PR only keeps plugin/config keys in the allow-list when the replacement SecretRef itself is env-backed.

## Solution Assessment

No. Scoping by migrated provider/auth targets is the right direction, but the implementation is too narrow for plugin/config targets migrated to file or exec SecretRefs.
