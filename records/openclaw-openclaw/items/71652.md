## Summary

Keep open: current main still lacks the local `tsgo` pressure guard, and the PR is useful, but the latest diff still samples host pressure before waiting on the shared heavy-check lock, so a queued run can start from stale pressure data.

## What This Changes

The PR adds a local `tsgo` host-pressure guard with memory/temp thresholds, worktree temp-dir routing, a force override, and focused script tests.

## Best Possible Solution

Land the guard in the existing wrapper/runtime seam with pressure sampled immediately after acquiring the lock and before spawning `tsgo`, plus focused regression coverage; leave exec-tool enforcement to the linked exec work.

## Reproduction Assessment

Yes, source-reproducible: hold the local heavy-check lock, let the wrapper sample pressure while healthy, then create memory/temp pressure before releasing the lock; the PR will spawn `tsgo` without rechecking. I did not execute that heavy queued-lock scenario in this read-only review.

## Solution Assessment

No. The wrapper/runtime direction and effective-temp-directory check are maintainable, but the pressure sample needs to move to the actual start point after lock acquisition.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Recheck pressure after acquiring the heavy-check lock:** `scripts/run-tsgo.mjs:31-34`
  - body: The wrapper computes `pressureGuardError` before `acquireLocalHeavyCheckLockSync`. If this process waits behind another heavy check, that reading can be minutes old by the time `tsgo` starts, so a host that became pressured during the wait still launches the heavy check. Check pressure after acquiring the lock and before spawning `tsgo`.
