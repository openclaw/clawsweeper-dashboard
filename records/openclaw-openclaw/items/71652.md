## Summary

Keep this PR open, but do not merge it as-is. The affected `tsgo` wrapper is a live local-check surface and current `main` does not already have the requested pressure guard, but the PR's new temp-space preflight checks `/tmp` after redirecting the actual heavy-check temp directory elsewhere, which leaves a real split-filesystem correctness gap.

## What This Changes

The PR adds a local `tsgo` heavy-check pressure preflight, redirects heavy-check temp variables into a worktree artifact directory, adds an `OPENCLAW_HEAVY_CHECK_FORCE` bypass, and extends focused run-tsgo tests.

## Best Possible Solution

Land the guard in the existing script wrapper/runtime seam after the temp-space preflight checks the effective heavy-check temp directory. If `/tmp` remains useful as a separate host-health signal, keep it explicit and do not let it replace validation of the actual write target.

## Reproduction Assessment

Yes. On a host where `/tmp` and `.artifacts/tmp/local-heavy-checks` or `OPENCLAW_LOCAL_HEAVY_CHECK_TMPDIR` are different filesystems, make one low on space and the other healthy; the PR's hard-coded `df -Pk /tmp` can then disagree with the filesystem `tsgo` actually writes to.

## Solution Assessment

No. The wrapper/runtime seam is the right place for this behavior, but the proposed patch is not the narrowest correct fix until the temp-pressure reader resolves the effective temp directory instead of always probing `/tmp`.
