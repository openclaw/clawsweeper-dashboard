## Summary

Keep open. The PR targets a real current-main Bedrock/stale-catalog image-capability bug, but the proposed global GPT-5 ID inference still has a concrete false-positive path: `openai-gpt-52` is marked text-only in the bundled Venice catalog/docs, while the PR head's `/gpt-5/` branch matches it and would enable image forwarding for stale explicit-only catalogs.

## What This Changes

The PR adds merge-time `input: ["text", "image"]` defaults for explicit-only model entries matching selected vision-capable model ID families, plus regression tests covering Claude, GPT, Gemini, Llama, and brand-new provider merge paths.

## Best Possible Solution

Keep the PR open and narrow the capability defaulting before merge: preserve explicit `input` arrays as authoritative, keep the Bedrock Claude stale-catalog repair, anchor or exception the GPT-5 family so `openai-gpt-52` stays text-only, and add regression coverage plus a credited changelog entry.

## Reproduction Assessment

Yes. On current main, `mergeProviderModels({ models: [] }, { models: [{ id: "us.anthropic.claude-opus-4-7" }] })` leaves `input` unset and `modelSupportsImages` returns false; on the PR head, the added regex matches `openai-gpt-52` even though current Venice metadata marks it text-only.

## Solution Assessment

No, not as currently written. The merge-time repair is a plausible narrow fix for stale Bedrock Claude catalogs, but the unqualified GPT-5 pattern is not conservative enough for provider-specific IDs; a tighter allowlist or provider-aware exception is safer.
