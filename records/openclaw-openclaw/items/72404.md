## Summary

Keep open. The PR addresses a real stale explicit-catalog image-capability bug, but the current head is not merge-ready because it regresses explicit input override preservation and still overmatches GPT-5 text-only Venice metadata.

## What This Changes

The PR adds merge-time `input: ["text", "image"]` defaults for explicit-only vision-looking model IDs, plus tests for Claude, GPT/OpenAI, Gemini, Llama, and brand-new provider merge paths.

## Best Possible Solution

Keep the merge-time stale-catalog repair, but preserve explicit `input` whenever it is configured and make vision-family inference conservative enough to match current provider metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main leaves an explicit-only Bedrock Claude row without `input`, and the image gate treats that as text-only; the PR head blockers are reproducible from the changed merge line and regex.

## Solution Assessment

No, not as currently written. The merge helper is the right narrow place for stale-catalog repair, but the patch must not undo explicit user/catalog input overrides and should not infer vision from broad GPT-5 substrings.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve explicit input overrides:** `src/agents/models-config.merge.ts:158`
  - body: This branch now always copies `implicitModel.input` when an explicit model ID matches discovery. Current main intentionally preserves configured `input` overrides here, with regression coverage for custom vision models; this PR would downgrade an explicit `['text','image']` row to a text-only implicit row, or override an explicit text-only opt-out when discovery says image. Keep the existing `"input" in explicitModel ? explicitModel.input : implicitModel.input` behavior for matched models.
  - confidence: 0.94
- **[P2] Narrow the GPT-5 capability match:** `src/agents/models-config.merge.ts:59`
  - body: The `/gpt-5/` alternative matches `openai-gpt-52`, but current Venice metadata marks that model `input: ['text']` while only adjacent Codex/newer GPT rows include image input. A stale explicit-only Venice row would get image forwarding enabled against current provider metadata, so anchor this to known image-capable IDs or add provider-aware matching plus negative coverage for `openai-gpt-52`.
  - confidence: 0.9
