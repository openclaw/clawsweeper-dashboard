## Summary

Keep open. The reported Bedrock/static-catalog vision regression is real on current main, and the PR is pointed at the right merge surface, but the current patch still over-infers image support: its `gpt-5` pattern also matches the bundled Venice `openai-gpt-52` model, which current metadata and docs mark text-only.

## What This Changes

The PR adds merge-time `input: ["text", "image"]` defaults for explicit-only model entries matching selected vision-capable ID patterns, plus regression tests for Claude, GPT, Gemini, Llama, and brand-new provider merge paths.

## Best Possible Solution

Keep the PR open and narrow the defaulting strategy before merge: preserve the Bedrock Claude explicit-only repair and explicit `input` overrides, but use exact/provider-aware vision inference or a tighter allowlist with negative coverage for `venice/openai-gpt-52`.

## Reproduction Assessment

Yes. On current main, an explicit-only Bedrock Claude model with no `input` remains without `input`, and the image gate treats that as text-only; on the PR head, the added GPT regex matches `openai-gpt-52`.

## Solution Assessment

No, not as currently written. Merge-time repair is a plausible narrow surface, but the PR should not infer image support from an unqualified `gpt-5` substring when existing provider metadata marks a matching model text-only.
