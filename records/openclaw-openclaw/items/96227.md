## Summary

Keep open: this patch is a focused attempt at the right bug, but it only instruments the non-stream `/v1/responses` branch and casts endpoint config as full runtime config, so the central HTTP-ingress diagnostics gap is not safely fixed.

## What This Changes

The PR adds a `model.usage` diagnostic emission block to `src/gateway/openresponses-http.ts` after non-stream `/v1/responses` agent runs.

## Best Possible Solution

Emit `model.usage` once from `agentCommandFromIngress` or a shared helper with the full runtime config and sibling metadata, then cover both HTTP endpoints, streaming modes, diagnostics-disabled behavior, and real subscriber proof.

## Reproduction Assessment

Yes at source level: current main routes both OpenAI-compatible HTTP handlers through `agentCommandFromIngress`, while only channel and cron sibling paths emit `model.usage`. I did not run a live HTTP subscriber repro in this read-only review.

## Solution Assessment

No. The endpoint-local patch is not the best fix because it misses chat completions and streaming Responses and cannot reliably honor top-level diagnostics config; the maintainable fix belongs at shared ingress finalization or a shared helper.
