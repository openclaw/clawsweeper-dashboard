## Summary

## What This Changes

The PR adds direct `model.usage` diagnostic emission to the non-streaming `/v1/responses` handler in `src/gateway/openresponses-http.ts`.

## Best Possible Solution

Centralize the emission at `agentCommandFromIngress` or a shared helper, preserve the canonical channel/cron event shape, and prove both HTTP endpoints in streaming and non-streaming modes.

## Reproduction Assessment

Yes. Source inspection shows this PR emits only in the non-streaming Responses handler, while current Chat Completions and Responses streaming paths still call ingress without a `model.usage` diagnostic; I did not run a live HTTP subscriber in this read-only review.

## Solution Assessment

No. A handler-local emitter is too narrow; the maintainable fix is a shared ingress emission path that uses `agentMeta` session fields and has endpoint/mode coverage.
