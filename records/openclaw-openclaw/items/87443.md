## Summary

This should stay open: current main and latest release still resolve sqlite-vec through generic Linux native packages, while live npm metadata has no musl-specific sqlite-vec artifact to select. The remaining work is a maintainer support/artifact decision before a safe code fix.

## What This Changes

The issue reports that OpenClaw’s sqlite-vec vector search fails on musl-based Linux systems because the selected native sqlite-vec binary is built for glibc.

## Best Possible Solution

Choose the musl sqlite-vec support policy, then either add libc-aware artifact resolution with focused tests and musl smoke proof, or document musl as extensionPath-only while preserving degraded fallback search.

## Reproduction Assessment

Yes from source inspection: current main resolves only generic linux-x64/linux-arm64 sqlite-vec packages, and the loader marks vector search unavailable when that native extension fails to load. I did not run the exact OpenWrt/iStoreOS command live.

## Solution Assessment

Unclear as an implementation path: a simple resolver map entry is not enough because the expected musl sqlite-vec npm package does not exist. The maintainable fix starts with a maintainer artifact/support decision.
