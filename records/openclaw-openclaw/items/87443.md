## Summary

This should stay open: current `main` and the latest release still resolve sqlite-vec through generic Linux artifacts with no musl-specific package path, while OpenClaw otherwise supports Alpine/musl as a runtime target. The remaining decision is how maintainers want to source or support a musl-compatible sqlite-vec artifact.

## What This Changes

The issue reports that sqlite-vec vector-store acceleration is unavailable on musl-based Linux because OpenClaw loads a glibc-built `vec0.so`.

## Best Possible Solution

Maintainers should decide the sqlite-vec artifact policy for musl Linux, then either add a tested musl-compatible resolution path or document musl as extensionPath-only while preserving fallback search.

## Reproduction Assessment

Yes from source inspection: OpenClaw supports musl/Alpine runtime setup, but sqlite-vec resolution only covers generic Linux x64/arm64 packages. I did not run the exact iStoreOS/OpenWrt command live.

## Solution Assessment

Unclear: adding a `linux-x64-musl` map entry is only correct if a trusted artifact exists, while vendoring or building one is a maintainer dependency and supply-chain decision.
