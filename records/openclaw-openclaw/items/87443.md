## Summary

This should stay open: current main and v2026.6.6 still use sqlite-vec 0.1.9 with only generic Linux native packages, and the live npm metadata still has no musl-specific sqlite-vec package. The remaining blocker is a maintainer artifact/support decision, not missing evidence from the reporter.

## What This Changes

The issue asks OpenClaw to support sqlite-vec vector search on musl-based Linux systems where the current glibc-linked vec0.so cannot load.

## Best Possible Solution

Choose the musl sqlite-vec support policy, then either add libc-aware artifact resolution with focused tests and musl smoke proof, or document musl as extensionPath-only while preserving fallback search.

## Reproduction Assessment

Yes from source inspection: current main resolves sqlite-vec only through generic linux-x64/linux-arm64 packages, and npm metadata has no musl package. I did not run the exact OpenWrt/iStoreOS command live.
