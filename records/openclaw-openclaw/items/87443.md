## Summary

Keep open: current main and v2026.6.9 still have no musl-aware sqlite-vec artifact path, and the live npm registry has no musl sqlite-vec package to select. The source-level failure is clear, but the fix needs a maintainer support and supply-chain decision before a safe implementation path exists.

## What This Changes

The issue asks for musl-compatible sqlite-vec loading so `openclaw memory status --deep` can report the builtin vector store as ready on OpenWrt/iStoreOS-like systems.

## Best Possible Solution

Maintainers should choose the sqlite-vec musl support policy, then either add libc-aware trusted artifact resolution with musl smoke proof or document musl users toward explicit `memorySearch.store.vector.extensionPath`/alternate memory backend guidance while preserving fallback search.

## Reproduction Assessment

Yes, at source level: current main still resolves musl Linux to generic sqlite-vec Linux packages, and the load-failure path marks the builtin vector store unavailable. I did not run a live musl host repro.

## Solution Assessment

Unclear. Adding a musl package branch is the best fix only if maintainers choose and can trust a musl sqlite-vec artifact; otherwise the maintainable path is documented explicit extensionPath support or an alternate memory backend recommendation.
