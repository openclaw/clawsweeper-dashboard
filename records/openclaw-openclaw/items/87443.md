## Summary

Keep this open: current main and the latest release still resolve sqlite-vec through generic glibc-oriented Linux packages, while live npm registry metadata still has no musl platform package to select. The remaining decision is whether OpenClaw should ship/support a trusted musl artifact or document musl as an explicit extensionPath-only setup.

## What This Changes

The issue asks OpenClaw to support a musl-compatible sqlite-vec vector-search binary for musl-based systems such as OpenWrt/iStoreOS.

## Best Possible Solution

Decide the musl support policy, then either add libc-aware resolution for a trusted musl sqlite-vec artifact with focused tests and musl smoke proof, or document musl as extensionPath-only while preserving fallback search.

## Reproduction Assessment

Yes, at source level: current main resolves generic linux-x64/linux-arm64 sqlite-vec packages and reports native load failures as unavailable vector-store status. I did not establish a live current-main musl repro in this read-only review.

## Solution Assessment

Unclear: simply adding a musl package name is only maintainable once a trusted musl artifact exists. The better first step is a maintainer support decision, followed by libc-aware resolution or explicit documentation of extensionPath-only support.
