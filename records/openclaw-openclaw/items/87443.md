## Summary

Keep open: current main and the latest release still select generic sqlite-vec Linux artifacts with no musl-aware package path, and the live registry has no musl sqlite-vec package to select. The bug is well diagnosed, but fixing it needs a maintainer support/artifact decision before an automated code change is safe.

## What This Changes

The issue reports sqlite-vec native extension loading failure on musl-based Linux systems and asks for a musl-compatible sqlite-vec artifact path.

## Best Possible Solution

Choose the sqlite-vec musl support policy, then either add libc-aware artifact resolution with focused tests and musl smoke proof, or document musl as requiring an explicit `memorySearch.store.vector.extensionPath` while preserving fallback search.

## Reproduction Assessment

Yes, at source level: current main resolves musl Linux to generic sqlite-vec Linux packages, and the reported `__memcpy_chk` relocation failure would make the vector-store probe return unavailable. I did not run a live musl host repro.

## Solution Assessment

Unclear. Adding a musl package branch is only the best fix if a trusted musl sqlite-vec artifact exists and maintainers choose to support it; otherwise the maintainable path is documented `extensionPath` support or a different memory backend recommendation.
