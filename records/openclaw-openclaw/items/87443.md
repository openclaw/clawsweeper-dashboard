## Label Justifications

- `impact:other`: The concrete impact is memory vector-store platform readiness, which is maintainer-visible but outside the narrower owned impact labels.

## Summary

Keep open: current main and the latest release still route musl Linux to the generic sqlite-vec Linux native packages, and the registry does not provide a musl sqlite-vec package for OpenClaw to select safely. The report remains valid, but the fix needs a maintainer artifact/support decision before implementation.

## What This Changes

The issue requests musl-compatible sqlite-vec vector-store support so memory status and vector search can load sqlite-vec on OpenWrt/iStoreOS-style systems.

## Best Possible Solution

Choose the sqlite-vec musl support policy, then either add libc-aware trusted artifact resolution with focused tests and musl smoke proof, or document musl as requiring an explicit store.vector.extensionPath while preserving fallback search.

## Reproduction Assessment

Yes, at source level: current main resolves musl Linux to generic sqlite-vec Linux packages, and the reported __memcpy_chk relocation failure would make the vector-store probe return unavailable. I did not run a live musl host repro.

## Solution Assessment

Unclear. Adding a musl package branch is only the best fix if a trusted musl sqlite-vec artifact exists and maintainers choose to support it; otherwise the maintainable path is documented extensionPath support or a different memory backend recommendation.
