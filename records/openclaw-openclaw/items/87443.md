## Summary

Keep this open: current main and v2026.6.6 still resolve sqlite-vec through generic Linux x64/arm64 artifacts with no musl package, while OpenClaw documents Alpine/musl as a supported install path. The existing fallback and extensionPath hint mitigate degraded recall, but they do not provide the requested musl-compatible sqlite-vec acceleration.

## What This Changes

The issue asks OpenClaw to make sqlite-vec vector acceleration work on musl Linux systems such as OpenWrt/iStoreOS instead of loading the glibc Linux binary.

## Best Possible Solution

Maintainers should choose the sqlite-vec musl artifact policy, then either add a tested libc-aware resolution path or document musl as extensionPath-only while preserving degraded fallback search.

## Reproduction Assessment

Yes from source inspection: current main maps Linux sqlite-vec to generic x64/arm64 packages and the locked/upstream package set has no musl variant. I did not run the exact iStoreOS/OpenWrt command live.

## Solution Assessment

Unclear: the requested musl binary is plausible, but adding a map entry is not sufficient without a trusted artifact. The safer path is a maintainer dependency/supply-chain decision first, then a tested libc-aware loader or explicit extensionPath-only documentation.
