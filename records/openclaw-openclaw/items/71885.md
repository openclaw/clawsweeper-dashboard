## Summary

Keep this PR open for review rather than close it: the SQLite binding failure is plausible if an object reaches the store, but the proposed fix normalizes too late and the contributor has not shown a real OpenClaw ingress path or after-fix behavior proof.

## What This Changes

The PR adds JSON serialization for non-string `TaskRecord.task` values at SQLite bind time and adds a direct SQLite store regression test for object task payloads.

## Best Possible Solution

Normalize task input before `TaskRecord` creation/update or deliberately widen the task contract across all consumers, then rebase and add redacted after-fix terminal/log proof for the real SQLite/OpenClaw path.

## Reproduction Assessment

No high-confidence end-to-end OpenClaw reproduction path is shown. Source inspection plus a direct `node:sqlite` bind probe show that an object reaching the current SQLite store would fail, but the real structured-task ingress path is still missing.

## Solution Assessment

No. SQLite-only serialization is too late for the current `TaskRecord.task: string` contract; the safer fix is boundary normalization before records enter memory and persistence, or an explicit contract widening with all consumers updated.
