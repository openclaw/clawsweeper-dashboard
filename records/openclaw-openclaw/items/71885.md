## Summary

Keep open. The PR targets a real SQLite bind failure for structured task values, but the current patch serializes only at the SQLite bind layer after task records have already been created, indexed, mirrored to flows, and emitted to downstream consumers that still require `TaskRecord.task` to be a string.

## What This Changes

The PR adds a SQLite bind helper that JSON-stringifies non-string `TaskRecord.task` values and a SQLite-store regression test for object-valued task payloads.

## Best Possible Solution

Keep `TaskRecord.task` as a string-only durable contract, normalize unknown runtime task inputs before lookup, merge, record construction, flow mirroring, and observer emission, and keep the SQLite store binding already-normalized strings. The useful SQLite regression should be retained after rebasing it onto the current test-state helper pattern, with a changelog entry for the persistence fix.

## Reproduction Assessment

Yes. A focused in-memory `node:sqlite` bind check reproduces object and array bind failures, and current-main source inspection shows the PR's normalization occurs after in-memory task record creation and downstream string consumers.

## Solution Assessment

No. Binding-time serialization is not the narrowest maintainable fix because it leaves non-string `TaskRecord.task` values alive in the registry and flow/status paths; normalizing at task-registry or runtime ingress preserves the existing contract and fixes persistence.
