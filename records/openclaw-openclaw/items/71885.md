## Summary

Keep open. The reported SQLite bind failure is real, but this PR serializes too late in the persistence layer while current main still treats `TaskRecord.task` as a string-only in-memory contract used by status and flow consumers.

## What This Changes

The PR adds a SQLite helper that JSON-stringifies non-string `TaskRecord.task` values and a regression test for storing object task payloads.

## Best Possible Solution

Keep `TaskRecord.task` string-only, normalize untrusted task inputs before record construction or at the runtime ingress, and cover SQLite plus status/flow consumers with a changelog entry.

## Reproduction Assessment

Yes. The SQLite object-bind failure reproduces with an in-memory `node:sqlite` check, and current main still binds `record.task` directly; the remaining gap is proving the runtime ingress that can pass a structured value.

## Solution Assessment

No. A SQLite-only serializer is not the best fix because structured values can enter the in-memory registry before persistence; the safer solution is normalization before `TaskRecord` creation or a deliberate full contract widening.
