## Summary

Keep open: the SQLite failure mode is plausible, but this PR is not mergeable as-is because it normalizes `TaskRecord.task` too late in the data path, the added regression test is stale against current `main`'s store-test harness, and contributor real behavior proof is still missing.

## What This Changes

The PR adds a SQLite bind-time serializer for non-string `TaskRecord.task` values and a regression test that stores an object task payload as JSON.

## Best Possible Solution

Keep `TaskRecord.task` string-only by normalizing untrusted task input before `createTaskRecord`, then add current-main regression coverage through registry persistence plus status or flow consumers that read task text.

## Reproduction Assessment

Yes for the narrow SQLite bind failure: an in-memory `node:sqlite` probe rejects an object parameter, and current source binds `record.task` directly on `main`. No high-confidence end-to-end OpenClaw ingress for structured task payloads is shown yet.

## Solution Assessment

No. SQLite-only serialization is too late because the registry and status/flow consumers still observe `TaskRecord.task` before or outside persistence; the safer fix is to normalize before record creation or deliberately widen the whole contract.
