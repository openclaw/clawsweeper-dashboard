## Summary

Keep open. The PR targets a real SQLite bind failure class, but the proposed fix normalizes only at persistence time, leaving `TaskRecord.task` able to remain non-string in the in-memory registry, task-flow sync, status rendering, and delivery paths if a structured runtime value reaches ingress.

## What This Changes

The PR adds a SQLite bind helper that JSON-stringifies non-string `TaskRecord.task` values and adds a direct SQLite-store regression test for object-valued task payloads.

## Best Possible Solution

Keep `TaskRecord.task` as the durable string contract, normalize unknown runtime task inputs before lookup/merge/record construction, and cover the registry ingress path plus affected status/flow consumers. The SQLite store should then bind an already-normalized string, with a changelog entry for the user-visible persistence fix.

## Reproduction Assessment

Yes. A focused in-memory SQLite bind check reproduces object and array bind failures, and source inspection shows the PR's normalization happens after the in-memory task record has already been created.

## Solution Assessment

No. Binding-time serialization is too late for the current contract; the safer fix is to coerce task values at task-registry or detached-runtime ingress and keep downstream `TaskRecord.task` consumers string-only.
