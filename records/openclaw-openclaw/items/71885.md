## Summary

Keep open. The reported SQLite bind failure is real, but the PR normalizes only the SQLite parameter value, leaving any structured task payload in the active registry record, observer event, and string-only task consumers; the prior ClawSweeper finding still applies after checking current main.

## What This Changes

The PR stringifies non-string `TaskRecord.task` values at SQLite bind time and adds a regression test that a structured task object persists as a JSON string.

## Best Possible Solution

Keep `TaskRecord.task` as a string contract and normalize unknown runtime task inputs at registry ingress, with regression coverage that exercises `createTaskRecord` and task status/list behavior so active records and persisted rows agree.

## Reproduction Assessment

Yes. A focused `node:sqlite` in-memory bind check reproduces object and array bind failures, and code inspection shows the PR would still leave a non-string `TaskRecord.task` in memory if such a payload reaches the registry.

## Solution Assessment

No. Binding-time serialization is too late; the narrow maintainable fix is to coerce before constructing or merging a `TaskRecord`, unless maintainers intentionally widen the type and update all consumers.
