## Summary

Keep open: the PR targets a real SQLite object-bind failure, but the patch normalizes non-string task values only at SQLite persistence while current main still treats `TaskRecord.task` as a string across registry, status, and flow consumers. The regression test also needs rebasing, and the external PR still has only test/lint proof rather than after-fix real behavior proof.

## What This Changes

The PR adds a SQLite bind helper that JSON-stringifies non-string `TaskRecord.task` values and a regression test for object task payloads.

## Best Possible Solution

Keep `TaskRecord.task` string-only by normalizing untrusted task input before record creation, then add focused coverage that exercises registry persistence and the status/flow consumers that read task text.

## Reproduction Assessment

Yes for the SQLite failure mode: current main binds `record.task` directly and a read-only in-memory `node:sqlite` probe rejects object parameters. The real runtime ingress for structured task payloads is not identified, so the end-to-end OpenClaw path still needs proof.

## Solution Assessment

No. SQLite-only serialization is too late in the data path; the maintainable fix is to normalize before `TaskRecord` creation, or intentionally widen the contract and update every consumer that assumes a string.
