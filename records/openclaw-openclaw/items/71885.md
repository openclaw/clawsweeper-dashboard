## Summary

Keep open. The PR addresses the reported SQLite bind failure class, but it serializes only at the persistence boundary, leaving a non-string `TaskRecord.task` in the in-memory registry and observer path even though current consumers still assume a string.

## What This Changes

The PR adds a SQLite bind helper that JSON-stringifies non-string `TaskRecord.task` values and a regression test that writes an object-valued task payload through the SQLite store.

## Best Possible Solution

Keep `TaskRecord.task` as the durable string contract and normalize unknown runtime task inputs at task-registry ingress before finding, merging, storing, emitting, or persisting records; only widen the type if all status, flow, command, and delivery consumers are updated together.

## Reproduction Assessment

Yes. A focused in-memory `node:sqlite` bind check reproduces object and array bind failures, and code inspection shows that a structured task value reaching `createTaskRecord` would still stay non-string in memory under this PR.

## Solution Assessment

No. Binding-time serialization is too late for the current contract; the narrower maintainable fix is to coerce before constructing or merging a `TaskRecord`, with persistence then receiving the already-normalized value.

## Review Findings

Overall correctness: patch is incorrect
