## Summary

Keep open: this is a plausible focused bug fix, but the PR should not merge as written because it normalizes only at SQLite binding while current task records and downstream status/flow consumers still assume `task` is a string. The regression test also needs rebasing onto current main's task-store test helper pattern, and the PR body only provides test/lint proof rather than real behavior proof.

## What This Changes

The PR adds a SQLite bind helper that JSON-stringifies non-string task values and a regression test for object task payloads.

## Best Possible Solution

Keep `TaskRecord.task` string-only, normalize untrusted task inputs before `TaskRecord` construction, and cover registry persistence plus status/flow consumers with focused regression tests.

## Reproduction Assessment

Yes. A read-only in-memory SQLite bind check reproduces object bind failure, and current main source shows `record.task` is bound directly while task records are expected to be string-valued.

## Solution Assessment

No. The PR's SQLite-only serialization is too late in the data path; the safer fix is to normalize at registry or detached-runtime ingress, or deliberately widen the full task contract and update all consumers.
