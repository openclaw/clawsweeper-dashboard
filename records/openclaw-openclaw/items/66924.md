## Summary

Keep open: current main still has the generic Gemma forward-compat template-inheritance path, and this PR is a focused fix attempt, but the proposed blanket 128k fallback is wrong for Gemma 4 and the external real-behavior proof gate is still missing.

## What This Changes

The PR adds a fixed 131,072-token Gemma context-window patch in the Google forward-compat resolver and a unit test expecting a Gemma 4 clone not to inherit the Gemini 1M template.

## Best Possible Solution

Keep the fix in the Google provider boundary, but derive Gemma fallback context from exact/catalog/family metadata when available, preserve configured metadata precedence, and add focused regression coverage plus redacted CLI proof.

## Reproduction Assessment

Yes, source-reproducible: with an exact Gemma registry miss and a Gemini flash template, current main's Gemma branch only patches reasoning while `cloneFirstTemplateModel` keeps the template `contextWindow`. The linked configured-list issue is separately closed because configured metadata now overrides that fallback in the list/runtime path.

## Solution Assessment

No. The branch changes the right provider-owned resolver, but hardcoding `131_072` for every Gemma clone is wrong for current Gemma 4 metadata; a metadata-backed fallback is narrower and safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Use exact/catalog/family metadata instead of locking a blanket 128k value.
