## Summary

Keep open for contributor and maintainer follow-up: current main does not include the PR's Gemma fallback-window patch, but the proposed 128k default conflicts with current Gemma 4 metadata and the PR still lacks real `models list` proof.

## What This Changes

The PR adds a 131,072-token context-window patch for Gemma forward-compat clones in the Google provider and a regression test for `gemma-4-26b-a4b-it`.

## Best Possible Solution

Keep the fix in the Google provider boundary, but derive fallback context from exact or Gemma-family metadata when available, preserve configured/discovered metadata precedence, and prove the visible model-list row after the change.

## Reproduction Assessment

Yes for the resolver-level fallback path by source inspection: current main maps missing `gemma-*` ids through Gemini flash templates, and the clone helper preserves template context unless patched or overridden. The exact configured `models list` case from the linked issue is already handled by configured metadata precedence and the current pi-ai catalog contains the renamed model id.

## Solution Assessment

No. The PR touches the right provider-owned path, but hardcoding 131,072 for every Gemma fallback conflicts with current Gemma 4 metadata; the safer solution should use exact or family metadata and keep configured overrides authoritative.
