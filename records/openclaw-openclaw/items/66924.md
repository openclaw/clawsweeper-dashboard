## Summary

Keep this PR open, but the current patch should not merge as-is. Current main still has the Gemma template-inheritance gap, and the PR targets the right Google provider boundary, but hardcoding 131,072 for every Gemma fallback under-reports the linked Gemma 4 26B A4B case, which is listed at a 256k-class context window.

## What This Changes

The PR adds a 131,072-token Gemma context-window patch to the Google forward-compat resolver and a regression test for a Gemma clone sourced from the Gemini flash template.

## Best Possible Solution

Keep the fix provider-owned in the Google plugin, but make the fallback family-aware: avoid inheriting Gemini flash's 1M context, preserve configured or discovered metadata whenever present, and use known Gemma catalog values for missed exact ids.

## Reproduction Assessment

Yes. A focused resolver check with modelId gemma-4-26b-a4b-it and a gemini-3-flash-preview template at 1,048,576 tokens reproduces the current-main inheritance path by inspection; the PR adds that shape as a unit test but asserts a too-small value.

## Solution Assessment

No. The PR chooses the right boundary and preserves the Gemma 4 reasoning patch, but one 131,072-token fallback for every Gemma model is not the narrowest correct solution for the linked Gemma 4 26B A4B case.
