## Summary

Keep this PR open, but do not merge it as-is. Current main still has the Gemma template-inheritance path, so the work is not obsolete; however, the PR hardcodes a 131,072-token fallback for every Gemma model while the linked Gemma 4 26B A4B report expects a 256k-class context window when exact metadata is unavailable or configured.

## What This Changes

The PR adds a 131,072-token Gemma context-window fallback in the Google forward-compat resolver and a regression test for a Gemma 4 model cloned from the Gemini flash template.

## Best Possible Solution

Keep the fix provider-owned in the Google plugin, but replace the one-size 128k fallback with Gemma-family or exact-id fallback metadata, preserve configured/discovered contextWindow precedence, add regression coverage for the Gemma 4 26B A4B case, and include the required changelog entry.

## Reproduction Assessment

Yes. By inspection, current main reproduces the standalone resolver failure when gemma-4-26b-a4b-it is cloned from a gemini-3-flash-preview template with contextWindow 1,048,576: the Gemma branch only patches reasoning, and the clone helper otherwise preserves the template contextWindow.

## Solution Assessment

No. The PR chooses the right provider boundary and test shape, but hardcoding 131,072 for every Gemma model is not the narrowest correct fix for the linked 256k-class Gemma 4 case; a family-aware fallback that still respects configured metadata is safer.
