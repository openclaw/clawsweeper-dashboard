## Summary

## What This Changes

The PR adds a 131,072-token context-window patch for all Google Gemma forward-compat clones and a regression test expecting Gemma not to inherit the Gemini 1M template context.

## Best Possible Solution

Keep the fix in the Google provider boundary, but derive Gemma fallback token metadata from authoritative exact or family metadata when available, preserve configured/catalog metadata precedence, and add focused regression coverage plus redacted CLI proof.

## Reproduction Assessment

Yes for the resolver-level failure by source inspection: with a missing `gemma-*` catalog row and a `gemini-3-flash-preview` template, current main clones the template and only patches Gemma 4 reasoning. I did not run a local CLI repro because this review is read-only.

## Solution Assessment

No. The PR touches the right provider-owned path, but a fixed 131,072 value for every Gemma clone underreports Gemma 4; the safer solution is metadata-backed fallback that preserves configured and discovered metadata.
