## Summary

Keep open: current main and the latest release still emit `memory.dream.completed` without a structured outcome, while deep dreaming failure paths only log errors outside the event journal. The gap is real, but the permanent outcome taxonomy and failure/interruption semantics need maintainer agreement before implementation.

## What This Changes

The issue requests structured outcome normalization for memory dreaming completion events so consumers can distinguish successful, partial, failed, and timed-out phases.

## Best Possible Solution

Define an additive memory-host dreaming outcome contract, update memory-core emissions and event tests for successful and failed phases, and keep broader started/interrupted crash recovery in https://github.com/openclaw/openclaw/issues/97692.

## Reproduction Assessment

Yes for the central schema gap: current source and tests show `memory.dream.completed` has no outcome field and failure paths do not write a structured failure event. I did not establish a live timeout reproduction, and the exact timeout configuration path needs separate verification.

## Solution Assessment

Unclear as written: adding an additive outcome field is the right likely seam, but maintainers should first choose the durable outcome taxonomy and keep crash-start/interruption recovery separate from this narrower event normalization.
