## Summary

Keep open: current main and v2026.6.10 still emit success-shaped `memory.dream.completed` events without outcome, status, or reason fields, and no related open PR fully owns this event contract. The remaining work is useful, but it needs maintainer direction because it extends the public memory-host event schema and overlaps crash-recovery and no-text fallback work.

## What This Changes

Request adds normalized outcome information to memory-core dreaming completion/failure events so dashboards, doctor checks, cron, and recovery code can distinguish completed, failed, timed-out, partial, or interrupted phases.

## Best Possible Solution

Define an additive memory-host event contract for dreaming outcomes, keep existing success-event consumers compatible, and coordinate failed/partial/interrupted semantics with https://github.com/openclaw/openclaw/issues/97692 plus the no-text recovery and diagnostic work.

## Reproduction Assessment

Yes, at source level: current main's event type and emitters lack outcome/status/reason fields, and failure/fallback paths log or write fallback text without appending failed outcome events. I did not run a live timeout or crash repro.

## Solution Assessment

Unclear as a direct bug-fix patch: an additive outcome field is likely the right direction, but the exact taxonomy and whether to add started/failed/interrupted records needs a maintainer-approved memory event contract.
