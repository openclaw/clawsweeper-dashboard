## Summary

Keep open. The PR targets a real MiniMax setup gap, but the current diff still makes a generic MiniMax model key an implicit MiniMax Search credential and removes some Coding/Token Plan clarity, so it needs revision before merge.

## What This Changes

The PR adds `MINIMAX_API_KEY` and onboarding metadata to MiniMax web-search provider metadata, updates MiniMax web-search docs/UI copy, and adjusts the provider registration test.

## Best Possible Solution

Land a revised MiniMax setup fix that exposes MiniMax Search in onboarding, keeps runtime/contract/docs metadata aligned, preserves clear Coding/Token Plan requirements and all accepted env vars, and only auto-detects generic `MINIMAX_API_KEY` after an entitlement gate or explicit maintainer-approved routing rule.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no explicit web-search provider, adding `MINIMAX_API_KEY` to the provider `envVars` makes `hasWebProviderEntryCredential()` return true and `resolveWebSearchProviderId()` select MiniMax by auto-detect order.

## Solution Assessment

No. The onboarding visibility part is useful, but treating every generic MiniMax model key as a search-capable credential is not the narrowest maintainable fix without a capability gate, lower priority, or explicit maintainer policy approval.
