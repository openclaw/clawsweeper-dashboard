## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested setup visibility and generic-key metadata, but the PR has unresolved maintainer-requested changes around MiniMax Token/Coding Plan wording, a false-positive auto-detection/data-routing concern, and no required changelog entry.

## What This Changes

The PR updates MiniMax web-search metadata, docs, manifest UI hints, and tests so `MINIMAX_API_KEY` is treated as a MiniMax Search credential and MiniMax Search appears during web-search setup.

## Best Possible Solution

Land a revised MiniMax web-search setup fix that exposes MiniMax Search in onboarding, keeps contract/runtime/manifest/docs metadata aligned, preserves clear Token/Coding Plan requirements, lists all accepted env vars in user-facing guidance, and either gates or carefully prioritizes generic `MINIMAX_API_KEY` auto-detection with focused provider-selection coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with no explicit web-search provider, the PR would make a non-empty `MINIMAX_API_KEY` satisfy `hasWebProviderEntryCredential()` and select MiniMax at `autoDetectOrder: 15`, ahead of later providers such as Gemini/Grok/Kimi/Perplexity.

## Solution Assessment

No. The onboarding visibility part is useful, but adding a generic MiniMax model key directly to auto-detection and softening Coding Plan wording is not the narrowest maintainable fix without a search-entitlement gate or an explicit maintainer decision on fallback behavior.
