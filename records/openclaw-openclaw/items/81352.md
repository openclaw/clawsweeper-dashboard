## Summary

Keep open: current main and the latest release still allow blank task owner keys and blank caller keys to compare equal, so the central hardening is still useful, but this external PR is not merge-ready because it lacks real behavior proof, is currently conflicting with main, and needs maintainer acceptance of the fail-closed behavior for restored blank-owner task rows.

## What This Changes

The PR changes task owner-scoped access to require both task and caller owner keys to normalize to non-empty matching strings, with regression coverage for blank caller keys and restored blank-owner task records.

## Best Possible Solution

Refresh the branch and land the non-empty owner-key guard only after redacted runtime proof is added and maintainers accept fail-closed restored blank-owner rows or choose a repair path.

## Reproduction Assessment

Yes, this is source-reproducible: current main compares two normalized optional owner keys, and the normalization helper returns undefined for blank strings. I did not run a live setup because this review is read-only.

## Solution Assessment

Yes, the central predicate is the narrowest maintainable fix; duplicating blank-key checks in callers would split the invariant. The unresolved pieces are contributor proof, conflict refresh, and the compatibility decision for restored blank-owner rows.
