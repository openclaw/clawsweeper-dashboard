## Summary

Keep open: the update addressed the earlier invalid-active-config and `.rejected.*` gaps and now has real CLI proof, but the new backup scanner still parses copied config artifacts as strict JSON even though OpenClaw config backups can be valid JSON5, leaving a concrete false-negative path for secret residue audit.

## What This Changes

The PR extends `openclaw secrets audit` to enumerate adjacent config backup/rejected files, scan them for plaintext secret fields without resolving backup-only refs, add focused tests, and document the new audit scope.

## Best Possible Solution

Keep the audit expansion, but parse backup/rejected config artifacts with the same JSON5-aware config parser as active config while still skipping ref resolution for backup-only refs.

## Reproduction Assessment

Yes, source-reproducible. A valid JSON5 `openclaw.json.bak` with a comment or trailing comma and a plaintext `models.providers.*.apiKey` is enumerated by the PR but rejected by `JSON.parse` before `collectConfigSecrets` can report it.

## Solution Assessment

No. The `secrets audit` surface and invalid/rejected artifact coverage are right, but backup artifacts should reuse OpenClaw's JSON5 config parser instead of the strict JSON helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Parse config backup artifacts as JSON5:** `src/secrets/audit.ts:276-280`
  - body: The new backup scan reads `.bak`/`.pre-update`/`.clobbered.*`/`.rejected.*` files through `readJsonObjectIfExists`, which uses `JSON.parse`. OpenClaw's active config is documented as JSON5, and `.bak`/`.pre-update`/`.clobbered.*` artifacts are copies of that authored config, so a valid backup with a comment or trailing comma is marked scanned and then ignored before plaintext findings are collected. Parse these artifacts with the same JSON5 config parser while still skipping backup-only ref resolution, and add a JSON5 backup fixture.
  - confidence: 0.9
