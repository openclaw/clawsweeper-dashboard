## Summary

Keep open. The direction is useful, but the patch is not merge-ready because backup scanning is skipped when the active config is invalid, rejected config payload artifacts are omitted, and the external PR only provides fixture proof rather than after-fix real behavior proof.

## What This Changes

The PR extends `openclaw secrets audit` to scan adjacent `openclaw.json` backup artifacts for plaintext secret fields, adds focused audit fixtures, and updates the secrets CLI docs.

## Best Possible Solution

Keep the audit expansion, but make backup-artifact scanning independent of active-config validity and cover all adjacent OpenClaw-owned config artifacts that can preserve secret-shaped payloads.

## Reproduction Assessment

Yes. Source inspection shows current main has no config-backup enumeration in `runSecretsAudit`, and the PR fixtures describe a clear `.bak` plaintext case, but this read-only review did not execute it.

## Solution Assessment

No. Adding the scan to `secrets audit` is the right surface, but the implementation should also run when the active config is invalid and include `.rejected.*` payload artifacts before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Scan backups when the active config is invalid:** `src/secrets/audit.ts:705-708`
  - body: Because this loop is inside `if (snapshot.valid)`, a malformed or clobbered active `openclaw.json` stops audit from checking adjacent backup files entirely. That misses plaintext residues in the recovery case this new scan is meant to cover, so move backup-artifact scanning out of the active-config-valid branch while still avoiding backup-only ref resolution.
  - confidence: 0.88
- **[P2] Include rejected config payload artifacts:** `src/secrets/storage-scan.ts:59-64`
  - body: OpenClaw saves failed config writes as `openclaw.json.rejected.*` beside the active config, and the troubleshooting docs pair those files with `.clobbered.*` for operator inspection. Since rejected payloads can contain the same secret-shaped fields, omitting them leaves `secrets audit` able to report clean while an adjacent rejected payload still contains plaintext credentials.
  - confidence: 0.82
