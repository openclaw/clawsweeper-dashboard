## Summary

Keep open: the update fixes the earlier rejected-payload and invalid-active-config gaps and now supplies real CLI proof, but the new backup scanner still parses copied `openclaw.json` artifacts as strict JSON even though OpenClaw config is documented and read as JSON5, leaving a concrete false-negative path for secret residues.

## What This Changes

The PR expands `openclaw secrets audit` to scan adjacent `openclaw.json` backup and rejected-payload artifacts, adds focused regression tests, and documents the new audit scope.

## Best Possible Solution

Keep the audit expansion, but parse adjacent config artifacts with OpenClaw's config JSON5 parser while preserving the current no-ref-resolution behavior for backup-only SecretRefs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: put a plaintext provider `apiKey` in a JSON5 `.bak` or `.pre-update` config artifact with a comment or trailing comma, and the PR's strict-JSON backup parser will skip it.

## Solution Assessment

No. Adding adjacent artifact scanning to `secrets audit` is the right surface, but config artifacts need the same JSON5 parsing contract as active `openclaw.json` before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse config artifacts as JSON5:** `src/secrets/audit.ts:276-279`
  - body: Because `openclaw.json` is documented and parsed as JSON5, backups created by the config backup paths can preserve comments or trailing commas from the copied source file. This new scan calls `readJsonObjectIfExists`, which uses strict `JSON.parse`, so those artifacts are added to `filesScanned` and then silently skipped without `PLAINTEXT_FOUND`; parse config artifacts with the same JSON5 parser and add a JSON5 backup fixture.
  - confidence: 0.89
