## Summary

Keep open: the PR expands a useful security audit path, but the current diff still has a false-negative for valid JSON5 config backups because backup artifacts are parsed with strict `JSON.parse` and skipped on parse errors.

## What This Changes

The branch adds `openclaw secrets audit` coverage for adjacent `openclaw.json` backup/rejected artifacts, tests those cases, and documents the new audit scope.

## Best Possible Solution

Keep the audit expansion, but parse backup/rejected config artifacts with OpenClaw's JSON5-aware config parser and add a JSON5 backup regression fixture.

## Reproduction Assessment

Yes, source-reproducible. A valid JSON5 `openclaw.json.bak` with a comment or trailing comma and a plaintext `models.providers.*.apiKey` is enumerated by the PR but rejected by `JSON.parse` before findings are collected.

## Solution Assessment

No. The audit surface and invalid/rejected artifact coverage are the right direction, but backup artifacts should reuse JSON5 config parsing while still skipping backup-only SecretRef resolution.
