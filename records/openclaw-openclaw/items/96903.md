## Summary

Keep open for normal review and landing: current main still reads xAI video create/poll success JSON with unbounded `response.json()`, while this PR is a focused security hardening that reuses the shared bounded reader, includes create/poll regressions, and has sufficient terminal proof.

## What This Changes

The PR routes xAI video create and poll success JSON responses through `readResponseWithLimit` with a 16 MiB cap and updates the xAI video tests to use streamed `Response` bodies plus oversized create/poll regressions.

## Best Possible Solution

Land the bounded-reader fix after maintainers accept the 16 MiB control-JSON cap and the exact-head CI/proof gates finish.

## Reproduction Assessment

Source-reproducible: current main's `readXaiVideoJson` uses `response.json()` for both submit and poll success bodies, and Undici's implementation fully reads the body before parsing. I did not run a current-main OOM repro.

## Solution Assessment

Yes, this is the best fix shape: one private helper covers both create and poll paths, reuses the existing bounded reader, and preserves xAI's malformed-response behavior. The shared `readProviderJsonObjectResponse` would also be plausible, but this wrapper keeps the established xAI error text with less churn.
