## Summary

This PR is still necessary because current main still routes xAI video create and poll success JSON through an unbounded `response.json()` helper, while the PR replaces that private helper with the existing bounded provider JSON reader. I found no line-level correctness blocker; the remaining maintainer decision is acceptance of the intentional 16 MiB fail-closed control-response cap.

## What This Changes

The PR changes the xAI video generation provider create/poll JSON reader to use the shared bounded provider JSON reader and updates xAI provider tests plus provider HTTP mocks to exercise streamed Response bodies.

## Best Possible Solution

Land the plugin-scoped bounded-reader fix for xAI video create and poll success JSON if maintainers accept the shared 16 MiB fail-closed control-response cap.

## Reproduction Assessment

Yes, at source level: current main calls `response.json()` in the xAI video helper used by both create and poll success responses. I did not run the oversized-body reproduction locally because this review is read-only, but the PR body includes terminal loopback proof plus a negative control.

## Solution Assessment

Yes, subject to maintainer acceptance of the cap: fixing the private xAI helper once and reusing the established Plugin SDK bounded reader is the narrow owner-boundary solution. A broader provider refactor or provider-specific duplicate reader would add more surface without solving a different problem.
