## Summary

Keep this PR open; the performance problem is credible, but this slim memo can serve stale plugin metadata after same-process inventory or manifest changes, and the PR lacks exact-branch real CLI proof.

## What This Changes

The branch adds a single-slot process memo for `loadPluginMetadataSnapshot`, clone-on-return behavior, lifecycle clearing, and focused memo regression tests.

## Best Possible Solution

Keep the performance optimization only if the memo key includes stat-based plugin discovery/install freshness like the broader related PR, or scope reuse to explicitly owned current snapshots, then supply PR-head CLI timing proof.

## Reproduction Assessment

Yes, at source level: two same-input calls can hit the new process memo before discovery or manifest scanning reruns, so same-process plugin inventory or manifest changes are not visible on the next metadata read. I did not mutate the checkout to run a filesystem repro.

## Solution Assessment

No; a process memo is plausible for this hot path only when its key carries the same plugin inventory and manifest freshness facts that current main checks by rescanning. The safer direction is to use the stat-based discovery fingerprint from the broader related PR or pass explicit snapshots through hot callers.
