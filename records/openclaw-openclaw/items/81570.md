## What This Changes

The PR adds a process-local memoization layer around plugin metadata snapshot loading, clears it at current metadata lifecycle boundaries, and adds memo regression tests.

## Best Possible Solution

Add the same effective registry/manifest freshness fingerprinting used by the broader direction in https://github.com/openclaw/openclaw/pull/81064, or avoid memoizing no-index discovery calls; then provide exact PR-head CLI timing proof plus stale-invalidation coverage.

## Reproduction Assessment

Partly: the original performance problem has source evidence and related CLI timing proof, but not exact PR-head real CLI proof. The blocking stale-cache issue is source-reproducible by calling `loadPluginMetadataSnapshot` twice without `params.index` after changing persisted registry or manifest inputs between calls.

## Solution Assessment

No. A process memo can be the right shape, but this narrow key needs effective persisted-registry and manifest/package file freshness, or it should not cache no-index discovery calls.
