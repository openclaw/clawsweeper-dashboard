## Summary

Keep open: this active draft perf hotfix has current-head production CLI proof, but the cache-hit path reuses mutable installed-index data from the memo and needs a small correctness fix before merge.

## What This Changes

The PR adds a process-local memo around `loadPluginMetadataSnapshot`, narrows memo eligibility/keying, adjusts recovered install-record handling, and adds plugin metadata snapshot and registry tests.

## Best Possible Solution

Keep the narrow process memo, but make cache-hit returns mutation-isolated or fully freeze the memoized installed-index data, while leaving broader freshness hardening to the existing follow-up.

## Reproduction Assessment

Yes, at source/proof level: current main rebuilds plugin metadata snapshots on repeated calls, and the PR body provides current-head beta6 CLI timing proof for the slow path. I did not run the timing commands in this read-only checkout.

## Solution Assessment

No, not as submitted: the narrow memo is the right hotfix direction, but cache hits must not expose mutable memoized installed-index records. Deep-cloning the cache hit or freezing all nested memoized index data is the safer narrow fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clone index records on cache hits:** `src/plugins/plugin-metadata-snapshot.ts:363`
  - body: The cache-hit path uses `cloneRecords: false`, so it returns shallow copies that still share `snapshot.index.plugins`, `snapshot.index.installRecords`, and diagnostic objects with the memoized snapshot. Unlike the old fresh-build path, any caller that mutates the returned index data can poison later metadata lookups in the same process; please deep-clone or freeze the memoized index data and add a regression test that mutates `snapshot.index.plugins[0]` or `installRecords` before the next cache hit.
  - confidence: 0.85
