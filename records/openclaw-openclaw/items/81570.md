## Summary

Keep open: the latest head fixes the earlier derived-registry memo problem, but the persisted-registry memo still bypasses current-main recovery of managed npm install records, and exact PR-head real behavior proof is still insufficient.

## What This Changes

The PR adds a single-slot in-process memo around plugin metadata snapshot loading, clone-on-return handling, cache-clearing hooks, and memo regression tests.

## Best Possible Solution

Keep the performance optimization only after the memo key covers recovered managed npm install-record freshness, or reuse the broader fingerprint approach from https://github.com/openclaw/openclaw/pull/81064, then add redacted exact PR-head CLI timing proof.

## Reproduction Assessment

Yes, at source level: first memoize a persisted registry, then add a recoverable managed npm plugin under the state directory without changing the persisted registry file. Current main would re-run the missing recovered-record check on the next metadata read; this PR can hit the memo first.

## Solution Assessment

No; an outer process memo is maintainable only if its key includes every freshness input that current main checks before accepting persisted metadata. The safer narrow fix is to fingerprint recovered managed npm install-record inputs or bypass the memo when that check can alter the registry source.
