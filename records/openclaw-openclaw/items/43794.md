## Summary

Keep open: current main and the latest release still allow plaintext credential storage and only mitigate with file permissions, SecretRefs, and limited legacy OAuth sidecar compatibility; the related auth-store sealing PR is open, conflicting, and only partial for this issue's keychain/master-password-at-rest request.

## What This Changes

Request default encrypted-at-rest storage for sensitive OpenClaw credentials, including config/auth-profile migration and keychain or master-password-backed decryption.

## Best Possible Solution

Keep this as the canonical security-owned design tracker for one credential-at-rest contract covering config, auth profiles, SecretRefs, keychain/master-password backends, migration, recovery, rotation, backups, and runtime plaintext boundaries.

Not applicable as a failing-runtime reproduction: this is a security feature/design request. Source and docs provide a high-confidence negative check that plaintext credentials remain supported and no default native keychain or master-password encryption contract exists on current main.

## Solution Assessment

Unclear for implementation: the goal is valid, but the best solution needs explicit security/product decisions on key management, migration, recovery, rotation, backups, compatibility, and where plaintext may exist after decryption.
