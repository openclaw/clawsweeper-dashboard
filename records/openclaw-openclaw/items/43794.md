## Summary

Keep open. Current main still supports plaintext credential persistence with opt-in SecretRefs and audit warnings, while the threat model continues to mark token theft from config storage as high residual risk; related PR #39059 is open/unmerged and #65239 was closed in favor of keeping this as the canonical item.

## What This Changes

Requests credential-at-rest encryption for sensitive config/auth fields using a keychain or master-password style mechanism while preserving backward compatibility.

## Best Possible Solution

Define and ship a security-owned credential-at-rest policy end to end, or explicitly document SecretRefs/external secret managers as the supported posture and update the threat model accordingly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: source and docs on current main show plaintext credentials are still supported, saved as JSON, preserved by onboarding paths, and reported by audit as PLAINTEXT_FOUND.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the storage contract; encryption at rest addresses the risk, but keychain/master-password, recovery, migration, and supported-surface scope need security/product approval.
