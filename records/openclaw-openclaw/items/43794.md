## Summary

Keep open: current main and the latest release reduce plaintext exposure with opt-in SecretRefs, but they do not implement encrypted-at-rest credential storage or a backward-compatible migration for existing credentials. ## What This Changes

The issue requests encrypted at-rest storage for config and auth credentials using system keychain or master-password decryption with backward-compatible migration.

## Best Possible Solution

Keep this as the canonical security tracker until maintainers choose and land a security-reviewed key-management plus migration design covering config, auth profiles, OAuth/rotating material, docs, and upgrade proof.

Yes, as a source-level negative check rather than a runtime bug: current docs and types show plaintext credentials still work, SecretRefs are opt-in, and OAuth/rotating auth material remains outside read-only SecretRef resolution.

## Solution Assessment

No current implementation is the complete best fix yet. The maintainable solution needs a security-reviewed keychain or passphrase model plus doctor/config migration and upgrade proof, not only another optional resolver example.
