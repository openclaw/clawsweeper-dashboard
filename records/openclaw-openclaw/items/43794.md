## Label Justifications

- `P1`: Credential-at-rest protection is a high-impact security/auth-provider gap already called out in the threat model, but it is not an active outage or confirmed emergency compromise.
- `impact:security`: The issue concerns API keys, tokens, backups, local files, and credential exposure from plaintext storage.
- `impact:auth-provider`: The affected data includes model/provider API keys, OAuth/static tokens, and auth-profile material used for provider access.

## Summary

Keep open. Current main still supports plaintext credential persistence, SecretRefs remain opt-in env/file/exec references, and the legacy OAuth sidecar path is read-only compatibility rather than the requested native encrypted-at-rest config storage; the related secret-broker proposal is adjacent but does not replace this tracker.

## What This Changes

The issue requests encrypted-at-rest storage for OpenClaw credentials and config secrets using system keychain or master-password support while preserving compatibility.

## Best Possible Solution

Keep this as the focused security-owned tracker for native encrypted-at-rest credential storage, and require a design that covers storage scope, migration, key management, recovery, rotation, SecretRef interop, backups, and runtime plaintext boundaries before implementation PRs land.

Not applicable as a failing-runtime reproduction: this is a security feature/design request. Source and docs give a high-confidence negative check that plaintext credentials remain supported and native encrypted-at-rest config storage is not implemented.

## Solution Assessment

Unclear for implementation. The goal is valid, but the safest path needs explicit decisions on keychain/master-password behavior, migration, recovery, rotation, backups, and plaintext runtime boundaries.
