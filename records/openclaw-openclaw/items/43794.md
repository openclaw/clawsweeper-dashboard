## Summary

Keep open: current main and v2026.6.6 support opt-in SecretRefs and auditing, but they still document plaintext credentials as valid and leave OAuth/rotating credential material outside SecretRef coverage. The linked closing PR is open and only adds a keychain exec SecretRef resolver, so it is not yet a merged encrypted-at-rest migration for existing config/auth stores.

## What This Changes

The issue requests encrypted-at-rest storage or keychain-backed migration for sensitive OpenClaw credentials currently stored in local config/auth files.

## Best Possible Solution

Land or revise a maintainer-approved credential storage design that covers OpenClaw-owned config and auth stores, defines keychain/passphrase behavior, provides doctor/migration and rotation guidance, and preserves SecretRef/plugin boundaries.

## Reproduction Assessment

Yes. Source and shipped docs show plaintext credential fields still work, SecretRefs are opt-in, and OAuth/rotating credentials remain outside SecretRef coverage, so the reported at-rest exposure is source-reproducible on current main and v2026.6.6.

## Solution Assessment

No. Closing this issue now is not the best solution because the linked PR is open and narrower than the requested backward-compatible encrypted-at-rest migration across OpenClaw-owned credential stores.
