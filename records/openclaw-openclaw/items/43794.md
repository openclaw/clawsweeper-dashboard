## Label Justifications

- `P1`: Credential-at-rest protection is a high-impact security/auth-provider design gap already called out in the threat model, but it is not an emergency outage.
- `impact:security`: The issue concerns API keys, tokens, config files, backups, and other sensitive data stored on disk.
- `impact:auth-provider`: The affected data includes provider API keys, OAuth/static tokens, and auth-profile material used for model/provider access.

## Summary

Keep open. Current main still treats native encryption at rest as an unresolved security roadmap item: SecretRefs reduce plaintext storage for supported opt-in surfaces, but plaintext config/auth values remain valid and the open auth-store sealing PR only covers a subset of the broader key-management and migration design.

## What This Changes

Request native encrypted-at-rest handling for OpenClaw credential/config secrets, with keychain or master-password decryption and backward-compatible migration.

## Best Possible Solution

Keep this as the canonical security-owned design tracker and require an explicit key-management, migration, recovery, rotation, backup, and plaintext-runtime boundary before merging native encryption work.

## Reproduction Assessment

Not applicable; this is a security feature/design request rather than a failing runtime path. Source and docs provide a high-confidence negative check that current main still permits plaintext credentials and only offers opt-in SecretRefs plus narrow legacy OAuth sidecar handling.

## Solution Assessment

Unclear for implementation. The goal is valid, but the maintainable path needs security-owned decisions around key management, migration, recovery, startup behavior, rotation, backups, and plaintext runtime exposure before code lands.
