## Summary

Keep open. Current main has SecretRefs and a narrower OAuth-profile encrypted secret payload path, but it still does not implement the requested native encryption-at-rest model for general config credentials with keychain/master-password handling, migration, recovery, and rotation.

## What This Changes

The issue requests built-in encryption-at-rest for sensitive OpenClaw config credentials using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Make a security-owned design decision for credential encryption at rest, then either implement a documented key-management, migration, recovery, rotation, redaction, and audit model or explicitly define SecretRefs/external vaults/process isolation as the supported boundary.

## Reproduction Assessment

Not applicable; this is a security feature request rather than a runtime bug. Static source/docs inspection gives a high-confidence verification path: SecretRefs and narrower OAuth-profile encryption exist, but general config credential encryption-at-rest is still absent and the threat model still recommends it.

## Solution Assessment

Unclear. Built-in encryption may be the right outcome, but the maintainable fix requires explicit key management, migration, recovery, rotation, and plaintext runtime-boundary decisions before code should land.
