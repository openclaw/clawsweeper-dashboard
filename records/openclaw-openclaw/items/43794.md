## Label Justifications

- `P1`: Credential-at-rest protection is a high-impact security/auth-provider gap documented in the threat model, even without evidence of active compromise.
- `impact:security`: The request concerns local storage of API keys, tokens, backups, and sensitive credential exposure.
- `impact:auth-provider`: The affected data includes provider API keys, OAuth/static tokens, gateway auth tokens, auth profiles, and SecretRef behavior.

## Summary

Keep open. Current main mitigates plaintext credential exposure with SecretRefs, owner-only permissions, and a doctor warning, but plaintext credentials are still a supported/default storage shape and the threat model still calls encryption at rest a high residual-risk recommendation.

## What This Changes

Request encrypted-at-rest storage for sensitive OpenClaw config and auth-profile credentials, using keychain or master-password decryption while preserving compatibility.

## Best Possible Solution

Keep this as the canonical security-owned tracker and design one encrypted-at-rest credential contract covering config, auth profiles, backups, SecretRefs, migration, recovery, rotation, and runtime plaintext boundaries before implementation PRs land.

## Reproduction Assessment

Not applicable as a failing-runtime reproduction: this is a security feature/design request. Source and docs give a high-confidence negative check that plaintext credentials remain supported on current main.

## Solution Assessment

Unclear for implementation. The goal is valid, but the best solution needs explicit security/product decisions on key management, migration, recovery, rotation, backups, compatibility, and where plaintext may exist after decryption.
