## Summary

Keep open. Current main has shipped opt-in SecretRefs for many credential surfaces, but it still does not provide native encryption-at-rest with keychain/master-password management, migration, and rotation; the threat model still calls that residual risk out explicitly.

## What This Changes

The issue requests encrypted-at-rest storage for sensitive OpenClaw credentials using a system keychain or master password while preserving compatibility.

## Best Possible Solution

Make a security-owned design decision for native credential encryption-at-rest, then either implement a documented key-management, migration, rotation, recovery, and audit model or explicitly define SecretRefs/external vaults/process isolation as the supported boundary.

## Reproduction Assessment

Not applicable as a security feature request rather than a runtime bug. Static inspection gives a high-confidence verification path: current main has SecretRefs but no native encrypted-at-rest credential store, and the threat model still recommends token encryption and rotation.

## Solution Assessment

Unclear. The requested direction is plausible, but built-in transparent encryption is not a safe automatic fix because the right key-management, migration, recovery, rotation, and agent-isolation model needs maintainer security/product judgment.
