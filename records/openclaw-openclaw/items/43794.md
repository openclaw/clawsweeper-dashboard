## Summary

Keep open. Current main still permits plaintext credential persistence, SecretRefs and file permissions are documented as mitigations rather than transparent encryption at rest, the threat model still lists token theft from config storage as a high residual risk, and related PR #39059 is open rather than merged.

## What This Changes

The issue requests built-in encryption or keychain/master-password handling for sensitive credentials stored in OpenClaw config and auth-profile files.

## Best Possible Solution

Define a security-owned credential-at-rest policy, then ship the chosen storage contract with migration, recovery, rotation guidance, docs, and regression tests for OpenClaw-owned config and auth stores.

## Reproduction Assessment

Yes: source and docs on current main show plaintext credentials are still supported and persisted as JSON, and the audit path reports PLAINTEXT_FOUND rather than preventing plaintext storage. I did not run a live write because this review was required to stay read-only.

## Solution Assessment

Unclear until maintainers choose the storage contract. Encryption at rest addresses the documented risk, but keychain versus passphrase, migration, recovery, rotation, SecretRef interaction, and supported-surface scope need secops/product approval before implementation.
