## Label Justifications

- `P1`: Credential-at-rest protection is a high-impact security/auth-provider gap already called out in the threat model, but it is not an active outage or confirmed emergency compromise.
- `impact:security`: The issue concerns API keys, tokens, backups, local files, and credential exposure from plaintext storage.
- `impact:auth-provider`: The affected data includes model/provider API keys, OAuth/static tokens, and auth-profile material used for provider access.

## Summary

Keep open. Current main still documents token theft from plaintext config/storage as a high residual risk, and the implemented SecretRef model is opt-in rather than native encrypted-at-rest credential storage.

## What This Changes

The issue requests native encryption at rest for OpenClaw credentials and sensitive config values, including backward-compatible migration to keychain or master-password backed storage.

## Best Possible Solution

Keep this as the canonical security-owned tracker and design a credential-at-rest contract that covers storage scope, migration, keychain/master-password handling, recovery, rotation, backups, SecretRef interop, and runtime plaintext boundaries before landing implementation PRs.

## Reproduction Assessment

Not applicable as a failing-runtime reproduction: this is a security feature/design request. Current-main source and docs give a high-confidence negative check that plaintext credentials remain supported and native encrypted-at-rest storage is not implemented.

## Solution Assessment

Unclear for implementation. The goal is valid, but the best solution needs explicit security/product decisions on key management, migration, recovery, rotation, backup behavior, and where plaintext may exist after decryption.
