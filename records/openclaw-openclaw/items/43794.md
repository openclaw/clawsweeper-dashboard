## Label Justifications

- `P1`: Credential-at-rest protection is a high-impact security/auth-provider gap documented in the threat model, even though there is no confirmed active compromise.
- `impact:security`: The request concerns local storage of API keys, tokens, config files, backups, and sensitive credential exposure.
- `impact:auth-provider`: The affected data includes model/provider API keys, OAuth/static tokens, gateway auth tokens, auth profiles, and SecretRef behavior.

## Summary

Keep open. Current main has opt-in SecretRefs, owner-only permissions, and a new doctor warning for plaintext secret-bearing config fields, but plaintext credentials are still a supported config/auth-profile state and the threat model still recommends token encryption at rest.

## What This Changes

Request encrypted-at-rest storage for sensitive config and auth credentials, with keychain or master-password decryption, backward-compatible migration, and rotation support.

## Best Possible Solution

Keep this as the canonical security-owned tracker and design one encrypted-at-rest credential contract covering config, auth profiles, backups, SecretRefs, migration, recovery, rotation, and runtime plaintext boundaries before implementation PRs land.

## Reproduction Assessment

Not applicable as a failing-runtime reproduction: this is a security feature/design request. Source and docs give a high-confidence negative check that plaintext credentials remain supported on current main.

## Solution Assessment

Unclear for implementation. The goal is valid, but the best solution needs explicit security/product decisions on key management, migration, recovery, rotation, backups, compatibility, and where plaintext may exist after decryption.
