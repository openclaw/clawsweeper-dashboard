## Summary

Keep open. Current main still has opt-in SecretRefs and a narrower OAuth-profile encrypted payload path, but it does not implement native encryption at rest for general config/static credentials with keychain or master-password handling, migration, recovery, and rotation; the related `.env` encryption request at https://github.com/openclaw/openclaw/issues/65239 was closed as a duplicate of this issue, so this remains the canonical tracker.

## What This Changes

The issue requests encryption-at-rest for sensitive OpenClaw configuration credentials using system keychain or a master password while preserving backward compatibility.

## Best Possible Solution

Make a security-owned design decision, then either implement a documented key-management, migration, recovery, rotation, redaction, and audit model or explicitly define SecretRefs/external vaults/process isolation as the supported boundary.

## Reproduction Assessment

Not applicable; this is a security feature request rather than a runtime bug. Static source and docs checks give a high-confidence verification path: SecretRefs and OAuth-profile encryption exist, but general credential encryption-at-rest is still absent.

## Solution Assessment

Unclear. Built-in encryption may be the right outcome, but the maintainable fix requires explicit key management, migration, recovery, rotation, and plaintext runtime-boundary decisions before code should land.
