## Label Justifications

- `P1`: Credential-at-rest protection is a high-impact security/auth-provider gap already documented in the threat model, but there is no confirmed active compromise or outage.
- `impact:security`: The issue concerns API keys, tokens, local config/auth files, backups, and sensitive credential exposure.
- `impact:auth-provider`: The affected data includes model/provider API keys, OAuth/static tokens, gateway auth tokens, and SecretRef/auth-profile behavior.

## Summary

Keep open. Current main has substantial SecretRef, audit, apply, and doctor-warning mitigation, but plaintext credential storage is still supported and sometimes default, so the requested native encrypted-at-rest credential storage and key-management/migration contract is not implemented.

## What This Changes

Request native encrypted-at-rest storage for sensitive OpenClaw config and auth credentials with keychain or master-password support and backward-compatible migration.

## Best Possible Solution

Keep this as the canonical security-owned tracker and design an encrypted-at-rest credential contract that covers storage scope, migration, keychain/master-password handling, recovery, rotation, backups, SecretRef interop, and runtime plaintext boundaries before implementation PRs land.

## Reproduction Assessment

Not applicable as a failing-runtime reproduction: this is a security feature/design request. Current-main source and docs give a high-confidence negative check that plaintext credentials remain supported and native encrypted-at-rest storage is not implemented.

## Solution Assessment

Unclear for implementation. The goal is valid, but the best solution needs explicit security/product decisions on key management, migration, recovery, rotation, backup behavior, compatibility, and where plaintext may exist after decryption.

## Security Review

Status: needs_attention

Summary: This remains security-sensitive credential-storage design work with no patch to audit; current main permits plaintext credential storage and offers SecretRefs as opt-in mitigation.

Concerns:

- **[medium] Define encryption key management first:** `docs/security/THREAT-MODEL-ATLAS.md:210`
  - body: Native encryption needs a clear keychain or master-password contract, fallback behavior, service startup semantics, recovery, migration, and rotation before implementation.
  - confidence: 0.91
- **[medium] Cover config files and backups, not only auth stores:** `src/config/io.ts:2152`
  - body: Current config persistence writes JSON with owner-only permissions and rotates backup artifacts, so auth-store-only sealing would not address the broader `openclaw.json` and backup exposure requested here.
  - confidence: 0.9

  - confidence: 0.87

  - body: SecretRefs currently resolve into an in-memory runtime snapshot; native decryption needs rules for which runtime paths, logs, diagnostics, snapshots, tools, and agents may see plaintext.
  - confidence: 0.84
