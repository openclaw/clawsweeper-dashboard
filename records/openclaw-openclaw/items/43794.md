## Summary

Keep open. Current main still leaves OpenClaw config and auth credentials plaintext-compatible, with SecretRefs as an opt-in externalization path; an open hardening PR covers only a subset of auth-store sealing and does not close the broader encrypted-at-rest design request.

## What This Changes

Request native encrypted-at-rest storage for sensitive OpenClaw config credentials using a keychain or master-password-backed migration path.

## Best Possible Solution

Keep this as the canonical security-owned design tracker, review partial sealing work as a subset, and define the supported key-management, migration, backup, rotation, and plaintext-runtime boundary before code lands.

Not applicable; this is a security feature/design request rather than a failing runtime path. Source and docs provide a high-confidence negative check that current main still permits plaintext credentials and only offers opt-in SecretRefs plus a narrow legacy OAuth sidecar path.

## Solution Assessment

Unclear for implementation. The goal is valid, but the maintainable path needs security-owned decisions around key management, migration, recovery, startup behavior, rotation, backups, and plaintext runtime exposure before code lands.
