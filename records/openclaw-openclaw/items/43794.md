## Summary

Keep open. Current main still treats SecretRefs as an opt-in alternative while plaintext credentials remain valid, so native encrypted-at-rest credential storage is unresolved security/product design work.

## What This Changes

Request native encrypted-at-rest storage for sensitive config credentials using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Make a security/product decision, then either implement a documented key-management, migration, recovery, rotation, and runtime-exposure model, or explicitly document SecretRefs and external vaults as the supported boundary.

## Reproduction Assessment

Not applicable; this is a security feature/design request rather than a failing runtime path. Source and docs provide a high-confidence negative check that main still permits plaintext credentials and only offers opt-in SecretRefs.

## Solution Assessment

Unclear for implementation. Native encryption may be the right end state, but the maintainable path needs security-owned decisions around key management, migration, recovery, startup, rotation, and plaintext runtime exposure before code lands.
