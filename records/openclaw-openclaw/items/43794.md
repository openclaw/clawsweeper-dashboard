## Summary

Keep open. Current main still treats encrypted-at-rest credential storage as unresolved security design work: SecretRefs are opt-in env/file/exec references, inline auth-profile secrets remain valid, and the threat model still lists plaintext token storage as a high residual risk.

## What This Changes

Request native encrypted-at-rest storage for OpenClaw credentials in config/auth files, using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Make a secops-owned product/design decision, then either implement a documented key-management, migration, recovery, rotation, and runtime exposure model, or explicitly define SecretRefs/external vaults/full-disk encryption as the supported boundary.

## Reproduction Assessment

Not applicable; this is a security feature/design request rather than a runtime bug. Source and docs checks show current main still permits plaintext config/auth credentials and offers opt-in SecretRefs rather than native encrypted-at-rest storage.

## Solution Assessment

Unclear for implementation. Native encryption may be the right end state, but the maintainable path needs a security-owned key-management, migration, recovery, service-startup, rotation, and plaintext runtime-boundary decision before code lands.
