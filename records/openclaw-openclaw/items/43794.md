## Summary

Keep open. Current main has mitigations such as SecretRefs, credential file permissions, and security docs, but it still supports plaintext credentials in config and auth-profile files, so the requested encrypted-at-rest credential storage and migration/key-management contract remains unresolved and security-sensitive.

## What This Changes

Request native encryption at rest for sensitive OpenClaw credentials using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Keep this as the canonical security-owned tracker and design a cohesive encrypted-at-rest credential contract covering config, auth profiles, backups, SecretRefs, migration, recovery, rotation, and runtime plaintext boundaries before implementation PRs land.

Not applicable as a failing-runtime reproduction: this is a security feature/design request. Source and docs give a high-confidence negative check that plaintext credential storage remains supported on current main.

## Solution Assessment

Unclear for implementation. The goal is valid, but the best solution needs explicit security/product decisions on key management, migration, recovery, rotation, backup behavior, compatibility, and where plaintext may exist after decryption.
