## Summary

Keep open. Current main still treats plaintext credential storage as supported behavior and documents SecretRefs, scrubbing, audit findings, and file permissions as mitigations rather than transparent encryption at rest; the related auth-store sealing PR is open and unmerged, so the requested security posture has not shipped.

## What This Changes

The issue requests encrypted-at-rest storage for sensitive credentials in OpenClaw config and auth files, with backward-compatible migration and keychain or master-password support.

## Best Possible Solution

Define and ship a security-owned credential-storage policy end to end: either transparent encrypted-at-rest storage for OpenClaw-owned credential files with backward-compatible migration, recovery, rotation, docs, and tests, or an explicit supported posture that mandates SecretRefs/external secret managers for at-rest secrecy and updates the threat model accordingly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for current behavior without mutating the checkout: current docs and source show plaintext credentials remain supported, persisted as JSON, and detected by `openclaw secrets audit` as PLAINTEXT_FOUND.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the credential-storage contract; the report identifies a real security gap, but the fix should be designed and reviewed as security work rather than attempted as a narrow automated patch.
