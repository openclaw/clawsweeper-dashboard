## Summary

Keep open: current main has opt-in SecretRefs, audit/apply helpers, and private-mode SQLite/config writes, but it still supports plaintext credential storage and does not implement credential encryption/keychain migration at rest. ## What This Changes

The issue requests encryption or keychain-backed storage for credentials currently persisted in OpenClaw config and auth files.

## Best Possible Solution

Choose and land a maintainer-reviewed credential-at-rest design that covers supported config/auth credential surfaces, documents the security boundary, and includes explicit upgrade or migration behavior for existing plaintext stores.

## Reproduction Assessment

Yes. Source and docs show plaintext remains a supported credential input, SecretRefs are opt-in, and auth-profile secrets are stored as JSON text in the agent SQLite database.

## Solution Assessment

Unclear. The requested direction addresses a real security gap, but the best fix needs maintainer/security choice between keychain SecretRefs, auth-store sealing, migration commands, recovery, and rotation semantics.
