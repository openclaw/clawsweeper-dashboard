## Summary

Keep open. Current main still supports plaintext credential storage and documents SecretRefs, scrubbing, audit findings, and file permissions as mitigations rather than transparent encryption at rest; the related auth-store sealing PR remains open and unmerged, while the newer duplicate #65239 was closed in favor of keeping this issue canonical.

## What This Changes

Request encrypted-at-rest storage for sensitive OpenClaw credential/config fields with backward-compatible migration and keychain or master-password support.

## Best Possible Solution

Define and ship a security-owned credential-storage policy end to end: either transparent encrypted-at-rest storage for OpenClaw-owned credential files with migration, recovery, rotation, docs, and tests, or an explicit supported posture that mandates SecretRefs/external secret managers for at-rest secrecy and updates the threat model accordingly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for current behavior without mutating the checkout: current docs and source show plaintext credentials remain supported, persisted as JSON, and reported by `openclaw secrets audit` as `PLAINTEXT_FOUND`.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the credential-storage contract; the report identifies a real security gap, but the fix should be designed and reviewed as security work rather than attempted as a narrow automated patch.
