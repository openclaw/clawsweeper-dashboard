## Summary

Keep open: current main still supports plaintext credential storage, with SecretRefs, file permissions, and doctor warnings as mitigations, and the related duplicate cleanup leaves this as the canonical security/product design tracker.

## What This Changes

The issue requests encrypted-at-rest storage for tokens, API keys, and other sensitive OpenClaw config/auth credentials with keychain or master-password support and backward compatibility.

## Best Possible Solution

Keep this as the canonical security-owned tracker and design one encrypted-at-rest credential contract covering keychain/master-password backends, migration, recovery, rotation, backups/rejected artifacts, auth profiles, SecretRef compatibility, and runtime plaintext boundaries.

Not applicable as a failing-runtime reproduction: this is a security feature/design request. Source and docs give a high-confidence negative check that plaintext values remain supported and native encryption is absent on current main.

## Solution Assessment

Unclear for implementation. The goal is valid, but the best solution needs explicit security/product decisions on key management, migration, recovery, rotation, backups, compatibility, and where plaintext may exist after decryption.
