## Summary

Keep open: current main and v2026.6.10 still allow plaintext credential storage, and the landed SecretRef system is an opt-in mitigation rather than a completed encrypted-at-rest credential design. Related work either covers a subset, remains open and blocked, or was closed as plugin/ClawHub scope.

## What This Changes

Request encryption at rest for sensitive OpenClaw config and auth credentials using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Land a maintainer/security-approved credential-at-rest design that covers openclaw.json, the active SQLite auth-profile store, doctor migration, key management, recovery, and rotation while keeping SecretRefs as a supported mitigation.

## Reproduction Assessment

Yes, as a source-level missing-capability check. Current source and docs show plaintext storage remains supported, SecretRefs are opt-in, and active auth-profile payloads are stored as JSON in SQLite rather than encrypted envelopes.

## Solution Assessment

Unclear. The request targets a real security gap, but maintainers still need to choose the key-management model, migration path, recovery behavior, and rotation guarantees before implementation.
