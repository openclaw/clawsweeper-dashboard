## Summary

Keep this open: current releases provide SecretRefs plus migration and plaintext-residue auditing, but native OS-keychain support remains in the open linked implementation PR and has not yet been proven through a live gateway flow. The existing framework mitigates the central risk for supported credentials, but it does not fully deliver the requested keychain-backed upgrade experience.

## What This Changes

Add encrypted or OS-keychain-backed storage and migration for credentials currently persisted in OpenClaw configuration and authentication stores.

## Best Possible Solution

Keep the canonical SecretRef architecture, land a security-reviewed OS-keychain provider or resolver with live gateway proof, and provide a documented migration/audit workflow that removes plaintext residue from every supported store without claiming coverage for excluded rotating credentials.

## Reproduction Assessment

Yes. Current source documentation explicitly identifies plaintext credential residue in `openclaw.json`, `auth-profiles.json`, `.env`, and generated model files, and tests exercise detection of those states.

## Solution Assessment

No, blanket in-place config encryption is not the best established direction. The narrower maintainable solution is the existing SecretRef model backed by a reviewed keychain resolver and a complete migration/audit flow, with excluded credential classes documented clearly.
