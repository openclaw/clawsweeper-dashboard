## Summary

Keep open. Current main still lacks native encrypted-at-rest storage for general OpenClaw config/auth credentials; SecretRefs are opt-in env/file/exec references, inline auth-profile credentials remain valid, and the legacy OAuth sidecar path explicitly migrates back to inline credentials rather than establishing a general keychain model.

## What This Changes

Request built-in encryption or keychain/master-password support so OpenClaw tokens and API keys are protected at rest while preserving compatibility.

## Best Possible Solution

Make a security-owned design decision, then implement a documented key-management, migration, recovery, rotation, redaction, and runtime plaintext-exposure model, or explicitly define SecretRefs/external vaults as the supported boundary.

## Reproduction Assessment

Not applicable; this is a security feature/design request rather than a runtime bug. Static source and docs checks provide high-confidence evidence that general encrypted-at-rest config/static credential storage is still absent.

## Solution Assessment

Unclear. Native encryption may be the right end state, but the maintainable solution needs key-management, migration, recovery, rotation, headless startup, and plaintext runtime-boundary decisions before implementation.
