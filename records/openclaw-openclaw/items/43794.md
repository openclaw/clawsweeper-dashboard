## Summary

Keep open: current main and v2026.6.6 have opt-in SecretRefs, audit, and scrub flows, but plaintext credential storage remains valid and the threat model still lists token encryption at rest as remaining security work. This issue is also paired with open implementation paths at https://github.com/openclaw/openclaw/pull/89107 and https://github.com/openclaw/openclaw/pull/39059, so it should stay open for maintainer security/product direction.

## What This Changes

Issue requests encryption or keychain-backed storage for OpenClaw credentials at rest, including migration from existing plaintext config/auth stores.

## Best Possible Solution

Keep this as the canonical security tracker until maintainers land or explicitly reject a design that covers key management, migration/doctor behavior, current auth-profile SQLite storage, config SecretRefs, OAuth/rotating credentials, and audit/docs proof.

## Reproduction Assessment

Yes. Source and docs show plaintext still works, SecretRefs are opt-in, and auth profile credentials can still persist as literal keys/tokens unless a ref is configured; no live install was run because this review is read-only.

## Solution Assessment

No single current path is clearly the best complete fix. SecretRefs are a useful mitigation, but the full request needs maintainer decisions on key ownership, migration, fail-closed behavior, OAuth boundaries, and whether the durable surface is core, plugin, or both.
