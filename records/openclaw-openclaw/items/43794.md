## Summary

Keep open. Current main still supports plaintext auth/config credential storage, docs frame SecretRefs, auditing, and file permissions as mitigations rather than transparent encryption at rest, the threat model still marks token theft from config storage as high residual risk, and related PR #39059 is open/unmerged while duplicate #65239 was closed in favor of this canonical item.

## What This Changes

Add built-in encryption, keychain, or master-password support for sensitive OpenClaw config and credential fields while preserving backward compatibility.

## Best Possible Solution

Define and ship a security-owned credential-at-rest policy end to end: either transparent encrypted storage for OpenClaw-owned credential files with migration, recovery, rotation, docs, and tests, or an explicit supported posture that requires SecretRefs/external secret managers and updates the threat model accordingly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: current source/docs show plaintext credentials remain supported and persisted as JSON, and the audit path reports them as PLAINTEXT_FOUND rather than preventing them.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the storage contract; encryption at rest addresses the risk, but keychain/master-password, recovery, migration, and supported-surface scope need security/product approval before implementation.
