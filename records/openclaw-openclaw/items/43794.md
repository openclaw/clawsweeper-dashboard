## Summary

Keep open. Current main has opt-in SecretRefs and redaction, but it still persists primary config and backups as plaintext JSON, and the related active work only partially covers auth-store sealing rather than the full encrypted-at-rest config/credential request.

## What This Changes

Request to encrypt sensitive config and credential fields at rest using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Design a security-owned encrypted-at-rest credential strategy covering primary config, credential stores, backups, service artifacts, migration, recovery, and rotation, while keeping SecretRefs as the current opt-in mitigation.

## Reproduction Assessment

Not applicable as a feature/security-hardening request rather than a runtime bug. Source inspection gives a high-confidence negative check: current main still serializes config JSON, copies backups verbatim, and offers opt-in SecretRefs rather than encrypted-at-rest config.

## Solution Assessment

Unclear. The requested direction is valid, but the exact implementation needs maintainer security design for key storage, migration, backup/service artifacts, recovery, and rotation before a narrow patch is safe.
