## Summary

Keep open: current main and the latest inspected release still contain the source-backed Codex app-server liveness gap where active turn-scoped requests suppress idle watchdogs, and non-tool server requests are not given the same timeout protection as tool calls. The exact Telegram/account-rate-limit wedge was not live-reproduced on current main, so the remaining work needs maintainer-reviewed liveness and timeout policy rather than conservative cleanup closure.

## What This Changes

The issue reports Codex app-server turns wedging user-facing Telegram sessions when the last observed notification is `account/rateLimits/updated`, with queued messages stuck behind the processing session until manual session deletion.

## Best Possible Solution

Define Codex app-server turn liveness so silent or never-settling turn-scoped requests release the session safely, distinguish intentional approval/user-input waits from stuck requests, and coordinate terminal-idle timeout budget/config policy with #85242/#85296.

## Reproduction Assessment

No live current-main 429 reproduction was established; source inspection shows current main and v2026.6.8 still have active-request watchdog gates, and non-tool turn request handlers can keep the counter positive until they settle.

## Solution Assessment

Unclear as a complete fix; relaxing the terminal-idle gate is plausible but needs maintainer judgment so legitimate approval, user-input, and MCP elicitation waits are not treated as stuck app-server requests.
