## Summary

Keep open: current main and the latest v2026.5.28 release still suppress the Codex terminal-idle watchdog while app-server turn requests are active, and the terminal-idle timeout is still not a plugin config surface. The exact 429-poisoning path was not live-reproduced in this review, and part of the reporter's main-turn timeout hypothesis is weaker on current source because `turn/start` already passes a request timeout.

## What This Changes

The issue reports a Codex app-server session wedge after rate-limit bursts where terminal-idle recovery remains gated by `activeAppServerTurnRequests > 0` and operators cannot tune the terminal-idle timeout.

## Best Possible Solution

Keep this open until the Codex app-server watchdog semantics are fixed with focused regression coverage, and make the terminal-idle config/default choice explicit rather than bundling it as an unreviewed compatibility change.

## Reproduction Assessment

No live high-confidence 429 reproduction was established in this read-only review. The source path is reproducible: current main and v2026.5.28 still gate terminal-idle scheduling/firing on `activeAppServerTurnRequests > 0`, and the terminal-idle timeout is not configurable.

## Solution Assessment

Unclear as a complete fix: relaxing the terminal-idle active-request gate is source-backed, but adding or lowering a terminal-idle config/default needs maintainer judgment and the main-turn no-timeout hypothesis is not confirmed on current source.
