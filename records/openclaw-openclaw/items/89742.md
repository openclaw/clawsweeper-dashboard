## Summary

Keep open: current main and v2026.6.9 still have the Codex app-server liveness gap where active turn-scoped app-server requests suppress completion and terminal idle watchdogs, and only dynamic tool-call server requests currently get the client-level timeout fallback. The exact 429-triggered shared-client wedge still needs live maintainer reproduction, but source inspection shows the central reported failure mode is not implemented away.

## What This Changes

The issue reports a Codex app-server liveness bug where a 429 rate-limit burst can leave shared channel and cron turns stuck because active turn-scoped requests suppress idle watchdog recovery.

## Best Possible Solution

Define a Codex app-server liveness policy that releases truly silent or never-settling turn-scoped server requests while preserving intentional human/MCP waits, then coordinate any terminal-idle timeout budget or config surface with the existing timeout tracker.

## Reproduction Assessment

No live current-main 429 reproduction was established. Source inspection gives a medium-confidence reproduction path for the remaining liveness hole: a turn-scoped server request that never settles can keep the active-request counter positive and suppress completion and terminal idle watches.

## Solution Assessment

Unclear as a complete fix. Relaxing the terminal-idle gate is plausible for stuck requests, but the best fix needs maintainer policy that distinguishes legitimate approval, user-input, and MCP waits from a wedged app-server and avoids adding a competing timeout knob casually.
