## Summary

Keep open: current main still has the source-backed Codex app-server liveness gap where active turn-scoped server requests suppress completion and terminal idle watchdogs until those requests settle. The exact 429-triggered Telegram wedge was not live-reproduced on current main, and current `turn/start` is now bounded by `params.timeoutMs`, so this needs maintainer-reviewed Codex liveness policy rather than conservative cleanup closure.

## What This Changes

The issue reports a Codex app-server liveness wedge where a rate-limit-triggered stuck turn leaves active request accounting nonzero and suppresses terminal-idle recovery.

## Best Possible Solution

Define Codex app-server turn liveness so silent or never-settling turn-scoped server requests release the session safely, while preserving intentional approval/user-input/MCP waits and coordinating terminal-idle timeout budget policy with the existing terminal-timeout tracker.

## Reproduction Assessment

No live current-main 429 reproduction was run. Source inspection gives a medium-confidence reproduction path for the remaining liveness hole: a turn-scoped server request that never settles can keep the active-request counter positive and suppress completion/terminal idle watches.

## Solution Assessment

Unclear as a complete fix. Relaxing the terminal-idle gate is plausible for stuck requests, but the best fix needs a maintainer policy that distinguishes legitimate approval/user-input/MCP waits from a wedged app-server and avoids adding a competing timeout knob casually.
