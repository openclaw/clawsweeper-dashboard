## Summary

Keep open. Current main still resolves Opus 4.7 adaptive effort from model id alone and emits `output_config.effort` without a GitHub Copilot provider/model compatibility rule; the related open PR only targets low/minimal and has already been reviewed as insufficient for this issue's high/xhigh evidence.

## What This Changes

The issue reports that GitHub Copilot-backed Claude Opus 4.7 rejects non-medium Anthropic adaptive `output_config.effort` values, causing schema failures and auth-profile cooldowns.

## Best Possible Solution

Implement a provider-aware Copilot Opus 4.7 adaptive-effort policy, preferably table-driven or based on preserved live catalog capabilities, with regression tests for low, minimal, high, and xhigh while leaving direct Anthropic Opus 4.7 behavior unchanged.

## Reproduction Assessment

Yes. A current-main source-level reproduction is a reasoning-enabled `github-copilot/claude-opus-4.7` request with high or xhigh: the transport maps it to adaptive `output_config.effort` and the Copilot wrapper does not normalize it; the issue provides live 400 logs for that payload.

## Solution Assessment

Unclear. Omitting or forcing `medium` matches this issue's logs, but the related open PR makes a narrower live-capability claim, so maintainers should choose the provider policy before automation hard-codes the behavior.
