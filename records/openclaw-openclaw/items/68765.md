## Summary

Keep this PR open. The reported gateway history truncation is real on current main, but the proposed branch is not safe to merge because it would regress active-branch transcript handling, replay reset/restore-cleared checkpoint history, duplicate retained compaction tails, and bypass existing transcript path containment.

## What This Changes

The PR adds a chained transcript resolver, regression tests, and a production-derived fixture so `readSessionMessages` can read configured main-session compaction checkpoint snapshots plus the live transcript as one continuous history.

## Best Possible Solution

Keep the useful repro evidence, but redesign the implementation around a bounded chain reader that preserves `SessionManager` active-branch reads, trims each snapshot at checkpoint boundaries, filters reset/restore epochs deliberately, and resolves checkpoint files through the existing session path containment contract. Reset-archive visibility should remain coordinated with #60409 and #73883 rather than becoming implicit default `chat.history` replay.

## Reproduction Assessment

Yes. The original truncation is reproducible from current main because `readSessionMessages` reads only one resolved file, while checkpoints store predecessor transcript paths. The proposed-branch regressions are also high-confidence from the source paths: reset and restore preserve checkpoint metadata, and the PR walks those paths unconditionally.

## Solution Assessment

No. The chained-read direction is valid, but this branch is not the best solution because it bypasses current active-branch filtering, ignores checkpoint boundaries, replays reset/restore-cleared content, and trusts raw stored paths instead of the canonical resolver.

Full review comments:

- **[P1] Preserve active-branch transcript reads:** `src/gateway/session-utils.fs.ts:287-288`
  - body: Current main reads tree-format transcripts through `SessionManager.getBranch()` so abandoned rewrite branches stay out of `chat.history`. The PR's multi-file loop reads JSONL lines directly, which can reintroduce stale abandoned entries and fail the existing active-branch regression guard.
  - confidence: 0.93
- **[P1] Stop replaying stale reset and restore epochs:** `src/gateway/session-utils.fs.ts:212-224`
  - body: The chain walker prepends every stored `preCompaction.sessionFile`. Since reset and restore both preserve checkpoint metadata on the new entry, default `chat.history` can resurface cleared history or duplicate restored content.
  - confidence: 0.92
- **[P1] Trim snapshots at checkpoint boundaries:** `src/gateway/session-utils.fs.ts:221-223`
  - body: Compaction checkpoints record `firstKeptEntryId` because the post-compaction transcript keeps an unsummarized tail. Concatenating full pre-compaction snapshots with the live transcript duplicates retained tail messages across normal checkpoint chains.
  - confidence: 0.86
- **[P2] Constrain transcript paths before reading them:** `src/gateway/session-utils.fs.ts:197-199`
  - body: The new `pushFile` path accepts checkpoint and entry paths with `path.resolve()` plus existence checks only. That bypasses the existing session-file containment rules and can merge unrelated readable JSONL files if persisted metadata is stale or corrupted.
  - confidence: 0.9
- **[P2] Prefer the canonical matching store row:** `src/gateway/session-utils.fs.ts:140-144`
  - body: The store scan returns the first matching `sessionId`, so legacy duplicate rows can select a stale non-main entry before the canonical main row with checkpoints is considered. In that case the intended compaction-chain fix silently falls back to the old truncating path.
  - confidence: 0.82
