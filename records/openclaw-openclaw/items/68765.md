## Summary

Keep open: the underlying gateway history problem is plausible and current main still reads only the active transcript segment for chat history, but this PR is not mergeable because it is stale against current async readers and introduces correctness/privacy regressions in checkpoint replay.

## What This Changes

The PR adds a main-session checkpoint-chain transcript resolver plus fixtures/tests so `readSessionMessages` concatenates pre-compaction snapshots with the live transcript.

## Best Possible Solution

Rework the chain reader around current main by defining active reset/restore epoch semantics, trimming snapshots at `firstKeptEntryId`, using contained transcript resolution, preserving active-branch filtering, and applying the result to async/recent chat-history readers.

## Reproduction Assessment

Yes, source-reproducible: current main selects a single active transcript for `chat.history` while checkpoint metadata stores pre-compaction transcript references. I did not run tests because this review was read-only.

## Solution Assessment

No. The chained-read direction is plausible, but this patch is not the best current solution because it misses current async readers and does not preserve branch, boundary, containment, or reset/restore semantics.

Full review comments:

- **[P1] Wire chained reads into the async history path:** `src/gateway/session-utils.fs.ts:281`
  - body: Current main's `chat.history` calls `readRecentSessionMessagesAsync`, while this PR only changes sync `readSessionMessages`. Merging this as-is leaves the gateway chat-history path described in the PR body reading only the live transcript segment.
  - confidence: 0.93
- **[P1] Preserve active-branch transcript filtering:** `src/gateway/session-utils.fs.ts:287-288`
  - body: This loop parses every JSONL line from every selected file, bypassing the current active-branch selection/indexing logic. Tree-format transcripts with abandoned rewrite branches will reintroduce stale sibling content into history.
  - confidence: 0.91
- **[P1] Stop replaying stale reset and restore epochs:** `src/gateway/session-utils.fs.ts:212-223`
  - body: The chain walk prepends every stored checkpoint snapshot. Reset and checkpoint restore preserve checkpoint metadata, so default history can resurface pre-reset messages or double-read restored content after a restore.
  - confidence: 0.92
- **[P1] Trim snapshots at compaction boundaries:** `src/gateway/session-utils.fs.ts:221-223`
  - body: Checkpoints record `firstKeptEntryId` because the post-compaction transcript retains an unsummarized tail. Concatenating each full pre-compaction snapshot with the live transcript duplicates retained turns and produces incorrect sequence counts.
  - confidence: 0.86
- **[P2] Constrain checkpoint paths before reading them:** `src/gateway/session-utils.fs.ts:196-198`
  - body: `pushFile` accepts checkpoint and entry paths using only `path.resolve()` plus existence checks. Corrupt or stale store metadata can make history ingest unrelated readable JSONL files instead of staying within session transcript ownership rules.
  - confidence: 0.9
