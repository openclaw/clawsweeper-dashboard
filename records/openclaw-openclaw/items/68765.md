## Summary

Keep open: the current-main truncation gap is source-reproducible, but this PR is not mergeable as-is because it is stale against the async `chat.history` path and still has correctness/privacy blockers around active-branch selection, reset/restore epoch replay, compaction boundaries, and path containment.

## What This Changes

The PR adds a main-session compaction-checkpoint transcript resolver, regression fixtures, and tests so `readSessionMessages` can read predecessor checkpoint files plus the live transcript.

## Best Possible Solution

Define the checkpoint-history replay contract first, then implement an async-aware, contained reader that preserves active-branch selection, trims at compaction boundaries, and respects reset/restore epochs or exposes recovery through an explicit opt-in path.

## Reproduction Assessment

Yes, source-reproducible: current main's `chat.history` calls the async recent-message reader, and that reader resolves a single transcript path while checkpoint metadata stores predecessor transcript references. I did not run tests because this review is read-only.

## Solution Assessment

No. The chain-reader direction may be useful, but this PR is not the best fix because it misses the current async gateway path and does not preserve branch, boundary, containment, or reset/restore contracts.

  - body: `chat.history` on current main calls `readRecentSessionMessagesAsync`, but this branch only installs the chained resolver under sync `readSessionMessages`.   - confidence: 0.93

  - body: This raw JSONL loop parses every line from every segment and bypasses the active-branch selection used by current transcript readers. Tree transcripts with abandoned rewrite branches can reintroduce stale sibling content into history.
  - confidence: 0.91
- **[P1] Filter checkpoint replay to the active reset or restore epoch:** `src/gateway/session-utils.fs.ts:212-223`
  - body: The loop prepends every stored checkpoint snapshot. Because reset and restore keep checkpoint metadata on the session entry, default history can resurface pre-reset messages or double-read restored content.
  - confidence: 0.92
- **[P1] Trim checkpoint snapshots at compaction boundaries:** `src/gateway/session-utils.fs.ts:221-223`
  - body: Checkpoints record `firstKeptEntryId` because the successor transcript keeps an unsummarized tail. Concatenating each full snapshot with the live transcript duplicates retained turns and produces incorrect sequence counts.
  - confidence: 0.86
- **[P2] Constrain checkpoint transcript paths before reading:** `src/gateway/session-utils.fs.ts:196-198`
  - body: `pushFile` trusts stored paths after `path.resolve()` plus existence only. Corrupt or stale store metadata can make history ingest unrelated readable JSONL files instead of staying within session transcript ownership rules.
  - confidence: 0.9
