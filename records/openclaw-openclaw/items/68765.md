## Summary

Keep open. The underlying gateway history bug is real on current main, but this PR is not mergeable as the best fix because it drops current active-branch transcript semantics, replays reset/restore-stale checkpoint snapshots, can duplicate retained compaction tails, and reads checkpoint paths without the existing containment rules.

## What This Changes

The PR adds a chained main-session transcript resolver, tests, and a production-derived fixture so `readSessionMessages` can concatenate compaction checkpoint snapshots with the live transcript.

## Best Possible Solution

Keep the PR open as useful evidence, but rework it around current main: preserve `SessionManager` active-branch reads, stitch checkpoint snapshots only within the active reset/restore epoch, trim segments at `firstKeptEntryId`, route all checkpoint paths through contained transcript resolution, update the bounded `chat.history` reader path, and add the required changelog entry.

## Reproduction Assessment

Yes. The PR head contains explicit reset/restore fixtures that demonstrate the replay-everything failure mode, and current main has an active-branch regression test that the PR head would bypass by parsing raw JSONL lines.

## Solution Assessment

No. The chained-read direction is plausible, but this patch is not the best solution because it bypasses current branch filtering, trusts raw checkpoint paths, and does not model compaction/reset/restore boundaries.

Full review comments:

- **[P1] Preserve active-branch transcript reads:** `src/gateway/session-utils.fs.ts:287-288`
  - body: Current main reads tree-format transcripts through `SessionManager.getBranch()` so abandoned rewrite branches stay out of history. This replacement loops over every JSONL line in each selected file, which reintroduces stale sibling branches into `readSessionMessages`.
  - confidence: 0.93
- **[P1] Stop replaying stale reset and restore epochs:** `src/gateway/session-utils.fs.ts:212-224`
  - body: This loop prepends every stored checkpoint snapshot. Reset and restore both preserve checkpoint metadata, so default history can resurface pre-reset messages or double-read restored content after a checkpoint restore.
  - confidence: 0.92
- **[P1] Trim snapshots at compaction boundaries:** `src/gateway/session-utils.fs.ts:221-223`
  - body: Each checkpoint records `firstKeptEntryId` because the post-compaction transcript keeps an unsummarized tail. Concatenating full pre-compaction snapshots with the live transcript duplicates retained tail messages and assigns incorrect sequence counts.
  - confidence: 0.86
- **[P2] Constrain checkpoint paths before reading them:** `src/gateway/session-utils.fs.ts:197`
  - body: `pushFile` accepts checkpoint and entry paths using only `path.resolve()` plus `existsSync()`. That bypasses the existing transcript path containment rules, so corrupted or stale store metadata can make history ingest unrelated readable JSONL files.
  - confidence: 0.9
- **[P2] Prefer the canonical matching store row:** `src/gateway/session-utils.fs.ts:140-144`
  - body: The store scan returns the first row with a matching `sessionId` and optional `sessionFile`. With legacy duplicate aliases, an earlier stale row can prevent the canonical main entry with the checkpoint chain from being used, leaving the truncation bug in place.
  - confidence: 0.82
