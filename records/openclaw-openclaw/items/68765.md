## Summary

Keep open. The reported history gap is source-reproducible, but the latest PR head still does not fix the `chat.history` gateway path and introduces default transcript replay behavior that conflicts with active-branch, reset/restore, checkpoint-boundary, and containment contracts.

## What This Changes

The PR adds a main-session compaction-checkpoint chain resolver to `readSessionMessages` plus gateway session utility tests and a redacted production-shaped fixture.

## Best Possible Solution

Define the checkpoint replay contract first, then implement a contained async-aware reader or explicit opt-in recovery path that preserves active-branch filtering, trims at compaction boundaries, and respects reset/restore epochs.

## Reproduction Assessment

Yes, source-reproducible: current main's `chat.history` uses a single-file async recent reader while session checkpoint metadata stores predecessor transcript references. I did not run a live compacted gateway session because this was a read-only review.

## Solution Assessment

No. The chain-reader direction may be useful, but this patch is not the best solution because it misses the async gateway path and does not preserve branch, boundary, containment, or reset/restore contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wire the chain reader into the async history path:** `src/gateway/session-utils.fs.ts:281`
  - body: `chat.history` calls `readRecentSessionMessagesAsync`, but this PR only routes sync `readSessionMessages` through the chained resolver. The reported gateway RPC will still read the live transcript segment only.
  - confidence: 0.93
- **[P1] Preserve active-branch filtering for each segment:** `src/gateway/session-utils.fs.ts:287-288`
  - body: This loop parses every JSONL line from each segment, bypassing the active-branch selection used by current readers. Tree transcripts with abandoned rewrite branches can reintroduce stale sibling content into history.
  - confidence: 0.91
- **[P1] Filter replay to the active reset or restore epoch:** `src/gateway/session-utils.fs.ts:212-223`
  - body: The loop prepends every stored checkpoint snapshot. Since reset copies `compactionCheckpoints` and restore preserves them, default history can resurface pre-reset messages or double-read restored content.
  - confidence: 0.92
- **[P1] Trim checkpoint snapshots at compaction boundaries:** `src/gateway/session-utils.fs.ts:221-223`
  - body: Checkpoints record `firstKeptEntryId` because the successor transcript keeps an unsummarized tail. Concatenating each full snapshot with the live transcript duplicates retained turns and produces wrong sequence counts.
  - confidence: 0.86
- **[P2] Constrain checkpoint transcript paths before reading:** `src/gateway/session-utils.fs.ts:197-198`
  - body: `pushFile` accepts stored paths after `path.resolve()` plus existence only. Corrupt or stale store metadata can make history ingest unrelated readable JSONL files instead of staying within session transcript ownership rules.
  - confidence: 0.9
