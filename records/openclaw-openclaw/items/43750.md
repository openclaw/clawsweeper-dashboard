## Label Justifications

- `P2`: The behavior can disrupt active sessions in multi-agent channels, but it does not make the core runtime broadly unusable.
- `impact:session-state`: One shared-channel command can independently rotate or reset multiple agents' session state.
- `impact:security`: The current authorization path does not prove that the user explicitly selected the agent whose context is reset.

## Summary

The issue remains necessary: current main authorizes and resets each already-routed agent session independently, with no explicit agent-target invariant, so shared-channel agents can each act on the same command. Explicit targeting fits project priorities, but the syntax and behavior of untargeted commands require a maintainer product and authorization decision.

## What This Changes

/new and /reset should support explicit agent targeting and avoid ambiguous multi-agent session resets in shared channels.

## Best Possible Solution

Resolve an explicit portable agent target before command dispatch, authorize that target, and let the existing reset handler mutate only the resulting session; in an ambiguous shared channel, an untargeted reset should fail with a selection prompt unless maintainers deliberately approve another cross-channel policy.

## Reproduction Assessment

Yes. Current-main source shows that each receiving agent authorizes the sender and resets its already-routed session without an explicit agent-target check, which provides a high-confidence source reproduction of the shared-channel ambiguity.

## Solution Assessment

No, not in full. Explicit targeting is the narrow maintainable direction, but choosing the highest-context agent, active participants, all agents, or a selection prompt for untargeted commands requires an explicit product and authorization decision.
