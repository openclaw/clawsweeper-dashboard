## Summary

Keep open: current main still lacks explicit `@agent` or native slash-option targeting for `/new` and `/reset`, and the production Telegram workaround comment confirms this is still a real multi-agent operations problem. The remaining work needs maintainer product and authorization decisions before it is a safe repair-lane candidate.

## What This Changes

Request `/new` and `/reset` support targeting a specific agent in multi-agent channels, with safer untargeted behavior and channel-gating parity.

## Best Possible Solution

Add an authorization-aware target resolver for session commands, with fail-closed ambiguity handling, native command options or documented text syntax, and focused tests/docs across Discord, Telegram, Slack, and text command paths.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the generic reset parser while `CommandTargetSessionKey` remains the already-routed or bound session. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks like the maintainable direction, but maintainers still need to decide syntax, untargeted UX, ambiguity behavior, and cross-agent reset permissions.
