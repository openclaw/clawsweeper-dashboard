## Label Justifications

- `P2`: The issue affects real multi-agent session management, but the blast radius is bounded to shared-channel command routing and reset behavior.
- `impact:session-state`: The reported behavior can reset or rotate the wrong agent session, or multiple sessions, in shared multi-agent channels.
- `impact:security`: Adding targeted shared-channel resets requires explicit authorization rules so users cannot reset agents or sessions they should not control.

## Summary

Keep open: current main still handles `/new` and `/reset` as commands for the already-routed session, while the requested cross-agent target selection needs maintainer decisions about UX, ambiguity, and authorization.

## What This Changes

Add support for targeting `/new` and `/reset` at a specific agent in shared multi-agent channels.

## Best Possible Solution

Define one authorization-aware target resolver for text and native session commands, fail closed on ambiguous targets, document the chosen syntax/default behavior, and cover Discord, Telegram, Slack, and shared reset tests.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the current reset parser and native-command helper without resolving a requested agent. I did not run a live multi-agent transport setup.

## Solution Assessment

Unclear: explicit target syntax is likely the right direction, but maintainers still need to choose the native option shape, untargeted behavior, ambiguity handling, and cross-agent reset authorization policy.
