## Label Justifications

- `P2`: The issue affects real multi-agent session management but has bounded channel/session blast radius and needs product/security review before implementation.
- `impact:session-state`: The reported behavior can reset or rotate the wrong agent session, or multiple sessions, in shared multi-agent channels.
- `impact:security`: Targeted shared-channel resets require explicit authorization so users cannot reset agents or sessions outside their control.

## Summary

Keep open: current main still treats `/new` and `/reset` as current routed-session commands with no requested-agent resolver, and the production Telegram workaround comment shows this is real multi-agent session-state pain. The fix still needs maintainer decisions on command UX, ambiguity handling, and cross-agent reset authorization.

## What This Changes

Request targeted `/new` and `/reset` handling so shared multi-agent channels can reset one selected agent instead of the wrong or multiple sessions.

## Best Possible Solution

Add one authorization-aware session-command target resolver shared by text and native command surfaces, fail closed on ambiguous or unauthorized targets, document the chosen syntax/default behavior, and prove Discord, Telegram, Slack, and shared reset coverage.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through current reset parsing and native command target helpers without resolving a requested agent. I did not run a live multi-agent transport setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization is likely the right direction, but maintainers still need to choose syntax, native command option shape, untargeted behavior, ambiguity handling, and cross-agent reset permissions.
