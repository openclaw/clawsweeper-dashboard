## Label Justifications

- `P2`: This is a valid multi-agent session-control problem with meaningful impact, but the affected setups are bounded multi-agent channel deployments.
- `impact:session-state`: Ambiguous `/new` or `/reset` commands can rotate or clear the wrong agent session in a shared channel.
- `impact:security`: Targeting rules determine which authorized channel users may reset another agent's state, so the fix crosses an authorization boundary.

## Summary

Keep open: current main and v2026.6.6 still route `/new` and `/reset` to the resolved/bound command session without a user-supplied agent target, and the remaining work needs a product and security decision rather than cleanup closure.

## What This Changes

The issue asks for `/new` and `/reset` to target a specific agent in multi-agent channels and avoid ambiguous resets.

## Best Possible Solution

Define one cross-channel command-targeting policy with explicit target syntax where supported, fail-closed or prompt-on-ambiguity behavior, and shared authorization tests for text and native command paths.

## Reproduction Assessment

Yes, source-level: current main builds `/new` and `/reset` as generic commands and reset handling uses route-derived `sessionKey` or `CommandTargetSessionKey` without parsing an `@agent` target. No live multi-agent channel was exercised in this read-only review.

## Solution Assessment

Unclear: explicit agent targeting is a plausible direction, but the best fix needs maintainer-approved syntax, ambiguity behavior, and cross-agent authorization rules before implementation.
