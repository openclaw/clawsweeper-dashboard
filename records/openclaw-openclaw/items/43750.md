## Summary

Keep open: current main still has no explicit target-agent resolution for `/new` or `/reset`, and the production Telegram workaround comment supports real multi-agent session-state pain. The remaining work needs maintainer product and authorization decisions before it is safe repair work.

## What This Changes

Request explicit `@agent` or native option targeting plus safer untargeted behavior for `/new` and `/reset` in multi-agent channels.

## Best Possible Solution

Add an authorization-aware session-command target resolver that supports explicit agent targeting for text and native command surfaces, fails closed on ambiguity, updates docs, and adds focused Discord, Telegram, Slack, and shared reset coverage.

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the current reset parser and native command helpers without resolving a requested agent. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks maintainable, but maintainers still need to choose syntax, untargeted UX, ambiguity behavior, and cross-agent reset permissions.
