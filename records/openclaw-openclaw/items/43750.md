## Summary

Keep open: current main still lacks explicit target-agent selection for `/new` and `/reset`, and the production Telegram workaround comment confirms this is real multi-agent session-state pain rather than a stale report. The remaining work needs product and authorization decisions before it is safe for an automated fix lane.

## What This Changes

Request explicit `@agent` or native agent targeting for `/new` and `/reset` in shared multi-agent channels, plus consistent channel-gating behavior.

## Best Possible Solution

Add an authorization-aware session-command target resolver with explicit agent targeting for text and native command surfaces, fail-closed ambiguity handling, docs, and focused Discord, Telegram, Slack, and shared reset coverage.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the reset parser and native command helpers without resolving a requested agent. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization is directionally maintainable, but maintainers still need to choose syntax, untargeted behavior, ambiguity handling, and cross-agent reset permissions.
