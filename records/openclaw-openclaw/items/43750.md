## Summary

Keep open: current main still has no explicit `@agent` or slash-option target resolver for `/new` or `/reset`; the recent Telegram production workaround comment reinforces that this remains a real multi-agent pain point, but the syntax, default no-target behavior, ambiguity handling, and reset authorization model need maintainer decisions before a safe fix can be scoped.

## What This Changes

Request support for targeted `/new` and `/reset` behavior in multi-agent channels, including explicit agent selection and safer untargeted defaults.

## Best Possible Solution

Add an authorization-aware target resolver for session commands, with fail-closed ambiguity handling and focused docs/tests for targeted and untargeted `/new` and `/reset` across native and text command paths.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through generic reset parsing while `CommandTargetSessionKey` remains the already-routed or bound session. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks like the maintainable direction, but maintainers still need to decide syntax, untargeted UX, ambiguity handling, and cross-agent reset authorization.
