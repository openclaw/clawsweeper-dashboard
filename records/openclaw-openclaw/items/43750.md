## Label Justifications

- `P2`: The issue affects real multi-agent session management with meaningful but bounded channel/session blast radius.
- `impact:session-state`: The reported behavior can reset or rotate the wrong agent session, or multiple sessions, in shared multi-agent channels.
- `impact:security`: Targeted resets in shared channels require explicit authorization rules so a user cannot reset agents or sessions they should not control.

## Summary

Keep open: current main still treats `/new` and `/reset` as current routed-session commands, with no target-agent resolver for text or native command surfaces; the production Telegram workaround comment reinforces that this is real multi-agent session-state pain, but the fix needs maintainer UX and authorization decisions first.

## What This Changes

The issue requests `@agent` targeting and safer default behavior for `/new` and `/reset` in shared multi-agent channels.

## Best Possible Solution

Add one authorization-aware session-command target resolver for text and native command surfaces, fail closed on ambiguous targets, document the chosen syntax/default behavior, and prove Discord, Telegram, Slack, and shared reset coverage.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through current reset parsing and native command target helpers without resolving a requested agent. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization is likely the right direction, but maintainers still need to choose syntax, native command option shape, untargeted behavior, ambiguity handling, and cross-agent reset permissions.
