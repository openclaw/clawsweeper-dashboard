## Summary

Keep open: current main still lacks explicit `@agent` or native slash-option targeting for `/new` and `/reset`, while the discussion includes a real production Telegram workaround that supports the multi-agent operations problem. This needs maintainer product and authorization decisions before it is safe repair-lane work.

## What This Changes

Request `/new` and `/reset` to target a named agent in multi-agent channels and avoid ambiguous resets.

## Best Possible Solution

Add an authorization-aware session-command target resolver with fail-closed ambiguity handling, native command options or documented text syntax, and focused Discord/Telegram/Slack/text-command coverage.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the generic reset parser while `CommandTargetSessionKey` remains the already routed or bound session. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks maintainable, but maintainers still need to decide syntax, untargeted UX, ambiguity behavior, and cross-agent reset permissions.
