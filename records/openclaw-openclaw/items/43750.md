## Summary

Keep open: current main still routes `/new` and `/reset` to the already-routed or bound session, with no resolver for an explicit `@agent` selector; the production Telegram workaround comment confirms this is still a real multi-agent pain point, but the syntax, no-target default, ambiguity handling, and cross-agent reset authorization need maintainer decisions before a repair PR is safe.

## What This Changes

The issue asks for `/new` and `/reset` to target a named agent in shared multi-agent channels, with safer untargeted behavior and channel-gating parity.

## Best Possible Solution

Add an authorization-aware target resolver for session commands, with fail-closed ambiguity handling and focused docs/tests for targeted and untargeted `/new` and `/reset` across native and text command paths.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` remain generic command input or reset tail while `CommandTargetSessionKey` stays the already-routed or bound session. I did not run a live multi-agent channel setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks like the maintainable direction, but maintainers still need to decide syntax, untargeted UX, ambiguity handling, and cross-agent reset authorization.
