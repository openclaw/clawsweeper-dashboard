## Label Justifications

- `P2`: This is a normal-priority multi-agent session-control problem with meaningful shared-channel impact but no confirmed outage or emergency data loss.
- `impact:session-state`: The reported behavior can start or reset the wrong agent session in a shared channel.
- `impact:security`: Ambiguous session-control commands in shared channels can affect an untargeted agent’s context, making this an authorization and safety concern.

## Summary

Keep open: current `main` has native command target-session plumbing, but it still does not provide user-facing `/new @agent` or `/reset @agent` selection, and the safe untargeted behavior remains a maintainer product/security decision.

## What This Changes

/new and /reset should support explicit agent targeting in multi-agent channels and avoid ambiguous session resets.

## Best Possible Solution

Define one channel-agnostic session-control targeting contract, then map Discord, Telegram, and Slack native command inputs onto it with explicit ambiguity handling for untargeted resets.

## Reproduction Assessment

Yes at source level: current command definitions and native command routing do not parse an agent target before choosing the reset session. I did not run a live multi-agent Discord or Telegram reproduction in this read-only review.

## Solution Assessment

No: the current `CommandTargetSessionKey` plumbing is not a complete solution because it only carries a session already selected by routing or binding. The best fix needs a maintainer-selected target syntax and no-target ambiguity policy before implementation.
