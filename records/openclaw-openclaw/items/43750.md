## Summary

Keep open: current main still lacks explicit `@agent` or native slash-option targeting for `/new` and `/reset`, and the issue comments include a real production Telegram workaround showing this is a practical multi-agent operations problem. The remaining work needs maintainer product and authorization decisions before it is safe repair-lane work.

## What This Changes

Request `/new` and `/reset` to target a selected agent in shared multi-agent channels and avoid ambiguous reset fan-out.

## Best Possible Solution

Add an authorization-aware session-command target resolver with fail-closed ambiguity handling, native command options or documented text syntax, and focused Discord, Telegram, Slack, and text-command coverage.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the generic reset parser while native command targeting remains the already routed or bound session. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks maintainable, but maintainers still need to decide syntax, untargeted UX, ambiguity behavior, and cross-agent reset permissions.
