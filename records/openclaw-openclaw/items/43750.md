- `impact:session-state`: The issue is about `/new` and `/reset` affecting the wrong agent session or multiple active sessions in shared channels.

## Summary

Keep open: current main and v2026.5.20 still route `/new` and `/reset` to the already selected command session, with no explicit `@agent` or native command target. The production Telegram workaround in the discussion supports the same multi-agent session-state problem, but the fix needs maintainer product and authorization decisions first.

## What This Changes

Request `/new` and `/reset` to support explicit agent targeting and safer untargeted behavior in multi-agent channels.

## Best Possible Solution

Add an authorization-aware session-command target resolver for text and native command surfaces, with explicit agent targeting, fail-closed ambiguity handling, docs, and focused Discord, Telegram, Slack, and shared reset coverage.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the current reset parser and native command helpers without resolving a requested agent. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks maintainable, but maintainers still need to choose syntax, untargeted UX, ambiguity behavior, and cross-agent reset permissions.
