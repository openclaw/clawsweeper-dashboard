## Summary

Keep open: current main still derives `/new` and `/reset` targets from channel routing or bindings, not from a user-facing agent selector. The KinthAI comment adds useful production evidence from shared Telegram channels, and the remaining work needs maintainer product and security direction for syntax, no-target behavior, and cross-agent authorization.

## What This Changes

The issue requests targeted `/new` and `/reset` command handling so multi-agent channels do not reset the wrong agent session.

## Best Possible Solution

Define one explicit, authorized agent-targeting contract for session reset/new commands across native and text channel surfaces, preserving route-bound defaults unless maintainers choose a stricter interactive or explicit-only default.

## Reproduction Assessment

Yes at source level: current command-routing code derives the target session from route/binding state, and issue comments report real multi-agent channel ambiguity. This read-only pass did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

No current implementation is the final solution. The existing `CommandTargetSessionKey` seam is useful, but user-facing target syntax, no-target behavior, and authorization need maintainer direction before a safe patch.
