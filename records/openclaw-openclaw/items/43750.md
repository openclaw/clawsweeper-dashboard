## Summary

Keep open: current main and the latest release still route `/new` and `/reset` to the already selected native/text session, with no explicit `@agent` or native command option targeting. The user reports and the Telegram workaround comment show a real multi-agent operations problem, but the remaining fix needs product and authorization decisions before it is safe automation work.

## What This Changes

Request `/new` and `/reset` support explicit agent targeting, safer untargeted behavior, and channel-gating parity in multi-agent channels.

## Best Possible Solution

Add an authorization-aware session-command target resolver with fail-closed ambiguity handling, native command options or documented text syntax, and focused Discord, Telegram, Slack, and text-command coverage.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the current reset parser and native command helpers without resolving a requested agent. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks maintainable, but maintainers still need to choose syntax, untargeted UX, ambiguity behavior, and cross-agent reset permissions.
