## Summary

Keep open: current main still lacks native `/new` or `/reset` agent-target parsing. The code routes native commands to the already resolved channel/topic/binding session, while extra command text such as `@forge` remains generic command input; the later Telegram production comment strengthens the report rather than making it stale.

## What This Changes

The issue requests explicit agent targeting for `/new` and `/reset`, a safer no-target default in multi-agent channels, and `/new` parity with existing channel gating behavior.

## Best Possible Solution

Add an explicit, authorization-aware native command target resolver for `/new` and `/reset` that maps configured agent ids/names/mentions/bindings to exactly one allowed session, rejects ambiguous or unauthorized targets, and documents the no-target policy across native command surfaces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `/new @forge` or `/reset @anvil` is parsed as generic command input, while `CommandTargetSessionKey` remains the already routed or bound channel/topic session.

## Solution Assessment

Unclear until maintainers choose the no-target policy. Explicit target parsing plus fail-closed authorization checks looks like the maintainable direction, but automation should not choose between prompt/reject/highest-context/all-agent defaults.
