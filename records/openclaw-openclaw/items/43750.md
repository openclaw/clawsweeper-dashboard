## Summary

Keep open: current main and v2026.5.20 still route `/new` and `/reset` to the already selected command session, with no explicit `@agent` or native command option target. The production Telegram workaround comment reinforces that this is real multi-agent session-state pain, but the fix needs product and authorization decisions before it is safe repair work.

## What This Changes

The issue requests agent-targeted `/new` and `/reset` behavior plus safer untargeted/channel-gating semantics in multi-agent channels.

## Best Possible Solution

Add an authorization-aware session-command target resolver with explicit agent targeting for text and native command surfaces, fail-closed ambiguity handling, docs, and focused Discord, Telegram, Slack, and shared reset coverage.

## Reproduction Assessment

Yes, at source level: `/new @forge` and `/reset @anvil` still flow through the current reset parser and native command helpers without resolving a requested agent. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

Unclear: explicit target parsing with fail-closed authorization looks maintainable, but maintainers still need to choose syntax, untargeted UX, ambiguity behavior, and cross-agent reset permissions.
