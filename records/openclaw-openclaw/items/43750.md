## Label Justifications

- `P2`: This is a meaningful multi-agent command/session safety issue with bounded blast radius and no evidence of an emergency runtime outage.
- `impact:session-state`: Ambiguous `/new` and `/reset` handling can start or reset the wrong agent's session/context in shared channels.
- `impact:security`: The requested fix involves authorization and explicit targeting for commands that can affect another agent's session state in a shared channel.

## Summary

Keep open: current main still routes `/new` and `/reset` to the route-derived or bound command session and does not expose a user-facing `@agent` or native agent selector. The remaining work needs maintainer product and security direction for target syntax, no-target behavior, and cross-agent reset authorization.

## What This Changes

Add explicit agent targeting and safer no-target behavior for `/new` and `/reset` in multi-agent channels.

## Best Possible Solution

Define one explicit, authorized agent-targeting contract for session reset/new commands across native and text surfaces, preserving current route-bound defaults unless maintainers choose an interactive or explicit-only default.

## Reproduction Assessment

Yes at source level: current command definitions and docs expose `/new` and `/reset` arguments but no agent selector, while channel adapters derive `CommandTargetSessionKey` from routing or bindings rather than command args. I did not run a live multi-agent Discord or Telegram setup.

## Solution Assessment

No current implementation is the final solution. The existing `CommandTargetSessionKey` seam is useful, but the missing user-facing selector plus authorization and default-behavior rules need maintainer direction before a safe patch.
