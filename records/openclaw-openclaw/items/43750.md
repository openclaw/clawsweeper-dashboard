## Summary

Keep open: current main and v2026.6.8 still route `/new` and `/reset` through the already resolved command/session target, with no explicit user-selected agent target. The report is valid, but the durable fix needs maintainer product and security direction for target syntax, ambiguous no-target behavior, and cross-agent authorization.

## What This Changes

The issue requests explicit per-agent targeting for `/new` and `/reset` in shared multi-agent channels, plus safer no-target behavior and channel-gating parity.

## Best Possible Solution

Define one shared text/native command-targeting contract for destructive session commands, then implement explicit target selection, ambiguous-target handling, authorization checks, and Discord/Telegram regression coverage together.

## Reproduction Assessment

Yes, at source level: current native command paths carry only route-derived session targets, not a user-selected `@agent` or `agent:` target. I did not run a live multi-agent Discord or Telegram replay in this read-only review.

## Solution Assessment

Unclear: explicit targeting is likely the right direction, but maintainers still need to choose the permanent syntax, ambiguous no-target behavior, and cross-agent authorization rule.
