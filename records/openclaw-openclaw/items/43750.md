## Summary

Keep open: current main still routes native `/new` and `/reset` to the already effective channel/topic/binding target and treats extra command text as normal command input, not as an agent selector. The later Telegram production comment strengthens the report, and related #58110 is broader multi-agent UI work rather than a replacement for this native command targeting gap.

## What This Changes

The issue asks for `/new` and `/reset` in multi-agent channels to support explicit agent targeting and avoid resetting unintended agents.

## Best Possible Solution

Add an explicit, authorization-aware native command target resolver for `/new` and `/reset` that can map configured agent ids/names/mentions/bindings to exactly one allowed session, rejects ambiguous or unauthorized targets, and applies a documented no-target policy consistently across Discord and Telegram.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `/new @forge` or `/reset @anvil` would be parsed as generic command input, while `CommandTargetSessionKey` remains the routed or bound channel/topic session.

## Solution Assessment

Unclear until maintainers choose the no-target policy. Explicit target parsing plus fail-closed authorization checks is the maintainable direction, but an automated repair should not pick between prompt/select/highest-context/all-agent defaults without product review.

## Risks / Open Questions

- The no-target default still needs maintainer product direction: fail closed, prompt, reset active participants, highest-context session, or keep current routed-session behavior have different safety and UX outcomes.
- Any target argument must preserve channel, guild, topic, binding, and sender authorization so it cannot reset an agent/session the sender could not otherwise control.
- Discord and Telegram likely need different syntax surfaces (`agent:` option vs mention/prefix text), so the shared core seam should be defined before implementation.
