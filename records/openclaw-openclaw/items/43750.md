- `P2`: This is a normal-priority multi-agent session-control problem with real shared-channel impact but no confirmed outage, data loss emergency, or crash-loop.
- `impact:session-state`: The reported behavior can start or reset the wrong agent session in a shared channel, directly affecting agent context state.

## Summary

Keep open: current main and the latest release still route `/new` and `/reset` through the already selected route, binding, or current session, not through a user-provided `@agent` target. The request is valid, but the safe syntax, no-target ambiguity behavior, and authorization boundary need maintainer product/security judgment before a fix PR is safe.

## What This Changes

Request `/new` and `/reset` to target a named agent in multi-agent channels and apply safe default/channel-gating behavior when no target is specified.

## Best Possible Solution

Define one channel-agnostic session-control targeting contract, then map Discord, Telegram, and Slack native/text command inputs onto it with explicit authorization and ambiguity handling.

## Reproduction Assessment

Yes, at source level for the missing explicit target path: current command definitions and native command routing do not parse an agent target before choosing the reset session. No live multi-agent Discord or Telegram reproduction was run in this read-only review.

## Solution Assessment

No, the requested targeting is not implemented yet and the proposed smart default is only one possible policy. The maintainable fix needs a maintainer-selected target syntax and fail-safe no-target behavior before implementation.
