## Label Justifications

- `P2`: This is a valid multi-agent session-control problem with meaningful impact, but it is limited to multi-agent channel deployments and needs design before repair.
- `impact:session-state`: Ambiguous `/new` or `/reset` handling can rotate or clear the wrong agent's session in a shared channel.
- `impact:security`: The requested targeting rule controls which channel users may reset another agent's state across an authorization boundary.

## Summary

Keep open: current main and v2026.6.6 still route `/new` and `/reset` to the already resolved command session and do not provide an `@agent` or native agent-target selector; the remaining work needs product and security direction before implementation.

## What This Changes

Request explicit `/new` and `/reset` agent targeting plus safe no-target behavior for multi-agent channel deployments.

## Best Possible Solution

Define a shared text/native command-targeting contract that fails closed or prompts when ambiguous, then implement channel-specific target UI/options with authorization and session-state regression coverage.

## Reproduction Assessment

Yes, source-level: current main builds `/new` and `/reset` as current-session commands and reset handling uses the route-derived `sessionKey` or `CommandTargetSessionKey` without parsing an `@agent` target. I did not run a live multi-agent Discord or Telegram channel repro in this read-only review.

## Solution Assessment

Unclear: explicit agent targeting is a plausible direction, but the best fix needs maintainer-approved syntax, ambiguity behavior, and cross-agent authorization rules before implementation.
