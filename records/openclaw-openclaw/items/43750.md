## Label Justifications

- `P2`: This is a valid multi-agent session-control problem with meaningful impact but a bounded deployment-specific blast radius.
- `impact:session-state`: Ambiguous `/new` or `/reset` commands can rotate or clear the wrong agent session in a shared channel.
- `impact:security`: Targeting rules determine which authorized channel users may reset another agent's state, so the fix crosses an authorization boundary.

## Summary

Keep open: current main and the latest release still bind `/new` and `/reset` to the already resolved route, bound session, or command session, with no user-provided `@agent` target; the remaining decision is product/security policy, not stale cleanup.

## What This Changes

Request to let `/new` and `/reset` target a specific agent in multi-agent channels and avoid ambiguous untargeted session resets.

## Best Possible Solution

Define one cross-channel command targeting policy: explicit target syntax where supported, fail-closed or prompt on ambiguous no-target resets, and shared authorization tests for text and native command paths.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` route `/new` and `/reset` through the current conversation route or bound session, with no parsed `@agent` target. No live multi-agent channel was exercised in this read-only pass.

## Solution Assessment

Unclear: `@agent` targeting is a plausible direction, but the best fix needs a maintainer-approved syntax, no-target ambiguity behavior, and authorization rule before implementation.
