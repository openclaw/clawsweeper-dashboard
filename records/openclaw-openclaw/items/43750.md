## Label Justifications

- `P2`: This is a normal-priority multi-agent shared-channel session-control gap with bounded but meaningful production impact.
- `impact:session-state`: Ambiguous `/new` or `/reset` handling can rotate or clear the wrong agent's session in a shared channel.

## Summary

Keep open: current main and latest stable v2026.6.10 still route `/new` and `/reset` through route-derived session targets, with no explicit user-selected agent target. The report remains valid, but the durable fix needs maintainer product/security direction for target syntax, ambiguous no-target behavior, and cross-agent authorization.

## What This Changes

Add explicit agent targeting for `/new` and `/reset` in multi-agent channels so destructive session commands do not affect the wrong agent.

## Best Possible Solution

Define one shared text/native command-targeting contract for destructive session commands, then implement explicit target selection, ambiguous-target handling, authorization checks, and cross-channel regression coverage together.

## Reproduction Assessment

Yes, at source level: current native command paths carry only route-derived session targets, and the reset handler treats trailing text as reset tail rather than an agent target. I did not run a live multi-agent Discord or Telegram replay in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract; explicit targeting is the right direction, but a one-channel patch would be too narrow without shared target and authorization semantics.
