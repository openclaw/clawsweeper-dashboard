## Summary

Keep open: the patch looks like a narrow source-backed fix for the preserved subagent fallback-notice gap, but it is still draft, expands user-visible fallback notice delivery while the visibility-policy counterpoint remains open, and the supplied proof does not demonstrate the changed preserved-subagent path in a real run.

## What This Changes

The PR changes `runReplyAgent` to emit model fallback notices for source-preserving subagent/inter-session fallback runs while keeping sticky fallback-state writes and cleared notices limited to non-preserving runs, with updated e2e coverage.

## Best Possible Solution

Land the notice fix only after maintainer sign-off on the fallback-notice visibility policy and real behavior proof for a preserved subagent/async run, while keeping #94919 open for the transport-policy work.

## Reproduction Assessment

Yes, source-reproducible: current main gates fallback notice generation on `!preserveUserFacingSessionState`, and the current e2e test asserts no `Model Fallback:` payload for `subagent_announce`. I did not run a live Z.AI/provider failure path in this read-only review.

## Solution Assessment

Mostly yes for the notice subproblem: the code change is a narrow owner-path fix that decouples notice emission from sticky state persistence. It still needs maintainer visibility-policy acceptance because #92473 asks for the opposite external-channel behavior, and it does not solve the transport knob requested in #94919.
