## What This Changes

The PR changes `runReplyAgent` so source-preserving fallback runs can emit the current run's fallback notice without persisting or clearing sticky user-facing fallback state, and updates the focused e2e coverage.

## Best Possible Solution

Land the notice slice only after redacted real preserved-run proof and maintainer acceptance of the fallback-notice visibility rule, while keeping the broader transport work tracked separately.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 gate fallback notice generation on `!preserveUserFacingSessionState`, and the current e2e coverage asserts no `Model Fallback:` payload for `subagent_announce`. I did not run a live Z.AI/provider failure path in this read-only review.

## Solution Assessment

Mostly yes for the notice subproblem: the diff is in the owner path and decouples notice emission from sticky state persistence. It still needs maintainer visibility-policy acceptance and real behavior proof, and it does not solve the separate provider transport request.
