## Summary

Keep open for manual maintainer review: current main still has the preserved-subagent fallback-notice gap, and the patch looks like a narrow fix for that slice, but it still needs behavior-specific real proof and a maintainer decision on fallback-notice visibility before merge.

## What This Changes

The branch changes `runReplyAgent` so source-preserving fallback runs can emit the current run's fallback notice without persisting or clearing sticky user-facing fallback state, and updates the e2e coverage for internal subagent announces.

## Best Possible Solution

Land the notice fix only after maintainer sign-off on fallback-notice visibility and redacted real behavior proof for a preserved subagent or async fallback run, while keeping the broader transport work tracked separately.

## Reproduction Assessment

Yes, source-reproducible: current main gates fallback notice generation on `!preserveUserFacingSessionState`, and the current e2e test asserts no `Model Fallback:` payload for `subagent_announce`. I did not run a live Z.AI/provider failure path in this read-only review.

## Solution Assessment

Mostly yes for the notice subproblem: the code change is a narrow owner-path fix that decouples notice emission from sticky state persistence. It still needs maintainer visibility-policy acceptance because the open external-channel issue asks for the opposite behavior, and it does not solve the transport knob requested by the linked report.
