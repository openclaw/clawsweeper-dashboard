## Summary

Keep open. Current main still leaves Feishu tenant-token/API HTTP calls on the Lark SDK Axios-backed default HTTP instance with only timeout/User-Agent wrapping, while the related proxy PR remains open and needs rework/proof plus an explicit Feishu egress-policy decision.

## What This Changes

The issue requests that Feishu tenant-token/API HTTP calls work when system HTTP proxy variables are configured, without surfacing the Lark SDK destructuring error.

## Best Possible Solution

Implement a Feishu-owned Lark HTTP adapter that preserves timeout/User-Agent behavior, prevents Axios proxy misrouting or double proxy handling, honors the chosen operator egress policy, and adds focused proxy tests plus troubleshooting docs.

## Reproduction Assessment

Yes at source level: Feishu probe and SDK tenant/API calls still flow through the Lark Axios-backed HTTP wrapper while proxy env can be active. I did not run a live Feishu credential/proxy setup in this read-only review.

## Solution Assessment

No for the issue's installed-SDK patch or an unconditional direct-bypass change. The safer direction is a Feishu plugin-owned, target-aware adapter at the Lark httpInstance seam that preserves operator proxy expectations.
