## Label Justifications

- `P1`: The report describes a broken official Feishu channel workflow for proxy-enabled users, blocking message delivery and startup identity resolution.
- `impact:security`: The remaining fix controls whether Feishu/Lark HTTPS egress honors or bypasses operator-managed proxy boundaries.

- `impact:auth-provider`: The failing path is Feishu/Lark tenant_access_token acquisition through the SDK HTTP client.

## Summary

Keep open. Current main still leaves Feishu tenant-token and SDK REST calls on the Lark SDK Axios-backed default HTTP instance with only timeout/User-Agent wrapping, and the latest 2026.5.20 report says a direct proxy bypass did not fully restore the gateway path.

## What This Changes

Feishu/Lark channel should keep message delivery working when HTTP proxy environment or managed proxy settings are configured instead of failing tenant-token acquisition.

## Best Possible Solution

Implement a Feishu-owned Lark HTTP adapter that preserves timeout/User-Agent behavior, avoids Axios proxy misrouting or double proxy handling, aligns REST and WebSocket egress with managed Proxyline policy, and adds focused proxy tests, docs, and redacted real Feishu proxy proof.

## Reproduction Assessment

Yes, at source level: configure Feishu with HTTP(S)_PROXY or managed proxy active, and probeFeishu reaches createFeishuClient and the Lark Axios-backed HTTP instance for token/API calls. I did not run live Feishu credentials through the reported local proxy.

## Solution Assessment

No for an installed-SDK patch or unconditional direct bypass. The maintainable fix is a target-aware Feishu HTTP adapter at the Lark httpInstance seam that preserves explicit operator proxy policy and covers the current SDK request behavior.
