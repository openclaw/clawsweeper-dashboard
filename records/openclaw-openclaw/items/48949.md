## Summary

Keep open. Current main still routes Feishu tenant-token and SDK REST calls through the Lark SDK default Axios-backed HTTP instance with only timeout/User-Agent wrapping, the related HTTPS proxy PR is closed unmerged, and newer reports show the proxy-bypass-only workaround may not cover the remaining Feishu SDK/request-construction failure.

## What This Changes

The issue reports Feishu tenant_access_token failures and blocked Feishu messaging when HTTP proxy environment variables are configured.

## Best Possible Solution

Settle a Feishu-owned REST/WebSocket egress policy, then implement a Lark HTTP adapter that preserves timeout/User-Agent behavior, managed Proxyline policy, NO_PROXY semantics or an explicit channel policy, and clear Feishu proxy diagnostics.

## Reproduction Assessment

Yes, at source level: current main still sends Feishu SDK token/API calls through the Axios-backed default HTTP instance while proxy env or managed proxy mode can be active. I did not run a live Feishu app through the reported local proxy.

## Solution Assessment

No for the installed-SDK patch or an unconditional direct-bypass change. The maintainable path is a target-aware Feishu HTTP adapter at the Lark httpInstance seam that preserves explicit operator proxy policy.
