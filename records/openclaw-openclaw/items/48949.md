## What This Changes

The issue asks OpenClaw to fix Feishu tenant access token and message delivery failures when Feishu/Lark SDK HTTP requests inherit incompatible HTTP(S) proxy settings.

## Best Possible Solution

Review and land, revise, or replace https://github.com/openclaw/openclaw/pull/86386 for the Feishu SDK REST/token proxy fix, then close this issue while tracking broader direct-vs-ambient proxy policy separately.

## Reproduction Assessment

Yes, source-reproducible: current main routes Feishu SDK REST/token HTTP through a timeout-only wrapper around the SDK default axios instance, while the SDK token path uses that httpInstance and axios reads proxy env unless proxy is false. I did not run a live Feishu account plus proxy scenario in this read-only review.

## Solution Assessment

Yes for the immediate implementation shape if maintainers accept the proxy contract: the linked PR keeps the change inside the Feishu plugin and reuses the shared proxy helper instead of mutating global axios. Broader direct-vs-ambient policy should stay with the related Feishu proxy-policy issues.
