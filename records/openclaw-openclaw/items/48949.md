## Summary

Keep open. Current main still leaves Feishu SDK HTTP token/API calls on the Lark Axios-backed default HTTP instance with only timeout/User-Agent wrapping, so the proxy-triggered tenant_access_token failure remains source-reproducible and needs an explicit Feishu proxy policy decision.

## What This Changes

The issue requests Feishu HTTP token/API calls to stop failing with tenant_access_token errors when ambient HTTP proxy variables route open.feishu.cn incorrectly.

## Best Possible Solution

Add a Feishu-owned HTTP proxy policy at the Lark SDK httpInstance seam that preserves timeout/User-Agent behavior, explicitly chooses direct versus ambient proxy behavior for Feishu/Lark/custom domains, honors NO_PROXY where applicable, and adds tests plus troubleshooting docs.

## Reproduction Assessment

Yes at source level. Current main delegates Feishu token/API POSTs through the Lark SDK default Axios instance without a Feishu proxy policy, and the published SDK can still turn a rejected token POST into the reported tenant_access_token destructuring error.

## Solution Assessment

No for the issue's blanket installed-SDK patch or unconditional direct-by-default suggestion. The maintainable solution is a target-aware Feishu plugin policy that coordinates with the existing proxy helper surface and preserves operator egress expectations.
