## Summary

Keep open. Current main still routes Feishu/Lark tenant-token and REST API calls through the Lark SDK Axios-backed default HTTP instance with only timeout/User-Agent wrapping, while managed proxy mode still injects HTTP(S)_PROXY and clears NO_PROXY; the remaining fix needs a Feishu egress-policy/security decision rather than cleanup closure.

## What This Changes

Fix Feishu channel tenant_access_token and message-send failures when HTTP(S) proxy environment settings misroute Lark SDK HTTPS requests.

## Best Possible Solution

Implement a Feishu-owned Lark HTTP adapter that prevents Axios proxy misrouting or double proxy handling while preserving timeout/User-Agent behavior, managed Proxyline/operator policy, HTTP/WebSocket consistency, focused proxy tests, troubleshooting docs, and redacted real Feishu proxy proof.

## Reproduction Assessment

Yes, at source level: the Feishu probe and SDK tenant/API calls still flow through the Lark Axios-backed HTTP wrapper while proxy env or managed proxy mode can be active. I did not run live Feishu credentials through the reported local proxy.

## Solution Assessment

No for the installed-SDK patch or an unconditional direct-bypass change. The maintainable path is a target-aware Feishu HTTP adapter at the Lark httpInstance seam that preserves explicit operator proxy policy.
