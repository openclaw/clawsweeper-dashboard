## Summary

Keep open: current main still lacks Feishu HTTP tenant-token/API proxy handling. The Feishu client wraps the Lark SDK HTTP instance for timeout/User-Agent only, while proxy env handling is currently wired for WebSocket clients; related PR #41943 is open but still needs changes before it can resolve this.

## What This Changes

The issue asks OpenClaw to prevent Feishu/Lark tenant-token and message API calls from failing with `tenant_access_token` errors when HTTP(S) proxy environment variables are configured.

## Best Possible Solution

Keep this issue open and resolve it by landing a corrected Feishu-owned HTTP adapter, likely via #41943 or a replacement, that uses target-aware proxy/NO_PROXY semantics and preserves the original token HTTP failure.

## Reproduction Assessment

Yes, source-backed rather than live-run: Feishu probe/API calls go through `createFeishuClient`, current main's HTTP wrapper lacks proxy-agent handling, and the upstream SDK still turns rejected tenant-token POSTs into the reported destructuring error.

## Solution Assessment

No, the proposed default Feishu-domain bypass is not the best direction. The maintainable fix should resolve the effective Feishu/Lark request URL, honor NO_PROXY, use HTTP CONNECT only when proxy env should apply, and preserve the underlying token HTTP error.
