## Summary

This issue should stay open: current `main` and the latest release still use a timeout-only Feishu Lark SDK HTTP wrapper, while the open linked PR at https://github.com/openclaw/openclaw/pull/86386 is the active implementation path for the reported proxy/token failure.

## What This Changes

The issue asks OpenClaw to make Feishu SDK HTTP/token requests work when the gateway process has HTTP proxy environment or managed proxy settings configured.

## Best Possible Solution

Resolve the Feishu egress policy and land or replace the linked PR so Lark SDK REST/token HTTP and WebSocket bootstrap/socket routing use one tested, maintainer-approved proxy contract.

## Reproduction Assessment

Yes, at source level: current main supplies the Lark SDK with a timeout-only HTTP wrapper, while the SDK token path uses that `httpInstance` and Axios reads ambient proxy env unless `proxy` is false. I did not run a live Feishu/proxy gateway in this read-only review.

## Solution Assessment

Unclear as a final product decision. The linked PR is the right owner boundary, but maintainers still need to accept the direct versus ambient versus managed proxy egress contract before merge.
