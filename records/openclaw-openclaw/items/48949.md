## Summary

Keep open: current main still routes Feishu SDK REST/token calls through the Lark SDK Axios-backed default HTTP instance without a Feishu-specific proxy policy, and the fresh 2026.5.20 report says a simple NO_PROXY or proxy:false bypass is not enough.

## What This Changes

The issue reports Feishu message delivery and tenant_access_token failures when the gateway inherits HTTP proxy settings.

## Best Possible Solution

Implement a Feishu-owned Lark HTTP adapter or explicit channel egress policy that keeps REST/token and WebSocket behavior consistent, preserves managed Proxyline security boundaries, adds focused proxy tests, updates docs, and proves the chosen mode with redacted real Feishu startup/send evidence.

## Reproduction Assessment

Yes, at source level: Feishu REST/token calls still go through the Lark SDK Axios default HTTP instance while proxy env or managed proxy mode can be active. I did not run a live Feishu app through the reported local proxies.

## Solution Assessment

No for the suggested installed-SDK patch or unconditional direct bypass. The maintainable fix needs an explicit Feishu egress policy at the Lark httpInstance seam, plus tests and real proxy proof.
