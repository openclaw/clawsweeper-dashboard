## Summary

Keep open. Current main still leaves Feishu tenant-token and SDK REST calls on the Lark SDK Axios-backed default HTTP instance with only timeout/User-Agent wrapping, while the correct direct-vs-proxy behavior is a Feishu egress-policy and security decision rather than cleanup closure.

## What This Changes

The issue requests fixing Feishu channel message delivery when ambient HTTP(S) proxy settings cause tenant_access_token acquisition to fail.

## Best Possible Solution

Implement a Feishu-owned Lark HTTP adapter that avoids Axios proxy misrouting or double proxy handling while preserving timeout/User-Agent behavior, managed Proxyline policy, HTTP/WebSocket consistency, focused proxy tests, docs, and redacted real Feishu proxy proof.

## Reproduction Assessment

Yes, at source level: current main still sends Feishu SDK token/API calls through the Axios-backed default HTTP instance while proxy env or managed proxy mode can be active. I did not run a live Feishu app through the reported local proxy.

## Solution Assessment

No for the installed-SDK patch or an unconditional direct-bypass change. The maintainable path is a target-aware Feishu HTTP adapter at the Lark httpInstance seam that preserves explicit operator proxy policy.
