## Summary

Keep open. Current main still has no Google Chat inbound reaction path: non-message webhook events either lack a supported reaction payload shape or return before Google Chat access checks, route resolution, session/system-event handling, and plugin-visible delivery.

## What This Changes

Add Google Chat inbound reaction handling so user-added and user-removed reactions delivered by Google Chat subscriptions can reach OpenClaw's runtime/plugin-visible event path.

## Best Possible Solution

Implement this inside the Google Chat plugin: parse official single and batch Workspace Chat reaction-created/deleted payloads, derive the affected space, message, user, and emoji, apply the Google Chat access/session routing model or a narrowly equivalent reaction path, enqueue durable session-scoped reaction system events, and document inbound subscription setup without folding in outbound OAuth work from #9764 or generic reaction-triggered turns from #17840.

## Reproduction Assessment

Yes for the current code path. A Google Chat webhook-like payload with a valid `space` and non-`MESSAGE` reaction event type reaches `processGoogleChatEvent` and exits at `extensions/googlechat/src/monitor.ts:59`; a live end-to-end repro still needs Google's exact Workspace Events payload shape.

## Solution Assessment

Unclear for the issue's proposed dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable repair is to mirror existing channel-owned reaction system events first, then add a plugin hook only if the current runtime contracts cannot represent Google Chat reaction metadata cleanly.
