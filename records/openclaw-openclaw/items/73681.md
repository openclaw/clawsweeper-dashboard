## Summary

Keep open. Current main still has no Google Chat inbound reaction path: the webhook parser accepts only message-shaped add-on normalization, the processor returns immediately for non-`MESSAGE` events, and there is no session/plugin-visible reaction delivery. The linked outbound OAuth issue and generic reaction-trigger issue are related but do not supersede this channel-specific inbound gap.

## What This Changes

Add Google Chat inbound reaction handling so reaction add/remove webhook events are parsed, authorized, routed, surfaced to sessions/plugins, and documented.

## Best Possible Solution

Implement the missing path inside the Google Chat plugin: parse official single and batch reaction event payloads, derive space/message/user/emoji from the reaction resource, apply the existing Google Chat access/session model or a narrowly equivalent reaction authorization path, enqueue deterministic session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth and generic reaction-triggered turns.

## Reproduction Assessment

Yes for the current code path: any parsed Google Chat event whose `type` or `eventType` is not `MESSAGE` reaches `processGoogleChatEvent` and returns immediately. A full live reproduction still needs a Google Workspace Events subscription that delivers reaction events to the webhook.

## Solution Assessment

Unclear for the issue's proposed dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable fix is channel-owned inbound reaction normalization and system-event delivery first, adding a plugin hook only if existing runtime contracts cannot preserve the needed metadata.
