## Summary

Keep open. Current main still accepts Google Chat webhook requests but has no inbound reaction handling path, so reaction-created/deleted events are not normalized, routed, or surfaced to sessions/plugins; the related OAuth and generic reaction-trigger issues do not supersede this channel-specific inbound gap.

## What This Changes

Add Google Chat inbound reaction support for `REACTION_ADDED` and `REACTION_REMOVED` webhook events, including normalization, routing, plugin/session visibility, and docs.

## Best Possible Solution

Implement the gap in the Google Chat plugin: parse official reaction-created/deleted webhook payloads, derive the affected space/message/user/emoji, apply the existing Google Chat access and routing model or a narrowly equivalent reaction path, enqueue session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth work.

## Reproduction Assessment

Yes for the current code path: a valid Google Chat webhook event with `eventType` or `type` other than `MESSAGE` reaches `processGoogleChatEvent` and returns immediately. A live end-to-end reproduction still depends on configuring Google Chat reaction event subscriptions and matching the official payload shape.

## Solution Assessment

No for making a dedicated `googlechat.onReaction` hook the mandatory first fix. The narrowest maintainable repair is channel-owned inbound reaction normalization and system-event delivery first, with a plugin hook added only if the existing runtime contracts cannot carry the needed metadata.
