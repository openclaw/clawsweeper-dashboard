## Summary

Keep open. Current main still accepts Google Chat webhook payloads but only routes `MESSAGE` events; reaction-created/deleted events have no parser branch, no inbound Google Chat reaction type, and no documented subscription or session/system-event delivery path. The related #9764 and #17840 issues cover outbound OAuth scope and generic reaction-triggered turns, not this channel-specific inbound drop.

## What This Changes

Add inbound Google Chat reaction handling so reaction add/remove webhook events reach the agent or plugin-visible pipeline instead of being ignored as non-message events.

## Best Possible Solution

Implement this inside the Google Chat plugin: parse official single and batch reaction event payloads, derive space/message/user/emoji/action, reuse or narrowly extend the existing Google Chat access and session routing model, enqueue deterministic session-scoped reaction system events, and document inbound reaction subscription setup. Avoid making this a core or outbound OAuth change unless the plugin cannot preserve required metadata through existing channel runtime seams.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any valid Google Chat event with a reaction event type and `space`: the webhook parser accepts non-`MESSAGE` types, then `processGoogleChatEvent` returns immediately before delivery. A live reproduction still needs a Chat app subscribed to reaction events.

## Solution Assessment

Yes for a channel-owned inbound reaction pipeline; unclear for adding a dedicated `googlechat.onReaction` hook as the first step. The narrower maintainable repair is to normalize and deliver reaction system events consistently with existing channel precedent, then add a plugin hook only if current runtime contracts are insufficient.
