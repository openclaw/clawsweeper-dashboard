## Summary

Keep open. Current main still routes Google Chat webhooks into a processor that returns for every non-`MESSAGE` event, so inbound reaction events remain invisible to agent/plugin event plumbing; the related OAuth and reaction-trigger issues do not cover this Google Chat parsing/routing gap.

## What This Changes

Request inbound Google Chat reaction webhook support so reaction add/remove events become agent/plugin-visible events.

## Best Possible Solution

Add channel-owned inbound reaction normalization inside the Google Chat plugin using documented single/batch payloads and existing system-event patterns, with docs/config aligned to other channels; keep outbound OAuth and reaction-triggered turns on their separate trackers.

## Reproduction Assessment

Yes. A source-level reproduction is any authenticated Google Chat webhook event whose `type` or `eventType` is not `MESSAGE`, such as a reaction event; current main dispatches it to `processGoogleChatEvent` and then returns before routing.

## Solution Assessment

Unclear as proposed. A Google Chat-specific `onReaction` hook may not be the narrowest first step; normalizing inbound reactions to the existing channel-owned system-event path is safer, with broader trigger/hook behavior handled by the existing reaction-trigger tracker.
