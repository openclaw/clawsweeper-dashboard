## Summary

Keep open. Current `main` still treats Google Chat webhook input as message-only and has no inbound reaction envelope, normalized reaction event, or Google Chat reaction plugin/runtime delivery path, so the reported gap remains real and is not superseded by the outbound OAuth or generic reaction-trigger issues.

## What This Changes

The issue requests Google Chat inbound reaction webhook support for reaction add/remove events, including plugin-visible routing and documentation.

## Best Possible Solution

Implement inbound reaction handling inside the Google Chat plugin boundary: parse documented reaction event payloads, preserve existing webhook security/access gates, enqueue deterministic session-scoped reaction system events, add focused tests, and document the required Google Chat event subscription separately from outbound OAuth work.

## Reproduction Assessment

Yes. A source-level reproduction is any Google Chat webhook event whose `type` or `eventType` is not `MESSAGE`: current `main` returns before routing and has no modeled reaction payload path. I did not perform a live Google Workspace webhook test in this read-only review.

## Solution Assessment

Mostly yes for channel-owned inbound reaction normalization, but unclear for adding a dedicated `googlechat.onReaction` hook as the first step. The narrower maintainable path is to deliver normalized reaction system/plugin events through existing contracts first, adding a new hook only if those contracts cannot carry the needed metadata.
