## Summary

Keep open. Current main still drops Google Chat inbound reaction events before they can become agent-visible system/plugin events, and the linked OAuth and generic reaction-trigger issues are adjacent rather than superseding.

## What This Changes

Add Google Chat inbound reaction handling so reaction add/remove events are parsed, authorized, routed, exposed to plugin or system-event consumers, and documented.

## Best Possible Solution

Implement this inside the Google Chat plugin boundary: parse documented single and batch reaction payloads, preserve existing webhook authentication/access gates, enqueue deterministic session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth work.

## Reproduction Assessment

Yes. A source-level reproduction is any Google Chat event whose `type` or `eventType` is not `MESSAGE`: current main returns before routing, and reaction payload fields are not modeled. I did not run a live Google Workspace subscription in this read-only review.

## Solution Assessment

Unclear for the requested dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable repair is channel-owned inbound reaction normalization and system-event delivery first, adding a plugin hook only if existing runtime contracts cannot carry the needed metadata.
