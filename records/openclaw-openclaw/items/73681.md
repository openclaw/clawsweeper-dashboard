## Summary

Keep open. Current main still exits on every non-MESSAGE Google Chat event, the parser/types/docs do not model inbound reaction payloads, and the open OAuth/turn-trigger trackers cover adjacent work rather than this channel-owned ingestion gap.

## What This Changes

Request Google Chat inbound reaction webhook handling so reaction-created/deleted events become agent-visible plugin or system events instead of being ignored.

## Best Possible Solution

Implement this inside the Google Chat plugin by normalizing documented reaction SpaceEvents into gated, session-scoped system events, with docs/config coverage, while leaving outbound OAuth and reaction-triggered turns to #9764 and #17840.

## Reproduction Assessment

Yes, source-level. Any authenticated Google Chat webhook event with a valid `space` and a `type`/`eventType` other than `MESSAGE` reaches `processGoogleChatEvent` and returns before routing; I did not live-verify a Workspace reaction subscription.

## Solution Assessment

No for adding a Google Chat-specific `onReaction` hook as the first step. The narrower maintainable path is channel-owned inbound reaction normalization plus the existing system-event pattern, with generic reaction-triggered turns handled by #17840.
