## Summary

Keep open. Current main still drops Google Chat non-message webhook events before any route, system-event, plugin, or agent plumbing can observe inbound reactions, and the linked OAuth/reaction-trigger trackers cover adjacent but different work.

## What This Changes

The issue requests inbound Google Chat reaction webhook handling so reaction add/remove events can reach plugin or agent-visible reaction plumbing.

## Best Possible Solution

Add plugin-owned Google Chat inbound reaction handling for current Workspace Events reaction payloads, normalize it into the existing system-event/notification pattern, and document setup separately from outbound OAuth and reaction-triggered turns.

## Reproduction Assessment

Yes, source-level. An authenticated Google Chat event with any non-`MESSAGE` type is dispatched to `processGoogleChatEvent`, which returns before routing or delivery, so a reaction event remains invisible on current main.

## Solution Assessment

Unclear as proposed. A dedicated `googlechat.onReaction` hook may be broader than the narrow first step; matching other channels by adding plugin-local reaction normalization and notification policy looks safer, with agent-turn behavior left to the existing reaction-trigger tracker.
