## Summary

Keep open. Current main still only processes message-shaped Google Chat inbound events and has no reaction payload model, normalization, access/session routing, or inbound docs; the linked OAuth and generic reaction-trigger issues are adjacent but do not supersede this Google Chat inbound gap.

## What This Changes

Add Google Chat inbound reaction handling so reaction-created/deleted events are parsed and routed instead of being dropped before any agent or plugin-visible delivery.

## Best Possible Solution

Implement this inside the Google Chat plugin: normalize official single and batch reaction event payloads, derive stable space/message/user/emoji identifiers, apply the existing Google Chat ingress access model or a narrowly equivalent reaction authorization path, enqueue deterministic session-scoped reaction system events, and document inbound reaction subscription setup separately from outbound reaction OAuth work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any accepted Google Chat event with `type` or `eventType` not equal to `MESSAGE`: current `processGoogleChatEvent` returns before delivery. A live proof still needs a Google Workspace Events reaction subscription/PubSub setup.

## Solution Assessment

Unclear for the requested dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable fix is Google Chat-owned reaction normalization and system-event delivery first, adding a public hook only if existing plugin/runtime contracts cannot carry the needed metadata.
