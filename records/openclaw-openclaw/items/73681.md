## Summary

Keep open. Current main still has the Google Chat inbound reaction gap: the webhook path accepts non-message event types but the processor returns before authorization, routing, or delivery, and there is no Google Chat reaction payload model or docs/config surface for inbound reaction notifications. The related OAuth and generic reaction-trigger issues remain adjacent, not superseding, work.

## What This Changes

The issue requests Google Chat inbound reaction events to be parsed, authorized, normalized, delivered to a runtime/plugin-visible path, and documented.

## Best Possible Solution

Implement the missing path inside the Google Chat plugin: parse official single and batch reaction event payloads, derive space/message/user/emoji metadata, authorize with the existing Google Chat access/session model, enqueue deterministic session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth work and generic reaction-triggered turns.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any parsed Google Chat event whose `type` or `eventType` is not `MESSAGE`: current `processGoogleChatEvent` returns immediately, and the parser/types do not model official reaction payloads. A live Google Workspace subscription was not run in this read-only review.

## Solution Assessment

Unclear for adding a dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable repair is channel-owned inbound reaction normalization and system-event delivery first, adding a plugin hook only if existing runtime contracts cannot preserve the needed metadata.
