## Summary

Keep open. Current main still drops Google Chat non-`MESSAGE` webhook events before access/routing and has no inbound reaction payload model or delivery path; the linked outbound OAuth and generic reaction-trigger issues are adjacent, not superseding fixes.

## What This Changes

The issue requests Google Chat inbound reaction event support so added/removed reactions are parsed, routed to plugin or agent-visible events, and documented.

## Best Possible Solution

Implement this in the Google Chat plugin boundary: parse documented single and batch reaction payloads, preserve existing webhook authentication and access gates, enqueue deterministic session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any Google Chat event whose `type` or `eventType` is not `MESSAGE`: current main returns before routing, and reaction payload fields are not modeled. I did not run a live Google Workspace subscription.

## Solution Assessment

Mostly yes for channel-owned inbound reaction normalization; unclear for a dedicated `googlechat.onReaction` hook as the first step. The narrower maintainable path is to deliver normalized reaction system/plugin events through existing contracts first, adding a new hook only if those contracts cannot carry the needed metadata.
