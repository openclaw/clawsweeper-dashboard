## Summary

Keep open. Current main still drops non-message Google Chat webhook events before they can become agent-visible system/plugin events, and the adjacent Google Chat OAuth and generic reaction-trigger issues do not supersede this inbound parsing/routing gap.

## What This Changes

Request inbound Google Chat reaction webhook support so reaction-created/deleted events are normalized and made visible to OpenClaw plugins or sessions.

## Best Possible Solution

Implement this inside the Google Chat plugin boundary: parse documented single and batch reaction payloads, preserve existing webhook auth/access gates, enqueue deterministic session-scoped reaction system events, add the needed config/docs/tests, and leave outbound OAuth plus reaction-triggered agent turns to their separate trackers.

## Reproduction Assessment

Yes. A source-level reproduction is any valid authenticated Google Chat event whose `type` or `eventType` is not `MESSAGE`: current main returns before routing, and reaction payload fields are not modeled. I did not run a live Workspace subscription.

## Solution Assessment

No for a Google Chat-specific `onReaction` hook as the first step. The narrower maintainable solution is channel-owned inbound reaction normalization and system-event delivery, with #17840 covering opt-in reaction-triggered turns.
