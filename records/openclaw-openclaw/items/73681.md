## Summary

Keep open. Current main still drops or rejects Google Chat inbound reaction events before they can become agent-visible system/plugin events, and the related OAuth and generic reaction-trigger issues do not cover this channel-owned ingress gap.

## What This Changes

Add Google Chat inbound reaction handling so reaction add/remove events are parsed, authorized, routed, exposed to plugin or system-event consumers, and documented.

## Best Possible Solution

Implement this in the Google Chat plugin boundary: parse official single and batch reaction event payloads, derive space/message/user/emoji metadata, preserve existing webhook authentication and access gates, enqueue deterministic session-scoped reaction system events, and document the subscription/setup path. Add a new plugin hook only if existing system-event and message hook contracts cannot carry the required metadata cleanly.

## Reproduction Assessment

Yes. A source-level reproduction is any non-`MESSAGE` Google Chat event that reaches `processGoogleChatEvent`: current main returns before routing, and official reaction payload fields are not modeled. A live Google Workspace subscription was not run in this read-only review.

## Solution Assessment

Unclear for the requested dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable fix is channel-owned inbound reaction normalization and system-event delivery first, with a new plugin hook only if the existing runtime contracts are insufficient.
