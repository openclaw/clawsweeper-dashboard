## Summary

Keep open. Current main still drops Google Chat inbound reaction events at the channel processor boundary, and the related OAuth and generic reaction-trigger issues are adjacent rather than replacements for this channel-specific ingress gap.

## What This Changes

The issue requests inbound Google Chat reaction webhook events to be parsed and routed to plugin or agent-visible reaction handling instead of being dropped as non-message events.

## Best Possible Solution

Implement this inside the Google Chat plugin boundary: parse documented single and batch reaction payloads, preserve existing webhook auth/access gates, route deterministic session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth work.

## Reproduction Assessment

Yes. A source-level reproduction is any Google Chat webhook event whose `type` or `eventType` is not `MESSAGE`: current main returns before routing it, and reaction payload fields are not modeled. I did not run a live Google Workspace subscription in this read-only review.

## Solution Assessment

Yes for channel-owned inbound reaction normalization and system-event delivery. A dedicated `googlechat.onReaction` hook is not clearly the first step unless existing system-event/plugin contracts cannot carry the required metadata.
