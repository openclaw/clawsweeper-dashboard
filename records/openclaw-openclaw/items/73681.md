## Summary

Keep open. Current main still has the Google Chat inbound reaction gap: message-shaped webhooks are handled, but non-message events either have no reaction parsing path or return before delivery, and the related OAuth and generic reaction-trigger issues do not supersede this channel-specific ingress bug.

## What This Changes

The issue requests Google Chat inbound reaction webhook events to be parsed, authorized, normalized, delivered to the runtime/plugin-visible path, and documented.

## Best Possible Solution

Implement the missing path inside the Google Chat plugin: parse official single and batch reaction event payloads, derive space/message/user/emoji metadata, authorize with the existing Google Chat access/session model, enqueue deterministic session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth work and generic reaction-triggered turns.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any parsed Google Chat event whose `type` or `eventType` is not `MESSAGE`: it reaches `processGoogleChatEvent` and returns immediately; official Workspace reaction payloads are also currently unmodeled by the parser/types. A live Google Workspace subscription was not run in this read-only review.

## Solution Assessment

Unclear for adding a dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable repair is channel-owned inbound reaction normalization and system-event delivery first, adding a plugin hook only if existing runtime contracts cannot preserve the needed metadata.
