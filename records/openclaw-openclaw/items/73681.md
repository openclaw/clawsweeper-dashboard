## Summary

Keep open. Current main still accepts Google Chat webhook payloads but drops non-`MESSAGE` events before any reaction normalization, access-policy routing, or agent-visible system/plugin event can happen, and the related OAuth and reaction-trigger issues are adjacent rather than superseding this inbound gap.

## What This Changes

Add Google Chat inbound reaction handling so reaction-created/deleted webhook events are parsed and routed as agent-visible or plugin-visible events.

## Best Possible Solution

Implement this inside the Google Chat plugin boundary: parse documented single and batch reaction payloads, preserve existing webhook auth/access gates, emit deterministic session-scoped reaction system events or the maintainer-approved equivalent, and leave outbound OAuth plus reaction-triggered turns to #9764 and #17840.

## Reproduction Assessment

Yes, source-level. Any authenticated Google Chat event that reaches `processGoogleChatEvent` with a non-`MESSAGE` type returns before routing, and current event types do not model reaction payloads; I did not establish a live Workspace reproduction.

## Solution Assessment

No for a Google Chat-specific `onReaction` hook as the first step. The narrower maintainable path is channel-owned inbound reaction normalization using the existing system-event pattern, with #17840 deciding whether reactions should wake agent turns.
