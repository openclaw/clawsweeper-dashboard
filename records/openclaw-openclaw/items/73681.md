## Summary

Keep open. Current main still drops Google Chat inbound reaction events: the webhook parser accepts non-message events with a space, but the processor returns before access checks, routing, or delivery unless the type is exactly `MESSAGE`. The related OAuth issue #9764 and generic reaction-trigger issue #17840 are adjacent, not replacements for this channel-owned ingress gap.

## What This Changes

Request Google Chat inbound reaction webhook events be parsed, authorized, and routed to runtime/plugin-visible reaction handling instead of being ignored.

## Best Possible Solution

Implement this inside the Google Chat plugin: parse official single and batch reaction event payloads, derive stable space/message/user/emoji metadata, preserve existing webhook authentication and access gates, emit deterministic session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth work and generic reaction-triggered turns.

## Reproduction Assessment

Yes. A source-level reproduction is any authorized Google Chat webhook payload with a valid `space` and a reaction event type: the webhook path can accept and acknowledge it, then `processGoogleChatEvent` returns because the event type is not `MESSAGE`. A live Google Workspace subscription was not run in this read-only review.

## Solution Assessment

Unclear for adding a dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable fix is channel-owned inbound reaction normalization and system-event delivery first, adding a plugin hook only if existing runtime/plugin contracts cannot carry the needed metadata.
