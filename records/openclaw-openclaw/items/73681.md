## Summary

Keep open. Current main still drops non-message Google Chat webhook events before they can become agent-visible reaction notifications, and the related OAuth and reaction-trigger issues do not supersede this inbound parsing/routing gap.

## What This Changes

Add inbound Google Chat reaction handling so reaction add/remove events are normalized and delivered as agent/plugin-visible notifications.

## Best Possible Solution

Implement this inside the Google Chat plugin boundary: parse documented reaction create/delete and batch payloads, preserve webhook auth/access gates, enqueue deterministic session-scoped reaction system events, add config/docs/tests, and leave outbound OAuth plus reaction-triggered turns to their separate trackers.

## Reproduction Assessment

Yes. A source-level reproduction is any authenticated Google Chat event whose `type` or `eventType` is not `MESSAGE`: current main returns before routing, and reaction payload fields are not modeled. I did not run a live Workspace subscription.

## Solution Assessment

Unclear for the requested Google Chat-specific `onReaction` hook. The narrower maintainable solution is channel-owned inbound reaction normalization and system-event delivery, consistent with the existing channel patterns, while #17840 covers opt-in reaction-triggered turns.
