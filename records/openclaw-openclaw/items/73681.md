## Summary

Keep open. The prior ClawSweeper keep-open review still matches current main: Google Chat webhook handling accepts message-shaped events but still has no inbound reaction parsing, authorization, routing, or session/plugin-visible delivery, and the related OAuth and reaction-trigger issues do not supersede this channel-specific gap.

## What This Changes

Add Google Chat inbound reaction handling so reaction add/remove webhook events are parsed, authorized, routed, surfaced to sessions/plugins, and documented.

## Best Possible Solution

Implement the missing path inside the Google Chat plugin: parse official single and batch reaction event payloads, derive the affected space, message, user, and emoji, apply the existing Google Chat access/session model or a narrowly equivalent reaction authorization path, enqueue deterministic session-scoped reaction system events, and document inbound subscription setup separately from outbound OAuth and generic reaction-triggered turns.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any parsed Google Chat event whose `type` or `eventType` is not `MESSAGE`: it reaches `processGoogleChatEvent` and returns immediately; a live end-to-end repro still needs a Workspace Events reaction subscription.

## Solution Assessment

Unclear for the issue's proposed dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable repair is channel-owned inbound reaction normalization and system-event delivery first, adding a plugin hook only if existing runtime contracts cannot preserve the needed metadata.
