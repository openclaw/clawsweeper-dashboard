## Summary

Keep open. Current main still accepts only Google Chat message events into the runtime path and has no parser, type model, config, docs, or tests for inbound reaction-created/deleted events, while official Google Workspace Events docs show Chat reaction single and batch events as subscribable payloads.

## What This Changes

Add inbound Google Chat reaction webhook handling so reaction add/remove events reach OpenClaw as normalized plugin-visible or session-scoped reaction events.

## Best Possible Solution

Implement this in the bundled Google Chat plugin: parse documented single and batch Workspace Chat reaction-created/deleted payloads, derive the affected space/message/user/emoji, apply the same access and session-routing model used for messages where possible, enqueue durable session-scoped reaction system events, and document the required Google Chat subscription setup without folding in outbound OAuth or generic reaction-triggered turns.

## Reproduction Assessment

Yes for a source-level reproduction: any parsed Google Chat event whose type is not `MESSAGE` exits at `extensions/googlechat/src/monitor.ts:59`. A live end-to-end reproduction was not run in this read-only pass, but the official Google docs confirm reaction event payloads exist for subscriptions.

## Solution Assessment

Unclear for the proposed dedicated `googlechat.onReaction` hook as the first step. The narrowest maintainable fix is to mirror existing channel-owned reaction system events first, then add a plugin hook only if the current runtime contracts cannot carry the needed Google Chat metadata.
