## Summary

Keep open. Current main still drops Google Chat non-message webhook events before any reaction payload can become a routed system/plugin-visible event, and the related OAuth and generic reaction-turn trackers do not supersede this channel-specific inbound parsing gap.

## What This Changes

Request inbound Google Chat reaction webhook events be parsed, normalized, and exposed to agents or plugins instead of being ignored.

## Best Possible Solution

Implement the capability inside the Google Chat plugin boundary by parsing documented reaction payloads, applying existing auth/access/routing checks, enqueueing deterministic reaction system events, and documenting the inbound notification setting.

## Reproduction Assessment

Yes, source-level. Any authenticated Google Chat event with `type`/`eventType` other than `MESSAGE` reaches the webhook processor and then returns at `processGoogleChatEvent`, so reaction payloads cannot be routed on current main.

## Solution Assessment

No for the exact Google-specific `onReaction` hook as the first step. The narrower maintainable path is channel-owned reaction normalization into the existing system-event pattern, with #17840 covering whether reactions should wake agent turns.
