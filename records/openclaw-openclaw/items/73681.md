## Summary

Keep open. Current main still has no inbound Google Chat reaction handling: authenticated non-`MESSAGE` events reach the Google Chat processor and return before routing, while the parser/types/docs only cover message events and outbound reaction actions; related #9764 and #17840 track outbound OAuth and reaction-trigger semantics rather than this inbound event gap.

## What This Changes

Request inbound Google Chat reaction webhook events to be parsed and exposed to plugin or agent event plumbing instead of being ignored.

## Best Possible Solution

Add Google Chat-owned inbound reaction support using documented Workspace Events payloads, normalize it to the existing reaction notification/system-event pattern, and document subscription/config separately from outbound OAuth work.

## Reproduction Assessment

Yes. A source-level reproduction is any authenticated Google Chat webhook event whose `type` or `eventType` is not `MESSAGE`; current main dispatches it to `processGoogleChatEvent`, which returns before routing or notification delivery.

## Solution Assessment

Unclear as proposed. A dedicated `googlechat.onReaction` hook may be broader than needed; the narrower maintainable direction is channel-owned normalization to existing system-event/reaction-notification semantics, with agent-turn triggering left to #17840.
