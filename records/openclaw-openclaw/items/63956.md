## Summary

Keep open: current main still has the reported Control UI streaming layout behavior, where committed stream segments and the live tail become separate stream items and each stream item renders as its own assistant chat group. The direct layout-only fix PR is closed unmerged, while adjacent open streaming PRs target duplicate text rather than the single-bubble grouping request.

## What This Changes

The issue requests that Control UI streaming assistant replies stay in one in-place assistant bubble/group instead of rendering multiple transient bubbles before final history collapse.

## Best Possible Solution

Implement a Control UI-only grouped rendering path for one active assistant run, interleaving committed text segments, visible tool cards, and the live tail inside one outer assistant group without changing gateway streaming protocol or persisted transcript shape.

## Reproduction Assessment

Yes. A source-level reproduction is to call buildChatItems with at least one streamSegments entry and a non-empty stream, then render those items: current main produces multiple kind:"stream" items and each becomes its own .chat-group.assistant.

## Solution Assessment

Yes. A UI-only stream-run item or equivalent single outer assistant group is the narrowest maintainable fix; changing gateway deltas, provider behavior, or transcript persistence would be broader than the reported layout bug.
