## Summary

Keep open: the current Control UI still emits committed stream segments and the live tail as separate stream items, and each stream item renders as a complete assistant chat group. The closed direct fix at https://github.com/openclaw/openclaw/pull/63986 was not merged, while the still-open adjacent streaming PR addresses duplicate text after tool boundaries rather than the single outer-bubble grouping requested here.

## What This Changes

The issue asks for Control UI streaming assistant responses to stay in one assistant bubble/group while the in-flight content updates.

## Best Possible Solution

Implement a Control UI-only grouped render path for one active assistant turn, keeping committed text segments, visible tool cards, and the live tail inside one outer assistant group without changing gateway streaming protocol or persisted transcript shape.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to build chat items with at least one streamSegments entry plus a non-empty live stream: current main emits multiple kind:"stream" items and renders each as its own .chat-group.assistant.

## Solution Assessment

Yes. A UI-only stream-run item or equivalent single outer assistant group is the narrowest maintainable fix; changing gateway deltas, provider behavior, transcript persistence, or adding config would be broader than the reported layout bug.
