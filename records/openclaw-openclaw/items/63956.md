## Summary

Keep open: current main still has the reported Control UI streaming layout path, where committed stream segments and the live tail become separate stream items and each stream item renders as its own assistant chat group. The direct layout PR is closed unmerged, and the active adjacent PR addresses duplicate text after tool boundaries rather than the single-bubble grouping request.

## What This Changes

The issue requests that Control UI streaming assistant responses remain in one updating assistant bubble/group during a run instead of rendering separate bubbles for stream fragments.

## Best Possible Solution

Implement a Control UI-only grouped render path for one active assistant run, keeping committed text segments, visible tool cards, and the live tail inside one outer assistant group without changing gateway streaming protocol or persisted transcript shape.

## Reproduction Assessment

Yes. A source-level reproduction is to build chat items with at least one streamSegments entry and a non-empty stream, then render them: current main creates multiple kind:"stream" items and each becomes its own .chat-group.assistant.

## Solution Assessment

Yes. A UI-only stream-run item or equivalent single outer assistant group is the narrowest maintainable fix; changing gateway deltas, provider behavior, transcript persistence, or adding config would be broader than the reported layout bug.

Current main commits pre-tool chatStream into chatStreamSegments at tool start, buildChatItems emits each committed segment plus the live tail as separate kind:"stream" items, and renderChat renders each stream item with renderStreamingGroup, causing multiple assistant groups during one in-flight turn. Implement a UI-only grouped active assistant run: committed segments, visible tool messages, reading indicator, and live tail should render inside one outer .chat-group.assistant.
