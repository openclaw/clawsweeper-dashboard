## Summary

Keep open. Current main still omits Telegram `humanDelay` wiring, and this PR is the canonical open implementation for the linked regression. The diff is narrow, follows the existing channel pattern, has focused test/changelog coverage, and prior maintainer triage already scoped it as a safe small merge rather than a fix for unrelated Telegram polling stalls.

## What This Changes

The PR re-exports `resolveHumanDelayConfig` through Telegram's local runtime barrel, passes the resolved value into Telegram block-stream dispatcher options, and adds a regression test plus changelog entry.

## Best Possible Solution

Land this PR, or an equivalent scoped patch, so Telegram block-stream replies use the existing agent-scoped `humanDelay` resolver through the local runtime barrel. Keep the scope limited to configured reply pacing for the linked regression and do not treat it as a fix for Telegram long-poll stalls.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with Telegram block streaming and `agents.defaults.humanDelay` or per-agent `humanDelay` configured, current main never supplies `options.humanDelay` to the shared dispatcher, so no configured inter-block delay can be applied.

## Solution Assessment

Yes. The PR uses the existing plugin-facing resolver and Telegram's local runtime barrel, then passes the value at the single shared dispatcher call site; that is narrower and more maintainable than adding Telegram-specific delay logic.
