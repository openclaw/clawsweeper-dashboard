## Summary

Keep open. Current main still omits Telegram `humanDelay` wiring, while this PR applies the existing cross-channel resolver at the right Telegram dispatch seam, includes coverage and changelog, and has maintainer triage identifying it as the preferred scoped fix for #68945.

## What This Changes

The PR re-exports `resolveHumanDelayConfig` through Telegram's local agent-runtime barrel, passes its result into Telegram block-stream dispatcher options, and adds a regression test plus changelog entry.

## Best Possible Solution

Land this PR, or an equivalent scoped patch, so Telegram block-stream replies use the existing agent-scoped `humanDelay` resolver through the local runtime barrel without expanding the fix into Telegram polling-stall behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with Telegram block streaming and configured `agents.defaults.humanDelay` or per-agent `humanDelay`, current main never supplies `options.humanDelay` to the shared dispatcher, so no configured inter-block delay can be applied.

## Solution Assessment

Yes. Passing `resolveHumanDelayConfig(cfg, route.agentId)` through Telegram's local runtime barrel into the shared dispatcher is the narrowest maintainable fix; adding Telegram-specific sleep logic would duplicate existing core behavior.
