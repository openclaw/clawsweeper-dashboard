## Summary

Keep open for normal maintainer merge review. Current main has a runner-level stripping wrapper, but the shared OpenAI Completions payload builder exported through the plugin SDK still emits tool fields without consulting the existing tool-support contract; this PR is a focused lower-layer fix with regression coverage and sufficient payload proof.

## What This Changes

The PR wraps OpenAI Completions tool emission and tool-history fallback with `supportsModelTools(model)`, adds focused regression tests, and records a changelog entry.

## Best Possible Solution

Land the builder-level guard with its regression tests while keeping the existing embedded-runner wrapper as defense in depth.

## Reproduction Assessment

Yes. Source inspection of current main shows the shared OpenAI Completions builder still emits `tools`, `tool_choice`, or `tools: []` without checking `supportsModelTools(model)`, so a `compat.supportsTools: false` model with tools or tool history has a deterministic bad-payload path.

## Solution Assessment

Yes. Reusing the existing `supportsModelTools(model)` helper inside the payload builder is the narrow maintainable fix because it covers exported builder/plugin-SDK callers while preserving current default tool-capable behavior.
