## Summary

Keep open for normal maintainer review: current main has a runtime wrapper that strips unsupported tool fields, but the exported OpenAI Completions payload builder still emits `tools` and the `tools: []` history fallback without checking the existing `supportsModelTools(model)` contract. The PR is focused, has sufficient builder-level proof, and I found no blocking correctness or security findings.

## What This Changes

Adds a `supportsModelTools(model)` guard to `buildOpenAICompletionsParams`, two regression tests, and a changelog entry for OpenAI Completions models with `compat.supportsTools: false`.

## Best Possible Solution

Land the builder-level guard and focused regression tests while keeping the existing wrapper as defense in depth, with the duplicate changelog text consolidated during landing.

## Reproduction Assessment

Yes. Source inspection of current main shows the exported payload builder still emits tool fields without checking `supportsModelTools(model)`, and the PR includes redacted after-fix builder output showing those fields absent for `compat.supportsTools: false`.

## Solution Assessment

Yes. Reusing the existing `supportsModelTools(model)` helper inside `buildOpenAICompletionsParams()` is the narrow maintainable fix because it covers direct builder callers without changing default tool-capable behavior.
