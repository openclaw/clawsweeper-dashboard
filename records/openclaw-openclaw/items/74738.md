## Summary

Keep open: current main has a runtime wrapper that strips unsupported OpenAI Completions tool fields, but the exported completions params builder still emits them; this PR correctly fixes that lower-level builder path, but the external PR still needs real behavior proof before merge.

## What This Changes

The PR wraps OpenAI Completions tool and tool-history payload emission in `supportsModelTools(model)`, adds two transport regression tests, and adds a changelog entry.

## Best Possible Solution

Land the builder-level `supportsModelTools()` guard with the regression tests so the exported transport helper and runtime stream path share the same compat contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. On current main, calling `buildOpenAICompletionsParams()` with `compat.supportsTools: false` and non-empty `context.tools`, or with detected tool history and no current tools, still reaches unguarded `params.tools` assignments.

## Solution Assessment

Is this the best way to solve the issue? Yes. Guarding the params builder is the narrow maintainable fix because the helper is exported through the provider transport SDK runtime path; the current wrapper remains useful defense in depth but is not a complete builder-level fix.
