## Summary

Keep this PR open, but it is not ready to merge: the warning is aimed at a real cron UX gap, yet the current patch can disagree with the gateway/runtime behavior it is trying to predict and it still lacks contributor real-behavior proof.

## What This Changes

The PR adds a non-blocking `cron edit --model` warning by loading local config/catalog data, building an allowed model set, parsing the CLI model value, and warning when the computed key is outside that set.

## Best Possible Solution

Add a gateway-backed or shared runtime validation path for `cron edit --model` so the warning is computed with the same target config and resolver that cron execution uses.

## Reproduction Assessment

Yes. Source inspection shows current main stores `--model` into the cron payload without an edit-time warning, while cron execution later rejects invalid payload models through `resolveAllowedModelRef`; I did not run a live CLI command in this read-only review.

## Solution Assessment

No. The PR moves in the right direction, but predicting runtime validity from local CLI config and `parseModelRef` is not the narrowest maintainable fix when cron edits can target a different gateway and runtime resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate against the target gateway:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: This warning reads local CLI config/catalog state before `cron.update`, but `cron edit` can target a remote gateway via `--url`/`--token`. If local and remote allowlists differ, the command can warn incorrectly or miss the warning for the job that will actually run.
  - confidence: 0.93
- **[P2] Reuse the cron runtime resolver:** `src/cli/cron-cli/register.cron-edit.ts:223-228`
  - body: This builds the allowlist with hard-coded defaults and checks `parseModelRef`, while cron execution uses `resolveAllowedModelRef` with runtime defaults, aliases, bare-id provider inference, auth-profile suffix stripping, and compat aliases. Valid cron models can still warn incorrectly, and invalid ones can still disagree with runtime validation.
  - confidence: 0.94
