## Summary

Keep open: current main still lacks edit-time feedback for disallowed cron model overrides, but this PR should not merge as-is because its warning can disagree with the gateway/runtime decision it is trying to predict.

## What This Changes

The PR adds a non-blocking `cron edit --model` warning by loading local config/model catalog and checking the supplied model against the allowed model set before sending `cron.update`.

## Best Possible Solution

Expose or reuse a gateway-side model validation contract so `cron edit --model` can emit a non-blocking warning that matches the runtime executor for local and remote gateways.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by comparing the PR's local `loadConfig()` plus `parseModelRef(DEFAULT_PROVIDER)` warning path with current cron execution's gateway-side `resolveAllowedModelRef` path.

## Solution Assessment

No. The warning direction is useful, but predicting runtime validation from local CLI state and a partial resolver is not the narrowest maintainable solution.
