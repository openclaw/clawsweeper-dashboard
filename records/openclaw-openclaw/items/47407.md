## What This Changes

The PR adds a non-blocking `cron edit --model` warning by loading local config/model catalog data and checking the requested model against the allowed model set before sending `cron.update`.

## Best Possible Solution

Add the warning or preflight at a gateway-owned runtime seam that uses the same config, catalog, resolved defaults, and `resolveAllowedModelRef` path as cron execution, with focused coverage for resolver edge cases.

## Reproduction Assessment

Yes, source-reproducible: current main accepts `cron edit --model` into `patch.payload.model`, while cron execution later rejects disallowed payload models through `resolveAllowedModelRef`. I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. The warning is useful, but this implementation reads local CLI config and partially reimplements model validation instead of using the gateway/runtime resolver that will execute the cron.
