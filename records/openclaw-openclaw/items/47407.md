## Summary

Keep open: the warning requested by this PR is not on current main, but the branch still approximates runtime validation from local CLI state, can disagree with the selected gateway, and lacks after-fix real behavior proof.

## What This Changes

The branch adds a non-blocking `cron edit --model` warning by loading local config/catalog data and comparing the requested model against the allowed model set before sending the cron update.

## Best Possible Solution

Add a gateway-backed cron model preflight or warning path that uses the same runtime config, catalog, defaults, aliases, and resolver as cron execution, with focused regression coverage and real CLI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: `cron edit --model` stores the model without edit-time feedback, while cron execution later calls `resolveAllowedModelRef` and can reject the payload model through the allowlist path.

## Solution Assessment

No. The warning is useful, but this implementation reads local CLI config and duplicates part of runtime validation instead of using the selected gateway/runtime resolver that will actually execute the cron.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate against the selected gateway runtime:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: This warning reads `loadConfig()` and `loadModelCatalog()` in the CLI process, but `cron edit` sends `cron.update` to the selected gateway and supports `--url`/`--token`. In remote-gateway setups, local and gateway allowlists can differ, so the warning can be wrong or absent for the job that will actually run.
  - confidence: 0.93
- **[P2] Reuse the runtime model resolver for the warning:** `src/cli/cron-cli/register.cron-edit.ts:220-228`
  - body: The warning builds `allowedKeys` with hard-coded defaults and checks `parseModelRef(model, DEFAULT_PROVIDER)` directly. Cron execution uses `resolveAllowedModelRef` with configured defaults, aliases, provider inference, auth-profile suffix stripping, and compatibility aliases, so valid models can still warn and invalid ones can be missed.
  - confidence: 0.94
