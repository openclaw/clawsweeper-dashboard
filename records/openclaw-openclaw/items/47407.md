## Summary

Keep this PR open, but do not merge it as-is: the requested warning is still useful and not implemented on current main, but the patch can disagree with the gateway/runtime validation path and lacks after-fix real behavior proof.

## What This Changes

The PR adds a non-blocking `openclaw cron edit --model` warning by loading local config/model catalog data and comparing the requested model to a computed allowed model set before `cron.update`.

## Best Possible Solution

Expose or reuse a target-gateway/runtime validation path for cron edit so the warning is computed with the same config, defaults, aliases, and allowlist resolver that will execute the cron.

## Reproduction Assessment

Yes. Source inspection shows current main stores `--model` into the cron payload without edit-time allowlist feedback, while cron execution later rejects invalid payload models through `resolveAllowedModelRef`; I did not run a live CLI command in this read-only review.

## Solution Assessment

No. The warning direction is useful, but predicting runtime validity from local CLI config and hard-coded defaults is not the narrowest maintainable fix when cron edits can target another gateway and runtime resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

  - body: This warning reads local CLI config/catalog state before `cron.update`, but `cron edit` can target a remote gateway via `--url` and `--token`. If local and remote allowlists differ, the command can warn incorrectly or miss the warning for the job that will actually run.
  - confidence: 0.93

Valid cron models can still warn incorrectly, and invalid ones can still disagree with runtime validation.
  - confidence: 0.94
