## Summary

Keep open. The PR addresses a real cron DX gap, but the branch predicts runtime allowlist behavior from local CLI state and partial model parsing, so it can warn incorrectly or miss the failure it is meant to predict. Current main still has no edit-time warning.

## What This Changes

The PR adds a non-blocking stderr warning in `openclaw cron edit --model` by loading local config/model catalog and checking the requested model against an allowed model set before calling `cron.update`.

## Best Possible Solution

Add a target-gateway validation path for `cron edit --model` that reuses the runtime model resolver with the gateway's config/catalog and resolved defaults, then keep the warning non-blocking if that remains the intended UX.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by comparing the PR's local `loadConfig()` plus `parseModelRef(DEFAULT_PROVIDER)` warning path with current cron execution's target-gateway `resolveAllowedModelRef` path; remote `--url`, aliases, bare IDs, and auth-profile suffixes are concrete mismatch cases.

## Solution Assessment

No. The warning direction is useful, but this implementation is not the best fix because it predicts a gateway runtime decision from local CLI state and a partial resolver instead of asking the same runtime contract that executes the cron job.

## Review Findings

  - body: The warning reads `loadConfig()` in the local CLI process before sending `cron.update`, but this command can edit a remote gateway via `--url`/`--token`. If local and remote allowlists differ, the new warning can be wrong or absent for the job that will actually run.
  - confidence: 0.93

Valid cron models can therefore warn incorrectly, and the warning can still disagree with runtime validation.
  - confidence: 0.94
