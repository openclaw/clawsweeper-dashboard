## Summary

Keep this PR open, but it is not ready to merge: the warning is aimed at a real cron model-selection footgun, yet the patch validates against local CLI state and a partial resolver instead of the gateway/runtime path that will execute the cron.

## What This Changes

The PR adds a non-blocking `cron edit --model` allowlist warning by loading local config/catalog data before sending `cron.update`.

## Best Possible Solution

Add a gateway-owned cron model warning or preflight that uses the same runtime config, catalog, resolved defaults, and `resolveAllowedModelRef` path as cron execution, with focused regression coverage for remote-gateway and resolver edge cases.

## Reproduction Assessment

Yes, source-reproducible: current main accepts `cron edit --model` into `patch.payload.model`, while cron execution later rejects disallowed payload models through `resolveAllowedModelRef`. I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. The warning is useful, but this implementation reads local CLI config and partially reimplements model validation instead of using the gateway/runtime resolver that will execute the cron.
