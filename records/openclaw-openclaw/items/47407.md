## Summary

Keep open: current main still lacks an edit-time cron model allowlist warning, but this PR remains incorrect because it approximates runtime validation from the local CLI process and lacks after-fix real behavior proof.

## What This Changes

The PR adds a non-blocking `cron edit --model` warning by loading local config/catalog data and comparing the requested model against the allowed model set before sending `cron.update`.

## Best Possible Solution

Add a gateway-owned cron model warning or preflight that uses the same runtime resolver, config, catalog, defaults, and alias handling as cron execution, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main accepts `cron edit --model` into `patch.payload.model`, while cron execution later rejects disallowed payload models through `resolveAllowedModelRef`; I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. The warning is useful, but this implementation reads local CLI config and duplicates only part of runtime validation instead of using the selected gateway/runtime resolver that will execute the cron.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate against the selected gateway runtime:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: This warning reads local CLI config/catalog data, but `cron edit` sends `cron.update` to the selected gateway and supports `--url`/`--token`. In remote-gateway setups, local and gateway allowlists can differ, so the warning can be wrong or absent for the job that will actually run.
  - confidence: 0.93
- **[P2] Reuse the cron execution resolver for this warning:** `src/cli/cron-cli/register.cron-edit.ts:220-228`
  - body: The warning builds `allowedKeys` with hard-coded defaults and checks `parseModelRef(model, DEFAULT_PROVIDER)`. Cron execution uses `resolveAllowedModelRef` with configured defaults, aliases, provider inference, auth-profile suffix stripping, and compatibility aliases, so valid models can still warn and invalid ones can be missed.
  - confidence: 0.94
