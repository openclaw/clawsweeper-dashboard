## Summary

Keep open. The PR targets a real cron UX gap, but the branch still validates with local CLI config and partial model parsing, so it can warn incorrectly or miss the runtime failure it is meant to predict. Current main has fail-closed cron execution semantics but still does not provide the requested edit-time warning.

## What This Changes

The PR adds a non-blocking stderr warning in `openclaw cron edit --model` by loading local config/model catalog and checking the requested model against an allowed model set before calling `cron.update`.

## Best Possible Solution

Add a target-gateway validation path for `cron edit --model` that reuses the runtime model resolver with the gateway's config/catalog and resolved defaults, then keep the warning non-blocking if that remains the intended UX. Cover aliases, bare IDs, auth-profile suffixes, remote gateway edits, and current fail-closed cron execution tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by comparing the PR's `loadConfig()` plus `parseModelRef(DEFAULT_PROVIDER)` warning path with current cron execution's target-gateway `resolveAllowedModelRef` path; remote `--url`, aliases, bare IDs, and auth-profile suffixes are concrete mismatch cases.

## Solution Assessment

No. The warning direction is useful, but this implementation is not the best fix because it predicts a gateway runtime decision from local CLI state and a partial resolver instead of asking the same runtime contract that executes the cron job.

Valid cron models can therefore warn incorrectly, and the warning can still disagree with runtime validation.
  - confidence: 0.94
