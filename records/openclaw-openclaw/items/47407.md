## Summary

Keep open: current `main` still lacks the requested edit-time warning, and this PR’s local CLI prediction can disagree with cron runtime behavior because cron edits target a gateway while execution validates with gateway-side defaults and the shared model resolver. The PR also still lacks real behavior proof.

## What This Changes

The PR adds a non-blocking `openclaw cron edit --model` warning when the specified model is outside the configured model allowlist.

## Best Possible Solution

Add a gateway-backed validation or preview path that uses the same cron runtime resolver and target gateway config, with coverage for aliases, bare ids, auth-profile suffixes, configured defaults, and remote gateway edits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current `cron edit --model` stores the model without an allowlist warning, while cron execution later validates payload models through `resolveAllowedModelRef` and can reject them.

## Solution Assessment

No. The warning is useful, but this PR predicts runtime behavior from local CLI config and a partial parser instead of the target gateway’s runtime resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

- **[P2] Validate against the target gateway:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: This warning reads local CLI config before `cron.update`, but `cron edit` can target a remote gateway with `--url`/`--token`. If local and remote allowlists differ, the command can warn incorrectly or miss the warning for the job that will actually run.
  - confidence: 0.93

Valid cron models can warn incorrectly, and the warning can still disagree with runtime validation.
  - confidence: 0.94
