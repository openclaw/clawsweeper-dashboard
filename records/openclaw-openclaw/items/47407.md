## Summary

The PR targets a real source-reproducible cron/model UX gap, but the proposed patch is not merge-ready because it validates against local CLI config instead of the selected gateway runtime and still does not reuse the full runtime model resolver. It also lacks after-fix real behavior proof for the CLI warning.

## What This Changes

The PR adds a non-blocking `openclaw cron edit --model` warning when the requested model is outside the configured model allowlist.

## Best Possible Solution

Add a gateway-owned cron model warning or preflight that uses the same runtime config, catalog, resolved defaults, and `resolveAllowedModelRef` path as cron execution, with focused regression coverage for remote-gateway and resolver-parity cases.

## Reproduction Assessment

Yes, source-reproducible: current main accepts `cron edit --model` into `patch.payload.model`, while cron execution later rejects disallowed payload models through `resolveAllowedModelRef`. I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. The warning is useful, but this implementation validates in the local CLI process and partially reimplements model resolution instead of using the gateway/runtime resolver that will execute the cron.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate against the selected gateway runtime:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: This warning is calculated from `loadConfig()` in the CLI process, but `cron edit` sends `cron.update` to the gateway selected by `--url`/`--token`. In remote-gateway setups the local allowlist can differ from the runtime that owns the job, so this can both miss doomed cron edits and warn on edits that would run.
  - confidence: 0.93
- **[P2] Reuse the runtime model resolver:** `src/cli/cron-cli/register.cron-edit.ts:228-229`
  - body: This path parses the raw option with hard-coded `DEFAULT_PROVIDER` and checks that key against `allowedKeys`. Cron execution resolves configured defaults and then calls `resolveAllowedModelRef`, which handles aliases, provider inference, auth-profile suffixes, and OpenRouter compatibility aliases; this warning will drift from the actual error condition.
  - confidence: 0.94
