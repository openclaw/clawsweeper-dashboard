## Summary

Keep open: the requested warning is useful, but this PR still predicts cron runtime validity from local CLI config instead of the selected gateway/runtime resolver and it lacks after-fix real behavior proof.

## What This Changes

The PR adds a non-blocking `cron edit --model` warning by loading local config/catalog data and checking the requested model against a computed allowlist before `cron.update`.

## Best Possible Solution

Expose or reuse a target-gateway/runtime validation path so `cron edit --model` computes any warning with the same config, catalog, aliases, defaults, and allowlist resolver that will execute the cron.

## Reproduction Assessment

Yes. Source inspection shows current main stores `--model` in the cron payload without edit-time feedback, while cron execution later rejects invalid payload models through `resolveAllowedModelRef`; I did not run a live CLI command in this read-only review.

## Solution Assessment

No. The warning direction is useful, but this implementation is not the narrowest maintainable fix because it predicts runtime validity from local CLI config and hard-coded defaults instead of the selected gateway/runtime resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate the warning against the selected gateway:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: This warning reads local CLI config before `cron.update`, but `cron edit` can target a remote gateway via `--url`/`--token`. If local and remote allowlists differ, the command can warn incorrectly or miss the warning for the job that will actually run.
  - confidence: 0.93
- **[P2] Reuse the runtime model resolver:** `src/cli/cron-cli/register.cron-edit.ts:220-228`

Valid cron models can still warn incorrectly, and invalid ones can still disagree with runtime validation.
  - confidence: 0.94
