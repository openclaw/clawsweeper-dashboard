## Summary

Keep open: current main still lacks the requested edit-time warning, and this PR's warning can disagree with the gateway that will actually run the job and with the runtime model resolver.

## What This Changes

This PR adds a non-blocking `cron edit --model` warning in the CLI by checking the supplied model against a locally loaded allowed model set before `cron.update`.

## Best Possible Solution

Expose or reuse a target-gateway model validation path so `cron edit --model` can warn with the same config, defaults, aliases, and auth-profile handling used at cron execution time.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by comparing the PR's local `loadConfig()` plus `parseModelRef(DEFAULT_PROVIDER)` warning path with the current gateway-side `resolveAllowedModelRef` execution path.

## Solution Assessment

No. The warning direction is useful, but a local pre-RPC prediction is not the narrowest maintainable solution because it can disagree with the target gateway and runtime resolver.

- **[P2] Validate against the target gateway:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: The warning reads config from the local CLI process, but `cron edit` can update a remote gateway via `--url`/`--token`. If local and remote allowlists differ, the command can warn incorrectly or miss the warning for the job that will actually run.
  - confidence: 0.93

  - confidence: 0.94
