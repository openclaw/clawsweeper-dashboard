## Summary

Keep open: current main still lacks an edit-time allowlist warning, but this PR predicts runtime validation from local CLI state and a partial resolver, so it can disagree with the gateway that will actually execute the cron job.

## What This Changes

The PR adds a non-blocking `cron edit --model` warning by loading config/model catalog in the CLI and checking the supplied model against the allowed model set before `cron.update`.

## Best Possible Solution

Add a gateway-backed validation or preview path that uses the same runtime model resolver and target gateway config as cron execution, with focused coverage for aliases, bare ids, auth-profile suffixes, and remote gateway edits.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by comparing the PR's local `loadConfig()` plus `parseModelRef(DEFAULT_PROVIDER)` warning path with current cron execution's gateway-side `cfgWithAgentDefaults` plus `resolveAllowedModelRef` path.

## Solution Assessment

No. The warning is useful, but this implementation is not the best fix because it predicts runtime behavior from the wrong config context and does not fully reuse the runtime resolver contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate against the target gateway:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: The warning reads local CLI config before sending `cron.update`, but this command can edit a remote gateway via `--url`/`--token`. If local and remote allowlists differ, the command can warn incorrectly or miss the warning for the job that will actually run.
  - confidence: 0.93
- **[P2] Reuse the runtime model resolver:** `src/cli/cron-cli/register.cron-edit.ts:227-228`

Valid cron models can warn incorrectly, and the warning can still disagree with runtime validation.
  - confidence: 0.94
