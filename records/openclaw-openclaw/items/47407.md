## Summary

Keep open: the requested cron warning is still not on current main, but this PR should not merge as-is because it validates with local CLI config and a partial parser instead of the selected gateway/runtime resolver, and it still lacks real behavior proof.

## What This Changes

The PR adds a non-blocking `cron edit --model` allowlist warning in `src/cli/cron-cli/register.cron-edit.ts` by loading local config/catalog data before `cron.update`.

## Best Possible Solution

Add a gateway-owned cron model warning or preflight that uses the same runtime config, catalog, defaults, and `resolveAllowedModelRef` path as cron execution, with focused regression coverage for remote-gateway and alias/default-provider cases.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts `cron edit --model` into `patch.payload.model`, while cron execution later rejects disallowed payload models through `resolveAllowedModelRef`; I did not run a live CLI repro in this read-only review.

## Solution Assessment

No. The warning is useful, but this implementation reads local CLI config and partially reimplements model validation instead of using the gateway/runtime resolver that will execute the cron.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

  - body: This warning reads `loadConfig()` in the local CLI process, but `cron edit` sends `cron.update` to the gateway chosen by `--url`/`--token`. If configs differ, the command can miss a cron that will fail or warn for one that will run.
  - confidence: 0.93

  - confidence: 0.94
