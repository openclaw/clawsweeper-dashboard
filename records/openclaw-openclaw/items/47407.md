## Summary

Keep open. The requested edit-time warning is still useful and is not implemented on current main, but this PR’s current implementation is not safe to merge because it predicts cron execution from local CLI config and a simplified model parser instead of the target gateway/runtime validation contract.

## What This Changes

The PR adds a non-blocking stderr warning in `openclaw cron edit --model` by loading local config/model catalog and checking the parsed model against the allowed model set before sending `cron.update`.

## Best Possible Solution

Add a gateway/runtime-backed validation path for `cron edit --model` so the warning uses the same target-gateway config and `resolveAllowedModelRef` semantics as cron execution, including aliases, bare refs, configured defaults, auth-profile suffixes, and remote gateway operation.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by comparing the PR’s local `loadConfig()` plus `parseModelRef()` warning path with cron execution’s target-gateway `resolveAllowedModelRef` path; remote gateway configs, aliases, bare model IDs, and auth-profile-suffixed refs are concrete mismatch cases.

## Solution Assessment

No. A warning is the right user-facing direction, but this implementation is not the best fix because it uses local CLI state and partial parsing instead of the gateway/runtime contract that actually decides whether the cron run will fail.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate against the target gateway config:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: The warning loads local config and catalog data before sending `cron.update`, but `cron edit` can target a remote gateway via `--url` and `--token`. When local and remote allowlists differ, the command can warn for a model that will run or stay quiet for one the gateway will reject, making the new feedback unreliable for remote edits.
  - confidence: 0.93
- **[P2] Reuse the runtime model resolver for warnings:** `src/cli/cron-cli/register.cron-edit.ts:227-228`
  - body: The check builds an allowed set but parses the raw CLI string with hard-coded defaults. Cron execution calls `resolveAllowedModelRef` with runtime-resolved defaults, alias resolution, bare-provider inference, and auth-profile suffix stripping, so valid aliases, bare allowlisted IDs, or `@profile` suffixed refs can receive false warnings.
  - confidence: 0.94
