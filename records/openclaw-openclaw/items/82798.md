## Summary

This PR should stay open: it is a focused fix for a real CLI SecretRef bug, but the current patch still lets provider-scoped web commands ask the gateway for broad target IDs while applying the path scope only afterward. That leaves a correctness gap for provider overrides and a concrete secrets-boundary concern before merge.

## What This Changes

The PR resolves local infer command SecretRefs before provider-backed execution, adds provider-scoped web search/fetch secret target selection, updates web search config snapshot selection, and adds regression tests plus a changelog entry.

## Best Possible Solution

Keep the provider-scoped resolution direction, but make the gateway resolution honor the selected allowed paths before resolving or applying assignments, then cover provider override and web-fetch fallback cases against the real gateway snapshot path.

## Reproduction Assessment

Yes. The linked issue provides a concrete infer web search command and current-main source shows that local web search/fetch read raw getRuntimeConfig before command SecretRef resolution.

## Solution Assessment

No. Resolving local infer SecretRefs before provider construction is the right direction, but this patch scopes allowed paths only after the gateway has resolved and returned broad target assignments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope gateway secret resolution before resolving web targets:** `src/cli/capability-cli.ts:1623-1624`
  - body: The new web command path passes provider-scoped `allowedPaths`, but `secrets.resolve` still receives only broad `targetIds`, and the client applies all returned assignments before the allowed-path analysis. With `--provider tavily` while the gateway snapshot is configured for another web provider, this can resolve/apply the wrong provider secret and leave the selected provider SecretRef treated as inactive rather than resolved. The gateway request or fallback path needs to carry the selected paths before any secret is resolved or assigned.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff narrows web command targets locally, but the gateway RPC can still resolve and return non-selected provider secrets before local filtering.

Concerns:

- **[medium] Gateway can over-resolve non-selected provider secrets:** `src/cli/capability-cli.ts:1623`
  - body: For provider-scoped web commands, `allowedPaths` is not sent to `secrets.resolve`; broad target IDs can cause the active gateway snapshot to return another provider's secret, which is then applied to the resolved config before scoped analysis. This weakens the intended selected-provider secret boundary.
  - confidence: 0.84
