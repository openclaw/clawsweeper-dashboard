## Summary

Keep open: the heartbeat leak is source-reproducible on current main, but this PR is not merge-ready because it lacks real behavior proof and still has correctness plus stale-architecture blockers.

## What This Changes

The PR adds heartbeat-mode mid-text `HEARTBEAT_OK` stripping and tests, plus older MiniMax provider-capability and transcript-policy context plumbing.

## Best Possible Solution

Refresh this into a focused current-main heartbeat change that strips all heartbeat-mode tokens from normalized text before the ack-length gate, preserves message-mode behavior, updates the heartbeat contract docs, and leaves MiniMax replay policy in provider hooks.

## Reproduction Assessment

Yes, from source inspection. Current main leaves a middle-only `HEARTBEAT_OK` unchanged, and heartbeat chat finalization calls that helper before suppressing or delivering output.

## Solution Assessment

No. The PR targets the right helper, but it should remove all heartbeat-mode tokens from normalized text, drop the stale provider-capability changes, and align the heartbeat docs before merge.

## Review Findings

Full review comments:

- **[P2] Remove remaining heartbeat tokens after edge stripping:** `src/auto-reply/heartbeat.ts:156-168`
  - body: The fallback only runs when `picked.didStrip` is false. If a heartbeat reply has one token at an edge and another in the middle, edge stripping sets `didStrip`, this branch is skipped, and a remaining `HEARTBEAT_OK` can still be delivered when the surrounding content exceeds `ackMaxChars`.
  - confidence: 0.88
- **[P2] Strip mid-text tokens from normalized text:** `src/auto-reply/heartbeat.ts:163-167`
  - body: The fallback splits raw `trimmed`, so wrapped middle tokens such as `<b>HEARTBEAT_OK</b>` or `**HEARTBEAT_OK**` can leave wrapper artifacts in delivered text and in the `ackMaxChars` gate. Use the normalized markup-stripped path before heartbeat-mode removal.
  - confidence: 0.85
- **[P2] Drop core MiniMax replay fallback:** `src/agents/provider-capabilities.ts:53-58`
  - body: Current main resolves replay policy through provider runtime plugins, and MiniMax already registers provider-owned replay hooks. Carrying this old core fallback forward would reintroduce provider-specific policy into core; remove the stale MiniMax/provider-capability commits when refreshing the branch.
  - confidence: 0.82
