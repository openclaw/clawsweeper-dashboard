## Label Justifications

- `P1`: The issue reports an active cross-channel workflow leak where real users can receive assistant intermediate text before the intended final response.
- `impact:message-loss`: The affected delivery path can send unintended intermediate messages and confuse or replace the intended final channel response.
- `impact:security`: Issue comments describe internal reasoning, operational context, and customer-facing intermediate content leaking to messaging recipients.

## Summary

Keep open: current main has partial safeguards for phased commentary, final-answer payloads, and message-tool-only opt-in, but ordinary unphased text can still be buffered and flushed before tool execution under automatic channel delivery, and cron direct delivery still preserves all successful text payloads.

## What This Changes

The issue asks OpenClaw to stop exposing assistant text produced between tool calls as visible messaging-channel replies.

## Best Possible Solution

Adopt a generic delivery-boundary policy that prevents unphased pre/inter-tool assistant text from reaching messaging channels while preserving explicit message-tool sends, final answers, errors, and documented fallback behavior with upgrade-aware tests.

## Reproduction Assessment

Yes at source level: current tests and handlers show buffered unphased assistant text is flushed before tool execution under automatic delivery, though no live transport reproduction was run in this read-only review.

## Solution Assessment

Not yet: existing commentary/final-answer filtering and message-tool-only options are useful partial fixes, but they do not close the default automatic delivery leak across source channels and cron variants.
