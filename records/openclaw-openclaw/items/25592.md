## Label Justifications

- `P1`: The issue affects real messaging-channel workflows where internal assistant text can be delivered to users during normal tool-heavy agent runs.
- `impact:message-loss`: The problem is incorrect message delivery: unintended intermediate assistant text appears as user-visible channel output.
- `impact:security`: Multiple reports describe internal reasoning, raw tool-call-like payloads, customer context, or operational traces crossing into external messaging channels.

## Summary

Keep open: current main has partial safeguards for commentary-phased and explicit message-tool-only paths, but the central cross-channel leak is still represented by source-level automatic direct/source reply behavior, cron announce payload selection that preserves multiple successful text payloads, and recent production reports plus open unmerged closing PRs.

## What This Changes

The issue reports that assistant text emitted before or between tool calls can be delivered as visible Slack, Telegram, iMessage, WhatsApp, or Discord channel messages instead of staying internal.

## Best Possible Solution

Keep this as the canonical bug and have maintainers choose a compatibility-safe delivery policy that suppresses non-final/unphased assistant text for channel-bound turns, with regression coverage for automatic direct chats, message_tool_only replies, cron announce delivery, commentary/final_answer phases, and failure fallback.

## Reproduction Assessment

Yes for the broad defect class from source inspection: current main still has automatic source-reply paths for direct/source chats and cron delivery can preserve multiple successful text payloads. I did not run a live Slack, Telegram, WhatsApp, or Discord repro in this read-only cleanup review.

## Solution Assessment

No current implementation is not yet the complete fix. The best remaining solution needs maintainer policy judgment because the most direct fixes change compatibility-sensitive visible-reply defaults and cron announce semantics.
