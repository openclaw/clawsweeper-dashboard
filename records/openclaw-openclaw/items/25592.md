## Summary

Keep open. Current main has important commentary-phase safeguards, but direct/source chats still default to automatic delivery and unphased assistant text can still enter the block-reply buffer and flush before tool execution; live GitHub state also shows open closing PRs for this issue.

## What This Changes

The issue asks OpenClaw to prevent assistant text produced before or between tool calls from being delivered as standalone messages to Slack, Telegram, iMessage, WhatsApp, and similar channels.

## Best Possible Solution

Keep this as the canonical channel-leak bug, review or supersede the two open implementation PRs, and land a maintainer-approved final-only or explicit-send policy with regression coverage for unphased pre-tool and inter-tool text.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: direct/source chats can still default to automatic delivery, unphased assistant text enters the block buffer, and a current test asserts that pre-tool buffered text is delivered when tool execution starts.

## Solution Assessment

No. Current main is only a partial solution; closing now would leave the automatic direct/source delivery path and unphased pre-tool text policy unresolved while open closing PRs still need maintainer disposition.
