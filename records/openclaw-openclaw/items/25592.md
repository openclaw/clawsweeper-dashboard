## Summary

Keep open. Current main and the latest release still leave automatic source replies and unphased assistant text paths that can surface pre-tool or inter-tool text to messaging recipients, while the linked implementation PRs remain open and only cover parts of the broader policy problem.

## What This Changes

Suppress assistant text produced before or between tool calls from external messaging channels unless it is the final intentional reply or an explicit message-tool send.

## Best Possible Solution

Keep this as the canonical channel-leak bug and land a maintainer-approved final-only or explicit-send delivery policy that preserves existing commentary/final-answer safeguards, with regression coverage and live transport proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: automatic source replies remain possible, unphased assistant text enters the block reply buffer, and a current test expects buffered pre-tool text to be delivered on tool execution start.

## Solution Assessment

No. Current main has important partial mitigations, but the best fix still requires a maintainer policy decision for direct/source delivery and unphased assistant text before or between tool calls.
