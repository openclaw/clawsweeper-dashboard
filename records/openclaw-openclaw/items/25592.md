## Label Justifications

- `P1`: The issue describes repeated user-visible leakage of internal assistant output into real messaging channels across production-style setups.
- `impact:message-loss`: The affected behavior sends messages that should remain internal, causing wrong or duplicate channel-visible replies.
- `impact:security`: Leaked assistant text can expose internal reasoning, tool traces, customer context, and operational details.
- `maturity:stable`: The cross-channel issue includes Discord reports, and the checked-in maturity scorecard marks Discord as M4 Stable.

## Summary

Keep open: current main and the latest release still allow automatic source delivery and still flush unphased buffered assistant text before tool execution, so the central leak class is not implemented away. Existing partial fixes and open PRs cover narrower variants, but they do not replace the canonical cross-channel policy bug.

## What This Changes

The issue asks OpenClaw to prevent assistant text produced before or between tool calls from being sent as visible Slack, Telegram, iMessage, WhatsApp, Discord, or similar channel messages.

## Best Possible Solution

Keep this as the canonical bug and have maintainers choose an upgrade-safe delivery policy, then land one focused fix that suppresses non-final/internal assistant text while preserving intentional message-tool sends and adding regression coverage for automatic and message-tool-only modes.

## Reproduction Assessment

Yes for source-level reproduction: current tests and handlers show ordinary assistant text can be buffered and flushed before a tool call when automatic delivery is active. I did not run a live Slack, Telegram, or Discord repro in this read-only review.

## Solution Assessment

Unclear as a policy choice: explicit send/message-tool-only semantics look safest for leakage, but changing automatic direct or group delivery defaults has compatibility risk. The best fix needs maintainer-selected upgrade behavior plus focused regression and live channel proof.
