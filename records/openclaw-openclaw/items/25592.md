## Label Justifications

- `P1`: Multiple recent production reports show an active cross-channel workflow regression exposing unintended assistant output to real users.
- `impact:message-loss`: The issue causes unintended, duplicated, or misclassified channel messages during tool-running turns.
- `impact:security`: Internal reasoning, operational narration, and raw tool payload content can be exposed to unintended channel participants.

## Summary

Current main contains important partial safeguards, but the canonical bug remains: direct chats can still use automatic bare-text delivery, unphased assistant chunks still enter block-reply delivery, and cron payload selection does not carry or filter reasoning state. Recent production reports and an open narrower implementation PR confirm that this issue remains necessary; resolving it requires an explicit compatibility decision rather than cleanup.

## What This Changes

Prevent intermediate assistant narration, reasoning, and tool-envelope text emitted around tool calls from becoming user-visible channel messages.

## Best Possible Solution

Keep this issue as the canonical cross-channel bug and define an upgrade-safe final-only delivery contract covering unphased text, reasoning payloads, direct chats, cron/announce, aborted turns, and streaming/final assembly, with focused regression tests and real channel proof for each affected path.

## Reproduction Assessment

Yes, at source level. Current main still combines automatic direct-chat delivery, unphased block-reply buffering, and cron payload selection without reasoning filtering; this read-only review did not perform a live channel run.

## Solution Assessment

No single proposed fix is sufficient. Explicit-send-only everywhere would break an established direct-chat contract, while commentary- or provider-only filtering misses other paths; maintainers should first choose a unified, upgrade-safe final-delivery policy.
