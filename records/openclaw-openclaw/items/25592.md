## Summary

Keep open: current main still has source-level paths where unphased assistant text can be buffered or delivered automatically, cron announce output can preserve multiple successful text payloads, and recent production comments plus open linked candidate PRs show the central leakage problem is still active.

## What This Changes

The issue reports that assistant text emitted before or between tool calls can be delivered as visible messages in Slack, Telegram, iMessage, Discord, WhatsApp, and cron announce paths instead of staying internal.

## Best Possible Solution

Keep this as the canonical product bug until maintainers choose and land a direct-chat, group/channel, and cron-safe final-only or message-tool-only delivery policy with regression coverage for unphased inter-tool text and existing automatic-reply compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: automatic visible-reply modes plus unphased assistant text buffering or cron multi-payload delivery can still produce visible non-final text; I did not run a live Slack/Telegram/Discord reproduction in this read-only review.

## Solution Assessment

Unclear. Suppressing all bare text would address the leak but is compatibility-sensitive, so the best fix needs a maintainer-approved policy that preserves or migrates existing automatic final-reply behavior while blocking internal/inter-tool text.
