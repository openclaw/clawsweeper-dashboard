## Label Justifications

- `P1`: Fresh production reports show unintended channel messages during normal interactive and autonomous workflows across major transports.
- `impact:message-loss`: The delivery pipeline misroutes intermediate content as extra outbound messages, corrupting the intended final-message flow.
- `impact:security`: Reported messages expose private reasoning, internal metadata, placeholders, and raw tool-call envelope content to channel recipients.

## Summary

Current main contains meaningful partial safeguards, but the issue remains necessary: direct replies can still be automatic, cron delivery preserves multiple deliverable payloads without a reasoning field, and fresh production reports cover paths not resolved by the phase-aware commentary guard. Open implementation PRs are candidates, not a merged canonical fix.

## What This Changes

Prevent intermediate assistant narration, reasoning, placeholders, and tool-envelope text from becoming user-visible messages during multi-tool channel workflows.

## Best Possible Solution

Adopt one canonical final-user-output invariant across interactive, streaming, aborted/yielded, and cron/isolated delivery, preserve explicit message-tool sends, and handle any direct-chat default change through a documented compatibility path with cross-channel regression coverage.

## Reproduction Assessment

Yes at source level: concrete multi-tool and cron traces map to current automatic direct-reply behavior, phase-dependent embedded filtering, and cron payload selection that lacks isReasoning. A live transport reproduction was not run in this read-only review.

## Solution Assessment

No single proposed approach is yet the best complete fix; prompt-level final tags, verbose-stream gating, and turn buffering each address only part of the affected surface, while the durable solution must enforce one output classification invariant across all delivery paths.
