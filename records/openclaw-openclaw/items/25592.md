## Label Justifications

- `P1`: The issue describes repeated production-visible leaks across messaging channels and cron announce flows that can expose internal agent text to real users.
- `impact:security`: Leaked reasoning, raw tool envelopes, execution traces, and private operational commentary can expose sensitive internal state in external channels.
- `impact:message-loss`: The failure is in channel delivery semantics: extra internal messages are delivered as user-visible replies and can also confuse the intended final-message flow.

## Summary

Keep open: current main has useful partial guards for commentary/final-answer phases, but the central channel-leak problem is still live through automatic visible-reply defaults, unphased pre-tool text buffering, and cron announce payload selection that does not carry reasoning metadata.

## What This Changes

The issue asks OpenClaw to stop delivering assistant text produced before or between tool calls as visible Slack, Telegram, Discord, iMessage, WhatsApp, or cron announce messages unless the output is an intentional final/source-visible reply.

## Best Possible Solution

Keep this as the canonical bug, review or close the linked implementation PRs, and choose a compatibility-safe source-reply policy with regression coverage for automatic direct/group/channel delivery, unphased pre-tool text, message-tool-only turns, and cron announce payloads.

## Reproduction Assessment

Yes at source level: current main has automatic source-reply defaults and a focused test proving pre-tool buffered text is flushed as a block reply before tool execution. I did not run a live Slack/Telegram/Discord reproduction in this read-only review.

## Solution Assessment

No, closing is not the best solution yet; the best path is a maintainer-selected delivery policy and review of the linked implementation PRs because the remaining behavior is compatibility-sensitive. A new ad hoc repair PR would be less useful while those candidates and product decisions are still open.
