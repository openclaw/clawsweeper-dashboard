## Summary

Keep open: current main has important phase-aware safeguards, but source, tests, docs, recent production comments, and open/conflicting implementation candidates show the broader channel-leak problem is not fully implemented on main or safely closeable.

## What This Changes

The issue asks OpenClaw to stop delivering assistant text produced before or between tool calls as visible Slack, Telegram, iMessage, WhatsApp, Discord, or cron announce messages unless it is an intentional final or explicit send.

## Best Possible Solution

Keep this as the canonical cross-channel leak bug, choose a maintainer-approved direct/source reply policy, and add generic regression coverage for unphased pre-tool text, message-tool sends, final-only behavior, reasoning payloads, and cron announce delivery before closing.

## Reproduction Assessment

Yes at source level, but not from a live channel run in this read-only review. Current tests assert buffered pre-tool text is flushed, current cron selection drops the reasoning marker, and recent comments report the same class on shipped 2026.6.x builds.

## Solution Assessment

No, the issue is not ready to close and the exact explicit-send direction still needs maintainer policy judgment. The best fix is a generic delivery-boundary rule that preserves compatible final replies where intended while suppressing unphased, reasoning, and inter-tool text across interactive and cron paths.
