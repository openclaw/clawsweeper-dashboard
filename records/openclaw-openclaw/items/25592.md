## Label Justifications

- `P1`: The issue describes current production leakage of internal assistant text into user-facing messaging channels across multiple transports.
- `impact:security`: Internal reasoning, tool payloads, and operational text can become visible to external channel users.
- `impact:message-loss`: The delivery path can emit unintended extra channel messages and obscure or duplicate the intended final reply.

## Summary

Keep open: current main still has source-level delivery paths where unphased assistant text can be buffered and delivered before tool execution, automatic visible replies remain the documented default for normal group/channel requests, and cron announce payload selection still does not carry or filter reasoning metadata. Recent June 2026 production reports and protected product/security labels make this still necessary rather than maintainer cleanup.

## What This Changes

The issue requests that assistant text produced before or between tool calls stay internal instead of being routed as visible Slack, Telegram, Discord, iMessage, WhatsApp, or cron announce messages.

## Best Possible Solution

Keep this as the canonical security/message-delivery bug while maintainers choose one visible-reply contract and implement it across automatic replies, message-tool-only runs, unphased inter-tool text, reasoning payloads, and cron announce delivery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction model: automatic visible replies remain possible, unphased pre-tool text is buffered and flushed to onBlockReply, and cron deliverable payload selection does not filter reasoning. I did not run a live Telegram, Discord, or Slack reproduction because this was a read-only review.

## Solution Assessment

No, current main is not the best final state yet; it contains important partial fixes but still needs a maintainer delivery-contract decision and cross-path regression coverage before this can close.
