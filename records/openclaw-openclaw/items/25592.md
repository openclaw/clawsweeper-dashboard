## Summary

Keep open. Current main and the latest release still contain automatic visible source-delivery paths plus an unphased assistant-text buffer/flush path that can deliver pre-tool or inter-tool text, and open implementation PRs are still linked to the same problem.

## What This Changes

The issue asks OpenClaw to prevent assistant text produced before or between tool calls from being delivered as visible messages in Slack, Telegram, iMessage, WhatsApp, and similar channels.

## Best Possible Solution

Keep this as the canonical channel-leak bug until maintainers choose and land a final-only or explicit-send delivery policy, with regression coverage for unphased pre-tool/inter-tool text and live transport proof for the selected behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: automatic source replies remain supported, unphased assistant text still enters the block reply buffer, and a current test expects buffered pre-tool text to be delivered on tool execution start.

## Solution Assessment

No. Current main has important partial mitigations, but the best fix still requires a maintainer-approved policy for unphased assistant text and automatic source delivery before or between tool calls.
