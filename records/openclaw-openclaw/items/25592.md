## Summary

Keep open. Current main has useful partial fixes, including final-payload collapse and commentary-phase suppression, but source inspection still shows an automatic direct/source delivery path where unphased assistant text can be buffered and flushed to channels before a tool call; two relevant closing PRs also remain open.

## What This Changes

The issue asks OpenClaw to stop sending assistant text produced before or between tool calls as visible Slack, Telegram, iMessage, WhatsApp, and similar channel messages.

## Best Possible Solution

Keep this as the canonical channel-leak bug, review the two open implementation PRs, and choose a direct/source-safe final-only or explicit-send policy with regression coverage for unphased assistant text before and between tool calls.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: direct chats can still default to automatic delivery, unphased assistant text is buffered, and the current pre-tool flush test asserts that buffered text is delivered when tool execution starts.

## Solution Assessment

No. Current main is better than older releases, but closing now would leave the direct/source automatic delivery and unphased pre-tool text policy unresolved; a maintainer should choose the compatibility path before landing a broad behavior change.
