## Summary

Keep open. Current main and v2026.5.20 still allow automatic source replies and unphased assistant text to enter block-reply delivery, while the directly relevant implementation PRs remain open and the remaining policy has compatibility impact.

## What This Changes

The issue requests suppressing bare assistant text emitted before or between tool calls so only intentional final or message-tool output reaches messaging channels.

## Best Possible Solution

Keep this as the canonical channel-leak bug and land a maintainer-approved final-only or explicit-send delivery policy with regression coverage and live transport proof.

Yes. A high-confidence source reproduction exists: automatic source replies remain possible, unphased assistant text enters the block reply buffer, and a current test expects buffered pre-tool text to be delivered on tool execution start.

## Solution Assessment

No. Current main has useful partial mitigations, but the complete fix still needs a maintainer policy decision for direct/source delivery and unphased assistant text before or between tool calls.
