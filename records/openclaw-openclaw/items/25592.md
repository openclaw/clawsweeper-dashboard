## Label Justifications

- `P1`: The issue affects real external messaging workflows and recent production reports show internal reasoning/tool text reaching users in Slack, Telegram, WhatsApp, and Discord.
- `impact:security`: The reported failures can expose internal reasoning, tool payloads, routing markers, and operational metadata to external channel recipients.
- `impact:message-loss`: The delivery pipeline can misroute intermediate assistant text as user-visible channel messages instead of keeping it internal.

## Summary

Keep open. Current main and v2026.6.5 still have automatic source-reply paths that can deliver unphased assistant text, plus cron delivery still drops reasoning metadata before selecting deliverable payloads; the issue also has recent production reports and open linked implementation candidates, so it is not safe to close.

## What This Changes

The issue asks OpenClaw to stop routing intermediate assistant text, reasoning, commentary, and raw tool-call-like text to external messaging channels before the final intentional reply.

## Best Possible Solution

Keep this as the canonical product/security bug until maintainers choose a direct-chat and cron-safe delivery policy, review the linked PRs, and add regression coverage for unphased pre-tool/inter-tool text plus reasoning payloads across streaming, final assembly, and cron announce delivery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: automatic source replies can still deliver unphased assistant text, and cron payload selection does not carry isReasoning. I did not run a live Slack/Telegram/Discord scenario in this read-only review.

## Solution Assessment

Unclear: current main has useful partial fixes, but the remaining best fix requires a maintainer policy decision for direct chats, normal group replies, and cron announce delivery rather than another one-sided suppression patch.
