## Label Justifications

- `P1`: The issue affects real external messaging workflows and has fresh production reports of internal text leaking into user-visible channels.
- `impact:security`: Reports include internal reasoning, raw tool-call markers, and command-like payloads appearing in external chats.
- `impact:message-loss`: The defect misroutes non-final assistant text into messaging channels as extra visible messages.

## Summary

Keep open. Current main and the latest stable release still allow automatic source replies and unphased assistant text paths that match the reported leak, and this issue has fresh production reports plus open linked implementation PRs that are not merged.

## What This Changes

The issue asks OpenClaw to stop sending intermediate assistant text between tool calls to external messaging channels and deliver only intentional final or explicit message-tool output.

## Best Possible Solution

Keep this as the canonical core delivery bug, choose a direct-chat-safe policy for final-only or message-tool-only delivery, and add regression coverage for unphased assistant text, commentary/reasoning payloads, cron announce delivery, and external channel paths.

## Reproduction Assessment

Yes. Source inspection shows automatic source-reply paths still exist and unphased assistant chunks can be buffered and flushed before tool execution; I did not run a live Slack/Telegram/Discord reproduction in this read-only review.

## Solution Assessment

Unclear as a single implementation choice. Explicit-send semantics would be robust, but the safer maintainer path is to choose a compatibility-aware delivery policy and land focused guards/tests across interactive and cron delivery rather than closing from partial fixes.
