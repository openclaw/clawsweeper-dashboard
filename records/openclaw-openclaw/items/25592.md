## Summary

Keep open. Current main still has automatic direct/source reply paths and a source-level path where unphased assistant text is buffered and flushed before tool execution; the issue also has active linked implementation PRs that need maintainer disposition.

## What This Changes

The issue asks OpenClaw to prevent intermediate assistant text between tool calls from being delivered as visible messages in Slack, Telegram, iMessage, WhatsApp, and similar channels.

## Best Possible Solution

Keep this as the canonical channel-leak bug and land a maintainer-approved final-only or explicit-send policy, with regression coverage for unphased pre-tool/inter-tool text and live transport proof for the selected behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: direct/source chats can still default to automatic delivery, unphased assistant text enters the block buffer, and the current test expects buffered pre-tool text to be delivered when tool execution starts.

## Solution Assessment

No. Current main is only a partial mitigation; resolving this needs a maintainer policy decision around direct/source delivery plus disposition of the linked implementation PRs.
