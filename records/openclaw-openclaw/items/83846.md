## Summary

Keep open: this PR targets a real WhatsApp diagnostics bug and now has copied installed-build output, but the patch still derives key diagnostics from `groups` alone and can misstate disabled or fail-closed group-policy states.

## What This Changes

This PR changes WhatsApp Web startup and unregistered-group drop diagnostics, adds a small listener-log helper with tests, and updates WhatsApp troubleshooting docs.

## Best Possible Solution

Make the listener and drop diagnostics use the effective resolved WhatsApp group policy facts, including disabled, allowlist, wildcard, sender-filter bypass, and account-scoped paths, while keeping runtime admission behavior unchanged.

## Reproduction Assessment

Yes. Source inspection shows current main still emits the misleading personal-only startup log while the existing group-policy source and tests show group observation and blocking states; I did not run a live WhatsApp session in this read-only review.

## Solution Assessment

No, not as patched. The PR targets the right logs/docs surface, but the maintainable fix should format diagnostics from the same resolved policy facts used by routing rather than from `groups` alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reflect disabled and fail-closed policies in startup logs:** `extensions/whatsapp/src/auto-reply/monitor/listener-log.ts:6-8`
  - body: `formatWhatsAppInboundListeningLog` only looks at `account.groups`, so configs with `groupPolicy: "disabled"` or `groupPolicy: "allowlist"` without groups or `groupAllowFrom` can still log that all groups are in scope. That replaces one misleading startup diagnostic with another in supported policy states; please derive this from the resolved group policy facts or use wording that does not overclaim admission.
  - confidence: 0.9
- **[P2] Avoid groups-only remediation for disabled drops:** `extensions/whatsapp/src/auto-reply/monitor/group-gating.ts:117-119`
  - body: This drop message tells operators to add the group or `"*"` to `channels.whatsapp.groups`, but `resolveChannelGroupPolicy` also returns disallowed for `groupPolicy: "disabled"`, including cases where the group or wildcard is already configured. The diagnostic should name the disabled policy case instead of sending users to the wrong config key.
  - confidence: 0.84
