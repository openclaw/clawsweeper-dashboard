## Summary

Keep open. Current main still has only adjacent auth/control-plane rate limiting, reactive provider cooldown/fallback behavior, and usage readouts; it does not implement configurable outbound API throttling, pre-dispatch queue/block behavior, token or cost budget enforcement, warning thresholds, or an `openclaw limits` CLI.

## What This Changes

Request configurable rate limiting, throttling, budget controls, observability, and CLI status/configuration for external API calls across providers and integrations.

## Best Possible Solution

Keep this as the canonical core design tracker and scope an opt-in outbound policy layer starting with model-provider dispatch, with explicit follow-ups for tools/search, channels/plugins, persistence, warnings, status/CLI UX, 429 backoff, and budgets.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/design request rather than a concrete runtime bug; the high-confidence gap check is source inspection of config, schema, transport, CLI, usage, and related issue/PR context.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The requested direction is valid, but implementing every proposed surface at once is broad; the safer maintainable path is a maintainer-reviewed core policy design with a narrow first enforcement boundary.
