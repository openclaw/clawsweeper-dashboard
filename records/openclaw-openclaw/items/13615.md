## Summary

Current main still lacks a unified configurable outbound rate-limit, queue, budget-enforcement, warning, and status/CLI capability. Existing code provides narrower reactive provider retry/fallback, usage reporting, gateway protection, and channel-specific mechanisms, while the live discussion keeps this as the canonical broad product-design tracker; the remaining work needs maintainer scoping rather than cleanup or an autonomous fix PR.

## What This Changes

Add configurable request and token throttling, cost-budget enforcement, warnings, observability, and CLI controls for model providers and other external API calls.

## Best Possible Solution

Define an opt-in model-provider dispatch policy as the first bounded core phase, using canonical SQLite-owned counters and explicit queue/block behavior; separately decide plugin/tool and channel transport hooks, then align schema, doctor/docs, status UX, and focused fresh-install and upgrade validation.

## Reproduction Assessment

Not applicable. This is a request for new cross-cutting operator controls; current-main source inspection provides high-confidence capability-gap evidence rather than a reproduction of broken documented behavior.

## Solution Assessment

No, not as one undifferentiated limiter for every external API. The maintainable path is to scope model-provider dispatch first, then add explicit owner hooks or separate policies for tools and channel transports only where their contracts require them.
