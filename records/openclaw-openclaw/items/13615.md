## Summary

Keep open. Current main still has reactive cooldown/failover/retry handling and usage readouts, but it does not implement configurable pre-dispatch outbound rate limits, queueing, token/cost budget enforcement, warning/status controls, or an `openclaw limits` CLI; the issue discussion also treats this as the canonical core tracker.

## What This Changes

The issue requests configurable rate limiting, throttling/backoff, cost budgets, observability, and `openclaw limits` CLI controls for outbound external API calls.

## Best Possible Solution

Keep this as the canonical design tracker for an opt-in outbound policy layer, starting with model-provider dispatch and explicitly scoping follow-ups for tools, channels/plugins, persistence, status UX, 429 backoff, warning thresholds, and token/cost budgets.
