## Summary

Keep open. Current main has adjacent reactive cooldown/fallback handling, provider usage readouts, and gateway/control-plane rate limits, but it still lacks the requested configurable outbound API rate limits, pre-dispatch queue/block behavior, token or cost budget enforcement, warning thresholds, and `openclaw limits` CLI. The existing maintainer discussion also treats this as the canonical broad tracker rather than work already shipped or superseded.

## What This Changes

The issue requests configurable rate limiting, throttling, cost controls, observability, and CLI status commands for outbound external API calls across model providers, web search, and messaging integrations.

## Best Possible Solution

Keep this as the canonical core design tracker. The desired end state is a maintainer-scoped, opt-in outbound policy layer that starts with model-provider dispatch, explicitly defines which tool/search/channel/plugin calls are in scope, and aligns schema, persistence, status/CLI UX, warning thresholds, 429 backoff behavior, token/cost budgets, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature/design request rather than a concrete runtime bug. The high-confidence gap check is current-main inspection of config types/schema, provider transport, CLI/docs surfaces, reactive fallback behavior, and related issue/PR context.

## Solution Assessment

Unclear. The requested direction is valid, but implementing every proposed surface at once is broad; the safer maintainable path is a core policy design with a narrow first enforcement boundary, likely model-provider dispatch, plus explicit follow-ups for other outbound surfaces.
