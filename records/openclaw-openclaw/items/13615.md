## Summary

Keep this issue open as the canonical product-design tracker: current main has reactive provider cooldown/fallback handling, usage reporting, and owner-specific retry helpers, but it still lacks configurable pre-dispatch throttling, queues, provider budgets, warning thresholds, and the proposed limits CLI. The closed, unmerged https://github.com/openclaw/openclaw/pull/13686 is useful prior art, while the narrower open issues cover only parts of the requested behavior.

## What This Changes

Add configurable outbound API rate limiting, throttling, token or cost budgets, usage visibility, alerts, and CLI controls across providers, tools, and channels.

## Best Possible Solution

Define a maintainer-approved, opt-in provider-dispatch policy first, using canonical SQLite-backed accounting where persistence is required; keep channel throttling with channel owners and decide explicitly whether cost budgets remain a separate gateway capability tracked by https://github.com/openclaw/openclaw/issues/42475.

## Reproduction Assessment

Not applicable: this issue requests a new cross-cutting control surface rather than reporting one already-defined behavior with a concrete current-main failure test. The motivating 429 and runaway-cost scenarios are credible, but they do not reduce the work to a single bug reproduction.

## Solution Assessment

No, not as one universal subsystem spanning every external call. The maintainable direction is to scope a provider-dispatch policy first, preserve owner-specific channel behavior, and treat gateway cost enforcement as an explicit related product decision.
