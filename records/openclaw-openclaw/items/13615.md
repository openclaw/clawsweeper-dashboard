## Summary

Keep open. Current main still has adjacent reactive rate-limit classification, cooldown/fallback, provider usage readouts, and some channel/security throttles, but it does not implement the requested configurable pre-dispatch outbound API limits, queueing, token/cost budget enforcement, warning/status UX, or `openclaw limits` CLI.

## What This Changes

The issue requests configurable rate limiting, throttling, cost budgets, observability, and CLI controls for OpenClaw external API calls across providers, tools, and channels.

## Best Possible Solution

Keep this as the canonical design tracker for an opt-in outbound policy layer whose first supported boundary is model-provider dispatch, with explicit follow-up scope for tools, search, and channel/plugin calls.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 lack the requested config/schema/CLI and pre-dispatch enforcement surfaces. This is a feature gap, not a broken existing-behavior bug with a failing runtime reproduction path.

## Solution Assessment

Unclear until maintainers scope it. The requested direction is plausible, but the best implementation needs an owner-approved boundary for provider dispatch first, then deliberate inclusion or exclusion of web-search, tools, and channel/plugin calls.
