## What This Changes

The PR adds an active reply-run guard to `runHeartbeatOnce` and a regression test that expects heartbeat to skip with `requests-in-flight` while the target session has a running reply operation.

## Best Possible Solution

Land this narrow active-reply heartbeat guard with regression coverage after redacted real Telegram or live-runtime proof, or close it only if the broader related PR merges first and fully covers the same race.

## Reproduction Assessment

Yes, at source level: seed a Telegram-backed session, enqueue a system event for that session, register a running reply operation for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main lacks the active reply-run guard that would skip that state.

## Solution Assessment

Yes for the narrow race. Reusing `resolveActiveReplyRunSessionId(sessionKey)` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the smallest maintainable fix; broader stale replay/default-target leakage should stay in the related heartbeat replay work unless that PR becomes the canonical landing path.
