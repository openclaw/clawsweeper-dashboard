## Summary

Keep open: this is a focused fix for a source-reproducible heartbeat/reply race and I found no blocking code-level issue, but the external PR still needs after-fix real Telegram or live-runtime proof before merge.

## What This Changes

The PR adds a `runHeartbeatOnce` guard that returns `requests-in-flight` when a reply run is active for the resolved session and adds a regression test for the idle-lane/active-reply race.

## Best Possible Solution

Land or cherry-pick the guard and regression test after redacted real Telegram or live-runtime proof and current checks are available.

## Reproduction Assessment

Yes, as a high-confidence source-level path: seed a Telegram-backed session, enqueue a system event, create a running `ReplyOperation` for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main has no active reply-run guard for that state.

## Solution Assessment

Yes. Reusing `resolveActiveReplyRunSessionId` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the narrowest maintainable fix; broader restart or async-completion hygiene should stay separate.
