## Summary

Keep open: current main still lacks the active reply-run heartbeat guard, and the PR is a narrow source-backed fix, but this external Telegram-visible bug-fix PR still needs after-fix real behavior proof before merge.

## What This Changes

The PR imports `resolveActiveReplyRunSessionId` into `runHeartbeatOnce`, skips heartbeat execution with `requests-in-flight` while a reply operation is active for the target session, and adds a focused regression test.

## Best Possible Solution

Land or cherry-pick this guard on current main with the regression test, refreshed checks, maintainer-owned changelog handling, and redacted real Telegram or live-runtime proof.

## Reproduction Assessment

Yes. A focused source-level reproduction is to seed a Telegram-backed session, enqueue a system event, create a running `ReplyOperation` for that session key, mock queue sizes as zero, and call `runHeartbeatOnce`; current main has no active reply-run guard for that state.

## Solution Assessment

Yes. Reusing the existing reply-run registry after heartbeat preflight and returning the established retryable `requests-in-flight` skip is the narrowest maintainable fix for this race; worker-thread or default-isolation changes belong to separate broader issues.
