## Summary

Keep open. Current main still lacks the active reply-run heartbeat guard, and this PR is a focused fix for a source-reproducible Telegram reply-loss race; it should not merge until after-fix real Telegram or equivalent live-runtime proof is supplied, and maintainers may also want to compare it with the broader overlapping work in https://github.com/openclaw/openclaw/pull/81313.

## What This Changes

The PR adds an active reply-run check to `runHeartbeatOnce` and a regression test that skips heartbeat execution while a reply operation is still running for the target session.

## Best Possible Solution

Land or replace the active reply-run heartbeat guard with regression coverage after redacted real Telegram or equivalent live-runtime proof is available, while keeping broader stale replay and wrong-target delivery fixes separate unless a broader PR intentionally subsumes this one.

## Reproduction Assessment

Yes, as a high-confidence source-level path: seed a Telegram-backed session, enqueue a system event, create a running reply operation for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main has no active reply-run guard for that state.

## Solution Assessment

Yes for the narrow active-reply race. Reusing `resolveActiveReplyRunSessionId` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is a small maintainable fix; broader replay or wrong-route cleanup should be handled by the overlapping broader PR or follow-up work.
