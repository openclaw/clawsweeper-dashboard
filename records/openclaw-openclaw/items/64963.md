## Summary

Keep open: current main still lacks the active reply-run heartbeat guard, and this PR is a focused source-level fix with no blocking code findings, but it still needs contributor-supplied real behavior proof before merge.

## What This Changes

The PR adds a `runHeartbeatOnce` guard that skips heartbeat execution while the target session has an active reply run, plus a regression test for the idle-lane active-reply case.

## Best Possible Solution

Land this narrow guard and regression test after redacted real Telegram or live-runtime proof, while tracking broader stale replay/default-target leakage in the related open reports.

## Reproduction Assessment

Yes, at source level: seed a Telegram-backed session, enqueue a system event for that session, register a running reply operation for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main lacks the active reply-run guard that would skip that state.

## Solution Assessment

Yes for the narrow race. Reusing `resolveActiveReplyRunSessionId(sessionKey)` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the smallest maintainable fix; broader stale replay issues should remain separate unless a wider PR fully covers them.
