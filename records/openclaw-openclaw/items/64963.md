## What This Changes

The PR adds an active reply-run guard to `runHeartbeatOnce` and a regression test for a heartbeat/system-event wake while a reply operation is still running.

## Best Possible Solution

Land or cherry-pick the guard with regression coverage after redacted real Telegram or live-runtime proof and refreshed checks are available.

## Reproduction Assessment

Yes, as a high-confidence source-level path: seed a Telegram-backed session, enqueue a system event, create a running `ReplyOperation` for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main has no active reply-run guard for that state.

## Solution Assessment

Yes. Reusing `resolveActiveReplyRunSessionId` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the narrowest maintainable fix; broader worker-thread or restart-continuation work should stay separate.
