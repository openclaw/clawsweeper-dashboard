## What This Changes

The PR adds an active-reply guard to `runHeartbeatOnce` that returns the existing retryable heartbeat busy skip, plus a regression test for an active `ReplyOperation` with an idle session lane.

## Best Possible Solution

Land this branch or an equivalent patch after adding one Unreleased Fixes changelog line and after final CI is green; keep the implementation as the narrow heartbeat-runner check against reply-run registry state rather than broadening this into a worker-thread or routing redesign.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the PR's focused test shape: seed a Telegram main session, enqueue a system event, create a running `ReplyOperation`, make `getQueueSize` return zero, and verify current main proceeds while the patch skips before invoking the reply path.

## Solution Assessment

Yes, with the changelog addition. The proposed guard runs after preflight has the canonical `sessionKey`, uses existing reply-run lifecycle state, and reuses the established retryable `requests-in-flight` contract, which is the narrowest maintainable fix for this race.
