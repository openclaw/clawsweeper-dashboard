## Summary

Keep open. Current main still only skips heartbeat on lane queue depth, while this patch adds the missing active reply-run guard; the code change looks correct, but the external PR still lacks after-fix real Telegram or live-runtime proof before merge.

## What This Changes

The PR adds an active reply-run check to `runHeartbeatOnce` and a regression test so heartbeat skips with `requests-in-flight` when the target session still has a running reply operation.

## Best Possible Solution

Land this narrow guard and regression after redacted real Telegram or live-runtime proof and current checks, or close it as superseded if the broader proven heartbeat replay fix lands first.

## Reproduction Assessment

Yes, as a high-confidence source-level path: seed a Telegram-backed session, enqueue a system event for that session, register a running reply operation for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main lacks the active reply-run guard that would skip that state.

## Solution Assessment

Yes for the narrow race. Reusing `resolveActiveReplyRunSessionId(sessionKey)` after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the smallest maintainable fix; broader stale replay hygiene should stay in the separate heartbeat replay work.
