## Summary

Keep this PR open: the change is a plausible focused fix, but the proposed MCP degraded cooldown does not retry within an already-live runtime, and the contributor proof does not show the failure paths that the PR changes.

## What This Changes

The PR marks stale Codex OAuth refresh failures as auth_permanent, adds optional MCP server degraded cooldown catalog state, accepts agents.defaults.gptChatBrevityGuard in config schema/types, and adds regression tests.

## Best Possible Solution

Keep the focused auth/MCP direction, but make degraded MCP catalogs expire or recompute after degradedUntil and require redacted real-run proof for the changed failure paths before merge.

## Reproduction Assessment

Yes, this is source-reproducible: current main lacks the proposed auth failure marking, has no MCP degraded cooldown, and the strict agent defaults schema does not accept gptChatBrevityGuard. I did not execute the repro tests because this review is read-only.

## Solution Assessment

No, not as-is: the fix direction is reasonable, but degraded catalog entries can stay cached past degradedUntil in a live session and the PR lacks sufficient real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Retry degraded MCP servers after cooldown expires:** `src/agents/pi-bundle-mcp-runtime.ts:415-423`
  - body: When an optional server fails, this stores a degraded entry in the runtime's cached catalog. `getCatalog()` returns that cached catalog before consulting the cooldown, so an active session can keep reporting the server as degraded after `degradedUntil` instead of retrying; invalidate or recompute the catalog when the degraded window expires.
