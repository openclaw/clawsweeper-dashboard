## Summary

Keep open: the ACPX handoff bug is real and the contributor supplied usable live proof, but the patch is not correct as written because mapping OpenClaw one-shot ensures to ACPX persistent mode can make stable one-shot sessions reuse a closed persistent ACPX record instead of creating a fresh backend session.

## What This Changes

The PR maps ACPX one-shot ensure calls to persistent delegate mode, adds ACPX regression coverage, and carries adjacent Matrix, session patch, and plugin config test updates.

## Best Possible Solution

Keep the ACPX client alive across the initial ensure-to-turn boundary while preserving OpenClaw one-shot freshness, close semantics, and focused regression coverage.

## Reproduction Assessment

Yes. Current main splits ensureSession from runTurn while acpx 0.7.0 closes non-persistent ensure clients, and the contributor supplied redacted after-fix live output for that handoff shape.

## Solution Assessment

No, not as-is. The owner-boundary is right, but rewriting oneshot to persistent changes ACPX record id and reuse semantics; the fix should retain a live client without making later stable one-shot sessions resume the old ACPX record.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve one-shot freshness when holding the ACPX client:** `extensions/acpx/src/runtime.ts:68`
  - body: Mapping every OpenClaw `oneshot` ensure to ACPX `persistent` makes acpx use the stable `sessionKey` as the record id and reuse that record on the next ensure. Stable one-shot bindings can then reopen the previously closed backend session instead of getting the fresh one-shot behavior that the control plane expects. Keep the client alive for the first turn without changing the reusable record semantics, or mark/reset the ACPX record so later one-shot ensures cannot reuse it.
  - confidence: 0.88
