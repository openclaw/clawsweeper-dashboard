## Summary

Keep this open. The report is recent, detailed, and still matches current main: Telegram direct replies enter the shared inbound reply path and then the embedded runner, which still performs model resolution, auth setup, tool materialization, system prompt construction, resource loading, and stream setup before the first assistant streaming callbacks can reach Telegram. The later Raspberry Pi data point and closed #75744 are useful context, but they do not supersede the broader embedded-runner startup/prep overhead reported here.

## What This Changes

The issue reports that Telegram direct chat replies spend roughly 25-31 seconds in embedded agent startup/prep stages before assistant streaming begins.

## Best Possible Solution

Add a measured warm-turn path in the shared embedded runner that reuses model/auth/tool/prompt/resource preparation only behind clear invalidation rules, while preserving per-turn channel authorization, tool policy, provider config, prompt, and session semantics. Prove the change with embedded stage summaries plus the Telegram RTT harness rather than a Telegram-specific bypass.

## Reproduction Assessment

Yes. The issue body and follow-up comment provide concrete stage traces, and current main still has the same Telegram-to-embedded-runner path and stage timing hooks; a maintainer can reproduce with repeated trivial Telegram DM messages under trace logging or with the RTT harness. I did not run the live channel/model path in this read-only review.

## Solution Assessment

Unclear. The requested caching directions are plausible, but the best fix should first isolate the hot stages with current-main metrics and then add narrowly invalidated shared caches or warm resources; a lightweight direct-chat shortcut would be riskier because it could skip auth, tool policy, or prompt construction semantics.
