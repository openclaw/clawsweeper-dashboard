## Summary

Keep this open. The report is recent, detailed, and still matches current main's Telegram direct-chat embedded-agent path: each turn enters the shared embedded runner and performs model resolution, auth setup, tool materialization, system-prompt assembly, session resource loading, and stream setup before any assistant streaming can begin. Current main adds Telegram RTT measurement support, but I did not find a merged runtime change that removes or caches the reported per-turn startup/prep work.

## What This Changes

Investigate and reduce Telegram direct-chat pre-stream latency caused by repeated embedded-agent startup and preparation work before model streaming begins.

## Best Possible Solution

Add a measured warm-turn path for the shared embedded runner that reuses invalidated model/auth/tool/prompt preparation where it is safe, while preserving per-turn channel authorization, tool policy, session state, and provider config semantics. Prove the reduction with embedded stage summaries plus the Telegram RTT harness rather than a Telegram-specific shortcut.

## Reproduction Assessment

Yes. The issue body gives four concrete traces against v2026.4.29, and current main still has the same stage timing hooks and Telegram-to-embedded-runner path; a maintainer can reproduce with repeated trivial Telegram DM messages under trace logging or the RTT harness. I did not run live Telegram/model traffic in this read-only review.

## Solution Assessment

Unclear. The suggested caching directions are plausible, but the safest fix should first isolate the hot stage with RTT/stage metrics and then add narrowly invalidated shared caches or warm resources, not a direct-chat bypass that skips policy or prompt setup.
