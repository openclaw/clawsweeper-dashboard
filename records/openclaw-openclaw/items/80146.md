## Summary

Keep open: this PR targets a real current-main cron run-history job-name leak, but the patch still trusts a request-supplied `agentId` on a read-scoped method and the supplied proof does not exercise real persisted/gateway run-history behavior.

## What This Changes

This PR changes `cron.runs(scope: "all")` to build job-name enrichment from paginated `cron.listPage()` results scoped by an optional/default agent id, adds an `agentId` protocol field, refreshes Swift models, and adds tests plus a changelog entry.

## Best Possible Solution

Derive the enrichment scope from trusted caller/session context, or attach job names only for jobs the caller is already authorized to list, then add a regression and redacted real gateway or persisted run-history proof.

## Reproduction Assessment

Yes. Source inspection shows current main builds all-scope job names from an unfiltered cron list and the run-log reader attaches/searches those names; the PR’s residual issue is also source-reproducible because the new scope is caller-controlled.

## Solution Assessment

No. `listPage()` is the right primitive, but using an untrusted request `agentId` is not the best access-control fix; the scope should come from trusted caller/session context or an explicitly authorized admin path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive cron run-name scope from the caller:** `src/gateway/server-methods/cron.ts:531`
  - body: The new `agentId` is trusted directly when building the all-scope `jobNameById` map. Because `cron.runs` is available to read-scoped clients and the schema accepts any non-empty string, a caller who knows or guesses another agent id can still get that agent's job names attached and searchable in run history. Derive this scope from trusted caller/session context, or require an admin-checked override, instead of taking it from the request.
  - confidence: 0.88
