## Summary

Keep open: this PR is a plausible narrow fix for a source-reproducible cron metadata leak, but it is still unmerged and the external PR proof is mock-only rather than proof from a real cron/gateway setup.

## What This Changes

The PR scopes all-scope `cron.runs` job-name enrichment through paginated agent-scoped cron list pages, adds `agentId` to `CronRunsParams`, refreshes Swift protocol models, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land this or an equivalent narrow fix after real behavior proof confirms job names are scoped in an actual cron/gateway run-history flow.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main: `cron.runs(scope: "all")` builds the job-name map from the unfiltered cron list and the run-log reader uses that map for search text and returned `jobName` fields.

## Solution Assessment

Mostly yes. Reusing `listPage()` for enrichment is the narrow maintainable fix for the named leak, but merge should wait for real behavior proof and maintainer/security confirmation of the unchanged global run-log rows.
