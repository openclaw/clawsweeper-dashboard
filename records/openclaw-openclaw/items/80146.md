## Label Justifications

- `P1`: The PR targets and still affects a cross-agent cron run-history metadata disclosure in a gateway read path.
- `merge-risk: 🚨 compatibility`: The diff intentionally changes all-scope run-history enrichment so entries outside the requested/default agent may no longer include names.
- `merge-risk: 🚨 security-boundary`: The diff adds a public request-selected `agentId` to a security-sensitive read path without proving caller authorization for that agent.

## Summary

Keep open: the targeted fix is valid, but the branch still lets a read-scoped `cron.runs` caller choose the job-name enrichment agent with a request field, and the supplied proof is handler/mock output rather than a real persisted run-history check.

## What This Changes

The PR adds optional `agentId` support to `cron.runs`, switches all-scope job-name enrichment to paginated `cron.listPage()`, updates docs/Swift protocol models/changelog, and adds a focused server-method test.

## Best Possible Solution

Use the existing `listPage()` primitive, but derive the enrichment scope from trusted caller/session context or an explicitly authorized admin path, then add persisted run-history proof while leaving broader cron isolation to https://github.com/openclaw/openclaw/issues/26370.

## Reproduction Assessment

Yes. Source inspection shows current main builds all-scope `cron.runs` names from an unfiltered job list, and the PR residual issue is source-reproducible because `p.agentId` drives the name map for an `operator.read` method.

## Solution Assessment

No. Reusing `listPage()` is the right primitive, but the scope should come from trusted caller/session context or an admin-checked override rather than a caller-selected request field.
