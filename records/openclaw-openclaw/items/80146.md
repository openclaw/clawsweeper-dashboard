## Summary

Keep open: the PR targets a real source-backed cron job-name disclosure, but it is not merge-ready because current main now validates cron RPCs from `packages/gateway-protocol`, the new client-supplied `agentId` is not a trusted security boundary, and the supplied proof does not show persisted two-agent behavior.

## What This Changes

The PR changes all-scope `cron.runs` job-name enrichment to use a paginated agent-scoped job lookup, adds an optional `agentId` RPC parameter with docs/generated Swift/test updates, adds handler regression tests, and edits the changelog.

## Best Possible Solution

Rebase onto current main, update the live Gateway protocol package, derive the job-name lookup scope from trusted caller context or an explicit authorized operator mode, remove the normal-PR changelog edit, and add redacted persisted two-agent proof plus focused regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main's all-scope `cron.runs` branch builds its name map from unfiltered `context.cron.list`, and the run-log reader stamps those names onto returned entries. I did not run a live two-agent reproduction in this read-only review.

## Solution Assessment

No, this is a useful direction but not the best mergeable fix yet: the scope should come from trusted caller or operator context, and the active `packages/gateway-protocol` contract must move with the handler. The PR also needs persisted two-agent proof before the security claim is merge-ready.
