## Label Justifications

- `P1`: The PR targets a cross-agent cron metadata disclosure in gateway run history.
- `merge-risk: 🚨 compatibility`: Merging changes all-scope `cron.runs` enrichment so entries outside the requested/default agent may lose previously returned job names.

## Summary

Keep open: this PR targets a real P1 cron metadata leak and mostly uses the right existing `listPage` contract, but it is not merge-ready because raw `agentId` can still bypass filtering when it is whitespace-only, the supplied proof is handler/mock output rather than real run-history behavior, and GitHub currently reports the branch as conflicting.

## What This Changes

The PR changes all-scope `cron.runs` job-name enrichment to use paginated agent-scoped cron list pages, adds an optional `agentId` protocol field, updates docs/Swift/changelog surfaces, and adds mocked regression tests.

## Best Possible Solution

Normalize or reject `cron.runs` `agentId` before lookup, preserve scoped job-name enrichment for the resolved agent, and land after redacted real run-history proof plus maintainer acceptance that broader run-row isolation stays with https://github.com/openclaw/openclaw/issues/26370.

## Reproduction Assessment

Yes. Current main is source-reproducible: all-scope `cron.runs` builds names from the unfiltered cron list, and the run-log reader uses that map in returned/searchable metadata; the PR-specific blank-`agentId` bypass is also source-reproducible from schema and normalization code.

## Solution Assessment

No, not yet. Reusing `listPage()` is the right narrow fix for normal values, but the PR must normalize or reject blank `agentId` before fallback and add coverage for that security boundary.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Normalize agentId before scoped lookup:** `src/gateway/server-methods/cron.ts:583-587`
  - body: This takes any present `agentId`, including whitespace-only values that pass `NonEmptyString`, and forwards it into `listPage`. `listPage` normalizes blank strings to `undefined` and only filters when the normalized id is truthy, so `cron.runs({ scope: "all", agentId: " " })` still builds an unfiltered `jobNameById` map and can expose other agents' job names. Trim/normalize the param first and fall back to the default agent or reject blank values, with a regression test.
