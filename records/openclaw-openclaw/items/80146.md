## Summary

Keep open: current main still has the reported `cron.runs(scope: "all")` job-name leak, and this PR is a plausible narrow fix, but it is not merge-ready until contributor real-behavior proof, conflict resolution, and maintainer acceptance of the scoped-enrichment compatibility boundary are handled.

## What This Changes

The PR changes all-scope `cron.runs` job-name enrichment to use paginated agent-scoped `cron.listPage()` with an optional `agentId`, plus schema, docs, generated Swift model, changelog, and regression-test updates.

## Best Possible Solution

Land this or an equivalent narrow name-enrichment fix after redacted real-behavior proof and maintainer security/compatibility confirmation, while leaving broader per-agent run-log isolation to https://github.com/openclaw/openclaw/issues/26370.

## Reproduction Assessment

Yes. Source inspection on current main shows all-scope `cron.runs` builds the job-name map from the unfiltered cron list, and the run-log reader exposes that map in search text and returned `jobName` fields.

## Solution Assessment

Mostly yes. Reusing the existing `listPage()` agent filter is the narrow maintainable fix for the named job-name leak, but maintainers still need to accept the narrower compatibility/security boundary and keep broader run-row isolation separate.
