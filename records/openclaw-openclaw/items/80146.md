## Label Justifications

- `P1`: The PR targets a cross-agent cron metadata disclosure in a gateway server method used by multi-agent deployments.
- `merge-risk: 🚨 compatibility`: The patch intentionally changes all-scope response enrichment semantics for entries outside the requested/default agent scope.

## Summary

Keep this PR open. The reported cron job-name leak is real and the patch direction matches the existing scoped cron list contract, but it still needs contributor real-behavior proof and maintainer acceptance of the narrower scoped-enrichment boundary before merge.

## What This Changes

The PR adds optional `agentId` support to `cron.runs`, switches all-scope job-name enrichment from unfiltered `cron.list()` to paginated agent-scoped `listPage()`, and updates protocol docs, generated Swift models, changelog, and regression tests.

## Best Possible Solution

Land this or an equivalent narrow fix after redacted real runtime proof and maintainer acceptance of the scoped-enrichment compatibility boundary, leaving full per-agent run-log isolation to https://github.com/openclaw/openclaw/issues/26370.

## Reproduction Assessment

Yes, source-reproducible: current main builds the all-scope `cron.runs` job-name map from the unfiltered cron list before all-scope run-log reads. I did not run a live repro because this review is read-only and the checkout must remain clean.

## Solution Assessment

Mostly yes: reusing `listPage()` keeps the fix aligned with the existing agent-scoped cron visibility path, and the optional `agentId` protocol field is additive. The remaining decision is whether maintainers accept scoped name enrichment while broader run-row isolation stays in the separate open issue.
