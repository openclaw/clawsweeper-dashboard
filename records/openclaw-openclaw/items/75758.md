## Summary

Keep open. Current main still can treat an expired or failed Feishu CardKit final update/close as successful, then suppress later final text delivery after the streaming session is marked closed.

## What This Changes

The issue requests graceful Feishu streaming-card timeout handling so the final result is still rendered after a long-running card session expires.

## Best Possible Solution

Keep the fix inside the Feishu plugin: verify CardKit final update/close responses and fall back to a static card or text reply before marking final delivery suppressed.

## Reproduction Assessment

Yes for a source-level reproduction: simulate a failed or expired CardKit final update/close that resolves through FeishuStreamingSession.close, then deliver a later final payload and observe dispatcher suppression. The live 10-minute Feishu timeout itself still needs external log proof.

## Solution Assessment

Yes. The narrow maintainable solution is to validate Feishu CardKit finalization and add Feishu-local fallback delivery; a broad core channel contract change is not justified by the evidence.
