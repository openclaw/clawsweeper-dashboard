## Summary

Keep open. Current main still lets Feishu CardKit final update/close failures look successful, then suppresses later final delivery, matching the reported hanging-card failure mode.

## What This Changes

The issue asks for Feishu streaming cards to deliver or attach the final result even after the long-running card stream times out.

## Best Possible Solution

Keep the repair inside the Feishu plugin: verify CardKit final update/settings success and fall back to a static card or text reply before marking the streaming reply closed or deduped.

## Repair Work Prompt

Fix the Feishu streaming-card timeout finalization bug reported at https://github.com/openclaw/openclaw/issues/75758. Keep the change inside the Feishu plugin unless a genuinely generic seam is required. Make CardKit final content update and streaming-mode close validate HTTP response success and Feishu JSON code where available, propagate or report finalization failure, and make the dispatcher avoid setting delivered/closed suppression state when finalization fails or expires. Add a fallback path that sends the final answer as a static Feishu card or text reply so the user still sees the final result. Add focused regression coverage for failed/expired CardKit final update/close and for a later final payload not being suppressed after failed streaming close. Add a single-line CHANGELOG.md fix entry under Unreleased/Fixes with appropriate human attribution if known. Do not change core channel contracts, unrelated Feishu tools, or dependency versions.
