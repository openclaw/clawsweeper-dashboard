## Summary

Keep open. Current main still has a focused Feishu streaming-card finalization gap: CardKit final update/close calls can appear successful without proving Feishu accepted the final content, while the dispatcher can then suppress later final-card delivery as already handled.

## What This Changes

The issue asks Feishu streaming cards to recover from the roughly 10-minute streaming-session timeout and still deliver the final result to the user.

## Best Possible Solution

Keep the fix inside the Feishu plugin: make CardKit update/close success explicit, check HTTP and Feishu API results, and only suppress normal final delivery after the final card update is verified. If the card session is expired or update/close fails, deliver the final text through a safe static-card or text fallback.

## Reproduction Assessment

Yes for a source-level/unit reproduction: configure Feishu card streaming, simulate an expired or failed CardKit final update/close, then deliver the final payload and observe that current dispatcher suppression can prevent fallback text/card delivery. The live 10-minute timeout path still needs an API-error fixture or log sample.

## Solution Assessment

Yes, a narrow Feishu plugin repair is the best direction. Core channel behavior does not need a new API; the maintainable fix is to make finalization success observable and gate duplicate suppression on that success.
