## Summary

Keep open. Current main still has a plausible Feishu streaming-card finalization gap: CardKit update/close failures can be swallowed while the dispatcher later suppresses normal final-text delivery after it believes the streaming card closed. The earlier ClawSweeper comment is aligned with current source and test evidence, and no related item in the provided context is a canonical fix for the timeout/final-result loss.

## What This Changes

The issue requests that Feishu streaming cards recover from long-running session timeout and still attach or deliver the final answer after the timeout.

## Best Possible Solution

Keep the fix inside the Feishu plugin. Make streaming-card finalization report verified success, validate CardKit update/close HTTP and Feishu API responses, and fall back to a static card or text reply when the final card update cannot be confirmed without reopening known duplicate-card regressions.

## Reproduction Assessment

Yes for a source-level reproduction: simulate a CardKit final update or close that fails or expires while `FeishuStreamingSession.close()` returns without a verified final card update, then deliver the final payload and observe the dispatcher suppression path. The exact live 10-minute timeout still needs a Feishu API fixture or log sample.

## Solution Assessment

Yes, the requested direction is the right maintainable shape if scoped to explicit Feishu plugin finalization and fallback delivery. A broad core change is unnecessary unless multiple channel plugins need the same verified-finalization contract.

Fix the Feishu streaming-card timeout/final-delivery gap from https://github.com/openclaw/openclaw/issues/75758. Keep the change inside the bundled Feishu plugin. Make `FeishuStreamingSession` validate CardKit content update and close responses (`response.ok` plus Feishu `code` where available) and return or throw enough status for the dispatcher to know whether final text actually landed. Add focused tests in `extensions/feishu/src/streaming-card.test.ts` and `extensions/feishu/src/reply-dispatcher.test.ts`, and add a user-facing `CHANGELOG.md` fix entry with proper reporter credit. Do not move Feishu-specific policy into core and do not change unrelated Feishu auth/onboarding/media behavior.
