## Summary

Keep open. Current main still has a source-level Feishu streaming-card timeout/finalization failure path: CardKit final update/close calls release responses without validating HTTP/API success, then the dispatcher can suppress later final text after marking the streaming card closed.

## What This Changes

The issue asks Feishu streaming cards to handle long-running session timeout and still attach the final result when the response completes.

## Best Possible Solution

Keep the fix inside the Feishu plugin by making CardKit final update/close report verified success and falling back to a static card or text reply before suppressing later final delivery.

## Reproduction Assessment

Yes at source level: mock an expired or failed CardKit final update/close while FeishuStreamingSession.close returns, then deliver the final payload and observe dispatcher suppression. I did not run a live 10-minute Feishu fixture or local tests in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to validate Feishu CardKit finalization and add a Feishu dispatcher fallback; a core channel contract change or new config knob is not supported by the evidence.
