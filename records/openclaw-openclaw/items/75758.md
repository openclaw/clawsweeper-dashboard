## Summary

Keep open. Current main still treats Feishu streaming-card close as successful without validating CardKit final update/close responses, then suppresses later final delivery after the session is considered closed.

## What This Changes

The issue requests Feishu streaming cards to recover after a long-running CardKit streaming session times out and still attach the final result to the card.

## Best Possible Solution

Keep the fix inside the Feishu plugin: make final CardKit update/close return verified success, and fall back to a static card or text reply before suppressing later final delivery when finalization fails or expires.

## Reproduction Assessment

Yes for a source-level reproduction: simulate a failed or expired CardKit final update/close while FeishuStreamingSession.close returns, then send the final payload and observe dispatcher suppression. The external 10-minute Feishu timeout still needs a live fixture or log sample.

## Solution Assessment

Yes. The narrow maintainable solution is to validate Feishu CardKit finalization and fall back in the Feishu dispatcher; a broad core channel contract change is not justified by the current evidence.
