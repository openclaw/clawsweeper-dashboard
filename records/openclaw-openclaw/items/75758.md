## Summary

Keep open. Current main still has a Feishu streaming-card finalization gap: CardKit final update/close responses are not verified, while the dispatcher can suppress later final-text delivery after it believes the streaming card closed.

## What This Changes

Feishu streaming cards should gracefully handle sessions that expire after long runs and still render the final response.

## Best Possible Solution

Keep the fix inside the Feishu plugin: verify CardKit final update/close success, and fall back to a static card or text reply when finalization fails or expires before suppressing later final delivery.

## Reproduction Assessment

Yes for a source-level reproduction: simulate a failed or expired CardKit final update/close while FeishuStreamingSession.close returns, then send the final payload and observe dispatcher suppression. The live 10-minute Feishu timeout still needs an external fixture or log sample.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if scoped to Feishu finalization and fallback delivery; a broad core channel change is not justified unless other plugins need the same contract.

Make FeishuStreamingSession verify CardKit content update and close/settings responses, including HTTP status and Feishu API code where available, and ensure close() reports failure when the final content was not confirmed.
