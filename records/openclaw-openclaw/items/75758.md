## Summary

Keep open. Current main still has a Feishu streaming-card timeout/finalization failure path: CardKit final update and close requests are treated as successful without validating HTTP or Feishu JSON success, and the dispatcher can then suppress the later final card text after marking streaming closed.

## What This Changes

The issue asks Feishu streaming cards to survive the roughly 10-minute CardKit timeout and still render the final AI result.

## Best Possible Solution

Keep the fix inside the Feishu plugin: verify CardKit final update/close success, propagate finalization failure to the dispatcher, and send a static card or text fallback before duplicate-final suppression can hide the final result.

## Reproduction Assessment

Yes, source-level reproduction is clear: mock an expired or failed CardKit update/close response while FeishuStreamingSession.close returns, then deliver the final payload and observe dispatcher suppression. A live 10-minute Feishu fixture would add external proof but is not needed to see the current failure path.

## Solution Assessment

Yes. The narrow maintainable fix is to validate Feishu CardKit finalization and fall back in the Feishu dispatcher; a broad core channel contract change is not justified by the evidence.
