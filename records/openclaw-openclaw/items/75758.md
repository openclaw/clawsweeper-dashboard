## Summary

Keep open. Current main still has a source-level Feishu streaming-card timeout/finalization failure path: CardKit final update/close responses are not validated, and later final-card text is suppressed once streaming is marked closed.

## What This Changes

The issue asks Feishu streaming cards to handle long-running CardKit session timeout and still attach the final result to the card.

## Best Possible Solution

Keep the fix inside the Feishu plugin by making final CardKit update/close report verified success and falling back to a static card or text reply before suppressing later final delivery.

## Reproduction Assessment

Yes for a source-level reproduction: mock a failed or expired CardKit final update/close while FeishuStreamingSession.close returns, then deliver the final payload and observe dispatcher suppression. A live 10-minute Feishu fixture would improve external proof but is not required to see the current failure path.

## Solution Assessment

Yes. The narrow maintainable solution is to validate Feishu CardKit finalization and fall back in the Feishu dispatcher; a broad core channel contract change is not justified by the evidence.

Fix the Feishu streaming-card timeout/finalization bug reported at https://github.com/openclaw/openclaw/issues/75758. Keep the change inside the Feishu plugin. Make CardKit final content update, note update if relevant, and streaming close verify HTTP success and Feishu JSON success instead of treating release-only fetch completion as success.
