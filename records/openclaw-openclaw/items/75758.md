## Summary

Keep open. Current main still has a plausible Feishu streaming-card finalization gap: CardKit update and close calls can return or fail without verified success, while the dispatcher later suppresses normal final text delivery once it believes the streaming card has closed. The related closed Feishu reports cover duplicate cards, duplicate text, and merge behavior, not this timeout/final-result-loss path.

## What This Changes

The issue asks the Feishu plugin to handle streaming-card timeout gracefully and attach the final result to the card, or otherwise deliver it, after a 10+ minute CardKit streaming session expires.

## Best Possible Solution

Keep the repair inside the Feishu plugin: make CardKit final update/close return verified success or explicit failure, and fall back to a static card or text delivery when final card finalization cannot be confirmed, while preserving existing duplicate-final suppression for successful closes.

## Reproduction Assessment

Yes for the source-level path: mock a failed or expired CardKit final update/settings PATCH while an active Feishu streaming session closes, then deliver the final payload and observe the dispatcher suppression path. The exact live 10-minute Feishu timeout still needs a fixture or log sample, but the current code path is concrete enough to reproduce in unit tests.

## Solution Assessment

Yes, the requested direction is the right maintainable shape when scoped to Feishu plugin finalization and fallback delivery. A broad core channel change is not needed unless another channel exposes the same verified-finalization contract gap.
