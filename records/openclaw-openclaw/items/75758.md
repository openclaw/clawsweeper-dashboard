## Summary

Keep open. Current main still has a plausible Feishu streaming-card finalization gap: the streaming session close path attempts to update the card with final text, but it does not verify Feishu/CardKit success and can swallow update or close failures, while the dispatcher later suppresses normal final-text delivery once it believes the streaming card was closed. That matches the reported symptom closely enough to warrant a focused Feishu plugin fix rather than cleanup closure.

## What This Changes

The issue asks Feishu streaming cards to recover from an approximately 10-minute CardKit streaming-session timeout and still attach or deliver the final response.

## Best Possible Solution

Keep the fix inside the Feishu plugin. Make CardKit finalization report whether the final text actually landed, treat expired/failed streaming-card updates as not finalized, and deliver the final response through a safe fallback such as a new static card or text reply without reopening duplicate-card regressions.

## Reproduction Assessment

Yes for a source-level/unit reproduction: enable Feishu card streaming, let the streaming session close path fail or expire, then deliver a final payload and observe that current dispatcher suppression can prevent any fallback final text. The live 10-minute Feishu timeout path still needs an API-error fixture or log sample.

## Solution Assessment

No, current main is not the best solution because it assumes streaming.close success even when CardKit update/close may fail. The narrower maintainable fix is to make finalization success explicit and suppress normal final delivery only after a verified final card update or an exact duplicate.
