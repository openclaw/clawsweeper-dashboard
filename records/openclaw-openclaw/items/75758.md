## Summary

Keep open. Current main still has a plausible Feishu streaming-card finalization gap: CardKit update/close calls can complete without verified success, while the dispatcher later suppresses normal final-text delivery after treating the streaming card as closed. The existing ClawSweeper comment is consistent with current source, tests, and related-item context; the nearby Feishu duplicate/merge reports are already closed and do not cover this timeout/final-result loss.

## What This Changes

The issue asks the Feishu plugin to handle streaming-card session timeout and still attach the final AI result to the card or otherwise deliver it after long-running responses.

## Best Possible Solution

Keep the fix inside the Feishu plugin. Make streaming-card finalization return a verified delivery result from CardKit update/close calls, then fall back to a static card or text reply when final card delivery cannot be confirmed, preserving duplicate-suppression behavior for genuinely successful closes.

## Reproduction Assessment

Yes for a source-level reproduction: simulate a CardKit final update or close that fails or expires while `FeishuStreamingSession.close()` resolves, then deliver a later final payload and observe `reply-dispatcher.ts` suppress normal final delivery. The live 10-minute Feishu timeout still needs service logs or a fixture to match the exact external failure shape.

## Solution Assessment

Yes, the requested direction is the right maintainable shape if scoped to Feishu plugin finalization and fallback delivery. A core channel change is not needed unless other plugins prove they need the same verified-finalization contract.
