## Summary

Keep open. Current main still has a Feishu plugin path where failed or expired CardKit finalization can be treated as successful, after which later final card text is suppressed.

## What This Changes

The issue requests Feishu streaming-card timeout handling so long-running final responses still appear after CardKit streaming expires.

## Best Possible Solution

Keep the fix inside the Feishu plugin by making CardKit final update/close success explicit and falling back to a static card or text reply before suppressing later final delivery.

## Reproduction Assessment

Yes for a source-level reproduction: mock a failed or expired CardKit final update/close while `FeishuStreamingSession.close` returns, then deliver the final payload and observe dispatcher suppression. A live Feishu timeout fixture would improve external proof but is not required to see the current failure path.

## Solution Assessment

Yes. The narrow maintainable solution is to validate Feishu CardKit finalization and fall back in the Feishu dispatcher; a broad core channel contract change is not justified by the evidence.
