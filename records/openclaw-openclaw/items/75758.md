## Summary

Keep open. Current main still has a source-level Feishu streaming-card timeout/finalization failure path: CardKit final update/close responses are not verified, and later final-card text can be suppressed once streaming is marked closed.

## What This Changes

Review Feishu streaming card timeout behavior where the final result may not be rendered after a long CardKit streaming session expires.

## Best Possible Solution

Keep the fix inside the Feishu plugin: verify CardKit final update/close success and fall back to a visible static card or text reply before suppressing late final text.

## Reproduction Assessment

Yes for a source-level reproduction: mock an expired or failed CardKit final update/close that still resolves through `FeishuStreamingSession.close`, then deliver a later final payload and observe dispatcher suppression. A live 10-minute Feishu fixture would improve external proof but is not required to see the current failure path.

## Solution Assessment

Yes. The narrow maintainable solution is to validate Feishu CardKit finalization in the Feishu plugin and use the existing static-card/text fallback path when finalization fails; a core channel contract change is not justified by the evidence.
