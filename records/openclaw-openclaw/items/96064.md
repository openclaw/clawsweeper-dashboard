## Label Justifications

- `P1`: The report describes a broken IRC channel workflow with repeated reconnects and ghost nicks affecting real channel use.
- `impact:crash-loop`: The source and reports point to an IRC restart/reconnect loop after nick collision recovery.

## Summary

Keep open: current main and the latest release still contain the static underscore fallback and a channel lifecycle path that can restart IRC after a successful connect, and the issue is now paired with an open fixing PR that should be reviewed before this issue closes.

## What This Changes

The issue requests that IRC nick-collision recovery stop repeatedly reconnecting with the same underscore-suffixed fallback nick.

## Best Possible Solution

Fix IRC so the monitor owns one live connection until abort, sends/probes avoid unnecessary duplicate transient nicks when a live client exists, and nick-collision fallback has deterministic bounded recovery coverage.

## Reproduction Assessment

Yes from source inspection, but not from a live server run. Current main still resets fallbackNickAttempted per connection, returns from monitorIrcProvider after connect, and lets the gateway auto-restart a resolved channel task.

## Solution Assessment

No, a suffix-only fix is likely too narrow. The maintainable fix should cover the IRC connection lifecycle and collision fallback together, or explicitly prove the lifecycle path is not involved for this fresh report.
