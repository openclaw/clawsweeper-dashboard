## Summary

Keep this open: the draft PR addresses a real plugin startup performance problem and supplies terminal timing proof, but the current fast cache-hit path appears to bypass the broader file-freshness verifier and can return stale plugin metadata after same-process persisted plugin file changes.

## What This Changes

The PR adds process-local memoization for `loadPluginMetadataSnapshot`, a new memo-key/fingerprint helper, lifecycle invalidation, and plugin registry regression tests around persisted install-record freshness.

## Best Possible Solution

Keep the process memo, but make the fast-hit path honor the same metadata freshness contract before running the focused memo/registry tests and broader plugin CI.

## Reproduction Assessment

Yes, at source/proof level. The PR body provides real terminal timing proof for the beta6 startup regression, and the review finding is source-reproducible from the fast-hit branch returning before the broad file-fingerprint key can run.

## Solution Assessment

No, not yet. A process-scoped memo is a plausible fix, but the current fast-key short-circuit weakens same-process plugin metadata freshness and should be corrected before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Keep fast cache hits fresh for persisted plugin files:** `src/plugins/plugin-metadata-snapshot.ts:217-220`
  - body: The new fast-key branch returns the cached snapshot as soon as `memoSlot.fastKey` matches, before the broad memo key is recomputed. That fast key only covers the persisted index and managed npm root package file, while the full key below fingerprints persisted manifest/source/setup/package files; after a same-process plugin file edit, the fast key can still match and return stale metadata. Please include the relevant file freshness in the hit check or fall through to the verifier when persisted plugin files may have changed.
  - confidence: 0.88
