## Summary

Keep open: this is a recent, focused performance PR with supplied real CLI timing proof, green relevant CI, and no blocking correctness or security finding from this read-only review. The main remaining gate is plugin-area maintainer review because the patch intentionally adds a process-local metadata cache in a control-plane area where freshness rules are strict.

## What This Changes

The PR adds a single-slot in-process memo for plugin metadata snapshots, expands discovery fingerprinting, clears the memo at metadata lifecycle boundaries, and adds regression tests for invalidation and aliasing.

## Best Possible Solution

Land the PR after plugin-area maintainer review confirms the process-local memo is the right exception to the metadata freshness rule, keeping clone-on-return and fingerprint invalidation coverage intact.

## Reproduction Assessment

Yes, at source/proof level: current main has repeated loadPluginMetadataSnapshot callers and no outer snapshot memo, and the PR body supplies before/after terminal timings for real CLI commands on a 104-plugin install. I did not recreate that large install in this read-only checkout.

## Solution Assessment

Likely yes: a single-slot process memo keyed by caller params plus discovery fingerprints, with cloned returns and lifecycle invalidation, is a narrow fix for repeated in-process rebuild cost. The safer alternative would be a narrower prepared-snapshot handoff, but the PR already covers the main freshness boundaries with focused tests and CI.
