## Summary

Keep open: this is an active draft implementation, current main does not have the requested process-local plugin metadata snapshot memo, and the related hotfix PR is still open rather than shipped. The latest broad branch has one blocking freshness issue: the fast cache-hit path can bypass the fuller file-backed verifier that is supposed to keep persisted plugin metadata fresh.

## What This Changes

The PR adds a process-local memo for `loadPluginMetadataSnapshot`, broadens plugin metadata fingerprints, clears the memo at metadata lifecycle boundaries, ignores vanished recovered install records, and adds plugin memo/registry tests.

## Best Possible Solution

Keep the process-scoped memo, but make the fast path preserve file-backed plugin metadata freshness before merge, then validate it with plugin memo, registry, type, lint, and broad CI coverage.

## Reproduction Assessment

Yes, at source/proof level: current main has repeated `loadPluginMetadataSnapshot` callers with no outer memo, and the PR body supplies real beta6 CLI timing proof. The review finding is also source-reproducible from the fast-hit branch bypassing the full file-backed verifier.

## Solution Assessment

No, not yet: a process-local memo is a reasonable fix direction, but the current fast-hit shortcut must not skip persisted plugin manifest/source/package freshness checks. A safer solution either includes those cheap file signatures in the fast key or falls back to full-key verification before returning cached metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Verify file-backed freshness before fast cache hits:** `src/plugins/plugin-metadata-snapshot.ts:217-220`
  - body: The fast-hit branch returns the cached snapshot as soon as the persisted index and npm package fingerprints match, before computing the full key that fingerprints persisted plugin manifest, source, setup, package, and diagnostic-source files. A same-process manifest or package metadata edit would now reuse stale metadata instead of reaching the existing stale-source checks, so include those file fingerprints in the fast key or fall back to the full verifier before returning.
  - confidence: 0.88
