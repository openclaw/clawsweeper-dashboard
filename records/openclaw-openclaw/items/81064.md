## What This Changes

The PR adds a process-local memo for plugin metadata snapshots, expands plugin metadata fingerprinting, clears memo state on metadata lifecycle boundaries, clones cached returns, and adds invalidation/aliasing tests.

## Best Possible Solution

Land the fingerprint-keyed, process-scoped plugin metadata snapshot memo after CI confirms the broad invalidation surface, keeping the cache within plugin control-plane ownership.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source/proof level: current main has many direct loadPluginMetadataSnapshot callers, and the PR body gives before/after terminal timings from a real 104-plugin install; I did not run that real install locally.

## Solution Assessment

Is this the best way to solve the issue? Likely yes: a single-slot memo keyed by caller inputs plus a stat fingerprint is narrower than a TTL cache and the tests exercise the main stale-read risks; full CI should validate the wide fingerprint surface.
