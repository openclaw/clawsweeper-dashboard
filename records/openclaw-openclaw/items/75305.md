## Summary

Keep open. Current main still sends the pre-compaction memory flush through a blank `transcriptPrompt`/runtime-only path, the nearest tests still assert that empty provider-facing prompt behavior, and open PR #75384 already references this issue with closing syntax, so this issue should wait for that PR or a maintainer-approved alternative to land.

## What This Changes

The issue reports that automatic pre-compaction memory flushes can submit an empty user message to Anthropic, causing a 400 error and Telegram session reset.

## Best Possible Solution

Keep the flush silent from the user's perspective while ensuring the provider-facing submitted user turn is non-empty, with focused tests for the strict-provider request shape; close this issue only after #75384 or an equivalent maintainer-approved fix lands.

## Reproduction Assessment

Yes. The source-level path is high confidence: memory flush passes `transcriptPrompt: ""`, runtime-context splitting turns that into a runtime-only `prompt: ""`, and the embedded runner submits that empty prompt.

## Solution Assessment

No. Current main is not the best solution because it preserves the blank provider-facing submission; the narrow maintainable fix belongs at the memory-flush/runtime prompt boundary without exposing internal housekeeping text to the user transcript.
