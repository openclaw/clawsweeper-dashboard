## Summary

Keep open. Current main still passes an empty transcript prompt for the pre-compaction memory flush, the embedded runner converts that into a runtime-only blank prompt, and the linked closing PR #75384 remains the implementation candidate rather than a merged fix. Related #75313/#75368 are the same strict-provider message-shape family, but they target session-file repair, not this memory-flush submission path.

## What This Changes

Fix the pre-compaction memory flush so strict providers such as Anthropic receive a non-empty user message while the housekeeping turn remains silent to the Telegram user.

## Best Possible Solution

Land PR #75384 or an equivalent narrow patch that sends the memory-flush directive as the provider-visible user prompt while preserving `silentExpected` and `NO_REPLY` delivery suppression, then close this issue through the merged fix.

Yes. The source-level reproduction path is high-confidence: memory flush passes `transcriptPrompt: ""`, runtime prompt resolution returns `prompt: ""` with `runtimeOnly: true`, the skip guard allows it, and the attempt submits that blank prompt.
