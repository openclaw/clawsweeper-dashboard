## Summary

Keep open. Current main still routes the pre-compaction memory flush through a blank `transcriptPrompt`/runtime-only path, tests still assert that blank behavior, and open PR #75384 already references this issue with closing syntax, so the issue should stay open until that PR is reviewed and resolved.

## What This Changes

The issue reports that automatic pre-compaction memory flushes can submit an empty provider-facing user message to Anthropic, causing a 400 error and Telegram session reset.

## Best Possible Solution

Keep the pre-compaction flush as silent housekeeping while ensuring the provider-visible user turn is non-empty, with tests covering the strict-provider request shape; close this issue only after the linked PR merges or a maintainer-approved alternative lands.

## Reproduction Assessment

Yes. The source-level reproduction path is high-confidence: memory flush passes `transcriptPrompt: ""`, runtime prompt resolution converts that to a runtime-only `prompt: ""`, and the embedded attempt submits that blank prompt instead of skipping it.

## Solution Assessment

No. Current main is not the best solution because it preserves the blank provider-facing submission; the maintainable direction is a narrow fix that keeps user delivery silent while sending a valid non-empty flush prompt to the provider.
