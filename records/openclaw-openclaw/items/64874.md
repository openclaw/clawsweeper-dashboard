## Summary

Keep open. The PR addresses a real current-main visibility gap around bootstrap truncation, but the branch is not merge-ready: the `/status` change targets an old builder surface, the edit hint uses a stale fixed 20,000-character threshold, and the user-facing diagnostics need policy-compliant release notes before landing.

## What This Changes

The PR adds edit-failure guidance to bootstrap truncation warnings, adds workspace bootstrap file health output to `/status`, and appends a large-file truncation note to edit mismatch errors.

## Best Possible Solution

Adapt the useful diagnostics to current main by rendering workspace-file health from `src/status/status-message.ts`, reusing `SessionSystemPromptReport` plus `analyzeBootstrapBudget` and resolved bootstrap limits, adding focused coverage for warning text/status output/edit mismatch diagnostics, and adding a changelog entry if this remains a user-facing fix.

## Reproduction Assessment

Yes. A workspace bootstrap file over the current 12,000-character default can be represented in `sessionEntry.systemPromptReport.injectedWorkspaceFiles`; current `/context` can report truncation from that data, while current `/status` never reads it and edit mismatch errors only append the current-content snippet.

## Solution Assessment

No. The requested direction is reasonable, but this branch is not the best current implementation because it edits the old status builder location and hardcodes 20,000 chars instead of deriving the active bootstrap limits from the current report/config-backed contract.
