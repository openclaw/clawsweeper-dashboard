## Summary

Keep open. Current main and the latest v2026.5.18 release still combine Google/Gemini tagged `<think>/<final>` prompting with native Gemini `thinkingConfig`, and the discussion has multiple affected-user confirmations without a merged fixing PR for this direct provider conflict.

## What This Changes

The issue asks OpenClaw to stop combining Google/Gemini tagged reasoning prompts with native Gemini thinking output because it can yield unclosed tags, empty post-tool turns, and wasted model calls.

## Best Possible Solution

Move direct Google/Gemini reasoning output to native whenever native Gemini thinking is sent, update the provider hooks and regression tests, and keep broader OpenRouter reasoning-field extraction under its separate canonical issue.

## Reproduction Assessment

Yes, at source level: current main deterministically resolves Google/Gemini to tagged reasoning mode while direct Google requests include native `thinkingConfig.includeThoughts`. I did not run a paid live Gemini loop in this read-only review.

## Solution Assessment

Yes. Fixing the provider-owned Google/Gemini reasoning-output contract is the narrow maintainable solution; generic empty-payload retries or broad OpenRouter reasoning-field extraction would leave the conflicting prompt/API contract intact.
