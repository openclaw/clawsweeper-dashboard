## Summary

Keep open. Current `main` and shipped `v2026.5.18` still route Google/Gemini through tagged `<think>/<final>` prompting while the direct Google transport sends native Gemini `thinkingConfig.includeThoughts`, so the reported provider-contract conflict remains source-reproducible and is not solved by the related OpenRouter reasoning-field work.

## What This Changes

The issue reports that Google/Gemini is forced into tagged reasoning while also using native Gemini thinking, causing unclosed `<think>` text, empty post-tool turns, and wasted paid model calls.

## Best Possible Solution

Move direct Google/Gemini reasoning output to native when native Gemini thinking is sent, update provider-family tests/docs, and preserve existing thought-signature plus function-call replay guarantees.

## Reproduction Assessment

Yes, at source level: current `main` deterministically resolves Google/Gemini to tagged reasoning mode while the direct Google transport sends native `thinkingConfig.includeThoughts`. I did not run a paid live Gemini loop in this read-only review.

## Solution Assessment

Yes. Fixing the provider-owned Google/Gemini reasoning-output contract is the narrow maintainable solution; downstream empty-payload handling or broad reasoning-field extraction would leave the conflicting prompt/API contract intact.
