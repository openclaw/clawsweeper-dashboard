## Summary

Keep open. Current main and v2026.5.18 still combine Google/Gemini tagged `<think>/<final>` prompting with native Gemini `thinkingConfig`, and the live discussion has multiple independent user confirmations with no merged closing PR. The later OpenRouter reasoning-field comment overlaps but is separately tracked by https://github.com/openclaw/openclaw/issues/74021.

## What This Changes

The issue requests aligning Google/Gemini reasoning output with native thinking so tagged reasoning does not cause unclosed tags, empty post-tool turns, and `payloads=0` failures.

## Best Possible Solution

Move direct Google/Gemini reasoning output to native when native Gemini thinking is sent, update the Google family tests/docs, and preserve thought-signature and tool replay guarantees.

## Reproduction Assessment

Yes, at source level: current main deterministically resolves Google/Gemini to tagged reasoning mode while the direct Google transport sends native `thinkingConfig.includeThoughts`. I did not run a paid live Gemini loop in this read-only review.

## Solution Assessment

Yes. Fixing the provider-owned Google/Gemini reasoning-output contract is the narrow maintainable solution; downstream empty-payload handling or broad reasoning-field extraction would leave the conflicting prompt/API contract intact.
