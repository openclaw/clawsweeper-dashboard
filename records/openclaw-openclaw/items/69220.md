## Summary

Keep open. Current main and the latest release still combine Google/Gemini tagged reasoning-output instructions with native Gemini thinking payloads, and no open fixing PR owns this canonical tracker.

## What This Changes

The issue asks OpenClaw to stop forcing tagged `<think>/<final>` reasoning for `google-generative-ai` because Gemini already uses native `thinkingConfig` and thought parts, causing empty post-tool turns.

## Best Possible Solution

Update the Google/Gemini provider reasoning-output contract so Gemini transports that send native thinking resolve to `native`, while preserving replay sanitation, thought-signature safeguards, and tool-result pairing coverage.

## Reproduction Assessment

Yes, at source level: current main deterministically resolves Google/Gemini to tagged mode while the direct Google transport sends native `thinkingConfig.includeThoughts`. The issue discussion adds multiple matching real-session reports, but I did not run a paid live Gemini loop.

## Solution Assessment

Yes. Fixing the provider-owned Google/Gemini reasoning-output mode is the narrow maintainable fix; only downstream empty-payload handling would leave the conflicting prompt/API contract intact.
