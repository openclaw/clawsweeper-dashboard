- `P1`: The issue breaks real Google/Gemini agent turns with empty post-tool responses and wasted paid model calls in active workflows.
- `impact:message-loss`: The reported `payloads=0` and final-less tagged output can suppress the user-visible assistant reply.

## Summary

Keep open. Current main and the v2026.5.18 release still combine Google/Gemini tagged `<think>/<final>` prompting with native Gemini `thinkingConfig`, the discussion has multiple independent affected-user confirmations, and no merged closing PR owns this direct Google/Gemini provider conflict.

## What This Changes

This issue requests changing Google/Gemini reasoning-output mode so native Gemini thinking is not combined with text-tag `<think>/<final>` prompting during tool-call turns.

## Best Possible Solution

Move direct Google/Gemini reasoning output to native whenever Gemini native thinking is sent, update the provider hooks, tests, and docs, and preserve thought-signature/tool replay guarantees.

## Reproduction Assessment

Yes, at source level: current main deterministically resolves Google/Gemini to tagged reasoning mode while direct Google requests include native `thinkingConfig.includeThoughts`. I did not run a paid live Gemini loop in this read-only review.

## Solution Assessment

Yes. Fixing the provider-owned Google/Gemini reasoning-output contract is the narrow maintainable solution; generic empty-payload retry or OpenRouter reasoning-field extraction would leave the conflicting prompt/API contract intact.
