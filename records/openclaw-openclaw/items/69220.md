## Summary

Keep open. Current main and the latest release still route Google/Gemini through tagged `<think>/<final>` prompting while the direct Google transport sends native Gemini `thinkingConfig.includeThoughts`, so the reported provider-contract conflict remains source-reproducible and is not superseded by the OpenRouter follow-up work.

## What This Changes

The issue asks OpenClaw to stop combining Google/Gemini tagged reasoning-output prompts with Gemini native thinking payloads because that can produce unclosed reasoning tags, empty post-tool turns, and `payloads=0` failures.

## Best Possible Solution

Move direct Google/Gemini reasoning output to native when native Gemini thinking is sent, update provider-family tests/docs, and keep existing thought-signature and tool replay guarantees intact.

## Reproduction Assessment

Yes, at source level: current main deterministically resolves Google/Gemini to tagged reasoning mode while the direct Google transport sends native `thinkingConfig.includeThoughts`. I did not run a paid live Gemini loop in this read-only review.

## Solution Assessment

Yes. Fixing the provider-owned Google/Gemini reasoning-output contract is the narrow maintainable solution; downstream empty-payload handling or broad reasoning-field extraction would leave the conflicting prompt/API contract intact.
