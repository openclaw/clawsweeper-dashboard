## Label Justifications

- `P1`: The report describes a real Gemini agent workflow that can burn large token budgets and end with empty or suppressed user-visible replies.
- `impact:message-loss`: The affected path can produce `payloads=0`, empty assistant turns, or discarded final text after tool use.
- `impact:session-state`: The issue concerns native thinking/tagged text state being persisted and replayed through session turns in a way that changes later turn behavior.

## Summary

Keep open. Current main and the latest release still combine Google/Gemini tagged reasoning-output prompting with native Gemini thinking payloads, and the discussion has multiple independent affected-user reports plus a closed duplicate that names this issue as the canonical tracker.

## What This Changes

The issue asks OpenClaw to stop forcing Google/Gemini into tagged `<think>/<final>` reasoning when native Gemini thinking is enabled because the combination can produce empty or suppressed turns.

## Best Possible Solution

Handle this at the Google provider boundary by making Gemini transports that send native `thinkingConfig` use native reasoning output, while preserving required thought-signature/tool replay behavior and leaving broader textSignature bloat to https://github.com/openclaw/openclaw/issues/48709.

## Reproduction Assessment

Yes, at source level: current main deterministically resolves Google/Gemini to tagged reasoning mode while the Google transport sends native `thinkingConfig.includeThoughts`. I did not run a paid live Gemini cron/tool-call loop in this read-only review.

## Solution Assessment

Unclear for automation. The provider-owned fix boundary is right, but maintainers should choose the exact Google tagged-versus-native compatibility path before automation changes the reasoning contract.
