## Summary

Keep open. Current main still combines Google/Gemini tagged reasoning-output instructions with native Gemini thinking payloads, the latest release still contains the tagged fallback, and related cleanup has treated this as the canonical tracker rather than a duplicate.

## What This Changes

The issue reports that Google/Gemini providers combine tagged `<think>/<final>` reasoning prompts with native Gemini thinking payloads, causing unclosed tags, empty post-tool turns, and wasted Gemini calls.

## Best Possible Solution

Update the Google/Gemini provider reasoning-output contract so Gemini transports that send native thinking resolve to `native`, while preserving replay sanitation, thought-signature safeguards, and tool-result pairing coverage.

## Reproduction Assessment

Yes, at source level: current main deterministically resolves Google/Gemini to tagged mode while the direct Google transport sends native `thinkingConfig.includeThoughts`. The issue discussion adds multiple matching real-session reports, but I did not run a paid live Gemini loop in this review.

## Solution Assessment

Yes. Fixing the provider-owned Google/Gemini reasoning-output mode is the narrow maintainable fix; only downstream empty-payload handling would leave the conflicting prompt/API contract intact.
