## Summary

Keep open: current main has the Feishu-side answer-plus-warning composition from https://github.com/openclaw/openclaw/pull/94428, but the core payload/tool-warning path can still emit a visible warning after a successful assistant reply when a retained mutating `lastToolError` survives.

## What This Changes

The issue asks OpenClaw to stop surfacing a misleading Feishu warning ahead of the assistant reply when a successful turn retains an exec `lastToolError`, especially for read-only exec output.

## Best Possible Solution

Keep this issue open until maintainers confirm the warning policy, reproduce the false `lastToolError` source on current main, and land a core payload/tool-error fix that still preserves Feishu answer-first delivery.

## Reproduction Assessment

No high-confidence controlled repro yet. Source inspection shows the current warning-emission path when a retained mutating `lastToolError` exists, but the false `lastToolError` trigger and the second Feishu run still need a current-main live or trajectory-backed reproduction.
