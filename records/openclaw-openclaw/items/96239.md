## Summary

Keep open: current main still has source-visible paths that can turn a recorded exec-like tool error into a user-visible warning payload alongside a successful assistant reply, and the Feishu dispatcher still treats final-shaped diagnostics as final delivery traffic. The related open issues and PRs cover adjacent slices, but none fully owns this specific false warning plus Feishu visibility report.

## What This Changes

The issue reports that a successful Feishu agent turn can queue a tool-warning card plus the real reply when exec stdout is recorded as an exec error/mutating action.

## Best Possible Solution

Keep this issue as the narrow canonical report, then fix the shared tool-error policy/classification so successful read-only exec output cannot create a user-visible failure warning, while coordinating the Feishu final-card ordering slice with the existing Feishu dispatcher work.

## Reproduction Assessment

Yes, at source level for the user-visible warning: a successful assistant reply plus lastToolError.mutatingAction=true still produces two final payloads. The exact successful-stdout-to-lastToolError trigger needs the reporter's trajectory or a focused synthetic regression before claiming a full current-main reproduction.

## Solution Assessment

Unclear as a single automatic fix: the likely solution crosses exec mutation classification, tool-result error attribution, and Feishu final diagnostic ordering. The safest path is a maintainer-approved policy plus focused tests for false read-only exec warnings and real mutating failures.
