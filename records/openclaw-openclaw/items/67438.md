## Summary

Keep open. Current main still routes Feishu new-app onboarding through scan-to-create and renders the QR with compact terminal output, while the public docs still tell users to scan that QR; the latest release keeps the same compact setting. The earlier repair PR was closed unmerged, and the newer related open PR only adds WSL font guidance while preserving `small: true`, so the reported scanability problem is not implemented on main.

## What This Changes

The issue reports that Feishu onboarding via `openclaw onboard` displays a terminal QR code that the Feishu mobile app does not scan on Windows 10.

## Best Possible Solution

Fix this in the Feishu plugin-owned setup path by making scan-to-create QR output reliably scannable, preferably by opting Feishu out of compact terminal QR rendering and adding a user-visible PNG or browser fallback when terminal rendering fails. Keep any WSL-specific font hint as a diagnostic aid, not the only repair, and align tests/docs if a fallback becomes user-visible.

## Reproduction Assessment

Unclear. The source-level reproduction path is high confidence because `openclaw onboard` / `openclaw channels login --channel feishu` reaches `runScanToCreate()` and compact `printQrCode()`, but the actual mobile scan failure still needs live Windows terminal plus Feishu/Lark app verification.

## Solution Assessment

No for the current main behavior. Keeping compact terminal output and only documenting or hinting around terminal font issues is not the narrowest reliable fix; the maintainable solution should change the Feishu QR output or provide a fallback in the plugin setup path.
