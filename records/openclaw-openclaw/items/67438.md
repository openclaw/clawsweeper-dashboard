## Summary

Keep open. Current main and the latest release still render Feishu scan-to-create with compact terminal QR output, while the reported setup path remains documented and reachable. The closed larger-QR PR was not merged, and the open WSL-font guidance PR is only a partial related candidate, not proof that the Windows onboarding scan failure is fixed.

## What This Changes

The issue reports that OpenClaw 2026.4.14 on Windows 10 shows a Feishu onboarding QR code from `openclaw onboard` that the Feishu mobile app does not scan.

## Best Possible Solution

Fix this in the Feishu plugin setup path by making scan-to-create QR output reliably scannable across common terminals, preferably by using a larger terminal QR and adding an openable PNG or URL fallback when terminal rendering is unreliable. If PR #72762 remains active, maintainers should either broaden it to cover the actual Feishu scan failure or close it as partial guidance and land a focused Feishu QR fix separately.

## Reproduction Assessment

Partially. The live scan failure still needs a Windows 10 terminal and Feishu mobile app to confirm end to end, but the source-level reproduction path to the suspect compact QR is clear: `openclaw onboard` -> channel setup -> Feishu setup wizard -> `runScanToCreate()` -> `printQrCode()` with `small: true`.

## Solution Assessment

No. Current main is not a solution, and the open WSL-font guidance PR is not the narrowest complete fix for a Windows command-window scan failure because it leaves the compact Feishu QR rendering unchanged.
