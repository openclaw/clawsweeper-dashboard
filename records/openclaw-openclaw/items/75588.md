## Summary

Keep open. This PR fills a real current-main Plugin SDK gap, but the stale-registry guard can still commit a registration-time scheduled cron job after that registry has been superseded, leaving an unloaded plugin's scheduled turn behind.

## What This Changes

The PR adds bundled-plugin `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` SDK APIs backed by cron jobs, plus cleanup hardening, API exports, tests, and a changelog entry.

## Best Possible Solution

Keep the additive cron-backed scheduled-turn seam, but tighten post-add commit/rollback so registration-time fire-and-forget schedules cannot survive registry replacement while preserving bundled-only access and the existing cron payload contract.

## Reproduction Assessment

Yes. A focused test can delay mocked `cron.add`, start `api.scheduleSessionTurn` during plugin registration, replace the active registry before the promise resolves, and then assert the old registry does not retain or leave behind the scheduled job.

## Solution Assessment

No. Reusing cron through a bundled-only SDK seam is the right direction, but the proposed stale-registry protection is incomplete for registration-time async schedules and should re-check active registry state before committing after `cron.add`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Recheck active registry for registration-time schedules:** `src/plugins/registry.ts:2360-2363`
  - body: `scheduledDuringRegistration` stays true for the whole async schedule operation, so a fire-and-forget call made during plugin registration can still commit after this registry has been replaced. The replacement cleanup runs before `cron.add` resolves and sees no runtime record yet; this branch then registers a cron job for the inactive plugin, so disable/reload cleanup can miss it and the scheduled turn can still fire. Gate the post-add commit on no active registry or this registry being active, and add a regression for replacement while `cron.add` is pending.
  - confidence: 0.86

## Repair Work Prompt

Fix the stale-registry lifecycle bug in PR #75588. The current `src/plugins/registry.ts` `scheduleSessionTurn` wrapper captures `scheduledDuringRegistration` and then lets post-`cron.add` commit succeed even when a newer active registry has replaced the one that started the schedule. Adjust the commit/rollback logic in `src/plugins/registry.ts` and/or `src/plugins/host-hook-workflow.ts` so registration-time scheduling remains allowed before activation, but a schedule resolving after registry replacement is rolled back and not registered. Add regression coverage in `src/plugins/contracts/scheduled-turns.contract.test.ts` that delays mocked `cron.add`, starts `api.scheduleSessionTurn` during plugin registration, replaces the active registry before resolving the add, and asserts the stale job is removed or not retained in runtime records.
