## Summary

Keep open. Current main still lacks the scheduled-turn SDK seam, and the PR is useful, but the PR head still omits delivery metadata on the default scheduling path, which can make cron.add reject default scheduled turns for multi-channel announce setups.

## What This Changes

This PR adds bundled-plugin scheduleSessionTurn and unscheduleSessionTurnsByTag SDK APIs backed by cron jobs, runtime cleanup, API exports, a changelog entry, and contract tests.

## Best Possible Solution

Land the bundled-only scheduled-turn API after the default scheduling path emits cron-valid delivery metadata and regression coverage exercises omitted deliveryMode against multi-channel cron validation.

## Reproduction Assessment

Not applicable as a user bug; this is a feature PR. The remaining review finding is source-reproducible from the PR delivery construction plus current cron normalization and validation behavior.

## Solution Assessment

No. The cron-backed bundled-plugin seam is a reasonable direction, but the PR should send a concrete default delivery such as announce with channel "last" when callers omit deliveryMode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Send default delivery metadata for scheduled turns:** `src/plugins/host-hook-workflow.ts:320-327`
  - body: `scheduleSessionTurn` only includes `delivery` when callers pass `deliveryMode`. When omitted, cron normalization turns this session-bound `agentTurn` job into announce delivery without a channel, and cron.add validation rejects that shape for installs with multiple announce channels. Please provide a tested default such as `{ mode: "announce", channel: "last" }` before calling `cron.add`.
  - confidence: 0.88
