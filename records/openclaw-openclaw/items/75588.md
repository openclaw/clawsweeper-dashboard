## Summary

Keep open. Current main still lacks the scheduled-turn SDK API, so the PR is useful, but the latest head still leaves the default `scheduleSessionTurn` delivery path cron-invalid for multi-channel installs.

## What This Changes

The PR adds bundled-plugin `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` SDK APIs backed by cron agent-turn jobs, tag cleanup, runtime cleanup, tests, API baseline updates, and a changelog entry.

## Best Possible Solution

Land the bundled-only scheduled-turn seam after the default delivery payload is made cron-valid and exact-head validation passes, leaving broader workflow docs and fixtures to the linked follow-up PRs.

## Reproduction Assessment

Yes for the PR finding: source inspection shows omitted `deliveryMode` skips the delivery object, while current cron normalization and validation reject the resulting default announce delivery when multiple channels are configured. Not applicable as a current-main user bug because this is a new feature PR.

## Solution Assessment

No. The cron-backed bundled-only seam is the right shape, but the patch should send a cron-valid default delivery such as `{ mode: "announce", channel: "last" }` when callers omit `deliveryMode`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Send default delivery metadata for scheduled turns:** `src/plugins/host-hook-workflow.ts:320-327`
  - body: When callers omit `deliveryMode`, this path omits `delivery` before `cron.add`. Current cron normalization turns session-bound `agentTurn` jobs without delivery into announce delivery, and the gateway validator rejects announce delivery without a channel when multiple channels are configured, so the default `scheduleSessionTurn` path fails for those installs. Please send a cron-valid default such as `{ mode: "announce", channel: "last" }`.
  - confidence: 0.88
