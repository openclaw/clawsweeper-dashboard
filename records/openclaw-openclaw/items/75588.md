## Summary

Keep open. Current main still does not expose the scheduled-turn SDK seam, and the closed predecessor stack means this PR is the active fresh-main review surface rather than obsolete cleanup. The latest diff still has a blocking SDK/Gateway contract mismatch around arbitrary cron payload extras, plus a required changelog gap.

## What This Changes

This PR adds `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` to the plugin SDK, wires bundled-plugin cron-backed scheduling and tag cleanup, refreshes the SDK API baseline hash, and adds scheduled-turn contract tests.

## Best Possible Solution

Keep the PR open, but align the scheduled-turn SDK with the cron protocol before merge: either add a typed metadata container that the cron schema accepts and preserves, or drop/narrow arbitrary `payloadExtras` until that protocol support exists. Add the required changelog entry once the public SDK surface is settled.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call the new helper with the PR’s tested `payloadExtras` such as `{ archetype: "approval-snapshot" }`; the helper sends that key to `cron.add`, and current main rejects unknown agent-turn payload fields through the closed cron schema.

## Solution Assessment

No. The general bundled-plugin scheduled-turn seam is a reasonable SDK boundary, but the proposed arbitrary payload extras path is not compatible with the existing Gateway cron contract; a schema-backed metadata field or a narrower SDK type is safer.
