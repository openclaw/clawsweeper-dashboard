## Summary

Keep open. The scheduled-turn seam is a useful additive SDK feature and current main lacks it, but the PR still has a P2 default-delivery bug: calls that omit `deliveryMode` can be rejected by cron validation on multi-channel installs.

## What This Changes

This PR adds bundled-plugin `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` SDK APIs backed by cron agent-turn jobs, runtime cleanup, API exports/baseline updates, a changelog entry, and contract tests.

## Best Possible Solution

Land the narrow bundled-only scheduled-turn API after fixing default delivery and exact-head validation, with broader workflow docs and fixtures staying in the linked follow-up PRs.

## Reproduction Assessment

Not applicable as a feature PR rather than a user bug. The current-main API gap is source-confirmed, and the review finding is source-reproducible from the PR delivery construction plus current cron validation behavior.

## Solution Assessment

No. The cron-backed bundled-only seam is the right direction, but the PR should first send a cron-valid default delivery, such as `announce` with `channel: "last"`, when callers omit `deliveryMode`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Send a concrete default delivery for scheduled turns:** `src/plugins/host-hook-workflow.ts:320-327`
  - body: `scheduleSessionTurn` only includes `delivery` when callers pass `deliveryMode`. When they omit it, `cron.add` normalizes this `session:*` `agentTurn` job to announce delivery without a channel, and the existing cron validator rejects that shape on installs with more than one announce channel. That makes the default API path fail for multi-channel users; please provide a tested default such as `{ mode: "announce", channel: "last" }` before calling `cron.add`.
  - confidence: 0.86
