## Summary

Keep open. Current main still lacks the scheduled-turn SDK seam, so this PR is not obsolete, but the proposed `payloadExtras` contract is incompatible with the current closed cron `agentTurn` payload schema and the public SDK change is missing the required changelog entry.

## What This Changes

The PR adds `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` to the plugin SDK, wires bundled plugins to schedule Gateway cron-backed session turns with tag cleanup, updates SDK exports/baselines, and adds focused contract tests.

## Best Possible Solution

Keep the scheduled-turn SDK seam direction, but align it with the existing cron protocol before merge: make any extra metadata schema-backed and reflected in Gateway protocol/client baselines, or drop `payloadExtras` until that protocol support exists. Add the required single-line changelog entry after the public API shape is final.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call the new helper with the PR's documented/tested `payloadExtras`, such as `{ archetype: "approval-snapshot" }`; the real `cron.add` path validates against the closed cron payload schema and rejects the extra key before creating the job.

## Solution Assessment

No. The host-mediated scheduling seam fits the plugin SDK boundary, but the proposed arbitrary payload extras are not compatible with the current cron contract; a typed schema change or narrower SDK type is the safer maintainable path.
