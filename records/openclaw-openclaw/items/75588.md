## Summary

Keep open: this is a useful additive Plugin SDK PR, but the current head still has a blocking restart-cleanup bug where replacement-plugin runtime-only scheduled turns can be removed immediately after activation.

## What This Changes

Adds bundled-plugin scheduled session-turn APIs, cron-backed scheduling and tag cleanup helpers, SDK exports, API baseline and changelog updates, and focused contract tests.

## Best Possible Solution

Keep the additive bundled-only SDK seam, but make restart cleanup distinguish old runtime-only scheduler records from replacement-generation records before landing.

## Reproduction Assessment

Yes. Source inspection shows replacement cleanup passes previous registry records, then the new fallback scans all live runtime jobs and can remove runtime-only scheduled turns created by the replacement plugin generation.

## Solution Assessment

No. The current PR direction is maintainable, but the cleanup fallback needs a generation/snapshot-aware boundary or equivalent preserve mechanism so it only removes stale runtime-only jobs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve new runtime-only scheduled jobs during restart cleanup:** `src/plugins/host-hook-runtime.ts:446-460`
  - body: When restart cleanup runs with previous registry records, this fallback scans all live runtime jobs for the plugin. `api.scheduleSessionTurn()` creates runtime-only jobs during replacement registration, and those IDs are not in `registryRecordKeys` or `preserveJobIds`, so a freshly scheduled cron job can be removed immediately after activation. Track the old generation/snapshot, or otherwise exclude replacement-generation runtime jobs before running this fallback.
  - confidence: 0.91
