## Summary

Keep open: the scheduled-turn SDK seam is useful and aligned with the existing host-hook direction, but the latest head still has a restart cleanup regression that can remove freshly scheduled replacement-generation jobs.

## What This Changes

The PR adds bundled-plugin SDK APIs to schedule and tag-unschedule future session turns via cron, with runtime cleanup, SDK exports, API baseline/changelog updates, and contract coverage.

## Best Possible Solution

Keep the cron-backed bundled-plugin seam, but make registry-replacement cleanup preserve replacement-generation runtime-only scheduled turns and cover that reload case with a focused regression test.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: a replacement plugin can schedule a runtime-only turn during registration, and the fallback cleanup path then removes records absent from previousRegistry.sessionSchedulerJobs and preserveSchedulerJobIds. I did not execute tests because this review was constrained to read-only inspection.

## Solution Assessment

No, not yet. Reusing cron behind a bundled-only SDK seam is the right direction, but the cleanup fallback needs a generation-aware guard before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve new-generation runtime jobs on restart:** `src/plugins/host-hook-runtime.ts:456-459`
  - body: During registry replacement, the replacement plugin can call `api.scheduleSessionTurn()` during registration and create runtime-only scheduler records. Those records are not in `previousRegistry.sessionSchedulerJobs`, and `collectSchedulerJobIds()` only preserves declared next-registry jobs, so this fallback loop calls their cleanup and deletes freshly scheduled cron jobs right after activation. Scope this fallback to old-generation records, or otherwise skip records created by the replacement generation.
  - confidence: 0.92

## Repair Work Prompt

Fix PR #75588's scheduled-turn cleanup regression. On registry replacement, cleanupPluginSessionSchedulerJobs({ records: previousRegistry.sessionSchedulerJobs, preserveJobIds }) must not clean runtime-only scheduler records created by the replacement generation via api.scheduleSessionTurn() during registration. Add a generation/cutoff/ownership guard, or pass explicit previous-runtime record identity, so fallback cleanup still removes genuinely old dynamic records on disable/reset but skips newer replacement-generation records. Add a regression test that schedules a session turn from the replacement plugin, runs cleanupReplacedPluginHostRegistry, asserts cron.remove was not called for the new job, and asserts listPluginSessionSchedulerJobs still contains it. Keep the bundled-only origin gate and existing cron payload defaults unchanged. No extra changelog entry is needed unless the public SDK behavior changes beyond fixing this PR bug.
