## Summary

Keep open: current main still lacks the scheduled-turn SDK APIs, and the PR head has one source-backed blocker in the default `scheduleSessionTurn` path. The split from the brittle #73384/#74483 stack is useful and builds on the merged #72287 host-hook foundation, but the omitted-delivery case still conflicts with current cron validation for multi-channel installs.

## What This Changes

This PR adds bundled-plugin `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` SDK APIs backed by cron agent-turn jobs, runtime cleanup, API exports, baseline updates, a changelog entry, and contract tests.

## Best Possible Solution

Land the narrow bundled-only scheduled-turn API after fixing omitted-delivery behavior and rerunning exact-head SDK, lint, type, and focused cron/plugin contract validation.

## Reproduction Assessment

Yes for the review finding: call `scheduleSessionTurn` without `deliveryMode` on a config with multiple announce channels. Source inspection shows the PR omits `delivery`, current cron normalization defaults that to announce, and current validation rejects announce delivery without a channel.

## Solution Assessment

No. The cron-backed bundled-plugin seam is the right direction, but the PR should send a cron-valid default delivery, such as `{ mode: "announce", channel: "last" }`, when callers omit `deliveryMode`, and cover that path against the cron add validator.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Send default scheduled-turn delivery metadata:** `src/plugins/host-hook-workflow.ts:320-327`
