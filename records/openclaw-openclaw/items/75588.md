## Summary

Keep open. Current main does not yet provide the scheduled-turn Plugin SDK methods, and this draft PR is the active fresh-main replacement slice for the closed consolidated stack. The latest head still has two concrete cron-contract blockers before merge: announce delivery omits a required channel on multi-channel installs, and tag cleanup scans only one paginated cron list page.

## What This Changes

The PR adds bundled-plugin Plugin SDK methods for scheduling future session turns and unscheduling them by tag, plus SDK exports, host cron wiring, a changelog entry, generated API baseline hashes, and contract tests.

## Best Possible Solution

Keep the additive host-mediated SDK seam, but align it with the existing cron contracts before merge: announce scheduled turns should use a cron-valid route such as channel "last" or an explicit channel contract, tag cleanup should page through all cron.list results, and both behaviors should be locked by focused plugin contract and cron server validation tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call scheduleSessionTurn with deliveryMode "announce" on a multi-channel config and current cron.add rejects the missing delivery.channel; create more than 200 cron jobs with matching tagged jobs after the first page and unscheduleSessionTurnsByTag misses them.

## Solution Assessment

No. The boundary is the right place for bundled workflow plugins, but this implementation is not the best fix until it matches cron delivery and pagination semantics; the safer alternative is a small repair in src/plugins/host-hook-workflow.ts plus focused tests.
