## Summary

Keep open. Current main does not expose the scheduled-turn plugin API, and this draft is a fresh replacement slice for the closed predecessor stack rather than obsolete cleanup. The proposed patch has one concrete runtime correctness bug in its advertised payload extras path plus a required changelog gap; the security pass did not find a concrete supply-chain or permission regression.

## What This Changes

This PR adds `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` to the plugin SDK, implements a bundled-plugin cron-backed scheduling helper with tag cleanup and stale-registry protection, exports the new types, refreshes the SDK API baseline hash, and adds scheduled-turn contract tests.

## Best Possible Solution

Keep the PR open, but align the new SDK contract with the existing cron protocol before merge: either add a typed, schema-backed metadata container that cron accepts and preserves, or remove/narrow `payloadExtras` from this slice until the protocol supports it. Add the required changelog entry once the public SDK surface is settled.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call the new helper with the PR's documented/tested `payloadExtras` such as `{ archetype: "approval-snapshot" }`; the real Gateway `cron.add` path validates against the closed cron payload schema and rejects the extra key before creating the job.

## Solution Assessment

No. The general host-mediated scheduling seam fits the plugin SDK boundary, but the proposed payload extras implementation is not compatible with the current cron contract; a typed schema change or a narrower SDK type is the maintainable fix.
