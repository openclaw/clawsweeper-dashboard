## Summary

Keep open. Current main still lacks the scheduled-turn Plugin SDK seam, and this PR is the active implementation slice, but the latest head has a concrete CI-breaking lazy-boundary regression from statically importing the gateway client path into plugin registry loads.

## What This Changes

The PR adds bundled-plugin `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` SDK APIs, a cron-backed host workflow helper, scheduler cleanup hardening, SDK exports/baseline updates, a changelog entry, and focused contract tests.

## Best Possible Solution

Keep the additive scheduled-turn SDK seam, but move gateway cron calls behind a lazy runtime/injected boundary so importing the plugin registry does not load the gateway client/config stack; then either prune completed one-shot runtime records or document/accept that lifecycle tradeoff before final validation.

## Reproduction Assessment

Not applicable as a feature PR rather than a user bug. The current-main gap is reproducible by source search, and the PR regression is reproducible from exact-head CI annotations showing the registry-to-gateway import chain failure.

## Solution Assessment

No, not in the current patch form. Reusing cron behind a bundled-plugin origin gate is the right direction, but the implementation should preserve plugin registry laziness and avoid static gateway client imports before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Lazy-load gateway calls from the scheduler workflow:** `src/plugins/host-hook-workflow.ts:2`
  - body: Because `registry.ts` now statically imports this module, this import pulls `src/agents/tools/gateway.ts` and then `src/gateway/call.ts` into plugin registry loads. Exact-head CI is already failing in the agentic control-plane runtime shard because an existing `config/io` mock does not provide `getRuntimeConfig`. Put the gateway call behind a lazy runtime import or injectable boundary so registry/test loads do not import the gateway client stack.
  - confidence: 0.87
- **[P3] Prune completed one-shot scheduler records:** `src/plugins/host-hook-workflow.ts:337-345`
  - body: `scheduleSessionTurn` registers every scheduled turn in host runtime state, while `at` and `delayMs` schedules default to `deleteAfterRun`. After cron auto-deletes a successful one-shot job, the runtime record remains until tag cleanup or plugin cleanup, so long-lived plugins can accumulate stale scheduler records.
  - confidence: 0.76
