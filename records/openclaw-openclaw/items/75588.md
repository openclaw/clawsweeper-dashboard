## Summary

Keep open. Current main still lacks this scheduled-turn SDK surface, and the PR is useful, but the patch has a P2 reload cleanup bug: dynamic turns scheduled during plugin registration can be removed by previous-registry cleanup because they are not represented in the next registry preserve set.

## What This Changes

This PR adds bundled-plugin `scheduleSessionTurn` and `unscheduleSessionTurnsByTag` SDK APIs backed by cron jobs, API exports, a changelog entry, and contract tests.

## Best Possible Solution

Land the scheduled-turn SDK seam after reload cleanup preserves newly registered dynamic scheduled turns and the focused exact-head validations are green.

## Reproduction Assessment

Not applicable as a user bug; this is a feature PR. The review finding is source-reproducible from the registry replacement path: `scheduleSessionTurn` creates runtime-only job records, while cleanup preserves only registry-listed job ids before scanning live records.

## Solution Assessment

No. The cron-backed bundled-only API direction is reasonable, but the patch should first preserve registration-time scheduled turns across plugin reload cleanup with focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve registration-scheduled turns across plugin reloads:** `src/plugins/host-hook-runtime.ts:456-459`
  - body: This cleanup pass scans live scheduler records for the plugin, but `scheduleSessionTurn` registers generated cron job IDs only in runtime state. On reload, a plugin can schedule a new turn during registration before previous-registry cleanup runs; because that new job is absent from `nextRegistry.sessionSchedulerJobs` and `preserveJobIds`, this loop can call its cleanup and remove the freshly scheduled cron job. Please add a reload regression and preserve or generation-scope these dynamic scheduled-turn records.
  - confidence: 0.84
