## Summary

Keep open. Current main still lacks the scheduled-turn Plugin SDK seam, so this draft PR is not obsolete or implemented, but the proposed patch has a blocking contract mismatch: arbitrary `payloadExtras` are passed into a closed `cron.add` agent-turn payload schema. The existing PR discussion already flagged the same payload-schema issue plus the missing changelog entry, and the security pass did not find a supply-chain or permissions regression.

## What This Changes

This PR adds public Plugin SDK methods for scheduling and unscheduling future session turns, a bundled-plugin cron-backed host helper with tag and stale-plugin cleanup, SDK exports/API baseline updates, and focused contract tests.

## Best Possible Solution

Keep the draft open, settle the scheduled-turn metadata contract against the existing cron protocol, then update the implementation and contract tests accordingly. After the SDK surface is final, add the required single-line Unreleased changelog entry and rerun the targeted plugin SDK, type, lint, and diff checks.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call the PR helper with the tested `payloadExtras` such as `{ archetype: "approval-snapshot" }`; the real `cron.add` Gateway path validates against the closed agent-turn payload schema and rejects the extra key before job creation.

## Solution Assessment

No. The host-mediated scheduling seam fits the plugin SDK boundary, but the arbitrary `payloadExtras` implementation is incompatible with current cron validation; the maintainable fix is either a schema-backed metadata container accepted by cron or removing/narrowing extras from this slice.
