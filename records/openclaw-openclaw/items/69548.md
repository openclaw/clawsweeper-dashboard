## Summary

Keep open: the feature is useful and current main does not already expose it, but the latest PR head still leaves stale explicit stagger values in place when `dreaming.staggerMs` is removed.

## What This Changes

The PR adds optional `dreaming.staggerMs` memory-core config, threads it into the managed dreaming cron schedule, and documents/changelogs the new jitter control.

## Best Possible Solution

Land a version that keeps the cron pass-through feature but makes set, unset, and `0` semantics explicit in code, tests, and docs.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows `desiredStaggerMs === undefined` skips the new schedule comparison, while current cron patching preserves an omitted `staggerMs`. The original multi-instance operational scenario was not live-reproduced.

## Solution Assessment

No. Reusing existing cron stagger infrastructure is the right direction, but this is not the best mergeable form until unset/removal semantics either reconcile correctly or are intentionally documented as requiring `staggerMs: 0`.
