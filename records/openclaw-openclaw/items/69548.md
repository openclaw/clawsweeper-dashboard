## Summary

Keep this PR open. Current main still does not expose `dreaming.staggerMs`, and the PR wires the option through the intended memory config, manifest, docs, and managed cron schedule path, but the latest diff still has a lifecycle bug: removing the config field can leave a previously persisted explicit stagger in place while the docs say unset returns to the cron default behavior.

## What This Changes

The PR adds optional `dreaming.staggerMs` memory-core config/docs/schema support and threads it through the memory host SDK and managed dreaming cron job schedule/logging.

## Best Possible Solution

Ship the feature with a clear lifecycle contract for `dreaming.staggerMs`: creating and updating should pass explicit values through, `0` should disable stagger, and removing the field should either reset the managed job to cron-default behavior or be explicitly documented and tested as preserve-on-removal semantics.

## Reproduction Assessment

Yes. A focused unit reproduction is an existing managed dreaming cron job with `schedule.staggerMs` set, followed by desired config without `dreaming.staggerMs`; the PR's patch builder can emit no schedule patch, and current cron patching preserves omitted stagger fields.

## Solution Assessment

No. Exposing the existing cron stagger through memory-core is the right direction, but the current PR is not the best complete solution until removal behavior matches the docs or the docs/tests intentionally define explicit `0` as the only clear path.
