## Summary

Keep this PR open: the new `dreaming.staggerMs` surface is useful and current head fixes the earlier `0` and blank-string parsing problems, but the patch still leaves a stale explicit stagger in place when the config field is removed.

## What This Changes

The PR adds `dreaming.staggerMs` to memory-core config/schema/docs/changelog and threads it through resolved dreaming config into the managed dreaming cron schedule.

## Best Possible Solution

Land a version that keeps the cron pass-through but makes set, unset, and `0` semantics explicit, either by resetting stale managed schedules to the cron default or by intentionally documenting a different removal workflow.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows `desiredStaggerMs === undefined` skips the schedule comparison, while `applyJobPatch` preserves omitted cron `staggerMs`. The original multi-instance operational scenario was not live-reproduced.

## Solution Assessment

No. Reusing the cron stagger infrastructure is the right direction, but this form is not mergeable until stale explicit stagger removal is either handled in code or deliberately accepted and documented by maintainers.
