## Summary

Keep open: this is a useful feature PR and current main/release still lack the memory-core `dreaming.staggerMs` surface, but the patch has a source-reproducible stale-stagger reset bug and lacks real behavior proof.

## What This Changes

The branch adds a memory-core `dreaming.staggerMs` config field to schema/docs/changelog and passes it from host config into the managed dreaming cron schedule.

## Best Possible Solution

Land a version that exposes the option with convergent set/unset/explicit-zero semantics, focused resolver/reconcile coverage, docs, and redacted real behavior proof.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows removing the config skips stagger reconciliation while current cron patching preserves omitted existing stagger values. The original multi-instance HA scenario was not live-reproduced.

## Solution Assessment

No. Reusing cron stagger infrastructure is the right direction, but this is not the best merge shape until stale explicit stagger removal is handled or deliberately documented as a maintainer-approved behavior.
