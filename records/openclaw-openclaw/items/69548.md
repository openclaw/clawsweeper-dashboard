## Summary

Keep open: the feature is useful and the PR now fixes the earlier `staggerMs: 0` and blank-string parser issues, but the current head can leave a stale explicit stagger after `dreaming.staggerMs` is removed and it still needs the required user-facing changelog entry.

## What This Changes

The PR adds `dreaming.staggerMs` to memory-core plugin metadata/docs, parses it in the memory host SDK, and passes it into the managed dreaming cron schedule.

## Best Possible Solution

Land a version that documents, implements, and tests the full `dreaming.staggerMs` lifecycle, including explicit `0`, unset/default behavior, removal/clear behavior, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create or update a managed dreaming cron job with an explicit `schedule.staggerMs`, then remove `dreaming.staggerMs`; the PR emits no clearing schedule patch and cron patching preserves the old value.

## Solution Assessment

No. Reusing the existing cron stagger infrastructure is the right direction, but the current patch is incomplete until removal semantics, tests, and the changelog are aligned.
