## Summary

Keep open. Current `main` still does not expose `dreaming.staggerMs`, and this PR is the active implementation path, but its current diff leaves stale explicit cron stagger values in place after the setting is removed even though the new docs say unset should return to the cron service default.

## What This Changes

The PR adds an optional `dreaming.staggerMs` config/schema/docs field and threads it through memory-core dreaming config into the managed dreaming cron schedule.

## Best Possible Solution

Ship the feature after the PR defines and tests the full config lifecycle for explicit values, `0`, unset, and removal. The most maintainable endpoint is for the docs, memory-core patch builder, and cron patch semantics to agree on whether field removal clears the override or whether `0` is the documented clear mechanism.

## Reproduction Assessment

Yes. A focused reproduction is an existing managed dreaming cron job with `schedule.staggerMs` set, followed by desired config where `dreaming.staggerMs` is removed; the PR emits no schedule patch and current cron patching preserves the old stagger on omit.

## Solution Assessment

No. The feature direction is appropriate because the cron subsystem already supports stagger windows, but the current PR is incomplete until the removal/clear semantics are either fixed or deliberately documented and covered by tests.
