## Summary

Keep open: the feature is useful and current main still lacks it, but the proposed patch has a source-reproducible stale-stagger bug and missing real behavior proof before merge.

## What This Changes

The PR adds a `dreaming.staggerMs` memory-core config option, docs/changelog text, and pass-through plumbing from host config into the managed dreaming cron schedule.

## Best Possible Solution

Land a version that keeps the config pass-through, preserves explicit `0`, and makes set/unset semantics converge with focused resolver/reconcile tests and user docs.

## Reproduction Assessment

Yes for the PR blocker: source inspection shows the PR skips stagger comparison when the desired config omits `staggerMs`, while current cron patching preserves an omitted existing stagger. The original HA stagger scenario was not live-reproduced.

## Solution Assessment

No. Reusing the cron stagger infrastructure is the right direction, but this patch is not the best merge shape until stale explicit stagger removal is handled in code or deliberately accepted and documented by maintainers.
