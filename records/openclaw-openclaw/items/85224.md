- `impact:crash-loop`: The issue is about process-level gateway availability during restart and update lifecycle handoff.

## Summary

Keep open: current main and v2026.5.20 still treat any non-blank macOS `XPC_SERVICE_NAME` as launchd supervision, and the update restart path exits after writing a supervised handoff, so the reported gateway-disappears path remains source-reproducible.

## What This Changes

The issue asks OpenClaw to stop treating inherited macOS launchd/XPC environment variables as proof that the gateway process itself is supervised by launchd.

## Best Possible Solution

Tighten Darwin supervisor detection so it proves this gateway owns a real OpenClaw LaunchAgent, using `OPENCLAW_LAUNCHD_LABEL`, expected gateway label matching, or a launchctl verification path, and cover inherited-XPC false positives in tests.

## Reproduction Assessment

Yes, at source level: Darwin with only `XPC_SERVICE_NAME` currently returns `launchd`, and the respawn helpers then take the supervised-exit path. A live macOS GUI/launchd replay was not performed in this read-only review.

## Solution Assessment

Yes, if the fix is scoped to supervisor detection and upgrade safety. The safest path is to reject inherited generic XPC state while preserving real OpenClaw LaunchAgent detection through an explicit marker, expected label match, or loaded-service check.
