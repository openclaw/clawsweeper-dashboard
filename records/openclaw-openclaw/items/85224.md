## Summary

Keep open: current main and the latest shipped release still classify inherited macOS `XPC_SERVICE_NAME` as launchd supervision, and the open linked fix PR is the active implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that macOS gateway update respawn can falsely choose the supervised launchd exit path when a gateway child inherits generic launchd/XPC environment from a parent app.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/85295 with the same narrow detector fix plus regression coverage for inherited `XPC_SERVICE_NAME` and the explicit `OPENCLAW_LAUNCHD_LABEL` launchd path.

## Reproduction Assessment

Yes for a source-level reproduction: on current main, Darwin plus only `XPC_SERVICE_NAME` makes `detectRespawnSupervisor` return `launchd`, and the update respawn helper then returns `mode: "supervised"`. A live macOS launchd/GUI run was not performed in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable direction is to stop treating generic inherited launchd/XPC environment as proof that the gateway itself is supervised and trust the OpenClaw-owned launchd marker for the respawn decision.
