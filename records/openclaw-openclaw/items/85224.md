## Summary

Keep open: current main and the latest shipped tag still source-reproduce the inherited `XPC_SERVICE_NAME` false-positive, and the linked fix PR is open rather than merged.

## What This Changes

Stop treating inherited macOS launchd/XPC environment as proof that the gateway itself is launchd-supervised during update respawn.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85295 or an equivalent narrow patch that trusts the OpenClaw-owned launchd marker for respawn supervision and adds inherited-XPC regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: current main classifies Darwin plus only `XPC_SERVICE_NAME` as `launchd`, and the update respawn helper then takes the supervised-exit path. I did not run a live macOS GUI/LaunchAgent replay in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is to stop using inherited generic launchd/XPC state as respawn-supervision proof and rely on the existing OpenClaw-owned launchd marker. The open linked PR follows that direction and should remain the implementation candidate until merged or rejected.
