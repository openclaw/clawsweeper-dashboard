## Summary

Keep open: current main and v2026.5.20 still classify any nonblank Darwin `XPC_SERVICE_NAME` as launchd supervision, and both restart helpers then take the supervised exit path. The report is source-reproducible with clear narrow fix targets rather than stale or duplicated by the related restart issues.

## What This Changes

The issue asks OpenClaw to stop treating inherited macOS `XPC_SERVICE_NAME` as proof that the gateway itself is registered as a launchd service during respawn/update restarts.

## Best Possible Solution

Narrow Darwin respawn-supervisor detection to an OpenClaw-owned launchd marker, optionally verified against a loaded label, while keeping generated LaunchAgents setting `OPENCLAW_LAUNCHD_LABEL` and adding regression tests for inherited XPC-only environments.

## Reproduction Assessment

Yes for a source-level reproduction: Darwin plus only `XPC_SERVICE_NAME` currently returns `launchd`, and the respawn helpers then take the supervised-exit path. A live macOS launchd/GUI replay was not run in this read-only review.

## Solution Assessment

Yes: the proposed direction is the narrowest maintainable fix if scoped to respawn-supervisor detection, because OpenClaw already writes `OPENCLAW_LAUNCHD_LABEL` for real gateway LaunchAgents.
