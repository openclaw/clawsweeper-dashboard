## Label Justifications

- `P1`: A macOS gateway update or SIGUSR1 restart can leave the gateway offline until manual LaunchAgent recovery.
- `impact:crash-loop`: The affected surface is gateway process availability during restart/update supervision.

## Summary

Keep open: current main and v2026.5.20 still classify macOS `XPC_SERVICE_NAME` alone as launchd supervision, so the reported gateway update/restart shutdown path remains source-reproducible and narrowly fixable.

## What This Changes

The issue requests narrowing macOS gateway respawn-supervisor detection so inherited launchd/XPC environment variables do not make an unmanaged child gateway exit as if launchd will restart it.

## Best Possible Solution

Narrow Darwin respawn-supervisor detection to an OpenClaw-owned gateway launchd identity, keep generated `OPENCLAW_LAUNCHD_LABEL` support, and add regression coverage showing inherited `XPC_SERVICE_NAME` no longer takes the supervised-exit path.

## Reproduction Assessment

Yes for a source-level reproduction: Darwin plus only `XPC_SERVICE_NAME` currently returns `launchd`, and the respawn helpers then take the supervised-exit path. I did not run a live macOS GUI/launchd replay in this read-only pass.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix if scoped to respawn-supervisor detection: trust the existing OpenClaw launchd marker for real gateway services and stop treating generic inherited XPC state as proof of gateway supervision.
