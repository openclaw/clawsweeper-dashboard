## Label Justifications

- `P1`: A macOS update or SIGUSR1 restart can silently take the gateway offline for real users when the process is not actually supervised by the gateway LaunchAgent.
- `impact:crash-loop`: The issue is about gateway process availability during restart/update handoff when the process exits expecting a supervisor that may not restart it.
- `impact:message-loss`: A missing gateway can leave chat bots, agents, and integrations offline, causing channel messages to be missed or undelivered.

## Summary

Keep open. Current main and v2026.5.20 still treat `XPC_SERVICE_NAME` alone as launchd supervision, so the reported macOS false-positive restart path remains source-reproducible; the prior merged launchd fix at https://github.com/openclaw/openclaw/pull/20555 solved the opposite launchd-thrash case but left this inherited-env failure mode real.

## What This Changes

The issue reports that macOS gateway update respawn can falsely choose the launchd-supervised path when generic launchd environment variables are inherited by a child gateway process.

## Best Possible Solution

Make macOS respawn supervision require a trusted OpenClaw gateway launchd marker or verified launchctl registration, keep LaunchAgent generation and doctor repair aligned, and add regression tests for the XPC-only inherited-env false positive.

## Reproduction Assessment

Yes at source level: `XPC_SERVICE_NAME` alone makes the Darwin detector return `launchd`, `respawnGatewayProcessForUpdate()` returns supervised, and the run loop writes a supervisor handoff before exiting. I did not run a live macOS launchd child-process repro in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to stop treating inherited generic XPC env as proof that this gateway process is registered with launchd. Current generated LaunchAgents already carry `OPENCLAW_LAUNCHD_LABEL`, and legacy compatibility can be handled by doctor repair or an explicit launchctl registration check.
