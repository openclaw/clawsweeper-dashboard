## What This Changes

The issue reports that macOS update respawn treats inherited `XPC_SERVICE_NAME` as launchd supervision, exits through the supervised handoff path, and can leave the gateway down.

## Best Possible Solution

Land one focused detector and regression-test fix that treats inherited macOS XPC state as unmanaged while preserving `OPENCLAW_LAUNCHD_LABEL` for real OpenClaw LaunchAgent supervision.

## Reproduction Assessment

Yes for a source-level reproduction: Darwin plus only inherited `XPC_SERVICE_NAME` currently returns `launchd`, and the update respawn helper then takes the supervised-exit path. A live macOS GUI/LaunchAgent replay was not run here.

## Solution Assessment

Yes, the narrow detector fix is the best direction because current service generation already provides an OpenClaw-owned launchd marker. The remaining action is to review and land one of the linked PRs rather than open another repair branch.
