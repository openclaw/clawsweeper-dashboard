## Summary

Keep open: the report is a fresh, source-plausible Telegram regression with user logs showing repeated outbound sends, and the related open report at https://github.com/openclaw/openclaw/issues/82329 covers part of the same preview/final-payload failure mode but not all of the cron/retry evidence here.

## What This Changes

The issue reports intermittent duplicate Telegram `sendMessage` calls for one assistant reply after OpenClaw 2026.5.12, especially on longer replies and cron-driven briefings.

## Best Possible Solution

Add preview/final-delivery idempotence so Telegram produces one visible final answer per assistant turn, with targeted coverage for partial-preview finals and cron direct delivery while preserving long-message chunking, unsent media, and genuine final-edit fallback behavior.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection shows a plausible current-main path for duplicate final payloads under Telegram partial preview, and the reporter supplied concrete gateway logs for separate outbound sends.

## Solution Assessment

No, a broad channel-adapter content-hash workaround is not the best first fix. The narrower maintainable fix should make preview/final payload delivery idempotent at the reply/Telegram delivery boundary, then add a guard only if cron retry proof shows it is still needed.
