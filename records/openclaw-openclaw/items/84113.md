## Label Justifications

- `P3`: The item is a low-urgency observability/product-contract split with canonical follow-up already open, not an urgent supported-workflow regression by itself.
- `impact:message-loss`: The reported symptom is a user turn that reaches the transcript/model path but is not visible in the live TUI chat log.
- `impact:session-state`: The live TUI view can diverge from the durable conversation transcript for externally or asynchronously appended turns.

## Summary

Close as superseded: the TUI live-rendering gap is already tracked by the open live transcript issue and implementation PR, while the external wake/live-session injection contract is tracked by the related security-sensitive API issue.

## What This Changes

The issue requests that externally injected user turns which reach the transcript/model path also appear live in the active TUI chat log with provenance and dedupe.

## Best Possible Solution

Track the external injection API under https://github.com/openclaw/openclaw/issues/83809 and the TUI live transcript rendering under https://github.com/openclaw/openclaw/issues/45388 / https://github.com/openclaw/openclaw/pull/73402, avoiding a wake-specific parallel event unless maintainers choose that contract.

## Reproduction Assessment

Yes for the source-level TUI gap: current main has no TUI subscription or dispatch path for `session.message`, and user rendering is limited to history reload/local sends. The exact wake-socket scenario is not a supported current-main path and remains part of the separate injection API decision.

## Solution Assessment

No as a standalone issue: the maintainable path is to land or replace the existing TUI `session.message` subscription work, and decide the external live-session injection contract separately. A new `wake.injected` event should not become a second source of truth unless maintainers explicitly choose it.
