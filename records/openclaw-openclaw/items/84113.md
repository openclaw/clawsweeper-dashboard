## Label Justifications

- `P3`: The request is a low-urgency wake-driven observability feature that depends on a new supported external injection contract rather than a documented behavior regression.
- `impact:message-loss`: The reported symptom is a user turn that reaches the transcript/model but is suppressed from the live TUI chat log.

## Summary

Keep open: current main does not implement the requested live rendering path, and the related external user-turn injection contract is still a product/security decision rather than a narrow automatic bug fix.

## What This Changes

The issue asks OpenClaw to render externally injected user turns in the TUI when a wake-socket style transport writes them into the active transcript.

## Best Possible Solution

Settle the supported external-injection contract with the related live-session injection issue, then render accepted external user turns in the active TUI session through the chosen scoped event with dedupe, provenance labeling, and regression tests.

## Reproduction Assessment

Yes for the UI gap from source: current main has no TUI dispatch path for `wake.injected` or `session.message`, and user-message rendering is only used for local sends and history reload. The exact wake-socket reproduction is not high-confidence because that injection surface is a local/proposed API, not a supported current-main feature.

## Solution Assessment

Unclear; the proposed wake-specific event may be useful, but the existing `session.message` transcript event path may be the narrower maintainable surface if maintainers accept external user-turn injection at all.
