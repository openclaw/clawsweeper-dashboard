## Summary

Keep open: current main still lacks the TUI live transcript rendering path, https://github.com/openclaw/openclaw/pull/73402 is already the open implementation candidate for the general TUI `session.message` gap, and the wake-specific injection/event contract remains a product/security decision tied to https://github.com/openclaw/openclaw/issues/83809.

## What This Changes

The issue requests live TUI rendering for externally injected user turns that already reach the transcript, including a proposed `wake.injected` event and source labeling.

## Best Possible Solution

Use the existing `sessions.messages.subscribe` / `session.message` contract for accepted transcript updates where possible, land or replace https://github.com/openclaw/openclaw/pull/73402 for the TUI subscriber lifecycle, and decide the external wake injection contract in https://github.com/openclaw/openclaw/issues/83809 before adding any wake-specific event.

## Reproduction Assessment

Yes for the source-level UI gap: current main has no TUI subscription or dispatch path for `session.message` or `wake.injected`, while history reload can render stored user turns. No for the exact wake-socket path, because that injection surface is a local/proposed API rather than supported current-main behavior.

## Solution Assessment

Unclear: the proposed wake-specific event may be useful, but the existing `session.message` transcript stream plus the open TUI subscription PR is the narrower maintainable path unless maintainers decide accepted wake injections need separate provenance events.
