## Summary

Keep open: current main still lacks the TUI subscription and dispatch path that would show accepted external transcript inserts live, and the wake-specific event/source contract depends on the related injection API decision rather than a narrow cleanup close.

## What This Changes

Request to broadcast accepted external user-turn injections and render them in openclaw-tui with active-session filtering, dedupe, and source labeling.

## Best Possible Solution

Use the existing `sessions.messages.subscribe` / `session.message` contract for live TUI transcript rendering where possible, keep the external injection/security contract in https://github.com/openclaw/openclaw/issues/83809, and add wake/source labeling only after that contract is settled.

## Reproduction Assessment

No for the exact wake-socket workflow: that injection path is a local/proposed surface, not supported current-main behavior. Yes at source level for the TUI gap: current main does not subscribe to or dispatch `session.message`/`wake.injected` events, while history reload can render stored user messages.

## Solution Assessment

Unclear. The proposed `wake.injected` event may be useful for provenance, but the existing `session.message` transcript subscription path and open TUI PR are the narrower maintainable rendering surface unless maintainers decide wake injections need distinct source metadata.
