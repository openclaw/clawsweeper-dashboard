## Summary

Keep open: current main still lacks a TUI subscription and handler for live transcript user-turn events, and the remaining path is split between an open TUI transcript PR and the external-injection product/security decision.

## What This Changes

The issue asks OpenClaw to render externally injected user turns in the live TUI chat log when those turns already reach the active conversation transcript.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/73402 for the generic live transcript rendering gap, then decide with https://github.com/openclaw/openclaw/issues/83809 whether wake-injected turns need a dedicated event, badge, or metadata contract.

## Reproduction Assessment

Yes for the source-level UI gap: current main has no TUI `session.message`/`wake.injected` dispatch or handler, and no TUI subscription call. The exact wake-socket flow is not a high-confidence runtime repro because the external injection surface is still a local/proposed API rather than supported current-main behavior.

## Solution Assessment

Unclear; a wake-specific event may be useful, but the existing `session.message` transcript event path and the open TUI subscription PR look like the narrower maintainable surface unless maintainers decide wake metadata needs a separate contract.
