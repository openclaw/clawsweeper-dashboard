## Label Justifications

- `P3`: The issue is a low-urgency wake-driven observability gap that depends on a new supported external-injection contract rather than an established core outage.
- `impact:message-loss`: The reported symptom is that an accepted user turn can be omitted from the live terminal TUI chat log.
- `impact:session-state`: The terminal TUI's visible state can diverge from the durable conversation transcript for externally injected turns.

## Summary

Keep open: current main and v2026.5.19 still leave the terminal TUI without a live transcript subscription or `session.message` dispatch path, while the wake-specific injection/event contract is still an open product and security decision tied to https://github.com/openclaw/openclaw/issues/83809. The open TUI live-transcript PR at https://github.com/openclaw/openclaw/pull/73402 is relevant but has not merged and does not by itself settle wake provenance/badge behavior.

## What This Changes

The issue requests a supported rendering path so externally injected user turns that reach the transcript also appear live in `openclaw-tui` with wake/source provenance.

## Best Possible Solution

Review or replace the generic terminal TUI `session.message` subscription fix, then decide wake-specific source/provenance rendering only after the external injection contract in https://github.com/openclaw/openclaw/issues/83809 is approved.

## Reproduction Assessment

Yes for the source-level terminal TUI visibility gap: current main has no TUI `session.message` subscription or dispatch path, while history reload can render stored user messages. No for the exact wake-socket flow, because that injection surface is still proposed/local rather than a supported current-main feature.

## Solution Assessment

Unclear as proposed: the existing `sessions.messages.subscribe` / `session.message` contract is likely the narrowest path for transcript updates, while a wake-specific event and provenance badge need a maintainer product/security decision first.
