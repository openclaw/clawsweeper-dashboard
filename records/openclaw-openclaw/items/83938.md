## Summary

The patch is a plausible fix for the linked backup restore bug, but this PR should stay open because the real-behavior proof is for the previous head and maintainers still need to accept the archive-size compatibility tradeoff from dereferencing hardlinks.

## What This Changes

The PR disables tar hardlink emission during backup creation, makes backup verification reject unsafe or missing hardlink targets, adds focused backup regression tests, and updates the changelog.

## Best Possible Solution

Land this direction after current-head proof is refreshed and maintainers accept the archive-size tradeoff, because regular file entries plus Link-target verification are a maintainable way to prevent archives that verify but fail broad extraction.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main lets node-tar emit Link entries while remapping only entry.path, and current backup verify ignores entry.linkpath; the linked issue supplies the macOS bsdtar failure shape.

## Solution Assessment

Yes, pending proof refresh. Dereferencing hardlinks during backup creation plus verifier checks for any remaining Link entries is narrower and safer than relying on restore tools to reinterpret unsafe tar link targets.
