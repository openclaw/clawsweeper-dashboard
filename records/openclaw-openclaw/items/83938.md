The PR makes backup creation dereference hardlinked files into regular archive entries, validates tar hardlink targets during backup verification, adds focused regression tests, and updates the changelog.

## Best Possible Solution

Land this direction after maintainer acceptance of the archive size/I/O and stricter verification tradeoff, keeping the hardlink emission and unsafe-target regression tests as the contract.

## Solution Assessment

Yes, with maintainer tradeoff acceptance. Dereferencing hardlinks at creation removes the restore-hostile archive shape, and verifier link-target checks prevent older or malformed archives from appearing healthy.

## Risks / Open Questions

- Dereferencing hardlinks can make hardlink-dense backups larger and increase archive write/read I/O, especially for state or workspace trees containing dependency directories.
- Backup verification will now fail archives with unsafe or missing Link targets that previously reported OK; that is likely intended, but it is still a user-visible compatibility change for older archives.
- The supplied real behavior proof is Linux-based; exact macOS bsdtar broad-extraction proof was not rerun, although the archive-shape proof removes the Link entries that caused the reported failure.
- The create-side fix relies on tar 7.5.15's internal linkCache behavior, so the regression tests are important upgrade guards for future tar changes.
