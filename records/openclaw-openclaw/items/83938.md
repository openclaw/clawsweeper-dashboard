## Label Justifications

- `P2`: This is a normal backup correctness bug with limited blast radius but meaningful recovery-confidence impact.
- `merge-risk: 🚨 compatibility`: Merging intentionally changes backup archive shape by replacing hardlink entries with regular file entries, which can increase size and I/O for existing hardlink-dense backups.

## Summary

Keep open for normal maintainer review: the current-head proof covers the central hardlink behavior and I found no blocking patch defect, but dereferencing hardlinks intentionally trades smaller hardlink-preserving archives for broader restore portability, so maintainers should accept that compatibility tradeoff before merge.

## What This Changes

The PR makes backup creation suppress tar hardlink entries, teaches backup verification to validate hardlink targets, adds hardlink regression tests, and records the fix in the changelog.

## Best Possible Solution

Land this direction after maintainer acceptance of the archive size/I/O tradeoff, keeping the tests that prove no Link entries are emitted and unsafe Link targets are rejected.

## Reproduction Assessment

Yes, source-reproducible: node-tar 7.5.15 emits Link entries from linkCache when files have nlink > 1, while current main remaps only entry.path and backup verify ignores entry.linkpath. The linked issue supplies the real macOS bsdtar failure shape.

## Solution Assessment

Yes, with maintainer tradeoff acceptance: disabling hardlink cache makes backup entries regular files, and verifier hardlink checks prevent older or malformed archives from looking healthy. A macOS bsdtar smoke would raise confidence but is not needed to identify the code path.
