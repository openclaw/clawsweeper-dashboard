## What This Changes

The PR adds `openclaw cron run --wait`, wait timeout and poll interval flags, exact `runId` filtering for `cron.runs`, protocol/docs/Swift model sync, and focused tests.

## Best Possible Solution

Land the opt-in wait mode if maintainers accept the CLI/protocol surface, keeping `cron.run` non-blocking while using exact `runId` history polling for wait semantics.

## Reproduction Assessment

Not applicable: this is an additive feature PR, not a broken existing behavior report. Source inspection of current main shows `cron run` returns after enqueue and `cron.runs` has no `runId` filter, while the PR body supplies after-fix live Gateway output.

## Solution Assessment

Yes, subject to maintainer approval: polling `cron.runs` by the returned manual `runId` is the narrow maintainable approach because it avoids changing the existing non-blocking Gateway `cron.run` RPC contract.
