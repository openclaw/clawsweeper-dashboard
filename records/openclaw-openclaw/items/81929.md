## Summary

Keep this PR open: the feature direction fits the existing non-blocking cron.run contract, but the proposed wait loop can exceed the documented timeout and the external PR currently has only test/mock proof rather than real Gateway cron behavior proof.

## What This Changes

The PR adds `openclaw cron run --wait`, timeout/poll flags, exact `runId` filtering for `cron.runs`, protocol/Swift model wiring, tests, and docs.

## Best Possible Solution

Keep the non-blocking Gateway RPC, but make the CLI wait loop cap sleeps to the remaining timeout, reject unsafe poll intervals, add focused coverage for that edge case, and require real cron-run proof before merge.

## Reproduction Assessment

Yes for the review finding by source inspection: if no terminal entry is returned and `pollIntervalMs` exceeds the remaining timeout, the loop sleeps the full interval before rechecking. No live Gateway cron reproduction was run in this read-only review.

## Solution Assessment

No, not yet. The feature boundary is maintainable, but the wait implementation must enforce the configured maximum wait time and the PR needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cap each wait sleep to the remaining timeout:** `src/cli/cron-cli/register.cron-simple.ts:70`
  - body: The loop checks `--wait-timeout` only before sleeping, then sleeps the full `pollIntervalMs`. With `--wait-timeout 1s --poll-interval 10m` and no terminal entry, the command will wait for the poll interval before noticing the timeout, violating the documented maximum wait time. Compute the remaining time and sleep only up to that amount, and reject or normalize a zero poll interval.
  - confidence: 0.9
