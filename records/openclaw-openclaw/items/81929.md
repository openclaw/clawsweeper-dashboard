## Summary

Keep open for maintainer handling: current main lacks this capability, the latest PR head addresses the poll-interval guardrail and has sufficient live proof, but this is an additive CLI/protocol feature with required checks still queued.

## What This Changes

This PR adds opt-in `cron run --wait` duration flags, exact `cron.runs` `runId` filtering, protocol/Swift/docs updates, and focused cron tests.

## Best Possible Solution

Land the opt-in wait mode if maintainers accept the CLI/protocol surface and checks finish green, while keeping `cron.run` itself non-blocking and using exact `runId` history polling.

## Reproduction Assessment

Not applicable: this is an additive feature PR, not a bug report. Source inspection shows current main lacks `--wait` and exact `runId` filtering, while the PR body provides after-fix live Gateway output for the new path.

## Solution Assessment

Yes, subject to maintainer approval: polling `cron.runs` by the returned `runId` is the narrow solution because it avoids reintroducing a blocking Gateway `cron.run` RPC. The latest head also rejects a zero poll interval before enqueueing.
