## Summary

Keep this PR open: it is a plausible additive CLI/protocol feature with supplied live Gateway proof, but the new wait loop can exceed the requested timeout by a full poll interval and the PR still has a failed required check.

## What This Changes

Adds opt-in `openclaw cron run --wait` with timeout and poll interval flags, exact `cron.runs --run-id` filtering, protocol/Swift/docs/tests/changelog updates.

## Best Possible Solution

Cap each wait-loop sleep to the remaining timeout, add a focused timeout regression, then land the opt-in wait mode if maintainers accept the public CLI/protocol surface and checks are green.

## Reproduction Assessment

Yes for the review finding: source inspection shows the loop checks timeout before sleeping and then sleeps the full poll interval. The overall PR is an additive feature, and the PR body supplies live after-fix Gateway output for the happy path.

## Solution Assessment

Mostly yes, but not as-is: polling `cron.runs` by the returned `runId` preserves the non-blocking `cron.run` contract. The wait loop should cap sleep to the remaining timeout so `--wait-timeout` remains a real bound.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Cap the wait sleep to the remaining timeout:** `src/cli/cron-cli/register.cron-simple.ts:77`
  - body: After an empty `cron.runs` page, the loop checks the timeout and then sleeps for the full poll interval. With `--wait-timeout 1s --poll-interval 10m`, a missing terminal entry can block for nearly 10 minutes instead of timing out around 1 second, which breaks the documented bounded wait behavior for shutdown hooks.
  - confidence: 0.89
