## Summary

Keep open: this is a useful additive feature with real proof and green checks, but the wait loop can overrun `--wait-timeout` by the full poll interval, so a small fix is needed before merge.

## What This Changes

The PR adds `openclaw cron run --wait` with timeout/poll flags, exact `cron.runs` `runId` filtering, protocol/Swift/docs updates, and focused tests.

## Best Possible Solution

Cap or race poll sleeps against the remaining wait timeout, add regression coverage for that edge, then land if maintainers accept the additive CLI/protocol surface.

## Reproduction Assessment

Yes for the review finding: source inspection shows the loop sleeps the full poll interval before rechecking elapsed time, so a large `--poll-interval` can exceed `--wait-timeout`. The main feature path has after-fix live Gateway proof in the PR body.

## Solution Assessment

No, not quite: polling `cron.runs` by exact `runId` is the right maintainable shape, but the wait loop should enforce the timeout boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Cap the wait sleep to the remaining timeout:** `src/cli/cron-cli/register.cron-simple.ts:77`
  - body: After a missed poll, this sleeps for the full `--poll-interval` before checking the deadline again. With `--wait-timeout 1s --poll-interval 10m`, the command can wait for about 10 minutes and even start another `cron.runs` RPC before timing out, so `--wait-timeout` is not actually a maximum wait. Cap the sleep to the remaining time or race the sleep/RPC loop against the deadline.
  - confidence: 0.86
