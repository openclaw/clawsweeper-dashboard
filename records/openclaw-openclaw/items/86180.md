## Label Justifications

- `P3`: The supported voice customization path is already shipped, and the remaining report is an unsupported config-key mismatch rather than a broad runtime outage.

## Summary

Current main and the latest release already support ElevenLabs voice customization through the documented `voiceId` field; the reported `voice` key is accepted only as provider-owned passthrough config and is not the OpenClaw/ElevenLabs TTS contract.

## What This Changes

The issue asks OpenClaw to honor a configured ElevenLabs voice for gateway Talk/TTS calls.

## Best Possible Solution

Keep the shipped `voiceId` contract and point users to `talk.providers.elevenlabs.voiceId` or `messages.tts.providers.elevenlabs.voiceId`, rather than adding a parallel `voice` alias.

## Reproduction Assessment

Yes for the exact unsupported-key behavior: source inspection shows `talk.providers.elevenlabs.voice` is preserved as provider-owned passthrough but the ElevenLabs provider reads `voiceId`. I did not run a live ElevenLabs call because this read-only review has no user API key, but the supported `voiceId` path is source- and release-proven.

## Solution Assessment

Yes: the maintainable solution is the existing `voiceId` field, which matches ElevenLabs' `voice_id` request path and is already documented and shipped. Adding a second `voice` config spelling would create a competing provider contract without a documented need.
