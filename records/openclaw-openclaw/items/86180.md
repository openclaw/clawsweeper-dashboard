## Summary

The report is a valid config-support mismatch, but not an OpenClaw source bug: ElevenLabs Talk TTS is documented and implemented with `voiceId`, while `voice` is only accepted as opaque provider-owned config and has no ElevenLabs semantics.

## What This Changes

The issue asks for `talk.providers.elevenlabs.voice` to select the ElevenLabs TTS voice used by Talk mode.

## Best Possible Solution

Keep the documented `voiceId` contract for ElevenLabs Talk TTS and direct users to that config key unless maintainers explicitly decide to add a `voice` alias.

## Reproduction Assessment

Yes. Source inspection shows `talk.providers.elevenlabs.voice` is preserved as an unknown provider-owned key but ignored by the ElevenLabs resolver, while `voiceId` is the implemented and documented path.

## Solution Assessment

No. Treating `voice` as an ElevenLabs alias would add a new public config compatibility path; the narrower maintainable answer is to use the shipped `voiceId` contract unless maintainers choose an alias intentionally.
