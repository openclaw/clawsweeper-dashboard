- `impact:auth-provider`: The issue concerns routing `image_generate` through configured provider ids, model refs, and credentials.

## Summary

Keep open: current main and v2026.6.6 still resolve `image_generate` through plugin-registered image-generation providers, so LiteLLM is covered but arbitrary custom `models.providers.<id>` entries remain unsupported without a provider plugin or a new core/config contract.

## What This Changes

Request `image_generate` support for custom or third-party OpenAI-compatible providers configured under `models.providers`, beyond the currently registered provider/plugin set.

## Best Possible Solution

Keep this as the canonical product/API design item: maintainers should choose between a guarded explicit custom-provider image capability, plugin-owned registration, or documentation that custom routes should use LiteLLM or a provider plugin.

## Reproduction Assessment

Yes. Source inspection shows a custom `provider/model` ref can become an image-generation candidate, then fail when the provider id is absent from the plugin image-provider registry; I did not run a live custom endpoint.

## Solution Assessment

Unclear. The report is valid, but the best fix needs a maintainer decision between auto-detection, an explicit config capability, or a plugin-owned image provider path; LiteLLM is already solved through the bundled plugin.
