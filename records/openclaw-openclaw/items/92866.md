## Label Justifications

- `impact:auth-provider`: The issue concerns provider/model routing and credential-bearing image-generation requests for configured custom provider IDs.

## Summary

Keep open: current main and the latest release still route `image_generate` only through plugin-registered image-generation providers; LiteLLM is covered, but arbitrary custom `models.providers.<id>` image endpoints remain a product/API decision.

## What This Changes

The issue requests routing `image_generate` through custom or third-party OpenAI-compatible providers declared in `models.providers`, with LiteLLM support as one acceptable subcase.

## Best Possible Solution

Define one canonical custom image-provider contract: either an explicit safe config/API opt-in for OpenAI-compatible image endpoints, or document LiteLLM and plugin-owned `registerImageGenerationProvider` as the supported path.

## Reproduction Assessment

Yes from source inspection: a custom `provider/model` image candidate can reach `generateImage`, then fail because the provider id is not registered through `imageGenerationProviders`. I did not run a live third-party endpoint in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. LiteLLM already has a plugin-owned route, while arbitrary custom providers need a deliberate choice between auto-detection, explicit config opt-in, or plugin-only capability registration.
