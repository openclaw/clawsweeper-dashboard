## Summary

Keep open: current main and v2026.6.8 solve the LiteLLM subcase, but arbitrary custom `models.providers.<id>` entries still do not become `image_generate` providers unless a plugin registers an image-generation capability for that provider id.

## What This Changes

The issue requests `image_generate` support for custom or third-party image providers configured under `models.providers`, including OpenAI-compatible proxies and LiteLLM-style routes.

## Best Possible Solution

Keep this open for maintainer product/API review, then implement one generic contract that routes custom image providers through the existing plugin/provider boundary and shared OpenAI-compatible image helper with explicit security and config semantics.

## Reproduction Assessment

Yes, for the source-level behavior: a custom `provider/model` can become an image-generation candidate, then current main fails at provider lookup because the provider id is not registered through `imageGenerationProviders`. I did not run a live third-party endpoint.

## Solution Assessment

Unclear until maintainers choose the product contract. The narrowest maintainable direction is likely a generic OpenAI-compatible image-provider path or explicit opt-in capability, not another provider-specific workaround or re-solving LiteLLM.
