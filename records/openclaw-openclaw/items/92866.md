## Label Justifications

- `P2`: This is a normal provider-routing feature gap with limited blast radius, not a core outage or urgent regression.
- `impact:auth-provider`: The issue is about routing an image-generation tool through configured provider ids, model refs, and credentials.

## Summary

Keep open: current main still routes `image_generate` through registered image-generation plugin capability providers, so arbitrary custom `models.providers.<id>` entries can still hit the reported missing-provider path; the LiteLLM subcase is already implemented, but the generic custom-provider contract remains a maintainer product/API decision.

## What This Changes

The issue requests support for routing `image_generate` through custom or third-party providers configured under `models.providers`, rather than only built-in image-generation providers.

## Best Possible Solution

Keep this open for maintainers to choose a generic custom-provider image-generation contract, then implement it with explicit auth/network semantics, docs, and focused source plus real endpoint validation.

## Reproduction Assessment

Yes, from source inspection: configure `agents.defaults.imageGenerationModel.primary` or the tool `model` override to `custom/model` without a registered image-generation provider, and current `generateImage` will produce the reported missing-provider error. I did not run a live third-party endpoint repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. The clean end state is either a generic OpenAI-compatible image-provider path with explicit security/auth rules, or a documented requirement to use LiteLLM or a plugin-owned `registerImageGenerationProvider` implementation.
