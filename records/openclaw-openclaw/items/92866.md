## Summary

Keep open: current main fixes the LiteLLM subcase, but arbitrary custom `models.providers.<id>` entries still do not become `image_generate` providers unless a plugin registers the image-generation capability.

## What This Changes

The issue requests routing `image_generate` through custom or third-party OpenAI-compatible providers configured under `models.providers`, including proxy/router providers.

## Best Possible Solution

Maintainers should choose a generic custom-provider image-generation contract, or explicitly document that users should use LiteLLM or plugin-owned `registerImageGenerationProvider` implementations.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `agents.defaults.imageGenerationModel.primary` to an unregistered custom `provider/model`, then the candidate reaches `generateImage` and fails provider lookup with the reported registry error.

## Solution Assessment

Unclear. LiteLLM is already solved by a plugin-owned provider, but arbitrary custom providers need a maintainer decision between auto-detection, explicit config opt-in, or requiring provider plugins.
