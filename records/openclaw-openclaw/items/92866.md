- `impact:auth-provider`: The issue concerns provider/model routing and credential-bearing image-generation requests for configured provider ids.

## Summary

Keep open: current main still does not let arbitrary `models.providers.<id>` entries register as `image_generate` providers. The LiteLLM subcase is now covered by a bundled plugin, and StepFun/Google Vertex have provider-specific follow-ups, but the broader custom-provider contract remains a maintainer product/API decision.

## What This Changes

Request support for custom or third-party OpenAI-compatible image endpoints in `image_generate`, instead of requiring a bundled/plugin-registered image provider id.

## Best Possible Solution

Define one canonical custom image-provider contract with explicit auth, endpoint, SSRF, fallback, and config semantics, or document LiteLLM/plugin-owned `registerImageGenerationProvider` as the supported path.

## Reproduction Assessment

Yes, from source inspection: a custom `provider/model` image candidate reaches `generateImage`, then fails when `getImageGenerationProvider` finds no plugin-registered image provider for that provider id. I did not run a live third-party endpoint in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. The existing plugin capability path is coherent, but first-class custom `models.providers` image generation needs a deliberate API/config decision rather than a narrow bug patch.
