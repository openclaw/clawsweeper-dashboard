- `impact:auth-provider`: The issue concerns provider/model routing and credential-bearing image-generation requests for configured custom provider IDs.

## Summary

Keep open: current main and latest v2026.6.8 still route `image_generate` only through registered image-generation capability providers, so the LiteLLM subcase is solved but arbitrary custom `models.providers.<id>` image endpoints remain a product/API decision.

## What This Changes

Request custom or third-party OpenAI-compatible `models.providers` entries to be usable by the shared `image_generate` tool instead of requiring a built-in or plugin-registered image provider.

## Best Possible Solution

Maintain one canonical decision here: either define a generic custom OpenAI-compatible image-provider contract with explicit auth, SSRF, endpoint, and config behavior, or document LiteLLM and plugin-owned `registerImageGenerationProvider` as the supported path.

## Reproduction Assessment

Yes from source inspection: a custom `provider/model` image candidate can be parsed, then current `generateImage` fails when `getImageGenerationProvider` finds no plugin-registered image provider for that provider ID. I did not run a live third-party endpoint in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. LiteLLM is already solved through a bundled plugin, while arbitrary custom providers require a deliberate choice between auto-detection, explicit config, or plugin-only capability registration.
