## Summary

Keep open: current main and v2026.6.9 still resolve `image_generate` through plugin-declared image-generation providers, not arbitrary custom `models.providers` entries. LiteLLM is already covered, and the StepFun work only partially overlaps, so the remaining generic custom-provider contract still needs maintainer direction.

## What This Changes

The issue requests `image_generate` support for custom and third-party OpenAI-compatible image providers configured under `models.providers`, instead of only plugin-registered image providers.

## Best Possible Solution

Define one canonical custom image-provider contract: either a safe explicit config/API opt-in for OpenAI-compatible image endpoints, or a documented plugin/ClawHub route for provider-owned image generation while keeping LiteLLM as the supported router path.

## Reproduction Assessment

Yes, at source level. A configured `imageGenerationModel.primary` such as `custom/image-01` reaches `generateImage`, then fails because `getImageGenerationProvider("custom")` only resolves plugin-registered image providers; I did not run a live third-party endpoint.

## Solution Assessment

Unclear until maintainers choose the product contract. The right owner boundary is either a deliberate core config/API extension or a documented plugin path, not an automatic registry fallback chosen without auth, SSRF, schema, and docs decisions.
