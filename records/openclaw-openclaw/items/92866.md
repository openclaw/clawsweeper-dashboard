## Label Justifications

- `P2`: This is a normal provider-routing feature gap with clear source evidence and limited blast radius, not an urgent outage or security emergency.
- `impact:auth-provider`: The issue concerns routing `image_generate` through configured provider ids, model choices, credentials, and provider HTTP settings.

## Summary

Keep open: current main and v2026.6.8 still do not register arbitrary `models.providers.<id>` entries as image-generation providers; LiteLLM is covered by a bundled plugin, but the remaining generic custom-provider behavior needs a maintainer API/config/security decision.

## What This Changes

Request support for routing `image_generate` through configured custom or third-party OpenAI-compatible model providers instead of only plugin-registered image-generation providers.

## Best Possible Solution

Keep this issue open for maintainers to choose the generic contract: either add an explicit OpenAI-compatible image capability for custom providers, define an opt-in config/schema path with security rules, or document that users should use LiteLLM or a plugin-owned image provider.

## Reproduction Assessment

Yes, from source inspection: configure `imageGenerationModel.primary` as an unregistered custom provider/model and `generateImage` will parse it as a candidate, then fail when the image provider registry cannot resolve that provider id. I did not run a live third-party endpoint repro.

## Solution Assessment

Unclear until maintainers choose the product contract. LiteLLM is already solved through a bundled plugin, while arbitrary custom providers need a deliberate decision between auto-detection, explicit opt-in config, or plugin-owned registration.
