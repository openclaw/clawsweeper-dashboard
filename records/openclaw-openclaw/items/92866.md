## Summary

Keep open: LiteLLM image generation is now supported, but current main and the latest release still require image-generation providers to come from plugin capability registration, so arbitrary custom `models.providers.<id>` OpenAI-compatible image endpoints still do not route through `image_generate`.

## What This Changes

Request support for custom or third-party OpenAI-compatible image-generation providers in `image_generate`, beyond the currently registered image provider plugins.

## Best Possible Solution

Define one maintainer-approved provider capability path for custom image generation, reusing the plugin-owned OpenAI-compatible image helper where possible and documenting whether arbitrary `models.providers` entries are supported or must use LiteLLM/a plugin.

## Reproduction Assessment

Yes from source inspection: a custom provider/model ref can become an image-generation candidate, then `generateImage` fails when `getImageGenerationProvider` cannot find that provider in the plugin capability registry. I did not run a live custom endpoint repro.

## Solution Assessment

Unclear until maintainers pick the contract; the LiteLLM route is already solved, while arbitrary custom provider ids need a product/API decision between auto-detection, explicit config opt-in, or plugin-owned registration.
