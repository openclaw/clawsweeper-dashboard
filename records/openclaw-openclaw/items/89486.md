## Summary

Keep this PR open, but the current patch is not merge-ready because it forwards a new helper parameter that no live heartbeat caller supplies. The linked context-engine bug can still occur on current heartbeat runs, and the supplied proof is unit-test output rather than a real heartbeat context-engine run.

## What This Changes

Adds optional `isHeartbeat` parameters to two context-engine `afterTurn` helper paths and conditionally forwards them when supplied.

## Best Possible Solution

Wire the heartbeat state at the embedded-run caller boundary into the context-engine lifecycle, add focused regression coverage, and include redacted real heartbeat-run proof before merge.

## Reproduction Assessment

Yes by source inspection: current main declares `afterTurn(params).isHeartbeat`, heartbeat runs carry `isHeartbeat` into the embedded run as a heartbeat trigger, and the `afterTurn` call still omits it unless an unwired helper parameter is supplied. I did not run a live heartbeat repro in this read-only review.

## Solution Assessment

No; forwarding `params.isHeartbeat` inside the helper is only a plausible partial fix because no real caller passes that parameter. The best fix is to derive the flag from the existing heartbeat trigger/run-kind at the caller boundary and cover that production path.
