## Summary

Keep open. Current main still reuses the pre-read startup config snapshot across in-process restart iterations, and https://github.com/openclaw/openclaw/pull/79951 is an open fix PR that explicitly references this issue, so the issue should remain open until that PR merges or is otherwise resolved.

## What This Changes

The issue asks OpenClaw to prevent in-process gateway restarts from reusing a stale `startupConfigSnapshotRead` and overwriting user config written after the original gateway boot.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79951, or an equivalent first-start-only snapshot fix, with regression coverage and real runtime proof that post-boot config writes survive an in-process restart.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: write config after the initial gateway pre-read, trigger an in-process restart, and the next loop iteration still injects the original snapshot instead of rereading disk.

## Solution Assessment

Yes. Gating the pre-read snapshot to the first gateway start is the narrowest maintainable fix because it preserves the cold-start optimization while making restart iterations use the normal disk snapshot loader.
