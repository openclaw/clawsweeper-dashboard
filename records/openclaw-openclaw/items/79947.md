## Summary

Keep open. Current main still reuses the same `startupConfigSnapshotRead` for every gateway-loop start callback, and the issue is already paired with an open closing PR, so the issue should remain open until that fix or an equivalent one merges.

## What This Changes

The issue asks OpenClaw to stop in-process gateway restarts from reusing a stale startup config snapshot that can overwrite user-written config after the first boot.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79951 or an equivalent first-start-only snapshot fix with regression proof, then close this issue after merge.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: write config after the initial CLI pre-read, trigger an in-process restart, and the next loop iteration still injects the original snapshot instead of rereading disk.

## Solution Assessment

Yes. Gating `startupConfigSnapshotRead` to only the first start preserves the cold-start optimization while making restart iterations use the normal disk snapshot loader.
