## Summary

Keep open. Current main still reuses a pre-read `startupConfigSnapshotRead` across every in-process restart start callback, and server startup honors that injected snapshot instead of rereading disk, so the reported config-loss bug remains a narrow valid issue.

## What This Changes

The issue reports that container/default in-process gateway restarts reuse a stale startup config snapshot and can overwrite user config written after the initial gateway boot.

## Best Possible Solution

Keep the first-start config pre-read optimization, but consume it only on the initial gateway start; later in-process restart iterations should reread current disk config and have focused regression plus Docker/SIGUSR1 proof when feasible.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: write config after the initial `gateway run` pre-read, trigger an in-process restart, and the next loop iteration still injects the original snapshot; I did not execute the Docker scenario in this read-only review.

## Solution Assessment

Yes. The proposed first-start-only use of `startupConfigSnapshotRead` is the narrowest maintainable fix because it preserves the cold-start optimization while forcing restart iterations through the normal disk snapshot loader.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79947. Current main reads `startupConfigSnapshotRead` once in `runGatewayCommand` and passes the same snapshot into every `runGatewayLoop` start callback, so in-process restarts can boot from stale config and later write back stale source state. Preserve the first-start pre-read optimization, but ensure later in-process restart iterations do not pass `startupConfigSnapshotRead` and therefore reread `openclaw.json` through `loadGatewayStartupConfigSnapshot`. Add focused regression coverage proving the first start can receive the pre-read snapshot while a second in-process restart start does not. Do not change config schema, restart policy, provider/plugin defaults, or unrelated writeback behavior. Add a user-facing changelog fix entry. Coordinate mentally with https://github.com/openclaw/openclaw/pull/44811, which overlaps runtime snapshot writeback but does not replace the restart-loop fix.
