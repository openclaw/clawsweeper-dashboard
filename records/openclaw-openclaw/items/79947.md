## Summary

Keep open. Current main still captures `startupConfigSnapshotRead` once before the gateway loop and passes that same snapshot into every restart start callback, while server startup honors that injected snapshot instead of rereading disk.

## What This Changes

The issue asks OpenClaw to make in-process gateway restarts reread the latest config instead of reusing the original startup snapshot and risking user config loss.

## Best Possible Solution

Keep the first-start config pre-read optimization, but ensure every later in-process restart startup rereads the current on-disk config and has regression coverage for a post-boot config change before SIGUSR1.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: write config after the initial `gateway run` pre-read, trigger an in-process restart, and the next loop iteration still injects the original snapshot. I did not execute the Docker scenario, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes, the proposed first-start-only use of `startupConfigSnapshotRead` is the narrowest maintainable fix because it preserves the cold-start optimization while forcing restart iterations through the normal disk snapshot loader.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79947. Current main reads `startupConfigSnapshotRead` once in `src/cli/gateway-cli/run.ts` before `runGatewayLoop`, then passes that same snapshot to every `startGatewayServer` call after in-process SIGUSR1 restart. Change the gateway run path so the CLI pre-read snapshot is supplied only for the first server start; subsequent in-process restart iterations must omit `startupConfigSnapshotRead` so `loadGatewayStartupConfigSnapshot` rereads the current `openclaw.json` from disk. Keep the first-start optimization and do not change the container restart mode, supervisor detection, broad config writer semantics, or unrelated stale-snapshot behavior. Add focused regression coverage, likely in `src/cli/gateway-cli/run.option-collisions.test.ts`, by stubbing/mocking `runGatewayLoop` or its start callback to invoke startup twice and asserting the first `startGatewayServer` call receives `startupConfigSnapshotRead` while the second omits it. If the test shape is awkward there, add a narrow CLI gateway restart-start test with the same assertion. Add a single-line `CHANGELOG.md` fix entry because this is user-visible config-loss behavior. Treat https://github.com/openclaw/openclaw/pull/44811 as adjacent context only; do not try to solve the broader runtime writeback PR in this fix.
