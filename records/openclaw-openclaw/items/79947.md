## Summary

Keep open because current main still has the reported stale snapshot path, and an open linked fix PR already targets the bug but has not merged.

## What This Changes

The issue reports that in-process gateway restarts reuse a startup config snapshot captured before the restart loop and can overwrite user config changes made after initial boot.

## Best Possible Solution

Keep the first-start pre-read optimization, but make every later in-process restart reread the current on-disk config and cover that with regression or live gateway proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: write config after the initial pre-read, trigger an in-process restart, and the repeated start callback still injects the original snapshot while startup skips the disk reread. I did not live-run the Docker container scenario during this read-only review.

## Solution Assessment

Yes, first-start-only use of `startupConfigSnapshotRead` is the narrowest maintainable fix because it preserves cold-start optimization while forcing restart iterations through the normal disk snapshot loader. The open linked PR should be reviewed and landed or rejected before opening another repair lane.
