## Summary

Keep open: current main still has the nodes CLI invalid-timeout gap and this PR targets the right runtime path, but the branch still changes the shared timeout error contract and has no after-fix real behavior proof.

## What This Changes

The branch routes the nodes CLI gateway timeout through `parseTimeoutMsWithFallback`, tightens timeout parsing to reject non-decimal strings, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Update the branch to reuse current main's `invalidTimeout()` wording while keeping strict positive integer parsing, then add redacted CLI output showing invalid nodes `--timeout` input fails before the gateway call.

## Reproduction Assessment

Yes, by source inspection. Current main forwards `Number(opts.timeout ?? 10_000)` from the nodes CLI runtime, and `callGateway` treats non-finite timeout values as unset, so invalid timeout input can bypass CLI validation.

## Solution Assessment

Mostly yes. Reusing the shared timeout parser in the nodes runtime is the narrow maintainable fix, but the PR should preserve current main's shared invalid-timeout error wording instead of adding a separate lower-case error path.
