## Summary

Keep open: the updated branch reuses the shared Windows codepage map, but its non-Windows fallback still returns the first GBK decode for every invalid UTF-8 buffer, so the advertised Big5/Shift_JIS/EUC-KR fallback policy is not real. The external PR also still lacks redacted real OpenClaw read-tool proof against an actual GBK file.

## What This Changes

Adds a session read-tool decoding helper, exports the Windows codepage map, routes read buffers through the helper, and adds GBK/BOM/UTF-8 tests.

## Best Possible Solution

Land one canonical narrow fix that reuses or extracts the strict-UTF-8-then-Windows-active-codepage behavior for the session read path, preserves non-Windows behavior unless a broader encoding policy is explicitly chosen, and includes real read-tool proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 decode session read-tool text buffers as UTF-8, and the reported GBK bytes reproduce mojibake under that decode. I did not run a live Chinese Windows OpenClaw session in this read-only review.

## Solution Assessment

No: the Windows-side direction is plausible, but the best fix should not add a non-Windows charset guessing loop that always returns the first successful GBK decode. The safer path is the existing active-codepage contract plus focused real proof.
