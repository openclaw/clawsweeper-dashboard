## Summary

Keep open: the runtime direction is now close to the right fix, but the PR still has an introduced Windows-locale test defect, insufficient real Windows read-tool proof, and multiple competing open branches for the same linked bug.

## What This Changes

The PR adds a read-tool file decoding helper, wires the session read tool through it, and adds encoding regression tests for BOM, UTF-8, non-Windows fallback, and win32 GBK behavior.

## Best Possible Solution

Land one canonical narrow read-tool fix that reuses the existing Windows active-codepage decoder, keeps non-Windows UTF-8 behavior stable, uses deterministic regression coverage, and has redacted real Windows read-tool proof before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 decode session read-tool buffers with UTF-8 only, and the linked GBK byte sample mojibakes under that path. I did not run a live Chinese Windows OpenClaw session in this read-only review.

## Solution Assessment

Mostly yes for the runtime direction: the revised branch delegates Windows invalid-UTF-8 decoding to the existing active-codepage helper, which is the right owner boundary for the reported bug. The test coverage must stop assuming every Windows host is CP936, and the PR still needs real Windows proof plus duplicate-PR coordination before merge.
