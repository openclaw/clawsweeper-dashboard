## What This Changes

The PR adds a `TrajectoryConfig` type and `OpenClawConfig.trajectory`, plus unrelated glob/wildcard handling for `contracts.tools` and a plugin contract test.

## Best Possible Solution

Retarget the branch to one concern: either implement a schema-backed `pi-trajectory-flush` cleanup timeout for #75839 with docs, tests, changelog, and real proof, or split `contracts.tools` wildcard support into a separate design-reviewed PR.

## Reproduction Assessment

## Solution Assessment

No. The current patch is not the narrowest maintainable solution because it neither wires the advertised timeout into the failing cleanup path nor keeps the unrelated plugin manifest wildcard feature separate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Register trajectory in the strict config schema:** `src/config/types.openclaw.ts:141`
  - body: Adding `trajectory?: TrajectoryConfig` to the TypeScript shape is not enough: `OpenClawSchema` is still `.strict()` and has no `trajectory` property, so the documented config example will be rejected by schema-backed validation.
  - confidence: 0.96
- **[P2] Wire flushTimeoutMs into pi-trajectory-flush:** `src/config/types.trajectory.ts:10`
  - body: This new `flushTimeoutMs` type is never read by the runtime or passed to `runAgentCleanupStep`, while the reported failure is the cleanup step timing out at the default 10000 ms. Users setting this option would still hit the same timeout.
  - confidence: 0.94
- **[P2] Match requested tools against contract globs:** `src/plugins/tools.ts:367-374`
  - body: For a plugin declaring `contracts.tools: ["xxx-*"]`, a request allowlist containing `xxx-test` will not load the plugin because this code compiles the allowlist and tests it against the manifest pattern string. The contract patterns need to be matched against requested tool names, not the other way around.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The trajectory type-only change has no supply-chain issue, but the unrelated wildcard contract change weakens plugin tool ownership checks without approved semantics.

Concerns:

- **[medium] Broad wildcard tool claims bypass exact ownership checks:** `src/plugins/tool-contracts.ts:23`

Because manifest tool ownership controls diagnostics and which plugin runtimes can own tools, this needs documented conflict and permission semantics before merge.
