## Summary

Keep open: the latest force-push fixed the earlier schema and final-timer problems, but the timer still fires only once instead of periodically, and the supplied terminal proof does not show `flushTimeoutMs` configured or trajectory flushing observed in a real run.

## What This Changes

The PR adds `trajectory.flushTimeoutMs` plus an environment fallback, registers the root config schema key, schedules timer-driven trajectory writer flushes, and adds a fake-timer runtime test.

## Best Possible Solution

If maintainers accept this knob, implement lifecycle-safe repeating flush semantics, or target the cleanup timeout directly if that is the real problem, then document the config/env surface, add a changelog entry, and provide real configured runtime proof.

## Reproduction Assessment

Yes, for the PR defect by source inspection: the PR head schedules one timeout at recorder initialization, but the callback never schedules the next tick. The added fake-timer test advances only one interval, so it does not exercise periodic behavior.

## Solution Assessment

No. A configurable flush policy may be useful, but this patch still does not implement repeated periodic flushes and needs docs, changelog, real changed-behavior proof, and maintainer confirmation that this is the right surface.
