## Summary

Keep open. The updated PR is focused on trajectory periodic flushing, but it still has blocking implementation gaps: the new config key is not accepted by the strict config schema, final flush can leave a live timer behind, and the external PR only provides fake-timer test output rather than real behavior proof.

## What This Changes

The PR adds `OpenClawConfig.trajectory.flushTimeoutMs`, resolves it from config or `OPENCLAW_TRAJECTORY_FLUSH_TIMEOUT_MS`, schedules periodic trajectory writer flushes, and adds a fake-timer unit test.

## Best Possible Solution

Retarget this to a schema-backed, documented trajectory flush policy with safe timer cleanup, and prove whether periodic flushing or a configurable `pi-trajectory-flush` cleanup timeout is the intended fix for the related operator reports.

## Reproduction Assessment

Yes for the PR defects by source inspection: the strict config schema does not include `trajectory`, and the proposed `flush()` path reschedules a timer after draining. I did not run tests because this is a read-only review.

## Solution Assessment

No. Periodic writer flushing may be useful, but this implementation is not the narrowest maintainable solution until the config contract, docs/changelog, timer lifecycle, and real configured-run proof are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Register the new config key in the schema:** `src/config/types.openclaw.ts:117`
  - body: The PR advertises `openclaw.json` support by adding `trajectory?: TrajectoryConfig`, but OpenClaw validates config with a strict `OpenClawSchema` and this branch does not add a `trajectory` property there. A user setting `trajectory.flushTimeoutMs` in the config file will fail schema-backed validation before the runtime can read it.
  - confidence: 0.96
- **[P2] Do not reschedule a live timer after final flush:** `src/trajectory/runtime.ts:417-420`
  - body: After `flush()` drains and removes the shared writer, this code immediately calls `scheduleFlush()` again. That creates a ref'ed `setTimeout` even when the recorder is done, so a configured one-shot CLI or cleanup run can stay alive until `flushTimeoutMs` expires; keep the timer active only while recording, or unref/cancel it on final cleanup.
  - confidence: 0.9
- **[P3] Document the new trajectory setting:** `src/config/types.trajectory.ts:1-10`
  - body: This adds a user-facing config/env surface, but the trajectory docs and changelog do not explain the key, env var, units, or operational tradeoff. Users trying to tune the timeout from the related reports would not have a supported configuration reference, and repo policy requires a changelog entry for user-facing features.
  - confidence: 0.86
