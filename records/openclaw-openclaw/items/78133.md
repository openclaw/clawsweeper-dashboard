## Summary

Keep open: the PR adds a new trajectory flush config/env surface, but the current patch is not mergeable because the config key is type-only, the timer behaves like an idle debounce, and the supplied proof does not show the new setting working in a real run.

## What This Changes

The branch adds `trajectory.flushTimeoutMs` and `OPENCLAW_TRAJECTORY_FLUSH_TIMEOUT_MS`, schedules timer-driven trajectory writer flushes, and adds a fake-timer recorder test.

## Best Possible Solution

Accept a flush knob only after maintainers confirm it is the right surface, then implement it through schema/docs/changelog with lifecycle-safe periodic semantics that continue during active event streams and stop after final flush.

## Reproduction Assessment

Yes for the PR defects by source inspection: the strict root config schema lacks `trajectory`, and the patch resets the timer on each event and schedules again after final flush. I did not run tests because this was a read-only review.

## Solution Assessment

No. A configurable flush policy may be useful, but this implementation is not the narrowest maintainable solution until schema/docs/changelog, timer semantics, lifecycle cleanup, and real changed-behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Register the new config key in the schema:** `src/config/types.openclaw.ts:120`
  - body: Adding `trajectory?: TrajectoryConfig` to the TypeScript type does not make it valid user config. Config files are validated through the strict `OpenClawSchema`, which has no `trajectory` root property, so `openclaw.json` with `trajectory.flushTimeoutMs` will be rejected before the runtime can read it.
  - confidence: 0.96
- **[P2] Keep periodic flushing independent of event cadence:** `src/trajectory/runtime.ts:393-397`
  - body: This cancels and restarts the timeout after every recorded event. A long reasoning run that keeps emitting events more frequently than `flushTimeoutMs` will never flush at that interval, so the advertised periodic in-flight flush behaves like an idle debounce.
  - confidence: 0.91
- **[P2] Stop scheduling timers after final flush:** `src/trajectory/runtime.ts:421-423`
  - body: `flush()` drains the writer and removes the shared writer entry, then schedules another timeout. A finalized one-shot recorder can keep a ref'ed timer alive until it fires, and that timer can call `writer.flush()` after the recorder has already been finalized.
  - confidence: 0.88
- **[P3] Document the new trajectory setting:** `src/config/types.trajectory.ts:1-10`
  - body: This adds a user-facing config and env surface, but the trajectory docs, config help, and changelog do not describe the key, env var, units, or operational tradeoff. Operators would have no supported reference for tuning it.
  - confidence: 0.86
