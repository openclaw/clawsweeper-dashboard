## Summary

Keep open: this PR needs changes before merge because the config key is type-only, the timer behaves like a debounce and can leave a post-flush timer alive, and the only proof is fake-timer unit output.

## What This Changes

Adds `trajectory.flushTimeoutMs` and `OPENCLAW_TRAJECTORY_FLUSH_TIMEOUT_MS` handling to the trajectory runtime, schedules timer-driven writer flushes, and adds a fake-timer test.

## Best Possible Solution

Make any trajectory-flush knob schema-backed, documented, and lifecycle-safe, then align it with the open cleanup-timeout/session-list work instead of landing a timer that does not match the reported behavior.

## Reproduction Assessment

Yes for the PR defects by source inspection: the strict root config schema does not include `trajectory`, and the PR timer cancels/restarts on every event and reschedules after final flush. I did not run tests because this was a read-only review.

## Solution Assessment

No. A configurable flush policy may be useful, but this implementation is not the best solution until schema/docs/changelog, timer semantics, lifecycle cleanup, and real configured-run proof are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Register the new config key in the schema:** `src/config/types.openclaw.ts:120`
  - body: The PR adds `trajectory?: TrajectoryConfig` to the TypeScript config type, but config files are validated through the strict `OpenClawSchema`. Since this branch does not add a `trajectory` schema property, `openclaw.json` with `trajectory.flushTimeoutMs` will be rejected before the runtime can read it.
  - confidence: 0.96
- **[P2] Keep periodic flushing independent of event cadence:** `src/trajectory/runtime.ts:393-397`
  - body: This code cancels and restarts the timeout after every recorded event, so a long run that keeps producing trajectory events more often than `flushTimeoutMs` will never flush at that interval. That makes the advertised periodic in-flight flush behave like an idle debounce instead.
  - confidence: 0.91
- **[P2] Do not reschedule a timer after final flush:** `src/trajectory/runtime.ts:421-423`
  - body: After `flush()` drains the writer and removes the shared writer entry, it immediately schedules another timeout. A configured one-shot run can keep a ref'ed timer alive until the timeout expires, and that timer can call `writer.flush()` after the recorder has already been finalized.
  - confidence: 0.88
- **[P3] Document the new trajectory setting:** `src/config/types.trajectory.ts:1-10`
  - body: This adds a user-facing config/env surface, but the trajectory docs, config help/labels, and changelog do not describe the key, env var, units, or operational tradeoff. Users from the related reports would not have a supported reference for tuning it.
  - confidence: 0.86
