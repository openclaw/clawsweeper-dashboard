## Summary

Keep open: the patch looks like a plausible narrow LM Studio auth fix and the earlier header-auth regression appears addressed, but the external PR still lacks after-fix proof from a real LM Studio setup.

## What This Changes

The PR resolves LM Studio configured API-key env templates through SecretInput handling, preserves Authorization-header auth when templates are unset, and adds focused runtime/setup regressions plus a changelog entry.

## Best Possible Solution

Keep this PR scoped to the LM Studio auth-resolution fix, require real LM Studio proof before merge, and leave native endpoint/fallback behavior to the linked issue or a separate product/API decision.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the env-template auth slice: on current main, `${LMSTUDIO_API_KEY}` is normalized to an unknown marker and returned as a literal API key instead of being resolved through SecretInput. I did not run a live LM Studio server in this read-only review.

## Solution Assessment

Yes for the auth slice: using the shared SecretInput resolver in the LM Studio provider is the narrow maintainable fix, and the later `allowUnresolved` handling preserves existing Authorization-header precedence. It is not a solution for the linked issue's separate endpoint fallback request.
