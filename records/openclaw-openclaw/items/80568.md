## Summary

Keep open: this targets a real LM Studio SecretInput bug, but the patch currently regresses existing header-only auth by throwing on an unset apiKey template before honoring an Authorization header, and the external PR still lacks real behavior proof.

## What This Changes

The branch changes LM Studio runtime API-key resolution to pass configured string apiKey values through SecretInput resolution, adds runtime regressions, and adds a changelog entry.

## Best Possible Solution

Keep the narrow env-template auth fix, but preserve header-only auth by deferring or ignoring unresolved apiKey templates when Authorization headers satisfy auth, with regression coverage and redacted real LM Studio proof before merge.

## Reproduction Assessment

Yes for the auth paths from source: current main treats arbitrary ${ENV} apiKey strings through the known-marker shortcut, and the PR’s new throw path would fail when headers.Authorization is configured with an unset apiKey template. I did not run a live LM Studio server reproduction.

## Solution Assessment

No, not as currently written. Resolving arbitrary env-template API keys is the right narrow fix, but the implementation should not throw before existing header-only auth can satisfy runtime/discovery auth.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91
