## Summary

Keep open: the PR addresses a real source-level LM Studio env-template auth gap, but current head still has a blocking discovery regression when profile/CLI discovery auth exists, and external real behavior proof is still missing.

## What This Changes

The PR updates LM Studio runtime/setup auth resolution to route configured apiKey env templates through SecretInput, preserve Authorization-header auth when templates are unresolved, add regressions, and add a changelog entry.

## Best Possible Solution

Keep the SecretInput-based auth fix, also allow unresolved configured apiKey templates when profile/CLI discovery auth already satisfies discovery, then merge only after redacted real LM Studio behavior proof is added.

## Reproduction Assessment

Yes, by source inspection: current main strips `${LMSTUDIO_API_KEY}` to `LMSTUDIO_API_KEY` and only treats registered env markers as env vars, so a custom template can become a literal bearer token. I did not run a live LM Studio reproduction in this read-only review.

## Solution Assessment

No as currently patched: resolving configured strings through SecretInput is the right direction, but discovery should also tolerate an unresolved configured template when `ctx.resolveProviderApiKey` already supplied discovery auth.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Skip unresolved templates when a discovery key exists:** `extensions/lmstudio/src/setup.ts:782`
  - body: `allowUnresolved` only follows Authorization-header auth, so an unset configured template returns `null` from discovery before `discoveryApiKey` can be used. This regresses profile/CLI-authenticated discovery when a stale optional `apiKey` template is present; treat the discovery key as satisfying discovery auth or defer the configured-template error until no alternate auth exists.
