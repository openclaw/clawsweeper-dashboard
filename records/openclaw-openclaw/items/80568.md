## Summary

Keep open: the PR targets a real LM Studio SecretInput gap, but it still lacks real behavior proof and introduces a likely header-auth fallback regression around unresolved `apiKey` templates.

## What This Changes

The PR changes LM Studio `apiKey` resolution to pass configured string keys through SecretInput resolution, adds env-template regression tests, and updates the changelog.

## Best Possible Solution

Keep the env-template `apiKey` fix, but preserve existing Authorization-header fallback semantics and add a regression for header auth plus an unset `apiKey` template before merge.

## Reproduction Assessment

Yes. A focused source-level repro is `headers.Authorization` plus `apiKey: "${LM_API_TOKEN}"` or another `${ENV}` template with the env unset: the PR throws in `resolveLmstudioConfiguredApiKey` before `resolveLmstudioRuntimeApiKey` can use the existing header-auth fallback.

## Solution Assessment

No. Resolving arbitrary env-template keys is the right direction, but the current implementation should defer or soften unresolved configured-key errors when an Authorization header already supplies LM Studio auth.
