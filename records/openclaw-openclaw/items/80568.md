## What This Changes

The PR resolves LM Studio configured `apiKey` strings through SecretInput, adjusts header-auth discovery handling, adds runtime/setup regressions, and updates the changelog.

## Best Possible Solution

Fix the discovery-key fallback regression, add focused coverage for unresolved optional `apiKey` templates with profile/CLI discovery auth, and then merge after redacted real LM Studio proof is provided.

## Reproduction Assessment

Yes, by source inspection. Current main can treat `${LMSTUDIO_API_KEY}` as a literal bearer token, and the PR head can return `null` before using an existing `discoveryApiKey` when that template is unset.

## Solution Assessment

No. Reusing SecretInput is the right direction, but tying unresolved-template tolerance only to Authorization headers is too narrow because discovery auth can also be satisfied by an existing profile/CLI discovery key.
