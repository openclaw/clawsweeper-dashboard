## Summary

I found no blocking correctness or security findings, and the supplied live gateway output plus focused regressions are sufficient for the changed boundary.

## What This Changes

The branch promotes complete agent-scoped `to` values into session-key selection for CLI gateway, generic gateway, and embedded agent paths, with regression coverage for dispatch, reset acknowledgements, and delivery sanitization.

## Best Possible Solution

Merge the selector-boundary fix after maintainer review and normal checks, keeping promotion limited to complete agent-scoped keys and preserving explicit `sessionKey` / `sessionId` precedence.

## Reproduction Assessment

Yes, source-reproducible: current main derives session selection from `sessionKey`/`sessionId` and later uses `to` as a delivery/turn-source value, so a full agent session key in `to` is not selected as the session. I did not run tests in the read-only checkout.

## Solution Assessment

Yes. Promoting only complete agent-scoped keys at the session-selection boundary and clearing the consumed `to` before delivery is the narrow owner-boundary fix; a WeChat-specific workaround or delivery-target parser change would be the wrong layer.
