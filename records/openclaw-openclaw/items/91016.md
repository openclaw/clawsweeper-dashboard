## Summary

Keep open: current `main` and the latest release do not prove the DeepSeek cache regression is fixed. Source inspection still shows DeepSeek model metadata is not opted into OpenAI-style `prompt_cache_key`, while DeepSeek's own cache contract is automatic prefix caching with `prompt_cache_hit_tokens`, so the right repair needs maintainer/provider proof rather than an automatic source-only change.

## What This Changes

The issue asks OpenClaw to restore DeepSeek prompt-cache effectiveness for upgraded users after v2026.6.1, including provider defaults, upgrade handling, and DeepSeek-appropriate cache behavior.

## Best Possible Solution

Keep this open for a provider-owner investigation with live DeepSeek repeated-prefix proof, then change the DeepSeek provider metadata, doctor/migration guidance, or cache-shaping strategy only if that proof shows it restores cache hits safely for upgraded and fresh configs.
