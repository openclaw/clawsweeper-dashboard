The issue remains necessary: current main still gates `prompt_cache_key` behind explicit compatibility metadata, bundled DeepSeek models do not set it, and another user corroborates the post-upgrade cache collapse. ## Best Possible Solution

Compare identical DeepSeek requests on 2026.5.7, 2026.6.1, and current main while capturing redacted serialized prefixes and DeepSeek's native cache-hit/miss counters, then make the narrow provider-owned prompt-stability or usage-parsing fix without globally enabling unsupported cache fields.

## Solution Assessment

No, automatically inserting `supportsPromptCacheKey` is not yet the best proven fix. DeepSeek's documented contract uses automatic prefix caching, so prompt serialization, stable-prefix construction, and usage parsing should be isolated before choosing the correction.
