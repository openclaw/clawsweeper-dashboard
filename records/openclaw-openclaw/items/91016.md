## Summary

Keep open: this is the earliest copy of a plausible v2026.6.1 DeepSeek prompt-cache regression, and current `main` plus the latest release still leave the direct DeepSeek catalog without the cache-key compat flag while the transport gates `prompt_cache_key` on that flag. The exact fix needs maintainer/provider proof because DeepSeek documents automatic prefix caching and `prompt_cache_hit_tokens`/`prompt_cache_miss_tokens`, not OpenAI `prompt_cache_key` or `prompt_tokens_details.cached_tokens`. ([api-docs.deepseek.com](https://api-docs.deepseek.com/guides/kv_cache))

## What This Changes

The issue reports that upgrading to v2026.6.1 broke DeepSeek V4 Flash prompt-cache reuse for an OpenAI-compatible completions setup, causing high cache misses and cost.

## Best Possible Solution

Keep this earliest report open as the canonical issue, close the later duplicate copies to it, and have maintainers verify DeepSeek V4 repeated-prefix behavior on v2026.6.1 versus current `main` before choosing a provider-local catalog/docs/request-shaping fix.

## Reproduction Assessment

No for the full DeepSeek 0-cache-hit claim: we have a source-reproducible missing-cache-key path in the bundled DeepSeek catalog, but not a live DeepSeek API trace proving that this is the cause. The reporter's field expectation also conflicts with DeepSeek's documented `prompt_cache_hit_tokens` usage fields.

## Solution Assessment

Unclear: adding `supportsPromptCacheKey` is a plausible narrow repair for OpenClaw's cache-key gate, but the best fix may instead be DeepSeek-specific docs, usage-field clarification, or provider-local request shaping after live API proof.
