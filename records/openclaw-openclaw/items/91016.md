## Summary

Keep this open: the reported upgrade regression is not fixed or disproven on current main or v2026.6.5. OpenClaw parses DeepSeek’s documented cache-hit counter, but the bundled DeepSeek live suite does not test repeated-prefix cache behavior, and official DeepSeek documentation indicates caching is automatic rather than controlled by OpenAI’s `prompt_cache_key`; the same-author English companion at https://github.com/openclaw/openclaw/issues/91018 tracks the identical unresolved report.

## What This Changes

Investigate and fix the reported DeepSeek prompt-cache regression after upgrading to OpenClaw 2026.6.1.

## Best Possible Solution

Use one canonical issue to add a DeepSeek repeated-prefix live regression that captures raw `prompt_cache_hit_tokens` and `prompt_cache_miss_tokens`, normalized usage, and stable-prefix request traces across v2026.5.7 and current main; repair prompt stability or provider-owned metadata only after that result identifies the actual regression.

## Reproduction Assessment

No, not for the full issue: the upgrade path and source gates are concrete, but a high-confidence reproduction requires repeated identical-prefix DeepSeek calls on current main with raw cache-hit and cache-miss counters. The corroborating user report raises confidence that the regression is real without identifying its precise cause.

## Solution Assessment

No, the proposed unconditional `supportsPromptCacheKey` default is not yet the best-supported fix because DeepSeek documents automatic prefix caching and the reporter says manual opt-in did not restore hits. The narrow maintainable solution must follow live evidence toward either prompt-prefix stability or provider-specific metadata.
