## Summary

Keep open: current main still has the LINE inbound media hang, and the PR is a focused repair with sufficient handler-level proof. The remaining blocker is maintainer acceptance of the fixed timeout and delivery tradeoff, not a concrete contributor-facing code defect.

## What This Changes

The PR adds a 30 second per-chunk idle timeout around LINE `getMessageContent` and the returned media stream, plus regression tests for stalled headers, stalled chunks, cleanup, and slow-progress media.

## Best Possible Solution

Land the focused LINE timeout if maintainers accept the fixed 30 second idle cutoff and non-abortable SDK tradeoff, leaving configurability or an overall deadline for follow-up only if real LINE traffic needs it.

## Reproduction Assessment

Yes from source inspection and the contributor's handler-level proof: current main awaits LINE media before `processMessage`, and stalled SDK headers or body reads have no deadline. I did not run a live LINE CDN stall reproduction in this read-only review.

## Solution Assessment

Yes, with a maintainer policy decision: a channel-local idle timeout is a narrow mitigation for the hang and matches the Telegram-style ceiling. If maintainers dislike fixed fail-open attachment behavior, the safer alternative is a configurable cutoff or overall deadline before merge.
