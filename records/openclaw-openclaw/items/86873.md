## Summary

Keep open for normal PR review: current main still has an unbounded LINE media download path and the patch is focused, but the external PR proof is mock-only and the new 30s attachment timeout has a message-delivery tradeoff maintainers should explicitly accept before merge.

## What This Changes

This PR adds a 30s per-chunk idle timeout around LINE inbound media downloads and tests stalled header/body stream behavior.

## Best Possible Solution

Land a LINE-owned timeout fix once the PR has real after-fix proof and maintainers accept the 30s idle attachment-skip behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main awaits getMessageContent and saveMediaStream without a deadline before processMessage. A never-settling response or stream therefore blocks LINE inbound dispatch on that path.

## Solution Assessment

Mostly yes: wrapping the LINE SDK promise and Readable with an idle deadline is a narrow owner-local fix and matches existing remote-media timeout patterns. The merge should still wait for real proof and maintainer acceptance of the 30s media-skip default.
