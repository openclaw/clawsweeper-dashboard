## Summary

Keep open: current main and the latest release still have the unbounded LINE media download path, while this PR is a focused fix with no concrete patch defect found. The remaining blocker is maintainer acceptance of the fixed 30 second LINE idle timeout and its attachment-delivery tradeoff.

## What This Changes

The PR adds a 30 second headers/per-chunk idle timeout around LINE inbound media downloads and regression tests for stalled headers, stalled chunks, iterator cleanup, and slow-progress streams.

## Best Possible Solution

Land the focused LINE timeout if maintainers accept the fixed idle cutoff; request a configurable cutoff or overall deadline only if maintainers want different delivery semantics.

## Reproduction Assessment

Yes. Current-main source awaits LINE media before processMessage with no deadline, and @line/bot-sdk exposes getMessageContent as a Readable without an AbortSignal hook; the PR body also supplies exact-head handler-level stall proof.

## Solution Assessment

Yes, with maintainer policy acceptance. A channel-local idle timeout is the narrowest maintainable mitigation I found; bypassing the LINE SDK for AbortSignal support or adding an overall/configurable deadline would be broader policy work.
