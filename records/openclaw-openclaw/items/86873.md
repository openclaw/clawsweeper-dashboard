## What This Changes

The PR adds a 30s per-chunk idle timeout around LINE inbound media downloads and regression coverage for stalled headers, stalled chunks, partial stalls, slow progress, and iterator cleanup.

## Best Possible Solution

Land the LINE-owned idle guard with regression tests once maintainers accept the fixed media-omission semantics, keeping broader WhatsApp/MSTeams, config-knob, or overall-deadline work separate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: make LINE getMessageContent or its returned stream never settle, and current main awaits downloadLineMedia before processMessage.

## Solution Assessment

Yes. The timeout belongs in the LINE-owned download helper, preserves slow-but-progressing transfers, and reuses the existing caller catch path; maintainers only need to accept the fixed 30s media-omission tradeoff.
