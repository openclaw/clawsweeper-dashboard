## Summary

Keep open: the LINE media timeout fixes a real inbound dispatch hang, but the patch still awaits `Readable` iterator cleanup after a timeout, and a local `Readable` probe shows that cleanup can remain pending before the error reaches the caller. ## What This Changes

This PR adds a hardcoded 30-second idle timeout around LINE inbound media `getMessageContent` and stream reads, plus focused `downloadLineMedia` regression tests.

## Best Possible Solution

Repair the timeout wrapper so cleanup cannot block the timeout rejection, cover the real Node `Readable` path, and then refresh the LINE handler proof before merge.

## Reproduction Assessment

Yes for the PR defect: source inspection plus a local Node `Readable` probe show that awaiting `iterator.return()` can remain pending after a timed-out read. The original LINE CDN stall is not live-reproduced here, but the current and proposed code paths are clear.

## Solution Assessment

No. The timeout direction is right, but the implementation must not await cleanup that can hang on the same stalled `Readable`; a fire-and-forget or otherwise bounded cleanup path with a real `Readable` regression is safer.
