## Summary

Keep this PR open for maintainer review. Current main still has an unbounded LINE inbound media download path, and the patch is a focused mitigation with sufficient terminal proof; the remaining decision is whether maintainers accept the fixed 30s per-chunk cutoff and skipped-attachment semantics.

## What This Changes

Adds a LINE-local 30s headers/per-chunk idle timeout around inbound media downloads and regression coverage for stalled headers/body streams.

## Best Possible Solution

Land the focused LINE-local idle timeout if maintainers accept the fixed 30s per-chunk policy; otherwise request a config or overall-deadline adjustment before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main awaits LINE `getMessageContent` and stream persistence before `processMessage` with no deadline, so a never-settling headers/body stream blocks dispatch; I did not run a live LINE CDN stall.

## Solution Assessment

Yes, with maintainer policy acceptance. A LINE-local idle timeout is the narrowest owner-boundary mitigation I found; bypassing the LINE SDK for abortable transport would be broader, while configurable or overall-deadline behavior is a maintainer policy choice.
