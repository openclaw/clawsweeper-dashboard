## Summary

Keep open. The documented `logging.file` placeholder still has a source-level gap on current `main`, and this PR targets the right area, but the branch needs a correctness fix for current home-prefix behavior plus real runtime proof before merge.

## What This Changes

The PR expands `YYYY-MM-DD` in configured `logging.file` paths and adds logger settings and log-tail regression tests.

## Best Possible Solution

Land a rebased narrow logging resolver that expands `~` and `YYYY-MM-DD` consistently for settings, writes, and tailing, with focused regression tests and redacted runtime proof.

## Reproduction Assessment

Yes, at source level: configure `logging.file` with `openclaw-YYYY-MM-DD.log`; current main can still expose the literal placeholder to settings and log tailing while writes use the dated active file. I did not run a live OpenClaw repro in this read-only review.

## Solution Assessment

No, not as submitted. Placeholder expansion is the right fix direction, but the PR must preserve current `~` expansion and include real after-fix runtime proof.
