## Summary

Keep open: the logging placeholder gap is still source-reproducible on current main, and this PR targets the right surface, but the submitted patch is not merge-ready because it drops current leading-tilde expansion and still lacks real behavior proof.

## What This Changes

The PR resolves `YYYY-MM-DD` in configured `logging.file` paths and adds logger settings plus log-tail regression tests.

## Best Possible Solution

Use one shared log-file resolver that expands a leading `~`, substitutes the host-local date for the documented template, preserves rolling pruning and log-tail behavior, and is covered by focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes, source-reproducible. Current docs show the dated `logging.file` pattern, while current settings/log-tail code can still follow the literal placeholder instead of the active dated file; I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. Placeholder resolution belongs in the logger path, but the patch must preserve current `~` expansion and be rebased with real behavior proof before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect
