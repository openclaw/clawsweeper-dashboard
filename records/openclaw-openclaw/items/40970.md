## Summary

Keep open. Current main and the latest release still leave configured `logging.file` values literal, so this PR is not obsolete. The branch is useful but not merge-ready because the new tests hard-code UTC calendar dates despite the documented host-local date contract, and the placeholder resolver is not shared with daemon status output.

## What This Changes

The PR adds `YYYY-MM-DD` placeholder resolution for configured logging file paths in `src/logging/logger.ts` and adds logger settings plus log-tail regression tests.

## Best Possible Solution

Land a focused shared logging-path resolver, likely in `src/logging/log-file-path.ts` or a nearby logging helper, and use it consistently for logger writes, log tailing, and daemon status while keeping date tests pinned to host-local calendar dates that are stable in every supported timezone.

## Reproduction Assessment

Yes. Current main reproduces by source inspection because a configured `logging.file` with `YYYY-MM-DD` is returned unchanged, and the PR test regressions reproduce with Node date probes under `America/Los_Angeles` and `Pacific/Honolulu`.

## Solution Assessment

No, not as written. The runtime direction is narrow and useful, but the resolver should be shared with status/log-tail surfaces and the tests should use local-safe instants or derive expected filenames from the same host-local date contract.
