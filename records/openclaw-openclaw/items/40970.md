## Summary

Keep this PR open: the direction is sound for current main because the logger can write dated rolling files, but log-tail/settings surfaces still see a literal `YYYY-MM-DD` placeholder. The remaining blocker I found is a required changelog entry for the user-facing logging fix.

## What This Changes

The PR resolves `YYYY-MM-DD` placeholders in configured logging file paths through logger settings/active-file resolution and adds focused logger settings and log-tail tests.

## Best Possible Solution

Land a narrow version that centralizes configured log-file date resolution for settings, writing, and tailing, keeps the focused tests, and adds the required changelog entry before merge.

## Reproduction Assessment

Yes, from source inspection. Current main returns the literal configured placeholder from logger settings while log tail only recognizes concrete dated filenames, so a configured `openclaw-YYYY-MM-DD.log` can write to the active dated file but tail the unresolved path.

## Solution Assessment

Yes, with one small process fix. Resolving the placeholder in the logger settings path is the narrowest maintainable approach because the log-tail and channel-log surfaces already consume `getResolvedLoggerSettings()`, but the PR needs a changelog entry before merge.
