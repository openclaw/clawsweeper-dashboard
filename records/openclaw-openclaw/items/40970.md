## What This Changes

The PR resolves `YYYY-MM-DD` placeholders in `logging.file` before exposing logger settings, building loggers, and tailing configured logs, with logger/log-tail tests.

## Best Possible Solution

Land a narrow logging fix that resolves the documented placeholder before settings are consumed by tail/startup surfaces, preserves existing rolling-file behavior, adds timezone-stable coverage, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes, from source inspection: with `logging.file` set to the documented `openclaw-YYYY-MM-DD.log` form, current main exposes the literal placeholder from `getResolvedLoggerSettings()` while log tailing only resolves already-expanded digit-dated basenames. I did not run the Vitest path because `pnpm` is unavailable.

## Solution Assessment

Mostly yes: resolving the placeholder in logger settings is the narrow shared seam for startup and log-tail consumers. The PR still needs the required changelog entry and a timezone-stable date fixture before merge.
