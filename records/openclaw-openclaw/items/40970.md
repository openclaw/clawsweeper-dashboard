## Summary

Keep open: current main still has the documented `logging.file` date-placeholder gap, but this PR should not merge yet because it regresses existing leading-tilde expansion and lacks required after-fix real behavior proof.

## What This Changes

The PR adds `YYYY-MM-DD` substitution for configured logging file paths and tests logger settings plus log-tail behavior.

## Best Possible Solution

Use one shared log-file resolver that expands a leading `~`, substitutes the host-local `YYYY-MM-DD` date for the documented template before writes and tails, preserves rolling pruning, and is covered by focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes, source-reproducible. Current docs advertise the dated `logging.file` pattern, while current settings and log-tail code can still follow the literal placeholder instead of the active dated file; no tests were run because this review is read-only.

## Solution Assessment

No, not as submitted. Placeholder resolution belongs in the logger/log-tail path, but the patch must preserve current `~` expansion, rebase cleanly, and include real behavior proof before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
