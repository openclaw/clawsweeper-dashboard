## Summary

Keep open: current main still has the documented `YYYY-MM-DD` logging-file gap, but this PR regresses leading-`~` expansion and lacks after-fix real behavior proof, so it needs contributor rework before merge.

## What This Changes

The PR changes logging file resolution so configured paths containing `YYYY-MM-DD` resolve to the current local date before writes and log-tail reads, with focused logger tests.

## Best Possible Solution

Use one shared log-file resolver that expands a leading `~`, substitutes the host-local `YYYY-MM-DD` date before writes and tails, preserves rolling pruning, and adds focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes, from source inspection: current docs advertise `logging.file` with `openclaw-YYYY-MM-DD.log`, while current main passes the configured value through settings and log tailing without substituting the placeholder. I did not run a filesystem repro in this read-only review.

## Solution Assessment

No, not as submitted. Placeholder resolution belongs in this logging path, but the patch must preserve existing `~` expansion, rebase cleanly, and include after-fix real behavior proof before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve home-prefix expansion in log-file resolution:** `src/logging/logger.ts:691-697`
