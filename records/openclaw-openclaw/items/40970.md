## Summary

Keep open: current main still leaves a literal `YYYY-MM-DD` path visible to logger settings and log-tail, while the PR’s latest helper regresses existing leading-tilde expansion and lacks real runtime proof.

## What This Changes

The PR adds `YYYY-MM-DD` date-placeholder resolution for configured logging file paths in `src/logging/logger.ts` and covers logger settings plus log-tail behavior with tests.

## Best Possible Solution

Use one shared logger path resolver that expands leading `~`, substitutes the host-local date, preserves rolling-log pruning behavior, and is covered by settings, tail, and tilde regression tests before requiring redacted real-run proof.

## Reproduction Assessment

Yes, source-reproducible. Current main documents the dated `logging.file` pattern, but logger settings can still return the literal placeholder and log-tail follows that unresolved path.

## Solution Assessment

No, not as currently proposed. Resolving the placeholder through a shared helper is the right direction, but the helper must preserve leading-tilde expansion and the PR needs real behavior proof before merge.
