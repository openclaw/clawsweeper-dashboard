## Summary

Keep open: current main still has a source-visible gap for the documented `logging.file` date placeholder in resolved settings/log tailing, but this PR is not mergeable as submitted because it regresses existing `~` expansion and lacks real behavior proof.

## What This Changes

The PR adds `YYYY-MM-DD` substitution for configured `logging.file` paths in logger resolution and adds logger-settings plus log-tail tests.

## Best Possible Solution

Centralize configured log-file resolution so settings, writes, `logs.tail`, and channel log reads use the same date-expanded and home-expanded active path, with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current docs advertise the dated `logging.file` pattern, while current settings and log-tail code can still route through the literal placeholder path; I did not run tests because this was a read-only review.

## Solution Assessment

No, not as submitted. Placeholder resolution belongs in the logger/log-tail path, but this implementation must preserve leading-`~` expansion and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve home expansion when resolving log files:** `src/logging/logger.ts:690-697`
