## Summary

Keep open: the underlying logging placeholder and log-tail gap is still present on current main, but this PR is not merge-ready because it conflicts with main, regresses the current `~` expansion contract, and still lacks contributor-supplied real behavior proof.

## What This Changes

The PR changes logging resolution and tests so `logging.file` paths containing `YYYY-MM-DD` resolve to the current local date for logger settings, writes, and log-tail reads.

## Best Possible Solution

Rebase the narrow logging fix onto current main, keep `expandHomePrefix` inside a shared resolver used by settings, writes, and tails, and add focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes, from source inspection: docs show `logging.file` with `openclaw-YYYY-MM-DD.log`, but current settings can return the literal placeholder and log tailing only recognizes digit-dated rolling names. I did not run a filesystem repro because this review is read-only.

## Solution Assessment

No, not as submitted. Placeholder resolution belongs in the logging path, but the branch needs to preserve leading `~` expansion, rebase cleanly, and include after-fix real behavior proof.
