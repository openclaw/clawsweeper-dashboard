## Summary

Keep open: current main still has a source-level gap for the documented `YYYY-MM-DD` logging.file placeholder in settings/log tailing, but this PR cannot merge as-is because it regresses leading-tilde expansion and lacks real behavior proof.

## What This Changes

The PR expands `YYYY-MM-DD` placeholders in configured `logging.file` paths and adds logger settings plus log-tail regression tests.

## Best Possible Solution

Land a rebased narrow logging resolver that expands `~` and `YYYY-MM-DD` consistently for settings, writes, and tailing, with focused regression tests and redacted runtime proof.

## Reproduction Assessment

Yes, at source level: docs advertise `openclaw-YYYY-MM-DD.log`, current main returns the configured file unchanged in settings, and log tailing only recognizes digit-dated rolling files. I did not run a live OpenClaw repro in this read-only review.

## Solution Assessment

No, not as submitted. Placeholder expansion is the right direction, but the resolver must preserve leading-tilde expansion and needs real after-fix runtime proof before merge.
