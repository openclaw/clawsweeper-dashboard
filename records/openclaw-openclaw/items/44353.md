## Summary

Keep open: current main has the generic fallback runner behavior needed once fallback candidates are present, but normal spawned subagents still resolve fallback candidates from agent/default model policy instead of the selected `subagents.model.fallbacks`, so the latest concrete reproduction remains valid.

## What This Changes

The issue asks OpenClaw to use configured fallback models when a primary provider/model fails, including Bedrock provider errors and subagent model fallback lists.

## Best Possible Solution

Thread the selected subagent fallback policy into normal spawned-subagent execution so `runWithModelFallback` receives the same fallback chain that selected the child primary.

## Reproduction Assessment

Yes, at source level. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` with `{ primary, fallbacks }`, spawn a subagent without an explicit model override, and the normal run path resolves fallbacks from agent/default model policy instead of the selected subagent fallback list.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the already-supported selected subagent fallback list into normal spawned-subagent execution, not to add another config option or broadly change provider-error classification.
