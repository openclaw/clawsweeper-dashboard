## Summary

Keep open: current main has fixed the isolated-cron subagent fallback gap from the related closed report, but normal spawned subagent runs still persist only an auto model override and then resolve fallback candidates from regular agent/default model config, not from the selected `subagents.model.fallbacks`; the current Bedrock provider hook also does not visibly classify the exact inference-profile-required error text.

## What This Changes

The issue requests that configured fallback models be tried when a primary provider fails, including Bedrock provider errors and subagent model fallback chains.

## Best Possible Solution

Preserve the selected `subagents.model.fallbacks` for normal spawned subagent runs and add exact provider-error classification for Bedrock inference-profile-required failures, while leaving cron preflight behavior to its separate canonical issue.

## Reproduction Assessment

Yes, at source level. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` with `{ primary, fallbacks }`, spawn a subagent without an explicit user model override, and current `agent-command` resolves fallback candidates from regular agent/default model config instead of the selected subagent model object.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the already-supported subagent fallback chain into `runWithModelFallback` for auto-selected spawned subagent runs and classify the exact Bedrock provider error, not to add a new config surface or broaden cron preflight behavior here.
