## Summary

Keep open: current main has generic fallback handling for unrecognized provider errors, but normal spawned subagent runs still lose the fallback list attached to the selected `subagents.model` config before `runWithModelFallback` builds candidates.

## What This Changes

The issue requests that configured model fallbacks be attempted when a provider-level failure occurs, especially for subagent model configs using Bedrock or other providers.

## Best Possible Solution

Carry the selected `subagents.model.fallbacks` into the normal spawned-subagent `runWithModelFallback` path, preserving explicit user model overrides and the existing cron fix semantics.

## Reproduction Assessment

Yes, at source level. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` with `{ primary, fallbacks }`, spawn a subagent without an explicit model override, and current `agent-command` resolves fallback candidates from regular agent/default model config instead of the selected subagent model object.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the already-supported subagent fallback chain into `runWithModelFallback` for normal spawned subagent runs, not to add a new config surface or rewrite provider failover policy.
