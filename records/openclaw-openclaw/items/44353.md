## Summary

Keep this open: current main still does not classify the reported Bedrock inference-profile-required error as failover-worthy, despite the related discussion saying the broader Bedrock inference-profile work shipped in v2026.4.5.

## What This Changes

The issue asks for configured fallback models to be tried when an AWS Bedrock primary model fails with the inference-profile provider error.

## Best Possible Solution

Add a narrow Bedrock-owned failover classifier for the inference-profile-required rejection, prove it advances to the configured fallback, and leave broader timeout/model-stack behavior to separate tracking.

## Reproduction Assessment

Yes, from source inspection rather than a live Bedrock call: the reported error string is not matched by the current Bedrock or generic classifiers, and the runner only advances to fallback on classified failover errors. A focused unit test using the exact message should reproduce the missing fallback path.

## Solution Assessment

Yes. The narrowest maintainable fix is the same direction as the closed proposed PR: map only the Bedrock on-demand-throughput / inference-profile-required shape to a failover reason such as model_not_found, with regression coverage, instead of broadening all provider “not supported” errors.

Fix https://github.com/openclaw/openclaw/issues/44353. Bedrock raw model IDs can fail with “Invocation of model ID ... with on-demand throughput is not supported. Retry your request with the ID or ARN of an inference profile...”; when fallbacks are configured, that exact provider error should enter model fallback instead of surfacing as a terminal session/subagent failure. Add a narrow Bedrock-owned classification in `extensions/amazon-bedrock/register.sync.runtime.ts` or the existing provider failover seam, mapping only the on-demand-throughput/inference-profile-required Bedrock shape to `model_not_found` or the existing failover reason best matching unavailable model routing. Add regression coverage proving the classifier recognizes the exact message and `runWithModelFallback` tries the configured next model. Do not broadly classify every “not supported” error, do not change Bedrock discovery/config migration, and do not attempt to solve the separate Kimi timeout/model_stack issue unless the same testable root cause is proven.
