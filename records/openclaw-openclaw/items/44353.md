## Summary

Keep open: current main can continue the fallback loop after unrecognized provider errors when candidates exist, but subagent-specific fallback arrays still do not reach the candidate resolver, so the reported subagent configuration remains broken.

## What This Changes

The issue asks OpenClaw to try configured fallback models when a primary provider such as AWS Bedrock fails with a provider-level error.

## Best Possible Solution

Preserve subagent-specific fallback arrays through spawn and runtime fallback resolution so automatic subagent sessions can pass the intended candidate list to `runWithModelFallback()` while explicit user model overrides remain strict.

## Reproduction Assessment

Yes, at source level. A config using `agents.defaults.subagents.model: { primary, fallbacks }` or per-agent `subagents.model.fallbacks` is reduced to the primary model, stored as an auto session override, and then resolved against only agent/default model fallbacks instead of the subagent fallback array.

## Solution Assessment

Yes. A narrow propagation fix is the best path because generic provider-error fallback already continues when candidates exist; broadening Bedrock-specific classification would not fix the missing subagent candidate list.
