## Summary

Keep open: current main has the generic fallback runner behavior needed to continue when fallback candidates exist, and the isolated cron subagent path was fixed separately, but normal spawned subagents still do not pass the selected `subagents.model.fallbacks` chain into the runner. The remaining bug is narrow fallback-policy routing for auto-selected spawned-subagent models.

## What This Changes

The issue asks OpenClaw to use configured model fallback chains when a provider-level model call fails, especially for Bedrock and spawned subagent model policies.

## Best Possible Solution

Thread the selected `subagents.model.fallbacks` policy into normal spawned-subagent execution so `runWithModelFallback` receives the same fallback chain that selected the subagent primary.

## Reproduction Assessment

Yes, at source level. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` with `{ primary, fallbacks }`, spawn a subagent without an explicit model override, and current normal run code resolves fallbacks from agent/default model policy instead of the selected subagent fallback list.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the already-supported selected subagent fallback list into `runWithModelFallback`, not to add a new config option or broadly change user-selected model strictness.
