## Summary

Keep open: current main has fixed related isolated-cron subagent fallback routing, but the normal spawned-subagent path still stores only the selected model and resolves fallbacks from the parent/default agent model policy instead of the selected `subagents.model.fallbacks` chain.

## What This Changes

The issue requests that provider-level model failures advance through the configured fallback chain, especially for Bedrock and spawned subagent model configs.

## Best Possible Solution

Carry the selected subagent model fallback policy into normal spawned-subagent runs, and add regression coverage that proves `subagents.model.fallbacks` reaches `runWithModelFallback` without changing explicit user override semantics.

## Reproduction Assessment

Yes, at source level. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` with `{ primary, fallbacks }`, spawn a subagent without an explicit model override, and current `agent-command` resolves fallbacks from regular agent/default model config before calling `runWithModelFallback`.

## Solution Assessment

Yes. The narrow maintainable fix is to thread the already-supported selected subagent fallback list through the normal spawned-subagent run path, not to add another config surface or broaden all provider errors blindly.
