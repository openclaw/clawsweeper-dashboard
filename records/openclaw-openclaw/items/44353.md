## Summary

Keep open: current main has the generic fallback runner behavior needed to continue when fallback candidates exist, but normal spawned subagents still do not pass the selected `subagents.model.fallbacks` chain into that runner. The remaining failure is now a narrow fallback-policy routing bug, not a request for a new config surface.

## What This Changes

The issue reports that provider-level failures, including Bedrock model ID errors, can terminate subagent runs instead of trying the fallback models configured for `subagents.model`.

## Best Possible Solution

Thread the selected subagent model fallback policy into normal spawned-subagent execution so `runWithModelFallback` receives the same fallback chain that selected the subagent primary, while preserving explicit user override strictness.

## Reproduction Assessment

Yes, at source level. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` with `{ primary, fallbacks }`, spawn a subagent without an explicit model override, and the normal run path resolves fallbacks from agent/default model policy instead of the selected subagent fallback list.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the already-supported selected subagent fallback list into `runWithModelFallback`, not to add another config option or broadly reclassify unrelated provider errors.
