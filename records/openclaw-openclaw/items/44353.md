## Summary

Keep this issue open. Current main has improved the generic provider-error fallback loop and Bedrock model discovery, but the reported subagent config path still drops `agents.defaults.subagents.model.fallbacks` and `agents.list[].subagents.model.fallbacks` before the runtime fallback runner receives candidates.

## What This Changes

The issue requests that configured fallback models be attempted when a primary provider fails with provider-level errors, especially for Bedrock inference-profile/model-ID failures in subagent runs.

## Best Possible Solution

Preserve subagent-specific fallback arrays through native automatic subagent spawn and runtime execution, while keeping explicit user model overrides strict. The fix should make `agents.defaults.subagents.model.fallbacks` and per-agent `subagents.model.fallbacks` available to `runWithModelFallback()` for child sessions, with focused regression coverage for provider errors and timeouts.

## Reproduction Assessment

Yes. The issue is reproducible by source-level tracing: an `{primary,fallbacks}` subagent model config is normalized to only its primary, the child session patch stores only that primary, and the runtime fallback resolver never reads subagent fallback config.

## Solution Assessment

Yes, a narrow subagent fallback propagation fix is the best path. Provider-error classification and Bedrock discovery are already partially addressed, so broadening Bedrock-specific classification would not fix the remaining missing-candidate path.
