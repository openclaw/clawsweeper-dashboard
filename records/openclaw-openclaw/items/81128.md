## Summary

Keep open: current main still has a source-backed bug where the ACP `sessions_spawn` contract accepts and forwards `streamTo: "parent"`, but transcript repair redacts that field to `__OPENCLAW_REDACTED__` and can make signed-thinking replay drop the assistant turn or replay invalid tool-call plumbing.

## What This Changes

The issue requests a fix for ACP `sessions_spawn` calls where `streamTo: "parent"` is rewritten by transcript redaction and parent-stream logging is lost.

## Best Possible Solution

Keep the ACP parent-stream contract and fix transcript repair so safe already-redacted `sessions_spawn` ACP fields remain replay-safe while `resumeSessionId` and attachment contents stay redacted.

## Reproduction Assessment

Yes at source level: current main deterministically redacts `sessions_spawn.streamTo` and uses object identity to reject mutated sibling tool calls during provider-owned thinking replay. I did not run the full Telegram/ACP live path, so the observed intermittency is supported by source inspection plus reporter runtime data rather than an end-to-end local repro.

## Solution Assessment

Yes, the best fix is narrow: adjust `session-transcript-repair` so already-redacted ACP routing fields do not force a new object, or otherwise keep safe enum routing out of schema-visible redaction, while preserving sensitive redaction. Disabling transcript redaction broadly would be the wrong solution.
