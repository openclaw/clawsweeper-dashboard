## Summary

Keep open: current main still has a source-backed mismatch where `sessions_spawn` accepts and forwards `streamTo: "parent"`, but the transcript/tool-call sanitizer rewrites that same field to `__OPENCLAW_REDACTED__`. The reporter also added fresh 5/5 affected live observations, and I found no merged fix or canonical duplicate that owns this exact redaction bug.

## What This Changes

The issue requests fixing ACP `sessions_spawn` argument redaction so `streamTo: "parent"` reaches validation and execution unchanged while sensitive fields remain redacted.

## Best Possible Solution

Keep the existing ACP streaming contract and make the `sessions_spawn` sanitizer schema-aware so safe `streamTo: "parent"` survives tool-call plumbing while `resumeSessionId` and attachment contents remain redacted.

## Reproduction Assessment

Yes, source-level: passing a `sessions_spawn` assistant tool-call block with `streamTo: "parent"` through `sanitizeToolCallInputs` deterministically rewrites it to the redaction sentinel, and stored/replay paths call that helper. I did not establish a live Telegram/ACP reproduction in this review.

## Solution Assessment

Yes: the narrow maintainable fix is to preserve the schema-safe `streamTo: "parent"` enum in tool-call arguments while continuing to redact actual sensitive fields. Broadly disabling transcript redaction would be too risky.
