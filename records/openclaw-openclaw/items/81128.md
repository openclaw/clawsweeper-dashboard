## Summary

Keep open: current main and the latest shipped release still redact `sessions_spawn.streamTo` to `__OPENCLAW_REDACTED__` in the transcript/tool-call sanitizer, while the tool schema and ACP docs define `streamTo: "parent"` as a legal value. The reporter's fresh 5/5 ACP dispatch data supports the same live-observability failure path, and no narrower merged or open PR appears to supersede this issue.

## What This Changes

The issue reports that legal `sessions_spawn({ runtime: "acp", streamTo: "parent" })` calls can be rewritten by redaction before validation, dropping ACP parent streaming and `streamLogPath` creation.

## Best Possible Solution

Keep the ACP parent-streaming contract and update `sessions_spawn` transcript sanitization so safe enum routing fields do not become schema-invalid redaction sentinels, with focused regression coverage for sanitizer, spawn-tool, and ACP stream paths.

## Reproduction Assessment

Yes for the core failure path: constructing a `sessions_spawn` tool-call block with `streamTo: "parent"` and passing it through `sanitizeToolCallInputs` is enough to show current main rewrites it to the redaction sentinel. I did not run the full Telegram/ACP live flow, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes, the narrow maintainable fix is to stop redacting the safe `streamTo: "parent"` enum in schema-visible tool-call plumbing while still redacting `resumeSessionId` and attachment content. Broadly disabling transcript redaction would be unsafe.
