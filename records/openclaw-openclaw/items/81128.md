## Summary

Keep open: current main still has a source-backed mismatch where `sessions_spawn` accepts and forwards `streamTo: "parent"`, but the transcript/tool-call sanitizer rewrites that same safe enum field to `__OPENCLAW_REDACTED__`; no linked or discovered PR currently closes this issue.

## What This Changes

The issue asks OpenClaw to stop an internal redaction layer from rewriting legal `sessions_spawn({ runtime: "acp", streamTo: "parent" })` arguments before validation.

## Best Possible Solution

Make the `sessions_spawn` sanitizer schema-aware so safe `streamTo: "parent"` remains valid in tool-call plumbing while sensitive resume IDs and attachment contents stay redacted.

## Reproduction Assessment

Yes at source level: `sanitizeToolCallInputs` on a `sessions_spawn` block containing `streamTo: "parent"` deterministically rewrites that field to the redaction sentinel under current main. I did not establish a live Telegram/ACP reproduction of the intermittent user flow.

## Solution Assessment

Yes: the narrow maintainable fix is to stop redacting the non-secret enum field into a schema-invalid sentinel while keeping `resumeSessionId` and attachment content redaction. Disabling the redaction layer globally would be too broad.
