## Summary

Keep open: current main still has a source-backed mismatch where `sessions_spawn` advertises and forwards `streamTo: "parent"`, but the transcript/tool-call sanitizer rewrites that same field to `__OPENCLAW_REDACTED__` for `sessions_spawn` blocks. That matches the reported failure mode closely enough to treat this as a real, narrow bug, even though I did not live-reproduce the intermittent Telegram/ACP run.

## What This Changes

A user reports that ACP `sessions_spawn` calls with `streamTo: "parent"` can be rewritten to the internal redaction sentinel, causing enum validation to reject an otherwise valid spawn and losing `streamLogPath` when the field is omitted.

## Best Possible Solution

Keep the shipped ACP streaming contract, but make the `sessions_spawn` redaction path schema-aware so safe enum routing fields do not become invalid sentinel values while genuinely sensitive fields remain redacted.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection shows the current sanitizer deterministically rewrites `sessions_spawn.streamTo` to the sentinel in persisted/replayed tool-call blocks, which is a strong root-cause path for the reported validation error.

## Solution Assessment

Yes, with a narrow boundary: preserve or safely omit `streamTo: "parent"` in schema-visible tool-call plumbing while continuing to redact `resumeSessionId` and attachment contents. Broadly disabling redaction would be the wrong fix.
