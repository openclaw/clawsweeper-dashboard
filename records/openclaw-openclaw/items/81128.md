## Summary

Keep open: current main still has a source-backed contract mismatch where `sessions_spawn` accepts and forwards `streamTo: "parent"`, but the transcript sanitizer rewrites that same field to the redaction sentinel in tool-call blocks. This is a narrow bug with a clear repair path, not a cleanup-close candidate.

## What This Changes

The issue reports that legal ACP `sessions_spawn` calls with `streamTo: "parent"` are intermittently rewritten to `__OPENCLAW_REDACTED__` before validation, losing parent streaming and `streamLogPath`.

## Best Possible Solution

Make the `sessions_spawn` sanitizer schema-aware so `streamTo: "parent"` remains valid in replay/tool plumbing while sensitive ACP resume IDs and attachment contents stay redacted, with focused regression coverage.

## Reproduction Assessment

Yes at source level: current `sanitizeToolCallInputs` deterministically rewrites `sessions_spawn.streamTo` to the sentinel, and existing tests assert that rewrite. I did not establish a live Telegram/ACP reproduction of the intermittent trigger.

## Solution Assessment

Yes: the narrow maintainable fix is to preserve or safely omit the non-secret `streamTo` enum in schema-visible tool-call plumbing while keeping redaction for genuinely sensitive fields. Broadly disabling transcript redaction would be too risky.
