## Summary

Keep open: current main still has a source-backed bug where ACP `sessions_spawn.streamTo: "parent"` is a documented legal value, but transcript repair still substitutes the redaction sentinel into that field and the signed-thinking replay path treats any re-sanitized block as unsafe by reference equality. The latest comments add useful live repro data and a plausible root-cause trace, and there is no merged PR or duplicate canonical item that resolves this exact ACP parent-stream replay/redaction failure.

## What This Changes

Fix ACP `sessions_spawn` replay/redaction so `streamTo: "parent"` continues to support parent streaming instead of becoming an invalid redaction sentinel or causing replay drops.

## Best Possible Solution

Keep the ACP parent-stream contract and make `sessions_spawn` transcript repair schema-aware/idempotent so safe `streamTo` handling cannot produce invalid replay/tool-call arguments while sensitive fields remain redacted.

## Reproduction Assessment

Yes at source level: current main still redacts `sessions_spawn.streamTo` to the sentinel and the signed-thinking replay gate rejects blocks whose sanitizer output is not reference-identical. I did not establish a live Telegram/ACP reproduction in this read-only pass.

## Solution Assessment

Yes in direction, but the safest fix is broader than blindly preserving all arguments: handle the non-secret `streamTo` enum without placing an invalid sentinel in schema-visible replay paths, and make already-redacted blocks idempotent while keeping `resumeSessionId` and attachments protected.
