## Summary

Keep open: current main still has a source-backed mismatch where `sessions_spawn` accepts and forwards `streamTo: "parent"`, but the transcript sanitizer rewrites that same field to `__OPENCLAW_REDACTED__` in persisted/replayed tool-call blocks.

## What This Changes

The issue reports intermittent ACP `sessions_spawn` failures when `streamTo: "parent"` is redacted before schema validation, which removes parent streaming and `streamLogPath` behavior.

## Best Possible Solution

Keep the ACP parent-stream contract, but make `sessions_spawn` transcript redaction schema-aware so `streamTo: "parent"` remains valid while `resumeSessionId` and attachment content stay redacted.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a high-confidence reproduction path: pass a `sessions_spawn` tool-call block with `streamTo: "parent"` through `sanitizeToolCallInputs`, and current main rewrites it to `__OPENCLAW_REDACTED__`.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve or safely omit the schema-valid non-secret `streamTo` field in tool-call plumbing while retaining redaction for actual sensitive fields, rather than disabling redaction broadly.
