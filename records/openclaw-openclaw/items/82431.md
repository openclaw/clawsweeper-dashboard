## Summary

Keep this PR open, but it is not merge-ready: the generic approval-action seam is useful, yet decision actions currently let plugin-supplied labels/styles drive approval buttons while the callback still resolves the canonical allow/deny decision. That makes protected-tool approval controls spoofable and needs a trust-boundary fix before maintainer API approval.

## What This Changes

The branch adds plugin approval action templates and `keepPendingWithoutRoute` across plugin hooks, Gateway protocol/schema, approval view/reply models, native channel adapters, Control UI, docs, and generated API/protocol baselines.

## Best Possible Solution

Canonicalize or validate decision-action display semantics before merge, then land the additive approval metadata seam only after maintainers accept the public plugin API and companion approval-resolution flow.

## Reproduction Assessment

Yes, source-reproducible for the PR finding: a plugin decision action can carry arbitrary `label`/`style`, and the pending view plus native adapters render those values while resolving the canonical decision callback. This is not a current-main bug report; it is a defect introduced by the proposed API surface.

## Solution Assessment

No: the proposed seam is useful, but decision actions should not let plugin-provided display text/style change the meaning of protected approval controls. The safer path is to derive decision labels/styles from the decision and keep custom plugin text on command actions or descriptive metadata.
