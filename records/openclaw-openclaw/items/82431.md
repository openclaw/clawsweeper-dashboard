## Summary

Keep open. The PR is a plausible additive subset of the related approval API request, but the patch is not merge-ready because command-only approval actions are fed into native approval adapters that still require decision actions, and the external real-behavior proof is test-only.

## What This Changes

The PR adds plugin approval action templates and keepPendingWithoutRoute through the hook contract, gateway schema and handler, reply payload metadata, approval view models, tests, and the SDK API baseline.

## Best Possible Solution

Land a narrow additive approval-action API after native approval clients handle command and decision actions explicitly, with real external-plugin approval proof; keep the broader remaining API work tracked by the linked issue.

## Reproduction Assessment

Yes. Source inspection shows a command-only action can enter PendingApprovalView.actions, while existing native adapters still map every action to a decision or decision-derived custom ID.

## Solution Assessment

No, not as written. The direction is maintainable, but the action model needs a discriminated contract and adapter updates before it is safe to expose across approval clients.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Update native approval adapters for command actions:** `src/infra/approval-view-model.types.ts:14-15`
  - body: Making `decision` optional allows `PendingApprovalView.actions` to contain command-only entries, but existing native approval runtimes still treat every action as a decision: Telegram/Matrix/QQ map `view.actions.map(action => action.decision)` into allowed decisions/keyboards and Discord builds custom IDs from `descriptor.decision`. A plugin command action like the new fixture will therefore produce `undefined` decisions or invalid buttons in those clients. Keep this view contract decision-only or update the native adapters to handle `kind: "command"` separately.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff changes approval UI/action semantics and needs tightening before those actions are trusted by native approval clients.

Concerns:

- **[medium] Discriminate approval action decisions from commands:** `src/gateway/protocol/schema/plugin-approvals.ts:13`
  - body: The new action contract carries semantic decisions and executable commands but does not enforce a decision for decision actions or keep command-only actions out of decision-only clients. Because these buttons resolve protected approval flows, the schema/view contract should make the two cases explicit before release.
  - confidence: 0.78
