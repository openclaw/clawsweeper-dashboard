## Summary

Keep open: this is a plausible generic plugin approval API slice, but the current PR head still turns Telegram command-only approval actions into inline callback buttons that are not routed as slash commands, and the supplied real-behavior proof is stale relative to the latest head.

## What This Changes

The PR adds plugin approval action templates and keepPendingWithoutRoute across before_tool_call, Gateway protocol/server handling, approval view/reply metadata, native channel adapters, generated Swift models, SDK API baseline, and focused tests.

## Best Possible Solution

Land the bounded approval action and no-route pending API after Telegram only renders runnable native buttons or implements a real routed callback for command actions, with refreshed current-head and Telegram proof; keep verified resolution reviewed separately at https://github.com/openclaw/openclaw/pull/82434.

## Reproduction Assessment

Yes for the PR review finding by source inspection: command-only plugin actions flow into Telegram inline callback buttons, while current Telegram callback handling does not replay arbitrary slash commands. The main PR request is an additive API feature, so current-main bug reproduction is not applicable.

## Solution Assessment

Not yet; the generic core/SDK direction is reasonable, but Telegram needs to filter command-only actions out of native callback buttons or add a routed callback path. The safer narrow fix is to keep command actions visible in text while only decision actions become Telegram approval buttons.
