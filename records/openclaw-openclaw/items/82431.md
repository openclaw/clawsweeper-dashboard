## Summary

Keep open: the PR is active and relevant, but current head still turns command-only plugin approval actions into Telegram callback buttons that cannot execute those plugin commands, and the supplied proof does not cover the current head or Telegram-visible path.

## What This Changes

The PR adds plugin approval action templates and no-route pending behavior, then carries the resulting command/decision metadata through Gateway, SDK reply payloads, approval view models, channel adapters, generated Swift protocol types, and focused tests.

## Best Possible Solution

Filter Telegram native buttons to decision-backed approval actions or add a real routed callback path for plugin commands, then land the bounded API after maintainer review and current-head real behavior proof while keeping verified resolution reviewed separately in https://github.com/openclaw/openclaw/pull/82434.

## Reproduction Assessment

Yes for the PR review finding by source inspection: PR head passes every action to Telegram inline buttons, while current-main Telegram callback handling only parses approval callbacks and structured native/plugin callbacks. Current-main bug reproduction is not applicable because this is an additive feature PR.

## Solution Assessment

No, not yet; the generic Gateway and SDK direction is plausible, but Telegram should not render command-only plugin actions as tappable callback buttons unless those callbacks are actually routed. The safer fix is to leave command actions in text and render only decision-backed approval buttons.
