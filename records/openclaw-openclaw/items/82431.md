## Summary

Keep open; the API direction is plausible, but the current branch still lets plugin-supplied decision command strings become Telegram and Slack native button callback payloads, so documented decision buttons can fail to resolve the pending approval.

## What This Changes

The PR adds plugin approval action templates and keepPendingWithoutRoute across the plugin hook, Gateway request schema, SDK/API surfaces, approval view models, Web UI, channel adapters, docs, generated Swift, and focused tests.

## Best Possible Solution

Land the host API only after native decision buttons are bound to canonical approval id/decision semantics and maintainers explicitly accept the approval API and no-route pending lifecycle.

## Reproduction Assessment

Yes for source-level reproduction: a decision action with a custom commandTemplate is rendered into Telegram and Slack native button callback data, while both callback handlers only direct-resolve payloads parsed as OpenClaw /approve commands.

## Solution Assessment

No, not yet; splitting command actions into text and decision actions into buttons is the right shape, but native decision buttons must not depend on arbitrary plugin command text for the built-in resolver.
