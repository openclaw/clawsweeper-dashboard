## Summary

Keep open: this PR addresses a real generic plugin approval API gap, but the latest head still allows inconsistent approval action metadata that can turn plugin-owned command actions into native approval buttons.

## What This Changes

The branch adds plugin approval action templates, keepPendingWithoutRoute, expanded approval commands, channel adapter rendering updates, docs, generated Swift protocol changes, SDK API baseline updates, and focused tests.

## Best Possible Solution

Make approval actions discriminated, render native controls only for decision actions, and then land the generic host API set with the companion approval-resolution and chat-metadata PRs if maintainers accept the public API.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the schema and expander permit kind: "command" with decision, while native adapters select approval buttons by decision presence. This is not a current-main bug reproduction; it is a source-reproducible patch contract issue.

## Solution Assessment

No, not yet: the generic host seam is a reasonable direction, but the public action metadata contract needs to reject or normalize inconsistent command/decision pairs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject command actions that carry decisions:** `src/gateway/protocol/schema/plugin-approvals.ts:17`
  - body: The schema accepts `decision` on every action, including `kind: "command"`, and the expander preserves it. Since native adapters choose approval buttons from `action.decision`, a plugin-owned verification command can become a built-in approval button instead of visible command text, bypassing the plugin command flow described in the docs. Please make the action schema/normalization discriminated, or require `kind === "decision" && decision` in native button filters.
  - confidence: 0.86

## Security Review
