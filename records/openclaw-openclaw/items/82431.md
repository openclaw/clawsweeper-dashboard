## Summary

Keep open. This is an active additive host/plugin API PR with supplied proof and companion work, but the diff still has a blocking Telegram approval-button issue: native decision buttons can carry plugin command callback data instead of staying on the approval resolver path.

## What This Changes

The PR adds plugin approval action templates and no-route pending behavior across before_tool_call, Gateway/protocol schemas, SDK rendering, native channel adapters, Web UI, docs, generated baselines, and focused tests.

## Best Possible Solution

Keep the additive generic host API approach, but fix Telegram decision button callback data to stay on the approval resolver path before normal maintainer API review and companion PR alignment.

## Reproduction Assessment

Yes for the review finding: source inspection shows Telegram decision buttons can be built from plugin commandTemplate values, while the Telegram callback resolver only treats /approve-shaped callback data as approval callbacks. I did not run a live Telegram click path in this read-only review.

## Solution Assessment

No, not quite: the generic host/plugin approval seam is the right direction, but Telegram native decision buttons need canonical approval callback data or validation that decision templates are /approve commands. Plugin-owned commands should remain visible text via kind: command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Encode Telegram decision callbacks as approvals:** `extensions/telegram/src/approval-handler.runtime.ts:106-108`
  - body: Decision actions are filtered before rendering, but the generated Telegram button still uses the plugin-provided `command` as callback data. The new test shows a decision action can emit `/agentkit deny ...`; Telegram only resolves approval callbacks that parse as `/approve <id> <decision>`, so that button falls through the plugin callback path instead of the approval resolver. Encode native decision buttons as `/approve ${approvalId} ${decision}` or reject non-approval command templates for `kind: "decision"`.
  - confidence: 0.91
