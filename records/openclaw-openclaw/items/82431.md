## Summary

Keep open: the additive plugin approval API is useful, but the latest head still renders command-only plugin approval actions as Telegram callback buttons, and the supplied proof is stale for the current head and does not cover the Telegram-visible path.

## What This Changes

The PR adds plugin-provided approval action templates and keepPendingWithoutRoute across before_tool_call, Gateway approval protocol/server handling, SDK reply metadata, channel adapters, generated Swift models, API baselines, and focused tests.

## Best Possible Solution

Land the bounded approval action and no-route pending API after Telegram renders only runnable native decision buttons or gains a real routed callback path for command actions, with refreshed current-head and Telegram proof; keep verified resolution reviewed separately at https://github.com/openclaw/openclaw/pull/82434.

## Reproduction Assessment

Yes for the PR review finding by source inspection: command-only plugin actions flow into Telegram inline callback buttons, while Telegram callback handling does not replay arbitrary slash commands. Current-main bug reproduction is not applicable because the PR is adding an API feature.

## Solution Assessment

Not yet; the generic core/SDK direction is plausible, but Telegram needs to filter command-only actions out of native callback buttons or implement a real callback route. The narrower safer fix is to keep command actions visible in text while only decision actions become Telegram approval buttons.
