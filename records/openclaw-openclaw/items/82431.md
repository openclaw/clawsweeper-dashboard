## What This Changes

The PR adds plugin approval action metadata and keepPendingWithoutRoute across before_tool_call hooks, Gateway/protocol schemas, SDK reply rendering, Web UI models, native channel adapters, docs, generated Swift/API baselines, and focused tests.

## Best Possible Solution

Keep the generic host/plugin approval metadata seam, fix Telegram decision callbacks to stay on the OpenClaw approval resolver, then land only after maintainer API acceptance and companion PR alignment.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the PR maps Telegram decision button callback data from action.command while current Telegram approval handling resolves only parsed /approve callbacks. The overall PR is an additive API feature rather than a bug reproduction.

## Solution Assessment

No, not quite. The generic plugin approval seam is the right direction, but Telegram decision buttons should use OpenClaw approval callbacks before merge and the public API shape still needs maintainer acceptance with the companion PRs.
