## Summary

Keep open: the latest head fixes the earlier Matrix fallback-text defect, but command-only plugin approval actions are still dropped from Discord and QQBot native approval delivery, so the PR needs focused adapter fixes before merge.

## What This Changes

The PR adds plugin approval action metadata and no-route pending behavior across Gateway, approval view/reply models, SDK API surface, generated Swift protocol, channel adapters, and focused tests.

## Best Possible Solution

Land the additive approval API after Discord and QQBot preserve command-only plugin approval actions in visible delivery payloads and focused adapter/API validation is green.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows Discord filters command-only actions out of the only action row, and QQBot renders plugin text from a helper that ignores actions while building the keyboard only from decision actions. Focused adapter tests with a command-only plugin approval view should reproduce the missing visible action.

## Solution Assessment

No as written. The generic API direction and Matrix follow-up are reasonable, but the native Discord and QQBot delivery paths need to surface command-only actions or provide an explicit supported fallback before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve Discord command-only approval actions:** `extensions/discord/src/approval-handler.runtime.ts:156-164`
  - body: For plugin approvals whose actions are command-only, this filter drops every action and returns no action row. The plugin approval container does not render `view.actions` anywhere else, so a Discord user never sees the plugin-owned command they need to run. Include those command actions in the visible message text or add a safe Discord representation instead of silently omitting them.
  - confidence: 0.8
- **[P2] Preserve QQBot command-only approval actions:** `extensions/qqbot/src/bridge/approval/handler-runtime.ts:141-142`
  - body: When a plugin approval supplies only command actions, `listDecisionActions(view.actions)` returns an empty array, while `buildPluginApprovalText(req)` never includes the custom commands. That leaves QQBot users with no visible action to take and sends an empty approval keyboard; thread command actions into the text and omit the keyboard when no decision actions exist.
  - confidence: 0.84
