## Summary

Keep open: the latest head still has a Telegram callback correctness issue, and the supplied real-behavior proof is stale relative to the current head and does not cover the Telegram-visible path.

## What This Changes

The PR adds plugin-provided approval action templates and no-route pending behavior across before_tool_call, Gateway plugin approval schema/server handling, SDK reply metadata, channel adapters, generated Swift models, API baselines, and focused tests.

## Best Possible Solution

Land the bounded approval action and no-route pending API after Telegram renders only runnable native decision buttons or gains a real routed callback path for command actions; keep verified resolution reviewed separately at https://github.com/openclaw/openclaw/pull/82434.

## Reproduction Assessment

Yes for the PR review finding by source inspection: command-only plugin actions flow into Telegram inline callback buttons, while current Telegram callback handling does not replay arbitrary slash commands. Current-main bug reproduction is not applicable because this PR adds a new API feature.

## Solution Assessment

Not yet; the generic core and SDK direction is plausible, but Telegram should not render command-only actions as native callback buttons unless they have a real callback route. The narrower safer fix is to keep command actions visible in text and only render decision actions as Telegram buttons.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Filter command-only Telegram approval actions from buttons:** `extensions/telegram/src/approval-handler.runtime.ts:102`
