## Summary

Keep open: the PR adds a useful public approval API surface, but latest head still has a Telegram correctness issue and the supplied proof is only checks/tests rather than real plugin or Telegram behavior.

## What This Changes

The PR adds plugin-provided approval action templates and keepPendingWithoutRoute across before_tool_call, Gateway plugin approval schema/server handling, SDK reply metadata, channel adapters, generated Swift models, API baselines, and focused tests.

## Best Possible Solution

Land the bounded approval action and no-route pending API after Telegram renders only runnable native decision buttons or gains a real routed callback path for command actions, with verified resolution reviewed separately at https://github.com/openclaw/openclaw/pull/82434.

## Reproduction Assessment

Yes for the PR review finding by source inspection: PR head turns every approval action into Telegram inline buttons, while current main treats button values as structured callback data rather than normal slash-command input. Current-main bug reproduction is not applicable because this PR adds a new API feature.

## Solution Assessment

Not yet; the generic approval action API direction is plausible, but Telegram should not render command-only actions as native callback buttons unless those callbacks are actually routed. The safer narrow fix is to keep command actions visible as text and only render decision-backed actions as Telegram buttons.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Filter command-only Telegram approval actions from buttons:** `extensions/telegram/src/approval-handler.runtime.ts:102`
