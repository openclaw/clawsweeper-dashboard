## Summary

Keep open: the API slice is useful and matches the linked external HITL plugin request, but the current head still has a blocking Telegram callback bug and the supplied real behavior proof is not current-head proof.

## What This Changes

The PR adds plugin approval action templates and keepPendingWithoutRoute across before_tool_call, Gateway protocol/server handling, approval view/reply models, native channel adapters, generated Swift models, SDK API baseline, and focused tests.

## Best Possible Solution

Land the bounded approval action and no-route pending API after Telegram renders command-only actions as text or routes them through a real callback, refreshed current-head proof is supplied, and the companion verified resolver remains reviewed separately at https://github.com/openclaw/openclaw/pull/82434.

## Reproduction Assessment

Yes for the review finding by source inspection: a command-only action flows into a Telegram inline callback button, while callback handling does not replay arbitrary slash-command callback data. The PR's main request is an additive API feature, so current-main bug reproduction is not applicable for that part.

## Solution Assessment

Not yet; the generic core/SDK surface is a reasonable direction, but Telegram should filter command-only actions out of native callback buttons or implement a real routed callback path. The safer narrow fix is to keep command actions visible in text while only decision actions become Telegram approval buttons.
