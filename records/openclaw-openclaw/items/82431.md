## Summary

Keep open: this is a useful focused API expansion for external HITL plugins, but the patch currently has a type/build regression in the Telegram approval path and only provides test/check proof rather than after-fix real behavior proof.

## What This Changes

The PR adds plugin approval action templates and keepPendingWithoutRoute to hook/RPC contracts, expands action commands with server approval IDs, threads action metadata through reply/view payloads, and updates focused tests plus the SDK API baseline hash.

## Best Possible Solution

Land a narrow additive approval API after preserving typed exec-decision actions for existing clients, updating affected approval clients where needed, and adding redacted real behavior proof from an actual plugin approval flow.

## Reproduction Assessment

Yes for the review finding: source inspection shows the optional decision type conflicts with the Telegram approval runtime's concrete allowedDecisions contract. The feature itself still lacks real behavior proof beyond tests and build checks.

## Solution Assessment

No as written. The API direction is sound, but the shared action type needs to avoid breaking existing exec approval consumers, and the PR needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep exec approval decisions concrete:** `src/infra/approval-view-model.types.ts:15`
  - body: Making `ApprovalActionView.decision` optional breaks the Telegram approval runtime: it maps `params.view.actions.map((action) => action.decision)` into `ExecApprovalPendingReplyParams.allowedDecisions`, so after this change that array is typed as possibly containing `undefined` and the extension build/typecheck should fail. Split command actions into a plugin-only type or narrow/filter before passing exec decisions.
