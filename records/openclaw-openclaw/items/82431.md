## Summary

Keep open: current main still lacks the generic plugin approval action and no-route pending API, and this PR addresses the linked external AgentKit need, but it should not merge until decision actions are validated against the stored allowed-decision set and the docs match the discriminated contract.

## What This Changes

The PR adds plugin approval action templates and keepPendingWithoutRoute to before_tool_call and plugin.approval.request, propagates expanded actions through Gateway, SDK, view-model, and native adapter paths, and updates docs/generated protocol/API baselines.

## Best Possible Solution

Land the generic host API set after enforcing the action/decision contract, keeping the companion verified-resolution and chat metadata PRs aligned, and tightening the public docs to match the SDK type.

## Reproduction Assessment

Not applicable as a current-main bug; this PR adds a new public plugin/Gateway API. The review finding is source-reproducible from the PR head because action decisions are stored independently from allowedDecisions while the resolver later rejects unavailable decisions.

## Solution Assessment

No, not quite: the generic API direction fits the linked external plugin need, but the PR should reject or normalize decision actions that are outside the stored allowedDecisions before exposing the contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Reject decision actions outside allowedDecisions:** `src/gateway/server-methods/plugin-approval.ts:119-120`
  - body: `allowedDecisions` is still the source of truth for `plugin.approval.resolve`, but action templates are expanded and stored without checking that a `kind: "decision"` action is allowed. A plugin can send `allowedDecisions: ["deny"]` plus an Allow Once decision action; native clients will render the button, then the resolver rejects the click as unavailable and leaves a dead approval control. Please validate decision actions against the resolved allowed-decision set before registering the request.
  - confidence: 0.86
- **[P3] Document action metadata as a discriminated union:** `docs/plugins/hooks.md:193-199`
  - body: The docs snippet still shows one action object with `kind: "decision" | "command"` and optional `decision`, but the SDK type and server validation require decision actions to include `decision` and command actions to omit it. Plugin authors following this snippet can build payloads that compile conceptually but fail with `INVALID_REQUEST`; please mirror the discriminated union shape from `src/plugins/hook-types.ts`.
  - confidence: 0.84
