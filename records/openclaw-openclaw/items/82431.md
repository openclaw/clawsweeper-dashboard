## Summary

Keep open: the latest head still exposes a new public approval API without matching plugin docs and still lets Telegram convert command-only plugin approval actions into native callback data. The proof is also check/test-only, so the external plugin/channel flow is not yet demonstrated.

## What This Changes

This PR adds plugin approval action templates and keepPendingWithoutRoute across before_tool_call, Gateway protocol/schema, SDK reply/view models, native channel adapters, generated Swift models, and focused tests.

## Best Possible Solution

Land after Telegram native callbacks are decision-only, docs/plugins/hooks.md documents the new approval action contract, and real external-plugin/channel proof demonstrates the flow.

## Reproduction Assessment

Yes for the PR findings by source inspection: the PR head still builds Telegram inline buttons from every action descriptor, and the PR head docs still show the old requireApproval shape. Current-main bug reproduction is not applicable because this PR adds a new API surface.

## Solution Assessment

Not yet; the generic API direction is reasonable, but the patch needs decision-only Telegram native callbacks, public docs for the contract, and real behavior proof before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Filter command actions out of Telegram native buttons:** `extensions/telegram/src/approval-handler.runtime.ts:102`
  - body: `buildPluginApprovalPendingReplyPayload` already puts plugin command actions in the message text, but this still converts every `view.actions` entry into Telegram inline `callback_data`. Command-only values such as `/agentkit approve ...` are not structured approval callbacks, so build native buttons only from decision-backed actions and leave command actions as visible text.
  - confidence: 0.91
- **[P2] Document the new approval action contract:** `src/plugins/hook-types.ts:484-491`
  - body: This adds public `before_tool_call.requireApproval.actions` and `keepPendingWithoutRoute` fields, plus Gateway request fields, while the plugin hook docs still show the old shape. Plugin authors need the docs to define command template expansion, decision-vs-command behavior, `allowedDecisions`, and no-route pending semantics before this API lands.
  - confidence: 0.88
