## Summary

Keep open. The additive approval API is plausible, but current head still renders command-only plugin approval actions as Telegram callback buttons even though Telegram callback handling does not replay those slash commands, and the submitted proof is checks-only rather than a real plugin or Telegram run.

## What This Changes

The PR adds plugin approval action templates and keepPendingWithoutRoute across before_tool_call, Gateway plugin approval schema/server handling, SDK reply metadata, channel adapters, generated Swift models, API baselines, and focused tests.

## Best Possible Solution

Land the bounded approval action and no-route pending API only after Telegram renders command actions through a real routed path or text fallback, the plugin-facing docs describe the new fields, and the paired verified-resolution contract is reviewed together with https://github.com/openclaw/openclaw/pull/82434.

## Reproduction Assessment

Yes for the PR review finding by source inspection: PR head builds Telegram inline buttons from all approval actions, while current main callback handling does not replay command-like callback_data as chat input. Current-main bug reproduction is not applicable because this PR adds a new API feature.

## Solution Assessment

Not yet; the generic approval action API direction is reasonable, but Telegram should not render command-only actions as native callback buttons unless those callbacks are actually routed. The safer narrow fix is to keep command actions visible as text or filter native buttons to decision-backed actions.

Filter command actions out of Telegram buttons or add a real routed callback path before rendering them natively.
