## Summary

Keep open. Current main already has account-aware Feishu runtime health/probe plumbing, but the Feishu setup/status fallback still probes only top-level channel credentials, and open PR #74060 explicitly targets this issue with closing syntax. The issue should stay open until that PR lands or a maintainer closes the paired implementation path.

## What This Changes

The issue requests that Feishu multi-account credentials under `channels.feishu.accounts.*` be recognized by `openclaw status --deep` health/status output instead of reporting Feishu as OFF/not configured.

## Best Possible Solution

Land a narrow Feishu plugin change that makes the setup/status fallback resolve the selected Feishu account before checking and probing credentials, with regression coverage for `defaultAccount` plus `accounts.*.appId/appSecret`; after that, let the linked PR close this issue.

## Reproduction Assessment

Yes for the deterministic source-level mismatch: a config with `channels.feishu.defaultAccount` and credentials only under `channels.feishu.accounts.<id>` makes Feishu configured, while the setup/status fallback still probes top-level credentials. I did not reproduce the exact terminal table with live Feishu credentials.

## Solution Assessment

Yes. Routing Feishu setup/status probing through the existing resolved-account path is the narrow plugin-owned fix and avoids adding a core special case.
