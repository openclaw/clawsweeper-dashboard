## Summary

Keep open. The report is valid enough to preserve because current main still has a Feishu setup/status probe path that reads top-level credentials, and open PR #74060 uses closing syntax for this same Feishu multi-account credential issue.

## What This Changes

The issue asks for Feishu health/status checks to recognize and probe credentials stored under `channels.feishu.accounts` with `defaultAccount`, instead of requiring top-level `channels.feishu.appId/appSecret`.

## Best Possible Solution

Review and either land PR #74060 or close it if a maintainer decides the current gateway health path is already sufficient. The desired end state is for Feishu setup/status and health probes to consistently use the selected resolved account, with regression coverage for account-scoped credentials.

## Reproduction Assessment

Unclear for the exact live `openclaw status --deep` OFF output on current main without real Feishu credentials. The source-level mismatch is deterministic: account-scoped credentials can make Feishu configured, while the setup/status probe still reads top-level credentials before verifying.

## Solution Assessment

Yes. Routing Feishu setup/status probing through the existing resolved-account path is the narrow plugin-owned fix and avoids adding a core special case.
