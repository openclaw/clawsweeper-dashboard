## Summary

Keep open. Current main has account-aware Feishu health/runtime plumbing, but the setup/status fallback still probes only top-level Feishu credentials, and open PR #74060 already targets this issue with closing syntax.

## What This Changes

The issue asks Feishu health/status to treat `channels.feishu.defaultAccount` plus `channels.feishu.accounts.*.appId/appSecret` as configured and probeable.

## Best Possible Solution

Land PR #74060 or an equivalent narrow Feishu plugin change that resolves the selected Feishu account before setup/status probing, with regression coverage for account-scoped credentials.

## Reproduction Assessment

Yes. A config with `channels.feishu.defaultAccount` and credentials only under `channels.feishu.accounts.<id>` exercises the remaining source-level mismatch: setup/status configured detection can see the account, while the fallback probe still starts from top-level credentials.

## Solution Assessment

Yes. Routing the Feishu setup/status probe through the existing resolved-account path is the narrow plugin-owned fix and avoids adding a core health special case.
