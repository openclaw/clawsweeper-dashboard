## Summary

Keep open. Current main has account-aware Feishu runtime health/probe plumbing, but the setup/status fallback still checks only top-level Feishu credentials before probing, and open PR #74060 explicitly targets this issue with closing syntax. Per cleanup policy, this issue should remain open until that implementation path lands or maintainers close it.

## What This Changes

The issue asks for Feishu health/status checks to treat `channels.feishu.defaultAccount` plus `channels.feishu.accounts.*.appId/appSecret` as configured instead of requiring top-level Feishu credentials.

## Best Possible Solution

Feishu's plugin-owned setup/status fallback should resolve the selected Feishu account before checking and probing credentials, with regression coverage for `defaultAccount` plus `accounts.*.appId/appSecret`; then the linked implementation PR can close this issue when merged.

## Reproduction Assessment

Yes for the deterministic source-level mismatch: account-scoped Feishu credentials make the resolved account configured, while `resolveStatusLines` still probes only top-level credentials. The exact live terminal Health table was not rerun with real Feishu credentials.

## Solution Assessment

Yes. Routing setup/status probing through the existing resolved-account path is the narrowest maintainable fix and keeps Feishu-specific behavior inside the Feishu plugin instead of adding a core special case.
