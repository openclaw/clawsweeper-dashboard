## Summary

Keep open. Current main still has a Feishu setup/status probe path that checks top-level `channels.feishu` credentials before probing, while the account resolver and gateway health path are account-aware; open PR #74060 already targets this issue with closing syntax, so the issue should stay open until that PR is reviewed and either landed or closed.

## What This Changes

The issue reports that `openclaw status --deep` can show Feishu as OFF/not configured when Feishu credentials are stored under `channels.feishu.accounts.*` with `defaultAccount` instead of top-level `channels.feishu.appId/appSecret`.

## Best Possible Solution

Review and either land PR #74060 or close it with maintainer rationale. The desired end state is for Feishu setup/status and health-style probes to consistently use the selected resolved account, with regression coverage for account-scoped credentials.

## Reproduction Assessment

Unclear for the exact live terminal OFF output without real Feishu credentials. The source-level mismatch is reproducible by inspection: account-scoped credentials can mark Feishu configured while the setup/status probe still reads only top-level credentials before probing.

## Solution Assessment

Yes. Routing the Feishu setup/status probe through `resolveFeishuAccount` is the narrow plugin-owned fix and matches the existing gateway health/runtime status account-resolution pattern.
