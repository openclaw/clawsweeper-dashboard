## Summary

Keep open. The issue is paired with open, unmerged PR #74060 using closing syntax, and current main still has a Feishu setup/status probe path that starts from top-level credentials even though account-aware Feishu health/probe plumbing exists elsewhere.

## What This Changes

Health/status should report Feishu as configured when credentials are stored under `channels.feishu.accounts.<id>` with `defaultAccount`, not only when `channels.feishu.appId/appSecret` are set at the top level.

## Best Possible Solution

Land PR #74060 or an equivalent narrow Feishu plugin change that resolves the selected Feishu account before setup/status probing, with regression coverage for `defaultAccount` plus `accounts.*.appId/appSecret`.

## Reproduction Assessment

Yes. A config with `channels.feishu.defaultAccount` and credentials only under `channels.feishu.accounts.<id>` exercises the source-level mismatch: setup/status configured detection can succeed, while the probe still starts from top-level `channels.feishu` credentials.

## Solution Assessment

Yes. Routing Feishu setup/status probing through the existing resolved-account path is the narrow plugin-owned fix and avoids adding a core health special case.
