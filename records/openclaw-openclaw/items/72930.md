## Summary

Keep open. Current main has account-aware Feishu gateway/runtime health pieces, but the Feishu setup/status fallback still starts its probe from top-level credentials, and open PR #74060 already targets this issue with closing syntax.

## What This Changes

The issue asks Feishu health/status to treat multi-account credentials under `channels.feishu.accounts.*` as configured and probeable instead of requiring top-level `appId`/`appSecret`.

## Best Possible Solution

Feishu setup/status should resolve the selected/default Feishu account before probing, then keep regression coverage for account-scoped credentials.

## Reproduction Assessment

Yes. A config with `channels.feishu.defaultAccount` and credentials only under `channels.feishu.accounts.<id>` exercises the source-level gap: configured detection can see the account, while setup/status probing still starts from top-level credentials.

## Solution Assessment

Yes. Reusing the existing Feishu resolved-account path is the narrow plugin-owned fix and avoids adding a core health special case.
