## Summary

Keep open. Current main has account-aware Feishu health probing, but the Feishu setup/status fallback still probes only top-level credentials, and open PR #74060 is explicitly linked with closing syntax for this issue. Per policy, the issue should stay open until that PR lands or is rejected.

## What This Changes

The issue reports that `openclaw status --deep` marks Feishu as OFF/not configured when credentials are configured only under `channels.feishu.accounts.*`.

## Best Possible Solution

Land PR #74060 or an equivalent plugin-owned change so all Feishu setup/status and health-adjacent probes consistently use the selected resolved account, with regression coverage for account-scoped credentials.

## Reproduction Assessment

Unclear for the exact live `openclaw status --deep` OFF row on current main without real Feishu credentials. The source-level mismatch is deterministic: account-scoped credentials make Feishu configured, while the setup/status fallback probe still reads only top-level credentials before verifying.

## Solution Assessment

Yes. Reusing the existing resolved-account Feishu path is the narrow maintainable fix and avoids adding a core special case for Feishu credential layout.
