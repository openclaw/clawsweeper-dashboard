## Summary

Keep open. The report still has an open, unmerged implementation candidate with closing syntax, and current main still shows the Feishu setup/status fallback using top-level credentials for its probe even though account-aware Feishu health/probe plumbing exists elsewhere.

## What This Changes

Health/status should treat Feishu as configured and probeable when credentials are stored under `channels.feishu.accounts` with `defaultAccount`, not only at top level.

## Best Possible Solution

Land PR #74060 or an equivalent narrow Feishu plugin change that resolves the selected Feishu account before setup/status probing, with regression coverage for `defaultAccount` plus `accounts.*.appId/appSecret`.

## Reproduction Assessment

Yes. A config with `channels.feishu.defaultAccount` and credentials only under `channels.feishu.accounts.<id>` exercises the source-level mismatch: configured detection can succeed, while setup/status probing still starts from top-level `channels.feishu` credentials.

## Solution Assessment

Yes. Routing Feishu setup/status probing through the existing resolved-account path is the narrow plugin-owned fix and avoids adding a core health special case.
