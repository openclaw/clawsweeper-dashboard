## Summary

Keep open. Current main has a real unguarded Telegram edit path, but this PR still miscounts escaped or markup-expanded payloads against Telegram's after-entity limit and would silently cut valid content; it also lacks the required changelog entry.

## What This Changes

The branch adds a Telegram edit-message truncation helper, applies it before `editMessageText` HTML/plain fallback calls, and adds send tests for over-limit edit payloads.

## Best Possible Solution

Update the Telegram plugin edit guard to enforce Telegram's post-entity 4096-character contract, preserve safe HTML boundaries where practical, add regression coverage for entity-expanded input, and record the user-facing fix in the changelog before merging.

## Reproduction Assessment

Yes. Source inspection shows current main can send over-limit `editMessageText` payloads, and the PR regression follows from `escapeHtml` expanding `&` to `&amp;` while the helper uses rendered HTML length and raw-text omitted counts.

## Solution Assessment

No, not as submitted. The Telegram edit path is the right boundary, but the implementation should count the text Telegram counts after entity parsing, avoid cutting valid under-limit content, and add the missing changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Base truncation on Telegram-counted text length:** `extensions/telegram/src/send.ts:215-218`
  - body: The helper truncates when `htmlText.length` exceeds 4096, but Telegram's `editMessageText` limit is after entity parsing and `omittedChars` is derived only from raw text length. For input like repeated `&` characters, rendered HTML can exceed 4096 while the parsed message is still valid, so this PR cuts content with no footer because the omitted count is zero. Base the guard and omitted count on Telegram-counted visible text and add an entity-expanded regression test.
  - confidence: 0.93
