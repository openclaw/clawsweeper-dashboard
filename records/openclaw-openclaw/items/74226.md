## Summary

Keep open: the underlying Telegram edit-message bug is real, but this PR still measures the wrong length contract and lacks real Telegram after-fix proof.

## What This Changes

The PR adds a Telegram edit-message truncation helper, applies it to `editMessageTelegram` HTML/plain fallbacks, and adds focused unit coverage in `send.test.ts`.

## Best Possible Solution

Keep the fix in the Telegram plugin, but measure and truncate against Telegram's post-entity parsed text limit, preserve valid markup-expanded edits, add a regression test, and verify in a real Telegram edit flow.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Current main passes rendered HTML and plain fallback text directly to `editMessageText`, while Telegram documents the edit text limit as 1-4096 characters after entities parsing.

## Solution Assessment

No, not as submitted. The Telegram edit path is the right owner boundary, but the implementation should count what Telegram counts after entity parsing and avoid over-truncating valid rendered markup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Count edit length after Telegram entity parsing:** `extensions/telegram/src/send.ts:215-218`
  - body: The guard fires when rendered HTML exceeds 4096, but Telegram's `editMessageText` limit is after entities parsing. A valid edit with escaped ampersands or long link hrefs can render above 4096 while visible text stays under the limit; this branch then truncates content and can omit the footer because `omittedChars` is raw-text based. Base the guard and omitted count on Telegram-counted text and cover entity-expanded input.
  - confidence: 0.93
