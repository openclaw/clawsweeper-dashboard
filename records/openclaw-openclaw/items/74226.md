## Summary

Keep open: current main still has the unguarded Telegram edit path and the PR targets the right boundary, but the submitted guard can truncate valid entity-expanded messages and the external PR lacks real Telegram after-fix proof.

## What This Changes

The branch adds a Telegram edit-message truncation helper, applies it to HTML/plain `editMessageText` calls, and adds unit coverage for over-limit edit payloads.

## Best Possible Solution

Implement the edit guard against Telegram's post-entity 4096-character contract, avoid truncating valid entity-expanded HTML, cover that regression, then verify with a real Telegram edit showing the truncation marker.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Current main passes rendered HTML and plain text directly to `editMessageText`, while Telegram caps edited text at 4096 characters after entity parsing.

## Solution Assessment

No, not as submitted. The Telegram edit path is the right owner boundary, but the implementation should count the text Telegram counts after entity parsing and include an entity-expanded regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Count Telegram edit length after entity parsing:** `extensions/telegram/src/send.ts:215-218`
  - body: The helper truncates whenever rendered HTML exceeds 4096, but Telegram counts `editMessageText` length after entity parsing. A valid message like repeated ampersands renders above 4096 as `&amp;` markup while parsed text stays under the limit, so this branch would drop visible content and may add no footer because `omittedChars` is based only on raw text. Base the guard and omitted count on Telegram-counted text and cover entity-expanded input. ([core.telegram.org](https://core.telegram.org/bots/api))
  - confidence: 0.93
