## Summary

Keep open. The PR targets a real Telegram editMessageText limit bug in current main, but the submitted helper still measures the cap using rendered HTML length and raw source length instead of Telegram's after-entities text length, so valid escaped or markup-expanded edits can be silently shortened before merge.

## What This Changes

The PR adds a Telegram edit payload truncation helper, wires it into `editMessageTelegram` for HTML and plain fallback edits, and adds unit coverage for over-limit and under-limit edit payloads.

## Best Possible Solution

Revise the Telegram plugin edit guard so it truncates only when the Telegram-counted text exceeds the edit limit, keeps HTML and plain fallback payloads within the API contract, always adds a visible marker when user-visible content is actually cut, and records the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. Current main can reproduce the original failure by calling `editMessageTelegram` with more than 4096 visible characters; the PR regression is reproducible with raw text at or below 4096 characters that expands in rendered HTML, such as repeated escaped ampersands.

## Solution Assessment

No. The Telegram plugin edit path is the right boundary, but the submitted implementation should base the guard, budget, and omitted-count logic on the post-entity text Telegram counts rather than rendered HTML markup length.
