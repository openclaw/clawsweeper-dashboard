## Summary

Keep open. The PR targets a real unguarded Telegram edit path, but the latest head still bases the guard on rendered HTML length instead of Telegram's post-entity text limit and is missing the required changelog entry.

## What This Changes

The branch adds a Telegram edit-message truncation helper, uses it in `editMessageTelegram` for HTML and plain fallback edits, and adds unit coverage for over-limit edit payloads.

## Best Possible Solution

Enforce Telegram's post-entity 4096-character edit limit in the Telegram plugin, avoid truncating valid under-limit HTML, add entity-expansion regression coverage, and add an Unreleased changelog Fixes entry.

## Reproduction Assessment

Yes. Source inspection shows current main can call `editMessageText` with over-limit text, and the PR regression is source-reproducible with entity-expanded input such as repeated `&` characters whose rendered HTML exceeds 4096 while visible text does not.

## Solution Assessment

No, not as submitted. The Telegram edit path is the right boundary, but the implementation should measure the text Telegram counts after entity parsing and include the missing changelog entry before merge.
