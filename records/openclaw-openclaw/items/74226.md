## Summary

Keep open. Current main still has the reported Telegram edit-message length gap, and this PR targets the right owner boundary, but the submitted helper still uses rendered HTML length and raw-source omitted counts in a way that can truncate entity-expanded short messages without any visible marker. The PR also needs the required user-facing changelog entry before merge.

## What This Changes

This PR adds a Telegram edit-message truncation helper, applies it to HTML and plain `editMessageTelegram` payloads, and adds unit tests for over-limit and under-limit edit payloads.

## Best Possible Solution

Revise the Telegram plugin edit guard so it caps both HTML and plain fallback edits according to the text length Telegram counts after entity parsing, preserves HTML boundaries where practical, always appends a visible marker when content is actually cut, adds a regression test for entity-expanded input, and records the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. Current main can be reproduced by calling `editMessageTelegram` with a payload over Telegram's edit limit or through the native-command progress-result edit path; the PR's remaining bug is reproducible with raw text at or below 4000 characters whose rendered HTML exceeds 4096 characters, such as repeated ampersands.

## Solution Assessment

No, not as submitted. The Telegram edit path is the right fix boundary, but the helper should base its truncation decision and footer accounting on Telegram-counted/plain text rather than rendered HTML markup length, then add the missing changelog entry.
