## Summary

Keep open. The PR targets a real Telegram edit-message limit bug, but the submitted helper truncates on rendered HTML length instead of Telegram's post-entity text length, so valid under-limit content can be cut without a visible marker, and the user-facing fix is missing the required changelog entry.

## What This Changes

This PR adds a Telegram edit-message truncation helper, applies it to HTML and plain `editMessageTelegram` payloads, and adds unit tests for over-limit edit payloads.

## Best Possible Solution

Update the Telegram plugin edit guard to cap HTML and plain fallback edits by Telegram-counted parsed text, preserve safe HTML boundaries where practical, always show a marker only when visible content is actually cut, add the missing regression test, and record the fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main can be reproduced by calling `editMessageTelegram` with text over Telegram's edit limit; the PR regression is reproducible with entity-expanded input such as many ampersands where raw parsed text is under 4096 but rendered HTML exceeds 4096.

## Solution Assessment

No, not as submitted. The Telegram edit path is the right fix boundary, but the helper should count the text Telegram counts after entity parsing rather than rendered HTML markup length, then add the required changelog entry.

- **[P2] Base truncation on Telegram-counted text length:** `extensions/telegram/src/send.ts:215-218`
