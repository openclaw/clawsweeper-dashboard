## Summary

Keep open. The PR targets a real Telegram edit-message failure that current main still has, but the latest PR head still truncates based on rendered HTML markup length instead of Telegram-counted text after entity parsing, so it can cut acceptable content and omit the visible truncation marker.

## What This Changes

The PR adds a Telegram edit-message truncation helper, applies it to HTML and plain `editMessageTelegram` payloads, and adds unit tests for over-limit and under-limit edit payloads.

## Best Possible Solution

Revise the Telegram plugin edit guard so it caps edits according to the text length Telegram counts after entity parsing, preserves HTML boundaries where practical, consistently truncates the plain fallback only when needed, always appends one visible footer when content is actually cut, and records the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. Current main can be reproduced by calling `editMessageTelegram` with a payload over Telegram's edit limit or through the native-command progress-result edit path; the PR regression is reproducible with raw text such as repeated ampersands whose rendered HTML exceeds 4096 while parsed text remains under the Telegram limit.

## Solution Assessment

No, not as submitted. The Telegram edit path is the right boundary, but the helper should base its guard and omitted-count accounting on Telegram-counted/plain text rather than rendered markup length, then add the missing changelog entry.
