## Summary

Keep this PR open, but do not merge it as submitted: the Telegram edit-message fix is aimed at the right bug, but the branch still limits rendered HTML length instead of Telegram's post-entity parsed text length and it lacks real Telegram behavior proof.

## What This Changes

The PR adds a Telegram edit-message truncation helper, wires it into `editMessageTelegram` for HTML and plain fallback edits, and adds focused `send.test.ts` coverage.

## Best Possible Solution

Keep the fix in the Telegram plugin, but count and truncate against Telegram's post-entity parsed text limit, preserve valid markup-expanded edits, add entity-expanded regression coverage, and verify in a real Telegram edit flow.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Current main sends rendered HTML and plain fallback text directly to `editMessageText`, while Telegram documents the edit text limit as 1-4096 characters after entities parsing.

## Solution Assessment

No, not as submitted. The Telegram edit path is the right owner boundary, but the implementation should count what Telegram counts after entity parsing and avoid over-truncating valid rendered markup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Count edit length after Telegram entity parsing:** `extensions/telegram/src/send.ts:215-218`
  - body: The guard fires when rendered HTML exceeds 4096, but Telegram's `editMessageText` limit is after entities parsing. Escaped characters or long link hrefs can make `htmlText` exceed 4096 while the visible Telegram text is still valid; this branch then truncates content and can omit the footer because `omittedChars` is raw-text based. Base the limit and omitted count on Telegram-counted text and cover entity-expanded input.
  - confidence: 0.93
