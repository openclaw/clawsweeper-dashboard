## Summary

Keep open. The PR targets a real unguarded Telegram edit path on current main, but the submitted guard counts rendered HTML source instead of Telegram parsed text and the user-facing fix is missing the required changelog entry.

## What This Changes

This PR adds a Telegram edit-payload truncation helper, applies it to HTML and plain `editMessageText` fallback calls, and adds send-path unit coverage for over-limit edits.

## Best Possible Solution

Keep the fix in the Telegram plugin edit path, but cap by Telegram-counted parsed text, avoid truncating entity-expanded under-limit messages, add regression coverage, and record the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main can be reproduced by calling `editMessageTelegram` with more than 4096 parsed characters; the PR regression is reproducible with entity-expanded input such as raw ampersands under 4096 characters whose rendered HTML exceeds 4096.

## Solution Assessment

No, not as submitted. The edit path is the right boundary, but the guard and omitted-count calculation should be based on Telegram parsed text length, with a changelog entry before merge.
