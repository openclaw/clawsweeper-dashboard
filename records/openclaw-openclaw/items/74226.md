## Summary

Keep open: current main still has the unguarded Telegram edit path, but this PR needs changes because the helper truncates on rendered HTML length rather than Telegram's after-entity text limit and it lacks the required changelog entry.

## What This Changes

The PR adds a Telegram edit-message truncation helper, applies it to HTML/plain `editMessageText` calls, and adds unit tests for over-limit edit payloads.

## Best Possible Solution

Enforce Telegram's after-entity 4096-character edit limit in the Telegram plugin, preserve valid entity-expanded content, add regression coverage for that case, and record the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current main can pass over-limit `editMessageText` payloads directly to Telegram, and the PR regression is source-reproducible with entity-expanded input such as repeated ampersands.

## Solution Assessment

No, not as submitted. The edit path is the right boundary, but the implementation should count the text Telegram counts after entity parsing and add the missing changelog entry before merge.
