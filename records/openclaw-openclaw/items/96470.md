## What This Changes

The PR changes before-tool-call tool diagnostics to emit `tool.execution.error` for returned soft-error results and adds a focused regression test.

## Best Possible Solution

Fix error-category derivation for non-status soft errors, then add redacted real diagnostic/exporter proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main emits `tool.execution.completed` for all non-throwing tool results while the shared predicate already classifies several returned result shapes as errors.

## Solution Assessment

No, not as currently written. Reusing `isToolResultError` at the producer is the right boundary, but the category helper must avoid reporting success-like status strings such as `completed` as the error category.

- **[P2] Do not report success statuses as error categories:** `src/agents/agent-tools.before-tool-call.ts:1527`
  - body: This uses `details.status` as the `errorCategory` for every soft error. Only use the status when it is itself a failure status, otherwise fall back to a real error category.
