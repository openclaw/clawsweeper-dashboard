## Summary

Keep open. The producer-side direction is right, but the current patch can emit success-like statuses such as `completed` as error categories for returned soft-error results, and the external PR still lacks redacted real diagnostic/exporter proof.

## What This Changes

The PR changes before-tool-call diagnostics to emit `tool.execution.error` for non-throwing tool results that satisfy `isToolResultError`, plus a focused regression test.

## Best Possible Solution

Keep the producer-side `isToolResultError` classification, derive `errorCategory` only from real failure statuses or a non-status fallback, and add redacted real diagnostic/exporter proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main emits `tool.execution.completed` for all non-throwing results, while the shared predicate and tests already classify several returned result shapes as errors.

## Solution Assessment

No, not as currently written. Reusing `isToolResultError` at the producer is the right boundary, but the category derivation must not report success-like statuses as error categories.
