## What This Changes

The issue asks for standalone debug proxy captures to redact sensitive headers with the same policy used by the runtime proxy path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82951 or an equivalent redaction fix, then decide separately whether old capture rows need purge or rotation guidance.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw proxy start` persists raw HTTP response, HTTP request, and CONNECT headers into `headersJson`; a live repro would start the proxy, send sensitive headers, and query `capture_events`.

## Solution Assessment

Yes. Reusing the existing sensitive-header redaction policy at the standalone proxy persistence sites is the narrow maintainable fix, but this issue should stay open until the linked PR or equivalent lands.
