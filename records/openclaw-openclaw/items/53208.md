## Summary

Keep open. Current main still lacks a WhatsApp-owned way to resolve a bare contact name like Jack from a trusted sender's DM, and the related contact-directory issue tracks only the lookup substrate, not the full trusted-relay authorization behavior.

## What This Changes

A trusted WhatsApp direct-message sender wants name-based relay requests like "tell Jack hello" to resolve and send to an approved WhatsApp contact.

## Best Possible Solution

Add a WhatsApp-plugin-owned, account-scoped alias or opt-in contact-directory resolver that maps approved names to authorized direct targets, rejects ambiguous matches, documents privacy/retention behavior, and coordinates with https://github.com/openclaw/openclaw/issues/38467.

## Reproduction Assessment

Yes, at source level: a bare WhatsApp name is neither an explicit E.164/JID target nor present as name metadata in current WhatsApp directory entries, and WhatsApp has no resolver fallback before delivery. I did not run a live WhatsApp relay session.

## Solution Assessment

Unclear for the suggested fix as written. The maintainable path is explicit WhatsApp alias/contact metadata with authorization and ambiguity rules, not guessing arbitrary names from conversational text.
