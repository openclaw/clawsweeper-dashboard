## Summary

Keep open. Current main still supports explicit WhatsApp E.164/JID outbound sends and same-provider cross-context delivery, but it still has no WhatsApp-owned name or alias resolver that would let an approved DM sender safely address a bare name like Jack.

## What This Changes

Allow an approved WhatsApp direct-message contact to ask OpenClaw to relay a message to another WhatsApp contact by name from that separate DM.

## Best Possible Solution

Add a WhatsApp-owned, account-scoped contact alias or approved contact-directory resolver that maps names to authorized direct targets, rejects ambiguous or unauthorized matches, and coordinates with #38467.

## Reproduction Assessment

Yes, source-level reproduction is clear: a WhatsApp target like Jack is neither an E.164/JID nor present as name metadata in the current WhatsApp directory, and WhatsApp has no resolver fallback before delivery. I did not run a live WhatsApp relay session.

## Solution Assessment

Unclear for the suggested fix as written. The desired relay behavior should be implemented through explicit WhatsApp alias or contact metadata plus authorization and ambiguity rules, not by guessing arbitrary names from prompt text or inbound display names.
