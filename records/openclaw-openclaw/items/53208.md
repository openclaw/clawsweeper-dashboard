## Summary

Keep open. Current main still does not implement the WhatsApp name or alias resolution needed for a trusted sender in a separate DM to say “tell Jack ...”, and the related contact-directory issue only covers part of the remaining product/privacy work.

## What This Changes

The issue asks for a trusted WhatsApp DM sender to relay a message to an approved contact by name instead of being told that the target chat is unavailable.

## Best Possible Solution

Add a WhatsApp-plugin-owned, account-scoped alias or opt-in contact-directory resolver with allowlist enforcement, ambiguity rejection, retention/deletion docs, and coordination with https://github.com/openclaw/openclaw/issues/38467.

## Reproduction Assessment

Yes at source level: a bare WhatsApp name is not an explicit E.164/JID target, current WhatsApp directory entries do not carry contact names, and there is no WhatsApp resolver fallback before delivery. I did not run a live WhatsApp relay session.

## Solution Assessment

Unclear for the suggested fix as written. The maintainable path is explicit WhatsApp alias/contact metadata with authorization, opt-in privacy controls, and ambiguity rules, not guessing arbitrary names from conversational text.
