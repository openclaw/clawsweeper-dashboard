## Label Justifications

- `P3`: This is an optional WhatsApp contact-name relay feature with product/privacy scope, not a broken documented core contract.
- `impact:message-loss`: The reported user-visible effect is that a trusted sender's requested outbound WhatsApp relay is not sent.
- `impact:security`: Any solution would change trusted relay authority and contact-name persistence/lookup, which touches authorization and sensitive contact data.

## Summary

Close as ClawHub/plugin scope. Current main and the latest shipped release still treat WhatsApp delivery targets as explicit E.164/JIDs, while bare contact-name relay needs an opt-in contact/alias resolver and trusted-relay policy that current plugin seams can support outside OpenClaw core.

## What This Changes

Request that a trusted WhatsApp DM sender be able to ask OpenClaw to relay a message to another contact by name and have it send a fresh outbound WhatsApp message.

## Best Possible Solution

Keep OpenClaw core and the official WhatsApp plugin scoped to explicit targets; publish opt-in contact-name or trusted-relay resolver experiments through ClawHub, or propose a plugin API only if an external implementation proves a missing seam.

## Reproduction Assessment

Yes at source level: a bare WhatsApp name like Jack is not an E.164/JID target, the current WhatsApp directory supplies allowFrom IDs without names, and WhatsApp has no resolver fallback for aliases. I did not run a live WhatsApp relay session.

## Solution Assessment

Yes for cleanup. The maintainable direction is an explicit opt-in contact/alias resolver with authorization and privacy rules outside core, not guessing bare names inside the bundled runtime.
