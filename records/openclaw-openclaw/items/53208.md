## Summary

Keep open. Current main still supports explicit WhatsApp E.164/JID outbound sends and same-provider cross-context delivery, but it does not provide a WhatsApp contact-name or alias resolver for a bare recipient like “Jack” from another approved DM. The related closed reports cover explicit target normalization and configured-channel selection, not this name-based trusted relay behavior.

## What This Changes

The issue requests that an approved WhatsApp direct-message contact be able to ask OpenClaw to relay a message to another approved WhatsApp contact by name.

## Best Possible Solution

Add a WhatsApp-owned configured contact alias or metadata path that maps approved names to approved direct WhatsApp targets, rejects ambiguous or unauthorized names, and covers an authorized DM sender relaying to another approved target without weakening the generic core cross-context policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with WhatsApp allowFrom entries containing only phone/JID IDs, a message action target like `Jack` reaches `resolveChannelTarget`, finds no directory name/handle match and no WhatsApp plugin fallback resolver, and returns an unknown-target error; no live WhatsApp run was performed because this review was read-only.

## Solution Assessment

No for a naive bare-name relay implementation. The maintainable fix needs explicit WhatsApp alias/contact metadata plus ambiguity and authorization rules, not automatic guessing from arbitrary prompt text or inbound display names.
