## Summary

Keep open. Current main still supports explicit WhatsApp E.164/JID outbound sends and allows same-provider cross-context delivery, but it still lacks a safe WhatsApp name or alias resolver for a bare recipient like "Jack" from another approved DM. The open WhatsApp contact-directory issue is related, but it does not fully settle this issue's trusted-relay authorization and ambiguity semantics.

## What This Changes

The issue asks OpenClaw to let an approved WhatsApp DM sender relay a message to another approved WhatsApp contact by name instead of claiming the target chat is unavailable.

## Best Possible Solution

Add a WhatsApp-owned, account-scoped contact alias or approved contact-directory path that maps names to approved direct WhatsApp targets, rejects ambiguous or unauthorized names, and coordinates with the broader contact-directory work in #38467 without weakening generic core cross-context policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on current main, a target like `Jack` is not a WhatsApp E.164/JID, WhatsApp directory peers are ID-only from `allowFrom`, and WhatsApp supplies no resolver fallback, so `resolveChannelTarget` returns an unknown-target error before delivery.

## Solution Assessment

Unclear for the issue's suggested fix as written. The behavior should be solved through explicit WhatsApp alias/contact metadata plus authorization and ambiguity rules, not by guessing arbitrary names from prompt text or inbound display names.
