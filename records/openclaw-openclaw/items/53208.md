## Summary

Keep open. Current main can send to explicit WhatsApp E.164/JID targets and does not block same-provider cross-context sends by default, but it still lacks a WhatsApp contact-name or alias resolver for a bare target such as “Jack” from another approved DM. The adjacent closed WhatsApp outbound fixes cover explicit target normalization and provider selection, not this name-based trusted relay path.

## What This Changes

Allow an approved WhatsApp direct-message contact to ask OpenClaw to relay a message by another approved contact's name to that WhatsApp recipient.

## Best Possible Solution

Add a WhatsApp-owned configured contact alias or metadata path that resolves approved names to approved direct WhatsApp targets, rejects ambiguous or unauthorized names, and adds regression coverage for an authorized DM sender relaying to another approved target without weakening the generic core cross-context policy.
