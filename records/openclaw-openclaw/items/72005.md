## Summary

Keep open. Current main still lacks the outbound footer/context-warning feature, and the open PR remains the relevant implementation path for #71984, but the reviewed head has concrete merge-blocking bugs in gateway delivery, dry-run state mutation, and context-warning threshold persistence.

## What This Changes

This PR adds server-side outbound footer stripping/rendering, context-warning config and session state, sendMessage wiring, generated schema updates, utility tests, and a new docs page.

## Best Possible Solution

Keep this PR as the implementation path for #71984, but revise it so every real delivery mode sends runtime-processed text, dry runs remain read-only, context warning eligibility is checked and recorded atomically, crossed thresholds cannot fire later out of order, and user-facing changelog/docs/tests cover those contracts.

## Reproduction Assessment

Yes. Static inspection of PR head 5bfcdd039a7a222e41f137a0c9b7c302e005ce5a gives a high-confidence reproduction path: gateway sends raw params.content after processedContent is computed, dryRun returns after the stateful hook, and threshold state is decided before the locked store update.

## Solution Assessment

No. The product direction is sound, but the current patch is not the best mergeable form until the outbound pipeline and session-state semantics are tightened; a focused revision in this PR or a narrow replacement branch is safer than a separate redesign.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Send processed content through gateway mode:** `src/infra/outbound/message.ts:337`
  - body: The hook result is only fed into outboundPlan; the gateway branch still sends message: params.content. Any channel with deliveryMode: "gateway" bypasses footer stripping/rendering and warning prepends, so fabricated model footers can still reach users.
  - confidence: 0.95
- **[P2] Skip the stateful hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: applyOutboundFooterHook runs before the dryRun early return, and the hook can persist contextWarningThresholdsTriggered. A dry run can therefore consume a one-shot warning without delivering a message, causing the next real send to omit it.
  - confidence: 0.92
- **[P2] Record all crossed thresholds when warning fires:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps across multiple thresholds, the hook persists only the selected highest threshold. Lower crossed thresholds remain unwarned and can fire later out of order, which breaks the once-per-threshold contract.
  - confidence: 0.91
- **[P2] Make warning check and record atomic:** `src/infra/outbound/footer-hook.ts:155-177`
  - body: The warning decision uses alreadyWarned before the session-store update. Two concurrent sends for the same session can both read the old state and both emit the same one-shot warning; re-check eligibility inside the locked update before recording.
  - confidence: 0.86
- **[P3] Add the user-facing changelog entry:** `CHANGELOG.md:5`
  - body: This is a user-facing feat with new config, outbound behavior, and docs, but the PR does not update CHANGELOG.md. Add an Unreleased Changes entry with allowed contributor credit before merge.
  - confidence: 0.88
