## Summary

Keep open. Current main still lacks the outbound footer/context-warning feature, and the PR is a useful implementation path for #71984, but the current patch has concrete correctness blockers in gateway delivery, dry-run state mutation, and one-shot threshold tracking.

## What This Changes

This PR adds server-side outbound footer stripping/rendering, context-warning config and session state, sendMessage wiring, generated schema updates, tests, and a new docs page.

## Best Possible Solution

Keep this PR or a narrow replacement as the implementation path, but revise it so processed text reaches every real delivery mode, dry runs stay read-only, warning eligibility is checked and recorded atomically, crossed thresholds cannot warn later out of order, and the user-facing changelog/tests document those contracts.

## Reproduction Assessment

Yes. Static inspection of PR head 5bfcdd039a7a222e41f137a0c9b7c302e005ce5a reproduces the blockers: gateway mode sends params.content, dryRun returns after the stateful hook, and warning state is decided before the locked store update.

## Solution Assessment

No. The product direction is sound, but the current patch is not the best mergeable form until the outbound pipeline and session-state semantics are tightened with focused fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Send processed content through gateway mode:** `src/infra/outbound/message.ts:337`
  - body: The hook result is only fed into the direct outbound plan; gateway mode still sends `message: params.content`. Gateway-delivered replies can therefore bypass footer stripping/rendering and warning prepends, leaving the core feature ineffective on those channels.
  - confidence: 0.95
- **[P2] Skip the stateful hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` runs before the `dryRun` early return, and the hook can persist `contextWarningThresholdsTriggered`. A dry run can consume a one-shot warning without delivering a message, so the next real send may omit it.
  - confidence: 0.92
- **[P2] Record all crossed thresholds when warning fires:** `src/infra/outbound/footer-hook.ts:195-198`
  - body: When usage jumps across multiple thresholds, the hook persists only the selected highest threshold. Lower crossed thresholds remain unwarned and can fire later out of order, breaking the once-per-threshold contract.
  - confidence: 0.91
- **[P2] Make warning check and record atomic:** `src/infra/outbound/footer-hook.ts:155-157`
  - body: The warning decision uses `alreadyWarned` before the session-store update. Two concurrent sends for the same session can both read the old state and both emit the same one-shot warning; re-check eligibility inside the locked update before recording.
  - confidence: 0.86
- **[P3] Add the user-facing changelog entry:** `CHANGELOG.md:5`
  - body: This is a user-facing feature with new config, outbound behavior, and docs, but the PR does not update `CHANGELOG.md`. Add an Unreleased Changes entry with allowed contributor credit before merge.
  - confidence: 0.88
