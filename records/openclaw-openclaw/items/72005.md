Full review comments:

- **[P1] Send processed content through gateway mode:** `src/infra/outbound/message.ts:244-249`
  - body: `processedContent` is computed here and used for the direct outbound plan, but the gateway branch later still sends `message: params.content`. Any channel using `deliveryMode: "gateway"` will still show fabricated model footers and miss server warnings, so the processed text needs to feed the gateway params and coverage.
  - confidence: 0.94
- **[P2] Keep dry runs from consuming warning state:** `src/infra/outbound/message.ts:244-249`
  - body: The hook runs before the `params.dryRun` return, and `applyOutboundFooterHook` can persist `contextWarningThresholdsTriggered`. A dry run can therefore consume a one-shot warning without delivering a message; skip stateful warning persistence for dry runs or move it after the dry-run branch.
  - confidence: 0.9
- **[P2] Record all crossed warning thresholds:** `src/infra/outbound/footer-hook.ts:195`
  - body: Persisting only `newThreshold` leaves skipped lower thresholds unwarned when usage jumps across multiple thresholds. After higher thresholds are recorded, a later turn can emit a lower warning out of order; persist every configured threshold up to the chosen value or return the full set from the evaluator.
  - confidence: 0.9
- **[P2] Recheck warning eligibility inside the store write:** `src/infra/outbound/footer-hook.ts:155-173`
  - body: `alreadyWarned` is read before the message text is built, then the store write happens later. Two concurrent sends for the same session can both prepend the same warning before either write records it; re-evaluate and claim the threshold inside the serialized `updateSessionStore` callback.
  - confidence: 0.84
