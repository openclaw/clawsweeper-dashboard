## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use status token semantics for footer values:** `src/infra/outbound/footer-hook.ts:101-107`
  - body: `loadFooterContext` treats `entry.contextTokens` as current usage and `agents.defaults.contextTokens` as the limit. Current status code uses `totalTokens` for usage and `contextTokens` as the window limit, so a normal session can render a false 100% footer and fire warnings immediately.
  - confidence: 0.96
- **[P1] Send processed content through gateway mode:** `src/infra/outbound/message.ts:337`
  - body: The hook result is only fed into the outbound plan, but gateway delivery still sends `message: params.content`. Gateway-mode channels will bypass footer stripping/rendering and context warnings, leaving the feature ineffective there.
  - confidence: 0.95
- **[P2] Skip the stateful hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` runs before the `dryRun` early return, and the hook can persist `contextWarningThresholdsTriggered`. A dry run can consume a one-shot warning without delivering any message.
  - confidence: 0.92
- **[P2] Record all crossed thresholds when warning fires:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps across multiple configured thresholds, the hook persists only the selected highest threshold. Lower crossed thresholds remain unwarned and can later fire out of order, breaking the once-per-threshold contract.
  - confidence: 0.91
- **[P2] Claim warning thresholds inside the store update:** `src/infra/outbound/footer-hook.ts:155-157`
  - body: Warning eligibility is computed from `ctx.entry` before the serialized store update. Two concurrent sends can both emit the same one-shot warning; recheck and claim the threshold inside the `updateSessionStore` mutator.
  - confidence: 0.86
