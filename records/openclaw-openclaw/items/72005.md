## Summary

Keep open: current main still lacks the server-rendered footer feature, and this PR remains the canonical implementation path for https://github.com/openclaw/openclaw/issues/71984, but the patch is not mergeable yet because gateway delivery bypasses the processed text, dry runs can mutate warning state, warning claims are incomplete/non-atomic, and contributor real behavior proof is missing.

## What This Changes

The branch adds a core outbound footer processor, messages config/session state for footer rendering and context warnings, sendMessage wiring, tests, generated schema, and docs.

## Best Possible Solution

Keep this as the canonical implementation path, but make the outbound hook reach every delivery mode, keep dry runs read-only, claim warning thresholds completely and atomically, preserve current-main payload planning, and require live channel proof before merge.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows gateway mode sends raw content, dry runs execute the stateful hook first, and warning eligibility is decided before the serialized store update. The feature itself still lacks live outbound-channel proof.

## Solution Assessment

No. The overall direction fits the linked user problem, but the current patch is not the best mergeable solution until the delivery, dry-run, warning-state, rebase, and proof gaps are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Send processed content through gateway delivery:** `src/infra/outbound/message.ts:244-249`
  - body: The hook result is only fed into the direct outbound plan, but the gateway branch still sends `message: params.content`. Gateway-mode channels will bypass footer stripping/rendering and warning prepends, leaving fabricated model footers and missing server-authored warnings there.
  - confidence: 0.95
- **[P2] Skip the stateful hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` runs before the dry-run return and can persist `contextWarningThresholdsTriggered`. A dry run can consume a one-shot warning without delivering a message, so the next real send may omit it.
  - confidence: 0.92
- **[P2] Record all crossed thresholds when warning fires:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps across multiple configured thresholds, the hook persists only the selected highest threshold. Lower crossed thresholds remain unwarned and can later fire out of order, breaking the once-per-threshold contract.
  - confidence: 0.91
- **[P2] Claim warning thresholds inside the store update:** `src/infra/outbound/footer-hook.ts:155-157`
  - body: Warning eligibility is computed from `ctx.entry` before the serialized store update. Two concurrent sends can both emit the same one-shot warning; recheck and claim the threshold inside the `updateSessionStore` mutator.
  - confidence: 0.86
