Overall confidence: 0.92

Full review comments:

- **[P1] Send processed content through gateway mode:** `src/infra/outbound/message.ts:244-249`
  - body: `processedContent` is only fed into the direct outbound payload plan, but the gateway branch still sends `params.content`. Any `deliveryMode: "gateway"` channel can still show fabricated footers and miss server warnings, so the processed payload must feed both direct and gateway sends after the current-main rebase.
  - confidence: 0.94
- **[P2] Skip stateful footer hooks during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: The hook runs before the `params.dryRun` return and can persist `contextWarningThresholdsTriggered`. A dry-run send can consume a one-shot warning without delivering it, so dry runs must not execute the state-mutating warning path.
  - confidence: 0.91
- **[P2] Record every crossed warning threshold:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps over multiple configured thresholds, the hook records only the selected highest threshold. Lower crossed thresholds remain unwarned and can fire later after higher warnings, violating the once-per-threshold session guarantee.
  - confidence: 0.9
- **[P2] Recheck warning eligibility inside the store update:** `src/infra/outbound/footer-hook.ts:187-195`
  - body: Warning eligibility is computed before `updateSessionStore`, so concurrent sends for the same session can both decide to emit the same one-shot warning. Re-evaluate or claim the threshold inside the store update so the check-and-set is atomic for the emitted warning.
  - confidence: 0.83
