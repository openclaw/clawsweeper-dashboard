## Summary

Keep open: this remains the canonical implementation path for the linked footer request, but the current PR is not mergeable because source inspection confirms delivery/state bugs and the external contributor has not posted real outbound-channel proof.

## What This Changes

The branch adds a server-side outbound footer and context-warning pipeline, messages config/session state, schema/docs, and unit tests for stripping model-authored footers and rendering runtime-authored status text.

## Best Possible Solution

Land this only after rebasing onto current outbound payload/durable delivery, applying processed text to every delivery mode, keeping dry runs read-only, making threshold claims complete and atomic, and adding redacted live outbound proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows gateway mode sends raw content, dry runs execute the stateful hook first, and warning state is decided before the serialized store update. The feature itself still lacks a posted real outbound-channel proof path.

## Solution Assessment

No. The direction fits the linked user problem, but the current patch is not the best mergeable form until the delivery pipeline, dry-run behavior, warning-state semantics, rebase conflicts, and proof gap are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Send processed content through gateway mode:** `src/infra/outbound/message.ts:244-249`
  - body: The hook result is only fed into the outbound plan, but gateway delivery still sends `message: params.content`. Gateway-mode channels will bypass footer stripping/rendering and warning prepends, leaving the feature ineffective there.
  - confidence: 0.95
- **[P2] Skip the stateful hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` runs before the dry-run return and can persist `contextWarningThresholdsTriggered`. A dry run can consume a one-shot warning without delivering a message, so the next real send may omit it.
  - confidence: 0.92
- **[P2] Record all crossed thresholds when warning fires:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps across multiple configured thresholds, the hook persists only the selected highest threshold. Lower crossed thresholds remain unwarned and can later fire out of order, breaking the once-per-threshold contract.
  - confidence: 0.91
- **[P2] Claim warning thresholds inside the store update:** `src/infra/outbound/footer-hook.ts:155-157`
  - body: Warning eligibility is computed from `ctx.entry` before the serialized store update. Two concurrent sends can both emit the same one-shot warning; recheck and claim the threshold inside the `updateSessionStore` mutator.
  - confidence: 0.86
