## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the processed text for gateway sends:** `src/infra/outbound/message.ts:337`
  - body: The hook result is only used for the direct outbound payload plan. The gateway branch still sends `params.content`, so gateway-delivered replies can show fabricated model footers and miss server-authored warnings even when the feature is enabled.
  - confidence: 0.94
- **[P1] Do not add a second usage footer config path:** `src/config/types.messages.ts:157-159`
  - body: Current main already supports runtime-authored `/usage full` rendering through `messages.usageTemplate`. Adding parallel `outboundFooter` and `contextWarning` config creates competing footer behavior unless maintainers explicitly approve and document why the existing usageState path is insufficient.
  - confidence: 0.89
- **[P2] Skip the stateful hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` runs before the `dryRun` return, and it can persist `contextWarningThresholdsTriggered`. A dry run can consume a one-shot warning without delivering any message, so the next real send may omit the warning.
  - confidence: 0.93
- **[P2] Record crossed thresholds when persisting a warning:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps across multiple thresholds, the hook records only the chosen highest threshold. Lower crossed thresholds remain unwarned and can fire later after higher warnings, breaking the once-per-threshold ordering the PR promises.
  - confidence: 0.88
- **[P2] Make warning emission atomic with the session write:** `src/infra/outbound/footer-hook.ts:155-177`
  - body: The warning decision reads `alreadyWarned` before processing and only writes after the warning text has already been emitted. Concurrent sends for the same session can both decide to warn from the same stale state and deliver duplicate one-shot warnings.
  - confidence: 0.81
