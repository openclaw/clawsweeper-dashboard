## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Send processed content through gateway mode:** `src/infra/outbound/message.ts:337`
  - body: The hook result is only fed into the outbound plan, but the gateway branch still sends `message: params.content`. Gateway-delivered replies can therefore bypass footer stripping/rendering and warning prepends, leaving the core feature ineffective on those channels.
  - confidence: 0.95
- **[P2] Skip the stateful hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` runs before the `dryRun` return, and the hook can persist `contextWarningThresholdsTriggered`. A dry run can consume a one-shot warning without delivering a message, so the next real send may omit it.
  - confidence: 0.92
- **[P2] Record all crossed thresholds when warning fires:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps across multiple thresholds, the hook persists only the selected highest threshold. Lower crossed thresholds remain unwarned and can fire later out of order, breaking the once-per-threshold contract.
  - confidence: 0.91
- **[P2] Make warning check and record atomic:** `src/infra/outbound/footer-hook.ts:155-157`
  - body: The warning decision uses `alreadyWarned` before the session-store update. Two concurrent sends for the same session can both read the old state and both emit the same one-shot warning; re-check eligibility inside the locked update before recording.
  - confidence: 0.86
- **[P2] Resolve footer limits like session status:** `src/infra/outbound/footer-hook.ts:36-41`
  - body: The hook only reads `agents.defaults.contextTokens` for `{context_limit}` and warning thresholds, while current status resolves selected/runtime model limits and configured caps. In common model-catalog setups the footer can render `?` and warnings never fire even though status knows the limit.
  - confidence: 0.82
- **[P3] Add the user-facing changelog entry:** `CHANGELOG.md:5`
  - body: This is a user-facing feature with new config, outbound behavior, and docs, but the PR does not update `CHANGELOG.md`. Add an Unreleased Changes entry with allowed contributor credit before merge.
  - confidence: 0.88
