Overall confidence: 0.88

Full review comments:

- **[P1] Send processed text through the gateway branch:** `src/infra/outbound/message.ts:337`
  - body: The hook output is only used to build `outboundPlan`; the gateway branch still sends `message: params.content`. Gateway channels can still show fabricated model footers and miss server-authored warnings, which breaks the main feature for any gateway-delivered channel.
  - confidence: 0.94
- **[P2] Skip the stateful hook for dry-run sends:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` runs before the `dryRun` early return, and that hook can persist `contextWarningThresholdsTriggered`. A dry run can consume a one-shot warning without sending a message, so the next real send may incorrectly omit it.
  - confidence: 0.9
- **[P2] Record all thresholds crossed by the warning:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps over multiple thresholds, `evaluateContextWarning` picks only the highest unwarned one and this update records only that value. Lower skipped thresholds remain unrecorded and can fire later after higher thresholds have already warned, violating the once-per-threshold contract.
  - confidence: 0.9
- **[P2] Resolve footer limits from the status telemetry path:** `src/infra/outbound/footer-hook.ts:37-41`
  - body: The PR claims the footer uses the same source as `session_status`, but this helper only reads `agents.defaults.contextTokens`. Current status resolves the active context limit from runtime snapshots, persisted session contextTokens, model config, channel overrides, and fallback state, so the footer can render wrong percentages or suppress warnings whenever the default is absent or stale.
  - confidence: 0.86
