## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Send processed content through gateway mode:** `src/infra/outbound/message.ts:252`
  - body: The hook result is only fed into the outbound plan, but gateway delivery still sends `message: params.content`. Gateway-mode channels will bypass footer stripping/rendering and warning prepends entirely.
  - confidence: 0.95
- **[P2] Skip the stateful footer hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` runs before the `dryRun` early return and can persist `contextWarningThresholdsTriggered`. A dry run can consume a one-shot warning without delivering any message.
  - confidence: 0.92
- **[P2] Record all crossed thresholds when warning fires:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps across multiple configured thresholds, the hook persists only the selected highest threshold. Lower crossed thresholds remain unwarned and can later fire out of order.
  - confidence: 0.91
- **[P2] Claim warning thresholds inside the store update:** `src/infra/outbound/footer-hook.ts:155-157`
  - body: Warning eligibility is computed from `ctx.entry` before the serialized store update. Two concurrent sends can both emit the same one-shot warning, so the check-and-claim needs to happen inside the mutator.
  - confidence: 0.86
