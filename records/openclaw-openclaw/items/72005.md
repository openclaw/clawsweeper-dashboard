Full review comments:

- **[P1] Send processed content through gateway delivery:** `src/infra/outbound/message.ts:244-249`
  - body: `processedContent` is only fed into the payload plan, while the gateway branch still sends `message: params.content`. Gateway delivery-mode channels therefore bypass footer stripping/rendering and warning prepends, so fabricated footers can still reach users through that path.
  - confidence: 0.94
- **[P2] Skip the stateful hook during dry runs:** `src/infra/outbound/message.ts:244-249`
  - body: The hook runs before the `dryRun` return and can persist `contextWarningThresholdsTriggered`. A dry-run send can consume a one-shot warning without delivering the message, causing the next real send to omit it.
  - confidence: 0.91
- **[P2] Record all crossed thresholds when warning fires:** `src/infra/outbound/footer-hook.ts:195`
  - body: When usage jumps across multiple configured thresholds, the hook persists only the chosen highest threshold. Lower crossed thresholds remain unwarned and can fire later after higher thresholds are already recorded, violating the once-per-threshold behavior.
  - confidence: 0.9
- **[P2] Claim warning thresholds inside the store update:** `src/infra/outbound/footer-hook.ts:156-177`
  - body: Warning eligibility is decided from `alreadyWarned` before entering `updateSessionStore`. Two concurrent sends for the same session can both decide to emit the same warning before either writer records the threshold, so users can see duplicate one-shot warnings.
  - confidence: 0.86
- **[P2] Reuse the status context-limit resolver:** `src/infra/outbound/footer-hook.ts:36-41`
  - body: The hook only reads `agents.defaults.contextTokens` for `contextLimit`, but `session_status` resolves model/provider context windows, persisted runtime caps, configured caps, transcript usage, and defaults. In common configs where `/status` knows the context window, this footer renders `?` and threshold warnings never fire.
  - confidence: 0.84
