## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Send the processed text through gateway sends:** `src/infra/outbound/message.ts:252`
  - body: This line feeds `processedContent` only into the outbound payload plan. The gateway branch below still sends `message: params.content`, so gateway-mode channels can still show fabricated footers and will miss the server-rendered footer/warning path even when configured.
  - confidence: 0.94
- **[P2] Skip the stateful hook for dry-run sends:** `src/infra/outbound/message.ts:244-249`
  - body: `applyOutboundFooterHook` can persist `contextWarningThresholdsTriggered`, but it runs before the `dryRun` early return. A preview send can consume a one-shot warning without delivering a message, causing the next real send to omit it.
  - confidence: 0.91
- **[P2] Persist all thresholds crossed by the emitted warning:** `src/utils/outbound-footer.ts:167-168`
  - body: The evaluator returns only the chosen highest threshold. If usage jumps from below 70% to 88%, only 85 is stored; after 95 is later stored, 70 remains eligible and can emit a stale lower warning, breaking the once-per-threshold guarantee.
  - confidence: 0.9
- **[P2] Claim warning thresholds atomically:** `src/infra/outbound/footer-hook.ts:155-157`
  - body: Warning eligibility is decided from `alreadyWarned` before `updateSessionStore` runs. Two concurrent sends for the same session can both read the old state, both prepend the same warning, and only then write the threshold.
  - confidence: 0.86
- **[P2] Resolve the context limit like status output:** `src/infra/outbound/footer-hook.ts:107`
  - body: The hook only uses `agents.defaults.contextTokens` as the limit, while current status output resolves active model/runtime/persisted/configured caps. In normal model-catalog setups without that default, the footer renders `?` and warnings never fire despite `session_status` having a real context limit.
  - confidence: 0.88
