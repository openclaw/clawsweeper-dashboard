## Summary

Keep open: current main still has enough split heartbeat default/status/scheduler logic that this observability mismatch is plausible, and I found no merged fix or shipped release that specifically aligns the reported `openclaw status` interval with delivered heartbeat cadence.

## What This Changes

The issue reports that `openclaw status` shows a 1h heartbeat interval while native Telegram heartbeats arrive on the 30m default when `heartbeat.every` is unset.

## Best Possible Solution

Make status report the same effective per-agent interval that the native heartbeat scheduler actually uses, with regression coverage for unset `heartbeat.every`, Codex runtime, and unrelated Anthropic/Claude auth profiles.

## Reproduction Assessment

No high-confidence live reproduction was run. The issue gives concrete v2026.5.6 evidence, and current source shows the relevant status/default/scheduler resolution paths, but a current-main gateway loop should still confirm the 30m delivery side before landing a fix.

## Solution Assessment

Unclear from source alone which interval should win in every mixed-auth setup, but the maintainable fix is to unify status and scheduler around one effective heartbeat resolver rather than changing display text independently.
