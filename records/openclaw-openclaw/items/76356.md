## Summary

Keep open. The PR addresses a real subagent startup observability bug, but the latest diff still lets the new startup-failure outcome be followed by a normal terminal end path, and the gateway session lifecycle path does not yet route `startup-failed` as a terminal event.

## What This Changes

The PR adds embedded subagent `first-progress` and `startup-failed` lifecycle events, routes startup failures into subagent and gateway status snapshots, and updates docs, changelog, and focused tests.

## Best Possible Solution

Merge a revision that makes `startup-failed` terminal across the embedded subscriber, subagent registry, `agent.wait`, and gateway session lifecycle surfaces, with regression coverage for late `end` noise after startup failure.

## Reproduction Assessment

Yes, from source inspection. The PR-added branch emits `startup-failed` and the added test asserts a follow-up `end`, while current consumers can treat a plain `end` as a successful or non-failed terminal state.

## Solution Assessment

No. The additive lifecycle design is the right direction, but the implementation should make startup failure a single stable terminal outcome or explicitly ignore/non-success later `end` events, and it should route `startup-failed` through server-chat session finalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep startup-failed from being overwritten by end:** `src/agents/pi-embedded-subscribe.handlers.lifecycle.ts:118-138`
  - body: This branch emits `startup-failed` and then continues into the existing terminal `end` emission; the added test even expects both callbacks. That later `end` can reach agent-job, subagent registry, and session surfaces as a normal completion or non-startup terminal state, overwriting the startup error this PR is trying to expose. Please return after startup failure, mark the later terminal event as failed, or ignore subsequent `end` events once startup has failed.
  - confidence: 0.91
- **[P2] Route startup-failed through session finalization:** `src/gateway/session-lifecycle-state.ts:6`
  - body: The mapper now recognizes `startup-failed`, but `server-chat.ts` only finalizes, persists, and broadcasts `end` and `error` phases. If `startup-failed` becomes the terminal event, session subscribers and persisted session state will not see the failure; with the current follow-up `end`, they see the wrong terminal state instead. Please update the server-chat phase filters and add coverage for this event.
  - confidence: 0.86
