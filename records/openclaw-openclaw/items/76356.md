## Summary

Keep open. The PR targets a valid subagent startup-hang bug, but the current branch is not yet safe to merge because `startup-failed` can be followed by a normal terminal event and is not fully routed through live `agent.wait` or Gateway session finalization.

## What This Changes

The PR adds `first-progress` and `startup-failed` lifecycle phases for embedded subagents, propagates startup failures into registry/agent-job/session snapshots, and updates docs, changelog, and tests.

## Best Possible Solution

Merge a revision that treats `startup-failed` as one stable terminal startup error across embedded runs, subagent registry, `agent.wait`, and Gateway session/chat lifecycle, with focused regression coverage and aligned docs/changelog.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main has only `start`/`end`/`error`, and the PR-added aborted-before-progress path emits `startup-failed` followed by `end`. I did not execute tests because this was a read-only review and `pnpm` is unavailable in the checkout environment.

## Solution Assessment

No. The additive lifecycle direction is reasonable for the linked bug, but this implementation needs to make startup failure terminal and route it through the same live wait and session finalization paths as other terminal outcomes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep startup-failed from being overwritten by end:** `src/agents/pi-embedded-subscribe.handlers.lifecycle.ts:118-138`
  - body: This branch emits `startup-failed` for an aborted run before any progress, but then falls through to the existing terminal `end`/`error` emission; the added test asserts the follow-up `end`. That later terminal event can overwrite the startup failure in registry/session consumers, so the startup-failed path needs to return or later events must be marked/ignored as the same failure.
  - confidence: 0.91
- **[P2] Route startup-failed through session finalization:** `src/gateway/session-lifecycle-state.ts:6`
  - body: The mapper accepts `startup-failed`, but `server-chat.ts` still only finalizes, persists, and broadcasts `end` and `error`. Once the startup failure is terminal, Control UI/session subscribers will miss it unless the server-chat lifecycle filters are updated.
  - confidence: 0.86
- **[P2] Teach live agent.wait to finish startup failures:** `src/gateway/server-methods/agent-job.ts:182-185`
  - body: The global agent-run listener records `startup-failed`, but the per-call `waitForAgentJob` listener still drops every phase except `end` and `error`. If the extra follow-up `end` is removed, an in-flight `agent.wait` can sit until its timeout instead of returning the startup failure.
  - confidence: 0.83
