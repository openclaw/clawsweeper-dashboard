## Summary

Keep open. The PR addresses a real subagent lifecycle gap, but the current patch can still turn a startup failure into a successful completion because it emits `startup-failed` and then falls through to the existing `end` event path.

## What This Changes

The PR adds embedded subagent `first-progress` and `startup-failed` lifecycle signals, wires them through subagent completion and gateway session/job snapshots, and updates docs, changelog, and focused tests.

## Best Possible Solution

Merge a version that makes `startup-failed` terminal across the registry and agent wait surfaces, with regression coverage for the exact `startup-failed` then `end` event sequence.

## Reproduction Assessment

Yes, from source inspection. Current main lacks the first-progress/startup-failed signals, and the PR-introduced ordering bug is visible because the new startup-failed branch is followed by the existing terminal end emission path.

## Solution Assessment

No. The additive lifecycle design is the right direction, but this implementation needs to prevent the follow-up `end` event from downgrading or racing the startup-failed outcome.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep startup-failed terminal after the follow-up end:** `src/agents/pi-embedded-subscribe.handlers.lifecycle.ts:147-167`
  - body: This branch emits `startup-failed` and then continues to the existing terminal `end` emission; the added test even asserts both lifecycle callbacks. The new registry branch records the startup error, but the following `end` event has no `aborted`/`yielded` flag, so it reaches the normal success path and can overwrite or race the failed outcome. Please return after the startup failure, mark the final event as non-success, or ignore later `end` events once a run is terminally startup-failed.
  - confidence: 0.9
