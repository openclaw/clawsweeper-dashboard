## Summary

Keep open. Current main still has source-backed transcript lock reclaim and startup cleanup behavior tied to the reported delayed/lost response pattern, while the focused related PR was closed unmerged and later session-store, timeout, and performance fixes do not prove this Telegram/tool-run lock cleanup symptom is fixed.

## What This Changes

The issue reports lost or delayed main-agent channel responses after tool execution when session lock cleanup removes transcript locks as dead-pid during active work.

## Best Possible Solution

Land an owner-reviewed session write-lock fix that protects active same-PID/no-starttime and active-turn transcript owners while preserving safe dead-PID, recycled-PID, non-OpenClaw-owner, and bounded stale-lock recovery.

## Reproduction Assessment

Yes at source level: current source and tests still show same-PID/no-starttime transcript locks can be reclaimed when not tracked in the in-process manager, and startup cleanup still removes stale .jsonl locks without active-turn awareness. The full Telegram delayed-delivery flow was not live-reproduced here.

## Solution Assessment

No current-main solution is complete. The narrow fix likely belongs in src/agents/session-write-lock.ts and adjacent cleanup/acquire tests, but it needs owner review because lock reclaim policy must balance active-owner safety against stale-lock recovery.
