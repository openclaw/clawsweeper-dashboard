## Summary

Keep open. Current main has useful mitigations for store-lock latency, write-lock acquire timeout, cleanup timeout, and restart recovery, but the specific Telegram/tool-run lock cleanup symptom is not proven fixed: current source still allows same-PID/no-starttime lock reclamation on acquire, startup cleanup still removes stale transcript locks without active-turn ownership, and the focused related PR was closed unmerged.

## What This Changes

The issue reports main-agent Telegram replies being delayed or lost after tool execution when stale session-lock cleanup removes a lock during an active run.

## Best Possible Solution

Land an owner-reviewed session-lock fix that protects active transcript owners from stale/dead-pid cleanup and same-PID/no-starttime reclaim while preserving safe dead-PID, recycled-PID, non-OpenClaw-owner, and bounded stale-lock recovery.

## Reproduction Assessment

Yes at source level, but not as a live Telegram run: current code and tests still preserve same-PID/no-starttime reclaim, and startup cleanup can remove stale transcript locks without active-turn ownership. The full delayed-delivery symptom was not live-reproduced here.

## Solution Assessment

No current-main solution is complete. The narrow code area is `src/agents/session-write-lock.ts` plus startup cleanup/recovery tests, but the stale-lock policy needs owner review because it balances active-owner safety against stale-lock recovery.
