## Summary

Keep open. Current main still has source-backed transcript lock cleanup/reclaim behavior tied to the reported response-loss pattern, while the focused related PR was closed unmerged and adjacent session-store work does not prove the Telegram/tool-run symptom is fixed.

## What This Changes

The issue reports main-agent replies being delayed or lost after tool execution when stale session lock cleanup removes or reclaims an active session lock.

## Best Possible Solution

Land an owner-reviewed session write-lock fix that protects active same-PID/no-starttime and active-turn transcript owners while preserving dead-PID, verified recycled-PID, non-OpenClaw-owner, and bounded stale-lock recovery.

## Reproduction Assessment

Yes for the core lock primitive: current source and tests show same-PID/no-starttime transcript locks can still be reclaimed when not tracked in the in-process manager, and startup cleanup still removes dead-pid .jsonl locks without active-turn awareness. The full Telegram delivery delay was not live-reproduced here.

## Solution Assessment

No current-main solution is complete. The narrow fix likely belongs in src/agents/session-write-lock.ts and its cleanup/acquire tests, but it needs owner review because lock reclaim policy must balance active-owner safety against stale-lock recovery.
