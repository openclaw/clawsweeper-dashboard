## Summary

Keep open. Current main includes useful adjacent mitigations for session-store contention and restart recovery, but the transcript write-lock acquisition path still preserves the same-PID/no-starttime reclaim behavior tied to the reported response-loss class, and the targeted fix PR for that lock behavior closed unmerged.

## What This Changes

The issue reports that after tool execution, a Telegram main-agent response can be written or pending but not delivered until a later user message because session lock cleanup or contention interferes with the active turn.

## Best Possible Solution

Land an owner-reviewed session transcript write-lock fix that protects live same-PID/no-starttime owners while preserving dead-PID, non-OpenClaw-owner, and verified recycled-PID recovery, with focused regression coverage.

## Reproduction Assessment

Yes at source level: current main can still reclaim an untracked current-PID transcript lock without starttime during acquisition, and the active test suite preserves that behavior. I did not establish a live Telegram end-to-end reproduction in this read-only review.

## Solution Assessment

No current solution is complete for this issue. The merged session-store and restart-recovery mitigations are useful, but the remaining transcript-lock fix needs owner review because it changes low-level stale-lock recovery semantics and a prior targeted PR closed unmerged.
