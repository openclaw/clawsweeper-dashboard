## Summary

Keep open: the current PR fixes the explicit `mode: "safeguard"` path and has addressed Greptile's type feedback, but it still misses documented provider-backed safeguard configs because the guard receives only the raw mode string.

## What This Changes

The PR disables Pi auto-compaction for raw safeguard mode, threads `compactionMode` through embedded and CLI compaction paths, reapplies the guard after resource reload, and adds focused Pi settings tests.

## Best Possible Solution

Land a narrow repair that shares one provider-aware safeguard resolver for embedded and CLI guard calls, preserves context-engine `ownsCompaction`, covers provider-only configs in tests, and records the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. A source-level reproduction exists with `agents.defaults.compaction.provider` set and `compaction.mode` omitted or `default`: current main loads the safeguard extension, but this PR's guard receives only the raw mode and does not disable Pi auto-compaction.

## Solution Assessment

No. The direction is close, but the guard should use the same provider-aware safeguard state as extension loading rather than threading only `compaction.mode`.
