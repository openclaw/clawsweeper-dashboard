## Summary

Keep open: the PR is a plausible fix for the linked safeguard-compaction bug, and later commits addressed the initial type issue, but the latest patch still uses only raw `agents.defaults.compaction.mode`. Current main also treats `agents.defaults.compaction.provider` as safeguard mode, so provider-backed safeguard would still leave Pi auto-compaction enabled and retain the same conflict. The PR is also missing the required user-facing changelog entry.

## What This Changes

The PR changes Pi auto-compaction guarding to disable Pi compaction for explicit safeguard mode, threads compaction mode through embedded and CLI paths, reapplies the guard after embedded resource reload, and adds focused `pi-settings` tests.

## Best Possible Solution

Land a narrow fix that centralizes the provider-aware “OpenClaw owns compaction” decision and uses it for both explicit safeguard mode and provider-forced safeguard mode, while preserving context-engine `ownsCompaction` behavior. Cover explicit safeguard, provider-backed safeguard with raw default or absent mode, default mode, and post-reload guard reapplication, then add the active-version changelog entry.

## Reproduction Assessment

Yes. The linked bug has concrete logs for explicit safeguard mode, and a source-level reproduction remains for provider-backed safeguard: set `agents.defaults.compaction.provider` without raw `mode: "safeguard"`; current extension resolution enables safeguard while the PR guard still sees only raw default/undefined mode.

## Solution Assessment

No. The PR is close, but passing raw `compaction.mode` is not the narrowest maintainable solution because it duplicates only part of the existing mode-resolution contract; sharing or exporting the resolved safeguard predicate is safer.
