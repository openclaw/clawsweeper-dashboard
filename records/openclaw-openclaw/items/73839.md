## Summary

Keep open: the PR addresses a real safeguard/Pi auto-compaction conflict, but the current diff still keys the guard off raw `compaction.mode` while current docs and code support provider-backed safeguard mode without that raw mode being set. It also still needs the required user-facing changelog entry.

## What This Changes

The PR adds a `compactionMode` parameter to the Pi auto-compaction guard, passes config compaction mode from embedded and CLI paths, reapplies the guard after resource reload, and adds focused `pi-settings` tests.

## Best Possible Solution

Land a narrow provider-aware ownership predicate for Pi auto-compaction: preserve context-engine `ownsCompaction`, treat the effective safeguard state as OpenClaw-owned compaction, reuse that logic in embedded and CLI guard calls, and document the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists with `agents.defaults.compaction.provider` set and `compaction.mode` unset or `default`: current code enables the safeguard extension, while the PR guard sees only the raw mode and would not disable Pi auto-compaction.

## Solution Assessment

No. The PR is close, but the maintainable solution should derive the guard from the same provider-aware safeguard resolution used to load the compaction extension instead of threading only the raw config mode string.
