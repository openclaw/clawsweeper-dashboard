## Summary

Keep open. Current main still lacks response-completion cue support and still uses a stable default web-push tag, so the PR is useful, but the latest head still has two merge blockers: it can play a completion cue for stream-only silent NO_REPLY finals, and it lacks the required user-facing changelog entry.

## What This Changes

The PR adds browser-local response-completion sound settings, a Communication Notifications UI, web-push subscribe/unsubscribe/test controls, service-worker tag/renotify changes, styling, storage updates, and focused UI tests.

## Best Possible Solution

Land the notification controls after preserving silent-output semantics for NO_REPLY stream-only finals, adding the Unreleased changelog entry, and rebasing or reconciling the overlapping Notifications-tab work so the final Control UI surface is single and coherent.

## Reproduction Assessment

Yes for the blocking regression. Current tests already model chatStream NO_REPLY plus a final event with no message as a silent completion; the PR head signals the cue from app-gateway using only payload.message, which is undefined in that path.

## Solution Assessment

No, not yet. The PR direction is maintainable and the prior Greptile findings appear addressed, but the cue trigger should know whether visible assistant output was produced before playing sound, and the user-facing feature needs a changelog entry.
