## Summary

Keep open. Current main fixes part of the misleading dispatcher-timeout wording, but the browser upload-with-trigger path is still stale-ref prone: the file chooser hook arms upload and then clicks the original ref without an internal re-resolution path, and the client upload helper still uses its own fixed 20s request timeout.

## What This Changes

The issue reports flaky browser file-chooser automation where upload arming and trigger clicking can split across requests, leaving stale refs and misleading timeout wording.

## Best Possible Solution

Keep this issue open until the browser plugin makes upload-with-trigger a single robust request: arm the file chooser and activate the trigger with shared timeout/abort context, preserve stale/action-specific errors, and add regression coverage. Keep #51395 separate because it tracks non-standard file-input fallback behavior rather than this stale-ref/timeout path.

## Reproduction Assessment

Yes. The issue provides concrete steps, and current main still has the matching code path: /hooks/file-chooser can accept ref, arm upload, then click that ref afterward.

## Solution Assessment

No. The current main behavior and related partial timeout work are not the best complete fix because they do not make upload+trigger atomic or preserve stale-ref-specific behavior through the trigger activation path.
