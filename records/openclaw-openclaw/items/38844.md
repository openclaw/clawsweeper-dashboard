## Summary

Keep open. Current main has partial related fixes for selector/batch actions and transient fetch-error wording, but the browser upload path still allows the fragile split arming/click flow, keeps a fixed 20s file-chooser client deadline, and has no agreed atomic upload-trigger contract for stale refs.

## What This Changes

The issue requests a robust browser file chooser flow that avoids stale-ref upload clicks and preserves browser-action errors instead of reporting healthy browser control as a service timeout.

## Best Possible Solution

Define and implement a browser upload-with-trigger contract that is atomic, abort-aware, covered by focused tests, and documented so callers avoid the fragile split ref workflow.

## Reproduction Assessment

Yes, source-reproducible. Current main still accepts paths-only upload arming followed by a separate ref click, the hook clicks an optional upload ref without a refresh contract, and the upload client still has a fixed 20,000ms wrapper deadline.

## Solution Assessment

No. Closing as implemented would be premature because selector/batch support and the open timeout PR only reduce symptoms; the remaining fix needs a maintainer-approved atomic upload-trigger and error-preservation contract.
