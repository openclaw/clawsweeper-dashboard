## Summary

Keep open. Current main has partial timeout-wording improvements and related selector/batch support, but the upload-with-trigger path still arms the chooser and clicks the original ref, and the upload client still uses the fixed 20s request timeout that open PR #74352 is trying to address.

## What This Changes

The issue asks for browser upload/file-chooser flows to be robust on dynamic pages by preferring atomic upload-and-trigger behavior and preserving stale-ref or action-timeout errors instead of service-outage wording.

## Best Possible Solution

Keep this open until the browser plugin handles upload-with-trigger as one robust operation with shared timeout/abort handling, stale-ref-specific errors, and regression coverage, while landing or superseding #74352 for the timeout-budget slice.

## Reproduction Assessment

Yes, from source inspection. Current main still has `/hooks/file-chooser` arm upload and then click the original `ref` without re-snapshot, timeout, or abort propagation, which matches the reported stale-ref path.

## Solution Assessment

No. Current main's dispatcher error mapping and selector/batch support are useful partial mitigations, but they do not fix the upload trigger path; #74352 addresses only the timeout-budget part.
