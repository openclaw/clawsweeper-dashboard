## Summary

Keep open. Current main still permits the split upload-arm plus later stale-ref click flow, still gives file-chooser arming a fixed 20s client deadline, and the linked timeout PR addresses only one narrower part of the broader atomic upload-trigger/error-preservation problem.

## What This Changes

The issue asks for browser file chooser uploads to avoid stale-ref split flows and preserve specific action errors instead of reporting healthy browser control paths as service timeouts.

## Best Possible Solution

Keep this as the canonical browser upload reliability issue until maintainers define the atomic upload-trigger contract, land or close the narrower timeout PR, and add regression coverage for preserved stale-ref/action errors.

## Reproduction Assessment

Yes, source-reproducible. Current main still allows paths-only upload arming followed by a separate stale-ref click, and the file-chooser client path still has a fixed 20,000ms wrapper deadline; I did not run a live dynamic-page upload flow.

## Solution Assessment

No, not yet. Selector/batch support and the open timeout PR reduce symptoms, but the maintainable solution needs an agreed atomic upload-trigger path that preserves stale-ref/action errors without turning them into transport outage guidance.
