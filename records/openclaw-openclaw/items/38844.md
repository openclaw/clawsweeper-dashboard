## Summary

Keep open: current main and the latest release fix the misleading local-dispatch timeout wrapper and support upload-plus-trigger in one request, but they still accept and document paths-only upload arming and still use a 20s file-chooser client timeout, so the remaining upload contract is not fully settled.

## What This Changes

The issue asks for Browser upload flows to avoid stale-ref split upload/click failures and to keep action timeouts specific instead of reporting browser-control outages.

## Best Possible Solution

Define the Browser upload contract, preserve paths-only arming only if it is intentional, steer agents toward ref/inputRef atomic upload when appropriate, and validate any timeout-budget change with a live dynamic-page proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source and docs show paths-only upload can be followed by a stale-ref click, but no live current-main trace proves the slow chooser-timeout symptom.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the atomic route is the right boundary for upload-plus-trigger, but maintainers still need to decide the supported paths-only contract and timeout budget before a fix PR is safe.
