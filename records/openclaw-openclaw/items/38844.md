- `impact:session-state`: The issue concerns stale browser element references and action state drift between separate tool calls.

## Summary

Keep open: current main and v2026.6.10 have the atomic upload route and improved local-dispatcher timeout/error handling, but `browserArmFileChooser` still uses a fixed 20s outer timeout and the Browser tool/docs still allow paths-only arming, so the upload reliability contract is not fully settled.

## What This Changes

The issue asks for Browser upload/file chooser flows to prefer atomic upload-plus-trigger behavior and to keep stale/action timeout errors distinct from browser-control transport outages.

## Best Possible Solution

Settle the Browser upload contract explicitly: prefer or require atomic upload with a trigger (`ref`, `inputRef`, or `element`) where appropriate, align the file-chooser transport timeout with the operation budget, and add focused tests plus live dynamic-page proof.

## Reproduction Assessment

No, not as a live reproduction in this review. Source inspection confirms the fixed 20s upload wrapper timeout and still-supported paths-only arming path, but the dynamic-page stale-ref race was not exercised against a real page.

## Solution Assessment

Not applicable to a submitted patch; the issue direction is valid, but the best fix needs maintainer approval for the Browser upload API and timeout tradeoff before implementation.
