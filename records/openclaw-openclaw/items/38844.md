## Summary

Keep open. Current main still preserves the split upload arm plus later click shape and the file-chooser client still has a fixed 20s transport timeout; related work only partially mitigates the stale-ref and timeout-reporting problem.

## What This Changes

Make browser upload/file chooser flows robust against stale trigger refs and keep stale-click/action timeout errors from being reported as browser-control service outages.

## Best Possible Solution

Keep this as the canonical browser upload reliability issue until maintainers define the atomic upload-trigger/stale-ref contract and land or replace the timeout-budget fix with regression coverage and real managed-browser proof.

## Reproduction Assessment

Yes, at source level: current main accepts paths-only upload arming followed by a separate click, and file-chooser arming still has a fixed 20s client timeout. I did not run a live dynamic-page managed-browser upload flow.

## Solution Assessment

No, not yet. Current main mitigates part of the timeout wording and an open PR targets the client timeout budget, but the durable fix still needs a maintainer-defined atomic upload-trigger/stale-ref contract.
