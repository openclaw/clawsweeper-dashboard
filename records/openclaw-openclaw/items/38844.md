## Summary

Keep open. Current main and the latest release still leave the browser upload-with-trigger path vulnerable to stale refs, while only partially improving dispatcher timeout wording.

## What This Changes

The issue requests a robust browser file chooser flow that avoids stale split upload/click failures and does not misreport healthy browser action failures as browser-control outages.

## Best Possible Solution

Keep this open until browser upload-with-trigger has shared timeout/abort handling, stale-ref-specific error preservation, and regression coverage for dynamic pages.

## Reproduction Assessment

Yes, from source inspection. Current main still arms upload and then clicks the original `ref`, which matches the stale-ref path on dynamic pages, although I did not run a live browser upload flow.

## Solution Assessment

No. Current main only partially improves dispatcher timeout wording; the remaining fix needs the upload-trigger path and timeout budget handled together, with #74352 treated as the related timeout slice.
