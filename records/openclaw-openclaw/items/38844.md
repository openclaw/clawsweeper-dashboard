## Summary

Keep open. Current main only partially addresses the misleading timeout wording; the browser upload flow can still be split across upload arming and a later stale-ref click, and the open timeout PR covers a narrower client-deadline symptom rather than the atomic upload-trigger behavior discussed in the issue.

## What This Changes

Report asks for browser upload/file chooser flows to avoid stale-ref split upload+click failures and preserve action-specific timeout errors.

## Best Possible Solution

Keep this open until browser upload with a trigger has a documented, atomic, abort-aware path that preserves stale-ref errors and has focused regression coverage; land the narrower timeout PR separately if accepted.

## Reproduction Assessment

Yes, source-reproducible. Current main still supports split upload arming followed by a later ref click, still uses a fixed 20-second upload-arm client deadline, and only partially improves dispatcher timeout wording; I did not run a live dynamic form page.

## Solution Assessment

Unclear. A timeout-only fix is too narrow; the maintainable solution needs maintainer agreement on the atomic upload-trigger contract and then a focused implementation that keeps stale-ref/action failures specific.
