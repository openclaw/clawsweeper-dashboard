## Label Justifications

- `P2`: One reported symptom is shipped as fixed, but a bounded browser upload reliability gap and timeout uncertainty remain.
- `impact:session-state`: The remaining failure mode allows an upload trigger action to use element state captured before a dynamic page re-render.

## Summary

Current main fixes the misleading local-timeout classification and supports atomic upload-plus-trigger requests, but the agent-facing contract still accepts paths-only arming without telling callers to prefer the atomic form, so the stale-ref workflow remains possible.

## What This Changes

Make browser file-chooser uploads robust against stale refs and keep local action timeout errors distinct from browser-control transport outages.

## Best Possible Solution

Keep paths-only arming only if it is an intentional public mode, clearly direct agents to send `ref` or `inputRef` with uploads by default, and use a live dynamic-page scenario to verify stale-ref specificity and the upload timeout budget before changing transport timing.

## Reproduction Assessment

Partly yes: current source still permits paths-only upload arming followed by a separate stale-ref click, but the misleading generic service-outage wording is fixed and the long-timeout symptom lacks a live current-main reproduction.

## Solution Assessment

Not yet—the atomic route is the right boundary, but the agent contract must make that path preferred without accidentally removing an intentional arming API, and timeout changes need live evidence.
