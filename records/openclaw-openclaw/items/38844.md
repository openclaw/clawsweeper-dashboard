## Summary

Keep open. Current main partially improves the misleading dispatcher-timeout wording, but it does not yet make upload-with-trigger robust: `/hooks/file-chooser` still arms upload and then clicks the original `ref`, without re-resolution, timeout forwarding, or abort-signal propagation. The newly linked open PR #74352 appears to address only the client timeout-budget slice, so the stale-ref/atomic-trigger part still needs maintainer follow-up.

## What This Changes

The issue requests a more robust browser upload/file chooser flow that prefers atomic upload-and-trigger handling and preserves stale-ref or action-specific errors instead of reporting healthy-browser failures as control-service timeouts.

## Best Possible Solution

Keep the browser upload behavior in the browser plugin and make upload-with-trigger a robust single request: propagate timeout and abort context into trigger activation, re-resolve or clearly reject stale refs with action-specific errors, add regression coverage for stale trigger refs and timeout preservation, and coordinate or separate the timeout-budget fix in #74352. Keep #51395 separate because it concerns non-standard file-input fallback behavior.
