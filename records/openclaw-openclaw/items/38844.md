## Summary

Keep this open. Current main and the latest release still only partially address the report: selector/batch work reduces stale-ref exposure, but the documented upload flow can still be split across calls and the upload hook client still uses a fixed 20s transport timeout.

## What This Changes

The issue reports flaky browser file upload/file chooser flows where split upload and stale-ref click steps can surface as misleading browser-control timeout errors.

## Best Possible Solution

Make the supported upload-trigger flow atomic at the model/tool and CLI/docs surface, while preserving file path guardrails and action-specific timeout errors.

## Reproduction Assessment

Yes at source level: current main still documents and supports a split upload-then-click path, and the hook client still has a fixed 20s timeout while the server-side chooser waiter defaults longer. I did not run a live dynamic-page repro in this read-only review.

## Solution Assessment

No for closing: current main only partially addresses the report, because selector/batch support and dispatcher-path wording reduce symptoms but do not make the primary upload-trigger flow atomic by default.
