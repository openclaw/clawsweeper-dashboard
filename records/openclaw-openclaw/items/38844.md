## Summary

Keep open. Current main has useful related improvements, but the browser upload path still exposes split arming plus stale-ref trigger behavior, keeps a fixed 20s upload client deadline, and does not yet define the atomic upload-trigger/error-preservation contract requested here.

## What This Changes

The issue asks for browser file chooser uploads to use a robust atomic trigger flow and to preserve action-specific stale-ref errors instead of surfacing misleading browser-control timeout messaging.

## Best Possible Solution

Keep this open until browser upload-with-trigger has an agreed atomic, abort-aware path that preserves stale-ref/action errors, plus focused coverage and docs that steer callers away from the fragile split flow.

## Reproduction Assessment

Yes, source-reproducible. Current main still permits paths-only upload arming followed by a separate stale-ref click, and the upload client wrapper still has a fixed 20,000ms deadline; I did not run a live dynamic page.

## Solution Assessment

No. Closing as implemented would be premature because selector/batch support and the open timeout PR only reduce symptoms; the narrow maintainable fix needs an agreed atomic upload-trigger and error-preservation contract.
