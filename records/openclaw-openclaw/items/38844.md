## Summary

Keep open. Current main still documents and exposes split upload arming plus stale-ref click paths, and the only open related implementation PR addresses the upload-hook timeout budget rather than the atomic upload-trigger/error-preservation behavior.

## What This Changes

The issue asks for browser upload/file chooser flows to avoid stale-ref split-click failures and keep action failures from being reported as browser-control transport outages.

## Best Possible Solution

Keep this open until browser upload-with-trigger has an agreed atomic, abort-aware path that preserves stale-ref errors and has focused regression coverage; review the narrower timeout PR separately.

## Reproduction Assessment

Yes, source-reproducible. On current main, the browser upload tool can still arm /hooks/file-chooser with only paths and then use a separate act click on a stale ref, while the upload client wrapper still has a 20,000ms deadline; I did not run a live dynamic page.

## Solution Assessment

No. Closing as implemented would be premature because selector/batch support and the open timeout PR only reduce or partially address symptoms; the maintainable fix needs an agreed atomic upload-trigger behavior and error-preservation contract.
