## Summary

Keep open: current main and v2026.6.10 partially address this by preserving local dispatcher timeout context and supporting upload-with-ref in one request, but the Browser upload contract is still unsettled because paths-only arming remains accepted and the agent client still uses a fixed 20s `/hooks/file-chooser` request timeout. The related selector/batch work and prior comments are useful context, but they do not fully close the stale-ref/upload-flow problem.

## What This Changes

The issue reports flaky Browser upload/file chooser automation where upload arming and trigger clicking can split across requests, stale refs can fail, and browser action failures can be surfaced as browser-control timeouts.

## Best Possible Solution

Define the Browser upload contract, keep or retire paths-only arming intentionally, align the hook client timeout with the chosen route behavior if needed, and require live dynamic-page proof plus focused Browser plugin tests before closing.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection still shows the relevant paths-only upload plus separate click shape and the fixed 20s file-chooser client timeout, but a dynamic-page trace is needed to prove the live failure today.

## Solution Assessment

Unclear: the existing upload-with-ref route is the right boundary for an atomic upload-plus-trigger path, but maintainers still need to decide whether paths-only arming remains supported and what timeout budget the Browser API should guarantee.
