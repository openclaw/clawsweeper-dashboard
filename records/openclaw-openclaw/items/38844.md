Keep open. Current main still allows paths-only upload arming followed by a separate stale-ref click, and the remaining timeout work is covered only by an open partial PR rather than a settled atomic upload/error-preservation contract.

## What This Changes

The issue reports flaky browser upload/file chooser automation where paths-only upload arming followed by a stale-ref click can be misreported as a browser control timeout.

## Best Possible Solution

Keep this as the canonical browser upload reliability issue until maintainers define the atomic upload-trigger/stale-ref contract and land or replace the timeout-budget fix with regression coverage and real managed-browser proof.

Yes, at source level: current main accepts paths-only upload arming followed by a separate click, and file-chooser arming still has a fixed 20s client timeout. I did not run a live dynamic-page managed-browser upload flow.

No, not yet. Selector/batch support and the open timeout PR mitigate parts of the problem, but the durable fix needs a maintainer-defined atomic upload-trigger contract and stale-ref/error-preservation coverage.
