## Summary

Keep open: current main and v2026.6.10 fix the misleading local-dispatch timeout wording and support upload-plus-trigger in one request, but paths-only upload arming is still accepted and the file-chooser client still uses a fixed 20s request timeout, so the Browser upload contract is not fully settled.

## What This Changes

The issue reports flaky browser file chooser uploads on dynamic pages when upload arming and trigger clicks are split, plus misleading stale-click timeout errors.

## Best Possible Solution

Define the Browser upload contract, keep or retire paths-only arming intentionally, steer agents toward atomic `ref` or `inputRef` upload when appropriate, and validate any timeout-budget change with live dynamic-page proof.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection still shows that paths-only upload can be followed by a separate stale-ref click, and the file-chooser client still has a 20s transport timeout.

## Solution Assessment

Unclear: the atomic upload-with-ref route is the right existing boundary for upload-plus-trigger, but maintainers still need to decide the supported paths-only contract and timeout budget before a fix PR is safe.
