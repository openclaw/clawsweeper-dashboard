## Summary

Keep open: current main and the latest release fix the misleading local-dispatch timeout wording and support upload-plus-trigger in one request, but the documented paths-only arming mode and fixed 20s file-chooser client timeout remain, so the Browser upload contract is not fully settled.

## What This Changes

The issue requests a more robust browser file chooser upload flow that avoids stale refs and preserves browser-action-specific timeout errors.

## Best Possible Solution

Define the Browser upload contract, keep or retire paths-only arming intentionally, steer agents toward atomic `ref` or `inputRef` upload when appropriate, and validate any timeout-budget change with live dynamic-page proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: current source and docs show that paths-only upload can still be followed by a separate stale-ref click, but I did not establish a live current-main trace for the 20s chooser-timeout symptom.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the atomic route is the right existing boundary for upload-plus-trigger, but maintainers still need to decide the supported paths-only contract and timeout budget before a fix PR is safe.
