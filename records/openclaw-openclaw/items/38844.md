## Summary

Keep open. Current main still permits paths-only upload arming followed by a separate stale-ref click, and the related open timeout PR only addresses the client timeout budget rather than settling the atomic upload-trigger contract.

## What This Changes

The issue asks OpenClaw to make managed-browser file chooser uploads robust against stale trigger refs and to keep browser action timeout errors specific instead of reporting healthy browser control as unreachable.

## Best Possible Solution

Define and test the browser upload contract, then land a focused fix that either makes upload plus trigger atomic and stale-safe or keeps split upload documented with specific timeout/error handling and managed-browser proof.

## Reproduction Assessment

Yes, at source level: current main accepts paths-only upload arming and still permits a separate later click on a snapshot ref that can go stale. I did not run a live dynamic-page managed-browser upload flow.

## Solution Assessment

No, not yet. Current main partially improves dispatcher timeout wording, and the open related PR targets timeout budget, but the durable fix still needs a maintainer-defined atomic upload-trigger or documented split-flow contract.
