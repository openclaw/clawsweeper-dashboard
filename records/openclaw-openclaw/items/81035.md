## Summary

Keep open: the useful outcome-threading approach still leaves one no-send path reporting success, so this PR should not merge until the delivered log is gated on actual visible outcomes.

## What This Changes

The PR threads Mattermost reply-delivery outcomes through final delivery, adds a no-visible-reply diagnostic helper with tests, wires diagnostic logging into the monitor, and updates the changelog.

## Best Possible Solution

Land the outcome-threading diagnostic after gating the success log strictly on text/media outcomes and adding monitor-level coverage for the empty no-send path.

## Reproduction Assessment

No high-confidence live Mattermost reproduction was established in this review. Source inspection shows current main can receive an `"empty"` delivery result, discard it, and still log a delivered final reply; the PR's own regression test/probe supports that no-send path.

## Solution Assessment

No, not yet. Threading the helper outcome is the right narrow direction, but the success log must be gated on `outcome === "text" || outcome === "media"` rather than every non-reasoning, non-violation outcome.
