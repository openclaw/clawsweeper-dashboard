## Summary

Keep open: the patch is on the right path, but it still emits the Mattermost delivered-success log for an `"empty"` no-send outcome when the new evaluator returns no violation. That leaves a false success signal in a no-visible-send path, so the PR needs a narrow fix before merge.

## What This Changes

The PR threads Mattermost reply-delivery outcomes into monitor logging, adds a no-visible-reply diagnostic helper with tests, and updates the changelog.

## Best Possible Solution

Land the outcome-threading diagnostic after tightening the delivered-success log to `"text"` and `"media"` outcomes only, then use https://github.com/openclaw/openclaw/issues/80501 for any broader visible fallback decision.

## Reproduction Assessment

No high-confidence live Mattermost reproduction was established. Source inspection does show current main can receive an `"empty"` result from `deliverTextOrMediaReply`, discard it, and still log a delivered final reply.

## Solution Assessment

No, not quite. Propagating the helper outcome and adding the diagnostic is the right narrow direction, but the success log should be gated on `outcome === "text" || outcome === "media"` and covered by focused regression testing.
