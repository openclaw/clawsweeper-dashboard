## Summary

Keep open: the PR is a useful fix path with sufficient real behavior proof, but current head introduces a blocking no-external-discovery side effect in auth loading and the PR has the protected `maintainer` label.

## What This Changes

The PR persists fresher managed external CLI OAuth credentials into `auth-profiles.json`, keeps bootstrap-only Codex CLI credentials runtime-only, and adds auth-store regression coverage plus a changelog entry.

## Best Possible Solution

Land the managed external CLI sync after preserving no-external-discovery paths with `syncExternalCli: false` and keeping the lock, race, and Codex bootstrap-only guards intact.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and PR proof: a stale persisted `anthropic:claude-cli` OAuth profile plus fresher Claude CLI credentials follows the external CLI bootstrap path; I did not rerun the local proof in this read-only sweep.

## Solution Assessment

Is this the best way to solve the issue? Not yet: persisting managed non-bootstrap CLI refreshes is the right narrow direction, but the current patch must first avoid external CLI reads and writes from no-external-discovery callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Respect external-auth opt-outs before syncing CLI credentials:** `src/agents/auth-profiles/store.ts:520-524`
  - body: `loadAuthProfileStoreForAgent()` now runs persisted external CLI sync for every non-readonly persisted-store load. Callers such as `ensureAuthProfileStoreWithoutExternalProfiles()` and `hasAuthForModelProvider({ discoverExternalCliAuth: false })` are meant to avoid external CLI discovery, but this path can still read Claude CLI credentials and rewrite `auth-profiles.json`; pass `syncExternalCli: false` through those no-external paths or otherwise keep sync limited to external-discovery callers.
  - confidence: 0.88
