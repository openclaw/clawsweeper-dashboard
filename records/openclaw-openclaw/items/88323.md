## Summary

Needs changes before merge: the PR has useful live Discord proof and focused tests for duplicate generated-media replays, but the outbound message-tool path can still mark an artifact delivered after a no-visible-result send and then suppress a legitimate retry.

## What This Changes

The branch adds a process-local generated-media delivery dedupe cache, applies it to final reply delivery, durable message sends, and outbound send actions, and adds regression tests around repeated generated artifact sends.

## Best Possible Solution

Cache generated artifacts only after a visible delivery result is known, then make the route-insensitive idempotency scope an explicit maintainer-approved behavior or tighten the key before merge.

## Reproduction Assessment

Yes. The linked report and PR proof give a high-confidence live Discord reproduction of duplicate top-level generated-image posts from one MEDIA artifact, and source inspection shows multiple delivery owners can replay the same media path.

## Solution Assessment

No, not yet. The overall guard is plausible, but executeSendAction must only record dedupe after visible delivery success; maintainers should also decide whether route-insensitive suppression is the desired permanent scope.
