## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep delivery route scope in generated-media dedupe keys:** `src/infra/outbound/generated-media-dedupe.ts:57-64`
  - body: `resolveGeneratedMediaDeliveryKeys` throws away every route field by writing `accountId`, `channel`, `to`, and `threadId` as `null`. Because this helper is now used by general outbound and durable message paths, the same generated artifact sent to a second target or thread within the TTL will be suppressed even though that is a legitimate delivery.
  - confidence: 0.92
- **[P1] Only cache outbound sends with a visible result:** `src/infra/outbound/outbound-send-service.ts:409-412`
  - body: This unconditional mark runs after `sendCoreMessage`, but best-effort durable sends can return with no `result` when the underlying send failed or was suppressed. That poisons the generated-media cache and can suppress the later final/replay path even though no visible message was delivered.
  - confidence: 0.87
- **[P1] Record dispatcher finals only after delivery succeeds:** `src/auto-reply/reply/dispatch-from-config.ts:2014-2017`
  - body: `sendFinalReply` only enqueues work; the dispatcher still runs `beforeDeliver` and `deliver` asynchronously and catches cancellation or failures afterward. Marking the artifact delivered immediately after queueing means a cancelled or failed queued final can block a retry for the TTL window.
  - confidence: 0.9
