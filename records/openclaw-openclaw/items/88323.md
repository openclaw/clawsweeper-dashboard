## Summary

Keep open. The PR addresses a real generated-media duplicate-delivery bug, but it still has a blocking message-delivery defect: the new cache can record an artifact after a no-visible/suppressed send and then suppress the next real retry; the branch is also conflicting and not maintainer-modifiable.

## What This Changes

The branch adds process-local generated-media delivery dedupe keyed by generated artifact paths across final reply, durable message sends, and message-action sends, with focused regression tests and small compatibility fixes.

## Best Possible Solution

Land a rebased or replacement fix that only records generated artifacts after visible delivery evidence and reconciles artifact-level dedupe with current main's route-aware reply/message-tool dedupe.

## Reproduction Assessment

Yes for the PR defect: source inspection shows executeSendAction can mark an artifact delivered after sendMessage returns a suppressed/no-visible result. The original Discord duplicate-delivery bug also has live proof in the PR body, though I did not run a fresh live Discord reproduction here.

## Solution Assessment

No. The shared dedupe boundary is plausible, but this implementation is not the best fix until cache writes depend on visible delivery evidence and artifact scope is reconciled with current main's route-aware behavior.
