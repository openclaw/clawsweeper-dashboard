Keep this issue open: current main and v2026.6.10 still let stale finalizer handoff env vars survive when no fresh value is present, and the open linked PR is the active fix candidate that should be reviewed before closing the issue.

## What This Changes

Ensure RPC post-core finalizer child processes receive only the current update run's source-config path and compatibility-host version, not stale inherited env values.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/94560; after a fix merges, close this issue as implemented by that PR.

Yes. Source inspection shows the finalizer child env is built from the parent env and stale handoff keys are not cleared when fresh values are absent; I did not execute the repro script in this read-only pass.

## Solution Assessment

Yes. Clearing only the internal finalizer handoff env vars before applying this run's values is the narrow owner-boundary fix, and the linked PR targets exactly that boundary.
