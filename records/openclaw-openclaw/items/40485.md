## Summary

Keep open. The requested Telegram behavior is valid, but this PR is currently incorrect because it changes removed legacy `src/telegram/*` files while current `main` runs the Telegram plugin from `extensions/telegram/src`; the active path still passes the configured reply mode and injects the command message id as the reply target.

## What This Changes

The PR changes legacy Telegram native-command delivery so direct-chat native `/new` and `/reset` use `replyToMode: "off"`, and adds a legacy `/new` regression test.

## Best Possible Solution

Port the narrow override into `extensions/telegram/src/bot-native-commands.ts`, with active-path tests for both `/new` and `/reset`, then either update this branch or replace it with a focused ClawSweeper PR that credits the source contribution.

## Reproduction Assessment

Yes. A code-level reproduction exists on current `main`: configure Telegram `replyToMode` to `first` or `all`, invoke native `/new` or `/reset` in a direct chat, and the active handler still supplies the triggering command `message_id` as `replyToId` while delivery only suppresses it in `off` mode.

## Solution Assessment

No. The behavior change is the right narrow fix, but this PR applies it to removed legacy files; the maintainable solution is to apply the same direct-chat reset override in the active Telegram plugin path and cover both reset commands.
