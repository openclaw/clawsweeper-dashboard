## Summary

Keep open. Current main still permits the stale-token reconnect loop when `AUTH_TOKEN_MISMATCH` happens without an actual cached device-token retry, and the shipped `chat.history` and Slack Socket Mode mitigations do not close that remaining auth-loop bug.

## What This Changes

The issue asks for Control UI stale gateway-token auth failures to stop retrying and for slow `chat.history` handling not to degrade Slack Socket Mode connectivity.

## Best Possible Solution

Patch browser and Node gateway reconnect gating so `AUTH_TOKEN_MISMATCH` reconnects only for an actually queued trusted cached-device-token retry, while preserving startup retry behavior and leaving `chat.history` timeout UX to the linked WebUI work.

## Reproduction Assessment

Yes for the remaining auth-loop portion: with storage cleared, an explicit shared token, and `AUTH_TOKEN_MISMATCH`, current browser test coverage expects a reconnect after 800 ms. The live Slack Socket Mode pong-timeout cascade was not reproduced here.

## Solution Assessment

Yes. The narrow maintainable fix is reconnect gating in the browser and Node gateway clients, preserving the one-shot cached-device-token recovery path and startup `UNAVAILABLE` retry behavior.
