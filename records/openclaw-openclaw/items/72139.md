## Summary

Keep open. Current main still contains a source- and test-backed `AUTH_TOKEN_MISMATCH` reconnect loop for the Control UI when no cached device-token retry is available; `chat.history` and Slack Socket Mode have mitigations, but the terminal stale-auth behavior requested here is not implemented yet.

## What This Changes

The issue asks OpenClaw to stop stale Control UI Gateway auth retries from looping indefinitely and to prevent `chat.history` latency from degrading Slack Socket Mode connectivity.

## Best Possible Solution

Land a focused reconnect-gating fix so browser and Node Gateway clients treat `AUTH_TOKEN_MISMATCH` as terminal unless a trusted cached-device-token retry was actually queued, while keeping the existing bounded `chat.history` and Slack timeout mitigations.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the remaining auth-loop portion: current main has a browser test that clears cached device auth, emits `AUTH_TOKEN_MISMATCH`, advances 800 ms, and expects reconnect; the Slack timing is corroborated by two reports but was not live-reproduced here.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrowest maintainable fix is reconnect gating in the Gateway browser and Node clients, with #44897 reviewed separately for frontend history timeout behavior.

## Repair Work Prompt

Fix the remaining auth retry loop for https://github.com/openclaw/openclaw/issues/72139. On current main, browser `GatewayBrowserClient` and Node `GatewayClient` must treat `AUTH_TOKEN_MISMATCH` as terminal unless a trusted cached-device-token retry was actually queued. Preserve the existing one-shot cached device-token retry on trusted endpoints, startup-unavailable reconnect behavior, non-auth reconnects, and existing device-token mismatch clearing. Add/update focused tests so the browser client does not reconnect on `AUTH_TOKEN_MISMATCH` when local storage has no stored device token, and the Node client pauses reconnect on a trusted endpoint when no stored token exists. Keep existing retry-with-stored-token tests green. Do not change Slack Socket Mode transport defaults and do not duplicate open PR #44897's frontend `chat.history` timeout work. Add a `CHANGELOG.md` fix entry unless maintainers classify the patch as test/internal only.
