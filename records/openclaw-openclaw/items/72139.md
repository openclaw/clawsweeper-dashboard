## Summary

Keep open. Current main still has a source- and test-backed `AUTH_TOKEN_MISMATCH` reconnect path when no cached device-token retry is available; `chat.history` and Slack Socket Mode have mitigations, but the terminal Control UI auth behavior requested here remains unresolved.

## What This Changes

The issue asks for stale Control UI Gateway auth to stop retrying indefinitely and for `chat.history` handling not to starve Slack Socket Mode pongs.

## Best Possible Solution

Land a focused reconnect-gating fix so browser and Node Gateway clients treat `AUTH_TOKEN_MISMATCH` as terminal unless a trusted cached-device-token retry was actually queued, while preserving the existing `chat.history` bounds and Slack timeout tuning.

## Reproduction Assessment

Yes for the remaining auth-loop portion: current main has a browser test that clears cached device auth, emits `AUTH_TOKEN_MISMATCH`, advances 800ms, and expects reconnect. The Slack pong timeout portion is corroborated by reporter logs and a macOS confirmation comment but was not live-reproduced here.

## Solution Assessment

Yes, the narrowest maintainable fix is reconnect gating in the Gateway browser and Node clients with focused tests. A frontend `chat.history` timeout or larger Slack ping timeout is useful mitigation, not the root fix for stale Control UI auth loops.
