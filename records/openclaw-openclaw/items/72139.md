## Summary

Keep open. Current main still has the reported root behavior: the Control UI can continue reconnecting after `AUTH_TOKEN_MISMATCH` when no cached device-token retry was actually attempted, and `chat.history` still falls back to a synchronous full-transcript read for tree/DAG transcripts. Slack Socket Mode timeout tuning and open PR #44897 are useful mitigations, but they do not remove the auth retry loop or the blocking transcript path.

## What This Changes

The issue asks OpenClaw to stop stale Control UI token retry loops and make `chat.history` nonblocking enough that UI auth failures cannot degrade Slack Socket Mode connectivity.

## Best Possible Solution

Fix both root paths in core: make `AUTH_TOKEN_MISMATCH` terminal when no trusted cached-device-token retry is available or after that retry is exhausted, and replace the tree/DAG transcript full-read fallback with a bounded active-branch history path or a yielding/off-thread read that preserves branch semantics. Keep Slack timeout tuning and PR #44897 as symptom-level mitigations, not the closure condition.

## Reproduction Assessment

Yes. The auth loop has a high-confidence reproduction path in the existing browser test that expects reconnect after `AUTH_TOKEN_MISMATCH` with no stored token, and source inspection shows the same trusted-endpoint gap in the Node client. The history starvation path is source-backed by the synchronous full-read fallback for tree/DAG transcripts on the `chat.history` RPC path.

## Solution Assessment

Yes. The best fix is a narrow core repair in the Control UI browser client, shared Node GatewayClient auth pause logic, and Gateway transcript reader; increasing Slack timeouts or adding only a frontend timeout is safer as mitigation but not sufficient as the product fix.
