## Summary

Keep open. Current main has a partial main-only fix for the transcript-read side, but it still preserves the stale-token reconnect path for Control UI when no cached device-token retry is available, which is the first reported root cause and is covered by a current test that expects the loop.

## What This Changes

The issue asks for Control UI stale-token auth failures to stop and prompt for operator action, and for `chat.history` to avoid synchronous transcript work that can starve Slack Socket Mode pongs.

## Best Possible Solution

Stop automatic reconnects for `AUTH_TOKEN_MISMATCH` whenever no trusted cached-device-token retry is actually available, and after the one allowed retry is exhausted. Preserve the current bounded `chat.history` tail-reader work, and close any remaining transcript starvation gap with a branch-aware bounded reader rather than a frontend timeout alone.

## Reproduction Assessment

Yes. Current source and `ui/src/ui/gateway.node.test.ts` reproduce the remaining auth loop by clearing stored device tokens, emitting `AUTH_TOKEN_MISMATCH`, and expecting a reconnect; current source also shows the generic gateway client has the same trusted no-stored-token pause gap.

## Solution Assessment

Yes for the keep-open path. The narrow maintainable fix is in the Control UI/browser gateway client plus generic gateway client reconnect gate, with transcript-reader preservation or follow-up bounded branch handling; PR #44897 is a symptom timeout, not the root fix.
