## Summary

Keep open. Current main still has a source-backed `AUTH_TOKEN_MISMATCH` reconnect loop when no cached device-token retry is available, and `chat.history` still falls back to synchronous full-transcript reads for tree/DAG transcripts. Slack Socket Mode timeout tuning and the related WebUI timeout PR are useful mitigations, but they do not close the reported root behavior.

## What This Changes

The issue asks OpenClaw to stop stale Control UI auth retry loops and prevent `chat.history` from blocking the Gateway event loop enough to disconnect Slack Socket Mode.

## Best Possible Solution

Fix both root paths in core: make `AUTH_TOKEN_MISMATCH` terminal when no trusted cached-device-token retry is actually available or after that retry is exhausted, and replace the tree/DAG transcript full-read fallback with a bounded or yielding active-branch history path that preserves branch semantics. Keep Slack timeout tuning and PR #44897 as symptom-level follow-ups, not the closure condition for this issue.

## Reproduction Assessment

Yes. The auth loop is reproduced by the existing browser test that expects a reconnect after `AUTH_TOKEN_MISMATCH` with no cached token, and source inspection shows the same trusted-endpoint gap in the Node GatewayClient. The history starvation path is source-backed for tree/DAG transcripts because the recent reader falls back to synchronous full-file `readSessionMessages()`.

## Solution Assessment

Yes. The narrow maintainable fix is in the Control UI/browser client, shared Node GatewayClient auth retry gate, and Gateway session transcript reader; increasing Slack Socket Mode timeouts or adding a frontend `chat.history` timeout only masks the availability problem.
