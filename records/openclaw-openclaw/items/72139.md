## Summary

Keep open. Current main still matches the two reported root paths: Control UI can continue reconnecting after `AUTH_TOKEN_MISMATCH` when no cached device-token retry was attempted, and `chat.history` still loads the whole transcript synchronously before applying display limits. The related WebUI timeout PR and Slack Socket Mode timeout tuning are mitigations, not fixes for these root causes.

## What This Changes

The issue requests that stale Control UI gateway authentication stop reconnecting indefinitely and that `chat.history` avoid blocking channel WebSocket responsiveness.

## Best Possible Solution

Implement the two root fixes in core: make `AUTH_TOKEN_MISMATCH` terminal when there is no trusted cached device-token retry available or once that retry is exhausted, and add a bounded recent transcript loading path for `chat.history` so large transcripts are not synchronously read in full before display limits are applied.

## Reproduction Assessment

Yes. Current source and an existing UI test reproduce the no-cached-token reconnect path, and source inspection shows `chat.history` still performs synchronous full-file transcript work before limits; the exact Slack pong timing should be rechecked after a patch.

## Solution Assessment

Yes. Fixing the Control UI auth gate and Gateway history loading path is the narrow maintainable solution; PR #44897's frontend timeout and Slack ping/pong tuning are symptom mitigations, not substitutes for the root fix.
