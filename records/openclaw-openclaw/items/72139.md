## Summary

Keep open. Current main still allows `AUTH_TOKEN_MISMATCH` reconnects when no cached device-token retry is available; the `chat.history` and Slack Socket Mode mitigations do not resolve the central stale-token retry loop.

## What This Changes

The issue asks OpenClaw to stop stale Control UI auth reconnect loops and prevent `chat.history`/auth failures from degrading Slack Socket Mode connectivity.

## Best Possible Solution

Patch browser and Node gateway reconnect gating so `AUTH_TOKEN_MISMATCH` only reconnects for an actually queued trusted cached-device-token retry, while leaving WebUI timeout UX to the linked WebUI PR.

## Reproduction Assessment

Yes for the remaining auth-loop portion from source: with storage cleared, an explicit shared token, and `AUTH_TOKEN_MISMATCH`, the current browser test expects a reconnect. The live Slack Socket Mode pong-timeout path was not reproduced here.

## Solution Assessment

Yes. The narrowest maintainable fix is reconnect gating in the browser and Node gateway clients while preserving the one-shot cached-device-token recovery path and startup retry behavior.
