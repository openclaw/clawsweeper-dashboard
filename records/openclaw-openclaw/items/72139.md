## Summary

Keep open. Current main still has the stale-token reconnect loop when `AUTH_TOKEN_MISMATCH` occurs and no cached device-token retry is actually queued; the existing `chat.history` and Slack Socket Mode mitigations do not close that central auth-loop behavior, and the open WebUI PR only covers part of the history-timeout UX.

## What This Changes

The issue asks OpenClaw to stop stale Control UI gateway auth retry loops and prevent slow `chat.history` handling from starving Slack Socket Mode pongs.

## Best Possible Solution

Make `AUTH_TOKEN_MISMATCH` terminal unless a trusted cached-device-token retry is actually queued, preserve the documented one-shot fallback, and keep the `chat.history` timeout/cancellation work with https://github.com/openclaw/openclaw/pull/44897 or a successor.

## Reproduction Assessment

Yes for the remaining auth-loop portion from source: current browser coverage clears storage, receives `AUTH_TOKEN_MISMATCH`, advances the backoff, and expects a reconnect. I did not establish a live Slack Socket Mode pong-timeout reproduction.

## Solution Assessment

Yes. The narrowest maintainable fix is reconnect gating in the browser and Node gateway clients while preserving the bounded cached-device-token retry and leaving the related WebUI `chat.history` timeout behavior to the separate open PR.
