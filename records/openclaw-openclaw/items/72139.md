## Summary

Keep open. Current main has meaningful mitigations for the `chat.history` and Slack Socket Mode timeout side of the report, but the stale Control UI token path still auto-reconnects after `AUTH_TOKEN_MISMATCH` when no cached device-token retry is actually available, so the requested terminal auth behavior remains unfixed.

## What This Changes

The issue asks OpenClaw to stop stale Control UI auth retry loops and keep `chat.history` from starving channel WebSocket connectivity.

## Best Possible Solution

Land a focused Gateway/UI repair that treats `AUTH_TOKEN_MISMATCH` as terminal unless a trusted cached-device-token retry is actually queued, and keep the existing bounded `chat.history` and Slack timeout changes as mitigations.

## Reproduction Assessment

Yes for the remaining auth-loop portion: current main has a browser test that clears cached device auth, emits `AUTH_TOKEN_MISMATCH`, and expects reconnect. The exact Slack disconnect timing is supported by issue logs and a second user report, but was not live-reproduced in this read-only pass.

## Solution Assessment

Yes, the narrowest maintainable fix is reconnect-gating in the browser and Node Gateway clients, with tests proving no reconnect when no cached-device-token retry is available. Increasing Slack timeouts or adding a frontend `chat.history` timeout is useful mitigation, not the root fix.

## Repair Work Prompt

Fix the remaining stale-token reconnect loop for https://github.com/openclaw/openclaw/issues/72139. On current main, `AUTH_TOKEN_MISMATCH` is only terminal after `deviceTokenRetryBudgetUsed` is true, so a Control UI or Node Gateway client with an explicit shared token but no cached device token keeps reconnecting. Update the browser `GatewayBrowserClient` and Node `GatewayClient` so `AUTH_TOKEN_MISMATCH` stops automatic reconnects unless a trusted cached-device-token retry is actually available and has been queued. Preserve the existing one-shot trusted cached-device-token recovery path and startup-unavailable reconnect behavior. Update tests that currently expect reconnect without a cached token, add Node coverage for the no-stored-token trusted endpoint case, and keep docs aligned only if behavior wording needs refinement. Do not change Slack provider behavior or broaden `chat.history` beyond necessary test assertions; current bounded async history reads are already a mitigation. Add a single-line `CHANGELOG.md` fix entry because this is user-facing availability/auth behavior.
