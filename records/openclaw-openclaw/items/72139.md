## Summary

Keep open. Current `origin/main` still has the stale-token reconnect loop for Control UI and the Node gateway client when `AUTH_TOKEN_MISMATCH` occurs without an actual cached-device-token retry; `chat.history` and Slack Socket Mode mitigations exist, but the central auth-loop behavior remains unresolved.

## What This Changes

The issue asks OpenClaw to stop stale Control UI gateway-auth reconnect loops and prevent slow `chat.history` handling from starving Slack Socket Mode pongs.

## Best Possible Solution

Make `AUTH_TOKEN_MISMATCH` terminal unless a trusted cached-device-token retry is actually queued, preserve the one-shot cached-token recovery path, and handle any remaining `chat.history` timeout UX through the linked WebUI work.

## Reproduction Assessment

Yes for the remaining auth-loop portion from source: with storage cleared, an explicit shared token, and `AUTH_TOKEN_MISMATCH`, the current browser test expects a reconnect. I did not establish a live Slack Socket Mode pong-timeout reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is reconnect gating in the browser and Node gateway clients while preserving the bounded cached-device-token retry and leaving the related WebUI `chat.history` timeout behavior to the open linked PR.
