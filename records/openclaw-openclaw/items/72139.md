## Summary

Keep open. Current main still source-encodes the stale Control UI/Gateway `AUTH_TOKEN_MISMATCH` reconnect loop when no cached device-token retry is available; `chat.history` and Slack reconnect mitigations exist, but the terminal auth behavior requested here remains unresolved.

## What This Changes

The issue asks for stale Control UI/Gateway auth failures to stop automatic reconnects, for `chat.history` not to starve the event loop, and for UI auth churn not to degrade Slack Socket Mode.

## Best Possible Solution

Make `AUTH_TOKEN_MISMATCH` terminal unless a trusted cached-device-token retry is actually queued, preserve the one-shot cached-token recovery path, and leave WebUI history-timeout UX to the related PR if it lands.

## Reproduction Assessment

Yes for the remaining auth-loop portion: current source and tests show a reconnect after `AUTH_TOKEN_MISMATCH` when no stored device-token retry is queued. The live Slack Socket Mode timing was not reproduced in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is reconnect gating in the browser and Node Gateway clients; `chat.history` response bounding and Slack ping/reconnect tuning should remain separate mitigations or validation context.
