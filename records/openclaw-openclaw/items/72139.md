## Summary

Keep open. Current main still treats `AUTH_TOKEN_MISMATCH` as reconnectable on trusted endpoints when no cached device-token retry was actually queued, so the stale Control UI token loop remains source-backed even though `chat.history` and Slack timeout mitigations have improved the broader symptom area.

## What This Changes

The issue asks OpenClaw to stop stale Control UI auth retry loops and keep `chat.history` from starving Slack Socket Mode connectivity.

## Best Possible Solution

Land a focused Gateway/UI reconnect-gating repair that makes `AUTH_TOKEN_MISMATCH` terminal unless a trusted cached-device-token retry is actually queued, then split any remaining Slack Socket Mode behavior into a narrower follow-up if it still reproduces.

## Reproduction Assessment

Yes for the remaining auth-loop portion: current main has a browser test that clears cached device auth, emits `AUTH_TOKEN_MISMATCH`, and expects reconnect. The Slack disconnect timing is supported by issue logs and a second user report, but was not live-reproduced here.

## Solution Assessment

Yes for the narrow repair direction. Reconnect gating in the browser and Node Gateway clients is the maintainable root fix for the proven loop; Slack timeout and frontend `chat.history` timeout changes are useful mitigations but not substitutes.

## Repair Work Prompt

Fix the remaining stale-token reconnect loop for https://github.com/openclaw/openclaw/issues/72139. On current main, `AUTH_TOKEN_MISMATCH` is only terminal after `deviceTokenRetryBudgetUsed` is true, so a Control UI or Node Gateway client with an explicit shared token but no cached device token keeps reconnecting. Update the browser `GatewayBrowserClient` and Node `GatewayClient` so `AUTH_TOKEN_MISMATCH` stops automatic reconnects unless a trusted cached-device-token retry is actually available and has been queued. Preserve the existing one-shot trusted cached-device-token recovery path and startup-unavailable reconnect behavior. Update tests that currently expect reconnect without a cached token, add Node coverage for the no-stored-token trusted endpoint case, and keep docs aligned only if behavior wording needs refinement. Do not change Slack provider behavior or broaden `chat.history` in this PR; current bounded async history reads are already a mitigation. Add a single-line `CHANGELOG.md` fix entry because this is user-facing availability/auth behavior.
