## Summary

Keep open. Current main has mitigations for Slack Socket Mode timeout tuning and async bounded `chat.history` reads, but it still preserves the stale-token reconnect path when `AUTH_TOKEN_MISMATCH` occurs without an available cached-device-token retry.

## What This Changes

The issue reports stale Control UI gateway tokens causing an infinite WebSocket retry loop and `chat.history` event-loop stalls that trigger Slack Socket Mode disconnects.

## Best Possible Solution

Make `AUTH_TOKEN_MISMATCH` terminal whenever no trusted cached-device-token retry is actually available or after the bounded retry fails, while preserving the current async bounded `chat.history` reader and Slack timeout tuning as mitigations.

## Reproduction Assessment

Yes for the auth-loop portion: current main has a browser test that clears localStorage, emits `AUTH_TOKEN_MISMATCH`, advances timers, and expects a reconnect. The precise Slack pong-timeout linkage is not live-reproduced here, but the issue and confirmation comment include repeated timing evidence.

## Solution Assessment

No, current main is not yet the best solution because the stale-token reconnect loop remains. The narrow maintainable fix is to gate reconnect on actual retry availability and add browser/Node tests for the no-cached-token mismatch case.
