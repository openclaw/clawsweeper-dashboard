## Summary

Keep open. Current main still has the stale gateway-token reconnect loop for Control UI and the Node gateway client when `AUTH_TOKEN_MISMATCH` occurs without an actual cached-device-token retry; `chat.history` and Slack Socket Mode mitigations exist, but the central auth-loop behavior remains unresolved.

## What This Changes

Stop stale Control UI gateway credentials from retrying indefinitely and prevent `chat.history`/Slack Socket Mode interactions from delaying channel responses.

## Best Possible Solution

Make `AUTH_TOKEN_MISMATCH` terminal unless a trusted cached-device-token retry is actually queued, preserve the one-shot recovery path, and keep the existing user-facing stale-token guidance.

## Reproduction Assessment

Yes for the remaining auth-loop portion from source: with storage cleared, an explicit shared token, and `AUTH_TOKEN_MISMATCH`, the current browser test expects a reconnect. The live Slack pong-timeout timing was not reproduced here.

## Solution Assessment

Yes. The narrowest maintainable fix is reconnect gating in the browser and Node gateway clients, while preserving the bounded cached-device-token retry and leaving `chat.history` timeout UX to the related WebUI work.
