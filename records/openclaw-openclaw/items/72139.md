## Summary

Keep open. Current main has partial mitigations, including Slack Socket Mode timeout tuning and bounded append-only transcript tail reads, but the source and tests still preserve the stale-token Control UI reconnect loop when no cached device-token retry is available. The related WebUI timeout PR is useful but only masks one symptom of delayed `chat.history` responses, not the auth-loop or remaining synchronous transcript-reader root paths.

## What This Changes

The issue asks OpenClaw to stop retrying Control UI WebSocket auth indefinitely on stale gateway tokens and to make `chat.history` non-blocking enough that UI history loads cannot starve Slack Socket Mode pong responses.

## Best Possible Solution

Fix the root paths in core: make token mismatch terminal when no trusted cached-device-token retry is actually possible or after that retry is exhausted, and make active-branch history reads bounded or yielding for rewritten transcripts. Keep Slack timeout tuning and the linked WebUI timeout PR as mitigations, not the closure condition.

## Reproduction Assessment

Yes for the auth-loop portion: the current browser test reproduces it by clearing localStorage, returning `AUTH_TOKEN_MISMATCH`, and expecting reconnect. The full Slack disconnect timing was not live-rerun here, but the reporter supplied repeated timing evidence and current source still has plausible event-loop starvation paths.

## Solution Assessment

Yes, the best fix is a narrow Gateway/UI repair rather than only increasing Slack timeouts or adding a frontend `chat.history` timeout. The latter can improve UX but does not prevent stale-token reconnect churn or synchronous Gateway work from affecting channel sockets.
