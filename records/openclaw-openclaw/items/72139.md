## Summary

Keep open. Current main still has a source- and test-backed Control UI `AUTH_TOKEN_MISMATCH` reconnect loop when no cached device-token retry is available; the `chat.history` and Slack Socket Mode sides have mitigations, but the requested terminal auth behavior remains unresolved.

## What This Changes

The issue asks for stale Control UI gateway auth to stop retrying and for `chat.history` handling not to starve Slack Socket Mode connectivity.

## Best Possible Solution

Land a focused reconnect-gating fix so browser and Node Gateway clients treat `AUTH_TOKEN_MISMATCH` as terminal unless a trusted cached-device-token retry was actually queued, while keeping the existing bounded `chat.history` and Slack timeout mitigations.

## Reproduction Assessment

Yes for the remaining auth-loop portion: current main has a browser test that clears cached device auth, emits `AUTH_TOKEN_MISMATCH`, advances 800 ms, and expects reconnect. The Slack timing is corroborated by two reports but was not live-reproduced here.

## Solution Assessment

Yes, the narrowest maintainable fix is reconnect gating in the Gateway browser and Node clients with focused tests. A frontend `chat.history` timeout or larger Slack ping timeout is useful mitigation, not the root fix for stale Control UI auth loops.
