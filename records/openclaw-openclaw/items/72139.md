## Summary

Keep open. Current main has useful mitigations for Slack Socket Mode timeout tuning and bounded append-only transcript tail reads, and the Control UI now surfaces token mismatch text, but source and tests still preserve the stale-token reconnect loop when no cached device-token retry is available. The remaining `chat.history` root path can still fall back to synchronous full active-branch reads for tree/DAG transcripts, so the reported availability problem is not fully fixed.

## What This Changes

The issue asks OpenClaw to stop retrying stale Control UI gateway auth indefinitely and make `chat.history` non-blocking enough that UI auth failures cannot starve Slack Socket Mode pongs.

## Best Possible Solution

Land a narrow core repair that makes `AUTH_TOKEN_MISMATCH` terminal whenever a trusted cached-device-token retry is not actually available or has already failed, updates browser and Node client tests to assert no reconnect loop, and replaces the remaining `chat.history` full active-branch fallback with a bounded or yielding branch-aware reader. Keep Slack timeout tuning and the WebUI timeout PR as mitigations, not the closure condition.

## Reproduction Assessment

Yes for the auth-loop portion: current main has a browser test that clears localStorage, emits `AUTH_TOKEN_MISMATCH`, and expects reconnect. The exact Slack pong-timeout timing is not reproduced here, but the issue includes repeated timing evidence and current main still has plausible event-loop starvation paths.

## Solution Assessment

Yes, the best fix is a focused Gateway/UI and transcript-reader repair rather than only increasing Slack Socket Mode timeouts or adding a frontend `chat.history` timeout. Those mitigations improve symptoms but do not prevent stale-token reconnect churn or synchronous Gateway work from affecting channel sockets.
