## Summary

Keep open. Current main still has a source- and test-backed stale-auth reconnect loop for trusted Control UI/Gateway clients when `AUTH_TOKEN_MISMATCH` occurs and no cached device-token retry was actually queued; `chat.history` and Slack Socket Mode now have mitigations, but the terminal auth behavior requested here is not implemented.

## What This Changes

Control UI with a stale gateway auth token should stop retrying and prompt for fresh auth, and `chat.history` should avoid starving Slack Socket Mode connectivity.

## Best Possible Solution

Make `AUTH_TOKEN_MISMATCH` terminal unless an actual cached device-token retry was queued, while keeping the existing bounded `chat.history` and Slack ping-timeout mitigations and tracking any separate Slack inbound-loss bug independently.

## Reproduction Assessment

Yes for the remaining auth-loop portion: current tests and source reproduce a stale shared-token mismatch with no cached device token and then assert/perform reconnect. The Slack disconnect timing is corroborated by two user reports but was not live-reproduced in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is reconnect gating in the browser and Node Gateway clients, with tests that flip the current no-stored-token mismatch expectation. Adding only frontend `chat.history` timeouts or larger Slack ping windows would be mitigation, not the root stale-auth fix.
