## Summary

Keep open: current main still documents agent-scoped tool deny policy, but the claude-cli bundle-MCP config path still writes a merged MCP server config without applying the effective agent tool policy. The prior embedded-Pi MCP filtering fixes do not cover this external CLI config-file path.

## What This Changes

The issue reports that per-agent `tools.deny` does not block MCP tools exposed to agents through the `claude-cli` bundle-MCP backend.

## Best Possible Solution

Apply the effective agent/provider tool policy before generated CLI MCP config is written or injected, preserving the OpenClaw loopback entry and adding claude-cli regression coverage plus live proof.

## Reproduction Assessment

Yes. The reporter gives a concrete 2026.5.7 claude-cli reproduction, and current source still shows the generated MCP config is built from existing/configured servers without an agent-policy filter; I did not live-run the external CRM MCP setup.

## Solution Assessment

Yes. Enforcing policy at the generated MCP config boundary is the narrowest maintainable fix because claude-cli reads callable MCP tools directly from that file; a docs-only clarification would leave the documented permission boundary unenforced.
