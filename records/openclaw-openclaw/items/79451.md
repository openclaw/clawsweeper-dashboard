## Summary

Keep open: current main still appears to leave the documented per-agent MCP deny policy unenforced for CLI backend MCP config generation, and the linked closing PR is open rather than merged.

## What This Changes

The issue requests that `agents.list[].tools.deny` block denied MCP servers/tools for the `claude-cli` backend generated MCP config.

## Best Possible Solution

Land a reviewed fix that applies effective agent/provider tool policy before writing CLI backend MCP configs, with regression coverage for deny-wins and exact allow semantics.

## Reproduction Assessment

Yes, source-reproducible: the docs define per-agent deny policy and current main's CLI bundle-MCP path merges and writes configured MCP servers without passing agent-scoped policy. I did not run the reporter's external CRM MCP setup.

## Solution Assessment

Yes: enforcing policy before writing the generated MCP config is the narrowest maintainable boundary because `claude-cli` reads tools from that file. The open linked PR is the right fix path to review and land before closing this issue.
