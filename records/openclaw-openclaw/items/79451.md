## Summary

Keep open. Current main still generates CLI backend MCP config without applying the documented deny-wins effective tool policy, and the only direct fix attempt is closed unmerged and conflicting, so there is no safe implemented or superseding path to close against.

## What This Changes

The issue reports that per-agent `tools.deny` does not block MCP servers exposed through the `claude-cli` backend, allowing denied MCP tools to remain callable.

## Best Possible Solution

Land a maintainer-approved security fix that applies deny-wins effective tool policy before generated CLI MCP config reaches backend clients, with exact allow/deny regression coverage and real backend proof for the chosen CLI modes.

## Reproduction Assessment

Yes, source-level: current main passes config into CLI MCP generation but never passes agent id or effective policy, so denied MCP servers remain in the generated config. I did not run a live Claude CLI turn in this read-only review.

## Solution Assessment

Yes on direction, but not yet in current code. Filtering the generated CLI MCP config before backend spawn is the right boundary; the exact whole-server versus exact-tool semantics need maintainer/security approval.
