## Summary

Keep open: current main still writes the generated CLI-backend MCP config without applying the effective agent tool policy, so the reported `claude-cli` permission bypass remains valid. The linked fixing PR is open and uses closing syntax, so this issue should stay open until that PR, or a replacement with the same boundary fix, lands.

## What This Changes

The issue asks OpenClaw to enforce per-agent `tools.deny` against MCP tools exposed to the `claude-cli` backend through generated MCP config files.

## Best Possible Solution

Land a generated-MCP-config enforcement fix that applies the same effective global/provider/profile/agent policy before CLI backends see MCP servers, while preserving deny-wins and not widening exact MCP tool allowlists; review https://github.com/openclaw/openclaw/pull/79480 or a narrow replacement to get there.

## Reproduction Assessment

Yes, source-reproducible: the reporter gives a concrete released-build `claude-cli` path, and current main still emits merged MCP config without the resolved agent tool policy. I did not live-run the external CRM MCP server.

## Solution Assessment

Yes in direction: enforcing policy before writing the CLI backend MCP config is the narrowest maintainable boundary because `claude-cli` reads tools from that file directly. The implementation still needs to avoid treating exact MCP tool allow entries as whole-server authorization.
