## Summary

Keep open: source inspection on current main still shows the documented per-agent deny policy is not applied before the claude-cli MCP config is generated, and the linked open fix PR should remain the implementation path until it merges.

## What This Changes

The issue asks for `agents.list[].tools.deny` to block denied MCP tools when a `claude-cli` backend reads the generated MCP config.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/79480 or an equivalent fix lands, with policy filtering applied before the generated Claude MCP config is written.

## Reproduction Assessment

Yes, source-reproducible: the reporter gives a concrete claude-cli path verified on 2026.5.7, and current main still writes merged MCP config without agent-scoped deny policy input. I did not run the external CRM setup live.

## Solution Assessment

Yes: applying the effective tool policy at the generated MCP config boundary is the narrowest maintainable solution because claude-cli reads those tools directly from that file. A docs-only clarification would leave the documented permission boundary unenforced.
