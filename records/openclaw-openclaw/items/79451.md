## Summary

Keep open: current main still builds CLI backend MCP config without effective per-agent tool-policy filtering, and the open closing PR is still under review with a security-sensitive matcher concern.

## What This Changes

`agents.list[].tools.deny` should prevent claude-cli backend agents from receiving or calling denied MCP server tools.

## Best Possible Solution

Land the linked fix PR or a successor that filters generated CLI backend MCP configs with the same effective policy semantics as embedded bundled tools, without widening exact tool allowlists.

## Reproduction Assessment

Yes. The issue gives a concrete released-version claude-cli reproduction, and current main still passes no agent/provider tool policy into the generated MCP config path before emitting merged servers.

## Solution Assessment

Yes, applying effective policy before emitting the CLI MCP config is the right boundary because external CLI backends consume that config directly. The current linked PR needs its exact-tool allowlist widening fixed before it is the best mergeable solution.
