## Summary

Keep open. Current `main` still merges and writes CLI backend MCP config without applying the documented effective tool policy, and the only direct fix attempt is closed unmerged and conflicting, so it is not a safe supersession target.

## What This Changes

The issue asks OpenClaw to enforce per-agent `tools.deny` for MCP servers exposed to the `claude-cli` backend.

## Best Possible Solution

Land a maintainer-approved security fix that applies the effective tool policy before generated CLI MCP config reaches the backend, with deny-wins regression coverage and real backend proof for the selected CLI modes.

## Reproduction Assessment

Yes, from source: a focused harness with a `claude-cli`-style backend, two MCP servers, and `agents.list[].tools.deny` would still flow both servers into the generated MCP config because no policy is passed to `prepareCliBundleMcpConfig`. I did not run a live Claude CLI turn in this read-only review.

## Solution Assessment

Yes on direction, but the exact enforcement shape needs maintainer/security approval. Filtering the generated MCP config before backend spawn is the right boundary; docs-only clarification would leave a silent permission bypass.
