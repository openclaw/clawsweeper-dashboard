## Summary

Keep open. Current main still writes the generated claude-cli MCP config from enabled/configured servers without applying the per-agent deny policy, and the only direct fix PR is closed unmerged after review found a permission-widening issue.

## What This Changes

The issue reports that per-agent tools.deny rules do not remove denied MCP servers from claude-cli backend MCP config generation, allowing agents to call blocked MCP tools.

## Best Possible Solution

Materialize a per-agent effective MCP config before invoking CLI backends, preserve deny-wins semantics, fail closed when the backend cannot enforce a requested restriction, and cover profile allow plus exact/wildcard deny cases.

## Reproduction Assessment

Yes, source-level. Current main still builds the generated CLI MCP config from enabled/configured servers without applying the agent's effective tools.deny policy, and the separate runtime toolsAllow guard does not cover config-level deny entries.

## Solution Assessment

No, a docs clarification alone is not the best fix. The maintainable direction is an owner-reviewed pre-handoff filter or fail-closed path that does not widen exact MCP allow entries.
