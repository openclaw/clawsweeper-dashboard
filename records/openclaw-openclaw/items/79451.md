## Summary

Keep open. Current main still builds and hands CLI backend MCP config to claude-cli without applying the documented per-agent deny-wins tool policy, and the only direct fix PR is closed unmerged, so there is no implemented or safely superseding path to close against.

## What This Changes

The issue asks OpenClaw to enforce `agents.list[].tools.deny` for MCP tools before claude-cli receives generated MCP configuration.

## Best Possible Solution

Land a maintainer-approved security fix that materializes an agent-effective, deny-wins MCP config before CLI backend handoff, with explicit semantics for whole-server versus per-tool policy and backend proof.

## Reproduction Assessment

Yes, source-level: current main passes config into CLI MCP generation but never passes an agent id or effective deny policy, so denied MCP servers can remain in the generated config. I did not run a live claude-cli turn in this read-only review.

## Solution Assessment

Yes on direction: filtering the generated CLI MCP config before backend spawn is the right boundary. The exact fail-closed, audit, and whole-server versus exact-tool semantics still need maintainer/security approval.
