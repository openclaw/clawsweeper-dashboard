## Summary

Keep open: current main and the latest release still generate CLI backend MCP config without applying the effective per-agent tool policy, so denied MCP servers can remain exposed to `claude-cli`.

## What This Changes

The issue asks OpenClaw to enforce `tools.deny` for MCP tools exposed through CLI backends such as `claude-cli`.

## Best Possible Solution

Thread the effective per-agent tool policy into CLI MCP config preparation and filter generated MCP server entries before the external CLI reads them, with tests for server denies, wildcard/tool-name denies, and profile `bundle-mcp` allow semantics across CLI backends.

## Reproduction Assessment

Yes, source-reproducible: configure `claude-cli` with an MCP server and `tools.deny`, then the current generated CLI MCP config path has no effective-policy filter before writing the config. I did not run a live Claude CLI turn in this read-only review.

## Solution Assessment

Yes: filtering before writing the CLI MCP config is the narrow maintainable fix because `claude-cli` reads the config directly. A docs-only clarification would leave the documented deny-wins policy unenforced.
