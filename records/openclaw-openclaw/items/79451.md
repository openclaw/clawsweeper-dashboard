## Summary

Keep open: the documented deny/per-agent tool-policy contract still does not appear to be applied when the claude-cli bundle-MCP path generates its MCP config, so the reported permission bypass remains valid on current main.

## What This Changes

Per-agent `tools.deny` should block MCP tools exposed to claude-cli through generated `--mcp-config` overlays.

## Best Possible Solution

Enforce the effective agent/provider tool policy before writing CLI backend MCP config, omitting denied MCP servers or tool prefixes while preserving the existing OpenClaw loopback scoping.

## Reproduction Assessment

Yes, source-reproducible: the reporter gives a concrete 2026.5.7 claude-cli command path, and current main still merges existing/configured MCP servers into the generated config without passing agent-scoped tool policy. I did not live-run the external CRM MCP setup.

## Solution Assessment

Yes: applying effective policy at the generated MCP config boundary appears to be the narrowest maintainable fix because claude-cli reads those tools directly from the config file. A docs-only clarification would leave a documented permission boundary unenforced.
