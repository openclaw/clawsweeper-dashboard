## Summary

Keep open: current main still does not apply the documented per-agent/effective tool policy before emitting the claude-cli MCP config, and the linked fixing PR is still open.

## What This Changes

The issue reports that `agents.list[].tools.deny` does not block denied MCP tools for agents using the `claude-cli` backend.

## Best Possible Solution

Land a reviewed version of https://github.com/openclaw/openclaw/pull/79480 or an equivalent fix that applies effective tool policy before writing CLI MCP config without widening exact MCP tool allowlists.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: claude-cli bundle MCP config generation merges existing and configured MCP servers without carrying agent-scoped tool policy into the generated config path.

## Solution Assessment

Yes. Enforcing the effective policy at the generated MCP config boundary is the narrow maintainable direction because claude-cli reads tools directly from that config; the issue should stay open while the linked PR is reviewed and landed.
