## Summary

Keep open: current main still does not apply the documented per-agent tool deny policy before writing CLI backend MCP configs, and an open implementation PR is already linked with closing syntax.

## What This Changes

The issue requests enforcement of `agents.list[].tools.deny` for MCP tools exposed through the `claude-cli` backend.

## Best Possible Solution

Review and land a fix like https://github.com/openclaw/openclaw/pull/79480 that applies effective tool policy before emitting CLI backend MCP configs while preserving deny-wins and fail-closed allowlist semantics.

## Reproduction Assessment

Yes, source-reproducible: docs define per-agent deny policy, and current main still writes merged CLI MCP config without effective agent policy. I did not live-run the external CRM MCP setup.

## Solution Assessment

Yes, applying policy at the generated MCP config boundary is the right direction because `claude-cli` reads the tool surface directly from that config. A docs-only clarification would leave a documented permission boundary unenforced.
