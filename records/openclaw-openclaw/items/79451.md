## Summary

Keep open. Current main still materializes the generated `claude-cli` MCP config from enabled bundled/configured servers without applying the agent's effective `tools.deny` policy, and the direct fix PR is closed unmerged after review found a permission-widening issue.

## What This Changes

The issue reports that per-agent `tools.deny` does not filter MCP tools handed to the `claude-cli` backend, allowing denied MCP tools to remain callable.

## Best Possible Solution

Materialize a per-agent effective MCP config before invoking CLI backends, preserve deny-wins semantics without widening exact allowlists, fail closed when requested restrictions cannot be enforced, and cover exact/wildcard/profile cases with tests plus real Claude CLI config proof.

## Reproduction Assessment

Yes, source-level. Current main still builds the generated CLI MCP config from enabled/configured servers without applying the agent's effective `tools.deny` policy, and the separate runtime `toolsAllow` guard does not cover config-level deny entries.

## Solution Assessment

No, a docs clarification alone is not the best fix. The maintainable direction is an owner-reviewed pre-handoff filter or fail-closed path that does not widen exact MCP allow entries.
