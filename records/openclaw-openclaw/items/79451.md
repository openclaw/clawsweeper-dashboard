## Summary

Keep open: the documented per-agent deny policy still is not applied to the current-main CLI bundle-MCP config path, and the linked closing PR is open rather than merged.

## What This Changes

The issue reports that `agents.list[].tools.deny` does not block MCP tools exposed through the `claude-cli` backend because the generated MCP config is not filtered by the agent tool policy.

## Best Possible Solution

Enforce the effective agent/provider/global tool policy before writing CLI backend MCP configs, omitting denied MCP servers and failing closed where per-tool allow semantics cannot be represented, then land a reviewed fix.

## Reproduction Assessment

Yes, source-reproducible with high confidence: the reporter gave a concrete 2026.5.7 `claude-cli` path, and current main still emits merged MCP server config without effective agent tool-policy input. I did not run the external CRM MCP setup locally.

## Solution Assessment

Yes: applying policy at the generated CLI MCP config boundary is the narrow maintainable fix because `claude-cli` reads those tools directly from the config file. This issue should stay open until https://github.com/openclaw/openclaw/pull/79480 or an equivalent reviewed fix lands.
