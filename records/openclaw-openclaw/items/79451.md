## Summary

Keep open: the documented per-agent tool-policy contract still appears broken on current main for CLI backend MCP config generation, and the linked fix PR is open rather than merged.

## What This Changes

The issue reports that `agents.list[].tools.deny` does not prevent `claude-cli` agents from receiving and calling denied MCP tools from generated MCP config.

## Best Possible Solution

Review and land a narrow fix that applies the effective agent/provider tool policy before emitting CLI backend MCP config, while preserving profile-level `bundle-mcp` semantics and exact-tool allow behavior.

## Reproduction Assessment

Yes, source-reproducible: current main still sends unfiltered merged MCP server config into the CLI backend path, and the reporter provided a concrete `claude-cli` reproduction verified on 2026.5.4 and 2026.5.7. I did not run the external CRM MCP server live.

## Solution Assessment

Yes: enforcing the effective policy before writing the generated CLI MCP config is the right boundary because `claude-cli` reads that config directly. A docs-only clarification would leave an existing documented permission boundary unenforced.
