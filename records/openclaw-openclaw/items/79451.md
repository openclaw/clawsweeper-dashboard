## Summary

Keep open: current main still writes the claude-cli MCP config without applying the documented effective tool policy, and the linked fix PR is open but not merged.

## What This Changes

The issue reports that per-agent `tools.deny` rules are not applied before claude-cli backend MCP config generation, allowing denied MCP tools to remain callable.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79480 or a replacement that filters the effective tool policy before writing CLI backend MCP config, with regression coverage for deny-wins and exact MCP allow semantics.

## Reproduction Assessment

Yes, source-reproducible: the reporter supplied a concrete 2026.5.7 claude-cli path, and current main still merges and emits MCP server config without effective tool-policy input. I did not run the external CRM MCP setup locally.

## Solution Assessment

Yes: enforcing effective policy at the generated CLI MCP config boundary is the narrowest maintainable fix because claude-cli reads tools from that file directly. The issue should remain open while the linked PR is reviewed and landed.
