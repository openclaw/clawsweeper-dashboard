## Summary

Keep open. The linked open PR #77240 directly closes this issue, and source inspection confirms the shared resolver mismatch for a providerless nested model id; the issue should stay open until that PR lands or is rejected.

## What This Changes

The issue requests consistent cron model-override allowlist handling when a payload model is checked without the OpenRouter provider prefix but matches a configured `agents.defaults.models` entry.

## Best Possible Solution

Review and land or reject #77240, or an equivalent narrow shared resolver fix with a regression for the OpenRouter nested model case.

## Reproduction Assessment

Yes, at source level: a providerless nested raw model such as `arcee-ai/trinity-large-preview:free` is parsed as provider `arcee-ai`, while the allowlist key remains `openrouter/arcee-ai/trinity-large-preview:free`. I did not run a live cron job in this read-only pass.

## Solution Assessment

Yes, the narrowest maintainable direction is a shared resolver regression and fix that infers only an unambiguous configured provider after the initial allowlist miss. The linked PR appears to target that boundary, but it is still open and needs review.
