## Summary

Keep open. Current main still has a source-level mismatch for providerless nested cron model refs: `payload.model` is passed to the shared resolver, and the resolver only infers a configured provider when the raw value has no slash, while the linked open PR #77240 is already the active closing implementation for this issue.

## What This Changes

Fix cron payload model overrides whose providerless nested model id is rejected despite a matching provider-prefixed `agents.defaults.models` allowlist entry.

## Best Possible Solution

Land or close the linked resolver fix PR after confirming its regression covers the cron-shaped `arcee-ai/trinity-large-preview:free` value and preserves ambiguous-model fail-closed behavior.

## Reproduction Assessment

Yes, at source level: a cron payload carrying `arcee-ai/trinity-large-preview:free` reaches `resolveAllowedModelRef`, and the shared resolver does not infer a configured provider for slash-containing raw values. I did not execute a live cron job in this read-only review.

## Solution Assessment

Yes at the intended boundary: resolving a unique configured provider and re-checking the existing allowlist is narrower than changing cron storage or broadening the allowlist. The linked PR should be reviewed as the implementation vehicle before this issue closes.
