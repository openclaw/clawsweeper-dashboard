## Summary

Keep open. Current main still parses a nested providerless cron model as provider `arcee-ai` before allowlist lookup, and open PR #77240 is the active closing fix candidate.

## What This Changes

The issue reports that cron model overrides can be rejected when the stored model id lacks the provider prefix but the configured allowlist key includes it.

## Best Possible Solution

Review and land PR #77240, or apply the same shared resolver fallback with the nested OpenRouter regression test and changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `agents.defaults.models` with `openrouter/arcee-ai/trinity-large-preview:free`, then resolve cron-style raw model `arcee-ai/trinity-large-preview:free`. Current main parses it as provider `arcee-ai` and rejects the `arcee-ai/trinity-large-preview:free` key against the `openrouter/...` allowlist.

## Solution Assessment

Yes, the best direction is a narrow shared model resolver fix, not cron storage migration. PR #77240 already targets that boundary and should be reviewed or replaced before closing this issue.
