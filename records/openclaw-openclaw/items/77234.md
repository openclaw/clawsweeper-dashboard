## Summary

Keep open. Current main still rejects the providerless nested OpenRouter form by parsing `arcee-ai` as the provider, and an open linked PR already carries the concrete fix for this issue.

## What This Changes

Cron model overrides whose stored value loses the `openrouter/` prefix should still match an `agents.defaults.models` allowlist entry such as `openrouter/arcee-ai/trinity-large-preview:free`.

## Best Possible Solution

Land or revise PR #77240 so shared model resolution retries a uniquely configured OpenRouter provider after the first allowlist miss while preserving ambiguous-match rejection.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call `resolveAllowedModelRef` with an allowlist containing `openrouter/arcee-ai/trinity-large-preview:free` and raw `arcee-ai/trinity-large-preview:free`; current main parses `arcee-ai` as the provider and misses the allowlist.

## Solution Assessment

Yes. The linked PR's shared-resolver fallback is narrower than changing cron storage because the failing value can already reach `resolveAllowedModelRef` in providerless nested form.
