## Summary

The report points at a real remaining cron/model allowlist gap on current main: slash-containing OpenRouter model IDs can be presented or stored without the `openrouter/` provider prefix, and the allowlist resolver then treats the first path segment as the provider and rejects the job.

## What This Changes

Cron model overrides should resolve OpenRouter-style slash-containing model IDs consistently against `agents.defaults.models` allowlist entries.

## Best Possible Solution

Teach model override resolution to match slash-containing model IDs against a unique configured allowlist entry, and add cron coverage for OpenRouter-style `provider/org/model:free` entries plus the stripped `org/model:free` form.

## Reproduction Assessment

Yes from source inspection: a cron payload model of `arcee-ai/trinity-large-preview:free` with only `openrouter/arcee-ai/trinity-large-preview:free` allowed resolves to key `arcee-ai/trinity-large-preview:free` and fails the allowlist check. I did not execute the Vitest path in this read-only environment.

## Solution Assessment

Yes: the narrow maintainable fix is in shared model reference resolution or cron model suggestion serialization, with regression coverage in cron/model-selection tests. It should not add a config knob or relax the allowlist broadly when multiple configured providers are ambiguous.
