## Summary

Keep this PR open: it targets a real Codex app-server silent-turn gap, but the added watchdog still resets from shared, unscoped app-server activity, so it can miss the reported concurrent-turn stall scenario.

## What This Changes

Adds a Codex app-server transport-idle timeout option/default, timer wiring, abort handling, and a regression test in `extensions/codex/src/app-server/run-attempt.*`.

## Best Possible Solution

Land a turn-scoped transport-idle watchdog that ignores unrelated shared-client traffic, keeps the existing completion and terminal watchdog semantics, and includes regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start run A, deliver a notification or request for thread B through the shared Codex app-server client before the new deadline, and run A's transport timer is refreshed even though its own turn is silent.

## Solution Assessment

No. Arming a short watchdog at `turn:start` is the right direction, but the reset path must be scoped to the active thread/turn or separated from global app-server activity before this can reliably fix the reported stall.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Scope transport-idle resets to the active turn:** `extensions/codex/src/app-server/run-attempt.ts:890`
  - body: The new transport watchdog is rescheduled from `touchTurnCompletionActivity`, but that helper is called before notification/request scoping. Because the shared app-server client fans out every server notification to every run handler, progress from another thread can keep this stalled turn's transport timer alive; in the reported case with other concurrent turns continuing, the lane can still wait for the long terminal fallback. Reset this watchdog only for current thread/turn activity, or keep a separate current-turn activity timestamp.
