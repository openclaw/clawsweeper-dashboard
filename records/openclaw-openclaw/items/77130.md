## Summary

Keep open: the PR targets a real Codex app-server idle-stall gap and adds useful timeout flag plumbing, but the current diff still lets unrelated shared-client traffic refresh the new 60s watchdog and it lacks the required changelog entry.

## What This Changes

Adds a Codex app-server transport-idle timeout option/default, wires timeout flag propagation through the event projector, and adds regression tests for transport, terminal, wallclock, and normal completion outcomes.

## Best Possible Solution

Land a turn-scoped transport-idle watchdog and timeout flag plumbing that ignore unrelated shared-client traffic, with concurrency regression coverage and an `Unreleased` Fixes changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start run A, deliver a notification or request for thread B through the shared Codex app-server client before the new deadline, and run A's transport timer is refreshed even though its own turn is silent.

## Solution Assessment

No. Arming a 60s transport watchdog and plumbing `idleTimedOut` are the right direction, but the reset path must be scoped to current thread/turn activity before the PR reliably fixes the reported stall.
