## Summary

Keep open: the PR targets a real Codex app-server silent-turn gap, but the current diff can still miss the stated concurrent-turn repro and does not surface the new timeout as an idle timeout for retry policy.

## What This Changes

Adds a 60s Codex app-server transport-idle watchdog in `runCodexAppServerAttempt` and a regression test for a turn that starts but receives no notifications.

## Best Possible Solution

Land a scoped Codex app-server fix that uses a short per-turn idle watchdog, ignores unrelated shared-client traffic, marks idle timeout result flags for retry policy, adds regression coverage, and includes the required changelog note.

## Reproduction Assessment

Yes from source inspection: current main starts only the 30-minute terminal idle watch at `turn:start`, while the 60s completion idle watch is tied to later app-server request activity. I did not execute the repro because this review is read-only.

## Solution Assessment

No; the added watchdog is the right direction, but the current patch resets it from unscoped shared-client traffic and leaves `idleTimedOut` false, so it is not yet the narrow maintainable fix for the stated behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope transport-idle activity to the active turn:** `extensions/codex/src/app-server/run-attempt.ts:890`
  - body: This schedules the new transport-idle timer from `touchTurnCompletionActivity`, but `handleNotification` calls that helper before checking whether the notification belongs to this run's `threadId`/`turnId`. The default app-server client is shared and fans every notification to every handler, so progress from another concurrent Codex turn can keep this run's transport-idle deadline alive. That is the PR's real-world case, where this conversation is silent while other turns continue, so the stuck lane can still wait for the 30-minute fallback.
  - confidence: 0.88
- **[P2] Surface transport stalls as idle timeouts:** `extensions/codex/src/app-server/run-attempt.ts:765-768`
  - body: The new timeout sets only the local prompt-message flag and calls `projector.markTimedOut()`. On current main the projector still returns `idleTimedOut: false`, and `runCodexAppServerAttempt()` only overrides `timedOut`, `aborted`, and `promptError`, while the embedded runner's same-model idle retry requires `timedOut && idleTimedOut`. These new 60s transport stalls will therefore surface as errors instead of taking the idle-timeout retry path described in the PR.
  - confidence: 0.83
- **[P3] Add the required changelog entry:** `extensions/codex/src/app-server/run-attempt.ts:108`
  - body: This is a user-facing Codex reliability fix, but the PR only changes implementation and tests. OpenClaw policy requires user-facing `fix` changes to include a `CHANGELOG.md` entry, so this should add a single-line fix note before merge.
  - confidence: 0.78
