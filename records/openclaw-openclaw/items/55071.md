## Summary

Keep open. Current main still does not implement skill-driven plugin context visibility: the proposed `metadata.openclaw.activates.plugins` contract is absent from skill metadata parsing, skill snapshots, docs, and request-time plugin tool resolution, so this remains an active public API/design request rather than conservative cleanup.

## What This Changes

The issue requests a new skill metadata contract where matched skills can activate plugin tool visibility per turn through `metadata.openclaw.activates.plugins`.

## Best Possible Solution

Keep this open for maintainer-owned design of a backward-compatible activation contract that parses skill activation metadata, carries activated plugin ids through snapshots or an equivalent request-time skill state, applies them during plugin tool planning, and documents/tests policy precedence.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/API request; the current-main verification path is code and docs inspection showing the requested activation contract and request-time wiring are absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The proposed direction fits the repo's manifest-first plugin trajectory, but the exact API and precedence rules need maintainer design review before implementation.
