## Summary

Keep open: current main still lacks the requested skill-driven plugin context visibility contract, and the newly landed skill visibility work is diagnostic groundwork rather than `metadata.openclaw.activates.plugins` implementation.

## What This Changes

The issue requests a new skill metadata activation contract so matched skills can expose plugin tool context dynamically per turn.

## Best Possible Solution

Keep this as a maintainer-owned public API design item that adds a documented activation contract, threads it through skill/request state, and applies it during plugin tool planning with focused precedence tests.

## Reproduction Assessment

## Solution Assessment

Is this the best way to solve the issue? Unclear. Plugin-level activation fits the manifest-first direction, but maintainers still need to settle default visibility, precedence, bundled-skill recursion, and compatibility semantics before implementation.
