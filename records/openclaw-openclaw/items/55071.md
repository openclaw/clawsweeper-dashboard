## Summary

Keep open: current main still lacks the requested skill-driven plugin context visibility contract, and the latest related maintainer comment says #75983 is only groundwork, not an implementation of `activates.plugins`.

## What This Changes

The issue requests a new `metadata.openclaw.activates.plugins` skill metadata contract that would make plugin tool schemas visible only when matched skills activate those plugins.

## Best Possible Solution

Design a backward-compatible public activation contract, parse it from skill metadata, thread activation state into request-time plugin tool planning, and document/test policy precedence before implementation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/API request; the high-confidence check is current-main source and docs inspection showing that `activates.plugins` is not parsed, stored, documented, or consumed during tool assembly.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Plugin-level activation fits the manifest-first direction, but the exact contract and security/deny precedence need maintainer design review before code changes.
