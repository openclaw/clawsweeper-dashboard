## Summary

Keep open: current main still lacks the requested skill-driven plugin context visibility contract, and the latest human follow-up says https://github.com/openclaw/openclaw/pull/75983 is only visibility/debuggability groundwork, not an `activates.plugins` implementation.

## What This Changes

Request to add a public skill metadata contract such as `metadata.openclaw.activates.plugins` so matched skills can expose plugin tools and plugin-bundled skills on demand.

## Best Possible Solution

Keep this open for maintainer design review of a backward-compatible skill activation contract that parses activation metadata, threads it through skill snapshots or matched-skill state, applies it during request-time plugin tool planning, and documents/tests deny/allow precedence.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/API request; the high-confidence check is current-main source and docs inspection showing `activates.plugins` is not parsed, stored, documented, or consumed during plugin tool assembly.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Plugin-level activation fits the existing manifest-first control-plane direction, but the exact skill contract and security/precedence behavior need maintainer API review before implementation.
