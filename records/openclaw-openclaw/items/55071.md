## Summary

Keep open: current main and latest release still lack the requested `metadata.openclaw.activates.plugins` skill metadata contract, and request-time plugin tool selection is still driven by manifest availability plus static tool policy rather than matched-skill activation. The merged `skills check --agent` work in PR #75983 is useful visibility groundwork, but its author explicitly noted that it does not implement skill-driven plugin loading.

## What This Changes

Request to add `metadata.openclaw.activates.plugins` so matched skills can expose plugin-owned tools and bundled skills on a per-turn basis.

## Best Possible Solution

Keep this as a maintainer-owned skill/plugin API design item; if accepted, implement a generic activation contract across skill parsing, prompt snapshots, plugin tool planning, docs, and focused compatibility tests.

Not applicable; this is a feature/API request, and source inspection gives a high-confidence absence proof on current main and `v2026.6.10`.

Unclear; `activates.plugins` is plausible, but the best fix needs maintainer agreement on metadata semantics, deny/allow precedence, plugin SDK boundaries, and bundled-skill behavior.
