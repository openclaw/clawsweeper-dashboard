## Summary

Keep open: current main still has a source-backed path where an accepted `sessions_spawn` result already marked `isError: true` is treated as a failure during compaction, and the related accepted-spawn terminal-success fix does not cover this legacy compaction noise.

## What This Changes

The issue requests that accepted `sessions_spawn` results be stored as non-errors and that compaction skip older accepted spawn results already persisted with `isError: true`.

## Best Possible Solution

Add a narrow canonical classification/compaction fix: accepted `sessions_spawn` payloads should be non-errors, while `forbidden` and `error` statuses stay failures, and legacy accepted results should be skipped by the compaction failure summary.

## Reproduction Assessment

Yes, source-reproducible: a focused test can pass a `toolResult` for `sessions_spawn` with `details.status: "accepted"` and `isError: true` into compaction and observe the false `## Tool Failures` entry. I did not run that test in this read-only review.

## Solution Assessment

Yes; the best fix is a narrow tool-result classification and compaction guard using the existing accepted-spawn predicate, not a config option or a broad exemption for all tool failures.
