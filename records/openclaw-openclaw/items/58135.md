## Summary

Keep open. Current main still has lower-level `promptMode` support, but the public `sessions_spawn` tool does not expose, parse, forward, test, or document a prompt-mode parameter; the related task-delivery bug was fixed separately and does not implement this requested public API.

## What This Changes

Expose an optional `promptMode` parameter on `sessions_spawn` so native sub-agents can use `full`, `minimal`, or `none` prompt composition for deterministic blank runs.

## Best Possible Solution

If maintainers approve making prompt modes public, add a native `sessions_spawn.promptMode` enum with existing defaults preserved, forward it safely into child runs, keep task/attachment delivery visible in `none` mode, and document it at `https://docs.openclaw.ai/tools/subagents`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: lower-level `agent` runs accept `promptMode`, while the current `sessions_spawn` schema, executor forwarding, native params, and docs omit it.

## Solution Assessment

Yes with a maintainer API caveat. Exposing the existing enum on native `sessions_spawn` is the narrowest implementation if the public-surface decision is approved, but it must explicitly preserve assigned task delivery when `promptMode: "none"` is selected.

## Risks / Open Questions

- Current docs describe `promptMode` as runtime-set and not user-facing, so exposing it through `sessions_spawn` needs an explicit public API decision.
- A naive `promptMode: "none"` forward can suppress the system prompt path that currently carries native sub-agent task context; task and attachment delivery need dedicated coverage.
- The implementation should define whether this parameter is native-subagent-only or also accepted/ignored for `runtime: "acp"`, which has separate harness prompt semantics.
