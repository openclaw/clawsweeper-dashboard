## Summary

Keep open. Current main and the latest release still expose promptMode only on the lower-level agent run API, not on the public sessions_spawn schema or native spawn forwarding path, so the requested spawn API behavior remains missing and needs maintainer product/API review.

## What This Changes

Request to add an optional promptMode parameter to sessions_spawn so callers can spawn sub-agents with full, minimal, or none prompt context.

## Best Possible Solution

If maintainers approve the API, add a bounded sessions_spawn promptMode enum that preserves the current default, specifies none-mode task and attachment semantics, updates docs, and adds regression coverage.

## Reproduction Assessment

Yes for source-level verification, not as a broken existing-behavior repro: current main and v2026.6.6 still omit promptMode from sessions_spawn while lower-level agent runs accept it.

## Solution Assessment

Unclear until maintainers settle the public prompt-mode contract. Exposing the existing enum is the narrow technical path if trusted/native sessions_spawn should own this control, but the public API and none-mode semantics need approval first.
