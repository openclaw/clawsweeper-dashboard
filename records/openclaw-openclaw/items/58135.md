## Summary

Keep open. Current main and v2026.6.10 still expose `promptMode` only on lower-level agent runs, not on the model-callable `sessions_spawn` schema or native spawn handoff, so this remains a useful but product-sensitive public API request.

## What This Changes

Expose an optional `promptMode` parameter on `sessions_spawn` so spawned sub-agents can request `full`, `minimal`, or `none` prompt composition while preserving current defaults.

## Best Possible Solution

If maintainers approve the API, add a validated optional promptMode enum to `sessions_spawn` and native spawn params, forward it to child agent runs, preserve the current minimal default, document none-mode semantics, and cover schema plus forwarding behavior in focused tests.

## Reproduction Assessment

Yes for source-level verification of the missing feature: `sessions_spawn` and `SpawnSubagentParams` do not include `promptMode`, while the lower-level child agent path accepts it. This is not a broken existing-behavior repro because the request adds public API surface.

## Solution Assessment

Unclear until maintainers approve the public prompt-mode contract. Exposing the existing enum is the narrow technical path if `sessions_spawn` should own this control; otherwise prompt mode should remain runtime-owned and the limitation should stay documented.
