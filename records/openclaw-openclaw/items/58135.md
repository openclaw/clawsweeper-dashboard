## Summary

Keep open. Current main and the latest stable release still do not expose `promptMode` on `sessions_spawn`; related task-delivery and override fixes closed adjacent problems, but this remains a public spawn API/product decision.

## What This Changes

Requests an optional `sessions_spawn.promptMode` parameter so callers can request `full`, `minimal`, or `none` prompt construction for spawned child sessions.

## Best Possible Solution

If maintainers approve the API, add a native `sessions_spawn.promptMode` enum with existing defaults preserved, define `none` semantics for task, routing, and attachments, document it, and add focused regression coverage.

## Reproduction Assessment

Yes for source-level verification, not as a strict runtime bug. Current main and `v2026.5.19` omit `promptMode` from the public `sessions_spawn` schema, native params, and forwarding path while lower-level agent runs accept it.

## Solution Assessment

Unclear until maintainers settle the public prompt-mode contract. Exposing the existing enum is the narrow technical path if approved, but `none` mode, attachments, task visibility, and ACP/trusted-caller semantics need to be specified first.
