## Summary

Keep open. Current main and `v2026.5.22` still omit `promptMode` from the public `sessions_spawn` schema, native spawn params, and forwarding path, while lower-level agent runs already accept `promptMode`; this remains a public spawn API/product decision.

## What This Changes

The issue requests an optional `sessions_spawn.promptMode` enum so spawned child agents can choose `full`, `minimal`, or `none` prompt assembly for deterministic blank workers.

## Best Possible Solution

If maintainers approve the API, add a native `sessions_spawn.promptMode` enum with current defaults preserved, define `none` semantics for task delivery/routing/attachments, update docs, and add focused regression coverage.

## Reproduction Assessment

Yes for source-level verification, not as a strict runtime bug. Current main and `v2026.5.22` omit `promptMode` from the public `sessions_spawn` schema, native params, and forwarding path while lower-level agent runs accept it.

## Solution Assessment

Unclear until maintainers settle the public prompt-mode contract. Exposing the existing enum is the narrow technical path if approved, but `none` mode, attachments, task visibility, and ACP/trusted-caller semantics need to be specified first.
