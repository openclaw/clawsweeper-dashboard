## Summary

Keep open. Current main still has lower-level `promptMode` support, but the public `sessions_spawn` tool does not expose or forward it, and docs still describe prompt modes as runtime-set rather than user-facing; this needs a maintainer API/product decision before implementation.

## What This Changes

The issue requests adding an optional `promptMode` enum parameter to `sessions_spawn` so child agents can be spawned with `full`, `minimal`, or `none` prompt construction.

## Best Possible Solution

If maintainers approve the public API, add a native `sessions_spawn.promptMode` enum with current defaults preserved, forward it into the child run, prove task and attachment visibility under `none`, and document native/ACP semantics.

## Reproduction Assessment

Yes for the current implementation state: source inspection shows lower-level `agent` runs accept `promptMode`, while the public `sessions_spawn` schema, forwarding path, native params, tests/docs surface, and docs omit it. This is a source-verifiable missing feature rather than a broken existing contract.

## Solution Assessment

Yes with a maintainer API caveat. Exposing the existing enum on native `sessions_spawn` is the narrow technical path if this should become public, but raw-mode task delivery, attachments, and ACP behavior need explicit semantics first.

- The public API needs to define whether this parameter is native-subagent-only or how it behaves with `runtime: "acp"`.
