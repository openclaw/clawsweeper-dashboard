## Summary

Keep open. The linked gateway-token precedence problem is real and current main still lacks this exact diagnostic, but the PR is not merge-ready because it warns on broad env/config presence instead of the narrower stale local token conflict, leaves the requested status surfaces uncovered, and needs the usual user-facing changelog/test follow-through.

## What This Changes

The PR adds a doctor warning, a deep security-audit finding, and doctor-security tests for configurations where both `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` are set.

## Best Possible Solution

Land a shared, redacted gateway auth-source diagnostic that detects an active local token-auth conflict between a durable state-dir `.env` `OPENCLAW_GATEWAY_TOKEN` and a distinct configured `gateway.auth.token`, reuses it in doctor, status, gateway status, and deep security audit, and preserves existing credential precedence without printing secret values.

## Reproduction Assessment

Yes. A source-level reproduction is clear: set local token auth with one `gateway.auth.token` value and a different `OPENCLAW_GATEWAY_TOKEN`, then compare env-first CLI/probe resolution with config-first gateway server auth; the PR's false-positive path is also reproducible with `gateway.auth.token: "${OPENCLAW_GATEWAY_TOKEN}"`.

## Solution Assessment

No. Warning rather than deleting credentials is the right direction, but this implementation should use a narrower shared predicate and cover the status surfaces where users encounter the mismatch.
