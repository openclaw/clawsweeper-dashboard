## Summary

Keep open. The PR addresses a real gateway token precedence problem, but it is not merge-ready because the new doctor/audit predicates are presence-only, can warn on intentional env-backed config, and do not cover the status surfaces requested by the linked report.

## What This Changes

The PR adds a doctor warning, a deep security-audit finding, and doctor-security tests for configurations with both `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token`.

## Best Possible Solution

Land a shared, redacted gateway auth-source diagnostic that detects an active local token conflict between a durable `OPENCLAW_GATEWAY_TOKEN` and a distinct configured `gateway.auth.token`, reuses it in doctor, status, gateway status, and deep security audit, and preserves env-template/SecretRef setups without printing secret values.

## Reproduction Assessment

Yes. A source-level reproduction is clear with local gateway token auth where `gateway.auth.token` has one value and `OPENCLAW_GATEWAY_TOKEN` has a different value; current main resolves local CLI/probe credentials env-first while gateway server auth is config-first. The PR false positive is also reproducible with `gateway.auth.token: "${OPENCLAW_GATEWAY_TOKEN}"` or matching values.

## Solution Assessment

No. Warning rather than deleting credentials is the right direction, but this implementation should use a source/value-aware predicate and cover the status/gateway-status surfaces where users encounter the mismatch.
