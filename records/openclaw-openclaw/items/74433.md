## Summary

Keep open. The PR targets a real auth-shadow diagnostic gap, but the current diff is too broad and incomplete: it checks any process environment token rather than the durable state-dir `.env` source from the linked report, can false-positive env-backed config tokens, and does not surface remediation in the status paths where users hit the failure.

## What This Changes

The PR adds a doctor warning, a deep security-audit finding, and doctor-security tests for configurations where `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` are both present.

## Best Possible Solution

Land a shared, redacted local-gateway auth-shadow diagnostic that detects a non-empty state-dir `.env` `OPENCLAW_GATEWAY_TOKEN` shadowing a distinct configured `gateway.auth.token`, reuses that logic across doctor, status, gateway status, and deep security audit, and leaves credential precedence unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is clear: set a local `gateway.auth.token` and a different durable state-dir `.env` `OPENCLAW_GATEWAY_TOKEN`; local client credential resolution can use the env token while gateway server auth resolves config-first. The PR-specific false positive is also reproducible by setting `gateway.auth.token` to `${OPENCLAW_GATEWAY_TOKEN}` or an env SecretRef and observing that the PR would report two competing sources.

## Solution Assessment

No. A warning is the right product direction, but this implementation is not the narrowest maintainable fix because it reads arbitrary process env, does not compare distinct sources, and omits the status surfaces from the linked request.
