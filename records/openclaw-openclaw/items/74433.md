Keep open. The diagnostic is a valid direction and current main still lacks the requested warning, but this PR is not merge-ready because the predicate is too broad and the linked status surfaces remain uncovered.

## What This Changes

The PR adds a doctor warning, a deep security-audit finding, and doctor-security tests when `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` are both configured.

## Best Possible Solution

Land a shared, redacted gateway-auth conflict diagnostic that detects active local token source divergence, uses it in doctor, status, gateway status, and deep audit, tests real-conflict and env-backed non-conflict cases, and adds the required changelog entry.

Yes. A source-level reproduction is clear from current credential precedence: use local token auth with one configured token and a different `OPENCLAW_GATEWAY_TOKEN`, then compare env-first CLI/probe resolution with config-first gateway auth; the PR false-positive is also reproducible with `gateway.auth.token: "${OPENCLAW_GATEWAY_TOKEN}"`.

## Solution Assessment

No. A warning is the right non-destructive approach, but this implementation should use a narrower source-aware predicate and wire the existing status warning surfaces before it can close the linked issue.
