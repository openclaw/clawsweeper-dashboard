## Summary

Keep open. The PR targets a real gateway auth precedence problem from #74271, but the current patch is not merge-ready because its predicate is broader than the reported stale local `.env` conflict, it omits the status surfaces where users hit the failure, and the new audit path lacks coverage and changelog follow-through.

## What This Changes

The PR adds a doctor security warning, a `gateway.env_token_overrides_config` security-audit finding, and doctor-security tests for overlapping `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` configuration.

## Best Possible Solution

Land a shared, redacted gateway auth-source diagnostic that detects an active local token conflict from a durable OpenClaw env source shadowing a distinct configured `gateway.auth.token`, reuses it across doctor, status, gateway status, and security audit, and keeps existing credential precedence unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure local token auth with a literal `gateway.auth.token` and a different `OPENCLAW_GATEWAY_TOKEN`, then compare env-first client/probe resolution with config-first gateway auth; the PR’s false-positive path is also reproducible with `gateway.auth.token: "${OPENCLAW_GATEWAY_TOKEN}"`.

## Solution Assessment

No. Warning instead of automatic cleanup is the right direction, but this implementation needs a narrower shared predicate and must surface the remediation on the status paths where the user-facing failure appears.
