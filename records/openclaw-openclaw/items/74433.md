## Summary

Keep open. The linked report describes a real auth-source diagnostic gap, and current main still has no `gateway.env_token_overrides_config` or equivalent warning, but this PR is not merge-ready because it over-detects non-conflicting env-backed config, does not surface the warning in the status paths named by #74271, and lacks coverage/changelog for the added audit/user-facing output.

## What This Changes

The PR adds a doctor warning, a deep security-audit finding, and doctor-security tests for cases where `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` are both configured.

## Best Possible Solution

Land a shared, redacted gateway auth-source diagnostic that detects an active local token conflict between a durable state-dir `.env` `OPENCLAW_GATEWAY_TOKEN` and a distinct configured `gateway.auth.token`, reuses it in doctor, status, gateway status, and deep security audit, and preserves existing credential precedence.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure local gateway token auth with a literal `gateway.auth.token` and a different `OPENCLAW_GATEWAY_TOKEN`, then compare env-first CLI/probe credential resolution with config-first gateway server auth; the PR false-positive path is also reproducible with `gateway.auth.token: "${OPENCLAW_GATEWAY_TOKEN}"`.

## Solution Assessment

No. Warning rather than automatic deletion is the right direction, but this implementation should use a narrower shared diagnostic and cover the status surfaces where users encounter the failure.
