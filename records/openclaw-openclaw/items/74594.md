## Summary

Keep open for maintainer product and security review. Current main still does not implement the requested skill capability declaration or `openclaw skills inspect` surface, and the related permission-manifest/security issues are broader open design threads rather than a clear superseding implementation.

## What This Changes

Add an optional skills-first `capabilities` declaration plus an `openclaw skills inspect <skill-id>` command that displays tools, files, and exec access without enforcing permissions.

## Best Possible Solution

Keep the RFC open until maintainers choose the public capability vocabulary and decide how it relates to #12219/#10827; if accepted, implement typed parsing, status propagation, human/JSON CLI output, docs, and focused parser/formatter/CLI tests.

## Reproduction Assessment

Not applicable as a runtime bug. Static inspection gives a high-confidence verification path: current main lacks the schema field, status payload, `inspect` subcommand, and docs coverage requested here.

## Solution Assessment

Unclear. The inspect-only v0 is a plausible narrow first step, but the best maintainable solution depends on maintainer agreement about the security-facing vocabulary and its overlap with broader permission-manifest work.
