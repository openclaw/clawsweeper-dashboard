## Summary

Keep open for maintainer design follow-up. Current main has skill metadata, load-time requirements, status reporting, and `openclaw skills` search/install/update/list/info/check surfaces, but it still does not define a skills capability manifest or an `openclaw skills inspect` command; the related permission-manifest/security threads are useful prior art rather than a clear replacement for this narrower inspect-only RFC.

## What This Changes

The issue requests an optional skills-first capability manifest plus an `openclaw skills inspect <skill-id>` command that displays tools/files/exec declarations without changing runtime behavior.

## Best Possible Solution

Keep this as an open RFC until maintainers decide the skills capability vocabulary and how it relates to #12219 and #10827. If accepted, implement a typed schema, parser/status propagation, `openclaw skills inspect` rendering, docs, and focused tests with explicit inspect-only wording.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/RFC. Static inspection gives a high-confidence verification path: current main lacks the requested schema field, parser/status propagation, docs, tests, and `openclaw skills inspect` command.

## Solution Assessment

Unclear. The inspect-only v0 is a plausible narrow first step, but the best maintainable solution depends on maintainer product/security agreement about the public permission vocabulary and whether it should remain separate from the broader manifest/security threads.
