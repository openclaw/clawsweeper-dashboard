## Summary

Keep open. Current main and the latest release still lack a skill capability declaration schema and a top-level `openclaw skills inspect <skill-id>` command; the related open issues are broader security-design trackers, not a safe cleanup replacement for this narrower inspect-first RFC.

## What This Changes

The issue requests an optional skill capability manifest plus a CLI inspection surface that displays declared tool, file, and exec access without changing runtime enforcement.

## Best Possible Solution

Adopt one maintainer-approved, versioned skill capability declaration vocabulary that can power inspect output now and later feed consent, grants, enforcement, and audit work without fragmenting skill metadata.

## Reproduction Assessment

Yes at source level: current skill metadata, CLI registration, and docs show no capability declaration or top-level capability inspect command. A failing runtime reproduction is not applicable because this is a feature RFC rather than a broken existing contract.

## Solution Assessment

Unclear as an implementation path: the inspect-only v0 is a reasonable first slice, but maintainers must first approve the public capability taxonomy, command naming, warning language, and coordination with the broader permission/enforcement trackers.
