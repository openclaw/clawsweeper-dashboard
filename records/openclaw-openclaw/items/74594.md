## Summary

Keep open for maintainer design/security review. Current main has skill metadata, load-time requirements, status reporting, and `openclaw skills` list/info/check surfaces, but it does not define a skills capability manifest or an `openclaw skills inspect` command. The related #12219, #10827, and #6276 discussions are useful prior art, but they are broader or security-enforcement-oriented rather than a clear terminal replacement for this narrower inspect-only RFC.

## What This Changes

The issue proposes an optional skills-first capability manifest and an `openclaw skills inspect <skill-id>` command for tools/files/exec visibility without runtime enforcement.

## Best Possible Solution

Keep this as an open RFC until maintainers choose the skill capability vocabulary and decide how it relates to #12219 and #10827. If accepted, the maintainable path is a typed schema/parser, status propagation, human and JSON CLI rendering, docs with explicit inspect-only wording, and focused parser/formatter/CLI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/RFC. Static inspection gives a high-confidence verification path: current main lacks the requested schema field, status propagation, docs, tests, and `openclaw skills inspect` command.

## Solution Assessment

Unclear. The inspect-only v0 is a plausible narrow first step, but the best maintainable solution depends on maintainer agreement about the public permission vocabulary and whether it should fold into the broader open skill security manifest work.
