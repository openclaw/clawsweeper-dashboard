## Summary

Keep open for maintainer product/security review. Current main still has skill metadata, parser, status, CLI, and docs surfaces, but no declared skill capability manifest and no `openclaw skills inspect` command; the related open discussions are useful prior art, but they are broader security-manifest threads rather than a merged or clearly canonical replacement for this narrower inspect-only RFC.

## What This Changes

The issue requests an optional skills-first capability manifest for tools/files/exec plus an `openclaw skills inspect <skill-id>` command, explicitly with no runtime enforcement in v0.

## Best Possible Solution

Keep the RFC open until maintainers choose the capability vocabulary and its relationship to #12219 and #10827. If accepted, implement it as a typed skill metadata/parser extension, propagate declared capabilities through skill status, add human and JSON CLI rendering, document that v0 is inspect-only, and cover parser/formatter/CLI behavior with focused tests.

## Reproduction Assessment

Not applicable as a runtime bug. Static inspection gives a high-confidence verification path: current main lacks the requested schema field, status propagation, docs, and `openclaw skills inspect` command.

## Solution Assessment

Unclear. An inspect-only v0 is a plausible narrow first step, but the best maintainable solution depends on maintainer agreement about public permission vocabulary and how it relates to the broader open security-manifest work.
