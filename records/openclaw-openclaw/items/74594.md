## Summary

Keep open for maintainer product/security follow-up. Current main still lacks optional skill capability metadata, capability propagation into skill status, an `openclaw skills inspect` command, and docs for inspect-only capability output; the related permission-manifest threads are useful prior art but not a settled replacement for this narrower RFC.

## What This Changes

Request an optional skills capability manifest plus `openclaw skills inspect <skill-id>` that displays declared tools/files/exec access without changing runtime enforcement.

## Best Possible Solution

Keep the RFC open until maintainers choose the skills capability vocabulary and warning model; if accepted, implement typed optional metadata parsing, status propagation, human and JSON inspect output, docs, and focused parser/formatter/CLI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/RFC. Static inspection on current main gives a high-confidence verification path: the requested schema field, status propagation, inspect subcommand, and docs are absent.

## Solution Assessment

Unclear. The inspect-only v0 is a plausible narrow first step, but the best maintainable solution depends on maintainer agreement about the vocabulary and how to avoid implying enforcement.
