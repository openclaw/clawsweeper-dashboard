## Summary

Keep open. Current main and the latest release still make `tool_result_persist` and `before_message_write` synchronous, with Promise-returning handlers explicitly warned and ignored, so the reported governance gap is real. The issue body's AxonFlow plugin context and SonicBotMan's deployment comment both describe the same missing awaited pre-persistence policy point rather than a stale or already-implemented request.

## What This Changes

The issue requests Promise-return support for `tool_result_persist`, with similar consideration for `before_message_write`, so plugins can asynchronously scan, redact, or block tool-result transcript entries before persistence.

## Best Possible Solution

Define a first-class awaited tool-result policy point for OpenClaw-owned transcript/model visibility, with explicit trust gating, timeout and failure behavior, priority ordering, block/redact semantics, shape bounds, final cap/redaction preservation, SDK docs, API baseline updates, and Pi/Codex regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Registering an async `tool_result_persist` handler currently reproduces the gap at source/test level: `src/plugins/hooks.sync-only.test.ts` asserts the Promise result is ignored and the original tool-result message is returned.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Promise support is the right capability direction for the reported governance use case, but the best implementation needs maintainer design around the synchronous session append path and the existing bundled-only high-trust middleware boundary.
