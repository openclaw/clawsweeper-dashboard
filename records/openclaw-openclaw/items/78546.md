## Summary

Keep open: current main still documents Claude as a supported ACP spawn target and still has the timeout/control path that the latest comments identify as the OpenClaw-side failure. The merged diagnostic work made the error visible, but the timeout/control fix remains in overlapping open PRs rather than on main.

## What This Changes

OpenClaw-managed Claude ACP sessions spawned through `sessions_spawn(runtime: "acp")` fail with `ACP_TURN_FAILED`/`ACP_BACKEND_UNSUPPORTED_CONTROL` while direct ACPX Claude runs work.

## Best Possible Solution

Land a narrow generic ACP control-handling fix that skips or fail-softs unsupported auto-injected timeout controls based on backend-advertised config keys, preserves strict behavior for explicit user config writes, and includes regression coverage plus live Claude ACP proof.

## Reproduction Assessment

Yes. The issue and comments provide direct ACPX-success versus OpenClaw-failure comparisons, and current main still routes Claude ACP spawns through code that can emit and reject unsupported `timeout` config controls; I did not run live Claude ACP locally.

## Solution Assessment

Yes. The best fix direction is generic ACP runtime-control handling rather than a Claude-only workaround: filter or fail-soft optional auto-injected controls while keeping explicit user `set_config_option` errors strict.
