## Summary

Keep open: current main still documents Claude ACP as a supported `sessions_spawn` target and still routes ACP runtime timeout controls into a path that current Claude ACP does not advertise or accept. The merged diagnostic PR made the failure visible as `ACP_BACKEND_UNSUPPORTED_CONTROL`, but the control-handling fix remains in overlapping open PRs rather than on `main`.

## What This Changes

Fix OpenClaw-managed Claude ACP `sessions_spawn` so it completes like direct ACPX instead of failing with `ACP_TURN_FAILED` or unsupported timeout-control errors.

## Best Possible Solution

Land a narrow generic ACP runtime-control fix that skips or fail-softs unsupported auto-injected timeout controls while preserving strict errors for explicit user config writes, then prove it with focused tests and a real Claude ACP spawn.

## Reproduction Assessment

Yes, with high confidence from source and live reporter evidence: current main still maps ACP `timeoutSeconds` into a pre-turn `setConfigOption` path, while current Claude ACP exposes only mode/model/effort config options and rejects unknown ids. I did not run a live Claude ACP setup locally in this read-only review.

## Solution Assessment

Yes, the best fix direction is generic ACP runtime-control handling rather than a Claude-only workaround: avoid sending unsupported auto-injected controls or tolerate classified optional timeout rejections, while keeping explicit user-driven config errors strict.
