## Summary

Keep open: current main still documents Claude as a supported ACP `sessions_spawn` target and still has the timeout/control path that the latest issue comments identify as the failing OpenClaw-side cause. The merged diagnostic PR made the failure clearer, but the timeout/control fix is still only present in open PRs, not on main.

## What This Changes

The issue reports that Claude ACP one-shot sessions fail through OpenClaw `sessions_spawn(runtime: "acp")` with `ACP_TURN_FAILED` while direct ACPX and direct Claude ACP runtime calls succeed.

## Best Possible Solution

Land a narrow generic ACP control-handling fix that skips or fail-softs unsupported auto-injected timeout controls while preserving strict errors for explicit user `set_config_option` calls, then add focused regression coverage and live Claude ACP proof.

## Reproduction Assessment

Yes, source/report-reproducible with high confidence: the issue and comments provide direct ACPX-success versus OpenClaw-failure comparisons, and current main still routes Claude `sessions_spawn` through code that emits and fail-fast rejects unsupported `timeout` config controls. I did not run live Claude ACP locally in this read-only review.

## Solution Assessment

Yes, the best fix direction is generic ACP runtime-control handling rather than a Claude-only workaround: skip unsupported advertised-key aliases or fail-soft optional auto-injected controls while keeping explicit user control writes strict. The open PRs should be reviewed against that boundary and proven with a real Claude ACP spawn.
