## Summary

Keep open: current main still documents Claude as a supported ACP `sessions_spawn` target, and the source still has the reported unsupported timeout/config-control path. The diagnostic PR made the failure clearer, but the root fix is still in open PRs rather than on main.

## What This Changes

The issue reports that OpenClaw-managed Claude ACP `sessions_spawn` fails with `ACP_TURN_FAILED` while direct ACPX Claude execution succeeds.

## Best Possible Solution

Land or replace a narrow generic ACP runtime-control fix that skips unsupported auto-injected timeout controls while preserving strict behavior for explicit user config writes, with focused tests and live Claude ACP proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessions_spawn` can still mirror `runTimeoutSeconds` into ACP `timeout` config and then fail before the turn when the backend advertises a strict key list without `timeout`; I did not run live Claude ACP locally.

## Solution Assessment

Yes. The best fix direction is generic ACP runtime-control filtering or fail-soft handling for optional auto-injected controls, not a Claude-only auth or cwd workaround; the open PRs should be reviewed against that boundary.
