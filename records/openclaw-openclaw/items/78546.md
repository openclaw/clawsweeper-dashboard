## Summary

Keep open: current main still documents Claude as a supported ACP `sessions_spawn` target, but the OpenClaw runtime-control path can still inject or forward unsupported `timeout` config controls before the first turn. The merged diagnostic PR made the failure clearer, but the behavioral fix is still in open/closed-unmerged PRs rather than on main.

## What This Changes

Fix Claude ACP `sessions_spawn` so OpenClaw-managed Claude ACP turns succeed like direct ACPX runs.

## Best Possible Solution

Land one narrow generic ACP control-handling fix that uses handle-aware advertised config keys and skips or fail-softs unsupported optional timeout controls, then prove a real Claude ACP `sessions_spawn` run and separately triage any remaining auth-env or model-passthrough failure.

## Reproduction Assessment

Yes. The report and comments give direct ACPX-success versus OpenClaw-failure comparisons, and current main still routes Claude ACP spawns through source that can emit or forward unsupported `timeout` config controls before the first turn; I did not run live Claude ACP locally.

## Solution Assessment

Yes, the best direction is a generic ACP runtime-control fix rather than a Claude-only workaround: skip unsupported auto-injected controls using handle-aware advertised keys while preserving strict failures for explicit user config writes.
