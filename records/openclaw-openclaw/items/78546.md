## Summary

Keep open: current main still documents Claude as a supported ACP `sessions_spawn` target, and the current code still auto-emits a `timeout` runtime config control that Claude ACP does not advertise; the merged diagnostic PR clarified the failure, but the functional timeout/control fix is still only in open PRs.

## What This Changes

The issue asks OpenClaw to make `sessions_spawn({ runtime: "acp", agentId: "claude" })` complete like direct ACPX instead of failing with `ACP_TURN_FAILED` or unsupported runtime-control errors.

## Best Possible Solution

Land a narrow generic ACP runtime-control fix that skips or fail-softs unsupported auto-injected timeout controls while preserving strict failures for explicit user config writes, then add focused regression coverage and live Claude ACP proof.

## Reproduction Assessment

Yes. The issue and comments provide direct ACPX-success versus OpenClaw-failure comparisons, and current main source still shows timeout control injection plus strict unsupported-key rejection; I did not run live Claude ACP locally in this read-only review.

## Solution Assessment

Yes. The best fix direction is generic ACP runtime-control handling rather than a Claude-only workaround, because the same optional timeout control can affect any strict ACP backend that does not advertise or implement it.
