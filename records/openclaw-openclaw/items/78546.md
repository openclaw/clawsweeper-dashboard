## Summary

Keep open: the current `main` code still exposes Claude as a supported ACP `sessions_spawn` target, while the reported failure is backed by multiple same-path confirmations and current source still applies/fails ACP runtime config controls before the first turn. Open fix PRs exist, so this should stay as the canonical issue until one lands.

## What This Changes

The issue reports that OpenClaw-managed Claude ACP `sessions_spawn` fails with `ACP_TURN_FAILED` even though direct ACPX and other ACP harnesses work.

## Best Possible Solution

Land one canonical narrow fix that prevents optional unsupported ACP runtime controls from breaking Claude turns, preserves manager-enforced timeouts, and adds regression coverage for Claude/non-Codex ACP behavior.

## Reproduction Assessment

Yes, as source/report-reproducible: the issue and comments compare direct Claude/ACPX success with OpenClaw `sessions_spawn(runtime: "acp", agentId: "claude")` failure, and current main still applies ACP runtime config controls before turns. I did not independently run live Claude ACP in this read-only review.

## Solution Assessment

Unclear because this issue has no patch, but the best direction is a narrow generic control-handling fix rather than a Claude-only workaround if unsupported optional controls can affect multiple ACP backends.
