## Summary

Keep open: current main still has a concrete Claude ACP failure surface for `sessions_spawn(runtime: "acp")`. The issue discussion now identifies the likely root cause as OpenClaw auto-applying an unsupported `timeout` ACP config option, and source inspection confirms current main still emits that option and fails fast when the backend advertises a strict config-key list; related fix PRs are open but unmerged.

## What This Changes

Make Claude ACP sessions spawned through OpenClaw `sessions_spawn(runtime: "acp")` complete like direct ACPX instead of failing with `ACP_TURN_FAILED`.

## Best Possible Solution

Land one narrow capability-aware ACP runtime-control fix that does not send auto-injected `timeout` unless the backend advertises it, keeps explicit user controls diagnosable, and adds regression plus live Claude ACP proof.

## Reproduction Assessment

Yes, as source/report-reproducible. Current main still routes Claude `sessions_spawn` through ACP runtime controls that emit `timeout`, while the inspected Claude ACP package only advertises `mode`, `model`, and conditional `effort` and rejects unknown config ids; I did not run the live Claude ACP smoke locally.

## Solution Assessment

Yes for the fix direction, but not yet for one specific open PR. The narrow maintainable solution is core capability-aware filtering or fail-soft handling for optional auto-injected timeout controls, preferably without a Claude-only special case in the acpx wrapper.
