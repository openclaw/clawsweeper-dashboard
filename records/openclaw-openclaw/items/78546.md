## Summary

Keep open: current main still documents Claude as a supported ACP `sessions_spawn` target, and the current ACP control path still turns timeout runtime options into backend config controls that Claude ACP does not advertise or accept. The diagnostic PR already landed and made the failure clearer, but the actual timeout/control fix is still represented by open replacement PRs rather than current main.

## What This Changes

The issue reports that `sessions_spawn({ runtime: "acp", agentId: "claude" })` fails through OpenClaw with `ACP_TURN_FAILED` even though direct ACPX and Claude ACP calls work.

## Best Possible Solution

Land a narrow generic ACP control-handling fix that skips or fail-softs unsupported auto-injected timeout controls while preserving strict errors for explicit unsupported user `set_config_option` calls, then cover it with focused tests and live Claude ACP proof.

## Reproduction Assessment

Yes, source/report-reproducible with high confidence: the issue comments provide direct ACPX-success versus OpenClaw-failure comparisons, and current source still routes Claude spawns through timeout config-control emission and forwarding. I did not run a live Claude ACP session in this read-only review.

## Solution Assessment

Yes, the best fix direction is generic ACP runtime-control handling rather than a Claude-only workaround, because the same optional timeout-control rejection can affect multiple ACP backends. The open PRs should be reviewed against that boundary and proven with a real Claude ACP spawn.
