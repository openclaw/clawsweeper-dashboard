## Summary

Keep open: current main still supports Claude ACP through `sessions_spawn`, and the ACP runtime-control path still emits or forwards unsupported timeout config controls that recent live comments tie to this failure. Several open PRs address parts of the problem, but none is merged on current main.

## What This Changes

The issue asks OpenClaw to make `sessions_spawn({ runtime: "acp", agentId: "claude" })` complete like direct ACPX instead of failing with `ACP_TURN_FAILED`.

## Best Possible Solution

Land a focused manager-level fix that skips or fail-softs unsupported auto-applied ACP timeout controls while preserving explicit user config errors, then add regression coverage and a live Claude ACP smoke.

## Reproduction Assessment

Yes, source/report-reproducible: multiple users supplied live Claude ACP failures with direct ACPX success, and current main still contains the timeout-control emission/strict rejection path tied to the latest error detail. I did not independently run the live Claude ACP harness here.

## Solution Assessment

No merged current-main solution exists yet. The narrowest maintainable direction appears to be in the ACP manager/runtime-options control path, with acpx-specific timeout skipping only as a narrower workaround.
