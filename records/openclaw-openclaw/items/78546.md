## Summary

Keep open: current main still documents Claude as a supported ACP `sessions_spawn` target and still contains the OpenClaw-side timeout/control path linked by the latest diagnostics. The merged error-detail work made the failure clearer, but the timeout/control fix remains in open PRs rather than on main, and the newest `ANTHROPIC_API_KEY` observation may be a separate follow-up once the timeout failure is fixed.

## What This Changes

The issue reports that OpenClaw-managed `sessions_spawn({ runtime: "acp", agentId: "claude" })` fails with `ACP_TURN_FAILED` while direct Claude ACPX CLI and Runtime API runs succeed.

## Best Possible Solution

Land a generic ACP control-handling fix that avoids failing turns on unsupported auto-injected timeout controls while preserving strict handling for explicit user controls, add focused regression coverage and live Claude ACP proof, and split the ambient API-key auth behavior into a narrower follow-up if it still reproduces.

## Reproduction Assessment

Yes. The issue and comments provide repeated direct-ACPX-success versus OpenClaw-managed-failure comparisons, and current source still shows the managed path emitting and rejecting the `timeout` config control before a turn; I did not run a live Claude ACP session locally.

## Solution Assessment

Yes, the best fix direction is generic ACP runtime-control handling rather than a Claude-only workaround. The open PRs should be reviewed against that boundary, with live Claude ACP proof and a separate auth-env follow-up if needed.
