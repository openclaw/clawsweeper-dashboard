Keep open: current main has improved ACP diagnostics and advertised-key aliasing, but the timeout/control failure path described in the latest issue discussion is still present in source and several overlapping fix PRs remain open rather than merged.

## What This Changes

Fix OpenClaw-managed Claude ACP `sessions_spawn` so it returns a Claude response instead of failing with `ACP_TURN_FAILED` or `ACP_BACKEND_UNSUPPORTED_CONTROL` while direct ACPX works.

## Best Possible Solution

Land one narrow ACP runtime-control fix that makes unsupported optional timeout controls non-fatal or avoids emitting them when no advertised alias exists, while preserving strict feedback for explicit user config writes and adding live Claude ACP proof.

## Reproduction Assessment

Yes. The issue has high-confidence report evidence plus a current-main source path: `sessions_spawn` can persist `timeoutSeconds`, the ACP manager emits a `timeout` config pair, and the control path throws when the Claude backend does not advertise that key.

## Solution Assessment

Yes. The maintainable direction is a generic ACP control-handling fix around optional auto-injected timeout controls, rather than another Claude-only workaround; maintainers should choose one of the open timeout-control PRs and require live proof.
