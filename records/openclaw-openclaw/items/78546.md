## Summary

Keep open: the diagnostic work improved the error surface, but current main still has the ACP runtime-control paths that can make Claude ACP `sessions_spawn` fail, and the linked fix PRs are still open or closed-unmerged rather than landed.

## What This Changes

The issue reports that Claude ACP works through direct ACPX but fails through OpenClaw-managed `sessions_spawn(runtime: "acp", agentId: "claude")` with `ACP_TURN_FAILED`.

## Best Possible Solution

Land or consolidate a narrow ACP fix that makes manager-injected optional timeout controls non-fatal, preserves strict errors for explicit user control writes and permission controls, handles unsupported `session/close` at normal close time, and includes live Claude ACP proof.

## Reproduction Assessment

Yes. The issue thread provides multiple live direct-ACPX-success versus OpenClaw-failure comparisons, and current main still emits `timeout` runtime controls and rejects non-advertised keys before the turn; I did not run live Claude ACP locally.

## Solution Assessment

Unclear as a merge path, but the best fix direction is clear: generic ACP control handling is safer than a skip-all capability change, and the final PR should also account for the close-time unsupported-control path reported later in the thread.
