## Summary

Keep open: current main still exposes Claude as a supported ACP `sessions_spawn` target and still routes timeout runtime options through the unsupported config-control path identified in the latest issue comments. The merged diagnostics made the cause clearer, but the actual timeout/control fix is still only in open PRs.

## What This Changes

The issue reports that Claude ACP `sessions_spawn` fails with `ACP_TURN_FAILED` even though direct ACPX CLI and ACPX Runtime API calls succeed.

## Best Possible Solution

Land a narrow generic ACP runtime-control fix that skips or fail-softs unsupported auto-injected timeout controls while preserving strict errors for explicit user `set_config_option` calls, with focused regression coverage and live Claude ACP proof.

## Reproduction Assessment

Yes. The report and later comments give a high-confidence source/report reproduction where direct Claude ACPX succeeds but OpenClaw `sessions_spawn(runtime: "acp", agentId: "claude")` fails, and current main still contains the timeout-control path that explains it.

## Solution Assessment

Yes. The narrow maintainable fix belongs in generic ACP runtime-control handling, not a broad Claude-only workaround; the open PRs should be reviewed against that boundary and proven with a real Claude ACP spawn.
