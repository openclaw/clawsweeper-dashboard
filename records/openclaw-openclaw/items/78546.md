## Summary

Keep open: current main still documents Claude as a supported ACP spawn target and still routes `sessions_spawn(runtime: "acp", agentId: "claude")` through timeout/config-control code that can reject or forward unsupported `timeout` controls. The merged diagnostic PR made the failure clearer, but the actual compatibility fixes are still in open PRs rather than on main.

## What This Changes

The issue reports that Claude ACP sessions launched through OpenClaw `sessions_spawn(runtime: "acp")` fail with `ACP_TURN_FAILED` while direct ACPX Claude runs succeed.

## Best Possible Solution

Keep this issue as the canonical open bug until a narrow ACP compatibility fix lands: skip or fail-soft unsupported auto-injected timeout controls while preserving strict errors for explicit user control writes, handle unsupported close where appropriate, and require live Claude ACP spawn proof before closing.

## Reproduction Assessment

Yes, source/report-reproducible with high confidence: current main still builds and applies a `timeout` ACP config option, the ACPX wrapper still forwards it for Claude, and the pinned Claude ACP package does not advertise or accept that option. I did not run a live Claude ACP session locally in this read-only review.

## Solution Assessment

Yes: the best direction is a generic ACP runtime-control compatibility fix, with real Claude ACP proof, rather than closing this issue or adding another repair job. The open PRs should be reviewed or consolidated against that boundary.
