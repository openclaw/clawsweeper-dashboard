## Summary

Keep open: current main still exposes Claude as a supported ACP `sessions_spawn` target, and the current timeout/control path still maps `timeoutSeconds` to a `timeout` config key that Claude ACP does not advertise. The merged diagnostic work made the root cause visible, but the timeout-control fix is still only present in open PRs or proposed follow-ups, not on main.

## What This Changes

Fix OpenClaw-managed Claude ACP `sessions_spawn` so Claude sessions complete when direct ACPX works instead of failing with `ACP_TURN_FAILED`.

## Best Possible Solution

Land a narrow ACP control-handling fix that skips or fail-softs unsupported auto-injected timeout controls while preserving strict errors for explicit user config writes, then add regression coverage and live Claude ACP proof.

## Reproduction Assessment

Yes. The issue and comments provide direct ACPX-success versus OpenClaw-failure comparisons, and current main still emits a `timeout` config option that the current Claude ACP dependency does not advertise; I did not run live Claude ACP locally.

## Solution Assessment

Yes. The best fix direction is generic ACP runtime-control handling for unsupported auto-injected controls, not a Claude-only core workaround; the open PRs should be reviewed against that boundary and proven with a real Claude ACP spawn.
