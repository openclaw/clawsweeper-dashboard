## Summary

Keep open: current main still documents Claude as a supported ACP target and still contains the OpenClaw-side timeout/config-control path that the latest reports identify as failing; related PRs target parts of the fix, but no merged main change resolves the central Claude ACP sessions_spawn failure yet.

## What This Changes

The issue requests that `sessions_spawn({ runtime: "acp", agentId: "claude" })` complete like direct ACPX Claude execution instead of failing with `ACP_TURN_FAILED`.

## Best Possible Solution

Land a generic ACP control-handling fix that skips or fail-softs unsupported auto-injected timeout controls and unsupported close controls while preserving strict errors for explicit user config writes, then prove it with focused regression coverage and live Claude ACP output.

## Reproduction Assessment

Yes. The issue and comments provide direct ACPX-success versus OpenClaw-failure comparisons, and current main still routes Claude ACP sessions through code that emits and fail-fast rejects unsupported `timeout` config controls; I did not run live Claude ACP locally.

## Solution Assessment

Yes for the general direction, but current main has not implemented it. The maintainable fix should be generic ACP runtime-control tolerance/filtering plus close-path handling, not a broad Claude-only workaround.
