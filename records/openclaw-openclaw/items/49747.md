## Summary

Keep open. Current main and v2026.6.6 still do not extract `slack` from `agent:<id>:slack:...` group-session keys, and the exact focused fix path remains the open but conflicting https://github.com/openclaw/openclaw/pull/49750.

## What This Changes

The issue asks OpenClaw to preserve Slack thread routing for A2A sessions by parsing agent-prefixed Slack session keys correctly and preventing base-session fallback from overriding valid target delivery context.

## Best Possible Solution

Keep this issue open until the focused Slack A2A routing fix is rebased against current main, reviewed, and landed with parser and delivery-context regression coverage.

## Reproduction Assessment

Yes, by source inspection. A `ctx.From` like `agent:builder:slack:channel:C123:thread:...` still makes `resolveGroupSessionKey()` inspect `parts[0]` as `agent`, then fall back to a webchat provider hint instead of extracting `slack`.

## Solution Assessment

Yes in direction, but the existing fix PR needs rebase and maintainer review. The best shape is a canonical agent-prefixed session-key parser plus field-level delivery-context fallback coverage, not a second parallel routing shim.
