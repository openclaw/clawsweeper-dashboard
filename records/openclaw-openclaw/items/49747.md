## Summary

Keep open. Current main and v2026.6.6 still derive the group-session provider from the first `ctx.From` segment, so an `agent:<id>:slack:channel:...` key can still fall back to `webchat`; the focused fix path is the still-open https://github.com/openclaw/openclaw/pull/49750.

## What This Changes

The issue reports that A2A Slack thread session keys can be persisted as webchat and later announce/restart delivery can route through the base session instead of the target thread.

## Best Possible Solution

Keep this issue open until the focused Slack A2A routing fix is reviewed against current main and lands with parser and delivery-context regression coverage.

## Reproduction Assessment

Yes, from source inspection rather than a live Slack run. `resolveGroupSessionKey()` still reads `parts[0]` as the provider, so an `agent:<id>:slack:channel:...` key with a webchat provider hint can still resolve through webchat on current main.

## Solution Assessment

The requested direction is likely right, but the exact open PR needs rebase/re-review because current main already partially hardened restart-sentinel fallback. The best fix is a canonical agent-prefixed session-key parser plus field-level delivery-context fallback coverage, not a second parallel routing shim.
