## Summary

Keep this open. Current main still contains both reported routing behaviors: agent-prefixed Slack channel keys can be resolved using the fallback provider, and restart/announce recovery still merges base-session routing for incomplete thread metadata. The provided timeline also identifies open PR #49750 as the active implementation candidate, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for A2A Slack thread routing to preserve agent-prefixed Slack session keys as Slack routes and to keep restart/announce fallback from replacing valid target delivery context with a base-session route.

## Best Possible Solution

Land #49750 or an equivalent narrow fix that makes group-session resolution use the shared agent-aware conversation parser, preserves explicit A2A target delivery context, and changes restart/announce recovery so base-session data only fills genuinely missing fields without redirecting a valid target route. Add focused regression coverage for agent-prefixed Slack thread metadata and restart/announce delivery routing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: call `resolveGroupSessionKey()` with `From=agent:builder:slack:channel:C123:thread:T`, `Provider=webchat`, and `ChatType=channel`; the resolver checks only the first segment and can return `webchat` instead of the encoded Slack provider.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix: reuse the existing agent-aware session-conversation parser and make restart/announce fallback field-level, rather than adding Slack-specific special cases across routing code.
