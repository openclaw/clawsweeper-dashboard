## Summary

Keep open: current main still has the agent-prefixed Slack parser gap, while later delivery-route work only partially overlaps. The direct fix path is the existing open paired PR, which needs review or refresh rather than a duplicate repair lane.

## What This Changes

The issue reports Slack A2A thread session keys being parsed as webchat and delivery fallback using the base session route instead of the target Slack thread.

## Best Possible Solution

Review or refresh the existing paired fix PR against current main, preserving current field-level fallback behavior while adding shared parser coverage for agent-prefixed Slack thread keys.

Yes, at source level: current main still reads the first `ctx.From` segment as the provider candidate, so an `agent:builder:slack:channel:...` key can fall back to a webchat provider hint. I did not run a live Slack A2A reproduction in this read-only review.
