## Summary

Keep open: current main and the latest release still leave the agent-prefixed Slack session-key parser path vulnerable to resolving as the provider hint, while the related open pull request is only a candidate and has not landed.

## What This Changes

The issue asks OpenClaw to preserve A2A Slack thread routing by parsing `agent:<id>:slack:...` keys as Slack routes and preventing base-session fallback from overriding valid target delivery context.

## Best Possible Solution

Refresh, review, and either land or replace https://github.com/openclaw/openclaw/pull/49750 with a focused parser-plus-routing fix, regression tests, and real Slack-thread proof.

## Reproduction Assessment

Yes, by source inspection: current main still reads the first `From` segment as the provider candidate, so an `agent:<id>:slack:channel:...` key can fall back to a webchat provider hint. I did not run the live Slack workflow in this read-only review.

## Solution Assessment

Yes for the requested direction: extracting the encoded provider and making fallback field-level is the right boundary to repair. The best landing path is still the existing candidate pull request or a narrow replacement after maintainer review and proof.
