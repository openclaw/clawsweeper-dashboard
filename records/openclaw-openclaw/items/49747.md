## Label Justifications

- `P2`: The report concerns a bounded Slack/A2A routing bug with meaningful but limited multi-agent workflow impact.
- `impact:session-state`: The reported failure can persist session metadata under the wrong transport route, drifting Slack thread state toward webchat.
- `impact:message-loss`: The reported announce/reply fallback can route messages to the wrong session or thread, making the intended Slack response disappear from the expected place.

## Summary

Keep open. Current main narrowed one symptom by rejecting direct `sessions_send` thread-session targets, but `resolveGroupSessionKey()` still derives the provider from the first `ctx.From` segment or the context fallback, and the same-author focused fix PR is still open rather than merged.

## What This Changes

The issue requests fixing A2A Slack thread session routing so agent-prefixed Slack keys do not normalize to webchat and fallback delivery cannot route announces or replies to the base session.

## Best Possible Solution

Keep the current direct-thread target guard, then land or replace the focused parser and delivery-context fix only if it preserves valid agent-prefixed Slack route parsing without reintroducing A2A traffic into human-facing thread targets.

## Reproduction Assessment

Yes, at source level: a `MsgContext` with `From="agent:builder:slack:channel:C123:thread:..."` and a webchat provider hint still reaches `resolveGroupSessionKey()` with `parts[0] === "agent"`, so the provider can fall back away from Slack. I did not run the full live Slack A2A thread scenario.

## Solution Assessment

Unclear as a final product path: the parser fix direction is plausible, but the open PR must be reconciled with the newer documented rule that direct `sessions_send` thread-session targets are invalid. The safer end state is parser correctness plus tests without weakening the parent-session coordination model.
