- `P2`: This is a normal-priority routing bug with limited but real impact on multi-agent Slack thread workflows.

## Summary

Keep open: current main and the latest release still parse `agent:<id>:slack:...` group keys from the first segment, so the central Slack-thread normalization bug remains; the related restart fallback path is partly improved on main, and the same author already has an open fix candidate that should be reviewed or refreshed.

## What This Changes

The issue requests stable A2A Slack thread routing by parsing agent-prefixed Slack session keys correctly and preventing base-session fallback from overriding explicit delivery context.

## Best Possible Solution

Review or refresh https://github.com/openclaw/openclaw/pull/49750 against current main, preserving the current field-level restart fallback while adding shared parser coverage for agent-prefixed Slack thread keys.

## Reproduction Assessment

Yes, at source level: current main still reads the first segment of `ctx.From` as the provider candidate, so an `agent:builder:slack:channel:...` key can fall back to a webchat provider hint. I did not run a live Slack A2A reproduction in this read-only review.

## Solution Assessment

Yes on direction, but not as a new queued fix: the maintainable fix belongs in the shared session-key parser plus focused restart/announce routing tests, and the existing open PR should be reviewed or rebased instead of starting a second repair lane.
