## Summary

Keep open: current main and the latest release still save Feishu downloaded media into the shared managed inbound bucket, the strict Feishu account schema still has no account-scoped inbound storage key, and the broader media-isolation issue is related but does not replace this Feishu account-boundary decision.

## What This Changes

Request account-scoped Feishu inbound media storage so multi-account deployments can avoid sharing one downloaded-media directory across trust boundaries.

## Best Possible Solution

Define a managed Feishu account-scoped inbound media contract that coordinates with the broader scoped-media tracker and covers saves, managed refs, local-root authorization, sandbox staging, cleanup, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: configure two Feishu accounts and current main still saves Feishu response media with the literal inbound bucket while the strict account schema has no scoped inbound setting. I did not run live multi-account Feishu traffic in this read-only review.

## Solution Assessment

Unclear as proposed: inboundDir addresses the symptom, but arbitrary per-account absolute directories are riskier than a managed scoped-media contract. The best fix should cover read paths, staging, cleanup, local-root policy, compatibility, and docs together.
