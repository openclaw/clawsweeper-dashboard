## Summary

Keep open. Current main still stores Feishu downloads from all configured accounts in the shared managed `inbound` bucket, and Feishu account config/docs still do not expose account-scoped inbound storage; #19330 is related but broader and does not replace this Feishu account-boundary request.

## What This Changes

The issue requests account-specific inbound media storage for multi-account Feishu setups so one account's attachments are not exposed through another account or agent media root.

## Best Possible Solution

Define account-scoped managed inbound buckets under OpenClaw-controlled media storage, then update Feishu saves plus shared media reference, access-root, staging, cleanup, docs, and tests without allowing arbitrary host paths.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure two Feishu accounts and both download paths still save via `saveMediaBuffer(..., "inbound", ...)`, while `channels.feishu.accounts.<id>.inboundDir` is not accepted by the strict account schema.

## Solution Assessment

Unclear. The requested `inboundDir` fixes the visible symptom, but the safer maintainable direction is a managed account-scoped media-store contract rather than user-configured absolute inbound directories.
