## Summary

Keep open: current main and the latest release still route Feishu inbound media through the shared `inbound` media bucket, Feishu has no account-scoped inbound storage setting, and the open cross-channel tracker does not replace this Feishu account-boundary decision.

## What This Changes

The issue requests an optional per-Feishu-account inbound media directory so media from separate Feishu accounts is isolated.

## Best Possible Solution

Define a backward-compatible managed Feishu account-scoped inbound media contract and thread that scope through saves, managed references, local access, sandbox staging, cleanup, docs, and regression tests without exposing arbitrary host paths by default.

## Reproduction Assessment

Yes, at source level: configure two Feishu accounts and current main still saves Feishu response media with the literal `inbound` bucket while the strict account schema has no scoped inbound setting. I did not run live multi-account Feishu traffic.

## Solution Assessment

Unclear as proposed: `inboundDir` addresses the symptom, but arbitrary per-account absolute directories are riskier than a managed scoped-media contract. The best fix should cover read paths, staging, cleanup, local-root policy, compatibility, and docs together.
