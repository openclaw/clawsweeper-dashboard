## Summary

Keep open: current main and v2026.6.6 still route Feishu inbound downloads into the shared managed `inbound` media bucket, the Feishu account schema has no account-scoped inbound storage key, and https://github.com/openclaw/openclaw/issues/19330 remains the broader cross-channel tracker rather than a replacement for this Feishu account-boundary decision.

## What This Changes

Request account-scoped inbound media storage for multi-account Feishu so attachments from different Feishu accounts do not share one global media directory.

## Best Possible Solution

Define a managed Feishu account-scoped inbound media contract that preserves and authorizes scope through saves, managed references, local roots, sandbox staging, cleanup, docs, and regression tests, while coordinating with the cross-channel tracker.

## Reproduction Assessment

Yes, at source level: configure two Feishu accounts and current main still saves Feishu response media with the literal `inbound` bucket while the strict account schema has no scoped inbound setting. This review did not run live multi-account Feishu traffic.

## Solution Assessment

Unclear as proposed: `inboundDir` addresses the symptom, but arbitrary per-account absolute directories are riskier than a managed scoped-media contract. The best fix should cover read paths, staging, cleanup, local-root policy, compatibility, and docs together.
