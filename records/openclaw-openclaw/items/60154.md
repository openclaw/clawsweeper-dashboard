## Label Justifications

- `P2`: This is a credible multi-account media privacy improvement with real but bounded blast radius and product/security design work remaining.
- `impact:security`: The report is about sensitive inbound media from one Feishu account being readable through a shared media root by another lower-trust agent.
- `impact:session-state`: Inbound media becomes part of agent/session context and can be mis-associated when all accounts share the same flat inbound pool.

## Summary

Keep open: current main and the latest release still route Feishu inbound media through the shared inbound bucket, reject scoped managed inbound references, and the open broad tracker at https://github.com/openclaw/openclaw/issues/19330 does not settle the Feishu account-level storage/config decision.

## What This Changes

Requests an optional per-Feishu-account inbound media directory so separate Feishu app accounts do not share the global inbound media store.

## Best Possible Solution

Define a managed, backward-compatible account-scoped Feishu inbound media contract coordinated with https://github.com/openclaw/openclaw/issues/19330, and avoid arbitrary account absolute paths unless maintainers deliberately approve that host-path policy.

## Reproduction Assessment

Yes, at source level: configure two Feishu accounts and current main still saves representative Feishu media through the shared inbound bucket while the strict account schema has no scoped inbound setting. I did not run a live multi-account Feishu deployment in this read-only review.

## Solution Assessment

Unclear as proposed. The requested inboundDir addresses the symptom, but a managed scoped-media contract is safer than arbitrary per-account absolute directories and must cover read paths, staging, cleanup, compatibility, and explicit sharing.
