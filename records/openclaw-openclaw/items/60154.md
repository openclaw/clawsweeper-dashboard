## Summary

Keep open. Current main still stores Feishu inbound media from all accounts in the shared media bucket and has no account-level `inboundDir` config; #19330 is relevant but remains broader per-agent media isolation work, not a complete substitute for this Feishu account-boundary request.

## What This Changes

Request account-specific inbound media storage for multi-account Feishu setups so attachments received by one account are not exposed through another agent's media roots.

## Best Possible Solution

Design account-scoped managed inbound buckets, likely under the existing media root, and update Feishu media saving plus the shared read/staging/reference paths so account isolation is enforced without introducing arbitrary host-path trust. The work should align with #19330 but still include Feishu account schema/docs/tests if Feishu needs account-specific configuration or migration behavior.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: configure two Feishu accounts and both media download paths still call `saveMediaBuffer(..., "inbound", ...)`, while the strict Feishu account schema rejects `channels.feishu.accounts.<id>.inboundDir`.

## Solution Assessment

Unclear. The requested `inboundDir` would address the visible Feishu symptom, but a validated account-scoped managed bucket or generic channel/account media-store seam is safer than allowing arbitrary absolute inbound directories.
