## What This Changes

The PR changes model fallback cooldown probe bookkeeping so a primary near-expiry probe does not consume the same-provider transient probe slot, and adds regression tests for that fallback chain behavior.

## Best Possible Solution

Keep the narrow runtime guard, repair the new tests to use the existing mocked auth-store setup, then require contributor real behavior proof and focused maintainer validation before merge.

## Reproduction Assessment

Yes. From source inspection, put the primary provider in near-expiry transient cooldown, make that primary probe fail, and include a later fallback on the same provider; current main can add that provider to `cooldownProbeUsedProviders` and skip the later fallback.

## Solution Assessment

Mostly yes. The `!isPrimary` runtime guard is the narrow maintainable fix because it preserves non-primary same-provider deduplication, but the submitted test implementation must be repaired before this PR is a good merge path.
