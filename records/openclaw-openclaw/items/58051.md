## Summary

Keep open. Current main still lets a primary transient cooldown probe consume the per-run provider probe slot, so the same-provider fallback bug remains; this PR's narrow guard change and regression coverage remain relevant, and I found no blocking PR-introduced correctness issue.

## What This Changes

The PR changes `runWithModelFallback` so primary cooldown probes neither check nor consume same-provider transient probe slots, and adds regression tests for same-provider fallback probing and non-primary deduplication.

## Best Possible Solution

Land the narrow current-main behavior change: primary cooldown probes should not consume `cooldownProbeUsedProviders`, while non-primary same-provider fallbacks should still deduplicate among themselves, with both regression tests preserved or strengthened.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a primary provider in transient cooldown near probe expiry, let that primary probe fail, then include a later same-provider fallback; current main records the primary provider in `cooldownProbeUsedProviders` and can skip the later fallback.

## Solution Assessment

Yes. The proposed `!isPrimary` guard is the narrowest maintainable fix for this edge case because it changes only the transient per-run provider-slot bookkeeping and preserves non-primary deduplication.
