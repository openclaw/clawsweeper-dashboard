## Summary

Keep open: current main still has the primary-probe slot consumption behavior, and the PR’s runtime fix is a narrow direction, but it still needs real behavior proof and its tests need to be rebased onto the current mocked auth-store harness before merge.

## What This Changes

The PR changes model fallback cooldown probe bookkeeping so primary probes do not consume the same-provider transient probe budget, and adds regression tests for that behavior.

## Best Possible Solution

Keep the narrow `!isPrimary` runtime guard, rework the new tests onto current `setAuthRuntimeStore`/`withTempAuthStore` helpers, then validate the real fallback chain with redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current main marks a transient cooldown probe provider without checking `isPrimary`, then records that provider after a failed probe, so a later same-provider fallback can hit the one-provider-per-run skip. I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes: excluding only primary candidates from transient provider-slot bookkeeping is the narrow maintainable fix because it preserves non-primary same-provider deduplication. The submitted PR is not merge-ready until the tests are rebased and real behavior proof is supplied.
