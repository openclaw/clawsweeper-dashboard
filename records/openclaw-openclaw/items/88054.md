## Summary

Keep this PR open: the targeted doctor false-positive fix is useful and now has positive live proof, but the current implementation can still suppress the configured SecretRef warning by authenticating the probe with fallback Gateway credentials instead of proving the configured SecretRef path is healthy.

## What This Changes

The PR adds a doctor Gateway status probe before emitting an unresolved `gateway.auth.token` SecretRef warning and adds unit coverage for inline-token and healthy-probe paths.

## Best Possible Solution

Keep the targeted suppression only when the probe proves the configured SecretRef/runtime path is healthy, or explicitly document and test a maintainer-approved fallback-auth suppression policy while preserving diagnostics for truly broken configured SecretRefs.

## Reproduction Assessment

Yes, at source level: the PR resolves the configured SecretRef with fallback disabled, then calls generic `callGateway`, which resolves local Gateway credentials from `process.env` with env-first precedence. I did not run the e2e path because this review is read-only.

## Solution Assessment

No. The branch is close, but the best fix needs either a probe that cannot bypass the configured SecretRef contract or an explicit maintainer decision that fallback-authenticated Gateway health is enough to suppress the warning.
