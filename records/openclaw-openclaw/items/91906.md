## Label Justifications

- `merge-risk: 🚨 message-delivery`: Merging as-is could suppress live partial replies for directly delivered subagent sessions even though final text may still arrive.

## Summary

Keep this PR open. The optimization is useful in detached subagent runs, but the patch currently gates suppression on the whole subagent lane even though current main has thread-bound/inline subagent launch paths that do have direct channel delivery, and the external PR proof is still test-only.

## What This Changes

The PR adds an internal `suppressLiveStreamOutput` flag that flows from subagent-lane attempt execution through the embedded runner subscription so streaming updates skip live preview parsing while final `message_end` text remains delivered.

## Best Possible Solution

Gate live-stream suppression on a true no-live-consumer condition, or explicitly prove and document that directly delivered subagent sessions are intended to become final-only, then add real subagent runtime proof.

## Reproduction Assessment

Source-reproducible: current main has native and ACP subagent paths that pass direct or inline delivery while still using `AGENT_LANE_SUBAGENT`, and the PR suppresses live parsing based only on that lane. I did not run a live thread-bound subagent scenario.

## Solution Assessment

No: the optimization is plausible for detached subagents, but lane-only suppression is too broad unless direct-delivery subagents are intentionally final-only. The safer fix is to key suppression off the absence of a live delivery consumer.
