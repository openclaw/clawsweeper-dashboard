## Summary

Keep open: the performance idea is plausible, but the branch suppresses live parsing for the whole subagent lane while current main still has direct and inline subagent delivery paths on that lane. The PR also still has only focused tests/lint as proof, not a real subagent run or benchmark.

## What This Changes

The PR threads a new internal `suppressLiveStreamOutput` flag from subagent-lane attempts through the embedded runner subscription so streaming updates skip live visible-text parsing while final `message_end` output remains delivered.

## Best Possible Solution

Gate live-stream suppression on an actual no-live-consumer condition, or explicitly approve final-only direct subagent behavior, then add real runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main has native and ACP subagent paths that pass direct or inline delivery while still using `AGENT_LANE_SUBAGENT`, and the PR suppresses live parsing based only on that lane. I did not run a live thread-bound subagent scenario.

## Solution Assessment

No: the optimization is plausible for detached subagents, but lane-only suppression is too broad unless direct-delivery subagents are intentionally final-only. The safer fix is to key suppression off the absence of a live delivery consumer.
