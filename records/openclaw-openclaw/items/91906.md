## Summary

This PR is still potentially useful, and current main does not already implement the suppression flag, but it is not merge-ready: the lane-only predicate can suppress live updates for directly delivered subagent sessions, the submitted proof is test-only, and GitHub currently reports the branch as conflicting with main.

## What This Changes

The PR threads an internal `suppressLiveStreamOutput` flag from subagent-lane attempts through the embedded runner subscription so streaming `message_update` handling skips live preview/reasoning parsing while `message_end` final text still emits.

## Best Possible Solution

Gate suppression on a real no-live-consumer condition, or get explicit maintainer approval for final-only directly delivered subagents, then rebase and add real runtime proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current native and ACP session spawns can deliver directly or inline while still using `AGENT_LANE_SUBAGENT`, and this PR suppresses live parsing based only on that lane. I did not run a live thread-bound subagent scenario.

## Solution Assessment

No. The optimization is plausible for detached subagents, but lane-only suppression is too broad unless maintainers intentionally approve final-only directly delivered child sessions; the safer fix is to key suppression off the absence of a live delivery consumer.
