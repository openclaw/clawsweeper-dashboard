## Summary

Keep open: the Gateway row/search fallback is the right owner boundary, but latest head still treats any parsed agent id other than literal `main` as non-default, so a configured default agent named differently can display and search omitted-account direct rows under the wrong account name; the supplied proof is also focused test output only.

## What This Changes

The PR adds Gateway direct-session row/search fallback to configured channel account names, expands session utility coverage, and stabilizes one `test-projects` helper-import routing assertion.

## Best Possible Solution

Land the shared Gateway row/search fallback after it compares parsed agent ids against `resolveDefaultAgentId(cfg)` and includes redacted real `sessions.list` handler or runtime proof for default and non-default account cases.

## Reproduction Assessment

Yes, at source level. Current main and the latest release return `origin.label` for opaque direct-session rows/search, and the remaining PR defect is visible from the candidate order when the configured default agent id is not `main`.

## Solution Assessment

No, not yet. The Gateway row/search layer and existing account `name` field are the maintainable fix boundary, but the ordering needs to use the configured default agent instead of only `DEFAULT_AGENT_ID`.
