## Summary

Keep open: the PR addresses a plausible hot-path PDF tool stall, but the proposed module cache can return stale model availability or the wrong PDF model for the same agent/workspace after config or auth-profile state changes.

## What This Changes

The PR adds a module-level PDF model-config cache keyed by agentDir/workspaceDir, exports a cache reset helper, adds a regression test, and records a changelog fix for #76644.

## Best Possible Solution

Keep the hot-path optimization, but make the cache dependency-aware or invalidated by config/auth-profile state, and avoid letting an initial null or old model selection persist across later valid runtime inputs.

## Reproduction Assessment

Yes, source-level reproduction is clear for the PR bug: call createPdfTool twice with the same agentDir/workspaceDir but different cfg or authProfileStore inputs, and the proposed cache will reuse the first resolver result. The original event-loop stall is also supported by the linked report's logs and the current hot-path resolver call.

## Solution Assessment

No, not as written: caching PDF model resolution is a reasonable direction, but keying only by agentDir/workspaceDir is not the narrowest maintainable fix because the resolver depends on config and auth-profile state. A safer fix should cache against those dependencies or invalidate when they change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Include config and auth state in the PDF cache:** `src/agents/tools/pdf-tool.ts:273-274`
  - body: The resolver result is not determined only by agentDir and workspaceDir: resolvePdfModelConfigForTool also reads cfg and authProfileStore to choose explicit pdfModel/imageModel values, default provider models, and auth-backed candidates.
