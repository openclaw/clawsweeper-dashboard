## Summary

Keep open: the trace-hook direction is plausible, but this PR is not ready because CI shows protocol/test type failures and the requester-run-id completion handoff is only partly wired.

## What This Changes

The PR adds a typed `agent_start` plugin hook, emits it before `llm_input`, and threads parent/requester run IDs through agent, subagent, Gateway, and embedded-runner paths.

## Best Possible Solution

Add the hook as an additive typed plugin lifecycle seam, but finish end-to-end run-id propagation through child spawn and completion handoff, refresh generated protocol artifacts, and prove it with focused coverage plus a real runtime trace/log.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source and CI evidence are high-confidence: current main lacks `agent_start`, and the PRâ€™s protocol/type failures reproduce in the published CI logs. I did not run a live Opik/subagent scenario locally.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the API direction is reasonable, but the patch leaves the requester-run-id completion path incomplete and generated/type surfaces failing. The safer path is to wire the existing `spawnedByRunId` contract end to end and regenerate/check protocol artifacts.
