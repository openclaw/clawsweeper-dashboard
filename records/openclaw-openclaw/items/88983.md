## Summary

Keep open: the PR targets a real auto-update bug, but the patch does not actually clear the inherited gateway service markers because `resolveCommandEnv` ignores `undefined` override values. The external real behavior proof gate is also still unmet.

## What This Changes

The PR attempts to clear gateway service marker env vars from the startup auto-update child command and expands the update-startup test expectation.

## Best Possible Solution

Strip the gateway service marker keys from the actual base environment passed to `runCommandWithTimeout` or change `resolveCommandEnv` deletion semantics with focused coverage, while preserving `OPENCLAW_AUTO_UPDATE=1` and proving the service-like update path.

## Reproduction Assessment

Yes, source-backed: the service environment sets the markers, the startup updater spawns through `runCommandWithTimeout`, and the update guard blocks when those markers remain. I did not run a live launchd/npm auto-update path in this read-only review.

## Solution Assessment

No. The fix belongs in the auto-update child env construction, but `undefined` overrides are not the current deletion contract; the patch should sanitize the inherited base env or deliberately update the command-env merge semantics with regression coverage.
