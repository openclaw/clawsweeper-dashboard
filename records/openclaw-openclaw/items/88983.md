## Summary

Keep open: the branch targets a real startup auto-update bug, but the patch does not actually remove inherited service markers because `undefined` env overrides are ignored after `process.env` is merged, and the posted proof is still only a mocked targeted test run.

## What This Changes

The PR adds env overrides intended to clear `OPENCLAW_SERVICE_MARKER` and `OPENCLAW_SERVICE_KIND` from startup auto-update child commands and updates the startup updater test expectation.

## Best Possible Solution

Sanitize the actual base environment passed to `runCommandWithTimeout`, or deliberately change `resolveCommandEnv` deletion semantics with focused coverage, while preserving `OPENCLAW_AUTO_UPDATE=1` and adding redacted real service-like update proof.

## Reproduction Assessment

Yes, source-backed: the service environment sets the markers, the startup updater spawns through `runCommandWithTimeout`, and the update guard blocks when those markers remain. I did not run a live launchd/npm auto-update path in this read-only review.

## Solution Assessment

No. The fix belongs in the auto-update child env construction, but `undefined` overrides are not the current deletion contract; the patch should sanitize inherited base env or deliberately update the command-env merge semantics with regression coverage.
