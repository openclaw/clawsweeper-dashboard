## What This Changes

The PR changes `sessions.usage` so aggregate totals and breakdowns are computed from all matched sessions while only the returned `sessions` rows are capped by `limit`.

## Best Possible Solution

Merge this narrow gateway fix after the contributor adds real behavior proof and exact-head validation confirms aggregate totals include all matched sessions while returned rows remain capped.

## Reproduction Assessment

Yes. Current main slices the discovered session list before loading usage and before aggregating totals, so a mocked or real store with more matched sessions than `limit` reproduces the under-count; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The PR uses the right owner-boundary fix by aggregating over the full matched session set while keeping `limit` scoped to returned rows, and it preserves the existing background cache path instead of adding a new config or runtime mode.
