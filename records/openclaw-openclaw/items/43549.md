## Summary

Current main still hides malformed shared session JSON by returning an empty store, while channel status does not diagnose that condition. Keep this issue open, but require maintainer judgment because the exact artifact and Telegram failure path remain unverified, and automatic reset could destroy valid session continuity; the related open recovery PR addresses backup/tmp recovery, not oversized-state diagnosis or status UX.

## What This Changes

Request detection and safe recovery for malformed or oversized persisted session state that can make Telegram appear broken.

## Best Possible Solution

Implement core-owned session-store health inspection with explicit parse, shape, and size results; expose it through doctor and relevant status output; and offer backup-first recovery only when the operator or a narrowly proven safe policy authorizes replacement.

## Reproduction Assessment

No. Current source proves malformed shared session JSON is silently treated as an empty store, but the issue lacks the exact artifact and a deterministic path from that condition to the reported Telegram outage.

## Solution Assessment

No. A Telegram-specific reset command or file handler would place shared session-store repair at the wrong boundary; the maintainable solution is core session diagnostics and backup-first doctor recovery, with channel status consuming that result where useful.
