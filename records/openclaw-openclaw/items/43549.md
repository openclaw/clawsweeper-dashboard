## Summary

Keep open. Current main still treats malformed or unreadable session-store JSON as an empty store, and Telegram/channel status does not report persisted session parse, size, or recovery diagnostics; the related open recovery PR is narrower and does not own the requested status/recovery UX.

## What This Changes

The issue asks OpenClaw to detect bad persisted Telegram session JSON in status/troubleshooting and provide a safe recovery path instead of requiring manual deletion.

## Best Possible Solution

Add non-mutating session-store health inspection for parse, schema, size, and recovery-source state; surface it through status/doctor, then add a maintainer-approved backup/reset or quarantine path for unsafe session state.

## Reproduction Assessment

No high-confidence live reproduction path is available. The report describes a real manual recovery, and current source shows malformed session-store JSON is silently treated as empty, but the exact preserved artifact/logs are missing.

## Solution Assessment

Unclear. Status/doctor diagnostics appear to be the right boundary, but automatic reset or quarantine needs maintainer policy because it can discard valid session and routing state.
