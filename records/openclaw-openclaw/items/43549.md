## Summary

Keep open: current main and v2026.5.19 still expose Telegram runtime/config/audit status only, not persisted Telegram JSON artifact health or safe reset guidance. The related generic session-store recovery PR may help a different state-loss path, but it does not close this Telegram-specific status and recovery UX gap.

## What This Changes

The issue requests Telegram status and recovery tooling that detects malformed, oversized, or poisoned persisted Telegram session state and guides a safe reset or quarantine flow.

## Best Possible Solution

Add a non-destructive Telegram state diagnostic that reports artifact path, size, parse/schema health, and explicit backup/reset guidance; define reset or quarantine policy before any automatic repair behavior.

## Reproduction Assessment

No high-confidence reproduction of the original wedge exists because the bad file path, size, and contents were not supplied. Source inspection does show current status lacks persisted-state diagnostics and invalid update-offset JSON falls back without operator guidance.

## Solution Assessment

Partly. A non-destructive status or doctor diagnostic with backup guidance is the safest first step; automatic self-heal, reset, or quarantine needs a maintainer policy decision first.
