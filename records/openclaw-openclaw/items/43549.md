## Summary

Keep open: current main still collapses malformed session stores to empty objects and Telegram status checks only runtime/config issues, so the requested status/recovery UX is not implemented. Related central session-store corruption work exists, but it does not cover the Telegram/status diagnostic surface yet.

## What This Changes

The issue asks OpenClaw to detect malformed or oversized Telegram-related session state in status output and guide operators through safe recovery instead of requiring manual deletion.

## Best Possible Solution

Add central read-only session-store health diagnostics and, if needed, a plugin/status hook so Telegram can surface the affected path, size, parse/schema state, and backup-first recovery guidance while folding generic store recovery into https://github.com/openclaw/openclaw/issues/77783 and https://github.com/openclaw/openclaw/pull/79910.

## Reproduction Assessment

No high-confidence runtime reproduction is available for the exact Telegram wedge. Source inspection does reproduce the diagnostic gap: malformed session-store JSON is swallowed as an empty store and Telegram status has no persisted-state validation path.

## Solution Assessment

Unclear. A non-destructive status/doctor diagnostic is the narrow maintainable first step, while automatic quarantine or a reset command needs maintainer policy because it can discard local session continuity.
