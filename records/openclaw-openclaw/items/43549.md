## Summary

Keep open. Current main still silently treats malformed session-store JSON as an empty store in the status/doctor paths, and the related recovery PR only covers a stale temp-artifact subset rather than Telegram-facing diagnosis or safe reset UX.

## What This Changes

The issue asks for Telegram status and recovery tooling to detect bad persisted Telegram session JSON and guide or perform safe reset/rebuild.

## Best Possible Solution

Add non-destructive session-store health diagnostics first, then add reset/quarantine only after maintainership settles backup and recovery semantics.

## Reproduction Assessment

Current source gives a medium-confidence reproduction path: malformed `sessions.json` is caught by `loadSessionStore` and surfaced to status/doctor as an empty store rather than as a diagnostic. A live Telegram wedge was not reproduced in this read-only review, so the exact bad artifact remains unconfirmed.

## Solution Assessment

No, not as an automated narrow fix yet. The safest direction is non-destructive status/doctor diagnostics first; reset or auto-quarantine needs maintainer decisions about backup, loss of continuity, and channel restart semantics.
