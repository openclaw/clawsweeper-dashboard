## Summary

Keep open. Current main has partial defenses around Telegram polling offsets and generic session-file repair, but I did not find the requested operator-facing status diagnosis for an oversized/malformed Telegram state artifact or a safe first-class recovery flow; the linked Keep My Claw ClawHub skill is a useful backup safety net, but it does not replace status/recovery UX in OpenClaw.

## What This Changes

The issue asks Telegram status and recovery tooling to detect malformed or oversized persisted session JSON and guide safe reset or self-healing.

## Best Possible Solution

Add a non-destructive Telegram/channel-state diagnostic that identifies the exact persisted artifact, reports size/parse/schema health, and points to an explicit backup-and-reset path; use generic session reset/doctor machinery when the artifact is an agent transcript rather than Telegram-owned polling state.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the original wedge, because the report lacks the exact corrupted file and contents; source inspection does confirm that current Telegram status does not perform the requested persisted-state file health checks.

## Solution Assessment

Is this the best way to solve the issue? The status/recovery direction is reasonable, but the safest first step is a non-destructive diagnostic plus explicit backup reset, not automatic state deletion without a maintainer decision.
