## Summary

Keep open: current main has partial Codex auth safeguards, but source inspection still shows stored session model/auth overrides can outrank changed defaults and survive `/new`, which matches the reported Telegram behavior closely enough to need a targeted fix.

## What This Changes

Telegram Codex sessions should stop staying pinned to stale OpenAI API-key auth when Codex OAuth or agent defaults have been reconfigured after an upgrade.

## Best Possible Solution

Add a narrow session or doctor repair that clears invalid/auto stale OpenAI API-key route pins for Codex sessions after config/auth changes while preserving deliberate user-selected direct OpenAI API sessions.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only pass. Source inspection does show a clear current path where stored session model/auth overrides can keep winning over changed defaults and survive `/new`.

## Solution Assessment

No, changing Telegram itself or forcing `openai-codex/*` is not the best fix because current docs make native Codex `openai/*` plus the Codex runtime. The safer fix is targeted session/doctor cleanup for stale pins and regression coverage around Codex runtime auth.
