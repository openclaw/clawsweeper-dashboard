## Summary

Keep open: current main still has the reported Control UI leak path, but this PR is dirty against main and the patch has two blocking correctness issues around history limiting and over-broad marker stripping.

## What This Changes

The PR changes Control UI chat rendering to hide tool/function/tool-result transcript plumbing and strip external-content security markers from extracted message text.

## Best Possible Solution

Port the useful behavior onto the current chat builder and extraction modules, filter non-renderable tool plumbing before applying the render limit, preserve live tool cards and mixed assistant content, and scope marker stripping to confirmed tool-result or wrapped external-content payloads with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass `buildChatItems` a history message that normalizes to `toolResult` with `showToolCalls: true`, and current main does not skip it; `extractText` also lacks any marker-stripping path.

## Solution Assessment

No. The intended behavior is the right narrow fix, but this patch is stale against current main and should be adjusted to filter before limiting history and avoid global redaction of normal chat text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Apply the history limit after filtering tool plumbing:** `ui/src/ui/views/chat.ts:1440-1441`
  - body: `historyStart` is still calculated from raw history before the new unconditional `continue`. In tool-heavy sessions, the UI can render far fewer than `CHAT_HISTORY_RENDER_LIMIT` visible user/assistant turns even though older visible turns would fit; filter non-renderable plumbing before applying the display limit or count visible messages when limiting.
  - confidence: 0.88
- **[P2] Scope external-marker redaction to tool-result paths:** `ui/src/ui/chat/message-extract.ts:42`
  - body: `processMessageText` now strips `SECURITY NOTICE` and `EXTERNAL_UNTRUSTED_CONTENT` markers for every role. Normal user or assistant messages that discuss those strings can be silently changed or become empty, so the sanitizer should be limited to confirmed tool-result or explicitly wrapped external-content payloads.
  - confidence: 0.86
