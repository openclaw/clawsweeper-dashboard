## Summary

Keep open: current main still has the Control UI leak path, but this PR is not merge-ready because it filters after the raw history limit, globally redacts marker text, and lacks external-contributor real behavior proof.

## What This Changes

The PR changes Control UI chat rendering to filter tool/function/tool-result history messages and strip external-content security markers from extracted message text.

## Best Possible Solution

A refreshed fix should filter non-renderable tool plumbing before history limiting in `build-chat-items.ts`, keep live tool cards, scope marker cleanup to confirmed external/tool payloads, and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass `buildChatItems` a history message that normalizes to `toolResult` with `showToolCalls: true`, and current main still does not skip it; `extractText` also lacks external marker stripping.

## Solution Assessment

No. The intended fix direction is right, but this patch filters after the raw history limit and applies marker stripping globally instead of limiting it to confirmed external/tool-result payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter tool plumbing before applying the history limit:** `ui/src/ui/views/chat.ts:1439-1441`
  - body: `historyStart` is calculated from raw history before the new unconditional filter. In tool-heavy sessions, dropping tool/toolResult entries after slicing can leave the UI showing far fewer real chat turns than the configured limit even though older user/assistant turns would fit.
