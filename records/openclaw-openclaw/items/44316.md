## Summary

Keep open: the PR targets a real Web UI leak, but it is not safe to merge as-is because current main has moved chat item construction into `ui/src/ui/chat/build-chat-items.ts` with a separate tool-call toggle, and the diff still has concrete regressions in redaction scope and history limiting.

## What This Changes

The PR filters tool-role/tool-call-only chat messages and strips `EXTERNAL_UNTRUSTED_CONTENT` and `SECURITY NOTICE` markers during Web UI text extraction.

## Best Possible Solution

Rework the fix on current main so normal chat hides tool plumbing, explicit tool-call inspection remains usable, and external-content/security markers are sanitized only on known tool-result or wrapped-payload display paths.

## Reproduction Assessment

Yes, from source inspection. A current-main history entry normalized as `toolResult` is included when `showToolCalls` is true, and expanded tool cards render raw `outputText`, so a wrapped `web_fetch` payload can expose the marker text without running the UI.

## Solution Assessment

No. The PR fixes part of the leak, but unconditional message filtering and global marker stripping are broader than the safest current-main solution; the fix should sanitize tool-result render paths and apply visible-history limits after filtering.

## Review Findings
