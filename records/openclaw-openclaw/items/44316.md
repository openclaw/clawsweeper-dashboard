## Summary

Keep open: this PR addresses a plausible Web UI leak, but the branch conflicts with current main, lacks real after-fix behavior proof, and still introduces source-level regressions around global redaction and render-limit ordering.

## What This Changes

The PR filters tool/tool-result/function chat records regardless of thinking visibility and strips external-content/security-notice markers in Web UI message extraction.

## Best Possible Solution

Rebase onto current main, implement the filtering in the current build-chat-items/message-extract paths with scoped marker stripping and filtering before the render cap, then add focused UI regression coverage plus real after-fix proof.

## Reproduction Assessment

Partially. The introduced regressions are source-reproducible from the PR head: marker stripping runs for every role and the render cap is computed before the new tool-message filter, but the original Web UI leak has no attached real-session proof.

## Solution Assessment

No. The direction is reasonable, but the current patch is overbroad, applies filtering after the history window is chosen, and targets a now-stale render path instead of the current main implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Scope marker stripping to tool-originated text:** `ui/src/ui/chat/message-extract.ts:42`
  - body: `extractText` now runs `stripExternalContentMarkers` for every role. A normal user or assistant message that discusses `SECURITY NOTICE:` or `EXTERNAL_UNTRUSTED_CONTENT` literally will be silently redacted or become empty, which changes visible chat history, search, and markdown export outside the tool-result leak this patch is trying to fix. Keep this stripping on known tool-result/wrapped-payload paths instead of all message text.
  - confidence: 0.84
- **[P2] Filter tool plumbing before applying the render cap:** `ui/src/ui/views/chat.ts:1440-1441`
  - body: The new unconditional `continue` runs after `historyStart` is computed from the raw history length. In a long session where the last 100 records include many tool/toolResult entries, those entries are dropped after they have already consumed the render window, so older visible user/assistant turns disappear even though they would fit. Filter tool plumbing before calculating the capped history window.
  - confidence: 0.79
