## Label Justifications

- `P1`: A deterministic current-main defect sends real users a hard-failure notice while an active subagent workflow continues successfully.
- `impact:message-loss`: The requester session and channel receive a false terminal notification that misrepresents the background work.
- `impact:session-state`: The task ledger becomes terminal and cancelled while the yielded subagent session remains active and resumable.

## Summary

Current main still violates the documented task-state contract: a cooperative `sessions_yield` is emitted with an explicit yield marker, but the task registry disregards it, finalizes the task as cancelled, and consequently sends a false cancellation notice. This issue is the only exact live report found and should remain open for a focused repair.

## What This Changes

Keep a depth-1 subagent’s background task non-terminal when it calls `sessions_yield` while awaiting descendants, and avoid sending a cancellation notice unless genuine cancellation occurred.

## Best Possible Solution

Treat lifecycle ends carrying `yielded: true` as cooperative, non-terminal task transitions, retain the running task record until the resumed subagent emits its actual terminal event, and add regression coverage proving no premature delivery and exactly one eventual settlement.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence deterministic path: yield emits `yielded: true` with abort metadata, task projection ignores the yield fact, and the abort maps to cancelled and terminal delivery.

## Solution Assessment

Yes. Repairing the generic lifecycle-to-task projection is narrower and more correct than suppressing the downstream message, changing WhatsApp behavior, or adding a new configuration option.
