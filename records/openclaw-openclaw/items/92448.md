## Label Justifications

- `P1`: A healthy user-requested multi-agent workflow is falsely reported as cancelled in the user channel, causing immediate confusion and wasted recovery turns.
- `impact:message-loss`: The requester receives a materially incorrect terminal cancellation message instead of accurate task state.
- `impact:session-state`: The task ledger becomes terminal and cancelled while the corresponding yielded subagent remains active and later succeeds.

## Summary

Current main still has the source-level conditions for this report: `sessions_yield` is normalized as non-terminal wait control flow, yet `cancelled` is the exceptional subagent task status that generates the exact requester-visible notice. The issue is a valid, bounded lifecycle bug and no related item currently owns this specific false-cancellation behavior.

## What This Changes

Keep a yielded depth-1 subagent’s background task non-terminal while descendants are active and prevent false cancellation notices to the requester.

## Best Possible Solution

Make detached subagent task settlement consume the canonical yielded/paused lifecycle state, keeping the task active until actual success, failure, timeout, or explicit cancellation and delivering cancellation text only for genuine cancellation.

## Reproduction Assessment

Yes, at high source confidence. The report supplies two deterministic runtime occurrences, and current main still combines internal yield-abort control flow with the cancelled-only subagent delivery path that produces the exact notice, although this review did not execute a live run.

## Solution Assessment

Yes. Repairing detached-task settlement at the subagent lifecycle boundary is the narrowest maintainable fix; suppressing or rewording the notice alone would leave persisted task state incorrect.
