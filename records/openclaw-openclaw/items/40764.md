## Summary

Keep this PR open, but not merge-ready. Current main still lacks an Android talk-mode language override, and the PR fixes the earlier TTS cache concern, but its latest head still introduces a user-visible false failure when the setting is changed before a gateway endpoint is cached.

## What This Changes

The PR adds an Android language override setting and threads it through preferences, STT intents, system TTS, device locale metadata, and Gateway connect options.

## Best Possible Solution

Adapt the language override onto the current Android voice stack, make offline preference changes status-neutral, cover the locale/connect behavior with focused Android tests, and add the required changelog entry.

## Reproduction Assessment

Yes for the review blocker. On the PR head, changing the Language setting before any successful connection or after Disconnect calls refreshGatewayConnection(), which writes the false "Failed: no cached gateway endpoint" status when connectedEndpoint is null.

## Solution Assessment

No, not as-is. The feature direction is reasonable, but the reconnect/status path should be guarded or split so preference persistence is independent from an available cached endpoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid reporting a gateway failure for offline language changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:640`
  - body: `setLanguageOverride()` always calls `refreshGatewayConnection()` after saving the preference. When the app has no cached endpoint, `refreshGatewayConnection()` sets the status to `Failed: no cached gateway endpoint`, so selecting a language before connecting or after Disconnect creates a false user-visible failure even though no reconnect can happen. Gate the refresh on a cached endpoint or make the disconnected refresh path a no-op.
  - confidence: 0.9

  - body: This is a user-facing Android feature, but the PR does not include a `CHANGELOG.md` entry. Repo policy requires user-facing `feat` changes to be recorded under the active version, with appropriate contributor credit.
  - confidence: 0.87
