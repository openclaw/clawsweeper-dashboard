## Summary

Keep this PR open, but not merge-ready. Current main still has no Android talk-mode language override, while the PR’s latest head addresses the earlier cached-TTS-language review but still has a user-visible offline status regression and lacks the required changelog entry.

## What This Changes

Adds an Android Settings language dropdown persisted as `node.languageOverride` and threads it into gateway locale, device info, speech recognition, and system TTS for talk and voice capture.

## Best Possible Solution

Land a rebased or replacement Android-only implementation that persists the override, applies it to STT/TTS and gateway/device locale payloads, leaves offline status unchanged when no reconnect is possible, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. Current-main verification shows no language override setting or preference, and the PR regression is reproducible by changing the new language setting before any cached endpoint exists or after disconnect, which routes into the existing `Failed: no cached gateway endpoint` branch.

## Solution Assessment

No, not as-is. The feature direction is maintainable, but the implementation should refresh desired connect options only when a cached endpoint exists, avoid false offline failures, add changelog coverage, and be rebased onto current Android main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid false failures when changing language offline:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:640`
  - body: Changing the new language setting always calls `refreshGatewayConnection()`. When the app has not connected yet, or after the user disconnects, that method has no cached endpoint and sets the visible status to `Failed: no cached gateway endpoint`, so a Settings-only change can put the app into a bogus failure state.
