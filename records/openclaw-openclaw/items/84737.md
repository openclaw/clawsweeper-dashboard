## Summary

Keep this PR open: the fix is narrowly aimed and has good default-config proof, but it still allows a documented Git prune-tags configuration to reintroduce the same moved-tag failure during dev updates.

## What This Changes

The PR changes dev-channel updates to fetch with `--no-tags`, adds targeted fetching for explicit dev tag refs, and adds focused update-runner tests.

## Best Possible Solution

Keep the focused update-runner direction, but make dev branch fetches opt out of both tag following and prune-tag refspecs while leaving stable/beta broad tag discovery unchanged.

## Reproduction Assessment

Yes. Current main uses the broad dev fetch command, the PR body proves the moved-tag failure class, and a scratch Git repro shows the PR command still fails when `fetch.pruneTags=true`.

## Solution Assessment

No. The direction is right, but the implementation should also suppress prune-tag refspecs, for example with `--no-prune-tags`, and add coverage for that documented Git configuration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Disable prune-tag refspecs for dev fetches:** `src/infra/update-runner.ts:844`
  - body: `--no-tags` only disables automatic tag following. With `fetch.pruneTags=true`, `git fetch --all --prune --no-tags` still fetches tag refspecs and reproduces the same `would clobber existing tag` failure this PR is meant to avoid. Add `--no-prune-tags` or an equivalent config override and cover that case in the update-runner tests.
  - confidence: 0.92
