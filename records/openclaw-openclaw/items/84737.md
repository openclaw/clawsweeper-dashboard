## What This Changes

The PR changes the git update runner and tests so dev-channel updates fetch branches with `--no-tags`, while explicit dev tag targets fetch only the requested tag ref.

## Best Possible Solution

Land this after redacted real behavior proof shows a dev update or local git-remote reproducer with a conflicting moved tag succeeds without broad tag fetching, while stable/beta tag-driven updates remain unchanged.

## Reproduction Assessment

Yes, source inspection plus the Git fetch contract gives a high-confidence reproduction path: a dev update on current main runs a broad `--tags` fetch before rebasing, so a conflicting remote tag can fail before the branch update continues. I did not run a live mutable update in this read-only review.

## Solution Assessment

Yes, the proposed fix is the narrow maintainable direction: make the branch-based dev fetch opt out of broad tag fetching and fetch only an explicitly requested dev tag before resolving it. The remaining gap is proof quality, not a known code-shape defect.
