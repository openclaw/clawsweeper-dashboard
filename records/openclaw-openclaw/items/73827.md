## Summary

Keep open. Current main still ignores Slack `<!subteam^...>` user-group mention tokens in the Slack plugin mention preflight, and the linked config-only PR was closed unmerged after maintainer feedback favoring an API-backed Slack `usergroups.users.list` membership check.

## What This Changes

Slack agents should treat Slack user-group mentions as explicit mentions when the bot user is a member of the mentioned user group.

## Best Possible Solution

Implement this in the Slack plugin as an API-backed membership check: extract Slack subteam tokens, call `ctx.app.client.usergroups.users.list` with bounded caching, treat the message as explicit only when `ctx.botUserId` appears in the returned user IDs, and update Slack setup docs, manifest scopes, tests, and changelog. The config-only allowlist from the closed PR should not be the primary behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with message text `<!subteam^S...> hello`, current main sets `hasAnyMention=false` and `explicitlyMentioned=false`, so a Slack room with `requireMention=true` reaches the no-mention skip path unless a separate mention regex happens to match.

## Solution Assessment

Yes. The maintainer-directed API-backed membership path is the narrowest maintainable fix because it avoids responding to user-group mentions unless Slack says the bot user is actually a member; the closed config-only PR is useful context but not the preferred solution.
