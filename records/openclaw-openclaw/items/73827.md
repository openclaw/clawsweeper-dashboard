## Summary

Keep open. Current main still only recognizes direct Slack `<@botUserId>` mentions, app_mention events, configured mention regexes, and implicit thread participation; it does not recognize `<!subteam^...>` user-group mentions, and the related config-only PR was closed unmerged after maintainer feedback favoring an API-backed membership check.

## What This Changes

Request that Slack channel mention gating wake agents when a Slack user-group mention targets a group containing their bot user.

## Best Possible Solution

Implement this inside the Slack plugin as an API-backed membership path: extract `<!subteam^S...>` tokens, resolve mentioned user groups with Slack `usergroups.users.list`, treat membership of `ctx.botUserId` as an explicit mention, cache and invalidate conservatively, and update Slack setup docs/scopes plus regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: in a `requireMention` Slack channel, text like `<!subteam^S123> hello` leaves both direct-mention facts false unless a separate regex matches, so current main reaches the `no-mention` skip path.

## Solution Assessment

Yes for the API-backed membership direction; no for the closed config-only allowlist as the primary fix. The membership check is the narrow maintainable path because it wakes only bots Slack reports as actual members of the mentioned user group.
