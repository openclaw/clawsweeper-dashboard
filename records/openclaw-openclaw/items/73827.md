## Summary

Keep open. Current main still only recognizes direct Slack `<@botUserId>` mentions, app_mention events, implicit thread participation, and configured mention regexes; it does not recognize `<!subteam^...>` Slack user-group tokens, and the related config-only PR was closed unmerged after maintainer feedback favoring an API-backed membership check.

## What This Changes

Slack user-group mentions should wake OpenClaw Slack agents that are members of the mentioned subteam instead of being dropped by mention gating.

## Best Possible Solution

Implement the fix inside the Slack plugin as an API-backed membership path: extract `<!subteam^S...>` tokens, call Slack `usergroups.users.list` for mentioned user groups, treat membership of `ctx.botUserId` as an explicit mention, cache and invalidate conservatively, and update Slack docs/setup scopes plus regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: in a `requireMention` Slack channel, text like `<!subteam^S123> hello` leaves both direct-mention facts false unless a separate regex matches, so current main reaches the `no-mention` skip path.

## Solution Assessment

Yes for the API-backed membership direction; no for the closed config-only allowlist as the primary fix. The membership check is the narrow maintainable path because it wakes only bots that Slack says are actually in the mentioned user group.
