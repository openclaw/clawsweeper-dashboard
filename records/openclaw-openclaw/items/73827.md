## Summary

Keep open. Current main still only recognizes direct Slack bot mentions, `app_mention` events, regex mention patterns, and implicit thread participation; it does not recognize Slack user-group `<!subteam^S...>` mentions, and the related config-only PR was closed unmerged after maintainer feedback favoring an API-backed membership check.

## What This Changes

The issue asks Slack channel mention gating to treat Slack user-group `<!subteam^S...>` mentions as wake signals for OpenClaw agents that are members of the referenced Slack user group.

## Best Possible Solution

Add a Slack-plugin-owned user-group membership mention path: parse subteam tokens, resolve membership through Slack Web API with bounded caching and fail-closed handling, update the Slack setup manifest/docs/tests/changelog, and keep shared core unchanged unless a generic seam is clearly needed.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: in a mention-gated Slack room, text like `<!subteam^S123> ping` does not match the current `<@...>` explicit mention checks, so `wasMentioned` remains false and the `no-mention` skip path is reached.

## Solution Assessment

Yes for the maintainer-directed API-backed fix; no for the closed config-only allowlist as the primary solution. Checking Slack membership via `usergroups.users.list` is narrower and safer because an agent wakes only when its bot user is actually in the referenced Slack user group.
