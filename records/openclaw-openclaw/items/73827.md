## Summary

Keep open. Current main still does not recognize Slack user-group `<!subteam^...>` mentions, so a mention-gated Slack channel can drop the reported message as `no-mention`; the related config-only PR was closed unmerged after maintainer feedback favoring an API-backed membership check.

## What This Changes

The issue requests that Slack user-group mentions wake OpenClaw agents that are members of the mentioned Slack user group.

## Best Possible Solution

Implement this inside the Slack plugin as an API-backed mention path: extract `<!subteam^S...>` tokens, resolve user-group members with Slack `usergroups.users.list`, treat membership of `ctx.botUserId` as an explicit mention, and update scopes, docs, tests, and changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: in a Slack room with `requireMention`, a message like `<!subteam^S123> hello` leaves `explicitlyMentioned` false, has no default regex match, and reaches the `no-mention` skip path.

## Solution Assessment

Yes for the maintainer-preferred API-backed membership direction; no for the closed config-only allowlist as the primary fix. The API-backed path is the narrow maintainable solution because it wakes only bots Slack reports as actual members of the mentioned user group.

## Repair Work Prompt

Fix the Slack user-group mention issue at https://github.com/openclaw/openclaw/issues/73827 in the Slack plugin. Current main drops Slack user-group mentions like `<!subteam^S123> hello` in `requireMention` channels because only direct `<@botUserId>` mentions, app_mention events, configured mention regexes, and implicit thread participation are recognized. Do not add a per-agent `groupChat.subteamMentions` config allowlist as the primary fix, do not move Slack-owned behavior into core beyond existing generic mention seams, and do not change unrelated channel behavior.
