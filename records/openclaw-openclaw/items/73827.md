## Summary

Keep open. Current main still only treats direct Slack `<@botUserId>` tokens, `app_mention` events, implicit thread participation, or configured mention regexes as mention signals; `<!subteam^S...>` user-group mentions are not recognized, and the related config-only PR was closed unmerged after maintainer feedback favoring an API-backed membership check.

## What This Changes

Request Slack channel messages containing `<!subteam^S...>` user-group mentions to wake OpenClaw agents whose bot users are members of the mentioned Slack user group.

## Best Possible Solution

Implement the fix inside the Slack plugin as an API-backed membership path: extract `<!subteam^S...>` tokens, use Slack `usergroups.users.list` to check whether `ctx.botUserId` is a member, cache bounded results, treat membership as an explicit mention, and update Slack setup docs, manifest scopes, tests, and changelog. Do not make the closed config-only allowlist the primary behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with text like `<!subteam^S123> hello` in a Slack room where mention gating is required, current main sets the direct mention facts false unless a separate regex matches, so the handler reaches the `no-mention` skip path.

## Solution Assessment

Yes for the remaining API-backed direction; no for the closed config-only proposal as the primary fix. The Slack membership check is the narrowest maintainable solution because it wakes only bots that are actually in the Slack user group.
