## Summary

Keep open. Current main still recognizes Slack direct bot mentions, app_mention events, configured mention regexes, and implicit thread participation, but not Slack user-group `<!subteam^S...>` tokens; the linked config-only PR was closed unmerged after maintainer feedback favored an API-backed membership check.

## What This Changes

The issue asks the Slack plugin to treat `<!subteam^S...>` user-group mentions as explicit mention signals for agents that belong to the mentioned Slack user group.

## Best Possible Solution

Implement the fix inside the Slack plugin as an API-backed membership path: extract `<!subteam^S...>` tokens, use Slack `usergroups.users.list` to determine whether `ctx.botUserId` is a member, cache bounded results with safe invalidation, treat confirmed membership as an explicit mention, and update Slack setup docs, manifest scopes, tests, and changelog. The closed config-only allowlist should not be the primary behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `<!subteam^S123> hello` in a Slack room where `requireMention` is true, current main sets the direct mention facts false unless a separate regex matches, so the handler reaches the `no-mention` skip path.

## Solution Assessment

Yes for the remaining API-backed direction; no for the closed config-only proposal as the primary fix. Checking Slack user-group membership is the narrower maintainable behavior because it wakes only bots that Slack says are members of the mentioned group.
