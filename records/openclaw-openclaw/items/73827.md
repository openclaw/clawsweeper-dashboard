## Summary

Keep open. Current main still treats Slack explicit mentions as direct `<@botUserId>` or `app_mention` signals plus configured mention regexes, with no Slack user-group/subteam path; the paired config-only PR was closed unmerged after maintainer feedback preferring an API-backed membership check.

## What This Changes

Slack user-group mentions like `<!subteam^S...>` should wake OpenClaw agents that are members of the mentioned Slack user group.

## Best Possible Solution

Implement the Slack plugin fix as the maintainer-preferred API-backed membership path: parse `<!subteam^S...>` tokens, check whether the current bot user is in each mentioned user group via Slack `usergroups.users.list`, treat confirmed membership as an explicit mention, and update Slack setup scopes/docs/tests/changelog. The closed config-only allowlist can remain a rejected alternative rather than the primary product behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with only `<!subteam^S123>` in a mention-gated Slack channel, current main never sets the explicit mention signal for the bot, so the room mention gate reaches the `no-mention` skip path.

## Solution Assessment

Yes for the API-backed direction, no for the closed config-only PR as the primary fix. Checking Slack membership is the narrowest maintainable behavior because it wakes only bots that are actually in the mentioned Slack user group.
