## Summary

Keep open. Current main still only treats direct Slack `<@botUserId>` tokens, `app_mention` events, implicit thread participation, and configured mention regexes as mention signals; it does not recognize Slack `<!subteam^...>` user-group mentions. The linked config-only PR was closed unmerged after maintainer feedback favoring an API-backed membership check, so this remains a narrow Slack plugin bug rather than a cleanup-close candidate.

## What This Changes

Slack user-group mentions should wake the OpenClaw agents that are actual members of the mentioned Slack user group.

## Best Possible Solution

Implement the fix inside the Slack plugin: extract `<!subteam^S...>` tokens, resolve each mentioned user group through Slack membership data, treat `ctx.botUserId` membership as an explicit mention, cache conservatively with subteam event invalidation or a short TTL, and update Slack setup/docs/tests/changelog without adding a config-only allowlist as the primary behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: in a Slack channel with `requireMention`, text like `<!subteam^S123> hello` leaves the direct mention facts false unless a separate regex happens to match, so current main reaches the `no-mention` skip path.

## Solution Assessment

Yes for the API-backed membership direction; no for the closed config-only allowlist as the primary fix. The membership check is the narrow maintainable path because it wakes only bots Slack reports as members of the mentioned user group.
