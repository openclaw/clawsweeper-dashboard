## Summary

Keep open: the shared dispatch boundary is the right area, but the patch still misses the operator-visible diagnostic requested by the linked reports and can misclassify healthy message-tool turns as violations.

## What This Changes

The PR adds a delivery-policy-violation helper, logs it from suppressed source-reply dispatch, adds unit tests, and updates the changelog.

## Best Possible Solution

Preserve the tool-only privacy contract, but emit a default-visible diagnostic only when a suppressed substantive final has no committed message-tool delivery evidence.

## Reproduction Assessment

Yes, source-reproducible: current main resolves group/channel rooms to message-tool-only delivery and suppresses ordinary finals, while the linked Telegram and Slack reports provide live examples of successful turns with no visible room reply. I did not run a live Telegram or Slack reproduction during this read-only review.

## Solution Assessment

No. The shared dispatch path is a reasonable place to surface the condition, but the patch should use committed message-tool delivery evidence and a default-visible warning or diagnostic event instead of a verbose-only payload-local log.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate the violation on absent message-tool delivery:** `src/auto-reply/reply/delivery-policy-violation.ts:10-15`
  - body: The new evaluator flags every suppressed non-empty final in `message_tool_only` mode, but the reported failure is specifically final text with no successful message-tool delivery. Healthy turns can still produce a private final summary after the message tool already sent the visible reply, so this would log false `delivery-policy-violation` lines for working behavior; pass committed messaging-tool delivery evidence into the diagnostic and return null when it exists.
  - confidence: 0.87
- **[P2] Emit the diagnostic outside verbose-only logging:** `src/auto-reply/reply/dispatch-from-config.ts:1579-1580`
  - body: The new signal goes through `logVerbose`, which only appears in file logs at debug/trace or with verbose console mode. The linked reports describe normal operator post-incident log inspection and explicitly ask for a non-verbose or hard diagnostic, so default installations will still have no grepable signal after the invisible turn.
  - confidence: 0.9
