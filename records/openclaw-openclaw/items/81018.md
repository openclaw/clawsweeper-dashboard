## Summary

Keep open: this is a focused diagnostic PR with supplied terminal proof and green CI, but the patch has a blocking false-positive path. The new suppress-branch check cannot tell whether the message tool already delivered, so successful tool-only turns that end with a private final summary would be logged as `delivery-policy-violation`.

## What This Changes

Adds a delivery-policy-violation helper and tests, calls it from the shared reply-dispatch suppression branch, and adds a changelog entry for logging suppressed `message_tool_only` final text.

## Best Possible Solution

Keep the shared diagnostic, but gate it on actual missing message-tool delivery while preserving the existing private-final contract; leave auto-repair or stronger log-level policy for a separate maintainer decision.

## Reproduction Assessment

Yes, at source level: current main resolves group/channel `message_tool` mode to suppressed automatic delivery and linked reports provide live Telegram/Slack examples. I did not run a live Telegram or Slack end-to-end reproduction in this read-only review.

## Solution Assessment

No. A diagnostic-only fix is a reasonable narrow direction, but this implementation should be gated on no successful message-tool delivery or moved to the runner boundary where `didSendViaMessagingTool` is available.
