## Summary

Keep open. The PR targets the right shared dispatch boundary, but the new signal is still emitted through verbose/debug-only logging, which does not satisfy the linked reports' request for a default-visible diagnostic when a successful turn produces no room-visible reply.

## What This Changes

The PR adds a delivery-policy-violation evaluator/formatter, logs that condition from the source-reply suppression branch, adds unit coverage, and updates the changelog.

## Best Possible Solution

Preserve the tool-only privacy contract, but surface this exact suppressed-final-text condition through a default-visible warning or diagnostic event with focused regression coverage.

## Reproduction Assessment

Yes, at source level: group/channel `message_tool_only` mode suppresses ordinary final delivery, and linked Telegram/Slack reports provide live examples where the run completed but no visible room reply was sent. I did not run a live Telegram or Slack reproduction during this read-only review.

## Solution Assessment

No. The shared dispatch boundary is the right place, but a verbose-only log does not solve the operator-visible diagnostic gap described by the reports; a non-verbose warning or structured diagnostic event is the narrower maintainable fix.
