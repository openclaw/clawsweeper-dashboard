## Summary

Keep open: this is a focused bug-fix PR for a real config validation failure, but the patch appears to downgrade blocked official memory-slot plugins to warnings and the PR has a protected maintainer label, so it should be repaired/reviewed rather than cleaned up.

## What This Changes

The PR changes config validation so a missing cataloged `memory-lancedb` slot warns with the official install hint, adds regression coverage, and updates the LanceDB memory plugin docs to require installation first.

## Best Possible Solution

Land a narrow validator/docs fix that soft-fails truly missing cataloged `memory-lancedb` installs while keeping unknown and blocked memory-slot IDs as hard validation errors.

## Reproduction Assessment

Yes. The current-main source path is clear: `validateConfigObjectWithPlugins()` with `plugins.slots.memory = "memory-lancedb"` and an empty manifest registry reaches the hard `pushMissingPluginIssue` call for the memory slot.

## Solution Assessment

No. The warning direction is the right narrow fix, but this implementation should not let blocked official external memory-slot candidates inherit the missing-install warning path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve hard failures for blocked memory slot plugins:** `src/config/validation.ts:1557-1559`
  - body: This sets `warnOnly` from catalog membership before `pushMissingPluginIssue` checks blocked plugin diagnostics. For an official external memory plugin that is present but blocked by the registry, such as a world-writable `memory-lancedb` candidate, the explicit memory slot now becomes `ok: true` with only a warning instead of the hard validation failure used for unsafe or unusable slot owners. Only downgrade the no-diagnostic missing official install case.
  - confidence: 0.86
