## Summary

Keep open: current main and the latest release still lack a binary-document guard for filesystem `read`, but this draft PR is not merge-ready because it is conflicting, lacks contributor real behavior proof, and adds full-buffer MIME sniffing outside the read-size budget.

## What This Changes

The PR adds MIME-based PDF/Office detection to the agent read wrapper, returns fallback text for detected binary documents, wires host/sandbox buffer reads, and adds a DOCX fallback regression test.

## Best Possible Solution

Rebase or replace this with one bounded read-path guard that prevents binary token waste, preserves read budgets, and coordinates parser-backed extraction with the document-extractor contract.

## Reproduction Assessment

Yes. Current main delegates non-image reads to the upstream read tool, which converts non-image buffers to UTF-8, and the PR's unbounded follow-up read is visible at `loadBuffer`/`fs.readFile`; I did not run a live binary read because this review is read-only.

## Solution Assessment

No, not as-is. The fallback direction is useful, but the implementation should use bounded detection and should be coordinated with the open document-extraction work before merge.
