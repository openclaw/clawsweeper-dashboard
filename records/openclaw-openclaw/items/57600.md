## Summary

Keep open. The PR targets a real missing document-read guard, but the patch is not mergeable as-is because MIME detection re-reads full user-controlled files outside the adaptive read cap, it lacks after-fix real behavior proof, and it overlaps the separate DOCX extraction design.

## What This Changes

The PR adds PDF/Office fallback handling to the `read` wrapper, injects host and sandbox buffer readers for MIME detection, and adds DOCX fallback regression coverage.

## Best Possible Solution

Use one canonical document-read path with bounded MIME sniffing or bounded extraction metadata, preserving adaptive read limits across host and sandbox reads and coordinating with the DOCX extraction design.

## Reproduction Assessment

Yes for the review finding. Source inspection of the PR diff shows `normalizeBinaryDocumentReadResult` awaits `readBufferForToolPath` before MIME detection, and both injected readers return full buffers.

## Solution Assessment

No. The fallback direction is useful, but this implementation should sniff only a bounded prefix or pass bounded MIME metadata through the existing read operation, with maintainer coordination alongside https://github.com/openclaw/openclaw/pull/57612.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound MIME sniffing reads:** `src/agents/pi-tools.read.ts:333`
  - body: The new normalization awaits a full buffer from `readBufferForToolPath` before calling `detectMime`, and both injected implementations read the full target file. Large PDFs or Office documents can therefore bypass the wrapper's adaptive read cap and spike memory after the normal read path already returned; sniff a bounded prefix or pass bounded MIME metadata through the read operation instead.
  - confidence: 0.86
