## Summary

Keep open: the PR targets a real token-waste problem, but the current patch introduces an unbounded full-file MIME sniff, has no after-fix real behavior proof, and overlaps the still-open DOCX extraction path that needs maintainer boundary judgment.

## What This Changes

The PR adds a PDF/Office guard to the `read` wrapper so supported binary document formats return a structured fallback instead of raw decoded text.

## Best Possible Solution

Land one bounded, maintainer-approved document-read path that prevents binary token waste without bypassing read caps, with parser-backed extraction coordinated through the document-extract/plugin contract.

## Reproduction Assessment

Yes by source inspection: current main still lets non-image `read` results flow through the upstream UTF-8 text path, and the PR review finding is visible at the added full-buffer MIME sniff call. I did not run a live file-read because this review is read-only.

## Solution Assessment

No, not as-is. The fallback direction is useful, but this implementation should sniff a bounded prefix or carry bounded MIME metadata from the existing read path, and it should be coordinated with the open DOCX extraction work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound MIME sniffing reads:** `src/agents/pi-tools.read.ts:333`
  - body: The new normalization awaits a full buffer from `readBufferForToolPath` before calling `detectMime`, and the host implementation reads the full target file. Large PDFs or Office documents can therefore bypass the wrapper's adaptive read cap and spike memory after the normal read path already returned; sniff a bounded prefix or pass bounded MIME metadata through the read operation instead.
  - confidence: 0.86

## Security Review

Status: needs_attention
