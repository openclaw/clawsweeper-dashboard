## Summary

Keep open: current main still lacks a PDF/Office document guard in `read`, the canonical linked issue remains open, and the broader DOCX extraction PR has not landed. This PR is useful directionally, but it still needs real behavior proof and a bounded MIME-sniffing implementation before merge.

## What This Changes

The PR adds PDF/Office MIME sniffing to the `read` wrapper, returns fallback text for detected binary documents, wires a host buffer reader, and adds a DOCX fallback regression test.

## Best Possible Solution

Land one bounded, maintainer-approved document-read path that prevents binary token waste without uncapped reads, with parser-backed extraction coordinated through the document-extractor/plugin contract.

## Reproduction Assessment

Yes by source inspection: current main's wrapper only normalizes images, while the pinned upstream read tool decodes non-image files as UTF-8 text. I did not run a live binary read because this cleanup review is read-only.

## Solution Assessment

No, not as-is. The fallback direction is useful, but the implementation should sniff a bounded prefix or reuse bounded MIME metadata, and parser-backed extraction should be coordinated with the open document-extraction work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound MIME sniffing reads:** `src/agents/pi-tools.read.ts:333`
  - body: `normalizeBinaryDocumentReadResult` calls `loadBuffer(filePath)` before MIME detection, and the host implementation backs that with `fs.readFile`. Large PDFs or Office documents can therefore bypass adaptive read paging and spike memory after the normal read already returned; sniff a bounded prefix or pass bounded MIME metadata through the read operation instead.
  - confidence: 0.86
