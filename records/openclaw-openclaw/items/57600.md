Keep open: the PR targets a real read-tool token-waste problem, but it is still an external draft with no real behavior proof, a main-branch conflict, and an introduced full-file MIME sniffing path that can bypass the existing read budget.

## What This Changes

This draft PR adds MIME-based PDF/Office fallback handling to the OpenClaw `read` wrapper, wires host and sandbox buffer reads, and adds a DOCX fallback regression test.

## Best Possible Solution

Land one bounded, maintainer-approved document-read path that prevents binary token waste without uncapped reads, coordinating parser-backed extraction through the existing document-extractor/plugin boundary.

## Reproduction Assessment

Yes by source inspection: current main's wrapper only normalizes images, while the pinned upstream read tool decodes non-image files as UTF-8 text. I did not run a live binary read because this review is read-only.

## Solution Assessment

No, not as-is. The fallback direction is useful, but the implementation should use bounded MIME detection or existing bounded metadata and be coordinated with the open document-extraction work.
