## Summary

Keep open for contributor and maintainer follow-up. The binary-document fallback direction is useful and current main still lacks it, but this external draft has no real behavior proof, introduces an unbounded full-file MIME sniff read, and overlaps the open DOCX extraction design in https://github.com/openclaw/openclaw/pull/57612.

## What This Changes

Adds PDF/Office MIME detection to the `read` wrapper, host/sandbox buffer readers, a structured binary-document fallback message, and DOCX fallback regression coverage.

## Best Possible Solution

Land one bounded, maintainer-approved document-read path that prevents binary token waste without bypassing read caps, with parser-backed extraction coordinated through the document-extract/plugin boundary.

## Reproduction Assessment

Yes for the PR review finding: source inspection of the PR diff shows the new normalization path fully reads the file before MIME detection. Current main inspection plus the pinned upstream read implementation also gives a high-confidence path for the original raw-binary behavior.

## Solution Assessment

No, not as-is. The safe-fallback direction is useful, but this implementation should sniff a bounded prefix or carry bounded MIME metadata from the existing read path, and it should be coordinated with the open DOCX extraction work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound MIME sniffing reads:** `src/agents/pi-tools.read.ts:333`
  - body: The new normalization awaits a full buffer from `readBufferForToolPath` before calling `detectMime`, and both injected implementations read the full target file. Large PDFs or Office documents can therefore bypass the wrapper's adaptive read cap and spike memory after the normal read path already returned; sniff a bounded prefix or pass bounded MIME metadata through the read operation instead.
  - confidence: 0.86
