## Summary

Keep open. The fallback direction is useful, but this draft is not safe to merge as-is because it overlaps the still-open DOCX extraction PR and re-reads target documents fully for MIME detection outside the existing read caps.

## What This Changes

The PR adds PDF/Office MIME fallback handling to the `read` wrapper, wires host and sandbox buffer reads for MIME detection, and adds a DOCX fallback regression test.

## Best Possible Solution

Use one canonical document-read path with bounded MIME sniffing or bounded extraction metadata, preserving adaptive read limits across host and sandbox reads and resolving how this relates to the DOCX extraction PR.

## Reproduction Assessment

Yes for the review finding. Source inspection of the PR diff shows a large PDF or Office read first goes through normal read handling and then calls a full-file buffer loader before `detectMime`.

## Solution Assessment

No. The fallback behavior is useful, but this implementation should sniff a bounded prefix or pass bounded MIME metadata through the read operation, and maintainers should choose the canonical path with the overlapping DOCX extraction PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound the MIME sniff read:** `src/agents/pi-tools.read.ts:326-327`
  - body: The new normalization awaits a full buffer from `readBufferForToolPath` before calling `detectMime`, and both host and sandbox callbacks implement that by reading the entire target file. Large PDFs or Office documents can therefore bypass the wrapper's adaptive read cap and spike memory after the normal read path already returned; sniff a bounded prefix or pass bounded MIME metadata through the read operation instead.
  - confidence: 0.86
