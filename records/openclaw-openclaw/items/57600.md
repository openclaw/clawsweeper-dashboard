## Summary

Keep open: the fallback direction is useful, but this draft still lacks real behavior proof, conflicts with current main, overlaps https://github.com/openclaw/openclaw/pull/57612, and adds an unbounded MIME-sniffing read path that can bypass the existing read budget.

## What This Changes

The PR adds MIME-based PDF and Office document detection to the OpenClaw read wrapper so matching files return a structured fallback instead of raw decoded binary text.

## Best Possible Solution

Land one bounded document-read path that prevents binary token waste without full-file sniffing, coordinated with the document-extractor/PDF tooling boundary and backed by real read proof.

## Reproduction Assessment

Yes by source inspection: current main only special-cases image handling, while the pinned upstream read tool decodes non-image files as UTF-8 text. I did not run a live binary read because this review is read-only.

## Solution Assessment

No, not as-is. The fallback direction is useful, but the implementation should sniff a bounded prefix or reuse bounded MIME metadata, and parser-backed extraction should be coordinated with the open document-extraction work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound MIME sniffing reads:** `src/agents/pi-tools.read.ts:333`
  - body: `normalizeBinaryDocumentReadResult` calls `loadBuffer(filePath)` before MIME detection, and the host implementation backs that with `fs.readFile`.
