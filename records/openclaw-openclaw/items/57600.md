## Summary

Keep open. Current main still lacks the requested binary-document read fallback, but this draft PR is not merge-ready because it re-reads the target file into memory for MIME detection, lacks real behavior proof, and overlaps the still-open DOCX extraction path.

## What This Changes

The PR adds PDF/Office MIME detection to the `read` wrapper, injects host and sandbox buffer readers, returns a structured fallback for binary documents, and adds DOCX fallback coverage.

## Best Possible Solution

Land one bounded document-read path that prevents binary token waste without bypassing read caps, and coordinate DOCX extraction with the document-extract design.

## Reproduction Assessment

Yes for the review finding. Source inspection of the PR diff shows `normalizeBinaryDocumentReadResult` awaits a full-buffer reader before MIME detection, while current main still has no PDF/Office read fallback.

## Solution Assessment

No. The fallback direction is useful, but this implementation should sniff a bounded prefix or carry bounded MIME metadata from the existing read path, with maintainer coordination alongside https://github.com/openclaw/openclaw/pull/57612.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound MIME sniffing reads:** `src/agents/pi-tools.read.ts:333`
  - body: The new normalization awaits a full buffer from `readBufferForToolPath` before calling `detectMime`, and both injected implementations read the full target file.
