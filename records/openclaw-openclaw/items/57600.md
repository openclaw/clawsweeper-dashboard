## Summary

Keep open. The PR targets a still-open read-tool fallback gap, but it cannot merge as-is because the diff introduces an unbounded full-file MIME sniff read, remains draft/conflicting, overlaps the open DOCX extraction PR, and lacks real after-fix behavior proof.

## What This Changes

The PR adds PDF/Office MIME fallback handling to the `read` wrapper, injects host and sandbox buffer readers for detection, and adds a DOCX fallback regression test.

## Best Possible Solution

Use one canonical bounded document-read path, either a safe fallback or maintainer-approved extraction, reusing MIME/extractor contracts without full-file re-reads.

## Reproduction Assessment

Yes for the review finding. Source inspection of the PR head shows a nonempty text result calls `loadBuffer(filePath)`, with host `fs.readFile` and sandbox `bridge.readFile` returning a full buffer before MIME detection.

## Solution Assessment

No. The fallback direction is useful, but this implementation should sniff a bounded prefix or pass bounded MIME metadata through the read operation, and maintainers should choose the canonical path alongside https://github.com/openclaw/openclaw/pull/57612.
