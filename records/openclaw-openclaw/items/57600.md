## Summary

Keep this PR open for maintainer handling, but the patch is not mergeable as-is: current main still lacks the requested document fallback, the related document-read work remains open, and this implementation adds an unbounded file-read path for MIME sniffing.

## What This Changes

The PR adds a PDF/Office MIME fallback to the `read` wrapper, wires host and sandbox buffer loaders, and adds one DOCX regression test.

## Best Possible Solution

Use one canonical document-read path with bounded MIME sniffing or bounded extraction metadata, covering both host and sandbox reads and preserving the existing adaptive read limits.

## Reproduction Assessment

Yes. For the review finding, a large PDF or Office document read would first go through the normal read path and then the PR's `normalizeBinaryDocumentReadResult` would call the full-file buffer loader before `detectMime`; this was verified from the diff rather than by running tests because the review is read-only.

## Solution Assessment

No. The fallback behavior is a useful narrow direction, but the implementation should sniff only a small prefix or reuse bounded read-operation metadata, and maintainers should decide how it relates to #57612 before landing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound the MIME sniff read:** `src/agents/pi-tools.read.ts:326-327`
  - body: The new normalization awaits a full buffer from `readBufferForToolPath` before calling `detectMime`, and both new host/sandbox callbacks implement that by reading the entire target file. Large PDFs or Office documents can therefore bypass the wrapper's adaptive read cap and spike memory after the normal read path already returned; sniff a bounded prefix or pass bounded MIME metadata through the read operation instead.
  - confidence: 0.86
