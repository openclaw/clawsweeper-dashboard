## Summary

Keep open. The fallback behavior is useful and current main still lacks it, but this draft PR is not mergeable as-is because it re-reads target documents fully for MIME detection, lacks after-fix real behavior proof, and overlaps the still-open native DOCX extraction work.

## What This Changes

The PR adds PDF/Office MIME fallback handling to the `read` wrapper, injects host and sandbox buffer readers for detection, and adds a DOCX fallback regression test.

## Best Possible Solution

Use one canonical document-read path with bounded MIME sniffing or bounded extraction metadata, preserving adaptive read limits across host and sandbox reads.

## Reproduction Assessment

Yes for the review finding. Source inspection of the PR head shows the read wrapper can receive nonempty text, then call a full-file buffer loader before `detectMime`.

## Solution Assessment

No. The fallback direction is useful, but this implementation should sniff a bounded prefix or pass bounded MIME metadata through the read operation, and maintainers should choose the canonical path alongside https://github.com/openclaw/openclaw/pull/57612.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound MIME sniffing reads:** `src/agents/pi-tools.read.ts:333`
  - body: The new normalization awaits a full buffer from `readBufferForToolPath` before calling `detectMime`, and both injected implementations read the full target file. Large PDFs or Office documents can therefore bypass the wrapper's adaptive read cap and spike memory after the normal read path already returned; sniff a bounded prefix or pass bounded MIME metadata through the read operation instead.
  - confidence: 0.86
