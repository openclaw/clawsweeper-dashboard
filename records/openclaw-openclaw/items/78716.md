## Summary

Keep open: this is a plausible fix for a real source-reproducible plugin tool dispatch bug, but the latest PR head is draft, lacks real behavior proof, and introduces blocking source-level defects.

## What This Changes

This PR changes `src/plugins/tools.ts` so cached plugin descriptor execution scans selected plugin tool factories and dispatches to the returned runtime tool whose name matches the cached descriptor.

## Best Possible Solution

Revise the cached wrapper to treat `declaredNames` as plugin-level ownership, dispatch by the actual returned runtime tool name, preserve matched-tool errors, add focused regression coverage, and include the required changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows two unnamed factories under the same `contracts.tools` list can share `declaredNames`, causing cached execution for the second tool to select the first factory and then throw `plugin tool runtime missing`.

## Solution Assessment

No. A wrapper-side dispatch fix is the right direction, but this PR’s latest head does not compile, still skips unnamed candidates in mixed registrations, and can hide matched-tool failures behind a runtime-missing error.
