## Summary

Keep open: the underlying cached plugin tool dispatch bug is valid and source-reproducible, but this PR still needs inspectable real behavior proof and current-head fixes before merge.

## What This Changes

The PR changes `src/plugins/tools.ts` so cached plugin descriptor execution scans plugin registry candidates and dispatches by the returned runtime tool name.

## Best Possible Solution

Revise the cached wrapper to use `declaredNames` only as plugin-level ownership, dispatch by actual runtime tool name without rerunning unrelated factories, add focused regression coverage plus a changelog entry, and verify with real plugin execution proof.

## Reproduction Assessment

Yes, source-reproducible. Current main stores shared plugin-level `declaredNames` on unnamed factories and the cached wrapper selects one entry before checking `runtimeTool.name`; the linked report at https://github.com/openclaw/openclaw/issues/78671 also includes sanitized live lossless-claw failures.

## Solution Assessment

No. Runtime-name dispatch in the cached wrapper is the right direction, but the current PR still reruns named factories in fallback, violates the repo catch-binding lint rule, lacks regression coverage/changelog, and has no inspectable real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Skip already-tried named candidates in fallback:** `src/plugins/tools.ts:591`
  - body: When a cached descriptor belongs to an unnamed registration and the plugin also has named registrations, this fallback iterates all `candidates`, including the named factories already invoked in the first pass. Those factories run at request time and can allocate resources or perform setup with side effects, so cached unnamed tool calls can double latency or effects; make the fallback try only unnamed or not-yet-tried candidates.
  - confidence: 0.9
- **[P2] Remove the unused catch binding:** `src/plugins/tools.ts:935`
  - body: This catch binds `error` but never reads it. The repo enables `unicorn/prefer-optional-catch-binding`, so this head will fail lint; use `catch {}` here and keep `catch (error)` only in the cold-load block that logs the error value.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/plugins/tools.ts:556`
  - body: This is a user-facing plugin execution fix, but the branch only changes `src/plugins/tools.ts`. Add a single-line `CHANGELOG.md` fix entry under the active version before merge.
  - confidence: 0.84
