## Summary

Keep open: the linked plugin dispatch bug is valid and the PR is pointed at the right runtime-name matching direction, but the latest head is not mergeable because it still leaves the cold-load catch path using an undefined `error` and lacks real cached-descriptor runtime proof.

## What This Changes

The PR changes `src/plugins/tools.ts` so cached plugin descriptor execution searches plugin tool candidates by returned runtime tool name instead of selecting one factory from plugin-level declared names.

## Best Possible Solution

Dispatch cached descriptors by actual runtime tool name while preserving cold-load error reporting, add a regression test for multiple unnamed factories under one manifest tool contract, add the changelog entry, and require redacted real runtime proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the cached wrapper can select the first unnamed factory via shared plugin-level `declaredNames`, and https://github.com/openclaw/openclaw/issues/78671 includes sanitized live `plugin tool runtime missing` failures for that shape.

## Solution Assessment

No. Matching by returned runtime tool name is the right fix direction, but this PR is not the best merge path until it restores the cold-load `catch (error)`, adds regression/changelog coverage, and supplies real after-fix plugin execution proof.
