## Summary

Keep open: the underlying cached plugin-tool bug is real and the wrapper-side fix direction is right, but the current PR head is draft, lacks inspectable real behavior proof, does not compile, and still introduces cached-dispatch regressions.

## What This Changes

This PR changes `src/plugins/tools.ts` so cached plugin descriptor execution iterates plugin tool factories and dispatches by the returned runtime tool name.

## Best Possible Solution

Revise the cached wrapper to treat `declaredNames` as plugin-level ownership, dispatch by the actual returned runtime tool name, preserve matched-tool errors, add focused regression coverage, and include the required changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main stores shared plugin-level `declaredNames` on unnamed factories and the cached wrapper selects the first matching entry before checking the returned runtime tool name.

## Solution Assessment

No. A wrapper-side runtime-name dispatch fix is the right direction, but this PR’s current head does not compile, skips unnamed candidates in mixed registrations, and can replace matched-tool errors with `plugin tool runtime missing`.

## Review Findings
