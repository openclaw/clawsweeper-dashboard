## Summary

Keep open: the branch targets the right source-reproducible cached plugin tool dispatch bug, but the latest diff still misses mixed named/unnamed registrations, catches too broadly, lacks the required changelog entry, and has no after-fix real behavior artifact.

## What This Changes

This PR rewrites cached plugin tool execution in src/plugins/tools.ts to scan a plugin's runtime tool factories and execute the returned tool whose runtime name matches the cached descriptor.

## Best Possible Solution

Keep the PR open and revise the wrapper to exact-match named registrations while still probing unnamed candidates by returned runtime tool name, preserve matched-tool errors, add regression coverage, and include the changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows function-style registrations can store empty names plus shared declaredNames, and cached execution then selects the first matching declared name before checking the returned runtime tool name.

## Solution Assessment

No. A wrapper-side dispatch fix is the right direction, but this diff should not drop unnamed candidates when named candidates exist and should not swallow the matched tool's malformed or synchronous execution errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Include unnamed candidates when named entries exist:** `src/plugins/tools.ts:563-566`
  - body: This only tries namedCandidates whenever the plugin has any named registration. A plugin can validly mix a named tool and an unnamed context factory under the same contracts.tools list; cached execution for the unnamed tool will skip its factory and still throw runtime missing. Match named candidates by the requested tool name, but keep unnamed factories in the fallback set that is checked by returned runtimeTool.name.
  - confidence: 0.9
- **[P2] Keep matched tool failures out of the candidate catch:** `src/plugins/tools.ts:568-584`
  - body: The try block also covers malformed-tool errors and the runtimeTool.execute call. If the matching tool is malformed or throws synchronously, this now catches that real failure, continues, and reports runtime missing instead of surfacing the tool error. Limit the catch to non-matching factory resolution, or rethrow once the returned runtime tool name matches the cached descriptor.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/plugins/tools.ts:556`
  - body: This is a user-facing plugin execution fix for cached descriptor tools, but the branch only changes src/plugins/tools.ts. Add a single-line CHANGELOG.md entry under the active version's fixes section before merge.
  - confidence: 0.86
