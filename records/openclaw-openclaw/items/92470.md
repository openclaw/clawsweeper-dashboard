## Summary

The issue is valid and still necessary: the published Windows page is healthy, but every Windows Hub asset link resolves through the latest stable release v2026.6.6, which lacks the installer and checksum assets that were present on v2026.6.5. The repository already has a signed-artifact promotion workflow, but no promotion run exists for v2026.6.6, so this should remain open until the intended Windows Hub build is promoted and the links return successfully.

## What This Changes

Restore the Windows Hub x64, ARM64, and checksum downloads referenced by the Windows documentation for the latest stable OpenClaw release.

## Best Possible Solution

Promote the intended signed Windows Hub build to v2026.6.6 under the existing stable asset names, verify all three public URLs, and land release validation that detects missing or stale Windows assets before a stable release is considered complete.

## Reproduction Assessment

Yes. On current main and the published documentation, each installer/checksum URL resolves to v2026.6.6 and returns HTTP 404 because those assets are absent from that release.

## Solution Assessment

No to a docs-only workaround. The narrow maintainable solution is to use the existing signed-artifact promotion workflow for v2026.6.6 and add release-completeness validation; https://github.com/openclaw/openclaw/pull/91089 may cover part of the prevention work but not the immediate repair.
