## Summary

Keep open. Current main and the latest release mitigate the original direct-Gemini trigger and the Google baseUrl footgun, but the shared strict sanitizer fallback described in the later discussion is still present and can return post-`<think>` text after `<final>` markers have been stripped, so the security-sensitive leak is not fully implemented away.

## What This Changes

The issue reports Gemini `<think>/<final>` reasoning tags leaking to WhatsApp and later Telegram when malformed or unclosed reasoning tags pass through strict visible-text sanitization.

## Best Possible Solution

Keep the direct-Gemini native reasoning behavior, then make a maintainer-approved shared sanitizer or stream-parser decision for unclosed `<think>` plus `<final>` leakage with WhatsApp and sibling-channel regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live current-main channel run. The current strict sanitizer still strips `<final>` markers before returning all text after an unclosed `<think>` opener when the visible result is otherwise empty.

## Solution Assessment

No, a WhatsApp-only splitter is likely too narrow while the shared fallback remains. The maintainable fix belongs in the shared sanitizer or stream parser, with WhatsApp/Telegram coverage proving the channel behavior.
