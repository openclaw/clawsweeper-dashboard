## Summary

Keep open: current main and the latest release still route WhatsApp text through the generic sanitizer without a WhatsApp reasoning-lane splitter, and the unclosed reasoning-tag fallback can return the hidden text the report calls out.

## What This Changes

The issue asks OpenClaw to stop WhatsApp from exposing Gemini `<think>` and `<final>` reasoning tags after the earlier Telegram-only reasoning-tag fix.

## Best Possible Solution

Add a WhatsApp-safe reasoning split/intercept step, preferably through a shared plugin-SDK text helper if Telegram and WhatsApp can use one cleanly, and cover closed, unclosed, streaming, and literal-code tag cases with focused channel tests.

## Reproduction Assessment

Yes for a source-level reproduction, but not a live one: current WhatsApp delivery runs strict reasoning stripping whose unclosed-tag fallback can return hidden content, and there is no WhatsApp splitter comparable to Telegram's. A live WhatsApp plus Gemini 2.5 Pro repro would still be useful before landing.

## Solution Assessment

Yes: keeping this open and fixing the WhatsApp delivery boundary is the narrowest maintainable path. The safer repair should avoid a broad generic sanitizer behavior change unless all affected channel and docs behavior is updated together.
