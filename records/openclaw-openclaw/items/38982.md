## What This Changes

The PR adds a runtime TUI `/theme` command, built-in mutable palettes, a searchable live-preview picker, terminal OSC color helpers, syntax color variants, command/theme tests, and an unrelated `.secrets.baseline` refresh.

## Best Possible Solution

Land one focused TUI theming path that preserves current light/dark startup detection, exposes a runtime theme API and `/theme` picker, live-previews safely, resets terminal OSC colors from shared cleanup on every exit path, recomputes already formatted chrome on preview/apply/revert, updates public TUI docs and changelog, and removes unrelated secret-scanning metadata churn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reset terminal colors from the shared exit path:** `src/tui/tui.ts:807`
  - body: `resetTerminalColors()` is only added to the signal/process-exit path, while `/exit`, `/quit`, and Ctrl+D go through `requestExit()` and never reset OSC 10/11. After applying a theme with `terminalBg`, normal exits can leave the user's terminal foreground/background changed; reset from the shared finish/requestExit cleanup path instead.
  - confidence: 0.87
- **[P2] Recompute TUI chrome after theme changes:** `src/tui/tui-command-handlers.ts:254`
  - body: `applyTheme(..., tui)` mutates formatter functions and requests a render, but the header, footer, and idle status are stored as already formatted `Text` strings. Preview, commit, and cancel can leave old ANSI colors in the TUI chrome until another state update recomputes them.
  - confidence: 0.88

  - body: The new runtime theme state initializes to `default`, dropping current main's `OPENCLAW_THEME` and `COLORFGBG` light-background detection. Users on light terminals would regress to the dark palette until manually running `/theme`; seed the runtime API from the existing detection result.
  - confidence: 0.84
