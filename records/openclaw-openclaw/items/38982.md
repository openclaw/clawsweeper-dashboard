## Best Possible Solution

## Solution Assessment

No. The picker direction is reasonable, but this patch is not the best merge shape until it rebases onto current theme detection, coordinates with https://github.com/openclaw/openclaw/pull/44382, refreshes cached TUI chrome, and removes unrelated baseline/dependency churn.

- **[P2] Seed runtime theme from current detection:** `src/tui/theme/theme.ts:191-192`
  - body: The branch initializes runtime theme state to `default`, so users with `OPENCLAW_THEME=light` or light `COLORFGBG` backgrounds would start in the dark palette until they manually run `/theme`. Seed the picker state from the current detection result instead of hard-coding `default`.
  - confidence: 0.9

  - body: `applyTheme` requests a render, but the header, footer, and idle status are already formatted strings set by `updateHeader`, `updateFooter`, and `renderStatus`. Recompute those cached strings on preview, apply, and cancel so live preview does not leave mixed old ANSI colors in the chrome.
  - confidence: 0.88

  - body: Current main removed direct TUI `cli-highlight` usage and no longer declares it as a root dependency, but this file imports it directly. A clean merge would rely on a transitive package or fail resolution; keep the current lightweight highlighter path or add a separately reviewed dependency change.
  - confidence: 0.84
- **[P3] Document the /theme command before shipping:** `src/tui/commands.ts:113-120`
  - body: This adds a user-facing `/theme` picker/list flow, but current public TUI docs only describe the existing `OPENCLAW_THEME` light/dark environment behavior. Add user docs for the new command and picker behavior before shipping the feature.
  - confidence: 0.8
