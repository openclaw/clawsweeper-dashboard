## Summary

Keep this PR open. Current main still lacks the requested runtime `/theme` command, `/theme list`, searchable picker, live preview, and cancel/commit flow, so the work is not obsolete; however the branch still needs maintainer follow-up because the proposed patch leaves preformatted TUI chrome in old colors, regresses current main's startup light/dark detection path unless rebased carefully, overlaps with open PR #44382, and includes unrelated secret-baseline churn.

## What This Changes

The PR adds a `/theme` slash command, six runtime TUI palettes, a searchable live-preview theme picker, terminal OSC color updates/reset handling, command/theme tests, and a `.secrets.baseline` refresh.

## Best Possible Solution

Land one focused TUI theming implementation that preserves current startup light/dark detection, adds a runtime theme API, registers `/theme` and `/theme list`, opens a searchable picker with live preview, commits on select, reverts on cancel, recomputes all already formatted TUI chrome on every preview/apply/revert, documents the public behavior at `https://docs.openclaw.ai/web/tui`, and carries targeted command/theme tests. If maintainers choose #44382 as canonical, this PR should be superseded only after that decision or merge path is explicit.

## Reproduction Assessment

Yes. The high-confidence path is source-level and manual: run the PR branch TUI, open `/theme`, move selection, and observe that header/footer/idle status text can remain in old ANSI colors because only `tui.requestRender()` runs after formatter mutation.

## Solution Assessment

No, not in its current form. The feature direction is reasonable, but the patch needs to preserve current-main startup theme detection, repaint preformatted chrome during preview/apply/revert, drop unrelated security-baseline churn, and be coordinated with #44382 before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Repaint TUI chrome after theme changes:** `src/tui/tui-command-handlers.ts:254`
  - body: `applyTheme(..., tui)` mutates formatter functions and requests a render, but the header, footer, and idle status are already stored as formatted strings via `Text.setText(...)` in `runTui`. A render reuses those old ANSI strings until another event recalculates them, so live preview and commit can leave mixed-theme chrome. Call the header/footer/status recomputation path after every preview, commit, and cancel.
  - confidence: 0.88
- **[P2] Preserve startup light-theme detection:** `src/tui/theme/theme.ts:191-192`
  - body: The new runtime theme state initializes unconditionally to `default`, but current main already honors `OPENCLAW_THEME` and `COLORFGBG` at startup and documents that behavior for light terminals. Landing this branch without carrying that detection into the new theme API would regress light-terminal readability until the user manually runs `/theme`.
  - confidence: 0.82
