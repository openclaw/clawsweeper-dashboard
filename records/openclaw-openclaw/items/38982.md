## Summary

Keep open, but this PR is not merge-ready: the feature is still absent from current main, and the branch has blocking regressions against current TUI theme detection, cached chrome repainting, dependency declarations, and contributor proof requirements.

## What This Changes

The PR adds a `/theme` slash command with a searchable live-preview picker, six runtime palettes, OSC terminal color helpers, syntax theme variants, focused tests, and an unrelated `.secrets.baseline` refresh.

## Best Possible Solution

Land one canonical TUI theming design that preserves current light/dark auto-detection, keeps mounted theme objects and cached chrome in sync, documents the user-facing command, and excludes unrelated secret baseline or dependency churn.

## Reproduction Assessment

Yes, for the blocking patch issues. Source inspection shows the PR head hard-codes default runtime theme state and only requests render on theme changes while current main stores header/footer/status as preformatted `Text` strings.

## Solution Assessment

No. The picker direction is reasonable, but this branch should be rebased onto current theme detection, coordinated with https://github.com/openclaw/openclaw/pull/44382, documented, and stripped of unrelated baseline churn before merge.

  - body: The branch initializes `palette` and `currentThemeName` to `default`, so users with `OPENCLAW_THEME=light` or a light `COLORFGBG` background start in the default dark palette until they manually run `/theme`.   - confidence: 0.84

  - body: This adds a user-facing `/theme` picker/list flow, but current public TUI docs only cover the existing `OPENCLAW_THEME` light/dark environment behavior. Add user docs for the new command and picker behavior before shipping the feature.
