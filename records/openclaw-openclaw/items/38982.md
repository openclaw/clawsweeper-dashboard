## Summary

Keep open. Current main still does not have a runtime `/theme` picker, but this PR is not merge-ready because it regresses current theme detection, leaves cached TUI chrome in stale colors during preview/apply, reintroduces direct `cli-highlight` usage after main removed that dependency, lacks inspectable real behavior proof, and overlaps with https://github.com/openclaw/openclaw/pull/44382.

## What This Changes

Adds a TUI `/theme` picker with six runtime palettes, live preview/cancel behavior, terminal OSC color helpers, syntax theme changes, tests, and an unrelated `.secrets.baseline` update.

## Best Possible Solution

Land one canonical, rebased TUI theming design that preserves current light/dark auto-detection, updates mounted theme objects and cached chrome during preview/apply/cancel, documents the user-facing command, and excludes unrelated baseline or dependency churn.

## Reproduction Assessment

Yes for the blocking patch issues: source inspection shows the PR head hard-codes default runtime theme state and relies on render-only theme changes while current main caches chrome as formatted `Text` strings. The feature request itself is not a bug reproduction case.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best merge shape until it is rebased onto current detection behavior, coordinated with https://github.com/openclaw/openclaw/pull/44382, documented, and stripped of unrelated baseline/dependency issues.
