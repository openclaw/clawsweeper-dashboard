## Summary

Keep open: current main still lacks the requested runtime `/theme` picker, so the PR is not obsolete. The branch is not merge-ready because it regresses current startup theme detection, does not repaint preformatted TUI chrome during live preview/apply/revert, misses normal-exit terminal color reset, and overlaps the broader open #44382 theme-system PR.

## What This Changes

The PR adds a runtime TUI `/theme` command, six mutable palettes, a searchable live-preview theme picker, terminal OSC color updates/reset helpers, syntax color variants, command/theme tests, and a `.secrets.baseline` refresh.

## Best Possible Solution

Land one focused TUI theming path that preserves current light/dark startup detection, exposes a runtime theme API and `/theme` picker, live-previews safely, resets terminal OSC colors from shared cleanup on every exit path, recomputes already formatted chrome on preview/apply/revert, updates public TUI docs and changelog, and removes unrelated secret-scanning metadata churn. If #44382 becomes canonical, this PR should be superseded only after that maintainer choice is explicit.

## Reproduction Assessment

Yes. The high-confidence path is source-level plus manual: run the PR branch TUI, switch to a theme with `terminalBg`, then exit via `/exit` or Ctrl+D and observe the OSC colors are not reset; also preview themes and observe header/footer/status can keep old ANSI strings because only `requestRender()` runs after formatter mutation.

## Solution Assessment

No. The feature direction is reasonable, but the current patch is not the best maintainable solution until it preserves current-main startup detection, resets terminal colors in shared exit cleanup, repaints preformatted chrome, drops unrelated baseline churn, adds changelog/docs coverage, and is coordinated with #44382.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reset terminal colors from the shared exit path:** `src/tui/tui.ts:807`
  - body: `resetTerminalColors()` is only added to the signal/process-exit path, while `/exit`, `/quit`, and Ctrl+D all go through `requestExit()` and never reset OSC 10/11. After applying a theme with `terminalBg`, normal exits can leave the user's terminal foreground/background changed; reset in the shared finish/requestExit path instead.
  - confidence: 0.87
- **[P2] Recompute TUI chrome after theme changes:** `src/tui/tui-command-handlers.ts:254`
  - body: `applyTheme(..., tui)` mutates formatter functions and requests a render, but the header, footer, and idle status are already stored as formatted `Text` strings. Preview, commit, and cancel can leave old ANSI colors in TUI chrome until an unrelated state update reruns `updateHeader`/`updateFooter`/`renderStatus`.
  - confidence: 0.88
- **[P2] Preserve startup theme detection:** `src/tui/theme/theme.ts:191-192`
  - body: The new runtime theme state initializes unconditionally to `default`, dropping current main's `OPENCLAW_THEME` and `COLORFGBG` light-background detection. Users on light terminals would regress to the dark palette until they manually run `/theme`; seed the new API from the existing detection path.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/tui/commands.ts:110-117`
  - body: This is a user-facing `feat` that adds a TUI slash command and picker, but the branch does not update `CHANGELOG.md`. Repo policy requires an active-release single-line entry with appropriate Thanks credit for new user-visible behavior.
  - confidence: 0.82
