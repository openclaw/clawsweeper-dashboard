## Summary

Keep open. Current main still lacks a Bedrock `requestMetadata` configuration/transport path, and the related open work covers automatic per-agent tagging rather than this arbitrary config surface.

## What This Changes

The issue requests exposing Bedrock `requestMetadata` in OpenClaw configuration for AWS cost allocation tagging.

## Best Possible Solution

Add a Bedrock-owned, validated `requestMetadata` surface with docs/tests and explicit merge precedence with any automatic per-agent metadata, after maintainers choose the public config shape.

## Reproduction Assessment

Yes. Source inspection on current main shows upstream pi-ai and AWS support the field, while OpenClaw’s Bedrock config schema, extra-param wrapper, and Bedrock provider wrapper do not expose or forward it.

## Solution Assessment

Unclear. The capability is valid, but the exact config path and merge/privacy policy need maintainer product/API judgment before a narrow implementation is safe.
