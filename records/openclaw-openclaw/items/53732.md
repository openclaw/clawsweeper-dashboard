## Summary

Keep open. Current main still embeds BOOT.md as ordinary prompt text, preserves that body across fallback attempts, and only strips reasoning tags from message-tool text fields; the linked open closing PR should be reviewed, landed, or closed before this issue is resolved.

## What This Changes

Request to prevent fallback models during BOOT.md startup runs from echoing BOOT.md instructions to users, especially through final replies or message-tool sends.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75128 or an equivalent narrow fix that marks BOOT.md content as internal runtime context and strips/suppresses that context before final replies, message-tool sends, and unsafe diagnostics while preserving legitimate boot notifications.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: BOOT.md content enters buildBootPrompt, fallback attempts preserve that body, and message-tool delivery currently lacks internal-context stripping.

## Solution Assessment

Yes. Keeping this issue open while the linked closing PR is reviewed is the narrowest maintainable path; a separate repair lane would duplicate active implementation work.
