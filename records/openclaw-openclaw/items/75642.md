## Summary

Keep open. The report is recent, unprotected, and matches current `main`: embedded Gateway runs still perform runtime plugin loading and full core/plugin tool construction before stream setup, and plugin tool factories are still invoked while building the per-run tool list. Existing related cleanup around startup/status plugin discovery does not cover this per-request `core-plugin-tools` hot path.

## What This Changes

Request to reduce Feishu/Gateway request latency by avoiding repeated plugin tool/spec loading during every simple agent request.

## Best Possible Solution

Keep the issue open and fix the request hot path so plugin runtime/tool metadata is reused or deferred without changing tool authorization semantics. The end state should keep plugin registry activation stable, avoid rebuilding expensive tool specs before first typing/stream setup, and preserve request-scoped executors for delivery, sandbox, sender, and policy context.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: enable embedded-run stage logging and send a normal Feishu message through the Gateway, then inspect the `[trace:embedded-run]` startup/prep stage summaries. The exact wall-clock delay still needs the reporter's live plugin/provider setup to quantify.

## Solution Assessment

Unclear because there is no proposed patch. Caching plugin specs is plausible, but the safest fix needs a narrow cache/defer boundary that preserves per-request tool policy and executor context rather than globally memoizing whole tool objects.
