## Summary

Keep open. Current main has already fixed several older plugin startup/schema reload paths, but this report is about the live per-request agent-turn path: `core-plugin-tools` still wraps `createOpenClawCodingTools()`, which resolves plugin tools every turn. The code reuses a compatible active registry, so this is not proven to be repeated full plugin module loading, but the request-time factory/materialization path remains plausible and needs targeted profiling before cleanup can close it.

## What This Changes

The issue requests reducing Gateway per-message latency from about 50 seconds to under 5 seconds by avoiding repeated request-time plugin tool/spec loading in Feishu/minimax Gateway runs.

## Best Possible Solution

Profile the Feishu/minimax agent-turn preparation path and then split cacheable plugin tool descriptors/specs from request-scoped execution context. Preserve the existing active-registry reuse, add context-safe invalidation keyed by registry/config/runtime version, and cover the fix with focused plugin-tool hot-path tests plus one gateway agent-turn timing guard.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The issue provides concrete stage timings and a plausible Feishu/minimax path, but this read-only review did not run live credentials or a benchmark; source inspection confirms the implicated request-time stage still exists.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Avoiding repeated full registry loads is already implemented, so the safer direction is not blanket tool caching but a measured, context-safe descriptor/cache seam for plugin tool materialization.
