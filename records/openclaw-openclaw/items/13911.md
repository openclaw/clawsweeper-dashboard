## Summary

Keep open. Current main and the latest v2026.6.6 release still do not expose a public per-channel sub-agent announce suppression policy, while current delivery code and tests continue to preserve requester-channel delivery and direct fallback behavior; the remaining work needs maintainer product/API direction because it overlaps broader announce routing work.

## What This Changes

The issue requests a public per-channel policy or config option to suppress or reroute sub-agent completion announcements away from noisy chat channels such as Telegram.

## Best Possible Solution

Define one backward-compatible sub-agent completion delivery policy that defaults to current behavior, covers per-channel suppression and related parent-routing semantics, and gates both requester-agent delivery and fallback sends after maintainer API direction.

## Reproduction Assessment

Yes, at source level: current main exposes no public per-channel sub-agent announce policy, no native sessions_spawn announce/routing parameter, and docs still describe token-based suppression. I did not run a live Telegram reproduction because the missing product surface is directly visible in source, docs, tests, and the latest release tag.

## Solution Assessment

Unclear. A channel denylist would address the Telegram/iMessage noise, but a single completion-delivery policy coordinated with the broader routing work is safer than adding a channel-only config knob that can drift from parent routing and fallback semantics.
