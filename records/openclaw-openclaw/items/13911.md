## Summary

Keep open. Current main has improved the announce path since earlier discussion, but it still does not expose a structural per-channel or per-spawn suppression policy; this remains a product/API decision tied to broader delivery-audience work.

## What This Changes

The issue requests a supported way to suppress or reroute sub-agent completion announcements per channel, especially for Telegram and iMessage sessions.

## Best Possible Solution

Add a backward-compatible completion delivery policy that defaults to current requester-chat delivery, supports channel and/or spawn-level suppressed or internal-only handling, and updates config schema/types, docs, and focused delivery tests.

## Reproduction Assessment

Yes, for the current behavior and missing capability. Source, docs, and tests on current main show no supported suppression key or public `sessions_spawn` announce policy, while top-level completion delivery still resolves external requester routes.

## Solution Assessment

Unclear pending maintainer API direction. The requested delivery-layer control belongs in the right area, but the exact primitive should be chosen alongside https://github.com/openclaw/openclaw/issues/8299, https://github.com/openclaw/openclaw/issues/64658, https://github.com/openclaw/openclaw/issues/27445, and https://github.com/openclaw/openclaw/issues/69492.
