## Summary

Keep open. Current main still lacks a public per-channel config policy or advertised spawn-time routing policy for sub-agent completion announces; the newer `expectsCompletionMessage: false` path is a hidden all-or-nothing completion skip, not the requested structural per-channel suppression/internal-only delivery control, and maintainer context keeps the design tied to #69492.

## What This Changes

The issue requests a config or spawn-time policy to suppress or reroute sub-agent completion announcements per channel so channel chats are not cluttered by raw sub-agent output.

## Best Possible Solution

Ship a backward-compatible announce audience/delivery policy that defaults to current behavior, can suppress or internalize completion announces by channel and/or spawn call, and runs before both announce-agent delivery and direct fallback sends.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: config/tool/docs surfaces expose no public per-channel suppression or reroute option, while delivery code and tests still preserve external requester delivery and direct fallback sends.

## Solution Assessment

Unclear. The requested direction is valid, but the best maintainable fix should align with #69492's audience/delivery classification instead of immediately adding a Telegram- or channel-specific flag; the hidden all-or-nothing completion skip is not sufficient.
