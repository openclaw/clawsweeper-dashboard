## Summary

Keep open. Current main and v2026.6.8 still lack a public per-channel sub-agent announce suppression policy, while top-level completion delivery still resolves user-facing targets and direct fallback sends; related suppression/routing/hook issues overlap but do not replace this per-channel request.

## What This Changes

The issue requests a structural per-channel or per-spawn way to suppress sub-agent completion announcements so Telegram/iMessage-style channel chats can avoid raw sub-agent completion noise.

## Best Possible Solution

Define one backward-compatible completion-delivery policy that defaults to current behavior, can suppress or internalize completions for selected channels, and applies before both requester-agent handoff and direct fallback delivery with schema, docs, and focused tests updated together.

## Reproduction Assessment

Yes, at source level: current main has no public per-channel sub-agent announce policy or `sessions_spawn` suppression/routing parameter, while delivery still resolves user-facing targets and direct fallback sends. I did not run a live Telegram/iMessage reproduction because the missing public surface is directly visible in source, docs, and the latest release tag.

## Solution Assessment

Unclear. A per-channel suppress knob matches the reported Telegram/iMessage pain, but the best maintainable fix is likely one completion-delivery policy coordinated with the open global suppression and parent-routing requests.
