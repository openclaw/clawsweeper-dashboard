## Label Justifications

- `impact:message-loss`: The requested policy controls whether sub-agent completion messages are delivered to, suppressed from, or rerouted away from user-facing channels.

## Summary

Keep open. Current main and the latest v2026.6.6 release still lack a structural per-channel or per-spawn policy for suppressing sub-agent completion announces, and the overlapping open trackers do not fully replace this channel-level product/API decision.

## What This Changes

Request a per-channel or broader config/API policy to suppress sub-agent completion announcements from selected chat channels.

## Best Possible Solution

Define one backward-compatible completion delivery policy that defaults to current channel delivery but can mark selected channel or spawn completions as internal-only or suppressed, then apply it before both the requester-agent handoff and direct fallback sends with schema, docs, and focused delivery tests updated consistently with the related routing trackers.

## Reproduction Assessment

Yes, at source level: current main exposes no public `sessions_spawn` announce/silent parameter and no `agents.defaults.subagents` or per-channel suppression policy, while docs still describe only exact token-based suppression. I did not run a live Telegram or iMessage scenario in this read-only review because the missing public surface is directly visible in source, docs, and tests.

## Solution Assessment

Not as a one-off Telegram-only key. The maintainable solution is a generic, backward-compatible delivery policy coordinated with the open deterministic suppression and parent-routing trackers so channel defaults, per-spawn overrides, and fallback delivery do not diverge.
