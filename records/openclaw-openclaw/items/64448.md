## Summary

Keep open: current main still sends Feishu image media through upload plus message create/reply without a stable provider idempotency field, and the related open rate-limit issue covers a different delivery failure mode rather than this reconnect duplicate-send report.

## What This Changes

The issue asks OpenClaw to prevent occasional duplicate Feishu image sends when WebSocket reconnect happens during image upload/delivery.

## Best Possible Solution

Thread a stable outbound idempotency key through the plugin outbound context and map it to Feishu/Lark message uuid for image create/reply sends, while preserving distinct keys for separate intentional sends.

## Reproduction Assessment

No high-confidence live reproduction was run for the WebSocket reconnect timing. Source inspection confirms the missing dedupe point, and a focused regression can simulate repeated post-upload sends once a stable outbound key is threaded.

## Solution Assessment

Yes. Provider-level uuid dedupe from a stable outbound id is the narrowest maintainable fix; disabling reconnect or adding a media URL in-flight lock would be broader and could hide legitimate sends.
