## Summary

Keep open. Current main still has the reported Feishu image duplicate-send gap: image media uploads are followed by Feishu message create/reply calls without a stable provider dedupe field, while the related open #70879 tracks rate-limit retry behavior rather than duplicate delivery after reconnect.

## What This Changes

The issue reports intermittent duplicate Feishu image delivery after Feishu websocket reconnects during media upload and requests idempotent media sends.

## Best Possible Solution

Keep the issue open and add a narrow Feishu media-delivery fix: thread a stable outbound send idempotency key through the generic plugin outbound context, map it to Feishu/Lark message uuid for image create/reply sends, and preserve intentional separate sends as distinct keys.

## Reproduction Assessment

Unclear for the live websocket timing: the report gives a concrete CLI command and log pattern, and current source confirms the missing dedupe point. A high-confidence automated reproduction can be built by simulating a repeated post-upload image send with the same outbound idempotency key.

## Solution Assessment

Yes. Provider-level uuid dedupe is the narrowest maintainable solution; disabling reconnect or adding a media URL in-flight lock would either avoid the provider contract or risk suppressing intentional separate sends.
