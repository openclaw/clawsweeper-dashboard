## Summary

Keep open: current main and the latest shipped release still send Feishu image media through upload plus `message.create`/`message.reply` without a stable Feishu `uuid`, and no outbound idempotency key reaches the Feishu plugin context. The closed broad Feishu duplicate report covered inbound reply-once dedupe, while the open rate-limit tracker is a separate delivery reliability issue.

## What This Changes

The issue reports occasional duplicate Feishu image sends when WebSocket reconnect happens during media upload.

## Best Possible Solution

Thread a stable outbound idempotency key through the plugin outbound context and map it to Feishu/Lark `im.message.create` and `im.message.reply` `uuid` fields for media sends, with regression coverage for distinct intentional sends.

## Reproduction Assessment

No high-confidence live reproduction was run for the WebSocket reconnect timing. Source inspection does show the current media path lacks the provider `uuid` field and the stable outbound key needed to make a repeated post-upload send safe.

## Solution Assessment

Yes. Provider-level `uuid` dedupe from a stable outbound send id is the narrowest maintainable fix; disabling reconnect or adding a media URL in-flight lock would be broader and could suppress legitimate sends.
