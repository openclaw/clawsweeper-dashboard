## Summary

Current main still lacks the Feishu outbound idempotency path this report needs: image upload is followed by message create/reply without the SDK-supported `uuid`, and the shared plugin outbound context does not carry a stable send key into Feishu media sends. Related Feishu issues cover inbound replay, rate-limit retries, and proactive pacing; they do not supersede this outbound duplicate-image bug.

## What This Changes

The issue reports occasional duplicate Feishu image sends when WebSocket reconnects during image upload/send, and asks for provider-level idempotency or equivalent dedupe.

## Best Possible Solution

Add a narrow Feishu media-delivery fix that threads a stable outbound send id through the plugin outbound context and maps it to Feishu/Lark message `uuid` for image create/reply sends while preserving distinct keys for intentional separate sends.

## Reproduction Assessment

No high-confidence live reproduction was run for the WebSocket reconnect timing. Source inspection confirms the missing dedupe point in current main, and a focused regression can simulate repeated post-upload image delivery using the same outbound idempotency key once that key is threaded.

## Solution Assessment

Yes. Provider-level `uuid` dedupe from a stable outbound idempotency key is the narrowest maintainable direction; disabling reconnect or locking by media URL would either risk delivery reliability or collapse legitimate separate sends.

## Repair Work Prompt

Fix the Feishu duplicate image-send bug from https://github.com/openclaw/openclaw/issues/64448. Current main uploads images in `extensions/feishu/src/media.ts` and then calls `client.im.message.create`/`reply` without the SDK-supported `data.uuid`; `ChannelOutboundContext` also has no stable outbound idempotency key for Feishu to consume. Add a backward-compatible plugin outbound context field carrying the stable send id where available, thread it through core outbound delivery/plugin message sends into Feishu `sendMediaFeishu`/`sendImageFeishu`, and set Feishu message `uuid` for image create/reply only when a stable key is present. Preserve intentional repeated sends as distinct keys; do not disable WebSocket reconnect and do not use a global media URL lock. Add focused regression coverage for Feishu media create/reply payloads and context propagation. Add a CHANGELOG entry because this is a user-visible delivery fix.
