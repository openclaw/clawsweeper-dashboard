## Summary

Keep open: current main still sends Feishu image media through upload plus `message.create`/`message.reply` without a stable Feishu `uuid`, so a repeated post-upload send is not provider-deduped. The related rate-limit retry issue remains a separate delivery reliability problem, not a duplicate-media fix.

## What This Changes

Request that Feishu image media sends be idempotent so WebSocket reconnect or retry behavior cannot deliver the same image twice.

## Best Possible Solution

Thread a stable outbound idempotency key through the plugin outbound context and map it to Feishu/Lark `im.message.create` and `im.message.reply` `uuid` fields for media sends, with regression tests for distinct intentional sends.

## Reproduction Assessment

No high-confidence live reproduction was run for the WebSocket reconnect timing. Source inspection shows the current media path lacks the provider `uuid` field needed to make a repeated post-upload message send safe.

## Solution Assessment

Yes. Provider-level `uuid` dedupe from a stable outbound send id is the narrowest maintainable fix; disabling reconnect or adding a media URL in-flight lock would be broader and could suppress legitimate sends.
