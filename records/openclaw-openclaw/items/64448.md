## Summary

Keep open: current main still sends Feishu image media through upload plus message create/reply without a stable Feishu `uuid`, while the linked rate-limit retry issue covers a different delivery failure mode.

## What This Changes

The issue reports occasional duplicate Feishu image sends during WebSocket reconnects and asks for provider-level idempotency on media sends.

## Best Possible Solution

Thread a stable outbound idempotency key through the plugin outbound context and map it to Feishu/Lark message `uuid` for media message create/reply sends, with regression tests for distinct intentional sends.

## Reproduction Assessment

No high-confidence live reproduction was run for the WebSocket reconnect timing. Source inspection shows the current media path lacks the provider idempotency field needed to make a repeated post-upload send safe.

## Solution Assessment

Yes. Provider-level `uuid` dedupe from a stable outbound id is the narrowest maintainable fix; disabling reconnect or adding a media URL in-flight lock would be broader and could suppress legitimate sends.
