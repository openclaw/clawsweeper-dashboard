## Summary

Keep open: current main still routes Feishu image media through upload plus message create/reply without a stable provider dedupe field, and the related open #70879 covers rate-limit retries rather than reconnect duplicate delivery.

## What This Changes

Request to prevent occasional duplicate Feishu image sends when WebSocket reconnect or retry behavior replays a media send.

## Best Possible Solution

Add a narrow Feishu media-delivery fix that threads a stable outbound send id through the plugin outbound context and maps it to Feishu/Lark message uuid for image create/reply sends while preserving separate intentional sends as distinct keys.

## Reproduction Assessment

No high-confidence live reproduction was run for the WebSocket reconnect timing. Source inspection does confirm the missing dedupe point, and a deterministic regression can simulate repeated post-upload image sends using the same outbound idempotency key once that key is threaded.

## Solution Assessment

Yes. Provider-level uuid dedupe from a stable outbound idempotency key is the narrowest maintainable fix; disabling reconnect or adding a media URL in-flight lock would risk hiding legitimate separate sends.
