## Summary

Keep open. Current main still uploads the Feishu image and sends it with `message.create` or `message.reply` without the SDK-supported `uuid`, and no stable outbound idempotency key is available in the plugin outbound context. The related retry and pacing issues are adjacent reliability work, not a superseding fix for this duplicate image-send path.

## What This Changes

The issue requests preventing occasional duplicate Feishu image sends during WebSocket reconnects by adding an idempotency guard to outbound media delivery.

## Best Possible Solution

Thread a stable outbound send id into the plugin outbound context and map it to Feishu/Lark `uuid` for image create/reply sends, with regression coverage proving duplicate retries share the key while separate sends do not.

## Reproduction Assessment

No high-confidence live reproduction was run for the WebSocket reconnect timing. Source inspection does show the missing dedupe point on current main, and a focused test can simulate repeated post-upload image delivery once a stable outbound key is threaded.

## Solution Assessment

Yes. Provider-level `uuid` dedupe from a stable outbound idempotency key is the narrowest maintainable direction; disabling reconnect or locking by media URL would either reduce delivery reliability or collapse legitimate separate sends.
