## Summary

Keep open. Current main still leaves the reported Feishu image duplicate-send path unresolved: image media is uploaded and then sent via Feishu message create/reply without a stable provider dedupe field, and the related open #70879 is about rate-limit retry rather than duplicate delivery after reconnect.

## What This Changes

The issue reports occasional duplicate Feishu image delivery when websocket reconnects occur during `openclaw message send --channel feishu --media`, and requests idempotent media sends.

## Best Possible Solution

Keep the issue open and implement a narrow Feishu media-delivery fix: propagate a stable outbound send idempotency key through the generic plugin outbound context, map it to Feishu/Lark message `uuid` for image create/reply sends, and preserve intentional separate sends as distinct keys.

## Reproduction Assessment

Unclear for the live websocket timing: the report gives a plausible command/log reproduction, and current source confirms the missing dedupe point, but the reconnect race itself was not executed here. A high-confidence automated reproduction can be built by simulating a repeated post-upload image send with the same outbound idempotency key.

## Solution Assessment

Yes, using Feishu's message-level `uuid` is the narrowest maintainable fix for duplicate delivery after reconnect/retry. A broad in-flight media URL lock would risk suppressing intentional separate sends and would not use the provider's own dedupe contract.
