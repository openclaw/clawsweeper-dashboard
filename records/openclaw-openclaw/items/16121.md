## Summary

Keep open. Current main and the latest release have partial raw `<tg-emoji>` HTML passthrough, but first-class Telegram custom emoji support is still missing from the shared Markdown/entity representation, message-tool inputs, entity-bearing sends, captions, edits, replies, and streaming paths; the related reaction PR is narrower and the message-tool duplicate was closed in favor of this issue.

## What This Changes

Request support for Telegram custom emojis in messages, captions, and replies, including custom emoji identifiers and send/display behavior.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving shipped raw HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, upstream-contract, shipped-release, and related-item inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw `<tg-emoji>` passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
