## Summary

Keep open. Current main and v2026.6.8 partially preserve raw Telegram `<tg-emoji emoji-id="...">` HTML, but the central outgoing entity/message-tool request remains unsolved: the shared message tool still has no `entities` or `caption_entities` input, and Telegram send/edit/caption/draft paths do not forward custom emoji entity arrays.

## What This Changes

Request support for Telegram custom emojis in messages, captions, and replies, including custom emoji identifiers and an outgoing path through OpenClaw's Telegram message delivery surface.

## Best Possible Solution

Define one supported Telegram custom-emoji/entity contract, then wire it through message-tool schema, Telegram channelData, send/edit/caption/draft delivery paths, docs, and focused regression tests while preserving current raw HTML fallback behavior.

## Reproduction Assessment

Yes, source-level: the message-tool schema has no entity fields, and Telegram send/edit/caption paths do not accept or forward `entities` or `caption_entities`. I did not run a live Telegram Bot API reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. Raw `<tg-emoji>` pass-through is useful partial support, but it does not solve plugin slash-command entity round-tripping, caption entities, or edit/draft delivery.
