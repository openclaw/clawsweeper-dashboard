## Summary

Keep open. Current main and v2026.6.8 preserve raw `<tg-emoji emoji-id="...">` HTML, but the central outgoing entity/message-tool request remains unsolved: the shared message tool has no custom-emoji entity fields, and Telegram send, caption, edit, draft, and outbound adapter paths still do not forward `entities` or `caption_entities`. The June 12 production report is useful confirming context for the plugin slash-command/message-tool surface.

## What This Changes

The issue requests first-class Telegram custom emoji support in outgoing messages, captions, and replies, including custom emoji identifiers and rendering.

## Best Possible Solution

Define one supported Telegram custom-emoji/entity contract, then wire it through message-tool schema, Telegram channelData, send/edit/caption/delivery paths, docs, and focused regression tests while preserving raw HTML fallback behavior.

## Reproduction Assessment

Yes, by source inspection: the message-tool schema has no entity fields and Telegram send/edit/caption paths do not accept or forward `entities` or `caption_entities`. I did not run a live Telegram Bot API reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. Raw `<tg-emoji>` pass-through is useful partial support, but it does not solve plugin slash-command entity round-tripping or caption/edit entity support.
