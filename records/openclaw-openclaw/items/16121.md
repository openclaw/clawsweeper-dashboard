## Summary

Keep open. Current main and the latest release preserve raw Telegram custom-emoji HTML, but first-class custom emoji support is still missing from the shared message-tool contract, Markdown IR, and Telegram entity/caption/edit/preview delivery paths.

## What This Changes

The issue requests Telegram custom emoji support for messages, captions, and replies, including custom emoji identifiers and an agent-usable send/display path.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving raw HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, upstream Telegram API, current-main, latest-release, and related-item inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw tg-emoji HTML passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
