## Summary

Keep open. Current main has shipped partial raw `<tg-emoji>` HTML passthrough, but the requested first-class Telegram custom emoji support is still incomplete across shared Markdown/entity representation, message-tool inputs, entity-bearing sends, captions, edits, replies, and streaming paths.

## What This Changes

Request Telegram custom emoji support for outgoing messages, captions, replies, and identifiers according to the current Telegram Bot API.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving shipped raw HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, upstream-contract, release, and related-item inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw `<tg-emoji>` passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
