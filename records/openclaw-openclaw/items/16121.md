## Summary

Keep open. Current main has partial raw `<tg-emoji emoji-id="...">` HTML passthrough, but the requested first-class Telegram custom emoji/entity support for message-tool input, sends, captions, replies, and shared representation is still incomplete.

## What This Changes

The issue requests first-class Telegram custom emoji support in Telegram messages, captions, replies, and agent-facing send APIs.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving the current HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high confidence that first-class outgoing custom emoji/entity support is still incomplete.

## Solution Assessment

Unclear until maintainers choose the public representation. Raw `<tg-emoji>` passthrough is useful partial support, but it is not the full maintainable solution for entity validation, message-tool input, captions, replies, chunking, and fallback policy.
