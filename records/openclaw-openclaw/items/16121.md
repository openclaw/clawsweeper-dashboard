## Summary

Keep open. Current main has useful partial support for preserving raw Telegram `<tg-emoji emoji-id="...">` HTML, but it still lacks first-class outgoing custom emoji/entity support through the shared message tool, Telegram send options, captions, replies/quotes, and API/docs contract.

## What This Changes

Request to support Telegram custom emoji identifiers and rendering in bot messages, captions, and replies according to the current Telegram Bot API.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message text, captions, replies/quotes, edits/previews, streaming, the shared message tool, docs, and focused tests while preserving current HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and upstream API inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete on current main.

## Solution Assessment

Unclear. Raw `<tg-emoji>` passthrough is a useful partial solution, but the maintainable fix needs an explicit API choice for shared IR, Telegram entity arrays, message-tool input, captions, replies, chunking, and fallback behavior.
