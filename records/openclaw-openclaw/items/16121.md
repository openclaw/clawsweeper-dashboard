## Summary

Keep open: v2026.6.6 and current main now preserve Telegram’s raw `<tg-emoji emoji-id="…">` HTML through text, caption, reply, and edit formatting paths, but the requested first-class agent-facing support remains absent and the existing maintainer discussion explicitly leaves the shared API representation undecided.

## What This Changes

Add first-class Telegram custom emoji support for outgoing messages, captions, and replies, including a usable representation for custom emoji identifiers.

## Best Possible Solution

Make the shipped safe `<tg-emoji>` path an explicit tested contract first, then add the narrowest typed authoring representation needed for agents while keeping Telegram entity encoding, limits, and fallback behavior inside the Telegram plugin.

## Reproduction Assessment

Yes at source level: current code proves raw `tg-emoji` passthrough exists but the Telegram send contract and shared message tool expose no first-class custom-emoji input. Live Telegram execution was not performed in this read-only review.

## Solution Assessment

No, adding only raw entity passthrough or only a new Markdown token is not yet proven best; maintainers should define the portable authoring contract and preserve the existing canonical HTML chunking and fallback paths.
