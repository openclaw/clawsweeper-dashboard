## Summary

Keep open. Current main has partial raw HTML support for `<tg-emoji emoji-id="...">`, but it still lacks first-class outgoing Telegram `entities`/`caption_entities` support through the shared message tool, Telegram action runtime, send/edit/caption paths, and Markdown IR, so the central request remains unresolved.

## What This Changes

The issue requests first-class Telegram custom emoji support for outgoing Telegram messages, captions, and replies, including custom emoji IDs/entities instead of text fallback.

## Best Possible Solution

Choose the public representation first, then implement validated Telegram custom emoji support through the Telegram plugin send/edit/caption/streaming paths and the shared message tool, preserving existing HTML fallback behavior with focused tests and docs.

## Reproduction Assessment

Yes. Source inspection shows the current message tool and Telegram action/send paths have no outgoing `entities` or `caption_entities` surface, while a recent user comment gives a concrete plugin slash-command path where entity arrays are dropped before reaching Telegram.

## Solution Assessment

No, the current partial raw HTML allowlist is not the complete solution. The best fix needs a maintainer-approved API shape for entity-bearing custom emoji support across the shared message tool and Telegram-owned delivery paths.
