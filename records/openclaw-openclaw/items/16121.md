## Summary

Keep open. Current main and v2026.6.10 partially preserve raw Telegram `<tg-emoji emoji-id="...">` HTML, but the central request remains: there is still no first-class message-tool or Telegram send/edit/caption entity surface for custom emoji IDs.

## What This Changes

Request Telegram custom emoji support across outgoing messages, captions, and replies, including custom emoji IDs and Telegram entity/HTML formatting.

## Best Possible Solution

Choose one maintainer-approved custom emoji representation, then wire it through the shared message tool, Telegram send/edit/caption/reply paths, docs, and focused regression tests while preserving the existing raw HTML fallback.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 expose no message-tool `entities`/`caption_entities` input and do not forward entity arrays through Telegram send/edit/caption paths. I did not run a live Telegram Premium custom emoji send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. Raw `<tg-emoji>` HTML support is useful partial coverage, but the best maintainable solution should define one typed entity/custom-emoji contract instead of adding an ad hoc bypass.
