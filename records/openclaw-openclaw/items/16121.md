## Summary

Keep this issue open: v2026.6.6 and current main can preserve raw Telegram `<tg-emoji>` HTML through text, caption, reply, and edit delivery, but OpenClaw still lacks the requested first-class agent-facing custom-emoji representation, and the existing discussion deliberately leaves that product/API choice unresolved.

## What This Changes

Add first-class Telegram custom emoji authoring and delivery for outgoing messages, captions, and replies.

## Best Possible Solution

Choose and document one canonical authoring contract: either explicitly support sanitized raw `<tg-emoji>` HTML with focused tests and user guidance, or add an offset-free typed custom-emoji representation that Telegram safely converts through chunking, captions, replies, edits, streaming, and fallback.

## Reproduction Assessment

Yes, at source level. Current code proves that sanitized raw HTML transport exists while a first-class custom-emoji message-tool or typed authoring representation remains absent.

## Solution Assessment

Unclear. Raw sanitized HTML is the narrowest existing mechanism and avoids caller-managed UTF-16 offsets, but maintainers must decide whether requiring Telegram-specific markup is an acceptable public agent-authoring contract.
