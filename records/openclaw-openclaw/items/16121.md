## Summary

Keep open. Current main has partial raw Telegram HTML passthrough for `<tg-emoji>`, but the requested first-class custom emoji support still lacks a shared representation, Telegram `entities`/`caption_entities` send surface, message-tool input, and caption/reply/edit/streaming integration.

## What This Changes

Request to add first-class Telegram custom emoji support for Telegram messages, captions, and replies.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving the current HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high confidence that first-class outgoing custom emoji/entity support is still incomplete.

## Solution Assessment

Not applicable as a patch review. Raw `<tg-emoji>` passthrough is useful partial support, but the maintainable solution still needs an explicit public representation and entity validation across send/caption/reply/edit paths.
