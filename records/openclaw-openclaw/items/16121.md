## Summary

Keep open. Current main and v2026.6.10 preserve part of the raw `<tg-emoji emoji-id="...">` HTML path, but the central request remains missing: there is no first-class custom emoji entity surface in the shared message tool or Telegram send/edit/caption paths, and the related items do not supersede this issue.

## What This Changes

The issue requests first-class Telegram custom emoji support for messages, captions, and replies.

## Best Possible Solution

Choose one maintainer-approved custom emoji representation, then wire it through the shared message tool, Telegram send/edit/caption/reply paths, docs, and focused regression tests while preserving the existing raw HTML fallback.

## Reproduction Assessment

Yes, source-level: current main exposes no message-tool `entities`/`caption_entities` input and does not forward entity arrays through Telegram send/edit/caption paths. I did not run a live Telegram Premium custom emoji send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. Raw `<tg-emoji>` HTML support is useful partial coverage, but the maintainable solution should define one typed custom emoji/entity contract instead of adding an ad hoc bypass.
