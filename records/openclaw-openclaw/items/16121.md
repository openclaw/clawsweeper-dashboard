## Summary

Keep open. Current main and the latest release preserve some raw `<tg-emoji>` HTML through Telegram rich-message rendering, but the core request still lacks a first-class entity/caption-entity/message-tool contract for custom emoji across sends, captions, edits, and plugin slash-command flows.

## What This Changes

The issue requests Telegram custom emoji support for messages, captions, and replies, including custom emoji identifiers and message-tool/send handling.

## Best Possible Solution

Choose the public API representation first, then implement validated Telegram custom emoji support through the Telegram plugin send/caption/edit/reply paths and the shared message tool, with docs and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. The message tool and Telegram schema/send surfaces still lack `entities` or `caption_entities`, although no live Telegram run was performed in this read-only review.

## Solution Assessment

Unclear until maintainers pick the API shape. Raw `<tg-emoji>` passthrough solves one path, but the remaining entity/caption/reply request needs a maintained contract rather than an allowlist-only patch.
