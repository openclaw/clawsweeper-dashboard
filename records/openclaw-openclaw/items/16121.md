## Summary

Keep open. Current main still only provides partial raw Telegram HTML passthrough for `<tg-emoji>`, while first-class outgoing custom emoji support through message-tool input, Telegram entities/caption_entities, captions, replies, and shared representation remains incomplete and already needs maintainer API-shape review.

## What This Changes

The issue requests Telegram custom emoji support for messages, captions, replies, identifiers, and outgoing display according to Telegram Bot API formatting support.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then implement validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving current HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high confidence that first-class outgoing custom emoji/entity support is still incomplete.

## Solution Assessment

Unclear until maintainers choose the public representation. Raw `<tg-emoji>` passthrough is useful partial support, but it is not the full maintainable solution for entity validation, message-tool input, captions, replies, chunking, and fallback policy.
