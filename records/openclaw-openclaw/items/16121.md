## Summary

Keep open. Current main and v2026.5.20 only preserve raw Telegram `<tg-emoji>` HTML, while first-class custom emoji entity support is still missing from the shared message-tool and Markdown IR contract plus Telegram send, caption, reply, edit, and streaming paths.

## What This Changes

Add first-class Telegram custom emoji support for messages, captions, replies, and custom emoji identifiers.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving raw HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, upstream-contract, shipped-release, and related-item inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw `<tg-emoji>` passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
