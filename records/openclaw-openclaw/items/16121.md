## Summary

Keep open. Current main now partially preserves raw Telegram `<tg-emoji emoji-id="...">` HTML, but the broader feature remains unresolved because OpenClaw still lacks a first-class custom emoji/entity representation in the shared message tool, Telegram send options, entity/caption entity paths, and documented API shape.

## What This Changes

Request support for Telegram custom emojis in OpenClaw messages, captions, and replies.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through messages, captions, replies/quotes, edits/previews, streaming, and the shared message tool while preserving the current HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and upstream API inspection give high confidence that first-class outgoing custom emoji/entity support is still incomplete on current main.

## Solution Assessment

Unclear. Raw `<tg-emoji>` passthrough is a useful partial solution, but it is not the full maintainable answer for message-tool input, entity validation, captions, replies/quotes, edits/previews, chunking, and fallback behavior.
