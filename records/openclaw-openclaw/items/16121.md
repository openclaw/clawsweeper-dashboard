## Summary

Keep open. Current main and v2026.6.6 preserve raw `<tg-emoji emoji-id="...">` HTML, but the central entity/message-tool request remains unsolved: the shared message tool has no custom-emoji entity fields and Telegram send, caption, edit, and durable-delivery paths still do not forward `entities` or `caption_entities`.

## What This Changes

Request support for Telegram custom emoji in outgoing messages, captions, replies, and agent/plugin message-tool sends.

## Best Possible Solution

Define one supported Telegram custom-emoji contract, then wire it through message-tool schema, Telegram channelData, send/edit/caption/delivery paths, docs, and focused regression tests while preserving raw HTML fallback behavior.

## Reproduction Assessment

Yes, by source inspection: `custom_emoji` entity arrays have no message-tool schema field and are not read or forwarded by Telegram send, caption, edit, or outbound adapter paths; no live Telegram run was performed.

## Solution Assessment

Unclear until maintainers choose the API shape. Raw `<tg-emoji>` pass-through is useful partial support, but it is not the best complete solution for plugin slash-command entity round-tripping or caption/edit entity support.
