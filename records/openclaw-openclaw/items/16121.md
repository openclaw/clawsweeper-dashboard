## Summary

Keep open. Current main and the latest release have partial raw `<tg-emoji>` HTML passthrough, but the requested first-class Telegram custom emoji support is still missing across shared Markdown/IR, entity-bearing message/caption send parameters, message-tool input, edits/previews, and streaming delivery paths.

## What This Changes

The issue requests first-class Telegram custom emoji support for messages, captions, and replies according to Telegram Bot API formatting and entity support.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving current HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, shipped-release, upstream-contract, and related-item inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw `<tg-emoji>` passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
