## Summary

Keep open. Current main and v2026.5.20 have useful raw `<tg-emoji>` HTML passthrough, but the requested first-class Telegram custom emoji/entity support still is not available through the shared message tool, Telegram send options, captions/entities, edits, or streaming paths.

## What This Changes

The issue requests first-class Telegram custom emoji support for outgoing messages, captions, replies, identifiers, and display behavior.

## Best Possible Solution

Define a stable custom emoji/entity contract, then implement validated Telegram support through the message tool, Telegram send/caption/reply/edit/streaming paths, docs, and focused regression tests while preserving raw HTML passthrough.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, shipped-release, upstream-contract, and related-item inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw `<tg-emoji>` passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
