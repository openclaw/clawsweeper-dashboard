## Summary

Current main and v2026.6.5 still preserve Telegram output as HTML without a typed custom-emoji identifier path from the message tool through text, caption, reply, edit, and streaming delivery. The closed entity-formatting PR is useful prior art but was not merged and does not supersede the remaining feature; maintainers still need to choose the durable shared-versus-Telegram API boundary.

## What This Changes

Add first-class Telegram custom emoji support for outgoing messages, captions, replies, edits, and streaming delivery while preserving custom emoji identifiers.

## Best Possible Solution

Define a narrow typed custom-emoji presentation contract, keep Telegram entity encoding and eligibility handling inside the Telegram plugin, and preserve it through chunking, captions, replies, edits, and streaming with HTML fallback unchanged for ordinary messages.

## Reproduction Assessment

Yes, at source level. Current main has no identifier-bearing custom emoji representation in the message tool or Markdown IR and no structured entity surface in Telegram text or caption delivery.

## Solution Assessment

No, merely allowlisting raw `<tg-emoji>` is not the best complete solution. The narrow maintainable approach is a typed presentation primitive with Telegram-owned serialization and coverage for every delivery path.

- Telegram eligibility constraints for bots using custom emoji need explicit validation and user-facing failure behavior.
