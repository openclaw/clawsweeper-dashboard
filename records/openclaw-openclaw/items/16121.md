## Summary

Keep open. Current main has partial raw Telegram HTML passthrough for `<tg-emoji>`, but the requested first-class custom emoji support still lacks a shared representation, `entities`/`caption_entities` send surface, message-tool input, and caption/reply/edit/streaming integration. https://github.com/openclaw/openclaw/issues/40848 is already closed as a duplicate of this issue, and https://github.com/openclaw/openclaw/pull/47562 is an open reactions-only PR that does not cover the message/caption/reply work.

## What This Changes

Request Telegram custom emoji support for messages, captions, and replies, including parsing/rendering custom emoji identifiers and sending them where standard emoji are supported.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives high confidence that first-class outgoing custom emoji/entity support is still incomplete.

## Solution Assessment

Not applicable as a patch review. Raw `<tg-emoji>` passthrough is useful partial support, but the maintainable solution still needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
