## Summary

Keep open. Current main partially preserves raw Telegram custom emoji HTML, but the shared message tool and Telegram send/edit/caption paths still cannot carry `entities` or `caption_entities`, so first-class outgoing custom emoji support remains unresolved.

## What This Changes

Request support for Telegram custom emoji rendering and handling in Telegram messages, captions, and replies, including custom emoji identifiers and message-tool sends.

## Best Possible Solution

Choose the public representation first, then wire validated `custom_emoji` entities through Telegram send/edit/caption/outbound paths and the shared message tool with docs and regression tests.

## Reproduction Assessment

Yes by source inspection: current schemas and send options omit entity arrays while Telegram sends and edits still use HTML parse mode; no live Telegram run was performed.

## Solution Assessment

The requested direction is valid, but the best fix is not another allowlist-only patch; it needs a maintained entity contract through the shared tool and Telegram plugin.
