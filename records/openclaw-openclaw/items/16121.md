## Summary

Keep open. Current main and v2026.5.12 have useful raw `<tg-emoji>` HTML passthrough, but the broader requested first-class outgoing Telegram custom emoji support is still missing from shared Markdown/IR, entity-bearing send/caption parameters, the message tool, edits/previews, and streaming delivery paths.

## What This Changes

Request first-class Telegram custom emoji support for outgoing messages, captions, replies, and agent-facing send surfaces.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving current HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, shipped-release, and related-item inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw `<tg-emoji>` passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
