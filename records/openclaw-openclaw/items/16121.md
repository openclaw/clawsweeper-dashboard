## Summary

Keep open. Current main and the latest release preserve raw Telegram `<tg-emoji>` HTML, but the first-class custom emoji/entity path requested here is still missing from the shared message-tool contract, Markdown IR, Telegram send/caption paths, and streaming/edit surfaces.

## What This Changes

Request support for Telegram custom emoji identifiers and rendering in outgoing messages, captions, and replies.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving raw HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, upstream Telegram API, current-main, latest-release, and related-item inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw `<tg-emoji>` passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
