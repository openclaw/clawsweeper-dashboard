## Summary

Keep open. Current main and the latest shipped tag have useful raw `<tg-emoji>` HTML passthrough, but they still do not provide first-class outgoing Telegram custom emoji support through a shared representation, entity-bearing send/caption parameters, the message tool, replies, edits, or streaming paths.

## What This Changes

Add first-class Telegram custom emoji support for messages, captions, replies, and shared message-tool sends.

## Best Possible Solution

Define a stable custom-emoji/entity contract, then wire validated Telegram support through message-tool sends, captions, replies/quotes, edits/previews, streaming, docs, and focused regression tests while preserving current HTML fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and release inspection give high confidence that first-class outgoing custom emoji/entity support remains incomplete on current main and v2026.5.12.

## Solution Assessment

Yes directionally, but not as a narrow automatic repair. Raw `<tg-emoji>` passthrough is useful partial support; the maintainable solution needs an explicit public representation and entity validation across send, caption, reply, edit, and streaming paths.
