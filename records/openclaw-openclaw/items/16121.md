## Summary

Keep open. Current main still lacks first-class outgoing Telegram custom emoji support for messages, captions, replies, edits/previews, and the shared message tool; related #47562 covers reaction work and the closed #13905/#40848 context does not ship the requested message/caption behavior.

## What This Changes

The issue requests Telegram custom emoji support in outgoing messages, captions, and replies, including custom emoji identifiers and rendering parity with standard emoji.

## Best Possible Solution

Define a small public custom-emoji/entity representation, then implement validated Telegram support across send, caption, reply, edit, delivery, draft streaming, and the shared message tool with focused docs and regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has no custom-emoji IR span, no outgoing entities/caption_entities send surface, and Telegram send/edit/delivery paths still rely on HTML parse_mode.

## Solution Assessment

Unclear. The requested direction is valid, but a raw <tg-emoji> passthrough would be too narrow because it would not cover message-tool input, captions, edits/previews, chunking, fallback, or UTF-16 entity validation.
