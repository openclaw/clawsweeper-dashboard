## Summary

Keep open: the PR targets a valid Windows exec-approvals persistence failure, but the proposed direct overwrite copy fallback can lose the existing approvals file on mid-copy errors, so it should not merge as-is.

## What This Changes

The branch adds an EPERM/EEXIST copy fallback for exec-approvals rename-overwrite failures, adds hard-link and cleanup tests, and updates the changelog for #77785.

## Best Possible Solution

Keep the Windows fallback, but make it failure-safe by preserving or restoring the existing exec-approvals.json if the copy path fails, with regression coverage for that error path.

## Reproduction Assessment

Yes for the relevant boundary: current main directly renames the temp file and the related report supplies Windows EPERM failures across shipped versions. I did not live-reproduce the native Windows locked-handle gateway case, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No, not yet: the fallback should not overwrite the live approvals file with fs.copyFileSync unless it also protects the old file from copy failure loss. The narrow maintainable solution is to keep the guarded EPERM fallback but add failure-safe preservation/restoration and a regression test for copy failure. ([nodejs.org](https://nodejs.org/download/release/v22.12.0/docs/api/fs.html?utm_source=openai))

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve approvals when fallback copy fails:** `src/infra/exec-approvals.ts:333`
  - body: `fs.copyFileSync(tempPath, filePath)` overwrites the live approvals file directly after rename fails. Node documents that this copy is not atomic and may remove the destination if an error occurs after opening it, so an ENOSPC or I/O failure can lose the existing approvals policy where current main would have left it intact; stage or restore the fallback so the old file survives copy errors. ([nodejs.org](https://nodejs.org/download/release/v22.12.0/docs/api/fs.html?utm_source=openai))
  - confidence: 0.94
