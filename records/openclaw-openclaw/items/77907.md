## Summary

Keep open: the fallback direction is useful and the latest head addresses the earlier restore and symlink-race blockers, but it still removes the post-write chmod on the normal rename path. Under a restrictive umask, that can make `exec-approvals.json` unreadable and cause OpenClaw to fall back to default `full`/`off` exec approvals.

## What This Changes

The PR adds a guarded Windows `EPERM`/`EEXIST` rename-overwrite fallback for exec approvals, regression tests for fallback safety, and a changelog entry for the linked Windows exec failure.

## Best Possible Solution

Keep the guarded fallback, but restore exact `0600` permissions after every successful write path and add restrictive-umask coverage for the normal rename path.

## Reproduction Assessment

Yes, source-level: current main directly renames the temp file over `exec-approvals.json`, and the linked Windows report gives concrete `EPERM` failures across shipped versions. The remaining PR regression is also source-reproducible from the removed chmod and Node's umask contract.

## Solution Assessment

No, not yet: the PR's fallback approach is narrow and the latest head fixes the earlier fd-safety issues, but the final file permission contract needs to be restored before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Restore permissions after successful rename:** `src/infra/exec-approvals.ts:717`
  - body: `renameExecApprovalsWithFallback` returns immediately on the normal rename path, and `writeExecApprovalsRaw` no longer runs the previous final chmod. `fs.writeFileSync(..., { mode: 0o600 })` is still subject to the process umask, so a service umask that masks owner bits can leave `exec-approvals.json` unreadable; `loadExecApprovals` then falls back to the default `full`/`off` policy instead of the persisted stricter approvals. Reapply `0600` after the successful write path and cover this with a restrictive-umask regression test.
  - confidence: 0.9
