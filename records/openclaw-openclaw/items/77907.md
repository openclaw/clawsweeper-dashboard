## Summary

Keep open for review/merge: current main still has a rename-only synchronous write path for exec approvals, and the latest PR head fixes the Windows fallback without the earlier permission, symlink, or copy-failure regressions.

## What This Changes

The PR replaces the exec approvals synchronous private-file-store write with a guarded temp-file rename that falls back to fd-validated copy on Windows EPERM/EEXIST, adds store regression tests, and updates the changelog.

## Best Possible Solution

Land the guarded exec-approvals writer and regression tests, then consider promoting the fallback into the shared fs-safe sync writer only if another store needs the same behavior.

## Reproduction Assessment

Yes, source-level: current main reaches a synchronous temp-file `renameSync` path for `exec-approvals.json`, and the linked Windows report gives concrete EPERM failures on shipped v2026.5.2 through v2026.5.4 builds. I did not run a native Windows locked-handle repro in this read-only review.

## Solution Assessment

Yes, the latest PR head is a narrow fix: it keeps the normal atomic rename path, falls back only on EPERM/EEXIST, validates the fallback fd, restores prior bytes on destructive copy failure, and reapplies owner-only permissions.
