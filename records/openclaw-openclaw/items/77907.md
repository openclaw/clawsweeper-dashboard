## Summary

Keep open: the PR addresses the source-reproducible Windows EPERM rename failure and fixed the earlier copy-failure data-loss concern with tests, but the latest fallback still has a symlink-swap TOCTOU before `copyFileSync`, so it is not merge-ready.

## What This Changes

The PR adds a guarded Windows rename-overwrite fallback for exec-approvals persistence, fallback cleanup and restoration tests, and a changelog entry for the related Windows EPERM issue.

## Best Possible Solution

Keep the Windows EPERM fallback, but make the fallback fail closed or write through a non-symlink-following path so the existing symlink and hard-link boundaries survive the copy path.

## Reproduction Assessment

Yes, for the relevant boundary: current main directly renames the temp file over `exec-approvals.json`, and the linked Windows report provides concrete EPERM failures across shipped versions. I did not live-run the native Windows locked-handle case, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No, not yet: the PR’s direction is narrow, but the current fallback validates the path and then opens it by name with `copyFileSync`, leaving a symlink-swap race. The safer solution is to keep the guarded fallback only if it preserves the existing path-safety guarantees under that race, with regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prevent symlink swaps before copying approvals:** `src/infra/exec-approvals.ts:360`
  - body: The fallback checks the destination, but `fs.copyFileSync(tempPath, filePath)` then opens the path by name. Make the fallback fail closed or avoid following swapped symlinks, and add a regression for that race.
