## Summary

Keep open: the Windows/npm lifecycle report is concrete, current main still lets the routed `status` path return without an explicit one-shot shutdown, and PR #74425 is an open closing PR rather than a merged fix.

## What This Changes

The issue reports that Windows npm CLI invocations such as `status` and `--version` leave short-lived `node.exe ... openclaw.mjs` processes alive after command completion.

## Best Possible Solution

Review and land PR #74425 or a narrower replacement that terminates short-lived Windows npm CLI invocations after `status` and root `--version` while preserving long-running gateway behavior.

## Reproduction Assessment

Yes. The reporter supplied concrete Windows 11 npm commands with process/timing evidence, and current main has a source-backed routed `status` path that returns without forced one-shot shutdown; I did not run a live Windows smoke from this Linux checkout.

## Solution Assessment

Unclear. A narrow one-shot CLI lifecycle fix is the right direction, but maintainers should confirm whether explicit `process.exit` or targeted cleanup of the lingering handle is the most maintainable implementation.
