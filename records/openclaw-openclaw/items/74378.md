## Summary

Keep open: the Windows npm `status` hang remains concrete, current main still lets the routed `status` success path return without a one-shot shutdown, and open PR #74425 references this issue with closing syntax but is not merged in the provided context.

## What This Changes

The issue asks for Windows npm CLI commands such as `status` and root version checks to terminate instead of leaving short-lived `node.exe ... openclaw.mjs` processes alive.

## Best Possible Solution

Review and land PR #74425 or a narrower replacement that shuts down short-lived Windows npm CLI invocations after `status` while preserving long-running gateway behavior.

## Reproduction Assessment

Yes. The reporter provided concrete Windows 11 npm commands with timing/process evidence, and current main has a source-backed routed `status` success path that returns without a forced one-shot process shutdown; I did not live-run Windows here.

## Solution Assessment

Unclear. A one-shot CLI lifecycle fix is the right boundary, but maintainers should decide whether the open PR's explicit `process.exit` approach or targeted cleanup of the lingering handle is more maintainable.
