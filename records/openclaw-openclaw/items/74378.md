## Summary

Keep open. The report has a concrete Windows/npm reproduction, current main still has a routed `status` path that returns without a success-path CLI shutdown, and open PR #74425 is already the implementation candidate that links this issue for closure.

## What This Changes

The issue reports Windows npm CLI commands such as `status` and `--version` leaving short-lived `node.exe ... openclaw.mjs` processes alive after command output.

## Best Possible Solution

Land PR #74425 or a narrow replacement that makes Windows npm one-shot CLI invocations such as `status` terminate promptly, preserves long-running gateway behavior, and adds focused coverage or Windows smoke proof for the lifecycle path.

## Reproduction Assessment

Yes. The reporter provided concrete Windows 11 npm commands with timing and process evidence, and current main has a source-backed `status` success path that returns without an explicit one-shot shutdown; I did not rerun Windows locally.

## Solution Assessment

Unclear. Keeping the issue open while PR #74425 is open is the right workflow, but maintainers should decide whether a wrapper-level explicit exit or narrower cleanup of the lingering handle is the best maintainable fix.
