## Summary

Keep open: the Windows npm CLI hang report is concrete, current main still has a routed `status` success path that returns without a one-shot shutdown, and open PR #74425 references this issue with closing syntax but is not merged.

## What This Changes

The issue asks OpenClaw CLI one-shot commands such as `status` and root `--version` to terminate cleanly on Windows npm installs instead of leaving `node.exe ... openclaw.mjs` processes alive.

## Best Possible Solution

Review and land PR #74425 or a narrower replacement that terminates short-lived Windows npm CLI invocations after `status` and root `--version` while preserving long-running gateway behavior.

## Reproduction Assessment

Yes. The reporter provided concrete Windows 11 npm commands with timing/process evidence, and current main has a source-backed routed `status` path that returns without an explicit one-shot shutdown; I did not live-run Windows from this checkout.

## Solution Assessment

Unclear. A narrow one-shot CLI lifecycle fix is the right direction, but maintainers should decide whether explicit `process.exit` or targeted cleanup of the lingering handle is the more maintainable implementation.
