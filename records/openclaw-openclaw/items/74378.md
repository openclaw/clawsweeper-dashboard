## Summary

Keep open: the Windows npm CLI hang report is concrete, current main does not contain a definitive merged `status` lifecycle fix, and open PR #74425 still references this issue with closing syntax but has not merged.

## What This Changes

Short-lived OpenClaw npm CLI commands on Windows should exit after `status` or `--version` instead of remaining as `node.exe ... openclaw.mjs` processes.

## Best Possible Solution

Review and land #74425 or a narrower replacement that makes Windows npm `status` and root `--version` invocations terminate cleanly without changing long-running gateway behavior.

## Reproduction Assessment

Yes. The reporter provided concrete Windows 11 npm commands with process timing evidence, and current main still has a routed `status` path that relies on natural event-loop drain rather than a proven one-shot shutdown; I did not live-run Windows here.

## Solution Assessment

Unclear. A narrow CLI lifecycle fix is the right direction, but maintainers should decide whether explicit `process.exit` or targeted cleanup of the lingering handle is the more maintainable implementation.
