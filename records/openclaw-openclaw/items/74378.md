## Summary

Keep open. This is a current Windows/npm CLI lifecycle regression with a concrete reproduction, current main still lets routed `status` return without normal-success termination, and open PR #74425 is a linked implementation candidate rather than a reason to close before merge.

## What This Changes

The issue requests that short-lived Windows npm CLI invocations such as `openclaw.mjs status` and version checks terminate promptly instead of leaving `node.exe` processes alive.

## Best Possible Solution

Review and land PR #74425 or a narrower replacement that makes short-lived Windows npm CLI commands terminate promptly after `status` and `--version`, with Windows/npm smoke coverage and no change to long-running gateway processes.

## Reproduction Assessment

Yes. The reporter provided concrete Windows 11 npm commands and timing/process evidence, and current main still shows the routed `status` success path returning without an explicit success shutdown.

## Solution Assessment

Unclear. Keeping this issue open while PR #74425 is open is the right state, but the final implementation should confirm whether explicit process exit or targeted cleanup of the lingering handle is the narrowest maintainable fix.
