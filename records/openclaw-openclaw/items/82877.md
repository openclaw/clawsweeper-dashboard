## Summary

Keep open: the linked bug is plausible, but this patch is not safe as written because it lets any basic cleanup text suppress the configured provider fallback, including the title-only SPA shell case that was previously fixed and documented. It also lacks after-fix real behavior proof from a real setup.

## What This Changes

The branch changes `web_fetch` so Readability failures try basic HTML cleanup before provider fallback, with updated error text and regression tests.

## Best Possible Solution

Revise the fix so simple-body pages can fall through to local cleanup without regressing configured provider extraction for JS-heavy or title-only shells, then update docs and tests for the chosen fallback-quality rule.

## Reproduction Assessment

No high-confidence live reproduction is available: the linked issue gives generic steps but no concrete sample or output. Source inspection does show the current provider-before-basic path and the PR’s mocked tests describe the failure shape.

## Solution Assessment

No: moving basic cleanup ahead of the provider and accepting any non-empty text reintroduces the title-only shell problem previously fixed for Firecrawl fallback. A safer solution needs a quality check or fallback strategy that preserves provider extraction for weak local output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
