## Summary

Keep open: current main and the latest release still derive the toggle from live launchd load state, and a false launchctl probe can clear the app LaunchAgent. The related landed fix removed KeepAlive/manual relaunch behavior, but it does not prove the after-update persistence problem is fixed.

## What This Changes

The issue asks for macOS Launch at login to remain enabled and auto-launch reliably across app restarts and Sparkle updates.

## Best Possible Solution

Define a durable user intent for Launch at login, reconcile the app plist/load state after startup and Sparkle updates, and prove it without reintroducing KeepAlive.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live: this review did not run a macOS Sparkle update, but source inspection shows a clear probe-false path where an existing LaunchAgent can be treated as off and then removed.

## Solution Assessment

Is this the best way to solve the issue? The existing KeepAlive removal is not enough; the best fix needs a maintainer-approved desired-state and reconciliation model for app launch-at-login across startup and updates.
