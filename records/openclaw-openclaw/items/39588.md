## Label Justifications

- `P2`: The issue is a user-visible macOS app persistence bug with limited blast radius and no evidence of a core runtime outage.
- `impact:session-state`: The affected behavior is persisted local app state drifting from LaunchAgent registration state.

## Summary

Keep open. Current main and the latest release still derive the macOS Launch at login toggle from live LaunchAgent health rather than durable user intent, so a stale or unloaded `ai.openclaw.mac` job can still make the setting appear off after an update.

## What This Changes

Ensure the macOS app’s Launch at login setting and `ai.openclaw.mac` registration persist across app restarts and app updates.

## Best Possible Solution

Persist the user’s Launch at login intent separately from launchd health, reconcile or re-register `ai.openclaw.mac` when intent is enabled, and keep the no-KeepAlive manual-quit behavior.

## Reproduction Assessment

No, not for the full signed Sparkle update and macOS login path. Source inspection does show a plausible current-main drift path where `LaunchAgentManager.status()` can return false and `AppState` copies that into the UI-bound setting.

## Solution Assessment

Yes, the narrow maintainable fix is to separate persisted user intent from launchd health, then repair or re-register the existing app LaunchAgent when intent is enabled. The related gateway LaunchAgent recovery does not solve this app toggle drift.
