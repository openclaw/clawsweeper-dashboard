## Summary

Keep open. Current main and the latest shipped tag still drive the macOS Launch at login UI from live LaunchAgent health rather than durable user intent, so the reported post-update drift remains source-backed.

## What This Changes

The issue reports that the macOS app's Launch at login setting appears to become disabled after app updates and no longer reliably auto-launches on login.

## Best Possible Solution

Persist the user's Launch at login intent separately from LaunchAgent health, reconcile or repair ai.openclaw.mac when that intent is enabled, and preserve the current no-KeepAlive manual-quit behavior.

## Reproduction Assessment

No high-confidence live reproduction of the full signed Sparkle update and logout/login path was performed. The source still shows a reproducible drift path: missing or unhealthy ai.openclaw.mac LaunchAgent state makes LaunchAgentManager.status() return false, and AppState copies that false value into the UI-bound setting.

## Solution Assessment

Yes, the best maintainable fix is narrow: separate persisted user intent from launchd health and reconcile the existing app LaunchAgent. The current shipped no-KeepAlive behavior should stay intact because it fixed a different manual-quit problem.
