## Summary

Keep open. Current main and latest v2026.6.1 still derive the visible macOS Launch at login setting from live LaunchAgent state, and the current source can turn an installed-but-unloaded app LaunchAgent into `set(false)`, which removes the plist instead of preserving user intent.

## What This Changes

The issue reports that the macOS app Launch at login preference appears to stop persisting after app updates.

## Best Possible Solution

Persist the user's launch-at-login intent separately from live launchd health, reconcile or re-register the app LaunchAgent after signed Sparkle updates when intent is enabled, and preserve explicit user-disable behavior.

## Reproduction Assessment

No high-confidence live update reproduction is available here. The source path is clear: an installed-but-unloaded app LaunchAgent makes `status()` return false, and AppState can propagate that false value into `set(false)`.

## Solution Assessment

No; the current implementation is not the best solution because it uses live launchd state as the setting source of truth. The safer fix is durable desired-state persistence plus launchd reconciliation after update/startup.
