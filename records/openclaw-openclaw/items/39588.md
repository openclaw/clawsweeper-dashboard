## Summary

Keep open. Current main still derives the macOS Launch at login toggle from live launchd status, so an unloaded or unhealthy `ai.openclaw.mac` LaunchAgent can make the setting appear disabled and can trigger the disable path again.

## What This Changes

The issue asks for the macOS app Launch at login preference to stay enabled and keep auto-launching across app restarts and Sparkle updates.

## Best Possible Solution

Persist the user's Launch at login intent separately from launchd health, reconcile or repair the app LaunchAgent when that intent is enabled, and keep the existing manual-quit behavior intact.

## Reproduction Assessment

No, not as a high-confidence live end-to-end reproduction of the signed Sparkle update and logout/login path. The current source does give a clear source-reproducible path: a missing or unloaded launchd registration makes `status()` false, and `AppState` feeds that false value back into the UI-bound setting.

## Solution Assessment

Yes as a direction. The narrow maintainable fix is to separate persisted user intent from launchd health and reconcile the existing macOS app LaunchAgent, rather than changing gateway LaunchAgent policy or reintroducing `KeepAlive`.
