- `P2`: This is a user-visible macOS app persistence bug with limited blast radius and no evidence of a core runtime outage.
- `impact:session-state`: The issue is about persisted local app state drifting from LaunchAgent registration state.

## Summary

Keep open. Current main still derives the macOS Launch at login toggle from live LaunchAgent health instead of durable user intent, so an update or stale launchd state can still make the UI appear off even when the user intended autostart to remain enabled.

## What This Changes

The issue asks for the macOS app's Launch at login setting to stay enabled across restarts, app updates, and the next macOS login.

## Best Possible Solution

Persist the user's Launch at login intent separately from launchd registration health, reconcile or re-register ai.openclaw.mac when intent is enabled, and preserve the current no-KeepAlive manual-quit behavior.

## Reproduction Assessment

No, not end to end: there is no high-confidence signed Sparkle update plus macOS login reproduction. Source inspection does show a current-main drift path where LaunchAgentManager.status() can return false and AppState copies that value into the UI-bound Launch at login setting.

## Solution Assessment

Yes, the narrow maintainable fix is to separate persisted user intent from launchd health, then repair or re-register the existing app LaunchAgent when intent is enabled. The related gateway LaunchAgent repair path does not solve this app toggle state drift.
