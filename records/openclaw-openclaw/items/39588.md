## Summary

Keep open. Current main and v2026.6.5 still derive the macOS Launch at login toggle from transient launchd loaded state and can write a false result back by removing the plist; the earlier merged fixes covered initialization deletion and manual-quit relaunch, not the durable intent/update drift reported here.

## What This Changes

The issue reports that macOS Launch at login does not reliably persist across app restarts or updates and may stop auto-launching after the user enables it.

## Best Possible Solution

Store launch-at-login intent independently from transient launchd loaded state, reconcile the app LaunchAgent after startup and Sparkle replacement, and verify both toggle continuity and actual next-login launch with a signed packaged app.

## Reproduction Assessment

No, not end to end with high confidence. Source inspection shows an installed-but-unloaded `ai.openclaw.mac` plist can read as false and be deleted through the setting writeback path, but the signed Sparkle update plus next-login sequence has not been reproduced on current main.

## Solution Assessment

No, a post-update reload alone is only a partial repair. The best fix is to separate durable user intent from transient launchd status and reconcile registration from that intent.
