## Summary

Keep open: current main and latest release still derive the macOS app Launch at login toggle from live launchd state and do not persist or reconcile a desired launch-at-login setting after Sparkle app replacement. The related merged LaunchAgent PR removed `KeepAlive`, but it did not solve the update-time persistence/UI drift reported here.

## What This Changes

The issue reports that the macOS app's Launch at login setting can appear disabled and stop autostarting after app updates.

## Best Possible Solution

Make the macOS app own a durable Launch at login desired-state contract and reconcile the app LaunchAgent after signed app replacement, with focused Swift coverage and signed update/login proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a plausible current-main drift path because the toggle is derived from live launchd state without a persisted desired-state reconciliation step, but a signed Sparkle update/login cycle is still needed.

## Solution Assessment

Unclear: repairing `AppState` and `LaunchAgentManager` is plausible, but the best fix depends on the maintainer choice between launchd-as-source-of-truth and persisted desired state with launchd repair.
