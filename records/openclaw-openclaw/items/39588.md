## Summary

Keep open. Current main still treats the macOS app Launch at login setting as live launchd state rather than durable user intent, so an unloaded/stale `ai.openclaw.mac` LaunchAgent can still make the UI look disabled and there is no app/Sparkle reconciliation path. The adjacent #40213 KeepAlive fix is present and shipped, but it only stopped relaunch-after-quit behavior and does not address this update-time persistence drift.

## What This Changes

The issue requests that the macOS app Launch at login preference remain enabled across restarts and Sparkle app updates, with UI state reconciled to the intended autostart behavior.

## Best Possible Solution

Keep the issue open and fix the app-owned Launch at login contract: persist the user's intended autostart preference, classify live launchd health separately from intent, repair or re-bootstrap `ai.openclaw.mac` on app startup and after Sparkle updates when intent is enabled, and preserve explicit user opt-out without reintroducing `KeepAlive`.

## Reproduction Assessment

Unclear for the full end-to-end update path: the report includes concrete local launchd/Sparkle evidence and current source gives a high-confidence source-level path for UI drift when live launchd state is missing or unloaded, but no signed Sparkle update plus logout/login reproduction was run here.

## Solution Assessment

Yes, the maintainable fix belongs in the macOS app LaunchAgent/state layer rather than gateway launchd or core update code. The #40213 KeepAlive-only change is a useful adjacent fix but is not sufficient for persisted intent or update-time reconciliation.
