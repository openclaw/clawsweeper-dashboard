## Summary

Keep open: current main and the latest release still derive the macOS app Launch at login toggle from live launchd state instead of a separately persisted desired preference, and I did not find a merged canonical fix for the update-related drift reported here. Adjacent work removed `KeepAlive` from the app LaunchAgent and addressed gateway LaunchAgent ownership, but that does not prove this update-correlated app auto-launch setting is fixed.

## What This Changes

The issue reports that the macOS app's Launch at login toggle and auto-launch registration stop persisting reliably after app updates.

## Best Possible Solution

Make the macOS app own a clear Launch at login desired-state model, reconcile the app LaunchAgent after signed app replacement/update, and prove the fresh-install plus update path with focused Swift tests and a live macOS smoke.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level partial yes: current code can drift because the UI follows live launchd status and disabling removes the plist, but the exact Sparkle-update sequence still needs a live macOS reproduction.

## Solution Assessment

Is this the best way to solve the issue? No close or automatic fix yet; the maintainable path is to decide the desired-state contract first, then reconcile launchd registration instead of relying only on `launchctl print` as the stored setting.
