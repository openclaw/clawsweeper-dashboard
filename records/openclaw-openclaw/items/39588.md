## Label Justifications

- `P2`: The shipped macOS behavior can silently disable an optional startup workflow after an update or unloaded-agent state, but it does not make the core runtime unusable.
- `impact:data-loss`: The issue concerns a persisted user preference and its LaunchAgent registration being silently reinterpreted or removed.

## Summary

Keep open. Current main and the latest v2026.6.5 release still derive the visible Launch at login preference from transient launchd loaded state, then write a false result back by deleting the existing plist; the report therefore remains necessary, although its Sparkle-update trigger still needs signed packaged-app reproduction.

## What This Changes

Preserve the macOS app’s Launch at login preference and working startup registration across application updates and subsequent logins.

## Best Possible Solution

Store launch-at-login intent independently of transient launchd state, reconcile the app registration after startup and Sparkle replacement, and verify both preference continuity and actual next-login launch with the signed packaged app.

## Reproduction Assessment

No, not end to end with high confidence. Current source establishes a reproducible installed-but-unloaded state that reads false and can delete the plist, but a signed Sparkle update followed by a login has not yet been exercised on current main.

## Solution Assessment

The reported direction is valid, but a post-update reload alone is not the best complete fix. The maintainable solution separates durable user intent from transient launchd status and then reconciles registration against that intent.
