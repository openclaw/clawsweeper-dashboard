## Summary

Keep open: current main and v2026.6.6 still derive the macOS Launch at login toggle from live launchd state, with no durable desired-state persistence or Sparkle post-update reconciliation. The related gateway LaunchAgent report at https://github.com/openclaw/openclaw/issues/28423 covers a different CLI/gateway entrypoint problem and its candidate PRs are closed unmerged, so it does not supersede this app preference drift report.

## What This Changes

The issue asks OpenClaw to keep the macOS app Launch at login setting enabled across app restarts and Sparkle app updates.

## Best Possible Solution

Define a durable macOS Launch at login desired-state contract and reconcile the app LaunchAgent after signed app replacement, with focused Swift coverage and signed update/login proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a plausible current-main path where a false live launchd status can be assigned into AppState.launchAtLogin and written back by the existing didSet side effect, but signed Sparkle update/login proof is still needed.

## Solution Assessment

Unclear. Repairing AppState and LaunchAgentManager is plausible, but the best fix depends on a maintainer decision about whether persisted desired state should override transient launchd status after app updates.
