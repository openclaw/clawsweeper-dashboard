## Summary

Keep open. Current main still derives the macOS Launch at login toggle from live launchd status, lacks a durable intent value for the existing setting, and the merged KeepAlive fix only addressed manual quit behavior, not update-time registration drift.

## What This Changes

The issue asks for macOS Launch at login to remain enabled and auto-launch reliably across app restarts and updates.

## Best Possible Solution

Persist the existing Launch at login user intent separately from launchd health, reconcile or repair ai.openclaw.mac when intent is enabled, and keep KeepAlive out of the desktop app LaunchAgent.

No high-confidence live reproduction of the full signed Sparkle update and logout/login path was performed. The current source still gives a source-reproducible drift path: launchd health failure makes status() return false, and AppState copies that into the visible toggle.

## Solution Assessment

Yes as a direction. The narrow maintainable fix is to separate persisted user intent from live launchd status and reconcile the existing app LaunchAgent, while leaving the already-shipped manual-quit KeepAlive behavior intact.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/39588 as a narrow macOS app bug. Current main derives the Launch at login UI toggle from LaunchAgentManager.status(), so a missing or unloaded ai.openclaw.mac LaunchAgent can erase the visible enabled state even when the user's intent was to keep launching at login. Add durable persistence for the existing Launch at login intent, use that intent for UI state, and reconcile or repair the ai.openclaw.mac LaunchAgent when intent is enabled and launchd health drifts. Preserve the shipped behavior from https://github.com/openclaw/openclaw/pull/40213: do not reintroduce KeepAlive for the desktop app LaunchAgent, and do not change gateway ownership or ai.openclaw.gateway behavior. Add focused Swift regression coverage for intent persistence/status drift and the generated plist invariants. Include a changelog entry because this is user-visible macOS behavior.
