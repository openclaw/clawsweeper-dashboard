## Summary

Keep open. Current main still makes the macOS Launch at login UI follow live launchd state rather than durable user intent, so update-time or unloaded LaunchAgent drift can still make the setting appear disabled even when the user intended autostart.

## What This Changes

Make the macOS app Launch at login setting remain enabled and continue autostarting after app updates.

## Best Possible Solution

Persist the user's launch-at-login intent in the macOS app, report live launchd health separately, and reconcile or re-bootstrap the desktop LaunchAgent on app startup and update completion when intent is enabled.

## Reproduction Assessment

Unclear for the full end-to-end update path: the issue includes concrete local launchd/Sparkle evidence, and current source gives a high-confidence path for UI drift when launchd state is missing or unloaded. I did not run a signed Sparkle update plus logout/login reproduction.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the macOS app LaunchAgent/state layer by separating persisted intent from live launchd health and reconciling the app LaunchAgent after startup or update; the #40213 KeepAlive-only fix is adjacent but insufficient.
