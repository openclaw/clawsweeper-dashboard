## Summary

Keep open. Current main still has the source-backed drift path: the macOS Launch at login toggle is UI-bound to `AppState.launchAtLogin`, which is initialized from live LaunchAgent health rather than durable user intent, and the latest release contains the same path.

## What This Changes

The issue reports that macOS Launch at login does not reliably remain enabled across app updates and the UI toggle can appear off despite LaunchAgent/login-item state.

## Best Possible Solution

Persist the user's Launch at login intent separately from launchd health, reconcile or repair `ai.openclaw.mac` when that intent is enabled, and preserve the current no-KeepAlive manual-quit behavior.

## Reproduction Assessment

No, there is not a high-confidence end-to-end reproduction of the signed Sparkle update and macOS login path. Source inspection does show a current-main drift path where `LaunchAgentManager.status()` can return false and `AppState` copies that into the UI-bound setting.

## Solution Assessment

Yes, the narrow maintainable direction is to separate persisted user intent from launchd health and reconcile the existing app LaunchAgent. Treating the gateway LaunchAgent update fixes as sufficient would not address the desktop app toggle drift.
