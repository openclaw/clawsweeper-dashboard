## Summary

Keep open: current main and the latest release still tie the macOS app Launch at login toggle to live launchd state, and I did not find a merged canonical fix for update-related desired-state drift. The adjacent KeepAlive and gateway LaunchAgent work helps nearby symptoms but does not prove this toggle now persists across signed app updates.

## What This Changes

The issue asks for macOS app Launch at login to remain enabled across restarts and app updates, with the UI toggle staying aligned with autostart behavior.

## Best Possible Solution

Make the macOS app own a clear Launch at login desired-state model, reconcile the app LaunchAgent after signed app replacement, and prove fresh-install plus update behavior on macOS.

## Reproduction Assessment

No high-confidence live reproduction for the exact Sparkle update sequence was established here. Source inspection does show a plausible current-main drift path because the UI follows launchd status and disabling removes the LaunchAgent plist rather than reading a durable desired preference.

## Solution Assessment

No automatic fix is not yet the best path because the desired-state contract needs maintainer choice first. The maintainable solution is to persist the intended Launch at login preference and reconcile launchd registration instead of treating `launchctl print` as the stored setting.
