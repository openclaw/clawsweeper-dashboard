## Summary

Keep open. Current main still lacks a first-class Gateway session envelope on `sessions.create` and `sessions.patch`, and same-author PR #74799 is an active implementation candidate rather than a cleanup-close reason before merge.

## What This Changes

The issue requests an additive session-owned runtime policy envelope on Gateway session create/patch RPCs, with persisted enforcement for tools, paths, bash commands, optional network scope, and later external authorization callbacks.

## Best Possible Solution

Keep this issue open until maintainers review and either land #74799 or choose a replacement design that adds a typed Gateway protocol field, persisted `SessionEntry` state, pre-execution fail-closed enforcement, create/patch coverage, tests, and docs. Network and external-callback claims should be added only if the implementation can enforce them reliably at the runtime boundary.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send an `envelope` property to `sessions.create` or `sessions.patch` on current main, and schema validation rejects it because both RPC schemas omit the field and disallow additional properties.

## Solution Assessment

Unclear. A session-owned envelope is a plausible additive solution for the Gateway-client gap, but the exact authorization boundary, path semantics, network scope, callback scope, and interaction with trusted policy/plugin hooks need maintainer security review; #74799 is the current candidate to evaluate.
