## Summary

Keep open. Current main still lacks a first-class Gateway session envelope, and same-author PR #74799 is already the active implementation candidate, so cleanup closure would race maintainer/security review.

## What This Changes

The issue requests an optional per-session runtime-policy envelope on Gateway `sessions.create` and `sessions.patch` to constrain tools, paths, bash commands, and network scope across a session's runs.

## Best Possible Solution

Keep this open until maintainers either land #74799 or choose a narrower accepted design that stores a typed session envelope, enforces it fail-closed before execution and after parameter rewrites, documents stable rejection semantics, and defers unenforceable network limits to runtime sandbox/proxy work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `sessions.create` or `sessions.patch` with `envelope`; current main rejects it because the closed TypeBox/AJV schemas omit that field.

## Solution Assessment

Unclear. A session-owned envelope is a plausible core Gateway solution, but the exact policy semantics and enforcement boundary need maintainer/security review before this should be considered solved.
