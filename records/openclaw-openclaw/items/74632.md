## Summary

Keep open. Current main still rejects an `envelope` on `sessions.create` and `sessions.patch` through closed Gateway schemas and has no session-owned enforcement path; #74799 is the paired open implementation candidate, so cleanup closure would race maintainer/security review.

## What This Changes

Requests an additive Gateway session envelope field on `sessions.create` and `sessions.patch` to persist and enforce per-session tool, path, and network policy.

## Best Possible Solution

Keep this open until maintainers land #74799 or choose a narrower accepted core design that stores a typed session envelope, enforces it fail-closed before execution and after parameter rewrites, documents stable rejection semantics, and treats network/external-callback policy as a reviewed follow-up if needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `sessions.create` or `sessions.patch` with `envelope`; current main rejects it because the closed TypeBox/AJV schemas omit that field.

## Solution Assessment

Unclear. A session-owned Gateway envelope is a plausible core solution, but the exact enforcement order, path matching, rejection semantics, and network/callback boundaries need maintainer/security review before this is considered solved.
