- `impact:security`: The requested envelope would become a tool-execution authorization boundary for paths, Bash, and possibly network behavior.
- `impact:session-state`: The missing capability is session-owned persisted policy state attached through Gateway session create/patch operations.

## Summary

Keep open. Current main still lacks the requested Gateway session-envelope contract, the same-author implementation at https://github.com/openclaw/openclaw/pull/74799 is still open, and the authorization semantics need maintainer/security review before this issue can be closed.

## What This Changes

Add a first-class per-session runtime-policy envelope to Gateway `sessions.create` and `sessions.patch` so clients can attach tool/path/network constraints to a session.

## Best Possible Solution

Keep this issue open while maintainers either land or replace https://github.com/openclaw/openclaw/pull/74799 with a reviewed, typed session-envelope design and defer network/callback semantics if they need a separate security boundary.

## Reproduction Assessment

Yes. Passing `envelope` to `sessions.create` or `sessions.patch` on current main is a high-confidence source-level reproduction because both closed schemas omit the field and AJV rejects unknown properties before storage.

## Solution Assessment

Unclear. A Gateway-owned session envelope is a plausible core direction, but maintainers still need to settle the policy semantics, enforcement layer, audit event shape, and whether network/callback behavior belongs in this same design.
