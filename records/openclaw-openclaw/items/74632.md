## Summary

Keep open. Current main still lacks the requested Gateway session-owned runtime envelope, and the same-author open PR #74799 is the active implementation candidate rather than a cleanup close or duplicate repair lane.

## What This Changes

The issue requests an additive per-session runtime-policy `envelope` on Gateway `sessions.create` and `sessions.patch`, persisted on session state and enforced before tool execution.

## Best Possible Solution

Keep the issue open while maintainers review #74799 or replace it with an accepted narrower design. The durable end state should be an additive protocol/schema/docs field, persisted `SessionEntry` state, pre-execution enforcement that still applies after policy/hook/approval rewrites, focused create/patch/tool-blocking coverage, and a separate explicit decision for network enforcement and out-of-process authorization callbacks.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send `envelope` in `sessions.create` or `sessions.patch` and current main rejects it because the TypeBox/AJV schemas omit the field with `additionalProperties: false`.

## Solution Assessment

Unclear. A session-owned envelope is a plausible narrow answer to the Gateway-client policy gap, but the exact enforcement layer and the network/callback portions need maintainer security review; #74799 is the active candidate to evaluate.
