## Summary

Keep open. Current main still lacks a first-class Gateway session envelope, and the same-author implementation PR at https://github.com/openclaw/openclaw/pull/74799 is open, conflicting, and security-sensitive, so closing this issue would preempt maintainer review. The May 22 performance comment is useful adjacent `sessions.create` context, but it is a separate performance question from the envelope API request.

## What This Changes

The issue requests an additive Gateway protocol field for `sessions.create` and `sessions.patch` so clients can attach per-session tool, path, Bash, network, and optional authorization policy.

## Best Possible Solution

Keep one reviewed Gateway/session policy design as the source of truth: land a rebased, security-reviewed version of the paired PR or replace it with a narrower approved envelope design, while deferring network/callback semantics if maintainers want a separate runtime boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `envelope` to `sessions.create` or `sessions.patch`; current main's closed schemas omit that field and AJV validation rejects unknown properties before storage.

## Solution Assessment

Unclear. A Gateway-owned session envelope is a plausible core solution, but maintainers still need to settle the exact authorization semantics, enforcement layer, audit event shape, and deferred network/callback scope before this becomes the best implementation path.
