## Summary

Keep open. Current main and v2026.6.8 still lack a first-class session-owned runtime envelope on `sessions.create` or `sessions.patch`; the linked implementation PR remains open, conflicted, and still needs maintainer/security product direction.

## What This Changes

The issue requests an additive gateway session envelope for `sessions.create` and `sessions.patch`, plus an optional external authorization callback for client-owned tool policy.

## Best Possible Solution

Keep this issue open as the canonical design anchor, then land a current-main, proof-backed version of the linked PR or explicitly split the envelope and external-callback pieces after maintainer/security review.

## Reproduction Assessment

Yes. Source inspection shows the current create/patch schemas omit `envelope` while rejecting unknown fields, and the Gateway handlers validate those schemas before processing; I did not run a live Gateway request in this read-only review.

## Solution Assessment

Unclear. A core Gateway/session envelope is plausible because plugin hooks cannot attach declarative policy through `sessions.create`, but the exact security semantics and optional callback/config surface need maintainer approval before this is the best fix.
