## Summary

Keep open. Current main still lacks a first-class session-owned runtime envelope in the Gateway session RPC contract, and same-author PR #74799 is the active implementation candidate rather than a cleanup-close reason.

## What This Changes

Request to add an optional per-session runtime envelope to `sessions.create` and `sessions.patch`, persist it on the session record, and enforce it on tool calls, with a possible external authorization callback follow-up.

## Best Possible Solution

Keep this issue open until maintainers review and either land #74799 or replace it with an accepted narrower design that adds a typed Gateway protocol field, persisted `SessionEntry` state, fail-closed pre-execution enforcement, create/patch/tool-blocking coverage, and docs. Network and external callback enforcement should be accepted only with a boundary that can reliably enforce those claims.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send `envelope` in `sessions.create` or `sessions.patch` and current main rejects it because the TypeBox/AJV schemas omit that field and disallow additional properties.

## Solution Assessment

Unclear. A session-owned envelope is a plausible additive solution for the Gateway-client gap, but the authorization boundary, path semantics, network constraints, and callback scope need maintainer security review; #74799 is the current candidate to evaluate.
