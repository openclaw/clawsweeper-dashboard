## Summary

Keep open. Current main still does not provide a first-class session-owned runtime envelope through the Gateway session RPC contract, and same-author PR #74799 is the active implementation candidate rather than a cleanup-close reason. The proposal also changes a tool-authorization boundary, so maintainer/security review is still needed before accepting a design.

## What This Changes

The issue requests an optional per-session runtime-policy envelope on `sessions.create` and `sessions.patch`, with pre-execution enforcement for tool, path, Bash, network, and optional external authorization constraints.

## Best Possible Solution

Keep this issue open until maintainers review and either land #74799 or replace it with an accepted narrower design that adds a typed Gateway protocol field, persisted `SessionEntry` state, fail-closed pre-execution enforcement, create/patch/tool-blocking coverage, and docs. Network and external callback enforcement should land only where OpenClaw can reliably enforce the claimed boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send `envelope` in `sessions.create` or `sessions.patch` on current main, and the closed TypeBox/AJV schemas reject it as an unexpected property before the handlers can persist anything.

## Solution Assessment

Unclear. A session-owned envelope is a plausible additive solution for the Gateway-client gap, but the exact authorization boundary, path semantics, network claims, and external callback behavior need maintainer/security review; #74799 is the current candidate to evaluate.
