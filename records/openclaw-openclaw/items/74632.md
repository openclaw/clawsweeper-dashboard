## Summary

Keep open. Current main still lacks a session-owned runtime envelope in the Gateway session RPC contract, and open PR #74799 is the active same-author implementation candidate rather than something to duplicate or close ahead of merge.

## What This Changes

The issue requests additive `sessions.create` and `sessions.patch` envelope fields for per-session tool/path/network policy enforcement, plus an optional external authorization callback.

## Best Possible Solution

Keep this issue open while maintainers review #74799 or replace it with an accepted narrower design. The durable end state should be an additive protocol field with typed schema/docs, persisted `SessionEntry` state, pre-execution enforcement that survives policy/hook/approval rewrites, focused create/patch/tool-blocking tests, and a separate explicit decision for network enforcement and external callbacks if they cannot be enforced reliably in the same layer.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send `envelope` in `sessions.create` or `sessions.patch` and current main rejects it because both TypeBox/AJV schemas omit the field and disable additional properties.

## Solution Assessment

Unclear. A session-owned envelope is a plausible narrow solution for the reported gap, but the exact enforcement layer, audit semantics, and network/callback scope need maintainer security review; #74799 is the active candidate to evaluate.
