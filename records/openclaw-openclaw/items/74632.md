## Summary

Keep open. Current main still lacks a first-class session-owned runtime envelope in the Gateway session RPC contract, and the same-author open PR #74799 is the active implementation candidate rather than a cleanup-close reason or a duplicate repair lane.

## What This Changes

Add an optional per-session runtime-policy envelope to Gateway `sessions.create` and `sessions.patch` so external clients can declare tool, path, command, and possibly network authorization for all runs in that session.

## Best Possible Solution

Keep the issue open until maintainers review and either land #74799 or replace it with an accepted narrower design. The durable end state should be an additive typed Gateway protocol field, persisted `SessionEntry` state, pre-execution enforcement that survives policy/hook/approval parameter rewrites, focused create/patch/tool-blocking tests, docs, and an explicit separate decision for network and out-of-process callback enforcement if those cannot be enforced reliably in the same layer.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: add `envelope` to a `sessions.create` or `sessions.patch` payload and current main rejects it because the AJV-backed schemas omit the field and set `additionalProperties: false`.

## Solution Assessment

Unclear. A session-owned envelope is a plausible narrow fix for the reported Gateway-client gap, but the network/callback scope and the authorization-boundary details need maintainer security review; #74799 is the active design candidate to evaluate.
