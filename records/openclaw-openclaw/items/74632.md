## Summary

Keep open. Current main still lacks a first-class Gateway session envelope, and same-author PR #74799 is already the active implementation candidate, so this needs maintainer/security review rather than cleanup closure.

## What This Changes

Add an optional per-session runtime-policy envelope to Gateway `sessions.create` and `sessions.patch` and enforce it for subsequent tool calls.

## Best Possible Solution

Keep this issue open until maintainers review and either land #74799 or replace it with an accepted narrower design that stores a typed session envelope, enforces it fail-closed before execution and after parameter rewrites, documents stable rejection semantics, and avoids unenforceable network promises.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send `envelope` in `sessions.create` or `sessions.patch`; current main rejects it because both TypeBox/AJV schemas omit that field and disallow additional properties.

## Solution Assessment

Unclear. A session-owned envelope is a plausible additive Gateway solution, but the exact authorization boundary, path semantics, network scope, callback behavior, and interaction with trusted policies/hooks need maintainer security review; #74799 is the current candidate to evaluate.

## Security Review

Status: needs_attention

Summary: The issue proposes a new session-level tool authorization boundary, so security review is required before accepting an implementation.

Concerns:

- **[medium] Review session authorization boundary:**
  - body: A persisted session envelope would participate in tool execution authorization. Maintainers should review fail-closed lookups, enforcement before execution and after trusted policy/hook/approval rewrites, path matching semantics, native runtime coverage, and stable denial events.
  - confidence: 0.9
- **[medium] Avoid unenforced network policy claims:**
  - body: The proposed `networkAccess` field is security-sensitive because hook-level command or URL scanning is not a complete egress boundary for Bash, SSH, or API tools. Network constraints should wait for sandbox/proxy-level enforcement or be explicitly out of scope.
  - confidence: 0.82
