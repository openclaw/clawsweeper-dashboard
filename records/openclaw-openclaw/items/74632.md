## Summary

Keep open. Current main and v2026.6.8 still lack a session-owned envelope on `sessions.create` or `sessions.patch`; the linked implementation PR is open but unmerged and still needs maintainer/security review, so this issue remains the canonical product/security design anchor.

## What This Changes

Request adds an optional session-owned runtime policy envelope to `sessions.create` and `sessions.patch`, plus a possible gateway authorization callback, so external Gateway clients can constrain tools, paths, commands, and network per session.

## Best Possible Solution

Keep this issue open as the canonical design anchor, then either land a maintainer-approved, proof-backed version of the linked implementation PR or explicitly narrow/close that direction after security review.

## Reproduction Assessment

Yes, source-level: current schemas reject unknown `envelope` fields, Gateway handlers validate those schemas, and current docs/release docs do not expose the capability. I did not run a live Gateway scenario in this read-only review.

## Solution Assessment

Unclear: a core Gateway/session envelope is plausible because current plugin hooks cannot attach declarative policy through `sessions.create`, but the exact security semantics and optional callback/config surface need maintainer approval before implementation is the best fix.
