## Summary

Keep open. Current main still lets the terminal doctor/gateway repair path install, start, or restart the standalone macOS gateway LaunchAgent without detecting that the Mac app owns the local runtime, and open PR #73974 already references this issue as its closing target.

## What This Changes

Prevent macOS CLI doctor/gateway repair from creating or restarting a standalone `ai.openclaw.gateway` LaunchAgent when the Mac app `ai.openclaw.mac` LaunchAgent owns the local gateway.

## Best Possible Solution

Make macOS gateway ownership explicit: terminal doctor/gateway repair should detect an installed or loaded app LaunchAgent and skip or defer standalone gateway install/start/restart with clear output, while app and CLI docs describe one canonical owner and tests cover duplicate LaunchAgent prevention.

## Reproduction Assessment

Yes for the code path: the issue has concrete macOS steps, and current source shows `doctor --fix` can still reach standalone LaunchAgent install or restart without checking `ai.openclaw.mac`. A live macOS launchd run was not performed here.

## Solution Assessment

Yes, the direction in open PR #73974 appears to be the right boundary: detect app ownership in CLI repair before service install/start/restart and add focused regression coverage. The issue should remain open until that PR is merged, replaced, or closed.
