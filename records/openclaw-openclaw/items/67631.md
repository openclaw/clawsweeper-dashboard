## Summary

Keep open. Current main still leaves the terminal doctor/gateway repair path able to install, start, or restart the standalone macOS Gateway LaunchAgent without detecting that the macOS app LaunchAgent owns the local runtime, and open PR #73974 is already the closing implementation candidate.

## What This Changes

The issue reports duplicate macOS launchd ownership when the app and CLI doctor/gateway repair paths both manage the local Gateway, causing duplicate processes and port conflicts.

## Best Possible Solution

Make macOS Gateway ownership explicit: doctor/gateway repair should detect an installed or loaded ai.openclaw.mac app LaunchAgent and skip or defer standalone ai.openclaw.gateway install/start/restart with clear output, backed by focused tests and docs.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current main shows the relevant doctor --fix path can still install or restart the standalone LaunchAgent without checking for ai.openclaw.mac; a live launchd run was not performed here.

## Solution Assessment

Yes as a direction. The narrowest maintainable fix is the approach already proposed in PR #73974: detect app ownership before CLI service repair and add focused regression coverage, then close this issue after that PR or an equivalent replacement lands.
