## Summary

Keep open. Current main and the latest release still leave terminal doctor/gateway repair paths able to install, bootstrap, start, restart, or rewrite the standalone `ai.openclaw.gateway` service without detecting an app-owned `ai.openclaw.mac` runtime, and the open closing PR for this same bug remains unmerged.

## What This Changes

Request that macOS app and CLI doctor/gateway repair paths avoid creating or repairing duplicate LaunchAgents so only one local gateway owns port 18789.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73974 or an equivalent corrected fix that requires app-specific launchd evidence before CLI install/start/restart/config repair, with focused regression coverage and docs.

## Reproduction Assessment

Yes, source-backed rather than live macOS. The reported steps align with current `maybeRepairGatewayDaemon`, `maybeRepairGatewayServiceConfig`, and Mac app launchd code paths that can operate on `ai.openclaw.gateway` while `ai.openclaw.mac` exists.

## Solution Assessment

Yes as a direction, but not by closing this issue now. The narrow maintainable solution is to land the linked PR or a replacement that validates app-owned launchd state before suppressing or performing CLI gateway repair.
