## Summary

Current `main` still has source-backed CLI doctor paths that can bootstrap, install, restart, or rewrite the default macOS Gateway LaunchAgent without an app-owned launchd guard. The issue is also paired with an open closing PR, so it should stay open until that PR or a corrected successor lands.

## What This Changes

The issue reports that the macOS app and CLI doctor flow can both manage LaunchAgents for the local gateway, creating duplicate gateway processes and port conflicts.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81725 or a corrected successor that guards every default CLI Gateway lifecycle mutation on a precise app-owned local-mode signal while preserving named profile services.

Yes. The current source path reaches default LaunchAgent bootstrap, install, start/restart, and service-config repair without an app-owned launchd guard, and the issue comments describe the matching conflict and workaround.
