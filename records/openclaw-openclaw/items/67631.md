## Summary

Keep open. Current main still lets the terminal doctor/gateway repair path install, bootstrap, start, or restart the standalone macOS Gateway LaunchAgent without detecting that the macOS app LaunchAgent owns the local runtime, and open PR #73974 is already the closing implementation candidate.

## What This Changes

Prevent the macOS app and npm CLI doctor/gateway repair flows from creating or running competing LaunchAgents and duplicate gateway processes on the same local port.

## Best Possible Solution

Make macOS gateway ownership explicit: CLI doctor/gateway repair should detect an installed or loaded `ai.openclaw.mac` app LaunchAgent and skip or defer standalone `ai.openclaw.gateway` install/start/restart with clear output, while docs and focused regression tests cover duplicate LaunchAgent prevention.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current main shows the relevant `doctor --fix` code path can still install or restart the standalone LaunchAgent without checking for `ai.openclaw.mac`; a live launchd run was not performed here.

## Solution Assessment

Yes as a direction. The narrowest maintainable fix is the direction already proposed in open PR #73974: detect app ownership before CLI service repair and add focused tests/docs, then close this issue only after that PR or an equivalent replacement lands.
