## Label Justifications

- `P1`: The issue reports a concrete Telegram authorization bypass that can trigger wrong bots and sessions in real multi-bot forum groups.
- `impact:security`: Disabled Telegram topic/group policy is an authorization boundary and native commands currently bypass that boundary.
- `impact:message-loss`: Commands sent in one forum topic can be processed or answered by non-owning bots instead of only the intended topic/account route.
- `impact:session-state`: The report describes stale topic session creation for bots that should have ignored the command.

## Summary

Keep open. Current main and the latest release still authorize Telegram native commands without topic/group `groupPolicy` overrides, and the linked open PR is the active fix path rather than a reason to close this issue before merge.

## What This Changes

The issue asks Telegram native slash commands in forum topics to respect topic-level `groupPolicy: "disabled"` so only the bot/account assigned to that topic handles the command.

## Best Possible Solution

Land or replace the linked PR with a fix that uses topic/group-aware policy resolution for native commands, silently drops commands for effective disabled topics, and adds focused regression coverage plus redacted Telegram forum proof.

## Reproduction Assessment

Yes, at source level. Configure a Telegram forum topic or inherited group override with `groupPolicy: "disabled"` while the account/default policy would otherwise allow commands, then send a native slash command; current main passes topic config into the helper but disables topic/group override resolution.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse the existing topic/group-aware policy resolver for native commands, preserve silent disabled-topic drops for non-owning bots, and cover that path with a focused test plus live Telegram proof.
