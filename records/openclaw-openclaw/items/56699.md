## Label Justifications

- `P1`: The issue reports a concrete Telegram command authorization bypass that can trigger wrong bots and sessions in real multi-bot forum groups.
- `impact:security`: Disabled Telegram topic/group policy is an authorization boundary and native commands currently bypass that boundary.
- `impact:message-loss`: Commands sent in one forum topic can be processed by non-owning bots instead of only the intended topic/account route.
- `impact:session-state`: The report describes stale topic session creation for bots that should have ignored the command.

## Summary

Keep open. Current main and the latest release still skip effective topic/group `groupPolicy` overrides for Telegram native commands, and an open linked PR with closing syntax is the active fix path rather than a reason to close this issue before merge.

## What This Changes

The issue asks for Telegram native slash commands in forum topics to respect the same effective topic/group `groupPolicy: "disabled"` filtering as regular messages.

## Best Possible Solution

Land or replace the linked PR with a fix that uses topic/group-aware policy resolution, silently drops effective disabled-topic commands for non-owning bots, and adds focused regression coverage plus redacted Telegram forum proof.

## Reproduction Assessment

Yes. A high-confidence source-level repro is to configure a Telegram forum topic or inherited group override with `groupPolicy: "disabled"` while the account/default policy would otherwise allow commands, then send a native slash command; current main passes the topic config into the helper but disables topic/group override resolution.

## Solution Assessment

Partly. Reusing the existing override-aware resolver is the narrow maintainable fix, but the final patch should preserve silent disabled-topic behavior for non-owning bots and add regression coverage plus real Telegram proof.
