## Label Justifications

- `P1`: The issue is a concrete Telegram authorization bypass that can trigger the wrong bots and sessions in real multi-bot forum groups.
- `impact:security`: Disabled topic/group policy is an authorization boundary, and native commands currently bypass that boundary.
- `impact:message-loss`: Commands sent in one topic are processed by all bots in the group instead of only the intended topic/account route.
- `impact:session-state`: The report describes stale topic session creation for bots that should have ignored the command.

## Summary

Keep open. Current main and the latest stable release still skip topic/group `groupPolicy` overrides for Telegram native commands, and the open linked fix PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks for Telegram native slash commands in forum topics to respect per-topic or per-group `groupPolicy: "disabled"` so non-owning bots do not process commands or create sessions.

## Best Possible Solution

Make Telegram native-command authorization use the effective topic/group policy path, silently drop commands where the effective topic/group policy is disabled, and add focused regression coverage plus redacted Telegram forum proof.

## Reproduction Assessment

Yes. A high-confidence source-level repro is to configure a Telegram forum topic or inherited group override with `groupPolicy: "disabled"` while account/default policy would otherwise allow commands, then send a native slash command; current main passes the topic config into the helper but disables topic/group override resolution.

## Solution Assessment

Partly. Reusing the existing override-aware resolver is the narrow maintainable fix, but the final patch should also preserve silent disabled-topic behavior for non-owning bots and add regression coverage plus real Telegram proof.
