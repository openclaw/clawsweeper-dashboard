## Label Justifications

- `P2`: The issue reports intermittent but workflow-breaking multi-agent orchestration failure with a bounded affected surface and no evidence of a core-wide outage.
- `impact:session-state`: A completed child can leave the parent orchestration session in a misleading waiting or terminal state because completion processing is not enforced.

## Summary

Current main still relies on a model-mediated follow-up turn after `sessions_yield`; it has no invariant requiring the parent to acknowledge a child completion before ending, so the reported orchestration reliability gap remains unresolved. Keep this open for a maintainer decision on completion-state semantics and an independent live reproduction.

## What This Changes

Require deterministic parent-agent handling of subagent completion after `sessions_yield` instead of relying on the model to notice and act on the next-message context.

## Best Possible Solution

Represent yielded orchestration as explicit runtime state with pending completion events and a defined acknowledgement or terminal rule, then validate it with deterministic multi-child tests and a same-provider live gateway scenario.

## Reproduction Assessment

Yes for the architectural failure mode: current main delivers completion as steering input or a new agent message without a runtime acknowledgement invariant. The reported frequency still needs an independent live reproduction using the same provider/model and a captured parent transcript.

## Solution Assessment

The requested stronger harness is directionally correct, but the best fix is not additional prompt wording; maintainers should first define an explicit pending-completion and terminal-state contract across yield, wake, cancellation, and multi-child behavior.
