## Summary

This issue should stay open: current main and v2026.6.5 still route an inactive yielded requester completion through a model-mediated inter-session agent turn, and delivery success does not prove the parent consumed the specific completion. The report is credible and source-backed, but the fix needs a maintainer decision on the deterministic completion-consumption contract rather than an automatic cleanup close.

## What This Changes

The issue requests a stronger sessions_yield/subagent completion mechanism so yielded parent agents cannot silently ignore completion events delivered as passive context.

## Best Possible Solution

Define a narrow runtime-owned pending-completion contract for yielded requester sessions that forces each child completion to be consumed, explicitly dismissed, or safely carried forward before the parent can finish.

## Reproduction Assessment

Yes at source level, but not as a fresh live run here: current main shows subagent completion is delivered as an inter-session agent turn and accepted based on the parent turn's visible or side-effect output. The reporter also supplied repeated v2026.6.5 observations where the parent ignored the completion.

## Solution Assessment

Unclear. Prompt strengthening alone is not the best fix; a runtime-owned pending-completion contract is more reliable, but maintainers must decide the allowed final, re-yield, and explicit-dismiss semantics before implementation.
