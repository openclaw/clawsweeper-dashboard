## Label Justifications

- `P2`: This is a credible core multi-agent reliability bug with limited orchestration scope, but it is not a core-wide outage or confirmed data-loss emergency.
- `impact:session-state`: The runtime can mark child work complete while the parent orchestration session terminates without incorporating that completion, leaving logical session state inconsistent.

## Summary

The report remains necessary: release v2026.6.5 and current main both end the yielded parent run and deliver completion through a new requester-agent turn, so delivery can succeed while the model fails to continue the orchestration loop. The failure is credible and source-backed, but the correct deterministic completion-consumption contract needs maintainer product judgment and overlaps without being fully subsumed by the older resume-session report at https://github.com/openclaw/openclaw/issues/81490.

## What This Changes

Require subagent completions received after `sessions_yield` to be deterministically consumed by the parent orchestration flow instead of relying only on a new model turn to notice passive context.

## Best Possible Solution

Add canonical runtime-owned pending-completion state for yielded requesters and prevent the resumed completion turn from yielding or terminating without explicitly consuming, resolving, or safely carrying forward each required completion event.

## Reproduction Assessment

Yes at the source-contract level: current main deterministically makes the yielded parent idle and starts a separate model-mediated completion turn, which cannot guarantee that the model processes the event. The reporter also supplies three same-scenario observations on v2026.6.5, but this read-only review did not independently run the model workflow.

## Solution Assessment

Unclear. Prompt strengthening alone is not the best fix; a runtime-owned completion state is narrower and more reliable, but maintainers must decide the allowed terminal and re-yield semantics before implementation.
