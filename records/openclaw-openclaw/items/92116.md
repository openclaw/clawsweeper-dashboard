## Summary

Current main intentionally ends the parent run after `sessions_yield` and routes a later subagent completion through a new direct agent turn; the code and tests prove transport and turn scheduling, but they do not guarantee that the model semantically consumes the completion or advances the orchestration plan. The report is therefore still necessary, though the requested guarantee needs a maintainer product decision because no harness can force correct model reasoning without defining a deterministic lifecycle and recovery policy.

## What This Changes

Require yielded parent agents to reliably process subagent completion events instead of depending solely on ordinary prompt-context interpretation.

## Best Possible Solution

Define a closed yielded-parent lifecycle in core: completion arrival atomically schedules exactly one structured continuation, records which completion must be handled, and transitions only to re-yielded, completed, or bounded-recovery states with dedupe and cross-provider tests.

## Reproduction Assessment

Yes at medium confidence: current source and tests reproduce the exact idle-parent/direct-completion architecture, and the reporter supplies repeated live traces showing divergent model outcomes. A controlled live current-main run is still needed for high-confidence behavioral reproduction because the silent ignore is probabilistic and the setup includes an external persona package.

## Solution Assessment

No, stronger prompt wording or ordinary passive context is not the best fix. The narrow maintainable direction is an explicit core lifecycle and recovery contract using the existing yielded state, but maintainers must first define what observable action counts as consuming a completion.
