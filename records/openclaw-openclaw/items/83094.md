## Summary

Keep open. The PR implements a useful narrow handoff primitive that matches the prior proposal direction, but it is a draft external feature PR with only test output as proof and a correctness issue where agent-start failures can report an accepted handoff.

## What This Changes

The PR adds a Phase 1 `sessions_send` handoff contract with stable handoff IDs, structured ack data, a best-effort JSONL ledger, control-token outcome recording, docs, and focused tests.

## Best Possible Solution

Keep the Phase 1 primitive narrow, fix the error-path ack semantics, add regression coverage for start failure, and land only after maintainer acceptance plus redacted real `sessions_send` proof.

## Reproduction Assessment

Yes for the review finding from source inspection: when the `agent` call throws, `startAgentRun` returns `status: "error"` while spreading the prebuilt accepted handoff. I did not run a live reproduction in this read-only review.

## Solution Assessment

No as currently patched. The implementation should return a rejected or failed handoff and record a terminal event when target run startup fails, and the PR still needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Return a non-accepted handoff when agent start fails:** `src/agents/tools/sessions-send-tool.ts:186`
  - body: `handoff` is built as `accepted` before either `startAgentRun` call, and this catch path now spreads that same object into the error result when `callGateway({ method: "agent" })` throws. That means a target run that was never accepted can return `status: "error"` with `handoff.status: "accepted"`, while the ledger only has the earlier queued `created` event. Build a rejected/failed handoff and record a terminal event for this path before returning the start error.
  - confidence: 0.88
