## Summary

Keep open: the current main branch still contains the reported Windows `claude-cli` spawn path and a plausible Claude stream-json empty-final path that can leave Telegram without a visible reply. The Windows spawn half is already tracked by a canonical open PR, but the silent-final half is not fully superseded.

## What This Changes

The issue reports that a first-time Windows Telegram setup using the Claude CLI backend either fails to spawn Claude or completes a Claude turn without sending a visible Telegram reply.

## Best Possible Solution

Land or replace the Windows shim resolver from the canonical process PR, then add a focused Claude stream-json regression that preserves visible assistant text when the terminal result is empty, with redacted Windows Telegram or equivalent runtime proof before closing this issue.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Windows Telegram setup was run here; source inspection strongly supports the Windows spawn failure and shows a plausible current-main empty-final path for the silent Telegram reply.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: finish the no-shell Windows command-resolution work through the existing process PR, and handle the empty Claude stream-json final at the parser/live-session boundary rather than changing Telegram delivery policy or adding a config knob.
