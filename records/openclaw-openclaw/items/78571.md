## Summary

Keep open: current main still leaves the Windows `claude-cli` launch path unresolved for the reported Telegram setup, and the later text-delta bridge only partially addresses the silent-output side of the report.

## What This Changes

The issue reports that a first-time Windows Telegram setup using Claude CLI cannot produce bot replies because Claude launch and output parsing fail through the gateway path.

## Best Possible Solution

Fix the Claude CLI launch/output path by resolving Windows Claude wrappers through the existing shell-free resolver, ensuring text-delta-only turns become visible or fail visibly, and documenting a complete Windows-safe `claude-cli` setup.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection is enough to reproduce the default Windows launch risk: the bundled backend emits bare `claude`, the CLI runner forwards it to the child supervisor, and only npm/pnpm/yarn/npx get Windows shim handling.

## Solution Assessment

Yes for the repair direction: the narrow maintainable fix belongs in the Claude CLI backend/process-launch/output-dispatch path, not Telegram-specific policy. A broad `shell: true` fallback is not the safest default compared with the existing Windows wrapper resolver.
