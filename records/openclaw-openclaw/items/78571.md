## Summary

Keep open: current main still leaves the reported Windows `claude-cli` launch path unresolved, while the newer text-delta bridge only partially addresses the silent-reply side of the report.

## What This Changes

The issue reports that a first-time Windows Telegram setup using the Claude CLI backend cannot produce a visible bot reply.

## Best Possible Solution

Resolve Claude CLI Windows wrappers through the existing shell-free spawn resolver, make Claude stream-json output produce a visible Telegram reply or explicit failure, and document a complete Windows-safe setup.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection is enough to reproduce the default Windows launch risk: the backend emits bare `claude`, both CLI runner paths pass it through, and the child adapter only special-cases package-manager shims.

## Solution Assessment

Yes for the repair direction, but not for closure: the narrow maintainable fix belongs in the Claude CLI launch/output path, preferably via the existing shell-free Windows resolver rather than a broad `shell: true` fallback.
