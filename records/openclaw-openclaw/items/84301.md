## Summary

Keep open: current main and the latest release still use a fixed Dream Diary narrative wait budget and fixed detached narrative concurrency, while slow serial local model backends can plausibly exceed that budget. The narrowest path is to wire the existing dreaming execution timeout shape through the narrative wait path and document the supported config surface.

## What This Changes

Request to make Dream Diary narrative timeout, and possibly narrative concurrency, configurable for slow or serial local model backends.

## Best Possible Solution

Honor a documented `dreaming.execution.defaults.timeoutMs` plus per-phase `execution.timeoutMs` for Dream Diary `waitForRun`, keep the 60s default, and defer concurrency tuning unless maintainers choose that broader policy.

## Reproduction Assessment

No high-confidence live reproduction is available from the issue, but the source path is clear: current main always waits 60000 ms for each Dream Diary narrative and logs/returns when the subagent result is not ok. The local-provider serialization claim still needs runtime logs or a focused regression test to prove the exact backend behavior.

## Solution Assessment

Yes for the timeout portion: the maintainable fix is to pass the already-parsed dreaming execution `timeoutMs` into `waitForRun`, update schema/docs, and add regression coverage. Concurrency should stay at the current default unless a separate maintainer decision says to expose it.
