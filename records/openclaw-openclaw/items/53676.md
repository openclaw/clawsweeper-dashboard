## Summary

Keep open. This is a plausible Google Chat bug-fix PR, but it is not merge-ready because the issuer-only auth path is not proven through the real google-auth-library verifier, the diagnostics portion is partially superseded by current main, and the PR still lacks real Google Chat behavior proof.

## What This Changes

The PR changes Google Chat webhook JWT issuer/email matching, auth rejection diagnostics, sender email fallback behavior, and related tests.

## Best Possible Solution

A rebased or replacement Google Chat fix should preserve current per-target diagnostics, validate real Google token shapes against google-auth-library behavior, keep add-on principal binding strict unless maintainers approve otherwise, and add focused sender-email fallback coverage.

## Reproduction Assessment

Yes for the PR review findings: source and dependency inspection show the issuer-only checks run behind `verifyIdToken()`'s issuer gate, and the PR diagnostic helper logs through `targets[0]`. Live Google Chat delivery was not exercised.

## Solution Assessment

No. Current main already has a better diagnostics implementation, and the issuer-only auth path needs real-token validation or a different verifier path before widening accepted claims.

## Review Findings

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-only acceptance checks run after `verifyIdToken()`, but google-auth-library 10.6.2 rejects issuers outside its default allowlist before returning a payload. Tokens with `iss: chat@system.gserviceaccount.com` or `service-...@gcp-sa-gsuiteaddons...` never reach this branch, so the mocked tests do not prove the intended production 401 fix.
  - confidence: 0.88

  - body: `logAuthFailure()` always selects `targets[0]` as the logger even though collected rejection reasons may belong to later accounts on the same webhook path. That can route diagnostics through the wrong account runtime sink in multi-account deployments.
  - confidence: 0.83
