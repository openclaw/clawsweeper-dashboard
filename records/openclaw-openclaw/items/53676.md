## Summary

Keep open: the branch is useful context, but it should not merge as-is because the issuer-only auth path is blocked by the pinned verifier and the diagnostics work is already superseded on current main.

## What This Changes

This PR changes Google Chat app-url/add-on token matching, adds webhook auth rejection diagnostics and tests, and falls back to event.user.email for sender personalization.

## Best Possible Solution

A replacement or carefully rebased Google Chat fix should preserve current per-target diagnostics, validate real token shapes against google-auth-library behavior, keep add-on principal binding strict, land the sender-email fallback with regression coverage, and coordinate broader docs/auth work through #65326.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows issuer-only checks after verifyIdToken() and a first-target diagnostic logger in the PR diff. Live Google Chat delivery was not exercised.

## Solution Assessment

No. The diagnostics change is superseded by current main, and the issuer-only auth path does not match the pinned verifier behavior; a rebased replacement should prove real JWT claim shapes first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-based acceptance runs only after verifyIdToken() returns, but google-auth-library 10.6.2 rejects issuers outside accounts.google.com, https://accounts.google.com, or the universe domain before returning a ticket. Tokens with iss: chat@system.gserviceaccount.com or service-...@gcp-sa-gsuiteaddons... never reach this branch, so the mocked tests do not prove the intended production 401 fix.
  - confidence: 0.88
- **[P2] Log failures through each checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: logAuthFailure() always selects targets[0] as the logger even though the collected rejection reasons may belong to later accounts on the same webhook path. In multi-account deployments, that routes diagnostics through the wrong runtime sink and weakens the per-account troubleshooting this change is meant to add.
  - confidence: 0.83
