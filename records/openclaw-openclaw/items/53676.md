## Summary

Keep open. This PR still has blocking correctness issues in the auth and diagnostics changes, while current main already covers the diagnostics part better through https://github.com/openclaw/openclaw/pull/71145; the sender-email fallback remains useful but needs a rebase and focused coverage.

## What This Changes

The PR changes Google Chat webhook auth matching, webhook auth rejection diagnostics, sender email fallback behavior, and related tests.

## Best Possible Solution

A rebased or replacement fix should preserve current per-target diagnostics, validate real Google token shapes against google-auth-library behavior, keep appPrincipal binding strict, and add focused sender-email fallback coverage.

## Reproduction Assessment

Yes for the PR review findings: source and dependency inspection show the issuer-only checks run behind verifyIdToken()'s issuer gate, and the PR diagnostic helper logs through targets[0]. Live Google Chat delivery was not exercised.

## Solution Assessment

No. Current main already has a better diagnostics implementation, and the issuer-only auth path needs real-token validation or a different verifier path before widening accepted claims.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-only acceptance checks run after verifyIdToken(), but google-auth-library@10.6.2 rejects issuers outside accounts.google.com, https://accounts.google.com, or the universe domain before returning a payload. Tokens with Chat or add-on service-account iss values never reach this branch, so the mocked tests do not prove the intended production 401 fix.
  - confidence: 0.88
- **[P2] Log failures through the checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: logAuthFailure() always selects targets[0] as the logger even though collected rejection reasons may belong to later accounts on the same webhook path. That can route diagnostics through the wrong account runtime sink in multi-account deployments.
  - confidence: 0.83
