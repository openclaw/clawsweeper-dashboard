## Summary

Keep open: this is a plausible Google Chat bug-fix PR, but it is not merge-ready because the issuer-only auth path is not proven against the real verifier, the diagnostics part is partially superseded by current main, and external real behavior proof is still missing.

## What This Changes

The PR changes Google Chat webhook JWT claim matching, auth rejection diagnostics, tests, and sender email fallback behavior.

## Best Possible Solution

A rebased or replacement Google Chat fix should preserve current per-target diagnostics, validate real Google token shapes or use the correct verifier path, keep add-on principal binding strict, and add focused sender-email fallback coverage.

## Reproduction Assessment

Yes for the PR review findings: source and dependency inspection show the issuer-only checks run behind `verifyIdToken()`'s issuer gate, and the PR diagnostic helper logs through `targets[0]`. Live Google Chat delivery was not exercised.

## Solution Assessment

No. The safest path is not to merge this branch as-is; preserve current main's diagnostics, validate the auth claim shapes against real Google behavior, and keep the useful sender fallback with regression coverage.

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-only acceptance checks run after `verifyIdToken()`, but google-auth-library 10.6.2 rejects issuers outside `accounts.google.com`, `https://accounts.google.com`, or the universe domain before returning a ticket.   - confidence: 0.88

  - body: `logAuthFailure()` always selects `targets[0]` as the logger even though collected rejection reasons may belong to later accounts on the same webhook path. That can route diagnostics through the wrong account runtime sink, and current main's per-target rejection logging avoids this problem.
  - confidence: 0.83
