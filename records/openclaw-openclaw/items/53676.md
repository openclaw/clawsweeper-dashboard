## Summary

Keep open. This PR still has blocking correctness and proof issues: the diagnostics work is partially superseded by the merged https://github.com/openclaw/openclaw/pull/71145 implementation, the issuer-only auth path is not validated against google-auth-library's real verifier behavior, and the external PR lacks real behavior proof.

## What This Changes

The PR changes Google Chat webhook JWT issuer/email matching, auth rejection diagnostics, sender email fallback behavior, and related tests.

## Best Possible Solution

A rebased or replacement Google Chat fix should preserve current account-specific diagnostics, validate real Google token claim shapes against google-auth-library behavior, keep add-on sub principal binding strict, and add focused sender-email fallback coverage.

## Reproduction Assessment

Yes for the PR review findings: source and dependency inspection show the issuer-only checks run behind verifyIdToken()'s issuer gate, and the diagnostic helper logs through targets[0]. Live Google Chat delivery was not exercised.

## Solution Assessment

No. Current main already has a better diagnostics implementation, and the issuer-only auth path needs real-token validation or a different verifier path before widening accepted claims.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:127-132`
  - body: The new issuer-only acceptance checks run after `verifyIdToken()`, but google-auth-library 10.6.2 rejects issuers outside `accounts.google.com`, `https://accounts.google.com`, or the universe domain before returning a ticket. Tokens with `iss` values like `chat@system.gserviceaccount.com` or `service-...@gcp-sa-gsuiteaddons...` never reach this branch, so the mocked tests do not prove the intended production fix.
  - confidence: 0.88
- **[P2] Log failures through the checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: `logAuthFailure()` always selects `targets[0]` as the logger even though the collected rejection reasons may belong to later accounts on the same webhook path. That can route diagnostics through the wrong account runtime sink in multi-account deployments.
  - confidence: 0.83
