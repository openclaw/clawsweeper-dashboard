## Summary

Keep open, but this PR is not merge-ready: current main already has the better per-target diagnostics, the issuer-only auth path is not proven against google-auth-library's real issuer gate, and the external PR still lacks real Google Chat behavior proof.

## What This Changes

The PR changes Google Chat webhook auth to accept issuer/email variants without email claims, adds auth rejection diagnostics and tests, and falls back sender email extraction to `event.user.email`.

## Best Possible Solution

A rebased or replacement Google Chat fix should preserve current per-target diagnostics, validate real Google token shapes against google-auth-library behavior, keep add-on principal binding strict, and add focused sender-email fallback coverage.

## Reproduction Assessment

Yes for the PR review findings: source and dependency inspection show the issuer-only checks run behind `verifyIdToken()`'s issuer gate, and the PR diagnostic helper logs through `targets[0]`. Live Google Chat delivery was not exercised.

## Solution Assessment

No. Current main already has a better diagnostics implementation, and the issuer-only auth path needs real-token validation or a different verifier path before widening accepted claims.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-only acceptance checks run after `verifyIdToken()`, but google-auth-library 10.6.2 rejects issuers outside `accounts.google.com`, `https://accounts.google.com`, or the universe domain before returning a ticket. Tokens with `iss: chat@system.gserviceaccount.com` or `service-...@gcp-sa-gsuiteaddons...` never reach this branch, so the mocked tests do not prove the intended production 401 fix.
  - confidence: 0.88
- **[P2] Log failures through the checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: `logAuthFailure()` always selects `targets[0]` as the logger even though the collected rejection reasons may belong to later accounts on the same webhook path. That can route diagnostics through the wrong account runtime sink in multi-account deployments and weakens the per-account troubleshooting this change is meant to add.
  - confidence: 0.83
