## Summary

Keep open. Current main partially supersedes the diagnostics portion, but this PR still contains an ineffective issuer-only auth change and a multi-account logging regression, while the remaining Google Chat/Add-ons compatibility work needs maintainer validation against real token shapes.

## What This Changes

The PR changes Google Chat webhook auth to accept issuer/email variants without email claims, adds auth rejection diagnostics and tests, and falls back sender email extraction to `event.user.email`.

## Best Possible Solution

A rebased or replacement Google Chat fix should keep the shipped per-target diagnostics and Add-ons body parsing, validate real JWT claim shapes against `google-auth-library`, preserve strict add-on principal binding, and land the sender-email fallback with focused tests and a changelog entry.

## Reproduction Assessment

Yes for the blocking PR defects: the PR diff shows issuer-only checks after `verifyIdToken()`, while the pinned verifier rejects those issuers first, and the diff shows diagnostics routed through `targets[0]`. Live Google Chat delivery was not exercised.

## Solution Assessment

No. This PR is not the best solution as written because its diagnostics are stale relative to main and its issuer-only auth path does not match the pinned verifier behavior; a maintainer-owned rebase or replacement is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens before accepting them:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new `iss` checks run only after `verifyIdToken()` returns, but `google-auth-library` 10.6.2's default `OAuth2Client` rejects issuers outside `accounts.google.com`, `https://accounts.google.com`, or the universe domain before returning a ticket. The mocked `iss: chat@system.gserviceaccount.com` and add-on service-account tests therefore do not prove the intended production 401 fix.
  - confidence: 0.88
- **[P2] Log failures through each checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: `logAuthFailure()` always selects `targets[0]` as the logger even though the collected rejection reasons may belong to later accounts on the same webhook path. In multi-account deployments, that routes diagnostics through the wrong runtime sink and weakens the per-account troubleshooting this change is meant to add.
  - confidence: 0.83
