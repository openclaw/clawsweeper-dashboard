## Summary

Keep open. Current main has superseded the diagnostic portion through #71145 and shipped it in v2026.4.29, but this PR is not implemented on main because the sender-email fallback is still absent; the auth change is also not merge-ready because issuer-only checks run after google-auth-library's issuer gate. The branch is dirty against main and overlaps open Google Chat Add-ons work, so it needs maintainer rebase and validation rather than cleanup closure.

## What This Changes

The PR branch changes Google Chat webhook auth to accept issuer/email variants without email claims, adds auth rejection logging and tests, and falls back sender email extraction to `event.user.email`.

## Best Possible Solution

A rebased or replacement Google Chat fix should keep the shipped per-target diagnostics, validate real token shapes against google-auth-library before widening accepted auth claims, preserve strict add-on principal binding, and land the sender-email fallback with focused tests and a user-facing changelog entry if it remains part of the fix.

## Reproduction Assessment

Yes for the blocking PR defects: the PR diff shows issuer-only acceptance after `verifyIdToken()`, while google-auth-library 10.6.2 rejects those issuers before returning a payload, and the PR diff shows diagnostics routed through `targets[0]`. Live Google Chat delivery was not exercised.

## Solution Assessment

No. This PR is not the best way to solve the remaining issue because its diagnostics are already superseded on main, its issuer-only auth path is incompatible with the pinned verifier, and the dirty branch needs maintainer consolidation with the active Add-ons work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-only checks run after `verifyIdToken()`, but google-auth-library 10.6.2 rejects issuers outside `accounts.google.com`, `https://accounts.google.com`, or the universe domain before returning a ticket. Tokens with `iss: chat@system.gserviceaccount.com` or `service-...@gcp-sa-gsuiteaddons...` therefore never reach this branch, so the mocked tests do not prove the intended production 401 fix.
  - confidence: 0.88
- **[P2] Log failures through each checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: `logAuthFailure()` always selects `targets[0]` as the logger even though the collected rejection reasons may belong to later accounts on the same webhook path. In multi-account deployments, that routes diagnostics through the wrong runtime sink and weakens the per-account troubleshooting this change is meant to add.
  - confidence: 0.83
