## Summary

Keep open: current main partially supersedes the diagnostics work, but the PR still contains an auth-path correctness problem and one small sender fallback that is not on main.

## What This Changes

The PR changes Google Chat webhook auth claim matching, adds auth rejection diagnostics, extends tests, and falls back sender email extraction to `event.user.email`.

## Best Possible Solution

A rebased or replacement Google Chat fix should preserve current per-target diagnostics, validate real token claim shapes against `google-auth-library`, keep add-on principal binding strict, and add focused coverage for any sender-email fallback.

## Reproduction Assessment

Yes for the PR review findings: source inspection of the diff and `google-auth-library` 10.6.2 shows the issuer-only path is behind a verifier gate that rejects those issuers. The first-target logger issue is also visible in the PR diff, but live Google Chat delivery was not exercised.

## Solution Assessment

No. The diagnostics part is superseded by current main, while the issuer-only auth change is not proven against the real verifier contract; the safer path is a maintainer-validated rebase or replacement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-based acceptance runs after `verifyIdToken`, but `google-auth-library` 10.6.2 verifies against its default issuer list before returning a ticket. Tokens with `iss: chat@system.gserviceaccount.com` or `service-...@gcp-sa-gsuiteaddons...` are rejected before this branch, so the mocked tests do not prove the intended production 401 fix.
  - confidence: 0.88
- **[P2] Log failures through each checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: `logAuthFailure` always selects `targets[0]` as the logger even though the collected rejection reasons may belong to later accounts on the same webhook path. That routes diagnostics through the wrong runtime sink in multi-account deployments and weakens the per-account troubleshooting this change is meant to add.
  - confidence: 0.83
