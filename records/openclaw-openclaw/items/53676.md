## Summary

Keep open for maintainer follow-up: the PR still contains a useful sender-email fallback, but its issuer-only auth path is not proven against the pinned verifier and the diagnostics work is partly superseded by https://github.com/openclaw/openclaw/pull/71145.

## What This Changes

The PR changes Google Chat app-url/add-on token matching, adds webhook auth rejection logging and tests, and falls back sender email extraction to event.user.email.

## Best Possible Solution

A rebased or replacement Google Chat fix should preserve current per-target diagnostics, validate real Google token shapes against google-auth-library, keep add-on principal binding strict, add focused sender-email fallback coverage, and coordinate broader Add-ons docs/parser work through https://github.com/openclaw/openclaw/issues/65326.

## Reproduction Assessment

Yes for the PR review findings: source and dependency inspection show the issuer-only checks run after verifyIdToken's issuer gate, and the PR diagnostic logger uses targets[0]. Live Google Chat delivery was not exercised.

## Solution Assessment

No. The diagnostics piece is already better solved on current main, and the issuer-only auth path needs real-token validation or a different verifier path before widening acceptance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-based acceptance checks run only after `verifyIdToken()` returns, but google-auth-library 10.6.2 verifies ID tokens with its default issuer allowlist first. Tokens with `iss: chat@system.gserviceaccount.com` or `service-...@gcp-sa-gsuiteaddons...` never reach this branch, so the mocked tests do not prove the intended production 401 fix.
  - confidence: 0.88
- **[P2] Log failures through the checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: `logAuthFailure()` always selects `targets[0]` as the logger even though collected rejection reasons may belong to later accounts on the same webhook path. That can route diagnostics through the wrong account runtime sink in multi-account deployments.
  - confidence: 0.83
