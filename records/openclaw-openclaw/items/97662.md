## Summary

Keep open: this is a valid Matrix transport OOM bugfix direction, but the patch is not merge-ready because it defaults dropped encrypted-media limits to 64 MiB instead of forwarding the caller’s configured `maxBytes`/`readIdleTimeoutMs`, and contributor proof remains test-only.

## What This Changes

The PR makes Matrix raw transport reads without `maxBytes` use the existing 64 MiB SDK response cap and adds mocked plus `node:http` transport tests for over-cap and under-cap responses.

## Best Possible Solution

Forward the encrypted-media download options into `MatrixClient.downloadContent`, keep or confirm the transport fallback cap only as defense-in-depth, and prove both configured-limit and omitted-limit paths.

## Reproduction Assessment

Yes, from source: current main routes raw Matrix requests without `maxBytes` to `Buffer.from(await response.arrayBuffer())`, and the encrypted-media adapter drops the options that would carry configured limits into that path.

## Solution Assessment

No, not yet: the transport fallback is useful defense-in-depth, but the best fix should also forward the existing caller options so configured Matrix media limits are honored before the 64 MiB fallback applies.

- **[P1] Forward caller media limits before defaulting raw downloads:** `extensions/matrix/src/matrix/sdk/transport.ts:368`
  - body: Defaulting omitted raw downloads to 64 MiB is a useful fallback, but the encrypted-media path currently drops the caller's `maxBytes` and `readIdleTimeoutMs` in `MatrixClient` before reaching transport. `downloadMatrixMedia` already passes the configured media limit into `decryptMedia`, and `crypto-facade` forwards those options to its `downloadContent` dependency; without forwarding them in `sdk.ts`, encrypted attachments can ignore lower account limits until 64 MiB and can reject higher configured limits at 64 MiB. Please forward the existing options and add coverage for the encrypted-media path before relying on this default.
