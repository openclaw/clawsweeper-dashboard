## Summary

Keep open: the latest head fixes the prior lint blocker and the raw transport cap is useful, but merge is still blocked by a Matrix encrypted-media option forwarding bug and by missing real Matrix runtime proof.

## What This Changes

The branch changes Matrix raw `performMatrixRequest` reads without `maxBytes` to use the existing 64 MiB cap and adds mocked plus `node:http` transport coverage.

## Best Possible Solution

Forward `maxBytes` and `readIdleTimeoutMs` through the Matrix crypto facade callback into `MatrixClient.downloadContent`, keep the 64 MiB fallback only for truly omitted raw callers, and add real Matrix runtime proof before merge.

## Reproduction Assessment

Yes for source reproduction: current main routes raw Matrix requests without `maxBytes` to `response.arrayBuffer()`, and the encrypted-media path passes options to `decryptMedia` but drops them at the MatrixClient crypto callback. I did not run a live Matrix repro in this read-only review.

## Solution Assessment

No, not yet: the transport fallback is the right defense-in-depth layer for truly omitted callers, but the best fix also preserves existing configured caller limits by forwarding options through the crypto facade before defaulting.
