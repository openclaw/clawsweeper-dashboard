## Summary

Keep open: the PR addresses a real Matrix raw-response OOM path, but it is not merge-ready because encrypted media still drops caller byte/idle-timeout options before the new default cap and the external-contributor proof is still test/lint output rather than real Matrix runtime evidence.

## What This Changes

The branch changes Matrix raw SDK transport reads with omitted `maxBytes` to use the existing 64 MiB SDK cap and adds mocked plus `node:http` tests for oversized and under-cap raw responses.

## Best Possible Solution

Forward `maxBytes` and `readIdleTimeoutMs` through the Matrix crypto-facade `downloadContent` wiring, keep the 64 MiB fallback for truly omitted raw callers, cover both paths in tests, and require redacted real Matrix runtime proof before merge.

## Reproduction Assessment

Yes for source reproduction: current main routes raw Matrix requests without `maxBytes` to `response.arrayBuffer()`, and caller-path review shows encrypted media options are dropped before transport. I did not run a live Matrix media repro in this read-only review.

## Solution Assessment

No, not yet. The transport fallback is the right defense layer for truly omitted callers, but the best fix must also preserve caller-specific media limits before applying that default.
