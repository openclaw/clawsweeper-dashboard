## Summary

Keep open; the PR fixes a real Matrix raw-response memory risk, but it is not merge-ready because it still drops configured encrypted-media download limits, introduces lint failures, and lacks real Matrix runtime proof.

## What This Changes

The PR makes Matrix raw transport reads without `maxBytes` use the existing 64 MiB SDK response cap and adds mocked plus `node:http` transport tests.

## Best Possible Solution

Forward encrypted-media download options into `MatrixClient.downloadContent`, keep the 64 MiB transport fallback only as defense-in-depth for truly omitted callers, fix the lint errors, and prove the real Matrix runtime path.

## Reproduction Assessment

Yes, from source: current main routes raw Matrix requests without `maxBytes` to `Buffer.from(await response.arrayBuffer())`, and the encrypted-media adapter drops the options that would carry configured limits into that path.

## Solution Assessment

No, not yet: the transport fallback is useful defense-in-depth, but the best fix should also forward existing caller options so configured Matrix media limits are honored before the 64 MiB fallback applies.

- **[P1] Forward caller media limits before defaulting raw downloads:** `extensions/matrix/src/matrix/sdk/transport.ts:368`
