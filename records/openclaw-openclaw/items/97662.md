## Summary

Keep open: the transport fallback is a useful bug-fix direction, but the branch still leaves the encrypted-media path dropping caller media limits before the new default cap and lacks real Matrix runtime proof, so it is not merge-ready.

## What This Changes

The PR changes Matrix raw transport reads without an explicit `maxBytes` to use the existing 64 MiB default cap and adds mocked plus `node:http` transport tests for over-limit and under-limit raw responses.

## Best Possible Solution

Forward `maxBytes` and `readIdleTimeoutMs` through MatrixClient's crypto-facade `downloadContent` callback, keep the 64 MiB fallback for truly omitted raw callers, and add redacted real Matrix runtime proof before merge.

## Reproduction Assessment

Yes for source reproduction: current main routes raw Matrix requests without `maxBytes` to `response.arrayBuffer()`, and caller-path review shows encrypted media options are dropped at the MatrixClient crypto callback. I did not run a live Matrix media repro in this read-only review.

## Solution Assessment

No, not yet. The transport fallback is the right defense layer for truly omitted callers, but the best fix must also preserve caller-specific media limits before applying that default.
