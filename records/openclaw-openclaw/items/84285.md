## Summary

Keep open: the source-level bug is real and the patch is focused with no blocking code finding, but this external PR still needs stronger after-fix real-environment proof before merge.

## What This Changes

The PR wraps async JSON reads in bounded retries for transient `File changed during read` errors and adds retry success/exhaustion coverage in `src/infra/json-files.test.ts`.

## Best Possible Solution

Land a central async JSON retry once the contributor supplies inspectable after-fix gateway proof and CI is green; longer term, reconcile the behavior with `@openclaw/fs-safe` if the dependency grows a native retry contract.

## Reproduction Assessment

Yes, source-reproducible: fs-safe can throw `File changed during read`, current `main` exposes those async JSON readers without retry, and device pairing reads `devices/paired.json` through `readJsonIfExists`. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Mostly yes: the wrapper is a narrow OpenClaw mitigation that preserves public reader contracts and covers the affected pairing path. A dependency-level fix would be cleaner long term, but the latest published fs-safe version still does not provide that retry behavior.
