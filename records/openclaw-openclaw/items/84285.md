## Summary

Close as superseded: the same JSON read race is already owned by the open dependency-level fs-safe fix, and this OpenClaw-local wrapper duplicates that boundary while introducing a strict-reader type regression.

## What This Changes

The PR replaces OpenClaw's async JSON helper re-exports with local retry wrappers for `File changed during read` and routes `tryReadJson` through `readJsonIfExists`.

## Best Possible Solution

Land and release the fs-safe fix, then update OpenClaw to that released `@openclaw/fs-safe` version while preserving OpenClaw's strict JSON read API contracts.

## Reproduction Assessment

Yes, by source inspection: fs-safe 0.2.4 can throw `File changed during read`, OpenClaw currently delegates JSON reads directly to fs-safe, and pairing-state reads sit on gateway authorization paths. I did not run a live concurrent gateway rewrite harness in this read-only review.

## Solution Assessment

No. The best fix is the dependency-level fs-safe retry already tracked in https://github.com/openclaw/fs-safe/pull/19, followed by an OpenClaw dependency bump, not a second local wrapper with a changed strict-read type contract.
