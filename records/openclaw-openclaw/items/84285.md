## Summary

Keep open, but do not merge as-is: the reported gateway JSON read race is valid, yet this patch retries after current `@openclaw/fs-safe` has already wrapped or swallowed the race error, so the claimed retry path does not actually run. The cleaner primitive fix is already tracked in https://github.com/openclaw/fs-safe/pull/19, with OpenClaw likely needing a dependency bump after that lands.

## What This Changes

The PR replaces direct re-exports of async JSON readers with OpenClaw wrappers that retry `File changed during read` up to three attempts.

## Best Possible Solution

Land the retry in `@openclaw/fs-safe` before JSON read errors are wrapped or swallowed, publish and bump OpenClaw to that dependency version, and keep the OpenClaw facade thin unless maintainers intentionally choose a tested local hotfix first.

## Reproduction Assessment

Yes, by source inspection: current main delegates JSON reads to `@openclaw/fs-safe`, fs-safe wraps the file-change race as `JsonFileReadError`, and pairing-state reads propagate through gateway authorization paths. I did not run a live multi-client concurrent rewrite harness in this read-only review.

## Solution Assessment

No. This PR retries at the wrong layer for current fs-safe behavior; the maintainable fix is to retry inside fs-safe before wrapping/swallowing errors, or to implement an OpenClaw hotfix that explicitly unwraps `cause` and avoids `tryReadJsonImpl` swallowing the race.
