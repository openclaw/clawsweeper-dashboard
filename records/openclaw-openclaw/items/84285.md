## Summary

Keep open: the latest force-push now handles the fs-safe cause chain, but it widens strict JSON read types exposed through the plugin SDK infra runtime and still needs stronger after-fix runtime proof before merge.

## What This Changes

The PR replaces async JSON helper re-exports with local wrappers that retry `File changed during read` up to three attempts and route `tryReadJson` through `readJsonIfExists`.

## Best Possible Solution

Preserve the existing strict reader contracts, add focused retry regression coverage or wait for https://github.com/openclaw/fs-safe/pull/19 and bump the dependency, then merge after redacted real runtime evidence is supplied.

## Reproduction Assessment

Yes, source-reproducible: `@openclaw/fs-safe@0.2.4` can throw `File changed during read`, OpenClaw currently re-exports non-retrying JSON readers, and device pairing reads `paired.json` on gateway authorization paths.

## Solution Assessment

No: the retry direction is plausible after the force-push, but the PR should preserve strict reader nullability contracts and either add local regression coverage or use the cleaner dependency-level fix.

## Review Findings

  - body: `readJson` and `readJsonFileStrict` used to re-export strict fs-safe readers whose contract is `Promise<T>` and throw on missing or invalid files.
