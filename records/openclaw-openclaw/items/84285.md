## Summary

Keep open: current main still has the one-shot async JSON read path, and this PR is a focused, source-backed fix with sufficient live-output proof and no blocking review findings. The only remaining maintainer consideration is whether to land this local mitigation before the upstream `@openclaw/fs-safe` retry fix ships.

## What This Changes

This PR wraps `readJson`, `readJsonIfExists`, and `tryReadJson` with bounded retries for transient `File changed during read` races and adds JSON helper retry coverage.

## Best Possible Solution

Land one bounded async JSON read retry path; prefer the `@openclaw/fs-safe` package-level fix when available, or merge this local mitigation now with an explicit follow-up to remove it after the dependency release.

## Reproduction Assessment

Yes, source-reproducible: current main reads pairing JSON through one-shot `@openclaw/fs-safe` async readers, and `fs-safe@0.2.7` throws `File changed during read` when file identity changes during open. I did not run a live concurrent gateway reproduction in this read-only review.

## Solution Assessment

Mostly yes: the PR is a narrow bounded retry that preserves parse and non-transient failures, but the cleaner long-term solution is the dependency-level retry in https://github.com/openclaw/fs-safe/pull/19.
