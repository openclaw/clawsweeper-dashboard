## Summary

Keep open for maintainer judgment: the merged quiet-session watchdog work addresses the original false reconnect class, but current main still does not expose the proposed public timeout field and the changelog explicitly preserves this PR as a configurable-timeout follow-up rather than treating it as fully superseded.

## What This Changes

The PR adds a `web.messageTimeoutMs` config field across config types, zod/generated schema, help labels, docs, WhatsApp reconnect resolution, and tests while preserving the 30-minute default.

## Best Possible Solution

Decide whether operators still need a tunable application-silence watchdog after the transport-activity fix; if yes, expose it through the current WhatsApp timing/config model with updated docs, generated schema artifacts, and focused watchdog regression tests.

## Reproduction Assessment

Yes for the original quiet-session reconnect symptom: current tests and source show the old inbound-silence path has been replaced by transport-aware watchdog behavior. For the remaining config request, reproduction is schema inspection: `web.messageTimeoutMs` is still absent on current main.

## Solution Assessment

No, not as-is against current main. The PR is useful source work, but after the transport-activity change the final solution needs maintainer agreement on whether to expose a timeout and how it maps to transport inactivity versus application silence.
