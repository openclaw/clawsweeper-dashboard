## Summary

Keep open. Current main still resolves MiniMax web_search credentials from configured webSearch apiKey, Coding Plan env vars, and MINIMAX_API_KEY, but not MINIMAX_OAUTH_TOKEN; the linked #65785 PR is an open unmerged implementation candidate with closing syntax, so this issue should remain open until that PR or an equivalent maintainer decision lands.

## What This Changes

The issue asks for MiniMax web_search to use the MiniMax OAuth token from MINIMAX_OAUTH_TOKEN for minimax-portal users instead of requiring a separate Coding Plan/API key.

## Best Possible Solution

Either land #65785 or an equivalent patch that updates the resolver, provider metadata, manifest UI hints, missing-credential message, docs, and regression tests if OAuth bearer tokens are valid for MiniMax Search; otherwise make the separate Coding Plan credential requirement explicit across the same surfaces and close the product expectation gap.

## Reproduction Assessment

Yes. With only MINIMAX_OAUTH_TOKEN present and no configured webSearch apiKey, MINIMAX_CODE_PLAN_KEY, MINIMAX_CODING_API_KEY, or MINIMAX_API_KEY, the current resolver returns undefined and web_search follows the missing_minimax_api_key path.

## Solution Assessment

Unclear. Adding MINIMAX_OAUTH_TOKEN is the narrow maintainable fix if MiniMax Search accepts the OAuth token, but if the vendor endpoint requires Coding Plan credentials, the safer solution is to preserve the separate-key requirement and align docs and UI copy.
