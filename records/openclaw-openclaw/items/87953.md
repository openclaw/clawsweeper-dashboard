## Summary

Keep open: current main and the latest release still allow a secure Control UI browser to hit normal device-pairing enforcement after trusted-proxy auth succeeds, which matches the reported 1008 pairing-required failure and conflicts with the trusted-proxy docs.

## What This Changes

The issue asks for trusted-proxy-authenticated Control UI WebSocket sessions to reach the dashboard without first-connection device pairing once proxy checks pass.

## Best Possible Solution

Align the trusted-proxy Control UI contract by either honoring trustedProxyAuthOk for operator Control UI pairing with focused security tests, or changing the docs if maintainers decide device-present browsers must still pair.

## Reproduction Assessment

Yes, by source inspection: HTTPS Control UI creates a device identity, trusted-proxy auth succeeds, shouldSkipControlUiPairing ignores trustedProxyAuthOk, and the unpaired device path can close with pairing required. I did not run a live Traefik/OAuth proxy reproduction in this read-only review.

## Solution Assessment

Unclear until maintainer/security review settles the contract. The narrow runtime repair is likely to make shouldSkipControlUiPairing honor trustedProxyAuthOk for operator Control UI only, with tests proving node and non-Control UI paths still require pairing.
