## Summary

Keep open: current main and the latest release still require pairing for trusted-proxy Control UI sessions with a browser device identity, while the open linked PR remains the candidate fix and needs maintainer/security review before this issue can close.

## What This Changes

Trusted-proxy Control UI sessions should reach the dashboard without first-connection device pairing when proxy auth headers pass validation.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/93581, then align runtime tests and trusted-proxy docs to one maintainer-approved Control UI pairing contract.

## Reproduction Assessment

Yes, by source inspection. Current main passes a trusted-proxy flag into the pairing decision, does not use it to skip pairing, and the gateway suite expects PAIRING_REQUIRED for a trusted-proxy Control UI device identity.

## Solution Assessment

Unclear until maintainer/security review settles the contract. If no-pairing SSO is intended, the linked gateway policy PR is the narrow implementation path; otherwise docs should state that first-time HTTPS dashboards still need pairing for operator scopes.
