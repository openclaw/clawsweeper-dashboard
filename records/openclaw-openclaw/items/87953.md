## Label Justifications

- `P1`: Trusted-proxy Control UI deployments can still be blocked by pairing after proxy authentication succeeds, which breaks a real user access workflow.
- `impact:security`: The issue is about whether reverse-proxy identity may replace device pairing as the Control UI operator access boundary.
- `impact:auth-provider`: The affected surface is gateway trusted-proxy authentication and Control UI WebSocket authorization.

## Summary

Keep open: current main and the latest release still preserve the trusted-proxy path that can reject a secure Control UI browser with an unpaired device identity, so the reported no-pairing SSO dashboard behavior remains unresolved and security-sensitive.

## What This Changes

The issue asks for trusted-proxy-authenticated Control UI users to reach the dashboard without first completing device pairing.

## Best Possible Solution

Align the runtime and docs around one trusted-proxy Control UI contract: either allow trusted-proxy-authenticated operator Control UI device identities to skip pairing with tight tests, or state that first-time HTTPS dashboards must still pair for operator scopes.

## Reproduction Assessment

Yes, by source inspection. The current UI creates a device identity in secure contexts, and the current gateway test asserts that an unpaired trusted-proxy Control UI device identity returns PAIRING_REQUIRED.

## Solution Assessment

Unclear until maintainer/security review settles the contract. If no-pairing SSO is desired, the narrow fix is to honor trustedProxyAuthOk only for operator Control UI pairing while preserving scope clearing and node/non-Control UI rejection; otherwise the docs should explicitly say first-time HTTPS dashboards must pair for operator scopes.
