## Label Justifications

- `impact:security`: The issue is about the trusted-proxy and device-pairing boundary that determines who may access the Control UI.

## Summary

Keep open: current main and the latest release still allow the reported mismatch between the trusted-proxy docs and Control UI pairing behavior. The source path shows secure Control UI browsers send a device identity, and the gateway still requires that identity to be paired even after trusted-proxy auth succeeds.

## What This Changes

The issue asks for trusted-proxy authenticated Control UI WebSocket sessions to reach the dashboard without first completing device pairing.

## Best Possible Solution

Align the trusted-proxy Control UI contract across docs, tests, and runtime: either allow verified trusted-proxy operator sessions with fresh browser identities to enter as documented, or update the docs with the required pairing behavior and supported workaround.

## Reproduction Assessment

Yes. The source path is high-confidence: secure Control UI creates a device identity, current trusted-proxy auth can succeed, and the current gateway test asserts that an unpaired trusted-proxy Control UI operator device gets a pairing-required response.

## Solution Assessment

Unclear until maintainers choose the intended auth contract. A narrow code fix could make shouldSkipControlUiPairing honor trustedProxyAuthOk for Control UI operators, but current tests assert the opposite on a security-sensitive boundary.
