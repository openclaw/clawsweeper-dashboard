## Label Justifications

## Summary

Keep open: the PR targets a real reply-delivery wedge, but the implementation only calls `fail()` on the superseded operation, which does not clear real dispatch operations that were retained until final delivery; the PR also needs after-fix real behavior proof.

## What This Changes

The PR bounds visible reply-turn waits for already-aborted active operations and adds regression tests for superseded versus healthy active runs.

## Best Possible Solution

Use the existing force-clear semantics, or an equivalent fail-plus-complete path, for aborted retained operations and add regression coverage for `retainFailureUntilComplete()` before providing redacted runtime or Telegram proof.

## Reproduction Assessment

Yes, at source level: dispatch-created reply operations call `retainFailureUntilComplete()`, and the registry contract keeps those operations active after `fail()`. The PR's fail-only retry path therefore cannot clear the real retained operation it is trying to reclaim.

## Solution Assessment

No. Bounding waits for already-aborted active operations is the right layer, but using `fail()` alone is not the narrow maintainable fix because retained operations require force-clear or complete semantics.
