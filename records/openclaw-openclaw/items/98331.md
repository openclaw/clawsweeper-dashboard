## Summary

Keep open: the bug is real, but the patch is not merge-ready because the reclaim path calls `fail()` only, which does not clear retained reply operations that dispatch uses for final delivery.

## What This Changes

The PR changes reply-turn admission to time out and reclaim already-aborted active reply operations, with regression tests for superseded and healthy active runs.

## Best Possible Solution

Use force-clear or complete-equivalent semantics for aborted retained operations, cover that retained-operation case in tests, then add redacted runtime or Telegram proof.

## Reproduction Assessment

Yes, at source level: dispatch-created reply operations call `retainFailureUntilComplete()`, and registry code plus tests show `fail()` alone leaves retained failures active. I did not run a live Telegram or provider scenario in this read-only review.

## Solution Assessment

No. Bounding waits for already-aborted active operations is the right layer, but `fail()` alone is not the narrow maintainable fix because retained operations require force-clear or complete semantics.
