## Summary

Keep open: the linked Feishu duplicate-reply problem is still present on current main, but this PR is not merge-ready because it is conflicting, lacks after-fix real Feishu proof, and still has blocking source-level review findings.

## What This Changes

The PR keeps Feishu streaming cards open across transient reply errors by adding clearText/clearStreamingProgress, a regression test, and bundled Card Kit markdown/reasoning sanitization changes.

## Best Possible Solution

Rebase and narrow the Feishu recovery fix so one streaming card survives transient reply errors, cleanup is serialized with retry delivery, stale session refs are reset safely, stream state stays raw, and redacted real Feishu proof is added.

## Reproduction Assessment

Yes, at source/test-harness level: current main closes active Feishu streaming on reply errors, the shared dispatcher does not await async error cleanup, and the linked report has sanitized failover logs showing duplicate visible replies. I did not run a live Feishu failover in this read-only pass.

## Solution Assessment

No. Keeping the card open through transient errors is the right direction, but this branch is not the best implementation until it is rebased, cleanup is serialized, stale refs are handled, sanitizer state mutation is removed or separated, and real proof is supplied.

  - body: `update()` sanitizes snapshots before `mergeStreamingText()` stores them as canonical state. A later raw snapshot or final, such as an incremental code span, can stop sharing a clean prefix and be concatenated or corrupted; keep state raw and sanitize only outbound Card Kit payloads.

  - confidence: 0.84

  - confidence: 0.83
