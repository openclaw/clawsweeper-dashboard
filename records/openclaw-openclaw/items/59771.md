## Summary

Keep open: current main still closes the active Feishu streaming card from reply `onError`, and this PR addresses a real duplicate-reply path, but the branch still has blocking state/race issues before it can merge.

## What This Changes

The PR adds Feishu streaming-session `clearText()` recovery, switches reply `onError` cleanup to clear buffers without closing the active streaming card, adds regression coverage, and bundles Feishu Card Kit markdown/reasoning sanitation changes.

## Best Possible Solution

Adapt the recovery as a focused Feishu fix: keep one streaming card alive through transient reply errors, serialize cleanup with retry delivery, reset closed/stale session refs safely, preserve current final dedupe behavior, and leave Card Kit sanitizer work separate or outbound-only.

## Reproduction Assessment

Yes. The PR-head sanitizer bug is reproducible from incremental snapshots such as `Here is `co` followed by `Here is `code``, and the cleanup race is source-backed by core not awaiting `onError`; the broader duplicate-reply problem is supported by #49381 and current main's error closeout path.

## Solution Assessment

No. Keeping the card open and clearing stale buffers is the right direction, but this branch is not the best implementation until recovery cleanup is serialized, stale session references are handled, sanitizer state mutation is removed or made outbound-only, and the required changelog entry is added.
