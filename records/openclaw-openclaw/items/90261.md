## Summary

Keep open. Current main and the latest release still have the `gateway.tools.allow` gap this PR is trying to harden, but the branch is not merge-ready because it changes a compatibility-sensitive Gateway tool boundary and still needs docs, real HTTP proof, and maintainer/security sign-off.

## What This Changes

The branch adds an owner-only deny list for `cron`, `gateway`, and `nodes` in Gateway tool resolution, threads RPC client admin scope into shared tool invocation, and adds HTTP/RPC regression tests.

## Best Possible Solution

Land the owner-only hardening only after maintainers explicitly accept the HTTP/RPC compatibility boundary, update the Gateway allowlist docs, and add redacted real Gateway HTTP proof for both denied and allowed paths.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.6.1` filter the default Gateway HTTP deny list through `gateway.tools.allow` without a later owner-only deny for `cron`, `gateway`, or `nodes`. I did not run the Gateway scenario in this read-only review.

## Solution Assessment

Unclear as merge-ready: centralizing the deny in `resolveGatewayScopedTools` is the right layer, but the PR also changes RPC behavior and leaves the public allowlist contract stale. The safer end state is explicit maintainer acceptance plus docs and live HTTP proof.
