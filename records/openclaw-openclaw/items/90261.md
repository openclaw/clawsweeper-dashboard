## Summary

Keep open: the security hardening is useful, but the PR is not merge-ready because the new deny is applied in shared tool resolution and unintentionally changes the Gateway RPC tools.invoke path, while the HTTP config semantics and real proof still need tightening.

## What This Changes

The branch adds an owner-only HTTP deny list for cron, gateway, and nodes, applies it during gateway-scoped tool resolution, and adds mocked HTTP invoke tests for non-owner denial and shared-secret owner access.

## Best Possible Solution

Keep the owner-only restriction on the actual HTTP endpoint, preserve or explicitly update RPC semantics with regression coverage, and document the new gateway.tools.allow owner/shared-secret requirement before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main lets gateway.tools.allow remove these names from the default deny list with no later owner-only filter, and the PR adds the filter in a shared path that RPC also uses; I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The runtime check is a plausible HTTP boundary, but the best fix must either pass adapter/owner context through the shared helper or otherwise avoid changing Gateway RPC unintentionally, then update docs and proof.

Full review comments:

- **[P1] Keep the owner-only gate off RPC tools.invoke:** `src/gateway/tool-resolution.ts:119-120`
  - body: `invokeGatewayTool` is shared by the HTTP endpoint and Gateway RPC, and the RPC adapter calls it without `senderIsOwner`. Since the helper still resolves with `surface: "http"`, this new deny treats every RPC `tools.invoke` caller as non-owner and hides explicitly allowed `cron`, `gateway`, and `nodes`, despite the PR saying the RPC adapter is out of scope. Thread adapter/owner context or apply this deny only to the actual POST `/tools/invoke` path.
  - confidence: 0.92
- **[P2] Document the new owner-only allow limit:** `src/security/dangerous-tools.ts:41`
  - body: This adds a second layer over `gateway.tools.allow` for `cron`, `gateway`, and `nodes`, but the public tools-invoke docs still say the allow list removes tools from the default HTTP deny list. Operators following those docs for trusted-proxy or private-ingress automation will see 404s without an explanation, so the docs/config reference need the new owner/shared-secret requirement before this behavior ships.
  - confidence: 0.86
