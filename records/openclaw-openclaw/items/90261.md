## Summary

Keep this PR open: the security-hardening direction is valid, but the current patch still over-applies the new owner-only HTTP deny to the shared RPC `tools.invoke` path, and the contributor proof remains test-only.

## What This Changes

The PR adds an owner-only deny list for `cron`, `gateway`, and `nodes` in Gateway tool resolution and adds regression tests for non-owner HTTP denial, shared-secret access, and inherited denies.

## Best Possible Solution

Land this as a direct-HTTP-only owner gate: keep RPC semantics unchanged or explicitly pass RPC owner context, update the `gateway.tools.allow` docs/audit contract, and add redacted live Gateway HTTP proof for both denied and allowed callers.

## Reproduction Assessment

Yes, by source inspection: configure `gateway.tools.allow` and agent policy for `nodes`, then call Gateway RPC `tools.invoke`; the RPC adapter passes no `senderIsOwner` and the shared resolver still uses `surface: "http"`, so the new deny filters the tool.

## Solution Assessment

No, not yet. The owner-only gate belongs on the direct HTTP caller boundary or on a caller-aware resolver mode, not on a shared `surface: "http"` path that also serves RPC without owner context.
