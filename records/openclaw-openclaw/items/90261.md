## Label Justifications

- `merge-risk: 🚨 compatibility`: Merging as-is can make previously allowed Gateway RPC and trusted-proxy HTTP tool invocations return `not_found`.

## Summary

Keep open: the security hardening is directionally useful, but this patch still lets the new owner-only deny hit the shared Gateway RPC `tools.invoke` path and leaves the changed HTTP allowlist contract without matching docs or real behavior proof.

## What This Changes

The PR adds an owner-only Gateway HTTP deny list for `cron`, `gateway`, and `nodes`, wires it into gateway-scoped tool resolution, and adds HTTP invoke regression tests.

## Best Possible Solution

Limit the owner-only deny to direct HTTP identity-bearing callers while preserving the intended RPC behavior, update the public allowlist docs/config guidance, and add redacted Gateway HTTP terminal proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current `tools.invoke` RPC calls the shared `invokeGatewayTool` without `senderIsOwner`, and that shared engine resolves tools with `surface: "http"`. With the PR's new `senderIsOwner !== true` deny, explicitly allowed owner-only tools are filtered before execution.

## Solution Assessment

No, this is not yet the best fix because it implements the HTTP boundary in a shared resolver without distinguishing the RPC adapter and without updating the documented allowlist contract. The safer shape is an explicit direct-HTTP owner-only policy path plus regression coverage for both HTTP and RPC.
