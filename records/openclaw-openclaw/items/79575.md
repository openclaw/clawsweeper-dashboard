## Summary

Keep open: this is useful Telnyx realtime voice work that is not implemented on current main, but the PR is not merge-ready because it logs a minted realtime stream URL token and now overlaps with a parallel open Telnyx implementation that needs maintainer direction.

## What This Changes

Adds Telnyx bidirectional media streaming support to voice-call realtime by allowing Telnyx config, adding optional media-stream provider hooks, starting a Telnyx stream after `call.answered`, accepting Telnyx WebSocket frames, and adding focused tests.

## Best Possible Solution

Select one Telnyx realtime design, keep the implementation inside the voice-call plugin boundary, redact stream-token logging, update user-facing docs, and land with focused voice-call realtime/provider checks.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source inspection confirms current main is Twilio-only, and the PR body provides live Telnyx runtime logs for the proposed behavior.

## Solution Assessment

Unclear as the final product direction: the provider-local approach is plausible, but this PR must stop logging the stream token and maintainers should choose between the post-answer `streaming_start` design here and the inline dial/answer design in https://github.com/openclaw/openclaw/pull/81024.
