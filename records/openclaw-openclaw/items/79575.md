## Summary

Keep open for maintainer review: current main still limits realtime Voice Call to Twilio, and this PR has convincing live Telnyx proof, but the head still logs a bearer-style realtime stream URL token and a parallel open Telnyx PR proposes a different carrier kickoff design that maintainers need to choose between.

The PR extends voice-call realtime support to Telnyx by adding media-stream start/stop provider hooks, Telnyx WebSocket start-frame handling, runtime stream startup on answered calls, validation changes, and focused tests.

## Best Possible Solution

Pick one Telnyx realtime architecture, redact stream-token logging, align the Voice Call docs, then land the chosen PR with focused voice-call tests and the existing live Telnyx proof.

## Reproduction Assessment

## Solution Assessment

No, not yet: the provider-local implementation is plausible because Telnyx documents both inline and `streaming_start` setup, but the current head should not log the full stream token and maintainers need to choose between this action-based path and the parallel inline implementation.
