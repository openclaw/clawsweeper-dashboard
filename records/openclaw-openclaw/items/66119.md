## Summary

Keep open. Current main still auto-starts the Signal plugin's `signal-cli` daemon without a non-mutating `registered:false` account-file preflight or recovery guard, and upstream `signal-cli` treats `registered` as persisted account state, so the report remains concrete and actionable.

## What This Changes

The issue requests protection against OpenClaw update/startup flows leaving a standalone Signal `signal-cli` account file with `registered:false`, including preserving account state, warning/failing closed, and backup guidance.

## Best Possible Solution

Keep the issue open and handle the mitigation in the bundled Signal plugin: add a Signal-owned, non-mutating startup preflight before daemon autostart where account-file location is known, fail closed with a clear warning on `registered:false`, add mocked regression coverage, and document backup/recovery guidance on the public Signal channel page.

## Reproduction Assessment

Unclear. The report gives concrete steps and current main still has a static path that would auto-start `signal-cli` without a guard, but a high-confidence live reproduction is unsafe because it may destroy or invalidate a real Signal account.

## Solution Assessment

No current solution is present on main. The narrowest maintainable direction is a non-mutating preflight plus docs first; automatic backups or recovery should wait for maintainer policy because they touch sensitive account key files.
