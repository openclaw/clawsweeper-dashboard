## Summary

Keep open. Current main still auto-starts native `signal-cli` without a non-mutating preflight for a persisted `registered:false` account file, and upstream `signal-cli` treats that state as not registered; the exact cause of the reported mutation is still unproven, so this needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue reports that upgrading OpenClaw from 2026.4.11 to 2026.4.12 caused a Signal `signal-cli` account file to contain `registered:false`, breaking the channel and possibly deleting the Signal account.

## Best Possible Solution

Add a Signal-owned, non-mutating startup preflight before native daemon autostart that fails closed on `registered:false`, documents recovery, and leaves backup or repair policy to an explicit maintainer security decision.

## Reproduction Assessment

Unclear for the full account-deletion report because reproducing it would risk a real Signal account. The narrower missing-guard behavior is source-reproducible from current `monitorSignalProvider` autostart plus upstream `signal-cli` rejecting persisted `registered:false` accounts.

## Solution Assessment

No current solution is present on main. The best direction is a narrow non-mutating preflight and docs update first; automatic backup or account-file repair needs maintainer security policy.
