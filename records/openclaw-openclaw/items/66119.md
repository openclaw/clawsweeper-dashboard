## Summary

Keep open. Current main still starts native `signal-cli` without a non-mutating guard for a persisted `registered:false` account file, and upstream `signal-cli` treats that state as not registered; the reported mutation/deletion mechanism is still unproven and the mitigation is security-sensitive.

## What This Changes

The issue asks OpenClaw to prevent or fail closed when Signal startup encounters `signal-cli` account state with `registered:false` after an update.

## Best Possible Solution

Add a Signal-owned, non-mutating startup preflight before native daemon autostart that fails closed on `registered:false`, documents recovery, and leaves backup or repair behavior to an explicit security/product decision.

## Reproduction Assessment

Unclear for the full account-deletion report because safely reproducing it would require risking a real Signal account. The narrower missing-guard behavior is source-reproducible from the current `monitorSignalProvider` autostart path plus upstream `signal-cli` rejecting persisted `registered:false` accounts.

## Solution Assessment

No current solution is present on main. A narrow non-mutating preflight is the best first fix; automatic backup, repair, or account-file mutation policy should wait for maintainer security review.
