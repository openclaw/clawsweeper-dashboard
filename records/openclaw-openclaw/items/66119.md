## Summary

Keep open. Current main still auto-starts native `signal-cli` without a non-mutating `registered:false` account-file preflight, and upstream `signal-cli` treats that persisted state as not registered; the exact mutation mechanism is not proven, so this needs maintainer security/product review rather than closure or automated repair.

## What This Changes

The issue asks OpenClaw's Signal plugin to avoid corrupting `signal-cli` account state and fail closed when account JSON is already `registered:false` during update/startup.

## Best Possible Solution

Add a Signal-owned, non-mutating startup preflight before native daemon autostart that fails closed on `registered:false`, reports a recovery path in the public Signal docs, and leaves any automatic backup or repair behavior to an explicit maintainer security decision.

## Reproduction Assessment

Unclear for the full account-deletion report because proving it would risk a real Signal account. The narrower missing-guard behavior is source-reproducible from `monitorSignalProvider` autostart plus upstream `signal-cli` rejecting `registered:false` accounts.

## Solution Assessment

No current solution is present on main. The best direction is a narrow non-mutating preflight and docs update first; automatic backups or account-file rewrites need maintainer security policy.
