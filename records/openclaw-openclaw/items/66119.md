## Summary

Keep open. Current main still auto-starts `signal-cli` without a non-mutating `registered:false` account-file preflight, and upstream `signal-cli` treats that state as not registered; the destructive live path is too risky to reproduce, but the missing guard remains actionable and security-sensitive.

## What This Changes

The issue asks OpenClaw to prevent Signal startup/update paths from proceeding when a `signal-cli` account file has `registered:false`, preserving account state and warning or recovering safely.

## Best Possible Solution

Add a Signal-owned, non-mutating account-state preflight before daemon autostart that fails closed on `registered:false`, emits redacted recovery guidance, and only adds backup or repair behavior after maintainers approve the secret-handling policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the destructive live account-deletion path, because proving it would risk a real Signal account; the narrower missing-guard behavior is source-reproducible from `monitorSignalProvider` autostart plus upstream `NotRegisteredException` on `registered:false`.

## Solution Assessment

Is this the best way to solve the issue? No current solution exists on main. The narrowest maintainable direction is a non-mutating fail-closed preflight and docs first; automatic backup or account-file rewrite behavior should wait for maintainer security/product policy.
