## Summary

Keep open. Current main still auto-starts the Signal daemon without a non-mutating `registered:false` account-file preflight, and upstream `signal-cli` treats that state as not registered, so the destructive-state guard remains actionable but policy-sensitive.

## What This Changes

The report asks the Signal plugin to prevent or warn on `signal-cli` account files with `registered:false` after updates before daemon autostart can worsen account loss.

## Best Possible Solution

Add a Signal-owned, non-mutating startup preflight before daemon autostart that fails closed on `registered:false`, documents recovery, and defers automatic backup or repair behavior until maintainers set the account-file security policy.

## Reproduction Assessment

Unclear for a high-confidence live reproduction: the report has concrete steps and current main has a source-visible autostart path with no guard, but safely proving account deletion would require risking a real Signal account. The narrower missing-guard behavior is source-reproducible from `monitorSignalProvider` plus the upstream `signal-cli` `NotRegisteredException` path.

## Solution Assessment

No current solution is present on main. The narrowest maintainable direction is a non-mutating preflight and docs first; automatic backups or account-file rewrites need maintainer security and product policy before implementation.
