## Summary

Keep open. Current main still auto-starts native `signal-cli` without a non-mutating guard for a persisted `registered:false` account file, and the full account-deletion mechanism remains unproven and security-sensitive rather than safely closeable.

## What This Changes

The issue reports that upgrading OpenClaw from 2026.4.11 to 2026.4.12 left a Signal `signal-cli` account file with `registered:false`, breaking the channel and allegedly deleting the Signal account.

## Best Possible Solution

Add a Signal-owned, non-mutating startup preflight before native daemon autostart that fails closed on `registered:false` with redacted recovery guidance; leave backup or repair behavior to an explicit security/product decision.

## Reproduction Assessment

No for the full account-deletion report: safely reproducing it could risk a real Signal account. The narrower missing-guard behavior is source-reproducible from current native autostart plus upstream `signal-cli` rejecting persisted `registered:false` accounts.

## Solution Assessment

No current solution is present on main. A narrow non-mutating preflight is the safest first fix, while automatic backup or repair of account files needs maintainer security review.
