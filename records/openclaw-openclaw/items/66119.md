## Summary

Keep open. Current main still has the same actionable gap: the Signal plugin can auto-start `signal-cli daemon` without a non-mutating preflight for a persisted `registered:false` account file, and the existing docs only advise manual backups rather than failing closed on this destructive state.

## What This Changes

The issue requests that OpenClaw avoid mutating Signal account data during updates, preserve or guard the `registered` flag, warn on `registered:false`, and consider backup/recovery behavior for `signal-cli` account state.

## Best Possible Solution

Handle this in the bundled Signal plugin by adding a Signal-owned, non-mutating startup preflight before daemon autostart where the account-file path is known, fail closed with a clear warning on `registered:false`, add mocked regression coverage, and document safe backup/recovery guidance on the public Signal channel page.

## Reproduction Assessment

Unclear. The report gives concrete steps and current main has a static path that would auto-start `signal-cli` without a guard, but a high-confidence live reproduction is unsafe because it may destroy or invalidate a real Signal account.

## Solution Assessment

No current solution is present on main. The narrowest maintainable direction is a non-mutating preflight and documentation first; automatic backup or repair behavior should wait for maintainer policy because it touches sensitive account key files.
