## Summary

Keep open. Current main still auto-starts native `signal-cli` without a non-mutating guard for a persisted `registered:false` account file, upstream `signal-cli` treats that state as not registered, and the reported account-file mutation/deletion mechanism remains unproven and security-sensitive rather than safely closeable.

## What This Changes

The issue asks OpenClaw to prevent Signal updates or restarts from corrupting or continuing with a `signal-cli` account file whose persisted `registered` flag is false.

## Best Possible Solution

Add a Signal-owned, non-mutating startup preflight before native daemon autostart that fails closed on `registered:false` with redacted recovery guidance; leave backup or repair behavior to an explicit security/product decision.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the full account-deletion report because safely reproducing it could risk a real Signal account; the narrower missing-guard behavior is source-reproducible from the current autostart path plus upstream `signal-cli` rejecting persisted `registered:false` accounts.

## Solution Assessment

Is this the best way to solve the issue? No current solution is present on main; a narrow non-mutating preflight is the safest first fix, while automatic backup or repair of account files needs maintainer security review.
