## Summary

This PR should stay open for maintainer review: the patch is a narrow, well-proven hardening of an unbounded provider-controlled response read, but it adds a new 4 MiB public plugin SDK runtime limit that maintainers should explicitly accept before merge.

## What This Changes

The branch routes live provider model catalog success JSON through `readResponseWithLimit` with a 4 MiB cap and idle timeout, and adds overflow/stall regression tests.

## Best Possible Solution

Land the bounded success-body read if maintainers accept 4 MiB as the SDK live-catalog limit; otherwise adjust or document the cap before merge.

## Reproduction Assessment

Yes, by source inspection: current main calls `response.json()` on a provider-controlled live catalog success body. The PR body also includes a Node 22 runtime harness and negative control showing the old path drains an oversized stream while the patched path cancels near the cap.

## Solution Assessment

Yes, with the compatibility decision called out: reusing `readResponseWithLimit` is the narrowest maintainable fix for this response-read boundary. The remaining question is whether maintainers accept 4 MiB as the public SDK limit.
