## Summary

Current main and the shipped v2026.5.7 release already implement the MiniMax usage endpoint routing in a safer form, with provider-baseUrl routing and regression coverage; merging this branch would reintroduce older endpoint and credential-routing behavior.

## What This Changes

This PR changes MiniMax usage polling to derive the remains endpoint host from provider baseUrl/env and updates unit tests to assert the request URL and method.

## Best Possible Solution

Keep the shipped v2026.5.7 implementation and close this overlapping branch instead of merging an older endpoint-routing variant.

## Reproduction Assessment

Yes. The old/branch behavior is source-reproducible from the PR diff, and current main has focused tests that verify the global, CN, configured-origin, and malformed-baseUrl request URLs.

## Solution Assessment

No for this PR branch; yes for current main. The shipped implementation routes by the configured MiniMax provider baseUrl, preserves the CN fallback, covers portal/API credential routing, and uses the current Token Plan endpoint path.

Full review comments:

- **[P2] Keep usage polling on the shipped Token Plan path:** `src/infra/provider-usage.fetch.minimax.ts:383`
  - body: The branch constructs requests to `/v1/api/openplatform/coding_plan/remains`, but current main now uses and tests `/v1/token_plan/remains` for both CN and global origins. Merging this branch would replace the shipped usage endpoint path with the older route.
  - confidence: 0.86
- **[P2] Route MiniMax usage tokens from the credential source:** `extensions/minimax/provider-registration.ts:247-252`
  - body: `resolveUsageAuth` can return a `minimax-portal` OAuth token, but this branch chooses the usage baseUrl from `ctx.provider` or `MINIMAX_API_HOST` instead of the portal provider config that produced the token. That can send portal credentials to the wrong regional MiniMax host when only `minimax-portal` is configured.
  - confidence: 0.78
