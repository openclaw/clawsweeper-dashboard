## Summary

Keep open. Current main still does not implement the PR's intended region-aware/global MiniMax usage host behavior, and the PR branch still has a blocking routing issue: it derives the usage endpoint from the generic `minimax` provider context even when usage auth selected a `minimax-portal` OAuth credential.

## What This Changes

The PR adds a `baseUrl` option to MiniMax usage polling, derives the remains endpoint origin from provider config or `MINIMAX_API_HOST`, and updates the MiniMax usage unit test to assert request URL and method.

## Best Possible Solution

Land one MiniMax-owned usage endpoint resolver that keeps `GET /v1/api/openplatform/coding_plan/remains`, chooses global vs CN from the actual credential/config source for `minimax`, `minimax-portal`, and `MINIMAX_API_HOST`, documents custom-origin behavior, and adds regression coverage for global API-key, CN API-key, and OAuth setups.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a config with `models.providers.minimax-portal.baseUrl` set to the CN host and a portal OAuth token; the PR diff still resolves the usage host from `models.providers.minimax` or the global/env fallback.

## Solution Assessment

No. The PR moves in the right direction by making the host configurable, but deriving it from `ctx.provider` is not the narrowest safe solution because usage auth can come from a different MiniMax provider id.
