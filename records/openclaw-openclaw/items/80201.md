## Summary

Keep open: current main still contains the reported requester-key and completion-origin routing boundaries, and the linked closing PR is open rather than merged.

## What This Changes

The issue reports that subagent completion announces spawned from `agent:main:main` can be delivered to a parallel Telegram-bound session instead of the originating parent session.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/80242 or an equivalent fix lands with same-account Telegram regression coverage and redacted runtime proof, then close from the merge.

## Reproduction Assessment

Yes, at source level: current main still passes the sandbox/peer key into `sessions_spawn` ownership and checks child bindings before requester bindings. I did not run live Telegram proof in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to record the real run session as requester ownership and prefer requester bindings before child fallback, rather than adding Telegram-specific routing policy.
