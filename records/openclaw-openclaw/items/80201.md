## Summary

Keep open: this is a fresh, detailed behavior bug against the existing push-based subagent completion contract, and current main still lacks an exact regression proving that a main-session spawn with a parallel Telegram-bound sibling cannot misroute the announce. Source inspection shows the intended requester key is captured and passed into delivery, but the delivery path still combines that key with channel route hints, so the report is plausible rather than obsolete or implemented.

## What This Changes

The issue requests that subagent completion announces spawned from `agent:main:main` be delivered to the originating main session instead of a parallel Telegram-bound session.

## Best Possible Solution

Add a focused regression for a main parent session with a parallel Telegram-bound sibling, then make subagent completion handoff preserve the recorded requester session as the transcript destination while using route hints only for external delivery.

## Reproduction Assessment

No high-confidence current-main reproduction was established. The issue includes detailed v2026.5.7 field evidence and source inspection found the plausible route-hint boundary, but not a failing test or live Telegram proof for this exact single-user misroute.

## Solution Assessment

Unclear until the exact regression is written. The likely maintainable fix is a narrow requester-session-key invariant in subagent announce delivery, not a Telegram transport rewrite or a broader delivery-context migration.
