Keep open: the report describes a current product contract violation with detailed release steps, and the closest related issues track different delivery failures rather than this single-user main-session-to-Telegram-session misroute.

## What This Changes

The issue reports that a subagent completion announcement spawned from `agent:main:main` can be routed to a parallel Telegram-bound session instead of the originating parent session.

## Best Possible Solution

Pin subagent completion handoff to the recorded requester/parent session key, then use delivery context only for outbound user delivery after the parent session has received the internal completion event.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. Source inspection and the detailed v2026.5.7 report make the route-mixing failure plausible, but a focused test or live Telegram probe is still needed.

## Solution Assessment

Unclear until a regression is written. The safest direction is a narrow parent-session-key fix in the subagent announce delivery path, not a broad Telegram transport rewrite.

## Repair Work Prompt

Fix the subagent completion announcement routing bug where a child spawned from `agent:main:main` can have its completion observed in a parallel Telegram-bound session instead of the originating parent session. Preserve the registered `requesterSessionKey`/parent session as the internal completion recipient; do not choose the internal recipient from `sourceChannel`, `lastChannel`, `deliveryContext`, or a sibling channel-bound session. Add a regression test that seeds both `agent:main:main` and `agent:main:telegram:default:direct:<chat_id>` (with Telegram delivery context/last-touched route), registers or runs a subagent completion for a child whose requester is `agent:main:main`, and asserts the gateway agent/session-history handoff targets `agent:main:main` and not the Telegram sibling. Keep core plugin-agnostic, avoid Telegram auth/transport policy changes, and add a `CHANGELOG.md` fix entry if behavior changes beyond tests.
