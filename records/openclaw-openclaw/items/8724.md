## Summary

Keep open: current main has adjacent provider request and idle-watchdog controls, but the requested strict-schema global/per-model non-resetting generation budget is still missing; related narrower timeout bugs do not supersede this canonical feature request.

## What This Changes

Request a typed global and per-model generation timeout so operators can bound long or looping model generations differently by model and fallback candidate.

## Best Possible Solution

Define one backwards-compatible timeout hierarchy, add a typed global/per-model generation budget at the agent model-catalog surface, resolve it per selected fallback candidate, enforce it with a non-resetting timer, and keep first-token/provider-validator/idle-timeout regressions in their narrower issues.

## Reproduction Assessment

Yes for the missing configuration/enforcement surface: current source shows the proposed per-model timeout key is outside strict schema and no selected-candidate generation budget is enforced. The original live Gemini looping scenario was not replayed in this read-only review.

## Solution Assessment

Unclear. A global/per-model timeout budget is a plausible direction, but maintainers should choose the public field name, units, disable behavior, and precedence against request, idle, first-token, run, and fallback timeouts before implementation.
