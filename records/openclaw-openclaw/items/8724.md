## Label Justifications

- `impact:auth-provider`: The issue concerns model-specific execution limits and the timing of provider/model fallback routing.

## Summary

The central capability remains absent on current main: OpenClaw has global run limits and provider-scoped request/idle guards, but no model-entry, non-resetting total-generation budget for actively streaming loops. Keep this issue as the canonical tracker, with maintainer product review required because adding another timeout knob changes a compatibility-sensitive config and fallback contract.

## What This Changes

Add configurable global and per-model total-generation timeouts that abort runaway model turns, log the timeout, and optionally allow fallback.

## Best Possible Solution

Adopt one typed, backwards-compatible total-generation budget at agent defaults with a per-model override only after defining precedence against the whole-run and provider/idle limits; resolve it for each fallback candidate and surface a timeout classification that fallback handling can consume without changing existing defaults.

## Reproduction Assessment

Yes at the source-contract level: current main has no per-model total-generation field, and its idle watchdog resets when chunks keep arriving, so active streaming loops bypass that guard. This read-only review did not run a live Gemini infinite-thinking reproduction.

## Solution Assessment

Unclear pending product direction. A non-resetting per-model budget is the narrowest mechanism for active streaming loops, but merely exposing the existing provider `requestTimeoutMs` would conflate HTTP/idle behavior with total-generation policy and would not fully solve the reported case.
