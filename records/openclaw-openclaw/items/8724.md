## Summary

Keep open. Current main still lacks a typed global/per-model non-resetting generation-timeout budget; it only has the existing run-level timeout, provider request timeout plumbing, and a reset-on-chunk idle watchdog, so the runaway streaming/thinking-loop case remains unsolved.

## What This Changes

The issue requests configurable total generation timeouts globally and per model so runaway model turns can be aborted, logged, and optionally routed through fallback handling.

## Best Possible Solution

Add a documented backwards-compatible generation-timeout contract at agent defaults and per-model catalog entries, resolve the effective budget for each selected fallback candidate, and enforce it with a non-resetting timer during active generation.

## Reproduction Assessment

Yes. The gap is reproducible by source/schema inspection: the proposed per-model key is rejected by strict config validation, and runtime enforcement still uses `params.timeoutMs` plus a reset-on-chunk idle watchdog.

## Solution Assessment

Yes in direction. A narrow core config/schema/docs change plus embedded-runner enforcement is the maintainable path; provider HTTP timeouts and first-token fail-fast behavior solve adjacent but different problems.
