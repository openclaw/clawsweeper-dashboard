## Summary

Keep open: current main and the latest release still expose only provider/run-level timeout controls, while the strict per-model agent schema does not accept the requested generation/request timeout keys. The prior discussion already treats this as the canonical remaining per-model timeout work, but the config contract and fallback semantics need maintainer product/API judgment before implementation.

## What This Changes

Request a global and per-model generation timeout so looping or slow model calls can abort or fail over without relying on one coarse run timeout.

## Best Possible Solution

Add a backwards-compatible agent default and per-model timeout contract, resolve it per fallback candidate, enforce it as a non-resetting generation budget, and document how it differs from provider request, idle, first-token, and whole-run timeouts.

## Reproduction Assessment

Yes for the config-contract gap: current source shows the strict per-model agent schema does not accept the requested timeout keys, and the runtime only has adjacent provider/run/idle timeout paths. No live model-loop reproduction was attempted.

## Solution Assessment

Unclear until maintainers choose the public config name and precedence semantics. The maintainable direction is a typed per-model/global generation budget, with first-token and provider-skip behavior kept as separate concerns.
