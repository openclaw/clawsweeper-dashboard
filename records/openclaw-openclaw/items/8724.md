## Summary

Keep open. Current main and the latest release still expose adjacent provider request/idle timeout and global run timeout controls, but not the requested typed global/per-model non-resetting generation budget under `agents.defaults.models`; the remaining work needs maintainer product direction before implementation.

## What This Changes

Add configurable global and per-model generation timeout budgets so long or looping model generations can be aborted and routed predictably.

## Best Possible Solution

Maintainers should choose the canonical timeout name, scope, and fallback semantics, then add the config/schema/docs/tests and a non-resetting attempt-level generation budget without replacing the existing provider idle/request timeout behavior.

## Reproduction Assessment

Yes, from source inspection: the strict agent model schema rejects the proposed per-model timeout keys, and the runner currently combines global run timeout with provider request/idle timeout rather than a non-resetting per-model generation budget. I did not run a live hung-provider reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the config contract. A per-model generation budget is plausible, but the best fix must avoid creating a competing source of truth with the existing provider timeout and first-token/idle-timeout behavior.
