## Summary

Keep open: current main still has adjacent run, provider-request, and idle watchdog timeouts, but not the requested typed global/per-model non-resetting generation timeout. The strict model catalog schema rejects the proposed key, and the embedded attempt path still aborts from the run-level timeout.

## What This Changes

The issue requests a global and per-model generation timeout configuration that aborts long-running model generation and can participate in fallback handling.

## Best Possible Solution

Add a backwards-compatible `generationTimeout` contract at the agent default and per-model catalog levels, resolve the effective budget for each selected fallback candidate, enforce it with a non-resetting timer during active generation, and classify/log timeout aborts for fallback handling while leaving first-token and provider-skip behavior separate.

## Reproduction Assessment

Yes by source inspection for the missing contract: the proposed per-model key is absent from types/docs and rejected by the strict schema, while attempts still use the run-level `params.timeoutMs`. I did not need a live provider hang to verify that current main lacks this configuration surface.

## Solution Assessment

Yes. A narrow core config and runner-enforcement change is the maintainable solution; provider request timeout and reset-on-chunk idle timeout address different failure modes and should not be overloaded.
