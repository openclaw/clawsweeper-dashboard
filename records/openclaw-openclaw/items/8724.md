## Summary

Keep open. Current main still has adjacent provider request/idle timeout controls, but it does not expose the requested typed global/per-model non-resetting generation timeout under `agents.defaults.models`, and the remaining config/fallback semantics need maintainer product direction.

## What This Changes

The issue requests a global and per-model generation timeout so long-running or looping model generations can be aborted, logged, and optionally routed to fallback models.

## Best Possible Solution

Maintainers should choose the canonical timeout name, scope, precedence, and fallback semantics, then add schema/types/docs/tests plus a non-resetting generation budget without replacing the existing provider request and idle-timeout behavior.

## Reproduction Assessment

Yes, from source inspection: the strict per-model agent config schema still rejects the proposed timeout keys, and the active runtime only wires provider request timeout into a reset-on-progress idle watchdog. I did not run a live hung-model reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the config contract. A per-model generation budget is plausible, but the best fix must avoid creating a competing timeout source of truth with provider request timeout, global run timeout, and first-token/idle behavior.
