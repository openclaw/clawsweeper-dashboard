## Summary

Keep open: current main still has adjacent run-level, provider-level, and idle-timeout controls, but not the requested typed global/per-model non-resetting generation-timeout contract. The issue is also already scoped as a maintainer/product decision, so it should not be swept as implemented or auto-repaired.

## What This Changes

The issue requests configurable global and per-model generation timeouts so selected models can abort or fail over when generation runs too long.

## Best Possible Solution

Add a documented global and per-model generation-timeout contract, resolve it for each selected fallback candidate, enforce it as a non-resetting total generation budget, and keep first-token/idle/provider-specific policy as separate scoped work.

## Reproduction Assessment

Yes for the missing configuration surface: source inspection shows the proposed agents.defaults.models timeout key is rejected by the strict schema and the runner still uses params.timeoutMs plus an idle watchdog. The exact live Gemini loop was not replayed.

## Solution Assessment

Unclear. A non-resetting per-model generation budget is the right direction for this issue, but maintainers still need to choose the public key, units, precedence, and interaction with existing provider/run/idle timeout behavior.
