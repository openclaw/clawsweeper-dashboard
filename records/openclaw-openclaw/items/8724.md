## Summary

Keep open: current main and the latest release still lack the requested typed global/per-model non-resetting generation timeout, while the existing provider timeout and idle watchdog only cover adjacent request/stream-idle behavior.

## What This Changes

The issue requests global and per-model generation timeout configuration so long or looping model generations can be aborted, logged, and possibly routed through fallback independently of provider request timeouts.

## Best Possible Solution

Keep this as the canonical maintainer-reviewed product/config decision, then add a typed generation-budget contract with explicit precedence, fallback semantics, docs/schema coverage, and a non-resetting enforcement point per selected model candidate.

## Reproduction Assessment

Yes for the missing config/runtime contract: current main and v2026.6.10 still reject the proposed per-model timeout keys and only wire provider request timeout into a reset-on-progress idle watchdog. I did not run a live hung-model reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. A per-model generation budget is plausible and source-backed, but the best fix must avoid creating a competing timeout source of truth with provider request timeout, global run timeout, and first-token/idle behavior.
