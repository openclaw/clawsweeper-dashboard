## Summary

Keep open. Current main still lacks the requested typed global/per-model non-resetting generation timeout; existing behavior is run-level `timeoutSeconds`, provider request timeout metadata, and a reset-on-chunk idle watchdog. The prior maintainer and ClawSweeper discussion correctly keeps first-token/provider-skip behavior as related but separate work.

## What This Changes

The issue requests a global and per-model generation timeout that can abort long-running model output loops and optionally trigger fallback.

## Best Possible Solution

Ship a documented backwards-compatible `generationTimeout` contract at agent defaults and model catalog entries, resolve the effective budget per selected fallback candidate, and enforce a non-resetting active-generation deadline with timeout classification and logging.

## Reproduction Assessment

Yes, source-reproducible. Against current main, the proposed per-model key is rejected by the strict schema and runtime still uses params.timeoutMs plus a reset-on-chunk idle watchdog; no live provider loop is needed to prove the missing config surface.

## Solution Assessment

Yes in direction. A narrow config/schema/docs change plus embedded-runner enforcement is the maintainable path; first-token timeout and provider-level skip behavior should remain separate work.

## Repair Work Prompt

Implement the per-model generation timeout requested in https://github.com/openclaw/openclaw/issues/8724. Add a backwards-compatible `generationTimeout` config contract at `agents.defaults` and `agents.defaults.models["provider/model"]`, choose and document the accepted duration format consistently with existing config conventions, update types, zod schema, generated schema/docs/help metadata as needed, and resolve the effective timeout for each selected fallback candidate before dispatching the embedded attempt. Enforce this as a non-resetting active-generation deadline; do not reuse the reset-on-chunk idle watchdog as the total-generation timer. Classify/log timeout aborts so fallback handling can treat them as timeouts. Keep first-token timeout and provider-level skip behavior out of scope; do not change `models.providers.<id>.timeoutSeconds` semantics. Add focused regression coverage for schema acceptance/rejection, effective timeout resolution, runner enforcement, and fallback compatibility. Because this is user-facing config, add a CHANGELOG.md entry unless maintainers decide it is internal-only.
