## Summary

Keep open. Current `main` and latest release still have adjacent provider request/idle timeout controls, but not the requested typed global/per-model non-resetting generation timeout, and the remaining timeout precedence/fallback behavior needs maintainer product direction.

## What This Changes

Request to add a global and per-model generation timeout so runaway model generations can be aborted and optionally fall back by model.

## Best Possible Solution

Maintainers should choose the canonical timeout name, scope, precedence, and fallback semantics, then add schema/types/docs/tests plus a non-resetting generation budget without replacing the existing provider request and idle-timeout behavior.

## Reproduction Assessment

Yes, from source inspection: the current strict per-model agent config schema still rejects the requested timeout keys, while runtime only consumes provider-level request timeout and a reset-on-progress idle watchdog. I did not run a live hung-model reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the config contract. A per-model generation budget is plausible, but the best fix must avoid creating a competing timeout source of truth with provider request timeout, global run timeout, and first-token/idle behavior.
