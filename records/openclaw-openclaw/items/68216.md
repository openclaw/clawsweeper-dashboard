## Summary

Keep open. Current main and the latest release still invoke the bundled Google Gemini CLI backend in headless `--prompt` mode without an explicit approval mode, while upstream Gemini CLI policy denies write/edit tools in non-interactive default mode; the focused `--yolo` PR was closed unmerged after surfacing a real security/product posture question.

## What This Changes

The issue asks for the `google-gemini-cli` OAuth backend to have working workspace file write/edit capability for files such as `IDENTITY.md` and `SOUL.md`.

## Best Possible Solution

Adopt a maintainer-approved Gemini CLI approval-mode mapping for headless runs, preserve explicit user backend overrides, update docs and regression coverage, and verify with a live OAuth edit smoke in a disposable workspace.

## Reproduction Assessment

Yes. This is source-reproducible without a live OAuth account: current main invokes `gemini --prompt` without an approval mode, and upstream Gemini CLI denies write/edit tools in non-interactive default approval mode.

## Solution Assessment

Unclear. The closed `--yolo` patch likely fixes the observable write failure, but the maintainable fix should first choose the intended Gemini native-tool approval posture, likely considering a narrower `auto_edit` or documented override path.
