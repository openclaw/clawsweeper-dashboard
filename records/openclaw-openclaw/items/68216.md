## Summary

Keep open. Current main still drives the Google Gemini CLI backend in headless `--prompt` mode without a Gemini approval mode, and the focused fix PR was closed unmerged after raising a real native-tool security posture question.

## What This Changes

The issue reports that `google-gemini-cli` can read OpenClaw workspace context but cannot write or edit files such as `IDENTITY.md` or `SOUL.md`.

## Best Possible Solution

Adopt a maintainer-approved Gemini CLI approval-mode mapping for headless runs, preserve explicit user backend args, document the security tradeoff, add regression coverage, and verify with a live OAuth edit smoke.

## Reproduction Assessment

Yes, by source and dependency contract rather than a live OAuth run. Current main invokes `gemini --prompt` without an approval mode, while Gemini CLI defaults to approval-prompt behavior and excludes prompt-requiring tools in non-interactive mode.

## Solution Assessment

Unclear. The closed PR's unconditional `--yolo` likely fixes the observable write failure, but the safer maintainable fix needs maintainer approval of the Gemini native-tool approval posture, possibly with `auto_edit` or a policy-derived mapping.
