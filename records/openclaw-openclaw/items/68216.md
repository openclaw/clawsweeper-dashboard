## Summary

Keep open. Current main and latest release still run the Google Gemini CLI backend in headless `--prompt` mode without an explicit approval mode, and the linked `--yolo` fix PR was closed unmerged after surfacing a real native-tool approval/security posture question.

## What This Changes

The issue reports that the Google Gemini CLI backend can read injected workspace context but cannot write or edit files such as `IDENTITY.md` when run through OpenClaw via OAuth.

## Best Possible Solution

Adopt a maintainer-approved Gemini CLI approval-mode mapping for headless runs, preserve explicit backend overrides, update docs and regression coverage, and verify with a live OAuth edit smoke in a disposable workspace.

## Reproduction Assessment

Yes. The source path is high-confidence: OpenClaw currently invokes Gemini CLI headlessly with `--prompt` and no approval mode, while upstream Gemini CLI defaults prompt-required native tools to denial in non-interactive mode.

## Solution Assessment

Unclear. The linked `--yolo` patch likely fixes the observable write failure, but the maintainable fix should first choose the intended Gemini native-tool approval and sandbox posture, possibly using a narrower `auto_edit` mapping or documented override path.
