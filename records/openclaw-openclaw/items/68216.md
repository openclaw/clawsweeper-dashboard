## Summary

Keep open. Current main and the latest release still run the Google Gemini CLI backend in headless `--prompt` mode without an explicit Gemini approval mode, and the prior focused fix was closed unmerged after surfacing a real native-tool approval posture question.

## What This Changes

The issue reports that the `google-gemini-cli` provider can read injected workspace files but cannot write files such as `IDENTITY.md` or `SOUL.md` during non-interactive OpenClaw runs.

## Best Possible Solution

Adopt a maintainer-approved Gemini CLI approval-mode mapping for headless runs, preserve explicit user backend overrides, update docs and regression coverage, and verify with a live OAuth edit smoke.

## Reproduction Assessment

Yes, source-reproducible without a live OAuth account: current main invokes `gemini --prompt` without an approval mode, while upstream Gemini CLI defaults to approval-prompt behavior and denies prompt-requiring actions in headless operation.

## Solution Assessment

Unclear. The closed `--yolo` patch likely fixes the observable write failure, but the maintainable fix should deliberately choose the Gemini native-tool approval posture, possibly using a narrower `auto_edit` or policy-derived mapping rather than all-tool auto-approval.
