## Summary

Keep open. Current main still starts the bundled `google-gemini-cli` backend with `--skip-trust`, JSON output, and `--prompt`, but without `--yolo` or `--approval-mode`, while the current regression test pins that no-approval-mode argv shape. Upstream Gemini CLI policy makes the reported behavior plausible in noninteractive `--prompt` runs, and the direct fix PR was closed unmerged.

## What This Changes

The issue requests that `google-gemini-cli` OAuth agent runs be able to write or edit workspace files such as `IDENTITY.md` instead of reporting that no file-writing tool is available.

## Best Possible Solution

Choose an explicit Gemini CLI headless approval posture, then update the Google plugin backend defaults, resume defaults, docs, and regression coverage together. The final patch should account for the newer `--skip-trust` compatibility problem and should be verified with a live OAuth smoke that edits a disposable `IDENTITY.md`.

## Reproduction Assessment

Yes. The current backend argv plus upstream Gemini CLI policy gives a high-confidence reproduction path for `gemini --prompt` where write/edit tools require approval and are denied or unavailable without an interactive TTY; a live OAuth smoke would be the final proof.

## Solution Assessment

Unclear. The closed `--yolo` PR likely fixes the observable read-only behavior, but it auto-approves all Gemini CLI tools; a narrower `--approval-mode auto_edit` or maintainer-chosen policy mapping may be safer if shell execution should not be broadened by default.
