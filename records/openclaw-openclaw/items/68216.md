## Summary

Keep open. Current main still launches the Google Gemini CLI backend in noninteractive `--prompt` mode without an explicit Gemini approval mode for edit/write tools, and the linked focused fix PR remains closed unmerged; the remaining decision is the maintainer-approved security/product posture for headless Gemini tool approval.

## What This Changes

The issue requests that the `google-gemini-cli` runtime regain working file write/edit capability for workspace files such as `IDENTITY.md` and `SOUL.md` during OpenClaw OAuth-backed Linux sessions.

## Best Possible Solution

Make an explicit maintainer decision for Gemini CLI headless approval behavior, then update the Google plugin backend defaults, resume defaults, docs, changelog, and regression coverage together.

## Reproduction Assessment

Yes. The current default argv plus Gemini CLI's approval-mode contract provide a high-confidence reproduction path: configure `google-gemini-cli`, run an OpenClaw turn asking to edit `IDENTITY.md`, and observe that the headless Gemini process lacks an edit/write approval mode.

## Solution Assessment

Unclear. The `--yolo` approach from the linked PR likely fixes the observable write failure, but a narrower `--approval-mode=auto_edit` or policy-mapped approval choice may be the more maintainable default.
