- `P1`: The issue breaks file-write tool use for real users on the Gemini CLI OAuth backend, making that agent workflow effectively read-only until the backend posture is decided.
- `impact:auth-provider`: The failure is specific to the `google-gemini-cli` OAuth/provider backend and its subprocess argv/default routing.

## Summary

Keep open. Current main and the latest shipped release still invoke Gemini CLI headlessly without an explicit approval mode, and the closed workaround PR confirms the root cause while leaving a native-tool approval/security posture decision unresolved.

## What This Changes

The issue asks OpenClaw's `google-gemini-cli` OAuth backend to expose working file write/edit capability for workspace files such as `IDENTITY.md`.

## Best Possible Solution

Choose and document a maintainer-approved Gemini CLI approval-mode mapping for headless runs, preserve explicit backend overrides, update docs/regression coverage, and verify with a live OAuth edit smoke in a disposable workspace.

## Reproduction Assessment

Yes, source-level. Current main invokes Gemini CLI with `--prompt` and no approval mode, while upstream Gemini CLI defaults to prompt-required approval behavior that cannot be satisfied in non-interactive mode.

## Solution Assessment

Unclear. The linked `--yolo` patch likely restores writes, but the best fix needs a maintainer-approved approval-mode and sandbox posture, potentially using the narrower `auto_edit` behavior rather than broad native-tool auto-approval.
