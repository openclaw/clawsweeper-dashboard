## Summary

Keep open. Current main and the latest checked release still launch the Google Gemini CLI backend in headless `--prompt` mode without `--approval-mode`, `auto_edit`, or `yolo`, while Gemini CLI documents approval modes for headless runs and the focused fix PR remains closed unmerged. 

## What This Changes

The issue asks OpenClaw to restore file write/edit capability for workspace files such as `IDENTITY.md` and `SOUL.md` when using the `google-gemini-cli` OAuth runtime.

## Best Possible Solution

Make an explicit maintainer decision for Gemini CLI headless approval behavior, then update the Google plugin backend defaults, resume defaults, docs, changelog, and regression coverage together.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main invokes `gemini --prompt` without an approval mode, and Gemini CLI's documented headless approval contract explains why a write/edit prompt can lose native edit tools; I did not run a live OAuth smoke.

## Solution Assessment

Unclear. The closed PR's `--yolo` direction likely fixes the observable write failure, but `--approval-mode=auto_edit` or an OpenClaw policy-mapped choice may be a narrower maintainable default.
