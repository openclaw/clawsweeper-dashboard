## Summary

Keep open: current `main` still has a source-reproducible Feishu monitor path where a partial `channelRuntime` can reach code that requires full debounce helpers. The open fix candidate is unmerged and still blocked on stronger real Feishu or gateway proof, so this issue is not implemented yet.

## What This Changes

Feishu inbound DM replies are reported as silently dropped while outbound Feishu pushes still deliver.

## Best Possible Solution

Review and land the narrow monitor runtime guard in https://github.com/openclaw/openclaw/pull/94013 once it has acceptable real Feishu or gateway startup proof, then close this issue as implemented by that merged fix.

## Reproduction Assessment

Yes, at source level: current `monitor.account.ts` accepts any truthy `params.channelRuntime`, and `monitor.message-handler.ts` immediately requires `channelRuntime.debounce`. I did not run a live Feishu account or gateway reproduction in this read-only review.

## Solution Assessment

Yes for the known partial-runtime failure: mirror the already-merged `bot.ts` `channelRuntime?.inbound` guard at monitor startup so the receive handler gets a full runtime. The solution should land through the open PR after proof, not through a separate issue close now.
