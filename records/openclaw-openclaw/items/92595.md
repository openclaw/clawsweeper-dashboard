## Label Justifications

- `P1`: The issue reports a regression that makes Feishu inbound replies fail for affected users while outbound pushes still work.
- `impact:message-loss`: The affected behavior is incoming Feishu messages being dropped before an agent response is generated or delivered.

## Summary

Keep open. Current main and the latest release still select any truthy Feishu monitor `channelRuntime` before the receive handler reads full debounce APIs, so a context-only runtime can still block inbound processing; https://github.com/openclaw/openclaw/pull/94013 is the active candidate fix and has not merged.

## What This Changes

The issue reports Feishu WebSocket DMs becoming write-only: outbound pushes arrive, but user replies do not trigger an OpenClaw agent response.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94013 or an equivalent monitor-runtime guard after real Feishu/gateway proof, then close this issue when the fix is merged.

## Reproduction Assessment

Yes, source-reproducible: current main can pass a context-only `channelRuntime`, the monitor selects it, and the receive handler immediately dereferences full debounce APIs. I did not run a live Feishu account repro in this read-only review.

## Solution Assessment

Yes for the known partial-runtime monitor failure: the best narrow fix is to mirror the already-merged Feishu bot-path `inbound` guard at the monitor runtime-selection boundary. Merge should still wait for real Feishu or gateway proof on the active PR.
