## Summary

Keep open. Current main still has a Feishu monitor startup path that accepts any truthy injected channel runtime, while the downstream handler requires full runtime APIs; the open fix PR is a candidate, but the issue should not close before that or an equivalent fix merges.

## What This Changes

Feishu WebSocket direct-message replies are reported as silently dropped while outbound Feishu pushes still deliver.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94013 after redacted Feishu or gateway proof, or land an equivalent monitor runtime guard with focused regression tests, then close this issue as implemented.

## Reproduction Assessment

Yes from source inspection, though not from a live Feishu run: current main can pass a partial channel runtime into monitor startup, and the receive handler immediately requires `debounce`. The reporter also provided concrete WebSocket DM steps and matching no-incoming-processing symptoms.

## Solution Assessment

Yes for the known source-level failure: guarding the monitor runtime selection the same way the merged bot path does is the narrow owner-boundary fix. The issue should stay open until that fix path merges with real Feishu or gateway proof.
