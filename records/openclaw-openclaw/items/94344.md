## Summary

Keep open: this is a useful candidate fix for the linked memory embedding issue, but the branch currently adopts the provider request trust contract only for `true` and leaves the documented explicit `false` denial ineffective on the embedding path.

## What This Changes

The PR makes the OpenAI-compatible embedding provider read the selected configured provider's `request.allowPrivateNetwork` flag when building its SSRF policy, and adds focused regression coverage.

## Best Possible Solution

Use one provider trust contract for configured OpenAI-compatible endpoints: default exact-host compatibility, explicit `true` for broader private-network access, and explicit `false` as a denial if memory embeddings are inheriting provider request policy.

## Reproduction Assessment

Yes, source-reproducible: current main builds the memory OpenAI-compatible embedding SSRF policy without reading `models.providers.<id>.request.allowPrivateNetwork`, matching the linked issue's missing explicit policy inheritance path. I did not run the live CLI reproduction because this review is read-only.

## Solution Assessment

No, not quite as-is. Reusing the existing provider request trust bit is the right layer, but the patch should either honor the existing explicit false opt-out semantics or get maintainer confirmation that memory embeddings intentionally differ.
