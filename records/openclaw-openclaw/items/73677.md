## Summary

Keep this issue open. Current `main` still drops Feishu/Lark custom sticker labels before they reach the agent-visible inbound body: plain text parsing returns only `parsed.text`, standalone sticker messages collapse to `<media:sticker>`, and rich-text `emotion` handling only preserves labels when Feishu sends supported post element fields. The existing ClawSweeper keep-open comment is still accurate against current `main`, and the recent maintainer mentions/subscriptions are a useful signal that this bundled Feishu plugin behavior needs owner follow-up rather than stale cleanup.

## What This Changes

Request that the Feishu/Lark channel preserve custom bracket-style stickers such as `[OK]`, `[完成]`, and `[调皮]` for the AI agent, either by mapping them to Unicode emoji or by retaining a text representation.

## Best Possible Solution

Implement the behavior in the Feishu plugin inbound parser: preserve verified custom-sticker labels or stable bracket-text fallbacks from raw event payloads, keep unknown sticker metadata visible as text rather than dropping it, and add focused tests for mixed text plus custom-sticker metadata, standalone sticker messages, rich-text emotion elements, Chinese labels, and unknown fallback behavior.

## Reproduction Assessment

Unclear for an end-to-end live Feishu reproduction because the issue does not include raw event JSON. Source-level reproduction is clear on current `main`: plain text events return only `parsed.text`, standalone sticker events render as `<media:sticker>`, and the agent body uses that parsed content directly.

## Solution Assessment

Yes, the right fix boundary is the Feishu plugin parser rather than OpenClaw core. A label/text-preservation fallback is safer than broad Unicode conversion until the plugin has a verified Feishu sticker-name mapping table.
