## Summary

Keep open. Current main still does not preserve Feishu/Lark custom sticker labels in the agent-visible inbound text: text events use only `parsed.text`, standalone sticker events become a generic media placeholder, and the rich-text emotion renderer only helps when Feishu supplies supported emotion fields. This remains a bundled Feishu plugin behavior gap, not a core cleanup close.

## What This Changes

Request to preserve Feishu/Lark custom bracket-style stickers in inbound agent context by converting them to Unicode emoji or stable text such as `[OK]`.

## Best Possible Solution

Implement this in the Feishu plugin inbound parser: preserve verified custom-sticker labels or bracket-text fallbacks from raw event payloads, keep unknown stickers visible as stable text, and add focused tests for mixed text plus sticker metadata, standalone sticker messages, rich-text emotion elements, Chinese labels, and unknown-label fallback behavior.

## Reproduction Assessment

Unclear for the exact live Feishu payload because the report does not include raw event JSON. Source-level reproduction is clear from current main: text parsing returns only `parsed.text`, and standalone sticker parsing yields `<media:sticker>` instead of `[OK]` or `[完成]`.

## Solution Assessment

Yes, the best boundary is a Feishu plugin parser fix rather than a core channel API change. The safest implementation should preserve verified label/text first and only add Unicode emoji mappings when the Feishu sticker names are confirmed.
