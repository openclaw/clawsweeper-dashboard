## Summary

Keep open: current main still normalizes Feishu/Lark sticker messages as generic media context and does not extract custom bracket labels or map them to Unicode. The issue is recent, no canonical duplicate or merged fix surfaced, and the remaining decision depends on real Feishu payload evidence plus plugin-owner/product policy for bracket text versus Unicode mapping.

## What This Changes

Request Feishu/Lark inbound custom stickers such as [OK] to be preserved for the agent as Unicode emoji or bracket text instead of being lost or reduced to generic media context.

## Best Possible Solution

Capture a real Feishu custom-sticker payload, then update the Feishu parser to preserve verified sticker labels in agent context before deciding whether to maintain a Unicode mapping.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the exact mixed `好的[OK]` report because the raw Feishu event payload is missing; source inspection does show standalone sticker labels are not extracted and are reduced to `<media:sticker>`.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the Feishu plugin parser is the right boundary, but bracket-text preservation should come before any curated Unicode mapping unless maintainers choose to own that mapping.
