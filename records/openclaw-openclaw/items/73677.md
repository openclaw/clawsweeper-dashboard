## Summary

Keep this open. Current main still has no path that converts Feishu/Lark custom-sticker meaning into Unicode emoji or bracket text: text messages return only the Feishu `text` field, sticker messages become a generic `<media:sticker>` placeholder plus media attachment, and the agent body is built from that parsed content. The remaining blocker is product/API confirmation of the raw custom-sticker payload and the desired label-versus-Unicode mapping policy.

## What This Changes

Preserve Feishu/Lark custom sticker meaning for agents by converting bracket-style stickers such as `[OK]`, `[完成]`, and `[调皮]` into Unicode emoji or visible text.

## Best Possible Solution

Fix this in the bundled Feishu plugin inbound parser after collecting a real custom-sticker event sample: preserve any verified text-ish label fields as bracket text, keep the existing media attachment path, avoid guessing Unicode equivalents unless backed by a maintained mapping, and add focused parser/body tests for mixed text plus sticker, standalone sticker, rich-text emotion, Chinese labels, and unknown-label fallback.

## Reproduction Assessment

Unclear: there is no raw Feishu event JSON for the reported `[OK]` or `[完成]` cases, so the exact live reproduction is not high-confidence. Source-level behavior is clear: current main has no label-preservation path beyond `parsed.text`, rich-text `emoji_type`, and generic sticker media placeholders.

## Solution Assessment

Yes on boundary, unclear on mapping details: the maintainable fix belongs in `extensions/feishu` rather than core. The safest approach is verified text preservation first; broad Unicode conversion should wait until a stable Feishu sticker-name source is identified.

## Risks / Open Questions

- The exact Feishu/Lark raw event JSON for custom bracket-style stickers is still missing, so a fix could accidentally target fields that the platform does not send.
- A Unicode conversion table may be brittle unless Feishu exposes a stable sticker name or a maintained mapping source; a text fallback may be safer than broad emoji normalization.
- If Feishu only exposes a `file_key` for custom stickers, OpenClaw may be limited to preserving a generic sticker placeholder and media attachment unless OCR/vision or user-maintained mappings are introduced.
