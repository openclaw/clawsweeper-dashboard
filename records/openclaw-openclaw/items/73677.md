## Summary

Keep this issue open. Current main still treats inbound Feishu/Lark stickers as generic media, and there is no implemented path that preserves custom bracket-style sticker labels or maps them to Unicode emoji before the agent sees the message. The right boundary is the Feishu plugin, but a safe fix needs a real custom-sticker event sample and an agreed fallback policy before guessing fields or maintaining a Unicode mapping.

## What This Changes

The issue requests preserving Feishu/Lark custom sticker meaning by converting bracket-style stickers such as [OK], [完成], and [调皮] into Unicode emoji or bracket-text for the agent.

## Best Possible Solution

Fix this in the bundled Feishu plugin inbound parser after collecting a real custom-sticker event sample: preserve any verified text-like sticker label as bracket text, keep the existing media attachment path, avoid broad Unicode conversion unless there is a maintained mapping, and add focused tests for mixed text plus sticker, standalone sticker, rich-text emotion, Chinese labels, and unknown-label fallback.

## Reproduction Assessment

Unclear: the source-level path is reproducible by following parseMessageContent for message_type "sticker", which yields <media:sticker>, but there is no raw Feishu event JSON for the reported [OK], [完成], or [调皮] payloads. That missing payload prevents a high-confidence live reproduction of the exact label-loss case.

## Solution Assessment

Yes on boundary, unclear on mapping details: the maintainable fix belongs in extensions/feishu rather than core. The safer solution is verified bracket-text preservation first, with Unicode mapping only if a stable Feishu sticker-name source and ownership policy are confirmed.
