## Summary

Keep this open. Current main still preserves Feishu sticker messages only as generic media placeholders and attachments, with no verified path that turns custom sticker meaning into bracket text or Unicode emoji. The issue is valid, but it needs a real raw Feishu/Lark custom-sticker event sample and plugin-owner mapping policy before a repair should guess at fields or emoji equivalents.

## What This Changes

The issue requests preserving Feishu/Lark custom sticker meaning for agents by converting bracket-style stickers such as [OK], [完成], and [调皮] into Unicode emoji or visible text.

## Best Possible Solution

Fix this in the bundled Feishu plugin inbound parser after collecting a real custom-sticker event sample: preserve verified text-ish label fields as bracket text, keep the existing media attachment path, avoid broad Unicode conversion until a stable mapping source exists, and add focused parser/body tests for mixed text plus sticker, standalone sticker, rich-text emotion, Chinese labels, and unknown fallback.

## Reproduction Assessment

Unclear: there is no raw Feishu event JSON for the reported [OK], [完成], or [调皮] cases, so the exact live reproduction is not high-confidence. Source inspection does show the current parser has no label-preservation path beyond parsed.text, rich-text emoji fields, and generic sticker media placeholders.

## Solution Assessment

Yes on boundary, unclear on mapping details. The maintainable fix belongs in extensions/feishu rather than core, but the exact fallback policy should be based on a verified Feishu payload instead of guessed Unicode mappings.
