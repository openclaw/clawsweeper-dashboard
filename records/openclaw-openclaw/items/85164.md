## Summary

Keep open: the feature is plausible and current main already emits Codex commentary as `kind: "preamble"` item events, but this PR still needs real Discord behavior proof and message-delivery fixes before merge.

## What This Changes

The PR adds an opt-in Discord progress-draft path that renders assistant commentary/preamble item events via a new `streaming.progress.commentary` setting.

## Best Possible Solution

Keep the opt-in Discord commentary path, but only render cleaned non-silent commentary, block commentary edits after final delivery starts, document the setting, and require redacted live Discord proof.

## Reproduction Assessment

Yes for the patch defects: source inspection shows raw commentary normalization without directive/silent-token filtering and no final-state guard in `pushCommentaryProgress`. No live Discord reproduction or proof was provided.

## Solution Assessment

Partly: consuming existing `preamble` item events is the right boundary, but the implementation needs delivery cleaning, final-state guards, docs, and real Discord proof before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter commentary before rendering drafts:** `extensions/discord/src/monitor/message-handler.draft-preview.ts:434-440`
  - body: `normalizeCommentaryProgressText` currently only trims and italicizes raw `progressText`. That means commentary such as `NO_REPLY` or `[[reply_to_current]]` would be shown in the temporary Discord draft, even though the linked acceptance criteria says silent tokens and inline delivery directives should stay hidden. Please strip delivery directives and suppress silent-only commentary before building the progress line.
  - confidence: 0.88
- **[P2] Stop commentary after final delivery starts:** `extensions/discord/src/monitor/message-handler.draft-preview.ts:273-274`
  - body: `pushToolProgress` returns once `finalReplyStarted` or `finalReplyDelivered` is set, and existing tests lock that late progress events do not edit the draft after final delivery. The new commentary path lacks that guard, so a late preamble event can call `startNow()` and `renderProgressDraft()` after the final answer has started. Please add the same guard and cover late commentary in the Discord process tests.
  - confidence: 0.84
