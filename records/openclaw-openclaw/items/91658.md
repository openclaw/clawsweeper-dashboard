## Summary

Keep open: the feature is useful and the updated body now includes inspectable live proof, but the patch is still not merge-ready because absent or invalid `clientContext` leaves retained same-session diagnostic state in place, so later diagnostic events can inherit stale caller context.

## What This Changes

The branch adds an optional gateway agent `clientContext` object, normalizes it, stores it in diagnostic session state, emits it on `message.queued` and `session.state`, updates tests, and regenerates Swift protocol models.

## Best Possible Solution

Keep the generic protocol seam, but make diagnostic `clientContext` run-scoped by clearing or replacing it at every gateway agent run start and covering same-session absent, invalid, and oversized cases.

## Reproduction Assessment

Yes. A source-level repro is to seed `clientContext` for a session, then start another run on the same session with absent or rejected context and emit `session.state` or `message.queued`; the old context remains because the new setter returns before clearing retained state.

## Solution Assessment

No. The generic gateway/protocol idea is plausible, but the current implementation stores per-run data in retained session state without clearing it, so the narrow maintainable fix is to clear/replace it on each run start or move it to a run-scoped diagnostic path.
