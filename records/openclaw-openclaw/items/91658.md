## Label Justifications

- `P2`: This is a normal-priority feature PR with a concrete pre-merge correctness and proof blocker, not an already-shipped urgent regression.
- `merge-risk: 🚨 session-state`: The diff stores run context on retained diagnostic session state, which can stale or mis-associate context across later same-session runs.
- `merge-risk: 🚨 security-boundary`: The new plugin-visible event field can carry caller identity or run metadata, and stale retention can expose a previous run's context to later diagnostic consumers.

## Summary

Keep open: the feature is useful, but the patch is not merge-ready because the new context is stored on long-lived diagnostic session state and is never cleared when a later run on the same session has no valid context, so plugins can see stale attribution data. The PR also needs inspectable real behavior proof beyond tests and a prose-only downstream claim.

## What This Changes

The PR adds an optional gateway agent `clientContext` object, normalizes it, seeds it into diagnostic session state, and emits it on `message.queued` and `session.state` diagnostic events with focused tests.

## Best Possible Solution

Keep the generic gateway/protocol idea, but make `clientContext` run-scoped by clearing or replacing it at every run start and proving same-session absent/invalid context cannot inherit stale data.

## Reproduction Assessment

Yes. A focused source-level repro is to seed context for a session, complete or idle the run, call the new seeder with `undefined` for the same session, then emit `session.state` or `message.queued`; the earlier context remains because the setter returns early and session state is retained.

## Solution Assessment

No. The generic protocol seam is plausible, but the current implementation uses retained session state for per-run data without a clear path, so the best fix is to make the context explicitly run-scoped or clear it on every run start.
