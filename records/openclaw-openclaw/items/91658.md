## Summary

Keep open: the live proof is useful, but the PR still has a merge-blocking lifetime bug where a same-session later run without valid `clientContext` can emit the previous run's context to diagnostic listeners.

## What This Changes

The PR adds an optional gateway `agent` `clientContext` request field, normalizes it, stores it in diagnostic session state, emits it on selected diagnostic events, and regenerates Swift protocol models.

## Best Possible Solution

Keep the additive protocol shape only if maintainers want this core API, but first make `clientContext` explicitly set or cleared for every gateway agent run and cover same-session context-to-absent/invalid cases.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: seed a context on a session, start a later same-session run whose normalized context is undefined, then emit `session.state` or `message.queued`; the helper returns without clearing the prior context.

## Solution Assessment

No, not yet; the core API may be a plausible seam, but the implementation must make context a per-run value by replacing or clearing it at each run boundary before it is safe to merge.
