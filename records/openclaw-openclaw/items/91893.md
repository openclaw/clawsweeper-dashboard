## Summary

Keep open: current main and the latest release still stage `.gemini` through unfiltered recursive `fs.cpSync`, and the open closing PR at https://github.com/openclaw/openclaw/pull/91907 is the active fix candidate.

## What This Changes

The issue reports that live test HOME staging recursively copies `.gemini` browser cache/profile data into temp homes and can fill disk space during live tests.

## Best Possible Solution

Land one scoped external-auth staging fix that preserves auth/config files while excluding generated browser cache/profile/recording trees, then close this issue and duplicates after the fix reaches main.

## Reproduction Assessment

Yes, source-level. With `OPENCLAW_LIVE_TEST=1`, current main reaches `stageLiveTestState`, copies `.gemini` from `LIVE_EXTERNAL_AUTH_DIRS`, and uses recursive `fs.cpSync` without a filter.

## Solution Assessment

Yes. A narrow live external-auth staging filter or allowlist with regression coverage is the maintainable fix; the best current path is to review and land or narrow the linked PR rather than queue another duplicate repair.
