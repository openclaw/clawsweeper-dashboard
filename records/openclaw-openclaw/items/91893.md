## What This Changes

The issue reports that live-test HOME staging recursively copies `.gemini` browser cache data into temporary homes and can fill disk during live tests.

## Best Possible Solution

Land one narrow fix PR that excludes generated browser cache/profile directories during external-auth staging while preserving real auth/config files, then let the issue close from the merged PR.

## Reproduction Assessment

Yes: source inspection gives a high-confidence path because live-test staging copies `.gemini` through unfiltered recursive `fs.cpSync`. I did not create a multi-GB cache locally, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes: filtering generated browser cache/profile directories only in live external-auth staging is the narrowest maintainable fix. The chosen PR should prove that real auth/config files still copy while generated browser trees do not.
