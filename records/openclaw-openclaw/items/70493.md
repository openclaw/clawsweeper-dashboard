## Summary

Keep open. Current main still routes sandboxed or isolated browser tool calls to the sandbox bridge by default, and that bridge registers the same Playwright-gated browser routes; recent CDP fallback work helps role snapshots but does not cover full navigate/act support or satisfy the maintainer request for isolated-session regression coverage.

## What This Changes

The issue requests Playwright-backed browser navigate, snapshot, and act support to work consistently in isolated sub-agent sessions, matching the main session/user-browser path.

## Best Possible Solution

Define the browser runtime owner for isolated sessions, then either make the sandbox browser bridge Playwright-capable or explicitly delegate browser work to the host/node owner without weakening sandbox policy; add an isolated sub-agent regression that proves navigate or snapshot no longer returns the Playwright-unavailable error when the owning browser runtime supports Playwright.

## Reproduction Assessment

Yes. A high-confidence path is to run a sandboxed or isolated sub-agent with the browser tool allowed, call browser navigate or AI snapshot through the sandbox bridge, and verify the current route does not hit the requirePwAi 501 path; I did not execute the live macOS path in this read-only review.

## Solution Assessment

Unclear because there is no proposed patch. The best fix needs maintainer choice between sandbox Playwright packaging and host/node browser delegation, with SSRF checks, bridge auth, and allowHostControl preserved either way.
