## Summary

Current main does not read `SPAWN_ALLOWLIST`, while the documented and tested authorization path is config-backed `subagents.allowAgents`; deciding whether to add a Docker/env alias for this security-sensitive allowlist needs maintainer judgment rather than automatic cleanup.

## What This Changes

The issue requests `SPAWN_ALLOWLIST=*` in a Docker deployment to allow configured agents to be targeted by `sessions_spawn`.

## Best Possible Solution

Keep the canonical behavior in `openclaw.json` via `agents.defaults.subagents.allowAgents` or per-agent `subagents.allowAgents`; if env-only Docker control is desired, add an explicit documented mapping with authorization tests.

## Reproduction Assessment

Yes, by source inspection: current main has no `SPAWN_ALLOWLIST` reader, and the `sessions_spawn` authorization path only consults config-backed `subagents.allowAgents`. I did not run a live Docker reproduction because this read-only review cannot start containers or write runtime state.

## Solution Assessment

No, treating `SPAWN_ALLOWLIST` as an existing bug-fix path is not the best solution because it is not part of the current documented contract. The safer direction is either support/docs guidance to use `openclaw.json`, or a maintainer-approved env-to-config feature with tests.
