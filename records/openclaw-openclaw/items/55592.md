## Summary

Keep open. Current main still has UI-local message deletion, so the PR is useful, but the branch is not merge-ready because the destructive truncate path maps the UI sequence through raw JSONL order instead of the active transcript branch shown in chat history.

## What This Changes

The PR adds a `sessions.truncate` Gateway RPC with protocol and Swift model wiring, operator scope registration, server-side transcript rewrite logic, and Control UI delete handling.

## Best Possible Solution

Keep this PR open and require the truncate implementation to resolve the selected UI message through the active transcript branch entry or id before rewriting the transcript, with regression coverage plus changelog and protocol-doc updates.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a tree transcript with an abandoned raw entry before the active `clean -> answer` branch: current chat history exposes active-branch seq 1/2, while the PR counts the abandoned raw line as seq 1.

## Solution Assessment

No. The requested direction is valid, but this implementation is not the narrowest safe solution until it maps the UI target through the active-branch transcript semantics before any destructive rewrite.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Map truncate seqs through the active branch:** `src/gateway/server-methods/sessions.ts:1387-1390`
  - body: The UI sends `__openclaw.seq` values assigned after chat history selects the active transcript branch, but this loop increments for every raw JSONL `message` or `compaction` line. In a branched transcript with abandoned entries before the active branch, seq N can map to the wrong raw line and truncate unintended history; resolve the target through the active branch entry/id before slicing.
  - confidence: 0.9
