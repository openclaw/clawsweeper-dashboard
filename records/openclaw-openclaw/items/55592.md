## Summary

Keep open: the backend-sync direction addresses a real local-only delete bug, but PR head maps UI-visible active-branch seqs to raw JSONL order, so branched transcripts can truncate the wrong history.

## What This Changes

The PR adds a `sessions.truncate` Gateway RPC with protocol and Swift model wiring, operator scope registration, server-side transcript rewrite logic, and Control UI delete handling.

## Best Possible Solution

Map truncate targets through the active transcript branch, preferably by stable record id or a shared active-record-to-raw-line helper, then add regression coverage plus changelog/protocol docs before merge.

## Reproduction Assessment

Yes, source-reproducible. A transcript with an abandoned `original` root followed by active `clean -> answer` returns UI seqs 1 and 2 for the active branch, while the PR's raw scan counts `original` as seq 1.

## Solution Assessment

No. The requested backend-synced delete direction is valid, but raw JSONL ordinal slicing is not the narrowest safe implementation for current active-branch transcript semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Map truncate seqs through the active branch:** `src/gateway/server-methods/sessions.ts:1387-1390`
  - body: The UI sends `__openclaw.seq` after chat history selects the active transcript branch, but this loop counts every raw JSONL `message` or `compaction` record. In a branched transcript with an abandoned raw entry before the active branch, deleting a visible message can slice at the wrong raw line and remove history the user did not choose; resolve the target through the active branch entry/id before slicing.
