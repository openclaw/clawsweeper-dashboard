## Summary

This PR should stay open: current main still has the UI-local delete behavior the PR is trying to fix, but the proposed destructive transcript mutation is not mergeable while it maps UI seq values against raw JSONL order instead of active transcript branch semantics.

## What This Changes

The PR adds a `sessions.truncate` Gateway RPC, protocol/Swift model wiring, operator scope registration, server-side transcript rewrite logic, and Control UI delete handling so message deletion truncates the persisted transcript instead of only hiding a group locally.

## Best Possible Solution

Keep the PR open and update `sessions.truncate` to resolve the selected UI message through the active transcript branch entry or a stable message id before rewriting the transcript, with regression coverage for branched transcripts.

## Reproduction Assessment

Yes. Source inspection plus the existing active-branch regression test gives a high-confidence reproduction path: a transcript with an abandoned raw entry before the active `clean -> answer` branch exposes UI seq 1/2 while the PR's raw scan can map those seq values to different JSONL lines.

## Solution Assessment

No. Backend-synced deletion is the right direction, but this implementation is not safe until the destructive cut target is mapped through active-branch transcript semantics instead of raw line order.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Map truncate seqs through the active branch:** `src/gateway/server-methods/sessions.ts:1387-1390`
  - body: The UI sends `__openclaw.seq` values assigned after chat history selects the active transcript branch, but this loop increments for every raw JSONL `message` or `compaction` line. In a branched transcript with abandoned entries before the active branch, seq N can map to the wrong raw line and truncate unintended history; resolve the target through the active branch entry/id before slicing.
  - confidence: 0.9
