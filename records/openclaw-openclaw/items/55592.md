## Summary

Keep open. The user problem is valid, but the proposed destructive truncate path is not merge-ready because it maps user-visible message seqs through raw JSONL order, which can delete the wrong history in branched transcripts.

## What This Changes

The PR adds a `sessions.truncate` Gateway RPC with protocol/Swift wiring and Control UI delete handling so message deletion truncates backend transcript history.

## Best Possible Solution

Keep the PR open and require `sessions.truncate` to resolve the selected UI message through the active transcript record or id before any rewrite, with targeted gateway/UI regression coverage and user-facing changelog/protocol-doc updates if the RPC remains public.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a tree transcript with an abandoned raw `original` entry before the active `clean -> answer` branch: current main exposes active seq 1/2, while the PR counts the abandoned raw line as seq 1.

## Solution Assessment

No. The direction is valid, but mapping user-visible seqs through raw JSONL is not the narrowest safe implementation; the truncate target should be resolved through active-branch transcript identity first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Map truncate seqs through the active branch:** `src/gateway/server-methods/sessions.ts:1387-1390`
  - body: The UI sends `__openclaw.seq` values assigned after chat history selects the active transcript branch, but this loop increments for every raw JSONL `message` or `compaction` line. In a branched transcript with abandoned entries before the active branch, seq N can map to the wrong raw line and truncate unintended history; resolve the target through the active branch entry or id before slicing.
  - confidence: 0.9
