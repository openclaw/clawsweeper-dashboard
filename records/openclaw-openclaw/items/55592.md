## What This Changes

The PR adds a `sessions.truncate` Gateway RPC and wires protocol, Swift model, gateway, and Control UI delete handling so message deletion can truncate the backend transcript.

## Best Possible Solution

Resolve deletion against the same active transcript entry identity used by history reads, or pass a stable transcript id/offset from the UI, then require real behavior proof after rebase.

## Reproduction Assessment

Yes. Source inspection shows current history assigns `__openclaw.seq` after active-branch transcript selection, while the PR maps that number over raw JSONL records in the truncate handler.

## Solution Assessment

No. A `seq`-only raw scan is not the best fix because it does not share the active transcript index contract; the safer path is stable entry id/offset targeting or active-index-based resolution.
