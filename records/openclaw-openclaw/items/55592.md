## Summary

Keep open: this PR addresses a real local-only delete gap, but the current patch still maps the UI-visible delete target through raw JSONL order, which can truncate the wrong history in branched transcripts.

## What This Changes

The PR adds a `sessions.truncate` Gateway RPC, protocol and Swift model wiring, and Control UI delete handling so deleting a chat message truncates the backend transcript from that message onward.

## Best Possible Solution

Resolve truncate targets through the active transcript branch, preferably by stable record id or a shared active-record-to-raw-line helper, then land the protocol/UI wiring with regression coverage, docs, and changelog.

## Reproduction Assessment

Yes, source-reproducible. A transcript with an abandoned `original` root followed by active `clean -> answer` returns UI seqs 1 and 2 for the active branch, while the PR's raw scan counts `original` as seq 1.

## Solution Assessment

No. The backend-synced delete direction is valid, but raw JSONL ordinal slicing is not the narrowest maintainable solution for current active-branch transcript semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Map truncate seqs through the active branch:** `src/gateway/server-methods/sessions.ts:1387-1390`
  - body: The UI sends `__openclaw.seq` after chat history selects the active transcript branch, but this loop counts every raw JSONL `message` or `compaction` record. In a branched transcript with an abandoned raw entry before the active branch, deleting a visible message can slice at the wrong raw line and remove history the user did not choose; resolve the target through the active branch entry/id before slicing.
  - confidence: 0.9
