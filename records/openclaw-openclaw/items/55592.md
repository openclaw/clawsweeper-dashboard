## Summary

Keep open: this PR addresses a real Control UI/backend delete gap, but the current head still maps UI-visible sequence numbers to raw JSONL order and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `sessions.truncate` Gateway RPC, protocol/Swift model wiring, and Control UI delete handling, with small CSS/import/test formatting changes.

## Best Possible Solution

Keep the backend-synced delete direction, but resolve the selected UI message through active transcript records or a stable record id before slicing, then add regression coverage and real behavior proof.

## Reproduction Assessment

Yes. Current main is source-reproducible from the localStorage-only delete path, and the PR-specific bug is source-reproducible by combining the existing active-branch fixture with the PR head raw-line seq scan.

## Solution Assessment

No. A backend-synced delete is the right direction, but raw ordinal truncation is not the best implementation for current parent-chain transcript semantics; the backend should map the visible active record before mutating the file.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Map truncate seqs through the active transcript branch:** `src/gateway/server-methods/sessions.ts:1387-1390`
  - body: The UI sends `__openclaw.seq` after chat history has selected the active parent chain, but this loop counts every raw JSONL `message` or `compaction` record. In a branched transcript with an abandoned raw entry before the active branch, deleting visible seq 1 can truncate at the abandoned record instead of the selected message, removing history the user did not choose. Resolve the target through the active branch entry or a stable record id before slicing the file.
  - confidence: 0.92
