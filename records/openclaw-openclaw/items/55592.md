## Summary

Keep this PR open, but it is not merge-ready. It addresses a real UI/backend transcript mismatch, yet the proposed destructive truncate path still maps a rendered active-branch sequence number onto raw JSONL order, which can delete the wrong history range.

## What This Changes

Adds a `sessions.truncate` Gateway RPC, protocol/Swift model wiring, and Control UI delete integration to truncate backend transcripts, plus minor style and formatting edits.

## Best Possible Solution

Keep the backend-synced delete direction, but resolve the selected active-branch transcript record or stable record id before slicing JSONL, then add regression coverage and redacted real Control UI proof.

## Reproduction Assessment

Yes. Source inspection shows current main delete is localStorage-only, and PR head is source-reproducible because rendered seq values are active-branch-visible while the truncate handler counts raw JSONL lines.

## Solution Assessment

No. Backend-synced delete is the right direction, but raw ordinal truncation is unsafe for current parent-chain transcript semantics; the backend should resolve a selected active record or stable record id before mutation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Map truncate seqs through active-branch records:** `src/gateway/server-methods/sessions.ts:1387-1390`
  - body: The UI sends `__openclaw.seq` from rendered history, but rendered history is active-branch-filtered while this loop counts every raw message/compaction line.
