## Best Possible Solution

Keep the backend-synced delete direction, but target a stable active-branch transcript record or offset shared by the history reader and truncate handler, add branched-transcript regression coverage, then require redacted Control UI plus transcript proof before merge.

## Reproduction Assessment

## Solution Assessment

No. Backend-synced deletion is the right direction, but this implementation should target a stable active-branch transcript id or offset rather than treating UI `seq` as raw file order.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Target the active transcript record before truncating:** `src/gateway/server-methods/sessions.ts:1389`
  - body: Control UI history seqs are assigned after `readSessionMessages*` selects the active transcript branch, so they can diverge from raw JSONL order when abandoned branch entries remain.
