## Summary

Keep this PR open, but it still needs changes before merge. Current main still has the local-only Control UI delete problem, and the PR fixes the right problem area, but its backend truncate mapping counts raw JSONL records instead of the active transcript branch that current history readers expose to the UI.

## What This Changes

The PR adds a `sessions.truncate` Gateway RPC, protocol/Swift wiring, and Control UI delete handling so message deletion truncates the backend transcript instead of only hiding UI groups locally.

## Best Possible Solution

Keep the backend-synced delete direction, but resolve truncate targets through the selected active transcript branch, preferably with a shared active-record-to-raw-line helper or stable record id, then add regression coverage for abandoned raw records before merge.

## Reproduction Assessment

Yes. Source inspection shows current main deletes only through `DeletedMessages` localStorage, and the PR bug is source-reproducible from an existing active-branch fixture where visible seq 1 maps to the second raw message record.

## Solution Assessment

No. Backend-synced delete is the right repair direction, but raw JSONL ordinal slicing is not the best implementation for current active-branch transcript semantics; the backend should map the UI-visible record to its raw transcript line before truncating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Map truncate seqs through the active branch:** `src/gateway/server-methods/sessions.ts:1387-1390`
  - body: The UI sends `__openclaw.seq` after chat history has selected the active transcript branch, but this loop counts every raw JSONL `message` or `compaction` record. In a branched transcript with an abandoned raw entry before the active branch, deleting visible seq 1 can truncate at the abandoned record instead of the selected message, removing history the user did not choose. Resolve the target through the active branch entry/id before slicing the file.
  - confidence: 0.92
