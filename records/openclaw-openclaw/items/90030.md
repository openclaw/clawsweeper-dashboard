## Summary

Keep open for maintainer review: current main and the latest release still have the unguarded QMD zero-hit forced-sync path, this PR is the stronger active fix candidate because it has live QMD proof, and I found no blocking patch defect. The remaining merge decision is the deliberate QMD freshness versus bounded-latency tradeoff.

## What This Changes

The PR changes memory_search so QMD zero-hit searches skip the generic forced sync-and-retry path while non-QMD backends keep the existing retry behavior.

## Best Possible Solution

Land the backend-specific guard if maintainers accept bounded QMD search latency over forced immediate zero-hit freshness; otherwise add an async or debounced QMD refresh path outside the tool call before merge.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 still await forced sync after zero memory results, and QMD sync awaits update work with a longer timeout budget. The PR and linked issue also provide live QMD evidence, but I did not run an independent large-corpus timing repro in this read-only pass.

## Solution Assessment

Yes with maintainer acceptance: the PR is the narrowest owner-local fix to keep QMD zero-hit update maintenance out of memory_search while preserving non-QMD retry. If forced freshness is a product contract, the safer alternative is an async or debounced QMD refresh outside the tool call.
