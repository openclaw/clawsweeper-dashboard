The patch looks correct as a narrow fix, but it is currently conflicting and needs maintainer acceptance of the QMD freshness tradeoff before merge.

## What This Changes

The PR changes `memory_search` so QMD zero-hit results skip the synchronous forced sync/retry path while other backends keep the retry behavior.

## Best Possible Solution

Land one refreshed canonical memory-core fix for the QMD zero-hit stall: either this narrow QMD guard if maintainers accept prompt empty QMD results, or a bounded/background refresh design if immediate freshness repair must remain.

## Reproduction Assessment

Yes, at source level: current main still awaits the zero-hit forced sync, and QMD sync delegates to update work with a much larger timeout budget. I did not run an independent live large-corpus QMD timing reproduction in this read-only review.

## Solution Assessment

Yes with maintainer acceptance: the QMD guard is a narrow owner-local mitigation that preserves non-QMD retry behavior. If immediate QMD freshness repair is considered required, a bounded, async, or debounced refresh path outside `memory_search` is the safer alternative.
