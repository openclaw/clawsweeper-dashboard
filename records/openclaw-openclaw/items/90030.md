## Summary

Keep open: current main and the latest release still have the unguarded QMD zero-hit forced-sync path, and this PR remains a viable candidate fix, but it is currently conflicting and changes a compatibility-sensitive freshness fallback.

## What This Changes

The PR adds a QMD backend guard around memory_search's zero-hit forced sync retry and adds a regression test asserting QMD zero-hit searches do not call sync.

## Best Possible Solution

Land a refreshed backend-aware zero-hit fix, or a bounded/nonblocking QMD refresh variant, so QMD retrieval cannot block interactive memory_search while non-QMD retry behavior remains intact.

## Reproduction Assessment

Yes at source level: current main still synchronously forces manager.sync({ reason: "search", force: true }) after zero-hit memory_search results, and QMD sync waits on update maintenance. The linked issue also includes live deployment observations, but I did not run an independent large-corpus QMD timing repro in this read-only pass.

## Solution Assessment

Yes with maintainer acceptance: the guard is the narrow memory-core tool fix and preserves non-QMD retry behavior. If immediate QMD freshness repair is required, a bounded or nonblocking refresh path is the safer alternative.
