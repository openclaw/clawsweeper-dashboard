## Summary

Keep open. ## What This Changes

The PR changes memory-core search so QMD zero-hit searches skip the synchronous forced sync retry while keeping non-QMD zero-hit retry behavior.

## Best Possible Solution

Use one refreshed, reviewed backend-aware QMD guard to fix the linked issue, preserving non-QMD zero-hit repair and keeping QMD refresh out of the interactive tool hot path.

## Reproduction Assessment

Yes at source level: current `main` calls `manager.sync({ reason: "search", force: true })` after zero-hit memory searches when sync exists, and QMD `sync` awaits update work. The linked issue also includes live 2026.6.5 before/after behavior, but I did not run an independent live QMD timing reproduction in this read-only pass.

## Solution Assessment

Yes with maintainer acceptance: a QMD-only guard is the narrow owner-local fix that keeps update work out of interactive search while preserving non-QMD repair. If maintainers want zero-hit freshness to remain contractual for QMD, the safer alternative is nonblocking refresh outside the tool call.
