## Summary

Keep open. This is a focused, proof-backed fix for the linked QMD memory_search stall, and current main plus the latest release still have the unguarded zero-hit forced-sync path; the remaining blockers are maintainer acceptance of the QMD freshness tradeoff and the PR's current dirty/conflicting merge state.

## What This Changes

The PR changes memory-core so QMD-backed zero-hit memory_search calls skip the synchronous forced sync-and-retry path while preserving the existing non-QMD retry and adding a regression test.

## Best Possible Solution

Land a rebased backend-aware guard if maintainers accept bounded QMD search latency over forced immediate freshness; otherwise require a nonblocking or debounced QMD refresh path before merge.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 await a forced manager sync after zero memory_search hits, while QMD sync awaits qmd update with a much larger timeout budget. The linked issue also supplies live v2026.6.5 timeout evidence, but I did not run a live QMD timing repro in this read-only pass.

## Solution Assessment

Yes, with maintainer acceptance: the guard is the narrowest owner-local fix to keep QMD update maintenance out of the interactive search path while preserving non-QMD zero-hit repair. If forced QMD freshness remains a contract, the safer alternative is a nonblocking or debounced refresh path.
