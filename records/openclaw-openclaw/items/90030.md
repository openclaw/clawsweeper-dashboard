## Summary

Keep open for maintainer review. The patch matches the source-proven QMD stall path and I found no blocking code defect, but merging intentionally changes QMD zero-hit freshness behavior.

## What This Changes

The PR changes `memory_search` so QMD zero-hit searches skip the generic forced sync-and-retry path while preserving the existing non-QMD retry behavior.

## Best Possible Solution

Land the backend-specific guard if maintainers accept bounded QMD search latency over forced immediate freshness; otherwise add an async or debounced QMD refresh path outside the tool call before merge.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.1` force-sync after zero-hit search, and QMD sync awaits update work. I did not execute an independent live QMD timing repro in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance: the guard is the smallest owner-local fix to keep QMD update work out of `memory_search` while preserving non-QMD zero-hit repair. If forced freshness remains a contract, a nonblocking refresh path is safer than the current synchronous fallback.
