## Label Justifications

- `P2`: This is a normal-priority bug fix for QMD-backed `memory_search` stalls with limited blast radius to the memory-core plugin.
- `merge-risk: 🚨 compatibility`: The PR deliberately removes QMD’s zero-hit forced sync-and-retry fallback, which can change upgrade behavior for users relying on that repair path.

## Summary

Keep open for maintainer review: current main and the latest release still have the QMD stall path, and this PR’s small guard, regression test, and live proof address it. The remaining issue is intentional compatibility risk: QMD zero-hit searches no longer force an immediate refresh/retry, so maintainers should explicitly accept that freshness tradeoff before merge.

## What This Changes

This PR changes `memory_search` so QMD zero-hit searches skip the generic forced sync/retry path while non-QMD backends keep the existing retry behavior.

## Best Possible Solution

Land the QMD-specific hot-path guard if maintainers accept returning zero hits instead of forcing `qmd update` inside the interactive search path; otherwise request an async or debounced QMD refresh follow-up before merge.

## Reproduction Assessment

Yes, from source inspection: current main and `v2026.6.1` call forced `sync` after zero hits, and QMD `sync` enters `runUpdate`. I did not run a local QMD reproduction in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance: this is a narrow owner-boundary fix that leaves the generic non-QMD retry intact and avoids QMD maintenance in the hot path. The safer alternative, if freshness must be preserved, is an async or debounced QMD refresh instead of blocking the search call.
