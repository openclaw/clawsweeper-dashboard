## Summary

Keep open for maintainer review: the patch is a narrow, source-backed fix for QMD zero-hit stalls, but it intentionally changes QMD zero-hit freshness semantics, so the remaining gate is maintainer acceptance of that tradeoff.

## What This Changes

The PR adds a QMD backend guard so zero-hit `memory_search` skips the generic forced sync/retry path, plus a regression test that QMD zero-hit searches do not call sync.

## Best Possible Solution

Land the backend-specific guard if maintainers accept prompt QMD zero-hit returns; otherwise change the PR to schedule or debounce QMD refresh outside the tool call.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.1` force-sync after zero-hit search, and QMD `sync` awaits update work. I did not run an independent live QMD timing reproduction in this read-only review, but the PR supplies after-fix live output.

## Solution Assessment

Yes, with maintainer acceptance: the guard is the smallest maintainable path to remove QMD update work from search while preserving non-QMD zero-hit retry. If immediate QMD freshness is required, an async or debounced refresh path is safer than blocking the tool call.
