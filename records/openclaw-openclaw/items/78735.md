Keep this PR open. The current-main gateway path does accept and forward the unsafe install override, and the PR appears to narrow that path correctly with no blocking diff-level finding, but it removes a documented gateway protocol/security behavior and should get human owner review before merge.

## What This Changes

The PR rejects `dangerouslyForceUnsafeInstall=true` for gateway `skills.install`, forces gateway skill installs to run with scans enforced, removes client-sent fields, and updates related docs/tests/changelog.

## Best Possible Solution

Land the gateway boundary hardening after gateway/security owner review, keeping the CLI plugin break-glass override separate from gateway-backed skill dependency installs.

## Reproduction Assessment

Yes from source inspection: current main accepts the optional boolean, forwards it through `skills.install`, and the scanner skips critical blocking when the value is true. I did not run a live gateway request in this read-only review.

## Solution Assessment

Yes in direction: rejecting `true` at schema/handler boundaries and forcing `false` into the installer is the narrow maintainable fix. Because it removes a documented protocol override, owner review is the safer merge path.
