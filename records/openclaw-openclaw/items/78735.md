## Summary

Keep this PR open but do not merge it as-is. The security direction is valid, but the patch hard-removes a gateway protocol field while an existing Control UI caller still sends that field, leaves public docs/changelog stale, and has no real gateway behavior proof from the contributor.

## What This Changes

The PR removes the `dangerouslyForceUnsafeInstall` field from the gateway `skills.install` schema, forces gateway skill installs to pass `dangerouslyForceUnsafeInstall: false`, and adds schema/handler tests.

## Best Possible Solution

Stop honoring unsafe gateway override requests while preserving compatibility for existing false-valued callers, update affected clients/docs/changelog, and verify a real gateway `skills.install` request path where `true` cannot bypass scans and normal installs still work.

## Reproduction Assessment

Yes. Current main is source-reproducible: the schema accepts `dangerouslyForceUnsafeInstall`, the gateway handler forwards it, and the scanner bypasses critical blocks when it is true; the patch-introduced Control UI regression is also visible from source because the UI always sends the flag as false.

## Solution Assessment

No. The security goal is right, but the current patch is not the best fix until it either updates every caller and protocol/doc surface for a breaking removal or keeps the legacy field accepted while refusing/ignoring unsafe true values.
