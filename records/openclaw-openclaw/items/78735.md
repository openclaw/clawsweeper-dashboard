## Summary

Keep this PR open: it addresses a real source-visible gateway scan bypass, but the patch still leaves one gateway client surface and two public docs pages advertising the removed override, and current-head real behavior proof is not accepted by the required proof gate.

## What This Changes

This PR changes gateway `skills.install` to reject `dangerouslyForceUnsafeInstall: true`, always pass `false` into skill installation, and updates web UI, docs, changelog, and gateway tests around that protocol change.

## Best Possible Solution

Enforce gateway install scans server-side, remove or neutralize the unsafe override from every gateway client and public doc surface, keep local CLI/plugin break-glass behavior separate, and merge only after current-head proof plus maintainer approval of the protocol/security compatibility choice.

## Reproduction Assessment

Yes for source reproduction: current main accepts and forwards the gateway override, and the scanner skips critical blocks when that flag is true. I did not run a live exploit, but the code path is direct.

## Solution Assessment

No, not yet. The runtime hardening is the right direction, but the protocol change is not fully synced across client/docs surfaces and current-head real behavior proof is still not accepted.
