## Summary

Keep open. The PR addresses the source-visible gateway scan bypass, but it still has blocking docs/changelog issues, mock-only proof for an external security-sensitive change, a dirty head, and an unresolved breaking gateway protocol decision.

## What This Changes

The PR removes `dangerouslyForceUnsafeInstall` from gateway `skills.install`, forces gateway installs to pass `dangerouslyForceUnsafeInstall: false`, updates the UI request/tests, and adds schema/handler tests plus protocol docs and changelog text.

## Best Possible Solution

Enforce security scans server-side for gateway installs, make an explicit compatibility/versioning choice for the protocol field removal, align all public docs and changelog text, and require redacted real gateway proof before merge.

## Reproduction Assessment

Yes for source reproduction: current main accepts `dangerouslyForceUnsafeInstall`, forwards it through gateway `skills.install`, and the scanner bypasses critical blocks when it is true. I did not establish a live exploit path.

## Solution Assessment

No, not yet. Forcing `false` in the gateway handler addresses the core bypass, but rejecting the field at the protocol boundary needs a compatibility/versioning decision, complete docs cleanup, and real runtime proof.
