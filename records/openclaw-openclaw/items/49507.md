## Summary

Keep open. Current main partially improves the no-account path through bound account lookup, but cron delivery can still choose a session-derived lastAccountId before that lookup, so the reported multi-account misrouting class remains source-reproducible and security-sensitive.

## What This Changes

The issue requests cron delivery to infer the correct channel account for a job agent when delivery.accountId is omitted, avoiding Feishu/Lark and other multi-account sends through the wrong account.

## Best Possible Solution

Define and land a focused cron account-routing policy where explicit accountId wins, isolated cron does not leak stale session accounts, bound agent/channel accounts are used when safe, and Feishu/Telegram multi-account cases have regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted delivery.accountId can flow through session lastAccountId before bound-account lookup, and current tests assert that precedence.

## Solution Assessment

Unclear. The reported misrouting is real, but the proposed raw agentId-to-accountId fallback is not the best fix because current docs and PR review treat agentId and accountId as separate identifier types.
