- `impact:security`: The issue reports scheduled messages being sent through the wrong account, which can cross account and trust boundaries.
- `impact:message-loss`: The reported failure is misrouted outbound channel delivery for cron announcements.
- `impact:session-state`: The source path can use persisted session account state ahead of the intended agent/channel binding.

## Summary

Keep open: current main and the latest release still let session-derived account state win before agent/channel binding for cron delivery, and the related direct agentId-to-accountId PRs were closed unmerged. The remaining fix needs a maintainer account-precedence and security policy decision.

## What This Changes

Requests cron announce delivery to infer the outbound account from agentId when delivery.accountId is omitted, so Feishu/Telegram jobs do not send from a stale or default account.

## Best Possible Solution

Keep explicit and preserved live delivery context authoritative, but prevent stale session lastAccountId from silently winning over an unambiguous agent/channel binding; fail closed or require delivery.accountId for ambiguous multi-account crons.

## Reproduction Assessment

Yes, source-level: current main can resolve an omitted cron delivery.accountId from stored session account state before consulting the agent/channel binding. I did not run a live Feishu or Telegram repro in this read-only review.

## Solution Assessment

No, the direct agentId == accountId fallback proposed in the closed PRs is too naive for configured bindings and existing session/live-context semantics. The better fix is an explicit precedence policy plus fail-closed handling for ambiguous multi-account delivery.
