## Label Justifications

- `P2`: Wrong-account cron delivery is a normal-priority bug with bounded but real multi-account impact.
- `impact:security`: The issue describes scheduled content being sent through the wrong trusted account in multi-account channel setups.
- `impact:message-loss`: Cron messages can be misrouted through the wrong outbound account or recipient context.
- `impact:session-state`: Stored session account state can override the intended agent/channel binding when `accountId` is omitted.

## Summary

Keep open: current main and the latest release still resolve cron account selection as explicit `accountId`, then session-derived `lastAccountId`, then agent/channel binding, and a current test explicitly locks the session-before-binding behavior reported here.

## What This Changes

The issue reports cron delivery using the wrong Feishu/Telegram account when `accountId` is omitted and asks for agent-aware account inference.

## Best Possible Solution

Define a maintainer-approved cron account precedence rule that keeps explicit `delivery.accountId` first, then uses the correct binding or stored-context behavior with regression coverage for stale `lastAccountId`, bindings, and explicit overrides.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current `resolveDeliveryTarget` lets session-derived account state win before agent/channel binding, and the current test suite asserts that behavior.

## Solution Assessment

No. Direct `agentId` to `accountId` inference is not the best fix because those identifiers are separate contracts; the maintainable fix is a binding-aware precedence rule with explicit upgrade behavior.
