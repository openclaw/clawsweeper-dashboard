## Label Justifications

- `P2`: The issue is source-reproducible and security-relevant, but limited to multi-account cron delivery rather than a broad outage.
- `impact:security`: Scheduled output can be sent through an unintended configured account in deployments with different trust boundaries.
- `impact:message-loss`: The affected cron message can be misrouted through the wrong channel account.
- `impact:session-state`: A stale session lastAccountId can influence later cron delivery identity.

## Summary

Keep open. Current main and v2026.5.18 still allow cron delivery without an explicit account to reuse a session-derived lastAccountId before the agent/channel binding, and the existing tests assert that precedence.

## What This Changes

Cron jobs with an agentId but no accountId can send through a stale or unintended Feishu/Telegram account; the request is to infer a safe outbound account automatically.

## Best Possible Solution

Define and test cron account-routing precedence: explicit delivery account wins, isolated cron does not leak stale session accounts, and Feishu/Telegram use safe bindings or channel defaults rather than raw id equality.

## Reproduction Assessment

Yes, at source level. An isolated cron job with omitted delivery.accountId can receive resolved.accountId from session lastAccountId before bound-account lookup, and the current test suite asserts that session-account wins.

## Solution Assessment

No, the issue's raw accountId = agentId proposal is not the best general fix because current docs treat those as distinct identifiers. The safer solution is a maintainer-approved resolver precedence policy with regression coverage for explicit account, stale session account, bindings, and defaults.
