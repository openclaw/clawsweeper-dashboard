## Label Justifications

- `P2`: The issue is source-reproducible and security-relevant, but scoped to multi-account cron delivery rather than a broad outage.
- `impact:security`: Scheduled output can be sent through an unintended configured account in deployments with different trust boundaries.
- `impact:message-loss`: The affected cron message can be misrouted through the wrong channel account.
- `impact:session-state`: A stale session lastAccountId can influence later cron delivery identity.

## Summary

Keep open. Current main and v2026.5.18 still choose a session-derived lastAccountId before bound account resolution for isolated cron delivery, and the tests assert that precedence, so this remains a valid security-sensitive account-routing bug rather than cleanup close.

## What This Changes

Cron jobs with an agentId but no explicit accountId should avoid sending through the wrong Feishu or other channel account by resolving account identity safely from agent/channel routing.

## Best Possible Solution

Define resolver precedence and fix cron so explicit account wins, omitted account does not inherit stale session account for isolated scheduled sends, and bound/default account selection is covered by regression tests.

## Reproduction Assessment

Yes, at source level. An isolated cron job with omitted delivery.accountId can receive resolved.accountId from session lastAccountId before bound-account lookup, and the current test suite asserts that session-account wins.

## Solution Assessment

No, the raw accountId = agentId proposal is not the best general fix because current docs treat those as distinct identifiers. The safer solution is a maintainer-approved resolver precedence policy with regression coverage for explicit account, stale session account, bindings, and defaults.
