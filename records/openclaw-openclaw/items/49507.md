## Summary

Keep open: current main and the latest release still only partially address the account-selection problem. Cron delivery now falls back to an agent/channel binding when no session account exists, but stored session account state still wins first and there is no maintainer-approved no-binding policy for multi-account cron sends.

## What This Changes

Cron jobs with an agentId but no delivery.accountId can send scheduled Feishu or Telegram output through a stale/default account instead of the intended agent account.

## Best Possible Solution

Define one canonical cron account contract: explicit delivery.accountId wins, safe agent/channel bindings are used when available, and ambiguous multi-account cron sends warn or fail closed instead of silently using stale session/default account state.

## Reproduction Assessment

Yes, source-level: current main still lets stored session account state outrank the agent binding and leaves ambiguous no-binding multi-account cases to channel/default behavior. I did not run a live Feishu or Telegram cron in this read-only review.

## Solution Assessment

No for the suggested direct accountId=agentId fix, because agent ids and channel account ids are different identifier types. The better fix needs a maintainer-approved binding/fail-closed account precedence policy with regression coverage.
