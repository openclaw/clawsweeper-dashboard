## Summary

Keep open: current main and the latest release still preserve session-derived account state ahead of agent-bound account resolution, so the Feishu/Telegram wrong-account cron path is not fully fixed and needs maintainer policy input before changing precedence.

## What This Changes

The issue reports cron deliveries using the wrong Feishu or Telegram account when `accountId` is omitted and requests automatic account selection from the job's agent context.

## Best Possible Solution

Define one canonical cron account-resolution contract: explicit `delivery.accountId` wins, safe agent/channel bindings are used when available, and multi-account cron delivery without a safe account warns or fails instead of silently using stale session/default account state.

## Reproduction Assessment

Yes, source-level: current main still lets stored session `lastAccountId` outrank the agent binding, and the adjacent test asserts that precedence. I did not run a live Feishu or Telegram cron in this read-only review.

## Solution Assessment

No for the proposed direct `accountId = agentId` fix: agent ids and channel account ids are different identifiers. A maintainable fix should use configured bindings or explicit account enforcement, with tests for stale session history and no-binding multi-account cases.
