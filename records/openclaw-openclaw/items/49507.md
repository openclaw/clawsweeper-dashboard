## Summary

Keep open. Current main still leaves cron delivery account routing unresolved for multi-account jobs: the resolver can use stale session `lastAccountId`, then bound-account config, but it does not implement the requested `agentId`-based account inference when no account is available, and the direct fix PRs are closed unmerged.

## What This Changes

The issue requests cron delivery to infer the correct outbound `accountId` from a job's `agentId` when `delivery.accountId` is omitted, instead of sending through a default or stale channel account.

## Best Possible Solution

Land a focused cron delivery identity-routing fix that keeps explicit `delivery.accountId` authoritative, resolves the job agent's bound outbound account before trusting stale cron session account state where appropriate, and documents or constrains any `agentId`-as-account fallback. Add regression coverage for explicit account, binding-derived account, stale session account, no binding, and Feishu/Telegram-style multi-account delivery.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists in the resolver: an isolated cron job with `agentId`, channel delivery, no explicit `accountId`, and no binding returns `undefined`, while a matching session can supply `lastAccountId`; current tests assert those behaviors.

## Solution Assessment

Unclear. The closed PRs' raw `accountId = agentId` fallback is directionally related but likely too broad; the safer fix is a tested precedence update that distinguishes explicit cron delivery, bound accounts, stale session state, and any documented compatibility fallback.
