## Label Justifications

- `P2`: Wrong-account cron delivery is a normal-priority but security-relevant scheduled messaging bug with a bounded channel-delivery blast radius.
- `impact:security`: The report and Telegram reproduction describe sensitive scheduled messages leaking through the wrong account boundary in multi-account setups.

## Summary

Keep open: current main has a partial binding-based fallback, but it still preserves session-derived account state ahead of the agent binding and does not implement a safe no-binding account resolution for this cron path. The related direct `accountId = agentId` PRs were closed unmerged, and no merged or open canonical PR safely owns this exact Feishu/Telegram cron account-selection problem.

## What This Changes

The issue requests cron delivery to choose the correct outbound channel account when `agentId` is set but `accountId` is omitted, instead of sending scheduled Feishu/Telegram messages under a stale or default account.

## Best Possible Solution

Define one canonical cron account-resolution contract: explicit `delivery.accountId` wins, configured agent/channel bindings are used when safe, and multi-account crons without a safe account should warn or fail closed instead of silently using stale session/default account state.

## Reproduction Assessment

Yes, source-level: current main still lets stored session account state outrank the agent binding and leaves no-binding cron deliveries without an inferred account, after which channel defaults can apply. I did not run a live Feishu or Telegram cron in this read-only review.

## Solution Assessment

No for the proposed direct `accountId = agentId` fix: the closed related PR discussion correctly flags agent ids and channel account ids as different identifiers. A maintainable fix should use configured bindings or explicit account enforcement, with tests for stale session history and no-binding multi-account cases.
