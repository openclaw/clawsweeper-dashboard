## Summary

Keep open. Current main still resolves cron delivery accounts from explicit delivery.accountId, session lastAccountId, then bound account config, but it does not implement the requested agentId-derived fallback; that account choice is passed directly into outbound delivery, so the reported multi-account misrouting class remains plausible.

## What This Changes

Cron jobs with an agentId but no accountId should avoid sending through the wrong Feishu or other channel account, with the proposed fallback of inferring accountId from agentId.

## Best Possible Solution

Land a focused cron account-routing fix that keeps explicit delivery.accountId authoritative, avoids stale session-account leakage for isolated cron delivery, and covers binding, default-account, and agent/account-id compatibility cases with regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: isolated cron channel delivery with agentId and no explicit accountId can use session lastAccountId, a binding, or no account, not an agentId-derived account.

## Solution Assessment

Unclear. The reported problem is valid, but the proposed raw agentId-to-accountId fallback is not clearly the safest fix because current docs model agentId and accountId as separate concepts.
