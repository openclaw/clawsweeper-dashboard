## Label Justifications

- `P2`: Wrong-account cron delivery is a normal-priority bug with bounded but real multi-account security and delivery impact.
- `impact:security`: The issue describes scheduled content being sent through the wrong trusted account in multi-account channel setups.
- `impact:message-loss`: Cron messages can be misrouted through the wrong outbound account or recipient context.
- `impact:session-state`: Stored session account state can override the intended agent/channel account when `accountId` is omitted.

## Summary

Keep open: current main and v2026.6.10 still let session-derived `lastAccountId` win before the agent/channel binding when cron delivery omits `accountId`, so the wrong-account cron delivery problem remains real and security-sensitive.

## What This Changes

The issue requests that cron delivery avoid using the wrong Feishu/channel account when `agentId` is set but `accountId` is omitted.

## Best Possible Solution

Define and implement a maintainer-approved cron account precedence rule that keeps explicit `delivery.accountId` first, then resolves the correct agent/channel binding or stored context deliberately with regression coverage.

## Reproduction Assessment

Yes, at source level: current main still lets session-derived `lastAccountId` win before the agent/channel binding, and the current test suite asserts that behavior. I did not run a live Feishu or Telegram multi-account cron reproduction.

## Solution Assessment

No: naive `agentId` to `accountId` inference is not the best fix because those identifiers are separate contracts. The maintainable fix is a binding-aware precedence rule with explicit upgrade/security behavior.
