## Summary

Keep open. Current main and the latest release still allow cron delivery without an explicit `delivery.accountId` to reuse a session-derived `lastAccountId` before consulting the agent/channel binding, and the current tests assert that precedence, so the reported multi-account misrouting class remains source-reproducible.

## What This Changes

The issue asks cron delivery to use the correct Feishu/channel account when `agentId` is set and `delivery.accountId` is omitted, with suggested `agentId` to `accountId` inference.

## Best Possible Solution

Define a cron account-routing policy where explicit account wins, isolated cron does not leak stale session accounts, safe agent/channel bindings or channel defaults are used when appropriate, and Feishu/Telegram multi-account cases have regression coverage.

## Reproduction Assessment

Yes, at source level. An isolated cron job with omitted `delivery.accountId` can receive `resolved.accountId` from session `lastAccountId` before bound-account lookup, and the current test suite asserts that precedence.

## Solution Assessment

No, the raw `agentId` to `accountId` fallback proposed in the issue is not the best general fix because current docs treat those as different identifiers. The safer solution is an explicit resolver precedence policy for explicit delivery account, trusted session state, bindings, and defaults.
