## Label Justifications

- `P2`: The issue is source-reproducible and security-relevant, but limited to multi-account cron delivery rather than a broad outage.
- `impact:security`: Scheduled output can be sent through an unintended configured account in deployments with different trust boundaries.
- `impact:message-loss`: Cron output can be delivered through the wrong channel account identity, which is a misrouted message-delivery problem.
- `impact:session-state`: A stale session `lastAccountId` can influence later cron delivery identity.

## Summary

Keep open. Current main and the latest release still allow isolated cron delivery without an explicit account to reuse a session-derived account before bound-account/default routing, and the safe fix needs a maintainer account-routing policy rather than raw `agentId` equals `accountId` inference.

## What This Changes

The issue asks cron announce delivery to choose the correct Feishu/channel account when `agentId` is set and `delivery.accountId` is omitted, originally suggesting `agentId` to `accountId` inference.

## Best Possible Solution

Define and test cron account-routing precedence: explicit account wins, isolated scheduled sends must not leak stale session accounts, and omitted accounts should resolve through safe bindings or documented explicit-account validation rather than raw id equality.

## Reproduction Assessment

Yes, at source level. An isolated cron job with omitted `delivery.accountId` can receive `resolved.accountId` from session `lastAccountId` before bound-account lookup, and current tests assert that session-account wins.

## Solution Assessment

No for the raw `agentId` to `accountId` fallback. The maintainable fix is a maintainer-approved resolver precedence policy with regression coverage for explicit account, stale session account, bindings, and defaults.
