## Label Justifications

- `P2`: Wrong-account cron delivery is a real multi-account correctness and security issue, but the blast radius is bounded to scheduled/channel delivery.
- `impact:security`: The report describes scheduled messages and permissions using a different account identity than the intended agent account.
- `impact:message-loss`: Cron output can be delivered through the wrong account or target, so the intended recipient may not receive the scheduled message.
- `impact:session-state`: The source path depends on stored session delivery metadata such as `lastAccountId` overriding the agent's intended account binding.

## Summary

Keep open: current main and the latest release still let session-derived `lastAccountId` beat the agent/channel binding for cron delivery, so the wrong-account path reported here remains possible and is not owned by any merged fix.

## What This Changes

Cron jobs with `agentId` but no explicit `accountId` can send scheduled messages through the wrong Feishu or channel account.

## Best Possible Solution

Adopt a maintainer-approved cron account precedence rule where explicit `delivery.accountId` wins and omitted accounts cannot silently reuse stale session identity over the intended agent binding; document and test the upgrade behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible: cron delivery still chooses explicit account, then session-derived account, then binding fallback, and a current test asserts session account precedence over binding.

## Solution Assessment

No for the literal `agentId` equals `accountId` suggestion. The maintainable fix should be binding-aware and explicit about session-vs-binding precedence, because that ordering is security- and upgrade-sensitive.
