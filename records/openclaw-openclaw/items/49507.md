## Label Justifications

- `P2`: The issue is source-reproducible and security-relevant, but limited to multi-account cron delivery rather than a broad outage.
- `impact:security`: Scheduled output can be sent through an unintended configured account in deployments with different trust boundaries.
- `impact:message-loss`: Cron output can be delivered through the wrong channel account identity, which is a misrouted message-delivery problem.
- `impact:session-state`: A stale session `lastAccountId` can influence later cron delivery identity.

## Summary

Keep open. Current main and v2026.5.20 still allow cron delivery without an explicit account to reuse a session-derived `lastAccountId` before bound-account or default-account routing, and the existing tests assert that precedence. The raw `agentId` to `accountId` fallback proposed here is not safely implied by current account contracts.

## What This Changes

Cron announce delivery should avoid sending through an unintended Feishu, Telegram, or similar channel account when `agentId` is set but `delivery.accountId` is omitted.

## Best Possible Solution

Define and test cron account-routing precedence: explicit account wins, isolated scheduled sends must not leak stale session accounts, and omitted accounts should resolve through safe bindings or documented defaults rather than raw id equality.

## Reproduction Assessment

Yes, at source level. An isolated cron job with omitted `delivery.accountId` can receive `resolved.accountId` from session `lastAccountId` before bound-account lookup, and the current test suite asserts that session-account wins.

## Solution Assessment

No for the raw `agentId` to `accountId` fallback. The maintainable fix is a maintainer-approved resolver precedence policy with regression coverage for explicit account, stale session account, bindings, and defaults.
