## Summary

Keep open. Current main and the latest release still allow cron delivery without an explicit accountId to reuse a session-derived lastAccountId before binding/default routing, and changing that precedence is security-sensitive enough to require maintainer policy.

## What This Changes

The issue asks cron delivery to avoid using the wrong Feishu account when agentId is set but delivery.accountId is omitted, with a suggested agentId-to-accountId fallback.

## Best Possible Solution

Define and document cron account-routing precedence so explicit account wins, isolated cron does not inherit unsafe stale session accounts, and safe agent/channel bindings or channel defaults are used consistently with Feishu/Telegram regression coverage.

## Reproduction Assessment

Yes, at source level. An isolated cron job with omitted delivery.accountId can take resolved.accountId from the session lastAccountId before bound-account lookup, and the current test suite asserts that precedence.

## Solution Assessment

No for the proposed raw agentId-to-accountId fallback, because current docs treat those as different identifiers. The safer solution is an explicit resolver precedence policy for explicit delivery account, trusted session state, bindings, and defaults.
