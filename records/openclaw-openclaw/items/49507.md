## Label Justifications

- `P2`: The issue is source-reproducible and security-relevant, but limited to multi-account cron delivery rather than a broad outage.
- `impact:security`: Scheduled output can be sent through an unintended configured account in deployments with different trust boundaries.
- `impact:message-loss`: The affected cron message can be misrouted through the wrong channel account.
- `impact:session-state`: A stale session `lastAccountId` can influence later cron delivery identity.

## Summary

Keep open. Current main and the latest tagged release still let isolated cron delivery without an explicit account reuse a session-derived `lastAccountId` before consulting agent/channel bindings, and the existing regression test asserts that precedence. The related Slack identity issue at https://github.com/openclaw/openclaw/issues/49313 was closed for display identity, not this account-selection problem, and the raw `agentId` to `accountId` PR at https://github.com/openclaw/openclaw/pull/49633 was closed unmerged with concrete compatibility concerns.

## What This Changes

Cron delivery should avoid sending scheduled Feishu, Telegram, or similar messages through the wrong account when a job has an `agentId` but no explicit `accountId`.

## Best Possible Solution

Define and test cron account-routing precedence: explicit delivery account wins, isolated cron must not leak stale session accounts across scheduled sends, and omitted accounts should resolve through safe bindings or documented channel defaults rather than raw id equality.

## Reproduction Assessment

Yes, at source level. An isolated cron job with omitted `delivery.accountId` can receive `resolved.accountId` from session `lastAccountId` before bound-account lookup, and the current test suite asserts that session-account wins.

## Solution Assessment

No, the raw `agentId` to `accountId` proposal is not the best general fix because docs and current config treat those as distinct identifiers. The safer path is a maintainer-approved resolver precedence policy with regression coverage for explicit account, stale session account, bindings, and defaults.
