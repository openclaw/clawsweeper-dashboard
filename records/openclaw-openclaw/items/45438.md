## Summary

Keep open. Current main fixed the session-store cache/load `structuredClone` hot path, but auth-profile cache, runtime snapshot, and runtime external OAuth overlay paths still clone through `structuredClone`, and maintainer discussion explicitly left this issue open for that remaining work plus benchmark or soak confirmation.

## What This Changes

The issue asks OpenClaw to remove `structuredClone` from session/auth store clone hot paths to prevent gateway native-memory growth and OOM crashes.

## Best Possible Solution

Finish or explicitly defer the auth-profile clone cleanup in the auth-profile owner area, add focused mutation-isolation/no-structuredClone coverage for persisted and runtime auth stores, and validate the large-store memory workload before closing.

## Reproduction Assessment

Yes for the reported issue class. The discussion includes production OOM evidence and a 500 sessions x 10KB x 2000 clone-cycle benchmark, while current source inspection confirms remaining auth-profile `structuredClone` paths; I did not rerun the benchmark because this review is read-only.

## Solution Assessment

Yes, keeping this open is the best current resolution. Current main is only a partial fix, and the remaining credential-sensitive auth-profile work plus the open implementation candidate need maintainer triage and memory validation before closure.
