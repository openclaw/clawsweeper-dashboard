## Summary

Keep open: current main still caches and single-flights bundled MCP runtimes by session id, while ordinary gateway agent runs preserve those runtimes and each runtime owns its own stdio child, so the duplicate same-config child-process leak remains possible.

## What This Changes

The issue asks OpenClaw to stop gateway lazy-loading from accumulating duplicate stdio MCP child processes for the same bundled MCP server/config across separate sessions.

## Best Possible Solution

Add a workspace/config-fingerprinted bundled MCP runtime ownership layer with single-flight startup and explicit leases, preserving existing cleanup paths without adding a process-table sweeper.

## Reproduction Assessment

Yes. The report includes concrete process/timing evidence, and current main has a static reproduction path: request bundled MCP runtimes for two different session ids with the same workspace/config and each id can own a separate stdio transport.

## Solution Assessment

Yes. A config-keyed single-flight runtime/lease manager is the narrow maintainable direction; killing duplicate processes after the fact would only mitigate the leak after startup cost and contention already happened.
