## Summary

Keep open: current main still keys bundled MCP runtimes and in-flight creation by session id, so separate gateway sessions with the same workspace and MCP config can each start the full stdio MCP server set; current TTL/run-end cleanup only mitigates after duplicate children already exist.

## What This Changes

The issue reports duplicate persistent stdio MCP child processes being spawned per gateway session for the same MCP configuration, causing memory and CPU growth.

## Best Possible Solution

Maintain one live bundled MCP runtime per compatible workspace/config fingerprint with single-flight creation and per-session leases, then tear it down deterministically on config changes, gateway cleanup, and lease/idle expiry.

## Reproduction Assessment

Yes. The reporter supplied concrete process/log evidence, and current source gives a high-confidence static path: two different session ids with the same workspace/config miss each other's session-keyed cache entries and each catalog startup can spawn the same stdio MCP children.

## Solution Assessment

Yes. A workspace/config-fingerprint single-flight runtime with leases is the narrow maintainable fix; cron cleanup, a shorter TTL, or run-end cleanup only reduces the leak after duplicate children already exist.
