## Summary

Keep open: current main still keys bundled MCP runtime reuse and in-flight creation by `sessionId`, so two different sessions with the same workspace/config can each create and retain their own stdio MCP child until run-end cleanup or idle eviction applies.

## What This Changes

Report that gateway bundled MCP lazy loading can spawn duplicate stdio MCP child processes for the same effective server config, causing CPU and memory leaks.

## Best Possible Solution

Add a workspace/config-fingerprinted bundled MCP runtime ownership layer with single-flight creation and leases so identical stdio server configs share one child while existing cleanup hooks still reap it.

## Reproduction Assessment

Yes. The issue has concrete live process/log evidence, and current main has a static reproduction path: request bundled MCP runtimes for two different session ids with the same workspace/config and each path can create its own stdio transport.

## Solution Assessment

Yes. A config-keyed single-flight runtime/lease manager is the narrow maintainable fix; an external process sweeper or shorter TTL would only clean up duplicates after they already exist.

## Repair Work Prompt

Fix the bundled MCP duplicate stdio child leak reported in https://github.com/openclaw/openclaw/issues/75621. Current main keys `SessionMcpRuntimeManager` caches and in-flight creation by `sessionId`; two sessions with the same workspace/effective MCP config can each spawn a stdio child. Implement a narrow runtime ownership/single-flight layer keyed by a stable workspace plus effective MCP server config fingerprint so identical configs share one runtime/child with leases. Preserve per-session routing/auth semantics, do not expose secrets in keys/logs, and keep existing cleanup behavior for run-end cleanup, session reset/delete, config reload, gateway shutdown, and idle eviction. Add focused tests proving two different session ids with the same config reuse one runtime/stdio startup and that config changes still create/dispose distinct runtimes. Update CHANGELOG.md because this is a user-facing fix. Do not change unrelated plugin loader, provider, or channel behavior.
