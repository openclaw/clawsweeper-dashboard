## Summary

Keep open: current main still keys bundled MCP runtime ownership and in-flight creation by session id, so separate sessions with the same workspace/config can still own separate stdio MCP children until cleanup. The issue has concrete process/log evidence through v2026.5.7, and the related shared-runtime PR is still open rather than merged into main.

## What This Changes

The issue requests preventing duplicate persistent stdio MCP child processes from being spawned for the same gateway parent and MCP config across lazy bundle-MCP sessions.

## Best Possible Solution

Land a canonical runtime/lease fix that prevents identical workspace/config stdio MCP runtimes from spawning duplicate children while preserving explicit session isolation where it is required, with regression coverage for cross-session reuse and teardown.

## Reproduction Assessment

Yes. The issue and follow-up comments include concrete process tables and gateway logs through v2026.5.7, and current source inspection shows separate session ids with the same workspace/config still create separate MCP runtimes that can spawn stdio children.

## Solution Assessment

Unclear. Shared workspace/config runtime reuse is plausible, but maintainers need to decide whether the bug fix should change the default runtime ownership contract, remain opt-in, or use a narrower lease inside the existing session model.
