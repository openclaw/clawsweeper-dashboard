## Summary

Keep this open. The reporter supplied concrete v2026.4.29 process/log evidence, and current main still has a session-id keyed bundled MCP runtime manager, so two different gateway sessions with the same workspace/config can own separate stdio MCP runtimes until idle cleanup or explicit session cleanup runs.

## What This Changes

This issue asks the gateway to prevent lazy bundled stdio MCP startup from creating persistent duplicate child processes for the same effective server config across sessions.

## Best Possible Solution

Implement a bundled MCP runtime ownership layer that single-flights and reuses live stdio runtimes by an effective workspace plus MCP config/server fingerprint, with reference-counted session bindings and explicit disposal on config reload, session deletion/reset, gateway shutdown, and idle expiry. Avoid command-name process sweeping as the primary fix.

## Reproduction Assessment

Yes. The reporter supplied process-table and gateway-log evidence, and current main has a focused static reproduction: request bundle-MCP materialization for two different session ids with the same workspace/config and observe that the manager creates separate stdio-backed runtimes.

## Solution Assessment

Yes, a config-keyed single-flight/reuse layer with reference-counted leases is the narrowest maintainable direction. Reaping by process name or relying only on the 10-minute session idle TTL would reduce symptoms but would not prevent duplicate startup work or the per-session accumulation pattern.
