## Summary

Keep open: current main and v2026.5.2 still cache and single-flight bundled MCP runtimes by `sessionId`, so separate agent sessions with the same workspace/config can each create a persistent stdio MCP child before idle eviction cleans anything up.

## What This Changes

The issue requests preventing the Gateway from leaving multiple lazy-spawned stdio MCP children alive for the same bundled MCP workspace/config across agent sessions.

## Best Possible Solution

Maintain one live compatible bundled MCP runtime per workspace/config fingerprint or retire the prior session-key runtime before replacement, with ref-counted leases and deterministic cleanup on config, session, and Gateway teardown paths.

## Reproduction Assessment

Yes. The issue includes concrete process/log evidence, and current source shows a high-confidence static path where two different session ids with the same workspace/config can each create their own stdio MCP transport.

## Solution Assessment

Yes. A manager-level single-flight/lease fix is the narrow maintainable path; cron cleanup or process-table killing only mitigates after duplicate children already exist.
