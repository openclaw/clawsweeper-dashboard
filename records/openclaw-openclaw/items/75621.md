## Label Justifications

- `P1`: The issue describes a gateway CPU/memory leak that delays real Telegram/agent workflows by 100+ seconds and can make the runtime unhealthy.
- `impact:crash-loop`: Duplicate MCP child processes create process-level availability pressure through CPU saturation, event-loop liveness warnings, fetch timeouts, and memory growth.
- `impact:session-state`: The bug is rooted in session-scoped MCP runtime ownership causing identical MCP config state to be duplicated across sessions.

## Summary

Keep open: current main and the latest release still key bundled MCP runtimes and in-flight creation by session id, so separate sessions with the same workspace and MCP config can still spawn separate stdio child sets. The related shared-runtime PR remains open and conflicting, and the remaining fix needs maintainer judgment on the MCP runtime isolation contract.

## What This Changes

The issue asks OpenClaw to stop lazy-spawning duplicate stdio MCP child processes for identical gateway MCP configurations because they leak CPU and memory across sessions.

## Best Possible Solution

Land a canonical MCP runtime ownership fix that prevents identical workspace/config stdio runtimes from spawning duplicate children, while preserving explicit session isolation where maintainers require it, with regression coverage and live gateway proof.

## Reproduction Assessment

Yes, source-reproducible by current main: two distinct session ids with the same workspace/config use different runtime keys, and each runtime can connect the full configured stdio MCP server set. The issue comments add concrete process tables and gateway timing across multiple MCP servers and versions.

## Solution Assessment

Unclear. A shared in-flight/runtime key by workspace and config fingerprint is plausible, but maintainers still need to choose the isolation contract: default reuse, opt-in shared scope, or a narrower lease/disposal change inside the existing session model.
