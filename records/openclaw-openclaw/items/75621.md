## Summary

Keep open: current main still keys bundled MCP runtime ownership and in-flight creation by session id, so separate gateway sessions with the same workspace and MCP config can each create persistent stdio children. The issue discussion now includes two independent server implementations showing the same pattern through OpenClaw 2026.5.3-1.

## What This Changes

The issue requests preventing duplicate lazy-spawned stdio MCP child processes for identical gateway MCP configuration.

## Best Possible Solution

Implement one single-flight bundled MCP runtime owner per normalized workspace/config fingerprint with per-session leases and deterministic disposal on run cleanup, config reload, session cleanup, and gateway shutdown.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call the bundled MCP runtime manager for two different session ids with the same workspace/config before idle TTL expires, and each id can own a separate runtime that starts the configured stdio MCP servers.

## Solution Assessment

Yes. A workspace/config-keyed single-flight runtime and lease manager is the narrow maintainable fix; lowering TTLs or killing duplicate processes after the fact only mitigates the leak after extra children already exist.
