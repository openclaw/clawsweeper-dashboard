## Label Justifications

- `P1`: The issue describes a gateway CPU/memory leak that can delay real Telegram and agent workflows by 100+ seconds.
- `impact:crash-loop`: Duplicate MCP child processes create process-level availability pressure through CPU saturation, liveness warnings, fetch timeouts, and memory growth.

## Summary

Keep open: current main and the latest release still key bundled MCP runtimes and in-flight creation by session id, so identical workspace/config stdio MCP server sets can still be created separately for different sessions. Current main has cleanup paths that reduce stale runtime lifetime in one-shot flows, but the central duplicate-child/reuse policy is not implemented, and the related shared-runtime PR is open but conflicting.

## What This Changes

The issue reports duplicate stdio MCP server children being lazy-spawned per gateway agent session for the same MCP configuration, causing memory growth, CPU pressure, and message latency.

## Best Possible Solution

Land a canonical MCP runtime ownership fix that prevents duplicate child sets for identical workspace/config stdio servers while preserving any maintainer-required session isolation boundary, with focused regression coverage and real gateway process-count proof.

## Reproduction Assessment

Yes. Source-reproducible: two distinct session ids with the same workspace/config use different runtime keys, and each runtime can connect the full configured MCP server set; the issue comments add process-table proof across multiple MCP servers and releases.

## Solution Assessment

Unclear. A shared in-flight/runtime key by workspace and config fingerprint is plausible, but maintainers still need to choose the isolation contract: default reuse, opt-in shared scope, or a narrower cleanup/reuse change inside the existing session model.
