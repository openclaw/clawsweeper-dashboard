## Summary

Keep this open. Current main has several MCP cleanup fixes, but the active bundled-MCP runtime manager is still keyed by `sessionId`, so it can keep separate stdio children for the same workspace/config when a later fresh session materializes the same MCP server before the old runtime is retired.

## What This Changes

The issue reports that the gateway can lazy-spawn duplicate long-lived stdio MCP child processes for one configured MCP server/config path, causing memory and CPU leaks and severe Telegram latency.

## Best Possible Solution

Keep the issue open and fix the bundled-MCP lifecycle so identical stdio server configs do not accumulate under one gateway while preserving MCP session isolation. A conservative direction is to retire or supersede stale runtimes when the same session key/config rolls to a new session id, and only add cross-session config-level sharing if maintainers explicitly bless that contract and document the concurrency/state implications.

## Reproduction Assessment

Yes. The reporter provided a concrete live `ps`/gateway-log reproduction, and current main source shows a reproducible code path: create two fresh session ids that both materialize the same stdio MCP config before idle cleanup, and each session runtime can connect its own child.

## Solution Assessment

Unclear. The suggested `Map<configKey, Promise<Child>>` addresses spawn duplication, but it may be broader than the current session-scoped MCP contract; a safer fix may retire stale same-session-key runtimes first and add tests before considering global process sharing.
