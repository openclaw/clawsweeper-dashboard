## Summary

Keep this open. Current main still scopes bundled MCP runtime reuse and in-flight creation to `sessionId`, while each runtime owns its own MCP catalog and stdio transport, so two different gateway sessions with the same workspace and MCP config can still create separate child processes. The reporter’s follow-up gives concrete per-session process evidence, and the code path matches it closely enough for a focused fix PR candidate.

## What This Changes

Prevent the Gateway bundled-MCP lazy-load path from spawning and retaining duplicate stdio MCP child processes for the same effective workspace and MCP config across sessions.

## Best Possible Solution

Add a Gateway-owned bundled-MCP runtime pool keyed by normalized effective workspace plus MCP server/config fingerprint, with shared in-flight startup, session leases, deterministic disposal hooks, and regression coverage proving two different session ids with the same config reuse one live stdio runtime.

## Reproduction Assessment

Yes. The reporter provided live process-table and Gateway-log evidence, and current main has a static reproduction path: call bundle-MCP materialization for two different session ids with the same workspace/config and observe two session-keyed runtimes that each can spawn their own stdio child.

## Solution Assessment

No, the current session-scoped TTL cleanup is not the best solution because it can leave duplicate children alive until idle expiry and still repeats startup work. The maintainable direction is config-keyed single-flight reuse with explicit per-session leases, not process-name sweeping.
