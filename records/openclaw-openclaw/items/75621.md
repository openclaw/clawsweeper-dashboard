## Summary

Keep open: current main still owns bundled MCP runtimes by session id, so separate gateway sessions with the same workspace and MCP config can create separate persistent stdio children until idle/session cleanup runs.

## What This Changes

The issue reports duplicate persistent stdio MCP child processes spawned by gateway bundle-MCP lazy loading across separate sessions for the same configured server.

## Best Possible Solution

Make bundled MCP runtime ownership single-flight by workspace/config/server fingerprint with session leases, while preserving deterministic disposal on config reload, session reset/delete, one-shot run end, and gateway shutdown.

## Reproduction Assessment

Yes. The reporter supplied concrete process/log evidence, and current main has a source-level reproduction path: two getOrCreateSessionMcpRuntime calls with different sessionId values but the same workspace/config can each create a separate runtime and stdio child set.

## Solution Assessment

Yes. A workspace/config-keyed single-flight runtime lease manager is the narrow maintainable fix; TTL tuning, cron cleanup, or killing younger PIDs only mitigates after duplicate children already exist.
