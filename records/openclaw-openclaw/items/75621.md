## Summary

Keep open. The reporter provided concrete v2026.4.29 process and timing evidence, and current main still caches bundled MCP runtimes by session id, so two different sessions with the same workspace and MCP config can create independent runtime catalogs and stdio children; the newer idle/shutdown/session cleanup paths mitigate leaks but do not provide cross-session single-flight or reuse.

## What This Changes

Prevent the Gateway bundled MCP lazy-load path from spawning multiple persistent stdio MCP children for the same effective workspace and MCP config across agent sessions.

## Best Possible Solution

Add a bundled MCP runtime ownership layer that single-flights and reuses live stdio runtimes by effective workspace plus normalized MCP server/config fingerprint, with reference-counted session bindings and explicit disposal on config reload, session reset/delete, gateway shutdown, and idle expiry. Process-name sweeping should remain only an external emergency mitigation, not the product fix.

## Reproduction Assessment

Yes. A static reproduction on current main is to request bundled MCP materialization for two different session ids with the same workspace and cfg; the manager will create two runtimes because both the live cache and createInFlight map are keyed by sessionId.

## Solution Assessment

Yes, the reporter's suggested config-keyed in-flight map is the right direction, but it should live as a runtime ownership layer with leases/refcounts rather than a raw child-process map. That is narrower and more maintainable than relying on the idle TTL or command-name process cleanup.
