## Summary

Keep open: the reporter's process-table and log evidence matches current main's session-id keyed bundled-MCP runtime cache, so separate sessions with the same workspace and MCP config can still create separate persistent stdio children. Prior related cleanup covers teardown and idle expiry, but not config-keyed cross-session reuse.

## What This Changes

Report a gateway bundled-MCP lifecycle bug where lazy loading can spawn duplicate stdio MCP children for the same effective config across sessions.

## Best Possible Solution

Add a workspace/config-fingerprinted bundled MCP runtime ownership layer with single-flight creation, explicit leases, and disposal on config reload, session deletion/reset, gateway shutdown, and idle expiry.

## Reproduction Assessment

Yes. The issue has concrete live process and log evidence, and current main has a static reproduction path: create or materialize bundle-MCP runtimes for two different session ids with the same workspace/config and observe that each runtime can start its own stdio transport.

## Solution Assessment

Yes. The reporter's suggested in-flight map is the right core shape, but it should be implemented as a config-keyed runtime lease manager rather than a command-name process sweeper.
