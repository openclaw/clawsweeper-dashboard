## Summary

Keep open: current main still keys bundled MCP runtime reuse and in-flight creation by `sessionId`, so two gateway sessions with the same workspace and MCP config can each create a persistent stdio child. The existing cleanup paths address shutdown, reset/delete, run-end opt-in cleanup, and idle expiry, but they do not provide config-keyed cross-session single-flight reuse.

## What This Changes

The issue reports duplicate persistent stdio MCP child processes spawned by the gateway for the same MCP server config across lazy-loaded agent sessions.

## Best Possible Solution

Add a workspace/config-fingerprinted bundled MCP runtime ownership layer with single-flight creation, leases, and disposal on config reload, session deletion/reset, gateway shutdown, and idle expiry.

## Reproduction Assessment

Yes. The issue has concrete live process/log evidence, and current main has a static reproduction path: create or materialize bundle-MCP runtimes for two different session ids with the same workspace/config and observe that each runtime can start its own stdio transport.

## Solution Assessment

Yes. The reporter's in-flight map is the right core shape, but the maintainable fix should be a config-keyed runtime lease manager rather than an external process sweeper or per-command special case.
