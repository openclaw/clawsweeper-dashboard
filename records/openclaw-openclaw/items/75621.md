## Summary

Keep open: current main still keys bundled MCP runtime ownership and in-flight creation by session id, so a later gateway session using the same workspace/config can spawn a second long-lived stdio child while the earlier one remains cached. The latest release tag contains the same session-id keyed manager, and the related closed reports do not supersede this narrower cross-session duplicate-child path.

## What This Changes

Prevent gateway bundled MCP lazy loading from spawning multiple persistent stdio MCP children for the same configured server/config.

## Best Possible Solution

Make bundled MCP runtime ownership single-flight and cleanup-aware for a stable sessionKey/workspace/config lease, with deterministic disposal or reuse before a replacement stdio child can spawn.

## Reproduction Assessment

Yes. The reporter supplied live process/log evidence, and source inspection shows a high-confidence path where different session ids with the same workspace/config use distinct runtime and in-flight cache keys.

## Solution Assessment

Yes. A stable keyed lease manager with regression coverage is the narrow maintainable fix; a cron/process-table cleanup only mitigates after duplicate children already exist.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75621. Current gateway bundled MCP runtime caching is keyed by sessionId, so a new session id for the same sessionKey/workspace/MCP config can lazy-spawn another persistent stdio MCP child before the old one is retired. Add a focused regression test in src/agents/pi-bundle-mcp-runtime.test.ts that creates or requests two runtimes for different session ids but the same sessionKey/workspace/config and proves only one live runtime/stdio transport remains, either by reusing the existing compatible runtime or retiring the previous one before creating a replacement. Implement the fix in src/agents/pi-bundle-mcp-runtime.ts and any necessary type/call-site adjustments, preserving cleanup on session reset/delete, gateway shutdown, config changes, one-shot run-end cleanup, and idle TTL. Do not share runtime state across unrelated session keys/workspaces unless the existing contracts prove it is safe. Update docs/changelog only if the user-facing MCP lifetime behavior changes.
