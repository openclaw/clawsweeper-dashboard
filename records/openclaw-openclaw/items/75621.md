## Summary

Keep open: current `main` and the latest release still key bundled MCP runtime ownership and in-flight creation by `sessionId`, so the duplicate workspace/config fan-out is not implemented away. The clean open candidate PR is useful, but it adds an opt-in shared runtime/config/security boundary and needs maintainer judgment rather than issue cleanup.

## What This Changes

The issue reports gateway-owned stdio MCP children being spawned more than once for identical MCP configs, causing memory growth, CPU pressure, and delayed channel replies.

## Best Possible Solution

Land one maintainer-approved MCP runtime ownership path that prevents duplicate identical stdio child sets while preserving default isolation unless maintainers intentionally accept an opt-in shared mode.

## Reproduction Assessment

Yes at source level, but not as a live process repro in this read-only review. Current `main` still keys MCP runtimes and in-flight creation by session id, while each runtime owns its own catalog and MCP server sessions; related issue proof also observed one runtime/tools-list per session for identical workspace/config.

## Solution Assessment

Unclear as a final product choice. Workspace/config-keyed sharing is the right implementation layer for duplicate child fan-out, but adding shared mode or changing defaults crosses session-isolation and security boundaries that maintainers must accept explicitly.
