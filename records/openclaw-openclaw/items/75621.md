## Summary

Keep open: current main and v2026.5.3 still use session-id keyed bundled MCP runtime ownership, and the issue now has independent 2026.5.3-1 evidence with another stdio MCP server. This remains a concrete gateway resource leak rather than the same per-message staging problem tracked by #75437.

## What This Changes

The issue requests de-duplicating or promptly reaping bundled MCP stdio children so repeated gateway sessions with the same MCP config do not accumulate duplicate child processes.

## Best Possible Solution

Use a stable workspace/config runtime key with single-flight creation and reference-counted session leases, while keeping explicit config reload, session delete/reset, one-shot run cleanup, and gateway shutdown disposal deterministic.

## Reproduction Assessment

Yes. The issue comments provide concrete process evidence, and current source shows a high-confidence path where different session ids with the same workspace/config can each create and retain a stdio-backed bundle-MCP runtime.

## Solution Assessment

Yes. A single-flight runtime manager keyed by the effective workspace/config fingerprint, with leases for active sessions and regression tests for cleanup paths, is narrower than cron-based process cleanup or lowering the idle TTL.

Update `src/agents/pi-bundle-mcp-runtime.ts` and related types/tests so live and in-flight runtime creation is single-flight by a stable effective runtime key such as workspace plus MCP config fingerprint, with session leases/aliases for active session ids and session keys.
