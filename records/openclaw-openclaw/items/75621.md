## Summary

Keep open: current main still uses session-id-scoped bundled MCP runtime ownership and in-flight creation, so separate agent sessions with the same workspace and MCP config can still create independent stdio children. The linked shared-runtime PR remains open and unmerged, and the remaining fix needs maintainer judgment on the MCP runtime isolation contract.

## What This Changes

Request to prevent the gateway bundle-MCP lazy path from accumulating duplicate stdio server processes for the same MCP config across agent sessions.

## Best Possible Solution

Land a canonical MCP runtime ownership fix that prevents identical workspace/config stdio runtimes from spawning duplicate children while preserving explicit session isolation where required, with regression coverage for reuse and teardown.

## Reproduction Assessment

Yes, by source inspection rather than a live run: two distinct session ids with the same workspace/config take separate cache keys and can each connect every configured stdio MCP server. The issue comments add concrete process tables and gateway logs across multiple MCP servers and versions.

## Solution Assessment

Unclear. A shared in-flight/runtime key by workspace and config fingerprint is plausible, but maintainers need to decide whether the fix should be default behavior, an opt-in runtime scope, or a narrower lease/disposal change inside the existing session model.
