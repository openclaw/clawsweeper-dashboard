## Summary

Keep this open. The reporter supplied concrete v2026.4.29 process and gateway-log evidence, and current main still keys bundled MCP runtime reuse and in-flight creation by session id, so two different gateway sessions with the same workspace/config can create separate stdio MCP children until idle cleanup, session cleanup, reload, or shutdown disposes them.

## What This Changes

The issue asks OpenClaw to prevent duplicate long-lived stdio MCP child processes for the same effective gateway MCP server configuration across lazy-loaded agent sessions.

## Best Possible Solution

Implement a bundled MCP runtime ownership layer that single-flights and reuses live stdio runtimes by effective workspace plus normalized MCP server/config fingerprint, with reference-counted session bindings and explicit disposal on config reload, session delete/reset, gateway shutdown, and idle expiry. Avoid command-name process sweeping as the primary fix.

## Reproduction Assessment

Yes. The reporter supplied process-table and gateway-log evidence, and current main has a focused static reproduction path: request bundle-MCP materialization for two different session ids with the same workspace/config and observe that the manager creates separate stdio-backed runtimes.

## Solution Assessment

Yes, the suggested config-keyed single-flight/reuse layer is the narrowest maintainable direction if it preserves lifecycle isolation through leases. Reaping by process name or relying only on the 10-minute idle TTL would reduce symptoms but would not prevent duplicate startup work across sessions.

Implement a narrow runtime ownership layer in the bundled MCP runtime manager that single-flights and reuses live stdio runtimes by effective workspace plus normalized MCP config/server fingerprint, with per-session bindings/leases and explicit disposal on config reload, session delete/reset, gateway shutdown, and idle expiry.
