## Summary

Keep open: current main and v2026.5.3 still key bundled MCP runtime ownership and in-flight creation by session id, so separate gateway sessions with the same workspace/config can create separate persistent stdio children. The reporter and a second commenter both provided process evidence matching that source path, including the newer 2026.5.3-1 report with a different MCP server.

## What This Changes

The issue reports duplicate lazy-spawned stdio MCP child processes per gateway session for the same configured MCP server, causing memory and CPU growth.

## Best Possible Solution

Maintain one live bundled MCP runtime per compatible workspace/config fingerprint with single-flight creation, session leases, and deterministic teardown on config reload, session cleanup, and gateway shutdown.

## Reproduction Assessment

Yes. I did not spawn live MCP processes in this read-only review, but the source path is high-confidence: two different session ids with the same workspace/config miss each other's session-id keyed cache and each catalog build can connect the same stdio servers.

## Solution Assessment

Yes. A workspace/config-fingerprinted single-flight runtime with leases is the narrow maintainable fix; lowering `mcp.sessionIdleTtlMs` only bounds cleanup after duplicate children already exist.
