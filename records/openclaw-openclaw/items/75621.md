## Summary

Keep open: current main still keys bundled MCP runtime ownership and in-flight creation by session id, so separate gateway sessions with the same workspace and MCP config can create separate persistent stdio children before idle eviction runs.

## What This Changes

The issue reports duplicate persistent stdio MCP child processes for the same gateway parent and config when bundled MCP runtimes are lazy-spawned across separate sessions.

## Best Possible Solution

Add a workspace/config-fingerprinted bundled MCP runtime ownership layer with single-flight startup and explicit leases, or an equivalent design that prevents duplicate stdio children without weakening session isolation.

## Reproduction Assessment

Yes. The report includes concrete process and timing evidence, and current main has a static reproduction path: request bundle-MCP runtimes for two different session ids with the same workspace/config and each id can own a separate stdio transport.

## Solution Assessment

Yes. A config-keyed single-flight runtime/lease manager is the narrow maintainable direction; a process-table sweeper would only mitigate the leak after duplicate children already exist.
