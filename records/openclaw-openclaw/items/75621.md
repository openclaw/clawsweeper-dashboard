## Summary

Keep this open: current main has important cleanup mitigations, but it still intentionally caches bundled MCP runtimes by session rather than by identical stdio config, so the same MCP server can still have multiple live children across sessions until explicit retirement or idle TTL. The linked shared-runtime PR is an unmerged, conflicting candidate and needs maintainer product/security judgment.

## What This Changes

The issue requests fixing duplicate stdio MCP child processes spawned under the gateway for the same MCP configuration, causing memory and CPU growth.

## Best Possible Solution

Keep the session-scoped cleanup improvements and make a maintainer decision on a bounded same-config MCP pooling/shared-runtime design that avoids child accumulation without silently weakening session isolation.

## Reproduction Assessment

Yes at source level for the remaining behavior: current main keys bundled MCP runtimes by session id, so separate sessions with the same workspace and MCP config can each own stdio MCP children until cleanup or idle TTL applies. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, not as an automated cleanup close or narrow fix PR. The likely fix direction is either the linked opt-in shared runtime or another pooling design, both of which need maintainer acceptance of the MCP session-isolation tradeoff.
