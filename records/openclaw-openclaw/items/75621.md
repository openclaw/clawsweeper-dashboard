## Summary

Keep open: current main still keys bundled MCP runtime ownership and in-flight creation by session id, so separate gateway sessions with the same workspace and MCP config can each create a persistent stdio child before idle cleanup runs.

## What This Changes

The issue reports duplicate persistent stdio MCP child processes for the same gateway parent and MCP config across separate lazy-loaded agent sessions.

## Best Possible Solution

Add a workspace/config-fingerprinted single-flight ownership or lease layer for bundled MCP runtimes so identical stdio configs do not duplicate children across sessions, while config changes, session reset, reload, shutdown, and idle cleanup still dispose safely.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: request bundled MCP runtimes for two different session ids with the same workspace/config, and the manager can create two cached runtimes because existing and in-flight state are keyed by session id.

## Solution Assessment

Yes. A manager-level config-keyed single-flight/lease design is the narrow maintainable fix; a cron or process-table sweeper would only clean up after duplicate children already exist.
