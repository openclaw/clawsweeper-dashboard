## Summary

Keep open: current main and v2026.5.2 still cache bundled MCP runtimes and in-flight creation by session id, so separate gateway sessions with the same workspace and MCP config can still create separate persistent stdio children before idle cleanup.

## What This Changes

The issue asks the Gateway to prevent duplicate persistent stdio MCP children for the same configured MCP server/config across separate agent sessions.

## Best Possible Solution

Maintain one live bundled MCP runtime per compatible workspace/config fingerprint with single-flight creation, session leases, and deterministic teardown on config reload, session reset/delete, gateway shutdown, and idle expiration.

## Reproduction Assessment

Yes. The reporter supplied concrete process/log evidence, and current source shows two different session ids with the same workspace/config can enter separate runtime-manager keys and each runtime can spawn a stdio child.

## Solution Assessment

Yes. A workspace/config-keyed single-flight runtime with leases is the narrow maintainable fix; process-table cron cleanup only mitigates after duplicate children already exist.

## Repair Work Prompt

Fix openclaw/openclaw#75621. Bundled MCP stdio runtimes can spawn duplicate persistent children when two Gateway agent sessions with different sessionIds use the same workspace and MCP config. In src/agents/pi-bundle-mcp-runtime.ts, introduce a deterministic compatible-runtime key, likely workspaceDir plus MCP config fingerprint and any required compatibility fields, and use it for single-flight creation/lookup so compatible sessions share one live runtime. Preserve existing same-session config-change recreation, one-shot cleanup, session reset/delete, config reload, idle sweep, and gateway shutdown semantics. Do not add a config option and do not implement process-table cleanup. Add regression coverage showing two different sessionIds/sessionKeys with identical workspace/config reuse the same runtime/in-flight promise, and that disposal occurs only when the shared runtime is no longer leased or when config compatibility changes. Update docs and CHANGELOG.md if the lifecycle wording changes.
