## Summary

Keep this open: the report is a concrete current v2026.4.29 failure, and current main still keys bundled MCP runtime reuse and in-flight creation by session id rather than by the stdio server/config tuple described in the report. Existing cleanup/reaping fixes cover stale or disposed runtimes, but they do not prevent two active sessions from owning separate stdio children for the same server config.

## What This Changes

The issue requests preventing the gateway from lazy-spawning persistent duplicate stdio MCP child processes for the same configured MCP server/config and reusing or single-flighting the existing healthy child instead.

## Best Possible Solution

Add an owned gateway/session MCP runtime reuse layer that single-flights and reuses live stdio MCP runtimes for the same effective workspace plus MCP config fingerprint, with reference-counted session bindings and disposal only after all users release or the idle TTL expires. Avoid command-name process sweeping; the fix should live in the bundled MCP runtime manager and transport lifecycle tests.

## Reproduction Assessment

Yes. The issue gives process-table and gateway-log evidence on v2026.4.29, and current main statically shows a focused repro path: create two session MCP runtimes with different session ids but identical workspace/config and observe that the manager permits two independent stdio transports.

## Solution Assessment

Unclear until implementation, but the suggested single-flight map is directionally right if it is keyed by the effective config/workspace and paired with safe refcounted disposal. Reusing by session id alone is not sufficient for the reported duplicate child pattern.
