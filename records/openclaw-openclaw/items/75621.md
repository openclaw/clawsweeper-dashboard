## Summary

Keep open: current main and v2026.5.7 still use session-id-keyed bundled MCP runtime ownership, so separate agent sessions with the same workspace and MCP config can each create persistent stdio children. The related shared-runtime PR is open and opt-in, so it is relevant follow-up work but not a reason to close this issue yet.

## What This Changes

Prevent the gateway from accumulating duplicate stdio MCP child processes for identical MCP server configs across agent sessions.

## Best Possible Solution

Review and either land or revise the shared-runtime work so compatible workspace/config fingerprints reuse one live bundled MCP runtime with deterministic lease cleanup, while preserving the intended isolation boundary.

## Reproduction Assessment

Yes. The reporter supplied concrete process/log evidence, and current source shows that two distinct session ids with the same workspace/config can each create a runtime whose catalog connection spawns stdio MCP children.

## Solution Assessment

Unclear until maintainer review of the isolation policy. A workspace/config-keyed single-flight runtime with leases is the narrow technical fix, but the open opt-in config approach needs a product decision before it can be treated as the final answer.
