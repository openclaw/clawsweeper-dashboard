## Summary

Keep open: current main and the latest release still use session-id-scoped bundled MCP runtime ownership, so separate gateway sessions with the same workspace/config can still own separate persistent stdio children. The open related PR at https://github.com/openclaw/openclaw/pull/79882 is a plausible implementation path, but it is unmerged, conflicting, and opt-in as written, so it does not close the default leak report.

## What This Changes

The issue asks OpenClaw to prevent duplicate persistent stdio MCP children for the same gateway parent and MCP config from accumulating across agent sessions.

## Best Possible Solution

Land a canonical runtime/lease fix that prevents identical workspace/config stdio MCP runtimes from spawning duplicate children while preserving explicit session isolation, with regression coverage and updated MCP docs.

## Reproduction Assessment

Yes. The reporters supplied concrete process/log evidence across graphiti, engram, and v2026.5.7, and source inspection shows that different session ids with the same workspace/config still get independent runtimes that can each spawn stdio children.

## Solution Assessment

No current-main fix is present. The shared-runtime PR is a plausible direction, but maintainers need to decide whether an opt-in config mode is enough or whether the default runtime ownership/teardown contract should change for this bug.
