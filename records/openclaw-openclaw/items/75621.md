## Summary

Keep open: current main still caches bundled MCP runtimes and in-flight creation by session id, so two gateway sessions with the same workspace/config can own separate persistent stdio children until idle eviction; the reporter's per-session process evidence remains actionable.

## What This Changes

The issue reports that gateway lazy loading can spawn multiple long-lived stdio MCP child processes for the same configured server, doubling CPU and memory until external cleanup or gateway restart.

## Best Possible Solution

Add a workspace/config-fingerprinted bundled MCP runtime ownership layer with single-flight creation, explicit leases, and the existing lifecycle cleanup hooks preserved.

## Reproduction Assessment

Yes. The issue has concrete live process/log evidence, and current main has a static reproduction path: request bundle-MCP runtimes for two different session ids with the same workspace/config and observe that each can create its own stdio transport.

## Solution Assessment

Yes. A config-keyed single-flight runtime/lease manager is the narrow maintainable fix; a process-table sweeper would only hide the leak after duplicate children already exist.
