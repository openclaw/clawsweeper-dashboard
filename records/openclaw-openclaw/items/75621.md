## Summary

Keep open: current main and the latest release still cache bundled MCP runtimes by session id, so separate sessions using the same workspace and MCP config can create separate persistent stdio children before idle eviction runs. The later commenter’s 2026.5.3-1 engram evidence also supports that this is not graphiti-specific.

## What This Changes

The issue asks OpenClaw to prevent gateway bundle-MCP lazy loading from accumulating duplicate stdio MCP child processes for the same effective server config across sessions.

## Best Possible Solution

Maintain one compatible live bundled MCP runtime per effective workspace/config fingerprint with single-flight creation, reference-counted session leases, and deterministic disposal on run, session, config reload, and gateway shutdown paths.

## Reproduction Assessment

Yes, by source inspection: request bundled MCP runtimes for two different session ids with the same workspace/config and the manager can create two distinct runtimes, each able to spawn stdio children. The reporter and follow-up comments provide matching process-level evidence on real gateways.

## Solution Assessment

Yes. A workspace/config-fingerprint single-flight runtime with leases is the narrow maintainable fix; reducing mcp.sessionIdleTtlMs or killing duplicate processes externally only bounds the leak after extra children already exist.

## Repair Work Prompt

Fix the bundle MCP duplicate stdio child leak described in openclaw/openclaw#75621. Current main caches SessionMcpRuntime objects and createInFlight promises by sessionId, so multiple sessions with the same effective workspace and mcp.servers config can each spawn a full set of stdio MCP children. Implement a narrow fix in the bundle MCP runtime manager: key compatible live runtimes and in-flight creation by an effective workspace/config fingerprint, add session leases/reference tracking so sessions can share the runtime safely, and keep deterministic teardown on session cleanup, one-shot run cleanup, config reload, and gateway shutdown. Add regression coverage proving two different session ids with the same workspace/config reuse one runtime/in-flight promise, config/workspace changes create a new runtime and dispose the old one when no longer leased, active leases prevent idle eviction, and catalog materialization still does not eagerly duplicate stdio transports. Update MCP lifecycle docs/changelog if the user-facing lifecycle changes. Do not change the MCP config schema, plugin API, provider behavior, or unrelated runtime-dependency staging paths.
