## Summary

Keep this open. The report is concrete and current main still keys bundled MCP runtime reuse and in-flight creation by `sessionId`, so two gateway sessions with the same workspace and MCP config can own separate stdio transports. Existing shutdown, reset, idle-TTL, and one-shot cleanup paths reduce stale-process leaks, but they do not prevent the duplicate same-config child process pattern described here.

## What This Changes

The issue requests that the gateway avoid accumulating duplicate stdio MCP child processes for the same effective MCP server configuration across lazy-loaded agent sessions.

## Best Possible Solution

Add a bundled MCP runtime ownership layer that single-flights and reuses a live stdio MCP runtime for the same effective workspace plus MCP config fingerprint, with leases/refcounts for session bindings and disposal only after all users release or the idle TTL expires. Avoid process-table sweeping; the gateway should own this lifecycle through the bundled MCP runtime manager and tests.

## Reproduction Assessment

Yes. The issue gives process-table and gateway-log evidence on v2026.4.29, and current main has a focused static repro path: request bundled MCP runtimes for two different session ids with the same workspace/config and observe that the manager can create separate stdio transports.

## Solution Assessment

Unclear until patched, but the suggested single-flight map is the right direction if keyed by the effective workspace/config and paired with refcounted disposal. Reusing by session id alone is not enough, and external cron/process cleanup is only a mitigation.
