## Summary

Keep open: the fix targets a real session-history bug, but the proposed whole-string tail slice can discard the pinned compaction summary when the post-compaction tail alone exceeds the cap, causing a different reseed context loss.

## What This Changes

The PR changes CLI session reseed prompt truncation to keep the recent rendered transcript tail, moves the truncation marker to the front, and adds a regression test.

## Best Possible Solution

Keep the recent-turn preservation, but make truncation structure-aware so a present compaction summary remains pinned while only the post-summary transcript tail is capped.

## Reproduction Assessment

Yes. Current main renders messages oldest-to-newest and then keeps the prefix, so an over-cap fixture with final marker turns will drop the markers; the PR's added test describes that source-level reproduction path.

## Solution Assessment

No. Tail-slicing is the right direction, but slicing the entire rendered string is not the best implementation because it can remove the compaction summary that the reseed loader deliberately places first.
