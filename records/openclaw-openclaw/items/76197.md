## Summary

Keep open. The PR targets a real `/status` auth-label mismatch, but the diff currently drops the `await` from async harness resolution, so the Codex runtime case still resolves auth against `openai` and the new regression path would fail.

## What This Changes

The PR changes `/status` to display the `openai-codex` OAuth profile for native Codex runtime `openai/gpt-*` sessions, with tests and a changelog entry.

## Best Possible Solution

Land a small update on this PR that awaits the harness resolver before deriving the auth-label provider, keeping the regression tests and changelog entry.

## Reproduction Assessment

Yes. The new Codex-harness test path calls `buildStatusText` without `resolvedHarness`, which hits the unawaited async resolver and leaves auth resolution on `openai`; I verified that from the patch and current source rather than running tests.

## Solution Assessment

No. The product direction is narrow and appropriate, but the current implementation must await `resolveStatusHarnessId` before using `effectiveHarness` for auth-provider selection.
