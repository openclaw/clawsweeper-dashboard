## Summary

Keep open: current main still has the gateway live transcript API but the TUI has no subscribe/unsubscribe client contract or `session.message` dispatch path, so the reported live-tail gap remains source-reproducible. This issue is also tied to open implementation work, especially https://github.com/openclaw/openclaw/pull/73402 for the canonical report and an external closing PR for this exact report, so it should not be closed before a fix lands.

## What This Changes

The issue reports that `openclaw tui --session ... --deliver` loads Telegram session history but does not live-tail inbound messages or agent replies.

## Best Possible Solution

Review, update, and land or replace https://github.com/openclaw/openclaw/pull/73402 as the scoped same-session TUI transcript subscription fix; keep broader bidirectional/mirror behavior on https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes. Source inspection on current main shows the TUI loads history and handles gateway chat events, but has no `sessions.messages.subscribe` client method or `session.message` dispatch/handler; the reporter's Ctrl+T/manual-refresh observation matches that path.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to wire TUI session mode to the existing `sessions.messages.subscribe` / `session.message` gateway contract, while the proposed bidirectional mirror mode is separate product work.
