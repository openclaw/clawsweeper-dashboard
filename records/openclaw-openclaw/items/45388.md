## Label Justifications

- `P2`: The issue is a real TUI live transcript bug with limited surface area and an open scoped fix path, not a core runtime emergency.
- `impact:message-loss`: Inbound channel messages and agent responses are not displayed in the active TUI until manual refresh or relaunch.
- `impact:session-state`: The bug is caused by the TUI session view not subscribing to the live transcript state for the attached session.

## Summary

Keep open: current main still has the gateway live transcript stream but the bundled TUI does not subscribe to it or dispatch `session.message` events. The remaining fix is already represented by the open maintainer PR at https://github.com/openclaw/openclaw/pull/73402 and an external closing PR, so this issue should stay open until a fix lands.

## What This Changes

Make `openclaw tui --session ... --deliver` live-tail the attached session so inbound Telegram messages and agent responses appear without manual refresh or relaunch.

## Best Possible Solution

Land or replace the scoped TUI subscription fix that consumes the existing `sessions.messages.subscribe` and `session.message` gateway contract, while keeping broader bidirectional channel mirroring on https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes. Source inspection on current main shows the TUI loads session history and handles chat/agent websocket events, but has no session-message subscription client method or `session.message` event dispatch path.

## Solution Assessment

Yes for the central bug. The maintainable fix is to wire TUI session mode to the existing scoped gateway transcript subscription contract; the requested bidirectional mirror mode is separate product work.
