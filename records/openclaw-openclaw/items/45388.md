## Label Justifications

- `P2`: This is a real user-facing TUI live transcript bug with bounded TUI/gateway blast radius and active focused fix paths.
- `impact:message-loss`: Inbound Telegram messages and agent replies are missing from the active TUI view until manual refresh or relaunch.
- `impact:session-state`: The visible TUI transcript can drift from the stored gateway session transcript even though history later contains the missing turns.

## Summary

Keep open: current main and the latest release still have the gateway live transcript subscription contract, but the bundled TUI does not subscribe to it or route `session.message` events, and linked fix work remains open.

## What This Changes

The issue asks for `openclaw tui --session ... --deliver` to live-tail inbound Telegram messages and agent replies instead of requiring manual refresh or relaunch.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73402 with a scoped TUI fix that consumes `sessions.messages.subscribe` and `session.message`, filters to the active session, unsubscribes on session changes/stop, defers history reload during active local streams, and keeps bidirectional mirror mode on https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes, source-level. Current main loads TUI history and routes only `chat`, `chat.side_result`, and `agent` events, while the gateway exposes subscribed `session.message` transcript events that the TUI never subscribes to or dispatches.

## Solution Assessment

Yes for the central bug. The narrow maintainable solution is to wire the TUI to the existing gateway transcript subscription contract, not to add polling, widen generic chat fanout, or bundle bidirectional mirror behavior into this repair.
