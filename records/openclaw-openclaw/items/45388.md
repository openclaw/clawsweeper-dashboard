## Summary

Keep open. Current main still has the Gateway transcript subscription contract, but the bundled TUI does not subscribe to or route `session.message`, so the live-tail bug remains; the issue is also already paired with open implementation work that should be reviewed rather than pre-closed.

## What This Changes

The issue requests that `openclaw tui --session ... --deliver` live-tail inbound Telegram messages and agent replies instead of only showing updates after manual refresh or relaunch.

## Best Possible Solution

Land a focused TUI fix, likely via #73402 or an equivalent replacement, that subscribes to the active session transcript stream, unsubscribes on session switch/shutdown, routes `session.message` only for the active session, and includes duplicate-prevention/history-overlap regression coverage.

## Reproduction Assessment

Yes. The issue gives a concrete command and current source provides a high-confidence code-path reproduction: Gateway transcript events exist, but the TUI neither subscribes to nor handles them.

## Solution Assessment

Yes for the implementation direction. The narrow maintainable solution is to use the existing `sessions.messages.subscribe` / `session.message` transcript contract rather than widening run-scoped `chat` event fanout for external channel traffic.
