## Summary

Keep open: current main still does not wire TUI session mode to the gateway's live transcript event stream, so inbound Telegram/user turns can remain invisible until history is reloaded. The prior target PR was closed unmerged, and the remaining open cross-repo PR is an implementation candidate rather than proof this is fixed.

## What This Changes

The issue asks for `openclaw tui --session ... --deliver` to live-tail inbound Telegram session messages and agent responses instead of requiring manual refresh or relaunch.

## Best Possible Solution

Wire TUI `--session` mode to the existing `sessions.messages.subscribe`/`session.message` transcript stream, update session subscriptions when the selected session changes, render incoming user turns without duplicating history, and leave bidirectional channel mirroring as a separate product discussion.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a TUI connected to a session cannot receive live inbound transcript messages because it never subscribes to or handles `session.message`. I did not execute the live Telegram scenario, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes for the narrow bug: the maintainable fix is to reuse the existing gateway session-message subscription API instead of adding a bespoke chat `user-message` event. The bidirectional mirror idea should be handled separately if maintainers want that mode.
