## Label Justifications

- `P2`: This is a normal-priority TUI live transcript regression with manual refresh or relaunch as a workaround and limited blast radius.
- `impact:message-loss`: Messages that exist in the durable session transcript or another channel can be absent from the live TUI view until refresh or relaunch.
- `impact:session-state`: The TUI's visible active session can drift from the Gateway-backed session transcript while the session remains open.

## Summary

Keep open: current main and the latest release still do not route live Gateway `session.message` transcript updates into the TUI, so the reported selected-session live-tail bug remains. The narrower canonical tracker is https://github.com/openclaw/openclaw/issues/38829, while the linked open fork PR is conflicting and follows an older broad chat-event approach.

## What This Changes

Make `openclaw tui --session ... --deliver` live-tail inbound selected-session messages and agent responses without relaunch or manual refresh.

## Best Possible Solution

Land a focused same-repo fix that subscribes the TUI to active-session `sessions.messages.subscribe` updates, routes `session.message` safely, prevents duplicate reload/render behavior, and leaves broader bidirectional mirroring to https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes, at source level: Gateway emits displayable transcript appends as `session.message`, while the current TUI does not subscribe to or dispatch that event. I did not run a live Telegram/TUI reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug: the clean fix is to consume the existing scoped `sessions.messages.subscribe` and `session.message` contract in the TUI. The broad chat-event fanout and bidirectional mirror ideas are not the best narrow bug fix.
