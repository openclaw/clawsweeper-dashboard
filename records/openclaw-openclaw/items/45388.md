## Label Justifications

- `P2`: This is a normal-priority user-visible TUI live-update bug with a manual refresh/relaunch workaround and a known implementation path.
- `impact:session-state`: The active TUI session view can drift behind the Gateway-backed transcript until manual refresh or relaunch.
- `impact:message-loss`: Inbound channel messages and assistant updates are effectively missing from the live TUI view.

## Summary

Keep open: current main and v2026.6.6 still expose Gateway live transcript events, but the bundled TUI does not subscribe to or handle `session.message` updates. The narrow fix is already represented by open implementation candidates, especially https://github.com/openclaw/openclaw/pull/73402, so this issue remains necessary until that path lands or is replaced.

## What This Changes

The issue requests real-time TUI updates for an explicitly opened Telegram-backed session instead of requiring manual refresh or relaunch.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73402 with a bounded TUI fix that subscribes after connect and session switches, unsubscribes stale sessions, filters to the active session, and refreshes history without clobbering active local streams.

## Reproduction Assessment

Yes, at source level: current main exposes and tests live `session.message` delivery, while the TUI has no subscription method, no dispatch branch, and no handler for those events. I did not run a live Gateway/TUI reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug: consuming the existing per-session Gateway transcript stream in the TUI is the maintainable owner-boundary fix. Bidirectional channel mirroring is separate product work tracked at https://github.com/openclaw/openclaw/issues/40678.
