## Summary

Keep open: current main and the latest release still do not wire TUI session mode to `sessions.messages.subscribe` / `session.message`, while the gateway already exposes and tests the live transcript stream. The remaining work is active in linked implementation candidates, so this issue should stay open until a scoped TUI subscription fix lands.

## What This Changes

Make `openclaw tui --session ... --deliver` live-tail the active session transcript so Telegram messages and agent responses appear without manual refresh or relaunch.

## Best Possible Solution

Implement a scoped TUI active-session transcript subscription lifecycle over the existing gateway contract, with subscribe/unsubscribe on connect and session switch, `session.message` handling, focused regression tests, and no broad chat-event fanout.

## Reproduction Assessment

Yes, source-reproducible. Current main loads `chat.history` and routes only `chat`, `chat.side_result`, and `agent`, while the gateway emits `session.message` only to subscribed connections and the TUI never subscribes.

## Solution Assessment

Yes for the narrow bug. The maintainable fix is to wire TUI session mode to the existing scoped `sessions.messages.subscribe` / `session.message` gateway contract; broad chat-event fanout or bidirectional mirroring would be a separate product choice.
