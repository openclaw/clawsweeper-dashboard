## Summary

Keep open: current main still has the TUI session history refresh path and Gateway transcript subscription machinery, but the bundled TUI does not subscribe to or handle live `session.message` transcript updates. The remaining work is already represented by open implementation candidates, especially https://github.com/openclaw/openclaw/pull/73402, so this issue should stay open until that path lands or is replaced.

## What This Changes

The issue reports that `openclaw tui --session "telegram:direct:<id>" --deliver` loads history but does not live-tail inbound Telegram messages or echoed agent replies until manual refresh or relaunch.

## Best Possible Solution

Land or replace the narrow TUI active-session transcript subscription fix, preserving session filtering and leaving broader cross-channel mirroring to the related product/security discussion.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main loads `--session` history and supports manual refresh, while the bundled TUI lacks a `session.message` subscription and handler for active transcript updates. I did not run a live Telegram/TUI session in this read-only review.

## Solution Assessment

Yes, the best narrow fix is to subscribe the TUI to active-session transcript updates and dedupe/filter by session; the separate bidirectional mirroring request should remain outside this bug fix.
