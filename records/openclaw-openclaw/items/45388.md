## Summary

Keep open. Current main still exposes Gateway transcript subscription events, but the bundled TUI does not subscribe to or route `session.message`; the issue is also paired with open implementation PRs that should be reviewed rather than pre-closed.

## What This Changes

Make `openclaw tui --session ... --deliver` live-tail the active Telegram-backed session, including inbound messages and agent replies, instead of requiring manual refresh or relaunch.

## Best Possible Solution

Review and land #73402 or an equivalent replacement that subscribes the TUI to active-session transcript events, filters by active session, unsubscribes on switch/shutdown, and prevents history-overlap duplicates; then close or supersede #45417 as appropriate.

## Reproduction Assessment

Yes. The issue gives a concrete command, and current source provides a high-confidence reproduction path: Gateway can emit `session.message`, but the TUI neither subscribes to nor routes that event.

## Solution Assessment

Yes for the implementation direction. The maintainable fix is to use the existing `sessions.messages.subscribe` / `session.message` transcript contract rather than broadening hidden external-channel `chat` fanout.
