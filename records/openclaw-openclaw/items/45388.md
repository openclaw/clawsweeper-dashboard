## Label Justifications

- `P2`: This is a real user-facing TUI live transcript bug with limited surface area and an open focused fix path.
- `impact:message-loss`: Inbound channel messages and agent replies are missing from the active TUI view until manual refresh or relaunch.
- `impact:session-state`: The visible TUI transcript can drift from the gateway session transcript even though stored history later contains the missing messages.

## Summary

Keep open: current main and v2026.5.19 still expose the gateway live transcript stream, but the bundled TUI does not subscribe to it or route `session.message` events. The issue has open implementation paths, so it should stay open until a scoped fix lands.

## What This Changes

The issue asks for `openclaw tui --session ... --deliver` to live-stream inbound Telegram messages and agent replies instead of only showing history on launch or manual refresh.

## Best Possible Solution

Land or replace the scoped TUI fix that consumes the existing `sessions.messages.subscribe` and `session.message` gateway contract, with active-session filtering, unsubscribe on session switch/stop, and regression coverage; keep bidirectional channel mirroring on https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes, source-level. Current main loads TUI history and handles chat/agent websocket events, while the gateway already emits subscribed `session.message` transcript events that the TUI does not subscribe to or dispatch.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to wire the TUI to the existing live transcript subscription contract rather than adding polling, widening generic chat fanout, or bundling the broader mirror-mode feature into this repair.
