## Label Justifications

- `P2`: This is a real user-facing TUI live transcript bug with limited TUI/gateway blast radius and active focused fix paths.
- `impact:message-loss`: Inbound Telegram messages and agent replies are missing from the active TUI view until manual refresh or relaunch.
- `impact:session-state`: The visible TUI transcript can drift from the stored gateway session transcript even though history later contains the missing messages.

## Summary

Keep open: current main and the latest release still expose the gateway live transcript subscription contract, but the bundled TUI does not subscribe to or dispatch `session.message` updates. The linked fix paths should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks for `openclaw tui --session ... --deliver` to live-tail Telegram session messages instead of requiring manual refresh or relaunch.

## Best Possible Solution

Land or replace the scoped TUI fix that consumes `sessions.messages.subscribe` and `session.message` with active-session filtering, unsubscribe on switch/stop, and regression coverage; keep bidirectional mirror mode on https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes, source-level. Current main loads TUI history through `chat.history` and routes only `chat`, `chat.side_result`, and `agent` websocket events, while the gateway already emits subscribed `session.message` transcript events that the TUI does not consume.

## Solution Assessment

Yes for the central bug. The narrow maintainable solution is to wire the TUI to the existing live transcript subscription contract, not to add polling, widen generic chat fanout, or bundle bidirectional mirror behavior into this repair.
