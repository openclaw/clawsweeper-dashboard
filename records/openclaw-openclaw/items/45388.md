## Label Justifications

- `P2`: This is a real user-facing TUI live transcript bug with limited TUI/gateway blast radius and active focused fix paths.
- `impact:message-loss`: Inbound channel messages and agent replies are missing from the active TUI view until manual refresh or relaunch.
- `impact:session-state`: The visible TUI transcript can drift from the stored gateway session transcript even though history later contains the missing messages.

## Summary

Keep open: current main and latest release still expose the gateway live transcript subscription contract, but the bundled TUI does not subscribe to `session.message` events or dispatch them. This is a source-reproducible TUI live transcript bug with active linked fix paths, so it should not be closed before a scoped implementation lands.

## What This Changes

The issue reports that `openclaw tui --session "telegram:direct:<id>" --deliver` loads Telegram session history but does not live-stream new inbound messages or agent replies.

## Best Possible Solution

Land or replace the scoped TUI fix that consumes the existing `sessions.messages.subscribe` and `session.message` gateway contract, with active-session filtering, unsubscribe on session switch/stop, and regression coverage; keep bidirectional channel mirroring on https://github.com/openclaw/openclaw/issues/40678.

## Reproduction Assessment

Yes, source-level. Current main loads TUI history and handles `chat`/`agent` websocket events, while the gateway already emits subscribed `session.message` transcript events that the TUI does not subscribe to or dispatch.

## Solution Assessment

Yes for the central bug. The narrow maintainable solution is to wire the TUI to the existing live transcript subscription contract, not to add polling, widen generic chat fanout, or bundle the broader mirror-mode feature into this repair.
