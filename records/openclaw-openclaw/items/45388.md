## Summary

Keep open: current main still does not connect TUI `--session` mode to the Gateway transcript event stream, so external Telegram/user turns can remain invisible until history reload even though assistant `chat` events may stream. The target fix PR is closed unmerged, and the open cross-repo PR is useful source work rather than proof that target `main` is fixed.

## What This Changes

The issue asks for `openclaw tui --session ... --deliver` to live-tail the attached session, including inbound Telegram/user turns and agent responses, without relaunch or manual refresh.

## Best Possible Solution

Wire TUI `--session` mode to `sessions.messages.subscribe`/`session.message`, update the subscription when the active session changes, render incoming transcript user turns without duplicating history, and keep bidirectional channel mirroring as a separate product decision.

## Reproduction Assessment

Yes. Source inspection on current main shows the TUI loads history and handles `chat`/`agent` events, while live transcript updates are delivered as `session.message` only to subscribed clients; the TUI neither subscribes to nor handles that event.

## Solution Assessment

Yes for the maintainable direction: the narrow fix should reuse the existing `sessions.messages.subscribe`/`session.message` transcript stream rather than inventing a new `chat` user-message state. The optional bidirectional mirror behavior from the comments should remain separate from this bug fix.
