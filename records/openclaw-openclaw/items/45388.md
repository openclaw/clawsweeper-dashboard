## Summary

Keep open: current main still has the gateway transcript subscription API but the TUI does not subscribe to, unsubscribe from, or dispatch `session.message` events, so the reported live-tail gap remains source-reproducible. The issue is also tied to open implementation candidates, including https://github.com/openclaw/openclaw/pull/73402 and https://github.com/BingqingLyu/openclaw/pull/640, so it should not be closed before a fix lands.

## What This Changes

The issue requests that `openclaw tui --session ... --deliver` live-tail inbound Telegram messages and agent replies instead of requiring manual refresh or relaunch.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/73402 as the scoped TUI fix using the existing `sessions.messages.subscribe` / `session.message` gateway contract, then close this issue and the canonical duplicate after merge.

## Reproduction Assessment

Yes, source-reproducible. Current main loads session history and handles `chat`/`agent` events but never subscribes to or dispatches `session.message`, while the gateway already emits that event for subscribed sessions.

## Solution Assessment

Yes for the narrow bug. The maintainable fix is to wire TUI session mode to the existing scoped gateway transcript subscription API; the older broad `chat` `user-message` approach and bidirectional mirroring idea are not the best current fix path.
