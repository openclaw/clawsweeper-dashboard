## Summary

Keep open: current main still lacks a TUI subscription/dispatch path for the gateway `session.message` transcript stream, so the reported live-tail gap remains source-reproducible. The issue is also paired with open implementation candidates, including the fork PR that closes this issue and the maintainer-labeled canonical fix for https://github.com/openclaw/openclaw/issues/38829, so it should not be closed before a fix lands.

## What This Changes

The issue reports that `openclaw tui --session ... --deliver` loads history but does not live-stream inbound Telegram messages or agent responses until refresh or relaunch.

## Best Possible Solution

Review, update, and land or replace https://github.com/openclaw/openclaw/pull/73402 as the scoped `sessions.messages.subscribe` / `session.message` TUI fix, then close this issue and the canonical duplicate after merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the TUI loads history but never subscribes to or handles `session.message`, which is enough to reproduce the missing live transcript path without running the live Telegram setup.

## Solution Assessment

Yes for the narrow bug: the maintainable fix is to wire TUI session mode to the existing scoped gateway transcript subscription API. The older broad `chat` `user-message` approach and the bidirectional mirroring idea are not the best current fix path.
