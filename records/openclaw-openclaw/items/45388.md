## Summary

Keep open: current main still has the Gateway transcript event contract but the bundled TUI does not subscribe to or route active-session `session.message` updates, so the reported live-tail bug remains. This issue also has an open linked fork PR, so it should not be closed before a safe same-repo fix lands or maintainers close/replace that candidate.

## What This Changes

The issue asks for `openclaw tui --session ... --deliver` to live-tail inbound channel messages and agent replies without relaunching or manual refresh.

## Best Possible Solution

Land a refreshed, security-reviewed same-repo fix that subscribes the TUI to active-session `sessions.messages.subscribe` updates, routes `session.message` safely, and leaves broader bidirectional mirroring to the existing product/security discussion.

## Reproduction Assessment

Yes, at source level: Gateway exposes and emits scoped `session.message` transcript updates, while current main TUI does not subscribe to or dispatch that event. I did not run a live Telegram/TUI terminal reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug: using the existing `sessions.messages.subscribe` and `session.message` contract is the clean owner-boundary fix. The broad chat-event fanout approach in the linked fork PR is not the best solution, and bidirectional mirroring belongs in the separate observe/mirror discussion.
