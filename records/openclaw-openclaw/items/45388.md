## Label Justifications

- `P2`: The bug breaks live cross-surface monitoring but preserves persisted history and has a reload or restart workaround.
- `impact:session-state`: The TUI's displayed transcript falls behind the canonical state of the attached session.
- `impact:message-loss`: Persisted inbound turns are omitted from the live TUI view until history is reloaded.

## Summary

Current main still loads the durable transcript without subscribing the TUI to live session-message events, which explains why reopening shows messages that the active view missed. Keep this issue open while the linked implementations are consolidated and a current-main-compatible fix receives real Gateway/TUI proof.

## What This Changes

Make a TUI attached to an existing channel session display new inbound messages and agent responses in real time without requiring a history refresh or restart.

## Best Possible Solution

Use the existing session-scoped transcript subscription API inside the TUI, with subscribe-on-connect/session-switch, unsubscribe-on-switch/exit, strict session filtering, and history reconciliation that preserves active local streaming; keep visible bidirectional channel delivery mirroring as separate product work.

## Reproduction Assessment

Yes. Current main loads durable history but lacks the TUI subscription and event-dispatch lifecycle needed to receive new transcript entries for the attached session, directly matching the reported refresh-after-reopen behavior.

## Solution Assessment

No for the older broadcast-based proposal. The narrowest maintainable solution is to consume the existing session-scoped transcript subscription API, as attempted by https://github.com/openclaw/openclaw/pull/73402, after refreshing it against current main.
