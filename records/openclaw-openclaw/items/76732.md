## Summary

This PR should stay open: it targets a real stale-image bug, but the latest diff uses process-wide mutable fresh-image state for provider replay, so overlapping attempts can make the privacy-sensitive pruning unreliable; the changelog entry also lands outside the existing Fixes section.

## What This Changes

The PR adds an aggressive history-image pruning option, applies it to prompt-build and provider replay when the current turn has images, adds coverage, and adds an Unreleased changelog note.

## Best Possible Solution

Keep the aggressive fresh-attachment pruning behavior, but make the replay pruning decision attempt-local and move the changelog note into the existing Fixes section.

## Reproduction Assessment

Yes for source-level reproduction. Current main preserves the last three completed image turns, and the PR head's concurrency problem follows from a module-global flag set before awaited prompt work and read later by provider replay transforms.

## Solution Assessment

No. The direction is the right narrow fix for the linked stale-image bug, but provider replay needs attempt-local state instead of process-wide mutable state.
