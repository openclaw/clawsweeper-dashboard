## Summary

Keep open: the PR targets a real source-reproducible privacy/reliability bug, but the current patch still uses process-global fresh-image state for provider replay and has no after-fix real Telegram/runtime proof.

## What This Changes

The PR adds aggressive history-image pruning for fresh attachment turns, threads it through prompt-build and provider replay, adds focused history-prune coverage, and adds a changelog note.

## Best Possible Solution

Keep the aggressive pruning direction, but make the fresh-image decision attempt-local across prompt-build and provider replay, preserve the raw transcript, and prove the fix with focused tests plus redacted Telegram/runtime evidence.

## Reproduction Assessment

Yes, at source level. Current main preserves recent prior image blocks and media markers in replay while separately loading current prompt images, so an image-heavy turn can still present old and new screenshots together.

## Solution Assessment

No. The direction is narrow and appropriate, but the PR’s provider replay path uses module-global state instead of attempt-local state, and the external PR still needs real behavior proof before merge.

Full review comments:

- **[P2] Keep fresh-image pruning state per attempt:** `src/agents/pi-embedded-runner/run/history-image-prune.ts:176-192`
