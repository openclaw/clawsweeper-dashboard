## Summary

This PR should stay open because the intended stale-image pruning fix is valid, but the current diff leaves the provider replay decision in shared module state and misplaces the changelog entry.

## What This Changes

The PR adds a history-image pruning override, applies it when the current turn has fresh images, wires provider replay to the same decision, adds coverage, and adds an Unreleased changelog entry.

## Best Possible Solution

Keep the aggressive pruning behavior, but scope the fresh-image decision to the current attempt or transform instance and move the user-facing note under Unreleased Fixes.

## Reproduction Assessment

Yes. Source inspection shows current main preserves the last three completed image turns and provider replay uses default pruning; the PR head then reads a shared module flag that can be overwritten before replay pruning.

## Solution Assessment

No. The direction is the right narrow fix, but provider replay needs attempt-local state instead of process-wide state, and the changelog should use the existing Fixes section.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep fresh-image pruning state per attempt:** `src/agents/pi-embedded-runner/run/history-image-prune.ts:176-178`
  - body: This module-level flag is shared by every installed replay transform in the process. If two embedded attempts overlap, another turn can flip the flag between setActiveTurnHasFreshImages() and activeSession.prompt(), so a fresh-image turn may still preserve stale images or an unrelated no-image turn may prune aggressively. Keep this state in the attempt/session transform closure or pass the pruning option through an attempt-local path.
