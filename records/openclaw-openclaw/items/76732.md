## Summary

Keep open: the PR targets a valid stale-image contamination bug, but the current branch still uses process-wide fresh-image state for provider replay and lacks after-fix real behavior proof.

## What This Changes

The PR adds an aggressive history-image pruning option for turns with fresh attachments, threads that decision into prompt-build and provider replay, adds a unit test, and adds a changelog entry.

## Best Possible Solution

Keep the aggressive pruning behavior, but scope the fresh-image decision to the current attempt or installed transform, preserve the raw transcript, and add isolation coverage plus real Telegram/runtime proof.

## Reproduction Assessment

Yes, at source level. Current main preserves recent image-bearing turns and provider replay uses the default pruning policy, while the PR head introduces a shared-state race that can be reasoned about without a live run.

## Solution Assessment

No. The direction is the right narrow fix, but provider replay needs attempt-local state instead of process-wide state, and the PR still needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep fresh-image pruning state per attempt:** `src/agents/pi-embedded-runner/run/history-image-prune.ts:176-192`
  - body: The new `activeTurnHasFreshImages` flag is shared by every installed replay transform in the process. If embedded attempts overlap, another turn can flip the flag between `setActiveTurnHasFreshImages()` and `activeSession.prompt()`, so a fresh-image turn may still preserve stale screenshots or an unrelated no-image turn may prune aggressively. Keep this state in the attempt/session transform closure or pass the pruning option through an attempt-local path.
  - confidence: 0.91
- **[P3] Use the existing Fixes changelog section:** `CHANGELOG.md:7-9`
  - body: The active changelog already groups user-facing fixes under Unreleased `### Fixes`, but this inserts a separate `### Bug fixes` section. Move the note into the existing section so release tooling and maintainer convention keep fixes grouped consistently.
  - confidence: 0.78
