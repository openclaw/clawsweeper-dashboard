## Summary

Keep this PR open: it addresses the valid image-contamination bug from #66702 and the latest head fixes the earlier hook/test/changelog gaps, but it now routes provider replay pruning through module-global turn state that can race between concurrent embedded attempts.

## What This Changes

The PR adds an optional history-image preservation override, applies aggressive pruning when the current turn has fresh images, adds a regression test, and adds an Unreleased changelog entry.

## Best Possible Solution

Keep the aggressive pruning decision scoped to the current attempt or installed transform instance, preserve the raw transcript unchanged, and place the user-facing note in the existing Unreleased Fixes section.

## Reproduction Assessment

Yes. Source inspection shows current main preserves the last three completed image turns, and the PR head stores fresh-image state in a module global that can be overwritten between the set call and provider replay pruning.

## Solution Assessment

No. The aggressive-pruning direction is right, but the replay path should receive attempt-local state rather than reading shared module state, and the changelog entry should use the existing Fixes section.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Keep fresh-image pruning state per attempt:** `src/agents/pi-embedded-runner/run/history-image-prune.ts:176-178`
  - body: This module-level flag is shared by every installed replay transform in the process. If two embedded attempts overlap, another turn can flip the flag between setActiveTurnHasFreshImages() and activeSession.prompt(), so a fresh-image turn may still preserve recent stale images or an unrelated no-image turn may prune aggressively. Keep this state in the attempt/session transform closure or pass the pruning option through an attempt-local path.
