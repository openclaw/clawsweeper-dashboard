## Summary

Keep open: this is a focused fix for a valid source-reproducible vision-history bug, but the PR is not merge-ready because provider replay uses process-wide fresh-image state and the external contributor has not provided real Telegram/runtime behavior proof.

## What This Changes

The PR adds a pruning option that removes all prior image/media history when the current turn has fresh attachments, wires it into prompt-build and provider replay, and adds a regression test plus changelog entry.

## Best Possible Solution

Keep the aggressive pruning behavior, but make the current-turn image decision attempt-local, preserve the raw transcript, and prove the behavior with focused tests plus real Telegram/runtime evidence.

## Reproduction Assessment

Yes, at source level. Current main preserves recent image-bearing turns and provider replay uses the default pruning policy, while the PR head’s shared-state race is visible from the changed source without a live run.

## Solution Assessment

No. The direction is the right narrow fix, but provider replay needs attempt-local state instead of process-wide state, and the PR still needs real behavior proof before merge.

Full review comments:

- **[P2] Keep fresh-image pruning state per attempt:** `src/agents/pi-embedded-runner/run/history-image-prune.ts:176-192`
