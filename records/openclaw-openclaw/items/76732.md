## Summary

This PR addresses a real stale-image contamination bug, but the current diff is not safe to merge because provider replay pruning depends on process-global mutable state that overlapping attempts can overwrite.

## What This Changes

The PR adds an aggressive history-image pruning option for fresh-image turns, wires it into prompt-build and provider replay, adds a regression test, and adds a changelog entry.

## Best Possible Solution

Keep the aggressive pruning behavior, but carry the fresh-image decision through attempt-local state or a transform closure and place the release note under Unreleased Fixes.

## Reproduction Assessment

Yes. Source inspection shows current main preserves the last three completed image turns and the linked report gives a concrete Telegram vision contamination path; I did not run tests because this was a read-only review.

## Solution Assessment

No. The intended pruning direction is right, but the provider replay path should not depend on a module-global flag shared across attempts.

- **[P2] Keep fresh-image pruning state per attempt:** `src/agents/pi-embedded-runner/run/history-image-prune.ts:176-178`
