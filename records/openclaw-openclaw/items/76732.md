## Summary

Keep open: the PR targets a valid stale-image bug, but the current head uses process-wide fresh-image state for provider replay, so overlapping embedded attempts can misapply a privacy-sensitive pruning decision.

## What This Changes

The PR adds configurable history-image pruning, applies aggressive pruning for image-bearing turns in prompt-build and provider replay paths, adds coverage, and adds a changelog entry.

## Best Possible Solution

Keep the aggressive pruning behavior, but scope the fresh-image decision to the current attempt or transform instance and move the changelog note into Unreleased Fixes.

## Reproduction Assessment

Yes, from source. Current main preserves the last three completed image turns and provider replay uses default pruning, while the PR head sets a shared flag before awaited prompt work so overlapping attempts can race.

## Solution Assessment

No. The direction is the narrow fix for #66702, but provider replay should use attempt-local state rather than a module global, and the changelog should follow the existing Fixes section.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Keep fresh-image pruning state per attempt:** `src/agents/pi-embedded-runner/run/history-image-prune.ts:176-178`
  - body: This module-level flag is shared by every installed replay transform in the process. If two embedded attempts overlap, another turn can flip the flag between `setActiveTurnHasFreshImages()` and `activeSession.prompt()`, so a fresh-image turn may still preserve stale images or an unrelated no-image turn may prune aggressively. Keep this state in the attempt/session transform closure or pass the pruning option through an attempt-local path.
