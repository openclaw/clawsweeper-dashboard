## Summary

Keep open: the PR addresses a real source-level bug, but the current patch still uses process-wide fresh-image state that can be overwritten by overlapping attempts, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds an aggressive history-image pruning option for fresh attachment turns, threads it through prompt-build and provider replay, adds coverage, and adds a changelog note.

## Best Possible Solution

Keep the aggressive pruning direction, but make the fresh-image decision attempt-local across prompt-build and provider replay, preserve the raw transcript, and prove it with focused tests plus redacted Telegram/runtime evidence.

## Reproduction Assessment

Yes, at source level. Current main preserves recent image-bearing turns and media markers in replay while separately loading current prompt images, so an image-heavy topic can still present old and new screenshots together.

## Solution Assessment

No. The direction is narrow and appropriate, but provider replay needs attempt-local state instead of a module-global flag, and the external PR still needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep fresh-image pruning state per attempt:** `src/agents/pi-embedded-runner/run/history-image-prune.ts:176-192`
  - body: The new activeTurnHasFreshImages flag is shared by every installed replay transform in the process. Overlapping embedded attempts can flip it between another attempt's setup and provider replay, so a fresh-image turn may still preserve stale screenshots or an unrelated no-image turn may prune aggressively. Store this in attempt/session-local state or pass the pruning option through an attempt-local path.
