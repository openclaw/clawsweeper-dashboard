## Summary

Keep this PR open because it targets a real WhatsApp auto-reply lifecycle gap, but the current branch is not merge-ready: current main already handles the `message_sending` cancel/rewrite part through the shared inbound `beforeDeliver` path, while this branch reintroduces delivery-accounting bugs and still lacks inspectable real WhatsApp proof.

## What This Changes

The PR runs WhatsApp auto-reply sending/sent hooks in `deliverWebReply`, threads session metadata from buffered inbound dispatch, exports `getGlobalHookRunner` through `hook-runtime`, and adds focused WhatsApp auto-reply tests.

## Best Possible Solution

Keep the shared `beforeDeliver` `message_sending` path and add a narrow WhatsApp auto-reply `message_sent` / internal `message:sent` parity path only after provider-accepted delivery, using the final delivered content.

## Reproduction Assessment

Yes, by source inspection. Current main covers inbound `message_sending` cancel/rewrite, but WhatsApp owner delivery paths can still record accepted media/block sends without the generic `message_sent` / internal `message:sent` parity; I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

No. The branch targets a real lifecycle gap, but it duplicates `message_sending` inside WhatsApp delivery and loses current-main cancellation, rewrite, and provider-accepted delivery contracts; a narrower sent-hook emitter on the remaining owner delivery path is safer.

  - body: When `message_sending` returns `cancel: true`, this function exits without telling `dispatchWhatsAppBufferedReply` delivery was skipped. The caller still marks the reply as sent and records sent text, so a plugin-cancelled WhatsApp reply can pollute echo suppression/history.

  - confidence: 0.88

  - body: This emits successful sent hooks after the media/text block even when the effective payload has no sendable text or media, such as a hook rewrite to an empty string. That reports `message_sent` success for a WhatsApp message that never reached the provider.
  - confidence: 0.9
