## Summary

Keep open. The requested WhatsApp sent-hook parity is still plausible, but this branch duplicates behavior now handled on current main and has merge-blocking correctness issues plus missing inspectable real behavior proof.

## What This Changes

The PR adds WhatsApp auto-reply `message_sending`, plugin `message_sent`, and internal `message:sent` handling plus focused tests and a hook-runtime SDK export.

## Best Possible Solution

Reuse current main’s shared `beforeDeliver`/`message_sending` path and add only the missing WhatsApp `message_sent` / `message:sent` parity for accepted media/block auto-reply deliveries.

## Reproduction Assessment

Yes. A hook-only plugin listening for `message_sent` or internal `message:sent` can trigger a WhatsApp media/block auto-reply; source inspection shows current main records provider-accepted sends on that owner delivery path without emitting those sent hooks.

## Solution Assessment

No. The goal is valid, but this branch solves too much in `deliverWebReply`; the maintainable fix is a narrower sent-hook parity change built on current main’s shared delivery and provider-accepted result contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Report cancelled sends to the dispatcher:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:93`
  - body: When `message_sending` returns `cancel: true`, this returns without telling `dispatchWhatsAppBufferedReply` that delivery was skipped. The caller still marks the reply as sent and records sent text, so a plugin-cancelled WhatsApp reply can pollute echo suppression/history.
  - confidence: 0.91
- **[P2] Return rewritten content to sent tracking:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:96`
  - body: A hook rewrite only updates this local `replyResult` copy. The caller continues to call `rememberSentText(payload.text, ...)` with the original payload, so self-echo suppression/history can track text that was not actually sent.
  - confidence: 0.88
- **[P2] Guard success hooks behind an accepted send:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: This emits successful sent hooks after the media/text block even when the effective payload has no sendable text or media, such as a hook rewrite to an empty string. That reports `message_sent` success for a WhatsApp message that never reached the provider.
  - confidence: 0.9
- **[P2] Keep sending hook failures in the delivery error path:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:77`
  - body: `runMessageSending` runs before the delivery `try` block, so a plugin hook exception rejects before delivery and before failed sent-hook emission. The generic outbound path catches `message_sending` hook failures and continues, so this path should preserve that contract.
  - confidence: 0.84
- **[P2] Update SDK contract artifacts or avoid the export:** `src/plugin-sdk/hook-runtime.ts:3`
  - body: Adding `getGlobalHookRunner` to `openclaw/plugin-sdk/hook-runtime` changes a public SDK subpath, but the branch does not update the API baseline/docs contract follow-through required for SDK changes. Either avoid the new export or update the SDK contract artifacts with the code change.
  - confidence: 0.76
