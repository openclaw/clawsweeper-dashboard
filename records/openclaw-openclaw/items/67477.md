## Summary

Keep this PR open. Current main already covers the shared inbound `message_sending` before-delivery path, but WhatsApp auto-reply delivery still lacks plugin `message_sent` and internal `message:sent` emission; the branch targets a real remaining gap, but the proposed patch is not safe to merge as-is because it misreports cancellation, rewritten content, empty sends, and hook-failure outcomes.

## What This Changes

The PR runs WhatsApp auto-reply hook handling in `deliverWebReply`, emits plugin `message_sent` and internal `message:sent`, threads session/account/conversation metadata through inbound dispatch, exports `getGlobalHookRunner` from `hook-runtime`, and adds focused tests.

## Best Possible Solution

Rebase and narrow the work around the still-missing WhatsApp auto-reply sent-hook parity. The final implementation should reuse the shared inbound `message_sending` path already on main, emit sent hooks only from accurate delivered or failed outcomes, preserve WhatsApp delivery result semantics, and keep SDK contract changes documented and checked.

## Reproduction Assessment

Yes. A hook-only plugin registering `message_sent` or internal `message:sent` can be exercised by sending a WhatsApp message that triggers an auto-reply; current main has no sent-hook call in the WhatsApp auto-reply delivery path, while the generic outbound path does.

## Solution Assessment

No, not as-is. The intended direction is right, but the patch duplicates now-shipped `message_sending` handling and does not correctly propagate cancellation, rewritten content, empty-send outcomes, or hook exceptions back to the dispatcher and hook telemetry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate canceled deliveries back to the dispatcher:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:93`
  - body: When `message_sending` returns `cancel: true`, `deliverWebReply` returns with no signal to `dispatchWhatsAppBufferedReply`. The caller still sets `didSendReply = true` and calls `rememberSentText`, so a plugin-canceled WhatsApp reply is recorded as sent and can pollute echo suppression/history. Return an explicit no-send/cancel result and have the caller skip sent tracking.
  - confidence: 0.91
- **[P2] Return rewritten content to sent-text tracking:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:96`
  - body: Hook rewrites only update the local `replyResult` copy. The dispatcher still records `payload.text` from the original payload after delivery, so self-echo suppression and history can track text that was not actually sent. Propagate the effective delivered text or return it in the delivery result.
  - confidence: 0.88
- **[P2] Avoid emitting success when no message was delivered:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: If a hook rewrites text to an empty string and there is no media, this function skips the text send path, sends no media, and still calls `emitMessageSentHooks(true)`. That reports a successful `message_sent` for an outbound message that never reached WhatsApp.
  - confidence: 0.9
- **[P2] Keep hook failures from aborting delivery:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:77`
  - body: `runMessageSending` runs before the `try` block, so a plugin hook exception rejects `deliverWebReply` before delivery or failed sent-hook emission. The canonical outbound path treats `message_sending` failures as non-blocking; keep WhatsApp auto-replies aligned by catching hook failures before delivery.
  - confidence: 0.84
- **[P2] Update the SDK contract artifacts for the new export:** `src/plugin-sdk/hook-runtime.ts:3`
  - body: Adding `getGlobalHookRunner` to `openclaw/plugin-sdk/hook-runtime` changes a public plugin SDK subpath. The repo policy for SDK seam expansion requires docs/API-baseline follow-through, so the PR should update the generated SDK API hash and relevant SDK docs or avoid adding this public export.
  - confidence: 0.76
