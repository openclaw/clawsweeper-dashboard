## Summary

Keep this PR open: the remaining WhatsApp auto-reply sent-hook parity is still a valid gap, but the branch is not mergeable as-is because current main already owns `message_sending` through the shared inbound `beforeDeliver` path and this PR reintroduces several delivery-accounting bugs.

## What This Changes

The branch wires WhatsApp auto-reply delivery through `message_sending`, emits plugin/internal sent hooks, threads session/account/conversation metadata, exposes `getGlobalHookRunner` from `plugin-sdk/hook-runtime`, and adds focused tests.

## Best Possible Solution

Reuse current main’s shared `beforeDeliver`/`message_sending` and durable final-delivery paths, then add only the missing WhatsApp media/block sent-hook parity based on actual accepted delivery results without expanding the SDK unless a documented public seam is truly needed.

## Reproduction Assessment

Yes. A hook-only plugin listening for `message_sent` or internal `message:sent` can trigger a WhatsApp auto-reply; source inspection shows current main records accepted WhatsApp media/block replies without a sent-hook emission on that owner delivery path.

## Solution Assessment

No. The goal is valid, but this branch solves too much in the channel-local delivery function and should be narrowed to sent-hook parity on top of current main’s shared `message_sending` path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Report canceled sends to the dispatcher:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:93`
  - body: When `message_sending` returns `cancel: true`, `deliverWebReply` returns without telling `dispatchWhatsAppBufferedReply` that nothing was delivered. The caller then sets `didSendReply = true` and records sent text, so a plugin-cancelled reply can pollute echo suppression/history.
  - confidence: 0.91
- **[P2] Propagate rewritten content to sent tracking:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:96`
  - body: Hook rewrites only update the local `replyResult` copy. The caller still records the original payload text after delivery, so self-echo suppression/history can track text that was not actually sent.
  - confidence: 0.88
- **[P2] Emit success only after a provider-accepted send:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: This emits a successful sent hook after the media/text section even when the effective payload has no sendable text or media, such as a hook rewrite to an empty string. That reports `message_sent` success for a WhatsApp message that never reached the provider.
  - confidence: 0.9
- **[P2] Keep sending hook failures from aborting delivery:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:77`
  - body: `runMessageSending` executes before the delivery `try` block, so a plugin hook exception rejects `deliverWebReply` before delivery and before failed sent-hook emission. The generic outbound path catches `message_sending` hook failures and continues delivery, so this path should preserve the same contract.
  - confidence: 0.84
- **[P2] Update SDK contract artifacts or avoid the export:** `src/plugin-sdk/hook-runtime.ts:3`
  - body: Adding `getGlobalHookRunner` to `openclaw/plugin-sdk/hook-runtime` changes a public SDK subpath, but the branch does not update the API baseline/docs contract follow-through required for public SDK changes. Either avoid the new export or update the SDK contract artifacts with the code change.
  - confidence: 0.76
