## Summary

Keep open: this PR targets a real WhatsApp hook lifecycle gap, but current main already implements the shared inbound `message_sending` cancel/rewrite path, while the branch is conflicting, lacks inspectable real WhatsApp proof, and still introduces delivery-accounting bugs.

## What This Changes

The branch adds WhatsApp auto-reply `message_sending`/`message_sent` hook calls, threads session/conversation metadata through buffered dispatch, exports `getGlobalHookRunner` from `hook-runtime`, and adds focused tests.

## Best Possible Solution

Keep the shared `beforeDeliver` sending-hook path and add only the missing WhatsApp owner/media delivery `message_sent` and internal `message:sent` parity after provider-accepted delivery, using the final delivered content.

## Reproduction Assessment

Yes, by source inspection. The PR head returns on cancel and rewrites only a local copy while its caller records success and the original payload; current main shows the shared cancel/rewrite path is already handled before delivery.

## Solution Assessment

No. The branch duplicates `message_sending` work now handled on main and should be narrowed to sent-hook parity on the remaining WhatsApp owner delivery path after accepted delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Report cancellation back to buffered dispatch:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:93`
  - body: When `message_sending` returns `cancel: true`, this function returns with no delivery result, but the caller still marks the reply as sent and records echo text. Return a skipped delivery result or otherwise propagate cancellation so buffered dispatch does not treat a plugin-cancelled WhatsApp reply as delivered.
  - confidence: 0.91
- **[P2] Return rewritten content to the dispatcher:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:96`
  - body: A hook rewrite updates only this local `replyResult` binding. `dispatchWhatsAppBufferedReply` still records the original `payload.text`, so self-echo/history tracking can store text that was not actually sent.
  - confidence: 0.88
- **[P2] Only emit sent hooks after accepted delivery:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: The media path emits successful sent hooks after the media/text block even when no sendable text or media was delivered, such as a rewrite to an empty string. Gate success on an accepted provider result instead of reaching this unconditional success call.
  - confidence: 0.9
- **[P2] Emit the actual fallback content:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:295-301`
  - body: When the first media send fails, `onError` can send a synthesized fallback reply with a warning, but `contentForSentHook` was captured before delivery. Hook consumers should see the final fallback text that the user received, not the stale pre-fallback content.
  - confidence: 0.82
