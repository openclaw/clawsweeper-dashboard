## Summary

Keep open, but not merge-ready: the WhatsApp hook gap is real, while this PR is conflicting, lacks inspectable real WhatsApp proof, duplicates current-main `message_sending` handling, and can mis-account cancelled or rewritten sends.

## What This Changes

The PR adds WhatsApp auto-reply `message_sending`/`message_sent` hook emission, threads session and conversation metadata into buffered delivery, exports `getGlobalHookRunner` from `hook-runtime`, and adds focused delivery tests.

## Best Possible Solution

Rebase onto current main and implement WhatsApp sent-hook parity at the provider-accepted delivery result or durable lifecycle layer, preserving the central `message_sending` path and avoiding an unapproved broad SDK export.

## Reproduction Assessment

Source-reproducible, not live-reproduced: the PR head returns from `deliverWebReply` on `message_sending` cancellation without telling the buffered dispatcher, and current main already has a central `beforeDeliver` hook path that the PR duplicates.

## Solution Assessment

No. The requested hook coverage is useful, but this implementation should be rebased and narrowed to sent-hook parity on the accepted-delivery path rather than adding a second local sending-hook pipeline and a broad public hook-runner export.

Full review comments:

- **[P1] Remove the duplicate message_sending path:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:76-77`
  - body: Current main already applies `message_sending` through the shared dispatcher `beforeDeliver` path before inbound reply delivery. Keeping this local call in WhatsApp delivery will either run the hook twice after rebase or split cancellation/rewrite behavior across two layers.
  - confidence: 0.88
- **[P1] Report hook cancellation to the dispatcher:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:92-93`
  - body: When `runMessageSending` returns `cancel: true`, this function returns with no skipped-delivery result. The buffered dispatcher can still mark the original payload as sent, so a plugin that cancels delivery can pollute echo/history accounting.
  - confidence: 0.9
- **[P2] Propagate rewritten content to sent accounting:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:95-96`
  - body: A hook rewrite only updates the local `replyResult`, while the caller still remembers the original payload text. That can make self-echo suppression and downstream history use text that was not actually delivered.
  - confidence: 0.86
- **[P2] Emit sent hooks from accepted delivery results:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: This success hook fires after the media path without tying the event to provider-accepted results or fallback text actually sent. Sent-hook consumers need the final delivered content and failure/suppression outcome, not the pre-delivery snapshot.
  - confidence: 0.82
- **[P1] Avoid exposing the raw global hook runner:** `src/plugin-sdk/hook-runtime.ts:3`
  - body: Adding `getGlobalHookRunner` to a public SDK subpath gives plugins a broad new way to synthesize hook events. This needs an explicit SDK contract decision, or a narrower helper that only exposes the WhatsApp sent-hook operation needed here.
  - confidence: 0.76
