## Summary

Keep open: current main already covers the linked message_sending cancellation path through shared beforeDeliver, but this PR still has a useful WhatsApp sent-hook parity goal and cannot merge as written because the branch is stale and breaks delivery accounting.

## What This Changes

The PR wires WhatsApp auto-reply delivery into message_sending, plugin message_sent, and internal message:sent hooks while threading session and conversation metadata through the WhatsApp dispatch path.

## Best Possible Solution

Keep the current shared beforeDeliver message_sending path and rework this PR into a narrow WhatsApp message_sent/internal message:sent parity patch that fires only after provider-accepted delivery with the final delivered content.

## Reproduction Assessment

Yes, by source inspection of the PR head. The new cancellation/rewrite path in deliverWebReply does not return an outcome, while the caller still records the original payload as sent.

## Solution Assessment

No. The branch should not reintroduce channel-local message_sending now that current main has the shared beforeDeliver path; it should be narrowed to provider-accepted WhatsApp sent-hook parity.
