## Summary

Keep this PR open because the remaining WhatsApp `message_sent` parity goal is still plausible, but the current branch is not mergeable: it reimplements `message_sending` in the WhatsApp delivery function even though current `main` now has a shared `beforeDeliver` path, and it reports sent-hook success without preserving current provider-accepted delivery semantics.

## What This Changes

The branch adds WhatsApp auto-reply `message_sending`, plugin `message_sent`, and internal `message:sent` handling, threads session metadata through inbound dispatch, exports `getGlobalHookRunner` from `hook-runtime`, and adds focused tests.

## Best Possible Solution

Add a narrow current-main WhatsApp sent-hook parity path for owner-delivered media/block auto-replies, emitted only after accepted delivery and using the final delivered content, while preserving the shared `beforeDeliver` `message_sending` flow.

## Reproduction Assessment

Yes, by source inspection: a hook-only plugin listening for `message_sent` or internal `message:sent` can trigger a WhatsApp media or block auto-reply, and current `main` still has owner delivery paths that record accepted sends without that sent-hook parity. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

No. The branch targets a real lifecycle gap, but it duplicates `message_sending` inside `deliverWebReply` and loses current-main cancellation, rewrite, and provider-accepted delivery contracts; the safer fix is a narrower sent-hook emitter on the remaining owner delivery path.

  - body: A hook rewrite only updates this local `replyResult` copy. The caller still records the original `payload.text`, so self-echo suppression/history can track text that was not actually sent.
  - confidence: 0.88
- **[P2] Only emit success hooks after accepted delivery:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: This emits successful sent hooks after the media/text block even when the effective payload has no sendable text or media, such as a hook rewrite to an empty string. That reports `message_sent` success for a WhatsApp message that never reached the provider.
  - confidence: 0.9
- **[P2] Keep sending hook failures in the delivery path:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:77`
  - body: `runMessageSending` executes before the delivery `try` block, so a plugin hook exception rejects before delivery and before failed sent-hook emission.   - confidence: 0.84
