## Summary

Keep open, but do not merge as-is. Current main already fixed the linked `before_dispatch` / `message_sending` cancel gap through the shared inbound dispatcher, while this PR still has useful WhatsApp `message_sent` parity work plus several correctness blockers and no inspectable real WhatsApp proof.

## What This Changes

The PR wires WhatsApp auto-reply `message_sending`, plugin `message_sent`, and internal `message:sent` hooks through web reply delivery, passes session/conversation metadata from buffered dispatch, exports `getGlobalHookRunner` from `hook-runtime`, and adds focused tests.

## Best Possible Solution

Keep the shared `beforeDeliver` `message_sending` path on main and add only the missing WhatsApp owner/media `message_sent` and internal `message:sent` parity after provider-accepted delivery, using the final delivered content.

## Reproduction Assessment

Yes, by source inspection of the PR head. Cancellation returns from `deliverWebReply` while the caller still records a sent reply, and current main shows the accepted-delivery/result propagation contract this path should preserve.

## Solution Assessment

No. The branch duplicates `message_sending` behavior now handled on main and should be narrowed to sent-hook parity on the remaining WhatsApp owner/media delivery path after accepted delivery.

Overall confidence: 0.88

Full review comments:

- **[P1] Return cancellation to buffered dispatch:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:93`
  - body: When `message_sending` returns `cancel: true`, this returns without telling `dispatchWhatsAppBufferedReply`; the caller still marks the reply sent and remembers the original text, so a plugin-cancelled WhatsApp reply is recorded as delivered.
  - confidence: 0.91
- **[P2] Propagate rewritten text to the dispatcher:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:96`
  - body: A hook rewrite updates only this local `replyResult`; the caller still records `payload.text`, so self-echo/history tracking can store text that was not actually sent.
  - confidence: 0.88
- **[P2] Gate sent hooks on accepted delivery:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: The media path emits successful sent hooks after the media/text block even when no sendable payload was delivered, such as a rewrite to an empty string. Downstream hooks should not observe a successful send that never happened.
  - confidence: 0.9
- **[P2] Use final delivered content for sent hooks:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:100`
  - body: `contentForSentHook` is captured before the media fallback path can send synthesized warning text, so hook consumers can observe stale content instead of what the user actually received.
  - confidence: 0.82
