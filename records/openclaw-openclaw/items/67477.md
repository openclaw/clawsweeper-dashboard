## Summary

Keep open: the hook parity problem is real, but this branch is not safe to merge as-is. Current main already handles `message_sending` through the shared `beforeDeliver` path, while this PR adds stale channel-local delivery logic that misreports cancellation, rewrites, and no-send outcomes, and it still lacks inspectable WhatsApp proof.

## What This Changes

The PR adds WhatsApp auto-reply `message_sending`, plugin `message_sent`, and internal `message:sent` handling, threads session metadata through inbound dispatch, exports `getGlobalHookRunner` from `hook-runtime`, and adds focused tests.

## Best Possible Solution

Rework the PR into a narrower current-main patch that keeps the shared `beforeDeliver` `message_sending` path and adds WhatsApp `message_sent` / `message:sent` parity only for provider-accepted auto-reply deliveries, with redacted real WhatsApp proof.

## Reproduction Assessment

Yes. A hook-only plugin listening for `message_sent` or internal `message:sent` can trigger a WhatsApp media/block auto-reply; source inspection shows current main records provider-accepted WhatsApp sends on that owner delivery path without emitting those sent hooks.

## Solution Assessment

No. The goal is valid, but this branch solves too much inside `deliverWebReply`; the maintainable fix is a narrower sent-hook parity change built on current main's shared delivery and provider-accepted result contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Report cancellation back to the dispatcher:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:93`
  - body: When `message_sending` returns `cancel: true`, this returns without telling `dispatchWhatsAppBufferedReply` that delivery was skipped. The caller still marks the reply as sent and records sent text, so a plugin-cancelled WhatsApp reply can pollute echo suppression/history.
  - confidence: 0.91
- **[P2] Propagate rewritten content to sent tracking:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:96`
  - body: A hook rewrite only updates this local `replyResult` copy. The caller continues to call `rememberSentText(payload.text, ...)` with the original payload, so self-echo suppression/history can track text that was not actually sent.
  - confidence: 0.88
- **[P2] Only emit success hooks after accepted delivery:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: This emits successful sent hooks after the media/text block even when the effective payload has no sendable text or media, such as a hook rewrite to an empty string. That reports `message_sent` success for a WhatsApp message that never reached the provider.
  - confidence: 0.9
- **[P2] Keep sending hook failures in the delivery path:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:77`
  - body: `runMessageSending` runs before the delivery `try` block, so a plugin hook exception rejects before delivery and before failed sent-hook emission. The generic outbound path fails open on `message_sending` hook errors, so this path should preserve that contract instead of aborting early.
  - confidence: 0.84
