## Summary

Keep this PR open: current main already covers the `message_sending` half through the shared inbound dispatcher seam, but WhatsApp auto-replies still do not emit `message_sent` / `message:sent`; the PR remains useful but is not mergeable as-is because it duplicates the now-shared path and loses delivery-result semantics.

## What This Changes

The PR wires WhatsApp auto-reply delivery through `message_sending`, emits plugin/internal sent hooks, threads session/account/conversation metadata, exports a hook-runner SDK helper, and adds focused tests.

## Best Possible Solution

Keep the shipped shared `message_sending` implementation and add only the missing WhatsApp auto-reply sent-hook parity, driven by actual delivery results and current normalized payload semantics.

## Reproduction Assessment

Yes. A hook-only plugin registering `message_sent` or an internal `message:sent` handler can trigger a WhatsApp auto-reply; current main delivers and tracks accepted WhatsApp replies without any sent-hook emission in that path.

## Solution Assessment

No, not as-is. The remaining goal is valid, but the patch should be narrowed to sent-hook parity and should reuse current main’s shared `message_sending` path instead of reimplementing it inside WhatsApp delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate canceled deliveries back to the dispatcher:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:93`
  - body: When `message_sending` returns `cancel: true`, `deliverWebReply` returns without telling the caller that no message was delivered. The dispatcher can then still record the reply as sent, so a plugin-canceled WhatsApp reply can pollute echo suppression/history; return an explicit no-send result and skip sent tracking.
  - confidence: 0.91
- **[P2] Return rewritten content to sent-text tracking:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:96`
  - body: Hook rewrites only update the local `replyResult` copy in `deliverWebReply`. The caller still tracks the original payload text after delivery, so self-echo suppression/history can record text that was not actually sent; propagate the effective delivered text or keep this on the shared `beforeDeliver` path.
  - confidence: 0.88
- **[P2] Avoid emitting success when nothing was delivered:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:309`
  - body: The patch emits a successful sent hook after the media/text section even when the normalized payload has no sendable text or media. That reports `message_sent` success for a WhatsApp message that never reached the provider, which can corrupt logging and downstream automations.
  - confidence: 0.9
- **[P2] Keep sending hook failures from aborting delivery:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:77`
  - body: `runMessageSending` runs before the delivery `try` block, so a plugin hook exception rejects `deliverWebReply` before delivery and before failed sent-hook emission. The generic outbound path catches `message_sending` hook failures and continues delivery, so keep WhatsApp auto-replies aligned with that contract.
  - confidence: 0.84
- **[P2] Update SDK API artifacts for the new export:** `src/plugin-sdk/hook-runtime.ts:3`
  - body: Adding `getGlobalHookRunner` to `openclaw/plugin-sdk/hook-runtime` changes a public plugin SDK subpath, but the PR does not update the generated SDK API hash/docs follow-through required for public contract changes. Either avoid the new public export or update the SDK contract artifacts and checks with the code change.
  - confidence: 0.76
