## Summary

Keep open. Current main partially satisfies the umbrella with HTTP /tools/invoke and SDK/RPC tools.invoke, but meaningful requested pieces remain: cron has no direct exec/tool payload, generic slash/CLI tool invocation is not present, and core coding-tool exposure over the Gateway direct-invoke surface is still under active product/security review.

## What This Changes

The issue requests direct OpenClaw tool invocation, such as slash/CLI/cron tool calls, without routing deterministic work through an LLM turn.

## Best Possible Solution

Keep the existing HTTP/RPC/SDK and skill-command direct-invoke paths, then resolve the remaining Gateway coding-tool exposure, cron direct-exec payload, and any generic slash/CLI surface as scoped security-reviewed follow-ups.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection gives a high-confidence state map: direct HTTP/RPC/SDK invoke exists, while cron direct exec/tool payloads, generic slash/CLI invocation, and coding-tool exposure are not fully implemented on current main.

## Solution Assessment

No, not as one broad umbrella change. The maintainable path is to keep the shared Gateway invoke helper and split the remaining public protocol, cron, CLI/slash UX, and coding-tool security surfaces into scoped owner-reviewed slices.
