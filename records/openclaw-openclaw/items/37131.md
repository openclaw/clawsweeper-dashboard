## Summary

Current main solves the SDK/API portion through a shared authenticated Gateway invocation path, but the broader request remains necessary: direct `exec` and host-file tools are intentionally denied, no direct cron payload exists, and the linked open PRs cover only selected tool families or a separate CLI plan workflow.

## What This Changes

Add supported chat, CLI, API/SDK, and cron paths for invoking explicit OpenClaw tools without an LLM inference turn.

## Best Possible Solution

Keep Gateway `tools.invoke` as the single execution boundary, define a maintainer-approved matrix of caller surfaces and tool risk classes, then add thin chat, CLI, or cron adapters incrementally with explicit opt-ins, audit coverage, upgrade-safe defaults, and canonical transcript handling.

## Reproduction Assessment

Not applicable; this is a feature request, and current-main source inspection establishes which portions exist and which remain intentionally unavailable.

## Solution Assessment

No, not as one unrestricted `/tool` or cron dispatcher; the narrowest maintainable solution is thin caller adapters over the existing Gateway boundary with separate explicit controls for each high-risk tool class.
