## Summary

Keep open. Current main has partial no-LLM direct invocation through Gateway HTTP, the SDK-facing `tools.invoke` RPC, and skill-owned `command-dispatch: tool`, but the requested umbrella still covers missing or incomplete surfaces: generic chat `/tool ...`, `openclaw agent --tool/--args`, direct cron tool payloads, and coding-tool availability on the current Gateway resolver. The related open items #63919 and #18160 remain useful narrower slices, while merged #74804 only covers the SDK RPC slice.

## What This Changes

Request support for invoking explicit OpenClaw tools from chat, CLI, and cron payloads without sending a normal LLM agent turn.

## Best Possible Solution

Keep this as the umbrella product/API request until maintainers either define one consistent direct-invocation contract across Gateway HTTP/RPC, CLI, cron, and chat commands, or deliberately split the remaining work into canonical narrower trackers. Any implementation should preserve default-deny behavior for high-risk tools and make approval/reporting semantics explicit.

## Reproduction Assessment

Not applicable as a bug reproduction. The current gaps are high-confidence by source inspection: cron has no direct tool payload, `openclaw agent` has no `--tool/--args`, generic `/tool ...` chat syntax is absent, and the current Gateway resolver still excludes coding tools.

## Solution Assessment

No; the current implementation is not yet the complete or best solution for the umbrella request. The merged SDK RPC is a good slice, but the remaining surfaces require maintainer-owned product and security design before a safe general implementation.

## Security Review

Status: needs_attention

Summary: The issue itself is security-sensitive because it proposes direct execution of tools that can reach command execution, filesystem mutation, session control, cron, Gateway config, and node relay surfaces.
