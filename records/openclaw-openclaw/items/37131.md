## Summary

Keep open. Current main covers the shared Gateway HTTP/RPC invocation engine and command cron jobs, but the requested generic CLI/slash surfaces, coding-tool exposure, and transcript/channel-result semantics are still incomplete and need security/product sequencing.

## What This Changes

The issue asks for a model-free path to invoke OpenClaw tools directly from chat/slash, CLI, and cron contexts with explicit arguments and session or channel result handling.

## Best Possible Solution

Keep the shared Gateway invoke engine as the only execution boundary, then land CLI, slash, coding-tool, and result-persistence surfaces as narrow security-reviewed changes with explicit owner/admin and config semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Current-main source, docs, tests, and live related-PR state give a high-confidence view of which requested surfaces exist and which remain missing.

## Solution Assessment

No, not as one broad implementation. The shared Gateway invoke path is the right foundation, but CLI, slash commands, cron/tool payloads, dangerous tool classes, delivery, and transcript persistence should remain separate security-reviewed decisions.
