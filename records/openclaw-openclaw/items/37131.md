## Summary

Keep open. Current main has the Gateway HTTP/RPC/SDK direct-invoke foundation, but the broader request still has meaningful open work for coding-tool HTTP exposure, CLI plan execution, cron/direct exec payloads, and session-spawn/API policy.

## What This Changes

Request direct invocation of OpenClaw tools from chat/slash commands, CLI, cron, and automation paths without a model round trip.

## Best Possible Solution

Keep the shared Gateway `tools.invoke` foundation and resolve the remaining HTTP coding-tool, CLI, cron, session-spawn, and startup-hook safety pieces through scoped security-reviewed follow-ups before closing the umbrella.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that RPC/SDK/HTTP foundations exist while direct cron payloads, generic CLI execution, and broader coding-tool exposure still depend on open follow-ups.

## Solution Assessment

No for a single umbrella implementation: this should not be solved by adding parallel dispatchers across every surface at once. The maintainable path is to reuse the Gateway policy/deny-list/hook pipeline and review each public invocation surface separately.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it can expose host command execution, filesystem mutation, session orchestration, message delivery, and scheduled automation outside a model turn.

Concerns:

- **[high] Preserve direct-invoke deny-list and hook boundaries:** `src/security/dangerous-tools.ts:9`
  - body: Current main default-denies high-risk HTTP tools such as exec, spawn, apply_patch, sessions_spawn, sessions_send, cron, gateway, and nodes; any expansion must preserve explicit opt-in, before_tool_call hooks, owner semantics, policy filtering, and auditability.
  - confidence: 0.92
- **[high] Define unattended cron execution policy first:** `src/gateway/protocol/schema/cron.ts:171`
  - body: Cron payload schemas currently allow only systemEvent and agentTurn; adding direct toolCall or exec payloads would schedule unattended host actions and needs explicit sandbox, approval, timeout, retry, output retention, cancellation, and delivery rules.
  - confidence: 0.88
