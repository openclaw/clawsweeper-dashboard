## Summary

Keep open. Current main and the latest release have the HTTP/RPC/SDK direct-invoke foundation, but the issue's broader request still has open product/security work for slash or chat invocation, CLI execution, cron tool payloads, and coding-tool exposure.

## What This Changes

The issue requests direct OpenClaw tool invocation from chat, slash command, CLI, and cron paths without requiring an LLM turn.

## Best Possible Solution

Keep the shipped HTTP/RPC/SDK foundation, then resolve the remaining slash/chat, CLI, cron, coding-tool, and session-spawn pieces through scoped security-reviewed follow-ups such as https://github.com/openclaw/openclaw/pull/63919, https://github.com/openclaw/openclaw/pull/82232, https://github.com/openclaw/openclaw/issues/18160, and https://github.com/openclaw/openclaw/issues/67832.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that HTTP/RPC/SDK direct invoke exists while slash/chat direct invoke, a generic direct CLI on main, cron toolCall/exec payloads, and broad coding-tool exposure remain absent or open follow-ups.

## Solution Assessment

No for the umbrella as one implementation: it crosses too many trust boundaries at once. The maintainable path is scoped maintainer/security review for the existing direct-invoke follow-ups, all reusing the Gateway policy and deny-list boundary instead of adding parallel dispatchers.

Status: needs_attention

Summary: The request is security-sensitive because it would expose command execution, filesystem mutation, session orchestration, message delivery, and persistent cron execution outside an LLM turn.
