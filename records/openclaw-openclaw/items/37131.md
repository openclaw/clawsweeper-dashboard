## Summary

Keep open. Current main partially covers the request with Gateway HTTP/RPC and SDK direct tool invocation, but the umbrella feature still lacks direct coding-tool wiring, a cron tool-call payload, and generic slash or CLI invocation surfaces.

## What This Changes

Request direct invocation of OpenClaw tools from chat/slash commands, CLI, and cron payloads without an intervening model turn.

## Best Possible Solution

Keep this as the umbrella tracker, review or replace the coding-tool HTTP/RPC slice, keep https://github.com/openclaw/openclaw/issues/18160 for cron direct exec, and split generic slash/CLI direct invocation into owner-approved follow-ups.

## Security Review

Status: needs_attention

Summary: The requested feature broadens direct access to high-risk tools and persisted automation paths, so the remaining design needs explicit security scoping.

Concerns:

- **[medium] Preserve direct-invoke guardrails for high-risk tools:** `docs/gateway/tools-invoke-http-api.md:41`
  - body: The current HTTP docs treat /tools/invoke as full operator access and the default deny list blocks RCE and mutation-capable tools such as exec, shell, fs_write, apply_patch, sessions_spawn, cron, gateway, and nodes. New slash, CLI, cron, or coding-tool paths need equivalent auth, policy, owner, approval, and audit rules.
  - confidence: 0.9
- **[medium] Keep persisted cron execution under owner/secops policy:** `.github/CODEOWNERS:36`
  - body: CODEOWNERS assigns the cron job validator to openclaw-secops, and a stored direct exec/tool payload would change the persisted host-command trust boundary rather than just adding a convenience wrapper.
  - confidence: 0.86
