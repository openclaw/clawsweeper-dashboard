## Summary

Keep open. Current main has direct Gateway HTTP/RPC, SDK, and scoped skill-command invocation, but the umbrella request still has meaningful unimplemented scope: Gateway direct invoke still omits coding tools, cron has no direct exec/tool payload kind, and broad slash/CLI direct execution needs product and security scoping.

## What This Changes

Request direct OpenClaw tool invocation from explicit chat, slash-command, CLI, or cron inputs without an LLM inference turn.

## Best Possible Solution

Keep the shipped HTTP/RPC/SDK path, then split the remaining work into security-reviewed follow-ups for Gateway coding tools, cron direct exec/tool payloads, and any generic slash or CLI surface.

## Reproduction Assessment

Not applicable. This is a feature request, and current-main source inspection gives a high-confidence state map: HTTP/RPC/SDK and skill-command direct dispatch exist, while cron direct execution, generic slash/CLI invocation, and Gateway coding-tool availability are incomplete.

## Solution Assessment

No. The maintainable solution is decomposed, security-reviewed slices rather than one broad direct-execution surface that mixes Gateway, cron, slash/CLI UX, and high-risk tool policy.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it broadens direct access to command, file, message, and persisted automation tools.

Concerns:

- **[medium] Preserve direct-invoke guardrails for high-risk tools:** `docs/gateway/tools-invoke-http-api.md:41`
  - body: The current HTTP endpoint is documented as full operator access and default-denies RCE and mutation-capable tools; any new slash, CLI, cron, or coding-tool path needs equivalent auth, owner, policy, approval, and audit rules.
  - confidence: 0.9

  - body: CODEOWNERS assigns cron job validation to openclaw-secops, and adding stored exec/tool payloads would change the persisted host-command trust boundary rather than only adding a convenience wrapper.
  - confidence: 0.86

  - body: Exec approval docs treat Gateway-authenticated callers as trusted operators and state approvals are not a per-user auth boundary, so direct scheduled or chat-triggered exec needs explicit policy rather than assuming approvals make it safe.
