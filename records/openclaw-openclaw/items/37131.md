## Summary

Keep open. Current main has the Gateway HTTP/RPC/SDK direct-invoke foundation, but the broader request still depends on unmerged or separate work for HTTP coding tools, CLI execution, cron payloads, and any generic chat/slash surface.

## What This Changes

Request direct tool invocation from chat/slash commands, CLI, and cron without a model round trip.

## Best Possible Solution

Keep the shared Gateway `tools.invoke` foundation and resolve the remaining surfaces through scoped, security-reviewed follow-ups for HTTP coding tools, CLI execution, cron payloads, and any generic slash/chat command.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main has partial direct-invoke support but not the full slash/CLI/cron/coding-tool surface requested.

## Solution Assessment

No for a single umbrella implementation. The safer path is to reuse the shared Gateway invoke/policy boundary and review the remaining public surfaces independently.

## Security Review

Status: needs_attention

Summary: The requested feature expands direct access to high-risk tool execution surfaces and needs explicit security review before implementation.

Concerns:

- **[high] Preserve direct-invoke deny-list and owner semantics:** `src/security/dangerous-tools.ts:9`
  - body: The current default HTTP deny list treats exec, shell, spawn, filesystem mutation, session orchestration, cron, gateway, and node relay tools as high-risk surfaces; any expansion should preserve explicit opt-in and fail-closed policy filtering.
  - confidence: 0.91
- **[high] Define unattended cron execution policy first:** `src/gateway/protocol/schema/cron.ts:171`
  - body: Adding cron `toolCall` or `exec` payloads would schedule host actions outside a live agent turn, so maintainers need policy for sandboxing, approvals, cwd/env, timeout, cancellation, retries, output retention, audit, and delivery.
  - confidence: 0.88
