## Summary

Keep open. Current main has direct HTTP, RPC, and SDK tool invocation foundations, but the request remains broader than what is implemented: slash-command/direct CLI invocation, cron tool payloads, direct coding-tool exposure, and session-spawn-style automation still need scoped product and security decisions.

## What This Changes

Request direct tool invocation via chat/slash command, CLI, and cron payloads without an LLM turn.

## Best Possible Solution

Keep the shipped HTTP/RPC/SDK direct-invoke foundation, then resolve slash, CLI, cron, coding-tool, and session-spawn expansion through scoped security-reviewed follow-ups.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that HTTP/RPC/SDK direct invoke exists while slash/CLI direct invocation and direct cron tool payloads remain absent on current main.

## Solution Assessment

No for the umbrella as written: one broad direct-invoke path would cross too many trust boundaries at once. The maintainable path is scoped review of the existing follow-ups for coding tools, CLI plan execution, cron direct exec, and direct session spawn.
