## Summary

Keep open. Current main has important direct-invocation foundations, but the umbrella request is only partially covered: SDK `tools.invoke`, HTTP `/tools/invoke`, and skill-specific tool-dispatch slash commands exist, while generic coding-tool HTTP access, CLI execution, cron `toolCall` payloads, direct session spawn, and generic chat `/tool` UX remain open or in follow-up work.

## What This Changes

Request to invoke OpenClaw tools directly from chat/slash commands, CLI or cron automation without requiring an LLM turn.

## Best Possible Solution

Keep the shared Gateway invoke foundation and land the remaining direct-invocation surfaces as scoped, security-reviewed changes with clear policy, audit, and upgrade behavior.

## Solution Assessment

No, not as one umbrella implementation. The maintainable path is to continue through the shared Gateway invoke boundary and review each public surface separately for security, policy, UX, and session-state behavior.
