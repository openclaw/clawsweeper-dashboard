## Summary

Keep open. Current main now covers two major parts of the request—generic policy-enforced Gateway HTTP/RPC invocation and no-model scheduled command execution—but the requested generic CLI/slash surfaces, broader coding-tool exposure, and transcript/channel-result semantics are still not merged; active related PRs own several of those pieces and the remaining work needs security/product sequencing.

## What This Changes

Add direct invocation of named OpenClaw tools with explicit arguments from chat, CLI, and cron without an LLM round-trip.

## Best Possible Solution

Keep the shared Gateway invocation engine as the sole execution boundary, land caller surfaces and tool classes as narrow changes with explicit opt-ins and principal checks, and define one canonical rule for whether direct results are returned only, delivered, or persisted into session history.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Current-main source, docs, and live related-PR state provide a high-confidence assessment of which requested surfaces exist and which remain missing.

## Solution Assessment

No, not as one broad implementation. Reusing the shared Gateway invoke path is the best foundation, but CLI, slash commands, cron, dangerous tool classes, delivery, and transcript persistence should be separate security-reviewed decisions rather than one parallel dispatcher.

## Risks / Open Questions

- A chat slash command that directly reaches tools would need a deliberate mapping from channel identity to Gateway principal and scope; reusing `operator.write` implicitly could expose high-power tools to unintended users.
- Command execution and host filesystem mutation cannot safely follow the same authorization assumptions as low-risk plugin tools; the active work explicitly defers RCE-class tools pending stronger owner/admin enforcement.
- The issue's proposed synthetic transcript insertion and direct channel reply behavior could create misleading session history or bypass normal delivery semantics unless maintainers choose one canonical result model.
- This umbrella overlaps multiple active PRs, so implementing it independently would risk duplicate APIs, conflicting config gates, and policy drift.
