Keep this PR open for maintainer review: it targets a real delegated-session tool-policy gap, but the latest patch still has a blocking ACP allow-list regression, lacks real behavior proof, and the PR carries the protected maintainer label.

## What This Changes

The PR adds inherited session tool allow/deny state and threads it through gateway tool resolution, sessions_spawn, native subagent and ACP spawns, session patching, Swift protocol models, and regression tests.

## Best Possible Solution

Preserve the inherited effective caller policy across child sessions while mapping ACP command capabilities through a documented, test-covered contract that does not disable ACP for normal restrictive profiles.

## Reproduction Assessment

Yes. Source inspection shows current main narrows the loopback tool surface but does not forward that restriction through sessions_spawn to ACP or native subagent children; I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The policy-inheritance direction is right, but the latest ACP allow-list check compares effective OpenClaw tool ids against extra synthetic ACP names, which blocks valid ACP delegation under restrictive profiles.
