## Summary

Keep open. The direction is useful for a real delegated-session policy gap, but the patch is not merge-ready because it still drops allow-list restrictions, leaves one process-deny bypass path, and the external PR lacks real after-fix behavior proof.

## What This Changes

The PR adds an inheritedToolDeny session field and threads explicit tool deny lists through gateway loopback resolution, sessions_spawn, subagent/ACP spawning, session patching, Swift protocol models, and regression tests.

## Best Possible Solution

Carry the caller's effective delegated-session policy, including allow-list exclusions and derived execution flags, into child session state and prove it with focused tests plus redacted real runtime output before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main filters loopback native tools only for the immediate MCP tool list and does not pass that restriction through sessions_spawn to child sessions. I did not run a live gateway reproduction because this review is read-only.

## Solution Assessment

No. The patch moves in the right direction, but inheriting only explicit deny entries is not enough for OpenClaw's allow-list policies, and inherited process denies also need to affect exec background construction before tools are filtered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Propagate allow-list restrictions to children:** `src/agents/pi-tools.ts:897`
  - body: This passes only `pluginToolDenylist` into delegated sessions. Because `collectExplicitDenylist` ignores `allow` policies, a caller restricted by `tools.allow`, a profile, provider policy, group policy, or sender policy can still spawn a child that regains tools not explicitly denied. Derive the inherited policy from the caller's effective tool surface, or persist/apply an inherited allow policy as well.
  - confidence: 0.88
- **[P2] Apply inherited process denies before building exec:** `src/agents/pi-tools.ts:557-563`
  - body: The new inherited policy is applied only in the final filtering pipeline, after `allowBackground` has already been computed. If a child inherits `process` as denied while `exec` remains allowed, `exec` is still constructed with background/yield support enabled, partially bypassing the inherited process restriction. Include the inherited policy in the `process` check before creating the exec tool.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch changes the command/tool execution boundary and leaves concrete ways for delegated sessions to retain broader capability than their caller.

Concerns:

- **[high] Allow-list policy can be lost during delegation:** `src/agents/pi-tools.ts:897`
  - body: Only explicit deny lists are persisted as inheritedToolDeny, so callers constrained by allow-list policies can create children with tools outside the parent's effective surface.
  - confidence: 0.88
- **[medium] Process deny does not disable exec backgrounding:** `src/agents/pi-tools.ts:557`
  - body: Inherited `process` denial is enforced after exec is constructed, so background/yield behavior can remain enabled for a child that should not have process-style control.
  - confidence: 0.84
