## Summary

Keep this issue open. Current main still has the reported mismatch: the documented/schema-supported subagent thinking default is consumed by runtime code, but the agent-facing gateway config mutation guard only allowlists the older top-level thinking defaults and still rejects the documented subagent path before forwarding `config.patch`.

## What This Changes

Allow `gateway config.patch` to update the documented `agents.defaults.subagents.thinking` subagent thinking default, or remove that path from the public config contract.

## Best Possible Solution

Land a narrow behavior fix that allowlists `agents.defaults.subagents.thinking` and, if the per-agent documented path remains supported, `agents.list[].subagents.thinking`, with regression tests proving those paths are allowed while adjacent subagent policy fields remain protected. Keep the existing shipped docs/runtime contract rather than removing the feature, because current schema and runtime already support it.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: the reported merge patch changes `agents.defaults.subagents.thinking`, `collectChangedConfigPaths` reports that leaf, and the current allowlist lacks any matching pattern, so the guard throws the exact protected-path error before the gateway write call.

## Solution Assessment

Yes. The narrowest maintainable fix is to add exact allowlist entries and regression coverage; removing docs/schema support would be a broader product rollback of a runtime-supported subagent default.
