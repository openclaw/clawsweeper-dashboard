## Summary

Keep open. Current main still has the mismatch: the subagent thinking default is documented, schema-backed, and consumed by the spawn path, but the agent-facing gateway tool's config mutation guard only allows the older `thinkingDefault` paths and rejects `agents.defaults.subagents.thinking` before forwarding `config.patch`.

## What This Changes

Allow the agent-facing `gateway` tool's `config.patch` action to update documented subagent thinking defaults, or remove that path from the supported config surface.

## Best Possible Solution

Keep this issue open and make a narrow guard change that permits only `agents.defaults.subagents.thinking` and, if the documented per-agent path is intended, `agents.list[].subagents.thinking`, with regression tests showing those leaves are forwarded while broader subagent policy fields stay protected.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: the reported patch changes the leaf `agents.defaults.subagents.thinking`, the current allowlist does not match that leaf, and `assertGatewayConfigMutationAllowed` throws before `config.patch` is called.

## Solution Assessment

Unclear pending maintainer security review. The likely maintainable fix is a narrow allowlist addition plus tests; a broader `agents.*.subagents.*` allow would be unsafe because neighboring fields control spawn policy and resource limits.
