## Summary

Keep open. Current main still exposes `message_sending` as a delivery-level hook without `stopReason`, `hasToolCalls`, or a typed agent completion context, while upstream agent run metadata already carries stop/tool-call signals. Related PR #39761 remains open and unmerged, so this issue should stay open for maintainer API review rather than being swept closed.

## What This Changes

Expose agent stop/tool-use context to the `message_sending` plugin hook so plugins can structurally suppress pre-tool-call preamble messages.

## Best Possible Solution

Add a backward-compatible typed agent completion context on `PluginHookMessageSendingEvent`, derive it from current `EmbeddedPiRunMeta`, and thread it through both agent command delivery and normal channel reply delivery before `applyMessageSendingHook()`. Cover the payload contract with focused delivery/hook tests and document the semantics in the public plugin hook docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available by inspecting current main: the hook event type and `applyMessageSendingHook()` payload do not expose stop/tool-call metadata, even though `EmbeddedPiRunMeta` contains the upstream signals.

## Solution Assessment

Unclear as a final API shape. The requested structural signal is the right class of fix, but maintainers should choose between direct metadata fields and the broader typed `agentContext` approach already explored in PR #39761.
