## Label Justifications

- `P2`: This is a normal-priority plugin API improvement with a clear user-visible channel symptom and bounded outbound hook surface.
- `impact:message-loss`: The report is about unwanted outbound preamble messages reaching channels because hooks cannot structurally identify and suppress them.
- `impact:session-state`: The requested hook metadata depends on carrying agent completion and tool-call state from the run into outbound delivery.

## Summary

Keep open. Current main and v2026.5.18 still omit stop/tool-call completion context from `message_sending`, and the related implementation candidate broadens tool metadata without exposing the specific `stopReason` requested here while it remains open and conflicting.

## What This Changes

The issue requests adding `stopReason` and optional tool-call state to `message_sending` hook metadata so plugins can suppress tool-use preamble messages structurally.

## Best Possible Solution

Define one backward-compatible typed completion/provenance context for `message_sending` on the active hook event and durable send paths, including `stopReason` and tool-call presence, with docs, focused contract tests, and real non-streaming channel proof.

## Reproduction Assessment

Yes for the hook-contract gap: current main and v2026.5.18 show `message_sending` lacks stop/tool-call context while agent run metadata already has `stopReason` and `pendingToolCalls`. I did not live-reproduce the Discord non-streaming ordering symptom.

## Solution Assessment

Unclear as proposed. The structural signal is the right direction, but maintainers should choose a typed public completion/provenance contract rather than only ad hoc metadata fields, and then wire every active delivery path consistently.
