## Summary

Keep open. Current main and the latest release still expose `message_sending` as delivery-level hook data without `stopReason`, `hasToolCalls`, or typed agent completion context, while the related implementation PR remains open and needs maintainer API-shape review.

## What This Changes

Request to expose stop/tool-call completion context in `message_sending` hook metadata so plugins can suppress tool-use preambles structurally.

## Best Possible Solution

Define one backward-compatible typed agent/completion context on the active hook event and durable send paths, with docs, contract tests, and real non-streaming channel proof.

## Reproduction Assessment

Yes for the hook-contract gap: current main and `v2026.5.12` show the hook event and runtime payload omit stop/tool-call context while agent run metadata already carries it. I did not live-reproduce the Discord ordering symptom.

## Solution Assessment

Unclear as the final API shape. The structural signal is the right direction, but maintainers should choose between direct metadata fields and a broader typed context, then wire every active durable delivery path consistently.
