## Summary

Keep open. Current main still exposes `message_sending` as delivery-level hook data without `stopReason`, `hasToolCalls`, or a typed agent completion context, and the related implementation PR remains open with API-shape and rebase concerns.

## What This Changes

Expose stop/tool-call completion context to the `message_sending` hook so plugins can distinguish tool preambles from final replies.

## Best Possible Solution

Define one backward-compatible typed completion/provenance context on the active hook event and durable send paths, with docs and contract tests covering agent-originated sends.

## Reproduction Assessment

Yes for the hook-contract gap: current source shows the hook event and runtime payload omit stop/tool-call context while agent run metadata already carries it. I did not live-reproduce the Discord message-ordering symptom.

## Solution Assessment

Unclear as a final API shape. The structural signal is the right direction, but maintainers should choose between direct `metadata` fields and the broader typed `agentContext` approach in https://github.com/openclaw/openclaw/pull/39761.
