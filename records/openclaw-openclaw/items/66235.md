## Summary

Keep open. Current main still exposes `message_sending` as delivery-level hook data without `stopReason`, `hasToolCalls`, or a typed agent completion context, while the related implementation candidate at https://github.com/openclaw/openclaw/pull/39761 remains open rather than merged.

## What This Changes

Expose agent completion or tool-call metadata such as `stopReason` or `hasToolCalls` to `message_sending` hook handlers so plugins can distinguish pre-tool text from final replies.

## Best Possible Solution

Land a backward-compatible typed completion context on the active hook event and durable send paths, with docs and contract tests covering agent-originated sends.

## Reproduction Assessment

Yes, source-level: current main omits stop/tool-call context from the hook event and runtime payload while agent run metadata already carries those signals. I did not perform a live Discord reproduction of the non-streaming ordering symptom.

## Solution Assessment

Unclear as a final API shape. The structural signal is the right direction, but maintainers should choose between direct metadata fields and the broader typed `agentContext` approach in https://github.com/openclaw/openclaw/pull/39761.
