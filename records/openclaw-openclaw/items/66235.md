## Summary

Keep open. Current main still exposes `message_sending` as delivery-level hook data without `stopReason`, `hasToolCalls`, or a typed agent completion context, and the related implementation candidate remains open with API-scope and proof concerns.

## What This Changes

Expose agent stop/tool-call completion context, such as `stopReason` or `hasToolCalls`, to `message_sending` hooks so plugins can filter tool-use preambles structurally.

## Best Possible Solution

Define and land one backward-compatible typed completion/provenance context on the active hook event and durable send paths, with docs and contract tests covering agent-originated sends.

## Reproduction Assessment

Yes, source-level: current main omits stop/tool-call context from the hook event and runtime payload while agent run metadata already carries those signals. I did not perform a live Discord reproduction of the non-streaming ordering symptom.

## Solution Assessment

Unclear as a final API shape. The structural signal is the right direction, but maintainers should choose between direct metadata fields and the broader typed `agentContext` approach already under discussion in https://github.com/openclaw/openclaw/pull/39761.
