## Summary

Keep open: this PR targets a valid source-reproducible cron cleanup bug, but the current patch still has a capped deferred-cleanup path that can leave isolated sessions undeleted, and the external after-fix real behavior proof is still missing.

## What This Changes

The PR adds descendant-aware deferral for isolated cron direct-delivery `deleteAfterRun` cleanup, plus cron docs, a changelog entry, and dispatch regression tests.

## Best Possible Solution

Make direct-delivery cleanup lifecycle-owned so one deduplicated cleanup owner remains active until the descendant tree drains, with regression coverage for descendants that outlast the initial wait and retry window.

## Reproduction Assessment

Yes. Source inspection on current main shows structured/threaded direct delivery reaches `cleanupDirectCronSessionIfNeeded`, and that helper deletes the parent cron session without checking active descendants.

## Solution Assessment

No. The PR moves the guard into the right helper, but the capped three-retry timer can stop before long descendants finish; a drain-triggered or lifecycle-owned cleanup path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep cleanup alive until descendants drain:** `src/cron/isolated-agent/delivery-dispatch.ts:531-533`
  - body: The deferred cleanup path stops scheduling retries after `DIRECT_CRON_CLEANUP_MAX_DEFERRED_RETRIES`. If a descendant run stays active past the initial wait and those retries, delivery has already returned and no owner re-enters this helper, so `deleteAfterRun` is permanently skipped for that isolated cron session.
