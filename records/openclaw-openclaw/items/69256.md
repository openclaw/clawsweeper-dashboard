## Summary

Keep open: this is a valid cron cleanup bug and the PR moves in the right direction, but the current head can still leave isolated sessions undeleted after the capped deferral window and it lacks required real after-fix proof.

## What This Changes

The PR adds descendant-aware deferral and scheduled retry for isolated cron direct-delivery deleteAfterRun cleanup, with docs, changelog text, and dispatch regression tests.

## Best Possible Solution

Make direct-delivery cleanup lifecycle-owned so one deduplicated cleanup path remains responsible until the descendant tree drains, then deletes exactly once, with coverage for descendants that outlast the current retry window.

## Reproduction Assessment

Yes. Source inspection of current main shows structured/threaded direct delivery reaches deliverViaDirectAndCleanup and then deletes the parent session without checking active descendants; I did not run a live Feishu cron in this read-only review.

## Solution Assessment

No. The PR adds the needed descendant guard in the right cleanup helper, but the capped three-retry timer is not the narrowest maintainable fix because it can stop before descendants finish.
