## Summary

Keep open: the PR targets a real source-reproducible cron cleanup bug, but latest head can still stop retrying before long-running descendants drain and the external real-behavior proof gate is still missing.

## What This Changes

The PR adds descendant-aware `deleteAfterRun` cleanup for isolated cron direct delivery, regression tests, cron docs, and changelog text.

## Best Possible Solution

Keep this PR open and make direct-delivery cleanup lifecycle-owned so one cleanup owner remains until descendants drain or a cron cleanup owner takes over, with an exhaustion regression test and redacted real-run proof before merge.

## Reproduction Assessment

Yes, as a source-level reproduction path. On current main, structured or threaded direct delivery with `deleteAfterRun: true` reaches direct cleanup and deletes the parent session without checking active descendants.

## Solution Assessment

No. The PR moves the fix into the right helper, but the fixed three-retry timer can stop before descendants finish; a drain-triggered or lifecycle-owned cleanup path is the narrower maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep cleanup retrying until descendants drain:** `src/cron/isolated-agent/delivery-dispatch.ts:531-533`
  - body: The deferred cleanup path stops scheduling retries once `deferredCleanupRetryCount` reaches `DIRECT_CRON_CLEANUP_MAX_DEFERRED_RETRIES`. If a subagent remains active beyond the initial wait plus those three retry intervals, delivery has already returned and no later owner re-enters cleanup, so `deleteAfterRun` is permanently skipped for that isolated cron session.
