## Label Justifications

- `P1`: The PR addresses a regression that can break real cron orchestrator-worker delivery by deleting the parent session while subagents are still active.
- `merge-risk: 🚨 compatibility`: The proposed capped retry can make existing `deleteAfterRun` cron configuration stop being honored for long-running descendant workflows.

## Summary

Keep open: the regression is credible and the branch moves cleanup in the right direction, but the current PR head can still abandon `deleteAfterRun` cleanup after a fixed retry budget, lacks external real behavior proof, and is conflicting with current main.

## What This Changes

The PR adds descendant-aware waiting, a deferred cleanup retry timer, regression coverage, docs, and a changelog entry for isolated cron direct-delivery `deleteAfterRun` cleanup.

## Best Possible Solution

Use one lifecycle-owned cleanup path that defers deletion while descendants are active and remains active until the run tree drains, with regression coverage for long-running descendants and redacted real Feishu/threaded direct-delivery proof.

## Reproduction Assessment

Yes for source-level reproduction, though I did not run a live Feishu scenario. Current main routes structured/threaded direct delivery through immediate cleanup while the sibling text path already waits under the run-session descendant tree.

## Solution Assessment

No. The PR guards the right cleanup helper, but a fixed retry cap can permanently skip cleanup; a drain-triggered or lifecycle-owned cleanup owner is the narrower maintainable fix.
