## Summary

Keep open: current main still lacks the descendant guard on direct cron cleanup, and this PR targets the right cron delivery helper, but the latest branch can still abandon `deleteAfterRun` cleanup after a fixed retry budget while descendants remain active. The existing review discussion correctly asked for a durable second-stage cleanup path; the current branch only partially addresses it.

## What This Changes

This PR adds descendant-aware deferred cleanup for isolated cron direct-delivery `deleteAfterRun` sessions, with regression tests, cron docs, and a changelog entry.

## Best Possible Solution

Keep this PR open and make direct-delivery cleanup lifecycle-owned: never delete the parent cron session while active descendants remain, but keep a deduplicated cleanup path alive until the descendant run tree drains or the cron timeout/cleanup owner takes over. Add a regression case where descendants stay active beyond the current retry budget and then drain.

## Reproduction Assessment

Yes. A focused unit path is clear: structured or threaded direct delivery with `deleteAfterRun: true` and active descendants reaches the direct cleanup path on current main and deletes the parent session without the text path’s descendant guard.

## Solution Assessment

No. The PR moves the fix into the right helper, but a fixed three-retry timer can silently give up before descendants finish; a drain-triggered or lifecycle-owned cleanup path is the narrower maintainable solution.
