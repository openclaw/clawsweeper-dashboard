## Summary

Keep open: the bug is real on current main and the PR is aimed at the right cron direct-delivery path, but the latest patch is not merge-ready because the deferred cleanup retry is capped and can still skip `deleteAfterRun` permanently for long-running descendants.

## What This Changes

This PR updates cron direct-delivery cleanup to wait for active descendant subagent runs before deleting isolated `deleteAfterRun` sessions, adds regression coverage and docs, and records a changelog entry.

## Best Possible Solution

Keep the PR open and adjust the direct-delivery cleanup path so `deleteAfterRun` cleanup is retried or triggered until the descendant run tree actually drains, without deleting the parent session while active descendants still need it.

## Reproduction Assessment

Yes. A focused unit path is clear: structured or threaded direct delivery with `deleteAfterRun: true` and `countActiveDescendantRuns(params.runSessionKey) > 0` reaches the direct cleanup path on current main and deletes the parent session without the text path’s subagent guard.

## Solution Assessment

No. The PR moves the fix into the right helper, but a fixed three-retry timer can still abandon cleanup while descendants are active; the safer solution is a drain-triggered or otherwise lifecycle-owned cleanup path that does not silently give up before the run tree finishes.
