## Summary

Keep open: current main still has the reported direct-delivery cleanup bug, but this branch can still abandon cleanup after a capped retry budget, and the external PR does not include observable after-fix proof from a real setup.

## What This Changes

The branch adds a descendant check plus capped deferred retries for isolated cron direct-delivery `deleteAfterRun` cleanup, with docs, changelog, and dispatch tests.

## Best Possible Solution

Make direct-delivery cleanup lifecycle-owned so one deduplicated cleanup owner remains active until the descendant tree drains, with regression coverage for descendants that outlast the initial wait and retry window.

## Reproduction Assessment

Yes, by source inspection rather than live execution: current main routes structured/threaded direct delivery through cleanup that deletes the parent session without a descendant guard, while the text path and docs already establish descendant-aware behavior.

## Solution Assessment

No. The PR moves the guard into the right helper, but the capped retry path can stop before long descendants finish; a lifecycle-owned or drain-triggered cleanup path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep cleanup retrying until descendants drain:** `src/cron/isolated-agent/delivery-dispatch.ts:531-533`

If a descendant run stays active past the initial wait and those retries, delivery has already returned and no owner re-enters this helper, so `deleteAfterRun` is permanently skipped for that isolated cron session.
