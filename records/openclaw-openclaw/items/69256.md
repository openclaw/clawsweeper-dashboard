## Summary

Keep open: the bug is valid, but the latest patch still has a bounded cleanup retry path that can abandon `deleteAfterRun` cleanup for long-running descendants, and the external PR lacks observable after-fix proof.

## What This Changes

This PR adds descendant-aware deferral for isolated cron direct-delivery `deleteAfterRun` cleanup, plus cron docs, a changelog entry, and dispatch regression tests.

## Best Possible Solution

Make direct-delivery cleanup lifecycle-owned: defer deletion while descendants are active, keep one deduplicated cleanup owner alive until the descendant tree drains or the cron cleanup owner takes over, and cover long descendants that outlast any initial wait.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: structured or threaded direct delivery with `deleteAfterRun: true` reaches direct cleanup, and current main deletes the parent session without checking active descendants.

## Solution Assessment

No. The PR moves the fix into the right helper, but the fixed three-retry timer can stop before descendants finish; a drain-triggered or lifecycle-owned cleanup path is the narrower maintainable solution.

## Review Findings
