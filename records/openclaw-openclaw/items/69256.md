## Summary

Keep open: the reported current-main bug is source-reproducible and the PR targets the right cron direct-delivery cleanup path, but the branch still has a lifecycle bug because cleanup retries stop after a fixed budget and can leave `deleteAfterRun` sessions undeleted when descendants run longer. The external PR also still lacks visible after-fix real behavior proof.

## What This Changes

The PR adds descendant-aware isolated cron direct-delivery `deleteAfterRun` cleanup with deferred retries, regression tests, docs, and a changelog entry.

## Best Possible Solution

Make direct-delivery cleanup lifecycle-owned or drain-triggered so one deduplicated cleanup owner remains active until the descendant run tree drains, then require redacted real Feishu or threaded direct-delivery proof before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows structured/threaded direct delivery reaches cleanup that deletes the parent session without a descendant guard, while the sibling text path and docs establish descendant-aware behavior.

## Solution Assessment

No. The PR moves the guard into the right helper and uses the run session key, but the fixed retry budget can stop before descendants finish; a lifecycle-owned or drain-triggered cleanup path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep cleanup alive until descendants drain:** `src/cron/isolated-agent/delivery-dispatch.ts:531-533`
  - body: The deferred cleanup path returns once `deferredCleanupRetryCount` reaches `DIRECT_CRON_CLEANUP_MAX_DEFERRED_RETRIES`. If a descendant stays active past the initial wait and those three 30s retries, delivery has already returned and no later owner re-enters this helper, so `deleteAfterRun` is permanently skipped for that direct-delivery cron session.
