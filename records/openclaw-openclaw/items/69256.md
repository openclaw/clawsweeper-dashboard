## Summary

Keep open: this PR targets a real current-main cron cleanup bug, but the latest branch can still permanently skip `deleteAfterRun` cleanup when descendants outlast the capped retry window, and it lacks observable after-fix proof from a real setup.

## What This Changes

The PR adds descendant-aware deferral for isolated cron direct-delivery `deleteAfterRun` cleanup, plus cron docs, changelog text, and dispatch regression tests.

## Best Possible Solution

Make direct-delivery cleanup lifecycle-owned so one deduplicated cleanup owner remains active until the descendant tree drains, with regression coverage for descendants that outlast the initial wait and retry window.

## Reproduction Assessment

Yes. Source inspection of current main shows structured/threaded direct delivery reaches `deliverViaDirectAndCleanup`, then `cleanupDirectCronSessionIfNeeded` deletes the parent session without checking active descendants.

## Solution Assessment

No. The PR moves the guard into the right helper, but a capped three-retry timer can stop before long descendants finish; a lifecycle-owned or drain-triggered cleanup path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep cleanup retrying until descendants drain:** `src/cron/isolated-agent/delivery-dispatch.ts:531-533`
  - body: The deferred cleanup path stops scheduling retries after `DIRECT_CRON_CLEANUP_MAX_DEFERRED_RETRIES`. If a descendant run stays active past the initial wait and those retries, delivery has already returned and no owner re-enters this helper, so `deleteAfterRun` is permanently skipped for that isolated cron session.
