## Label Justifications

- `P1`: The issue concerns internal agent replies leaking into visible webchat and corrupting routing in real multi-agent workflows.
- `impact:security`: The report is about an internal/visible boundary where private inter-session replies can become user-visible.
- `impact:message-loss`: The announce path can misdeliver or duplicate inter-session replies instead of keeping them internal.
- `impact:session-state`: The report says target sessions can inherit webchat channel and deliveryContext state from the caller.

## Summary

Keep open: current main and v2026.6.6 still start the normal A2A announce path after waited sessions_send replies for non-parent peer sessions, and related live issues route remaining strict-internal leakage back here; maintainers need to decide the security/product contract before implementation.

## What This Changes

The issue asks OpenClaw to keep visible-webchat-originated sessions_send handoffs to internal sessions internal-only, without leaking raw replies or rebinding the target session to webchat delivery context.

## Best Possible Solution

Define the waited peer-session sessions_send delivery contract, then align the tool, A2A announce flow, tests, and docs so strict internal handoffs cannot leak raw replies or rebind internal targets while intentional peer follow-up remains explicit.

## Reproduction Assessment

Yes, source-reproducible: current main still builds an internal sessions_send run with deliver false, returns the waited reply inline, and starts the A2A announce flow for non-parent peer sessions. I did not run the full visible webchat scenario in this read-only review.

## Solution Assessment

Unclear until maintainers settle the strict-internal handoff contract. The narrowest maintainable path is a behavior decision first, then code, tests, and docs that prevent duplicate or visible announce delivery for strict internal sends without removing intentional peer follow-up delivery.
