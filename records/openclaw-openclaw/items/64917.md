## Summary

Keep open. Current main still has the reported visible/internal boundary risk: `sessions_send` uses a non-deliverable target run but seeds it with the WebChat internal channel, persists request delivery hints before the external-delivery branch, and then starts the A2A announce path that can call Gateway `send`.

## What This Changes

The issue reports that `sessions_send` from a visible WebChat-backed session can leak internal target replies through announce delivery and persist WebChat delivery context on the target session.

## Best Possible Solution

Define and enforce an internal/non-deliverable `sessions_send` boundary that does not seed WebChat delivery context or emit UI-visible/A2A announce output, while preserving documented peer-session announce delivery where intended.

## Reproduction Assessment

Yes, at source level. A WebChat-backed caller can invoke `sessions_send` to a fresh internal target and the current code path still seeds WebChat delivery hints, leaves the run visible by default, and starts A2A announce delivery.

## Solution Assessment

Unclear until maintainers confirm the intended semantics. The narrow maintainable fix would suppress WebChat delivery-context seeding and visible announce/UI output for internal or non-deliverable target runs without breaking normal peer A2A announce delivery.
