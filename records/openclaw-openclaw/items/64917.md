## Summary

Keep open. Current main still has the reported boundary problem: `sessions_send` starts its target run with `deliver: false`, but still marks the tool result as announce-capable and starts the A2A announce flow, while the internal channel constant is `webchat` and the Gateway can persist request channel hints onto the target session entry. Existing coverage only protects already-established external routes, not the fresh/internal target case described here.

## What This Changes

The issue requests that `sessions_send` from a visible WebChat-backed session to an internal session keep replies and routing internal instead of announcing output or binding the target session to WebChat.

## Best Possible Solution

Make `sessions_send` preserve strict internal boundaries for non-deliverable/internal targets: do not seed fresh target sessions with WebChat delivery context, and do not run or expose an announce path unless there is an explicit deliverable target or documented external handoff. Preserve the established-route behavior already covered for Discord/Telegram-style sessions.

## Reproduction Assessment

Yes. The issue's steps are plausible against current main from static inspection: a visible WebChat caller can invoke `sessions_send`, which uses `channel: webchat` internally, persists request channel hints, and still starts the announce flow. A focused regression test should mock a WebChat caller sending to a fresh internal session and assert no visible send/broadcast and no WebChat deliveryContext on the target.

## Solution Assessment

Unclear as a product decision, but the current implementation is not the best solution for strict internal delegation. The safer direction is a narrow code fix that makes `sessions_send` internal by default for internal/non-deliverable targets while preserving documented announce behavior for explicit external session routes; if maintainers intend announce semantics, docs and tool metadata need to say that plainly.
