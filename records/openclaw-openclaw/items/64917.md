## Summary

Keep open. Current main still has the reported boundary risk: `sessions_send` starts the target run as non-deliverable but with the WebChat internal channel, Gateway persists request delivery hints before delivery policy handling, and the A2A announce path can still send target-produced output to a visible delivery target. The related PR is open and addresses only requester-side announce routing, not the whole WebChat delivery-context and visibility boundary.

## What This Changes

The issue reports that `sessions_send` from a visible WebChat-backed session can leak internal target replies through the announce path and bind internal targets to WebChat delivery context.

## Best Possible Solution

Define and enforce a non-deliverable/internal `sessions_send` boundary that does not seed WebChat delivery context or emit UI-visible/A2A announce output for internal target runs, while preserving documented peer A2A announce delivery where maintainers intend it.

## Reproduction Assessment

Yes, at source level. A WebChat-backed caller can send to a fresh internal target and current main still seeds WebChat channel hints, persists request delivery context, defaults run visibility to visible, and starts an A2A announce path that can call Gateway `send`.

## Solution Assessment

Unclear until maintainers settle the strict-internal versus peer-announce semantics. The narrow direction is to suppress WebChat delivery-context seeding and visible announce/UI output for internal or non-deliverable target runs without breaking normal peer A2A announce delivery.
