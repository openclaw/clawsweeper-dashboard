## Summary

Keep open. Current main still has the reported WebChat/internal boundary risk: `sessions_send` starts the target with `deliver: false` on the internal WebChat channel, persists request delivery hints before delivery policy, and still leaves A2A announce/live UI paths available unless the narrow owned-child skip applies.

## What This Changes

The issue requests that `sessions_send` from a visible WebChat-backed session to an internal target keep the target run and reply internal, without WebChat delivery-context rebinding or announce leakage.

## Best Possible Solution

Define and enforce a strict internal boundary for internal or non-deliverable `sessions_send` target runs while preserving documented peer-session A2A announce delivery where it is explicitly intended.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `sessions_send` from a WebChat-backed caller to a fresh internal target and assert no visible chat/agent broadcast or Gateway `send` occurs and no WebChat delivery context is persisted.

## Solution Assessment

Unclear until maintainers confirm the intended semantics. The narrow maintainable direction is to suppress WebChat delivery-context seeding and UI-visible announce/broadcast paths for internal or non-deliverable `sessions_send` runs while leaving documented peer A2A delivery intact.

## Security Review

Status: needs_attention

Summary: This issue reports an existing visible/internal information-boundary risk in the `sessions_send` WebChat and A2A announce path.

Concerns:

- **[medium] Internal replies can surface through visible UI events:** `src/gateway/server-chat.ts:599`
  - body: Gateway chat event handling defaults unknown run contexts to Control-UI-visible and broadcasts assistant/chat events unless the run is explicitly hidden, which can expose internal target replies.
  - confidence: 0.84
- **[medium] Fresh targets can inherit WebChat delivery context:** `src/gateway/server-methods/agent.ts:1016`
  - body: The Gateway agent handler stores request delivery hints before checking `deliver: true`, so the internal WebChat channel used by `sessions_send` can become persisted on a fresh target session.
  - confidence: 0.84
- **[medium] A2A announce remains active after internal sends:** `src/agents/tools/sessions-send-tool.a2a.ts:156`
  - body: After reading the target reply, `sessions_send` still starts the A2A flow, which can generate an announce and deliver it through Gateway `send` when a target resolves.
  - confidence: 0.8
