## Label Justifications

- `P1`: The report concerns internal agent replies leaking into visible webchat and corrupting routing for real multi-agent workflows.
- `impact:security`: The issue is about an internal/visible boundary where private inter-session replies can become user-visible.

- `impact:session-state`: The report says target sessions can inherit webchat channel and deliveryContext state from the caller.

## Summary

Keep open: current main and v2026.6.6 still let normal peer `sessions_send` return a waited reply inline while reporting pending announce delivery and starting the A2A announce path. A later duplicate close also routes remaining strict-internal handoff leakage back to this issue, so this remains the canonical maintainer/security decision point rather than cleanup fodder.

## What This Changes

The issue requests strict internal-only behavior for `sessions_send` from visible webchat-backed sessions so target replies do not leak through announce delivery or bind target sessions to webchat context.

## Best Possible Solution

Define the waited peer-session `sessions_send` delivery contract, then align the tool, A2A announce flow, tests, and docs so strict internal handoffs cannot leak raw replies or rebind internal targets to visible webchat context while intentional peer follow-up behavior remains explicit.

## Reproduction Assessment

Yes, source-reproducible: current main still returns waited replies inline and then starts the normal A2A announce path for non-parent peer sessions. I did not run the full visible webchat scenario in this read-only review.

## Solution Assessment

Unclear until maintainers settle the strict-internal handoff contract. The narrowest safe path is a behavior decision first, then code and test changes that prevent duplicate or visible announce delivery for strict internal sends without accidentally removing intentional peer follow-up delivery.
