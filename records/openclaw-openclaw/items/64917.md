## Summary

Keep open: current main and v2026.6.6 still keep the normal `sessions_send` A2A announce path active after a waited inline reply, while the remaining strict-internal handoff and webchat rebinding behavior is still the canonical open discussion for this issue.

## What This Changes

The issue asks OpenClaw to keep `sessions_send` from visible webchat to an internal target fully internal, without leaking raw target replies or rebinding the target session to webchat delivery context.

## Best Possible Solution

Define the waited peer-session `sessions_send` contract, then align code, docs, and tests so internal-only handoffs cannot leak or rebind visible webchat state while any intentionally visible A2A follow-up remains explicit and structurally safe.

## Reproduction Assessment

Yes, source-reproducible: current main still returns waited `sessions_send` replies inline and starts the A2A announce path for non-parent peer sessions, with tests asserting that behavior. I did not run the full live webchat reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers settle the contract: strict internal suppression matches the security report, but current docs and tests intentionally preserve normal peer A2A follow-up delivery. The best path is a maintainer security/product decision followed by a narrow code, test, and docs update.
