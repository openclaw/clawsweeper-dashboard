## Summary

Keep open. Current main still starts the target `sessions_send` run as non-deliverable while carrying WebChat channel hints, persists request delivery context before the delivery policy check, and can still send target-produced A2A announce output to visible routes; the related open PRs are unmerged and cover competing partial directions.

## What This Changes

The issue asks for `sessions_send` from visible WebChat-backed sessions to internal sessions to stay internal and avoid rebinding target sessions to WebChat delivery context.

## Best Possible Solution

Define and enforce a non-deliverable/internal `sessions_send` boundary that does not seed WebChat delivery context or emit UI-visible A2A announce output for internal target runs, while preserving intended peer A2A announce delivery.

## Reproduction Assessment

Yes, at source level. A visible WebChat-backed caller can send to a fresh internal target; current main still seeds WebChat channel hints, persists delivery context, defaults run visibility to visible, and starts an A2A announce path that can call Gateway `send`.

## Solution Assessment

Unclear until maintainers settle the strict-internal versus peer-announce semantics. The narrow maintainable direction is to suppress WebChat delivery-context seeding and visible announce/UI output for internal or non-deliverable target runs without breaking documented peer A2A delivery.
