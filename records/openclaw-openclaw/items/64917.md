## What This Changes

The issue asks for `sessions_send` from visible webchat to internal sessions to keep replies internal and avoid rebinding target sessions to webchat delivery context.

## Best Possible Solution

A maintainer should define the waited peer-session `sessions_send` delivery contract, then align the tool, A2A announce flow, route handling, tests, and docs so strict internal handoffs cannot leak raw replies while intentional follow-up remains explicit.

Yes, at source level: current main still starts a hidden webchat-tagged nested run, returns waited replies inline, and starts the A2A announce flow for non-skipped peer sends. I did not run the full visible webchat scenario in this read-only review.

Unclear until maintainers settle the strict-internal handoff contract. The safest path is a maintainer behavior decision first, followed by focused regression coverage for strict internal handoff, normal peer follow-up, and delivery-context preservation.
