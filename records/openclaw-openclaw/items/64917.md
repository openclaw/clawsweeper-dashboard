## Summary

Keep open: current main and the latest release still combine hidden `sessions_send` execution with the normal peer A2A announce path, and related reports route the remaining strict-internal leakage/contract decision back here rather than to a safer canonical replacement.

## What This Changes

The issue asks for `sessions_send` from a visible webchat-backed session to an internal session to keep replies internal and avoid rebinding the target session to webchat delivery context.

## Best Possible Solution

Define the waited peer-session `sessions_send` delivery contract, then align the tool, A2A announce flow, tests, and docs so strict internal handoffs cannot leak raw replies or rebind internal targets while intentional peer follow-up remains explicit.

## Reproduction Assessment

Yes. Current main is source-reproducible: the tool starts a hidden webchat-tagged nested run, returns waited replies inline, and still starts the A2A announce flow for non-parent peer sessions; I did not run the full visible webchat scenario.

## Solution Assessment

Unclear until maintainers settle the strict-internal handoff contract. The narrowest maintainable path is a behavior decision first, then code, tests, and docs that prevent visible leakage without accidentally removing intentional peer follow-up delivery.
