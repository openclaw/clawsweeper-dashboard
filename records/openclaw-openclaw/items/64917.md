## Summary

Keep open: current main and v2026.6.10 still have the hidden webchat-tagged sessions_send run followed by the A2A announce path, and the open related PR only covers a narrower channel-derivation slice without settling the strict internal-delivery boundary.

## What This Changes

The issue requests that sessions_send from a visible webchat-backed caller to an internal target keep replies internal and avoid rebinding the target session to webchat.

## Best Possible Solution

Define the waited peer-session sessions_send delivery contract, then align the tool, A2A announce flow, gateway route handling, tests, and docs so strict internal handoffs cannot leak raw replies or rebind targets while intentional peer follow-up remains explicit.

## Reproduction Assessment

Yes for a source-level reproduction: current main still starts a hidden webchat-tagged nested run, returns waited replies inline, and starts the A2A announce flow for non-skipped peer sends. I did not run a live visible webchat scenario in this read-only review.

## Solution Assessment

Unclear until maintainers settle the strict-internal handoff contract. The safest path is a maintainer behavior decision first, then a focused implementation with regression coverage for strict internal handoff, normal peer follow-up, and delivery-context preservation.
