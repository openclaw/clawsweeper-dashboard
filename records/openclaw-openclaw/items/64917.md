## Summary

Keep open: current main and the latest release still preserve the source path where normal peer `sessions_send` returns a waited reply inline, marks announce delivery pending, and runs the A2A announce flow; related cleanup has also routed remaining strict-internal handoff leakage back to this issue as the canonical maintainer/security decision point.

## What This Changes

The issue asks OpenClaw to keep `sessions_send` handoffs from visible webchat to internal sessions internal-only, without leaking raw target replies or rebinding target session delivery context to webchat.

## Best Possible Solution

Define the waited peer-session `sessions_send` contract, then align the tool, A2A announce flow, session delivery-context seeding, tests, and docs so strict internal handoffs cannot leak or rebind while intentional peer-visible follow-up remains explicit.

## Reproduction Assessment

Yes, source-reproducible: current main still creates an internal webchat-tagged target run, returns waited replies inline, and starts the normal A2A announce path for non-parent peer sessions. I did not run the full visible webchat scenario in this read-only review.

## Solution Assessment

Unclear until maintainers settle the strict-internal handoff contract. The safest path is a behavior decision first, then code, tests, and docs that prevent leaks without silently removing intentional peer follow-up delivery.
