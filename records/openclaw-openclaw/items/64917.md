## Label Justifications

- `P1`: The report concerns a user-visible channel delivery boundary where internal session replies can surface in WebChat and session routing state can drift.
- `impact:security`: The issue is about internal orchestration replies crossing into a visible user-facing WebChat path.
- `impact:message-loss`: The A2A announce path can misroute or duplicate session replies through the wrong visible target.
- `impact:session-state`: The reported and source-supported failure includes target sessions inheriting WebChat delivery context.

## Summary

Keep open. Current main and v2026.5.18 still run `sessions_send` targets as `deliver: false` with the internal WebChat channel hint, can follow with A2A announce delivery, and can persist request delivery context before the delivery-policy check; related open PRs address pieces but have not landed a complete boundary fix.

## What This Changes

The issue asks for `sessions_send` from visible WebChat-backed sessions to internal sessions to remain internal, avoid raw visible announce replies, and avoid binding target sessions to WebChat delivery context.

## Best Possible Solution

Define the strict-internal versus peer-A2A `sessions_send` contract, then enforce that non-deliverable internal target runs neither seed visible WebChat delivery context nor emit visible announce output while preserving intentional peer-session announces.

## Reproduction Assessment

Yes, at source level. Current main starts the target run as non-deliverable with a WebChat channel hint, can enter an A2A announce path that calls Gateway `send`, and persists request delivery context before delivery-policy checks.

## Solution Assessment

Unclear until maintainers settle strict-internal versus peer-announce semantics. The maintainable direction is to suppress WebChat context seeding and visible announce output for internal or non-deliverable target runs without breaking documented peer A2A delivery.
