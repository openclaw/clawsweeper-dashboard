## What This Changes

The issue reports that `sessions_send` from a visible webchat-backed session to an internal session can leak target replies through the announce path and bind the target session to webchat.

## Best Possible Solution

Maintainers should define the waited peer-session `sessions_send` delivery contract, then align the tool, A2A announce flow, route handling, tests, and docs so strict internal handoffs cannot leak raw replies while intentional follow-up remains explicit.

## Reproduction Assessment

Yes, at source level: current main still starts a hidden webchat-tagged nested run, returns waited replies inline, and starts the A2A announce flow for non-skipped peer sends. I did not run the full visible webchat scenario in this read-only review.

Unclear until maintainers settle the strict-internal handoff contract. A narrow channel derivation fix helps one symptom, but the safer full solution needs focused regression coverage for strict internal handoff, normal peer follow-up, and delivery-context preservation.
