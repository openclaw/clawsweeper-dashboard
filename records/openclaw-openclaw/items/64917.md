## What This Changes

The issue asks for `sessions_send` from visible webchat to an internal session to remain internal, avoid visible raw replies, and avoid rebinding the target session to webchat delivery context.

## Best Possible Solution

Define the waited peer-session `sessions_send` delivery contract, then align the tool, A2A announce flow, gateway route handling, tests, and docs so strict internal handoffs cannot leak raw replies or rebind targets while intentional peer follow-up remains explicit.

Yes for a source-level reproduction: current main still starts a hidden webchat-tagged nested run, returns waited replies inline, and starts the A2A announce flow for non-skipped peer sends. I did not run the full visible webchat scenario in this read-only review.
