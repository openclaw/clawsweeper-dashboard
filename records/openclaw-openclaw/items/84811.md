## Label Justifications

- `P1`: A spawn-enabled agent can revive a removed agent id with default runtime permissions, making this a current security-relevant agent workflow issue.
- `impact:security`: The report concerns authorization and sandbox/tool-policy boundaries for delegated agent spawning.
- `impact:session-state`: The behavior can create or continue session state under agent ids that no longer have a corresponding configured agent entry.

## Summary

Keep open: current main still allows explicit allowlist-only sub-agent ids to appear in `agents_list` and pass `sessions_spawn`, and a recent merged hardening PR deliberately preserved that compatibility path, so this needs maintainer/security policy review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to reject or clearly distinguish sub-agent allowlist entries whose agent config has been removed from `agents.list[]`.

## Best Possible Solution

Define the registry contract for explicit `allowAgents` entries, then either fail closed for stale allowlist-only targets with migration/docs/doctor coverage or keep compatibility while adding clear `agents_list` warnings and a security audit/doctor warning.

## Reproduction Assessment

Yes, at source level. Current tests explicitly accept allowlisted-but-unconfigured `agentId` values and `agents_list` marks such ids as `configured: false`; I did not run a live gateway because this review is read-only.

## Solution Assessment

Unclear pending maintainer decision. Fail-closed registry validation is the security-clean direction, but it would reverse a recently documented compatibility choice for explicit allowlist entries.
