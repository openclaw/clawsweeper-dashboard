## Label Justifications

- `P1`: The issue concerns a security-relevant spawn boundary where a removed agent ID can still create live sessions under default policy.
- `impact:security`: The report is about authorization semantics for which agent IDs a session may spawn and whether removed agents remain usable.
- `impact:session-state`: The reported behavior can create and persist sessions under an unintended or decommissioned agent identity.

## Summary

Keep open: current main still lists explicit allowlisted-but-unconfigured agent IDs and accepts them in `sessions_spawn`, while the merged wildcard hardening intentionally preserved that compatibility path, so this needs maintainer/security judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to reject or clearly flag `allowAgents` entries whose agent configs were removed so `agents_list` and `sessions_spawn` do not treat them as normal spawn targets.

## Best Possible Solution

Define the explicit allowlist contract: either require explicit targets to exist in `agents.list[]` with a tested compatibility/deprecation path, or keep explicit orphan IDs supported while making `agents_list`, `sessions_spawn`, docs, and `doctor --fix` clearly expose and clean up orphaned entries.

## Reproduction Assessment

Yes, from source inspection and existing tests. Current main has tests that expect both `agents_list` to mark an allowlisted unconfigured ID as `configured: false` and `sessions_spawn` to accept an explicitly allowlisted unconfigured target; I did not run a live spawn because this review is read-only.

## Solution Assessment

Unclear. Registry rejection would close the reported boundary, but current main intentionally preserved explicit unconfigured IDs as a compatibility path, so maintainers should choose the contract before an automated code change.
