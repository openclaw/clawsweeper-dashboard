## Summary

Keep open. Current main still queues delivered isolated cron awareness with `trusted: false`, and the untrusted system-event paths can still force `senderIsOwner=false` before owner-only tool policy filters `cron`, `gateway`, and `nodes`; #72201 is an active unmerged audience/provenance proposal, not a shipped fix.

## What This Changes

Fix owner DM turns after isolated cron `announce` delivery so cron awareness does not strip owner-only tools from the next legitimate owner request.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for post-delivery cron awareness so the main session gets hidden context without treating the next owner DM as an untrusted turn, while leaving generic untrusted-event owner downgrades intact.

## Reproduction Assessment

Yes. The report includes a detailed live trace on `openclaw@2026.4.23`, and current main still has the source-level chain from cron awareness `trusted: false` through owner downgrade and owner-only tool filtering; I did not run a live Discord cron cycle in this read-only review.

## Solution Assessment

Unclear. Setting all owner-created cron output to trusted or weakening `ForceSenderIsOwnerFalse` globally is probably too broad; the safer direction is a narrow cron-awareness provenance or internal-audience exception like the open #72201 proposal, after maintainer security review.
