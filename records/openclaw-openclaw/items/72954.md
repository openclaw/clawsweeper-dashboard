## Summary

Keep open. Current main still has the reported source-level chain, and the related event-audience PR is still open and explicitly preserves owner downgrades for trusted:false prompt content, so there is no implemented or superseding fix to close against.

## What This Changes

The issue requests preventing cron-fired announce awareness from causing the next owner DM to lose owner-only tools such as cron, gateway, and nodes.

## Best Possible Solution

Make an explicit maintainer decision for cron-awareness provenance, then either document and preserve the one-turn downgrade as the trust boundary or add a narrow audited lane for delivered cron awareness without weakening arbitrary untrusted system-event gating.

## Reproduction Assessment

Yes, source-reproducible. Current main still queues delivered isolated cron awareness as trusted:false, maps untrusted pending or drained system events to non-owner context, and filters owner-only tools from that turn.

## Solution Assessment

Unclear. Trusting or exempting this cron-awareness event would fix the UX, but the maintainable solution needs maintainer security judgment because trusted:false is also the prompt-injection boundary for owner-only tools.
