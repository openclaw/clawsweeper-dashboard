## Summary

Keep open. Current main still contains the reported chain for explicit isolated cron announce delivery: cron awareness is queued as untrusted, untrusted queued or drained system events force owner downgrade, and owner-only tool filtering removes `cron`, `gateway`, and `nodes`; the related #72201 audience proposal is open but not present on main.

## What This Changes

The issue requests preserving owner-only tools on the next verified owner DM after an isolated cron announce delivery queues awareness text.

## Best Possible Solution

Land a narrow provenance- or audience-aware lane for delivered isolated cron awareness so it can inform a verified owner turn without exposing owner-only tools to arbitrary untrusted prompt text.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current main still has a source-level reproduction path through explicit cron awareness queueing, untrusted event inspection/draining, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. The issue’s direct `trusted: true` option may be too broad for the existing security model; a #72201-style internal-audience or provenance exception looks narrower and safer than weakening `ForceSenderIsOwnerFalse` globally.
