## Summary

Keep open. Current main still contains the reported cron-awareness-to-owner-downgrade chain, and the related open audience-field PR is not merged, so the owner-only tool flicker remains unresolved.

## What This Changes

The issue asks OpenClaw to stop isolated cron `announce` delivery awareness from stripping owner-only tools on the next verified owner DM.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for isolated cron announce awareness so owner-created cron output can inform the next verified owner turn without rendering as user-facing untrusted text, while unrelated untrusted external events continue to force non-owner tool policy.

## Reproduction Assessment

Yes. The issue includes a concrete live trace from `openclaw@2026.4.23`, and current main still has the same source-level path from cron awareness queueing through owner downgrade and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting all owner-created cron output may fix the symptom, but the safer maintainable direction is a scoped internal/audience or provenance marker rather than weakening `ForceSenderIsOwnerFalse` globally in command auth.
