## Summary

Keep open. Current main still has the reported source-level chain: isolated cron announce awareness is queued as untrusted, heartbeat turns that into `ForceSenderIsOwnerFalse`, command auth hard-downgrades owner identity, and final owner-only filtering removes `cron`, `gateway`, and `nodes`; the related #72201 audience proposal is still not present on main.

## What This Changes

The issue asks OpenClaw to stop owner-created isolated cron `announce` deliveries from hiding owner-only tools on the next verified owner DM.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for isolated cron announce awareness so owner-created cron output can inform the next verified owner turn without downgrading arbitrary untrusted system events.

## Reproduction Assessment

Yes. The report includes a concrete live trace, and current main still has a source-level reproduction path through untrusted cron awareness queueing, pending-event inspection, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting owner-created cron output may be correct for this producer, but weakening `ForceSenderIsOwnerFalse` globally is riskier than a scoped cron-awareness provenance marker or internal-audience event lane.
