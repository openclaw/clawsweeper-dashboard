## Summary

Keep open. Current main still contains the reported source-level chain from delivered isolated cron awareness through an untrusted-event owner downgrade and owner-only tool filtering, while the active related audience PR is not merged and its current direction still preserves owner downgrades for `trusted: false` prompt content.

## What This Changes

The issue asks OpenClaw to avoid stripping owner-only tools on the next owner turn after an isolated cron job delivers announce output into the owner chat.

## Best Possible Solution

Make an explicit maintainer decision for cron-awareness provenance: either keep the one-turn downgrade as the security boundary, or add a narrow trusted/provenance lane for delivered isolated cron awareness without letting arbitrary untrusted system events bypass owner-only gating.

## Reproduction Assessment

Yes, source-reproducible. Current main still queues delivered isolated cron awareness as `trusted: false`, maps untrusted pending or drained system events to a forced non-owner turn, and then filters owner-only tools from that turn.

## Solution Assessment

Unclear. The smallest UX fix would be to trust or exempt this cron-awareness event, but the safer maintainable path needs maintainer security judgment because `trusted: false` is also the prompt-injection boundary for owner-only tools.
