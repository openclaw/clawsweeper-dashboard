## Summary

Keep open. Current main still contains the reported chain: isolated cron announce awareness is mirrored into the main session as untrusted, heartbeat converts untrusted pending events into `ForceSenderIsOwnerFalse`, command auth hard-overrides owner status, and owner-only tools are filtered. The report is valid, but the fix changes trust and owner-only authorization semantics and overlaps the open #72201 event-audience proposal, so it needs maintainer direction rather than automated cleanup.

## What This Changes

The issue asks OpenClaw to stop owner-created isolated cron announce delivery from downgrading the next verified owner DM and stripping owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so output from an owner-created cron job can inform the next verified owner turn without stripping owner-only tools, while externally influenced untrusted system events continue to force non-owner tool policy. The #72201 internal-audience design is a plausible fit if maintainers accept that contract.

## Reproduction Assessment

Yes. The issue includes a concrete live trace from `openclaw@2026.4.23`, and current main has a source-level reproduction path through cron awareness queueing, untrusted event detection, owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting all owner-created cron output is plausible, but changing `command-auth.ts` so owner identity generally overrides `ForceSenderIsOwnerFalse` is not the safest fix; a cron-awareness provenance marker or internal-audience event lane is narrower.
