## Summary

Keep open. Current main still has the reported chain: isolated cron announce awareness is queued as untrusted, heartbeat downgrades the next run when untrusted events are pending, command auth hard-overrides owner identity, and owner-only policy removes `cron`, `gateway`, and `nodes`; the related #72201 event-audience proposal has not landed.

## What This Changes

Prevent isolated cron `announce` delivery awareness from stripping owner-only tools on the next verified owner DM after a cron-fired delivery.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for isolated cron announce awareness so owner-created cron output can inform the next verified owner turn without downgrading arbitrary untrusted system events.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current main still has a source-level reproduction path through untrusted cron awareness queueing, pending-event inspection, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting owner-created cron output may be correct for this producer, but broadly weakening `ForceSenderIsOwnerFalse` is riskier than a scoped cron-awareness provenance marker or internal event audience.
