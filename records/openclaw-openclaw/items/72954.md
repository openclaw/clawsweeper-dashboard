## Summary

Keep open. Current main still contains the reported source-level chain from isolated cron awareness being queued as untrusted through forced non-owner authorization and owner-only tool filtering, and the related audience PR is still open and does not settle the trust/security decision.

## What This Changes

The issue requests that owner-created isolated cron announce delivery not strip owner-only tools from the next owner DM after the delivered cron awareness event is queued.

## Best Possible Solution

Make an explicit maintainer decision for cron-awareness provenance: either keep the one-turn downgrade as the security boundary, or add a narrow audited provenance lane for delivered owner-created cron awareness while preserving downgrades for arbitrary untrusted events.

## Reproduction Assessment

Yes, source-reproducible. Current main queues delivered isolated cron awareness with `trusted: false`, maps untrusted pending or drained events to forced non-owner status, and then filters owner-only tools from that turn.

## Solution Assessment

Unclear. Trusting cron awareness or bypassing the force flag would address the UX symptom, but the maintainable fix needs a scoped provenance rule so untrusted prompt content cannot regain owner-only tools broadly.
