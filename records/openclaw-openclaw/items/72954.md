## Summary

Keep open. Current main still contains the reported trust/authorization chain: isolated cron announce awareness is mirrored into the main session as untrusted, untrusted pending or drained system events force non-owner context, and the owner-only tool policy removes `cron`, `gateway`, and `nodes` from that turn. The bug is valid, but the repair changes security-sensitive trust semantics and overlaps the open #72201 event-audience approach, so it needs maintainer direction rather than conservative cleanup closure.

## What This Changes

The issue reports that isolated cron `announce` delivery queues untrusted awareness into the main DM session, causing the next verified owner message to lose owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Land a narrow provenance- or audience-aware fix so owner-created isolated cron announce awareness can inform the next verified owner turn without exposing owner-only tools to arbitrary untrusted system events. The #72201 hidden internal-audience path is a plausible end state if maintainers accept that contract; otherwise the fix should be scoped to cron awareness provenance, not general owner-resolution override.

## Reproduction Assessment

Yes. The issue provides a live `openclaw@2026.4.23` trace, and current main has a source-level reproduction path through cron awareness queueing, untrusted event detection, owner downgrade, and owner-only tool filtering; I did not run a live Discord/Telegram cron in this read-only review.

## Solution Assessment

Unclear. Trusting all owner-created cron output is plausible, but the safer maintainable solution is likely narrower: distinguish cron awareness provenance or adopt the event-audience/internal-context approach from #72201 while preserving general untrusted-event owner downgrades.
