## Summary

Keep open. Current main still has the reported chain: isolated cron announce awareness is queued as untrusted, heartbeat converts pending untrusted events into `ForceSenderIsOwnerFalse`, command auth hard-downgrades owner identity, and owner-only tool policy removes the named tools.

## What This Changes

The issue reports that isolated cron `announce` delivery queues an untrusted main-session awareness event, causing the next verified owner DM to lose owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for isolated cron announce awareness so owner-created cron output can inform the next verified owner turn without granting owner-only tools to arbitrary untrusted system events.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current main still has the source-level path through untrusted cron awareness queueing, pending-event inspection, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting all owner-created cron output directly may be too broad; the #72201-style internal-audience or provenance exception looks safer than weakening `ForceSenderIsOwnerFalse` globally.
