## Summary

Keep open. Current main still contains the reported source-level chain: isolated cron announce awareness is queued as untrusted, heartbeat converts untrusted pending events into `ForceSenderIsOwnerFalse`, command authorization hard-downgrades owner identity, and owner-only tool policy then removes `cron`, `gateway`, and `nodes`. The closest active path is the open #72201 event-audience proposal, but closing this issue before that or an equivalent maintainer-approved trust/audience fix lands would lose a valid security-sensitive regression report.

## What This Changes

The issue reports that isolated cron `announce` output queued as an untrusted main-session system event causes the next verified owner DM to lose owner-only tools through `ForceSenderIsOwnerFalse`.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so output from an owner-created cron job can inform the next verified owner turn without stripping owner-only tools, while externally influenced untrusted system events continue to force non-owner tool policy. Reviewing and landing #72201 or an equivalent scoped event-lane fix is the best current path.

## Reproduction Assessment

Yes. The issue includes a concrete live trace from `openclaw@2026.4.23`, and current main still has a source-level reproduction path through cron awareness queueing, untrusted pending-event detection, owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting all owner-created cron output is plausible for this report, but weakening `ForceSenderIsOwnerFalse` generally is not the safest fix; a cron-awareness provenance marker or hidden internal-audience lane is narrower.
