## Summary

Keep open. Current main still contains the reported source-level chain: delivered isolated cron awareness is queued as an untrusted system event, that untrusted pending event forces non-owner context, and owner-only tools are then filtered from the turn; the related audience PR is still open and does not yet settle the trust boundary.

## What This Changes

The issue asks for cron-delivered isolated announce awareness not to strip owner-only tools from the next valid owner DM.

## Best Possible Solution

Make an explicit maintainer decision for cron-awareness provenance, then either preserve the one-turn downgrade as the security boundary or add a narrow audited lane for delivered cron awareness without weakening arbitrary untrusted system-event gating.

## Reproduction Assessment

Yes, source-reproducible. Current main still queues delivered isolated cron awareness as trusted:false, maps untrusted pending or drained system events to ForceSenderIsOwnerFalse, and then filters owner-only tools from that turn.

## Solution Assessment

Unclear. Trusting or exempting this cron-awareness event would fix the UX, but the maintainable solution needs maintainer security judgment because trusted:false is also the prompt-injection boundary for owner-only tools.
