## Summary

Keep open. Current main still has the source-level chain that queues isolated cron awareness as untrusted context, downgrades owner status when that context is consumed, and then filters owner-only tools; the related audience/provenance proposal in #72201 is still not on main.

## What This Changes

Fix isolated cron `announce` delivery so owner-created cron output does not cause the next owner DM to lose owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for cron awareness that has already been delivered to the user, while preserving global owner downgrades for arbitrary untrusted system events.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still queues cron awareness as `trusted: false`, drains it as `System (untrusted)`, forces owner status false, and then filters owner-only tools. I did not run a live Discord or Telegram replay in this read-only review.

## Solution Assessment

Unclear. The requested outcome is valid, but the safest fix needs maintainer trust-model judgment; a narrow internal-audience or provenance exception is safer than making verified owner identity bypass all untrusted-event downgrades.
