## Summary

Keep open. Current main still has the reported source-level chain: isolated cron announce awareness is queued as untrusted, heartbeat converts pending untrusted events into a forced non-owner context, command auth treats that as a hard owner override, and owner-only tool policy removes `cron`, `gateway`, and `nodes`. The active #72201 event-audience PR looks like the nearest proposed direction, but it is open and no equivalent audience/provenance lane has landed on main.

## What This Changes

Request to prevent isolated cron `announce` delivery awareness from stripping owner-only tools on the next verified owner DM after a cron-fired delivery.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for isolated cron announce awareness so output from an owner-created cron job can become hidden runtime context, or otherwise avoid downgrading a verified owner turn, while arbitrary untrusted system events continue to force non-owner tool policy. Reviewing and landing #72201 or an equivalent scoped internal-context path is the cleanest current direction.

## Reproduction Assessment

Yes. The issue includes a concrete live trace from `openclaw@2026.4.23`, and current main still has a source-level reproduction path through untrusted cron awareness queueing, pending-event inspection, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting owner-created cron output may be correct for this producer, but weakening `ForceSenderIsOwnerFalse` broadly is not the safest fix; a scoped cron-awareness provenance marker or internal event audience is narrower.
