## Summary

Keep open. Current main still has the reported trust/authorization chain: delivered isolated cron awareness is queued as untrusted, heartbeat converts pending untrusted events into a forced owner downgrade, command auth honors that as a hard override, and final tool policy removes the owner-only `cron`, `gateway`, and `nodes` tools.

## What This Changes

The issue asks for owner-created isolated cron announce output to avoid downgrading the next verified owner turn and stripping owner-only tools.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for isolated cron announce awareness so owner-created cron output can inform the next verified owner turn without granting owner-only tools to arbitrary untrusted system events.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current main still has the source-level path through untrusted cron awareness queueing, pending-event inspection, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. A scoped internal-audience/provenance marker looks safer than weakening `ForceSenderIsOwnerFalse`, but the final trust semantics need maintainer review because the same flag protects webhook and exec-event paths.
