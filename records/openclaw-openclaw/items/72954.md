## Summary

Keep open. Current main still has the reported source-level chain: delivered isolated cron awareness is queued as `trusted: false`, untrusted queued events force non-owner context, and owner-only tools are then filtered; the active audience PR is related but is still open and does not settle the trust/authorization decision.

## What This Changes

The issue asks OpenClaw to stop cron-delivery awareness events from stripping owner-only tools from the next valid owner turn.

## Best Possible Solution

Make an explicit maintainer security decision for cron-awareness provenance, then implement a narrow audited path for delivered cron awareness without weakening arbitrary untrusted system-event gating.

## Reproduction Assessment

Yes, source-reproducible. Current main still queues delivered isolated cron awareness as `trusted: false`, maps untrusted pending or drained system events to forced non-owner status, and filters owner-only tools from that turn.

## Solution Assessment

Unclear. Trusting or exempting cron-awareness events would fix the UX, but the best maintainable fix needs maintainer security judgment because the same trust flag is also the prompt-injection boundary for owner-only tools.
