## Summary

Keep open. Current main still has the reported source-level chain: isolated cron announce awareness is queued as untrusted, heartbeat turns that pending untrusted event into `ForceSenderIsOwnerFalse`, command auth hard-overrides owner identity, and owner-only policy removes `cron`, `gateway`, and `nodes`; the overlapping #72201 audience proposal is still open rather than merged.

## What This Changes

Prevent owner-created isolated cron announce awareness events from downgrading the next verified owner DM and hiding owner-only tools.

## Best Possible Solution

Add a narrow provenance- or audience-aware internal context lane for isolated cron announce awareness so delivered owner-created cron output can inform the next verified owner turn without weakening the generic untrusted-event owner downgrade.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current main still has a source-level reproduction path through untrusted cron awareness queueing, pending-event inspection, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting owner-created cron output may be correct for this producer, but broadly weakening `ForceSenderIsOwnerFalse` is riskier than a scoped cron-awareness provenance marker or the internal event-audience lane proposed in #72201.

Status: needs_attention

  - body: A repair should not generally let verified owner identity override `ForceSenderIsOwnerFalse`; otherwise externally influenced `System (untrusted)` content could be processed in the same turn as owner-only tools.   - confidence: 0.92
