## Summary

Keep open. Current main still contains the reported chain: isolated cron announce awareness is queued as untrusted, heartbeat maps untrusted pending events to an owner downgrade, command auth hard-overrides owner identity, and owner-only tool policy then removes `cron`, `gateway`, and `nodes`.

## What This Changes

Cron announce awareness from isolated owner-created jobs should not cause the next legitimate owner DM to lose owner-only tools.

## Best Possible Solution

Add a narrow provenance- or audience-aware internal awareness path for delivered isolated cron output while preserving owner downgrades for arbitrary untrusted system events.

## Reproduction Assessment

Yes. The source-level path is present on current main: cron awareness is queued as `trusted: false`, heartbeat converts that pending event into `ForceSenderIsOwnerFalse`, command auth forces `senderIsOwner=false`, and owner-only tool filtering removes the reported tools.

## Solution Assessment

Unclear. Trusting all owner-created cron output or letting owner identity always bypass `ForceSenderIsOwnerFalse` is likely too broad; a scoped internal-audience or provenance exception is the safer maintainable direction.
