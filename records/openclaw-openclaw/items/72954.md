## Summary

Keep open. Current main still has the reported chain: explicit isolated cron awareness is queued as `trusted: false`, heartbeat converts pending untrusted events into `ForceSenderIsOwnerFalse`, command auth hard-overrides owner identity, and owner-only policy removes the affected tools.

## What This Changes

The issue requests a fix for isolated cron announce awareness so owner-created cron output does not cause the next verified owner DM to lose owner-only tools.

## Best Possible Solution

Add a narrow provenance- or audience-aware lane for isolated cron announce awareness so owner-created cron output can inform the next verified owner turn without granting owner-only tools to arbitrary untrusted system events.

## Reproduction Assessment

Yes. The issue includes a concrete live trace, and current main still has the source-level path through untrusted cron awareness queueing, pending-event inspection, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. The issue’s option to trust owner-created cron output may be too broad; a #72201-style internal-audience or provenance exception looks safer than weakening `ForceSenderIsOwnerFalse` globally.
