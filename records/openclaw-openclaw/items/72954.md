## Summary

Keep open. Current main still contains the reported behavior: isolated cron announce awareness is queued into the main session as untrusted, untrusted system-event text downgrades owner status for the next run, and non-owner policy removes owner-only tools such as cron, gateway, and nodes. The related #72201 audience proposal looks relevant but is still open and not present on current main, so this should remain with maintainers rather than be closed as implemented or duplicate.

## What This Changes

Owner-created isolated cron announce output should not cause the next verified owner DM to lose owner-only cron, gateway, and node tools.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so an already-delivered owner-created cron result can inform the next verified owner turn without exposing user-facing untrusted text or stripping owner-only tools. Preserve the existing downgrade for externally influenced untrusted events; #72201 is a plausible implementation direction if maintainers accept the internal-audience contract.

## Reproduction Assessment

Yes. The issue provides a concrete live Discord trace, and current main has a source-level reproduction path from untrusted cron awareness queueing through owner downgrade and owner-only tool filtering, with tests locking key pieces of that behavior.

## Solution Assessment

Unclear. Trusting owner-created cron output is plausible for this specific awareness event, but changing command auth so owner identity generally overrides `ForceSenderIsOwnerFalse` would be too broad; an internal-audience or cron-provenance exception is the narrower maintainable fix.
