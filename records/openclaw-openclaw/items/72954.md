## What This Changes

The issue asks for owner-created isolated cron announce delivery not to leave an untrusted pending event that strips `cron`, `gateway`, and `nodes` from the next owner DM.

## Best Possible Solution

Make an explicit maintainer decision for cron-awareness provenance, then either preserve the one-turn downgrade as the security boundary or add a narrow audited lane for delivered cron awareness without weakening arbitrary untrusted system-event gating.

## Reproduction Assessment

Yes, source-reproducible. Current `main` still queues delivered isolated cron awareness as `trusted: false`, maps untrusted pending events to `ForceSenderIsOwnerFalse`, and then filters owner-only tools from that turn.

## Solution Assessment

Unclear. Trusting or exempting this cron-awareness event would fix the UX, but the maintainable solution needs maintainer security judgment because `trusted: false` is also the prompt-injection boundary for owner-only tools.
