## Summary

Keep open. Current main has narrowed the default/implicit cron delivery path so a no-`to` last-target DM no longer queues awareness, but explicit isolated cron announce delivery still queues untrusted main-session awareness that can drive owner downgrade and owner-only tool filtering; the active #72201 audience/provenance direction is not merged.

## What This Changes

The issue requests that cron-fired isolated `announce` delivery not cause the next legitimate owner DM to lose owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Land a narrow provenance or internal-audience handling for post-delivery cron awareness, preserving owner downgrades for arbitrary untrusted events and adding regression coverage for explicit and implicit delivery cases.

## Reproduction Assessment

Yes for the remaining explicit-target path: source and tests show explicit isolated cron announce awareness is queued as `trusted: false`, heartbeat maps that to `ForceSenderIsOwnerFalse`, and command auth/tool policy then strip owner-only tools. The exact no-`to` implicit path from the old live trace is partially mitigated on current main.

## Solution Assessment

Unclear. Trusting all owner-created cron output is likely too broad; the safer solution is a narrow internal-audience or provenance exception for post-delivery cron awareness, like the direction in #72201, without weakening generic untrusted-event downgrades.
