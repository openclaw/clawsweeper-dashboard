## Summary

Keep open. Current main still contains the reported source-level chain, and the remaining fix changes trust metadata and owner-only authorization semantics rather than being conservative cleanup. The open #72201 event-audience proposal appears to be the closest active path, but it needs maintainer/security review before this issue can be closed.

## What This Changes

The issue reports that isolated cron `announce` output is queued as an untrusted main-session system event, causing the next verified owner DM to lose owner-only tools through `ForceSenderIsOwnerFalse`.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so output from an owner-created cron job can inform the next verified owner turn without stripping owner-only tools, while externally influenced untrusted system events continue to force non-owner tool policy. Reviewing and landing #72201 or an equivalent scoped event-lane fix looks like the best current path.

## Reproduction Assessment

Yes. The issue includes a concrete live trace from `openclaw@2026.4.23`, and current main still has a source-level reproduction path through cron awareness queueing, untrusted pending-event detection, owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting all owner-created cron output is plausible, but generally weakening `ForceSenderIsOwnerFalse` in command auth is not the safest fix; a cron-awareness provenance marker or hidden internal-audience lane is narrower.
