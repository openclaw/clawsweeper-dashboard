## Summary

Keep open. Current main still has the reported source-level chain: isolated cron announce awareness is queued as untrusted, heartbeat turns downgrade ownership when untrusted pending events are present, command auth treats that downgrade as a hard owner override, and owner-only tool policy removes `cron`, `gateway`, and `nodes`. The nearest active path is the open #72201 event-audience proposal, but that has not landed on main and needs maintainer/security review before this issue can close.

## What This Changes

The issue asks OpenClaw to stop owner-created isolated cron `announce` delivery awareness from causing the next verified owner DM to lose owner-only tools.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so output from an owner-created cron job can inform the next verified owner turn without stripping owner-only tools, while arbitrary untrusted system events continue to force non-owner tool policy. Reviewing and landing #72201 or an equivalent scoped internal-context lane is the cleanest current direction.

## Reproduction Assessment

Yes. The issue includes a concrete live trace from `openclaw@2026.4.23`, and current main still has a source-level reproduction path through cron awareness queueing, untrusted pending-event detection, hard owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting owner-created cron output may be correct for this producer, but weakening `ForceSenderIsOwnerFalse` broadly is not the safest fix; a scoped cron-awareness provenance marker or internal event audience is narrower.
