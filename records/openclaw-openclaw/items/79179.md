## Summary

Keep open: current main still allows a numeric MCP server key to become the provider-visible prefix of a bundle MCP tool name, so the reported Moonshot/Kimi rejection has a source-backed root cause and a narrow fix path.

## What This Changes

Ensure bundle MCP tool names derived from numeric or punctuation-leading server keys start with a provider-accepted letter.

## Best Possible Solution

Add a canonical leading-letter rule in bundle MCP name sanitation, preserve deterministic collision/truncation behavior, and update regression coverage plus user-facing docs/changelog if exposed names change.

## Reproduction Assessment

Yes at source level: sanitizeServerName("12306") returns a digit-leading prefix and buildSafeToolName then exposes a digit-leading full tool name. I did not live-call Moonshot because no Moonshot/Kimi credential was visible in env or ~/.profile.

## Solution Assessment

Yes: fixing the bundle MCP server-name sanitization path is the narrowest maintainable solution because that is where config keys become provider-visible tool prefixes. A Moonshot-only transport workaround would leave invalid names for other strict providers.

## Repair Work Prompt

Fix the bundle MCP tool-name bug where a configured MCP server key such as `12306` produces provider-visible names such as `12306__query-tickets`, which Moonshot/Kimi rejects because function names must start with a letter. Keep the fix in the bundle MCP naming path, preferably `src/agents/pi-bundle-mcp-names.ts`; do not add a config option or a provider-specific transport workaround. Preserve deterministic ordering, collision suffixes, the 30-character server prefix cap, the 64-character full-name cap, and the original server/tool values used for runtime calls. Add regression tests for numeric and other non-letter-leading server keys, including materialized tool names and truncation/collision behavior if impacted. Update `docs/plugins/bundles.md` and `CHANGELOG.md` if exposed naming behavior changes.
