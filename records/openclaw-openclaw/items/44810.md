## Summary

Keep open. Current main still has the OpenClaw-visible failure mode: the ACP parent relay emits the generic interactive-input stall notice after elapsed time since assistant output, and the current ACP agent bridge still drops non-text runtime events that would let the relay distinguish prompt-submitted/no-first-output from an actual input wait.

## What This Changes

The issue requests clearer ACP/Codex diagnostics when a child session has accepted a prompt but produces no output because the gateway child environment lacks proxy or network access.

## Best Possible Solution

Land a focused ACP diagnostics patch that carries prompt-submitted and runtime-activity signals through the current ACP manager, agent command/dispatch paths, task progress, and parent relay, then classifies no-first-output-after-prompt separately from interactive waits.

## Reproduction Assessment

Yes for the OpenClaw-visible failure mode: current source still emits the exact generic interactive-wait stall notice for a no-output ACP child run. The upstream proxy-stripped Codex network failure itself remains environment-dependent and was not live-run here.

## Solution Assessment

Yes: the narrow maintainable fix is additive diagnostics/state propagation through the current ACP runtime boundary, not a broad proxy policy change or resurrecting the old closed PR diff.
