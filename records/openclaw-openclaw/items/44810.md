## Summary

Current main still matches the reported ACP/Codex no-output diagnostic gap: parent-owned ACP runs can still emit the generic 60s “may be waiting for interactive input” message, while prompt-submitted/runtime-activity/error signals are not bridged into the parent relay well enough to distinguish a network/proxy stall from a true interactive wait. The earlier implementation PR #44949 is closed unmerged, and broader issue #29195 is related but does not make this narrower proxy/no-first-output report obsolete.

## Best Possible Solution

Keep this issue open and land a focused ACP diagnostics fix: bridge prompt-submitted/runtime-activity/error state from ACP turns into parent-owned run events, update the no-output watchdog to classify prompt-sent/no-first-output separately from interactive wait, surface terminal runtime/network errors, and add redacted proxy-env/log-path diagnostics plus targeted tests. Use #44949 as prior art but keep the new patch smaller and current-main focused.
