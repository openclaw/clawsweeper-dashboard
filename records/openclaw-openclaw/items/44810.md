## Summary

Keep open. Current main still emits the misleading generic ACP parent-stream stall notice when a child run produces no assistant output, and the bridge still lacks prompt-submitted/no-first-output state to distinguish a network/proxy failure from an interactive wait.

## What This Changes

Requests that ACP/Codex child runs launched from a proxy-stripped Gateway surface a concrete network/runtime failure instead of a generic silent no-output or interactive-input stall.

## Best Possible Solution

Land a focused ACP diagnostics fix that carries prompt-submitted/runtime-activity signals through the ACP manager, agent command, auto-reply dispatch, task progress, and parent relay, then classifies no-first-output-after-prompt separately from true interactive waits.

## Reproduction Assessment

Yes for the OpenClaw-visible failure mode: source inspection shows the relay still emits the exact generic no-output/interactive-input notice when no assistant output arrives. The full proxy-stripped Codex upstream failure is environment-dependent and was not live-run here.

## Solution Assessment

Yes, the best fix is a narrow additive diagnostics/state propagation patch rather than closing this as stale or folding it into the broader ACP reliability issue. #44949 is useful prior art but should be revived as a smaller current-main change with redacted diagnostics and contract tests.
