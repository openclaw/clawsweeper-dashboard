## Summary

Keep open. Current main still emits the misleading generic ACP parent-stream stall message when a child run produces no assistant output, and the ACP bridge still does not preserve enough prompt-submitted/no-first-output state to classify the proxy/network failure mode. The related implementation PR was closed unmerged, and the broader ACP reliability issue does not supersede this concrete repro.

## What This Changes

Request clearer ACP/Codex diagnostics when a gateway-launched child session has network/proxy failure after prompt submission, instead of reporting a generic interactive-input stall.

## Best Possible Solution

Land a focused ACP diagnostics fix that preserves prompt-submitted/runtime-activity signals through the ACP manager, agent command, auto-reply dispatch, task progress, and parent relay, then classify no-first-output-after-prompt separately from true interactive waits and surface terminal runtime/network errors with redacted environment diagnostics.

## Reproduction Assessment

Yes for the OpenClaw-visible failure mode: current main can reproduce the misleading parent notice by starting an ACP parent relay and allowing no assistant output until the watchdog fires. The full proxy-stripped Codex upstream failure is environment-dependent, but the code path still emits the exact reported symptom.

## Solution Assessment

Yes, the best repair is a narrow additive diagnostics/state propagation patch rather than closing this as stale or relying on the broader ACP reliability issue. #44949 is useful prior art, but should be revived as a smaller current-main change with redaction and contract tests.
