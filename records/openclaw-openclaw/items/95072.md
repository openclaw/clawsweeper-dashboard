## Summary

Keep open for maintainer judgment: current main and the latest release intentionally keep unmanaged Windows `/restart` on the in-process fallback, and changing it to detached respawn would reverse a shipped no-orphan safety decision.

## What This Changes

The issue asks for unmanaged Windows `/restart` to create a fresh gateway process and PID instead of using the in-process restart fallback.

## Best Possible Solution

Maintainership should decide the unmanaged Windows restart contract, then either keep in-process restart with clearer status/docs or implement a Windows-safe full-process handoff with targeted tests and live Windows proof.

## Reproduction Assessment

Yes, at source level: current main returns disabled for win32 without Scheduled Task markers, and the run loop then uses the in-process restart fallback, so PID preservation follows from the code path. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No/unclear: replacing the branch with the update detached-respawn pattern would satisfy fresh-PID semantics, but it reverses a recent shipped safety decision; the best fix depends on the intended product contract.
