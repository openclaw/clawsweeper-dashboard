## Summary

Keep open: current main and latest release still use env-only Windows supervisor detection and fall back to in-process restart when the running gateway lacks Scheduled Task markers; the open closing PR is the active implementation candidate and should be reviewed rather than closing this issue early.

## What This Changes

Request to make Windows `/restart` produce a real fresh-PID gateway restart when a Scheduled Task exists but the running gateway lacks Scheduled Task environment markers.

## Best Possible Solution

Review and land a narrow Windows-safe schtasks detection/restart fix with live PID/uptime proof, or explicitly document and keep the in-process fallback if maintainers reject fresh-PID semantics for this setup.

## Reproduction Assessment

Yes, at source level: current main returns disabled for `win32` without Scheduled Task markers, and the run loop then takes the in-process restart fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

Unclear: the original detached-respawn suggestion is not the best path because the closed detached-respawn PR found orphan risk; the schtasks-probe PR is safer but still needs maintainer review and real Windows proof.
