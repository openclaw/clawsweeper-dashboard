## Summary

Keep open: current main still detects Windows Scheduled Task supervision only from environment markers, so an env-less manually started Windows gateway can take the in-process restart fallback. The open linked PR is the current candidate fix, but it is not merged and still needs maintainer review/proof before this issue can close.

## What This Changes

The issue asks for Windows `/restart` to use a real supervised/full-process restart when a Scheduled Task exists but the running gateway lacks Scheduled Task environment markers.

## Best Possible Solution

Review and, if needed, repair https://github.com/openclaw/openclaw/pull/95095 so it detects the exact Scheduled Task that the restart handoff will run, covers default/profile/custom task names, and includes real Windows proof before this issue closes.

## Reproduction Assessment

Yes, at source level: `/restart` emits SIGUSR1, current Windows supervisor detection returns null without env markers, and the run loop then takes the in-process fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No/unclear for the original detached-respawn proposal because it conflicts with the shipped no-orphan safety fallback. The better path is the open schtasks-detection candidate, but it needs exact task-name handling, maintainer review, and Windows proof before this issue is solved.
