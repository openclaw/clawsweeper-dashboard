## Summary

Keep open: current main and the latest release still keep env-less Windows `/restart` on the in-process fallback, and the open closing PR is the active implementation candidate rather than a merged fix.

## What This Changes

The issue reports that Windows gateway `/restart` can report success while staying in the same Node process when no Scheduled Task supervisor markers are present.

## Best Possible Solution

Review and land a Windows-safe Scheduled Task detection fix with exact task-name coverage and real Windows PID/uptime proof, or explicitly keep the shipped in-process fallback contract.

## Reproduction Assessment

Yes, at source level: current main returns disabled for `win32` without Scheduled Task markers, and the run loop then takes the in-process restart fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No/unclear: the original detached-respawn proposal is not the best fix because it reopens orphan-child risk. The current schtasks-probe direction is safer, but it needs exact task-name handling and live Windows proof.
