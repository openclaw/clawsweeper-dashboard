## Summary

Keep open: current main and v2026.6.8 still intentionally keep unmanaged Windows `/restart` on the in-process fallback, while the detached-respawn PR was closed after confirming the orphaned-child risk.

## What This Changes

The issue requests unmanaged Windows `/restart` to replace the gateway process so PID and uptime reset instead of falling back to an in-process restart.

## Best Possible Solution

Maintainers should decide the unmanaged Windows restart contract, then either keep the in-process fallback with clearer status/docs or implement a Windows-safe full-process handoff with focused tests and live proof.

## Reproduction Assessment

Yes, at source level: current main returns disabled for unmanaged `win32` without Scheduled Task markers, and the run loop then performs the in-process restart fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No for the originally proposed detached-respawn shape: the linked PR was closed after confirming it reintroduces the orphaned-child risk. The best fix depends on a maintainer-approved Windows-safe handoff or an explicit decision to keep in-process semantics.
