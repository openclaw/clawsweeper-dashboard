## Summary

Keep open: current main and the latest release still rely on Windows restart supervisor environment markers, and the open linked PR is the active candidate fix for the env-missing Scheduled Task case.

## What This Changes

The issue requests that Windows `/restart` use a real gateway restart path when a Scheduled Task exists but the running gateway lacks service marker environment variables.

## Best Possible Solution

Land or reject the focused linked PR that teaches supervisor detection about the Windows Scheduled Task without reviving unsafe unmanaged detached respawn.

## Reproduction Assessment

Yes from source: on win32 with no Scheduled Task/env markers, current main returns no supervisor and disables detached respawn, which drives the run loop into the in-process fallback. I did not run a live Windows Scheduled Task scenario in this read-only review.

## Solution Assessment

Unclear until the linked PR is reviewed; direct schtasks probing is the plausible owner-boundary fix, but restart fallback behavior is compatibility-sensitive. The original detached-respawn proposal is not the best path because current code intentionally avoids orphaned Windows children.
