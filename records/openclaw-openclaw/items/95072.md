## Summary

Keep open: current main and the latest release still take the in-process fallback for env-less Windows restarts, and https://github.com/openclaw/openclaw/pull/95095 is an open closing PR for the safer Scheduled Task detection path.

## What This Changes

The issue asks Windows `/restart` to use the Scheduled Task restart path when an installed task exists even if the running gateway lacks service environment markers, so PID/uptime change instead of a hot in-process restart.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/95095 with exact Scheduled Task detection and real Windows proof, then close this issue after merge.

## Reproduction Assessment

Yes, source-level: current main detects Windows Scheduled Task supervision only from env markers; with none present, ordinary restart returns disabled and the run loop takes the in-process fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

Unclear: reintroducing unmanaged detached respawn is not the best fix because it reopens orphan-child risk. The safer path is exact-task schtasks detection with profile/custom task coverage and Windows PID/uptime proof, represented by the open linked PR.
