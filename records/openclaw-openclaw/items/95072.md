## Summary

Keep open: current main and `v2026.6.10` still rely on Windows supervisor environment markers, and the open linked PR is the active candidate fix for the env-missing Scheduled Task case. Because https://github.com/openclaw/openclaw/pull/95095 uses closing syntax for this issue and is still open, this issue should remain open until that PR is reviewed, landed, or rejected.

## What This Changes

The issue requests that Windows `/restart` use the Scheduled Task restart path when a Scheduled Task exists even if the running gateway lacks supervisor environment markers.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/95095 after resolving the Scheduled Task name and Windows proof concerns; do not revive unsafe unmanaged detached respawn.

## Reproduction Assessment

Yes from source inspection: on Windows with no supervisor environment markers, current main returns no supervisor and `restartGatewayProcessWithFreshPid()` disables detached respawn, which drives the run loop into the in-process fallback. I did not run a live Windows Scheduled Task scenario in this read-only review.

## Solution Assessment

Unclear until the linked PR is reviewed; direct Scheduled Task probing is the plausible owner-boundary direction, but restart fallback behavior is compatibility-sensitive. The original detached-respawn proposal is not the best path because current code intentionally avoids orphaned Windows children.
