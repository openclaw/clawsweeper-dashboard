## Summary

Keep open: current main and the latest release still use env-marker-only Windows supervisor detection and fall back to an in-process restart for this env-less path, while an open closing PR remains the candidate fix.

## What This Changes

The issue asks OpenClaw to make Windows `/restart` use a safe Scheduled Task restart path when the gateway was started without service environment markers, instead of reporting success while PID and uptime stay unchanged.

## Best Possible Solution

Review and land a Windows-safe Scheduled Task detection fix with exact task-name coverage and live PID/uptime proof, or explicitly keep the shipped in-process fallback contract.

## Reproduction Assessment

Yes, source-level: current main returns disabled for `win32` without Scheduled Task markers, and the run loop then takes the in-process restart fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No/unclear: the original detached-respawn proposal is not the best fix because it reopens orphan-child risk. The linked schtasks-probe direction is safer, but it still needs exact task-name handling, maintainer acceptance, and live Windows proof.
