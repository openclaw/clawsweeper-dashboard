## Summary

Keep open for maintainer judgment: current main and the latest release intentionally keep unmanaged Windows `/restart` on the in-process fallback, while the requested detached respawn would reverse a recent safety decision meant to avoid orphaned gateway processes.

## What This Changes

Request `/restart` on native Windows without Scheduled Task markers to replace the gateway process with a fresh PID instead of completing as an in-process restart.

## Best Possible Solution

Decide the product contract first: either keep unmanaged `/restart` as an in-process lifecycle restart with clearer status/docs, or add a Windows-safe full-process handoff with targeted tests and real Windows proof instead of a raw detached spawn.

## Reproduction Assessment

Yes, at source level. Current main returns disabled for `win32` without Scheduled Task markers and the run loop then uses the in-process restart fallback, so PID preservation follows directly from the current code path.

## Solution Assessment

Unclear. The proposed detached spawn is plausible for fresh PID semantics, but it reverses a recent safety decision; a maintainer should choose whether the command contract is fresh-process restart or clearly documented in-process restart for unmanaged Windows.
