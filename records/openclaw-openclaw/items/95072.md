## Summary

Keep open: current main and the latest release still take the in-process fallback on Windows without Scheduled Task markers, but the requested fresh-PID detached restart reverses an explicit safety contract and needs maintainer product/upgrade judgment rather than an automatic cleanup close.

## What This Changes

The issue requests that Windows `/restart` without Scheduled Task supervision perform a fresh-PID gateway restart instead of falling back to an in-process restart.

## Best Possible Solution

Maintainers should first decide the unmanaged Windows restart contract, then either keep the in-process fallback with clearer status/UI wording or add a tested safe fresh-PID handoff that does not orphan the gateway.

## Reproduction Assessment

Yes, at source level: current main returns `disabled` for Windows without Scheduled Task markers, and the run loop then performs an in-process restart. I did not run a live Windows foreground gateway reproduction in this read-only review.

## Solution Assessment

No, not as proposed without maintainer approval: replacing the disabled branch with plain detached spawn conflicts with the current safety contract and tests. A safer solution needs an explicit Windows fresh-PID handoff design or a decision to keep in-process restart as intended.
