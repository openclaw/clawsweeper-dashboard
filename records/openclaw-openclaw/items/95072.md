## Summary

Keep open: current main and v2026.6.9 still use env-only Windows Scheduled Task detection, so env-less `/restart` remains on the in-process fallback, while https://github.com/openclaw/openclaw/pull/95095 is the active open fix candidate.

## What This Changes

The issue reports that env-less Windows `/restart` under an installed Scheduled Task falls back to in-process restart, leaving PID and uptime unchanged.

## Best Possible Solution

Review and land a Windows-safe Scheduled Task detection fix with profile/custom task-name coverage and real `/restart` then `/status` proof, or explicitly document that env-less Windows restarts are intentionally in-process.

## Reproduction Assessment

Yes, at source level: current main returns no `schtasks` supervisor for env-less Windows and then keeps ordinary restart on the in-process fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

Unclear: probing `schtasks` is safer than detached respawn, but the active PR still needs task-name alignment and live Windows proof before it is the best fix.
