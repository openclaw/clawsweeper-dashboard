## Summary

Keep open: current main still uses the in-process fallback for env-less Windows restarts, and https://github.com/openclaw/openclaw/pull/95095 is an open closing PR for the safer Scheduled Task detection path.

## What This Changes

The issue requests that Windows `/restart` produce a fresh gateway process when a Scheduled Task exists but service environment markers are missing.

## Best Possible Solution

Review and land a Windows-safe Scheduled Task detection fix with exact task-name coverage and real Windows PID/uptime proof, or explicitly keep the shipped in-process fallback contract.

## Reproduction Assessment

Yes, source-level: current main returns disabled for `win32` without Scheduled Task markers, and the run loop then takes the in-process restart fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No/unclear: the original detached-respawn proposal is not the best fix because it reopens orphan-child risk. The current schtasks-probe direction is safer, but it needs exact task-name handling and live Windows proof.
