## Summary

Keep open: current main and the latest release still take the in-process fallback for env-less Windows restarts, and https://github.com/openclaw/openclaw/pull/95095 is the open candidate fix for this issue.

## What This Changes

This issue asks for Windows `/restart` to replace the gateway process instead of reporting success while leaving PID and uptime unchanged.

## Best Possible Solution

Review or revise https://github.com/openclaw/openclaw/pull/95095 into a Windows-safe schtasks detection fix that queries the exact task name and proves PID/uptime changes, or explicitly keep the in-process fallback contract.

## Reproduction Assessment

Yes. Source-level inspection shows env-less Windows supervisor detection returns null, `restartGatewayProcessWithFreshPid()` returns disabled, and the run loop then performs the in-process fallback; I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No/unclear. The original detached-respawn proposal is not the best fix because it reopens orphan-child risk; the schtasks-probe direction in the linked PR is safer but still needs exact task-name handling and Windows proof.
