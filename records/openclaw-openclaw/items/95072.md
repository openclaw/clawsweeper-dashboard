## Summary

Keep open: current main still reproduces the env-less Windows Scheduled Task detection gap, and https://github.com/openclaw/openclaw/pull/95095 is an open closing PR for the safer schtasks-probe direction. The issue should not be closed before that PR is reviewed, proven on Windows, and either landed or rejected.

## What This Changes

The issue reports that Windows `/restart` can fall back to an in-process restart with unchanged PID/uptime when a Scheduled Task exists but the running gateway lacks Scheduled Task environment markers.

## Best Possible Solution

Review and either land the linked Windows-safe Scheduled Task detection PR with real PID/uptime proof, or explicitly keep and document the shipped in-process fallback contract.

## Reproduction Assessment

Yes, source-level: current main returns disabled for `win32` without Scheduled Task markers, and the run loop then takes the in-process restart fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

Unclear: probing `schtasks` is a safer direction than detached respawn, but the linked PR still needs exact task-name handling and real Windows proof before this issue can close.
