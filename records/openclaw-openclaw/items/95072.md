## Summary

Keep open: current main and the latest release still have env-only Windows Scheduled Task detection and the Windows no-marker `/restart` path falls back to in-process restart; the open linked PR is the active candidate fix rather than a merged implementation.

## What This Changes

The issue reports that Windows `/restart` can report success while using an in-process restart that leaves the gateway PID and uptime unchanged when Scheduled Task markers are missing.

## Best Possible Solution

Review and either land the linked Windows-safe Scheduled Task detection PR with real PID/uptime proof, or explicitly keep and document the shipped in-process fallback contract.

## Reproduction Assessment

Yes, at source level: current main returns no Windows `schtasks` supervisor without env markers, then returns `disabled` and takes the run-loop in-process restart fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

Unclear: probing `schtasks` is a safer direction than detached respawn, but the linked PR still needs exact task-name handling, maintainer review, and real Windows proof before it is the best fix.
