## What This Changes

The PR passes active config into plugin doctor contract discovery and migration paths, adds load-path doctor coverage, and also introduces `api.runtime.llm.complete` with new plugin config, SDK types, docs, and tests.

## Best Possible Solution

Yes. Current main source shows doctor contract discovery calls the plugin manifest registry without active config, while the documented plugin metadata scope includes `plugins.load.paths`, so a config-only load-path plugin contract is missed by that path.

## Solution Assessment

No for the PR as currently shaped. The doctor config pass-through is the right narrow direction, but bundling the new plugin LLM completion runtime and config policy into the same PR is not the smallest maintainable fix without explicit owner/security approval.
