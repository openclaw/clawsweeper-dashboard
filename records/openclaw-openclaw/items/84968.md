## Summary

Keep open: current main still creates one skills watcher per workspace/agent even when those watchers include the same shared managed or extra skills directory, and the reported EMFILE cascade is consistent with the current Chokidar-backed implementation.

## What This Changes

The issue asks OpenClaw to stop per-agent skills watcher fan-out on shared skills directories, or otherwise document/disable that scaling limit.

## Best Possible Solution

Implement a gateway-level skills watcher registry keyed by normalized watched root/path set, with subscriber fan-out to affected workspace snapshot versions and focused regression coverage for multiple workspaces sharing the same managed or extra skills directory.

## Reproduction Assessment

Yes for the watcher fan-out: source inspection shows each distinct workspace gets its own watcher while sharing global and extra skills roots. I did not run a live 28-agent EMFILE repro in this read-only review.

## Solution Assessment

Yes: deduplicating shared skills watchers and fanning out snapshot invalidations is the narrowest maintainable fix; raising `LimitNOFILE` or disabling `skills.load.watch` should remain a workaround, not the product behavior.

## Repair Work Prompt

Fix the skills watcher fan-out reported in https://github.com/openclaw/openclaw/issues/84968. In `src/agents/skills/refresh.ts`, replace the per-workspace-only watcher ownership with a bounded shared watcher registry for common watched roots/path sets so multiple workspaces/agents watching the same managed, personal, extra, or plugin skills directory do not create duplicate Chokidar watchers. Preserve per-workspace snapshot versioning and `watch-targets` refresh semantics: a shared event should bump every subscribed workspace that depends on that watched root, while changing one workspace's target set should unsubscribe only that workspace and close the underlying watcher only when no subscribers remain. Keep `skills.load.watch=false` behavior working. Add focused tests in `src/agents/skills/refresh.test.ts` for two different workspaces sharing `CONFIG_DIR/skills` or the same `skills.load.extraDirs` and assert Chokidar is not duplicated for the shared root, both workspaces receive refresh events, and cleanup closes watchers when the last subscriber leaves. Add a changelog fix entry because this is user-visible gateway reliability work. Do not change bundle-MCP runtime policy or session recovery behavior except for tests needed to prove the watcher fix.
