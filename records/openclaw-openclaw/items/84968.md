## Summary

Keep open: current `main` and the latest shipped release still create one Chokidar skills watcher per workspace, even though each watcher includes the same managed, personal, extra, and plugin skill roots. The reported EMFILE cascade is source-backed and not covered by the adjacent open watcher cleanup or polling PRs.

## What This Changes

The issue asks OpenClaw to stop fanning out per-agent file watchers over shared skills directories because that can exhaust file descriptors and leave sessions stuck after bundle-MCP disposal.

## Best Possible Solution

Implement a gateway-level skills watcher registry that deduplicates normalized shared watch roots and fans change events out to subscribed workspaces while preserving workspace-specific snapshot versions and filters.

## Reproduction Assessment

Yes for the watcher fan-out path: source inspection shows each workspace gets its own watcher while sharing managed, personal, extra, and plugin skill roots. I did not run a live 28-agent EMFILE repro in this read-only review.

## Solution Assessment

Yes. Deduplicating shared skills watchers with subscriber fan-out is the narrow maintainable fix; raising `LimitNOFILE` or disabling `skills.load.watch` should remain an operator workaround, not the default product behavior.
