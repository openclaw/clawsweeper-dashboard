## Summary

Keep open: this targets a real sandbox security bug, but the patch accepts newly created skill mount sources before the existing canonical-root containment check and still lacks real first-launch sandbox proof.

## What This Changes

The PR changes `src/agents/sandbox/workspace-mounts.ts` so missing workspace skill mount source directories are created recursively and immediately included as read-only mount sources.

## Best Possible Solution

Create or materialize missing skill mount source directories before returning mounts, but always re-stat and run canonical-root containment validation after creation, then cover missing-directory and symlink-parent cases with tests plus real first-launch sandbox proof.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` return false when the skills mount source is missing, which makes `resolveReadOnlyWorkspaceSkillMounts` omit that overlay. I did not run a live Docker first-launch repro in this read-only review.

## Solution Assessment

No. Creating the directory is a plausible direction, but returning true before the existing canonical containment check is not safe; the repair should validate the created path before accepting it as a mount source.
