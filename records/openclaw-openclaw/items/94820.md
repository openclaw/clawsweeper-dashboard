## Summary

Keep open: this targets a real current and shipped sandbox bug, but merging as-is is unsafe because the new mkdir path accepts a mount source before the canonical-root guard runs, and the PR still lacks real first-launch sandbox proof.

## What This Changes

The branch changes sandbox workspace skill mount resolution so missing host skill directories are created recursively and then included as read-only mount sources.

## Best Possible Solution

Create or materialize missing skill mount source directories before returning mounts, but always re-stat and run canonical-root containment validation after creation, then cover missing-directory plus symlink-parent cases with tests and real first-launch sandbox proof.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 return false when the skills mount source is missing, which makes resolveReadOnlyWorkspaceSkillMounts omit that overlay. I did not run a live Docker first-launch repro in this read-only review.

## Solution Assessment

No. Creating the directory is a plausible direction, but returning true before the existing canonical containment check is not the safest implementation; the best fix must validate the created path before accepting it as a mount source.
