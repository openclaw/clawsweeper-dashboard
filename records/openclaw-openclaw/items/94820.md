## Summary

Keep open: the PR targets a real sandbox first-launch protection gap, but the submitted patch is not merge-ready because it accepts newly created mount sources before the existing canonical-root validation and still lacks first-launch Docker/sandbox proof.

## What This Changes

The PR changes sandbox skill mount-source filtering so missing host directories are created inside `isExistingWorkspaceSkillMountSource` and then treated as read-only bind mount sources.

## Best Possible Solution

Materialize missing protected skill roots before mount resolution, or revalidate after mkdir, while preserving canonical-root and symlink-parent coverage plus redacted first-launch sandbox proof.

## Reproduction Assessment

Yes at source level: current main returns false for missing skill roots, which filters out read-only overlays before Docker and browser container creation. I did not run a live first-launch Docker repro in this read-only review.

## Solution Assessment

No. Creating missing directories is a plausible fix direction, but returning true before canonical-root validation is not the safest implementation; setup-time materialization or post-mkdir revalidation is the narrower maintainable shape.
