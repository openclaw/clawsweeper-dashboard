## Summary

Keep this PR open, but do not merge it as written. It targets a real Telegram group text-command drop for explicitly configured groups with no effective sender allowlist, but the patch also authorizes `/status` and other text commands for per-group `groupPolicy: "open"` senders that current tests and docs keep unauthorized.

## What This Changes

The PR changes Telegram inbound text-command gating so the access-group gate is skipped whenever the effective command allowlist has no entries.

## Best Possible Solution

Keep the targeted Telegram fix idea, but represent the already-approved explicit group separately from the sender allowlist. Preserve sender allowlist enforcement, `commands.allowFrom`, owner-only command checks, and the current per-group-open unauthorized-sender guard, then add regression coverage for explicit groups, wildcard groups, open groups, and configured sender allowlists.

## Reproduction Assessment

Yes. The current-main static path reproduces the reported drop for an explicitly configured group with no effective sender allowlist: group policy can allow the group, `allowForCommands.hasEntries` is false, `resolveControlCommandGate` blocks `/status`, and `resolveTelegramInboundBody` returns null. The PR regression is also covered by the existing per-group-open `/status` denial test.

## Solution Assessment

No. The patch is small, but `allowForCommands.hasEntries` is not the right switch because empty allowlists also occur in open-group configurations where current tests and docs keep text commands unauthorized. A safer fix should pass an explicit group-policy authorization fact into the command gate or add a separate authorizer for dedicated configured-group entries only.
