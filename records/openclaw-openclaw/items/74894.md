## Summary

Keep this PR open, but do not merge it as written. It targets a real Telegram group `/status` regression, yet the patch disables the early command gate for every empty effective group allow-list, including per-group open configurations that current tests and docs keep unauthorized for random senders.

## What This Changes

The PR changes Telegram inbound text-command gating so the access-group command gate is skipped when the effective command allow-list has no entries.

## Best Possible Solution

Keep the targeted Telegram fix direction, but model “this group was explicitly configured and allowed” separately from “this sender allow-list has entries.” Preserve sender allow-list enforcement, `commands.allowFrom`, owner-only checks, and the per-group-open unauthorized-sender guard, then add regression coverage for explicit groups with no sender allow-list, wildcard groups, per-group open groups, and configured sender allow-lists.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction for the linked bug: an explicitly configured Telegram group can pass group access with no effective sender allow-list, then `/status` hits the shared control-command gate with no configured authorizer and is dropped. The PR regression is also reproducible from existing coverage that expects `/status` from a random sender in a per-group open group to receive no reply.

## Solution Assessment

No. Skipping access-group enforcement whenever `allowForCommands.hasEntries` is false is too broad because empty allow-lists also occur in per-group open configurations where current docs and tests keep text commands unauthorized. The safer fix is a narrower explicit-group authorizer or equivalent policy fact, not disabling the whole gate for empty lists.
