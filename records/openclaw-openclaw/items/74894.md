## Summary

Keep this PR open, but do not merge it as written: it targets a real Telegram group `/status` no-reply regression, yet the patch broadens command authorization for open groups with no effective sender allow-list.

## What This Changes

The PR changes Telegram inbound text-command gating so access-group command enforcement is skipped when the effective command allow-list has no entries.

## Best Possible Solution

Keep the targeted Telegram fix idea, but carry an explicit dedicated-group authorization fact into the command gate instead of treating every empty sender allow-list as authorized.

## Reproduction Assessment

Yes. The source-level reproduction is a per-group `groupPolicy: "open"` setup with no effective sender allow-list and `/status` from a random sender; current tests expect no reply, while this patch would make `commandAuthorized` true.

## Solution Assessment

No. The patch conflates dedicated configured groups with open groups; the safer fix is to preserve sender/owner command auth and add a narrow authorizer only for the explicitly matched group case that caused the linked regression.
