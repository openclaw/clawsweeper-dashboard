## Summary

Keep this PR open, but do not merge it as written. The linked Telegram group `/status` regression is valid on current main, while this patch fixes it by treating every empty effective group command allow-list as authorized, which also broadens command access for per-group `groupPolicy: "open"` setups that current tests and docs still keep restricted.

## What This Changes

The PR changes Telegram inbound text-command gating so the access-group command gate is skipped when the effective command allow-list has no entries.

## Best Possible Solution

Keep the targeted Telegram fix idea, but represent the already-approved explicit group separately from the sender allow-list. Preserve sender allow-list enforcement, `commands.allowFrom`, owner-only command checks, and the current per-group-open unauthorized-sender guard, then add regression coverage for explicitly configured groups with no sender allow-list, wildcard groups, open groups, and configured sender allow-lists.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: an explicitly configured Telegram group can pass group access with no effective sender allow-list, then `/status` hits the shared control-command gate with no configured authorizer and `resolveTelegramInboundBody` returns `null`; the PR regression is covered by the existing per-group-open `/status` denial test.

## Solution Assessment

No. Skipping access-group enforcement whenever `allowForCommands.hasEntries` is false is too broad, because empty allow-lists also occur in per-group open configurations where current tests and docs keep text commands unauthorized; a safer fix should pass an explicit group-policy authorization fact or add a narrower Telegram authorizer for dedicated configured-group entries only.
