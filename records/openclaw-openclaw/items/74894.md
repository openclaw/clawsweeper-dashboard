## Summary

Keep this PR open, but do not merge it as written. The linked Telegram group `/status` regression is real on current `main`, but the patch fixes it by treating every empty effective group command allowlist as authorized, which also broadens text-command access for per-group `groupPolicy: "open"` configurations that current tests and docs still keep restricted.

## What This Changes

This PR changes Telegram inbound text-command gating so the access-group gate is skipped when the effective sender command allowlist has no entries.

## Best Possible Solution

Keep the targeted Telegram fix idea, but model the already-approved explicit group separately from the sender allowlist. Preserve sender allowlist enforcement, `commands.allowFrom`, owner-only command checks, and the current per-group-open unauthorized-sender guard, then add regression coverage for explicitly configured groups with no sender allowlist, wildcard groups, open groups, and configured sender allowlists.

## Reproduction Assessment

Yes. Current `main` has a high-confidence static reproduction: an explicitly configured Telegram group with no effective sender allowlist can pass group access, then `/status` hits `resolveControlCommandGate` with no configured authorizer and `resolveTelegramInboundBody` returns `null`. The PR regression is also covered by the existing per-group-open `/status` denial test.

## Solution Assessment

No. The patch is small, but `allowForCommands.hasEntries` is not the right switch because empty allowlists also occur in open-group configurations where current tests and docs keep text commands unauthorized. A safer fix should pass an explicit group-policy authorization fact into the command gate or add a narrower Telegram authorizer for dedicated configured-group entries only.
