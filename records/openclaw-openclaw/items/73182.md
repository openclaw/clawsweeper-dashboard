## Summary

Keep open. Current main has partial mitigations, but the billing-impacting default is still non-off for Claude 4.6 and generic reasoning-capable catalog models, cron jobs without explicit payload thinking still fall through to that policy, and the Microsoft Teams renderer still lacks the shared reasoning-payload suppression guard.

## What This Changes

The issue requests opt-in API thinking defaults for Claude/reasoning-capable models, global agent defaults and startup/doctor warnings for thinking, cron inheritance fixes, and separation of API thinking from user-visible reasoning delivery.

## Best Possible Solution

Define one explicit policy across runtime and cron: API thinking is off unless explicitly configured or required by the provider/model, per-agent and global defaults are honored consistently, reasoning visibility is a separate opt-in delivery choice, and every chat surface uses the shared reasoning-payload guard before rendering hidden reasoning.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: resolve an unconfigured Claude 4.6 or reasoning-capable catalog model, run an isolated cron agentTurn without payload thinking, or pass a reasoning-marked payload through the Teams renderer and observe the non-off or unsuppressed path.

## Solution Assessment

No. Current main is not the best final solution because it preserves non-off implicit API thinking for several reasoning-capable paths and leaves Teams without the shared reasoning suppression guard; the safer direction is a maintainer-approved provider policy plus targeted cron and Teams fixes.

## Risks / Open Questions

- Changing implicit thinking defaults may regress providers or models that genuinely require a non-off API thinking value unless provider-required behavior is modeled separately from provider-supported behavior.
- The Teams delivery fix is security-sensitive because it must suppress hidden reasoning while preserving explicitly requested reasoning surfaces where the product intentionally supports them.
- The cron fix crosses agent default resolution, payload overrides, and model-selection behavior, so maintainer policy needs to decide whether default-agent per-agent config, global defaults, or payload fields win in each path.
