## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Authenticate the extension bridge socket:** `extensions/browser/src/browser/extension-bridge-server.ts:268`
  - body: `background-utils.js` derives a relay HMAC for `/extension`, but the bridge upgrade path never reads or verifies that token and accepts any `/extension` WebSocket before allowing `agent.request` to call the node-host emitter. That means any local client that can reach the loopback port can pose as the extension and originate node-attributed turns or CDP frames; validate the token or an equivalent Origin/capability before `handleExtension` is entered.
  - confidence: 0.9
- **[P2] Do not send a fake loopback gateway token:** `extensions/chrome-extension/sidepanel.js:181-182`
  - body: When no token is configured, loopback connections replace the empty token with `test-token-for-container` and later send it as `auth.token`. Current gateway auth treats a wrong token as a mismatch and records a failure, while missing credentials are handled separately, so token-auth local gateways will fail and may be rate-limited instead of prompting the user to configure the real token.
  - confidence: 0.86
- **[P1] Keep node event emission off the broad SDK surface:** `src/plugin-sdk/gateway-runtime.ts:36-40`
  - body: `plugin-sdk/gateway-runtime` is a public plugin subpath, but this exports both `emitNodeGatewayEvent` and `registerNodeGatewayEventEmitter`, letting plugins replace or use the process-global node emitter as an authenticated node event source. The browser bridge already has an `onAgentRequest` injection point; keep registration internal or add a narrow maintainer-approved SDK contract with docs and contract coverage before exposing this security-sensitive API.
  - confidence: 0.82
