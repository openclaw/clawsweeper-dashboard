## Summary

Keep open. Current main still has a source-level path where durable managed env values are converted into `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` metadata and then removed before launchd writes the owner-only env file, and the provided GitHub context shows open closing PR #75507 already targets this exact bug.

## What This Changes

The issue asks OpenClaw to make macOS launchd service-env regeneration export the actual managed environment values, not only the `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` metadata list, so env-backed gateway auth tokens are present at startup.

## Best Possible Solution

Review and land PR #75507, or an equivalent narrow fix, so launchd keeps durable managed values available for the owner-only env file while preserving systemd behavior, avoiding plist secret serialization, adding focused regression coverage, and including a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: state-dir `.env` values enter `durableEnvironment`, are added to the install-plan environment, then `writeManagedServiceEnvKeysToEnvironment` deletes the actual managed values before launchd serializes the remaining environment entries.

## Solution Assessment

Yes. The best fix belongs in the daemon install plan to launchd service-env handoff, not gateway auth resolution, because the gateway can only resolve env-backed auth once the managed service process actually receives the token value.
