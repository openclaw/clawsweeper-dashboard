## Summary

Keep open; current main still has a slot-blind stale-plugin repair path that can remove `plugins.allow` and `plugins.entries` while leaving a fatal `plugins.slots.memory` reference behind.

## What This Changes

Ensure `openclaw doctor --fix` keeps stale plugin cleanup consistent with `plugins.slots.*` references so repaired config still validates.

## Best Possible Solution

Make stale plugin repair slot-aware and atomic: either recover the selected plugin before cleanup or remove/warn on matching stale `plugins.slots.*` references in the same repair so the resulting config validates.

## Reproduction Assessment

Yes, source-reproducible. A config with a missing plugin in `plugins.allow`, `plugins.entries`, and explicit `plugins.slots.memory` follows code that removes allow/entry refs but leaves the slot, after which validation reports `plugins.slots.memory: plugin not found`.

## Solution Assessment

Yes, the requested direction is the right repair boundary. The narrow maintainable fix is to extend the existing stale-plugin doctor repair to handle slot references consistently, rather than adding a new config mode or changing plugin install behavior.

## Repair Work Prompt

Fix openclaw/openclaw#76551 together with #76550. In current main, `openclaw doctor --fix` stale-plugin cleanup can remove a missing plugin from `plugins.allow` and `plugins.entries` while leaving `plugins.slots.memory` or other slot refs pointing at the same missing id, so `openclaw config validate` still fails. Keep the fix scoped to doctor/config recovery: update `src/commands/doctor/shared/stale-plugin-config.ts` so stale slot references are handled atomically with allow/entry cleanup, preferably by removing or warning on slot refs for the same truly missing plugin when plugin discovery is healthy, while preserving valid installed/known slot owners. Do not change package-manager install behavior from #76441. Add regression coverage for allow+entries+slots and for the validation outcome; update `docs/cli/doctor.md` and `CHANGELOG.md` if the user-facing repair behavior changes.
