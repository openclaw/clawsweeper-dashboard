## Summary

Keep open; current main still has a slot-blind stale-plugin repair path that can remove `plugins.allow` and `plugins.entries` while leaving a fatal `plugins.slots.memory` reference behind. The open sibling #76550 tracks the same recovery gap from the slot-cleanup side, so the two reports should be fixed together rather than closed pre-merge.

## What This Changes

Request that `openclaw doctor --fix` handle stale plugin ids atomically when `plugins.slots.*` still references the same missing plugin.

## Best Possible Solution

Make stale-plugin doctor repair slot-aware and atomic: clear/reset matching stale slot references in the same write, keep channel cleanup intact, and add regression coverage plus docs/changelog updates.

## Reproduction Assessment

Yes, source-reproducible. A config with the same missing plugin id in `plugins.allow`, `plugins.entries`, and explicit `plugins.slots.memory` follows code that removes allow/entry refs but leaves the slot, after which validation reports `plugins.slots.memory: plugin not found`.

## Solution Assessment

Yes, the requested repair boundary is appropriate. The narrow maintainable fix is to extend the existing stale-plugin doctor repair to handle slot references consistently, not to add a new config mode or change broad plugin install semantics.
