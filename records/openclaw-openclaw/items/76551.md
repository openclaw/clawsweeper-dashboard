## Summary

Keep open; current main still has a slot-blind stale-plugin repair path that can remove `plugins.allow` and `plugins.entries` while leaving a fatal `plugins.slots.memory` reference behind. The open sibling #76550 covers the complementary slot-cleanup gap, so these should be fixed together rather than closed pre-merge.

## What This Changes

`openclaw doctor --fix` should repair stale plugin config atomically when `plugins.slots.*` still references the plugin being removed from `plugins.allow` or `plugins.entries`.

## Best Possible Solution

Make stale plugin repair slot-aware and atomic, using the existing uninstall-style slot reset/removal contract so a repaired config validates after doctor runs.

## Reproduction Assessment

Yes, source-reproducible. A config with a missing plugin in `plugins.allow`, `plugins.entries`, and explicit `plugins.slots.memory` follows code that removes allow/entry refs but leaves the slot, after which validation reports `plugins.slots.memory: plugin not found`.

## Solution Assessment

Yes, the requested repair boundary is appropriate. The narrow maintainable fix is to extend the existing stale-plugin doctor repair to handle slot references consistently, not to change plugin install mechanics or add a new config mode.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76551 together with the related stale-slot recovery gap in https://github.com/openclaw/openclaw/issues/76550. Current main removes missing plugin ids from `plugins.allow` and `plugins.entries` in `doctor --fix` while leaving `plugins.slots.memory` pointing at the same missing plugin, so `config validate` hard-fails. Make stale plugin repair slot-aware and atomic: either preserve allow/entry refs when a slot still intentionally depends on that id, or reset/remove stale slot references in the same repair; prefer behavior consistent with plugin uninstall slot resets.
