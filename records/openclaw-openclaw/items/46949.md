## Summary

Keep open: current main still lacks the ACP pressure-reclaim path, and the PR head remains a plausible fix direction but has blocking correctness issues plus missing real behavior proof.

## What This Changes

The PR makes ACP session-limit enforcement asynchronous, releases cached oneshot handles after startup identity reconciliation, evicts dormant oneshot and stale runtimes under pressure, and adds ACP manager regression tests.

## Best Possible Solution

Land a bounded ACP manager fix that releases non-running oneshot handles and reclaims stale cached runtimes only while still at capacity, with hard probe timeouts, current capacity/idleness rechecks, focused tests, and redacted live ACP proof.

## Reproduction Assessment

Yes, source-level: current main counts `runtimeCache.size()` in `enforceConcurrentSessionLimit`, while status reads and startup identity reconciliation can leave oneshot handles cached. I did not run a live ACPX reproduction in this read-only review.

## Solution Assessment

No. The fix direction is appropriate, but this PR needs a hard status-probe timeout, capacity and idleness rechecks, and real after-fix proof before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a hard timeout for status probes:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This starts a 5s AbortController timer but awaits `getStatus` directly. The `AcpRuntime` contract only accepts an optional signal and does not require adapters to honor abort, so a hung adapter can block session initialization under capacity pressure; race the probe against a timer.
  - confidence: 0.87
- **[P2] Recheck capacity after oneshot eviction:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After dormant oneshot eviction frees a slot, this still enters stale probing. A single new session can evict an unrelated idle persistent runtime even though capacity is already available; recheck cache size before stale probing.
  - confidence: 0.84
- **[P2] Recheck capacity before clearing stale runtimes:** `src/acp/control-plane/manager.core.ts:1297`
  - body: While waiting for the candidate actor queue, another close or eviction can already bring the cache below the limit, but this still clears and closes the candidate. Revalidate pressure inside the actor lock before clearing.
  - confidence: 0.8
- **[P2] Recompute blind idle age inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288`
  - body: The blind fallback uses `candidate.idleMs` captured before waiting on `actorQueue`. If another actor-scoped operation touched the session first, a backend without `getStatus` can be evicted using stale idle age; recompute current idle age inside the lock.
  - confidence: 0.78
