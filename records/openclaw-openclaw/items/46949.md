## What This Changes

The PR makes ACP session-limit enforcement release cached one-shot runtime handles and probe or evict idle stale runtimes before failing at max concurrent sessions, with ACP manager regression tests.

## Best Possible Solution

Land a rebased ACP manager fix that frees dormant one-shot handles only while capacity remains exhausted, uses hard bounded status probes, rechecks pressure and idleness at eviction time, and includes focused tests plus redacted real ACP/acpx proof.

## Reproduction Assessment

Yes, source-level: current main can cache pending ACP runtimes and rejects new sessions solely from `runtimeCache.size()` once `acp.maxConcurrentSessions` is reached. I did not run a live ACPX pressure reproduction in this read-only review.

## Solution Assessment

No. The direction is appropriate, but this PR needs a hard `getStatus` timeout, pressure and idleness rechecks, a rebase, and real after-fix proof before it is the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a hard status-probe timeout:** `src/acp/control-plane/manager.core.ts:1282`
  - body: Under pressure this awaits `cached.runtime.getStatus` directly and only aborts through an optional signal. A runtime that ignores `signal` can hang every new session attempt at capacity, so race the probe against a real timeout before continuing limit enforcement.
  - confidence: 0.87
- **[P2] Skip stale probing once one-shot cleanup frees capacity:** `src/acp/control-plane/manager.core.ts:1105`
  - body: `evictDormantOneshotRuntimesUnderPressure` may already free a slot, but the code immediately probes stale persistent runtimes before checking `runtimeCache.size()`. A single new session can therefore evict unrelated idle context even after capacity is available.
  - confidence: 0.84
- **[P2] Recheck pressure before clearing stale candidates:** `src/acp/control-plane/manager.core.ts:1297`
  - body: While a candidate is queued or being probed, another close can bring the cache below the limit. The callback still clears and closes the candidate, causing avoidable context loss unless it revalidates capacity before clearing.
  - confidence: 0.8
- **[P2] Recompute idle age under the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-`getStatus` fallback uses `candidate.idleMs` from the initial snapshot. If another actor-scoped operation touched the session before this callback runs, the manager can evict a backend using stale idle age.
  - confidence: 0.78
