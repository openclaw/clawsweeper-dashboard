## Summary

Keep open. Current main still lacks pressure-time reclamation for dormant ACP oneshot handles, but the PR head has blocking lifecycle defects around unbounded status probes and eviction revalidation under changing capacity.

## What This Changes

The PR makes ACP concurrent-session enforcement asynchronous, adds dormant oneshot and stale-runtime pressure eviction, releases oneshot handles after startup identity reconciliation, and adds ACP manager regression tests.

## Best Possible Solution

Land or replace this PR with a current-main ACP manager fix that reclaims dormant oneshot cache entries with hard-bounded probes and capacity/idleness revalidation, while leaving broader orphan-process cleanup to #68916.

## Reproduction Assessment

Yes. Current main can be reproduced by filling `acp.maxConcurrentSessions` and opening another ACP session; the PR-specific defects can be exercised with a `getStatus` promise that ignores `signal` and with cache capacity or idleness changing while stale eviction waits in the actor queue.

## Solution Assessment

No, not as written. The direction is narrow and maintainable, but the PR needs hard timeout and revalidation guards before stale-pressure cleanup is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Add a hard timeout around status probes:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This awaits `cached.runtime.getStatus(...)` directly and relies on the adapter honoring the optional abort signal. If a backend ignores `signal`, session initialization under pressure can hang indefinitely instead of freeing a slot or failing fast, so race the probe against a real timer and handle late errors separately.
  - confidence: 0.88
- **[P2] Recheck capacity after dormant oneshot eviction:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After dormant oneshot eviction frees a slot, this still runs stale-pressure eviction before checking the cache size. A single new session can therefore evict an additional idle runtime even though pressure was already relieved.
  - confidence: 0.86
- **[P2] Stop stale eviction once capacity is relieved:** `src/acp/control-plane/manager.core.ts:1297-1298`
  - body: The actor-queue callback can run after another session has already closed and brought the cache below the configured limit. Without a size check immediately before `clear` and `close`, this path can evict a usable idle session even though the new session no longer needs a slot.
  - confidence: 0.81
- **[P2] Recompute blind idleness inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-`getStatus` fallback decides staleness from `candidate.idleMs`, which was captured before waiting for `actorQueue.run`. If another actor-scoped operation touched the runtime while this callback was queued, the PR can still evict it based on stale idle time.
  - confidence: 0.8
