## Summary

Keep open. Current main still has the ACP runtime-cache pressure problem this PR targets, but the PR head has blocking correctness issues in the new eviction path and the contributor has not attached real after-fix ACP/acpx proof.

## What This Changes

The PR makes ACP session-limit enforcement reclaim cached oneshot and stale runtime handles under pressure, closes oneshot handles after startup identity reconciliation, and adds ACP manager regression tests.

## Best Possible Solution

Land a bounded ACP manager fix that releases non-running oneshot handles and reclaims stale handles only while still at capacity, with hard probe timeouts, current eligibility rechecks, focused regression tests, and redacted live proof.

## Reproduction Assessment

Yes, source-level: current main counts `runtimeCache.size()` for `maxConcurrentSessions`, while status reads and startup identity reconciliation can leave oneshot handles cached. I did not run a live ACPX reproduction in this read-only review.

## Solution Assessment

No. The fix direction is appropriate, but the current PR should hard-timeout status probes, recheck capacity and idleness under the actor lock, and include real after-fix ACP proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a hard status-probe timeout:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This starts a 5s `AbortController` timer but awaits `getStatus` directly. The runtime contract only accepts an optional signal and does not require adapters to honor it, so a hung adapter can block session initialization under capacity pressure; race the probe against a real timeout.
  - confidence: 0.87
- **[P2] Recheck capacity after oneshot eviction:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After dormant oneshot eviction frees a slot, this still enters stale probing. A single new session can evict an unrelated idle persistent runtime even though capacity is already available; recheck cache size before stale probing.
  - confidence: 0.84
- **[P2] Recheck pressure before clearing stale runtimes:** `src/acp/control-plane/manager.core.ts:1297`
  - body: While waiting for the candidate actor queue, another close or eviction can already bring the cache below the limit, but this still clears and closes the candidate. Revalidate pressure inside the actor lock before clearing.
  - confidence: 0.8
- **[P2] Recompute blind idle age inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288`
  - body: The blind fallback uses `candidate.idleMs` captured before waiting on `actorQueue`. If another actor-scoped operation touched the session first, a backend without `getStatus` can be evicted using stale idle age; recompute current idle age inside the lock.
  - confidence: 0.78
