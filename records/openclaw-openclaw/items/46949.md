## Summary

Keep open. The PR targets a real source-proven ACP runtime-cache leak, but the current head still has correctness blockers and lacks required real behavior proof.

## What This Changes

The PR makes ACP session-limit enforcement asynchronous so it can evict dormant oneshot and stale cached runtimes under pressure, closes oneshot handles after startup identity reconciliation, and adds ACP manager tests.

## Best Possible Solution

Land a bounded ACP manager fix that releases non-running oneshot handles, evicts stale runtimes only while still at capacity, uses hard probe timeouts, and includes focused tests plus redacted ACP/acpx proof.

## Reproduction Assessment

Yes, source-level. Current main counts cached runtime handles for maxConcurrentSessions, and startup reconciliation can ensure a pending oneshot runtime handle without a oneshot-specific release path; I did not run a live ACPX pressure reproduction in this read-only review.

## Solution Assessment

No. The fix direction is appropriate, but the current PR head should add hard status-probe timeouts, recheck capacity and idleness under the actor lock, avoid extra evictions once pressure is relieved, and include real after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a hard status-probe timeout:** `src/acp/control-plane/manager.core.ts:1282`
  - body: Under pressure this awaits cached.runtime.getStatus directly and only aborts through an optional signal. Since AcpRuntime.getStatus is not required to honor signal, an adapter that hangs can stall every new session attempt at capacity; race the probe against a real timeout so the manager always progresses.
  - confidence: 0.87
- **[P2] Skip stale probing after oneshot eviction frees capacity:** `src/acp/control-plane/manager.core.ts:1105`
  - body: evictDormantOneshotRuntimesUnderPressure may already free a slot, but this immediately probes stale persistent runtimes before checking runtimeCache.size(). A single new session can evict an unrelated idle persistent runtime even though capacity is already available.
  - confidence: 0.84
- **[P2] Recheck pressure before clearing stale candidates:** `src/acp/control-plane/manager.core.ts:1297`
  - body: While this candidate waits in actorQueue.run, another close or eviction can bring the cache below the limit. The callback still clears and closes the candidate, causing unnecessary context loss; revalidate that the manager is still at capacity before clearing.
  - confidence: 0.8
- **[P2] Recompute idle age under the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-getStatus fallback uses candidate.idleMs from the snapshot taken before the actor lock. If another actor-scoped operation touched the session first, this can evict a no-status backend using stale idle age; recompute current idle time before treating it as stale.
  - confidence: 0.78
