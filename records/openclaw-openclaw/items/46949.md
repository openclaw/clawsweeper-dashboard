## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a hard status-probe timeout:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This awaits getStatus directly and only asks the runtime to abort via signal. If a backend ignores that signal, every new session attempt at capacity can hang indefinitely, so race the probe against a real timeout before continuing limit enforcement.
  - confidence: 0.87
- **[P2] Skip stale probing after oneshot cleanup frees capacity:** `src/acp/control-plane/manager.core.ts:1105`
  - body: Oneshot cleanup may already bring the cache below maxConcurrentSessions, but the code still probes stale persistent runtimes. A single new session can evict unrelated idle context even after capacity is available.
  - confidence: 0.84
- **[P2] Recheck pressure before clearing stale candidates:** `src/acp/control-plane/manager.core.ts:1297`
  - body: While the candidate waits in actorQueue.run, another close or eviction can bring the cache below the limit. The callback still clears and closes the candidate, causing avoidable context loss unless capacity is revalidated before clearing.
  - confidence: 0.8
- **[P2] Recompute idle age under the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-getStatus fallback uses candidate.idleMs from the pre-lock snapshot. If another actor-scoped operation touched the session first, this can evict a no-status backend using stale idle age.
  - confidence: 0.78
