## Summary

Keep open. Current main still has the ACP runtime-cache counting path this PR targets, but the proposed pressure-eviction code needs correctness fixes and contributor real behavior proof before merge.

## What This Changes

This PR makes ACP concurrent-session enforcement asynchronously evict dormant oneshot and stale cached runtime handles, with ACP manager regression tests.

## Best Possible Solution

Land a bounded ACP manager fix that releases non-running oneshot handles and reclaims stale cached runtimes only while still at capacity, with focused regression tests and redacted live ACP proof; keep the broader reset/process cleanup tracked at https://github.com/openclaw/openclaw/issues/68916.

## Reproduction Assessment

Yes, source-level: current main can cache oneshot runtime handles through status/startup paths, and maxConcurrentSessions counts runtimeCache.size. I did not execute a live ACPX reproduction in this read-only review.

## Solution Assessment

No. The fix direction is appropriate, but the current PR should add a real timeout for status probes and recheck capacity/idleness before evicting under pressure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a real timeout for status probes:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This path starts a 5s AbortController timer but still awaits getStatus directly. If an ACP runtime accepts the optional signal but does not observe it, enforceConcurrentSessionLimit can hang indefinitely under capacity pressure. Race the probe against a timer so session creation always progresses.
  - confidence: 0.87
- **[P2] Skip stale probing once oneshot eviction frees capacity:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After evicting dormant oneshot handles, the code immediately probes stale runtimes even when the cache is already below the configured limit. That can evict an unrelated idle persistent session during one new-session request, causing avoidable context loss. Recheck runtimeCache.size() against the limit before calling stale-pressure eviction.
  - confidence: 0.82
- **[P2] Recompute blind idle age inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288`
  - body: The no-getStatus fallback uses candidate.idleMs from the initial snapshot after waiting for actorQueue.run. A queued actor operation can touch the cache before this callback runs, but the stale snapshot can still evict the now-recent handle. Re-read lastTouchedAt/current idle age inside the actor callback before marking it stale.
  - confidence: 0.78
