## Summary

Keep this PR open. Current main still has the ACP max-session cache pressure behavior the PR is trying to address, but the PR head has bounded correctness blockers around status-probe timeouts and over-eviction, so it should be repaired on the PR branch or replaced with a narrow current-main patch rather than merged as-is.

## What This Changes

The PR updates the ACP session manager to release cached oneshot runtime handles after startup identity reconciliation, reclaim dormant or stale cached runtimes under max-concurrent-session pressure, and add ACP manager regression tests.

## Best Possible Solution

Land a current-main ACP manager lifecycle fix that reclaims dormant oneshot cache entries without hanging session initialization or evicting usable runtime context, while leaving the broader orphan-process cleanup tracked separately.

## Reproduction Assessment

Yes. Current main has a focused manager-level proof by filling `acp.maxConcurrentSessions` and opening another ACP session; the PR-specific defects can be reproduced with a `getStatus` promise that ignores `signal` and with cache capacity or idleness changing while stale eviction waits in the actor queue.

## Solution Assessment

No, not as written. The design direction is maintainable, but the patch needs hard timeout and revalidation guards so pressure cleanup cannot hang initialization or evict an idle session after pressure has already been relieved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Add a hard timeout around status probes:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This awaits `cached.runtime.getStatus(...)` directly and relies on the adapter honoring the optional abort signal. If a backend ignores `signal`, any session initialization that reaches this pressure path can hang indefinitely instead of freeing a slot or failing fast, so race the probe against a real timer and observe late errors separately.
  - confidence: 0.88
- **[P2] Recheck capacity after dormant oneshot eviction:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After dormant oneshot eviction frees a slot, this still runs stale-pressure eviction before checking the cache size. A single new session can therefore evict an additional idle runtime even though pressure was already relieved.
  - confidence: 0.86
- **[P2] Stop stale eviction once capacity is relieved:** `src/acp/control-plane/manager.core.ts:1297-1298`
  - body: The actor-queue callback can run after another session has already closed and brought the cache below the configured limit. Without a size check immediately before `clear`/`close`, this path can evict a usable idle session even though the new session no longer needs a slot.
  - confidence: 0.81
- **[P2] Recompute blind idleness inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-`getStatus` fallback decides staleness from `candidate.idleMs`, which was captured before waiting for `actorQueue.run`. If another actor-scoped operation touched the runtime while this callback was queued, the PR can still evict it based on stale idle time.
  - confidence: 0.8
