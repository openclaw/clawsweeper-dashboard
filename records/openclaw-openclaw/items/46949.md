## Summary

Keep open. Current main still lacks the dormant oneshot pressure-eviction behavior, but this PR cannot merge as-is because the stale-pressure path can hang on an adapter that ignores abort signals, can over-evict sessions after pressure is already relieved, and lacks after-fix real ACP/acpx behavior proof.

## What This Changes

The branch updates the ACP session manager to release reconciled or dormant oneshot runtimes under max-session pressure and adds manager regression tests.

## Best Possible Solution

Land a rebased manager fix that releases dormant oneshot handles while bounding status probes and revalidating pressure, idleness, and actor state immediately before clearing any cached runtime.

## Reproduction Assessment

Yes, for the source-level manager cache path: current main counts cached runtimes against acp.maxConcurrentSessions and does not release startup-reconciled oneshot handles, so a seeded dormant oneshot cache at capacity can still block a new session. I did not run a live ACP/acpx process reproduction.

## Solution Assessment

No, not as submitted. The manager-side fix direction is appropriate, but the stale-pressure implementation needs hard timeouts and final capacity/idleness rechecks before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bound status probes with a hard timeout:** `src/acp/control-plane/manager.core.ts:1282`
  - body: The pressure probe awaits `cached.runtime.getStatus(...)` directly and only aborts via `AbortController`. Since the AcpRuntime contract does not require adapters to honor `signal`, a stuck status call can hang every session creation that hits capacity; race the probe against a manager-owned timeout so this path always progresses.
  - confidence: 0.88
- **[P2] Stop after oneshot eviction relieves capacity:** `src/acp/control-plane/manager.core.ts:1105`
  - body: `enforceConcurrentSessionLimit` always runs stale-runtime eviction after the dormant oneshot pass. If the oneshot pass already frees a slot, starting one new session can still evict an unrelated idle session; recheck `runtimeCache.size()` before calling the stale-pressure path.
  - confidence: 0.84
- **[P2] Recompute blind-idle eligibility before eviction:** `src/acp/control-plane/manager.core.ts:1288`
  - body: For runtimes without `getStatus`, the stale decision uses `candidate.idleMs` from the initial snapshot after waiting for the actor queue. If queued actor work touched the session first, this can evict a no-longer-idle runtime; recompute current idle time inside the callback before marking it stale.
  - confidence: 0.82
- **[P2] Recheck capacity before clearing stale candidates:** `src/acp/control-plane/manager.core.ts:1297`
  - body: While this callback waits behind other actor-scoped work, another session can close and bring the cache below the limit. The code still clears and closes the candidate, causing avoidable context loss under load; check capacity again immediately before `runtimeCache.clear`.
  - confidence: 0.8
