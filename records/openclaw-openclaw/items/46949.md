## Summary

Keep open. Current main still has the ACP runtime-cache pressure problem, and this PR is a useful fix direction, but the branch still has blocking eviction correctness issues, is conflicting, and lacks required after-fix real behavior proof.

## What This Changes

The PR changes ACP session-limit enforcement to release startup-reconciled oneshot handles, evict dormant oneshot and stale runtime-cache entries under pressure, and add ACP manager regression tests.

## Best Possible Solution

Land a rebased ACP manager fix that releases non-running oneshot handles and evicts stale cached runtimes only while capacity is still exhausted, with hard status-probe timeouts, focused tests, and redacted ACP/acpx proof.

## Reproduction Assessment

Yes, source-level. Current main counts runtimeCache.size() for maxConcurrentSessions and startup reconciliation can ensure/reconcile pending oneshot handles without releasing them; I did not run a live ACPX pressure reproduction in this read-only review.

## Solution Assessment

No. The fix direction is appropriate, but PR head should add hard status-probe timeouts, recheck capacity and idleness under the actor lock, avoid extra evictions once pressure is relieved, and include real after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a hard status-probe timeout:** `src/acp/control-plane/manager.core.ts:1282`
  - body: Under pressure this awaits cached.runtime.getStatus directly and only aborts through an optional signal. A runtime that ignores signal can hang every new session attempt at capacity, so race the probe against a real timeout before continuing limit enforcement.
  - confidence: 0.87
- **[P2] Skip stale probing after oneshot cleanup frees capacity:** `src/acp/control-plane/manager.core.ts:1105`
  - body: evictDormantOneshotRuntimesUnderPressure may already free a slot, but the code immediately probes stale persistent runtimes before checking runtimeCache.size(). A single new session can therefore evict unrelated idle context even after capacity is available.
  - confidence: 0.84
- **[P2] Recheck pressure before clearing stale candidates:** `src/acp/control-plane/manager.core.ts:1297`
  - body: While a candidate waits in actorQueue.run, another close or eviction can bring the cache below the limit. The callback still clears and closes the candidate, causing avoidable context loss unless it revalidates capacity before clearing.
  - confidence: 0.8
- **[P2] Recompute idle age under the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-getStatus fallback uses candidate.idleMs from the snapshot taken before the actor lock. If another actor-scoped operation touched the session first, this can evict a no-status backend using stale idle age.
  - confidence: 0.78
