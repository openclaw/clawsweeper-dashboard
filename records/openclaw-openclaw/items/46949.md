## Summary

Keep open. The PR targets a real ACP cache/counting bug that still exists on current main, but the branch still lacks after-fix real behavior proof and has pressure-eviction correctness issues that should block merge.

## What This Changes

The PR makes ACP concurrent-session enforcement release cached oneshot handles after startup/status paths, adds pressure eviction for stale cached runtimes, and adds ACP manager regression tests.

## Best Possible Solution

Land a bounded ACP manager fix that releases non-running oneshot handles and reclaims stale cached runtimes only while still at capacity, with focused regression tests and redacted live ACP proof; keep reset/process cleanup in the related open issue.

## Reproduction Assessment

Yes, source-level: current main counts `runtimeCache.size()` in `enforceConcurrentSessionLimit`, while status and startup reconciliation paths can leave oneshot handles cached. I did not run a live ACPX reproduction in this read-only pass.

## Solution Assessment

No. The fix direction is appropriate, but this branch needs a hard status-probe timeout and capacity/idleness rechecks before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a real timeout for status probes:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This starts a 5s AbortController timer but awaits `getStatus` directly. Since the runtime contract only accepts an optional signal and does not guarantee adapters honor it, a runtime that ignores abort can hang session initialization under capacity pressure; race the probe against a timer.
  - confidence: 0.87
- **[P2] Skip stale probing once oneshot eviction frees capacity:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After dormant oneshot eviction, this always enters stale probing even if the cache is already below `maxConcurrentSessions`. One new session can evict an unrelated idle persistent runtime and lose context; recheck `runtimeCache.size()` against the limit before stale probing.
  - confidence: 0.82
- **[P2] Recompute blind idle age inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288`
  - body: The no-`getStatus` fallback uses `candidate.idleMs` from the initial snapshot after waiting on `actorQueue`. A queued operation can touch the cache first, so re-read `lastTouchedAt` or current idle age inside the lock before blind eviction.
  - confidence: 0.78
