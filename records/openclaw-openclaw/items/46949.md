## Summary

Keep open. Current main and v2026.5.12 still reject ACP sessions at the runtime-cache limit without pressure cleanup, while this PR is a plausible fix direction but remains blocked by concrete eviction-safety findings, a dirty branch, and missing real ACP/acpx proof.

## What This Changes

The PR makes ACP session-limit enforcement async, releases startup-reconciled oneshot handles, evicts dormant oneshot and stale runtime-cache entries under pressure, and adds ACP manager regression tests.

## Best Possible Solution

Land a rebased ACP manager fix that releases non-running oneshot handles and reclaims stale cached runtimes only while capacity is still exhausted, with hard probe timeouts, actor-locked eligibility checks, focused tests, and redacted real ACP/acpx proof.

## Reproduction Assessment

Yes, source-level: current main counts runtimeCache.size() against acp.maxConcurrentSessions and rejects at capacity without a pressure cleanup path. I did not run a live ACPX pressure reproduction in this read-only review.

## Solution Assessment

No. The fix direction is appropriate, but the PR head should enforce hard status-probe timeouts, recheck capacity/idleness before clearing candidates, avoid extra evictions once pressure is relieved, and include real after-fix proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce a hard status-probe timeout:** `src/acp/control-plane/manager.core.ts:1282`
  - body: Under pressure this awaits cached.runtime.getStatus directly and only aborts through an optional signal. A runtime that ignores signal can hang every new session attempt at capacity, so race the probe against a real timeout before continuing limit enforcement.
  - confidence: 0.87
- **[P2] Skip stale probing after oneshot cleanup frees capacity:** `src/acp/control-plane/manager.core.ts:1105`
  - body: evictDormantOneshotRuntimesUnderPressure may already free a slot, but the code immediately probes stale persistent runtimes before checking runtimeCache.size(). A single new session can therefore evict unrelated idle context even after capacity is available.
  - confidence: 0.84
- **[P2] Recheck pressure before clearing stale candidates:** `src/acp/control-plane/manager.core.ts:1297`
  - body: While a candidate waits in actorQueue.run, another close or eviction can bring the cache below the limit. The callback still clears and closes the candidate, causing avoidable context loss unless it revalidates capacity before clearing.
  - confidence: 0.8
- **[P2] Recompute idle age under the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-getStatus fallback uses candidate.idleMs from the snapshot taken before the actor lock. If another actor-scoped operation touched the session first, this can evict a no-status backend using stale idle age.
  - confidence: 0.78
