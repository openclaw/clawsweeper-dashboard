## Summary

Current main still has the exact active-wake provenance gap, and the PR feeds a real downstream session-state consumer rather than adding inert metadata. The implementation is likely correct, but an external contributor’s tests and lint are not real behavior proof, and the branch also broadens reply-run queue option types beyond the provenance field actually consumed by active steering.

## What This Changes

The PR carries typed `inputProvenance` through active subagent wake steering, queued `AgentSession` user messages, and reply-run fallback queueing, with regression coverage for retry, stale-thread, generated-media, and fallback paths.

## Best Possible Solution

Keep provenance construction at the subagent-wake owner, forward only the metadata the active steering backend consumes, and demonstrate a running active requester wake whose queued message carries `subagent_announce` provenance while retaining the established user-facing session state.

## Reproduction Assessment

Yes at source level: current main drops the known provenance when steering an active requester, while the execution path requires that provenance to recognize `subagent_announce` as state-preserving. A failing end-to-end runtime symptom has not been reproduced for this narrower claim.

## Solution Assessment

Mostly yes: attaching provenance at queued user-message construction is the narrowest effective boundary and the wake owner knows the source fact. The queue option contract should remain limited to fields with an actual consumer unless owners deliberately approve the broader surface.
