## Summary

Keep open: current main still has diagnostic chat send timing and aggregate usage/throughput surfaces, but it does not expose first-token or total response latency under assistant messages in the Chat UI.

## What This Changes

Requested behavior is to show first-output latency, total latency, and optionally token/throughput details under assistant messages in the Chat UI.

## Best Possible Solution

Add a canonical assistant-turn timing contract that feeds both live chat events and history reloads, then render it through the existing compact assistant message metadata UI.

## Reproduction Assessment

Yes, by source inspection: current main renders message timestamp plus context/usage/model metadata and records diagnostic send timing, but no first-token or total-latency metadata is present in chat event/history schemas or assistant group rendering.

## Solution Assessment

Unclear pending product direction. The maintainable path is to choose a canonical persisted or event-backed timing owner before adding UI chrome, rather than deriving the display from transient client request timing only.
