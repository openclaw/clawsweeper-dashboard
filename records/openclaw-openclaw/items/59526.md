## Summary

Keep this open: current main has a real auth-profile override pipeline for typed `/model ...@<profileId>` selections, but the shared `/models` picker data and channel button/menu paths still expose only provider/model/runtime choices, not account/profile choices.

## What This Changes

Request to let `/models` choose a model, choose the backing GPT/Claude account or auth profile, and show the active account without manually editing model strings.

## Best Possible Solution

Add a generic, privacy-aware auth-profile choice model to the shared `/models` picker data, render it in supported channel pickers, and apply selections through the existing `authProfileOverride` session field rather than adding GPT/Claude-specific account state.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence current-main gap: typed `/model ...@<profileId>` is supported, but `/models` channel picker data and callbacks do not carry profile/account choices.

## Solution Assessment

Unclear until maintainers settle the UX/privacy policy for displaying account labels in chat. The narrow maintainable direction is to reuse the existing auth-profile store, display-label helper, and session override pipeline with provider-generic tests.
