## Summary

Keep open: current main and v2026.6.10 still expose profile pinning through typed `/model <model>@<profile>`, but `/models` picker data and Telegram/Discord picker callbacks remain model/provider/runtime oriented rather than first-class account/profile selection. The related open preservation PR is useful adjacent work, but it preserves existing auth-profile state and does not add the requested picker-side account choice or active-account UX.

## What This Changes

The issue requests account/profile switching directly from `/models` picker buttons for GPT/OpenAI and Claude/Anthropic-style provider accounts.

## Best Possible Solution

Define a shared `/models` account/profile picker contract that lists compatible auth profiles for the selected provider/model, shows the active profile, and persists through the existing model/auth session helpers across channels.

## Reproduction Assessment

Yes, at source level for the missing capability: current `/models` channel data and Telegram/Discord picker callbacks carry model/provider/runtime state, while account/profile pinning only exists through typed `/model ...@profile`. I did not run a live chat flow because the missing picker state is directly visible in current source.

## Solution Assessment

Unclear until maintainers choose the UX shape. The maintainable direction is to extend the shared picker/profile contract rather than add raw model-string editing or channel-specific account hacks.
