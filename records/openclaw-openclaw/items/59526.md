## Summary

Keep open: current main supports typed `/model <model>@<profileId>` and lower-level profile override persistence, but the documented `/models` button/dropdown picker still exposes provider/model selection without account/profile choices.

## What This Changes

The issue requests adding account selection to the `/models` UI so users can switch GPT and Claude accounts from model buttons without editing raw model strings.

## Best Possible Solution

Extend the existing picker/directive flow so provider, model, and account/profile choices are selected and persisted together through the shared session override and auth-profile compatibility contracts.

## Reproduction Assessment

Yes, at source level: the documented `/models` picker path persists provider/model/runtime choices but does not expose or persist an auth-profile/account selection. I did not run a live chat flow because this read-only review found the missing capability in the current source path.

## Solution Assessment

Unclear until maintainers choose the UX shape. The best technical direction is not raw model-string editing; it is extending the existing picker/directive path to carry a validated profile override alongside model selection.
