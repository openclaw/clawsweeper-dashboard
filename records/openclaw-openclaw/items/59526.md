## Summary

Keep open: current main and v2026.6.10 still expose typed `/model <model>@<profile>` plus model-only `/models` picker callbacks, but not first-class account/profile choice from the picker. The related open PR only preserves existing auth-profile overrides during model selection; it does not add the requested account-switching UX.

## What This Changes

Request to add account/profile selection and current-account visibility to `/models` buttons for GPT/OpenAI and Claude/Anthropic-backed models.

## Best Possible Solution

Define a shared `/models` account/profile picker contract that lists compatible auth profiles for the selected provider/model, shows the active profile, and persists through existing session/model auth helpers across channels.

## Reproduction Assessment

Yes, at source level. Current main passes model-only data into `/models` channel pickers and keeps account/profile pinning on the typed `/model ...@profile` path, so the requested picker-side account choice is visibly absent without a live chat run.

## Solution Assessment

Unclear until maintainers choose the UX and compatibility semantics. The maintainable direction is a shared picker/profile contract rather than raw model-string editing or channel-specific account hacks.
