## Summary

Keep open: current main has typed `/model ...@<profileId>` support and can display an active auth label in text surfaces, but the shared `/models` picker contracts and channel callbacks still expose provider/model/runtime choices only, not selectable account/auth-profile choices.

## What This Changes

Request account/profile switching directly from the `/models` picker for GPT and Claude-backed models.

## Best Possible Solution

Add a provider-generic, privacy-aware auth-profile choice model to shared `/models` picker data and channel SDK contracts, then apply selections through the existing `authProfileOverride` persistence path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; this is a feature request, and source inspection gives high confidence that current `/models` picker payloads and callbacks still lack selectable auth-profile/account choices.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the account-label UX/privacy policy; the maintainable direction is provider-generic picker data backed by existing auth-profile/session override plumbing, not GPT/Claude-specific picker state.
