## Summary

Keep open. Current main still routes forced-document Telegram MP4 media through the video sender path, and the live GitHub state shows https://github.com/openclaw/openclaw/pull/80405 is an open closing PR for the narrow fix, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that Telegram `message send --media <file.mp4> --force-document` route through `sendDocument` instead of `sendVideo`, with compatible failure handling for callers.

## Best Possible Solution

Review and land one focused PR that makes forced-document Telegram video sends use `sendDocument` with live Telegram proof, then split fallback behavior only if it remains reproducible and maintainable.

## Reproduction Assessment

Yes for the routing bug: source inspection on current main shows `video/mp4` selects `sendVideo` before the document fallback even when `forceDocument` is set. I did not run a live Telegram send in this read-only pass.

## Solution Assessment

Yes for the explicit override: adding `!forceDocument` guards around video-note selection, video probing, and the regular `sendVideo` branch is the narrow maintainable fix. Automatic fallback from a failed typed endpoint should be considered separately because ambiguous network failures can risk duplicate sends.
