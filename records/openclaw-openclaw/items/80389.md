## Summary

Keep open: current main and the latest shipped tag still route `video/mp4` through Telegram `sendVideo` before the document branch, so the reported `--force-document` video override is not implemented. The direct `sendDocument` workaround is consistent with Telegram's documented general-file endpoint, but automatic fallback after ambiguous network failures needs duplicate-safe handling.

## What This Changes

The issue asks Telegram `message send --media` to deliver MP4s reliably by honoring `--force-document`, optionally falling back to `sendDocument`, and returning a non-zero exit code on final send failure.

## Best Possible Solution

Implement explicit `forceDocument` document routing for Telegram video media with focused regression coverage, and only add automatic fallback for error classes that are proven safe from duplicate sends.

## Reproduction Assessment

Yes for the routing bug by source inspection: current main routes `video/mp4` to `sendVideo` even when `forceDocument` is true. The live Telegram network failure and exit-code symptom were not reproduced in this read-only review.

## Solution Assessment

Partly: explicit `--force-document` video routing is the narrow maintainable fix. Auto-falling back from `sendVideo` to `sendDocument` should be limited or deferred until the ambiguous-send duplicate risk is handled.
