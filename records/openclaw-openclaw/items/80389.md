## Summary

Keep open. Current main still routes Telegram video media through `sendVideo` before the document fallback when `forceDocument` is set, and the open closing PR at https://github.com/openclaw/openclaw/pull/80405 is the active fix path.

## What This Changes

The issue asks Telegram `message send --media` to honor `--force-document` for MP4/video media, add document fallback after typed-send failure, and report final send failures with a non-zero exit.

## Best Possible Solution

Review and land or replace the focused PR that makes forced-document videos skip video-specific probing/routing and use `sendDocument`; then track fallback and exit-code behavior separately if it remains reproducible.

## Reproduction Assessment

Yes for the routing bug: source inspection on current main shows `forceDocument` is passed through, but video media still selects `api.sendVideo` before the document branch. I did not run a live Telegram send.

## Solution Assessment

Yes for the explicit override: guarding video probing and video sender selection behind `!forceDocument` is the narrow maintainable fix. Automatic fallback from a failed typed send is a separate product decision because ambiguous network failures can risk duplicate sends.
