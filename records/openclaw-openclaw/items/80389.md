## Summary

Keep open. Current main still routes Telegram video media through `sendVideo` before the document fallback even when `forceDocument` is set, and https://github.com/openclaw/openclaw/pull/80405 is an open closing PR for the narrow fix, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for Telegram `message send --media` to deliver MP4s by honoring `--force-document`/`sendDocument`, optionally falling back from typed media endpoints, and surfacing final send failures with a nonzero exit.

## Best Possible Solution

Review and land one focused PR that makes forced-document Telegram video sends choose `sendDocument` with live Telegram proof, then split any fallback or exit-code follow-up if it remains reproducible.

## Reproduction Assessment

Yes for the routing bug: source inspection on current main shows `kind === "video"` selects `sendVideo` without a `forceDocument` guard, so a mocked MP4 send would reproduce the selected method. I did not run a live Telegram server send in this read-only pass.

## Solution Assessment

Mostly yes: the linked PR's `!forceDocument` guards around video-note selection, video probing, and `sendVideo` are the narrow maintainable fix for the explicit override. Automatic fallback from failed typed endpoints should be considered separately because ambiguous network failures can risk duplicate sends.
