## Summary

Keep open: current main still routes Telegram `video/mp4` media through `sendVideo` before the document branch, even when `forceDocument` is set. The related final-reply media issue is a separate path, and current CLI error wrapping appears to exit non-zero for thrown send failures, so the narrow repair should start with the explicit video document override.

## What This Changes

The issue asks for Telegram `message send --media` to deliver MP4 media reliably by honoring `--force-document`, optionally falling back to `sendDocument`, and surfacing final failures as non-zero exits.

## Best Possible Solution

Implement explicit `forceDocument` document routing for Telegram video media with focused regression coverage, and only add automatic fallback after a duplicate-safe error policy is defined.

## Reproduction Assessment

Yes for the routing bug by source inspection: current main maps `video/mp4` to the video branch and calls `sendVideo` before the document branch can see `forceDocument`. The live Telegram network failure and exit-code symptom were not reproduced.

## Solution Assessment

Partly: explicit `--force-document` video routing is the narrow maintainable fix. Broad automatic fallback from `sendVideo` to `sendDocument` should wait until ambiguous-send duplicate risk is handled.
