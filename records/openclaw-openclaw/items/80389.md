## Summary

Keep open. Current main still has a source-reproducible Telegram video `--force-document` routing bug, and an open closing PR already targets the narrow fix, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks for Telegram `message send --media <file.mp4> --force-document` to route through `sendDocument`, optionally fall back from typed media sends, and fail with a non-zero exit status when media delivery fails.

## Best Possible Solution

Review and land the focused Telegram PR that makes forced-document video sends select `sendDocument`, then track typed-endpoint fallback or exit-code behavior separately if still needed.

## Reproduction Assessment

Yes for the routing bug by source inspection: current main selects `sendVideo` for `kind === "video"` before the document branch and without a `forceDocument` guard. I did not run a live Telegram upload in this read-only review.

## Solution Assessment

Yes for the explicit override: a narrow guard that lets forced-document videos use `sendDocument` is the maintainable fix. Automatic fallback and non-zero exit behavior should be evaluated as separate follow-ups because fallback after ambiguous network failures can risk duplicate sends.
