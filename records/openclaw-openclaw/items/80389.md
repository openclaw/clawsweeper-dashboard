## Summary

Keep open. Current main still selects Telegram `sendVideo` for video media before the document fallback even when `forceDocument` is set, and https://github.com/openclaw/openclaw/pull/80405 is an open closing PR for the focused fix, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks for Telegram `message send --media` to honor `--force-document` for MP4/video media, optionally fall back to `sendDocument` on typed endpoint failure, and return non-zero when delivery fails.

## Best Possible Solution

Review and land the focused PR that makes forced-document Telegram videos skip video-specific probing/routing and use `sendDocument`; split fallback and exit-code follow-ups if they remain reproducible after that fix.

## Reproduction Assessment

Yes for the routing bug: source inspection on current main shows a `forceDocument: true` video still selects `sendVideo`, so a focused mock of `sendMessageTelegram` with MP4 media would reproduce the selected method. I did not live-send to Telegram in this read-only review.

## Solution Assessment

Yes for the explicit override: guarding the video probe and video sender behind `!forceDocument` is the narrow maintainable fix. Automatic fallback from `sendVideo` failure should be a separate decision because failed network sends may not prove Telegram rejected the media before delivery.
