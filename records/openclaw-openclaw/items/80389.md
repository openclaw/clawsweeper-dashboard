## Summary

Keep open. Current `main` still routes Telegram video media through `sendVideo` before the document fallback even when `forceDocument` is true, and https://github.com/openclaw/openclaw/pull/80405 is an open closing PR for the focused fix.

## What This Changes

The issue asks for Telegram MP4 media sent with `--force-document` to use `sendDocument` instead of `sendVideo`, with fallback and failure reporting considered as follow-ups.

## Best Possible Solution

Review and land one focused PR that makes forced-document Telegram videos bypass video probing and `sendVideo`, then track fallback or exit-code behavior separately if those symptoms remain.

## Reproduction Assessment

Yes for the routing bug: current source shows a `forceDocument: true` video still selects the Telegram video sender before document fallback. I did not live-send to Telegram, so the network failure and exit-code parts are not fully reproduced here.

## Solution Assessment

Yes for the explicit override: guarding video probing and video sender selection behind `!forceDocument` is the narrow maintainable fix. Automatic fallback from a failed typed endpoint should be reviewed separately because ambiguous network failures can risk duplicate sends.
