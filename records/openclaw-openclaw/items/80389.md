## Summary

Keep open. Current main still routes Telegram video media through the video sender before the document fallback even when `forceDocument` is set, and an open closing PR already targets the focused routing fix.

## What This Changes

The issue requests that Telegram `.mp4` sends respect `--force-document`/`sendDocument` and surface media-send failures to callers instead of silently routing through `sendVideo`.

## Best Possible Solution

Review and land the focused PR that makes forced-document Telegram videos skip video-specific probing/routing and use `sendDocument`; split fallback and exit-code follow-ups if they remain reproducible after that fix.

## Reproduction Assessment

Yes for the routing bug: source inspection on current main shows a `forceDocument: true` video still selects `sendVideo`, so a focused mock of `sendMessageTelegram` with MP4 media would reproduce the selected method. I did not live-send to Telegram in this read-only review.

## Solution Assessment

Yes for the explicit override: guarding video-note selection, video probing, and the `sendVideo` branch behind `!forceDocument` is the narrow maintainable fix. Automatic fallback from failed typed endpoints should be handled separately because ambiguous network failures can risk duplicate sends.
