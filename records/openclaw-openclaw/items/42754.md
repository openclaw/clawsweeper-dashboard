## Label Justifications

- `P2`: Feishu interactive-card payload loss affects real plugin workflows but is scoped to one channel plugin.
- `impact:data-loss`: Structured selector and form values from Feishu callbacks are discarded before plugins or agents can use them.
- `impact:message-loss`: The synthetic channel message created from the card callback omits part of the inbound user interaction payload.

## Summary

Keep open: current main and v2026.5.22 still drop Feishu card-action sibling fields before agent dispatch, while overlapping open PRs already cover the implementation path and need a canonical payload-contract decision.

## What This Changes

The issue asks Feishu card action callbacks to preserve action.option, action.options, and action.form_value so plugin and agent handlers can receive selector and form data.

## Best Possible Solution

Land one canonical Feishu plugin patch that preserves selected sibling action fields at the parser boundary, carries them through decoder/fallback/handler dispatch, keeps legacy command handling compatible, covers the chosen payload contract with tests, includes redacted live Feishu proof, and then closes the related issues after merge.

## Reproduction Assessment

Yes, source-level. A raw card.action.trigger payload with action.option, action.options, or action.form_value is normalized into only action.value and action.tag, then fallback dispatch serializes only action.value.

## Solution Assessment

No final implementation is on main yet. A handler-only change would be incomplete; the maintainable fix should start at parseFeishuCardActionEventPayload and align the event type, decoder/fallback behavior, handler dispatch, tests, and selected payload shape.
