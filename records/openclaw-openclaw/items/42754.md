## Label Justifications

- `P2`: The missing fields block structured Feishu card workflows but are limited to one bundled channel plugin.
- `impact:data-loss`: Dropdown, multi-select, and form values are discarded before plugins or agents can consume them.
- `impact:message-loss`: The synthetic inbound message omits meaningful parts of the user's card interaction payload.

## Summary

Keep open: current main and v2026.6.6 still normalize Feishu card callbacks to action.value/tag only, so selector/form fields are not agent-visible; two overlapping open PRs already cover the work but need a canonical payload contract and live proof before this issue can close.

## What This Changes

Request that Feishu card action callbacks pass dropdown, multi-select, and form fields through to plugins instead of only action.value.

## Best Possible Solution

Land one Feishu plugin patch that preserves the raw callback fields at the parser boundary, carries them through the decoder/handler without bypassing structured-action safeguards, defines one command-compatible payload contract, and includes focused tests plus redacted live Feishu proof.

## Reproduction Assessment

Yes. Source inspection shows live card.action.trigger callbacks are normalized to value/tag only and the fallback dispatch reads only action.value; this review did not run a live Feishu callback.

## Solution Assessment

No current implementation is on main. A handler-only change is too narrow; the fix should start at the parser boundary and preserve the existing structured-action validation and command-routing behavior.
