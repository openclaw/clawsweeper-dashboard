## Label Justifications

- `P2`: Feishu interactive-card payload loss affects real plugin workflows but is scoped to one channel plugin.
- `impact:data-loss`: Structured user selections and form values from Feishu callbacks are discarded before plugins or agents can use them.
- `impact:message-loss`: The synthetic channel message created from the card callback omits part of the inbound user interaction payload.

## Summary

Keep open: current main and the latest release still drop the Feishu card-action sibling fields before agent dispatch, and live related PRs already cover the repair path while maintainers choose the canonical callback payload contract.

## What This Changes

The issue requests preserving Feishu card callback `action.option`, `action.options`, and `action.form_value` so dropdown, multi-select, and form submissions reach plugins or agents.

## Best Possible Solution

Land one canonical Feishu plugin patch that preserves selected sibling action fields from parser through decoder/fallback/handler dispatch, keeps legacy command handling compatible, covers the chosen payload contract with tests, includes redacted live Feishu proof, and then closes this issue after merge.

## Reproduction Assessment

Yes, source-level. A raw `card.action.trigger` payload with `action.option`, `action.options`, or `action.form_value` is normalized into only `action.value` and `action.tag`, then fallback dispatch serializes only `action.value`.

## Solution Assessment

No final implementation is on main yet. A handler-only change would be incomplete; the maintainable fix should start at `parseFeishuCardActionEventPayload` and align the event type, decoder/fallback behavior, handler dispatch, tests, and selected payload shape.
