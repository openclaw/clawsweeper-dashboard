## Summary

Keep open. Current main still reconstructs Feishu card action events with only `action.value` and `action.tag`, and the decoder/fallback path only reads `event.action.value`, so `action.option`, `action.options`, and `action.form_value` are still dropped before dispatch. The issue remains valid and has an open same-author implementation candidate, #42908, while the newer #74191 is closed unmerged.

## What This Changes

The issue requests preserving Feishu card callback sibling fields `action.option`, `action.options`, and `action.form_value` so plugins can receive dropdown, multi-select, and form submission data.

## Best Possible Solution

Land a Feishu-owned end-to-end fix that preserves the raw sibling fields in `parseFeishuCardActionEventPayload`, widens the decoder/handler types, defines the fallback command/JSON format, and adds focused tests for dropdown, multi-select, and form submission callbacks. #42908 is the active starting point, but it likely needs review and rebase or replacement before merge.

## Reproduction Assessment

Yes. A raw Feishu card-action payload with `action.value` plus `action.option`, `action.options`, or `action.form_value` will be normalized by current `parseFeishuCardActionEventPayload` into an event containing only `value` and `tag`, and the downstream fallback then reads only `action.value`.

## Solution Assessment

Yes, the best fix belongs in the Feishu plugin path rather than core: preserve the callback fields at the parser boundary and carry them through the existing decoder/handler contract with tests. Closing before an implementation lands would hide a valid same-author issue/PR pair.
