## Summary

Keep this open: current main still models and forwards only `action.value`/`tag` for Feishu card actions, so the reported callback fields are not preserved, and an open same-author PR already covers the targeted fix.

## What This Changes

Request that Feishu card action callbacks preserve `action.option`, `action.options`, and `action.form_value` so plugins can receive dropdown, multi-select, and form submission data.

## Best Possible Solution

Land a narrow Feishu parser/handler/test change that preserves these callback fields without changing existing button behavior, then close this issue after the implementation merges.

## Reproduction Assessment

Yes, from source inspection: a Feishu payload containing `action.option`, `action.options`, or `action.form_value` loses those sibling fields when `parseFeishuCardActionEventPayload` returns only `action.value` and `tag`. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the narrowest maintainable direction is to extend the Feishu-owned parser/type/handler path and add focused regression tests. The open PR should settle the exact synthetic message format before merge.
