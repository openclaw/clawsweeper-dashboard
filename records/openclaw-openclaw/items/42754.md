## Label Justifications

- `P2`: This is a normal-priority Feishu channel payload-preservation issue with real interactive-card workflow impact but limited blast radius.
- `impact:data-loss`: Feishu callback fields carrying user selections and form values are discarded before plugins or agents can use them.
- `impact:message-loss`: The synthetic channel message created from the card callback omits part of the user interaction payload.

## Summary

Keep open: current main and v2026.5.20 still normalize Feishu card callbacks down to action.value/action.tag before agent dispatch, while open Feishu PRs already cover competing fix paths that need a maintainer payload-contract decision.

## What This Changes

Preserve Feishu card callback sibling fields such as action.option, action.options, and action.form_value so dropdown, multi-select, and form interactions can reach plugins or agents.

## Best Possible Solution

Land one canonical Feishu plugin patch that preserves selected sibling action fields at the parser boundary, carries them through decoder/fallback/handler dispatch, keeps legacy command handling compatible, covers the chosen payload contract with tests, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, source-level. A raw card.action.trigger payload with action.option, action.options, or action.form_value is normalized into only action.value and action.tag, then fallback dispatch serializes only action.value.

## Solution Assessment

Partly: preserving the sibling fields is the right direction, but a handler-only change would be incomplete. The maintainable fix should start at parseFeishuCardActionEventPayload and align the event type, decoder/fallback behavior, handler dispatch, tests, and selected payload shape.
