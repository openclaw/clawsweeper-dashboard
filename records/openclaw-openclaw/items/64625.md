## Summary

Keep open. Current `main` still matches the reported Slack custom-invoke bug: the bundled Slack plugin uses the custom invoke path, core supplies media access on the top-level channel action context, and the Slack wrapper forwards only `toolContext` when `options.invoke` is present. The related #64632 fix PR is closed and unmerged, so the remaining work is still actionable.

## What This Changes

The issue requests that Slack `upload-file` preserve agent-scoped `mediaLocalRoots` and `mediaReadFile` when the bundled Slack plugin routes message actions through its custom invoke callback.

## Best Possible Solution

Fix the bundled Slack plugin by building one trusted action context from `toolContext` plus `ctx.mediaLocalRoots` and `ctx.mediaReadFile`, then pass that context through both the custom invoke and fallback runtime paths with focused regression coverage for `upload-file`.

## Reproduction Assessment

Yes. A source-level reproduction is to call `createSlackActions` with `options.invoke` and a `ChannelMessageActionContext` containing `mediaLocalRoots`/`mediaReadFile`; current code forwards only `ctx.toolContext`, so the Slack upload runtime receives no agent-scoped roots.

## Solution Assessment

Yes. The narrow maintainable fix is the issue's proposed wrapper-boundary change: forward the trusted, core-computed media access fields into the custom invoke path while leaving the local-media guard intact.
