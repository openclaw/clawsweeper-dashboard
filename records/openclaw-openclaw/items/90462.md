## Summary

Keep open. Current main has improved reset/status/provenance paths for auto fallback selections, but code and docs still intentionally persist automatic fallback as a session override and I did not find source coverage that treats LM Studio "Model is unloaded" as a failed fallback candidate, so the reported provider/session-state trap still needs maintainer follow-up.

## What This Changes

The issue asks OpenClaw to keep automatic model fallback from trapping a Telegram topic on an unavailable LM Studio model and to expose reliable reset and diagnostic recovery.

## Best Possible Solution

Define the intended auto-fallback lifetime, make automatic pins observable and bounded, ensure model reset flows clear stale auto pins, classify unloaded local-provider models as failed fallback candidates, and add focused regression coverage.

## Reproduction Assessment

No live Telegram/LM Studio repro was run. Source inspection shows matching auto-persisted fallback and reset paths, but the exact unloaded-model loop still needs focused live or unit reproduction.

## Solution Assessment

Unclear. The best fix is broader than adding a reset command because it needs to preserve user-selected model semantics while making automatic fallback pins observable, bounded, and self-clearing when the fallback fails.
