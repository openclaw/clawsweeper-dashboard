## Summary

Keep open. This is a plausible and proof-backed feature, but the patch makes denylist a public exec security/config mode while the macOS host/settings code still cannot decode, preserve, or enforce that mode.

## What This Changes

The PR adds an exec denylist security mode, default curl/wget deny rules, logging, docs, tests, and Control UI support for editing denylist rules.

## Best Possible Solution

Land denylist only after the TS gateway/node runtime, web UI, macOS host/settings path, docs, and tests all share the same exec approvals contract and enforcement semantics.

## Reproduction Assessment

Not applicable. This is a feature PR rather than a bug report; the contributor did provide live after-fix proof for the TS/web path.

## Solution Assessment

No. The TS/web implementation direction is plausible, but exposing denylist as a public exec policy before the macOS host can decode, preserve, and enforce it leaves the product contract split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Add macOS exec-approval denylist support:** `src/config/types.tools.ts:302`
  - body: This line makes `denylist` a supported exec security value, but the macOS app still decodes `ExecSecurity` with only `deny`, `allowlist`, and `full`, and its approvals file model has no `denylist` field. Files containing `security` or `askFallback` set to `denylist` fail to decode on that host path, while files containing denylist arrays can be read/saved without preserving or enforcing those rules. Please update the macOS approvals model, settings UI, resolver, and system.run enforcement before exposing the new mode.
  - confidence: 0.91
