## Summary

Keep this PR open: the main denylist path has strong proof and the later commits addressed the earlier native macOS parity concern, but a gateway headless fallback path can still treat `askFallback: "denylist"` as prechecked without actually evaluating the denylist when the prompt requirement comes from `exec-approvals.json`.

## What This Changes

The PR adds an exec denylist security mode across exec approval config, gateway/node/macOS enforcement, Control UI, documentation, changelog, and tests.

## Best Possible Solution

Resolve denylist fallback through one shared host policy context, only mark fallback as prechecked when `evaluateExecDenylist` actually ran on the final command/env/workdir, and keep TS gateway/node plus native macOS behavior covered by focused tests.

## Reproduction Assessment

Yes for the blocking finding by source inspection: configure gateway approvals with `ask: "always"`, `askFallback: "denylist"`, and a matching denylist while tool/default ask stays off; the PR precheck skips denylist evaluation but the fallback path still marks it prechecked. I did not run a mutable gateway repro in this read-only review.

## Solution Assessment

No, not yet: the main denylist implementation is the right direction, but the fallback path needs to reuse the resolved host policy or carry an explicit evaluated precheck result before this is merge-safe.
