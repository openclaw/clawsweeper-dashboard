## Summary

Keep this PR open, but it is not ready to merge: the TS gateway/node and web UI portions add a plausible denylist mode, while the macOS host path still cannot decode, preserve, or enforce the same public exec-approval contract.

## What This Changes

The branch adds an exec denylist mode, denylist rule normalization/evaluation, default curl/wget blocking, logging, docs, tests, and Control UI support for editing denylist rules.

## Best Possible Solution

Land denylist only after the TS gateway/node runtime, web UI, macOS host/settings path, docs, and tests all share one exec approvals contract and enforcement model.

## Reproduction Assessment

Not applicable for the feature request itself; the blocking review issue is source-reproducible from the PR head. TS config/protocol accept denylist while the macOS ExecSecurity/Codable models and native system.run path still do not.

## Solution Assessment

No. The TS/web implementation direction is plausible, but exposing denylist as public exec policy before the macOS host can decode, preserve, and enforce it leaves a split security contract.
