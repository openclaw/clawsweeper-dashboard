## Summary

Close: current main already implements the Telegram markdown-mode chunking behavior with the original plain-text fallback preserved, while this PR branch is superseded by that safer implementation.

## What This Changes

The PR changes Telegram direct text sends so default markdown payloads are rendered to Telegram HTML and sent through the chunked text path, with a changelog entry.

## Best Possible Solution

Close this PR and keep the current main implementation that chunks raw markdown through the existing planner while preserving plain-text fallback semantics.

## Reproduction Assessment

Yes. The mocked sendMessageTelegram tests on current main exercise a 5000-character default markdown payload and assert multiple <=4000-character HTML sends with buttons only on the final chunk.

## Solution Assessment

No for this PR branch; yes for current main. The narrow maintainable fix is to pass the original raw text into the chunk planner and let the planner handle HTML rendering plus fallback text, which current main does.
