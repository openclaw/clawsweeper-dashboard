## Summary

Keep open. The PR targets a real Telegram markdown-mode chunking bug, but the current patch regresses the existing HTML-parse fallback by using rendered HTML as the plain fallback for markdown sends.

## What This Changes

The PR adds a changelog entry and routes Telegram direct text sends through the chunked send path after rendering markdown to Telegram HTML.

## Best Possible Solution

Keep the PR open and adjust the chunk helper or call site so markdown mode chunks rendered Telegram HTML while using the original text as the plain fallback, preserving existing HTML-mode behavior.

## Reproduction Assessment

Yes. A mocked sendMessageTelegram test can reproduce current main sending one oversized default-markdown payload, and a mocked HTML parse error for _oops_ reproduces the PR regression by expecting the fallback to remain _oops_ rather than rendered tags.

## Solution Assessment

No. Rendering markdown before chunking is the right direction, but the implementation should pass an explicit plain fallback into the chunk plan instead of deriving it from rendered HTML.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve the original markdown fallback:** `extensions/telegram/src/send.ts:967`
  - body: This call passes rendered HTML into `sendChunkedText`, and `buildChunkedTextPlan` derives `plainText` from that same argument when `opts.plainText` is absent. If Telegram rejects the HTML parse, markdown sends now fall back to literal rendered tags like `<i>oops</i>` instead of the original `_oops_`, breaking the existing parse-fallback contract. Pass the original text as the fallback while chunking the rendered HTML.
  - confidence: 0.93
