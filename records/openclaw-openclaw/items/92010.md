## Summary

The provider proposal is useful and roadmap-aligned, but current main already exposes the generic provider-plugin capabilities it requires; repository policy sets a high bar for bundling optional integrations in core.

## What This Changes

Add CloudSigma TaaS as an OpenAI-compatible multi-model provider with API-key authentication, catalog discovery, and a static fallback catalog.

## Best Possible Solution

CloudSigma should publish and maintain an npm-distributed provider plugin in its own repository, use the documented authentication and catalog hooks, and list it through ClawHub; a narrower core issue is appropriate only if implementation exposes a missing generic SDK capability.

## Reproduction Assessment

Not applicable; this issue proposes a new provider integration rather than reporting broken existing behavior. Its required technical shape is source-verifiable through the provider SDK documentation and the Novita implementation.

## Solution Assessment

No, bundling another core plugin is not the best route under current repository policy. An externally maintained provider plugin is narrower, already supported, and preserves the intended ownership boundary.
