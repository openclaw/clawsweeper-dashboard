## Repair Work Prompt

Fix the isolated cron long-attempt lane progress bug reported in https://github.com/openclaw/openclaw/issues/94033. Current main updates `laneTaskProgressAtMs` only through execution phase/run-progress callbacks, while the awaited `runEmbeddedAttemptWithBackend` can remain busy in a long tool execution or non-streaming output path until the `cron-nested` command-lane sliding timeout fires. Implement a bounded liveness repair in the embedded-runner/cron lane path so real active tool execution or streaming work keeps the lane watchdog from firing prematurely, while preserving cron-owned absolute `timeoutSeconds`, abort propagation, cleanup, and stale-lane release behavior. Do not add config or change cron defaults. Compare the adjacent pre-execution issues/PRs, but do not fold their watchdog phase fix into this PR unless the same code change is required. Add focused fake-timer/regression coverage for long in-flight attempt progress and for a truly stuck attempt still timing out or being cleaned up as designed; update docs only if the user-visible timeout contract changes. CHANGELOG.md is release-owned and must not be edited.
