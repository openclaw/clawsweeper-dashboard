## Summary

Keep open. Current main has stronger release validation, beta-first release mechanics, staged update safety, and rollback guidance, but it still does not expose an owner-set production-ready, early-adopter, or known-issues indicator in release notes, workflow metadata, update docs, or the update CLI; the adjacent open PR is heuristic pre-upgrade review rather than canonical release readiness metadata.

## What This Changes

The issue asks OpenClaw to add a per-release production-readiness or known-issues stability indicator so production users can decide when to upgrade.

## Best Possible Solution

Define a release-owned readiness taxonomy and criteria in the release process, require one canonical metadata value for each release, and surface it first in GitHub release notes; later update CLI/UI surfaces can consume the same metadata if maintainers want runtime visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update docs, update CLI registration, latest release notes, and targeted term searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable path should start with release-owned metadata rather than inferred heuristics alone.
