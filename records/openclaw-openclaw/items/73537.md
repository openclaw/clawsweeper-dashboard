## Summary

Keep open. Current main has stable/beta/dev release lanes, release validation profiles, staged update hardening, and rollback/update documentation, but it still does not expose an official per-release Production-ready, Early adopter, Known issues, or equivalent readiness marker; related release-stability discussions explicitly route the broader policy question here.

## What This Changes

The issue requests a visible per-release production-readiness/stability indicator in release notes, GitHub release metadata, or release metadata so production users can judge update risk.

## Best Possible Solution

Define a maintainer-owned readiness taxonomy and source-of-truth release metadata, require release notes to display it, and optionally expose the same value to update CLI/UI consumers.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, workflow, latest-release, and related-thread inspection gives high-confidence verification that current main lacks the requested official per-release readiness marker.

## Solution Assessment

Unclear until release owners define the public criteria. The requested direction is plausible, but the maintainable implementation should be an owner-set release metadata field rather than heuristic risk inference alone.
