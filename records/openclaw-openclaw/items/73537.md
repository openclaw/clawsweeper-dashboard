## Summary

Keep this open. Current main has stronger release lanes, beta-first validation, update dry-run/status surfaces, staged package verification, and rollback guidance, but it still does not expose the requested per-release production-ready, early-adopter, or known-issues indicator in release notes, workflow metadata, update CLI, or public update docs. The related open PR #61973 overlaps with upgrade trust but proposes heuristic pre-upgrade risk review rather than canonical release-owner readiness metadata.

## What This Changes

Add a user-visible production-readiness stability indicator to each OpenClaw release so operators can decide whether to update production installations.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one canonical source for each release, and expose it first in GitHub release notes. Update/status CLI or UI surfaces can consume that same metadata later if maintainers want runtime visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update CLI registration, update docs, release-note generation scripts, and targeted term searches show no per-release production-readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release-owned metadata rather than heuristic-only update review output.
