## Summary

Keep this open. Current main has stable/beta/dev lanes, release validation, package-update safeguards, rollback docs, and adjacent update-safety proposals, but it still does not expose an owner-set per-release Production-ready, Early adopter, Known issues, or equivalent readiness marker.

## What This Changes

The issue requests a visible per-release production-readiness/stability indicator in release notes, GitHub release metadata, or similar release metadata.

## Best Possible Solution

Establish a release-manager-owned readiness taxonomy and make it a required release note or release metadata value, then let update CLI/UI surfaces consume that canonical value.

## Reproduction Assessment

Not applicable as a feature request. Current-main source and doc checks show release channels, validation profiles, and update commands, but no visible per-release readiness marker.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable path should start with owner-set release metadata rather than inferred update-risk heuristics alone.
