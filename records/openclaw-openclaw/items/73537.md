## Summary

Keep this open. Current main has beta/stable/dev release lanes, broad validation profiles, package-update safeguards, and rollback docs, but it still does not expose an owner-set per-release production-readiness marker in release notes, release workflow inputs, release metadata, or the update CLI; the open #61973 PR is adjacent pre-upgrade heuristic work, not canonical release readiness metadata.

## What This Changes

Request to add a per-release production-readiness or stability indicator such as Production-ready, Early adopter, or Known issues to release notes, labels, or release metadata.

## Best Possible Solution

Define a release-owned readiness taxonomy and criteria, require one canonical metadata value for each release, surface it at the top of GitHub release notes, and let CLI/UI consumers read that same source if needed.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update docs, update CLI registration, changelog, and targeted term searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable path should start with release-owned metadata rather than inferred update-review heuristics alone.
