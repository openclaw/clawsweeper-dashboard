## Summary

Keep open. Current main has stronger release validation, beta-first flow, staged update safety, rollback guidance, and update status/wizard surfaces, but it still does not expose an owner-set per-release Production-ready, Early adopter, or Known issues marker. The related open PR is a heuristic pre-upgrade risk review, not canonical release readiness metadata.

## What This Changes

The issue requests a visible production-readiness/stability indicator on OpenClaw releases so production users can decide whether to upgrade.

## Best Possible Solution

Define one release-owned readiness taxonomy and source of truth, require that value during release creation, and surface it first at the top of GitHub release notes; update CLI/UI consumers can follow once the metadata contract exists.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update docs, update CLI registration, and latest release metadata do not expose the requested marker.

## Solution Assessment

Unclear until release owners choose the public criteria and commitment level. The requested direction is plausible, but the maintainable path should start with canonical release metadata rather than inferred changelog heuristics alone.
