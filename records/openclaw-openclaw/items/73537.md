## Summary

Keep open. Current main has stronger release validation, beta-first channel semantics, staged update verification, dry-run/status update surfaces, and rollback guidance, but it still does not expose the requested per-release production-ready / early-adopter / known-issues indicator in release notes, release metadata, or update CLI/UI surfaces. Related PR #61973 overlaps with upgrade trust, but it proposes heuristic pre-upgrade review rather than release-owner-supplied readiness metadata.

## What This Changes

Request a release-level production-readiness stability indicator in release notes, release metadata, or update surfaces so production users can tell when a version is safe to upgrade to.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one canonical source for each release, and expose it first in GitHub release notes; update/status surfaces can consume the same metadata later if maintainers want CLI visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release policy docs, update docs, update CLI registration, release workflows, changelog, and exact term searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start from release-owner metadata rather than heuristic-only upgrade risk scoring.
