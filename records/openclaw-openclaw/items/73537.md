## Summary

Keep open. Current main has stronger release validation, beta-first channel semantics, update dry-run/status surfaces, staged package verification, and rollback guidance, but it still does not expose the requested per-release production-ready / early-adopter / known-issues indicator in release notes, release metadata, or update UI/CLI surfaces. The related open PR #61973 overlaps with upgrade trust but proposes heuristic pre-upgrade review rather than release-owner-supplied readiness metadata.

## What This Changes

Request to add a per-release production-readiness indicator, such as production-ready, early-adopter, or known-issues, to OpenClaw release notes, release labels/tags, or release metadata.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one canonical source for each release, and expose it first in GitHub release notes; update/status surfaces can consume the same metadata later if maintainers want CLI visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: docs index, release/update docs, update CLI registration, release workflows, changelog, and exact release-body term checks show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release process metadata rather than a heuristic-only update command.
