## Summary

Keep this open. Current main has beta-first release lanes, broad release validation, staged package-update safeguards, and rollback docs, but it still does not expose an owner-set per-release Production-ready, Early adopter, Known issues, or equivalent readiness marker in release notes, release workflow metadata, update docs, or the update CLI; the related open PR is an adjacent heuristic pre-upgrade review, not canonical release readiness metadata.

## What This Changes

Request to add a release-visible stability indicator such as Production-ready, Early adopter, or Known issues so production users can choose safer update windows.

## Best Possible Solution

Define a release-owned readiness taxonomy and criteria, require one canonical metadata value for each release, surface it at the top of GitHub release notes, and optionally let update CLI/UI consumers read that same source.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update docs, update CLI registration, changelog, and targeted term searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start with owner-set release metadata rather than inferred update-review heuristics alone.
