## Summary

Keep open. Current main has stronger release validation, beta-first channel semantics, update dry-run/status surfaces, staged package verification, and rollback guidance, but it still does not expose the requested per-release production-ready, early-adopter, or known-issues indicator in release notes, release metadata, or update CLI/docs surfaces. The related open PR #61973 overlaps with upgrade trust, but it proposes heuristic pre-upgrade risk review rather than release-owner-supplied readiness metadata.

## What This Changes

Request a visible production-readiness stability indicator for each OpenClaw release, such as production-ready, early-adopter, or known-issues status in release notes, labels, metadata, or update surfaces.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one canonical source for each release, and expose it first in GitHub release notes. Update/status CLI or UI surfaces can consume that same metadata later if maintainers want runtime visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, channel docs, update docs, update CLI registration, release workflow inputs, changelog structure, and targeted term searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release-owned metadata rather than a heuristic-only update command.
