## Summary

Keep open: current main has release lanes, validation, and safer update mechanics, but it still lacks an official per-release readiness marker, and related stability-policy threads have been routed here as the canonical discussion.

## What This Changes

The issue requests a visible Production-ready, Early adopter, Known issues, or equivalent stability indicator for each OpenClaw release.

## Best Possible Solution

Define a release-owned readiness taxonomy with documented criteria, require it in release notes or release metadata during publish, and let update CLI/UI surfaces consume that official value later.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, workflows, latest release notes, and update CLI/docs show validation lanes but no official per-release readiness label.

## Solution Assessment

Unclear until release owners define the public promise. The requested direction is plausible, but the maintainable solution should start with owner-set release metadata rather than heuristic update-risk inference alone.
