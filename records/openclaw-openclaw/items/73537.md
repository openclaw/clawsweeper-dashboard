## Summary

Keep open. Current main has stronger release validation, beta-first mechanics, staged update safety, and rollback guidance, but it still does not expose an owner-set per-release production-readiness, early-adopter, or known-issues indicator; the related open PR is a heuristic pre-upgrade review command rather than canonical release metadata.

## What This Changes

The issue requests a visible stability/readiness indicator on each OpenClaw release so production operators can tell whether to update, wait, or avoid a release with known issues.

## Best Possible Solution

Define a release-owned readiness taxonomy and criteria, require one canonical metadata value during release creation, and surface it first at the top of GitHub release notes; CLI or UI update surfaces can consume the same metadata later if maintainers want runtime visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update docs, update CLI registration, and targeted searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable path should start with release-owned metadata instead of inferred heuristics alone.
