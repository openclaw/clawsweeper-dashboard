## Summary

Keep this open. Current main has release lanes, validation profiles, package safeguards, update status, and rollback/update docs, but it still does not expose an owner-set per-release readiness marker; the related update-review and Update Manager items do not supersede this narrower release-policy request.

## What This Changes

The issue requests a visible per-release production-readiness or stability marker in release notes, GitHub labels/tags, or release metadata.

## Best Possible Solution

Define a release-owned readiness taxonomy with criteria and require it in release notes or metadata, then let update CLI/UI surfaces consume that source if desired.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: targeted source, docs, workflow, and update CLI inspection shows no visible per-release readiness marker.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The maintainable direction should start with owner-set release metadata or release-note policy, not inferred update-risk heuristics alone.
