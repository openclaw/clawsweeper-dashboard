## Summary

Keep this open. Current main has stable/beta/dev lanes, beta-first validation, release checks, update safeguards, and rollback docs, but it still does not expose an official per-release Production-ready, Early adopter, Known issues, or equivalent readiness marker; related broad stability threads were explicitly routed here as the canonical policy discussion.

## What This Changes

The issue requests a visible production-readiness or stability marker on OpenClaw releases so production operators can decide whether to upgrade.

## Best Possible Solution

Release owners should define a small readiness taxonomy with criteria, make it part of the release publish/release-note process, and let update CLI/UI surfaces consume that same canonical value if maintainers want it visible before upgrade.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, latest release notes, update docs, and update CLI registration show validation and channel mechanics but no official per-release readiness marker.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should be owner-set release metadata/release-note content rather than inferred heuristics alone.
