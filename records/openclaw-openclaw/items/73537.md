## Summary

Keep this open. Current main has beta-first release mechanics, broader release validation, staged package update safeguards, and rollback docs, but it still does not expose an owner-set Production-ready, Early adopter, Known issues, or equivalent per-release stability indicator in release notes, release workflow metadata, update docs, or the update CLI; the related open PR is heuristic pre-upgrade review rather than canonical release readiness metadata.

## What This Changes

The issue requests a visible production-readiness stability indicator for each OpenClaw release.

## Best Possible Solution

Define a release-owned readiness taxonomy and criteria, require one canonical metadata value for each release, and surface it first at the top of GitHub release notes before optionally wiring CLI/UI consumers to the same source.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update docs, update CLI registration, changelog, and targeted term searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start with owner-set release metadata rather than inferred update-review heuristics alone.
