## Summary

Keep this open. Current main has stable/beta/dev release lanes, beta-first validation, package-update safeguards, and rollback docs, but it still does not expose an owner-set per-release Production-ready, Early adopter, Known issues, or equivalent readiness marker; related #61973 and #65011 are adjacent update-safety surfaces rather than canonical release readiness metadata.

## What This Changes

The issue requests a visible production-readiness or stability indicator on OpenClaw releases so operators can decide whether to upgrade production installs.

## Best Possible Solution

Define a release-owned readiness taxonomy and require a readiness value in the release notes or release metadata for every release, then let update CLI/UI surfaces consume that same source if maintainers want client-side display.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior. Current-main verification is high confidence because release docs, release workflows, update docs, and update CLI registration expose channels and validation gates but no per-release readiness marker.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start with owner-set release metadata rather than inferred update-review heuristics alone.
