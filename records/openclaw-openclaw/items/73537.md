## Summary

Keep this open. Current main has stronger release validation, beta-first channel semantics, staged package update verification, rollback guidance, and update status/dry-run surfaces, but it still does not expose the requested per-release production-ready / early-adopter / known-issues indicator in GitHub release notes, release metadata, update docs, or CLI/update surfaces. Related PR #61973 is adjacent update-trust work, but it is open, unmerged, and describes heuristic pre-upgrade risk review rather than a canonical release-readiness label.

## What This Changes

The issue requests a per-release production-readiness stability indicator in release notes, release metadata, or update surfaces.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one canonical source for each release, and expose it first in GitHub release notes; CLI/status/update surfaces can consume the same metadata later if maintainers want in-product visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: docs index, release/update docs, update CLI registration, release workflow inputs, changelog, and exact term searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release process metadata instead of heuristic-only update output.
