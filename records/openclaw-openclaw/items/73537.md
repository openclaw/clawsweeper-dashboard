## Summary

Keep open. Current main has stronger release validation, beta-first release mechanics, staged update safety, and rollback guidance, but it still does not expose an owner-set production-ready, early-adopter, or known-issues indicator in release notes, workflow metadata, update docs, or the update CLI. The related open PR #61973 is adjacent upgrade-trust work, but it proposes heuristic pre-upgrade risk review rather than canonical release-readiness metadata.

## What This Changes

Request to add a user-visible production-readiness or stability indicator to each OpenClaw release so production operators can decide whether to upgrade.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one canonical source for each release, and expose it first in GitHub release notes. Update/status CLI or UI surfaces can consume the same metadata later if maintainers want runtime visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update docs, update CLI registration, changelog structure, and targeted term searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release-owned metadata; #61973's heuristic update review would be complementary rather than a substitute.
