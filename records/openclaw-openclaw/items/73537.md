## Summary

Keep this open. Current main has beta-first release lanes, validation profiles, package-update safeguards, staged rollout and rollback guidance, and an adjacent open pre-upgrade review PR, but it still lacks an owner-set per-release Production-ready, Early adopter, Known issues, or equivalent readiness marker in release notes, release workflows, update docs, or the update CLI.

## What This Changes

The issue requests a visible production-readiness/stability indicator on each OpenClaw release so production users can decide whether to upgrade.

## Best Possible Solution

Define a release-owned readiness taxonomy and criteria, require one canonical metadata value for each release, surface it at the top of GitHub release notes, and optionally let update CLI/UI consumers read that same source.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, release workflows, update docs, update CLI registration, latest release notes, and targeted term searches still show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start with owner-set release metadata rather than inferred update-review heuristics alone.
