## Summary

Keep this open. Current main has stable/beta/dev lanes, beta-first validation, package-update safeguards, and rollback docs, but it still does not expose an owner-set per-release Production-ready, Early adopter, Known issues, or equivalent readiness marker; related #61973 and #65011 are adjacent update-safety surfaces rather than canonical release readiness metadata.

## What This Changes

Request a production-readiness stability indicator on OpenClaw releases so production operators can decide whether to update.

## Best Possible Solution

Define a release-owned readiness taxonomy and required metadata value per release, surface it at the top of GitHub release notes, and let update CLI/UI consumers read that same source if maintainers want client-side display.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: targeted searches plus release docs, release workflows, update docs, and update CLI registration show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start with owner-set release metadata rather than inferred update-review heuristics alone.
