## Summary

Keep open. Current main and v2026.6.8 still route multi-character CJK trigram tokens through MATCH with only a catch-path LIKE fallback, and the open related PRs leave the CJK decomposition/ranking decision unresolved.

## What This Changes

The issue requests better CJK keyword fallback behavior in memory-core so multi-character CJK queries are decomposed instead of matched only as one exact LIKE substring.

## Best Possible Solution

Land one maintainer-approved memory-core fallback strategy with focused CJK regression coverage and redacted real memory_search proof, then close this issue after the selected PR merges.

## Reproduction Assessment

Yes, at source/dependency level: current main routes long CJK trigram tokens through MATCH and only uses broader LIKE fallback on MATCH exceptions, and an in-memory SQLite trigram probe reproduced the zero-row MATCH case. I did not run a full Ollama-backed memory_search scenario in this read-only review.

## Solution Assessment

Unclear: the bug is real, but the best fix depends on maintainer selection of the CJK fallback decomposition and ranking semantics. A safe final path should repair manager-search.ts with focused CJK tests and real memory_search proof, while coordinating with the separate hybrid minScore PR.
