## Summary

Keep open. Current main and the latest release still route multi-character CJK trigram queries through the MATCH path and only use the broader LIKE fallback on MATCH exceptions, while the overlapping open PRs have not merged and do not safely replace this narrower fallback-semantics issue.

## What This Changes

The issue reports wrong or irrelevant memory-core keyword results for multi-character CJK queries when FTS5 trigram search falls back to LIKE matching.

## Best Possible Solution

Keep this issue as the narrow acceptance target until maintainers choose and merge a CJK term-decomposition plus empty-MATCH fallback strategy with focused tests and real memory-search proof.

## Reproduction Assessment

Yes at source level: current keyword planning and fallback code still match the reported long-CJK MATCH and exception-only LIKE fallback behavior. I did not run a live Ollama-backed memory index scenario in this read-only review.

## Solution Assessment

No landed solution is best yet. The related open PRs overlap the cluster, but this issue remains the narrow target for deciding split-term CJK LIKE fallback semantics.
