## Summary

Keep open. Current main and v2026.6.6 still route multi-character CJK trigram queries through MATCH and only use LIKE fallback after MATCH exceptions, while the open related PRs still need maintainer choice on CJK term splitting, ranking, and latency proof.

## What This Changes

The issue requests a memory-core CJK fallback fix so multi-character trigram queries can match relevant partial CJK content instead of exact whole-phrase LIKE or empty MATCH results.

## Best Possible Solution

Use or replace one related PR with a maintainer-approved CJK term decomposition plus empty-MATCH fallback, add focused memory-core regression coverage and latency proof, then close this issue after merge.

## Reproduction Assessment

Yes, at source level. Current main shows multi-character CJK trigram tokens going through MATCH and LIKE fallback only on MATCH exceptions; I did not run a live Ollama-backed memory index in this read-only review.

## Solution Assessment

No landed solution is best yet. The best path is to repair or replace the related open PRs after maintainers choose the CJK split-term, ranking, and latency semantics.
