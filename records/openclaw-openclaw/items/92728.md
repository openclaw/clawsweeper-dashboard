## Summary

Keep open. Current main and the latest release still keep long CJK trigram tokens on the MATCH path and only use the LIKE fallback on MATCH exceptions, while the related open PRs are not merged and still need maintainer judgment on the CJK split-term and fallback semantics.

## What This Changes

The issue requests changing memory-core CJK keyword fallback behavior so multi-character CJK queries do not require an exact whole-phrase or whole-token match.

## Best Possible Solution

Use an existing related PR or a narrow replacement to land one maintainer-approved CJK term decomposition and empty-MATCH fallback strategy, with regression coverage and bounded performance proof, then close this issue after merge.

## Reproduction Assessment

Yes, at source level. Current main shows long CJK trigram tokens going through MATCH and LIKE fallback only on exceptions; this read-only review did not run a live Ollama-backed memory index reproduction.

## Solution Assessment

No landed solution is best yet. The best path is to repair or replace the related open PRs only after maintainers choose the split-term, scoring, and latency semantics for the CJK fallback.
