## Summary

The central behavior is still present on current main and in the latest tagged release: long CJK trigram tokens are not decomposed for LIKE matching, and fallback still does not address the reported precision problem. The issue should remain open because the related open PR explicitly leaves this CJK LIKE precision problem unresolved and ClawSweeper has already marked the fix shape as needing maintainer/product review.

## What This Changes

The issue requests a memory-core CJK LIKE fallback strategy that decomposes multi-character CJK queries so keyword search returns relevant memory rows instead of exact-phrase or issue-discussion rows.

## Best Possible Solution

Keep this issue open as the canonical CJK LIKE precision decision, then land one maintainer-approved manager-search fix with focused CJK regression coverage and latency/relevance proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current `planKeywordSearch` still routes long CJK trigram tokens through MATCH, and `searchKeyword` only runs LIKE fallback on MATCH exceptions rather than empty MATCH results. I did not run a full live Ollama-backed `memory_search` setup in this read-only review.

## Solution Assessment

Unclear as an implementation path: keeping the issue open is right, but the final fix should be a maintainer-approved CJK decomposition and fallback-ranking policy rather than a one-off exact phrase or broad OR fallback.
