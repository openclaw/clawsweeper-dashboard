## Summary

Keep open. Current main and the latest release still return hook 4xx/408/413 validation failures without a hook warning log, and the only direct fix PR was closed unmerged with no newer canonical replacement found.

## What This Changes

The issue requests warning-level journal entries with stable rejection reasons for hook gateway requests that return non-2xx responses.

## Best Possible Solution

Add a centralized, secret-safe hook rejection logging helper in the gateway hook request handler, use stable branch-specific reason codes for rejected hook ingress paths, and cover body, direct-hook, and mapped-hook rejects with focused regression tests.

## Reproduction Assessment

Yes, at source level. The malformed JSON path reaches `readJsonBody`, sends a 400/408/413 response, and returns before any `logHooks.warn`; direct and mapped validation branches show the same no-log return pattern.

## Solution Assessment

Yes. Handler-level secret-safe rejection diagnostics are the narrow maintainable fix; docs or plugin changes would not restore server-side visibility for rejected hook ingress requests.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/69192. Current `src/gateway/server/hooks-request-handler.ts` returns hook ingress 4xx/408/413/404/405/401 failures without a warning-level hook journal entry except for auth throttling and thrown mapping exceptions. Add a centralized secret-safe rejection logging helper in the hook request handler and call it for non-2xx hook requests before returning. Use stable branch-specific reason codes for body read failures, query-token rejection, method/path/auth rejection, direct wake/agent validation failures, mapped validation failures, channel/agent/session-policy failures, and mapped-prefix failures. Do not log hook tokens, Authorization headers, raw request bodies, rendered messages, recipients, arbitrary payload fields, or secrets; prefer bounded metadata such as path, status, reason, content type, body type/byte count when safely available, and existing trace/request context if already available. Add focused regression coverage in hook server/request-handler tests that asserts warning logs for malformed JSON, direct validation rejection, mapped `hook mapping requires message`, and session/prefix rejection paths. Add a `CHANGELOG.md` fix entry because this changes user-visible gateway diagnostics. Do not introduce a new config option or move hook ownership out of the gateway handler.
