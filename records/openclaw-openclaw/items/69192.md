## Summary

Keep open. Current main still has hook rejection branches that return 4xx/408/413 without a `logHooks.warn` journal entry, the latest release has the same behavior, and the only direct fix PR was closed unmerged.

## What This Changes

The issue asks the gateway to write secret-safe warning journal lines with stable rejection reason codes whenever hook requests return non-2xx responses.

## Best Possible Solution

Add a centralized secret-safe hook rejection logging helper in the gateway hook request handler, use stable reason codes for the main rejection branches, and cover body, direct-hook, and mapped-hook rejects with focused regression tests.

## Reproduction Assessment

Yes, at source level. The malformed JSON path reaches `readJsonBody`, sends a 400/408/413 response, and returns before any `logHooks.warn`; adjacent tests already prove the status paths but not the missing journal diagnostics.

## Solution Assessment

Yes. Handler-level secret-safe rejection diagnostics are the narrow maintainable fix; docs or plugin changes would not restore server-side visibility for rejected hook ingress requests.
