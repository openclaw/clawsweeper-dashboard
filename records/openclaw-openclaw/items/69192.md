## Summary

Keep open. Current main still returns hook 400/408/413 validation failures from the JSON reader, direct wake/agent validation, and mapped-hook validation without a warning journal entry; the only warning calls in the handler are auth throttling and thrown mapping exceptions. The related fix PR #69267 was closed unmerged, so it does not supersede this issue.

## What This Changes

Add warning-level gateway journal diagnostics for non-2xx hook request rejections, with stable rejection reason codes and less ambiguous mapped-hook validation failures.

## Best Possible Solution

Add a narrow hook-rejection logging helper in the gateway hook request handler and cover the main rejection branches with tests, preserving current statuses and response bodies except for narrowly distinguished mapping errors where reliable.

## Reproduction Assessment

Yes. The malformed JSON repro maps directly to `readJsonBody` returning `ok: false`, followed by a 400 `sendJson` response without a `logHooks.warn` call; existing tests already cover the status path but not the journal line.

## Solution Assessment

Yes. Handler-level diagnostics with focused tests is the narrow maintainable fix; a docs-only change or plugin surface would not restore server-side visibility for rejected ingress requests.
