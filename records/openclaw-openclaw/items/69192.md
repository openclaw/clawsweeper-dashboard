## Summary

Keep open. Current main still returns hook 4xx/408/413 responses from multiple ingress validation branches without a warning journal entry; the only warning calls in the handler are for auth throttling and thrown mapping exceptions. The related fix PR #69267 attempted this exact logging work but was closed unmerged, so it does not supersede the issue.

## What This Changes

Add warning-level gateway diagnostics for non-2xx hook rejections, with stable rejection reason codes and less ambiguous mapped-hook validation failures.

## Best Possible Solution

Add a narrow hook-rejection logging helper in the gateway hook request handler and cover the main rejection branches with tests. Keep responses and auth behavior unchanged, use stable reason codes plus safe metadata, and only split mapping error strings where the code can reliably distinguish the failing branch.

## Reproduction Assessment

Yes. The malformed JSON repro maps directly to `readJsonBody` returning an error and the handler sending a 400 response without any `logHooks.warn` call; current tests already exercise the 400 status path.

## Solution Assessment

Yes. Handler-level diagnostics plus focused tests are the maintainable fix; a plugin or docs-only change would not restore server-side journal visibility for rejected ingress requests.
