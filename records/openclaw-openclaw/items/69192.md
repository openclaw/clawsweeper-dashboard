## Summary

Keep open. Current main and the latest release still have hook rejection branches that return 4xx/408/413 without a `logHooks.warn` journal entry, and the only linked fix PR was closed unmerged.

## What This Changes

The issue requests secret-safe warning-level gateway journal entries, with specific rejection reasons, for hook requests that fail validation or otherwise return non-2xx responses.

## Best Possible Solution

Add a small secret-safe hook rejection logging helper in the gateway hook request handler, use stable reason codes for the main rejection branches, and cover malformed body, direct-hook, and mapped-hook rejects with focused regression tests.

## Reproduction Assessment

Yes. The malformed JSON path reaches `readJsonBody`, immediately sends a 400/408/413 response, and returns without `logHooks.warn`; adjacent tests already exercise the response status paths but not journal diagnostics.

## Solution Assessment

Yes. Handler-level rejection diagnostics with careful redaction is the narrow maintainable fix; docs or plugin changes would not restore server-side visibility for rejected ingress requests.
