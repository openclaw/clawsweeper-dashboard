## Summary

This should stay open: current main and the latest release still use the fuzzy-normalized whole-file content as both the write base and diff base, so a fuzzy edit can silently rewrite untouched lines and hide those changes from the preview/model diff.

## What This Changes

The issue reports that fuzzy `edit` matching rewrites unrelated file content outside the matched edit range and hides those collateral changes from the model-facing diff.

## Best Possible Solution

Fix the fuzzy path so normalized text is used only to find or validate matches while writes and diffs remain based on the original LF-normalized file; add regression coverage for untouched Unicode, trailing whitespace, and the displayed diff.

## Reproduction Assessment

Yes. Source inspection on current main shows the reported path directly: any fuzzy match can switch the whole base to normalizeForFuzzyMatch, and the issue body provides a focused Vitest reproduction; I did not create that test file because this review is read-only.

## Solution Assessment

Yes. The suggested direction is the right maintainable boundary: preserve the original file as the write/diff base and limit fuzzy normalization to match discovery, with tests covering both output bytes and preview honesty.
