## Summary

Keep this open: current main and the latest stable release still contain the whole-file fuzzy-normalization write/diff path, and the linked closing fix at https://github.com/openclaw/openclaw/pull/90060 is still open rather than merged.

## What This Changes

Fuzzy edit matching should locate approximate oldText without normalizing and rewriting unrelated file content or hiding collateral changes in the edit diff.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/90060 with the scoped fuzzy-offset fix and regression coverage, then close this issue as fixed.

## Reproduction Assessment

Yes. Source inspection on current main shows that any fuzzy match can switch baseContent to normalizeForFuzzyMatch(normalizedContent), after which edit.ts writes and diffs from that base; I did not run the supplied Vitest because this review is read-only.

## Solution Assessment

Yes. The best fix is to keep the original LF-normalized content as the write/diff base and use fuzzy normalization only for match discovery plus offset mapping, which the linked PR already implements for review.
