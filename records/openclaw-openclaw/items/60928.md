## Summary

Keep open. The PR addresses a real still-unimplemented CLI skills snapshot refresh gap, but the proposed patch should not merge as-is because it starts a persistent skills watcher in one-shot CLI runs and the branch is now dirty against current main.

## What This Changes

The PR adds a forced skills snapshot refresh flag for CLI agent runs, calls the skills watcher and snapshot-version bump before rebuilding snapshots, and extends the agent command test to assert those calls.

## Best Possible Solution

Keep the manual refresh idea, but adapt it to current main by bumping the skills snapshot version only for the trusted CLI agent entrypoint before reading the version, without starting the persistent watcher from one-shot CLI or changing ingress watcher lifecycle.

## Reproduction Assessment

Yes. The PR body gives a concrete extraDirs plus reused session reproduction, and current main code confirms the CLI path can reuse a persisted skillsSnapshot solely from the stored snapshot version unless some other lifecycle has bumped it.

## Solution Assessment

No. Forcing a manual snapshot-version bump in the CLI path is the right direction, but starting ensureSkillsWatcher there is not the narrowest maintainable fix because chokidar watchers are persistent by default and this one-shot path has no teardown.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid starting the skills watcher in one-shot CLI runs:** `src/agents/agent-command.ts:506`
  - body: The new ensureSkillsWatcher call runs in the agent command path and creates a Chokidar watcher with persistent:true by default, but this CLI flow has no teardown before returning. Normal `openclaw agent --local` runs can then keep the Node event loop alive after printing the reply, contrary to the one-shot CLI contract. The stale snapshot fix only needs the manual version bump before reading the snapshot version, so skip the watcher here or close it before return.
