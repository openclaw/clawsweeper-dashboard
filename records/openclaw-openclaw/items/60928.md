## Summary

Keep open: the PR addresses a real CLI stale-skills-snapshot bug, but the current patch is not merge-ready because it starts the persistent skills watcher from the one-shot CLI path and needs to be ported onto current main's refresh-state seam.

## What This Changes

The PR makes `openclaw agent` force a skills snapshot refresh for CLI runs and adds focused assertions in the agent command test.

## Best Possible Solution

Land a CLI-only manual skills snapshot version bump before the snapshot version read, preserving watcher ownership in long-lived session paths.

## Reproduction Assessment

Yes. Current main source shows cached `skillsSnapshot` reuse when the version/filter match, and the PR body provides a fixed-session extra-dir CLI before/after verification; I did not run the live flow in this read-only review.

## Solution Assessment

No. The manual version bump is the right fix direction, but starting `ensureSkillsWatcher` in the one-shot CLI path is not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid starting the skills watcher from CLI agent runs:** `src/agents/agent-command.ts:503`
  - body: The added `ensureSkillsWatcher` call creates a chokidar watcher in the one-shot CLI path with no corresponding teardown. Since `--local` and embedded fallback runs are documented as one-shot, and chokidar watchers are persistent by default, this can keep `openclaw agent --local` alive after the reply; keep the manual version bump but avoid starting the watcher here.
  - confidence: 0.91
