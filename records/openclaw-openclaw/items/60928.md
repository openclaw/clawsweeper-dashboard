## Summary

Keep open: this is a focused fix for a real stale skills-snapshot CLI path, but the current PR should not merge because it starts the persistent skills watcher from a one-shot CLI run with no teardown.

## What This Changes

The PR adds a forced skills snapshot refresh for trusted `openclaw agent` CLI runs and extends the agent command test to assert the refresh calls.

## Best Possible Solution

Land a CLI-only manual skills snapshot version bump using the refresh-state seam, while leaving watcher lifecycle ownership in the long-lived gateway/session path.

## Reproduction Assessment

Yes. Source inspection shows current main can reuse a persisted `skillsSnapshot` when the version has not advanced, and the PR body provides fixed-session `skills.load.extraDirs` before/after CLI proof with after-fix live output.

## Solution Assessment

No. The manual snapshot version bump is the right narrow fix, but starting `ensureSkillsWatcher` from the one-shot CLI path conflicts with the documented lifecycle and should be removed or explicitly torn down.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Avoid starting the persistent watcher in one-shot agent runs:** `src/agents/agent-command.ts:503`
  - body: The new `ensureSkillsWatcher` call starts a chokidar watcher before the snapshot version read, but the CLI `--local` path is documented as one-shot and this flow does not close the watcher. Chokidar defaults `persistent` to true, so this can keep `openclaw agent --local` alive after printing the reply; keep the manual version bump, but do not start the watcher from this path.
  - confidence: 0.92
