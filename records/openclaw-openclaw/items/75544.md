## Summary

Keep open. This draft, maintainer-labeled PR is useful but not ready to merge because the approve fallback still mutates local pairing state before proving it is the same request named by the active Gateway.

## What This Changes

The PR adds a request-id-aware guard for `openclaw devices` local pairing fallback, regression tests for list/approve mismatch cases, and an Unreleased changelog entry.

## Best Possible Solution

Update the approve fallback to fail closed before local mutation whenever the Gateway supplies a request id that does not exactly match the explicit local approval target, and add a regression test for the different-local-request case.

## Reproduction Assessment

Yes. A focused mock path can reproduce it: make the Gateway reject approval with `device pairing required (requestId: req-profile)`, make local state contain `req-default`, then run explicit approve for `req-default`; the patched code approves local state before checking the gateway id. I did not run tests because this review was read-only.

## Solution Assessment

No. The list-side guard is on the right path, but approve should validate the Gateway request id before calling `approveDevicePairing`, or otherwise only approve the exact request named by the Gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate the gateway request before approving locally:** `src/cli/devices-cli.ts:221-224`
  - body: When the Gateway error names `req-profile` but the operator passes `req-default`, this line approves `req-default` from local state before any comparison with the Gateway request. If that local request exists, the new mismatch error never runs and the CLI still mutates the wrong profile/state-dir, which is the case this PR is trying to prevent. Check the Gateway request id before calling `approveDevicePairing`, or require it to match the explicit request id.
  - confidence: 0.9
