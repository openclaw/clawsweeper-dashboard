## Summary

Keep open. This maintainer-labeled, member-authored PR is not eligible for cleanup closure, and the proposed patch still has a blocking approve-fallback ordering bug: it can approve a local pending request before validating that it is the request named by the active Gateway.

## What This Changes

The PR adds request-id-aware local fallback guards for `openclaw devices`, regression tests for fallback state mismatches, and a changelog entry.

## Best Possible Solution

Keep the list-side guard, but make explicit approve fail closed before local mutation whenever the Gateway supplies a request id that differs from the operator’s explicit approval target or is absent from local state; then add a mismatch regression and refresh the branch for automerge.

## Reproduction Assessment

Yes. A focused mocked path can reproduce it by making the Gateway reject with `device pairing required (requestId: req-profile)`, making local state contain `req-default`, and running explicit approve for `req-default`; the PR code approves locally before checking the Gateway id.

## Solution Assessment

No. The list-side change is the right direction, but the approve path should validate the Gateway request id before calling `approveDevicePairing`, or otherwise load/check local state first and only mutate the exact Gateway-named request.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate the gateway request before approving locally:** `src/cli/devices-cli.ts:221-226`
  - body: When the Gateway error names `req-profile` but the operator passes an existing local `req-default`, this line approves `req-default` before comparing against `fallback.details.requestId`. That mutates the wrong local pairing store and skips the mismatch guard, so the approve fallback needs to fail closed before local approval.
  - confidence: 0.9
