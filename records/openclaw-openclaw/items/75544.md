## Summary

Keep open. This is a maintainer-labeled PR from a MEMBER author, and the current head still has a blocking approve-fallback ordering bug: it can mutate local pairing state before validating the Gateway-reported request id.

## What This Changes

The PR adds request-id-aware local pairing fallback checks for `openclaw devices list/approve`, regression coverage for mismatch cases, and an Unreleased changelog entry.

## Best Possible Solution

Update the approve fallback to fail closed before local mutation whenever the Gateway supplies a request id that does not exactly match the explicit approval target, add a regression for Gateway `req-profile` with local `req-default`, then rebase the branch and rerun the targeted CLI/device checks.

## Reproduction Assessment

Yes. A focused mock path can reproduce it: make the Gateway reject approval with `device pairing required (requestId: req-profile)`, make local state contain `req-default`, then run explicit approve for `req-default`; the patched code approves locally before comparing the Gateway id.

## Solution Assessment

No. The list-side guard is the right direction, but approve should validate the Gateway request id before calling `approveDevicePairing`, or otherwise only approve the exact request named by the Gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate the gateway request before approving locally:** `src/cli/devices-cli.ts:221-224`
  - body: When the Gateway error names `req-profile` but the operator passes `req-default`, this line calls `approveDevicePairing(requestId, ...)` before comparing against `fallback.details.requestId`. If `req-default` exists locally, the fallback approves and mutates the wrong store, and the new mismatch error never runs. Check the Gateway request id before local approval.
  - confidence: 0.9
