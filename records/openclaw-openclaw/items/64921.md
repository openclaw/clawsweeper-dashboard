## Summary

Keep open. Current main still routes plugin text-transform reads through the createRequire-based active runtime helper, while the documented write side and CLI/provider prompt consumers expect registered transforms to be applied; the related fix PR #65331 remains open and unmerged.

## What This Changes

The issue reports that plugin-registered text transforms are not applied in the main process because runtime lookup cannot access the active plugin registry.

## Best Possible Solution

Fix the runtime text-transform lookup so bundled main-process code can read the active plugin registry through a direct import or another stable generic seam, keep the external-plugin fallback if still needed, and add focused regression coverage for CLI and provider prompt application.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: a plugin can register `textTransforms`, but current main resolves them through a helper that only require-probes stable runtime filenames before the CLI/provider prompt paths consume the result.

## Solution Assessment

Yes as a direction. The narrow maintainable fix is to make `resolveRuntimeTextTransforms()` read the current active registry in the bundled main process while retaining any needed fallback, then cover that behavior with targeted tests rather than changing unrelated plugin loading.
