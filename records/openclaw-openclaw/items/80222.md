## Summary

Keep open: current main still has the early preflight-compaction bug and this PR is a focused fix with real behavior proof, but the patch should be adjusted because it can bypass explicit context-token caps.

## What This Changes

The PR changes preflight compaction thresholding to consider persisted `SessionEntry.contextTokens` and adds a regression test for long-context false-positive compaction.

## Best Possible Solution

Preserve explicit configured caps while using persisted session context windows only as an authoritative larger runtime signal when no lower cap applies.

## Reproduction Assessment

Yes. Source inspection shows current main can threshold preflight compaction against a 128k resolved window while the active session entry records 272k, and the PR's added test models that false-positive path; I did not run tests because this review is read-only.

## Solution Assessment

No. The general direction is right, but `Math.max(...)` is too broad because it treats persisted session context as higher priority than explicit context-token caps that current docs and tests define as budget limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve explicit context-token caps:** `src/auto-reply/reply/agent-runner-memory.ts:498-501`
  - body: This `Math.max(...)` lets a stale or previously larger `SessionEntry.contextTokens` override `agentCfgContextTokens`. That regresses the existing cap contract: if an operator lowers an agent/session budget to 128k while the entry still carries 272k, preflight compaction can be skipped until the larger threshold and the next run may exceed the configured budget. Keep persisted session context as a larger runtime signal only when it does not bypass an explicit configured cap, and add a regression case for that cap.
  - confidence: 0.83
