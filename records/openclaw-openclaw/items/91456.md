## Summary

Keep open. Current main and v2026.6.6 intentionally keep a Telegram lane guarded when a timed-out spooled handler does not settle after reply-fence abort, matching the supplied live failure signature; the June 12 buffered-replay fix improves adjacent processing but does not remove this terminal state. Likely related people are Josh Avant, who introduced and recently extended this recovery path, and Peter Steinberger, who has substantial history in the same polling and backlog code.

## What This Changes

Fix Telegram direct-message ingress lanes that can remain guarded after a timed-out or non-abortable handler, preventing later messages from reaching the agent session.

## Best Possible Solution

Make the timed-out update terminal and fence every late reply, delivery, and session mutation from its old handler, then release the process-global lane guard and drain later durable updates in order without deleting dead-letter evidence.

## Reproduction Assessment

Yes. The discussion provides a live released-build reproduction with the exact shipped diagnostic, and current-main source plus focused regression coverage provides a concrete path using a handler that ignores reply abort beyond the grace period.

## Solution Assessment

No. Making sendChatAction best-effort only addresses one trigger, and clearing tombstones addresses the wrong ownership layer; the maintainable fix must establish a terminal late-effect fence before releasing the process-global lane guard.

- Deleting failed tombstones broadly, as proposed in the related open PR, could discard durable failure evidence or permit unsafe replay without resolving the active-handler guard.
