## Label Justifications

- `P1`: A single stuck Telegram handler can make an active direct-message workflow appear dead until the whole gateway process is restarted.
- `impact:message-loss`: Later Telegram updates can remain spooled indefinitely instead of reaching the agent while the lane guard persists.
- `impact:session-state`: Queued direct messages are not incorporated into the intended agent session during the guarded state.

## Summary

Current main and the latest release still preserve the exact guarded-lane failure when a timed-out Telegram handler does not settle after reply abort. The June 12 buffered-replay fix improves adjacent deferred delivery ownership but does not release a permanently hung active handler, and the new live report confirms provider restarts do not recover it.

## What This Changes

Fix Telegram direct-message lanes that remain guarded after a timed-out or non-aborting handler, preventing later inbound messages from reaching the agent session.

## Best Possible Solution

Add a bounded stale-handler isolation contract that terminally fails the stuck update, fences all obsolete reply/session/delivery effects, releases the lane, and drains later durable updates exactly once across provider restart and full gateway lifetime.

## Reproduction Assessment

Yes, at source level with high confidence: current main contains the exact diagnostic branch and a focused test that blocks the next same-lane update until the old handler settles. The discussion also provides a live released-version reproduction, although this read-only review did not repeat the fault injection on current main.

## Solution Assessment

No, the outbound-only suggestions are not sufficient because the same terminal state is independently reached by an inbound handler that never settles. The best fix must isolate stale handler side effects before bounded lane release, not merely ignore chat-action failures or delete the guard.
