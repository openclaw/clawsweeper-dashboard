## Summary

Keep open. Current main and v2026.6.6 still contain the guarded Telegram lane behavior, and the newer live comment broadens the failure from outbound network timeouts to handler-timeout/orphaned-reply paths that can leave later DMs spooled but undelivered.

## What This Changes

The issue reports Telegram direct-message lanes staying guarded after send or handler timeouts, delaying or dropping later inbound DMs.

## Best Possible Solution

Define a Telegram-owned recovery contract that fails or releases one wedged update, drains later same-lane inbound messages, and reconciles or safely quarantines uncertain outbound sends without adding a broad fallback stack.

## Reproduction Assessment

Yes at source level: current source and tests show a timed-out Telegram lane can stay guarded with a later same-lane update pending, and the issue now has a live comment with the same signature. I did not run a live Telegram network or handler-timeout reproduction in this read-only review.

## Solution Assessment

Unclear as written: making only sendChatAction best-effort is too narrow because the live repro reaches the same state through handler timeout. The best fix should cover lane release/fail-one-update behavior and outbound delivery reconciliation together.
