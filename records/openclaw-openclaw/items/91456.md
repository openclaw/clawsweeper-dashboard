- `impact:message-loss`: The affected user-visible behavior is delayed, missing, or never-processed Telegram direct messages after a lane becomes guarded.
- `impact:session-state`: The failure is tied to per-lane handler and session/reply-fence state remaining guarded across later updates and provider restarts.

## Summary

Keep open. Current main still has the Telegram isolated-ingress guarded-lane path, adjacent tests preserve the later-same-lane pending-update behavior, and related open work is not a safe canonical fix yet.

## What This Changes

The issue reports Telegram polling direct-message lanes staying guarded after outbound send or handler timeouts, delaying or dropping later inbound DMs.

## Best Possible Solution

Fix Telegram isolated-ingress recovery so a failed or unabortable turn cannot indefinitely block later same-lane inbound updates, while preserving dead-letter and outbound reconciliation semantics with live Telegram proof.

## Reproduction Assessment

Yes at source level: current tests and code show a timed-out Telegram lane can remain guarded with a later same-lane update pending. I did not run a live Telegram network-timeout reproduction.

## Solution Assessment

Unclear as an implementation path: the issue is real, but a sendChatAction-only fix is too narrow because the same guarded-lane end state is also reported from handler timeout. The best fix should address lane release/restart and outbound reconciliation together.
