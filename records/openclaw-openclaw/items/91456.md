## Label Justifications

- `P1`: The issue describes a real Telegram DM workflow where inbound messages can stop reaching the agent after lane timeout/recovery failures.
- `impact:message-loss`: The observable failure is delayed or missing Telegram direct messages in the target agent session.
- `impact:session-state`: The failure is tied to stale per-lane handler/session guard state that can persist across later updates.

## Summary

Keep open. Current main and the latest release still contain source- and test-backed behavior where a timed-out Telegram spooled handler can keep the lane guarded and leave later same-lane updates pending; the merged https://github.com/openclaw/openclaw/pull/92281 fixed a related buffered replay path, but not this stale active-handler path.

## What This Changes

Prevent Telegram direct-message lanes from remaining guarded after send or handler timeouts so later inbound DMs continue reaching the agent session.

## Best Possible Solution

Fix Telegram isolated-ingress lane recovery so a timed-out or aborted handler cannot indefinitely block later same-lane DMs, while preserving duplicate-send safety through explicit reconciliation and visible channel status.

## Reproduction Assessment

Yes, at source level: current tests and source show a timed-out Telegram lane can remain guarded with a later same-lane update pending. I did not run a live Telegram network-timeout reproduction.

## Solution Assessment

Unclear. The related buffered replay fix is useful, but the remaining best fix needs a maintainer-chosen contract for stale active-handler release, retry, and duplicate-send safety.
