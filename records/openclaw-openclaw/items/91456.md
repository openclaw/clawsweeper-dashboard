## Label Justifications

- `P1`: The issue describes a real Telegram DM workflow where later inbound messages can stop reaching the agent until process-level recovery.
- `impact:message-loss`: The observed failure is delayed or missing Telegram direct messages despite updates reaching the gateway/spool path.
- `impact:session-state`: The failure is tied to persistent lane/session guard state and stuck handler state across later updates and restarts.

## Summary

Keep open. Current main and the latest release still contain the Telegram isolated-ingress path that keeps a timed-out same-lane handler guarded, and the adjacent regression test explicitly expects the later update to remain pending; the merged spooled-buffered replay PR did not close this issue, and the open tombstone-clearing PR is not a safe canonical fix.

## What This Changes

The issue asks OpenClaw to prevent Telegram polling DM lane timeouts or stuck reply handlers from blocking or dropping later direct messages.

## Best Possible Solution

Fix the Telegram isolated-ingress recovery contract so one timed-out or unabortable handler fails that update without indefinitely guarding the lane, while preserving dead-letter/tombstone semantics and adding live Telegram proof.

## Reproduction Assessment

Yes at source level, but not as a fresh live current-main run. Current-main tests and source show a timed-out same-lane Telegram update can leave the later update pending, and issue comments provide sanitized live signatures from released builds.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix needs maintainer agreement on the lane-release and reconciliation contract; the open tombstone-clearing PR is not the safest solution.
