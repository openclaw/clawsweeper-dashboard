## Label Justifications

- `P1`: The issue describes a real Telegram DM workflow where inbound messages can stop reaching the agent until recovery or a full gateway restart.
- `impact:message-loss`: The observed failure is delayed or missing Telegram direct messages after a guarded lane blocks later updates.
- `impact:session-state`: The failure is tied to Telegram lane/session guard state that remains active across later same-lane updates.

## Summary

Keep open. Current main has partial Telegram isolated-ingress recovery, and direct typing cues are now best-effort in at least the early DM path, but the central guarded-lane failure remains: a timed-out same-lane spooled handler can stay in the process-global active-handler map when the handler does not settle, leaving later updates pending.

## What This Changes

The issue reports that Telegram direct-message lanes can stay guarded after send or handler timeouts, causing later direct messages to be delayed or dropped.

## Best Possible Solution

Keep this open and have Telegram owners define a bounded lane-release and delivery-reconciliation contract that lets later inbound DMs drain after a failed update without duplicate sends or silent same-lane ordering breaks.

## Reproduction Assessment

Yes at source level: current main has a same-lane blocked update path and a test that leaves the later update pending when the old handler does not stop after timeout. I did not run a live Telegram network-timeout repro in this read-only review.

## Solution Assessment

Unclear as an automated fix: making sendChatAction best-effort is only a partial mitigation, while the durable solution needs an owner-approved policy for releasing guarded lanes and reconciling uncertain outbound deliveries.
