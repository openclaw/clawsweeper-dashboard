## Summary

Keep open. Current main still has browser-plugin timeout/control propagation gaps that match the report: `navigate` accepts schema-level controls but drops them before the client and route layers, while snapshot timeout propagation is still absent on main and only partially covered by the open #75702 subset PR. https://github.com/openclaw/openclaw/issues/72653

## What This Changes

The issue requests a browser-plugin fix so `browser open`, `browser snapshot`, and `browser navigate` do not time out when the configured CDP endpoint is reachable and direct Playwright CDP works.

## Best Possible Solution

Implement a browser-plugin-owned timeout/load-state propagation fix for navigate and snapshot, validate the open-tab remote CDP budget path, coordinate with #75702 for the snapshot subset, and preserve existing SSRF guards, existing-session limitations, and default timeout values unless tests justify a change.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path with a healthy CDP endpoint and direct Playwright success; current-main inspection also gives a unit-level reproduction for dropped `timeoutMs`/`loadState` controls on navigate and snapshot.

## Solution Assessment

Yes. A targeted browser plugin propagation repair is the narrow maintainable solution; a blanket timeout increase, generic retry loop, or core migration would leave the route/helper gaps and misleading transport timeout behavior intact.
