## Summary

Keep open. Current main still has source-level browser plugin timeout/control gaps matching this report: navigate drops caller timeout/load-state controls, snapshot does not carry a timeout through the agent/tool/route/helper stack, and the Chrome MCP open path still uses fixed internal budgets even when the outer request has a timeout.

## What This Changes

The issue asks for browser open, snapshot, and navigate to stop timing out when CDP status and direct Playwright CDP connections are healthy.

## Best Possible Solution

Fix this inside the browser plugin by threading normalized timeoutMs/loadState through open, navigate, and snapshot boundaries while preserving SSRF checks, existing-session limits, and current default budgets unless tests justify a narrower default change.

## Reproduction Assessment

Yes, source-level. The issue gives a concrete live CDP reproduction, and current main statically shows the dropped browser tool controls for navigate and snapshot plus fixed Chrome MCP open budgets; I did not run the live Ubuntu setup.

## Solution Assessment

Yes. A targeted browser-plugin propagation and timeout-boundary repair is the narrow maintainable solution; a blanket timeout increase or retry loop would leave the route/helper gaps and misleading failure mode intact.
