## Summary

Keep open. Current main still leaves the reported browser timeout class unresolved: a green CDP readiness probe does not exercise the later operation paths, navigate still drops caller timeout/load-state controls, snapshot still uses a fixed client timeout, and open can still hit downstream navigation waits. The narrower open issues #47203 and #72934 overlap with navigate and snapshot symptoms, but they do not fully supersede this broader open/snapshot/navigate report.

## What This Changes

The issue reports that browser open, snapshot, and navigate time out even though the same CDP endpoint passes status, curl, and direct Playwright checks.

## Best Possible Solution

Keep this open until maintainers either make it the canonical broad CDP operation-timeout tracker or split it into linked open, navigate, and snapshot work. The end state should wire operation-specific timeout/load-state behavior through the browser tool, client, route, and Playwright/CDP layers while keeping navigation SSRF checks intact and improving timeout classification.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path with a local CDP endpoint, curl and direct Playwright success, and OpenClaw browser operation timeouts; current-main source still matches the timeout-prone boundaries described.

## Solution Assessment

Unclear for the issue as written because it bundles open, navigate, and snapshot failures. The safest technical direction is targeted timeout/load-state plumbing and clearer error classification, not a blanket timeout increase or broad CDP transport rewrite.
