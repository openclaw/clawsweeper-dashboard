## Summary

Keep open. Current main still matches the reported working-CDP-but-tool-timeout pattern: navigate and snapshot drop caller controls, and open only applies timeout at the outer request boundary while internal browser operations keep fixed budgets; the related snapshot PR is useful but does not cover this broader issue.

The issue requests that browser open, navigate, and snapshot work through OpenClaw when the configured CDP endpoint is reachable and direct Playwright CDP control succeeds.

## Best Possible Solution

## Solution Assessment

Fix the browser plugin timeout/control propagation bug reported in https://github.com/openclaw/openclaw/issues/72653. Current main accepts `timeoutMs` and `loadState` at the browser tool surface, but open/navigate/snapshot do not propagate those controls end to end. Keep the fix inside `extensions/browser/`; preserve existing default timeout values unless tests prove a specific boundary needs slack; do not relax SSRF policy, add new config, or move plugin-owned behavior into core. Coordinate with https://github.com/openclaw/openclaw/pull/75702: if it is still open, avoid duplicating its snapshot-only work; if it has merged, build on it. Add regression coverage for host and node proxy paths showing explicit and default timeoutMs reach navigate/open/snapshot route/helper layers, and loadState reaches the navigate implementation where applicable. Add a single-line user-facing CHANGELOG.md fix entry unless the final diff is tests-only.
