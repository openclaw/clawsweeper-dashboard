## Summary

Keep open. Current main still has the browser operation-timeout class described here: CDP readiness can pass while navigate and snapshot flow through fixed or dropped timeout controls, and the narrower open reports #47203 and #72934 do not fully supersede this broader open/snapshot/navigate report.

## What This Changes

The issue reports that `browser open`, `browser snapshot`, and `browser navigate` time out through OpenClaw even though the same local CDP endpoint is reachable and direct Playwright CDP usage works.

## Best Possible Solution

Keep this open as the broad browser CDP operation-timeout tracker, or explicitly split it into linked open, navigate, and snapshot work. The end state should propagate operation-specific timeout and load-state controls through the browser tool, client, HTTP routes, and Playwright/CDP layers while preserving navigation SSRF guards and producing errors that distinguish service reachability from operation timeout.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with a local CDP endpoint, green `browser status`, successful curl/direct Playwright CDP checks, and OpenClaw browser operation timeouts; current-main source still contains the same timeout-prone boundaries.

## Solution Assessment

No current solution is present. The safer direction is targeted timeout/load-state propagation plus clearer timeout classification, not a blanket timeout increase or a broad CDP transport rewrite.
