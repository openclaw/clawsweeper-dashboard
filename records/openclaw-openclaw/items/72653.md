## Summary

Keep open. Current main still has browser-plugin timeout propagation gaps after a green CDP readiness probe: navigate drops caller timeout controls before the proxy/client and route layers, and snapshot still lacks end-to-end timeout wiring through the tool, client query, route plan, Chrome MCP snapshot, and Playwright snapshot helpers. The related #75702 work is useful but explicitly scoped to #72934's snapshot subset and does not supersede this broader Linux/CDP open+navigate+snapshot report.

## What This Changes

The issue reports that OpenClaw browser open, snapshot, and navigate time out on Ubuntu even though browser status shows CDP ready and direct Playwright CDP access works.

## Best Possible Solution

Implement an extension-owned browser plugin fix that normalizes and threads operation timeouts and navigation wait-state controls through the agent tool, CLI/client, node proxy, /navigate and /snapshot routes, Chrome MCP helpers, and Playwright helpers while preserving SSRF pre/post navigation guards and existing-session limits.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path with a local CDP endpoint, successful curl/direct Playwright CDP checks, and OpenClaw browser action timeouts; current-main source still matches timeout-prone boundaries for navigate and snapshot.

## Solution Assessment

Yes, the best direction is a targeted timeout/load-state propagation repair inside the browser plugin. A blanket timeout increase or broad CDP transport rewrite would be less maintainable and would not cover the dropped-control seams shown in current main.
