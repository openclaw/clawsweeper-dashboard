## Summary

Keep open. Current main still has the same browser operation-timeout gaps after a green CDP readiness probe: navigate drops timeout/load-state controls, snapshot keeps a fixed client timeout, and remote/open paths can still hit Playwright navigation waits. The related #47203 and #72934 reports cover narrower navigate and snapshot subsets, but they do not fully supersede this broader open/snapshot/navigate report.

## What This Changes

The issue reports that the browser tool times out on open, snapshot, and navigate even though the same CDP endpoint passes status, curl, and direct Playwright checks.

## Best Possible Solution

Keep this open until maintainers either make it the canonical broad CDP operation-timeout tracker or split it into linked open, navigate, and snapshot work. The end state should propagate operation-specific timeout/load-state controls through the browser tool, client, route, and Playwright/CDP layers while preserving navigation SSRF guards and clearer timeout errors.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with a local CDP endpoint, curl and direct Playwright success, and OpenClaw browser operation timeouts; current-main source still matches the timeout-prone boundaries described.

## Solution Assessment

No current solution is present. The safest fix direction is targeted timeout/load-state propagation plus clearer timeout classification, not a blanket timeout increase or a broad CDP transport rewrite.
