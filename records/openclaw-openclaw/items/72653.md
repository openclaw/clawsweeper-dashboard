## Summary

Keep open. Current main still has browser operation-timeout gaps after a green CDP readiness probe: the agent navigate path drops timeout/load-state controls, the route still invokes Playwright navigation without those controls, Playwright keeps default page.goto wait semantics, and snapshot still uses a fixed 20s client timeout. The related #47203 and #72934 reports cover narrower navigate and snapshot subsets, but they do not supersede this broader open/navigate/snapshot CDP timeout report.

## What This Changes

The issue reports browser open, navigate, and snapshot operations timing out even though the configured CDP endpoint is reachable and direct Playwright CDP usage works.

## Best Possible Solution

Implement a browser-plugin fix that propagates operation-specific timeout and navigation wait-state controls through the agent tool, client, node proxy, route, and Playwright/CDP layers, with clearer timeout classification and regression coverage, while preserving existing SSRF/navigation-result guards.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with a local CDP endpoint, curl and direct Playwright success, and OpenClaw browser operation timeouts; current-main source still matches the timeout-prone boundaries described.

## Solution Assessment

No current solution is present. The narrow maintainable direction is targeted timeout/load-state propagation and clearer error classification, not a blanket timeout increase or broad CDP transport rewrite.
