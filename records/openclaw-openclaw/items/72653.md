## Summary

Keep open. Current main still matches the reported browser timeout class: navigate drops the public timeout/load-state controls, snapshot still lacks end-to-end timeout propagation, and existing-session open can still run fixed internal Chrome MCP budgets under an outer tool timeout.

## What This Changes

The issue asks for browser open, snapshot, and navigate to stop timing out when the configured CDP endpoint is otherwise reachable and works through direct Playwright/CDP checks.

## Best Possible Solution

Fix this inside the browser plugin by threading normalized timeout/load-state controls through open, navigate, and snapshot, preserving SSRF checks and existing default budgets unless targeted regression tests justify a narrower default change.

## Reproduction Assessment

Yes, source-level. The report gives a concrete live CDP setup, and current main still statically reproduces the dropped timeout/load-state controls and fixed internal browser budgets; I did not run the live Ubuntu CDP environment.

## Solution Assessment

Yes. A targeted browser-plugin propagation and timeout-boundary repair is the narrow maintainable solution; increasing global timeouts or adding generic retries would leave the broken action seams intact.
