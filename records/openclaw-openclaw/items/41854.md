## What This Changes

The branch adds Groq onboarding/auth-choice, CLI flag, credential/env handling, default model/catalog wiring, detect-secrets allowlist churn, and tests across core commands, agents, docs, and secrets files.

## Best Possible Solution

Add Groq API-key setup in `extensions/groq` with plugin auth runtime, manifest `providerAuthChoices` CLI metadata, default-model wiring, focused onboarding tests, and redacted real onboarding proof.

Yes for the PR defects by source inspection. The branch declares `--groq-api-key` without forwarding `opts.groqApiKey`, and current main shows the replacement plugin path still has `auth: []` for Groq.

## Solution Assessment

No. The maintainable solution is plugin-owned Groq setup using the existing provider auth contracts, not adding a second core-owned Groq catalog and onboarding path.
