## Summary

Keep open. Current main has shipped Groq provider/catalog/docs through the bundled plugin, but this PR still targets native onboarding/auth support and the branch is not mergeable as-is because it hard-codes stale core provider wiring and leaves the new dedicated CLI flag non-functional.

## What This Changes

The PR adds Groq auth-choice, onboarding flag, credential storage, default model/provider config, env-var mapping, tests, and secret-baseline placeholder changes in core command and agent provider files.

## Best Possible Solution

Replace this with a narrow plugin-owned Groq setup path in `extensions/groq`: manifest auth-choice CLI metadata plus a Groq API-key auth method/default model, with focused provider-auth/onboarding tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection shows the PR declares a new `--groq-api-key` flag without the matching old CLI adapter pass-through, and current main confirms provider flags now need manifest `optionKey`/`cliFlag`/`cliOption` metadata.

## Solution Assessment

No. The PR's core hard-coded provider catalog and auth-choice approach is no longer the maintainable fix after Groq/provider ownership moved into plugins; a plugin-owned setup implementation is the safer path.
