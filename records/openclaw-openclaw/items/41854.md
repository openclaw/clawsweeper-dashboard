## Summary

Keep open: current main has Groq provider/catalog/docs support, but this PR is stale against the plugin-owned provider architecture and does not safely deliver the dedicated onboarding/CLI auth path as-is.

## What This Changes

The PR adds Groq onboarding/auth support, including picker entries, a `--groq-api-key` flag, credential/env handling, default model configuration, and tests.

## Best Possible Solution

Implement Groq API-key setup in the bundled Groq plugin using the manifest/provider-auth surfaces, with targeted onboarding and CLI coverage.

## Reproduction Assessment

Yes. Source inspection shows current main has Groq catalog/env metadata but no plugin-owned Groq auth method or manifest CLI flag metadata, and the PR’s prior review comments show its old `--groq-api-key` path was not forwarded by the real CLI adapter.

## Solution Assessment

No. The PR’s static core-provider approach is not the best fix on current main; the maintainable path is to add Groq auth setup to `extensions/groq` and let the existing generic onboarding/CLI machinery consume it.
