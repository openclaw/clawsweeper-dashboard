## Summary

Keep open: current main still has a real Groq onboarding/auth gap, but this PR is not merge-ready because it lacks real behavior proof and implements the remaining work through stale core-owned provider paths.

## What This Changes

The PR adds Groq auth/onboarding wiring, a `groq-api-key` choice, a `--groq-api-key` flag, credential/default-model handling, static Groq catalog entries, tests, and secret-baseline/docs placeholder changes.

## Best Possible Solution

Implement Groq API-key auth in `extensions/groq` with a plugin-owned `ProviderAuthMethod`, non-interactive handler, and manifest CLI metadata, while keeping the manifest model catalog as the source of truth.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch adds `--groq-api-key` without the matching PR-head CLI adapter wiring, and current main's Groq provider still has `auth: []`. I did not run live onboarding because this review is read-only.

## Solution Assessment

No. The PR hard-codes Groq through stale core catalog/onboarding paths; the narrower maintainable solution is plugin-owned Groq auth/setup metadata with the manifest catalog as the source of truth.
