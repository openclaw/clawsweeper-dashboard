## Summary

Keep open. Current main still has only partial bundled Groq support, while this PR’s implementation is stale against the current plugin-owned provider architecture and should not merge as-is.

## What This Changes

The PR adds hardcoded Groq API-key onboarding, a `--groq-api-key` flag, credential/env mapping, default Groq model/provider config, tests, and unrelated secret-scan baseline/allowlist edits.

## Best Possible Solution

Rework the feature in `extensions/groq`: add manifest `providerAuthChoices` for `groq-api-key` and `--groq-api-key`, add a plugin-owned `modelCatalog` for Groq text models, expose runtime API-key auth/default-model setup through the provider plugin helper, and cover the picker, non-interactive flag, env-ref/profile storage, and catalog behavior with focused tests.

## Reproduction Assessment

Yes. Source inspection on current main is enough: Groq has no manifest `providerAuthChoices`/`modelCatalog`, the runtime registers `auth: []`, and the current CLI/picker paths consume plugin metadata for these surfaces.

## Solution Assessment

No. This PR is not the best way to solve the issue because it hardcodes Groq in old core command/model-provider paths; the maintainable fix is plugin-owned metadata and runtime setup inside `extensions/groq`.
