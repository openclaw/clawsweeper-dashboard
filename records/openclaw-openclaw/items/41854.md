## Summary

Keep open: the Groq onboarding goal is still useful, but this PR is stale against current main’s plugin-first provider architecture and includes unrelated secret-scanner churn. Current main already has a bundled Groq plugin with setup/catalog/env metadata, but the missing work belongs in `extensions/groq` as plugin-owned API-key auth and manifest flag metadata, not as new core provider tables.

## What This Changes

The PR adds Groq API-key onboarding, provider picker entries, non-interactive flags, env-var mapping, default model config, core model-provider wiring, docs placeholder edits, and tests.

## Best Possible Solution

Rebuild the change inside `extensions/groq`: declare `providerAuthChoices` for `groq-api-key` and `--groq-api-key`, add a Groq API-key auth method with the shared plugin helper, keep model data in the Groq manifest catalog, and drop the unrelated secret-scan changes.

## Reproduction Assessment

Yes. Source inspection on current main shows Groq has setup/catalog/env metadata but no runtime auth method, and onboard provider flags are emitted only from manifest choices with explicit flag metadata.

## Solution Assessment

No. The PR hardcodes Groq into stale core command/model-provider paths; the narrow maintainable solution is plugin-owned auth metadata plus a Groq plugin API-key method.
