## Summary

Keep open: this PR is the canonical remaining thread for Groq onboarding after https://github.com/openclaw/openclaw/issues/21442 was closed in favor of it, and current main still does not fully implement executable Groq API-key setup. The submitted branch is stale against the plugin-owned provider setup architecture, has blocking review findings, and lacks required real behavior proof.

## What This Changes

The PR wires Groq into core onboarding/auth-choice flags, non-interactive API-key handling, credential/env mapping, default model/provider config, static provider catalog data, docs secret placeholders, and tests.

## Best Possible Solution

Rework Groq setup in `extensions/groq` with plugin-owned `providerAuthChoices`, a `createProviderApiKeyAuthMethod` auth method, generated `--groq-api-key` metadata, focused onboarding coverage, docs alignment, and redacted real onboarding proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch declares `--groq-api-key` without forwarding `opts.groqApiKey`, and it duplicates Groq provider catalog data in core while current main owns Groq metadata in the plugin. I did not run onboarding because this is a read-only review.

## Solution Assessment

No. The submitted core-static implementation is not the best current path; the maintainable fix is plugin-owned Groq API-key setup and catalog/default-model metadata through the existing provider plugin contract.
