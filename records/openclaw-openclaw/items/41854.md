## Summary

Keep open: current main has Groq plugin/docs/manifest catalog metadata, but the dedicated Groq auth/onboarding path still is not complete, and the linked feature request was closed in favor of this PR. The submitted branch is not mergeable as-is because it lacks real behavior proof and hard-codes provider catalog/auth wiring into core paths that current main treats as plugin-owned.

## What This Changes

The branch adds Groq auth/onboarding support through core command/provider config paths, including `groq-api-key`, `--groq-api-key`, `GROQ_API_KEY`, credential storage, default model config, static catalog entries, tests, and secret-baseline churn.

## Best Possible Solution

Implement Groq auth in `extensions/groq` with a plugin-owned API-key ProviderAuthMethod, non-interactive handler, and manifest `providerAuthChoices` CLI metadata, while keeping the manifest model catalog as the source of truth.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch declares `--groq-api-key` without the matching PR-head CLI adapter wiring, and current main's Groq plugin has setup metadata but no runtime auth method. I did not run live onboarding because this review is read-only.

## Solution Assessment

No. The submitted branch solves the gap by hard-coding Groq into legacy core provider/onboarding paths; the narrower maintainable solution is plugin-owned Groq auth/setup metadata with the manifest catalog as the source of truth.
