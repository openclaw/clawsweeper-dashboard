## Summary

This PR should stay open as useful source work, but it is not mergeable as-is. Current main has a bundled Groq plugin, docs, and manifest model catalog, yet the Groq runtime still lacks an API-key auth method and the PR solves that through stale core wiring plus unrelated secret-scanner churn instead of the current plugin-owned provider path.

## What This Changes

The PR hardcodes Groq API-key onboarding, default model/provider config, implicit provider loading, tests, and secret-scanner baseline/allowlist edits across core command, agent, docs, and secrets files.

## Best Possible Solution

Rebuild the change inside `extensions/groq`: add plugin-owned API-key auth using the provider SDK helpers, declare `providerAuthChoices` with `groqApiKey` and `--groq-api-key`, wire the default Groq model through plugin-owned onboarding/config helpers, keep the manifest catalog as the model source, and remove all unrelated secret-scanner churn.

## Reproduction Assessment

Yes. Source inspection is enough: current main exposes Groq manifest setup/catalog metadata but the runtime provider has `auth: []`, and onboard CLI flags are generated only from manifest choices with explicit flag metadata.

## Solution Assessment

No. The PR hardcodes Groq in old core command/model-provider paths; the maintainable solution is a plugin-owned Groq auth method and manifest metadata consistent with the current provider architecture.
