## What This Changes

Request to reduce cold-start plugin scan overhead by caching or otherwise eliminating repeated synchronous `fs.existsSync` calls in plugin discovery paths.

## Best Possible Solution

Implement a small lifecycle-owned, test-clearable plugin scan existence cache or precomputed existence-fact pass across `bundle-manifest`, `bundled-dir`, and SDK alias scanning, avoiding a broad process-global stale filesystem cache.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection confirms the synchronous probe pattern, and the issue includes Windows CPU-profile steps, but current-main Windows profiling is still missing.

## Solution Assessment

Yes, with a narrower cache boundary than the original process-global proposal. The plugin policy supports bounded lifecycle-owned caches, so the fix should scope cached existence facts to the scan/metadata lifecycle and include regression tests plus profiling proof.
