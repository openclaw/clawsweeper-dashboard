## Summary

Keep open: current main still has uncached synchronous filesystem existence probes in the plugin manifest and bundled plugin scan paths, so the reported cold-start optimization has not landed.

## What This Changes

Request to reduce plugin discovery cold-start cost by caching repeated `fs.existsSync` checks in plugin manifest discovery and bundled plugin directory scanning.

## Best Possible Solution

Add a plugin-scan-owned cached existence helper with scan lifetime or explicit invalidation, wire only the hot discovery probes through it, and keep manifest/discovery freshness guarantees covered by focused tests.

## Reproduction Assessment

Yes for the structural issue: current main still exposes repeated direct `fs.existsSync` calls through `detectBundleManifestFormat`, `loadBundleManifest`, `resolveBundledPluginsDir`, and `resolveBundledPluginScanDir`. The exact Windows cold-start self-time should be re-profiled before claiming the reported reduction.

## Solution Assessment

Unclear as originally proposed: caching repeated existence checks is a plausible narrow fix, but an unbounded process-level cache is only acceptable if install/reload/doctor paths invalidate it or the helper is scoped to one scan flow.
