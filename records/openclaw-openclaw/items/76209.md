## Summary

Keep open: current main still has uncached synchronous existence checks in the cited plugin scan paths, and the optimization is narrow enough to pursue with careful cache lifetime handling.

## What This Changes

Request to reduce cold-start plugin discovery cost by caching repeated `fs.existsSync` checks in plugin manifest and bundled directory scan paths.

## Best Possible Solution

Add a small plugin-scan-owned cached existence helper with explicit scan lifetime or invalidation, wire only the hot discovery probes through it, and cover cache hits plus stale-state boundaries with focused Vitest tests.

## Reproduction Assessment

Yes for the structural issue: current main exposes repeated `fs.existsSync` calls through `detectBundleManifestFormat`, `loadBundleManifest`, `resolveBundledPluginsDir`, and `resolveBundledPluginScanDir`. The exact 25.4s Windows self-time should still be re-profiled before claiming the reported reduction.

## Solution Assessment

Unclear as written: caching repeated existence checks is plausible, but a process-level cache is only the best fix if install/reload/doctor paths can invalidate it or the helper is scoped to one scan flow.
