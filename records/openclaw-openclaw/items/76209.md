## Summary

Keep open: current main still has direct synchronous existence checks in the cited plugin scan paths and no cached helper, while #67000 tracks broader warm-session startup work rather than this narrower filesystem probe optimization.

## What This Changes

Request to reduce plugin discovery cold-start cost by caching repeated `fs.existsSync` probes in plugin manifest and bundled-plugin scan paths.

## Best Possible Solution

Add a scan-scoped or invalidatable cached existence helper for the hot plugin discovery probes, with focused tests covering cache hits and stale-state boundaries.

## Reproduction Assessment

Yes for the structural issue: current main exposes repeated direct `fs.existsSync` calls through `detectBundleManifestFormat`, `loadBundleManifest`, `resolveBundledPluginsDir`, and `resolveBundledPluginScanDir`. The reported Windows wall-time should be re-profiled before claiming the same reduction.

## Solution Assessment

Unclear as a process-level cache: caching repeated existence checks is plausible, but the maintainable fix should respect plugin metadata freshness by using explicit scan lifetime or invalidation.
