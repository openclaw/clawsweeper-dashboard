## Summary

Keep open: the PR targets a real Feishu topic-thread prompt gap, but the current diff appends an unconditional message-tool instruction that conflicts with existing automatic group reply modes and it lacks the required changelog entry.

## What This Changes

Adds Feishu topic-scope prompt helpers/tests and appends a `thread-reply` message-tool hint to topic-scoped Feishu group sessions.

## Best Possible Solution

Keep the fix Feishu-owned, but make the topic hint delivery-mode aware or explicitly conditional on choosing the message tool, then add a focused changelog fix entry and prompt coverage for automatic fallback.

## Reproduction Assessment

Yes. A source-level reproduction exists with a Feishu `group_topic` or `group_topic_sender` group using `messages.groupChat.visibleReplies: "automatic"`, or with the message tool unavailable: current main sends the final reply through the threaded dispatcher while the PR still tells the model to call the message tool.

## Solution Assessment

No. The extension-local prompt override is the right boundary, but the current wording is too broad; conditional wording or gating on message-tool-only delivery is the narrower maintainable fix.
