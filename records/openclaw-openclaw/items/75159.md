## Summary

Keep this PR open. It targets a real Feishu topic-thread bug and stays in the right extension-owned area, but the current patch appends a message-tool instruction for every topic-scoped group turn, including automatic-delivery turns where the shared prompt tells the model not to use the message tool. The PR also needs the required user-facing changelog entry before merge.

## What This Changes

The PR adds Feishu topic-scope prompt helpers and tests, then appends a thread-reply message-tool hint to `GroupSystemPrompt` for `group_topic` and `group_topic_sender` sessions.

## Best Possible Solution

Keep the fix Feishu-owned, but make the topic guidance compatible with delivery mode. A corrected patch should emit the override only when visible output must use the message tool, or phrase it conditionally so automatic-delivery sessions still follow the shared instruction to reply normally while any explicit message-tool send uses `thread-reply`.

## Reproduction Assessment

Yes. A source-level reproduction exists by combining a Feishu `group_topic` or `group_topic_sender` group with automatic visible replies or message-tool-unavailable fallback: current main will automatically thread the final reply, while the PR’s unconditional hint also tells the model to call the message tool.

## Solution Assessment

No, not as written. The extension-local prompt approach is directionally right for message-tool-only turns, but the proposed implementation is too broad and needs delivery-mode gating or conditional wording plus a changelog entry.
