## Summary

Keep open: current main and the latest release still preserve `type: "custom"` in this OpenAI-compatible Anthropic stream wrapper, while the canonical OpenAI payload builders project OpenClaw tools to function-tool payloads, so this remains a narrow provider-payload bug.

## What This Changes

The issue asks OpenClaw to convert Anthropic-family custom tool definitions, `tool_choice`, and `allowed_tools` entries to OpenAI function-tool form before sending them through the OpenAI-compatible bridge.

## Best Possible Solution

Update this compatibility wrapper to project Anthropic-style custom tools and their choices into the same function-tool shape used by OpenClaw's canonical OpenAI builders, with focused regression coverage and no new config surface.

## Reproduction Assessment

Yes, at source level: the current wrapper and its tests still emit `type: "custom"` for the reported definition, `tool_choice`, and allowed-tool paths. I did not run a live OpenAI 400 reproduction in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to align this bridge with the canonical OpenAI function-tool projection and update the focused wrapper tests. A broader provider-routing or config change is unnecessary for the reported payload-shape bug.
