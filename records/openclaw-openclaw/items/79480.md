## Summary

Keep open: the PR addresses the main deny/profile CLI MCP path, but the current matcher still widens an exact MCP tool allow entry into access to the whole MCP server, which is a permission-boundary regression.

## What This Changes

The PR threads effective tool policies into CLI bundle MCP config generation and adds regression coverage for per-agent deny rules and profile-allowed bundle MCP servers.

## Best Possible Solution

Keep the pre-write CLI MCP filtering, but separate server-wide/profile/plugin authorization from exact per-tool allow entries; exact allows should fail closed unless a downstream CLI per-tool filter is added.

## Reproduction Assessment

Yes, source-level. At PR head, `allow: ["mcp__twenty-crm__read_contact"]` reaches the `mcp__<server>__` prefix branch and emits the whole server, while existing embedded policy coverage keeps exact MCP allowlists narrow.

## Solution Assessment

No. Applying effective policy before writing CLI MCP config is the right boundary, but this implementation still conflates exact per-tool allow entries with server-wide authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not widen exact MCP tool allowlists:** `src/agents/bundle-mcp-config.ts:82-83`
  - body: This treats any `mcp__<server>__...` policy entry as permission to emit the whole MCP server. Please distinguish server-wide entries from exact tool allows, or fail closed when the downstream config cannot enforce per-tool allows.
