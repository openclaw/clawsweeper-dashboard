## Summary

Keep open: the PR addresses the reported CLI bundle-MCP policy boundary, but current head still widens exact MCP tool allow entries into full-server access for CLI backends, which is a permission-boundary bug.

## What This Changes

The PR threads effective tool policy into CLI bundle MCP config generation and adds regression coverage for per-agent denies and profile-allowed bundle MCP servers.

## Best Possible Solution

Keep the pre-write CLI bundle MCP filtering, but treat only server-wide/profile/plugin entries as server authorization; exact per-tool allows should fail closed unless downstream CLI per-tool filtering is added.

## Reproduction Assessment

Yes, source-level. At PR head, allow: ["mcp__twenty-crm__read_contact"] reaches the mcp__<server>__ prefix branch and emits the entire server, while embedded policy coverage keeps exact MCP allowlists narrow.

## Solution Assessment

No. Applying policy before writing CLI MCP config is the right boundary, but the matcher still conflates exact per-tool allows with server-wide authorization.
