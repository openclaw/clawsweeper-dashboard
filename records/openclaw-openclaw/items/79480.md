## Summary

Keep this PR open: it fixes the right CLI MCP config boundary and has sufficient real behavior proof, but the latest head still turns exact MCP tool allow entries into full-server access, which is a security-sensitive regression.

## What This Changes

The PR adds effective tool-policy filtering before generated CLI bundle MCP configs are written and adds regression coverage for denied and profile-allowed MCP servers.

## Best Possible Solution

Keep the generated-config enforcement boundary, but only treat profile/plugin/server-wide entries as server authorization and fail closed for exact MCP tool allows unless the downstream CLI can enforce per-tool filtering.

## Reproduction Assessment

Yes, source-level. The linked issue gives a concrete released-build claude-cli reproduction, current main writes merged CLI MCP config without policy input, and the PR body shows the fixed deny/profile path through a real Claude Code MCP health check.

## Solution Assessment

No. Applying policy before emitting the CLI MCP config is the right boundary, but the current matcher is too broad because exact MCP tool allows still expose every tool on that server.
