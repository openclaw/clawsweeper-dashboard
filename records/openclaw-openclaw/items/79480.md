## Summary

Keep open: the PR fixes the reported deny path and now has real CLI proof, but the new matcher still broadens exact MCP tool allow entries to the whole MCP server, which is not the documented effective-policy behavior.

## What This Changes

The PR computes effective tool policies for CLI bundle MCP preparation, filters generated MCP server configs, preserves `bundle-mcp` profile allows, and adds regression coverage.

## Best Possible Solution

Enforce CLI MCP config policy without widening exact tool allowlists: preserve profile and server-wide allows, keep deny entries fail-closed at server level, and add a changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. The linked report gives a concrete released-build `claude-cli` reproduction, and current main still writes merged CLI MCP config without passing agent-scoped tool policy into that path.

## Solution Assessment

No. Applying policy before emitting the CLI MCP config is the right boundary, but the current matcher is not the best implementation because exact MCP tool allow entries can expose every tool on that server.
