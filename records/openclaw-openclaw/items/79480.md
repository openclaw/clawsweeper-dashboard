## Summary

Keep open: the PR puts CLI bundle MCP filtering at the right boundary and includes useful real behavior proof, but the latest head still widens exact MCP tool allowlists into full-server access for CLI backends.

## What This Changes

The PR threads effective tool policy into CLI bundle MCP config generation and adds regression coverage for per-agent denies and profile-allowed bundle MCP servers.

## Best Possible Solution

Keep the pre-write CLI bundle MCP filtering, but treat only server-wide/profile/plugin entries as server authorization; exact per-tool allows should fail closed unless a downstream CLI per-tool filter is added, with regression coverage for that case.

## Reproduction Assessment

Yes, source-level. At PR head, allow: ["mcp__twenty-crm__read_contact"] reaches the mcp__<server>__ prefix branch and emits the entire server, while current embedded policy coverage keeps exact MCP allowlists narrow.

## Solution Assessment

No. Applying policy before writing CLI MCP config is the right boundary, but this matcher still conflates exact per-tool allows with server-wide authorization.
