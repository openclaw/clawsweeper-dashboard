## Summary

Keep open: this PR targets a real documented CLI MCP deny-policy gap and includes sufficient real behavior proof, but the current head still widens exact MCP per-tool allow entries into full-server CLI access, so it needs a narrow policy matcher and regression-test fix before merge.

## What This Changes

The PR resolves effective tool policies during CLI runner preparation, filters generated bundle MCP config before writing it for CLI backends, and adds regression coverage for per-agent deny/profile-allowed MCP servers.

## Best Possible Solution

Keep the pre-write CLI MCP filtering, but only treat server-wide, plugin/profile, wildcard, or explicit whole-server policy entries as whole-server authorization; exact per-tool allows should fail closed unless CLI backends gain real per-tool filtering.

## Reproduction Assessment

Yes, source-level: at PR head, `allow: ["mcp__twenty-crm__read_contact"]` reaches the `mcp__<server>__` prefix branch and emits the entire server, while existing embedded policy coverage keeps exact MCP allowlists narrow. I did not run a live external MCP setup in this read-only review.

## Solution Assessment

No. Applying effective policy before writing CLI MCP config is the right boundary, but the current matcher still conflates exact per-tool allow entries with server-wide authorization.
