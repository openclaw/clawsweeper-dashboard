## Summary

Keep open: this is a plausible security bug fix, but the patch introduces a blocking profile allowlist regression and the external PR only provides test-based proof, not real CLI backend behavior proof.

## What This Changes

The PR adds policy-based filtering of merged bundle MCP server configs before CLI backend config emission, with regression tests for MCP deny and allow matching.

## Best Possible Solution

Update this PR so CLI MCP config filtering preserves the same policy semantics as embedded tool filtering, including profile `bundle-mcp` plugin-level allow entries, then add regression coverage and real behavior proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main merges and writes CLI MCP config without applying tool-policy filtering, and the linked issue gives a concrete `claude-cli` reproduction on released builds. I did not run a live `claude-cli` MCP setup in this read-only review.

## Solution Assessment

No, the proposed server-name allowlist matcher is not yet the best solution because it misses established profile/plugin metadata semantics such as `bundle-mcp`. The safer direction is to reuse or mirror the shared policy pipeline semantics for bundle MCP server metadata before emitting CLI config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve bundle-mcp profile allowlists:** `src/agents/bundle-mcp-config.ts:91`
  - body: The `coding` and `messaging` profiles intentionally allow bundle MCP through the `bundle-mcp` plugin-level entry, and embedded filtering keeps those tools via plugin metadata. This new allowlist path only checks entries against server-name candidates, so `tools.profile: "coding"` makes every CLI MCP server fail this `allow.some(...)` check and emits an empty MCP config. Please map `bundle-mcp`/plugin-level allow entries to bundle MCP servers or otherwise reuse the existing policy semantics.
  - confidence: 0.88
