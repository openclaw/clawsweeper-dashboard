## Summary

Keep open: the fix direction and supplied proof cover the reported deny/profile path, but PR head still widens exact MCP tool allow entries into whole-server CLI access, which is a permission-boundary regression.

## What This Changes

The branch threads effective tool policies into CLI bundle MCP config generation, filters MCP servers before writing CLI configs, and adds regression coverage for deny/profile bundle MCP behavior.

## Best Possible Solution

Keep the pre-write CLI MCP filtering, but authorize a CLI MCP server only for server-wide/profile/plugin entries; exact per-tool allows should fail closed unless the downstream CLI path can enforce per-tool filtering.

## Reproduction Assessment

Yes, source-level. At PR head, allow: ["mcp__twenty-crm__read_contact"] reaches the mcp__<server>__ prefix branch and emits the entire server, while current embedded policy coverage keeps exact MCP allowlists narrow.

## Solution Assessment

No. Applying effective policy before writing CLI MCP config is the right boundary, but this matcher still conflates exact per-tool allows with server-wide authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not widen exact MCP tool allowlists:** `src/agents/bundle-mcp-config.ts:82-83`
  - body: This treats any `mcp__<server>__...` policy entry as permission to emit the whole MCP server. Please distinguish server-wide entries from exact tool allows, or fail closed when the downstream config cannot enforce per-tool allows.
