## Summary

Keep open: the PR is aimed at the right CLI bundle-MCP boundary and includes sufficient after-fix live output, but the latest head still treats exact MCP tool allow entries as authorization to expose an entire MCP server, which would weaken the same permission boundary this patch is fixing.

## What This Changes

The PR adds tool-policy filtering before generated CLI bundle MCP configs are written and adds regression coverage for denied and profile-allowed MCP servers.

## Best Possible Solution

Keep the generated-config enforcement boundary, but only treat profile/server-wide entries as server authorization and fail closed for exact MCP tool allowlists unless the downstream CLI can enforce per-tool filtering.

## Reproduction Assessment

Yes. The linked issue gives a concrete released-build claude-cli reproduction, current main lacks policy input before writing the CLI MCP config, and the PR body's live output exercises the intended fixed path.

## Solution Assessment

No. Applying policy before emitting the CLI MCP config is the right boundary, but the current matcher is too broad because exact MCP tool allows still expose every tool on that server.
