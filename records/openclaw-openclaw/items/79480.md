## Summary

Keep open: current main still lacks CLI bundle-MCP policy enforcement, and this PR is the right fix area, but the current head still widens exact MCP tool allow entries into server-level authorization.

## What This Changes

The PR resolves effective tool policies for CLI bundle-MCP config generation, filters generated MCP server configs by those policies, and adds deny/profile regression coverage.

## Best Possible Solution

Keep the pre-write CLI bundle-MCP filtering, but treat only server-wide/profile/plugin entries as server authorization; exact per-tool allows should fail closed unless downstream per-tool filtering is added.

## Reproduction Assessment

Yes, source-level: at PR head, allow: ["mcp__twenty-crm__read_contact"] reaches the mcp__<server>__ prefix branch and emits the entire server. Current embedded policy coverage keeps exact MCP allowlists narrow.

## Solution Assessment

No. Applying policy before writing CLI MCP config is the right boundary, but this matcher still conflates exact per-tool allows with server-wide authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not widen exact MCP tool allowlists:** `src/agents/bundle-mcp-config.ts:82-83`
  - body: This treats any `mcp__<server>__...` policy entry as permission to emit the whole MCP server. Please distinguish server-wide entries from exact tool allows, or fail closed when the downstream config cannot enforce per-tool allows.
