## Summary

Keep open: the PR is aimed at the right CLI bundle-MCP policy boundary and its real behavior proof covers the deny/profile case, but the current matcher still treats exact MCP tool allow entries as permission to emit an entire server, which is a permission-boundary regression.

## What This Changes

The PR applies effective tool policy before writing CLI backend bundle MCP configs and adds regression coverage for per-agent denies plus profile-allowed bundle MCP servers.

## Best Possible Solution

Keep the pre-write CLI bundle MCP filtering, but treat only server-wide, wildcard, profile, or plugin policy entries as server authorization; exact per-tool allows should fail closed unless a downstream CLI per-tool filter is added.

## Reproduction Assessment

Yes, source-level: at PR head, allow: ["mcp__twenty-crm__read_contact"] reaches the mcp__<server>__ prefix branch and emits the entire server. The embedded policy matcher and tests show exact allowlists are supposed to stay narrow.

## Solution Assessment

No. Applying policy before writing CLI MCP config is the right boundary, but the current matcher conflates exact per-tool allows with server-wide authorization.
