## Summary

Keep this PR open. The generated-config enforcement boundary is the right fix direction and the PR supplies real proof for the deny/profile path, but latest head still has a blocking policy-widening bug: an exact MCP tool allow entry can cause the entire CLI MCP server to be emitted.

## What This Changes

The PR applies effective tool-policy filtering to generated CLI backend bundle MCP configs and adds regression coverage for deny and profile-allow behavior.

## Best Possible Solution

Keep the CLI config filtering boundary, but only treat profile/plugin/server-wide entries as server authorization and fail closed for exact per-tool allows unless a downstream CLI can enforce per-tool filtering.

## Reproduction Assessment

Yes, source-level. Current main emits merged CLI MCP configs without effective policy, and latest PR head can be checked with a narrow filter input where allow: ["mcp__twenty-crm__read_contact"] still emits the whole twenty-crm server.

## Solution Assessment

No. Applying effective policy before writing CLI MCP config is the right boundary, but the current matcher is too broad because exact MCP tool allow entries become full-server access.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not widen exact MCP tool allowlists:** `src/agents/bundle-mcp-config.ts:82-83`

Please distinguish server-wide entries from exact tool allows, or fail closed when the downstream config cannot enforce per-tool allows.
