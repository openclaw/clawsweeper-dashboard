## Summary

Keep open: the PR fixes the reported CLI MCP deny path in the right area, but PR head still has a permission-boundary bug where an exact MCP tool allow entry authorizes the whole server for CLI backends.

## What This Changes

The PR computes effective tool policy for CLI bundle MCP config generation, filters generated MCP servers before writing backend config, and adds regression tests for denies and profile-allowed bundle MCP servers.

## Best Possible Solution

Keep the pre-write CLI MCP filtering, but only treat server-wide/profile/plugin entries as server authorization; exact per-tool allows should fail closed unless downstream CLI per-tool filtering is added.

## Reproduction Assessment

Yes, source-level. At PR head, allow: ["mcp__twenty-crm__read_contact"] reaches the mcp__<server>__ prefix branch and emits the entire server, while embedded policy coverage keeps exact MCP allowlists narrow.

## Solution Assessment

No. Applying policy before writing CLI MCP config is the right boundary, but the matcher still conflates exact per-tool allows with server-wide authorization.
