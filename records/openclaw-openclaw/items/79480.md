## Summary

Needs changes before merge: the PR applies policy at the right CLI bundle-MCP boundary, but the current matcher still lets an exact MCP tool allow entry authorize the entire server config, which is a permission-boundary regression.

## What This Changes

Threads effective tool policies into CLI bundle-MCP config generation and adds regression coverage for deny rules and profile-allowed bundle MCP servers.

## Best Possible Solution

Keep the pre-write CLI MCP filtering, but authorize server config only from server-wide, profile, or plugin entries; exact per-tool allows should fail closed unless a downstream CLI per-tool filter is added.

## Reproduction Assessment

Yes, source-level. At PR head, `allow: ["mcp__twenty-crm__read_contact"]` reaches the `mcp__<server>__` branch and emits the whole server, while the embedded policy test keeps exact MCP allowlists narrow.

## Solution Assessment

No. Applying policy before writing CLI MCP config is the right boundary, but the current matcher conflates exact per-tool allow entries with server-wide authorization.

## Review Findings
