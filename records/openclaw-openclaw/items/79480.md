## Summary

Keep open: the PR fixes the reported deny/profile path in the right area, but its latest matcher still turns exact MCP tool allow entries into whole-server access for CLI backends, which is a permission-boundary regression.

## What This Changes

The PR applies effective tool policies before generating CLI backend bundle-MCP config files, preserving profile-allowed bundle MCP servers while removing denied servers.

## Best Possible Solution

Keep the pre-write CLI bundle-MCP filtering, but treat only server-wide/profile/plugin entries as server authorization; exact per-tool allow entries should fail closed unless a downstream CLI per-tool filter is added.

## Reproduction Assessment

Yes, source-level: at PR head, allow: ["mcp__twenty-crm__read_contact"] reaches the mcp__<server>__ prefix branch and emits the whole server, while current embedded policy tests keep exact MCP allowlists narrow.

## Solution Assessment

No. Applying policy before writing CLI MCP config is the right boundary, but this matcher still conflates exact per-tool allows with server-wide authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P1] Do not widen exact MCP tool allowlists:** `src/agents/bundle-mcp-config.ts:82-83`
