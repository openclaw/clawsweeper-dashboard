## Summary

Keep open: this is a protected maintainer-labeled member PR, and the product direction matches the linked Control UI blank-page report, but the new fallback test is order-dependent under the repo’s non-isolated UI Vitest setup.

## What This Changes

The PR adds a static HTML Control UI mount fallback, troubleshooting docs/changelog text, and focused coverage for the fallback.

## Best Possible Solution

Land a static bootstrap/docs fallback for an unregistered `openclaw-app`, with regression coverage that does not depend on the shared custom element registry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible, but not live-reproduced: current main has no static fallback before module registration, and `openclaw-app` is only defined by the module-loaded app code.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes for the product behavior, but not this exact patch until the new test is made independent of the shared `customElements` registry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Make the fallback test independent of custom-element state:** `ui/src/ui/mount-fallback.test.ts:39-44`
  - body: The UI unit project runs with `isolate: false`, and existing tests can import `app.ts`, which defines `openclaw-app` for the shared jsdom worker. Once that registry entry exists, this first case no longer represents an unregistered app: the fallback script sees `customElements.get('openclaw-app')` and keeps the panel hidden, so the `hidden === false` assertion can fail depending on file scheduling. Use a fresh window/DOM or otherwise avoid relying on the process-global registry for the negative case.
  - confidence: 0.84
