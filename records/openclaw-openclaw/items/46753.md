## Summary

Keep open: the cache-cap idea addresses real unbounded maps on current main, but this branch also bundles a large web_search rewrite that breaks current plugin registry and plugin-owned credential contracts, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds oldest-first caps to several agent cache maps and also tightens tool schemas while replacing the registry-backed web_search implementation with a large in-tool provider implementation.

## Best Possible Solution

Narrow the change to cache caps with insertion-only eviction and focused regression coverage, while leaving web_search routing and credential handling on the current plugin/runtime contracts or splitting any schema hardening into a separate reviewed PR.

## Reproduction Assessment

No, not for the long-running leak in a live daemon. Source inspection shows the current unbounded maps and source-reproducible PR regressions, but the actual memory-growth scenario and after-fix behavior were not run in a real setup.

## Solution Assessment

No. The cache-cap direction is reasonable, but this PR is not the best solution because it bundles a core web_search rewrite that bypasses established plugin registry and credential contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:1906`
  - body: This hard-coded provider selection bypasses the current runtime/plugin provider registry. Providers outside the local list, including bundled firecrawl/exa/tavily/duckduckgo and custom runtime providers covered by current tests, will no longer execute through web_search.
  - confidence: 0.93
- **[P1] Read web_search credentials from plugin-owned config:** `src/agents/tools/web-search.ts:910`
  - body: The new provider resolvers only look at legacy tools.web.search.* config or ambient env vars. Current providers store configured credentials under plugins.entries.<plugin>.config.webSearch.*, so existing configured Brave/Gemini/Grok/Kimi/Perplexity users can get missing-key results despite having valid credentials.
  - confidence: 0.91
- **[P2] Preserve sub-second recording durations:** `src/agents/tools/nodes-tool.ts:167-168`
  - body: Adding minimum: 1000 to durationMs rejects values from 0 through 999 ms even though current main only caps the maximum at 300000. That is an unrelated behavior tightening for nodes screen/camera recording and contradicts the PR's no behavior change claim.
  - confidence: 0.86
- **[P2] Evict only when inserting a new workspace file key:** `src/agents/workspace.ts:92`
  - body: The workspace cache eviction runs before every set, including refreshes for a key that is already cached. At capacity, refreshing a changed tracked file evicts an unrelated entry and then overwrites the same key, causing avoidable churn and a smaller effective cache.
  - confidence: 0.84
- **[P3] Apply one model-cache cap policy:** `src/agents/context.ts:82`
  - body: Configured context-window writes bypass the new cap, while discovered writes enforce it. A large configured model list can push the shared cache over the cap and then let a later discovered insert evict a configured entry arbitrarily.
  - confidence: 0.72
