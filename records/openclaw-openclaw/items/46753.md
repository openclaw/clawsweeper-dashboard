## Summary

Keep open: the underlying cache-cap problem is still present on current main, but this PR is not mergeable as-is because it bundles a large web_search/provider rewrite that would regress current runtime providers and plugin-owned credentials.

## What This Changes

The PR adds oldest-first caps to bootstrap, workspace-file, cache-trace, and model-context maps while also changing tool schemas and replacing the web_search wrapper with a direct hard-coded implementation.

## Best Possible Solution

Land a narrow current-main patch that caps the affected cache maps with regression coverage while preserving the existing web_search provider registry, plugin-owned credential paths, and unrelated tool schemas.

## Reproduction Assessment

Yes, by source inspection. Current main inserts distinct session keys, workspace file paths, cache trace paths, and model IDs into module-level Maps without a size check; this review did not run a heap loop because it was read-only.

## Solution Assessment

No. The cache-cap idea is the right target, but this PR is not the best way to solve it because the unrelated web_search/schema rewrite introduces blocking regressions; the safer path is a focused replacement patch on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep web_search on the runtime provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The new `SEARCH_PROVIDERS` list restricts the agent tool to five hard-coded providers and the replacement no longer calls the current runtime/provider registry. Current main supports bundled and runtime providers through `resolveRuntimeWebSearchProviders`/`resolvePluginWebSearchProviders`, including firecrawl, tavily, searxng, exa, duckduckgo, and custom providers, so those configurations would stop working or route incorrectly.
  - confidence: 0.94
- **[P1] Resolve web_search credentials from plugin config:** `src/agents/tools/web-search.ts:1898-1914`
  - body: `createWebSearchTool` now derives provider configs from `tools.web.search` and reads keys from those legacy subobjects or env vars. Existing onboarding, docs, and provider contracts store credentials under `plugins.entries.<provider>.config.webSearch.*`, so users with valid plugin-owned config would get missing-key errors after this change.
  - confidence: 0.92
- **[P2] Preserve sub-second node clip durations:** `src/agents/tools/nodes-tool.ts:165-169`
  - body: The schema adds `minimum: 1000` to `durationMs`, but current main only caps the maximum and existing camera helper coverage accepts 200 ms clip payloads. Agent callers that previously requested short clips would now fail schema validation without this PR documenting or justifying that behavior change.
  - confidence: 0.84
- **[P2] Evict workspace cache entries only for new keys:** `src/agents/workspace.ts:92-93`
  - body: This eviction runs before every refreshed file write. When the cache is full and `params.filePath` is already present with a changed identity, it deletes an unrelated oldest entry and then overwrites the existing key, shrinking effective cache capacity and causing avoidable disk churn.
  - confidence: 0.87
