## Summary

Keep open: current main still has unbounded agent cache maps, but this PR is dirty against main and mixes the useful cache-cap work with broad unrelated web-search and tool-schema changes that break existing provider/config contracts.

## What This Changes

This PR adds oldest-first size caps to several agent runtime caches and also rewrites or tightens multiple agent tool schemas, including a large inline web_search provider implementation.

## Best Possible Solution

Rework this as a narrow cache-cap patch on current main, preserving bootstrap refresh and web-search provider contracts, with focused eviction tests and no unrelated schema/provider rewrite.

## Reproduction Assessment

Yes, at source level. Current main has long-lived global Maps for session/bootstrap, workspace file contents, cache-trace writers, and model context tokens with no size cap, so creating many distinct keys in a daemon can grow them indefinitely.

## Solution Assessment

No, not as proposed. The cache-cap idea is narrow and maintainable, but this PR should be split or replaced because it is merge-conflicted and changes web-search provider, credential, and tool-schema behavior outside the memory-leak fix.

Full review comments:

- **[P1] Keep web_search on the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: This hard-coded provider list drops the current runtime provider registry path. Current main exposes and tests runtime/custom providers plus bundled firecrawl, exa, tavily, and duckduckgo; after this change those providers are no longer eligible for web_search execution, so existing plugin-backed setups break.
  - confidence: 0.93

  - body: These resolvers only read legacy tools.web.search fields or process env values. Current doctor migrations and provider contracts store bundled search credentials under plugins.entries.<plugin>.config.webSearch.*, so users with canonical plugin config would get missing API key results despite being configured.
  - confidence: 0.91
- **[P2] Preserve refreshed bootstrap snapshots:** `src/agents/bootstrap-cache.ts:22-29`
  - body: The PR branch still returns the cached bootstrap files for an existing session before reloading the workspace. Current main intentionally reloads and compares the files each turn so AGENTS/BOOTSTRAP edits are picked up; a rebase that keeps this shape would reintroduce stale bootstrap behavior.
  - confidence: 0.86
- **[P2] Keep sub-second recording durations valid:** `src/agents/tools/nodes-tool.ts:165-168`
  - body: Adding minimum: 1000 makes durationMs values from 0-999 fail schema validation even though current main only caps the maximum and the runtime clamps/handles numeric values. That is a behavior change unrelated to cache eviction and can break existing camera/screen recording calls.
  - confidence: 0.82
- **[P3] Evict only when inserting new workspace cache keys:** `src/agents/workspace.ts:91-93`
  - body: This eviction runs before every set, including refreshes of a file path already present in the cache. At capacity, updating an existing file evicts an unrelated oldest entry and then overwrites the same key, reducing effective cache usefulness and causing avoidable disk reads.
  - confidence: 0.79
