## Summary

Keep open: the cache-cap idea is plausible because current main still has source-visible unbounded maps, but this branch is not mergeable as-is because it mixes the cache change with stale bootstrap-cache behavior, a web_search provider-registry rewrite, credential-path regressions, schema tightening, and no real daemon proof.

## What This Changes

The branch adds oldest-first caps to several agent-runtime cache maps and also changes many tool schemas plus the web_search implementation.

## Best Possible Solution

Land a narrow cache-cap change with focused eviction coverage while preserving current bootstrap refresh behavior, web_search provider-registry delegation, plugin-owned credential paths, and existing schema compatibility.

## Reproduction Assessment

No high-confidence runtime reproduction is present. Source inspection shows the targeted maps are uncapped on current main, but neither this review nor the PR body demonstrates heap growth or eviction behavior in a long-running daemon.

## Solution Assessment

No. Simple caps may be a reasonable fix direction, but this PR is not the best solution because it bundles unrelated provider rewrites and schema changes and is stale against current bootstrap-cache behavior.

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve bootstrap refresh behavior:** `src/agents/bootstrap-cache.ts:23-24`
  - body: The PR head still returns any cached session entry without reloading workspace bootstrap files. Current main now reloads each turn and only reuses the previous snapshot when the workspace and file contents match, so merging this stale branch would make long-running sessions miss AGENTS/bootstrap edits again.
  - confidence: 0.9
- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The new hard-coded `SEARCH_PROVIDERS` list replaces registry-backed execution, so runtime and bundled providers outside this list stop working even though current tests require `runWebSearch` to execute active plugin providers.
  - confidence: 0.93
- **[P1] Read search keys from plugin-owned config:** `src/agents/tools/web-search.ts:553-563`
  - body: These local resolvers read legacy `tools.web.search.*` fields or env vars directly, but current provider docs and implementations use plugin-owned paths such as `plugins.entries.<plugin>.config.webSearch.*`; configured users can get missing-key errors after this change.
  - confidence: 0.9
- **[P2] Preserve sub-second recording durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding `minimum: 1000` rejects `durationMs` values below one second, while current main only enforces the 300,000 ms maximum. That validation tightening is unrelated to cache caps and breaks previously accepted node camera clip inputs.
  - confidence: 0.86
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92-93`
  - body: This eviction runs before every set, including same-key refreshes. At capacity, refreshing a changed file can evict an unrelated entry and then overwrite the existing key, reducing effective cache size without lowering memory for the refreshed key.
  - confidence: 0.8
- **[P3] Apply model-cache limits consistently:** `src/agents/context.ts:82`
  - body: Configured context-window entries still bypass the new `MODEL_CACHE` cap, so a large configured model set can overfill the map and later have configured entries evicted by the discovered-model path.
  - confidence: 0.73
