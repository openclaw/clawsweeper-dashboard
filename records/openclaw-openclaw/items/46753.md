## Summary

Keep open: current main still has the unbounded cache maps this PR is trying to address, but this branch is not merge-safe because it also rewrites `web_search` away from the runtime provider registry and tightens unrelated tool validation.

## What This Changes

The PR adds oldest-first caps to four agent runtime caches and also changes multiple agent tool schemas, including replacing the `web_search` wrapper with a hard-coded provider implementation.

## Best Possible Solution

Land a narrow updated or replacement PR that caps the cache maps with regression coverage while preserving the existing `web_search` runtime provider and plugin config contracts.

## Reproduction Assessment

Yes, at source level. Current main has module-global `Map` caches with no eviction, and the PR-specific regressions are visible by comparing PR head's hard-coded `web_search` resolver with current main's registry-backed runtime tests.

## Solution Assessment

No. Cache caps are the right direction, but this branch should be narrowed to that fix and should not replace provider-owned `web_search` runtime/config behavior or silently tighten unrelated node recording validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route web_search through the runtime registry:** `src/agents/tools/web-search.ts:25`
  - body: This hard-coded provider list replaces the registry-backed `web_search` wrapper, so bundled/runtime providers outside these five IDs, such as firecrawl, exa, tavily, duckduckgo, and custom providers, stop being selectable. Keep provider execution behind `runWebSearch`/runtime provider discovery instead of moving provider implementations into this core tool file.
  - confidence: 0.96
- **[P1] Read web search credentials from plugin config:** `src/agents/tools/web-search.ts:553-563`
  - body: The new resolver reads top-level or legacy `tools.web.search.*` fields for provider keys, but current onboarding/doctor paths store provider-owned credentials under `plugins.entries.<plugin>.config.webSearch.*`. Existing configured Brave/Gemini/Grok/Kimi/Perplexity users can therefore get missing-key errors even though their plugin config is valid.
  - confidence: 0.91
- **[P2] Preserve sub-second node recording durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding `minimum: 1000` to `durationMs` rejects calls that current main accepts and forwards, including short sub-second camera or screen recordings. This is an unrelated behavior change for a cache-cap PR; preserve the existing maximum-only contract unless a separate product decision changes it.
  - confidence: 0.89
- **[P2] Evict workspace cache only for new entries:** `src/agents/workspace.ts:92`
  - body: The eviction runs before every `workspaceFileCache.set`, including stale refreshes for a key that is already cached. At capacity, refreshing an existing file deletes an unrelated oldest entry and then overwrites the same key, shrinking effective cache size and causing avoidable disk churn; only evict when inserting a new key.
  - confidence: 0.86
- **[P3] Apply the model cache cap consistently:** `src/agents/context.ts:52-58`
  - body: The new cap is only enforced on discovered model inserts, while `applyConfiguredContextWindows` still writes into the same `MODEL_CACHE` unconditionally. A large configured model list can push the map past the stated cap and then have entries evicted later by the discovered path.
  - confidence: 0.78
