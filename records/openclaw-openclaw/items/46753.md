## Summary

Keep open: current main still has the uncapped agent-runtime maps, so the memory-growth concern remains valid, but this branch is not mergeable as-is because it lacks real behavior proof and introduces regressions plus unrelated security-sensitive web_search changes.

## What This Changes

This PR adds size caps to four agent-runtime Map caches and also changes many agent tool schemas plus replaces the web_search runtime/provider path with a local implementation.

## Best Possible Solution

Replace this with a narrow cache-cap patch limited to the affected caches, preserving bootstrap refresh and registry-backed web_search, with focused eviction tests and redacted long-running daemon/cache proof.

## Reproduction Assessment

No high-confidence runtime reproduction is present. Source inspection shows current main's target maps are still unbounded, but there is no daemon or heap-growth proof for the leak or the proposed eviction behavior.

## Solution Assessment

No. Cache caps are a plausible fix direction, but this PR mixes unrelated provider/schema changes and regresses current bootstrap and web_search contracts; a narrow cache-only patch is the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve bootstrap refresh behavior:** `src/agents/bootstrap-cache.ts:23-24`
  - body: Current main reloads bootstrap files before reusing a cached snapshot so long-running sessions pick up workspace edits. This branch returns the cached entry before reloading, so later bootstrap changes can be missed.
  - confidence: 0.9
- **[P1] Route web_search through the runtime provider registry:** `src/agents/tools/web-search.ts:1906`
  - body: This local provider selection no longer consults the runtime/plugin provider registry. Configured providers outside the hard-coded set, including custom runtime providers, can stop working or fall back to the wrong provider.
  - confidence: 0.93
- **[P1] Read search keys from plugin-owned config:** `src/agents/tools/web-search.ts:910`
  - body: These resolvers only read local scoped config or env vars, while current providers store configured credentials under plugin-owned paths such as plugins.entries.google.config.webSearch.apiKey. Existing configured users can get missing-key errors.
  - confidence: 0.9
- **[P2] Preserve sub-second recording durations:** `src/agents/tools/nodes-tool.ts:167`
  - body: Adding minimum: 1000 rejects durationMs values below one second, while current main only enforces the 300000 ms maximum. That validation tightening is unrelated to cache caps and breaks previously accepted node recording inputs.
  - confidence: 0.86
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92-93`
  - body: This evicts before every set, including same-key refreshes. At capacity, refreshing a changed file can delete an unrelated entry and then overwrite the existing key, causing avoidable cache churn and disk reads.
  - confidence: 0.8
- **[P3] Apply model-cache limits consistently:** `src/agents/context.ts:52`
  - body: Configured context-window entries still bypass the new cap, so a large configured model set can overfill the map and later have configured entries evicted by the discovered-model path.
  - confidence: 0.73
