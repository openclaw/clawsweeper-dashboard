## Summary

Keep open. The cache-cap change is plausible, but the branch also rewrites web_search outside the provider registry, breaks plugin-owned provider/config behavior, tightens an unrelated nodes schema, has failing CI, and lacks real after-fix behavior proof.

## What This Changes

The PR adds oldest-entry eviction caps to several agent-runtime caches and also changes agent tool schemas plus web_search provider/auth execution.

## Best Possible Solution

Keep the cache eviction work narrow, preserve the existing provider registry and plugin-owned credential contracts, and remove unrelated tool-schema/API behavior changes unless they are handled in separate focused PRs.

## Reproduction Assessment

Yes for the review blockers: the PR source and live CI show provider auto-detect regressions for Firecrawl/Tavily, and the schema diff shows sub-second durationMs values are newly rejected. For the original cache leak, the unbounded maps are source-visible but I did not run a long-lived daemon reproduction in this read-only review.

## Solution Assessment

No. The cache caps should be implemented without replacing the web_search runtime registry or tightening unrelated tool schemas; the safer path is a small cache-only patch plus focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The new hard-coded provider list excludes current plugin-provided web_search providers such as Firecrawl, Exa, Tavily, DuckDuckGo, and installed custom providers. This breaks the documented provider registry contract and is already reflected by PR CI failures where Firecrawl/Tavily auto-detect falls back to Brave.
  - confidence: 0.93
- **[P1] Preserve plugin-owned web search credentials:** `src/agents/tools/web-search.ts:553-563`
  - body: The new local auth resolvers read legacy tools.web.search fields and env vars, but current provider contracts store credentials under plugins.entries.<plugin>.config.webSearch.*. Users with configured plugin-owned keys will get missing-key errors even though the selected provider is configured.
  - confidence: 0.9
- **[P2] Keep sub-second camera clips valid:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding minimum: 1000 to durationMs rejects previously valid camera_clip durations below one second. That is a behavior change outside the cache fix and contradicts the PR's no behavior changes claim.
  - confidence: 0.82
