## Summary

Keep open. The memory-growth problem is still real on current main, but this PR is not safe to merge as-is because it mixes the cache caps with a large web_search rewrite and unrelated tool-schema tightening, introducing provider registry, SecretRef, and cache-refresh regressions.

## What This Changes

The PR adds FIFO size caps to four agent runtime Map caches and also changes agent tool schemas plus web_search provider and credential handling.

## Best Possible Solution

Land a narrow cache-only replacement on current main that caps the four intended Maps with deterministic oldest-entry eviction only when inserting new keys, preserves provider/SecretRef and tool-schema behavior, and adds focused regression tests for cache size, same-key refresh, and configured/discovered context-window interactions.

## Reproduction Assessment

Yes. Current main has module-level Maps in the reported paths with distinct-key writes and no maximum size; repeated distinct session keys, workspace paths, trace files, or model IDs reproduce unbounded growth at source level.

## Solution Assessment

No. The cache-cap direction is appropriate, but this PR is not the best way to solve it because it bundles unrelated provider, credential, and schema changes with the memory fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route web_search through the runtime provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The PR replaces the registry-backed web_search path with hard-coded provider handling. Current main supports bundled and runtime providers outside this list, so firecrawl, exa, tavily, duckduckgo, or custom plugin providers can stop working or fall back incorrectly.
  - confidence: 0.93
- **[P1] Read web_search keys from plugin-owned SecretRefs:** `src/agents/tools/web-search.ts:553-563`
  - body: These resolvers use legacy tool config or env vars instead of the plugin-owned webSearch config and resolved runtime SecretRefs. Existing users with migrated plugin config can get missing-key failures even though their provider is configured.
  - confidence: 0.9
- **[P2] Evict workspace cache entries only for new keys:** `src/agents/workspace.ts:92`
  - body: The eviction helper runs before every workspaceFileCache.set, including refreshes of a path already in the map. At capacity, refreshing an existing file evicts an unrelated oldest entry and reduces effective cache capacity.
  - confidence: 0.88
- **[P2] Preserve sub-second node clip durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding minimum: 1000 to durationMs rejects 0-999 ms values that current main accepts up to the existing 300000 ms maximum. That is an unrelated behavior change for node camera/screen operations.
  - confidence: 0.82
- **[P3] Apply the model cache cap consistently:** `src/agents/context.ts:52-58`
  - body: Only the discovered-model path enforces the new model-cache cap, while configured context windows write to the same cache without a guard. A large configured model list can exceed the cap and later have configured entries evicted.
  - confidence: 0.76
