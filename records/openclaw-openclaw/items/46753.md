## Summary

Keep open: the cache-capping goal is valid from source inspection, but this branch is not mergeable as-is because it mixes in an unrelated core `web_search` provider rewrite that bypasses the current plugin/runtime provider registry and plugin-owned credential paths.

## What This Changes

The branch adds oldest-first caps to four global agent caches and also changes many agent tool schemas, including a full core `web_search` provider rewrite.

## Best Possible Solution

Land a narrow cache-only fix on current main that preserves the plugin/runtime provider contract, guards eviction on new-key insertions, adds focused regression tests, and leaves web-search/schema hardening to separate owner-reviewed work.

## Reproduction Assessment

Yes, source-level: current main has module-level Maps in the reported cache paths without size caps. I did not run a long-lived daemon memory test, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. Oldest-first caps are a reasonable direction, but this PR is not the best implementation because it mixes the cache fix with unrelated provider/schema behavior changes; a cache-only patch with tests is the maintainable route.

  - confidence: 0.88
