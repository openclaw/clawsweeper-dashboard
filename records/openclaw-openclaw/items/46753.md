## Summary

Keep open. The reported memory-growth problem is real on current main, but this PR is not safe to merge as-is because it mixes the cache caps with unrelated web_search provider/credential rewrites and schema tightening that introduce concrete regressions.

## What This Changes

The PR adds FIFO size caps to four agent runtime caches while also rewriting web_search and tightening many agent tool schemas.

## Best Possible Solution

Land a narrow cache-only replacement on current main that caps the four intended Maps, evicts only on new-key inserts, preserves existing web_search/provider/SecretRef/schema behavior, and adds focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. Current main has module-level Maps in the reported paths with distinct-key writes and no maximum size; repeated distinct session keys, workspace paths, trace files, or model IDs reproduce unbounded growth at source level.

## Solution Assessment

No. The cache-cap direction is right, but this PR is not the best implementation because it bundles unrelated web_search, credential, and schema changes with regressions into the memory patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep web_search on the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The rewrite replaces current main's `runWebSearch`/runtime-provider path with hard-coded provider clients. Current main tests require custom runtime providers and bundled Firecrawl/Tavily/DuckDuckGo providers to execute through the active plugin registry, so configured providers would be dropped or rejected.
  - confidence: 0.93
- **[P1] Keep web_search credentials on plugin SecretRefs:** `src/agents/tools/web-search.ts:553-563`
  - body: The new resolver reads legacy tool config/env paths instead of plugin-owned `webSearch` config and runtime-resolved SecretRefs. Current onboarding and doctor migrations store provider keys under `plugins.entries.<plugin>.config.webSearch`, so migrated users can see missing-key failures despite configured credentials.
  - confidence: 0.9
- **[P2] Evict workspace entries only for new paths:** `src/agents/workspace.ts:92`
  - body: The helper runs before every `workspaceFileCache.set`. When a cached file is refreshed at capacity, this deletes an unrelated oldest entry and then overwrites the same key, reducing effective capacity and adding avoidable disk reads.
  - confidence: 0.88
- **[P2] Preserve sub-second node clip durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding `minimum: 1000` to `durationMs` rejects 0-999 ms clips that current main accepts with only the existing maximum. That is unrelated to cache caps and can break node media callers.
  - confidence: 0.82
- **[P3] Apply the model cache cap consistently:** `src/agents/context.ts:52-58`
  - body: Only `applyDiscoveredContextWindows` enforces the new model-cache cap, while `applyConfiguredContextWindows` writes to the same cache unguarded. Large configured model lists can exceed the cap and later have configured entries evicted by discovered-model inserts.
  - confidence: 0.76
