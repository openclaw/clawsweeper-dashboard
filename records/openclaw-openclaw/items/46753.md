## Summary

Keep open: the cache-capping goal is useful, but this PR is not safe to merge because it bundles a stale inline `web_search` rewrite that breaks registry-backed providers and plugin-owned credentials, plus unrelated schema tightening that changes existing tool behavior.

## What This Changes

The PR caps four global agent runtime maps and also adds tool-schema bounds/descriptions, including a large inline `web_search` provider implementation.

## Best Possible Solution

Replace or rework this PR as a narrow current-main cache-cap patch that preserves the provider registry, plugin-owned credential resolution, bootstrap refresh semantics, and existing tool validation contracts.

## Reproduction Assessment

Yes. Source inspection shows the target caches are module-level `Map` instances without size caps on current main, and the PR regressions are source-reproducible by comparing the hard-coded branch code with current registry tests and config migration tests.

## Solution Assessment

No. Oldest-first caps are a reasonable narrow direction, but this PR is not the best implementation because it mixes in a large unrelated `web_search` rewrite and behavior-changing schema bounds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The PR replaces the registry-backed `web_search` path with a fixed provider list. Current main expects runtime providers such as custom, Firecrawl, Exa, Tavily, and DuckDuckGo to execute through the plugin registry, so selections outside this hard-coded list will stop working or fall back incorrectly.
  - confidence: 0.93
- **[P1] Keep plugin-owned web search credentials wired:** `src/agents/tools/web-search.ts:905-911`
  - body: These provider-specific resolvers only read legacy `tools.web.search.*` fields or environment variables. Current migration and provider code store configured keys under `plugins.entries.<provider>.config.webSearch`, so users with valid plugin-owned config can receive missing-key errors after this change.
  - confidence: 0.9
- **[P2] Do not reject sub-second camera clips:** `src/agents/tools/nodes-tool.ts:165-168`
  - body: Adding `minimum: 1000` to `durationMs` rejects values from 0-999 ms that the current schema accepts. That is an unrelated validation break for existing node camera workflows and is not covered by the cache-cap goal.
  - confidence: 0.87
- **[P3] Evict workspace files only for new keys:** `src/agents/workspace.ts:90-93`
  - body: The eviction call runs before every refreshed `workspaceFileCache.set`, including updates to a key already in the map. At capacity, refreshing a changed file evicts an unrelated oldest entry and then overwrites the same key, shrinking the effective cache and causing avoidable disk reads.
  - confidence: 0.78
