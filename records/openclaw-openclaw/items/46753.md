## Summary

Keep open, but do not merge as-is: source inspection supports the cache-leak concern, while the branch mixes that fix with a large `web_search` rewrite and schema tightening that break current provider/config contracts and lacks real behavior proof.

## What This Changes

The branch caps four agent-runtime global maps and also adds broad tool-schema bounds/descriptions plus a large direct `web_search` provider implementation.

## Best Possible Solution

Land a narrow cache-cap change that preserves the current web-search runtime/provider registry and plugin config contracts, guards same-key cache refreshes, and leaves schema hardening for separate review.

## Reproduction Assessment

No for a live daemon reproduction: current main has source-visible unbounded `Map` caches, but the PR body says long-running daemon behavior was not verified and this read-only review did not run one.

## Solution Assessment

No. The narrow maintainable solution is cache caps with regression coverage and same-key insertion guards, not a broad `web_search` provider rewrite and unrelated schema tightening.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: This new local provider list only supports five hard-coded providers and no longer executes providers from the active runtime/plugin registry. Current main tests require custom/runtime providers and runtime-selected providers to run through `runWebSearch`, so Firecrawl, Tavily, and third-party search providers would stop working.
  - confidence: 0.92
- **[P1] Read search keys from plugin-owned config:** `src/agents/tools/web-search.ts:553-563`
  - body: These new resolvers read legacy `tools.web.search.*` fields or env vars directly, but current provider docs and tests use `plugins.entries.<provider>.config.webSearch.*` plus provider-level fallbacks. Users configured through current onboarding can now get missing-key errors even though their provider is configured.
  - confidence: 0.9
- **[P2] Preserve sub-second recording durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding `minimum: 1000` to `durationMs` rejects values below one second that current main accepts for camera and screen recording. That is an unrelated validation tightening for a cache-cap PR and can silently break existing agent calls.
  - confidence: 0.84
- **[P2] Evict only when inserting a new workspace cache key:** `src/agents/workspace.ts:92`
  - body: The eviction helper runs before every cache write, including refreshes for a file path already present in the map. At capacity, a changed tracked file evicts an unrelated oldest entry and then overwrites the existing key, causing unnecessary churn and reducing effective cache utility.
  - confidence: 0.78
