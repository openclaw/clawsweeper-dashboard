## Summary

Keep open: the cache-cap direction is useful, but this PR is not mergeable as-is because it regresses bootstrap refresh, bypasses the registry-backed web_search/provider credential contract, tightens existing node duration validation, lacks real daemon/cache proof, and broadens the web_search secrets/network surface outside the stated memory fix.

## What This Changes

The branch caps four agent-runtime cache maps and also adds broad tool-schema bounds/descriptions plus a large direct web_search provider implementation.

## Best Possible Solution

Refresh or replace this with a narrow cache-cap patch that preserves bootstrap refresh and registry-backed web_search, adds focused eviction tests, and includes redacted daemon/cache proof.

## Reproduction Assessment

No high-confidence runtime reproduction is present. Source inspection shows the target maps are still uncapped on current main, but the PR does not demonstrate heap growth or eviction behavior in a long-running daemon.

## Solution Assessment

No. Cache caps are a plausible fix direction, but this implementation bundles unrelated provider/schema changes and regresses existing contracts; a narrow cache-only patch is the safer maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve bootstrap refresh behavior:** `src/agents/bootstrap-cache.ts:23-24`
  - body: Current main reloads bootstrap files each turn and only reuses the cached snapshot when the workspace and file contents still match. This branch returns the cached session entry before reloading, so long-running sessions can miss later workspace bootstrap edits.
  - confidence: 0.9
- **[P1] Route web_search through the runtime provider registry:** `src/agents/tools/web-search.ts:1903-1906`
  - body: The local provider selection no longer consults the runtime/plugin provider registry, so configured providers outside the hard-coded list stop working even though current code expects registry-backed runtime providers.
  - confidence: 0.93
- **[P1] Read search keys from plugin-owned config:** `src/agents/tools/web-search.ts:905-910`
  - body: These resolvers read legacy tools.web.search.* fields or env vars, but current bundled providers store credentials under plugin-owned paths such as plugins.entries.google.config.webSearch.apiKey. Existing configured users can get missing-key errors after this change.
  - confidence: 0.9
- **[P2] Preserve sub-second recording durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding minimum: 1000 rejects durationMs values below one second, while current main only enforces the 300000 ms maximum. That validation tightening is unrelated to cache caps and breaks previously accepted node recording inputs.
  - confidence: 0.86
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92-93`
  - body: This evicts before every set, including same-key refreshes. At capacity, refreshing a changed file can delete an unrelated entry and then overwrite the existing key, causing unnecessary cache churn and disk reads.
  - confidence: 0.8
- **[P3] Apply model-cache limits consistently:** `src/agents/context.ts:82`
  - body: Configured context-window entries still bypass the new cap, so a large configured model set can overfill the map and later have configured entries evicted by the discovered-model path.
  - confidence: 0.73
