## Summary

Keep open: the cache-growth concern is real from source inspection, but this PR is not mergeable as-is because it lacks real runtime proof, is stale against current bootstrap behavior, and bundles unrelated web-search/schema changes that introduce regressions.

## What This Changes

The PR adds oldest-first caps to several global agent-runtime cache maps while also changing many tool schemas and replacing the web_search runtime implementation.

## Best Possible Solution

Split this into a narrow cache-cap change with focused eviction tests and runtime proof, preserving current bootstrap refresh behavior, registry-backed web_search, plugin-owned credential paths, and existing schema compatibility.

## Reproduction Assessment

No high-confidence runtime reproduction is present. Source inspection shows the targeted maps are uncapped on current main, but the PR body does not demonstrate heap growth or eviction behavior in a long-running daemon.

## Solution Assessment

No. Simple caps may be a reasonable fix direction, but this PR is not the best solution because it combines the cache change with unrelated provider rewrites and schema validation changes while lagging current bootstrap-cache behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve bootstrap refresh behavior:** `src/agents/bootstrap-cache.ts:23-24`
  - body: Current main reloads bootstrap files each turn and reuses the cached snapshot only when the workspace and file contents still match. This branch returns the cached session entry before reloading, so long-running sessions can miss later AGENTS/bootstrap edits.
  - confidence: 0.9
- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The hard-coded provider list bypasses `runWebSearch` and the runtime/plugin provider registry, so configured providers outside this list stop working even though current tests require active runtime providers to execute.
  - confidence: 0.93
- **[P1] Read search keys from plugin-owned config:** `src/agents/tools/web-search.ts:553-563`
  - body: These local resolvers read legacy `tools.web.search.*` fields or env vars, but current providers store credentials under plugin-owned paths such as `plugins.entries.google.config.webSearch.apiKey`. Existing configured users can get missing-key errors after this change.
  - confidence: 0.9
- **[P2] Preserve sub-second recording durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding `minimum: 1000` rejects `durationMs` values below one second, while current main only enforces the 300,000 ms maximum. That validation tightening is unrelated to cache caps and breaks previously accepted node recording inputs.
  - confidence: 0.86
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92-93`
  - body: This evicts before every set, including same-key refreshes. At capacity, refreshing a changed file can delete an unrelated entry and then overwrite the existing key, reducing effective cache size without lowering memory for the refreshed key.
  - confidence: 0.8
- **[P3] Apply model-cache limits consistently:** `src/agents/context.ts:82`
  - body: Configured context-window entries still bypass the new cap, so a large configured model set can overfill the map and later have configured entries evicted by the discovered-model path.
  - confidence: 0.73
