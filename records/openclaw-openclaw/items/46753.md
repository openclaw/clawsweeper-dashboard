## Summary

Keep open: current main still has the source-level unbounded-cache concern, but this PR has blocking regressions, broad unrelated web_search/security-surface changes, and no real after-fix runtime proof.

## What This Changes

The branch adds oldest-first caps to four global agent-runtime cache maps while also changing many agent tool schemas and replacing registry-backed web_search with local provider code.

## Best Possible Solution

Split this into a narrow cache-cap change with focused eviction tests and real runtime proof, preserving bootstrap refresh, registry-backed web_search, plugin-owned credential paths, and existing schema compatibility.

## Reproduction Assessment

No high-confidence runtime reproduction is present. Source inspection shows the target maps are uncapped on current main, but the PR body does not demonstrate heap growth or eviction behavior in a long-running daemon.

## Solution Assessment

No. Cache caps may be a reasonable fix direction, but this branch is not the best solution because it combines that fix with unrelated provider rewrites and schema validation changes while lagging current bootstrap-cache behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve bootstrap refresh behavior:** `src/agents/bootstrap-cache.ts:23-24`
  - body: Current main reloads bootstrap files each turn and only reuses the cached snapshot when the workspace and file contents still match. This branch returns the cached session entry before reloading, so long-running sessions can miss later workspace bootstrap edits.
  - confidence: 0.9
- **[P1] Route web_search through the runtime provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The hard-coded provider list bypasses runWebSearch and the runtime/plugin provider registry, so configured providers outside this list stop working even though current tests cover registry-backed runtime providers.
  - confidence: 0.93
- **[P1] Read search keys from plugin-owned config:** `src/agents/tools/web-search.ts:553-563`
  - body: These resolvers read legacy tools.web.search.* fields or env vars, but current bundled providers store credentials under plugin-owned paths such as plugins.entries.google.config.webSearch.apiKey. Existing configured users can get missing-key errors after this change.
  - confidence: 0.9
- **[P2] Preserve sub-second recording durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding minimum: 1000 rejects durationMs values below one second, while current main only enforces the 300000 ms maximum. That validation tightening is unrelated to cache caps and breaks previously accepted node recording inputs.
  - confidence: 0.86
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92-93`
  - body: This evicts before every set, including same-key refreshes. At capacity, refreshing a changed file can delete an unrelated entry and then overwrite the existing key, shrinking effective cache size and causing unnecessary disk reads.
  - confidence: 0.8
- **[P3] Apply model-cache limits consistently:** `src/agents/context.ts:82`
  - body: Configured context-window entries still bypass the new cap, so a large configured model set can overfill the map and later have configured entries evicted by the discovered-model path.
  - confidence: 0.73
