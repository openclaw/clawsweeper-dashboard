## Summary

Keep open: current main still has source-visible unbounded cache maps, but this PR is not mergeable as-is because it adds unrelated provider/schema changes, breaks existing web_search contracts, and lacks real long-running behavior proof.

## What This Changes

The PR adds oldest-first caps to four agent-runtime cache maps while also tightening many tool schemas and replacing web_search provider delegation with a local implementation.

## Best Possible Solution

Land a narrow cache-cap change with focused eviction tests, while preserving web_search provider-registry delegation, plugin-owned credential paths, and existing schema compatibility.

## Reproduction Assessment

No. Source inspection shows current main has unbounded maps in the targeted cache paths, but this review did not reproduce heap growth in a long-running daemon and the PR body says that was not verified.

## Solution Assessment

No. Cache caps are a reasonable direction, but this PR is not the best fix because it mixes them with unrelated provider rewrites and schema tightening that break current contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The replacement implementation no longer calls the runtime/plugin provider registry, so bundled and third-party providers outside the hard-coded list stop working even though current tests require active registry providers to execute through runWebSearch.
  - confidence: 0.92
- **[P1] Read search keys from plugin-owned config:** `src/agents/tools/web-search.ts:553-563`
  - body: These resolvers read legacy tools.web.search fields or environment variables directly, but current onboarding and providers use plugin-owned config paths; configured users can now get missing-key errors.
  - confidence: 0.9
- **[P2] Preserve sub-second recording durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding minimum: 1000 rejects sub-second durationMs values that current main accepts, and that validation tightening is unrelated to the cache-cap change.
  - confidence: 0.84
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92`
  - body: This eviction runs before every set, including same-key refreshes; at capacity, refreshing an existing file can evict an unrelated entry and shrink effective cache size without reducing memory.
  - confidence: 0.78
- **[P3] Apply model-cache limits consistently:** `src/agents/context.ts:82`
  - body: Configured context-window entries bypass the same MODEL_CACHE cap used for discovered entries, so a large configured model set can overfill the map and later evict configured values from the discovered path.
  - confidence: 0.72
