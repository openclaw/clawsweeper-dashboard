The PR adds optional `sessionId` Runtime prompt metadata, renders it as `session=<uuid>`, wires one command prompt path, adds a focused assertion, and updates the changelog.

## Best Possible Solution

Thread the opaque session UUID from the canonical current or target session through every Runtime prompt path, add focused coverage, and require redacted real behavior proof showing the resulting Runtime line.

## Reproduction Assessment

Not applicable as a bug reproduction: this is additive Runtime prompt metadata, not a broken existing contract. Source inspection gives a high-confidence behavior check that current main lacks `session=` metadata and that the PR head's real prompt paths are incomplete.

## Solution Assessment

No, not yet. Exposing the opaque `sessionId` is safer than exposing raw `sessionKey`, but the PR should also wire embedded prompts and use the target session entry for command Runtime metadata before merge.
