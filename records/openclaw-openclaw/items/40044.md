## Summary

Keep this PR open. Current main still does not expose session identity in the system prompt Runtime line, so the contribution is not obsolete, but the branch is incomplete because it wires the new field only through the command-system prompt path while the normal embedded runner Runtime assembly still omits the available `params.sessionId`.

## What This Changes

The PR adds optional `sessionId` Runtime metadata, renders `session=<uuid>` in the system prompt Runtime line, wires it for command-system prompts, adds a focused runtime-line assertion, and adds a changelog entry.

## Best Possible Solution

Land one additive prompt-visible session metadata contract across the shared Runtime type/renderer and every production prompt assembly path, including the embedded runner and command-system prompts, with focused tests plus docs and changelog kept aligned.

## Reproduction Assessment

Yes. Current main can be checked by inspecting `buildRuntimeLine` and the two `buildSystemPromptParams` production call sites: neither `sessionId` nor `sessionKey` is rendered today, and the embedded runner has `params.sessionId` available but does not pass it into Runtime metadata.

## Solution Assessment

No. The direction is reasonable, but this branch is not the narrowest complete fix because normal embedded agent runs would still omit `session=<uuid>`; the safer path is to wire the chosen identifier through both command and embedded Runtime assembly before merge.
