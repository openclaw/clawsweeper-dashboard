## Summary

Keep open: the requested prompt metadata is still absent from current main, and this PR is not merge-ready because it misses the main embedded prompt path, can report the wrapper session in command prompts, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR adds optional `sessionId` Runtime prompt metadata, renders it as `session=<uuid>`, wires one command-prompt runtime value, adds a runtime-line test assertion, and adds a changelog entry.

## Best Possible Solution

Thread the opaque session UUID through every real prompt runtime path from the canonical current or target session, with focused coverage for embedded prompts and target-session command prompts while keeping raw session keys out of provider-visible prompts.

## Reproduction Assessment

Not applicable as a bug reproduction: this is additive prompt metadata. Source inspection gives a high-confidence behavior check that current main lacks `session=` Runtime metadata and that the PR branch still misses or misattributes important prompt paths.

## Solution Assessment

No. `sessionId` is the safer direction versus raw `sessionKey`, but the patch should also thread the field through embedded prompts and use `targetSessionEntry?.sessionId` for command Runtime metadata.

## Review Findings

Overall correctness: patch is incorrect

Regular agent prompts would continue to omit the advertised session UUID, so pass it through that runtime path and add focused coverage.
  - confidence: 0.91

Commands targeting another session can advertise the wrapper session UUID; use `targetSessionEntry?.sessionId` and cover the target-session case.
  - confidence: 0.88
