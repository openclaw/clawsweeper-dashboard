## Summary

Keep open: the PR is useful and current main still lacks Runtime session metadata, but the branch is not merge-ready because it wires only part of the prompt surface and can use the wrong session entry for command prompts.

## What This Changes

The PR adds optional `sessionId` Runtime metadata in shared system-prompt types/rendering, wires one command-system-prompt path, adds a runtime-line assertion, and adds a changelog entry.

## Best Possible Solution

Land one additive Runtime metadata contract that exposes the non-secret session UUID consistently through the shared renderer and every production prompt assembly path, with docs and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection is enough: current main has no `session=` Runtime segment, and the PR diff still leaves the embedded runner path unwired while using the wrong command-session source in one path.

## Solution Assessment

No. The shared renderer/type direction is reasonable, but the PR is not the best complete fix until it uses the resolved command target session entry and threads `params.sessionId` through embedded Runtime metadata too.

  - body: The PR passes `params.sessionEntry?.sessionId`, but this function already resolves `targetSessionEntry` from `sessionStore` and uses it for the target session's bootstrap and tool metadata. When a command targets a session whose store entry differs from the wrapper entry, the Runtime line would show the wrong session UUID.
  - confidence: 0.86

  - confidence: 0.88
