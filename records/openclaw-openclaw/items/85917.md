## Summary

Keep this PR open, but it should not merge as-is: the central early-registration fix is the right direction for the reported abort race, yet the patch creates a duplicate-send path that can return `in_flight` and then never receive a durable terminal result if attachment prep fails.

## What This Changes

The PR moves `chat.send` abort-controller registration before attachment preparation and adds gateway chat tests for duplicate sends and abort-during-prep behavior.

## Best Possible Solution

Keep the early abort-controller registration, but also persist a terminal `chat:<runId>` error result on attachment-prep failure and add focused coverage for the duplicate `in_flight` failure case before merge.

## Reproduction Assessment

Yes. Source inspection shows current main awaits attachment preparation before registering the abort controller, while `chat.abort` returns `aborted:false` when the controller entry is absent.

## Solution Assessment

No, not as submitted. Early registration is the maintainable fix direction, but the attachment-prep failure path needs a terminal dedupe write so duplicate callers do not remain stuck on `in_flight`.
