## Summary

Keep open: the branch targets a valid Feishu queue/session bug, but it is merge-conflicting, proof-missing, and still computes queue keys before native topic hydration, so it can break same-topic FIFO.

## What This Changes

The PR threads Feishu config into sequential-key resolution so grouped Feishu messages can queue by the same group, sender, topic, or topic-sender identity used for session routing.

## Best Possible Solution

Refresh or replace this with a narrow Feishu-only fix that derives queue keys from the same hydrated session identity used by handling, preserves chat-wide stop and btw lanes, removes the changelog edit, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 compute normal Feishu sequential keys at chat scope while session routing can use group_topic or group_topic_sender peer IDs. I did not run a live Feishu before-fix scenario in this read-only review.

## Solution Assessment

No. Reusing Feishu group-session identity is the right direction, but this branch computes the queue key from the raw event before native topic hydration and lacks real behavior proof, so it is not the safest fix yet.
