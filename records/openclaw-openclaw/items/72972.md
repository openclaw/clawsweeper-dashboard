## Summary

Keep open: current main still uses chat-wide Feishu sequential keys while group routing can split the same chat into scoped peers; this PR applies the right narrow fix and I found no blocking functional or security issue.

## What This Changes

The PR threads Feishu account config into inbound handler registration, derives group-message queue keys from `resolveFeishuGroupSession`, and adds sequential-key coverage plus a changelog entry.

## Best Possible Solution

Merge this targeted Feishu fix after maintainer approval, keeping `resolveFeishuGroupSession` as the single source of truth for group peer ids and keeping the branch current with main.

## Reproduction Assessment

Yes. Configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`, then send ordinary group messages in different roots or threads within the same chat; current main computes the same chat-wide queue key before routing them to separate scoped peers.

## Solution Assessment

Yes. Reusing the existing group-session resolver and passing the resolved account config into queue-key selection is the narrowest maintainable fix, and it preserves the existing `/stop` and `/btw` lanes.
