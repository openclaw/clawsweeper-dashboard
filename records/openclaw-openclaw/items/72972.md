## Summary

Keep open for maintainer review and merge consideration. Current main still serializes ordinary Feishu group messages by chat before routing them into scoped group sessions, while the PR aligns the queue key with the existing group-session resolver and I found no concrete blocking bug in the latest diff.

## What This Changes

The PR passes resolved Feishu account config into inbound handler registration, derives group-message sequential keys through the existing group-session resolver, adds sequential-key coverage for group scopes and overrides, and adds a changelog fix entry.

## Best Possible Solution

Merge this targeted Feishu fix after normal maintainer approval, keeping `resolveFeishuGroupSession` as the single source of truth for Feishu group peer ids and preserving the existing control and side-question lanes.

## Reproduction Assessment

Yes. Configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`, then send ordinary group messages in different roots or threads within the same chat; current main computes the same chat-wide sequential key before routing them to separate scoped peers.

## Solution Assessment

Yes. Reusing the existing group-session resolver and threading the resolved account config into queue-key selection is the narrowest maintainable fix because it avoids duplicating Feishu topic/sender fallback logic.
