## Summary

Keep open. Current main still serializes normal Feishu inbound group messages at the chat level even though the routing layer can split those messages by configured sender/topic group sessions; the PR aligns those two paths and I found no concrete correctness or security blocker in the supplied latest diff.

## What This Changes

The PR threads Feishu account config into receive-handler registration, scopes Feishu group-message sequential keys through the existing group-session resolver, adds focused sequential-key coverage, and adds a changelog fix entry.

## Best Possible Solution

Land this targeted queue-key alignment after normal maintainer review and green required checks, keeping `resolveFeishuGroupSession` as the single source of truth for Feishu group peer ids while preserving the stable per-chat control and `/btw` lanes.

## Reproduction Assessment

Yes. Configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"` and send two ordinary group messages in different topics or roots in the same chat; current main routes them to different peers but enqueues both under the same chat-level key.

## Solution Assessment

Yes. Reusing the existing Feishu group-session resolver is the narrowest maintainable fix because it avoids duplicating topic/sender fallback rules and keeps routing and serialization aligned.
