## Summary

Keep open. Current main still queues normal Feishu group messages by whole chat even when routing already splits the conversation by configured group topic or sender scope; this PR is the right code direction, but it still needs the required user-facing changelog entry before merge.

## What This Changes

The PR threads Feishu account config into message-handler registration, changes Feishu group-message sequential keys to reuse group-session peer resolution, adds Feishu sequential-key scope tests, and updates Matrix approval runtime test fixtures.

## Best Possible Solution

Land the targeted queue-key alignment after adding a single Unreleased Feishu fix entry and getting green relevant CI, keeping `resolveFeishuGroupSession` as the single source of truth for Feishu group peer IDs so routing and serialization remain aligned.

## Reproduction Assessment

Yes. A code-level reproduction is clear: configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`, send two normal group messages in different roots/topics, and current main queues both under `feishu:<accountId>:<chatId>` even though routing resolves different peer IDs.

## Solution Assessment

Yes for the code path. Reusing the existing Feishu group-session resolver is the narrowest maintainable fix because it preserves `/stop` and `/btw` chat-level lanes while avoiding duplicate topic/sender fallback logic; the patch still needs the required changelog entry.
