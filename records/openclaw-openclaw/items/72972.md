## Summary

Keep open: the PR targets a real Feishu queue-scoping bug, but the patch needs one correctness fix for native `topic_group` starter events and still needs contributor-supplied real Feishu behavior proof before merge.

## What This Changes

The PR threads Feishu account config into sequential-key selection, derives group queue keys from `resolveFeishuGroupSession`, adds Feishu sequential-key tests, and updates the changelog.

## Best Possible Solution

Revise queue-key selection so it cannot diverge from routed peers for native `topic_group` starters with missing `thread_id`, then land the focused Feishu fix after redacted real behavior proof is added.

## Reproduction Assessment

Yes, from source inspection. Configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`; current main computes the chat-wide queue key before routing can split messages into scoped peers, and current tests show native topic starters can be hydrated to a thread id only after that point.

## Solution Assessment

No, not as-is. Reusing `resolveFeishuGroupSession` is the right direction for populated topic ids, but the PR resolves the queue key before native topic hydration and can assign a starter and follow-up in the same topic to different lanes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep native topic starters on the hydrated lane:** `extensions/feishu/src/sequential-key.ts:35-41`
  - body: Current main hydrates missing `thread_id` for `topic_group` starters before routing them to `oc-group:topic:<thread>`, but this queue key is resolved earlier and lets `resolveFeishuGroupSession` fall back to `message_id`. A starter with no `root_id`/`thread_id` and its follow-up with `thread_id` can then run on different queue keys while routing to the same session. Keep this case on a conservative lane or move hydration before key selection, and add `topic_group` starter coverage.
  - confidence: 0.9
