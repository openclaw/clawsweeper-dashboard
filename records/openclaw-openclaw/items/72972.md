## Summary

Keep open for normal PR review and merge gating. Current main still queues ordinary Feishu group messages by chat even when routing splits the same chat by topic or sender, and the PR aligns queue keys with the existing group-session resolver without a concrete blocking finding.

## What This Changes

The PR threads Feishu account config into inbound handler registration, resolves ordinary group-message queue keys through the existing Feishu group-session resolver, adds focused sequential-key coverage, and adds a changelog fix entry.

## Best Possible Solution

Land this targeted queue-key alignment after maintainer review, keeping `resolveFeishuGroupSession` as the single source of truth for Feishu group peer ids while preserving per-chat control and `/btw` lanes.

## Reproduction Assessment

Yes. Configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"` and send ordinary group messages in different roots/topics in the same chat; current main routes them to different peers but enqueues both under the same chat key.

## Solution Assessment

Yes. Reusing the existing group-session resolver is the narrowest maintainable fix because it avoids duplicating topic/sender fallback logic and keeps routing and serialization aligned.
