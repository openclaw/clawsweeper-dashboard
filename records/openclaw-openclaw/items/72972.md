## Summary

Keep open. Current main still queues normal Feishu inbound messages at the whole-chat level while routing can already split group sessions by sender/topic, so this PR addresses a real mismatch. The Feishu code direction looks right; the remaining concrete repair is the required user-facing changelog entry before merge.

## What This Changes

The PR threads Feishu account config into receive-handler queue-key resolution, scopes normal group message queue keys through Feishu group-session peer resolution, adds Feishu sequential-key scope tests, and updates Matrix approval test fixtures.

## Best Possible Solution

Land the targeted queue-key alignment after adding one Unreleased Feishu fix entry and getting green relevant CI, keeping `resolveFeishuGroupSession` as the single source of truth so Feishu routing and serialization stay aligned.

## Reproduction Assessment

Yes. A code-level reproduction is clear: configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`, send two normal group messages in different roots/topics, and current main queues both under `feishu:<accountId>:<chatId>` even though routing resolves different peer ids.

## Solution Assessment

Yes for the code path. Reusing the existing Feishu group-session resolver is the narrowest maintainable fix because it preserves `/stop` and `/btw` chat-level lanes while avoiding duplicated topic/sender fallback logic; the PR still needs the required changelog entry.
