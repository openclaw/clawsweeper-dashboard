## Summary

Keep open for normal maintainer review and merge consideration. Current main still computes ordinary Feishu group-message queue keys at the chat level before routing those same messages into scoped group sessions, and the PR is a narrow fix that reuses the existing group-session resolver with no concrete review findings.

## What This Changes

The PR threads Feishu account config into inbound handler registration, changes `getFeishuSequentialKey` to derive group-message queue keys from `resolveFeishuGroupSession`, adds sequential-key scope tests, and adds a changelog entry.

## Best Possible Solution

Merge this targeted Feishu fix after normal maintainer approval, keeping `resolveFeishuGroupSession` as the single source of truth for Feishu group peer ids and preserving the existing control and side-question queue lanes.

## Reproduction Assessment

Yes. Configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`, then send ordinary group messages in different roots or threads within the same chat; current main computes the same chat-wide sequential key before routing them to separate scoped peers.

## Solution Assessment

Yes. Reusing the existing group-session resolver and threading resolved account config into queue-key selection is the narrowest maintainable fix because it avoids duplicating Feishu topic/sender fallback logic.
