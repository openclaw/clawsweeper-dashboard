## Summary

Keep this PR open. Current main implements the basic Feishu session-account precedence and unknown-account fallback, but it still does not implement the PR discussion’s final channel-origin guard; the submitted branch adds that missing guard, but it is currently dirty against main and needs maintainer rebase/adaptation before it can land.

## What This Changes

The PR changes Feishu tool account resolution and Feishu tool call sites so explicit accountId wins, Feishu-origin session account context can beat configured defaultAccount, non-Feishu inherited context is ignored, and routing tests cover fallback, unknown, normalized, and non-Feishu cases.

## Best Possible Solution

Land the channel-origin guard in the Feishu plugin: explicit tool accountId first; normalized inherited account only when ctx.messageChannel is Feishu and the id maps to an enabled Feishu account; configured channels.feishu.defaultAccount next; then normal Feishu default selection. The rebased patch should thread messageChannel through wiki, drive, perm, bitable, doc/media, and scopes resolution and keep the current runtime/SecretRef handling intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu config with accounts a and b plus defaultAccount b, then resolving a Feishu tool from a non-Feishu context such as messageChannel "slack" with agentAccountId "a"; current main lacks the channel guard and can select account a before the configured default.

## Solution Assessment

Yes in direction, but not as currently mergeable. The PR’s final approach is the narrow maintainable fix because the SDK already provides messageChannel, but the branch needs a rebase/adaptation onto current main before it is safe to land.
