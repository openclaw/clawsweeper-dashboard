## Summary

Keep this PR open. Current main and the latest inspected release already implement the basic Feishu session-account-over-default precedence, but they still lack the PR discussion’s final channel-origin guard, so the PR remains useful and should get maintainer rebase/adaptation rather than cleanup closure.

## What This Changes

The PR changes Feishu tool account resolution and Feishu tool call sites so explicit accountId wins, Feishu-origin session account context can beat configured defaultAccount, non-Feishu inherited context is ignored, and targeted routing/doc account tests cover those cases.

## Best Possible Solution

Land the channel-origin guard in the Feishu plugin: explicit tool accountId first, normalized inherited account only when ctx.messageChannel is Feishu and the id maps to an enabled Feishu account, configured channels.feishu.defaultAccount next, then normal Feishu default selection. The rebased patch should thread messageChannel through wiki, drive, perm, bitable, doc/media, and scopes resolution with targeted tests for Feishu and non-Feishu contexts.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu config with accounts a and b plus defaultAccount b, then resolving a Feishu tool from a non-Feishu context such as messageChannel "slack" with agentAccountId "a"; current main lacks the channel guard and can select account a before the configured default.

## Solution Assessment

Yes in direction, but the current PR branch needs maintainer rebase/adaptation. The final approach is the narrow maintainable fix because the SDK already provides messageChannel and the behavior belongs in the Feishu plugin account resolver.
