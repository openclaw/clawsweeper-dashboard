## Summary

Keep open. Current main already implements the core Feishu tool precedence order, but this PR still carries meaningful unique work around Feishu-origin session gating and normalization that is not present on main, so it should be rebased and proven rather than swept closed.

## What This Changes

The PR changes Feishu tool account resolution so explicit accountId wins, then Feishu-origin session account context, then configured defaultAccount fallback, with targeted routing tests.

## Best Possible Solution

Rebase the PR onto current main, preserve only the still-missing Feishu-origin session guard and normalization coverage, then add redacted real Feishu runtime proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the remaining risk: current main passes ctx.agentAccountId into Feishu tools without checking ctx.messageChannel, so a non-Feishu context with a colliding Feishu account id can still select that tenant. I did not run tests because this review is read-only and test commands may create artifacts.

## Solution Assessment

Yes for direction: scoping inherited account precedence to Feishu-origin tool contexts is the narrowest maintainable fix for the remaining cross-channel collision risk. The implementation needs a rebase onto current main and real behavior proof before it is merge-ready.
