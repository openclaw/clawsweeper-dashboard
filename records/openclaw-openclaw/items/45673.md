## Summary

Keep open: current main already covers the original session-over-default Feishu precedence, but this PR still carries a distinct channel-scoping fix and is not merge-ready because the branch would drop newer resolver protections and lacks inspectable real behavior proof.

## What This Changes

The PR threads `messageChannel` through Feishu tool account resolution and updates Feishu routing/doc tests so Feishu-origin sessions can prefer contextual accounts while non-Feishu contexts fall back to configured `defaultAccount`.

## Best Possible Solution

Rebase the channel-scoped account selection onto current main’s resolver, preserve enabled-account and runtime SecretRef behavior, add non-Feishu same-id collision coverage, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main passes both channel and account context into plugin tool context, but Feishu account resolution only receives `agentAccountId`, so a non-Feishu session whose account id matches a configured Feishu account can select that Feishu account over `defaultAccount`. I did not run tests in this read-only review.

## Solution Assessment

No, not as-is. The channel-scoping direction is narrow and plausible, but this branch must be rebased so it keeps current main’s enabled-account and runtime SecretRef protections.
