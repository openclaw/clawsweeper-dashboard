## Summary

Keep open. Current main still classifies a sustained zero-delay sample as degraded from ELU/CPU thresholds alone, the current unit test locks in that false-positive shape, and the linked fix PR was closed unmerged.

## What This Changes

The issue asks for gateway event-loop health to avoid reporting degraded for idle zero-delay samples where ELU and CPU ratios saturate at 1.0.

## Best Possible Solution

Keep the diagnostic fields, but make zero-delay samples non-degraded and add focused regression coverage without changing protocol shape or adding config.

## Reproduction Assessment

Yes, source-reproducible: the current unit harness models `utilization: 1`, `cpuCoreRatio: 1`, `delayP99Ms: 0`, and `delayMaxMs: 0`, and expects `degraded: true`. I did not run a live Docker/WebSocket gateway in this read-only review.

## Solution Assessment

Yes, the narrow fix belongs in `src/gateway/server/event-loop-health.ts` plus its tests. The fix should cover both ELU-only and CPU-only zero-delay paths; the adjacent open sampling PR handles a different rapid-probe symptom.
