## Summary

Keep open. Current main has a partial post-release guard for sub-second CPU/ELU samples, but the central false positive remains: after a sustained sample with `delayP99Ms: 0` and `delayMaxMs: 0`, the monitor still marks `eventLoop.degraded` from ELU/CPU alone.

## What This Changes

The issue asks OpenClaw to stop reporting gateway event-loop health as degraded for idle no-delay samples where ELU/CPU look saturated.

## Best Possible Solution

Keep the existing diagnostic fields, but make `degraded` and `reasons` require evidence of actual event-loop blocking, with regression coverage for idle no-delay ELU=1.0 samples and no protocol/schema expansion.

## Reproduction Assessment

Yes, from source inspection: the current unit harness already models `utilization: 1`, `cpuCoreRatio: 1`, and zero delay, and expects `degraded: true`. I did not run a live Docker/macOS gateway in this read-only review, so this is source-reproducible rather than live-reproduced here.

## Solution Assessment

Yes, the narrow maintainable fix is in `src/gateway/server/event-loop-health.ts`: preserve the metrics but stop treating ELU/CPU-only saturation with zero delay as degraded. Channel status transport and protocol shape can remain unchanged.
