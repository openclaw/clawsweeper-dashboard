## Summary

Keep this issue open. Current main has adjacent safeguards for config validation, managed package-update service stops, and the cited context-engine regression, but it still does not provide a first-class `openclaw gateway preflight [--json]` command or a shared no-bind startup validation path that exercises Gateway plugin/context-engine startup before accepting traffic.

## What This Changes

Request a Gateway preflight command that validates config, startup plugins, context-engine resolution, and port availability without binding the Gateway port or mutating session state.

## Best Possible Solution

Add `openclaw gateway preflight [--json]` as a Gateway CLI command backed by a shared startup validation routine that parses and validates config, resolves auth/SecretRefs, loads the same startup plugin/context-engine registration paths in an explicit no-bind/no-mutation mode, probes configured bindability without accepting connections, returns stable human/JSON results with exit code 0/1, and documents the command at the public Gateway CLI page before composing it into safe update flows.

## Reproduction Assessment

Yes. The absence is reproducible by source and docs inspection on current main: no `gateway preflight` command is registered or documented, and `gateway run` still proceeds into `startGatewayServer()` and `startListening()`.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is sound, but the maintainable fix is a shared no-bind/no-mutation startup validation path, not a parallel schema-only checker or a dry run that calls full startup side effects.
