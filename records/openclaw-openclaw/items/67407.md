## Summary

Keep this issue open. Current main has adjacent config validation, update hardening, gateway restart preflight, and the cited lossless-claw regression fix, but it still lacks a first-class `openclaw gateway preflight [--json]` command or shared no-bind/no-mutation startup validation path.

## What This Changes

The issue requests a first-class `openclaw gateway preflight [--json]` command that validates config, plugin and context-engine startup paths, and port availability without binding or mutating runtime state.

## Best Possible Solution

Add `openclaw gateway preflight [--json]` behind a shared, side-effect-aware Gateway startup-validation routine, then compose safe update flows on top after the primitive is stable and documented.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new command and startup-validation feature request. Source and docs inspection give high confidence that current main still lacks the requested standalone Gateway preflight surface.

## Solution Assessment

Unclear because there is no patch yet. The requested command is a plausible product surface, but the maintainable solution should split the real startup path into a no-bind/no-mutation validation phase rather than adding a parallel schema-only checker or invoking full startup blindly.
