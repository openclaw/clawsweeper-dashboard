## Summary

Keep this issue open. Current main has adjacent config checks, restart-safe active-work preflight, update hardening, and the cited context-engine regression fix, but it still lacks the requested first-class `openclaw gateway preflight [--json]` command and shared no-bind/no-mutation startup validation path.

## What This Changes

Add `openclaw gateway preflight [--json]` to validate config, plugin/context-engine startup, and port bindability without accepting traffic or mutating session state.

## Best Possible Solution

Define a shared side-effect-aware Gateway startup-validation routine, expose it as `openclaw gateway preflight [--json]`, document the operator contract, then compose safe update flows on top after the primitive is stable.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new Gateway CLI/startup-validation feature request. Source and docs inspection give high confidence that current main still lacks the requested standalone no-bind preflight surface.

## Solution Assessment

Unclear until maintainers choose the no-mutation startup boundary and JSON/reporting contract. The requested command is plausible, but the maintainable solution should split real Gateway startup into a reusable validation phase rather than adding a parallel schema-only checker.
