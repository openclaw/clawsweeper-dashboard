## Summary

Keep this issue open. Current main still lacks a first-class `openclaw gateway preflight [--json]` command or shared no-bind startup validation path; the existing preflight-like checks cover config/auth only, while real Gateway startup still proceeds into plugin/session startup and listener binding.

## What This Changes

Request a Gateway preflight command that validates config, plugin/context-engine startup, and port availability without binding the port or mutating session state.

## Best Possible Solution

Add a shared Gateway startup-validation routine with explicit no-bind and no-session-mutation modes, expose it as `openclaw gateway preflight [--json]`, document it, and compose safe update flows on top only after that primitive is stable.

## Reproduction Assessment

Yes. Source inspection on current main is enough: there is no registered or documented `gateway preflight` command, and the foreground path still calls `startGatewayServer()` followed by `startListening()`.

## Solution Assessment

Unclear because there is no patch yet. The requested command is the right product surface, but the maintainable implementation should split the real startup path into a side-effect-aware validation phase rather than adding a parallel schema-only checker or blindly invoking full startup.
