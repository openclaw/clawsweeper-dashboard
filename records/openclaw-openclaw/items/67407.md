## Summary

Keep this open. Current main still lacks a first-class `openclaw gateway preflight [--json]` surface or a shared no-bind startup validation routine; the existing service preflight is config/future-version only, and the related lossless-claw context-engine regression is fixed separately without implementing this requested Gateway preflight.

## What This Changes

Add a standalone Gateway preflight command that validates config, plugin/context-engine startup resolution, and port bindability without binding the listener or mutating session state.

## Best Possible Solution

Add `openclaw gateway preflight [--json]` as a Gateway CLI command backed by a shared startup-validation routine with explicit no-bind and no-session-mutation modes, then document it on the Gateway CLI page and compose safe update flows on top of that validated primitive later.

## Reproduction Assessment

Yes. The absence is reproducible by source inspection on origin/main: no preflight command is registered or documented, and the foreground Gateway path still calls `startGatewayServer()` followed by listener binding.

## Solution Assessment

Unclear because no patch exists yet. The requested direction is sound, but the best implementation is a shared, side-effect-aware startup validation path rather than a parallel schema-only validator or a blind dry run of full startup.
