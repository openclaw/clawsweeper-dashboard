## Summary

Keep this open. Current main has adjacent config/startup safeguards and the cited lossless-claw regression is fixed separately, but it still lacks the requested first-class `openclaw gateway preflight [--json]` command and shared no-bind/no-mutation Gateway startup validation path; the related open scanner/framework issues are broader coordination points, not replacements.

## What This Changes

The issue requests a Gateway preflight command that validates config, context-engine/plugin startup paths, and port availability without binding the Gateway port or mutating session state.

## Best Possible Solution

Add `openclaw gateway preflight [--json]` backed by a shared Gateway startup-validation routine that parses and validates config, resolves auth/SecretRefs, exercises plugin/context-engine registration in an explicit no-mutation mode, probes bindability without serving traffic, emits redacted stable results, and documents the operator contract before composing it into update flows.

## Reproduction Assessment

Not applicable: this is a Gateway CLI/startup-validation feature request, not broken existing behavior on current main. Source, docs, live issue, and PR searches give high confidence that the standalone no-bind command is absent.

## Solution Assessment

Unclear until maintainers choose the no-mutation startup boundary, JSON schema, diagnostics, and redaction contract. The requested command is plausible, but the maintainable solution should split real Gateway startup into a reusable validation phase rather than adding a parallel schema-only checker.
