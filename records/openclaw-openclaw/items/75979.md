## Summary

Keep open for maintainer/security review. The bug is valid and the propagation direction is reasonable, but the patch exposes the elevated snapshot as a caller-supplied Gateway `agent` RPC field, which can bypass the intended elevated allowlist/approval boundary if not restricted to the internal follow-up path.

## What This Changes

The PR threads `ExecElevatedDefaults` through exec approval follow-up targets, the Gateway `agent` schema/RPC, agent command options, and `runAgentAttempt`, with focused follow-up forwarding tests.

## Best Possible Solution

Keep the narrow elevated-default propagation, but make the snapshot internal-only to exec approval follow-up calls or revalidate it server-side against live policy before forwarding; preserve per-command approvals, add negative coverage for caller-supplied snapshots, and add the changelog entry.

## Reproduction Assessment

Yes. The linked issue gives a concrete Telegram allow-once flow, and current source shows the follow-up run enters embedded execution without `bashElevated`, causing the next elevated exec to fail at the defaults-based gate before approval registration.

## Solution Assessment

No. Passing an availability snapshot is a plausible fix boundary, but accepting it as an unguarded public `AgentParamsSchema` field is not the safest way to solve the issue; the value should be internal-only or validated from live sender/config state.
