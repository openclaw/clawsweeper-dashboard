## Summary

Keep this PR open: it fixes a real systemd stale-drop-in failure and the prior overmatch review finding was addressed, but the current strip logic can still leave a bare `ExecStart=` reset behind when comments or blank lines separate the reset from the stale OpenClaw command, making the service invalid after staging.

## What This Changes

The PR removes stale OpenClaw gateway/node `ExecStart` overrides from same-unit systemd drop-ins after staging and adds focused systemd tests for stale override cleanup and non-service preservation.

## Best Possible Solution

Keep the drop-in cleanup, but remove the reset and managed OpenClaw service command as a semantic override group that tolerates comments and blank lines while preserving unrelated settings and non-target `ExecStart` commands.

## Reproduction Assessment

Yes from source and supplied live output: current main rewrites only the main unit, systemd drop-ins are parsed after it, and the PR body shows a stale drop-in launching `2026.5.12` after a `2026.5.18` update. I did not run local tests because this was a read-only review.

## Solution Assessment

No, not yet. Cleaning stale managed drop-ins is the right direction and the first classifier bug was fixed, but the strip logic still needs to preserve correctness when reset/comment/command groups are not adjacent.
