## Summary

Keep open: the reported CLI-backed cron regression is real and the proof is useful, but this patch fixes it at the wrong layer by converting every CLI runtime `toolsAllow` request into a full-tool run, including explicit restrictions that current main intentionally fails closed.

## What This Changes

The PR changes CLI runner preparation to warn and clear any runtime `toolsAllow` for CLI backends instead of throwing, and updates the unit test to expect success.

## Best Possible Solution

Preserve the explicit-policy fail-closed guard in `prepareCliRunContext` and repair only auto-default or provably marker-lost cron `toolsAllow` values before CLI dispatch.

## Reproduction Assessment

Yes at source level. Current main rejects any defined CLI runtime `toolsAllow`, and the cron caller already shows which auto-default values should be dropped versus which explicit restrictions should reach that guard.

## Solution Assessment

No. Clearing `toolsAllow` in `prepareCliRunContext` is broader than the reported cron-default regression; the maintainable fix is to repair the upstream default-marker gap while keeping explicit restrictions fail-closed.
