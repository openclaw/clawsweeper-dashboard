## Summary

Keep open. Current main now has built-in code-safety scanning for installed skills/plugins, but that implementation is still code-only: it scans JS/TS-family files, explicitly excludes markdown, and the deep security audit for installed skills delegates to that same scanner. The requested `SKILL.md`-aware correlation is therefore not implemented on main.

## Evidence

- **Built-in scanner is code-only:** `skill-scanner` only marks `.js`, `.ts`, `.mjs`, `.cjs`, `.mts`, `.cts`, `.jsx`, and `.tsx` as scannable, and its built-in rules look for code patterns like `fetch`, `readFile`, and `process.env`; there is no markdown or `SKILL.md` parsing path.

## Risks / Open Questions

- Closing this would drop a still-valid hardening request: the current implementation can catch `process.env` in code, but not intent split across `SKILL.md` plus otherwise-benign code.
- Any future fix needs to define scope carefully across at least three surfaces: gateway-backed install scanning, `openclaw security audit --deep`, and the separate `openclaw skills install <slug>` ClawHub download path.
