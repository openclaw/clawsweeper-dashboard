## Summary

Keep open: the diagnostic feature is useful and has real-output proof, but the latest PR head still reports diagnostics from a different session-store point than the maintenance decision it claims to explain, and it adds unused diagnostic work to non-JSON maintenance output.

## What This Changes

The PR adds stale-running task decision/reason diagnostics to `openclaw tasks maintenance --json`, including retained/reconcile reasons and durable cron recovery filtering tests.

## Best Possible Solution

Keep the JSON diagnostic shape, but compute it only for JSON output and against the same task/session state used by the task-maintenance decision, with regression coverage for stale session pruning in preview or apply mode.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR computes diagnostics after session maintenance and before the JSON branch, while session maintenance mutates the store in apply mode and prunes only a clone in preview mode. I did not run tests because this review is read-only.

## Solution Assessment

No as written; the additive diagnostic surface is reasonable, but the implementation should be lazy for JSON output and tied to the same session-store snapshot that task maintenance used.
