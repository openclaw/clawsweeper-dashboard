## What This Changes

The branch threads optional `toolCallId` through Control UI stream segment types, adjusts chat item sorting to keep a matching live tool card after its stream segment, adds a focused regression test, and removes an unused Telegram rich-message parameter.

## Best Possible Solution

Keep the renderer-boundary approach, replace the fractional timestamp adjustment with deterministic causal ordering for stream/tool pairs, and land only after redacted real WebChat/Gateway proof.

## Reproduction Assessment

Yes at source level, but not by a live browser run here: current main snapshots stream segments with `Date.now()` and then sorts those segments against tool event timestamps, so an older tool timestamp can move the tool before its causal assistant text.

## Solution Assessment

No as written: `buildChatItems` is the right layer to consume the existing `toolCallId` signal, but the fractional timestamp bump should be replaced with deterministic causal ordering and proven in real WebChat.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve causal order when segment timestamps collide:** `ui/src/ui/chat/build-chat-items.ts:388`
  - body: The PR moves a matched tool to `predecessorTimestamp + 0.001`. When two stream segments share the same millisecond timestamp, both matched tools can end up after all same-timestamp streams, so a turn can render stream1, stream2, tool1, tool2 instead of stream1, tool1, stream2, tool2. Please sort matched stream/tool pairs by an explicit causal order and add a two-segment same-timestamp regression.
  - confidence: 0.84
