## Summary

Keep this PR open, but it is not merge-ready: the change is in the right WebChat renderer path, yet the timestamp adjustment can still misorder same-millisecond multi-tool segments and the contributor proof is unit-test-only.

## What This Changes

The PR adds optional tool-call metadata to WebChat stream segments and uses it while sorting chat items so a matching live tool card stays after the stream segment that introduced it.

## Best Possible Solution

Keep the renderer-boundary fix, but make the causal stream/tool ordering deterministic for same-timestamp multi-tool turns and require real WebChat proof before landing.

## Reproduction Assessment

Yes at source level: current main uses `Date.now()` for stream segment timestamps and then sorts rendered items by timestamp, so an older tool event timestamp can move the tool card ahead of its causal assistant text. I did not run a live browser/Gateway reproduction.

## Solution Assessment

No as written: consuming `toolCallId` in `buildChatItems` is the right layer, but the fractional timestamp bump should be replaced or supplemented with deterministic causal tie ordering for multi-segment same-millisecond cases.

Full review comments:

- **[P2] Preserve causal order when segment timestamps collide:** `ui/src/ui/chat/build-chat-items.ts:388`
  - body: The new adjustment moves every matched tool to `predecessorTimestamp + 0.001`. Because the producer records segment `ts` with millisecond `Date.now()`, two live stream segments can share the same timestamp; then the later stream segment still sorts before the adjusted first tool, leaving a multi-tool turn ordered like stream1, stream2, tool1, tool2. Please make same-timestamp stream/tool pairs preserve their insertion/causal order and add a regression with two matching segments that share a timestamp.
