- `impact:session-state`: Dreaming can persist low-value material into `MEMORY.md`, altering the long-term context and state supplied to future agent runs.

## Summary

Current main still permits dreaming-only daily recurrence to satisfy `minUniqueQueries`, and no related issue or PR owns a fix. Focused tests and dreaming docs intentionally describe query/day context diversity while the CLI promises distinct queries, so this needs a memory-core owner decision before implementation.

## What This Changes

Require `minUniqueQueries` to count genuinely distinct recall queries rather than allowing recall-day diversity to satisfy the deep-promotion gate.

## Best Possible Solution

Select an explicit promotion policy, then make the setting name and help text match it and update ranking, apply-time revalidation, explain output, and regression tests together; if genuine recall is required, keep day recurrence only in the separate consolidation score.

## Reproduction Assessment

Yes, at source level: synthetic daily signals feed recall-day diversity, both promotion gates accept that diversity, and a focused current test expects promotion with zero recalls. A live nine-agent fleet reproduction was not needed to establish the narrow gate behavior.

## Solution Assessment

Unclear. The suggested expression change is the narrowest implementation of the CLI’s distinct-query wording, but current tests and dreaming documentation encode broader query/day context diversity, so an owner must first choose the intended product contract.
