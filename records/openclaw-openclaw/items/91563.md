## Summary

Keep open: current main and the latest release still let recall-day diversity satisfy `minUniqueQueries` in both ranking and apply, while CLI/docs describe the option as a distinct query count and tests intentionally preserve daily-only promotion behavior.

## What This Changes

The issue asks for memory-core Dreaming deep promotion to count genuinely distinct recall queries for `minUniqueQueries` instead of using recall-day diversity as a fallback.

## Best Possible Solution

Maintainers should decide whether `minUniqueQueries` means organic distinct query hashes or broader cross-day context diversity, then align rank/apply gates, CLI/docs, and regression tests around that contract.

## Reproduction Assessment

Yes, at source level: current main stores query hashes and recall days separately, then lets recall-day length satisfy the `minUniqueQueries` gate in both rank and apply. I did not replay the full fleet workload from the report.

## Solution Assessment

Unclear. The suggested code change is narrow, but it conflicts with current comments and tests that intentionally let repeated dreaming-only daily signals clear default promotion gates, so maintainers should settle the policy first.
