## Summary

The PR addresses a real Gemini CLI selected-profile bug at the right private backend boundary, but it is not merge-ready because the refresh path can replace the raw OAuth access token with OpenClaw's formatted Google OAuth API-key payload before writing Gemini CLI credentials. It also leaves a smaller cleanup gap for partially-created temporary Gemini homes and needs maintainer auth/security compatibility signoff for the fail-closed provider/env behavior.

## What This Changes

The PR adds a private `google-gemini-cli` auth bridge that forwards selected OpenClaw OAuth/API-key profiles into an isolated Gemini CLI home, clears conflicting auth environment, rejects Vercel Gateway profiles on this backend, and adds regression tests.

## Best Possible Solution

Keep the private backend bridge, but preserve raw refreshed OAuth material for Gemini CLI, clean up failed temp-home setup, then rerun redacted live OAuth proof and get maintainer auth/security compatibility signoff.

## Reproduction Assessment

Yes, source inspection shows current `main` does not forward selected OpenClaw auth profiles into the Gemini CLI subprocess even though the backend hook exists for private execution preparation.

## Solution Assessment

No, the private backend bridge is the right ownership boundary, but this implementation is not yet the best fix because it can feed formatted OpenClaw OAuth output into Gemini CLI's raw token field.
