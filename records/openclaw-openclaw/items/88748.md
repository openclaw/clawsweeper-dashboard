## Summary

Keep open: this is a meaningful fix for a source-reproducible Gemini CLI auth bug, but it is not merge-ready because it expands the public CLI backend SDK to raw OAuth credential forwarding without a documented/approved contract and has only test/source proof, not a live Gemini OAuth subprocess run.

## What This Changes

The PR adds opt-in CLI backend auth-profile forwarding, refreshes selected Gemini OAuth credentials, writes them to an isolated `GEMINI_CLI_HOME` for the Gemini subprocess, and adds focused tests.

## Best Possible Solution

Land only after the credential bridge is documented or narrowed to an approved boundary, temp-home cleanup is hardened, and redacted live Gemini OAuth subprocess proof confirms the selected profile drives execution.

## Reproduction Assessment

Yes, source-reproducible: current main drops auth-profile forwarding for CLI providers and the Gemini CLI backend has no `prepareExecution` bridge, matching the linked report that selected OpenClaw profiles do not control the external runtime.

## Solution Assessment

No: the bridge shape is plausible and matches Gemini CLI credential paths, but the public raw-token SDK exposure and absent live OAuth proof make this not the best merge-ready form yet.
