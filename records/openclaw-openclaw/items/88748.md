## Summary

Keep open: the branch fixes part of Gemini CLI profile selection and credential materialization, but it still does not force Gemini CLI to use OAuth for non-interactive runs, broadens the public CLI backend SDK to carry raw OAuth token material, and drops provider-auth alias inputs from one runtime auth-plan call.

## What This Changes

The PR forwards selected `google-gemini-cli` OAuth profiles into CLI execution, refreshes the selected profile, and writes Gemini-compatible OAuth credentials into an isolated `GEMINI_CLI_HOME` before spawning `gemini`.

## Best Possible Solution

Keep the plugin-owned Gemini credential materialization idea, but set Gemini's OAuth auth selection explicitly, avoid or explicitly approve the public raw-token SDK expansion, restore alias-plan inputs, and require an authenticated redacted `gemini --prompt` proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main does not forward a selected `google-gemini-cli:*` profile into the Gemini CLI backend and the existing backend has no credential-home bridge. I did not run a live failing Gemini prompt in this review, and the PR's current proof still does not show an authenticated prompt consuming the isolated profile.

## Solution Assessment

No, not yet: the PR is a plausible owner-boundary direction, but it misses Gemini's auth-type contract and expands the SDK security boundary too broadly. A safer fix would keep the bridge narrow, restore alias inputs, and prove a real authenticated Gemini CLI request after the patch.

Full review comments:

- **[P1] Set Gemini's OAuth auth type before spawning:** `extensions/google/cli-backend.ts:98-101`
  - body: The bridge writes `.gemini/oauth_creds.json` and sets `GEMINI_CLI_HOME`, but Gemini CLI 0.40.1 only chooses personal OAuth in non-interactive mode when settings select `oauth-personal` or `GOOGLE_GENAI_USE_GCA=true`. With a clean isolated home, `gemini --prompt` can exit before reading this credential file, and the supplied proof only ran `gemini --version`; set the auth type and prove an authenticated prompt path.
  - confidence: 0.94
- **[P1] Keep raw OAuth material out of the public CLI SDK:** `src/plugins/cli-backend.types.ts:29-42`
  - body: This exports an `authCredential` shape containing access, refresh, and ID token fields through `openclaw/plugin-sdk/cli-backend`, so any CLI backend plugin that opts in can receive raw OAuth token material. That is a new public security boundary; keep the bridge internal/narrow or land it only with explicit maintainer security approval plus the matching SDK docs/baseline/tests.
  - confidence: 0.88
- **[P2] Preserve provider alias inputs in the runtime auth plan:** `src/agents/command/attempt-execution.ts:536-546`
  - body: This second `buildAgentRuntimeAuthPlan` call no longer receives `params.metadataSnapshot` or `providerAuthAliasesEnabled`, while the resolver uses those inputs to decide provider/auth alias lookup. Non-CLI runs can now compute auth forwarding from a different plugin snapshot or alias-enable state than the selection path above; pass the same inputs through both calls.
  - confidence: 0.82
