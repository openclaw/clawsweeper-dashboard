## Summary

This PR still contains useful Gemini CLI OAuth bridge work, but it is not merge-ready: the latest head leaves one upstream Gemini credential-source env bypass and adds an unauthorized root pnpm override.

## What This Changes

The branch adds a private google-gemini-cli OAuth bridge that forwards selected OpenClaw OAuth profiles into an isolated GEMINI_CLI_HOME, updates CLI auth profile selection, adds regression tests, and adds a root pnpm override for esbuild.

## Best Possible Solution

Land the private Gemini auth bridge only after clearing all upstream Gemini credential-source env that can override staged OAuth and removing or separately authorizing the dependency override.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source inspection: current main and v2026.6.6 lack the Gemini CLI prepareExecution bridge, and upstream Gemini CLI can ignore staged oauth_creds.json when GEMINI_FORCE_ENCRYPTED_FILE_STORAGE is inherited.

## Solution Assessment

Is this the best way to solve the issue? No, not yet. The private owner-local prepareExecution bridge is the right layer, but it must clear the encrypted-storage env bypass and drop or explicitly authorize the unrelated dependency override before merge.
