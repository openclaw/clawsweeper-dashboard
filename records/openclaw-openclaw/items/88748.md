## Best Possible Solution

Isolate the exact Gemini CLI OAuth store used by 0.40.1, preserve existing auth alias snapshot inputs, and land token forwarding only behind a documented maintainer-approved SDK/security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Force Gemini OAuth storage into the isolated home:** `extensions/google/cli-backend.ts:100`
  - body: `GEMINI_CLI_HOME` is not the OAuth store used by Gemini CLI 0.40.1: the dependency loads OAuth from hybrid keychain storage and migrates legacy `oauth_creds.json` from `homedir()/.gemini`. With only this env var, an auth-consuming run can still use ambient credentials or migrate the selected refresh token into the global Gemini keychain. The bridge needs to isolate the dependency's actual storage path, for example by controlling HOME and forcing file storage, then prove an auth-consuming command uses it.
  - confidence: 0.95
- **[P1] Keep auth alias inputs on the runtime plan:** `src/agents/command/attempt-execution.ts:536-546`
  - body: This call used to receive the same `metadataSnapshot` and `providerAuthAliasesEnabled` inputs as the harness selection above it. Dropping them means the runtime auth plan can reload provider aliases or ignore a disabled-plugin state, so embedded-provider auth can be resolved against a different alias set than the rest of the attempt.
  - confidence: 0.88
- **[P1] Settle the token-forwarding SDK contract first:** `src/plugins/cli-backend.types.ts:29-42`
  - body: This public plugin SDK type exposes access, refresh, and ID token fields to any CLI backend that opts into credential forwarding, but the PR does not document the trust boundary, redaction requirements, or whether this should be public rather than bundled-Google-only. That is a security-sensitive plugin API expansion and should be narrowed or explicitly approved before merge.
  - confidence: 0.86
