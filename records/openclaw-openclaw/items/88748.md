- `P1`: The bug can make Gemini runs use a different account or stale credential source than the OpenClaw-selected profile indicates.

- `merge-risk: 🚨 security-boundary`: The bridge materializes access and refresh tokens into a temporary filesystem credential file for an external process.

## Summary

The PR fixes a source-reproducible Gemini account-selection bug with a private, owner-scoped credential bridge and convincing authenticated proof; no concrete patch defect remains, but the intentional credential-source change and token-handling boundary warrant explicit maintainer approval before merge.

## What This Changes

The PR selects and refreshes the applicable OpenClaw Gemini OAuth profile, materializes it in an isolated Gemini CLI home for execution, and makes auth epochs profile-owned while that bridge is active.

## Best Possible Solution

Merge the private Google-backend bridge after maintainers explicitly approve and document that selected OpenClaw profiles govern Gemini CLI execution, while preserving ambient Gemini behavior when no profile is selected and keeping focused isolation, cleanup, and refresh tests.

## Reproduction Assessment

Yes. Current source shows selected OpenClaw profile state and ambient Gemini CLI credentials are separate authorities, and the PR supplies an authenticated after-fix run through the bridge.

## Solution Assessment

Yes technically: the private backend preparation hook is narrower and safer than exposing a general credential-forwarding Plugin SDK API, and it matches Gemini CLI's upstream credential contract; maintainers still need to approve the intentional account-source policy.

## Risks / Open Questions

- For runs with a selected `google-gemini-cli` profile, merging intentionally changes the credential authority from ambient `~/.gemini` state to the selected OpenClaw profile; maintainers should approve that upgrade-visible account-source behavior.
- The bridge copies access and refresh tokens into a temporary file for the subprocess. The implementation uses restrictive permissions and cleanup, but this remains an auth-provider security boundary that ordinary green CI cannot settle.
- Gemini CLI may refresh its temporary credential file during a run, but that subprocess-side copy is discarded during cleanup; OpenClaw instead refreshes and persists its profile before each run.
