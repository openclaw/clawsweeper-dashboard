## Summary

Keep open: the private Gemini CLI bridge is the right layer and the PR has strong live proof, but PR head still leaves an upstream Gemini encrypted-storage mode that can ignore the staged credential file.

## What This Changes

The PR forwards selected `google-gemini-cli` OAuth profiles into the Gemini CLI subprocess by refreshing the profile and preparing an isolated `GEMINI_CLI_HOME` with Gemini-compatible OAuth files.

## Best Possible Solution

Keep the private Gemini `prepareExecution` bridge, but clear `GEMINI_FORCE_ENCRYPTED_FILE_STORAGE`, cover it in the auth bridge test, and rerun focused validation before merge.

## Reproduction Assessment

Yes. Source inspection of PR head plus upstream Gemini CLI v0.46.0 shows `GEMINI_FORCE_ENCRYPTED_FILE_STORAGE=true` makes Gemini load `OAuthCredentialStorage` before the staged `oauth_creds.json`.

## Solution Assessment

No, not yet. The owner-local private bridge is the right layer, but it must also clear the upstream encrypted-storage switch before the staged profile is authoritative.
