## Summary

## What This Changes

The PR forwards selected `google-gemini-cli` OAuth profiles into CLI runs, refreshes/loads credentials during CLI preparation, writes an isolated `GEMINI_CLI_HOME/.gemini/oauth_creds.json`, and adds CLI backend auth-forwarding SDK fields.

## Best Possible Solution

Land a Gemini-owned OAuth bridge only after credential forwarding is explicitly gated, documented, covered by SDK/contract tests, and proven with a redacted authenticated Gemini CLI prompt/request.

Yes, from source inspection: current main has no Gemini CLI `prepareExecution` bridge and the CLI run path blocks normal auth-profile forwarding for CLI providers. I did not run a live authenticated Gemini repro in this read-only review.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Gate raw auth credentials before calling prepareExecution:** `src/agents/cli-runner/prepare.ts:331`
  - body: `prepareExecution` is a public plugin hook, but this now receives the selected `authCredential` for any backend with an effective profile, even when that backend did not opt into credential forwarding. That can expose OAuth access/refresh/id token material to existing or third-party CLI hooks; only resolve and pass credentials after an explicit, documented opt-in boundary is satisfied.
  - confidence: 0.91
- **[P2] Make the auth-forwarding flag match the public SDK contract:** `src/agents/command/attempt-execution.ts:214-222`
  - body: `acceptsAuthProfileForwarding` is exported as a generic CLI backend capability, but this gate returns false before reading backend config unless the provider is exactly `google-gemini-cli`. That makes the new public flag silently unusable for other CLI backends; either keep the bridge Gemini-internal or honor the generic opt-in with docs and tests.
  - confidence: 0.86
