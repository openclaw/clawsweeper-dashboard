## Label Justifications

- `P1`: The affected workflow can authenticate successfully in OpenClaw yet execute with an expired or different ambient Gemini account.
- `merge-risk: 🚨 compatibility`: Selected-profile runs stop relying on ambient Gemini CLI state and may require users with incomplete profiles to re-authenticate.

## Summary

This PR is still necessary and appears functionally correct: current main keeps OpenClaw’s selected Gemini OAuth profile separate from the external CLI’s ambient credentials, while the branch adds a private, provider-specific bridge with canonical refresh, restrictive temporary storage, cleanup, focused tests, and authenticated live proof. Maintainer review remains appropriate because merging intentionally stops ambient-credential fallback for bridged runs and moves refresh tokens across a subprocess boundary.

## What This Changes

The PR refreshes a selected `google-gemini-cli` OAuth profile, materializes it in a private per-run Gemini CLI home, and makes Gemini subprocess authentication profile-owned rather than ambient.

## Best Possible Solution

Keep the bridge private to the bundled Google backend, use the canonical refreshed OpenClaw profile as the sole auth owner, preserve restrictive per-run credential storage and unconditional cleanup, and explicitly approve the intentional removal of ambient Gemini credential fallback before landing through the intended release path.

## Reproduction Assessment

Yes at source level. Current main’s selected OpenClaw profile and the external Gemini CLI’s ambient credential store are independent, and the linked report describes the resulting wrong-account or expiry mismatch; this review did not execute the failing current-main scenario itself.

## Solution Assessment

Yes, subject to maintainer approval of the compatibility change. A private bundled-backend preparation hook using the canonical profile refresh and existing cleanup contract is narrower and more maintainable than exposing a generic secret-forwarding Plugin SDK API.
