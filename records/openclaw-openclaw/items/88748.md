## Summary

## What This Changes

## Best Possible Solution

Keep the private Google backend bridge, fix OAuth staging to preserve raw refreshed credential fields, add full-path regression coverage, then require refreshed redacted live Gemini CLI proof plus maintainer auth/security signoff before merge.

## Reproduction Assessment

Yes. Current main and v2026.6.6 have a static `google-gemini-cli` backend without `prepareExecution`, while Gemini CLI reads its own home/auth environment, so the selected OpenClaw OAuth profile is source-reproducible as ignored.

## Solution Assessment

No, not as submitted. The private Google-owned bridge is the right layer, but the full prepare path can replace raw OAuth access material with provider-formatted API-key JSON before writing Gemini CLI's `oauth_creds.json`.

Full review comments:

- **[P1] Forward raw OAuth material to Gemini CLI:** `src/agents/cli-runner/prepare.ts:336`
  - body: For `google-gemini-cli` OAuth profiles, `resolveApiKeyForProfile` returns the provider-formatted API-key string; Google OAuth formats that as JSON, so this branch overwrites `authCredential.access` before the backend writes `oauth_creds.json`. A normal selected OAuth profile can therefore hand Gemini CLI JSON as `access_token`, breaking the bridge this PR adds.
  - confidence: 0.94

  - body: `createIsolatedGeminiCliHome` creates and chmods `tempHome` before `.gemini/settings.json` is written, but errors in `chmod`, `mkdir`, or `writeFile` escape before the callers' cleanup handlers know the path. Wrap the helper setup in cleanup-on-error so failed preparation does not leave stale temp homes.
  - confidence: 0.79
