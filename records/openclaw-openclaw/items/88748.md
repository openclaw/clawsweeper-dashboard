## Summary

Keep open: this targets a real Gemini OAuth runtime bug, but current head is not merge-ready because the isolated Gemini home lacks the auth-type signal Gemini requires for non-interactive runs, the SDK change exposes raw OAuth token material, and the proof still stops short of an authenticated Gemini request.

## What This Changes

The PR forwards selected OpenClaw OAuth profiles into the `google-gemini-cli` CLI backend, refreshes/loads credentials, and writes Gemini-compatible OAuth credentials into an isolated CLI home before spawning `gemini`.

## Best Possible Solution

Land a narrower Gemini-owned bridge that sets the auth method Gemini CLI requires, preserves existing provider-alias routing inputs, avoids or explicitly approves public raw-token SDK exposure, and includes redacted authenticated Gemini CLI proof.

## Reproduction Assessment

Yes from source inspection: current main does not forward selected `google-gemini-cli` OAuth profiles into the spawned CLI, and this PR's new bridge can be checked against Gemini CLI 0.40.1's non-interactive auth path. I did not run live Google OAuth in this read-only review.

## Solution Assessment

No. The bridge is a plausible direction, but it is not the best mergeable shape until it selects Gemini's OAuth auth mode, keeps auth alias planning stable, and narrows or explicitly approves the raw-token SDK contract.
