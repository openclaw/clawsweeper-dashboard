## Summary

Keep open: the branch targets a real Gemini CLI auth-profile gap, but it still exposes a raw credential-forwarding Plugin SDK contract without docs/API/security approval, only honors the new generic forwarding flag for Gemini, and the supplied proof does not show an authenticated Gemini request using the selected OpenClaw profile.

## What This Changes

The branch forwards selected `google-gemini-cli:*` OAuth profiles into Gemini CLI preparation, writes an isolated per-run `GEMINI_CLI_HOME/.gemini/oauth_creds.json`, and extends CLI backend plugin types and tests for that bridge.

## Best Possible Solution

Land this only after maintainers choose the SDK boundary: either keep raw credential forwarding private to the bundled Gemini bridge, or publish a documented/tested Plugin SDK contract with security guidance, then require redacted proof of an authenticated Gemini request using the selected profile.

## Reproduction Assessment

Yes at source level: current main blocks CLI auth-profile forwarding and the Gemini CLI backend has no auth bridge, so selected OpenClaw profiles cannot control the subprocess path. I did not run a live failing Gemini prompt against current main.

## Solution Assessment

No as-is: the plugin-owned Gemini bridge is the right ownership direction, but the public raw credential SDK contract, Gemini-only capability gate, and missing authenticated proof need resolution before this is the best fix.
