## Summary

Keep open. Current main and v2026.6.6 still lack the Gemini CLI selected-profile bridge, but this PR is not merge-ready because the current head can still write provider-formatted Google OAuth JSON into Gemini CLI's `access_token` field despite the useful redacted live proof.

## What This Changes

The PR adds a private `google-gemini-cli` auth bridge that stages selected OpenClaw OAuth or API-key profiles into an isolated Gemini CLI home and routes canonical Google CLI runs to a selected CLI auth profile.

## Best Possible Solution

Keep the private Google-owned bridge, preserve raw refreshed OAuth fields for Gemini CLI staging, add full-path regression coverage, then refresh redacted live Gemini CLI proof and get maintainer auth/security signoff.

## Reproduction Assessment

Yes. Current main and v2026.6.6 have a static `google-gemini-cli` backend without `prepareExecution`, while Gemini CLI reads its own home/auth environment, so the selected OpenClaw OAuth profile is source-reproducible as ignored.

## Solution Assessment

No, not as submitted. The private Google-owned bridge is the right layer, but the full prepare path can replace raw OAuth access material with provider-formatted API-key JSON before writing Gemini CLI's `oauth_creds.json`.

- **[P1] Forward raw OAuth material to Gemini CLI:** `src/agents/cli-runner/prepare.ts:336`
  - body: For `google-gemini-cli` OAuth profiles, `resolveApiKeyForProfile` returns the provider-formatted API-key string; Google OAuth formats that as JSON, so this branch overwrites `authCredential.access` before the backend writes `oauth_creds.json`. A normal selected OAuth profile can therefore hand Gemini CLI JSON as `access_token`, breaking the bridge this PR adds.
  - confidence: 0.94

Wrap this helper setup in cleanup-on-error or move cleanup ownership into the helper.
  - confidence: 0.78
