## Summary

Keep open: current head is not merge-ready because the new CLI auth-profile forwarding gate fails prod typechecking, and the OAuth bridge still does not force Gemini CLI away from native keychain credentials, so the supplied proof does not cover the highest-risk ambient-auth path.

## What This Changes

The PR adds a private Gemini CLI auth bridge that stages selected OpenClaw OAuth/API-key profiles into an isolated Gemini CLI home and adjusts CLI auth-profile selection for canonical Google models routed through `google-gemini-cli`.

## Best Possible Solution

Keep the private Google backend bridge, but carry the forwarding capability through the resolved backend contract, set it for `google-gemini-cli`, force upstream file-backed credential storage for staged OAuth, and refresh live proof before merge.

## Reproduction Assessment

Yes. Current main lacks the Gemini CLI `prepareExecution` bridge while upstream Gemini CLI reads its own home/keychain state, and the PR head has a source-level type failure in the new forwarding gate.

## Solution Assessment

No, not yet. A private Google backend bridge is the right layer, but this implementation needs the resolved capability contract fixed and native-keychain isolation proven before it is the best fix.
