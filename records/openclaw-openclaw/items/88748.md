## Label Justifications

## Summary

Keep open: the PR targets a real Gemini CLI OAuth gap, but this head still publishes a raw auth-profile forwarding SDK surface without a documented/baselined security contract, the public opt-in is only honored for Google, and the supplied proof does not exercise an authenticated Gemini request.

## What This Changes

The branch forwards selected OpenClaw `google-gemini-cli:*` OAuth profiles into Gemini CLI backend preparation and writes a per-run `GEMINI_CLI_HOME/.gemini/oauth_creds.json`, while adding CLI backend auth-forwarding fields to the plugin SDK.

## Best Possible Solution

Keep the Gemini-owned bridge, but either make the raw credential handoff private to the bundled Google backend or publish a narrow documented/baselined SDK contract with sanitized credential shape, generic opt-in behavior, focused tests, and authenticated Gemini CLI proof.

## Reproduction Assessment

Yes for the source-level bug: current `main` can select/load an auth profile but does not pass credential material to `prepareExecution`, while the Google backend has no bridge to prepare a Gemini credential home. I did not run a live authenticated Gemini request locally.

## Solution Assessment

No. The owner-local Gemini bridge is a plausible fix and the Gemini CLI source supports `GEMINI_CLI_HOME`, but the PR currently broadens the public SDK/security contract more than it proves and leaves the generic opt-in behavior inconsistent.

  - confidence: 0.9
