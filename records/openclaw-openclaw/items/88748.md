## Summary

Keep open: this is a real Gemini CLI auth-profile bug fix with useful live proof, but the current head is not merge-ready because it forwards raw auth credentials through the generic public CLI backend hook, can still stage provider-formatted OAuth material as Gemini CLI access_token, and carries a workspace dependency-policy change that needs explicit approval or removal.

## What This Changes

The branch adds a private Google Gemini CLI auth bridge that forwards selected OpenClaw auth profiles into an isolated Gemini CLI home, clears competing auth env, updates CLI auth selection tests, and changes pnpm workspace dependency policy.

## Best Possible Solution

Keep the Google-owned bridge, but restrict credential material to the bundled Gemini path, preserve raw OAuth fields for oauth_creds.json, clean failed temp-home setup, and remove or explicitly approve the workspace dependency-policy change before merge.

## Reproduction Assessment

Yes. The bug is source-reproducible: current main lacks a Gemini CLI prepareExecution bridge while upstream Gemini CLI reads auth from GEMINI_CLI_HOME or the user .gemini state, so selected OpenClaw profiles can diverge from subprocess auth.

## Solution Assessment

No for the current head. The owner-local Gemini bridge is the right layer, but this implementation must keep credentials out of the generic plugin hook, preserve raw OAuth material, and settle the dependency-policy delta first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep raw credentials out of the generic CLI hook:** `src/agents/cli-runner/prepare.ts:358-360`
  - body: This adds `authCredential` to the object passed to every CLI backend `prepareExecution` callback, even though the public context type does not expose credential material. Any backend with this hook can now inspect or log selected OAuth/API-key secrets; keep the secret-bearing object on a private bundled-Gemini path or otherwise gate it so third-party CLI backends never receive it.
  - confidence: 0.93
- **[P1] Preserve raw OAuth fields for Gemini CLI staging:** `src/agents/cli-runner/prepare.ts:239-244`
  - body: `resolveApiKeyForProfile` returns the provider runtime API-key string; for Google OAuth that can be the JSON payload from `formatGoogleOauthApiKey`, not the raw bearer token. Writing that value into `oauth_creds.json.access_token` can make the selected OAuth path fail, so keep the refreshed profile's raw `access`/`refresh`/`expires` fields for OAuth staging.
  - confidence: 0.9
- **[P1] Remove or explicitly approve the esbuild workspace override:** `pnpm-workspace.yaml:96`
  - body: This auth bridge PR still changes `pnpm-workspace.yaml` dependency policy by adding esbuild minimum-release-age exclusions and an `overrides.esbuild` pin. That is unrelated supply-chain surface for this fix unless maintainers explicitly approve the exact release-branch dependency delta.
  - confidence: 0.86
- **[P3] Clean up failed Gemini home setup:** `extensions/google/cli-backend.ts:147-157`
  - body: `createIsolatedGeminiCliHome` creates `tempHome` before chmod, mkdir, and settings writes, but failures in those setup steps occur before the caller's cleanup try/catch. Wrap setup after `mkdtemp` in a local try/catch that removes `tempHome` before rethrowing.
  - confidence: 0.82
