## Summary

Keep open: the PR addresses a real `/compact` UX bug, but the patch is not mergeable as written because it feeds context-window limits into `currentTokenCount`, which the compactor treats as observed live usage and can return or persist as `tokensBefore`.

## What This Changes

The PR changes manual `/compact` reply formatting, adds a JSONL file-size fallback for stale session totals, and forwards a token count into manual compaction.

## Best Possible Solution

Land a corrected `/compact` fix that keeps the label guard and stale-total fallback, omits `currentTokenCount` unless real live usage is available, preserves current-main context-budget resolution, and adds focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main still formats the arrow whenever both token counts are present and uses the stale stored total for the context summary, while the PR body includes copied Telegram before/after output for the small-session case.

## Solution Assessment

No, not as written. The label guard and stale-total fallback are the right direction, but deriving `currentTokenCount` from context-window metadata is not a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Stop passing context-window tokens as usage:** `src/auto-reply/reply/commands-compact.ts:128-134`
  - body: `observedContextTokens` is built from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are context-window budgets, then passed as `currentTokenCount`. The compactor treats that field as live prompt usage and can return or persist it as `tokensBefore`, so manual `/compact` can report/checkpoint a bogus before count instead of fixing misleading labels. Omit this field unless a real usage measurement is available.
