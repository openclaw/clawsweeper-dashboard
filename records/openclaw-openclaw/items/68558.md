## Summary

Keep open: the PR addresses a real `/compact` reply bug, but the current patch still passes a model context-window value as `currentTokenCount`, which the compactor contract defines as live prompt usage and then reuses in diagnostics/checkpoints.

## What This Changes

The branch changes `src/auto-reply/reply/commands-compact.ts` to guard the `before → after` compact label, estimate stale session totals from JSONL file size, and forward an observed token value to embedded compaction.

## Best Possible Solution

Land a corrected patch that keeps the label guard and stale-total fallback, omits `currentTokenCount` unless true live prompt usage is available, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still renders the unconditional `before → after` label path, and the PR body includes copied before/after Telegram output for the small-session case.

## Solution Assessment

No, not as currently patched. The owner module is right for the label and stale-total fixes, but forwarding `contextTokens` as `currentTokenCount` violates the compactor contract and should be removed unless real usage is available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass only live usage tokens to compaction:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` is derived from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are model context-window limits, but `currentTokenCount` is documented and consumed as caller-observed live prompt usage. Because the compactor returns and persists this value as `tokensBefore` and uses it in token-after estimation, manual `/compact` can now report a bogus before count near the model window and skew diagnostics; omit this field unless a real usage snapshot is available.
  - confidence: 0.94
