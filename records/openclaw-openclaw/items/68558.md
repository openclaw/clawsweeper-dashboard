## Summary

Keep this PR open: the formatter guard addresses a real `/compact` output bug that is still present on current `main` and v2026.6.8, but the branch is conflicting and its token-accounting changes can mix context-window budgets and rough file-size estimates into user-visible/session token state.

## What This Changes

The PR modifies `src/auto-reply/reply/commands-compact.ts` to suppress non-decreasing `/compact` before/after arrows, forward a derived `currentTokenCount`, and estimate context usage from session JSONL file size.

## Best Possible Solution

Port only the `before > after` formatter guard onto current `main`, add focused tests, keep `currentTokenCount` and `contextTokenBudget` separate, and handle stale usage estimation only through a separately proven canonical token source.

## Reproduction Assessment

Yes, at source level: current `main` and v2026.6.8 still render a `tokensBefore → tokensAfter` label whenever both values are non-null. A compact result like `tokensBefore: 0` and `tokensAfter: 36` therefore still has the reported misleading output path.

## Solution Assessment

No, not as submitted. The formatter clamp is the useful part, but the best fix is to port that guard onto current `main` without treating context-window budgets or rough file-size estimates as canonical usage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Pass live usage tokens, not context windows:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` is derived from `params.contextTokens` or persisted `contextTokens`, which are context-window budget values, but `currentTokenCount` is the live prompt-token usage contract. Passing a 272k/1M window here can make manual `/compact` report or checkpoint a huge `tokensBefore` value for a small session instead of actual session usage.
  - confidence: 0.93
- **[P2] Do not let file size override fresh token totals:** `src/auto-reply/reply/commands-compact.ts:224`
  - body: `Math.max(storedTotal, fileEstimate)` lets a rough JSONL bytes/4 estimate replace a fresh stored token total whenever transcript JSON is large, especially with tool results. Use this only when the stored total is missing or known stale, or keep the canonical stored/session result so the context summary does not overstate usage.
  - confidence: 0.86
