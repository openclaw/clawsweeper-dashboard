## Summary

Keep this PR open: current main and the latest release still have the misleading `/compact` label path, but this branch is not merge-ready because it mixes context-window budgets into usage-token accounting and adds a broad JSONL byte estimate fallback.

## What This Changes

The PR changes `src/auto-reply/reply/commands-compact.ts` to suppress non-decreasing compact arrows, forward a token count into manual compaction, and estimate context usage from the session JSONL file when stored totals look stale.

## Best Possible Solution

Port the non-decreasing label guard onto current main while keeping context budget and observed usage tokens separate; only add a stale-token fallback if maintainers define a narrow stale-state contract and cover it with tests.

## Reproduction Assessment

Yes. Current main and v2026.6.10 still render the before/after arrow whenever both compaction token fields are non-null, so a compact result like 0 to 36 remains source-reproducible.

## Solution Assessment

No, not as submitted. The formatter guard is the right narrow idea, but forwarding context-window values as live usage and letting JSONL byte estimates override stored totals is not the safest fix path.

- **[P1] Pass live usage tokens, not context windows:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from `params.contextTokens` or persisted `contextTokens`, which are context-window budgets, but `currentTokenCount` is the observed live prompt-token count and is later used for compaction metrics and checkpoint `tokensBefore`.   - confidence: 0.93
- **[P2] Keep byte estimates from overriding stored totals:** `src/auto-reply/reply/commands-compact.ts:224`
  - body: `Math.max(storedTotal, fileEstimate)` lets a rough JSONL bytes/4 estimate replace a fresh stored token total whenever transcript JSON is larger, especially with tool results. Use an estimate only when the stored total is missing or explicitly proven stale.
  - confidence: 0.86
