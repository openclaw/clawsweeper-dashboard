## Summary

This PR is a plausible fix for a real `/compact` reply bug, but it still has a blocking contract problem: it forwards a context-window limit as `currentTokenCount`, which the compactor treats as observed live prompt usage.

## What This Changes

The branch changes `src/auto-reply/reply/commands-compact.ts` to guard `/compact` before/after labels, estimate stale session totals from the JSONL file size, and forward a computed token count to embedded compaction.

## Best Possible Solution

Land a corrected patch that keeps the label guard and stale-total fallback, omits `currentTokenCount` unless true live prompt usage is available, and adds focused regression tests.

## Reproduction Assessment

Yes, at source level. Current `main` still has the unconditional before/after label path and no file-size fallback, and the PR body includes copied before/after Telegram output for the small-session case.

## Solution Assessment

No, not as written. The label guard and stale-total fallback are in the right owner module, but forwarding `contextTokens` as `currentTokenCount` violates the compactor contract and should be removed unless a real usage snapshot is available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass only live usage tokens to compaction:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are model context-window limits, but `compactEmbeddedPiSession.currentTokenCount` is defined as observed live prompt usage and is returned as `tokensBefore`/used for token-after estimation and checkpoints. Passing the window size can make `/compact` report bogus before counts; omit this field unless a real usage count is available.
