## Summary

Keep open: current main still has the misleading manual `/compact` reply path, but this PR is not correct as written because it forwards a context-window budget as live prompt usage and is stale against newer current-main context-budget handling.

## What This Changes

The PR updates manual `/compact` reply formatting to suppress misleading before/after labels, add a JSONL file-size fallback for stale token totals, and pass a token count into compaction.

## Best Possible Solution

Land a corrected `/compact` fix that keeps the label guard and stale-total fallback, omits `currentTokenCount` unless true live prompt usage is available, preserves current-main context-budget resolution, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still formats the arrow whenever both token counts are present and still uses the stale stored-total path for the context summary, while the PR body includes copied Telegram before/after output for the small-session case.

## Solution Assessment

No, not as written. The label guard and stale-total fallback are the right direction, but deriving `currentTokenCount` from context-window metadata violates the compactor contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass live usage tokens, not context windows:** `src/auto-reply/reply/commands-compact.ts:128-134`
  - body: `observedContextTokens` is derived from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are context-window budgets, then forwarded as `currentTokenCount`. The compactor treats that field as live prompt usage and can persist or return it as `tokensBefore`, so manual `/compact` can still report bogus before counts. Omit this field unless a real usage measurement is available.
  - confidence: 0.94
