## Summary

Keep open: the UX bug is real and the branch has useful fixes and proof, but the current patch passes context-window limits as live compaction usage, which can produce bogus before counts and skew diagnostics.

## What This Changes

The PR changes manual `/compact` reply formatting, forwards a token count into embedded compaction, and estimates stale session totals from the JSONL file size.

## Best Possible Solution

Land a corrected patch that keeps the label guard and stale-total display fallback, omits `currentTokenCount` unless true live prompt usage is available, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main can be exercised with a focused command test that mocks `compactEmbeddedPiSession` to return `tokensBefore <= tokensAfter`, which renders the misleading arrow today; the PR body also includes copied before/after Telegram output.

## Solution Assessment

No. The label guard and stale-total fallback are in the right owner module, but forwarding `contextTokens` as `currentTokenCount` violates the compactor contract because that field expects live prompt usage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass only live usage tokens to compaction:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` is derived from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are context-window limits, but `currentTokenCount` is documented as caller-observed live prompt tokens and is returned as `tokensBefore`/used for token-after estimation. Passing a window size can make `/compact` report the model window as the before count and skew diagnostics; omit this field unless a real usage snapshot is available.
  - confidence: 0.94
