## Summary

Keep this PR open, but do not merge it as-is. The underlying `/compact` reply bug is real on current `main`, and the label/context-summary parts of the PR are directionally useful, but the branch forwards the model context window as `currentTokenCount`, which violates the compactor contract and can produce inflated `tokensBefore` diagnostics.

## What This Changes

The PR modifies `src/auto-reply/reply/commands-compact.ts` to guard non-decreasing compaction labels, pass a token count into manual compaction, and estimate stale session totals from the JSONL file size.

## Best Possible Solution

Land a corrected version that keeps the non-decreasing-label guard and stale-total display fallback, but only passes `currentTokenCount` when the manual handler has a true fresh prompt-token usage value; otherwise omit it and let the compactor use transcript estimates. Add focused coverage for the non-decreasing label and stale-token display path plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A focused test can mock `compactEmbeddedPiSession` to return `tokensBefore <= tokensAfter` and verify the reply label; current main's label code renders the arrow unconditionally in that case. The PR body also gives a concrete Telegram stale-total session reproduction.

## Solution Assessment

No. The PR addresses the visible label bug, but the `currentTokenCount` forward uses the context-window field, which is the wrong contract for compaction diagnostics and token-after sanity checks. The safer fix is to omit that forward unless an actual observed prompt-token count is available.
