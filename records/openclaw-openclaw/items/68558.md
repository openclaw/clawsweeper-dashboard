## Summary

Keep this PR open: the reported `/compact` output problem is real on current `main`, and the branch fixes part of it, but the current head still passes the model context window as `currentTokenCount`, which violates the compactor contract and can produce wrong diagnostics or token-after checks.

## What This Changes

The PR modifies `src/auto-reply/reply/commands-compact.ts` to change manual `/compact` reply labeling, pass a token count into compaction, and estimate stale session totals from the JSONL file size.

## Best Possible Solution

Land a corrected patch that keeps the non-decreasing-label guard and stale-total display fallback, omits `currentTokenCount` unless true live prompt usage is available, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. A focused unit test can mock `compactEmbeddedPiSession` to return `tokensBefore <= tokensAfter` and current `main` will render the misleading arrow; the PR body also provides copied Telegram before/after output for a stale-total session.

## Solution Assessment

No. The label guard and stale-total fallback are in the right owner module, but passing `contextTokens` as `currentTokenCount` is not the narrow maintainable fix because that field expects live prompt usage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass only live usage tokens to compaction:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are model context-window limits. `compactEmbeddedPiSession.currentTokenCount` is caller-observed live prompt usage and is reused for `tokensBefore` and token-after estimation, so this can make manual `/compact` report the window size as the before count and skew compaction diagnostics. Omit this field unless the handler has a real usage snapshot.
  - confidence: 0.94
