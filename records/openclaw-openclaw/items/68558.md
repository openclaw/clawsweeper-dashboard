## Summary

Keep open: the PR addresses a real misleading `/compact` reply, but the current branch forwards the model context window as `currentTokenCount`, which violates the compactor contract and can create bogus before-token diagnostics.

## What This Changes

The PR changes `src/auto-reply/reply/commands-compact.ts` to suppress non-reducing `/compact` labels, estimate stale totals from session-file size, and pass a derived token count into manual compaction.

## Best Possible Solution

Land a corrected compact-command fix that keeps the label guard and stale-total fallback, removes the context-window `currentTokenCount` forward unless real usage is available, and adds focused tests.

## Reproduction Assessment

Yes, at source level. Current main still renders the before/after label whenever both counts are present, and the PR body includes copied Telegram output for the small-session case.

## Solution Assessment

No, not as written. The label and stale-total fallback are the right direction, but forwarding `contextTokens` as `currentTokenCount` is not the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass only live usage tokens to compactor:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are context-window caps, but `compactEmbeddedPiSession.currentTokenCount` is caller-observed live prompt usage and is persisted or returned as `tokensBefore`. Passing the window size can make manual `/compact` report or checkpoint bogus before counts, so omit this field unless the command has a real usage snapshot.
  - confidence: 0.94
