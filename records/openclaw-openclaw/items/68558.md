## Summary

Keep open: the central `/compact` formatter bug is still present on current main and in the latest release, but this branch is dirty against current main and its token-accounting changes would mix context-window budgets with observed usage state.

## What This Changes

The PR changes the manual `/compact` reply path to suppress non-decreasing compact arrows, pass a token count into compaction, and estimate displayed context usage from session JSONL size when stored totals look stale.

## Best Possible Solution

Port the non-decreasing label guard onto current main while keeping context budget and observed usage tokens separate; only add a stale-token fallback if maintainers can prove a narrow stale-state contract and test it.

## Reproduction Assessment

Yes. Current main and v2026.6.10 still render the before/after arrow whenever both compaction token fields are non-null, so a compact result like 0 to 36 remains source-reproducible.

## Solution Assessment

No, not as submitted. The formatter guard is the right narrow idea, but forwarding context-window values as live usage and letting JSONL byte estimates override totals is not the safest fix path.

- **[P1] Pass live usage tokens, not context windows:** `src/auto-reply/reply/commands-compact.ts:169`
