## Summary

Keep open: this is a useful, source-reproducible `/compact` reply bug fix, but the PR still passes context-window metadata as `currentTokenCount`, which the compactor treats as live prompt usage for metrics, checkpoints, and returned `tokensBefore`.

## What This Changes

The PR changes manual `/compact` reply formatting to suppress non-reducing before/after labels, estimate stale session totals from the session JSONL file, and pass a derived token count into compaction.

## Best Possible Solution

Land a corrected `/compact` reply fix that keeps the label guard and stale-total display fallback, omits `currentTokenCount` unless a real live usage snapshot is available, and adds focused command tests.

## Reproduction Assessment

Yes, at source level. Current main emits the before/after label whenever both counts exist, and the PR body includes copied Telegram output for the small stale-total session case.

## Solution Assessment

No, not as written. The label guard and stale-total fallback are the right direction, but context-window metadata must not be passed as `compactEmbeddedPiSession.currentTokenCount`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass only live usage tokens to the compactor:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are context-window limits, but `compactEmbeddedPiSession.currentTokenCount` is caller-observed prompt usage and is reused for hook metrics, checkpoints, and returned `tokensBefore`. On large-window models this can make manual `/compact` report a bogus before count and skew the post-compaction sanity baseline; omit this field unless manual compact has a real usage snapshot.
  - confidence: 0.94
