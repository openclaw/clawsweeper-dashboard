## Summary

Keep open: the visible `/compact` reply bug still exists on current main and the PR contains useful formatting/fallback work, but the patch forwards context-window budgets as live usage tokens, which violates the compactor contract and can keep the token labels misleading.

## What This Changes

This PR changes manual `/compact` reply formatting to hide non-shrinking token arrows, estimates stale session totals from the JSONL file size, and forwards a token count into compaction.

## Best Possible Solution

Land a corrected patch that keeps current-main context-budget resolution, guards the label when `before <= after`, estimates stale totals from the session file, omits `currentTokenCount` unless true usage is available, and adds focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main builds `Compacted (before → after)` whenever both token counts exist, and the PR body includes copied Telegram before/after `/compact` output for a small session.

## Solution Assessment

No, not as written. The label guard and file-size fallback are the right direction, but forwarding context-window metadata as `currentTokenCount` is not the narrow correct fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass live usage tokens to compaction:** `src/auto-reply/reply/commands-compact.ts:166`
  - body: `observedContextTokens` is derived from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are model context-window budgets, but `currentTokenCount` is caller-observed prompt usage consumed by compaction metrics and before-count output. Passing the window can make manual `/compact` report bogus before counts, so omit this field unless a real usage measurement is available.
