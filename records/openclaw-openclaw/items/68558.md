## Summary

Keep open: this is a real user-facing `/compact` bug, but the PR is not mergeable as written because it passes context-window budgets as `currentTokenCount`, which the compactor treats as live prompt usage and reuses for `tokensBefore`. A narrow repair should preserve the useful label guard and stale-total fallback while removing that bad forward and rebasing onto current main.

## What This Changes

The PR adds a session-file token estimate, changes `/compact` labels to avoid a before-to-after arrow when compaction did not reduce tokens, and forwards a token count to manual compaction.

## Best Possible Solution

Land a corrected `/compact` fix that keeps the label guard and stale-total fallback, omits `currentTokenCount` unless a real live usage measurement is available, preserves current-main context-budget resolution, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main still renders the compaction arrow whenever both token counts are present and still bases the context summary on `tokensAfter` or stored totals only. The PR body also includes copied Telegram before/after output for the small stale-total session.

## Solution Assessment

No, not as written. The label guard and stale-total fallback are the right direction, but deriving `currentTokenCount` from context-window metadata is not the narrow maintainable fix.
