## What This Changes

The PR changes manual `/compact` reply formatting by adding a before/after label guard, forwarding a derived token count into compaction, and estimating context usage from the session file.

## Best Possible Solution

Port the formatter guard onto current main, keep context budget and live usage as separate inputs, and leave broader stale/unknown Codex context accounting to the canonical session-state issue.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 still render a `tokensBefore → tokensAfter` label whenever both values are non-null, so a small-session result like `0` and `36` can produce the misleading output.

## Solution Assessment

No, this is not the best fix as submitted. The narrow maintainable fix is to keep the formatter clamp, preserve current main compact runtime behavior, and avoid treating context-window budgets or rough file-size estimates as canonical session usage.

Full review comments:

- **[P1] Pass usage tokens, not context windows:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from `params.contextTokens` or persisted `contextTokens`, which are context-window budget values, but `currentTokenCount` is the live prompt-token usage contract. Passing a 272k/1M window here can make manual `/compact` report or checkpoint a huge `tokensBefore` value for a small session instead of actual session usage.
  - confidence: 0.91
- **[P2] Do not let file size override fresh token totals:** `src/auto-reply/reply/commands-compact.ts:224`
  - body: `Math.max(storedTotal, fileEstimate)` lets a rough JSONL bytes/4 estimate replace a fresh stored token total whenever transcript JSON is large, especially with tool results. Use this only when the stored total is missing or known stale, or keep the canonical stored/session result so the context summary does not overstate usage.
  - confidence: 0.84
