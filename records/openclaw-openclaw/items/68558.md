## Summary

Keep open: the `/compact` UX fix is useful, but the patch currently feeds model context-window capacity into a compactor field that is meant for live prompt usage, which can make manual compaction diagnostics and checkpoints report capacity as usage.

## What This Changes

The PR changes `src/auto-reply/reply/commands-compact.ts` to guard compact before/after labels, pass a derived token count into compaction, and estimate stale context totals from session file size.

## Best Possible Solution

Land a corrected `/compact` reply fix that preserves the label guard and stale-total context fallback, only passes `currentTokenCount` when a real live usage snapshot exists, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main formats the before/after compact label whenever both counts exist, and the PR body includes copied Telegram before/after output for a small stale-total session.

## Solution Assessment

No, not as written. The label guard and stale-total fallback are the right direction, but `currentTokenCount` must not be populated from context-window metadata; omit it unless manual `/compact` has a real live usage snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass only live usage tokens to the compactor:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from the model context-window cap, but `compactEmbeddedPiSession.currentTokenCount` is caller-observed prompt usage and is reused as compaction metrics/checkpoint `tokensBefore`. Passing the window cap can make manual `/compact` report and persist capacity as usage; omit this field unless the command has a real usage snapshot.
