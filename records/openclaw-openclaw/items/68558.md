## Summary

Keep open. The reported /compact reply bug is valid and the branch contains useful UI fixes, but it is not mergeable as written because it forwards context-window values as live usage tokens and the branch now conflicts with current main's newer context-budget resolver.

## What This Changes

The branch modifies `src/auto-reply/reply/commands-compact.ts` to change manual `/compact` label formatting, add a JSONL file-size token estimate fallback, and forward a token count into the compactor.

## Best Possible Solution

Land a corrected current-main patch that keeps the non-decreasing label guard and stale-total fallback, avoids passing context-window budgets as `currentTokenCount`, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still renders the arrow whenever both counts exist, and the PR body includes copied Telegram before/after output for the small-session case.

## Solution Assessment

No, not as written. The formatting guard and stale-total fallback are the right direction, but deriving `currentTokenCount` from context-window metadata violates the compactor contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass live usage tokens, not context windows:** `src/auto-reply/reply/commands-compact.ts:128-134`
  - body: `observedContextTokens` comes from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are context-window budgets, then gets forwarded as `currentTokenCount`. The compactor treats that field as caller-observed live prompt usage and can persist or return it as `tokensBefore`, so manual `/compact` can still report bogus before counts; omit it unless a real usage measurement is available.
  - confidence: 0.94
