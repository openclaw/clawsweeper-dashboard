## Summary

Keep this PR open, but it still needs changes before merge: current main still has the misleading `/compact` label path, while the PR incorrectly forwards context-window budgets as live prompt-token usage.

## What This Changes

The PR changes manual `/compact` reply formatting, adds a JSONL file-size token estimate fallback, and forwards an observed token count into the compactor.

## Best Possible Solution

Land a corrected current-main patch that guards the compaction label, estimates stale totals from the session file, preserves context-budget resolution, avoids `currentTokenCount` unless real usage is available, and adds regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still formats the arrow whenever both counts exist, and the PR body includes copied Telegram before/after output for the small-session case.

## Solution Assessment

No, not as written. The label guard and stale-total fallback are the right direction, but deriving `currentTokenCount` from context-window metadata violates the compactor contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass live usage tokens, not context windows:** `src/auto-reply/reply/commands-compact.ts:128-134`
  - body: `observedContextTokens` comes from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are context-window budgets, then gets forwarded as `currentTokenCount`. The compactor treats that field as caller-observed live prompt usage and can persist or return it as `tokensBefore`, so manual `/compact` can still report bogus before counts. Omit this field unless a real usage measurement is available.
  - confidence: 0.94
