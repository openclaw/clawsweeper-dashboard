## Summary

Keep open: the reported `/compact` label bug is real, but the PR still forwards context-window metadata as `currentTokenCount`, which violates the compactor contract and can create new misleading `tokensBefore` metrics.

## What This Changes

The branch changes manual `/compact` reply formatting, adds a JSONL file-size fallback for stale session totals, and passes a token count into compaction.

## Best Possible Solution

Land a corrected patch that keeps the non-shrinking label guard and stale-total fallback, preserves current-main context-budget resolution, and omits `currentTokenCount` unless manual `/compact` has real observed usage.

## Reproduction Assessment

Yes, at source level. Current main formats the misleading before/after arrow whenever both token counts exist, and the PR body provides copied Telegram before/after output for the small-session case.

## Solution Assessment

No, not as written. The label guard and stale-total fallback are the right direction, but forwarding context-window metadata as live prompt usage is not the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass live usage tokens to compaction:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are context-window budgets, but `currentTokenCount` is consumed as live prompt usage and returned/persisted as `tokensBefore`. This can make manual `/compact` report a huge bogus before count, so omit this field unless the command has a real observed usage value.
