## Summary

Keep open: the bug is real and the useful parts of the patch are in the right module, but the PR still has a blocking correctness issue because it passes model context-window tokens into a compactor parameter reserved for live prompt usage.

## What This Changes

The branch changes manual `/compact` reply formatting in `src/auto-reply/reply/commands-compact.ts`, forwards a token count to compaction, and estimates stale session totals from the JSONL file size.

## Best Possible Solution

Land a corrected patch that keeps the non-decreasing-label guard and stale-total display fallback, omits `currentTokenCount` unless true live prompt usage is available, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. A focused test can mock `compactEmbeddedPiSession` to return `tokensBefore <= tokensAfter` and current main will render the misleading arrow; the PR body also provides copied Telegram before/after output.

## Solution Assessment

No. The label guard and stale-total fallback are a good direction, but passing `contextTokens` as `currentTokenCount` violates the compactor contract and should be removed unless the handler has a real live usage snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass only live usage tokens to compaction:** `src/auto-reply/reply/commands-compact.ts:169`
  - body: `observedContextTokens` comes from `params.contextTokens` or `targetSessionEntry.contextTokens`, which are the model context-window budget. `compactEmbeddedPiSession.currentTokenCount` is documented and consumed as live prompt usage, so this can make manual `/compact` report the window size as `tokensBefore` and skew token-after diagnostics. Leave this field unset unless this handler has an actual usage snapshot.
  - confidence: 0.94
