## Summary

Keep open. Current main still only partially covers the bundled report: configured agent names are present in the gateway row and Agents selector, but other Control UI identity/display paths do not uniformly honor `agents.list[].name`, the topbar still has no token/cost/cache readout, and the reported QClaw Windows Electron installer deletion path is not present in this source tree.

## What This Changes

The issue requests fixes for QClaw Windows upgrades deleting the installed app, Control UI agent display honoring configured/custom names, and restoring topbar token usage visibility.

## Best Possible Solution

Split the report into focused owner paths: route the QClaw Windows installer deletion to the QClaw installer/package owner with the supplied logs and registry detail, decide the intended Control UI precedence among ui.assistant, agents.list[].name, agents.list[].identity, and IDENTITY.md, and handle topbar token usage through the existing topbar/usage UX track rather than mixing it with installer recovery.

## Reproduction Assessment

Partially yes: the in-repo UI gaps are reproducible by source inspection because configured names reach gateway rows while runtime/default identity and some UI paths still use different precedence, and the topbar lacks a usage readout. No high-confidence in-repo reproduction exists for the QClaw upgrade deletion because the Windows Electron installer is not in this checkout.

## Solution Assessment

No for one automated OpenClaw core fix. The safer solution is maintainer triage that splits external installer ownership from focused Control UI name-precedence and topbar usage changes, then lands each with targeted tests once the intended UX contract is explicit.
