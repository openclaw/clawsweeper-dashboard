## Summary

Keep open: current main still lacks an explicit Telegram `allowBots` config surface in types, strict schema, UI/help metadata, docs, and group/topic runtime policy. Current code can route Telegram bot-to-bot traffic through BotFather Bot-to-Bot Communication Mode plus sender allowlists, and the Plaiground ClawHub skill is a useful Discord workaround, but the requested public parity knob remains a feature/config product decision and both related implementation PRs are closed unmerged.

## What This Changes

Request adding Telegram `allowBots` support for account/group/topic group traffic so bot-authored messages can trigger OpenClaw similarly to Discord and Slack.

## Best Possible Solution

Maintain a single clear Telegram product contract: either add scoped `allowBots` semantics across account/group/topic config and docs, or document BotFather Bot-to-Bot Communication Mode with sender allowlists as the supported path and close the parity request afterward.

## Reproduction Assessment

Yes for the repository gap: source inspection shows current main still has no accepted Telegram `allowBots` config field and no runtime `allowBots` policy. I did not run a live two-bot Telegram group reproduction in this read-only review.

## Solution Assessment

Unclear. Adding `allowBots` is the requested parity direction, but current main already has sender allowlists and Telegram now exposes Bot-to-Bot Communication Mode, so maintainers should decide whether a new knob is better than documenting the existing mechanism.
