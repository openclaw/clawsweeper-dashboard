## Summary

Keep this issue open. Current `main` still lacks a Telegram `allowBots` config/schema/docs/runtime surface, and the Plaiground ClawHub skill mentioned in the discussion is a useful Discord workaround rather than bundled Telegram behavior; Telegram’s own docs now make this a maintainer product decision because Bot-to-Bot Communication Mode and loop safeguards need to be documented with any config knob. ([core.telegram.org](https://core.telegram.org/bots/features))

## What This Changes

Add a Telegram `allowBots` configuration and runtime policy for group/topic bot-authored messages to match bot-message controls available in other channels.

## Best Possible Solution

Maintain Telegram as the owner boundary: either add `allowBots` consistently across account/group/topic config, schema, generated metadata/help, UI hints, docs, and inbound filtering with loop-safe defaults, or document the Telegram Bot-to-Bot Mode boundary and recommend Discord/ClawHub or another integration path.

## Reproduction Assessment

Yes for the missing source surface: current main rejects Telegram `allowBots` in strict schema and has no runtime bot-sender policy to consult. A live Telegram test should still verify Bot-to-Bot Communication Mode requirements for the exact group/topic workflow. ([core.telegram.org](https://core.telegram.org/bots/features))

## Solution Assessment

Unclear. A narrow parity flag is maintainable only if it is tied to Telegram’s Bot-to-Bot Communication Mode and loop safeguards; otherwise the safer result is documentation plus an alternate supported integration path.
