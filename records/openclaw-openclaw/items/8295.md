## Summary

Keep open: current main still lacks a Telegram `allowBots` or explicit bot-sender config contract, and the newer Telegram bot-to-bot/Guest Mode path makes this a product and security design item rather than conservative cleanup.

## What This Changes

The issue requests Telegram group bot-to-bot opt-in support, comparable to Discord/Slack `allowBots`, for multi-bot group interactions.

## Best Possible Solution

Define a Telegram-owned bot-sender policy that covers account/group/topic opt-in, peer authorization, loop prevention, docs, generated config metadata/help, and live Telegram QA; then decide whether that public option is named `allowBots` or a safer sender-type/peer policy.

## Reproduction Assessment

Yes for the repository gap: source inspection shows no accepted Telegram `allowBots` or explicit bot-sender policy field, and the group/topic runtime has no sender-type branch. I did not run a live two-bot Telegram setup in this read-only review.

## Solution Assessment

Unclear. A simple `allowBots` knob may be part of the answer, but Telegram bot-to-bot and Guest Mode support needs maintainer decisions on config shape, account routing, authorization, and loop prevention before implementation.
