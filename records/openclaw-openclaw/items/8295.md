## Summary

Keep open: current main still lacks a Telegram `allowBots` config contract, and the newer Telegram Bot-to-Bot/Guest Mode behavior makes the remaining work a product and security design rather than stale cleanup.

## What This Changes

Request Telegram group support for bot-authored messages via `allowBots` parity with Discord/Slack so multiple OpenClaw bots can respond to one another when mentioned.

## Best Possible Solution

Define one Telegram bot-to-bot contract, then implement scoped account/group/topic opt-in, Guest Mode handling, explicit sender/account authorization, loop safeguards, docs, generated config metadata, and live QA coverage.

## Reproduction Assessment

Yes for the repository gap: source inspection shows current main still has no accepted Telegram `allowBots` field and no Guest Mode-specific handler. I did not run the live two-bot Telegram setup in this read-only review.

## Solution Assessment

Unclear. A simple `allowBots` knob is plausible, but Telegram's current Bot-to-Bot and Guest Mode behavior requires maintainer decisions on config shape, account routing, authorization, and loop prevention before code changes. ([core.telegram.org](https://core.telegram.org/bots/features))
