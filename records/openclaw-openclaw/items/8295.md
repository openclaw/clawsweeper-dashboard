## Summary

Keep open. Current main still lacks Telegram `allowBots` in the Telegram config contract, strict schema, generated metadata/help, docs, and group/topic runtime policy; the related implementation PRs were closed unmerged, and the Plaiground ClawHub skill discussed in the issue is a Discord workaround rather than bundled Telegram behavior. Official Telegram docs now describe Bot-to-Bot Communication Mode for group chats, but they also call out required loop safeguards, so this still needs a Telegram-owned product/runtime decision before cleanup can close it. ([core.telegram.org](https://core.telegram.org/bots/features))

## What This Changes

Add Telegram `allowBots` support for group/topic bot-authored messages to match other channels and enable multi-bot setups.

## Best Possible Solution

Add a Telegram-owned implementation after maintainers confirm the bot-to-bot policy, covering account/group/topic config, strict schema, generated metadata/help, UI hints, docs, runtime filtering, and regression tests; otherwise document Telegram groups/topics as unsupported for this mode and keep the Discord Plaiground workaround as an alternative.

## Reproduction Assessment

Yes for the missing source surface: current main has no Telegram `allowBots` key in types/schema/metadata/docs and no group/topic runtime bot-sender policy to consult. I did not establish a live Telegram multi-bot failure path because that depends on Bot-to-Bot Communication Mode and group privacy/admin setup.

## Solution Assessment

Unclear. A parity flag is plausible, but the maintainable solution needs explicit Telegram semantics for Bot-to-Bot mode, mention/allowlist precedence, same-bot echoes, and loop-safe defaults before implementation.
