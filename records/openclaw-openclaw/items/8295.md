## Summary

Keep this issue open. Current main still lacks Telegram `allowBots` in the Telegram config types, strict schema, generated metadata/help surface, docs, and group/topic inbound policy; the related Plaiground ClawHub skill is a useful Discord workaround but does not ship bundled Telegram behavior. The only remaining caveat is product-level: Telegram group bot-to-bot delivery may be limited by the Bot API, so maintainers should decide whether to implement a loop-safe config gate for delivered bot-authored updates or document the Telegram boundary instead.

## What This Changes

Add Telegram `allowBots` support for groups/topics so bot-authored Telegram messages can be handled with parity to Discord, Slack, and Google Chat.

## Best Possible Solution

Maintain Telegram as the owner boundary: either add `allowBots` consistently across account/group/topic config types, strict schema, generated metadata/help, docs, UI hints, and inbound group/topic filtering with loop-safe defaults and regression tests, or document that Telegram Bot API groups cannot support the requested bot-to-bot workflow and point users to Discord/ClawHub or a separate MTProto-style integration path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: add `allowBots` under `channels.telegram.groups` or a topic and the strict Telegram schema has no such key, while the runtime group gate has no bot-sender policy to consult. A live Telegram test would still be needed to confirm exactly what bot-authored group updates Telegram delivers in practice.

## Solution Assessment

Unclear. A narrow parity flag is maintainable only if Telegram actually delivers the target bot-authored updates; otherwise the best solution is documentation plus a supported alternate integration path rather than a misleading no-op setting.
