## Summary

Keep open: current main still lacks Telegram `allowBots` across config types, strict schema, generated metadata/help, docs, and group/topic runtime policy. The Plaiground ClawHub skill mentioned in discussion is a useful Discord workaround, but the related Telegram PRs are closed unmerged and do not ship the requested bundled Telegram behavior.

## What This Changes

The issue requests Telegram group/topic `allowBots` support so bot-authored messages can participate in multi-bot Telegram group workflows.

## Best Possible Solution

Keep this issue open and either add a scoped Telegram plugin config/runtime/docs/tests implementation after validating group bot-message delivery, or document Telegram's platform boundary and point users to Telegram channels or the existing Discord Plaiground workaround.

## Reproduction Assessment

Yes, for the repository gap: source inspection shows current main has no accepted Telegram `allowBots` config field and no group/topic runtime bot-sender policy. I did not establish a live Telegram group delivery reproduction with two bot accounts.

## Solution Assessment

Unclear. Adding `allowBots` across Telegram config/schema/runtime/docs/tests is the likely parity direction, but maintainers should first validate whether Telegram groups can deliver other bot messages reliably; if not, the maintainable solution is to document that boundary and steer users to Telegram channels or the Discord Plaiground workaround.
