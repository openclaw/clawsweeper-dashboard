## Summary

Keep open: current main still lacks a Telegram `allowBots` config contract, and the newer Telegram bot-to-bot/Guest Mode discussion makes this a product and security design item rather than conservative cleanup.

## What This Changes

The issue requests a Telegram group/topic `allowBots` option so bot-authored Telegram messages can participate with parity to Discord, Slack, and Google Chat.

## Best Possible Solution

Define one Telegram-owned bot-sender policy, then implement scoped account/group/topic opt-in, loop safeguards, docs, generated config metadata/help, and live Telegram QA coverage while coordinating with the broader bot-to-bot and multi-account routing items.

## Reproduction Assessment

Yes for the repository gap: current source shows no accepted Telegram `allowBots` field and no bot-sender policy branch in the group/topic runtime. I did not run a live two-bot Telegram BotFather setup in this read-only review.

## Solution Assessment

Unclear. A simple `allowBots` knob may be part of the answer, but Telegram's newer bot-to-bot/Guest Mode behavior needs maintainer decisions on config shape, account routing, sender authorization, loop prevention, and diagnostics before implementation.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because enabling bot-authored Telegram traffic changes authorization and loop-risk boundaries.

Concerns:

- **[medium] Define bot-sender authorization before opt-in:** `extensions/telegram/src/bot-handlers.runtime.ts:703`
  - body: Accepting bot-authored Telegram messages can let one bot cause another account or agent to process group traffic unless sender, destination account, group/topic, and mention/reply rules are explicit.
