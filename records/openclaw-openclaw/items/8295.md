## Summary

Keep open: current main still lacks a Telegram `allowBots` config surface and the remaining work now also overlaps Telegram's new Bot-to-Bot/Guest Mode semantics and multi-account routing, so this needs maintainer product/security direction rather than conservative cleanup.

## What This Changes

The issue requests Telegram group/topic bot-to-bot participation through an `allowBots`-style configuration option matching Discord, Slack, and Google Chat.

## Best Possible Solution

Define one Telegram bot-to-bot product contract, then implement either scoped `allowBots` semantics across account/group/topic config and runtime or document BotFather modes plus sender allowlists as the supported path.

## Reproduction Assessment

Yes for the repository gap: source inspection shows current main still has no accepted Telegram `allowBots` field and no runtime `allowBots` bot-sender policy. I did not run a live two-bot Telegram setup in this read-only review.

## Solution Assessment

Unclear. The requested `allowBots` knob is plausible, but Telegram's new Bot-to-Bot and Guest Mode behavior means maintainers should first choose the public config, routing, authorization, and loop-prevention semantics.

## Security Review

Status: needs_attention

Summary: The issue itself is security-sensitive because enabling bot-authored Telegram traffic changes authorization and loop-risk boundaries.

Concerns:

- **[medium] Define bot-sender authorization before enabling traffic:** `extensions/telegram/src/bot-handlers.runtime.ts:617`
  - body: Accepting `from.is_bot` messages can let one bot cause another account or agent to process group traffic unless the destination account, sender, and group/topic authorization rules are explicit.
  - confidence: 0.86
- **[medium] Add explicit loop-prevention rules:** `extensions/telegram/src/bot-handlers.runtime.ts:2033`
  - body: Telegram's current message handler only skips same-bot echoes; cross-bot participation needs dedupe, mention/reply constraints, and rate/depth limits so multiple bots cannot trigger unbounded replies.
  - confidence: 0.84
