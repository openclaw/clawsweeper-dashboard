## Summary

Keep open. Current main still lacks Telegram `allowBots` in the config types, strict schema, generated metadata/help, docs, and group/topic runtime policy; the related implementation PRs were closed unmerged, and the Plaiground ClawHub skill in the discussion is a useful Discord workaround rather than bundled Telegram behavior.

## What This Changes

The issue requests a Telegram group/topic `allowBots` option so multiple bot instances can intentionally interact in shared Telegram groups, matching other channel support.

## Best Possible Solution

Add a Telegram-owned implementation only after confirming the intended Telegram bot-to-bot policy, covering account/group/topic config, strict schema, generated metadata/help, UI hints, docs, runtime filtering, and regression tests; otherwise document Telegram as unsupported for this mode and keep the Discord Plaiground workaround as an alternative.

## Reproduction Assessment

Yes for the missing source surface: current main has no Telegram `allowBots` key in types/schema/metadata/docs and no group runtime bot-sender policy to consult. I did not establish a live Telegram multi-bot failure path because that depends on Bot-to-Bot Communication Mode setup.

## Solution Assessment

Unclear. A parity flag is likely maintainable only if maintainers confirm Telegram Bot-to-Bot mode semantics, mention/allowlist precedence, same-bot echo filtering, and loop safeguards; otherwise the safer solution is explicit documentation plus an alternate integration path.
