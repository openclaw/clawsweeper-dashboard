## Summary

Keep this issue open. Current main still lacks a Telegram `allowBots` config/schema/metadata/docs/runtime surface, the related implementation PRs were closed unmerged, and the Plaiground ClawHub skill in the discussion is a useful Discord workaround rather than bundled Telegram behavior.

## What This Changes

Add a Telegram `allowBots` option for groups/topics so bot-authored messages can participate in multi-bot Telegram group workflows with parity to Discord/Slack.

## Best Possible Solution

Keep this open and land a Telegram-owned fix that adds `allowBots` consistently across account/group/topic config, strict schema, generated metadata/help, UI hints, docs, runtime filtering, and regression tests; if Telegram group bot-to-bot support is intentionally out of scope, document that boundary and keep the Discord Plaiground ClawHub workaround as an alternative.

## Reproduction Assessment

Yes for the missing source surface: current main has strict Telegram schemas with no `allowBots` key and group runtime has no bot-sender policy to consult. I did not establish a live Telegram failure path because the remaining behavior depends on Telegram Bot-to-Bot Communication Mode setup.

## Solution Assessment

Unclear. A narrow parity flag is maintainable only if it is tied to Telegram’s Bot-to-Bot mode, mention/allowlist handling, same-bot echo filtering, and loop safeguards; otherwise the safer solution is explicit documentation plus an alternate supported integration path.
