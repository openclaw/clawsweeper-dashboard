## Summary

Keep open. Current main and the latest release still throw on LLM-default empty Discord modal fields, so the underlying bug is real, but this PR is not merge-ready because it weakens modal validation, can drop container styling, and needs to be ported to the current parser split. Open PR #46455 overlaps conceptually but is still open, so it is coordination context rather than a superseding fix.

## What This Changes

The PR changes the Discord component parser to treat empty auto-filled component specs as absent and adds regression tests for empty text, empty blocks, empty modal fields, and valid modal fields.

## Best Possible Solution

Implement empty-default tolerance in the current Discord parser while keeping strict errors for malformed present `modal.fields` values and preserving meaningful `container` metadata when top-level content supplies the message text. Add focused regression coverage for empty LLM defaults, malformed modal fields, and container metadata with content fallback.

## Reproduction Assessment

Yes. A `sendMessage` action with object-shaped `components` containing `modal: { fields: [] }` reaches `readDiscordComponentSpec` on current main and throws before plain-text fallback can run.

## Solution Assessment

No. The requested behavior is correct, but this patch is not the best fix because it silently accepts malformed modal fields, can drop container-only component specs, and targets the older parser layout instead of `components.parse.ts`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject malformed modal fields:** `extensions/discord/src/components.ts:600`
  - body: The new guard only parses modal fields when `fieldsRaw` is a non-empty array, so a present malformed value like `fields: "name"` or `fields: {}` is silently treated as no modal. That regresses the current validation contract and can fall through to a plain message while discarding an intended form; keep empty arrays as absent, but still reject present non-array field values.
  - confidence: 0.9
- **[P2] Preserve container-only component specs:** `extensions/discord/src/components.ts:623-624`
  - body: This null guard ignores `components.container`, but `sendMessage` adds top-level `content` as the component text fallback after parsing. A payload with `components: { container: { accentColor: "#5865F2" } }` plus message content now returns `null`, skips the component sender, and drops the intended container styling.
  - confidence: 0.87
