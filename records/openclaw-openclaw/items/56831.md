## Summary

Keep open. The reported Discord parser bug is still present on current main and in the latest release, but this PR is not merge-ready because it patches the older parser layout and introduces two behavior regressions around malformed modal fields and container-only component metadata.

## What This Changes

The PR changes Discord component parsing so empty LLM-filled component objects return null and adds component parser regression tests.

## Best Possible Solution

Implement empty-default tolerance in the current Discord parser while keeping strict validation for malformed present modal field values and preserving container metadata when top-level content supplies the message text. Coordinate with #46455 if the broader poll/components sanitization work lands first.

## Reproduction Assessment

Yes. A `sendMessage` action with object-shaped `components` such as `{ modal: { fields: [] } }` reaches `readDiscordComponentSpec` on current main and throws before the plain-text send path can run.

## Solution Assessment

No. The requested behavior is correct, but this PR is not the best current fix because it targets the older parser file layout and would silently accept malformed modal fields while dropping container-only component specs that should still render through the component sender.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject malformed modal fields:** `extensions/discord/src/components.ts:600`
  - body: The new guard only parses modal fields when `fieldsRaw` is a non-empty array, so a present malformed value like `fields: "name"` or `fields: {}` is silently treated as no modal. That regresses the current validation contract and can fall through to a plain message while discarding an intended form; keep empty arrays as absent, but still reject present non-array field values.
  - confidence: 0.9
- **[P2] Preserve container-only component specs:** `extensions/discord/src/components.ts:623-624`
  - body: This null guard ignores `components.container`, but `sendMessage` adds top-level `content` as the component text fallback after parsing. A payload with `components: { container: { accentColor: "#5865F2" } }` plus message content now returns `null`, skips the component sender, and drops the intended container styling.
  - confidence: 0.87
