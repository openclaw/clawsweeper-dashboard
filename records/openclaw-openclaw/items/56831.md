## Summary

Keep open. The reported Discord parser bug is still present on current main, but this PR has blocking correctness issues and is stale against the split `components.parse.ts` implementation: it silently accepts malformed modal fields and can drop container-only component styling when message text is supplied by the send fallback.

## What This Changes

The PR changes Discord component parsing and tests so empty LLM-default component specs return `null` instead of throwing on empty modal fields.

## Best Possible Solution

Implement empty-default tolerance in `components.parse.ts` while preserving strict errors for malformed present modal fields and keeping meaningful container metadata on the component path when content is supplied by fallback.

## Reproduction Assessment

Yes. On current main, a `sendMessage` action with `components: { modal: { fields: [] } }` reaches `readDiscordComponentSpec` and throws before the plain-text send fallback can run.

## Solution Assessment

No. The requested behavior is right, but this patch is not the best fix because it targets the stale parser location, relaxes malformed-field validation, and can drop container styling that current code treats as meaningful.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject malformed modal fields:** `extensions/discord/src/components.ts:600`
  - body: The new guard only parses modal fields when `fieldsRaw` is a non-empty array, so a present malformed value like `fields: "name"` or `fields: {}` is silently treated as no modal. That regresses the current validation contract and can fall through to a plain message while discarding an intended form; keep empty arrays as absent, but still reject present non-array field values.
  - confidence: 0.9
- **[P2] Preserve container-only component specs:** `extensions/discord/src/components.ts:623-624`
  - body: This null guard ignores `components.container`, but `sendMessage` adds top-level `content` as the component text fallback after parsing. A payload with `components: { container: { accentColor: "#5865F2" } }` plus message content now returns `null`, skips the component sender, and drops the intended container styling.
  - confidence: 0.87
