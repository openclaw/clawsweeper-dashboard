## Summary

Keep open. The underlying Discord send failure still exists on current main and the latest release, but this PR is not merge-ready because it relaxes malformed modal validation and can drop container-only component styling; it also needs to be ported to the current split parser file.

## What This Changes

The PR branch modifies the Discord component parser to treat empty text, blocks, and modal defaults as absent and adds regression tests for empty component specs.

## Best Possible Solution

Implement the empty-default tolerance in `extensions/discord/src/components.parse.ts` while preserving strict errors for present malformed `modal.fields` values and keeping `container` metadata meaningful when text is supplied by the send action fallback. Add focused parser and send-action regression coverage.

## Reproduction Assessment

Yes. A `sendMessage` action with object-shaped `components` containing `modal: { fields: [] }` passes the object-key guard and reaches the current parser throw before plain-text fallback can run.

## Solution Assessment

No. The requested behavior is correct, but this patch is not the best solution because it silently accepts malformed modal fields, can drop container-only component specs, and targets the old parser layout instead of the current `components.parse.ts` file.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject malformed modal fields:** `extensions/discord/src/components.ts:600`
  - body: The new guard only parses modal fields when `fieldsRaw` is a non-empty array, so a present malformed value like `fields: "name"` or `fields: {}` is silently treated as no modal. That regresses the current validation contract and can fall through to a plain message while discarding an intended form; keep empty arrays as absent, but still reject present non-array field values.
  - confidence: 0.9
- **[P2] Preserve container-only component specs:** `extensions/discord/src/components.ts:623-624`
  - body: This null guard ignores `components.container`, but `sendMessage` adds top-level `content` as the component text fallback after parsing. A payload with `components: { container: { accentColor: "#5865F2" } }` plus message content now returns `null`, skips the component sender, and drops the intended container styling.
  - confidence: 0.87
