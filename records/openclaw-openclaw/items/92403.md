## Label Justifications

- `impact:session-state`: The affected behavior is UI access to earlier persisted transcript state within the same session.

## Summary

Current main and v2026.6.5 both limit WebChat to the newest 100 history entries, while upward scrolling only expands the already-loaded render set, so the earliest messages remain unreachable in long conversations. The open pagination implementation at https://github.com/openclaw/openclaw/pull/87111 is the appropriate existing fix path, but it is currently conflicting and has not merged, so this issue remains necessary until that work is refreshed, reviewed, and landed.

## What This Changes

Restore access to the earliest messages in long WebChat conversations when the user scrolls upward.

## Best Possible Solution

Land additive cursor-based `chat.history` pagination that keeps the fast recent-tail startup, loads older pages on demand, prepends and deduplicates by stable transcript identity, and preserves the WebChat viewport across ordinary and tool-heavy long conversations.

## Reproduction Assessment

Yes, at source level with high confidence: create a WebChat transcript exceeding 100 displayed entries, reload it, and the client requests only the newest 100 while upward scrolling never fetches an older page. The reporter’s screenshots corroborate the visible cutoff, although a fresh live-browser reproduction would strengthen pre-merge validation.

## Solution Assessment

Yes, additive pagination on the existing `chat.history` boundary is the best direction because it preserves bounded startup payloads while making the complete transcript reachable. Raising the fixed limit alone would postpone the cutoff and increase payload cost without solving the underlying access problem.
