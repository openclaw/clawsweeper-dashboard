## Summary

The report is plausible on current main: Control UI loads only the newest 100 history entries, then upward scrolling progressively renders only that fetched subset. The issue does not yet prove whether this fixed bound, transcript retention, or a browser-specific scroll failure caused the reported regression, so it should remain open pending a focused reproduction.

## What This Changes

Investigate why the earliest messages in a long Control UI conversation can no longer be reached by scrolling upward.

## Best Possible Solution

Provide cursor-based older-history pagination through the existing gateway and Control UI history path, preserve the visible scroll anchor while prepending each page, and cover both the 100-entry boundary and continued upward loading with focused tests.

## Reproduction Assessment

No, not at high confidence. Source inspection exposes a deterministic 100-entry cutoff that could explain the symptom, but the report does not establish conversation length, underlying transcript contents, browser behavior, or a current-main end-to-end failure.

## Solution Assessment

Unclear pending gateway contract inspection and reproduction. If the 100-entry cutoff is confirmed, extending the existing history path with cursor pagination is preferable to merely increasing the limit or adding a second transcript source.
