## Summary

Keep open: current main still lacks the requested runtime markers, but this branch is not merge-ready because it has source-reproducible env-marker correctness issues, missing real behavior proof, and unresolved overlap with newer exec-marker PRs.

## What This Changes

The branch adds `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` injection for exec, ACP client, and TUI local-shell child processes, with docs and tests.

## Best Possible Solution

Define one maintained runtime-marker contract, land the canonical exec implementation, and only extend ACP client and TUI local-shell after rebasing onto current main with reserved-marker scrubbing, docs, tests, and real behavior proof.

## Reproduction Assessment

Yes for source-level PR findings: the PR head shows the approved gateway follow-up still drops `agentId`, and the new resolver preserves inherited session markers when the current context lacks one. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best final shape until the public env contract is chosen, stale markers are scrubbed, explicit agent identity is forwarded consistently, and overlapping exec-marker PRs are reconciled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec follow-ups:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveExecRuntimeEnv` falls back to deriving an id from the session key, which becomes `main` for legacy/global keys. The approved gateway follow-up path has `params.agentId` but still calls `runExecProcess` without it, so an approved command from a non-main legacy/global context can expose `OPENCLAW_AGENT_ID=main` instead of the caller's explicit agent id.
  - confidence: 0.88
- **[P2] Scrub stale runtime markers before injection:** `src/agents/bash-tools.exec-runtime.ts:265-268`
  - body: The resolver starts from `...baseEnv` and only writes `OPENCLAW_SESSION_KEY` when a current key exists. A stale parent or caller-provided marker can therefore survive into a context with no bound session, misrepresenting the runtime context that these markers are meant to describe.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, install, release, or package-resolution change was found, but the diff exposes session metadata through inherited child-process environments and needs stale-marker handling.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:268`
  - body: The patch makes `OPENCLAW_SESSION_KEY` available to exec, ACP client, and TUI local-shell children. Environment variables are inherited by descendants and can appear in logs, traces, crash reports, shell debugging, or process inspection, so maintainers should explicitly accept the exposure and redaction expectations before making it public.
  - confidence: 0.8

  - body: Because the new resolver spreads inherited env before conditional injection, an old or caller-provided `OPENCLAW_SESSION_KEY` can remain visible when the current runtime has no session key, creating a spoofed context marker.
  - confidence: 0.84
