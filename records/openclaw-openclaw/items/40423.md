## Summary

Keep open. The feature is still absent from current main and the PR remains useful, but it should not land as-is because one exec approval follow-up path can still lose explicit agent identity, the session-key env exposure needs maintainer policy, and the ACP portion is stale against the current helper split.

## What This Changes

The PR adds OPENCLAW_AGENT_ID and OPENCLAW_SESSION_KEY runtime environment markers to exec, ACP client, and TUI local shell child processes, plus matching docs and tests.

## Best Possible Solution

Rework this as one current-main runtime-marker contract: preserve explicit agent-id precedence in every exec path, decide and document whether session-key env exposure is default or opt-in, adapt ACP changes to client-helpers, keep TUI local-shell context lazy and current, and reconcile the exec subset with #72980 before landing.

## Reproduction Assessment

Yes for the blocking review finding: the source path is clear because ProcessGatewayAllowlistParams has agentId but its approval follow-up runExecProcess call does not pass it, while the proposed resolver can fall back to main for missing or legacy keys. A focused test should run an approved gateway/safe-bin path with an explicit non-main agentId and a legacy/global session key and assert the spawned env keeps that explicit agent id.

## Solution Assessment

No, this PR is not the best landing shape as-is. The direction is reasonable, but it needs rebase/rework, explicit session-key exposure policy, full agentId propagation, and overlap reconciliation before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid defaulting unthreaded agent identity to main:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveExecRuntimeEnv` falls back to deriving an agent id from the session key, which returns `main` for missing or legacy keys. The gateway approval follow-up path still calls `runExecProcess` without forwarding `agentId`, so approved commands for non-main/global contexts can receive `OPENCLAW_AGENT_ID=main`, breaking the deterministic identity contract this PR adds.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff has no dependency or CI supply-chain change, but it deliberately exposes session keys to child-process environments and needs explicit maintainer policy before merge.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:267`
  - body: The patch injects `OPENCLAW_SESSION_KEY` into exec, ACP client, and TUI local-shell children. Environment variables are inherited by subprocesses and often appear in logs, traces, shell debugging, and crash reports, so default exposure, caller override behavior, and redaction expectations should be settled before this becomes a public contract.
  - confidence: 0.73
