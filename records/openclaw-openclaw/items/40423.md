## Summary

Keep open: current main still lacks these runtime identity markers, and this PR has unique ACP/TUI scope beyond the open exec-only follow-ups, but it needs maintainer contract review, correctness fixes, and real behavior proof before merge.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec, ACP client, and TUI local-shell child-process environments, with docs and tests.

## Best Possible Solution

Define one trusted runtime-marker contract: land or replace the canonical exec path through https://github.com/openclaw/openclaw/pull/73349 or a successor with scrubbing and proof, then separately decide whether ACP client and TUI local shell should expose the same markers.

## Reproduction Assessment

Yes for source-level PR findings: the PR diff shows stale-marker preservation, and current main shows the approved gateway follow-up path still lacks explicit `agentId` forwarding. I did not run tests because this is a read-only review.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the best final shape until trusted marker semantics are chosen, stale markers are scrubbed, explicit agent identity is forwarded consistently, overlapping exec PRs are reconciled, and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec follow-ups:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: The new resolver falls back to deriving an id from the session key, but the approved gateway follow-up path still calls `runExecProcess` without `params.agentId`. In legacy/global non-main contexts, an approved command can expose `OPENCLAW_AGENT_ID=main` instead of the caller's explicit id, breaking routing scripts.
  - confidence: 0.88
- **[P2] Scrub reserved runtime markers before injection:** `src/agents/bash-tools.exec-runtime.ts:265-268`
  - body: The helper spreads `baseEnv` and only overwrites `OPENCLAW_SESSION_KEY` when a current key exists. A stale parent or caller-provided marker can survive into an unbound context; ACP and TUI helpers need the same reserved-marker handling to keep trusted markers from being spoofed.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff exposes session metadata through child-process environments and needs explicit trusted-marker, spoofing, and redaction semantics before merge.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:268`
  - body: The patch makes `OPENCLAW_SESSION_KEY` available to exec, ACP client, and TUI local-shell children. Environment variables are inherited by descendants and can appear in logs, traces, shell debugging, or process inspection, so maintainers should explicitly accept the exposure and document that session keys are routing context, not authorization tokens.
