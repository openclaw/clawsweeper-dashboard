## Summary

Keep open. Current main still only exposes shell-type markers, the active exec-focused implementation at https://github.com/openclaw/openclaw/pull/73349 does not cover this PR's ACP client or TUI local-shell scope, and this branch still has correctness, security-policy, and real-behavior-proof blockers.

## What This Changes

The branch adds `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` injection for exec, ACP client, and TUI local-shell child processes, plus docs and tests.

## Best Possible Solution

Approve one trusted env-marker contract with override/clearing semantics and routing-only documentation, land the exec path through the canonical implementation, then decide whether ACP client and TUI local shell should receive the same markers.

## Reproduction Assessment

Yes for the PR review findings by source inspection: the diff preserves inherited reserved markers, and the current approved gateway follow-up path still lacks explicit `agentId` forwarding. I did not run tests because this review is read-only.

## Solution Assessment

No as-is. The direction is useful, but the branch should not merge until reserved marker clearing, explicit agent forwarding, session-key exposure policy, canonical scope, conflicts, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec runs:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveAgentIdFromSessionKey` falls back to `main` when no explicit agent id is provided, but the approved gateway follow-up path still reaches `runExecProcess` without forwarding `params.agentId`. Approved commands from non-main/global agents can therefore expose the wrong `OPENCLAW_AGENT_ID`; thread the explicit id through that path and cover an explicit-id mismatch.
  - confidence: 0.88
- **[P2] Clear reserved markers before injecting runtime values:** `src/agents/bash-tools.exec-runtime.ts:265-268`
  - body: The resolver spreads `baseEnv` before conditional session injection, so stale or caller-provided `OPENCLAW_SESSION_KEY` can survive when no current key exists; the ACP and TUI helpers use the same pattern. Delete or overwrite reserved marker keys before returning child envs so the values are always the trusted current context.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No dependency or workflow change was found, but the patch intentionally exposes session-routing metadata through inherited child-process environments.

Concerns:

- **[medium] Approve session marker exposure:** `src/agents/bash-tools.exec-runtime.ts:268`
  - body: `OPENCLAW_SESSION_KEY` would become visible to exec, ACP client, and TUI local-shell children. Environment variables are inherited by descendants and can appear in logs or process inspection, so maintainers should confirm these keys are routing context rather than standalone authorization secrets and document redaction expectations.
  - confidence: 0.84
- **[low] Prevent stale marker spoofing:** `src/agents/bash-tools.exec-runtime.ts:265`
  - body: Because reserved markers are copied from `baseEnv` before conditional injection, an old or caller-provided marker can remain visible when the current runtime has no bound session key or agent context.
  - confidence: 0.86
