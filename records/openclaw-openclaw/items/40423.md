## Summary

Keep open: current main still only exposes the existing `OPENCLAW_SHELL` markers, while this PR adds a broader trusted runtime marker contract that still has blocking source-level issues, missing real behavior proof, and overlap with a narrower exec-focused PR.

## What This Changes

The PR adds `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` to exec, ACP client, and TUI local-shell child-process environments, with related docs and tests.

## Best Possible Solution

Define one trusted runtime-marker contract: land or rework the canonical exec path through https://github.com/openclaw/openclaw/pull/73349 or a successor, then separately decide whether ACP client and TUI local shell should expose the same markers.

## Reproduction Assessment

Yes for the PR findings: the PR head shows stale marker preservation and the approved gateway follow-up path still omits explicit `agentId` forwarding. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but this branch should not merge until trusted marker semantics are settled, reserved markers are scrubbed or overwritten consistently, explicit agent identity is forwarded through approved exec paths, and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec follow-ups:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: The new fallback returns `main` for missing or legacy session keys, but the approved gateway follow-up path on this branch still calls `runExecProcess` without `params.agentId`. Non-main global or legacy agents can therefore expose `OPENCLAW_AGENT_ID=main` after approval; thread `agentId` into that path and cover an explicit-id mismatch.
  - confidence: 0.88
- **[P2] Scrub reserved runtime markers before injection:** `src/agents/bash-tools.exec-runtime.ts:265-268`
  - body: The resolver spreads `baseEnv` and only writes `OPENCLAW_SESSION_KEY` when a current key exists. A parent or caller-provided `OPENCLAW_SESSION_KEY` can survive into an unbound child process, and the ACP/TUI helpers use the same pattern for reserved markers; clear or overwrite these markers before injecting trusted values.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff exposes session metadata through child-process environments and needs explicit trust, spoofing, and redaction semantics before merge.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:268`
  - body: The patch makes `OPENCLAW_SESSION_KEY` available to exec, ACP client, and TUI local-shell children. Environment variables are inherited by descendants and can appear in logs, shell tracing, crash reports, or process inspection, so maintainers should explicitly confirm that these keys are routing context rather than authorization secrets and document redaction expectations.
  - confidence: 0.8
- **[low] Prevent stale marker spoofing:** `src/agents/bash-tools.exec-runtime.ts:265`
  - body: Because reserved markers are copied from `baseEnv` before conditional injection, an old or caller-provided marker can remain visible when the current runtime has no bound session key or agent context.
  - confidence: 0.86
