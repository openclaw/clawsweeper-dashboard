## Summary

Keep open. Current main still only exposes the existing `OPENCLAW_SHELL` markers, the overlapping exec-only work is still open in a separate PR, and this broader branch still has correctness, security-contract, rebase, and real-behavior-proof blockers before it can merge.

## What This Changes

The PR adds `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` injection for exec, ACP client, and TUI local-shell child processes, with related docs and tests.

## Best Possible Solution

Land one trusted exec env-marker contract through the canonical exec PR or a successor, then separately decide whether ACP client and TUI local shell should expose the same markers.

## Reproduction Assessment

Yes for the PR findings: the PR head source shows stale marker preservation and the approved gateway follow-up path still omits explicit `agentId` forwarding. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but this branch should not merge until trusted marker semantics are settled, reserved markers are scrubbed or overwritten consistently, explicit agent identity is forwarded through approved exec paths, and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec follow-ups:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveAgentIdFromSessionKey` falls back to `main` for missing or legacy keys, but the approved gateway follow-up path on this branch still calls `runExecProcess` without `params.agentId`. Non-main global or legacy agents can therefore expose `OPENCLAW_AGENT_ID=main` after approval; thread `agentId` into that path and cover an explicit-id mismatch.
  - confidence: 0.88
- **[P2] Clear reserved markers before injecting trusted values:** `src/agents/bash-tools.exec-runtime.ts:265-268`
  - body: The resolver spreads `baseEnv` and only writes `OPENCLAW_SESSION_KEY` when a current key exists. A parent or caller-provided `OPENCLAW_SESSION_KEY` can survive into an unbound child process, and the ACP/TUI helpers use the same pattern; clear or overwrite these reserved markers before returning the child env.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff does not change dependencies or workflows, but it intentionally exposes session-routing metadata through inherited child-process environments and needs maintainer approval of that contract.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:268`
  - body: `OPENCLAW_SESSION_KEY` would become visible to exec, ACP client, and TUI local-shell children. Environment variables are inherited by descendants and can appear in logs, shell tracing, crash reports, or process inspection, so maintainers should explicitly confirm that these keys are routing context rather than authorization secrets and document redaction expectations.
  - confidence: 0.8
- **[low] Prevent stale marker spoofing:** `src/agents/bash-tools.exec-runtime.ts:265`
  - body: Because reserved markers are copied from `baseEnv` before conditional injection, an old or caller-provided marker can remain visible when the current runtime has no bound session key or agent context.
  - confidence: 0.86
