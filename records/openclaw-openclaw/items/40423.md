## Summary

Keep open. The runtime-marker direction is useful, but current main still lacks these markers and this PR still has blocking source-level issues, missing real behavior proof, and an unresolved security/product decision around exposing session routing metadata.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec, ACP client, and TUI local-shell child environments, with docs and tests.

## Best Possible Solution

Approve one trusted env-marker contract for exec first, then intentionally decide whether ACP client and TUI local shell should expose the same markers with reserved-key clearing and redaction docs.

## Reproduction Assessment

Yes for the PR review findings by source inspection: the PR diff shows reserved markers can survive from `baseEnv`, and the approved gateway follow-up path still omits explicit `agentId` forwarding. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but this branch should not merge until reserved markers are cleared or overwritten consistently, approved exec follow-ups preserve explicit agent identity, session-key exposure is approved, and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec follow-ups:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveAgentIdFromSessionKey` returns `main` when `agentId` is missing, and the approved gateway follow-up path still reaches `runExecProcess` without `params.agentId`. Approved commands run by non-main/global agents can therefore expose `OPENCLAW_AGENT_ID=main`; thread the explicit id through that path and cover an explicit-id mismatch.
  - confidence: 0.88
- **[P2] Clear reserved markers before returning child env:** `src/agents/bash-tools.exec-runtime.ts:265-268`
  - body: The resolver spreads `baseEnv` and only writes `OPENCLAW_SESSION_KEY` when a current key exists, so stale or caller-provided markers can survive into unbound children. The ACP/TUI helpers use the same pattern; delete or overwrite reserved keys before injecting trusted values.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow change, but it exposes session-routing metadata through inherited child-process environments and currently allows stale marker carry-over.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:268`
  - body: `OPENCLAW_SESSION_KEY` would become visible to exec, ACP client, and TUI local-shell children. Environment variables are inherited by descendants and can appear in logs or process inspection, so maintainers should confirm these keys are routing context rather than standalone authorization secrets and document redaction expectations.
  - confidence: 0.8
- **[low] Prevent stale marker spoofing:** `src/agents/bash-tools.exec-runtime.ts:265`
  - body: Because reserved markers are copied from `baseEnv` before conditional injection, an old or caller-provided marker can remain visible when the current runtime has no bound session key or agent context.
  - confidence: 0.86
