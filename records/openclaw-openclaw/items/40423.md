## Summary

Keep this PR open for maintainer review: current main still lacks the requested agent/session env markers beyond the existing OPENCLAW_SHELL contract, but the branch has correctness and security blockers and overlaps with https://github.com/openclaw/openclaw/pull/73349, https://github.com/openclaw/openclaw/pull/72980, and https://github.com/openclaw/openclaw/pull/76053.

## What This Changes

The branch adds OPENCLAW_AGENT_ID and OPENCLAW_SESSION_KEY injection for exec, ACP client, and TUI local-shell child processes, with docs and tests.

## Best Possible Solution

Define one maintained runtime-marker contract, reconcile it with the open exec-marker PRs, then apply it consistently to the chosen exec, ACP client, and TUI surfaces with stale-marker scrubbing, explicit agent-id precedence, docs, tests, and real behavior proof.

## Reproduction Assessment

Yes for source-level PR findings. The gateway approval follow-up path has params.agentId but does not pass it to runExecProcess, while the new resolver falls back to main for legacy/global session keys; I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is reasonable, but this branch should be rebased, consolidated with the related exec-marker PRs, and fixed for explicit agent-id forwarding plus stale marker handling before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec follow-ups:** `src/agents/bash-tools.exec-runtime.ts:261`
  - body: The new resolver falls back to a session-key-derived id, which becomes `main` for legacy/global keys. The gateway approval follow-up path has `params.agentId` but still calls `runExecProcess` without it, so approved commands from a non-main legacy/global context can expose `OPENCLAW_AGENT_ID=main`.
  - confidence: 0.88
- **[P2] Scrub stale session markers before injection:** `src/agents/bash-tools.exec-runtime.ts:263-266`
  - body: The resolver starts from inherited `baseEnv` and only overwrites `OPENCLAW_SESSION_KEY` when a current key exists. A stale parent or caller-provided marker can therefore survive in a context with no bound session, contradicting the runtime-marker contract and misrouting downstream scripts.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, install, release, or supply-chain change was found, but the diff exposes session-routing metadata through inherited subprocess environments and leaves stale marker spoofing possible.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:266`
  - body: The patch injects `OPENCLAW_SESSION_KEY` into exec, ACP client, and TUI local-shell children. Environment variables can be inherited by descendants and captured in logs, traces, crash reports, shell debugging, or process inspection, so maintainers should explicitly accept the exposure and redaction expectations before making it a public contract.
  - confidence: 0.8
- **[low] Stale session markers can survive when context is absent:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: Because the resolver spreads `baseEnv` before conditionally adding the current session key, an inherited or caller-provided `OPENCLAW_SESSION_KEY` can remain visible when the current runtime has no session key, misrepresenting the active context.
  - confidence: 0.82
