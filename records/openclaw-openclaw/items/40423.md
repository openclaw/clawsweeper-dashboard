## Summary

Keep open. Current main still only injects `OPENCLAW_SHELL` on these paths, so the requested runtime markers are not implemented, but this PR is not merge-ready because the diff can preserve stale reserved markers, misses an approved exec follow-up path, lacks real behavior proof, and overlaps the newer exec-focused work at https://github.com/openclaw/openclaw/pull/73349 while retaining unique ACP/TUI scope.

## What This Changes

The PR adds runtime environment injection of `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` for exec, ACP client, and TUI local shell, with docs and tests.

## Best Possible Solution

Land one trusted env-marker contract for exec first, then make an explicit maintainer decision on whether ACP client and TUI local shell should expose the same markers under the same reserved-key semantics.

## Reproduction Assessment

Yes for the PR findings by source inspection: the PR diff shows reserved markers can survive from `baseEnv`, and the approved gateway follow-up path still omits explicit `agentId` forwarding. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but this branch should not merge until reserved markers are scrubbed or overwritten consistently, approved exec follow-ups preserve explicit agent identity, the session-key exposure policy is approved, and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec follow-ups:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveAgentIdFromSessionKey` returns `main` when `agentId` is missing, and this branch only threads `agentId` into the direct `runExecProcess` call. The approved gateway follow-up path still reaches `runExecProcess` without `params.agentId`, so non-main global or legacy agents can expose `OPENCLAW_AGENT_ID=main` after approval; thread the ID through that path and cover an explicit-ID mismatch.
  - confidence: 0.88

  - body: The resolver spreads `baseEnv` and only sets `OPENCLAW_SESSION_KEY` when a current key exists. A parent or caller-provided marker can survive into unbound children, and the ACP/TUI helpers use the same pattern; delete or overwrite reserved markers before injecting trusted values.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff does not change dependencies or workflows, but it intentionally exposes session-routing metadata through inherited child-process environments and needs maintainer approval of that contract.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:268`
  - body: `OPENCLAW_SESSION_KEY` would become visible to exec, ACP client, and TUI local-shell children. Environment variables are inherited by descendants and can appear in logs, shell tracing, crash reports, or process inspection, so maintainers should confirm these keys are routing context rather than authorization secrets and document redaction expectations.
  - confidence: 0.8

  - body: Because reserved markers are copied from `baseEnv` before conditional injection, an old or caller-provided marker can remain visible when the current runtime has no bound session key or agent context.
  - confidence: 0.86
