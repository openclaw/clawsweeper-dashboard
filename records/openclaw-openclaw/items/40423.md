## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec runs:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveAgentIdFromSessionKey` falls back to `main` when no explicit agent id is supplied, but the approved gateway follow-up path still calls `runExecProcess` without forwarding `params.agentId`. Approved commands from non-main/global agents can therefore expose the wrong `OPENCLAW_AGENT_ID`; thread the explicit id through that path and cover an explicit-id mismatch.
  - confidence: 0.88
- **[P2] Clear reserved markers before injecting runtime values:** `src/agents/bash-tools.exec-runtime.ts:265-268`
  - body: The resolver spreads `baseEnv` before conditional session injection, so stale or caller-provided `OPENCLAW_SESSION_KEY` can survive when no current key exists; the ACP and TUI helpers use the same pattern. Delete or overwrite reserved marker keys before returning child envs so values always reflect the trusted current context.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces inherited session-routing metadata in child-process environments and needs an explicit trusted-marker and redaction contract before merge.

Concerns:

- **[medium] Approve session marker exposure:** `src/agents/bash-tools.exec-runtime.ts:268`
  - body: `OPENCLAW_SESSION_KEY` would be visible to exec, ACP client, and TUI local-shell children. Environment variables are inherited by descendants and can appear in logs or process inspection, so maintainers should confirm these keys are routing context rather than standalone authorization secrets and document redaction expectations.
  - confidence: 0.84

  - confidence: 0.86
