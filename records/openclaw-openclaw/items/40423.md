## Summary

Keep open: current main still lacks these runtime identity/session markers, and the PR is useful, but the latest patch is not merge-ready because it still drops explicit agent identity on the gateway approval exec path and is dirty against current main's ACP helper split.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec, ACP client, and TUI local-shell child processes, with docs and tests.

## Best Possible Solution

Port the feature onto current main as a small runtime-env contract: centralize marker resolution, thread explicit `agentId` through every exec run path, update ACP helper/TUI plumbing, docs, changelog, and regression coverage.

## Reproduction Assessment

Yes for the review finding from source inspection: `params.agentId` is available in the gateway approval path, but that `runExecProcess` call still omits it while the new helper can fall back to `main`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The direction is maintainable, but the branch should first be rebased onto current main, port ACP env handling to `client-helpers.ts`, and fix the missing gateway approval `agentId` propagation.
