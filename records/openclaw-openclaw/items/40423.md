## Summary

Keep open. Current main still lacks the requested agent/session env markers, and this PR is not merge-ready because it defines a new public runtime/security contract, has source-level defects in marker integrity and `agentId` forwarding, lacks real behavior proof, and overlaps with newer exec-marker work that should be consolidated.

## What This Changes

The PR adds `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` injection for exec, ACP client, and TUI local-shell child processes, plus docs and tests.

## Best Possible Solution

Define one maintained runtime-marker contract: land or replace the canonical exec path tracked by https://github.com/openclaw/openclaw/issues/66705 and https://github.com/openclaw/openclaw/pull/73349, then separately decide whether ACP client and TUI local-shell should expose the same trusted markers with scrubbing, docs, tests, and proof.

## Reproduction Assessment

Yes for source-level PR findings: the PR head shows the approved gateway follow-up path still drops `agentId`, and the new resolver preserves inherited session markers when the current context lacks one. I did not run tests because this was a read-only review.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best final shape until the public env contract is chosen, stale markers are scrubbed, explicit agent identity is forwarded consistently, overlapping exec-marker PRs are reconciled, and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Forward agent IDs through approved exec follow-ups:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveExecRuntimeEnv` falls back to deriving an id from the session key, which becomes `main` for legacy/global keys. The approved gateway follow-up path has `params.agentId` but still calls `runExecProcess` without it, so an approved command from a non-main legacy/global context can expose `OPENCLAW_AGENT_ID=main` instead of the caller's explicit agent id.
  - confidence: 0.88
- **[P2] Scrub stale runtime markers before injection:** `src/agents/bash-tools.exec-runtime.ts:265-268`
  - body: The resolver starts from `...baseEnv` and only writes `OPENCLAW_SESSION_KEY` when a current key exists. A stale parent or caller-provided marker can therefore survive into a context with no bound session, misrepresenting the runtime context these markers are meant to describe; the ACP and TUI helpers need the same reserved-marker handling.
  - confidence: 0.84

## Security Review
