## Summary

Keep open: current main does not implement these runtime identity markers, and the PR still needs rework before merge because the approved gateway exec follow-up path can mislabel explicit non-main agent contexts and the session-key env contract needs maintainer policy review.

## What This Changes

The PR adds `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` to exec, ACP client, and TUI local-shell child-process environments, with docs and tests.

## Best Possible Solution

Land a current-main runtime-marker contract after maintainer policy approval, preserving explicit agent-id precedence across every exec path, rebasing onto the current ACP helper layout, and reconciling session-key scope with #72980.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: in the PR's added env resolver, run an approved gateway exec follow-up with `agentId` set to a non-main value and a legacy/global session key, then observe the spawned env can fall back to `OPENCLAW_AGENT_ID=main` because the follow-up path does not forward `agentId`.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best landing shape until it forwards explicit agent identity through the approval follow-up path, rebases onto current ACP client helpers, and settles the session-key exposure policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve explicit agent IDs in approved exec runs:** `src/agents/bash-tools.exec-runtime.ts:263`
  - body: `resolveExecRuntimeEnv` falls back to a session-key-derived id, which becomes `main` for missing or legacy keys, but the gateway approval follow-up path still calls `runExecProcess` without forwarding `agentId`. Approved commands for a non-main/global context can therefore receive `OPENCLAW_AGENT_ID=main`, breaking the deterministic identity contract this PR adds.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: No dependency, CI, or package supply-chain change is visible, but exposing session keys through child-process environments is security-sensitive and needs maintainer policy before merge.

Concerns:

- **[medium] Define session-key exposure policy:** `src/agents/bash-tools.exec-runtime.ts:267`
  - body: The patch injects `OPENCLAW_SESSION_KEY` into exec, ACP client, and TUI local-shell children. Environment variables are inherited by subprocesses and may appear in logs, shell tracing, crash reports, or process inspection, so default exposure and redaction expectations should be decided before this becomes a public contract.
  - confidence: 0.74
