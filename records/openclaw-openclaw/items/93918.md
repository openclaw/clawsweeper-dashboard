## Label Justifications

- `P1`: The report describes a broken native chat approval workflow that can stall real agent turns and force repeated approvals.
- `impact:session-state`: The approved exec result can fail to reconnect with the originating agent turn, leaving session state pending or out of sync.
- `impact:message-loss`: The approval outcome and exec lifecycle result can miss the intended agent session, forcing a manual nudge through the channel.
- `impact:security`: Exec approvals are the operator-controlled command execution boundary, so the remaining fix must preserve approval-to-command/session binding.

## Summary

Keep open: current main still limits inline gateway exec approval waiting to WebChat, so native Telegram approvals remain on the async follow-up path; the open fix candidate at https://github.com/openclaw/openclaw/pull/93949 specifically targets this issue and is not merged yet.

## What This Changes

Request that native chat `/approve` resolutions resume the original agent turn with exec lifecycle output instead of requiring an extra user message that can mint a new approval id.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/93949, keeping the fix scoped to native approval-capable channels while leaving broader approval scheduling semantics to https://github.com/openclaw/openclaw/issues/70953.

## Reproduction Assessment

Yes at source level, but not by a live run in this review. Current main admits inline gateway approval waiting only for WebChat, while Telegram resolves approvals through the gateway and then uses the detached follow-up path described by the report.

## Solution Assessment

Likely yes as a fix direction: extending the existing WebChat inline approval precedent to native approval-capable channels is narrower than inventing a new resume mechanism. The exact PR should still be reviewed for security and sibling-channel coverage before merge.
