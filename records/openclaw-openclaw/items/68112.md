## Summary

Keep open: current main still has the startup scheduler-death failure, but this PR head predates current cron startup behavior and would regress deferral and maintenance semantics unless it is rebased or replaced carefully.

## What This Changes

The PR wraps cron startup missed-job catch-up in error handling, adds partial-state and zombie running-marker cleanup, and adds cron resilience tests.

## Best Possible Solution

Land a rebased or replacement startup resilience fix that always arms the cron timer after catch-up or final-persist failures while preserving startup agent-turn deferral, interrupted-run emission, and maintenance recompute semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to inject a `persist()` or `saveCronStore()` failure during startup catch-up: current main awaits `runMissedJobs()` before reaching `armTimer(state)`, and the catch-up persist calls can throw.

## Solution Assessment

No, not as currently mergeable. The direction is right, but the branch must be adapted to current main so it does not drop `deferAgentTurnJobs: true` or the maintenance-only startup recompute path.
