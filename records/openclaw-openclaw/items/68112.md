## Summary

Keep open: current main still has the startup throw-before-arm gap, so the useful fix is not implemented, but this branch is conflicting, lacks real behavior proof, and adds a duplicate stale-marker cleanup policy that needs reconciliation before merge.

## What This Changes

The PR wraps cron startup `runMissedJobs()` in a catch path, adds timeout-based `runningAtMs` zombie cleanup during timer ticks, and adds startup/zombie regression tests.

## Best Possible Solution

Rebase onto current main, keep the startup catch-up timer-arm fix narrow, and integrate any timeout-aware stale-marker behavior into `recomputeNextRunsForMaintenance()` with real gateway proof.

## Reproduction Assessment

Yes at source level. Current `start()` awaits `runMissedJobs()` before `armTimer(state)`, and `runMissedJobs()` can propagate persistence failures from startup planning or finalization; I did not run a live gateway repro in this read-only review.

## Solution Assessment

No, not as submitted. The startup catch guard is plausible, but the branch needs a current-main rebase and should reuse the existing maintenance cleanup owner instead of adding a second running-marker reaper.
