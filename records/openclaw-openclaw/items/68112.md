## Summary

Keep this PR open, but it is not mergeable as-is: current main still has the startup catch-up failure shape, while the branch drops newer startup deferral/recompute semantics and lacks external real behavior proof.

## What This Changes

The PR wraps startup `runMissedJobs()` with catch/repair logic, adds timeout-based zombie `runningAtMs` cleanup, and adds cron resilience tests.

## Best Possible Solution

Land a rebased or replacement startup resilience fix that always reaches timer arming after catch-up persistence failures while preserving startup agent-turn deferral, interrupted-run emission, and maintenance recompute behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to inject a cron-store persistence failure during `planStartupCatchup()` or `applyStartupCatchupOutcomes()`, because current main awaits that work before the later timer-arming block.

## Solution Assessment

No, not in the current PR shape. The fix direction is right, but the branch must be rebased and adapted to preserve current main's startup deferral and maintenance recompute contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Preserve startup agent-turn deferral:** `src/cron/service/ops.ts:138-140`
  - body: Current main passes `deferAgentTurnJobs: true` so overdue isolated agent-turn cron jobs are rescheduled out of the gateway startup window. This branch wraps `runMissedJobs()` but calls it with only `skipJobIds`, so merging it as-is would reintroduce immediate startup execution for those jobs and break the documented/tested startup protection.
  - confidence: 0.9

The branch's final startup block still uses full `recomputeNextRuns(state)`, which can silently skip missed work after a caught catch-up failure.
  - confidence: 0.84
