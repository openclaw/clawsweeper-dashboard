## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve interrupted-run state during catch-up recovery:** `src/cron/service/ops.ts:161-169`
  - body: This catch path clears `runningAtMs`/`nextRunAtMs` directly after startup catch-up fails. Current main's startup recovery records persisted running markers as interrupted failed runs, updates delivery/failure state, disables one-shots, and emits a finished event; bypassing that path can hide interrupted runs and change retry or delivery behavior. Rework the resilience guard onto the current recovery helper instead of deleting markers in a parallel path.
  - confidence: 0.84
- **[P2] Reuse the shared running-marker cleanup policy:** `src/cron/service/timer.ts:697-712`
  - body: The new timeout-based `clearZombieRunningMarkers` adds a second runtime policy for clearing persisted running state beside `recomputeNextRunsForMaintenance`/`normalizeJobTickState`. That makes finite-timeout jobs become runnable under different rules depending on which cron path touches them, increasing duplicate or suppressed delivery risk. The timer path should reuse the existing maintenance cleanup semantics or intentionally replace them in one shared owner.
  - confidence: 0.78
