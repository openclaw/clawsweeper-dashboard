## Summary

Keep this open. Current main still treats cron `delivery.channel` as string-only, so a persisted `{ id: "telegram" }` value is dropped and can fall back to `last`; the related fix PR was closed without merging.

## What This Changes

The issue requests that cron jobs configured for Telegram delivery do not lose their explicit channel and route through another configured channel such as Feishu.

## Best Possible Solution

Canonicalize object-shaped cron delivery channel values at cron delivery parsing and persisted-job normalization while preserving existing string behavior and provider-prefix validation.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a persisted cron job whose `delivery.channel` is `{ id: "telegram" }`: current main normalizes that to undefined and either falls back during delivery planning or deletes it during persisted normalization.

## Solution Assessment

Yes. The narrowest maintainable fix is cron-local compatibility parsing for `{ id: string }`, with regression coverage, without Telegram/Feishu special cases or a global string coercion change.
