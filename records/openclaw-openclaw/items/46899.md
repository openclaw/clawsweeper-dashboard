## Summary

Keep open: current main and the latest release still use string-only cron `delivery.channel` normalization, and the open linked fix at https://github.com/openclaw/openclaw/pull/91162 has not merged yet.

## What This Changes

Cron jobs configured for Telegram delivery should honor the configured channel instead of falling back to session-derived Feishu routing.

## Best Possible Solution

Refresh and review the linked parser-boundary fix, keep runtime `delivery.channel` canonical as a string, and merge only after proof shows a Telegram-configured cron job delivers to Telegram with another channel enabled.

## Reproduction Assessment

Yes for the source-level path: current main drops non-string `delivery.channel` values during cron normalization, then announce routing can fall back to `last`. I did not establish a live Telegram/Feishu reproduction from the reporter's exact stored job row.

## Solution Assessment

Yes, the best fix is at the parser/normalization boundary so alternate persisted input becomes the canonical string before runtime routing. The linked PR takes that shape, but this issue should remain open until that or an equivalent fix merges.
