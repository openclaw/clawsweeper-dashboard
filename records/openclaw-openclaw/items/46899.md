## Summary

Keep this open: current main and the latest release still have the source-proven cron delivery path where a non-string persisted `delivery.channel` can be erased and later resolved through `last`, while the direct repair attempt was closed unmerged.

## What This Changes

The issue reports that cron jobs configured for Telegram delivery can be routed through Feishu instead.

## Best Possible Solution

Normalize or preserve `{ id: string }` channel values at the cron delivery boundary, or reject them without falling back to `last`, with generic regression coverage and no Telegram- or Feishu-specific routing.

## Reproduction Assessment

Yes, at source level: a persisted job with `delivery.channel` shaped as `{ id: "telegram" }` is rejected by the string-only parser, deleted during hydration, and then can enter the `last` fallback path. I did not run a live scheduled Telegram/Feishu delivery in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local delivery channel normalization or fail-closed rejection plus regression tests, not channel-specific Telegram or Feishu branching.
