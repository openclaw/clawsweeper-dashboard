## Label Justifications

- `P1`: A source-proven cron routing regression can send scheduled output through the wrong channel path and block real user message delivery.
- `impact:message-loss`: The issue is specifically about scheduled Telegram cron messages being misrouted to Feishu and not reaching the intended recipient.

## Summary

Keep open. Current main and the latest release still reject object-shaped cron `delivery.channel` values and can delete the explicit Telegram channel during hydration, leaving the run to enter `last` routing where Feishu or another configured channel may be selected.

## What This Changes

Cron jobs configured for Telegram delivery are reported to lose their explicit delivery channel and fall back to Feishu with a Telegram-style target ID.

## Best Possible Solution

Normalize `{ id: string }` channel values at the cron delivery boundary, or fail closed before `last` fallback, with generic regression coverage and no Telegram- or Feishu-specific routing.

## Reproduction Assessment

Yes, at source level: a persisted job with `delivery.channel` shaped as `{ id: "telegram" }` is rejected by parsing/hydration and then reaches `last` fallback routing. I did not run a live scheduled Telegram/Feishu delivery in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local channel normalization or fail-closed rejection plus regression tests, not Telegram- or Feishu-specific branching.
