## Summary

Keep this open. Current main and v2026.5.12 still parse cron `delivery.channel` through string-only normalization, so an object-shaped persisted channel can be dropped and the delivery path can fall back through `last`; the direct fix PR was closed unmerged and no replacement was found.

## What This Changes

The issue asks OpenClaw to preserve explicit Telegram cron delivery instead of misrouting scheduled output through Feishu or another fallback channel.

## Best Possible Solution

Normalize or preserve `{ id: string }` channel values at the cron delivery boundary, or reject them safely without falling back to `last`, with generic regression coverage and no Telegram- or Feishu-specific branching.

## Reproduction Assessment

Yes, at source level: a persisted job whose `delivery.channel` is `{ id: "telegram" }` fails the string-only parser, is deleted during normalization, and then resolves through `last` fallback. I did not run a live scheduled cron delivery in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local delivery input normalization plus regression tests, not special-casing Telegram or Feishu behavior.
