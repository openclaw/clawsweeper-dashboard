Keep this open. Current main and the latest release still drop an object-shaped cron `delivery.channel` before delivery planning, which matches the linked closed fix PR's explanation for Telegram jobs falling back to `last` and reusing a Feishu route; no merged replacement fix was found.

## What This Changes

The issue reports cron jobs configured for Telegram delivery being routed through Feishu when both channels are enabled.

## Best Possible Solution

Make cron delivery parsing preserve the intended channel for `{ id: string }` persisted values, or fail safely without falling back to `last`, with focused regression coverage and no Telegram/Feishu-specific branching.

## Reproduction Assessment

Yes for the source-level compatibility path: a persisted job with `delivery.channel` set to `{ id: "telegram" }` is normalized to no channel and then resolves through `last`. The exact raw config from the reporter is not available, so I did not mark this as a live reproduced run.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local parsing/normalization for `{ id: string }` channel values, while preserving existing string behavior and avoiding channel-specific special cases.
