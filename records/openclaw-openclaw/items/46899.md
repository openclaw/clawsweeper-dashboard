## Summary

Keep open. Current main and v2026.5.20 still treat object-shaped cron `delivery.channel` values as absent, so a persisted Telegram channel can be erased and the run can fall into `last` routing where Feishu or another configured channel may be selected.

## What This Changes

The issue reports cron jobs configured for Telegram delivery being routed through Feishu after the explicit Telegram delivery channel is lost.

## Best Possible Solution

Handle `{ id: string }` channel values generically at the cron delivery boundary, or fail closed before fallback routing, while preserving valid string-channel behavior and coordinating with the implicit-`last` admission guard.

## Reproduction Assessment

Yes, at source level: a persisted job with `delivery.channel` shaped as `{ id: "telegram" }` is rejected by parsing/hydration and then reaches `last` fallback routing. I did not run a live scheduled Telegram/Feishu delivery in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local channel normalization or fail-closed rejection plus regression tests, not Telegram- or Feishu-specific branching.
