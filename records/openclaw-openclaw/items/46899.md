## Summary

Keep open. Current main still loses object-shaped persisted cron `delivery.channel` values during hydration, so the job can fall back to implicit `last` routing and select Feishu or another configured channel instead of Telegram; the latest release has the same core parsing behavior.

## What This Changes

The issue reports Telegram-configured cron delivery being routed through Feishu when a scheduled job runs.

## Best Possible Solution

Normalize `{ id: string }` cron delivery channels generically or fail closed before fallback routing, with regression coverage for persisted-job hydration and `last` fallback behavior.

## Reproduction Assessment

Yes, at source level: a persisted job with `delivery.channel` shaped as `{ id: "telegram" }` is rejected by parsing/hydration, loses the channel field, and can then reach implicit `last` fallback routing. I did not run a live scheduled Telegram/Feishu delivery in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local channel normalization or fail-closed rejection plus regression tests, not Telegram- or Feishu-specific branching.
