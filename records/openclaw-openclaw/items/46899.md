## Summary

Keep this open. Current main still treats cron `delivery.channel` as string-only, so a persisted value like `{ id: "telegram" }` is normalized away or resolved as `last`; the linked fix PR was closed without merging.

## What This Changes

The issue asks for cron jobs configured for Telegram delivery to keep using Telegram instead of falling back to another configured channel such as Feishu.

## Best Possible Solution

Canonicalize cron delivery channel values shaped like `{ id: string }` at the delivery planning and persisted-job normalization boundaries, with no Telegram/Feishu special cases and no global string-coercion broadening.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a persisted cron job whose `delivery.channel` is `{ id: "telegram" }`: current main normalizes that to undefined and can route as `last` while preserving the Telegram-style target.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local compatibility parsing for `{ id: string }`, preserving existing string behavior and avoiding provider-specific routing policy.
