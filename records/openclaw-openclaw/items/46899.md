## Summary

Keep this open. Current main and v2026.5.7 still normalize cron delivery through string-only channel parsing, so a persisted object-shaped `delivery.channel` can be dropped and announce delivery can fall back through `last`; the linked direct fix PR was closed unmerged and no canonical replacement was found.

## What This Changes

Cron jobs configured for Telegram delivery should not fall back to Feishu or another last-used channel when cron delivery normalization loses the intended channel.

## Best Possible Solution

Preserve `{ id: string }` cron delivery channel values, or fail safely without falling back to `last`, with generic cron regression coverage and no Telegram- or Feishu-specific branching.

## Reproduction Assessment

Yes, for the source-level compatibility path: a persisted job with `delivery.channel` set to `{ id: "telegram" }` parses to undefined, is deleted during store hydration, and can then resolve through `last`. I did not mark it live-reproduced because the reporter did not provide the raw persisted cron JSON and this read-only review did not run a cron job.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local delivery parsing/normalization for object-shaped channel values, plus regression tests, while preserving existing string behavior and avoiding channel-specific special cases.
