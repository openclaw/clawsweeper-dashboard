## Label Justifications

- `P1`: A source-proven cron routing regression can send scheduled output through the wrong channel path and block real user message delivery.

## Summary

Keep open. Current main and the latest release still parse cron `delivery.channel` as string-only and delete object-shaped persisted channel values before delivery resolution, so the reported Telegram-to-Feishu fallback path remains source-reproducible.

## What This Changes

Cron delivery configured for Telegram should not lose its channel and fall back to Feishu or implicit `last` routing for scheduled jobs.

## Best Possible Solution

Normalize `{ id: string }` channel values at the cron delivery boundary, or fail closed before fallback routing, while preserving valid string-channel behavior and coordinating with the implicit-`last` admission guard.

## Reproduction Assessment

Yes, at source level: a persisted job with `delivery.channel` shaped as `{ id: "telegram" }` is rejected by parsing/hydration and then reaches `last` fallback routing. I did not run a live scheduled Telegram/Feishu delivery in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local channel normalization or fail-closed rejection plus regression tests, not Telegram- or Feishu-specific branching.
