## Summary

Keep open: current main and the latest release still use string-only cron delivery channel parsing, while the linked closing PR is open and conflicting rather than merged.

## What This Changes

Cron jobs with an explicit Telegram `delivery.channel` should deliver via Telegram instead of losing that channel and falling back to another configured channel such as Feishu.

## Best Possible Solution

Refresh and review the linked parser-boundary fix or an equivalent narrow PR, keeping runtime `delivery.channel` canonical as a string and proving a Telegram cron job delivers to Telegram with another channel enabled.

## Reproduction Assessment

Yes for the source-level path: current main drops non-string `delivery.channel` values during cron normalization, and the planner can then fall back to `last`. I did not establish a live Telegram/Feishu reproduction from the reporter's exact stored job row.

## Solution Assessment

Yes, the best fix shape is parser-boundary normalization that recovers the canonical channel id before runtime routing. This issue should stay open until the linked PR or an equivalent fix merges with real delivery proof.
