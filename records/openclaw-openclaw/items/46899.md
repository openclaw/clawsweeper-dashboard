## Summary

Keep open: current main and the latest release still parse cron `delivery.channel` as string-only, so object-shaped persisted channel references can be dropped and delivery can fall back to implicit `last` routing; the linked fix PR is still open and should own the next step.

## What This Changes

Cron jobs configured for Telegram delivery can lose the explicit channel and route scheduled output through Feishu or other implicit last-channel state.

## Best Possible Solution

Refresh and prove the focused parser-boundary fix so cron normalization recovers persisted `{ id }` channel references while keeping runtime `delivery.channel` canonical as a string.

## Reproduction Assessment

Yes for the source-level path: current main drops non-string `delivery.channel` values during cron normalization, then announce planning can fall back to `last`. I did not establish a live Telegram/Feishu reproduction from the reporter's exact stored job.

## Solution Assessment

Yes, the linked parser-boundary repair is the narrow maintainable fix because it normalizes alternate persisted input before runtime and keeps the runtime type canonical. The issue should stay open until that PR is refreshed, proven, and merged or replaced.
