## Summary

Keep this issue open: current main still drops non-string cron `delivery.channel` values before delivery planning, the latest release has the same parser behavior, and the linked fix at https://github.com/openclaw/openclaw/pull/91162 is still open rather than merged.

## What This Changes

Cron jobs configured for Telegram delivery can route scheduled output through Feishu or last-channel state instead of honoring the configured Telegram channel.

## Best Possible Solution

Review, rebase, prove, and merge https://github.com/openclaw/openclaw/pull/91162 or an equivalent parser-boundary fix, then close this issue once the fix is on main.

## Reproduction Assessment

Yes for a source-level path: current main drops non-string persisted `delivery.channel` values, then announce routing can fall back to `last`. I did not establish a live Telegram/Feishu reproduction from the reporter's exact stored job row.

## Solution Assessment

Yes, the best current fix shape is to normalize the alternate channel reference at cron delivery ingress while keeping runtime `delivery.channel` canonical as a string. The linked PR targets that boundary but still needs real delivery proof and merge readiness.
