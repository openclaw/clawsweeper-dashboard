## Summary

Keep this open. Current main still has the lossy cron delivery path: non-string `delivery.channel` values are dropped during normalization/planning, so a persisted `{ id: "telegram" }` channel can become implicit `last` routing and select another channel such as Feishu; the direct fix PR was closed unmerged.

## What This Changes

The issue reports cron announce delivery configured for Telegram being routed through Feishu when the delivery channel is lost or mis-normalized.

## Best Possible Solution

Normalize or preserve `{ id: string }` channel values at the cron delivery boundary, or fail closed without falling back to `last`, with generic regression coverage and no Telegram- or Feishu-specific branching.

Yes. The narrow maintainable fix is cron-local delivery channel normalization or fail-closed rejection plus regression tests, not Telegram- or Feishu-specific routing logic.
