## Summary

Keep this open. Current main and the latest release still have the lossy cron delivery path: non-string `delivery.channel` values can be dropped during normalization/planning and then routed through `last` fallback, matching the reported Telegram-to-Feishu misroute; the direct fix PR at https://github.com/openclaw/openclaw/pull/46907 was closed unmerged.

## What This Changes

The issue asks for explicit Telegram cron delivery to stay on Telegram instead of falling back to Feishu when cron delivery channel data is not preserved correctly.

## Best Possible Solution

Preserve or reject invalid stored explicit `delivery.channel` values before fallback selection, with regression coverage proving an explicit channel cannot silently become `last`.

## Reproduction Assessment

Yes, at source level. A persisted job with `delivery.channel: { id: "telegram" }` goes through store load normalization, loses the channel, and then enters the `last` fallback path; I did not run a live Telegram/Feishu cron delivery.

## Solution Assessment

Yes. The narrow maintainable fix is generic cron delivery boundary normalization or fail-closed validation with regression tests, not Telegram- or Feishu-specific routing logic.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/46899. Persisted or loosely-created cron jobs can carry `delivery.channel` as an object such as `{ id: "telegram" }`; current cron delivery parsing/normalization drops that value and can fall back to `last`, which may send a Telegram target through another channel such as Feishu. Implement a generic cron-local fix: normalize supported object-shaped channel values wherever cron delivery channel strings are parsed, or fail closed with a clear validation error instead of deleting the explicit channel and falling back to `last`. Do not add Telegram- or Feishu-specific branching, do not add startup migrations, and keep legacy persisted repair in `openclaw doctor --fix` if persistent rewrites are needed. Add focused regression coverage for delivery planning, persisted job normalization/load, and target resolution so explicit channels cannot silently become fallback routes. Include a user-facing changelog entry for the bug fix.
