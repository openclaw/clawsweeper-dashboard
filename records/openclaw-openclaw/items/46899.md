## Summary

Keep this open. Current main still drops non-string cron `delivery.channel` values during parsing/hydration, so a persisted object-shaped Telegram channel can fall back to `last` routing and select another channel such as Feishu; the direct fix PR was closed unmerged.

## What This Changes

The issue reports cron announce delivery configured for Telegram being routed through Feishu after the delivery channel is lost.

## Best Possible Solution

Normalize or preserve `{ id: string }` channel values at the cron delivery boundary, or reject them without falling back to `last`, with generic regression coverage and no Telegram- or Feishu-specific routing.

## Reproduction Assessment

Yes, at source level: a persisted job with `delivery.channel` shaped as `{ id: "telegram" }` is rejected by the string-only parser, deleted during hydration, and then enters the `last` fallback path. I did not run a live scheduled Telegram/Feishu delivery in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is cron-local delivery channel normalization or fail-closed rejection plus regression tests, not channel-specific Telegram or Feishu branching.
