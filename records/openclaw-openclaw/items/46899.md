## Summary

Keep open: current main and the latest release still parse cron `delivery.channel` as a string-only field and delete rejected object-shaped channel values, while the only matching fix is the unmerged linked PR. Because that PR is still open and referenced as the closing fix, this issue is still necessary.

## What This Changes

The issue reports cron delivery configured for Telegram being routed through Feishu because the configured delivery channel is lost before runtime delivery planning.

## Best Possible Solution

Land PR https://github.com/openclaw/openclaw/pull/91162 or an equivalent focused parser-boundary fix, keeping runtime `delivery.channel` canonical as a string and proving multi-channel cron delivery routes to Telegram.

## Reproduction Assessment

Yes for the source-level path: current main drops non-string `delivery.channel` values during cron normalization, and the delivery planner can then fall back to `last`. I did not establish a live Telegram/Feishu reproduction from the reporter's exact stored job.

## Solution Assessment

Yes for the likely fix shape, but not solved yet: normalizing recoverable alternate input at the cron delivery parser boundary is the narrow maintainable path. The issue should remain open until the linked PR or an equivalent fix merges with proof.
