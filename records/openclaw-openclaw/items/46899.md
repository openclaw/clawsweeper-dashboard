## Summary

Keep open: current main still drops object-shaped `delivery.channel` values before resolving the cron delivery plan, so the closed unmerged related PR did not land the root fix.

## What This Changes

The issue asks cron announce delivery to honor Telegram as the configured channel instead of falling back to another channel such as Feishu.

## Best Possible Solution

Teach the current cron delivery planner to unwrap object-shaped channel IDs before falling back to `last`, with focused regression coverage and no channel-plugin-specific special case.

## Reproduction Assessment

Yes at source level. A focused test can call `resolveCronDeliveryPlan()` with `delivery.channel` shaped as `{ id: "telegram" }` and currently observes the planner losing the explicit Telegram channel before `last` fallback.

## Solution Assessment

Yes. The narrow maintainable fix is in cron delivery normalization/planning, not in the Telegram or Feishu plugins, and should preserve the existing strict create/update schema unless maintainers want broader input acceptance.
