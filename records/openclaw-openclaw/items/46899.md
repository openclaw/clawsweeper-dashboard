## Summary

Keep this open. Current main still uses string-only cron delivery channel parsing, so a persisted object-shaped channel like `{ id: "telegram" }` is dropped during delivery planning or store normalization and can fall back to `last`; the linked fix pull request was closed unmerged and no canonical replacement was found.

## What This Changes

Cron delivery should honor a job configured for Telegram instead of falling back to Feishu when persisted delivery data is object-shaped.

## Best Possible Solution

Accept `{ id: string }` channel values in cron-owned delivery parsing and persisted-job normalization, preserving existing string behavior and adding focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a persisted cron job with `delivery.channel` set to `{ id: "telegram" }`: current main normalizes that to undefined or deletes it, then downstream delivery treats the channel as `last`.

## Solution Assessment

Yes. The narrowest maintainable fix is to add cron-local compatibility parsing for `{ id: string }` in the runtime planner and persisted normalizer, without Telegram/Feishu special cases or a global string coercion change.
