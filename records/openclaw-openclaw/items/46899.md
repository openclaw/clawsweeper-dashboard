## Summary

Keep this open. Current main still preserves explicit string-shaped Telegram delivery, but it does not handle the object-shaped persisted `delivery.channel` value described by the linked fix attempt; that value is treated as non-string, then either falls back to `last` at delivery-plan resolution or is removed during persisted-job normalization.

## What This Changes

The issue asks OpenClaw to preserve a cron job's configured Telegram delivery channel instead of routing scheduled output through the previous Feishu channel when `delivery.channel` is malformed or persisted as an object.

## Best Possible Solution

Canonicalize object-shaped cron delivery channel values such as `{ id: "telegram" }` at the cron delivery boundary, preserve them through persisted-job normalization and store load, and cover both direct resolver behavior and persisted normalization with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to pass a cron job with `delivery.channel = { id: "telegram" }`: current resolver code normalizes that to undefined and announces through `last`, while persisted normalization deletes the channel before runtime use.

## Solution Assessment

Yes. The narrowest maintainable fix is cron-local compatibility parsing for `{ id: string }`, without Telegram/Feishu special cases or a broad global string-coercion change.

## Repair Work Prompt

Fix the cron delivery channel mismatch reported at https://github.com/openclaw/openclaw/issues/46899. Current main accepts string `delivery.channel` values, but object-shaped persisted values like `{ id: "telegram" }` are treated as non-strings: `resolveCronDeliveryPlan()` falls back to `last`, and persisted-job normalization can delete the channel on load.
