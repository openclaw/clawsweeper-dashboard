## Summary

Keep this open. Current main still drops object-shaped persisted `delivery.channel` values, so a Telegram cron job stored as `{ id: "telegram" }` can fall back to `last` and route through another configured channel; the related fix PR was closed without merging.

## What This Changes

The issue reports cron announce delivery using Feishu instead of Telegram when a scheduled job is configured for Telegram delivery.

## Best Possible Solution

Canonicalize object-shaped cron delivery channel values such as `{ id: string }` at the cron delivery boundary and persisted-job normalization, with regression coverage and no Telegram/Feishu special cases.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a cron job whose `delivery.channel` is `{ id: "telegram" }`: current main normalizes that to undefined, then either falls back to `last` during delivery planning or deletes it during persisted-job normalization.

## Solution Assessment

Yes. The narrowest maintainable fix is cron-local compatibility parsing for `{ id: string }`, preserving current string behavior without a global coercion change or provider-specific routing special case.
