## Summary

Keep open: the branch is useful and now has real proof for the initial process-hidden `exec` behavior, but it changes a documented tool-policy contract and still leaves quiet successful background completions with no `process` poll path unless the existing empty-success notification work lands or this branch handles it.

## What This Changes

The branch decouples `exec` auto-backgrounding from `process` tool visibility, adds hidden-process follow-up text and timeoutless rejection, updates exec docs, and adds process-denied regression tests.

## Best Possible Solution

Land this only after maintainers accept the process-hidden finite-timeout contract and the branch either makes quiet successful completions observable without `process` or is explicitly sequenced behind the existing empty-success notification work.

## Reproduction Assessment

Yes, source-level: current main sets exec backgrounding from the effective `process` tool policy and docs state that process-denied exec ignores `yieldMs`/`background`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. Decoupling the policy may be acceptable, but the implementation should preserve observable completion for quiet successes and needs maintainer acceptance of the new restricted-tool contract.

## Review Findings
