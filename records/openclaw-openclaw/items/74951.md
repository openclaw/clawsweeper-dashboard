## Summary

Keep open. The PR targets a real exec-turn blocking problem, but the submitted patch broadens process-denied background execution without guarding `timeout: 0`, leaves a contradictory background-process doc in place, and lacks the required changelog entry for a user-facing fix.

## What This Changes

The PR decouples exec auto-backgrounding from `process` tool visibility, adjusts hidden-process follow-up text/docs, and adds regression coverage for a long exec command when `process` is denied.

## Best Possible Solution

Land a revised PR only if maintainers accept process-hidden backgrounding: preserve or require a finite timeout when `process` is unavailable, reject timeoutless backgrounding in that mode, align both exec docs, add a changelog entry crediting the contributor, and keep focused regression coverage for both the allowed and rejected paths.

## Reproduction Assessment

Yes. The focused reproduction is `createOpenClawCodingTools` with `tools.deny: ["process"]`, then a long `exec` call with a short `yieldMs`; the unsafe variant adds `timeout: 0`, which current runtime code and existing tests show removes the supervisor timeout.

## Solution Assessment

No, not as submitted. Decoupling backgrounding from `process` visibility may be acceptable, but the maintainable version needs a finite-timeout guard or rejection path before returning an unmanaged background session without the `process` tool.
