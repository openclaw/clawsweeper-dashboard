## Summary

Keep open. The PR addresses a real long-running exec pain point, but the submitted behavior broadens process-denied exec backgrounding without preserving a finite lifecycle guard, and it leaves the background-process docs contradictory.

## What This Changes

The PR decouples exec auto-backgrounding from `process` tool availability, adds hidden-process follow-up messaging, updates the exec docs, and adds regression coverage for a long process-denied exec call.

## Best Possible Solution

Land a corrected version only if maintainers accept the new process-denied contract: keep the clearer hidden-process messaging, preserve or require a finite timeout when `process` is unavailable, and update both exec/background-process docs plus regression coverage for that guard.

## Reproduction Assessment

Yes. Use `createOpenClawCodingTools` with `tools.deny: ["process"]`, then call `exec` with a long command and small `yieldMs`; current main runs synchronously, while the PR expects `status: "running"`. The unsafe variant is the same setup with `timeout: 0`, which current runtime/tests prove disables the supervisor timeout.

## Solution Assessment

No, not as submitted. Decoupling backgrounding from process visibility may be acceptable, but the maintainable fix should not let process-denied background sessions opt out of the only remaining automatic stop.
