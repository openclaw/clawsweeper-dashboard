## Summary

Keep open: the PR addresses a plausible remaining long-running exec path, but it changes a documented restricted-tool contract and still has a blocking completion-visibility issue for quiet successful commands when `process` is hidden.

## What This Changes

The PR decouples finite-timeout exec backgrounding from `process` tool visibility, adds hidden-process follow-up wording and timeoutless rejection, updates docs, and adds process-denied regression tests.

## Best Possible Solution

Land this only after maintainers accept the restricted-tool policy and the branch guarantees an observable completion path for process-hidden background sessions or rejects cases that cannot be observed.

## Reproduction Assessment

Yes, source-level: current main derives exec backgrounding from `process` visibility and current docs/tests lock process-denied exec as synchronous despite `yieldMs`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. Decoupling finite-timeout backgrounding may be acceptable, but the patch should preserve quiet-success completion visibility without `process` and needs maintainer approval for the new restricted-tool contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve quiet completion visibility without process:** `src/agents/pi-tools.ts:707`
  - body: This enables process-hidden exec backgrounding while the returned text tells the model to rely on completion wakes or timeout. A quiet command can finish successfully before timeout with no output, and current notification code suppresses that wake unless `notifyOnExitEmptySuccess` is true; because `process` is hidden, there is no poll path to confirm completion. Force an observable empty-success notification for this mode, reject it, or sequence this behind the empty-success notification work.
  - confidence: 0.9
