## Summary

Keep open: current main and the latest release still have the fixed Codex native-thread startup guard and compaction invalidation behavior, while the semantic ownership, lifecycle diagnostics, and preservation work is still in open stacked PRs that need maintainer sequencing.

## What This Changes

The issue requests semantic ownership for long-running Codex app-server native thread/bootstrap cache reuse so compatible context-engine sessions do not rotate solely on a fixed native-token guard.

## Best Possible Solution

Review and land or supersede the stacked Codex native-thread lifecycle PRs, then split any remaining `/doctor` or status UX into a narrower follow-up once the semantic preservation behavior is settled.

## Reproduction Assessment

Yes for a source-level reproduction of the motivating behavior: current main still rotates startup bindings at a fixed 70k native/session token threshold before active context-engine setup, and compaction still clears bindings. No high-confidence live Discord/Codex slowdown reproduction was established in this read-only pass.

## Solution Assessment

Unclear as one monolithic change: the semantic ownership direction matches the existing `thread_bootstrap` contract, but the safer path is staged maintainer-reviewed PRs for guard ordering, configurability, diagnostics, and compaction/rollover preservation.
