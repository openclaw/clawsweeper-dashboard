## Summary

Keep open: current main has the semantic thread-bootstrap reuse foundation and a model-aware token fuse, but the issue's central request is not complete. Token-pressured thread_bootstrap bindings can still cold-start, context-engine native compaction still clears projection state instead of preserving compatible bindings, and the remaining diagnostics/preservation work is in open conflicting PRs that need maintainer sequencing.

## What This Changes

Request semantic Codex native-thread ownership so compatible long-running thread_bootstrap sessions reuse bindings across token pressure, rollover, compaction, and diagnostics instead of cold-starting on blunt native-token rotation.

## Best Possible Solution

Review and sequence the open lifecycle diagnostics and semantic-preservation PRs, then keep the final design to Codex-supported semantic binding preservation/status diagnostics without bypassing real app-server overflow or stale-thread rejection.

## Reproduction Assessment

Yes for the remaining source-level behavior: current main has a test that expects a token-pressured thread_bootstrap binding to start a fresh thread, and compaction code clears projection state before native compaction; no live Discord/Codex slowdown reproduction was run.

## Solution Assessment

Unclear as one broad implementation: semantic ownership matches the thread_bootstrap contract, but compaction/rollover preservation and diagnostics touch Codex app-server lifecycle and session state, so the best path is maintainer review of the open PRs rather than a new automated fix PR.
