## Summary

Keep open: current main and the latest release still have the unresolved source-level behavior, while the direct diagnostics and semantic-preservation PRs are open and conflicting, so no close reason is satisfied.

## What This Changes

The issue asks OpenClaw to treat Codex app-server native threads as semantic projection caches for long-running context-engine sessions, with rotation, diagnostics, compaction, and rollover keyed by projection ownership rather than a hard native-token threshold.

## Best Possible Solution

Review and sequence the open lifecycle diagnostics and semantic-preservation PRs, then keep the final design to Codex-supported semantic binding preservation and status diagnostics without bypassing real app-server overflow or stale-thread rejection.

## Reproduction Assessment

Yes for source-level reproduction: current main has a test expecting a token-pressured `thread_bootstrap` binding to cold-start, and the compaction code clears projection metadata before native compaction. I did not run a live long-running Discord/Codex reproduction in this read-only review.

## Solution Assessment

Unclear as one broad implementation: semantic ownership matches the `thread_bootstrap` contract, but compaction/rollover preservation and lifecycle diagnostics touch session-state and privacy boundaries, so maintainer sequencing of the open PRs is the best path.
