## Summary

Keep this open: current main still lacks a supported Docker setup offline rerun mode. The reporter’s linked patch is useful workflow evidence, but the core script still builds or pulls before the setup steps airgapped operators need to reuse.

## What This Changes

Request an explicit `--offline` flag for `scripts/docker/setup.sh` so airgapped hosts can rerun setup with a preloaded Docker image.

## Best Possible Solution

Add a Bash-compatible opt-in offline mode to Docker setup, document it on the Docker install page, and cover the main-image-present, main-image-missing, and sandbox warning cases in the existing Docker setup e2e harness.

## Reproduction Assessment

Yes. Static source inspection on current main is enough: there is no `--offline` argument path, and the script still reaches build or pull before the supported rerun steps an airgapped operator wants to reuse.

## Solution Assessment

Yes. An explicit opt-in flag is the narrowest maintainable direction because it preserves default online behavior while allowing a preloaded image to continue through the existing setup routine.

## Repair Work Prompt

Implement the requested `--offline` mode for `scripts/docker/setup.sh` as a focused Docker setup feature. Parse a Bash 3.2-compatible `--offline` flag near the top of the script without changing default behavior. When offline mode is set, skip the main image build/pull and require `docker image inspect "$IMAGE_NAME"` to succeed, with a clear error pointing operators to `docker load` if the image is missing. Keep the rest of setup unchanged: env validation, `.env` upserts, data-dir permission fix-up, onboarding unless skipped, gateway config sync, compose startup, sandbox overlay/config handling. For `OPENCLAW_SANDBOX`, skip sandbox image build in offline mode and warn if `openclaw-sandbox:bookworm-slim` is missing rather than failing, while avoiding a silently broken enabled sandbox state. Add focused coverage in `src/docker-setup.e2e.test.ts`, update `docs/install/docker.md`, and add a user-facing changelog entry if this proceeds as maintainer-owned landing work. Do not change Podman setup or revive `OPENCLAW_DOCKER_PLATFORM` from the separate closed cross-platform thread.
