## Summary

Keep this open: current main still has no supported offline Docker setup rerun mode, and the request remains a focused opt-in feature rather than an obsolete or duplicated item.

## What This Changes

Add an opt-in `--offline` mode to `scripts/docker/setup.sh` so airgapped operators can reuse preloaded Docker images while still running setup, env sync, permissions, onboarding, gateway config, compose startup, and sandbox handling.

## Best Possible Solution

Add a Bash 3.2-compatible `--offline` flag that preserves default online behavior, requires the main image via `docker image inspect`, warns for a missing optional sandbox image, and documents/tests the rerun flow.

## Reproduction Assessment

Yes. Static source inspection on current main shows no `--offline` argument path, and the setup script still reaches `docker build` or `docker pull` before the rerun steps the reporter needs.

## Solution Assessment

Yes. An explicit opt-in flag is the narrow maintainable shape, but the linked external patch should be treated as workflow evidence rather than copied directly because it changes default local-build and compose-startup semantics.

For https://github.com/openclaw/openclaw/issues/70443, implement a narrow `--offline` mode in `scripts/docker/setup.sh`. Parse the flag in a Bash 3.2-compatible way, preserve byte-for-byte-equivalent default online behavior, skip main image build/pull only when the flag is passed, require `docker image inspect "$OPENCLAW_IMAGE"` to succeed with a clear `docker load`-oriented error when missing, and for `OPENCLAW_SANDBOX` skip the sandbox image build in offline mode with a warning if `openclaw-sandbox:bookworm-slim` is absent. Do not copy the linked workaround as-is: do not make offline behavior implicit for `openclaw:local`, do not add `--force-recreate`, and do not introduce an env-var alias unless maintainers explicitly request it. Update `src/docker-setup.e2e.test.ts` to pass setup args and cover default online behavior, offline preloaded main image success, missing main image failure, and optional sandbox warning behavior; update `docs/install/docker.md` and add a single-line `CHANGELOG.md` Changes entry for the new Docker setup flag.
