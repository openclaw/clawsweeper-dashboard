## Summary

Keep open: current main still has source-reproducible default-enabled cron creation, agent/default model inheritance for cron jobs without `payload.model`, and only reason-agnostic recurring-job backoff. The related open items cover narrower cron defaults, guardrail, and provenance slices, but they do not fully supersede this broader consent/cost/quota policy issue.

## What This Changes

The issue requests safeguards so externally registered recurring cron jobs do not auto-enable, do not silently inherit costly default models, and can be paused by quota or cost guardrails.

## Best Possible Solution

Approve a cron registration and provenance contract that separates operator-created jobs from external/agent/plugin registrations, then implement opt-in activation, cron-specific model/default policy, doctor/update visibility, reason-aware guardrails, docs, and regression coverage together.

## Reproduction Assessment

Yes for the source-level current-main path: cron add/normalization and service creation still make omitted `enabled` active, and isolated agent-turn cron model selection still falls through to agent/default model selection when `payload.model` is absent.
