## Label Justifications

- `P1`: The issue describes background scheduled jobs consuming expensive provider/model quota without explicit user consent, and current source still permits that path.
- `impact:auth-provider`: The affected behavior centers model/provider selection and quota usage for scheduled background agent turns.

## Summary

Keep open: current main and the latest release still allow cron jobs from API/service/plugin paths to become active by default, and plugin scheduled turns can create agent-turn cron jobs without a per-job model or operator consent boundary.

## What This Changes

The issue requests explicit activation, update/doctor visibility, model-cost guardrails, and quota/circuit-breaker behavior for externally registered cron jobs.

## Best Possible Solution

Define a cron registration contract that distinguishes operator-created jobs from external, plugin, agent, or synced jobs, then implement opt-in activation, cron-specific model policy, doctor/update visibility, quota guardrails, docs, and regression coverage together.

## Reproduction Assessment

Yes for the source-level path: omitted enabled still defaults active, plugin scheduled turns create enabled agent-turn jobs, and missing payload.model falls through to agent/default model selection. I did not live-reproduce the reported quota burn or the separate interactive usage regression.
