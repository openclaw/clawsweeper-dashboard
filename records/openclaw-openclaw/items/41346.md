## Summary

Keep open. Current main and the latest release still allow externally/plugin-created cron jobs to be enabled immediately, still let agent-turn cron jobs without an explicit model resolve through agent/default model selection, and still lack a quota/usage circuit-breaker policy for recurring background work.

## What This Changes

The issue asks OpenClaw to require explicit activation and cost/usage guardrails for externally registered recurring cron jobs that can inherit expensive default models.

## Best Possible Solution

Land a maintainer-approved cron safety design that records registration provenance, differentiates operator versus external or agent-created schedules, gates activation/model defaults, and adds quota-aware pause/notification with migration, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still defaults omitted cron enabled state to true, plugin scheduled turns pass enabled true, and agentTurn jobs without payload.model resolve through configured defaults; I did not run a live quota-burning scenario in this read-only review.

## Solution Assessment

Unclear as a direct implementation. The requested safeguards are directionally aligned, but maintainers need to choose the product/security boundary between operator cron, plugin or agent scheduling provenance, model-cost policy, and usage circuit-breaking.
