## Summary

Keep open: current `main` still has the reported core behavior, and the requested safeguards require cron API/product policy decisions rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to add safeguards so externally registered recurring cron jobs do not silently auto-enable, inherit expensive default models, or run without usage guardrails.

## Best Possible Solution

Define a source-aware cron registration policy, then add compatible confirmation or disabled-by-default behavior for external jobs plus explicit model and usage guardrails.

## Reproduction Assessment

Yes for the core current behavior by source inspection: cron add normalization and job creation default missing `enabled` to true, and isolated cron uses the agent/default model when no per-job model is set. I did not establish a live runaway-usage reproduction in this read-only review.

## Solution Assessment

Unclear as one patch: the requested safeguards span registration identity, update/doctor UX, model-cost policy, and quota thresholds. The maintainable direction is to split source-aware cron enablement/confirmation from model-cost and usage circuit-breaker enforcement.
