## Summary

Keep open: current main still defaults cron additions without `enabled` to enabled and falls back to the agent/default model when no job model is supplied; existing guards cover errors/timeouts, not confirmation or quota/cost circuit breaking.

## What This Changes

Request externally registered recurring cron jobs to default disabled, require confirmation/dry-run visibility, avoid silently inheriting expensive default models, and pause on excessive usage.

## Best Possible Solution

Define a source-aware cron registration policy, then add compatible confirmation/disabled-by-default behavior plus model and usage guardrails without breaking explicit user-created reminders.

## Reproduction Assessment

Yes for the core current behavior by source inspection: cron add normalization and job creation default missing `enabled` to true, and isolated cron uses the agent/default model when no per-job model is set. I did not establish a live runaway-usage reproduction in this read-only review.

## Solution Assessment

Unclear as a single fix: the requested safeguards are directionally reasonable but span API policy, doctor/update UX, model-cost classification, and quota thresholds. The maintainable path is to split this into a source-aware cron default/confirmation design and separate cost/usage enforcement work.
