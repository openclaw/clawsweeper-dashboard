## Summary

Keep open. Current main and v2026.6.6 still support the reported source-level behavior: externally/plugin/agent-created cron jobs can be enabled immediately, agentTurn cron jobs without an explicit model still fall through to agent/default model selection, and recurring cron still lacks a usage/quota circuit-breaker policy.

## What This Changes

The issue asks OpenClaw to add consent, model-cost, dry-run, and usage circuit-breaker guardrails for externally registered recurring cron jobs.

## Best Possible Solution

Define one cron safety policy for external/plugin/agent registration provenance, default activation, explicit model or cost warnings, doctor dry-run visibility, and quota-aware pause/notification behavior, then land it with docs, protocol/state updates, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 still default omitted cron enabled to true, plugin scheduled turns pass enabled true, agent-facing cron docs say enabled defaults true, and agentTurn jobs without payload.model resolve through agent/default model selection; I did not run a live quota-burning scenario.

## Solution Assessment

Unclear as a single implementation. The requested safeguards are directionally aligned, but maintainers need to choose the product/security boundary between operator cron, plugin or agent scheduling provenance, model-cost policy, and usage circuit-breaking.
