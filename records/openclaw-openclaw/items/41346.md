## Summary

Keep open: current main still treats omitted cron `enabled` as active and still resolves agent-turn cron jobs without `payload.model` through agent/default model selection. The remaining work is broader than the linked cron-defaults tracker because it also needs external-registration consent, doctor/update visibility, model-cost policy, and usage guardrails.

## What This Changes

Request external/plugin-registered cron jobs to require explicit activation, avoid silent expensive default-model inheritance, surface doctor/update dry-run diffs, and add usage circuit breakers.

## Best Possible Solution

Define a source-aware cron registration contract that separates user-created from external/plugin-created jobs, then add explicit activation, cron-scoped model/default or warning policy, doctor/update dry-run visibility, quota guardrails, docs, and regression coverage together.

## Reproduction Assessment

Yes for the core current behavior by source inspection: omitted cron `enabled` still becomes true, and omitted cron `payload.model` still resolves through the agent/default model path. I did not reproduce the reported overnight quota burn in a live provider setup during this read-only review.

## Solution Assessment

Unclear as an automatic patch: the requested safeguards change public cron registration semantics, CLI/doctor UX, config schema, model-cost policy, and quota enforcement. The maintainable solution needs maintainer approval and should reconcile the overlapping cron-default tracker before implementation.
