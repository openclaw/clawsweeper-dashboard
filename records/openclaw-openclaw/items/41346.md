## Summary

Keep open. Current main and the latest release still have the central source-level behavior: cron creates default to enabled, plugin scheduled turns explicitly register enabled agentTurn jobs, agentTurn jobs without a per-job model still resolve through agent/default model selection, and recurring cron has generic backoff/alerts rather than a consent, provenance, cost, or quota circuit-breaker policy.

## What This Changes

The issue requests guardrails so externally or plugin registered cron jobs do not silently enable recurring model-backed work, inherit expensive defaults, or continue without usage protection.

## Best Possible Solution

Define one cron safety policy for external/plugin/agent registration provenance, default activation, explicit model or cost warnings, doctor/update visibility, and quota-aware pause/notification behavior, then land it with protocol, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: omitted cron `enabled` still defaults to true, plugin scheduled turns pass `enabled: true`, and agentTurn jobs without payload.model still resolve through default model selection. I did not run a live quota-burning scenario because this is a read-only review and the costly symptom depends on real provider usage.

## Solution Assessment

Unclear as a single implementation. The requested safeguards are directionally aligned, but maintainers need to choose the product/security boundary between operator cron, plugin or agent scheduling provenance, model-cost policy, and quota circuit-breaking.
