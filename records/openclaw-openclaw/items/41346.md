## Summary

Keep open: current main still treats newly created cron jobs as enabled by default and isolated cron jobs without an explicit payload model still resolve through agent/default model selection. The exact bundled Antfarm-era report is not present in current source, and current memory-core dreaming is opt-in, but the requested external-registration safeguards remain unresolved core/API policy work.

## What This Changes

The issue requests safer defaults and guardrails for externally registered polling cron jobs, including disabled-by-default activation, update/doctor diffs, model cost warnings, dry-run summaries, and usage circuit breakers.

## Best Possible Solution

Define a provenance-aware cron registration policy for plugin/external jobs, with pending or disabled-by-default activation, explicit model/cost summaries, and a documented quota/usage guard that preserves current explicit user-created cron behavior where appropriate.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call cron.add or normalizeCronJobCreate without enabled and without payload.model for an isolated agentTurn job, and current main produces an enabled job that resolves through agent/default model selection. I did not run a live gateway scenario because this review was constrained to read-only inspection.

## Solution Assessment

No, this is not a safe narrow bug-fix automation item. The maintainable fix needs a backward-compatible cron/plugin API design for external-job provenance, activation UX, and model/usage policy rather than directly flipping the global createJob default.
