## Label Justifications

- `P1`: Persistent background cron jobs can immediately consume expensive provider quota without confirmation, which is an urgent real-user workflow and cost regression.
- `impact:security`: The issue concerns persistent scheduled automation without explicit consent or trustworthy provenance.
- `impact:auth-provider`: The reported failure depends on cron agent turns inheriting expensive default provider/model selection.

## Summary

Keep open. Current main and v2026.6.8 still have the central behavior: cron creates default-enabled jobs, plugin scheduled turns explicitly register enabled agentTurn jobs without a per-job model, model selection falls back to configured defaults, and recurring cron uses generic backoff rather than consent, provenance, cost, or quota circuit-breaker policy.

## What This Changes

Request guardrails so external, plugin, or agent-created cron jobs do not auto-enable, silently inherit expensive model defaults, or run without usage and quota circuit breakers.

## Best Possible Solution

Define one cron safety policy for external, plugin, and agent registration provenance, default activation, explicit model or cost warnings, doctor/update visibility, and quota-aware pause/notification behavior, then land protocol, docs, and tests together.

## Reproduction Assessment

Yes at source level: omitted enabled defaults to true, plugin scheduled turns pass enabled true, and agentTurn jobs without payload.model resolve through configured defaults.

## Solution Assessment

Unclear as a single implementation; the safeguards are aligned, but maintainers need to choose the boundary across operator cron, plugin and agent scheduling provenance, model-cost policy, and quota circuit breaking.
