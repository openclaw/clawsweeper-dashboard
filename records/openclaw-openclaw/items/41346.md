## Summary

Keep open. Current main and v2026.6.8 still allow cron jobs to be created enabled by default, plugin scheduled turns explicitly create enabled agentTurn cron jobs, missing per-job models resolve through agent/default model selection, and recurring cron still lacks the requested consent/provenance/cost/quota guard policy.

## What This Changes

The issue requests safer cron defaults and guardrails for externally registered polling jobs, including explicit activation, update/doctor visibility, model-cost checks, and usage circuit breaking.

## Best Possible Solution

Define one core cron safety policy for external/plugin/agent registration provenance, default activation, explicit model or cost warnings, doctor/update visibility, and quota-aware pause/notification behavior, then land it with protocol, docs, migration, and regression coverage.

## Reproduction Assessment

Yes, at source level: omitted cron enabled still defaults to true, plugin scheduled turns pass enabled: true, and agentTurn jobs without payload.model still resolve through default model selection. I did not run a live quota-burning scenario because this is a read-only review and the costly symptom depends on real provider usage.

## Solution Assessment

Unclear as a single implementation. The requested safeguards are directionally aligned, but maintainers need to choose the product/security boundary between operator cron, plugin or agent scheduling provenance, model-cost policy, and quota circuit-breaking.
