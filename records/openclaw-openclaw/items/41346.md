## Summary

Keep open: current main and the latest release still allow externally or plugin-created cron jobs to become active by default, and agent-turn cron jobs without a per-job model still resolve through agent/default model selection. The remaining fix is policy-sensitive because consent, provenance, cron-specific defaults, and quota/circuit-break behavior all change public scheduling semantics.

## What This Changes

The issue requests safer cron registration defaults, explicit activation and update visibility, model-cost guardrails, and usage circuit breakers for externally registered polling jobs.

## Best Possible Solution

Approve a cron registration and provenance contract that separates operator-created jobs from external, agent, plugin, or synced registrations, then implement opt-in activation, cron-specific model/default policy, doctor/update visibility, reason-aware guardrails, docs, and regression coverage together.

## Reproduction Assessment

Yes for the source-level path: cron add and service creation default omitted `enabled` to active, and isolated agent-turn model selection falls through to agent/default selection when no per-job model is present.
