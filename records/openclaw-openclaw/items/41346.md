## Summary

Keep open: current main and the latest release still default newly created cron jobs to enabled and still let isolated agent-turn cron jobs fall through to agent/default model selection when the job has no payload model. The issue also has protected security/product-review signals and overlaps open scheduling/circuit-breaker RFCs, so it is not conservative cleanup-closeable.

## What This Changes

The issue requests safer defaults and guardrails for externally registered recurring cron jobs that can auto-run on expensive default models without confirmation or usage limits.

## Best Possible Solution

Define a maintainer-approved cron safety contract: provenance-aware externally or agent-created jobs require explicit activation/confirmation, model/cost defaults are visible and bounded, and circuit breakers coordinate with existing retry and provider-cooldown policy.

## Reproduction Assessment

Yes, for the central source path: current main normalizes omitted cron `enabled` to true, `createJob` also defaults it to true, and isolated agent-turn model selection starts from agent/default model selection when `payload.model` is absent. I did not run a live quota-burn scenario because this cleanup review is read-only.

## Solution Assessment

No, not as a single autonomous fix: the requested direction is valid, but the best solution needs a maintainer security/product decision across cron API defaults, provenance, confirmation UX, migration behavior, and cost circuit-breaker semantics.
