## Summary

Keep open: current main still has the core mechanics reported, because cron creations default to enabled and isolated agent-turn jobs without a per-job model resolve through agent/default model selection. The requested consent, dry-run, model-cost, and quota circuit-breaker behavior is not implemented and needs maintainer product/security direction.

## What This Changes

Request safer handling for externally registered recurring cron jobs: default disabled or confirmed, explicit model/cost guard, dry-run visibility, and quota circuit breaker.

## Best Possible Solution

Define a cron registration safety policy for plugin/external recurring agent-turn jobs, then implement the chosen consent/default-disabled/model-cost/quota guard with focused regression tests and docs.

## Reproduction Assessment

Yes for the core source path: creating a recurring agentTurn cron job without enabled/model still normalizes to enabled true and resolves through agent/default model selection. The live quota-exhaustion impact was not reproduced against a provider session in this read-only review.

## Solution Assessment

Unclear; the requested safeguards are directionally appropriate, but they bundle several policy choices across cron defaults, plugin trust, doctor/update UX, model-cost taxonomy, and quota enforcement. A maintainer should choose the supported contract before implementation.
