## Label Justifications

- `P1`: The report describes a real user-facing runaway cron cost/quota failure caused by background jobs starting without explicit consent or cost guardrails.
- `impact:security`: The issue is about persistent externally registered scheduled work running without clear user confirmation, provenance, or guardrails.

## Summary

Keep open. Current main and the latest release still create cron jobs as enabled unless callers explicitly disable them, isolated agent-turn crons still fall through to agent/default model selection when no payload model is set, and recurring failures still use generic retry/backoff rather than a durable reason-aware circuit breaker.

## What This Changes

Request safer handling for externally registered cron jobs: default disabled or confirmed creation, explicit model and cost safeguards, update/doctor diffing, and usage circuit breakers.

## Best Possible Solution

Keep this open as a maintainer-owned cron safety design and implement provenance or confirmation for externally registered jobs, explicit model/cost policy, durable guard state, client-visible schema, docs, and regression coverage together.

## Reproduction Assessment

Yes, at source level. Current main shows omitted cron enabled defaults to true, omitted cron payload models fall through to agent/default model selection, and recurring failures use generic backoff without durable circuit-breaker state; I did not run a live expensive-provider reproduction.

## Solution Assessment

No, the current implementation is only partial mitigation. The best fix needs a maintainer-approved cron safety policy across registration provenance/confirmation, model cost defaults, update/doctor visibility, durable guard state, schema/docs, and tests.
