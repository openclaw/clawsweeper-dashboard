## Summary

Keep open. Current main and v2026.6.10 still allow create-time cron jobs to default enabled, plugin scheduled turns explicitly register enabled agentTurn jobs, missing cron payload models resolve through agent/default model selection, and related open issues cover only parts of the consent, provenance, cost, and quota guardrail problem.

## What This Changes

The issue requests confirmation/default-disabled behavior for externally registered cron jobs, explicit model/cost safeguards for recurring agentTurn jobs, and usage circuit breakers for runaway scheduled automation.

## Best Possible Solution

Define one core cron safety policy for external/plugin/agent registration provenance, default activation, explicit model or cost warnings, doctor/update visibility, and quota-aware pause/notification behavior, then land it with protocol, docs, migration, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still defaults omitted cron enabled to true, plugin scheduled turns pass enabled: true, and agentTurn jobs without payload.model can resolve through agent/default model selection; I did not run a live provider quota-burning scenario.

## Solution Assessment

Unclear as a single implementation. The requested safeguards are directionally aligned, but maintainers need to choose the security/product boundary across operator cron, plugin or agent scheduling provenance, model-cost policy, and quota circuit-breaking.
