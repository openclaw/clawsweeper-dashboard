## Summary

Keep open: current main still defaults newly added cron jobs without `enabled` to enabled and still starts isolated cron model selection from the configured default model, while the requested safeguards require cron API/product policy rather than conservative cleanup.

## What This Changes

The issue requests disabled-by-default external cron registration, confirmation/dry-run visibility, explicit model or cost warnings, and usage circuit breakers for recurring polling jobs.

## Best Possible Solution

Define a source-aware cron registration policy that preserves intentional user-created jobs while requiring explicit activation, model choice, and quota-aware guardrails for externally registered recurring agent-turn jobs.

## Reproduction Assessment

Yes for the core current behavior by source inspection: `cron.add` normalizes omitted `enabled` to true, `createJob` repeats that fallback, and isolated cron starts from the configured default model unless an override applies. I did not live-run the quota-burn scenario in this read-only review.

## Solution Assessment

Unclear as one direct fix: the requested direction is reasonable, but it spans external registration identity, compatibility, doctor/update UX, model-cost classification, and provider quota thresholds. A maintainable solution should split policy design from narrow implementation slices.
