## Summary

Keep open: current main still defaults omitted cron `enabled` to active, lets agent-turn cron jobs without `payload.model` fall through to agent/default model selection, and only applies generic recurring-job error backoff rather than the requested consent, model-cost, or quota circuit-breaker policy. The related open issues cover narrower model-default, guardrail, and provenance slices, but they do not fully supersede this broader consent and cost-control issue.

## What This Changes

The issue asks OpenClaw to prevent externally registered recurring cron jobs from auto-enabling, inheriting expensive default models, and running without cost or quota guardrails.

## Best Possible Solution

Approve a cron registration and provenance contract that separates operator-created jobs from external/agent/plugin registrations, then implement opt-in activation, cron-specific model/default policy, doctor/update visibility, reason-aware guardrails, docs, and regression coverage together.

Yes for the source-level current-main path: cron add/normalization and service creation still make omitted `enabled` active, and isolated agent-turn cron model selection still falls through to agent/default model selection when `payload.model` is absent.
