## Label Justifications

- `P1`: The source-reproducible cron behavior can create persistent automated model calls with expensive defaults and no user-facing guardrail, affecting real background workflows.
- `impact:security`: The issue concerns unconfirmed externally registered scheduled work and provenance/activation boundaries for persistent automation.
- `impact:auth-provider`: The reported failure mode depends on model choice, quota exhaustion, provider usage, and default model inheritance.

## Summary

Keep open. Current main and the latest release still create cron jobs as enabled by default, bundled scheduled-turn hooks explicitly register enabled agentTurn jobs, and cron agentTurn model selection still falls back to the agent/default model when no per-job model is set; related open issues cover parts of the safety story but not this whole consent/model/circuit-breaker problem.

## What This Changes

The issue requests safer handling for externally registered cron jobs: disabled-by-default or confirmation before activation, explicit low-cost model policy, dry-run/update visibility, and usage circuit breakers.

## Best Possible Solution

Keep this issue open until maintainers define and implement one coherent cron safety policy for external/agent/plugin registration provenance, activation consent, explicit model selection, and quota/cost guardrails with docs and regression coverage.

## Reproduction Assessment

Yes at source level. Current main defaults missing cron enabled to true, plugin scheduled turns pass enabled true, and agentTurn cron runs without payload.model resolve through agent/default model selection; I did not run a live quota-burning scenario.

## Solution Assessment

Unclear. The requested safeguards are directionally aligned, but the best fix likely needs a maintainer-chosen split between operator cron, agent/plugin scheduling provenance, default model policy, and reason/usage circuit-breaking rather than a single unconditional default flip.
