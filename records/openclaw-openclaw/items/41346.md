## Label Justifications

- `P1`: Latest shipped behavior permits recurring unattended inference that can rapidly consume a real user's paid model quota.
- `impact:security`: Persistent jobs share a creation contract that lacks a trustworthy operator-versus-agent or external creator distinction.
- `impact:auth-provider`: Jobs without an explicit payload model inherit configured provider and model defaults, directly controlling the reported cost and quota impact.

## Summary

Keep open: current main and the latest v2026.6.5 release still enable cron jobs when creation omits `enabled` and still begin isolated cron model selection from the configured agent default. The requested remedy crosses activation compatibility, trustworthy creator provenance, model-cost policy, doctor/update UX, and quota enforcement, so it needs explicit maintainer security and product decisions rather than stale cleanup.

## What This Changes

Add explicit activation, provenance, model-cost, update visibility, and usage safeguards for externally created recurring inference jobs.

## Best Possible Solution

Add gateway-owned, non-forgeable creator provenance first, then define an upgrade-safe activation policy for non-operator recurring jobs; coordinate cron model defaults with https://github.com/openclaw/openclaw/issues/57638 and provenance/security semantics with https://github.com/openclaw/openclaw/issues/71712, while treating quota enforcement as a separately approved usage-policy surface.

## Reproduction Assessment

Yes at source level: create a recurring `agentTurn` job without `enabled` or `payload.model`; current main enables it and begins model selection from the configured agent default. The exact historical integration sync can vary, but the underlying unsafe default path is clear and is also present in v2026.6.5.

## Solution Assessment

No, not as one undifferentiated implementation: changing every cron to disabled would break operator workflows, while cost estimation and quota breaking introduce separate policy. The narrowest maintainable first step is trusted creator provenance followed by an explicit, compatibility-reviewed activation rule for non-operator recurring jobs.
