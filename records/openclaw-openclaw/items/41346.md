## Summary

Keep open: current main still enables omitted cron `enabled` fields by default, still resolves omitted cron models through agent/default model selection, and still lacks the requested external-registration confirmation, cron-scoped cost policy, or usage circuit breaker. The related open cron-default issues cover parts of the model/delivery/defaults space but do not fully supersede the broader consent and runaway-usage policy here.

## What This Changes

Request source-aware safeguards so externally registered polling cron jobs do not auto-enable and spend expensive inherited model quota without explicit user consent or cost limits.

## Best Possible Solution

Define a source-aware cron registration policy that separates user-created jobs from external/plugin-created jobs, then implement explicit enablement/confirmation, cron-scoped model defaults or warnings, and gateway quota guardrails under one documented contract.

## Reproduction Assessment

Yes for the core current behavior by source inspection: omitted cron `enabled` still becomes true, and omitted cron `payload.model` still resolves through the agent/default model path. I did not reproduce the reported overnight quota burn in a live provider setup during this read-only review.

## Solution Assessment

Unclear as a single automatic fix: the requested safeguards change public cron registration semantics, CLI/doctor UX, config schema, model-cost policy, and quota enforcement. The maintainable solution needs maintainer approval and should reconcile the overlapping cron-default trackers before implementation.
