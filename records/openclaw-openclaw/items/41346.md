## Summary

Keep open: current main still has the reported default-active and default-model cron behavior, and the remaining work needs maintainer product/API design rather than cleanup closure. The narrower related issues cover cron defaults and scheduling provenance pieces, but they do not fully supersede external-registration consent, update/doctor visibility, model-cost policy, and usage guardrails.

## What This Changes

The issue asks OpenClaw to prevent externally registered polling cron jobs from silently enabling expensive model usage without confirmation, model policy, dry-run visibility, or circuit breakers.

## Best Possible Solution

Approve a source-aware cron registration contract that separates operator-created jobs from external/plugin/agent-created jobs, then add explicit activation, cron-scoped model/default or warning policy, doctor/update dry-run visibility, quota guardrails, docs, and regression coverage together.

## Reproduction Assessment

Yes for the core current behavior by source inspection: omitted cron `enabled` still becomes active, and omitted cron `payload.model` still resolves through agent/default model selection. I did not live-reproduce the overnight quota burn or the broader v2026.3.7 session-token regression.
