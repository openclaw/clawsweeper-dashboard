## Summary

Keep open: current main still auto-enables cron jobs when `enabled` is omitted, still resolves missing cron job models through the configured agent/default model path, and still lacks the broader external-registration consent, cron cost policy, and usage circuit-breaker design. The related cron defaults issue is useful context but does not supersede the full consent and runaway-usage policy reported here.

## What This Changes

The issue requests explicit activation and cost/usage guardrails for external or plugin-registered polling cron jobs instead of silently inheriting active state and global model defaults.

## Best Possible Solution

Define a source-aware cron registration contract that separates user-created and external/plugin-created jobs, then implement explicit activation, cron-scoped model/default or warning policy, doctor/update dry-run visibility, quota guardrails, docs, and regression coverage together.

## Reproduction Assessment

Yes for the core current behavior by source inspection: omitted cron `enabled` still becomes true, and omitted cron `payload.model` still resolves through the agent/default model path. I did not reproduce the reported overnight quota burn in a live provider setup during this read-only review.

## Solution Assessment

Unclear as an automatic patch: the requested safeguards change public cron registration semantics, CLI/doctor UX, config schema, model-cost policy, and quota enforcement. The maintainable solution needs maintainer approval and should reconcile the overlapping cron-default tracker before implementation.
