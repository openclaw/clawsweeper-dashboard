## Summary

Keep open: current main and the latest shipped release still default omitted cron jobs to enabled and allow cron agent turns to inherit agent/default model selection, while the requested consent, model-cost, dry-run, and quota guardrails require maintainer product/API decisions.

## What This Changes

The issue asks OpenClaw to require explicit activation and cost guardrails for externally registered polling cron jobs that currently auto-enable and inherit default models.

## Best Possible Solution

Approve a cron registration policy that distinguishes external registrations from user-created jobs, then implement opt-in activation, explicit cron model/default handling, doctor/update dry-run visibility, quota guardrails, docs, and regression coverage as one coherent contract.

## Reproduction Assessment

Yes for the core source path: omitted cron enabled values still become active, and omitted cron payload.model values still flow through agent/default model selection. I did not establish a live reproduction of the reported quota burn or the separate interactive usage regression.

## Solution Assessment

Unclear as an automatic patch: the requested safeguards change public cron registration semantics, CLI/doctor/update UX, model-cost policy, config defaults, and quota enforcement. The safer path is a maintainer-approved contract that also coordinates with https://github.com/openclaw/openclaw/issues/57638 for cron model/delivery defaults.
