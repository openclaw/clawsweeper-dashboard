## Label Justifications

- `P1`: The issue describes real background cron activity consuming expensive provider/model quota without consent, and current main still permits the source-level behavior.
- `impact:auth-provider`: The affected behavior centers model selection, provider quota usage, and provider/model guardrails for scheduled background jobs.

## Summary

Keep open: current main and the latest release still default omitted cron jobs to enabled, cron jobs without `payload.model` still fall through to agent/default model selection, and recurring errors still use fixed backoff with only partial `lastErrorReason` observability. The related open issues cover cron defaults, reason-aware guardrails, and provenance slices, but they do not fully replace this broader consent, cost, and quota policy issue.

## What This Changes

The issue asks OpenClaw to require safer activation, model selection, dry-run visibility, and usage guardrails for externally registered polling cron jobs.

## Best Possible Solution

Approve a cron registration and provenance contract that separates operator-created jobs from external/agent/plugin registrations, then implement opt-in activation, cron-specific model/default policy, doctor/update visibility, reason-aware guardrails, docs, and regression coverage together.

## Reproduction Assessment

Yes for the source-level current-main path: omitted `enabled` still becomes active in normalization/service creation, and isolated agent-turn cron model selection still falls through to agent/default model selection when `payload.model` is absent. I did not live-reproduce the reported quota burn or the separate interactive usage regression.

## Solution Assessment

No as an automatic patch: the requested safeguards change public cron registration semantics, config/schema, CLI/doctor/update UX, model-cost policy, and quota enforcement.
