## Summary

Keep open: current main and the latest shipped tag still default missing cron `enabled` fields to true and still fall back to agent/default model selection when no per-job model is set. Related open issues cover cron defaults, agent scheduling provenance, and gateway budgets in pieces, but they do not fully supersede the requested confirmation, default-disabled external registration, model-cost, and usage circuit-breaker policy.

## What This Changes

The issue requests safeguards so externally registered polling cron jobs do not silently enable, inherit expensive default models, or run without usage limits.

## Best Possible Solution

Define one source-aware cron registration policy, then split implementation into compatible disabled-by-default or confirmation behavior for external jobs, cron-scoped model defaults, and gateway-level cost or quota guardrails that share the same documented contract.

## Reproduction Assessment

Yes for the core current behavior by source inspection: missing cron `enabled` still defaults to true, and isolated cron falls back to agent/default model selection when no per-job model is set. I did not establish a live runaway quota burn in this read-only review.

## Solution Assessment

Unclear as one patch: the requested safeguards span cron registration identity, update/doctor confirmation UX, model-cost policy, and quota thresholds. The maintainable path is to split source-aware enablement/confirmation from cron defaults and budget enforcement after maintainer approval.
