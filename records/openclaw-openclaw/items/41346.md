## Summary

Keep open: current main still has source-reproducible default-enabled cron creation and default-model inheritance, with only partial diagnostics and no quota/cost guardrail. Related open items track narrower cron defaults, guardrail, and provenance slices rather than fully superseding this issue, so this needs maintainer product/API review.

## What This Changes

The issue requests cron safety policy changes so externally registered polling jobs do not start enabled by default, do not silently inherit expensive agent defaults, and are bounded by doctor/dry-run/cost/quota guardrails.

## Best Possible Solution

Approve a cron registration policy that distinguishes operator-created jobs from external/agent/plugin registrations, then implement opt-in activation, explicit cron model/default handling, doctor/update dry-run visibility, reason-aware usage guardrails, docs, and regression coverage as one contract.

Yes for the source-level core path: cron.add/default normalization and createJob still make omitted enabled values active, and isolated cron model selection still falls through to agent/default model selection when payload.model is absent.
