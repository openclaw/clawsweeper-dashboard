## Summary

Keep open. Current main still accepts cron creation without an explicit disabled/consent state, normalizes omitted `enabled` to true, has enabled-by-default CLI and agent-tool paths, and includes bundled managed cron code that creates or re-enables recurring jobs. Isolated cron jobs without `payload.model` still resolve through the agent/default model path, and I found usage/cost telemetry but no origin-aware consent flow, expensive inherited-model guard, doctor/update estimate, or quota circuit breaker.

## What This Changes

Request safeguards so externally or plugin-registered recurring cron jobs require user activation or consent, avoid expensive inherited default models, expose doctor/update dry-run estimates, and pause non-critical polling on excessive usage.

## Best Possible Solution

Implement an origin-aware cron safety contract: persist creator/origin and managed-job intent, require activation or auditable consent for newly registered external recurring jobs, surface cadence/resolved model/model source/estimated daily call volume in doctor/update previews, warn or block expensive inherited defaults unless recurring jobs set `payload.model`, and add configurable quota pause behavior for non-critical recurring work.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a `cron.add` request that omits `enabled` is valid and becomes enabled through normalization/service creation, while a recurring isolated job without `payload.model` follows the documented model precedence to agent/default selection. I did not run a live token-burn scenario because this review is read-only.

## Solution Assessment

Unclear for the full safeguard list as written. The direction is sound, but a blanket default-disabled flip alone is not the best fix because current main and bundled managed jobs rely on existing cron activation semantics; the safer path is origin-aware, migration-aware consent plus separate model-cost and quota policies.
