## Summary

Keep open. Current main still accepts cron creation without an explicit disabled/consent state, normalizes omitted `enabled` values to true, has managed plugin code that creates and re-enables recurring cron jobs, and resolves isolated cron jobs without `payload.model` through agent/default model selection; I did not find a consent plan, dry-run estimate, expensive inherited-model guard, or quota circuit breaker.

## What This Changes

Request origin-aware safeguards so plugin/external recurring cron jobs do not silently auto-enable, inherit costly default models, or run without quota/cost controls.

## Best Possible Solution

Design an origin-aware cron safety contract: persist owner/origin metadata, require explicit activation or a trusted activation plan for newly registered external/plugin recurring jobs, expose cadence/resolved model/model source/estimated daily calls in doctor/update dry-runs, guard expensive inherited defaults, and add configurable quota/cost pausing for non-critical polling.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron.add` accepts omitted `enabled`, normalization and job creation default it to true, and memory-core managed dreaming creates or repairs recurring isolated jobs as enabled; I did not run a live token-burn test.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is sound, but the safest fix is not a blanket default-disabled change; it should be origin-aware, migration-aware, and explicit about trusted managed jobs versus external/plugin jobs requiring user activation.
