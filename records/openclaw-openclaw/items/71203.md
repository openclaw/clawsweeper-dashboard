## Summary

Keep open. The PR addresses a real stale-cache bug, but current head still drops explicit empty provider-discovery options for CLI-only configurations and the contributor has not provided real startup/cache-refresh proof.

## What This Changes

The PR adds a gateway startup helper that refreshes generated `models.json` caches for the default agent directory and configured agent directories, plus focused startup regression tests.

## Best Possible Solution

Keep the separate startup refresh helper, but preserve scoped metadata-only discovery even when the provider list is empty, add CLI-only regression coverage, and require redacted real startup proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: current main skips startup models.json preparation when no default primary model is configured and otherwise refreshes only the default agent dir, matching the linked zed/kim stale-cache report.

## Solution Assessment

Is this the best way to solve the issue? No for the current PR head. The independent helper is the right direction, but it must preserve explicit empty provider-discovery scope and include a focused CLI-only regression before it is safe to merge.
