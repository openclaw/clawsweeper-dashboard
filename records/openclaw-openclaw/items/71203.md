## Summary

Keep open. The branch implements the requested configured-agent startup refresh direction, but the latest head still drops scoped metadata-only provider discovery for CLI-only configurations and only supplies test/format output rather than real after-fix behavior proof.

## What This Changes

The branch adds a gateway startup helper that refreshes models.json for the active/default agent directory and configured agent directories, and expands gateway startup tests for custom agents.

## Best Possible Solution

Revise this PR so startup refreshes unique default/configured agent dirs while always preserving scoped entries-only provider discovery, including empty provider lists, then add real startup/cache-refresh proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source inspection: current main skips startup models.json preparation when no default primary model is configured and otherwise prepares only the default agent dir, matching the linked zed/kim stale-cache report.

## Solution Assessment

Is this the best way to solve the issue? No for the latest PR head. The separate helper is the right direction, but it must keep explicit empty provider-discovery options scoped and supply real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Pass empty provider discovery scope:** `src/gateway/server-startup-post-attach.ts:163-170`
  - body: When every configured/default model provider is CLI-backed, providerDiscoveryProviderIds becomes [], but this branch converts the scoped startup options to {}. That makes ensureOpenClawModelsJson run without the empty scope, entries-only flag, or startup timeout before channel startup, so CLI-only gateways can discover/load unrelated provider runtimes and block readiness.
