## Summary

Keep open: this is a plausible fix for a real startup cache bug, but the current PR head still drops the explicit empty provider-discovery scope in CLI-only configs and lacks contributor-supplied real startup proof.

## What This Changes

The PR adds a gateway startup helper to refresh default and configured agent `models.json` files and extends startup tests for custom-agent and no-default-primary cases.

## Best Possible Solution

Keep the separate startup refresh helper, but always pass scoped metadata-only discovery options even when the provider list is empty, then add CLI-only regression coverage and redacted real startup proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main skips startup cache refresh when no default primary model is configured and otherwise refreshes only the default agent dir; the PR defect is also source-reproducible with an all-CLI provider list producing `providerDiscoveryProviderIds: []`.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted. The independent startup helper is the right direction, but it must preserve explicit empty provider-discovery scope and include a CLI-only regression before merge.

## Review Findings

Overall correctness: patch is incorrect
