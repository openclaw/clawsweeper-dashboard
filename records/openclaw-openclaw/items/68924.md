## Summary

Keep this PR open: the wording cleanup is plausible, but the latest head still adds user-visible cost-window hints with source-inspectable correctness problems and lacks contributor-supplied real UI proof.

## What This Changes

The PR renames Control UI cost labels to usage cost and passes `usageCostSummary` into overview cards to show 7-day and 30-day cost hints.

## Best Possible Solution

Keep the clearer wording if maintainers want it, but compute overview cost hints from complete calendar windows over one cost source, hide hints when cost coverage is incomplete, preserve the current provider-quota card, and add redacted real UI proof.

## Reproduction Assessment

Yes. Source inspection is enough: the PR labels whatever `usageCostSummary.daily` rows were loaded as `7d` and `30d`, while current main defaults the selected range to today and emits sparse daily rows.

## Solution Assessment

No. The wording change is reasonable, but the implementation should use complete calendar windows from one cost source, respect missing-cost coverage, and be rebased over the current provider-quota overview card.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Compute real windows before labeling cost hints:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: The new hint labels whatever `usage.cost` daily rows were loaded as `7d` and `30d`. Since `loadUsage` requests the selected start/end range and current main defaults that range to today, a fresh dashboard can show a one-day total as both 7-day and 30-day spend.
  - confidence: 0.91
- **[P2] Sum calendar days instead of populated entries:** `ui/src/ui/views/overview-cards.ts:80-84`
  - body: `usageCostSummary.daily` only contains days with usage on current main, but `.slice(-days)` takes the last N populated rows. Intermittent activity can make the advertised 7-day or 30-day total include spend from much older calendar days.
  - confidence: 0.88
- **[P2] Use one cost scope for value and hint:** `ui/src/ui/views/overview-cards.ts:97-103`
  - body: The card value prefers `usageResult.totals`, while the hint comes from `usageCostSummary.daily`. `sessions.usage` is capped at 1000 sessions but `usage.cost` is not, so high-volume profiles can display a value and hint from different datasets.
  - confidence: 0.86
- **[P2] Hide window hints when costs are incomplete:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: This branch shows dollar-window hints whenever daily rows exist, but the cost summary separately tracks missing cost entries. Providers or transcripts without cost reporting can therefore surface misleading `$0.00` windows instead of falling back to token/message context.
  - confidence: 0.84
