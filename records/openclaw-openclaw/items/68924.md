## Summary

Keep open: current main has not implemented this UI change, and the live PR still has source-reproducible defects in the new overview cost hint. Earlier i18n/model-auth feedback appears addressed, but the 7d/30d hint is still misleading and the external PR still lacks real UI proof.

## What This Changes

The branch renames Control UI cost labels to Usage Cost, threads `usageCostSummary` into overview cards, and adds 7-day and 30-day usage-cost hints to the overview cost card.

## Best Possible Solution

Keep the clearer wording only if maintainers want it, but compute the overview hint from complete calendar windows over one consistent cost data scope and fall back when cost coverage is incomplete.

## Reproduction Assessment

Yes for the PR branch via source inspection: PR head renders fixed `7d`/`30d` labels from whatever selected-range daily rows were loaded, while current main defaults the selected usage range to today-to-today. This is not a current-main bug because the patch has not merged.

## Solution Assessment

No. The direction is useful, but this implementation should not treat sparse selected-range entries as fixed calendar windows or mix capped `sessions.usage` totals with uncapped `usage.cost` hints.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Compute real windows before showing 7d and 30d hints:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: The hint labels selected-range daily rows as fixed 7d/30d windows. `loadUsage` sends the currently selected start/end dates to `usage.cost`, and current main defaults both dates to today, so a fresh dashboard can show one-day spend as 7d and 30d.
  - confidence: 0.9
- **[P2] Sum calendar days instead of populated entries:** `ui/src/ui/views/overview-cards.ts:80-84`
  - body: `usageCostSummary.daily` only contains days with usage, so `.slice(-days)` counts the last N active entries. Intermittent usage can make the advertised 7d or 30d total include older spend outside that calendar window.
  - confidence: 0.88
- **[P2] Use one cost scope for the card value and hint:** `ui/src/ui/views/overview-cards.ts:97-103`
  - body: The card value prefers `usageResult.totals`, while the new hint is derived from `usageCostSummary.daily`. `sessions.usage` is capped at 1000 sessions but `usage.cost` is not, so high-volume workspaces can see a value and hint from different scopes.
  - confidence: 0.84
- **[P2] Suppress window hints when cost coverage is incomplete:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: This branch shows dollar-window hints whenever daily rows exist, but cost aggregation tracks missing costs separately with `missingCostEntries`. Providers or transcripts without cost reporting can therefore display false `$0.00` windows instead of falling back to token/message context.
  - confidence: 0.86
