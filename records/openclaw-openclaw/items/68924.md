## Summary

Keep this PR open. Current main does not implement the requested overview wording or usage cost summary card, and the PR is still a useful narrow implementation, but the proposed hint logic would mislabel selected-range data as fixed 7-day and 30-day windows and can show false-zero costs when cost coverage is incomplete.

## What This Changes

The PR renames cost-related Control UI copy to “Usage Cost”, threads `usageCostSummary` into overview rendering, and changes the overview cost card hint to show 7-day and 30-day usage cost totals.

## Best Possible Solution

Keep the feature direction, but revise the card to compute real calendar-window summaries from one consistent cost source, gate or fall back when cost data is incomplete, preserve i18n strings, and add focused UI coverage for default, intermittent, and missing-cost cases.

## Reproduction Assessment

Yes. The introduced PR issues can be reproduced by loading the overview with the default today-only usage dates and any nonempty `usageCostSummary.daily`, then with intermittent daily entries or entries carrying `missingCostEntries`; the card will still render fixed 7d/30d cost hints.

## Solution Assessment

No. The proposed direction is reasonable, but the implementation should use calendar-aware windows or explicitly selected-range labels, keep the card value and hint on the same data scope, and fall back to token/message context when cost coverage is incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Compute actual calendar windows for cost hints:** `ui/src/ui/views/overview-cards.ts:136-138`
  - body: The hint prints `7d` and `30d` whenever `costDaily` has entries, but overview refresh calls `loadUsage` with the current usage date filters, whose defaults are both today. On a fresh dashboard these values are one-day totals with fixed multi-day labels, so the cost card misrepresents spend.
  - confidence: 0.9
- **[P2] Sum calendar days, not populated entries:** `ui/src/ui/views/overview-cards.ts:80-84`
  - body: `daily` only contains days that had usage entries, so `.slice(-days)` takes the last N active days rather than the last N calendar days. Intermittent usage across a wider range will make the displayed 7d/30d totals include older spend.
  - confidence: 0.88
- **[P2] Fall back when cost coverage is incomplete:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: `costDaily.length > 0` can be true even when entries have token usage but missing cost data, because missing costs are tracked via `missingCostEntries` and total cost remains zero. In that case the new hint can show `$0.00` windows instead of the prior token/message context, which is misleading.
  - confidence: 0.86
