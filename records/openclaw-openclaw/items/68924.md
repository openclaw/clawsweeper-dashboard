## Summary

Keep open: this is a focused Control UI wording and overview-card change, but the current PR head still has source-level correctness issues in the new usage-cost hints and lacks real behavior proof before merge.

## What This Changes

The PR renames Control UI cost labels to Usage Cost, passes usageCostSummary into overview cards, and adds 7-day and 30-day usage-cost hints to the overview cost card.

## Best Possible Solution

Keep the clearer wording if maintainers want it, but make the overview cost hints range-aware, calendar-day based, cost-coverage aware, and sourced from one consistent cost dataset before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the default selected usage range can be today-to-today while the new card labels any returned daily cost entries as 7d and 30d.

## Solution Assessment

No. The direction is useful, but this implementation should not label selected-range or sparse daily entries as fixed calendar windows, and it should avoid mixing capped sessions.usage totals with uncapped usage.cost hints.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Compute real windows before showing 7d and 30d hints:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: The hint always prints `7d` and `30d`, but overview loading passes the selected usage dates to `usage.cost`, and those dates default to today-to-today. A normal fresh dashboard can therefore present one-day data as 7-day and 30-day spend.
  - confidence: 0.9
- **[P2] Sum calendar days instead of populated entries:** `ui/src/ui/views/overview-cards.ts:80-84`
  - body: `usageCostSummary.daily` only contains days with usage, so `.slice(-days)` takes the last N active entries. With intermittent usage, the displayed 7-day or 30-day totals can include older spend outside the advertised calendar window.
  - confidence: 0.88
- **[P2] Use one cost scope for the value and hint:** `ui/src/ui/views/overview-cards.ts:97-103`
  - body: The main card value prefers `usageResult.totals`, while the new hint is derived from `usageCostSummary.daily`. `sessions.usage` is capped at 1000 sessions but `usage.cost` is not, so high-volume workspaces can see a card value that disagrees with the hint's data scope.
  - confidence: 0.84
- **[P2] Suppress cost-window hints when coverage is incomplete:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: Daily entries can exist while cost data is missing, tracked separately by `missingCostEntries`. In that case this branch can show `$0.00` windows instead of falling back to token/message context, which is misleading for providers or transcripts without cost reporting.
  - confidence: 0.86
