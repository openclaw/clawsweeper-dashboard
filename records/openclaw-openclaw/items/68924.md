## Summary

Keep open: the UI wording direction is plausible, but the branch is not merge-ready because source inspection still shows cost-window correctness bugs, contributor proof is missing, and the live PR is currently conflicting with main.

## What This Changes

The PR renames usage-cost labels in the web UI and wires `usageCostSummary` into overview cards so the cost card can show 7-day and 30-day hints.

## Best Possible Solution

Keep the wording direction only if maintainers want it, and compute overview cost hints from a consistent dense calendar-window source that falls back when cost coverage is incomplete.

## Reproduction Assessment

Yes, via source inspection: PR head renders fixed `7d` and `30d` labels from selected-range `usageCostSummary.daily`, while current main defaults the usage range to today and can emit sparse daily rows. I did not run a browser session in this read-only review.

## Solution Assessment

No. The wording may be acceptable, but the implementation should compute complete calendar windows from one cost source, gate on complete cost coverage, and rebase onto the current overview-card code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Compute real windows before labeling cost hints:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: The new hint labels whatever `usage.cost` daily rows were loaded as `7d` and `30d`. Since `loadUsage` requests the selected start/end range and current main defaults that range to today, a fresh dashboard can show a one-day total as both 7-day and 30-day spend.
  - confidence: 0.91
- **[P2] Sum calendar days instead of populated entries:** `ui/src/ui/views/overview-cards.ts:80-84`
  - body: `usageCostSummary.daily` on current main is sparse, but `.slice(-days)` takes the last N populated rows. Intermittent activity can make the advertised 7-day or 30-day total include spend from much older calendar days.
  - confidence: 0.88
- **[P2] Use one cost scope for value and hint:** `ui/src/ui/views/overview-cards.ts:97-103`
  - body: The card value prefers `usageResult.totals`, while the hint comes from `usageCostSummary.daily`. `sessions.usage` is capped at 1000 sessions but `usage.cost` is not, so high-volume profiles can display a value and hint from different datasets.
  - confidence: 0.86
- **[P2] Hide window hints when costs are incomplete:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: This branch shows dollar-window hints whenever daily rows exist, but the cost summary separately tracks missing cost entries. Providers or transcripts without cost reporting can therefore surface misleading `$0.00` windows instead of falling back to token/message context.
  - confidence: 0.84
