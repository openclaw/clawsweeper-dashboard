## Summary

Keep this PR open. Current main still has the older dashboard wording and does not thread `usageCostSummary` into overview cards, so the contribution is not obsolete, but the proposed cost-card logic still has user-visible correctness regressions around date-window labeling, sparse daily data, mixed data scopes, and incomplete cost coverage.

## What This Changes

The PR renames Control UI cost copy to “Usage Cost”, passes usage cost summary data into overview cards, and adds 7-day and 30-day cost hints to the overview cost card.

## Best Possible Solution

Keep the clearer usage-cost wording, but revise the overview cost card so its main value and hint come from one consistent cost source, the hint labels match actual calendar coverage or selected range, sparse days are handled as zero-use calendar days, incomplete cost coverage falls back to token/message context, and focused UI coverage locks those cases down.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by following the default overview load: `usageStartDate` and `usageEndDate` initialize to today, `loadUsage` requests `usage.cost` for that range, and the PR renders fixed `7d`/`30d` labels whenever any daily entry exists. Additional fixtures with intermittent daily entries and `missingCostEntries` reproduce the other review findings.

## Solution Assessment

No. The direction is useful, but this implementation is not the narrowest maintainable solution because it labels selected-range data as fixed calendar windows, sums active entries instead of calendar days, mixes capped and uncapped usage sources, and does not gate cost hints on cost coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Compute actual windows before labeling cost hints:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: The hint always prints `7d` and `30d`, but overview loading passes the current usage date filters to `usage.cost`; those filters default to today-to-today. On a normal fresh dashboard, this can present one-day data as 7-day and 30-day spend.
  - confidence: 0.9
- **[P2] Sum calendar days instead of populated entries:** `ui/src/ui/views/overview-cards.ts:80-84`
  - body: `usageCostSummary.daily` only contains days with usage, so `.slice(-days)` takes the last N active days. For intermittent usage, the displayed 7-day or 30-day totals can include older spend outside the advertised calendar window.
  - confidence: 0.88
- **[P2] Use one cost scope for card value and hint:** `ui/src/ui/views/overview-cards.ts:97-103`
  - body: The main value prefers `usageResult.totals`, while the hint uses `usageCostSummary.daily`. `sessions.usage` is capped at 1000 sessions but `usage.cost` is not, so high-volume workspaces can see a card value that disagrees with the hint's data scope.
  - confidence: 0.84
- **[P2] Suppress cost-window hints when coverage is incomplete:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: Daily entries can exist while cost data is missing, tracked separately by `missingCostEntries`. In that case this branch can show `$0.00` windows instead of the prior token/message context, which is misleading for providers or transcripts without cost reporting.
  - confidence: 0.86
