## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Derive cost-window labels from the loaded range:** `ui/src/ui/views/overview-cards.ts:138`
  - body: On overview refresh, `loadOverview` calls `loadUsage` with the current usage date state, which defaults to today. This means a normal overview load can render `7d` and `30d` even though `usage.cost` was only requested for one day; request real 7/30-day summaries or label the hint as the loaded range.
  - confidence: 0.92
- **[P2] Sum calendar days instead of sparse entries:** `ui/src/ui/views/overview-cards.ts:83`
  - body: `usage.cost` currently builds `daily` only for days with records, so `slice(-7)` can span much older intermittent usage and overstate a recent 7-day total. Base the windows on date boundaries, or require a dense daily series before showing fixed recent-window totals.
  - confidence: 0.9
- **[P2] Gate cost hints on complete cost coverage:** `ui/src/ui/views/overview-cards.ts:137-138`
  - body: Cost aggregation tracks `missingCostEntries`, but this branch shows 7d/30d dollar hints whenever `daily` is non-empty. Providers or transcripts without pricing can therefore show `$0.00` windows as if costs were known; fall back to token/message context or an incomplete-cost hint when costs are missing.
  - confidence: 0.88
- **[P2] Keep the card value and hint on one usage scope:** `ui/src/ui/views/overview-cards.ts:97-103`
  - body: The main value still prefers `sessions.usage` totals, which are requested with `limit: 1000`, while the new hint comes from uncapped `usage.cost`. Large workspaces can see a card total and 7d/30d hint from different datasets; use one summary source or make the cap/scope explicit.
  - confidence: 0.84
