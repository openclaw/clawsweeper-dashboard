## Summary

Keep open: the UI wording cleanup is plausible, but the PR is conflicting, lacks contributor-supplied real Control UI proof, and the new overview cost-window hints can misstate current usage data.

## What This Changes

The PR renames Control UI cost labels to Usage Cost and threads usageCostSummary into overview cards to show 7-day and 30-day cost hints.

## Best Possible Solution

Keep the clearer wording if maintainers want it, but rebase onto current main, compute cost windows from complete calendar ranges over one cost source, suppress dollar windows when cost coverage is incomplete, and attach redacted UI proof.

## Reproduction Assessment

Yes. Source inspection is enough: the PR labels whatever usageCostSummary.daily rows were loaded as 7d and 30d, while current main defaults the selected usage range to today and emits sparse daily rows.

## Solution Assessment

No. The wording change is reasonable, but the implementation should use complete calendar windows from one cost source, respect missing-cost coverage, and be rebased over the current provider-quota overview card.

- **[P2] Compute real windows before labeling cost hints:** `ui/src/ui/views/overview-cards.ts:136-139`

  - confidence: 0.88

  - body: The card value can come from usageResult.totals while the hint comes from usageCostSummary.daily. sessions.usage is capped at 1000 sessions but usage.cost is not, so high-volume workspaces can see inconsistent cost totals.
  - confidence: 0.86
- **[P2] Hide window hints when costs are incomplete:** `ui/src/ui/views/overview-cards.ts:136-139`
  - body: This branch shows dollar-window hints whenever daily rows exist, but the summary separately tracks missingCostEntries. Providers or transcripts without cost reporting can therefore surface misleading $0.00 windows instead of falling back to token/message context.
  - confidence: 0.84
