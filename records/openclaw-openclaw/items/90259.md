## Summary

Keep this PR open. The feature is still useful and not implemented on current main, but the proposed carryover resolver has a blocking session-state bug because it can choose a compaction summary from an abandoned branch instead of the active transcript branch.

## What This Changes

This PR adds family-history retrieval for chat history/session tools, injects reset-ancestor compaction summaries into embedded-agent prompts when active context lacks one, and includes related protocol snapshot and lint-suppression cleanup.

## Best Possible Solution

Keep the family-history API and carryover idea, but derive carryover only from each ancestor transcript's active branch/session context and land it after the dependent family-history surface is documented and reviewed.

## Reproduction Assessment

Yes, at source level: current main records `usageFamilySessionIds` during reset/rollover but current `chat.history` and embedded prompts only use the active transcript. I did not run a live repro because this review is read-only.

## Solution Assessment

No, not as currently implemented: the PR uses the right LLM-boundary hook, but the resolver must honor active transcript branch semantics instead of scanning every parsed entry for the newest compaction.
