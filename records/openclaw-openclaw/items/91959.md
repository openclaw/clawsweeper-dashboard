## Summary

The report is valid and appears to share the same provider-stream root cause already tracked in the older canonical issue: Responses-compatible Bedrock endpoints emit successive cumulative `final_answer` message items, while current aggregation preserves each as a separate assistant text block. Keep this issue open for now because the canonical issue is also open and no merged fix exists; maintainers should consolidate the two reports and implement one provider-boundary repair.

## What This Changes

Prevent cumulative Bedrock Mantle Responses message snapshots from being concatenated into a massively duplicated final assistant reply.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/89531 as the canonical tracker, preserve this report's Mantle reproduction evidence there, and fix the Responses provider boundary to replace only cumulative snapshots of the same logical final-answer item while retaining legitimate distinct message items.

## Reproduction Assessment

Yes. The issue supplies an exact released-version command, two affected models, reasoning-on/off comparisons, and quantitative duplicate counts; current source also exposes the aggregation path that preserves multiple cumulative message items, though a raw event capture would refine the identity rule.

## Solution Assessment

Partly. Last-wins replacement is the right direction for cumulative snapshots, but it should be keyed to the same logical output item or an explicit Bedrock-compatible transport contract rather than applied to all Responses message items.
