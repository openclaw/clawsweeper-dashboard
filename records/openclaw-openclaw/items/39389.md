## Summary

Keep open: the PR targets a real Feishu documentation gap, but the added wording still risks misdocumenting routing because “default account” can be read as `channels.feishu.defaultAccount`, while inbound routing uses the literal default routing bucket unless `accountId: "*"` is set.

## What This Changes

The PR adds Feishu `match.accountId` guidance and an `accountId: "*"` wildcard routing example to `docs/channels/feishu.md`.

## Best Possible Solution

Update the Feishu page to explain that omitted `match.accountId` maps to OpenClaw's literal default routing bucket, separate that from outbound `channels.feishu.defaultAccount`, and show `accountId: "*"` for cross-account Feishu bindings.

## Reproduction Assessment

Yes. The docs defect is reproducible by comparing the PR wording with `src/routing/resolve-route.ts`, the route tests, and Feishu inbound's concrete `account.accountId` call path.

## Solution Assessment

No. A Feishu-specific clarification is the right direction, but this patch should revise the wording so it does not conflate the inbound default routing bucket with Feishu's outbound `defaultAccount` setting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clarify the default routing bucket:** `docs/channels/feishu.md:586`

Inbound routing maps omitted `accountId` to OpenClaw's literal default routing bucket while Feishu passes concrete runtime account ids, so multi-account users can read this as matching their configured default account when it will not.
