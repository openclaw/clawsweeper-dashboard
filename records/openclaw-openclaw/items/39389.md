## Summary

Keep open: the Feishu docs gap is real, but this PR's new wording can still mislead multi-account users by conflating omitted `match.accountId` with `channels.feishu.defaultAccount`; it also lacks contributor-provided real behavior proof.

## What This Changes

The PR adds a Feishu channel docs bullet for `match.accountId` and a wildcard `accountId: "*"` example for routing one peer across Feishu account instances.

## Best Possible Solution

Revise the Feishu page to name the literal `default` routing bucket separately from `channels.feishu.defaultAccount`, keep the wildcard example, and validate the rendered docs.

## Reproduction Assessment

Yes. The docs defect is source-reproducible by comparing this PR's new sentence with the resolver, route tests, and Feishu inbound `account.accountId` call path.

## Solution Assessment

No. A Feishu-specific clarification is the right surface, but the proposed `default account` wording should be tightened before merge because it can be read as the Feishu outbound `defaultAccount` setting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Name the literal default routing bucket:** `docs/channels/feishu.md:586`
  - body: This says omitted `match.accountId` matches the "default account," but this page also defines `channels.feishu.defaultAccount` for outbound selection. Inbound routing indexes omitted account scope into the literal `default` bucket while Feishu passes concrete account ids, so users with `defaultAccount: "main"` can configure bindings that silently do not match.
  - confidence: 0.9
