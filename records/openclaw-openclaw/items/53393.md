## Summary

Keep open. Current main covers the original model-catalog hot-reload symptom, but the PR’s broader registry refactor is not implemented on main and the branch introduces a context-window cache race by clearing an in-flight loader without a generation guard.

## What This Changes

This PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches against model config changes, wires the registry into Gateway hot reload, and adds registry unit tests.

## Best Possible Solution

Do not merge this branch as-is. Either close it as superseded for the original model-catalog bug, or ask for a narrow current-main revision that invalidates context-window/config-derived state with a generation or cancellation guard and focused regression coverage.

## Reproduction Assessment

Yes. The original model-catalog symptom is reproducible from the linked issue steps, and the PR-specific bug is reproducible by triggering hot reload while `ensureContextWindowCacheLoaded()` is in flight, allowing the old async loader to repopulate the cache after invalidation.

## Solution Assessment

No. A registry could be a maintainable direction, but this implementation clears context state without preventing stale async writes; the current model-catalog generation guard is the safer pattern to mirror.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Guard stale context loads before clearing the cache:** `src/agents/context.ts:94`
  - body: Resetting `loadPromise` during hot reload allows a pre-reload `ensureContextWindowCacheLoaded()` run to continue after `MODEL_CONTEXT_TOKEN_CACHE.clear()` and write old model windows back into the cache. Add a generation or cancellation guard so pre-reload loads cannot repopulate stale entries after invalidation.
  - confidence: 0.88
