## Summary

Keep open: current main covers the original model-catalog hot-reload symptom, but this broader registry refactor is not implemented on main and the PR can reintroduce stale context-window cache data during an in-flight reload.

## What This Changes

The PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Either close this refactor in favor of current main's targeted model-catalog handling, or rework it with generation-guarded context-window invalidation before merge.

## Reproduction Assessment

Yes, by source inspection. Clearing `loadPromise` while `ensureContextWindowCacheLoaded()` is in flight lets the older async task resume after hot reload and repopulate `MODEL_CONTEXT_TOKEN_CACHE` with pre-reload data.

## Solution Assessment

No. A registry could be maintainable, but the context-window invalidation path needs the same generation/cancellation protection used by the Gateway model-catalog refresh before this is safe to merge.
