## Summary

Keep open for human review. Current main covers the linked model-catalog hot-reload bug with targeted invalidation, but this broader refactor is not implemented on main, introduces a source-level context-window race, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR adds an agents config-derived cache registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's targeted model-catalog reload path unless maintainers want a broader registry; if they do, add generation/cancellation protection to context-window invalidation and require runtime proof.

## Reproduction Assessment

Yes, for the PR-introduced race by source inspection: the invalidator can clear context state while a pre-reload `ensureContextWindowCacheLoaded()` task resumes after awaits and writes stale data back. I did not run tests because this review is read-only.

## Solution Assessment

No, not in this form. A registry could be maintainable if maintainers want it, but this diff needs a context-window generation guard and real runtime proof before replacing current main's targeted invalidation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Guard context-window loads before clearing cache:** `src/agents/context.ts:94-95`
  - body: The new invalidator clears `MODEL_CONTEXT_TOKEN_CACHE` and resets `loadPromise` while an existing `ensureContextWindowCacheLoaded()` task may still be running. That older task can resume after the models.json/discovery awaits and write stale context-window data back into the cleared cache, so add a generation or cancellation guard before async loads write shared context state.
  - confidence: 0.88
