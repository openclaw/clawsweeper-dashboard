## Summary

Keep open: the branch still has a blocking context-window cache race and lacks contributor real-behavior proof; current main already covers the linked model-catalog hot-reload bug, so the remaining broader registry is a maintainer architecture decision.

## What This Changes

The PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, and calls it from Gateway hot reload.

## Best Possible Solution

Keep current main's targeted model-catalog hot-reload handling unless maintainers explicitly want a broader registry; if they do, add generation/cancellation protection to context-window invalidation and require real runtime proof.

## Reproduction Assessment

Yes for the PR-introduced race, by source inspection. The proposed invalidator can clear shared context state while an older ensureContextWindowCacheLoaded task resumes after awaits and writes stale data back.

## Solution Assessment

No, not in this form. A registry may be maintainable, but this implementation needs a context-window generation guard and maintainer approval before replacing current main's targeted invalidation.
