## Summary

Keep open. Current main already covers the original model-catalog hot-reload symptom, but this PR's broader registry refactor has not landed and the proposed context-window invalidation still has a stale async-loader race called out in the existing review discussion.

## What This Changes

This PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, wires the registry into Gateway hot reload, and adds registry unit tests.

## Best Possible Solution

Rework against current main with a generation-guarded context/config-derived invalidation path, or close the refactor if maintainers prefer the existing targeted model-catalog fix plus a narrower follow-up.

## Reproduction Assessment

Yes, by source inspection. Triggering hot reload while the PR's `ensureContextWindowCacheLoaded()` path is in flight can let the old async task write stale context windows back after the invalidator clears the cache.

## Solution Assessment

No. A registry could be maintainable, but this implementation should not clear context-window state without a generation or cancellation guard; the current Gateway model-catalog generation guard is the safer pattern to mirror.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Guard stale context loads before clearing cache:** `src/agents/context.ts:94`
  - body: Resetting `loadPromise` during hot reload allows a pre-reload `ensureContextWindowCacheLoaded()` task to continue after `MODEL_CONTEXT_TOKEN_CACHE.clear()` and write old model windows back into the cache. Add a generation or cancellation guard so pre-reload loads cannot repopulate stale entries after invalidation.
  - confidence: 0.88
