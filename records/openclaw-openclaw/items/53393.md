## Summary

Keep open: the linked model-catalog hot-reload bug is already addressed on current main, but this broader registry PR is still not safe to merge because it introduces a source-visible context-window cache race, lacks real runtime proof, conflicts with main, and needs maintainer judgment on the registry architecture.

## What This Changes

The PR adds a config-derived cache registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's targeted model-catalog invalidation unless maintainers explicitly choose a registry; any registry version should add generation/cancellation guards for context-window loads and include real hot-reload proof.

## Reproduction Assessment

Yes, for the PR-introduced bug by source inspection: start `ensureContextWindowCacheLoaded()`, invalidate after its awaits begin, and the older task can write old context-window data back into the cleared cache. I did not execute a runtime repro because this review is read-only.

## Solution Assessment

No, not in this form. A registry could be maintainable if maintainers want that architecture, but this branch needs a context-window generation/cancellation guard and real runtime proof before it should replace current targeted invalidation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Guard in-flight context-window cache loads:** `src/agents/context.ts:93-95`
  - body: The new invalidator clears `MODEL_CONTEXT_TOKEN_CACHE` and resets `loadPromise` while an older `ensureContextWindowCacheLoaded()` can still be awaiting models.json or discovery. When that task resumes, it writes context entries from the old `cfg` back into the cleared cache, so removed or changed model context windows can persist after hot reload; add a generation or cancellation guard before async loads write shared context state.
  - confidence: 0.88
