## Summary

Keep open for maintainer handling: the linked model-catalog bug is already fixed on current main, but this PR is a broader refactor with a source-proven context-window race, missing real behavior proof, a conflicting branch, and an architecture choice about replacing targeted invalidation with a registry.

## What This Changes

The PR adds a config-derived cache registry, registers model catalog, models.json readiness, and context-window caches for models config changes, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's targeted model-catalog invalidation unless maintainers explicitly want a broader registry; if they do, add generation/cancellation protection to context-window invalidation and require runtime proof.

## Reproduction Assessment

Yes, for the PR-introduced bug by source inspection: start a context-window load, invalidate after its awaits begin, and the older task can write stale context data back into the cleared cache. I did not run tests because this review is read-only.

## Solution Assessment

No, not in this form. A registry could be maintainable if maintainers want that architecture, but this branch needs a context-window generation guard and real runtime proof before it can replace current targeted invalidation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Guard in-flight context-window loads:** `src/agents/context.ts:93-95`

Add a generation or cancellation guard before async loads write shared context state.
