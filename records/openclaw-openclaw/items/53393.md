## Summary

Keep this PR open, but it should not merge as-is: the broader registry refactor is partly superseded by current main's targeted model-catalog reload handling and the PR introduces a source-reproducible context-window cache race.

## What This Changes

The PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's guarded model-catalog refresh, and pursue a broader config-derived cache registry only if the context-window invalidation is generation-guarded and maintainers still want the refactor.

## Reproduction Assessment

Yes. By source inspection of the PR diff and current `ensureContextWindowCacheLoaded()` flow, clearing `loadPromise` while an older async load is in flight lets that older task resume after hot reload and write stale context-window entries back into the shared cache.

## Solution Assessment

No. A registry could be maintainable, but this implementation needs generation or cancellation protection for context-window invalidation and a rebase around current main's targeted model-catalog hot-reload handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
