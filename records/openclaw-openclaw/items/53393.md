## Summary

Keep open: current main already fixes the linked model-catalog hot-reload bug, but this PR's broader registry is not on main and the branch still has a source-level context-window race, missing real behavior proof, and merge conflicts.

## What This Changes

The PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's targeted model-catalog hot-reload fix unless maintainers explicitly want a broader registry; if they do, add generation/cancellation protection for context-window invalidation and require runtime proof.

## Reproduction Assessment

Yes, for the PR-introduced race by source inspection: start a context-window load, invalidate after its awaits begin, and the older task can write stale context data back into the cleared cache. I did not run tests because this review is read-only.

## Solution Assessment

No, not in this form. A registry could be maintainable if maintainers choose that architecture, but this branch needs a context-window generation guard, conflict resolution, and real runtime proof before replacing current targeted invalidation.
