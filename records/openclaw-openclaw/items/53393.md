## Summary

Keep open: current main already covers the linked model-catalog hot-reload bug, but this broader registry PR still needs maintainer architecture judgment, real behavior proof, and a fix for a context-window cache race introduced by the branch.

## What This Changes

The PR adds a config-derived cache registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's targeted model-catalog invalidation unless maintainers want a broader registry; if they do, add generation/cancellation protection to context-window invalidation and require runtime proof.

## Reproduction Assessment

Yes, for the PR-introduced race by source inspection: start a context-window load, invalidate after its awaits begin, and the older task can write stale context data back into the cleared shared cache. I did not run tests because this review is read-only.

## Solution Assessment

No, not in this form. A registry could be maintainable if maintainers want that architecture, but the branch needs a context-window generation guard and real runtime proof before it can replace current targeted invalidation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Guard context-window loads before clearing cache:** `src/agents/context.ts:94-95`

Add a generation or cancellation guard before async loads write shared context state.
