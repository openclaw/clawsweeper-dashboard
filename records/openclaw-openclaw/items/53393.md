## Summary

Keep open for maintainer handling. Current main has a targeted model-catalog hot-reload fix for the linked user-visible bug, but it does not implement this PR’s broader config-derived cache registry; the PR is also stale against current runtime-state code and has a concrete context-window cache race that should block merging as-is.

## What This Changes

This PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches against `models` config changes, calls the registry from Gateway hot reload, and adds registry unit tests.

## Best Possible Solution

Do not merge this branch as-is. Either close it after maintainer confirmation that the linked model-catalog bug is sufficiently covered by current main, or replace it with a narrow current-main patch for context-window cache invalidation that uses a generation guard and targeted regression coverage.

## Reproduction Assessment

Yes for the original model-catalog symptom: the linked issue gives concrete steps, and current main has source/tests for model-related hot reload cache clearing. Yes for the PR-specific race in review form: trigger hot reload while `ensureContextWindowCacheLoaded()` is in flight, then observe stale context-window data written after the clear; the PR does not include a regression test for that path.

## Solution Assessment

No. A registry could be a maintainable direction, but this PR clears `loadPromise` without preventing the old async context loader from writing stale data after invalidation; current main's generation-guarded model-catalog approach is safer for that cache.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Guard stale context loads before clearing the cache:** `src/agents/context.ts:94`
  - body: Resetting `loadPromise` during hot reload lets an already-running `ensureContextWindowCacheLoaded()` continue after `MODEL_CONTEXT_TOKEN_CACHE.clear()` and write old model windows back into the cache. Add a generation/cancellation guard so pre-reload loads cannot repopulate stale entries after invalidation.
