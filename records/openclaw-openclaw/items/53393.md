## Summary

Keep open: this PR is not safe to merge as-is because the new context-window invalidator can clear cache state while an older async context load later repopulates stale entries. Current main already has targeted, generation-guarded model-catalog hot-reload handling, so the broader registry refactor needs maintainer direction before repair work.

## What This Changes

The PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep the targeted main implementation unless maintainers want the registry refactor; if they do, add generation-guarded context-window invalidation and focused regression coverage before merge.

## Reproduction Assessment

Yes, by source inspection. In the PR, hot reload clears `MODEL_CONTEXT_TOKEN_CACHE` and `loadPromise` while an older `ensureContextWindowCacheLoaded()` task can still resume after awaits and write old discovered/configured entries back.

## Solution Assessment

No. A registry could be maintainable, but the context-window invalidation path needs a generation or cancellation guard like the Gateway model-catalog refresh before this is safe to merge.
