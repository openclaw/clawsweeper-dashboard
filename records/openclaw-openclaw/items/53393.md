## Summary

Keep open for maintainer handling: the linked model-catalog hot-reload bug is already solved on current main, but this broader refactor is not merge-ready because it introduces an unguarded context-window cache invalidation race and still lacks real behavior proof.

## What This Changes

The PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's targeted, guarded model-catalog hot-reload handling unless maintainers explicitly want a broader registry; if they do, rework context-window invalidation with a generation guard and require real runtime proof.

## Reproduction Assessment

Yes for the PR-introduced race, by source inspection: an older `ensureContextWindowCacheLoaded()` task can resume after awaits and repopulate the shared cache after the new invalidator clears it. The original linked model-catalog hot-reload bug is already addressed on current main.

## Solution Assessment

No, not in this form. A registry could be maintainable, but the proposed context-window invalidation needs generation/cancellation protection and maintainers should decide whether the broader refactor is still wanted after current main's targeted fix.
