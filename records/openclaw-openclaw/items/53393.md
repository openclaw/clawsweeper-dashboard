## Summary

Keep open, but do not merge as-is. Current main already handles the original model-catalog hot-reload symptom with targeted guarded refresh behavior, while this broader refactor still introduces an unguarded context-window cache race and lacks external real behavior proof.

## What This Changes

The branch adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's guarded model-catalog handling unless maintainers explicitly want the broader registry; if they do, rework context-window invalidation with a generation guard and add real behavior proof.

## Reproduction Assessment

Yes, by source inspection of the PR diff. The new context-window invalidator can clear shared state while an older ensureContextWindowCacheLoaded() task is still able to resume after awaits and write stale context data back.

## Solution Assessment

No, not in current form. A registry could be maintainable, but the context-window path needs generation/cancellation protection and maintainers should first decide whether this refactor is still wanted after main's targeted fix.
