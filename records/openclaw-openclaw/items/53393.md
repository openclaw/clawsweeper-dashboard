## Summary

Keep open: the linked model-catalog bug is already handled on current main, but this refactor still has a blocking context-cache race, no contributor real behavior proof, and a refactor-only architecture choice for maintainers.

## What This Changes

The PR adds a config-derived cache invalidation registry, registers model catalog, models.json readiness, and context-window caches, and calls the registry from Gateway hot reload.

## Best Possible Solution

Keep current main's targeted model-catalog hot-reload handling unless maintainers explicitly want the broader registry; if they do, add generation/cancellation protection to context-window invalidation and require real runtime proof.

## Reproduction Assessment

Yes for the PR-introduced race, by source inspection. The proposed invalidator can clear shared context state while an older ensureContextWindowCacheLoaded task resumes after awaits and writes stale data back.

## Solution Assessment

No, not in this form. A registry may be maintainable if maintainers want it, but this implementation needs a context-window generation guard and runtime proof before replacing current main's targeted invalidation.
