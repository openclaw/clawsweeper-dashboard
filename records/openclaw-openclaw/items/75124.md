## Summary

Keep open. Current main still resolves tool-dispatched skill slash commands through the raw OpenClaw tool factory and only applies owner-only filtering, while normal agent turns go through the layered effective tool-policy pipeline. The linked closing PR #75125 is open and unmerged, so this issue should stay tracked until that or an equivalent security-reviewed fix lands.

## What This Changes

Require user-invocable skills using `command-dispatch: tool` to honor the same session-effective tool policy as normal agent turns.

## Best Possible Solution

Review and land #75125 or an equivalent narrow patch that resolves inline tool-dispatched skills through the shared `createOpenClawCodingTools(...)` effective-policy path, with regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure a restricted profile or leaf subagent, add a user-invocable skill that dispatches to a denied tool such as `sessions_spawn`, and current main resolves the tool from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. Reusing `createOpenClawCodingTools(...)` with the canonical runtime-policy session and target session metadata is the narrowest maintainable fix; duplicating the profile, group, sandbox, and subagent filters in the inline path would risk future drift.
