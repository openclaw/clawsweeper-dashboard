## Summary

Keep open: current `main` still routes `command-dispatch: tool` skill slash commands through raw OpenClaw tool construction plus owner-only filtering, while the documented and normal agent path applies profile, group, sandbox, sender, and subagent policy. The remaining fix path is already covered by overlapping open PRs, especially https://github.com/openclaw/openclaw/pull/78525 and https://github.com/openclaw/openclaw/pull/46793, so this should stay open until a canonical implementation lands.

## What This Changes

Skill slash commands using `command-dispatch: tool` should honor the same effective tool-policy pipeline as normal agent turns.

## Best Possible Solution

Land or replace one canonical policy-aware inline dispatch fix so skill tool commands reuse the effective tool surface, with regression coverage for allow/deny, group, sandbox, sender, and subagent policy layers.

## Reproduction Assessment

Yes. Source inspection on current main shows the inline skill tool path still constructs raw tools and applies only owner-only filtering; a restricted `tools.deny` or leaf-subagent skill slash-command test would exercise the failing path.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve inline skill dispatch through the shared effective-tool policy path, not to add another one-off filter after raw tool construction.
