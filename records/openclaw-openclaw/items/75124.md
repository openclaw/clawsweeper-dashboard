## Summary

Keep open. Current main still matches the reported policy gap: inline tool-dispatched skill commands build the raw OpenClaw tool set and apply only owner-only filtering, while the normal session tool path uses the effective profile, group, sandbox, provider, and subagent policy pipeline. The linked fix PR #75125 is open and unmerged, so this issue should remain open until that or an equivalent reviewed fix lands.

## What This Changes

Ensure `command-dispatch: tool` skill slash commands resolve tools through the same effective tool-policy pipeline as normal session turns.

## Best Possible Solution

Review and land #75125 or an equivalent narrow patch that resolves inline tool-dispatched skill commands through `createOpenClawCodingTools(...)`, forwards the same runtime policy context as normal session turns, and adds regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: define a user-invocable skill with `command-dispatch: tool` and `command-tool: sessions_spawn`, invoke it from a restricted or leaf-subagent session, and current main resolves it from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing effective tool builder and policy pipeline for inline tool dispatch instead of duplicating raw tool construction in the inline-action branch.
