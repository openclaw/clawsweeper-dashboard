## Summary

Current main still has the reported policy gap: inline skill tool dispatch builds raw OpenClaw tools and then only applies owner-only filtering, while the normal coding-tool path applies profile, group, sandbox, and subagent policy. Because open PR #75125 uses closing syntax for this issue, the issue should stay open until that PR, or a replacement, lands.

## What This Changes

The issue asks for `command-dispatch: tool` skill slash commands to use the same effective tool policy as normal agent turns.

## Best Possible Solution

Review and land #75125 or a narrow replacement that switches inline tool dispatch to `createOpenClawCodingTools` with canonical session, group, sender, sandbox, model, and subagent context plus regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence reproduction path: define a user-invocable skill with `command-dispatch: tool` for a tool filtered out of the effective policy, then invoke it through the inline branch that constructs raw tools and only owner-filters them.

## Solution Assessment

Yes as a fix direction, but it is not on main yet. The narrow maintainable solution is to reuse the same effective coding-tool builder and context that normal command/tool surfaces already use, then keep this issue open until the linked PR merges.
