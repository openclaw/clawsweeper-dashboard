## Summary

Current main still has the reported policy mismatch: inline skill tool dispatch builds the raw OpenClaw tool set and only applies owner-only filtering, while the normal command/tool surface uses the full coding-tool policy pipeline. This issue should stay open because #75125 is an open closing PR for the same bug and has not merged yet.

## What This Changes

The issue asks for `command-dispatch: tool` skill slash commands to resolve tools through the same effective tool-policy pipeline as normal agent turns.

## Best Possible Solution

Land a narrow fix that makes inline tool-dispatch skill commands build their tool list through `createOpenClawCodingTools(...)` with the canonical runtime-policy session context and regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a user-invocable `command-dispatch: tool` skill currently reaches `createOpenClawTools(...)` plus owner-only filtering instead of the effective coding-tool policy pipeline. I did not run a live slash-command scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing `createOpenClawCodingTools(...)` pipeline with the same session/group/sender/model context used by normal command handling. Current main has not done that yet, and #75125 is the active candidate implementation.
