## Summary

Keep open. Current main still builds inline tool-dispatched skill commands from raw `createOpenClawTools(...)` and applies only owner-only filtering; #75125 is an open, unmerged closing PR, so this issue should remain tracked until that or an equivalent fix lands.

## What This Changes

The issue asks inline `command-dispatch: tool` skill slash commands to honor the same effective tool policy as normal agent turns.

## Best Possible Solution

Review and land #75125, or replace it with a narrow equivalent patch that routes inline tool-dispatched skill commands through the canonical effective-policy tool builder and adds regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: use a restricted profile or leaf subagent, add a user-invocable `command-dispatch: tool` skill targeting a denied tool such as `sessions_spawn`, and current main resolves it from the raw tool set before only owner-only filtering.

## Solution Assessment

Yes. Reusing `createOpenClawCodingTools(...)` with the canonical policy session is the narrowest maintainable fix; duplicating profile, group, sandbox, and subagent filters in the inline path would create another drift point.
