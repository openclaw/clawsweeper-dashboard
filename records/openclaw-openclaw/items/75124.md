## Summary

Keep open. Current main still has the reported authorization-policy gap: inline `command-dispatch: tool` skill slash commands build the raw OpenClaw tool set and only apply owner-only filtering, while normal agent turns use the effective tool-policy pipeline. The linked fix PR #75125 is open and unmerged, so this issue should stay open until that PR or an equivalent reviewed security hardening patch lands.

## What This Changes

Skill slash commands using `command-dispatch: tool` should honor the same effective tool policy as normal session tool construction rather than building raw tools.

## Best Possible Solution

Review and land #75125 or an equivalent narrow patch that routes inline tool-dispatched skill commands through `createOpenClawCodingTools(...)` with the canonical runtime-policy session key and target-session group/subagent metadata, plus regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: define a user-invocable skill with `command-dispatch: tool` targeting a restricted tool such as `sessions_spawn`, invoke it from a restrictive or leaf-subagent session, and current main resolves the tool from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. The best fix direction is to reuse the existing effective tool-policy pipeline instead of duplicating partial filtering in inline dispatch; the open PR #75125 is the current implementation candidate that should be reviewed for that boundary.
