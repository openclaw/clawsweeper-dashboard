## Summary

Keep open. Current main still has the reported authorization-policy gap: inline skill slash commands with `command-dispatch: tool` build raw tools and only apply owner-only filtering, while normal command/tool surfaces use the effective tool-policy pipeline. The linked closing PR #75125 is open and unmerged, so this issue should remain open until that PR or an equivalent reviewed fix lands.

## What This Changes

The issue requests that tool-dispatched skill slash commands honor the current session's effective tool policy instead of resolving tools from the raw OpenClaw tool set.

## Best Possible Solution

Review and land #75125 or an equivalent narrow patch that resolves inline tool-dispatched skill commands through the session-effective `createOpenClawCodingTools(...)` path, with regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: configure a restricted session or leaf subagent, add a user-invocable skill whose frontmatter dispatches to a restricted tool such as `sessions_spawn`, and current main resolves it from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. The best direction is to reuse the existing effective tool-policy builder for inline tool dispatch; duplicating profile, group, sandbox, and subagent filters in this branch would be more fragile.
