## Summary

Keep open. Current main still matches the reported authorization-policy gap, and the linked closing PR is open and unmerged, so this issue should remain tracked until that fix or an equivalent security-reviewed patch lands.

## What This Changes

The issue asks that skill slash commands using `command-dispatch: tool` honor the current session’s effective tool policy instead of dispatching through a raw tool set.

## Best Possible Solution

Review and land #75125 or an equivalent narrow patch that resolves inline tool-dispatched skill commands through the same session-effective `createOpenClawCodingTools(...)` path used by normal agent turns, with regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: configure a restricted profile or leaf subagent, add a user-invocable skill that dispatches to a denied tool such as `sessions_spawn`, and current main resolves that tool from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. Reusing the existing effective tool-policy builder is the narrowest maintainable fix; duplicating profile, group, sandbox, and subagent filters in the inline path would risk future drift.
