## Summary

Keep open. Current main still resolves inline `command-dispatch: tool` skill commands from the raw OpenClaw tool set and only applies owner-only filtering; #75125 is an open, unmerged closing implementation candidate, so this issue should remain tracked until that or an equivalent fix lands.

## What This Changes

The issue asks for `command-dispatch: tool` skill slash commands to honor the same effective tool policy as normal agent turns.

## Best Possible Solution

Review and land #75125, or replace it with a narrow equivalent patch that routes inline tool-dispatched skill commands through `createOpenClawCodingTools(...)` using the canonical runtime policy session and regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: in a restricted profile or leaf subagent, a user-invocable `command-dispatch: tool` skill for a denied tool such as `sessions_spawn` is resolved from raw `createOpenClawTools(...)` instead of the filtered coding-tool surface.

## Solution Assessment

Yes. Reusing `createOpenClawCodingTools(...)` is the narrowest maintainable fix because it already owns owner-only, profile, provider, group, sandbox, and subagent policy filtering.
