## Summary

Keep open: current main still routes `command-dispatch: tool` skill slash commands through raw OpenClaw tool construction plus owner-only filtering, while the documented effective-policy path applies profile, group, sandbox, and subagent restrictions. The remaining work is already covered by overlapping open fix PRs, especially https://github.com/openclaw/openclaw/pull/78525.

## What This Changes

The issue requests that user-invocable skill slash commands using `command-dispatch: tool` honor the same effective tool policy as normal agent turns.

## Best Possible Solution

Land or replace one canonical policy-aware inline dispatch fix so skill tool commands reuse the effective tool surface, with regression coverage for allow/deny, group, sandbox, and subagent policy layers.

## Reproduction Assessment

Yes. Source inspection on current main shows the inline skill tool path still constructs raw tools and applies only owner-only filtering; a restricted `tools.deny` or leaf-subagent skill slash-command test would exercise the failing path.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve inline skill dispatch through a shared effective-tool helper or the normal policy-aware tool pipeline, not to add another ad hoc filter after raw tool construction.
