## Summary

Keep open. Current main still matches the reported authorization-policy gap: inline skill slash commands using `command-dispatch: tool` build the raw OpenClaw tool set and only apply owner-only filtering, while the normal session tool path applies profile, provider, group, sandbox, and subagent policy. The linked fix PR #75125 is open and unmerged, so this issue should remain open until that PR or an equivalent reviewed fix lands.

## What This Changes

The issue requests that user-invocable skills with `command-dispatch: tool` use the same session-effective tool policy as normal agent turns instead of raw tool construction.

## Best Possible Solution

Review and land #75125 or an equivalent narrow patch that resolves inline tool-dispatched skill commands through the session-effective tool surface, with regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: add a user-invocable skill with `command-dispatch: tool` and a restricted tool such as `sessions_spawn`, invoke it from a restrictive or leaf-subagent session, and current main resolves the tool from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing `createOpenClawCodingTools(...)` effective policy pipeline for inline tool dispatch rather than duplicating raw tool construction; current main has not done that yet.
