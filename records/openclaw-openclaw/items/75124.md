## Summary

Keep open: current main still lets `command-dispatch: tool` skill slash commands build the raw OpenClaw tool set and then apply only owner-only filtering, so the effective profile, group, sandbox, and subagent policy pipeline is still bypassed. A related hook-hardening PR has merged, but the active full-policy fix remains open at https://github.com/openclaw/openclaw/pull/78525.

## What This Changes

Requests that `command-dispatch: tool` skill slash commands honor the current session's effective tool policy instead of dispatching from the raw tool list.

## Best Possible Solution

Land or replace the focused policy-aware inline dispatch fix so skill tool commands reuse the same effective tool surface as normal turns, with regression coverage for allow/deny, group, sandbox, and subagent policy layers.

## Reproduction Assessment

Yes. Source inspection on current main shows a direct `createOpenClawTools(...)` plus owner-only filter path for skill tool dispatch; a focused restricted `tools.deny` or leaf-subagent skill slash-command test would exercise the failing path.

## Solution Assessment

Yes. The narrow maintainable direction is to resolve inline skill dispatch through a policy-aware helper or shared effective-tool pipeline, not to add another ad hoc filter after raw tool construction.
