## Summary

Keep open. Current main still routes user-invocable `command-dispatch: tool` skill slash commands through the raw OpenClaw tool factory and only owner-only filtering, while the normal agent/tool prompt path applies the layered effective tool-policy pipeline. The linked closing PR #75125 is open and unmerged, so this issue should remain tracked until that fix or an equivalent security-reviewed patch lands.

## What This Changes

The issue requests that tool-dispatched skill slash commands honor the same effective tool policy as normal agent turns, including profile, group/channel, sandbox, and subagent restrictions.

## Best Possible Solution

Review and land #75125, or an equivalent narrow patch, so inline tool-dispatched skill commands resolve tools through `createOpenClawCodingTools(...)` with the current session's canonical policy context and focused regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: in a restricted profile or leaf subagent session, add a user-invocable `command-dispatch: tool` skill targeting a denied tool such as `sessions_spawn`; current main resolves it from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. Reusing the existing effective tool-policy builder is the narrowest maintainable direction; duplicating profile, group, sandbox, and subagent filters in the inline path would risk future drift.
