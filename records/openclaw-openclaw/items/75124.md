## Summary

Keep open. Current main at 8093ae60292f4d9009b3d48f234b39b5cbc538e5 still matches the reported authorization-policy gap: inline skill slash commands using `command-dispatch: tool` build raw OpenClaw tools and only apply owner-only filtering, while normal agent turns use the layered effective tool-policy pipeline. The linked fix PR #75125 is open and unmerged, so this issue should remain open until that PR or an equivalent reviewed fix lands.

## What This Changes

Skill slash commands using `command-dispatch: tool` should resolve requested tools through the same session-effective tool-policy pipeline as normal agent turns.

## Best Possible Solution

Review and land #75125 or an equivalent narrow patch that resolves tool-dispatched skill slash commands through the session-effective tool surface, with regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: configure a restricted or leaf-subagent session, add a user-invocable skill whose `command-dispatch: tool` targets a restricted tool such as `sessions_spawn`, and current main will resolve it from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: reuse the same effective policy resolver used by normal agent turns and `/tools`, rather than maintaining a separate raw-tool dispatch path for user-invocable skill commands.
