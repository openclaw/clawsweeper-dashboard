## Summary

Keep open. Current main still matches the reported authorization-policy gap: inline skill slash commands with `command-dispatch: tool` build the raw OpenClaw tool set and only apply owner-only filtering, while normal agent turns apply profile, provider, group, sandbox, and subagent policy. The linked closing PR #75125 is open and unmerged, so this issue should remain open until that PR or an equivalent reviewed fix lands.

## What This Changes

The issue reports that user-invocable skill slash commands using `command-dispatch: tool` can bypass the current session's effective tool policy.

## Best Possible Solution

Review and land #75125 or an equivalent narrow patch that resolves inline tool-dispatched skill commands through the same session-effective `createOpenClawCodingTools(...)` pipeline used by normal agent turns, with regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: configure a restricted session or leaf subagent, add a user-invocable skill whose frontmatter dispatches to a restricted tool such as `sessions_spawn`, and current main resolves that tool from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

Yes. Routing inline tool dispatch through the existing effective tool-policy builder is the narrowest maintainable direction; duplicating profile, group, sandbox, and subagent filters in the inline path would be riskier.
