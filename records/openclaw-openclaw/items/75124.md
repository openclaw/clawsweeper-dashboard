## Summary

Keep open. Current main still resolves `command-dispatch: tool` skill slash commands from the raw OpenClaw tool factory and only applies owner-only filtering, so the documented effective tool policy can still be bypassed; #75125 is the open closing implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that `command-dispatch: tool` skill slash commands bypass the session's effective tool policy by resolving raw tools and only applying owner-only filtering.

## Best Possible Solution

Review and land #75125, or replace it with a narrow equivalent patch that routes inline tool-dispatched skill commands through `createOpenClawCodingTools(...)` with regression coverage for restricted policies.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a restricted session or leaf subagent, add a user-invocable `command-dispatch: tool` skill targeting a denied tool such as `sessions_spawn`, and the inline branch resolves it from raw tools before only owner filtering.

## Solution Assessment

Yes. Reusing the existing effective tool-policy builder is the narrowest maintainable fix; duplicating policy filters inside the inline branch would leave another drift point.
