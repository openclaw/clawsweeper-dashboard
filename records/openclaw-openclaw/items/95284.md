## Summary

Keep open: this is a focused provider-boundary fix for a real source-reproducible reasoning/message-delivery bug, but the branch is too narrow because it only seals native reasoning before text and leaves the reasoning-to-tool-call lane able to start before `thinking_end`. It also still lacks redacted real DeepSeek/OpenClaw behavior proof, and several open candidate PRs target the same linked issue without a merged proof-positive canonical fix yet.

## What This Changes

The PR changes the OpenAI-completions stream adapter so an open native thinking block is finished before the first visible text delta, with an idempotent guard to avoid finishing the same block twice.

## Best Possible Solution

Adopt one canonical provider-boundary fix that emits exactly one `thinking_end` before text or tool-call lanes, covers event order in provider tests, and includes redacted real DeepSeek/OpenClaw proof before merge.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: feed OpenAI-completions chunks with `reasoning_content` followed by `content`, and current main opens thinking but does not emit `thinking_end` until the end-of-stream finish loop.

## Solution Assessment

No for this branch as the best fix. The provider adapter is the right layer, but this patch is too narrow until it covers tool-call starts, adds event-order tests, and supplies real DeepSeek/OpenClaw proof.
