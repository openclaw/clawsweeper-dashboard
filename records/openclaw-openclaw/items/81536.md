## Summary

Keep open: the reported stdout JSON corruption is valid, and the branch targets the right startup gap, but the current implementation is too broad and would regress existing parse-only/pass-through `--json` behavior.

## What This Changes

The branch imports `routeLogsToStderr` in `runCli` and calls it immediately after console capture when argv contains `--json`, before lazy plugin command registration.

## Best Possible Solution

Keep the PR open and change the implementation so early plugin-registration diagnostics are kept off stdout only for actual JSON-output commands, while parse-only and pass-through `--json` behavior remains unchanged.

## Reproduction Assessment

Yes. Current source shows lazy plugin registration happens before `parseAsync`/`preAction`, and the PR body gives a concrete `openclaw memory search ... --json` reproduction with after-fix terminal output.

## Solution Assessment

No. The direction is right, but the implementation should use the existing command-aware JSON-output contract or temporarily scope routing to the plugin-registration window instead of permanently flipping stderr routing for any argv `--json` token.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep early stderr routing scoped to JSON output mode:** `src/cli/run-main.ts:647-648`
  - body: This flips stderr routing as soon as any argv token looks like `--json`, before Commander knows the selected command. That is broader than the current contract: `config set --json` is parse-only and tests assert it must not route logs, while pass-through tokens after `--` should also be ignored. Scope the early routing to actual JSON-output mode or only around plugin registration, preserving `--` handling.
  - confidence: 0.9
