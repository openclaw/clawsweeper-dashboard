## Summary

Keep open. On current main, `claude-cli` still has no path that translates `exec-approvals.json` allowlist entries into Claude Code's internal Bash permission layer; the code only maps OpenClaw YOLO to Claude `bypassPermissions` and otherwise launches Claude directly with its own permission handling. Related closed PRs #68682 and #70723 addressed forced bypass defaults, not this allowlist-forwarding request.

## What This Changes

Request that agents using the `claude-cli` backend honor OpenClaw `tools.exec.security: "allowlist"` and `exec-approvals.json` for Claude Code internal exec.

## Best Possible Solution

Keep this open for a security/design decision, then implement an Anthropic plugin-owned permission bridge that maps the effective OpenClaw exec policy to a supported Claude Code permission mechanism or diagnoses `allowlist` as unsupported/fail-closed for that backend. It should cover global and per-agent policy, fresh/resumed/live Claude sessions, docs, and regression tests for allowed and denied commands.

## Reproduction Assessment

Yes. The issue provides concrete config and command steps, and current-main static inspection gives a high-confidence reproduction of the missing bridge: the Claude CLI launch path does not read or evaluate `exec-approvals.json` before Claude Code handles internal Bash/tool execution.

## Solution Assessment

Unclear. Forwarding the policy is the right end state, but the exact mechanism needs maintainer review against Claude Code's current permission APIs; a safer interim may be to fail closed or warn for `claude-cli` plus `security: "allowlist"` until a real bridge exists. ([]())
