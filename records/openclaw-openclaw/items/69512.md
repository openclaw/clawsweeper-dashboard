## Summary

Keep open. Current `origin/main` at `15dc7d3f57eab7fe42022e6d25512c6acf9bf6a2` still has the same gap described in the prior ClawSweeper comment: Claude CLI sessions get YOLO permission mapping and MCP tool allowlisting, but not OpenClaw `exec-approvals.json` allowlist enforcement.

## What This Changes

The issue requests forwarding OpenClaw exec allowlist policy into Claude CLI backend sessions so Claude Code internal exec obeys the same allow/deny decisions as native exec.

## Best Possible Solution

Implement a Claude CLI-owned permission bridge for fresh, resumed, and live stdio sessions, or fail closed with clear diagnostics when equivalent OpenClaw allowlist enforcement is unavailable.

## Reproduction Assessment

Yes. The issue provides concrete live steps, and current main statically reproduces the missing bridge because the Claude CLI path launches without evaluating `exec-approvals.json` while native gateway exec does evaluate it.

## Solution Assessment

Unclear. Forwarding or fail-closed behavior is the right end state, but the safest design needs maintainer review against Claude Code's permission APIs and OpenClaw's effective allowlist semantics.
