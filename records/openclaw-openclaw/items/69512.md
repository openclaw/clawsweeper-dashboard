## Summary

Keep open. Current main and the latest release still do not forward `exec-approvals.json` allowlist rules into Claude CLI native tool decisions; the newer related control-request PR is relevant but still open, reviewed as security-sensitive, and does not yet provide a clean allowlist-parity implementation.

## What This Changes

The issue asks OpenClaw to forward per-agent `exec-approvals.json` allowlist policy into Claude CLI backend sessions so native Claude tool execution matches OpenClaw exec enforcement.

## Best Possible Solution

Land a security-reviewed Anthropic CLI permission bridge that uses OpenClaw's canonical effective exec policy and approvals state for Claude native tool decisions, or fail closed with clear diagnostics where equivalent allowlist semantics cannot be represented safely.

## Reproduction Assessment

Yes, at source level. Current main still leaves allowlist-mode Claude CLI sessions without `exec-approvals.json` permission forwarding while native OpenClaw exec continues to evaluate and deny allowlist misses.

## Solution Assessment

Unclear. The requested parity is the right end state, but maintainers need to choose whether Claude native tools should get command-by-command allowlist decisions, fail closed with MCP guidance, or use another documented Claude permission-control surface.
