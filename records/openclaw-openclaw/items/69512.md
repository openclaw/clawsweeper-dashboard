## Summary

Keep open: current main and v2026.6.10 still do not forward OpenClaw exec allowlist decisions into Claude CLI native tool use. The runtime now fails closed for restrictive policy, which is safer than the original fail-open behavior, but it still does not provide allowlist parity and the remaining fix needs maintainer security/product direction.

## What This Changes

Request to forward `exec-approvals.json` and `tools.exec.security: "allowlist"` semantics into `claude-cli` backend sessions so allowed commands run and non-allowed commands are denied consistently across runners.

## Best Possible Solution

Choose one maintainer-approved design that preserves OpenClaw approval and audit semantics across Claude CLI native tool use, then land focused tests and redacted live Claude CLI proof for restrictive allowlist/on-miss and full/off modes.

## Reproduction Assessment

Yes, from source: current main denies Claude CLI native Bash for restrictive exec policy rather than evaluating the command against the allowlist, while native gateway exec still evaluates `exec-approvals.json`. I did not run a live Claude CLI turn in this read-only review.

## Solution Assessment

Unclear. Forwarding allowlist decisions into Claude native permission handling is a plausible solution, but maintainers still need to choose the permanent security boundary versus an MCP approval route or explicit unsupported fail-closed behavior.
