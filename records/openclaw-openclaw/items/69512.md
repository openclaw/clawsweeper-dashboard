## Summary

Keep open: current main and v2026.6.6 now fail closed for restrictive Claude CLI native tool requests, but they still do not evaluate `exec-approvals.json` allowlist entries for allowed Claude CLI Bash commands, and the remaining fix is a security-sensitive product decision.

## What This Changes

The issue requests forwarding OpenClaw exec allowlist policy into `claude-cli` backend sessions so allowlisted commands run and non-allowlisted commands are denied consistently.

## Best Possible Solution

Keep the fail-closed default, then design a maintainer-approved path that either evaluates Claude native Bash requests against OpenClaw’s canonical exec policy or exposes a default-off MCP exec route that preserves the existing approval and audit machinery.

## Reproduction Assessment

Yes, at source level: current main’s Claude live `can_use_tool` handler denies all restrictive policies without evaluating the requested command against the allowlist, while native OpenClaw exec still evaluates `approvals.allowlist`. I did not rerun a live Claude CLI session.

## Solution Assessment

Unclear: fail-closed behavior is safer than the original silent bypass, but it is not the requested allowlist parity and the best permanent path needs maintainer security/product direction.
