## Summary

Keep this open. Current main still does not bridge OpenClaw exec allowlist policy into Claude Code internal tool execution, so the reported runner mismatch remains valid and security-sensitive.

## What This Changes

Forward OpenClaw `exec-approvals.json` allowlist policy into `claude-cli` backend sessions so Claude Code internal exec honors the same per-agent exec policy as native OpenClaw exec.

## Best Possible Solution

Keep the issue open for maintainer security/design review, then implement a Claude CLI-owned permission bridge for fresh, resumed, and live stdio sessions or fail closed with clear diagnostics when equivalent enforcement is unavailable.

## Reproduction Assessment

Yes. The issue provides a concrete live reproduction, and current source statically reproduces the missing bridge because the Claude CLI backend is launched without reading or evaluating `exec-approvals.json`.

## Solution Assessment

Unclear. Forwarding policy is the right end state, but the safest implementation needs maintainer review against Claude Code's supported `allowedTools`/`disallowedTools`/permission-prompt mechanisms and fail-closed semantics.
