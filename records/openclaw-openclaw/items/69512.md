## Summary

Keep this open. Current main still maps only OpenClaw YOLO exec policy to Claude Code `bypassPermissions`; I did not find a path that forwards or enforces `exec-approvals.json` allowlist entries for Claude Code's internal exec/tool execution, so the reported policy mismatch remains valid and security-sensitive.

## What This Changes

Forward OpenClaw `exec-approvals.json` allowlist policy into `claude-cli` backend sessions so Claude Code internal exec honors the same per-agent exec policy as native OpenClaw exec.

## Best Possible Solution

Keep the issue open for a maintainer security/design decision, then implement a Claude CLI-owned permission bridge that maps OpenClaw's effective global and per-agent exec policy into a supported Claude Code permission mechanism for fresh, resumed, and live stdio sessions. If Claude Code cannot support equivalent enforcement, OpenClaw should clearly warn or fail closed for `claude-cli` agents configured with `security: "allowlist"`, and docs/tests should cover that contract.

## Reproduction Assessment

Yes. The issue provides concrete reproduction steps and current source gives a high-confidence static reproduction of the missing bridge: the Claude CLI backend normalizes Claude permission args and launches Claude without reading or evaluating `exec-approvals.json`; I did not run a live Claude Code session.

## Solution Assessment

Unclear. Forwarding the policy is the right end state, but the mechanism needs maintainer review against Claude Code's current permission contract; a safer alternative may be to reject or warn on `claude-cli` plus allowlist mode until equivalent enforcement is available.
