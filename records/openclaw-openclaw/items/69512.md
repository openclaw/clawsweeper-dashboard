## Summary

Keep open. Current main still does not forward `exec-approvals.json` allowlists into Claude CLI native Bash execution; it only maps OpenClaw YOLO policy to Claude `bypassPermissions`, while allowlist-mode Claude sessions remain on Claude Code's own permission path.

## What This Changes

Request to make Claude CLI backend sessions enforce the same OpenClaw exec allowlist decisions as native OpenClaw exec.

## Best Possible Solution

Add an Anthropic plugin-owned permission bridge for fresh, resumed, and live Claude CLI sessions that maps OpenClaw exec policy to Claude native tool decisions, or intentionally fail closed with clear diagnostics when equivalent allowlist semantics cannot be represented safely.

## Reproduction Assessment

Yes. The issue's steps are backed by current source: allowlist-mode Claude CLI args do not receive allowlist rules, while native OpenClaw exec evaluates `exec-approvals.json`; I did not run a live Claude Code session in this checkout.

## Solution Assessment

Unclear. Bridging the policy is the right end state, but maintainers need to decide whether Claude native Bash should be command-by-command allowlisted, denied/fail-closed, or routed through a documented Claude permission-control surface.
