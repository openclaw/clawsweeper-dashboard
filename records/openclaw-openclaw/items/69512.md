## Summary

Keep open. Current main still does not forward OpenClaw `exec-approvals.json` allowlists into Claude CLI native Bash execution; the merged Claude permission work maps only YOLO policy to `bypassPermissions`, and the related open control-request PR does not implement allowlisted command execution.

## What This Changes

Request that Claude CLI backend sessions honor OpenClaw `exec-approvals.json` allowlists for backend-internal exec calls.

## Best Possible Solution

Add an Anthropic plugin-owned permission bridge for fresh, resumed, and live Claude CLI sessions that maps OpenClaw exec policy to Claude Code native tool decisions, or fail closed with clear diagnostics when equivalent allowlist semantics cannot be represented safely.

## Reproduction Assessment

Yes. Source inspection and the issue's live steps show current main leaves allowlist-mode Claude CLI sessions on Claude Code's own permission path while native OpenClaw exec evaluates `exec-approvals.json`; I did not run a live Claude Code session in this read-only review.

## Solution Assessment

Unclear. Bridging or failing closed is the right end state, but maintainers need to choose whether Claude native Bash should be denied, YOLO-only, or checked command-by-command through Claude's permission prompt surface.
