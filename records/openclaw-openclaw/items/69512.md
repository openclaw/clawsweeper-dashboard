## Summary

Keep open: current main and v2026.6.10 still do not forward OpenClaw exec allowlist decisions into Claude CLI native tool use. The runtime now fails closed for restrictive policy, which is safer than the original fail-open report, but allowlisted commands are still not evaluated on that backend and the open related PR only targets a narrower ask-on-miss path.

## What This Changes

Request to make Claude CLI backend sessions honor OpenClaw `exec-approvals.json`, `tools.exec.security: "allowlist"`, and related safe-bin policy instead of relying on Claude Code's separate native permission mode.

## Best Possible Solution

Choose one maintainer-approved Claude CLI permission boundary that preserves OpenClaw approval and audit semantics, then land focused tests plus redacted live Claude CLI proof for restrictive allowlist/on-miss and full/off modes.

Yes, source-level: current main denies Claude CLI native Bash for restrictive exec policy rather than evaluating the command against the allowlist, while native gateway exec still evaluates `exec-approvals.json`. I did not run a live Claude CLI turn in this read-only review.

## Solution Assessment

Unclear. Forwarding allowlist decisions into Claude native permission handling is plausible, but maintainers still need to choose the permanent security boundary versus an MCP approval route or explicit unsupported fail-closed behavior.
