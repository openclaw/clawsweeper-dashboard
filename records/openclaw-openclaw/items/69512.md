## Summary

Keep open. Current main still does not bridge OpenClaw `exec-approvals.json` allowlists into Claude CLI internal Bash execution; the current implementation only maps YOLO exec policy to Claude `bypassPermissions`.

## What This Changes

Request to forward OpenClaw exec allowlist policy into `claude-cli` backend sessions so Claude Code internal exec follows the same allow/deny decisions as native OpenClaw exec.

## Best Possible Solution

Implement an Anthropic plugin-owned permission bridge for fresh, resumed, and live Claude CLI sessions, or fail closed with clear diagnostics when equivalent OpenClaw allowlist semantics cannot be represented.

## Reproduction Assessment

Yes, by source inspection and the issue’s live steps. Current main keeps Claude permission mode unset for allowlist policy while native gateway exec evaluates `exec-approvals.json`; I did not run a live Claude Code session in this read-only review.

## Solution Assessment

Unclear. Forwarding or fail-closing the policy is the right end state, but the exact Claude Code permission-rule or prompt-tool mapping needs maintainer security design before implementation.
