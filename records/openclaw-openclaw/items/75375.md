## Summary

Keep open. Current main still runs boot-md BOOT.md work inside the in-process gateway startup hook path and awaits the per-agent boot agent turn; the gateway dispatch uses a delayed `void` call, but there is still no boot-md timeout or isolation once the hook begins running.

## What This Changes

The issue asks for boot-md startup work to stop making the gateway appear unavailable during restarts when BOOT.md agent work is slow.

## Best Possible Solution

Run boot-md on a boot-md-owned bounded background or isolated path after gateway readiness, preserving per-agent workspace dedupe, session-key routing, silent reply behavior, outbound delivery, and session mapping cleanup.

## Reproduction Assessment

Yes. Configure boot-md with a BOOT.md whose agent turn is slow or pending; current main follows `runBootChecklist -> runStartupTasks -> runBootOnce -> agentCommand` and awaits that chain inside the hook handler.

## Solution Assessment

Partly yes. A boot-md-specific detached or budgeted execution path is the narrow maintainable direction, but a broad change to all startup tasks or relying only on the existing top-level `void triggerInternalHook` would not address the in-process slow-run boundary.
