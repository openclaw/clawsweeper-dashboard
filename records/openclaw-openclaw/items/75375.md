## Summary

Keep open. Current main still runs bundled boot-md BOOT.md work inside the Gateway process through an awaited handler/task/agent-command path with no boot-specific timeout or isolation, so a slow boot turn can still make the restart window look unavailable even though the startup hook is fired in the background.

## What This Changes

Request that bundled boot-md run BOOT.md startup work without making Gateway clients appear offline during restarts.

## Best Possible Solution

Run BOOT.md execution on a bounded background path after Gateway readiness, preserving per-agent workspace dedupe, session-key routing, silent reply behavior, and session mapping cleanup.

## Reproduction Assessment

Yes. Enable boot-md with a BOOT.md whose agent turn is slow or pending; current main schedules the gateway startup hook in-process and then awaits runStartupTasks -> runBootOnce -> agentCommand with no boot-specific budget or isolation.

## Solution Assessment

Yes, with a narrow refinement: detach or budget boot-md execution specifically after Gateway readiness rather than making all startup tasks fire-and-forget.

## Repair Work Prompt

Fix the boot-md startup availability bug reported in https://github.com/openclaw/openclaw/issues/75375. Current main fires gateway:startup in the background, but the bundled boot-md handler still awaits runStartupTasks -> runBootOnce -> agentCommand in the Gateway process with no boot-specific budget, so a slow BOOT.md turn can saturate the event loop during restart. Implement a narrow fix that runs boot-md BOOT.md execution after Gateway readiness on a bounded background path, or adds an equivalent boot-md-specific timeout/isolation boundary. Preserve per-agent workspace dedupe, per-agent session-key routing, silent reply semantics, non-success logging, and session mapping cleanup. Avoid broad changes to shared runStartupTasks unless the new API is backwards-compatible and restart-sentinel behavior remains covered. Add focused regression coverage proving slow boot-md work does not keep Gateway startup/readiness or client handshakes pending, and add a CHANGELOG.md Fixes entry crediting the human reporter as appropriate.
