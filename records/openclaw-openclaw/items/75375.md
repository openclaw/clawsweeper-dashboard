## Summary

Keep open. Current main has a delayed, unawaited gateway startup hook dispatch, but boot-md still awaits BOOT.md agent work inside the gateway process, and the normal gateway agent RPC path has a stronger accept/yield/abort pattern that boot-md does not use. The recent report is concrete and distinct from the earlier boot-md delivery/routing issue.

## What This Changes

The issue requests that bundled boot-md BOOT.md startup work run detached or bounded so gateway restarts accept UI, Telegram, SSE, and health clients immediately.

## Best Possible Solution

Keep the boot-md feature, but move BOOT.md execution onto a detached post-ready agent-run path that yields before heavy runner setup, records/logs completion, supports bounded abort/timeout, and preserves per-agent workspace dedupe plus the existing message-tool routing semantics.

## Reproduction Assessment

Yes. Enable boot-md with a BOOT.md that triggers a slow provider/tool/network operation, restart the gateway, and poll UI/Telegram/health or gateway RPCs while the BOOT run starts; source inspection shows that BOOT work is still awaited through in-process agentCommand.

## Solution Assessment

No. Current main's unawaited 250ms startup-hook timer is only a partial mitigation; the safer solution is to reuse or extract the gateway agent accepted/yield/background-run pattern, with boot-specific completion logging and cancellation.

Fix https://github.com/openclaw/openclaw/issues/75375. Current main schedules the internal gateway:startup hook on a timer, but bundled boot-md still awaits runStartupTasks, and runBootOnce awaits agentCommand directly inside the gateway process. Prefer reusing or extracting the gateway agent accepted/yield/background-task pattern from src/gateway/server-methods/agent.ts; otherwise add a boot-specific post-ready scheduler that yields before heavy agent setup, records/logs completion, and applies a bounded timeout/abort so a stuck BOOT.md cannot monopolize the gateway.
