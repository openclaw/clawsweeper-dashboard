## Summary

Keep open. Current main still runs boot-md BOOT.md work in the Gateway process through the awaited `runStartupTasks -> runBootOnce -> agentCommand` path; the Gateway startup dispatcher uses `setTimeout`/`void triggerInternalHook`, but there is still no boot-specific detachment, worker isolation, or short budget around the boot agent turn itself.

## What This Changes

The issue asks for boot-md startup work to run in the background or under a hard budget so Gateway restarts remain responsive while BOOT.md is executing.

## Best Possible Solution

Run boot-md agent execution on a bounded background path that cannot starve Gateway readiness or request handling, while preserving per-agent workspace dedupe, session-key routing, silent reply behavior, outbound delivery, and session mapping cleanup.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: enable boot-md with a slow BOOT.md and the scheduled `gateway:startup` handler follows `runStartupTasks -> runBootOnce -> agentCommand` in the Gateway process. I did not run a live restart in this read-only review.

## Solution Assessment

Yes, the narrow maintainable repair is boot-md-specific detached or budgeted execution after readiness. Changing global internal-hook dispatch semantics would be broader and could affect unrelated hooks.

## Repair Work Prompt

Fix the boot-md startup availability bug. Current main still runs BOOT.md agent turns in the Gateway process through `gateway:startup`, with `boot-md` awaiting `runStartupTasks -> runBootOnce -> agentCommand`; make the boot-md agent work bounded/background so HTTP/WS/UI/channel responsiveness is not held by a slow BOOT.md run. Keep the change boot-md-specific unless a small generic seam is necessary, and do not broadly change global internal-hook ordering or semantics. Preserve per-agent workspace dedupe, per-agent session-key routing, missing/empty/failed BOOT.md logging, silent reply behavior, outbound message delivery, and session mapping restore cleanup. Add regression coverage showing startup/hook dispatch can return while a slow boot run remains pending, and add a user-facing `CHANGELOG.md` fix entry.
