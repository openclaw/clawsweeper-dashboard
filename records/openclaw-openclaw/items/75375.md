## Summary

Keep open. Current main still lacks a bounded or detached boot-md execution path; the startup hook eventually runs BOOT.md through the Gateway process and awaits the agent run, with no boot-specific timeout or off-process boundary. The report is concrete and recent enough for a focused repair PR.

## What This Changes

Request that the bundled boot-md gateway startup hook run BOOT.md work without making the Gateway appear unavailable during restarts.

## Best Possible Solution

Move BOOT.md execution onto a bounded background path that cannot monopolize client-facing Gateway responsiveness during restart, and add regression coverage proving a long boot run does not hold Gateway readiness or client handshakes. Preserve the existing boot prompt, per-agent workspace dedupe, and session cleanup semantics.

## Reproduction Assessment

Yes. The issue gives version, platform, config, steps, user-visible symptoms, and event-loop metrics; static inspection confirms BOOT.md work still awaits the agent run through the boot-md task chain. I did not perform the live restart scenario because this review was read-only.

## Solution Assessment

No current solution is present on main. The existing fire-and-forget startup dispatcher is helpful but not sufficient by itself because boot-md still runs unbounded in the Gateway process once dispatched; the safer fix is a boot-md-specific detached or budgeted execution path with tests.
