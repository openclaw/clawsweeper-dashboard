## Summary

Keep open. Current main still runs the boot-md BOOT.md agent turn through an awaited internal startup hook chain with no boot-specific detached worker or hard budget, so the reported restart availability risk remains actionable.

## What This Changes

Request that boot-md run BOOT.md startup work without holding the gateway startup hook path or making clients observe an outage.

## Best Possible Solution

Run BOOT.md execution on a bounded background path after Gateway readiness while preserving per-agent scope, workspace dedupe, session-key routing, silent delivery, and session mapping cleanup.

## Reproduction Assessment

Yes. Configure boot-md with a BOOT.md whose agent turn is slow or pending; current main's handler -> runStartupTasks -> runBootOnce chain awaits through agentCommand, and existing tests assert the awaited path.

## Solution Assessment

Yes. A boot-md-specific detached or budgeted execution path is the narrow maintainable direction; changing all startup tasks or all internal hooks would be broader and riskier.

## Repair Work Prompt

Fix the boot-md restart availability bug reported in https://github.com/openclaw/openclaw/issues/75375. Keep the change narrowly scoped to boot-md/Gateway startup behavior: BOOT.md work should run after Gateway readiness on a bounded background path so slow or stuck agent turns do not hold the internal startup hook path or make clients observe a restart outage. Preserve per-agent scope, workspace dedupe, session-key routing, deliver=false behavior, outbound message-tool semantics, and session mapping cleanup. Do not make all startup tasks or all internal hooks fire-and-forget without proving restart-sentinel and other startup task semantics remain correct. Add focused regression coverage for a slow/pending BOOT.md run returning promptly from the startup hook boundary, and add a single-line CHANGELOG.md fix entry because this is user-facing availability behavior.
