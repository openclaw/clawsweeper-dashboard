## Summary

Keep open. Current main still runs boot-md BOOT.md agent work through an awaited internal-hook task path with no boot-md-specific readiness boundary or hard timeout, so the reported restart availability risk remains actionable.

## What This Changes

The issue asks for boot-md BOOT.md startup execution to run off the gateway readiness path or under a hard budget so restarts remain responsive.

## Best Possible Solution

Boot-md should schedule BOOT.md agent turns after Gateway readiness or through an explicitly bounded background task, preserving per-agent behavior and cleanup while surfacing failures through logs.

## Reproduction Assessment

Yes. Enable boot-md with a BOOT.md that performs a slow or pending agent turn, restart the Gateway, and observe readiness/UI/channel behavior while current main follows handler -> runStartupTasks -> runBootOnce -> agentCommand without a hard budget.

## Solution Assessment

Yes. The narrow maintainable fix is boot-md-specific detachment or budgeting after readiness; making all startup tasks or all internal hooks fire-and-forget would be riskier because other startup users expect awaited ordered execution.
