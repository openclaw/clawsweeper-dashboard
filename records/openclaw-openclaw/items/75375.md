## Summary

Keep open. Current main still has the reported boot-md execution shape: the gateway startup hook is dispatched asynchronously, but once fired the boot-md handler awaits the startup-task runner, and the boot task awaits `agentCommand` without a boot-specific timeout, off-process boundary, or detached background budget. The issue is recent, concrete, source-confirmed, and not covered by the related open boot-md prompt-disclosure PR.

## What This Changes

The issue requests that bundled boot-md BOOT.md startup work run without making gateway restart availability depend on a full agent run.

## Best Possible Solution

Move BOOT.md execution onto a bounded detached/background path that cannot keep client-facing gateway readiness dependent on a full agent run, while preserving per-agent workspace dedupe, existing BOOT.md prompt semantics, session mapping restore behavior, and outbound message-tool delivery. Add regression coverage that proves long boot work does not block gateway readiness or client handshakes.

## Reproduction Assessment

Yes. The issue gives concrete version, platform, config, restart steps, user-visible symptoms, and event-loop metrics, and static inspection of current main still shows the boot-md startup path awaiting the full `agentCommand` chain. I did not run a live gateway restart because this review is read-only.

## Solution Assessment

No current main solution is present. The narrowest maintainable solution is a boot-md-specific detached or budgeted execution path with focused gateway startup/readiness tests, rather than treating the existing fire-and-forget hook dispatch as sufficient.
