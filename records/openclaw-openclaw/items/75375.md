## Summary

Keep open. Current main still lacks a boot-md-specific detached/background boundary or timeout budget: the startup hook dispatch is delayed/fire-and-forget, but once invoked the boot-md handler waits through serial startup tasks and the BOOT.md agent run. The issue is recent, concrete, and not superseded by the adjacent open BOOT.md prompt-disclosure PR, so this is a focused repair candidate rather than a cleanup close.

## What This Changes

The issue requests that bundled boot-md BOOT.md execution stop making the gateway appear unavailable during restarts by moving boot work onto a bounded background path.

## Best Possible Solution

Move BOOT.md execution onto a bounded detached path that cannot keep client-facing gateway readiness unavailable during restart, while preserving per-workspace dedupe, per-agent session routing, BOOT.md prompt semantics, and message-tool delivery. Add regression coverage for long boot runs and document any new timeout/background behavior if it becomes user-visible.

## Reproduction Assessment

Yes. The issue gives version, platform, config, steps, visible symptoms, and event-loop metrics; source inspection confirms the boot path still awaits BOOT.md agent execution through the startup-task chain. I did not run the live restart scenario in this read-only review.

## Solution Assessment

Yes, the requested direction is the right maintainable fix: boot work should be bounded or detached from gateway readiness. The implementation should avoid broad hook semantic changes unless tests prove a generic startup-task budget is safe.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75375. Current main still lets bundled boot-md run BOOT.md through an awaited startup-task/agentCommand chain after gateway:startup dispatch, which can make restarts appear unavailable while a long BOOT.md agent run is active. Implement a bounded detached/background path for boot-md so gateway client readiness is not held by BOOT.md work. Preserve existing behavior for per-agent workspace resolution, workspace dedupe, BOOT.md prompt semantics, session mapping restore, and outbound message-tool delivery. Prefer a boot-md/startup-task-specific boundary or timeout over changing all internal hook semantics unless the generic change is clearly covered. Do not conflate this with PR #75128, which is about BOOT.md prompt disclosure. Add focused regression tests proving a long boot run does not hold gateway startup/client readiness, and keep existing boot-md delivery behavior intact. Add a user-facing CHANGELOG.md fix entry crediting Thanks @adzendo. Do not touch dependencies, lockfiles, unrelated provider/channel defaults, or broad plugin hook contracts.
