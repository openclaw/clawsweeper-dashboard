## Summary

Keep open. Current main still runs boot-md BOOT.md work through the in-process internal hook runner and awaits the boot agent turn inside that hook; the gateway schedules the startup hook asynchronously, but there is no boot-specific timeout or isolation to keep a slow agent run from degrading restart availability.

## What This Changes

The issue asks for boot-md BOOT.md execution during gateway startup to be detached or budgeted so restarts do not make clients appear unavailable while the boot agent runs.

## Best Possible Solution

Run boot-md on a bounded background path after Gateway readiness, preserving per-agent workspace dedupe, session-key routing, silent reply behavior, outbound delivery, and session mapping cleanup.

## Reproduction Assessment

Yes. Configure boot-md with a BOOT.md whose agent turn is slow or pending; current main follows runBootChecklist -> runStartupTasks -> runBootOnce -> agentCommand and awaits that promise inside the hook handler.

## Solution Assessment

Yes. A boot-md-specific detached or budgeted execution path is the narrow maintainable fix; broad startup-task fire-and-forget semantics would risk unrelated restart-sentinel behavior.
