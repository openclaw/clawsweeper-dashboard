## Summary

Keep open. Current main still has the boot-md startup hook awaiting the BOOT.md task chain through `runStartupTasks` and `runBootOnce`, with no boot-specific background boundary or hard runtime budget once the work starts in the Gateway process. The report is recent, concrete, and narrow enough for a focused availability fix.

## What This Changes

Request that bundled `boot-md` BOOT.md startup work run detached or bounded so long agent runs cannot make Gateway clients appear offline after restart.

## Best Possible Solution

Add a boot-md-owned detached or bounded background execution path that returns promptly from the startup hook, records/logs completion and failures, and preserves per-agent workspace dedupe, session-key routing, silent reply behavior, and session mapping cleanup. Add regression coverage with a deliberately long boot task proving Gateway startup/client readiness is not held by BOOT.md execution.

## Reproduction Assessment

Yes. The issue gives version, platform, config, restart steps, client symptoms, event-loop metrics, and a workaround; source inspection confirms the BOOT.md work is still awaited through the boot-md task chain on current main.

## Solution Assessment

Yes. The suggested detached or hard-budgeted boot-md execution is the narrow maintainable direction; a broad hook-dispatcher rewrite is not required to address this specific availability bug.

## Repair Work Prompt

Fix the boot-md startup availability bug reported in https://github.com/openclaw/openclaw/issues/75375. Current main still awaits BOOT.md agent execution through `src/hooks/bundled/boot-md/handler.ts` -> `src/gateway/startup-tasks.ts` -> `src/gateway/boot.ts`, so a slow BOOT.md run can make Gateway clients appear offline after restart. Implement a boot-md-specific detached or bounded background execution path that returns promptly from the `gateway:startup` hook while preserving per-agent workspace dedupe, per-agent session-key routing, silent reply semantics, session mapping snapshot/restore cleanup, and non-success logging. Avoid changing generic internal hook semantics or restart-sentinel behavior unless a small documented seam is clearly necessary. Add focused regression coverage with a long/pending mocked `runBootOnce` proving the boot-md startup hook or Gateway startup dispatch does not wait for BOOT.md completion, plus coverage that background failures are logged. Add a user-facing `CHANGELOG.md` Fixes entry thanking @adzendo. Validate with targeted tests and formatting; run `pnpm check:changed` in Testbox before handoff if runtime files change.
