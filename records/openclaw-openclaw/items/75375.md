## Summary

Keep open. Current main still awaits boot-md per-agent BOOT.md execution through the startup-task runner and `agentCommand`, with no boot-specific background boundary or hard timeout, so the reported restart availability problem remains actionable.

## What This Changes

The issue asks for bundled boot-md Gateway startup work to run in the background or under a hard budget so Gateway clients can connect immediately after restart.

## Best Possible Solution

Move BOOT.md execution onto a bounded background path that cannot monopolize Gateway responsiveness during restart, while preserving per-agent workspace dedupe, session-key routing, silent reply semantics, and session mapping cleanup. Add regression coverage that a long boot run does not hold Gateway readiness or client handshakes.

Yes. The issue provides version, config, restart steps, client symptoms, event-loop metrics, and a workaround; static inspection confirms current main still has an awaited boot-md agent-run chain. A live restart was not run because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable direction is a boot-md-specific detached or budgeted execution path, plus tests; a broad plugin API change is not needed for the reported bundled-hook availability bug.

Fix https://github.com/openclaw/openclaw/issues/75375. The bundled `boot-md` `gateway:startup` hook currently awaits per-agent BOOT.md agent runs through `runStartupTasks` and `runBootOnce`, which can make the Gateway appear unavailable during restarts when BOOT.md performs slow network-bound work. Implement a bounded background path so Gateway readiness/client handshakes are not held by boot-md work. Keep the fix narrow to boot-md/startup-task/Gateway startup surfaces unless a small generic seam is clearly needed. Preserve per-agent workspace dedupe, per-agent session-key routing, silent reply semantics, non-success boot logging, and session mapping restore. Add regression coverage proving a long BOOT.md run does not block Gateway startup hook dispatch/readiness. Add a single-line `CHANGELOG.md` fix entry for this user-facing availability bug with `Fixes #75375. Thanks @adzendo.` Do not disable boot-md and do not introduce bundled-extension IDs into generic core paths.
