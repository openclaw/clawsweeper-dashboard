## Summary

Keep open. Current main still runs boot-md BOOT.md work inside the Gateway process by awaiting the boot startup tasks and the agent turn, with no boot-specific timeout or isolation; the shipped release named in the report predates any fix and main still shows the same behavior.

## What This Changes

The issue asks for boot-md startup work to stop making Gateway restarts appear unavailable while a slow BOOT.md agent run is in progress.

## Best Possible Solution

Keep boot-md enabled but run its BOOT.md agent work on a bounded background path after Gateway readiness, preserving per-agent workspace dedupe and session mapping cleanup.

## Reproduction Assessment

Yes. Enable boot-md with a BOOT.md whose agent turn stays slow or pending; current main follows gateway:startup -> runBootChecklist -> runStartupTasks -> runBootOnce -> agentCommand and awaits that chain inside the hook path.

## Solution Assessment

Yes. A boot-md-specific detached or budgeted execution path is the narrowest maintainable fix; changing all startup-task or internal-hook semantics would risk unrelated restart and lifecycle behavior.
