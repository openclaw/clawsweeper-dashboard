## Summary

Keep open. Current main still lacks a bounded or detached boot-md execution path; source, tests, and docs show gateway startup still dispatches boot-md to a task chain that awaits each BOOT.md agent run, matching the concrete recent availability report.

## What This Changes

The issue asks for boot-md BOOT.md startup work to run without making Gateway restarts appear unavailable to Telegram, UI, SSE, and health-check clients.

## Best Possible Solution

Move BOOT.md execution onto a bounded background path that cannot monopolize client-facing Gateway responsiveness during restart, while preserving the existing boot prompt, per-agent workspace dedupe, per-agent session routing, and session mapping cleanup. Add regression coverage proving a long boot run does not hold Gateway readiness or client handshakes.

## Reproduction Assessment

Yes. The issue gives version, platform, config, steps, user-visible symptoms, and event-loop metrics; static inspection confirms BOOT.md work still awaits the agent run through the boot-md task chain. I did not perform the live restart scenario because this review was read-only.

## Solution Assessment

No current solution is present on main. The suggested fire-and-forget direction is useful, but the safer maintainable fix is a boot-md-specific detached or budgeted execution path with tests that prove Gateway responsiveness during a long BOOT.md run.
