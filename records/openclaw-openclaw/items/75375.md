## Summary

Keep open. Current main still runs bundled boot-md BOOT.md agent work through the Gateway startup hook path with no boot-specific background boundary or hard budget, so the reported restart availability risk remains actionable.

## What This Changes

The issue requests moving bundled boot-md BOOT.md execution off the startup availability path or bounding it so Gateway clients can connect promptly during restarts.

## Best Possible Solution

Run BOOT.md execution on a bounded background path after Gateway readiness, preserving per-agent workspace dedupe, session-key routing, silent reply semantics, and mapping cleanup.

## Reproduction Assessment

Yes. Configure boot-md with a BOOT.md whose agent turn remains slow or pending; current main's handler -> runStartupTasks -> runBootOnce chain awaits that work, and existing tests assert the awaited path rather than readiness isolation.

## Solution Assessment

Yes. Detaching or budgeting boot-md specifically is the narrow maintainable fix; broad fire-and-forget changes to all internal hooks would need extra lifecycle and error-handling proof.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75375. The bundled boot-md Gateway startup hook currently allows slow BOOT.md agent turns to keep the internal startup hook path busy and run heavy work in the Gateway process during restart. Make BOOT.md execution run on a bounded background path after Gateway readiness so UI, Telegram, and SSE handshakes stay responsive while the boot ping finishes. Prefer a boot-md-specific scheduler or timeout around runBootChecklist/runBootOnce; preserve per-agent scope, workspace dedupe, session-key routing, silent reply semantics, warning/debug logging, and session mapping restoration. Avoid changing generic internal hook dispatch or runStartupTasks semantics unless the narrow fix requires it and tests cover existing restart-sentinel behavior. Add regression coverage proving the Gateway startup hook dispatch/startup sidecar path returns before a delayed boot run settles while failures/skips are still observed. Add a user-facing CHANGELOG.md fix entry.
