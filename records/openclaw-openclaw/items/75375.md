## Summary

Keep open. Current main still runs boot-md BOOT.md work through the in-process gateway:startup hook and awaits the boot agent turn without boot-specific detachment or a short timeout, so the restart availability bug remains actionable.

## What This Changes

Request that bundled boot-md run BOOT.md startup work without making the Gateway appear unavailable during restarts.

## Best Possible Solution

Run boot-md on a bounded background path after Gateway readiness while preserving per-agent workspace dedupe, session-key routing, silent reply behavior, outbound delivery, and session mapping cleanup.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: enable boot-md with a slow BOOT.md, then gateway:startup follows runBootChecklist -> runStartupTasks -> runBootOnce -> agentCommand and awaits the boot turn. I did not run a live restart because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is boot-md-specific detached or budgeted execution after readiness. Changing global internal-hook dispatch semantics would be broader and risk unrelated hook behavior.
