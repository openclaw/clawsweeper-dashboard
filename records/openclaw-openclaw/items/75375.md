## Summary

Keep open. Current main still runs boot-md BOOT.md work inside the in-process gateway startup hook path and awaits the boot agent turn without a boot-specific isolation or short timeout, so the restart availability bug remains actionable.

## What This Changes

Request boot-md gateway-startup BOOT.md execution to run detached or bounded so the gateway stays responsive immediately after restart.

## Best Possible Solution

Run boot-md on a bounded background path after Gateway readiness while preserving per-agent workspace dedupe, session-key routing, silent reply behavior, outbound delivery, and session mapping cleanup.

## Reproduction Assessment

Yes. Configure boot-md with a BOOT.md whose agent turn is slow or pending; current main follows runBootChecklist -> runStartupTasks -> runBootOnce -> agentCommand and awaits that promise inside the hook handler.

## Solution Assessment

Yes, a boot-md-specific detached or budgeted execution path is the narrow maintainable fix; changing all startup-task or internal-hook semantics would risk unrelated restart-sentinel and hook behavior.
