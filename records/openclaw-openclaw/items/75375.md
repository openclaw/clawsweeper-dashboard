## Summary

Keep open. Current main still runs bundled `boot-md` through an awaited in-process chain with no boot-specific timeout, abort signal, or background task boundary; the existing `setTimeout` only defers hook dispatch and does not isolate a long BOOT.md agent run from the Gateway process. The report is fresh, detailed, unprotected, and not superseded by a merged fix or linked open PR in the provided context.

## What This Changes

The issue asks for bundled `boot-md` Gateway startup work to run in a bounded detached/background path so BOOT.md checks cannot make the Gateway appear unavailable after restart.

## Best Possible Solution

Keep the existing boot-md semantics but move BOOT.md execution onto a bounded background path that cannot monopolize Gateway responsiveness: preserve per-agent workspace dedupe, session-key routing, silent reply behavior, outbound message delivery, and session mapping cleanup while adding regression coverage for a long boot run during restart.

## Reproduction Assessment

Yes. The issue gives version, platform, config, restart steps, client symptoms, event-loop metrics, and a workaround; static inspection confirms the current-main boot-md path still awaits `boot-md -> runStartupTasks -> runBootOnce -> agentCommand` with no boot-specific timeout.

## Solution Assessment

Yes, the requested direction is the right shape. A boot-md-specific detached or budgeted execution path is narrower and safer than changing all hook dispatch semantics, though shared timeout support could be reused if it preserves existing hook contracts.
