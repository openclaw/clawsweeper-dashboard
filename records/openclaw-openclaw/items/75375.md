## Summary

Keep open. Current main still has the same awaited boot-md execution chain described in the report: the gateway startup hook path eventually awaits each BOOT.md agent run with no boot-specific background boundary or hard budget, so the availability bug remains actionable.

## What This Changes

The issue requests making bundled boot-md BOOT.md gateway-startup work non-blocking or time-bounded so Gateway clients remain responsive during restarts.

## Best Possible Solution

Move BOOT.md execution onto a bounded background path that cannot monopolize Gateway responsiveness during restart, while preserving the existing boot prompt, per-agent workspace dedupe, session-key routing, silent reply semantics, and session mapping cleanup. Add regression coverage proving a long boot run does not hold Gateway readiness or client handshakes.

## Reproduction Assessment

Yes. The issue gives version, platform, config, restart steps, client symptoms, event-loop metrics, and a workaround; static inspection confirms BOOT.md work still awaits the agent run through the boot-md task chain.

## Solution Assessment

Yes for the requested direction, but no current implementation is present on main. The narrow maintainable fix is a boot-md-specific detached or budgeted execution path with tests, not a broad plugin API change.
