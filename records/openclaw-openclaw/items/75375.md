## Summary

Keep open. Current main still has the awaited boot-md execution chain described in the report: the bundled startup hook awaits the startup-task runner, the runner awaits each BOOT.md task, and runBootOnce awaits the agent run with no boot-specific background boundary or short budget. The existing ClawSweeper comment already identified this path, and the current re-check shows no relevant code change since then.

## What This Changes

The issue requests making bundled boot-md BOOT.md gateway-startup work non-blocking or time-bounded so Gateway clients stay responsive during restarts.

## Best Possible Solution

Move BOOT.md execution onto a bounded background path that cannot monopolize Gateway responsiveness during restart, while preserving the existing boot prompt, per-agent workspace dedupe, session-key routing, silent reply semantics, and session mapping cleanup. Add regression coverage proving a long boot run does not hold Gateway readiness or client handshakes.

## Reproduction Assessment

Yes. The issue gives version, platform, config, restart steps, client symptoms, event-loop metrics, and a workaround; static inspection confirms BOOT.md work still awaits the agent run through the boot-md task chain.

## Solution Assessment

Yes for the requested direction, but no implementation is present on main. The narrow maintainable fix is a boot-md-specific detached or budgeted execution path with tests, not a broad plugin API change.

Implement a bounded background/detached execution path for BOOT.md startup work so Gateway readiness and client handshakes are not held by a long run.
