## Summary

Keep open. The report is concrete, recent, and current main still has the same boot-md execution shape reported against v2026.4.29: gateway startup dispatch is delayed/fire-and-forget, but the boot-md handler still waits through the boot task runner and `agentCommand` without a boot-specific budget or off-process/background boundary. Since that fire-and-forget dispatcher already existed in the reported release, it is not enough evidence to close this as fixed.

## What This Changes

The issue asks for the bundled `boot-md` gateway startup hook to stop making restarts appear unavailable while BOOT.md agent work runs.

## Best Possible Solution

Move BOOT.md execution onto a bounded detached path that cannot keep the gateway's client-facing event loop unavailable during restart, while preserving per-agent workspace dedupe, existing BOOT.md prompt semantics, and outbound message-tool delivery. Add regression coverage that proves gateway readiness/client handshakes are not held by a long boot run and document any new timeout/background-task behavior.

## Reproduction Assessment

Yes. The issue gives version, platform, config, steps, user-visible symptoms, and event-loop metrics; static inspection confirms the configured boot path awaits `agentCommand` through the startup-task chain. I did not run the live restart scenario because this review was read-only.

## Solution Assessment

No current solution is present on main. The existing fire-and-forget gateway dispatcher already existed in v2026.4.29, so the safer repair is a boot-md-specific bounded background execution path or dispatcher budget with tests, not a close based on current behavior.
