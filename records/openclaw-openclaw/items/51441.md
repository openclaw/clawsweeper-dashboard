## Summary

Keep open. Current main has related status and fallback plumbing, but it still lacks a durable resolved backend model/provider contract in OpenAI-compatible transport output, embedded agent metadata, persisted session rows, and normal session_status details.

## What This Changes

Request to expose the actual backend model/provider used by routing proxies in session status, session persistence, and agent-visible runtime context.

## Best Possible Solution

Define one durable resolved-backend identity contract, then thread it from OpenAI-compatible responses through run metadata, session persistence, Gateway rows, and session_status without overwriting the user-selected model state.

## Reproduction Assessment

Yes, source-level only: current main lacks first-class resolved backend fields in the session row, embedded agent metadata, and normal session_status details. I did not establish a live routing-proxy reproduction in this read-only pass.

## Solution Assessment

Unclear pending maintainer direction. Capturing response.model is plausible, but the best fix needs a deliberate contract for requested versus active versus resolved model/provider and fallback semantics.
