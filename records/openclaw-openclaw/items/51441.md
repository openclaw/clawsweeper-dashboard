## Summary

Keep open. Current main and the latest stable release still do not carry response-side backend model identity from OpenAI-compatible transports into agent metadata, session persistence, GatewaySessionRow, or session_status output; the related open items at https://github.com/openclaw/openclaw/issues/22021, https://github.com/openclaw/openclaw/issues/7006, and https://github.com/openclaw/openclaw/issues/9016 are adjacent HTTP-header or cost surfaces, not a replacement for this runtime/session metadata request.

## What This Changes

Expose requested-versus-resolved backend model/provider/fallback metadata in session_status, session rows, and agent runtime text for OpenAI-compatible routing proxies.

## Best Possible Solution

Add an additive selected-versus-resolved model metadata contract at the OpenAI-compatible transport boundary, persist it separately from selected model state, and surface it consistently in status/runtime APIs while leaving LiteLLM tagging as a separate policy decision.

## Reproduction Assessment

Yes. The feature gap is source-reproducible: OpenAI-compatible response types expose model fields, but current main does not carry them into agent meta, persistence, GatewaySessionRow, or session_status; no live LiteLLM setup is needed to prove the missing plumbing.

## Solution Assessment

Yes for the additive selected-versus-resolved metadata direction. The exact API field names, text rendering, runtime injection wording, and optional LiteLLM tags still need maintainer product/API review before implementation.
