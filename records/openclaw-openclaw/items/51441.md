## Summary

Keep open. Current main still ignores returned backend model identifiers from OpenAI-compatible responses, keeps only one provider/model identity in agent and session metadata, and related open items cover adjacent transport-header, cost, or active-status display slices rather than replacing this session_status/runtime metadata feature.

## What This Changes

Expose the actual backend model returned by routing proxies separately from the requested alias in agent metadata, session rows, session_status, and runtime context.

## Best Possible Solution

Define an additive requested-vs-resolved model metadata contract, capture returned backend IDs at the OpenAI-compatible transport boundary, persist them without overwriting the requested alias, expose them through status/session surfaces, and treat LiteLLM tagging as a separate follow-up decision.

## Reproduction Assessment

Yes for the observable feature gap. A high-confidence source reproduction is an OpenAI-compatible proxy response where response.model or chunk.model differs from the requested alias; current main reads IDs/usage/status but has no resolved backend field to carry that value onward.

## Solution Assessment

Yes, the additive selected-vs-resolved metadata direction is the narrow maintainable solution. The exact public field names, status wording, runtime injection, and optional LiteLLM tags still need maintainer API/UX review before implementation.
