## Summary

Keep open. Current main and the latest release still have partial response-model and fallback concepts, but they do not carry routing-proxy backend identity through OpenAI-compatible Responses transport, agent metadata, persisted session rows, Gateway rows, and normal session_status output.

## What This Changes

Request to expose requested versus resolved backend model/provider and fallback state in agent runtime metadata, persisted session state, Gateway rows, CLI/status output, and session_status.

## Best Possible Solution

Define a maintainer-approved selected-versus-resolved backend metadata contract, then propagate it from OpenAI-compatible responses through agent meta, session storage, Gateway rows, session_status, and status/runtime text with focused source and live proxy tests.

## Reproduction Assessment

Yes, source-level: current main has responseModel in lower-level message types and one sibling transport, but the Responses transport, EmbeddedAgentMeta, persistSessionUsageUpdate, GatewaySessionRow, and session_status output still lack the requested propagation. I did not run a live routing proxy in this read-only pass.

## Solution Assessment

Unclear until maintainer direction: capturing upstream response.model is likely part of the right fix, but the best solution needs an explicit durable API/storage contract and selected-versus-resolved display semantics first.
