## Summary

Keep open. Current main still has no separate requested-vs-resolved backend model pipeline for OpenAI-compatible responses, and the related open items #22021 and #9016 cover adjacent HTTP visibility and cost metadata rather than this session/runtime metadata feature.

## What This Changes

Expose the actual backend provider/model from OpenAI-compatible responses in session metadata, `session_status`/status output, and agent runtime context while preserving the requested alias.

## Best Possible Solution

Add an additive metadata path that captures returned backend IDs in the OpenAI-compatible transports, carries them separately through agent meta and session persistence, exposes them in status/session APIs, and leaves proxy-specific tagging for a follow-up decision.

## Reproduction Assessment

Yes. A source-level reproduction is an OpenAI-compatible stream where `response.model` or `chunk.model` differs from the requested alias; current main has no carrier from those fields into agent meta, session persistence, or `GatewaySessionRow`.

## Solution Assessment

Yes. Capturing the returned model at the transport boundary and carrying it as separate resolved metadata is the narrow maintainable direction; the LiteLLM tag correlation should be split unless maintainers explicitly want it in the first change.
