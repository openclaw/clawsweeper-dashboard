## Summary

Keep open. Current main has fixed several stale status-display bugs, including the related internal handoff attribution bug, but it still does not expose a durable resolved backend model/provider contract for OpenAI-compatible routing proxies in session rows, session_status, or agent runtime metadata.

## What This Changes

Request resolved backend provider/model and fallback state in session status, session rows, persisted run metadata, and agent runtime context for LiteLLM/OpenRouter-style routing proxies.

## Best Possible Solution

Maintainers should choose one provider/session-state contract for resolved backend identity, then implement it through the OpenAI-compatible transport, run metadata persistence, Gateway session projection, and session_status output with focused upgrade-safe tests.

## Reproduction Assessment

Yes, source-level only: current main lacks first-class resolved backend fields in the session row and agent metadata, and OpenAI-compatible transport output still starts from the configured model id. I did not run a live routing-proxy request in this read-only pass.

## Solution Assessment

Unclear pending maintainer direction. Capturing `response.model` is plausible, but the durable API needs a clear decision on field names, fallback semantics, provider derivation, and how it differs from existing selected/active/fallback display state.
