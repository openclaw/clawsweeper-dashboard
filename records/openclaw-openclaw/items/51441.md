## Summary

Keep open: current main still has selected/active runtime model plumbing, but it does not carry response-side resolved backend identity from OpenAI-compatible transports into agent metadata, session persistence, Gateway session rows, or session_status/runtime output. The KeepMyClaw and CloudGPU comments add useful validation targets, and the related open issues are adjacent rather than replacements.

## What This Changes

The issue asks OpenClaw to capture the actual backend model/provider returned by OpenAI-compatible routing proxies, persist it, and surface it through session_status and runtime context.

## Best Possible Solution

Define an additive selected-versus-resolved backend identity contract at the OpenAI-compatible transport boundary, persist it separately from selected runtime model state, and surface it consistently through session_status, Gateway rows, CLI status, and agent runtime context while treating provider correlation tags as a separate policy decision.

## Reproduction Assessment

Yes. Source inspection on current main shows OpenAI-compatible response model data is not carried into agent metadata, session persistence, Gateway rows, or session_status output.

## Solution Assessment

Yes for the additive selected-versus-resolved metadata direction, but the exact public fields, status/runtime wording, and optional provider tags need maintainer product/API review before implementation.
