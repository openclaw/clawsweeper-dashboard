## Summary

Keep open: current main still exposes the selected/runtime model but not a separate provider-reported backend model for LiteLLM/OpenRouter-style routing proxies, and adjacent open items do not fully supersede the session-store and agent-runtime request.

## What This Changes

Expose provider-reported resolved model/provider and fallback state through session persistence, `session_status`, status output, and agent runtime context.

## Best Possible Solution

Add an additive core observability path that preserves the configured model while separately recording provider-reported backend identity when available, with focused transport, persistence, status, runtime-context, docs, and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was run; this is a feature request rather than a documented-behavior bug. Source inspection does confirm the current main path has no separate resolved backend model field from OpenAI-compatible responses through session status.

## Solution Assessment

Unclear pending maintainer API decision. Capturing provider-reported `response.model` is the likely narrow implementation direction, but the durable field contract and fallback semantics should be agreed before coding.
