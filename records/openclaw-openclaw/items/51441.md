## Summary

Keep open: current main and the latest release still do not capture provider-returned backend identity from OpenAI-compatible responses into agent metadata, session persistence, Gateway session rows, or session_status/CLI output. Existing winner/fallback trace helps OpenClaw-owned fallback paths, but it does not solve routing-proxy backend observability; the KeepMyClaw and CloudGPU comments add useful validation targets rather than making the core request obsolete.

## What This Changes

Request to capture provider-returned backend model/provider identity from routing proxies and expose selected, resolved, and fallback metadata through session persistence, session_status, CLI status, and agent runtime context.

## Best Possible Solution

Define an additive selected-versus-resolved backend identity contract at the OpenAI-compatible transport boundary, persist it separately from selected runtime model state, and expose it consistently through session_status, Gateway rows, CLI status, and agent runtime context while treating provider correlation tags as a separate policy decision.

## Reproduction Assessment

Yes, source-level. Current main ignores response.model in the OpenAI-compatible completed-response path and lacks resolved backend fields in agent metadata, session persistence, Gateway rows, and session_status output.

## Solution Assessment

Yes for the additive selected-versus-resolved metadata direction, but the exact public fields, status/runtime wording, and optional provider tags need maintainer product/API review before implementation.
