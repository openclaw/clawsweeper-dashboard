## Summary

Keep open. Current main still lacks a separate requested-vs-resolved backend model pipeline: OpenAI-compatible returned model IDs are ignored, session persistence stores only one model/provider identity, and related open issues cover adjacent HTTP-header or cost-display slices rather than this session_status/runtime metadata feature.

## What This Changes

Expose requested-vs-resolved backend model/provider/fallback metadata from OpenAI-compatible responses through agent meta, session persistence, session_status, CLI status, and runtime context.

## Best Possible Solution

Add an additive requested-vs-resolved model metadata pipeline that captures returned backend IDs, keeps the requested alias intact, persists resolved backend fields, exposes them in status/session surfaces, and handles LiteLLM tagging separately.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an OpenAI-compatible proxy stream where response.model or chunk.model differs from the requested alias; current main reads usage/status from those events but has no resolved backend field to carry that value.

## Solution Assessment

Yes. The requested additive metadata path is the narrow maintainable direction, with optional LiteLLM request tags deferred until the core resolved-model contract is settled.
