## Summary

Keep open: current main still does not carry response-side backend model identity from OpenAI-compatible transports into agent metadata, session persistence, Gateway session rows, or session_status/runtime surfaces. Related open items cover HTTP headers, OpenRouter cost, and active-status display, but none replace this selected-versus-resolved session/runtime metadata request.

## What This Changes

The issue requests exposing requested-versus-resolved backend model/provider metadata in session status, persisted session rows, and agent runtime context for routing-proxy deployments.

## Best Possible Solution

Add an additive selected-versus-resolved model metadata contract at the OpenAI-compatible transport boundary, persist it separately from selected model state, and surface it consistently in status/runtime APIs while treating outbound proxy tags as a separate policy decision.

## Reproduction Assessment

Yes, source-level. OpenAI-compatible response types expose model fields, but current main does not carry them into transport output, agent meta, persistence, GatewaySessionRow, or session_status; no live LiteLLM setup is needed to prove the missing plumbing.

## Solution Assessment

Yes for the additive selected-versus-resolved metadata direction. The exact public API names, rendering rules, runtime injection text, and optional LiteLLM tagging policy still need maintainer product/API review before implementation.
