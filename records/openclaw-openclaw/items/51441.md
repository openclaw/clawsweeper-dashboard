## Summary

Keep open. Current main still has selected/runtime and OpenClaw-managed fallback model surfaces, but it does not capture OpenAI-compatible proxy-returned backend model IDs or persist/expose requested-vs-resolved model identity for session_status or agent runtime context. The related open issues are adjacent transport/cost visibility requests, not a complete canonical replacement for this session metadata feature.

## What This Changes

The issue requests capturing OpenAI-compatible proxy-resolved backend model/provider metadata and exposing it through session persistence, session_status, status output, and agent runtime context.

## Best Possible Solution

Add an additive requested-vs-resolved model metadata pipeline: capture returned model IDs from OpenAI Responses and Chat Completions streams, thread them through assistant output and agent metadata without replacing the requested model identity, persist resolved backend fields on sessions, expose them in GatewaySessionRow, session_status, and status summaries, and add focused proxy-style tests for silent routing/fallback visibility.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an OpenAI-compatible streaming response where response.model or chunk.model differs from the requested alias; current main initializes output.model from model.id and never reads those returned model fields, so downstream agentMeta and session_status cannot observe the backend.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if implemented additively. Existing selected-vs-active fallback display should remain, but it is not a substitute for proxy-resolved response metadata; optional LiteLLM tagging can be deferred.
