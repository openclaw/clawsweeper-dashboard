## Summary

Current main still does not expose response-side resolved backend identity from OpenAI-compatible transports into agent metadata, session persistence, or session_status/runtime surfaces, so this should stay open for maintainer API/product review rather than be swept as implemented or duplicate.

## What This Changes

Request to add selected-versus-resolved backend model/provider metadata for routing-proxy responses and surface it through session_status, session rows, persisted usage, and agent runtime context.

## Best Possible Solution

Add an additive resolved backend identity contract at the OpenAI-compatible transport boundary, persist it separately from selected model state, and surface it consistently in status/runtime APIs while treating provider-specific tagging as a separate policy decision.

## Reproduction Assessment

Yes. The feature gap is source-reproducible from current main: OpenAI-compatible response/model fields are not carried into agent metadata, session persistence, GatewaySessionRow, or session_status output.

## Solution Assessment

Yes for the additive selected-versus-resolved metadata direction, but the exact API field names, status formatting, runtime injection wording, and optional provider tags need maintainer product/API review before implementation.
