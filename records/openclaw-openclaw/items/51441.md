## Summary

Keep open. Current main still drops returned backend model identifiers from OpenAI-compatible responses, carries only one provider/model identity through agent/session metadata, and the related open issues cover adjacent cost or HTTP-header surfaces rather than replacing this session_status/runtime metadata request.

## What This Changes

Expose the actual backend model/provider returned by routing proxies through session persistence, session_status, CLI status, and agent runtime context.

## Best Possible Solution

Define an additive requested-versus-resolved model metadata contract, capture returned backend IDs at the OpenAI-compatible transport boundary, persist them without overwriting selected model state, and expose them consistently through status/session/runtime surfaces while leaving LiteLLM tagging as a follow-up decision.

## Reproduction Assessment

Yes, source-reproducible for the observable feature gap: OpenAI-compatible response and chunk types have model fields, but current main has no resolved-model carrier from transport output into agent meta, session persistence, GatewaySessionRow, or session_status. No live LiteLLM setup was needed because the missing plumbing is visible in source.

## Solution Assessment

Yes for the additive selected-versus-resolved metadata direction; it is the narrow maintainable way to keep the requested alias while exposing actual backend routing. The exact field names, status wording, runtime injection text, and optional LiteLLM tags still need maintainer API/UX review before implementation.
