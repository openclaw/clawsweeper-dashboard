## Summary

Keep open: current main still creates Feishu synthetic reaction message IDs and can pass them unchanged into Feishu API `message_id` fields. The active in-repo implementation candidate is https://github.com/openclaw/openclaw/pull/77653, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Normalize Feishu synthetic reaction message IDs by stripping the `:reaction:<emoji>:<uuid>` suffix before using them in Feishu API calls.

## Best Possible Solution

Land one Feishu-plugin fix that preserves synthetic IDs for internal dedupe/session identity while normalizing IDs at every Feishu API `message_id` boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows OpenClaw constructs `om_*:reaction:<emoji>:<uuid>` IDs, can select them as reply targets, and forwards them unchanged into Feishu API `message_id` fields, matching the reporter's 400/code 99992354 log.

## Solution Assessment

Yes. Feishu-local API-boundary normalization is the narrow maintainable fix because internal synthetic turn identity remains intact while Feishu receives valid base open message IDs.
