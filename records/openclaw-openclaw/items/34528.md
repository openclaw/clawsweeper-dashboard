- `impact:message-loss`: A suffixed reaction message ID can make Feishu API calls fail instead of delivering the expected reply, typing indicator, reaction, media, or related channel action.

## Summary

Keep open: current main still lets Feishu synthetic reaction IDs reach Feishu API `message_id` fields, and the issue has an open closing PR that should land or be replaced before cleanup closes it.

## What This Changes

Strip Feishu synthetic `:reaction:<emoji>:<uuid>` suffixes from message IDs before using them in Feishu API calls.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77653 with a Feishu-local API-boundary normalizer that preserves synthetic internal reaction IDs while sending only base open message IDs to Feishu.

## Reproduction Assessment

Yes. Source inspection on current main shows Feishu reaction events create suffixed synthetic IDs and reply, typing, reaction, pin, and media API paths pass raw IDs into Feishu `message_id` fields, matching the reported 400/code 99992354 failure.

## Solution Assessment

Yes in direction. The maintainable fix is API-boundary normalization inside the Feishu plugin, while keeping synthetic reaction IDs for internal dedupe/session identity; the open closing PR is the current canonical path.
