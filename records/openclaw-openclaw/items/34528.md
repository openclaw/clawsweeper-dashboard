## Summary

Keep open: current main and v2026.6.5 still synthesize Feishu reaction message IDs with a `:reaction:` suffix and pass API-bound message IDs without a normalizer. The open linked fix PR is the right current owner, so this issue should stay paired with that PR until it lands or is replaced.

## What This Changes

Normalize Feishu reaction-suffixed synthetic message IDs to base Open Message IDs before Feishu API calls while preserving synthetic IDs for internal reaction-event identity.

## Best Possible Solution

Land or replace the open Feishu normalization PR after conflict resolution and real Feishu proof, with one shared API-bound message-ID normalizer covering replies, reads, edits, media, reactions, pins, typing, and streaming cards.

## Reproduction Assessment

Yes, from source: reaction events create a suffixed synthetic `message_id`, and the reply/typing/API paths pass that value to Feishu without stripping the suffix. I did not run a live Feishu workspace reproduction in this read-only review.

## Solution Assessment

Yes: preserving the synthetic ID for OpenClaw event identity while normalizing only Feishu API-bound Open Message IDs is the narrowest maintainable boundary. The open PR follows that shape, but it still needs real environment proof and conflict resolution before landing.
