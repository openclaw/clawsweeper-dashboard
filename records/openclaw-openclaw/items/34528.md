## Summary

Keep open: current main and v2026.6.10 still create Feishu reaction-suffixed synthetic IDs and can pass them to Feishu API message_id paths; https://github.com/openclaw/openclaw/pull/77653 is the open fix candidate but has not merged.

## What This Changes

The issue asks the Feishu plugin to strip synthetic `:reaction:<emoji>:<uuid>` suffixes from message IDs before calling Feishu APIs.

## Best Possible Solution

Refresh, prove, and land https://github.com/openclaw/openclaw/pull/77653 or a narrow replacement that normalizes Feishu API-bound message IDs while preserving synthetic reaction IDs for internal dedupe/session identity.

## Reproduction Assessment

Yes at source level. Current main constructs suffixed Feishu reaction message IDs, falls back to that ID as the reply target, and passes reply IDs directly into Feishu API `message_id` paths; I did not run a live credentialed Feishu flow in this read-only review.

## Solution Assessment

Yes for the fix direction. The maintainable shape is a Feishu-plugin API-boundary normalizer, not removing synthetic IDs globally, and the open PR is the current candidate once refreshed and proven.
