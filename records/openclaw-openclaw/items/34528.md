## Summary

Keep open: current main still lets Feishu synthetic reaction message IDs reach API-bound reply and typing paths, and https://github.com/openclaw/openclaw/pull/77653 is an open closing implementation candidate rather than a merged fix.

## What This Changes

Strip OpenClaw's synthetic Feishu `:reaction:<emoji>:<uuid>` suffix before using message IDs in Feishu API calls.

## Best Possible Solution

Land one Feishu-plugin fix that preserves synthetic IDs for internal dedupe/session state but normalizes IDs at every Feishu API boundary, then close this issue through the merged PR.

## Reproduction Assessment

Yes, by source inspection plus the reporter's log. Current main constructs `om_*:reaction:<emoji>:<uuid>` IDs, falls back to them as reply targets, and forwards them unchanged into Feishu `message_id` API fields.

## Solution Assessment

Yes. Feishu-local API-boundary normalization is the narrow maintainable fix because it leaves internal synthetic turn identity intact while sending valid base open message IDs to Feishu.
