## Summary

Keep open: current main still synthesizes Feishu reaction message IDs with the reaction suffix and sends raw message IDs into Feishu API path fields. The open canonical fix path is https://github.com/openclaw/openclaw/pull/77653, so this issue should close only after that PR, or a replacement, lands.

## What This Changes

The issue asks Feishu to strip synthetic `:reaction:<emoji>:<uuid>` suffixes before using message IDs in Feishu API calls.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77653 with a Feishu-local API-boundary normalizer that preserves synthetic reaction IDs internally while sending base open message IDs to Feishu endpoints.

## Reproduction Assessment

Yes. Source inspection shows current main creates `om_*:reaction:<emoji>:<uuid>` IDs and forwards reply, typing, and reaction message IDs directly into Feishu `message_id` path fields, matching the reporter's 400 error.

## Solution Assessment

Yes in direction. The narrow maintainable solution is Feishu API-boundary normalization while preserving the synthetic ID for internal event identity and dedupe.
