- `impact:message-loss`: Reaction-triggered Feishu replies and actions can fail when OpenClaw sends a synthetic reaction message ID back to Feishu APIs.

## Summary

Keep open: current main and v2026.6.8 still allow Feishu reaction-suffixed synthetic IDs to reach Feishu API `message_id` paths, and https://github.com/openclaw/openclaw/pull/77653 is the open fix candidate but has not merged.

## What This Changes

Report asks the Feishu channel to strip synthetic `:reaction:<emoji>:<uuid>` suffixes from message IDs before using them in Feishu API calls.

## Best Possible Solution

Land or replace the canonical Feishu-plugin fix so API-bound Feishu message IDs are normalized while internal synthetic reaction IDs remain available for dedupe and session identity.

## Reproduction Assessment

Yes. The source path on current main creates a suffixed Feishu reaction `message_id` and can reuse it as the default reply target passed to Feishu APIs; I did not run a live credentialed Feishu reaction flow in this read-only review.

## Solution Assessment

Yes for the fix direction. Normalizing only at Feishu API boundaries is narrower than removing synthetic internal IDs, and the current open PR is the right candidate once it is rebased and proven against a real Feishu flow.
