## Summary

Keep open: current main still creates Feishu synthetic reaction message IDs and forwards raw message IDs into Feishu API fields, while the canonical fixing PR is open and unmerged.

## What This Changes

Strip Feishu `:reaction:<emoji>:<uuid>` suffixes before using synthetic reaction message IDs in Feishu API calls.

## Best Possible Solution

Land a Feishu-plugin API-boundary normalizer that preserves synthetic reaction IDs for internal identity while sending only base open message IDs to Feishu.

## Reproduction Assessment

Yes. Current main source creates `om_*:reaction:<emoji>:<uuid>` IDs, can use them as reply targets, and forwards raw IDs into Feishu `message_id` API fields, matching the reported 400/code 99992354 failure.

## Solution Assessment

Yes in direction. The narrow maintainable solution is Feishu-local API-boundary normalization; the linked open PR targets that path and should be reviewed, proven, and landed or replaced before this issue closes.
