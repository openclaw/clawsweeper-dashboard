## Summary

Keep open: current main and the latest release still pass Feishu synthetic reaction message IDs to API-bound `message_id` paths, and the live linked fix PR is still open/conflicting rather than merged.

## What This Changes

Feishu reaction events can generate synthetic `message_id` values with `:reaction:<emoji>:<uuid>` suffixes that then reach Feishu API calls and fail as invalid `open_message_id` values.

## Best Possible Solution

Land a narrow Feishu plugin fix that normalizes API-bound open message IDs across reply, read, edit, media, reaction, pin, typing, and streaming paths while preserving synthetic IDs for internal turn identity.

## Reproduction Assessment

Yes at source level: current main creates a suffixed Feishu reaction `message_id`, defaults reply targeting to that ID, and passes reply IDs directly into Feishu API paths. I did not run a live credentialed Feishu reaction flow in this read-only review.

## Solution Assessment

Yes, the best fix shape is to normalize only API-bound Feishu open message IDs and keep synthetic IDs for OpenClaw turn/dedupe identity. The current open PR follows that direction but still needs conflict resolution and real behavior proof before merge.
