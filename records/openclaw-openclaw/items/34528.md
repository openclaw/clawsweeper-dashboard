## Summary

Keep open: current main still has the Feishu synthetic reaction ID flow and raw Feishu API `message_id` call sites, and https://github.com/openclaw/openclaw/pull/77653 is still open with closing syntax for this issue.

## What This Changes

Feishu reaction event IDs with a `:reaction:<emoji>:<uuid>` suffix should be normalized to the base Open Message ID before Feishu API calls.

## Best Possible Solution

Land or replace the open Feishu PR with one Feishu-local API-boundary normalizer that keeps synthetic IDs internal while sending only base Open Message IDs to Feishu endpoints.

## Reproduction Assessment

Yes from source inspection. Current main creates `om_*:reaction:<emoji>:<uuid>` IDs, can use them as reply targets, and forwards raw IDs into Feishu API `message_id` fields, matching the reported 400/code 99992354 failure.

## Solution Assessment

Yes in direction. The narrow maintainable solution is Feishu-local normalization at API boundaries while preserving synthetic IDs internally; the linked open PR should be reviewed, proven, and landed or replaced before this issue closes.
