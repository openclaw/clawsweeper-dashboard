## Summary

Keep open. Current main still creates Feishu synthetic reaction message IDs with the reported `:reaction:<emoji>:<uuid>` suffix and still passes raw message IDs into Feishu API `message_id` path parameters; the issue is also paired with open target PR #39558 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to strip Feishu synthetic reaction suffixes from message IDs before using them in Feishu API calls.

## Best Possible Solution

Keep this issue open and land, rebase, or replace the open Feishu plugin fix so every Feishu API-boundary `message_id` path parameter receives a base Open Message ID while internal routing, dedupe, and session keys can continue using the synthetic reaction ID where needed.

## Reproduction Assessment

Yes. The report gives live Feishu steps, and current main provides a high-confidence static reproduction path: a reaction event produces a suffixed ID, `ctx.messageId` preserves it, and reply/typing/reaction/media API calls can pass it as `path.message_id`.

## Solution Assessment

Yes. The issue’s proposed direction is the right maintainable fix when scoped to Feishu API boundaries; current main has not implemented it, and open PR #39558 appears to be the active implementation candidate to review or replace.
