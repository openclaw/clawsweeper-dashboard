## Summary

Keep open: current main still creates Feishu synthetic reaction message IDs and forwards raw message IDs into Feishu API `message_id` fields. The canonical in-repo fix path is still the open, unmerged https://github.com/openclaw/openclaw/pull/77653, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests stripping Feishu `:reaction:<emoji>:<uuid>` suffixes from message IDs before those IDs are sent to Feishu API endpoints.

## Best Possible Solution

Land a Feishu-plugin API-boundary normalizer that preserves synthetic reaction IDs for internal identity while sending only base open message IDs to Feishu endpoints.

## Reproduction Assessment

Yes, source-reproducible. Current main creates `om_*:reaction:<emoji>:<uuid>` IDs, can use them as reply targets, and forwards raw IDs into Feishu `message_id` API fields, matching the reported 400/code 99992354 failure.

## Solution Assessment

Yes in direction, but the fix should land through the existing canonical PR or a replacement. The narrow maintainable solution is Feishu-local API-boundary normalization rather than changing internal synthetic reaction identity.
