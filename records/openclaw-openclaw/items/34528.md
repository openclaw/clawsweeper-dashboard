## Summary

Keep open: current main still lets synthetic Feishu reaction message IDs reach reply, typing, and send API paths, and open PR #39558 is the active closing implementation candidate rather than a merged fix.

## What This Changes

Strip Feishu synthetic `:reaction:<emoji>:<uuid>` suffixes from API-facing message IDs so reaction-triggered replies and typing indicators use the original `om_...` message ID.

## Best Possible Solution

Land or replace #39558 with a narrow Feishu plugin fix that strips synthetic reaction suffixes before Feishu API calls while preserving suffixed IDs for internal dedupe and session keys.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu testing. Current main constructs a suffixed reaction `message_id`, falls back to it as the reply target, and passes it unchanged to Feishu reply and typing API paths.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-local message-id normalization at API boundaries or before reply dispatch, with regression tests proving synthetic IDs remain internal only.
