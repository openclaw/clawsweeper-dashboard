## Summary

Keep open: current main still constructs Feishu synthetic reaction message IDs and lets that suffixed value flow into reply and typing API paths, while open PR #39558 is the active unmerged closing implementation candidate.

## What This Changes

Strip Feishu synthetic reaction suffixes from message IDs before those IDs are used in Feishu API calls.

## Best Possible Solution

Land or replace #39558 with a narrow Feishu plugin fix that normalizes API-facing message IDs while preserving synthetic IDs for internal dedupe and session state.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu testing. Current main constructs a suffixed reaction `message_id`, falls back to it as the reply target, and passes it unchanged to Feishu reply and typing API paths.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-local ID normalization at API boundaries or before reply dispatch, with regression coverage proving synthetic IDs remain internal only.
