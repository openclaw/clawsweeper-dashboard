## Summary

Keep this issue open: current main and the latest release still allow a synthetic Feishu reaction message ID to reach API-bound reply paths, while the same-repository fix remains an open, conflicting PR that still needs maintainer handling.

## What This Changes

The issue requests stripping OpenClaw's synthetic Feishu `:reaction:<emoji>:<uuid>` suffix before using Feishu message IDs in API calls.

## Best Possible Solution

Land a rebased, live-proven Feishu-local normalizer at API boundaries while preserving synthetic reaction IDs for internal turn identity, then close this issue from the merged fix.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main creates `om...:reaction:...` synthetic IDs, falls back to that ID as the reply target, and sends it as the Feishu API `message_id` path parameter. I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Yes, the best fix shape is to normalize at Feishu API boundaries while keeping synthetic IDs internal for reaction turn identity and dedup. The active open PR follows that direction, but it needs maintainer review, conflict resolution, and real behavior proof before merge.
