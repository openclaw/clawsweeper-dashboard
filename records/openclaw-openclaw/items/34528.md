## Summary

Keep open: current main and the latest release still pass synthetic Feishu reaction IDs into Feishu `message_id` API fields, and the issue is paired with the open canonical closing PR at https://github.com/openclaw/openclaw/pull/77653.

## What This Changes

Strip Feishu synthetic `:reaction:<emoji>:<uuid>` suffixes from message IDs before using them in Feishu API calls.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77653 with a Feishu-local API-boundary normalizer that preserves synthetic internal IDs while sending only base open message IDs to Feishu endpoints.

## Reproduction Assessment

Yes. Source inspection on current main shows Feishu reaction events produce `om_*:reaction:<emoji>:<uuid>` IDs and reply/typing/reaction API paths pass raw IDs into `message_id`, matching the reporter's Feishu 400/code 99992354 log.

## Solution Assessment

Yes in direction. The narrow maintainable fix is Feishu API-boundary normalization, not removing internal synthetic reaction identity; the linked open PR is the canonical path but still needs proof and conflict resolution.
