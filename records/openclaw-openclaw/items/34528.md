- `impact:message-loss`: Reaction-triggered Feishu replies and actions can fail when OpenClaw sends a synthetic reaction message ID back to Feishu APIs.

## Summary

Keep open: current main and the latest release still allow Feishu reaction-suffixed synthetic message IDs to reach Feishu API message-id paths, and the open linked fix candidate has not landed.

## What This Changes

The issue requests stripping Feishu synthetic `:reaction:<emoji>:<uuid>` suffixes from API-bound message IDs while preserving internal reaction event identity.

## Best Possible Solution

Land or replace the focused Feishu-plugin fix so every Feishu API-bound message ID uses the base `open_message_id` while internal synthetic reaction IDs remain available for dedupe and session identity.

## Reproduction Assessment

Yes. Current main creates a suffixed Feishu reaction `message_id`, can fall back to that ID as the reply target, and passes reply targets directly into Feishu API `path.message_id`; no live credentialed Feishu run was performed in this read-only review.

## Solution Assessment

Yes for the fix direction. Normalizing only at Feishu API boundaries is narrower than removing the synthetic internal IDs, and https://github.com/openclaw/openclaw/pull/77653 is the current candidate once refreshed and proven.
