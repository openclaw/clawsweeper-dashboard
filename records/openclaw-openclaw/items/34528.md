## Summary

Keep open: current main still has the Feishu reaction suffix flow described by the report, and open PR #39558 is the active closing implementation candidate rather than a merged fix.

## What This Changes

Strip or normalize Feishu synthetic `:reaction:<emoji>:<uuid>` message-id suffixes before using those IDs in Feishu API calls.

## Best Possible Solution

Land or supersede #39558 with a narrow Feishu plugin fix that normalizes API-facing reaction message ids while preserving suffixed internal ids for dedupe and session tracking.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu testing. Current main constructs a suffixed reaction `message_id`, falls back to it as the reply target, and passes it unchanged to Feishu reply/typing API paths.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-local message-id normalization at API boundaries or before reply dispatch, with regression tests that prove synthetic ids stay available only for internal dedupe/session keys.
