- `impact:message-loss`: A suffixed reaction message ID can make Feishu API calls fail instead of delivering the expected reply, typing indicator, reaction, media, or related channel action.

## Summary

Keep open: current main and the latest release still create Feishu reaction synthetic IDs with a `:reaction:` suffix and pass unnormalized IDs into Feishu `message_id` API fields, while the remaining canonical fix path is an open unmerged PR.

## What This Changes

The issue asks for Feishu API-bound message IDs from reaction events to be normalized from synthetic `om_*:reaction:<emoji>:<uuid>` IDs back to the base open message ID before API calls.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77653 with a Feishu-local API-boundary normalizer that preserves synthetic reaction IDs internally and sends only base open message IDs to Feishu.

## Reproduction Assessment

Yes. Source inspection on current main shows reaction events create suffixed synthetic IDs and Feishu API-bound paths pass raw IDs into `message_id` fields, matching the reported 400/code 99992354 failure.

## Solution Assessment

Yes in direction. The maintainable fix is Feishu API-boundary normalization that keeps synthetic reaction identity internal, and the open linked PR is the current canonical implementation path.
