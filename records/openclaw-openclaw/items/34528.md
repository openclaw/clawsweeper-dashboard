## Summary

Keep open: current main still lets Feishu synthetic reaction message IDs flow into API-facing reply and reaction paths, while https://github.com/openclaw/openclaw/pull/77653 is an open closing implementation candidate rather than a merged fix.

## What This Changes

The issue reports Feishu synthetic reaction message IDs leaking into Feishu API calls and asks to strip the `:reaction:<emoji>:<uuid>` suffix before API use.

## Best Possible Solution

Land or replace a single Feishu-local normalization fix that keeps synthetic IDs for internal dedupe/session state but sends only base open message IDs to Feishu APIs.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu testing. Current main constructs a suffixed reaction `message_id`, falls back to it as the reply target, and passes it unchanged into Feishu reply, typing, and reaction API paths.

## Solution Assessment

Yes. Normalizing Feishu open message IDs at API boundaries, while preserving synthetic IDs internally, is the narrow owner-local fix and matches the active PR direction.
