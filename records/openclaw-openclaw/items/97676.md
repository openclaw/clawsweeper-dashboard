## Summary

Keep open: the PR is aimed at the right false-positive, but the current patch can suppress stored-secret findings when the referenced env value is itself authored in `config.env` or `config.env.vars`, and the posted CLI proof does not show the resolved-env path that reproduces the bug.

## What This Changes

The PR changes security-audit stored-secret checks to consult authored config snapshot values before suppressing gateway password and hooks token findings, and adds regression tests for env-ref and plaintext cases.

## Best Possible Solution

Keep the parsed-authored-value fix, but only suppress these findings for externally supplied env refs; config-owned env values should still warn or have equivalent audit coverage, with real CLI proof for both external env refs and config.env-backed secrets.

## Reproduction Assessment

Yes. Source inspection on current main shows the audit passes only resolved `cfg` into `collectSecretsInConfigFindings`, so `${VAR}` authored strings are gone before `looksLikeEnvRef` when the env var is set.

## Solution Assessment

No. The fix is close, but it is not the best solution until it distinguishes external env references from env values stored in `config.env` or `config.env.vars`.
