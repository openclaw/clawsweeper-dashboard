## Summary

Keep open: the PR addresses the main external-env false positive and the latest proof is useful, but the patch still appears incomplete for `$include`-owned config because it reads only root `ConfigFileSnapshot.parsed` before falling back to the already substituted source view.

## What This Changes

The PR changes security-audit stored-secret checks for gateway passwords and hooks tokens to use authored config values when deciding whether `${VAR}` env references should suppress findings.

## Best Possible Solution

Carry include-resolved pre-substitution authored values or field provenance into `collectSecretsInConfigFindings`, then cover root, `$include`, config-owned env, and plaintext cases for both affected check IDs.

## Reproduction Assessment

Yes. Source inspection on current main shows the existing audit checks resolved config values, and source inspection of the PR head shows the remaining `$include` gap because root `parsed` does not carry include-resolved authored strings.

## Solution Assessment

No. The fix is on the right security-audit boundary, but it is not the best complete fix until it carries include-resolved authored values or equivalent field provenance instead of only root parsed paths.
