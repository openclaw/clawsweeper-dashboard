## Summary

Keep open: the latest head addresses the earlier root/config-directory `$include` gap, but the new authored-value lookup still re-resolves includes without the config reader's `OPENCLAW_INCLUDE_ROOTS` allowlist, so a supported shared include root can keep the old stored-secret false positive.

## What This Changes

The PR updates security-audit stored-secret checks for gateway password and hooks token so authored `${VAR}` references can suppress false-positive findings while config-owned and plaintext secret values still warn.

## Best Possible Solution

Thread the same include-root policy used by the config reader into the authored-value audit lookup, or reuse a snapshot/provenance value that was resolved by the config reader itself.

## Reproduction Assessment

Yes at source level: current-main security audit checks consume resolved credential strings, and the latest PR head still has a source-reproducible edge case for `$include` files reached through `OPENCLAW_INCLUDE_ROOTS`. I did not run a live CLI repro because this review is read-only.

## Solution Assessment

No. The fix is on the right security-audit boundary, but it should use the same include-root contract as the config reader before it can be the best maintainable solution.
