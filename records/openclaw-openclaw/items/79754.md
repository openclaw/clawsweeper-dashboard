## Summary

Current main still appears affected: Bedrock runtime advertises adaptive Claude 4.6 thinking, but directive/session validation still cannot load a Bedrock provider-policy artifact, and the only linked fix PR was closed unmerged.

## What This Changes

The issue reports that Bedrock Claude 4.6 rejects `/think:adaptive` and remaps an `adaptive` default to `medium` because the Bedrock provider policy artifact is missing.

## Best Possible Solution

Add a Bedrock-owned provider-policy API that reuses Bedrock-aware Claude 4.6/4.7 matching for prefixed refs, then cover directive and session validation so adaptive is accepted and preserved for Bedrock Claude 4.6.

## Reproduction Assessment

Yes from source inspection: `/think:adaptive` and stored `thinkingLevel` validation flow through `isThinkingLevelSupported`, but current main has no Bedrock provider-policy artifact for its adaptive Claude 4.6 profile. I did not run a live AWS Bedrock session.

## Solution Assessment

Yes: the narrow maintainable fix is to expose Bedrock's existing provider-owned thinking profile through a lightweight Bedrock policy artifact, not a core Bedrock special case or broad runtime plugin load.
