## Summary

Keep open: current main still has a source-proven gap where Amazon Bedrock's full provider runtime advertises Claude 4.6 adaptive thinking, but the lightweight provider-policy path used by thinking validation cannot load a Bedrock policy artifact.

## What This Changes

The issue asks for `/think:adaptive` and `agents.defaults.thinkingDefault: "adaptive"` to work for Amazon Bedrock Claude 4.6 models.

## Best Possible Solution

Add an Amazon Bedrock lightweight provider-policy artifact backed by the same Claude thinking-profile helper as the runtime registration, then cover Bedrock Claude 4.6 through the policy, directive, and session-default paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Bedrock Claude 4.6 supports `adaptive` in the runtime provider, but the directive/session validation path falls back to a missing `extensions/amazon-bedrock/provider-policy-api.*` artifact and therefore sees only the base profile.

## Solution Assessment

Yes. The narrowest maintainable fix is extension-owned: expose Bedrock's thinking profile through a lightweight provider-policy API and share the resolver so runtime and validation cannot drift.
