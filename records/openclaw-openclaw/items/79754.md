## Summary

Keep open: current main still has a Bedrock-specific gap where the runtime hook advertises Claude 4.6 adaptive thinking, but directive/session validation cannot see that policy because Amazon Bedrock lacks the lightweight provider-policy artifact used by the fallback path.

## What This Changes

The issue requests that Bedrock Claude 4.6 accept `/think:adaptive` and preserve `thinkingDefault: "adaptive"` instead of remapping it to `medium`.

## Best Possible Solution

Add a lightweight Bedrock provider-policy surface backed by the same Claude thinking-profile logic as the runtime hook, with regression coverage for directive handling, session patch/default behavior, and provider-public-artifact resolution.

## Reproduction Assessment

Yes, source-reproducible with high confidence: the current validation path needs either an active provider registry entry or a provider-policy artifact, and Amazon Bedrock has neither for the lightweight fallback even though its runtime hook supports adaptive Claude 4.6 thinking. I did not establish a live AWS Bedrock reproduction.

## Solution Assessment

Yes: the narrow maintainable fix is to keep this provider-owned by adding a Bedrock `provider-policy-api` artifact that shares the runtime profile helper, rather than adding a core Bedrock special case or broad runtime plugin loading.
