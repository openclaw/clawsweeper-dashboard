## Summary

This should stay open: current `main` still has a Bedrock-specific gap between the runtime provider hook, which advertises adaptive Claude 4.6 thinking, and the directive/session validation path, which cannot see Bedrock policy because the lightweight provider-policy artifact is missing.

## What This Changes

The issue requests that `amazon-bedrock` Claude 4.6 accept `/think:adaptive` and preserve `thinkingDefault: "adaptive"` instead of remapping it to `medium`.

## Best Possible Solution

Keep the policy plugin-owned by adding a lightweight Bedrock provider-policy surface that shares the same Claude thinking profile logic as the runtime hook, with regression tests for directive handling and session patch/default behavior.

## Reproduction Assessment

Yes from source inspection: `/think:adaptive` and stored `thinkingLevel` validation flow through `isThinkingLevelSupported`, but Bedrock has no `provider-policy-api` fallback for its adaptive Claude 4.6 profile. I did not run a live AWS Bedrock session.

## Solution Assessment

Yes: the narrow maintainable fix is to expose Bedrock's existing provider-owned thinking profile through a lightweight plugin policy artifact and reuse that logic, rather than adding a core Bedrock special case or broad runtime plugin loading.
