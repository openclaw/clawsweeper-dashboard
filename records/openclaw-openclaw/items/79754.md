## Summary

Current main still appears to have the reported Bedrock thinking-policy gap: the Bedrock runtime hook advertises adaptive Claude 4.6 thinking, but directive/session validation can only see active provider hooks or bundled `provider-policy-api` artifacts, and Amazon Bedrock has no such artifact on main. The linked fix PR is closed unmerged, so this issue still needs a narrow follow-up.

## What This Changes

The issue asks OpenClaw to accept `/think:adaptive` and preserve `thinkingDefault: "adaptive"` for Amazon Bedrock Claude 4.6 models instead of rejecting or remapping them to `medium`.

## Best Possible Solution

Add a plugin-owned Amazon Bedrock provider-policy surface that shares Bedrock-aware Claude thinking matching with the runtime hook, with regression coverage for directive handling and session default/remap paths.

## Reproduction Assessment

Yes, from source inspection on current main. The documented expected behavior exists, the Bedrock runtime hook supports adaptive Claude 4.6 profiles, and the validation path falls back to a missing Bedrock provider-policy artifact, so it will see only the base profile.

## Solution Assessment

Yes. The narrow maintainable fix is to expose Bedrock’s existing provider-owned thinking profile through a Bedrock policy artifact with Bedrock-aware model ID matching, not to add core Bedrock special cases or broad request-time provider loading.
