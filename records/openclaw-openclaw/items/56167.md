## Summary

Keep this PR open. Current main still lacks the Feishu outbound-level silent-token guard, so the PR is not obsolete, but the proposed patch remains incomplete because `sendMedia` can still forward `NO_REPLY` through its no-media fallback path; the overlapping open PR #44828 is not a merged or clearly canonical replacement.

## What This Changes

The PR imports `isSilentReplyText` into the Feishu outbound adapter, suppresses exact `NO_REPLY` in `sendText` and the leading text branch of `sendMedia`, and adds Feishu outbound tests for exact/whitespace token suppression, substantive text passthrough, and media-caption suppression.

## Best Possible Solution

Complete one canonical Feishu fix that suppresses exact silent-token text at every Feishu outbound text boundary, preserves media delivery when the caption is silent, covers the no-`mediaUrl` fallback, and adds focused regression tests before landing either this PR or the consolidated replacement.

## Reproduction Assessment

Yes. On the PR head, `feishuOutbound.sendMedia({ text: "NO_REPLY", mediaUrl: undefined, ... })` skips the new leading-text branch and then reaches the unguarded no-media fallback, which calls `sendOutboundText` with the raw token.

## Solution Assessment

No. The direction and SDK helper are appropriate, but the patch is not the best complete fix because it only guards the first `sendMedia` text branch; the safer maintainable solution is to normalize/suppress exact silent text before every Feishu text send path that can receive adapter input.
