## Summary

Keep open. Current main still has no exec-approvals denylist or full-command deny rule, and an allowlisted shell/interpreter path can still satisfy the allowlist path before any deny-before-allow command-content policy exists. The CRE comments are useful prior art, while #6615 overlaps but is broader and does not yet replace this issue’s specific shell-wrapper bypass requirement.

## What This Changes

Add exec-approvals deny patterns that inspect full command content before allowlist decisions so dangerous shell-wrapper payloads can be mechanically blocked.

## Best Possible Solution

Define an additive exec approvals deny-rule model that runs before broad allow decisions and consistently matches canonical command text, argv, and shell-wrapper inline payloads across gateway and node execution, with protocol, CLI, UI, docs, and regression tests updated together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure allowlist mode with an allowed shell path such as `/bin/zsh`, then request `zsh -c "openclaw gateway stop"`; current main has no denylist schema or deny-before-allow branch to reject that command content.

## Solution Assessment

Unclear. Full-command deny patterns evaluated before allowlist are directionally appropriate, but the maintainable solution needs security design around normalization, shell parsing, matching grammar, compatibility, and overlap with broader denylist proposals.
