## Summary

Keep open. Current main still has no exec-approvals denylist or command-content deny rule, and the allowlist decision path can still treat an allowlisted shell/interpreter path as satisfied before any full-command deny check exists. The CRE comments are useful prior art, while #6615 is overlapping but broader and does not yet replace this issue’s specific shell-wrapper bypass requirement.

## What This Changes

Add exec-approvals deny patterns that match full command content before allowlist decisions, so dangerous inner commands cannot run through an allowlisted shell path.

## Best Possible Solution

Define an additive exec approvals deny-rule model that is evaluated before broad allow decisions and consistently matches canonical command text, argv, and shell-wrapper inline payloads across gateway and node execution. The finished design should update the protocol schema, approvals CLI, exec-policy CLI, Control UI, docs, and focused regression tests, while coordinating terminology and behavior with the broader denylist work in #6615.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure allowlist mode with an allowed shell path such as /bin/zsh, then request a shell-wrapper payload like `zsh -c "openclaw gateway stop"`; current main has no denylist schema or deny-before-allow branch to reject that command content.

## Solution Assessment

Unclear. Full-command deny patterns evaluated before allowlist are directionally appropriate for the reported bypass, but the best implementation needs maintainer security design around normalization, shell payload parsing, matching grammar, compatibility, and overlap with the broader denylist proposal.
