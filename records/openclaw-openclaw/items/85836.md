## Summary

Close as implemented: current main has the merged update/SIGUSR1 hardening that prevents the reported package-swap lazy-import deadlock, routes global package `update.run` through a managed service handoff, and carries `update.run` through the restart path instead of a config reload.

## What This Changes

The issue reports that macOS global-npm LaunchAgent `update.run` leaves the old Gateway process running after SIGUSR1 instead of loading the newly installed bundle.

## Best Possible Solution

Keep the current managed-update handoff plus eager lifecycle-runtime SIGUSR1 hardening, and use https://github.com/openclaw/openclaw/issues/85246 for any remaining live macOS UI update proof.

## Reproduction Assessment

No current-main failing reproduction was established; the supplied logs match the pre-fix package-swap lazy-import failure fixed by the merged SIGUSR1 lifecycle-runtime hardening. Current source now routes the update path through process replacement or managed-service handoff instead of reload-only recovery.

## Solution Assessment

Yes. The current solution is the maintainable fix for this failure mode because it avoids mutating package-manager installs inside the live Gateway process and makes the SIGUSR1 restart path resilient to swapped hashed chunks.
