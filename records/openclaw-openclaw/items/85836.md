## Summary

Close: current main and shipped v2026.5.20 already route supervised global package-manager update.run through a managed-service handoff outside the live Gateway process and treat update.run as an update-process respawn, so the reported soft-reload path no longer applies.

## What This Changes

The issue reports that UI-driven update.run on macOS global-npm LaunchAgent installs sends SIGUSR1 but soft-reloads instead of replacing the Gateway process after disk bundle changes.

## Best Possible Solution

Keep the shipped managed-service handoff and update-process respawn behavior; reopen only with v2026.5.20 or newer logs showing the handoff still leaves the old Gateway process running.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains. The historical logs are detailed, but current main and v2026.5.20 no longer run supervised global package updates inside the live Gateway process.

## Solution Assessment

Yes. Moving package-manager update work outside the live Gateway process and treating update.run as an update-process respawn is the maintainable fix for bundle files changing on disk.
