## Summary

Keep open: current `main` still has a source-backed mismatch where the status command resolves status/memory/channel SecretRef targets but not gateway auth targets, while the standalone security audit does resolve gateway auth targets before running the same no-auth finding logic.

## What This Changes

The issue reports that human-readable `openclaw status --deep` can show an unauthenticated Gateway warning even when token auth is configured through a SecretRef.

## Best Possible Solution

Add gateway probe/auth SecretRef targets to status command resolution and cover the human `status --deep` embedded-audit path so resolved token/password auth never produces `gateway.http.no_auth`.

## Reproduction Assessment

Yes, from source inspection: status scans resolve `getStatusCommandSecretTargetIds`, which omits `gateway.auth.*`, while the standalone security audit resolves those targets before running the no-auth check. I did not execute a live Gateway repro in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to align status command SecretRef target selection with the Gateway auth used by its probe and embedded audit, not to weaken the audit rule that correctly requires a resolved shared secret.
