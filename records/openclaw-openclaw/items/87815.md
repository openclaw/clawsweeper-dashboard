## Summary

Keep open: current `main` still has a source-backed mismatch where `status --deep` resolves status/memory/channel SecretRef targets but not `gateway.auth.token` or `gateway.auth.password`, then runs the embedded security audit against that partially resolved config.

## What This Changes

The issue asks for `openclaw status --deep` to report gateway auth consistently when token or password auth is configured through a SecretRef.

## Best Possible Solution

Resolve active `gateway.auth.token` and `gateway.auth.password` SecretRefs for the status command path before the embedded security audit runs, with focused regression coverage and no config/schema change.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `gateway.auth.mode` with a token or password SecretRef, then run human `openclaw status --deep` so the embedded audit sees an unresolved auth value. I did not execute the live CLI in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to include active gateway auth SecretRef targets in the status command’s read-only resolution path or otherwise pass a gateway-auth-resolved config into the embedded audit. This should be covered by status-target and human deep-status regression tests rather than changing the security audit finding itself.
