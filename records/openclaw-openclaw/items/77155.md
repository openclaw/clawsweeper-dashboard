## Summary

Keep open: current main still omits allow-only `plugins.allow` IDs from the 2026.5.2 release install repair, while stale plugin cleanup can remove those IDs afterward. PR #77193 is the linked implementation path and is still open, so this issue should wait for that PR to merge or be rejected.

## What This Changes

Request that `openclaw doctor --fix` preserve and install official external plugins, such as `lobster`, when they appear only in `plugins.allow` during the 2026.5.2 one-time migration.

## Best Possible Solution

Review and land or reject PR #77193; the desired end state is a release migration that includes only official external catalog matches from `plugins.allow` while keeping arbitrary allow-only IDs excluded.

## Reproduction Assessment

Yes, by source inspection: an older config with `plugins.allow: ["lobster"]` and no material `plugins.entries.lobster` would not pass `lobster` to the release install repair, and stale cleanup can then remove it. I did not run doctor because this was a read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to add allow-only IDs only when they resolve through the official external plugin catalog before stale cleanup runs. That matches PR #77193’s stated boundary and avoids installing arbitrary allow-list entries.
