## What This Changes

The issue reports isolated cron agent turns aborting early because the pre-execution watchdog can stay armed when no `before_agent_reply` hook emits an execution phase.

## Best Possible Solution

Land one canonical cron watchdog fix that treats known setup progress as liveness while preserving the true no-progress guard, then let the merged PR close this issue and retire overlapping PRs.

Yes, source-level: current main still gates the key `before_agent_reply` phase on hook presence and clears the watchdog only for execution-stage progress or `firstModelCallStarted`. I did not run a live scheduled cron job in this read-only review.
