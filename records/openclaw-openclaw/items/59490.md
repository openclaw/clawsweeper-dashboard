## Summary

Keep open: current main still lacks the requested `cron add --dry-run` feature, and this PR remains the canonical implementation path. The code shape looks focused, but the external PR still needs after-fix real CLI proof before merge, and the new flag should be covered in the cron docs.

## What This Changes

The PR adds `--dry-run` to `openclaw cron add`, prints resolved create params as text or JSON, skips the `cron.add` RPC, and adds CLI tests for those paths.

## Best Possible Solution

Land the CLI-only dry-run implementation after the contributor posts real CLI proof and the cron reference documents the new flag, while preserving validation-before-output and avoiding scheduler or gateway contract changes.

## Reproduction Assessment

Not applicable as a bug reproduction, but the feature gap is high-confidence from source inspection: current main has no `--dry-run` option and always calls `cron.add` after params construction.

## Solution Assessment

Mostly yes. Short-circuiting after params construction and before the gateway RPC is the narrow maintainable solution; richer next-fire previews can be a follow-up unless a maintainer wants that in this first flag.
