## Summary

Keep open. The PR is protected by the `security` label and still has a blocking consistency issue: prefix-constrained hook session-key overrides are demoted in the direct hook finding, but the full audit aggregate still reports the same setting as WARN, so the false-positive reduction is incomplete. The multi-user heuristic portion otherwise matches the linked report.

## What This Changes

The PR changes security audit collectors, docs, tests, and changelog text so prefix-constrained hook session-key overrides report as INFO and the multi-user heuristic only fires when an open or wildcard broadening signal is present.

## Best Possible Solution

Apply one maintainer-approved severity policy consistently across the direct hook finding, the aggregate dangerous-config summary, docs, changelog, and a full-audit regression. The multi-user heuristic change can remain focused on requiring open or wildcard broadening signals.

## Reproduction Assessment

Yes. A full audit path using `hooks.allowRequestSessionKey=true` with non-empty `hooks.allowedSessionKeyPrefixes` still reaches `config.insecure_or_dangerous_flags` at WARN through the aggregate dangerous-flag collector.

## Solution Assessment

No. The multi-user heuristic change is a narrow fit, but the hook fix is incomplete unless the aggregate dangerous-config warning is aligned or deliberately documented as separate policy.
