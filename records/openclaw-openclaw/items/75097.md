## Summary

Keep open. The PR addresses a real false-positive report and the multi-user heuristic change looks directionally narrow, but it is not merge-ready because the hook severity downgrade still leaves a WARN-level aggregate dangerous-config finding for the same prefix-constrained setup. The security label also requires explicit maintainer handling.

## What This Changes

The PR demotes constrained hook session-key override audit findings, narrows the multi-user heuristic to broadening signals, updates security docs/tests/changelog, removes the WhatsApp qrcode dependency, and loosens Codex web-search location schema handling.

## Best Possible Solution

Apply one maintainer-approved severity policy consistently across the direct hook collector, aggregate dangerous-config collector, docs, changelog, and a full-audit regression test, while keeping the multi-user heuristic focused on open or wildcard broadening signals if maintainers accept that policy.

## Reproduction Assessment

Yes. A config with hooks enabled, `hooks.allowRequestSessionKey=true`, and non-empty `hooks.allowedSessionKeyPrefixes` follows `runSecurityAudit` through `collectGatewayConfigFindings`, which still emits `config.insecure_or_dangerous_flags` at WARN before the direct hook collector can return INFO.

## Solution Assessment

No. The multi-user heuristic portion appears to solve the reported false positive narrowly, but the hook false-positive fix is incomplete until the aggregate WARN path is aligned or deliberately preserved with docs and a full-audit regression test.
