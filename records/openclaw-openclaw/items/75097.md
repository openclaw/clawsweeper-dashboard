## Summary

Keep open. This is an active security-labeled PR and the patch has a blocking consistency gap: at PR head the direct hook finding is demoted to INFO for constrained prefixes, but the full audit still emits a WARN through the aggregate dangerous-config flag path for the same setting.

## What This Changes

The PR changes security audit code, tests, docs, and changelog text so constrained hook session-key overrides report as INFO and the multi-user heuristic requires an open or wildcard broadening signal.

## Best Possible Solution

Have a maintainer decide the intended severity policy for prefix-constrained hook session-key overrides, then align the full audit path, docs, changelog, and regression tests so the shipped behavior matches that policy before merging.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a full `runSecurityAudit` on a config with `hooks.allowRequestSessionKey=true` and non-empty `hooks.allowedSessionKeyPrefixes` still reaches the aggregate WARN path even after the PR's direct finding downgrade.

## Solution Assessment

No. The multi-user heuristic change looks narrow, but the hook fix is incomplete because it updates the named hook finding without updating or documenting the full-audit aggregate warning that still fires for the same mitigated setting.
