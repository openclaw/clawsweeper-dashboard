## Summary

Keep open. The PR addresses a real false-positive class, but the hook downgrade is incomplete because the full audit path can still emit a WARN-level aggregate dangerous-config finding for the same prefix-constrained setup, and the PR carries the protected `security` label.

## What This Changes

The PR demotes prefix-constrained hook session-key override audit findings, narrows multi-user heuristic signals, updates security audit docs/tests/changelog, removes the WhatsApp `qrcode` dependency, and moves Matrix pinned-DM owner resolution to the live ingress allowlist.

## Best Possible Solution

Apply one maintainer-approved severity policy consistently across the direct hook finding, aggregate dangerous-config collector, docs, changelog, and a full-audit regression test, while keeping the multi-user heuristic focused on open or wildcard broadening signals if maintainers accept that policy.

## Reproduction Assessment

Yes. A config with `hooks.enabled=true`, `hooks.allowRequestSessionKey=true`, and non-empty `hooks.allowedSessionKeyPrefixes` follows `runSecurityAudit` through the aggregate dangerous-config path, which still emits a WARN for `hooks.allowRequestSessionKey=true` before the PR's direct hook finding is demoted.

## Solution Assessment

No. The multi-user heuristic change is a narrow fit for #26859, but the hook fix is not complete until the aggregate WARN is aligned or intentionally documented and tested as a separate policy.
