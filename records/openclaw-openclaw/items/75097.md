## Summary

Keep open. This security-labeled PR still has a blocking consistency issue: the direct hook finding is demoted to INFO for constrained prefixes, but a full audit still emits a WARN-level dangerous-config aggregate for the same mitigated setting.

## What This Changes

The PR demotes prefix-constrained hook sessionKey override audit findings to INFO, narrows the multi-user heuristic to broadening channel signals, updates security docs/changelog/tests, and adjusts Codex web-search user-location config parsing.

## Best Possible Solution

Pick one security-audit severity policy for prefix-constrained hook session keys and apply it consistently across the direct hook collector, dangerous-config aggregate collector, docs, changelog, and a full `runSecurityAudit` regression; then land the multi-user heuristic narrowing if maintainers accept that policy direction.

## Reproduction Assessment

Yes. A config with `hooks.enabled=true`, `hooks.allowRequestSessionKey=true`, and non-empty `hooks.allowedSessionKeyPrefixes` follows `runSecurityAudit` through `collectGatewayConfigFindings`, where `hooks.allowRequestSessionKey=true` still produces `config.insecure_or_dangerous_flags` at WARN before the direct hook collector emits INFO.

## Solution Assessment

No. The multi-user heuristic change appears narrow and covered, but the hook false-positive fix is incomplete until the aggregate dangerous-config path is aligned or deliberately preserved with matching docs and tests.
