## Summary

Keep open. The PR is a plausible fix for #26859, but it is security-labeled and the hook severity change is incomplete: the direct hook finding is demoted to INFO for constrained prefixes, while the full audit still has an unconditional WARN-level dangerous-flag aggregate for `hooks.allowRequestSessionKey=true`.

## What This Changes

The branch demotes constrained hook session-key audit findings, narrows the multi-user heuristic to broadening channel signals, updates security docs/changelog/tests, and adjusts Codex web-search config tests plus a tool-display script import.

## Best Possible Solution

Keep this PR open, align the full-audit dangerous-flag aggregate with the direct hook severity policy or explicitly document and test that a separate WARN remains intentional, then land after security review if maintainers accept the narrowed multi-user heuristic.

## Reproduction Assessment

Yes. A config with `hooks.allowRequestSessionKey=true` and non-empty `hooks.allowedSessionKeyPrefixes` can be traced through full `runSecurityAudit`: the gateway dangerous-flag aggregate emits a WARN before the PR's direct hook collector emits INFO.

## Solution Assessment

No. The multi-user heuristic narrowing is focused and tested, but the hook false-positive fix is not complete until the aggregate warning path is aligned or deliberately retained with docs and full-audit regression coverage.
