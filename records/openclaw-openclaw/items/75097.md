## Summary

Keep open. The PR addresses a real false-positive report, but the current patch still leaves a full `openclaw security audit` WARN for the prefix-constrained hook case through the aggregate dangerous-config collector, and the `security` label requires explicit maintainer handling before merge.

## What This Changes

The PR changes security audit code, docs, tests, and changelog text so constrained hook session-key overrides report as INFO and the multi-user heuristic requires an open or wildcard broadening signal.

## Best Possible Solution

Align the entire security audit path around one maintainer-approved severity policy for prefix-constrained hook session-key overrides, then update the direct finding, aggregate dangerous flag behavior, docs, changelog, and a full-audit regression together.

## Reproduction Assessment

Yes. A full `runSecurityAudit` on a config with `hooks.allowRequestSessionKey=true` and non-empty `hooks.allowedSessionKeyPrefixes` still reaches the aggregate WARN path through `collectGatewayConfigFindings` even after the PR demotes the direct hook finding.

## Solution Assessment

No. The multi-user heuristic change appears focused, but the hook fix is incomplete because it changes the named hook finding without aligning the full-audit aggregate warning that still fires for the same mitigated setting.
