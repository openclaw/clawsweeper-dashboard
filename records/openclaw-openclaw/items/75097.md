## Summary

Keep open. The PR targets a real security-audit false-positive report and the multi-user heuristic change is directionally supported, but the hook severity downgrade is incomplete because the full audit path still emits a WARN-level aggregate dangerous-config finding for the same prefix-constrained setup; the protected `security` label also requires maintainer handling.

## What This Changes

The PR updates security audit collectors, docs, tests, and changelog text to downgrade prefix-constrained hook session-key overrides to INFO, require open or wildcard broadening signals for the multi-user heuristic, and removes an unused WhatsApp `qrcode` dependency.

## Best Possible Solution

Apply one maintainer-approved severity policy consistently across the direct hook finding, aggregate dangerous-config collector, docs, changelog, and a full-audit regression, while keeping the multi-user heuristic focused on open or wildcard broadening signals.

## Reproduction Assessment

Yes. A config with `hooks.allowRequestSessionKey=true` and a non-empty `hooks.allowedSessionKeyPrefixes` follows the full audit path through `collectGatewayConfigFindings`, where current main still records `hooks.allowRequestSessionKey=true` as a WARN-level aggregate dangerous flag.

## Solution Assessment

No. The multi-user heuristic change is a narrow fit for the linked report, but the hook fix is not the best complete solution until the aggregate warning semantics are aligned or intentionally documented as separate policy.
