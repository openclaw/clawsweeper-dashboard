## Summary

Keep open. The protected `security` label requires maintainer handling, and the current patch still leaves a full `openclaw security audit` path that reports `hooks.allowRequestSessionKey=true` as a WARN-level dangerous flag even when the PR demotes the direct hook finding to INFO.

## What This Changes

The PR demotes prefix-constrained hook session-key audit findings to INFO, requires broadening signals for the multi-user heuristic, updates security audit docs/tests/changelog, and removes the unused WhatsApp `qrcode` dependency.

## Best Possible Solution

Apply one maintainer-approved severity policy consistently across the direct hook finding, aggregate dangerous-config collector, docs, changelog, and a full-audit regression test, while keeping the multi-user heuristic focused on open or wildcard broadening signals if maintainers accept that behavior.

## Reproduction Assessment

Yes. A config with `hooks.enabled=true`, `hooks.allowRequestSessionKey=true`, and non-empty `hooks.allowedSessionKeyPrefixes` follows `runSecurityAudit` through the aggregate dangerous-config path, which still emits a WARN for `hooks.allowRequestSessionKey=true` before the PR's direct hook finding is demoted.

## Solution Assessment

No. The multi-user heuristic change appears narrow and well-covered, but the hook false-positive fix is incomplete until the aggregate WARN path is aligned or explicitly documented and tested as a separate policy.
