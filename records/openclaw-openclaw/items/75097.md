## Summary

Keep open: the PR addresses a real #26859 false-positive path, but the hook warning change is incomplete because full audits still emit a WARN-level aggregate for the same prefix-constrained setting, and the protected security label needs maintainer/security review before merge.

## What This Changes

This PR downgrades prefix-constrained hook session-key override findings to INFO, narrows the multi-user heuristic to broadening channel signals, and updates security docs, tests, and the changelog.

## Best Possible Solution

Align the aggregate dangerous-flag path with the constrained-prefix policy, or document and test the aggregate WARN as intentional, then land after security review.

## Reproduction Assessment

Yes. A config with `hooks.allowRequestSessionKey=true` and non-empty `hooks.allowedSessionKeyPrefixes` can be traced through full `runSecurityAudit`: the gateway config collector still sees `hooks.allowRequestSessionKey=true` and emits WARN before the hook hardening collector emits the PR's INFO finding.

## Solution Assessment

No. The multi-user heuristic narrowing is focused, but the hook false-positive fix is incomplete until the aggregate audit path either matches the new INFO policy or is deliberately retained with docs and regression coverage.
