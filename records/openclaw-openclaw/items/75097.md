## Summary

Keep open. The PR is protected by the `security` label and still has a blocking audit-severity inconsistency: the direct hook finding is demoted to INFO for constrained prefixes, but the full audit still emits a WARN-level aggregate dangerous-config finding for the same setting.

## What This Changes

The PR changes security audit code, docs, tests, and changelog text so constrained hook session-key overrides report as INFO and the multi-user heuristic requires an open or wildcard broadening signal.

## Best Possible Solution

Land this only after the maintainer-approved severity policy is applied consistently across the direct hook finding, the aggregate dangerous-config summary, docs, changelog, and a full-audit regression. The multi-user heuristic portion otherwise appears focused on the reported false positive.

## Reproduction Assessment

Yes. A full `runSecurityAudit` on a config with `hooks.enabled=true`, `hooks.allowRequestSessionKey=true`, and non-empty `hooks.allowedSessionKeyPrefixes` still reaches the aggregate WARN path through `collectGatewayConfigFindings`, even though the direct hook finding is INFO.

## Solution Assessment

No. The multi-user heuristic change looks like a narrow fit, but the hook fix is incomplete because it changes one finding's severity without aligning the full-audit aggregate warning for the same mitigated setting.
