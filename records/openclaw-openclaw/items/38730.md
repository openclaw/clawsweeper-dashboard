## Summary

Keep open because current main now addresses the reported cron run-session accumulation with automatic 24-hour pruning and transcript cleanup, but the issue also requests a high-frequency schedule warning that is still absent and is owned by an open, conflicting linked PR. The original OpenClaw 2026.3.2 incident has not been reproduced against current main, so the remaining work needs a maintainer decision on whether schedule frequency is the right warning signal and current runtime proof before landing.

## What This Changes

Add safeguards for cron-related session and context pressure through automatic run-session cleanup, stronger diagnostics, and warnings for high-frequency schedules.

## Best Possible Solution

Keep automatic per-run session pruning as the canonical root fix, then require the linked PR or a replacement to justify its warning signal and threshold with a current-main cron workload that demonstrates actionable session or context pressure across CLI and Gateway creation/update paths.

## Reproduction Assessment

No. The incident evidence is from OpenClaw 2026.3.2, while current main now automatically reaps completed isolated cron sessions; no live or test reproduction currently ties frequent schedules to silent reply loss on current main.

## Solution Assessment

Partly. Automatic retention is the narrow root-cause fix for unbounded completed-run accumulation, but the proposed universal under-30-minute warning is only a plausible mitigation and needs evidence that frequency is the correct signal rather than workload, retained tokens, or session count.
