## Summary

Keep open: current main mitigates cron run-session buildup with retention and a timer-driven reaper, but high-frequency `every` schedule advisories are still missing from the checked creation/update paths, and open replacement PR #76473 carries closing syntax for this issue.

## What This Changes

Request guardrails and warnings so high-frequency cron jobs do not silently accumulate cron sessions or context pressure.

## Best Possible Solution

Keep the shipped retention cleanup and land or replace #76473 with a shared, non-blocking high-frequency `every` schedule advisory across cron create/update entry points.

## Reproduction Assessment

Yes for the remaining missing guardrail: source inspection shows short `everyMs` schedules can be accepted through CLI, Gateway, Control UI, and agent-tool paths without a dedicated advisory. The original context-exhaustion/no-reply incident was not independently replayed.

## Solution Assessment

Yes for the current direction: a shared non-blocking advisory is the narrowest compatible guardrail, while the shipped retention reaper handles the session buildup portion. Context-threshold diagnostics should be separate follow-up work if maintainers still want them.
