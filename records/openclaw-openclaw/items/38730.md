## Summary

Keep open. Current main mitigates part of the cron session-retention problem, but the requested high-frequency `every` schedule advisory is still absent from the CLI, Gateway, Control UI, and agent-tool add/update paths, and #38731 remains an open same-author implementation PR with closing syntax for this issue.

## What This Changes

The issue asks for cron guardrails around session accumulation, including run-session TTL cleanup, context/session pressure warnings, short-interval schedule warnings, and stronger cleanup behavior.

## Best Possible Solution

Keep the shipped cron run-session retention mitigation and land or replace #38731 with a shared, non-blocking advisory for high-frequency `every` schedules that is applied consistently across CLI add/edit, Gateway add/update, Control UI add/edit, and agent-created cron jobs. If maintainers still want context-threshold diagnostics, track that as a narrower follow-up after the schedule guardrail lands.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the missing guardrail: source inspection shows short `everyMs` values remain valid and creation/update paths do not warn. The original production no-reply/context-pressure incident is supported by the report but was not independently replayed here.

## Solution Assessment

Is this the best way to solve the issue? Yes for the remaining actionable part: a shared non-blocking advisory preserves compatibility while making risky high-frequency schedules visible at creation and update time. Closing this before #38731 or an equivalent fix lands would leave a requested guardrail missing.
