## Summary

Keep open: current main has cron run-session retention and reaping, but it still accepts sub-30-minute cron schedules without the requested advisory, and https://github.com/openclaw/openclaw/pull/76473 is an open closing PR that has not landed.

## What This Changes

The issue asks for guardrails around high-frequency cron jobs and session accumulation, including retention/cleanup, warnings, and diagnostics before context pressure makes agents stop replying.

## Best Possible Solution

Review, prove, and land or replace https://github.com/openclaw/openclaw/pull/76473 for the warning slice, keep the cron session reaper, and track broader session-store bloat or context diagnostics in the canonical follow-up.

## Reproduction Assessment

Partly. Source inspection shows current main accepts CLI `--every 15m` and Gateway `{ kind: "every", everyMs: 900000 }` schedules without an advisory, but the original production `replies=0` incident was not replayed.

## Solution Assessment

Yes for the focused direction, but not for closing now. A shared non-blocking advisory across CLI add/edit and Gateway add/update is maintainable, while the issue should remain open until the linked closing PR lands or maintainers choose a different scope.
