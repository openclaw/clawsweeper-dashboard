## Summary

Keep open: current main already prunes expired isolated cron run sessions, but sub-30-minute `every` schedules still have no advisory across the checked creation/update paths, and open replacement PR #76473 is the active closing implementation candidate.

## What This Changes

The issue requests cron session-retention cleanup plus warnings or guardrails for high-frequency cron schedules that can create context/session pressure.

## Best Possible Solution

Keep the shipped cron run-session retention cleanup and add one shared, non-blocking advisory for short `every` schedules wherever cron jobs are created or updated.

## Reproduction Assessment

Yes for the remaining missing guardrail: source inspection shows short `everyMs` schedules are accepted in CLI, Gateway, Control UI, and agent-tool paths without a warning. The original production context-exhaustion/no-reply incident was not independently replayed.

## Solution Assessment

Yes for the remaining actionable direction: a shared non-blocking advisory is the narrowest compatible guardrail, and #76473 is already the active closing implementation candidate. Context-threshold diagnostics should be handled separately if maintainers still want them.
