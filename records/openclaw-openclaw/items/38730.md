## Summary

The issue is partially resolved in the shipped v2026.6.5 release: isolated cron-run sessions now default to 24-hour retention and are automatically pruned, with documented cleanup controls. It should remain open because https://github.com/openclaw/openclaw/pull/76473 is an open PR that explicitly closes this issue, and the separate claim that cron accumulation causes main-session context exhaustion and silent `replies=0` behavior has not been reproduced on current main.

## What This Changes

Request cron-run session retention, effective cleanup, context-pressure warnings, and high-frequency schedule warnings to prevent silent agent non-response.

## Best Possible Solution

Retain the shipped cron-session reaper, require current-main real behavior proof for the linked warning PR, and narrow any follow-up to a reproducible context-pressure or silent-reply defect with diagnostics at the actual agent/session boundary.

## Reproduction Assessment

Unclear. The 2026.3.2 incident provides concrete logs and session counts, but v2026.6.5 now ships automatic cron-run retention, and there is no current-main reproduction proving that cron entries inflate the main transcript or cause `replies=0`.

## Solution Assessment

No. A high-frequency warning is a reasonable mitigation, but it is not the best complete solution without validating the threshold and tracing the remaining context-pressure or silent-reply failure to its owning code path.
