## Summary

Keep this PR open: the flag is useful and the proof covers the happy path, but the latest diff still reads a caller-supplied path inside the shared outbound runner, which bypasses the existing sandbox/media access boundary for non-CLI callers.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, reads UTF-8 file content into the send message, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the CLI flag, but resolve file content at the CLI boundary or through a policy-backed text-file resolver before shared action handling, with regression coverage for exact content and sandbox bypass attempts.

## Reproduction Assessment

Yes. Current main source has no `--message-file` option, and the PR head source makes the sandbox bypass path clear by reading `messageFile` in shared `handleSendAction` while `tools.message` forwards raw params and `sandboxRoot` to the same runner.

## Solution Assessment

No. The user-facing flag is the right shape, but the implementation should not read arbitrary files inside the shared outbound runner; the safer fix is CLI-owned resolution or an existing policy-backed file access path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep message-file reads inside the CLI boundary:** `src/infra/outbound/message-action-runner.ts:632`
  - body: `handleSendAction` is shared by CLI and `tools.message`, and the tool path forwards raw params plus `sandboxRoot` into `runMessageAction`. Reading `messageFile` here with bare `fs.readFile` lets any caller that can supply that param read and send host files outside the sandbox/media policy. Resolve the file in CLI-owned input handling or through the same policy-backed resolver used for local media.
  - confidence: 0.9
