## Summary

Keep open: the feature is useful and the supplied terminal proof is sufficient for the happy path, but the PR still reads `messageFile` with unrestricted `fs.readFile` inside the shared outbound runner, which can bypass the sandbox/media access boundary for non-CLI callers.

## What This Changes

The PR adds a `--message-file <path>` option to `openclaw message send`, updates CLI docs and changelog text, and adds send-validation coverage for file content and error handling.

## Best Possible Solution

Keep the CLI flag, but resolve file content at the CLI boundary or through a policy-backed text-file resolver before shared action handling, with regression coverage for exact content and sandbox bypass attempts.

## Reproduction Assessment

Yes. Current main has no `--message-file` flag, and the PR head's security issue is source-reproducible because `handleSendAction` reads `messageFile` directly while the agent tool path forwards raw params and `sandboxRoot` into the same runner.

## Solution Assessment

No. The user-facing flag is a reasonable shape, but arbitrary path reads should not happen inside shared outbound handling; CLI-owned resolution or a policy-backed resolver is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep message-file reads inside the CLI boundary:** `src/infra/outbound/message-action-runner.ts:632`
  - body: `handleSendAction` is shared by CLI and agent `tools.message`, and the tool path forwards raw params plus `sandboxRoot` into `runMessageAction`. Reading `messageFile` here with bare `fs.readFile` lets non-CLI callers that can supply the param read and send host files outside the sandbox/media policy; resolve the file in CLI-owned input handling or through the existing policy-backed file access path.
  - confidence: 0.9
