## Summary

Keep open: the flag is a useful CLI feature, but the latest patch still reads the supplied file inside shared outbound send handling, which keeps the sandbox/media-policy bypass concern alive for non-CLI callers that can pass raw action params.

## What This Changes

Adds a `--message-file <path>` flag to `openclaw message send`, reads UTF-8 file contents into the send body, and updates docs, tests, and the changelog.

## Best Possible Solution

Keep the CLI flag, but resolve file content at the CLI input boundary or through a policy-backed text-file resolver before shared action handling, with regression coverage for exact content and sandbox bypass attempts.

## Reproduction Assessment

Yes. Current main lacks `--message-file`, and the PR head source makes the remaining security regression reproducible by inspection: `handleSendAction` reads `messageFile` with bare `fs.readFile` while tool callers can forward raw params plus `sandboxRoot`.

## Solution Assessment

No. The user-facing flag is the right shape, but reading arbitrary files in the shared outbound runner is not the narrowest maintainable solution; the safer fix is CLI-owned resolution or an existing policy-backed file access path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep message-file reads out of shared send handling:** `src/infra/outbound/message-action-runner.ts:632`

Reading `messageFile` here with bare `fs.readFile` lets a non-CLI caller that can supply that param read host files outside the sandbox/media policy and send their contents. Resolve the file at the CLI boundary or through the same policy-backed access layer used for local media.
