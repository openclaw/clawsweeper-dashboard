## Summary

Keep open: the CLI feature is useful, but the patch still resolves a caller-provided file path inside the shared outbound send runner, which is also reachable from agent/tool dispatch, and the external PR still needs actual CLI-path behavior proof before merge.

## What This Changes

This PR adds `--message-file <path>` to `openclaw message send`, documents it, reads UTF-8 file content for send actions, adds send-validation tests, and updates the changelog.

## Best Possible Solution

Keep the CLI flag, but resolve the file only in the CLI helper path, pass sanitized `message` downstream, update docs/changelog/tests, and require real CLI proof before merge.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this PR adds a new CLI input mode, and current main only supports inline `--message`, media, and presentation inputs. Source inspection confirms shell expansion happens before OpenClaw receives argv.

## Solution Assessment

No. The CLI flag is a reasonable direction, but file resolution belongs at the CLI boundary before `messageCommand` / `runMessageAction`, not in the shared outbound runner used by agent/tool dispatch.
