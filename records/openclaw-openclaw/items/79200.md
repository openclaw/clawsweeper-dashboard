## Summary

Keep open: the feature is useful, but the current patch still reads `messageFile` inside the shared outbound runner, which gives agent/tool callers a hidden host-file read path, and the external proof does not show the actual CLI flag working end to end.

## What This Changes

The PR adds a `--message-file <path>` option to `openclaw message send`, reads UTF-8 file content into outbound send params, and updates docs, changelog, and send-validation tests.

## Best Possible Solution

Implement file reading in the CLI helper path, pass only `message` into `runMessageAction`, strip `messageFile` from generic runner/tool/gateway params, then align docs, changelog credit, tests, and real CLI proof.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this PR adds a new CLI input mode, and current main only supports inline `--message`, media, and presentation inputs. Source inspection confirms the flag is missing; shell expansion happens before OpenClaw receives argv.

## Solution Assessment

No. The CLI flag is a reasonable direction, but resolving file paths inside the shared outbound runner is not the narrowest maintainable or safest implementation.
