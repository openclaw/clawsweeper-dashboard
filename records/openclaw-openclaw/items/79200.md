## Summary

Keep open: the feature is useful and the supplied terminal proof covers the happy path, but the PR still reads `messageFile` inside shared outbound handling, which can bypass the existing sandbox/media policy boundary for non-CLI callers.

## What This Changes

The PR adds a `--message-file <path>` option to `openclaw message send`, reads UTF-8 file content into the send message, and updates docs, changelog, and send-validation tests.

## Best Possible Solution

Keep the CLI flag, but resolve file content at the CLI boundary or through a policy-backed text-file resolver before shared action handling, with regression coverage for exact content and sandbox bypass attempts.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows PR head reads `messageFile` with `fs.readFile` in shared `handleSendAction`, while `tools.message` forwards raw params and `sandboxRoot` into the same runner. I did not run a live exploit; the defect is source-reproducible from the PR head.

## Solution Assessment

No. The user-facing flag is the right shape, but file reads should happen in CLI-owned input handling or a policy-backed resolver, not in the shared outbound runner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep message-file reads inside the CLI boundary:** `src/infra/outbound/message-action-runner.ts:632`
  - body: `handleSendAction` is shared by CLI and `tools.message`, and the tool path forwards raw params plus `sandboxRoot` into the same runner. Reading `messageFile` here with bare `fs.readFile` lets non-CLI callers that can supply this param read and send host files outside the sandbox/media policy; resolve the file in CLI-owned input handling or through a policy-backed resolver.
  - confidence: 0.9
