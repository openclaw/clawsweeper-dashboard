## Summary

Keep open: the feature is useful and the contributor supplied real terminal proof, but the patch currently reads `messageFile` inside the shared send runner, which lets non-CLI message-tool params bypass the existing sandbox/media file-access controls.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, reads UTF-8 file contents into the send body, and updates docs, changelog, and send-validation tests.

## Best Possible Solution

Keep the flag, but resolve file contents at the trusted CLI command boundary or through the existing sandbox/media access policy before assigning to `message`, with regression coverage that agent/tool calls cannot read outside their allowed roots.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the new `messageFile` branch uses raw `fs.readFile` in a shared runner reached by the model-facing message tool. For the feature request itself, current-main absence of `--message-file` is source-confirmed rather than a runtime failure.

## Solution Assessment

No. The CLI flag is the right direction, but implementing file reads in `handleSendAction` is not the narrowest safe fix because it bypasses the existing sandbox/media file-access path for non-CLI callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep message-file reads out of the shared tool path:** `src/infra/outbound/message-action-runner.ts:627`
  - body: `handleSendAction` is also reached by the model-facing `message` tool, which forwards raw args and `sandboxRoot` into `runMessageAction`. Reading `messageFile` here with `fs.readFile` bypasses the existing sandbox/media-root checks used for `media` and `filePath`, so a tool call with an extra `messageFile` key can read a host path and send the contents as text. Resolve the file at the CLI boundary, or run it through the same sandbox/local-root policy before use.
  - confidence: 0.87
