## Summary

Keep open: the feature is useful and the supplied terminal proof is sufficient, but the latest PR head still reads arbitrary file paths from shared outbound send handling, which crosses the existing sandbox/media access boundary.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, updates docs and changelog text, and adds send-validation coverage for file-backed message bodies.

## Best Possible Solution

Keep the CLI flag, but resolve the file at the CLI boundary or through a policy-backed text-file resolver before shared message-action handling receives the final message string.

## Reproduction Assessment

Yes. Current main lacks the flag, and the PR-head diff makes the blocking boundary issue source-reproducible by adding a bare file read inside shared send handling.

## Solution Assessment

No. The user-facing flag is the right shape, but the implementation should not add unrestricted host file reads to `message-action-runner.ts`; CLI-owned resolution or policy-backed access is safer.

## Review Findings
