## Summary

Keep open: the feature is useful and not on main, but the latest PR head still reads `messageFile` with `fs.readFile` inside shared outbound handling, creating a file-access policy bypass for non-CLI callers before merge.

## What This Changes

The PR adds a `--message-file <path>` option to `openclaw message send`, updates CLI docs and changelog text, and adds send-validation coverage for file-backed message bodies.

## Best Possible Solution

Keep the CLI flag, but resolve file content at the CLI boundary or through a policy-backed text-file resolver, then add sandbox/tool-path regression coverage for the boundary.

## Reproduction Assessment

Yes. Current main lacks the flag, and the PR-head source clearly shows the new file-read path in shared send handling while the agent message tool forwards params into the same runner.

## Solution Assessment

No. The flag is the right user-facing shape, but reading arbitrary paths inside shared outbound handling is not the narrowest maintainable implementation; the file read should stay CLI-owned or use existing policy-backed access.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep message-file reads inside the file-access boundary:** `src/infra/outbound/message-action-runner.ts:632`

Reading `messageFile` here with bare `fs.readFile` lets any non-CLI caller that can carry this param read and send host files outside the sandbox/media policy, so resolve the file at the CLI boundary or through the same policy-backed access path used for local media.
