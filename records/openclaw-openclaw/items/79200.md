## Summary

Keep open: the feature is still useful and neither current main nor the latest release exposes `--message-file`, but this PR is not merge-ready because the live PR is dirty/conflicting and current main moved send-body construction into the shared `buildSendPayloadParts` path.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, reads UTF-8 file content as the send body, updates CLI docs, adds outbound send validation tests, and edits `CHANGELOG.md`.

## Best Possible Solution

Land a refreshed, changelog-free implementation that resolves `messageFile` before the shared send-body validation/normalization path, preserves file content exactly, and explicitly handles or defers the linked issue's truncation criterion.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection confirms current main and `v2026.6.6` have no `--message-file` flag and read send bodies from `message`; the shell-expansion corruption itself depends on caller shell and quoting.

## Solution Assessment

Is this the best way to solve the issue? Likely yes if maintainers accept the new public CLI flag, because a file-backed input path is narrower than changing channel adapters or relying on shell quoting; the implementation needs a rebase into current `buildSendPayloadParts`.
