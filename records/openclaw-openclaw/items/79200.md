## Summary

Keep open: the feature direction is useful, but the current patch still reads a caller-supplied path inside the shared outbound runner, which is reachable from non-CLI message action paths. The external proof also remains too weak for merge because it does not show an actual `openclaw message send --message-file` invocation or the preserved payload text on the latest head.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, reads UTF-8 file content as the message body in the shared outbound send runner, and updates docs, changelog, and send-validation tests.

## Best Possible Solution

Resolve `--message-file` only in the CLI helper, pass the resulting `message` string into existing dispatch, strip path-only options before generic runner/tool/gateway paths, then keep docs, changelog credit, tests, and after-fix CLI proof aligned.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this PR adds a new CLI input mode, and current main only documents and registers inline `--message`, media, and presentation inputs. Source inspection confirms the missing flag; shell expansion corruption happens before OpenClaw receives the argument.

## Solution Assessment

No in the current form. The CLI flag is a reasonable direction, but file reading belongs at the CLI boundary so non-CLI tool, gateway, and plugin paths do not inherit hidden host-file read behavior.
