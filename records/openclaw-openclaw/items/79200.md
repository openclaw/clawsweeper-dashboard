## Summary

Keep open: the feature direction is useful, but the current PR still reads an arbitrary host file inside the shared outbound send runner, which is reachable from the agent message tool path, not only from the CLI flag.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, documents it, updates the changelog, and adds send-validation tests for reading UTF-8 message text from a file.

## Best Possible Solution

Resolve `--message-file` in the CLI/message command boundary, pass only hydrated `message` text into `runMessageAction`, and keep the shared agent/outbound runner host-file-free.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR reads `messageFile` in `handleSendAction`, and current main's agent message tool forwards raw params into that shared runner. The original request itself is a new CLI input mode, so current-main bug reproduction is not applicable beyond the shell-expansion limitation outside OpenClaw.

## Solution Assessment

No. The flag is a reasonable feature direction, but the file read should happen at the CLI boundary before shared outbound dispatch so agent/tool callers cannot trigger host-file reads through hidden params.
