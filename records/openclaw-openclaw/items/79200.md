## Summary

Keep open: the feature is useful and active, but this PR is not merge-ready because the file read is implemented in the shared outbound send runner reachable from non-CLI message action paths, the external proof still does not show exact payload preservation through the CLI flag, and the changelog credit is missing.

## What This Changes

The PR adds a `--message-file <path>` option to `openclaw message send`, reads UTF-8 file contents in the outbound send runner, documents the flag, adds send-validation tests, and records a changelog entry.

## Best Possible Solution

Resolve `--message-file` at the CLI boundary, pass only the resulting message string into the existing runner/gateway/tool path, preserve file content exactly in regression tests, update docs/changelog credit, and provide after-fix terminal or CLI proof that shows the preserved payload with private details redacted.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this PR adds a new CLI input mode, and current main only exposes inline `--message`. Source inspection confirms the missing feature; the shell-expansion hazard itself is outside OpenClaw once callers use command substitution.

## Solution Assessment

No in the current form. The CLI flag direction is reasonable, but the file read should happen before generic message action dispatch so tool, gateway, and plugin paths cannot inherit a hidden host-file read capability.
