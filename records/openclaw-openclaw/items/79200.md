## Summary

Keep open: this is the active implementation candidate and the duplicate alternative PR is closed, but the current patch still puts the file read in shared outbound action handling, which can bypass the existing sandbox/media file-access boundary.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, documents it, adds a changelog entry, and adds send-validation tests for file content and error handling.

## Best Possible Solution

Keep the CLI flag, but resolve file content at the CLI boundary or through a policy-backed text-file resolver before shared outbound action handling, with regression coverage for exact content and sandbox bypass attempts.

## Reproduction Assessment

Yes. Current main lacks the flag, and the PR head source clearly shows the new file-read path inside the shared runner while the tool path forwards raw params and `sandboxRoot` into that runner.

## Solution Assessment

No. The user-facing flag is the right shape, but reading arbitrary file paths in the shared outbound runner is not the safest implementation; the read should happen in CLI-owned handling or behind the existing file-access policy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep message-file reads inside the CLI boundary:** `src/infra/outbound/message-action-runner.ts:632`
  - body: `handleSendAction` is shared by CLI and agent tool execution, and `createMessageTool` forwards raw params plus `sandboxRoot` into `runMessageAction`. Reading `messageFile` here with bare `fs.readFile` lets a non-CLI caller that can supply that field read and send host files outside the sandbox/media policy. Resolve the file in CLI-owned input handling or through a policy-backed resolver before this shared handler.
  - confidence: 0.9
