## Summary

Keep this PR open for revision: the feature is plausible, but the current implementation would still mutate file contents after reading them and the supplied proof does not show the actual delivered payload bytes.

## What This Changes

The PR adds a `--message-file <path>` option to `openclaw message send`, reads that file into the send action, adds send-validation tests, and updates the changelog.

## Best Possible Solution

Keep the additive flag only if maintainers want this CLI surface, but handle file content on a separate path that skips inline-message trimming and escape normalization, assert exact payload text in tests, and document the flag on the public message CLI page.

## Reproduction Assessment

Yes for the review finding by source inspection: after the PR assigns file content to `params.message`, current-main code trims message strings and rewrites literal `\\n`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The PR is a reasonable direction, but routing file bytes through the existing inline `--message` normalization is not the narrowest maintainable implementation for an exact file-input escape hatch.
