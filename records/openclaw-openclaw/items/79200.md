## Summary

Keep open: the feature is useful, but the latest PR head still reads `messageFile` inside the shared outbound send runner, creating a host-file-read path outside the CLI boundary, and the supplied proof does not exercise the actual `openclaw message send --message-file` CLI path.

## What This Changes

This PR adds `--message-file <path>` to `openclaw message send`, reads UTF-8 file content for the message body, updates docs/changelog, and adds send-validation tests.

## Best Possible Solution

Resolve `--message-file` in the CLI helper layer, pass only sanitized `message` text into `messageCommand`/`runMessageAction`, keep the shared runner host-file-free, and add terminal proof for the real CLI command.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR head reads `messageFile` in `handleSendAction`, while current main's agent message tool forwards params into that shared runner. The original request is a new CLI input mode, so there is no current-main OpenClaw bug reproduction beyond the shell-expansion limitation of passing arbitrary file content through `--message`.

## Solution Assessment

No. The CLI flag is a reasonable direction, but reading the file inside the shared runner is not the narrowest maintainable fix; the file read should happen before shared outbound dispatch.
