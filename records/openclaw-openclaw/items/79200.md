## Summary

Keep open: the feature is useful and the happy-path proof is credible, but the patch reads `messageFile` inside the shared message action runner with a bare host `fs.readFile`, which bypasses the existing sandbox/media read policy for non-CLI callers.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, plus validation, tests, docs, and a changelog entry.

## Best Possible Solution

Keep the CLI flag, but resolve file content at the CLI boundary or through the existing sandbox/media read policy before shared action handling, with regression coverage for exact content and sandbox bypass attempts.

## Reproduction Assessment

Yes. Current main source shows no `--message-file` option and only inline `message` text handling; the PR proof shows the new temp-file happy path, and the sandbox bypass concern is source-reproducible from the shared tools.message call path.

## Solution Assessment

No. The user-facing flag is the right shape, but the file read belongs in CLI-owned input handling or a policy-backed resolver, not as an unrestricted read inside shared `handleSendAction`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep message-file reads inside file access policy:** `src/infra/outbound/message-action-runner.ts:632`
  - body: `handleSendAction` is used by CLI, gateway, and `tools.message`; `createMessageTool` forwards raw args plus `sandboxRoot` into this shared runner. Reading `messageFile` here with bare `fs.readFile` lets any caller that can supply that param read and send arbitrary host files, bypassing the sandbox/media local-root checks that existing media paths go through. Resolve the file in the CLI layer before calling the shared action, or read it through the existing sandbox/media access policy.
  - confidence: 0.9
