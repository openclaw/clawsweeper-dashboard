## Summary

Keep open: the feature is useful and the PR has adequate terminal proof, but the current patch still reads caller-provided files inside shared message-action handling, which is a security boundary regression for non-CLI callers.

## What This Changes

The PR adds `--message-file <path>` to `openclaw message send`, updates CLI docs and the changelog, and adds send-validation coverage for file input, exact content preservation, mutual exclusion, and missing files.

## Best Possible Solution

Keep the CLI flag, but resolve file content at the CLI boundary or through an existing policy-backed text/media resolver before shared message action handling, with sandbox bypass regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main lacks `--message-file`, and the PR head shows the new security-sensitive behavior clearly: a shared send runner reads `messageFile` directly while the message tool can pass raw params and a sandbox root into that runner.

## Solution Assessment

No. The user-facing flag is a good direction, but the implementation should not read arbitrary file paths in shared outbound handling; the narrower maintainable fix is CLI-side resolution or a policy-backed file resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep message-file reads behind the CLI/policy boundary:** `src/infra/outbound/message-action-runner.ts:632`
  - body: `handleSendAction` is shared by CLI and the agent message tool, which forwards raw params and `sandboxRoot` into `runMessageAction`. Reading `messageFile` here with bare `fs.readFile` lets any caller that can supply that param read host files outside the sandbox/media policy and send their contents. Resolve the file in CLI input handling, or route it through the existing policy-backed file access boundary.
  - confidence: 0.9
