## Summary

Keep open: the CLI flag is useful, and the author supplied follow-up runner-level validation, but the latest patch still reads `messageFile` inside the shared outbound runner that is also reached by agent/tool dispatch.

## What This Changes

This PR adds `--message-file <path>` to `openclaw message send`, documents it, updates the changelog, and adds outbound send-validation coverage for reading UTF-8 message text from a file.

## Best Possible Solution

Resolve `--message-file` in the CLI helper layer, pass sanitized `message` text into `messageCommand` / `runMessageAction`, keep the shared runner host-file-free, and add actual CLI terminal proof.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR head reads `messageFile` in `handleSendAction`, while current main's agent message tool forwards raw params into that shared runner. The original request is a new CLI input mode, so there is no current-main bug reproduction beyond the known shell-expansion limitation outside OpenClaw.

## Solution Assessment

No. The flag is a reasonable feature direction, but file reading should happen at the CLI boundary before shared outbound dispatch so agent/tool callers cannot trigger host-file reads through hidden params.
