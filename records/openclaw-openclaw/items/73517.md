## Summary

Keep open. Current main still performs the task-maintenance backing-session lookup per stale task, and the same-author closing PR #74406 is still open and unmerged in the provided GitHub context.

## What This Changes

Prevent task registry maintenance from repeatedly loading/cloning session stores and deriving CLI chat type for stale records from `runs.sqlite`.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so one maintenance pass reuses session-store reads and duplicate CLI chat-type derivations while preserving fresh recovery-hook rechecks.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The original live CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass lookup-cache direction is the narrowest maintainable fix for this report because it targets repeated lookup cost without changing task schema, retention policy, startup restore, or automatic SQLite quarantine behavior.
