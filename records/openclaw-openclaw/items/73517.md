## Summary

Keep open. Current main still has the reported repeated task-maintenance lookup path, and same-author PR #74406 is open and unmerged with closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that task registry maintenance stop entering sustained high CPU when stale or large `runs.sqlite` state causes repeated backing-session and chat-type lookups.

## Best Possible Solution

Land PR #74406 or an equivalent narrow change that reuses session-store reads and duplicate CLI chat-type derivations within one maintenance pass while preserving the fresh recheck around recovery hooks and changed tasks. Keep malformed SQLite quarantine or automatic rebuild behavior on the separate #71689 track.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The full live CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass lookup-cache direction is the narrowest maintainable fix because it targets repeated lookup cost without changing task schema, retention, startup restore, or automatic database quarantine policy.
