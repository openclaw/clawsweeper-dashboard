## Summary

Keep open. Current main still performs session-store lookups inside each stale subagent/CLI task check, so the reported repeated load/clone path remains, and open PR #74406 is the active same-author implementation candidate with closing syntax rather than a reason to close this issue before merge.

## What This Changes

The issue requests preventing gateway task registry maintenance from sustaining high CPU when stale `runs.sqlite` task records repeatedly reload or clone backing session state.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so one maintenance pass reuses backing-session store reads and duplicate CLI chat-type results while preserving the fresh recheck around recovery hooks and changed task records. Keep malformed SQLite quarantine or rebuild behavior in the separate #71689 track.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing one child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` calls. The full live high-CPU magnitude still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix for this issue because it targets repeated lookup cost without changing task schema, retention, startup restore, or corruption recovery policy. The current lightweight chat-type parser is useful partial hardening but does not remove the repeated store clone path.
