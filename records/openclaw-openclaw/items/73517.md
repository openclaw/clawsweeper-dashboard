## Summary

Keep open. Current main still reloads and clones session-store state through the task-maintenance backing-session check for each stale subagent/CLI task; the issue also has the same-author closing PR #74406 open and unmerged in the provided context, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Prevent Gateway task-registry maintenance from sustaining high CPU on stale `runs.sqlite` data by reusing expensive backing-session lookups or otherwise bounding pathological maintenance work.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow per-maintenance-pass lookup cache, while preserving the fresh recovery-hook recheck and leaving broader malformed-SQLite quarantine to the separate registry-corruption track.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same child session store or channel session key, run `runTaskRegistryMaintenance`, and assert repeated `loadSessionStore`/chat-type work. The exact live CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked per-pass backing-session cache is the narrowest maintainable direction because it removes repeated synchronous session-store load/clone work without changing task schema, retention policy, startup restore, or automatic SQLite recovery behavior.
