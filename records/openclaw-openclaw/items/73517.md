## Summary

Keep open. Current main still performs task-maintenance backing-session checks by calling the session-store load and CLI chat-type derivation path per stale task, and the same-author closing PR #74406 remains open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Prevent Gateway task registry maintenance from burning CPU by repeatedly reloading or cloning session stores and deriving chat type while reconciling stale task records in `runs.sqlite`.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so one maintenance pass reuses session-store reads and duplicate CLI chat-type derivations while preserving the fresh recheck around recovery hooks and changed tasks. Leave malformed SQLite quarantine or rebuild behavior to the separate #71689 track.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share the same child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The live CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass lookup-cache direction is the narrowest maintainable fix for this issue because it targets repeated lookup cost without changing task schema, retention policy, startup restore, or automatic quarantine behavior.
