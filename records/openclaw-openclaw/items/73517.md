## Summary

Keep open. Current main still performs the reported per-task session-store lookup/clone work during stale subagent and CLI task reconciliation, and the linked closing PR remains the active implementation candidate rather than a merged fix.

## What This Changes

Task registry maintenance should avoid sustained gateway CPU burn when stale task records in `runs.sqlite` repeatedly hit the same session-store and CLI chat-type lookup paths.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so a single maintenance pass reuses session-store reads and duplicate CLI chat-type derivations while preserving the fresh recovery-hook recheck. Keep broader malformed SQLite quarantine/rebuild behavior tracked separately in #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI tasks sharing the same child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The original full high-CPU reproduction depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass lookup context is the narrowest maintainable fix for the reported hot path. It avoids changing task schema, retention policy, startup database recovery, or automatic quarantine behavior.
