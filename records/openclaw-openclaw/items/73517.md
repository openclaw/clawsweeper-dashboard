## Summary

Keep open. Current main still has the repeated per-task session-store lookup/clone path for stale subagent and CLI task reconciliation, and the linked closing PR is still the active implementation candidate rather than a merged fix. Main does have a lightweight chat-type parser and batch yielding, but those do not remove the repeated store lookup cost described by the report.

## What This Changes

Task registry maintenance should avoid sustained high CPU when stale task records repeatedly hit the same session-store and CLI chat-type lookup paths.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so one maintenance pass reuses session-store reads and duplicate CLI chat-type derivations while preserving the fresh recheck around recovery hooks and changed tasks. Keep broader malformed SQLite recovery/quarantine work in #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The original sustained CPU profile still depends on the reporter's private task registry data.

## Solution Assessment

Yes, the per-maintenance-pass lookup context proposed by PR #74406 is the narrowest maintainable fix for the reported hot path. It avoids changing task schema, retention policy, startup database recovery, or automatic quarantine behavior.
