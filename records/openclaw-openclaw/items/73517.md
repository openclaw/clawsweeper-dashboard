## Summary

Keep open. Current main still performs the task-maintenance backing-session lookup per stale subagent/CLI task, no per-maintenance-pass cache context exists, and the paired PR #74406 is still the implementation candidate in the provided context.

## What This Changes

Task registry maintenance should avoid repeated session-store loads/clones and duplicate chat-type derivation when stale task rows share the same backing session data.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so one maintenance pass reuses session-store reads and duplicate CLI chat-type derivations while preserving fresh recovery-hook rechecks.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same child session store or channel session key, run `runTaskRegistryMaintenance()`, and count repeated `loadSessionStore` plus chat-type derivation calls. The original live CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes. The linked per-maintenance-pass lookup-cache direction is the narrowest maintainable fix because it targets repeated lookup cost without changing task schema, retention policy, startup restore, or automatic SQLite quarantine behavior.
