## Summary

Keep open. Current main still performs backing-session checks per stale task, including repeated session-store loads/clones and CLI chat-type derivation, and the same-author closing PR #74406 is still the appropriate implementation path in the provided context.

## What This Changes

Task registry maintenance should avoid sustained high CPU when stale runs.sqlite entries repeatedly hit expensive session-store and chat-type lookup paths.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so each maintenance pass reuses session-store reads and duplicate CLI chat-type results while preserving fresh recovery-hook rechecks.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same child session store or channel session key, run runTaskRegistryMaintenance, and count repeated loadSessionStore and chat-type checks. The original full CPU profile still depends on the reporter's private runs.sqlite state.

## Solution Assessment

Yes, the linked per-maintenance-pass lookup-cache direction is the narrowest maintainable fix because it targets repeated lookup cost without changing task schema, retention policy, startup restore, or automatic SQLite quarantine behavior.
