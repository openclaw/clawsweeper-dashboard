## Summary

Keep open. Current main has addressed the plugin-walking chat-type side path, but it still resolves and clones session-store data per stale subagent/CLI task; the linked closing PR #74406 remains open rather than merged into main.

## What This Changes

The issue requests preventing gateway task registry maintenance from sustaining high CPU when stale task records repeatedly trigger expensive session-store lookups against `runs.sqlite` state.

## Best Possible Solution

Review, repair, and land PR #74406 or an equivalent narrow patch that caches backing-session lookups per maintenance pass while preserving the lightweight chat parser and wedged-subagent recovery semantics.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same child session store, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` calls. The original full CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes in direction: a pass-local backing-session lookup cache is the narrowest maintainable fix because it targets repeated synchronous lookup and clone cost without changing task schema, retention, statuses, or user config.
