## Summary

Keep open. Current main still performs the repeated per-task session-store lookup/clone path described in the report, and the linked closing PR remains open and unmerged.

## What This Changes

The issue asks OpenClaw to prevent gateway task registry maintenance from burning CPU when stale `runs.sqlite` task records repeatedly load or clone the same session state.

## Best Possible Solution

Land PR #74406 or an equivalent narrow task-maintenance change that reuses session-store reads and duplicate CLI chat-type derivations within a maintenance pass while preserving the fresh recheck around recovery hooks and changed tasks. Keep automatic malformed SQLite quarantine or rebuild behavior scoped to #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share the same child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The original sustained CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes. A per-maintenance-pass backing-session lookup context is the narrowest maintainable direction because it targets repeated lookup cost without changing task schema, retention policy, startup restore, or automatic quarantine behavior.
