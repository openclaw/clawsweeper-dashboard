## Summary

Keep open. Current main still has the per-task session-store lookup and chat-type derivation path described by the report, and the same-author closing PR #74406 remains the active unmerged implementation candidate rather than a reason to close this issue.

## What This Changes

The issue requests preventing gateway task registry maintenance from burning CPU on stale or large `runs.sqlite` state by bounding or caching repeated session/backing-state checks.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so one maintenance pass reuses session-store reads and duplicate CLI chat-type derivations while preserving the fresh recheck around recovery hooks and changed tasks. Keep malformed SQLite quarantine/rebuild behavior in the separate #71689 track.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share the same child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The original sustained CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix for this issue because it targets repeated lookup cost without changing task schema, retention policy, startup restore, or automatic quarantine behavior.
