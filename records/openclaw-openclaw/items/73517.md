## Summary

Keep open: current main still has the repeated per-task session-store lookup/clone path described by the report, and the linked closing PR is still the active unmerged implementation candidate.

## What This Changes

The issue asks OpenClaw to prevent gateway task registry maintenance from sustaining high CPU on stale `runs.sqlite` state by avoiding repeated session-store loads/clones and duplicate chat-type derivation during one maintenance pass.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow change, so one maintenance pass reuses session-store reads and duplicate CLI chat-type derivations while preserving the fresh recheck around recovery hooks and changed tasks. Keep broader malformed SQLite quarantine/rebuild behavior tracked separately in #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share the same child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The original sustained CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked PR's per-maintenance-pass lookup context is the narrowest maintainable direction for this issue because it targets the repeated lookup cost without changing task schema, retention policy, startup restore, or automatic quarantine behavior.
