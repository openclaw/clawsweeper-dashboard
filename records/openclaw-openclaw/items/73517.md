## Summary

Keep open. Current main still resolves backing sessions per stale task and clones cached session stores on each lookup, while the same-author closing PR #74406 is still open and unmerged in the provided context.

## What This Changes

The issue requests preventing gateway task registry maintenance from hot-looping when stale task records repeatedly trigger expensive session-store and chat-type lookups.

## Best Possible Solution

Review and land PR #74406, or an equivalent narrow task-maintenance change, so each maintenance pass reuses session-store reads and duplicate CLI chat-type results while preserving fresh recovery-hook rechecks.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same child session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type checks. The original sustained CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass lookup-cache direction is the narrowest maintainable fix for this issue because it targets repeated lookup cost without changing task schema, retention policy, startup restore, or automatic SQLite quarantine behavior.
