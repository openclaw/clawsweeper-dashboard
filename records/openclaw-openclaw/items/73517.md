## Summary

Keep open. Current main still has the reported per-task session-store lookup/clone path during stale subagent and CLI task reconciliation, and the open closing PR is still an implementation candidate rather than a merged fix.

## What This Changes

Prevent gateway task registry maintenance from hot-looping on stale `runs.sqlite` data by avoiding repeated session-store reloads/clones and duplicate chat-type derivation during one maintenance pass.

## Best Possible Solution

Land PR #74406 or an equivalent narrow task-maintenance change that reuses session-store reads and duplicate CLI chat-type derivations within one maintenance pass while preserving the fresh recheck around recovery hooks and changed tasks; keep broader malformed SQLite recovery or quarantine work in #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share one session store or channel session key, run `runTaskRegistryMaintenance`, and assert repeated `loadSessionStore` and chat-type derivation calls on current main. The full sustained CPU reproduction still requires the reporter's private `runs.sqlite` data.

## Solution Assessment

Yes. A per-maintenance-pass lookup context, as proposed in #74406, is the narrowest maintainable fix for this hot path because it avoids task schema, retention, startup recovery, and automatic quarantine changes.
