## Summary

Keep open. Current main still has the repeated task-maintenance lookup shape described by the CPU profile, and the linked closing PR is still an open implementation candidate rather than a merged fix.

## What This Changes

Task registry maintenance should avoid sustained high CPU when stale task records repeatedly check the same backing session store and CLI chat type.

## Best Possible Solution

Review and land PR #74406 or an equivalent narrow task-maintenance change that reuses backing-session store reads and duplicate CLI chat-type derivations within one maintenance pass while preserving the fresh recheck around recovery hooks and changed tasks. Broader malformed-database quarantine should remain in the separate task-registry corruption track such as #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and `deriveSessionChatTypeFromKey` calls. The full production CPU profile still requires the reporter's private task registry database or a sanitized equivalent.

## Solution Assessment

Yes, the per-maintenance-pass cache direction in PR #74406 is the narrowest maintainable fix for the reported hot path because it avoids schema, retention, startup repair, and automatic database quarantine changes. Current main's batch yielding helps responsiveness but does not remove the repeated lookup and clone cost.
