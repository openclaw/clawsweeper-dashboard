## Summary

Keep open: current main still downloads Telegram documents and forwards their media metadata, but Office-style attachments are filtered as binary before shared extraction and the bundled document extractor is still PDF-only. The related DOCX implementation PRs were closed unmerged, so the central requested capability remains unresolved.

## What This Changes

The issue requests text extraction for Telegram document attachments so agents can read files such as PDF, DOCX, TXT, and other document formats from message context.

## Best Possible Solution

Finish this through the shared document extraction path with maintainer-approved Office MIME/extractor support, bounded parser resource limits, untrusted-content wrapping, and focused Telegram plus media/input-file coverage.

## Reproduction Assessment

Yes. Current source shows Telegram document attachments reach MediaPaths and MediaTypes, while Office MIME types are treated as binary and skipped before extraction; no live Telegram run was performed in this read-only review.

## Solution Assessment

No to a Telegram-only parser or config toggle. The maintainable solution should extend the shared document-extractor/media path so Telegram, input_file, and other attachment surfaces use one bounded document capability.
