## Summary

Keep open: current main and v2026.6.5 download and forward Telegram document media, and text/PDF extraction exists, but Office-style document text extraction is still skipped before extractor lookup; the related DOCX PR is closed unmerged and the broader read-tool issue does not supersede this Telegram/media-context request.

## What This Changes

The issue requests extracting text from Telegram document attachments so agents receive document contents instead of only media placeholders.

## Best Possible Solution

Finish the capability through the shared document-extractor and media-understanding pipeline with explicit Office MIME support, bounded parsing, preserved untrusted-content wrapping, Telegram/shared-media tests, and docs, while coordinating any filesystem read-tool behavior with https://github.com/openclaw/openclaw/issues/35406.

## Reproduction Assessment

Yes. Current main forwards Telegram document media into shared media handling, but Office MIME types are excluded from default support and skipped as binary before extractor lookup, with tests asserting that skip.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable direction is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
