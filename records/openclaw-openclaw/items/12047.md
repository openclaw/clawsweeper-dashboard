## Summary

Keep open: current main and v2026.5.22 still download and forward Telegram document media, and text/PDF extraction exists, but Office-style document text extraction remains intentionally skipped before extractor lookup. The related DOCX implementation candidate at https://github.com/openclaw/openclaw/pull/57612 is still open, conflicting, and narrower than the full Office attachment request.

## What This Changes

Request that Telegram document attachments have extractable text added to agent-visible context instead of only a `<media:document>` placeholder.

## Best Possible Solution

Finish the capability through the shared document-extractor and media-understanding pipeline with explicit Office MIME support, bounded parsing, preserved untrusted-content wrapping, Telegram/shared-media tests, and docs.

## Reproduction Assessment

Yes, source-reproducible. Current main forwards Telegram document media into shared media handling, but Office MIME types are excluded from default support and skipped as binary before extractor lookup, with tests asserting that skip.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable direction is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
