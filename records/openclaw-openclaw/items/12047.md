## Summary

Keep open: current main handles Telegram document download and shared text/PDF extraction, but Office-style document text extraction is still intentionally blocked before extractor lookup and the related DOCX implementation remains open with security and product questions.

## What This Changes

The issue requests extracting text from Telegram document attachments and including it in agent context instead of leaving only a media placeholder.

## Best Possible Solution

Finish the capability through the shared document-extract/media pipeline with explicit Office MIME support, bounded ZIP/XML extraction, preserved binary-safety checks, untrusted-content wrapping, Telegram/shared-media coverage, and public docs; review or adapt https://github.com/openclaw/openclaw/pull/57612 rather than adding a Telegram-only parser.

## Reproduction Assessment

Yes, source-reproducible. Current main forwards Telegram document media into shared context, but Office MIME types are excluded from default file support and skipped as binary before extractor lookup, with regression coverage asserting that skip.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable path is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
