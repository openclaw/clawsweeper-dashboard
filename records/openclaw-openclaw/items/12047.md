## Summary

Keep open. Current main handles Telegram document download and shared text/PDF extraction, but the requested Office-document part is still missing: DOCX/XLSX/PPTX and legacy Word MIME types are filtered as binary before extraction, while the related DOCX PR remains open and unmerged.

## What This Changes

The issue requests extracting text from Telegram document attachments so agents can read files such as DOCX, PDF, TXT, and Markdown instead of seeing only a document media placeholder.

## Best Possible Solution

Finish this through the shared document-extraction pipeline rather than a Telegram-only path: review or adapt #57612, add bounded Office MIME/extractor support where maintainers are comfortable, preserve binary-safety and untrusted-content wrapping, and cover Telegram handoff plus shared media/input-file behavior with focused tests and docs.

## Reproduction Assessment

Yes. A Telegram DOCX/XLSX/PPTX attachment can be traced from msg.document into MediaPath/MediaTypes, then into applyMediaUnderstanding where Office MIME types are skipped before extraction; current tests assert the skip behavior for XLSX and legacy Word/OLE files.

## Solution Assessment

No for a Telegram-specific extractor. The maintainable solution is shared document-extraction support with explicit parser, MIME-scope, decompression/XML, and resource-limit decisions so Telegram, input_file, read, and other media paths behave consistently.
