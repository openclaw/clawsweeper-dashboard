## Label Justifications

- `P2`: The missing capability affects common document workflows but has bounded attachment and channel impact.
- `impact:message-loss`: Telegram receives and stores the file, but its readable content is omitted from the agent-visible message context.

## Summary

Current main still downloads Telegram documents but intentionally limits shared extraction to text, JSON, and PDF, so Office documents remain unreadable; the broader read-tool issue is related but does not fully own Telegram attachment ingestion.

## What This Changes

Extract readable text from Telegram document attachments and include it in the agent message context.

## Best Possible Solution

Extend the shared document-extractor contract with a deliberately bounded set of Office formats so Telegram and other attachment entry points gain the capability without channel-specific parsing or a new Telegram config switch.

## Reproduction Assessment

Yes. Current source shows Telegram documents reach the shared attachment pipeline, while Office MIME types are deliberately filtered as binary before extraction.

## Solution Assessment

No to a Telegram-specific extractor, shell-tool fallback, or new channel setting. The narrow maintainable solution is shared format-aware extraction with explicit untrusted-input and resource-limit contracts.
