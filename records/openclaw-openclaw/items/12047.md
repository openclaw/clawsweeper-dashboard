## Summary

Keep open. Current main still solves only part of the requested document flow: Telegram document attachments are downloaded and forwarded into the shared media context, and text/PDF extraction exists there, but Office formats remain outside the default supported MIME/extractor path and are still guarded as binary before extraction. The related DOCX implementation remains open, so there is no merged current-main fix for the central Office-document capability requested here.

## What This Changes

The issue asks for Telegram document attachments such as DOCX, PDF, TXT, and Markdown to have their text extracted and included in the agent message context instead of only a media placeholder.

## Best Possible Solution

Finish this in the shared document extraction pipeline with explicit supported Office MIME types, bounded extractors, preserved binary-safety checks, untrusted-content wrapping, Telegram/shared-media coverage, and docs for supported formats.

## Reproduction Assessment

Yes, from source inspection. Telegram forwards saved document media into the shared context, but Office MIME types are not in the default allowed document set and are skipped as binary before extractor lookup, with a regression test asserting that behavior.

## Solution Assessment

No for a Telegram-specific extractor. The maintainable solution is shared document extraction so Telegram, input_file/read flows, and future plugins use the same MIME support, limits, and untrusted-content boundaries.
