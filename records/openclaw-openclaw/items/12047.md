## Summary

Keep open: current main and the latest release still download and forward Telegram document media, but Office-style document text extraction remains blocked by MIME allowlists and binary-safety gates, and the related DOCX implementation candidate is still open with security and product-boundary concerns.

## What This Changes

The issue requests Telegram document attachment text extraction so agents can read documents instead of receiving only media placeholders.

## Best Possible Solution

Finish this through the shared document-extract/media pipeline with explicit Office MIME support, bounded ZIP/XML parsing, preserved untrusted-content wrapping, Telegram/shared-media coverage, and docs; review or adapt https://github.com/openclaw/openclaw/pull/57612 rather than adding a Telegram-only parser.

## Reproduction Assessment

Yes, source-reproducible. Current main forwards Telegram document media into shared context, but Office MIME types are excluded from default file support and skipped as binary before extractor lookup, with tests asserting that skip.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable solution is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
