## Summary

Keep open. Current main downloads Telegram documents and extracts text/PDF content through the shared media path, but Office document formats remain intentionally skipped before extraction, and the active DOCX implementation candidate is still open.

## What This Changes

Request text extraction for Telegram document attachments so document contents reach the agent context instead of only a media placeholder.

## Best Possible Solution

Finish this in the shared document-extraction pipeline by reviewing or adapting #57612, defining Office MIME/parser/resource limits, preserving binary-safety and untrusted-content wrapping, and documenting supported formats.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram upload. A Telegram DOCX/XLSX/PPTX attachment is downloaded and forwarded as media context, then Office MIME types hit the binary skip path before extraction; current tests assert that skip behavior.

## Solution Assessment

No for a Telegram-specific extractor. The maintainable solution is shared document extraction so Telegram, OpenResponses input_file, read/media paths use the same MIME support, limits, and untrusted-content boundaries.
