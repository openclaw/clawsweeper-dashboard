## Summary

Keep open: current main and the latest release cover Telegram document download plus text/PDF extraction, but Office-style document attachments are still excluded from the shared extraction path and the related DOCX implementation PRs are closed unmerged.

## What This Changes

Requested Telegram document attachments to have extractable text included in agent context instead of only a media placeholder.

## Best Possible Solution

Implement maintainer-approved Office document support through the shared document-extractor and media-understanding/input_file paths, preserving binary-safety checks, untrusted-content wrapping, focused Telegram coverage, and docs.

## Reproduction Assessment

Yes, at source level: saved Telegram document paths and MIME types flow into shared media context, but Office MIME types are classified as binary and skipped before extraction. I did not run a live Telegram bot reproduction in this read-only review.

## Solution Assessment

No to a Telegram-only parser or default config toggle. The maintainable solution is shared document extraction so Telegram, input_file, and adjacent attachment surfaces use one bounded document capability.
