## Summary

Keep open: current main and v2026.5.20 download and forward Telegram document media, and shared text/PDF extraction exists, but Office-style document attachments are still excluded before extractor lookup; the related DOCX implementation PR is open and unresolved.

## What This Changes

The issue requests Telegram document attachment text extraction so agents can read formats such as DOCX, PDF, TXT, and Markdown instead of only seeing document placeholders.

## Best Possible Solution

Finish this through the shared document-extractor and media-understanding pipeline with explicit Office MIME support, bounded ZIP/XML parsing, preserved untrusted-content wrapping, Telegram/shared-media coverage, and docs.

## Reproduction Assessment

Yes, source-reproducible. Current main forwards Telegram document media into shared media handling, but Office MIME types are excluded from default support and skipped as binary before extractor lookup, with tests asserting that skip.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable solution is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
