## Summary

Keep open. Current main downloads Telegram documents and can extract text/PDF attachments through the shared media path, but Office documents are still deliberately filtered as binary before extraction and the related DOCX PR remains open and unmerged.

## What This Changes

Request to extract text from Telegram document attachments and include that text in agent context instead of only showing a `<media:document>` placeholder.

## Best Possible Solution

Finish this in the shared document-extraction pipeline, preserving binary-safety checks and applying the behavior consistently across Telegram, input_file, read, and other attachment paths.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run. A Telegram Office attachment is downloaded and forwarded as MediaPath/MediaTypes, then the shared media-understanding path skips Office MIME types before extraction, with current tests asserting that skip behavior.

## Solution Assessment

No for a Telegram-specific extractor. The maintainable path is shared document-extractor support with explicit MIME, parser, and resource-limit policy so all attachment entry points behave consistently.
