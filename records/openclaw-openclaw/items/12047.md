## Summary

Keep open: current main and v2026.5.20 download and forward Telegram document media and support text/PDF extraction, but Office-style attachments are still excluded by MIME/default support and skipped before extraction; the related DOCX implementation at https://github.com/openclaw/openclaw/pull/57612 remains open with proof, parser-boundary, and resource-limit concerns.

## What This Changes

The issue requests extracting text from Telegram document attachments so agents can read document contents instead of only seeing media placeholders.

## Best Possible Solution

Finish this through the shared document-extractor and media-understanding pipeline with explicit Office MIME support, bounded ZIP/XML parsing, preserved untrusted-content wrapping, Telegram/shared-media coverage, and docs; review or adapt https://github.com/openclaw/openclaw/pull/57612 rather than adding a Telegram-only parser.

## Reproduction Assessment

Yes, source-reproducible. Current main forwards Telegram document media into shared media handling, but Office MIME types are excluded from default file support and skipped as binary before extractor lookup, with tests asserting that skip.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable solution is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
