## Summary

Keep open. Current main handles Telegram document download/metadata handoff and shared text/PDF extraction, but Office-style documents are still excluded or skipped before extraction, and the DOCX implementation candidate remains open with security/product questions.

## What This Changes

The issue requests text extraction for Telegram document attachments so agents can read files such as DOCX, PDF, TXT, and Markdown sent through Telegram.

## Best Possible Solution

Finish the capability through the shared document-extract/media pipeline with explicit Office MIME support, bounded ZIP/XML extraction, preserved binary-safety checks, untrusted-content wrapping, Telegram/shared-media coverage, and public docs; review or adapt https://github.com/openclaw/openclaw/pull/57612 rather than adding a Telegram-only parser.

## Reproduction Assessment

Yes, source-reproducible. Telegram forwards saved document media into shared context, but Office MIME types are excluded from default file support and skipped as binary before extractor lookup, with regression coverage asserting that skip.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable path is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
