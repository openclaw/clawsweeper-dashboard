## Summary

Keep open. Current main handles Telegram document download and shared text/PDF extraction, but Office-style document text extraction is still blocked before extractor lookup; prior review comments and the open DOCX PR are useful context, but they do not resolve the remaining capability.

## What This Changes

Request to extract text from Telegram document attachments such as TXT, Markdown, PDF, DOCX, XLSX, and PPTX and include it in agent context instead of only a media placeholder.

## Best Possible Solution

Finish this through the shared document-extract/media pipeline with explicit supported Office MIME types, bounded extraction, preserved binary-safety checks, untrusted-content wrapping, Telegram/shared-media coverage, and docs.

## Reproduction Assessment

Yes, for the remaining gap from source inspection. Telegram forwards saved document media into shared context, but Office MIME types are outside the default file allowlist and are skipped as binary before extractor lookup, with regression coverage asserting that behavior.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable path is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
