## Summary

Keep open. Current main handles Telegram document download and shared text/PDF extraction, but Office documents remain unsupported, and the new document-extractor runtime can load bundled artifacts only. The earlier DOCX implementation at https://github.com/openclaw/openclaw/pull/57612 closed unmerged, leaving a real core plugin-loading and security-policy decision before this capability can be implemented cleanly.

## What This Changes

Add text extraction for document attachments received through Telegram so agents can read and process formats such as PDF, TXT, Markdown, and Office documents.

## Best Possible Solution

Add a registry-aware external document-extractor loading contract with enforceable resource limits, then provide Office formats through a vetted plugin; bundle only formats that maintainers explicitly choose to own.

## Reproduction Assessment

Yes. Current source shows Telegram preserves document metadata and passes downloaded media onward, while the bundled extractor supports only PDF and the resolver cannot load external document extractors.

## Solution Assessment

No, Telegram-specific parsing or a new channel configuration flag is not the best solution. The narrow maintainable direction is a secure registry-aware extractor plugin seam followed by bounded Office-format implementations.
