## Summary

Keep open. Current main downloads Telegram document media and extracts text/PDF through shared paths, but Office-style MIME types are still intentionally skipped before extraction and the active DOCX implementation PR remains open and conflicting.

## What This Changes

Request text extraction for Telegram document attachments so agents can read files instead of only seeing a media placeholder.

## Best Possible Solution

Finish this through the shared document-extraction pipeline with bounded Office MIME/extractor support, preserved binary-safety checks, untrusted-content wrapping, focused tests, and docs for supported formats.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram upload. A Telegram Office document is downloaded and forwarded as media context, then Office MIME types hit the binary skip path before extraction, with regression coverage asserting that skip behavior.

## Solution Assessment

No for a Telegram-specific extractor. The maintainable solution is shared document extraction so Telegram, `input_file`, read/media paths, and future plugins use the same MIME support, limits, and untrusted-content boundaries.
