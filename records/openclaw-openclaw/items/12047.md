## Summary

Keep open. Current main handles Telegram document download and shared text/PDF extraction, but Office document text extraction remains explicitly skipped and the related DOCX implementation PR is still open/unmerged.

## What This Changes

The issue requests text extraction for Telegram document attachments so agents can read files such as DOCX, PDF, TXT, and Markdown instead of receiving only a document placeholder.

## Best Possible Solution

Finish this in the shared document-extraction pipeline rather than a Telegram-only path: review or adapt #57612, define parser/dependency/resource limits for trusted Office MIME scope, preserve untrusted-content wrapping and binary-safety checks, and document the supported formats.

## Reproduction Assessment

Yes from source inspection. A Telegram DOCX/XLSX/PPTX attachment flows from msg.document into MediaPath/MediaTypes, then Office MIME types are skipped before extraction; current tests assert that skip behavior.

## Solution Assessment

No for a Telegram-specific extractor. The maintainable solution is shared document extraction with explicit Office parser, MIME-scope, and resource-limit decisions so Telegram, input_file, read, and other media paths behave consistently.
