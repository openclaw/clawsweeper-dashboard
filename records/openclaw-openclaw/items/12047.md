## Summary

Keep open: current main downloads and forwards Telegram document media, and shared text/PDF extraction exists, but Office-style document extraction is still blocked by MIME allowlists, binary-safety gates, and an open unmerged DOCX PR.

## What This Changes

Request to extract text from Telegram document attachments and include it in agent-visible message context.

## Best Possible Solution

Finish this through the shared document-extractor/media-understanding pipeline with explicit Office MIME support, bounded ZIP/XML parsing, preserved untrusted-content wrapping, Telegram/shared-media coverage, and docs; review or adapt https://github.com/openclaw/openclaw/pull/57612 rather than adding a Telegram-only parser.

## Reproduction Assessment

Yes, source-reproducible. Current main forwards Telegram document media into shared media handling, but Office MIME types are excluded from default file support and skipped as binary before extractor lookup, with tests asserting that skip.

## Solution Assessment

No for a Telegram-specific extractor or config-only toggle. The maintainable solution is shared document extraction so Telegram, input_file/read flows, and future plugins reuse the same MIME support, limits, and untrusted-content boundaries.
