## Summary

Keep open. Current main downloads and forwards Telegram documents, and it already handles text/PDF-style extraction through shared paths, but Office-format extraction remains missing: DOCX/XLSX/PPTX MIME families are outside the default file MIME set, are treated as binary before extractor dispatch, and the related DOCX implementation PR remains open with security/product-boundary concerns.

## What This Changes

The issue requests extracting text from Telegram document attachments so agents can read files such as PDF, TXT, DOCX, XLSX, and PPTX instead of receiving only a media placeholder.

## Best Possible Solution

Finish this through the shared document extraction pipeline: review or replace the open DOCX PR with a bounded document-extract plugin implementation, add explicit Office MIME support only where limits are enforceable, preserve untrusted-content wrapping, and cover Telegram plus shared input-file/read behavior with focused tests and docs.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram upload. A Telegram Office document reaches shared media context, then Office MIME types are classified as binary and skipped before document extraction, with tests asserting that skip behavior.

## Solution Assessment

No for a Telegram-specific extractor or unbounded parser. The maintainable solution is shared document extraction so Telegram, input_file, read/media paths, and future plugins use the same MIME support, limits, and untrusted-content boundaries.
