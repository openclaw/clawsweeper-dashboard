## Summary

Keep open. The core Telegram attachment download path is present, but current `main` still does not extract Office documents: DOCX/XLSX/PPTX MIME types are not in the default supported file MIME set, the shared media path treats the relevant Office MIME families as binary before extractor dispatch, and current tests explicitly preserve those skips while a related DOCX implementation PR remains open.

## What This Changes

The issue requests text extraction for Telegram document attachments so agents can read uploaded PDFs, DOCX, TXT, Markdown, and similar files instead of seeing only a media placeholder.

## Best Possible Solution

Finish this in the shared document extraction pipeline with bounded Office MIME/extractor support, preserved binary-safety checks, untrusted-content wrapping, focused Telegram/shared-media tests, and docs for supported formats.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram upload. A Telegram Office document is downloaded and forwarded as media context, then Office MIME types hit the binary skip path before extraction, with regression coverage asserting that skip behavior.

## Solution Assessment

No for a Telegram-specific extractor. The maintainable solution is shared document extraction so Telegram, `input_file`, read/media paths, and future plugins use the same MIME support, limits, and untrusted-content boundaries.
