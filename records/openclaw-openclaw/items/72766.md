## Summary

Keep this PR open. Current main still has the heartbeat session-store race the PR targets, and the code change is the right narrow direction, but the branch needs the required changelog entry plus normal maintainer rebase and Testbox validation before merge.

## What This Changes

The PR changes `src/infra/heartbeat-runner.ts` so heartbeat task timestamp writes and delivered-heartbeat dedupe writes use `updateSessionStore` instead of a separate load/mutate/save session-store snapshot.

## Best Possible Solution

The desired end state is for heartbeat session-store writes that preserve existing session metadata to use the locked re-read store path, with a single-line Unreleased/Fixes changelog entry and focused coverage for task timestamp and dedupe writes under concurrent inbound metadata updates.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to let heartbeat read a session-store snapshot, perform a concurrent inbound metadata update through the locked store path, then let heartbeat save the stale snapshot and observe fields such as route metadata, `messageCount`, `lastMessageAt`, or `updatedAt` being lost.

## Solution Assessment

Yes. Reusing `updateSessionStore` is the narrowest maintainable code fix because it applies the existing lock and re-read contract to the two unsafe heartbeat writes without adding a new concurrency mechanism; the remaining mechanical gap is the changelog entry.
