## Summary

Keep this PR open. Current main still has the stale heartbeat session-store load/mutate/save writes, and the PR applies the existing locked `updateSessionStore` helper to the two unsafe heartbeat paths without introducing a blocking correctness or security issue.

## What This Changes

The branch updates `src/infra/heartbeat-runner.ts` so heartbeat task timestamp and delivered-heartbeat dedupe writes use `updateSessionStore` instead of separate `loadSessionStore` and `saveSessionStore` snapshots.

## Best Possible Solution

Land a rebased version of this focused helper-based fix, preferably with one regression test that interleaves a heartbeat store write with a session metadata update and proves metadata survives; then treat #41254 as superseded for the dedupe-only slice.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to let heartbeat read a session-store snapshot, make a concurrent inbound metadata or route update through the locked store path, then let heartbeat save the stale snapshot and observe metadata fields being lost on current main.

## Solution Assessment

Yes. Reusing `updateSessionStore` is the narrowest maintainable fix because it applies the existing session-store lock and re-read contract to the two unsafe heartbeat writes without adding a new concurrency mechanism.
