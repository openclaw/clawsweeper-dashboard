## Summary

Keep this PR open. Current main still has the stale session-store load/mutate/save pattern in both heartbeat writes, and the PR's switch to `updateSessionStore` matches the existing lock-and-re-read contract; the remaining maintainer cleanup is the required changelog entry before merge.

## What This Changes

The PR changes heartbeat task-timestamp and delivered-heartbeat dedupe writes in `src/infra/heartbeat-runner.ts` to mutate the session store through `updateSessionStore` instead of a separate `loadSessionStore`/`saveSessionStore` snapshot.

## Best Possible Solution

Land the locked `updateSessionStore` heartbeat write fix with a single-line `Unreleased` / `Fixes` changelog entry and avoid changing the broader session-store locking model unless a separate regression proves it is needed.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to let heartbeat read a session-store snapshot, perform a concurrent inbound metadata update through `recordSessionMetaFromInbound` or `updateLastRoute`, then let heartbeat save the stale snapshot and observe metadata such as `messageCount`, route data, `lastMessageAt`, or `updatedAt` being lost.

## Solution Assessment

Yes. Reusing `updateSessionStore` is the narrowest maintainable code fix because it applies the existing lock and re-read contract to the two unsafe heartbeat writes without introducing another synchronization mechanism; the PR still needs the mechanical changelog follow-up.
