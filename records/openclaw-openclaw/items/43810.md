## Summary

Keep this PR open, but not merge-ready. Current main still defines DM pairing replies as one-time per newly created request, while this patch changes the shared helper to redisclose an existing active pairing code on every repeat message and also makes creation callbacks/result semantics fire as if a new request was created.

## What This Changes

The PR changes shared DM pairing challenge issuance so an existing pending pairing request with a stored code sends the pairing reply again instead of returning early.

## Best Possible Solution

Keep the current one-time pairing-code disclosure contract, or land only a maintainer-approved resend design that preserves real creation semantics and updates shared/channel tests plus public docs.

## Reproduction Assessment

Yes. Source inspection is high-confidence: `upsertChannelPairingRequest` returns `{ code, created: false }` for existing pending requests, and the PR's `!code` guard would fall through to send the reply and return `created: true`.

## Solution Assessment

No. The one-line guard is too broad; if repeat resends are approved, the fix should preserve `created`/`onCreated` as first-creation signals and add matching docs plus shared/channel regression tests.
