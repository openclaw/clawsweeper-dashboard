## Summary

Keep open. Current main still intentionally suppresses repeat DM pairing replies, and this PR’s one-line shared-helper change is not ready because it reuses the creation path for repeated active-code delivery across channels.

## What This Changes

The PR changes the shared pairing challenge guard so an existing pending request with a nonempty code sends another pairing reply on repeat `/start` or DM triggers instead of returning `{ created: false }`.

## Best Possible Solution

Model repeat delivery as an explicit pairing reminder path that preserves creation-only signals, applies cooldown or suppression policy, and updates shared/channel tests, security docs, and changelog together.

## Reproduction Assessment

Yes. Mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`: current main returns `{ created: false }` without sending, while this PR falls through to callback/reply delivery and a created-style result.

## Solution Assessment

No. The one-line guard change conflates first-time creation with reminder delivery; the safer maintainable path is a named reminder contract with preserved creation semantics and explicit abuse controls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve creation-only side effects:** `src/pairing/pairing-challenge.ts:31`
  - body: Changing this guard to `!code` lets repeat pending requests returning `{ code, created: false }` continue into `onCreated`, reply delivery, and a `{ created: true, code }` result. Channel callers use those signals for first-request logging or sends, so repeated messages would be reported and handled as new pairing requests.
  - confidence: 0.93
- **[P2] Define repeat-code delivery policy before resending:** `src/pairing/pairing-challenge.ts:31`
  - body: Current tests and docs say repeated DMs do not resend codes, while Matrix has a separate reminder path with cooldown semantics. This shared guard change would resend the same valid pending code on every repeat trigger without a named reminder policy, cooldown, or updated security contract.
  - confidence: 0.9
