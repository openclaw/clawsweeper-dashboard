## Summary

Keep open: this PR requests a user-visible pairing reminder feature, but the current shared helper, tests, and security docs intentionally keep repeat pending requests from sending another code. The one-line guard change would also make repeat requests look newly created to callers.

## What This Changes

The PR changes `issuePairingChallenge` to return early only when no code exists, so existing pending requests with a code fall through to callback and reply delivery.

## Best Possible Solution

Add an explicit pairing reminder path that preserves first-creation semantics, defines cooldown or suppression policy, and updates shared/channel tests, security docs, and changelog together.

## Reproduction Assessment

Yes. From source, mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`: current main returns `{ created: false }` without sending, while this PR would fall through to callback and reply delivery.

## Solution Assessment

No. The one-line guard change conflates first-time creation with reminder delivery; a named reminder contract with explicit abuse controls is the maintainable path.
