## Summary

Keep this PR open, but it is not merge-ready: current main intentionally sends DM pairing codes only when a new request is created, while this one-line change would resend existing pending codes and break the shared creation contract.

## What This Changes

The PR changes shared DM pairing challenge issuance to resend a stored pending pairing code when an existing request still has a code.

## Best Possible Solution

Keep the current one-time contract unless maintainers explicitly approve repeat-code disclosure; if approved, implement a repeat-send path that returns `created: false`, keeps `onCreated` first-request-only, and updates shared tests plus docs.

## Reproduction Assessment

Yes for the PR behavior delta: source inspection shows `{ code, created: false }` would fall through after the patch, and the inspected Telegram screenshot shows repeated `/start` replies. This is not a current-main bug reproduction because current docs and tests define the no-resend behavior.

## Solution Assessment

No as proposed: the one-line guard changes shared helper policy and creation semantics together. The safer design is a deliberate repeat-send path, only after maintainer approval, that preserves `created` and `onCreated` as first-request-only signals.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve creation semantics when resending pending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: With this guard, an existing pending request that returns `{ code, created: false }` falls through to `onCreated` and then returns `{ created: true, code }`.
