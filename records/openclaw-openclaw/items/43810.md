## Summary

Keep open: the PR addresses a real Telegram UX pain point, and the screenshot shows the proposed visible behavior, but the patch changes shared pairing policy in a way current docs/tests reject and makes creation-only signals fire on repeat requests. Maintainers need to decide the resend policy before this can merge.

## What This Changes

The branch changes `issuePairingChallenge` to resend an existing pending pairing code whenever the store returns a code, instead of only when a request was newly created.

## Best Possible Solution

If repeat pairing reminders are desired, add an explicit reminder contract that preserves creation-only signals, defines cooldown or suppression behavior, and updates shared tests, docs, and live Telegram proof together.

## Reproduction Assessment

Yes. From source, make `upsertPairingRequest` return `{ code: "ABCD", created: false }`: current main returns without sending, while this PR would send the reply and fire `onCreated`; the existing shared test covers the current no-send contract.

## Solution Assessment

No. The one-line guard change is not the best solution because it changes shared creation semantics and repeat-code disclosure policy without a deliberate reminder path.
