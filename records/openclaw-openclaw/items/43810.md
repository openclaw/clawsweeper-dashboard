## Summary

Keep open for maintainer judgment: the UX goal is plausible and has visible Telegram proof, but it changes documented shared DM pairing-code disclosure policy and the current one-line patch repeats first-creation side effects across channels.

## What This Changes

This PR changes `src/pairing/pairing-challenge.ts` so existing pending pairing requests with a stored code resend the pairing reply instead of returning without a response.

## Best Possible Solution

Approve or reject repeat pending-code delivery as an explicit shared pairing policy; if approved, implement a separate resend branch that preserves `created` and `onCreated` semantics and update tests/docs.

## Reproduction Assessment

Yes for the PR behavior delta and review blocker, not as a current-main bug. Source inspection shows current main returns before replying for `{ code, created: false }`, while the branch falls through, and the PR screenshot shows repeated Telegram `/start` replies after the patch.

## Solution Assessment

No. The one-line guard conflates repeat delivery with first creation; the safer implementation is a separate resend path that keeps first-request side effects guarded by `created` after the policy is approved.
