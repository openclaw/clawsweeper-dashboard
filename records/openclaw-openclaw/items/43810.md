## Summary

Keep open for maintainer review: the UX goal is understandable and has visible Telegram proof, but the patch changes shared pairing semantics by treating existing requests as newly issued replies and removes the current bounded code-send behavior without an approved policy.

## What This Changes

The branch changes `src/pairing/pairing-challenge.ts` to send pairing replies whenever the pairing store returns a code, rather than only when a new request is created.

## Best Possible Solution

If repeated reminders are accepted, add an explicit bounded resend branch that preserves `created: false`, keeps `onCreated` creation-only, updates docs/tests, and applies the policy across shared pairing paths.

## Reproduction Assessment

Yes for the proposed patch regression: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and `{ created: true, code }`. The existing shared test models that source path, and the screenshot shows the intended repeated Telegram reply behavior.

## Solution Assessment

No. A one-line guard replacement conflates creation with resend behavior and removes the current once-per-new-request boundary; a bounded resend path with explicit maintainer-approved semantics is safer.
