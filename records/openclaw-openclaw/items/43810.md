## Summary

Keep open: the UX request is plausible, but this branch is not merge-ready because it changes shared creation semantics, conflicts with the current documented/tested repeat-request boundary, and needs an explicit bounded resend policy before broadening pairing-code replies across channels.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending request with a stored code resends the pairing reply on repeat `/start` or DM attempts.

## Best Possible Solution

If maintainers want pending-code resend, add an explicit resend branch that preserves `created: false`, keeps `onCreated` creation-only, applies a cooldown or cap, and updates shared tests, channel coverage, docs, and redacted Telegram proof together.

## Reproduction Assessment

Yes. Source repro is high confidence: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR falls through into reply sending, `onCreated`, and a created result; the inspected Telegram screenshot also shows repeated `/start` replies.

## Solution Assessment

No. The one-line guard change is not the best way to solve the issue because it conflates creation and resend semantics; a bounded, explicit pending-code resend path is the safer maintainable solution.
