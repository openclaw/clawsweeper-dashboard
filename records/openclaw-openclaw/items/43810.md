## Summary

Keep open for maintainer review: the UX goal is plausible, but the current patch conflicts with documented and tested no-resend behavior and broadens valid pairing-code delivery without preserving creation-only semantics.

## What This Changes

The PR changes `issuePairingChallenge` to send a pairing reply whenever the store returns a nonempty code, including existing pending requests.

## Best Possible Solution

Model repeat delivery as an explicit pairing reminder path that preserves first-creation semantics, applies cooldown or suppression policy, and updates shared/channel tests, security docs, and changelog together.

## Reproduction Assessment

Yes. Mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`: current main returns `{ created: false }` without sending, while this PR falls through to callback/reply delivery and a created-style result.

## Solution Assessment

No. The one-line guard change conflates first-time creation with reminder delivery; the safer solution is a named reminder contract with preserved creation semantics and explicit abuse controls.
