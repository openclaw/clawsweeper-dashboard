## Summary

The PR targets a real Telegram UX pain point, and the attached screenshot shows the intended repeated `/start` replies, but the one-line shared-helper change is not merge-ready because it turns repeat pending requests into creation events and removes the documented once-per-new-request resend limit across all shared DM pairing channels.

## What This Changes

The branch changes `src/pairing/pairing-challenge.ts` so pairing replies are sent whenever `upsertPairingRequest` returns a code, not only when it creates a new request.

## Best Possible Solution

If repeated reminders are accepted, add an explicit bounded resend branch that preserves `created: false`, keeps `onCreated` creation-only, documents the resend policy, and updates focused shared plus channel tests.

## Reproduction Assessment

Yes for the proposed patch regression: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and a `{ created: true, code }` return. The screenshot also demonstrates the intended repeated Telegram reply behavior after the change.

## Solution Assessment

No. A one-line guard replacement conflates creation with resend behavior; the maintainable path needs explicit resend semantics and an approved exposure limit.
