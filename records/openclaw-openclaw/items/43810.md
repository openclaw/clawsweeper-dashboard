## Summary

Keep open: the PR proposes a real UX change, but the one-line shared pairing guard change conflates first creation with repeat resend behavior and broadens pairing-code exposure without an approved resend policy.

## What This Changes

The PR changes `issuePairingChallenge` so existing pending pairing requests with a stored code resend the pairing reply on repeated `/start` or DM messages before approval.

## Best Possible Solution

Define an explicit, bounded pending-code resend path only if maintainers want that policy, keep `onCreated` creation-only, and update the shared tests, channel tests, docs, and security wording together.

## Reproduction Assessment

Yes. Source repro is high confidence: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR falls through into reply sending, `onCreated`, and `{ created: true, code }`; the inspected Telegram screenshot also shows repeated `/start` replies.

## Solution Assessment

No. The one-line guard change is not the best solution because it mixes creation and resend semantics; the safer fix is an explicit resend branch with an approved throttle or cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When the store returns an existing `{ code, created: false }`, this guard now falls through, calls `onCreated`, sends a reply, and returns `{ created: true, code }`. Split any repeat-code path so existing requests keep `created: false` and creation-only side effects stay one-time.
  - confidence: 0.95
