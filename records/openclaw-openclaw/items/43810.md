## Summary

Keep this PR open, but not merge-ready: the UX goal is understandable and the screenshot proves the behavior, but the one-line shared-helper change conflicts with current docs/tests and changes pairing creation/security semantics across channels.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a stored code sends the pairing reply again instead of returning early.

## Best Possible Solution

Keep the current one-time contract unless maintainers approve repeat-code disclosure; if approved, add an explicit repeat-reply path that preserves `created: false`, keeps `onCreated` first-request-only, and updates shared tests plus docs.

## Reproduction Assessment

Yes for the PR behavior delta: source inspection shows `{ code, created: false }` would fall through after the patch, and the PR screenshot shows repeated Telegram `/start` replies. This is not a current-main bug reproduction because current docs and tests define the no-resend behavior.

## Solution Assessment

No. The one-line shared-helper change is not the best fix because it changes return and `onCreated` semantics instead of adding an explicit, approved repeat-reply path with tests and documentation.
