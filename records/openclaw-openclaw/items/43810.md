## Summary

Keep open: the UX goal is plausible, but this patch changes shared DM-pairing semantics with unbounded repeat code replies and creation-only callbacks firing on repeats, so it needs maintainer policy/security review before merge.

## What This Changes

The PR changes `issuePairingChallenge` to resend a stored pending pairing code when an unapproved sender repeats `/start`.

## Best Possible Solution

Define an explicit bounded resend policy that preserves `created`/`onCreated` semantics, then update shared tests, affected channel expectations, and pairing docs to match that policy.

## Reproduction Assessment

Yes for the PR-introduced behavior: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and a `{ created: true, code }` result. The inspected Telegram screenshot also shows repeated `/start` replies after the patch.

## Solution Assessment

No. The one-line guard change is not the best fix because it changes shared callback/state semantics and lacks an explicit resend bound or channel-scope policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When the store returns an existing `{ code, created: false }`, this guard now sends a reply, invokes `onCreated`, and returns `{ created: true, code }`. A resend path should keep `onCreated` and the returned `created` state tied to actual new requests.
  - confidence: 0.95
- **[P2] Bound repeated pairing-code replies:** `src/pairing/pairing-challenge.ts:31`

  - confidence: 0.9
