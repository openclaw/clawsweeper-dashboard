## Summary

Keep open: the UX request is concrete and current main has not adopted it, but the patch is not merge-ready because it changes a documented shared DM pairing policy and repeats first-creation side effects.

## What This Changes

The PR changes the shared DM pairing challenge guard so existing pending requests with a stored code resend the pairing reply instead of returning without a response.

## Best Possible Solution

Decide whether repeat pending-code delivery is an approved shared pairing policy; if approved, implement an explicit resend branch that preserves first-created side effects and update tests/docs.

## Reproduction Assessment

Yes for the PR behavior delta and review blocker, not as a current-main bug. Source inspection shows current main returns without replying for `{ code, created: false }`, while the branch falls through, and the PR screenshot shows repeated Telegram `/start` replies after the patch.

## Solution Assessment

No. The one-line guard conflates repeat delivery with first creation; the safer implementation is a separate resend path that keeps `created` and `onCreated` semantics intact after the policy is approved.
