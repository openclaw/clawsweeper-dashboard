## Summary

Keep open for maintainer decision, but the patch is not merge-ready: repeat pairing replies are a feature change from the current shared contract, and the one-line guard change breaks creation semantics, tests/docs, and CI.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a reusable code sends a pairing reply instead of returning early.

## Best Possible Solution

If maintainers want this UX, add an explicit resend-existing-code path that preserves `created` and `onCreated` semantics, then update shared tests, affected channel expectations, and public docs.

## Reproduction Assessment

Yes for the PR defect: source inspection shows an existing `{ code, created: false }` request falls through under the patch, sends another reply, calls creation-side effects, and returns `{ created: true, code }`. The requested resend behavior itself is a feature change from the current documented contract.

## Solution Assessment

No. The one-line guard change is not the best fix because it changes shared creation semantics and leaves tests, channel expectations, and docs inconsistent; an explicit resend path is safer if the product direction is approved.
