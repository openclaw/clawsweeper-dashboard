## Summary

Keep open: the feature has visible Telegram proof, but the one-line patch changes shared pairing creation semantics and pairing-code exposure policy without a resend contract, tests/docs, or throttling.

## What This Changes

The PR changes `issuePairingChallenge` to resend a pending pairing code on repeat `/start` or DM pairing attempts by replacing the `created` guard with a `code` guard.

## Best Possible Solution

If repeat resends are accepted, add an explicit resend branch that preserves `created: false`, keeps `onCreated` creation-only, updates focused tests/docs, and defines a resend cooldown or cap.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows `{ code: "ABCD", created: false }` would fall through after the guard change, call `onCreated`, send a reply, and return `{ created: true, code }`. The user-visible resend behavior is also supported by the PR screenshot.

## Solution Assessment

No. Replacing the guard alone is not the best fix because it changes shared creation semantics and the documented resend policy without a distinct resend path, tests/docs, or abuse limit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ code, created: false }`, this guard now falls through, calls `onCreated`, sends a reply, and returns `{ created: true, code }`. If resends are desired, add a separate resend branch that keeps `created` false and only calls `onCreated` for actual new requests.
  - confidence: 0.94

  - body: The shared helper currently sends pairing messages only for new requests, matching the documented roughly once-per-hour behavior. This guard sends the same pending code on every repeat message across shared pairing channels, so the feature needs an explicit cooldown or cap before merge.
  - confidence: 0.88
