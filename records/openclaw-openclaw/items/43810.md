## Summary

Keep open: the PR has useful Telegram screenshot proof for a real UX change, but it changes documented shared DM pairing-code disclosure policy and the one-line guard also breaks first-creation semantics across callers.

## What This Changes

The PR changes `issuePairingChallenge` to resend a stored pairing code for an existing pending request by returning only when no code is available.

## Best Possible Solution

End state should be an explicit, approved repeat-resend policy in the shared pairing flow that preserves first-request signals and updates tests plus security docs.

## Reproduction Assessment

Yes for the proposed behavior delta, not as a current-main bug. With an upsert result of `{ code: "ABCD", created: false }`, current main returns without replying while this PR enters the reply path.

## Solution Assessment

No. If repeat delivery is approved, the maintainable fix is a separate resend branch that keeps `created` and `onCreated` tied to first creation rather than broadening the existing path.
