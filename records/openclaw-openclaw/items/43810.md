## Summary

Keep this PR open, but the current patch should not merge: the UX goal is plausible and the screenshot proves repeat Telegram replies, yet the one-line guard change changes shared creation semantics and repeats pairing codes without an explicit resend policy.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a stored code receives another pairing-code reply instead of returning early.

## Best Possible Solution

If repeat sends are accepted, add an explicit resend branch that returns `created: false`, gates `onCreated` to true creations, updates tests/docs, and defines a cooldown or cap.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows `{ code, created: false }` would fall through to `onCreated`, send a reply, and return `created: true`. Current main itself matches the documented no-resend behavior.

## Solution Assessment

No. Replacing the guard alone is not the best fix because it changes shared creation semantics and security posture; the maintainable path needs an explicit resend policy plus tests and docs.
