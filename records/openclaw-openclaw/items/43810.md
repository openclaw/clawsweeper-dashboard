## Summary

Keep open, but the current patch is not mergeable: it changes a documented shared DM-pairing suppression policy into unconditional repeat code delivery and also turns creation-only callbacks/results into repeat-request signals.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending request with a code can send another pairing reply instead of returning early.

## Best Possible Solution

Add an explicit pairing reminder/resend contract, with creation-only callbacks preserved, cooldown or suppression policy defined, and shared/channel tests plus docs updated together if maintainers accept the behavior.

## Reproduction Assessment

Yes. From source, mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`: current main returns `{ created: false }` without sending, while this PR would fall through to callback and reply delivery; I did not run tests because this review is read-only.

## Solution Assessment

No. The one-line guard change conflates first-time pairing creation with reminder delivery; a named reminder path with explicit abuse controls is the maintainable solution.
