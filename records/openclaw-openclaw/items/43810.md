## Summary

Keep open: the UX goal is plausible, but this is a shared DM pairing access-control policy change and the current branch is not merge-ready. Current main, tests, and public docs still define no-repeat behavior for existing pending pairing requests, while this patch would also re-run first-request side effects.

## What This Changes

The PR changes shared pairing challenge handling so an existing pending request with a stored code receives another pairing reply instead of returning without a reply.

## Best Possible Solution

Approve or reject repeat pairing-code delivery as an explicit shared pairing policy, then implement it as a separate resend branch that preserves first-request semantics and updates tests/docs.

## Reproduction Assessment

Yes for the proposed behavior delta and review blocker, not as a current-main bug. Source inspection shows the branch falls through for `{ code, created: false }`, while current main returns before replying.

## Solution Assessment

No. If repeat delivery is approved, the maintainable fix is a separate resend path that keeps `created` and `onCreated` first-request side effects distinct from repeat replies.
