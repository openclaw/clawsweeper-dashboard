## Summary

Keep open: the requested repeat-code UX is understandable and the PR includes visible Telegram proof, but this one-line shared-helper change is not mergeable because it changes creation-side-effect semantics and leaves the current tests/docs contract inconsistent.

## What This Changes

The PR changes `issuePairingChallenge` to send a pairing reply whenever `upsertPairingRequest` returns a code, including existing pending requests.

## Best Possible Solution

If maintainers want this UX, add an explicit resend-existing-code path that preserves `created: false` and calls `onCreated` only for newly created requests, then update shared tests, channel expectations, and docs.

## Reproduction Assessment

Yes for the PR defects: source inspection shows an existing `{ code, created: false }` request falls through under the patch, sends another reply, calls creation-side effects, and returns `{ created: true, code }`. The requested repeat-reply behavior itself is a feature change from the current documented contract.

## Solution Assessment

No. The one-line guard change is not the best fix because it changes shared creation semantics and leaves tests, channel expectations, and docs inconsistent; an explicit resend path is safer if the product direction is approved.
