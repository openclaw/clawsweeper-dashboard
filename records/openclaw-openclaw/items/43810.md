## Summary

Keep open: this PR targets a plausible UX improvement, but the current one-line guard change is not mergeable because it treats repeat pairing replies like newly created requests and repeats pairing codes without an explicit resend policy.

## What This Changes

The PR changes shared pairing challenge issuance so an existing pending request with a stored code receives another pairing-code reply instead of returning early.

## Best Possible Solution

If repeat sends are accepted, add an explicit resend branch that keeps `created: false`, gates creation side effects to actual new requests, updates tests/docs, and defines a cooldown or abuse policy.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows `{ code, created: false }` would fall through to `onCreated`, send a reply, and return `created: true`. Current main itself matches the documented no-resend behavior.

## Solution Assessment

No. Replacing the guard alone changes shared creation semantics and security posture; the maintainable fix needs an explicit resend path plus policy and tests.
