## Summary

Keep open: the requested resend behavior is a plausible UX feature, but this patch is not mergeable because it changes shared pairing semantics, conflicts with current tests/docs, and needs an approved resend/security policy first.

## What This Changes

The PR changes the shared DM pairing helper so an existing pending pairing code is sent again on repeat unapproved messages.

## Best Possible Solution

Define a bounded resend policy that preserves creation-state semantics, then update the shared helper, channel expectations, tests, and pairing docs together.

## Reproduction Assessment

Yes for the PR-introduced behavior: source inspection shows an existing `{ code, created: false }` pending request would now resend, invoke `onCreated`, and return the created path. I did not run tests because this review is read-only.

## Solution Assessment

No. The one-line guard change is not the best solution because it changes shared callback/result semantics and unbounds repeated pairing-code replies without a documented policy.

## Review Findings
