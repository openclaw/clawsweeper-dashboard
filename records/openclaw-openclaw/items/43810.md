## Summary

Keep open, but do not merge as-is. The UX goal is understandable and the screenshot shows repeated Telegram replies, but current main documents and tests new-request-only pairing replies, and this one-line shared helper change breaks `created`/`onCreated` semantics across channels.

## What This Changes

This PR changes `issuePairingChallenge` to send a pairing reply whenever `upsertPairingRequest` returns a non-empty code, including existing pending requests.

## Best Possible Solution

If maintainers want repeat-code replies, implement them deliberately in shared pairing while preserving `created`/`onCreated`, then update regression tests and user docs to match the accepted all-channel policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: when `upsertPairingRequest` returns `{ code: "ABCD", created: false }`, the PR falls through to send a reply, call `onCreated`, and return `{ created: true, code }`; current main has a focused test for the opposite behavior.

## Solution Assessment

No. The one-line guard change is too broad because it conflates repeat replies with new creation; the safer path is to preserve creation semantics and update tests/docs only if maintainers accept the feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve the creation state on repeat replies:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns an existing code with `created: false`, this guard now falls through to `return { created: true, code }`. That makes repeat requests report a new challenge and breaks the current repeat-request contract; resend support should return the real creation state or use a broader result shape with updated coverage.
  - confidence: 0.96
- **[P2] Keep `onCreated` limited to first issuance:** `src/pairing/pairing-challenge.ts:31`
  - body: The new guard lets existing pending requests reach `params.onCreated`, so repeat `/start` messages now fire first-creation side effects in shared callers such as Telegram, Slack, and Signal. If repeat replies are accepted, keep the reply path separate and call `onCreated` only when `created` is true.
  - confidence: 0.93
