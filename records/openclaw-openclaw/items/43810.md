## Summary

Keep open: this PR proposes a real UX change, but current main, tests, docs, and the latest release define the opposite no-resend pairing policy, and the one-line patch changes shared creation semantics without maintainer approval.

## What This Changes

The branch changes `src/pairing/pairing-challenge.ts` so repeat unapproved pairing requests resend an existing pairing code whenever the store returns a code.

## Best Possible Solution

If maintainers approve repeat code delivery, add an explicit resend branch that preserves first-creation semantics, updates tests and security docs, and validates Telegram plus shared channel behavior.

## Reproduction Assessment

Yes for the proposed behavior delta: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, current main returns before replying while this PR sends the reply path. This is a policy change, not a current-main bug.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it also changes `onCreated` and return semantics; the maintainable path is a separate repeat-reply branch after maintainer approval.
