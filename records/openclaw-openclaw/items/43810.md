## Summary

Keep open: the request is a plausible UX feature, and the screenshot shows the intended repeated Telegram reply, but the one-line shared-helper change conflicts with the documented once-per-new-request behavior and breaks `created`/`onCreated` semantics across all DM pairing channels.

## What This Changes

The PR changes `issuePairingChallenge` so repeat unapproved `/start` messages resend the pairing reply whenever the existing upsert returns a code.

## Best Possible Solution

If maintainers want repeat pairing replies, implement it explicitly in the shared pairing helper while preserving `created` and `onCreated` as first-time-only signals, then update regression tests and pairing docs for the new cadence.

## Reproduction Assessment

Yes, source-level. Current main returns before sending for `{ created: false, code }`, while this PR falls through, sends the reply, fires `onCreated`, and returns a newly-created result for the same existing-request state.

## Solution Assessment

No. The intended UX can be useful, but the one-line guard change is not the best implementation because it broadens callback semantics, misreports the `created` result, and leaves tests/docs on the old contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Return the actual created state on repeats:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ created: false, code: "ABCD" }`, this guard now falls through and the function returns `{ created: true, code }`. That misreports an existing pending request as newly created; resend support should preserve the original `created` value in the result.
  - confidence: 0.92
- **[P2] Keep onCreated limited to new requests:** `src/pairing/pairing-challenge.ts:31`
  - body: For an existing request with a code, the new guard lets control reach `onCreated`, so repeat `/start` messages run first-time side effects in Telegram, Slack, Signal, and other callers. Guard `onCreated` with the original `created` flag if repeat replies are sent.
  - confidence: 0.91
- **[P2] Update the repeat-request contract tests:** `src/pairing/pairing-challenge.ts:31`
  - body: This changes the tested contract for existing pairing requests, but the current test still expects no reply and `{ created: false }`. The branch should update the test to the intended resend behavior after the created/onCreated semantics are fixed.
  - confidence: 0.88
