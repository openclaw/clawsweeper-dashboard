## Summary

Keep open: this is a plausible Telegram pairing UX feature, but the current one-line change is not mergeable because it changes shared creation semantics and repeatedly exposes the same pending pairing code without an approved resend boundary.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a stored code receives another pairing reply instead of returning silently.

## Best Possible Solution

If maintainers want pending-code resend, add an explicit bounded resend branch that preserves `created: false`, keeps `onCreated` creation-only, and updates shared tests, channel coverage, docs, and security wording together.

## Reproduction Assessment

Yes. Source repro is high confidence: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR guard falls through into reply sending, `onCreated`, and a created result; the Telegram screenshot also shows repeated `/start` replies.

## Solution Assessment

No. The one-line guard change is not the best way to solve this because it mixes creation and resend semantics; the safer path is an explicit resend policy with a throttle or cap.
