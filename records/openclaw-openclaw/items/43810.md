## Summary

Keep open: the PR targets a real UX improvement, but the current one-line patch changes shared DM pairing-code disclosure policy and breaks the existing creation semantics that callers and tests rely on.

## What This Changes

The PR changes `issuePairingChallenge` to resend a stored pending pairing code when an existing request returns a code, even when the request was not newly created.

## Best Possible Solution

If repeat code delivery is approved, implement it as an explicit resend branch that preserves first-request `created` and `onCreated` semantics, then update the shared tests and security docs to match the policy.

## Reproduction Assessment

Yes for the proposed behavior delta, not as a current-main bug. With `upsertPairingRequest` returning `{ code, created: false }`, current main returns without replying while this PR sends the reply path.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it also changes creation callbacks and return semantics; a separate approved resend branch is the narrower maintainable solution.
