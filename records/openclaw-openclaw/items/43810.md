## Summary

Keep open and do not merge as-is. The resend UX is understandable, but this one-line shared helper change breaks the current first-creation contract, would fire creation-only side effects on repeat pending requests, and would repeatedly disclose active pairing codes without the explicit cooldown/reminder policy already used by Matrix.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending request with a nonempty code sends another pairing reply instead of returning `{ created: false }`.

## Best Possible Solution

Model repeat delivery as an explicit pairing reminder path that preserves `onCreated` for first issuance, applies cooldown or suppression policy, and updates shared/channel tests, security docs, and changelog together.

## Reproduction Assessment

Yes. Mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`: current main returns `{ created: false }` without sending, while this PR falls through to callback/reply delivery and a created-style result.

## Solution Assessment

No. The one-line guard change conflates first-time creation with reminder delivery; the safer solution is a named reminder contract with preserved creation semantics and explicit abuse controls.
