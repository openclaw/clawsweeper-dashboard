## Summary

Keep open: the branch is not merge-ready because it changes the shared DM pairing helper from creation-only replies to repeat-code resend without preserving creation semantics, adding bounds, or updating the documented/tested cross-channel contract.

## What This Changes

This PR changes `issuePairingChallenge` so an existing pending request with a stored code falls through to send another pairing reply instead of returning on `created: false`.

## Best Possible Solution

If maintainers want this UX, add an explicit bounded resend path that preserves `created: false`, keeps `onCreated` creation-only, updates shared tests/docs/channel coverage, and uses redacted Telegram proof.

## Reproduction Assessment

Yes. Source reproduction is high confidence: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR falls through into reply sending, `onCreated`, and a `{ created: true, code }` result; the inspected Telegram screenshot also shows repeated `/start` replies.

## Solution Assessment

No. The one-line guard change conflates first creation with repeat resend; a maintainable solution needs an explicit resend branch and a maintainer decision about bounds and scope.
