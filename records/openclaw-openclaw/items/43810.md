## Summary

This PR should stay open for maintainer policy review: it changes the shared pairing-code resend behavior, and the current one-line implementation would also conflate repeat replies with first-time creation semantics.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a stored code sends another pairing reply instead of returning early.

## Best Possible Solution

If maintainers approve repeat code delivery, implement it as a shared pairing contract change that preserves first-creation semantics, updates tests/docs, and keeps Telegram proof attached to the policy change.

## Reproduction Assessment

Yes for the behavioral difference in source: have `upsertPairingRequest` return `{ code: "ABCD", created: false }`; current main returns early, while this PR sends the reply path. The broader question is policy, not a broken existing contract.

## Solution Assessment

No. The one-line guard reaches the desired resend behavior, but it also changes `onCreated` and return semantics; a maintainable fix would separate repeat reply delivery from first-time creation after maintainer approval.
