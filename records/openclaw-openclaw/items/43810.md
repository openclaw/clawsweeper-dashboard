## Summary

Keep open: the UX goal is plausible, but the one-line shared-helper change breaks the existing creation-only contract, broadens pairing-code exposure, and needs maintainer policy approval before a repair is safe.

## What This Changes

The branch changes `issuePairingChallenge` so any non-empty pairing code triggers a reply, including existing pending requests.

## Best Possible Solution

Define an explicit shared or Telegram-specific resend policy, preserve creation-only result and callback semantics, and align tests/docs with the approved contract.

## Reproduction Assessment

Yes for the PR branch behavior by source inspection: `{ code, created: false }` now falls through to reply handling. This is not a reproduced current-main bug because current `main` intentionally returns before replying for existing pending requests.

## Solution Assessment

No. The one-line guard demonstrates the requested UX, but the maintainable fix needs an approved bounded resend policy and must keep creation callbacks/results tied to actual new requests.
