## Summary

Keep open: the requested repeat-code behavior is a user-facing pairing policy change, and the current one-line patch is not merge-ready because it folds repeat replies into first-creation semantics used by shared callers and tests.

## What This Changes

The PR changes shared pairing challenge issuance so an existing pending request with a stored code can receive the pairing reply again instead of returning without a reply.

## Best Possible Solution

If maintainers approve repeat code delivery, implement it as an explicit resend branch that preserves first-request `created`/`onCreated` semantics and updates shared tests plus security docs.

## Reproduction Assessment

Yes for the proposed behavior delta, not as a current-main bug: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, current main returns before replying while this PR sends the reply path.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it also changes `onCreated` and return semantics; the maintainable path is a separate repeat-reply branch after maintainer approval.
