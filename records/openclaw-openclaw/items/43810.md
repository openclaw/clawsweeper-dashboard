## Summary

Keep this PR open for maintainer decision. The UX gap is real and the Telegram screenshot demonstrates the desired repeat reply, but the patch changes a documented shared DM pairing-code disclosure policy and broadens first-request side effects across all shared pairing callers.

## What This Changes

The PR changes `issuePairingChallenge` to send a pairing reply whenever `upsertPairingRequest` returns a code, including existing pending requests.

## Best Possible Solution

End state should be an explicitly approved resend policy in the shared pairing flow that handles repeat delivery as its own branch, preserves `created` and `onCreated` first-request semantics, and updates tests plus docs.

## Reproduction Assessment

Yes for the proposed behavior delta and review blocker, not as a current-main bug. Source inspection shows the PR falls through for `{ code, created: false }`, while current main returns before replying.

## Solution Assessment

No. If repeat delivery is approved, the maintainable fix is a separate resend branch that keeps first-request side effects and return semantics distinct from repeat replies.
