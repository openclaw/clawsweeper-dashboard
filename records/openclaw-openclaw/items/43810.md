## Summary

Keep this PR open for maintainer review, but do not merge the current patch: it turns a shared creation-only pairing helper into an unbounded resend path, changes callback/result semantics, and expands repeated pairing-code exposure without an approved policy.

## What This Changes

The PR changes `src/pairing/pairing-challenge.ts` to send a pairing reply whenever `upsertPairingRequest` returns a code instead of only when it creates a new request.

## Best Possible Solution

Define a bounded resend policy in the shared pairing contract, keep `created` and `onCreated` tied to first creation, then update helper tests, docs, and Telegram-visible expectations together.

## Reproduction Assessment

Yes for the proposed PR behavior, but not as a current-main bug. Source inspection shows an existing `{ code, created: false }` falls through on the PR branch, and the screenshot shows repeat Telegram `/start` replies; current main returns before sending on `created: false`.

## Solution Assessment

No. The one-line guard demonstrates the desired resend UX, but it changes shared creation semantics and lacks a bounded resend policy or channel boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation-only semantics:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns an existing pending request with `{ code, created: false }`, this guard now falls through, invokes `onCreated`, sends a reply, and returns `{ created: true, code }`. Any resend path should keep creation-only callbacks and the returned `created` flag tied to actual new requests so channel logging and one-time side effects do not run repeatedly.
  - confidence: 0.95
- **[P2] Bound repeated pairing-code replies:** `src/pairing/pairing-challenge.ts:31`
  - body: This sends the same pending pairing code on every repeat sender message until expiry, while current docs and tests limit pairing replies to newly created requests. Add an approved cooldown, cap, or channel-specific policy before moving this into the shared helper.
  - confidence: 0.9
