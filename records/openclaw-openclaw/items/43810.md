## Summary

Keep open for maintainer product/security review. Current `main` intentionally suppresses repeat pairing replies, while this PR changes shared pairing behavior across channels and would repeatedly deliver active pending codes without a documented resend policy.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing code is sent again instead of returning early when the sender already has a request.

## Best Possible Solution

If maintainers want repeat pairing reminders, add an explicit resend/reminder contract that preserves creation-only signals, defines cooldown or suppression behavior, and updates shared tests, docs, and Telegram proof together.

## Reproduction Assessment

Yes. From source, make `upsertPairingRequest` return `{ code: "ABCD", created: false }`: current `main` returns before sending, while this PR would send the reply and fire `onCreated`; I did not run tests because this review is read-only.

## Solution Assessment

No. The one-line guard change is not the best way to solve the issue because it changes shared creation semantics and repeat-code disclosure policy without defining a deliberate reminder path.
