## Summary

Keep open, but this branch is not mergeable as-is: it changes a creation-only pairing helper into a repeat-send path without preserving creation semantics or defining a resend policy, while current docs and tests still describe one pairing reply per new request.

## What This Changes

The PR changes the shared pairing challenge guard so pending users with an existing pairing code receive another pairing-code reply on repeated DM start attempts.

## Best Possible Solution

If repeat sends are accepted, implement an explicit resend path that preserves `created: false`, gates `onCreated` to true creations, updates tests and docs, and documents a resend throttle or cap.

## Reproduction Assessment

Yes for the proposed patch regression. Source inspection shows `{ code: "ABCD", created: false }` would fall through to `onCreated`, send a reply, and return `created: true`, while current main and the latest release keep the no-resend behavior.

## Solution Assessment

No. Replacing the guard alone is not the best fix because it changes shared creation semantics and pairing-code exposure policy without tests, docs, or a resend limit.
