## Summary

Keep open, but do not merge as-is: the screenshot proves the intended repeated Telegram reply, while the one-line shared helper change conflicts with the current tests/docs and reuses first-request `onCreated` semantics across DM channels.

## What This Changes

The PR changes the shared pairing challenge guard so repeat pending requests with a stored code send another pairing reply instead of returning without a response.

## Best Possible Solution

If maintainers want repeat-code delivery, add an explicit resend branch that preserves `{ created: false }` and only runs first-request side effects for newly created requests, then update shared channel tests/docs and Telegram proof.

## Reproduction Assessment

Yes for the PR behavior delta: source inspection shows `{ code, created: false }` falls through after the patch, and the attached screenshot shows repeated Telegram `/start` replies. Current main tests and docs establish the opposite existing contract.

## Solution Assessment

No. The one-line guard changes shared creation semantics; a safer solution would make resend behavior explicit and keep `onCreated` tied to new requests.
