## Summary

Keep open for maintainer judgment: the requested repeat-code resend is a real user-visible feature, but the current one-line guard change breaks creation semantics and removes the documented once-per-new-request boundary without an approved cooldown or cap.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a stored code receives another pairing reply on each repeat `/start` or DM before approval.

## Best Possible Solution

If maintainers want pending-code resend, add an explicit bounded resend path that preserves `created: false`, keeps `onCreated` creation-only, and updates shared tests, channel coverage, docs, and security wording together.

## Reproduction Assessment

Yes. Source repro is high confidence: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR falls through into reply sending, `onCreated`, and a created result; the inspected Telegram screenshot also shows repeated `/start` replies.

## Solution Assessment

No. The one-line guard change is not the best solution because it mixes creation and resend semantics; the safer path is an explicit resend branch with an approved throttle or cap.
