## Summary

Keep open: this is a plausible UX feature, but current main, tests, and docs intentionally use a one-reply-per-new-request DM pairing contract, and this diff incorrectly turns repeat pending requests into created events across shared channel call sites.

## What This Changes

The PR changes the shared pairing challenge helper so an existing pending pairing code is sent again on repeated `/start` or DM messages.

## Best Possible Solution

If maintainers want repeat pairing-code replies, define that as the cross-channel policy, resend existing codes while preserving `created` and `onCreated` semantics for new requests only, and update focused tests plus the pairing docs.

## Reproduction Assessment

Yes, source-reproducible for the current behavior and PR conflict. Current main returns before sending on `created: false`, and helper/channel tests plus docs encode one reply per new pending request; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The behavior may be useful, but a one-line guard change is not the narrow maintainable solution because it changes `onCreated` and return-value semantics and leaves the shared tests/docs contract unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep creation callbacks tied to new requests:** `src/pairing/pairing-challenge.ts:31`
  - body: With the new guard, an existing pending `{ code, created: false }` falls through to `params.onCreated?.({ code })` and returns `{ created: true, code }`. Callers use that creation signal for one-time pairing logs or side effects, so repeat `/start` or DM messages would be treated as fresh requests; resend the reply without changing the actual creation signal.
  - confidence: 0.94
- **[P2] Update the repeat-request contract and tests:** `src/pairing/pairing-challenge.ts:31`
  - body: The helper and channel tests still assert that an already-pending request does not send another pairing reply, and the public pairing docs say the bot sends only when a new request is created. If repeat replies are intended, update the shared contract, focused tests, channel tests, and docs in the same change so CI and user-visible behavior agree.
  - confidence: 0.9
