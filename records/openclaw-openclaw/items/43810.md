## Summary

Keep open: the PR targets a plausible Telegram pairing UX improvement, but the current patch changes shared pairing semantics without an approved resend policy and would repeat creation-only side effects across channels.

## What This Changes

The branch changes `issuePairingChallenge` to send a pairing reply whenever the upsert returns a code, including existing pending requests.

## Best Possible Solution

Define a bounded resend path, shared or Telegram-specific, that preserves creation-only `created` and `onCreated` semantics, updates tests/docs, and includes redacted Telegram proof.

## Reproduction Assessment

Yes for the PR branch behavior, but not as a current-main bug. Source inspection shows `{ code, created: false }` falls through on the PR branch, while current main returns before sending on `created: false`; the screenshot shows repeated Telegram replies.

## Solution Assessment

No. The one-line guard demonstrates the requested UX, but the maintainable solution needs an explicit resend policy and must keep creation callbacks/results tied to actual new requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation-only callback semantics:** `src/pairing/pairing-challenge.ts:31`
  - body: When an existing pending request returns `{ code, created: false }`, this guard now falls through, calls `onCreated`, sends a reply, and returns `created: true`. Keep `onCreated` and the returned `created` flag tied to actual new requests so channel logs and one-time side effects do not repeat on every `/start`.
  - confidence: 0.95
- **[P2] Add a resend policy before exposing pending codes repeatedly:** `src/pairing/pairing-challenge.ts:31`
  - body: This sends the same pending pairing code on every repeat sender message until expiry, while current docs and tests bound pairing replies to newly created requests. A resend path needs an approved cooldown, cap, or channel-specific policy before changing the shared helper.
  - confidence: 0.9
- **[P2] Update the repeat-request contract test:** `src/pairing/pairing-challenge.ts:31`
  - body: The existing test for `{ code: "ABCD", created: false }` expects no reply and `{ created: false }`; this guard makes that case send a reply and return `created: true`. If the product decision is to support resends, update the test and contract together so CI reflects the intended behavior.
  - confidence: 0.88
