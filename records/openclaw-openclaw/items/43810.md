## Summary

Keep this PR open, but it is not ready to merge: the one-line guard change breaks the established creation-only pairing contract and needs maintainer policy approval for repeat-code exposure before repair or landing.

## What This Changes

The PR changes `src/pairing/pairing-challenge.ts` so an existing pending pairing code is replied again instead of returning silent `created: false`.

## Best Possible Solution

If repeated reminders are accepted, add an explicit bounded resend branch that preserves `created: false`, keeps `onCreated` creation-only, updates docs/tests, and applies the policy across shared pairing paths.

## Reproduction Assessment

Yes for the proposed patch regression: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and `{ created: true, code }`. The existing shared test models that source path, and the screenshot shows the intended repeated Telegram reply behavior.

## Solution Assessment

No. The one-line guard replacement conflates creation with resend behavior; a bounded resend path with maintainer-approved semantics is safer.
