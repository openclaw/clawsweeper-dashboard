## Summary

Keep this PR open but not merge-ready: it proposes a useful Telegram pairing UX change, but the one-line shared-helper guard change breaks the current pairing contract by treating existing pending requests like newly created ones and by repeating creation side effects across shared DM pairing channels.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing code is replied to again whenever a code exists, instead of returning when the request was not newly created.

## Best Possible Solution

If repeat sends are accepted, implement an explicit resend branch that preserves `created: false`, gates creation side effects to new requests, updates tests/docs, and defines a cooldown or limit policy.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows `{ code, created: false }` falls through to reply and creation side effects. There is no current-main bug reproduction because current tests and docs intentionally suppress repeat shared DM pairing replies.

## Solution Assessment

No. A one-line guard replacement is not the best solution because it changes shared creation semantics and security behavior without a resend contract, cooldown policy, or matching tests and docs.

## Review Findings

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ code, created: false }`, this guard now falls through to `onCreated`, sends a reply, and returns `{ created: true, code }`. A resend should not look like a newly created request or repeat creation side effects; keep `created` false and gate `onCreated` to new requests.
  - confidence: 0.94

  - body: Current tests and docs say already-pending requests are not sent another shared pairing message. If repeat sends are intended, the shared helper needs updated tests, docs, channel expectations, and abuse limits rather than changing the guard alone.
  - confidence: 0.88
