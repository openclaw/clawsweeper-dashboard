## Summary

Keep open: this PR proposes a real UX change, but current main intentionally documents and tests one-time pairing-code delivery, and the one-line patch changes shared access-control semantics without preserving first-creation behavior.

## What This Changes

The PR changes `issuePairingChallenge` so existing pending pairing requests with a stored code receive another pairing reply instead of returning early.

## Best Possible Solution

End state should be an explicitly approved resend policy in the shared pairing flow, with a separate repeat-delivery branch that preserves `created`/`onCreated` first-request semantics and updates tests plus docs.

## Reproduction Assessment

Yes for the PR-introduced behavior change and review blocker: source inspection shows `{ code, created: false }` falls through under the patch. This is not a current-main bug because current main returns before replying.

## Solution Assessment

No. If repeat delivery is approved, the maintainable fix is a distinct resend path that keeps first-request side effects and return contracts separate.
