## Summary

Keep open for maintainer review: this is a real feature proposal, but the one-line guard change conflicts with the current shared no-resend contract and changes pairing-code exposure semantics across DM-capable channels.

## What This Changes

The PR changes shared pairing challenge handling so an existing pending pairing code is sent again on repeat `/start` messages.

## Best Possible Solution

If maintainers want repeat sends, add an explicit resend branch that preserves `created: false`, gates creation side effects to newly created requests, updates shared tests/docs, and defines resend throttling expectations.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows `{ code, created: false }` would fall through, send a reply, fire creation side effects, and return `created: true`. There is no current-main bug reproduction because current docs and tests define the no-resend behavior.

## Solution Assessment

No. Replacing the guard alone is not the best way to solve this because it changes shared creation semantics and documented security behavior without a resend contract, tests, docs, or cooldown policy.

A resend should not look like a newly created request or repeat creation side effects; split the resend path and keep `created` false.
