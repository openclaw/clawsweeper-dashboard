## Summary

Keep open for maintainer policy review, but the PR is not safe to merge as-is. Current main, tests, and public security docs define the no-repeat-resend behavior, while this one-line change alters shared DM access-control behavior and breaks first-creation semantics across channels.

## What This Changes

The PR changes `src/pairing/pairing-challenge.ts` so a stored pairing code is resent when an existing pending pairing request has a code.

## Best Possible Solution

If approved, implement repeat resend as an explicit shared pairing policy branch that preserves `created: false` and first-request `onCreated` semantics, with updated tests and security docs.

## Reproduction Assessment

Yes, source-reproducible for the proposed behavior delta: current main returns before sending when upsert returns `{ code: "ABCD", created: false }`, while this PR falls through. This is not a current-main bug because docs and tests currently specify no repeat resend.

## Solution Assessment

No. The one-line guard change is not the best fix because it broadens `onCreated` and the returned `created` value; an approved implementation should use a separate resend path.
