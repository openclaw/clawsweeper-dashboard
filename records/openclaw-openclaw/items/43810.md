## Summary

Keep open: this is a plausible UX feature, but current main intentionally sends a DM pairing reply only when a new request is created. The one-line shared-helper change would make repeat pending requests look newly created and needs maintainer product/security direction before merge.

## What This Changes

The PR changes the shared DM pairing helper so an existing pending pairing code is replied to again instead of returning when `created` is false.

## Best Possible Solution

If repeat sends are accepted, implement an explicit resend branch that preserves `created: false`, gates creation side effects to new requests, updates tests/docs for the shared contract, and defines resend throttling expectations.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows `{ code, created: false }` would fall through, call reply/creation side effects, and return `created: true`. There is no current-main bug reproduction because current tests and docs intentionally do not resend until a new request is created.

## Solution Assessment

No. Replacing the guard alone is not the best solution because it changes shared creation semantics and documented security behavior without a resend contract, tests, docs, or cooldown policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ code, created: false }`, this guard now falls through to `onCreated`, sends a reply, and returns `{ created: true, code }`. A resend should not look like a newly created request or repeat creation side effects; keep `created` false and gate `onCreated` to new requests.
  - confidence: 0.94
- **[P2] Update the shared resend contract:** `src/pairing/pairing-challenge.ts:31`
  - body: Current tests and docs say already-pending requests are not sent another pairing message. If repeat sends are intended, the shared helper needs updated tests, docs, channel expectations, and abuse limits rather than changing the guard alone.
  - confidence: 0.88
