## Summary

Keep open; the UX goal is plausible and the screenshot shows the intended Telegram behavior, but the patch is not mergeable because it changes shared pairing semantics, would break the existing repeat-request contract, and lacks an approved resend/cooldown policy.

## What This Changes

This PR changes `issuePairingChallenge` so repeat pairing attempts with an existing pending code send another pairing reply instead of returning early.

## Best Possible Solution

If repeat sends are approved, implement an explicit resend branch that preserves `created` semantics, gates `onCreated` to true creations, adds a cooldown or cap, and updates the shared tests and pairing docs.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection of the diff and current store contract shows `{ code, created: false }` would fall through, call `onCreated`, send a reply, and return `{ created: true, code }`. The user-visible desired behavior is also shown in the Telegram screenshot.

## Solution Assessment

No. The one-line guard change is not the best fix because current docs and tests define creation-only sends; a maintainable solution needs an explicit resend branch plus rate/exposure policy.
