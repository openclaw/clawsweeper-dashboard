## Summary

Keep open, but do not merge as-is: the requested repeat-resend UX is not implemented on main, and this one-line shared-helper change expands pairing-code exposure while changing creation-only callback semantics without an approved resend policy.

## What This Changes

The PR changes `src/pairing/pairing-challenge.ts` so a pairing reply is sent whenever the upsert returns a code, including existing pending requests.

## Best Possible Solution

Define a bounded resend policy in the shared pairing contract or a Telegram-specific path, keep `created` and `onCreated` creation-only, then update tests, docs, and Telegram-visible proof together.

## Reproduction Assessment

Yes for the PR branch behavior, but not as a current-main bug. Source inspection shows `{ code, created: false }` falls through on the PR branch, while current main returns before sending on `created: false`; the screenshot shows the intended repeated Telegram replies.

## Solution Assessment

No. The one-line guard demonstrates the desired UX, but the maintainable fix needs an explicit bounded resend policy and must preserve creation-only callback/result semantics.
