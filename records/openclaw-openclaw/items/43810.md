## Summary

Keep open: this is a real feature proposal with visible Telegram proof, but the one-line patch is not mergeable because it changes the documented pairing-code resend policy and conflates existing requests with new creations.

## What This Changes

The PR changes `issuePairingChallenge` to resend a pairing reply whenever a stored code exists instead of only when a new request is created.

## Best Possible Solution

If maintainers accept repeat code delivery, implement it as a documented pairing contract that resends without firing first-created callbacks or returning `created: true` for existing requests.

## Reproduction Assessment

Yes. Source inspection shows current main returns before `sendPairingReply` for `{ code, created: false }`, while the store returns that shape for an existing pending request; the screenshot shows the proposed after-fix Telegram behavior.

## Solution Assessment

No. The current patch gets repeat replies by changing the guard, but it also marks existing requests as created and leaves docs, tests, and callback semantics inconsistent; a safer implementation would preserve `created` semantics and explicitly update the pairing contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve created semantics when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ code, created: false }` for an existing request, this guard now falls through, calls `onCreated`, and returns `{ created: true, code }`. Callers use `onCreated` for first-time pairing request side effects, so a resend feature needs to send the reply without reporting a new request.
  - confidence: 0.95
