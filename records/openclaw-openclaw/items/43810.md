## Summary

Keep open, but this PR is not merge-ready: the screenshot proves the requested Telegram repeat reply, while the one-line shared-helper change conflicts with the documented no-resend pairing contract and breaks the `created` boundary used by callers.

## What This Changes

The PR changes `src/pairing/pairing-challenge.ts` so an existing pending pairing request with a stored code sends another pairing reply instead of returning silently.

## Best Possible Solution

Keep the current no-resend contract unless maintainers explicitly approve repeat-code disclosure; if approved, implement a repeat-send path that returns `created: false`, keeps `onCreated` first-request-only, and updates shared tests plus security docs.

## Reproduction Assessment

Yes for the PR behavior delta: source inspection shows `{ code, created: false }` falls through after the patch, and the inspected Telegram screenshot shows repeated `/start` replies. This is a feature request rather than a current-main bug reproduction.

## Solution Assessment

No. The one-line guard change is not the best solution because it changes shared access-control semantics; an approved implementation should separate repeat reply delivery from first-request creation and update the contract tests/docs.
