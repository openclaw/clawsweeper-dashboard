## Summary

Keep open, but this patch should not merge as-is: it changes shared DM pairing policy without preserving creation semantics or a resend bound, while current tests, docs, and shipped behavior define repeat pending requests as silent until a new request is created.

## What This Changes

The PR changes `issuePairingChallenge` to resend a pairing-code reply whenever `upsertPairingRequest` returns a code, including existing pending requests.

## Best Possible Solution

Add a maintainer-approved bounded resend path that preserves `created: false` on repeats, keeps `onCreated` creation-only, and updates shared tests, docs, and channel expectations.

## Reproduction Assessment

Yes for the proposed patch regression by source inspection: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the new `!code` guard falls through to reply sending, `onCreated`, and `{ created: true, code }`. I did not run the suite because this review is read-only.

## Solution Assessment

No. The one-line guard replacement conflates first creation with resend behavior and removes the documented resend boundary; the safer path is an explicit bounded resend branch after maintainer policy approval.

## Review Findings

  - confidence: 0.95
