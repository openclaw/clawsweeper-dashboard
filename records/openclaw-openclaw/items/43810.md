## Summary

Keep open, but this PR is not merge-ready. The UX goal is useful, but the one-line guard change violates the current docs/tests, changes shared creation-only callback/result semantics, and needs maintainer policy on repeated pairing-code exposure before a narrower fix can land.

## What This Changes

The PR changes `issuePairingChallenge` to resend a pairing reply whenever `upsertPairingRequest` returns a code, even for an existing pending request.

## Best Possible Solution

Define a bounded pending-code resend policy, preserve creation-only callback and result semantics, then update the shared helper, tests, docs, and Telegram-visible expectations together.

## Reproduction Assessment

Yes for the PR behavior, but not as a current-main bug. By source inspection, a stored `{ code, created: false }` falls through on the PR branch, and the screenshot shows repeated Telegram `/start` replies after the change; current main intentionally returns without resending.

## Solution Assessment

No. The guard change demonstrates the desired Telegram behavior, but it changes shared auth semantics and sends the same pending code on every repeat message without an approved resend policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns an existing pending request with `{ code, created: false }`, this guard now falls through, invokes `onCreated`, and returns `{ created: true, code }`. Any resend path should keep creation-only callbacks and the returned `created` flag tied to actual new requests so channel logging or one-time side effects do not run repeatedly.
  - confidence: 0.95
- **[P2] Bound repeated pairing-code replies:** `src/pairing/pairing-challenge.ts:31`
  - body: This sends the same pending pairing code on every repeat sender message until expiry, while current docs and tests limit pairing replies to newly created requests. Add an approved cooldown, cap, or channel-specific policy before moving this into the shared helper.
  - confidence: 0.9
