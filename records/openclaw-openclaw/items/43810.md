## Summary

Keep this PR open, but the current patch is not safe to merge: it turns a shared creation-only pairing helper into an unbounded resend path across channels and changes the `created`/`onCreated` semantics.

## What This Changes

The PR changes shared DM pairing challenge issuance to resend an existing pending pairing code whenever a code is present instead of only when a new request is created.

## Best Possible Solution

If repeated reminders are accepted, add an explicit bounded resend branch that preserves `created: false`, keeps `onCreated` creation-only, and updates focused tests plus pairing docs.

## Reproduction Assessment

Yes for the proposed patch regression: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and a `{ created: true, code }` return. The screenshot also demonstrates the intended repeated Telegram reply behavior after the change.

## Solution Assessment

No. A single guard replacement conflates request creation with resend behavior; the maintainable fix needs explicit resend semantics and a bounded exposure policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ code, created: false }`, this guard now falls through, calls `onCreated`, sends a reply, and returns `{ created: true, code }`. If resends are desired, split the existing-request path so it can send a reminder while keeping `created` false and `onCreated` creation-only.
  - confidence: 0.95

  - confidence: 0.9
