## Summary

Keep this PR open, but it is not ready to merge: the one-line guard change alters documented shared DM pairing policy and repeats first-request side effects across pairing callers.

## What This Changes

The PR changes shared DM pairing challenge handling so existing pending requests with a stored code resend the pairing reply on repeated `/start` messages.

## Best Possible Solution

Approve or reject repeat pending-code delivery as an explicit shared pairing policy; if approved, implement a separate resend branch that preserves `created` and `onCreated` semantics and update tests/docs.

## Reproduction Assessment

Yes for the PR behavior delta and review blocker, not as a current-main bug. Source inspection shows current main returns before replying for `{ code, created: false }`, while the branch falls through, and the PR screenshot shows repeated Telegram `/start` replies after the patch.

## Solution Assessment

No. The one-line guard conflates repeat delivery with first creation; the safer implementation is a separate resend path that keeps first-request side effects guarded by `created`.
