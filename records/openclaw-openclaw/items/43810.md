## Summary

Keep open. The PR addresses a plausible UX request, but current main deliberately sends DM pairing replies only for new requests, and this one-line shared-helper change breaks creation semantics, tests/docs, and broadens repeated pairing-code exposure without maintainer policy.

## What This Changes

This PR changes `issuePairingChallenge` so a pending pairing request with any stored code resends the pairing reply instead of returning early when `created` is false.

## Best Possible Solution

If maintainers want pending-code resend, implement an explicit bounded resend branch that preserves `created: false`, keeps `onCreated` creation-only, updates shared tests/docs/channel coverage, and uses redacted Telegram proof.

## Reproduction Assessment

Yes. Source reproduction is high confidence: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR falls through into reply sending, `onCreated`, and a `{ created: true, code }` result; the inspected Telegram screenshot also shows repeated `/start` replies.

## Solution Assessment

No. The one-line guard change conflates creation and resend semantics; a maintainable fix would make resend explicit, bounded, and separate from first-creation side effects.
