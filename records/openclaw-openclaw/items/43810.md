## Summary

Keep open for maintainer judgment: the PR addresses a real Telegram UX pain point and has visible proof, but it changes shared documented pairing-code disclosure behavior and the one-line implementation breaks the `created` contract across channels.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending request with a stored code falls through to send another pairing reply instead of returning early.

## Best Possible Solution

If maintainers approve repeat delivery, update the shared pairing contract to resend replies while returning `created: false`, guard `onCreated` to first creation only, and update tests plus security docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`; current main returns early while the PR falls through and sends a reply.

## Solution Assessment

No. The one-line guard change achieves repeat delivery, but it also returns `created: true` and fires `onCreated` for repeat requests; a contract-level change is the safer solution if the policy is accepted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve created semantics when resending codes:** `src/pairing/pairing-challenge.ts:31`
