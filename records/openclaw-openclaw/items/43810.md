## Summary

Keep open: the requested resend behavior is plausible, but this one-line shared-helper change breaks existing creation semantics and needs a maintainer-approved resend policy before it is mergeable.

## What This Changes

The PR changes the shared pairing challenge guard so an existing pending request with a code sends another pairing reply instead of returning when the request was not newly created.

## Best Possible Solution

Define a bounded pending-code resend policy, preserve creation-only callback and result semantics, then update the shared helper, tests, docs, and affected channel expectations together.

## Reproduction Assessment

Yes. By source inspection, a stored `{ code, created: false }` result now falls through on the PR branch; the screenshot also shows repeated Telegram `/start` messages receiving code replies after the patch.

## Solution Assessment

No. The guard change demonstrates the desired visible behavior, but it widens shared auth behavior without a resend bound and changes `onCreated`/return semantics that callers currently treat as creation-only.
