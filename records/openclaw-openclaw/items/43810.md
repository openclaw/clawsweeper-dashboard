## Summary

Keep open for maintainer decision. The requested resend UX is plausible and has Telegram screenshot proof, but the one-line helper change breaks the shared pairing contract and changes creation semantics across every channel using the helper.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a code no longer returns early when `created` is false.

## Best Possible Solution

If maintainers want this UX, implement an explicit resend-existing-code path that preserves `created` and `onCreated` semantics, then update shared tests and public docs.

## Reproduction Assessment

Yes for the PR defect: source inspection and PR CI show an existing `{ code, created: false }` request falls through under the patch, sends another reply, calls creation-side effects, and returns `{ created: true, code }`. The requested resend behavior itself is a feature change from the current documented contract.

## Solution Assessment

No. The one-line guard change is not the best fix because it changes creation semantics and leaves shared tests, channel expectations, and public docs inconsistent.
