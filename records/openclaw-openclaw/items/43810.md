## Summary

Keep this PR open, but it is not mergeable as-is: the one-line guard change makes repeat pairing requests look newly created and removes the documented resend boundary without a maintainer-approved cooldown or cap.

## What This Changes

The PR changes shared DM pairing challenge handling so existing pending pairing requests with a stored code resend the pairing-code reply on every repeated `/start` or DM before approval.

## Best Possible Solution

Add a bounded resend path for pending DM pairing codes only if maintainers approve it; keep `onCreated` creation-only, update shared/channel tests, and align docs/security wording.

## Reproduction Assessment

Yes. Source repro is high confidence: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR falls through into reply sending, `onCreated`, and `{ created: true, code }`; the inspected Telegram screenshot confirms the visible repeated-code behavior.

## Solution Assessment

No. The one-line guard change conflates first creation with resend behavior; the safer path is an explicit, bounded resend branch after maintainers decide the allowed repeat-code policy.
