## Summary

Keep open: this is a focused user-visible feature/policy change, but current main still documents and tests no-resend behavior for pending pairing requests, and the one-line guard change breaks first-creation semantics for shared channel callers.

## What This Changes

The branch changes `src/pairing/pairing-challenge.ts` so an existing pending pairing request with a stored code sends the pairing reply again instead of returning without a reply.

## Best Possible Solution

If maintainers approve repeat code delivery, implement it as an explicit resend branch that preserves `created`/`onCreated` first-request semantics and updates the shared tests plus security docs.

## Reproduction Assessment

Yes for the proposed behavior delta, not as a current-main bug: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, current main returns before replying while this PR sends the reply path.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it also changes `onCreated` and return semantics; the maintainable path is a separate repeat-reply branch after maintainer approval.
