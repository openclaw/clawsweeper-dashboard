## Summary

Keep open: this is a plausible UX feature, but the current one-line shared guard is not merge-ready because it changes creation semantics, contradicts current docs/tests, and needs a maintainer decision on pending-code resend policy.

## What This Changes

The branch changes `issuePairingChallenge` to send pairing replies whenever the pairing store returns a code, including existing pending requests.

## Best Possible Solution

Define a bounded resend path, shared or Telegram-specific, that preserves creation-only `created` and `onCreated` semantics, updates tests/docs, and uses redacted Telegram proof.

## Reproduction Assessment

Yes for the PR branch behavior, but not as a current-main bug. Source inspection shows `{ code, created: false }` falls through on the branch, while current main returns before sending on `created: false`; the screenshot shows repeated Telegram `/start` replies.

## Solution Assessment

No. The one-line guard demonstrates the requested UX, but the maintainable solution needs an explicit resend policy and must keep creation callbacks/results tied to actual new requests.
