## Summary

Keep open: the requested repeated pairing-code delivery is a real user-visible feature request, but the current one-line patch changes the shared pairing contract and has a correctness bug that should block merge until maintainers approve the policy change.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending request with a stored code can receive another pairing reply instead of returning early.

## Best Possible Solution

If maintainers approve repeat delivery, update the shared pairing contract to resend replies while preserving `created: false`, call `onCreated` only on first creation, and align tests plus security docs with the new policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: mock `upsertPairingRequest` to return `{ code, created: false }` and current main returns before sending a reply, while the PR screenshot shows repeated Telegram `/start` replies after the patch.

## Solution Assessment

No. The one-line guard change achieves repeat delivery, but it incorrectly reports creation state and fires first-created callbacks; a contract-level change with tests and docs is the safer solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve created semantics when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ code, created: false }` for an existing request, this falls through, calls `onCreated`, and returns `{ created: true, code }`. If repeat delivery is approved, send the reply while keeping `created` false and only firing first-create callbacks.
  - confidence: 0.95
