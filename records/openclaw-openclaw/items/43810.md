## Summary

Keep open for maintainer review: the UX request is coherent and has visible Telegram proof, but the current one-line patch is not merge-ready because it changes documented shared DM pairing policy and conflates repeat replies with first-request side effects.

## What This Changes

The PR changes the shared `issuePairingChallenge` guard so a returned pairing code, rather than the `created` flag, controls whether the pairing reply path runs.

## Best Possible Solution

End state should be an explicitly approved repeat-delivery policy in the shared pairing flow, with a separate resend branch that preserves `created` and `onCreated` semantics and updates tests plus docs.

## Reproduction Assessment

Yes for the proposed PR behavior and review blocker, not as a current-main bug. Source inspection shows `{ code, created: false }` falls through after the PR guard while current main returns before replying, and the screenshot shows repeated Telegram replies after the patch.

## Solution Assessment

No. The one-line guard is not the best solution because it treats repeat delivery as new creation; a maintainable fix would split resend behavior from first-request side effects and only proceed after policy approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation semantics when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When an existing pending request returns `{ code, created: false }`, this guard now falls through to the creation path, calls `onCreated`, sends a reply, and returns `{ created: true, code }`. If repeat delivery is approved, handle it in a separate branch so first-request side effects and tests still see `created: false`.
