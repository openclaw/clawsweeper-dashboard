## Summary

Keep open: current main intentionally keeps repeat pending DM pairing requests quiet, and this PR changes a shared helper in a way that breaks the existing repeat-request contract and creation-callback semantics.

## What This Changes

The PR changes the shared DM pairing challenge guard so repeat pending requests with an existing code receive another pairing-code reply.

## Best Possible Solution

Add a maintainer-approved bounded resend path that preserves `created: false` on repeats, keeps `onCreated` creation-only, and updates shared tests, docs, and channel expectations.

## Reproduction Assessment

Yes for the proposed patch regression: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the branch falls through to `sendPairingReply`, `onCreated`, and `{ created: true, code }`. Current tests and docs define the existing repeat-request behavior without running the suite.

## Solution Assessment

No. The one-line guard replacement conflates first creation with resend behavior; a safer solution needs an explicit bounded resend branch after maintainer policy approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When the store returns an existing `{ code, created: false }`, this guard now falls through, calls `onCreated`, sends a reply, and returns `{ created: true, code }`. Split any repeat-code branch so resends keep `created: false` and creation side effects remain one-time.
  - confidence: 0.95
- **[P2] Bound repeated pairing-code replies:** `src/pairing/pairing-challenge.ts:31`
  - body: Current main and the pairing docs limit pairing messages to newly created requests, roughly once per hour per sender. This change lets an unapproved sender trigger the same live code reply on every repeat DM across shared pairing channels, so the feature needs an explicit cooldown or cap before merge.
  - confidence: 0.9
