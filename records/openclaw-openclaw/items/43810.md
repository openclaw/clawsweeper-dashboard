## Summary

Keep this PR open, but the current patch is not mergeable as-is. The UX goal is plausible, yet the one-line shared pairing helper change conflicts with current docs/tests, changes creation-only callback/result semantics across channels, and broadens pairing-code exposure without an accepted resend policy.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending sender with a pairing code receives another pairing reply instead of returning as an already-existing request.

## Best Possible Solution

Define a bounded pending-code resend policy, preserve creation-only callback/result semantics, then update the shared helper, tests, docs, and affected channel expectations together.

## Reproduction Assessment

Yes. By source inspection, `{ code, created: false }` from the store bypasses the PR branch guard and reaches reply sending; the screenshot also shows repeated Telegram `/start` replies after the change.

## Solution Assessment

No. The patch demonstrates the desired visible behavior, but it is not the best solution because it changes shared auth semantics without a resend bound and reports repeat sends as newly created pairing requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns an existing pending request with `{ code, created: false }`, this guard now falls through, invokes `onCreated`, and returns `{ created: true, code }`. Any resend path should keep creation-only callbacks and results tied to actual new requests so channel side effects do not run repeatedly.
  - confidence: 0.95
- **[P2] Bound repeated pairing-code replies:** `src/pairing/pairing-challenge.ts:31`
  - body: This sends the same pending pairing code on every repeat sender message until expiry, while current docs and tests limit sends to new requests. Add an accepted cooldown, cap, or channel-specific policy before changing this shared path.
  - confidence: 0.9
