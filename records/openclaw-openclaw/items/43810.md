## Summary

Keep open, but this is not merge-ready: the one-line shared guard change makes existing pending pairing requests look newly created, repeats live code replies without an approved bound, and conflicts with current docs/tests.

## What This Changes

This PR changes shared DM pairing challenge issuance to send a pairing reply whenever the store returns a code instead of only when a new request is created.

## Best Possible Solution

Define a bounded resend policy for pending pairing codes, preserve creation-only callback/result semantics, then update the shared helper, docs, tests, and channel expectations together.

## Reproduction Assessment

Yes for the PR-introduced behavior: source inspection shows `{ code, created: false }` now falls through into reply sending, `onCreated`, and a created-looking result. The Telegram screenshot also demonstrates repeated `/start` messages receiving code replies after the patch.

## Solution Assessment

No. The one-line guard change is not the best solution because it changes shared creation semantics and lacks a maintainer-approved resend bound or channel policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns an existing pending request with `{ code, created: false }`, this guard now falls through, invokes `onCreated`, and returns `{ created: true, code }`. Any resend path should keep creation-only callbacks and results tied to actual new requests so channel side effects do not run repeatedly.
  - confidence: 0.95
- **[P2] Bound repeated pairing-code replies:** `src/pairing/pairing-challenge.ts:31`
  - body: This resends the same pending pairing code on every repeat sender message until expiry across the shared helper, while current docs and tests limit sends to new requests. Add an approved cooldown, cap, or channel-specific policy before changing this shared path.
  - confidence: 0.9
