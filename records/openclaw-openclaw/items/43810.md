## Summary

Keep open for maintainer review. Current main still documents and tests no repeat resend, while this PR changes shared pairing-code disclosure and currently breaks the first-creation contract used by callers.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending request with a stored code enters the pairing reply path instead of returning without a reply.

## Best Possible Solution

If maintainers approve repeat code delivery, implement it as an explicit resend branch that preserves first-request `created` and `onCreated` semantics, then update shared tests and docs to match the approved policy.

## Reproduction Assessment

Yes for the proposed behavior delta, not as a current-main bug. With `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, current main returns without replying while this PR sends the reply path.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it also changes `onCreated` and return semantics; a separate approved repeat-resend branch is the maintainable path.
