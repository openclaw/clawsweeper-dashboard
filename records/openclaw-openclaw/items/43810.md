## Summary

Keep open. The PR is a plausible UX feature, but it is not mergeable as-is because it changes documented pairing-code disclosure policy and conflates repeat requests with newly-created pairing requests across shared DM pairing code.

## What This Changes

The PR changes `issuePairingChallenge` to send a pairing reply whenever `upsertPairingRequest` returns a code, including existing pending requests.

## Best Possible Solution

If maintainers approve repeat code delivery, land it as an explicit shared pairing contract update that preserves `onCreated` semantics, updates tests and security docs, and keeps Telegram proof attached.

## Reproduction Assessment

Yes for the proposed behavior delta: have `upsertPairingRequest` return `{ code: "ABCD", created: false }`; current main returns early, while this PR would send the reply path. This is a policy change, not a current-main bug.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it also changes `onCreated` and return semantics; a maintainable fix should separate repeat replies from first-time creation after maintainer approval.

## Review Findings
