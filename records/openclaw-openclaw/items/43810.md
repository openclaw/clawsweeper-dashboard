## Summary

Keep open for maintainer judgment: the requested resend behavior is a user-visible feature that conflicts with the documented/current no-resend policy, and this branch changes the shared pairing helper in a way that breaks creation semantics across channel callers.

## What This Changes

The PR changes one guard in `issuePairingChallenge` so an existing pending pairing request with a reusable code sends another pairing reply instead of returning early.

## Best Possible Solution

Decide the resend policy first; if accepted, implement it by separating reply resend behavior from request-creation semantics, preserving `created`/`onCreated`, and updating regression tests plus docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: when `upsertPairingRequest` returns `{ code, created: false }`, the PR guard falls through to reply, `onCreated`, and a created-looking return value; I did not run tests because this review was read-only.

## Solution Assessment

No. Replacing the guard alone is not the best way to solve it because it conflates reusable-code resend behavior with new request creation; a maintainable fix would preserve creation semantics separately from any accepted resend policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve created semantics when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: Changing the guard to `!code` lets existing requests with `{ code, created: false }` fall through to `onCreated`, `sendPairingReply`, and the final `{ created: true, code }` return. That makes repeat `/start` messages look like new pairing requests to shared callers; keep `created` and `onCreated` tied to actual creation while separately gating any approved resend behavior.
  - confidence: 0.93
