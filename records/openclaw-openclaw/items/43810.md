## Summary

Keep open: the requested UX is understandable and the screenshot shows it working in Telegram, but current main deliberately documents and tests one-time DM pairing-code delivery, and this shared-helper change would alter creation semantics across channels without maintainer policy approval.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending request with a stored code sends a pairing reply instead of returning early.

## Best Possible Solution

Keep the current one-time contract unless maintainers approve repeat-code disclosure; if approved, add an explicit repeat-reply path that preserves `created: false`, keeps `onCreated` first-request-only, and updates shared tests plus docs.

## Reproduction Assessment

Yes for the PR behavior delta: source inspection shows `{ code, created: false }` would fall through after the patch, and the PR screenshot shows repeated Telegram `/start` replies. This is not a current-main bug reproduction because current docs and tests define the no-resend behavior.

## Solution Assessment

No as proposed: the one-line guard couples repeat replies to first-request semantics. A maintainable solution needs an approved policy plus a repeat-send branch that preserves `created` and `onCreated` semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve creation semantics when resending pending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: With this guard, an existing pending request that returns `{ code, created: false }` falls through to `onCreated` and returns `{ created: true, code }`.   - confidence: 0.95
