## Summary

Keep open: this is a plausible UX feature with real Telegram proof, but it changes documented pairing-code resend policy and shared creation semantics, so it needs maintainer approval and a safer implementation before merge.

## What This Changes

The PR changes shared pairing challenge issuance to resend an existing pending pairing code when an unapproved sender sends `/start` again.

## Best Possible Solution

If maintainers approve repeat code delivery, implement it as an explicit resend branch that preserves first-request `created` and `onCreated` semantics, then update shared tests and security docs.

## Reproduction Assessment

Yes for the proposed behavior delta, not as a current-main bug: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, current main returns before replying while this PR sends the reply path.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it also changes `onCreated` and return semantics; the maintainable path is a separate repeat-reply branch after maintainer approval.
