## Summary

Keep open: the requested repeated-code UX is plausible and the screenshot shows it, but the one-line shared helper change still misreports repeat requests as newly created, re-fires creation side effects, and removes the documented once-per-new-request throttle without a bounded policy.

## What This Changes

The PR changes shared pairing challenge issuance so repeat pending pairing requests with an existing code receive another pairing-code reply.

## Best Possible Solution

Add a maintainer-approved bounded resend path that preserves `created: false` for repeats, keeps `onCreated` creation-only, and updates shared tests, docs, and affected channel expectations.

## Reproduction Assessment

Yes for the proposed patch regression: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and `{ created: true, code }`. The existing shared test models that source path, and the PR screenshot shows the requested repeated Telegram reply behavior.

## Solution Assessment

No. The one-line guard replacement conflates first creation with resend behavior; a safer solution needs an explicit bounded resend branch after maintainer policy approval.
