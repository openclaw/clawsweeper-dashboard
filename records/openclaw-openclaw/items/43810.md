## Summary

Keep open, but do not merge as-is: the PR proposes a plausible UX feature, but the one-line guard change alters the shared pairing helper’s creation semantics, contradicts current tests/docs, and needs maintainer policy for repeat pairing-code sends and cooldown behavior.

## What This Changes

The PR changes `issuePairingChallenge` to resend an existing pending pairing code when a user repeats `/start` instead of returning early.

## Best Possible Solution

If repeat sends are accepted, implement an explicit resend path that preserves `created: false`, calls `onCreated` only for true creations, adds focused tests/docs, and defines a resend cooldown or cap.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows `{ code: "ABCD", created: false }` would fall through to `onCreated`, send a reply, and return `created: true`. The user-visible resend behavior is also shown in the PR screenshot.

## Solution Assessment

No. Replacing the guard alone is not the best fix because it changes shared creation semantics and pairing-code exposure policy without tests, docs, or a resend limit.
