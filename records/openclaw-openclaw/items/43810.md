## Summary

Keep this PR open, but not merge-ready: the UX goal is understandable, and the attached Telegram screenshot shows the intended after-fix behavior, but the one-line shared helper change breaks the existing creation-only contract and needs a maintainer-approved resend policy before it can safely land.

## What This Changes

The PR changes `issuePairingChallenge` so a stored pending pairing code is sent again instead of returning when `created` is false.

## Best Possible Solution

Define an explicit bounded resend policy that preserves `created`/`onCreated` semantics, then update the shared helper tests, affected channel expectations, and pairing docs to match it.

## Reproduction Assessment

Yes for the PR-introduced behavior: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and a `{ created: true, code }` result. The attached Telegram screenshot also shows repeated `/start` replies after the patch.

## Solution Assessment

No. The one-line guard change is not the best fix because it changes shared callback/result semantics and lacks an explicit resend bound or channel-specific policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When the store returns an existing `{ code, created: false }`, this guard now sends a reply, invokes `onCreated`, and returns `{ created: true, code }`. A resend path should keep `onCreated` and the returned `created` state tied to actual new requests.
  - confidence: 0.95
- **[P2] Bound repeated pairing-code replies:** `src/pairing/pairing-challenge.ts:31`
  - body: This resends the same pending pairing code on every repeat sender message until expiry across the shared helper, while current docs and tests limit sends to new requests. Add a maintainer-approved cooldown, cap, or channel-specific policy before changing this shared path.
  - confidence: 0.9
