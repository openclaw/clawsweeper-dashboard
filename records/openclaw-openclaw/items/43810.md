## Summary

Keep open: the PR addresses a plausible Telegram UX request and has visual proof, but it changes documented shared pairing-code disclosure policy and is not correct as written because repeat replies are conflated with first-time creation.

## What This Changes

The PR changes `issuePairingChallenge` to resend a pairing reply whenever the pairing store returns a code instead of only when a request was newly created.

## Best Possible Solution

If maintainers approve repeat code delivery, implement it as an explicit shared pairing contract that preserves first-creation semantics, updates tests and security docs, and keeps the Telegram proof attached.

## Reproduction Assessment

Yes for the proposed behavior delta: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, current main returns before replying while this PR would send the reply path. This is a policy change, not a current-main bug.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it also changes `onCreated` and return semantics; the maintainable path is a separate repeat-reply branch after maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve creation semantics when resending codes:** `src/pairing/pairing-challenge.ts:31`
