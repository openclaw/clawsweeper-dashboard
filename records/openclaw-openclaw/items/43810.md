## Summary

Keep open: the PR targets a plausible Telegram-visible UX improvement, but current main intentionally documents and tests no repeat pairing-code resend, and the one-line guard changes shared creation/side-effect semantics across DM pairing paths.

## What This Changes

The PR changes `issuePairingChallenge` so pending pairing requests with an existing code send another pairing reply instead of returning without a reply.

## Best Possible Solution

If maintainers approve repeat code delivery, implement it as an explicit resend branch that preserves `created`/`onCreated` first-request semantics and updates the shared tests plus security docs.

## Reproduction Assessment

Yes for the proposed behavior delta and review finding: an upsert result `{ code: "ABCD", created: false }` falls through on the PR branch, while current main returns before replying. It is not a current-main bug because docs and tests currently define no-repeat resend behavior.

## Solution Assessment

No. The one-line guard reaches repeat delivery, but it conflates reply-sent with request-created semantics; the maintainable path is a separate repeat-reply branch after policy approval.
