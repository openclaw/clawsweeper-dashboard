## Summary

Keep this PR open, but it is not merge-ready: the user problem is plausible, yet the patch changes shared pairing semantics, repeatedly exposes the same pending code without an approved bound, and trips existing creation-only expectations.

## What This Changes

The PR changes shared DM pairing challenge issuance to send a pairing reply whenever the store returns a code, including existing pending requests, instead of only when a request is newly created.

## Best Possible Solution

Define a maintainer-approved bounded resend policy that preserves creation-only callback/result semantics, then update the shared helper, affected channel expectations, docs, and tests together.

## Reproduction Assessment

Yes for the PR-introduced behavior: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and a `{ created: true, code }` result. The PR screenshot also shows repeated Telegram `/start` replies after the patch.

## Solution Assessment

No. The one-line guard change is not the best way to solve this because it changes shared callback/result semantics and lacks an explicit resend bound or channel-specific policy.

## Review Findings
