## Summary

Keep open: the UX request is plausible, but this one-line shared pairing change is not merge-ready because it changes the documented once-per-new-request behavior, reports repeat sends as newly created challenges, and needs maintainer direction on resend limits before it can safely land.

## What This Changes

The PR changes shared DM pairing issuance so repeat `/start` messages with an existing pending code receive another pairing-code reply.

## Best Possible Solution

If maintainers want repeat sends, add an explicit resend branch that preserves `created: false`, gates creation side effects to new requests, updates tests/docs for the shared contract, and defines resend throttling expectations.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows `{ code, created: false }` would fall through, call reply and creation side effects, and return `created: true`. There is no current-main bug reproduction because current tests and docs define the no-resend behavior.

## Solution Assessment

No. Replacing the guard alone is not the best way to solve this because it changes shared creation semantics and documented security behavior without a resend contract, tests, docs, or cooldown policy.

A resend should not look like a newly created request or repeat creation side effects; keep `created` false and gate `onCreated` to new requests.
