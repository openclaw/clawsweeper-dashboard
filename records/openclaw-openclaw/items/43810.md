## Summary

Keep open. The PR has useful Telegram screenshot proof for the proposed UX, but current main, tests, and public security docs still define one-time DM pairing replies; the patch also changes shared pairing-code disclosure and misreports existing requests as newly created.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a stored code sends the pairing reply again instead of returning early.

## Best Possible Solution

If maintainers approve repeat delivery, update the shared pairing contract to resend replies while preserving `created: false`, firing `onCreated` only on first creation, and aligning docs/tests across channels.

## Reproduction Assessment

Yes. Source inspection shows current main returns before sending a reply for `{ code, created: false }`, and the store returns that shape for existing pending requests; the screenshot shows the proposed after-fix Telegram behavior.

## Solution Assessment

No. The one-line guard change resends the code, but it also fires first-created callbacks and returns `created: true` for existing requests; the safer path is an explicit contract change with docs/tests and a callback guard.
