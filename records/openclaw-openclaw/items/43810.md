## Summary

Keep open for maintainer review: the requested repeat-code behavior is a plausible UX feature, but the one-line patch changes shared pairing-code disclosure contrary to current docs/tests and repeats first-request side effects through `onCreated`.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending request with a stored pairing code falls through to send another pairing reply instead of returning without a response.

## Best Possible Solution

If maintainers approve repeat-code delivery, add an explicit resend branch that preserves first-created semantics, updates shared channel tests/docs, and validates the affected DM channels.

## Reproduction Assessment

Yes for the PR behavior delta and blocker: source inspection shows `{ code, created: false }` falls through after the patch, while current tests/docs assert no resend. The inspected screenshot also shows repeated Telegram `/start` replies after the patch.

## Solution Assessment

No. The one-line guard conflates repeat-code delivery with first-request creation; a safer fix would make resend behavior explicit and keep `onCreated` tied to new requests.
