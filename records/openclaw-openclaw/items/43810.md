## Summary

Keep open for maintainer/product review: the UX may be useful and the screenshot shows the intended Telegram behavior, but the branch is not mergeable because it changes a documented shared DM pairing contract and creation-side-effect semantics without updating tests, docs, or the broader channel contract.

## What This Changes

The PR changes the shared DM pairing challenge helper to resend an existing pending pairing code instead of returning early when the request already exists.

## Best Possible Solution

If maintainers approve the UX, add an explicit resend-existing-code path that preserves `created` and `onCreated` semantics, then update the shared tests, affected channel expectations, and public pairing docs together.

## Reproduction Assessment

Yes for the PR defect: source inspection shows an existing `{ code, created: false }` request falls through under the patch, sends another reply, calls creation-side effects, and returns `{ created: true, code }`. The requested resend behavior is a feature change from the current documented contract, not a broken existing behavior reproduction.

## Solution Assessment

No. The one-line guard change is not the best fix because it changes shared creation semantics and leaves tests, channel expectations, and docs inconsistent; an explicit resend branch is safer if maintainers approve the product direction.
