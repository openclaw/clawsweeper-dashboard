## Summary

Keep open: the requested repeat pairing-code delivery is a plausible feature, but the patch changes shared pairing security behavior and breaks the existing created-only contract.

## What This Changes

The PR changes `issuePairingChallenge` to resend a pairing reply when an existing pending request has a stored code.

## Best Possible Solution

If repeat delivery is approved, update the shared pairing contract to resend replies while preserving `created: false`, firing `onCreated` only for first creation, and updating docs/tests across affected channels.

## Reproduction Assessment

Yes. Source inspection shows current main returns before sending a reply for `{ code, created: false }`, and the PR screenshot shows the proposed repeated Telegram `/start` behavior after the patch.

## Solution Assessment

No. The one-line guard change resends the code, but it also returns `created: true` and fires first-created callbacks for existing requests; an explicit shared-contract change is safer.

## Review Findings

Overall correctness: patch is incorrect
