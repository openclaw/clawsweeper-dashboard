## Summary

Keep open: the UX goal is understandable, but the one-line patch changes the shared DM pairing contract across channels, conflicts with current tests/docs, and needs an explicit pairing/security policy decision before merge.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending unapproved sender with a stored pairing code receives the pairing reply again instead of returning early.

## Best Possible Solution

Keep the current one-time contract unless maintainers approve repeat-code disclosure; if approved, add an explicit repeat-reply path that preserves `created: false`, keeps `onCreated` first-request-only, and updates shared tests plus docs.

## Reproduction Assessment

Yes for the PR behavior delta: source inspection shows `{ code, created: false }` would fall through after the patch, and the inspected screenshot shows repeated Telegram `/start` replies. This is not a current-main bug reproduction because current tests/docs define the no-resend behavior.

## Solution Assessment

No as proposed: the one-line guard couples repeat replies to first-request semantics. A maintainable solution needs an approved policy plus a repeat-send branch that preserves `created` and `onCreated` semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve creation semantics when resending pending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: With this guard, an existing pending request that returns `{ code, created: false }` falls through to `onCreated` and returns `{ created: true, code }`. Current callers and tests use `created` as the first-request boundary, so repeat `/start` messages would duplicate pairing-request side effects and make downstream code treat an old request as newly created; if repeat replies are approved, keep the return value false and guard `onCreated` on the original `created` flag.
  - confidence: 0.95
