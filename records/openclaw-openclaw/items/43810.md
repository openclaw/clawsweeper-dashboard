## Summary

Keep open: this is a real repeat-code UX request, but current main still defines one pairing reply per newly created request and the PR changes shared pairing behavior without preserving creation semantics.

## What This Changes

The PR changes `issuePairingChallenge` so existing pending requests with a stored code receive another pairing reply instead of returning without replying.

## Best Possible Solution

If maintainers accept repeat-code replies, implement them deliberately in shared pairing while keeping `created` and `onCreated` tied to first issuance, then update tests and docs for the accepted policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR falls through to send a reply, call `onCreated`, and return a created result where current main returns `{ created: false }`.

## Solution Assessment

No. The one-line guard change is not the best fix because it conflates repeat replies with first-creation semantics; a safer implementation would separate repeat sending from creation state and update coverage/docs only after policy approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve the creation state on repeat replies:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns an existing request with `{ code, created: false }`, this guard now falls through to `return { created: true, code }`. That makes repeat requests report a new challenge and breaks the current repeat-request contract; resend support should preserve the real creation state or use an updated result shape with tests.
  - confidence: 0.96
- **[P2] Keep onCreated first-issuance only:** `src/pairing/pairing-challenge.ts:31`
  - body: The changed guard lets existing pending requests reach `params.onCreated`, so repeat `/start` messages now fire first-creation side effects in shared callers such as Telegram, Slack, and Signal. If repeat replies are accepted, keep the reply path separate and call `onCreated` only when `created` is true.
  - confidence: 0.94
