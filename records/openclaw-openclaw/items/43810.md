## Summary

Keep open for maintainer review: the UX request is plausible and the PR includes real Telegram proof, but the one-line shared helper change conflicts with current docs/tests and changes pairing-code disclosure plus first-created side effects across all DM channels.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a stored code resends the pairing reply instead of returning without a response.

## Best Possible Solution

If maintainers approve repeat-code delivery, implement an explicit resend branch that preserves first-created semantics, update docs/tests, and validate the shared DM channels.

## Reproduction Assessment

Yes for the PR behavior delta and blocker: source inspection shows current main returns without replying for `{ code, created: false }`, while the branch falls through. The PR screenshot also shows after-patch repeated Telegram `/start` replies.

## Solution Assessment

No. The one-line guard conflates repeat-code delivery with first-request creation; a maintainable fix would make resend behavior explicit and keep `onCreated` tied to new requests.
