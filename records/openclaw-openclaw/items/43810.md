## Summary

This PR should remain open for maintainer judgment, but the current patch is not merge-ready because the one-line guard change turns repeat sends into newly-created results and fires first-created side effects across the shared pairing path.

## What This Changes

The PR changes `issuePairingChallenge` so an existing pending pairing request with a stored code can receive another pairing reply instead of returning early.

## Best Possible Solution

If maintainers approve repeat delivery, update the shared pairing contract to resend replies while preserving `created: false`, call `onCreated` only for first creation, and align tests plus security docs with the new policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`; current main returns early while the PR falls through and sends a reply.

## Solution Assessment

No. The one-line guard change achieves repeat delivery, but it incorrectly reports creation state and fires first-created callbacks; a contract-level change with tests and docs is the safer solution.
