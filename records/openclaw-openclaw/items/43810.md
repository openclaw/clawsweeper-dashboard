## Summary

Keep open, but this PR is not merge-ready. The screenshot proves the requested Telegram repeat reply, but the one-line shared helper change conflicts with the documented one-time pairing contract and makes repeat pending requests look newly created across all pairing channels.

## What This Changes

The PR changes the shared pairing challenge guard so existing pending requests with a stored code send another pairing reply instead of returning without a response.

## Best Possible Solution

Keep the current no-resend contract unless maintainers explicitly approve repeat-code disclosure; if approved, implement it as an explicit repeat-send path that preserves `created: false`, guards first-request side effects, and updates tests/docs.

## Reproduction Assessment

Yes for the PR behavior delta: source inspection shows `{ code, created: false }` falls through after the patch, and the inspected Telegram screenshot shows repeated `/start` replies. This is a feature request rather than a current-main bug reproduction.

## Solution Assessment

No. The single guard change is not the best solution because it changes shared access-control semantics; an approved implementation should separate repeat reply delivery from first-request creation and update the contract tests/docs.
