## Summary

Keep open: the PR targets a real Telegram-visible UX gap, but the current one-line change is not merge-ready because it changes shared pairing creation semantics and removes the documented no-repeat resend behavior without a policy limit.

## What This Changes

The PR changes `issuePairingChallenge` so pending pairing requests with an existing code resend a pairing reply instead of returning without sending.

## Best Possible Solution

If repeat resends are accepted, implement an explicit resend/reminder branch that preserves `created: false`, keeps `onCreated` creation-only, adds a cooldown or cap, and updates focused tests plus pairing docs.

## Reproduction Assessment

Yes for the proposed patch regression: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending and creation callbacks. The screenshot also shows the intended repeated Telegram reply behavior after the change.

## Solution Assessment

No. A single guard replacement is not the best fix because it conflates creation with resend behavior and skips the documented resend policy boundary; a dedicated resend path with throttling is safer.
