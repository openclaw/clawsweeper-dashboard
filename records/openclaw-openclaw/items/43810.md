## Summary

This is not a cleanup-close candidate: the PR proposes a real user-visible resend feature, but the current patch breaks the shared pairing creation contract and needs maintainer policy on repeated live-code exposure before merge.

## What This Changes

The PR changes shared DM pairing challenge handling so existing pending requests with a stored code resend the pairing-code reply on every repeated `/start` or DM before approval.

## Best Possible Solution

Add a maintainer-approved bounded resend path that preserves creation semantics, keeps `onCreated` creation-only, updates shared tests and docs, and defines an exposure limit for repeated pending-code replies.

## Reproduction Assessment

Yes for the proposed patch regression by source inspection: with `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the changed guard falls through into reply sending, `onCreated`, and `{ created: true, code }`. The visible Telegram screenshot also demonstrates the intended repeated-code behavior after the patch.

## Solution Assessment

No. The proposed one-line guard change conflates first creation with resend behavior; the safer solution is an explicit, bounded resend branch after maintainers decide the allowed resend policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When the store returns an existing `{ code, created: false }`, this guard now falls through, calls `onCreated`, sends a reply, and returns `{ created: true, code }`. Split any repeat-code branch so resends keep `created: false` and creation side effects remain one-time.
  - confidence: 0.95

  - body: Current main and the pairing docs limit pairing messages to newly created requests, roughly once per hour per sender. This change lets an unapproved sender trigger the same live code reply on every repeat DM across shared pairing channels, so the feature needs an explicit cooldown or cap before merge.
  - confidence: 0.9
