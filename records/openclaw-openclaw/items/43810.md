## Summary

Keep this PR open for maintainer judgment, but the patch is not safe to merge as-is because it changes a documented shared pairing contract and makes repeat requests look like newly created requests to all shared callers.

## What This Changes

This PR changes `issuePairingChallenge` so an existing pending request with a stored code sends another pairing reply instead of returning early.

## Best Possible Solution

If repeat sends are accepted, add an explicit resend path that preserves `created: false`, fires `onCreated` only for newly created requests, updates shared/channel tests and docs, and considers abuse controls.

## Reproduction Assessment

Yes for the PR behavior and regression path. The screenshot shows repeated Telegram `/start` replies, and source inspection shows `{ code, created: false }` would now fall through to reply, `onCreated`, and a created-looking return value; I did not run tests because this review is read-only.

## Solution Assessment

No. Replacing the guard alone is not the best way to solve it because it changes the shared creation contract; a separate resend path should keep creation side effects tied to `created`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ code, created: false }` for an existing pending request, this guard now falls through to `onCreated`, `sendPairingReply`, and returns `{ created: true, code }`. That makes a resend look like a new request to every shared caller; preserve `created: false` and gate creation side effects on `created` even if a resend path is added.
  - confidence: 0.94
- **[P2] Update the resend contract with the behavior change:** `src/pairing/pairing-challenge.ts:31`
  - body: The helper is a shared contract, and current shared, Slack, Signal, Discord tests plus pairing docs say pending requests do not resend until a new request is created. If repeat replies are intended, update the affected contract, tests, and docs with this behavior instead of changing only the shared guard.
  - confidence: 0.88
