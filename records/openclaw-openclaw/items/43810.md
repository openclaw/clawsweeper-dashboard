## Summary

Keep open, but not merge-ready: the UX request is plausible and the screenshot demonstrates the Telegram behavior, but the one-line helper change contradicts current docs/tests, makes resends report as newly created, and needs a shared resend policy.

## What This Changes

The PR changes `issuePairingChallenge` so any stored pairing code is sent again instead of returning early when the pairing request already exists.

## Best Possible Solution

If repeat sends are accepted, add an explicit shared DM pairing resend path that preserves `created: false`, gates `onCreated` to new requests, updates docs/tests, and defines resend throttling expectations.

## Reproduction Assessment

Yes for the proposed patch regression: source inspection shows an existing pending `{ code, created: false }` falls through after the guard change. Current `main` intentionally returns without replying, so there is no current-main bug reproduction for the requested new behavior.

## Solution Assessment

No. Replacing the guard alone is not the best solution because it changes shared creation semantics; an explicit resend branch with tests, docs, and policy is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When `upsertPairingRequest` returns `{ code, created: false }`, this guard now falls through to `onCreated`, sends a reply, and returns `{ created: true, code }`. A resend should not look like a newly created request or repeat creation side effects; keep `created` false and gate `onCreated` to new requests.
  - confidence: 0.94

  - body: Current docs and tests say already-pending requests are not sent another pairing message. If repeat sends are intended, this shared helper needs updated tests/docs and channel expectations rather than changing the guard alone.
