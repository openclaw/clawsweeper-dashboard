## Summary

Keep open, but not mergeable as-is: the requested pairing-code resend is a feature policy change, and the patch bypasses current docs/tests plus creation-only callback semantics without a bounded resend policy.

## What This Changes

The PR changes shared DM pairing challenge issuance so any non-empty pairing code, rather than only a newly created request, causes a pairing reply to be sent.

## Best Possible Solution

Define an explicit shared or channel-specific pending-code reminder policy with cooldown/cap, keep `created` and `onCreated` tied to true new requests, and align tests/docs plus Telegram proof with that contract.

## Reproduction Assessment

Yes for the PR branch behavior by diff/source inspection and the attached Telegram screenshot. This is not a current-main bug reproduction because current main intentionally returns on `!created` and docs say repeat DMs do not resend until a new request is created.

## Solution Assessment

No. The one-line guard proves the requested UX, but the maintainable solution needs an approved bounded reminder policy and must preserve creation-only callback/result semantics.
