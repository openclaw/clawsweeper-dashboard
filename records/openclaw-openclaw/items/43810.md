## Summary

Keep open. Current main still intentionally sends DM pairing codes only when a new shared pairing request is created, while this PR changes that contract without preserving creation-only callbacks, updating the shared/channel tests, or defining a cross-channel resend/cooldown policy for repeated disclosure of pending codes.

## What This Changes

The PR changes `src/pairing/pairing-challenge.ts` so an existing pending pairing request with a nonempty code sends another pairing reply instead of returning early when `created` is false.

## Best Possible Solution

If repeat pairing reminders are desired, add an explicit reminder path that keeps `created` and `onCreated` creation-only, sends existing codes through named reminder behavior with cooldown or suppression, and updates the shared helper contract, affected channel tests, docs, and changelog together.

## Reproduction Assessment

Yes. Mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`, or send two unauthorized DMs where the second upsert returns `created: false`; current main sends once, while this PR falls through and sends again.

## Solution Assessment

No. The one-line guard change is not the best way to solve the UX issue because it conflates request creation with reminder delivery; the safer alternative is an explicit reminder contract with preserved creation-only callbacks and cooldown policy.
