## Summary

Keep this PR open, but it is not merge-ready: the requested resend UX is still absent from current main, while the one-line patch changes shared pairing creation semantics and leaves the helper contract, tests, docs, and cross-channel side effects inconsistent.

## What This Changes

The PR changes `src/pairing/pairing-challenge.ts` so an existing pending pairing code can trigger another pairing reply instead of returning early.

## Best Possible Solution

If maintainers want this UX, implement an explicit resend-existing-code path that preserves `created` and `onCreated` semantics, then update the shared helper tests, affected channel expectations, and public pairing docs.

## Reproduction Assessment

Yes for the PR defects: source inspection of the patched guard shows an existing `{ code, created: false }` request falls through, calls `onCreated`, sends another reply, and returns `created: true`. The requested repeat-reply behavior itself has screenshot proof, but it is a feature change from the current documented contract.

## Solution Assessment

No. The one-line guard change is not the best solution because it changes shared creation semantics; a safer approved implementation would preserve `created`, gate `onCreated` to new requests, and update tests/docs/contracts for repeat replies.
