## Summary

Keep open: this branch still proposes a behavior change that is not on main, but current docs and tests define creation-only pairing replies and the one-line guard change conflates repeat sends with new requests.

## What This Changes

This PR changes the shared pairing challenge helper to resend a stored pending pairing code when an unapproved sender repeats `/start`.

## Best Possible Solution

If maintainers want this UX, implement an explicit bounded resend branch that preserves creation state and callback semantics, then update shared/channel tests and docs with redacted Telegram proof.

## Reproduction Assessment

Yes for the PR-introduced behavior. With `upsertPairingRequest` returning `{ code: "ABCD", created: false }`, the PR falls through into reply sending, `onCreated`, and a `{ created: true, code }` result; the inspected Telegram screenshot also shows repeated `/start` replies.

## Solution Assessment

No. The one-line guard change changes shared state and callback semantics; a maintainable fix needs an explicit resend path and a policy decision about bounds and channel scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve creation state when resending codes:** `src/pairing/pairing-challenge.ts:31`
  - body: When the store returns an existing `{ code, created: false }`, this guard now runs creation side effects and returns `{ created: true, code }`. A resend branch should keep `onCreated` and the returned `created` state tied to actual new requests.
  - confidence: 0.95

  - confidence: 0.9
