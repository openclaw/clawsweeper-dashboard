## Summary

Keep open. Current main still does not implement the requested WhatsApp reconnect backfill: it only consumes Baileys upsert events, skips stale `append` history, keeps full socket history sync disabled, and reconnect success only drains outbound retries. The recent #76175 cross-reference is useful and appears to be the active implementation candidate, so this issue should stay open until that work is reviewed, landed, or rejected.

## What This Changes

Request WhatsApp recovery for messages missed during a Web connection outage/reconnect window, especially monitored group messages.

## Best Possible Solution

Review #76175 against the current source behavior and provider semantics, then land a bounded WhatsApp-owned reconnect recovery path or an explicit gap-warning fallback if reliable provider backfill is unavailable.

## Reproduction Assessment

Yes for the source-level current behavior: current main has no reconnect backfill path and its test suite intentionally skips five-minute-old `append` history messages. A live outage/reconnect experiment is still needed to validate the exact WhatsApp provider semantics.

## Solution Assessment

Unclear. The linked configurable offline-message approach may address some missed `append` events, but the requested gap backfill needs maintainer validation that it covers reconnect windows safely or should instead use a cursor/history-sync mechanism plus warning behavior.
