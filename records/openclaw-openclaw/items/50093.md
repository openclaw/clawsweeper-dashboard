## Summary

Keep open. Current main and the latest release still skip older WhatsApp reconnect catch-up messages and have no per-chat cursor, Baileys history-sync listener, or bounded missed-message backfill path; the open related PRs need maintainer policy choice and live WhatsApp proof before this can close.

## What This Changes

Request backfilling WhatsApp messages missed during disconnect and reconnect windows for active monitored chats.

## Best Possible Solution

Implement bounded reconnect recovery inside the WhatsApp plugin: persist eligible chat state, validate Baileys history or append replay semantics, cap lookback and volume, dedupe by message ID before enqueue, preserve startup-history safety, and warn clearly when backfill is unavailable.

## Reproduction Assessment

Yes, for a source-level reproduction: emit a WhatsApp `messages.upsert` event of type `append` with a timestamp more than 60 seconds before socket connection time, and current main skips it before enqueue. I did not establish a live WhatsApp outage reproduction in this read-only review.

## Solution Assessment

Unclear. The WhatsApp plugin is the correct boundary, but maintainers still need to choose between broader history sync, a widened reconnect catch-up window, opt-in replay config, or explicit gap warnings when provider replay cannot be trusted.
