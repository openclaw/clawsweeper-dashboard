## Summary

Keep open. Current main and the latest release still only process recent WhatsApp `append` upserts and replay OpenClaw's own durable inbound journal; they do not request or consume WhatsApp reconnect history for messages that never reached OpenClaw, and the open candidate PRs are conflicting or proof-blocked rather than merged fixes.

## What This Changes

Request WhatsApp reconnect handling to backfill or clearly report messages missed during outage windows for monitored chats.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers choose and merge a bounded WhatsApp reconnect replay, backfill, or gap-warning policy with live proof.

## Reproduction Assessment

No for the full live WhatsApp outage/backfill path; I did not reproduce a real reconnect gap. Yes at source level: a `messages.upsert` event with type `append` and a timestamp more than 60 seconds before socket connect is still skipped, and current tests assert that behavior.

## Solution Assessment

No, current main is not the complete solution because it lacks a provider-history backfill or gap-warning path for messages OpenClaw never saw. The open PRs are plausible candidates for narrower policies, but maintainers still need to choose the replay boundary and require live WhatsApp proof.
