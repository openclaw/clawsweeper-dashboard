## Summary

Keep open. Current main and the latest release still skip stale WhatsApp `append` catch-up messages after a 60-second connect grace and do not implement a reconnect history/backfill path; open candidate PRs exist, but neither has merged and the remaining choice is maintainer policy plus live WhatsApp proof.

## What This Changes

Request WhatsApp reconnect recovery so monitored-chat messages sent during an outage are replayed, processed, or explicitly reported as unrecoverable after reconnection.

## Best Possible Solution

Keep this as the canonical WhatsApp reconnect missed-message tracker and land one plugin-owned bounded recovery policy with live proof: validated replay for eligible monitored chats, durable dedupe/read-receipt handling, and a clear unrecoverable-gap warning when backfill cannot be trusted.

## Reproduction Assessment

Yes, source-level: current main skips `append` messages older than the 60-second pre-connect guard, and the append-upsert test asserts a five-minute append is not delivered. I did not run a live WhatsApp outage replay in this read-only review.

## Solution Assessment

Unclear. The WhatsApp plugin inbound/reconnect path is the right owner boundary, but maintainers still need to choose between default bounded replay, opt-in replay, or explicit gap warnings and require live WhatsApp proof.
