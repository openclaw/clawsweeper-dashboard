## Summary

Keep open. Current main and v2026.5.18 still lack a full WhatsApp missed-message backfill, per-chat cursor, or `messaging-history.set` processing path; existing code only handles Baileys upsert events and skips `append` backlog older than the 60-second connect grace window, while the open implementation PRs cover competing partial approaches that need maintainer review.

## What This Changes

Request reconnect-time WhatsApp backfill so messages missed during WhatsApp Web outages are recovered or logged for monitored chats.

## Best Possible Solution

Review the open WhatsApp PRs and land a plugin-owned bounded reconnect recovery design that filters to eligible chats, persists or dedupes by cursor/message ID, caps replay, handles Baileys history limitations, and documents fallback warnings.

## Reproduction Assessment

Yes for source-level current behavior: current main has no per-chat reconnect cursor/history listener/config path and its append regression test expects a five-minute-old catch-up message to be skipped. I did not run a live WhatsApp outage reproduction.

## Solution Assessment

Unclear. The WhatsApp plugin is the right boundary, but maintainers need to choose between recent `append` replay, opt-in offline-message handling, `messaging-history.set` cursor backfill, and warning-only behavior when provider backfill is unreliable.
