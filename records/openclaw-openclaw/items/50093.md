## Summary

Keep open. Current main still lacks bounded WhatsApp reconnect backfill: it only consumes Baileys upsert events, skips append history older than the 60-second reconnect grace window, keeps full socket history sync disabled, and has no per-chat cursor or history replay path. The newly noted PR #21719 is useful source context, but it is closed unmerged and only proposed an opt-in recent-append path rather than proving the requested missed-message backfill is implemented.

## What This Changes

Request to recover WhatsApp messages missed during Web connection outages by backfilling monitored-chat history after reconnection.

## Best Possible Solution

Implement bounded reconnect recovery inside the WhatsApp plugin with per-account/per-chat last-seen state, a validated Baileys history mechanism for eligible chats, capped replay volume and age, message-ID dedupe before enqueueing, and an explicit operator warning when provider backfill cannot be trusted.

## Reproduction Assessment

Yes for current code-level behavior: a five-minute-old append message is covered by a regression test that expects no onMessage call, and current source has no reconnect history listener or cursor path. A live WhatsApp outage/reconnect experiment is still needed to validate the provider-side history mechanism and exact missed-message delivery behavior.

## Solution Assessment

Unclear. The cursor-plus-history direction is plausible, and PR #21719 is useful prior art for opt-in recent append handling, but the safest fix needs maintainer validation of WhatsApp/Baileys history semantics plus explicit replay, privacy, dedupe, and fallback-warning decisions.
