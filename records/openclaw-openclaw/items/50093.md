## Summary

Keep open. Current main still lacks WhatsApp reconnect backfill: inbound delivery is limited to Baileys upsert events, stale `append` history remains intentionally skipped, full socket history sync is disabled, and reconnect handling only drains outbound deliveries.

## What This Changes

The issue requests bounded WhatsApp missed-message backfill after reconnect for monitored chats, especially groups, using a per-chat cursor/history replay or at least an operator warning when replay is unavailable.

## Best Possible Solution

Implement bounded reconnect recovery inside the WhatsApp plugin: persist last-seen state per account and eligible chat, validate a Baileys history mechanism for monitored chats, cap lookback and replay volume, dedupe by message ID before enqueueing, and emit a clear warning when reliable provider backfill is unavailable.

## Reproduction Assessment

Yes for current code-level behavior: an `append` upsert with a timestamp five minutes before connect is covered by tests and is expected not to call `onMessage`. A live WhatsApp outage/reconnect experiment is still needed to validate provider-side history availability and exact missed-message behavior.

## Solution Assessment

Unclear. The requested cursor-plus-history direction is plausible, but the safest solution needs maintainer validation of Baileys history semantics; if reliable replay is unavailable, a bounded gap-warning fallback may be the maintainable interim path.
