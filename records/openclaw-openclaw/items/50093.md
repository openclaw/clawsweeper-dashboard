## Summary

Keep open. Current main still does not backfill WhatsApp messages missed during a Web reconnect gap: inbound delivery is driven by Baileys upsert events, stale append history is intentionally skipped, full socket history sync is disabled, and reconnect success only drains outbound delivery retries. The BlueBubbles catch-up item is useful related context, but it is channel-specific and does not supersede this WhatsApp request.

## What This Changes

Request to recover WhatsApp messages missed during Web connection outages by backfilling monitored-chat history after reconnection.

## Best Possible Solution

Implement bounded reconnect recovery inside the WhatsApp plugin: persist last-seen state per account and eligible chat, validate the Baileys history mechanism for monitored chats, cap lookback and replay volume, dedupe by message ID before enqueueing, and emit a clear warning when reliable provider backfill is unavailable.

## Reproduction Assessment

Yes for current code-level behavior: the append-upsert regression test shows a five-minute-old `append` message is expected not to call `onMessage`, and current source has no reconnect backfill path. A live WhatsApp outage/reconnect experiment is still needed to validate the provider-side history mechanism and exact missed-message behavior.

## Solution Assessment

Unclear. The requested cursor-plus-history direction is plausible and aligns with Baileys history APIs, but the safest fix requires maintainer validation of provider semantics plus explicit privacy, replay-limit, dedupe, and fallback-warning decisions.
