## Summary

Keep open. Current main and the latest shipped release still skip stale WhatsApp `append` catch-up messages, do not subscribe to Baileys `messaging-history.set`, and the two related open PRs remain unmerged competing repair paths that need maintainer policy and live proof review.

## What This Changes

The issue requests WhatsApp reconnect recovery that backfills or otherwise accounts for messages missed while the Web connection was down.

## Best Possible Solution

Land one bounded WhatsApp-plugin recovery design that filters to eligible monitored chats, caps replay, dedupes by message ID, preserves startup-history safety, documents Baileys limitations, and includes real reconnect proof or an explicit gap warning when replay cannot be trusted.

## Reproduction Assessment

Yes for source-level reproduction: emit a WhatsApp `messages.upsert` event of type `append` with a timestamp more than 60 seconds before `connectedAtMs`, and current main skips it before enqueue. I did not run a live WhatsApp outage reproduction.

## Solution Assessment

Unclear. The WhatsApp plugin is the right boundary, but maintainers still need to choose whether the safest fix is widened `append` replay, a `messaging-history.set` path, opt-in config, gap warnings, or a combination with live Baileys proof.
