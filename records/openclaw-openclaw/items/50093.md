## Summary

Keep open. Current main still processes only Baileys-delivered `notify`/`append` upserts, skips stale `append` messages older than the 60-second connect grace window, and has no per-chat reconnect backfill cursor or history-sync path; the open WhatsApp PRs cover competing policies that need maintainer choice and live proof.

## What This Changes

Request WhatsApp reconnect recovery that backfills or otherwise handles monitored-chat messages missed while the Web connection was down.

## Best Possible Solution

Choose one bounded WhatsApp-plugin reconnect recovery policy that filters to eligible monitored chats, caps replay, dedupes by message ID, preserves startup-history safety, documents Baileys limits, and proves replay or explicit gap-warning behavior with a real reconnect test.

## Reproduction Assessment

Yes for source-level reproduction: emit a WhatsApp `messages.upsert` event of type `append` with a timestamp more than 60 seconds before the socket connection time, and current main skips it before enqueue. I did not establish a live WhatsApp outage reproduction in this read-only review.

## Solution Assessment

Unclear. The WhatsApp plugin is the right boundary, but maintainers still need to choose whether the solution is broader history backfill, a widened reconnect catch-up window, an opt-in config, or a gap warning when provider replay cannot be trusted.
