## Summary

Keep open. Current main still lacks WhatsApp reconnect backfill, per-chat cursoring, or a history-sync listener; it only processes live `messages.upsert` events and continues to skip stale `append` backlog after a 60-second guard. Two unmerged PRs are active implementation candidates, so this should stay open until maintainers choose and land or reject the reconnect-replay policy.

## What This Changes

Request that the WhatsApp plugin backfill messages missed during a dropped Web connection after reconnection.

## Best Possible Solution

Review the open WhatsApp PRs and land a bounded plugin-owned reconnect recovery path that caps replay, filters to eligible chats, dedupes by message ID, and documents whether broader history-sync cursoring remains out of scope.

## Reproduction Assessment

Yes for source-level current behavior: main still has no reconnect-time backfill/cursor/history listener and intentionally skips stale `append` catch-up messages. I did not establish a live WhatsApp outage reproduction in this read-only review.

## Solution Assessment

Unclear. The WhatsApp plugin is the right ownership boundary, but maintainers still need to choose whether the best fix is focused recent-append replay, opt-in offline-message config, a `messaging-history.set` cursor design, or warning-only behavior when reliable backfill is unavailable.
