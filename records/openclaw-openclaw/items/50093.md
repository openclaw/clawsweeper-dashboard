## Summary

Keep open. Current `main` still has no WhatsApp reconnect backfill or per-chat cursor path: it only handles Baileys `messages.upsert`, skips older `append` history, keeps full socket history sync disabled, and reconnect success only drains outbound deliveries. The active implementation candidate is https://github.com/openclaw/openclaw/pull/76175, so this issue should remain open until that PR is reviewed, landed, or rejected.

## What This Changes

The issue requests WhatsApp reconnect recovery so messages sent to monitored chats during an outage are backfilled or otherwise surfaced after reconnection.

## Best Possible Solution

Review and land a bounded WhatsApp-plugin reconnect recovery path that persists eligible last-seen state, validates the Baileys history mechanism, caps replay, dedupes by message ID, and emits a clear gap warning when reliable backfill is unavailable.

## Reproduction Assessment

Yes for source-level current behavior: the checked-out WhatsApp monitor has no reconnect backfill path and intentionally skips older `append` history messages. I did not establish a live WhatsApp outage reproduction in this read-only review.

## Solution Assessment

Unclear. The WhatsApp plugin is the right ownership boundary, but the exact best fix still depends on maintainer validation of Baileys history semantics and replay policy before choosing between configurable append replay, `messaging-history.set`, explicit fetch, and warning-only fallback behavior.
