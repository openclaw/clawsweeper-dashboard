## Summary

Keep open. Current main and the latest release still skip WhatsApp `append` catch-up messages older than the 60-second connect grace window and do not implement a per-chat reconnect backfill/cursor path; the linked open PRs are plausible fix candidates but still need maintainer policy choice and live WhatsApp proof.

## What This Changes

The issue asks OpenClaw to recover or process WhatsApp messages missed while the WhatsApp Web/Baileys connection is disconnected and then reconnects.

## Best Possible Solution

Land one WhatsApp-plugin reconnect recovery design that preserves startup-history safety, bounds replay to eligible monitored chats, dedupes by message ID, validates Baileys history behavior with live proof, and emits a clear warning when provider backfill is unavailable.

## Reproduction Assessment

Yes, source-level: emit a WhatsApp `messages.upsert` event of type `append` with a timestamp more than 60 seconds before the socket connection time, and current main skips it before enqueue. I did not establish a live WhatsApp outage reproduction in this read-only review.

## Solution Assessment

Unclear as a final product choice. The WhatsApp plugin is the right boundary, but maintainers still need to choose between a narrow reconnect grace window, opt-in offline replay config, broader history sync/cursor handling, or explicit gap warnings when backfill cannot be trusted.
