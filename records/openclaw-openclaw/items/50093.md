## Summary

Keep open. Current main still has no full WhatsApp missed-message backfill, per-chat cursor, or `messaging-history.set` processing path; it only accepts `messages.upsert` events and skips stale `append` catch-up messages after the short startup guard. The active implementation candidates are https://github.com/openclaw/openclaw/pull/80642 and https://github.com/openclaw/openclaw/pull/76175, so the issue should remain open until maintainers review, land, or reject those approaches.

## What This Changes

Request WhatsApp to recover messages missed during Web disconnects by replaying or backfilling active monitored chats after reconnect.

## Best Possible Solution

Review the open WhatsApp PRs and land a bounded plugin-owned reconnect recovery path that caps replay, filters to eligible chats, dedupes by message ID, and documents whether broader `messaging-history.set`/cursor backfill remains out of scope.

## Reproduction Assessment

Yes for source-level current behavior: current main has no reconnect-time backfill/cursor/history listener and intentionally skips stale `append` catch-up messages. I did not establish a live WhatsApp outage reproduction in this read-only review.

## Solution Assessment

Unclear. The WhatsApp plugin is the right ownership boundary, but maintainers still need to choose whether the best fix is the focused recent-append replay, an opt-in offline-message config, a `messaging-history.set` cursor design, or a warning-only fallback when reliable backfill is unavailable.
