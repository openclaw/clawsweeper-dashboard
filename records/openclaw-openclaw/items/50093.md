## Summary

Keep open. Current main still skips stale WhatsApp `append` catch-up messages, has no reconnect-history/backfill listener or cursor, and the linked open PRs propose different replay policies that need maintainer choice and live WhatsApp proof.

## What This Changes

Request backfilling WhatsApp messages missed during disconnect/reconnect windows for active monitored chats.

## Best Possible Solution

Implement bounded reconnect recovery inside the WhatsApp plugin: track eligible chat state, validate Baileys history or append replay semantics, cap lookback and replay volume, dedupe before enqueue, preserve startup-history safety, and emit a clear warning when backfill is unavailable.

## Reproduction Assessment

Yes, for a source-level reproduction: emit a WhatsApp `messages.upsert` event of type `append` with a timestamp more than 60 seconds before socket connection time, and current main skips it before enqueue. I did not establish a live WhatsApp outage reproduction in this read-only review.

## Solution Assessment

Unclear. The WhatsApp plugin is the correct boundary, but maintainers still need to choose between broader history sync, a widened reconnect catch-up window, opt-in replay config, or explicit gap warnings when provider replay cannot be trusted.
