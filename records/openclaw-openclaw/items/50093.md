## Summary

Keep open. Current main still lacks a WhatsApp reconnect backfill/cursor path: it only consumes Baileys upsert events, skips stale append history, keeps full socket history sync disabled, and reconnect success only drains outbound delivery retries. The linked open PR #76175 is the active implementation candidate, so this issue should remain open until that work is reviewed, landed, or rejected.

## What This Changes

Request that WhatsApp recover messages missed during Web connection outages by backfilling monitored chats after reconnect.

## Best Possible Solution

Review #76175 against current source behavior and provider semantics, then land a bounded WhatsApp-owned reconnect recovery path or an explicit gap-warning fallback if reliable provider backfill is unavailable.

## Reproduction Assessment

Yes for source-level current behavior: current main has no reconnect backfill path and its test suite intentionally skips five-minute-old append history messages. I did not establish a live provider outage reproduction in this read-only review.

## Solution Assessment

Unclear. The requested direction is reasonable, but the safest fix needs maintainer validation of Baileys/WhatsApp Web history semantics and replay policy before choosing between cursor/history sync, configurable append replay, and warning-only fallback behavior.
