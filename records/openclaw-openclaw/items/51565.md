## Summary

Keep open. Current main still does not implement the PR's intended WhatsApp per-agent ack emoji fallback, and this PR remains useful, but the branch needs maintainer/contributor follow-up before merge because the WhatsApp ack path has moved on and the SDK export/test typing choices still need review.

## What This Changes

The PR adds a WhatsApp-specific ack emoji resolver, wires WhatsApp ack reactions through it, exports it from `channel-feedback`, and adds resolver tests for object-shaped WhatsApp ack configs and agent identity fallback.

## Best Possible Solution

Keep this PR open and revise it against current main: integrate a WhatsApp-specific resolver into the current ack handle path, preserve `reactionLevel` gating and removal-after-reply behavior, honor account-level precedence, skip `messages.ackReaction`, keep no-config and empty-emoji no-reaction semantics, and cover the cases with typed tests.

## Reproduction Assessment

Yes. A focused source-level reproduction is enough: on current main, `channels.whatsapp.ackReaction: { direct: true, group: "mentions" }` produces `emoji === ""`, and the WhatsApp ack gate returns false, so no per-agent identity emoji reaction is sent.

## Solution Assessment

Unclear as currently branch-stale, but the WhatsApp-specific resolver direction is narrower than using the generic `resolveAckReaction` chain. The safest solution is to rebase it over current main's ack reaction handle flow and decide whether the helper belongs in the public SDK or WhatsApp-local code.
