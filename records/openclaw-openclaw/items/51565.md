## Summary

This PR should stay open because the underlying WhatsApp ack-reaction gap is still visible on current main, but the branch is stale/dirty and the submitted shape would drop newer WhatsApp ack-handle behavior unless it is repaired before merge.

## What This Changes

The PR adds WhatsApp ack emoji fallback resolution, routes WhatsApp ack reactions through that resolver, exports the resolver through `plugin-sdk/channel-feedback`, and adds identity resolver tests for object-format ack configs.

## Best Possible Solution

Implement a WhatsApp-local resolver that derives emoji and scope from the effective account config, skips global `messages.ackReaction`, preserves `reactionLevel`, group activation, ack handles/removal, and adds focused tests plus changelog coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows a valid WhatsApp `ackReaction` object without `emoji` becomes an empty string, and the WhatsApp ack gate suppresses blank emoji reactions.

## Solution Assessment

No as submitted. The WhatsApp-specific resolver direction is useful, but it needs to be rebased into the current async ack-handle flow and kept out of core identity policy where possible.
