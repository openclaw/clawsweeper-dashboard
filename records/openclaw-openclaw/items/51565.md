## Summary

Keep open. The reported WhatsApp behavior is still present on current main, and the PR addresses a valid typed config shape, but the current PR head is stale against the newer WhatsApp ack-reaction handle flow and needs revision before it is safe to merge.

## What This Changes

This PR adds a WhatsApp-specific ack emoji resolver, exports it through `channel-feedback`, routes WhatsApp ack reactions through that resolver, and adds identity resolver tests for object-shaped WhatsApp `ackReaction` configs and agent identity fallback.

## Best Possible Solution

Revise the PR into a current-main-compatible WhatsApp ack resolver: preserve `reactionLevel` gating and `AckReactionHandle` removal-after-reply behavior, resolve account/channel object emojis, fall back to the responding agent identity only when WhatsApp ack config is present without an emoji, skip `messages.ackReaction`, keep tests typed, and add the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: on current main, a valid WhatsApp `ackReaction` object without `emoji` produces an empty `emoji`, and the WhatsApp ack gate then suppresses the reaction.

## Solution Assessment

No as currently submitted. The resolver direction is appropriate, but the branch must be rebased onto the current async ack-handle flow and the public SDK seam should be kept narrow or adjusted during that repair.
