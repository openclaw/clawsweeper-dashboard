## Summary

Keep open: current main has not implemented the per-agent WhatsApp fallback, but this branch is conflict-stale, lacks real WhatsApp proof, and still needs maintainer agreement on the visible fallback policy and public SDK surface.

## What This Changes

The PR adds a WhatsApp ack-emoji resolver, exports it through `plugin-sdk/channel-feedback`, uses it in WhatsApp ack sending, and adds resolver tests for object-shaped WhatsApp ack configs.

## Best Possible Solution

Rework this on the current WhatsApp plugin path as a local effective ack resolver that preserves the ack handle lifecycle, derives emoji/direct/group from one resolved config, documents the intentional fallback, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes at source level: current main reads only `ackConfig?.emoji`, so a WhatsApp `ackReaction` object without `emoji` becomes an empty emoji and is suppressed. A live WhatsApp reproduction or after-fix proof is still missing.

## Solution Assessment

No, not as submitted. The maintainable solution is a WhatsApp-local effective config resolver on top of the current ack lifecycle, not a stale branch that broadens the SDK with WhatsApp-specific policy.
