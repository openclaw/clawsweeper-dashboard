## Summary

Keep open: current main still has the reported WhatsApp partial-ack behavior, but this PR is not mergeable as submitted because it is stale against the current ack-handle flow, mixes account-level emoji with channel-level scope, moves WhatsApp policy into core identity, and lacks real WhatsApp behavior proof.

## What This Changes

The PR adds WhatsApp ack emoji fallback resolution, exports it through `plugin-sdk/channel-feedback`, routes WhatsApp ack sending through it, and adds identity resolver tests for object-form `ackReaction` configs.

## Best Possible Solution

Implement a WhatsApp-owned effective ack config resolver that preserves `reactionLevel`, group activation, ack removal, account scope, and the skip of `messages.ackReaction`, with focused tests and real WhatsApp proof.

## Reproduction Assessment

Yes, by source inspection. On current main, a valid WhatsApp `ackReaction` object without `emoji` resolves to a blank emoji and `shouldAckReactionForWhatsApp` suppresses the reaction.

## Solution Assessment

No, not as submitted. The WhatsApp-specific resolver direction is useful, but it should be rebased into the current async ack-handle flow and kept inside the WhatsApp plugin or a narrow generic seam.
