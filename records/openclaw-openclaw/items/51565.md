## Summary

Keep open. The PR targets a real WhatsApp ack-reaction gap, but the branch is stale against current main and needs rework around the async ack handle flow, account-scoped config, extension/core boundaries, tests, and changelog coverage before merge.

## What This Changes

The branch adds WhatsApp ack emoji fallback resolution, routes WhatsApp ack reactions through that resolver, exports it through `channel-feedback`, and adds identity resolver tests for object-format ack configs.

## Best Possible Solution

Revise the change into a current-main-compatible WhatsApp ack resolver that preserves `reactionLevel`, group activation, ack removal, account context, and WhatsApp’s skip of `messages.ackReaction`, with focused tests and changelog coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows a valid WhatsApp `ackReaction` object without `emoji` becomes an empty string, and the WhatsApp ack gate suppresses blank emoji reactions.

## Solution Assessment

No as submitted. The WhatsApp-specific resolver direction is useful, but it needs to be rebased into the current async ack-handle flow and kept aligned with account-scoped config and extension-boundary rules.
