## Summary

Keep open: the per-agent WhatsApp ack fallback is still not implemented on current main, but this branch is conflicting and needs rework before it is a safe merge candidate.

## What This Changes

The PR adds a WhatsApp ack emoji resolver that falls back from account/channel ackReaction objects to the agent identity emoji, exports it through channel-feedback, wires WhatsApp ack sending to it, and adds resolver tests.

## Best Possible Solution

Rework this as a WhatsApp-local effective ack resolver that derives emoji and direct/group scope from the same account/channel config, preserves current ack-handle cleanup, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Unclear for a live setup, but source inspection gives a reproducible path: current main turns a WhatsApp ackReaction object without `emoji` into an empty emoji and suppresses the ack. No redacted live WhatsApp session proof is present.

## Solution Assessment

No, not as submitted. The maintainable path is a plugin-local effective WhatsApp ack config resolver that preserves the current handle lifecycle and does not broaden the SDK with one-channel policy.
