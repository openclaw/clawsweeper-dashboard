## Summary

Keep open: the user problem is still valid on current main, but the branch needs changes before merge because it targets an older WhatsApp ack flow, crosses the plugin/core boundary, and lacks after-fix real WhatsApp behavior proof.

## What This Changes

The PR adds WhatsApp ack emoji fallback resolution, exports it through `channel-feedback`, routes WhatsApp ack reactions through it, and adds identity resolver tests for object-format ack configs.

## Best Possible Solution

Revise this into a current-main-compatible WhatsApp-owned ack resolver that preserves `reactionLevel`, group activation, ack removal, account context, and WhatsApp’s skip of `messages.ackReaction`, with focused tests and real behavior proof.

## Reproduction Assessment

Yes by source inspection. A valid WhatsApp config with `ackReaction` present but no `emoji` makes `emoji` blank in `maybeSendAckReaction`, and the WhatsApp ack gate then suppresses the reaction.

## Solution Assessment

No as submitted. The WhatsApp-specific resolver direction is useful, but it needs to be rebased into the current async ack-handle flow and kept inside the WhatsApp plugin boundary or a narrow generic seam.
