## Summary

Keep open: current main still suppresses WhatsApp ack reactions when `ackReaction` has `direct`/`group` but no `emoji`, so the central request is not implemented. The PR is not merge-ready because it is conflicting, lacks live WhatsApp proof, mixes account-level emoji resolution with channel-level scope flags, and promotes WhatsApp-specific policy through a broad SDK barrel.

## What This Changes

This PR adds a WhatsApp-specific ack emoji resolver in agent identity, exports it through `plugin-sdk/channel-feedback`, uses it for WhatsApp ack reactions, and adds resolver tests for object-shaped `ackReaction` configs.

## Best Possible Solution

Rework this against current main as a WhatsApp-local effective ack resolver that derives emoji and scope from the same merged account config, preserves the current ack lifecycle, and has live redacted WhatsApp proof.

## Reproduction Assessment

Yes at source level: current main resolves WhatsApp ack emoji from `ackConfig?.emoji ?? ""`, and the shared gate suppresses empty emojis. I did not run a live WhatsApp transport reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The best repair is a WhatsApp-local effective config resolver on the current ack lifecycle, not a stale branch that broadens the public SDK with WhatsApp-specific policy.
