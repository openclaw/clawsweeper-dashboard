## Summary

Keep open: the reported WhatsApp per-agent ack gap is real from current source, but this PR is not merge-ready because it puts WhatsApp-specific resolution in core/SDK, lacks the required changelog entry, and still needs rebase against current main.

## What This Changes

The PR adds object-aware ack emoji resolution, introduces a WhatsApp-specific ack emoji resolver, routes WhatsApp ack reactions through it for agent identity fallback, and adds identity tests.

## Best Possible Solution

Land a rebased patch that keeps WhatsApp-specific fallback logic in the WhatsApp plugin or behind a generic core helper, preserves no-config and no-global-message behavior, covers account/direct/group cases, and records the user-visible change.

## Reproduction Assessment

Yes, from source inspection. Current main reads only `channels.whatsapp.ackReaction.emoji`, and `shouldAckReactionForWhatsApp` suppresses empty emoji values, so a WhatsApp ackReaction object without `emoji` cannot fall back to the routed agent identity emoji.

## Solution Assessment

No. The intended behavior is reasonable, but the current PR puts a WhatsApp-named resolver in core identity and the broad SDK barrel; a narrower WhatsApp-owned helper or generic object-emoji helper plus changelog is the maintainable path.
