## Summary

Keep open: the reported WhatsApp behavior is plausible from source, but the branch is stale and conflicting with current ack cleanup, changes partial-config behavior, exposes channel-specific policy through a broad SDK barrel, and only has unit-test proof.

## What This Changes

The PR adds a WhatsApp ack emoji resolver that falls back from account/channel ackReaction objects to the responding agent’s identity emoji, exports it via channel-feedback, wires WhatsApp ack sending to it, and adds resolver tests.

## Best Possible Solution

Implement the fallback as current-main WhatsApp behavior that uses the effective account/channel ack config, preserves `AckReactionHandle` cleanup, keeps the resolver WhatsApp-local or behind a narrow seam, documents the intentional partial-config fallback, and includes unit plus redacted live WhatsApp proof.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection does show the current-main path for `ackReaction` objects without `emoji`: WhatsApp resolves an empty emoji and suppresses the ack reaction.

## Solution Assessment

No, not as currently proposed. The useful behavior should be reworked against current main, kept at the WhatsApp ownership boundary or a narrow seam, and proven in a real WhatsApp setup before merge.
