## Summary

Keep this PR open, but it is not mergeable as submitted: the useful WhatsApp per-agent ack fallback is not on current main, while the branch is stale against the current async ack handle/removal flow, puts WhatsApp policy into core identity/SDK surface, and still lacks contributor real-behavior proof.

## What This Changes

The branch adds WhatsApp ack emoji fallback from account/channel object configs to the responding agent identity emoji, exports that resolver through `channel-feedback`, and adds resolver tests.

## Best Possible Solution

Implement a WhatsApp-owned effective ack resolver in the plugin flow, preserving account-scoped config, reactionLevel, group activation, ack removal, and the documented skip of `messages.ackReaction`.

## Reproduction Assessment

Yes, by source inspection. Current main accepts a WhatsApp `ackReaction` object without `emoji`, resolves it to a blank emoji, and then suppresses the ack reaction; no live WhatsApp reproduction was run here.

## Solution Assessment

No, not as submitted. The useful behavior should be rebased into the current WhatsApp ack-handle path and kept plugin-local or behind a narrow generic seam instead of adding WhatsApp-specific policy to core identity and a broad SDK barrel.
