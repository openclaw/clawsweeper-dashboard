## Summary

Keep open: the WhatsApp per-agent ack emoji behavior is still a plausible improvement, but this branch is not mergeable as submitted because it is stale against current main's ack lifecycle flow, mixes emoji resolution with stale scope config, moves WhatsApp policy into core identity/SDK, and lacks contributor real WhatsApp proof.

## What This Changes

The PR makes WhatsApp ack reactions fall back from account/channel object configs without an emoji to the responding agent identity emoji, exports that resolver through channel-feedback, and adds resolver tests.

## Best Possible Solution

Implement a WhatsApp-owned effective ack resolver in the plugin flow that computes emoji/direct/group from the same account/channel config, preserves reactionLevel, group activation, and removeAckAfterReply, and keeps core identity channel-agnostic.

## Reproduction Assessment

Yes, by source inspection: current main accepts a WhatsApp ackReaction object without emoji, resolves the emoji to an empty string, and then suppresses the ack reaction. No live WhatsApp session was exercised here.

## Solution Assessment

No, not as submitted. The useful behavior should be rebased into the current WhatsApp AckReactionHandle path and kept plugin-local or behind a narrow generic seam instead of adding WhatsApp-specific policy to core identity and a broad SDK barrel.
