## Summary

Keep open: the PR targets a real WhatsApp ack-reaction gap on current main, but the submitted branch is conflicting, predates the current async ack-handle/removal flow, lacks contributor real WhatsApp proof, and moves WhatsApp-specific policy into core/SDK surface.

## What This Changes

The PR adds WhatsApp ack emoji fallback resolution so configured WhatsApp ack reactions can use the responding agent’s identity emoji, plus related SDK exports and resolver tests.

## Best Possible Solution

Implement a WhatsApp-owned effective ack resolver in the plugin flow, preserving account-scoped config, reactionLevel, group activation, ack removal, and the documented skip of `messages.ackReaction`.

## Reproduction Assessment

Yes, by source inspection. Current main accepts WhatsApp ack config objects with optional `emoji`, reads a blank emoji when it is absent, and then suppresses the ack reaction.

## Solution Assessment

No, not as submitted. The useful behavior should be rebased into the current WhatsApp ack-handle path and kept inside the WhatsApp plugin or a narrow generic seam rather than core identity policy.
