## Summary

Keep open: the WhatsApp per-agent ack emoji gap is still source-reproducible on current main, but this external PR is conflicting/stale, lacks live WhatsApp proof, and needs maintainer judgment on the fallback policy and SDK/core boundary before it can merge.

## What This Changes

The PR adds a WhatsApp ack emoji fallback resolver, exports it through `channel-feedback`, wires WhatsApp ack sending to use it, and adds resolver tests.

## Best Possible Solution

Implement a WhatsApp-owned effective ack resolver in the plugin flow that resolves emoji, direct/group scope, account overrides, reactionLevel, status reactions, and removeAckAfterReply together, then require redacted live WhatsApp proof for the changed behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows valid WhatsApp `ackReaction` objects without `emoji` resolve to an empty string and are suppressed, but no live WhatsApp session was exercised in this review.

## Solution Assessment

No, not as submitted. The useful behavior should be rebased into the current WhatsApp ack/status reaction path and kept plugin-local or behind a narrow generic seam instead of adding WhatsApp-specific policy to core identity and a broad SDK barrel.
