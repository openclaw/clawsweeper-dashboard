## Summary

Keep open. The copied-snippet hardening is useful, but the diff still drops existing Signal exec target-mode reaction binding because the current Signal exec renderer emits the manual fallback payload with `/approve` and `Full id:` but no standalone `ID:` header, while the new parser rejects that shape. The added proof is still a Vitest/local JSON-RPC path, not redacted real Signal service/device proof.

## What This Changes

This PR changes Signal approval-reaction extraction to require a matching canonical `ID:` header before appending reaction hints or registering reaction targets, and adds parser plus send-path tests.

## Best Possible Solution

Keep the copied-snippet guard, preserve exec target-mode binding by emitting a canonical `ID:` header for the Signal exec pending payload or by a narrow verified fallback parser, then require redacted real Signal service/device proof.

## Reproduction Assessment

Yes. Source inspection shows current main emits Signal exec target-mode text with `/approve` and `Full id:` but no standalone `ID:` header, while the PR parser rejects that text before send-time binding.

## Solution Assessment

No in current form. Header gating is the right copied-snippet mitigation, but the best fix must preserve existing exec target-mode reaction binding before merge.
