## Summary

Keep open: the PR is still a useful Signal approval-reaction bug fix, but the current head drops the new approval hint before the shared outbound channel handler, and the proof remains local-endpoint-only with maintainer compatibility/proof judgment still required.

## What This Changes

The PR makes Signal approval reaction binding require approval-owned delivery plus a canonical ID header, adds ID headers to Signal approval prompts, threads an outbound approval hint, and adds focused tests.

## Best Possible Solution

Add `hint?: ChannelOutboundPayloadHint` to `ChannelHandlerParams`, pass `params.hint` into `createChannelHandler`, keep the stricter ID-header gate, and merge only after maintainer accepts the compatibility/proof boundary.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `createChannelOutboundContextBase` reads `params.hint`, but `ChannelHandlerParams` and the `createChannelHandler` call never carry that field, so the Signal adapter cannot receive the approval-pending hint.

## Solution Assessment

No, not yet. The canonical ID-header gate is a reasonable fix shape, but the PR must wire the trusted approval hint through the shared outbound handler and prove the target-mode Signal path, not only direct `sendMessageSignal` calls.
