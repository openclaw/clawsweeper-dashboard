## Summary

Keep open: current main and v2026.6.9 still keep SecretRef-managed dashboard URLs non-tokenized, do not register a token-print/copy recovery flag, and `config get` returns the redacted config object rather than resolving plaintext. The only linked fix PR is closed unmerged after review found a functional blocker and unresolved security-boundary concerns, so this remains a maintainer-owned auth recovery decision.

## What This Changes

The issue requests an operable way for users with SecretRef-managed gateway tokens to recover Control UI access after browser sessionStorage is lost.

## Best Possible Solution

Land one maintainer-approved dashboard auth recovery path: preferably session/exchange-token auth, or an explicit copy-token command that never logs the bearer and ships with focused tests, docs, and real Control UI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a SecretRef-managed `gateway.auth.token` makes `openclaw dashboard` copy/open a non-tokenized URL, and `config get` does not resolve plaintext SecretRefs.

## Solution Assessment

Unclear until maintainers choose the security boundary. A copy-token command is a plausible interim fix, but session/exchange-token auth may be the cleaner long-term solution.
