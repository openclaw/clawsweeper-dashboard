## Summary

Keep open: current main and the latest release still leave SecretRef-managed dashboard tokens non-tokenized with no supported plaintext recovery path, while the only linked fix PR is closed unmerged after functional proof and security-boundary concerns.

## What This Changes

The issue reports that SecretRef-managed `gateway.auth.token` values leave users without an operable documented way to recover the Control UI login token after browser sessionStorage is lost.

## Best Possible Solution

Land one maintainer-approved dashboard recovery path: preferably session or one-time exchange auth, or a focused explicit copy-token flow that never logs the bearer and updates tests and docs.

## Reproduction Assessment

Yes. Source inspection shows that SecretRef-managed tokens are resolved but marked `secretRefConfigured`, so `openclaw dashboard` copies/opens a non-tokenized URL while `openclaw config get gateway.auth.token` reads the redacted config object instead of plaintext.

## Solution Assessment

Unclear until maintainers choose the security boundary. A copy-token command is a plausible interim fix, but a session or one-time exchange flow may be the cleaner end state for avoiding plaintext handling.
