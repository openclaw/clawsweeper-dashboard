## Summary

Keep open: current main and v2026.6.9 still leave SecretRef-managed dashboard users without a plaintext recovery path, and the only linked fix PR closed unmerged after functional and security-boundary review blockers.

## What This Changes

The issue asks for an operable way to authenticate the Control UI when `gateway.auth.token` is SecretRef-managed and `openclaw config get gateway.auth.token` returns the SecretRef object instead of a usable token.

## Best Possible Solution

Land one maintainer-approved dashboard auth recovery path: preferably session/exchange-token auth, or an explicit copy-token command that never logs the bearer and ships with focused tests, docs, and real Control UI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a SecretRef-managed `gateway.auth.token` and missing browser sessionStorage, `openclaw dashboard` copies/opens a non-tokenized URL while `config get` reads redacted config rather than resolving plaintext.

## Solution Assessment

Unclear until maintainers choose the security boundary. A copy-token flag is a plausible interim fix, but the closed linked PR was not sufficient; session/exchange-token auth may be the cleaner end state.
