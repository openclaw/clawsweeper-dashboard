## Summary

Keep open: current main and v2026.6.9 still leave SecretRef-managed dashboard tokens without an operable recovery path after browser sessionStorage is lost. The only linked fix PR, https://github.com/openclaw/openclaw/pull/93089, closed unmerged after review found a functional blocker and unresolved security-boundary concerns.

## What This Changes

The issue reports that SecretRef-managed gateway dashboard tokens cannot be recovered through the documented CLI path after the Control UI browser session is lost.

## Best Possible Solution

Land one maintainer-approved dashboard auth recovery path: preferably session/exchange-token auth, or an explicit copy-token command that never logs the bearer and ships with focused tests, docs, and real Control UI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a SecretRef-managed `gateway.auth.token` and missing browser sessionStorage, `openclaw dashboard` copies/opens a non-tokenized URL, while `config get` returns the redacted config object rather than plaintext.

## Solution Assessment

Unclear until maintainers choose the security boundary. A copy-token flag is a plausible interim fix, but the closed linked PR was not sufficient; session/exchange-token auth may be the cleaner end state.
