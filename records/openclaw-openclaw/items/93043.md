## Summary

Keep open: current main and v2026.6.9 still leave SecretRef-managed dashboard tokens without an operable recovery path, and the linked fix PR closed unmerged after functional and security-boundary review blockers.

## What This Changes

The issue requests a dashboard/auth recovery path for SecretRef-managed gateway tokens because `openclaw dashboard` opens a non-tokenized URL and `openclaw config get gateway.auth.token` returns the SecretRef object instead of plaintext.

## Best Possible Solution

Land one maintainer-approved dashboard auth recovery path: preferably session/exchange-token auth, or an explicit copy-token command that never logs the bearer and ships with focused tests, docs, and real Control UI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a SecretRef-managed `gateway.auth.token` and missing browser sessionStorage, `openclaw dashboard` copies/opens a non-tokenized URL while `config get` returns config data rather than the plaintext bearer.

## Solution Assessment

Unclear until maintainers choose the security boundary. A session/exchange-token flow is the cleaner durable solution; an explicit copy-token command can be acceptable only with no logging or URL exposure, docs, tests, and real Control UI proof.
