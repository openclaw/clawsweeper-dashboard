## Label Justifications

## Summary

Keep open: current main and v2026.6.9 still leave SecretRef-managed dashboard tokens non-tokenized, `config get` redacts or returns the config object instead of a resolved bearer, and there is no viable merged or open PR that owns the recovery path.

## What This Changes

The issue reports that users with SecretRef-managed `gateway.auth.token` cannot recover a plaintext token through the CLI after losing Control UI session storage.

## Best Possible Solution

Land one maintainer-approved dashboard recovery path, preferably session or exchange-token auth, or a tightly scoped copy-token command with no bearer logging, focused tests, docs, and redacted real Control UI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a resolved SecretRef-managed `gateway.auth.token`, `openclaw dashboard` copies/opens a non-tokenized URL, while `openclaw config get gateway.auth.token` reads the redacted config rather than returning the bearer.

## Solution Assessment

Unclear until maintainers choose the security boundary. A copy-token command is a plausible interim fix, but the closed linked PR was not sufficient; a session or exchange-token handoff may be the cleaner long-term solution.
