## Label Justifications

- `P1`: A default-onboarding SecretRef token can leave users unable to authenticate to the primary dashboard after browser session storage is lost.
- `impact:auth-provider`: The failure is in gateway shared-secret auth and SecretRef resolution/retrieval for the dashboard.
- `impact:security`: The fix must preserve the gateway token secrecy boundary while making recovery operable.

## Summary

Keep open: current main and latest v2026.6.6 still suppress tokenized dashboard URLs for SecretRef-managed gateway tokens, while the documented/config/UI recovery paths do not provide a supported plaintext or secure handoff path for the local file-provider token.

## What This Changes

The issue reports that SecretRef-managed gateway dashboard tokens cannot be retrieved as plaintext through the documented CLI path after Control UI session storage is lost.

## Best Possible Solution

Add a maintainer-approved secure recovery path for SecretRef-managed dashboard auth, ideally session or exchange-token based; if an interim path is needed, make it copy-only or otherwise non-logging and update dashboard docs, troubleshooting, onboarding copy, UI hints, and focused tests together.

## Reproduction Assessment

Yes. Source inspection shows a resolved SecretRef token still sets `secretRefConfigured: true`, `openclaw dashboard` therefore opens/copies a non-tokenized URL, and the current docs/UI guidance points to paths that do not produce plaintext for that case.

## Solution Assessment

No. The report’s diagnosis is sound, but the best fix needs a maintainer security/UX decision; session or exchange-token auth is cleaner than printing plaintext, while an interim copy-only command or local-provider-specific hint would need careful non-logging proof.
