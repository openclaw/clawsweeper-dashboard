## Summary

Keep this issue open: current `main` and the latest release still lack an operable SecretRef-managed dashboard token recovery path, and the open linked PR is only a candidate fix, not a merged resolution.

## What This Changes

The issue asks for a secure, documented way to authenticate the Control UI when `gateway.auth.token` is SecretRef-managed and `openclaw config get gateway.auth.token` returns the reference object instead of plaintext.

## Best Possible Solution

Land one maintainer-approved recovery path: preferably session/exchange-token dashboard auth, or a tested copy-only CLI path that never logs the bearer and updates the dashboard docs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a SecretRef-managed `gateway.auth.token` resolves with `secretRefConfigured: true`, `openclaw dashboard` copies/opens a non-tokenized URL, and `config get` reads the redacted config value rather than resolving plaintext. I did not run a live CLI/browser repro in this read-only review.

## Solution Assessment

No final best fix is selected yet. The issue is valid, but maintainers need to choose between session/exchange-token auth and an explicit guarded token-copy command before treating the linked PR as the right long-term solution.
