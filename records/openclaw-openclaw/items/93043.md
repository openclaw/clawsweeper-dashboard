## Summary

Keep open. Current main and v2026.6.8 still suppress SecretRef-managed dashboard token delivery and do not provide a plaintext recovery path, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks for an operable way to authenticate the Control UI when `gateway.auth.token` is SecretRef-managed and `openclaw config get gateway.auth.token` returns only the SecretRef object.

## Best Possible Solution

Keep this issue open until maintainers land one secure dashboard recovery path and close it through the linked PR or a narrower replacement.

## Reproduction Assessment

Yes. Source inspection shows SecretRef-managed gateway tokens remain non-tokenized in `openclaw dashboard`, and the documented config getter still returns configuration rather than a plaintext SecretRef value; I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

Unclear. A `--print-token` or clipboard-copy path could be an acceptable interim fix, but the best long-term solution may be session or exchange-token dashboard auth that avoids exposing the shared bearer.
