## Summary

Keep open: current main and the latest release still keep SecretRef-managed dashboard tokens out of URLs and do not provide a merged CLI/session recovery path; the linked copy-token PR is closed unmerged after security/proof concerns.

## What This Changes

The issue asks for an operable, secure recovery path when a SecretRef-managed gateway token is needed for Control UI login after browser sessionStorage is lost.

## Best Possible Solution

Keep this issue open for a maintainer-approved dashboard auth recovery path, preferably exchange/session auth or a tested and documented explicit copy-token mode that never logs the bearer.

## Reproduction Assessment

Yes, at source level: configure `gateway.auth.token` as a resolvable SecretRef, clear Control UI sessionStorage, and `openclaw dashboard` will copy/open only the clean dashboard URL while `config get` returns the ref-shaped config value. I did not run a live dashboard flow because this review is read-only and the source path is already deterministic.

## Solution Assessment

Unclear. The issue correctly identifies that `config get` is not the right plaintext recovery path, but the permanent fix needs a maintainer security decision between session/exchange-token auth and an explicit opt-in bearer copy flow.
