## Summary

Keep open. Current main and the latest release still leave SecretRef-managed dashboard tokens without an operable recovery path after browser sessionStorage is lost, and the linked open fix PR is the active implementation candidate rather than a reason to close this issue.

## What This Changes

The issue asks for a secure, documented way to authenticate the Control UI when `gateway.auth.token` is managed by SecretRef and `config get` only returns the reference object.

## Best Possible Solution

Keep this issue open until maintainers land one approved recovery path: preferably session/exchange-token dashboard auth, or a tested copy-token command that never logs the bearer and updates the dashboard docs.

## Reproduction Assessment

Yes at source level. Configure `gateway.auth.token` as a resolved SecretRef, clear Control UI sessionStorage, and `openclaw dashboard` will copy/open a clean URL while `openclaw config get gateway.auth.token` reads the stored config value rather than resolving the secret plaintext.

## Solution Assessment

Unclear until maintainers choose the security boundary. Session/exchange-token auth is the cleaner end state, while an interim `--print-token` command can be acceptable only if it is opt-in, tested, documented, and proven not to leak the bearer.
