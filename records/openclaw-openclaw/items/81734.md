## Summary

Keep open. Current main still has the reported native-first Codex state: Pi-shaped workspace tools are filtered from the Codex dynamic surface, the old compatibility profile is rejected/removed, and the fallback path does not appear to emit an operator-visible runtime/tool-surface warning. The related QA work adds detection coverage but is not a product fix.

## What This Changes

The issue asks OpenClaw to prevent, opt out of, or visibly signal Pi-to-Codex fallback/tool-surface changes after the Codex dynamic-tools compatibility profile was removed.

## Best Possible Solution

Make cross-runtime fallback visible and intentional with an operator-facing runtime/tool-surface warning or validation plus Codex prompt/tool-vocabulary guidance, and only reintroduce duplicate dynamic tools if maintainers explicitly choose that compatibility policy.

## Reproduction Assessment

Yes from source inspection and related live reports, but not by a live run in this review. Current main strips the named Pi-shaped tools under Codex and fallback candidates can cross into different runtime behavior without a dedicated runtime-boundary signal.

## Solution Assessment

Unclear as a final product fix. Reinstating the old openclaw-compat profile would reverse the native-first Codex policy; the narrower maintainable direction appears to be explicit runtime-boundary signaling plus Codex prompt/tool-vocabulary guidance, validated by the related canary.
