## Summary

Keep open: the branch targets a valid ACP stdout-cleanliness bug, but the current diff still regresses documented ACP client invocations and does not make the standalone server entrypoint's forced-stderr flag effective for console writes.

## What This Changes

The PR adds ACP server and CLI log-to-stderr routing plus startup regression tests for stdout-safe ACP JSON transport.

## Best Possible Solution

Land a focused implementation that routes captured console logs to stderr before CLI registration only for ACP bridge invocations, leaves every `acp client` stdout path intact, and makes direct ACP server runs stdout-safe.

## Reproduction Assessment

Yes. The current-main CLI gap is source-reproducible by tracing `runCli()` from `enableConsoleCapture()` through command/plugin registration before any ACP-specific forced-stderr call, and the PR regression is source-reproducible with `openclaw acp client --server-args --url ...` through `getCommandPositionalsWithRootOptions()`.

## Solution Assessment

No. Early ACP routing is the right direction, but the detector must recognize real `acp client` mode before consuming client/server-args options, and the standalone server path must enable or require console capture before relying on forced stderr routing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve acp client server-args on stdout:** `src/cli/run-main.ts:157`
  - body: The new client detector treats `--server-args` as a normal single-value flag. The documented `openclaw acp client --server-args --url ...` form passes an option-shaped value, so the helper returns `null`, `isAcpClient` stays false, and interactive client `console.log` output is redirected to stderr.
  - confidence: 0.86
- **[P2] Enable capture for the standalone ACP server:** `src/acp/server.ts:18`
  - body: `routeLogsToStderr()` only flips the forced-stderr flag; ordinary `console.*` output is redirected only after console capture is enabled. The direct server entrypoint calls `serveAcpGateway()` without capture setup, so direct ACP server runs can still put console output on stdout.
  - confidence: 0.78
