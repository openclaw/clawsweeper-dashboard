## Summary

Keep open: this targets a real ACP stdout hygiene bug, but the current branch still has source-backed correctness blockers and lacks after-fix real behavior proof.

## What This Changes

The PR routes ACP bridge console logging to stderr in `serveAcpGateway()` and early CLI startup, with unit coverage for ACP log routing.

## Best Possible Solution

Land a refreshed ACP/CLI fix that routes bridge diagnostics to stderr before stdout-sensitive startup work, preserves interactive client stdout including variadic `--server-args`, and installs console capture for direct server entrypoints before relying on stderr routing.

## Reproduction Assessment

Yes, source-reproducible. The documented `openclaw acp client --server-args --url ...` path flows through the PR-head detector with `isAcpClient` false, and the direct server entrypoint sets the stderr flag without installing console capture.

## Solution Assessment

No. Early stderr routing is directionally right, but this branch still mishandles variadic ACP client arguments and does not make the direct server entrypoint install console capture before relying on the stderr flag.
