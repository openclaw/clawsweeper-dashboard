## Summary

Keep open. Current main already has the server-side ACP stderr guard, but the PR still targets an unmerged CLI startup window before command/plugin registration, and the proposed detector would regress documented interactive `acp client --server-args` usage.

## What This Changes

The PR routes ACP bridge logs to stderr in server and CLI startup paths and adds ACP server/CLI regression tests.

## Best Possible Solution

Land a focused current-main implementation that routes captured console logs to stderr only for real ACP bridge invocations before command/plugin registration, preserves every interactive `acp client` stdout path, and handles the standalone server entrypoint explicitly.

## Reproduction Assessment

Yes. The PR regression is source-reproducible by tracing `openclaw acp client --server-args --url ...` through the proposed detector, and the current-main CLI gap is visible by inspecting `runCli()` before command registration.

## Solution Assessment

No. The CLI guard is the right direction, but this PR's parser is not robust for documented client invocations and the direct server entrypoint still does not make the forced-stderr flag effective for console writes.
