## Summary

Keep open: the PR is aimed at a real ACP stdout hygiene bug, but the current branch still has merge-blocking behavior gaps and lacks contributor real behavior proof.

## What This Changes

The PR routes ACP bridge logging to stderr in `serveAcpGateway()` and early CLI startup, with startup and CLI tests for ACP stderr routing and client exceptions.

## Best Possible Solution

Land a refreshed ACP/CLI fix that routes bridge diagnostics to stderr before startup side effects, preserves interactive `acp client` stdout including variadic server args, and installs console capture before relying on stderr routing.

## Reproduction Assessment

Yes, by source inspection: PR-head `--server-args` parsing leaves option-shaped server args unconsumed, and direct `serveAcpGateway()` sets the stderr-routing flag without installing console capture.

## Solution Assessment

No. Early stderr routing is the right direction, but this branch needs a client-aware variadic parser and direct-entrypoint capture setup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve ACP client server args on stdout:** `src/cli/run-main.ts:159`
  - body: `--server-args` is variadic, but this detector treats it as a single value flag. For `openclaw acp client --server-args --url ...`, `--url` is not consumed as a server arg, `isAcpClient` stays false, and interactive client `console.log` output is redirected to stderr.
  - confidence: 0.86

  - body: `routeLogsToStderr()` only flips the flag consumed by the patched console wrapper. The standalone server path calls `serveAcpGateway()` directly without `enableConsoleCapture()`, so direct runs can still leave normal console output on stdout.
  - confidence: 0.78
