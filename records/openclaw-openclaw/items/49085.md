## Summary

Keep open. Current main already shipped the server-side ACP stdout protection, but this PR also contains still-unmerged CLI startup protection before command registration, and the proposed client-mode detector still has a concrete regression for documented `acp client --server-args` usage.

## What This Changes

The PR routes ACP server logs to stderr, adds an ACP-specific `runCli()` stderr-routing guard before command registration, and adds startup/CLI tests for bridge versus client stdout behavior.

## Best Possible Solution

Land a focused current-main implementation that routes logs to stderr only for the ACP bridge before command/plugin registration, preserves all interactive `acp client` stdout cases, and either fixes or clearly scopes the standalone server entrypoint behavior.

## Reproduction Assessment

Yes. The PR regression is reproducible by tracing the documented `openclaw acp client --server-args --url ...` path through the proposed detector; the current-main gap is reproducible by static inspection of `runCli()` showing no ACP stderr route before `buildProgram()` and command registration.

## Solution Assessment

No. The server-side change is already shipped and the CLI guard is the right direction, but the proposed parser is not robust enough for documented client invocations and the direct server entrypoint still needs a clearer capture strategy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve acp client server-args on stdout:** `src/cli/run-main.ts:157`
  - body: The new client detector treats `--server-args` as a normal single-value flag. The documented `openclaw acp client --server-args --url ...` form passes an option-shaped value, which `getCommandPositionalsWithRootOptions` rejects via `isValueToken`; `isAcpClient` then stays false and `routeLogsToStderr()` runs for interactive client mode, moving client `console.log` updates to stderr.
  - confidence: 0.86
- **[P2] Enable capture for the standalone ACP entrypoint:** `src/acp/server.ts:18`
  - body: `routeLogsToStderr()` only flips the force-stderr flag; ordinary `console.log` and `console.info` are redirected only after `enableConsoleCapture()` patches the console. The direct `src/acp/server.ts` main path calls `serveAcpGateway()` without that capture setup, so direct ACP server runs can still put console output on stdout despite this new call.
  - confidence: 0.78
