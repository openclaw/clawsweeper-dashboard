## Summary

Keep open: current main independently has the server-side stderr-routing portion, but this PR is still not mergeable because it lacks real behavior proof and has source-reproducible P2 blockers in ACP client detection and standalone server capture.

## What This Changes

The PR routes ACP bridge logs to stderr in `serveAcpGateway` and CLI startup, with regression tests for ACP stdout log routing.

## Best Possible Solution

Land a fresh ACP/CLI fix that routes bridge diagnostics to stderr before stdout-sensitive startup work, preserves all `acp client` stdout behavior including variadic `--server-args`, and installs console capture for direct server runs before relying on stderr routing.

## Reproduction Assessment

Yes, source-reproducible. The PR regression follows from `openclaw acp client --server-args --url ...` through the branch detector, and the standalone server gap follows from `routeLogsToStderr()` requiring console capture before it can redirect `console.*`.

## Solution Assessment

No. Early stderr routing is the right direction, but this branch still mishandles variadic ACP client arguments and does not make the direct server entrypoint install console capture before relying on the stderr flag.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

  - confidence: 0.78
