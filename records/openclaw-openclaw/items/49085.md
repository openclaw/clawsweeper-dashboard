## Summary

Keep open: this is a valid ACP stdout-hygiene fix attempt, but the current PR is not mergeable as-is because it still has source-backed ACP client stdout regressions, a standalone server capture gap, and no after-fix real behavior proof. Current main already contains the server-side `serveAcpGateway()` stderr-routing portion, but not the full CLI startup protection this PR is trying to add.

## What This Changes

The PR routes ACP bridge diagnostics to stderr in `serveAcpGateway()` and early CLI startup, with unit coverage for ACP stdout log routing.

## Best Possible Solution

Land a fresh ACP/CLI fix that routes bridge diagnostics to stderr before stdout-sensitive startup work, preserves all `acp client` stdout behavior including variadic `--server-args`, and installs console capture for direct server runs before relying on the stderr flag.

## Reproduction Assessment

Yes, source-reproducible. `openclaw acp client --server-args --url ...` flows through the PR detector with `isAcpClient` false, and direct `src/acp/server.ts` runs set the stderr flag without installing console capture.

## Solution Assessment

No. Early stderr routing is the right direction, but this branch still mishandles variadic ACP client arguments and does not make the direct server entrypoint install console capture before relying on the stderr flag.

## Review Findings

Overall correctness: patch is incorrect

  - confidence: 0.78
