## Summary

Keep this PR open for normal review rather than cleanup: current main has already shipped the server-side part of the ACP stdout protection, but the PR branch still has unresolved correctness findings around `acp client` detection and direct server capture, plus no contributor-provided real behavior proof.

## What This Changes

The PR routes ACP bridge logging to stderr in server and CLI startup paths and adds regression tests for ACP stdout JSON hygiene.

## Best Possible Solution

Land a focused ACP/CLI fix that preserves `acp client` stdout behavior, routes bridge diagnostics to stderr before any stdout-sensitive startup work, and installs console capture for direct ACP server runs before relying on forced stderr routing.

## Reproduction Assessment

Yes, source-reproducible. The PR regression follows from `openclaw acp client --server-args --url ...` through the PR detector, and the direct server gap follows from `routeLogsToStderr()` requiring console capture before it can redirect `console.*`.

## Solution Assessment

No. Early stderr routing is the right direction, but this branch still mishandles variadic `acp client` arguments and does not make the direct server entrypoint install console capture before relying on the stderr flag.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve acp client server args on stdout:** `src/cli/run-main.ts:159`
  - body: The detector treats `--server-args` as a normal single-value flag, but `openclaw acp client --server-args --url ...` passes an option-shaped value. That makes `getCommandPositionalsWithRootOptions()` return null, so `isAcpClient` stays false and the interactive client's `console.log` output is redirected to stderr.
  - confidence: 0.86

  - confidence: 0.78
