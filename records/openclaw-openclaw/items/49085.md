## Summary

Keep this PR open, but it is not merge-ready: current main already covers the standalone server-side stderr flag, while the PR still has correctness gaps in ACP client detection and direct-entrypoint console capture, plus missing real behavior proof.

## What This Changes

The PR routes ACP bridge logging to stderr in `serveAcpGateway()` and early `runCli` startup, with regression tests for bridge/client routing.

## Best Possible Solution

Land a refreshed ACP/CLI fix that routes bridge diagnostics to stderr before startup side effects, preserves interactive `acp client` stdout including variadic `--server-args`, and installs console capture before relying on stderr routing.

## Reproduction Assessment

Yes, by source inspection rather than execution: PR-head ACP client detection misclassifies `--server-args --url ...`, and the direct `serveAcpGateway()` entrypoint sets the stderr flag without installing console capture.

## Solution Assessment

No. Early stderr routing is the right direction, but this branch needs client-aware variadic parsing and direct-entrypoint capture setup before merge.
