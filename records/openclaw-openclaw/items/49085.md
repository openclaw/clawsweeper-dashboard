## Summary

Keep open. Current main has the server-side ACP stderr routing, but this PR still targets an unmerged CLI startup gap before command/plugin registration, and the proposed PR parser regresses documented interactive `acp client --server-args` usage.

## What This Changes

The PR routes ACP bridge logs to stderr, adds an ACP-specific `runCli()` pre-registration stderr guard, and adds ACP server/CLI regression tests.

## Best Possible Solution

Land a focused current-main implementation that routes captured console logs to stderr only for real ACP bridge invocations before command/plugin registration, preserves every interactive `acp client` stdout path, and explicitly handles or documents the standalone server entrypoint behavior.

## Reproduction Assessment

Yes. The PR regression is reproducible by tracing `openclaw acp client --server-args --url ...` through the proposed detector, and the current-main gap is reproducible by static inspection of `runCli()` before command registration.

## Solution Assessment

No. The server-side change is already present, and the CLI guard is the right direction, but this PR's parser is not robust enough for documented client invocations and the direct server entrypoint still lacks console capture.
