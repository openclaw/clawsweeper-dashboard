## Summary

Keep open: this PR targets a real ACP stdout-cleanliness bug, and current main only covers the server-side stderr flag/test; the PR branch still has source-backed P2 regressions and lacks contributor real behavior proof.

## What This Changes

This PR routes ACP bridge console logging to stderr from `serveAcpGateway()` and early CLI startup, with startup and CLI tests for ACP stderr routing and client exceptions.

## Best Possible Solution

Land a refreshed current-main ACP/CLI fix that protects bridge stdout before startup diagnostics, preserves interactive client stdout including variadic `--server-args`, and enables console capture wherever stderr routing is relied on.

## Reproduction Assessment

Yes, by source inspection rather than a live run: PR head misclassifies `openclaw acp client --server-args --url ...` and the direct `serveAcpGateway()` entrypoint sets the stderr flag without console capture.

## Solution Assessment

No. Early stderr routing is the right direction, but this branch needs a parser that preserves ACP client variadic args and direct-entrypoint capture before it can land.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve ACP client server args on stdout:** `src/cli/run-main.ts:159`
  - body: `--server-args` is variadic, but this detector treats it as a single value flag. For `openclaw acp client --server-args --url ...`, the option-shaped server arg is not consumed, `isAcpClient` stays false, and interactive client console output is redirected to stderr.
  - confidence: 0.86
- **[P2] Enable capture before standalone ACP stderr routing:** `src/acp/server.ts:18`
  - body: `routeLogsToStderr()` only flips the flag consumed by the patched console wrapper. The standalone server path calls `serveAcpGateway()` directly without `enableConsoleCapture()`, so direct runs can still leave normal console output on stdout.
  - confidence: 0.78
