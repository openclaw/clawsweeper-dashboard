## Summary

Keep open. The PR targets a real ACP stdout hygiene bug and current main already contains the server-side stderr-routing piece, but this branch is still not correct as-is: it can redirect supported `acp client --server-args --url ...` output to stderr, the standalone server path still lacks console capture, and external real behavior proof is missing.

## What This Changes

The PR routes ACP bridge logging to stderr in server and CLI startup paths and adds ACP startup/CLI regression tests.

## Best Possible Solution

Land or replace this with a focused ACP/CLI fix that installs console capture for direct ACP server runs, routes bridge diagnostics before stdout-sensitive startup, and preserves interactive `acp client` stdout behavior.

## Reproduction Assessment

Yes. The PR regression is source-reproducible through `openclaw acp client --server-args --url ...`, and the direct server gap follows from `routeLogsToStderr()` requiring console capture before it can affect `console.*` output.

## Solution Assessment

No. Routing ACP bridge diagnostics to stderr is the right direction, but this branch mishandles variadic client server arguments and does not make the direct server entrypoint install console capture before relying on the stderr flag.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve acp client server args on stdout:** `src/cli/run-main.ts:159`
  - body: `--server-args` is variadic and docs show it receiving option-shaped values like `--url`. Treating it as a normal value flag makes `getCommandPositionalsWithRootOptions()` return null for that supported client invocation, so `isAcpClient` stays false and the interactive client's `console.log` output is redirected to stderr.
  - confidence: 0.86
- **[P2] Enable capture for the standalone ACP server:** `src/acp/server.ts:18`
  - body: `routeLogsToStderr()` only flips the flag consumed by patched console forwarding; it does not patch `console.*` itself. The standalone module path calls `serveAcpGateway()` directly without `enableConsoleCapture()`, so direct ACP server runs can still write console output to stdout.
  - confidence: 0.78
