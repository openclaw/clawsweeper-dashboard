## Summary

Keep open: this PR targets a real linked cron delivery-loss bug, but the patch is not merge-ready because it bypasses the fatal structured-error guard after the helper has already selected fatal error payloads, and the PR still has no after-fix real cron proof.

## What This Changes

The PR adds a finalization-side check in `src/cron/isolated-agent/run.ts` so fatal structured-error delivery suppression is skipped when a later non-error text payload follows the last error payload.

## Best Possible Solution

Move the recovered-output rule into `resolveCronPayloadOutcome` before fatal payload selection, preserve the fatal structured-error suppression path, add focused direct-delivery regression coverage, and require redacted real cron proof before merge.

## Reproduction Assessment

Yes, at source level: the linked issue gives a reduced scheduled isolated cron repro, and current main shows the helper can leave a recovered Feishu/direct-delivery run fatal before `dispatchCronDelivery` is reached. I did not run the live WSL2/Feishu setup in this read-only review.

## Solution Assessment

No. The proposed check is in the wrong layer because it runs after fatal classification and fatal delivery payload selection; the maintainable fix is to classify recovered output correctly in the helper before finalization gates dispatch.
