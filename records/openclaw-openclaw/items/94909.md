## Summary

Keep open: this PR targets a real cron delivery bug, but the patch is not merge-ready because the recovery decision runs after payload classification has already selected fatal state and fatal delivery payloads, and the external PR lacks real behavior proof.

## What This Changes

The PR changes isolated cron finalization to bypass the fatal structured-error no-dispatch branch when later non-error payload text appears after the last error payload.

## Best Possible Solution

Move the recovered-output rule into `resolveCronPayloadOutcome` so it clears fatal state and selects the successful delivery payload before finalization decides dispatch, then cover the direct-delivery `finalAssistantVisibleText` case with real cron proof.

## Reproduction Assessment

Yes at source level: the linked issue documents the scheduled isolated-agent state, and current `resolveCronPayloadOutcome` plus `finalizeCronRun` show the pre-dispatch fatal branch can still block delivery for direct-delivery recovered warning output. I did not run the live WSL2/Feishu scheduled setup.

## Solution Assessment

No. The proposed check is in the wrong layer: it runs after the helper has already chosen fatal state and fatal delivery payloads, while the maintainable fix is to classify recovered output correctly in the helper before finalization.
