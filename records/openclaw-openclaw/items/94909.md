## Summary

Keep open: the PR targets a real cron delivery bug, but this patch is in the wrong layer and can bypass the fatal structured-error suppression that currently prevents error payloads from being announced as normal completion output.

## What This Changes

The PR adds a finalization-side check that skips the fatal structured-error pre-dispatch branch when a non-error text payload appears after the last error payload.

## Best Possible Solution

Move the recovered-output decision into `resolveCronPayloadOutcome`, keep genuine fatal structured errors on the non-dispatch path, add direct-delivery regression coverage, and require redacted real cron proof.

## Reproduction Assessment

Yes at source level: the linked issue documents the scheduled isolated-agent state, and current `resolveCronPayloadOutcome` plus `finalizeCronRun` show the pre-dispatch fatal branch can still block delivery for direct-delivery recovered warning output. I did not run the live WSL2/Feishu scheduled setup.

## Solution Assessment

No. The proposed check is in the wrong layer because it runs after the helper has already chosen fatal state and fatal delivery payloads; the safer fix is to classify recovery and select recovered output in the helper before finalization.
