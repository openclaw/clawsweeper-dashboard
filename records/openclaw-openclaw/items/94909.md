## Summary

Keep open: this PR targets a real recovered cron delivery bug, but the patch changes finalization after the payload resolver has already marked the run fatal and selected fatal error text for delivery. It also still lacks after-fix real cron or delivery proof.

## What This Changes

The PR changes `src/cron/isolated-agent/run.ts` so cron delivery is attempted when a later non-error text payload appears after an earlier error payload.

## Best Possible Solution

Move recovery detection into `resolveCronPayloadOutcome` before fatal state and delivery payloads are selected, preserve true fatal structured-error suppression, add focused regression coverage, and require redacted scheduled-run or delivery proof before merge.

## Reproduction Assessment

Yes at source level: the linked issue documents a reduced scheduled run, and current main still has the pre-dispatch fatal structured-payload return after resolver classification. I did not run the live WSL2/Feishu scheduled setup in this read-only review.

## Solution Assessment

No. The bug should be solved in `resolveCronPayloadOutcome` before fatal state and delivery payloads are selected; this PR bypasses the later guard after the helper has already chosen the fatal error payload.
