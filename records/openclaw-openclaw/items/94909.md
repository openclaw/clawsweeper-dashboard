## Summary

Keep open: this PR targets a valid cron delivery bug, but the proposed finalization bypass runs after the resolver has already selected fatal status and fatal delivery payloads, so it can send the very error text the existing guard suppresses.

## What This Changes

The PR adds a post-resolution check in `src/cron/isolated-agent/run.ts` that bypasses the fatal structured-error delivery skip when a later non-error payload follows an error payload.

## Best Possible Solution

Move recovery into `resolveCronPayloadOutcome` before fatal state and delivery payloads are selected, preserve true fatal structured-error suppression, add focused regression coverage, and require redacted scheduled-run proof before merge.

## Reproduction Assessment

Yes at source level: the linked canonical issue documents a reduced scheduled run, and current main still has the resolver/finalization path that classifies recovered output as fatal before dispatch. I did not run the live WSL2/Feishu scheduled setup in this read-only review.

## Solution Assessment

No. The bug should be solved in `resolveCronPayloadOutcome` before fatal state and delivery payloads are selected; this PR bypasses the later guard after the helper has already chosen the fatal error payload.
