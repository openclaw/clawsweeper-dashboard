## Summary

Keep open: the trajectory cap fix is useful, but the current patch can lose runtime capture rows when multiple recorders for the same session flush from stale in-memory snapshots, and the PR still needs real behavior proof before merge.

## What This Changes

This PR replaces stop-on-full runtime trajectory capture with a capped JSONL window that atomically rewrites the runtime sidecar and adds rotation and directory-permission regression coverage.

## Best Possible Solution

Land a bounded-window trajectory writer only after it preserves events across overlapping recorders for the same session and includes focused regression coverage plus real run/export proof.

## Reproduction Assessment

Yes, from source inspection rather than execution in this read-only review. A focused repro can create two recorders for the same session, flush one so the map entry is deleted, write and flush a newer recorder, then flush the older writer and observe newer rows disappear.

## Solution Assessment

No. A rolling window is a plausible fix, but replacing the whole sidecar from per-recorder stale state is not safe for OpenClaw's existing overlapping fallback and attempt recorder path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid replacing the sidecar from stale recorder state:** `src/trajectory/runtime.ts:317`
  - body: The new writer snapshots `lines` into `content` and later replaces the whole sidecar. OpenClaw already has overlapping recorders for one session: `agent-command.ts` keeps a fallback recorder alive while each embedded attempt creates and flushes its own recorder for the same `sessionFile`; those flushes remove the shared writer from the cache, so the older fallback writer can later flush a stale snapshot and overwrite newer attempt events. Keep one canonical writer/refcount or reload and merge before replacement, and add a regression for that fallback-plus-attempt path.
  - confidence: 0.86
