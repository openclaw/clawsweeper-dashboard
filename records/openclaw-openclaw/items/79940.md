## Summary

Keep open: this PR targets a real cron run-history metadata leak, but the patch still trusts a caller-supplied `agentId` for a read-scoped `cron.runs` request, so the name leak remains reachable for guessed or known non-default agents.

## What This Changes

The PR adds optional `agentId` to `cron.runs`, uses paginated `cron.listPage()` for all-scope job-name enrichment, adds a focused handler test, updates the validator test, and adds a changelog entry.

## Best Possible Solution

Derive the visible job-name set from authenticated caller context, allow an explicit admin override only when authorized, and add a cross-agent regression proving hidden job names are neither returned nor searchable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main builds all-scope job names from an unfiltered cron list, and the run-log reader attaches those names to returned entries and query matching.

## Solution Assessment

No. Reusing `listPage()` is the right primitive, but honoring `params.agentId` directly is not the best fix for an access-control bug; the scope should come from caller authorization or an admin-only override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not trust agentId for cron run-name lookup:** `src/gateway/server-methods/cron.ts:531`
  - body: This still lets any read-scoped `cron.runs` caller choose the agent used for all-scope job-name enrichment. Because `cron.runs` is exposed under `operator.read` and `listPage()` only filters by the requested value, a caller can set `agentId` to another known agent and have that agent's job names attached or searched in global run history. Please derive the allowed agent from authenticated caller context or gate explicit overrides on admin authorization.
  - confidence: 0.9
