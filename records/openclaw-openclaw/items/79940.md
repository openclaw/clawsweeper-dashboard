## Summary

Keep open: the PR is a focused fix for a real cron run-history metadata leak, but the proposed access-control boundary still trusts a request-supplied `agentId`, so a read-capable caller can ask for another known agent's job-name enrichment.

## What This Changes

The PR adds optional `agentId` support to `cron.runs`, switches all-scope job-name enrichment from `cron.list()` to paginated `cron.listPage()`, adds regression coverage, refreshes Swift protocol models, and records a changelog fix.

## Best Possible Solution

Derive the job-name lookup scope from trusted caller/session context, with an explicit admin override if needed, and add a regression proving one read-scoped caller cannot enrich or search another agent's cron job name.

## Reproduction Assessment

Yes. Source inspection reproduces the current-main leak via unfiltered `context.cron.list()` and the PR's residual issue via a caller-controlled `agentId` on a read-scoped `cron.runs` request.

## Solution Assessment

No. `listPage()` is the right primitive, but using an untrusted request `agentId` is not the best access-control fix; the scope should come from authenticated/session context or an explicitly authorized admin path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive the cron run-name scope from the caller:** `src/gateway/server-methods/cron.ts:531`
  - body: The new `agentId` is trusted directly when building the all-scope `jobNameById` map. Because `cron.runs` is readable with `operator.read` and the schema accepts any non-empty string, a caller can request another known agent id and still get that agent's job names attached and searchable in run history. Derive this scope from the authenticated/session caller, or require an admin-checked override, instead of taking it from the request.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff touches an access-control-sensitive metadata leak and still leaves job-name visibility controlled by an untrusted request field.

Concerns:

- **[medium] Request-selected agentId can expose cron job names:** `src/gateway/server-methods/cron.ts:531`
  - body: The patch narrows the enrichment map only to the `agentId` supplied in the request. Since `cron.runs` is read-scoped, that does not prove the caller owns or may view that agent, so cross-agent job-name disclosure remains possible for known or guessable agent ids.
