## Summary

Keep open. Current main and v2026.6.10 still lack the requested Redis/BullMQ-style distributed job orchestration, shared worker claiming, dead-letter semantics, queue observability, and horizontal worker coordination; related open issues cover narrower cron, session-storage, runtime-split, and subagent-placement slices rather than superseding this broader architecture input.

## What This Changes

The issue proposes adding a Redis/BullMQ-style distributed job orchestration layer with separate workers, retries, delayed and scheduled jobs, dead-letter handling, and observability for horizontally scaled OpenClaw deployments.

## Best Possible Solution

Keep this issue open as the broad horizontal job-orchestration architecture tracker until maintainers decide whether OpenClaw should add a backend-neutral distributed scheduler/job seam, fold pieces into related cron/session/runtime trackers, or document the ScalyClaw-style Redis/BullMQ approach as an external/plugin pattern.

## Reproduction Assessment

Yes for source confirmation, not as a bug reproduction: current main uses in-process command lanes, Gateway-local cron execution, and SQLite-backed local ledgers, so the absence of a distributed Redis/BullMQ-style worker queue is directly visible in code and docs.

## Solution Assessment

Unclear as a direct implementation: Redis/BullMQ is useful prior art, but the best OpenClaw path needs maintainer direction on backend-neutral API shape, core versus plugin ownership, config surface, credentials, failure semantics, and upgrade behavior.
