- `impact:session-state`: The proposal is about coordinating stateful background, task, session, and worker activity across multiple OpenClaw instances.

## Summary

Keep open for maintainer architecture review: current main still uses process-local queues, Gateway-local cron state, and local task/flow ledgers, while this proposal asks for distributed queue-backed worker coordination. The linked ScalyClaw Redis/BullMQ prototype and older closed BullMQ PR are useful prior art, but related open issues only cover narrower cron, session-storage, and Gateway-federation slices.

## What This Changes

The issue proposes adding Redis/BullMQ-style horizontal scaling with centralized job queues, worker processes, retries/backoff, delayed and cron jobs, dead-letter handling, and queue observability.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, external session storage, Gateway federation, config/schema/docs, validation, and the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, release-tag, related-issue, and prototype inspection give high confidence that OpenClaw lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
