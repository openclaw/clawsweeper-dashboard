## Summary

Keep open for maintainer architecture review: current main still uses process-local queues, Gateway-local cron scheduling, file-backed cron state, and local task ledgers, so the requested distributed Redis/BullMQ-style orchestration and horizontal worker coordination layer is not implemented.

## What This Changes

The issue proposes adding a Redis/BullMQ-style distributed job orchestration layer with separate workers, retries/backoff, delayed and scheduled jobs, dead-letter handling, observability, and horizontal scaling support.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, while coordinating the cron HA, session storage, and Gateway federation slices already tracked separately.

## Reproduction Assessment

Not applicable; this is a feature and architecture proposal rather than a reproducible defect. Source and docs inspection gives high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art from the linked ScalyClaw prototype, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one concrete backend.
