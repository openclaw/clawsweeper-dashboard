## Summary

Keep open for maintainer architecture review: current main and the latest release still use in-process queue lanes, Gateway-local cron state, and local task/flow ledgers, while this proposal asks for distributed Redis/BullMQ-style job orchestration and horizontal worker coordination.

## What This Changes

Proposal to add a Redis/BullMQ-backed distributed job orchestration layer with separate workers, retries/backoff, delayed and scheduled jobs, dead-letter handling, and queue observability.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, while keeping Tasks/TaskFlow canonical and coordinating cron HA, external session storage, Gateway federation, worker pairing, config/schema/docs, validation, and the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, release-tag, related-issue, and prototype inspection give high confidence that OpenClaw lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
