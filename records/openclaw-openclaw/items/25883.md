## Summary

Keep open for maintainer architecture review: current main still uses process-local command lanes, Gateway-local cron/state, and local SQLite task ledgers, while this proposal asks for distributed job orchestration and worker coordination. The linked ScalyClaw prototype is useful prior art, but related open issues only cover narrower cron HA, external session storage, and Gateway federation slices rather than a shipped replacement.

## What This Changes

The issue proposes a Redis/BullMQ-style distributed job orchestration layer with separate workers, retries, delayed/cron jobs, dead-letter handling, and observability for horizontally scaled OpenClaw deployments.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, external session storage, Gateway federation, config/schema/docs, validation, and ScalyClaw as prior art.

## Reproduction Assessment

Not applicable; this is a feature and architecture proposal rather than a reproducible defect. Source and docs inspection gives high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
