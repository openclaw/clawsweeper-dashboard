## Summary

Keep open for maintainer architecture review: current main and v2026.5.22 still use process-local queues, Gateway-local cron state, and local task/flow ledgers, while this proposal asks for distributed Redis/BullMQ-style job orchestration and horizontal worker coordination. The ScalyClaw prototype and older closed Redis/BullMQ PRs are useful prior art, but the remaining work is not implemented and the related open issues cover only narrower cron, session-storage, and Gateway-federation slices.

## What This Changes

The issue proposes adding a Redis/BullMQ-style distributed job orchestration layer for background work, delayed jobs, scheduled jobs, retries, observability, and horizontally scaled workers.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, while keeping Tasks/TaskFlow canonical and coordinating cron HA, external session storage, Gateway federation, config/schema/docs, validation, and the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, release-tag, related-issue, and prototype inspection give high confidence that OpenClaw lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
