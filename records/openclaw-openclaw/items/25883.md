## Summary

Keep open for architecture review: current main still uses process-local command lanes, Gateway-local cron/state, local task ledgers, and isolated multi-Gateway guidance, while this proposal asks for distributed Redis/BullMQ-style worker coordination. The ScalyClaw prototype is useful prior art, and the related open issues cover narrower cron, session-storage, and Gateway-federation slices rather than superseding this broader scaling request.

## What This Changes

The issue proposes adding Redis/BullMQ-style distributed job orchestration, worker processes, retries, dead-letter handling, scheduling, and visibility for OpenClaw horizontal scaling.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, external session storage, Gateway federation, config/schema/docs, validation, and the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, related-issue, and prototype inspection give high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
