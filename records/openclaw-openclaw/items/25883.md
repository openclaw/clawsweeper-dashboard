## Summary

Keep open for maintainer architecture review. Current main still uses in-process queues, Gateway-local cron scheduling/state, and local task persistence while the issue asks for a distributed Redis/BullMQ-style orchestration layer; related comments and open issues treat this as broad horizontal-scaling input, not cleanup to close.

## What This Changes

The issue proposes a Redis/BullMQ-style horizontal scaling architecture with shared queues, dedicated workers, delayed and scheduled jobs, retries/backoff, dead-letter handling, and observability.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, session storage, Gateway federation, config, docs, validation, and the ScalyClaw prototype as prior art before implementation.

## Reproduction Assessment

Not applicable; this is a feature and architecture proposal rather than a reproducible defect. Source and docs inspection gives high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
