## Summary

Keep open for maintainer architecture review: current main still uses process-local queues, Gateway-local cron/state, and local task ledgers, while this request asks for distributed Redis/BullMQ-style orchestration and worker coordination. The related open issues track narrower cron, session, and Gateway federation slices rather than superseding this broader horizontal-scaling proposal.

## What This Changes

Request a Redis/BullMQ-style distributed job orchestration layer with worker processes, retries/backoff, delayed and cron jobs, dead-letter handling, and observability for horizontal scaling.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, external session storage, Gateway federation, config/schema/docs, validation, and the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable; this is a feature and architecture proposal rather than a reproducible defect. Source and docs inspection gives high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
