## Summary

Keep open for maintainer architecture review: current main and v2026.5.22 still use in-process command lanes, Gateway-local cron, and local task/flow ledgers, while this proposal asks for distributed Redis/BullMQ-style queueing, worker coordination, retries, dead-letter semantics, and observability. The linked ScalyClaw prototype and older unmerged BullMQ PR are useful prior art, but related open issues cover narrower slices rather than replacing this umbrella.

## What This Changes

The issue proposes Redis/BullMQ-style distributed job orchestration, worker coordination, retries, delayed jobs, cron scheduling, and observability for horizontally scaled OpenClaw deployments.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, while coordinating cron HA, external session storage, Gateway federation, distributed runtime work, config/schema/docs, validation, and the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, release-tag, related-item, and prior PR inspection give high confidence that OpenClaw lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
