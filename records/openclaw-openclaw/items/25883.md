## Summary

Keep open for maintainer architecture review: current main and v2026.5.22 still use in-process command lanes, Gateway-local cron files/state, and local task/flow ledgers, while this proposal asks for distributed Redis/BullMQ-style job orchestration and horizontal worker coordination.

## What This Changes

The issue proposes adding a Redis/BullMQ-style distributed job orchestration layer with separate workers, retries/backoff, delayed/cron jobs, dead-letter handling, and queue observability.

## Best Possible Solution

Make a maintainer architecture decision for a generic or plugin-capable distributed scheduler/job backend while coordinating the narrower cron HA, external session storage, Gateway federation, task ledger, config/schema/docs, validation, and ScalyClaw prototype inputs.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, release-tag, related-issue, and live GitHub inspection give high confidence that OpenClaw lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
