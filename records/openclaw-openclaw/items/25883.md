## Summary

Keep open for maintainer architecture review. Current main still uses in-process command lanes, Gateway-local/file-backed cron state, and local SQLite task ledgers, while the issue discussion treats #13613 as only the narrower cron-HA overlap.

## What This Changes

Request to add Redis/BullMQ-style distributed job orchestration with workers, retries/backoff, delayed and cron jobs, dead-letter handling, and observability for horizontal scaling.

## Best Possible Solution

Make an explicit architecture decision: either design a generic distributed scheduler/job-backend contract, possibly plugin-backed, or document this as external deployment/plugin prior art while coordinating the cron HA subset with #13613.

## Reproduction Assessment

Not applicable as a feature/architecture proposal. The current-main verification path is source-backed: queue, cron, config, task registry, and dependency surfaces show the requested distributed orchestration layer is absent.

## Solution Assessment

Unclear. Redis/BullMQ is useful prior art, but the narrowest maintainable direction needs maintainer architecture judgment around a generic core/plugin contract rather than importing one backend directly into today’s local scheduler paths.
