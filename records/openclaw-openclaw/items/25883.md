## Summary

Current main still lacks distributed job claiming or horizontally coordinated workers: command lanes and cron serialization are explicitly process-local, while SQLite-backed cron/task state remains local runtime persistence rather than a shared queue. The related open issues cover cron HA, shared sessions, and gateway federation only as narrower slices, so this umbrella proposal still needs a maintainer architecture decision rather than closure.

## What This Changes

Introduce Redis-backed distributed job orchestration with independent workers, delayed and scheduled jobs, retries, dead-letter handling, prioritization, and centralized job observability.

## Best Possible Solution

Define a backend-neutral distributed execution contract for durable enqueueing, leasing, retries, scheduling, cancellation, and observability; then decide whether a Redis/BullMQ backend is an external plugin or an explicitly supported core deployment mode, with narrower issues owning cron HA, shared sessions, and federation.

## Reproduction Assessment

Not applicable: this requests a new distributed deployment and execution capability rather than reporting broken behavior against an existing contract. Current source inspection nevertheless gives high confidence that the requested cross-instance queue and claiming behavior is absent.

## Solution Assessment

No, not as a direct Redis/BullMQ integration yet. The prototype is useful prior art, but the maintainable repository solution first needs a backend-neutral ownership contract and an explicit core-versus-plugin decision.
