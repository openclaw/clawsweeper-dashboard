## Summary

Keep open for maintainer architecture review: current main and v2026.5.22 still use process-local queues, Gateway-local cron state, and local task/flow ledgers, while this proposal asks for distributed Redis/BullMQ-style job orchestration and horizontal worker coordination. The ScalyClaw prototype and the closed BullMQ cron pull request are useful prior art, but related open issues only cover narrower cron, session-storage, Gateway-federation, distributed-runtime, and worker-auth slices.

## What This Changes

Request a Redis/BullMQ-style distributed job orchestration layer with workers, retries/backoff, delayed and cron jobs, dead-letter semantics, and queue observability for horizontal scaling.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, while preserving current Tasks/TaskFlow contracts and coordinating the narrower cron HA, external session storage, Gateway federation, and distributed runtime trackers.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, release-tag, related-issue, and prototype inspection give high confidence that OpenClaw lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
