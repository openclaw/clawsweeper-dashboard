## Summary

Keep open. Current main still has process-local command lanes, Gateway-local cron scheduling, file-backed cron state, and local SQLite/in-memory task ledgers; it does not implement the requested Redis/BullMQ-style distributed queue, shared worker claiming, dead-letter handling, or horizontal worker coordination. The existing maintainer-facing comment also keeps this as architecture input, with #13613 covering only the narrower multi-gateway cron coordination slice.

## What This Changes

The issue proposes adding Redis/BullMQ-style distributed job orchestration so background work, delayed jobs, retries, cron, and worker coordination can scale horizontally.

## Best Possible Solution

Keep this as a maintainer architecture item: decide whether OpenClaw should grow a generic distributed scheduler/job backend abstraction, likely with plugin-capable seams for shared queue/state, worker claiming, retries/backoff, dead-letter semantics, cron support, and observability; otherwise document Redis/BullMQ-style orchestration as an external deployment or plugin pattern while preserving the linked ScalyClaw prototype as prior art and coordinating the cron HA slice with #13613.
