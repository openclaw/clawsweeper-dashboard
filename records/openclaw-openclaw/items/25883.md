## Summary

Current main has improved durable SQLite records for cron and background tasks, but execution ownership remains Gateway- and process-local: command lanes are in-memory, cron runs inside one Gateway, and no shared worker claim, distributed retry/dead-letter, or cross-instance orchestration contract exists. The linked issues cover narrower cron coordination, gateway federation, and session storage and all remain open, so this broader architecture proposal is still useful input but requires a maintainer product decision rather than an automated implementation.

## What This Changes

Add a distributed job orchestration layer with shared queues, independent workers, delayed and scheduled jobs, retries, failure handling, and observability for horizontally scaled deployments.

## Best Possible Solution

Define a maintainer-approved, backend-neutral distributed execution architecture with explicit work claiming, idempotency, scheduler ownership, state consistency, retry/dead-letter policy, observability, and upgrade semantics, then split implementation into coordinated canonical slices and keep Redis/BullMQ as one possible backend rather than the core contract.

## Reproduction Assessment

Not applicable. OpenClaw does not currently document horizontal distributed execution as a supported capability; source inspection instead confirms the requested capability is absent by design rather than a reproducible regression.

## Solution Assessment

No. Redis/BullMQ is useful prior art, but directly integrating that stack is not yet proven to be the narrowest maintainable solution; maintainers should first choose a backend-neutral core/plugin boundary and coordinate the existing cron, session, and gateway scaling work.
