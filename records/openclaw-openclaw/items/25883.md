## Summary

Keep open for maintainer architecture review. Current main still has process-local command lanes, Gateway-local cron scheduling, local cron state files, and local SQLite task ledgers; it does not implement the requested Redis/BullMQ-style distributed job orchestration, worker claiming, dead-letter handling, or horizontal coordination. The existing maintainer-facing comment also keeps this open and treats #13613 as only a narrower cron-HA overlap.

## What This Changes

Request to add a horizontally scalable job orchestration layer with Redis/BullMQ-style queues, dedicated workers, retries/backoff, dead-letter handling, scheduled jobs, and observability.

## Best Possible Solution

Keep this as an architecture decision. Maintainers should decide whether OpenClaw needs a generic distributed scheduler/job backend contract, possibly plugin-backed, that covers shared queue/state, worker claiming, retries/backoff, dead-letter semantics, cron support, and observability; otherwise document Redis/BullMQ-style orchestration as an external deployment or plugin pattern while preserving ScalyClaw as prior art and coordinating the cron HA subset with #13613.

## Reproduction Assessment

Not applicable as a feature/architecture request. The current-main verification path is source-backed: inspect queue, cron, config, task registry, and plugin surfaces to confirm the requested distributed orchestration layer is absent.

## Solution Assessment

Unclear. Redis/BullMQ may be useful prior art, but the best maintainable direction needs maintainer architecture judgment around a generic core/plugin contract rather than importing one backend directly into the current local scheduler paths.
