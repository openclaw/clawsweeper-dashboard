## Summary

Keep open. Current main still uses process-local queues, Gateway-local cron scheduling, and local task/flow stores, so the requested distributed job orchestration layer remains unresolved and needs maintainer architecture review.

## What This Changes

The issue proposes a Redis/BullMQ-style distributed job orchestration layer with separate workers, retries/backoff, delayed and cron jobs, dead-letter handling, and observability.

## Best Possible Solution

Keep the architecture issue open until maintainers decide the distributed scheduler/job-backend boundary, preserve ScalyClaw as prior art, and coordinate #13613 for the cron-only HA slice.

## Reproduction Assessment

Not applicable. This is an architecture feature proposal, and source/docs review shows current main lacks a distributed queue/worker backend rather than exposing a failing current behavior with reproduction steps.

## Solution Assessment

Unclear. Redis/BullMQ is plausible prior art, but the maintainable path should first decide whether OpenClaw needs a generic distributed scheduler/job backend, a plugin API seam, or external deployment guidance.
