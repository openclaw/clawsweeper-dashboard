## Summary

Keep open: current main and v2026.6.10 still use in-process command lanes, Gateway-local cron execution, local SQLite task ledgers, and path/accessor-based session storage, with no Redis/BullMQ-style distributed queue, worker claiming, dead-letter backend, or horizontal worker coordination. The issue remains useful architecture input and needs maintainer product review rather than closure or an automated fix PR.

## What This Changes

The issue proposes adding Redis/BullMQ-style distributed job orchestration with external workers, retries/backoff, delayed and scheduled jobs, dead-letter handling, and queue observability for horizontally scaled OpenClaw deployments.

## Best Possible Solution

Define a backend-neutral scheduler/job orchestration boundary first, then decide whether implementation belongs in core, a plugin API, or documented external deployment guidance while coordinating the narrower cron HA, external session storage, and federation trackers.

## Reproduction Assessment

No bug reproduction is applicable; this is an architecture feature proposal. Source and release inspection give high confidence that current main lacks the requested distributed queue/backend contract.

## Solution Assessment

No for direct Redis/BullMQ core integration as written. The maintainable path is a maintainer-approved backend-neutral boundary that fits existing cron, task, session, plugin, config, and security ownership.
