## Summary

Keep open for maintainer architecture review. Current main still uses process-local queues, Gateway-local cron scheduling, local cron state files, and SQLite-backed task ledgers; it does not implement the requested Redis/BullMQ-style distributed queue, worker coordination, dead-letter handling, or centralized job orchestration, and prior maintainer context kept this as broader architecture input rather than a cleanup close.

## What This Changes

The issue proposes Redis/BullMQ-backed job orchestration with separate workers, retries/backoff, delayed and scheduled jobs, dead-letter handling, and observability for horizontally scaled OpenClaw deployments.

## Best Possible Solution

Decide a generic distributed scheduler/job backend direction, likely plugin-capable, and coordinate the narrower cron HA slice with #13613 while preserving the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable; this is a feature and architecture proposal rather than a reproducible defect. Source and docs inspection does give high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ may be useful input, but the maintainable OpenClaw path is likely a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one concrete backend.
