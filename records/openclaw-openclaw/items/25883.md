## Summary

Keep open for architecture review: current main still has process-local queues, Gateway-local cron scheduling, local cron/task/session state, and no distributed worker-claiming or Redis/BullMQ-style orchestration layer. The issue discussion and related open issues treat this as broad horizontal-scaling input rather than cleanup that can be closed or safely queued as a narrow fix.

## What This Changes

The issue proposes a Redis/BullMQ-style distributed job orchestration layer for background jobs, delayed/cron work, retries, dead-letter handling, visibility, and horizontally scaled workers.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, session storage, Gateway federation, config, docs, validation, and ScalyClaw as prior art before implementation.

## Reproduction Assessment

Not applicable; this is a feature and architecture proposal rather than a reproducible defect. Source and docs inspection gives high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
