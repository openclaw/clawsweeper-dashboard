## Summary

Keep open for maintainer architecture review: current main and v2026.5.20 still use process-local queues, Gateway-local cron scheduling/state, and local task ledgers, while this proposal asks for distributed Redis/BullMQ-style worker coordination. The ScalyClaw prototype and prior closed BullMQ/minions PRs are useful prior art, but related open issues cover narrower cron, session-storage, and Gateway-federation slices rather than superseding this broader scaling request.

## What This Changes

The issue proposes Redis/BullMQ-style distributed job orchestration, dedicated workers, retries/backoff, dead-letter handling, cron support, and observability for horizontal scaling.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, external session storage, Gateway federation, config/schema/docs, validation, and the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, release-tag, related-issue, and prototype inspection give high confidence that OpenClaw lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
