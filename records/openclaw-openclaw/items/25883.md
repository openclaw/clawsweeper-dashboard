## Summary

Keep open for maintainer architecture review. Current main still uses process-local queues, Gateway-local cron scheduling/state, and local SQLite task/flow stores, while the request asks for distributed Redis/BullMQ-style orchestration and the discussion already treats the ScalyClaw prototype as architecture input rather than cleanup to close.

## What This Changes

The issue proposes Redis/BullMQ-backed horizontal job orchestration with separate workers, retries/backoff, delayed and scheduled jobs, dead-letter handling, and observability.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, session storage, Gateway federation, config, docs, validation, and ScalyClaw as prior art.

## Reproduction Assessment

Not applicable; this is a feature and architecture proposal rather than a reproducible defect. Source and docs inspection gives high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
