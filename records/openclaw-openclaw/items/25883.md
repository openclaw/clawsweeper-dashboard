## Summary

Keep open: current main and v2026.6.5 still use an in-process command queue, Gateway-local cron execution, and SQLite-backed task/cron ledgers rather than the requested Redis/BullMQ-style distributed job orchestration and worker coordination layer. The open related issues cover narrower scaling slices, but none supersedes this broader architecture proposal.

## What This Changes

The issue requests horizontal scaling support through a Redis/BullMQ-style distributed job orchestration layer with workers, retries, delayed/cron jobs, failure handling, and observability.

## Best Possible Solution

Make a maintainer architecture decision for a generic or plugin-capable distributed scheduler/job backend, or document Redis/BullMQ-style orchestration as an external deployment/plugin pattern while coordinating the open cron HA, session storage, and Gateway federation trackers.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, release-tag, and live GitHub inspection give high confidence that OpenClaw lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
