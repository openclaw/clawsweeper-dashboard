## Summary

Keep open for maintainer architecture review: current main and the latest release still use process-local command queues, Gateway-local cron scheduling/state, and local SQLite task/flow ledgers, while this proposal asks for distributed Redis/BullMQ-style worker coordination. The ScalyClaw prototype plus the closed BullMQ/minions PRs are useful prior art, but the related open issues cover narrower cron, session-storage, and Gateway-federation slices rather than superseding this broader scaling request.

## What This Changes

Requests a Redis/BullMQ-style distributed job orchestration layer with shared queues, worker processes, delayed/cron jobs, retries, dead-letter behavior, and queue observability for horizontally scaled OpenClaw deployments.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, external session storage, Gateway federation, config/schema/docs, validation, and the ScalyClaw prototype as prior art.

## Reproduction Assessment

Not applicable: this is a feature and architecture proposal rather than a reproducible defect. Source, docs, latest-release, related-issue, and prototype inspection give high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
