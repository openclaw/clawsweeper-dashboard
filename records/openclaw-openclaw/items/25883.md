## Summary

Keep open for maintainer architecture review: current main still uses in-process queues, Gateway-local cron scheduling/state, local task ledgers, and isolated multi-Gateway profiles, while the request asks for distributed Redis/BullMQ-style orchestration and worker coordination.

## What This Changes

The issue proposes adding Redis/BullMQ-backed horizontal job orchestration with separate workers, retries/backoff, delayed and scheduled jobs, dead-letter handling, and observability.

## Best Possible Solution

Make a maintainer architecture decision for a generic distributed scheduler/job backend direction, likely plugin-capable, then coordinate cron HA, external session storage, Gateway federation, config/schema/docs, validation, and ScalyClaw as prior art.

## Reproduction Assessment

Not applicable; this is a feature and architecture proposal rather than a reproducible defect. Source and docs inspection gives high confidence that current main lacks the requested distributed queue and worker-coordination layer.

## Solution Assessment

Unclear until maintainers choose the architecture. Redis/BullMQ is useful prior art, but the maintainable OpenClaw path likely needs a generic scheduler/job backend abstraction or documented plugin/deployment pattern before committing to one backend.
