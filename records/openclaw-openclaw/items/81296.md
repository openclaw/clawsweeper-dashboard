## Summary

Keep open: the pack-before-install approach is narrow and the supplied macOS live output is enough real behavior proof, but the patch introduces new failing update step names that currently fall through to `unexpected-error` for JSON/control-plane callers.

## What This Changes

This PR pre-packs npm Git/GitHub update specs into a temporary tarball before the staged global npm install and adds updater/CLI coverage for that path.

## Best Possible Solution

Keep the pre-pack staged install design, but land it with failure reason/progress handling and regression coverage for pack and pack-verify failures.

## Reproduction Assessment

Yes for the review finding by source inspection: a non-zero `global update pack` or `global update pack verify` failedStep falls through to `unexpected-error`. The original macOS npm failure was not reproduced locally in this read-only review, but the current-main source path and supplied live output show the affected update flow.

## Solution Assessment

Mostly yes: pre-packing only npm Git/GitHub specs and feeding the tarball into the existing staged updater is the narrowest maintainable approach I found. It should not merge until the new failure steps are classified as package-manager install failures and covered by tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Classify pack-step failures as global install failures:** `src/infra/package-update-steps.ts:187`
  - body: When `npm pack` exits non-zero or the verify step cannot find exactly one tarball, this helper returns failed step names `global update pack` or `global update pack verify`. `normalizeFallbackFailureReason` does not recognize either name, so those package-manager failures are reported as `unexpected-error` instead of `global-install-failed` to JSON/control-plane callers. Please add the new step names to the failure mapping and cover the pack-failure path.
  - confidence: 0.87
