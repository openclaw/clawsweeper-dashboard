## Summary

Keep open for normal PR review: current main still sends npm GitHub/git specs directly to `npm i -g`, while this PR adds a focused pack-then-install path with coverage and real command-output proof. I found one nonblocking cleanup issue where a thrown pack step can leave the temporary pack directory behind.

## What This Changes

The PR pre-packs npm GitHub/git source update specs into a temporary tarball before staged global npm install and adds update-runner, CLI, and package-update-step coverage.

## Best Possible Solution

Land the targeted npm git-source prepack path after fixing or consciously accepting the temporary-directory cleanup edge case, preserving the staged swap and existing pnpm source-update behavior.

## Reproduction Assessment

Yes, from source and supplied proof: current main builds the direct `npm i -g --prefix <stage> github:...` command, and the PR body gives a concrete macOS failure plus the passing pack/install command sequence. I did not rerun that macOS reproduction locally in this read-only review.

## Solution Assessment

Yes, with one minor cleanup fix: prepacking only npm GitHub/git source specs is a narrow package-manager-boundary workaround that reuses the existing staged install, verification, fallback, and swap flow.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86

Full review comments:

- **[P3] Clean up the pack directory when packing throws:** `src/infra/package-update-steps.ts:185-186`
  - body: `prepareNpmGitSourceInstallSpec` creates `packDir` before awaiting `runStep`, but the outer `finally` only learns that path after the helper returns. If spawning `npm pack` throws, for example when the npm command cannot be spawned, the update fails before `packedInstallDir` is assigned and the new temp directory is left behind. Assign cleanup ownership before awaiting the pack step or catch and remove `packDir` inside this helper.
  - confidence: 0.78
