## Summary

This is a plausible CLI updater bug fix, but the patch is incomplete: it pre-packs only a subset of npm git source specs while the updater accepts broader npm package-spec forms that would still take the failing direct install path.

## What This Changes

The PR changes npm-managed update staging so GitHub/git install specs are packed to a temporary tarball before the staged global install, with tests for the new pack-and-install flow.

## Best Possible Solution

Keep the PR open and make the prepack decision cover every npm git source spec form accepted by the updater, with regression coverage for full git URLs, git@ URLs, and GitHub shorthands.

## Reproduction Assessment

Yes, at source level: current main sends GitHub/git update specs directly to npm i -g --prefix, and the PR body gives a concrete macOS command showing that direct path failing with spawn /bin/sh ENOENT. I did not rerun the macOS command in this review.

## Solution Assessment

No. Packing before staged install is the right direction for the built-in github: spec, but the detector should cover npm's documented git source forms before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prepack every npm git source spec:** `src/infra/package-update-steps.ts:124-126`
  - body: The new detector only recognizes github:, git+..., and git: specs, but npm also treats full git URLs, git@ URLs, and GitHub username/repo shorthands as git package specs. Because the updater passes explicit specs containing :// or # through to the same global npm path, those forms still skip npm pack and keep the failing direct npm i -g --prefix behavior. Please classify the same git forms npm accepts, or delegate this detection to npm-package-arg/pacote, and add coverage.
  - confidence: 0.86
