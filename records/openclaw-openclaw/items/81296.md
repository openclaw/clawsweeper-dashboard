## Summary

Keep open: the prepack approach is useful and the supplied proof covers the GitHub shortcut path, but the git-spec gate misses other npm-supported Git package-spec forms that OpenClaw accepts through `--tag <spec>`, so the same direct staged npm install failure can remain.

## What This Changes

The PR adds a pre-pack step for npm Git/GitHub update specs, installs the resulting tarball through the staged npm update flow, and extends updater tests.

## Best Possible Solution

Keep the staged prepack design, but classify npm Git specs consistently with npm's package-spec contract while preserving direct registry, tarball, file, and alias installs.

## Reproduction Assessment

Yes. The current-main source path maps `--tag main` to a GitHub package spec and passes it directly to staged `npm i -g --prefix`, and the PR body includes macOS before/after terminal output for the failing and passing npm sequence.

## Solution Assessment

No. Prepacking is the right shape, but this implementation should cover every npm-documented Git package-spec form that OpenClaw accepts through `--tag <spec>`, not only `github:` and `git+`/`git:` prefixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prepack every npm git package spec:** `src/infra/package-update-steps.ts:124-127`
  - body: The new gate only matches `github:`, `git+...`, and `git:` specs. npm's package-spec contract also treats forms like `owner/repo#ref`, `https://github.com/...git#ref`, and `git@github.com:...git#ref` as git specs, and OpenClaw accepts arbitrary `--tag <spec>`, so those update paths would still use direct `npm i -g --prefix ...` and can hit the same failure this PR is meant to avoid. Please classify git specs with npm-package-arg-equivalent logic or cover these forms explicitly while leaving tarball URLs direct.
  - confidence: 0.88
