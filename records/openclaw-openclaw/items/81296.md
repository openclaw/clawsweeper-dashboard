## Summary

Keep open: this is a useful update-path fix with real terminal proof, but the current head still misses npm-supported hosted GitHub HTTPS specs without a `.git` suffix, so removing the existing fail-fast rejection would leave a valid git source form on the direct staged install path.

## What This Changes

The PR removes the current OpenClaw source-package rejection, pre-packs npm GitHub/git update specs into a temporary tarball before staged global install, and updates CLI docs, changelog, and tests.

## Best Possible Solution

Keep the pack-before-install design, but align the detector and regression tests with npm-package-arg hosted GitHub forms, including HTTPS GitHub URLs without `.git`, before removing the source-target rejection.

## Reproduction Assessment

Yes at source level: npm's parser classifies hosted GitHub HTTPS specs without `.git` as git specs, while the PR detector returns false for those same inputs. I did not run a full branch-built `openclaw update` in this read-only review.

## Solution Assessment

No, not yet. Prepacking git source specs is the right repair, but the implementation needs detector/test coverage for hosted GitHub HTTPS URLs without `.git` before removing the current validation path.
