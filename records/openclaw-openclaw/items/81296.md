## Summary

Keep open: this is a useful updater fix, but the current PR head still misses npm git package-spec forms, so some source update targets would fall through to the direct staged npm install path while the docs promise GitHub/git specs are pre-packed.

## What This Changes

The PR removes the current early rejection for OpenClaw GitHub/git package update targets, pre-packs detected npm git specs into a temporary tarball before staged npm install, and updates CLI docs, tests, and changelog text.

## Best Possible Solution

Keep the pack-before-install design, but classify every npm-supported git package spec form that OpenClaw accepts before removing the early source-target rejection, with focused regression tests for the missed forms.

## Reproduction Assessment

Yes at source level: current main rejects OpenClaw source package targets before update, and the PR body proves the central `github:` pack/install sequence succeeds. The remaining blocker is also source-reproducible from the PR regex versus npm's documented package-spec forms.

## Solution Assessment

No, not yet. Prepacking git source specs is the right repair, but the detector must cover npm's GitHub shorthand, SCP-style SSH, and plain GitHub HTTPS git forms or explicitly keep those forms rejected before merge.
