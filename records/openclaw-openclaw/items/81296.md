## Summary

Keep open: the PR targets a real npm Git/GitHub update-path bug and the pack-then-install direction is plausible, but the patch currently leaves the new pack failure path classified as `unexpected-error` and CI is failing existing update-flow tests that were not updated for the extra pack step.

## What This Changes

The branch adds an npm pack step for Git/GitHub global update specs, installs the produced tarball into the staged npm prefix, cleans the pack directory, and adds package-update-step coverage.

## Best Possible Solution

Keep the pack-before-staged-install approach if it remains the narrowest npm workaround, but wire the new step into update-runner failure taxonomy and existing CLI/update test harnesses before merge.

## Reproduction Assessment

Yes for the npm-level path: the PR body gives direct failing and passing commands, and current source maps GitHub specs to the staged npm update path. I did not independently run a full branch-built `openclaw update` in this read-only review.

## Solution Assessment

Unclear until the blockers are fixed: the npm pack/tarball path matches the npm contract and the reported workaround, but the proposed patch currently misses failure classification and existing update-flow test updates.
