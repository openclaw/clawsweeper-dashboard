## Label Justifications

- `P1`: The issue reports real Feishu and Telegram agent/channel workflows producing runaway repeated messages until user intervention.
- `impact:message-loss`: The core symptom is duplicated visible channel messages during failed tool-call retry loops.
- `impact:crash-loop`: The agent can remain stuck in a runaway repeated tool-call loop, affecting session availability.

## Summary

Keep open. Current main has useful partial mitigations, including disabled embedded Pi SDK auto-retry and committed-message duplicate suppression, but the default rolling detector is still off and deterministic validation failures still lack a low terminal per-turn cap for the reported Feishu/Telegram visible-message loop.

## What This Changes

Prevent agents from repeatedly retrying failed tool calls and sending duplicate visible channel messages when tool arguments are invalid.

## Best Possible Solution

Keep this as the concrete bug thread and land a maintainer-scoped runtime guard for deterministic validation failures plus duplicate visible-message suppression, while leaving broader self-healing architecture to https://github.com/openclaw/openclaw/issues/60767.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source/report level: the Feishu missing-command path and Telegram corroboration are concrete, and current main still shows default-off rolling detection plus no low terminal validation-failure cap; I did not replay live channels in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No patch is under review. The maintainable path is a focused maintainer-approved guard rather than a broad default-on or self-healing change without settled thresholds and channel-noise policy.
