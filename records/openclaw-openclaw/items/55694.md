## Summary

Keep open. Current main has useful loop-safety pieces, but the reported default Feishu/Telegram repeated-message path is still not fully fixed because rolling loop detection remains opt-in and required-parameter validation does not enforce a terminal per-turn retry cap.

## What This Changes

The issue reports agents repeatedly retrying failed tool calls and sending duplicate visible messages, and asks for validation, retry limits, backoff, and quieter failure reporting.

## Best Possible Solution

Keep this as the concrete bug thread and land a maintainer-scoped runtime guard for deterministic validation failures and duplicate visible-progress suppression, while leaving broader self-healing architecture to https://github.com/openclaw/openclaw/issues/60767.
