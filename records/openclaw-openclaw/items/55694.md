## Summary

Keep open. Current main still has opt-in rolling loop detection and warning-only generic repeats, so the reported default Feishu/Telegram repeated-message tool-failure loop is not proven fixed; overlapping open work needs a maintainer policy choice.

## What This Changes

The issue reports failed tool calls looping with repeated user-visible messages and asks for validation-error classification, retry limits, backoff, and reduced message noise.

## Best Possible Solution

Land one canonical core runtime guard that treats deterministic validation failures as terminal within a run, caps repeated identical tool+args+error outcomes, and suppresses duplicate visible progress for the same failing action while preserving explicit polling semantics.

## Reproduction Assessment

Yes. The issue gives a concrete Feishu `exec` missing-parameter path with Telegram corroboration, and current source shows the default detector is off while enabled generic repeats remain warning-only.

## Solution Assessment

No. The issue’s direction is valid, but the best fix is a focused maintainer-selected loop/error guard rather than a broad retry/backoff/self-heal change without settled default and threshold policy.
