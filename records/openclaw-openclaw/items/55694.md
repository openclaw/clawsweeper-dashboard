## Summary

Keep open: current main and v2026.6.8 still leave the reported default-path failure possible because tool-argument validation errors are turned into immediate tool-result errors before the before-tool-call loop detector can run, and the general loop detector remains opt-in.

## What This Changes

The issue reports repeated failed tool calls causing an agent loop and repeated channel-visible messages.

## Best Possible Solution

Adopt a maintainer-approved default runtime policy that catches repeated validation-error tool results before they can loop, suppresses duplicate channel-visible progress messages, and keeps the existing opt-in detector behavior compatible for high-power workflows.

## Reproduction Assessment

Yes at source level: a missing required tool argument is returned as an immediate error tool result before the before-tool-call detector can run, and the agent loop continues unless later policy stops it. No live Feishu or Telegram reproduction was run.

## Solution Assessment

No: the existing opt-in detector is useful for calls that reach the hook, but it does not cover pre-hook validation failures or settle the default channel-visible suppression policy.
