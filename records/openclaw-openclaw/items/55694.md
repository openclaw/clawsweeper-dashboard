## Summary

Keep open. Current main still has only partial safeguards: tool-loop detection is documented and implemented as opt-in, generic repeated calls remain warning-only until the later global no-progress breaker when enabled, and the reported default Feishu/Telegram repeated-message loop is not proven fixed.

## What This Changes

The issue requests core safeguards to classify deterministic tool-call validation failures, limit repeated retries, and suppress repeated user-visible progress messages when a tool keeps failing with the same error.

## Best Possible Solution

Land a focused core runtime guard that treats deterministic validation failures such as missing required tool parameters as non-retryable within a run, caps repeated identical tool+args+error outcomes, and suppresses repeated visible progress delivery for the same failing action while keeping legitimate polling thresholds explicit and tested.

## Reproduction Assessment

Yes. The report gives a concrete Feishu `exec` missing-parameter path plus Telegram corroboration, and current source/tests reproduce the guard gap by showing loop detection is off by default and generic repeats are warning-only when enabled.

## Solution Assessment

No. Current main is not the complete solution; the safer path is a maintainer-selected, no-progress-aware tool execution guard rather than relying on opt-in docs, guidance text, or a broad default-on change without threshold review.
