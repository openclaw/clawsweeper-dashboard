## Summary

Keep open: current main and the latest release have useful loop-detection pieces, but the rolling detector that would cover the reported default-path repeated failed tool calls is still opt-in, and deciding default blocking thresholds plus channel-visible noise suppression needs maintainer product judgment.

## What This Changes

The issue reports repeated failed tool calls causing agent retry loops and repeated user-visible messages in channels such as Feishu and Telegram.

## Best Possible Solution

Keep this as the specific channel-visible repeated-tool-failure tracker until maintainers choose the default policy for rolling loop detection and message-noise suppression; the end state should extend the existing loop-detection path rather than add a broad parallel self-healing framework.

## Reproduction Assessment

Yes from source for the remaining gap: the rolling detector exists but is disabled unless configured, so the no-config path can still avoid the pre-tool breaker. No live Feishu or Telegram repro was run in this read-only sweep.

## Solution Assessment

Unclear as a product decision: enabling the existing rolling detector by default would be the narrowest technical fix, but thresholds and channel-noise behavior need maintainer approval because this changes default runtime behavior.
