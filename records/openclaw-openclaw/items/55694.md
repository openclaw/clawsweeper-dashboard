## Summary

Keep open: current main still leaves the reported default-path failure possible because schema validation failures become immediate error tool results before loop detection can run, while the rolling detector is documented and tested as opt-in.

## What This Changes

The issue asks OpenClaw to stop repeated failed tool-call loops and suppress duplicate channel-visible messages when tool arguments or executions fail repeatedly.

## Best Possible Solution

Keep this open until core has a maintainer-approved default policy that catches repeated validation/tool failures, blocks the runaway turn, and emits at most one structured user-facing failure across channel runtimes.

## Reproduction Assessment

Yes at source level: a missing or invalid tool argument is converted into an immediate error tool result before the before-tool-call detector can run, and the rolling detector remains default-off. I did not run a live Feishu or Telegram repro in this read-only pass.

## Solution Assessment

Unclear as written: the useful existing detector is not enough because validation failures can bypass it and default behavior is a product choice. The safer solution is a maintainer-approved default runtime policy with explicit thresholds and channel-message suppression semantics.
