## Summary

Keep open. Current main has partial loop-detection machinery and better required-parameter guidance for some tools, but the guard is still disabled by default, generic repeated calls are warning-only until a later no-progress breaker when explicitly enabled, and the reported default Feishu/Telegram message-spam loop is not proven fixed.

## What This Changes

The issue requests core safeguards that classify deterministic tool-call validation failures, cap repeated retries, and suppress repeated user-visible progress messages when a tool keeps failing with the same error.

## Best Possible Solution

Land a focused core runtime guard that treats deterministic validation failures such as missing required tool parameters as non-retryable within a run, caps repeated identical tool+args+error outcomes, and prevents repeated visible progress-message delivery for the same failing action while preserving configurable thresholds for legitimate polling.

## Reproduction Assessment

Yes for the core guard gap: the report provides concrete Feishu logs and Telegram corroboration, and current tests/source show loop detection is off by default. The exact channel-level message flood is model-dependent, but the missing default hard stop is reproducible by repeated missing-parameter exec attempts under default config.

## Solution Assessment

No, current main is not the complete solution: opt-in loop detection and guidance text improve diagnostics but do not enforce the requested default retry cap or message-noise suppression. A narrower maintainable fix should be in the agent tool execution guard, with maintainer-chosen defaults and thresholds.
