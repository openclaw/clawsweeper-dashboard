## Summary

Keep open: current main and the pinned upstream edit tool only partially address this broad request. Recovery hints, LF handling, trailing-whitespace/Unicode fuzzy normalization, and smaller-oldText prompt guidance exist, but broad ASCII whitespace/blank-line normalization, an explicit preflight contract, and `read keyword=` positioning remain unresolved tool/API decisions.

## What This Changes

Request broadening edit/read tooling with whitespace-tolerant matching, preflight validation, line/keyword read positioning, and guidance for smaller edits.

## Best Possible Solution

Define a maintainer-approved read/edit tool contract, preferably with upstream alignment, that specifies matching normalization, ambiguity handling, preview/diff correctness, read positioning params, and user-facing guidance before implementation.

## Reproduction Assessment

Yes, at source level. Current main plus the pinned upstream package show the requested broad whitespace fallback and `read keyword=` positioning are absent, while only narrower recovery and fuzzy-normalization behavior exists.

## Solution Assessment

Unclear. The requested direction addresses real friction, but the broad normalization and read-schema changes are not the best maintainable fix until ambiguity, preview, and compatibility rules are decided.
