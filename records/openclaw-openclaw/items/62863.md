## Summary

Keep open: current main still only partially addresses the request. OpenClaw adds recovery/readback hints around edit failures and now pins pi-coding-agent 0.71.1, whose upstream edit tool has LF handling plus narrow fuzzy normalization, but the requested broader whitespace-collapsing fallback, explicit preflight contract, `read keyword=` positioning, and small-edit guidance remain unresolved product/API decisions.

## What This Changes

Request to make the edit/read tools more tolerant and easier to recover from via whitespace-normalized edit matching, preflight checks, line/keyword read positioning, and small-edit guidance.

## Best Possible Solution

Define and land a maintainer-approved read/edit tool contract, preferably upstream or with a narrow documented OpenClaw wrapper, covering normalization, uniqueness, preview/diff correctness, read positioning params, and user-facing guidance.

## Reproduction Assessment

Yes. A static source reproduction is clear: `read` accepts path/offset/limit only, and `edit` still requires a unique oldText match after only LF/trailing-whitespace/Unicode fuzzy normalization, so collapsed ASCII spaces/tabs or blank-line differences can still miss; no mutating live test was run because this was a read-only review.

## Solution Assessment

Unclear. The requested direction addresses real friction, but broad normalization is not yet the narrowest maintainable fix without a contract for ambiguous matches, previews, diffs, and read parameter compatibility.
