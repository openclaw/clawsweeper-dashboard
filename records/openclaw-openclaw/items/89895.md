## Summary

Keep open: the PR addresses the right stale UI problem, but the new guard is too narrow and can hide Stop during a real active run when local stream text is null. The supplied proof is still unit-test output rather than real Control UI browser/gateway behavior.

## What This Changes

The PR narrows Control UI chat's in-progress badge and Stop button to local send/stream activity and adds focused chat-view regression tests.

## Best Possible Solution

Keep the stale session-list-only fix, but make the visible abort guard include true local active-run state such as `chatRunId`/tool-run activity and add real Control UI proof.

## Reproduction Assessment

Yes for source-level review: current main can render stale abortability from session-list state, and the PR's new predicate can be checked against known active null-stream states in the chat/tool paths. I did not run a live browser/gateway repro in this read-only review.

## Solution Assessment

No. Guarding only on `sending || stream !== null` is too narrow because local active runs can have `chatRunId` set while `chatStream` is null; the best fix separates stale session-list-only abortability from local active-run abortability.
