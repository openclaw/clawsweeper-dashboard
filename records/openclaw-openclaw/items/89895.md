## Summary

Keep open: this PR targets a real Control UI false-busy bug, but the current guard can hide Stop while a local streamless tool run is still active and the PR still has only test-output proof for a visible browser behavior.

## What This Changes

This PR gates the Control UI WebChat In progress badge and Stop button on local send/stream state and adds focused render tests for stale abortability.

## Best Possible Solution

Carry an explicit local active-run signal, such as `chatRunId`, separately from session-list abortability so stale session rows do not force busy UI while current streamless runs remain abortable, then add coverage and real Control UI proof.

## Reproduction Assessment

Yes, at source level. Current main can have `chatRunId` set while `chatStream` is null during tool rendering, and this PR's new gate would treat that state as not active.

## Solution Assessment

No. The stale-session split is the right direction, but `sending || stream !== null` is not the best boundary because it drops legitimate local active-run states that should keep Stop visible.
