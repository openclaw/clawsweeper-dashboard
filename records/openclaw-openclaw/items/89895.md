## Summary

Keep this PR open, but do not merge it as written: the stale completed-state problem is real, yet the new guard is too narrow and can hide Stop/In progress during real active runs whose `chatRunId` remains set while `chatStream` is null. The supplied proof is also only a boolean simulation, not a real Control UI browser/gateway run.

## What This Changes

The PR changes Control UI chat rendering so the in-progress badge and Stop button require local `sending || stream !== null` activity in addition to abortability.

## Best Possible Solution

Separate stale session-list-only abortability from local active-run abortability: visible Stop/In progress should remain for sending, stream, or local active `chatRunId`, while stale session-list state alone should not force the completed UI busy state.

## Reproduction Assessment

Yes for the PR regression from source inspection: active runs can keep `chatRunId` while `chatStream` is null, and the patched guard would hide Stop in that state. The original sticky completed-state issue still needs real current-main UI proof for merge confidence.

## Solution Assessment

No. The fix should distinguish stale session-list-only abortability from local active-run abortability, rather than treating `sending || stream !== null` as the whole live-run signal.
