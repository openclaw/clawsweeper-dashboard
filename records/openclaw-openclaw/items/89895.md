## Summary

Keep open: the PR addresses the stale completed-state symptom, but the new guard is still too narrow because it hides visible abort controls for an active local run when `chatRunId` is set and `chatStream` is null. Real browser/gateway proof is also still missing.

## What This Changes

The branch narrows the Control UI chat In progress badge and Stop button to local send/stream activity and adds focused tests for stale session-list-only abortability.

## Best Possible Solution

Split stale session-list abortability from local active-run state: suppress the completed-response false busy UI, but keep visible abort controls active for a local `chatRunId`, then add focused regression coverage and redacted browser/gateway proof.

## Reproduction Assessment

Yes: source inspection shows `canAbort` can be true from a local `chatRunId` while `sending` is false and `stream` is null. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

No: this is a plausible stale-state mitigation, but it is not the best fix because it uses send/stream presence as the only local activity signal. The safer shape is to preserve local `chatRunId` abortability and only suppress stale session-list-only activity after completion.
