## Summary

Keep this PR open, but do not merge it as-is: it targets the right stale Control UI abort-state bug, yet the new guard drops an existing local active-run state where `chatRunId` is still set while `chatStream` is null.

## What This Changes

The PR changes Control UI chat rendering so the visible in-progress badge and Stop button require local send/stream activity in addition to abortability.

## Best Possible Solution

Separate stale session-list abortability from local live-run activity by passing an explicit local active-run signal, such as `Boolean(state.chatRunId)`, into the chat view and using it with `sending || stream !== null` for the visible Stop button and badge.

## Reproduction Assessment

Yes from source: current main has active `chatRunId` states with `chatStream === null`, while the PR only treats `sending || stream !== null` as active. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

No. The PR points at the right stale-session bug, but the guard is too narrow because it omits local `chatRunId` as an active-run signal.
