## Summary

Keep open: the reporter provided real-provider evidence, and current main still treats DeepSeek DSML tool markup on the `openai-completions` stream as visible or filtered text instead of executable tool calls. The newer Microsoft Foundry default helps fresh DeepSeek V4 configs, but existing or explicitly configured `openai-completions` models can still hit the reported path.

## What This Changes

Recover complete DeepSeek/Foundry DSML tool-call text into executable `toolCall` blocks when native `tool_calls` are missing.

## Best Possible Solution

Add a narrowly gated DeepSeek/Foundry DSML recovery path on the OpenAI-compatible completions stream that preserves native `tool_calls`, promotes only complete valid DSML invocations, and leaves malformed DSML as non-executable text or diagnostics.

## Reproduction Assessment

Yes. The reporter supplied a concrete live CLI command and current source shows the same openai-completions path only emits native `choiceDelta.tool_calls`, while DSML text is filtered or diagnosed as text; I did not re-run live Foundry credentials.

## Solution Assessment

Yes, if implemented narrowly. A conservative recovery layer in `processOpenAICompletionsStream` or an equivalent provider-specific wrapper is more complete than relying on the current Foundry default-to-responses mitigation, because existing completions configs remain supported.
