## Summary

Keep open: current main still has the reported OpenAI-completions stream gap, and no canonical merged PR now owns this DSML-text-to-tool-call recovery. The existing code filters DeepSeek DSML text and handles native streamed `tool_calls`, but it does not promote complete DSML `invoke` markup into executable OpenClaw tool calls.

## What This Changes

Recover DeepSeek/Foundry DSML tool markup emitted as assistant text into executable tool calls on the OpenAI completions stream path.

## Best Possible Solution

Add a narrowly gated DeepSeek/Foundry DSML recovery path on the OpenAI-compatible completions stream that preserves native `tool_calls`, promotes only complete valid DSML invocations, and leaves malformed DSML as non-executable text or diagnostics.

## Reproduction Assessment

Yes. The reporter supplied a concrete live CLI command, and current main shows the same source path only emits native `choiceDelta.tool_calls` while DSML content is filtered as text; I did not rerun live Foundry credentials.

## Solution Assessment

Yes, if implemented narrowly. A conservative recovery layer in `processOpenAICompletionsStream` or an equivalent provider-specific wrapper is more complete than relying on the Foundry Responses default because explicit and existing completions configs remain supported.
