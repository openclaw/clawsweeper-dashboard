## Label Justifications

- `P1`: The report describes a real provider/operator workflow where intended tool calls are emitted as text and therefore not executed.

## Summary

Current main still only strips DeepSeek DSML text and builds executable tool calls from native `choiceDelta.tool_calls`, so the reported Foundry DeepSeek V4 text-to-tool-call recovery remains necessary.

## What This Changes

The issue asks OpenClaw to recover complete DeepSeek/Foundry DSML tool markup emitted as assistant text into executable `toolCall` blocks on the `openai-completions` stream path.

## Best Possible Solution

Add a narrow DeepSeek/Foundry DSML recovery step in `processOpenAICompletionsStream` that synthesizes tool calls only for complete, parseable DSML tool blocks, preserves native tool-call behavior, and leaves malformed output non-executable.

## Reproduction Assessment

Yes for a source-level reproduction: a DeepSeek-compatible completions chunk with DSML `<invoke>` content and no native `tool_calls` will pass through the text filter path and never append a `toolCall`. I did not rerun the live Foundry command in this read-only review.

## Solution Assessment

Yes, the best fix is a conservative ingestion-layer recovery in the OpenAI completions stream parser, not a channel formatter workaround. The parser should reject incomplete or non-object DSML so visible examples do not become executable actions.
