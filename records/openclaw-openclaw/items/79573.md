## Summary

Current main already routes the Android chat composer through `sendChat` into an operator-backed `ChatController` that calls `chat.send`, so the reported 2026.5.6 behavior is obsolete on the inspected source.

## What This Changes

The issue requests fixing Android chat sends so pressing Send emits a `chat.send` RPC and preserves the optimistic message instead of only refreshing `chat.history`.

## Best Possible Solution

Keep the current operator-backed Android chat send path; if a released APK built from current main still reproduces this, track that as a new Android build/distribution regression with fresh logs.

## Reproduction Assessment

No for current main: the report has credible 2026.5.6 logs, but the inspected source now routes Send to `chat.send` through the operator session. I did not establish a failing current-main Android path.

## Solution Assessment

Yes: the current implementation is the narrow maintainable path because the UI send action delegates to the Android chat controller, and that controller emits the documented `chat.send` RPC instead of a history refresh.
