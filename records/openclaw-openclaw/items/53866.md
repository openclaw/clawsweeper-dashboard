## Summary

Keep open: this is a useful Firecrawl onboarding feature, but current main has not implemented it and this branch still targets the old command-owned setup path instead of the provider-owned setup seam. It also lacks contributor real-behavior proof and has concrete correctness issues in credential persistence, disabled-state handling, and test teardown.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, writes returned API keys into search/fetch config, and adds tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin as a web-search provider runSetup, persist SecretRef-aware plugin-scoped webSearch/webFetch credentials, add focused provider/search-flow tests, update Firecrawl docs, and include live redacted proof.

## Reproduction Assessment

Yes, for the blocking PR review findings: source inspection shows the branch adds Firecrawl-specific setup to the command path while current main delegates provider setup through runSetup, and the teardown, disabled-state, legacy-config, and spinner issues are visible in the diff. I did not exercise a live Firecrawl browser-auth session.

## Solution Assessment

No. The feature is useful, but the maintainable solution is a Firecrawl-owned runSetup implementation with plugin-scoped credential writes, not a vendor-specific branch in core onboarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move Firecrawl setup behind the provider hook:** `src/commands/onboard-search.ts:406-414`
  - body: Current main routes web-search setup through provider entries and invokes entry.runSetup. Adding Firecrawl-specific browser auth inside the command bypasses that provider-owned seam and targets the pre-refactor implementation; move the helper and setup flow into the Firecrawl provider instead.
  - confidence: 0.91
- **[P2] Persist Firecrawl fetch credentials under plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: This helper writes the returned key to tools.web.fetch.firecrawl.apiKey, but current Firecrawl fetch config is owned by plugins.entries.firecrawl.config.webFetch.apiKey and the tools.web.fetch.firecrawl.* shape is legacy migration/compatibility only. New browser-auth credentials should be stored in the plugin-scoped path.
  - confidence: 0.9
- **[P2] Return the enabled config after browser auth succeeds:** `src/commands/onboard-search.ts:426`
  - body: When web search was previously disabled, this success path stores the key and tells the user Firecrawl is connected, then preserveDisabledState leaves tools.web.search.enabled as false. Manual key entry for the same active setup enables search, so browser auth should not leave the user with a connected-but-disabled provider.
  - confidence: 0.83
- **[P2] Restore fetch in teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-30`
  - body: afterEach calls vi.unstubAllGlobals() and then immediately re-stubs fetch, so the suite leaves a mocked global fetch installed for later files in non-isolated runs. Let beforeEach install the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-118`
  - body: Once prompter.progress returns, an unexpected throw before either spin.stop call falls into the outer catch and leaves the terminal spinner active. Wrap the wait block so failures stop the spinner before returning null.
  - confidence: 0.78
