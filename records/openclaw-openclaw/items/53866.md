## Summary

Keep this PR open as a useful Firecrawl browser-auth feature, but do not merge this branch as-is: current main has moved web-search setup behind provider hooks, while this branch adds Firecrawl-specific core onboarding code and introduces credential-storage regressions.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs returned keys into Firecrawl config, and adds onboarding/auth tests.

## Best Possible Solution

Implement Firecrawl browser sign-in as a Firecrawl-owned web-search provider runSetup with a plugin-owned PKCE helper, SecretInput/SecretRef-aware writes to plugin-scoped webSearch and webFetch credentials, regression tests, and Firecrawl docs.

## Reproduction Assessment

Yes. The blocking problems are source-reproducible from the PR diff and current main: the branch stores a browser-returned raw key after SecretRef resolution and adds Firecrawl-specific setup to a core command wrapper that now delegates to provider runSetup.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable fix is a Firecrawl provider implementation behind runSetup with plugin-scoped, SecretRef-aware credential persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor SecretRef mode before storing browser keys:** `src/commands/onboard-search.ts:422`
  - body: `resolveSearchSecretInput` can convert the search credential to an env SecretRef, but this success path still passes the raw browser-auth API key into the Firecrawl fetch sync helper. A setup running with `secretInputMode: "ref"` can therefore persist plaintext credentials despite the operator requesting references only.
  - confidence: 0.91
- **[P2] Move Firecrawl setup behind the provider hook:** `src/commands/onboard-search.ts:405-414`
  - body: This adds a Firecrawl-specific browser-auth branch to shared onboarding. Current main routes web-search setup through `src/flows/search-setup.ts` and calls each provider's `runSetup`, so this flow should live in the Firecrawl provider instead of core setup.
  - confidence: 0.89
- **[P2] Persist Firecrawl fetch credentials under plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: The helper writes returned keys to legacy `tools.web.fetch.firecrawl.apiKey`, which current Firecrawl config and docs treat as compatibility input. New setup should update `plugins.entries.firecrawl.config.webFetch.apiKey` or a SecretRef through the provider-owned path.
  - confidence: 0.9
- **[P2] Restore fetch in teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-29`
  - body: After `vi.unstubAllGlobals()` this immediately re-stubs `fetch`, so the suite can leave a mocked global fetch installed for later files in non-isolated runs. Let `beforeEach` install the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-115`
  - body: Once `prompter.progress` returns, an unexpected throw before either `spin.stop` call falls into the outer catch and leaves the terminal spinner active. Guard the wait block so the spinner is always stopped.
  - confidence: 0.78
