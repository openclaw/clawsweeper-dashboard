## Summary

Keep open. The feature is valid, but the current branch should not merge as-is because current main routes web-search setup through provider-owned hooks, while the PR hard-codes Firecrawl setup in the command layer and introduces a plaintext credential-storage regression in SecretRef mode.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs the returned key into Firecrawl fetch config, and adds auth/onboarding tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the bundled Firecrawl plugin as a provider-owned `runSetup`, keep the PKCE helper under `extensions/firecrawl`, persist web-search and web-fetch credentials only through plugin-scoped SecretInput/SecretRef-aware paths, and document browser sign-in after that implementation lands.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The blocking issues are visible by comparing the PR diff with current main: the branch writes a raw browser-auth key after SecretInput resolution and targets command-owned setup, while current main exposes provider-owned `runSetup` and SecretRef-aware credential paths.

## Solution Assessment

Is this the best way to solve the issue? No. Browser auth is a good feature direction, but the maintainable solution is provider-owned Firecrawl setup with plugin credential paths rather than a Firecrawl-specific branch in the shared command wrapper.

  - body: `resolveSearchSecretInput` can return an env SecretRef, but this success path still passes the raw `auth.apiKey` into the Firecrawl fetch sync helper.   - confidence: 0.91

  - body: This adds a Firecrawl-specific browser-auth branch to the shared onboarding command. Current main routes web-search setup through `src/flows/search-setup.ts` and calls each provider's `runSetup`, so this flow should live in the Firecrawl provider instead of core setup.
  - confidence: 0.88
- **[P2] Persist Firecrawl fetch keys to plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: The helper writes to legacy `tools.web.fetch.firecrawl.apiKey`, which current docs and doctor migration treat as compatibility input. New setup should update `plugins.entries.firecrawl.config.webFetch.apiKey` or a SecretRef through the provider-owned path.
  - confidence: 0.9
- **[P2] Restore fetch in test teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-30`
  - body: After `vi.unstubAllGlobals()` this immediately re-stubs `fetch`, so the suite leaves a mocked global fetch installed for later files in non-isolated runs. Let `beforeEach` install the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the progress spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-115`

  - confidence: 0.78
