## Summary

Keep this PR open, but the current branch is not merge-ready. The feature is still useful, but current main now routes web-search setup through provider-owned hooks, and this branch stores browser-returned Firecrawl credentials through legacy/plaintext paths.

## What This Changes

Adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs returned keys into Firecrawl config, and adds onboarding/auth tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl web-search provider `runSetup`, persist only plugin-owned `webSearch`/`webFetch` credentials with SecretRef-aware helpers, then add docs and a changelog entry.

## Reproduction Assessment

Yes. The blocking problems are source-reproducible from the PR diff and current main: the branch adds Firecrawl-specific setup to the shared command wrapper and syncs a raw browser API key after SecretRef resolution.

## Solution Assessment

No. Browser sign-in is a reasonable feature, but the maintainable fix is a Firecrawl provider `runSetup` implementation using plugin-scoped, SecretInput-aware credential persistence.
