## Summary

Keep open: the Firecrawl browser sign-in feature is reasonable, but this branch should not merge as-is because current main now routes web-search setup through provider-owned hooks, the PR writes new Firecrawl fetch credentials to legacy config, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs returned keys into search/fetch config, and adds tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin as a web-search provider runSetup, persist SecretRef-aware plugin-scoped webSearch/webFetch credentials, and document the flow after real behavior proof exists.

## Reproduction Assessment

Yes, for the blocking PR review findings: source inspection of the PR head shows Firecrawl-specific setup added to the old command implementation while current main delegates provider setup through runSetup, and the test teardown/spinner issues are directly visible. I did not exercise a live Firecrawl browser-auth session.

## Solution Assessment

No. The feature is useful, but the maintainable solution is a Firecrawl-owned runSetup implementation with plugin-scoped credential writes, not new vendor branches in core onboarding.
