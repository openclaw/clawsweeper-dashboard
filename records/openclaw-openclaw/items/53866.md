## Summary

Keep open as a valid feature request, but this branch should not merge as-is: current main moved provider-specific search setup behind `runSetup`, while the PR adds Firecrawl logic back into the command wrapper, writes legacy Firecrawl fetch config, and still lacks real behavior proof.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper under `src/commands`, wires it into web-search onboarding, syncs Firecrawl search/fetch keys, and adds command tests.

## Best Possible Solution

Implement browser sign-in as the Firecrawl plugin's `WebSearchProvider.runSetup`, persist SecretRef-aware plugin-scoped `webSearch` and `webFetch` credentials, update focused tests/docs, and include redacted live proof.

## Reproduction Assessment

Yes, for the PR review findings: source inspection of current main and the PR diff shows the command-wrapper bypass, legacy config write, fetch teardown leak, and spinner gap. I did not run a live Firecrawl browser-auth session.

## Solution Assessment

No. The feature is useful, but the maintainable solution is a Firecrawl-owned `runSetup` implementation with plugin-scoped credential writes, not a provider-specific branch in core onboarding.
