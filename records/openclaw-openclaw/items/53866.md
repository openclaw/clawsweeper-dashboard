## Summary

Keep open as a useful feature request, but this branch should not merge as-is because current main routes provider-specific web-search setup through the provider `runSetup` seam, Firecrawl credentials are now plugin-scoped, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a core-command Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs Firecrawl keys into web_fetch config, and adds command tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin's `runSetup`, persist plugin-scoped SecretInput-aware `webSearch` and `webFetch` credentials, update focused tests/docs, and attach redacted live proof.

## Reproduction Assessment

Yes, for the PR review findings: source inspection of current main and the PR diff shows the command-wrapper bypass, legacy config write, test teardown leak, and spinner gap. I did not run a live Firecrawl browser-auth session.

## Solution Assessment

No. The feature is useful, but the maintainable path is a Firecrawl-owned `runSetup` implementation with plugin-scoped credential writes, not a provider-specific branch in core onboarding.

Full review comments:

- **[P2] Move Firecrawl browser auth into the provider setup hook:** `src/commands/onboard-search.ts:403-414`
  - body: Current main routes provider-specific web-search setup through `PluginWebSearchProviderEntry.runSetup`; adding Firecrawl browser auth in the command wrapper bypasses that seam and reintroduces provider behavior into core onboarding.
  - confidence: 0.92
- **[P2] Persist Firecrawl fetch keys under plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: The helper writes browser-auth keys to `tools.web.fetch.firecrawl.apiKey`, but that path is legacy compatibility now; new Firecrawl fetch credentials should be written under `plugins.entries.firecrawl.config.webFetch.apiKey`.
  - confidence: 0.9
- **[P2] Restore fetch in teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-30`
  - body: `afterEach` restores globals and then immediately re-stubs `fetch`, leaving the mock installed for later files in non-isolated Vitest runs. Teardown should restore globals and let `beforeEach` install the mock per test.
  - confidence: 0.86
- **[P3] Stop the spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-118`
  - body: After `prompter.progress` returns, an unexpected throw before either `spin.stop` call falls into the outer catch and can leave terminal progress running. Wrap the wait block so failures stop the spinner before returning `null`.
  - confidence: 0.78
