## Summary

Keep open. Firecrawl browser auth is a useful onboarding direction, but this PR is not mergeable as-is because current main now routes web-search setup through provider-owned `runSetup`, while the branch hard-codes Firecrawl in the shared command path and can persist raw browser-auth credentials outside SecretRef-aware plugin config.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs returned keys into Firecrawl fetch config, and adds onboarding/auth tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the bundled Firecrawl plugin as a provider-owned `runSetup`, keep PKCE/browser helpers under `extensions/firecrawl`, persist web-search and web-fetch credentials only through plugin-scoped SecretInput/SecretRef-aware paths, and document the flow after it lands.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The blocking problems are visible from the PR diff against current main: the branch writes a raw browser-auth key after SecretInput resolution and targets command-owned setup while current main exposes provider-owned `runSetup` and plugin credential paths.

## Solution Assessment

Is this the best way to solve the issue? No. The feature direction is good, but the narrow maintainable solution is Firecrawl-owned setup through `PluginWebSearchProviderEntry.runSetup`, not a Firecrawl-specific branch in the shared command wrapper.
