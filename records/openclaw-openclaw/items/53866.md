## Summary

Keep open: the Firecrawl browser-auth setup is a useful feature request and is not implemented on current main, but this branch is not the right merge shape because current main routes web-search setup through provider-owned runSetup and Firecrawl credentials now belong in plugin-scoped config. The external PR also lacks real after-fix behavior proof.

## What This Changes

The branch adds a Firecrawl PKCE browser-auth helper, invokes it from web-search onboarding, writes returned keys into search/fetch config, and adds tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin as a web-search provider runSetup, persist SecretRef-aware plugin-scoped webSearch/webFetch credentials, and document the flow after real behavior proof exists.

## Reproduction Assessment

Yes, for the blocking PR review findings: source inspection shows the branch adds Firecrawl-specific setup to the command path while current main delegates provider setup through runSetup, and the teardown/spinner issues are directly visible in the diff. I did not exercise a live Firecrawl browser-auth session.

## Solution Assessment

No. The feature is useful, but the maintainable solution is a Firecrawl-owned runSetup implementation with plugin-scoped credential writes, not new vendor branches in core onboarding.
