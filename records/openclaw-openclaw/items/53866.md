## Summary

Keep open: the feature is valid, but this branch should not merge as-is because it adds Firecrawl-specific setup in the core command path, stores browser-returned credentials in the legacy fetch path, and can bypass SecretRef expectations. A maintainer review already says to leave it open as a valid request while rewriting it around the provider-owned setup seam.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires browser sign-in into web-search onboarding, syncs returned keys into Firecrawl fetch/search config, and adds tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin as a web-search provider runSetup, store SecretRef-aware plugin-scoped webSearch/webFetch credentials, and document the flow after it lands.

## Reproduction Assessment

Yes, for the blocking review findings. They are source-reproducible from the PR diff against current main: the branch writes the raw browser-auth key into Firecrawl fetch config after SecretInput handling and adds Firecrawl-specific setup in a command wrapper that now delegates to provider runSetup.

## Solution Assessment

No. The feature is reasonable, but the maintainable solution is a Firecrawl-owned runSetup implementation with plugin-scoped, SecretRef-aware credential persistence instead of new Firecrawl branches in core onboarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Honor SecretRef mode before storing browser keys:** `src/commands/onboard-search.ts:422`
  - body: `resolveSearchSecretInput` can return an env SecretRef for the search credential, but this success path then passes the raw `auth.apiKey` into the Firecrawl fetch sync helper. A setup running with `secretInputMode: "ref"` would still write plaintext credentials to config.
  - confidence: 0.91
- **[P2] Move Firecrawl setup behind the provider hook:** `src/commands/onboard-search.ts:403-414`
  - body: This adds a Firecrawl-specific browser-auth branch to shared onboarding. Current main routes web-search setup through `src/flows/search-setup.ts` and invokes each provider's `runSetup`, so this vendor flow should live in the Firecrawl provider instead of core setup.
  - confidence: 0.89
- **[P2] Persist Firecrawl fetch credentials under plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: The helper writes to legacy `tools.web.fetch.firecrawl.apiKey`, which current Firecrawl config and docs treat as compatibility input. New setup should update `plugins.entries.firecrawl.config.webFetch.apiKey` or a SecretRef through the provider-owned path.
  - confidence: 0.9
- **[P2] Restore fetch in teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-29`
  - body: After `vi.unstubAllGlobals()` this immediately re-stubs `fetch`, so the suite leaves a mocked global fetch installed for later files in non-isolated runs. Let `beforeEach` install the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-115`
  - body: Once `prompter.progress` returns, an unexpected throw before either `spin.stop` call falls into the outer catch and leaves the terminal spinner active. Wrap the wait block so the spinner is always stopped on failures.
  - confidence: 0.78
