## Summary

Keep open: the feature is still useful, but this branch conflicts with the provider-owned setup boundary and current Firecrawl credential contracts, and maintainer discussion already asks for a rewrite rather than merging this shape.

## What This Changes

The branch adds a Firecrawl PKCE browser-auth helper, wires Firecrawl browser sign-in into web-search onboarding, syncs returned keys into Firecrawl fetch config, and adds tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin as a web-search provider runSetup, persist only SecretRef-aware plugin-scoped webSearch and webFetch credentials, and document the user flow after it lands.

## Reproduction Assessment

Yes, for the blocking review findings. The issues are source-reproducible from the PR diff against current main: the branch writes raw browser-auth keys after SecretInput handling and puts Firecrawl-specific setup in a command wrapper that now delegates to provider runSetup.

## Solution Assessment

No. The requested feature is reasonable, but the narrow maintainable solution is a Firecrawl-owned runSetup implementation with plugin-scoped, SecretRef-aware credential persistence rather than new Firecrawl branches in core onboarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Honor SecretRef mode before storing browser keys:** `src/commands/onboard-search.ts:422`
  - body: `resolveSearchSecretInput` can return an env SecretRef, but this success path still passes the raw `auth.apiKey` into the Firecrawl fetch sync helper. A setup running with `secretInputMode: "ref"` would persist plaintext credentials even though the operator requested references only.
  - confidence: 0.91
- **[P2] Move Firecrawl setup behind the provider hook:** `src/commands/onboard-search.ts:403-414`
  - body: This adds a Firecrawl-specific browser-auth branch to shared onboarding. Current main routes web-search setup through `src/flows/search-setup.ts` and calls each provider's `runSetup`, so this flow should live in the Firecrawl provider instead of core setup.
  - confidence: 0.89
- **[P2] Persist Firecrawl fetch credentials under plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: The helper writes to legacy `tools.web.fetch.firecrawl.apiKey`, which current Firecrawl config and docs treat as compatibility input. New setup should update `plugins.entries.firecrawl.config.webFetch.apiKey` or a SecretRef through the provider-owned path.
  - confidence: 0.9
- **[P2] Restore fetch in teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-29`
  - body: After `vi.unstubAllGlobals()` this immediately re-stubs `fetch`, so the suite leaves a mocked global fetch installed for later files in non-isolated runs. Let `beforeEach` install the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-115`
  - body: Once `prompter.progress` returns, an unexpected throw before either `spin.stop` call falls into the outer catch and leaves the terminal spinner active. Wrap the wait block so the spinner is always stopped.
  - confidence: 0.78
