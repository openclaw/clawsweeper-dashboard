## Summary

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper under `src/commands`, wires it into web-search onboarding, syncs Firecrawl keys, and adds command tests.

## Best Possible Solution

Implement browser sign-in as Firecrawl plugin-owned `WebSearchProvider.runSetup`, persist SecretRef-aware plugin-scoped `webSearch` and `webFetch` credentials, update focused tests/docs, and include redacted live proof.

## Reproduction Assessment

Yes for the PR review findings: current main source plus the PR diff show the branch bypasses the provider-owned setup seam and uses legacy Firecrawl fetch config. I did not run a live Firecrawl browser-auth session.

## Solution Assessment

No. The feature is useful, but the current branch is not the best solution because Firecrawl-specific browser auth belongs in the Firecrawl provider `runSetup` hook, not in the core command wrapper.

  - body: Current main routes provider-specific web-search setup through `PluginWebSearchProviderEntry.runSetup`; adding Firecrawl browser auth in the command wrapper bypasses that seam and reintroduces provider behavior into core onboarding.

- **[P2] Restore fetch in teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-30`
  - body: `afterEach` restores globals and then immediately re-stubs `fetch`, leaving the mock installed for later files in non-isolated Vitest runs. Teardown should restore globals and let `beforeEach` install the mock per test.
  - confidence: 0.86

  - body: After `prompter.progress` returns, an unexpected throw before either `spin.stop` call falls into the outer catch and can leave terminal progress running. Wrap the wait block so failures stop the spinner before returning `null`.
  - confidence: 0.78
