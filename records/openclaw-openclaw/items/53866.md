## Summary

Keep open: the feature is useful, but this branch targets an obsolete command-owned setup path and would introduce a credential-storage regression against current SecretRef and plugin-boundary behavior.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs returned keys into Firecrawl config, and adds onboarding/auth tests.

## Best Possible Solution

Implement Firecrawl browser sign-in as a Firecrawl-owned web-search provider runSetup that persists only SecretInput/SecretRef-aware plugin-scoped webSearch and webFetch credentials, then document the flow.

## Reproduction Assessment

Yes. The blocking problems are source-reproducible from the PR diff and current main: the branch writes a raw browser-auth key after SecretInput resolution and adds Firecrawl-specific setup to a command wrapper that now delegates to provider runSetup.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable solution is a Firecrawl provider implementation behind runSetup with plugin-scoped, SecretRef-aware credential persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor SecretRef mode before storing browser keys:** `src/commands/onboard-search.ts:422`
  - body: `resolveSearchSecretInput` can return an env SecretRef, but this success path still passes the raw `auth.apiKey` into the Firecrawl fetch sync helper. A setup running with `secretInputMode: "ref"` would persist plaintext credentials even though the operator requested references only.
  - confidence: 0.91
- **[P2] Move Firecrawl setup behind the provider hook:** `src/commands/onboard-search.ts:405-414`
  - body: This adds a Firecrawl-specific browser-auth branch to shared onboarding. Current main routes web-search setup through `src/flows/search-setup.ts` and calls each provider's `runSetup`, so this flow should live in the Firecrawl provider instead of core setup.
  - confidence: 0.89
- **[P2] Persist Firecrawl fetch credentials under plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: The helper writes to legacy `tools.web.fetch.firecrawl.apiKey`, which current Firecrawl config and docs treat as compatibility input. New setup should update `plugins.entries.firecrawl.config.webFetch.apiKey` or a SecretRef through the provider-owned path.
  - confidence: 0.9
- **[P2] Restore fetch in teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-29`
  - body: After `vi.unstubAllGlobals()` this immediately re-stubs `fetch`, so the suite leaves a mocked global fetch installed for later files in non-isolated runs. Let `beforeEach` install the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-115`
  - body: Once `prompter.progress` returns, an unexpected throw before either `spin.stop` call falls into the outer catch and leaves the terminal spinner active. Wrap the wait block so the spinner is always stopped.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI surface, but it introduces a concrete credential-storage regression for Firecrawl browser-auth keys.

Concerns:

- **[high] Plaintext key bypasses SecretRef mode:** `src/commands/onboard-search.ts:422`
