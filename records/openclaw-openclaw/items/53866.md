## Summary

Keep open: the feature is useful, but this branch is not mergeable as-is because current main routes provider-specific web-search setup through the provider hook, while the PR adds Firecrawl-specific command logic, writes legacy fetch config, has blocking behavior/test issues, and lacks required real behavior proof.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs returned keys into search/fetch config, and adds tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin as a web-search provider runSetup, persist SecretRef-aware plugin-scoped webSearch/webFetch credentials, add focused provider/search-flow tests and docs, and include redacted live proof.

## Reproduction Assessment

Yes, for the PR review findings: source inspection of current main and the PR diff shows the command-path bypass, legacy credential write, disabled-state return, fetch teardown leak, and spinner gap. I did not run a live Firecrawl browser-auth session.

## Solution Assessment

No. The feature is valid, but the maintainable solution is a Firecrawl-owned runSetup implementation with plugin-scoped credential writes, not a vendor-specific branch in core onboarding.

Full review comments:

- **[P2] Move Firecrawl setup behind the provider hook:** `src/commands/onboard-search.ts:403-414`
  - body: Current main runs provider-specific web-search setup through PluginWebSearchProviderEntry.runSetup from src/flows/search-setup.ts. Adding Firecrawl browser auth directly in src/commands/onboard-search.ts reintroduces vendor behavior into the command wrapper and misses the provider-owned setup path; move this flow into the Firecrawl provider instead.
  - confidence: 0.91
- **[P2] Persist Firecrawl fetch credentials under plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: The helper stores the browser key in tools.web.fetch.firecrawl.apiKey, but current Firecrawl fetch config is plugins.entries.firecrawl.config.webFetch.apiKey and tools.web.fetch.firecrawl.* is legacy compatibility. New browser-auth credentials should be written to the plugin-owned webFetch path.
  - confidence: 0.9
- **[P2] Return the enabled config after browser auth succeeds:** `src/commands/onboard-search.ts:426`
  - body: This success path returns preserveDisabledState(config, next), so a previously disabled web_search remains disabled after the user completes browser auth and sees Firecrawl connected. Manual key entry enables search for the same setup flow, so browser auth should not leave the provider connected but inactive.
  - confidence: 0.83
- **[P2] Restore fetch in teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-30`
  - body: afterEach calls vi.unstubAllGlobals() and then immediately re-stubs fetch, so this file leaves a mocked global fetch installed for later files in non-isolated runs. Let beforeEach install the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-118`
  - body: Once prompter.progress returns, an unexpected throw before either spin.stop call falls into the outer catch and leaves the terminal spinner active. Wrap the wait block so failures stop the spinner before returning null.
  - confidence: 0.78
