## Summary

Keep this issue open. Current main has provider/model tagging in the qa-lab mock, but the dispatcher still does not use that tag to produce provider-specific scenario plans, and the parity workflows still compare OpenAI and Anthropic through the same mock lane rather than a real-provider agentic parity CI lane.

## What This Changes

Request to make the qa-lab mock dispatcher differentiate OpenAI and Anthropic provider plans, or add a secret-gated live-frontier parity lane for real provider comparison.

## Best Possible Solution

Land the deterministic Option A first: compute a provider variant inside the mock dispatcher, make at least three agentic parity scenarios intentionally return different valid plans for OpenAI and Anthropic, and pin those differences in qa-lab mock server tests. Treat a real-provider parity workflow as a separate maintainer-approved follow-up.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by inspecting buildResponsesPayload(), the Anthropic adapter, the source-discovery mock tests, and the parity workflows; they show the mock lanes still share the same scenario plan path. I did not run tests because this review was required to remain read-only and avoid cache/artifact writes.

## Solution Assessment

Yes, scoped Option A is the best first repair. It is narrow, deterministic, and testable inside the qa-lab mock; Option B can be useful later but should not be bundled into an automated fix because it introduces secret-bearing CI and API budget concerns.
