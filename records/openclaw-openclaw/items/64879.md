## Summary

Keep open. Current main records provider variants for qa-lab mock requests, but the scenario dispatcher still does not use that variant to choose different plans, and the QA parity workflows still run both candidate and baseline through `--provider-mode mock-openai`.

## What This Changes

Request provider-aware qa-lab mock dispatch for GPT-vs-Claude parity scenarios, or a live-frontier parity lane that compares real providers.

## Best Possible Solution

Land deterministic Option A first: feed a normalized provider variant into `buildResponsesPayload()`, make at least three agentic parity scenarios intentionally diverge by provider, and pin those plans in qa-lab mock tests; keep live-frontier parity as a separate policy decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by inspecting the mock dispatcher, Anthropic adapter, source-discovery test, and QA workflows: both provider lanes still share the same prompt-driven mock plan path.

## Solution Assessment

Yes. Scoped Option A is the narrowest maintainable first fix because it is deterministic, local to qa-lab, and testable without introducing secret-bearing live CI in the same change.
