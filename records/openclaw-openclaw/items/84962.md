## Summary

Keep this PR open, but it is not merge-ready: the current patch hardcodes one Railway origin, moves active config away from the persistent Railway volume, broadens forwarded-header trust without evidence, and still lacks real Railway proof.

## What This Changes

The PR adds a Railway deployment config, copies `deploy/` into the Docker runtime image, documents `OPENCLAW_CONFIG_PATH`, and changes the Docker workspace dependency manifest stage from bind mounts to `COPY`.

## Best Possible Solution

Keep Railway on persistent per-deployment config under `/data`, populate `gateway.controlUi.allowedOrigins` from the actual Railway domain, and only trust a verified minimal Railway ingress proxy source.

## Reproduction Assessment

Yes, at source level: a Railway origin other than the hardcoded staging hostname reaches `checkBrowserOrigin` and returns `origin not allowed`. I did not run a live Railway deployment in this read-only review.

## Solution Assessment

No. The patch hardcodes one deployment, points active config at an image path, and bundles broad proxy trust; the safer solution is persistent per-deployment config plus verified narrow proxy trust.
