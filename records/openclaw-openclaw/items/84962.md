## Summary

Keep open: this targets a real Railway Control UI failure, but the current patch is not merge-ready because it still fails non-staging/custom Railway origins, moves active config off the persistent volume, broadens proxy trust without evidence, and has no real Railway proof.

## What This Changes

The PR adds a Railway config with a hardcoded Control UI origin and trusted proxy CIDR, copies `deploy/` into the Docker runtime image, documents `OPENCLAW_CONFIG_PATH`, and changes one Docker workspace-deps bind-mount step to `COPY`.

## Best Possible Solution

Keep Railway on a persistent deployment config path, populate `gateway.controlUi.allowedOrigins` from the actual Railway domain, and only trust a verified minimal Railway ingress proxy source.

## Reproduction Assessment

Yes, source-level reproduction is clear: a Railway origin other than the hardcoded staging hostname reaches `checkBrowserOrigin` and returns `origin not allowed`. I did not run a live Railway deployment.

## Solution Assessment

No. The patch hardcodes one deployment, points active config at an image path, and bundles broad proxy trust; the safer solution is persistent per-deployment config plus verified narrow proxy trust.
