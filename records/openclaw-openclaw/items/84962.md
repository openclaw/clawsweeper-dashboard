## Summary

Keep open: the PR targets a real Railway Control UI origin problem, but the current diff is not merge-ready because it hard-codes one Railway domain, moves active config away from persistent storage, adds a broad trusted-proxy CIDR without proof, and lacks real Railway behavior evidence.

## What This Changes

The branch adds a Railway-specific bundled config with a hardcoded Control UI origin and trusted proxy CIDR, copies `deploy/` into the Docker runtime image, documents `OPENCLAW_CONFIG_PATH` for Railway, and changes the workspace-deps Dockerfile inputs from bind mounts to `COPY`.

## Best Possible Solution

Keep the Railway fix per-deployment and persistent: write or document a `/data` config containing the deployment's actual Railway origin, and only add trusted proxy settings after maintainer security review or Railway-backed proof of the ingress source.

## Reproduction Assessment

Yes, source-reproducible: a Railway deployment opened from a generated or custom origin not present in `gateway.controlUi.allowedOrigins` reaches the current origin check and gets `origin not allowed`. I did not run a live Railway deployment.

## Solution Assessment

No. The PR changes the right surface, but a bundled one-host config plus broad proxy trust is not the narrow maintainable fix; the safer path is persistent per-deployment config with the actual Railway origin and reviewed proxy settings.
