## Summary

## What This Changes

The branch adds a bundled Railway gateway config, copies `deploy/` into the Docker runtime image, switches one Docker workspace manifest extraction step from bind mounts to `COPY`, and documents `OPENCLAW_CONFIG_PATH` for Railway.

## Best Possible Solution

Use the existing config keys to seed or document a per-deployment `/data` config whose allowed origin matches the actual Railway domain and whose trusted proxy list is minimal, proven, or operator-supplied.

## Reproduction Assessment

Yes at source level: current Railway docs send users to generated or custom Railway domains, while `checkBrowserOrigin` rejects public origins that are not explicitly allowlisted. I did not run a live Railway deployment in this read-only review.

## Solution Assessment

No. A hardcoded bundled config is not the best fix; the maintainable path is per-deployment persistent config plus a proven minimal proxy trust boundary.
