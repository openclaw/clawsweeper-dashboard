## What This Changes

The branch adds a bundled Railway config with a hardcoded Control UI origin and trusted proxy CIDR, copies `deploy/` into the Docker image, documents `OPENCLAW_CONFIG_PATH` for Railway, and changes the Docker workspace-deps input from bind mounts to `COPY`.

## Best Possible Solution

Keep Railway on a persistent deployment config path, populate `gateway.controlUi.allowedOrigins` from the actual Railway domain, and only trust a verified minimal Railway ingress proxy source.

## Reproduction Assessment

Yes, source-level reproduction is clear: a Railway origin other than the hardcoded staging hostname reaches `checkBrowserOrigin` and returns `origin not allowed`. I did not run a live Railway deployment.

## Solution Assessment

No. The proposed fix hardcodes one deployment and moves active config out of the persistent volume; the maintainable fix is per-deployment persistent config plus verified narrow proxy trust.
