## Summary

Keep open: the Railway origin problem is real at source level, but this patch is not merge-ready because it hard-codes one Railway hostname, moves active config away from persistent storage, broadens trusted-proxy scope without provider proof, and still has no real Railway proof.

## What This Changes

The PR changes Docker manifest staging, adds a bundled Railway gateway config, copies deploy assets into the runtime image, and documents using that bundled config through `OPENCLAW_CONFIG_PATH`.

## Best Possible Solution

Use the existing config keys to seed or document a per-deployment persistent `/data` config whose origin matches the actual Railway domain and whose proxy trust is minimal, proven, or operator-supplied, then verify `/openclaw` in a real Railway deployment.

## Solution Assessment

No. A hard-coded bundled config is not the best fix; the maintainable path is per-deployment persistent config plus a proven minimal proxy trust boundary.

Full review comments:

- **[P1] Do not hard-code one Railway origin:** `deploy/railway/openclaw.json:6`
  - body: The Railway guide says each service can use a generated or custom domain, but this config only allows one staging hostname. Any other Railway domain still reaches the current origin check and fails with `origin not allowed`, so the origin needs to come from the actual deployment or operator config.
  - confidence: 0.94
- **[P1] Keep Railway config on persistent storage:** `docs/install/railway.mdx:66`
  - body: The docs promise `openclaw.json` survives redeploys under `/data`, but this required `OPENCLAW_CONFIG_PATH` points at a file under `/app`. Runtime edits and custom-domain fixes would live outside the Railway volume and can disappear on redeploy; seed or point to a config under `/data` instead.
  - confidence: 0.91
- **[P1] Do not trust the whole CGNAT range as a proxy:** `deploy/railway/openclaw.json:10`
  - body: `gateway.trustedProxies` makes forwarded-client headers authoritative when the remote address matches the list, and current docs require only actual proxy IPs. Without Railway-specific proof that every `100.64.0.0/10` source is the intended ingress boundary, this broad CIDR weakens the proxy trust boundary.
  - confidence: 0.89
- **[P3] Preserve manifest-only dependency staging:** `Dockerfile:30-31`
  - body: Replacing bind-mounted manifest extraction with full `COPY packages` and bundled-plugin directory copies makes source-only package or extension edits invalidate the dependency install layer the Dockerfile is designed to preserve. If Railway cannot use bind mounts, use a manifest-only copy strategy rather than copying whole trees into the dependency stage.
  - confidence: 0.76
- **[P3] Route Railway deploy config through labeler:** `deploy/railway/openclaw.json:1`
  - body: This PR adds a runtime deploy config under `deploy/railway`, but labeler only routes `deploy/fly.private.toml` under the Docker label. Future changes to this config alone can miss Docker or ops/security review labels unless `deploy/railway/**` or a broader `deploy/**` rule is added.
  - confidence: 0.78
