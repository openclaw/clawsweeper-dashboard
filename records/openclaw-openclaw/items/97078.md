## Summary

Keep open: the amended PR now follows the opt-in provider shape, but it is not merge-ready because `firecrawl-free` can still send configured Firecrawl credentials, reports/caches results as `firecrawl`, and lacks after-fix OpenClaw proof.

## What This Changes

The PR adds an opt-in `firecrawl-free` web_search provider, allows keyless Firecrawl search requests through the client, expands `firecrawl_search` options, and updates plugin metadata, docs, catalog data, doctor migration metadata, and tests.

## Best Possible Solution

Keep the two-provider opt-in design, but make `firecrawl-free` truly anonymous, preserve `firecrawl-free` in result/cache metadata, document Firecrawl keyless limitations, and add real OpenClaw behavior proof before merge.

## Reproduction Assessment

Yes for the blocking review findings: source inspection shows `firecrawl-free` passes full config into a client that resolves and sends configured Firecrawl keys, and the same client hard-codes result provider identity to `firecrawl`. A live unauthenticated Firecrawl search from this environment was rejected, so positive end-to-end behavior remains unproven.

## Solution Assessment

No. The opt-in two-provider direction matches current policy, but the implementation is not the best fix until the free provider has a true no-credential path, distinct result/cache identity, accurate docs/body text, and real OpenClaw proof.
