## Summary

Keep open. Current main still treats `contextInjection` as a defaults-only setting: the runtime resolver reads only `agents.defaults`, the embedded run paths call it without an agent id, and the per-agent config schema/docs do not expose `agents.list[].contextInjection`.

## What This Changes

Request per-agent `agents.list[].contextInjection` with fallback to `agents.defaults.contextInjection` and then the existing `"always"` default.

## Best Possible Solution

Add a narrow per-agent override in the shared agent config contract and make embedded bootstrap/compaction resolution use `agents.list[].contextInjection ?? agents.defaults.contextInjection ?? "always"`.

## Reproduction Assessment

Yes. A config containing `agents.list[].contextInjection` is not part of the strict `AgentEntrySchema`, and even a typed bypass would be ignored because the resolver reads only `agents.defaults.contextInjection`.

## Solution Assessment

Yes. The proposed per-agent config field is the smallest maintainable fix for mixed strict/advisory agents; the related bootstrap budget knobs should stay out of scope unless maintainers explicitly want a broader config pass.
