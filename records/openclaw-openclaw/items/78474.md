## Summary

Keep this PR open: it targets a real channel package-state probe bug, but the proposed path rewrite still appears to fail the loader boundary and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds a built-artifact path rewrite in `package-state-probes` plus a unit test so bundled package-state probes prefer matching `dist/extensions` JavaScript artifacts over source TypeScript files.

## Best Possible Solution

Keep the source fallback, but resolve or pass the matching built plugin root when using a `dist/extensions` artifact, cover the real loader boundary in regression tests, and add after-fix gateway startup proof before merge.

## Reproduction Assessment

Yes, source-backed: current main checks source TypeScript before built JavaScript for extensionless probes, and https://github.com/openclaw/openclaw/issues/78462 includes Matrix and WhatsApp logs showing source `auth-presence.ts` load failures. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. Rewriting only the module path is not enough because the real loader still validates it against the old source root; the safer fix should carry a matching dist boundary/root or resolve from the bundled built root directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep the dist artifact inside the loader boundary:** `src/channels/plugins/package-state-probes.ts:168`
  - body: When `resolvedPath` points at `/repo/dist/extensions/...`, this call still keeps the source `rootDir` (`/repo/extensions/...`). `loadChannelPluginModule` validates the module path under that root before requiring it, so the built artifact is rejected and the checker still logs a load failure. Pass the matching dist plugin root with the rewritten path, or resolve from the bundled built root directly.
  - confidence: 0.88
