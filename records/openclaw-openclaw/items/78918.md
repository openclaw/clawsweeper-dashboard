## Summary

Keep open: current main still has the colon-only active-memory guard, while Google Chat legitimately uses slash-containing `spaces/<spaceId>` values and the bundled public-surface dirName validator still throws on `/`. The closed related fix at https://github.com/openclaw/openclaw/pull/78924 was not merged, although the reporter later added useful branch-level proof that it worked.

## What This Changes

The issue requests fixing active-memory recall for Google Chat sessions whose slash-containing `spaces/<spaceId>` value can be treated as a bundled plugin dirName.

## Best Possible Solution

Land a narrow active-memory fix that keeps bundled dirName validation intact, treats slash-bearing `spaces/<id>` values as conversation identity rather than runnable channel names, and adds regression coverage plus changelog.

## Reproduction Assessment

Yes, source-backed: a Google Chat `spaces/<id>` value can pass the current colon-only active-memory resolver and then hit the bundled dirName validator, which rejects `/` with the reported error. I did not live-run the Docker Google Chat path.

## Solution Assessment

Yes: the narrow maintainable fix is to filter or normalize slash-bearing conversation identifiers in active-memory channel-context resolution, not to relax the public-surface directory validator or change Google Chat identifiers.
