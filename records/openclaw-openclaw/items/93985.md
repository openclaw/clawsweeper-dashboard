## Summary

Keep open: this PR targets a real Docker duplicate-mount failure, but the branch currently makes user binds override protected read-only skill mounts, which is a sandbox security and compatibility decision that should not land without maintainer approval or a safer precedence-preserving fix.

## What This Changes

The branch filters read-only workspace skill mounts out when configured Docker/browser binds target the same container path, applies that filtered mount list to shell/browser/fs bridge modeling, updates sandbox tests, and adds a demo proof script.

## Best Possible Solution

Land one canonical sandbox mount fix that removes duplicate Docker targets across shell Docker, browser Docker, config hashing, and filesystem bridge modeling while keeping protected read-only skill mounts authoritative by skipping or rejecting colliding user binds.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and v2026.6.10 append custom binds before protected read-only skill mounts in shell and browser Docker create args, so an allowed overlapping bind can produce duplicate Docker targets. I did not run a live Docker daemon in this read-only review.

## Solution Assessment

No. Filtering duplicate targets is the right area, but this branch drops protected overlays; the safer fix keeps protected mounts authoritative and skips or rejects the colliding user bind unless maintainers explicitly choose user-bind precedence.
