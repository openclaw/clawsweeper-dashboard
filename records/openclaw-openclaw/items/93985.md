## Summary

Keep open: the PR addresses a real Docker sandbox duplicate-mount bug and includes terminal Docker proof, but it reverses the current protected read-only skill mount invariant by making the user bind authoritative over skill roots.

## What This Changes

The PR filters read-only workspace skill mounts when Docker/browser binds target the same container path, updates filesystem mount modeling and sandbox tests, and adds a demo script.

## Best Possible Solution

Land one canonical sandbox mount fix that removes duplicate Docker targets across shell Docker, browser Docker, config hashing, and filesystem bridge modeling while keeping protected read-only skill mounts authoritative unless maintainers explicitly choose a different security contract.

## Reproduction Assessment

Yes. Current main and v2026.6.10 append custom binds before read-only skill mounts in shell and browser Docker create args, so an allowed overlapping bind can produce duplicate Docker destinations; I did not run a live Docker daemon in this read-only review.

## Solution Assessment

No. Filtering duplicate targets belongs in sandbox mount policy, but this branch drops protected read-only overlays; the safer fix skips or rejects colliding user binds unless maintainers explicitly accept user-bind precedence.
