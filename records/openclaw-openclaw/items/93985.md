## Summary

Keep open: this PR targets a real source-reproducible Docker duplicate-mount bug and now includes plausible terminal proof, but the patch removes protected read-only skill mounts when a user bind collides, which reverses the existing sandbox skill protection contract.

## What This Changes

The branch filters read-only workspace skill mounts against configured Docker/browser binds, threads that filtered list into filesystem path modeling, updates sandbox tests, and adds a sandbox bind-filter demo script.

## Best Possible Solution

Land one canonical sandbox mount fix that removes duplicate Docker targets across shell Docker, browser Docker, config hashing, and filesystem bridge modeling while preserving protected read-only skill mounts, preferably by skipping or rejecting the colliding user bind.

## Reproduction Assessment

Yes. Current main and v2026.6.10 append custom binds before read-only skill mounts in shell and browser Docker create args, so a dangerous reserved-target bind can produce duplicate Docker targets; I did not run a live Docker daemon in this read-only review.

## Solution Assessment

No. Filtering duplicate targets is the right area, but this branch makes user binds authoritative over protected skill mounts; the safer fix preserves protected mounts and removes or rejects the colliding user bind.
