## Summary

Keep open. The PR targets a real user-facing CLI problem, but the current patch is not merge-ready: the new warning path only handles thrown usage-load failures, while the current Claude `user:profile` failure path normally resolves as a provider error snapshot and is still formatted as a raw `HTTP 403` line. The classifier also suppresses diagnostics for any Claude 403 without requiring an explicit `user:profile` signal, and the user-facing CLI fix needs a changelog entry.

## What This Changes

The branch adds structured text-mode warnings for `openclaw channels list` usage-load exceptions, adds focused command tests and docs for that warning, and includes small MiniMax test/docs ordering cleanups.

## Best Possible Solution

The text `channels list` path should classify explicit Claude `user:profile` failures whether they arrive as thrown load failures or resolved provider error snapshots, keep raw diagnostics for non-scope failures, and align tests, docs, and changelog. The separate JSON fallback behavior should remain tracked by #59472 unless intentionally folded in.

## Reproduction Assessment

Yes. A mocked Claude OAuth 403 response with `scope requirement user:profile` and no valid Claude web session makes `fetchClaudeUsage` return a `ProviderUsageSnapshot.error`; `channelsListCommand` then passes that resolved summary to `formatUsageReportLines`, so the PR's catch block is not reached.

## Solution Assessment

No. The proposed catch-only warning is not the narrowest correct fix because it misses resolved snapshot errors; the safer approach is to classify explicit scope errors at the text usage-formatting boundary while preserving raw diagnostics for other 403s.
