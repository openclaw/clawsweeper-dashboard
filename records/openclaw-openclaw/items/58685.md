## Summary

Keep open. Current main still has the raw Claude usage-error path this PR is trying to improve, but the patch is not merge-ready because its new warning only handles thrown usage-load failures and can hide useful diagnostics for non-scope Claude 403s.

## What This Changes

The PR adds CLI warnings for `openclaw channels list` usage-fetch failures, targeted usage-error tests, a channels docs update, and small unrelated docs/test ordering cleanups.

## Best Possible Solution

Handle usage failure messaging in a shared CLI formatting path that classifies both thrown load failures and resolved `ProviderUsageSnapshot.error` entries, only suppresses raw details for explicit `user:profile` scope errors, and updates docs, tests, and changelog together.

## Reproduction Assessment

Yes. A mocked Claude OAuth 403 body containing `scope requirement user:profile` with no valid Claude web session makes `fetchClaudeUsage` return a `ProviderUsageSnapshot.error`; `channelsListCommand` then formats that snapshot through `formatUsageReportLines` instead of entering the PR's catch block.

## Solution Assessment

No. The catch-only patch is not the narrowest correct fix because the common Claude scope path resolves successfully with an error snapshot; the safer approach is to classify explicit scope errors at the snapshot-formatting boundary while preserving raw diagnostics for other 403s.
