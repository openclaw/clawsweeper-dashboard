## Summary

Keep open. Current main still gives gateway-executed plugin message actions an explicit 10,000 ms caller timeout, the CLI media send path does not expose or pass a longer timeout, and the issue is already paired with open unmerged closing PR #74169.

## What This Changes

The issue asks for `openclaw message send --channel wecom --media <~10MB zip>` to complete instead of failing with `gateway timeout after 10000ms`.

## Best Possible Solution

Review, revise, or replace #74169 with a scoped core fix that gives gateway-executed plugin media sends a longer or configurable message-action timeout and adds regression coverage, without adding WeCom-specific policy to core.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter's CLI command plus current main's explicit `message.action` 10,000 ms wrapper means any gateway-executed plugin media send that takes longer than 10 seconds can reproduce the observed error.

## Solution Assessment

No, current main is not the best solution because the failing timeout path remains. A targeted message-action or media-send timeout fix with tests is safer than relying only on a generic handshake default change.
