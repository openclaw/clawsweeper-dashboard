## Summary

Keep open. Current `main` still has a concrete 10,000 ms caller-side timeout on gateway-executed plugin message actions, the CLI send path does not expose or pass a longer media-send timeout, and open unmerged PR #74169 is already the paired implementation candidate for this issue.

## What This Changes

The issue reports that `openclaw message send --channel wecom ... --media <about 10MB zip>` fails with `gateway timeout after 10000ms` and requests successful large-file delivery.

## Best Possible Solution

Review, revise, or replace #74169 with a scoped generic core fix that gives gateway-executed plugin media sends a longer or configurable message-action timeout, plus regression coverage for a delayed gateway plugin send over 10 seconds, without adding WeCom-specific policy to core.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter's CLI command plus current `main`'s explicit `message.action` 10,000 ms wrapper means any gateway-executed plugin media send that takes longer than 10 seconds can reproduce the observed error.

## Solution Assessment

No, current `main` is not the best solution because it still has the failing timeout path. The safer direction is a targeted message-action/media-send timeout path with tests; changing only a generic handshake default is not clearly sufficient while the runner supplies an explicit 10,000 ms timeout.
