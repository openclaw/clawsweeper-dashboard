## Summary

Keep open. Current main still gives gateway-executed plugin message actions an explicit 10,000 ms caller timeout, the CLI media send path does not pass a longer timeout or expose a timeout flag, and the linked closing PR #74169 is still open and unmerged.

## What This Changes

The issue requests that `openclaw message send --channel wecom --media <10 MB file>` complete instead of failing with a 10,000 ms gateway timeout.

## Best Possible Solution

Review, revise, or replace #74169 with a scoped core fix that gives gateway-executed plugin media sends a longer or configurable message-action timeout, plus regression coverage for a delayed gateway plugin send over 10 seconds, without adding WeCom-specific policy to core.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter's CLI command plus current main's explicit `message.action` 10,000 ms wrapper means any gateway-executed plugin media send that takes longer than 10 seconds can reproduce the observed error.

## Solution Assessment

No, current main is not the best solution because it still has the failing timeout path. A targeted message-action/media-send timeout path with tests is safer than relying only on a generic handshake default bump.
