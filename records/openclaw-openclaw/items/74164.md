## Summary

Keep open. Current main still sends gateway-executed plugin message actions with an explicit 10000 ms timeout, and an open closing PR is already paired with the report.

## What This Changes

The issue reports `openclaw message send --channel wecom --media <10 MB zip>` timing out through the local gateway after 10000 ms on OpenClaw 2026.4.15.

## Best Possible Solution

Land or replace the linked PR with a narrow fix that makes gateway-executed message actions use the intended configurable longer timeout, with coverage for the no-config `message send` path.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: the CLI supplies gateway options without a timeout, the runner defaults to 10000 ms, and gateway-mode plugin message actions pass that timeout into `message.action`; live WeCom access would only prove the full upload end to end.

## Solution Assessment

Unclear for the linked PR as described, because changing only `resolveGatewayCallTimeout` may not affect message-action callers that already pass an explicit 10000 ms timeout. The safer fix should cover `resolveGatewayActionOptions` or stop forcing the shorter default for gateway-executed sends.
