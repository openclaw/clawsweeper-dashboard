## Summary

Keep open. Current main still routes gateway-executed CLI message sends through a 10000 ms default, and the issue is already paired with an open unmerged closing PR, so it should stay open until that implementation lands or is replaced.

## What This Changes

The issue requests that `openclaw message send --channel wecom --media <large file>` succeed instead of timing out after 10000 ms.

## Best Possible Solution

A narrow fix should make gateway-executed message sends use the intended longer, handshake-aligned timeout and cover the no-config CLI path with regression tests.

## Reproduction Assessment

Yes. Source-level current-main reproduction is high confidence: CLI message send omits a timeout override, the message-action runner fills in `10_000`, and gateway-executed plugin sends use that RPC path.

## Solution Assessment

Unclear for the linked PR as described. Changing only `src/gateway/call.ts` may not affect callers that already pass `timeoutMs: 10_000`, so the safer fix should cover the message-action timeout resolver or stop forcing the shorter default there.
