## Summary

Keep open. Current main still applies an explicit 10,000 ms timeout to gateway-executed plugin message actions, the CLI media send path does not pass a longer timeout or expose a timeout flag, and the linked closing PR #74169 is still open and unmerged.

## What This Changes

The issue reports that `openclaw message send --channel wecom --media <~10MB zip>` fails through the local gateway with `gateway timeout after 10000ms` instead of sending the file.

## Best Possible Solution

A scoped core fix should let gateway-executed plugin media sends wait long enough via an explicit message-action timeout path or documented configuration, with regression coverage for a delayed gateway plugin send and without adding WeCom-specific policy to core.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter's CLI command plus current main shows `message.action` is called with an explicit 10,000 ms wrapper timeout; a delayed gateway-executed plugin send over 10 seconds or the reported 10 MB WeCom upload exercises that path.

## Solution Assessment

No for treating a generic default bump as proven complete. The safer solution is to adjust or make configurable the message-action/media-send timeout boundary and add a focused regression test showing a gateway plugin send can exceed 10 seconds without failing.
