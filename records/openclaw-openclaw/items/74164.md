## Summary

Keep open. Current main still routes the reported `openclaw message send --media` path through an explicit 10000 ms `message.action` gateway timeout, and the linked closing PR at https://github.com/openclaw/openclaw/pull/74169 remains open and does not cover that explicit message-action timeout path.

## What This Changes

The issue reports that sending an approximately 10 MB media file through `openclaw message send --channel wecom` fails with a 10000 ms gateway timeout.

## Best Possible Solution

Review, revise, or replace the active timeout PR so gateway-executed message actions use the shared timeout policy or a deliberate CLI timeout while preserving explicit caller overrides and adding focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. The reported CLI command enters `messageCommand` without a timeout override, `resolveGatewayActionOptions` fills `10_000`, and gateway-mode plugin sends pass that explicit timeout to the `message.action` RPC.

## Solution Assessment

No, not yet. Raising only the generic no-timeout gateway fallback misses the reported path because message-action dispatch already supplies `timeoutMs: 10_000`; the fix needs to cover that resolver or intentionally thread a larger command timeout.
