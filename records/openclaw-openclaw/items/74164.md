## Summary

Keep open. Current main still routes gateway-executed plugin message actions through an explicit 10,000 ms caller timeout, the CLI media send path does not pass a longer timeout or expose a timeout flag, and the linked closing PR #74169 is still open and unmerged.

## What This Changes

The issue asks for large WeCom media sends through `openclaw message send` to complete instead of failing with a 10,000 ms gateway timeout.

## Best Possible Solution

Review, revise, or replace #74169 with a scoped core fix that gives gateway-executed plugin media sends an explicit longer or configurable message-action timeout, plus regression coverage for a delayed gateway plugin send over 10 seconds, without adding WeCom-specific policy to core.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter's CLI command plus current main's explicit `message.action` 10,000 ms wrapper means any gateway-executed plugin media send that takes longer than 10 seconds can reproduce the observed error.

## Solution Assessment

No, current main is not the best solution because it still has the failing timeout path. The safest fix is a targeted message-action/media-send timeout path with tests; a generic handshake default bump alone is not clearly sufficient while the runner supplies an explicit 10,000 ms timeout.
