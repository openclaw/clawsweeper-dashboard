## Summary

Keep open. Current `main` still has a source-level path that can force gateway-executed `message.action` sends through a 10000 ms wrapper timeout, and this issue is already paired with open PR #74169.

## What This Changes

The issue reports that sending a roughly 10 MB file through the external WeCom plugin with `openclaw message send --media` times out after 10000 ms.

## Best Possible Solution

Land or replace the linked PR with a narrow fix that gives gateway-executed message sends an appropriately longer default timeout, with coverage for the no-config/no-env `message send --media` path.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the CLI send command reaches the gateway message-action runner with no explicit timeout, and the runner supplies 10000 ms to the gateway call.

## Solution Assessment

Unclear. Raising only the generic gateway fallback may be insufficient because the message-action runner currently passes an explicit 10000 ms timeout; the maintainable fix should cover that send/action timeout boundary.
