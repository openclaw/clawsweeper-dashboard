## Summary

Keep open. Current main still sends gateway-executed plugin message actions with an explicit 10,000 ms timeout, the CLI media send path does not expose or forward a longer timeout, and the linked closing PR is still only an open implementation candidate rather than merged behavior.

## What This Changes

The reporter asks for `openclaw message send --channel wecom --media` to send a roughly 10 MB file successfully instead of failing with `gateway timeout after 10000ms`.

## Best Possible Solution

Add a generic, documented timeout budget for gateway-executed message actions, especially media sends, and cover it with a regression test for a delayed plugin `message.action` response. Keep WeCom-specific upload behavior in the external plugin and avoid hardcoding WeCom policy in core.

## Reproduction Assessment

Yes for the core timeout boundary. A gateway-executed channel plugin `send` action that takes more than 10 seconds, including the reporter's roughly 10 MB WeCom upload, follows the current `message.action` path with an explicit 10,000 ms wrapper timeout.

## Solution Assessment

No for the current main behavior, and unclear for the linked fallback-only PR. Because the message-action runner passes an explicit 10,000 ms timeout, changing only the generic fallback in `src/gateway/call.ts` may not fix this CLI media-send path.
