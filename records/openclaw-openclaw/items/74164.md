## Summary

Keep open. Current main still has the 10000 ms timeout on the CLI gateway-executed message action path, and an open closing PR is already the implementation lane for this report.

## What This Changes

The issue reports that `openclaw message send --channel wecom --media <10MB zip>` times out through the local gateway after 10000 ms.

## Best Possible Solution

Land or replace the linked PR with a narrow fix that makes gateway-executed message sends use the intended longer timeout, including the no-config/no-env CLI path.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: `openclaw message send` reaches `message.action` with an explicit default `timeoutMs: 10000`; a live WeCom account is only needed for end-to-end upload proof.

## Solution Assessment

Unclear for the linked PR as described, because changing only the generic fallback may not affect callers that already pass `timeoutMs: 10000`. The safer fix should cover the message action timeout resolver or stop forcing the shorter default for gateway-executed sends.
