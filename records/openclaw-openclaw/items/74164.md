## Summary

Keep open. Current main still routes gateway-executed plugin message sends through an explicit 10,000 ms caller timeout, the CLI media-send path does not pass or expose a longer timeout, and the issue is already paired with open unmerged closing PR #74169.

## What This Changes

Allow sending an approximately 10 MB file through the WeCom channel via `openclaw message send --media` without the CLI gateway call timing out after 10 seconds.

## Best Possible Solution

Review, revise, or replace #74169 with a scoped core fix that gives gateway-executed plugin media sends an explicit longer or configurable message-action timeout, plus regression coverage for a delayed gateway plugin send over 10 seconds, without adding WeCom-specific policy to core.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter's CLI command plus current main's explicit `message.action` 10,000 ms wrapper means any gateway-executed plugin media send that takes longer than 10 seconds can reproduce the observed error.

## Solution Assessment

No, current main is not the best solution because it still has the failing timeout path. The safer fix is a targeted message-action/media-send timeout path with tests; a generic handshake-default bump alone is not clearly sufficient while the runner supplies an explicit 10,000 ms timeout.
