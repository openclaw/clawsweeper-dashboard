## Summary

Keep open. Current main still applies a 10000 ms timeout to gateway-executed message actions when the CLI sends through a plugin without a timeout override, and the linked closing PR remains open and unmerged.

## What This Changes

The issue reports that `openclaw message send --channel wecom --media ...` fails for an approximately 10 MB file with `gateway timeout after 10000ms`.

## Best Possible Solution

Review, land, or replace the linked timeout PR with a fix that aligns gateway-executed message actions with the shared timeout policy while preserving explicit caller overrides and adding focused regression coverage.

## Reproduction Assessment

Yes by source inspection. The reported CLI command enters `messageCommand` without a timeout override, `resolveGatewayActionOptions` fills `10_000`, and gateway-mode plugin sends pass that value to the `message.action` RPC.

## Solution Assessment

Unclear for the linked PR as currently scoped. Changing only the generic gateway fallback may not fix callers that already pass `timeoutMs: 10_000`, so the durable solution should cover the message-action timeout resolver or CLI gateway options too.
