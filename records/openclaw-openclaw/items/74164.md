## Summary

Keep open. Current main still routes gateway-executed plugin message actions through an explicit 10,000 ms client timeout, the CLI `message send` path does not pass a longer timeout or expose a timeout flag, and the linked closing PR #74169 is still open and unmerged in the provided issue context.

## What This Changes

The issue reports that sending an approximately 10 MB file through the WeCom channel with `openclaw message send --media` fails with `gateway timeout after 10000ms`.

## Best Possible Solution

Add a scoped, channel-agnostic timeout path for gateway-executed plugin media sends, either by passing an explicit longer action timeout from the message CLI/tool path or by documenting/configuring that boundary, with regression coverage for a delayed gateway plugin send. Keep WeCom-specific upload policy in the WeCom plugin.

## Reproduction Assessment

Yes. The reporter's command plus current main's source path gives a high-confidence core reproduction boundary: a gateway-executed plugin `send` taking longer than 10 seconds will hit the explicit `message.action` wrapper timeout.

## Solution Assessment

No for a default fallback-only fix. Because `runMessageAction` currently passes an explicit 10,000 ms timeout into the gateway call, the safer fix is to adjust the message-action/media-send timeout boundary itself and cover that path with a focused test.
