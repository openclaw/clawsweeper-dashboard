## Summary

Keep open. Current main still sends the reported CLI media-send path through a core `message.action` gateway call with an explicit 10000 ms timeout, and the open closing PR at https://github.com/openclaw/openclaw/pull/74169 is unmerged and does not cover that explicit message-action timeout path.

## What This Changes

The issue reports that `openclaw message send` times out after 10000 ms when sending an approximately 10 MB media file through the external WeCom channel.

## Best Possible Solution

Review, revise, or replace the linked timeout PR so gateway-executed message actions use the shared timeout policy while preserving explicit caller overrides and adding focused regression coverage.

## Reproduction Assessment

Yes by source inspection. The reported CLI command enters `messageCommand`, receives no timeout override, resolves `gateway.timeoutMs` to `10_000`, and sends that explicit value through the `message.action` gateway RPC.

## Solution Assessment

No for the linked PR as currently scoped. Raising only the generic no-timeout gateway fallback is not sufficient because the reported message-action path already passes an explicit `timeoutMs: 10_000`.
