## Summary

Keep open. Current main still applies a 10000 ms timeout to gateway-executed plugin message actions on the reported CLI path, and the linked closing PR remains open and unmerged.

## What This Changes

The issue reports that `openclaw message send --channel wecom --media <10MB zip>` fails with `gateway timeout after 10000ms` instead of sending the file successfully.

## Best Possible Solution

Review, land, or replace the linked timeout PR with a fix that aligns gateway-executed message-action timeouts with the shared timeout policy while preserving explicit caller overrides and adding focused regression coverage.

## Reproduction Assessment

Yes by source inspection. The reported CLI command enters `messageCommand` without a timeout override, `resolveGatewayActionOptions` fills `10_000`, and gateway-mode plugin sends pass that value to the `message.action` RPC.

## Solution Assessment

No merged solution exists yet. The active linked PR is the right implementation lane, but the durable fix should cover the message-action timeout resolver or CLI gateway options, not only the generic no-timeout gateway fallback.
