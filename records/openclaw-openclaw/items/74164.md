## Summary

Keep open. Current main still sends gateway-executed plugin message actions with an explicit 10,000 ms wrapper timeout, and the linked closing PR is open rather than merged; the source also shows that changing only the generic gateway-call fallback would not cover this CLI path because the message-action runner already passes a timeout value.

## What This Changes

The issue reports a regression where `openclaw message send --channel wecom ... --media <10 MB zip>` times out at the gateway after 10,000 ms instead of delivering the file.

## Best Possible Solution

Keep this issue open until #74169 is revised, landed with a fix that actually covers the explicit `message.action` timeout path, or replaced with a narrow generic change for gateway-executed plugin media sends. The fix should avoid WeCom-specific core policy and should add focused regression coverage for a gateway plugin send that takes longer than 10 seconds.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter supplied an exact CLI command and current main shows gateway-executed plugin sends wait on `message.action` under an explicit 10,000 ms wrapper timeout; a delayed gateway plugin action or the reported large upload exercises the same path.

## Solution Assessment

No for a generic default-only timeout bump as the complete solution. The maintainable fix needs to adjust the explicit message-action/media-send timeout boundary or provide an intentional scoped override, because this CLI path already passes `timeoutMs: 10_000`.
