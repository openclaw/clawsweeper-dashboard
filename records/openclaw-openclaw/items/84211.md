## What This Changes

The branch keeps the Codex app-server `message` dynamic tool direct across source-reply delivery modes and adds regression coverage for resuming a Codex thread when the mode toggles to `message_tool_only`.

## Best Possible Solution

Land a Codex-owned fix that keeps `message` direct across delivery modes, keeps `message_tool_only` enforcement per turn, and either proves or explicitly accepts the one-time existing-binding rotation risk.

Yes from source inspection: current main changes the `message` tool spec between automatic and `message_tool_only`, and thread lifecycle compares those fingerprint bytes before resume. I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes: stabilizing `message` at the bridge is a narrow fix that preserves per-turn visible-reply instructions while keeping other OpenClaw tools searchable. The remaining question is upgrade safety for bindings that already stored the old deferred-message fingerprint.
