## Summary

Keep open. Current main still handles Feishu bot added/removed subscription events by parsing them and writing logs only; the hook docs and typed hook surface do not document or expose the requested Feishu membership hook events.

## What This Changes

The issue requests hook events for Feishu bot added and removed group-chat membership events, including chat, account, and timestamp context.

## Best Possible Solution

Add a Feishu-owned hook emission path for bot added and removed events, preserve the existing logs, document the new hook keys/context, and cover the dispatcher with focused Feishu monitor tests. If maintainers prefer a generic channel lifecycle convention, make that hook type additive and reusable rather than hard-coding an untyped bypass.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-verifiable: invoking the registered Feishu bot added/deleted handlers on current main would only reach the log branches in `extensions/feishu/src/monitor.account.ts`.

## Solution Assessment

Yes, exposing these already-received Feishu subscription events through the existing hook pipeline is the narrowest maintainable direction. The safer implementation should stay in the Feishu plugin plus the documented hook contract, without adding polling or moving Feishu-specific policy into core.
