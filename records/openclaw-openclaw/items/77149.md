## Summary

Keep open: current main still suppresses normal final replies in default group/channel message-tool mode when the message tool is available, and this local-LLM non-invocation case is distinct from the already-closed #74868 unavailable-tool fix. Changing that fallback would override documented private-by-default group behavior, so it needs maintainer product judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to avoid silently losing Discord and Telegram channel replies from local LLM providers when the model returns plain text instead of calling the message tool.

## Best Possible Solution

Maintainers should choose the product policy for tool-available-but-not-called local-provider turns, then implement either a narrow compatibility path or stronger diagnostics without weakening group/channel privacy by accident.

## Reproduction Assessment

Yes. Source inspection and tests show that default group/channel turns use `message_tool_only`, suppress normal final delivery, and only fall back when the message tool is unavailable or config is set to `automatic`.

## Solution Assessment

Unclear. A plain-text fallback would restore legacy local-LLM behavior, but an unconditional fallback conflicts with the documented private-by-default channel policy; a safer path may be explicit config, doctor/provider guidance, or an opt-in heuristic after maintainer review.
