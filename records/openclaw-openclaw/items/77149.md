## Summary

Keep open: current main still has the reported tool-available-but-not-called path, where default group/channel turns suppress normal final replies even when the model generated assistant text. That behavior is now documented as intentional privacy/noise policy, so changing the fallback for local providers needs maintainer product judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to avoid silently suppressing plain-text Discord/Telegram channel replies from local LLMs under the default group/channel message-tool-only reply mode.

## Best Possible Solution

Maintainers should choose the product policy for tool-available-but-not-called local-provider turns, then implement either a narrow compatibility path or stronger in-product diagnostics without accidentally weakening group/channel privacy defaults.

## Reproduction Assessment

Yes. Source inspection and regression tests show that default group/channel turns use `message_tool_only`, suppress normal final delivery, and only fall back when the message tool is unavailable or config is set to `automatic`.

## Solution Assessment

Unclear. A plain-text fallback would restore legacy local-LLM behavior, but an unconditional fallback conflicts with the documented private-by-default channel policy; a safer path may be explicit config, doctor/provider guidance, or an opt-in heuristic after maintainer review.
