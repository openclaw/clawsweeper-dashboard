## Summary

Keep open: the branch addresses a real source-level sticky fallback path, but the central behavior is currently documented and tested as the compatibility default, so the unconditional rollback needs maintainer policy approval plus external real behavior proof before merge.

## What This Changes

This PR changes agent fallback/session accounting to roll back auto fallback overrides after successful runs, separate selected-versus-actual model persistence, update status output, docs, tests, and the changelog.

## Best Possible Solution

Preserve durable fallback as the compatibility default unless maintainers explicitly choose a per-turn or TTL recovery policy, then implement that contract with schema/docs/tests and real fallback proof.

## Reproduction Assessment

Yes by source inspection: current main documents and tests durable auto fallback, while session usage persists the actual fallback runtime and the successful embedded-run path returns before rollback. I did not run a live fallback session in this read-only review.

## Solution Assessment

No as currently framed: immediate rollback is a plausible mode, but making it unconditional flips the existing default. The safer path is a maintainer-approved fallback persistence policy, or a chosen TTL alternative, with compatible docs and tests.
