## Summary

Keep this PR open. The runtime fix targets a real current-main sticky fallback path, but the patch changes documented model-failover semantics without updating the model failover docs, which still describe auto fallback overrides as sticky across later turns.

## What This Changes

The PR rolls back successful fallback candidate session overrides, separates selected session model persistence from actual fallback usage accounting in reply and follow-up runs, and adds focused tests plus an Unreleased changelog entry.

## Best Possible Solution

Update the model failover documentation to match the new success-rollback semantics, then land this narrow runtime/accounting fix after maintainer review and changed-gate validation.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: a successful non-selected fallback attempt can persist fallback override state, and session usage persistence writes the actual fallback provider/model into stored session identity.

## Solution Assessment

No, not yet. The runtime approach is narrow and likely the right fix, but it should not land while docs/concepts/model-failover.md still documents the inverse sticky auto-fallback behavior.
