## Summary

Keep this PR open for maintainer review. Current main still has the sticky fallback state path the branch targets: successful fallback attempts return before clearing temporary auto overrides, and session usage persistence still writes the actual fallback runtime model into the stored session identity. The patch is narrow, has a changelog entry and focused coverage, and GitHub check-runs on the PR head show no failures; I found no discrete blocking review finding.

## What This Changes

The PR rolls back automatic fallback model overrides after successful fallback attempts, separates stored session routing from actual fallback usage attribution in reply and follow-up accounting, and adds focused tests plus an Unreleased changelog entry.

## Best Possible Solution

Land this PR or an equivalent narrow fix after maintainer review and normal merge validation, so fallback candidate selection remains temporary, future turns retry the selected primary model, and usage/cost stay attributed to the actual fallback runtime.

## Reproduction Assessment

Yes. The source-level reproduction is clear on current main: a successful fallback candidate can persist an auto override before the attempt, return without success rollback, and then usage persistence writes the fallback runtime provider/model into the session identity.

## Solution Assessment

Yes. The patch addresses the bug at the two narrow state boundaries that create it: fallback candidate rollback in the runner and selected-versus-actual model persistence in session accounting, without changing fallback policy or provider behavior.
