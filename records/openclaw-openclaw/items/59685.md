## Summary

Keep open: the PR is aimed at a valid status false-negative and now includes real terminal proof, but the latest head still misses legacy provider-only Gateway memory payloads and only wires the fallback through the normal human status path.

## What This Changes

The PR adds a live Gateway memory-status fallback, splits doctor memory runtime health from embedding health, restores active memory plugin registry state, updates tests, and adds a changelog entry.

## Best Possible Solution

Land a narrowed fix that trusts live Gateway provider/runtime evidence consistently across human and JSON/all status surfaces while preserving cold standalone plugin loading and keeping embedding readiness separate from runtime health.

## Reproduction Assessment

Yes. Source inspection of current main plus related #57256 shows the failure path: local memory snapshot is null, the renderer prints unavailable, and current doctor status can expose live provider evidence without a separate runtime field.

## Solution Assessment

No, not as-is. Trusting live Gateway memory evidence is the right direction, but the PR should also treat legacy provider evidence as healthy and route the fallback before early status surface returns.
