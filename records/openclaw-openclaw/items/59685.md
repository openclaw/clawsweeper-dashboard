## Summary

Keep open: the PR addresses a real memory-status false-negative path, but it still lacks required after-fix real behavior proof and has source-level gaps that can leave false negatives in supported status surfaces.

## What This Changes

The PR adds a Gateway memory-status fallback to status output, splits doctor memory runtime versus embedding health, restores cached plugin side effects on active-registry reuse, and updates tests plus changelog.

## Best Possible Solution

Land a narrow rebased fix that trusts live Gateway memory runtime/provider evidence when local memory inspection is inconclusive, preserves memory-slot-scoped loading, covers human and JSON status surfaces, and includes real behavior proof.

## Reproduction Assessment

Yes, source-reproducible. The linked bug and follow-up plugin report describe live memory operations succeeding while diagnostics report unavailable, and current main still lacks a renderer fallback to live Gateway memory evidence when local memory inspection is inconclusive.

## Solution Assessment

No, not as-is. Trusting live Gateway memory evidence is the right direction, but the patch should cover all status surfaces and legacy provider-only Gateway payloads before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor legacy provider-only gateway status:** `src/commands/status.command-sections.ts:166`
  - body: The fallback treats a legacy `doctor.memory.status` payload as healthy only when `embedding.ok` is true. Older gateways can return a non-empty provider with a skipped embedding probe (`checked: false`, `ok: false`), and this PR calls the method without `deep`/`probe` params, so mixed-version setups can still render unavailable even though the live gateway reports a memory provider.
  - confidence: 0.82
- **[P2] Cover the all-status memory fallback:** `src/commands/status.command.ts:200`
  - body: The new gateway fallback is wired only through the normal status path and gated by `opts.deep`. Human `status --all` returns through `statusAllCommand` before this block, and JSON all uses the fast JSON scan, so commands that run local memory inspection can still report the configured plugin as unavailable when the live gateway is healthy.
  - confidence: 0.83
