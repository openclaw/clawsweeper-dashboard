## Summary

Keep open: current main still lacks the live Gateway memory-status fallback and runtime-vs-embedding split, so this PR is not obsolete; the branch is useful but still has two status-surface correctness blockers.

## What This Changes

The PR adds a Gateway memory-status fallback for status output, splits doctor memory runtime versus embedding health, reuses/restores active memory plugin registry state, updates tests, and adds a changelog entry.

## Best Possible Solution

Land a narrowed fix that trusts live Gateway memory runtime/provider evidence across the relevant human and JSON status surfaces, preserves slot-scoped plugin loading, and keeps embedding readiness separate from runtime health.

## Reproduction Assessment

Yes, source-reproducible. Current main still renders an enabled memory plugin with a missing local snapshot as unavailable and has no live Gateway memory fallback or runtime health field, matching the linked false-negative reports.

## Solution Assessment

No, not as-is. Trusting live Gateway memory evidence is the right direction, but the PR should also honor legacy provider-only payloads and wire the fallback through all status surfaces that can otherwise report the same false negative.
