## Summary

Keep this PR open: the resolver/session-store fix is still needed on current main, but the latest branch still models Claude CLI compaction through an `agentHarnessId: "claude-cli"` path that current docs and runtime code say is not the normal Claude CLI route.

## What This Changes

The PR maps `claude-cli` context-token resolution to canonical Anthropic provider config and adds resolver, session-store, and CLI compaction regression tests.

## Best Possible Solution

Land the resolver/session-store budget fix only after the compaction regression follows the normal Claude CLI `ownsNativeCompaction` no-op path, or after maintainers explicitly accept the deterministic proof boundary.

## Reproduction Assessment

Yes. Source inspection shows CLI runs persist `provider: claude-cli`, session-store asks the context resolver for that runtime provider, and current main’s resolver does not consult canonical `anthropic` config before the fixed Anthropic 1M fallback.

## Solution Assessment

No, not fully as presented. The resolver/session-store boundary is the right fix layer, but the compaction regression should match current Claude CLI ownership semantics instead of manufacturing `agentHarnessId: "claude-cli"`.
