## Summary

Keep open: this PR is the active candidate fix for https://github.com/openclaw/openclaw/issues/80933, and current main still misses the canonical Anthropic `contextTokens` lookup for `claude-cli` session persistence. The patch appears functionally correct from source review and supplied terminal proof, but the persisted session-state and runtime-provider mapping semantics need maintainer acceptance before merge.

## What This Changes

The PR makes context-token resolution consult canonical Anthropic configuration for `claude-cli` before the fixed Anthropic fallback and adds resolver, session-store, and CLI compaction regression tests.

## Best Possible Solution

Land after maintainer acceptance of the `claude-cli` to Anthropic budget semantics, or first switch the resolver to one owner-approved runtime-to-canonical provider source while keeping the focused regressions.

## Reproduction Assessment

Yes, by source inspection: current main stamps CLI runs as `claude-cli`, then resolves and persists `contextTokens` using that runtime provider without checking canonical `models.providers.anthropic`. I did not run tests because this review is read-only and must not create artifacts.

## Solution Assessment

Mostly yes: fixing the context-token resolver before session-store persistence is the right boundary for the bug. The remaining best-fix question is whether maintainers accept the narrow hard-coded alias or want the existing CLI backend `modelProvider` binding to own this mapping.
