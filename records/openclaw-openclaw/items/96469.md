## Summary

Keep open for maintainer review and possible landing: current main still lacks an unreadable-config write blocker, and this PR implements the focused guard with sufficient live-output proof and no blocking code findings. The only material caveat is that it intentionally changes unreadable-config writes to fail closed, so maintainers should accept that operator-facing behavior before merge.

## What This Changes

The PR adds a `readError` marker to config snapshots, blocks full-file config writes when the base snapshot came from an unreadable config, and adds EACCES regression coverage that preserves the live file and writes a rejected artifact.

## Best Possible Solution

Land the focused config write guard after maintainers accept the fail-closed behavior, while leaving the broader sudo/wrong-UID ownership policy in https://github.com/openclaw/openclaw/issues/78493.

## Reproduction Assessment

Yes. Current main returns a fallback snapshot after EACCES and current write blockers do not include an unreadable-base reason; the PR body also includes before/after live output for createConfigIO/writeConfigFile. I did not rerun commands in this read-only pass.

## Solution Assessment

Yes for the config-clobber half. The guard lives at the config IO writer and rejected-artifact boundary, which covers doctor/update callers without per-caller checks; sudo ownership remediation should remain separate.
