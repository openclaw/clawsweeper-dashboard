## Summary

This PR remains useful and reviewable: it addresses the config-clobber half of the linked unreadable-config incident with a focused fail-closed guard and regression coverage. The only maintainer-visible blocker is accepting the compatibility tradeoff that existing doctor/update writes after an unreadable config now reject and require operator recovery instead of continuing with best-effort state.

## What This Changes

The branch adds `readError` to config snapshots, treats an unreadable base config as an always-blocking full-file write reason, and expands EACCES config IO tests.

## Best Possible Solution

Land the scoped config write guard if maintainers accept the fail-closed operator behavior, while keeping the wrong-UID update/ownership policy tracked separately in https://github.com/openclaw/openclaw/issues/78493.

## Reproduction Assessment

Yes, source inspection and the linked production report give a high-confidence path: an existing `openclaw.json` becomes unreadable, the loader returns a fallback snapshot, and a later full-file write can serialize a skeletal config over the rich file on current main. I did not run a live sudo/systemd or LaunchAgent reproduction during this read-only review.

## Solution Assessment

Yes, with a maintainer compatibility decision: recording the read failure on the config snapshot and blocking full-file writes at the existing config IO write gate is the narrowest maintainable fix for the clobber path. The broader sudo/wrong-UID prevention remains a separate product decision in the linked issue.
