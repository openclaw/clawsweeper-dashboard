## Summary

This issue should stay open: current `main` and the latest release still define the gateway connect-frame `auth.*` fields as unbounded protocol strings, while the linked fixing PR is open and unmerged. The central work is actionable, but it should continue through https://github.com/openclaw/openclaw/pull/77538 rather than a duplicate repair lane.

## What This Changes

The issue asks OpenClaw to reject oversized gateway connect-frame `auth.*` fields at the protocol schema boundary before they reach auth-handling code.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77538, or replace it with an equivalent schema-boundary fix that adds explicit auth-field bounds, documents the protocol contract, and preserves legitimate in-spec connect requests.

## Reproduction Assessment

Yes. Source inspection shows current `ConnectParamsSchema` accepts each connect-frame auth field as plain `Type.String()` with no schema length bound, and the pre-auth handler validates that schema before auth handling.

## Solution Assessment

Yes, with one maintainer decision still open. Bounding the protocol schema is the narrowest effective layer, but the chosen 256/4096 caps and documentation need security/compatibility acceptance.
