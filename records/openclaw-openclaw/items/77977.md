## Summary

Keep open: current main and the latest release still accept unbounded connect-frame auth strings at the protocol schema, and the open linked PR is the active candidate fix. This issue should close only after that PR lands or maintainers choose a replacement fix.

## What This Changes

The issue requests schema-level size bounds for gateway connect-frame `auth.*` fields so oversized pre-auth payloads are rejected before auth handling.

## Best Possible Solution

Review and land the linked gateway schema-boundary PR after gateway/security compatibility acceptance of the chosen caps, or replace it with a narrower equivalent fix.

## Reproduction Assessment

Yes. Source inspection shows current main accepts unbounded connect auth strings at schema validation, then passes accepted bootstrap/device/shared-secret candidates into pre-auth auth handling and length-scaling secret comparison paths.

## Solution Assessment

Yes, with maintainer cap acceptance still needed. Bounding `ConnectParamsSchema` is the narrowest effective layer because it rejects oversized values before auth resolution and token comparison; the unresolved part is whether the exact caps are the desired protocol compatibility line.
