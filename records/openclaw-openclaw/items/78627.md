## Summary

Keep open for maintainer triage: current main shows the relevant thinking/provider paths, but it does not prove that a newly onboarded non-Azure NVIDIA custom provider auto-sends `medium` without an explicit default or customized model metadata. The requested provider-level disable would change thinking precedence and should be decided deliberately.

## What This Changes

The issue asks OpenClaw to avoid sending thinking parameters to an NVIDIA OpenAI-compatible custom provider/model that rejects them, preferably with a per-provider disable.

## Best Possible Solution

Keep the existing per-model `params.thinking: "off"` workaround documented and decide whether provider/model capability metadata should clamp explicit thinking defaults for unsupported custom providers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. Source inspection shows how a non-off explicit/session/global thinking level can reach the runtime and how per-model `params.thinking: "off"` disables it, but the NVIDIA hang was not live-reproduced and the reporter's effective config is missing.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Adding a provider-level disable may be useful, but the narrowest maintainable path may be documenting or improving the existing per-model override unless maintainers decide that provider capability metadata should override explicit thinking defaults.
