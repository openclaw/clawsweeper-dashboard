## Summary

This PR should remain open but is not merge-ready: the fallback can reuse a process-global resolved credential for a different configuration that shares the provider id and managed marker, the supplied evidence is unit-test-only, and another open PR overlaps the same auth policy.

## What This Changes

The PR makes the shared custom-provider API-key resolver fall back to a resolved runtime-config snapshot for managed SecretRef markers and adds three resolution and isolation tests.

## Best Possible Solution

Keep one shared managed-SecretRef resolver, bind snapshot reuse to canonical whole-source config applicability, cover the same-marker/different-config case, and demonstrate actual file-backed preparation through availability and final resolution.

## Reproduction Assessment

Yes at source level: current main rejects the managed marker despite resolved runtime snapshot state, and the proposed same-marker cross-config exposure follows directly from the new guard. This PR has not supplied a real current-branch runtime reproduction.

## Solution Assessment

No in its current form. Updating the shared resolver is the narrowest useful boundary, but snapshot reuse must follow the canonical whole-config applicability contract rather than matching only the provider id and marker class.
