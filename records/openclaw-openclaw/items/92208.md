## Summary

The updated guard still does not isolate the process-global snapshot from another config that uses the same provider id and the same `secretref-managed` marker; that is the normal managed-SecretRef shape, so the patch can return the other config’s resolved credential. The availability path does call the changed resolver, so this branch covers both availability and final resolution, but the credential-boundary defect and missing real runtime proof block merge.

## What This Changes

Adds a runtime-config snapshot fallback for managed custom-provider API keys when the snapshot source has a non-secret marker, plus resolution and cross-config tests.

## Best Possible Solution

Bind runtime-snapshot reuse to the exact source configuration or canonical SecretRef identity, cover the same-marker cross-config case, and land one implementation with redacted file-backed preparation proof.

## Reproduction Assessment

Yes at source level: the linked released report and current auth chain show the managed marker returning no usable key, and the changed resolver is used by the fast availability check. This PR does not provide real after-fix execution.

## Solution Assessment

No in its current form; snapshot fallback is the right boundary, but applicability must use exact source-config or SecretRef identity rather than the shared marker value.
