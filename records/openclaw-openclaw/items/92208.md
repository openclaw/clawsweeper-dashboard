## Summary

The reported v2026.6.5 auth regression is valid and the shared-helper approach reaches the relevant availability and resolution callers, but this branch must not merge as written because it can return the active process snapshot’s resolved credential to an unrelated config that happens to use the same provider id. The overlapping https://github.com/openclaw/openclaw/pull/92113 demonstrates the needed source-snapshot applicability contract and stronger real-runtime proof, but remains open rather than a safe supersession target.

## What This Changes

The PR makes custom-provider API-key resolution fall back to a resolved runtime config snapshot and adds a unit regression test for the `secretref-managed` marker.

## Best Possible Solution

Resolve managed custom-provider SecretRefs through one shared helper that first proves the runtime snapshot applies to the caller’s source config, use it consistently across availability and final resolution, and cover both successful file-backed resolution and same-provider cross-config isolation.

## Reproduction Assessment

Yes. The linked issue provides a concrete v2026.6.5 configuration and failure path, and source inspection shows the managed marker is rejected before resolved snapshot data is available to the custom-provider auth path; this review did not execute the live setup itself.

## Solution Assessment

No, not as written. Moving the fallback into the shared helper is appropriately narrow, but it must enforce the repository’s source-snapshot applicability contract before returning resolved secret material.
