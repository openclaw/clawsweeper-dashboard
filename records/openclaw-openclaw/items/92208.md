## Summary

The branch addresses the released custom-provider SecretRef regression and now feeds both availability and final resolution through the changed helper, but its source guard still permits credential reuse by an unrelated config that uses the same provider id and managed marker. The overlapping sibling PR is not yet a safe superseding target because it is open and its current applicability logic also needs careful review; this PR additionally remains blocked on real file-backed proof.

## What This Changes

This PR adds runtime-snapshot resolution for managed custom-provider API keys, scoped by the snapshot’s source marker, with success and cross-config isolation unit tests.

## Best Possible Solution

Use the existing runtime-snapshot applicability contract to require the caller config to match the recorded source config, cover two unrelated same-provider configs that both use the managed marker, and retain one canonical availability-and-resolution implementation with file-backed SecretRef proof.

## Reproduction Assessment

Yes at source level: the linked v2026.6.5 configuration reaches a helper that rejects the managed marker, and that helper gates both availability and final credential resolution. This review did not execute the real gateway setup.

## Solution Assessment

No in its current form: centralizing the fallback in this helper is maintainable, but snapshot applicability must use the existing source-config matching contract rather than marker presence alone.
