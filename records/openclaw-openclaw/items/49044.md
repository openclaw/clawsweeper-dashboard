## Summary

Keep this PR open. The requested per-account SOUL capability is still not implemented on current main, and the proposed patch is useful directionally but incomplete: it persists a new account config key without adding the validated schema/runtime/bootstrap contract that would make that key save safely and affect agent prompts.

## What This Changes

The PR adds a `--soul` option to `channels add`, stores `soulFile` under channel account config from CLI/onboarding, adjusts channel setup registry snapshot handling, and adds related tests.

## Best Possible Solution

Rework the feature as one end-to-end account-scoped bootstrap contract: add a documented `soulFile` account schema where supported, validate workspace-relative markdown filenames/paths before persistence, preserve default `SOUL.md` fallback, pass channel/account context into bootstrap resolution and prompt assembly, keep prompt cache identity deterministic for scoped files, update docs/changelog, and cover default plus named-account behavior with focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on the PR branch: invoke the new `channels add --soul` path for an allowed channel such as Slack, then the patch writes `soulFile` even though current strict account schemas do not accept it and current runtime bootstrap still reads fixed `SOUL.md`. I did not run tests or the branch because this review is read-only.

## Solution Assessment

No. Persisting `soulFile` from CLI/onboarding is not the best fix by itself because it creates a config surface before defining validation, guarded path resolution, runtime loading, prompt behavior, and cache semantics. The safer alternative is to land the schema, guarded reader, prompt integration, docs, and tests together.
