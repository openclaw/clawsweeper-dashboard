## Summary

Keep open: current main still lacks per-account SOUL support, but this PR is not merge-ready because it writes a new account config key without the matching schema/runtime bootstrap contract and still lacks real behavior proof.

## What This Changes

The PR adds `channels add --soul`, onboarding prompts, account-scoped `soulFile` config writes, and setup-plugin snapshot tests for per-account SOUL files.

## Best Possible Solution

Implement a complete account-scoped SOUL contract across config schemas/types, CLI and onboarding, channel/account-aware bootstrap resolution and cache keys, safe fallback behavior, focused tests, and redacted live channel proof.

## Reproduction Assessment

Yes for the blocking defects by source inspection: the PR writes `soulFile`, current main validates persisted config against strict account schemas, and bootstrap still reads only the canonical `SOUL.md`. I did not run tests because this is a read-only review.

## Solution Assessment

No. The PR adds write/onboarding surfaces before the schema and runtime reader exist; the safer path is to land the full account-scoped SOUL contract together.
