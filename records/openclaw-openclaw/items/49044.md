## Summary

Keep open: current main still only supports the fixed workspace `SOUL.md`, and this PR is not merge-ready because it persists a new account `soulFile` field without schema, runtime bootstrap, prompt/cache, path-validation, or current onboarding-flow support.

## What This Changes

The PR adds `channels add --soul`, onboarding prompts, account-level `soulFile` writes, setup snapshot handling, and tests for channel setup/SOUL CLI flow.

## Best Possible Solution

Define per-account SOUL as an end-to-end account bootstrap contract with strict schema support, safe workspace-relative markdown path validation, default `SOUL.md` fallback, channel/account-aware prompt loading and cache keys, docs/changelog, and focused default plus named-account tests.

## Reproduction Assessment

Yes. Static source inspection is enough: the PR writes `channels.<channel>.accounts.<id>.soulFile`, while current main validates strict account schemas that do not declare it and runtime bootstrap still loads only fixed workspace `SOUL.md`.

## Solution Assessment

No. Persisting the field from CLI/onboarding is only a partial solution; the maintainable fix needs the schema, guarded path resolution, runtime prompt integration, account-aware cache semantics, docs, and tests to land together.
