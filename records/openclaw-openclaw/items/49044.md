## Summary

Keep open. The feature is still useful and current main does not implement per-account SOUL files, but the proposed patch is incomplete for merge because it persists a new account config key before adding the schema, guarded runtime reader, prompt/cache semantics, and current onboarding-flow integration needed to make it valid and effective.

## What This Changes

The PR adds a `--soul` option and onboarding prompts for account-scoped SOUL file paths, plus channel setup snapshot compatibility tests.

## Best Possible Solution

Rework this as an end-to-end account-scoped bootstrap contract: add documented `soulFile` account schema support where intended, validate workspace-relative Markdown filenames or paths, preserve default `SOUL.md` fallback, pass channel/account context into bootstrap resolution and prompt assembly, keep cache identity account-aware, update current onboarding/CLI docs, and cover default plus named-account behavior with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main has no `--soul` flag, no `soulFile` config schema or runtime reader, and fixed `SOUL.md` bootstrap loading; no runtime test was run because this was a read-only review.

## Solution Assessment

No. The proposed patch is not the best way to solve the issue yet because it writes a new config value before the schema and runtime behavior make that value valid, guarded, and observable. The safer solution is to land schema, guarded resolution, prompt/cache behavior, docs, and tests together.

  - body: The new branch stores `soulFile` under `channels.<channel>.accounts.<id>`, but the allowed channel account schemas are strict and do not declare this key. Users choosing a custom SOUL file will hit config validation failure until the public config contract includes the field.

  - confidence: 0.87

  - body: `opts.soul.trim()` is accepted as an arbitrary path-like string even though this value is intended to drive prompt file injection. Reject absolute paths, traversal, separators where unsupported, and non-workspace targets before persisting it.
  - confidence: 0.78
- **[P2] Move onboarding changes into the current setup flow:** `src/commands/onboard-channels.ts:857`
  - body: Current main's `src/commands/onboard-channels.ts` is only a re-export barrel; the real setup flow now lives in `src/flows/channel-setup.ts`. The SOUL prompt step needs to be rebased into that flow or onboarding will not run it on current main.
