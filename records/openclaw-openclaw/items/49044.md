## Summary

Keep this PR open. Current main still lacks per-account SOUL behavior, and the branch is incomplete because the new account field is not part of the validated config/runtime prompt contract and the onboarding edit targets an outdated setup file.

## What This Changes

The branch adds `channels add --soul`, account-level `soulFile` persistence in CLI/onboarding, channel setup snapshot typing/tests, and related command tests.

## Best Possible Solution

Define per-account SOUL as an end-to-end account bootstrap contract: schema, guarded workspace-relative path resolution, default `SOUL.md` fallback, channel/account-aware bootstrap and prompt/cache behavior, docs/changelog, and focused default plus named-account tests.

## Reproduction Assessment

Yes. A static source reproduction is sufficient: follow the new `channels add --soul` path for an allowed channel and the patch persists `soulFile`, while current strict schemas do not accept that key and current bootstrap still reads only fixed workspace `SOUL.md`.

## Solution Assessment

No. Persisting `soulFile` from CLI/onboarding is not sufficient by itself; the maintainable solution needs schema, guarded runtime loading, prompt/cache semantics, docs, and tests in the same contract.
