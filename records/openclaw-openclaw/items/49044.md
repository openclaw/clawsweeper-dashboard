## Summary

Keep this PR open. Current main still does not provide per-account SOUL behavior, and the proposed branch remains incomplete because the new account field is not part of the validated config/runtime prompt contract and the onboarding code no longer matches the current setup flow.

## What This Changes

The PR adds `channels add --soul`, account-level `soulFile` writes from CLI/onboarding, channel setup snapshot adjustments, and related command/setup tests.

## Best Possible Solution

Define per-account SOUL as an end-to-end account bootstrap contract: schema, safe workspace-relative path validation, default `SOUL.md` fallback, channel/account-aware bootstrap and prompt/cache behavior, docs/changelog, and focused tests for default plus named-account flows.

## Reproduction Assessment

Yes. A static reproduction is sufficient: follow the new `channels add --soul` path for an allowed channel and the patch persists `soulFile`, but current strict schemas do not accept that key and current bootstrap still reads only fixed workspace `SOUL.md`.

## Solution Assessment

No. Persisting `soulFile` from CLI/onboarding is not the narrowest maintainable solution by itself; the safer path is to land schema, guarded path resolution, runtime prompt integration, cache semantics, docs, and tests together.
