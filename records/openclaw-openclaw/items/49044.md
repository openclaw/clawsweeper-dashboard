## Summary

Keep open: current main still lacks per-account SOUL support, and this PR is not merge-ready because it writes a new account `soulFile` field without the matching config schema/runtime reader and still lacks real behavior proof.

## What This Changes

The PR adds `--soul` handling and onboarding prompts that store `soulFile` under channel accounts, plus setup snapshot compatibility tests.

## Best Possible Solution

Rework this as a complete account-scoped SOUL contract on current main: schema/types/docs, CLI and onboarding writes, channel/account-aware bootstrap resolution and caching, fallback/path guards, focused tests, and redacted live channel proof.

## Reproduction Assessment

Yes, by source inspection: the PR writes `soulFile`, current main validates config writes against strict account schemas that do not include that key, and bootstrap loading still uses canonical `SOUL.md` only. I did not execute mutating commands in the read-only checkout.

## Solution Assessment

No. The current PR is not the best merge shape because it adds CLI/onboarding writes without the matching schema and runtime/bootstrap reader; the safer path is to restore the full contract from https://github.com/openclaw/openclaw/pull/42861 on top of current main.
