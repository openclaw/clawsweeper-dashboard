## Summary

Keep open: current main still lacks per-account SOUL support, but this PR is not merge-ready because it only persists the new field in setup paths and leaves schema validation, runtime prompt loading, path safety, cache semantics, and the current onboarding flow unresolved.

## What This Changes

The branch adds `channels add --soul`, account-level `soulFile` writes, onboarding SOUL prompts, channel setup snapshot adjustments, and focused tests.

## Best Possible Solution

Define per-account SOUL as an end-to-end account bootstrap contract with strict schema support, safe workspace-relative markdown path validation, default `SOUL.md` fallback, channel/account-aware loading and cache keys, docs, and focused runtime plus setup tests.

## Reproduction Assessment

Yes. Static source inspection is enough: the PR writes `channels.<channel>.accounts.<id>.soulFile`, while current main validates strict account schemas that do not declare it and runtime bootstrap still loads only fixed workspace `SOUL.md`.

## Solution Assessment

No. Persisting a new field from CLI/onboarding is only a partial solution; the maintainable fix needs schema, guarded path resolution, runtime prompt integration, account-aware cache semantics, docs, and tests to land together.
