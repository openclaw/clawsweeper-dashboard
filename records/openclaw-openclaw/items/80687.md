## Summary

Keep open: the PR addresses a real source-visible gap for music/video generation default timeouts, but the current patch also changes music reference-image fetch timing by reusing the configured provider timeout before `loadReferenceImages`. The existing maintainer-label also means this should stay in explicit maintainer handling rather than cleanup closure.

## What This Changes

The PR wires configured music/video generation `timeoutMs` defaults into tool and runtime provider requests, adds focused tests, updates tool docs, and adds a changelog entry.

## Best Possible Solution

Carry the configured default only into the provider generation timeout, keep reference-input loading on its explicit per-call or existing fetch timeout behavior, and cover both paths with focused tests.

## Reproduction Assessment

Yes, by source inspection: current main accepts `timeoutMs` in model config but music/video provider requests only receive `params.timeoutMs` when the caller supplies it. I did not run a live reproduction because this review is read-only.

## Solution Assessment

No, not as currently patched. The runtime default propagation is the right direction, but the music tool needs separate provider and reference-fetch timeout variables so model defaults do not change input loading behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep model defaults out of reference-image fetches:** `src/agents/tools/music-generate-tool.ts:642`
  - body: This fallback is assigned before the shared `timeoutMs` is passed into `loadReferenceImages`, so a call that omits `timeoutMs` now uses `musicGenerationModel.timeoutMs` for reference-image fetch aborts instead of the existing 30s fetch default. Split the provider timeout from the reference-fetch timeout so only the generation request inherits the model default.
  - confidence: 0.94
