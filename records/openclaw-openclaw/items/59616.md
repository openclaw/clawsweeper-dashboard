## Summary

Keep open. Current main still watches only the active config file path, while include resolution and recursive include-path scanning exist but are not wired into the gateway reloader.

## What This Changes

The issue asks for gateway config hot reload to watch files referenced by `$include`, not only the root config file.

## Best Possible Solution

Land a narrow gateway reloader change that synchronizes Chokidar’s watched set with the valid `$include` graph after startup and each accepted snapshot, while preserving current include security checks and reload planning behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear: start the reloader with a root config containing a `$include`, edit only the included file, and current main has no watcher on that path to schedule a reload.

## Solution Assessment

Yes. The issue’s proposed direction is the narrow maintainable fix: reuse the existing include-path collector and diff paths into `watcher.add()`/`watcher.unwatch()` without changing include resolution semantics.

Fix the config hot-reload bug where edits to `$include`-referenced files do not trigger reloads. In `src/gateway/config-reload.ts`, keep the Chokidar watched set synchronized with the currently accepted include graph: collect include paths from the current valid root/source config on startup and after each successful accepted snapshot, add newly referenced files with `watcher.add()`, and unwatch files removed from the graph. Preserve existing debounce, in-process write dedupe, missing/invalid config handling, last-known-good promotion/recovery, reload planning, and include resolver guardrails; do not broaden allowed include roots or change `$include` merge semantics. Use PR #59632 only as context because it is closed unmerged. Add focused tests in `src/gateway/config-reload.test.ts` for included-file change scheduling, removed include unwatching, and graceful include-scan/watch failures.
