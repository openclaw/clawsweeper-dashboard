## Summary

Keep open: current main still lacks passive context-window usage in the system-prompt Runtime line, and the closed related items consolidate this as the canonical tracker rather than a shipped fix.

## What This Changes

Request adding a passive context-window usage percentage or count to the per-turn system-prompt Runtime metadata so agents do not need to call `session_status` each turn.

## Best Possible Solution

Add an optional `context=NN%` Runtime field backed by fresh session total/context-window metadata, omit it when unavailable or stale, and cover formatting plus runner propagation with focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main shows the requested passive `context=` Runtime field and carrier types are absent while existing usage is available through `/status` and `/context`.

## Solution Assessment

Yes; the narrow maintainable solution is prompt/runtime plumbing from existing fresh session usage metadata. `/status`, `/context`, response footers, and workspace-sync plugins are useful adjacent surfaces but do not give the agent passive per-turn Runtime context awareness.

## Repair Work Prompt

Implement context-window usage in the system-prompt Runtime line for embedded runs. Derive an optional context percent only from fresh session data: use the active session entry via `resolveFreshSessionTotalTokens()` plus a positive context-window value, guard `NaN`/Infinity/negative values, round for display, and clamp to a reasonable display range. Thread the optional field through `agent-runner-execution`/`runEmbeddedPiAgent` params, embedded run params/types, `buildSystemPromptParams`, `RuntimeInfoInput`, and `buildRuntimeLine()`. Render `context=NN%` in the Runtime line when available and omit it when data is missing, stale, or invalid. Keep the field in the existing volatile Runtime section below the prompt cache boundary; do not add a new status command, config option, or broad context-accounting rewrite. Update the system-prompt docs if the Runtime field list changes. Add focused tests for formatting, stale-data omission, reply-runner propagation, and embedded prompt assembly. Add a changelog entry under the active Changes section because this is user-visible prompt behavior.
