## What This Changes

The issue reports that resolving an exec approval from an external channel plugin loses the original channel target and sends the post-approval agent reply to webchat instead of the originating channel.

## Best Possible Solution

Replace the static native approval inline gate with a capability-backed check for the initiating channel, and add a fake external channel plugin regression test that preserves the original turn-source target.

## Reproduction Assessment

Yes, at source level: current main gates inline exec approval resumption through a bundled-channel constant while the report uses an external channel plugin with native approval capability. I did not live-run the Lansenger plugin flow.

## Solution Assessment

No, current main is not a complete solution for this issue because the merged native-chat fix covers bundled channels only. The narrow fix is to base the inline decision on the registered channel approval capability rather than adding Lansenger-specific policy.
