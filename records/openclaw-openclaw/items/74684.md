## Summary

Keep open. Current main still omits first-class ACP PID/process metadata from the ACP spawn result, tool return boundary, runtime/status contracts, persisted ACP metadata, and documented sessions_spawn response; open PR #74756 is unmerged and references this issue with closing syntax, so the issue should stay open until that PR or an equivalent design lands.

## What This Changes

Request exposing PID/process metadata for ACP sessions spawned by sessions_spawn so operators can correlate child session keys with local adapter or child-agent processes.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that records availability, adapter/wrapper/child process identity when known, persistence/status exposure, and explicit close/cancel behavior without making manual raw-PID killing the implicit API.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the ACP spawn result, tool boundary, runtime/status contracts, persisted metadata, docs, and tests do not expose typed PID/process correlation metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but maintainers should decide whether the durable contract should be a single optional pid or a typed process-metadata object with lifecycle and availability semantics.
