## Summary

Keep open. Current main still lacks first-class ACP process/PID correlation in the `sessions_spawn` response, ACP runtime/status contracts, persisted ACP metadata, `session_status` output, and operator docs; the issue is also paired with open PR #74756 using closing syntax, so it should remain open until that PR or an equivalent design lands.

## What This Changes

Expose process/PID metadata for ACP sessions spawned through `sessions_spawn` so operators can correlate and manage running child processes.

## Best Possible Solution

Land #74756 or a replacement additive contract that persists structured ACP process metadata and exposes it through spawn/status surfaces without making manual raw-PID matching the primary control path.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the spawn result, tool boundary, runtime/status types, persisted metadata, `session_status` tool, and ACP docs all omit first-class process correlation.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the maintainable shape is likely structured process metadata with explicit unavailable semantics rather than only a single optional `pid`.
