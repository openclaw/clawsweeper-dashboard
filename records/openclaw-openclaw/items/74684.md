## Summary

Keep open. Current main still lacks first-class ACP process/PID correlation metadata in the ACP spawn result, runtime/status contracts, persisted session metadata, and ACPX wrapper path. The issue is also paired with open, unmerged PR #74756, so it should remain open until that PR or an equivalent design lands.

## What This Changes

Expose process metadata or PIDs for ACP sessions spawned through `sessions_spawn` so operators can correlate, inspect, and safely stop running child process trees.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that covers unavailable metadata, adapter/wrapper/harness identifiers, persistence, status exposure, and explicit cancellation/close behavior.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the accepted ACP spawn result, tool return boundary, runtime/status types, persisted ACP session metadata, and ACPX wrapper all omit first-class PID/process correlation metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the narrowest maintainable fix may be a typed process metadata object rather than only an optional `pid` field, depending on ACPX and child harness lifecycle semantics.
