## Summary

Keep open. Current main still lacks a first-class, typed PID/process correlation contract on `sessions_spawn`, ACP runtime/status types, and persisted ACP session metadata, and open PR #74756 already references this issue with closing syntax.

## What This Changes

Expose local ACP adapter or child-agent process metadata for sessions spawned via `sessions_spawn(runtime="acp")` so operators can correlate running child sessions with OS processes.

## Best Possible Solution

Land #74756 or a replacement that adds an additive ACP process-metadata contract from ACPX through spawn, status, and persisted metadata with clear unavailable semantics and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessions_spawn` returns only logical IDs and delivery metadata while the ACP runtime/status and persisted metadata types lack typed process fields; no live ACP run was needed to prove the missing contract.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the maintainable fix should distinguish adapter and child-agent PID(s), define unavailable cases, and decide whether termination belongs in the same API surface.
