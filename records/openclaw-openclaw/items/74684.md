## Summary

Keep open. Current main still omits first-class ACP PID/process correlation from the ACP spawn result, runtime/status contracts, persisted session metadata, and ACP status output, and open PR #74756 already references this issue with closing syntax.

## What This Changes

Expose first-class ACP process/PID correlation for `sessions_spawn` ACP child sessions so operators can inspect or terminate running ACP adapters safely.

## Best Possible Solution

Review and land #74756 or a replacement that adds an additive ACP process-metadata contract through spawn, status, and persisted metadata without relying on manual process-table matching.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the accepted spawn result, tool return path, runtime/status contracts, persisted metadata, and ACP status output all omit first-class PID/process correlation.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the maintainable fix should distinguish adapter and child-agent process metadata and define unavailable cases rather than only adding a raw optional PID.
