## Summary

Keep open. Current main still omits first-class ACP PID/process correlation from the spawn result, runtime/status contracts, persisted ACP metadata, and ACP troubleshooting docs; open PR #74756 already references this issue with closing syntax, so the issue should remain open until that PR or an equivalent design lands.

## What This Changes

Expose process metadata or PIDs for ACP sessions spawned by `sessions_spawn` so operators can correlate running child processes with OpenClaw sessions.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that persists available/unavailable process details and exposes them through spawn and status.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the accepted ACP spawn result, runtime/status types, persisted metadata, ACPX wrapper, and docs all omit first-class PID/process correlation.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the maintainable shape may be a structured process metadata object rather than a single optional `pid`, depending on ACPX wrapper and harness lifecycle semantics.
