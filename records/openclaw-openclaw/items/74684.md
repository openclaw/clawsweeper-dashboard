## Summary

Keep open. Current main and the latest release still do not expose typed ACP process metadata through `sessions_spawn`, session metadata, or ACP status surfaces; ACPX now tracks process PIDs internally for cleanup, but that is not a first-class observability/control API.

## What This Changes

Expose ACP child process metadata or PIDs through `sessions_spawn`, status, or session records so operators can correlate running ACP sessions with local processes.

## Best Possible Solution

Define a typed ACP process-metadata contract with roles, availability, and staleness semantics, source it from ACPX lease/session-record data, then expose it through spawn/status/docs with explicit visibility policy.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main and `v2026.5.12` omit PID/process metadata from the ACP spawn, runtime/status, and session metadata contracts while ACPX only keeps PID data internally for cleanup.

## Solution Assessment

Unclear. The requested direction is useful, but a raw optional `pid` field is not the best long-term fix; the maintainable path is a typed process-metadata contract with visibility and stale-PID rules.
