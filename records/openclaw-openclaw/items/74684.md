## Summary

Keep open. Current main still lacks a first-class typed ACP process-metadata surface in `sessions_spawn`, persisted ACP session metadata, and `session_status`; ACPX has internal PID data for cleanup/status strings, but the linked PID PR closed unmerged and no current OpenClaw API contract exposes adapter/child process IDs safely.

## What This Changes

The issue requests first-class PID/process metadata for ACP sessions spawned through `sessions_spawn(runtime="acp")`.

## Best Possible Solution

Define typed ACP process metadata with process role, availability, and staleness semantics, then propagate it through ACPX, ACP runtime/status types, persisted session metadata, `sessions_spawn`, docs, and focused coverage.

## Reproduction Assessment

Yes. Source inspection is enough: the ACP spawn result, status types, and session metadata on current main and the latest release do not expose typed PID/process metadata, even though ACPX tracks PID data internally.

## Solution Assessment

Unclear. The requested direction is useful, but a raw optional `pid` alone is not the best solution; the maintainable path is a typed process-metadata contract with roles, visibility rules, stale/unavailable semantics, and ACPX-backed data.
