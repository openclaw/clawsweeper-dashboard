## Summary

Keep open. Current main still does not expose first-class ACP process/PID metadata through the `sessions_spawn` ACP result, ACP runtime/status contracts, persisted session metadata, or `/acp status`; the issue is also paired with open unmerged PR #74756, so it should remain open until that PR or an equivalent replacement lands.

## What This Changes

The issue requests process/PID metadata for ACP sessions spawned through `sessions_spawn` so operators can correlate a running child session with local adapter and child-agent processes.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that represents available and unavailable adapter/harness process identifiers, persists that metadata, exposes it through `sessions_spawn` and ACP status surfaces, and leaves kill/cancel behavior on explicit control APIs.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the ACP spawn result, runtime/status contracts, persisted session metadata, ACPX wrapper, status formatter, and docs all omit first-class process/PID correlation metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but maintainers should decide whether the final contract is a single optional PID or a typed metadata object that can represent multiple ACP process layers safely.
