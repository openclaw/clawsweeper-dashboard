## Summary

Keep open. Current main still lacks first-class ACP process/PID correlation in the `sessions_spawn` ACP result, ACP runtime/status contracts, persisted ACP session metadata, and operator-facing status surface; open PR #74756 already references this issue with closing syntax, so the issue should stay open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to expose process correlation metadata, such as ACP adapter/child PIDs or a first-class close/kill path, for ACP sessions spawned through `sessions_spawn`.

## Best Possible Solution

Land #74756 or a replacement that adds an additive ACP process-metadata contract through ACPX runtime integration, spawn results, status, and persisted metadata with clear unavailable semantics.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the accepted spawn result, tool boundary, runtime/status contracts, and persisted metadata all omit first-class PID/process correlation.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the maintainable fix should distinguish adapter and child-agent process metadata and define unavailable cases rather than only adding a raw optional PID.
