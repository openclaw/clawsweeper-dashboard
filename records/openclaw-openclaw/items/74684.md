## Summary

Keep open. Current main still does not expose first-class ACP PID/process metadata through the ACP spawn result, runtime/status contracts, persisted ACP session metadata, or ACPX wrapper path, and the provided timeline shows open unmerged PR #74756 already targets the same issue with closing syntax.

## What This Changes

The issue requests OS PID/process correlation metadata or a first-class control path for ACP child processes spawned through `sessions_spawn(runtime="acp")`.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that covers unavailable metadata, adapter/wrapper/harness identifiers, persistence, status exposure, and explicit cancellation/close behavior without making manual raw-PID killing the implicit API.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the accepted ACP spawn result, tool return boundary, runtime/status types, ACP session metadata, and ACPX wrapper all omit first-class PID/process correlation metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the narrowest maintainable solution may be a typed process metadata object rather than a single optional `pid` field, depending on ACPX and child harness lifecycle semantics.
