## Summary

Keep open. Current `main` still lacks first-class ACP process correlation in `sessions_spawn`, `session_status`, and persisted session metadata; ACPX now tracks PIDs internally for cleanup, but that is not exposed as a typed OpenClaw contract, and the linked PID PR closed unmerged.

## What This Changes

The issue requests first-class process metadata or control for ACP sessions spawned via `sessions_spawn` so operators can correlate running child sessions to OS processes.

## Best Possible Solution

Define a typed ACP process-metadata contract, preferably with adapter and child-agent roles plus unavailable/stale semantics, then propagate it through ACPX, OpenClaw runtime/status types, `sessions_spawn`, persisted ACP metadata, docs, and focused coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current `main` and the latest release return only logical ACP session identifiers and do not persist or expose typed PID/process metadata through spawn, status, or session records.

## Solution Assessment

Unclear. The requested direction is useful, but a raw optional `pid` alone is not the best fix; the safer solution is a typed process metadata contract with role, visibility, staleness, and unavailable semantics.
