## Summary

Keep open. Current main still lacks first-class ACP PID/process correlation, and open PR #74756 already references this issue with closing syntax, so the issue should remain open until that PR or an equivalent replacement lands.

## What This Changes

Expose process metadata for ACP sessions spawned through `sessions_spawn` so operators can correlate live child sessions with local adapter or harness processes.

## Best Possible Solution

Land #74756 or a replacement that threads additive ACP process metadata through ACPX, runtime handle/status, persisted session metadata, and the spawn result while documenting unavailable cases.

## Reproduction Assessment

Yes. Static inspection of current main is enough to reproduce the reported gap because the accepted spawn result, runtime/status types, stored ACP metadata, and docs only expose logical session identifiers or arbitrary detail bags.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the maintainable fix should distinguish adapter and harness process metadata instead of relying only on one optional raw `pid`.
