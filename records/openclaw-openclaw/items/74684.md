## Summary

Keep open. Current main still lacks first-class ACP PID/process correlation in the spawn result, runtime/status contracts, and persisted ACP session metadata, and open PR #74756 already references this issue with closing syntax.

## What This Changes

Expose ACP process metadata through `sessions_spawn`, status, and session metadata so operators can correlate running ACP child sessions with OS processes.

## Best Possible Solution

Review and land #74756 or a replacement that adds an additive ACP process-metadata contract through ACPX, spawn, status, and persisted metadata with clear unavailable semantics.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the spawn result, tool boundary, runtime/status contracts, persisted metadata, and troubleshooting docs all omit first-class PID/process correlation.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the maintainable fix should distinguish adapter and child-agent process metadata and define unavailable cases rather than only adding a raw optional PID.
