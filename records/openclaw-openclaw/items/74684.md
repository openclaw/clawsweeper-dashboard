## Summary

Keep open. Current main still lacks first-class typed PID/process metadata in the ACP spawn result, runtime/status contracts, persisted ACP session metadata, and bundled ACPX wrapper path. The issue is also paired with open, unmerged PR #74756 using closing syntax, so it should stay open until that PR or an equivalent design lands.

## What This Changes

Request to expose ACP child process PID or structured process-correlation metadata from `sessions_spawn`, session records, and status surfaces for running ACP sessions.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that covers unavailable metadata, adapter/wrapper/harness identifiers, persistence, status exposure, and explicit cancellation/close behavior without making manual raw-PID killing the implicit API.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the ACP spawn result, tool return boundary, runtime/status types, persisted ACP session metadata, and sessions list/status surfaces do not expose first-class PID/process correlation metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the narrowest maintainable solution may be a typed process metadata object with lifecycle semantics rather than only a single optional `pid` field.
