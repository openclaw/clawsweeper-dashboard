## Summary

Keep open. Current main still lacks first-class ACP PID/process metadata in the `sessions_spawn` ACP result, runtime/status contracts, and persisted ACP session metadata, and the issue is already paired with open unmerged PR #74756 for the same behavior.

## What This Changes

Expose process/PID correlation metadata for ACP sessions spawned via `sessions_spawn(runtime="acp")` so operators can inspect or terminate the right running child process.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that can represent unavailable metadata, adapter/wrapper/harness process identifiers, persistence, status exposure, and explicit cancellation or close behavior without making raw PID killing the implicit API.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the accepted ACP spawn result, tool return boundary, runtime/status types, persisted metadata, tests, and docs all omit first-class PID/process correlation metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but maintainers should decide whether the durable contract is a single optional PID or a typed process-metadata object with explicit availability and lifecycle semantics.
