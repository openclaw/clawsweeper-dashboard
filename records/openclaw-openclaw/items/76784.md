## Summary

Keep open: the PR fixed the earlier missing changelog and broadened cache invalidation, but the lifecycle path can still bump the cache version before its async session-store write finishes, letting an immediate list cache stale lifecycle rows for the TTL.

## What This Changes

The PR adds bounded and in-flight-coalesced caching for `sessions.list`, keeps default async session rows lightweight, updates gateway tests, adds a changelog entry, and relaxes two script signal typings.

## Best Possible Solution

Keep the lightweight cached list path, but advance cache freshness only after the backing session row mutation has committed, ideally through a shared writer-side version signal covered by lifecycle regression tests.

## Reproduction Assessment

Yes. From source inspection, cache a list, trigger a chat lifecycle event, and issue another identical `sessions.list` after the PR bumps the version but before `persistGatewaySessionLifecycleEvent` finishes; that request can cache the old row under the new version.

## Solution Assessment

No. The cache/version approach is maintainable, but the lifecycle bump must be ordered after the async store write or tied to the session-store writer so the cache generation cannot certify stale data.
