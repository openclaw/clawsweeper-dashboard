## Summary

Keep open: the PR still has a correctness blocker because cached `activeMinutes` results can be reused for 60 seconds even though the active cutoff moves with `Date.now()`. The existing ClawSweeper comment identified the same issue after the latest force-push, and current main has no equivalent whole-result cache.

## What This Changes

The PR adds bounded caching and in-flight coalescing for Gateway `sessions.list`, keeps default async rows lighter, extends gateway/session tests and changelog, and relaxes two script signal shim types.

## Best Possible Solution

Keep the cache for stable list calls, but bypass it or make it time-aware for `activeMinutes` requests and add focused regression coverage for rows aging past the active window.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cache an `activeMinutes` list while a row is still inside the active window, wait until its `updatedAt` falls below the moving cutoff, then repeat the identical request before the cache TTL expires.

## Solution Assessment

No. The mutation-version cache is maintainable for stable list calls, but `activeMinutes` depends on wall-clock time and needs a bypass, shorter time-bound TTL, or time-bucketed key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Bypass the list cache for activeMinutes queries:** `src/gateway/server-methods/sessions.ts:156`
  - body: `activeMinutes` results are time-relative because `listSessionsFromStoreAsync` computes the cutoff from `Date.now()`. Caching an identical request for the full 60 second TTL lets a row that was active when cached keep appearing after it ages out, until the cache expires. Bypass the cache for `activeMinutes` or make that cache key/TTL time-aware.
  - confidence: 0.9
