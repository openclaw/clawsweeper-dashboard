## Summary

Keep open: the latest PR head still has a source-proven stale-cache race in lifecycle-driven session rows, so it needs a narrow repair before merge.

## What This Changes

The PR adds a bounded `sessions.list` result cache with in-flight coalescing, keeps default list rows lighter, updates session tests/changelog, and relaxes two script signal shim types.

## Best Possible Solution

Keep the lightweight list cache, but anchor cache freshness to completed session-store mutations or a reliable store-writer version signal, with regression coverage for lifecycle persistence ordering.

## Reproduction Assessment

Yes, from source inspection: cache an identical `sessions.list` result, receive a lifecycle event, call `sessions.list` before `persistGatewaySessionLifecycleEvent` finishes, then call it again after the writer resolves. The second call can reuse the pre-write snapshot under the already-advanced version until TTL expiry.

## Solution Assessment

No. Whole-result caching and in-flight coalescing are reasonable, but this implementation should invalidate after the session-store write is visible, not immediately after scheduling an unawaited persistence task.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bump the list cache after lifecycle writes:** `src/gateway/server-chat.ts:358`
  - body: The new bump runs immediately after firing `persistGatewaySessionLifecycleEvent` as an unawaited promise. If a `sessions.list` request arrives before `updateSessionStoreEntry` finishes, it can cache the old lifecycle/status row under the new version; after the write completes there is no second bump, so identical calls can serve stale rows until the 60s TTL. Move the bump/invalidation to persistence completion or a store-writer seam, or avoid caching while lifecycle writes are in flight.
  - confidence: 0.91
