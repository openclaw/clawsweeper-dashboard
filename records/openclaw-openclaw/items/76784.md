## Summary

This PR should stay open because the latest head still has a merge-blocking cache correctness issue for time-relative `sessions.list` queries. The existing ClawSweeper comment remains valid after the latest force-push.

## What This Changes

The PR adds bounded caching and in-flight coalescing for Gateway `sessions.list`, keeps default async list rows lighter, extends session tests, updates the changelog, and relaxes two script signal shim types.

## Best Possible Solution

Keep the cache for stable list calls, but bypass it for `activeMinutes` requests or make those cache entries time-aware, with regression coverage for a row aging past the active window.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cache an `activeMinutes` list while a row is inside the window, wait until its `updatedAt` falls below the moving cutoff, then repeat the identical request before the 60 second cache TTL expires.

## Solution Assessment

No. The general mutation-version cache is a maintainable optimization for stable list calls, but time-relative `activeMinutes` filtering needs a bypass, time-bucketed key, or bounded TTL tied to the active cutoff.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Bypass the cache for activeMinutes lists:** `src/gateway/server-methods/sessions.ts:156`
  - body: `activeMinutes` results depend on `Date.now()`, but this cache key treats the value as a stable request parameter and the wrapper reuses the result for up to 60 seconds. A session that was active when first cached can keep appearing after it ages past the cutoff until the cache expires, even with no store mutation. Please bypass this cache for `activeMinutes` requests or make that key/TTL time-aware.
  - confidence: 0.9
