## Summary

Keep this PR open: the sessions-list caching direction is useful, but the latest diff still invalidates before async lifecycle persistence is durable, so identical `sessions.list` calls can serve stale lifecycle/status rows until the cache TTL.

## What This Changes

The PR adds a bounded/coalesced `sessions.list` result cache, keeps default rows lighter by skipping transcript usage fallback, updates gateway/session tests and changelog, and relaxes two script signal shim types.

## Best Possible Solution

Keep the lightweight list-row cache, but tie freshness to a mutation signal that changes after store writes are visible, or include a reliable completed-writer/store-state key.

## Reproduction Assessment

Yes. From source inspection, cache an identical `sessions.list`, trigger a lifecycle start/end event, call `sessions.list` before the unawaited persistence completes, then call it again after the writer resolves; the second call can reuse pre-write rows under the already-bumped version until TTL expiry.

## Solution Assessment

No. Whole-result caching is a reasonable optimization, but this implementation should invalidate after the session-store write is visible, not immediately after scheduling an async persistence task.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Invalidate after lifecycle persistence completes:** `src/gateway/server-chat.ts:358`
  - body: The new version bump runs immediately after firing `persistGatewaySessionLifecycleEvent` as an unawaited promise. If `sessions.list` arrives before `updateSessionStoreEntry` finishes, it can cache the old store snapshot under the new version; after the write completes there is no second bump, so identical calls can serve stale lifecycle/status rows until the 60s TTL. Move the bump/invalidation to persistence completion or the store-writer seam, or avoid caching while lifecycle writes are in flight.
  - confidence: 0.91
