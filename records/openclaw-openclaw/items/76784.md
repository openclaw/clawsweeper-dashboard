## Summary

Keep open: the performance direction is useful, but the PR introduces stale `sessions.list` rows because the new 60s result cache is invalidated only by the local sessions handler while chat, agent, and session-event paths can update/broadcast session state independently. It also needs the required changelog entry for a user-facing perf change.

## What This Changes

The PR adds a bounded `sessions.list` result cache with in-flight coalescing, keeps default list rows lighter by skipping transcript usage fallback, extends session-list tests, and relaxes two script shim signal types.

## Best Possible Solution

Keep the lightweight row optimization, but make result-cache invalidation central to every session store, transcript, and lifecycle mutation or key the cache on a reliable store/version signal before merging.

## Reproduction Assessment

Yes, from source inspection: cache an identical `sessions.list` request, then update or create a session through a chat/agent/session-event path that broadcasts `sessions.changed` outside `server-methods/sessions.ts`; the next identical list can reuse the old cached result until the 60s TTL.

## Solution Assessment

No. Caching the whole list result can be maintainable, but only if invalidation is centralized across all session mutation paths or tied to a store/version signal; this PR invalidates only one local helper.
