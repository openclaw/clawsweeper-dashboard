## Summary

Keep this PR open: it addresses the earlier changelog and broad invalidation concerns, but the new lifecycle invalidation still advances the cache version before the async session-store write is visible, which can cache pre-write rows under the post-bump key for the 60s TTL.

## What This Changes

The PR adds bounded caching and in-flight coalescing for `sessions.list`, keeps default async rows lightweight, updates gateway/session tests and changelog, and relaxes two script signal shim types.

## Best Possible Solution

Keep the lightweight list-row cache, but tie cache freshness to a store mutation/version signal that changes after writes are durable or include a reliable completed-writer/store-state key.

## Reproduction Assessment

Yes, from source inspection: cache an identical `sessions.list`, handle a lifecycle start/end event, call `sessions.list` before the unawaited persistence completes, then call it again after the writer resolves. The second call can reuse pre-write rows cached under the already-bumped version until TTL expiry.

## Solution Assessment

No. Whole-result caching is a reasonable optimization, but this implementation should invalidate after the session-store write is visible, not immediately after scheduling an async persistence task.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bump cache version after lifecycle writes:** `src/gateway/server-chat.ts:358`
  - body: The new version bump happens immediately after firing `persistGatewaySessionLifecycleEvent` as an unawaited promise. If a `sessions.list` request arrives before `updateSessionStoreEntry` finishes, it builds and caches the old store snapshot under the new version; after the write completes there is no second bump, so identical calls can serve stale lifecycle/status rows until the 60s TTL. Move the bump/invalidation to the persistence completion or store-writer seam, or avoid caching while lifecycle writes are in flight.
  - confidence: 0.91
