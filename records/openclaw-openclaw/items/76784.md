## Summary

Keep open: the performance direction is useful and current main does not already have whole-result `sessions.list` caching, but the current diff introduces a blocking stale-result case for time-relative `activeMinutes` queries.

## What This Changes

The PR adds bounded result caching and in-flight coalescing around Gateway `sessions.list`, keeps default async rows lighter, updates gateway/session tests and changelog, and relaxes two script signal shim types.

## Best Possible Solution

Keep the cache for stable list parameters, but bypass or make it time-aware for `activeMinutes` requests and add regression coverage for rows aging past the active window.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cache an `activeMinutes` list while a row is still inside the window, wait until its `updatedAt` falls below the moving cutoff, then repeat the identical request before the 60 second cache TTL expires.

## Solution Assessment

No, not as currently implemented. The mutation-version cache is a narrow maintainable optimization for stable list calls, but `activeMinutes` depends on `Date.now()` and needs to bypass the cache or use a time-aware key/TTL.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bypass the list cache for activeMinutes queries:** `src/gateway/server-methods/sessions.ts:156`
  - body: `activeMinutes` results are time-relative: `listSessionsFromStoreAsync` computes the cutoff from `Date.now()`. Caching an identical request for the full 60s TTL means a row that was active when cached can keep appearing after it ages out, until the cache expires. Bypass the result cache for `activeMinutes` or make the key/TTL time-aware.
