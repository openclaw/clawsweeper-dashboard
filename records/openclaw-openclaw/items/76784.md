## Summary

Keep open: the PR is a useful gateway performance optimization, but it still introduces a P2 cache-invalidation race where lifecycle/status rows can stay stale for the cache TTL after an unawaited lifecycle persistence write.

## What This Changes

The branch adds bounded whole-result caching and in-flight coalescing for gateway `sessions.list`, keeps default list rows lighter, updates related gateway/session tests and changelog, and relaxes two script signal shim types.

## Best Possible Solution

Keep the lightweight list-row cache, but tie freshness to a store mutation signal that changes after writes are durable or suppress caching while lifecycle writes are in flight.

## Reproduction Assessment

Yes, from source inspection: cache an identical `sessions.list`, handle a lifecycle start/end event, call `sessions.list` before the unawaited persistence completes, then call it again after the writer resolves. The second call can reuse pre-write rows cached under the already-bumped version until TTL expiry.

## Solution Assessment

No. Whole-result caching is a reasonable optimization, but this implementation should invalidate after the session-store write is visible, not immediately after scheduling an async persistence task.
