## Summary

Keep open: the latest PR head still has a cache correctness blocker for time-relative `activeMinutes` session-list queries, even though the optimization direction is otherwise narrow and relevant.

## What This Changes

The PR adds bounded `sessions.list` result caching and in-flight coalescing, keeps default list rows lighter, extends gateway/session tests and changelog, and relaxes two local script signal shim types.

## Best Possible Solution

Keep the cache for stable list queries, but bypass it or make it time-aware for `activeMinutes` requests and cover rows aging past the active window.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cache an `activeMinutes` result while a row is inside the window, let its `updatedAt` age past the moving cutoff before the 60-second cache TTL expires, then repeat the identical request and receive a stale row.

## Solution Assessment

No. The mutation-version cache is a maintainable optimization for stable list calls, but `activeMinutes` depends on wall-clock time and needs a bypass, shorter time-bound TTL, or time-bucketed key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Bypass the list cache for activeMinutes queries:** `src/gateway/server-methods/sessions.ts:156`
  - body: `activeMinutes` results are time-relative because `listSessionsFromStoreAsync` filters against `Date.now()`. Caching an identical request for the full 60-second TTL lets a row that was active when cached keep appearing after it ages out, until the cache expires. Bypass this cache path for `activeMinutes` or make the cache key/TTL time-aware.
