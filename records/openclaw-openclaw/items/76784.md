## Summary

Keep open: the PR is useful, but the session-list cache can still serve stale lifecycle/status rows because the new invalidation is triggered before the async lifecycle store write is durable.

## What This Changes

The PR adds bounded whole-result caching and in-flight coalescing for `sessions.list`, keeps default async rows lighter, updates gateway/session tests and changelog, and loosens two script signal shim types.

## Best Possible Solution

Keep the lightweight list-row cache, but make cache freshness follow completed session-store mutations rather than scheduled lifecycle persistence work.

## Reproduction Assessment

Yes. From source inspection, cache an identical `sessions.list`, trigger a lifecycle start/end event, call `sessions.list` before the unawaited persistence finishes, then call it again after the writer resolves; the second call can reuse pre-write rows until TTL expiry.

## Solution Assessment

No. Whole-result caching is a reasonable optimization, but this implementation should invalidate after the session-store write is visible or use a store mutation signal that changes only after writes complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bump cache version after lifecycle writes:** `src/gateway/server-chat.ts:358`
  - body: The new bump runs immediately after launching `persistGatewaySessionLifecycleEvent` as an unawaited promise. If `sessions.list` runs before `updateSessionStoreEntry` finishes, it can cache the old store rows under the already-bumped version; after the write resolves there is no second bump, so identical requests can serve stale lifecycle/status rows for the 60s TTL. Move invalidation to persistence completion or the store-writer seam.
  - confidence: 0.91
