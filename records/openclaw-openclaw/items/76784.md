## Summary

Keep open: the PR still has a blocking cache freshness issue because the new `sessions.list` result cache is invalidated only from one local sessions helper while current main has other session-change broadcasters; it also needs the required changelog entry.

## What This Changes

The PR adds bounded whole-result caching and in-flight coalescing for `sessions.list`, skips default transcript usage fallback in async list rows, extends tests, and adjusts two script shim signal types.

## Best Possible Solution

Keep the lightweight list-row optimization, but make cache freshness central or store-version-backed and add the Unreleased changelog entry before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: cache one identical `sessions.list` call, update session state through `server-session-events`, `server-chat`, or `server-methods/agent`, then repeat the call before the 60s TTL and receive stale rows.

## Solution Assessment

No. Whole-result caching can be maintainable, but clearing only from the local sessions helper is not enough; a shared session-change invalidation seam or reliable store-version key is the narrower safe fix.
