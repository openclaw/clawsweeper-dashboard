## Summary

This PR reverses a newly established current-main contract: `--channel beta` intentionally falls back to latest when beta is missing or older, while `--tag beta` provides raw dist-tag selection. The linked report does not include registry evidence showing that this fallback caused the stale beta target, so changing the documented compatibility behavior is not currently justified.

## What This Changes

The PR always resolves the beta channel to the npm beta dist-tag, removes beta-to-latest fallback handling from the update command, and updates resolver tests accordingly.

## Best Possible Solution

Retain the current beta fallback unless maintainers explicitly decide that persistent beta selection must always follow the raw dist-tag; if that policy changes, align both docs pages and prove fresh-install, newer-stable-to-beta, missing-beta, downgrade-confirmation, and persisted-channel behavior.

## Reproduction Assessment

Partly yes: the beta-to-latest fallback is clear and reproducible from current source, but the linked beta-3-versus-beta-4 failure lacks captured registry values proving that this comparison caused it.

## Solution Assessment

No. Deleting the fallback reverses a current documented contract without establishing the linked report's root cause; registry publication should be fixed at its owner boundary if the beta dist-tag itself was stale.
