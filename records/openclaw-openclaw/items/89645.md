## Summary

Keep open: the patch removes a shipped and documented beta-channel fallback, so it is not a cleanup-close candidate and it is not merge-ready without a maintainer decision on channel semantics plus stronger proof.

## What This Changes

The PR removes npm beta-to-latest fallback resolution, updates resolver tests to expect raw beta dist-tag selection, and deletes CLI dry-run/downgrade handling for that fallback.

## Best Possible Solution

Preserve the fallback contract unless maintainers explicitly approve raw persistent beta semantics; if they do, align docs, CLI/startup tests, and upgrade/downgrade proof before merge.

## Reproduction Assessment

Partly yes: source inspection shows current main can choose `latest` instead of beta when beta compares older, and the PR removes that path. The original beta-3-versus-beta-4 report is not live-reproduced because before output and historical dist-tag values are missing.

## Solution Assessment

No. Deleting the fallback is a compatibility policy change, not the narrowest proven repair for a stale beta result; preserving fallback while diagnosing metadata freshness is safer unless maintainers choose a new contract.
