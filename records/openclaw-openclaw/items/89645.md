## Summary

Keep open, but this PR is not merge-ready: it removes a documented beta-channel fallback that current main, tests, docs, startup auto-update, and the latest release still preserve, and the remaining decision is maintainer-owned compatibility policy rather than a safe cleanup close.

## What This Changes

The PR changes npm beta channel resolution so explicit beta updates return the beta dist-tag directly and removes CLI fallback/downgrade handling for beta-to-latest fallback.

## Best Possible Solution

Preserve the shipped beta fallback unless maintainers explicitly approve raw persistent beta selection; if policy changes, align docs and cover dry-run, missing-beta, newer-stable downgrade, --yes, and startup auto-update paths.

## Reproduction Assessment

Partly yes: source inspection shows current main can choose latest instead of beta when beta compares older, and live registry headers support a possible stale metadata window. The exact historical beta-3 versus beta-4 result is not high-confidence reproduced because the PR lacks before output and captured old dist-tag values.

## Solution Assessment

No. Deleting the fallback is a compatibility policy change, not the narrowest repair for a stale beta result; the safer path is to preserve fallback and diagnose metadata freshness unless maintainers intentionally change the public channel contract.
