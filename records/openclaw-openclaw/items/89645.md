## Summary

Keep open, but this PR is not merge-ready: it changes the documented persistent beta-channel contract and can affect existing package or auto-update users, while the linked issue still needs a maintainer decision about fallback semantics versus registry freshness.

## What This Changes

The PR removes beta-to-latest fallback from npm channel resolution, updates resolver tests to expect raw beta dist-tag selection, and removes the CLI fallback note/downgrade suppression.

## Best Possible Solution

Preserve the shipped beta fallback unless maintainers explicitly approve raw persistent beta selection; if policy changes, align docs and add fresh-install, missing-beta, newer-stable downgrade, --yes, and auto-update coverage.

## Reproduction Assessment

Partly yes: source inspection shows current main can choose latest instead of beta when beta compares older, and live registry headers show beta/latest responses are cacheable for 300 seconds. The exact historical beta-3-versus-beta-4 result is not live-reproduced because before output and old dist-tag values were not captured.

## Solution Assessment

No. Deleting the fallback is a compatibility policy change, not the narrowest repair for a stale beta result; the safer path is to preserve fallback and diagnose metadata freshness, or intentionally change the public contract with docs and upgrade proof.
