## Summary

Keep open, but the patch should not merge as-is: it removes the documented beta-to-latest fallback for persistent `--channel beta` updates, while current `main`, the latest release, docs, tests, and git-channel behavior still preserve that fallback contract.

## What This Changes

The PR removes package beta-channel fallback to `latest` and updates resolver tests so `--channel beta` always targets the npm `beta` dist-tag.

## Best Possible Solution

Preserve the shipped beta fallback unless maintainers explicitly approve raw persistent beta selection; if the policy changes, align docs and add fresh-install, missing-beta, newer-stable downgrade, `--yes`, and auto-update coverage.

## Reproduction Assessment

Partly yes: source inspection reproduces the current beta fallback path and the PR's removal of it. No high-confidence live repro ties the linked beta-3-versus-beta-4 report to this fallback comparison because before output and npm tag values are missing.

## Solution Assessment

No. Deleting the fallback is a compatibility policy change, not the narrowest repair for a stale beta result; the safer path is to preserve fallback and diagnose metadata freshness, or intentionally change the public contract with docs and upgrade proof.
