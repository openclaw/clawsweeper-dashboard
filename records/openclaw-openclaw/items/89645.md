## Summary

Keep open, but do not merge as-is: the patch replaces the shipped and documented beta-channel fallback contract with raw beta dist-tag selection, which can change existing update and auto-update behavior. The linked report remains valid context, but this PR needs maintainer approval for the semantics change plus stronger before/after proof.

## What This Changes

The PR removes beta-to-latest fallback from npm channel resolution, updates resolver tests to expect raw beta dist-tag selection, and removes the CLI fallback note and downgrade-risk suppression.

## Best Possible Solution

Preserve the shipped beta fallback unless maintainers explicitly approve raw beta-channel semantics; if they do, align docs and add fresh-install, missing-beta, newer-stable downgrade, --yes, and startup auto-update proof.

## Reproduction Assessment

Partly yes: source inspection shows current main can choose latest instead of beta when beta compares older, and live npm registry headers show beta/latest responses are cacheable for 300 seconds. The exact historical beta-3-versus-beta-4 result is not high-confidence reproduced because the report and PR proof lack captured before output and old dist-tag values.

## Solution Assessment

No. Deleting the fallback is a compatibility policy change, not the narrowest proven repair for a stale beta result; the safer path is to preserve the fallback and diagnose registry freshness, or intentionally change the contract with docs and upgrade proof.
