## Summary

Keep open: the PR is a focused patch, but it removes a shipped and documented beta-channel fallback rather than proving the linked stale-beta symptom needs raw beta dist-tag semantics. Current main, docs, tests, and the latest release still define beta as preferring beta with latest/stable fallback, so this needs maintainer product/compatibility approval before merge.

## What This Changes

The PR changes update resolution so `openclaw update --channel beta` returns the beta npm dist-tag directly and removes the CLI fallback note/downgrade suppression for beta-to-latest fallback.

## Best Possible Solution

Preserve the shipped beta fallback unless maintainers explicitly approve raw persistent beta selection; if that policy changes, align docs and add fresh-install, downgrade, `--yes`, dry-run, and startup auto-update coverage.

## Reproduction Assessment

Partly yes: source inspection shows current main can choose latest instead of beta when beta compares older, matching the behavior this PR removes. The exact historical beta-3-versus-beta-4 result is not reproduced today because current registry beta is newer than latest and no before output was captured.

## Solution Assessment

No. Deleting the fallback is a compatibility policy change, not the narrowest proven repair for a stale beta result; the safer path is to preserve fallback and diagnose metadata freshness, or intentionally change the public contract with docs and upgrade proof.
