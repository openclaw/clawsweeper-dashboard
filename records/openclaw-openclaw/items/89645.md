## Summary

Keep this PR open, but it is not merge-ready: the patch removes a shipped and documented beta-to-latest fallback, while the linked report still lacks registry/output evidence proving that this fallback caused the stale beta result.

## What This Changes

The PR changes package update resolution so `openclaw update --channel beta` always targets the npm `beta` dist-tag instead of falling back to `latest` when beta is missing or older.

## Best Possible Solution

Preserve the shipped beta fallback unless maintainers explicitly choose raw persistent beta selection; if that policy changes, update both docs pages and add fresh-install, missing-beta, newer-stable downgrade, `--yes`, and auto-update coverage.

## Reproduction Assessment

Partly yes: the current fallback behavior is source-reproducible through `resolveNpmChannelTag`, tests, and docs. No high-confidence reproduction yet ties the linked beta-3-versus-beta-4 report to this resolver because registry values and before output are missing.

## Solution Assessment

No. Deleting the fallback is a policy change, not the narrowest fix for the reported stale beta result; the safer path is to keep fallback and diagnose the release/dist-tag owner boundary, or explicitly change the public contract with docs and upgrade proof.
