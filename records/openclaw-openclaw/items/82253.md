## Summary

Keep open: current main and the latest release still lack per-channel Slack `replyToMode`, but this PR is not merge-ready because it applies the new setting only to inbound automatic replies while same-channel Slack tool sends can still use account/chat-type threading.

## What This Changes

The PR adds an optional Slack per-channel `replyToMode` config key, applies it during inbound message preparation, updates generated schema/types/docs, and adds focused Slack tests.

## Best Possible Solution

Carry one effective Slack reply mode through inbound routing and same-channel Slack tool delivery, then align docs, tests, generated metadata, and redacted real Slack proof around that chosen behavior.

## Reproduction Assessment

Yes. Source inspection shows PR-head inbound routing uses `channelConfig.replyToMode`, while Slack tool auto-threading still derives `replyToMode` from account/chat type and current main action runtime uses that context to inject `threadTs`.

## Solution Assessment

No. The inbound override is useful but too narrow unless the same effective mode reaches same-channel Slack tool delivery, or the docs explicitly scope the setting to inbound automatic replies only.
