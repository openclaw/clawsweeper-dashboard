## Summary

Keep open: this is a focused Slack feature PR, but the current patch is incomplete because it applies the per-channel mode during inbound preparation while final reply and message-tool threading paths still re-resolve from account/chat-type config.

## What This Changes

Adds Slack per-channel `replyToMode` config/schema/docs/generated metadata and applies it in inbound Slack preparation, including regex mention reroute coverage.

## Best Possible Solution

Carry the resolved Slack reply mode from inbound preparation through the finalized turn context and shared threading contracts so final replies, follow-ups, and Slack message-tool sends all honor per-channel precedence.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR resolves `replyToMode` on the prepared Slack routing path, while `ChannelThreadingContext`, final reply resolution, and Slack tool context still lack a resolved-mode field. I did not run live Slack or tests in this read-only review.

## Solution Assessment

No. The feature is useful, but the patch should propagate the resolved mode through `FinalizedMsgContext`/`ChannelThreadingContext` and prefer it in final reply/tool threading instead of only changing preparation.
