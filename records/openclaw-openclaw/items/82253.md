## Summary

This PR should stay open: it adds useful Slack config and the supplied proof is adequate for the preparation path, but source inspection shows the resolved per-channel mode still does not reach shared final-reply and message-tool threading. The related PR https://github.com/openclaw/openclaw/pull/57309 points at the same propagation boundary.

## What This Changes

The PR adds Slack per-channel `replyToMode` config/schema/docs/generated metadata and applies it while preparing inbound Slack routing, including regex mention reroutes.

## Best Possible Solution

Carry the resolved Slack reply mode from inbound preparation into the finalized turn context and channel threading contracts, then prefer it in final reply filtering and Slack message-tool threading with focused regression coverage.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR resolves the per-channel mode in preparation, while final reply filtering and Slack tool context still resolve from account/chat-type state. I did not run live Slack or tests in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but the patch needs to propagate the resolved mode through the shared reply/tool threading path instead of stopping at `PreparedSlackMessage`.
