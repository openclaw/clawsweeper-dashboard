## Summary

Keep open: the latest head fixes the earlier Slack regex-reroute gap and now supplies usable real-behavior proof, but the patch still leaves one reply-mode path incomplete because message-tool/threading context can re-resolve the account mode instead of the per-channel override.

## What This Changes

Adds optional Slack per-channel `replyToMode` to config/schema/docs/generated metadata and uses it in inbound Slack routing, including the regex mention reroute path, with tests.

## Best Possible Solution

Carry the resolved Slack `replyToMode` into the finalized inbound context and channel threading context so auto replies, message-tool sends, followups, and tests all honor the same per-channel precedence.

## Reproduction Assessment

Yes for the review finding via source inspection: the PR computes a per-channel mode for prepared Slack routing, while current shared threading/tool context only receives chat type and re-resolves account mode. I did not run PR tests or live Slack messages in this read-only sweep.

## Solution Assessment

No. The routing change is close, but the maintainable fix must also pass the pre-resolved per-channel mode through `ctxPayload`/`ChannelThreadingContext` and cover message-tool/threading context behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Propagate the resolved mode into tool threading:** `extensions/slack/src/monitor/message-handler/prepare-routing.ts:179`

Store the resolved mode on the finalized context and pass it through the channel threading context, with a test for the message-tool path.
