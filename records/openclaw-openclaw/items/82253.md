## Summary

Keep open: this is a focused Slack feature PR that is not already implemented on current main or in v2026.6.6, but it is not merge-ready because the new per-channel mode affects automatic inbound replies while same-channel Slack message/upload tool delivery can still use the account/chat-type reply mode.

## What This Changes

The PR adds `channels.slack.channels.<id>.replyToMode` to Slack config/schema/docs/generated metadata and uses resolved channel config in inbound Slack reply routing, including regex mention reroute.

## Best Possible Solution

Use one effective Slack reply mode for both automatic inbound replies and same-channel message/upload tool delivery, then provide current-head redacted proof for both paths before merge.

## Reproduction Assessment

Yes for the blocking PR defect by source inspection. At the PR head, inbound routing reads `channelConfig?.replyToMode`, while `buildSlackThreadingToolContext` still resolves account/chat-type mode and `action-runtime` threads same-channel sends when that mode is `all`.

## Solution Assessment

No. The PR is a plausible inbound-reply change, but the best fix is to carry the effective Slack reply mode across automatic replies and same-channel message/upload tools, or explicitly scope the docs to inbound-only behavior.
