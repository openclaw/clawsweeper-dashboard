## Best Possible Solution

Share or move the omitted-input catalog fallback into the canonical model resolver path, using the same provider-scoped matching semantics, then cover both omitted-input success and provider-scoped explicit text-only rejection.

## Reproduction Assessment

Yes, by source inspection: current main accepts provider-scoped configured model ids, while the PR helper checks only raw id equality before merging catalog image input. I did not run tests because this was a read-only checkout review.

## Solution Assessment

No: the branch is a plausible workaround, but the safer fix belongs in or behind the canonical model resolver so explicit input detection and catalog fallback use one matching contract.
