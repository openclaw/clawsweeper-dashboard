## Summary

Keep this issue open. Current main still has a WhatsApp-plugin-owned automated recipient path that can explain the report without proving a literal JSON config mutation: heartbeat/cron recipient resolution merges configured WhatsApp allowFrom with the persisted pairing allow-store, and current tests explicitly expect pairing-store-only numbers to be returned as automated recipients. The linked explicit outbound allowlist bypass is a separate closed issue and does not cover this remaining heartbeat/cron delivery behavior.

## What This Changes

The issue reports that an unknown WhatsApp phone number appeared in allowFrom and received Eye Rest Reminder cron messages despite the user running dmPolicy=allowlist with only three authorized numbers.

## Best Possible Solution

Keep the fix inside the WhatsApp plugin: automated heartbeat/cron delivery in strict allowlist setups should use explicit --to, the active session only when it is also configured in allowFrom, or configured allowFrom itself; pairing-store approvals should remain DM access state unless the user explicitly opts into using them as automation recipients, and status/log output should label config entries separately from pairing-store entries.

## Reproduction Assessment

Yes for the code-level automated-send path: the current heartbeat-recipient tests reproduce pairing-store entries being returned as WhatsApp automation recipients. The literal claim that the JSON allowFrom config was silently mutated is not reproduced by the inspected code.

## Solution Assessment

Yes, the narrow WhatsApp-plugin-owned fix described in the maintainer comment is the best path. A core migration or broad config rewrite would miss the boundary: the risky behavior is delivery recipient resolution, not normal config editing.
