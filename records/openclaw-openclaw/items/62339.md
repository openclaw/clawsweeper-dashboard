## Summary

Keep open: current main still has a WhatsApp automation recipient path that merges configured `allowFrom` with the pairing allow-store, so cron/heartbeat delivery can include pairing-store-only numbers even though normal setup/config paths do not show a silent `allowFrom` mutation.

## What This Changes

The issue reports an unauthorized WhatsApp phone number appearing in `allowFrom` and receiving Eye Rest Reminder cron output despite a strict allowlist.

## Best Possible Solution

Keep the fix in the WhatsApp plugin and make strict allowlist automation recipients explicit: `--to`, configured `allowFrom`, or an active session only when it is also configured; status/log output should distinguish configured recipients from pairing-store DM approvals.

## Reproduction Assessment

Yes for the code-level automated-send path: current resolver code and tests show pairing-store entries being returned as WhatsApp automation recipients. No high-confidence reproduction exists for a literal `openclaw.json` mutation.

## Solution Assessment

Yes, the narrow WhatsApp-plugin-owned fix is the best direction. A broad core migration or config rewrite would miss the observed boundary, which is recipient resolution for automated delivery.
