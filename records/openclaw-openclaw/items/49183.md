## Summary

Keep open for maintainer product/security review. Current main and v2026.5.19 still have provider, sender, group, and inherited tool-policy primitives, but they do not expose a scope-keyed tool-policy config or thread authenticated Gateway operator scopes into effective agent tool access.

## What This Changes

Request adding session/operator-scope-aware tool policy configuration that connects Gateway authenticated scopes to agent tool access resolution.

## Best Possible Solution

Define a reviewed operator/session-scope tool-policy contract, thread authenticated Gateway scopes into the shared resolver, document precedence against sender/provider/owner-only/inherited gates, and add regression coverage for distinct read/write/admin tool surfaces.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new authorization/config surface. Source and release inspection give a high-confidence verification path that current main and v2026.5.19 lack scope-keyed tool policy inputs and caller-scope policy resolution.

## Solution Assessment

Unclear. The proposed toolsByScope direction is plausible, but maintainers still need to choose the config home, enforcement layer, precedence rules, and trust-model relationship with sender identity and owner-only tools.
