## Summary

Keep open. Current main still does not connect gateway operator scopes to agent tool-policy resolution; it only has adjacent global, provider, agent, group, sender, subagent, gateway, sandbox, and owner-only controls, so the RFC remains a valid security-sensitive product decision.

## What This Changes

The issue requests session-level tool policies that use gateway operator scopes or sender identity to restrict the tools available inside an agent session.

## Best Possible Solution

Keep the RFC open until maintainers define and document a typed session/operator-scope policy input, thread authenticated gateway scopes into the shared tool-policy path, and add regression coverage for read/write/admin callers without weakening owner-only gates.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/RFC. There is a high-confidence source verification path showing no toolsByScope config field, no caller-scope input in the resolver, and only senderIsOwner derived from operator.admin in the gateway agent path.

## Solution Assessment

Unclear. The proposed toolsByScope shape is plausible, but the best solution needs maintainer product/security judgment because it crosses gateway auth, channel sender identity, tool-policy merge precedence, and owner-only tool semantics.
