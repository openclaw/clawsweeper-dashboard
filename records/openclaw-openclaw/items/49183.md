## Summary

Keep open for maintainer product/security review. Current main and v2026.5.22 have sender, group, provider, subagent, and inherited tool-policy primitives, but they still do not expose a scope-keyed tool-policy contract or thread authenticated Gateway operator scopes into effective agent tool access.

## What This Changes

Requests a per-session tool-policy layer that maps authenticated Gateway operator scopes to effective agent tool availability.

## Best Possible Solution

Define a reviewed operator/session-scope tool-policy contract, thread authenticated Gateway scopes into the shared resolver, document precedence against sender/provider/owner-only/inherited gates, and add regression coverage for distinct read/write/admin tool surfaces.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new authorization/config surface. Source and release checks give high confidence that current main and v2026.5.22 lack scope-keyed tool policy inputs and caller-scope policy resolution.

## Solution Assessment

Unclear. The proposed toolsByScope direction is plausible, but maintainers still need to choose the config home, enforcement layer, precedence rules, and trust-model relationship with sender identity and owner-only tools.
