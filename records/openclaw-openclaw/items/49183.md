## Summary

Keep open: current main and v2026.6.6 still have gateway operator scopes plus global, provider, agent, group, sender, subagent, and inherited tool-policy layers, but no operator-scope-to-tool-policy bridge or `toolsByScope` config. Related direct-session and Feishu slices are closed unmerged, while broader tenant/RBAC issues are adjacent rather than a complete replacement.

## What This Changes

The issue requests session-level tool-policy overrides keyed by gateway operator scopes, with gateway scope handoff into agent tool resolution.

## Best Possible Solution

Keep the current trust model until maintainers choose the scope/tool precedence; if accepted, implement additive config/schema, gateway scope handoff, a policy-pipeline layer, docs, and regression tests as narrow PRs.

## Reproduction Assessment

Not applicable: this is an RFC for a new authorization/config capability, not a broken existing behavior report. Source, docs, release-tag checks, and related items give high-confidence evidence that the requested scope-to-tool-policy bridge is still absent.

## Solution Assessment

Unclear: `toolsByScope` is a plausible implementation shape, but maintainers first need to decide whether gateway scopes should become a runtime tool-authorization boundary and how that layer composes with existing sender/group/agent policies.
