## Summary

Keep open: current main and v2026.6.10 still have gateway operator scopes and layered agent tool policies, but they do not bridge operator scopes into runtime tool-policy resolution or expose the requested `toolsByScope` config. Existing sender/group policies, session-envelope work, and multi-tenant RBAC trackers are useful adjacent paths, but they do not settle the security/product boundary for this RFC.

## What This Changes

Request to add session-level tool policy overrides keyed by gateway operator scope or sender identity so different sessions/users can have different agent tool access.

## Best Possible Solution

Decide the durable authorization model first, then either add a typed, documented scope-aware tool-policy layer with upgrade-safe precedence or route this need to sender/group policies, session envelopes, or the broader tenant RBAC design.

## Reproduction Assessment

Not applicable. This is an RFC for a new authorization/config capability; source and release inspection show the requested scope-to-tool-policy bridge is absent rather than a broken existing path.

## Solution Assessment

Unclear. `toolsByScope` is a plausible slice, but current docs define operator scopes as a trusted-domain control-plane guardrail, so maintainers should choose between scope-aware tool policy, session envelopes, sender/group policies, and tenant RBAC before implementation.
