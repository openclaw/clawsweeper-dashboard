## Summary

Keep open for maintainer product/security review. Current main and v2026.5.22 still have Gateway operator scopes plus global, per-agent, provider, group, and sender tool policies, but no scope-keyed tool-policy contract or authenticated Gateway-scope input to effective agent tool access.

## What This Changes

The issue requests a new per-session tool-policy config/auth bridge that maps Gateway operator scopes or sender identity to effective agent tool access.

## Best Possible Solution

Define a reviewed operator/session-scope tool-policy contract, thread authenticated Gateway scopes into the shared resolver, document precedence against sender/group/provider/owner-only/inherited gates, and add regression coverage for distinct read/write/admin tool surfaces.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new authorization/config surface. Source and release checks give high confidence that current main and v2026.5.22 lack scope-keyed tool-policy inputs and caller-scope policy resolution.

## Solution Assessment

Unclear. The proposed toolsByScope direction is plausible, but maintainers still need to decide the config home, enforcement layer, precedence rules, and trust-model relationship with sender identity and owner-only tools.
