## Summary

Keep open: current main has sender-scoped and inherited tool-policy primitives, but no operator-scope-keyed config or caller-scope input, and the remaining authorization boundary needs maintainer product/security review.

## What This Changes

The issue proposes per-session or scope-based tool-policy overrides so authenticated Gateway operator scopes can restrict agent tool access.

## Best Possible Solution

Define a reviewed operator/session-scope policy contract, thread authenticated Gateway scopes into the shared tool-policy path, document precedence against sender and owner-only gates, and add regression coverage for distinct read/write/admin tool surfaces.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new authorization/config surface. Source inspection gives a high-confidence verification path: current main has sender-scoped policies, but no scope-keyed config field or caller-scope parameter in effective tool-policy resolution.

## Solution Assessment

Unclear. The proposed `toolsByScope` direction is plausible, but maintainers still need to choose the config home, merge precedence, enforcement layer, and interaction with sender identity and owner-only tools.
