## Summary

Keep open. Current main still does not bridge authenticated gateway operator scopes into shared agent tool-policy resolution; it has adjacent global, provider, agent, group/sender, subagent, gateway, sandbox, and owner-only controls, but not the RFC’s operator-scope policy boundary.

## What This Changes

This issue requests session-level tool-policy overrides keyed by gateway operator scopes so multi-user deployments can restrict agent tools per connected caller.

## Best Possible Solution

Define a reviewed session/operator-scope policy contract, thread authenticated gateway scopes into the shared tool-policy path, and add coverage proving distinct read/write/admin tool surfaces without weakening owner-only gates.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new policy surface. Source inspection gives a high-confidence verification path: current main has no scope-keyed config field, no scope parameter in the resolver, and only converts operator.admin to senderIsOwner in the gateway agent path.

## Solution Assessment

Unclear. The proposed toolsByScope shape is plausible, but the best fix needs maintainer product/security judgment on config shape, merge precedence, enforcement layer, and interaction with sender and owner-only policy.
