## Summary

Keep open: current main and v2026.6.6 have global/provider/agent/group/sender tool-policy layers plus gateway method scopes, but not the requested operator-scope-to-tool-policy bridge. The remaining work is a security/config product decision about whether gateway scopes should become a per-session tool authorization boundary.

## What This Changes

The issue requests session-level tool policy overrides that connect gateway operator scopes with agent tool availability for multi-user deployments.

## Best Possible Solution

Keep this RFC open for maintainer-approved security design, then split any approved direction into narrow PRs for config/schema, gateway identity handoff, tool-policy resolution, docs, and focused tests.

## Reproduction Assessment

Not applicable: this is an RFC for a new authorization/config capability, not a broken existing behavior report. Source and docs provide high-confidence evidence that current main lacks the requested operator-scope tool-policy bridge.

## Solution Assessment

Unclear: toolsByScope is a plausible shape, but it is not yet proven to be the best solution because maintainers first need to decide the trust boundary and precedence model. The currently supported alternatives are sender/group/agent tool policies or separate gateways for stronger separation.
