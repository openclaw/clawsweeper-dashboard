## Summary

Keep open: current main and the latest release have sender/group/agent tool-policy layers and gateway method scopes, but not the requested operator-scope-to-tool-policy bridge. Related discussion still treats this RFC as the focused RBAC follow-up, and the trust-model/config change needs maintainer security and product direction.

## What This Changes

Request to add per-session tool-policy overrides that connect gateway operator scopes to the agent tool-access resolver.

## Best Possible Solution

Keep this RFC open for a maintainer-approved security design, then split any approved work into narrow PRs for config/schema, gateway identity handoff, agent tool-policy resolution, docs, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature/RFC, not a broken existing behavior report. Source and docs provide high-confidence capability evidence that current main lacks a toolsByScope/operator-scope tool-policy bridge.

## Solution Assessment

Unclear: toolsByScope is a plausible design, but it is not safe to implement automatically before maintainers decide whether operator scopes should become a per-session tool-authorization boundary. The current supported path remains sender/group/agent tool policies or separate gateways for stronger trust separation.
