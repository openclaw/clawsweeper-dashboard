## Summary

Keep open for maintainer product and security review. Current main and the latest release have provider, sender, group, inherited, and HTTP operator-scope primitives, but they still do not expose a scope-keyed tool-policy config or thread authenticated Gateway operator scopes into effective agent tool access.

## What This Changes

The issue requests a new per-session tool-policy contract that can restrict agent tools based on Gateway operator scopes, with possible later sender-based overrides.

## Best Possible Solution

Define a reviewed operator/session-scope tool-policy contract, thread authenticated Gateway scopes into the shared resolver, document precedence against sender/provider/owner-only/inherited gates, and add fail-closed tests for distinct read/write/admin tool surfaces.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new authorization/config surface. Source and release inspection give a high-confidence verification path that current main and v2026.5.20 lack scope-keyed tool policy inputs and caller-scope policy resolution.

## Solution Assessment

Unclear. The proposed toolsByScope direction is plausible, but maintainers still need to choose the config home, enforcement layer, precedence rules, and trust-model relationship with sender identity and owner-only tools.
