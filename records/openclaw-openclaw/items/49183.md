## Summary

Current main still does not implement the RFCâ€™s operator-scope-to-agent-tool-policy bridge. It has mature global, provider, agent, group, sender, subagent, sandbox, gateway, and owner-only tool filtering, but the policy resolver is not passed gateway operator scopes and the config schema does not expose a `toolsByScope` or equivalent session-scope override.

## What This Changes

Add session-level tool policy overrides that connect gateway operator scopes to agent tool resolution for multi-user deployments.

## Best Possible Solution

Keep this RFC open until maintainers settle and document a typed session/operator-scope-aware policy input in the shared tool-policy path, with gateway-auth scopes threaded into agent/tool resolution and regression coverage proving read/write/admin callers get distinct tool surfaces without weakening owner-only gates.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence verification path for the missing feature: search and inspect the config types, zod schema, gateway `agent` handler, method scopes, and tool-policy resolvers for `toolsByScope` or caller-scope inputs.

## Solution Assessment

Unclear. The proposed `toolsByScope` design is plausible, but the best fix needs maintainer product/security judgment because it crosses gateway auth, channel sender identity, shared tool-policy resolution, and owner-only tool semantics.
