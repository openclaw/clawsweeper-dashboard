## Summary

Keep open. Current main has useful sender-scoped and inherited tool-policy primitives, but it still does not bridge authenticated gateway operator scopes into the agent tool-policy resolver; the remaining RFC is a security/product design decision, not stale cleanup.

## What This Changes

The issue requests session-level tool filtering, likely via scope-keyed policy such as `toolsByScope`, so gateway operator scopes can restrict the tools available to an agent session.

## Best Possible Solution

Define a reviewed operator/session-scope policy contract, thread authenticated gateway scopes into the shared tool-policy path, document precedence against sender and owner-only gates, and add regression coverage for distinct read/write/admin tool surfaces.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC for a new authorization/config surface. Source inspection gives a high-confidence verification path: current main has sender-scoped policies, but no scope-keyed config field or caller-scope parameter in effective tool-policy resolution.

## Solution Assessment

Unclear. The proposed `toolsByScope` shape is plausible, but maintainers still need to choose the config home, merge precedence, enforcement layer, and interaction with sender identity and owner-only tools.
