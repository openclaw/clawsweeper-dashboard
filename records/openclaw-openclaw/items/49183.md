## Summary

Keep open: current main and v2026.6.6 still have gateway method scopes plus global, provider, agent, group, sender, subagent, and inherited tool-policy layers, but not the requested bridge from operator scopes into effective agent tool access. The related implementation slices are closed unmerged or open adjacent work, so this remains a maintainer security/config design decision rather than cleanup-close material.

## What This Changes

Add session-level tool policy filtering that connects gateway operator scopes, such as `operator.read` or `operator.admin`, to the agent runtime tool policy applied for that session.

## Best Possible Solution

Keep this RFC open until maintainers choose the trust model and policy precedence, then split any approved direction into narrow additive PRs for config/schema, gateway scope handoff, runtime policy resolution, docs, and focused tests.

## Reproduction Assessment

Not applicable: this is an RFC for a new authorization/config capability, not a broken existing behavior report. Source, docs, latest-release checks, and related items give high-confidence evidence that the requested operator-scope tool-policy bridge is still absent.

## Solution Assessment

Unclear: `toolsByScope` is a plausible implementation shape, but maintainers first need to decide whether gateway scopes should become a runtime tool-authorization boundary and how that layer composes with existing sender/group/agent policies.
