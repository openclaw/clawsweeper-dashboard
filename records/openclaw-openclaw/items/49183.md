## Summary

Keep this open: current main still lacks the requested operator-scope-to-agent-tool-policy bridge, and the remaining work is a security/product design decision rather than a narrow cleanup close.

## What This Changes

Add session-level tool filtering that maps gateway operator scopes to agent tool policy so read/write/admin sessions can receive different runtime tool surfaces.

## Best Possible Solution

Keep this RFC open until maintainers choose and document a typed session/operator-scope-aware policy input, thread gateway auth scopes into shared tool resolution, and add regression coverage proving distinct read/write/admin tool surfaces without weakening owner-only gates.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence source verification path: inspect the config types, schema/search results, gateway agent handler, method scopes, and tool-policy resolvers for missing operator-scope policy input.

## Solution Assessment

Unclear. The proposed `toolsByScope` direction is plausible, but the best implementation needs maintainer security/product judgment because it crosses gateway auth, shared tool-policy resolution, channel sender identity, HTTP surfaces, and owner-only semantics.
