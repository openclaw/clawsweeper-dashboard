## Label Justifications

- `P1`: The issue describes a user-facing Discord component workflow where acknowledged clicks can stall for minutes before the agent-visible flow or reply progresses.
- `impact:message-loss`: The reported failure delays or effectively suppresses Discord component interaction/reply delivery in the channel pipeline.

## Summary

Keep open: current main and v2026.6.9 still contain the reported `send_attempt_started` no-reconciliation recovery path, and Discord component replies still flow through the durable outbound delivery path that can enter that state. The report is source-supported but the full 6-12 minute Discord stall still needs live proof and a maintainer-approved ambiguous-send recovery contract.

## What This Changes

The issue asks OpenClaw to make Discord button/select component interactions deliver promptly instead of stalling for minutes with outbound delivery stuck in `send_attempt_started`.

## Best Possible Solution

Live-prove the Discord component stall, then implement a maintainer-approved recovery path that keeps acknowledged component clicks from remaining silently ambiguous without allowing unsafe duplicate sends.

## Reproduction Assessment

Yes at source level: a queued Discord outbound reply can enter `send_attempt_started`, and without `reconcileUnknownSend` the recovery path refuses blind replay. The full 6-12 minute live Discord component stall was not reproduced in this read-only review.

## Solution Assessment

Unclear as a final implementation choice. The current fail-closed recovery branch protects against duplicate sends, so the best fix needs maintainer direction on Discord reconciliation, timeout/failure surfacing, or guarded retry semantics.
