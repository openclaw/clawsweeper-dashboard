## Summary

Keep open: current main and v2026.6.10 still route Discord component replies through durable outbound delivery while Discord lacks unknown-send reconciliation, so `send_attempt_started` entries still hit the fail-closed recovery path instead of a prompt, verified component reply recovery. The adjacent Discord reconnect-drain PR is open and proof-blocked, and it does not replace this component-interaction stall report.

## What This Changes

The issue reports Discord button/select callbacks stalling for 6-12 minutes after entering `send_attempt_started`, despite successful Discord ACK and gateway receipt.

## Best Possible Solution

Add a Discord-owned ambiguous-send recovery path or session-visible failed-delivery signal for component reply delivery, backed by live Discord proof, while keeping blind duplicate sends impossible.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the source-level `send_attempt_started` recovery refusal is clear on current main, but the full 6-12 minute live Discord component timing pattern still needs live proof.

## Solution Assessment

Is this the best way to solve the issue? No for closing or autonomous repair now; the right fix needs maintainer direction on Discord unknown-send reconciliation, duplicate-safe retry, or session-visible failure signaling.
