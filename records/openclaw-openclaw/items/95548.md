## Summary

## What This Changes

The issue reports Discord component button/select callbacks that ACK but then stall 6-12 minutes in gateway delivery with queue entries stuck in `send_attempt_started`.

## Best Possible Solution

Prove the Discord component stall in a live or controlled gateway setup, then add a maintainer-approved Discord ambiguous-send reconciliation or timeout recovery path that avoids both stuck component replies and blind duplicate sends.

## Reproduction Assessment

Yes for the source-level failure path: current main still routes Discord component replies through durable outbound delivery and still refuses recovery for `send_attempt_started` without adapter reconciliation. No live Discord run in this read-only review reproduced the full 6-12 minute timing pattern.

## Solution Assessment

Unclear as a final implementation choice. The narrowest safe fix needs maintainer direction on Discord unknown-send reconciliation, pending/backoff behavior, or timeout/failure surfacing because the current fail-closed behavior protects against duplicate sends.
