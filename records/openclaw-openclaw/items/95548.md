## Summary

Keep open: current main and v2026.6.9 still have the reported `send_attempt_started` refusal path, Discord component replies flow through durable outbound delivery, and Discord does not advertise unknown-send reconciliation. The report is source-reproducible but still needs live Discord proof and maintainer direction before a repair is safe.

## What This Changes

The issue reports Discord button/select interactions that acknowledge in Discord but stall for minutes before reaching the agent/session delivery flow.

## Best Possible Solution

Prove the Discord component stall in a live or controlled gateway setup, then implement a maintainer-approved recovery path that prevents acknowledged component clicks from remaining in ambiguous outbound delivery without session-visible recovery.

## Reproduction Assessment

Yes, source-level for the recovery refusal: a queued delivery with `recoveryState: "send_attempt_started"` and no `reconcileUnknownSend` hits the same refusal path. The full 6-12 minute Discord component stall was not live-reproduced in this read-only review.

## Solution Assessment

Unclear as a final implementation choice. The narrowest safe fix needs maintainer direction on Discord unknown-send reconciliation, timeout/failure surfacing, or guarded retry semantics because the current fail-closed behavior protects against duplicate sends.
