## Label Justifications

- `P1`: A real user reports hidden reasoning text being delivered as a visible Telegram heartbeat reply, which is an urgent channel delivery regression.
- `impact:message-loss`: The issue is about heartbeat selecting and delivering the wrong visible channel payload instead of suppressing the final `HEARTBEAT_OK` path.

## Summary

Keep this issue open: current `main` and the latest release still select the last outbound heartbeat payload without excluding reasoning, and two open PRs explicitly close this issue but have not merged yet.

## What This Changes

The issue asks OpenClaw to stop heartbeat from delivering reasoning/thinking payloads as the main visible reply when `includeReasoning` is false.

## Best Possible Solution

Land one canonical heartbeat reasoning-selector fix that reuses the shared reasoning classifier and preserves explicit `includeReasoning` delivery, then close this issue after the merge.

## Reproduction Assessment

Yes. A current-main reply array like `[{ text: "HEARTBEAT_OK" }, { text: "hidden", isReasoning: true }]` is source-reproducible: the selector scans backward, treats the reasoning payload as outbound content, and the runner can deliver it as the main heartbeat reply.

## Solution Assessment

Yes. The narrow maintainable fix is in heartbeat reply selection and the shared reasoning classifier boundary, not in Telegram-specific delivery, with runner coverage to preserve opt-in `includeReasoning` behavior.
