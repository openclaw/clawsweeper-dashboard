## Label Justifications

- `P1`: The report describes private reasoning text being delivered into a real Telegram heartbeat workflow, which is a broken user-facing channel behavior.
- `impact:message-loss`: The issue is about the wrong heartbeat payload being delivered to a channel instead of suppressing reasoning and honoring the final reply.

## Summary

Keep open: current main and the latest release still have the reported heartbeat selector behavior, and two open PRs already carry closing references for the same fix path. The issue should remain open until one canonical PR lands.

## What This Changes

The issue asks OpenClaw to stop selecting heartbeat reasoning payloads as the visible main reply when `includeReasoning` is false or unset.

## Best Possible Solution

Review and land one canonical selector-layer fix, preferably the branch that reuses the existing SDK reasoning classifier and preserves opt-in `includeReasoning` delivery, then close this issue through the merged PR.

## Reproduction Assessment

Yes by source inspection. Current main selects the last outbound heartbeat payload without a reasoning check, so a result array ending in `{ isReasoning: true, text: ... }` can return that reasoning payload as the main reply.

## Solution Assessment

Yes. The best fix is a narrow heartbeat selector change that skips SDK-classified reasoning payloads while preserving the existing `includeReasoning: true` delivery lane; the issue should stay paired with the open closing PRs rather than spawning another repair job.
