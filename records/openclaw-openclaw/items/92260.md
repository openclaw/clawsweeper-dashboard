## Label Justifications

- `P2`: A heartbeat can send unintended reasoning text to a user-visible channel, but the bug is scoped to the heartbeat reply path.
- `impact:message-loss`: The issue is about incorrect channel delivery: reasoning can be selected as the visible heartbeat reply while the intended final heartbeat payload is suppressed.

## Summary

Keep open: current main and the latest release still select the last outbound heartbeat payload without excluding reasoning, while two open PRs already reference this issue with closing syntax and should remain the implementation path.

## What This Changes

The issue asks OpenClaw to stop heartbeat main-reply selection from delivering reasoning payloads when `includeReasoning` is unset or false.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92356 or an equivalent maintainer-preferred fix that skips reasoning in the heartbeat main-reply selector while preserving explicit `includeReasoning` delivery.

## Reproduction Assessment

Yes from source inspection: a `replyResult` array ending with `{ text: "hidden", isReasoning: true }` after `{ text: "HEARTBEAT_OK" }` will currently select the reasoning payload. I did not run a live Telegram/provider reproduction in this read-only review.

## Solution Assessment

Yes: the best fix is the narrow selector/runner path already proposed by the linked PRs, with regression coverage for default-hidden reasoning and explicit `includeReasoning: true` delivery. Adding a new config option or channel-specific workaround would be the wrong layer.
