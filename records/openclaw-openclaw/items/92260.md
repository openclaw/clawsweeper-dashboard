## What This Changes

The issue reports heartbeat selecting a reasoning/thinking payload as the visible channel reply when `includeReasoning` is false.

## Best Possible Solution

Land one canonical heartbeat reasoning selector fix that reuses the shared SDK reasoning classifier and preserves the explicit `includeReasoning` delivery path, then close this issue through the merge.

## Reproduction Assessment

Yes. A source-level repro is clear: a reply array ending with `{ text: "thinking", isReasoning: true }` is selected over the earlier final payload because the selector only checks outbound content.

## Solution Assessment

Yes, the best fix belongs at heartbeat reply selection while preserving the explicit `includeReasoning` reasoning-delivery path. The open PRs already target that layer, so this issue should wait for maintainer review and landing rather than a separate repair job.
