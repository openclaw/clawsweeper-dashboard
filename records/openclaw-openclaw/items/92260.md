## Summary

Current main still violates the documented heartbeat contract: the main-reply selector can choose a trailing reasoning payload even though reasoning delivery defaults off. Keep this issue open because two clean, mergeable PRs explicitly close it; https://github.com/openclaw/openclaw/pull/92263 currently provides the stronger end-to-end runner coverage and preserves the explicit opt-in path.

## What This Changes

Prevent heartbeat runs from delivering a reasoning payload as the main visible reply when `includeReasoning` is false.

## Best Possible Solution

Use one canonical internal fix—preferably the proof-positive shape in https://github.com/openclaw/openclaw/pull/92263—that selects the last non-reasoning main payload, preserves separate opt-in reasoning delivery, and covers flagged, formatted, scalar, array, silent-token, and visible-final cases at selector and runner levels.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction shape: pass `[{ text: "HEARTBEAT_OK" }, { text: "thinking", isReasoning: true }]` and current main selects the second payload. This read-only review did not run a live xAI-to-Telegram heartbeat.

## Solution Assessment

Yes at the ownership boundary: filtering the heartbeat main-reply selection is narrower and more maintainable than Telegram-specific suppression. The best implementation also reuses established reasoning classification and verifies that explicit opt-in reasoning delivery still works.
