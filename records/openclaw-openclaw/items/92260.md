## Label Justifications

- `impact:message-loss`: The selector can choose reasoning text instead of the intended final heartbeat reply, causing the wrong visible message to be delivered.

## Summary

Keep open: current main and latest release still select the last outbound heartbeat payload without excluding reasoning, and the open linked PR is the active fix path.

## What This Changes

The issue requests heartbeat main-reply selection to skip reasoning payloads when `includeReasoning` is false so thinking text is not delivered as the visible channel reply.

## Best Possible Solution

Land one canonical selector/runner fix that uses the shared reasoning classifier for main-reply selection while preserving the explicit `includeReasoning` Thinking path.

## Reproduction Assessment

Yes, at source level. A reply result shaped as final `HEARTBEAT_OK` followed by `{ text, isReasoning: true }` makes current `resolveHeartbeatReplyPayload` return the reasoning payload because it scans backward and checks only outbound content.

## Solution Assessment

Yes. The narrow maintainable solution is to change heartbeat main-reply selection so SDK-classified reasoning is not selected as the visible reply while preserving the separate `includeReasoning` Thinking path.
