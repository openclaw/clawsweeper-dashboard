## Summary

Keep open: current main and the latest release still select the heartbeat main reply without excluding reasoning payloads, and https://github.com/openclaw/openclaw/pull/92356 is the open candidate that should close this issue after merge.

## What This Changes

The issue reports that heartbeat reply selection can deliver a reasoning payload as the visible Telegram reply when `includeReasoning` is false.

## Best Possible Solution

Review and land one canonical selector-layer fix, preferably the open candidate PR if maintainers accept it, then close this issue as implemented.

## Reproduction Assessment

Yes, source-level: a reply array ending with `{ isReasoning: true, text: ... }` passes current selector checks and can become the main heartbeat reply. I did not run a live Telegram/provider reproduction in this read-only review.

## Solution Assessment

Yes, fixing the selector is the right layer because the docs make reasoning delivery opt-in and the runner already has a separate reasoning lane. The open candidate PR also keeps the runner and selector classifiers aligned through the shared SDK helper.
