## Summary

Current main already routes isolated cron `agentTurn` jobs onto fresh run-scoped session identity, which addresses the reported embedded session takeover path; the proof is main-only and not in the latest release tag shown in this checkout.

## What This Changes

The issue reports that isolated cron `agentTurn` jobs fail with `EmbeddedAttemptSessionTakeoverError` while the embedded prompt lock is released.

## Best Possible Solution

Keep the current main implementation and ship it; if the failure reproduces on a build containing `a7d2d9c6df002d718badc5500cf52f516d97d248` or later, reopen with job JSON and failing run diagnostics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence failure class: the embedded runner raises this error when its session file changes while the prompt lock is released. I did not run the macOS cron scenario locally, but current main has focused source and regression-test coverage for the isolated cron session identity path.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: isolated cron jobs get fresh run-scoped session identity, while explicit `session:<id>` cron jobs still keep their requested persistent session.
