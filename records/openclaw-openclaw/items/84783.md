## Summary

Keep this open: current `main` still has the same source-level Moonshot model-resolution path that can miss the fast `skipPiDiscovery` attempt and fall back into `models.json` generation before dispatch, while the reporter added shipped-release logs, TUI logs, and a config workaround that matches that path.

## What This Changes

Fix slow native Moonshot model resolution before Discord/TUI dispatch when no explicit `models.providers.moonshot` config is present.

## Best Possible Solution

Resolve bundled/native Moonshot model refs from provider-owned dynamic resolution or the existing static catalog during the `skipPiDiscovery` fast path, keeping broad `models.json` discovery as a later fallback.

## Reproduction Assessment

Yes at source level: native Moonshot without explicit `models.providers.moonshot` can miss the fast `skipPiDiscovery` resolution and enter `ensureOpenClawModelsJson`, matching the reporter's shipped-release and TUI traces. I did not perform a live Discord/TUI Moonshot run in this read-only pass.

## Solution Assessment

Yes in direction: the maintainable fix is to use the existing provider-owned/static-catalog model facts during fast resolution, not to require users to add explicit Moonshot config. The repair should prove the fast path with focused regression coverage.
