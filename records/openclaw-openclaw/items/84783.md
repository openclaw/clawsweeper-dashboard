## Summary

Keep this open: current `main` and the shipped 2026.5.19 code still route native Moonshot embedded runs from the lean `skipPiDiscovery` lookup into broad `models.json` generation before dispatch when no explicit `models.providers.moonshot` entry is configured, matching the reporter's 10-30s traces and workaround.

## What This Changes

Report asks OpenClaw to avoid a 10-30s native Moonshot model-resolution delay before Discord/TUI dispatch when no explicit `models.providers.moonshot` entry is configured.

## Best Possible Solution

Resolve known bundled/static Moonshot model refs before broad `models.json` generation in the embedded runner, keeping the explicit `models.providers.moonshot` block as a workaround rather than a requirement.

## Reproduction Assessment

Yes, at source level: run an embedded Discord/TUI turn with `moonshot/kimi-k2.6` and no explicit `models.providers.moonshot` model row, and current code can fall into `ensureOpenClawModelsJson` before dispatch. I did not perform a live credentialed Moonshot run.

## Solution Assessment

Yes in direction: the maintainable fix is to use the existing static-catalog or provider-owned dynamic resolution path before broad discovery, without requiring a new config knob or the user's explicit provider metadata workaround.
