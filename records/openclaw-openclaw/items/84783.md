## Summary

Keep this open: current main still has a source-level path where native Moonshot can miss the initial skip-PI model resolution and fall back to broad models.json/provider discovery before dispatch; the recent remote-TUI startup perf fix is adjacent but does not address embedded Discord/TUI dispatch resolution.

## What This Changes

The issue reports that native Moonshot runs can spend roughly 30 seconds in model-resolution before Discord or TUI dispatch starts.

## Best Possible Solution

Resolve bundled/native provider model refs through targeted provider-owned or static manifest metadata before broad models.json preparation, with regression coverage proving moonshot/kimi-k2.6 dispatch does not enter cross-provider discovery.

## Reproduction Assessment

No, not as a live current-main reproduction in this read-only pass. The reporter's logs plus current source show a credible path where native Moonshot misses the first skip-PI resolution and then falls into broad models.json/provider discovery.

## Solution Assessment

Yes in direction: the narrow maintainable fix is to use the existing provider-owned/static catalog data before broad discovery, without requiring users to add a new Moonshot config option.
