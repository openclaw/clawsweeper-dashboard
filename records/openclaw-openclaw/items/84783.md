## Summary

Keep this open: current main and v2026.6.10 still gate bundled static-catalog fallback to plugin-harness transport, so native Moonshot/Google runs can still miss the lean model-resolution pass and fall into broad discovery before dispatch. Earlier direct fix PRs are closed unmerged, and the newer open cron static-catalog PR is adjacent but does not cover this first-turn native-provider latency path.

## What This Changes

The issue reports slow first-turn native provider model resolution before Discord/TUI dispatch when provider model metadata is not configured inline.

## Best Possible Solution

Land a maintainer-approved, provider-agnostic native-provider fast path that resolves exact bundled static-catalog models before broad discovery only when provider/runtime metadata precedence remains safe, with redacted before/after dispatch timing proof.

## Reproduction Assessment

Yes at source level: current main still leaves native providers without bundled static-catalog fallback on the skip-discovery fast pass, then calls `ensureOpenClawModelsJson()`. I did not run a live Moonshot/Google Discord or TUI repro in this read-only review.

## Solution Assessment

No current fix is ready: a Moonshot-only resolver is too narrow after the Google reproduction, while the broad static-first PR raised provider-precedence compatibility concerns. The best path is a narrowed or capability-gated native fast path with live dispatch proof.
