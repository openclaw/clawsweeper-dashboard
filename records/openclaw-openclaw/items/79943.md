## What This Changes

The issue asks how user-configured `mcp.servers` tools can be exposed to Anthropic agent turns when OpenClaw 2026.5.7 stays on `route=native` and documented runtime-selection config is rejected.

## Best Possible Solution

Prove the Anthropic/Telegram path with a minimal stdio MCP server, then align schema/docs/runtime behavior so a documented runtime-policy shape exposes the generated MCP tool name in the actual provider request and tool inventory.

## Reproduction Assessment

## Solution Assessment

Unclear: switching off `route=native` is probably not the right fix because that log is provider HTTP routing inside the embedded runner. The maintainable fix is either schema/docs alignment for runtime policy or a runtime propagation fix proven by a live MCP tool appearing in the Anthropic turn.
