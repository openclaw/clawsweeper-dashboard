## Summary

Keep open: this is a fresh, detailed external bug report against documented MCP/runtime behavior, and current source does not give a safe close reason without a live current-main Anthropic plus Telegram reproduction.

## What This Changes

The reporter says OpenClaw 2026.5.7 launches a configured `mcp.servers` stdio server but does not expose its tool to Anthropic API-key agent turns, and the documented runtime-policy config shapes are rejected or ignored.

## Best Possible Solution

Run the minimal stdio MCP server scenario on current main with Anthropic API-key auth and Telegram, then fix the narrow failing layer: bundle-MCP propagation/tool-policy diagnostics if tools are still absent, or config/docs guidance if current main works with model-scoped runtime policy.

## Reproduction Assessment

No: the issue gives a detailed 2026.5.7 reproduction with logs, but I did not establish a high-confidence current-main live reproduction. Source inspection confirms the intended MCP and runtime-policy paths exist, while one provider-scoped config path remains plausibly confusing.

## Solution Assessment

Unclear: switching off `route=native` is probably not the right fix because current source treats PI as the fallback harness for unclaimed provider/model turns and the native Anthropic route is a provider transport. The maintainable fix depends on whether current main fails to materialize `bundle-mcp` tools or only needs corrected runtime-policy/config guidance.
