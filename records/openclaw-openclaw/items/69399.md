## Summary

Keep open. Current main still lacks the requested opt-in fallback notice config, and this PR is a plausible implementation path, but it is not merge-ready because the new opt-in tests do not pass the config through the queued-run path and the config help omits the cleared-notice behavior.

## What This Changes

The PR adds `agents.defaults.notifyOnModelFallback` to config types/schema/help/docs and changes `runReplyAgent` to prepend fallback and fallback-cleared notices outside verbose mode, with e2e coverage.

## Best Possible Solution

Land one canonical default-false fallback-notice implementation that validates the production queued-config path, documents both fallback and cleared notices, and includes real behavior proof for a normal-mode fallback reply.

## Reproduction Assessment

Yes, source-reproducible. Current main gates both user-visible fallback notices on `verboseEnabled`, and current e2e tests assert verbose-off suppression while lifecycle events still emit.

## Solution Assessment

No, not as currently patched. The opt-in direction is narrow, but the tests need to pass the opt-in through the queued config used by production and the config help/generated schema should describe both fallback and cleared notices.
