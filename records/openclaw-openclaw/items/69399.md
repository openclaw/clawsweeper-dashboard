## Summary

Keep this PR open. Current main still has verbose-only fallback notices and no fallback-notice config key, while the PR is an active implementation for the linked feature request; however, the proposed patch is not merge-ready because the new e2e cases do not actually pass the opt-in flag through the queued run config, and the user-facing feature lacks a changelog entry.

## What This Changes

The PR adds an `agents.defaults.notifyOnModelFallback` opt-in, config type/schema/help/docs updates, reply-runner guards for fallback and fallback-cleared notices outside verbose mode, and e2e coverage for the fallback notice lifecycle.

## Best Possible Solution

Land one canonical fallback-notice implementation that keeps default behavior unchanged, exposes a clearly named opt-in config, documents both fallback and fallback-cleared notices, includes a changelog entry, and has e2e coverage that passes the config through the same queued-run path used in production.

## Reproduction Assessment

Yes. Current main has source and e2e evidence for verbose-off suppression, and the PR test failure path is statically reproducible: `createMinimalRun` passes `config: {}` while the new normal-mode tests only set a runtime snapshot whose source does not match that config.

## Solution Assessment

No, not yet. The general opt-in approach is plausible and narrower than a new notification system, but the PR needs the queued-config test setup fixed, a changelog entry added, and maintainer agreement on the config scope before merge.
