## Summary

Keep open: current main still lacks the requested opt-in behavior, and this PR is a plausible narrow direction, but PR head still needs a queued-config test fix, consistent config help, contributor real behavior proof, and maintainer agreement on the new config surface.

## What This Changes

The PR adds `agents.defaults.notifyOnModelFallback`, widens fallback and fallback-cleared reply notices outside verbose mode when that option is true, and updates config schema/help, docs, generated config metadata, and e2e tests.

## Best Possible Solution

A default-false fallback-notice option should land only after maintainers accept the config surface and the code, docs, help text, and queued-config tests all describe the same fallback and cleared-notice behavior.

## Reproduction Assessment

Yes for the current behavior gap: source inspection shows current main emits fallback lifecycle events while reply-visible fallback notices are guarded by `verboseEnabled`, and no `notifyOnModelFallback` key exists. I did not run the e2e path because this is a read-only review.

## Solution Assessment

No, not as currently patched. The opt-in approach is narrow, but the queued production config path must be covered correctly and config help should match the cleared-notice behavior before this is the best maintainable fix.
