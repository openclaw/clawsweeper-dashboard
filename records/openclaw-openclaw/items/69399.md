## Summary

Keep open. The feature direction is plausible, but this PR is not merge-ready because the new notify-on-fallback tests do not pass the config into the queued run, real behavior proof is still missing, and the public config/notification shape overlaps related open fallback-notification work.

## What This Changes

The PR adds a default-off `agents.defaults.notifyOnModelFallback` option, widens fallback and fallback-cleared reply notices outside verbose mode when enabled, and updates config schema/help, docs, generated hashes, and e2e coverage.

## Best Possible Solution

Land one coordinated default-false fallback-notice implementation after the config path is tested correctly, fallback and cleared notices are documented consistently, real behavior proof is attached, and maintainers approve the config surface.

## Reproduction Assessment

Yes for the current behavior gap. Source inspection shows current main emits fallback lifecycle events but prepends fallback and cleared notice payloads only under `verboseEnabled`, and no independent fallback-notice config key exists.

## Solution Assessment

No, not as currently patched. The default-false opt-in direction is narrow, but the PR needs the queued-run config path fixed, help/generated descriptions aligned with cleared-notice behavior, real proof attached, and maintainer config-surface approval.
