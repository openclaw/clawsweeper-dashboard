## Summary

Keep this PR open: it targets a real Memory Wiki lint false positive in the right plugin-owned code path, but the patch needs a small correctness fix, a required changelog entry, and inspectable real behavior proof before merge.

## What This Changes

The PR updates Memory Wiki linting to resolve bridge-relative source links, ignore transcript reply directives, and add a regression test for those cases.

## Best Possible Solution

Keep the extension-local bridge-relative resolver, but make link normalization tolerant of raw Markdown targets, match only documented reply directives, add the changelog entry, and require inspectable CLI proof before merge.

## Reproduction Assessment

Yes from source inspection: current main only validates generated wiki path forms while bridge pages preserve original bridgeRelativePath values, and the wikilink extractor treats reply directive text as link targets. I did not run lint because this review is constrained to read-only inspection.

## Solution Assessment

No as written: resolving bridge-relative targets in the Memory Wiki plugin is the right fix boundary, but the implementation should not let raw imported Markdown links crash lint and it needs the repo-required release note and proof gate satisfied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard percent decoding for raw bridge links:** `extensions/memory-wiki/src/lint.ts:69-71`
  - body: Bridge source pages preserve raw Markdown links, and those can contain unescaped percent characters like `100% done.md`. `decodeURIComponent` throws `URIError` for that input, so this change can make `openclaw wiki lint` abort instead of warning on a bridge page with ordinary raw source text; decode defensively or fall back to the normalized raw target.
  - confidence: 0.89
- **[P3] Match only supported reply directives:** `extensions/memory-wiki/src/lint.ts:53-54`
  - body: This regex also treats `[[reply_to]]` as a directive because the suffix group is optional, but OpenClaw only documents `reply_to_current` and `reply_to:<id>`. That masks a real broken wikilink target instead of reporting it, so tighten the matcher to the documented forms.
  - confidence: 0.77
- **[P3] Add the required changelog entry:** `extensions/memory-wiki/src/lint.ts:80-82`
  - body: This is a user-facing fix to `openclaw wiki lint`, but the PR only changes Memory Wiki code and tests. Please add a single-line `CHANGELOG.md` Fixes entry so the shipped lint behavior is recorded.
  - confidence: 0.86
