## Summary

Keep open: current main still has the reported block-streaming Markdown defects, but this PR head remains incorrect because preserved separators are still trimmed during delivery, separator-inclusive chunks can exceed `maxChars`, the table integration test contradicts its own cap, and table detection misses markdown-it-valid table forms.

## What This Changes

The PR adds table-span detection to `EmbeddedBlockChunker`, preserves paragraph/newline separators in raw chunks, adds chunker/subscribe/table-span tests, and documents/changelogs the streaming behavior.

## Best Possible Solution

Repair or replace this PR so shared block streaming preserves boundary semantics through delivery, protects markdown-it-compatible pipe tables, honors channel caps, and covers both raw chunker and subscribe-level behavior.

## Reproduction Assessment

Yes, at source level. Current main drops paragraph separators in `EmbeddedBlockChunker`, trims delivered block replies in `emitBlockChunk`, and has no table-safe break path before channel rendering.

## Solution Assessment

No. The shared chunker/parser surface is the right place to fix this, but this PR still needs a narrow repair for delivery trimming, cap accounting, table fixture expectations, and markdown-it-compatible table recognition.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve separators through delivery:** `src/agents/pi-embedded-block-chunker.test.ts:142-143`
  - body: The new regression proves raw `EmbeddedBlockChunker` output, but production delivery still trims block replies before saving and invoking `onBlockReply`. A trailing newline preserved here can still disappear from visible streaming output, so the fix needs delivery-level coverage and a path that carries the boundary through `emitBlockChunk`.
  - confidence: 0.91
- **[P2] Count preserved separators against maxChars:** `src/agents/pi-embedded-block-chunker.ts:219-223`
  - body: This guard checks only the text before the paragraph separator, but the emitted slice includes `paragraphBreak.length` as well. A blank-line separator with spaces or extra newlines can therefore emit a chunk longer than `maxChars`, violating the documented channel cap behavior.
  - confidence: 0.9
- **[P2] Align the table fixture with maxChars:** `src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.streams-soft-chunks-paragraph-preference.test.ts:53`
  - body: This test sets `maxChars` to 25 but expects the full four-row table to be delivered as one block. Since the PR now has row/max fallback behavior for oversized tables, either raise the cap above the table length or assert the row-safe chunk sequence.
  - confidence: 0.91
- **[P2] Accept one- and two-dash table delimiters:** `src/markdown/table-spans.ts:52`
  - body: The delimiter regex requires at least three hyphens per cell, while markdown-it's table rule accepts one or more. Valid short-delimiter tables will not be recognized as table spans and can still split mid-table.
  - confidence: 0.86
- **[P3] Recognize one-column pipe tables:** `src/markdown/table-spans.ts:47-52`
  - body: After trimming edge pipes, a one-column table row has one cell, so the `cells.length >= 2` checks reject it. Those valid tables remain unprotected unless the row and delimiter helpers allow the one-cell case with a non-empty cell.
  - confidence: 0.78
