## Summary

Keep this PR open for a narrow repair. Current main still has the reported block-streaming Markdown boundary problems, but this PR is not merge-ready because the preserved separators are still trimmed before delivery, the new separator-preserving path can exceed the configured cap, and the table parser/test coverage misses valid table cases.

## What This Changes

The PR adds pipe-table span parsing, wires table-aware and separator-preserving breaks into `EmbeddedBlockChunker`, updates focused chunker/streaming tests, and adds streaming docs plus a changelog entry.

## Best Possible Solution

Land one shared block-streaming fix that preserves Markdown boundary semantics through the actual `subscribeEmbeddedPiSession` delivery surface, enforces channel caps, protects valid GFM pipe-table spans, and keeps the docs, changelog, and focused regression tests aligned with #42106, #66614, and #66605.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for stripped paragraph separators and missing table protection, and the PR-specific defects are reproducible by tracing the PR head through `emitBlockChunk`, the `maxChars: 25` table fixture, and the table delimiter/cell checks.

## Solution Assessment

No. The shared chunker is the right surface, but this branch needs a narrow repair so preserved separators survive delivery, separator-inclusive chunks respect caps, valid table forms are detected, and the integration test matches the row-safe cap behavior.

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Cover the delivery path that trims separators:** `src/agents/pi-embedded-block-chunker.test.ts:142-143`
  - body: The new regression only proves raw `EmbeddedBlockChunker` output. In the production streaming path, `emitBlockChunk` still calls `.trimEnd()` before saving `assistantTexts` and invoking `onBlockReply`, so a preserved `\n` or `\n\n` is removed before visible block delivery and concatenated updates can still collapse paragraphs. Add delivery-level coverage and carry the boundary through that path.
  - confidence: 0.91
- **[P2] Cap preserved separators before emitting:** `src/agents/pi-embedded-block-chunker.ts:219-223`
  - body: This guard compares only `paragraphBreak.index - start` to the cap, but the emitted slice extends through `paragraphBreak.index + paragraphBreak.length`. A blank-line separator with spaces or extra newlines can therefore produce a chunk longer than `maxChars`, violating the documented channel cap behavior.
  - confidence: 0.9
- **[P2] Make the table fixture fit the cap:** `src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.streams-soft-chunks-paragraph-preference.test.ts:53`
  - body: This test sets `maxChars` to 25 but expects the 61-character table to be delivered as one block reply. The PR chunker has explicit row/max fallback when the limit lands inside a table, so the fixture should either raise the cap above the table length or assert the row-safe chunk sequence.
  - confidence: 0.91
- **[P2] Accept short table delimiter cells:** `src/markdown/table-spans.ts:52`
  - body: This regex requires at least three hyphens per delimiter cell, so valid pipe tables using one or two hyphens, such as `| - | - |`, are not recognized and can still split mid-table. Relax the delimiter check and add coverage for the short form.
  - confidence: 0.82
- **[P3] Recognize one-column pipe tables:** `src/markdown/table-spans.ts:47-52`
  - body: After trimming edge pipes, a row like `| Name |` has one cell, so the `cells.length >= 2` checks reject valid one-column pipe tables. Those tables remain unprotected from mid-table splits unless the row and delimiter helpers allow the one-cell case with a non-empty cell.
  - confidence: 0.78
