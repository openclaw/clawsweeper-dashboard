## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cap preserved separators before emitting:** `src/agents/pi-embedded-block-chunker.ts:219-223`
  - body: The new flushOnParagraph branch checks only the text before the separator against paragraphLimit, then emits through paragraphBreak.index + paragraphBreak.length. A blank-line separator at the max boundary can therefore produce a chunk longer than maxChars, violating the documented channel cap clamp.
  - confidence: 0.9
- **[P2] Align the table integration fixture with the cap:** `src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.streams-soft-chunks-paragraph-preference.test.ts:53`
  - body: This test sets maxChars to 25 but expects the 61-character table to arrive as one onBlockReply. The PR's chunker falls back to row/max splits when a table exceeds the cap, so the fixture should either raise the cap above the table length or assert the capped row-safe sequence.
  - confidence: 0.91
- **[P2] Accept shorter table delimiter cells:** `src/markdown/table-spans.ts:52`
  - body: The delimiter regex requires at least three hyphens per cell, so compact valid delimiter rows such as | - | - | are not recognized and remain eligible for mid-table splitting. Relax the delimiter check and cover the short form in table-spans tests.
  - confidence: 0.82
- **[P3] Recognize one-column pipe tables:** `src/markdown/table-spans.ts:47-52`
  - body: After trimming edge pipes, one-column rows have one cell, but both row and delimiter helpers require at least two. Valid one-column pipe tables remain unprotected by the new table-safe chunking path.
  - confidence: 0.78
