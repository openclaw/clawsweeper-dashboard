## Summary

Keep open: current main still has the shared block-streaming boundary bug, and this PR is the active replacement path, but the head still has concrete merge blockers around delivery trimming, maxChars accounting, and table-parser/test coverage.

## What This Changes

The PR adds table-span detection to `EmbeddedBlockChunker`, preserves raw newline/paragraph separators in emitted chunks, and adds chunker/subscribe tests plus streaming docs and changelog text.

## Best Possible Solution

Repair and land or replace the shared block-streaming fix so separators survive subscribe delivery, pipe tables are protected using markdown-it-compatible rules, `maxChars` remains honored, and the subscribe-level table guard passes.

## Reproduction Assessment

Yes, at source level. Current main routes raw assistant text through `EmbeddedBlockChunker`, consumes paragraph separators before delivery, trims trailing block reply text, and has no table-safe break path.

## Solution Assessment

No. The shared chunker is the right surface, but this PR needs a narrow repair so preserved separators survive delivery, separator-inclusive chunks respect caps, and valid pipe-table forms are covered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve separators through delivery:** `src/agents/pi-embedded-block-chunker.test.ts:142-143`
  - body: The new regression only proves raw `EmbeddedBlockChunker` output, but production delivery still calls `.trimEnd()` in `emitBlockChunk` before saving and invoking `onBlockReply`. A preserved trailing newline can still be stripped before visible block delivery, so add delivery-level coverage and carry the boundary through that path.
  - confidence: 0.91
- **[P2] Count preserved separators against maxChars:** `src/agents/pi-embedded-block-chunker.ts:219-223`
  - body: This guard checks only `paragraphBreak.index - start`, while the emitted slice extends through `paragraphBreak.index + paragraphBreak.length`. A blank-line separator with spaces or extra newlines can therefore emit a chunk longer than `maxChars`, violating the documented channel cap behavior.
  - confidence: 0.9
- **[P2] Align the table fixture with the cap:** `src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.streams-soft-chunks-paragraph-preference.test.ts:53`
  - body: This test sets `maxChars` to 25 but expects the full 61-character four-row table to be delivered as one block reply. The PR chunker has explicit row/max fallback behavior, so the fixture should either raise the cap above the table length or assert the row-safe chunk sequence.
  - confidence: 0.91
- **[P2] Accept short table delimiters:** `src/markdown/table-spans.ts:52`
  - body: The delimiter regex requires at least three hyphens per cell, but markdown-it's table rule accepts one or more. Valid short delimiter rows such as `| - | - |` will not be recognized as table spans and can still split mid-table.
  - confidence: 0.84
- **[P3] Recognize one-column pipe tables:** `src/markdown/table-spans.ts:47-52`
  - body: After trimming edge pipes, a one-column table row has one cell, so the `cells.length >= 2` checks reject it. Those markdown-it-valid tables remain unprotected unless the row and delimiter helpers allow the one-cell case with a non-empty cell.
  - confidence: 0.78
