The PR adds table-span detection and separator-preserving break logic to EmbeddedBlockChunker, plus streaming tests, docs, and a changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve separators through delivery:** `src/agents/pi-embedded-block-chunker.test.ts:142-143`
  - body: The new regression proves raw EmbeddedBlockChunker output, but production delivery still calls .trimEnd() in emitBlockChunk before saving and invoking onBlockReply. A preserved trailing newline can still be stripped before visible block delivery, so add delivery-level coverage and carry the boundary through that path.
  - confidence: 0.91
- **[P2] Count preserved separators against maxChars:** `src/agents/pi-embedded-block-chunker.ts:219-223`
  - body: This guard checks only paragraphBreak.index - start, while the emitted slice extends through paragraphBreak.index + paragraphBreak.length. A blank-line separator with spaces or extra newlines can therefore emit a chunk longer than maxChars, violating the documented channel cap behavior.
  - confidence: 0.9
- **[P2] Align the table fixture with maxChars:** `src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.streams-soft-chunks-paragraph-preference.test.ts:53`
  - body: This test sets maxChars to 25 but expects the full four-row table to be delivered as one block reply. The PR chunker now has explicit row/max fallback behavior, so the fixture should either raise the cap above the table length or assert the row-safe chunk sequence.
  - confidence: 0.91
- **[P2] Accept one- and two-dash table delimiters:** `src/markdown/table-spans.ts:52`
  - body: The delimiter regex requires at least three hyphens per cell, but markdown-it's table rule accepts one or more. Valid short delimiter rows will not be recognized as table spans and can still split mid-table.
  - confidence: 0.84
- **[P3] Recognize one-column pipe tables:** `src/markdown/table-spans.ts:47-52`
  - body: After trimming edge pipes, a one-column table row has one cell, so the cells.length >= 2 checks reject it. Those tables remain unprotected unless the row and delimiter helpers allow the one-cell case with a non-empty cell.
  - confidence: 0.78
