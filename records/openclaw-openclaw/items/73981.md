## Summary

Keep open. Current main still has the underlying separator and table-splitting bugs, but this PR head has blocking correctness issues around delivery trimming, chunk caps, table fixture expectations, and table-span coverage.

## What This Changes

The PR adds pipe-table span detection and separator-preserving chunking to `EmbeddedBlockChunker`, with focused streaming tests, docs, and a changelog entry.

## Best Possible Solution

Land a repaired shared block-streaming fix that preserves Markdown separators through `subscribeEmbeddedPiSession`, protects valid pipe-table spans, respects channel caps, and keeps focused tests/docs aligned.

## Reproduction Assessment

Yes, at source level. Current main emits paragraph chunks before the separator, skips the separator afterward, and has no table-span safe-break path; I did not run tests during this read-only review.

## Solution Assessment

No. The shared chunker is the right surface, but this PR needs a narrow repair so preserved separators survive delivery, separator-inclusive chunks respect `maxChars`, and valid table forms are covered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Cover the delivery path that trims separators:** `src/agents/pi-embedded-block-chunker.test.ts:142-143`
  - body: The new regression proves raw `EmbeddedBlockChunker` output, but production delivery still calls `.trimEnd()` in `emitBlockChunk` before saving and invoking `onBlockReply`. A preserved trailing newline can still be stripped before visible block delivery, so add delivery-level coverage and carry the boundary through that path.
  - confidence: 0.91
- **[P2] Cap preserved separators before emitting:** `src/agents/pi-embedded-block-chunker.ts:219-223`
  - body: This guard checks only `paragraphBreak.index - start`, while the emitted slice extends through `paragraphBreak.index + paragraphBreak.length`. A blank-line separator with spaces or extra newlines can therefore emit a chunk longer than `maxChars`, violating the documented channel cap behavior.
  - confidence: 0.9
- **[P2] Make the table fixture match the cap:** `src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.streams-soft-chunks-paragraph-preference.test.ts:53`
  - body: This test sets `maxChars` to 25 but expects the full table to be delivered as one block reply. The PR chunker now has explicit row/max fallback behavior, so the fixture should either raise the cap above the table length or assert the row-safe chunk sequence.
  - confidence: 0.91
- **[P2] Accept short table delimiter cells:** `src/markdown/table-spans.ts:52`
  - body: The delimiter regex requires at least three hyphens per cell, so valid short delimiter cells such as `| :-: | :-: |` are not recognized as table spans and can still split mid-table. Relax the delimiter check and add coverage for the short form.
  - confidence: 0.82
- **[P3] Recognize one-column pipe tables:** `src/markdown/table-spans.ts:47-52`
  - body: After trimming edge pipes, a one-column table row has one cell, so the `cells.length >= 2` checks reject it. Those tables remain unprotected unless the row and delimiter helpers allow the one-cell case with a non-empty cell.
  - confidence: 0.78
