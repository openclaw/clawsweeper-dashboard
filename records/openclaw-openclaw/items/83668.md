## Summary

Current main already exposes role-bearing conversation history through persisted transcript messages, `chat.history`/`sessions_history`, SDK run output types, and plugin hook message contexts, so the requested provenance capability is covered on main.

## What This Changes

The issue requests explicit role/source metadata for conversation history messages so skills and integrations can distinguish user, assistant, and tool-originated entries.

## Best Possible Solution

Keep the current role-bearing transcript/history/hook contracts, and open a narrower follow-up only if a specific skill or plugin surface still drops these fields.

## Reproduction Assessment

No; this is a feature request rather than a reproducible bug. Source inspection on current main shows the requested role metadata already exists in persisted history, SDK output, tool history, and plugin hook message contexts.

## Solution Assessment

Yes; the current role-bearing message contract is the narrow maintainable solution for distinguishing user, assistant, and tool-originated history without adding a second provenance schema. A future change should target only a concrete surface that demonstrably loses those fields.
