## Summary

Keep open: current main and v2026.6.10 still use per-agent built-in memory SQLite indexes, and the linked shared-store PR is only a candidate implementation with unresolved storage, migration, session-boundary, and SDK-contract blockers.

## What This Changes

The issue requests a memory index design change so agents sharing the same workspace/source directories reuse one vector store instead of maintaining duplicate per-agent stores.

## Best Possible Solution

Design and land a first-class shared memory index model with canonical source-keyed storage, agent-scope filtering, watcher ownership, upgrade/rebuild behavior, and regression coverage.

## Reproduction Assessment

Yes, source-level: current main still resolves the built-in memory store path from `agentId` and caches managers by `agentId`, so identical workspaces are not source-keyed. I did not run a live multi-agent reindex because this review is read-only.

## Solution Assessment

Unclear as a final design: source-keyed indexing targets the real duplication, but the final implementation needs maintainer decisions on storage identity, migration/rebuild behavior, agent privacy, and watcher lifecycle.
