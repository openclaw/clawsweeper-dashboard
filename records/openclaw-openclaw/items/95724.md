## Summary

Keep open: current main and v2026.6.9 still use per-agent builtin memory SQLite indexes, and the requested source-directory keyed shared index remains a real but product-sensitive memory-core design change.

## What This Changes

Request to share builtin memory vector indexes across agents that resolve the same workspace/source set instead of storing one index per agent database.

## Best Possible Solution

Design a first-class memory-core shared-index model keyed by canonical source set, with explicit agent-scope filtering, watcher ownership, upgrade/rebuild behavior, tests, and docs.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.9 resolve builtin memory storage through agentId and cache managers by agentId. I did not run a live multi-agent reindex because this was a read-only review.

Unclear as an implementation direction: the source-key idea targets the real waste, but it needs maintainer sign-off on storage layout, migration/rebuild behavior, privacy scope, and watcher lifecycle before code should be queued.
