## Summary

Keep open: current main and the latest release still key the built-in memory index by agent, and the prior shared-memory PR is closed unmerged rather than a viable canonical fix. The report is source-backed, but the fix changes durable memory ownership, per-agent isolation, watcher lifecycle, and upgrade behavior, so it needs maintainer architecture review before implementation.

## What This Changes

The issue requests source-directory keyed shared memory vector indexes so same-workspace agents do not duplicate embeddings, chunks, storage, or file watchers.

## Best Possible Solution

Design a first-class shared memory corpus/index owner in memory-core or the memory host SDK, with explicit per-agent visibility metadata, upgrade/doctor handling, and focused tests for same-workspace sharing, diverged sources, provider keys, and existing per-agent stores.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 compute the memory database path from `agentId`, and the docs describe per-agent memory index storage. I did not run a live two-agent indexing scenario in this read-only review.

## Solution Assessment

Unclear as an implementation direction: source-keyed sharing addresses the waste, but the best design needs maintainer choice around per-agent isolation, provider-key separation, extra path semantics, and migration rather than a narrow automatic patch.
