## Summary

Keep open for maintainer review: this PR appears to fix the isolated-cron feedback loop without breaking normal A2A ping-pong, with sufficient live-output proof and focused regression coverage. The remaining concern is merge-time message-delivery risk because the change intentionally changes which session receives follow-up A2A turns.

## What This Changes

The branch adds an isolated-cron requester gate in `sessions_send` so fire-and-forget A2A announces pass `maxPingPongTurns: 0` for cron runs, plus regression tests for cron, normal, and cron-like requester keys.

## Best Possible Solution

Land one canonical isolated-cron `sessions_send` fix that suppresses requester ping-pong for canonical cron runs while preserving target announce and normal requester ping-pong; this PR currently matches that shape with broader proof than the alternate open candidate.

## Reproduction Assessment

Yes, from source and supplied proof rather than a fresh local run. Current main passes positive `maxPingPongTurns` into a callee that steps the requester, and the PR body includes before/after Node output showing the cron requester step disappears while normal ping-pong remains.

## Solution Assessment

Yes. Forcing `maxPingPongTurns` to zero only for isolated cron requesters is narrower than skipping the whole A2A announce flow or gating on `timeoutSeconds`, and it keeps normal requester behavior intact.
