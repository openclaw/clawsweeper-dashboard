## Summary

Keep open. The PR targets a real prompt-customization gap, but the patch is not mergeable because it exposes new config without schema/runtime wiring, lacks real behavior proof, and needs maintainer/security judgment before allowing config to remove Safety prompt guidance.

## What This Changes

The PR adds `agents.defaults.systemPrompt.identityLine` and `disabledSections` to the config type, direct `buildAgentSystemPrompt` support/tests, and a global OAuth test mock.

## Best Possible Solution

Define one maintainer-approved prompt customization contract, then implement schema, generated docs/help, runtime prompt assembly, precedence, tests, and explicit guardrails for Safety end to end.

## Reproduction Assessment

Yes for the PR blockers. Source inspection is enough: the patch adds a config type and direct builder parameters, but the strict schema and production prompt callers do not accept or use the new nested defaults.

## Solution Assessment

No. The PR is not the best mergeable shape yet; the safer path is a maintainer-approved prompt customization API with complete schema/runtime/docs coverage and a specific decision on whether Safety can be disabled.
