## Summary

Keep open: the PR proposes a real prompt-customization feature, but it is not mergeable because it exposes a new config shape only in TypeScript/direct prompt-builder tests while current main’s strict schema and configured runtime prompt paths do not wire that shape through. The ability to disable Safety also needs explicit maintainer API/security judgment and real runtime proof before merge.

## What This Changes

Adds optional `agents.defaults.systemPrompt.identityLine` and `disabledSections` fields, direct `buildAgentSystemPrompt` section gating tests, and a global OAuth test mock.

## Best Possible Solution

Define one maintainer-approved prompt customization contract, then implement schema, docs/config metadata, runtime resolution across CLI/embedded/auto-reply/compaction paths, tests, and explicit Safety guardrails end to end.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the strict runtime schema lacks the new key and the configured prompt resolver/runtime callers do not read the proposed defaults. The requested prompt customization itself is a feature rather than a current-main bug reproduction.

## Solution Assessment

No. Direct renderer parameters and isolated builder tests are not the best mergeable shape; the safer path is a maintainer-approved config/API contract with complete schema, runtime, docs, tests, and Safety policy coverage.
