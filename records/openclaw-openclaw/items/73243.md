## Summary

Keep open; current main still has the stuck-session recovery gap, but this PR cannot merge as-is because the abort completion path can hide failed recovery and emit misleading abort telemetry, with docs/changelog still missing.

## What This Changes

The branch adds a configurable diagnostics stuck-session abort threshold, session.aborted diagnostics/OTel handling, recovery return values, config schema/reload updates, and focused tests.

## Best Possible Solution

Land a corrected version that retries when recovery returns false, emits session.aborted only for the still-current successfully recovered stuck state, and documents the new config/telemetry before closing the linked bug.

## Reproduction Assessment

Yes. Current main can be reproduced by holding a diagnostic session in processing past diagnostics.stuckSessionWarnMs: source shows it emits session attention and conservative recovery only, while the PR has fake-timer coverage for the intended abort path but lacks the false-recovery and natural-recovery race cases.

## Solution Assessment

No, not as-is. The overall direction is the narrow core fix for #71127, but the heartbeat must treat recovered === false as retryable and must gate abort telemetry/state reset on the latest state still matching the original stuck run.
