## Summary

Keep open. Current main still lacks the requested stuck-session abort threshold, so this PR is not obsolete, but the current PR head still has two blocking correctness issues: it can mark a session idle after recovery reports no actual release, and it emits `session.aborted` before confirming the stuck state is still current. It also needs public docs and changelog coverage for the new config/diagnostic surface.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session recovery, `session.aborted` diagnostics/OTel handling, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a corrected version of this PR, or a narrow replacement, that aborts/reclaims genuinely stuck sessions after the configured threshold, retries instead of going idle when recovery returns false, emits abort telemetry only for the still-current stuck state, and documents the new config and telemetry before the linked bug is closed.

## Reproduction Assessment

Yes. Current main can be checked by putting a diagnostic session in `processing` past `diagnostics.stuckSessionWarnMs`: the heartbeat emits `session.stuck` and calls conservative recovery, but there is no abort threshold or public `stuckSessionAbortMs` config. The PR adds a happy-path fake-timer test, but still needs regression coverage for `recovered === false` and the natural-recovery telemetry race.

## Solution Assessment

No, not as-is. The direction is the narrow core fix for the linked stuck-session reports, but the implementation must gate idle transition and `session.aborted` emission on a confirmed current state and successful recovery, then update docs/changelog.
