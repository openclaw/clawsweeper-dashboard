## Summary

Keep open. The PR targets a real stuck-session recovery gap, but the current head is not ready because the abort path can mark a session idle without confirmed recovery, can emit misleading abort telemetry for a stale recovery, and omits required docs/changelog updates for the new public config and telemetry surface.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session abort/recovery, `session.aborted` diagnostics/OTel handling, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a corrected, rebased implementation that adds a configurable active stuck-session abort threshold on top of current main's session-attention recovery model, only marks sessions idle after confirmed current recovery, emits accurate telemetry, and documents the new config and OTel events.

## Reproduction Assessment

Yes. Current-main source inspection is enough to reproduce the remaining gap: a `processing` session past `diagnostics.stuckSessionWarnMs` can be classified and conservatively recovered, but there is no `diagnostics.stuckSessionAbortMs` threshold for active stuck-run aborts; the PR adds fake-timer coverage but still needs failure and race assertions.

## Solution Assessment

No, not as-is. The direction is the narrow maintainable fix for the linked stuck-session reports, but the PR must gate idle transition and abort telemetry on a current, successful recovery and align with current main's newer recovery runtime and docs.

Overall correctness: patch is incorrect

- **[P2] Keep processing when stuck recovery returns false:** `src/logging/diagnostic.ts:772-776`
  - body: `recoverStuckSession` can resolve `false` for duplicate/in-flight recovery, active reply work, active lane work, or a no-op. This handler still marks the diagnostic state `idle` and drops `queueDepth`, which can stop future retry while the underlying work remains blocked; only transition to idle after recovery is confirmed.
  - confidence: 0.9

If the old run recovers naturally and new work starts before the recovery promise resolves, state is protected by the guard but a misleading `session.aborted` event is still emitted for current healthy work.
  - confidence: 0.88

  - body: This adds public config `diagnostics.stuckSessionAbortMs` plus a new `session.aborted` diagnostics/OTel surface, but the PR does not update the user-facing configuration, queue/agent-loop, OTel docs, or changelog. Add those before landing the user-facing behavior change.
  - confidence: 0.86
