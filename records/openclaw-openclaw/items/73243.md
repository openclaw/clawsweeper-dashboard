## Summary

Keep open. The PR targets a real current-main gap, but the latest head still has correctness blockers in the abort completion path and is missing the public docs/changelog updates required for the new config and telemetry surface.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session recovery, `session.aborted` diagnostics/OTel handling, config schema/reload updates, and focused diagnostics tests.

## Best Possible Solution

Land a corrected version that aborts genuinely timed-out stuck sessions, only marks recovery complete after the still-current state is confirmed and recovery succeeded, retries false/failure cases, and documents the config and telemetry surface.

## Reproduction Assessment

Yes. Current main can be checked by putting a diagnostic session in `processing` past `diagnostics.stuckSessionWarnMs`: the heartbeat classifies/logs liveness and only attempts conservative stale-lane recovery, with no abort threshold. The PR blockers are also reproducible with mocks returning `false` and with a delayed recovery after natural session recovery.

## Solution Assessment

No, not as-is. The direction is the narrow maintainable fix for the linked stuck-session bug, but the PR needs to gate telemetry/state reset on current state plus successful recovery and update public docs/changelog before merge.
