## Summary

Keep open. Current main still lacks the requested stuck-session abort threshold, and the PR is the right general direction for the linked stuck-session reports, but the current head has two concrete correctness bugs: it can mark diagnostics idle after an unsuccessful recovery and it can emit abort telemetry before confirming the stale stuck state is still current. The new public config/telemetry surface also needs docs and changelog coverage before merge.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session recovery, `session.aborted` diagnostics/OTel handling, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a corrected version of this PR, or a narrow replacement, that aborts or reclaims genuinely stuck sessions after a configured threshold, retries when recovery returns false, emits abort telemetry only for the still-current stuck state, and documents the new config and telemetry surface before the linked bug is closed.

## Reproduction Assessment

Yes. Current main can be verified by leaving a diagnostic session in `processing` past `diagnostics.stuckSessionWarnMs`: the heartbeat emits `session.stuck` and conservative recovery runs, but there is no `stuckSessionAbortMs` config or active stuck-run abort threshold. The PR's added fake-timer tests cover the happy path but not the false-recovery or stale-telemetry races.

## Solution Assessment

No, not as-is. The requested direction is the narrow maintainable fix, but the implementation should transition to idle and emit `session.aborted` only after confirming the same stuck state is current and recovery actually succeeded; failed or no-op recovery should clear the abort flag for retry.
