## Summary

Keep open. The linked stuck-session bug is real, but this branch is not ready: current main now has overlapping recovery semantics, and the PR still has source-visible P2 bugs that can mark active sessions idle and emit misleading abort telemetry.

## What This Changes

The branch adds `diagnostics.stuckSessionAbortMs`, stuck-timeout heartbeat recovery, `session.aborted` diagnostics/OTel telemetry, config schema/reload updates, and focused tests.

## Best Possible Solution

Converge on one recovery contract: current main's active-work/stale-bookkeeping checks plus any maintainer-approved public abort threshold, documented with regression coverage.

## Reproduction Assessment

Yes for the PR defects from source inspection: the recovery runtime returns `false` for retry/no-op cases, while the heartbeat still marks the diagnostic state idle and emits telemetry before confirming the stale state is current.

## Solution Assessment

No, not as-is. The direction is plausible, but it must be rebased onto current main's recovery contract and must gate state reset and abort telemetry on a true recovery of the still-current stuck run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep processing state when recovery returns false:** `src/logging/diagnostic.ts:918-922`
  - body: `recoverStuckDiagnosticSession` returns `false` for active reply work, active lane tasks, duplicate/no-op recovery, and failures, but this code still marks the diagnostic state idle after any resolved value. That can stop later heartbeat retries while the actual session or lane remains blocked; only transition to idle after `recovered === true` and clear the abort flag for retry otherwise.
  - confidence: 0.91
- **[P2] Emit abort telemetry after the stale-state guard:** `src/logging/diagnostic.ts:909`
  - body: `logSessionAbortRequested` runs before checking that the same stuck processing state is still current. If the old run naturally recovers and new work starts before recovery resolves, the state guard prevents clobbering but a misleading `session.aborted` event has already been emitted.
  - confidence: 0.88
- **[P3] Document the public abort threshold and event:** `src/config/types.base.ts:293-294`
  - body: This adds public `diagnostics.stuckSessionAbortMs` config and `session.aborted` diagnostics/OTel behavior, but the user-facing Gateway config and OTel docs still omit them and the changelog edit is unrelated. Add docs and a real changelog fix entry before landing.
  - confidence: 0.84
