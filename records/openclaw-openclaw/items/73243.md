## Summary

Keep open. Current main still lacks the requested abort threshold, so the PR is not obsolete, but the proposed patch still has correctness blockers in the stuck-session recovery path and is missing required docs/changelog coverage for the new public config key.

## What This Changes

This PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-driven stuck-session abort/recovery, `session.aborted` diagnostics and OTel recording, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a corrected version that treats the abort threshold as an actual recovery threshold for abortable embedded and reply runs, leaves diagnostic state in `processing` for retry when no recovery happened, emits abort telemetry only after confirming the same stale state is still current, and updates the public docs plus changelog for the new config key.

## Reproduction Assessment

Yes. Current main can be verified by a fake-timer heartbeat with a session in `processing` past `diagnostics.stuckSessionWarnMs`: it emits `session.stuck` and calls conservative recovery, but there is no `stuckSessionAbortMs` config or abort-threshold path. The PR adds coverage for the happy abort path, but still needs coverage for active reply runs, false recovery, and stale-state telemetry suppression.

## Solution Assessment

No, not as-is. The direction is narrow and relevant, but the patch should abort active reply work at the timeout threshold, only mark diagnostic state idle after a real recovery, move `session.aborted` emission behind the stale-state guard, and publish the config/docs/changelog surface together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Abort reply work at the timeout threshold:** `src/logging/diagnostic-stuck-session-recovery.runtime.ts:80`
  - body: When `resolveActiveEmbeddedRunSessionId` finds an active reply operation but there is no embedded handle, this branch still returns `false` even for `stuck-timeout` recovery. That leaves reply-only stuck runs un-aborted, so the new threshold does not recover one of the paths the PR claims to fix.
  - confidence: 0.86
- **[P2] Only mark sessions idle after real recovery:** `src/logging/diagnostic.ts:772`
  - body: `recoverStuckSession` can resolve `false` when no run or lane was actually released. This line still marks the diagnostic state idle and clears the queue depth, which stops retries while the underlying run/lane may remain blocked.
  - confidence: 0.9
- **[P2] Move abort telemetry behind the state guard:** `src/logging/diagnostic.ts:763`
  - body: `session.aborted` is emitted before checking whether the same stuck state is still current. If the old run naturally finishes and new work starts before recovery resolves, the guard protects state but the telemetry still reports an abort for healthy new work.
  - confidence: 0.88
- **[P3] Document the new public config key:** `src/config/types.base.ts:282`
  - body: This adds public `diagnostics.stuckSessionAbortMs`, but the public configuration reference and changelog still describe only warning behavior. Add the docs and changelog entry before landing the user-facing fix.
  - confidence: 0.86
