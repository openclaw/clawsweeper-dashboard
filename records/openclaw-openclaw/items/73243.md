## Summary

Keep open. The stuck-session recovery problem is valid and this PR is working in the right area, but the current branch can still hide active stuck work by marking diagnostics idle after a false/no-op recovery and can emit misleading abort telemetry after a natural-recovery race; the new public config/event also needs docs and a real changelog entry.

## What This Changes

The PR adds a diagnostics.stuckSessionAbortMs threshold, attempts stuck-session abort/recovery, emits session.aborted diagnostics/OTel telemetry, updates config schema/reload wiring, and adds focused tests.

## Best Possible Solution

Land a corrected stuck-session abort path that only marks idle and emits abort telemetry after confirmed recovery of the same stuck state, retries false/no-op recoveries, and documents the config/event with a real changelog entry.

## Reproduction Assessment

Yes. A fake-timer diagnostic heartbeat scenario can hold a session in processing past diagnostics.stuckSessionWarnMs on current main to show warning/conservative recovery only, and the PR already adds a focused abort-threshold test path that exposes the intended behavior.

## Solution Assessment

No, not as-is. The direction is the narrow maintainable fix for the linked stuck-session bug, but the heartbeat must treat recovered === false as retryable and gate telemetry/state reset on the latest state still matching the original stuck run.
