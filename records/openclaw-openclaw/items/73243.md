## Summary

Keep this PR open: it addresses a real stuck-session recovery gap, but the latest head still has two source-level correctness blockers around false recovery and stale telemetry, plus missing public docs/changelog for the new config and diagnostics event.

## What This Changes

The PR adds a diagnostics-controlled stuck-session abort threshold, recovery wiring, session.aborted diagnostics/OTel telemetry, config schema support, reload behavior, and regression tests.

## Best Possible Solution

Land a corrected version that fits current-main liveness recovery semantics, retries when recovery returns false, emits abort telemetry only for the still-current recovered run, and documents the public config, OTel signal, and changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the proposed abort threshold, and PR-head source shows the false-recovery and stale-telemetry races without needing a live stuck gateway.

## Solution Assessment

No, not as-is. The overall direction is a narrow fix for the linked stuck-session bug, but the implementation must not mark idle or emit session.aborted unless recovery actually succeeded for the same still-current processing state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep processing state when recovery returns false:** `src/logging/diagnostic.ts:1091-1095`
  - body: `recoverStuckDiagnosticSession` returns `false` for duplicate/no-op recovery, active reply work, active lane tasks, no active work, and caught failures. This handler still marks the diagnostic state idle and clears queue depth after any resolved value, which can stop retries while the stuck work remains blocked; only transition to idle when `recovered === true`, and clear the abort flag for retry otherwise.
  - confidence: 0.91
- **[P2] Emit abort telemetry after the stale-state guard:** `src/logging/diagnostic.ts:1082`
  - body: `logSessionAbortRequested` runs before checking that this is still the same stuck processing state. If the old run naturally recovers and new work starts before recovery resolves, the guard protects state but a misleading `session.aborted` event has already been emitted for current healthy work.
  - confidence: 0.88
- **[P3] Document the public abort config and event:** `src/config/types.base.ts:312-313`
  - body: This adds public `diagnostics.stuckSessionAbortMs` config and `session.aborted` diagnostics/OTel behavior, but the gateway config docs, OTel docs, and changelog are not updated. Generated schema/help text is useful, but it does not cover the user-facing docs and release-note surface for this behavior change.
  - confidence: 0.84
