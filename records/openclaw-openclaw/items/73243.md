## Summary

Keep this PR open: the branch is aimed at the right diagnostics recovery path for the linked stuck-session bug, but the latest head still has concrete runtime and public-surface blockers before it is safe to merge.

## What This Changes

Adds `diagnostics.stuckSessionAbortMs`, stuck-session abort telemetry, recovery return values, config schema/reload updates, diagnostics-otel handling, and focused stuck-session tests.

## Best Possible Solution

Land a corrected version that treats `recovered === false` as retryable, gates abort telemetry on the still-current stuck run, and documents the new config, OTel signal, and changelog entry before closing the linked bug.

## Reproduction Assessment

Yes, from source inspection. Current main has stuck-session warning and limited recovery paths but no configurable abort threshold or `session.aborted` transition, while the PR adds fake-timer coverage for the intended abort path.

## Solution Assessment

No, not as-is. The overall design is the narrow core fix, but the implementation must not mark the state idle or emit abort telemetry unless recovery actually succeeded for the same still-current stuck run.

## Review Findings

Full review comments:

- **[P2] Keep processing state when recovery returns false:** `src/logging/diagnostic.ts:1091-1095`
  - body: `recoverStuckDiagnosticSession` returns `false` for active reply work, active lane tasks, duplicate/no-op recovery, no active work, and caught failures. This branch still marks the diagnostic state idle and clears queue depth after any resolved value, which can stop retries while the real stuck work remains blocked; only transition to idle when recovery returns `true`, and clear the abort flag for retry otherwise.
  - confidence: 0.91

If the old run naturally recovers and new work starts before recovery resolves, the guard protects state but a misleading `session.aborted` event has already been emitted for current healthy work.
  - confidence: 0.88

Generated schema/help text is useful but does not satisfy the user-facing docs and changelog surface for this behavior change.
