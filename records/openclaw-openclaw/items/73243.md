## Summary

Keep open: current main still lacks the requested abort threshold, but this PR has correctness issues in the abort completion path and still needs public docs/changelog updates before it can safely close the linked stuck-session reports.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session recovery, `session.aborted` diagnostics/OTel handling, config schema/reload updates, and focused tests.

## Best Possible Solution

Land a corrected, rebased version that aborts only genuinely timed-out stuck work, retries when recovery does not succeed, emits accurate abort telemetry, and documents the new public config/OTel surface.

## Reproduction Assessment

Yes. Current main can be exercised with a diagnostic session left in `processing` beyond `diagnostics.stuckSessionWarnMs`; it emits stuck-session diagnostics and conservative recovery, but has no abort threshold or `session.aborted` event.

## Solution Assessment

No, not as-is. The direction is the narrow maintainable fix, but the abort completion handler must gate telemetry and idle transition on the still-current state and `recovered === true`, with docs and changelog added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep processing state when abort recovery fails:** `src/logging/diagnostic.ts:772`
  - body: `recoverStuckSession` can resolve `false` when active work or a lane was not actually recovered, but this handler still marks the session `idle` and clears queue depth. That can hide the still-blocked work and prevent a later abort retry; keep the state processing and clear `stuckAbortRequestedAt` unless recovery returned `true`.
  - confidence: 0.9
- **[P2] Emit abort telemetry after the stale-state guard:** `src/logging/diagnostic.ts:763`
  - body: `logSessionAbortRequested` runs before checking whether this is still the same processing state. If the old run naturally recovers and new work starts before the promise resolves, state is protected by the guard but a misleading `session.aborted` event is already emitted for healthy current work.
  - confidence: 0.88
- **[P3] Document the new abort config and telemetry:** `src/config/types.base.ts:282`
  - body: This adds public `diagnostics.stuckSessionAbortMs` config and a `session.aborted` diagnostics/OTel event, but the public configuration/OTel docs and changelog still describe warning-only behavior or omit the new surface. Add those updates before landing the user-facing fix.
  - confidence: 0.86
