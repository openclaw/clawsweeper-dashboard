## Summary

Keep open. The PR addresses a real current-main gap and is on the right code path, but the current head still has correctness blockers: a false/no-op recovery can be recorded as idle recovery, and abort telemetry can fire for a stale recovery race before the state guard runs.

## What This Changes

The branch adds `diagnostics.stuckSessionAbortMs`, abort recovery wiring, `session.aborted` diagnostics/OTel handling, config schema/reload updates, and focused diagnostics tests.

## Best Possible Solution

Land a corrected version that retries when recovery returns false, emits `session.aborted` only after confirming the same stuck state was actually recovered, and documents the public config/telemetry with a real changelog entry.

## Reproduction Assessment

Yes. Current main can be reproduced or checked by holding a diagnostic session in `processing` past `diagnostics.stuckSessionWarnMs`: source shows warning/classification plus conservative recovery only, while the PR has fake-timer coverage for the happy abort path but not the false-recovery and natural-recovery race paths.

## Solution Assessment

No, not as-is. The overall direction is the narrow maintainable fix for the linked stuck-session bug, but the caller must distinguish successful recovery from retry/no-op and gate telemetry/state reset on the latest state still matching the original stuck run.
