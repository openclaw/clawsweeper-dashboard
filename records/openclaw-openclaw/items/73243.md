## Summary

Keep open. Current main still lacks the requested active stuck-session abort threshold, and this PR is the right general direction, but the current head still has correctness and documentation blockers before it should land.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session recovery, `session.aborted` diagnostics/OTel handling, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a corrected version of this PR, or a narrow replacement, that aborts or reclaims genuinely stuck sessions after a configured threshold, retries on no-op recovery, emits abort telemetry only for the still-current stuck state, and documents the new public surface.

## Reproduction Assessment

Yes. Current main can be verified by putting a diagnostic session in `processing` past `diagnostics.stuckSessionWarnMs`: the heartbeat emits stuck-session diagnostics and conservative recovery, but there is no public abort threshold or active stuck-run abort path. The PR adds happy-path fake-timer coverage but still needs false-recovery and stale-telemetry race coverage.

## Solution Assessment

No, not as-is. The direction is the narrow maintainable fix for the linked reports, but the implementation should transition to idle and emit `session.aborted` only after confirming the same stuck state is current and recovery actually succeeded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep processing state when recovery fails:** `src/logging/diagnostic.ts:772-773`
  - body: `recoverStuckSession` can resolve `false` when active reply/lane work remains or no lane/run was released, but this branch still marks diagnostics `idle` and clears `queueDepth`. That stops later abort retries while the real work can remain blocked; only transition idle after `recovered === true` and clear the abort flag on false.
  - confidence: 0.9
- **[P2] Emit abort telemetry after the stale-state guard:** `src/logging/diagnostic.ts:763`
  - body: `logSessionAbortRequested` runs before checking that the same stuck processing state is still current. If the old run naturally recovers and new work starts before recovery resolves, state is protected by the guard but a misleading `session.aborted` event has already been emitted for healthy current work.
  - confidence: 0.88
- **[P3] Document the abort config and telemetry:** `src/config/types.base.ts:282`
  - body: This adds public `diagnostics.stuckSessionAbortMs` config and `session.aborted` diagnostics/OTel behavior, but the PR does not update the configuration, queue/agent-loop, OTel docs, or changelog. Add those user-facing updates before landing the behavior change.
  - confidence: 0.86
