## Summary

Keep open. Current main has partial mitigations around outbound hook cancellation, commentary-phase suppression, and isolated-cron stale acknowledgement retry, but it still lacks the requested runtime-owned fake-progress classifier and execution-state-backed pre-send gate across user-facing delivery paths.

## What This Changes

The issue requests a runtime pre-send policy layer that classifies outbound assistant messages, blocks unsupported acknowledgements and commentary leakage, suppresses repeated fake-progress messages, enforces progress-debt timeouts, and adds tests/docs for those behaviors.

## Best Possible Solution

Define a first-class outbound policy boundary and durable execution-state contract, then implement fake-progress enforcement with typed ack/progress/blocker/completion classification, repeat suppression, progress-debt faulting, structured metrics, and coverage across shared outbound, channel adapters, agents, subagents, ACP, and cron.

## Reproduction Assessment

Unclear for a high-confidence live end-to-end reproduction in this read-only pass. The issue body gives concrete direct-chat and Telegram-topic reports, and current source inspection shows a plausible remaining path because final user-visible acknowledgement text can still pass through delivery unless an existing hook/plugin cancels it.

## Solution Assessment

Yes directionally: a runtime-enforced pre-send policy backed by durable execution state is the right class of solution for this failure mode. The exact implementation is not yet a safe narrow repair because maintainers need to choose the shared outbound boundary, state source, and plugin/core contract first.
