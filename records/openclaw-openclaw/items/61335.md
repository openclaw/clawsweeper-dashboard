## Summary

Keep open: the DST freeze root cause is still present on current main, but this PR needs changes because it predates current token-bucket-first selected-hour filtering and still lacks contributor-owned real Control UI proof.

## What This Changes

The PR changes Control UI Usage hour slicing and selected-hour filtering to use monotonic next-hour-boundary stepping across DST transitions and adds node-side regression tests.

## Best Possible Solution

Adapt the PR to replace only the Usage hour-slice walker and the legacy selected-hour span fallback with monotonic next-hour stepping, preserve precise token-bucket selected-hour behavior, add current-main DST regression coverage, and include redacted real Control UI proof.

## Reproduction Assessment

Yes for the root cause: current main still uses the DST-sensitive `setToHourEnd` loop, and a focused Node probe shows the local fallback hour can move the cursor backward. I did not run the full browser Usage-page freeze scenario.

## Solution Assessment

No. Monotonic next-hour stepping is the right fix direction, but this PR head is not the best merge shape because it bypasses the token-bucket selected-hour helper introduced on current main; the safer fix is to make that helper's span fallback and the shared hour-slice walker DST-safe.
