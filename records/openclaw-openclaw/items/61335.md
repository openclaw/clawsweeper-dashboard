## Summary

Keep open: this PR targets a valid source-reproducible Control UI Usage DST freeze, but the current patch bypasses the selected-hour token-bucket contract on current main and still lacks contributor-provided real Control UI proof.

## What This Changes

The PR changes Control UI Usage hour slicing and selected-hour filtering to use monotonic next-hour boundary stepping across DST transitions and adds node-side regression tests.

## Best Possible Solution

Apply monotonic next-hour-boundary stepping inside the existing Usage hour-slice walker and selected-hour fallback while preserving precise token-bucket-first filtering and adding focused DST coverage.

## Reproduction Assessment

Yes, source-level. Current main still uses the backward-moving `setToHourEnd` path, and a Sydney DST fallback probe shows that operation can resolve one millisecond before the cursor; I did not run a full browser freeze repro.

## Solution Assessment

No. Monotonic next-hour stepping is the right direction, but the PR should preserve `sessionTouchesSelectedHours` token-bucket-first behavior and make only the span fallback and hour-slice walker DST-safe.

## Review Findings
