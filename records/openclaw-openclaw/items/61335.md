## Summary

Keep open: the DST freeze is reproducible and the patch direction is useful, but the latest branch is still incorrect against current main because selected-hour rendering would bypass precise token-bucket filtering; it also needs the required changelog entry before merge.

## What This Changes

The PR changes Control UI Usage hour slicing and selected-hour filtering to use monotonic DST-safe next-hour stepping and adds node-side DST regression coverage.

## Best Possible Solution

Keep the monotonic hour-boundary algorithm, but wire it into current `forEachSessionHourSlice` and `sessionSpanTouchesSelectedHours` while leaving `sessionTouchesSelectedHours` as the selected-hour entry point, with focused DST tests and a changelog entry.

## Reproduction Assessment

Yes. Current main still has the `setToHourEnd` cursor path, and a Sydney DST fallback Node check shows the old local hour-end calculation can resolve before the cursor.

## Solution Assessment

No. Monotonic next-hour stepping is the right fix, but this PR applies it through a span-only selected-hour helper instead of preserving current main's token-bucket-first helper and updating only its fallback walker.
