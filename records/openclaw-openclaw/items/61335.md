## Summary

Keep open. The DST freeze remains valid on current main, but the PR needs an update before merge because current main has since added precise token-bucket selected-hour filtering that the PR's span-only helper would bypass.

## What This Changes

The PR changes Control UI Usage hour slicing and selected-hour filtering to use monotonic next-hour boundary stepping across DST transitions and adds node-side regression tests.

## Best Possible Solution

Update the Usage metrics implementation on current main so `forEachSessionHourSlice` and the fallback path inside `sessionTouchesSelectedHours` share a monotonic next-hour-boundary helper, while preserving precise token-bucket filtering and adding focused DST fallback/spring-forward coverage near the existing Usage tests.

## Reproduction Assessment

Yes. Current main still has the `setToHourEnd` plus `cursor = nextMs + 1` fallback path, and the prior ClawSweeper read-only Sydney DST check showed that this can resolve an hour end earlier than the cursor.

## Solution Assessment

No. The monotonic next-hour-boundary direction is the right fix, but this PR should be adapted to current main instead of replacing the newer token-bucket-first selected-hour helper with a span-only helper.
