## Summary

Keep open: current main still has the DST-sensitive Usage hour walker, but this PR is not merge-ready because the head bypasses current token-bucket selected-hour semantics and still lacks real Control UI proof.

## What This Changes

The PR changes Usage metrics hour slicing and selected-hour filtering to use monotonic next-hour boundary stepping across DST transitions, with node-side regression tests.

## Best Possible Solution

Keep the monotonic next-hour-boundary fix, but wire it into the hour-slice walker and the selected-hour fallback while preserving token-bucket-first filtering and adding focused DST regression coverage plus real Control UI proof.

## Reproduction Assessment

Yes, source-level. Current main still uses `setToHourEnd` plus `cursor = nextMs + 1`, and a Sydney fallback probe shows the old local hour-end operation can resolve one millisecond before the cursor.

## Solution Assessment

No. Monotonic next-hour stepping is the right underlying fix, but this PR should preserve current `sessionTouchesSelectedHours` token-bucket semantics and only make the fallback span walker DST-safe.
