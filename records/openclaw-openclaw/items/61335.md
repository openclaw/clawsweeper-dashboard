## Summary

Keep this PR open: the DST freeze is still real on current main, but the branch needs to be adapted because it would bypass current main's precise token-bucket selected-hour filtering.

## What This Changes

The PR changes Control UI Usage hour slicing and hour-filtered rendering to use monotonic DST-safe next-hour stepping and adds node-side DST regression coverage.

## Best Possible Solution

Apply monotonic next-hour-boundary stepping inside current main's `forEachSessionHourSlice` and the `sessionSpanTouchesSelectedHours` fallback while keeping `sessionTouchesSelectedHours` token-bucket-first behavior and focused DST tests.

## Reproduction Assessment

Yes. Current main still has the `setToHourEnd` cursor path, and a Sydney DST fallback Node check shows the old local hour-end calculation can resolve before the cursor.

## Solution Assessment

No. The monotonic boundary helper is the right direction, but the selected-hour integration should be applied inside the current helper rather than replacing it with a span-only path.
