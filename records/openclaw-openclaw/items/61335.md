## Summary

Keep open. The DST cursor bug is real and the PR has a useful direction, but the current branch is not mergeable as-is because it drops token-bucket-first selected-hour filtering, is conflicting against current main, and still lacks contributor real-behavior proof.

## What This Changes

This PR changes Control UI Usage hour aggregation and hour filtering to step by monotonic next-hour boundaries across DST transitions and adds node-side DST regression tests.

## Best Possible Solution

Keep the monotonic DST-safe boundary helper, but integrate it into current main's hour-slice and selected-hour fallback walkers while preserving token-bucket-first filtering and focused DST regression coverage.

## Reproduction Assessment

Yes, source-level. Current main still advances Usage hour slices with `setToHourEnd(...); cursor = nextMs + 1`, and a Sydney DST fallback probe shows the hour end can resolve one millisecond before the cursor.

## Solution Assessment

No. Monotonic next-hour stepping is the right underlying fix, but this PR should preserve `sessionTouchesSelectedHours` token-bucket semantics and make only the fallback span walker DST-safe.
