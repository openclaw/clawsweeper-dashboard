## Summary

Keep this PR open: the reported Usage DST freeze is valid, but the branch is not ready because it still drops current main's precise token-bucket selected-hour semantics and lacks after-fix real Control UI proof.

## What This Changes

This PR replaces Usage hour slicing and selected-hour filtering with monotonic next-hour boundary stepping and adds node-side DST regression tests.

## Best Possible Solution

Keep the monotonic next-hour-boundary fix, but apply it to the hour-slice walker and the `sessionSpanTouchesSelectedHours` fallback while preserving token-bucket-first filtering and adding focused DST regression coverage.

## Reproduction Assessment

Yes, source-level. Current main still uses `setToHourEnd` plus `cursor = nextMs + 1`, and a Sydney fallback probe shows that boundary can resolve one millisecond before the cursor.

## Solution Assessment

No. Monotonic boundary stepping is the right underlying fix, but this PR should preserve `sessionTouchesSelectedHours` token-bucket semantics and make only the fallback span walker DST-safe.
