## Summary

Keep this open. Current main has a useful Telegram-specific supersession fence, but the core queue/final-delivery contract still queues newer same-session input for a later turn while preserving the active run's completed final payload for delivery on generic paths.

## What This Changes

The issue requests a pre-send same-session queue check so completed replies can be suppressed or deferred when newer inbound messages are pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending follow-ups immediately before eligible visible final delivery, suppresses or defers stale final payloads, and carries enough suppressed reply/tool context into the next collected turn.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with collect/followup, a newer same-session inbound message during an active run is queued, while finalization schedules the drain and returns the active run's final payload unchanged.

## Solution Assessment

No, not as a simple one-line queue-depth check. The maintainable solution needs explicit opt-in semantics for side effects, streaming previews, routed delivery, hook ordering, and how suppressed context is presented to the next turn.
