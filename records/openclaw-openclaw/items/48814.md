## Summary

Keep this issue open. Current main still treats newer same-session input as a queued follow-up while preserving the active run's completed final payload for delivery, and the later Telegram stale-reply work only fences authorized abort supersession rather than general pending-message suppression.

## What This Changes

Request an opt-in delivery-time queue policy that suppresses or defers a completed final reply when newer same-session inbound messages are already pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending follow-ups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should define streaming preview cleanup, side-effect disclosure, routed delivery behavior, plugin hook order, and whether content-aware classification is a later feature.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with collect or followup, a newer same-session inbound message during an active run takes the enqueue-followup path while the active run still returns finalizeWithFollowup(finalPayload), preserving the completed final payload for delivery.

## Solution Assessment

No, not as a one-line blanket queue-size check. The maintainable solution should be an opt-in queue/delivery policy with explicit semantics for side effects, streaming previews, routed delivery, hook ordering, and later content-aware classification.
