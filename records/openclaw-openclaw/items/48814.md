## What This Changes

Request to suppress completed-but-stale foreground replies when a newer same-session user message is pending before channel delivery.

## Best Possible Solution

Keep the shared foreground freshness guard on main and open narrower follow-ups only for content-aware routing or background-branch cancellation semantics beyond visible stale final suppression.

## Reproduction Assessment

Yes, at source/regression level: the merged regression test deterministically starts an older same-session dispatch, starts a newer one, releases the older final, and asserts the older delivery is cancelled. I did not run a live Telegram or WhatsApp race in this read-only review.

## Solution Assessment

Yes. A shared buffered-dispatch freshness guard is the narrow maintainable solution because it covers the common foreground delivery path instead of adding channel-specific or queue-mode-specific stale checks.
