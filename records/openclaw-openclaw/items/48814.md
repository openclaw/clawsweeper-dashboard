## Summary

Keep this issue open. Current main still documents and implements collect/followup as a later queued turn, while the active run’s completed final payload is preserved for delivery; the newer inbound message is not used as a pre-send stale-reply suppression signal. The discussion also raises real design constraints around side effects, streaming previews, routed delivery, and async branch supersession, so this needs maintainer product/API design before a safe repair PR.

## What This Changes

Request an opt-in pre-send same-session queue check that suppresses or defers a completed reply when newer inbound messages are already pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending follow-ups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define streaming preview cleanup, side-effect disclosure, routed delivery behavior, plugin hook ordering, and whether content-aware classification is a later layer.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with `collect` or `followup`, a newer same-session inbound message during an active run takes the `enqueue-followup` path, while the active run still returns `finalizeWithFollowup(finalPayloads)`, preserving the completed final payload for delivery.

## Solution Assessment

No, not as a one-line blanket queue-size check. The requested direction is valid, but the maintainable solution should be an opt-in queue/delivery policy with explicit semantics for side effects, streaming previews, routed delivery, and hook ordering.
