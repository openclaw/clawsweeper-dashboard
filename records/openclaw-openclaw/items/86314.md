## Summary

Keep open: current main still does not carry `channels.openclaw-weixin.enabled=true` through the gateway startup plugin plan, and two open draft PRs already own the startup-planner repair with closing syntax.

## What This Changes

The issue reports that an installed external Weixin channel can complete QR login but is missing from the gateway runtime registry, so `channels.start` returns `invalid channels.start channel` and messages are not delivered.

## Best Possible Solution

Land one canonical startup-planner fix that treats enabled-only external channel config as startup intent while preserving external plugin trust and disablement boundaries.

## Reproduction Assessment

Yes, source-reproducible: current startup planning still derives startup channel ids through a helper that ignores `{ enabled: true }` alone, and `channels.start` still rejects channels absent from the active registry. I did not run a live WSL2 Weixin setup in this read-only review.

## Solution Assessment

Yes: the best fix direction is a generic gateway startup-planner repair, not a Weixin-specific special case. The implementation should preserve the external-plugin trust boundary while making enabled-only channel config reach the runtime load path.
