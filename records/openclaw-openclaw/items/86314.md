## Label Justifications

- `P1`: The issue describes a real channel workflow where an installed messaging channel cannot start after login, blocking users from receiving Weixin messages.
- `impact:message-loss`: If the Gateway never starts the Weixin channel account, inbound Weixin messages cannot be delivered to OpenClaw.

## Summary

Keep open: current main and v2026.6.6 still do not make enabled-only external channel config participate in Gateway startup planning, and the remaining fix is already owned by open draft PRs that explicitly close this issue.

## What This Changes

The issue reports that `@tencent-weixin/openclaw-weixin` can complete QR login but is not registered in the Gateway runtime on WSL2, so `channels.start` rejects `openclaw-weixin` as an invalid channel.

## Best Possible Solution

Land the canonical startup-planner fix that treats enabled-only external channel config as startup intent without widening untrusted workspace plugin activation, then close this issue after the merged proof is on main.

## Reproduction Assessment

Yes, this is source-reproducible: current startup planning still derives configured channel ids through a helper that ignores `{ enabled: true }` alone, which matches the reported `channels.start` invalid-channel path. I did not run a live WSL2 Weixin setup in this read-only review.

## Solution Assessment

Yes, the best fix direction is the linked startup-planner repair, not a Weixin-specific special case. The implementation should align Gateway startup intent with explicit channel config while preserving the workspace-plugin trust boundary.
