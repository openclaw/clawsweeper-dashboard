## Summary

Keep open: current main and v2026.6.8 still do not treat `channels.openclaw-weixin.enabled=true` alone as gateway startup intent, and two open closing PRs already own the startup-planner repair.

## What This Changes

The issue reports that an installed and enabled external Weixin channel plugin can complete QR login but fail `channels.start` because the running gateway registry has not loaded the plugin.

## Best Possible Solution

Land one canonical startup-planner fix that treats explicit enabled-only external channel config as startup intent without widening untrusted workspace plugin activation, then close this issue against the merged PR.

## Reproduction Assessment

Yes, source-reproducible: current startup planning still derives configured channel ids through a helper that ignores `{ enabled: true }` alone, while `channels.start` still fails when the active registry cannot resolve the plugin channel. I did not run a live WSL2 Weixin setup in this read-only review.

## Solution Assessment

Yes: the best fix direction is the linked startup-planner repair, not a Weixin-specific special case or the merged entry-only partial fix. The implementation should align Gateway startup intent with explicit channel config while preserving the external-plugin trust boundary.
