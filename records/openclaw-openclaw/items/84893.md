Keep open: current main still has the source-visible legacy `openai-codex` OAuth sidecar gap on the read-only embedded auth paths, and the related fix PR is open rather than landed. ## What This Changes

The issue reports that upgrading from 2026.5.12 to 2026.5.19 makes Codex unusable on a headless VPS via both the legacy `openai-codex/gpt-5.5` route and the newer `codex/gpt-5.5` route.

## Best Possible Solution

Land or split a focused auth-store fix that rehydrates legacy `openai-codex` OAuth sidecars in read-only runtime loaders, then re-test the 2026.5.12-to-current headless upgrade path and track any remaining Cloudflare 403 as a narrower follow-up.

## Reproduction Assessment

Yes, with source-level confidence for the auth-binding half: a sidecar-backed `openai-codex` OAuth profile can load without token material through current read-only runtime helpers, and PI credential conversion then drops the credential. The Cloudflare leg remains environment-dependent and was not live-reproduced here.

## Solution Assessment

Yes for the auth-binding root cause: the maintainable fix belongs in the shared auth-store loader boundary, not in user re-auth guidance or route-specific config workarounds. The Cloudflare behavior should be re-tested after that fix rather than assumed resolved.
