Keep open: current main and the latest release still leave the sidecar-backed `openai-codex` OAuth path unresolved in the read-only/secrets-runtime loader used by embedded runs, and the likely auth fix remains an open unmerged PR. ## What This Changes

The issue reports that upgrading from OpenClaw 2026.5.12 to 2026.5.19 makes Codex unusable on a headless VPS through both the legacy `openai-codex/gpt-5.5` route and the newer `codex/gpt-5.5` route.

## Best Possible Solution

Land or split a focused fix that rehydrates legacy `openai-codex` OAuth sidecars in the read-only/secrets-runtime loaders, then verify the 2026.5.12-to-current headless upgrade path and track any remaining Cloudflare 403 as a narrower follow-up.

## Reproduction Assessment

Yes, with source-level confidence: a sidecar-backed `openai-codex` OAuth profile can load without token material through the current read-only/secrets-runtime helpers, and PI credential conversion then drops the provider credential. I did not execute the live VPS/OAuth path, and the `codex/gpt-*` Cloudflare leg remains environment-dependent.

## Solution Assessment

Yes for the auth-binding root cause: the maintainable fix belongs in the shared auth-store loader boundary, not in route-specific workarounds or user re-auth instructions. The Cloudflare behavior should be verified after that fix rather than treated as solved by configuration changes alone.
