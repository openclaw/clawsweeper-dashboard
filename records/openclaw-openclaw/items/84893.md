## Summary

Keep open: current main still has the source-visible auth gap reported for upgraded legacy OpenAI Codex OAuth sidecars, and the promising related fix remains an open draft PR. The separate headless `codex/gpt-*` Cloudflare 403 path is not proven resolved by current main either.

## What This Changes

Investigate a 2026.5.19 regression where upgraded headless VPS Codex setups fail via legacy `openai-codex` OAuth binding and a newer `codex` route Cloudflare 403.

## Best Possible Solution

Land or split a focused fix that rehydrates legacy `openai-codex` OAuth sidecars in the read-only runtime loaders, then verify the 2026.5.12-to-current headless upgrade path and file a narrower follow-up only if the `codex/gpt-*` Cloudflare 403 still reproduces.

## Reproduction Assessment

Yes for the auth-binding half: current source shows sidecar-backed `openai-codex` OAuth profiles can load without token material through the embedded read-only helpers and then be dropped during PI credential conversion. The Cloudflare 403 leg is environment-dependent and was not live-reproduced in this review.

## Solution Assessment

Yes for the auth-binding root cause: the maintainable fix belongs in the shared auth-store loader boundary, not in route-specific config workarounds or user re-auth instructions. The Cloudflare behavior should be verified after that auth fix lands.
