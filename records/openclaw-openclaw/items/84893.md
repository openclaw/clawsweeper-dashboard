## Summary

Keep open: this is a fresh, well-instrumented upgrade regression, and current main still leaves legacy sidecar-backed OpenAI Codex OAuth unresolved on the read-only/secrets-runtime auth paths implicated by the report. The related open fix PR is promising but unmerged, and the separate `codex/gpt-*` Cloudflare 403 path still needs live headless proof after the auth fix lands.

## What This Changes

Upgrading from 2026.5.12 to 2026.5.19 should keep Codex usable on a headless VPS through the existing `openai-codex` OAuth profile or another supported Codex route.

## Best Possible Solution

Land or split a focused fix that rehydrates legacy `openai-codex` OAuth sidecars in the read-only/secrets-runtime loaders, then verify the 2026.5.12-to-current headless upgrade path and file a narrower follow-up only if the `codex/gpt-*` Cloudflare 403 still reproduces.

## Reproduction Assessment

Yes, with source-level confidence: a sidecar-backed `openai-codex` OAuth profile can load without token material through the current read-only/secrets-runtime helpers, and PI credential conversion then drops the provider credential. I did not execute the live VPS/OAuth turn; the `codex/gpt-*` Cloudflare leg remains environment-dependent.

## Solution Assessment

Yes for the auth-binding root cause: the maintainable fix belongs in the shared auth-store loader boundary, not in route-specific workarounds or user re-auth instructions. The `codex/gpt-*` Cloudflare behavior should be verified after that fix rather than used as the only resolution path.
