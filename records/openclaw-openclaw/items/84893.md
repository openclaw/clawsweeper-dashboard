## Summary

Keep open: current main and the shipped 2026.5.19 release still contain the read-only auth-store path that can load legacy `openai-codex` OAuth sidecars without token material, and the likely fix is still an open draft PR. The separate `codex/gpt-*` Cloudflare 403 leg remains environment-dependent and should be re-tested after the auth fix lands.

## What This Changes

The issue reports that upgrading a headless VPS from OpenClaw 2026.5.12 to 2026.5.19 makes Codex unusable through both the legacy `openai-codex/gpt-5.5` OAuth route and the newer `codex/gpt-5.5` route.

## Best Possible Solution

Land or split a focused auth-store fix that rehydrates legacy `openai-codex` OAuth sidecars in read-only runtime loaders, then verify the 2026.5.12-to-current headless upgrade path and track any remaining Cloudflare 403 separately.

## Reproduction Assessment

Yes, with source-level confidence: an `openai-codex` OAuth profile that only has a legacy `oauthRef` can flow through read-only runtime loaders without token material, then be dropped during PI credential conversion. I did not run the live VPS/OAuth request, and the Cloudflare leg remains environment-dependent.

## Solution Assessment

Yes for the auth-binding half: the linked auth-store sidecar rehydration direction is the narrow maintainable fix, not route-specific config workarounds or user re-auth guidance. The Cloudflare behavior should be verified after that fix lands before deciding whether more transport work is needed.
