## What This Changes

The PR catches malformed frontmatter only during vault-wide page summary reads, records a per-page frontmatter error, and teaches memory-wiki lint to report that page as `invalid-frontmatter`.

## Best Possible Solution

Land the narrow page-summary/lint fix after adding redacted real behavior proof that `wiki_apply` and `wiki_lint` complete while the bad page is reported as invalid frontmatter.

## Reproduction Assessment

Yes, source-reproducible: current main parses frontmatter with `YAML.parse`, compile stops on the first summary error, and both affected tools call compile. I did not execute the repro locally because this review had to keep the checkout read-only and dependencies are not installed.

## Solution Assessment

Yes, this looks like the best fix shape: tolerate malformed YAML only in vault-wide summaries and lint reporting, while leaving direct edit-target parsing strict so existing metadata is not silently dropped on writes. The broader alternative in https://github.com/openclaw/openclaw/pull/96130 is riskier because it catches parsing inside the shared parser used by mutation paths.
