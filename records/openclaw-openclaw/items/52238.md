## Label Justifications

- `P1`: The issue reports live Feishu group/topic messages being routed to a direct or web/main session, breaking replies for real users.

## Summary

Keep open: current main and v2026.6.10 still classify the reported Feishu `oc_...` event with `chat_type: "p2p"` as non-group, while current p2p direct/thread behavior makes the proposed broad `oc_` or root/thread heuristic unsafe without a verified Feishu discriminator.

## What This Changes

The issue asks Feishu topic-group messages delivered as `p2p` to route as group/topic messages instead of direct web/main sessions.

## Best Possible Solution

Keep this as the canonical Feishu inbound group/direct classification bug and land a Feishu-local shared classifier based on verified topic/OpenChat metadata with regressions for true p2p direct and p2p thread cases.

## Reproduction Assessment

Yes, source-reproducible: a reported Feishu event with `chat_type: "p2p"` reaches current main with `isGroup === false` and follows the direct route. I did not run a live Feishu/Lark replay in this read-only review.

## Solution Assessment

No, the originally suggested broad `root_id`/`thread_id` or `oc_` prefix-only classifier is not the best fix. The safer solution needs a verified Feishu topic/OpenChat discriminator plus tests proving true p2p direct and p2p thread cases remain direct.
