## Summary

Keep open. Current main still builds the announce rewrite turn from child completion metadata plus a delivery instruction, with no explicit bounded requester conversation context in the internal event contract or delivery handoff; the related account-routing fixes do not address this context-grounding bug.

## What This Changes

The issue requests that subagent and cron completion announce replies be grounded in the parent/requester session context so the generated user-facing update stays relevant to the active conversation.

## Best Possible Solution

Add an explicit bounded requester-context handoff for announce rewrites, keep child output marked as untrusted data, preserve the existing delivery/idempotency routing behavior, and document/test the contract for subagent, isolated cron, active steered, queued, and dormant direct-agent paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: build an announce flow and observe that the trigger is derived from child task_completion fields plus the rewrite instruction, then forwarded unchanged by active and direct delivery paths. I did not run live Feishu or Telegram checks because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, the requested direction is the right fix boundary, but it should not be raw transcript injection. The safer implementation is a bounded sanitized requester summary or equivalent requester-session context handoff with regression tests proving the rewrite turn is grounded without leaking or bloating context.
