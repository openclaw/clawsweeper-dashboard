## Summary

Keep open. Current main still builds the announce rewrite turn from child completion result/status/stats plus a reply instruction, and the active/direct delivery paths forward that trigger without an explicit bounded requester conversation handoff. The Feishu cron report is concrete, the later Telegram stale-session note is useful adjacent routing evidence, and the issue remains actionable rather than implemented, duplicate, or stale.

## What This Changes

The issue requests that sub-agent and cron announce completion rewrites be grounded in the parent/requester conversation context so completion replies do not become out of context.

## Best Possible Solution

Add an explicit bounded requester-context handoff for announce rewrite turns, while preserving the child result as untrusted data and keeping existing idempotency, direct delivery, queue, fallback, nested-subagent, and dormant-session behavior intact. The contract should define what requester context is available for subagent, cron, active steered, and direct-agent paths, with docs and regression tests covering the boundary.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: construct a subagent or cron completion announce and observe that the trigger is generated from child task_completion fields plus the rewrite instruction, then forwarded unchanged by active and direct delivery paths.

## Solution Assessment

Yes, a narrow announce-handoff repair is the right direction. The safer implementation is not raw last-N transcript injection; it should pass a bounded, sanitized requester summary or an existing requester-session context reference at delivery time.
