## Summary

Keep open. Current main still builds the announce rewrite turn from child completion metadata, result text, stats, and a rewrite instruction, then forwards that same trigger through active-session and direct-delivery paths without an explicit bounded requester-conversation handoff. The related #74913 direction is useful for the follow-up “agent forgot what it sent” half, but it does not fully resolve initial rewrite grounding for this issue.

## What This Changes

Ensure subagent and cron announce rewrite turns have enough bounded parent/requester context to produce completion replies grounded in the active conversation.

## Best Possible Solution

Add an explicit, bounded requester-context handoff for announce rewrite turns while preserving the child result as untrusted data and keeping existing idempotency, queue/direct fallback, nested-subagent, dormant-session, and cron descendant behavior intact. The contract should define what requester context is available for subagent, cron, active steered, and direct-agent paths, with docs and regression tests covering the boundary.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: construct a subagent or cron completion announce and observe that the trigger is generated from task_completion fields plus the rewrite instruction, then forwarded unchanged by active and direct delivery paths.

## Solution Assessment

Yes, a narrow announce-handoff repair is the best direction. The safer fix is not raw last-N transcript injection; it should pass a bounded, sanitized requester summary or existing requester-session context reference, while #74913 can separately handle delivered-message transcript mirroring.
