## Summary

Keep open. Current main still builds the announce rewrite turn from the child completion event and forwards that same trigger through active, queued, and direct delivery paths; I found no requester conversation summary or bounded context handoff in the event type, gateway schema, docs, or regression tests. The Feishu cron report is concrete, and the later Telegram stale-session comment is useful adjacent evidence, but the context-grounding bug remains actionable rather than closeable.

## What This Changes

Request a fix so subagent and cron completion announce rewrites are grounded in the parent or requester conversation context instead of only the child task result.

## Best Possible Solution

Add an explicit bounded requester-context handoff for announce rewrite turns, preserving child results as untrusted data and keeping idempotency, direct fallback, queued delivery, nested subagent, cron, and dormant-session behavior intact. The contract should define what context is available for subagent, cron, active steered, direct-agent, and ACP/plain-prompt paths, with docs and regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: runSubagentAnnounceFlow creates a task_completion event from child completion fields, buildAnnounceSteerMessage renders only that event, and delivery forwards the same trigger unchanged. I did not run live channel tests because the review was required to keep the checkout read-only.

## Solution Assessment

No, not as the raw last-N-message injection suggested in the issue. The maintainable fix should pass a bounded, sanitized requester summary or an explicit requester-context reference at delivery time, then prove the rewrite turn is grounded without copying unbounded transcript text.
