## Summary

Keep open. Current main still builds the announce rewrite turn from child completion fields plus a rewrite instruction, and both active-session steering and direct delivery forward that same trigger without an explicit bounded requester-context handoff. The Feishu cron report is concrete, and the Telegram stale-session comment is useful adjacent routing evidence, but the context-grounding bug remains actionable rather than implemented or stale.

## What This Changes

Ensure subagent and cron completion announce replies are grounded in the parent/requester session context so user-facing updates do not drift out of the active conversation.

## Best Possible Solution

Add an explicit bounded requester-context handoff for announce rewrite turns, preserving the child result as untrusted data and keeping existing idempotency, direct delivery, queue, fallback, nested-subagent, and dormant-session behavior intact. The contract should define what requester context is available for subagent, cron, active steered, and direct-agent paths, with docs and regression tests covering the boundary.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: construct a subagent or cron completion announce and observe that the trigger is generated from child task_completion fields plus the rewrite instruction, then forwarded unchanged by active and direct delivery paths.

## Solution Assessment

Yes, a narrow announce-handoff repair is the best direction. The safer implementation is not raw last-N transcript injection; it should pass a bounded, sanitized requester summary or an existing requester-session context reference at delivery time, with tests proving the rewrite turn is grounded without leaking excessive transcript content.
