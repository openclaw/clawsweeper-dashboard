## Summary

Keep open. Current main has adjacent routing, display, and transcript hygiene fixes, but the announce rewrite path still builds the requester handoff from child completion data plus a rewrite instruction, without an explicit bounded requester-context handoff; related items such as https://github.com/openclaw/openclaw/pull/74913 and https://github.com/openclaw/openclaw/pull/76767 cover nearby transcript or Telegram supersession slices, not this grounding bug.

## What This Changes

The issue requests that subagent and cron announce handoffs preserve enough requester conversation context so parent sessions do not rewrite child completion results out of context.

## Best Possible Solution

Add a typed, sanitized, size-bounded requester-context and turn-freshness handoff to announce rewrite turns, with regression coverage for active, queued, direct, cron, and nested requester paths.

## Reproduction Assessment

Yes, source-level. Construct a subagent or cron task_completion announce on current main and inspect that triggerMessage/internalEvents contain child result fields plus the rewrite instruction, without a bounded requester-context field; live channel proof was not run.

## Solution Assessment

Yes. The narrow maintainable fix is an explicit bounded requester-context summary or typed session-context reference in the announce handoff; raw transcript splicing or relying on unrelated Telegram/cron fixes would leave this bug class exposed.
