## Summary

Keep open. Current main still builds the subagent/cron announce rewrite turn from child completion data plus a rewrite instruction and forwards that same trigger through active-session and direct delivery paths without an explicit bounded requester-conversation handoff; the recent #74913 discussion is useful but only covers the delivered-message mirroring/follow-up half, not the original context-grounding problem.

## What This Changes

The issue requests that sub-agent or cron announce replies be grounded in the parent/requester conversation context instead of being generated from the child result alone.

## Best Possible Solution

Add an explicit, bounded requester-context handoff for announce rewrite turns, with the child result still treated as untrusted data and with delivery routing, idempotency, nested-subagent, dormant-session, active-session, and cron descendant behavior preserved. #74913 can handle delivered-message mirroring separately, but this path still needs the rewrite turn to be grounded before the user-facing completion reply is generated.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: construct a subagent or cron completion announce and observe that the trigger is generated from task_completion fields plus the rewrite instruction, then forwarded unchanged by both active-session and direct delivery paths.

## Solution Assessment

Yes, the narrow announce-handoff direction is the right fix, but the suggested raw last-N transcript injection should be tightened. The maintainable version is a bounded, sanitized requester summary or explicit requester-session context reference with regression tests proving the rewrite turn is grounded without copying excessive transcript content.
