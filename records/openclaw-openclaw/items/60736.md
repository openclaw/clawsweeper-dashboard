## Summary

Keep open. Current main and the latest shipped release only cover a partial Google Vertex path: native routing works for authorized_user ADC with GOOGLE_CLOUD_LOCATION, but the requested service-account/metadata ADC, CLOUD_ML_REGION/default location behavior, and complete config/onboarding path remain unfinished; open implementation work at https://github.com/openclaw/openclaw/pull/60860 and https://github.com/openclaw/openclaw/pull/83971 still targets this cluster, with useful production proof from a related commenter.

## What This Changes

The issue requests complete Google Vertex AI support in the Google provider, including OAuth/ADC handling, region-aware endpoint routing, mode distinction, and onboarding/config support.

## Best Possible Solution

Land one Google-plugin-owned Vertex implementation that preserves native endpoint routing, supports standard ADC modes, resolves project/location with documented fallbacks, accepts native config, wires setup/onboarding, and includes focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes, source-level. With GOOGLE_CLOUD_PROJECT set but no GOOGLE_CLOUD_LOCATION, current main throws instead of using CLOUD_ML_REGION or a default; with a service_account ADC file, the runtime reader and provider gate reject the credentials before native Vertex bearer auth can run.

## Solution Assessment

No, current main is only a partial solution. The maintainable path is to review, repair, land, or supersede the linked Google Vertex PR work with one coordinated provider-owned implementation rather than close this issue now.
