## Summary

Keep open: this PR addresses a source-backed Telegram TTS ordering regression and includes sufficient live Telegram proof, but the patch introduces a blocking hook/transcript correctness issue for successful voice sends. The protected `maintainer` label also rules out cleanup close.

## What This Changes

The PR finalizes already-streamed Telegram text in place when delayed TTS/media arrives, sends the media as a durable follow-up, preserves voice fallback/buttons, and adds tests plus a changelog entry.

## Best Possible Solution

Land a patched version that keeps streamed Telegram text in place, sends late audio/media as a follow-up, preserves buttons, and exposes `spokenText` only when fallback text is actually sent.

## Reproduction Assessment

Yes, source-backed: current main bypasses preview finalization for media-bearing final payloads and clears the unfinalized stream, matching the linked Telegram TTS timing report. I did not run a live current-main repro in this read-only review.

## Solution Assessment

No, not yet: the late-media split is the right narrow direction, but the proposed `spokenText` handling should remain hidden unless Telegram voice delivery actually falls back to text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep hidden spokenText out of successful voice sends:** `extensions/telegram/src/bot/delivery.replies.ts:844-845`
  - body: `spokenText` is only the fallback copy for the media-only voice follow-up, but this initializes `contentForSentHook` from it before `deliverMediaReply` knows whether `sendVoice` failed. When voice delivery succeeds, the PR now passes that hidden text to `message_sent` hooks and `transcriptMirror`; in the late-media path the preview text was already mirrored, so the session transcript can duplicate the answer. Start with visible `reply.text` only and assign `visibleFallbackText` after fallback text is actually sent.
  - confidence: 0.86
