## What This Changes

The PR marks the running Gateway restart prompt in doctor repair as requiring interactive confirmation and adds regression tests for non-interactive running-service skip and stopped-service start behavior.

## Best Possible Solution

Merge the narrow prompt-policy fix once maintainers accept that unattended doctor repair skips already-running Gateway restarts and explicit gateway commands or interactive doctor own that lifecycle action.

## Reproduction Assessment

Yes. Source inspection shows current main can reach `service.restart()` for a running Gateway after a failed health/probe path because `doctor --fix --non-interactive` auto-approves runtime repair prompts unless they require interactive confirmation; I did not run a live failing current-main Gateway reproduction in this read-only review.

## Solution Assessment

Yes. The PR changes the narrow running-service restart prompt that creates the SIGTERM risk while leaving stopped-service start repair on the existing non-interactive repair path; the broader alternative of making all service lifecycle prompts interactive-only was too broad for stopped-service recovery.
