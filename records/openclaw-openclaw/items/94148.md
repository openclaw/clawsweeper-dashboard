## Summary

Keep open: this is the focused candidate fix for the linked running-Gateway doctor repair bug, and current main plus the latest release still have the unguarded non-interactive restart path. The patch looks correct, but it intentionally changes unattended Gateway repair semantics, so maintainer acceptance of that compatibility tradeoff is still needed before merge.

## What This Changes

Adds interactive-only confirmation to the running Gateway restart prompt in doctor daemon repair and adds regression tests that non-interactive repair skips running restarts while still starting stopped services.

## Best Possible Solution

Land the narrow prompt-policy change only after maintainers accept that running-Gateway restarts require an interactive or explicit lifecycle action while stopped-service recovery stays automatic.

## Reproduction Assessment

Yes. Source inspection shows current main can reach `service.restart()` for a running Gateway after failed health/probe handling because non-interactive repair auto-approves runtime repair prompts unless they require interactive confirmation.

## Solution Assessment

Yes for the code shape: marking only the running-service restart prompt interactive-only is the narrow fix while preserving stopped-service start repair. The remaining question is maintainer acceptance of the compatibility tradeoff.
