## What This Changes

The PR marks the doctor Gateway running-service restart prompt as requiring interactive confirmation and adds regression coverage that non-interactive repair skips running restarts while still starting stopped services.

## Best Possible Solution

Merge the narrow lifecycle prompt-policy fix once maintainers accept the non-interactive restart compatibility tradeoff and normal merge gates stay green.

## Reproduction Assessment

Yes. The linked issue and current main source show `doctor --fix --non-interactive` can reach `service.restart()` for a loaded running Gateway after a failed health check because `confirmRuntimeRepair` auto-approves repair-mode runtime repairs unless the prompt requires interactive confirmation.

## Solution Assessment

Yes. The PR changes the narrow running-service restart prompt that creates the SIGTERM risk while leaving stopped-service start repair on the existing non-interactive repair path; the broader alternative of making all service lifecycle prompts interactive-only already proved too broad for stopped-service recovery.
