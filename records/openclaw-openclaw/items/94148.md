## Summary

Keep open: current main and the latest release still have the unguarded non-interactive running-Gateway restart path, and this PR is the focused open candidate fix. The patch looks correct with no blocking findings, but it changes unattended Gateway repair semantics and still needs maintainer acceptance of that compatibility tradeoff before merge.

## What This Changes

The PR prevents `doctor --fix --non-interactive` from restarting an already-running Gateway while preserving stopped-service auto-start and adding focused daemon-flow tests.

## Best Possible Solution

Land the narrow lifecycle split once maintainers accept that explicit or interactive Gateway lifecycle commands own running-service restarts, while stopped-service recovery remains automatic in non-interactive repair mode.

## Reproduction Assessment

Yes, source-level: current main can reach the running-service `service.restart()` path after a failed Gateway health/probe path, and repair-mode prompts are auto-approved unless they are explicitly guarded. I did not run a live supervised Gateway reproduction in this read-only review.

## Solution Assessment

Yes, this is the narrowest maintainable fix shape I found: it gates only explicit non-interactive running-service restarts while preserving stopped-service start repair and explicit repair/`--yes` restart behavior. The remaining question is maintainer acceptance of the compatibility tradeoff.
