## Summary

Keep open for maintainer review: the patch is narrow and correct, and the behavior is not on current main or the latest release, but it intentionally changes unattended doctor repair semantics for already-running Gateways.

## What This Changes

The PR marks the running Gateway restart repair prompt as interactive-only and adds regression tests while preserving non-interactive start of a stopped loaded Gateway service.

## Best Possible Solution

Land the narrow prompt-policy change after maintainers explicitly accept that already-running Gateway restarts require an interactive or explicit restart path, while stopped-service starts remain auto-repaired.

## Reproduction Assessment

Yes. Source inspection shows current main can reach service.restart() for a running Gateway because repair-mode confirmRuntimeRepair auto-approves prompts unless they are marked interactive-only, and the PR body supplies live output showing the patched running Gateway PID stayed unchanged.

## Solution Assessment

Yes. This is the narrowest maintainable fix shape I found: mark only the running-service restart prompt interactive-only while leaving stopped-service start repair on the existing non-interactive recovery path; if maintainers want unattended restarts, that should be an explicit opt-in path.
