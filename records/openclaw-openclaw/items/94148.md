## Summary

Keep open: current main and the latest release still auto-approve the running-Gateway restart prompt in non-interactive repair mode, while this focused PR adds the missing interactive-only guard and regression tests. The patch looks correct, but the merge still needs maintainer acceptance of the compatibility tradeoff for unattended workflows that expected implicit restarts.

## What This Changes

The PR changes doctor Gateway daemon repair so non-interactive fix mode skips restarting a running Gateway while still auto-starting a stopped service.

## Best Possible Solution

Land the narrow prompt-policy split only after maintainers accept that running-Gateway restarts should be an explicit lifecycle action while stopped-service recovery stays automatic.

## Reproduction Assessment

Yes, source-level: current main can reach `service.restart()` for a running Gateway after a failed health path, and non-interactive repair mode auto-approves that prompt unless it is marked interactive-only.

## Solution Assessment

Yes as a code shape, with a product caveat: marking only the running-service restart prompt interactive-only is the narrowest fix I found, but maintainers still need to accept the compatibility tradeoff.
