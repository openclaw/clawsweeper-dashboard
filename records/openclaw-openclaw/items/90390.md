## Summary

Keep open: current main and v2026.6.9 still use the 30-minute Discord component callback default and 24-hour cap, while the linked open PR is the active fix candidate and still needs maintainer acceptance of the longer default/cap tradeoff.

## What This Changes

The issue requests changing Discord component callbacks to better support async review workflows by raising the unset default TTL from 30 minutes to 7 days and the existing `agentComponents.ttlMs` cap to 30 days.

## Best Possible Solution

Resolve the TTL default and cap policy in https://github.com/openclaw/openclaw/pull/90394 or a narrow replacement, then close this issue after the accepted change lands.

## Reproduction Assessment

Yes, at source level: unset Discord component sends pass no `ttlMs` into the registry, and current main falls back to 30 minutes with a 24-hour schema cap. I did not run a multi-day live Discord click path, but the default and cap behavior are directly visible in source and docs.

## Solution Assessment

Unclear: changing the Discord plugin default and existing cap is the narrow implementation boundary if maintainers accept the policy. The exact 7-day default and 30-day maximum remain product/security tradeoffs, so the best solution depends on maintainer acceptance or revised values.
