Keep open: current main and v2026.6.9 still use the 30-minute Discord component callback default and 24-hour cap, while the linked open PR is only a candidate and still needs maintainer acceptance of the longer stale-action window.

## What This Changes

The issue requests raising Discord component callback lifetime defaults from 30 minutes to 7 days and the bounded override cap from 24 hours to 30 days for async review workflows.

## Best Possible Solution

Resolve the TTL default and cap policy through the linked PR or a narrow replacement, then close this issue after the accepted values, docs, config metadata, and tests land.

## Reproduction Assessment

Yes, at source level: unset Discord component sends fall back to a 30-minute registry TTL, and current schema/docs cap the override at 24 hours. I did not run a live multi-day Discord click path in this read-only review.

## Solution Assessment

Unclear: the Discord plugin registry/config/docs surface is the narrow implementation boundary if maintainers accept the policy. The exact 7-day default and 30-day cap remain product and security tradeoffs rather than an established bug-fix contract.
