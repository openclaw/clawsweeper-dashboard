## Best Possible Solution

Land this sender-level gate after adding focused iMessage send coverage and redacted macOS/imsg SIP proof; keep the separate TTS schema note out of this PR.

Yes for source reproduction: current main forwards `payload.replyToId` into iMessage final delivery, and `sendMessageIMessage` serializes it as bridge reply metadata without checking `actions.reply`. I did not establish a live SIP-enabled macOS reproduction in this read-only review.

## Solution Assessment

Yes: the sender boundary is the narrowest maintainable fix because direct, durable, and monitor iMessage sends funnel through `sendMessageIMessage` before bridge parameters are built. Adding a new `replyToMode` config is unnecessary for this bug because the existing `actions.reply` setting already expresses the private threaded-reply gate.
