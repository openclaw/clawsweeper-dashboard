## Summary

Keep open. Current main still has a source-backed delivery gap where a non-empty routed reply can be reported as successful after outbound delivery returns zero results, which can skip the delivery-mirror transcript append and match the Telegram message-loss symptom.

## What This Changes

The issue asks OpenClaw to verify and retry delivery for assistant responses in sessions whose deliveryContext points at a Telegram chat or topic.

## Best Possible Solution

Make the shared routed reply/outbound delivery contract require an actual provider delivery result or explicit durable retry/claim handoff before a non-silent final reply is counted as delivered.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. Yes for a focused source-level path: make deliverOutboundPayloads return [] for a non-empty routed final reply, then routeReply reports success without a provider message id or mirror append.

## Solution Assessment

Yes, keeping this open is the right maintainer path. The narrow repair belongs in the shared route-reply/outbound contract and dispatch fallback coverage, not in a Telegram-only watchdog or stale-lock cleanup workaround.
