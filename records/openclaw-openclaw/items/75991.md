## Summary

Keep open. Current main still has a source-backed silent-delivery path: routed non-empty replies can be counted as successfully delivered when outbound delivery returns an empty result set, which also skips the delivery-mirror transcript append.

## What This Changes

The issue asks OpenClaw to verify and retry CLI backend replies for Telegram delivery contexts instead of letting transcript-only assistant responses miss Telegram delivery.

## Best Possible Solution

Update the shared routed reply/outbound delivery contract so non-silent final replies require an actual provider delivery result or an explicit durable retry/claim outcome before the turn is marked delivered or mirrored.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. Yes for a focused source-level path: make deliverOutboundPayloads return [] for a non-empty routed final reply, then routeReply and its dispatch callers currently report success without a provider message id or mirror append.

## Solution Assessment

Yes, keeping this open for a narrow shared delivery-contract fix is the best path. A Telegram-only watchdog or stale-lock cleanup workaround would not address the cross-channel routeReply success semantics.

## Repair Work Prompt

Fix openclaw/openclaw#75991. Current main can treat a non-empty routed final reply as delivered when deliverOutboundPayloads returns [] (for example because the outbound queue entry was claimed by another owner), which also skips delivery-mirror append. Update the shared routeReply/outbound delivery contract so non-silent final replies with a deliveryContext are not counted as delivered unless there is an actual provider delivery result, or an explicit durable retry/claim outcome that guarantees another owner will deliver/mirror it. Preserve the duplicate-send protection from #70386; do not remove the claimed-by-other-owner guard or blindly re-send from both owners. Cover normal dispatch and ACP routed delivery callers, plus mirror behavior. Add a CHANGELOG.md fix entry because this is user-visible message delivery. Do not make this Telegram-only unless the shared contract already proves other channels cannot hit the same path.
