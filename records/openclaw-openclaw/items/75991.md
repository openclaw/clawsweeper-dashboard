## Summary

Keep open. Current main still has a source-backed silent-success delivery gap: a visible routed reply can receive an empty outbound result, be reported as successful, skip delivery-mirror append, and prevent Telegram fallback from firing.

## What This Changes

Ensure CLI-backend assistant responses for sessions with Telegram delivery contexts are actually delivered or retried instead of only appearing in the transcript.

## Best Possible Solution

Make non-silent routed final delivery require an actual provider delivery result or an explicit durable claim/retry outcome before marking the turn delivered, with Telegram fallback and delivery-mirror behavior covered by tests.

## Reproduction Assessment

Yes for the focused source-level bug: mock deliverOutboundPayloads to return [] for a visible routed reply and routeReply currently reports ok: true, after which ACP delivery can mark the turn delivered. The full intermittent lock cascade remains unclear without live logs.

## Solution Assessment

Yes. Keeping this open for a shared delivery-contract fix is the narrowest maintainable path; the launchd watchdog only masks lock contention and does not prevent false successful route reports.

## Repair Work Prompt

Fix the routed delivery false-success gap for https://github.com/openclaw/openclaw/issues/75991. A visible non-silent routed reply must not be marked delivered merely because routeReply returned ok when deliverOutboundPayloads produced no provider result. Preserve the existing queue-claim/dedupe behavior from #70386: an empty result can mean another recovery/drain path owns the queue entry, so model that as an explicit durable claim/retry handoff or otherwise make it observable without re-sending duplicates. Likely changes are in routeReply and ACP routed delivery, with Telegram fallback coverage. Add focused tests showing that a visible routed final with no outbound result does not set deliveredFinalReply/deliveredVisibleText or suppress Telegram fallback, while a legitimate claimed-by-other-owner/queued handoff does not reintroduce duplicate sends. Add a one-line CHANGELOG.md fix entry because this is user-facing. Do not change bundled plugin boundaries, provider defaults, or unrelated delivery queue policy.
