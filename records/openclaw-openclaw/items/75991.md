## Summary

Keep open. Current main still has a source-backed silent delivery gap: a routed non-empty reply can be treated as delivered when outbound delivery returns no provider result, which can suppress Telegram fallback and leave no delivery-mirror transcript entry.

## What This Changes

Ensure CLI backend assistant replies for sessions with a Telegram deliveryContext are delivered or retried instead of only landing in the session transcript.

## Best Possible Solution

Make routed non-silent final delivery require either a real provider delivery result or an explicit durable retry/claim outcome before marking the turn delivered, with Telegram fallback and delivery-mirror behavior covered by tests.

## Reproduction Assessment

Unclear for the complete intermittent lock cascade. A focused source-level reproduction is available by mocking deliverOutboundPayloads to return [] for a non-empty routed final reply and asserting routeReply/ACP delivery do not report a delivered Telegram-visible response.

## Solution Assessment

Yes, keeping this open is the right path. The narrowest maintainable fix is in the shared route-reply/outbound delivery contract with Telegram fallback coverage, not the launchd stale-lock workaround.
