## Summary

Keep open. Current `main` still has a source-backed silent-delivery gap: a routed non-empty reply can be treated as delivered when outbound delivery returns zero results, which skips the delivery-mirror append and can match the Telegram loss described here.

## What This Changes

The issue reports intermittent loss of `claude-cli` assistant replies to Telegram delivery contexts after transcript generation, especially around long tool chains and session lock contention.

## Best Possible Solution

Update the shared routed reply/outbound delivery contract so non-silent final replies require an actual provider delivery result or an explicit durable retry/claim outcome before the turn is marked delivered, with mirror and Telegram fallback coverage.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. Yes for a focused source-level path: make `deliverOutboundPayloads` or the queue claim return `[]` for a non-empty routed final reply, then `routeReply` and its dispatch consumers currently report success without a provider message id or mirror append.

## Solution Assessment

Yes, keeping this open is the best current outcome. The narrow fix belongs in the shared route-reply/outbound contract plus dispatch fallback coverage, not in a Telegram-only watchdog or stale-lock cleanup workaround.
