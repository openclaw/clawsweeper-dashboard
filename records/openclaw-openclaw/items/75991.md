## Summary

Keep open. Current main still has a plausible silent-success delivery gap: routed replies can return success when the outbound layer reports zero delivered results, which also prevents the delivery-mirror append and can stop Telegram fallback logic from firing.

## What This Changes

The issue reports intermittent Telegram message loss for Claude CLI-backed turns where the assistant response reaches the session transcript but is not delivered or mirrored to the Telegram delivery context.

## Best Possible Solution

Make non-silent routed final delivery prove an actual outbound result or durable retry before reporting success, preserve or add stable mirror metadata for session-backed replies, and let Telegram fallback/retry when delivery does not produce a real result.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. A high-confidence source-level reproduction exists for the missing-delivery signal: mock `deliverOutboundPayloads` to return `[]`, then `routeReply` reports success with no message id and no mirror can be appended.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is in the shared route-reply/outbound delivery contract plus Telegram fallback coverage, not a Telegram-only watchdog workaround.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75991. Current main can treat a routed Telegram final reply as successfully queued even when `deliverOutboundPayloads` returns an empty result array; because the outbound mirror append only runs when results are non-empty, this can leave a Claude CLI-backed assistant response in the transcript with no delivery-mirror row and no Telegram fallback. Implement a narrow fix so non-silent routed final delivery only reports success after at least one actual outbound result or an explicit durable retry/queue state that guarantees delivery. Preserve existing silent/empty-payload behavior. Ensure session-backed route replies pass stable mirror metadata/idempotency where appropriate, and do not weaken the existing outbound recovery work tracked by #51628. Add regression coverage proving `routeReply` does not return ok for empty non-silent delivery results, Telegram dispatch sends fallback or retries when route delivery produces no actual result, and normal successful Telegram/Slack delivery still mirrors exactly once. Add a user-facing CHANGELOG.md Fixes entry crediting @Lead2Legacy. Do not change Claude CLI provider selection, Telegram auth/pairing policy, or broad session-lock timeout defaults in this repair.
