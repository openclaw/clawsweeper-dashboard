## Summary

Keep open. Current main still has a source-backed silent-delivery gap: routed replies can be marked successful when outbound delivery returns no delivered result, which prevents the missing-delivery condition from reaching Telegram fallback and transcript mirroring.

## What This Changes

Ensure CLI backend replies for sessions with a Telegram delivery context are actually delivered or retried instead of only being written to the session transcript.

## Best Possible Solution

Make non-silent routed final delivery prove a real outbound result or durable retry before reporting success, preserve mirror metadata for session-backed replies, and let Telegram fallback when no delivered result exists.

## Reproduction Assessment

Yes for the source-level delivery gap: mock or force `deliverOutboundPayloads` to return `[]`, then `routeReply` reports success and routed dispatch can mark a final as queued without any mirror append. The full field cascade with Telegram, session lock contention, and Claude CLI remains unclear without a live reproduction.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is in the shared route-reply/outbound delivery contract plus Telegram fallback coverage, not a Telegram-only stale-lock watchdog.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75991. Current main can silently treat routed delivery as successful when `deliverOutboundPayloads` returns an empty result array: `routeReply` returns `ok: true`, `dispatchReplyFromConfig`/ACP routed delivery count that as a queued final, `deliver.ts` only appends `delivery-mirror` rows when results are non-empty, and Telegram skips its no-visible-response fallback when `queuedFinal` is true. Implement a narrow reliability fix so non-silent routed final delivery with a session delivery context is not considered successful unless there is a real outbound delivery result or a durable retry path that preserves mirror metadata. Make Telegram fallback/retry fire when routed delivery produces no visible result. Cover both the generic route-reply/outbound contract and Telegram dispatch behavior; include direct agent command mirror handling only if it is part of the same narrow fix. Do not add a Telegram-only launchd/watchdog workaround or startup-time core migration. Add a single-line `CHANGELOG.md` fix entry, omitting bot/maintainer thanks if no human contributor should be credited.
