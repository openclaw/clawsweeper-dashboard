## Summary

Keep open. Current main still has a source-backed silent-delivery gap: a routed non-empty reply can be treated as successfully queued/delivered when outbound delivery returns zero results, which can skip both delivery-mirror append and Telegram fallback.

## What This Changes

The issue reports intermittent Telegram message loss where Claude CLI backend replies appear in the session transcript but are not mirrored or delivered through the Telegram delivery context.

## Best Possible Solution

Update the shared routed reply/outbound delivery contract so non-silent final replies require either an actual provider delivery result or an explicit durable retry/claim outcome before the turn is marked delivered, with Telegram fallback and delivery-mirror behavior covered by tests.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. Yes for a focused source-level path: make deliverOutboundPayloads or the queue claim return [] for a non-empty routed final reply, then routeReply and its dispatch consumers currently report success without a provider message id or mirror append.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is in the shared route-reply/outbound delivery contract plus Telegram fallback coverage, not a Telegram-only watchdog or stale-lock cleanup workaround.
