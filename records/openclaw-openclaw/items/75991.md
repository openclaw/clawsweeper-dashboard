## Summary

Keep open. Current main still has a source-backed routed-reply loss path: a non-empty final reply can be counted as delivered when outbound delivery returns zero results, which skips the delivery-mirror append and prevents Telegram's fallback from firing.

## What This Changes

The issue requests verification and retry for CLI backend responses routed to a Telegram delivery context when the transcript has a final assistant reply but no delivered Telegram mirror.

## Best Possible Solution

Make non-silent routed final replies require a concrete provider delivery result or explicit durable retry/owned-claim outcome before they count as delivered, with mirror and Telegram fallback coverage.

## Reproduction Assessment

Yes for a focused source-level path: have deliverOutboundPayloads return [] for a non-empty routed final reply, such as the claimed-by-other-owner path. The broader macOS lock-timeout cascade remains unverified live.

## Solution Assessment

Yes, fixing the shared route-reply/outbound delivery contract is the narrow maintainable path. A Telegram-only watchdog or stale-lock cleanup mitigates contention but does not close the silent success gap.

## Repair Work Prompt

Fix the routed final-reply silent delivery success gap described by https://github.com/openclaw/openclaw/issues/75991. Keep the change narrow: non-silent routed final replies should not be counted as delivered when deliverOutboundPayloads returns zero provider results unless there is an explicit durable retry/owned-claim outcome that guarantees delivery. Preserve the existing no-duplicate-send behavior for queue entries claimed by recovery/drain. Add regression coverage for routeReply returning a failure or retryable outcome on zero delivery results, dispatch-from-config not setting queuedFinal/counts for that path, Telegram fallback firing when no visible delivery occurred, and delivery-mirror behavior staying tied to actual delivery.
