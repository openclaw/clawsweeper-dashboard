## Summary

Keep open. Current main still has a shared routed-delivery gap where a non-empty Telegram-bound reply can be reported as delivered when outbound delivery returns zero results, skipping the delivery-mirror append and matching the silent-loss symptom.

## What This Changes

Ensure CLI backend replies for sessions with a Telegram delivery context are actually delivered or retried instead of only being written to the session transcript.

## Best Possible Solution

Make the shared routed reply/outbound contract require a provider delivery result or explicit durable retry/claim state for non-silent final replies, with mirror and Telegram fallback coverage.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. Yes for a focused source-level path: make deliverOutboundPayloads return [] for a non-empty routed final reply, then routeReply reports ok: true and the delivery mirror is skipped.

## Solution Assessment

Yes, keeping this open for a shared contract fix is the best current path. A Telegram-only watchdog or stale-lock cleanup does not guarantee final reply delivery or transcript mirroring.
