## Summary

Keep open. Current main still has a source-backed silent-success delivery gap: a routed non-empty reply can be marked delivered when outbound delivery returns zero results, which prevents delivery-mirror append and can bypass Telegram fallback handling.

## What This Changes

The report asks OpenClaw to verify and retry CLI backend assistant responses for sessions with a Telegram deliveryContext instead of leaving only JSONL transcript rows.

## Best Possible Solution

Make routed non-silent final delivery require either an actual outbound result or a recorded durable retry/claim outcome before marking the turn delivered, with Telegram fallback and delivery-mirror behavior covered by tests.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. A high-confidence source-level reproduction exists: mock deliverOutboundPayloads to return [] for a non-empty routed final reply, and routeReply/ACP delivery currently report success without a provider message id or mirror append.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is in the shared route-reply/outbound delivery contract plus Telegram fallback coverage, not a Telegram-only launchd watchdog workaround.

## Repair Work Prompt

Fix the delivery reliability gap from https://github.com/openclaw/openclaw/issues/75991. Current main can treat a non-empty routed CLI/ACP final reply as delivered when deliverOutboundPayloads returns []: routeReply reports ok with no messageId, dispatch-acp-delivery marks final/visible delivery complete, delivery-mirror is not appended, and Telegram fallback may not fire. Adjust the shared route-reply/outbound delivery contract so non-silent routed final delivery either proves at least one actual provider result/message id or records a durable retry/claim outcome before success is propagated. Preserve the #70386 duplicate-send guard for concurrent queue drains; do not add launchd/watchdog behavior and do not weaken delivery-mirror dedupe/filtering. Add focused tests for routeReply empty-result failure or durable-retry handling, ACP routed delivery not marking visible/final delivered on unproven delivery, Telegram fallback behavior, and existing outbound queue-claim semantics. Add a CHANGELOG.md fix entry unless maintainers classify the final patch as pure internal/test-only.
