## Summary

Keep open. Current main still has a source-backed silent-delivery gap: a non-empty routed final reply can be treated as queued when outbound delivery returns no results, which can skip both the delivery-mirror transcript append and Telegram fallback delivery.

## What This Changes

The issue requests reliable Telegram delivery and retry behavior for CLI backend assistant responses in sessions with a Telegram `deliveryContext`, especially after lock or cleanup timeout cascades.

## Best Possible Solution

Update the shared routed reply/outbound delivery contract so non-silent final replies require an actual provider delivery result or an explicit durable retry/claim outcome before the turn is marked delivered, with Telegram fallback and delivery-mirror tests covering the zero-result path.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. Yes for a focused source-level reproduction: make `deliverOutboundPayloads` or the active queue claim return `[]` for a non-empty routed final reply, then current main reports success without a provider message id or mirror append.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is in the shared route-reply/outbound delivery contract plus Telegram fallback coverage, not a Telegram-only watchdog or stale-lock cleanup workaround.
