## Summary

Keep open. Current main still has the source-backed silent-success gap described in the report: routed replies can report success when outbound delivery returns zero delivered results, which prevents delivery-mirror append and can bypass Telegram fallback handling.

## What This Changes

The issue reports intermittent Telegram message loss for Claude CLI-backed turns where the assistant response reaches the session transcript but is not delivered or mirrored to the Telegram delivery context.

## Best Possible Solution

Make non-silent routed final delivery prove an actual outbound result or durable retry before reporting success, preserve stable mirror metadata for session-backed replies, and let Telegram fallback or retry when delivery produces no real result.

## Reproduction Assessment

Unclear for the full intermittent lock cascade. A high-confidence source-level reproduction exists: mock deliverOutboundPayloads to return [], then routeReply reports ok with no message id and no mirror can be appended.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is in the shared route-reply/outbound delivery contract plus Telegram fallback coverage, not a Telegram-only watchdog workaround.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75991. Current main can treat a routed Telegram final reply as successfully queued even when deliverOutboundPayloads returns an empty result array; because the outbound mirror append only runs when results are non-empty, this can leave a Claude CLI-backed assistant response in the transcript with no delivery-mirror row and no Telegram fallback. Implement a narrow fix so non-silent routed final delivery only reports success after at least one actual outbound result or an explicit durable retry/queue state that guarantees delivery. Preserve existing silent, reasoning-suppressed, and empty-payload behavior. Ensure session-backed route replies pass stable mirror metadata/idempotency where appropriate, and coordinate with the delivery recovery work tracked by #51628 rather than weakening it. Add regression coverage proving routeReply does not return ok for empty non-silent delivery results, Telegram dispatch sends fallback or retries when route delivery produces no actual result, and normal successful Telegram/Slack delivery still mirrors exactly once. Add a user-facing CHANGELOG.md Fixes entry crediting @Lead2Legacy. Do not change Claude CLI provider selection, Telegram auth/pairing policy, or broad session-lock timeout defaults in this repair.
