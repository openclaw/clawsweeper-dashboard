## Summary

Keep open: this PR is a plausible focused bug fix, but the current head still has a blocking ACP visibility semantics issue, a weak regression assertion, no contributor real-behavior proof, and a conflicting branch state.

## What This Changes

The PR skips requester-session wake/steering for Discord completion delivery and gates ACP resolved-session identity notices on `acp.stream.tagVisibility.session_info_update`, with focused tests.

## Best Possible Solution

Preserve direct delivery to bound Discord ACP threads through the existing delivery contracts, and gate identity notices with the shared ACP projection visibility helper plus focused regression coverage and redacted live proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the relevant current-main paths: completion delivery can hit requester-session wake before direct delivery, and the identity notice ignores ACP tag visibility, but the Discord/ACP runtime behavior still needs real proof.

## Solution Assessment

No, not yet. The Discord direct-delivery direction is plausible, but the notice suppression should use `isAcpTagVisible(resolveAcpProjectionSettings(cfg), "session_info_update")`, the test should prove the notice preconditions, and the PR needs contributor real-behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Honor resolved ACP tag visibility for the notice:** `src/auto-reply/reply/dispatch-acp.ts:282-286`
  - body: The new guard only suppresses the resolved identity notice when `tagVisibility.session_info_update` is explicitly `false`. Use the shared resolved visibility helper here and cover the unset-config case.

Assert that the dispatch still produced an expected final reply, or otherwise prove the notice conditions are met, before checking that the identity notice text is absent.
  - confidence: 0.76
