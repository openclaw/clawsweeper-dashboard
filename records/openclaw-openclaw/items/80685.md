## Best Possible Solution

Keep a narrow fix that preserves Discord ACP thread completion delivery while reusing the shared ACP tag-visibility resolver/defaults and adding non-vacuous regression coverage plus redacted real-behavior proof.

## Reproduction Assessment

No, not with high-confidence live proof. Source inspection of current main shows the reported paths: requester-session wake runs before Discord direct completion delivery, and the resolved identity notice is emitted without consulting ACP tag visibility.

## Solution Assessment

No, not yet. The Discord direct-delivery direction is plausible, but the ACP notice suppression should use `isAcpTagVisible(resolveAcpProjectionSettings(cfg), "session_info_update")` semantics and the test should prove the identity-notice path was actually exercised.
