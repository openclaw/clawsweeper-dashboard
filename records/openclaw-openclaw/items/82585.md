## Summary

Keep open: the PR targets a plausible Synology Chat webhook normalization problem, but it changes default behavior through a new config option and still lacks after-fix real Synology Chat proof for this broader replacement patch.

## What This Changes

The PR adds `channels.synology-chat.triggerWord`, threads it through Synology Chat account resolution, and changes webhook text cleanup to strip only that configured prefix when stripping leaves non-empty text.

## Best Possible Solution

Keep the fix inside the Synology Chat plugin, but land it only after maintainers accept the explicit `triggerWord` contract, the config cleanup gap is fixed, and the contributor adds redacted live webhook proof for no-trigger Synology behavior.

## Reproduction Assessment

Yes for the source-level failure path: a POST payload where `text` equals `trigger_word`, or where `trigger_word` is the first word of `text`, will be stripped by current main before delivery. I did not execute a live Synology webhook run in this read-only review.

## Solution Assessment

Unclear. The plugin-owned sanitizer is the right boundary, but the proposed default behavior and new config contract need maintainer approval and the new config field should participate in account cleanup.
