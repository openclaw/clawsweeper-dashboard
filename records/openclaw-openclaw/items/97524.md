## Summary

Keep open: current main and v2026.6.10 still do not have a browser WebSpeech provider path for the Dashboard Talk button, and resolving the mismatch needs a maintainer decision about browser-native dictation versus provider-backed Talk/WebChat voice input.

## What This Changes

The issue requests browser WebSpeech-style dictation from the Dashboard Talk button when `webSpeech` is configured.

## Best Possible Solution

Settle the Talk/WebChat voice-input boundary with maintainers, then either fold this into the canonical provider-backed voice work, intentionally revive browser-native dictation, or update docs/UI so unsupported `webSpeech` expectations are not presented as current behavior.

## Reproduction Assessment

Yes at source level for the mismatch: current UI creates realtime Talk sessions through Gateway RPCs and has no browser WebSpeech path. I did not reproduce the Windows 11/Chrome microphone scenario live.

## Solution Assessment

No as a narrow bug patch: adding a direct `webkitSpeechRecognition.start()` call would bypass the current realtime Talk/provider boundary. The maintainable solution needs a maintainer-approved browser-native dictation or provider-backed STT direction first.
