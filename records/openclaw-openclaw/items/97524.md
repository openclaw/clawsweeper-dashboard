## Summary

Keep open: current main and v2026.6.10 do not implement a `webSpeech` Talk provider or any `webkitSpeechRecognition` path for the Dashboard Talk button, but the right fix is a maintainer product/security boundary decision rather than an automatic bug patch.

## What This Changes

The issue reports that the Dashboard Talk button with a `webSpeech` talk provider detects microphone audio but never calls `webkitSpeechRecognition.start()` on Windows 11/Chrome with an external speakerphone.

## Best Possible Solution

Settle the WebChat voice-input boundary in the canonical voice discussion, then either document/remove the stale WebSpeech expectation or implement an approved browser SpeechRecognition or Gateway/provider-backed STT path with privacy, docs, and real browser proof.

## Reproduction Assessment

Yes at source level for the mismatch: current main and v2026.6.10 have no WebSpeech provider path for the Talk button, so a config expecting `webSpeech` cannot cause `webkitSpeechRecognition.start()` to run. I did not run a Windows/Chrome live reproduction.

## Solution Assessment

No as a narrow bug fix: forcing the Talk button to call WebSpeech would bypass the current realtime Talk/provider boundary. The safer solution is a maintainer-approved WebChat voice-input direction plus cleanup of the stale docs expectation.
