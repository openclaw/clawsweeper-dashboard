## Summary

Keep open: current main and the latest release still make workspace-derived plugin discovery part of plugin loader cache identity, while source and tests show workspace-local plugin isolation is intentional. Related install-record and runtime-registry reuse work reduced adjacent hot paths, but the safe cross-workspace cache contract still needs maintainer review and current live profiling.

## What This Changes

Request to reduce redundant plugin-loader cold loads by narrowing workspaceDir's role in PluginLoaderCacheState cache identity or adding an operator cache-size tuning path.

## Best Possible Solution

Define a maintainer-approved cache identity and compatibility-aware reuse path that shares global, bundled, and installed plugin work only when workspace-local discovery and plugin scope are proven safe, then validate it with negative workspace-local-plugin coverage and a current gateway profile.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection does reproduce the central cache-key condition: workspaceDir still changes discovery roots, cache identity, and active-registry compatibility.

## Solution Assessment

No, unconditional workspaceDir removal is not the best safe fix because workspace-local plugin discovery is current behavior. A compatibility-aware reuse design that preserves workspace isolation is the safer direction.
