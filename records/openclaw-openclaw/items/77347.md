## Summary

Keep open: current main still keys plugin runtime registry reuse by workspace-derived discovery inputs, and the report describes a concrete performance problem, but workspace identity is also part of documented plugin discovery and active-registry compatibility, so a safe fix needs maintainer design rather than mechanical removal.

## What This Changes

The issue requests reducing redundant plugin loader cold loads by removing unnecessary workspace granularity from the runtime registry LRU key or adding a tunable cache size.

## Best Possible Solution

Design a workspace-aware reuse rule that shares registries only when resolved discovery inputs and runtime workspace state are provably equivalent, with regression coverage for shared and isolated workspace cases.

## Reproduction Assessment

Yes, at source level: current main still passes `options.workspaceDir` into the loader cache key, and tests assert that changing the workspace prevents runtime registry reuse. I did not run a live timing benchmark because this was a read-only review.

## Solution Assessment

Unclear: reducing redundant cold loads is the right direction, but simply removing `workspaceDir` is not proven safe against current workspace plugin discovery and registry compatibility contracts. A conditional compatibility check over resolved roots would be safer.
