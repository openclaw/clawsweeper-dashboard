## Summary

Keep open. Current main intentionally treats workspace changes as cache-shaping because workspaces can contribute different `.openclaw/extensions` plugin sets, but the reported cold-load churn is plausible enough to need a deliberate cache-design follow-up rather than cleanup closure.

## What This Changes

The issue requests changing plugin loader cache granularity so identical global/plugin configurations can reuse warm registry entries across agent workspaces.

## Best Possible Solution

Design the cache so globally identical loads can share warm state only when workspace-local plugin roots and activation/runtime inputs are proven equivalent, while preserving workspace plugin isolation and adding regression coverage for both cases.

## Reproduction Assessment

Yes for the cache-key divergence: current main builds the key from workspace-derived plugin roots, and existing tests expect cross-workspace active-registry reuse to fail. No live timing profile was run for the reported cold-load duration.

## Solution Assessment

No for the proposed raw removal of `workspaceDir`: current discovery supports workspace-local plugins, so the key is not universally redundant. A maintainable solution should separate shared/global cache inputs from workspace-specific discovery rather than flattening all workspaces together.

## Risks / Open Questions

- Removing `workspaceDir` from the key outright could reuse a registry across workspaces that have different `.openclaw/extensions` contents.
- The reported 12-16s cold-load timing was not live-profiled in this read-only review, so the performance impact is accepted from the issue report rather than independently measured.
- A safer fix likely needs a cache partition/fingerprint design that distinguishes workspace-local plugin roots without fragmenting purely global loads.
