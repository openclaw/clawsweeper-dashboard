## Summary

Keep open: current main and the latest release still keep workspace-derived discovery roots in plugin loader cache identity, and current tests/compatibility checks show workspace-local plugin isolation is intentional. Adjacent fixes reduced install-record reads and some runtime registry reuse misses, but they do not close the workspace cache-key design question.

## What This Changes

The issue requests reducing redundant plugin loader cold loads by making plugin registry cache reuse less workspace-specific when plugin configuration is otherwise identical.

## Best Possible Solution

Define an owner-approved cache identity and reuse path that shares global, bundled, and installed plugin work only when workspace-local discovery is proven irrelevant, then validate it with negative workspace-local-plugin coverage and a current gateway profile.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection does reproduce the central cache-key condition: workspaceDir still changes discovery roots, cache identity, and active-registry compatibility.

## Solution Assessment

No, the requested unconditional workspaceDir removal is not the best safe fix because workspace-local plugin discovery is current behavior. A compatibility-aware reuse design that preserves workspace-local isolation is the safer direction.
