## Summary

Still necessary: current main keeps workspace-derived plugin roots, scope, and runtime/load-option modes in plugin loader cache identity, while tests deliberately prevent cross-workspace reuse. The report and follow-up comment provide credible latency evidence, but the safe fix needs a compatibility-aware cache/reuse design rather than simply removing `workspaceDir`.

## What This Changes

The issue asks to reduce redundant plugin loader cold loads caused by workspace- and scope-granular plugin registry cache keys.

## Best Possible Solution

Design a compatibility-aware registry reuse rule that shares loaded registries only when workspace roots, manifests, activation inputs, runtime mode, plugin scope, and config/trust inputs are proven equivalent, with isolation tests for unsafe workspaces.

## Reproduction Assessment

Yes at source level, but not by a live current-main timing run: current main still hashes workspace-derived roots and `onlyPluginIds` into loader cache identity, and tests intentionally reject cross-workspace reuse.

## Solution Assessment

Unclear as proposed: reducing redundant cold loads is the right fix direction, but simply removing `workspaceDir` is unsafe because workspace-local plugin discovery is a current isolation boundary.
