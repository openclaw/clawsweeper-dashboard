## Summary

Keep open: current main still keys runtime plugin registry reuse by workspace, and the report identifies a plausible hot-path performance problem, but workspace roots are also documented and implemented as part of plugin discovery scope, so a safe fix needs maintainer cache-identity design.

## What This Changes

Optimize plugin loader cache reuse so equivalent plugin configurations across workspaces do not trigger redundant cold runtime loads.

## Best Possible Solution

Design workspace-aware runtime registry reuse that shares entries only when discovery roots, plugin fingerprints, install records, and runtime workspace compatibility are provably equivalent, then cover both shared and isolated workspace cases in tests.

## Reproduction Assessment

Yes, at source level: current main passes `workspaceDir` into the plugin load cache key, and tests assert that changing workspaces prevents registry reuse. I did not run a live timing benchmark because this was a read-only review and `pnpm` is unavailable in the checkout.

## Solution Assessment

Unclear: the requested optimization is reasonable, but directly removing `workspaceDir` is not proven safe because workspace plugin roots are documented and implemented as part of discovery scope. A conditional reuse rule based on discovery equivalence is the safer direction.
