## Summary

Keep open: current main still keys plugin runtime cache reuse by workspace-derived discovery context, and the reported performance problem is plausible, but workspace roots can change the discovered plugin set and current tests encode workspace isolation, so this needs maintainer design rather than a mechanical cleanup close.

## What This Changes

The issue asks to avoid redundant plugin-loader cold loads by sharing cache entries across workspaces when plugin configuration is otherwise identical.

## Best Possible Solution

Design a workspace-aware reuse rule that shares loaded runtime registries only when workspace plugin discovery and runtime compatibility are provably identical, with regression coverage for both shared and isolated workspace cases.

## Reproduction Assessment

Yes, at source level: current main still includes `workspaceDir` in the discovery context used for the plugin loader cache key, and tests assert that changing workspaces prevents registry reuse. I did not run a live timing benchmark because this review was read-only.

## Solution Assessment

Unclear: the optimization goal is valid, but simply removing `workspaceDir` is not proven safe because workspace plugin roots and active registry workspace compatibility are current contracts. A conditional reuse design based on identical discovery inputs would be safer.
