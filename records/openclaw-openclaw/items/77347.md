- `impact:crash-loop`: The reported synchronous cold plugin loads can block the gateway event loop for many seconds, which is an availability/hang-class impact.

## Summary

Keep open: current main and the latest release still keep workspace-scoped plugin discovery in the loader cache identity, and that protects real workspace-local plugin loading. Adjacent install-record and metadata-snapshot fixes reduce related overhead, but they do not settle the safe cross-workspace reuse design or the remaining `onlyPluginIds`/activation-flag fanout reported in the comments.

## What This Changes

The issue asks OpenClaw to avoid redundant plugin-loader cold loads by reconsidering `workspaceDir` and related load-option fanout in the plugin registry cache key.

## Best Possible Solution

Define an owner-approved cache identity and reuse path that shares global, bundled, and installed plugin work across workspaces only when workspace-local discovery is proven irrelevant, with negative coverage for workspace-local plugins and current gateway profiling.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. The report includes strong field logs, and current source proves workspace-distinct cache identity plus workspace mismatch guards, but the exact latency delta still needs a current gateway profile after adjacent fixes.

## Solution Assessment

No, unconditional `workspaceDir` removal is not the best safe fix because workspace-local plugin discovery is real. A conditional reuse or split-cache design that preserves workspace-local isolation is the safer path.
