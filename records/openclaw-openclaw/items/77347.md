## Summary

Keep open: current `main` has important partial fixes for scoped plugin-loader reuse, but the remaining workspace-cache identity question is still intentionally preserved by source and tests, and the newer reporter data shows related `onlyPluginIds`/activation fanout is still an active performance concern rather than a cleanup close.

## What This Changes

The issue requests reducing redundant plugin-loader cold loads by removing or narrowing `workspaceDir` and related scope inputs from the runtime plugin registry cache identity.

## Best Possible Solution

Design a workspace-aware reuse rule that shares loaded registries only when discovery roots, manifest snapshots, activation inputs, runtime mode, and requested plugin scope are provably compatible, while keeping workspace-local plugin isolation intact.

## Reproduction Assessment

Yes, at source level: current `main` still includes workspace-derived roots and `onlyPluginIds` in plugin-loader cache compatibility, and tests assert that cross-workspace reuse is rejected. I did not run a live gateway timing benchmark for the reported 7s to 16s stalls.

## Solution Assessment

Unclear: the requested optimization direction is valid, but simply removing `workspaceDir` is not the best safe fix because workspace-local plugin discovery is a current contract. A conditional reuse design with regression coverage for both same-input sharing and workspace isolation is safer.
