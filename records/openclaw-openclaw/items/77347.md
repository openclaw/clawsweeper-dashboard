## Summary

Keep open: current main still keys runtime plugin registry reuse by workspace-derived discovery context, and the reported performance problem is plausible, but workspace roots are documented and tested as part of plugin discovery and runtime registry compatibility, so a safe fix needs maintainer design rather than deleting `workspaceDir` from the cache key.

## What This Changes

Request to reduce redundant plugin-loader cold loads by allowing identical plugin configurations to reuse cached runtime registries across agent workspaces.

## Best Possible Solution

Design a workspace-aware reuse rule that shares registries only when workspace plugin discovery roots, load paths, activation metadata, and active runtime state are provably equivalent, with regression coverage for both shared and isolated workspace cases.

## Reproduction Assessment

Yes, at source level: current main passes `options.workspaceDir` into the plugin loader cache key, and tests assert that changing workspaces prevents registry reuse. I did not run a live timing benchmark because this review was read-only.

## Solution Assessment

Unclear: the optimization direction is reasonable, but simply removing `workspaceDir` is not the best proven fix because workspace plugin discovery and active registry compatibility currently depend on it. A conditional reuse design or richer equivalence check would be safer.
