## Summary

Keep open: current main still includes workspace-derived discovery roots and plugin scope in plugin-loader cache identity, while docs and tests show workspace plugins are a real isolation boundary. Related open fixes address parts of the hot path, but they do not yet settle the remaining workspace/cache-compatibility design.

## What This Changes

The issue asks OpenClaw to avoid redundant plugin-loader cold loads caused by workspace and plugin-scope cache-key granularity.

## Best Possible Solution

Design and land a compatibility-aware loader reuse rule that shares registries only when discovery roots, config/trust, activation inputs, runtime mode, install records, and requested plugin scope are provably compatible.

## Reproduction Assessment

Yes, at source level: current main still includes workspace-derived roots and `onlyPluginIds` in plugin-loader cache compatibility, and tests explicitly reject cross-workspace reuse. I did not run a live gateway timing benchmark for the reported stalls.

## Solution Assessment

No, not as a blanket `workspaceDir` removal. The safer direction is a conditional reuse policy that preserves workspace-local plugin discovery and load-options compatibility while avoiding redundant cold loads for equivalent runtime contexts.
