## Label Justifications

- `P2`: This is a credible performance and availability bug in the plugin loader hot path, but it is bounded to loader/cache design and needs maintainer shaping rather than emergency handling.
- `impact:crash-loop`: The reported synchronous cold loads block the Node event loop for many seconds, creating an availability/hang-style impact for gateway turns.

## Summary

Keep open: current main still makes workspace-derived plugin roots part of the plugin loader cache identity and explicitly rejects cross-workspace reuse, while existing code only mitigates scoped-plugin churn within a compatible workspace.

## What This Changes

The issue asks OpenClaw to avoid redundant plugin loader cold loads by not keying otherwise identical registry cache entries on `workspaceDir`, or by adding a loader-cache tuning fallback.

## Best Possible Solution

Design a workspace-safe cache reuse path that shares full or superset registries only when discovery roots, install records, activation metadata, load options, and workspace plugin roots are proven equivalent; treat cache-size tuning as a fallback, not the primary fix.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection shows the workspace-sensitive cache identity and the discussion provides concrete latency traces, but current-main live gateway proof is still missing.

## Solution Assessment

No, simply removing `workspaceDir` is not proven to be the best fix because workspace roots affect plugin discovery and metadata snapshot compatibility. The safer direction is conditional workspace-safe registry reuse with focused tests and profiling.
