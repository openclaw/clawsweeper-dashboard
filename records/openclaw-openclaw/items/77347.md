## Summary

Keep open: current main still keys plugin loader registry reuse by workspace, and the report identifies a plausible hot-path performance problem, but workspace identity is also a documented plugin discovery and runtime isolation input, so a safe fix needs maintainer design rather than a mechanical key removal.

## What This Changes

Request to stop `workspaceDir` from fragmenting plugin loader LRU cache entries, or make cache capacity tunable, to avoid repeated cold plugin loads across agent workspaces.

## Best Possible Solution

Design a workspace-aware reuse rule that shares loaded registries only when discovery roots and workspace-dependent runtime state are provably identical, with regression coverage for both shared and isolated workspace cases.

## Reproduction Assessment

Yes, at source level: current main passes `options.workspaceDir` into the loader cache key and tests assert that changing the workspace prevents registry reuse. I did not run a live timing benchmark in this read-only review.

## Solution Assessment

Unclear: the optimization direction is valid, but simply removing `workspaceDir` is not proven safe because workspace-specific plugin roots and active runtime registry compatibility are current contracts. A conditional reuse design keyed by actual discovery/runtime compatibility would be safer.
