## Summary

Keep open: current main still includes the workspace-derived plugin root in the loader cache identity and rejects active-registry reuse across different workspace directories. The merged install-record work mitigates adjacent hot-hit overhead, but it does not settle the owner decision for safe cross-workspace registry reuse when workspace-local plugins may exist.

## What This Changes

The issue requests narrowing or removing `workspaceDir` from the plugin loader LRU cache key so identical plugin setups across multiple workspaces do not trigger redundant cold plugin loads.

## Best Possible Solution

Define an owner-approved cache identity that reuses global, bundled, and installed plugin work across workspaces only when workspace discovery is proven absent, with regression coverage for workspace-local plugins and a current gateway latency check.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main reproduction was run, but source inspection shows that different workspace directories still produce distinct cache identity and active-registry compatibility rejects cross-workspace reuse.

## Solution Assessment

Is this the best way to solve the issue? Unclear; removing `workspaceDir` globally is not the best safe fix because workspace discovery is real, but a conditional cross-workspace reuse path or split cache identity could solve the reported performance problem.
