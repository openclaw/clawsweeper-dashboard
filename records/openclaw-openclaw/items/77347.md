## Summary

Keep open: current main and v2026.6.8 still include workspace-derived plugin discovery roots in the loader cache identity, and that protects real workspace-local plugin loading. Adjacent install-record and dispatch-registry reuse fixes reduce the reported latency class, but they do not prove a safe cross-workspace reuse design or close the remaining cache-key question.

## What This Changes

The issue requests reducing redundant plugin-loader cold loads caused by workspace-scoped and option-scoped plugin registry cache keys.

## Best Possible Solution

Define an owner-approved cache identity and reuse path that shares global, bundled, and installed plugin work only when workspace-local discovery is proven irrelevant, then validate it with negative workspace-local-plugin coverage and a current gateway profile.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection does reproduce the central cache-key condition: workspaceDir still changes discovery roots and cache identity, while current comments provide credible field timing that needs refreshed profiling.

## Solution Assessment

No, the requested unconditional workspaceDir removal is not the best safe fix because workspace-local plugin discovery is current behavior. A split-cache or compatibility-aware reuse design that preserves workspace-local isolation is the safer direction.
