## Summary

Keep open: current main still partitions plugin runtime registry cache keys by workspace-derived discovery roots, and the reported performance concern is source-reproducible, but workspace plugins and active registry compatibility make a wholesale key removal unsafe without maintainer design.

## What This Changes

Request to reduce redundant plugin loader cold loads across agent workspaces by avoiding unnecessary `workspaceDir` cache-key partitioning or adding a tunable loader cache size.

## Best Possible Solution

Design a workspace-aware reuse rule that shares runtime registries only when discovery roots, activation inputs, and runtime workspace state are provably identical, with regression coverage for both shared and isolated workspace cases.

## Reproduction Assessment

Yes, at source level: current main passes `workspaceDir` into the plugin load cache key, and tests assert that changing the workspace prevents registry reuse. I did not run a live timing benchmark in this read-only sweep.

## Solution Assessment

Unclear: the optimization direction is valid, but simply removing `workspaceDir` is not proven safe because workspace plugin roots and active-registry workspace compatibility are current contracts. A conditional reuse design is the safer path.
