## Summary

Keep open: current main has partial scoped active-registry reuse, but it still treats workspace identity and several activation/scope permutations as loader-cache compatibility inputs. The report and the later v2026.5.7 timing comment identify a credible hot-path performance problem, while current source shows that blanket removal of `workspaceDir` would cross a real workspace-plugin discovery boundary.

## What This Changes

The issue asks OpenClaw to avoid redundant cold plugin registry loads when equivalent plugin configurations are used across workspaces or scope/activation permutations.

## Best Possible Solution

Design workspace-aware registry reuse around explicit discovery fingerprints and prepared runtime handles: share only when workspace roots, config/activation inputs, runtime mode, and requested plugin capabilities are proven compatible, with regression coverage for both safe scoped reuse and required workspace isolation.

## Reproduction Assessment

Yes, at source level: current main still derives the loader cache key from `workspaceDir` and `onlyPluginIds`, and tests preserve workspace-incompatible reuse while allowing only narrow compatible scoped reuse. I did not run a live timing benchmark in this read-only review.

## Solution Assessment

No for the proposed blanket `workspaceDir` removal: workspace-specific plugin discovery is a current contract. The safer fix is conditional reuse or prepared-registry reuse when discovery and activation inputs are identical or provably compatible.
