## Summary

Keep open: current main still keys plugin runtime registry reuse by workspace, and the report identifies a real hot-path/performance concern, but the same workspace value also drives workspace plugin discovery and runtime registry compatibility, so a safe fix needs maintainer design rather than a mechanical removal.

## What This Changes

The issue requests reducing redundant plugin-loader cold starts across agent workspaces by removing or narrowing `workspaceDir` from the plugin loader LRU cache key, or by making the cache size configurable.

## Best Possible Solution

Design a workspace-aware reuse rule that shares loaded registries only when workspace plugin discovery and workspace-dependent runtime state are provably identical, with regression coverage for both shared and isolated workspace cases.

## Reproduction Assessment

Yes, at source level: current main passes `options.workspaceDir` into the plugin load cache key and tests assert that changing the workspace prevents registry reuse. I did not run a live timing benchmark because this was a read-only review.

## Solution Assessment

Unclear: the requested optimization direction is valid, but simply removing `workspaceDir` is not proven safe because workspace-specific plugin roots and active registry workspace compatibility are current contracts. A narrower conditional reuse design would be safer.
