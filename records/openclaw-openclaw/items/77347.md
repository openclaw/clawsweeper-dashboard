## Summary

Keep open: current main still includes workspace-derived plugin discovery inputs in the runtime loader cache identity, and the report identifies a plausible hot-path performance problem. A mechanical removal is not safe because workspace plugin roots, discovery precedence, and active runtime compatibility are current contracts.

## What This Changes

The issue asks OpenClaw to avoid redundant cold plugin loads across workspaces by removing or relaxing `workspaceDir` from the plugin loader LRU cache key, or by making the cache size configurable.

## Best Possible Solution

Design a workspace-aware reuse rule that shares loaded registries only when discovery roots, candidate inventory, activation inputs, and workspace-dependent runtime state are provably identical, with regression coverage and profiling.

## Reproduction Assessment

Yes, source-level: current main passes `workspaceDir` into the plugin discovery/cache fingerprint and tests assert that changing workspace prevents registry reuse. The timing impact was not live-profiled in this read-only review.

## Solution Assessment

Unclear: the requested optimization direction is valid, but removing `workspaceDir` from the key is not the safest fix because current docs and tests treat workspace plugins as part of discovery and compatibility. A conditional reuse strategy based on identical resolved discovery inputs is safer.
