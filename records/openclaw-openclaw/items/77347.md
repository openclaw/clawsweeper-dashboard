## Summary

Keep open: current main still separates plugin loader runtime cache by workspace, and the report identifies a plausible hot-path performance problem, but workspace plugin discovery and active registry compatibility are current contracts that make a mechanical removal unsafe.

## What This Changes

Requests making plugin loader cache identity less workspace-specific, or adding a cache-size knob, to avoid redundant cold plugin loads across otherwise identical agent workspaces.

## Best Possible Solution

Add a workspace-aware reuse strategy that shares loaded registries only when discovery roots, install/config inputs, and runtime workspace state are equivalent, with coverage for both reuse and isolation cases.

## Reproduction Assessment

Yes, at source level: current main passes `workspaceDir` into the loader cache key and tests assert that changing the workspace prevents registry reuse. I did not run a live timing benchmark because this review was read-only.

## Solution Assessment

Unclear: the optimization direction is valid, but simply removing `workspaceDir` is not proven safe because workspace plugin roots and active registry workspace compatibility are current contracts. A conditional reuse design is the safer path.
