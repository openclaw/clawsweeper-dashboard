## Summary

Keep open: current Control UI Markdown Preview can leave an explicit `/Users/...` Markdown link as a same-origin HTTPS anchor, while the existing assistant-media download path is separate, authenticated, and root-allowlisted. This is a valid narrow UI bug, but the fix should not widen local-file access.

## What This Changes

The issue asks Control UI Markdown Preview to stop turning local absolute artifact links into misleading Gateway routes and instead render them inertly, route them through a safe artifact path, or show an explicit hint.

## Best Possible Solution

Treat host-local filesystem paths in Markdown as intentionally non-navigable text or hinted anchors, and only translate them to assistant-media URLs when they are already inside the existing authenticated, allowlisted media roots.

## Reproduction Assessment

Yes from source inspection: the Markdown Preview calls `toSanitizedMarkdownHtml`, and that sanitizer resolves `/Users/...` against the Gateway origin as an allowed HTTPS URL. I did not run a live browser click in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to make obvious local filesystem Markdown hrefs inert or route only already-allowed media sources through the existing assistant-media path. Adding broad local download access would be the wrong boundary for this bug.
