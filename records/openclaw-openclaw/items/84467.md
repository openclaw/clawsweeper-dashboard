## Summary

Keep open: current main still renders root-relative host filesystem paths in Markdown Preview as same-origin Gateway links, while the intentional local-file serving path is the ticketed assistant-media route.

## What This Changes

The issue requests safer Control UI handling for Markdown links that look like local filesystem artifact paths, such as `/Users/...`, so they do not navigate as misleading Gateway routes.

## Best Possible Solution

Make obvious host-local filesystem Markdown hrefs inert or intentionally convert only already-authorized artifacts to the existing assistant-media flow, without broadening local-file access from the Gateway.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: render `[example.docx](/Users/...)` in Markdown Preview and click it; the sanitizer keeps the href as an allowed same-origin HTTPS URL.

## Solution Assessment

Yes. The narrow maintainable solution is to make local filesystem-looking Markdown links non-navigable or route only ticketed artifacts through the existing assistant-media boundary, not to serve arbitrary local paths.
