## Summary

Keep this PR open, but it is not merge-ready: the central fix direction is useful, current `main` still has the hostless `isPortBusy` probe, and this branch is one of several candidates for the same open bug; however the patch collapses an inconclusive multi-host probe into `false` and the supplied proof is only test output, not the real IPv4-only listener behavior after the change.

## What This Changes

The PR exports `checkPortInUse` from `src/infra/ports-inspect.ts` and changes `isPortBusy` in `src/cli/ports.ts` to call that multi-endpoint probe.

## Best Possible Solution

Reuse the multi-endpoint probe for `isPortBusy`, but preserve a closed status mapping where `busy` is true, `free` is false, and `unknown` surfaces as an error or diagnostic before merging one canonical candidate for the open bug.

## Reproduction Assessment

No live reproduction was run in this read-only review, but the current-main source path is high-confidence: `isPortBusy` still does a hostless bind and `forceFreePortAndWait` exits when that helper reports false.

## Solution Assessment

No, not as-is. Reusing `checkPortInUse` is the right owner-boundary direction, but the implementation should not collapse `unknown` into false and still needs real after-fix proof.
