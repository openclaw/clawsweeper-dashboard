## Summary

The patch is a focused, contract-consistent fix for a valid cron lifecycle bug: the system event has already been enqueued, so an unavailable immediate heartbeat should not turn the completed one-shot run into a skipped result. The queued fallback re-enters the normal heartbeat policy rather than bypassing disabled, quiet-hours, or busy checks. The PR still needs after-fix proof from a real patched gateway before merge.

## What This Changes

The PR makes main-session system-event cron runs return success and request a queued fallback wake when the immediate heartbeat is skipped, with regression coverage for disabled heartbeats.

## Best Possible Solution

Keep the proposed event-result semantics, directly assert that the delete-after-run job is absent from the list or store, and capture one patched gateway execution showing both the wake outcome and deleted SQLite row.

## Reproduction Assessment

Yes at the source and released-behavior level: the supplied CLI and SQLite capture matches the current code path that records a skipped result and retains the disabled row. A current-main reproduction was not independently executed during this read-only review.

## Solution Assessment

Yes, this is the narrowest maintainable fix because it makes the cron result reflect the already-successful event enqueue and reuses the existing heartbeat queue instead of adding a second wake mechanism. The regression should additionally assert deletion directly.
