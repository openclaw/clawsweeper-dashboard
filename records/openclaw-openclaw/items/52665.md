## Summary

Keep open. Current main still lacks apt retry or fix-missing handling in the active sandbox-common build path, and the related implementation PR is closed without merge, so the requested reliability hardening remains valid.

## What This Changes

Request to add bounded apt-get retry handling and fix-missing behavior to the sandbox-common Docker image build so transient network or mirror failures are less likely to break setup.

## Best Possible Solution

Land a focused Dockerfile hardening change that adds bounded apt retry handling to `Dockerfile.sandbox-common` and the cold-start base image path, with a source-level test that locks the retry contract. Treat `--fix-missing` cautiously: either omit it in favor of fail-closed retries or add it only with evidence that missing requested packages cannot silently produce an incomplete sandbox image.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: inspect current `Dockerfile.sandbox-common` and `Dockerfile.sandbox` and observe that their apt update/install layers have no retry or fix-missing options; the live network failure itself is intermittent.

## Solution Assessment

Unclear as proposed. Bounded `Acquire::Retries` is the narrowest maintainable fix and matches existing repo precedent, but `--fix-missing` is not clearly the best default because apt can hold back requested packages instead of failing loudly.
