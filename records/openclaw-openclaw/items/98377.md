## Summary

Keep open: this is a fresh, detailed report against an existing MCP-native OAuth gateway path, and current main plus v2026.6.11 still rely on the long-lived session MCP transport/SDK refresh behavior without focused coverage for expired-token refresh in that persistent runtime. The open auth-profile PR overlaps remote MCP token refresh, but it explicitly leaves `auth: "oauth"` without `oauth.authProfileId` on the MCP SDK provider path, so it does not supersede this issue.

## What This Changes

The issue reports that long-running gateway MCP transports using MCP-native `auth: "oauth"` stop after access-token expiry even though `openclaw mcp probe` can refresh the stored token.

## Best Possible Solution

Add a focused regression for MCP-native `auth: "oauth"` without `oauth.authProfileId` in the long-lived session MCP runtime, then fix the reactive or proactive refresh path without adding a new config surface or requiring manual re-login while a valid refresh token exists.
