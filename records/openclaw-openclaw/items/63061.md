## Summary

Keep open. The source-backed mismatch is real and the latest PR head fixes the earlier normalized-guard and dev-profile test gaps, but the patch still changes documented profile isolation and state-sharing semantics without maintainer contract approval, aligned docs, or after-fix real behavior proof.

## What This Changes

This PR changes CLI profile state resolution so explicit non-sentinel `OPENCLAW_HOME` makes profiled commands use `<OPENCLAW_HOME>/.openclaw` instead of `<OPENCLAW_HOME>/.openclaw-<profile>`, with related tests.

## Best Possible Solution

Define the `OPENCLAW_HOME` versus profile precedence once, then align CLI path resolution, daemon/config defaults, CLI help, public docs, and regression tests to that contract.

## Reproduction Assessment

Yes, at source level. Current main applies `--profile` before CLI startup, derives `<OPENCLAW_HOME>/.openclaw-work` in CLI profile setup, while config defaults derive `<OPENCLAW_HOME>/.openclaw` when `OPENCLAW_STATE_DIR` is absent; I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear as submitted. The code change matches current config defaults, but it is not clearly the best fix until maintainers choose whether explicit `OPENCLAW_HOME` should override profile isolation or daemon/config resolution should follow the profile-derived state.
