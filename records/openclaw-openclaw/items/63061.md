## Summary

Keep this PR open. Current main still derives a profile-suffixed CLI state directory under explicit `OPENCLAW_HOME`, while the generic config resolver defaults that same home to an unsuffixed `.openclaw`; the PR remains relevant, but it is not merge-ready without a maintainer decision on precedence plus docs/help/changelog updates.

## What This Changes

The PR changes CLI profile state-directory resolution so explicit `OPENCLAW_HOME` uses `<OPENCLAW_HOME>/.openclaw` without a profile suffix, and adds regression tests for named, dev, and sentinel home values.

## Best Possible Solution

Decide the precedence contract explicitly. If `OPENCLAW_HOME` is meant to define the physical state root, land a revised version of this PR with updated CLI help, docs, changelog, and any needed service-path alignment; if named profiles must remain isolated even inside one `OPENCLAW_HOME`, keep the current suffixing behavior and document that profile suffixing wins.

## Reproduction Assessment

Yes. Static current-main inspection is enough to reproduce the mismatch: `applyCliProfileEnv` derives `<OPENCLAW_HOME>/.openclaw-work`, while `resolveStateDir` defaults the same explicit home to `<OPENCLAW_HOME>/.openclaw`; I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The code change is narrow and plausible if `OPENCLAW_HOME` should override profile suffixing, but the PR still needs the public contract updated and the profile isolation implications accepted before it is the best maintainable fix.
