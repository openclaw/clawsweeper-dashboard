## Summary

Keep open. Current main still has the CLI/daemon path mismatch, and the PR is plausible but not merge-ready because it changes the documented profile isolation contract without docs/changelog updates or explicit maintainer acceptance of the state-sharing boundary.

## What This Changes

The PR changes CLI profile state-dir resolution so explicit `OPENCLAW_HOME` uses `<OPENCLAW_HOME>/.openclaw` without a profile suffix, adds a normalized explicit-home helper, and updates profile tests.

## Best Possible Solution

Make the precedence contract explicit: either land the unsuffixed `OPENCLAW_HOME` behavior with updated help/docs/changelog and accepted state-sharing implications, or keep profile suffixing and fix daemon/CLI parity by propagating explicit state/config paths consistently.

## Reproduction Assessment

Yes. Static current-main inspection reproduces the mismatch: `applyCliProfileEnv` derives `<OPENCLAW_HOME>/.openclaw-work`, while `resolveStateDir` defaults the same explicit home to `<OPENCLAW_HOME>/.openclaw`; I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The code change is narrow and technically plausible, but it changes a documented isolation behavior and needs docs/changelog plus maintainer acceptance before it is the best maintainable fix.
