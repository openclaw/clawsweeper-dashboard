## Summary

Keep open: the source-level CLI/gateway path mismatch still exists on current main, but the patch remains blocked because it changes the documented profile isolation contract under explicit OPENCLAW_HOME and lacks contributor real behavior proof.

## What This Changes

The PR changes CLI profile state resolution so explicit OPENCLAW_HOME makes profiled CLI commands use <OPENCLAW_HOME>/.openclaw instead of <OPENCLAW_HOME>/.openclaw-<profile>.

## Best Possible Solution

Define the intended OPENCLAW_HOME versus profile precedence once, then align CLI path resolution, daemon/config defaults, help text, docs, and regression tests to that contract.

## Reproduction Assessment

Yes, at source level. Current main applies profile env before dispatch, derives <OPENCLAW_HOME>/.openclaw-work for --profile work, while config defaults resolve <OPENCLAW_HOME>/.openclaw; I did not run a live gateway.

## Solution Assessment

Unclear as submitted. The code change matches current config defaults, but it chooses OPENCLAW_HOME over profile isolation without first updating the documented operator contract or getting maintainer acceptance.
