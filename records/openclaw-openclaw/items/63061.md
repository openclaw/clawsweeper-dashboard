## Summary

Keep open. The underlying mismatch is source-reproducible, but this PR still needs a maintainer decision on whether explicit `OPENCLAW_HOME` should override profile isolation, plus real after-fix behavior proof from the contributor.

## What This Changes

This PR changes CLI profile state resolution so explicit non-sentinel `OPENCLAW_HOME` makes profiled commands use `<OPENCLAW_HOME>/.openclaw` and adds focused profile tests.

## Best Possible Solution

Define the `OPENCLAW_HOME` versus profile precedence once, then align CLI path resolution, daemon/config defaults, help/docs, and regression tests to that contract.

## Reproduction Assessment

Yes, at source level. Current main applies the profile env before CLI dispatch, derives `<OPENCLAW_HOME>/.openclaw-work` for `--profile work`, while config defaults resolve `<OPENCLAW_HOME>/.openclaw`; I did not run a live gateway.

## Solution Assessment

Unclear as submitted. The code change matches current config defaults, but it chooses `OPENCLAW_HOME` over profile isolation without first updating the documented operator contract or getting maintainer acceptance.
