## Summary

Keep open. Current main still has the reported Feishu message-tool fallback behavior, and the PR is a useful direction, but the proposed diff drops message text when any presentation block renders, crosses the existing lazy runtime boundary, and misses the required user-facing changelog entry.

## What This Changes

The PR exports Feishu outbound card-rendering helpers and uses them from the Feishu message-tool presentation-card builder.

## Best Possible Solution

Share a lightweight Feishu presentation-card renderer between the message-tool and outbound paths, preserve text before block elements, keep runtime imports lazy, and cover button plus select fallback behavior in tests and changelog.

## Reproduction Assessment

Yes. Invoke the Feishu message action with both `text` or `message` and a `presentation` button block, or add the same case to `extensions/feishu/src/channel.test.ts`; current main renders fallback markdown while `extensions/feishu/src/outbound.test.ts` shows the native button behavior to match.

## Solution Assessment

No. Reusing the outbound card-element behavior is the right direction, but the current PR should preserve message text and move the reusable pure rendering logic out of `outbound.ts` so `channel.ts` does not eagerly load runtime-heavy send/client code.
