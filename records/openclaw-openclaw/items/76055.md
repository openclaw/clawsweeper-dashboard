## Summary

Keep open: the reported Feishu message-tool rendering bug is valid and this PR targets the right path, but the current diff would drop mixed message text and eagerly import outbound runtime code; it also lacks after-fix real behavior proof.

## What This Changes

The PR exports Feishu outbound card element helpers and uses one from the Feishu message-tool presentation card builder so buttons/select blocks render as native card elements.

## Best Possible Solution

Extract or share a lightweight Feishu presentation-card helper for message actions and outbound payloads, preserve mixed text before block elements, keep runtime imports lazy, and add focused channel/outbound regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Feishu `message` tool send with `presentation.blocks[].type === "buttons"` currently enters `buildFeishuPresentationCard`, and the existing channel test expects markdown fallback instead of a native action element.

## Solution Assessment

No. Reusing the native card-element behavior is the right direction, but this implementation drops `text/message` whenever any block renders and crosses the lazy runtime boundary by importing `outbound.js` from the channel entry module.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Preserve text before rendered presentation blocks:** `extensions/feishu/src/channel.ts:158-164`
  - body: The new builder only uses `params.fallbackText` when no block yields an element.   - confidence: 0.89
