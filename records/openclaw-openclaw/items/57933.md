## Summary

Keep open. Current main still lacks the requested per-session active-turn heartbeat/status model: sessions expose coarse run status and change events, but not periodic liveness, turn phases, stall detection, heartbeat endpoints, or Control UI live health indicators. Related health/heartbeat work is adjacent rather than a canonical replacement for this feature.

## What This Changes

Request adding per-session heartbeat/status metadata for active turns, exposed through Gateway session APIs or subscriptions and rendered as live health indicators in the Control UI sessions list.

## Best Possible Solution

Add an additive, typed turn-health contract in the Gateway that derives state from the active run lifecycle, reuses authenticated session subscriptions for push updates, exposes a bounded snapshot through sessions.list or a new read RPC, and renders compact status badges/tooltips in the Sessions view. Defer auto-recovery until the health signal and permissions model are proven.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but we have a high-confidence verification path: inspect the current session schema, RPC list, session event payloads, and Control UI sessions table and confirm the requested heartbeat/status surface is absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the public API shape and thresholds; the safest direction is an additive Gateway-owned turn-health snapshot pushed through existing authenticated session subscriptions, with any remediation actions left out of the first pass.
