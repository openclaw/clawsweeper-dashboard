## Summary

Keep this PR open: the proposed direction matches the linked bug, but the main queue-cleanup guard misses the real wrapped error path, so the fix would not reliably stop WhatsApp listener-down sends from remaining pending for recovery replay.

## What This Changes

The PR adds a terminal outbound dispatch error for WhatsApp listener-down sends and moves those queue entries to `failed/` so delivery recovery does not replay them after restart.

## Best Possible Solution

Keep the terminal-failure design, but detect the terminal marker through the `OutboundDeliveryError` wrapper and prove the fixed WhatsApp listener-down path with focused regression tests plus redacted real runtime evidence.

## Reproduction Assessment

Yes. The linked issue provides concrete live steps and logs, and current-main source shows a listener-down platform-send failure is queued before send and then left pending via `failDelivery`; I did not run the live WhatsApp flow locally.

## Solution Assessment

No. Treating listener-down as terminal is the right narrow fix, but this implementation checks the outer wrapped error instead of the original terminal cause, so it should unwrap or preserve the terminal marker before deciding between `moveToFailed` and `failDelivery`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Unwrap terminal errors before leaving entries pending:** `src/infra/outbound/deliver.ts:1228`
  - body: The WhatsApp send path throws the new terminal error from inside platform send, but `deliverOutboundPayloadsCore` wraps platform-send failures in `OutboundDeliveryError` before this cleanup block runs. As written, `isOutboundDispatchTerminalError(err)` is false here, so listener-down sends still call `failDelivery` and remain pending for recovery replay. Check the wrapped `cause` or otherwise preserve the terminal marker before deciding between `moveToFailed` and `failDelivery`.
