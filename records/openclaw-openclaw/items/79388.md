## Summary

Keep open: this PR targets a real current-main outbound/WhatsApp recovery bug and the code direction looks sound, but the external PR still lacks inspectable after-fix real behavior proof before merge.

## What This Changes

The PR tags WhatsApp listener-unavailable sends as terminal outbound dispatch failures, moves those queue entries to `failed/`, treats legacy listener-down recovery errors as permanent, exports the helper through the plugin SDK, and adds regression tests plus changelog text.

## Best Possible Solution

Land a rebased version of the terminal pre-flight failure fix after the contributor adds redacted live proof showing listener-unavailable sends are not replayed by recovery.

## Reproduction Assessment

Yes. Current-main source shows the listener-down WhatsApp send throws before platform results are returned and the queue cleanup path keeps that entry pending via `failDelivery`; the linked issue adds live logs for the duplicate replay sequence, though I did not run a live WhatsApp repro here.

## Solution Assessment

Yes. Tagging listener-down as a terminal pre-flight dispatch failure and moving it out of the pending queue is narrower than changing WS response semantics, while preserving recovery for crash-mid-send cases.
