## Summary

Keep open: the branch is a plausible, focused fix for a source-reproducible WhatsApp delivery-recovery bug and I found no blocking code-review findings, but this external PR still lacks after-fix real behavior proof from the live WhatsApp/gateway flow.

## What This Changes

The PR tags WhatsApp listener-down outbound failures as terminal, moves those queue entries to `failed/` instead of retrying them, handles legacy recovery entries, exports the helper through the plugin SDK, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land this branch or an equivalent terminal-preflight-error fix after the contributor adds redacted after-fix proof showing an `UNAVAILABLE` WhatsApp send no longer replays after gateway restart.

## Reproduction Assessment

Yes at source level, but not as a live local reproduction in this read-only review. Current `main` enqueues before dispatch, WhatsApp listener-down throws a plain error, and the cleanup path records that error with `failDelivery`, leaving the entry pending for recovery.

## Solution Assessment

Yes, subject to the proof gate. A tagged terminal pre-flight error is the narrowest maintainable fix because it keeps crash-mid-send recovery intact while making the existing listener-down `UNAVAILABLE` response final.
