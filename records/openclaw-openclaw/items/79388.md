## Summary

Keep open: this PR targets a real WhatsApp duplicate-send bug, but the recovery fallback makes listener-down replay errors permanent and can drop otherwise retryable queued deliveries. It also still lacks acceptable after-fix real behavior proof from the contributor's setup.

## What This Changes

The PR adds a terminal outbound dispatch error, uses it for WhatsApp listener-unavailable sends, moves those queue entries out of pending recovery, adds a legacy no-listener recovery pattern, and updates tests plus changelog.

## Best Possible Solution

Keep the tagged terminal error for live WhatsApp dispatch failures, but replace the broad permanent recovery pattern with a discriminator that cannot drop retryable recovery entries, then add real after-fix WhatsApp proof before merge.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main enqueues before send, WhatsApp listener absence throws a plain error, and the catch path records failure while leaving the entry pending. I did not run a live WhatsApp reproduction in this read-only review.

## Solution Assessment

No. The terminal live-dispatch error is the right direction, but making the no-listener text globally permanent in recovery is too broad; ordinary recovery listener-down failures should remain retryable unless the entry is known to be a terminal live-send failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep listener-down recovery errors retryable:** `src/infra/outbound/delivery-queue-recovery.ts:81`
  - body: `PERMANENT_ERROR_PATTERNS` is applied to the error thrown by the recovery replay itself, not only to stored legacy entries. With this pattern, a valid queued WhatsApp delivery that hits a temporarily missing Web listener during startup or reconnect is moved straight to `failed/`, bypassing the existing backoff/reconnect-drain path and dropping the message. Scope any legacy cleanup to entries known to come from a terminal live send, and keep ordinary recovery listener-down errors retryable.
  - confidence: 0.88
