## Summary

Keep open: the future WhatsApp listener-down fix is directionally right, but the legacy recovery hunk does not do what the PR and changelog claim, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds an outbound terminal dispatch error, uses it for WhatsApp no-listener sends, moves terminal queue entries to failed, extends recovery permanent-error matching, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the typed terminal-error path for future listener-down sends, but either make legacy stored lastError handling effective and well-scoped or remove the legacy claim/regex before merge.

## Reproduction Assessment

Yes from source inspection and the linked live logs. Current main enqueues before dispatch, stores no-listener failures as pending queue state, and recovery can later replay the entry once WhatsApp is available.

## Solution Assessment

No, not yet. The typed terminal error is the narrow fix for future sends, but the legacy recovery addition tests the new pattern only after attempting replay, so it should be fixed or scoped out before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle stored no-listener failures before replay:** `src/infra/outbound/delivery-queue-recovery.ts:81`
  - body: The new permanent pattern only runs after a recovery send fails. Check the stored entry before replaying, or remove/scope the legacy-clearing claim.
