## Summary

Keep open. The terminal-error approach looks right for new WhatsApp listener-down sends, but the legacy recovery defense is incomplete and the external real-behavior proof is still insufficient for merge.

## What This Changes

The PR tags WhatsApp listener-down dispatch failures as terminal, moves those queued sends to failed instead of pending, adds a recovery permanent-error pattern, tests, and a changelog entry.

## Best Possible Solution

Keep the terminal-error design, add a pre-dispatch legacy lastError check that moves WhatsApp listener-down pending entries to failed, then attach redacted real WhatsApp proof showing recovery no longer duplicates messages.

## Reproduction Assessment

Yes, from source and the linked issue logs. I did not run a live WhatsApp repro in this read-only review, but current and shipped code paths show how a listener-down dispatch is left pending and later recovered.

## Solution Assessment

No as submitted. Tagged terminal dispatch errors are the narrow maintainable fix for new sends, but the legacy path needs to inspect existing lastError before replay instead of relying only on errors thrown during a retry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle legacy listener-down entries before retrying:** `src/infra/outbound/delivery-queue-recovery.ts:81`
  - body: The new permanent pattern only runs after a recovery attempt throws. Pending entries written by older releases already have `lastError` containing the listener-down message; once WhatsApp is connected again, recovery/reconnect drain can deliver them successfully before this pattern is consulted, so the duplicate replay still happens for the legacy entries the PR says it clears. Move matching `currentEntry.lastError` entries to `failed/` before calling `drainQueuedEntry`.
  - confidence: 0.91
