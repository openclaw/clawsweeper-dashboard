## Summary

Close as implemented on main. The exact Feishu dispatcher `mediaLocalRoots` parameter from this PR is not present, but the observable user problem is already solved by the shared reply-media normalizer: local reply media is validated with agent-scoped access, staged into managed media storage, and only then delivered through the Feishu dispatcher.

## What This Changes

The PR imports `getAgentScopedMediaLocalRoots`, adds an optional `mediaLocalRoots` parameter to the Feishu reply dispatcher, computes scoped roots at both Feishu bot dispatcher call sites, and forwards them to `sendMediaFeishu`.

## Best Possible Solution

Keep the shipped shared reply-media staging design and close this Feishu-specific plumbing PR as redundant. Any remaining Feishu account-config semantics should stay with the separate local-roots configuration work rather than adding duplicate channel-local reply plumbing.

## Reproduction Assessment

Yes. The original reproduction path is an agent reply carrying local media; current main traces that through Feishu `dispatchReplyFromConfig` into `createReplyMediaPathNormalizer`, and regression tests cover absolute workspace media staging before delivery.

## Solution Assessment

No. The PR is a reasonable narrow plumbing fix, but the current main implementation is the better maintained solution because it centralizes validation and staging for reply media across channels before Feishu receives a managed path.
