## Summary

Keep open: current main and the latest release still route Android `camera.list` through `ProcessCameraProvider.getInstance()`, so the metadata path can initialize CameraX before a user-requested capture. No merged fix, open paired PR, or canonical duplicate currently owns the remaining work.

## What This Changes

The issue requests that Android camera capability/listing metadata avoid eager CameraX initialization so opening the app does not trigger the camera privacy indicator before capture.

## Best Possible Solution

Make Android `camera.list` use `CameraManager`/`CameraCharacteristics` metadata without touching CameraX, keep `ProcessCameraProvider` only in `camera.snap` and `camera.clip`, and prove on a real Android 13+ device that app launch stays camera-idle until capture.

## Reproduction Assessment

Partly: source inspection gives a clear current-main path where invoking Android `camera.list` calls `ProcessCameraProvider.getInstance()`. No live Android run proves the full app-open privacy-dot trigger or which caller invokes `camera.list` on reopen.

## Solution Assessment

Yes for the confirmed code path: replacing Android `listDevices()` with Camera2 metadata is the narrowest maintainable fix while preserving CameraX for actual capture. The app-open caller should be verified with live proof before changing gateway or agent command policy.
