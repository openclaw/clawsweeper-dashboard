## Summary

Keep open: current main and v2026.6.8 still route Android `camera.list` through `ProcessCameraProvider.getInstance()`, so the metadata path remains capable of initializing CameraX before an explicit photo or clip. The app-open trigger still needs live Android proof because the connect advertisement path itself only builds command/capability lists from flags.

## What This Changes

The issue requests delaying Android CameraX initialization until `camera.snap` or `camera.clip` and making `camera.list`/capability reporting metadata-only.

## Best Possible Solution

Make Android `camera.list` use `CameraManager`/`CameraCharacteristics` metadata without touching CameraX, keep `ProcessCameraProvider` only in `camera.snap` and `camera.clip`, and verify that app launch stays camera-idle until capture.

## Reproduction Assessment

No high-confidence live reproduction is established here. Source inspection proves that invoking Android `camera.list` initializes `ProcessCameraProvider`, but it does not prove the reported app-open sequence invokes `camera.list`.

## Solution Assessment

Yes for the confirmed code path: replacing `listDevices()` with `CameraManager` metadata is the narrowest maintainable fix while preserving CameraX for actual capture. The app-open caller should be verified before changing gateway command policy.
