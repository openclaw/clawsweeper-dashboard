## Label Justifications

- `P2`: The issue reports a privacy-sensitive Android camera behavior with a narrow affected command surface and no evidence of data loss or core runtime outage.
- `impact:security`: The report concerns premature camera hardware access and Android's privacy indicator, which is a sensitive device-access boundary.

## Summary

Keep open: current main and the latest release still route Android `camera.list` through `ProcessCameraProvider.getInstance()`, so a metadata request can initialize CameraX before an intentional photo or clip; the app-open trigger still needs real Android proof.

## What This Changes

The issue requests delaying Android CameraX initialization so app launch and camera metadata listing do not activate camera hardware before `camera.snap` or `camera.clip`.

## Best Possible Solution

Make Android `camera.list` use `CameraManager`/`CameraCharacteristics` metadata only, keep CameraX provider initialization inside `camera.snap` and `camera.clip`, and verify on a real Android 13+ device that app launch stays camera-idle until capture.

## Reproduction Assessment

No high-confidence live reproduction for the full app-open behavior was established. Source inspection does reproduce the narrower `camera.list` path that initializes `ProcessCameraProvider`, which is enough to keep the issue open.

## Solution Assessment

Yes for the confirmed code path: replacing Android `camera.list` with Camera2 metadata is the narrowest maintainable fix. The app-open caller should be verified before changing gateway command policy or advertisement behavior.
