## Summary

Keep open: current main and v2026.6.8 still route Android `camera.list` through `ProcessCameraProvider.getInstance()`, so the metadata path can initialize CameraX before an actual capture. The full app-open trigger still needs real-device proof because Android connect advertisement itself builds commands/caps from flags rather than calling `listDevices()`.

## What This Changes

The issue requests delaying Android CameraX initialization until `camera.snap` or `camera.clip` and making `camera.list` metadata-only.

## Best Possible Solution

Make Android `camera.list` use `CameraManager`/`CameraCharacteristics` metadata without touching CameraX, keep `ProcessCameraProvider` only in `camera.snap` and `camera.clip`, and verify on a real Android 13+ device that launch stays camera-idle until capture.

## Reproduction Assessment

No for the full app-open symptom: source inspection shows Android `camera.list` reaches `ProcessCameraProvider`, but connect advertisement itself only serializes runtime flags. Real Android 13+ proof is still needed to confirm what invokes `camera.list` on reopen.

## Solution Assessment

Yes for the confirmed metadata path: replacing Android `listDevices()` with metadata-only `CameraManager` logic is the narrow maintainable fix while preserving CameraX capture behavior. If the app-open trigger is an automatic gateway or agent `camera.list`, that caller should be verified before changing gateway policy.
