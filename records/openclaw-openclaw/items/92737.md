## Summary

Keep open: current main still routes Android `camera.list` through `ProcessCameraProvider.getInstance()`, so the requested metadata-only camera listing is not implemented. The app-open green-dot trigger is not fully proven from source because node connect advertisement now uses runtime flags only, but a meaningful privacy-sensitive camera metadata bug remains.

## What This Changes

The issue requests lazy Android camera initialization so app open, capability advertisement, and `camera.list` do not touch camera hardware before `camera.snap` or `camera.clip`.

## Best Possible Solution

Make Android `camera.list` use CameraManager/CameraCharacteristics metadata without touching CameraX, keep `ProcessCameraProvider` only in capture commands, and verify on a real Android 13+ device that app launch stays camera-idle until capture.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established for the reported app-open green-dot symptom, because I did not run a real Android device and current node connect advertisement only reads flags. Source inspection does reproduce the remaining `camera.list` problem: invoking that command still calls `ProcessCameraProvider.getInstance()`.

## Solution Assessment

Yes for the confirmed code path: replacing `listDevices()` with CameraManager metadata while keeping CameraX in `snap` and `clip` is the narrow maintainable fix. A live run should still confirm whether any app-open caller invokes `camera.list` immediately after reconnect.
