## Label Justifications

- `P2`: The issue reports a privacy-sensitive Android camera behavior with a narrow affected command surface and no evidence of data loss or core runtime outage.
- `impact:security`: The report is about premature camera hardware access and Android's privacy indicator, which is a sensitive-device-access boundary.

## Summary

Keep open: current main and the latest release still route Android `camera.list` through `ProcessCameraProvider.getInstance()`, so the metadata path can initialize CameraX before an actual photo or clip. The app-open handshake portion is not fully proven from source because connect advertisement itself uses runtime flags, so the fix should make listing metadata-only and then verify the reported Android privacy-indicator behavior on a real device.

## What This Changes

Delay Android CameraX provider initialization until capture commands and keep camera capability/listing metadata from opening camera hardware.

## Best Possible Solution

Make Android `camera.list` use `CameraManager`/`CameraCharacteristics` metadata without touching CameraX, keep `ProcessCameraProvider` only in `camera.snap` and `camera.clip`, and verify on a real Android 13+ device that app launch stays camera-idle until capture.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show that invoking Android `camera.list` currently initializes `ProcessCameraProvider`, but source alone does not prove the app-open handshake invokes `camera.list`.

## Solution Assessment

Yes for the confirmed code path: replacing `listDevices()` with `CameraManager` metadata while keeping CameraX in `snap`/`clip` is the narrow maintainable fix. If the app-open trigger comes from an automatic gateway or agent `camera.list`, that caller should be verified rather than changing gateway command policy first.
