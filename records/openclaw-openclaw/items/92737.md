## Label Justifications

- `P2`: The issue reports privacy-sensitive Android camera behavior with a narrow affected command surface and no evidence of data loss or core runtime outage.
- `impact:security`: The report concerns premature camera hardware access and Android's privacy indicator, which is a sensitive device-access boundary.

## Summary

Keep open: current main and v2026.6.8 still route Android `camera.list` through `ProcessCameraProvider.getInstance()`, so metadata listing can initialize CameraX before an intentional photo or clip; the full app-open trigger still needs real Android proof.

## What This Changes

The issue reports that the Android app initializes CameraX during app open or camera metadata listing, causing Android’s camera privacy indicator before an explicit capture command.

## Best Possible Solution

Make Android `camera.list` use `CameraManager`/`CameraCharacteristics` metadata only, keep CameraX provider initialization inside `camera.snap` and `camera.clip`, and verify on a real Android 13+ device that app launch stays camera-idle until capture.

## Reproduction Assessment

No high-confidence live reproduction exists for the full app-open behavior in this read-only review. Source inspection does reproduce the narrower `camera.list` path that reaches `ProcessCameraProvider.getInstance()`, which is enough to keep the issue open.

## Solution Assessment

Yes for the confirmed code path: replacing Android `camera.list` with Camera2 metadata is the narrowest maintainable fix. The app-open caller should be verified before changing gateway command policy or advertisement behavior.
