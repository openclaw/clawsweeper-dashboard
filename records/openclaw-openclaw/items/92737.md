## Summary

Keep open: current main and the latest release still route Android `camera.list` through `ProcessCameraProvider.getInstance()`, so a metadata command can initialize CameraX before a requested capture. The exact app-open trigger still needs real-device proof, but the source-backed privacy issue remains unresolved.

## What This Changes

The issue asks for Android camera capability/listing to avoid eager CameraX initialization on app open and reserve camera hardware access for `camera.snap` or `camera.clip`.

## Best Possible Solution

Make Android `camera.list` use `CameraManager`/`CameraCharacteristics` metadata without touching CameraX, keep `ProcessCameraProvider` only in capture paths, and prove on Android 13+ that app launch stays camera-idle until capture.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection does show that invoking Android `camera.list` currently initializes `ProcessCameraProvider`, but source alone does not prove the app-open caller path.

## Solution Assessment

Yes for the confirmed code path: replacing Android `listDevices()` with `CameraManager` metadata is the narrow maintainable fix while preserving CameraX for `snap` and `clip`. The app-open trigger should be verified before changing gateway command policy or advertisement behavior.
