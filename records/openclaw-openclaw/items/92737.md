## Label Justifications

- `P2`: The issue reports a privacy-sensitive Android camera behavior with a focused command surface and no evidence of data loss or core runtime outage.
- `impact:security`: The report is about premature camera hardware access and Android's privacy indicator, which is a sensitive-device-access boundary.

## Summary

Keep open: current main and v2026.6.6 still route Android `camera.list` through CameraX `ProcessCameraProvider`, so the metadata command can initialize the camera stack before photo or clip capture. The exact app-open trigger is not fully proven from source, so the remaining work needs a metadata-only Android implementation plus real-device privacy-indicator proof.

## What This Changes

The issue requests lazy Android camera initialization so opening the app or listing camera metadata does not trigger the Android camera privacy indicator before `camera.snap` or `camera.clip`.

## Best Possible Solution

Make Android `camera.list` use metadata-only Camera2 APIs, keep CameraX provider initialization only in `camera.snap` and `camera.clip`, and prove on a real Android 13+ device that reopening the app stays camera-idle until capture.

## Reproduction Assessment

No, not for the full app-open claim. Source inspection does show that invoking Android `camera.list` currently initializes `ProcessCameraProvider`, but source alone does not prove that app open or connect advertisement invokes `camera.list`.

## Solution Assessment

Yes for the confirmed code path: replacing `listDevices()` with Camera2 metadata while keeping CameraX in `snap` and `clip` is the narrow maintainable fix. If a gateway or agent automatically invokes `camera.list` on reconnect, that caller should be verified with real-device proof before changing command policy.
