## Label Justifications

- `P1`: Unexpected camera hardware activation is a user-visible privacy regression in the Android companion app.
- `impact:security`: The issue concerns camera hardware activation and Android's privacy indicator before a user-requested capture.

## Summary

Keep open: current main and the latest release still route Android `camera.list` through `ProcessCameraProvider.getInstance()`, and Android documentation says the first CameraX provider initialization enumerates/queries cameras and communicates with camera hardware. I found no duplicate issue or open PR that owns this exact privacy regression; the only uncertainty is the exact app-open trigger, because source inspection shows advertisement itself is registry-only while `camera.list` is the concrete hardware-touching path. ([developer.android.com](https://developer.android.com/media/camera/camerax/configuration?utm_source=openai))

## What This Changes

The issue asks for Android camera capability and listing paths to avoid initializing CameraX or showing the camera privacy indicator until an actual capture command runs.

## Best Possible Solution

Keep Android capability advertisement pure, move `camera.list` to Camera2 metadata via `CameraManager`/characteristics, and reserve CameraX provider and lifecycle binding for `camera.snap` and `camera.clip`.

## Reproduction Assessment

Partially: source inspection shows `camera.list` initializes CameraX through `ProcessCameraProvider.getInstance()`, which matches the reported root cause. I did not run an Android 13+ device to verify the app-open green dot or identify the automatic caller that invokes `camera.list` after connect.

## Solution Assessment

Yes, likely: replacing `camera.list` with a Camera2 metadata path is the narrowest maintainable fix while keeping CameraX for actual capture. Adding a new toggle or disabling camera listing would be a weaker workaround.
