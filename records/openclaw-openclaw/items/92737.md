## Summary

Keep open: current main and the latest release still route Android `camera.list` through CameraX initialization, so the privacy-sensitive metadata path remains unresolved even though the exact app-open trigger still needs live Android proof.

## What This Changes

The issue requests delaying Android CameraX initialization until actual capture and making camera capability/list metadata avoid opening the camera stack.

## Best Possible Solution

Make Android `camera.list` use `CameraManager`/`CameraCharacteristics` metadata, keep `ProcessCameraProvider` only in `camera.snap` and `camera.clip`, and verify on Android 13+ that launch stays camera-idle until capture.

## Reproduction Assessment

No high-confidence live reproduction is available for the app-open privacy dot. Source inspection does show that invoking Android `camera.list` currently initializes `ProcessCameraProvider`, while the connect handshake itself only advertises flags.

## Solution Assessment

Yes for the confirmed metadata path: replacing `listDevices()` with `CameraManager` metadata is the narrow maintainable fix. The app-open trigger should be verified before changing gateway command policy or broader node behavior.
