## Summary

Keep open. The PR is useful, but it is not mergeable as written because raw RRF scores are far below the existing hybrid `minScore` thresholds, and the new user-facing config key is not fully reflected in the canonical docs/changelog surfaces.

## What This Changes

The PR adds an opt-in `weighted`/`rrf` memory hybrid fusion config, RRF merge logic and tests in `memory-core`, and related config schema/help/type updates.

## Best Possible Solution

Keep the opt-in config shape, but normalize RRF into the existing 0-1 score contract or make thresholding fusion-aware, add manager-level regression coverage, and update canonical docs, changelog, and generated config baseline artifacts.

## Reproduction Assessment

Yes. A static reproduction follows from enabling `fusion: "rrf"` with the default `minScore`: even a top vector+keyword hit scores about 0.016 and is filtered out by the existing 0.35 strict threshold and 0.3 keyword fallback floor.

## Solution Assessment

No. Raw RRF is reasonable for ordering, but using the raw rank-fusion value as the existing filtered `score` is not maintainable unless the score is normalized or manager thresholding becomes fusion-aware.
