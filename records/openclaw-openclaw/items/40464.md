## Summary

Keep open: the PR targets a real config data-loss report, but current main does not contain this queue/fresh-read implementation and the PR branch is stale against the current config writer. As proposed, it would drop newer write contracts around destructive-write guards, runtime snapshot finalization, canonical write notifications, and write options, so it is not safe to merge unchanged or close as implemented.

## What This Changes

The PR adds a module-level config write queue, changes ownerDisplaySecret auto-persistence to re-read and write inside the queued section, and adds queued-write/runtime-snapshot regression tests.

## Best Possible Solution

Keep one coherent config persistence path: if maintainers still want queueing and ownerDisplaySecret fresh-read behavior, layer it around the current module-level writer while preserving destructive-write protection, source snapshot projection, refresh handling, persistedHash/afterWrite notifications, skip/log flags, and dedicated regression coverage. If shipped clobber recovery is considered sufficient, close the PR as superseded with that rationale.
