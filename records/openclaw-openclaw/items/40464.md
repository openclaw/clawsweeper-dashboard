## Summary

Keep this PR open, but do not merge it as-is. Current main mitigates the reported config-wipe symptom with destructive-write rejection and last-known-good recovery, yet it does not implement this PR's queue/fresh-reread prevention, and the branch is stale against newer config-write contracts.

## What This Changes

The PR serializes module-level config writes, re-reads before auto-persisting `commands.ownerDisplaySecret`, and adds queued-write/runtime-snapshot regression tests.

## Best Possible Solution

Keep one coherent config persistence path: if queueing still lands, wrap the current module-level writer without losing destructive-write protection, source projection, runtime refresh, listener notification, and output/log controls; otherwise close the PR as superseded by the shipped recovery strategy.

## Reproduction Assessment

Yes. The linked bug gives a concrete restart/truncation path, and the PR adds targeted race tests; this review verified the relevant source and diff contracts rather than executing tests.

## Solution Assessment

No as-is. A queue plus fresh-read owner-secret persist can be a reasonable direction, but this branch needs to be rebased and reworked around the current writer contracts before merge.
