## Summary

Keep open. Current main still leaves the combined dashboard token-delivery failure path without safe manual token recovery guidance, and the linked closing PR #72802 is still open while the alternative #73227 is closed unmerged.

## What This Changes

The issue reports that `openclaw dashboard` prints a dashboard URL without the gateway token, leaving WSL/headless users unable to authenticate when automatic token delivery fails.

## Best Possible Solution

Land #72802 or an equivalent narrow dashboard command change that adds safe manual auth recovery guidance only when clipboard/browser token delivery fails, while keeping tokenized URLs limited to clipboard/browser delivery and preserving the existing redaction tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a plaintext `gateway.auth.token`, failed clipboard copy, and `openclaw dashboard --no-open`; current main logs only the bare URL plus `Use the URL above`, while tests already pin that behavior.

## Solution Assessment

Yes. The best fix direction is narrow and owner-local: add guidance naming `OPENCLAW_GATEWAY_TOKEN` / `gateway.auth.token` in the failed-delivery path without logging the token value or fragment marker.
