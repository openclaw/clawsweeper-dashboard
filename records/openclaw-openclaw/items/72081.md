## Summary

Keep open. Current main still has the headless/WSL fallback gap: when the tokenized dashboard URL cannot be delivered through clipboard or browser, the command logs only the clean URL plus a generic hint, and the open closing PR #72802 remains the implementation candidate. The earlier comment pointing to #73227 is useful context, but that PR is closed and unmerged.

## What This Changes

The issue reports that `openclaw dashboard` prints a dashboard URL without the gateway token on WSL/headless Linux, leaving the user unable to authenticate when automatic delivery fails.

## Best Possible Solution

Land or revise PR #72802 so failed clipboard/browser delivery shows a redaction-preserving recovery hint that names the token sources without logging the token value or fragment marker.

## Reproduction Assessment

Yes. Configure a literal `gateway.auth.token`, force clipboard copy and browser opening to fail, then inspect `dashboardCommand` logs; current main emits the clean URL and generic SSH/no-open fallback without manual token-source guidance.

## Solution Assessment

No, not as originally requested. Printing the tokenized URL would violate the current no-token-in-logs contract, so the narrow maintainable fix is a safe fallback hint that points to `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` without revealing credentials.
