## Summary

Keep open. Current main still sends the tokenized dashboard URL only to clipboard/browser delivery and lacks a manual recovery hint when those channels fail; the issue is already paired with open closing PR #72802, while PR #73227 is closed and unmerged.

## What This Changes

The reporter wants `openclaw dashboard` to surface gateway auth token access when the printed dashboard URL lacks a token on WSL/headless runs.

## Best Possible Solution

Land or revise PR #72802 so failed clipboard/browser delivery shows a redaction-preserving recovery hint, with tests and changelog, without logging the token value or `#token=`.

## Reproduction Assessment

Yes. Configure a literal `gateway.auth.token`, make clipboard delivery fail and browser support/opening fail or use `--no-open`, then inspect `dashboardCommand` logs; current main emits the clean URL and generic fallback without token-source recovery guidance.

## Solution Assessment

No, printing the tokenized URL in runtime logs is not the best fix because the current security contract forbids token values and `#token=` in logs. The narrow maintainable solution is a manual-auth hint naming `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` without revealing the secret.
