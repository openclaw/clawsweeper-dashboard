## Summary

Keep open. Current main still leaves the reported headless/WSL dashboard path without safe manual token recovery guidance when both clipboard and browser delivery fail, and the open closing PR remains the right implementation candidate.

## What This Changes

The issue asks for `openclaw dashboard` to provide a usable token-authenticated access path when the command prints only a non-tokenized Gateway URL in WSL/headless environments.

## Best Possible Solution

Review and land #72802, or an equivalent narrow dashboard command/test change, so failed clipboard/browser delivery prints safe manual recovery guidance naming `OPENCLAW_GATEWAY_TOKEN` or `gateway.auth.token` while preserving the existing token redaction contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a plaintext `gateway.auth.token`, `copyToClipboard` returning false, and either `--no-open` or unavailable browser support; current main then logs only a bare URL plus generic fallback/hint text.

## Solution Assessment

Yes. The best fix direction is narrow and owner-local: preserve dashboard token redaction, add manual auth guidance only when automatic token delivery fails, and pin that behavior in `src/commands/dashboard.links.test.ts`.
