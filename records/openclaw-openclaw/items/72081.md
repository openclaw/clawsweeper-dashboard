## Summary

Keep open. Current main still has the reported headless/WSL failure mode, and an open unmerged PR already uses closing syntax for this issue, so the right maintainer action is to review, land, or replace that PR rather than close early.

## What This Changes

The issue asks for `openclaw dashboard` on WSL/headless installs to provide a usable token-authenticated Control UI access path instead of only printing a clean gateway URL.

## Best Possible Solution

Review and land or replace PR #72802 with a redaction-preserving dashboard fallback: when a literal token exists but clipboard and browser delivery both fail, show users where to retrieve or set the token without logging the token value or the `#token=` marker, and cover that branch in `src/commands/dashboard.links.test.ts`.

## Reproduction Assessment

Yes. Configure a literal `gateway.auth.token`, make `copyToClipboard` return false, and make browser opening unavailable; current main then logs the clean URL plus generic SSH or `--no-open` text without naming `OPENCLAW_GATEWAY_TOKEN`, `gateway.auth.token`, or `openclaw config get gateway.auth.token`.

## Solution Assessment

No, printing the tokenized URL in runtime logs is not the best fix because current tests intentionally prevent token values and `#token=` fragments from reaching captured logs. The maintainable solution is a manual recovery hint that names token sources without exposing the secret.
