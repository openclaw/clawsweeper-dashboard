## Summary

Keep open. Current main still leaves plaintext-token dashboard users stranded when clipboard and browser delivery fail, and the open closing PR remains the right implementation candidate; the fix must add recovery guidance without logging the token value or fragment marker.

## What This Changes

The issue asks for `openclaw dashboard` to provide usable gateway token authentication guidance when the printed dashboard URL is non-tokenized.

## Best Possible Solution

Review and land #72802, or an equivalent narrow dashboard command/test change, so failed clipboard/browser delivery prints safe manual recovery guidance naming `OPENCLAW_GATEWAY_TOKEN` or `gateway.auth.token` while keeping tokenized URLs limited to automatic delivery paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a plaintext `gateway.auth.token`, `copyToClipboard` returning false, and either `--no-open` or unavailable browser support; current main then logs only the bare URL plus a generic fallback/hint.

## Solution Assessment

Yes. The best fix direction is narrow and owner-local: keep dashboard token redaction intact, add manual auth guidance only when automatic token delivery fails, and cover that path in `dashboard.links.test.ts`.
