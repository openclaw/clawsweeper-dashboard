## Summary

Keep this issue open. Current main still has the reported failed-delivery gap: the dashboard command keeps the token out of terminal logs, but when clipboard delivery fails and the browser path is disabled or unavailable, it only prints a bare URL plus generic fallback text instead of safe manual auth recovery guidance. The open closing PR remains the right implementation candidate to review or land.

## What This Changes

The issue requests that `openclaw dashboard` provide a usable token-authentication path instead of leaving users with only a bare dashboard URL in WSL/headless environments.

## Best Possible Solution

Land #72802 or an equivalent narrow dashboard command/test change that detects `includeTokenInUrl && !copied && !opened`, prints safe manual recovery guidance naming `OPENCLAW_GATEWAY_TOKEN` or `gateway.auth.token`, and preserves the existing token redaction guarantees.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a literal `gateway.auth.token`, failed `copyToClipboard`, and either `--no-open` or unavailable browser support; current main then logs only the bare URL plus a generic fallback or SSH hint.

## Solution Assessment

Yes. The best fix direction is narrow and owner-local: keep tokenized URLs limited to clipboard/browser delivery, add safe manual auth guidance only after both delivery paths fail, and cover that path in `src/commands/dashboard.links.test.ts`.
