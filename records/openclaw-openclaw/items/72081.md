## Summary

Keep this issue open. Current `main` still leaves the reported headless/WSL failure path without a literal-token recovery hint, but an open PR already uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and either landed or closed.

## What This Changes

The issue asks for `openclaw dashboard` on WSL/headless installs to provide usable token-authentication information when it prints the dashboard URL.

## Best Possible Solution

Add or land a narrow dashboard fallback that, only when tokenized delivery cannot reach clipboard or browser, tells users where to retrieve or provide the shared token without logging the token value or fragment marker; cover that failed-delivery path in the existing dashboard link tests.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to configure a literal `gateway.auth.token`, make `copyToClipboard` return false, make browser support/opening fail, and observe that current `dashboardCommand` logs only the bare URL plus the generic SSH/no-open hint.

## Solution Assessment

No, printing a tokenized URL directly to runtime logs would regress the CVE-safe behavior now covered by tests. The maintainable fix is a CVE-safe recovery hint like the open PR proposes, not restoring token fragments in logged output.
