## Summary

Keep open. Current main still lacks a safe manual-auth recovery hint when the dashboard token URL cannot be delivered through clipboard or browser, and open PR #72802 is already the linked implementation candidate with closing syntax.

## What This Changes

The issue reports that `openclaw dashboard` prints a dashboard URL without a usable gateway token path on WSL/headless environments.

## Best Possible Solution

Land or revise PR #72802 so failed clipboard/browser delivery shows a redaction-preserving recovery hint, with tests and changelog, without ever logging the token value or `#token=`.

## Reproduction Assessment

Yes. Configure a literal `gateway.auth.token`, mock clipboard delivery to fail and browser support/opening to fail, then inspect `dashboardCommand` logs; current main emits the clean URL and generic SSH/no-open fallback without a manual token-source hint.

## Solution Assessment

No, not as originally requested. Printing the tokenized URL would conflict with the current no-token-in-runtime-logs tests, so the narrow maintainable fix is a CVE-safe hint that names `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` without revealing the token or fragment marker.
