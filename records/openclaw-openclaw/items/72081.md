## Summary

Keep open. Current main still reproduces the reported stranded dashboard-auth path, and the provided discussion plus remote refs show #72802 remains an open closing PR, so this issue should wait for that PR or an equivalent narrow fix to land.

## What This Changes

The issue requests that `openclaw dashboard` provide a usable gateway-token recovery path when headless/WSL users only receive a bare Control UI URL.

## Best Possible Solution

The dashboard command should add safe recovery guidance only when tokenized automatic delivery fails, naming `OPENCLAW_GATEWAY_TOKEN` / `gateway.auth.token` as recovery sources while preserving the existing no-token-in-logs contract and covering the path in dashboard link tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a plaintext `gateway.auth.token`, `copyToClipboard` returning false, and either `--no-open` or unsupported browser opening; current main then logs the bare URL plus a generic fallback without telling the user where to recover the token.

## Solution Assessment

Yes. The narrowest maintainable direction is the owner-local dashboard command/test change described by #72802: preserve redaction, add manual auth guidance only after both automatic delivery channels fail, and avoid broader auth or plugin changes.
