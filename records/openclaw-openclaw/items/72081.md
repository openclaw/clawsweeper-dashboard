## Summary

Keep open. Current `main` still has the reported headless/WSL failure mode: the command intentionally logs a clean dashboard URL and sends the tokenized URL only to clipboard/browser delivery, but when those delivery paths fail it falls back to SSH/no-open hints without a manual token recovery hint. PR #72802 is open, unmerged, and uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and either landed or closed.

## What This Changes

The issue asks for `openclaw dashboard` to provide usable gateway token authentication when the printed dashboard URL lacks the token in WSL/headless use.

## Best Possible Solution

Land or revise the narrow dashboard fallback from PR #72802: when a literal token exists but clipboard and browser delivery both fail, show a recovery hint pointing to `OPENCLAW_GATEWAY_TOKEN`, `gateway.auth.token`, or `openclaw config get gateway.auth.token` without logging the token value or `#token=`, and cover that branch in `dashboard.links.test.ts`.

## Reproduction Assessment

Yes. Configure a literal `gateway.auth.token`, make `copyToClipboard` return false and browser opening unavailable, then inspect `dashboardCommand` output: current `main` logs the clean URL plus SSH/no-open fallback text without a manual token source hint.

## Solution Assessment

No. The literal requested behavior of printing a tokenized URL is not the safest fix because current tests intentionally prevent token values and `#token=` fragments from reaching runtime logs; a redaction-preserving fallback hint is the maintainable path.
