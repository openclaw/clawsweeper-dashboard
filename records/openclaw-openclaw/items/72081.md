## Summary

Keep open. Current main still has the reported dashboard token-delivery failure when clipboard and browser delivery both fail, and the open closing PR remains the right implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue reports that `openclaw dashboard` prints a dashboard URL without a gateway token in WSL/headless conditions, leaving the user without a manual authentication path.

## Best Possible Solution

Land #72802 or an equivalent narrow dashboard command/test change that adds safe manual recovery guidance when tokenized clipboard/browser delivery fails, while keeping token values out of logs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a plaintext `gateway.auth.token`, failed `copyToClipboard`, and either `--no-open` or unavailable browser support; current main then logs only the clean URL plus generic fallback text.

## Solution Assessment

Yes. The best fix direction is narrow and local to the dashboard command/tests: add a safe hint naming `OPENCLAW_GATEWAY_TOKEN` or `gateway.auth.token` only after delivery fails, without printing the token or `#token=`.
