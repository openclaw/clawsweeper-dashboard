## Summary

Keep open: this is a useful partial fix for the linked macOS Tahoe OS-version bug, but the branch still mixes the Darwin OS name with the macOS product version in prompts, leaves the compaction prompt caller unchanged, moves the `sw_vers` probe ahead of the cache, and lacks real macOS proof.

## What This Changes

The PR changes Darwin OS-summary `release` to use `sw_vers -productVersion`, routes CLI and embedded-attempt runtime OS metadata through that value, and updates the OS-summary test expectation.

## Best Possible Solution

Land one cached macOS runtime/display helper that preserves shipped non-Darwin prompt shape, covers CLI, embedded attempt, and compaction, and explicitly decides whether the status JSON `OsSummary.release` semantics should change before merging.

## Reproduction Assessment

Yes, from source inspection and reporter evidence: current main passes raw `os.release()` into runtime prompt paths, and the linked issue provides macOS 26.5.1 versus Darwin 25.5.0 host output. I did not run a live macOS 26 host in this read-only review.

## Solution Assessment

No. The PR is a plausible partial fix, but the maintainable fix should use a dedicated cached runtime OS display helper, cover compaction, and make the status JSON `release` compatibility decision explicit.
