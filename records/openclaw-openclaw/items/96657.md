## Summary

Keep open: current main still lets Feishu resource tools select `channels.feishu.defaultAccount` before any binding-derived account when `ctx.agentAccountId` is absent, so the reported non-inbound tool-execution account drift remains real and security-sensitive.

## What This Changes

The issue reports that Feishu Bitable/doc/wiki/drive/permission tool calls on non-inbound turns can use the global Feishu default account instead of the agent's bound Feishu account.

## Best Possible Solution

Define a maintainer-approved Feishu tool account precedence rule that keeps explicit and valid contextual accounts first, uses a safe binding-derived or enabled-family account before global default, and covers cron, relay, and heartbeat paths with regression plus live or redacted runtime proof.

## Reproduction Assessment

Yes at source level: a Feishu tool invocation with no valid `ctx.agentAccountId` reaches `resolveImplicitToolAccountId` and can return `channels.feishu.defaultAccount` before any enabled-family fallback. I did not run a live multi-tenant Feishu cron, sessions_send, or heartbeat reproduction.

## Solution Assessment

Unclear. Binding-aware precedence is likely the right direction, but the exact ordering across explicit tool account, turn context, session-derived account, agent binding, enabled-family fallback, and global default is security- and compatibility-sensitive.
