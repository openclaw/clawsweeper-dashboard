## Summary

Keep open: current main and v2026.6.10 still have the Feishu tool-account resolver ordering described in the issue, so accountless non-inbound tool calls can reach the global Feishu default before any binding-aware selection. The source-level bug is clear, but the repair changes security- and compatibility-sensitive account precedence.

## What This Changes

The issue requests binding-aware Feishu tool-account selection for non-inbound turns so resource tools use the agent-bound Feishu account instead of the global default.

## Best Possible Solution

Define a maintainer-approved Feishu tool account precedence rule that keeps explicit and valid contextual accounts first, uses a safe binding-derived or enabled-family account before global default, and covers cron, relay, and heartbeat paths with regression plus live or redacted runtime proof.

## Reproduction Assessment

Yes at source level: a Feishu tool invocation with no valid ctx.agentAccountId reaches resolveImplicitToolAccountId and can return channels.feishu.defaultAccount before any enabled-family fallback. A live multi-account Feishu cron, relay, or heartbeat reproduction was not run in this read-only review.

## Solution Assessment

Unclear. Binding-aware precedence is likely the right direction, but the exact ordering across explicit tool account, turn context, agent binding, enabled-family fallback, and global default is security- and compatibility-sensitive.
