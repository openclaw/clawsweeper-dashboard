## Summary

Keep open: current main and v2026.6.10 still let Feishu tool calls without a valid contextual account choose the global Feishu default before any binding-aware selection. Because the fix changes cross-tenant credential precedence and fallback behavior, maintainers still need to choose the security and compatibility policy.

## What This Changes

The issue requests Feishu resource tools to resolve the agent-bound Feishu account, rather than global defaultAccount, during non-inbound turns such as cron, relay, and heartbeat.

## Best Possible Solution

Define a maintainer-approved Feishu tool account precedence rule that preserves explicit and valid contextual accounts, derives safe binding or tool-capable accounts before global default, and covers cron, relay, and heartbeat with focused regression plus live or redacted proof.

## Reproduction Assessment

Yes at source level: a Feishu tool call with no valid ctx.agentAccountId reaches the shared resolver and can return channels.feishu.defaultAccount before any binding-aware account selection. I have not run a live multi-tenant Feishu cron, relay, or heartbeat reproduction.

## Solution Assessment

Unclear. Binding-aware precedence is likely the right owner-boundary fix, but the exact ordering across explicit tool params, turn context, agent bindings, capability scan, and global default is security- and compatibility-sensitive.
