## Summary

Keep open: current main and v2026.6.10 still let Feishu resource tools choose global `channels.feishu.defaultAccount` when tool context lacks a valid account, and there is no merged canonical fix for binding-aware tool account precedence yet.

## What This Changes

The issue reports that Feishu resource tools use global `defaultAccount` instead of the agent's bound Feishu account when tool execution lacks inbound account context.

## Best Possible Solution

Define a maintainer-approved Feishu tool account precedence rule that keeps explicit and valid contextual accounts first, uses a safe binding-derived or enabled-family account before global default, and covers cron, relay, and heartbeat paths with regression plus live or redacted runtime proof.

## Reproduction Assessment

Yes at source level: a Feishu tool invocation with no valid `ctx.agentAccountId` reaches `resolveImplicitToolAccountId` and can return `channels.feishu.defaultAccount` before any enabled-family fallback. Full cron, `sessions_send`, and heartbeat proof still needs live or focused runtime validation.

## Solution Assessment

Unclear. Binding-aware precedence is likely the right direction, but the exact ordering across explicit tool account, turn context, agent binding, enabled-family fallback, and global default is security- and compatibility-sensitive.
