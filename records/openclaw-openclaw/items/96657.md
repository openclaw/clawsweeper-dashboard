## Summary

Keep open: current main and the latest release still let Feishu resource tools fall back to `channels.feishu.defaultAccount` before any agent binding-aware account selection, so the reported cross-tenant tool-execution bug remains real and security-sensitive.

## What This Changes

The issue asks for Feishu resource tools to use the agent's bound Feishu account on non-inbound turns instead of falling back to the global Feishu default account.

## Best Possible Solution

Define a maintainer-approved Feishu tool account precedence rule that preserves explicit and valid contextual accounts, resolves the current agent's Feishu binding before global default, keeps safe enabled-family fallback behavior, and proves cron, relay, and heartbeat paths with focused regression and live or redacted runtime evidence.

## Reproduction Assessment

Yes at source level: a Feishu tool call with no valid `ctx.agentAccountId` reaches `resolveImplicitToolAccountId` and can return `channels.feishu.defaultAccount` before any binding-aware selection. I did not run a live multi-account Feishu cron, relay, or heartbeat reproduction.

## Solution Assessment

Unclear: binding-aware precedence is likely the right owner-boundary fix, but the exact ordering across explicit tool params, turn context, agent bindings, enabled-family fallback, and global default needs maintainer security and compatibility direction.
