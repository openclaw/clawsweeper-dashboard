## Summary

Keep open: current main and the latest release still have a Feishu tool resolver path that uses the global Feishu `defaultAccount` when no trusted `ctx.agentAccountId` reaches the tool, and choosing where to add binding-aware account context is security- and compatibility-sensitive.

## What This Changes

The issue requests binding-aware Feishu tool account selection for non-inbound turns so resource tools do not use the global `channels.feishu.defaultAccount` instead of the agent's bound Feishu account.

## Best Possible Solution

Define one maintainer-approved Feishu tool account precedence contract, thread the resolved binding-aware account into each non-inbound turn context where possible, and leave global `defaultAccount` as an intentional no-context fallback with focused regression tests.

## Reproduction Assessment

Yes at source level: when `ctx.agentAccountId` is absent, current Feishu tool resolution still reaches `channels.feishu.defaultAccount` before any binding-aware account selection. I did not run a live Feishu multi-tenant cron, sessions_send, or heartbeat reproduction.

## Solution Assessment

Unclear. Binding-aware precedence is the right product direction, but the best implementation layer may be shared turn-context resolution rather than making the Feishu resolver independently read bindings.
