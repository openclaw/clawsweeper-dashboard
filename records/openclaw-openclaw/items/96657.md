## Summary

Keep open: current main and v2026.6.10 still let Feishu tool account resolution choose the global `defaultAccount` before any binding-aware or enabled-tool-family fallback when plugin tool context lacks `agentAccountId`; changing that precedence is security- and compatibility-sensitive, so this is not safe cleanup to close.

## What This Changes

The issue reports that Feishu resource tools on non-inbound turns can use the global Feishu default account instead of the agent-bound account, causing cross-tenant permission failures.

## Best Possible Solution

Adopt a documented Feishu tool account precedence rule that keeps explicit/contextual account first, uses a valid agent binding or enabled-family account before global default, and adds regression plus redacted live multi-account proof.

## Reproduction Assessment

Yes at source level: a Feishu tool call with no explicit `accountId` and no `ctx.agentAccountId` reaches `channels.feishu.defaultAccount` before any binding-aware choice. The live cron/session/heartbeat matrix is not yet reproduced end to end.

## Solution Assessment

Unclear. Binding-aware precedence is the likely direction, but the exact ordering against explicit account, contextual/session account, enabled-family fallback, and global default needs maintainer security and compatibility approval.
