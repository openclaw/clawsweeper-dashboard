## Summary

Keep open: current main and `v2026.6.10` still have a Feishu tool-account resolver path where a non-contextual tool call can return `channels.feishu.defaultAccount` before any binding-aware account choice, and changing that precedence is security- and compatibility-sensitive.

## What This Changes

The issue asks for Feishu resource tools to resolve an agent-bound Feishu account, not the global default account, during non-inbound turns such as cron, inter-session relay, and heartbeat work.

## Best Possible Solution

Define and implement a maintainer-approved Feishu tool account precedence rule that keeps explicit and valid turn context first, uses an unambiguous binding-derived account before global fallback, and covers unbound and ambiguous cases with focused regression tests.

## Reproduction Assessment

Yes, at source level: a Feishu tool invocation with no explicit `accountId` and no `ctx.agentAccountId` reaches the resolver path that returns `channels.feishu.defaultAccount` before any binding-aware account selection. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

Unclear as a final implementation until maintainers choose the account-precedence contract. Binding-aware precedence is the right direction, but the exact ordering and ambiguity behavior are security- and compatibility-sensitive.
