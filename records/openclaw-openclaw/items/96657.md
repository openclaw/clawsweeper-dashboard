## Summary

Keep open: current main and v2026.6.10 still let Feishu tool account resolution choose global `channels.feishu.defaultAccount` before any binding-aware account, and the protected security/product labels require maintainer direction before changing fallback precedence.

## What This Changes

The issue reports Feishu resource tools using the global default account instead of the agent-bound Feishu account on non-inbound turns such as cron, inter-session relay, and heartbeat.

## Best Possible Solution

Define a maintainer-approved Feishu tool account precedence rule that keeps explicit and valid contextual accounts first, derives the correct agent-bound Feishu account before global default, and proves cron, relay, heartbeat, CLI, and embedded paths.

## Reproduction Assessment

Yes at source level: a Feishu tool invocation without a valid `ctx.agentAccountId` reaches `resolveImplicitToolAccountId` and can return `channels.feishu.defaultAccount` before any binding-aware account. I did not run a live Feishu multi-tenant cron, relay, or heartbeat reproduction.

## Solution Assessment

Unclear. Binding-aware precedence is likely the right owner-boundary fix, but the exact ordering across explicit account, turn context, agent binding, enabled-family fallback, and global default is security- and compatibility-sensitive.
