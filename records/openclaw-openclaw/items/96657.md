## Summary

Keep open: current main and v2026.6.10 still have Feishu tool account resolution that consults explicit/contextual account state and then the global Feishu default before any enabled-family fallback, with no binding-aware tool resolver step. The exact cron/session/heartbeat trigger matrix needs maintainer confirmation because current cron has some account propagation, but the plugin resolver gap is real and security-sensitive.

## What This Changes

The issue reports Feishu resource tools choosing the global Feishu default account on non-inbound turns instead of the Feishu account bound to the running agent.

## Best Possible Solution

Define and implement a maintainer-approved Feishu tool account precedence rule that keeps explicit/contextual account choices first, uses a valid agent binding or enabled-family account before global default, and covers stale/missing context with regression and live or redacted runtime proof.

## Reproduction Assessment

Yes for the resolver-level bug: current source and v2026.6.10 show a no-context Feishu tool call will choose global defaultAccount before enabled-family fallback and without consulting bindings. Unclear for every reported trigger because current cron has partial binding/account propagation that needs live or focused trigger proof.

## Solution Assessment

Unclear. Binding-aware precedence is the likely right direction, but the exact ordering against explicit account, session-derived account, enabled-family fallback, and global default needs maintainer security/compatibility approval.
