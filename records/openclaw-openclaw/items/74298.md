## Summary

Keep open. Current main does not show an OpenClaw path that strips `TELEGRAM_BOT_TOKEN` or `DISCORD_BOT_TOKEN`, but the requested migration hardening is still incomplete: the migration guide lacks a copied `.env` token-integrity checklist, and doctor/status do not provide a dedicated migration preflight for expected channel credentials.

## What This Changes

The issue requests investigation of missing Telegram/Discord bot tokens after an rsync-based machine migration, plus doctor preflight and migration documentation for copied credential integrity.

## Best Possible Solution

Document a migration checklist that verifies copied state-dir `.env` credential key presence and recommends `openclaw status --all` or channel probes after cutover, then add doctor/status diagnostics only where OpenClaw can infer an expected configured credential from channel state, account config, or SecretRef metadata without warning on unused optional channels.

## Reproduction Assessment

No, there is not a high-confidence reproduction path for the reported rsync line stripping against current main. Source and test inspection show OpenClaw preserves and consumes these token env names when present; the remaining reproducible gap is documentation/diagnostic coverage, not a demonstrated file-copy mutation.

## Solution Assessment

Unclear for the broad doctor request. A targeted docs update plus narrowly scoped diagnostics for configured channel credentials would be maintainable; a hard-coded missing-env-var check for two optional fallback names would likely produce false positives.
