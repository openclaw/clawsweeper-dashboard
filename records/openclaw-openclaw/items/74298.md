## Summary

Keep open. Current main does not show an OpenClaw code path that strips `TELEGRAM_BOT_TOKEN` or `DISCORD_BOT_TOKEN` from the trusted state-dir `.env`, but the requested doctor/docs follow-up is still valid and narrow: migration docs do not tell operators to verify state-dir `.env` channel tokens, and doctor still lacks a general secret-safe preflight for missing env-backed Telegram/Discord channel credentials.

## What This Changes

Request to investigate reported rsync migration token loss, add doctor preflight coverage for env-backed Telegram/Discord channel tokens, and document state-dir `.env` migration verification.

## Best Possible Solution

Add a generic, secret-safe doctor migration preflight that reports enabled/configured channel accounts whose env SecretRefs or default env fallbacks are unavailable, then update the migration guide with a post-copy checklist for state-dir `.env` and channel credential availability. Keep this in doctor/channel plugin diagnostic seams and docs, not startup migrations or automatic secret repair.

## Reproduction Assessment

Unclear for the exact rsync-stripping claim: current source and tests show OpenClaw preserving trusted state-dir `.env` keys. The user-visible failure mode is reproducible by configuring Telegram or Discord with an env-backed token and running diagnostics or startup with that env var missing, which current inspection can classify as missing or configured unavailable.

## Solution Assessment

Yes for the doctor/docs direction; no for an rsync-specific OpenClaw code path based on current evidence. The narrow maintainable fix is to surface existing credential availability state in doctor and document migration verification without logging or rewriting secrets.
