## Summary

Keep open. Current main does not show an OpenClaw code path that strips `TELEGRAM_BOT_TOKEN` or `DISCORD_BOT_TOKEN` from the state-dir `.env`, but the requested migration hardening is still not complete: the migration guide lacks a copied-secret integrity checklist, and current doctor/status diagnostics only cover configured or unavailable credentials rather than expected env keys lost during a machine move.

## What This Changes

The issue requests investigation of missing Telegram/Discord bot tokens after rsync migration, plus `openclaw doctor` preflight coverage and migration documentation for `.env` token verification.

## Best Possible Solution

Keep the issue open for a scoped migration-hardening decision: document state-dir `.env` token-key verification and post-cutover `openclaw status --all` or channel probes in the migration guide, then add doctor/status checks only where OpenClaw can infer an expected credential from enabled channel config, explicit SecretRefs, token files, or a migration inventory without warning on every optional env var.

## Reproduction Assessment

No, there is not a high-confidence reproduction for the reported rsync stripping against current main. The high-confidence check is source/docs inspection: OpenClaw preserves and consumes these env names when present, while migration docs and preflight coverage remain incomplete.

## Solution Assessment

Unclear. A blanket `doctor` check for missing optional bot-token env vars is not the best default; a targeted docs update plus diagnostics for explicitly configured or enabled credentials would be narrower and less misleading.
