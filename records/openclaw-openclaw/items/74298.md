## Summary

Keep open. Current main still does not show an OpenClaw code path that strips Telegram or Discord bot-token lines from the trusted state-dir `.env`, but the requested migration hardening is not complete: the migration guide lacks a copied-secret integrity checklist, and doctor/status do not provide a dedicated migration preflight for expected channel credentials.

## What This Changes

The issue requests investigation and migration hardening after a VPS copy reportedly left `TELEGRAM_BOT_TOKEN` and `DISCORD_BOT_TOKEN` absent from the copied `.env`, including doctor preflight checks and migration documentation.

## Best Possible Solution

Update the migration guide with a state-dir `.env` token-key verification checklist and post-cutover `openclaw status --all` or channel probe guidance, then add doctor/status warnings only where OpenClaw can infer an expected credential from configured channel state, token files, or SecretRefs.

## Reproduction Assessment

No, there is not a high-confidence reproduction path for the reported rsync line stripping against current main. Source and test inspection show the key names are preserved and consumed when present; a live transfer showing the lines removed would be needed to reproduce that part.

## Solution Assessment

Unclear as written. A targeted docs update plus diagnostics tied to configured credentials is maintainable, but a broad doctor check for optional env-var names would likely create false positives.
