## Summary

Keep open. Current main does not show an OpenClaw path that strips `TELEGRAM_BOT_TOKEN` or `DISCORD_BOT_TOKEN` from the state-dir `.env`, but the requested migration hardening is still incomplete: the migration guide lacks a copied-secret integrity check, and doctor/status diagnostics only cover configured or unavailable credentials rather than env fallback keys lost during a machine move.

## What This Changes

Request to investigate reported rsync loss of Telegram/Discord bot-token env lines during machine migration, add redacted preflight or doctor validation for expected env keys, and document `.env` migration checks.

## Best Possible Solution

Add migration guidance that explicitly verifies copied state-dir `.env` key presence with redacted commands, and add doctor/status diagnostics only where OpenClaw can infer an expected channel credential from configured SecretRefs, enabled channel config, durable service env metadata, or a migration inventory without warning on every optional channel env var.

## Reproduction Assessment

No, there is not a high-confidence reproduction for rsync silently stripping these lines against current main. The high-confidence check is source/test inspection: OpenClaw’s current dotenv parser and token consumers preserve/use these env names, while the migration docs and preflight coverage remain incomplete.

## Solution Assessment

Unclear. A blanket doctor check for `TELEGRAM_BOT_TOKEN` and `DISCORD_BOT_TOKEN` is not the best default because those env fallbacks are optional; a redacted migration checklist plus targeted diagnostics for explicitly configured or inferable credentials would be narrower.
