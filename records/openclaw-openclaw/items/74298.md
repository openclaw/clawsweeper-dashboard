## Summary

Keep open. Current main does not show an OpenClaw path that strips `TELEGRAM_BOT_TOKEN` or `DISCORD_BOT_TOKEN` from the state-dir `.env`, but the requested migration hardening is still not fully covered: the migration guide still lacks a token-integrity checklist, and doctor/status behavior does not provide a dedicated migration preflight for expected copied channel credentials.

## What This Changes

The issue asks OpenClaw to investigate reported rsync migration loss of Telegram/Discord bot token lines, add a doctor preflight for expected `.env` keys, and document `.env` migration best practices.

## Best Possible Solution

Keep the issue open for a scoped migration-hardening decision: document state-dir `.env` token-key verification and post-cutover `openclaw status --all` or channel probes in the migration guide, then add doctor/status checks only where OpenClaw can infer an expected credential from enabled/configured channel state or explicit SecretRefs without warning on every optional env var.

## Reproduction Assessment

No, there is not a high-confidence reproduction for the reported rsync line stripping against current main. The high-confidence check is source/docs inspection: current code preserves and consumes the token env names when present, while the migration guide and preflight coverage remain incomplete.

## Solution Assessment

Unclear for the requested broad doctor check. A targeted docs update plus narrowly scoped diagnostics for configured credentials would be maintainable; a blanket missing-env-var warning would likely be noisy and could mislead users.
