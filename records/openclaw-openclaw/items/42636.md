## Summary

Keep this PR open: the performance goal still looks valid, and current main still uses a 200-message Control UI hydration/render window. The proposed branch is stale against the active render path, so it needs a refresh before it can solve the issue cleanly.

The PR reduces the Control UI initial chat history request and render window from 200 to 80 messages by adding a shared UI constant, changing the chat controller and view, and updating UI tests.

## Best Possible Solution

Refresh the change against current main by using one shared initial chat history/render window constant, wiring it through `loadChatHistory()` and `buildChatItems()`, keeping or intentionally splitting input-history recall alignment, and adding direct coverage in the controller and chat item builder tests.

No. The PR's intent is reasonable, but the proposed patch is not the best current implementation because it changes the old view-local render path while current main uses the extracted `buildChatItems()` module. The safer fix is a small rebase/refresh that updates the active builder and its tests.
