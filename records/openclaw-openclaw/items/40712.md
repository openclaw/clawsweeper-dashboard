## Summary

Keep open: current main still keeps arbitrary in-flight assistant generation in process-local buffers and explicitly refuses to resume transcript tails ending in assistant output. Current code has partial mitigations for restart-aborted sessions and completed final-delivery replay, but the central requested behavior remains a product/architecture decision.

## What This Changes

Request durable persistence, recovery, or user notification for assistant message generation interrupted by a gateway restart.

## Best Possible Solution

Keep this as the focused tracker for durable in-flight assistant generation recovery, then split work across checkpointing, restart marking, boot-time recovery, delivery replay, and channel-safe interruption messaging.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: assistant deltas are buffered in memory, shutdown clears that state, and restart recovery rejects transcript tails ending in assistant output.

## Solution Assessment

Unclear. Durable persistence is the right direction, but the JSONL status sketch needs maintainer review against existing abortedLastRun markers, pending-final-delivery replay, the open drain-manifest producer, and channel UX semantics.
