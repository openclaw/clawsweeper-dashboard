## Summary

Keep open. Current main has useful mitigations for controlled restarts and explicit aborts, but the requested durable restart recovery for in-progress assistant generation is still missing: assistant-tail partial output is explicitly treated as non-resumable, and the newer partial-retention path only covers RPC or /stop aborts with in-memory buffers.

## What This Changes

The issue requests durable persistence and recovery of in-progress assistant responses when the gateway restarts or crashes during generation.

## Best Possible Solution

Keep this issue open as the specific tracker for assistant-generation recovery. The maintainable end state is a SessionManager-aware checkpoint/status model plus startup recovery that distinguishes generating, completed-not-delivered, sent, and interrupted states, coordinated with the broader restart-continuation RFC and drain-manifest work.

## Reproduction Assessment

Yes. The current unit test in src/agents/main-session-restart-recovery.test.ts reproduces the remaining source-level behavior: a restart-aborted session ending in partial assistant output fails recovery instead of resuming.

## Solution Assessment

No. The current mitigations are not the full solution, and the issue's raw per-delta JSONL rewrite proposal is risky unless it preserves the SessionManager parent chain and provider replay contracts; a structured checkpoint/recovery model is the safer direction.
