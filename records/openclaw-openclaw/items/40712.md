## Summary

Keep open. Current main has restart-drain and explicit-abort mitigations, but durable recovery of an in-progress assistant response after a gateway restart is still missing; the current recovery path explicitly treats assistant-tail partial output as non-resumable, and live stream buffers remain process-local.

## What This Changes

Request durable persistence and startup recovery for in-progress assistant responses when the gateway restarts or crashes.

## Best Possible Solution

Implement a SessionManager-aware checkpoint/status model plus boot-time recovery that distinguishes generating, completed-not-delivered, sent, and interrupted states while coordinating with #60864 and #74873.

## Reproduction Assessment

Yes, at source level. Current main has a unit test for a restart-aborted session ending in assistant partial output and expects recovery to fail without calling the gateway; I did not run a live restart because this was a read-only review and pnpm is unavailable here.

## Solution Assessment

No, current main is not the full solution. The drain, restart-continuation, and explicit-abort paths reduce loss in narrower cases, but the requested durable assistant-generation recovery needs a structured checkpoint/recovery design rather than raw transcript rewrites.
