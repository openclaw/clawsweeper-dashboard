## Summary

Keep open: current main mitigates clean restarts by draining active work and has restart-sentinel/session-delivery/main-session recovery pieces, but it still does not durably persist or resume arbitrary in-flight assistant generation after crash, forced restart, or drain timeout. The linked drain-manifest PR is still open and explicitly covers only the shutdown-side producer for a later recovery consumer.

## What This Changes

Request durable recovery for assistant message generation interrupted by a gateway restart, including partial state persistence, delivery status tracking, and boot-time handling for incomplete generations.

## Best Possible Solution

Keep this as the restart-generation recovery umbrella until maintainers choose one durable design that extends the existing drain, sentinel, delivery queue, and restart-aborted session recovery paths while respecting the SessionManager transcript DAG contract.

## Reproduction Assessment

Yes, for the remaining crash, forced-restart, and drain-timeout behavior by source inspection: live assistant deltas remain process-local and assistant partial transcript tails are failed rather than resumed. A normal clean restart may now drain active work, so the repro should target a crash, force path, or response longer than the configured drain window.

## Solution Assessment

Unclear. The requested JSONL status-flushing approach is plausible, but the safer maintainable direction is to build on the existing restart drain, sentinel, session-delivery queue, and SessionManager transcript recovery paths instead of adding a parallel raw JSONL status system.
