## Label Justifications

- `P2`: This is a real restart-time reliability gap with meaningful user impact, but it needs product and architecture scoping before implementation.
- `impact:data-loss`: The report is about generated assistant content disappearing when the gateway restarts mid-turn.
- `impact:message-loss`: The user-visible failure is an incomplete or missing channel reply after restart.
- `impact:session-state`: The missing durable checkpoint leaves transcript and session state inconsistent with the interrupted generation.

## Summary

Keep open: current main and the latest shipped tag have partial restart recovery, but assistant streaming deltas are still process-local and assistant-tailed transcripts are intentionally failed rather than replayed, so durable mid-generation persistence/resume is not implemented.

## What This Changes

Request durable persistence and recovery for in-progress assistant generation so gateway restarts can resume, complete, or notify instead of dropping partial replies.

## Best Possible Solution

Define a durable in-flight generation checkpoint and recovery contract that composes with abortedLastRun, pendingFinalDelivery, restart drain, and channel-safe interruption UX, then split it into focused gateway/agent changes.

## Reproduction Assessment

Yes for a source-level reproduction. Live assistant deltas are buffered in ChatRunState maps, shutdown clears that state, and the recovery tests expect assistant-tailed transcripts to fail rather than resume.

## Solution Assessment

Unclear. Durable persistence or interruption recovery is the right direction, but the JSONL status sketch needs maintainer review against current abortedLastRun, pendingFinalDelivery, restart-drain, idempotency, and channel UX semantics.
