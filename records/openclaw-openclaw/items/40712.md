## Label Justifications

- `P2`: This is a real restart-time reliability gap with meaningful user impact, but it needs product and architecture scoping before implementation.
- `impact:data-loss`: The report is about generated assistant content disappearing when the gateway restarts mid-turn.
- `impact:message-loss`: The user-visible failure is an incomplete or missing channel reply after restart.
- `impact:session-state`: The missing durable checkpoint leaves transcript and session state inconsistent with the interrupted generation.

## Summary

Keep open: current main and the latest shipped release still keep live assistant generation in process-local buffers, clear those buffers during shutdown, and intentionally fail assistant-tailed restart recovery instead of resuming it. Related restart-recovery issues and the open drain-manifest PR cover adjacent pieces, but they do not supersede this focused mid-generation recovery request.

## What This Changes

The issue requests durable persistence and resume or recovery handling for assistant responses interrupted by a gateway restart.

## Best Possible Solution

Define a durable in-flight generation checkpoint and recovery contract that composes with abortedLastRun, pendingFinalDelivery, restart sentinels, drain behavior, and channel-safe interruption UX, then split it into focused gateway and agent changes.

## Reproduction Assessment

Yes, for a source-level reproduction. Live assistant deltas are buffered in ChatRunState maps, shutdown clears that state, and the recovery tests expect assistant-tailed transcripts to fail rather than resume.

## Solution Assessment

Unclear. Durable persistence or interruption recovery is the right direction, but the JSONL status sketch needs maintainer review against current abortedLastRun, pendingFinalDelivery, restart-sentinel, idempotency, privacy, and channel UX semantics.
