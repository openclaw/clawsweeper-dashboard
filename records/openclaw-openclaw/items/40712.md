## Summary

Keep open: current main has several restart-loss mitigations, but it still does not durably persist or resume arbitrary in-flight assistant generation after a crash, forced restart, or drain timeout. The remaining work is a feature/design umbrella rather than a narrow bug fix.

## What This Changes

The issue requests durable persistence and recovery of in-progress assistant generation across gateway restarts.

## Best Possible Solution

Keep this as the umbrella for durable in-flight generation recovery, building on safe restart drain, durable final delivery, restart-aborted session recovery, and channel-safe resume semantics.

## Reproduction Assessment

Yes, by source inspection for the remaining crash, force-restart, and drain-timeout cases: assistant deltas are process-local and assistant partial transcript tails are failed rather than resumed. A runtime repro should target a forced restart, crash, or response that outlives the configured drain window because clean safe restarts may now drain active work.

## Solution Assessment

Unclear. The issue's JSONL status-flushing sketch is plausible, but the narrower maintainable direction is likely to extend the existing restart drain, durable final-delivery, restart-aborted recovery, and follow-up/channel delivery contracts instead of adding a parallel raw JSONL status system.
