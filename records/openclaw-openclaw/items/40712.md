## Summary

Keep open: current main has restart draining and post-restart main-session recovery, but it still does not implement durable partial assistant-generation persistence/resume, and the related manifest PR is open and only covers part of the requested design.

## What This Changes

Persist and recover in-flight assistant generation so gateway restarts can resume or complete interrupted replies instead of losing partial output.

## Best Possible Solution

Keep this issue as the umbrella until maintainers finish a durable recovery design, likely by landing or replacing the drain-manifest work and adding a boot-time consumer plus regression coverage for partial assistant tails and channel delivery.

## Reproduction Assessment

Yes, by source inspection rather than a live run: chat deltas are held in process-local maps and cleared on shutdown, while current recovery resumes only selected transcript tails. The user-facing scenario can be live-validated with a long response plus `openclaw gateway restart`, but this read-only review did not run it.

## Solution Assessment

Unclear. The requested JSONL status flushing is plausible, but current main already has restart draining and main-session recovery, and the safer path is to align any durable partial-generation work with the open drain-manifest/recovery design instead of adding a parallel transcript schema ad hoc.
