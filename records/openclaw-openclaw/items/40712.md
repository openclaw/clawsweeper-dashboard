## Summary

Keep open: current main has partial restart recovery, but mid-stream assistant deltas are still process-local and assistant-tailed transcripts are intentionally non-resumable, so the requested durable generation persistence/replay is not implemented.

## What This Changes

The issue requests durable persistence and resume/recovery for in-progress assistant message generation after a gateway restart, including partial streaming state and startup delivery handling.

## Best Possible Solution

Define a durable in-flight generation checkpoint and recovery contract that composes with abortedLastRun, pendingFinalDelivery, safe restart drain, and channel-safe interruption UX, then split it into focused gateway/agent changes coordinated with the open restart recovery RFCs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: live assistant deltas are buffered in ChatRunState maps, the shutdown path clears that state, and assistant-tailed transcripts are failed rather than resumed; I did not run a live restart in this read-only review.

## Solution Assessment

Unclear. Durable persistence or interruption recovery is the right direction, but the issue's JSONL status sketch needs maintainer review against current abortedLastRun, pendingFinalDelivery, safe restart drain, hidden resume, and channel UX semantics.
