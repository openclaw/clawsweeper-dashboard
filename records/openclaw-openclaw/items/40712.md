## Summary

Keep open: current main mitigates clean restarts by draining active work and has restart-sentinel/session-lock recovery pieces, but it still does not durably persist and resume arbitrary in-flight assistant generation after crash, forced restart, or drain timeout. The related drain-manifest PR remains open and only covers part of the requested design.

## What This Changes

The issue requests durable recovery or resumption of in-progress assistant replies after gateway restart, including persistent streaming state, message status, and boot-time recovery.

## Best Possible Solution

Keep this as the umbrella until maintainers align one restart-recovery design that extends the existing drain, sentinel, session-lock, and abort-partial paths without ad hoc transcript schema changes.

## Reproduction Assessment

Yes, by source inspection for crash, forced restart, or restart drain timeout: live assistant deltas are process-local and cleared on close, while restart recovery refuses assistant partial tails. Ordinary clean restarts now drain active work for a bounded budget, so the current repro should specify a crash/force/timeout or a response longer than the drain window.

## Solution Assessment

Unclear. The requested JSONL status flushing is plausible, but the transcript is a parentId DAG with a documented appendMessage boundary, so the safer solution is to extend the existing restart-sentinel/drain-manifest/session-recovery design rather than add a parallel raw JSONL status path.
