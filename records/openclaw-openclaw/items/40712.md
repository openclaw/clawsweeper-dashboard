## Summary

Keep open: current main mitigates clean restarts by draining active work and has limited post-restart main-session recovery, but it still does not provide durable partial assistant-generation persistence or resume after forced restarts, drain timeouts, or crashes. The related drain-manifest PR is open and explicitly covers only part of the design.

## What This Changes

Request durable persistence and recovery for in-progress assistant generation so interrupted gateway restarts can resume or deliver the response.

## Best Possible Solution

Keep this issue as the umbrella until maintainers settle one durable recovery design that extends the existing drain and main-session recovery paths with tested partial assistant-tail and channel-delivery behavior.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current main stores live assistant deltas in process-local maps that are cleared on shutdown, and restart recovery rejects assistant partial tails. A live check would be a long response interrupted by a forced restart, drain timeout, or crash.

## Solution Assessment

Unclear. The requested JSONL status flushing is plausible, but the safer maintainable solution should align with the existing restart drain, main-session recovery, and the open drain-manifest work instead of adding a parallel schema ad hoc.
