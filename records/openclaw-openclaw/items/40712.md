## Summary

Keep open: current main has partial restart-loss mitigations, but arbitrary in-flight assistant generation is still not durably persisted or resumed after crash, forced restart, or shutdown-drain timeout. The linked drain-manifest PR is a useful partial implementation, not a complete fix for the product behavior.

## What This Changes

Persist and resume or complete in-progress LLM responses when the gateway restarts during generation.

## Best Possible Solution

Keep this as the umbrella for durable in-flight generation recovery, building on restart drain, pending final delivery, restart-aborted recovery, and channel-safe resume/notification semantics.

## Reproduction Assessment

Yes. Source inspection shows assistant deltas remain in process-local chatRunState buffers that are cleared on shutdown, and restart recovery intentionally fails a transcript tail ending in an assistant partial.

## Solution Assessment

Unclear. The issue's JSONL status-flushing sketch is plausible, but the safer direction is likely to extend the existing restart drain, durable final-delivery, restart-aborted recovery, and channel delivery contracts rather than adding a parallel raw transcript-status system.
