## Summary

Keep open: current main mitigates clean restarts with active-work draining, restart-aborted session recovery, and durable final-delivery capture, but it still does not durably persist or resume arbitrary in-flight assistant generation after crash, forced restart, or drain timeout.

## What This Changes

The issue requests durable recovery or resumption of in-progress assistant message generation when the gateway restarts during streaming.

## Best Possible Solution

Keep this as the umbrella for durable in-flight generation recovery until maintainers choose a design that builds on restart drain, restart-aborted session recovery, durable final delivery, and channel-safe resume semantics.

## Reproduction Assessment

Yes, by source inspection for the remaining crash, force-restart, and drain-timeout cases: assistant deltas remain process-local and assistant partial transcript tails are failed rather than resumed. A clean restart may now drain active work, so a runtime repro should target forced restart, crash, or a response longer than the configured drain window.

## Solution Assessment

Unclear. The issue's JSONL status-flushing sketch is plausible, but the safer maintainable path is probably to extend the existing restart drain, durable final-delivery, restart-aborted recovery, and channel delivery queue contracts instead of adding a parallel raw JSONL status system.
