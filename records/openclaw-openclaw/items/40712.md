## Summary

Keep open: current main has useful restart-loss mitigations for completed deliveries and marked restart-aborted sessions, but arbitrary in-flight assistant generation is still process-local and assistant partial transcript tails are intentionally not resumed. The related drain-manifest PR is open and partial, so it is not a reason to close this umbrella feature.

## What This Changes

The issue requests durable persistence, recovery, or user notification for in-progress assistant message generation interrupted by a gateway restart.

## Best Possible Solution

Keep this as the focused umbrella for durable in-flight assistant generation recovery, then split implementation into restart-time marking/checkpointing, boot-time recovery, final-delivery replay, and channel-safe user notification semantics.

## Reproduction Assessment

Yes, source inspection provides a high-confidence current-main reproduction path: assistant deltas are stored in process-local chatRunState buffers that are cleared on shutdown, and restart recovery fails transcript tails ending in assistant partials.

## Solution Assessment

Unclear: the JSONL status-flushing sketch is plausible, but the better maintainable direction is likely to extend existing restart-aborted markers, delivery queues, and recovery contracts instead of adding a parallel raw transcript status system.
