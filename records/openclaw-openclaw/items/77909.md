## Summary

Keep open: current main still routes fetched Feishu merge_forward sub-messages through a formatter that has no interactive-card extraction path, so the reported `[interactive]` data-loss behavior remains source-reproducible.

## What This Changes

The issue asks OpenClaw to preserve readable Feishu interactive/card content inside merge_forward messages instead of reducing it to placeholder text.

## Best Possible Solution

Add a Feishu-plugin-local merge_forward fix that shares the interactive-card text extraction path, covers representative card/table payloads, and preserves existing fallback behavior.

## Reproduction Assessment

Yes, at source level. A fetched merge_forward sub-message with `msg_type: "interactive"` reaches `formatSubMessageContent()` on current main and falls through to `[interactive]`; no live Feishu tenant was exercised here.

## Solution Assessment

Yes, the narrow maintainable direction is to wire merge_forward sub-message formatting through a shared Feishu interactive-card parser with focused regression tests. A core/channel API change is not needed for this reported behavior.
