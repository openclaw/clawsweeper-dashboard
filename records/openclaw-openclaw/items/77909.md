## Summary

Keep open: current main still fetches Feishu merge_forward API items, but interactive sub-messages go through a merge_forward-specific formatter with no interactive case and still become `[interactive]`; the related interactive-card parser work is separate and not wired into this path.

## What This Changes

The issue requests preserving original text from Feishu merge_forward sub-messages, especially card/table/interactive content, instead of placeholder output.

## Best Possible Solution

Centralize or expose the Feishu interactive-card content extractor inside the plugin, use it for merge_forward sub-messages, and add focused regression coverage for interactive/table card payloads returned by `im.message.get`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a fetched merge_forward item with `msg_type: "interactive"` reaches `formatSubMessageContent` and falls through to `[interactive]`; I did not run a live Feishu workspace reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-local parser reuse plus regression tests, not a core channel API change or new Feishu permission.
