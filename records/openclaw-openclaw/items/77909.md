## Summary

Keep open. Current main and the latest release still route Feishu `merge_forward` sub-messages with `msg_type: "interactive"` to the generic `[interactive]` formatter, while the only linked fix PR closed unmerged and only replaced that with another placeholder.

## What This Changes

The issue asks the Feishu plugin to preserve original text from `merge_forward` messages containing cards, tables, or interactive content instead of delivering `[interactive]` or a placeholder.

## Best Possible Solution

Fix Feishu merge_forward expansion inside the plugin by requesting full card content when fetching message details and reusing the existing interactive-card parser for sub-messages, with focused regression tests and live Feishu proof.

## Reproduction Assessment

Yes, from source inspection: a fetched merge_forward item with `upper_message_id` and `msg_type: "interactive"` reaches `formatSubMessageContent` and current main returns `[interactive]`. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution because the merge_forward formatter still has a placeholder-only fallback while `send.ts` already has a parser for extractable interactive-card text. The closed placeholder PR was also too narrow because it would still strip the original card content.
