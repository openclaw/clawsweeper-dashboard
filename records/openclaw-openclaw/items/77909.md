## Summary

Keep open: current main still routes fetched Feishu merge_forward sub-messages through a formatter that has no `interactive` case, so the reported `[interactive]` content loss remains source-reproducible. The direct fix attempt at https://github.com/openclaw/openclaw/pull/77917 was closed unmerged, and no exact open closing PR currently resolves this path.

## What This Changes

The issue requests that Feishu merge_forward sub-messages preserve extractable text from interactive cards, tables, and similar card payloads instead of rendering `[interactive]`.

## Best Possible Solution

Add a Feishu-plugin-local merge_forward fix that shares interactive-card text extraction, covers representative card/table payloads, and preserves existing fallback behavior.

## Reproduction Assessment

Yes, at source level. A fetched merge_forward sub-message with `msg_type: "interactive"` reaches `formatSubMessageContent()` on current main and falls through to `[interactive]`; no live Feishu tenant was exercised here.

## Solution Assessment

Yes. The narrow maintainable direction is to wire merge_forward sub-message formatting through a shared Feishu interactive-card parser with focused regression tests; a core/channel API change is not needed for this reported behavior.
