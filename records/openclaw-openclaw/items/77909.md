## Summary

Keep open: current main still has a Feishu merge_forward-specific parser path that turns interactive sub-messages into placeholders, and the related interactive-card parser work is not wired into that path.

## What This Changes

The issue reports that Feishu merge_forward messages containing cards or tables are fetched but collapsed to placeholders like `[interactive]` instead of readable original content.

## Best Possible Solution

Fix the Feishu plugin parser so merge_forward sub-messages reuse the existing interactive-card extraction path and add focused regression coverage for interactive/table card payloads.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a fetched merge_forward sub-message with `msg_type: "interactive"` reaches `formatSubMessageContent` and falls through to `[interactive]`. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to wire merge_forward sub-message formatting through the existing Feishu interactive-card parser, then cover this path with targeted plugin tests. A broader core/channel API change is not needed for the reported behavior.
