## Summary

Keep open. Current main and the latest release still route Feishu merge_forward interactive sub-messages through a formatter that falls back to `[interactive]`, and the linked fixing PR is open but not yet merged.

## What This Changes

The issue reports that Feishu merge_forward messages containing interactive cards or tables are fetched from the API but rendered to the agent as `[interactive]` or another placeholder instead of original content.

## Best Possible Solution

Land or replace the linked Feishu plugin PR with a narrow fix that reuses the existing interactive-card text extraction for merge_forward sub-messages and proves the forwarded card/table content reaches the agent.

## Reproduction Assessment

Yes. A source-level repro is clear: a fetched merge_forward item with `upper_message_id` and `msg_type: "interactive"` reaches `formatSubMessageContent` and current main returns `[interactive]`; I did not run a live Feishu tenant reproduction.

## Solution Assessment

Yes, a narrow Feishu plugin parser fix is the right direction, but current main does not have it yet. The linked PR should share the existing interactive-card parser and add proof rather than only returning a card title or placeholder.
