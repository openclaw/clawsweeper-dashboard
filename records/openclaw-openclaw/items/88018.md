## Summary

Keep this PR open: current main still has the legacy-session save gate that can treat an unreadable target store as empty, while the PR’s latest head adds the missing target-readability guard and fixes the earlier false `result.changes` report. I found no blocking patch defect, but the migration behavior is compatibility/session-state sensitive because corrupt-target users now need operator repair instead of silent overwrite.

## What This Changes

The PR changes legacy session migration to preserve a corrupt target session store and legacy backup instead of overwriting target data, with a regression test for that corrupt-target path.

## Best Possible Solution

Land the guard if maintainers accept the safer manual-repair behavior for corrupt target stores, keeping the existing happy path unchanged and preserving both data sources on the corrupt-target path.

## Reproduction Assessment

Yes. The source path is clear without running tests here: current main parses a corrupt target as `{ ok: false, store: {} }` but still saves when the legacy parse succeeds, and the PR’s regression test encodes the corrupt-target setup through `runLegacyStateMigrations`.

## Solution Assessment

Yes. The target-readability guard is a narrow fix for the destructive write, and the follow-up commit correctly limits reported session changes to committed saves; the remaining question is maintainer acceptance of the manual-repair migration behavior.
