## Summary

Keep open: the data-preserving guard is directionally right, but as written it can report a direct-chat session migration even when the corrupt-target path skips the target store save, so the PR needs a small result-reporting fix before merge.

## What This Changes

The PR adds a target-store readability guard to legacy session migration and a regression test so corrupt target session stores are warned about and left untouched instead of overwritten.

## Best Possible Solution

Preserve the corrupt target and legacy store, but only report direct-chat session migration after an actual target-store save and cover that result in the corrupt-target regression test before merge.

## Reproduction Assessment

Yes, by source inspection: current main treats a corrupt target store as `{ store: {}, ok: false }`, then still saves when the legacy store parses successfully. I did not run the probe because this was a read-only review, but the PR body also supplies before/after live output.

## Solution Assessment

No as submitted. The core guard is the right fix shape, but the result reporting must be tied to the committed save path so doctor/startup logs do not claim an unsaved session migration.
