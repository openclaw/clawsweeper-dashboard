## Best Possible Solution

Define one guarded context-assembly mutation contract that matches current prompt assembly, applies explicit trust policies, clones and validates message payloads, documents the SDK surface, and is tested alongside the related hook proposals.

## Solution Assessment

No. A guarded assembly hook may be useful, but this patch is not the best mergeable shape until it rebases onto current prompt assembly, enforces the raw-context trust boundary, clones and validates hook payloads, documents the public contract, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate before_assemble with prompt-injection policy:** `src/plugins/hook-types.ts:177`
  - body: `before_assemble` can replace the model-visible message list, but the latest PR head classifies it only as a conversation hook. A non-bundled plugin with `allowConversationAccess=true` and `allowPromptInjection=false` can still rewrite or inject context, bypassing the policy used for other prompt-mutating hooks.
  - confidence: 0.92
- **[P2] Pass cloned messages into assembly hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2605`
  - body: The hook payload passes `activeSession.messages` directly to plugin code. A handler that mutates the array and then returns void or throws can still alter session state outside the returned-result contract; clone and normalize the payload, then only apply validated returned messages.
  - confidence: 0.86
- **[P2] Report the runtime system prompt to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2886`
  - body: `after_assemble` is described as observing the final model input, but it receives `systemPromptText` rather than the runtime-context-composed system prompt current main now builds for hook/model submission. Observability plugins would miss context visible to the model.
  - confidence: 0.9
