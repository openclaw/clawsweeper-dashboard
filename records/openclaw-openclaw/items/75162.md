## Summary

Keep open: current main does not implement these assembly hooks, and the PR is potentially useful source work, but it should not merge until the public API, security policy, and correctness issues are resolved.

## What This Changes

The PR adds typed `before_assemble` and `after_assemble` plugin hooks, HookRunner methods, embedded-runner call sites around context assembly, and a stray `extensions/aicm` lockfile importer.

## Best Possible Solution

Define the intended core extension point first, then implement it with explicit hook policy classification, honest model-input semantics, SDK/docs/contract tests, and a changelog entry, or consolidate the capability with #61485/context-engine if that is the maintainable path.

## Reproduction Assessment

Yes. Static review is sufficient: current main lacks the hook names, and the latest PR diff adds raw-message hooks without the existing policy classifications, without accumulated event evolution, and without a complete final model-input payload.

## Solution Assessment

No. A guarded assembly extension point may be reasonable, but this patch is not the best implementation because it bypasses current safety gates and does not accurately implement the stated sequential or final-context behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate assembly hooks through hook policies:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks can read and replace full conversation messages, but the diff only adds their names to the generic hook list. Because they are absent from the prompt-injection and conversation-access hook classifications, a non-bundled plugin can register raw transcript hooks without the existing `allowPromptInjection` and `allowConversationAccess` gates.
  - confidence: 0.93
- **[P2] Pass accumulated messages to each assembly handler:** `src/plugins/hooks.ts:1318-1325`
  - body: `runBeforeAssemble` promises chained transformations, but `runModifyingHook` invokes every handler with the original event. A later plugin therefore sees the unmodified message list and can undo an earlier plugin's deletion or reorder when it returns its own `messages`.
  - confidence: 0.9
- **[P2] Expose the actual model input to after_assemble:** `src/agents/pi-embedded-runner/run/attempt.ts:2878`
  - body: The `after_assemble` event is described as the final context before the LLM call, but the payload only includes `activeSession.messages`. The actual request also includes system/runtime prompt text and image context, so observers will miss part of the model submission.
  - confidence: 0.84
- **[P3] Remove the stale AICM lockfile importer:** `pnpm-lock.yaml:234`
  - body: The PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/plugins/hook-types.ts:106-107`
  - body: This is a user-facing plugin API feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires user-facing `feat` changes to include one before merge.
  - confidence: 0.86
